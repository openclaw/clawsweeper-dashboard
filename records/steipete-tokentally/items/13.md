## What This Changes

This PR updates the OpenRouter catalog pricing converter to parse string or numeric USD-per-token prices and replaces the OpenRouter tests with string, numeric, zero, and malformed-price cases.

## Best Possible Solution

Land the string-price OpenRouter fix while preserving the existing numeric-input contract, or make the numeric semantic change an explicit maintainer-approved breaking change with focused tests and migration notes.

## Reproduction Assessment

Yes, from source and live-data inspection: current main only accepts numeric OpenRouter prices, while the live OpenRouter catalog currently returns priced models with string prompt/completion values, so the current guard accepts none.

## Solution Assessment

Partially: parsing OpenRouter string prices as USD per token is the narrow maintainable fix, but silently treating numeric catalog values as per-token is not clearly the safest upgrade path for the exported converter.
