## Summary

Keep this issue open: current main intentionally provides diagnostics only and still marks encrypted-JSON import unsupported. The requested implementation is planned in the repository specification, but its Keychain identity, helper packaging, command UX, and plaintext-lifetime boundaries require explicit product and security approval before coding.

## What This Changes

Add an explicitly invoked macOS encrypted-JSON unlock/import flow for modern Granola profiles, with controlled Keychain prompts and strict plaintext-handling rules.

## Best Possible Solution

Approve one explicit-only command contract, then implement a narrowly scoped macOS helper that returns only requested decrypted data in memory and validate successful import, denied prompts, and secret-free output on a redacted live profile.

## Reproduction Assessment

Yes at the source level: current main deterministically reports encrypted JSON as unimplemented, and regression tests enforce that diagnostic-only behavior. End-to-end reproduction still requires the real encrypted-only macOS profile described in the issue.

## Solution Assessment

Yes directionally: a separate explicitly invoked helper is the narrowest architecture consistent with the repository’s existing security policy. The exact command, application identity, packaging strategy, and plaintext rules must be approved before implementation.
