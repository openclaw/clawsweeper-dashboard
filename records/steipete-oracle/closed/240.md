## Summary

Keep open: current main and v0.13.0 still lack generated-bundle-aware readiness for extensionless ChatGPT file tiles, while this PR threads explicit bundle metadata into the pre-send gate with focused regressions and live browser proof. I found no blocking patch defect; the remaining merge risk is that this still depends on ChatGPT’s external DOM shape.

## What This Changes

The PR marks Oracle-generated browser bundles, passes that metadata into ChatGPT pre-send attachment readiness, accepts stem-only bundle tiles only for generated bundles, and adds matcher regressions plus a changelog entry.

## Best Possible Solution

Land the narrow generated-bundle marker and matcher change after normal maintainer checks, while keeping future ChatGPT DOM drift covered by focused browser smokes and matcher tests.

## Reproduction Assessment

Yes, this is source-reproducible: current main passes only plain filenames into the readiness matcher, so an extensionless `attachments-bundle` tile cannot satisfy the exact/duplicate/ellipsis checks for `attachments-bundle.txt` or `.zip`. The PR body also includes authenticated browser proof of the edge case and the fixed behavior.

## Solution Assessment

Yes, the proposed path is the narrowest maintainable fix I found: carry an explicit generated-bundle marker from bundle creation and allow stem-only matching only for that marked path. It avoids relaxing normal user-file matching while preserving existing duplicate-renamed and ellipsized behavior.
