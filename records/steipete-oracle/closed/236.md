## Summary

Keep open: current main still separates API `--followup` from launch-time browser follow-ups, and the saved browser conversation recovery path is read-only. The request remains a real browser-session continuity gap, but it needs maintainer direction on the public CLI contract before implementation.

## What This Changes

Allow a new CLI invocation to continue a stored ChatGPT browser session as the next turn in the same conversation, with hydration-safe resume behavior and better reattach guidance.

## Best Possible Solution

Add a maintainer-approved browser resume follow-up path that reopens the saved ChatGPT conversation, waits for hydrated prior turns to stabilize, submits the new prompt and files as the next turn, and preserves existing API follow-up behavior.

## Reproduction Assessment

Yes, from source inspection: `--followup` rejects non-API engines before session creation, while `--browser-follow-up` is only consumed by the active browser run. I did not run a live ChatGPT smoke because browser live tests are opt-in and the source path is clear.

## Solution Assessment

Unclear until maintainers choose the CLI contract: extending `--followup` for browser sessions is coherent for users, but it changes the current API-only boundary. The narrow maintainable path is to share saved-URL recovery and browser submit primitives while preserving API response-id behavior.
