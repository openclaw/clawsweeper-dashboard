## Summary

Keep this PR open: it has useful proof and tests for the generated-bundle path, but the proposed matcher still relaxes readiness for any user upload named `attachments-bundle.txt` or `.zip`, which contradicts the PR’s stated boundary.

## What This Changes

The PR relaxes ChatGPT browser attachment readiness for Oracle-generated `attachments-bundle.txt` and `.zip` chips and adds focused expression tests plus a changelog note.

## Best Possible Solution

Carry a generated-bundle marker from bundle creation through the browser submission/readiness APIs so only Oracle-created bundles get stem-only matching, while user files keep exact or existing duplicate-renamed matching.

## Reproduction Assessment

Yes at source level: current main does not accept a stem-only `attachments-bundle` chip for `attachments-bundle.txt`, and the PR tests demonstrate the intended generated-bundle path. I did not run a live ChatGPT regression in this read-only review.

## Solution Assessment

No: the direction is right, but basename-derived `generatedBundle` is not the narrowest maintainable boundary. The safer fix is to pass explicit bundle metadata into the readiness matcher.
