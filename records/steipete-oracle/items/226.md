## Summary

Keep open. The recovery idea is useful, but the patch should not merge as-is because the fallback can reopen stale shell or external URLs in a persistent browser profile, and the PR body does not include inspectable real behavior proof for the after-fix recovery path.

## What This Changes

The PR adds default-on `oracle session <id> --harvest/--live` recovery by relaunching Chrome from saved session metadata, plus `--no-recover`, tests, and a changelog entry.

## Best Possible Solution

Land a narrower recovery path that only reopens validated ChatGPT conversation URLs, prefers a valid harvested conversation URL over stale runtime shell URLs, preserves cleanup on retry failures, and includes redacted real recovery proof.

## Reproduction Assessment

Yes for the blocking findings: source inspection shows the recovery selector accepts any `http` candidate and prefers `runtime.tabUrl`, while current main's recoverability helper rejects non-conversation ChatGPT shell URLs and external URLs. I did not run browser smokes in this read-only review.

## Solution Assessment

No as written. The maintainable path is to share or mirror the existing recoverable-conversation checks, prefer a validated conversation URL, and wrap the recovered retry in cleanup before accepting the default-on behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.83

Full review comments:

- **[P2] Validate recovery URLs before relaunching Chrome:** `src/browser/recoverConversation.ts:18-20`
  - body: Current main only treats ChatGPT conversation URLs or conversation IDs as recoverable and explicitly rejects home/project shell URLs and external URLs. This resolver returns the first `http` candidate and prefers `runtime.tabUrl`, so a stale `https://chatgpt.com/`, project shell URL, or external metadata URL can be opened in the manual-login profile and then persisted as an empty/detached harvest instead of recovering the saved response. Reuse the existing recoverability logic and prefer a valid conversation URL.
  - confidence: 0.87
- **[P3] Wrap recovered harvest retries in cleanup:** `src/cli/browserTabs.ts:267-274`
  - body: After recovery launches Chrome and assigns `recoveredChrome`, the retry harvest runs before the `finally` block that kills it when `closeAfterRecover` is true. If that retry throws, the internal close option is ignored and the newly launched Chrome is left running; move the retry under the cleanup `try/finally`.
  - confidence: 0.78
