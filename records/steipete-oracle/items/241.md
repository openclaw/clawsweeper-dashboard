## Label Justifications

- `P1`: The report describes a latest-release browser-mode login failure that can block logged-in remote Chrome users on every run.
- `impact:auth-provider`: The affected behavior is ChatGPT session/auth detection in the browser engine.

## Summary

Keep open: current main still treats plain `/backend-api/me` 401/403 responses as authoritative logout signals, and existing tests assert that 401 plus authenticated DOM must fail, so the reported browser-mode false logout remains actionable.

## What This Changes

The issue asks to stop browser mode from rejecting logged-in remote Chrome sessions when `/backend-api/me` returns 401/403 without bearer auth.

## Best Possible Solution

Update the browser login probe to use a cookie-authenticated session signal or a strongly gated 401/403 DOM fallback, preserving auth-page/login-CTA failures and adding regression coverage for remote Chrome login detection.

## Reproduction Assessment

Source-level yes, live no: current main has a clear unit-level path where 401 plus authenticated DOM returns `ok: false`, matching the reported failure condition. A live ChatGPT session would still be needed to confirm the endpoint now returns 401 for logged-in users broadly.

## Solution Assessment

Yes, the login probe is the right repair surface. The safest fix is likely to probe `/api/auth/session` for a cookie-authenticated user or only accept 401/403 through the existing strong DOM gate with focused regression tests.
