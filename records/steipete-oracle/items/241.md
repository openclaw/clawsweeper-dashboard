## Summary

Current main still treats plain `/backend-api/me` 401/403 responses as authoritative logout signals even when the page has strong authenticated-DOM evidence. The implementation, focused tests, and browser-mode documentation all preserve the reported failure, and no newer canonical issue or PR owns the remaining fix.

## What This Changes

Correct browser-mode login detection when a signed-in ChatGPT session receives 401/403 from the cookie-only backend probe.

## Best Possible Solution

Use a stable cookie-authenticated session signal such as `/api/auth/session`, or admit plain 401/403 only behind the existing strong authenticated-DOM gate, while preserving authoritative rejection of auth pages, visible login CTAs, and composer-only guest shells.

## Reproduction Assessment

Yes at source level: current main and its focused tests deterministically reject plain 401/403 despite authenticated DOM, matching the reporter's endpoint-level reproduction. A live signed-in remote-Chrome run was not performed during this read-only review.

## Solution Assessment

Yes in scope: a narrow probe correction with the existing DOM safety gates is the maintainable solution. Probing `/api/auth/session` is preferable if a real browser smoke confirms its stable response contract; otherwise the guarded 401/403 fallback is the smaller repair.
