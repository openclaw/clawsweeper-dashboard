## Label Justifications

- `P1`: The regression prevents affected logged-in remote-Chrome users from starting any browser-mode run.
- `impact:auth-provider`: Oracle misclassifies a valid ChatGPT browser session as unauthenticated because of its backend auth probe.

## Summary

Current main still treats plain `/backend-api/me` 401/403 responses as authoritative logout signals despite authenticated DOM evidence, matching the reported remote-browser failure. The fallback shipped in v0.13.0 deliberately preserved this gap, so the issue remains necessary.

## What This Changes

Fix browser-mode login detection when a logged-in ChatGPT session receives 401/403 from the cookie-only `/backend-api/me` probe.

## Best Possible Solution

Use `/api/auth/session` as the primary cookie-authenticated signal if stable, with the existing DOM checks as a guarded fallback; preserve auth-page and login-CTA rejection and cover 401/403, guest composer, and remote-session cases in tests and a live browser smoke.

## Reproduction Assessment

Yes at source level: a 401/403 response with authenticated DOM deterministically fails on current main, and focused tests explicitly encode that outcome. The reporter also supplied a concrete live-console path, though this review did not independently run it.

## Solution Assessment

Yes in direction, but probing `/api/auth/session` is preferable to broadly normalizing all 401/403 responses because it directly verifies the cookie-authenticated session; a guarded DOM fallback remains useful for transient endpoint failures.
