## Summary

Keep this PR open: it targets a real browser model-selection failure, but the proposed menu-filter helper still lets unrelated Radix menus satisfy picker-open detection, so the main live failure can persist.

## What This Changes

The PR changes ChatGPT browser model selection to scope picker detection to real model-switcher rows, avoid Thinking/Instant confusion, and add model-selection tests.

## Best Possible Solution

Make picker-open detection and option scanning require an actual model-switcher menu when any non-picker Radix menus are present, while preserving a carefully scoped text-only fallback for real picker rows that lack test ids.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: on current main, a visible non-picker '[role="menu"]' or '[data-radix-collection-root]' causes ensureMenuOpen to treat the menu as open and findBestOption to scan unrelated menu items. I did not run a live ChatGPT browser smoke because live OpenAI browser tests are opt-in and can take several minutes.

## Solution Assessment

No, the PR moves in the right direction but the shared queryPickerMenus fallback still lets non-picker menus satisfy the open-menu check. The narrow fix is to separate strict picker-open detection from any text-only fallback used after a real picker root is known.
