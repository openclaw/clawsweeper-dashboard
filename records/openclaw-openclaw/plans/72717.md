# Coding Plan for openclaw/openclaw#72717: Feature Request: SQLite FTS index for wiki_search to improve synthesis query performance

Source report: [records/openclaw-openclaw/items/72717.md](records/openclaw-openclaw/items/72717.md)

## Summary

Keep open: current `main` still implements `memory-wiki` search as digest/file-backed Markdown scanning with no SQLite/FTS index, no index refresh path, and no stale-index fallback behavior. The related https://github.com/openclaw/openclaw/issues/75312 is a separate shared-memory runtime bug, not a duplicate of this performance feature.

## Plan

Implement https://github.com/openclaw/openclaw/issues/72717 as a plugin-local, derived SQLite FTS cache for `memory-wiki` search. Keep Markdown pages and compiled digests as the source of truth; the FTS database must be a rebuildable read cache. Add hash or mtime stale checks, a rebuild path through `wiki compile`, and refresh/invalidation through `wiki_apply`, ingest, ChatGPT import/rollback, and imported-source sync paths that call compile. `wiki_search` should query the FTS cache first when available and current, preserve existing ranking/provenance fields as much as possible, and fall back to current file/digest scanning when SQLite/FTS is missing, stale, corrupt, or unsupported. Do not import core `src/**` internals from `extensions/memory-wiki`; keep the plugin boundary clean and keep Node and Bun runtime paths working. Update docs and add a `CHANGELOG.md` entry because this is user-visible search behavior/performance work. Treat https://github.com/openclaw/openclaw/issues/75312 as related-but-separate shared-memory fallback work, not part of this fix.
