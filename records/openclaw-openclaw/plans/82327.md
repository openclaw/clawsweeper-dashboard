# Coding Plan for openclaw/openclaw#82327: Codex app-server turns do not emit tool progress into Mattermost progress drafts

Source report: [records/openclaw-openclaw/items/82327.md](records/openclaw-openclaw/items/82327.md)

## Summary

Keep open. Current main still has a source-level gap for Codex DM turns on Mattermost: Codex defaults source replies to message-tool delivery, the shared dispatcher suppresses progress callbacks unless the channel opts in, and Mattermost does not set that opt-in even though it has draft progress handlers.

## Plan

Fix https://github.com/openclaw/openclaw/issues/82327. Mattermost progress drafts should receive Codex app-server `stream: "tool"` and item progress during message-tool-only/source-suppressed Codex turns. Add a scoped `allowProgressCallbacksWhenSourceDeliverySuppressed` opt-in to the Mattermost draft/progress reply options only when an editable draft/progress stream is active, mirroring Telegram's intent without changing final visible-reply policy or enabling standalone progress chatter. Add focused regression coverage proving Mattermost `streaming.mode: "progress"` forwards `onToolStart`/`onItemEvent` callbacks when `sourceReplyDeliveryMode: "message_tool_only"`, and that callbacks remain disabled when draft streaming/tool progress is off. Do not broaden Codex harness defaults, do not change `messages.visibleReplies`, and preserve existing finalization/delete behavior.
