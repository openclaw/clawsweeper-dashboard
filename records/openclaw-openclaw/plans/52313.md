# Coding Plan for openclaw/openclaw#52313: gateway restart/update can fail to come back when respawn reuses unstable package-manager paths

Source report: [records/openclaw-openclaw/items/52313.md](records/openclaw-openclaw/items/52313.md)

## Summary

Keep open: current main and v2026.5.18 still let the update-mode detached respawn reuse `process.execArgv + process.argv.slice(1)`, so a package-manager versioned OpenClaw `dist/*` realpath can still be spawned after package replacement. Ordinary unmanaged restarts were improved by https://github.com/openclaw/openclaw/pull/83138, but the `update.run` child respawn path remains unresolved; https://github.com/openclaw/openclaw/pull/52319 is useful prior work, not a merged fix.

## Plan

Fix https://github.com/openclaw/openclaw/issues/52313 by adding a focused update-respawn entrypoint planner. In `src/infra/process-respawn.ts`, stop blindly using `process.execArgv + process.argv.slice(1)` for package-manager OpenClaw `dist/*` realpaths during `respawnGatewayProcessForUpdate`; rewrite pnpm/OpenClaw package realpaths back to a stable `openclaw.mjs` or package-root entrypoint where safe. Preserve dev/source entrypoints such as `src/entry.ts`, preserve Bun/source-checkout behavior, and do not undo the ordinary unmanaged in-process restart behavior from https://github.com/openclaw/openclaw/pull/83138. Use existing service entrypoint helpers/patterns where they fit, add regression tests for pnpm versioned realpath rewriting, package-root fallback, dev passthrough, and current update health fallback, and add a single changelog fix entry if the behavior changes user-visible restart reliability.
