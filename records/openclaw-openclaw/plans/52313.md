# Coding Plan for openclaw/openclaw#52313: gateway restart/update can fail to come back when respawn reuses unstable package-manager paths

Source report: [records/openclaw-openclaw/items/52313.md](records/openclaw-openclaw/items/52313.md)

## Summary

Keep open: the current checkout and latest release still build detached gateway respawns from `process.execPath`, `process.execArgv`, and `process.argv.slice(1)`, so an OpenClaw pnpm versioned `dist/*` realpath can still be reused after a package update. The closed reference PR usefully explored the fix, but it was not merged and no related open issue fully supersedes this stable-entrypoint problem.

## Plan

Fix https://github.com/openclaw/openclaw/issues/52313 with a narrow respawn-entrypoint planner. Current `src/infra/process-respawn.ts` reuses `process.execArgv` and `process.argv.slice(1)`, so packaged gateways launched through pnpm versioned realpaths can respawn into a path removed by self-update. Rewrite OpenClaw package-manager realpaths such as `node_modules/.pnpm/openclaw@*/node_modules/openclaw/dist/*` to a stable package wrapper or symlinked package entrypoint such as `node_modules/openclaw/openclaw.mjs`; when a package root is known, prefer `<packageRoot>/openclaw.mjs`; preserve dev/source entrypoints like `src/entry.ts` and avoid broad changes to custom-supervisor restart policy or package-swap behavior tracked by other issues. Use https://github.com/openclaw/openclaw/pull/52319 only as reference, not as a branch to replay, because it carried broad unrelated churn. Add focused regression tests in `src/infra/process-respawn.test.ts`, consider reusing or extracting existing stable-entrypoint logic from `src/daemon/program-args.ts` / `src/daemon/gateway-entrypoint.ts`, preserve the update health fallback in `src/cli/gateway-cli/run-loop.ts`, and add a single-line `CHANGELOG.md` fix entry crediting the human reporter when appropriate.
