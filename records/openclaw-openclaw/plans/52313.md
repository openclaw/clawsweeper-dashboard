## Summary

Keep open: current main has fixed some adjacent restart availability problems, but update-mode respawn still builds detached children from the live Node executable and current argv without planning a stable OpenClaw entrypoint. The linked stable-wrapper PR that directly addressed this was closed unmerged, and the currently open adjacent PR only covers managed service entrypoint selection.

## Plan

Fix https://github.com/openclaw/openclaw/issues/52313. Current update-mode respawn in `src/infra/process-respawn.ts` reuses `process.execArgv` and `process.argv.slice(1)`, so a pnpm versioned OpenClaw realpath can be reused after a package swap. Add a focused respawn entrypoint planner for `respawnGatewayProcessForUpdate()` / `spawnDetachedGatewayProcess()` that rewrites OpenClaw pnpm realpaths and known package roots to stable wrapper/install entrypoints, preserves `src/entry.ts` dev/source and Bun behavior, preserves supervisor and `OPENCLAW_NO_RESPAWN` semantics, and does not change ordinary unmanaged restart behavior from https://github.com/openclaw/openclaw/pull/83138. Add regression tests in `src/infra/process-respawn.test.ts` for pnpm versioned realpath rewrite, package-root wrapper fallback, dev/source passthrough, and current Windows/update semantics; reuse `src/daemon/gateway-entrypoint.ts` helpers or factor a small generic helper if needed. Coordinate with adjacent service-entrypoint work in https://github.com/openclaw/openclaw/pull/66502 rather than duplicating conflicting candidate ordering. Add a `CHANGELOG.md` fix entry because this is user-visible gateway availability.
