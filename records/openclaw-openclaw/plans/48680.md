# Coding Plan for openclaw/openclaw#48680: [Bug] Model fallback treats HTTP 403 business rejection as 'candidate_succeeded', skipping remaining fallback models

Source report: [records/openclaw-openclaw/items/48680.md](records/openclaw-openclaw/items/48680.md)

## Summary

Keep open. Current main still has a source-reproducible fallback gap for the reported provider business-denial shape: the fallback wrapper only fails a normal result when the result classifier returns a classification, and the current embedded classifier/CJK failover patterns still do not cover the reported `success:false` / `CE-011` / Chinese denial markers.

## Plan

Fix https://github.com/openclaw/openclaw/issues/48680. Current main does not classify provider business-denial payloads like `{"success":false,"message":"当前ak因违规请求被禁止访问该模型...","code":"CE-011"}` as failover-worthy, so an embedded run can return normally with `isError=true` and the outer model fallback logs `candidate_succeeded` instead of trying the next model. Add the narrow semantic classification in the embedded/failover boundary, likely `src/agents/pi-embedded-helpers/failover-matches.ts`, `src/agents/pi-embedded-helpers/errors.ts`, or `src/agents/pi-embedded-runner/result-fallback-classifier.ts`. Preserve guards for visible replies, outbound delivery evidence, hook blocks, aborts, and side-effectful attempts; do not broaden this into retrying all unknown `isError` payloads. Add regression coverage for the reported CE-011/Chinese denial text and for `runWithModelFallback + runEmbeddedPiAgent` advancing to the next candidate. Include a `CHANGELOG.md` fix entry unless maintainers direct otherwise.
