# Coding Plan for openclaw/openclaw#78727: [Bug]: CLI device pairing: scope upgrades trigger infinite request loop; token rotation drops prior scopes

Source report: [records/openclaw-openclaw/items/78727.md](records/openclaw-openclaw/items/78727.md)

## Summary

Keep open. Current main still has a source-visible token-rotation gap: approval merges `approvedScopes`, but the rotated role token is minted from the pending request scopes first, so an upgrade can drop previously usable scopes and keep the client in another upgrade/request cycle.

## Plan

Fix https://github.com/openclaw/openclaw/issues/78727. In `src/infra/device-pairing.ts`, make `approveDevicePairing` mint rotated role tokens from the merged role-scoped approved baseline instead of replacing token scopes with only the latest pending request scopes, while preserving the existing caller-scope containment checks and not adding a master-token bypass for `openclaw mcp serve`. Add regression coverage in `src/infra/device-pairing.test.ts` for an already-paired operator device whose token has an existing scope such as `operator.pairing` or `operator.read`, then approve an MCP-style upgrade with additional operator scopes and assert both `approvedScopes` and `tokens.operator.scopes` contain the union. Add a Gateway/MCP-facing regression if feasible, likely around `src/mcp/channel-server.test.ts` or `src/gateway/server.auth.control-ui.suite.ts`, proving the approved upgrade settles instead of creating another pending request. Add a `CHANGELOG.md` fix entry because this is user-facing. Do not weaken scope approval, do not silently auto-approve upgrades, and do not change `gateway.nodes.pairing.autoApproveCidrs` semantics.
