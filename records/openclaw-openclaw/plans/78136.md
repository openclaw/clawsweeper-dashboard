## Summary

Keep open: current main still has the readiness mismatch described here. The command queue can reject new work while `gatewayDraining` is true, but `/readyz` is still driven only by startup, channel, skip, cache, and event-loop checks; the only linked fix PR was closed unmerged.

## Plan

Fix the gateway readiness/drain mismatch reported in https://github.com/openclaw/openclaw/issues/78136. Add a public command-queue runtime accessor for the current `gatewayDraining` flag, wire it into `createReadinessChecker()` from `startGatewayServer()`, and make readiness return `ready: false` with a stable `gateway-draining` failing reason while the command queue is rejecting new work for restart drain. Keep `/healthz` as shallow liveness, avoid a new config option, and do not change Docker healthcheck semantics. Ensure the transient draining result is not cached in a way that stays red after `resetAllLanes()` clears the flag. Add regression tests for the command-queue accessor, readiness during drain, cache transition from draining to ready, and the HTTP `/readyz` status/body if feasible. Also inspect the SIGUSR1 duplicate restart path and add or preserve coverage that `resetAllLanes()` clears the drain flag after in-process restart iterations. Add a concise changelog fix entry if this repair lane owns changelog updates.
