# Coding Plan for openclaw/openclaw#70533: Plugin discovery loads all dist/extensions/ manifests at boot regardless of tools.allow (~500 MB structural heap)

Source report: [records/openclaw-openclaw/items/70533.md](records/openclaw-openclaw/items/70533.md)

## Summary

Keep open. Current main still builds the plugin metadata snapshot and installed-index manifest registry before it knows the gateway startup plugin scope, so disabled or unallowed bundled plugin manifests can still be materialized during the control-plane path.

## Plan

Fix https://github.com/openclaw/openclaw/issues/70533 by scoping plugin metadata/installed-index manifest reconstruction before manifest reads. Derive a conservative gateway startup/control-plane plugin-id scope from existing activation semantics, pass it through loadPluginLookUpTable/loadPluginMetadataSnapshot into loadPluginManifestRegistryForInstalledIndex, and preserve configured channels, providers, root plugin config, slots, setup diagnostics, plugins.allow/plugins.deny, and workspace/plugin-load-path behavior. Do not add broad persistent discovery caches or change runtime plugin activation policy. Add focused regression coverage proving unneeded installed-index/bundled manifests are not loaded when the startup scope is small, and coordinate with the broader repeated-rescan tracker https://github.com/openclaw/openclaw/issues/78461 rather than solving that whole issue here. Add a CHANGELOG entry for the user-visible startup/memory performance fix if this becomes a repair PR.
