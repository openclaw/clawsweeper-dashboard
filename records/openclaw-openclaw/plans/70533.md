# Coding Plan for openclaw/openclaw#70533: Plugin discovery loads all dist/extensions/ manifests at boot regardless of tools.allow (~500 MB structural heap)

Source report: [records/openclaw-openclaw/items/70533.md](records/openclaw-openclaw/items/70533.md)

## Summary

Keep open. Current main still builds the plugin metadata snapshot and manifest registry before deriving the narrowed startup plugin plan, and the 2026.5.12 user datapoint says discovery still scales with the bundled plugin tree despite `plugins.allow`.

## Plan

Fix https://github.com/openclaw/openclaw/issues/70533 by reducing bundled/plugin manifest discovery work before unused plugins are read and validated. Carry a conservative plugin-id scope from existing plugin activation/config semantics into `loadPluginMetadataSnapshot` and `loadPluginManifestRegistryForInstalledIndex` before manifest candidates are materialized; do not add a new config option and do not treat `tools.allow` literals as the only plugin-id source. Preserve manifest-first validation, channel/provider auto-enable behavior, slots, disabled/deny handling, bundled discovery compatibility, and external installed plugin behavior. Add focused regression tests proving an allowlisted startup/config path passes `pluginIds` to the installed-index registry loader and still includes required configured providers, channel owners, memory/context slots, and explicitly enabled external plugins. Add a user-facing changelog fix entry during landing. Validate with narrow Vitest files first, then Testbox/Crabbox `pnpm check:changed`; include live before/after RSS or manifest-open-count proof on a large bundled-plugin install if feasible.
