# Coding Plan for openclaw/openclaw#70533: Plugin discovery loads all dist/extensions/ manifests at boot regardless of tools.allow (~500 MB structural heap)

Source report: [records/openclaw-openclaw/items/70533.md](records/openclaw-openclaw/items/70533.md)

## Summary

Keep open. Current main still builds the plugin metadata snapshot and installed-index manifest registry without passing a narrowed plugin-id scope, so startup/request planning can only filter after the manifest registry has already been reconstructed.

## Plan

Fix OpenClaw plugin discovery so restrictive plugin/tool configs can reduce metadata manifest reads before validation. Carry a conservative plugin-id scope into `loadPluginMetadataSnapshot` and `loadPluginManifestRegistryForInstalledIndex` for Gateway startup/reload and relevant control-plane paths, using existing semantics for `plugins.allow`, `plugins.entries.*.enabled`, configured channels/providers/agent harnesses/memory slots/hooks, and `tools.allow` rather than a literal tools-only filter. Preserve manifest-first validation for plugins that may still be activated by config, keep disabled/denied diagnostics scoped, avoid persistent metadata caches, and do not change the public plugin API. Add focused regression tests proving unneeded bundled manifests are not read/validated when a safe scope is available and proving compatibility exceptions still load. Add a single changelog fix entry because this is user-visible performance/availability work.
