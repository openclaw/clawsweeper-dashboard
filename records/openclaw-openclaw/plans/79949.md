## Plan

Fix https://github.com/openclaw/openclaw/issues/79949. Current WhatsApp inbound handling accepts each image separately, media messages bypass the inbound debouncer, and queue.mode=steer injects a concurrent media message as text-only, so later images in a forwarded multi-image burst can be lost to the LLM. Keep the fix owned by the WhatsApp plugin unless a small generic media-carrying seam is truly needed. Implement rapid same-account/conversation/sender media coalescing or an equivalent queued-turn media preservation path that guarantees all images are delivered to a single agent turn as media. Extend WebInboundMessage/buildWhatsAppInboundContext to carry MediaPaths/MediaTypes if needed. Preserve access control, dedupe, read receipts, quoted-media download behavior, group mention gating, mediaMaxMb/saveMediaBuffer guardrails, and channel-owner boundaries. Add regression coverage that emits two image messages about 200ms apart and asserts one agent dispatch sees both media paths/images; also cover queue.mode=steer so the second media is not swallowed. Add a user-facing changelog fix entry.
