## Summary

Keep open: current main and the latest release still bypass WhatsApp inbound debouncing for media, and active-run steering accepts only text, so the reported second-image loss remains a valid WhatsApp delivery bug.

## Plan

Fix the WhatsApp issue where rapid multi-image forwarded messages can deliver only the first image to the agent when media bypasses inbound debounce and active queue steering is text-only. Keep the scope to WhatsApp media loss for this issue; do not implement the broader timestamp batching request in https://github.com/openclaw/openclaw/issues/11703 or per-group debounce config from https://github.com/openclaw/openclaw/issues/54565. Prefer a WhatsApp-owned short media-burst coalescing path keyed by account, conversation, and sender, or a lossless fallback that prevents media-bearing turns from being accepted by text-only active steering. Preserve all image media facts as ordered current-turn images/MediaPaths, keep control commands and non-media immediate-flush behavior safe, avoid adding new config unless maintainers explicitly require it, and update docs only if default media batching semantics change. Add regression coverage for two rapid image/jpeg messages from the same sender producing one agent-visible turn with both images, plus coverage that queue.mode steer no longer drops the second image.
