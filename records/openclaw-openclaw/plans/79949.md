# Coding Plan for openclaw/openclaw#79949: WhatsApp: multi-image forwarded messages only deliver the first image to the LLM

Source report: [records/openclaw-openclaw/items/79949.md](records/openclaw-openclaw/items/79949.md)

## Summary

Keep open. Current main and the latest release still have a source-visible path where WhatsApp media messages bypass inbound debounce and active `queue.mode: "steer"` injects only text into the running agent, so a rapid second image can be accepted without its image payload reaching the LLM.

## Plan

Fix https://github.com/openclaw/openclaw/issues/79949. Current main lets WhatsApp media messages bypass inbound debounce, then active `messages.queue.mode: "steer"` sends only `followupRun.prompt` into the active embedded session, so image payloads from rapid forwarded WhatsApp images can be lost. Implement a focused fix that preserves every image attachment for rapid WhatsApp media messages: either add a WhatsApp-owned short media burst coalescer before the first agent turn, or make active-run steering carry `followupRun.images`/`imageOrder` through `queueEmbeddedAgentMessageWithOutcomeAsync` and `steerActiveSessionWithOptionalDeliveryWait`. Prefer no new config. Do not implement the broader timestamp-batching request from https://github.com/openclaw/openclaw/issues/11703 or the per-group debounce config request from https://github.com/openclaw/openclaw/issues/54565. Do not revive the closed split-dispatch shape from https://github.com/openclaw/openclaw/pull/80043 unless its dedupe and production media-bypass issues are fixed. Add focused tests proving two rapid WhatsApp image messages both reach the agent as image payloads, and that text-only debounce, control commands, locations, quotes, read receipts, and collect-mode image batching keep their existing behavior. No CHANGELOG edit.
