# Coding Plan for openclaw/openclaw#58411: sessions_spawn lacks --bind here equivalent — agent cannot bind ACP session to existing Discord thread

Source report: [records/openclaw-openclaw/items/58411.md](records/openclaw-openclaw/items/58411.md)

## Summary

Keep open: current main still exposes only `thread` for ACP `sessions_spawn`, and the Discord path still maps that to child-thread placement, while `/acp spawn --bind here` has current-conversation binding semantics that agents cannot request. The related Telegram issue at https://github.com/openclaw/openclaw/issues/41004 was fixed for `thread:true` channel placement, but it does not cover this Discord current-bind API gap.

## Plan

Implement an additive ACP `sessions_spawn` current-conversation binding option for the existing Discord `/acp spawn --bind here` parity gap. Add a tool parameter such as `bindTo: "current"` for `runtime: "acp"`, thread it through `SpawnAcpParams`/`spawnAcpDirect`, and bind the new ACP session to the requester’s current conversation using the same capability checks, current-placement validation, normalized conversation reference, and existing-binding ownership rules used by `/acp spawn --bind here`. Preserve current `thread: true` behavior and tests: it should still create or bind the channel’s normal child/thread placement for Discord. Update focused tests for the new current-bind tool path and for the unchanged child-thread path, update ACP/subagent/Discord docs, and add a changelog entry because this is user-facing tool API behavior. Stop if the implementation would require a broader CLI/API redesign or a new config policy knob.
