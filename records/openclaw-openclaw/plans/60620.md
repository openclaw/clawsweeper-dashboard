## Summary

Keep open. Current main still reloads the full startup plugin scope after detecting any deferred configured channel plugin, so a non-deferred startup plugin can still be activated once during setup-runtime startup and again during the post-listen full-runtime reload.

## Plan

Fix https://github.com/openclaw/openclaw/issues/60620. Current main computes deferredConfiguredChannelPluginIds separately but still passes the full startupPluginIds list into reloadDeferredGatewayPlugins after listen, causing non-deferred startup plugins to be activated twice when any configured deferred channel plugin is present. Scope the post-listen full-runtime reload to deferredConfiguredChannelPluginIds and preserve or merge the already attached non-deferred registry state, gateway methods, services, and discovery. Add focused regression coverage for a mixed startup plan with one non-deferred plugin and one deferred configured channel plugin. Do not add plugin-specific idempotency guards or broaden plugin discovery/loading. Include a concise CHANGELOG.md fix entry if the landing workflow expects one.
