# Coding Plan for openclaw/openclaw#70347: Cron outer timeout should emit lifecycle.error so sessions.json finalizes immediately

Source report: [records/openclaw-openclaw/items/70347.md](records/openclaw-openclaw/items/70347.md)

## Summary

Keep open. Current main still aborts and cleans up timed-out isolated cron runs, but the timeout-winning branch returns a cron error without emitting the terminal lifecycle event that would drive synchronous session-store finalization; the related open PR overlaps direct cron session status persistence but does not fix this lifecycle-event path.

## Plan

Fix https://github.com/openclaw/openclaw/issues/70347 on current main. When an isolated cron agent run has already started and the outer cron timeout wins, emit a terminal lifecycle error event carrying the active `sessionId`/`sessionKey`, timeout error text, `errorKind: "timeout"`, `stopReason: "timeout"`, and timestamps so the existing gateway lifecycle persistence path finalizes the session row immediately. Keep the fix in the cron/gateway lifecycle boundary; do not add a second direct `sessions.json` writer from cron and do not revive the direct-status approach from https://github.com/openclaw/openclaw/pull/56624. Add focused regression coverage showing the timed-out started run emits/persists the terminal lifecycle state, and guard against duplicate or missing finalization if the embedded runner later settles. Add a changelog entry for the user-visible cron reliability fix.
