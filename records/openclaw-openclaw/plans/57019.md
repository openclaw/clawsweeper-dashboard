## Summary

Keep open. Current main has the fs-safe snapshot-safe release mitigation, but the OpenClaw session wrapper still catches `file_lock_stale`, re-reads the lock path, and removes it by path before retrying, so the remaining race is still source-reproducible in the session-level recovery path.

## Plan

Fix https://github.com/openclaw/openclaw/issues/57019 in the session write-lock implementation. Preserve the existing fs-safe manager adoption and existing stale/dead-PID/recycled-PID/non-OpenClaw-owner/orphan-self recovery behavior, but make the session-level `file_lock_stale` recovery snapshot-safe so it cannot delete a newer lock file by path after inspecting an older lock. Prefer wiring session stale removal through fs-safe `staleRecovery: "remove-if-unchanged"` plus an approval predicate, or add an equivalent observed raw/stat guard around `removeReportedStaleLockIfStillStale`; do not change lock timeout policy, add config, or broaden the lock lifecycle.
