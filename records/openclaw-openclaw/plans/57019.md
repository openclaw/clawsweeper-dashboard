## Summary

Keep open. Current main has the fs-safe snapshot-safe release mitigation, but the OpenClaw session wrapper still handles `file_lock_stale` by re-reading and path-removing a same-PID/no-starttime lock before retrying, so a same-session concurrent acquire can still remove a fresh lock through the higher-level recovery path.

## Plan

Fix the session write-lock stale recovery race reported at https://github.com/openclaw/openclaw/issues/57019. Current main uses @openclaw/fs-safe for snapshot-safe release, but src/agents/session-write-lock.ts still handles file_lock_stale by re-inspecting and path-removing lockPath in removeReportedStaleLockIfStillStale.
