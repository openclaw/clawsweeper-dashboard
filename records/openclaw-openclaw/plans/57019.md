## Summary

Keep open. Current main removed the original local HELD_LOCKS release implementation, but the session wrapper still has a source-backed stale-recovery path that can classify same-PID/no-starttime locks as reclaimable and then delete the lock by path before retrying.

## Plan

Fix the session write-lock stale-recovery race reported in https://github.com/openclaw/openclaw/issues/57019. Keep the change scoped to session lock stale cleanup: avoid deleting a lock file by path after fs-safe has observed a stale snapshot unless the file is still the same observed lock, preferably by using @openclaw/fs-safe's staleRecovery: "remove-if-unchanged" plus shouldRemoveStaleLock or an equivalent snapshot guard. Preserve existing dead-PID, recycled-PID, non-OpenClaw-owner, malformed/payload-less, and intentional orphan-self recovery behavior. Add focused regression tests in src/agents/session-write-lock.test.ts for stale retry/removal not unlinking a changed or freshly acquired lock, including the same-PID/no-starttime class where feasible. Add a CHANGELOG.md Fixes entry because this protects user-visible transcript durability. Do not change session timeout defaults, add config, or redesign the lock lifecycle.
