# Coding Plan for openclaw/openclaw#20934: [Feature]: REST endpoint for session management in HTTP API integrations

Source report: [records/openclaw-openclaw/items/20934.md](records/openclaw-openclaw/items/20934.md)

## Summary

Keep open. Current main still lacks the requested REST-native session-management endpoints for HTTP integrations; the existing Gateway session RPC/schema support and adjacent HTTP helpers do not expose reset/status/compact/model/reasoning under `/v1/sessions/*`, and the prior REST PR is closed unmerged.

## Plan

Implement https://github.com/openclaw/openclaw/issues/20934 as a focused authenticated Gateway HTTP session-management surface. Add `/v1/sessions/reset`, `/v1/sessions/status`, `/v1/sessions/compact`, `/v1/sessions/model`, and `/v1/sessions/reasoning` or a tightly equivalent documented shape that maps onto existing `sessions.reset`, `sessions.list`/describe/status semantics, `sessions.compact`, and `sessions.patch` for `model` and `reasoningLevel`. Reuse existing Gateway auth helpers and method-scope checks; do not re-enable slash-command interception or set `CommandAuthorized` in `/v1/chat/completions`. Avoid the closed PR pitfalls from https://github.com/openclaw/openclaw/pull/23941: do not pass unsupported `key` fields into strict `sessions.list`, do not use broad `/v1/sessions` prefix matching that catches `/v1/sessionsXYZ`, and validate reset `reason` against the existing allowed values. Add focused route/auth/request/response tests, update public Gateway/OpenAI-compatible HTTP docs, and add a user-facing changelog entry. Coordinate naming with the read-only tracker at https://github.com/openclaw/openclaw/issues/47826 without collapsing this mutation-focused issue into that one.
