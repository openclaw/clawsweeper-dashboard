# Coding Plan for openclaw/openclaw#70903: Persistent file-based provider cooldown blocks user for hours after billing recovery

Source report: [records/openclaw-openclaw/items/70903.md](records/openclaw-openclaw/items/70903.md)

## Summary

Keep open. Current main still has the core recovery gap: billing-disabled profiles remain persisted for long windows, same-auth fallback chains can avoid far-from-expiry probes, and configure/plugin auth refresh paths can upsert fresh credentials without clearing stale usage state.

## Plan

Fix https://github.com/openclaw/openclaw/issues/70903 as a narrow runtime recovery bug. Keep persisted auth-profile cooldown/disabled protection, but add targeted recovery so billing-disabled chains whose configured fallbacks share the same unusable auth path can probe on the existing throttle instead of waiting hours, while preserving normal fallback preference when there is a real alternate provider/profile path. Also clear matching provider/profile usage state after successful configure/plugin auth refresh paths that call runProviderPluginAuthMethod, using the existing clearAuthProfileCooldown/listProfilesForProvider pattern from models auth login. Do not add a new config option, do not remove persisted auth-state generally, do not broadly re-enable retries for auth_permanent or rate_limit, and coordinate with but do not depend on https://github.com/openclaw/openclaw/pull/74909. Add focused regression tests for same-auth fallback billing recovery and configure/plugin auth stale-state cleanup. Include a changelog entry if the repair changes user-visible recovery behavior.
