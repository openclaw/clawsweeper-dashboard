# Coding Plan for openclaw/openclaw#50621: Cron systemEvent job times out after ~960s even though agent runs in main session

Source report: [records/openclaw-openclaw/items/50621.md](records/openclaw-openclaw/items/50621.md)

## Summary

Keep open: current main and the latest release still apply the generic cron job timeout to main-session `systemEvent` jobs while `wakeMode: "now"` waits for `runHeartbeatOnce`, so the false `cron: job execution timed out` state remains source-reproducible. Related PRs that tried to adjust this exact timeout were closed unmerged, and current open cron PRs I checked do not own this focused fix.

## Plan

Fix https://github.com/openclaw/openclaw/issues/50621. Current main still runs main-session `payload.kind="systemEvent"` / `wakeMode="now"` jobs under the generic cron timeout while awaiting `runHeartbeatOnce`, so long main-session work can be delivered or completed but the cron run records `cron: job execution timed out`. Make the main-session `systemEvent` wake-now status boundary explicit: record success after event delivery plus verified wake scheduling/start, or otherwise avoid racing the long main-session agent turn against `DEFAULT_JOB_TIMEOUT_MS`. Preserve the https://github.com/openclaw/openclaw/issues/652 guarantee that wake-now jobs actually wake or schedule the main session; do not weaken isolated `agentTurn` timeout cleanup, provider timeout handling, or existing `timeoutSeconds` semantics. Add focused regression coverage for the reported path and update docs/changelog only if the public cron behavior changes.
