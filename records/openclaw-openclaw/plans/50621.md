# Coding Plan for openclaw/openclaw#50621: Cron systemEvent job times out after ~960s even though agent runs in main session

Source report: [records/openclaw-openclaw/items/50621.md](records/openclaw-openclaw/items/50621.md)

## Summary

Keep open: current main and the latest shipped tag still route main-session `systemEvent` wake-now jobs through the generic cron timeout while awaiting the main heartbeat, so the false `cron: job execution timed out` status remains source-reproducible.

## Plan

Fix the cron main-session `systemEvent` timeout/status bug described by https://github.com/openclaw/openclaw/issues/50621. Current main gives non-`agentTurn` cron jobs the generic `DEFAULT_JOB_TIMEOUT_MS`, starts that timeout immediately for main-session jobs, then awaits `runHeartbeatOnce` for `wakeMode: "now"`; long main-session work can be delivered successfully but still be recorded as `cron: job execution timed out`. Keep the fix narrow in `src/cron/service/timeout-policy.ts`, `src/cron/service/timer.ts`, and focused cron tests. Preserve the wake-now guarantee from https://github.com/openclaw/openclaw/issues/652 that cron verifies the heartbeat is started/scheduled before recording success, and do not regress isolated `agentTurn` timeout cleanup or add a new config option unless maintainers explicitly request it. Add regression coverage for a main `systemEvent` wake-now job whose heartbeat promise outlives the generic cron timeout and ensure the run is not marked failed solely for that timeout after delivery/wake has been established. Update docs or changelog only if the user-visible cron contract changes.
