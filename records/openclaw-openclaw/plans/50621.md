# Coding Plan for openclaw/openclaw#50621: Cron systemEvent job times out after ~960s even though agent runs in main session

Source report: [records/openclaw-openclaw/items/50621.md](records/openclaw-openclaw/items/50621.md)

## Summary

Keep open: current main and the latest release still apply the generic cron timeout to main-session `systemEvent` jobs while `wakeMode: "now"` awaits `runHeartbeatOnce`, so the false timeout status remains source-reproducible. The duplicate reports and closed fix attempts preserve useful context, but no open focused PR currently owns this exact cron timeout boundary.

## Plan

Fix https://github.com/openclaw/openclaw/issues/50621. Current main enqueues main-session `systemEvent` cron events, then `wakeMode: "now"` awaits `runHeartbeatOnce` inside `executeJobCoreWithTimeout`; `resolveCronJobTimeoutMs` still gives non-`agentTurn` jobs the 10-minute default and `normalizeCronJobPayload` deletes `timeoutSeconds` from `systemEvent`, so long main-session work can be recorded as `cron: job execution timed out` after the event was delivered. Implement a bounded fix in the cron runtime/status boundary: preserve the wake-start guarantee from https://github.com/openclaw/openclaw/issues/652, keep isolated `agentTurn` timeout cleanup unchanged, and avoid adding a broad liveness-timeout feature or public config knob unless the existing contract cannot be preserved. Add regression coverage for a pending/long `runHeartbeatOnce` path and for the preserved non-main/isolated timeout behavior. Add a changelog entry because this is user-visible cron behavior.
