# Coding Plan for openclaw/openclaw#50621: Cron systemEvent job times out after ~960s even though agent runs in main session

Source report: [records/openclaw-openclaw/items/50621.md](records/openclaw-openclaw/items/50621.md)

## Summary

Keep open: current main and the latest release still let a wake-now main-session `systemEvent` cron job sit inside the generic cron timeout while awaiting `runHeartbeatOnce`, so the false `cron: job execution timed out` status remains source-reproducible. The duplicate report and prior closed PRs all point to the same focused cron timeout boundary, and no open focused fix PR currently owns it.

## Plan

Fix https://github.com/openclaw/openclaw/issues/50621 by preventing main-session `systemEvent` cron jobs using `wakeMode: "now"` from being recorded as `cron: job execution timed out` while the delivered main-session agent turn continues. Preserve the https://github.com/openclaw/openclaw/issues/652 guarantee that wake-now actually wakes or starts/schedules the main session; do not change isolated `agentTurn` timeout cleanup. Prefer an explicit status boundary after event delivery plus verified wake scheduling/start over adding a new public config knob. Use the closed unmerged PRs https://github.com/openclaw/openclaw/pull/50670, https://github.com/openclaw/openclaw/pull/50698, and https://github.com/openclaw/openclaw/pull/50723 only as root-cause background, not as a required patch shape. Add regression coverage with a deferred `runHeartbeatOnce`/fake-timer path proving the cron run is not marked error when the main heartbeat remains pending past `DEFAULT_JOB_TIMEOUT_MS`, and keep existing busy-skip and isolated timeout tests green. Add a `CHANGELOG.md` fix entry if the repair changes user-visible behavior.
