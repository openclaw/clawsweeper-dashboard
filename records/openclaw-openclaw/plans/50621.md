# Coding Plan for openclaw/openclaw#50621: Cron systemEvent job times out after ~960s even though agent runs in main session

Source report: [records/openclaw-openclaw/items/50621.md](records/openclaw-openclaw/items/50621.md)

## Summary

Keep open: current main and the latest shipped release still route main-session `systemEvent` wake-now jobs through the generic cron timeout while awaiting the main heartbeat, so the false `cron: job execution timed out` status remains source-reproducible.

## Plan

Fix https://github.com/openclaw/openclaw/issues/50621. Current main-session cron `systemEvent` jobs with `wakeMode: "now"` enqueue the event, await `runHeartbeatOnce`, and remain under the generic 10-minute cron timeout, so long active main-session work can complete while cron records `cron: job execution timed out`. Implement a narrow fix that preserves the wake-now heartbeat-start guarantee from https://github.com/openclaw/openclaw/issues/652 while making the main-session `systemEvent` status boundary explicit: success should be based on delivery and verified wake scheduling/start, not waiting for the entire long main heartbeat. Do not add a new config option as the primary fix, and do not weaken isolated `agentTurn` timeout cleanup or watchdog behavior. Add focused regression coverage for main-session wake-now `systemEvent` exceeding the generic timeout and update docs/changelog only if the user-visible behavior text changes.
