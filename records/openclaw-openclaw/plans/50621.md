## Summary

Keep open: current main and the latest release still apply the generic cron watchdog to main-session `systemEvent` jobs while `wakeMode: "now"` waits on heartbeat work, so the false `cron: job execution timed out` state remains source-reproducible. The duplicate reports and closed unmerged fix PRs add useful confirmation, but no merged fix or open canonical PR currently resolves this issue.

## Plan

Fix https://github.com/openclaw/openclaw/issues/50621. Current main gives `sessionTarget: "main"` + `payload.kind: "systemEvent"` + `wakeMode: "now"` the generic 10-minute cron timeout even though that path awaits `runHeartbeatOnce` and can block on real main-session agent work. Implement the narrow timeout-policy repair so this blocking path uses `AGENT_TURN_SAFETY_TIMEOUT_MS`, while preserving the generic `DEFAULT_JOB_TIMEOUT_MS` for non-blocking `systemEvent` jobs such as `wakeMode: "next-heartbeat"` and preserving existing isolated `agentTurn` explicit timeout/cleanup behavior. Add focused regression coverage for the new timeout-policy branch and its inverse, keep normalization/type behavior unchanged unless required by the narrow fix, update `CHANGELOG.md` if user-visible, and stop if the solution would require a new user config option or broader cron product decision.
