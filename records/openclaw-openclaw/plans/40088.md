# Coding Plan for openclaw/openclaw#40088: memory_search: chokidar file watcher silently stops delivering events, index goes stale

Source report: [records/openclaw-openclaw/items/40088.md](records/openclaw-openclaw/items/40088.md)

## Summary

Keep open. Current main has part of the watcher resilience work, but the default sync interval is still disabled and interval ticks still do not mark memory dirty, so a watcher that stops after a successful sync can still leave the index stale. The exact fix PRs were closed unmerged, and the open recursive-memory PR overlaps stale search timing but does not cover the watcher-death fallback.

## Plan

Fix the watcher-death stale-index fallback for this issue without taking over the recursive-memory/QMD/docs work in https://github.com/openclaw/openclaw/pull/72748. Implement a focused memory-core change: keep explicit `agents.defaults.memorySearch.sync.intervalMinutes: 0` as an opt-out, set a modest non-zero default such as 15 minutes, and make interval sync set the memory dirty flag or otherwise trigger the existing lightweight `syncMemoryFiles` hash comparison before calling `sync({ reason: "interval" })` so modified files are discovered even when chokidar stopped emitting events. Preserve the existing watcher error listener and avoid full forced reindexing on every interval. Add regression coverage for default interval resolution, explicit `0` override, interval tick causing a memory-file scan when watcher events are absent, and the non-throwing watcher error listener. Add a single changelog Fixes entry crediting the human reporter where appropriate. Do not change QMD recursive watcher roots, temporal-decay path parsing, or published recursive-memory docs unless maintainers decide to combine with the open recursive-memory PR.
