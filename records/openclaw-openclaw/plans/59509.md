# Coding Plan for openclaw/openclaw#59509: feat: support per-model location override for google-vertex provider

Source report: [records/openclaw-openclaw/items/59509.md](records/openclaw-openclaw/items/59509.md)

## Summary

Keep open. Current main and v2026.5.7 still expose only transport-level Vertex `options.location` plus `GOOGLE_CLOUD_LOCATION`; per-model params are accepted, but the Google plugin has no hook that maps `params.location` into Vertex transport options, so mixed global/regional Vertex models under one project remains a valid feature gap.

## Plan

Implement explicit per-model Google Vertex location override for https://github.com/openclaw/openclaw/issues/59509. Use the existing `agents.defaults.models["google-vertex/<model>"].params.location` config path; do not add a top-level `location` model schema key and do not add automatic preview-model defaults unless maintainers ask. Add a Google-plugin-owned `extraParamsForTransport` mapping in the provider hook path that forwards a non-empty string `params.location` only for Google Vertex transport/model requests, preserving existing `options.location` and `GOOGLE_CLOUD_LOCATION` fallback behavior. Add focused regression coverage showing two `google-vertex` model entries can produce different transport `location` options, keep direct Gemini `cachedContent` behavior intact, update Google provider docs, and add a user-facing changelog entry. If the broader Google Vertex routing PR lands first, rebase/adapt to its final integration point instead of duplicating provider setup.
