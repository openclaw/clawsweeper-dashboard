## Summary

Keep open. Current main still has a canonical provider normalizer, but the Usage/Control UI paths continue to use raw provider strings for aggregates, filters, suggestions, query matching, exports, and visible badges, so documented aliases such as `z.ai`/`z-ai` and `aws-bedrock` can still display or group separately from their canonical providers.

## Plan

Fix the Control UI Usage provider canonicalization bug described in https://github.com/openclaw/openclaw/issues/47840. Reuse the existing `normalizeProviderId` contract so documented aliases such as `z.ai`/`z-ai` display and group as `zai`, and `bedrock`/`aws-bedrock` display and group as `amazon-bedrock`. Scope the fix to derived Usage API/UI boundaries: aggregate grouping keys, display labels, provider filters, query suggestions/query matching, session badges, and CSV/export-style output. Preserve raw stored session/config fields and do not add a new `bailian` alias or docs contract without maintainer approval. Add focused regression coverage for alias grouping/display/filter/query behavior and a user-facing changelog entry if the fix changes visible UI behavior.
