## Summary

Keep open. Current main still has a source-reproducible CLI fallback retry gap: later fallback candidates can be built from a captured session entry that still contains the reusable CLI binding, and resume-mode CLI execution still submits prompt text to that backend session.

## Plan

Fix https://github.com/openclaw/openclaw/issues/54192. Current main can reuse a stale CLI session binding across fallback candidates: agent-command.ts passes the captured sessionEntryForAttempt into each runAgentAttempt, attempt-execution.ts clears params.sessionEntry/sessionStore after failed reused Claude CLI turns but the outer fallback callback does not refresh its entry, and executePreparedCliRun/buildCliArgs still submit prompt text in resume mode. Add a CLI-owned fallback retry repair that avoids reusing a backend session after the first attempt may have appended the prompt, while preserving the original task through a fresh/reseeded CLI session or a backend-confirmed transcript repair. Do not implement a blind prompt-skip that can drop the task when the first attempt failed before persistence. Add focused regression coverage for timeout/rate-limit fallback retry with an existing cliSessionId and assert the later CLI candidate does not resubmit to the stale backend session. Keep the change in the agent/CLI runner boundary; do not change plugin API, CLI backend manifest schema, or unrelated embedded-runner behavior. Add a changelog entry if the final fix is user-visible.
