# Coding Plan for openclaw/openclaw#54192: CLI runner: add orphaned-user-message repair for fallback retries

Source report: [records/openclaw-openclaw/items/54192.md](records/openclaw-openclaw/items/54192.md)

## Summary

Keep open. Current main still lacks a CLI-owned orphaned-user-message repair for fallback retries: the fallback loop can re-enter the CLI path with a reusable backend session, and resume-mode CLI execution still sends the retry prompt into that backend session.

## Plan

Fix https://github.com/openclaw/openclaw/issues/54192 in one focused PR. The bug is that fallback retries can re-enter the CLI runner with an existing reusable CLI backend session and still submit the retry prompt, which can duplicate the user turn in the backend session. Add a CLI-owned repair or safe fresh-session reset/reseed path for fallback retries that prevents duplicate backend user turns without dropping the task. Preserve the existing prompt-preservation behavior from PR 55632/PR 66029, preserve fresh-session reseed behavior, do not add a config option, and avoid a naive prompt skip unless the backend transcript is proven to already contain the prompt. Likely touch src/agents/command/attempt-execution.ts, src/agents/cli-runner.ts, src/agents/cli-runner/execute.ts, and focused tests under src/agents/command/attempt-execution.cli.test.ts and/or src/agents/cli-runner.reliability.test.ts. Add regression coverage for timeout/rate-limit fallback retry with a reused CLI session id and assert the duplicate prompt is not submitted while the original task is still available to the successful retry.
