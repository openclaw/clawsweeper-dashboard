# Coding Plan for openclaw/openclaw#48045: [Bug]: Browser tool silently discards downloads and throws \"Download is starting\" error when using CDP connection

Source report: [records/openclaw-openclaw/items/48045.md](records/openclaw-openclaw/items/48045.md)

## Summary

Keep open: current main and the latest release still send direct browser `/navigate` attachment URLs through URL-only Playwright navigation, so Playwright's `Download is starting` abort remains a failure path. The shipped managed-download work is useful adjacent coverage for explicit and click-triggered downloads, but it does not add `/navigate` download metadata or a guarded success path for attachment responses.

## Plan

Fix https://github.com/openclaw/openclaw/issues/48045 in the browser plugin. Make Playwright-backed `/navigate` handle allowed `Content-Disposition: attachment` or equivalent download-triggering navigations over CDP by observing a real download around guarded navigation, treating Playwright's `Download is starting` abort as success only when that download is captured, persisting via the existing managed downloads directory/safe-write path, and returning additive `download` metadata from `/navigate`. Preserve SSRF/redirect-chain checks and blocked-navigation cleanup; do not use nonexistent Playwright APIs such as `BrowserContext.setDefaultDownloadPath`, do not skip policy checks, and do not double-save downloads owned by explicit `/wait/download` or `/download` waiters. Add focused regression coverage for attachment navigation success, non-download navigation errors still throwing, policy-denied redirects remaining fail-closed, and explicit waiter ownership. Add a changelog entry because this is user-visible.
