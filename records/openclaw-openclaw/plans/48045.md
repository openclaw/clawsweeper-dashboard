## Summary

Keep open: current main and the latest release still leave direct browser `/navigate` attachment URLs on a URL-only Playwright path that rethrows Playwright's `Download is starting` abort. The shipped managed-download work is useful adjacent context, but it covers click-triggered/explicit download handling rather than this direct navigation-download response path.

## Plan

Fix the direct browser `/navigate` attachment-download path for https://github.com/openclaw/openclaw/issues/48045. In `extensions/browser/src/browser/pw-tools-core.snapshot.ts`, `extensions/browser/src/browser/pw-session.ts`, and the `/navigate` route, make navigation to a `Content-Disposition: attachment` response succeed only when a real download is observed and saved under the existing guarded downloads directory, then return additive structured download metadata. Preserve SSRF/navigation-policy behavior from `gotoPageWithNavigationGuard` and `assertPageNavigationCompletedSafely`; do not swallow policy denials or skip redirect-chain checks. Do not revive the closed approaches from https://github.com/openclaw/openclaw/pull/48132 or https://github.com/openclaw/openclaw/pull/28978 that used nonexistent APIs, cycles, or unsafe success handling. Add focused regression coverage for direct attachment navigation, keep explicit `/wait/download` and `/download` waiter behavior intact, and add a user-visible changelog Fixes entry if the PR changes released behavior.
