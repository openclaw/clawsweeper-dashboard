# Coding Plan for openclaw/openclaw#48045: [Bug]: Browser tool silently discards downloads and throws \"Download is starting\" error when using CDP connection

Source report: [records/openclaw-openclaw/items/48045.md](records/openclaw-openclaw/items/48045.md)

## Summary

Keep open. Current main includes adjacent managed/click download capture, but the direct Playwright `/navigate` path still returns only URL metadata and rethrows non-policy `page.goto()` errors, so an attachment navigation can still fail instead of reporting a captured download.

## Plan

Fix https://github.com/openclaw/openclaw/issues/48045 on current main. Implement a localized browser-plugin repair for direct `/navigate` to attachment/download URLs on the Playwright/CDP path: observe or arm the next download around `gotoPageWithNavigationGuard`, treat Playwright's download-starting navigation abort as success only when a real download is captured, persist through the existing guarded OpenClaw downloads directory, and return structured download metadata from `/navigate`. Preserve SSRF/navigation-policy checks, including redirect/final-result validation; do not repeat the #28978 approach that skipped final validation. Do not use the old #48132 `BrowserContext.setDefaultDownloadPath()` or reverse-import private download helpers. Add focused regression coverage for attachment navigation and update existing browser route/helper tests as needed. Add a changelog entry if the repair is user-visible.
