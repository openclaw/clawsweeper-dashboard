# Coding Plan for openclaw/openclaw#48045: [Bug]: Browser tool silently discards downloads and throws \"Download is starting\" error when using CDP connection

Source report: [records/openclaw-openclaw/items/48045.md](records/openclaw-openclaw/items/48045.md)

## Summary

Keep open: current main and v2026.5.19 still leave direct browser `/navigate` attachment URLs on a URL-only Playwright path that rethrows Playwright's download-starting navigation abort. The shipped managed-download work is useful adjacent context, but it covers explicit/click-triggered downloads rather than this direct navigation-download response path.

## Plan

Fix https://github.com/openclaw/openclaw/issues/48045 in the browser plugin only. Implement direct `/navigate` download handling for CDP-backed Playwright pages: arm or observe a real download around `gotoPageWithNavigationGuard`, treat Playwright's `Download is starting` abort as success only when a real download is captured, save the artifact under the existing managed downloads directory, and return additive `download` metadata from `/navigate` while keeping normal navigation responses compatible. Preserve SSRF/navigation policy enforcement, including redirect-chain and blocked-target handling; do not revive the rejected approaches from https://github.com/openclaw/openclaw/pull/28978 or https://github.com/openclaw/openclaw/pull/48132 such as skipping final validation, reverse-importing download helpers, `BrowserContext.setDefaultDownloadPath`, or default-saving while an explicit waiter owns the download. Add focused regression coverage for Content-Disposition attachment navigation and blocked navigation behavior. Add a changelog entry if this becomes a user-facing fix.
