# Coding Plan for openclaw/openclaw#48045: [Bug]: Browser tool silently discards downloads and throws \"Download is starting\" error when using CDP connection

Source report: [records/openclaw-openclaw/items/48045.md](records/openclaw-openclaw/items/48045.md)

## Summary

Keep open: current main and v2026.5.20 still leave direct browser `/navigate` attachment URLs on a URL-only Playwright path that rethrows Playwright's download-starting abort. The managed-download rewrite discussed in https://github.com/openclaw/openclaw/pull/64558 is useful adjacent context, but it covers click-triggered and explicit-waiter downloads rather than this direct navigation response path.

## Plan

Fix the browser plugin direct navigation download path for https://github.com/openclaw/openclaw/issues/48045. When `/navigate` targets a response that starts a browser download, arm or observe the real Playwright/CDP download around `gotoPageWithNavigationGuard`, treat Playwright's `Download is starting` abort as success only when a real download was captured, save through the existing guarded downloads directory, and return additive structured download metadata from `/navigate` while preserving the current URL response. Preserve SSRF redirect and blocked-target fail-closed behavior; do not reintroduce the old https://github.com/openclaw/openclaw/pull/48132 approach of `BrowserContext.setDefaultDownloadPath`, reverse-importing download helpers, or double-saving while explicit waiters are active. Reuse existing download saver/output-root semantics where possible, add focused regression coverage for Content-Disposition attachment navigation plus policy-denied navigation, and add a user-facing changelog fix entry.
