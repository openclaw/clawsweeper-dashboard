# Coding Plan for openclaw/openclaw#48045: [Bug]: Browser tool silently discards downloads and throws \"Download is starting\" error when using CDP connection

Source report: [records/openclaw-openclaw/items/48045.md](records/openclaw-openclaw/items/48045.md)

## Summary

Keep open: current main and the latest release still route browser `/navigate` attachment URLs through URL-only Playwright navigation, so the Playwright `Download is starting` abort remains a failure path. The landed managed-download work is useful adjacent coverage for explicit and click-triggered downloads, but it does not add guarded `/navigate` download metadata or a success path for attachment responses.

## Plan

Fix the browser-plugin CDP `/navigate` download bug reported in https://github.com/openclaw/openclaw/issues/48045. In `extensions/browser/src/browser/pw-tools-core.snapshot.ts` and related session/download helpers, arm or observe a real Playwright download around guarded navigation, treat Playwright's `Download is starting` abort as success only when a download was captured, persist through the existing guarded managed downloads directory, and return additive download metadata from the `/navigate` route. Preserve SSRF/navigation policy checks, redirect-chain denial behavior, retry handling for detached/closed targets, and explicit `wait/download` or `download` waiter ownership; do not reintroduce `BrowserContext.setDefaultDownloadPath`, reverse imports from download helpers into session code, broad CDP fallback stacks, or unconditional error swallowing. Add focused regression coverage for attachment navigation and policy-denied navigation. Add a concise changelog fix entry only if this repair lane is maintainer-owned for landing.
