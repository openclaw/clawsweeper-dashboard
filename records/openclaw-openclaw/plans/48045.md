# Coding Plan for openclaw/openclaw#48045: [Bug]: Browser tool silently discards downloads and throws \"Download is starting\" error when using CDP connection

Source report: [records/openclaw-openclaw/items/48045.md](records/openclaw-openclaw/items/48045.md)

## Summary

Keep open: current main and the latest release still route direct browser `/navigate` attachment URLs through URL-only Playwright `page.goto()` handling that rethrows Playwright's download abort. The adjacent managed-download work is useful, but it covers explicit and click-triggered downloads rather than this direct navigation response path.

## Plan

Fix https://github.com/openclaw/openclaw/issues/48045 in the browser plugin only. Current `extensions/browser/src/browser/pw-tools-core.snapshot.ts` routes `/navigate` through guarded `page.goto()` and rethrows Playwright's `Download is starting` abort; current `extensions/browser/src/browser/routes/agent.snapshot.ts` can only return URL metadata. Implement direct attachment-navigation download handling by arming/observing a real download around guarded navigation, treating the download-starting abort as success only when a download is captured, saving via the existing managed downloads directory/safe output helpers, and returning additive `download` metadata from `/navigate`. Preserve existing SSRF/redirect policy checks and do not revive unsafe closed PR shapes from https://github.com/openclaw/openclaw/pull/28978 or https://github.com/openclaw/openclaw/pull/48132. Add focused regression coverage for a `Content-Disposition: attachment` navigation and for blocked/denied navigation still failing closed. Changelog is expected because this is user-visible browser behavior.
