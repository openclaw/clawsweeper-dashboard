## Summary

Keep open: current main still leaves direct browser `/navigate` attachment downloads on a URL-only path, while the shipped managed/click download repair does not handle the `page.goto()` download-starting failure path or return download metadata.

## Plan

Fix https://github.com/openclaw/openclaw/issues/48045 on current main. Implement direct browser `/navigate` handling for URLs that trigger a file download through Playwright/CDP: persist the actual download under the existing guarded browser downloads directory, return additive structured download metadata from `/navigate` such as `download.triggered` plus the saved path/filename when available, and do not turn unrelated navigation errors into success. Preserve current SSRF/navigation-policy behavior, including redirect/final-result checks and blocked-target cleanup. Reuse the existing browser plugin download helpers and output-root guards where possible; do not add a config option, do not revive the old `BrowserContext.setDefaultDownloadPath()` approach from https://github.com/openclaw/openclaw/pull/48132, and do not double-save downloads while explicit waiters are armed. Add focused regression coverage for attachment navigation, normal navigation, policy-denied navigation, and explicit waiter coexistence. Add a changelog entry because this is user-visible.
