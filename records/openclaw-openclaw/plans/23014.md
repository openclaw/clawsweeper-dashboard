# Coding Plan for openclaw/openclaw#23014: [Feature]: Add .m2a audio format support for Telegram and media handling

Source report: [records/openclaw-openclaw/items/23014.md](records/openclaw-openclaw/items/23014.md)

## Summary

Keep open: current main still has no `.m2a` support hits, the shared audio classifier and MIME map omit it, Telegram filename-based documents still start as `<media:document>`, and secondary WebChat/UI/memory audio maps also omit the format. The only related implementation at https://github.com/openclaw/openclaw/pull/23260 is closed unmerged and only covered part of the needed surface, while the existing maintainer review already scoped this as follow-up work.

## Plan

Implement `.m2a` audio-format support for the existing media handling paths. Treat https://github.com/openclaw/openclaw/issues/23014 as the canonical issue and https://github.com/openclaw/openclaw/pull/23260 as partial background only. Add a verified `.m2a` extension-to-MIME mapping to shared media helpers, include `.m2a` in the appropriate filename audio/voice checks, make Telegram filename-based audio documents resolve to audio placeholders/preflight when saved content is audio or the filename proves audio, and update the duplicated WebChat/UI/media-note/memory multimodal audio extension maps. Add focused regression coverage for generic or absent Telegram MIME metadata with an `.m2a` document filename. Do not add a config option, do not broaden media policy beyond `.m2a`, and verify the `.m2a` MIME mapping rather than copying `.m4a` behavior blindly. Add a user-facing changelog entry because this is new user-visible format support.
