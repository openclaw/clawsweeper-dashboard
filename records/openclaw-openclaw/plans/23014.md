# Coding Plan for openclaw/openclaw#23014: [Feature]: Add .m2a audio format support for Telegram and media handling

Source report: [records/openclaw-openclaw/items/23014.md](records/openclaw-openclaw/items/23014.md)

## Summary

Keep open: current main still has no `.m2a` support hits and still omits the extension from the shared audio classifier, voice-compatible extension set, Telegram document placeholder path, WebChat/UI audio maps, auto-reply/media-understanding checks, and memory multimodal audio list. The linked implementation PR was closed unmerged and only touched part of the required surface.

## Plan

Fix https://github.com/openclaw/openclaw/issues/23014 by adding `.m2a` as a recognized audio format across OpenClaw media handling. First verify the correct `.m2a` MIME mapping from an upstream/standard source or the repo MIME dependency; if it maps to `audio/mpeg`, add the mapping deliberately in `src/media/mime.ts` and cover `mimeTypeFromFilePath()`, `detectMime()`, `extensionForMime()` if needed, and `isAudioFileName()`. Preserve the existing `originalFilename` detection in `saveMediaBuffer()`/`saveMediaStream()` and add a regression for generic content type plus `.m2a` original filename. Reclassify Telegram `document` messages whose filename/MIME resolves to audio so placeholders/preflight use audio behavior, using plugin-safe `openclaw/plugin-sdk/media-mime` helpers or a documented SDK export rather than importing core internals from `extensions/telegram`. Update the WebChat/UI audio maps, auto-reply/media-understanding audio checks, memory multimodal audio list, focused tests, and `CHANGELOG.md`. Do not add config, broad media refactors, unrelated format churn, or bundled-plugin core boundary violations.
