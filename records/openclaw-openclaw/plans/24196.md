Source report: [records/openclaw-openclaw/items/24196.md](records/openclaw-openclaw/items/24196.md)

## Summary

Keep open. Current main still has scalar Discord upload paths and one-file send loops; the newer component/media-gallery surface is not the requested native multi-file attachment batching for ordinary Discord messages.

## Plan

Implement native multi-file sending for Discord outbound messages. Keep existing scalar `media`/`path`/`filePath`/`mediaUrl` plus `filename` behavior working, accept ordered multi-attachment input through the existing reply/payload/mediaUrls flow or a minimal additive plugin-facing seam, load every file through the existing outbound media helpers and access checks, send up to Discord's verified per-message attachment limit in one `files` array, and split overflow into follow-up messages. Preserve `mediaMaxMb`, `mediaAccess`/`mediaLocalRoots`/`mediaReadFile`, reply-to consumption, silent sends, text chunking, forum/thread behavior, component/file attachment constraints, and voice-message single-audio behavior. Update focused Discord/shared outbound tests, docs/channels/discord.md, and CHANGELOG.md. Do not broaden non-Discord channel fan-out behavior unless an additive contract is required and tested.
