# Coding Plan for openclaw/openclaw#24196: [Feature]: Allow max allowed file attachments on discord messages

Source report: [records/openclaw-openclaw/items/24196.md](records/openclaw-openclaw/items/24196.md)

## Summary

Keep open. Current main still does not implement native multi-file Discord message delivery: upstream reply/send payloads can carry multiple media references, but the Discord action/runtime/upload path remains scalar and the shared fallback sends media one file at a time.

## Plan

Implement native multi-file Discord message sending for https://github.com/openclaw/openclaw/issues/24196. Keep the fix owned by the bundled Discord plugin and avoid a broad shared outbound rewrite unless a minimal additive seam is unavoidable. Preserve existing scalar `media`/`path`/`filePath`/`filename` behavior, accept ordered multi-attachment input from `mediaUrls`/structured attachments, load each file through existing outbound media helpers and access/size checks, send up to the verified Discord per-message attachment cap in one `files` array, and split overflow into follow-up messages. Preserve reply-to consumption, text chunking, components/embeds, silent/suppress-embed behavior, mediaMaxMb, no-data-URL voice handling, and existing component/file attachment semantics. Update focused Discord/shared tests, `docs/channels/discord.md`, and `CHANGELOG.md`; do not add a new config option unless maintainer policy explicitly asks for one.
