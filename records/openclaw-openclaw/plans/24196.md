# Coding Plan for openclaw/openclaw#24196: [Feature]: Allow max allowed file attachments on discord messages

Source report: [records/openclaw-openclaw/items/24196.md](records/openclaw-openclaw/items/24196.md)

## Summary

Keep open. Current main can carry multiple outbound media references, but Discord delivery still uses scalar `mediaUrl` paths or sequential one-file sends, so the requested native multi-file Discord message behavior remains unimplemented.

## Plan

Implement native multi-file Discord outbound sending for this issue. Keep the fix Discord-owned: preserve existing scalar `media`/`path`/`filePath`/`mediaUrl`/`filename` behavior, accept ordered multi-attachment input already carried as `mediaUrls` or structured send attachments, load each file through the existing outbound media helpers/access checks, send up to the verified Discord per-message attachment cap in one `files` array, and split overflow into follow-up messages. Do not add a new `maxAttachments` config option unless maintainers explicitly request it, do not move policy into core, and do not bypass media size/access restrictions, reply-to handling, text chunking, component/file attachment semantics, or voice-message restrictions. Update focused Discord/shared outbound tests, `docs/channels/discord.md`, and `CHANGELOG.md`. Treat https://github.com/openclaw/openclaw/issues/1155 as older duplicate context and https://github.com/openclaw/openclaw/issues/53641 as related buffer-attachment context, but do not broaden this PR into the direct buffer false-success fix unless it is naturally required by the chosen implementation.
