Keep open. Current main still routes ordinary Discord uploads through scalar `mediaUrl` paths or sequential one-file sends, so native multi-file Discord message delivery remains unimplemented and no related PR appears to supersede this issue.

## Plan

Implement native multi-file sending for https://github.com/openclaw/openclaw/issues/24196 in the bundled Discord plugin. Accept ordered multi-attachment input while preserving existing single `media`/`path`/`filePath`/`filename` behavior; load each file through existing outbound media helpers and access checks; send up to the verified Discord per-message attachment cap in one `files` array; split overflow into follow-up messages; preserve reply-to, text chunking, silent/suppressEmbeds, threadName, component/file attachment, voice-media, and media size behavior. Update focused Discord/source tests, `docs/channels/discord.md`, and add a user-facing `CHANGELOG.md` fix/feature entry.
