# Coding Plan for openclaw/openclaw#24196: [Feature]: Allow max allowed file attachments on discord messages

Source report: [records/openclaw-openclaw/items/24196.md](records/openclaw-openclaw/items/24196.md)

## Summary

Keep open. Current main still represents multiple outbound media items upstream, but Discord delivery still funnels them through scalar `mediaUrl`/`filename` sends or one-file-at-a-time sequencing, so the requested native multi-file Discord message behavior remains unimplemented.

## Plan

Implement native multi-file Discord attachment sending for this issue. Keep the fix owned by the Discord plugin unless a tiny additive generic seam is unavoidable. Accept ordered multi-attachment input from existing message-tool/outbound `mediaUrls` and Discord action paths while preserving existing scalar `media`/`path`/`filePath`/`mediaUrl`/`filename` behavior. Load every file through the existing outbound media helpers and access checks, respect configured Discord media size caps, send up to the verified Discord per-message attachment cap in one `files` array, split overflow into follow-up messages, and keep reply-to, text chunking, components/embeds, silent/suppressEmbeds, forum-thread, and voice-message behavior intact. Verify the current Discord API attachment count/request-size contract before hard-coding limits. Update focused Discord/outbound tests, docs/channels/discord.md if the user-visible behavior needs documentation, and CHANGELOG.md for the user-facing enhancement. Do not broaden behavior for unrelated channels.
