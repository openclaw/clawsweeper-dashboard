## Summary

Current main and v2026.6.5 still send ordinary Discord media through a scalar one-file request, while the only documented multi-file path is the separate interactive `media-gallery` surface; the requested native batching remains useful and unimplemented.

## Plan

Implement native multi-file batching for ordinary Discord outbound messages for https://github.com/openclaw/openclaw/issues/24196. Keep ownership in the bundled Discord plugin: extend the adapter/send boundary to accept ordered attachments while preserving existing one-item `mediaUrl`/`filename` behavior, load every attachment through the existing outbound media access and size-cap helpers, send each platform-supported count batch in one Discord `files` array, and split overflow into ordered follow-up messages. Preserve reply targeting, caption/text chunking, components and `attachment://` references, forum/thread behavior, receipts, and existing behavior for other channels. Do not add a config option, change generic cross-channel fan-out without a required narrow seam, or edit CHANGELOG.md. Add focused unit coverage for one file, multiple files in one request, overflow batching/order, per-file names and guards, and relevant component/reply interactions. Include redacted real Discord after-fix proof in the PR body.
