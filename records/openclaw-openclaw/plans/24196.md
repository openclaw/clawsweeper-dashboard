## Summary

Keep open: current main and v2026.6.6 still expose Discord outbound uploads through scalar `mediaUrl`/`filename` paths, and multi-media payloads are still sequenced into separate sends rather than one native multi-file Discord message.

## Plan

Fix https://github.com/openclaw/openclaw/issues/24196 by adding native multi-file Discord outbound batching in the bundled Discord plugin. Accept ordered multi-attachment input from existing `mediaUrls`/structured attachment paths while preserving current single `media`/`path`/`filePath`/`filename` behavior. Load each file through the existing outbound media helpers and access checks, send up to the verified Discord per-message file limit in one `files` array, split overflow into follow-up messages, and keep replyTo, text chunking, silent/suppressEmbeds, embeds/components, receipt ids, and voice-message behavior deterministic. Prefer a Discord-owned implementation; do not change generic shared outbound fan-out semantics for other channels unless a minimal additive plugin-facing seam is truly required. Update focused Discord/outbound tests and docs; do not edit CHANGELOG.md for this normal fix, but include release-note context in the PR body.
