Source report: [records/openclaw-openclaw/items/24196.md](records/openclaw-openclaw/items/24196.md)

## Summary

Keep open. Current main still accepts multiple media references upstream, but regular Discord send/action paths reduce delivery to scalar `mediaUrl` calls or sequential one-file sends, so the native multi-file Discord message behavior remains unimplemented.

## Plan

Implement native multi-file sending for the bundled Discord plugin. Preserve existing scalar `media`/`mediaUrl`/`path`/`filePath`/`filename` behavior, collect ordered multiple attachment media references where they already exist, load each file through existing outbound media helpers/access checks, send up to the verified Discord per-message attachment cap in one `files` array, split overflow into follow-up messages, and keep voice-message, reply-to, text chunking, component/file block, silent, embed, and media size cap behavior intact. Do not add a new config option unless a maintainer explicitly asks. Update focused Discord/shared outbound tests, docs/channels/discord.md, and CHANGELOG.md for the user-facing change.
