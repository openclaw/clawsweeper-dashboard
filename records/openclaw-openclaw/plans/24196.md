## Summary

Keep open. Current main still accepts and sends Discord media through scalar `mediaUrl`/`filename` paths or one-file-at-a-time media sequencing, so the requested native multi-attachment Discord message behavior remains unimplemented.

## Plan

Implement native multi-file Discord sending for this issue. Preserve existing scalar `media`/`mediaUrl`/`path`/`filePath`/`filename` behavior, accept ordered multi-attachment input from existing `attachments`/`mediaUrls` surfaces, load each file through existing outbound media helpers and access checks, send batches in one Discord `files` array up to the verified Discord cap, split overflow into follow-up messages, and keep voice-message/component/embed/text-chunking/reply behavior intact. Prefer Discord-owned code over a broad shared outbound rewrite unless a small additive helper is needed. Update focused Discord tests, docs, and a user-facing changelog entry; do not add a config option unless maintainers explicitly decide tunable limits are needed.
