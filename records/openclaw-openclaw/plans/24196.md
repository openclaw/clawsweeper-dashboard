# Coding Plan for openclaw/openclaw#24196: [Feature]: Allow max allowed file attachments on discord messages

Source report: [records/openclaw-openclaw/items/24196.md](records/openclaw-openclaw/items/24196.md)

## Summary

Keep open. Current main still has the same Discord delivery gap: multi-media payloads can be represented upstream, but Discord sends still pass through scalar `mediaUrl`/`filename` paths or one-file-at-a-time sequencing.

## Plan

Implement native multi-file Discord outbound sending for this issue. Keep the fix owned by the bundled Discord plugin where possible: accept ordered multi-attachment input while preserving existing single `media`/`path`/`filePath`/`mediaUrl`/`filename` behavior, load every file through the existing outbound media helpers and access checks, send files together up to the verified current Discord per-message cap, and split overflow into follow-up messages. Do not broaden other channels' generic fan-out behavior unless a minimal additive plugin-facing seam is truly needed. Preserve reply-to behavior, text chunking, components/embeds behavior, audio-as-voice restrictions, configured media size caps, and media access safeguards. Add focused regression tests and update Discord docs plus changelog for the user-visible change.
