Keep open. Current main still carries multiple media references upstream, but the Discord send/action path remains scalar or fans media out as one file per message, so the requested native multi-file Discord message behavior is not implemented.

## Plan

Implement native multi-file Discord sends for https://github.com/openclaw/openclaw/issues/24196. Preserve existing single `media`/`mediaUrl`/`path`/`filePath`/`filename` behavior, voice handling, components/embeds, reply-to behavior, text chunking, media access checks, local-root/read-file handling, and existing shared outbound behavior for other channels. Accept ordered multi-attachment input from `mediaUrls` and message-tool `attachments`, load each file through the existing Discord media helpers, send batches up to the verified Discord per-message attachment cap in one `files` array, and split overflow into follow-up messages. Do not add a new config option or broad core outbound rewrite unless the local Discord implementation cannot express the behavior cleanly. Update focused Discord/outbound tests, docs, and changelog.
