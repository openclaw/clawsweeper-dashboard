## Summary

Keep open: current main still has a source-backed Codex app-server path where a completed raw assistant response is collected but the embedded run waits for `turn/completed`, then records a timeout error and suppresses the generic assistant event that Discord and Telegram delivery depend on. Related issues cover adjacent timeout/stall classes, but this issue adds a narrower final-response-not-delivered case that is not implemented by the merged quiescent-turn fix.

## Plan

Fix the Codex app-server embedded-run delivery gap reported in https://github.com/openclaw/openclaw/issues/82343. Current main collects assistant text from `rawResponseItem/completed` but, if no `turn/completed` arrives, times out and suppresses the generic assistant event that Discord/Telegram delivery uses. Add a narrow, well-guarded Codex app-server runner/projector fix so confirmed final raw assistant output can complete/deliver when terminal turn completion is missing. Preserve existing behavior for transitional/status/commentary raw assistant items, user/developer abort markers, completed `agentMessage` fallback, and non-assistant `custom_tool_call_output` watchdog handling. Do not change Discord or Telegram channel send code unless source evidence proves a channel-specific bug. Update focused regression tests and docs/changelog if user-visible behavior changes. Coordinate with https://github.com/openclaw/openclaw/issues/82274, https://github.com/openclaw/openclaw/issues/81114, and the open https://github.com/openclaw/openclaw/pull/81152 so the fix does not duplicate or conflict with progress-aware timeout work.
