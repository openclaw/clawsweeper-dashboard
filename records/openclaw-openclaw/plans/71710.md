# Coding Plan for openclaw/openclaw#71710: [Bug]: openclaw-agent ignores SIGTERM under cron, accumulates hung process chains and exhausts host RAM/swap

Source report: [records/openclaw-openclaw/items/71710.md](records/openclaw-openclaw/items/71710.md)

## Summary

Keep open. Current main still lacks a one-shot `openclaw agent` SIGTERM/SIGINT path that can cancel the pending Gateway request, abort the accepted Gateway run, or abort embedded fallback/local execution, and the latest comment adds a related but separable post-completion hang data point rather than resolving the SIGTERM case.

## Plan

Fix https://github.com/openclaw/openclaw/issues/71710 without broad supervisor refactors. Add a bounded one-shot `openclaw agent` shutdown path for SIGTERM/SIGINT: the CLI should create a caller-owned AbortController, pass cancellation through `callGateway`/`GatewayClient`, observe the accepted `agent` run id while still waiting for the final response, and on process signal abort the pending Gateway wait plus invoke the existing Gateway run abort path when the run id is known. Also wire the same signal into embedded fallback and `--local` agent execution so local one-shot runs can drain and exit. Add focused tests for pending Gateway call cancellation, accepted-run abort after an accepted ack, embedded fallback/local abort propagation, and preservation of existing session-lock cleanup behavior. Document external cron wrappers with `timeout -k <delay> <duration>` as a backstop, but do not make docs-only the fix. Do not change unrelated process supervisor policy from https://github.com/openclaw/openclaw/issues/66399 except where needed to interoperate with this CLI signal bridge; do not add a new user-facing config option unless maintainers explicitly request it. Add a changelog entry because this is user-visible reliability behavior.
