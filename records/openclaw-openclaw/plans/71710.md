# Coding Plan for openclaw/openclaw#71710: [Bug]: openclaw-agent ignores SIGTERM under cron, accumulates hung process chains and exhausts host RAM/swap

Source report: [records/openclaw-openclaw/items/71710.md](records/openclaw-openclaw/items/71710.md)

## Summary

Keep open. Current main still has the one-shot `openclaw agent` path waiting on a Gateway final response without a process-signal abort bridge, even though the Gateway-side `agent` abort seam exists; the cron-facing CLI docs also still lack a `timeout -k` backstop for external supervisors.

## Plan

Fix https://github.com/openclaw/openclaw/issues/71710 narrowly. Add SIGTERM/SIGINT handling for one-shot `openclaw agent` so a pending Gateway-backed run and any embedded/local fallback receive caller cancellation and the CLI exits within a bounded grace window. Extend `callGateway`/`GatewayClient.request` only as needed to accept a caller `AbortSignal` and surface the accepted `agent` run id while `expectFinal` waits; when the CLI has an accepted run, abort it through the existing Gateway `chat.abort`/agent abort-controller seam with the same session/run identity and without weakening authorization. Thread the same cancellation into embedded `agentCommand` fallback/local execution. Add focused regression coverage for Gateway pending final, accepted-run abort, embedded fallback/local abort, and cleanup of signal handlers. Document `timeout -k <delay>` or equivalent as a recommended cron backstop for external supervisors. Preserve existing `expectFinal`, timeout, auth scope, Gateway fallback, and `chat.abort` behavior; do not implement the broader exec supervisor SIGTERM work tracked by https://github.com/openclaw/openclaw/issues/66399 in this PR. Add a changelog entry because this is user-visible reliability behavior.
