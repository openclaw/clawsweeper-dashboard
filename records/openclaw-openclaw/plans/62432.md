# Coding Plan for openclaw/openclaw#62432: [Bug]: Xiaomi/MiMo sessions can repeatedly relaunch exec after 'Command still running' instead of switching to process poll

Source report: [records/openclaw-openclaw/items/62432.md](records/openclaw-openclaw/items/62432.md)

## Summary

Current main still accepts a repeated identical long-running exec by starting a new process session before returning the non-terminal running result; the only repeated-call protection I found is opt-in loop detection, which remains disabled by default.

## Plan

Fix the default repeated-running exec fan-out reported in https://github.com/openclaw/openclaw/issues/62432. Add a provider-agnostic guard in the exec/process layer so an identical in-flight long-running exec from the same relevant session/scope/workdir/command does not spawn another process after `Command still running`; instead return the existing running session or a clear `process(action="poll")` requirement. Keep matching narrow so legitimate intentional duplicate commands are not globally denied. Do not make this Xiaomi/MiMo-specific and do not rely on `tools.loopDetection.enabled`, which is disabled by default. Add focused regression coverage for the default disabled-loop-detection path and the exec/process registry behavior. Include a changelog entry if the runtime/user-visible behavior changes.
