# Coding Plan for openclaw/openclaw#85869: Plugin install should guard WhatsApp plugin/core version compatibility

Source report: [records/openclaw-openclaw/items/85869.md](records/openclaw-openclaw/items/85869.md)

## Summary

Keep open: current main and the latest release still let the direct npm/package install path rely on `openclaw.install.minHostVersion` while WhatsApp advertises a newer OpenClaw peer/plugin API requirement, so the reported version-skew install remains source-reproducible. The later issue comment also confirms this as a real production pain point and points to a compatibility-guard pattern rather than changing the product direction.

## Plan

Fix https://github.com/openclaw/openclaw/issues/85869 with one focused PR. Current main lets a floating `openclaw plugins install @openclaw/whatsapp` pass when `openclaw.install.minHostVersion` is satisfied even though the package declares a newer OpenClaw/plugin API peer requirement. Add a generic direct npm/package install compatibility guard before install persistence; prefer using existing semver/compat helpers and avoid a WhatsApp-only special case. Align WhatsApp official/package metadata if needed, and add regression coverage for a host version like 2026.5.10-beta.1 installing a package whose `peerDependencies.openclaw` or `openclaw.compat.pluginApi` requires a newer host. Do not change plugin distribution strategy or add a new config option. Add a changelog entry if the user-visible CLI install error changes.
