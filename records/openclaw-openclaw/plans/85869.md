# Coding Plan for openclaw/openclaw#85869: Plugin install should guard WhatsApp plugin/core version compatibility

Source report: [records/openclaw-openclaw/items/85869.md](records/openclaw-openclaw/items/85869.md)

## Summary

Keep open: current main and the latest release still allow the reported WhatsApp version-skew path because direct npm/package install enforces only `openclaw.install.minHostVersion`, while the package advertises a newer OpenClaw/plugin API requirement. The later commenter also corroborated this as a real production pattern, so this remains a narrow source-backed compatibility bug rather than cleanup-close material.

## Plan

Fix https://github.com/openclaw/openclaw/issues/85869 by adding a generic package compatibility guard for external plugin installs. The bug: a floating `openclaw plugins install @openclaw/whatsapp` can install a package whose `openclaw.compat.pluginApi` or `peerDependencies.openclaw` requires a newer OpenClaw core than the running host, because the direct npm/local install path currently only enforces `openclaw.install.minHostVersion`. Add or reuse a shared compatibility helper that checks declared plugin API/OpenClaw compatibility before persisting/installing a package, and apply it to npm/local package install plus non-bundled manifest-registry loading where appropriate. Keep the fix generic; do not special-case WhatsApp, do not add a new config option, do not auto-update packages during gateway startup, and do not compare all third-party package versions directly to the gateway version. Add focused regression tests for the WhatsApp-style case where minHostVersion is satisfied but pluginApi/peer dependency is too new, plus compatible-package pass cases and ClawHub parity where useful. Update docs only if the public manifest/install compatibility contract changes; changelog is expected because this is user-visible install behavior.
