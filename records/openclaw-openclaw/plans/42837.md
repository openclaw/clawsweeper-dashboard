## Summary

Keep open: current main still creates Feishu dynamic-agent bindings without `match.accountId`, while named-account routing only considers bindings for the active account or wildcard account, so named/non-default Feishu DM traffic can still fall through to the main agent. The merged docs PR documents the limitation rather than fixing it, and the latest release still contains the same account-less dynamic binding creation path.

## Plan

Fix Feishu dynamicAgentCreation routing for named/non-default accounts without changing non-Feishu routing or adding new config. Current behavior: `extensions/feishu/src/bot.ts` routes Feishu DMs with `account.accountId`, but `extensions/feishu/src/dynamic-agent.ts` checks and writes dynamic bindings using only `match.channel` and `match.peer`; `src/routing/resolve-route.ts` treats account-less bindings as default-account-only, so named-account DMs can fall back to `agent:main`. Update the Feishu dynamic-agent helper and its DM caller so lookup and newly written bindings are account-aware for the active Feishu account, while preserving legacy account-less behavior for the default account and avoiding repeated duplicate bindings. Add focused regression coverage for a named Feishu account DM that creates or finds the dynamic binding and routes to the per-user agent, plus coverage that default-account legacy bindings still work. Do not change group routing, broadcast behavior, session defaults, docs-only behavior, unrelated channels, or CHANGELOG.md.
