# Coding Plan for openclaw/openclaw#42837: [Bug]: Feishu dynamicAgentCreation feature not working

Source report: [records/openclaw-openclaw/items/42837.md](records/openclaw-openclaw/items/42837.md)

## Summary

Keep open: current main still creates Feishu dynamic-agent direct bindings without `match.accountId`, while DM and comment routing re-resolves with the runtime `account.accountId`, so named-account traffic can miss the generated binding and fall back to `agent:main`. The closed related code PR identified the right account-aware shape, while the open docs PR documents the default-account behavior and does not fix named accounts.

## Plan

Fix https://github.com/openclaw/openclaw/issues/42837 as a focused Feishu plugin bug fix. Current main passes `account.accountId` into Feishu route resolution but `maybeCreateDynamicAgent` does not accept an account id, detects existing direct bindings only by channel/peer, and writes generated bindings without `match.accountId`; named-account DMs/comments can therefore miss the generated binding and route to `agent:main`. Update `extensions/feishu/src/dynamic-agent.ts` to accept `accountId`, use `DEFAULT_ACCOUNT_ID`/`normalizeAccountId` from `openclaw/plugin-sdk/account-id`, treat legacy account-less bindings as default-account-only and `accountId: "*"` as any-account, and write normalized `match.accountId` for both missing-binding and new-agent paths. Update `extensions/feishu/src/bot.ts` and `extensions/feishu/src/comment-handler.ts` to pass the runtime account id while preserving the current `configWritesAllowed`, `replaceConfigFile`, and `afterWrite` behavior. Add/update focused tests for named-account DM and comment dynamic creation, default-account legacy behavior, wildcard matching if covered, and the full new-agent creation path. Do not change core routing, group routing, broadcast behavior, provider/model routing, or sender-derived workspace path-template semantics. Add a user-facing changelog entry for the Feishu routing fix unless maintainers say otherwise.
