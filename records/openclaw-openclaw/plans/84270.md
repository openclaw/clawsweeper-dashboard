## Summary

Keep open: current main still bases completion cleanup on the later parent announce delivery result, so a parent `NO_REPLY` after an already-delivered Slack completion can still be treated as undelivered and retried.

## Plan

Fix the subagent completion delivery accounting bug reported at https://github.com/openclaw/openclaw/issues/84270. In current main, completion cleanup only trusts the later parent announce result; when the child already delivered the Slack requester-thread completion and the parent later returns exact NO_REPLY without a message(send), cleanup retries and can suspend at retry-limit. Add a route/thread-scoped committed delivery evidence contract so cleanup marks the run delivered, clears pending final delivery, avoids incrementing announceRetryCount, and does not log give-up when the same requester route/thread already has successful completion delivery evidence. Preserve existing retry behavior when no matching evidence exists, and do not make generic NO_REPLY alone count as delivered. Keep the fix in the subagent announce/registry lifecycle boundary; avoid broad channel rewrites. Add focused regression coverage for successful prior Slack/message-tool or delivery-mirror evidence plus parent NO_REPLY, and add a changelog entry if the final patch is user-visible.
