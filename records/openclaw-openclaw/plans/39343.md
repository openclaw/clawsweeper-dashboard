# Coding Plan for openclaw/openclaw#39343: Feature Request: Image batching / media group buffering at gateway layer

Source report: [records/openclaw-openclaw/items/39343.md](records/openclaw-openclaw/items/39343.md)

## Summary

Keep open. The Telegram album buffering/config half is on current main, but the LINE plugin still processes each downloadable media webhook event independently; the LINE SDK exposes optional image-set metadata that OpenClaw does not use, and there is no LINE media batching config or docs surface.

## Plan

Implement the remaining LINE media batching behavior for https://github.com/openclaw/openclaw/issues/39343. Keep the change inside the LINE plugin boundary; do not add a generic gateway-layer media buffer and do not change Telegram album behavior. Add an optional LINE config key for media batching delay, defaulting to existing behavior unless there is already a local convention that clearly supports an opt-in default. Prefer LINE message.imageSet.id/index/total when present, and fall back to an account + conversation + sender + short time-window key for clients without imageSet metadata. Buffer downloadable LINE media, flush on timer and before same-key text/non-media dispatch, aggregate media into one LineInboundContext, and preserve access gating, webhook replay dedupe, mediaMaxMb handling, group history, and reply-token fallback semantics. Add focused tests for same-sender image batching, separation by account/conversation/sender, text-triggered flush, optional imageSet grouping, and disabled/default behavior. Update LINE schema/types/docs/config metadata and add a changelog entry because this is user-visible.
