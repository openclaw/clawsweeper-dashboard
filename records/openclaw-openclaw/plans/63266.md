# Coding Plan for openclaw/openclaw#63266: Docker to host migration can leave workspace-local and workspace-local/local paths split, breaking local secrets assumptions

Source report: [records/openclaw-openclaw/items/63266.md](records/openclaw-openclaw/items/63266.md)

## Summary

Keep open. Current main still has the legacy workspace detector as a no-op, so doctor/status cannot warn about the reported active child workspace with parent-held operational files, and the docs still do not explain the Docker-to-host parent/child workspace split.

## Plan

Fix the Docker-to-host split workspace diagnostic without changing runtime secret semantics. Add a narrow read-only detector for an active child workspace like `~/.openclaw/workspace-local/local` when the parent `~/.openclaw/workspace-local` contains meaningful operational markers such as bootstrap files, memory files, skills, config-like directories, or operator secret folders. Surface the active workspace and suspicious parent-held files through doctor/status; do not silently move files, do not treat parent-level `secrets/homeassistant.md` as an official SecretRef location, and keep any repair/migration opt-in. Update docs to clarify canonical host workspace and Docker bind-mount locations after migration, including the parent/child split troubleshooting guidance. Add focused regression coverage for the detector/status warning and keep existing no-warning behavior for a single active workspace.
