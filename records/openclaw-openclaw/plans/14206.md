## Summary

Keep open. Current main and v2026.6.5 have Feishu retry-after-rate-limit handling, but they still lack the requested proactive outbound minimum-interval/pacing configuration across Feishu send paths.

## Plan

Implement a narrow Feishu-plugin proactive outbound pacing layer for https://github.com/openclaw/openclaw/issues/14206. Add a conservative, documented config surface under channels.feishu and per-account accounts.<id> that controls the minimum interval between Feishu outbound message API calls. Keep ownership inside extensions/feishu; do not add a core-wide limiter, a new plugin SDK seam, or unrelated provider/channel behavior. Route all burst-capable Feishu delivery surfaces through the same per-account pacing helper: chunked normal replies, card replies, no-visible-reply fallback, comment-thread replies, message-tool outbound text/card paths, media message create/reply after upload, and streaming-card initial send. Preserve existing retry-after-rate-limit behavior in requestFeishuApi; pacing should happen before send attempts and should not replace retry handling. Update config schema, plugin manifest metadata, Feishu docs, and focused tests for top-level and per-account config inheritance plus affected send paths. Do not edit CHANGELOG.md.
