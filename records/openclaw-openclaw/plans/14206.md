## Summary

Current main and the latest release still lack the requested proactive Feishu outbound pacing/minimum-interval configuration; the merged Feishu retry work improves recovery after rate-limit responses but does not prevent bursty sends before Feishu sees them.

## Plan

Implement Feishu proactive outbound pacing for https://github.com/openclaw/openclaw/issues/14206. Add a Feishu-owned minimum-send-interval scheduler/config surface at top level and per account with a conservative default that does not surprise existing users; wire it around the shared Feishu message delivery boundary so chunked replies, comment replies, message-tool outbound, media/card sends, fallbacks, and streaming-card initial sends all share the intended pacing. Do not replace or weaken the existing requestFeishuApi retry/backoff handling from https://github.com/openclaw/openclaw/pull/89659, do not add a broad core rate-limit framework, and do not edit CHANGELOG.md. Update Feishu schema, manifest/config metadata, docs, and focused tests; stop and ask for maintainer direction if the implementation would require changing global message chunking semantics or adding cross-provider limits.
