## Summary

Keep open: current main still lacks a Feishu outbound pacing/minimum-interval configuration, and the Feishu reply, comment, outbound, media, and streaming-card paths still send directly through Feishu APIs without a shared proactive pacing helper. The broader rate-limit and retry/backoff trackers are related context but do not supersede this Feishu-specific outbound pacing request.

## Plan

Implement Feishu outbound message pacing for https://github.com/openclaw/openclaw/issues/14206. Add a Feishu-owned per-account pacing helper and configurable minimum interval, with top-level and accounts.<id> support, schema/manifest/docs updates, and focused regression tests. Apply the helper consistently to burst-prone Feishu delivery surfaces: normal reply chunks, comment-thread replies, message-tool outbound text/cards, media fallback sends and media message sends, and streaming-card create/update/close operations where they count as outbound Feishu API calls. Keep the change inside the Feishu plugin unless a small generic helper is already available through the plugin SDK; do not introduce a broad core rate-limit framework or solve retry/backoff here. Leave https://github.com/openclaw/openclaw/issues/13615 for cross-provider rate limiting and https://github.com/openclaw/openclaw/issues/70879 for retry/backoff after Feishu rate-limit errors. Add a user-facing CHANGELOG.md entry because this adds config/behavior, but keep attribution policy-compliant. Stop and return for maintainer review if the implementation requires a new core plugin API or a product decision beyond conservative config naming/defaults.
