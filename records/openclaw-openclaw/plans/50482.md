## Summary

Keep open. Current main and the latest release still collapse Telegram voice notes into generic audio/transcript context without an agent-visible voice-origin or source-modality marker, and the latest discussion usefully narrows the preferred shape to generic provenance fields rather than a Telegram-only flag.

## Plan

Implement https://github.com/openclaw/openclaw/issues/50482 as a backwards-compatible inbound provenance feature. Add a generic current-message provenance contract, preferably a source_modality enum plus a bounded source_traits bag, to the shared channel/MsgContext path; populate Telegram voice notes before resolveTelegramPrimaryMedia/STT collapses them into generic audio; render the new fields in inbound user metadata; add regression coverage for Telegram voice notes versus ordinary audio/text and update relevant SDK/channel docs. Coordinate naming with https://github.com/openclaw/openclaw/issues/40838, but do not solve broad sender/bot metadata or expose raw Telegram Message objects. Add a maintainer-owned changelog entry if this lands as a user-facing agent-context change.
