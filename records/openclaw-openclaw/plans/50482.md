## Summary

Keep open. Current main and v2026.5.19 still collapse Telegram voice notes into generic audio/transcript context, and the shared inbound context plus rendered metadata do not expose voice/source-modality provenance; the broader Telegram metadata issue can inform naming but does not replace this focused request.

## Plan

Implement https://github.com/openclaw/openclaw/issues/50482 as a narrow additive agent-context feature. Add a generic, backward-compatible source-modality contract rather than a Telegram-only `is_voice` field: carry `source_modality` and a small `source_traits` bag from Telegram before `msg.voice` is flattened to generic audio, expose it through the shared channel turn/context types and finalized `MsgContext`, render it in inbound metadata in the dynamic user conversation-info path, and document the new field. Cover Telegram voice notes versus ordinary audio/text and preserve existing STT behavior, media placeholders, prompt-cache stability for the system metadata block, and channel/plugin ownership boundaries. Coordinate naming with https://github.com/openclaw/openclaw/issues/40838; do not expose raw Telegram message objects or add config knobs. Add a concise `CHANGELOG.md` Changes entry because this changes agent-visible context.
