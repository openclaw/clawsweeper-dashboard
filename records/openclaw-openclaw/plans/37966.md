## Summary

Keep this issue open: current main and v2026.5.12 still route the bundled LiteLLM provider through `openai-completions`, where explicit LiteLLM Anthropic `cacheRetention` neither resolves to stream options nor triggers payload-level `cache_control` injection. The related custom `anthropic-messages` fixes are merged, but the LiteLLM OpenAI-compatible proxy path reported here remains unhandled and the prior LiteLLM-specific PRs are closed unmerged.

## Plan

Fix https://github.com/openclaw/openclaw/issues/37966 on current main. Implement LiteLLM prompt-cache payload handling for Anthropic-backed LiteLLM models on the bundled `openai-completions` path. Prefer the LiteLLM provider/plugin boundary: use public `openclaw/plugin-sdk/provider-stream` helpers such as `streamWithPayloadPatch` and `applyAnthropicEphemeralCacheControlMarkers` rather than importing core internals from `extensions/litellm`. Gate injection to explicit cache enablement only (`cacheRetention: "short"` or `"long"`, and legacy `cacheControlTtl: "5m"` or `"1h"` if still supported); do not inject when config is absent, invalid, or `cacheRetention: "none"`. Restrict Anthropic detection to clear Claude/LiteLLM Anthropic refs such as `claude-*`, `anthropic/claude-*`, and `anthropic.claude-*`; do not affect GPT, Gemini, or arbitrary LiteLLM models. Preserve existing direct Anthropic, Bedrock, custom `anthropic-messages`, and OpenRouter behavior. Add focused regression tests for string and array system content, short/long positive cases, `none` opt-out, absent config, legacy TTL if applicable, and non-Anthropic LiteLLM models. Add a single-line changelog fix entry because this is user-visible. Stop and ask if the implementation appears to require a new public plugin API or a broader product decision.
