# Coding Plan for openclaw/openclaw#37966: [Bug]: cacheRetention ignored for LiteLLM-proxied Anthropic models

Source report: [records/openclaw-openclaw/items/37966.md](records/openclaw-openclaw/items/37966.md)

## Summary

Keep open: current main and the latest shipped release still leave bundled LiteLLM on the OpenAI-compatible route, where explicit `cacheRetention` for `litellm/claude-*` does not resolve to Anthropic cache behavior and no LiteLLM-specific payload wrapper injects `cache_control` markers. The related LiteLLM fix attempts are closed unmerged, and the open OpenRouter PR covers a different provider path.

## Plan

Fix https://github.com/openclaw/openclaw/issues/37966 on current main. The observable bug is that `agents.defaults.models["litellm/claude-opus-4-6"].params.cacheRetention` is ignored on bundled LiteLLM because LiteLLM uses `openai-completions`, `resolveCacheRetention()` does not classify that route as Anthropic cache eligible, and only OpenRouter injects Anthropic `cache_control` payload markers. Implement a narrow LiteLLM Anthropic prompt-cache wrapper, preferably through the LiteLLM provider-owned `wrapStreamFn` seam using SDK-exported/shared payload helpers instead of hardcoding broad LiteLLM policy in generic core. Inject Anthropic `cache_control` markers only for LiteLLM model IDs that clearly target Claude/Anthropic and only when effective retention is `short` or `long`; `cacheRetention: "none"` must not inject markers. Do not widen arbitrary OpenAI-compatible proxies or non-Anthropic LiteLLM models. Reuse existing `applyAnthropicEphemeralCacheControlMarkers`/payload policy where appropriate, add regression coverage for string and array system/developer content, absent config, `none`, `short`, `long`, and non-Anthropic LiteLLM models, update `docs/providers/litellm.md` and `docs/reference/prompt-caching.md` if behavior changes, and add a single-line `CHANGELOG.md` fix entry.
