Keep this issue open: current `main` still routes bundled LiteLLM through the `openai-completions` transport, while the current resolver and Anthropic `cache_control` injector do not cover that default LiteLLM Anthropic proxy path.

## Plan

Fix https://github.com/openclaw/openclaw/issues/37966. Current main still routes bundled LiteLLM through `api: "openai-completions"`, so explicit `agents.defaults.models["litellm/claude-..."] .params.cacheRetention` does not produce Anthropic `cache_control` markers. Implement the fix at the LiteLLM provider boundary, preferably with a `wrapStreamFn` in `extensions/litellm/index.ts` using public `openclaw/plugin-sdk/provider-stream` helpers rather than importing core internals directly. Inject Anthropic cache markers only when the selected LiteLLM model ref looks Anthropic/Claude-like and effective `cacheRetention` is `"short"` or `"long"`; do not inject for absent config, `cacheRetention: "none"`, invalid values, or non-Anthropic LiteLLM models. Keep core plugin-agnostic, avoid broad resolver changes for all `openai-completions` providers unless required by the implementation, and coordinate with https://github.com/openclaw/openclaw/pull/82973 if it lands first. Add focused regression coverage for LiteLLM string/array system message payloads, non-Anthropic models, no config, and explicit `none`; add a user-facing changelog fix entry if the repair PR is prepared for landing.
