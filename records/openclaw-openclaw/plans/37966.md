## Summary

Keep this issue open: current main and the latest release still leave bundled LiteLLM on the `openai-completions` path without a LiteLLM-specific Anthropic `cache_control` injector, so explicit `cacheRetention` remains ineffective for the reported setup.

## Plan

Fix https://github.com/openclaw/openclaw/issues/37966. Current main ignores explicit `agents.defaults.models["litellm/claude-opus-4-6"].params.cacheRetention` because bundled LiteLLM is `openai-completions`, `resolveCacheRetention` only admits direct/custom `anthropic-messages` and Google families, and the Anthropic `cache_control` payload wrapper is OpenRouter-scoped. Implement a LiteLLM-owned wrapper through the provider/plugin stream surface: for `provider === "litellm"` and Claude-like LiteLLM model ids (`claude-*`, `anthropic/claude-*`, `anthropic.claude*`), inject Anthropic `cache_control: { type: "ephemeral" }` into eligible system/developer content blocks only when effective `cacheRetention` is `"short"` or `"long"`; preserve absent config and `cacheRetention: "none"` as no marker injection. Do not globally enable Anthropic markers for arbitrary OpenAI-compatible providers, do not duplicate OpenRouter long-TTL work, and do not change non-Anthropic Bedrock semantics. Add focused regression tests for enabled short/long, explicit none, absent config, non-Anthropic LiteLLM models, and string/array system content. Add a changelog entry if this becomes a user-facing bug-fix PR under maintainer landing policy.
