# Coding Plan for openclaw/openclaw#37966: [Bug]: cacheRetention ignored for LiteLLM-proxied Anthropic models

Source report: [records/openclaw-openclaw/items/37966.md](records/openclaw-openclaw/items/37966.md)

## Summary

Keep open: current main still does not honor explicit `cacheRetention` for the default LiteLLM Claude path because LiteLLM models are registered as OpenAI-compatible completions, while the existing cache-retention and `cache_control` marker paths only cover direct Anthropic, Bedrock, custom `anthropic-messages`, Google, and OpenRouter-specific routes.

## Plan

Fix https://github.com/openclaw/openclaw/issues/37966 on current main. The observable bug is that explicit `agents.defaults.models["litellm/claude-opus-4-6"].params.cacheRetention` is ignored because LiteLLM registers Claude through `openai-completions`, `resolveCacheRetention` does not classify that path, and no LiteLLM payload wrapper injects Anthropic `cache_control` markers. Implement the narrow provider-owned fix: add or reuse a stream-wrapper helper for LiteLLM Anthropic/Claude-like models that injects `cache_control` into system/developer message content only when effective extra params explicitly request caching (`cacheRetention: "short"` or `"long"`, or legacy cacheControlTtl values), skips injection for `cacheRetention: "none"` and absent/invalid config, and does not affect non-Anthropic LiteLLM models or arbitrary OpenAI-compatible providers. Prefer keeping LiteLLM-specific detection in `extensions/litellm` via `wrapStreamFn` and public `openclaw/plugin-sdk/*` helpers; if a small shared helper is needed, expose it through the plugin SDK rather than deep-importing core internals. Coordinate with any landed changes from https://github.com/openclaw/openclaw/pull/79370 so long TTL/none semantics are consistent, and add a user-facing changelog fix entry if this changes shipped behavior. Do not broaden implicit prompt caching defaults or add a new config key.
