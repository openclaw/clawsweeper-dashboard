# Coding Plan for openclaw/openclaw#65235: [UX] Suppress tool parameter validation errors from being sent to user chat surfaces

Source report: [records/openclaw-openclaw/items/65235.md](records/openclaw-openclaw/items/65235.md)

## Summary

Keep open. Current main still routes mutating validation-shaped tool failures through the final user-facing warning path, and the related closed retryability PR did not implement chat-surface suppression for this UX bug.

## Plan

Fix https://github.com/openclaw/openclaw/issues/65235. Current main still sends mutating tool parameter-validation failures to user chat as warning payloads because validation failures are serialized as generic tool errors and ToolErrorSummary has no validation/internal-only classification. Add a typed classification for missing/malformed tool-parameter validation failures, thread it through the adapter/tool execution end state into ToolErrorSummary, and suppress only that class in buildEmbeddedRunPayloads. Cover write/read/edit required-param failures and exec missing-command or equivalent malformed-parameter failures where the current path can be classified safely. Preserve visible warnings for real unresolved mutating side-effect failures, preserve verbose details where appropriate for non-validation failures, and do not implement the broader configurable pre-dispatch validation feature from https://github.com/openclaw/openclaw/issues/56771. Do not revive the ad-hoc retryable-only approach from https://github.com/openclaw/openclaw/pull/68070 unless it becomes a typed contract that also drives chat visibility. Add focused regression tests and a CHANGELOG.md Fixes entry unless maintainers decide this is release-note-exempt.
