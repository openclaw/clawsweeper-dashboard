# Coding Plan for openclaw/openclaw#61569: [Bug]: Custom node ID via `openclaw run node --node-id` not honored

Source report: [records/openclaw-openclaw/items/61569.md](records/openclaw-openclaw/items/61569.md)

## Summary

Keep this issue open. Current main and the latest shipped tag still forward the CLI override as `client.instanceId` but resolve gateway node identity from `device.id` or `client.id`, so the custom node id is still not usable for node lookup or exec-node routing.

## Plan

Fix https://github.com/openclaw/openclaw/issues/61569. The CLI node host already accepts `openclaw node run --node-id` and sends it as `connect.client.instanceId`, but gateway node identity paths still use `connect.device?.id ?? connect.client.id`. Add a shared resolver for gateway node ids where a trimmed non-empty `connect.client.instanceId` wins, then use it consistently for `NodeRegistry.register`, `reconcileNodePairingOnConnect` and its paired-node lookup, known-node catalog/list/describe behavior, `node.pending.drain`, and `node.invoke.result` caller validation. Preserve existing device-id and client-id fallbacks, device-token auth, node-pairing approval semantics, and approved command/capability behavior. Use the prior closed PRs https://github.com/openclaw/openclaw/pull/61587 and https://github.com/openclaw/openclaw/pull/61719 only as context; do not copy their incomplete shape blindly. Add focused regression tests for instanceId winning over device id and client id, blank/whitespace fallback, pairing reconciliation, pending drain, invoke-result mismatch validation, and CLI nodes list/describe where feasible. This is user-visible, so include a concise `CHANGELOG.md` fix entry unless maintainer landing policy says to defer it.
