# Coding Plan for openclaw/openclaw#61569: [Bug]: Custom node ID via `openclaw run node --node-id` not honored

Source report: [records/openclaw-openclaw/items/61569.md](records/openclaw-openclaw/items/61569.md)

## Summary

Keep this issue open. Current `main` still advertises and forwards `--node-id` as `connect.client.instanceId`, but the gateway still registers, reconciles, catalogs, and drains node work under `device.id` or `client.id`, so the custom ID remains unusable for node lookup and exec routing.

## Plan

Fix https://github.com/openclaw/openclaw/issues/61569 by making custom node ids from `openclaw node run/install --node-id` authoritative wherever the gateway identifies a node. Add a shared resolver for `ConnectParams`/connected clients that returns trimmed non-empty `connect.client.instanceId` before falling back to `connect.device?.id` and `connect.client.id`; use it in `NodeRegistry.register`, `reconcileNodePairingOnConnect`, node catalog/list/describe inputs as needed, pending work drain/enqueue/reconnect checks, and invoke-result/system-run validation paths that compare node ids. Preserve existing fallback behavior when `instanceId` is absent or blank, preserve device-token and node-pairing authorization semantics, do not bump the gateway protocol version because `client.instanceId` already exists, and avoid broad node/pairing refactors. Add focused regression tests for custom id lookup, fallback order, pairing metadata, pending work, and exec-node resolution. Add a changelog fix entry in the active release section if this is prepared as a maintainer fix PR.
