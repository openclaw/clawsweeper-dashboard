# Coding Plan for openclaw/openclaw#61569: [Bug]: Custom node ID via `openclaw run node --node-id` not honored

Source report: [records/openclaw-openclaw/items/61569.md](records/openclaw-openclaw/items/61569.md)

## Summary

Keep this issue open. Current main still forwards the CLI override as `connect.client.instanceId`, but gateway node registration, paired-node lookup/reconciliation, pending-work drain, and invoke-result validation still key the node from `device.id` or `client.id`, so the custom id remains unresolvable by `node.list`, `node.describe`, and node invocation paths.

## Plan

Fix the current-main bug where `openclaw node run --node-id <id>` is serialized as `connect.client.instanceId` but gateway node identity still resolves from `device.id ?? client.id`. Add a shared gateway helper that returns a trimmed non-empty `connect.client.instanceId` before falling back to `connect.device?.id` and `connect.client.id`, then use it consistently for `NodeRegistry.register()`, WebSocket paired-node lookup, `reconcileNodePairingOnConnect()`, `node.pending.drain`, and `node.invoke.result` caller checks. Add focused regression coverage proving a node connected with `instanceId: "my-mac-node"` appears in `node.list`, is accepted by `node.describe`, can be invoked/result-matched by that id, and drains pending work under that id. Do not change CLI option names, docs semantics, device-pairing security policy, or introduce a new config option. Add a single `CHANGELOG.md` fix entry if the repair changes user-visible behavior.
