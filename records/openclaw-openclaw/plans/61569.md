# Coding Plan for openclaw/openclaw#61569: [Bug]: Custom node ID via `openclaw run node --node-id` not honored

Source report: [records/openclaw-openclaw/items/61569.md](records/openclaw-openclaw/items/61569.md)

## Summary

Keep this issue open. Current main still forwards the CLI override as `connect.client.instanceId`, but the gateway continues to key live registration, pairing reconciliation, pending-node work, and invoke-result validation from `device.id` or `client.id`, so the custom id remains unreliable for node lookup and execution.

## Plan

Fix https://github.com/openclaw/openclaw/issues/61569. `openclaw node run --node-id <id>` is documented and already sent as `connect.client.instanceId`, but gateway node identity still falls back to `device.id`/`client.id` in several paths. Add a small shared gateway helper that resolves the effective node id with trimmed non-empty `connect.client.instanceId` first, then device id, then client id. Use it consistently in node registration, node pairing reconciliation/lookup, last-connect metadata, node catalog/list/describe dedupe if needed, pending node pull/ack/drain behavior, and `node.invoke.result` caller validation. Do not weaken device-token or pairing authorization checks, do not add a new config option, and avoid broad node/pairing refactors. Add focused regression coverage for custom node id with device identity present, including describe/list and invoke/result or pending paths as practical. Add a user-facing changelog fix entry if the landing workflow expects one.
