# Coding Plan for openclaw/openclaw#30381: chatCompletions: ignore request model when x-openclaw-agent-id header is present

Source report: [records/openclaw-openclaw/items/30381.md](records/openclaw-openclaw/items/30381.md)

## Summary

Keep open. Current main and the latest release still select the explicit agent header, then reject a generic OpenAI request-body model before dispatch; the production SlurpBot Ops comments support that this is still an active compatibility bug, and the related PR is closed unmerged.

## Plan

Fix https://github.com/openclaw/openclaw/issues/30381. In OpenAI-compatible gateway requests, when `x-openclaw-agent-id` or `x-openclaw-agent` resolves to a valid configured agent, do not treat the request-body `model` as an agent selector and do not reject generic OpenAI model ids like `gpt-4`; the selected agent should use its configured model unless `x-openclaw-model` is supplied. Do not allow unknown explicit agent ids to fall back to `main`. Preserve `model: "openclaw"`, `model: "openclaw/<agentId>"`, `model: "agent:<agentId>"`, default-agent routing, invalid explicit selector errors, and `x-openclaw-model` allowlist enforcement. Likely touch `src/gateway/http-utils.ts`, `src/gateway/openai-http.ts`, `src/gateway/openresponses-http.ts`, their focused tests, and docs only if the public contract needs clarification. Add or update a user-facing `CHANGELOG.md` fix entry if the repair changes shipped behavior.
