# Coding Plan for openclaw/openclaw#70879: Feishu plugin lacks retry logic for API rate limit errors

Source report: [records/openclaw-openclaw/items/70879.md](records/openclaw-openclaw/items/70879.md)

## Summary

Keep open. Current main and v2026.5.20 still treat Feishu outbound rate-limit send failures as terminal in text/card, media, and initial streaming-card delivery paths; the existing backoff classifier is only wired to typing indicators, while proactive pacing, duplicate media idempotency, and delivery-failure notification are separate open items.

## Plan

Fix the Feishu outbound delivery retry gap reported at https://github.com/openclaw/openclaw/issues/70879. Add a Feishu-owned bounded retry/backoff helper for retryable delivery failures from message create/reply sends, including thrown SDK/Axios-style errors and nonzero API response codes after checking current Feishu/Lark error-code contracts. Apply it to text/card sends in extensions/feishu/src/send.ts, media sends in extensions/feishu/src/media.ts, and the initial streaming-card message send in extensions/feishu/src/streaming-card.ts; reuse or factor the existing typing backoff classifier only if the names and semantics stay accurate. Keep the fix plugin-local, do not add a new config option, do not implement proactive pacing from https://github.com/openclaw/openclaw/issues/14206, do not solve media idempotency from https://github.com/openclaw/openclaw/issues/64448, and do not add the broader delivery-failure notification feature from https://github.com/openclaw/openclaw/issues/46830. Add focused Vitest coverage proving retryable create/reply failures are retried and bounded, non-retryable errors remain terminal, withdrawn-message fallback behavior still works, and media retries do not introduce duplicate successful sends. Add a CHANGELOG.md fix entry because this changes user-visible Feishu delivery reliability.
