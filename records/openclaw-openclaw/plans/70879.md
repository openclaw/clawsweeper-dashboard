# Coding Plan for openclaw/openclaw#70879: Feishu plugin lacks retry logic for API rate limit errors

Source report: [records/openclaw-openclaw/items/70879.md](records/openclaw-openclaw/items/70879.md)

## Summary

Keep open. Current main still sends Feishu delivery create/reply calls as one-shot operations for text, cards, media, and initial streaming cards; the existing backoff handling is scoped to typing indicators, and the related pacing and duplicate-media reports track different remaining work.

## Plan

Fix https://github.com/openclaw/openclaw/issues/70879 by adding bounded Feishu-plugin-owned retry/backoff for outbound delivery sends that fail with verified retryable Feishu/Lark rate-limit or flow-control responses. Keep the scope inside the Feishu plugin: do not add a core retry layer, do not add a public pacing config, and do not implement the proactive minimum-interval feature tracked by https://github.com/openclaw/openclaw/issues/14206. Verify retryable error/response shapes from official Feishu/Lark docs or SDK source before coding. Apply the retry helper consistently to sendMessageFeishu/sendCardFeishu reply and create flows, media message create/reply after upload, and initial streaming-card message create/reply where safe. Preserve withdrawn-reply fallback behavior and guard against duplicate-delivery regressions, especially the media idempotency risk tracked by https://github.com/openclaw/openclaw/issues/64448. Add focused regression tests for thrown retryable errors and nonzero retryable responses, plus non-retryable errors remaining terminal. Add a CHANGELOG.md fix entry if user-visible behavior changes.
