## Summary

Keep open. Current main still treats Feishu delivery rate-limit or flow-control failures as terminal across text/card, media, and initial streaming-card sends; the existing retryable/backoff classification is only wired to typing indicators, and the related Feishu pacing and duplicate-media reports track different remaining work.

## Plan

Fix https://github.com/openclaw/openclaw/issues/70879 by adding Feishu-plugin-owned bounded retry/backoff for outbound message delivery only. Read the Feishu/Lark docs or SDK source for retryable error/status shapes before hard-coding the classifier; cover explicit rate-limit/flow-control responses or errors such as HTTP 429 and known Feishu API rate-limit codes, but do not retry ambiguous post-send network failures. Apply the helper consistently to text/card create/reply sends, media post-upload message create/reply sends, and the initial streaming-card message send. Keep proactive pacing/config out of scope for https://github.com/openclaw/openclaw/issues/14206 and avoid broad media idempotency changes beyond what is needed to keep retries bounded and safe around https://github.com/openclaw/openclaw/issues/64448. Add focused regression tests for thrown SDK errors and nonzero SDK responses, update a concise CHANGELOG.md fix entry, and do not change core plugin APIs unless source inspection proves the Feishu plugin cannot own the helper locally.
