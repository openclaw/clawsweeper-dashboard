# Coding Plan for openclaw/openclaw#70879: Feishu plugin lacks retry logic for API rate limit errors

Source report: [records/openclaw-openclaw/items/70879.md](records/openclaw-openclaw/items/70879.md)

## Summary

Keep open. Current main still treats Feishu message delivery rate-limit responses as terminal send failures across text/card, media, and initial streaming-card sends; the existing backoff handling is limited to typing indicators, and the related pacing and duplicate-media issues track different work.

## Plan

Fix https://github.com/openclaw/openclaw/issues/70879 by adding Feishu-plugin-owned bounded retry/backoff for outbound message delivery rate-limit/flow-control failures. Scope the change to Feishu create/reply delivery sends in extensions/feishu: text/card sends, media post-upload sends, and initial streaming-card sends. Verify Feishu/Lark retryable response and thrown-error shapes against official docs or @larksuiteoapi/node-sdk source/types before choosing codes; include 429 and documented Feishu rate-limit/quota/flow-control codes where appropriate. Keep retries bounded with conservative exponential backoff/jitter, preserve existing withdrawn-reply fallback behavior, and avoid adding a core retry layer, public config option, or proactive pacing for https://github.com/openclaw/openclaw/issues/14206. Do not try to solve duplicate media idempotency from https://github.com/openclaw/openclaw/issues/64448, but make sure the retry design does not make that risk worse. Add focused Vitest regression coverage for retrying retryable send failures and not retrying non-retryable failures across send.ts/media.ts/streaming-card.ts, update nearby typing/shared classifier tests if factoring code, and add a concise CHANGELOG.md fix entry.
