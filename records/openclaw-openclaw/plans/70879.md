# Coding Plan for openclaw/openclaw#70879: Feishu plugin lacks retry logic for API rate limit errors

Source report: [records/openclaw-openclaw/items/70879.md](records/openclaw-openclaw/items/70879.md)

## Summary

Keep open. Current main and the latest shipped release still treat Feishu delivery-side rate-limit failures as terminal for text/card, media, and initial streaming-card sends; the only existing Feishu backoff classifier is wired to typing reactions, not outbound delivery.

## Plan

Fix Feishu outbound delivery retry for https://github.com/openclaw/openclaw/issues/70879. Add a Feishu-owned bounded retry/backoff helper for retryable delivery failures from message create/reply calls, including thrown SDK/Axios-style errors and nonzero Feishu response codes such as 11232, 99991400, and HTTP/code 429 after checking Feishu docs/SDK shapes. Apply it to text/card send paths in extensions/feishu/src/send.ts, media delivery sends in extensions/feishu/src/media.ts, and the initial streaming-card message send in extensions/feishu/src/streaming-card.ts as appropriate. Reuse or factor the existing typing backoff classification rather than duplicating stale code, but do not change core retry policy, add a public config option, add proactive pacing, or attempt to solve the separate duplicate-media idempotency issue at https://github.com/openclaw/openclaw/issues/64448 or the broader delivery-notification feature at https://github.com/openclaw/openclaw/issues/46830. Add focused regression tests for thrown retryable errors, nonzero retryable responses, terminal non-retryable responses, and bounded retry exhaustion; include a concise CHANGELOG.md Fixes entry because this is user-visible delivery behavior.
