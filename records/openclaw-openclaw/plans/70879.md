# Coding Plan for openclaw/openclaw#70879: Feishu plugin lacks retry logic for API rate limit errors

Source report: [records/openclaw-openclaw/items/70879.md](records/openclaw-openclaw/items/70879.md)

## Summary

Keep open. Current main and the latest shipped tag still treat Feishu message-delivery rate-limit failures as terminal across text/card, media, and initial streaming-card sends; the existing backoff classifier is scoped to typing indicators, and the related pacing/idempotency items track separate work.

## Plan

Fix https://github.com/openclaw/openclaw/issues/70879 by adding bounded Feishu-plugin retry/backoff for outbound delivery sends that fail with explicit retryable Feishu/Lark rate-limit or transient API errors. Keep the fix local to the Feishu plugin; do not add a core retry layer, public config option, or proactive pacing in this PR. Factor a small delivery retry helper or retryable classifier, apply it to text/card sendMessageFeishu/sendCardFeishu create/reply paths, media sendImageFeishu/sendFileFeishu create/reply paths, and the initial streaming-card message send. Preserve withdrawn-message reply fallback behavior and thread safety. Check the current Feishu/Lark docs/SDK error shapes before choosing retryable codes, include 2200, 11232, HTTP/SDK 429 where supported, and respect Retry-After when available. Retries must be bounded and idempotency-aware; do not introduce non-idempotent media retry behavior that worsens https://github.com/openclaw/openclaw/issues/64448. Add focused regression tests for thrown retryable SDK/Axios errors and nonzero retryable response codes succeeding on a later attempt, plus non-retryable failures remaining terminal. Add a single CHANGELOG.md fix entry if code changes land.
