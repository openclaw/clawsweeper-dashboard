# Coding Plan for openclaw/openclaw#70879: Feishu plugin lacks retry logic for API rate limit errors

Source report: [records/openclaw-openclaw/items/70879.md](records/openclaw-openclaw/items/70879.md)

## Summary

Keep open: current main still treats Feishu outbound rate-limit delivery failures as terminal on the main text/card, media, and initial streaming-card send paths, while the existing backoff classifier is scoped to typing reactions.

## Plan

Fix https://github.com/openclaw/openclaw/issues/70879 by adding Feishu-plugin-owned bounded retry/backoff for retryable outbound delivery failures. Keep the repair local to the Feishu plugin; do not add a core retry layer, a new public config option, or proactive pacing work. Factor a small retryable-error classifier from Feishu response objects and thrown SDK/Axios-style errors, apply the retry wrapper to message create/reply sends for text/cards, media sends, and the initial streaming-card send, and preserve withdrawn/not-found reply fallback semantics. Add focused regression tests that prove retryable nonzero responses and thrown retryable errors retry before success and that non-retryable errors still fail once. Keep proactive pacing in https://github.com/openclaw/openclaw/issues/14206, duplicate media idempotency in https://github.com/openclaw/openclaw/issues/64448, and user-facing failure notifications in https://github.com/openclaw/openclaw/issues/46830 unless the maintainer explicitly expands this fix.
