# Coding Plan for openclaw/openclaw#70879: Feishu plugin lacks retry logic for API rate limit errors

Source report: [records/openclaw-openclaw/items/70879.md](records/openclaw-openclaw/items/70879.md)

## Summary

Keep open. Current main still handles Feishu delivery sends as one-shot message create/reply calls for text/cards, media, reply fallback, and initial streaming cards; existing backoff classification is scoped to typing reactions and the related pacing and duplicate-media issues remain distinct rather than superseding this retry bug.

## Plan

Fix https://github.com/openclaw/openclaw/issues/70879 in the Feishu plugin only. Implement a bounded retry/backoff helper for outbound Feishu message delivery calls when the SDK throws or returns retryable rate-limit/flow-control errors; include HTTP/SDK 429 and existing Feishu backoff codes, and confirm any additional requested codes such as 2200 or 11232 against Feishu docs/source before adding them. Apply the helper consistently to sendMessageFeishu/sendCardFeishu reply and direct sends, reply fallback direct sends, image/file message delivery after upload, and the initial streaming-card message send. Reuse or extract typing backoff classification where it keeps behavior aligned. Do not add a core-wide retry layer or a public config option. Preserve withdrawn/not-found reply fallback behavior, keep retries bounded with jitter/backoff, and account for https://github.com/openclaw/openclaw/issues/64448 so retries do not amplify duplicate delivery. Add regression tests for both thrown SDK/Axios errors and non-zero Feishu response bodies, and add a user-facing changelog entry for the Feishu reliability fix.
