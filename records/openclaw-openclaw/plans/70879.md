# Coding Plan for openclaw/openclaw#70879: Feishu plugin lacks retry logic for API rate limit errors

Source report: [records/openclaw-openclaw/items/70879.md](records/openclaw-openclaw/items/70879.md)

## Summary

Keep open. Current main and v2026.5.20 still make Feishu text/card, media, and initial streaming-card delivery sends as one-shot calls; the existing rate-limit/backoff classifier is scoped to typing indicators, and there is no closing PR for this delivery retry gap.

## Plan

Fix Feishu outbound delivery rate-limit handling for https://github.com/openclaw/openclaw/issues/70879. Add a bounded Feishu-owned retry/backoff helper for retryable API send failures such as HTTP 429 and Feishu retryable response/error codes, and apply it to message create/reply delivery paths for text/card, media, and initial streaming-card sends as appropriate. Reuse or factor existing typing backoff classification where it fits, but keep the fix local to the Feishu plugin and avoid adding a new public config option or core retry layer. Preserve withdrawn/not-found reply fallback semantics and avoid retry behavior that could worsen duplicate media delivery tracked separately. Add focused regression tests for thrown SDK errors and nonzero Feishu response codes. Add a changelog entry because this changes user-visible Feishu delivery reliability.
