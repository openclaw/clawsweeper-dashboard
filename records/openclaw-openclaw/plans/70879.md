# Coding Plan for openclaw/openclaw#70879: Feishu plugin lacks retry logic for API rate limit errors

Source report: [records/openclaw-openclaw/items/70879.md](records/openclaw-openclaw/items/70879.md)

## Summary

Keep open. Current main and the latest shipped tag still make one terminal Feishu delivery attempt for text/card replies, media sends, and initial streaming-card sends, while existing backoff handling is limited to typing indicators and related open issues track proactive pacing, idempotency, or generic delivery-failure notification separately.

## Plan

Fix Feishu outbound delivery rate-limit retries for this issue. Add a Feishu-owned bounded retry/backoff helper for retryable delivery failures from message.create/message.reply responses and thrown SDK/Axios errors, using current Feishu docs/SDK types to classify codes such as 2200, 11232/11247-style flow-control failures, 99991400/99991403, and HTTP or SDK 429. Apply it consistently to text/card create/reply sends in extensions/feishu/src/send.ts, media message sends in extensions/feishu/src/media.ts, and initial streaming-card sends in extensions/feishu/src/streaming-card.ts where safe. Preserve withdrawn/not-found reply fallback behavior, do not add a new public config option, do not implement proactive pacing from https://github.com/openclaw/openclaw/issues/14206, and do not implement generic delivery-failure notification from https://github.com/openclaw/openclaw/issues/46830. Keep retries bounded, logged only as appropriate, and idempotency-aware; avoid adding non-idempotent media retries that worsen https://github.com/openclaw/openclaw/issues/64448. Add focused regression tests for thrown retryable errors, nonzero retryable responses, exhausted retries, successful retry, and non-retryable failures. Add a CHANGELOG.md fix entry because this is user-visible channel reliability work.
