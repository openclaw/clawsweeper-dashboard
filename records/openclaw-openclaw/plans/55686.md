# Coding Plan for openclaw/openclaw#55686: [Feature Request] Make command options clickable buttons in Feishu (e.g., /reasoning)

Source report: [records/openclaw-openclaw/items/55686.md](records/openclaw-openclaw/items/55686.md)

## Summary

Keep open: current main still returns command-option status replies such as `/reasoning` as plain `Options:` text, and Feishu's normal reply pipeline does not currently turn those options into native presentation/card buttons. Existing Feishu card support is useful infrastructure, while the adjacent callback-field work does not close this command-option UX gap.

## Plan

Fix https://github.com/openclaw/openclaw/issues/55686 with a narrow Feishu command-option button UX. Implement clickable buttons for known command-option/menu status replies such as `/reasoning` without parsing arbitrary agent-authored `Options:` text into executable actions. Prefer producing shared `presentation` button metadata from directive/menu helpers or another tightly validated internal command-option helper, then make Feishu's normal inbound reply pipeline render those buttons as native Feishu cards through the existing `MessagePresentation` and `ocf1` quick-action callback path. Keep readable text fallback for channels and contexts without buttons. Preserve Feishu streaming, thread reply, media, mention, and duplicate-final behavior. Coordinate with but do not absorb the separate callback-field work in https://github.com/openclaw/openclaw/issues/42754 and https://github.com/openclaw/openclaw/pull/42908. Add focused regression tests for `/reasoning`/directive options, Feishu rendering/callback routing, and fallback behavior; add a changelog entry because this is user-visible.
