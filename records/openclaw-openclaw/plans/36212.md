## Summary

Keep open. Current main still captures the provider message ID at enqueue, but the collect-mode drain and queued followup runner still do not propagate it into trusted per-item metadata or runtime currentMessageId.

## Plan

Fix https://github.com/openclaw/openclaw/issues/36212. Preserve the original provider message ID for queued followup messages in the shared queue path, not in Feishu-specific workaround code. Current main captures `FollowupRun.messageId` in `src/auto-reply/reply/get-reply-run.ts`, but collect drains in `src/auto-reply/reply/queue/drain.ts` drop it and `src/auto-reply/reply/followup-runner.ts` does not pass it to `runEmbeddedPiAgent.currentMessageId`. Add normalized and bounded trusted per-item message_id metadata to collect prompts, propagate an appropriate queued `messageId` on emitted followup runs, and forward queued `messageId` as `currentMessageId` for queued runtime execution. Preserve existing routing/auth batching behavior, avoid new config, avoid Feishu-only policy, and add regression tests for collect prompt metadata, blank/missing IDs, summary/individual drain behavior as applicable, and followup-runner currentMessageId forwarding. Be careful with multi-message collect batches: keep per-item IDs available for explicit replies and do not silently lose IDs; if an aggregate currentMessageId is used, make the rule explicit and covered by tests. Bound prompt-rendered IDs to avoid token bloat from oversized incoming IDs.
