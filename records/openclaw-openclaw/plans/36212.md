# Coding Plan for openclaw/openclaw#36212: Queued messages should preserve original platform message_id

## Summary

Keep open. Current main still captures the provider message ID when queuing a followup, but collect-mode drain and queued followup execution do not carry that ID into the collected prompt or runtime `currentMessageId`, and the targeted fix attempt was closed unmerged.

## Plan

Fix this issue by preserving original platform message IDs through queued followup execution. Current main captures `FollowupRun.messageId` in `src/auto-reply/reply/get-reply-run.ts`, but collect-mode synthetic runs in `src/auto-reply/reply/queue/drain.ts` drop it and `src/auto-reply/reply/followup-runner.ts` does not pass it to `runEmbeddedPiAgent.currentMessageId`. Implement the fix in the shared queue/followup path, not as a Feishu-specific workaround and without adding config. Normalize blank IDs away, preserve trusted IDs in collect prompt metadata if exposing them to the agent, and forward the resolved queued ID as runtime `currentMessageId`. Add regression coverage for collect prompt/messageId propagation and followup-runner `currentMessageId` forwarding; update `CHANGELOG.md` because this is user-visible.
