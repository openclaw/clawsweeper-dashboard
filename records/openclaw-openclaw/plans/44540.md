# Coding Plan for openclaw/openclaw#44540: Hybrid search BM25 component penalizes multimodal (image/audio) results

Source report: [records/openclaw-openclaw/items/44540.md](records/openclaw-openclaw/items/44540.md)

## Summary

Keep open: current main still applies the default hybrid weight penalty to vector-only results, and multimodal chunks do not carry modality through the search/merge contract, so the reported image/audio ranking failure remains plausible and source-reproducible.

## Plan

Fix the current-main memory-core bug where multimodal image/audio chunks indexed through memorySearch.extraPaths and Gemini Embedding 2 are penalized by the default hybrid BM25 weighting when they only have vector relevance. Carry an explicit modality/content kind from MemoryFileEntry/buildMultimodalChunkForIndexing through MemoryChunk, the chunks schema/search rows, and mergeHybridResults. For non-text multimodal chunks with vectorScore > 0 and no meaningful keyword score, do not treat missing BM25 as negative evidence; rank them by the vector signal while preserving existing text hybrid behavior and keyword-only fallback behavior. Add focused regression coverage in hybrid tests and memory index/manager tests using mocked Gemini structured inputs to show image/audio vector hits surface under default 0.7/0.3 hybrid weights when competing with text chunks. Do not add a config option, do not globally replace hybrid fusion, and do not weaken BM25 for normal text chunks. Include a changelog entry because this is user-visible memory_search behavior.
