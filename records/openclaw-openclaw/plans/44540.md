# Coding Plan for openclaw/openclaw#44540: Hybrid search BM25 component penalizes multimodal (image/audio) results

Source report: [records/openclaw-openclaw/items/44540.md](records/openclaw-openclaw/items/44540.md)

## Summary

Keep open. Current main still documents Gemini multimodal memory indexing and default hybrid BM25+vector search, but the built-in memory index drops the image/audio modality before search and the hybrid ranker still applies the same weighted BM25 formula to vector-only media hits.

## Plan

Fix https://github.com/openclaw/openclaw/issues/44540 by carrying explicit multimodal chunk modality through the built-in memory index and hybrid ranking path. Add an internal modality field to the memory chunk schema/contracts as needed, persist it from MemoryFileEntry during writeChunks, return it through searchVector/searchKeyword/mergeHybridResults, and make image/audio vector candidates use their vector score without the BM25 text-weight penalty while preserving existing BM25+vector scoring for text chunks. Include focused regression coverage for a multimodal vector-only candidate under default 0.7/0.3 hybrid weights and for normal text hybrid scoring. Avoid adding a new user config knob or changing global default weights. Add a user-facing changelog entry because this changes memory_search ranking behavior.
