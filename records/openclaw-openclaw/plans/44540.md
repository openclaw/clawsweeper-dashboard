# Coding Plan for openclaw/openclaw#44540: Hybrid search BM25 component penalizes multimodal (image/audio) results

Source report: [records/openclaw-openclaw/items/44540.md](records/openclaw-openclaw/items/44540.md)

## Summary

Keep open. Current main still knows image/audio modality before indexing, but the persisted chunks/search rows do not carry it into hybrid ranking; vector-only media hits get `textScore: 0` and are still weighted by the default hybrid formula, so the reported ranking bug remains source-reproducible.

## Plan

Fix https://github.com/openclaw/openclaw/issues/44540 by preserving multimodal image/audio modality through builtin memory indexing and hybrid ranking. Persist a modality/content-kind field in the memory index schema, write it from MemoryIndexEntry when chunks are stored, return it from vector/keyword search rows and MemorySearchResult plumbing, and update mergeHybridResults so non-text media vector hits are ranked by vector relevance instead of being diluted by missing BM25 textScore. Preserve current BM25+vector weighting for text chunks, preserve FTS-only behavior, and do not infer modality from file paths in the ranker. Handle existing indexed media rows with an explicit safe refresh/reindex treatment. Add focused regression tests showing image/audio vector hits can surface under default 0.7/0.3 hybrid weights, plus schema/search plumbing coverage. Include a user-facing changelog Fixes entry if this becomes a landed fix.
