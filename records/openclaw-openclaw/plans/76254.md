## Summary

Keep open. Current main and the latest release still return after `process.ppid` on non-Linux platforms, so macOS nested callers can still leave a gateway grandparent outside the stale-process exclusion set; the reporter's follow-up narrows this to a latent correctness bug while the live SIGTERM loop is tracked separately.

## Plan

Fix https://github.com/openclaw/openclaw/issues/76254 by extending `getSelfAndAncestorPidsSync()` so Darwin/BSD callers can exclude transitive ancestors, not only `process.ppid`. Keep the Linux `/proc/<pid>/status` path intact, do not change stale-cleanup call-site policy, keep the walk bounded by `MAX_ANCESTOR_WALK_DEPTH`, and use the existing `ps -p <pid> -o ppid=` precedent without adding dependencies. Add a focused regression test that stubs `process.platform` to `darwin`, stubs the direct parent to a plugin host, mocks parent lookup so a gateway grandparent is discovered, and verifies `findGatewayPidsOnPortSync()` excludes that grandparent while still returning an unrelated stale gateway PID. Include a changelog entry because this is user-visible macOS gateway restart correctness.
