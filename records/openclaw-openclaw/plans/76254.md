# Coding Plan for openclaw/openclaw#76254: macOS: getSelfAndAncestorPidsSync only walks /proc on Linux, leaving #68451 unfixed for grandchild callers

Source report: [records/openclaw-openclaw/items/76254.md](records/openclaw-openclaw/items/76254.md)

## Summary

Keep open. Current `main` and the latest `v2026.5.7` release still stop `getSelfAndAncestorPidsSync()` at `process.ppid` on non-Linux platforms, so a macOS nested caller can leave a gateway grandparent outside the stale-process exclusion set; the reporter's follow-up correctly narrows this to a latent correctness bug while https://github.com/openclaw/openclaw/issues/76259 owns the live SIGTERM loop.

## Plan

Fix https://github.com/openclaw/openclaw/issues/76254 by extending `getSelfAndAncestorPidsSync()` so non-Linux Unix platforms, especially macOS/Darwin, can walk beyond `process.ppid` when excluding caller ancestors from stale-gateway cleanup. Keep the existing Linux `/proc` path, avoid adding a runtime override hook, keep the walk bounded and cycle-safe, and use the existing `ps -p <pid> -o ppid=` precedent or a small shared helper rather than changing stale-cleanup call-site policy. Add focused tests in `src/infra/restart-stale-pids.test.ts` that stub Darwin, model `gateway -> plugin-host -> sidecar`, and prove the gateway grandparent is excluded while unrelated stale gateway PIDs still return. Preserve existing Windows and Linux coverage, include a user-facing changelog fix entry, and do not attempt to fix the separate kill-tree behavior tracked by https://github.com/openclaw/openclaw/issues/76259.
