## Plan

Fix https://github.com/openclaw/openclaw/issues/76831. The bug is that with `messages.tts.auto: "inbound"`, a WhatsApp text inbound followed by an audio inbound during an active default `messages.queue.mode=steer` run can produce a final reply that answers the audio prompt but is delivered as plain text because final TTS still sees the original text-originated dispatch context. Add a deterministic regression for text-then-audio steering into an active run, then carry the steered audio/reply-origin metadata forward as a sticky `inboundAudio` signal or explicit active reply-origin update so final TTS receives `inboundAudio=true` when any consumed steered inbound was audio. Keep this as a bug fix: do not add a new config option, do not change TTS provider selection, and preserve ordinary non-steered audio and text-only steering behavior.
