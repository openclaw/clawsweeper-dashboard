## Summary

Keep open: current main still appears to allow this stale-prefix npm update failure because npm target-root resolution only trusts the package-root-derived prefix when a prefix-local npm binary exists, and otherwise falls back to `npm root -g`.

## Plan

Fix the stale npm-prefix updater bug reported in https://github.com/openclaw/openclaw/issues/79087. Current main can resolve the invoked OpenClaw package root, but npm update target selection still falls back to `npm root -g` when the package-root-derived prefix lacks `<prefix>/bin/npm`, causing staging under a stale `/usr/lib/node_modules`. Implement a narrow fix so the package update flow can prefer the trusted invoked package root for standard npm global layouts like `<prefix>/lib/node_modules/openclaw`, while preserving the existing generic guard against inferring arbitrary npm ownership from path shape alone. Include dry-run/JSON diagnostics when the invoked package root and npm-reported global root disagree. Add focused regression tests for the user-prefix/stale-prefix case and keep pnpm/bun behavior unchanged. Do not add a new config option or move this into docs only.
