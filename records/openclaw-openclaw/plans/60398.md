# Coding Plan for openclaw/openclaw#60398: gateway install fails with error 5 when home directory is on external APFS volume

Source report: [records/openclaw-openclaw/items/60398.md](records/openclaw-openclaw/items/60398.md)

## Summary

Keep open. Current main and v2026.5.18 still derive the macOS LaunchAgent plist path from HOME and pass that same path to launchctl bootstrap, so an external-APFS home can still hit the reported error 5 path; the merged launchd hardening PR fixed adjacent stdio/log failures but not the external-home plist location.

## Plan

Fix https://github.com/openclaw/openclaw/issues/60398. Implement a shared macOS LaunchAgent plist path resolver that keeps the normal HOME/Library/LaunchAgents path for ordinary homes but chooses a launchd-safe boot-volume /Users/<shortname>/Library/LaunchAgents path when HOME is on an external APFS volume. Use the resolver consistently in launchd install/stage/start/restart, detached restart handoff, update restart helper, service audit/status inspection, and the macOS app LaunchAgent snapshot if that surface still reads the plist directly. Preserve custom OPENCLAW_LAUNCHD_LABEL and OPENCLAW_PROFILE behavior, do not add a config flag, and keep the related stdout/stderr/stdin fixes from https://github.com/openclaw/openclaw/pull/82844 intact. Add focused regression tests for normal HOME, external HOME, custom labels/profiles, restart/update fallback, and audit/status paths; include a changelog entry because this fixes user-visible macOS gateway install behavior.
