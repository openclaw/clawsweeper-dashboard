# Coding Plan for openclaw/openclaw#60398: gateway install fails with error 5 when home directory is on external APFS volume

Source report: [records/openclaw-openclaw/items/60398.md](records/openclaw-openclaw/items/60398.md)

## Summary

Keep open. Current main and the latest shipped release still resolve the macOS Gateway LaunchAgent plist under the effective HOME directory and pass that same path to launchctl bootstrap, so an external-APFS home can still hit the reported launchd error path; I found no merged or open PR that supersedes this issue.

## Plan

Fix the macOS gateway LaunchAgent external-HOME bug reported in https://github.com/openclaw/openclaw/issues/60398. Add a shared LaunchAgent plist-location resolver that keeps normal HOME behavior but, when the resolved HOME is on an external `/Volumes/...` style location, writes and bootstraps the plist from the launchd-safe boot-volume user LaunchAgents directory. Use that resolver everywhere the gateway plist path is read, written, bootstrapped, audited, restarted, or displayed: install/stage/uninstall/repair/restart, detached restart handoff, update restart helper, service audit/status paths, and the macOS app LaunchAgent snapshot if needed. Preserve custom labels and profiles, avoid changing service-env or log-path policy unless required, and treat https://github.com/openclaw/openclaw/issues/40207 as adjacent evidence rather than folding its StandardOutPath fix into this PR. Add focused regression tests for normal HOME and external-HOME path selection plus bootstrap callers, and include a changelog entry because this is user-visible gateway install behavior. Stop and ask for maintainer input if the implementation would require a new config flag or a broader macOS account-layout policy.
