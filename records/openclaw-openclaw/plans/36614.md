# Coding Plan for openclaw/openclaw#36614: per-channel-peer: updateLastRoute still contaminates main session with channel-specific delivery context

Source report: [records/openclaw-openclaw/items/36614.md](records/openclaw-openclaw/items/36614.md)

## Summary

Keep open: current main still has direct-message channel write paths that pass the shared main session into `updateLastRoute` while the documented `per-channel-peer` route contract isolates those DMs by channel and peer.

## Plan

Fix https://github.com/openclaw/openclaw/issues/36614. With `session.dmScope: per-channel-peer`, direct-message route resolution returns an isolated session key and `lastRoutePolicy: "session"`, but iMessage, Signal, LINE, Matrix, Mattermost, and Discord component paths still build `updateLastRoute` with `route.mainSessionKey`, contaminating `agent:<id>:main` with channel delivery context. Update the remaining direct-message write sites to use `resolveInboundLastRouteSessionKey({ route, sessionKey: <inbound session key> })` or an equivalent central contract, preserving `dmScope: main`, group/thread metadata, explicit main-session routes, and the pinned-main-owner guard only when the update target is the main session. Use Slack, Telegram, and WhatsApp as current examples. Do not implement the downstream message-tool fallback from https://github.com/openclaw/openclaw/pull/82969 as part of this upstream fix. Add focused regression coverage proving isolated DM sessions do not write channel delivery metadata to `agent:<id>:main`; add a user-facing changelog fix entry if the landing policy requires one.
