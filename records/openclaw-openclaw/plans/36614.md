## Summary

Keep this open: current main still has iMessage and several other inbound channel paths building direct-message `updateLastRoute` payloads with `mainSessionKey`, while `recordInboundSession` persists `deliveryContext` to that requested target. The related fix attempts are closed and unmerged, and the merged related PR only addressed the read-side WebChat stale-route leakage, not this write-side contamination.

## Plan

Fix https://github.com/openclaw/openclaw/issues/36614. Current main keeps `session.dmScope: per-channel-peer` route sessions isolated, but iMessage, Signal, LINE, Matrix, Mattermost, and Discord component/direct inbound `updateLastRoute` payloads still target `route.mainSessionKey`. Update the affected channel paths to use `resolveInboundLastRouteSessionKey({ route, sessionKey: ctxPayload.SessionKey ?? route.sessionKey })` or the equivalent shared policy so session-scoped DMs persist `deliveryContext` to the active route/thread session while shared-main DMs still write to `agent:main:main`. Preserve existing pinned-main-owner guards and do not change WebChat `chat.send` fallback behavior or session config semantics. Add focused regression coverage for iMessage and representative affected channels; update the changelog as a user-facing bug fix if the landing workflow expects one.
