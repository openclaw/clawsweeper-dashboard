# Coding Plan for openclaw/openclaw#64609: Telegram multi-account: group/topic systemPrompt silently ignored due to inconsistent config resolution

Source report: [records/openclaw-openclaw/items/64609.md](records/openclaw-openclaw/items/64609.md)

## Summary

Keep open. Current main and latest release v2026.5.7 still have the Telegram resolver mismatch: multi-account account config drops channel-level groups before prompt/skills/topic context is built, while mention policy still falls back to channel-level groups; the linked fix PR was closed without merging.

## Plan

Fix https://github.com/openclaw/openclaw/issues/64609. In Telegram multi-account setups, channel-level `channels.telegram.groups` prompt/skills/topic config is dropped because `mergeTelegramAccountConfig` intentionally does not inherit groups, while mention policy still falls back to channel-level groups. Implement a prompt/skills/topic-only fallback for Telegram group context resolution from account-scoped groups to channel-level groups, preserving the multi-account routing/access-control guard from https://github.com/openclaw/openclaw/issues/30673. Do not make `resolveTelegramAccount()` inherit channel-level groups wholesale in multi-account mode, and do not broaden `enabled`, `allowFrom`, `groupPolicy`, or delivery/routing semantics. Add focused regression coverage for multi-account channel-level group/topic `systemPrompt` and `skills`, including the path that feeds `GroupSystemPrompt`; cover native command prompt/skills behavior if the shared resolver is used there. Add a user-facing changelog entry if this becomes the fix PR.
