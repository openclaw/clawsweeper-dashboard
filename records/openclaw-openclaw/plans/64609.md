# Coding Plan for openclaw/openclaw#64609: Telegram multi-account: group/topic systemPrompt silently ignored due to inconsistent config resolution

Source report: [records/openclaw-openclaw/items/64609.md](records/openclaw-openclaw/items/64609.md)

## Summary

Keep open. Current main and latest release v2026.5.7 still have the Telegram multi-account resolver mismatch: prompt/skills/topic context only sees merged account groups, while mention policy can still fall back to channel-level groups; the linked fix PR was closed without merging.

## Plan

Fix https://github.com/openclaw/openclaw/issues/64609. In Telegram multi-account setups, channel-level `channels.telegram.groups` currently reaches `requireMention` via group-policy fallback but is dropped from prompt/skills/topic context because `resolveTelegramGroupConfig` reads only the merged account config. Add a scoped Telegram group prompt resolver that falls back from `channels.telegram.accounts.<accountId>.groups` to `channels.telegram.groups` for `systemPrompt`, `skills`, and topic prompt/skill resolution only. Do not reintroduce wholesale channel-level `groups` inheritance in `mergeTelegramAccountConfig`; keep the https://github.com/openclaw/openclaw/issues/30673 routing/access-control guard intact, and do not let channel-level `enabled`, `allowFrom`, or `groupPolicy` bypass account-scoped access semantics. Include focused regression coverage for two accounts plus channel-level group/topic `systemPrompt` and `skills`, including account-scoped groups taking precedence. Consider the closed reference implementation at https://github.com/openclaw/openclaw/pull/64650, but adapt to current main. Add a single user-facing `CHANGELOG.md` fix entry if the repair changes behavior.
