# Coding Plan for openclaw/openclaw#83863: ACP/Codex child tasks can be marked succeeded with progress-only output and no final deliverable

Source report: [records/openclaw-openclaw/items/83863.md](records/openclaw-openclaw/items/83863.md)

## Summary

Keep open. Current main partially handles structured `startTurn` failures, but the remaining source paths still allow a completed ACP turn or subagent ok outcome to carry only progress text into successful task state while required completion delivery can fail separately.

## Plan

Fix https://github.com/openclaw/openclaw/issues/83863 by making ACP/Codex child tasks that expect a final completion artifact stop reporting plain success when the only captured assistant output is interim/progress text or required completion delivery exhausts retries. Keep the fix bounded to ACP/subagent/task finalization contracts: prefer structured final-output, runtime result, tool/final artifact, or delivery-state facts over broad text-prefix heuristics; reuse existing blocked terminal outcomes; do not add a new config option. Preserve valid short text-only final answers. Add regressions for legacy ACP progress-only done, startTurn completed with missing final artifact if the contract can identify it, progress-only output after file/tool mutation, required completion delivery failure, and a real final report that remains succeeded. Review closed PR https://github.com/openclaw/openclaw/pull/79567 as a rejected heuristic direction, not as source to copy. Check related issues https://github.com/openclaw/openclaw/issues/44925, https://github.com/openclaw/openclaw/issues/52249, and https://github.com/openclaw/openclaw/issues/83836 only to avoid conflating broader delivery, wake, or encrypted-content replay failures. Add a changelog Fixes entry if the repair changes user-visible task/completion behavior. Do not change provider defaults, channel routing, task retention, or unrelated task registry semantics.
