# Coding Plan for openclaw/openclaw#80970: openclaw sessions cleanup crashes: validateSessionId rejects agent:<id>:main session key (regex excludes ':')

Source report: [records/openclaw-openclaw/items/80970.md](records/openclaw-openclaw/items/80970.md)

## Summary

Keep open: current main still lets `sessions cleanup --fix-missing` feed a key-shaped `sessionId` such as `agent:main:main` into strict transcript-id validation, so malformed legacy store rows can crash cleanup before they can be pruned or reported.

## Plan

Fix https://github.com/openclaw/openclaw/issues/80970. `openclaw sessions cleanup --fix-missing` currently calls `resolveSessionFilePath(entry.sessionId, entry, ...)`; legacy or malformed rows can have `sessionId` equal to a session-store key such as `agent:main:main`, which makes strict transcript-id validation throw and abort cleanup. Preserve strict transcript filename validation in `src/config/sessions/paths.ts`; do not broadly allow colons in transcript session ids. Update the cleanup path, likely `src/config/sessions/cleanup-service.ts`, so malformed/key-shaped stored `sessionId` values are handled as cleanup data errors: if a valid persisted `sessionFile` can be safely used, keep/report it; otherwise `--fix-missing` should mark/prune or otherwise report the row without crashing. Add regression coverage for dry-run and applied cleanup with a row like `"agent:main:main": { "sessionId": "agent:main:main", "updatedAt": ... }`, plus a control case where `agent:main:main` is only the store key and the transcript id is safe. Also consider the related `agent:main:explicit:agent:main:main` shape mentioned in https://github.com/openclaw/openclaw/issues/77011. Coordinate with https://github.com/openclaw/openclaw/issues/50248 and https://github.com/openclaw/openclaw/pull/69954 so the stale-sessionFile canonical fallback and invalid-sessionId handling do not conflict. Add a user-facing changelog entry for the cleanup crash fix; do not take on broader subagent retention semantics from https://github.com/openclaw/openclaw/issues/47975.
