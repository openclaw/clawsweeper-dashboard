# Coding Plan for openclaw/openclaw#38580: Feature request: Read pinned Telegram messages via bot API

Source report: [records/openclaw-openclaw/items/38580.md](records/openclaw-openclaw/items/38580.md)

## Summary

Keep open: current main still lacks a Telegram pinned-message read action. The generic `list-pins` action exists and Telegram has write-side pin helpers, but the Telegram plugin does not advertise, gate, or route a read backed by `getChat`; the two related implementation PRs are closed unmerged.

## Plan

Implement the narrow on-demand Telegram pinned-message read requested in https://github.com/openclaw/openclaw/issues/38580. Add Telegram plugin-owned `list-pins` / `listPins` message-action support backed by grammY/Bot API `getChat`, returning an empty pins array when there is no current pinned message and a single current pinned message when Telegram exposes `pinned_message`. Add a Telegram `actions.pins` gate consistently in config types, zod schema, discovery, runtime checks, docs, and tests. Reuse existing target parsing/chat resolution patterns and keep this in the Telegram plugin boundary; do not add automatic session-start context injection and do not implement the separate pin/unpin write-action request from https://github.com/openclaw/openclaw/issues/30518. Use the closed unmerged attempts https://github.com/openclaw/openclaw/pull/38613 and https://github.com/openclaw/openclaw/pull/38718 only as context, not as a direct patch. Add focused regression tests and a user-facing changelog entry because this is a new message action. Validation should include targeted Vitest through `node scripts/run-vitest.mjs`, oxfmt check for touched files, and redacted live Telegram proof or Telegram E2E proof showing `list-pins` returns the current pinned message without exposing chat IDs, tokens, phone numbers, or private endpoints.
