# Coding Plan for openclaw/openclaw#38580: Feature request: Read pinned Telegram messages via bot API

Source report: [records/openclaw-openclaw/items/38580.md](records/openclaw-openclaw/items/38580.md)

## Summary

Keep open: current main still lacks a Telegram pinned-message read action. Telegram has generic `list-pins` action vocabulary and write-side pin helpers, but the Telegram plugin does not advertise, gate, or route a pinned-message read backed by `getChat`, and the two related implementation attempts are closed unmerged.

## Plan

Implement on-demand Telegram pinned-message reads for https://github.com/openclaw/openclaw/issues/38580. Add Telegram plugin-owned support for the existing `list-pins` message action, backed by Telegram Bot API `getChat` and the current `pinned_message` field; return an empty `pins` array when no pinned message exists and one normalized pin when Telegram returns one. Keep the change in the Telegram plugin/action surfaces: `extensions/telegram/src/channel-actions.ts`, `extensions/telegram/src/action-runtime.ts`, `extensions/telegram/src/send.ts`, Telegram config schema/types, focused tests, docs, and `CHANGELOG.md`. Add a consistent `channels.telegram.actions.pins` gate, defaulting in line with other enabled read/write action gates unless codebase patterns clearly require a different default. Do not add automatic session-start context injection, scheduled refresh, global core policy, or a new core action name. Reuse existing Telegram account/token/target resolution and request diagnostics; prefer grammY API primitives and existing helper seams over a custom raw HTTP wrapper. Preserve plugin/core boundary rules and do not import core internals from extension production code. Validation should include targeted Vitest files, oxfmt check for touched files, and redacted real Telegram proof showing `list-pins` returns the current pinned message in a real bot/chat.
