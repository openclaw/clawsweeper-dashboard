# Coding Plan for openclaw/openclaw#69443: [Bug] Subagent RPC callback to WeChat session key routed to main session instead

Source report: [records/openclaw-openclaw/items/69443.md](records/openclaw-openclaw/items/69443.md)

## Summary

Keep open. Current main and the latest release still choose the Gateway `agent` run session from `sessionKey`, `sessionId`, or the default agent session before `to` is treated as delivery metadata, so a canonical session key supplied only in `to` can still fall back to the main session.

## Plan

Fix the Gateway `agent` RPC routing bug reported at https://github.com/openclaw/openclaw/issues/69443. A request with `to` set to a canonical `agent:<id>:...` session key and no `sessionKey` or `sessionId` must not silently run in the agent main session. Implement this generically in the Gateway/session resolution boundary: either promote canonical session-key-shaped `to` values to the explicit requested session only when `sessionKey` and `sessionId` are absent, or reject that shape with a clear validation error. Preserve existing precedence for explicit `sessionKey` and `sessionId`, preserve ordinary delivery-target `to` semantics, and do not add WeChat-specific core logic. Add focused regression coverage for a session-key-shaped `to` value in the Gateway handler and, if the shared command resolver is changed, in the command session resolver tests. Add a user-facing changelog fix entry if behavior changes.
