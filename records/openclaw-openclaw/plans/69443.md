# Coding Plan for openclaw/openclaw#69443: [Bug] Subagent RPC callback to WeChat session key routed to main session instead

Source report: [records/openclaw-openclaw/items/69443.md](records/openclaw-openclaw/items/69443.md)

## Summary

Keep open. Current main and the latest release still choose the Gateway agent run session from `sessionKey`, `sessionId`, or the agent default before `to` is treated as delivery metadata, so a canonical `agent:<id>:...` value supplied only in `to` can still fall back to the main session.

## Plan

Fix the Gateway agent RPC/session resolver bug where a canonical session key such as `agent:main:openclaw-weixin:direct:peer@im.wechat` supplied only in `to` can silently route the run to `agent:main:main`. Preserve explicit `sessionKey` and `sessionId` precedence. Add generic handling for canonical `agent:<id>:...` values in `to` when no explicit selector exists, or reject that shape clearly; do not add WeChat-specific core logic and do not change ordinary E.164/direct delivery targets. Add regression coverage for the Gateway handler and CLI/session resolver, plus a user-facing changelog fix entry if behavior changes.
