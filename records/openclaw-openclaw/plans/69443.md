## Summary

Keep open. Current main still selects the Gateway `agent` run session from `sessionKey`, `sessionId`, or the default agent session before treating `to` as delivery metadata, so a canonical session key supplied only in `to` is not honored as the target session.

## Plan

Fix https://github.com/openclaw/openclaw/issues/69443. In the Gateway `agent` method, a request whose `to` is a canonical `agent:<id>:...` session key and whose `sessionKey` and `sessionId` are absent currently selects the default agent session before treating `to` as delivery metadata. Implement a generic fix in the Gateway/session resolver so this shape either targets the canonical session key explicitly or is rejected with a clear validation error; preserve precedence for explicit `sessionKey` and `sessionId`, preserve normal delivery-only `to` values, and do not add WeChat-specific logic to core.
