## Summary

Keep open. Current main and the latest release still select the Gateway agent run session from `sessionKey`, `sessionId`, or the agent default before `to` is treated as delivery metadata, so a canonical `agent:<id>:...` value supplied only in `to` can still fall back to the main session.

## Plan

Fix https://github.com/openclaw/openclaw/issues/69443. Preserve explicit `sessionKey` and `sessionId` precedence, reject mismatched agent ids, avoid WeChat-specific core logic, and do not reinterpret provider-native `to` strings. Add focused regression coverage for the Gateway handler and CLI/session resolver behavior; include a changelog entry if user-visible behavior changes.
