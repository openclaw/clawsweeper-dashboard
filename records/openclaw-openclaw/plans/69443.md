## Summary

Keep open. Current main and v2026.5.18 still select the Gateway agent run session from `sessionKey`, `sessionId`, or an agent-main default before `to` is used as delivery metadata, so a canonical `agent:<id>:...` value supplied only in `to` can still miss the intended WeChat-backed session. The adjacent open work for https://github.com/openclaw/openclaw/issues/41483 targets explicit `--agent --channel --to` per-recipient derivation and does not close this raw Gateway `to`-as-session-key path.

## Plan

Fix https://github.com/openclaw/openclaw/issues/69443. Current main selects the Gateway agent run session before treating `params.to` as delivery metadata, so a request with no explicit `sessionKey` or `sessionId` and `to` equal to a canonical `agent:<id>:...` session key can run in the default/main session instead of the intended channel session. Add a generic resolver guard in the Gateway agent path, and shared CLI/session helpers if needed: preserve explicit `sessionKey` and `sessionId` precedence; when `to` is a well-formed canonical agent session key and no explicit selector exists, select/canonicalize that session or fail with a clear invalid-request error; do not reinterpret ordinary delivery targets; do not add WeChat-specific core logic; coordinate with the adjacent open work at https://github.com/openclaw/openclaw/pull/73403 without broadening into raw channel-recipient key derivation.
