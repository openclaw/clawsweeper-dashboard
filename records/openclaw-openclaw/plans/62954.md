## Summary

Keep open: current main still has the auto-reply heartbeat stale-display path, because heartbeat turns can persist their runtime provider/model into the shared session entry that UI/history/status resolvers prefer.

## Plan

Fix the auto-reply heartbeat runtime-model bleed reported here and in https://github.com/openclaw/openclaw/issues/74217. Current main preserves heartbeat runtime model only in the agent-command session-store path; the auto-reply path still passes `modelUsed`/`providerUsed` from heartbeat runs into `persistRunSessionUsage` and `persistSessionUsageUpdate`, which then writes `entry.modelProvider` and `entry.model`. Thread an `isHeartbeat` or `preserveRuntimeModel` flag through `src/auto-reply/reply/session-run-accounting.ts` into `src/auto-reply/reply/session-usage.ts`, pass it from both `runReplyAgent` and `createFollowupRunner`, and for heartbeat turns preserve the existing persisted runtime model/provider/context fields in both the usage and model/context-only branches. Keep normal non-heartbeat persistence behavior unchanged, keep cost snapshot semantics from https://github.com/openclaw/openclaw/issues/69347 unchanged, and do not change CLI session binding or heartbeat cost attribution without maintainer direction. Add focused regression coverage for heartbeat persistence in `src/auto-reply/reply/session.test.ts` and caller coverage proving the heartbeat flag is forwarded from agent-runner and followup-runner. Add a single user-facing changelog fix entry crediting the human reporter if appropriate.
