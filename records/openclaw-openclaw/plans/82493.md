# Coding Plan for openclaw/openclaw#82493: Subagent completion delivered as Interactive Card invisible to non-Claude models

Source report: [records/openclaw-openclaw/items/82493.md](records/openclaw-openclaw/items/82493.md)

## Summary

Keep open: the ordinary subagent announce path already carries text, but current rich source-reply handling can preserve presentation/interactive/channelData-only payloads while the durable transcript/model-visible mirror only records text or media names. That leaves a plausible current-main gap for non-Claude coordinators when a completion is represented as a rich card rather than plain text.

## Plan

Fix the reported non-Claude subagent completion visibility bug without changing model-specific policy. Current main preserves rich source replies (`presentation`, `interactive`, `channelData`) as content, but `src/agents/pi-embedded-runner/run/payloads.ts` only mirrors `payload.text`/`mediaUrls` into `sourceReplyTranscriptMirror`, and `src/config/sessions/transcript.ts` drops mirrors with no resolved text. Add a model-agnostic plain-text fallback for rich source replies/subagent completion handoff, preferably reusing existing `renderMessagePresentationFallbackText`, `resolveInteractiveTextFallback`, or a small shared helper. Preserve rich card delivery for UI/channel rendering, do not make WebChat/TUI generic outbound channels, do not add config or provider/model branching, and do not conflate this with Feishu raw-card readback tracked separately at https://github.com/openclaw/openclaw/issues/78289. Add focused regression coverage for a `message_tool_only` rich payload with no `text` proving chat.history/subagent announce can recover readable completion text, plus coverage that normal text/media behavior remains unchanged. Add a changelog fix entry because this is user-visible.
