## Summary

Keep this PR open: it addresses a real approval UX bug and has credible terminal proof, but the patch introduces a double-summarization path for native approval delivery that can distort the hidden-command metadata shown to operators.

## Plan

Repair PR https://github.com/openclaw/openclaw/pull/80141. The current patch summarizes exec approval command text in both `buildPendingApprovalView` and `buildExecApprovalPendingReplyPayload`, so Telegram/Matrix native paths that pass `view.commandText` into the shared payload builder can re-summarize the truncation marker and misreport hidden content. Make summarization single-pass or idempotent while preserving the full sanitized pending approval record, approval matching, `systemRunPlan`, bindings, and allowed-decision semantics. Add focused regression coverage for the native payload path, especially Matrix and Telegram, showing that an eight-line escaped-newline command reports the original hidden-line/char counts once. Do not broaden exec approval policy, persistence, or config behavior; the broader Allow Always persistence cluster is separate.
