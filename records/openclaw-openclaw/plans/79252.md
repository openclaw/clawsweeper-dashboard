## Summary

Keep open: current main and v2026.6.10 still compare the global breaker against the current tool/args no-progress streak, so cross-tool no-progress loops can still evade the documented global breaker. Earlier same-problem PRs are closed unmerged, and the currently open related work tracks adjacent loop-detection problems rather than this exact session-global counter bug.

## Plan

Fix https://github.com/openclaw/openclaw/issues/79252 by making `globalCircuitBreakerThreshold` aggregate stable no-progress outcomes across the current run/session history instead of only the candidate tool name and args hash. Keep the fix in the loop detector and before-tool-call boundary; do not add a new config option, do not change default thresholds, and do not fold in the separate consecutive-error-cascade feature from https://github.com/openclaw/openclaw/issues/75923 or the exec volatility issue from https://github.com/openclaw/openclaw/issues/93917. Add regression coverage proving that after the threshold is reached across repeated stable no-progress outcomes, a different next tool is blocked by `global_circuit_breaker`, while unrelated one-off or changing-progress calls do not trip it. Changelog is release-owned; do not edit `CHANGELOG.md` for this normal fix.
