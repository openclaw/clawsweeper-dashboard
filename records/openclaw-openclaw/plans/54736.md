# Coding Plan for openclaw/openclaw#54736: isSilentReplyText regex fails on doubled NO_REPLY tokens

Source report: [records/openclaw-openclaw/items/54736.md](records/openclaw-openclaw/items/54736.md)

## Summary

Keep open. Current `main` and `v2026.5.19` still use exact-only silent-token matching, so `NO_REPLY\n\nNO_REPLY` misses shared suppression and can be normalized into a visible `NO_REPLY` before channel delivery.

## Plan

Fix https://github.com/openclaw/openclaw/issues/54736 by updating the shared silent-reply classification/normalization path so payloads made only of repeated whitespace-separated silent tokens, such as `NO_REPLY\n\nNO_REPLY`, are treated as silent and not delivered. Keep the fix in the shared auto-reply token predicate or normalization contract, not in a channel plugin. Preserve the earlier https://github.com/openclaw/openclaw/issues/19537 behavior: substantive replies that merely contain or end with `NO_REPLY` must still deliver after safe token stripping, not be fully suppressed. Add focused regression coverage for repeated `NO_REPLY`, mixed-case repeated tokens, and a custom token such as `HEARTBEAT_OK`; include normalization/runtime delivery coverage if the predicate change alone does not prove the leak path. Coordinate with but do not absorb the broader reasoning-preface work in https://github.com/openclaw/openclaw/pull/73153 unless maintainers explicitly ask; https://github.com/openclaw/openclaw/pull/83996 is closed unmerged and can be used only as prior art. Do not add a new config option or channel-specific workaround.
