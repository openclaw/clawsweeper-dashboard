## Summary

Keep open. Current main and v2026.5.20 still use exact-only silent-token matching, so a doubled token-only payload falls through shared suppression and can leave a visible `NO_REPLY` after normalization; related PRs are closed or still open/unmerged, so the core bug is not implemented on main.

## Plan

Fix the repeated silent-token-only leak from https://github.com/openclaw/openclaw/issues/54736. In the shared auto-reply token classifier or normalization path, treat payloads made only of one or more whitespace-separated silent tokens as silent, including custom tokens, while preserving the earlier false-positive boundary where substantive replies ending with `NO_REPLY` are delivered after stripping rather than fully suppressed. Keep the change narrow to repeated token-only payloads unless maintainers explicitly decide to include reasoning-prefaced or markdown-wrapped variants. Coordinate with https://github.com/openclaw/openclaw/pull/73153, https://github.com/openclaw/openclaw/pull/80930, and the closed unmerged https://github.com/openclaw/openclaw/pull/83996 to avoid conflicting semantics. Likely files are `src/auto-reply/tokens.ts`, `src/auto-reply/tokens.test.ts`, and possibly `src/auto-reply/reply/normalize-reply.ts` plus adjacent delivery tests. Add focused regression tests for `NO_REPLY\n\nNO_REPLY`, mixed-case repeated tokens, custom token repeated payloads, and preservation of substantive text ending in `NO_REPLY`. Add a changelog entry only if maintainers treat this as user-facing during landing.
