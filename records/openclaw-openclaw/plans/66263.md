## Summary

Keep open: current main still has the Codex native compaction accounting gap, because a successful native compaction can return no `tokensAfter`, the session store loses the explicit stale/unknown marker, and transcript fallback can still re-promote older usage as fresh.

## Plan

Fix the Codex native compaction accounting bug described at https://github.com/openclaw/openclaw/issues/66263. On current main, native Codex compaction can complete without a trustworthy `tokensAfter`; `sessions.compact` then deletes freshness state, and transcript fallback can later reuse pre-compaction usage as fresh current context.

Boundary:
- Keep the fix focused on Codex native compaction/session accounting and status/session fallback behavior.
- Capture a finite app-server post-compaction token count as `tokensAfter` if the native notification/result payload provides it.
- If no reliable `tokensAfter` is available, persist an explicit unknown/stale post-compaction state instead of deleting the freshness marker into ambiguity.
- Add a compaction marker or equivalent boundary so transcript-derived usage does not cross native compaction and resurrect pre-compaction totals as fresh.
- Do not add a config flag or product policy change.
- Coordinate with open adjacent work at https://github.com/openclaw/openclaw/pull/69270, https://github.com/openclaw/openclaw/pull/72285, and https://github.com/openclaw/openclaw/pull/82027 before duplicating broader compaction/status changes.
