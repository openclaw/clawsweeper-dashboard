## Summary

Keep open. Current main still has generic plugin config validation paths that can emit the reported missing-required-field error when a required plugin config is observed as missing, and there is still no UnraidClaw source or focused regression proving valid plugin config survives startup/catalog reload phases without repeated invalid-config logs.

## Plan

Fix the valid-plugin-config startup/catalog spam reported in https://github.com/openclaw/openclaw/issues/63234. Keep the scope generic: preserve resolved `plugins.entries.<id>.config` for required native/tool plugins across gateway startup, active registry reuse, `tools.catalog`, and post-ready plugin tool resolution. Add focused regression coverage with a synthetic required-config plugin that would fail if the config is later observed as `{}` or if repeated `[plugins] <id> invalid config` logs are emitted after a successful registration. Do not add an UnraidClaw special case, do not add a config option, and do not weaken the current fail-closed contract for genuinely invalid plugin config. Add a single-line changelog Fixes entry if the behavior change is user-visible.
