# Coding Plan for openclaw/openclaw#29707: [Feature Request]: Native Prompt Caching Support for Amazon Nova (Bedrock)

Source report: [records/openclaw-openclaw/items/29707.md](records/openclaw-openclaw/items/29707.md)

## Summary

Keep open: current main still treats direct Amazon Nova Bedrock model IDs as non-cacheable, while current AWS Bedrock docs document prompt caching support for Nova and recommend explicit prompt caching for cost savings and more consistent performance. This is valid provider work, not conservative cleanup to close.

## Plan

Fix https://github.com/openclaw/openclaw/issues/29707 by adding native Amazon Nova prompt-caching support in the bundled Amazon Bedrock provider. Keep the fix provider-owned: recognize AWS-supported Nova direct, regional, geo, and global IDs such as amazon.nova-2-lite-v1:0, amazon.nova-pro-v1:0, amazon.nova-lite-v1:0, amazon.nova-micro-v1:0, us./eu. prefixed IDs, and global.amazon.nova-2-lite-v1:0 where AWS documents support. Preserve the non-Anthropic no-cache path for unsupported Bedrock models. Do not rely on pi-ai cacheRetention pass-through alone because pi-ai 0.73.1 only injects cachePoint for selected Claude IDs; add Nova-safe payload shaping in the Amazon Bedrock plugin or a shared helper. Avoid mapping cacheRetention: "long" to ttl: "1h" for Nova unless current AWS docs or live proof confirm it; current AWS docs show 5-minute TTL for Nova model cards. Update regression tests, docs/reference/prompt-caching.md, docs/providers/bedrock.md if needed, and CHANGELOG.md. Stop and report back if implementation would require a new public plugin SDK seam or a root dependency change.
