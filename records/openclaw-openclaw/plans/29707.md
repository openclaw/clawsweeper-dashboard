# Coding Plan for openclaw/openclaw#29707: [Feature Request]: Native Prompt Caching Support for Amazon Nova (Bedrock)

Source report: [records/openclaw-openclaw/items/29707.md](records/openclaw-openclaw/items/29707.md)

## Summary

Keep open: current main and the latest shipped release still exclude direct Amazon Nova Bedrock model IDs from cache-retention eligibility and force them through `cacheRetention: "none"`. The earlier linked Nova `cachePoint` failure in https://github.com/openclaw/openclaw/issues/20815 and workaround PR https://github.com/openclaw/openclaw/pull/20866 explain why the guard exists, but this feature request now needs a provider-owned Nova prompt-caching implementation rather than cleanup closure.

## Plan

Implement native prompt-caching support for Amazon Nova models in the bundled Amazon Bedrock provider. Preserve explicit `cacheRetention` for supported direct, region-prefixed, and global `amazon.nova-*` Bedrock model IDs; inject only AWS/Nova-compatible `cachePoint` checkpoints into the Converse payload; keep unsupported non-Anthropic/non-Nova Bedrock models on the current no-cache path; and do not reuse the Claude `ttl: "1h"` mapping for Nova unless the AWS contract or a live request proves it is valid. Update focused tests for retention eligibility, provider wrapper behavior, payload shaping, and config defaults as appropriate, then update public prompt-caching/Bedrock docs and add a user-facing changelog entry. Do not move provider-specific Nova policy into core except for the existing shared eligibility seam needed by the runtime.
