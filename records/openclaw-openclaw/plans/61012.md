## Summary

Keep open: current main and v2026.5.18 still route token-gated Telegram message actions through the raw token resolver, so omitted accountId does not honor the configured Telegram default account despite the existing documented resolver semantics.

## Plan

Fix the Telegram default-account routing bug from https://github.com/openclaw/openclaw/issues/61012. Current main still resolves token-gated actions in extensions/telegram/src/action-runtime.ts with resolveTelegramToken(cfg, { accountId }), so omitted accountId bypasses channels.telegram.defaultAccount and can fail or use the wrong implicit default token. Route sendMessage, poll, react, deleteMessage, editMessage, sendSticker, createForumTopic, and editForumTopic token lookup through resolveTelegramAccount while preserving explicit unknown-account fail-closed behavior from extensions/telegram/src/token.ts. Add/update regression coverage in extensions/telegram/src/action-runtime.test.ts for omitted accountId plus channels.telegram.defaultAccount using a named account token and no top-level Telegram token. Use the closed PRs as prior art, but keep the final patch owner-bounded and do not fold in duplicate-token validation or broader warning/product changes unless the code path requires it.
