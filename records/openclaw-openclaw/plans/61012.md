The prior discussion usefully narrowed this from a client-cache claim to the omitted-account `openclaw message send` / action-runtime bug.

## Plan

Fix https://github.com/openclaw/openclaw/issues/61012 narrowly in the Telegram plugin. In `extensions/telegram/src/action-runtime.ts`, stop resolving token-gated message actions with raw `resolveTelegramToken(cfg, { accountId })` when `accountId` is omitted; use the existing `resolveTelegramAccount` semantics so `channels.telegram.defaultAccount` and configured account tokens are honored. Preserve explicit account behavior, especially unknown explicit account IDs in multi-account configs refusing channel-level fallback. Add focused regression coverage in `extensions/telegram/src/action-runtime.test.ts` for omitted `accountId` plus named `channels.telegram.defaultAccount` across token-gated actions, and keep existing `accounts.ts`, `token.ts`, `send.ts`, and `src/commands/message.ts` behavior green. Add a user-facing changelog fix entry if the repair changes runtime behavior.
