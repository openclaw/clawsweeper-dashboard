# Coding Plan for openclaw/openclaw#63920: Expose Slack message_action and global shortcut events via channels.slack

Source report: [records/openclaw-openclaw/items/63920.md](records/openclaw-openclaw/items/63920.md)

## Summary

Keep this open: current main and the latest release still wire Slack block actions and modal lifecycle events, but not Bolt shortcut events, so registered Slack global and message shortcuts still have no OpenClaw system-event path. The request fits the Slack plugin boundary and is narrow, with the main implementation constraints being immediate Slack ACK and a clear no-channel routing choice for global shortcuts.

## Plan

Implement Slack global shortcut and `message_action` shortcut routing for this issue. Keep the work inside the Slack plugin: add a focused `registerSlackShortcutHandler` path under `extensions/slack/src/monitor/events/`, wire it from `registerSlackInteractionEvents`, call `ack()` before any routing or agent work, preserve app/team mismatch dropping and Slack interaction auth behavior, summarize/sanitize payloads with the existing interaction formatter, and enqueue a compact `Slack interaction: ...` or clearly parallel shortcut system event with stable `sessionKey`, `contextKey`, and delivery context where available. Message shortcuts should include channel/message/user context; global shortcuts should use a deliberate user-scoped or existing canonical no-channel route and test that choice. Add focused Vitest coverage for global shortcut delivery, message shortcut delivery, no `ctx.app.shortcut` support, mismatch drops, ACK ordering, and session/context keys. Update `docs/channels/slack.md` to list shortcut routing. Do not add a new config option, broad plugin shortcut registration API, full modal workflow API, or unrelated Slack message actions in this PR.
