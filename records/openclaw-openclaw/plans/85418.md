## Summary

Current main and the latest shipped release still use a 2-second synchronous Chrome `--version` probe and collapse timeout/null results into the reported doctor warning. No exact duplicate or open fixing PR was found, so this should stay open as a narrow queueable browser-plugin doctor bug.

## Plan

Fix the Chrome MCP doctor false warning reported in https://github.com/openclaw/openclaw/issues/85418. In `extensions/browser/src/browser/chrome.executables.ts`, avoid cold-spawning macOS Google Chrome for version detection when possible, for example by deriving the `.app` bundle from known Google Chrome/Canary executable paths and reading `Contents/Info.plist` `CFBundleShortVersionString` with a bounded system helper before falling back to `--version`; also make the fallback tolerate realistic macOS cold-cache latency. Preserve the existing `readBrowserVersion(executablePath): string | null` SDK-facing signature unless a richer internal-only diagnostic is clearly needed, and do not add a config option or move browser ownership into core. Add focused tests in the browser helper and doctor readiness tests for macOS plist-first behavior, slow/unreadable version fallback behavior, and the existing missing/too-old/detected paths. Update docs only if the user-facing doctor wording changes; add a changelog entry during landing if the repair is user-visible.
