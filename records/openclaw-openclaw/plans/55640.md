# Coding Plan for openclaw/openclaw#55640: Bug: messaging tool suppress mechanism skips when tool fails then retries in same turn, causing duplicate delivery

Source report: [records/openclaw-openclaw/items/55640.md](records/openclaw-openclaw/items/55640.md)

## Summary

Keep open. Current main still has the same failed-send/retry gap: failed message-tool sends discard pending evidence, in-flight retry evidence is not consulted by message_end or final payload dedupe, and the related closed/open items cover different messaging-tool paths.

## Plan

Fix https://github.com/openclaw/openclaw/issues/55640. Current main discards failed message-tool pending evidence and only dedupes message_end/final payloads against committed successful sends, so a failed send followed by an in-flight retry with the same route/content can deliver both the retry tool send and the assistant text. Implement a narrow lifecycle fix in the embedded messaging-tool bookkeeping/dedupe path: suppress duplicate same-route/same-content message_end/final delivery while a retry send is in flight, but preserve fallback delivery if every send attempt fails. Do not globally treat all pending sends as delivered, and do not regress distinct post-tool commentary behavior such as https://github.com/openclaw/openclaw/issues/76915. Add regression coverage for fail first, retry start, message_end before retry success, and all-fail fallback. Include a user-facing changelog entry for the duplicate delivery fix.
