## Summary

Keep open: current main and v2026.5.12 still flatten Telegram forum-topic origin targets on the normal inbound context, audio preflight context, and mention-skip hook context even when thread metadata is available. Existing helper, docs, and upstream type contracts support preserving the non-General topic in the routing target, and adjacent open Telegram topic PRs do not cover these inbound context builders.

## Plan

Fix https://github.com/openclaw/openclaw/issues/83302 by preserving topic-qualified Telegram origin targets in inbound context builders. In `extensions/telegram/src/bot-message-context.session.ts`, use the existing `buildTelegramRoutingTarget(chatId, threadSpec)` contract for `reply.originatingTo` when a non-General forum topic is known, keeping General topic chat-scoped. In `extensions/telegram/src/bot-message-context.body.ts`, preserve a topic-qualified `OriginatingTo`/`originatingTo` for audio preflight and mention-skip `message_received` hook contexts when `replyThreadId` or `resolvedThreadId` is known. Add focused regression coverage for non-General forum topics and preserve existing DM/General behavior. Do not duplicate the send-fallback or async/cron route work already covered by https://github.com/openclaw/openclaw/pull/76920 and https://github.com/openclaw/openclaw/pull/81885. Add a concise `CHANGELOG.md` Fixes entry because this is user-visible Telegram delivery behavior.
