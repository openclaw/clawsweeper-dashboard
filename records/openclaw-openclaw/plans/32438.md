# Coding Plan for openclaw/openclaw#32438: [Feature]: Add a formatted log file when JSON logs are hard to read and Control UI is unusable

Source report: [records/openclaw-openclaw/items/32438.md](records/openclaw-openclaw/items/32438.md)

## Summary

Keep open. Current main and latest stable still persist Gateway file logs as one JSONL stream; the shipped CLI fallback helps read that JSONL when local RPC is unavailable, but it does not create the requested persistent formatted companion file. The earlier duplicate is closed, and the related implementation PRs are closed unmerged rather than viable canonical fixes.

## Plan

Implement a focused OpenClaw logging enhancement for this issue: when file logging is enabled, derive a human-readable companion log path from the active logging.file by inserting -formatted before the extension, write one redacted plain-text line per JSONL event, and leave the existing JSONL file, logs.tail, Control UI parsing, and CLI JSON/pretty consumers unchanged. Keep the implementation in the core logger path, do not add a new config key unless the existing no-config plan proves unsafe, and explicitly handle rolling daily file changes, rotation, pruning, and append failures without blocking logging. Update operator docs for the derived path and lifecycle. Do not edit CHANGELOG.md. Stop and ask for maintainer direction if the safest implementation requires an opt-out config surface or changes the JSONL contract.
