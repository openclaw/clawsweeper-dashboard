# Coding Plan for openclaw/openclaw#32438: [Feature]: Add a formatted log file when JSON logs are hard to read and Control UI is unusable

Source report: [records/openclaw-openclaw/items/32438.md](records/openclaw-openclaw/items/32438.md)

## Summary

Keep open. Current main still writes a single JSONL Gateway file log and only formats it at CLI/UI read time, while the paired implementation PRs are closed unmerged and the merged date-boundary log discovery work is adjacent rather than a formatted sidecar.

## Plan

Implement a focused formatted-log sidecar for the Gateway file logger. When file logging is enabled, derive a human-readable companion path from the active `logging.file` by inserting `-formatted` before the extension, write one redacted plain-text line per event beside the unchanged JSONL file, and do not change `logs.tail`, Control UI, CLI JSONL consumers, or the existing logging config contract. Preserve current rolling-date and max-file-byte behavior for the JSONL file, and make the sidecar lifecycle bounded and documented so it cannot grow indefinitely or drift from the active log. Add regression coverage for sidecar path derivation, redaction, child/subsystem logger formatting, rolling date changes, rotation/pruning behavior, and unchanged JSONL output. Update `docs/logging.md`, `docs/gateway/logging.md`, `docs/gateway/configuration-reference.md` if needed, and add a user-facing changelog entry. Stop and ask for maintainer review if the implementation would require a new config option, changing the JSONL log contract, or a broader logging policy decision.
