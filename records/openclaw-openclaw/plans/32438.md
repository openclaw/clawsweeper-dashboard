# Coding Plan for openclaw/openclaw#32438: [Feature]: Add a formatted log file when JSON logs are hard to read and Control UI is unusable

Source report: [records/openclaw-openclaw/items/32438.md](records/openclaw-openclaw/items/32438.md)

## Summary

Keep open. Current main still persists Gateway file logs as JSONL only, with readable formatting done by the CLI/Control UI at read time rather than by a durable formatted sidecar; the related implementation PRs are closed unmerged, and the merged rolling-log fallback work is adjacent but does not satisfy this request.

## Plan

Implement a focused replacement for https://github.com/openclaw/openclaw/issues/32438. Add a human-readable formatted companion log file derived from the active `logging.file` path without changing the existing JSONL file format, CLI/Control UI JSONL consumers, or adding a new config key unless maintainers explicitly request one. The sidecar should write one redacted plain-text line for each main file-log event, handle rolling date changes and JSONL rotation/pruning deliberately, avoid unbounded formatted-log growth, and preserve existing logging redaction. Use the closed PRs as reference only: https://github.com/openclaw/openclaw/pull/27446, https://github.com/openclaw/openclaw/pull/32442, and https://github.com/openclaw/openclaw/pull/29932. Add focused regression coverage for derived path naming, child logger subsystem/message formatting without raw JSON prefix noise, redaction, rotation/date-boundary behavior, and unchanged JSONL consumers. Add a user-facing changelog entry and update logging docs.
