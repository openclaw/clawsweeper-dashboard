# Coding Plan for openclaw/openclaw#54409: [Bug]: Feishu plugin: per-chat serial queue prevents messages.queue.mode = \"collect\" from batching queued messages

Source report: [records/openclaw-openclaw/items/54409.md](records/openclaw-openclaw/items/54409.md)

## Summary

Keep open. Current main and the latest shipped release still route ordinary Feishu same-chat messages through the sequential queue before the gateway can observe active-run state, so collect-mode follow-ups can still become separate `run-now` turns; the related Feishu timeout work explicitly did not address this behavior.

## Plan

Fix https://github.com/openclaw/openclaw/issues/54409. The Feishu receive path currently serializes ordinary same-chat messages through the full `handleMessage`/agent turn before they can reach active-run queue policy, so `messages.queue.mode = "collect"` cannot batch mid-run follow-ups. Add a regression that keeps the first same-chat Feishu dispatch pending and proves later ordinary messages can reach the gateway/follow-up queue while the active run is still active. Implement the fix in the Feishu ingress scheduling boundary, preserving FIFO for the first normal message, dedupe, inbound debounce merging, `:control` and `:btw` out-of-band lanes, and the `taskTimeoutMs` starvation guard. Do not change global queue semantics or add a new config option unless the regression proves the shared contract is wrong. Add a user-facing changelog entry during landing.
