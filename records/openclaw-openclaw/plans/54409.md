# Coding Plan for openclaw/openclaw#54409: [Bug]: Feishu plugin: per-chat serial queue prevents messages.queue.mode = \"collect\" from batching queued messages

## Summary

Keep open. Current main and the latest release still await Feishu's same-chat sequential queue around the full message handling and reply dispatch path, so later ordinary same-chat messages can enter the shared queue only after the prior run is no longer active and `collect` can still miss batching them.

## Plan

Fix https://github.com/openclaw/openclaw/issues/54409. Feishu currently awaits the per-chat sequential queue around the full `handleMessage` / `dispatchReplyFromConfig` lifecycle, so later ordinary same-chat messages reach the active-run queue only after the first run is inactive and `messages.queue.mode: "collect"` cannot batch them. Implement a bounded Feishu ingress repair so ordinary same-chat follow-up messages can reach the shared active-run collect queue while preserving Feishu ordering, dedupe, inbound debounce behavior, `:control` and `:btw` lanes, and the task-timeout starvation guard from https://github.com/openclaw/openclaw/pull/76687. Do not remove the sequential queue wholesale or change global queue policy unless the Feishu seam cannot be made correct. Add focused regression coverage for same-chat Feishu collect-mode follow-ups and keep existing sequential-key/queue tests green. Add a user-facing changelog fix entry if the PR is prepared for landing.
