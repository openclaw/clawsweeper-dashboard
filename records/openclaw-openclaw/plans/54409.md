# Coding Plan for openclaw/openclaw#54409: [Bug]: Feishu plugin: per-chat serial queue prevents messages.queue.mode = \"collect\" from batching queued messages

Source report: [records/openclaw-openclaw/items/54409.md](records/openclaw-openclaw/items/54409.md)

## Summary

Keep open. Current main and the latest shipped release still route ordinary Feishu same-chat messages through the per-key sequential queue before the gateway sees active-run state, so collect-mode messages can still miss the shared follow-up batching path; the related timeout fix at https://github.com/openclaw/openclaw/pull/76687 addressed starvation but deliberately left this behavior unresolved.

## Plan

Fix https://github.com/openclaw/openclaw/issues/54409 and, if the same root cause is covered, https://github.com/openclaw/openclaw/issues/67793. Feishu currently awaits the per-key sequential queue around the full `handleMessage`/agent reply turn, so ordinary same-chat messages sent during an active run reach gateway queue policy only after `isActive` is false and `messages.queue.mode = "collect"` cannot batch them. Refactor the Feishu inbound dispatch boundary so normal same-chat follow-up messages can reach the existing active-run follow-up queue while preserving same-chat FIFO for the first/claim path, dedupe, inbound debounce, `:control` and `:btw` out-of-band lanes, and the queue timeout guard from https://github.com/openclaw/openclaw/pull/76687. Do not add a new user config option or move Feishu-specific policy into core unless a small SDK/contract seam is strictly necessary. Add focused regression coverage for the Feishu queue path plus existing queue collect policy, and add a changelog entry because this is user-visible channel behavior.
