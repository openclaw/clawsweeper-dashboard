# Coding Plan for openclaw/openclaw#54409: [Bug]: Feishu plugin: per-chat serial queue prevents messages.queue.mode = \"collect\" from batching queued messages

Source report: [records/openclaw-openclaw/items/54409.md](records/openclaw-openclaw/items/54409.md)

## Summary

Keep open. Current main and the latest release still serialize normal same-chat Feishu messages before they reach the gateway active-run queue, so `messages.queue.mode = "collect"` can still miss the documented batching path; the related starvation timeout work was merged but explicitly scoped this active-run collect behavior out.

## Plan

Fix the Feishu active-run collect-mode batching bug tracked at https://github.com/openclaw/openclaw/issues/54409. Current Feishu normal same-chat messages are serialized through `createSequentialQueue` and `await enqueue(sequentialKey, task)` before reaching the gateway active-run follow-up queue, so later messages arrive after `isActive` is false and run as separate turns. Implement a Feishu-owned dispatch-boundary repair so normal follow-ups can reach the shared `messages.queue.mode = "collect"` path while preserving dedupe, inbound debounce merging, first-message FIFO, and the existing `:control` and `:btw` out-of-band lanes. Add focused regression coverage for the active-run collect shape and do not change global queue semantics unless the focused Feishu test proves a missing shared contract. Add a changelog entry if the final change is user-visible runtime behavior.
