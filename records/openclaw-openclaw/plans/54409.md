## Summary

Keep open. Current main and the latest release still await Feishu's same-chat sequential queue around full message handling, so later ordinary same-chat messages can miss the active-run window where `collect` would enqueue and batch them.

## Plan

Fix the Feishu active-run collect batching bug reported at https://github.com/openclaw/openclaw/issues/54409. Current Feishu receive handling awaits the same-chat sequential queue around the full `handleMessage`/agent run path, so later ordinary same-chat messages do not reach shared queue policy until the first run is inactive and `messages.queue.mode = "collect"` returns `run-now`. Keep the fix in the Feishu plugin boundary unless a failing test proves shared queue code is wrong. Preserve same-chat ordering for claim/dedupe work, inbound debounce merging, control-command and `/btw` out-of-band lanes, and the timeout behavior from https://github.com/openclaw/openclaw/pull/76687. Add a focused regression that models one long-running Feishu dispatch followed by multiple ordinary same-chat messages and proves they enter the gateway/follow-up collect path while the run is still active. Add or update changelog for the user-visible Feishu fix. Do not change global queue-mode semantics or add a new config option unless maintainer review explicitly asks for that direction.
