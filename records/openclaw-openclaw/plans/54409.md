# Coding Plan for openclaw/openclaw#54409: [Bug]: Feishu plugin: per-chat serial queue prevents messages.queue.mode = \"collect\" from batching queued messages

Source report: [records/openclaw-openclaw/items/54409.md](records/openclaw-openclaw/items/54409.md)

## Summary

Keep open. Current main and v2026.5.18 still run normal Feishu same-chat messages through a per-key sequential queue before they reach the gateway active-run policy, so collect-mode follow-ups can still arrive after the active run has ended; the linked timeout work addressed starvation but left this collect behavior unresolved.

## Plan

Fix Feishu collect-mode batching for normal same-chat messages. Current `createFeishuMessageReceiveHandler` awaits `createSequentialQueue` over the full `handleMessage` / `core.channel.turn.run` lifecycle, so later normal messages only reach `resolveActiveRunQueueAction` after the active run is gone and `messages.queue.mode = "collect"` cannot batch them. Implement a Feishu-owned ingress/dispatch-boundary fix so later normal same-chat messages can reach the shared active-run follow-up queue while the first run is active. Preserve first-message FIFO, Feishu dedupe and inbound debounce behavior, `recordProcessedMessage`/release semantics, the `:control` and `:btw` lanes from https://github.com/openclaw/openclaw/pull/64324, and the task-timeout starvation guard from https://github.com/openclaw/openclaw/pull/76687. Do not add a new config option or broad global gateway policy unless the Feishu boundary proves impossible; stop for maintainer review if a shared SDK/gateway contract change is required. Add focused regression coverage and a single-line `CHANGELOG.md` fix entry.
