# Coding Plan for openclaw/openclaw#54409: [Bug]: Feishu plugin: per-chat serial queue prevents messages.queue.mode = \"collect\" from batching queued messages

Source report: [records/openclaw-openclaw/items/54409.md](records/openclaw-openclaw/items/54409.md)

## Summary

Keep open. Current main and the latest shipped release still route ordinary same-chat Feishu messages through the sequential queue before the gateway can observe active-run state, so collect-mode follow-ups can still arrive as separate turns; the merged Feishu timeout fix explicitly scoped this behavior out.

## Plan

Fix https://github.com/openclaw/openclaw/issues/54409. Current Feishu ingress awaits the per-chat sequential queue across the full `handleFeishuMessage` and agent turn, so same-chat messages sent while a run is active only reach the gateway after the active run is no longer active and `messages.queue.mode = "collect"` cannot batch them. Implement a Feishu-owned dispatch-boundary fix so ordinary active-run follow-ups can reach the shared followup/collect queue while preserving same-chat FIFO for the first claim/dedupe path, inbound debounce merging, duplicate suppression, `:control` and `:btw` lanes, and the queue task-timeout guard added for https://github.com/openclaw/openclaw/issues/70133. Do not change global queue semantics unless a focused regression proves the shared queue code is wrong. Add regression coverage that holds the first same-key Feishu dispatch pending and proves later ordinary same-chat messages enter the active-run followup/collect path instead of waiting for the first full agent turn to finish. Add a user-facing changelog fix entry if this becomes a maintained repair PR.
