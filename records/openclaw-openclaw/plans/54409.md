# Coding Plan for openclaw/openclaw#54409: [Bug]: Feishu plugin: per-chat serial queue prevents messages.queue.mode = \"collect\" from batching queued messages

Source report: [records/openclaw-openclaw/items/54409.md](records/openclaw-openclaw/items/54409.md)

## Summary

Keep open. Current main and the latest shipped release still run normal Feishu same-chat messages through the per-chat sequential queue before they reach the gateway active-run queue, so `messages.queue.mode = "collect"` can still miss batching; the merged timeout fix at https://github.com/openclaw/openclaw/pull/76687 addressed starvation for https://github.com/openclaw/openclaw/issues/70133 and was explicitly scoped away from this collect-mode behavior.

## Plan

Fix https://github.com/openclaw/openclaw/issues/54409. Current Feishu normal same-chat messages are serialized through `createSequentialQueue` around the full `handleFeishuMessage`/agent turn, so later messages only reach the gateway after the first run ends and `messages.queue.mode = "collect"` cannot batch them. Implement a Feishu-owned dispatch-boundary fix that lets normal same-chat follow-ups reach `dispatchReplyFromConfig` while the first run is still active. Preserve Feishu message dedupe, inbound debounce merging, first-message ordering, `:control` abort lane behavior, `/btw` lane behavior, comments/reactions, and the timeout fix from https://github.com/openclaw/openclaw/pull/76687. Add focused regression coverage for the active-run collect scenario and update `CHANGELOG.md` because this is user-visible. Do not change global queue semantics or add config knobs unless a failing test proves the gap is shared outside Feishu.
