## Summary

Keep open. Current `main` and the latest stable release still have a source-visible plain-status path where service summaries can reach unbounded `systemctl` subprocesses, so the central `openclaw status` hang is not proven fixed by the separate `status --deep` Docker-audit fix.

## Plan

Fix https://github.com/openclaw/openclaw/issues/84698 by bounding the plain `openclaw status` and `openclaw status --json` service-summary probes. Current source routes `statusCommand` through `resolveStatusRuntimeSnapshot` -> `resolveStatusRuntimeDetails` -> `resolveStatusServiceSummaries` -> `readServiceStatusSummary` -> `readGatewayServiceState`, where Linux systemd helpers call `execSystemctl`/`execFileUtf8` without a timeout. Thread the status timeout, or a clearly derived bounded service-probe budget, into gateway/node service summary reads; make hung service-manager calls fail soft to `unknown`/degraded service rows instead of blocking the whole command. Do not add a new config/env option, do not weaken the merged `status --deep` Docker audit timeout from https://github.com/openclaw/openclaw/pull/85476, and do not edit `CHANGELOG.md` for the repair PR. Add regression coverage with a non-settling service/systemctl double and include proof that default and JSON status return degraded output instead of timing out.
