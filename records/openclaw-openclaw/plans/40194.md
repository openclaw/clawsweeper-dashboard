## Summary

Keep this open. Current main still routes Feishu group messages through an account-aware route lookup, but the canonical non-DM session key drops the account identity, so multiple Feishu accounts in the same group can collide on the same session.

## Plan

Fix Feishu multi-account group session isolation for https://github.com/openclaw/openclaw/issues/40194. Current main passes accountId from extensions/feishu/src/bot.ts into resolveAgentRoute, but src/routing/session-key.ts drops accountId for non-direct peers, so two Feishu accounts in the same group can collide on agent:<agent>:feishu:group:<chat>. Implement account-scoped non-direct session identity for Feishu multi-account group routes, preferably using existing accountId rather than Feishu appId and preserving current single-default-account keys where feasible. Keep binding matching on plain group/chat ids; do not require users to put app ids, app secrets, or account suffixes into bindings. Add regression coverage for two Feishu accounts in the same chat producing distinct route/session keys and for preserving default-account/single-account behavior. Stop and route back if the only viable fix requires a new user-facing config option or broad cross-channel session migration. Include a changelog entry if the fix is user-facing.
