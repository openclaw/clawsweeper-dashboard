# Coding Plan for openclaw/openclaw#40194: [Feishu] 飞书群多机器人身份混乱问题

Source report: [records/openclaw-openclaw/items/40194.md](records/openclaw-openclaw/items/40194.md)

## Summary

Keep open: current main still builds Feishu group session identity from the chat-scoped group peer, so multiple Feishu app accounts in the same group can share the same persisted session despite account-aware routing inputs.

## Plan

Fix https://github.com/openclaw/openclaw/issues/40194. Current main passes Feishu `accountId` to route resolution, but default Feishu group sessions keep the peer ID as the raw `chat_id`, and non-direct session keys ignore `accountId`, so multiple Feishu accounts/apps in the same group can share `agent:<agent>:feishu:group:<chat>`. Implement a Feishu-owned account-aware group session/routing identity for normal group, topic, sender, and topic-sender scopes while preserving legacy single-account behavior or adding a clear compatibility path. Add focused regression tests showing two Feishu accounts in the same chat produce isolated routes/session keys and do not mix bot identity. Keep the fix inside Feishu-owned routing/session construction unless a minimal typed core seam is required; do not make a broad generic non-direct session-key migration for every channel, and do not touch unrelated Feishu tool account routing or `sessions_send` behavior. Add a one-line user-facing changelog entry if behavior changes.
