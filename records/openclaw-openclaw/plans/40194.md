## Summary

Keep open: current main still builds Feishu group session identity from the group chat peer while the shared non-direct session-key path omits accountId, so multiple Feishu apps in the same group can still collide on one group session despite account-scoped routing support.

## Plan

Fix https://github.com/openclaw/openclaw/issues/40194. Current Feishu group messages use a chat-scoped peer id and the shared non-direct session-key path ignores accountId, so multiple Feishu accounts/apps in the same group can share `agent:<agent>:feishu:group:<chat>`. Implement account-aware Feishu group session identity in the Feishu-owned path where possible, preserving legacy single-account behavior and avoiding a global group-key change unless it is explicitly necessary. Add regression coverage for two Feishu accounts in the same group producing isolated session keys/routes/account metadata, while keeping existing `groupSessionScope`, topic, sender, and broadcast behavior stable. Update docs and changelog for the user-visible bug fix. Do not change unrelated channel routing, ACP thread binding behavior, or Feishu broadcast dedup semantics.
