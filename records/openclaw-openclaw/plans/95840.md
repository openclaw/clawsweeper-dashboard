## Summary

Keep open: current main and the latest release still hard-gate cache-ttl pruning and cache-ttl marker writes through eligibility that rejects OpenAI, so the reported OpenAI idle-gap pruning path remains disabled.

## Plan

Fix the OpenAI cache-ttl pruning eligibility bug reported in https://github.com/openclaw/openclaw/issues/95840. Current main accepts `agents.defaults.contextPruning.mode: "cache-ttl"` but skips OpenAI because `isCacheTtlEligibleProvider` returns false, which prevents both the context-pruning extension and `openclaw.cache-ttl` marker writes from activating. Implement the narrowest owner-boundary fix so OpenAI automatic-prefix-cache models can use the existing `cache-ttl` pruning mode without adding a new config option and without sending Anthropic `cache_control` to OpenAI payload builders. Prefer separating pruning/marker eligibility from Anthropic cache-control eligibility if a direct `isCacheTtlEligibleProvider("openai", ...)` change would affect unrelated call sites. Preserve existing Anthropic, Google, OpenRouter, Moonshot/ZAI, and unsupported-provider semantics. Add focused regression tests for OpenAI eligibility/pruning factory activation/marker writes, and update docs only if the supported-provider wording changes. Do not edit `CHANGELOG.md`. Treat https://github.com/openclaw/openclaw/issues/95610 as related context only; do not attempt the broader per-turn prefix-stability repair in this PR.
