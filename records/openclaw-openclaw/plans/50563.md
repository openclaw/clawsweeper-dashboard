# Coding Plan for openclaw/openclaw#50563: Message loss on gateway crash (ack-before-process)

Source report: [records/openclaw-openclaw/items/50563.md](records/openclaw-openclaw/items/50563.md)

## Summary

Keep open: current main and the latest stable release still send WhatsApp read receipts before enrichment, dedupe claim/commit, and the awaited inbound handler, so the reported loss window remains real at source level.

## Plan

Fix https://github.com/openclaw/openclaw/issues/50563 on current main. In the bundled WhatsApp plugin, stop acknowledging inbound WhatsApp messages before they are durably safe: delay `sock.readMessages` until an accepted inbound turn is either written to a durable replay boundary or `options.onMessage` has completed and inbound dedupe finalization has succeeded. Keep existing `sendReadReceipts: false` and self-chat behavior. Be careful with debounce batches, retryable inbound errors, media/enrichment failures, stale append history, and append messages that are skipped before `onMessage`; do not broaden this into the general gateway restart persistence work tracked by https://github.com/openclaw/openclaw/issues/51620. Update focused tests that currently assert eager read receipts, add a regression proving `onMessage` failure does not send a read receipt, update WhatsApp docs if the documented behavior changes, and add a single-line changelog entry because this is user-visible reliability behavior.
