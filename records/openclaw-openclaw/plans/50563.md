# Coding Plan for openclaw/openclaw#50563: Message loss on gateway crash (ack-before-process)

## Summary

Keep open: current main and v2026.5.19 still send WhatsApp read receipts before enrichment, durable handoff, dedupe commit, or `onMessage` delivery. Ryce's KeepMyClaw comment is useful complementary crash-state context, but it does not replace a bundled WhatsApp plugin fix for the acknowledgment boundary.

## Plan

Fix https://github.com/openclaw/openclaw/issues/50563 in the bundled WhatsApp plugin. Current behavior calls `sock.readMessages` in `extensions/whatsapp/src/inbound/monitor.ts` before accepted inbound messages are durably stored, dedupe-finalized, or delivered to `onMessage`; a crash in that window can silently lose a message. Implement a plugin-owned durable accepted-inbound handoff/replay or equivalent persist-before-read contract before `readMessages`, and update the read-receipt ordering so accepted messages are either durably recoverable or fully handled/dedupe-finalized before the user-visible read receipt. Keep the fix scoped to WhatsApp inbound monitor/dedupe behavior; do not broaden into the all-channel gateway restart tracker at https://github.com/openclaw/openclaw/issues/51620 and do not add a core API unless a generic seam is truly required. Preserve self-chat read-receipt behavior, append-history handling, debounce batching, retryable inbound errors, media/enrichment failures, and duplicate-reply prevention. Add or update focused tests around read receipt ordering, durable/replay behavior, dedupe commit/release, and existing read receipt cases; update WhatsApp docs and CHANGELOG.md if behavior or operator-visible persistence changes. Confirm Baileys read receipt semantics from the pinned `baileys` source and keep any on-disk message content/metadata privacy-conscious.
