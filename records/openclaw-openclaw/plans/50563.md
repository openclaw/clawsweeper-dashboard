# Coding Plan for openclaw/openclaw#50563: Message loss on gateway crash (ack-before-process)

Source report: [records/openclaw-openclaw/items/50563.md](records/openclaw-openclaw/items/50563.md)

## Summary

Keep open: current main and the latest shipped release still send WhatsApp read receipts before accepted inbound messages reach enrichment, dedupe commit, or `onMessage` delivery, and there is no durable WhatsApp inbound replay boundary. Dependency inspection also shows Baileys read-message calls are read receipts, so the fix should use local durable handoff/replay rather than assuming WhatsApp will redeliver after a crash.

## Plan

Fix the WhatsApp ack-before-process crash-loss window described in https://github.com/openclaw/openclaw/issues/50563. Keep the change owned by the bundled WhatsApp plugin unless source inspection proves a small generic SDK seam is required. Do not rely on WhatsApp redelivery after Baileys receipts; add or reuse a plugin-local durable inbound handoff/replay point before `sock.readMessages` and before accepted handler side effects, then send read receipts only after accepted messages are durably recorded or successfully delivered through `onMessage` and dedupe finalization. Update focused WhatsApp monitor/dedupe tests so eager read receipts are no longer asserted before accepted delivery or durable handoff. Do not implement a broad gateway snapshot system, cross-provider catch-up program, or new config option unless maintainers explicitly choose that product direction.
