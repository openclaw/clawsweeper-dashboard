## Summary

Keep open: current main still sends WhatsApp read receipts before durable inbound storage, inbound dedupe finalization, or handler delivery, so the reported crash-loss window remains. Ryce's KeepMyClaw context is useful complementary crash-state recovery, but it does not replace a plugin-owned acknowledgment ordering fix.

## Plan

Fix https://github.com/openclaw/openclaw/issues/50563 in the bundled WhatsApp plugin. Current main calls `sock.readMessages` in `extensions/whatsapp/src/inbound/monitor.ts` before append filtering, enrichment, inbound dedupe claim/commit, and debounced `onMessage` delivery; a crash in that window can mark the WhatsApp message read without processing it. Move the read-receipt boundary so accepted inbound messages are marked read only after they are durably stored for replay or after successful `onMessage` delivery plus inbound dedupe finalization. Preserve self-chat/no-read-receipt behavior, access-control filtering, append catch-up semantics, media/enrichment failure behavior, debounce batching, and retryable inbound error release semantics. Do not add a user-facing config option unless maintainers explicitly request one. Update focused WhatsApp monitor tests to prove read receipts are not sent before accepted handler delivery/dedupe finalization and that skipped/rejected messages do not get new premature receipts. Add a concise changelog entry for the user-visible reliability fix and docs only if the read-receipt behavior text needs clarification.
