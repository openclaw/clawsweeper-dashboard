## Summary

Current main and the latest release still have the reported WhatsApp ack-before-process window: accepted inbound messages can be marked read before durable storage, dedupe finalization, or handler delivery. Ryce's KeepMyClaw comment is useful complementary crash-state recovery context, but it does not replace fixing the bundled WhatsApp plugin's acknowledgment ordering.

## Plan

Fix the WhatsApp plugin ack-before-process crash window reported at https://github.com/openclaw/openclaw/issues/50563. Keep the change scoped to the bundled WhatsApp plugin: delay `sock.readMessages` until an accepted inbound message is durably stored for replay or has successfully reached `onMessage` and inbound dedupe finalization. Preserve existing self-chat and `sendReadReceipts: false` privacy behavior, append-history filtering semantics, retryable inbound error release behavior, debounce batching, media/enrichment handling, and existing public config keys. Do not add a new core API or config option unless the current plugin/SDK surfaces make the minimal fix impossible; stop for maintainer review if that happens. Update focused WhatsApp monitor tests so read receipts are not sent before the new safe boundary, and add a user-facing changelog entry for the reliability fix.
