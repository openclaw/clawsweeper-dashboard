# Coding Plan for openclaw/openclaw#79353: xai provider not registering x_search / code_execution tools even with valid Grok model

Source report: [records/openclaw-openclaw/items/79353.md](records/openclaw-openclaw/items/79353.md)

## Summary

Keep open: current main still has a mismatch between xAI auth-profile discovery and the xAI tool credential gates, so valid Grok chat auth can leave `x_search` and `code_execution` unregistered.

## Plan

Fix https://github.com/openclaw/openclaw/issues/79353. Current main can treat an xAI auth profile as sufficient manifest auth for plugin tool availability, but the xAI tool factories and execution auth resolver only accept `XAI_API_KEY`, `plugins.entries.xai.config.webSearch.apiKey`, or legacy `tools.web.search.grok.apiKey`. Align `x_search` and `code_execution` registration/execution with xAI auth profiles created by `openclaw onboard --auth-choice xai-api-key`, preserving the existing env/plugin/legacy key paths. Do not add a new config option. Keep xAI-specific behavior in the xAI plugin unless a small generic plugin-tool auth context seam is required. Do not expose raw auth-profile secrets in descriptors, caches, logs, manifests, or public metadata. Add focused regression coverage for auth-profile-only xAI setup and update docs/changelog if user-visible auth wording changes.
