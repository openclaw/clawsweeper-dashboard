## Summary

Keep open. Current main still resolves non-media Feishu text messages to the raw `message_id`, and the receive, debounce, and final persistent dedupe gates all consume that key, so same-logical private-chat text deliveries with different IDs can still be processed twice; the related closed reconnect and media reused-ID items cover different cases.

## Plan

Fix https://github.com/openclaw/openclaw/issues/46778 with a Feishu-local logical dedupe key for duplicate text/private-chat deliveries where Feishu changes `message_id`. Keep the media behavior from https://github.com/openclaw/openclaw/issues/75057 intact: distinct media attachments with reused IDs must still process separately via `file_key`/`image_key`. Implement the shared key in `extensions/feishu/src/dedupe-key.ts` or a local helper so receive-time claims, debounce dedupe, and `handleFeishuMessage` final persistent dedupe all use the same key. Use a conservative fingerprint such as chat/thread context, sender ID, `create_time`, and normalized text content; do not use bare content-only dedupe. Add regression coverage for two p2p text events with the same sender/chat/timestamp/content but different `message_id` values dispatching once, and keep/extend coverage proving same-ID distinct media variants still dispatch. Add a single user-facing changelog fix entry under the active release section. Do not add a config option, core API, or broad Feishu refactor.
