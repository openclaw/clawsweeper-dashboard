# Coding Plan for openclaw/openclaw#46778: 飞书消息重复推送，message_id 去重失效

Source report: [records/openclaw-openclaw/items/46778.md](records/openclaw-openclaw/items/46778.md)

## Summary

Keep open. Current main and the latest shipped release still derive non-media Feishu text dedupe from the raw `message_id`, so the reported same-logical-message deliveries with different IDs can still pass the receive and final persistent dedupe gates.

## Plan

Fix https://github.com/openclaw/openclaw/issues/46778 as a narrow Feishu-local bug. Current text messages without media use only `event.message.message_id` as the dedupe key, so the same logical Feishu private-chat text event can be processed twice when Feishu re-delivers it with a different ID. Add a logical text retry fingerprint in `extensions/feishu/src/dedupe-key.ts` using stable event fields such as account-independent sender id, chat id, root/thread context, create_time, message_type, and normalized text/content hash; use that key consistently through receive claims, debounce suppression, persistent freshness checks, and final `finalizeFeishuMessageProcessing`. Preserve the existing media-aware `message_id + file_key/image_key` behavior for image/file/audio/video/post media, and do not move policy into core or add a config option. Add focused regression coverage for two p2p text events with same sender/chat/create_time/content and different `message_id` values, including receive-handler behavior and final persistent dedupe behavior; keep existing media dedupe tests green. Add a single user-facing `CHANGELOG.md` fix entry under the active `### Fixes` section, without thanking bot accounts.
