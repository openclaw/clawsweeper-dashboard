## Summary

Keep open: current main and the latest release still dedupe ordinary Feishu text messages by the resolved `message_id` key, so the reported same-message/different-`message_id` DM duplicate path remains source-reproducible. The linked Feishu mega-PR mentions a create-time dedupe approach, but it is open, broad, and currently flagged as unsafe to merge, so it is not a close reason.

## Plan

Fix the Feishu inbound duplicate-message bug where the same text delivery can arrive with a different message_id and bypass dedupe. Keep the work inside extensions/feishu, preserve the existing media-aware dedupe key behavior for attachments, do not add config/env knobs, do not add standalone cache files, and do not copy the broad open Feishu enhancement PR wholesale. Add a bounded retry identity for ordinary text events using documented Feishu fields such as account/chat/sender/create_time/message_type/normalized content and, where useful, top-level event_id; make it lossless so distinct same-content messages outside the tight retry identity are still processed. Add focused regression coverage for same sender/chat/create_time/content with different message_id being processed once, and for distinct nearby messages not being dropped.
