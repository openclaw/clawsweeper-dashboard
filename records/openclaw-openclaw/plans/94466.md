## Summary

Keep open: current main and v2026.6.8 still have Gateway managed-image blocks with relative `url`/`openUrl` fields, while the native SwiftUI chat model/render path does not preserve or render those image fields, so the macOS WebChat media-action regression is not implemented or superseded.

## Plan

Fix native macOS SwiftUI WebChat handling of Gateway managed outgoing image blocks. Current Gateway emits `type: "image"` blocks with relative `/api/chat/media/outgoing/.../full` `url` and `openUrl`; current `OpenClawChatMessageContent` drops those fields and `ChatMessageBody` only renders `file`/`attachment`. Add a narrow model/rendering/transport path that preserves image metadata, fetches or resolves relative Gateway media URLs using existing authenticated Gateway context, and implements Open/Download/Copy for native chat images. Do not make managed media unauthenticated, do not add config/env options, do not change browser Control UI remote Markdown policy, and do not bundle unrelated document-media work. Add focused Swift regression coverage and, if Gateway media code is touched, run the managed-image tests. No CHANGELOG edit.
