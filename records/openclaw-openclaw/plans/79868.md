# Coding Plan for openclaw/openclaw#79868: Embedded run silently uses fallback provider without logging \"model fallback decision\" when primary is unreachable mid-dispatch

Source report: [records/openclaw-openclaw/items/79868.md](records/openclaw-openclaw/items/79868.md)

## Summary

Keep open. Current main still preserves auto fallback selections across later embedded runs and only emits `model fallback decision` from the active fallback loop, so a reused auto fallback can dispatch first without the requested gateway log line.

## Plan

Fix https://github.com/openclaw/openclaw/issues/79868 by adding explicit telemetry when an embedded auto-reply run starts from a stored auto fallback selection instead of the configured primary. Keep current fallback persistence behavior and do not add a fallbackPersistence/TTL/config policy in this repair; avoid mislabeling `modelOverrideSource: "user"` or legacy user overrides as fallback. Prefer a small helper or log/trajectory emission at the model-selection or embedded-runner boundary where `modelOverrideSource: "auto"`, stored provider/model, and origin provider/model are available. Add regression coverage proving an auto fallback override reused on a later embedded run emits an operator-visible fallback decision or equivalent event, while user/legacy overrides do not. Coordinate with the related open TTL PR so the fix remains compatible if that policy later lands. Add a changelog entry if the repository’s user-visible fix policy requires it.
