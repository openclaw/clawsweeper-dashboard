# Coding Plan for openclaw/openclaw#79868: Embedded run silently uses fallback provider without logging \"model fallback decision\" when primary is unreachable mid-dispatch

Source report: [records/openclaw-openclaw/items/79868.md](records/openclaw-openclaw/items/79868.md)

## Summary

Keep open. Current main still logs `model fallback decision` for fallback-loop failures, but a persisted `modelOverrideSource: "auto"` fallback can become the next embedded run's first candidate and bypass the failed-primary log path; the related https://github.com/openclaw/openclaw/issues/64902 tracks transcript audit persistence, not this gateway-log gap.

## Plan

Fix https://github.com/openclaw/openclaw/issues/79868 by adding explicit, sanitized gateway fallback telemetry when an embedded reply run starts from an auto-persisted fallback model (`modelOverrideSource: "auto"`) while the configured primary differs. Keep the existing fallback persistence and candidate ordering intact. Do not change strict `modelOverrideSource: "user"` behavior, do not add a config option, and do not broaden fallback eligibility. Prefer implementing at the auto fallback-selection boundary or model fallback decision emission boundary so operators get a `model fallback decision` or equivalent log line explaining that the configured primary is being bypassed due to a reused auto fallback selection. Add focused regression coverage for an auto fallback session state that dispatches the fallback first and verifies telemetry, plus negative coverage that user-selected overrides are not reported as fallback reuse. Related context: https://github.com/openclaw/openclaw/issues/64902 is adjacent transcript audit work and should not be solved unless the chosen implementation naturally reuses the same sanitized fallback-step metadata. Changelog expected because this is user-facing observability.
