# Coding Plan for openclaw/openclaw#79868: Embedded run silently uses fallback provider without logging \"model fallback decision\" when primary is unreachable mid-dispatch

Source report: [records/openclaw-openclaw/items/79868.md](records/openclaw-openclaw/items/79868.md)

## Summary

Keep open. Current main still preserves auto fallback overrides and can start a later embedded turn on the fallback provider; the fallback decision log path only fires for skips, failures, cooldown probes, or successes after earlier fallback activity, so the reported silent reused-fallback path remains plausible.

## Plan

Fix https://github.com/openclaw/openclaw/issues/79868 by adding explicit fallback telemetry when an embedded reply run starts from an auto-sourced persisted fallback selection instead of the configured primary. Preserve current sticky auto fallback behavior and do not add a config option or change strict user override semantics. Prefer a `model fallback decision` or equivalent fallback-step/lifecycle event that identifies the requested/origin primary, the reused fallback candidate, `modelOverrideSource: "auto"`, run/session/lane context, and that this is a reused auto fallback rather than a newly failed user override. Add focused coverage proving a session with `modelOverrideSource: "auto"` and a fallback provider/model emits the new diagnostic while a `modelOverrideSource: "user"` selection does not. Coordinate with, but do not implement, the separate per-turn persistence feature in https://github.com/openclaw/openclaw/issues/80502.
