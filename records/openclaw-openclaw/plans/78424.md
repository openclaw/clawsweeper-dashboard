# Coding Plan for openclaw/openclaw#78424: sessions.resolve returns INVALID_REQUEST for sessionKey=\"current\" despite system prompt explicitly recommending it

Source report: [records/openclaw-openclaw/items/78424.md](records/openclaw-openclaw/items/78424.md)

## Summary

Keep open: current main still documents `sessionKey="current"` for the caller's current session, but the gateway `sessions.resolve` RPC still receives only raw resolve params and treats `key: "current"` as a literal store key, so the failed `No session found: current` probe remains unsolved at the WS layer.

## Plan

Fix https://github.com/openclaw/openclaw/issues/78424 by making direct gateway `sessions.resolve({ key: "current" })` resolve the caller's current session instead of returning/logging `INVALID_REQUEST: No session found: current`. Keep the fix gateway-owned and narrow: thread or derive a trustworthy current session binding from the WS/request context, preserve existing literal session key/sessionId named `current` precedence, preserve spawnedBy/visibility and deleted-agent checks after resolving the effective key, and avoid adding config/env/protocol version churn unless unavoidable. Coordinate with the related WebChat wrong-session report at https://github.com/openclaw/openclaw/issues/89773, but do not broaden the PR into unrelated session-status UI behavior. Add regression coverage for direct gateway `sessions.resolve` with `key: "current"`, no failed literal probe, existing literal-current behavior, and wrapper/session-status behavior that should remain unchanged.
