# Coding Plan for openclaw/openclaw#54529: Telegram: apiRoot local Bot API server receives updates but never dispatches to agent sessions

Source report: [records/openclaw-openclaw/items/54529.md](records/openclaw-openclaw/items/54529.md)

## Summary

Keep open: current main and v2026.6.6 still persist Telegram polling offsets by account/token only while polling uses apiRoot separately, so switching the same bot token between the official and a local Bot API root can reuse an incompatible offset; webhook cleanup also still treats non-network 400 errors such as Logged out as fatal setup failures.

## Plan

Fix issue #54529 in the Telegram plugin without adding a new config option. Make Telegram update-offset state endpoint-aware for channels.telegram.apiRoot changes: normalize the effective apiRoot, include it in the persisted offset identity or rotation check, reset/delete stale offsets when the same bot token switches between official and local Bot API roots, and cover both default and named accounts. Keep existing broad apiRoot/media behavior intact. Also add a narrow migration-cooldown handling path so deleteWebhook 400 Logged out during the official-to-local Bot API logOut window does not crash-loop the polling startup when getUpdates can proceed. Add focused regression tests for official-to-local and local-to-official offset switching, unchanged same-endpoint behavior, existing bot-token rotation behavior, and deleteWebhook Logged out handling. Update Telegram docs only if there remains a manual operator step or user-visible behavior to document.
