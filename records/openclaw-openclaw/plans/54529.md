# Coding Plan for openclaw/openclaw#54529: Telegram: apiRoot local Bot API server receives updates but never dispatches to agent sessions

Source report: [records/openclaw-openclaw/items/54529.md](records/openclaw-openclaw/items/54529.md)

## Summary

Keep open. Current main and v2026.5.22 still scope Telegram polling offsets to account/token rather than normalized apiRoot, and deleteWebhook still only tolerates recoverable network failures, so the reported local Bot API migration path remains source-reproducible.

## Plan

Fix Telegram local Bot API apiRoot migration for https://github.com/openclaw/openclaw/issues/54529. In the Telegram plugin, make persisted polling offsets endpoint-aware using the normalized apiRoot as part of the stored identity or rotation check; when the stored endpoint differs from the current account config, log a concise stale-offset message, delete/reset the old offset, and start polling without sending the old official-api update_id to the local server. Preserve existing account/token rotation behavior and do not add a new config option. Also make startup deleteWebhook tolerate only the known Telegram 400 Logged out response during local Bot API migration while keeping unrelated 4xx errors fatal. Add focused regression coverage for offset endpoint rotation and deleteWebhook Logged out handling; include a test that shows isolated ingress receives null initialUpdateId after an apiRoot rotation. Do not conflate this with the separate in-flight update durability work in https://github.com/openclaw/openclaw/pull/73993. Add a changelog fix entry because this is user-visible Telegram behavior.
