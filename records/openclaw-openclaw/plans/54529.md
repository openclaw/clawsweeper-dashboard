# Coding Plan for openclaw/openclaw#54529: Telegram: apiRoot local Bot API server receives updates but never dispatches to agent sessions

Source report: [records/openclaw-openclaw/items/54529.md](records/openclaw-openclaw/items/54529.md)

## Summary

Keep open. Current main still scopes Telegram polling offsets to account and token identity, but not the configured Bot API endpoint, so a same-token switch from the official API to a local `apiRoot` can keep polling from a stale higher watermark; the `deleteWebhook` startup path also still lacks a narrow recoverable path for the migration-time logged-out rejection.

## Plan

Fix https://github.com/openclaw/openclaw/issues/54529 in the Telegram plugin. Persist a normalized Bot API endpoint identity with update-offset state so same-token switches between the official API and a local `channels.telegram.apiRoot` reset or ignore stale watermarks; thread that identity through monitor read/write and both isolated and non-isolated polling paths. Add a narrow recoverable `deleteWebhook` handling path for the local Bot API migration `400: Logged out` cooldown without treating ordinary token/auth/client rejections as recoverable. Do not add a new config option or change core plugin APIs. Add regression coverage for same-token endpoint rotation, isolated worker initial offset behavior, non-isolated tracker floor behavior if needed, and the specific deleteWebhook logged-out classification. Add a user-facing changelog entry because this fixes Telegram message delivery/startup behavior.
