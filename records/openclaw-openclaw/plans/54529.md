## Summary

Keep open. Current main still has the Telegram apiRoot plumbing, but the reported local Bot API migration failure remains source-reproducible because persisted polling offsets are not scoped to the configured Bot API endpoint and the deleteWebhook startup path only downgrades recoverable network failures.

## Plan

Fix https://github.com/openclaw/openclaw/issues/54529 in the Telegram plugin. Preserve existing apiRoot config shape; do not add a new config option or core SDK seam. Store a normalized Telegram Bot API endpoint identity with update-offset state so switching the same account/token between the official API and a self-hosted/local apiRoot does not reuse an incompatible lastUpdateId. Treat legacy offset records without endpoint identity conservatively, similar to the botId migration. Add focused regression coverage for same bot/account with different apiRoot values and for unchanged apiRoot retaining the offset. Also handle the local Bot API migration deleteWebhook 400 Logged out case narrowly during polling startup without treating bad tokens or arbitrary 4xx failures as recoverable. Check overlap with https://github.com/openclaw/openclaw/pull/77075 before touching bot-update-tracker behavior. Add a CHANGELOG.md fix entry because this is user-visible Telegram behavior.
