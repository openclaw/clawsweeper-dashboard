# Coding Plan for openclaw/openclaw#54529: Telegram: apiRoot local Bot API server receives updates but never dispatches to agent sessions

Source report: [records/openclaw-openclaw/items/54529.md](records/openclaw-openclaw/items/54529.md)

## Summary

Keep open. Current main still preserves Telegram polling offsets by account and bot token, but not by the configured Bot API endpoint, so switching the same token from the official API to a local apiRoot can keep polling above the local server's update_id range; the deleteWebhook startup path also still only treats network-style failures as recoverable.

## Plan

Fix https://github.com/openclaw/openclaw/issues/54529 in the Telegram plugin. Scope Telegram update offset state to the normalized Bot API apiRoot identity so switching the same bot token between the official API and a local apiRoot discards or separates stale watermarks; thread that endpoint identity through read/write/delete from monitor and both isolated and non-isolated polling paths. Add a narrow recoverable classification for the local Bot API migration-time deleteWebhook 400 Logged out response without making arbitrary 4xx token/auth failures recoverable. Do not add a user config option or move logic into core. Add regression coverage for same-token endpoint switches, isolated ingress initialUpdateId, non-isolated tracker floors, and deleteWebhook logged-out startup behavior; include a maintainer-owned changelog Fixes entry if this becomes a fix PR.
