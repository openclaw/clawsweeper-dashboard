# Coding Plan for openclaw/openclaw#54529: Telegram: apiRoot local Bot API server receives updates but never dispatches to agent sessions

Source report: [records/openclaw-openclaw/items/54529.md](records/openclaw-openclaw/items/54529.md)

## Summary

Keep open. Current main and the latest release still document Telegram `apiRoot`, but polling offsets are persisted without endpoint identity and `deleteWebhook` startup cleanup still only treats network-style failures as recoverable, so the local Bot API migration path can still lose inbound updates or restart-loop.

## Plan

Fix https://github.com/openclaw/openclaw/issues/54529. In the Telegram plugin, make update-offset persistence include a normalized Bot API endpoint identity so same-token switches between the official API and a local apiRoot do not reuse an incompatible update_id watermark. Thread the endpoint identity through monitor read/write, isolated ingress startup, and non-isolated polling startup as needed. Add a narrow recoverable classification for the local Bot API migration `deleteWebhook` 400 `Logged out` response without weakening ordinary auth failures. Add focused regression tests for endpoint-scoped offsets, monitor wiring, isolated worker initial offset selection, and deleteWebhook migration handling. Do not add a new user config option or move behavior into core.
