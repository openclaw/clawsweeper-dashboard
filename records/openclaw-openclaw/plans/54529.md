# Coding Plan for openclaw/openclaw#54529: Telegram: apiRoot local Bot API server receives updates but never dispatches to agent sessions

Source report: [records/openclaw-openclaw/items/54529.md](records/openclaw-openclaw/items/54529.md)

## Summary

Keep open. Current main and the latest release still persist Telegram polling offsets by account and bot-token identity, but not by the configured Bot API endpoint, so switching the same token from the official API to a local apiRoot can keep polling from a stale higher watermark; the deleteWebhook startup path also still only treats network-style failures as recoverable.

## Plan

Fix https://github.com/openclaw/openclaw/issues/54529 in the Telegram plugin. Current bug: update-offset-store scopes persisted polling offsets to accountId and botToken only, while apiRoot is threaded separately into polling; a same-token switch from official Telegram Bot API to a local Bot API server can reuse a stale higher watermark and skip local updates. Expected repair: include a normalized Bot API endpoint identity in the offset state or offset key, pass it through monitor read/write and both isolated and non-isolated polling startup, treat endpoint changes as a rotation/reset, and add regression coverage. Also handle the narrow local Bot API migration deleteWebhook logged-out rejection as recoverable without masking ordinary invalid-token/auth failures. Keep the fix inside extensions/telegram; do not add a new config option or change core plugin APIs.
