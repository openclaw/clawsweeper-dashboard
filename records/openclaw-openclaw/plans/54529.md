# Coding Plan for openclaw/openclaw#54529: Telegram: apiRoot local Bot API server receives updates but never dispatches to agent sessions

Source report: [records/openclaw-openclaw/items/54529.md](records/openclaw-openclaw/items/54529.md)

## Summary

Keep open. Current main and the latest release still expose Telegram `apiRoot`, but the persisted polling offset is scoped to account/token identity rather than endpoint identity, and startup cleanup still treats a Telegram `400: Logged out` `deleteWebhook` rejection as fatal rather than a recoverable local Bot API migration state.

## Plan

Fix https://github.com/openclaw/openclaw/issues/54529 in the Telegram plugin. Make the update-offset store endpoint-aware for same-token `apiRoot` switches by persisting a normalized endpoint identity or otherwise invalidating/separating stale watermarks when `apiRoot` changes; preserve existing bot-id/token-fingerprint rotation and legacy v1/v2 behavior. Thread the endpoint identity through monitor read/write and isolated polling without changing core plugin APIs or adding a user config option. Narrowly classify Telegram `deleteWebhook` `400: Logged out` during local Bot API migration as recoverable while preserving real auth failures. Add regression coverage for official-to-local lower update ids and the logged-out cleanup path, update changelog if user-facing, and avoid broad Telegram dispatch refactors unless a test proves a separate root cause.
