# Coding Plan for openclaw/openclaw#71865: Auth login blocked by size-drop guard when openclaw.json was created by PowerShell (verbose/BOM format)

Source report: [records/openclaw-openclaw/items/71865.md](records/openclaw-openclaw/items/71865.md)

## Summary

Keep open. Current main and the latest release still route models auth login through the shared config writer that compares canonical next JSON bytes against the previous raw file bytes, so a PowerShell/BOM-formatted config can still false-positive the size-drop guard.

## Plan

Fix the shared config writer false positive for this issue: a PowerShell/BOM or otherwise heavily formatted but semantically valid `openclaw.json` must not make `openclaw models auth login` lose the config update because canonical two-space output is less than half of the previous raw bytes. Keep the fix in the shared config writer rather than adding an auth-only bypass. Preserve hard destructive guards such as `gateway-mode-removed`, last-good comparisons, rejected payload behavior for real clobbers, schema validation, and existing explicit `allowConfigSizeDrop` semantics for intentional deletes. Add focused regression coverage for a raw-large/BOM-formatted config that is semantically unchanged except for the auth/model config mutation, and include the older models-set false-positive shape if practical. Do not implement the separate public bypass requests from https://github.com/openclaw/openclaw/issues/81828 or https://github.com/openclaw/openclaw/issues/82124 in this fix. Add a user-facing changelog fix entry.
