# Coding Plan for openclaw/openclaw#71865: Auth login blocked by size-drop guard when openclaw.json was created by PowerShell (verbose/BOM format)

Source report: [records/openclaw-openclaw/items/71865.md](records/openclaw-openclaw/items/71865.md)

## Summary

Keep open. Current main and latest release `v2026.5.7` still compare canonical next config bytes against the previous raw file bytes, so a BOM-prefixed or heavily formatted PowerShell-created `openclaw.json` can still be rejected during auth-login persistence as a formatting-only `size-drop`.

## Plan

Fix the shared config writer false positive where a valid BOM-prefixed or heavily formatted PowerShell-created `openclaw.json` can be rejected as `size-drop` when auth login or another config mutation rewrites the same semantic content as canonical two-space JSON. Keep the destructive-write guard fail-closed for real clobbers, but make formatting-only/BOM-only shrink safe in the writer rather than as an auth-specific bypass. Add regression coverage for the PowerShell/BOM-style auth/config write path and for preserving true destructive shrink rejection. Coordinate with https://github.com/openclaw/openclaw/pull/79299 and https://github.com/openclaw/openclaw/pull/79549 if they are still open, but do not solve unrelated best-effort overwrite or merge-patch preservation work here. Do not add a new config option or bypass all size-drop checks. Add a changelog entry because the fix is user-facing.
