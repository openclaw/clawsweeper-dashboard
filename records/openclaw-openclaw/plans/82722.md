# Coding Plan for openclaw/openclaw#82722: Heartbeat polls still interrupt model responses during streaming (not just tool calls)

Source report: [records/openclaw-openclaw/items/82722.md](records/openclaw-openclaw/items/82722.md)

## Summary

Keep open: current main has guards for busy command/session lanes and already-active reply runs, but the webchat `chat.send` path registers a chat run before dispatch creates the reply operation, and the heartbeat runner does not consult that active chat-run state. The reported timing gap is source-plausible and is not covered by the earlier session-lane fix.

## Plan

Fix the heartbeat/webchat race reported in https://github.com/openclaw/openclaw/issues/82722. Heartbeat polls must return a retryable `requests-in-flight` skip while the target session has an active or just-started webchat/chat.send run, even before the async dispatch has created a `ReplyOperation` or occupied the embedded session lane. Prefer a single shared active-session/run activity contract consumed by `src/infra/heartbeat-runner.ts`; do not add heartbeat config, change delivery semantics, or consume queued heartbeat/system events before the defer decision. Add focused regression coverage for a registered chat.send/pre-reply-operation heartbeat tick plus existing busy-lane/active-run guards.
