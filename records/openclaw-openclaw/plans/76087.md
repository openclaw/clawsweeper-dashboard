## Summary

Keep open: current main and the latest release still run restart-continuation session-delivery recovery as a one-shot startup timer. Failed delivery attempts persist retry metadata and backoff state, but no scheduler invokes recovery again during normal gateway operation after the immediate drain fails.

## Plan

Fix https://github.com/openclaw/openclaw/issues/76087. Add a generic retry scheduler for restart-continuation session-delivery entries, not Telegram-specific retry policy. The fix should retry entries after isSessionDeliveryEligibleForRetry backoff without requiring another gateway startup, respect maxRetries/idempotency/entriesInProgress/drainInProgress, and move max-retry or permanent failures to failed/ with useful diagnostics. Add regression coverage proving an entry that fails the immediate restart-continuation drain is retried after backoff during normal gateway operation and is not double-delivered when a drain is already in progress. Likely files are src/gateway/server-runtime-services.ts, src/gateway/server-restart-sentinel.ts, src/infra/session-delivery-queue-recovery.ts, and their tests. Add a CHANGELOG.md Fixes entry.
