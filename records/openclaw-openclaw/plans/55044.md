# Coding Plan for openclaw/openclaw#55044: [Bug]: ignored an explicit stop/no-action instruction

Source report: [records/openclaw-openclaw/items/55044.md](records/openclaw-openclaw/items/55044.md)

## Summary

Keep open: current main and latest release still use `abortedLastRun` as both the explicit stop/no-action abort marker and the restart-recovery eligibility marker, so a running session with a resumable transcript tail can still receive the hidden recovery continuation path. The adjacent open PRs are useful context, but they do not specifically close the explicit user-abort versus restart-recovery state split.

## Plan

Fix https://github.com/openclaw/openclaw/issues/55044 by separating explicit user-abort intent from automatic restart-recovery eligibility. The default behavior must keep `/stop` and standalone abort/no-action phrases durable and must not inject a synthetic resume message after an explicit user stop, while true gateway-restart interruptions should still recover. Prefer a small session-entry field or structured marker that records restart-owned recovery state separately from `abortedLastRun`; migrate current restart marking/recovery code to the new marker and keep existing stop/abort semantics intact. Add focused regression coverage showing explicit `/stop` or natural-language no-action abort on a still-running session is not recovered, and true restart-marked sessions still recover. Coordinate with open session-state PRs https://github.com/openclaw/openclaw/pull/83803 and https://github.com/openclaw/openclaw/pull/83176 if their state normalization touches the same fields. Add a changelog entry if the fix changes user-visible stop/recovery behavior. Do not disable main-session restart recovery broadly, do not add a user config knob, and do not make core depend on a channel-specific workaround.
