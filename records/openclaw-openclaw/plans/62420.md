# Coding Plan for openclaw/openclaw#62420: Bug: in-turn auto-compaction can bypass memoryFlush because memoryFlush only runs pre-turn

Source report: [records/openclaw-openclaw/items/62420.md](records/openclaw-openclaw/items/62420.md)

## Summary

Keep open. Current main and the latest release still run memoryFlush only before the main reply turn, while completed in-turn compactions are counted after execution without re-entering any memory-flush path.

## Plan

Fix https://github.com/openclaw/openclaw/issues/62420. The bug is that agents.defaults.compaction.memoryFlush can be bypassed when embedded Pi auto-compaction completes during the main reply or queued follow-up turn after the pre-turn runMemoryFlushIfNeeded call. Add a bounded shared runtime path that detects newly completed compaction cycles after incrementRunCompactionCount and runs or audits the same safe memoryFlush behavior once per new compaction cycle for both main replies and follow-ups. Preserve existing memoryFlush guard semantics for disabled plans, heartbeat, CLI backends, read-only workspaces, and one-flush-per-compaction-cycle metadata. Avoid broad product/config changes. Add regression coverage for main reply and follow-up compaction paths, plus memory metadata updates, and do not regress the adjacent every-other-cycle fix from https://github.com/openclaw/openclaw/pull/51421. Add a changelog entry if the patch changes user-visible behavior.
