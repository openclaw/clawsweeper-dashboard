## Summary

Keep open. Current main and the latest release still have the reported ordering gap: memory flush runs only before the reply turn, while in-turn compaction is counted after the embedded run without re-entering memory flush. The merged adjacent fix at https://github.com/openclaw/openclaw/pull/51421 fixed the every-other-cycle counter bug, but it did not add coverage for compactions that complete during the main or follow-up turn.

## Plan

Fix https://github.com/openclaw/openclaw/issues/62420. Current behavior runs runMemoryFlushIfNeeded only before the reply turn, then records in-turn embedded Pi auto-compactions after the run in main replies and queued follow-ups without re-running or auditing memoryFlush. Add a narrow shared fix so completed auto-compaction cycles trigger the same memoryFlush bookkeeping/write opportunity once per new compaction cycle for both src/auto-reply/reply/agent-runner.ts and src/auto-reply/reply/followup-runner.ts. Prefer a shared helper near session-run-accounting/agent-runner-memory rather than duplicating logic. Do not add a new config option, do not make this a docs-only clarification, and do not broaden into unrelated memory-core/plugin behavior. Add or update focused regression tests for main reply and follow-up post-run compaction accounting, and add a user-facing CHANGELOG.md Fixes entry if this is prepared as a maintainer repair PR.
