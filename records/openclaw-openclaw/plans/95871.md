## Summary

Current main still has the cron timeout/core-result race described in this issue: `executeJobCoreWithTimeout` can return the timeout-shaped cron error when the timeout marker wins, even though the Codex app-server layer can record a native missing tool-result failure underneath. Newer merged Codex fixes reduce related native-tool false positives and model-fallback confusion, but they do not make the cron-visible diagnostic preserve the underlying `missing_tool_result` cause.

## Plan

Fix https://github.com/openclaw/openclaw/issues/95871 by preserving the underlying embedded Codex incomplete-turn / `missing_tool_result` cause in cron-visible diagnostics when an isolated cron agent run times out after the embedded run has already recorded that cause. Work at the cron/agent boundary, likely `src/cron/service/timer.ts`, `src/cron/isolated-agent/run.ts`, and diagnostics helpers, and keep the existing wall-clock timeout guard for true stalls. Do not add new config, provider routing, or broad fallback behavior. Treat https://github.com/openclaw/openclaw/issues/95873 and https://github.com/openclaw/openclaw/pull/95943 as adjacent attribution work rather than the same fix. Add focused regression coverage for the timeout-vs-core-result race and for preserving an embedded fatal error/failure signal in the cron result. Stop and report if reproducing the native Codex `web_search` trigger itself requires live credentials or a broader product decision.
