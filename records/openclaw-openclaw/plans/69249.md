# Coding Plan for openclaw/openclaw#69249: Gateway restart can silently abort an in-flight Discord turn, with no automatic recovery message to the user

Source report: [records/openclaw-openclaw/items/69249.md](records/openclaw-openclaw/items/69249.md)

## Summary

Keep open: current main and latest release still resume restart-aborted main sessions through a hidden `deliver: false` recovery turn, while Discord abort handling exits without a same-channel recovery notice. The new 2026.5.19 user report strengthens the source-level evidence that recovery can write the answer to the transcript without delivering it to Discord.

## Plan

Fix https://github.com/openclaw/openclaw/issues/69249. Current main resumes restart-aborted main sessions via `src/agents/main-session-restart-recovery.ts` with `deliver: false`, so Discord/channel-backed recovery can write the recovered reply into the transcript without delivering it to the original channel. Implement a bounded, channel-agnostic fix that uses canonical persisted external route or delivery context for externally routable main sessions to either send an idempotent visible interruption notice or best-effort resume delivery; preserve hidden/session-only recovery for internal or non-routable sessions. Coordinate with https://github.com/openclaw/openclaw/pull/74687's channel-backed manual-recovery direction and do not add a Discord-only fallback branch unless shared routing cannot express the behavior. Add focused regression coverage for Discord-shaped route metadata, non-routable/internal sessions, and abort/no-visible-notice behavior; add a user-facing CHANGELOG entry.
