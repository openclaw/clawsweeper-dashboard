# Coding Plan for openclaw/openclaw#69249: Gateway restart can silently abort an in-flight Discord turn, with no automatic recovery message to the user

Source report: [records/openclaw-openclaw/items/69249.md](records/openclaw-openclaw/items/69249.md)

## Summary

Keep open: current main still resumes restart-aborted main sessions through a hidden `deliver: false` gateway call, so recovered Discord output can be written to the transcript without reaching the channel. The latest user comment reports the same shape on OpenClaw 2026.5.19, and the related open PRs do not yet close this focused Discord-visible recovery gap.

## Plan

Fix https://github.com/openclaw/openclaw/issues/69249. Current main resumes restart-aborted main sessions via `src/agents/main-session-restart-recovery.ts` with `deliver: false`, so Discord-backed recovery can write the final answer to the transcript without delivering it to the original Discord DM/channel. Implement a narrow, channel-agnostic repair that preserves hidden recovery for internal/non-routable sessions but, when a visible delivery route exists from persisted `deliveryContext` or last-route metadata, either delivers the recovered final answer best-effort or posts an idempotent interruption/recovery notice to the original route. Coordinate with open related PRs, especially https://github.com/openclaw/openclaw/pull/74687 and https://github.com/openclaw/openclaw/pull/84382; do not blindly set every restart recovery call to `deliver: true`, do not add a Discord-only policy branch unless no generic route exists, and do not change plugin SDK/API contracts. Add focused regression coverage for a Discord/channel-backed main session where restart recovery produces a final assistant reply or recovery notice that is routed outward rather than only appended to the transcript. Add a changelog entry if the fix is user-visible.
