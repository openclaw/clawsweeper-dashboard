# Coding Plan for openclaw/openclaw#64030: cron: encode sessionKey when deriving jobId to support channel-native conversation IDs

Source report: [records/openclaw-openclaw/items/64030.md](records/openclaw-openclaw/items/64030.md)

## Summary

Keep open. Current main still rejects slash-bearing cron `session:<id>` targets even though custom session targets are documented and cron job IDs are UUID-backed; the related fix at https://github.com/openclaw/openclaw/pull/64057 was closed unmerged.

## Plan

Fix https://github.com/openclaw/openclaw/issues/64030 so cron accepts channel-native session keys containing `/` or `\` in `session:<id>` targets and when binding `current`. Current main already uses UUID cron job IDs, so do not derive run-log filenames from raw session keys. Treat cron session-target payloads as opaque nonempty, non-NUL runtime session keys; preserve strict run-log `jobId` filename validation in `src/cron/run-log.ts`. Update tests that currently expect path-separator rejection to accept a DingTalk-style key such as `session:agent:main:dingtalk:group:cid3tmd4xb19xjfk/wogxwy2a==`, keep empty/NUL rejection coverage, and keep unsafe run-log job ID rejection coverage. Do not normalize `buildAgentSessionKey`, change persisted global session keys, or add a config option. Add a user-visible changelog fix entry.
