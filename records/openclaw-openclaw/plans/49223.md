# Coding Plan for openclaw/openclaw#49223: WhatsApp inter_session delivery requests can be wrongly suppressed as REPLY_SKIP / ANNOUNCE_SKIP / NO_REPLY

Source report: [records/openclaw-openclaw/items/49223.md](records/openclaw-openclaw/items/49223.md)

## Summary

Keep open: current main and the latest release still expose the source path where `sessions_send` delivery turns are marked as inter-session data, broad skip/silent guidance remains available, and exact control tokens suppress visible delivery before the requested WhatsApp group message can be emitted.

## Plan

Fix https://github.com/openclaw/openclaw/issues/49223 as a narrow `sessions_send` inter-session delivery-precedence bug. Current main marks target turns as `inter_session`, tells the model they are not direct end-user instructions, and gives broad `REPLY_SKIP`/`ANNOUNCE_SKIP`/`NO_REPLY` exits; explicit allowed requests to post a normal message into the target channel can therefore be suppressed. Add a narrow prompt/runtime guard so explicit target-channel delivery requests via `sessions_send` are not treated as coordination-only chatter, while preserving legitimate skip/no-reply behavior for coordination status and ping-pong control. Prefer refining existing `input-provenance` and `sessions-send` prompt/control-token contracts over adding a public typed mode unless the implementation proves a small internal metadata seam is necessary. Add focused regression coverage for explicit delivery intent and forbidden exact control-token outcomes. Do not change channel authorization policy, do not weaken inter-session provenance, and do not make all inter-session turns visible by default. Changelog is expected if user-visible behavior changes.
