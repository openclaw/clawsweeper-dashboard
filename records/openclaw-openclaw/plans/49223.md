# Coding Plan for openclaw/openclaw#49223: WhatsApp inter_session delivery requests can be wrongly suppressed as REPLY_SKIP / ANNOUNCE_SKIP / NO_REPLY

Source report: [records/openclaw-openclaw/items/49223.md](records/openclaw-openclaw/items/49223.md)

## Summary

Keep open: current main has partial hardening for same-session and external-route delivery, but it still lets an explicit `sessions_send` handoff become an `inter_session` turn whose model-facing guidance and A2A control-token handling can suppress a real outbound group message.

## Plan

Fix https://github.com/openclaw/openclaw/issues/49223 by making explicit `sessions_send` / `inter_session` delivery requests into an existing group/channel session prefer normal outbound delivery over `NO_REPLY`, `REPLY_SKIP`, or `ANNOUNCE_SKIP`. Keep the change in core sessions_send/A2A prompt or tool-result metadata; do not add a new public config option or broad public API mode unless source tests prove it is required. Preserve legitimate anti-ping-pong/silent behavior for coordination chatter and parent-owned child sessions. Add focused regression tests around `src/agents/tools/sessions-send-tool.a2a.ts`, `src/agents/tools/sessions-send-helpers.ts`, and the sessions_send gateway/tool tests showing explicit delivery intent posts a normal message while coordination chatter can still skip. Coordinate with currently open sessions_send work such as https://github.com/openclaw/openclaw/pull/75369 if it touches the same trust boundary. Do not edit CHANGELOG.md.
