## Summary

Keep open: current main still has the reported source-level precedence problem. `sessions_send` only accepts a free-text message, marks it as `inter_session`, and the A2A prompts still broadly offer `REPLY_SKIP`/`ANNOUNCE_SKIP` while the runtime suppresses those control replies before visible delivery; the open related PR targets requester-side announce routing, not the WhatsApp explicit-delivery intent described here.

## Plan

Fix the source-reproducible bug where `sessions_send`/`inter_session` delivery requests into a WhatsApp group can be treated as coordination chatter and suppressed as `REPLY_SKIP`, `ANNOUNCE_SKIP`, or `NO_REPLY`. Keep the fix narrow: update the A2A/inter-session prompt guidance and, if needed, control-token handling so explicit requests to post/send/reply with a normal channel message outrank generic ping-pong/silent heuristics when policy allows. Do not add a broad new config option. Do not treat requester-side announce routing from https://github.com/openclaw/openclaw/pull/57697 as the whole fix, but preserve current nested-lane, threadId, provenance, and control-echo protections. Add focused regression coverage that simulates a WhatsApp group target or equivalent channel session receiving an explicit delivery request and forbids `REPLY_SKIP`, `ANNOUNCE_SKIP`, and `NO_REPLY` from suppressing the intended visible send. Add a single-line `CHANGELOG.md` fix entry if the behavior changes user-visible delivery.
