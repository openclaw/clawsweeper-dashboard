# Coding Plan for openclaw/openclaw#97578: [Bug]: OpenClaw CLI should handle node lacks exec approvals capability gracefully

Source report: [records/openclaw-openclaw/items/97578.md](records/openclaw-openclaw/items/97578.md)

## Summary

Keep open: current main still forwards `exec.approvals.node.get/set` directly to the selected node without first checking that the connected node advertises `system.execApprovals.get/set`, so the reported raw TypeError path is still plausible and the requested structured unsupported-capability error is not implemented.

## Plan

Fix the OpenClaw issue where `openclaw approvals get --node <id> --json` can surface a raw TypeError when the connected node lacks `system.execApprovals.get/set`. Add a narrow gateway-side guard in the `exec.approvals.node.get/set` relay path before `nodeRegistry.invoke`, using the connected node session's advertised commands; return a structured unsupported-capability error that is actionable in text and JSON. Keep raw `node.invoke system.execApprovals.*` forbidden, do not add a config option, do not enable remote approvals management for unsupported nodes, and avoid broad Windows node pairing/payload refactors already covered by related maintainer PRs. Add focused regression coverage for missing `system.execApprovals.get` and `system.execApprovals.set`, plus CLI JSON behavior if needed. No CHANGELOG.md edit.
