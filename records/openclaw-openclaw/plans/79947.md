## Summary

Keep open. Current main still reuses a pre-read `startupConfigSnapshotRead` across every in-process restart start callback, and server startup honors that injected snapshot instead of rereading disk, so the reported config-loss bug remains a narrow valid issue.

## Plan

Fix https://github.com/openclaw/openclaw/issues/79947. Current main reads `startupConfigSnapshotRead` once in `runGatewayCommand` and passes the same snapshot into every `runGatewayLoop` start callback, so in-process restarts can boot from stale config and later write back stale source state. Preserve the first-start pre-read optimization, but ensure later in-process restart iterations do not pass `startupConfigSnapshotRead` and therefore reread `openclaw.json` through `loadGatewayStartupConfigSnapshot`. Add focused regression coverage proving the first start can receive the pre-read snapshot while a second in-process restart start does not. Do not change config schema, restart policy, provider/plugin defaults, or unrelated writeback behavior. Add a user-facing changelog fix entry.
