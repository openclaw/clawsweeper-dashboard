## Summary

Keep open. Current main and the latest shipped release still bind the macOS Launch at login toggle to live LaunchAgent health rather than durable user intent, so stale launchd/update state can still make the setting appear off after the user enabled autostart.

## Plan

Fix https://github.com/openclaw/openclaw/issues/39588. In the macOS app, keep Launch at login user intent durable across app restarts and Sparkle/app-bundle updates instead of deriving the UI toggle solely from live launchd health. Keep the fix scoped to apps/macos app LaunchAgent state; do not change CLI gateway ownership or doctor repair behavior. Add a stable UserDefaults-backed intent key, migrate existing enabled launchd state carefully so existing users are not silently disabled, initialize AppState from intent, and reconcile/re-register ai.openclaw.mac with the current Bundle path when intent is enabled but the plist/job is missing or stale. Preserve the current LaunchAgent contract: RunAtLoad=true, no KeepAlive, and disabling autostart removes the plist without killing the running app. Add focused Swift coverage for intent persistence/reconciliation and keep the existing no-KeepAlive test green.
