# Coding Plan for openclaw/openclaw#61041: Feature Request: Auto-resize images exceeding 5MB limit instead of rejecting

Source report: [records/openclaw-openclaw/items/61041.md](records/openclaw-openclaw/items/61041.md)

## Summary

Keep open. Current main and the latest release still reject oversized direct chat image attachments in the shared gateway parser before downstream sanitizer resizing can run; the related iOS PR only covers client-side composer preflight, and broader media-limit configurability remains a separate design track.

## Plan

Implement the narrow server-side path for this issue: oversized valid image attachments handled by src/gateway/chat-attachments.ts should be resized/recompressed before the MAX_IMAGE_BYTES rejection, then accepted when the output fits, while still failing explicitly when it cannot fit safely. Reuse existing image-ops/tool-image primitives or a small shared helper; preserve base64 validation before unsafe decode, verify decoded size, cleanup-on-failure semantics, MIME handling, offloadedRefs/imageOrder behavior, and MAX_IMAGE_INPUT_PIXELS protection. Add or update regression coverage in src/gateway/chat-attachments.test.ts and adjacent image sanitizer/image-ops tests as needed. Do not add a new public media.image config contract in this PR; leave that to https://github.com/openclaw/openclaw/issues/67031, and do not change the iOS client-side scope owned by https://github.com/openclaw/openclaw/pull/73710. Add a changelog entry if the final PR is user-facing.
