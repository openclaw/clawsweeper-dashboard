## Summary

Keep open. Current main and the latest v2026.5.19 release still reject oversized shared gateway image attachments before any server-side resize/recompress path runs; the iOS client-side image processor is useful related work but does not cover the shared gateway parser path reported here.

## Plan

Fix https://github.com/openclaw/openclaw/issues/61041 by adding bounded server-side resize/recompression for oversized image chat attachments in the shared gateway parser. Keep the change scoped to the parseMessageWithAttachments path and tests: when an image is above MAX_IMAGE_BYTES but within the existing general chat attachment ceiling, attempt to resize/recompress through the existing media image-ops/media-services abstraction, recompute bytes/MIME/base64, and then continue through the existing inline/offload flow. Preserve base64 validation, MIME sniffing, cleanup on failed offload, text-only offload behavior, general maxBytes rejection, and MAX_IMAGE_INPUT_PIXELS safety; do not import sharp directly into the gateway parser and do not add a new media.image config surface because that belongs to https://github.com/openclaw/openclaw/issues/67031. Update regression coverage so the current oversized-image rejection test proves a reducible image is accepted/resized, and add a failure case for an image that cannot be reduced safely. Include a changelog entry if the landing workflow requires one for this user-visible behavior change.
