## Summary

Keep open. Current main still has the source-level failure path: same-session Gateway agent runs serialize through the session lane, lane-wait facts are emitted only as diagnostics, and the CLI waits on a generic final Gateway response or timeout fallback without actionable lane-busy metadata.

## Plan

Fix https://github.com/openclaw/openclaw/issues/54622. When a Gateway-backed `openclaw agent` dispatch waits behind an active same-session lane long enough to exceed the wait/CLI timeout path, surface an actionable lane-busy or lane-wait timeout error/result to the CLI and JSON path with lane, waitedMs, queueAhead and/or activeAhead, and operator guidance. Keep same-session serialization intact, do not add a new concurrency policy or `--force-queue` mode unless maintainers explicitly choose it, and do not classify unrelated Gateway transport timeouts as lane busy. Likely boundary: `src/process/command-queue.ts`, `src/process/command-queue.types.ts`, `src/agents/pi-embedded-runner/run.ts`, `src/gateway/server-methods/agent.ts`, `src/gateway/protocol/schema/agent.ts`, `src/commands/agent-via-gateway.ts`, focused tests, `docs/cli/agent.md`, and `CHANGELOG.md`. Add regression coverage for a queued same-session agent call and preserve the generic gateway timeout fallback for non-lane failures.
