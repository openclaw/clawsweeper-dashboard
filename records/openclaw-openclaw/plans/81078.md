## Summary

Keep open: current main still suppresses text-only final payloads after any successful block stream without proving that the final text was already sent, so the reported WhatsApp-visible truncation remains actionable.

## Plan

Fix https://github.com/openclaw/openclaw/issues/81078. In `buildReplyPayloads`, stop dropping all text-only final payloads merely because block streaming succeeded. Suppress a final payload only when `blockReplyPipeline.hasSentPayload(payload)` or `directlySentBlockKeys` proves that exact/rendered final payload was already delivered, while preserving existing behavior for post-stream error payloads, unsent media after streamed text, block-sent media dedupe, messaging-tool dedupe, TTS/media payloads, and heartbeat/silent filtering. Add or update focused regression coverage in `src/auto-reply/reply/agent-runner-payloads.test.ts` for the reported case: `didStream()=true`, `isAborted()=false`, `hasSentPayload()=false`, text final payload should be delivered. Keep WhatsApp config behavior unchanged; only update `extensions/whatsapp/src/auto-reply/monitor/inbound-dispatch.test.ts` if the repair needs channel-path coverage. Add a user-facing changelog entry for the final-reply delivery fix.
