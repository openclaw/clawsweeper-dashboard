## Summary

Keep open: current main still contains the reported blanket successful-stream final-payload suppression path, and the focused test suite still asserts that an unsent final text payload is dropped after `didStream()` with no abort.

## Plan

Fix https://github.com/openclaw/openclaw/issues/81078. In `buildReplyPayloads`, stop dropping all final payloads solely because block streaming succeeded; suppress a final payload only when `blockReplyPipeline.hasSentPayload(payload)` or `directlySentBlockKeys` proves that same payload was already delivered. Preserve current behavior for unsent media after streamed text, already block-sent media, error payloads, silent turns, TTS/audio voice payloads, and messaging-tool dedupe. Add/update focused regression coverage in `src/auto-reply/reply/agent-runner-payloads.test.ts` for a partial streamed text followed by a longer distinct final text where `hasSentPayload` is false and the final is delivered. Add a user-facing changelog fix entry unless maintainer landing handles it. Do not add a new config option, do not change provider/channel retry policy, and do not broaden WhatsApp-specific behavior beyond the generic reply-payload suppression path.
