## Summary

Keep open: current main still has the successful-block-stream path that suppresses text-only final payloads without checking whether that final text was actually sent, and the focused test suite currently codifies that loss-of-final behavior.

## Plan

Fix https://github.com/openclaw/openclaw/issues/81078. In `src/auto-reply/reply/agent-runner-payloads.ts`, stop dropping all text-only final payloads merely because block streaming succeeded; suppress a final payload only when `blockReplyPipeline.hasSentPayload(payload)` or `directlySentBlockKeys` proves the same final payload was delivered. Preserve current behavior for errors, unsent media after streamed text, already-sent media stripping, TTS/audio-as-voice handling, messaging-tool dedupe, and aborted-stream fallback. Update focused regression coverage in `src/auto-reply/reply/agent-runner-payloads.test.ts`, and add/update WhatsApp-facing coverage in `extensions/whatsapp/src/auto-reply/monitor/inbound-dispatch.test.ts` if needed to prove partial block delivery followed by a longer completed final reply is delivered. Add a user-facing changelog fix entry. Do not add a new config option or change WhatsApp blockStreaming semantics.
