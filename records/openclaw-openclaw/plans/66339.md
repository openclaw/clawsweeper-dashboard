# Coding Plan for openclaw/openclaw#66339: memory search can hit QMD SQLite lock contention during normal runtime

Source report: [records/openclaw-openclaw/items/66339.md](records/openclaw-openclaw/items/66339.md)

## Summary

Keep open. Current main still coordinates QMD update work only inside one Node process and QMD searches only wait on the same manager instance, while CLI/startup paths can create separate transient managers that race a running gateway against the same QMD SQLite store.

## Plan

Fix https://github.com/openclaw/openclaw/issues/66339 as a focused QMD concurrency bug. Add bounded per-agent cross-process coordination in `extensions/memory-core` so gateway/background QMD update or embed maintenance and foreground `openclaw memory search`/index/doc lookup do not race the same QMD SQLite store and surface `SQLITE_BUSY` during normal use. Preserve the current public backend/config contract; do not add a new config knob unless the existing internal timeout/lock helpers require one, and do not fold this into the separate remote embedding transport issue at https://github.com/openclaw/openclaw/issues/71784. Add regression coverage for separate transient managers/process-like ownership, update QMD docs only if user-visible behavior changes, and add a changelog entry because this affects user-visible memory reliability.
