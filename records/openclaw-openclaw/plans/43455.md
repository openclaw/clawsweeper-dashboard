# Coding Plan for openclaw/openclaw#43455: Browserbase integration broken: static cdpUrl returns 404, needs session API

Source report: [records/openclaw-openclaw/items/43455.md](records/openclaw-openclaw/items/43455.md)

## Summary

Keep open. Current main still documents Browserbase as a static bare WebSocket CDP profile and still connects through the generic CDP attach path, while Browserbase's current public API documents session creation via REST and a returned `connectUrl`.

## Plan

Fix https://github.com/openclaw/openclaw/issues/43455 in one focused browser-plugin PR. Current main documents Browserbase as `wss://connect.browserbase.com?apiKey=...`, but Browserbase's current Sessions API creates a session with `POST https://api.browserbase.com/v1/sessions` and returns a `connectUrl`. Implement Browserbase session lifecycle in the browser plugin, not core: detect the documented Browserbase profile shape or add a narrowly owned helper, create a session before CDP attach using the API key already supplied by the Browserbase config path, connect Playwright/CDP to the returned `connectUrl`, and release only sessions OpenClaw created. Preserve generic Browserless/remote CDP behavior and SSRF/redaction behavior. Do not add a broad provider framework, do not log raw API keys or returned signed URLs, and do not change unrelated CDP fallback semantics. Update focused tests, `docs/tools/browser.md`, and `CHANGELOG.md`. Treat https://github.com/openclaw/openclaw/issues/78337 as adjacent SecretRef context, not part of this fix unless a tiny interface choice is unavoidable; stop for maintainer review if the fix requires a new public config contract or SecretRef semantics.
