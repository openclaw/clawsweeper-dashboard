# Coding Plan for openclaw/openclaw#43455: Browserbase integration broken: static cdpUrl returns 404, needs session API

Source report: [records/openclaw-openclaw/items/43455.md](records/openclaw-openclaw/items/43455.md)

## Summary

Keep open. Current main still advertises and routes Browserbase as a static generic WebSocket CDP profile, while Browserbase’s current docs describe creating a session through the Sessions API and using the returned connection URL. ([docs.browserbase.com](https://docs.browserbase.com/reference/api/create-a-session?utm_source=openai))

## Plan

Fix Browserbase browser profile support for https://github.com/openclaw/openclaw/issues/43455. Implement Browserbase-specific session lifecycle in the browser plugin only: create a Browserbase session through the current official Browserbase Sessions API before attaching, use the returned CDP connect URL, and request release only for sessions OpenClaw created. Preserve existing generic HTTP, `/json/version`, direct ws/wss CDP behavior, SSRF checks, and redaction. Do not add Browserbase defaults or policy to core. Coordinate with https://github.com/openclaw/openclaw/issues/78337 rather than solving credential-at-rest concerns by requiring long-lived API keys in cleartext `cdpUrl`. Add mocked API-contract tests, update `docs/tools/browser.md`, and add a `CHANGELOG.md` fix entry; live Browserbase validation is preferred only with redacted credentials.
