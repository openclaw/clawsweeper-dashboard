# Coding Plan for openclaw/openclaw#43455: Browserbase integration broken: static cdpUrl returns 404, needs session API

Source report: [records/openclaw-openclaw/items/43455.md](records/openclaw-openclaw/items/43455.md)

## Summary

Keep open. Current main still documents Browserbase as a static bare WebSocket CDP profile and still routes that URL through generic CDP attach, while Browserbase’s current docs describe creating a session through the Sessions API and using the returned `connectUrl`. ([docs.browserbase.com](https://docs.browserbase.com/reference/api/create-a-session?utm_source=openai))

## Plan

Fix https://github.com/openclaw/openclaw/issues/43455. In the browser plugin, add Browserbase session lifecycle support for the documented Browserbase profile without changing generic remote CDP behavior. Prefer using the existing documented `wss://connect.browserbase.com?apiKey=...` input as the compatibility trigger: create a Browserbase session via `POST https://api.browserbase.com/v1/sessions` with `X-BB-API-Key`, connect Playwright/CDP to the returned `connectUrl`, cache/reuse it consistently with existing remote CDP session handling, and clean up only sessions OpenClaw created according to Browserbase lifecycle rules. Preserve externally supplied Browserbase `connectUrl`/signed URLs as direct CDP endpoints. Do not leak API keys, signing keys, or returned URLs in errors, config responses, logs, snapshots, or tests. Update `docs/tools/browser.md`, focused tests, and `CHANGELOG.md`; stop for maintainer input if the fix requires a new public config contract beyond the existing documented URL.
