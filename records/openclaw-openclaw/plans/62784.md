# Coding Plan for openclaw/openclaw#62784: [Bug] Multi-agent Slack: concurrent Socket Mode connections cause pong timeout storm and gateway SIGKILL

Source report: [records/openclaw-openclaw/items/62784.md](records/openclaw-openclaw/items/62784.md)

## Summary

Keep this open. Current main has useful Slack Socket Mode mitigations, but it still hands off Slack account monitors without awaiting or staggering the socket startup, and the Slack path still reaches `app.start()` without per-account startup desynchronization.

## Plan

Fix https://github.com/openclaw/openclaw/issues/62784. Add a Slack-owned, abortable startup desynchronization before Socket Mode opens when more than one enabled Slack Socket Mode account is configured. Keep the fix inside the Slack plugin if possible; do not add a new config option, do not change global channel startup policy, and do not take on the broader event-loop/worker-thread work tracked by https://github.com/openclaw/openclaw/issues/58519 or the global startup policy in https://github.com/openclaw/openclaw/pull/78437 unless maintainers explicitly choose that direction. Preserve current native reconnect behavior (`autoReconnectEnabled: true`), the 15s default Socket Mode ping timeout, non-recoverable auth handling, and existing reconnect backoff. Add focused regression coverage proving multi-account Slack Socket Mode startup is desynchronized and abort-safe before `app.start()` is called. Add a single-line changelog Fixes entry because this is user-visible reliability work.
