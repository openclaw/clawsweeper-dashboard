# Coding Plan for openclaw/openclaw#33413: Slack: Show tool-level progress in assistant thread status

Source report: [records/openclaw-openclaw/items/33413.md](records/openclaw-openclaw/items/33413.md)

## Summary

Keep open: current main and the latest release still use Slack's native assistant-thread status only for static reply start/clear, while tool-start events update reactions and draft-preview progress but do not update `assistant.threads.setStatus`. The closed prototype at https://github.com/openclaw/openclaw/pull/33420 confirms a viable Slack-local wiring shape, and the adjacent open config request at https://github.com/openclaw/openclaw/issues/67129 does not supersede this tool-level status behavior.

## Plan

Implement the Slack native assistant-thread tool-status feature for https://github.com/openclaw/openclaw/issues/33413. Wire the existing Slack `replyOptions.onToolStart` path in `extensions/slack/src/monitor/message-handler/dispatch.ts` so that, when the normal typing/status lifecycle is active and a status thread exists, Slack calls `ctx.setSlackThreadStatus` with a compact human-readable label for the current tool instead of leaving the native status at only `is typing...`. Reuse existing tool-display/progress helpers where practical instead of duplicating a large tool map, dedupe repeated labels to avoid status spam, keep status cleanup reliable on success/error/idle, and do not change Slack draft-preview progress defaults or introduce the custom `loading_messages` config tracked by https://github.com/openclaw/openclaw/issues/67129. Add focused Slack tests that prove start status, tool status update, repeated-label dedupe, missing-thread no-op, and final clear behavior. Add a CHANGELOG entry if the repo policy requires one for this user-visible Slack feature.
