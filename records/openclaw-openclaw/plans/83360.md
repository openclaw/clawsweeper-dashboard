# Coding Plan for openclaw/openclaw#83360: [Bug]: auto-update can never succeed under systemd — updater is spawned as a child of the gateway it needs to restart

Source report: [records/openclaw-openclaw/items/83360.md](records/openclaw-openclaw/items/83360.md)

## Summary

Keep open: current main still has the conflicting updater path. The gateway auto-update scheduler launches the CLI updater as a normal child of the gateway, while the update CLI still blocks package updates when the active gateway PID is in the updater ancestry; the open marker-stripping PR is related but does not by itself solve this systemd process-tree failure.

## Plan

Fix the managed-gateway auto-update failure reported at https://github.com/openclaw/openclaw/issues/83360 without weakening the manual package-update safety guards from https://github.com/openclaw/openclaw/pull/75819. Route `runAutoUpdateCommand` through a supervisor-owned managed-service handoff or equivalent systemd-safe path that runs outside the gateway process tree, account for the related marker-env issue at https://github.com/openclaw/openclaw/issues/78492 and open PR https://github.com/openclaw/openclaw/pull/78494, add focused regression coverage for the auto-update path and the preserved guard behavior, add a user-facing changelog fix entry, and stop if the implementation would require a new config knob or a product-policy change. Do not make `openclaw update` generally ignore gateway service markers or ancestor PID checks.
