# Coding Plan for openclaw/openclaw#83360: [Bug]: auto-update can never succeed under systemd — updater is spawned as a child of the gateway it needs to restart

Source report: [records/openclaw-openclaw/items/83360.md](records/openclaw-openclaw/items/83360.md)

## Summary

Keep open: current main still has the reported conflict between the gateway auto-update scheduler and the updater's gateway-ancestry safety guard, so auto-update remains unable to complete from a standard systemd-managed gateway process tree.

## Plan

Fix https://github.com/openclaw/openclaw/issues/83360. Current main's gateway auto-update scheduler launches `openclaw update --yes --channel ... --json` as a gateway child, and the update CLI correctly refuses package updates when the active gateway PID is in the updater ancestry. Implement a service-safe auto-update handoff outside the live gateway process tree, preferably by reusing or extending the existing managed-service update handoff pattern rather than weakening the manual `openclaw update` ancestry guard. Preserve `KillMode=control-group` and the safety refusal for operator/manual updates from gateway-owned children. Reconcile with the narrower marker-env PR https://github.com/openclaw/openclaw/pull/78494; do not rely only on clearing service markers because the PID ancestry guard remains. Add focused regression coverage for the auto-update scheduler path and the guarded CLI path, update docs/changelog if user-facing behavior changes, and stop if the repair would require a new config knob or maintainer product decision.
