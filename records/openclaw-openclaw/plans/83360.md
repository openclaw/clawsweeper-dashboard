# Coding Plan for openclaw/openclaw#83360: [Bug]: auto-update can never succeed under systemd — updater is spawned as a child of the gateway it needs to restart

Source report: [records/openclaw-openclaw/items/83360.md](records/openclaw-openclaw/items/83360.md)

## Summary

Keep open: current main still has the reported conflict. The gateway auto-update loop launches the updater from inside the gateway process tree, while the update CLI still refuses package updates when the managed gateway PID is an ancestor; the related marker-env issue and PR do not by themselves fix this systemd ancestry path.

## Plan

Fix https://github.com/openclaw/openclaw/issues/83360. Current main's gateway auto-update scheduler launches `openclaw update --yes --channel ... --json` via `runCommandWithTimeout`, so under the standard systemd user service the updater remains in the gateway process tree and hits the CLI ancestry guard before package install. Implement a service-safe auto-update handoff for managed services, preferably by reusing/extending the existing managed-service update handoff or by using a systemd-owned transient unit/oneshot for Linux. Preserve the manual `openclaw update` guard in `src/cli/update-cli/update-command.ts`; do not make generic `runCommandWithTimeout` detached by default, and do not weaken `getSelfAndAncestorPidsSync` blocking. Coordinate with the marker-env path in https://github.com/openclaw/openclaw/issues/78492 and https://github.com/openclaw/openclaw/pull/78494, because that overlap is related but not sufficient for the PID-ancestry failure. Add focused regression coverage and a changelog entry if the behavior change is user-visible. Stop and ask for maintainer review if the fix would require a new public config option or a broader update architecture rewrite.
