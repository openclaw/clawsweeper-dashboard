# Coding Plan for openclaw/openclaw#83360: [Bug]: auto-update can never succeed under systemd — updater is spawned as a child of the gateway it needs to restart

Source report: [records/openclaw-openclaw/items/83360.md](records/openclaw-openclaw/items/83360.md)

## Summary

Keep open: current main still launches the gateway auto-updater as a normal gateway child process, while the update CLI still blocks package updates when the active gateway PID is in the updater ancestry. The related open marker-stripping PR is useful context, but it does not solve this systemd process-tree failure mode.

## Plan

Fix the documented gateway auto-update bug where `update.auto.enabled=true` under the standard systemd user service launches `openclaw update` as a gateway child and the CLI ancestry guard rejects it. Use the existing managed-service update handoff pattern where possible so the auto-update package replacement runs outside the live gateway process tree; preserve the manual gateway-ancestry/package-update refusal and avoid weakening `openclaw update` safety checks. Coordinate with the related marker-inheritance work in https://github.com/openclaw/openclaw/issues/78492 and https://github.com/openclaw/openclaw/pull/78494 rather than duplicating a conflicting fix. Add focused regression coverage for the auto-update handoff/default path and update docs/changelog only if behavior text changes. Validate with targeted Vitest plus real Linux systemd user-service package-update proof using `update.auto.enabled=true` and a newer package target.
