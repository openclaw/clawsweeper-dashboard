## Summary

Keep open: current main still has the reported conflict. The gateway scheduler runs the updater as a gateway child, while the update CLI refuses package updates when the active gateway PID is in the updater ancestry; the related marker-stripping PR is useful context but does not solve the systemd ancestry failure.

## Plan

Fix the managed-service auto-update failure described in https://github.com/openclaw/openclaw/issues/83360, while reconciling the related marker-inheritance thread https://github.com/openclaw/openclaw/issues/78492 and open PR https://github.com/openclaw/openclaw/pull/78494. Preserve the manual `openclaw update` safety guard for updater processes owned by the live gateway; do not make gateway-owned manual package updates mutate the install tree. Move configured gateway auto-update onto a service-safe handoff, preferably reusing or extending the existing managed-service update handoff instead of adding an unrelated fallback stack. Add focused regression tests for auto-update child ownership/handoff, marker env handling, and the CLI ancestry block. Update docs/changelog only if user-visible behavior changes. Validate with the narrow Vitest files first, then require real Linux systemd user-service package-update proof with `update.auto.enabled=true` before landing.
