## Summary

Keep open. Current main still permits the stale-token reconnect loop when `AUTH_TOKEN_MISMATCH` happens without an actual cached device-token retry, and the shipped `chat.history` and Slack Socket Mode mitigations do not close that remaining auth-loop bug.

## Plan

Fix https://github.com/openclaw/openclaw/issues/72139 by tightening browser and Node gateway reconnect gating for `AUTH_TOKEN_MISMATCH`. On a trusted endpoint, reconnect automatically only when a cached-device-token retry has actually been queued; if no stored device token is available, stop the reconnect loop and surface the auth failure through the existing close/pause callbacks. Preserve the one-shot cached-device-token fallback, startup `UNAVAILABLE` reconnect behavior, pairing-required wait retry behavior, and existing Slack Socket Mode timeout/config mitigations. Add/update regression tests for browser no-cached-token mismatch, Node no-cached-token mismatch, successful one-shot cached-token retry, and exhausted retry behavior. Do not change Slack config semantics or solve the separate WebUI `chat.history` timeout PR in this fix. Add a concise `CHANGELOG.md` fix entry because this is user-visible.
