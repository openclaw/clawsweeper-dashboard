# Coding Plan for openclaw/openclaw#71140: Telegram outbound replies leak/drop react/media directives instead of applying native reactions/GIFs

Source report: [records/openclaw-openclaw/items/71140.md](records/openclaw-openclaw/items/71140.md)

## Summary

Keep open. Current main and the latest release still lack the reaction directive path: shared directive parsing does not recognize `[[react...]]`, outbound normalization does not produce Telegram reaction channel data, and Telegram delivery/adapter paths do not execute reaction-only payloads. The media/GIF side is partly supported already, with a separate open media-ordering PR, but the native reaction leak/drop remains a distinct bug.

## Plan

Fix https://github.com/openclaw/openclaw/issues/71140 with a focused replacement PR. Parse `[[react:...]]` and `[[react_to_current:...]]` in the shared reply directive path, strip those directives from visible text, preserve the reaction as structured outbound data such as `channelData.telegram.reaction`, allow reaction-only Telegram replies when a valid reply target exists, and call the existing Telegram reaction primitive from both the Telegram bot delivery path and `sendTelegramPayloadMessages` without changing unrelated media/GIF behavior. Use https://github.com/openclaw/openclaw/pull/71141 as useful source context but fix the reaction option type contract noted there. Do not duplicate the separate MEDIA-ordering work in https://github.com/openclaw/openclaw/pull/44281 except where tests need to prove reactions compose with existing media sends. Add focused parser/outbound/Telegram adapter and delivery tests, add a user-facing `CHANGELOG.md` fix entry, and provide redacted live Telegram proof showing reaction-only and text-plus-reaction replies.
