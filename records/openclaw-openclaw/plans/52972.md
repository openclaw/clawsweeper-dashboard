# Coding Plan for openclaw/openclaw#52972: Bug: Incorrect 'I did not schedule a reminder' note appended to messages after scheduling cron reminders

Source report: [records/openclaw-openclaw/items/52972.md](records/openclaw-openclaw/items/52972.md)

## Summary

Keep open: current main and the latest release still only count native `cron` tool adds as `successfulCronAdds`; a successful `openclaw cron add --at` through CLI/shell scheduling can create the job without setting the same-turn signal that suppresses the contradictory reminder note.

## Plan

Fix https://github.com/openclaw/openclaw/issues/52972 by adding a narrow verified same-turn scheduling signal for successful cron creation through assistant shell/CLI paths. Current main only increments `successfulCronAdds` for native `toolName === "cron"` add actions in `src/agents/pi-embedded-subscribe.handlers.tools.ts` and `extensions/codex/src/app-server/dynamic-tools.ts`, while `openclaw cron add --at` creates jobs through Gateway `cron.add` and can leave the reply guard in `src/auto-reply/reply/agent-runner.ts` thinking no reminder was scheduled. Use https://github.com/openclaw/openclaw/pull/53024 as prior art, but do not use a loose raw substring match that treats `echo cron add` as proof of scheduling; require the safest available success evidence from the CLI/tool result path. Add focused regression tests for successful exec/bash CLI cron add suppressing the note, unrelated successful commands not incrementing, and failed cron-add commands not incrementing. Keep the change scoped: do not add a config option, broadly strip the note from delivered text, or change reminder-commitment regex policy. Add a user-visible changelog entry during landing if the maintainer workflow requires it.
