# Coding Plan for openclaw/openclaw#59470: Bug: openclaw gateway call times out for plugin gateway methods even when the method executes successfully

Source report: [records/openclaw-openclaw/items/59470.md](records/openclaw-openclaw/items/59470.md)

## Summary

Keep open. Current `main` and latest release `v2026.5.19` still have the source-backed timeout path: plugin gateway handlers are awaited, but a returned payload is not converted into a Gateway RPC response, and no linked closing PR or canonical duplicate owns this exact bug.

## Plan

Fix https://github.com/openclaw/openclaw/issues/59470. Plugin gateway RPC methods that return a value without calling `respond(...)` execute but leave `openclaw gateway call` waiting until timeout. In current `main`, `handleGatewayRequest` invokes the handler inside `withPluginRuntimeGatewayRequestScope` but ignores the return value; the registry stores handlers raw; the WebSocket path sends `{ type: "res" }` only through the passed `respond` callback. Add a backward-compatible adapter in `src/gateway/server-methods.ts` so a non-undefined returned value sends one success response when `respond` has not already been called. Preserve existing callback-style handlers, accepted/final or multi-response behavior, rate-limited/startup-gated paths, and method scope/registry behavior. Update the public `GatewayRequestHandler` type and plugin SDK docs as needed, add regression coverage for both return-style and explicit `respond(...)` plugin gateway handlers, and include a single `CHANGELOG.md` fix entry.
