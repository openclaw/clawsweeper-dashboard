## Summary

Keep open: current main and the latest release still have a source-reproducible Slack outbound mirror path where an inherited child reply timestamp can win over the conversation's existing thread session and create a fresh mirror-only session entry.

## Plan

Fix https://github.com/openclaw/openclaw/issues/96535 with a narrow Slack outbound mirror routing change. Source evidence shows inherited same-thread Slack replies can carry a child `replyTo` timestamp into `prepareOutboundMirrorRoute`; Slack currently skips auto-thread recovery when `replyToId` is already a timestamp, then `buildThreadAwareOutboundSessionRoute` chooses `replyToId` before `threadId` or the recovered current session. Make Slack-owned routing canonicalize inherited same-thread mirror sessions to the root/current thread session while preserving explicit `replyTo`/`threadId` targets, `topLevel`/`threadId:null` opt-outs, D... DM canonicalization, MPIM behavior, and ordinary top-level channel-session policy. Prefer changing Slack threading/route inputs or Slack-specific precedence over a broad generic precedence change unless tests prove the generic contract should change. Add regression coverage for a Slack channel thread where current session is `agent:<agent>:slack:channel:<cid>:thread:<rootTs>` and inherited child reply timestamp must not create `:thread:<childTs>`. Do not edit `CHANGELOG.md`.
