## Summary

Keep open: the report matches a current source-level gap in Linux gateway duplicate-service handling. Current `main` has partial guardrails for system-level services, but the canonical user-plus-system systemd pair can still evade the duplicate scan and then trigger the stale-PID cleanup loop described by the reporter.

## Plan

Fix the Linux gateway duplicate-supervisor bug reported in https://github.com/openclaw/openclaw/issues/79375. Current `main` filters out the canonical user `openclaw-gateway.service` from extra-service scans and only checks system-level gateway services when the user service is missing, so an upgrade can leave both user and system units supervising the same port. Add a focused repair that detects the canonical user+system systemd pair for the same profile/port, surfaces it in `openclaw doctor` and deep status/diagnostics, and offers confirmed doctor/update repair that stops/disables and backs up or clearly reports the stale user unit. Do not add a new config option or restart-via flag, do not silently delete units outside confirmed repair policy, and do not wholesale adopt the stale broader approaches from https://github.com/openclaw/openclaw/pull/57276 or https://github.com/openclaw/openclaw/pull/68909. Add regression coverage for both scopes existing at once and, if touched, for stale-PID cleanup not killing a recognized managed peer. Add a user-facing changelog entry because this fixes a Linux upgrade/runtime failure.
