# Coding Plan for openclaw/openclaw#79375: Upgrade leaves stale user-level systemd unit, dueling services kill each other on Linux

Source report: [records/openclaw-openclaw/items/79375.md](records/openclaw-openclaw/items/79375.md)

## Summary

Keep open: current main still has the reported Linux systemd split-brain gap. The default user unit and system unit can coexist without being reported as an extra service, and the startup stale-process cleanup can still kill a sibling supervised gateway on the same port.

## Plan

Fix https://github.com/openclaw/openclaw/issues/79375. Current main filters canonical `openclaw-gateway.service` out of `findExtraGatewayServices` and only checks `findSystemGatewayServices()` when the user service is missing, so a stale loaded user unit plus a system unit is not detected and service-mode stale PID cleanup can make them fight. Add canonical Linux user+system systemd duplicate detection for the same gateway profile/unit/port in doctor/status, surface clear cleanup guidance, and offer confirmed doctor cleanup that stops/disables and backs up/removes the stale user unit through existing service-repair policy. Do not broaden system-scoped lifecycle support, automatic sudo management, or the open PR https://github.com/openclaw/openclaw/pull/57276 scope. Add focused regression coverage for the loaded user+system canonical pair, active/inactive handling, cleanup output, and stale PID behavior if changed. Add a user-facing changelog entry if behavior changes.
