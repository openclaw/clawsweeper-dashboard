# Coding Plan for openclaw/openclaw#39476: A2A sessions_send: target agent can call sessions_send back, causing duplicate messages

Source report: [records/openclaw-openclaw/items/39476.md](records/openclaw-openclaw/items/39476.md)

## Summary

Keep open. Current main and the latest release still expose requester session identity to A2A target prompts, still run nested `sessions_send` reply/announce turns, and still have no scoped deny/stub preventing the target agent from calling `sessions_send` back to the requester.

## Plan

Fix https://github.com/openclaw/openclaw/issues/39476 by adding a scoped internal guard for reverse `sessions_send` during existing A2A nested runs. The bug is that the target agent receives requester session identity and can call `sessions_send` back to the requester while the original tool already returns the target reply, causing duplicate channel messages. Keep the fix bounded to `sessions_send` A2A target/reply/announce nested execution; do not add a public dispatch-only mode, do not remove the documented A2A conversation flow globally, and coordinate with but do not depend on https://github.com/openclaw/openclaw/pull/50720. Prefer a well-formed stub/error result such as `A2A_REVERSE_CALL_BLOCKED` over silently removing the tool, so model tool-use structure remains valid and attempts can be observed. Add focused regression coverage showing the target-side nested run cannot reverse-call the requester while normal top-level `sessions_send` and inline result delivery still work. Update docs only if the user-visible behavior or error text changes.
