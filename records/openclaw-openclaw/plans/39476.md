# Coding Plan for openclaw/openclaw#39476: A2A sessions_send: target agent can call sessions_send back, causing duplicate messages

Source report: [records/openclaw-openclaw/items/39476.md](records/openclaw-openclaw/items/39476.md)

## Summary

Keep open. Current main and v2026.5.20 still expose requester session context to A2A target prompts, run reply/announce nested turns with `sourceTool: "sessions_send"`, and construct `sessions_send` for normal non-embedded agent toolsets without a scoped reverse-call guard.

## Plan

Fix the `sessions_send` A2A duplicate-message bug where a target agent can call `sessions_send` back to the requester during nested A2A processing. Add a scoped internal guard so target/reply/announce nested runs sourced by `sessions_send` cannot use `sessions_send` to reverse-call the requester; preserve inline tool-return delivery, existing peer A2A reply/announce behavior, and the requester-owned ACP/native child skip. Do not add a public dispatch/private-delivery schema option for this bug, do not delete the A2A ping-pong/announce lifecycle wholesale, and do not change unrelated channel delivery behavior. Add regression coverage that proves the nested A2A target toolset denies or omits `sessions_send` while ordinary non-A2A sessions_send remains available. Add a maintainer-owned changelog fix entry if the repair changes user-visible A2A behavior.
