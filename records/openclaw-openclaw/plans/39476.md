# Coding Plan for openclaw/openclaw#39476: A2A sessions_send: target agent can call sessions_send back, causing duplicate messages

Source report: [records/openclaw-openclaw/items/39476.md](records/openclaw-openclaw/items/39476.md)

## Summary

Keep open. Current main still exposes the requester session key to the A2A target and runs target/requester A2A steps without a scoped `sessions_send` denial, so the reverse-send duplicate path remains source-reproducible; the open dispatch-mode request is adjacent but does not replace this bug.

## Plan

Fix the A2A reverse-send duplicate bug for https://github.com/openclaw/openclaw/issues/39476. Add an internal scoped tool-availability guard so a `sessions_send` target run, and any nested A2A reply/announce turns that are part of that flow, cannot call `sessions_send` back to the requester. Preserve current conversation-mode reply-back and target announce behavior unless the only change is blocking reverse tool calls; do not implement the separate dispatch/handoff mode from https://github.com/openclaw/openclaw/issues/44309. Expected implementation likely needs an internal agent-run/tool-deny field or equivalent runtime parameter carried from `createSessionsSendTool` through the gateway/agent runner into tool construction, plus regression tests proving the target A2A run does not expose `sessions_send` while ordinary sessions still do. Add or update docs only if the behavior becomes user-visible, and add a user-facing `CHANGELOG.md` Fixes entry if repo policy requires it.
