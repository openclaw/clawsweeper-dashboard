## Summary

Keep open. Current main and the latest release still classify `browser.profiles.*` under the browser plugin's broad `browser` restart prefix, so hot-mode `config.patch` can still schedule SIGUSR1; the linked `novita-sandbox` skill is useful real-world context, but this remains a core/browser-plugin reload metadata bug rather than ClawHub-only work.

## Plan

Fix https://github.com/openclaw/openclaw/issues/43803. Current main still classifies `browser.profiles.*` under `browserPluginReload.restartPrefixes: ["browser"]`; in hot mode, `config.patch` builds a restart-required plan and schedules SIGUSR1. Keep the fix owner-bounded: prefer granular browser plugin reload metadata using the existing `restartPrefixes`/`hotPrefixes`/`noopPrefixes` contract so profile/default-profile changes that the browser runtime can refresh avoid gateway restarts, while structural browser service settings still restart. Add/update regression coverage for `browser.profiles.<name>.cdpUrl` and `browser.defaultProfile` through `buildGatewayReloadPlan` and the `config.patch` restart scheduling path. Do not add a new config option, do not hard-code a core browser allowlist if plugin metadata can express it, and preserve existing `commands.restart`, reload-mode, sentinel, and shared-auth behavior. Add a user-facing `CHANGELOG.md` fix entry unless maintainers decide this is internal-only.
