# Coding Plan for openclaw/openclaw#43803: [BUG] config.patch still sends SIGUSR1 for hot-reloadable paths (browser.profiles.*), bypassing reload mode

Source report: [records/openclaw-openclaw/items/43803.md](records/openclaw-openclaw/items/43803.md)

## Summary

Keep open. Current `main` and the latest release still route `browser.profiles.*` through the browser plugin's broad `browser` restart prefix, so a profile-only `config.patch` can still be classified as restart-required and reach the direct SIGUSR1 scheduling path.

## Plan

Fix https://github.com/openclaw/openclaw/issues/43803. Browser profile config changes such as `browser.profiles.<name>.cdpUrl` should not force a gateway SIGUSR1 restart when the runtime can hot/no-op refresh them. Keep the fix in the browser plugin/reload metadata boundary where possible; do not hard-code browser profile exceptions in `src/gateway/server-methods/config-write-flow.ts`. Be careful that current reload matching is first-prefix-wins, so adding a specific noop/hot prefix after a broad `restartPrefixes: ["browser"]` will not work unless the broad prefix is removed/narrowed or rule specificity is fixed with coverage. Preserve restarts for structural browser service settings that really require plugin/service re-registration. Add/update regression coverage for reload planning and `config.patch` restart scheduling, keep the existing browser profile hot-refresh test passing, update `docs/tools/browser.md` to distinguish structural browser config from profile-only changes, and add a changelog fix entry if required by current release policy.
