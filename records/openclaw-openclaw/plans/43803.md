## Summary

Keep open. Current main still lets `browser.profiles.*` changes reach the restart-required path because the browser plugin registers the whole `browser` tree as a restart prefix; the earlier broad `config.patch` restart bug is fixed, but this browser-profile granularity bug remains.

## Plan

Fix the remaining browser-profile config reload bug tracked by https://github.com/openclaw/openclaw/issues/43803. Current main still registers the browser plugin with `restartPrefixes: ["browser"]`, so `browser.profiles.*` and likely `browser.defaultProfile` are classified as gateway-restart changes and hot-mode `config.patch` can schedule SIGUSR1. Implement a plugin-owned granular reload registration, not a hard-coded core allowlist: preserve restart behavior for structural browser service settings such as `browser.enabled` or other service-affecting root fields, but make profile/default-profile changes hot or no-op according to the browser runtime refresh behavior. Update focused tests in the browser plugin metadata test, gateway reload-plan tests, and config.patch restart scheduling tests. Align docs only if the final behavior changes documented wording; do not add a new config option or widen core/plugin boundaries.
