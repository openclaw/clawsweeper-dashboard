## Summary

Keep open. Current main and the latest release still let a top-level Feishu card-action `open_message_id` win over `context.open_message_id`, so a temporary `card-action-c-...` callback id can still become the streaming reply target; the adjacent open card-update PR is related but does not resolve or close this report.

## Plan

Fix https://github.com/openclaw/openclaw/issues/56818 by normalizing Feishu card-action callback ids so `context.open_message_id` is preferred when the top-level `open_message_id` is a temporary `card-action-c-...` callback id. Keep the existing SDK-flat top-level real `open_message_id` behavior, keep the no-real-id plain-send fallback, and do not pull in the unrelated in-place card update behavior from https://github.com/openclaw/openclaw/pull/64027. Add a focused regression in the Feishu card-action lifecycle tests for a payload containing both top-level `open_message_id: "card-action-c-..."` and `context.open_message_id: "om_original"`, asserting the reply dispatcher receives `om_original`. Add a user-facing changelog fix entry unless maintainers decide to handle changelog at landing.
