# Coding Plan for openclaw/openclaw#52115: [Feature]: memory_search returns stale/invalid results when gateway is long-running

Source report: [records/openclaw-openclaw/items/52115.md](records/openclaw-openclaw/items/52115.md)

## Summary

Keep open: current main and the latest release still start dirty builtin memory search sync in the background, then query the existing index, so the reported first-search stale or ghost-path behavior remains a valid memory-core freshness bug.

## Plan

Fix https://github.com/openclaw/openclaw/issues/52115 in memory-core, not in gateway or CLI wrappers. Current builtin `MemoryIndexManager.search()` only awaits `sync({ reason: "search", force: true })` for an empty index, then calls `startAsyncSearchSync()` which fire-and-forgets dirty sync before querying existing rows; this lets the first `memory_search` after memory file create/delete/rename return stale or ghost paths. Make dirty on-search memory/session sync complete, or otherwise serialize safely within the existing memory_search deadline, before querying results. Preserve existing provider fail-closed behavior, paused index identity handling, search timeout cancellation, and QMD-specific behavior unless a focused sibling test proves it should share the same invariant. Update the async-search test that currently encodes non-awaited sync and add regression coverage for first search after a memory file change and delete. Coordinate with adjacent refs https://github.com/openclaw/openclaw/issues/14770, https://github.com/openclaw/openclaw/issues/56862, https://github.com/openclaw/openclaw/issues/40088, and https://github.com/openclaw/openclaw/issues/90361, but do not add a new config option or broad memory lifecycle refactor.
