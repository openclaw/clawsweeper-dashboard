# Coding Plan for openclaw/openclaw#54014: [Bug]: OPENCLAW_HOME not respected

Source report: [records/openclaw-openclaw/items/54014.md](records/openclaw-openclaw/items/54014.md)

## Summary

Keep open. Current main and the latest v2026.5.18 release still only partially honor the documented OPENCLAW_HOME contract: core state/config paths use the effective home in some cases, but the nested .openclaw edge case, default dev/git checkout, and installer bootstrap/onboarding defaults still source-reproduce.

## Plan

Fix the OPENCLAW_HOME path-resolution bug tracked by this issue. Scope the repair to internal OpenClaw defaults: preserve current parent-home semantics, add a narrow guard so OPENCLAW_HOME values whose basename is an existing state dir name such as .openclaw or legacy state names are not nested again, update both resolveStateDir() and resolveConfigDir(), and make default git/update/install bootstrap paths derive from OPENCLAW_HOME when OPENCLAW_GIT_DIR, OPENCLAW_STATE_DIR, or OPENCLAW_CONFIG_PATH are not explicitly set. Do not mechanically replace every HOME or os.homedir use; leave intentional OS-account behavior such as shell profile probing, host tilde expansion, personal ~/.agents skills, and Trash paths alone and document those exceptions. Keep the profile-precedence and sandbox-tilde PRs separate unless the central helper change naturally covers them. Add focused regression tests and user-facing docs/changelog coverage for the path behavior change.
