# Coding Plan for openclaw/openclaw#40840: sessions.delete is not a guaranteed hard-kill boundary after normal agent runs

Source report: [records/openclaw-openclaw/items/40840.md](records/openclaw-openclaw/items/40840.md)

## Summary

Keep open: current main and the latest release still have a source-visible lifecycle gap where `sessions.delete` can remove/archive a row, then a normal agent-run finalizer can write the same session key back from stale run state. The prior ClawSweeper comment correctly identified this as a narrow session-store race, and no linked closing PR or canonical replacement issue currently owns the remaining fix.

## Plan

Fix https://github.com/openclaw/openclaw/issues/40840 by making explicit `sessions.delete` win over stale in-flight normal `agent` finalization. Current main deletes and archives the row in `src/gateway/server-methods/sessions.ts`, but `src/agents/command/session-store.ts` can build from the caller snapshot/defaults and write `store[sessionKey] = merged` after the fresh writer store no longer has that key. Add a narrow store/lifecycle guard so a finalizer for a session that existed at run start does not recreate it after explicit deletion; preserve legitimate first-run session creation, heartbeat/runtime model behavior, and existing `sessions.delete` cleanup semantics. Do not add a new `sessions_kill` RPC, config option, or broad product surface. Add focused regression coverage for delete during the agent finalization window, and update `CHANGELOG.md` because this is user-visible Gateway session behavior.
