# Coding Plan for openclaw/openclaw#40840: sessions.delete is not a guaranteed hard-kill boundary after normal agent runs

Source report: [records/openclaw-openclaw/items/40840.md](records/openclaw-openclaw/items/40840.md)

## Summary

Keep open: current main still has a source-reproducible stale-finalizer gap where normal post-run session persistence can recreate a missing `sessionKey` after `sessions.delete` succeeds, and no merged or open PR now owns the narrow fence. This should stay scoped to the session-store write boundary; the broader child-process hard-kill request remains separate.

## Plan

Fix the stale-finalizer race where `sessions.delete` can remove a session row but a normal agent run's post-run metadata finalizer later recreates the same `sessionKey`. Keep the scope narrow: add a lifecycle-owned generation, tombstone, or compare-before-write fence shared by the delete path and `updateSessionStoreAfterAgentRun`, so finalizers from the deleted generation cannot write past a successful delete while legitimate later activity can create a fresh session. Do not add a new `sessions_kill` API, a config option, or broader child-process termination behavior; that broader work remains tracked at https://github.com/openclaw/openclaw/issues/12554. Add focused regression coverage for finalizer-after-delete behavior and keep existing restart-recovery guarded persistence and `sessions.delete` lifecycle cleanup semantics intact. Do not edit CHANGELOG.md.
