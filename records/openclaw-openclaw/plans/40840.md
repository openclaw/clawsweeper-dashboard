## Summary

Keep open. Current main and v2026.6.6 still allow the ordinary post-agent-run metadata writer to synthesize a missing session row after `sessions.delete` has removed it, so the narrow delete-fence bug remains actionable.

## Plan

Fix the `sessions.delete` stale-finalizer bug for https://github.com/openclaw/openclaw/issues/40840. Current main removes the row in `src/gateway/server-methods/sessions.ts`, but `src/agents/command/session-store.ts:updateSessionStoreAfterAgentRun` can later recreate the same key from a stale in-memory fallback entry. Add a narrow fence so ordinary post-run metadata persistence does not recreate a missing/deleted row for the same `sessionKey`/`sessionId`, while preserving legitimate session creation before a run and session-id rotation/compaction behavior. Prefer the existing session accessor/store writer boundary; do not add a new public hard-kill API, config option, or long-lived runtime compatibility shim. Add focused regression coverage proving a normal post-run update does not resurrect a row removed by delete, and keep existing preserve-user-facing and lifecycle tests passing. No CHANGELOG.md edit.
