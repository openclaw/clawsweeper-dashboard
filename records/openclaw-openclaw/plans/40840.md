# Coding Plan for openclaw/openclaw#40840: sessions.delete is not a guaranteed hard-kill boundary after normal agent runs

Source report: [records/openclaw-openclaw/items/40840.md](records/openclaw-openclaw/items/40840.md)

## Summary

Keep open. Current main and the latest release still have a source-level gap where ordinary post-run session-store finalization can write a missing session key after `sessions.delete` removes it; the existing stale-lifecycle guard only covers a narrower user-facing recovery path, and the related session-kill issue tracks broader process termination rather than this delete fence.

## Plan

Fix https://github.com/openclaw/openclaw/issues/40840 by making `sessions.delete` a hard store boundary against stale same-run finalizers. Focus on current core session lifecycle code: stale post-run metadata/pending-delivery/lifecycle cleanup must not recreate a session row after the same `sessionKey` and owning `sessionId` were deleted. Do not add a new `sessions_kill` command, config option, or broad process-kill API; leave process termination design to https://github.com/openclaw/openclaw/issues/12554. Preserve legitimate new-session creation, compaction rotation, and heartbeat/preserve-user-facing behavior. Add focused regression coverage that simulates a normal agent run finalizer racing after `sessions.delete`, and prove the deleted key stays absent while ordinary new runs still persist their session rows. No CHANGELOG.md edit.
