## Summary

Keep open. Current main still clears the TUI editor before any busy-run guard and still routes normal chat submissions to `sendMessage`, whose normal path reaches `client.sendChat` without rejecting an existing active run; the direct fix proposal at https://github.com/openclaw/openclaw/pull/45586 closed unmerged, while https://github.com/openclaw/openclaw/pull/59800 covered adjacent pending-send reconciliation rather than this submit-time guard.

## Plan

Fix the TUI input-swallow bug reported at https://github.com/openclaw/openclaw/issues/45326. Add a submit-time guard before `createEditorSubmitHandler` clears the editor so normal chat messages are blocked while `state.activeChatRunId` or `state.pendingOptimisticUserMessage` indicates an active/pending run; preserve the typed draft, avoid adding it to history, do not call `sendMessage`, and show a concise busy hint that points users to Esc or `/abort`. Preserve existing slash-command and bang-line behavior unless a focused test proves a specific command must be blocked. Add regression tests for normal submit during an active run preserving the draft and not sending, plus coverage that slash commands and bang lines keep their intended routing. Include a single-line `CHANGELOG.md` fix entry for the user-visible TUI behavior. Do not change gateway queue semantics, provider behavior, or unrelated TUI rendering.
