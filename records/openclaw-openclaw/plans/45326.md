## Summary

Keep open. Current main and the latest release still clear the TUI draft before any active-run decision, production wiring gives the submit helper no busy-state predicate, and normal chat send still reaches the gateway without rejecting an existing active run; the direct fix PR was closed unmerged.

## Plan

Fix the TUI busy-submit bug tracked by https://github.com/openclaw/openclaw/issues/45326. Current behavior clears normal chat input before checking whether a run is already active or pending, then `sendMessage` can call `client.sendChat` without rejecting `state.activeChatRunId`; this can swallow or replay stale text. Add a TUI-local guard before clearing the editor so normal chat submitted while `activeChatRunId`, `pendingChatRunId`, or an equivalent busy pending-send state is active is not sent and the draft stays visible. Keep Esc as the explicit abort path per `docs/web/tui.md`; do not turn Enter into implicit abort and do not broaden gateway queue semantics. Preserve intended `/btw` side-message behavior and existing slash/bang command behavior unless a command is explicitly rejected, in which case do not lose its draft. Add focused regression coverage for active-run submit preservation, pending-run/transitional submit preservation if applicable, and no second chat send. Add a user-facing fix entry to `CHANGELOG.md` if required by the current active version section. Use the closed prior PR only as reference, not as a wholesale cherry-pick.
