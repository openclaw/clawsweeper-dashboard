## Summary

Keep open: current main still has the reported Telegram DM-topic session-key split. A proactive outbound send mirrors into a `:thread:<topicId>` session, while an inbound reply in the same DM topic routes to `:thread:<chatId>:<topicId>`, so cron-started conversations can lose context.

## Plan

Fix https://github.com/openclaw/openclaw/issues/80212 by aligning Telegram direct DM-topic outbound session-route computation with inbound DM-topic routing. Keep the change in the Telegram plugin or the narrow outbound session helper only if it remains generic: proactive sends to a Telegram DM topic under `dm.threadReplies: "inbound"` and `session.dmScope: "per-account-channel-peer"` should mirror transcript/session metadata under `agent:<agent>:telegram:<account>:direct:<chatId>:thread:<chatId>:<topicId>`, while delivery still sends Telegram's numeric `message_thread_id`. Add focused regression coverage proving outbound and inbound routes match for DM topics, including a named account/per-account scope case. Do not add a cron `sessionKey` override, generic aliasing, or broad session migration unless the narrow alignment cannot work. Add a changelog entry because this is user-visible Telegram session continuity behavior.
