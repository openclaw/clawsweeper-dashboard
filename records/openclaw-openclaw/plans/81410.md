# Coding Plan for openclaw/openclaw#81410: Gateway lifecycle commands target stale SUDO_USER scope from root shell instead of root systemd user service

Source report: [records/openclaw-openclaw/items/81410.md](records/openclaw-openclaw/items/81410.md)

## Summary

Keep open. Current main still treats effective-root plus a non-root `SUDO_USER` as a sudo-to-root command and targets `systemctl --machine <sudo-user>@ --user` before trying root's direct `--user` manager, so the reported root-owned systemd user service can be missed.

## Plan

Fix https://github.com/openclaw/openclaw/issues/81410. In the Linux systemd gateway lifecycle path, prevent stale `SUDO_USER`/`SUDO_UID`/`SUDO_GID` from making an already-root shell manage the sudo caller's user manager when the command is clearly operating on root's own systemd user service. Keep the existing genuine sudo-to-root behavior from https://github.com/openclaw/openclaw/pull/72655 intact. Prefer a small change in `src/daemon/systemd.ts` with focused tests in `src/daemon/systemd.test.ts`: cover euid 0, `HOME=/root`, `USER=root`, `LOGNAME=root`, `XDG_RUNTIME_DIR=/run/user/0`, stale non-root `SUDO_USER`, and a root unit path, and assert lifecycle commands use direct `systemctl --user` for the root service. Also keep or add coverage showing `sudo openclaw ...` still targets the invoking user's machine scope when that is the intended behavior. Do not add a config option. Add a changelog entry if the repair changes user-visible gateway lifecycle behavior.
