# Coding Plan for openclaw/openclaw#81410: Gateway lifecycle commands target stale SUDO_USER scope from root shell instead of root systemd user service

Source report: [records/openclaw-openclaw/items/81410.md](records/openclaw-openclaw/items/81410.md)

## Summary

Keep open: current main still has the reported source-level bug. The Linux systemd resolver sends effective-root commands with a non-root `SUDO_USER` to the sudo caller's user manager, while unit discovery still uses the current `HOME`, so a root-owned user unit under `/root` can be discovered but lifecycle/status calls target the wrong systemd user scope.

## Plan

Fix the Linux gateway systemd user-scope resolver so `openclaw gateway start/stop/status` run as effective root with `HOME=/root`, `USER=root` or `LOGNAME=root`, `XDG_RUNTIME_DIR=/run/user/0`, and stale non-root `SUDO_USER`/`SUDO_UID`/`SUDO_GID` target root's direct systemd user manager when managing the root-owned user unit under `/root/.config/systemd/user/openclaw-gateway.service`. Preserve the intended `sudo openclaw ...` behavior where root should manage the invoking sudo user's user service. Add focused regression coverage in `src/daemon/systemd.test.ts` and status/lifecycle coverage only if needed; avoid adding a config knob or changing system-scoped unit behavior from `https://github.com/openclaw/openclaw/pull/57276`. Include a changelog entry if the fix is user-facing under current release policy, and obtain live Linux systemd proof before landing.
