# Coding Plan for openclaw/openclaw#81410: Gateway lifecycle commands target stale SUDO_USER scope from root shell instead of root systemd user service

Source report: [records/openclaw-openclaw/items/81410.md](records/openclaw-openclaw/items/81410.md)

## Summary

Keep open. Current main and the latest shipped release still prefer a non-root `SUDO_USER` machine scope whenever the process is effectively root, so a root-owned systemd user service can be mis-targeted; the related sudo fixes do not cover this stale-root-shell case.

## Plan

Fix https://github.com/openclaw/openclaw/issues/81410. In the shared Linux systemd lifecycle code, ensure `openclaw gateway start/stop/restart/status` do not prefer a stale non-root `SUDO_USER` machine scope when the process is effectively root and the resolved service context is root-owned (`HOME=/root`, `USER`/`LOGNAME=root`, root user unit path, or equivalent trusted context). Preserve the intentional sudo-to-root behavior from https://github.com/openclaw/openclaw/pull/72655 and the `sudo -u` stale-SUDO_USER behavior. Keep the fix in `src/daemon/systemd.ts` and tests; do not add a new config option, do not broaden into system-scoped unit support, and do not pass service EnvironmentFile secrets to `systemctl`. Add focused regressions for the stale root-shell case plus no-regression assertions for existing sudo-to-root and `sudo -u` cases. Add a changelog entry if the fix is user-visible.
