# Coding Plan for openclaw/openclaw#81410: Gateway lifecycle commands target stale SUDO_USER scope from root shell instead of root systemd user service

Source report: [records/openclaw-openclaw/items/81410.md](records/openclaw-openclaw/items/81410.md)

## Summary

Keep open: current main still routes effective root plus a non-root `SUDO_USER` to the sudo user's `--machine ... --user` scope before root's direct `--user` manager, so the reported root-owned user-service stop/start/status mismatch remains actionable. The related system-unit PR is adjacent Linux systemd work, but it does not resolve this stale root-shell user-service case.

## Plan

Fix https://github.com/openclaw/openclaw/issues/81410 on current main. The bug is Linux gateway lifecycle/status routing when effective uid is root, `HOME=/root`, `USER=root`/`LOGNAME=root`, `XDG_RUNTIME_DIR=/run/user/0`, and stale non-root `SUDO_USER`/`SUDO_UID`/`SUDO_GID` remain in the environment: `openclaw gateway stop/start/status` should target root's systemd user service under `/root/.config/systemd/user`, not the stale sudo user's `--machine <user>@ --user` scope. Keep the change focused in the daemon systemd resolver/status path; preserve the intentional sudo-to-root and sudo-u behavior covered by existing tests from https://github.com/openclaw/openclaw/pull/72655; do not implement broad `/etc/systemd/system` support or add a new config option. Add/update regression tests for stop/restart/is-enabled/runtime status with the stale root-shell env, and keep systemd user-unit docs behavior aligned. Changelog is expected because this is user-visible Linux gateway lifecycle behavior. Validate with `node scripts/run-vitest.mjs src/daemon/systemd.test.ts`, `node scripts/run-vitest.mjs src/cli/daemon-cli/lifecycle-core.test.ts src/cli/daemon-cli/status.gather.test.ts`, and `git diff --check`; collect live Linux root systemd user-service proof for stop/start/status with stale `SUDO_*` before landing if feasible.
