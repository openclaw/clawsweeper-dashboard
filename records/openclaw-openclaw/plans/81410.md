# Coding Plan for openclaw/openclaw#81410: Gateway lifecycle commands target stale SUDO_USER scope from root shell instead of root systemd user service

Source report: [records/openclaw-openclaw/items/81410.md](records/openclaw-openclaw/items/81410.md)

## Summary

Keep open: current main still has a source-level path that matches the report. The Linux systemd resolver treats any effective-root process with a non-root `SUDO_USER` as a sudo-to-root invocation and sends lifecycle/status calls to `systemctl --machine <sudo-user>@ --user`, while the unit discovery path can still read the root-owned user unit from `/root`, producing the wrong service target and contradictory status output.

## Plan

Fix https://github.com/openclaw/openclaw/issues/81410 on current main. In `src/daemon/systemd.ts`, adjust the systemd user-scope selection so an effective-root shell with root service context (`USER=root`/`LOGNAME=root`, `HOME=/root`, `XDG_RUNTIME_DIR=/run/user/0`, or a resolved root-owned user unit) does not let stale non-root `SUDO_USER` redirect lifecycle/status commands to `systemctl --machine <sudo-user>@ --user`. Preserve the intentional sudo-to-root behavior from https://github.com/openclaw/openclaw/pull/72655 for real `sudo openclaw ...` flows, and preserve sudo-u target-user behavior. Add focused regression tests in `src/daemon/systemd.test.ts` for stop/restart/is-enabled/runtime or status-relevant calls under stale sudo root-shell env, plus guard tests showing existing sudo behavior remains. Do not add a config flag and do not fold in the system-scoped `/etc/systemd/system` work from https://github.com/openclaw/openclaw/pull/57276 unless a tiny shared helper is unavoidable. Add a user-facing changelog fix entry. Before landing, collect redacted live Linux proof if possible: root-owned `~/.config/systemd/user/openclaw-gateway.service`, stale `SUDO_*`, then `openclaw gateway stop/start/status` targets root's user manager and no longer reports disabled while the root listener is active.
