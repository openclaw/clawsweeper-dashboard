## Summary

Keep open: current main still routes effective root plus a non-root `SUDO_USER` to the sudo caller's systemd user scope before trying root's direct `--user` manager, so the reported root-owned user-service lifecycle mismatch remains actionable.

## Plan

Fix https://github.com/openclaw/openclaw/issues/81410 as a narrow gateway daemon systemd bug. Current main routes effective root plus stale non-root `SUDO_USER` to `systemctl --machine <sudo-user>@ --user`, even when `HOME=/root`, `USER=root`, `LOGNAME=root`, `XDG_RUNTIME_DIR=/run/user/0`, and the managed unit path is `/root/.config/systemd/user/openclaw-gateway.service`; stop/start/status can then target the sudo user's scope while root's user service remains active. Update the resolver/lifecycle path so root-owned user-service operations use root's direct `systemctl --user` manager when the effective service context is root's user unit. Preserve intentional sudo-to-root and sudo-u behavior from https://github.com/openclaw/openclaw/pull/72655, including user-bus fallback behavior; do not broaden this into `/etc/systemd/system` support, do not add a config option, and do not change unrelated system-scoped-unit policy from https://github.com/openclaw/openclaw/pull/57276. Add focused regressions in `src/daemon/systemd.test.ts` for stale non-root `SUDO_USER` with root HOME/USER/LOGNAME/XDG_RUNTIME_DIR and keep adjacent lifecycle/status coverage as needed. Add an active `CHANGELOG.md` Fixes entry if the landing workflow expects one for this user-visible gateway fix.
