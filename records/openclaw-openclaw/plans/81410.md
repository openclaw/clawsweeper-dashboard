# Coding Plan for openclaw/openclaw#81410: Gateway lifecycle commands target stale SUDO_USER scope from root shell instead of root systemd user service

Source report: [records/openclaw-openclaw/items/81410.md](records/openclaw-openclaw/items/81410.md)

## Summary

Keep open: current main and v2026.5.18 still route effective-root plus a non-root `SUDO_USER` to the sudo caller's machine-scoped systemd user manager before root's direct `--user` manager, so the reported root-owned user-service lifecycle mismatch remains actionable.

## Plan

Fix https://github.com/openclaw/openclaw/issues/81410 as a narrow Linux gateway systemd user-scope bug. In `src/daemon/systemd.ts`, adjust user-scope resolution so an effective-root process managing root's user unit, such as `HOME=/root`, `USER=root`, `LOGNAME=root`, and `XDG_RUNTIME_DIR=/run/user/0`, does not let stale non-root `SUDO_USER` redirect stop/start/status to `systemctl --machine <sudo-user>@ --user`. Preserve the intentional sudo-to-root and sudo-u behavior added by https://github.com/openclaw/openclaw/pull/72655, and do not broaden this into system-scoped `/etc/systemd/system` support from https://github.com/openclaw/openclaw/pull/57276. Add focused regression coverage in `src/daemon/systemd.test.ts` for the stale root-shell case and existing sudo paths. Run the focused Vitest wrapper commands and `git diff --check`; include live Linux root systemd user-service proof if feasible before landing. No changelog is needed unless maintainers want one during landing.
