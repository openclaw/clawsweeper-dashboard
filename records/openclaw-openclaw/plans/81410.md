# Coding Plan for openclaw/openclaw#81410: Gateway lifecycle commands target stale SUDO_USER scope from root shell instead of root systemd user service

Source report: [records/openclaw-openclaw/items/81410.md](records/openclaw-openclaw/items/81410.md)

## Summary

Keep open: current main and the latest release still prefer the non-root `SUDO_USER` machine scope whenever the effective UID is root, so a stale sudo environment can still misroute root-owned user-service lifecycle commands. The related system-scope work fixes `/etc/systemd/system` units, but it does not cover this root-owned user unit case.

## Plan

Fix https://github.com/openclaw/openclaw/issues/81410. In the Linux systemd gateway adapter, preserve the intentional sudo-to-root behavior from https://github.com/openclaw/openclaw/pull/72655 for normal `sudo openclaw ...` lifecycle commands, but do not let stale `SUDO_USER` override an installed root-owned user service when the active environment and resolved unit identify root as the service owner (`HOME=/root`, root user identity/runtime dir, unit under `/root/.config/systemd/user`). Keep the change inside the daemon systemd lifecycle/scope-selection boundary unless source review proves a caller adjustment is necessary. Add focused regression tests for stale root-shell `SUDO_*` env plus existing sudo caller behavior. Do not add new CLI flags, config keys, broad fallback stacks, or CHANGELOG.md edits.
