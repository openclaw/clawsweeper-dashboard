# Coding Plan for openclaw/openclaw#81410: Gateway lifecycle commands target stale SUDO_USER scope from root shell instead of root systemd user service

Source report: [records/openclaw-openclaw/items/81410.md](records/openclaw-openclaw/items/81410.md)

## Summary

Keep open. Current main still has a source-level path that treats any effective-root process with a non-root `SUDO_USER` as sudo-to-root and routes gateway stop/restart/status systemctl calls to that user's systemd manager, which matches the reported stale-root-shell failure and is not covered by the existing sudo-u regression tests.

## Plan

Fix https://github.com/openclaw/openclaw/issues/81410. In `src/daemon/systemd.ts`, adjust systemd user-scope selection so a root shell whose environment clearly points at root's own user manager and root-owned unit path is not redirected to a stale non-root `SUDO_USER`; preserve the intentional sudo-to-root and sudo-u behavior from https://github.com/openclaw/openclaw/pull/72655. Keep the change scoped to daemon systemd user-service lifecycle/status; do not fold in system-scoped `/etc/systemd/system` work from https://github.com/openclaw/openclaw/pull/57276 unless a tiny shared helper is necessary. Add focused `src/daemon/systemd.test.ts` coverage for effective uid 0, `HOME=/root`, `USER=root`, `LOGNAME=root`, `XDG_RUNTIME_DIR=/run/user/0`, stale non-root `SUDO_USER`, and an existing root user unit, covering stop/restart/status or the underlying `execSystemctlUser` call shape. Add a user-facing changelog fix entry if repo policy still requires it for the repair PR. Stop and ask for maintainer input if the only viable fix requires a new CLI flag/config option or a broad root-execution policy change.
