# Coding Plan for openclaw/openclaw#81410: Gateway lifecycle commands target stale SUDO_USER scope from root shell instead of root systemd user service

Source report: [records/openclaw-openclaw/items/81410.md](records/openclaw-openclaw/items/81410.md)

## Summary

Keep open: current main still routes effective root plus non-root `SUDO_USER` straight to the sudo caller's machine-scoped user manager, so the reported root-owned systemd user-service stop/start/status mismatch remains actionable.

## Plan

Fix https://github.com/openclaw/openclaw/issues/81410. In current main, effective root plus stale non-root SUDO_USER causes `openclaw gateway stop/start/status` to target `systemctl --machine <sudo-user>@ --user` even when the root-owned user unit under `/root/.config/systemd/user/openclaw-gateway.service` is the configured service. Keep the change inside the Linux daemon systemd resolver/control path, prefer root's direct `--user` manager when the effective service context is root's user unit, and preserve the intentional sudo-to-root and sudo-u behaviors covered by existing tests from https://github.com/openclaw/openclaw/pull/72655. Add focused regression coverage for `HOME=/root`, `USER=root`, `LOGNAME=root`, `XDG_RUNTIME_DIR=/run/user/0`, effective uid 0, and stale non-root `SUDO_USER`, and verify status/control paths do not mix root unit discovery with the sudo caller's systemctl scope. Coordinate with but do not implement the system-scoped `/etc/systemd/system` behavior from https://github.com/openclaw/openclaw/pull/57276. Add a concise user-facing changelog fix entry.
