# Coding Plan for openclaw/openclaw#81410: Gateway lifecycle commands target stale SUDO_USER scope from root shell instead of root systemd user service

Source report: [records/openclaw-openclaw/items/81410.md](records/openclaw-openclaw/items/81410.md)

## Summary

The issue should stay open: current main still routes effective-root processes with a non-root `SUDO_USER` directly to that sudo user's systemd user scope, and the reported root-shell shape is not covered by the existing sudo-user regressions.

## Plan

Fix https://github.com/openclaw/openclaw/issues/81410. Current main treats effective uid 0 plus non-root SUDO_USER as sudo-to-root and immediately targets `systemctl --machine <sudoUser>@ --user`, so an already-root shell with HOME=/root, USER=root, LOGNAME=root, XDG_RUNTIME_DIR=/run/user/0, and a root-owned `/root/.config/systemd/user/openclaw-gateway.service` can stop/start/status the wrong user manager. Implement the fix in the shared daemon systemd lifecycle path, probably `src/daemon/systemd.ts`, so this already-root/root-owned-unit context uses direct `systemctl --user` for root. Preserve the intended behaviors from https://github.com/openclaw/openclaw/pull/72655: true sudo-to-root should still support the invoking user's scope, and `sudo -u` target-user cases must not let stale SUDO_USER override USER/LOGNAME. Add focused regression coverage in `src/daemon/systemd.test.ts` for restart/stop/status availability or runtime checks with stale sudo variables and root env. Avoid adding config flags or startup-time migrations. Add a user-facing changelog fix entry if this lane is handled as maintainer-owned work.
