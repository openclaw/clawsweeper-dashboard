# Coding Plan for openclaw/openclaw#92129: [Bug]: Telegram isolated ingress deletes spooled updates when the turn fails — errors are swallowed so handleUpdate never rejects (silent message loss)

Source report: [records/openclaw-openclaw/items/92129.md](records/openclaw-openclaw/items/92129.md)

## Summary

Keep open. Current main still deletes a claimed Telegram isolated-ingress spool entry when `bot.handleUpdate()` resolves, while OpenClaw's Telegram message paths can log a dispatch failure and return normally, violating the documented retryable-handler contract and leaving a credible silent message-loss path.

## Plan

Fix https://github.com/openclaw/openclaw/issues/92129 in the Telegram plugin. Telegram isolated polling must not clean-delete a claimed spooled update when actual message processing failed but the handler swallowed the error. Keep the fix local to Telegram; avoid new config, generic channel API changes, or broad runtime refactors. Prefer threading an explicit per-update processing outcome from the bot/update tracker or spooled replay context into `#handleClaimedSpooledUpdate`, or another narrow Telegram-owned signal, so successful dispatch deletes the spool entry while failed dispatch releases the claim or dead-letters with an intentional reason. Preserve existing message-dispatch dedupe from https://github.com/openclaw/openclaw/pull/85208 and handler-timeout tombstone behavior from https://github.com/openclaw/openclaw/pull/83505. Add focused regression tests showing a spooled Telegram message whose dispatch/process path throws is not deleted just because the handler resolved, plus coverage for the intended retry/dead-letter behavior. Do not edit CHANGELOG.md.
