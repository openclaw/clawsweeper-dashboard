# Coding Plan for openclaw/openclaw#7406: [Feature]: Human-readable Telegram topic names in session dropdown

Source report: [records/openclaw-openclaw/items/7406.md](records/openclaw-openclaw/items/7406.md)

## Summary

Keep open: current main still captures Telegram forum topic names, but the value stops at Telegram inbound metadata and is not carried into stored session display metadata, gateway session rows, or the Control UI selector. Recent discussion also treats this issue as the canonical tracker; the linked implementation PRs are closed unmerged and the open UI cleanup PR is adjacent, not a complete fix.

## Plan

Fix https://github.com/openclaw/openclaw/issues/7406 by adding a narrow generic thread/session display-label path for known thread/topic names. Reuse the existing Telegram topic-name cache and TopicName source; have Telegram forum-topic inbound context populate a generic ThreadLabel or explicit session display-label value, persist it through session metadata without clobbering explicit user labels, expose it through gateway sessions.list rows, and make the Control UI session dropdown prefer that explicit label. Do not parse raw Telegram session keys in the UI and do not add a Telegram-specific core formatting branch if a generic thread-label contract can carry the data. Add focused regression tests for Telegram context, session metadata persistence, gateway row projection, and Control UI selector rendering. Because this is user-visible, add a changelog entry during landing and include redacted real Telegram plus Control UI proof showing a forum-topic session row/selector displays the human topic name.
