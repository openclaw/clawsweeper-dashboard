# Coding Plan for openclaw/openclaw#7406: [Feature]: Human-readable Telegram topic names in session dropdown

Source report: [records/openclaw-openclaw/items/7406.md](records/openclaw-openclaw/items/7406.md)

## Summary

Keep open. Current main already learns Telegram forum topic names and exposes them to prompt metadata, but it still does not promote the known topic name into persisted session display metadata, gateway session rows, or the Control UI session selector.

## Plan

Fix the open Telegram forum topic session-label issue. Current main already caches TopicName in the Telegram plugin and includes it in prompt metadata, but sessions.list and the Control UI selector still only see label/displayName/origin labels that do not include the human topic name. Add a narrow generic explicit thread/session display-label path rather than a Telegram-specific UI parser: have Telegram populate the generic label from cached TopicName for forum-topic sessions, persist/expose it through session metadata and GatewaySessionRow, and make the Control UI session selector prefer it while preserving existing non-Telegram and non-topic labels. Do not add a user config option, do not hardcode Telegram key parsing in the UI as the primary fix, and do not change delivery/session routing semantics. Add focused regression coverage for Telegram topic context, session metadata persistence, gateway session row display, and Control UI selector rendering. Include a user-facing changelog entry during the repair branch if the repository landing workflow expects one.
