# Coding Plan for openclaw/openclaw#7406: [Feature]: Human-readable Telegram topic names in session dropdown

Source report: [records/openclaw-openclaw/items/7406.md](records/openclaw-openclaw/items/7406.md)

## Summary

Keep open: current main and the latest release still learn Telegram forum topic names, but that value stops at Telegram inbound context and prompt metadata instead of becoming stored session display metadata, a gateway session row label, or a Control UI picker label.

## Plan

Fix https://github.com/openclaw/openclaw/issues/7406 by carrying known Telegram forum topic names into session display labels. Reuse the existing cached Telegram TopicName value from extensions/telegram/src/bot-message-context.ts and extensions/telegram/src/bot-message-context.session.ts; do not add a Telegram API polling path, a new per-topic config name field, or Telegram-only UI string parsing. Add a generic session display/thread-label metadata path in the session metadata/types layer, persist it from Telegram forum-topic context, expose it through gateway sessions.list rows, and make the Control UI session selector/shared display resolver prefer the explicit label while preserving existing label/displayName behavior for non-Telegram sessions. Coordinate with the adjacent UI cleanup in https://github.com/openclaw/openclaw/pull/68850 if it is still open, but do not treat that PR as a complete fix. Add focused regression tests for Telegram TopicName propagation, session metadata/gateway row projection, and UI display resolution. Add a user-facing changelog entry because this changes visible session labels. Attach redacted real behavior proof showing a known Telegram forum topic name in the Control UI session picker.
