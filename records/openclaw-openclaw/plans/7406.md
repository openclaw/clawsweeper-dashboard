## Summary

Keep open. Current main has the Telegram topic-name data source, but the known topic name still only reaches inbound context/prompt metadata and is not promoted into persisted session label/displayName data used by sessions.list and the Control UI selector.

## Plan

Fix the canonical Telegram forum topic session-label issue: current Telegram context can learn cached TopicName, but sessions.list and the Control UI session dropdown still show generic/raw topic session labels. Implement a narrow generic session display-label/thread-label path rather than a Telegram-specific UI parser or hardcoded topic map. Prefer wiring Telegram forum TopicName into the existing ThreadLabel/session display metadata path, persist it without clobbering existing human labels on later ordinary messages, expose it in gateway session rows, and make the Control UI selector prefer the explicit human label while preserving non-Telegram behavior. Coordinate with adjacent UI cleanup in https://github.com/openclaw/openclaw/pull/68850, but do not treat that PR as a complete fix. Add focused regression coverage for Telegram topic-name context, session metadata persistence, gateway row projection, and Control UI session-option labels. Do not add a per-topic config name option unless a maintainer asks for that product surface; no changelog is required until maintainer landing unless the repair lane policy asks for one.
