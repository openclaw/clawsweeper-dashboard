## Summary

Keep open. Current main still emits `multiple_gateways` from the number of reachable probe transports alone, while the SSH tunnel path can also keep the configured remote URL as a second target and the collected gateway self identity is not used to distinguish one gateway from two.

## Plan

Fix https://github.com/openclaw/openclaw/issues/69135. `openclaw gateway probe` currently emits `multiple_gateways` whenever more than one target is reachable, even when `sshTunnel` and `configRemote` are two transports to the same gateway. Keep probing both transports, but in `src/commands/gateway-status/output.ts` group reachable targets by confirmed gateway self identity before emitting `multiple_gateways`. Use the existing `GatewayStatusProbedTarget.self` from `pickGatewaySelfPresence`; preserve the warning for genuinely distinct identities and for unknown/null identities where same-gateway cannot be proven. Add focused Vitest coverage, likely in `src/commands/gateway-status/output.test.ts`, for two reachable targets with the same non-null self identity not emitting `multiple_gateways`, and for distinct identities still warning. Do not add a config option, do not skip the configured remote probe, and do not broaden gateway discovery behavior. Add a changelog fix entry because this is user-visible CLI behavior.
