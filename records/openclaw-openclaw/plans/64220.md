# Coding Plan for openclaw/openclaw#64220: [Bug]: Control UI can leave \"Compacting context...\" stuck even when no compaction is running

Source report: [records/openclaw-openclaw/items/64220.md](records/openclaw-openclaw/items/64220.md)

## Summary

Keep open. Current main still has a source-reproducible stale UI path: a compaction `start` can set the active banner, and if the matching `end` is missed, terminal chat/reconnect cleanup clears run/tool state without clearing the compaction/fallback indicator state.

## Plan

Fix the stale Control UI compaction/fallback indicator issue tracked at https://github.com/openclaw/openclaw/issues/64220. Current main can set `compactionStatus.phase = "active"` on a compaction `start`; if the matching `end` is missed, terminal chat/reconnect/send-failure cleanup clears run/tool state but not the compaction/fallback status or timers, so `Compacting context...` remains renderable. Add a shared timer-aware cleanup helper for compaction/fallback UI status, call it from terminal chat cleanup, reconnect/reset, session or agent switch, and send-failure paths as appropriate, and add focused regression tests for a missed compaction `end` followed by terminal chat cleanup. Do not change backend compaction semantics, protocol payloads, provider behavior, or add a config option. Add a user-facing changelog fix entry if the repair lands.
