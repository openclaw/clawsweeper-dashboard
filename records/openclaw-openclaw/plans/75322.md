# Coding Plan for openclaw/openclaw#75322: buildExecEventPrompt: 'user delivery disabled' branch races runtime silent-reply detection, causes incomplete-turn errors

## Summary

Keep open. Current main and the latest release still have the legacy internal exec-completion prompt asking for `HEARTBEAT_OK only` without `NO_REPLY` or `<final>` guidance, while strict final-tag handling can erase a bare `NO_REPLY` before the incomplete-turn guard runs.

## Plan

Fix https://github.com/openclaw/openclaw/issues/75322 by aligning the legacy `buildExecEventPrompt` internal/no-output paths with the existing silent-reply contract. Update only the non-`useHeartbeatResponseTool` exec-completion prompt paths that currently say `Reply HEARTBEAT_OK only` for no output or `deliverToUser=false`; prefer explicit `<final>NO_REPLY</final>` prompt wording or an equally narrow runtime guard that prevents strict-final zero-payload retries. Preserve structured `heartbeat_respond` behavior when `useHeartbeatResponseTool` is true, preserve user-relay exec summaries and missing-output failure wording, and do not broaden this into a general cron/job no-reply API or pull in unrelated sentinel-output filtering from https://github.com/openclaw/openclaw/pull/73785. Add focused regression coverage for `src/infra/heartbeat-events-filter.test.ts` and the strict-final empty-turn path, and add a user-facing changelog Fixes entry.
