## Summary

Keep open. Current main still does not auto-select the only enabled account when the message tool omits `accountId`; the shared runner preserves only explicit/default/bound IDs, and WhatsApp still resolves omitted sends through the default-account helper. The closed related implementation PR is useful context, but it did not merge and no narrower canonical item has replaced this issue.

## Plan

Fix omitted-account fallback for the message tool after channel selection. Preserve explicit `params.accountId`, `input.defaultAccountId`, and target-bound agent binding behavior first; otherwise use the selected channel plugin config adapter to prefer an enabled/configured default account, auto-select exactly one enabled/configured account, and throw clear errors for zero or multiple enabled/configured accounts. Keep the policy generic in shared outbound code rather than adding a WhatsApp-only shim. Add shared channel-selection/message-runner coverage plus WhatsApp regression tests, update routing/message docs, and add a user-facing changelog fix entry if the final change is user-visible.
