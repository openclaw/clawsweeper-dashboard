## Summary

Keep open. Current main still has the reported routing gap: omitted message-tool account IDs are preserved only from explicit/default/bound sources, then WhatsApp falls back to the default-account helper even when that account is disabled.

## Plan

Fix https://github.com/openclaw/openclaw/issues/20756. Implement a shared outbound message-account resolver after channel selection so omitted `accountId` preserves existing precedence first: explicit `params.accountId`, `input.defaultAccountId`, then target-bound agent binding. Only when those are absent, use the selected channel plugin config adapter to prefer an enabled configured default account, otherwise auto-select exactly one enabled/configured account. Throw clear errors when zero enabled/configured accounts or multiple enabled/configured accounts remain. Do not hard-code WhatsApp-only routing policy in core, and do not change unrelated delivery, channel selection, or config migration behavior.
