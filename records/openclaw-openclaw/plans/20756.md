# Coding Plan for openclaw/openclaw#20756: message tool should auto-select the only enabled account

Source report: [records/openclaw-openclaw/items/20756.md](records/openclaw-openclaw/items/20756.md)

## Summary

Keep open. Current main and the latest shipped tag still preserve only explicit, input-default, or target-bound account IDs in the message runner, so omitted WhatsApp `accountId` can still fall back to a disabled `default` account; the related implementation attempt at https://github.com/openclaw/openclaw/pull/20801 was closed unmerged and no canonical replacement is open.

## Plan

Fix https://github.com/openclaw/openclaw/issues/20756. Implement a shared outbound message-account resolver after channel selection, not a WhatsApp-only patch. Preserve explicit `params.accountId`, `input.defaultAccountId`, and target-bound agent account IDs first. When no account is selected, use the selected channel plugin config adapter to prefer an enabled and configured default account, then auto-select exactly one enabled and configured account; throw clear errors for zero or multiple enabled/configured accounts. Use existing adapter hooks (`listAccountIds`, `resolveAccount`, `defaultAccountId`, `isEnabled`, `isConfigured`) and avoid hard-coded channel IDs. Add or update shared runner/channel-selection coverage plus WhatsApp regression coverage for disabled `default` with one enabled named account, update routing/message docs, and add a user-facing `CHANGELOG.md` fix entry. Treat https://github.com/openclaw/openclaw/pull/20801 as source context only; do not copy its fallback-to-default-on-missing-plugin behavior without checking current owner-boundary rules.
