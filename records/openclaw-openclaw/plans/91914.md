## Summary

Current main still creates the foreground delivery fence before per-session turn admission. An older delivery waits while a newer generation is active, while that newer same-session turn can simultaneously wait for the older reply run to finish. The closest regression test settles the newer mocked dispatch immediately and therefore does not cover this admission-ordering cycle, so the issue remains necessary.

## Plan

Fix https://github.com/openclaw/openclaw/issues/91914 so a newer same-session inbound queued in reply-turn admission cannot block or suppress the active turn’s later originating-channel delivery. Trace `dispatchInboundMessageWithBufferedDispatcher`, the foreground fence lifecycle, `dispatchReplyFromConfig`, and `admitReplyTurn`; align fence participation with actual turn admission or otherwise eliminate the opposing wait without weakening the rule that a newer successfully visible reply suppresses stale older delivery. Add focused regression coverage using the real same-session reply-run registry and admission path: hold the first turn’s delivery pending, start a second turn for the same session and target, prove the second remains queued without blocking the first delivery, then prove the second runs and delivers after the first completes.
