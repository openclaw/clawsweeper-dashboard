# Coding Plan for openclaw/openclaw#80862: Telegram uses multiple sendMessage previews instead of editMessageText, leaving stale reasoning messages

## Summary

Keep open: current main and the latest release still contain the reasoning split path that calls `forceNewMessage()` before preserving or deleting the prior Telegram preview id, and the related fix PRs were closed unmerged.

## Plan

Fix https://github.com/openclaw/openclaw/issues/80862 as a narrow Telegram reasoning-preview lifecycle bug. Current main sets `splitReasoningOnNextStream` in `onReasoningEnd`, then the next `onReasoningStream` calls `reasoningLane.stream?.forceNewMessage()` and `resetDraftLaneState(reasoningLane)` before the previous preview id is deleted; `createTelegramDraftStream.forceNewMessage()` resets `streamMessageId`, so the old Telegram preview can be orphaned. Implement the fix in the Telegram plugin boundary, likely around `extensions/telegram/src/bot-message-dispatch.ts` and/or `extensions/telegram/src/draft-stream.ts`, so a previous reasoning preview is actually deleted or archived before a new reasoning preview is created. Do not add config, change `/reasoning` semantics, edit `CHANGELOG.md`, or broaden unrelated progress-mode behavior. Add focused regression coverage for `onReasoningStream` -> `onReasoningEnd` -> second `onReasoningStream`, proving the first preview is cleared/deleted or otherwise not orphaned and the next preview still streams correctly. Be careful: simply replacing the direct call with `rotateLaneForNewMessage(reasoningLane)` is not sufficient unless the real draft stream still preserves or deletes the previous `streamMessageId` before `forceNewMessage()` resets it. Validate with `node scripts/run-vitest.mjs extensions/telegram/src/bot-message-dispatch.test.ts extensions/telegram/src/draft-stream.test.ts` and, before merge, provide real Telegram DM proof with private chat details and tokens redacted.
