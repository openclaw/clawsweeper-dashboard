## Summary

Keep open. Current main still immediately queues WhatsApp connected/disconnected lifecycle system events into the resolved session, and there is no threshold, verbosity, reconnect coalescing, or surface-aware policy on the exposed config/docs surface; the closed banner-filter PR was explicitly superseded by this issue, not a fix for it.

## Plan

Fix the open request to suppress or throttle transient WhatsApp/channel connection status system events. Implement the policy in the WhatsApp-owned lifecycle enqueue path first, not as a broad shared prompt string filter. Add backward-compatible channel/account config such as a lifecycle/status-events verbosity mode and/or notifyAfterMs threshold, coalesce reconnects that recover before the threshold, and preserve durable operator-action notices for logged-out, conflict, auth-required, stopped-channel, and max-retry states. Update config types/schema/help/docs and add focused regression coverage for transient reconnect suppression plus preservation of terminal outage notices. Do not hard-code unrelated channel IDs in core and do not change shared system-event queue semantics unless an additive generic seam is clearly needed. Add a changelog entry because this changes user-visible prompt noise/config behavior.
