# Coding Plan for openclaw/openclaw#80254: Session not restored after Gateway crash — new session created instead

Source report: [records/openclaw-openclaw/items/80254.md](records/openclaw-openclaw/items/80254.md)

## Summary

Keep open. Current main has partial WebChat session-continuity and restart-recovery machinery, but it does not provide a high-confidence fix for this Gateway-crash case: the UI can still reset a token-only reconnect to `main`, and backend restart recovery only resumes sessions that were explicitly marked as running and `abortedLastRun`.

## Plan

Fix https://github.com/openclaw/openclaw/issues/80254. On current main, WebChat/Gateway reconnect after a crash can plausibly fall back to `main` or a fresh session even when the previous transcript/store row still exists. Add a focused failing regression first, then make the narrowest fix so Control UI restores the previous selected WebChat session on reconnect/restart when the session selection, store row, or transcript is still valid. Likely starting points: `ui/src/ui/app-settings.ts` token-only URL handling, `ui/src/ui/app-gateway.ts` `applySessionDefaults`/hello reconnect flow, `ui/src/ui/storage.ts` scoped persisted selection, `ui/src/ui/controllers/chat.ts`, and only if the regression proves it, `src/agents/main-session-restart-recovery.ts` or `src/gateway/server-startup-post-attach.ts`. Do not add a new config option, do not implement the broad graceful-restart manifest/cron recovery work from https://github.com/openclaw/openclaw/issues/57425, and do not reset legitimate user-selected sessions to `main` unless the session is truly invalid. Add/update focused tests and a user-visible changelog entry if behavior changes.
