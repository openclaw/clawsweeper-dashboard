## Summary

Keep open: current main still lacks a transcript-file-scoped active embedded prompt holder, so the same session JSONL can be entered by another logical run while the original prompt lock is released. The older fallback and owned-write fixes reduce related symptoms, but they do not prevent duplicate prompt holders from racing on the same file.

## Plan

Fix https://github.com/openclaw/openclaw/issues/85913 on current main. Current main still keys active embedded runs by sessionId/sessionKey and releases the session write lock during provider I/O, so a second logical run can enter the same resolved session JSONL and later trip EmbeddedAttemptSessionTakeoverError. Add a resolved transcript-file-scoped active prompt holder guard before provider streaming, and update stuck-session recovery so it does not abort/re-enter/re-fire against a session file that still has an active prompt holder; wait, steer, fail cleanly, or allocate a fresh session according to the existing recovery contracts. Preserve the existing takeover fence and the owned-write publication behavior from https://github.com/openclaw/openclaw/pull/84250; do not broaden trust of external session-file edits, do not change heartbeat.isolatedSession defaults, and do not change provider fallback semantics from https://github.com/openclaw/openclaw/pull/83550. Add focused regression coverage for same-file duplicate holders, stuck-session recovery with an active holder, and the relevant heartbeat/channel busy-lane path.
