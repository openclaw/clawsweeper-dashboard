## Plan

Fix the Telegram idle-account restart loop tracked by https://github.com/openclaw/openclaw/issues/83264. Current main passes `channels.telegram.timeoutSeconds` into the isolated ingress worker, while `fetchJson` aborts `getUpdates` after `TELEGRAM_GET_UPDATES_REQUEST_TIMEOUT_MS` (45s); when configured `timeoutSeconds` is higher, idle long polls can abort before returning `[]`, so no `poll-success` marks liveness and the gateway health monitor can restart the account as disconnected. Keep the fix inside the Telegram plugin owner boundary. Make isolated polling use a bounded Bot API long-poll timeout that cannot outlive the existing request guard, preserve the current behavior that successful empty `getUpdates` responses count as liveness, and do not add a new healthMonitor config flag or change generic gateway health policy.
