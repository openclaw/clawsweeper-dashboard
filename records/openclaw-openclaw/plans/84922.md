## Summary

Keep open: current main still filters unsupported cron payload kinds out of the loaded cron store, and a later full persist can rewrite `jobs.json` without those rows. The warning path now makes this partly visible, but the durable-job data-loss path remains source-reproducible.

## Plan

Fix https://github.com/openclaw/openclaw/issues/84922 by making routine cron-store load/save lossless for persisted jobs whose `payload.kind` is not currently executable, such as `command` or `agentmessage`. Keep the runtime contract strict: do not add direct command cron execution, do not add a new CLI/schema payload kind, and do not claim to fix https://github.com/openclaw/openclaw/issues/84923 or https://github.com/openclaw/openclaw/issues/18160. Preserve unsupported raw rows across full `persist(state)` writes while still warning operators and still skipping them for runtime scheduling/execution. Add regression coverage that seeds `jobs.json` with valid jobs plus unsupported payload-kind rows, triggers the load plus full-persist path, and asserts the unsupported rows remain in `jobs.json`; keep doctor migration behavior explicit for truly invalid schedule/payload shapes. Add a changelog entry because this prevents user-visible persisted cron job loss.
