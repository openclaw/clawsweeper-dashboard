## Summary

Keep open. Current main still has Feishu degraded approval paths where interactive approval payloads become label-only text, and the generic forwarded approval text still uses a literal `/approve <id>` placeholder; the open interpolation PR helps one piece but does not cover Feishu document-comment degradation or channel-specific fallback guidance.

## Plan

Fix https://github.com/openclaw/openclaw/issues/69754 by preserving Feishu native cards for supported chat targets while improving text fallbacks for degraded Feishu/document-comment surfaces. On current main, `extensions/feishu/src/outbound.ts` routes Feishu document-comment interactive/presentation payloads through `renderMessagePresentationFallbackText`, and `src/interactive/payload.ts` emits only button labels, so approval button values like `/approve req_1 allow-once` disappear. Add approval-aware fallback copy that explains native buttons are unavailable on that Feishu surface and includes exact allowed `/approve <id> <decision>` commands; prefer a shared renderer/helper over Feishu-only string scraping. Account for open PR https://github.com/openclaw/openclaw/pull/78793: do not duplicate the generic `request.id` interpolation if it lands, and rebase/use its behavior. Update Feishu outbound and shared interactive payload tests, add Feishu/approval docs for degraded approval UX, and add a changelog entry if user-visible copy changes. Do not add a config option, new approval transport, new auth policy, or Feishu-specific core dependency.
