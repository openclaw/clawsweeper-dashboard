# Coding Plan for openclaw/openclaw#51393: Feature: browser tool evaluate should support statements (var/let/const)

Source report: [records/openclaw-openclaw/items/51393.md](records/openclaw-openclaw/items/51393.md)

## Summary

Keep open: current main and v2026.5.22 still parse managed browser evaluate input as a parenthesized expression, so `var`/`let`/`const` statement bodies remain unsupported. The closed browser exec/evaluate request explicitly left this narrower behavior as follow-up, and the FSB comment gives useful prior art for treating input as a function body while preserving the existing timeout wrapper.

## Plan

Implement statement-body support for browser act kind=evaluate in the browser plugin. Preserve current expression/function inputs and existing timeout, abort, navigation guard, SSRF, evaluateEnabled, ref/no-ref, batch, and existing-session constraints. Treat useful prior art from https://github.com/openclaw/openclaw/issues/51393#issuecomment-4508600259 positively: callers should be able to use `const` and `return` naturally while the tool owns the async wrapper and timeout. Add focused tests for managed Playwright evaluate with declaration statements and returns, plus compatibility tests for existing expression/function examples. Check Chrome MCP existing-session parity through `evaluateChromeMcpScript` or document/test the intended limitation. Do not add a separate browser.exec command, do not broaden evaluate defaults, and keep changelog/docs aligned because this is user-visible browser tool behavior.
