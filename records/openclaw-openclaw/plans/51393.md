# Coding Plan for openclaw/openclaw#51393: Feature: browser tool evaluate should support statements (var/let/const)

Source report: [records/openclaw-openclaw/items/51393.md](records/openclaw-openclaw/items/51393.md)

## Summary

Keep open. Current main and v2026.5.12 still parse managed browser evaluate input as parenthesized expressions, so declaration statements like `var`, `let`, and `const` remain unsupported without the reported IIFE workaround; the related browser exec/evaluate thread was closed while explicitly leaving this statement-body request as a follow-up.

## Plan

Implement statement-body support for browser evaluate in the browser plugin without adding a new core tool, config option, or force-click behavior. Preserve existing expression and function-source semantics first: function sources should still be called with the element for ref evaluate and without arguments for page evaluate; expressions and IIFEs should still return the same values; runtime errors should not be hidden by a statement fallback; async results must keep the current Promise.race timeout behavior. Add shared private parsing/execution logic for the Playwright page and ref evaluators, support declaration statement bodies such as `var x = 1; return x`, `let x = 1; return x`, and `const x = 1; return x`, and clarify whether existing-session Chrome MCP evaluate can support the same behavior or should be documented/rejected consistently. Add focused regression tests and update browser evaluate docs/CLI wording plus CHANGELOG for the user-visible feature. Do not implement the separate force-click request tracked elsewhere.
