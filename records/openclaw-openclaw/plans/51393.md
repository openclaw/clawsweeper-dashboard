## Summary

Keep open: current main and the latest release still parse managed browser evaluate input as a parenthesized expression, so statement bodies with var/let/const still fail. The closed browser exec thread explicitly left this narrower request as follow-up, and the recent FSB prior-art comment supports a focused function-body style repair rather than closing this as solved.

## Plan

Fix https://github.com/openclaw/openclaw/issues/51393 by teaching browser evaluate to accept statement/function-body input such as `const x = 1; return x` without requiring an IIFE. Keep the change inside the browser plugin evaluate implementation and tests; preserve existing expression input, function input, ref input, async results, timeout/abort handling, evaluateEnabled gating, batch execution, and navigation/SSRF guards. Do not add a separate browser.exec command, do not change default evaluate security policy, and do not add a config option. Check the existing-session Chrome MCP path and either support the same body semantics there or keep/document a clear driver limitation if upstream requires function source. Add focused regression coverage and a user-facing changelog entry.
