# Coding Plan for openclaw/openclaw#45952: Webchat: messages lost during WebSocket reconnect (no client-side queue/ACK)

Source report: [records/openclaw-openclaw/items/45952.md](records/openclaw-openclaw/items/45952.md)

## Summary

Keep open: current main still has a source-proven WebChat data-loss window during transient WebSocket reconnects. Normal sends are discarded before capture when the UI is disconnected, in-flight RPCs are rejected on socket close without replay, and the existing server ACK/idempotency contract is not yet wired into a durable Control UI outbound queue.

## Plan

Fix the WebChat reconnect-window message-loss bug reported in https://github.com/openclaw/openclaw/issues/45952. Add a focused Control UI outbound-send queue for normal `chat.send`: capture the message, attachment snapshot or explicit attachment failure state, session key/session id, and a stable idempotency key before transport; do not drop the draft when the Gateway is transiently disconnected or when the pending `chat.send` RPC is rejected by socket close; retry unACKed sends after reconnect hello; remove queued sends only after the `chat.send` ACK/duplicate response confirms receipt; and show bounded failed-delivery feedback after retry exhaustion. Keep this separate from the broader refresh/draft/history persistence work in https://github.com/openclaw/openclaw/issues/51549 and https://github.com/openclaw/openclaw/pull/73154, and coordinate with but do not depend on https://github.com/openclaw/openclaw/pull/75776. Do not add a config option or new gateway protocol unless the existing documented ACK/idempotency contract proves insufficient. Add focused regression coverage for disconnected normal submit queueing, pending RPC rejection followed by reconnect flush using the same idempotency key, no duplicate agent turn for same-key retry, and attachment retry/fail-visible behavior.
