# Coding Plan for openclaw/openclaw#45952: Webchat: messages lost during WebSocket reconnect (no client-side queue/ACK)

Source report: [records/openclaw-openclaw/items/45952.md](records/openclaw-openclaw/items/45952.md)

## Summary

Keep open: current main still has a source-reproducible WebChat data-loss window during transient WebSocket reconnects. Normal sends are not captured before the connected-state gate, reconnect-window RPCs are rejected and cleared without replay, and the existing server ACK/idempotency contract is not yet used by a durable Control UI outbound queue.

## Plan

Fix https://github.com/openclaw/openclaw/issues/45952 on current main. Implement a focused Control UI outbound-send retry queue for normal `chat.send` only: capture the message, attachments, session identity, and stable idempotency key before transport; keep unACKed items until the `chat.send` ACK resolves; retry transient `gateway not connected`/socket-close failures after reconnect hello with the same key; and show bounded failed-delivery feedback after retry exhaustion. Do not broaden this into localStorage refresh/draft/history persistence and do not change the gateway protocol unless the existing ACK/idempotency contract proves insufficient. Preserve attachment payload bytes for retry or fail visibly instead of replaying metadata-only sends. Add regression coverage for disconnected submit, pending RPC rejection during socket close, reconnect flush with same idempotency key, and no duplicate agent turn when ACK was lost. Add a user-facing changelog entry if the fix changes visible behavior.
