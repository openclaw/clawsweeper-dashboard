## Summary

Keep open: current main and the latest release still drop the normal WebChat send path during ordinary reconnect windows. The source path returns before queueing while disconnected, rejects in-flight RPCs on socket close, and only resumes the chat queue for seq-gap reconnects, while the documented `chat.send` ACK/idempotency contract is already available for a focused UI retry fix.

## Plan

Fix https://github.com/openclaw/openclaw/issues/45952 with a focused Control UI WebChat reconnect-send repair. Add an outbound normal `chat.send` queue/record that captures the composer payload before transport, preserves a stable idempotency key across retry, retries transient closed/not-connected or socket-close failures after the next reconnect hello, and shows bounded failed-delivery feedback when retry budget is exhausted. Keep the scope to transient WebSocket reconnect delivery; do not implement broad browser-refresh/history/draft persistence, do not change the gateway protocol unless the existing `chat.send` ACK/idempotency contract proves insufficient, and do not persist attachment data longer than needed for the transient retry. Re-check related open PRs https://github.com/openclaw/openclaw/pull/73154 and https://github.com/openclaw/openclaw/pull/75776 before finalizing to avoid conflicting behavior. Add focused regression tests for disconnected submit capture, in-flight RPC close/reconnect replay with the same idempotency key, retry failure feedback, and the existing busy-run queue behavior. Add a user-facing changelog entry if the fix lands through the maintainer repair lane.
