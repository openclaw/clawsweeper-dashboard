# Coding Plan for openclaw/openclaw#45952: Webchat: messages lost during WebSocket reconnect (no client-side queue/ACK)

Source report: [records/openclaw-openclaw/items/45952.md](records/openclaw-openclaw/items/45952.md)

## Summary

Keep open: current main still has a source-reproducible WebChat data-loss window for normal sends during transient WebSocket reconnects. The server already has an ACK/idempotency contract, but the Control UI does not retain a normal `chat.send` payload/key until ACK or replay it after reconnect.

## Plan

Fix https://github.com/openclaw/openclaw/issues/45952 as a focused WebChat data-loss bug. In Control UI, normal `chat.send` submits must be captured before transport with a stable idempotency key, retained until the server ACK resolves, retried after transient WebSocket close/reconnect hello, and surfaced as failed after bounded retry/error conditions. Keep the fix scoped to transient reconnect delivery for normal sends; do not implement broad browser refresh queue/draft persistence from https://github.com/openclaw/openclaw/issues/51549 or active-run snapshot recovery from https://github.com/openclaw/openclaw/pull/41067 unless required by this bug. Reuse the existing `chat.send` ACK/idempotency contract and avoid gateway protocol changes unless source inspection proves the current contract is insufficient. Preserve attachments safely for transient reconnect or show explicit failed-delivery feedback when replay is impossible. Add focused regression tests for disconnected submit queueing/replay, pending RPC rejection on close, same-key retry after ACK loss, and non-duplication when the server already accepted the send. Add a changelog entry if the fix changes user-visible WebChat behavior.
