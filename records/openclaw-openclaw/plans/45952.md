# Coding Plan for openclaw/openclaw#45952: Webchat: messages lost during WebSocket reconnect (no client-side queue/ACK)

## Summary

Keep open: current main still has a source-reproducible WebChat data-loss window for normal sends during transient WebSocket reconnects. The server ACK/idempotency contract already exists, but the Control UI does not retain a normal `chat.send` payload/key until ACK or replay it after reconnect.

## Plan

Fix https://github.com/openclaw/openclaw/issues/45952 with a focused Control UI WebChat outbound-send retry path. Capture normal `chat.send` payloads before transport, assign and retain a stable idempotency key until the server ACK resolves, retry transient `gateway not connected` / socket-close failures after successful reconnect hello, and surface bounded failed-delivery state if retries cannot complete. Use the existing `chat.send` ACK/idempotency contract; do not add a new gateway protocol unless source proof shows the current contract is insufficient. Keep scope to transient reconnect delivery for normal sends; do not solve browser refresh draft/history persistence or broad localStorage queue persistence in this PR. Handle attachments explicitly: preserve recoverable payload bytes for transient retry or fail the queued message visibly without sending metadata-only attachments. Add focused regression coverage for disconnected submit capture, pending RPC close/reconnect replay with the same idempotency key, no duplicate sends after lost ACK, and seq-gap/busy queue behavior staying intact. Add a concise changelog Fixes bullet if the repair branch is landing-ready and user-facing.
