# Coding Plan for openclaw/openclaw#45952: Webchat: messages lost during WebSocket reconnect (no client-side queue/ACK)

Source report: [records/openclaw-openclaw/items/45952.md](records/openclaw-openclaw/items/45952.md)

## Summary

Keep open: current main still has a source-proven WebChat reconnect-window message-loss path. Normal sends are not captured while disconnected, in-flight `chat.send` RPCs are rejected on socket close without replay, and the existing server ACK/idempotency contract is not yet wired into a durable Control UI outbound queue.

## Plan

Fix the WebChat reconnect-window message-loss bug tracked by https://github.com/openclaw/openclaw/issues/45952. Add a focused Control UI outbound-send queue for normal `chat.send`: capture the message, attachments metadata/payload references as appropriate, session key, and a stable idempotency key before attempting transport; keep it pending until the gateway ACKs; retry transient `gateway not connected` / socket-close failures after reconnect hello; and surface bounded failed-delivery state if retries are exhausted. Reuse the existing server `chat.send` ACK/idempotency contract; do not bump gateway protocol, do not add a config option, and do not try to solve the broader browser refresh/history/draft persistence scope from https://github.com/openclaw/openclaw/issues/51549 in the same PR. Be careful with attachments: do not persist large data URLs or private payloads to localStorage, and make transient reconnect behavior explicit in tests. Add or update focused regression tests around disconnected normal submit capture, pending RPC rejection/retry with the same idempotency key, reconnect hello draining, and failure feedback. Add a `CHANGELOG.md` Fixes entry if the PR changes user-visible behavior.
