## Summary

Keep open. Current main still has a TUI reconnect/backfill gap: TUI does one reconnect history load and handles live chat/agent events, but it still has no transcript-event or event-gap recovery path for a post-restart reply that lands after that load.

## Plan

Fix https://github.com/openclaw/openclaw/issues/68970 by adding TUI-owned reconnect/gap transcript backfill for the currently watched session. Use existing gateway APIs such as `sessions.messages.subscribe` or the established session-event subscription path so a persisted `session.message` append after gateway restart causes TUI to reload the current session history. Defer reloads while an active local/chat run is streaming, following the Web UI `session.message` deferral pattern, and flush once idle. Also make event gaps trigger an appropriate backfill/reload path. Keep the change scoped to TUI reconnect/backfill behavior; do not change heartbeat generation, delivery-context routing, provider/channel policy, or add a config option. Add focused regression tests for post-reconnect transcript appends, gap-triggered reload, unrelated-session transcript events, and active-run deferral. Add a `CHANGELOG.md` fixes entry if the repair changes user-visible behavior.
