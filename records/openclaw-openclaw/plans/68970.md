# Coding Plan for openclaw/openclaw#68970: TUI misses post-restart heartbeat reply, while the same reply is visible in Web UI

Source report: [records/openclaw-openclaw/items/68970.md](records/openclaw-openclaw/items/68970.md)

## Summary

Keep open. Current main still leaves the reported TUI recovery gap: TUI only consumes live chat/side-result/agent events, does one reconnect history load, and only uses reconnect watchdog recovery when a tracked active run exists; it does not backfill idle post-reconnect transcript updates the way Control UI does.

## Plan

Fix https://github.com/openclaw/openclaw/issues/68970 by adding TUI reconnect/backfill parity for watched-session transcript updates. Current main handles only `chat`, `chat.side_result`, and `agent` events in `src/tui/tui.ts`; it does one reconnect `loadHistory()` and only recovers reconnect-stale active runs via the watchdog. Add a TUI-owned path that reloads history when a `session.message` update or event gap indicates the current session may have changed after reconnect, reusing the existing `sessions.messages.subscribe` Gateway method if appropriate. Defer or coalesce reloads while `state.activeChatRunId`/optimistic local state is active so user messages are not raced or erased, following the Control UI pattern. Do not change session routing semantics, provider behavior, or the watchdog behavior already merged in https://github.com/openclaw/openclaw/pull/74224 except as needed to integrate the new backfill path. Add focused regression coverage for: current-session transcript event triggers history reload when idle; transcript event is deferred during active run and replayed after terminal/silent-final cleanup; other-session transcript events are ignored; `onGap` triggers a safe history refresh or reconnect/backfill for the current watched session. Add a changelog entry because this is user-visible TUI behavior.
