# Coding Plan for openclaw/openclaw#68970: TUI misses post-restart heartbeat reply, while the same reply is visible in Web UI

Source report: [records/openclaw-openclaw/items/68970.md](records/openclaw-openclaw/items/68970.md)

## Summary

Keep open. Current main still has the same TUI reconnect/backfill gap: TUI does one reconnect history load and handles live chat/agent events, but it does not handle `session.message` transcript events or reload on websocket sequence gaps the way Web UI does.

## Plan

Fix the TUI reconnect/backfill bug from https://github.com/openclaw/openclaw/issues/68970. Current main only loads history once on reconnect and TUI ignores `session.message` transcript events and websocket sequence gaps, so a post-restart heartbeat reply appended after the reconnect load can be visible in Web UI but missing from TUI. Add TUI-owned transcript/gap recovery for the watched session using existing gateway `sessions.messages.subscribe` / `session.message` APIs or an equivalent existing contract. Defer history reload while a local/active chat run is streaming, matching the Web UI pattern, and avoid rewriting heartbeat, delivery-context, or session-routing behavior. Add focused regression coverage for TUI event dispatch/backfill, gap recovery, and any GatewayChatClient subscribe/unsubscribe wrapper needed. Add a `CHANGELOG.md` Fixes entry if the implementation changes user-visible TUI behavior.
