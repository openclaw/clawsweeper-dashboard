# Coding Plan for openclaw/openclaw#66399: Process supervisor: graceful signal escalation and drain timeout for exec tool

Source report: [records/openclaw-openclaw/items/66399.md](records/openclaw-openclaw/items/66399.md)

## Summary

Keep open. Current main and v2026.5.18 still route supervisor timeout/cancel through `adapter.kill("SIGKILL")`, and the child adapter still immediately calls `child.kill("SIGKILL")` in that branch, so the direct child does not get the requested graceful cleanup window.

## Plan

Fix the supervisor cancellation bug tracked at https://github.com/openclaw/openclaw/issues/66399. Current main routes timeout/manual cancellation through `createProcessSupervisor` -> `requestCancel` -> `cancelAdapter` -> `adapter.kill("SIGKILL")`; the child adapter then calls `killProcessTree(...)` but immediately follows with `child.kill("SIGKILL")`, so direct children do not get a real SIGTERM cleanup window. Implement a focused SIGTERM -> grace-period -> SIGKILL policy for supervisor-managed child and PTY runs, keep the hard fallback and wait/drain fallback, make cancellation idempotent, unref and clear timers on settle/dispose, and preserve the service-managed `detached:false` process-group guard from the child adapter. Reuse or align with `src/process/kill-tree.ts` instead of adding broad fallback stacks. Add/update regression tests for no-output timeout, overall timeout, replaceExistingScope/manual cancel, child/PTY adapter escalation, and wait/drain fallback behavior. Do not touch unrelated channel/plugin behavior. Add a single `CHANGELOG.md` fix entry if this is user-facing in the active version section.
