# Coding Plan for openclaw/openclaw#66399: Process supervisor: graceful signal escalation and drain timeout for exec tool

Source report: [records/openclaw-openclaw/items/66399.md](records/openclaw-openclaw/items/66399.md)

## Summary

Keep open. Current main still routes supervisor timeout/cancel through `adapter.kill("SIGKILL")`; the adapter has some wait-fallback and process-tree grace behavior, but the direct supervised child still does not receive a bounded SIGTERM-first grace window before hard kill.

## Plan

Fix the process-supervisor timeout/cancel behavior tracked by https://github.com/openclaw/openclaw/issues/66399. Implement SIGTERM-first cancellation in `src/process/supervisor/supervisor.ts`: on timeout/manual cancellation, send `adapter.kill("SIGTERM")`, wait a short unref'ed grace period, and if the run has not settled, escalate through the existing `adapter.kill("SIGKILL")` path so current process-tree cleanup and wait-fallback behavior remain intact. Do not make the fix SIGTERM-only. Preserve Windows and PTY semantics, and do not redesign session locks, gateway CLI signal handling, or SIGKILL diagnostics in this PR. Add/update focused regression coverage for SIGTERM first, no SIGKILL when the process exits during grace, SIGKILL after grace when it remains running, and existing child/PTY wait fallback behavior as needed. Add a single-line `CHANGELOG.md` fix entry because this is user-visible reliability behavior. Validation should include the focused supervisor/adapter/kill-tree tests and a changed gate in Testbox before handoff.
