# Coding Plan for openclaw/openclaw#68916: [Bug]: ACP oneshot sessions leave orphaned processes — session reset does not clean up child ACP session keys

Source report: [records/openclaw-openclaw/items/68916.md](records/openclaw-openclaw/items/68916.md)

## Summary

Keep open. Current main and the latest release still close only the target ACP session during reset/delete, while parent-owned child ACP sessions are persisted with lineage metadata that the reset path does not enumerate.

## Plan

Fix https://github.com/openclaw/openclaw/issues/68916. Current main closes only the reset/deleted ACP target key in `src/gateway/session-reset-service.ts`; add parent-owned child cleanup by enumerating ACP entries whose `entry.spawnedBy` or `entry.parentSessionKey` matches the canonical parent session being reset/deleted, then close each child with the existing bounded ACP cancel/close helper. Avoid duplicate closes for the parent key, do not add PID exposure, startup reaper policy, broad process-name killing, or a new config option. Add focused regression coverage in `src/gateway/server.sessions.reset-cleanup.test.ts` and `src/gateway/server.sessions.delete-lifecycle.test.ts` that seeds parent plus child ACP entries and asserts both parent and child close/cancel calls, while unrelated ACP sessions are untouched. Add a user-facing changelog fix entry if repo policy requires it for this runtime behavior change.
