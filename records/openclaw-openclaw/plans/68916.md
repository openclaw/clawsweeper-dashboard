## Summary

Keep open. Current main has adjacent ACP process-tree and task-maintenance cleanup, but the parent reset path still closes only the reset target ACP key and does not synchronously enumerate child ACP sessions owned by that parent.

## Plan

Fix https://github.com/openclaw/openclaw/issues/68916 narrowly in the parent session reset path. In `src/gateway/session-reset-service.ts`, make parent `/new` and `/reset` cleanup enumerate ACP session-store entries via `listAcpSessionEntries({ cfg })`, select child ACP entries whose `entry.spawnedBy` or `entry.parentSessionKey` matches the canonical reset parent, and close each child key through the existing ACP cancel/close cleanup path with the reset reason. Add a regression test in `src/gateway/server.sessions.reset-cleanup.test.ts` that seeds a parent session plus child ACP entries and proves resetting the parent closes the child ACP keys. Do not broaden this PR into PID exposure, completed-run fallback policy, acpx startup reaping, or process ownership changes; those are tracked separately. Add a single-line changelog fix entry because this is user-visible cleanup behavior.
