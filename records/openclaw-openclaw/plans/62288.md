# Coding Plan for openclaw/openclaw#62288: Browser: prefer managed openclaw lane; existing-session attach brittle at DevToolsActivePort; improve fallback and diagnostics

Source report: [records/openclaw-openclaw/items/62288.md](records/openclaw-openclaw/items/62288.md)

## Summary

Keep open. Current main covers the managed default and has improved direct Chrome MCP attach errors, but the canonical diagnostics work is still incomplete: profile listing still drops Chrome MCP failure reasons, and Chrome MCP tab selection can still treat browser-internal pages as normal tabs because internal-target filtering only applies to CDP paths.

## Plan

Fix the browser existing-session attach/fallback diagnostics issue in the bundled browser plugin. Preserve the current managed `openclaw` default and explicit `profile="user"` behavior; do not implement a new Chrome extension bridge or change Desktop packaging policy. Add structured Chrome MCP attach/list failure information that survives through `ProfileStatus`, `/profiles`, browser status/doctor, and CLI profile/status output, distinguishing transport/attach failure from page readiness and tab-count failures. Filter browser-internal Chrome MCP `list_pages` results (`chrome://`, `chrome-untrusted://`, `devtools://`, `edge://`, `brave://`, `vivaldi://`, `opera://`) before `listTabs`, tab aliases, and default `ensureTabAvailable` selection, using the existing CDP filter pattern. Add regression coverage for Chrome MCP attach/list failure reason propagation and for a Chrome MCP tab list where an internal page precedes a real tab. Keep larger FSB-style owned-tab/extension-bridge ideas out of scope. Add a changelog entry if the diagnostics are user-visible.
