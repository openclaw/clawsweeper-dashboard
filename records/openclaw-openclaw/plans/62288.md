# Coding Plan for openclaw/openclaw#62288: Browser: prefer managed openclaw lane; existing-session attach brittle at DevToolsActivePort; improve fallback and diagnostics

Source report: [records/openclaw-openclaw/items/62288.md](records/openclaw-openclaw/items/62288.md)

## Summary

Keep open. Current main already prefers the managed `openclaw` browser lane and has partial Chrome MCP diagnostics, but the canonical existing-session repair is still incomplete because attach failure reasons are not carried through profiles/doctor output and Chrome MCP page selection can still treat browser-internal pages as normal tabs.

## Plan

Fix the canonical browser existing-session attach/fallback diagnostics issue. Keep `openclaw` as the managed default and do not build a new extension bridge. In the browser plugin, preserve Chrome MCP attach failure reason codes through status, profiles, browser doctor, and CLI output so failures distinguish missing Chrome, wrong profile/userDataDir, approval/remote-debugging, DevToolsActivePort, npx/spawn, transport, and no-tabs/page-tool failures. Filter browser-internal Chrome MCP pages such as chrome://, devtools://, edge://, brave://, vivaldi://, and opera:// before default target selection, matching the CDP target filter behavior; explicit target IDs may still be handled by existing target resolution. Add focused regression tests for mocked attach/list failures and internal-page-before-real-tab selection. Coordinate related Desktop-specific npx/EasyClaw cases only by preserving reason codes; do not solve OAuth or packaging-specific product work in this PR. Add a changelog entry if the final PR is user-visible.
