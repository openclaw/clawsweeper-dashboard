## Summary

Keep open. Current main and the latest release still render Telegram fenced code blocks as static `<pre><code>` HTML without preserving the fence language, while the related WebChat copy-button work is a separate renderer path.

## Plan

Implement https://github.com/openclaw/openclaw/issues/10519 by preserving a sanitized fenced-code language for Telegram HTML output. Expected behavior: a fenced block such as ```js renders as `<pre><code class="language-js">...`, while unfenced or indented code blocks remain valid and existing escaping, file-reference wrapping, chunk splitting, and Telegram parse-mode behavior are preserved. Prefer an additive shared Markdown IR/render-aware chunking change if it can keep the plugin-sdk text-runtime contract backwards-compatible; otherwise use a narrow Telegram-local fence-aware path. Do not interpolate raw `token.info`; extract/normalize only the first language token and HTML-attribute escape it. Add or update focused tests for `markdownToTelegramHtml`, `markdownToTelegramHtmlChunks`, file-reference wrapping inside code, and render-aware chunking. Add a user-facing `CHANGELOG.md` entry unless maintainers decide this is internal-only. Stop and ask for maintainer review if the implementation requires a broader public SDK redesign.
