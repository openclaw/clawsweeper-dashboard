# Coding Plan for openclaw/openclaw#10519: Telegram: Add language class to code blocks for native copy button support

Source report: [records/openclaw-openclaw/items/10519.md](records/openclaw-openclaw/items/10519.md)

## Summary

Keep open. Current main and the latest release still render Telegram fenced code blocks as static `<pre><code>` blocks, the existing formatter test asserts the no-language-class output, and the related WebChat copy-button work is a separate UI renderer path.

## Plan

Fix https://github.com/openclaw/openclaw/issues/10519 by preserving fenced Markdown language metadata for Telegram HTML output. Expected behavior: `markdownToTelegramHtml("```bash\necho hello\n```")` should emit `<pre><code class="language-bash">...`, while unfenced/indented code blocks keep plain `<pre><code>`. Keep the fix owner-scoped to Telegram/shared Markdown rendering: either add additive optional metadata to `MarkdownStyleSpan`/`MarkdownIR` and preserve it through slicing/chunking, or implement a narrow Telegram fence-aware renderer if that avoids a wider contract change. Sanitize/normalize the first fence info token before putting it in an HTML attribute; do not emit raw `token.info`. Preserve existing escaping, file-reference wrapping, blockquote/spoiler/link nesting, chunk-limit behavior, and Telegram HTML fallback behavior. Add focused regression coverage for fenced language classes, no-language fences, unsafe fence info, chunked code blocks, and file-reference wrapping; add a user-facing changelog entry.
