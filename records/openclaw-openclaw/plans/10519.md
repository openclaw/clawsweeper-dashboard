## Summary

Keep open. Current `main` and the latest tagged release still render Telegram fenced code blocks as plain `<pre><code>` without preserving the fence language, and the closed related PR attempts were not merged into `main`.

## Plan

Implement the Telegram fenced-code language preservation requested by https://github.com/openclaw/openclaw/issues/10519. Preserve a sanitized first fence-language token through the shared Markdown IR/render-aware chunking path, or use a narrowly scoped Telegram renderer only if the shared IR change becomes unsafe. Emit `class="language-X"` for safe Telegram fenced code blocks, leave unlabeled or invalid fences as `<pre><code>`, and do not loosen existing Telegram HTML escaping, file-reference wrapping, parse-mode fallback, or chunk splitting. Treat closed PRs https://github.com/openclaw/openclaw/pull/83623, https://github.com/openclaw/openclaw/pull/83770, and https://github.com/openclaw/openclaw/pull/83817 as useful prior attempts, not merged fixes. Add focused regression tests for labeled, unlabeled, invalid, escaped, wrapped, and chunked code-block behavior. Include a concise `CHANGELOG.md` entry for the user-facing Telegram formatting change if this is an OpenClaw-owned repair PR. After the patch, collect redacted real Telegram proof that a fenced code block reaches Telegram with copy-button-compatible code markup; redact private details such as tokens, phone numbers, chat IDs, IPs, and non-public endpoints.
