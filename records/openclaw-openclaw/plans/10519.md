# Coding Plan for openclaw/openclaw#10519: Telegram: Add language class to code blocks for native copy button support

Source report: [records/openclaw-openclaw/items/10519.md](records/openclaw-openclaw/items/10519.md)

## Summary

Keep open. Current main and the latest release still render Telegram fenced code blocks as static `<pre><code>` blocks without preserving the fence language, while Telegram’s documented HTML format supports `class="language-..."` on preformatted code.

## Plan

Implement Telegram fenced-code language preservation for https://github.com/openclaw/openclaw/issues/10519. Current `markdownToTelegramHtml("```js\nconst x = 1;\n```")` emits `<pre><code>...`; expected Telegram HTML should emit a sanitized `class="language-js"` on fenced code blocks while preserving current behavior for unfenced code blocks. Prefer carrying optional fence-language metadata through the shared Markdown IR/render-aware chunking path if that can be done as an additive text-runtime contract; use a narrow Telegram-local renderer only if shared IR expansion is clearly excessive. Do not change unrelated Telegram send behavior, parse mode fallback, file-reference wrapping, table rendering, or WebChat UI behavior. Add focused tests for fenced language output, no-language fences, HTML escaping/attribute sanitization, file-reference wrapping inside code blocks, and chunked code blocks. Add a user-facing changelog entry.
