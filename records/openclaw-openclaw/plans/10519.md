## Summary

Keep open. Current main still renders Telegram fenced code blocks with a static `<pre><code>` marker, the formatter test still asserts output without a language class, and the latest release has the same behavior; the WebChat copy-button path is separate and does not change Telegram Bot API HTML.

## Plan

Fix https://github.com/openclaw/openclaw/issues/10519 by preserving fenced code language identifiers for Telegram HTML output. Current main renders fenced blocks as `<pre><code>...`; expected output for a fenced language such as `bash` or `js` is `<pre><code class="language-bash">...` or `<pre><code class="language-js">...`. Implement this in the Telegram markdown render/chunk path with sanitized language metadata, preferably as an additive shared Markdown IR/render marker capability if needed, or a narrowly scoped Telegram-local path if that avoids wider SDK churn. Preserve existing Telegram HTML escaping, file-reference wrapping, parse_mode HTML compatibility, and render-aware chunk splitting. Do not change the WebChat copy-button renderer as part of this fix. Add/update regression coverage for fenced language blocks, unfenced/indented code blocks, escaping of unsafe language strings, file-reference wrapping, and chunk-limit behavior. Add a user-facing changelog entry because Telegram output changes.
