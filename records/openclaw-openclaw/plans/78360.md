# Coding Plan for openclaw/openclaw#78360: [Bug]: TUI stream watchdog marks quiet active runs idle in 2026.5.4

Source report: [records/openclaw-openclaw/items/78360.md](records/openclaw-openclaw/items/78360.md)

## Summary

Keep open: current main and the latest release still let the TUI watchdog clear an active run to idle and emit the reported prompt based on stream silence alone, while the gateway already has active-run state that the TUI does not consume for this decision.

## Plan

Fix the TUI quiet-active-run watchdog bug reported in https://github.com/openclaw/openclaw/issues/78360. Scope the repair to TUI session info and watchdog behavior: carry an authoritative active-run signal such as `sessions.list.hasActiveRun` into the TUI state/session info path, and before the streaming watchdog clears `activeChatRunId`, sets idle, or logs the taking-longer prompt, refresh or consult that state. If the current session still has an active run, keep/rearm the working/streaming state and do not append the prompt. Only clear to idle when the run is terminal, no longer active, or the stream is genuinely stale. Preserve existing recovery for lost final events, reconnect gaps, aborted/error/final runs, local `/btw` handling, deferred history refreshes, and concurrent-run ownership. Do not add a public config option, change CLI watchdog defaults, or implement the separate timeout-policy request in https://github.com/openclaw/openclaw/issues/68596. Add focused regression coverage in the TUI watchdog tests and any needed session-info/gateway active-run tests. Add a single-line `CHANGELOG.md` fix entry thanking @flashosophy if this user-facing fix is prepared for landing.
