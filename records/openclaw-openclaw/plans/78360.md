# Coding Plan for openclaw/openclaw#78360: [Bug]: TUI stream watchdog marks quiet active runs idle in 2026.5.4

Source report: [records/openclaw-openclaw/items/78360.md](records/openclaw-openclaw/items/78360.md)

## Summary

Keep open: current main and the latest release still allow the TUI watchdog to clear an active run and show the reported timeout prompt based on stream silence alone, while the gateway already exposes active-run state that the TUI does not currently carry into its session info path.

## Plan

Fix the TUI streaming watchdog false-idle path for quiet active runs. Current main still clears `activeChatRunId`, sets idle, and logs `This response is taking longer than expected. Send another message to continue.` from `src/tui/tui-event-handlers.ts` solely because no chat/tool event arrived before the watchdog timer. Use the existing gateway `sessions.list.hasActiveRun` signal, or an equivalent authoritative active-run state already exposed by the gateway, before clearing the active run: refresh or consume session info on watchdog expiry, keep or rearm streaming when the current session still has an active run, and only set idle/prompt when the run is actually no longer active or a terminal event is known. Add focused fake-timer regression coverage for a quiet active run and preserve the existing tool-event/reconnect behavior. Do not add a public config option as the primary fix; the open configurable-timeout request is separate. Add a changelog entry because this is user-visible TUI behavior.
