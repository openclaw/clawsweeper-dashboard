## Summary

Keep open: current main still rejects exact hosted custom-scheme origins, but this PR should not merge as written because it parses configured allowlist entries as URLs and can authorize a stripped origin from a path, query, or userinfo-bearing value.

## Plan

Repair PR https://github.com/openclaw/openclaw/pull/38290 or create a narrow replacement branch for the same bug. Preserve the observable fix: browser WebSocket clients with `Origin: chrome-extension://<id>` or other hosted custom schemes should pass when that exact `scheme://host` value is listed in `gateway.controlUi.allowedOrigins`. Keep the security boundary strict: configured allowlist entries must remain literal full origins or `*`; do not parse `https://control.example.com/path`, query strings, or userinfo-bearing values into broader allowed origins. Add or update focused tests in `src/gateway/origin-check.test.ts` for exact custom-scheme allow, mismatch reject, and negative path/query/userinfo allowlist cases; keep or port the integration coverage in `src/gateway/server.auth.browser-hardening.test.ts`. Do not implement scheme wildcards such as `chrome-extension://*`; leave that product/security decision to https://github.com/openclaw/openclaw/issues/39739. User-facing bug fix likely needs a maintainer-owned changelog entry at landing, not a contributor blocker.
