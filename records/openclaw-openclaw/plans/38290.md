## Summary

Keep open: current main and the latest release still have the exact-origin custom-scheme bug, and this PR contains useful proof, but the patch is not safe to merge as written because it broadens configured allowlist interpretation and permits empty-host opaque origins.

## Plan

Repair or replace this PR as a narrow Gateway origin-check fix. Preserve the useful behavior: exact hosted custom-scheme Origins such as `chrome-extension://abcdefghijklmnop`, `app://localhost`, `tauri://localhost`, and `electron://localhost` should match only when the exact `scheme://host` value is present in `gateway.controlUi.allowedOrigins`. Do not add scheme wildcards or new config. Keep configured allowlist entries exact trimmed/lowercased strings except `*`, so path-bearing entries like `https://trusted.example/app` do not begin authorizing `https://trusted.example`. Reject opaque origins without a non-empty host, such as `file://`, `file:///tmp/x`, and `data:`. Preserve current private-same-origin, local-loopback, trusted-proxy HTTP, and host-header fallback behavior. Add focused unit coverage in `src/gateway/origin-check.test.ts` and a WebSocket/browser-hardening regression in `src/gateway/server.auth.browser-hardening.test.ts`; include HTTP caller coverage only if needed by the implementation. Do not edit `CHANGELOG.md`.
