## Plan

Repair https://github.com/openclaw/openclaw/pull/38290 or create a narrow replacement branch for the same fix. The valid bug is that exact hosted custom-scheme browser origins such as `chrome-extension://<id>`, `app://localhost`, or `tauri://localhost` cannot match `gateway.controlUi.allowedOrigins` because current `parseOrigin()` compares `URL.origin === "null"`. Fix only the parser/matcher boundary in `src/gateway/origin-check.ts`: preserve exact configured allowlist strings as trimmed lowercase values, keep `*` behavior unchanged, keep literal `Origin: null` rejected, and fall back to `scheme://host` only for incoming hosted custom-scheme origins with a host. Add/update regression coverage in `src/gateway/origin-check.test.ts` for accept, reject, and no-broadening cases such as `allowedOrigins: ["https://trusted.example/path"]` not authorizing `Origin: https://trusted.example`. Preserve or adapt the Gateway WebSocket handshake coverage in `src/gateway/server.auth.browser-hardening.test.ts`. Do not add scheme-level wildcard support; that remains tracked separately at https://github.com/openclaw/openclaw/issues/39739. Do not change token/device auth, Host-header fallback, private same-origin behavior, docs policy, or unrelated Gateway code. Changelog is maintainer landing work unless explicitly requested.
