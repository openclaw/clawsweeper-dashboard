# Coding Plan for openclaw/openclaw#7433: Improve WhatsApp/Baileys group message reliability

Source report: [records/openclaw-openclaw/items/7433.md](records/openclaw-openclaw/items/7433.md)

## Summary

Keep open. Current main and the latest release still lack the requested Baileys `getMessage`, socket-level `cachedGroupMetadata`, proactive group/participant update handlers, and LID update wiring; the merged reconnect work and newer durable inbound state are useful partial context but do not solve the central retry/cache request.

## Plan

Fix the WhatsApp/Baileys reliability gaps tracked by this issue. Keep the change plugin-local under `extensions/whatsapp`; do not add core API, config surface, or unbounded/persistent message-content storage. Wire Baileys `getMessage` into socket construction using a bounded account-scoped recent-message content store that is populated early enough for sender-side retransmit requests, pass existing group metadata state through `cachedGroupMetadata`, and add proactive handlers for `groups.upsert`, `groups.update`, `group-participants.update`, and `lid-mapping.update` without creating a stale parallel LID source of truth. Preserve the existing reconnect fallback behavior from https://github.com/openclaw/openclaw/pull/73732 and the durable inbound state behavior on current main. Add focused regression tests for socket options, retry lookup, group cache event updates, participant updates, and LID update handling. Changelog is maintainer-owned; do not edit `CHANGELOG.md` unless explicitly asked.
