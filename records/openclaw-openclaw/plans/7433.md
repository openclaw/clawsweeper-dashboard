# Coding Plan for openclaw/openclaw#7433: Improve WhatsApp/Baileys group message reliability

Source report: [records/openclaw-openclaw/items/7433.md](records/openclaw-openclaw/items/7433.md)

## Summary

Keep open: current main still omits the requested Baileys retry and group-cache hooks, and the merged reconnect work is explicitly a partial fallback rather than the broader `getMessage`, `cachedGroupMetadata`, group event, participant event, and LID update wiring requested here. The issue discussion adds useful operational context, especially that `getMessage` needs content available before send acknowledgment to satisfy sender-side retry requests.

## Plan

Fix https://github.com/openclaw/openclaw/issues/7433 in the WhatsApp plugin only. Wire Baileys retry/cache reliability hooks without changing core SDK contracts or adding new public config: pass a bounded, account-scoped `getMessage` callback to `makeWASocket`; pass socket-level `cachedGroupMetadata`; keep the existing reconnect metadata fallback; subscribe to `groups.upsert`, `groups.update`, `group-participants.update`, and `lid-mapping.update` where supported by Baileys 7.0.0-rc10; and reuse Baileys' existing LID signal repository/mapping files rather than creating a stale parallel identity store. Keep recent message content retention privacy-preserving and in-memory by default unless maintainers explicitly approve persistence. Avoid duplicating reconnect/offline backfill work tracked by https://github.com/openclaw/openclaw/issues/50093 and https://github.com/openclaw/openclaw/pull/76175. Add focused regression tests for the socket options and event handlers, update the active changelog if behavior changes, and stop for maintainer review if the implementation would require a new config option or product policy choice.
