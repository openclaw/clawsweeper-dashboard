# Coding Plan for openclaw/openclaw#82525: channel.telegram start-account: getMe retries block event loop during api.telegram.org slow periods

Source report: [records/openclaw-openclaw/items/82525.md](records/openclaw-openclaw/items/82525.md)

## Summary

Keep open. Current main only partially mitigates the reported Telegram startup slow-path: it limits startup probes to two slots, but `startAccount` still waits on `getMe` with a 15s startup budget and the probe still retries within that budget, so the slow-api/event-loop report is not already fixed or safely duplicate.

## Plan

Fix the Telegram startup slow-api bug reported in https://github.com/openclaw/openclaw/issues/82525 in a narrow PR. Current main already has `withTelegramStartupProbeSlot` with concurrency 2 and a 15s startup `getMe` budget; make slow or intermittent `api.telegram.org/getMe` stop blocking `channels.telegram.start-account` across multi-account gateway startup. Stay within the Telegram plugin and channel startup surfaces; do not add a new user-facing config option or broad Gateway scheduler rewrite. Preserve 401 invalid-token fatal behavior, preserve `openclaw channels status --probe` longer health-check behavior, and avoid regressing slow-but-healthy Docker/Windows Telegram setups from https://github.com/openclaw/openclaw/issues/76459. Consider a startup-specific short budget/backoff/stagger or stricter limiter, and ensure a timed-out startup probe does not immediately trigger another blocking grammY `bot.init()` `getMe` on the hot path without backoff. Add focused tests for multi-account slow probes, timeout/backoff behavior, successful `botInfo` handoff, and 401 handling. Add a changelog entry if the runtime behavior changes.
