# Coding Plan for openclaw/openclaw#78289: [Bug] Feishu plugin: message read returns \"[Interactive Card]\" instead of actual card content

## Summary

Keep open: current main still routes Feishu single-message reads and thread-history reads through API calls that omit Feishu's documented `card_msg_content_type=user_card_content` query parameter, so interactive cards can still degrade to `[Interactive Card]`. The direct fix PR is closed unmerged, while the remaining work is a narrow Feishu plugin bug fix.

## Plan

Fix https://github.com/openclaw/openclaw/issues/78289 in the Feishu plugin only. Add `card_msg_content_type: "user_card_content"` to the single-message read and thread-history list requests so interactive cards return the original card JSON; preserve existing parser fallbacks and do not fold in the separate fallback-parser scope tracked at https://github.com/openclaw/openclaw/issues/60380. Because `@larksuiteoapi/node-sdk@1.64.0` types do not expose this query parameter, use a narrow SDK-safe wrapper/raw request adapter or bounded local type rather than a broad `any`. Add focused tests asserting both request shapes and existing card extraction behavior. Add a single-line `CHANGELOG.md` Fixes entry for the user-visible Feishu read behavior. Validate with the focused Feishu test and changed-check lane; attach redacted live Feishu read proof if credentials are available.
