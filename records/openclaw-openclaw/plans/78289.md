# Coding Plan for openclaw/openclaw#78289: [Bug] Feishu plugin: message read returns \"[Interactive Card]\" instead of actual card content

Source report: [records/openclaw-openclaw/items/78289.md](records/openclaw-openclaw/items/78289.md)

## Summary

Keep open: current main still reads Feishu single messages and thread history without Feishu's documented full-card query parameter, so interactive cards can still arrive as fallback text. The direct fix PR is closed unmerged, and the broader open Feishu parser PR does not close this exact fetch-layer bug.

## Plan

Fix https://github.com/openclaw/openclaw/issues/78289 in the Feishu plugin only. In `extensions/feishu/src/send.ts`, make both `getMessageFeishu` and `listFeishuThreadMessages` request Feishu's documented `card_msg_content_type=user_card_content` so interactive-card reads can receive original card JSON. Because `@larksuiteoapi/node-sdk` 1.64.0 does not type this query key on `im.message.get` or `im.message.list`, use a local SDK-safe/raw `client.request` adapter or a narrowly typed wrapper rather than adding excess properties to the typed SDK payload. Preserve existing parser fallbacks and do not add a new config option, core channel behavior, or broad parser refactor. Add focused tests in `extensions/feishu/src/send.test.ts` asserting both get/list request the full-card parameter and still parse existing full-card JSON; keep broader fallback parser work coordinated with https://github.com/openclaw/openclaw/issues/60380. Add one user-facing `CHANGELOG.md` Fixes entry. If Feishu credentials are available, include redacted live proof of `message(action=read)` or thread history reading an interactive card.
