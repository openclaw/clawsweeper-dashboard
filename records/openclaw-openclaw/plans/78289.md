## Summary

Keep open: current main still routes Feishu `message(action=read)` through helpers that omit any card-content return-format parameter, so interactive card reads can still fall back to `[Interactive Card]`. The direct fix attempt at https://github.com/openclaw/openclaw/pull/78302 is closed unmerged, and https://github.com/openclaw/openclaw/issues/54972 was closed as duplicate/superseded in favor of this issue, so this remains the canonical open tracker.

## Plan

Fix the Feishu plugin bug where `message(action=read)` and thread-history reads return `[Interactive Card]` instead of actual interactive card text. Work only in the Feishu plugin/readback surface unless a small local SDK-safe request helper is needed. First validate the upstream Feishu/Lark contract for `card_msg_content_type` and the correct value; this issue cites `user_card_content`, while related prior work and the external Lark/OpenClaw parser use `raw_card_content`, and `@larksuiteoapi/node-sdk@1.63.1` types do not expose either field on typed `im.message.get`/`list` payloads. Add the validated parameter to both `getMessageFeishu` and `listFeishuThreadMessages` through a type-safe or intentionally isolated raw-request boundary, preserve existing parser fallbacks, and add focused tests asserting the request parameters and full-card parsing. Include a `CHANGELOG.md` Fixes entry. Check the overlapping open Feishu interactive-card PR before editing and do not make broad core, channel SDK, or unrelated parser refactors.
