# Coding Plan for openclaw/openclaw#78289: [Bug] Feishu plugin: message read returns \"[Interactive Card]\" instead of actual card content

Source report: [records/openclaw-openclaw/items/78289.md](records/openclaw-openclaw/items/78289.md)

## Summary

Keep open: current main still routes Feishu `message(action=read)` and thread-history reads through fetch calls that omit Feishu's documented full-card return-format parameter, so interactive-card reads can still degrade to `[Interactive Card]`. The direct fix PR is closed unmerged, and the remaining work is a narrow Feishu plugin bug fix rather than stale cleanup.

## Plan

Fix https://github.com/openclaw/openclaw/issues/78289. In the Feishu plugin, make `message(action=read)` and Feishu thread-history reads request full interactive-card JSON from Feishu by using the documented `card_msg_content_type: "user_card_content"` query parameter for both message get and message list. Because `@larksuiteoapi/node-sdk` 1.64.0 types do not expose this param, add a small local typed/raw request adapter or a tightly documented wrapper that demonstrably forwards the query; do not change core plugin APIs, non-Feishu channels, outbound card rendering, or broad parser semantics. Add/update regression tests in `extensions/feishu/src/send.test.ts` that assert both request shapes include the parameter and that richer card content still parses. Add a single-line `CHANGELOG.md` Fixes entry because the behavior is user-visible. Keep parser-only fallback gaps scoped to https://github.com/openclaw/openclaw/issues/60380. If Feishu credentials are available, attach redacted live proof of reading an interactive card after the fix.
