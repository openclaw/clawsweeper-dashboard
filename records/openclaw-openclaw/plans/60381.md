# Coding Plan for openclaw/openclaw#60381: browser tool: add force parameter for click and expose evaluate action

Source report: [records/openclaw-openclaw/items/60381.md](records/openclaw-openclaw/items/60381.md)

## Summary

Keep open: current main and the latest release already expose browser `evaluate`, but the requested opt-in `force` click flag is still absent from the browser action schema, normalized request type, route normalizer, CLI/docs surface, and Playwright click forwarding path. The FSB comments are useful prior art for keeping forced clicks explicit and auditable instead of making them the default.

## Plan

Implement only the remaining force-click option for the bundled browser plugin. Current main and `v2026.6.5` already implement browser `evaluate`, so do not rework the evaluate path except to avoid regressions. Add an explicit `force?: boolean` click field to nested and flattened `BrowserToolSchema`, `BrowserActRequest`, `/act` normalization, Browser CLI click command if CLI parity is intended, docs/tool guidance, and Playwright `locator.click`/`locator.dblclick` forwarding. Preserve the default behavior when `force` is omitted or false. Make forced click auditable in wording: normal click remains the default, and `force: true` is a last-resort explicit bypass for covered/intercepted elements. For existing-session Chrome MCP profiles, either implement a supported equivalent with focused tests or reject `force` with a clear unsupported-action contract. Add focused tests for schema exposure, normalizer carry-through, default false/omitted behavior, Playwright forwarding for click and doubleClick, CLI argument parsing if added, and existing-session support/rejection. Do not edit `CHANGELOG.md`; put release-note context in the PR body.
