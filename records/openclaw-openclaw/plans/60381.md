# Coding Plan for openclaw/openclaw#60381: browser tool: add force parameter for click and expose evaluate action

Source report: [records/openclaw-openclaw/items/60381.md](records/openclaw-openclaw/items/60381.md)

## Summary

Keep open: current main implements the gated `evaluate` action, but the requested opt-in `force` click parameter is still missing from the browser tool schema, normalized action type, CLI, and Playwright click forwarding path. The recent FSB comments are useful prior art for making forced clicks explicit and auditable instead of a default interaction mode.

## Plan

Implement the remaining `force?: boolean` browser click support for https://github.com/openclaw/openclaw/issues/60381. Keep the existing gated `evaluate` action intact. Add an explicit opt-in `force` field to the browser act schema, `BrowserActRequest` click type, normalization, CLI `openclaw browser click` command, managed Playwright `clickViaPlaywright` path, docs, browser-automation skill guidance, and focused tests. Forward `force` to Playwright `locator.click` and, if supporting `doubleClick`, to `locator.dblclick`; preserve default behavior when `force` is absent or false. Do not make force click the default, do not weaken `browser.evaluateEnabled`, and do not bypass current navigation/SSRF/dialog guards. For Chrome MCP existing-session profiles, either implement a tested semantic equivalent or return a clear unsupported error when `force` is requested; document whichever behavior is chosen. Add a user-facing changelog entry under the active Changes section.
