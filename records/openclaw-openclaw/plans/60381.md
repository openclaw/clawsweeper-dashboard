# Coding Plan for openclaw/openclaw#60381: browser tool: add force parameter for click and expose evaluate action

Source report: [records/openclaw-openclaw/items/60381.md](records/openclaw-openclaw/items/60381.md)

## Summary

Keep open. Current main already has the requested gated `evaluate` action, but the `force` click option is still missing from the browser act schema, request type, normalizer, CLI surface, and Playwright click forwarding path; the May 12 comment also adds useful guidance that force clicks should stay explicit and auditable.

## Plan

Fix https://github.com/openclaw/openclaw/issues/60381 by adding an explicit opt-in `force?: boolean` option for browser `act` click requests. Keep the existing gated `evaluate` implementation; do not fold in statement-style evaluate work from https://github.com/openclaw/openclaw/issues/51393. Add `force` to the browser tool schema, `BrowserActRequest` click type, `/act` normalizer, legacy flattened act keys if needed, Playwright `clickViaPlaywright` options, `/act` dispatch, CLI click command if the CLI exposes click options, docs, browser automation skill guidance, and regression tests. Preserve the default non-force click path; document force as a fallback after normal click failure with an explicit reason/ref or selector in user guidance. For existing-session Chrome MCP clicks, either implement equivalent behavior if the driver supports it or reject/document the option clearly instead of silently ignoring it. Add a user-facing changelog entry unless maintainers decide this remains internal.
