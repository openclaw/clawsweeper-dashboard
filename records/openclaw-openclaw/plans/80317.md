# Coding Plan for openclaw/openclaw#80317: TTS OpenAI provider: MP3 responseFormat not voice-compatible for Telegram, unlike Edge TTS

## Summary

Keep open: current main still has the reported OpenAI-provider gate, so an explicit OpenAI `responseFormat: "mp3"` can synthesize MP3 for Telegram while failing to mark the result as voice-compatible, even though Telegram routing and shared media policy already accept MP3 voice sends.

## Plan

Fix https://github.com/openclaw/openclaw/issues/80317. Current main's OpenAI speech provider sets `voiceCompatible` only for `responseFormat === "opus"`, so Telegram TTS with `messages.tts.providers.openai.responseFormat: "mp3"` falls back to `sendAudio` even though Telegram/shared media policy accepts MP3 voice sends. Update `extensions/openai/speech-provider.ts` so voice-note target MP3 is voice-compatible too, preferably by reusing the shared `isVoiceCompatibleAudio` policy with the provider's computed file extension or by using an explicit `opus|mp3` allowlist. Keep WAV/PCM non-voice-compatible unless a real transcode path is added. Add/update regression tests in `extensions/openai/speech-provider.test.ts` for `responseFormat: "mp3"` on `voice-note` returning `voiceCompatible: true`, and keep the existing WAV/non-compatible behavior covered. If needed, add a speech-core/Telegram-path test proving MP3 OpenAI TTS sets `audioAsVoice`. Add a single active-version `CHANGELOG.md` Fixes entry because this is user-visible. Do not change Telegram auth/routing policy, add a new config option, or introduce broad audio transcoding.
