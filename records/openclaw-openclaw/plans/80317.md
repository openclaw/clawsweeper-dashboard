# Coding Plan for openclaw/openclaw#80317: TTS OpenAI provider: MP3 responseFormat not voice-compatible for Telegram, unlike Edge TTS

Source report: [records/openclaw-openclaw/items/80317.md](records/openclaw-openclaw/items/80317.md)

## Summary

Keep open: current main and the latest stable tag still mark OpenAI TTS voice-note output as compatible only for Opus, so configured MP3 output still misses Telegram voice delivery despite the shared Telegram/media path already accepting MP3.

## Plan

Fix https://github.com/openclaw/openclaw/issues/80317. On current main, `extensions/openai/speech-provider.ts` marks OpenAI TTS voice-note output as `voiceCompatible` only for `responseFormat === "opus"`; when a Telegram voice-note request uses an OpenAI-compatible provider configured for `mp3`, speech-core does not set `audioAsVoice` and Telegram falls back to `sendAudio`. Update the OpenAI provider so `responseFormat === "mp3"` is voice-compatible for `voice-note` targets, matching the existing shared Telegram/media MP3 compatibility. Keep WAV and other non-compatible formats false, do not add config options, and do not change cross-channel policy. Add regression coverage in `extensions/openai/speech-provider.test.ts` for `target: "voice-note"` plus `responseFormat: "mp3"` returning `outputFormat: "mp3"`, `fileExtension: ".mp3"`, and `voiceCompatible: true`; preserve the existing WAV false case. Reference the closed unmerged source PR https://github.com/openclaw/openclaw/pull/80326 for intent, but implement against current main. Add a changelog entry because this changes user-visible Telegram TTS behavior. Validate with the focused OpenAI provider test and routing tests if touched.
