# Coding Plan for openclaw/openclaw#62891: [v4.5] Session-store reconciliation on startup blocks main process, causing full fallback chain timeout for cron/LLM requests

Source report: [records/openclaw-openclaw/items/62891.md](records/openclaw-openclaw/items/62891.md)

## Summary

Keep open. Current main still starts or restarts cron without waiting for delivery-capable channel transport readiness, while the docs promise startup cron deferral out of the channel-connect window, so the Discord/SIGUSR1 race remains source-reproducible.

## Plan

Fix the cron/channel readiness race tracked by https://github.com/openclaw/openclaw/issues/62891. Current main can start cron after post-ready maintenance without checking delivery-capable channel transport readiness, and config.patch/SIGUSR1 hot reload restarts cron before affected channels restart. Implement a bounded generic cron delivery-readiness gate for initial startup and hot reload: wait for configured, enabled, delivery-capable channel accounts that cron could deliver through to become connected, or time out with clear diagnostics without wedging cron for disabled/unconfigured/skipped/unmanaged channels. Keep the fix in core gateway/cron/channel readiness seams; do not add a new user-facing config option unless absolutely necessary, do not rely on fixed sleeps or session-store lock timing, and do not try to solve broad LLM fallback behavior from https://github.com/openclaw/openclaw/issues/43374 or inbound Discord replay from https://github.com/openclaw/openclaw/issues/52577 in the same PR. Add focused regression tests for startup and hot reload ordering plus a slow Discord/channel readiness case. Changelog is expected because this is user-visible reliability behavior.
