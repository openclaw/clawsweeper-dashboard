## Summary

Keep open: current main still has a source-visible race where cron can start or restart without waiting for delivery-capable channel transport readiness, so the Discord cron/SIGUSR1 failure window remains actionable.

## Plan

Fix https://github.com/openclaw/openclaw/issues/62891 by adding a bounded generic delivery-readiness gate before cron can start on initial gateway post-ready activation and before cron is restarted during config.patch/SIGUSR1 hot reload. Use existing channel runtime snapshots/connected status rather than fixed sleeps; skip channels/accounts that are disabled, unconfigured, manually stopped, skipped by OPENCLAW_SKIP_CHANNELS/OPENCLAW_SKIP_PROVIDERS, or not delivery-capable; log useful timeout diagnostics and continue safely after the bounded wait. Keep the fix scoped to cron/channel readiness and do not change fallback timeout budgeting, provider transport policy, ACPX startup architecture, or add a public config option unless maintainers explicitly request it. Add regression coverage for startup and hot reload ordering, plus Discord-style connected-status handoff behavior, and add a user-facing changelog fix entry if landing policy requires one.
