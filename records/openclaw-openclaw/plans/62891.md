## Summary

Keep open. Current main and the latest shipped tag still start or restart cron without waiting for delivery-capable channel transport readiness, so the Discord/SIGUSR1 cron race remains source-reproducible.

## Plan

Fix https://github.com/openclaw/openclaw/issues/62891 by adding a bounded generic cron delivery-readiness gate for initial gateway startup and config.patch/SIGUSR1 hot reload. Cron must not begin executing delivery-capable due jobs until configured channel transports that can receive deliveries are connected, or until a bounded timeout logs clear diagnostics and proceeds safely. Keep the fix in generic gateway/cron/channel readiness seams; do not add provider-specific policy to core, do not rely on fixed sleeps or session-store lock timing, and stop if the implementation would require a new product/config decision. Add focused regression tests proving startup cron waits past sidecar handoff for channel readiness and hot reload does not restart cron before affected channel restart/readiness handling. Add a CHANGELOG.md fix entry because this is user-visible reliability behavior.
