## Summary

Keep open. Current main still starts or restarts cron without a bounded wait for delivery-capable channel transport readiness, so the Discord/config.patch cron race remains source-reproducible; the existing startup deferral only covers missed agent-turn catch-up jobs.

## Plan

Fix the cron/channel readiness race reported in https://github.com/openclaw/openclaw/issues/62891. Current main can start cron in post-ready maintenance and restart cron during config.patch/SIGUSR1 before configured delivery-capable channel transports, especially Discord, have reported connected=true. Add a bounded generic readiness gate around initial cron activation and hot-reload cron restart that waits for configured delivery-capable channel accounts to become connected, skips disabled/unconfigured/skipped/unmanaged channels, times out with clear diagnostics, and does not add a new config option unless the existing architecture already has an appropriate policy knob. Keep the fix scoped to gateway cron activation/reload and channel readiness; do not change model fallback budgeting, provider transport policy, Telegram concurrency, or unrelated session-lock behavior. Add focused regression coverage for post-ready startup and hot reload ordering, plus any channel readiness helper tests. Add a changelog entry if the final change is user-visible.
