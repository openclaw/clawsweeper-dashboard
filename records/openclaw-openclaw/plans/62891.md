# Coding Plan for openclaw/openclaw#62891: [v4.5] Session-store reconciliation on startup blocks main process, causing full fallback chain timeout for cron/LLM requests

Source report: [records/openclaw-openclaw/items/62891.md](records/openclaw-openclaw/items/62891.md)

## Summary

Keep open. Current main still has the cron/channel-readiness race described in the issue: startup cron is only deferred until sidecars return, and config.patch hot reload restarts cron before affected channels are restarted, while Discord transport readiness is recorded later through the channel status path.

## Plan

Fix the cron/channel-readiness race tracked by https://github.com/openclaw/openclaw/issues/62891. Add a bounded generic delivery-readiness gate for cron startup and config.patch/SIGUSR1 hot reload so due cron jobs that may deliver to chat channels do not run before configured delivery-capable channel transports have connected or a clear timeout has elapsed. Keep the fix core-generic; do not hard-code Discord policy, do not rely on the sessions.json lock, and do not add a fixed sleep as the primary mechanism. Preserve behavior for disabled/unconfigured/skipped/unmanaged channels and for installations with no channel delivery. Add focused regression coverage for startup post-attach/post-ready cron activation, hot reload ordering when restartCron and restartChannels are both true, and channel handoff versus connected status. Keep the broader LLM fallback timeout issue out of scope. Add or prepare a single-line changelog Fixes entry if this is landed as a user-visible reliability fix.
