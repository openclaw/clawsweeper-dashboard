## Summary

Keep open. Current main still defers cron only until sidecar/post-attach handoff, and hot reload still starts cron before channel restarts, so the Discord transport readiness race remains source-reproducible.

## Plan

Fix https://github.com/openclaw/openclaw/issues/62891. The observable bug is that cron scheduled jobs can run after gateway sidecar handoff but before delivery-capable channel transports, especially Discord, are connected; config.patch hot reload also restarts cron before restarting affected channels. Implement a bounded, plugin-agnostic cron delivery-readiness gate for initial startup and cron hot reload using channel manager runtime snapshots/connected status. Do not add a new config knob unless a maintainer explicitly asks, do not rely on fixed sleeps or session-store locks, and keep core independent of Discord-specific policy. Add regression tests for initial startup deferral and hot reload ordering/readiness wait. Before handoff, run the focused gateway/cron tests and oxfmt on touched files; live Discord restart proof is recommended before landing.
