# Coding Plan for openclaw/openclaw#50186: [Feature]: Surface systemd gateway hygiene issues in doctor/status and consider safer KillMode defaults

Source report: [records/openclaw-openclaw/items/50186.md](records/openclaw-openclaw/items/50186.md)

## Summary

Keep open. Current main and v2026.5.7 already use `KillMode=control-group` for generated Linux gateway units, but doctor/status still do not collect or expose `KillMode`, `TasksCurrent`, or `MemoryCurrent`, so the central observability request remains valid.

## Plan

Implement the remaining diagnostic portion of https://github.com/openclaw/openclaw/issues/50186. Add read-only Linux systemd hygiene collection for the managed gateway service: request and parse `KillMode`, `TasksCurrent`, `MemoryCurrent`, and keep existing `MainPID` behavior; carry these fields through `GatewayServiceRuntime`; print them in `openclaw gateway status --deep`/JSON where service runtime details are shown; and add conservative doctor/status warnings or notes when `KillMode=process` combines with unusually high task or memory counts. Do not change the generated systemd unit default, which already uses `KillMode=control-group`, and do not add a config option. Include focused unit tests for parsing, runtime query properties, status output, and doctor audit/reporting. Add a concise changelog entry because this is user-visible diagnostic output.
