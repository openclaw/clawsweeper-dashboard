## Summary

Keep open. Current main and v2026.5.18 already generate Linux gateway units with `KillMode=control-group`, but `openclaw doctor` and `openclaw gateway status --deep` still do not query, parse, carry, or print `KillMode`, `TasksCurrent`, or `MemoryCurrent`, so the central observability request remains valid.

## Plan

Implement the remaining focused diagnostics for https://github.com/openclaw/openclaw/issues/50186. Add read-only Linux systemd hygiene collection for the local gateway service so `openclaw gateway status --deep` and `openclaw doctor` can expose `KillMode`, `TasksCurrent`, `MemoryCurrent`, and `MainPID` when available. Warn conservatively for suspicious combinations such as `KillMode=process` with high task or memory counts, and include operator next-step commands like `systemctl --user show openclaw-gateway.service -p KillMode -p TasksCurrent -p MemoryCurrent` and `systemd-cgls --user-unit openclaw-gateway.service`. Do not change generated `KillMode=control-group`, do not add MemoryMax/TasksMax guardrails here, and do not silently migrate existing units. Add focused tests for parsing, status JSON/text output, and doctor warnings; add a changelog entry if the final PR is user-visible.
