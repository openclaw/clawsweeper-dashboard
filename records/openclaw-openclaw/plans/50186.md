# Coding Plan for openclaw/openclaw#50186: [Feature]: Surface systemd gateway hygiene issues in doctor/status and consider safer KillMode defaults

Source report: [records/openclaw-openclaw/items/50186.md](records/openclaw-openclaw/items/50186.md)

## Summary

Keep open: current main and the latest release already use `KillMode=control-group` for generated Linux systemd units, but doctor/status still do not collect or display `KillMode`, `TasksCurrent`, or `MemoryCurrent`, so the central observability request remains actionable.

## Plan

Implement the remaining diagnostics from https://github.com/openclaw/openclaw/issues/50186. Keep scope to observability: extend the Linux systemd runtime/service audit path to collect `KillMode`, `TasksCurrent`, `MemoryCurrent`, and `MainPID` where available; carry those values through the shared `GatewayServiceRuntime`/status data; surface them in `openclaw doctor` and `openclaw status --deep` human and JSON output; emit conservative warnings and concrete next-step commands when `KillMode=process` combines with suspicious task or cgroup memory counts. Do not add a process reaper, do not silently migrate existing units, and do not change the current generated `KillMode=control-group` default unless a maintainer explicitly asks. Add focused Vitest coverage for parsing, DTO propagation, doctor/status formatting, and warning thresholds; update docs/changelog if user-visible output changes.
