# Coding Plan for openclaw/openclaw#50186: [Feature]: Surface systemd gateway hygiene issues in doctor/status and consider safer KillMode defaults

Source report: [records/openclaw-openclaw/items/50186.md](records/openclaw-openclaw/items/50186.md)

## Summary

Keep open: current main already uses `KillMode=control-group` for generated Linux gateway units, but doctor/status still do not collect or report `KillMode`, `TasksCurrent`, or `MemoryCurrent`, so the requested hygiene visibility remains missing.

## Plan

Implement the diagnostic visibility requested by https://github.com/openclaw/openclaw/issues/50186. Add Linux systemd hygiene collection for the gateway service by extending the systemd runtime query/parser and `GatewayServiceRuntime` to carry `KillMode`, `TasksCurrent`, `MemoryCurrent`, and `MainPID` where available. Surface these fields in `openclaw gateway status --deep` human output and JSON, and add conservative doctor/status warnings with next-step commands when `KillMode=process` plus high task or cgroup memory values indicate degraded service hygiene. Keep scope to read-only diagnostics and warnings: do not change generated `KillMode` defaults, do not silently migrate existing units, do not add a configurable kill mode, and do not implement a process reaper or Codex sidecar lifecycle fix. Treat https://github.com/openclaw/openclaw/issues/84413 as related context only, not part of this fix. Add focused tests for parsing, DTO propagation, status formatting, and doctor warning output; update docs/changelog only if the user-visible status/doctor output changes require it.
