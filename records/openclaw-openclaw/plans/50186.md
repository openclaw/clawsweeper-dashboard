# Coding Plan for openclaw/openclaw#50186: [Feature]: Surface systemd gateway hygiene issues in doctor/status and consider safer KillMode defaults

Source report: [records/openclaw-openclaw/items/50186.md](records/openclaw-openclaw/items/50186.md)

## Summary

Keep open: current main and v2026.5.22 already use `KillMode=control-group` for generated Linux systemd units, but doctor/status still do not surface `KillMode`, `TasksCurrent`, or `MemoryCurrent`, so the central diagnostics request remains actionable.

## Plan

Implement the remaining observability portion of https://github.com/openclaw/openclaw/issues/50186. Add Linux user-systemd gateway hygiene diagnostics to `openclaw doctor` and/or `openclaw status --deep`: collect `KillMode`, `TasksCurrent`, `MemoryCurrent`, and `MainPID` from `systemctl --user show openclaw-gateway[-profile].service`; surface raw values in status/deep output or structured runtime summary; and add a focused doctor warning when `KillMode=process` combines with unusually high task or memory counts. Include actionable commands such as `systemctl --user show ... -p KillMode -p TasksCurrent -p MemoryCurrent -p MainPID`, `systemd-cgls --user-unit ...`, and restart guidance. Preserve the current generated-unit default `KillMode=control-group`; do not silently rewrite existing units or change repair policy unless the existing explicit doctor repair flow already asks the operator. Add focused unit tests for systemd show parsing, status formatting, and doctor service audit/warning behavior. Update docs if user-visible output changes; add a changelog entry if the implementation is user-facing.
