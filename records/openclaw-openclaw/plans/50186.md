# Coding Plan for openclaw/openclaw#50186: [Feature]: Surface systemd gateway hygiene issues in doctor/status and consider safer KillMode defaults

Source report: [records/openclaw-openclaw/items/50186.md](records/openclaw-openclaw/items/50186.md)

## Summary

Keep open: current main and v2026.5.19 already use `KillMode=control-group` for generated Linux gateway units, but doctor/status still do not collect or expose `KillMode`, `TasksCurrent`, or `MemoryCurrent`, so the requested hygiene visibility remains missing.

## Plan

Implement the remaining diagnostic half of https://github.com/openclaw/openclaw/issues/50186. Add Linux user-systemd hygiene collection for the gateway service without changing generated unit defaults or silently migrating existing units. Extend `readSystemdServiceRuntime`/`parseSystemdShow` and `GatewayServiceRuntime` to carry `KillMode`, `TasksCurrent`, `MemoryCurrent`, and `MainPID` where available; surface raw values in `openclaw gateway status --deep`, `openclaw status --deep` JSON/human service summaries, and `openclaw doctor` gateway notes. Emit conservative warnings when `KillMode=process` combines with unusually high task count or memory, and include actionable commands such as `systemctl --user show openclaw-gateway.service -p KillMode -p TasksCurrent -p MemoryCurrent` and `systemd-cgls --user-unit openclaw-gateway.service`. Add focused tests for parser/runtime DTO/status printing/doctor notes, preserve current `KillMode=control-group` default, avoid changing repair/migration policy, and add a concise changelog entry if maintainers treat the diagnostic output as user-facing.
