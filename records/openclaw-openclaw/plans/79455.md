## Summary

Keep open: current main still maps Telegram direct/private topic sends to `message_thread_id` and falls back by stripping that field, while the reporter's live Bot API probe and current Telegram docs indicate `direct_messages_topic_id` is now needed for direct messages chat topics. The related open work at https://github.com/openclaw/openclaw/issues/79448 and https://github.com/openclaw/openclaw/pull/79410 covers adjacent fallback/status behavior, not this field-selection mismatch.

## Plan

Fix https://github.com/openclaw/openclaw/issues/79455 in the Telegram plugin. Current main sends direct/private topic ids as `message_thread_id` and then falls back threadless on `message thread not found`; add support for Telegram Bot API `direct_messages_topic_id` for direct messages chat topics without widening forum-topic threadless fallback or blindly changing every DM/private-chat topic. Verify the current Bot API shape: inbound messages may expose `direct_messages_topic.topic_id` and chats may expose `is_direct_messages`; carry a small typed thread kind through `extensions/telegram/src/bot-message-context.ts`, `extensions/telegram/src/reply-parameters.ts`, and the delivery/send helpers so outbound text/media/poll/draft paths use the right field. Update tests that currently assert DM topics always use `message_thread_id`, add regression coverage for `direct_messages_topic_id`, keep forum/private-chat `message_thread_id` cases covered, and add a user-facing `CHANGELOG.md` fix entry. Stop and ask for maintainer input if the implementation would require a new config knob, dependency upgrade, or changing the shipped forum fail-closed policy.
