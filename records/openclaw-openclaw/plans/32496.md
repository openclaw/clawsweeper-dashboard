## Summary

Keep open. Current main still accepts arbitrary model params and offers `params.extra_body` as an advanced workaround, but it does not directly normalize or forward `frequency_penalty` or `presence_penalty`; the related implementation PRs were closed unmerged, and the broader generation-parameter issues do not replace this narrow request.

## Plan

Implement direct finite-number passthrough for `frequency_penalty` and `presence_penalty` from accepted model/agent `params` into supported OpenAI-compatible request payloads. Keep the scope narrow: do not add a new `generation` config block, do not broaden to `top_p`, `top_k`, or vendor-specific penalties unless already required for the two-field implementation, and preserve `params.extra_body` as the advanced escape hatch. Add regression coverage for direct params, non-finite/non-number rejection, existing `onPayload` chaining, and payload behavior on OpenAI-compatible routes; update public docs and CHANGELOG for the user-visible feature.
