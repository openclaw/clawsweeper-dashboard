# Coding Plan for openclaw/openclaw#63272: feat(cli): `skills uninstall` design proposal + prior art reconciliation

## Summary

Keep open: current main and v2026.6.8 still lack the CLI uninstall surface, and source inspection confirms stale ClawHub lockfile entries can reinstall manually deleted skills during `skills update --all`. The older feature request is closed in favor of this issue, while the open Gateway RPC PR is adjacent and does not provide the requested CLI command.

## Plan

Implement `openclaw skills uninstall <slug>` for workspace/shared ClawHub skill lifecycle. Add a CLI command under `src/cli/skills-cli.ts` that resolves the same target workspace rules as install/update, supports dry-run plus noninteractive confirmation, removes only the safe `skills/<slug>` directory, and removes the matching `.clawhub/lock.json` entry. Use existing lifecycle helpers such as `resolveWorkspaceSkillInstallDir`, `normalizeTrackedSkillSlug` or stricter requested-slug validation as appropriate, and `readClawHubSkillsLockfile`/`writeClawHubSkillsLockfile`; do not mutate sandbox copies, bundled skills, dashboard UI, plugin config, `openclaw.json`, or CHANGELOG.md. Coordinate semantics with https://github.com/openclaw/openclaw/pull/69319: require safe path containment, preserve idempotent not-installed behavior, and avoid deleting unrelated/manual skill folders unless the implemented CLI explicitly proves and documents that policy. Update focused tests in the skills lifecycle and CLI command/format suites, update `docs/cli/skills.md`, and include release-note context in the PR body rather than editing `CHANGELOG.md`.
