# Coding Plan for openclaw/openclaw#53890: Feature: Default outbound topic/thread binding per agent for Telegram

Source report: [records/openclaw-openclaw/items/53890.md](records/openclaw-openclaw/items/53890.md)

## Summary

Keep open because current main and the latest release still lack a Telegram config-level outbound topic binding; topic config remains inbound/session routing only, and proactive outbound threading has no per-agent default-topic lookup.

## Plan

Implement https://github.com/openclaw/openclaw/issues/53890 as a focused Telegram feature. Add a topic-level outbound default, likely `outbound?: boolean`, to Telegram topic config type/schema/docs; pass the active `agentId` into the channel `resolveAutoThreadId` contract from send and poll actions; update the Telegram resolver so explicit `threadId`, `threadId: null`, `topLevel: true`, and `:topic:` targets keep precedence, current-context auto-threading remains second, and a configured per-agent topic default is used only as fallback. Add validation or deterministic handling for duplicate outbound defaults for the same group/agent. Keep Telegram-specific policy in the Telegram plugin and avoid core bundled Telegram defaults beyond the typed resolver seam. Add regression tests for schema acceptance/rejection, explicit-topic precedence, fallback outbound default resolution, and poll/send propagation; update Telegram docs and changelog for the user-facing config feature.
