## Summary

Keep open. Current main still sends `stream_options.include_usage` on every OpenAI Chat Completions streaming request even when the resolved compat data says streaming usage is unsupported, so the Mindie-style strict-parameter 422 remains source-reproducible; related strict message-key work helps a different rejected-payload class but does not cover this top-level request parameter.

## Plan

Fix the Mindie/OpenAI-compatible 422 path by making OpenAI Chat Completions request construction respect the existing streaming-usage compatibility contract for strict backends without hardcoding Mindie. The fix should let a model that explicitly opts out with `models.providers.<provider>.models[].compat.supportsUsageInStreaming: false` avoid sending unsupported `stream_options.include_usage`, while preserving usage tracking for provider-declared compatible local/proxy backends such as vLLM, LM Studio, Ollama, llama.cpp-style paths where the existing tests expect usage requests, and any explicit `true` override. Do not add a Mindie-specific provider branch, do not remove tool/message strictness behavior already covered by `compat.supportsTools` and `compat.strictMessageKeys`, and do not revive the blanket one-line gate from https://github.com/openclaw/openclaw/pull/79958. Add focused regression tests for the negative strict-backend opt-out and the positive known/local compatible paths; update docs or config help only if the fix changes documented config semantics. Add a changelog entry because this is user-visible provider compatibility.
