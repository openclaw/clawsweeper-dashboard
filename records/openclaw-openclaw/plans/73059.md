# Coding Plan for openclaw/openclaw#73059: [Bug]: Windows linked source install can skip Control UI auto-build through junction-launched scripts/ui.js

Source report: [records/openclaw-openclaw/items/73059.md](records/openclaw-openclaw/items/73059.md)

## Summary

Keep open. Current main and the latest release still use the raw direct-execution path comparison in `scripts/ui.js`, while gateway first-start Control UI asset building still depends on that wrapper and no merged canonical PR fixes the junction-launched entrypoint case.

## Plan

Fix the Windows linked-source Control UI build no-op reported in https://github.com/openclaw/openclaw/issues/73059. Current `scripts/ui.js` only calls `main()` when `path.resolve(process.argv[1])` exactly equals `fileURLToPath(import.meta.url)`, which can fail when launched through `%APPDATA%\npm\node_modules\openclaw` as a junction to the repo. Update direct-execution detection in `scripts/ui.js` to canonicalize both the entry path and module path using realpath with a resolved-path fallback, or an equivalent helper that works from this JS script. Add focused coverage in `test/scripts/ui.test.ts` for a linked/junction-style `argv[1]` that still runs the main path. Preserve the existing Windows cmd.exe launcher safeguards and do not broaden the UI dependency install behavior. Add a `CHANGELOG.md` fix entry because this affects source/global Windows installs. The adjacent closed PR https://github.com/openclaw/openclaw/pull/52291 covered Windows `pnpm.cmd` quoting and should not be treated as the fix for this direct-execution bug.
