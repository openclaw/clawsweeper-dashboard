# Coding Plan for openclaw/openclaw#78326: [Bug]: `parseStandalonePlainTextToolCallBlocks` doesn't recognize OpenAI Harmony-format tool calls (gpt-oss-120b on LM Studio)

Source report: [records/openclaw-openclaw/items/78326.md](records/openclaw-openclaw/items/78326.md)

## Summary

Keep open: current `main` and the latest release still only parse bracket-style plain-text tool-call blocks, while the LM Studio wrapper depends on that parser to promote text back into structured tool calls. The report is source-reproducible and narrow, but it is additive Harmony-format compatibility rather than already-documented bracket behavior.

## Plan

Fix https://github.com/openclaw/openclaw/issues/78326. Current `src/plugin-sdk/tool-payload.ts` only parses `[tool]` JSON `[END_TOOL_REQUEST]` / `[/tool]` blocks, while gpt-oss on LM Studio can surface Harmony text such as `commentary to=read code {"path":"/path/to/file","line_start":1,"line_end":400}` or `<|channel|>commentary to=read code<|message|>{...}`. Add additive Harmony-format support without regressing bracket parsing: require a registered/allowed tool name, require a JSON object, preserve the existing max payload limit, and keep parsing standalone-only. Also update `extensions/lmstudio/src/stream.ts` so Harmony-looking text remains buffered until `done` and does not leak as text before promotion. Add focused regression coverage in `src/plugin-sdk/tool-payload.test.ts` and `extensions/lmstudio/src/stream.test.ts` for shorthand Harmony, delimiter form, allowed-tool rejection, malformed JSON, existing bracket behavior, and no raw text events after promotion. Update `docs/gateway/local-models.md` and `CHANGELOG.md` if the behavior is user-facing. Do not add a config flag or broaden tool execution beyond the existing allowed-tool-name gate.
