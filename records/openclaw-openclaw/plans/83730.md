## Summary

Keep open: this is a fresh Feishu DM delivery regression with concrete logs, and current main plus the latest shipped tag still route p2p reply delivery through the inbound `oc_...` chat id even though the direct-user identity is the sender `open_id`. The related Feishu visible-reply and P2P-thread reports are useful context, but they do not cover this exact `queuedFinal=true, replies=1` plus `SUBSCRIPTION_NOT_FOUND` failure.

## Plan

Fix Feishu p2p DM final reply delivery for https://github.com/openclaw/openclaw/issues/83730. Current main receives p2p events with `chat_id=oc_...` and `sender.open_id=ou_...`, but direct final replies go through `createFeishuReplyDispatcher({ chatId: ctx.chatId, skipReplyToInMessages: true })`, so `sendMessageFeishu` sends to the `oc_...` chat id as `receive_id_type=chat_id`; the reporter sees `SUBSCRIPTION_NOT_FOUND`. Keep the change inside the Feishu plugin. Prefer routing ordinary non-thread p2p final replies to the sender direct target (`user:${ctx.senderOpenId}` or equivalent open_id path) or another Feishu-supported safe direct-send path. Do not change global `messages.visibleReplies` defaults, Codex provider policy, group policy, or unrelated channel routing. Preserve existing group/topic behavior and coordinate with the open P2P-thread repair path so p2p `thread_id`/`root_id` behavior is not regressed. Add focused regression tests that prove a plain p2p DM final reply does not target bare `oc_...` as `chat_id`, and that group/topic and p2p-thread routing still use the intended reply/thread anchors. Add a user-facing changelog fix entry if this repair PR is created.
