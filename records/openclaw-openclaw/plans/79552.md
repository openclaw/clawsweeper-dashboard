## Summary

Keep open: current main still exposes the Android reconnect race, and both linked fix PRs are closed unmerged, so notification events can still be lost before the Gateway connect handshake completes.

## Plan

Fix the Android gateway reconnect notification-loss bug tracked by https://github.com/openclaw/openclaw/issues/79552. Keep the Gateway protocol unchanged: the first WebSocket frame must remain `connect`. In Android, prevent `sendNodeEvent` and `sendNodeEventDetailed` from sending through a connection before the connect RPC/ACK has completed; prefer publishing `currentConnection` only after `conn.connect()` returns ready metadata, or use an explicit handshake-complete readiness guard if that better fits the existing lifecycle. Preserve notification events that arrive during reconnect with a bounded queue and/or active-notification reconciliation after reconnect; respect existing notification forwarding filters, avoid unbounded memory growth, and avoid duplicate agent wakeups. Add focused Android regression coverage for a delayed connect ACK and for notification events being retained/flushed or reconciled after reconnect. Include a user-facing changelog entry if the repair workflow owns changelog updates; otherwise note that landing needs one. Do not loosen server-side handshake validation or add a config flag.
