Keep open: current main and v2026.5.12 still expose the Android reconnect race, and both linked fix PRs are closed unmerged. The reporter's local patch comment is useful confirmation of the right direction: Android should wait for a completed gateway connect handshake and preserve notification events that arrive during reconnect.

## Plan

Fix https://github.com/openclaw/openclaw/issues/79552 on current main. Keep the Gateway protocol contract unchanged. In Android, prevent node.event sends from using a GatewaySession connection until the connect RPC has completed successfully, then preserve notifications.changed events that arrive during reconnect with bounded queueing/retry or active-notification reconciliation after reconnect. Reuse the useful direction from https://github.com/openclaw/openclaw/pull/79565 and https://github.com/openclaw/openclaw/pull/79927, but avoid the incomplete path that merely returns false before handshake and still drops notification events. Add focused Android regression coverage for sendNodeEvent/sendNodeEventDetailed during the pre-handshake window and notification replay/flush behavior. Preserve existing notification forwarding filters, quiet-hours policy, and rate limiting; do not change the gateway server to accept pre-connect node.event frames.
