## Summary

Keep open: current main still routes retryable heartbeat busy skips through the shared 1 second wake retry, so the reported active-conversation stall remains a source-reproducible regression rather than a stale or implemented item.

## Plan

Fix the heartbeat busy-skip retry regression reported in https://github.com/openclaw/openclaw/issues/40611. Current main treats requests-in-flight/lanes-busy heartbeat skips as retryable and the wake layer retries them after DEFAULT_RETRY_MS=1000ms, which can cause heartbeat work to grab the session path immediately after an active conversation clears. Add a heartbeat-owned, reason-specific debounce/backoff for retryable busy skips while preserving the existing fast 1s retry for non-busy handler errors. Prefer a typed/internal retry reason or retry delay flow over broad new config; only add public config/schema/docs if maintainers explicitly choose that product surface. Keep the fix in core heartbeat scheduling/wake code, do not change Telegram-specific behavior, do not implement the broader idle-timeout feature from https://github.com/openclaw/openclaw/issues/14051, and do not reintroduce full-interval drift from https://github.com/openclaw/openclaw/issues/29762. Add/update focused regression tests proving busy skips do not advance the full interval, do not retry at 1 second, and ordinary retryable errors still use the fast retry. Changelog is needed only if the repair is user-visible under repository policy.
