## Summary

Keep open: current main has related image-history and replay hygiene, but it does not yet add a final Responses `image_url` sanitizer, sanitize raw image-like text in the Codex app-server mirrored-history path, or quarantine poisoned Codex thread bindings.

## Plan

Fix https://github.com/openclaw/openclaw/issues/82878 as a focused user-visible bug fix. Add a shared sanitizer for OpenAI/Azure/OpenAI-Codex Responses-shaped payloads that runs immediately before every `responses.create` call and in pi-ai/onPayload wrapper paths when the payload has a Responses-style `input` array, regardless of model metadata. It should validate `input_image`/`image_url` fields and `function_call_output.output[*]`, accept only valid `data:image/<type>;base64,<payload>` and explicitly supported URL forms, drop or replace malformed image records with a safe placeholder, and never log raw payloads. Reuse or extract the existing gateway image URL/base64 validation ideas from `src/gateway/openai-http.ts` and `src/media/input-files.ts` where practical. Also sanitize Codex app-server mirrored history before thread start/resume projection: replace raw JPEG/PNG/GIF/WebP-looking base64 text and invalid image data URL text with a placeholder, strip invalid image records, log only counts/paths/hashes, and clear/quarantine the existing `.codex-app-server.json` binding via the existing binding helper only when definite image poison is detected. Do not add a config option, do not broaden plugin APIs, do not print raw base64/data URLs, and avoid changing unrelated history-image pruning policy from https://github.com/openclaw/openclaw/pull/76732. Add focused regression coverage for malformed `function_call_output.output[*].image_url`, raw JPEG-like base64 in tool-result text, metadata-missing Responses-shaped payloads, Codex mirrored-history sanitization, and binding clearing on detected poison. Add a single-line `CHANGELOG.md` Fixes entry because this is user-visible session recovery behavior.
