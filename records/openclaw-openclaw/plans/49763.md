## Summary

Keep open. Current main and the latest release still let Ctrl+O replace a busy TUI activity status with non-busy tool-toggle text, which drives the status renderer down the path that stops the loader.

## Plan

Fix the OpenClaw TUI loading spinner freeze/loss when Ctrl+O toggles tool output during an active run. Current main already blocks overlapping normal message submits via canSubmitTuiChatMessage/createEditorSubmitHandler, so do not reopen queue semantics or allow overlapping normal sends. Preserve the busy loader when activityStatus is streaming/running/waiting/finishing context and Ctrl+O changes tool expansion; if a tool-toggle notice is needed, keep it separate from the run-owned busy activity status or otherwise avoid driving renderStatus into the non-busy loader-stop branch. Add focused regression coverage that Ctrl+O/tool expansion during a busy run leaves the status busy/loader-owned, and that non-busy Ctrl+O still gives user feedback. Include a user-facing CHANGELOG.md fix entry if the repair branch owns changelog updates. Do not touch plugin APIs, provider routing, or backend run lifecycle.
