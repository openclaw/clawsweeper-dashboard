## Summary

Keep open: this is a real Google Chat threading bug and the PR has solid proof plus passing CI, but the latest patch still has two same-target guard gaps that can inject the inbound space's thread into a different outbound Google Chat target.

## Plan

On https://github.com/openclaw/openclaw/pull/74235, keep the existing Google Chat thread-resource normalization design but fix the remaining cross-target explicit-remap bug. In `extensions/googlechat/src/actions.ts`, only map an explicit Google Chat message-resource `replyTo` equal to `currentMessageId` onto `inboundThreadId` when the `to` target is the current Google Chat space; otherwise leave it unthreaded or reject it as unsupported. In `extensions/googlechat/src/channel.adapters.ts`, make `resolveAutoThreadId` require the same current target before returning the inbound thread for explicit current-message `replyToId` as well as omitted `replyToId`. Add focused regression coverage in `extensions/googlechat/src/actions.test.ts` and `extensions/googlechat/src/channel.test.ts` for explicit current-message replyTo with `to: spaces/BBB` while current context is `spaces/AAA`. Do not move Google Chat resource-name handling into core or add a config option.
