## Summary

Keep open. Current main still has the reported classification gap: the embedded-run timeout path can abort an active model stream and downstream failover/logging still normalizes timeout-shaped assistant failures to `timeout`/408 without carrying request-started or first-byte observations into the decision.

## Plan

Fix the openai-codex SSE post-start timeout misclassification reported in https://github.com/openclaw/openclaw/issues/66561. Add model-call lifecycle state at the embedded-runner boundary, at least request/model call started and response/first-byte observed, and preserve it through `EmbeddedRunAttemptResult`, assistant failover classification, failover observation/logging, and status/reason selection so a local abort after upstream bytes started is not logged or surfaced as an indistinguishable no-response `timeout`/408. Keep the scope narrow: do not change Codex base URL/transport defaults, do not broadly redesign model fallback, and coordinate with https://github.com/openclaw/openclaw/issues/60388 / https://github.com/openclaw/openclaw/pull/62682 so terminal run-budget fallback behavior is not duplicated. Add focused tests for a timed-out assistant attempt with model call started and first-byte/response bytes observed, plus the existing plain timeout case. Update changelog if the user-visible error/log classification changes.
