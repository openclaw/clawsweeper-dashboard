## Summary

Keep open. Current main still has the timeout-shaped abort path described by the report: model-call first-byte/progress data exists only as diagnostics, while the embedded attempt result and assistant failover boundary still classify bare timed-out aborts as timeout/408 without knowing whether the upstream stream had already started.

## Plan

Fix https://github.com/openclaw/openclaw/issues/66561. Current main records `timeToFirstByteMs` and `responseStreamBytes` only in model-call diagnostic events, while `EmbeddedRunAttemptResult` and assistant failover only receive `aborted`, `timedOut`, `idleTimedOut`, compaction timeout, and tool-timeout flags. Add a narrow lifecycle signal that can be carried from the model-call wrapper or run attempt into failover observation and user/log classification so a local abort after upstream stream progress is not reported as a plain no-response timeout/408. Preserve legitimate no-response idle timeouts, run-budget timeouts during compaction/tool execution, auth profile rotation, and existing fallback behavior unless lifecycle state proves the stream had already started. Do not implement the terminal-abort fallback-stop behavior from https://github.com/openclaw/openclaw/pull/62682 or the OpenAI SSE sanitizer fix from https://github.com/openclaw/openclaw/pull/76324; only coordinate with those surfaces. Add focused regression tests for diagnostic lifecycle propagation, failover observation normalization, assistant failover decision/status, and a post-start abort case. Add a changelog entry if the surfaced behavior changes for users/operators.
