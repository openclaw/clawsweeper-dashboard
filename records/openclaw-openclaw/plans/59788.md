# Coding Plan for openclaw/openclaw#59788: [Feature]: Start and persist routed agent sessions from outbound WhatsApp sends

Source report: [records/openclaw-openclaw/items/59788.md](records/openclaw-openclaw/items/59788.md)

## Summary

Keep open: current main still mirrors outbound sends only after an agent has already been selected, and the CLI/Gateway paths still choose explicit/session/default agent ids before consulting channel/account route bindings. The outbound-first non-default WhatsApp account workflow remains unimplemented.

## Plan

Implement route-aware outbound-first sessions for https://github.com/openclaw/openclaw/issues/59788. Add a shared outbound agent resolver used before outbound session-route derivation in CLI `openclaw message send`, Gateway send, and the shared action runner. Required behavior: explicit `agentId` and explicit/provided `sessionKey` keep precedence; otherwise infer the agent from existing channel/account/peer routing bindings using the same DM scoping semantics as inbound routing; then fall back to the default agent. Persist the outbound opener into the resolved agent session transcript via the existing mirror path. Keep the solution generic across channels, do not add WhatsApp-only core logic, and do not add a new flag/config option unless the existing routing contract cannot express the behavior. Add focused regression coverage for non-default account-routed outbound sends and keep existing default-agent/session-key behavior intact. Add a changelog entry because this is user-visible outbound/session behavior.
