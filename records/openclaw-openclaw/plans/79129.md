## Plan

Fix the CLI performance bug described in https://github.com/openclaw/openclaw/issues/79129: plain `openclaw status --json --timeout 1000` should not repeatedly or broadly discover bundled plugin manifests before returning a small gateway/status JSON payload. Current main routes through `src/commands/status-json.ts` -> `src/commands/status.scan.fast-json.ts`, where `hasConfiguredChannelsForReadOnlyScope()` can load installed channel manifest records through `src/plugins/channel-presence-policy.ts` and `src/plugins/plugin-registry-contributions.ts`. Keep the fix narrow: prefer passing/reusing one prepared metadata or manifest view inside the status scan, or use a cheaper configured-channel predicate for plain JSON status; preserve installed-index freshness and do not add a persistent global metadata cache. Add focused regression coverage showing plain status JSON does not invoke broad manifest/registry loaders unnecessarily while preserving configured-channel and env-only channel detection semantics. Check related reports https://github.com/openclaw/openclaw/issues/77348, https://github.com/openclaw/openclaw/issues/78100, and https://github.com/openclaw/openclaw/issues/78461 before implementation to avoid duplicating broader gateway/plugin-metadata work.
