# Coding Plan for openclaw/openclaw#81122: Bug: Twilio voice-call can get stuck in hold music after failed/no-stream call

Source report: [records/openclaw-openclaw/items/81122.md](records/openclaw-openclaw/items/81122.md)

## Summary

Keep open: current main still marks an inbound Twilio call active as soon as stream TwiML is returned, but there is no pending-stream timeout for the case where the media WebSocket never starts; the later inbound call can still be sent to the hold queue.

## Plan

Fix https://github.com/openclaw/openclaw/issues/81122 by adding provider-owned pending-stream cleanup for Twilio inbound stream TwiML. In `extensions/voice-call/src/providers/twilio.ts`, when `decision.activateStreamCallSid` adds a call SID to `activeStreamCalls`, start a short pending-stream timer that removes that SID only if no media stream was registered. Cancel the timer from `registerCallStream`, `unregisterCallStream`, terminal status callback cleanup, and `hangupCall`. Preserve current queue behavior for genuinely active streams, do not add a queue config option, and avoid changing non-Twilio providers. Add focused tests in `extensions/voice-call/src/providers/twilio.test.ts` for a no-start inbound call clearing so a later inbound call streams instead of receiving hold-queue, plus tests that a normally registered active stream still queues later inbound calls. Add a user-facing changelog fix entry if this is promoted as a maintainer repair PR.
