# Coding Plan for openclaw/openclaw#81122: Bug: Twilio voice-call can get stuck in hold music after failed/no-stream call

Source report: [records/openclaw-openclaw/items/81122.md](records/openclaw-openclaw/items/81122.md)

## Summary

Keep open: current main still marks an inbound Twilio call active as soon as stream TwiML is returned, but has no pending-stream timeout if the WebSocket start never arrives; that can leave stale state and route later inbound calls to the hold queue.

## Plan

Fix https://github.com/openclaw/openclaw/issues/81122. In the voice-call plugin's Twilio provider, add provider-owned pending stream startup tracking for inbound calls that receive stream TwiML via activateStreamCallSid. Start a short timeout when adding the SID to activeStreamCalls; cancel and remove that pending timer in registerCallStream, unregisterCallStream, terminal status callback cleanup, and hangup cleanup. If the timeout fires and callStreamMap has no registered stream for that SID, remove it from activeStreamCalls so the next inbound call can stream. Preserve the existing queue behavior while a real stream is active; do not add a queue config option or change core/plugin SDK boundaries. Add focused Vitest coverage for stream TwiML with no WebSocket start timing out and allowing the next inbound call to stream, plus keep the existing cleanup-path behavior covered. Add a single-line CHANGELOG Fixes entry if preparing the PR for landing.
