## Summary

Keep open: current main and the latest release still mark an inbound Twilio call active when stream TwiML is generated, but there is no pending-stream timeout for the case where Twilio never sends a media-stream start, so the stale state can still route a later caller to the hold queue.

## Plan

Fix https://github.com/openclaw/openclaw/issues/81122 by adding provider-owned pending-stream cleanup for inbound Twilio stream TwiML. In `extensions/voice-call/src/providers/twilio.ts`, when `activateStreamCallSid` is added because stream TwiML was returned, start a short pending-stream TTL; clear that timer/state when `registerCallStream` succeeds, and also clear it from existing unregister, terminal status, and hangup cleanup paths. Preserve the existing queue behavior for genuinely active streams, do not add a new config option unless maintainers explicitly ask, and keep the fix within the Twilio voice-call provider boundary. Add focused fake-timer regression coverage in `extensions/voice-call/src/providers/twilio.test.ts` proving a later inbound call streams after the pending no-start TTL, while an actually registered active stream still queues the next call. Add a user-facing `CHANGELOG.md` fix entry if this is promoted as a maintainer repair PR.
