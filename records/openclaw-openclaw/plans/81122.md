# Coding Plan for openclaw/openclaw#81122: Bug: Twilio voice-call can get stuck in hold music after failed/no-stream call

Source report: [records/openclaw-openclaw/items/81122.md](records/openclaw-openclaw/items/81122.md)

## Summary

Keep open: current main still has the stale no-start window. Twilio inbound stream TwiML marks a call SID active before any media stream registers, and there is no pending-stream timeout to clear that state if the WebSocket never starts.

## Plan

Fix https://github.com/openclaw/openclaw/issues/81122 by adding provider-local pending-stream cleanup for Twilio inbound stream TwiML. When generateTwimlResponse activates an inbound call SID, start a short pending-stream timer; cancel it when registerCallStream sees that call SID, and clear it from unregisterCallStream, terminal status callback cleanup, and hangupCall. If the timer fires before a stream registers, remove only that pending call SID from activeStreamCalls so later inbound calls can stream. Preserve the existing call-waiting queue behavior for genuinely active streams, do not add a new config option, and do not change unrelated Twilio auth, webhook verification, or TTS fallback behavior. Add focused Vitest coverage for the no-WebSocket-start path and cancellation on successful stream registration; add a user-facing Fixes changelog entry during landing if the release workflow requires it.
