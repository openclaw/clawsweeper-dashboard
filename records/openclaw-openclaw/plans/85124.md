# Coding Plan for openclaw/openclaw#85124: Bug: Anthropic API-key auth shows 'invalid bearer token' in status --usage, but the key works for inference

Source report: [records/openclaw-openclaw/items/85124.md](records/openclaw-openclaw/items/85124.md)

## Summary

Keep open: current main and the latest release still have a source-level path where an Anthropic API-key-only profile can be selected for provider usage and then sent to Anthropic's OAuth usage endpoint, producing the misleading bearer-token failure described here.

## Plan

Fix https://github.com/openclaw/openclaw/issues/85124. On current main, Anthropic API-key-only profiles can be returned by the generic provider-usage fallback and then sent to fetchClaudeUsage, which calls Anthropic's OAuth usage endpoint with Authorization: Bearer and surfaces a misleading invalid bearer token error. Keep the fix in the provider-usage/auth boundary, not CLI output formatting. Prefer a generic provider-owned usage-auth shape that lets a provider with a usage hook decline fallback, or an equally narrow owner-boundary fix; do not break API-key usage for providers whose usage APIs support API keys such as Z.ai, MiniMax, or Xiaomi. Add regression coverage for Anthropic API-key-only profiles resolving to no usage auth/no status usage error, Anthropic OAuth/token profiles still using fetchClaudeUsage, and status/gateway usage surfaces avoiding the misleading error. Treat https://github.com/openclaw/openclaw/issues/4614 as related OAuth/setup-token context only, not the same bug. Add a CHANGELOG.md Fixes entry for the user-visible status/auth fix if the repair lane owns changelog updates. Do not change Anthropic inference auth, setup-token behavior, or broad provider fallback semantics beyond what the tests require.
