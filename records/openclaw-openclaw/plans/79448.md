# Coding Plan for openclaw/openclaw#79448: [Bug]: Telegram DM topic-route fallback silently masks routing regression — `lastDeliveryStatus: delivered` for fallback-to-main delivery

Source report: [records/openclaw-openclaw/items/79448.md](records/openclaw-openclaw/items/79448.md)

## Summary

Keep open: current main still collapses a Telegram threadless retry success into ordinary cron delivery success, so operators cannot tell a correctly routed topic delivery from fallback-to-main delivery in persisted cron state.

## Plan

Fix https://github.com/openclaw/openclaw/issues/79448. Current main retries Telegram sends without `message_thread_id` when Telegram returns `message thread not found`, but the successful retry is reported to cron as ordinary `delivered`. Add a narrow durable signal for threadless fallback success, preferably by adding fallback metadata to the Telegram send/outbound result and propagating it into cron delivery state/run logs as a visible warning status such as `delivered-fallback` or an equivalent explicit fallback field. Update protocol validators, run-log parsing/filtering, CLI/UI display, docs, and changelog if a public field or enum changes. Add regression coverage where a cron Telegram `delivery.to` like `123:topic:42` resolves to `to=123` plus `threadId=42`, the first send rejects with `400: Bad Request: message thread not found`, the threadless retry succeeds, and the cron state/run entry is distinguishable from a clean topic delivery. Do not remove the fallback, do not add a config option, and do not broaden forum retry behavior here; https://github.com/openclaw/openclaw/issues/79408 tracks the forum no-retry case separately.
