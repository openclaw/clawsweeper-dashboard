# Coding Plan for openclaw/openclaw#65792: [Feature]: Support `message_sending` hook for webchat/WebSocket delivery path

Source report: [records/openclaw-openclaw/items/65792.md](records/openclaw-openclaw/items/65792.md)

## Summary

Keep open. Current main still sends WebChat final assistant output directly from the Gateway chat event path without invoking the existing `message_sending` hook, while the shared outbound delivery path already applies that hook; related open issues are broader or sibling hook-coverage trackers, not replacements for this narrow WebChat parity gap.

## Plan

Fix https://github.com/openclaw/openclaw/issues/65792 by wiring the existing plugin `message_sending` hook into WebChat final assistant delivery. Keep the change narrow: touch the Gateway chat final path and focused tests; do not attempt the broader outbound lifecycle observer, `message_sent`, internal hook, TUI, or HTTP API hook gaps. In `src/gateway/server-chat.ts`, make final WebChat delivery call the global hook runner only when `message_sending` hooks are registered, pass `channelId: "webchat"`, `conversationId`/`to` as the session key, plus `sessionKey` and run correlation, apply rewritten content to both `broadcast("chat", ...)` and `nodeSendToSession(...)`, and isolate hook failures so delivery still proceeds. If a hook returns `cancel: true`, suppress the delivered final assistant message and ensure the Control UI does not preserve the already-streamed stale text through its final-event fallback. Add regression coverage in `src/gateway/server-chat.agent-events.test.ts` for rewrite, cancel, and hook failure; update `ui/src/ui/controllers/chat.test.ts` only if UI cancellation semantics require it. No changelog unless maintainers decide this user-facing fix should ship immediately.
