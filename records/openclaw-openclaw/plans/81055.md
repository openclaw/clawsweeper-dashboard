# Coding Plan for openclaw/openclaw#81055: [Bug]: `openclaw status --json` emits `[secrets] …` diagnostic lines on stdout before the JSON document, breaking `jq` / `JSON.parse` consumers (sibling of closed `#72962`)

## Summary

Keep open. Current `main` still has a narrow route-first stdout gap: `runCli` tries routed commands before console capture is enabled, so the existing JSON-mode stderr flag does not protect `runtime.log` diagnostics emitted during `status --json` secret resolution.

## Plan

Fix https://github.com/openclaw/openclaw/issues/81055. Current route-first `status --json` can run before `enableConsoleCapture()` is installed, while `resolveCommandConfigWithSecrets` emits `[secrets]` diagnostics through `runtime.log`/`console.log`; this can put diagnostics on stdout before the JSON payload. Keep machine-readable stdout reserved for the JSON document when command secret resolution emits diagnostics for unresolved SecretRefs or gateway fallback. Prefer a generic fix for routed JSON command diagnostics or the shared command-secret diagnostic path; preserve existing non-JSON diagnostic behavior, keep `secretDiagnostics` in the JSON payload, and do not add config knobs or change SecretRef resolution policy. Add focused regression coverage for routed `status --json` with command-secret diagnostics proving stdout starts with parseable JSON and diagnostics are not written before it.
