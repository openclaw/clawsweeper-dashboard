# Coding Plan for openclaw/openclaw#68944: [Bug]: CLI commands hang at WebSocket gateway handshake

## Summary

Keep open: current main and the latest release still have the source-proven post-challenge failure shape where `sendConnect()` can throw before emitting the signed connect request, and that error is still swallowed as a debug-only parse error.

## Plan

Fix https://github.com/openclaw/openclaw/issues/68944 so gateway-backed CLI calls do not hang when `sendConnect()` fails after `connect.challenge` but before emitting the signed `connect` request. In `src/gateway/client.ts`, propagate connect assembly failures such as payload, public-key, or signing errors through `onConnectError`, close or settle the handshake appropriately, and narrow `handleMessage()` so dispatch errors are not swallowed as debug-only parse errors. In `src/gateway/call.ts`, wire `onConnectError` into `executeGatewayRequestWithScopes()` so one-shot CLI calls reject with an actionable error instead of waiting for the outer timeout. Add focused regression tests that simulate a synchronous throw before the connect request is sent and assert no silent hang, while preserving the existing successful Ed25519 nonce challenge flow. Keep the current signed nonce protocol; do not add HMAC fallback, broad auth semantics, or a new config option. Coordinate with nearby Gateway close/diagnostic work in https://github.com/openclaw/openclaw/pull/54475 and https://github.com/openclaw/openclaw/pull/65641, and add a concise changelog fix entry if landing policy requires it.
