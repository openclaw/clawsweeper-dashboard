# Coding Plan for openclaw/openclaw#68944: [Bug]: CLI commands hang at WebSocket gateway handshake

Source report: [records/openclaw-openclaw/items/68944.md](records/openclaw-openclaw/items/68944.md)

## Summary

Keep open: current main and the latest release still have the source-proven post-challenge failure shape where `sendConnect()` can throw before emitting the signed connect request, and that exception is still swallowed as a debug-only parse error.

## Plan

Fix https://github.com/openclaw/openclaw/issues/68944. Gateway-backed CLI commands can hang after `connect.challenge` when `GatewayClient.sendConnect()` throws synchronously while assembling the signed connect request before `request("connect", params)` is emitted. Keep the signed nonce/device identity protocol; do not add HMAC fallback or a new config option. In `src/gateway/client.ts`, separate JSON parse errors from event dispatch/connect assembly failures and make post-challenge `sendConnect()` failures call `onConnectError`, close/settle the connection, and avoid leaving the client in a silent wait state. In `src/gateway/call.ts`, wire one-shot CLI calls to observe connect assembly errors so `openclaw status`, `sessions`, `doctor`, and similar gateway-backed commands fail with a useful diagnostic instead of hanging. Add focused regression tests that inject a synchronous failure after `connect.challenge` and before connect request emission, plus call-path coverage showing `callGateway` rejects with the diagnostic. Coordinate with the related gateway close/diagnostic work in https://github.com/openclaw/openclaw/pull/54475 and https://github.com/openclaw/openclaw/pull/65641; preserve existing behavior for valid signed connects, startup retries, stored device tokens, and challenge timeouts. Add a user-facing changelog fix entry unless a maintainer says release notes are being handled separately.
