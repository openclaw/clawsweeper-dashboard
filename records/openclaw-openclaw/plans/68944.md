## Summary

Keep open: current main and the latest shipped tag still allow a post-challenge `sendConnect()` assembly failure to be swallowed by the broad message handler catch after the challenge timer is cleared, so gateway-backed CLI commands can wait for the outer timeout without a useful connect-auth error. The mistaken Google Chat cross-reference and older closed handshake-timeout/package-walk reports cover different intervals.

## Plan

Fix https://github.com/openclaw/openclaw/issues/68944 by making GatewayClient post-challenge connect assembly failures visible and terminal. Keep the existing Ed25519 `connect.challenge`/`device.nonce` protocol; do not add HMAC fallback, a new config option, or a protocol change. Wrap or restructure `sendConnect()` so synchronous failures while selecting auth, building the payload, signing, deriving the public key, or building connect params call `onConnectError`, close/reject promptly, and do not leave `connectSent`/challenge timers in a stuck state. Narrow `handleMessage()` so non-JSON-parse failures are not logged as debug-only parse errors. Add focused regression coverage that injects a synchronous post-challenge failure before `request("connect", params)` and proves the CLI/call path fails promptly with a useful error instead of waiting for the outer timeout. Add a CHANGELOG.md fix entry because this is user-visible CLI availability.
