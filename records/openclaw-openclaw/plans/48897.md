# Coding Plan for openclaw/openclaw#48897: [Bug] TUI input breaks on German keyboard layout: duplicated printable characters and broken AltGr (@, €)

Source report: [records/openclaw-openclaw/items/48897.md](records/openclaw-openclaw/items/48897.md)

## Summary

Keep open. Current main still delegates TUI editor printable input to pi-tui, and the pinned `@earendil-works/pi-tui@0.74.0` decoder still rejects printable Kitty CSI-u events when Alt/Ctrl modifier bits are present, which keeps the reported German AltGr `@`/`€` path unresolved.

## Plan

Fix the German-layout TUI input bug reported at https://github.com/openclaw/openclaw/issues/48897. Current main uses `@earendil-works/pi-tui@0.74.0`; `src/tui/tui.ts` delegates editor input to pi-tui and `src/tui/components/custom-editor.ts` falls through to `super.handleInput(data)`, while pi-tui `decodeKittyPrintable()` rejects printable CSI-u events that carry Alt/Ctrl modifier bits. Implement a narrow AltGr-aware printable insertion path or upgrade to an upstream pi-tui release if one fixes the contract. Preserve existing Alt/Ctrl shortcuts such as Alt+Enter, Alt+Up, Ctrl+C, Ctrl+D, and normal keybinding matching; do not broaden insertion for arbitrary modified shortcut events. Add focused regression coverage for German AltGr `AltGr+Q -> @`, `AltGr+E -> €`, and the duplicate printable examples where feasible. Add a `CHANGELOG.md` fix entry because this is user-visible. Validation should include targeted TUI tests plus manual or Testbox proof in a Kitty/CSI-u-capable terminal with a German layout showing `?`, `Ö`, `Ü`, `Ä`, `ß`, `AltGr+Q`, and `AltGr+E` inserted exactly once.
