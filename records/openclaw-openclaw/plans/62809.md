# Coding Plan for openclaw/openclaw#62809: Recurring isolated cron jobs have unstable first-turn OpenAI prompt-cache hits across runs

Source report: [records/openclaw-openclaw/items/62809.md](records/openclaw-openclaw/items/62809.md)

## Summary

Keep open. Current main still derives OpenAI-family prompt-cache affinity from the per-run session id, while isolated cron deliberately creates a fresh session id and run-scoped session key for each run, so the reported first-turn cross-run cache instability remains unresolved rather than covered by the adjacent prompt-prefix issues.

## Plan

Fix https://github.com/openclaw/openclaw/issues/62809 by adding stable OpenAI-family prompt-cache affinity for recurring isolated cron jobs. Derive an internal cache-affinity key from the stable cron job/session identity and effective provider/model, pass it through cron execution into embedded/OpenAI transport options, and use it for `prompt_cache_key` when cache retention is enabled. Do not reuse the isolated transcript `sessionId`, do not remove the `:run:<sessionId>` session key behavior, and do not add the optional public config/static-file/first-turn-template ideas in this PR. Add regression coverage showing two isolated runs of the same cron job keep different runtime session ids/session keys but send the same OpenAI prompt-cache key for the same model, with model/provider changes separated if the key includes them. Keep adjacent issues https://github.com/openclaw/openclaw/issues/64217 and https://github.com/openclaw/openclaw/issues/71973 out of scope except where the new key must coexist with their prompt-prefix work.
