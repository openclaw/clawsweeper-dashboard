# Coding Plan for openclaw/openclaw#62809: Recurring isolated cron jobs have unstable first-turn OpenAI prompt-cache hits across runs

Source report: [records/openclaw-openclaw/items/62809.md](records/openclaw-openclaw/items/62809.md)

## Summary

Keep open. Current main still derives OpenAI-family prompt-cache affinity from the isolated cron run session id, which is intentionally fresh per run, so the reported first-turn cross-run cache instability remains unresolved and is not covered by the adjacent prompt-prefix issues.

## Plan

Fix https://github.com/openclaw/openclaw/issues/62809 by adding stable OpenAI-family prompt-cache affinity for recurring isolated cron jobs without weakening isolated runtime sessions. Add an internal cache-affinity key derived from stable cron job identity plus effective model/provider context, preferably hashed/non-PII, and pass it through the embedded run/provider options so OpenAI Responses and opted-in OpenAI-compatible Completions use it for `prompt_cache_key`. Preserve the existing run-scoped `sessionId`, `runId`, `sessionKey`, transcript isolation, cleanup behavior, and ambient-context isolation. Do not add a user-facing config option unless source review proves it is necessary. Add focused tests proving isolated cron still creates fresh run sessions while OpenAI transport receives a stable cache key across runs for the same job/model and a different key when the model or job identity changes. Include a `CHANGELOG.md` fix entry if landing policy treats this user-visible cache/cost behavior as release-noteworthy.
