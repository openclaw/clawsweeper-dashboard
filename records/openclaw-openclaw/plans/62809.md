# Coding Plan for openclaw/openclaw#62809: Recurring isolated cron jobs have unstable first-turn OpenAI prompt-cache hits across runs

Source report: [records/openclaw-openclaw/items/62809.md](records/openclaw-openclaw/items/62809.md)

## Summary

Keep open. Current main still uses the isolated cron run session id as the OpenAI prompt-cache affinity key, while isolated cron deliberately creates a fresh session id per run; the newer related report was closed as a duplicate of this issue, and the adjacent open prompt-prefix issues do not cover this cross-run key-affinity gap.

## Plan

Fix https://github.com/openclaw/openclaw/issues/62809 by adding stable cross-run OpenAI prompt-cache affinity for recurring isolated cron jobs. Preserve isolated cron session ids, run session keys, transcript files, cleanup, and run-scoped state; do not reuse the runtime session id to get cache hits. Thread a separate internal cache-affinity key from cron preparation/execution into the embedded runner/provider transport boundary, and make OpenAI Responses and opted-in Completions use that key for prompt_cache_key when cacheRetention is enabled. The key should be stable across runs of the same cron job only when the effective provider/model and stable prompt context are compatible; avoid a public config option unless the existing contracts already have a natural field. Add focused regression coverage proving two isolated runs keep different session ids but send the same provider cache key, and keep adjacent prompt-prefix drift issues out of scope. Add a changelog entry if the resulting PR is user-facing.
