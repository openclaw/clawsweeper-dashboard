## Summary

Keep open. Current main and the latest shipped release still route isolated recurring cron OpenAI prompt-cache affinity through the fresh per-run session id, and the closed payload-key report explicitly points here as the canonical tracker for the remaining repair.

## Plan

Fix https://github.com/openclaw/openclaw/issues/62809. Current isolated cron runs intentionally get fresh per-run session ids, but OpenAI Responses and opted-in Completions use that run session id as prompt_cache_key, so recurring jobs cannot maintain cross-run cache affinity. Add a separate internal cron cache-affinity key, derived from job id plus effective provider/model and stable prompt/payload prefix context, and thread it through cron preparation, run execution, embedded-run stream options, and OpenAI request builders. Preserve isolated transcript/session cleanup, keep cacheRetention='none' suppressing prompt_cache_key, respect compat.supportsPromptCacheKey, and do not add a user-facing config option unless implementation proves it is required. Treat https://github.com/openclaw/openclaw/issues/64217 and https://github.com/openclaw/openclaw/issues/71973 as adjacent non-goals. Add focused regression coverage showing two isolated runs keep different session ids/session keys but produce the same internal cache-affinity key for the same job/model/prefix, and add or update a CHANGELOG.md Fixes entry if the repair lane owns user-facing changelog work.
