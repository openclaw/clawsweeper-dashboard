# Coding Plan for openclaw/openclaw#44294: Preserve structured ACP backend error kinds instead of mapping all errors to `end_turn`

Source report: [records/openclaw-openclaw/items/44294.md](records/openclaw-openclaw/items/44294.md)

## Summary

Keep open. Current main still drops non-refusal Gateway `errorKind` values at the ACP prompt boundary: `timeout`, `rate_limit`, `context_length`, and `unknown` resolve as plain `end_turn`, and `finishPrompt` returns no `PromptResponse._meta` metadata.

## Plan

Fix this issue by preserving structured Gateway chat error kinds through the ACP bridge without inventing unsupported ACP stop reasons. In `src/acp/translator.ts`, keep `errorKind: "refusal"` mapped to ACP `stopReason: "refusal"`, keep other error kinds ACP-compatible as `end_turn`, and include a documented `PromptResponse._meta` value that exposes the backend error kind for `timeout`, `rate_limit`, `context_length`, and `unknown`. Add or update focused tests in `src/acp/translator.error-kind.test.ts` and any affected stop-reason tests so non-refusal error kinds assert both `stopReason: "end_turn"` and the metadata. Update `docs/cli/acp.md` to document the metadata contract, and add a changelog entry if the change is user-visible. Do not change the ACP SDK, do not add custom values to the ACP `StopReason` enum, and do not refactor Gateway error classification beyond what is needed for the ACP projection.
