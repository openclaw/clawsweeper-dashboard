# Coding Plan for openclaw/openclaw#54242: [Bug]: backup archives can contain hardlink targets with '..' that fail broad extraction on macOS tar

Source report: [records/openclaw-openclaw/items/54242.md](records/openclaw-openclaw/items/54242.md)

## Summary

Keep open. Current main and v2026.5.7 still remap only tar entry paths and verify only entry paths, while `tar@7.5.15` stores hardlink targets separately in `linkpath`; no open or merged PR was found that fixes this specific gap.

## Plan

Fix https://github.com/openclaw/openclaw/issues/54242 as a narrow backup correctness bug. Current `openclaw backup create` remaps only tar `entry.path`, while hardlink targets are stored in `entry.linkpath`; `openclaw backup verify` also validates only entry paths. Update backup creation to avoid restore-unsafe hardlink targets, either by rewriting link targets relative to the archive root or by dereferencing unsafe hardlinks, and update backup verification to reject unsafe `ReadEntry.linkpath` values for hardlink entries. Add focused regression coverage with a hardlinked payload whose generated tar link target would otherwise contain `..`. Do not implement general configurable exclude rules, service-aware backup lifecycle, or a broad restore command in this PR. Add a user-facing `CHANGELOG.md` fix entry.
