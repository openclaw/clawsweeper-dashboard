# Coding Plan for openclaw/openclaw#54242: [Bug]: backup archives can contain hardlink targets with '..' that fail broad extraction on macOS tar

Source report: [records/openclaw-openclaw/items/54242.md](records/openclaw-openclaw/items/54242.md)

## Summary

Keep open. Current main and the latest release still rewrite only tar entry paths during backup creation and verify only tar entry paths, while the hardlink target lives in `linkpath`; the related exclude and backup-design items do not close this restore-safety gap.

## Plan

Fix https://github.com/openclaw/openclaw/issues/54242. Current backup creation remaps only `entry.path` in `src/infra/backup-create.ts`, while `tar@7.5.15` stores hardlink targets separately in `entry.linkpath`; current `backup verify` in `src/commands/backup-verify.ts` also validates only entry paths. Implement a bounded fix so backup archives either emit restore-safe hardlink targets or dereference/rewrite unsafe hardlink entries, and make verification reject unsafe tar link targets that contain traversal segments or would fail common broad extraction. Add focused regression coverage for a hardlinked payload whose generated tar `linkpath` would otherwise contain `..`, including create and verify behavior. Do not implement configurable exclude patterns, a broad restore command, or service-aware backup redesign in this PR; keep those in the related feature/design issues. Add a user-facing changelog entry for the backup restore-safety fix.
