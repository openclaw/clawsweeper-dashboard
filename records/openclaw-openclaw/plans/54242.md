# Coding Plan for openclaw/openclaw#54242: [Bug]: backup archives can contain hardlink targets with '..' that fail broad extraction on macOS tar

Source report: [records/openclaw-openclaw/items/54242.md](records/openclaw-openclaw/items/54242.md)

## Summary

Keep open. Current `main` and latest release still remap only tar entry paths and verify only tar entry paths, while `tar@7.5.15` stores hardlink targets separately in `linkpath`; the related symlink de-duplication and exclusion/design threads do not close this hardlink target safety gap.

## Plan

Fix the backup hardlink target safety bug reported in https://github.com/openclaw/openclaw/issues/54242. Current `createBackupArchive` remaps `entry.path` in `src/infra/backup-create.ts` but leaves tar hardlink `entry.linkpath` untouched, and `backupVerifyCommand` in `src/commands/backup-verify.ts` validates only entry paths. Implement a narrow restore-safe solution: either rewrite hardlink targets to valid archive-relative targets when the target is also in the archive, or dereference/emit file contents for unsafe hardlinks; in all cases, `backup verify` must reject unsafe tar link targets that contain traversal or resolve outside the archive root. Add focused regression coverage for a hardlinked fixture whose raw tar `linkpath` would contain `..`, including create-time archive output and verify rejection of a crafted unsafe archive. Do not add configurable excludes, restore commands, service lifecycle behavior, or broad backup redesign in this PR; those remain in the related feature/design issues. Add a user-facing `CHANGELOG.md` fix entry if the repair lane is preparing this for landing.
