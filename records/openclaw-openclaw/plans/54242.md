# Coding Plan for openclaw/openclaw#54242: [Bug]: backup archives can contain hardlink targets with '..' that fail broad extraction on macOS tar

Source report: [records/openclaw-openclaw/items/54242.md](records/openclaw-openclaw/items/54242.md)

## Summary

Keep open. Current main and latest release still rewrite and verify only tar entry paths, while `tar@7.5.15` stores hardlink targets separately in `linkpath`; the related exclusion, symlink, and backup-design threads do not close this hardlink target restore-safety gap.

## Plan

Fix https://github.com/openclaw/openclaw/issues/54242. Current `openclaw backup create` remaps only tar `entry.path`, and `openclaw backup verify` validates only entry paths, while `tar@7.5.15` stores hardlink targets separately in `entry.linkpath`; a verified archive can then fail broad extraction on macOS `bsdtar` when a hardlink target contains `..`. Keep the fix bounded to backup archive emission and verification: either emit restore-safe hardlink targets, dereference/reject unsafe hardlink entries, and make `backup verify` inspect link targets as well as entry paths. Add regression coverage for a hardlinked payload whose generated tar `linkpath` would otherwise contain traversal segments. Do not implement configurable excludes or a new restore command in this PR; those belong to the related design issues. Add a single-line `CHANGELOG.md` fix entry because this is user-visible backup behavior.
