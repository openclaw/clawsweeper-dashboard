## Summary

Keep open: current main still has a cached default memory manager that can be mutated into fallback-active embedding provider state, and the inspected hot-reload path does not clear or re-resolve that manager, so the reported no-recovery behavior remains distinct from the already-closed identity-guard issues.

## Plan

Fix https://github.com/openclaw/openclaw/issues/96534. The in-Gateway memory_search path uses the default cached MemoryIndexManager; after an embedding operation failure activates a configured fallback provider, the manager stores fallbackFrom/fallbackReason/provider/providerKey and does not re-resolve the primary provider after it recovers or on soft config reload. Add a narrow recovery/invalidation fix in memory-core or the Gateway reload path so the configured provider can be used again without a full process restart. Preserve the existing index identity guard: do not allow vector search with a provider/model identity that does not match the stored index metadata, and do not add a new config option. Include a regression test that simulates primary provider failure, fallback activation to a different identity, primary recovery or reload, and successful recovery without process restart; include coverage for not reintroducing cross-model vector results.
