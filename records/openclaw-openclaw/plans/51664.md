## Summary

Keep open: current main and the latest release still route WebChat code-block Copy through `navigator.clipboard.writeText` with an ignored rejection path, while the two Edge fallback PRs were closed unmerged and the newer open textContent PR addresses a different payload-selection bug.

## Plan

Fix https://github.com/openclaw/openclaw/issues/51664 by adding a shared Control UI clipboard helper for text copy. It should try `navigator.clipboard.writeText` first, then fall back to a hidden textarea plus `document.execCommand('copy')`, with cleanup in `finally` and robust selection such as `setSelectionRange`. Wire WebChat code-block Copy through that helper and only show the copied state after a successful copy. Also consider moving `ui/src/ui/chat/copy-as-markdown.ts` onto the same helper so Control UI copy paths do not drift. Before editing, check whether https://github.com/openclaw/openclaw/pull/73633 has merged; if it has, preserve its rendered `<pre><code>` text resolution and add fallback copying around it rather than replacing it. Add focused regression tests for Clipboard API rejection and fallback success/failure. Add a user-facing changelog fix entry. Do not change gateway, channel, provider, or markdown-renderer behavior beyond the clipboard-copy path.
