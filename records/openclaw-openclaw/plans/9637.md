## Summary

Keep open: current main still lacks a plain-text TUI accessibility mode, the linked implementation PR was closed unmerged, and source inspection shows config/schema/docs plus TUI render paths still emit emoji or Unicode glyphs without a switch.

## Plan

Implement the accessibility request from https://github.com/openclaw/openclaw/issues/9637 on current main. Add a default-off `ui.accessibility.plainText` or clearly equivalent config option to `OpenClawConfig`, zod validation, generated config labels/help/schema docs, and the public configuration reference. Load the option once at TUI startup and thread it through `ChatLog`, `ToolExecutionComponent`, selector/theme wrappers, waiting/status text, and tool summary formatting so enabled mode uses deterministic ASCII alternatives for tool emoji and decorative Unicode glyphs such as arrows, separators, bullets, and ellipses. Treat PR https://github.com/openclaw/openclaw/pull/9685 as prior art, but rework against current main and cover the broader current glyph set including local `SearchableSelectList`, `FilterableSelectList`, settings cursor/theme, waiting status, and the pi-tui SelectList wrapper seam. Do not change channel reaction emoji behavior, agent identity emoji behavior, provider/tool protocol semantics, or unrelated UI surfaces. Add focused tests and a user-facing changelog entry.
