## Summary

Keep open: current `origin/main` still lacks a DeepSeek V4 stream-wrapper fallback for unowned OpenAI-compatible proxy providers, the linked implementation PR was closed unmerged, and a later comment adds live Fireworks evidence that the problem still affects released installs.

## Plan

Fix https://github.com/openclaw/openclaw/issues/79608. On current main, a custom `models.providers` OpenAI-compatible proxy that exposes DeepSeek V4 has no owning provider runtime plugin, so `wrapProviderStreamFn` returns no DeepSeek V4 wrapper and replayed assistant messages can miss the required `reasoning_content`. Add a narrow fallback at the provider stream hook/runtime seam using the existing `createDeepSeekV4OpenAICompatibleThinkingWrapper`; apply it only for `openai-completions` DeepSeek V4 routes and avoid importing `extensions/deepseek` internals into core. Cover exact ids such as `deepseek-v4-flash` / `deepseek-v4-pro`, and evaluate the reported Fireworks path form `accounts/fireworks/models/deepseek-v4-pro` carefully; if that first-turn timeout proves to require response normalization rather than replay payload shaping, stop and report it as a separate follow-up. Add focused regression tests for custom unowned proxy selection, non-DeepSeek/non-OpenAI negative cases, and the shared payload patch. Update changelog if the fix is user-visible, and do not change unrelated provider behavior.
