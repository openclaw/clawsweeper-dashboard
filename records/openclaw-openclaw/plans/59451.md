# Coding Plan for openclaw/openclaw#59451: [Feature]: Add file-based fallback search when `node:sqlite` is unavailable

Source report: [records/openclaw-openclaw/items/59451.md](records/openclaw-openclaw/items/59451.md)

## Summary

Keep open: current main and v2026.5.18 still return an unavailable `memory_search` payload when the SQLite-backed manager cannot initialize, and the related open PR only covers the narrower error-message UX, not file-based recall preservation.

## Plan

Implement the fallback requested by https://github.com/openclaw/openclaw/issues/59451. When builtin memory search cannot initialize specifically because `node:sqlite` is unavailable, `memory_search` should fall back to a simple deterministic keyword scan over only the agent's allowed memory files (`MEMORY.md` and `memory/*.md`, plus any existing supported memory file constraints if reused from host helpers), return bounded snippets, and clearly mark the response as fallback keyword search rather than vector search. Do not change provider quota, embedding failure, QMD transient fallback, or existing in-database FTS/MATCH fallback behavior; coordinate with https://github.com/openclaw/openclaw/pull/69199 if it has landed. Add focused regression coverage for the missing-`node:sqlite` path and path/symlink safety, and do not add a new config option unless maintainers explicitly request one.
