## Summary

Keep open: current main and v2026.5.18 still return an unavailable `memory_search` payload when the SQLite-backed manager cannot initialize, and the linked messaging PR only covers error copy, not file-based recall preservation.

## Plan

Implement https://github.com/openclaw/openclaw/issues/59451 as a narrow memory-core fallback: when builtin memory manager creation fails specifically because `node:sqlite` is unavailable, have `memory_search` scan allowed agent memory files (`MEMORY.md` and `memory/*.md`, plus only the same configured extra memory paths already allowed by memory read helpers), return deterministic keyword-ranked snippets, and label the response/debug mode as fallback keyword search. Do not change provider quota handling, generic embedding failures, QMD transient fallback behavior, or the existing SQLite/FTS/vector search paths. Coordinate with the open messaging PR surface in `buildMemorySearchUnavailableResult` but do not treat the messaging-only PR as resolving this issue. Add focused regression tests for missing-node:sqlite fallback hits, no fallback for non-SQLite errors, path/symlink constraints, and result labeling; add a changelog entry because this is user-visible memory recall behavior.
