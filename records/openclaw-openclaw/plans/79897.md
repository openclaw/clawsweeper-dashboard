## Summary

Keep open: current main still has a source-plausible llama.cpp/OpenAI-compatible streaming usage loss when the final usage or timings chunk is sent after `[DONE]`. The linked upstream PR is closed unmerged and proposed removing the usage request rather than preserving the usage data that `/status` and compaction need; the newer live reproduction comment strengthens the report.

## Plan

Fix https://github.com/openclaw/openclaw/issues/79897. Current main already sends `stream_options: { include_usage: true }` and can normalize llama.cpp-style `usage`/`timings` when such data reaches OpenClaw, but OpenAI Node 6.37.0 stops yielding SSE data after `data: [DONE]`, so llama.cpp usage/timings emitted after `[DONE]` can be dropped and saved as zero. Implement a narrow fix in the OpenAI-completions streaming/guarded transport path so post-DONE usage or timings are preserved into assistant output and embedded-runner/session usage. Do not gate off `stream_options.include_usage`, do not require users to set `compat.supportsUsageInStreaming`, do not weaken `request.allowPrivateNetwork`/SSRF behavior, and avoid double-counting when providers already report usage before `[DONE]`. Add focused regression coverage with a mock SSE stream that sends a usage/timings chunk after `[DONE]`, plus embedded subscriber/run usage coverage as needed. Add a `CHANGELOG.md` fix entry because this is user-visible. Validation should include targeted OpenAI transport/provider-fetch/embedded usage tests and, if feasible, redacted real llama.cpp proof showing nonzero saved usage or `/status`.
