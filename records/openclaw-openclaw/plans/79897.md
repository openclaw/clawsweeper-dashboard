# Coding Plan for openclaw/openclaw#79897: OpenAI-compatible streaming with llama.cpp saves zero usage (stream closed before final usage chunk)

Source report: [records/openclaw-openclaw/items/79897.md](records/openclaw-openclaw/items/79897.md)

## Summary

Keep open: current main still has a source-reproducible path where llama.cpp usage emitted after `[DONE]` is lost. Later embedded routing fixes help requests reach OpenClaw's transport, but the remaining issue is that OpenClaw records usage only from chunks yielded by OpenAI SDK 6.37.0, whose SSE iterator ignores messages after `[DONE]`.

## Plan

Fix https://github.com/openclaw/openclaw/issues/79897 on current main. Preserve `stream_options: { include_usage: true }` for OpenAI-completions streaming, and do not solve this by gating the request on `supportsUsageInStreaming`. The bug is that llama.cpp can emit usage/timings after `data: [DONE]`, while OpenAI SDK 6.37.0 stops yielding after `[DONE]`, so the final usage never reaches `processOpenAICompletionsStream()` or embedded session usage. Implement a narrow transport/parser or ingestion-boundary fix that lets late llama.cpp usage/timings reach normalized assistant/session usage, while preserving standard OpenAI-style streams and avoiding double-counting when usage is already present before `[DONE]`. Add focused regression coverage for a stream with content/finish, `[DONE]`, then a usage-only or timings-only SSE block, plus coverage that normal before-DONE usage is not double-counted. Changelog entry expected because this is user-visible `/status` and compaction accounting behavior.
