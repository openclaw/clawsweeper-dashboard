# Coding Plan for openclaw/openclaw#56610: [Feature]: Discord delivery queue / retry on WebSocket reconnect (parity with WhatsApp)

Source report: [records/openclaw-openclaw/items/56610.md](records/openclaw-openclaw/items/56610.md)

## Summary

Keep open: current main still has the generic outbound delivery queue and WhatsApp/Telegram reconnect drains, but Discord has no READY/reconnect hook that drains queued outbound deliveries, so the reported Discord message-loss gap remains actionable.

## Plan

Fix https://github.com/openclaw/openclaw/issues/56610. Current main persists outbound delivery attempts before Discord sends and has drainPendingDeliveries for channel reconnect recovery, but Discord gateway READY/reconnect paths never drain channel="discord" queue entries, unlike WhatsApp and Telegram. Add a Discord-plugin-owned reconnect drain after gateway readiness/reconnect that calls openclaw/plugin-sdk/delivery-queue-runtime drainPendingDeliveries, scopes by normalized Discord account id and channel, and reuses existing deliverOutboundPayloadsInternal through the SDK runtime. Do not add channels.discord.delivery.queue or a new config option unless maintainers explicitly request it. Preserve existing Discord REST retry behavior; only use bypassBackoff for the verified reconnect/no-active-listener failure class. Add focused coverage for the Discord lifecycle/startup drain hook, account-scoped queue selection, backoff behavior, and no duplicate replay while a live delivery is claimed. Keep the scope outbound-only and do not mix in the inbound reconnect-backfill work from https://github.com/openclaw/openclaw/pull/81226. Add a changelog entry because this is user-visible message delivery behavior.
