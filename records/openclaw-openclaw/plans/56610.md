# Coding Plan for openclaw/openclaw#56610: [Feature]: Discord delivery queue / retry on WebSocket reconnect (parity with WhatsApp)

Source report: [records/openclaw-openclaw/items/56610.md](records/openclaw-openclaw/items/56610.md)

## Summary

Keep open: current main still has Discord durable outbound sends and the shared pending-delivery drain API, but Discord does not call the drain on Gateway READY/RESUMED or reconnect, so queued outbound messages can remain stranded after the reported reconnect window. The related open PR addresses inbound missed-message backfill and reply hydration, not this outbound delivery-queue drain.

## Plan

Fix Discord outbound delivery recovery during Gateway reconnect windows for https://github.com/openclaw/openclaw/issues/56610. Add a Discord-plugin-owned READY/RESUMED or reconnect drain that imports `drainPendingDeliveries` from `openclaw/plugin-sdk/delivery-queue-runtime`, selects queued entries with `entry.channel === "discord"` and matching normalized account id, and bypasses backoff only for reconnect/no-listener-shaped failures where safe. Do not add a new public config option unless required by existing config contracts. Preserve the existing active-delivery claim and unknown-platform-outcome protections so reconnect drains cannot duplicate live sends. Coordinate with the adjacent inbound reconnect-backfill PR at https://github.com/openclaw/openclaw/pull/81226 if it touches the same listener/startup files. Add focused regression coverage for Discord startup/lifecycle drain registration and shared delivery-queue reconnect behavior; add a changelog entry because this is user-visible message-delivery reliability.
