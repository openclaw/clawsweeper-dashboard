# Coding Plan for openclaw/openclaw#70330: WebChat can silently rotate agent:main:main after gateway restart, hiding prior session/checkpoints

Source report: [records/openclaw-openclaw/items/70330.md](records/openclaw-openclaw/items/70330.md)

## Summary

Keep open. Current main has fixed part of the reconnect path by carrying the history-backed `sessionId`, but the issue’s remaining safety gap is still present: ordinary `chat.send` still has a reset-command path without an explicit session-control flag or reset audit payload, and recent comments add a session-not-found reconnect variant that is not covered by the current UI behavior.

## Plan

Fix the WebChat session continuity/audit bug reported in https://github.com/openclaw/openclaw/issues/70330. Scope the PR to existing behavior: ordinary WebChat reconnect/page-refresh sends must keep using the `sessionId` returned by `chat.history`, a generic `chat.send` must not silently perform ambiguous session control without explicit provenance, and any reset/new path used by WebChat should expose enough typed metadata for old session id, new session id, trigger/source, and archived transcript status to be visible/auditable. Add regression coverage for an existing `agent:main:main` session A, reconnect/history/list, then plain `chat.send("continue")` preserving A and not producing a reset archive. Also cover the `/new` or `/reset tail` ambiguity so the reset marker/provenance is not lost. Keep reset policy defaults and config shape unchanged; do not build a full archive browser unless a very small existing sessions surface can expose the audit marker. Add a changelog entry because this is user-visible WebChat behavior.
