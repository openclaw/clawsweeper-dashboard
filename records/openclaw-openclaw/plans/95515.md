## Summary

Keep open: current `main` and v2026.6.9 still contain a source-reproducible path where the doctor fallback migration can write `groupAllowFrom` to an external `email` channel account with no bundled schema metadata. A closed-unmerged same-topic PR confirms the likely repair shape, but it is not a viable canonical fix.

## Plan

Fix the doctor/upgrade path that copies `allowFrom` into `groupAllowFrom` for external or unknown channel schemas. The reported failure is that upgrading from 2026.6.8 to 2026.6.9 writes `channels.email.accounts.default.groupAllowFrom`, which the email plugin schema rejects and can prevent Gateway startup. Keep the fix in the doctor/config migration boundary; do not add a runtime compatibility shim and do not broaden the email schema. Use https://github.com/openclaw/openclaw/pull/95518 as context only because it is closed unmerged. Likely make `maybeRepairGroupAllowFromFallback` require explicit schema/capability proof before writing root or account `groupAllowFrom`, then add regression coverage for an email-like external channel with `accounts.default.allowFrom` that must remain unchanged. Preserve existing supported migrations for bundled channels whose schemas accept `groupAllowFrom`. Do not edit `CHANGELOG.md`.
