Keep open. Current main still has the reported Telegram reply-media limit gap: the configured cap is computed and direct sends use it, but final reply and native-command reply delivery still call media loading without `maxBytes`; the two related fix PRs are closed unmerged.

## Plan

Fix https://github.com/openclaw/openclaw/issues/46023 with a narrow Telegram plugin change. Current main computes `mediaMaxBytes` in `extensions/telegram/src/bot-core.ts`, but final model replies and native-command replies do not pass that cap into `deliverReplies`, `deliverMediaReply`, or `buildOutboundMediaLoadOptions`, so `loadWebMedia` runs without `maxBytes`. Add focused regression tests proving configured `mediaMaxMb` reaches `loadWebMedia` for final media replies and native-command reply delivery; preserve current defaults when no cap is configured.
