## Summary

Keep open. Current main and the latest release still document and compute Telegram mediaMaxMb, but the final-reply and native-command reply delivery paths do not pass any media cap into outbound media loading; the two direct fix PRs are closed unmerged.

## Plan

Fix this issue by threading the already-computed Telegram mediaMaxBytes through the existing reply delivery paths. Carry the value from bot setup into createTelegramMessageProcessor/dispatch deliveryBaseOptions and registerTelegramNativeCommands/buildCommandDeliveryBaseOptions, add an optional mediaMaxBytes field to deliverReplies and deliverMediaReply, and pass it as maxBytes to buildOutboundMediaLoadOptions while preserving mediaLocalRoots, mediaLoader test injection, direct send.ts behavior, and current defaults. Add regression tests proving configured mediaMaxMb reaches loadWebMedia for standard final media replies and native-command media replies; include a one-line CHANGELOG.md Fixes entry if this user-visible fix is prepared as a PR.
