# Coding Plan for openclaw/openclaw#46023: [Bug]: Telegram final replies with media ignore configured mediaMaxMb in outbound delivery path

Source report: [records/openclaw-openclaw/items/46023.md](records/openclaw-openclaw/items/46023.md)

## Summary

Keep open. Current main still has the reported Telegram reply-media limit gap: the Telegram plugin computes the configured cap and the direct send path passes it to media loading, but final reply and native-command reply delivery still call `loadWebMedia` without a configured `maxBytes`; the two direct fix PRs are closed unmerged.

## Plan

Fix https://github.com/openclaw/openclaw/issues/46023 as a narrow Telegram plugin bug. Current main computes `mediaMaxBytes` in `extensions/telegram/src/bot-core.ts`, but final model replies and native-command replies do not carry it into `deliverReplies`, `deliverMediaReply`, or `loadWebMedia`. Carry the existing computed value through `createTelegramMessageProcessor`, `dispatchTelegramMessage`, `registerTelegramNativeCommands` / `buildCommandDeliveryBaseOptions`, `deliverReplies`, and `deliverMediaReply`, then call `buildOutboundMediaLoadOptions({ mediaLocalRoots, maxBytes: mediaMaxBytes })`. Preserve current defaults and behavior when the value is absent; do not add new config, change the default, or fold in unrelated proxy-transport work from https://github.com/openclaw/openclaw/pull/47387. Add focused regression tests proving `loadWebMedia` receives the configured cap for final media replies and native-command media replies, and that the undefined/default path remains unchanged. Add a one-line `CHANGELOG.md` fix entry for the user-visible Telegram behavior change.
