# Coding Plan for openclaw/openclaw#16387: Feature: Dynamic header templating for custom LLM providers (enables Fireworks.ai prompt caching)

Source report: [records/openclaw-openclaw/items/16387.md](records/openclaw-openclaw/items/16387.md)

## Summary

Keep open: current main and the latest release still pass configured provider headers through as static strings, and the related dynamic-header implementation PRs are closed unmerged. The current provider turn-state hook can add provider-owned headers, but it does not resolve custom-provider config placeholders for Fireworks, LiteLLM, or similar proxy use cases.

## Plan

Implement dynamic provider-header templating for custom model-provider config. Add a narrow allowlisted resolver for `models.providers.*.headers` and `models.providers.*.request.headers` that resolves `{{session.key}}`, `{{agent.id}}`, and `{{run.id}}` at model-request dispatch for normal turns and compaction. Preserve existing static header behavior, SecretRef handling, auth injection, request timeout/proxy/TLS handling, attribution headers, provider-plugin turn-state headers, and caller-header precedence. Do not add Fireworks-specific core policy or broaden this into arbitrary template evaluation. Use the closed PRs as background only; produce a clean bounded OpenClaw implementation with focused tests and docs. Add a changelog entry if the landing policy treats this user-facing config feature as release-notable.
