# Coding Plan for openclaw/openclaw#16387: Feature: Dynamic header templating for custom LLM providers (enables Fireworks.ai prompt caching)

Source report: [records/openclaw-openclaw/items/16387.md](records/openclaw-openclaw/items/16387.md)

## Summary

Keep open: current main and the latest release still treat configured provider headers as literal strings, and the provider turn-state hook only supports provider-owned native headers rather than the requested generic config placeholders. The useful prior implementation attempts are closed unmerged, and the open ZAI-specific stickiness PR is adjacent but does not cover Fireworks/LiteLLM dynamic header templating.

## Plan

Implement the feature requested in https://github.com/openclaw/openclaw/issues/16387 as a narrow OpenClaw provider-header resolver. Add an allowlisted dispatch-time resolver for configured `models.providers.*.headers` and `models.providers.*.request.headers` values supporting only `{{session.key}}`, `{{agent.id}}`, and `{{run.id}}`; preserve static values, SecretRef resolution, auth injection, attribution headers, retries, provider-owned stream hooks, and normal/compaction request behavior. Do not add arbitrary template evaluation, broad fallback stacks, Fireworks-only hardcoding, or a new plugin API unless source inspection proves the existing dispatch path cannot carry the needed context. Use prior closed PRs https://github.com/openclaw/openclaw/pull/16448 and https://github.com/openclaw/openclaw/pull/18213 only as background, not as code to apply blindly. Update docs for the new placeholder contract and add a changelog entry because this is user-facing config behavior. Validate with focused tests for provider headers, request.headers, normal turns, compaction, precedence, SecretRef behavior, and OpenAI transport headers; if credentials are available, include redacted Fireworks or LiteLLM proof showing the dynamic header is sent with a session-stable value.
