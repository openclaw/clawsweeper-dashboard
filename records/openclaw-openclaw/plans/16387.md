# Coding Plan for openclaw/openclaw#16387: Feature: Dynamic header templating for custom LLM providers (enables Fireworks.ai prompt caching)

Source report: [records/openclaw-openclaw/items/16387.md](records/openclaw-openclaw/items/16387.md)

## Summary

Keep open: current main still normalizes and merges configured provider headers as literal strings. The provider-owned turn-state hook and `prompt_cache_key` support are useful adjacent paths, but they do not provide the requested generic config placeholder contract for custom provider headers, and the related implementation PRs are closed unmerged.

## Plan

Implement the dynamic provider-header templating feature requested in https://github.com/openclaw/openclaw/issues/16387. Add a narrow, allowlisted resolver for configured `models.providers.*.headers` and `models.providers.*.request.headers` values that supports `{{session.key}}`, `{{agent.id}}`, and `{{run.id}}` at model-request dispatch time. Preserve existing static header behavior, model/provider/request header precedence, SecretRef resolution and stripping, auth override injection, attribution headers, retries, provider-owned stream hooks, and compaction behavior. Use the closed unmerged PRs https://github.com/openclaw/openclaw/pull/16448 and https://github.com/openclaw/openclaw/pull/18213 only as background; do not copy stale precedence or broad templating behavior. Update docs for the public provider config surface and add focused regression tests for literal static headers, each supported placeholder, unknown placeholders, request.headers, normal turns, compaction, SecretRef markers, and auth/attribution precedence. Do not change unrelated provider defaults or introduce a Fireworks-only workaround unless the generic config resolver proves impossible.
