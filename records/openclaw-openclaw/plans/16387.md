# Coding Plan for openclaw/openclaw#16387: Feature: Dynamic header templating for custom LLM providers (enables Fireworks.ai prompt caching)

Source report: [records/openclaw-openclaw/items/16387.md](records/openclaw-openclaw/items/16387.md)

## Summary

Keep open: current main still treats configured provider headers as literal strings, the Fireworks plugin only applies a Kimi payload wrapper, and the prior exact implementation PRs were closed unmerged. The discussion provides concrete Fireworks and Langfuse/LiteLLM use cases, and no open PR currently closes this runtime interpolation gap.

## Plan

Implement dynamic provider-header templating for this issue. Add an allowlisted resolver for configured model-provider headers at model request dispatch so models.providers.*.headers and models.providers.*.request.headers can use {{session.key}}, {{agent.id}}, and {{run.id}}. Preserve existing static header behavior, SecretRef resolution, auth overrides, caller header precedence, OpenClaw attribution headers, provider-owned stream hooks, retries, compaction, and normal agent turns. Do not add provider-specific Fireworks policy to core and do not broaden plugin APIs unless the current dispatch path cannot carry the required context. Use the closed dynamic-header PRs only as background, rebase the design on current provider-request-config plumbing, add focused regression tests and docs, and include a changelog entry for the user-visible config capability.
