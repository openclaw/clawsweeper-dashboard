# Coding Plan for openclaw/openclaw#79915: liveness warning fires during cold-start blocking (no startup grace window)

Source report: [records/openclaw-openclaw/items/79915.md](records/openclaw-openclaw/items/79915.md)

## Summary

Keep open: current main still starts diagnostic liveness at gateway startup without a startup-grace guard, so queued/open diagnostic work can still emit a WARN-level liveness warning during cold-start blocking. The related startup-grace PRs were closed unmerged, and the open threshold-tuning PR does not cover this missing grace window.

## Plan

Implement the narrow diagnostic startup-grace fix for https://github.com/openclaw/openclaw/issues/79915. Add an internal `startupGraceMs` option to `StartDiagnosticHeartbeatOptions`, pass `60_000` from `src/gateway/server.impl.ts`, always call `sampleLiveness`/`sampleDiagnosticLiveness` every heartbeat so event-loop-delay metrics and CPU/ELU baselines reset, and suppress only `diagnostic.liveness.warning` event plus Gateway WARN/debug emission while inside startup grace. Add focused regression coverage in `src/logging/diagnostic.test.ts` asserting the sampler is called during grace, no liveness event/WARN is emitted during grace with queued work, and warnings resume after grace expires. Preserve current idle/debug-only and active/queued liveness behavior. Do not add `OPENCLAW_STARTUP_GRACE_MS` or a config schema option unless a maintainer explicitly requests that separate product surface. Add a user-facing changelog fix entry with appropriate non-bot credit.
