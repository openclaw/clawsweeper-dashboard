# Coding Plan for openclaw/openclaw#1210: Images from Discord stored as base64 in session transcripts

Source report: [records/openclaw-openclaw/items/1210.md](records/openclaw-openclaw/items/1210.md)

## Summary

Keep open: current main still has mitigation layers, but not the requested durable transcript claim-check for inbound Discord images. Source inspection shows Discord attachments are saved as media paths, then hydrated into `type: "image"` base64 blocks for the Pi prompt, and the session persistence guard appends non-tool user messages without rewriting those image blocks to compact media references.

## Plan

Fix https://github.com/openclaw/openclaw/issues/1210 by adding a backward-compatible transcript claim-check for inbound image content. Current main saves Discord media to disk, renders `[media attached: ...]` prompt notes, hydrates those notes into `type: "image"` base64 blocks, and then persists the Pi user message without replacing the image data. Implement the fix in the session persistence/context boundary so user image blocks are persisted as compact media reference metadata instead of raw `image.data`, while bounded recent references can still hydrate for model context. Keep legacy inline-base64 transcripts readable; do not add a new config option, do not add startup/load-time core migrations, and do not solve this as display-only `chat.history` redaction. Coordinate with the existing Gateway `media://inbound/<id>` claim-check pattern and with open display PR https://github.com/openclaw/openclaw/pull/43982. Add focused regression coverage proving new persisted user messages do not contain base64 image data, prompt hydration still works from references, and old inline image transcripts remain readable/prunable. Add a user-facing changelog entry because this fixes a session-bricking storage bug. Before landing, provide live Discord media roundtrip proof or the closest maintained media roundtrip proof with private details redacted.
