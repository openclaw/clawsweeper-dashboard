## Summary

Keep open: current main still has replay/display/offload mitigations, but the central session persistence path can still write inbound user `image.data` blocks into JSONL when a channel turn is submitted with images. The exact persistence PRs are closed unmerged, and the open Gateway history PR is a display projection change rather than the durable transcript claim-check requested here.

## Plan

Fix https://github.com/openclaw/openclaw/issues/1210 by adding a backward-compatible transcript persistence claim-check for inbound/user image content blocks. Keep the fix in the session persistence or runner boundary, not in a Discord-only special case unless that is the only safe first step. Persist compact media reference/path/metadata instead of raw `type: "image", data: ...` for new user turns, hydrate bounded current/recent references when building model context, and keep legacy inline-base64 transcript entries readable. Do not add a new config option. Add focused regression coverage proving persisted JSONL no longer contains the original base64 for an inbound image turn and that replay/context still gets the usable image. Add a user-facing CHANGELOG.md fix entry.
