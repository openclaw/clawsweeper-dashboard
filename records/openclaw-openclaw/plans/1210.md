Source report: [records/openclaw-openclaw/items/1210.md](records/openclaw-openclaw/items/1210.md)

## Summary

Keep open: current main still mitigates old image replay and Gateway display bloat, but it does not implement the requested durable transcript claim-check for inbound image content. The Discord/media path can still hydrate saved media into `type: "image"` blocks for the Pi prompt, and the session guard persists user messages without rewriting those image blocks to references.

## Plan

Fix https://github.com/openclaw/openclaw/issues/1210 by adding a backward-compatible transcript claim-check for inbound user image content. Do not implement a redaction-only write-time deletion that breaks retry/recovery; persist durable media reference metadata for image blocks that already have a saved media path/reference, hydrate bounded recent references when constructing model context, and keep legacy inline-base64 transcripts readable. Coordinate with the existing `media://inbound` / `MediaPath` patterns from https://github.com/openclaw/openclaw/pull/55513 and the display-only chat.history work in https://github.com/openclaw/openclaw/pull/43982. Likely files include `src/agents/session-tool-result-guard.ts`, `src/agents/session-tool-result-guard-wrapper.ts`, `src/agents/pi-embedded-runner/run/images.ts`, `src/agents/pi-embedded-runner/run/history-image-prune.ts`, `src/auto-reply/media-note.ts`, Discord media tests under `extensions/discord/src/monitor`, and gateway chat media tests if display projection needs alignment. Add regression tests proving a Discord/media user turn persists compact refs instead of raw `image.data`, legacy inline-base64 transcripts still replay safely, current-turn hydration still reaches vision models, and older image history remains bounded. Add a user-facing changelog entry. Stop and ask for maintainer review if the fix needs a new public config knob, a breaking transcript schema, or media retention policy changes.
