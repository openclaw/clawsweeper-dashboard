## Summary

Keep open: current main still has mitigation layers, but not the durable session-transcript claim-check requested here. Discord media is saved to disk, then hydrated into base64 `image` blocks for the Pi prompt, and the guarded persistence path still appends user messages without rewriting those blocks to compact media references; the issue discussion also includes live Discord forensic proof from v2026.3.24.

## Plan

Fix https://github.com/openclaw/openclaw/issues/1210 by adding a backward-compatible session-transcript claim-check for inbound image content. New Discord/channel image turns should not persist raw `type: "image"` base64 data in session JSONL; persist compact media reference metadata, hydrate only bounded recent references during model-context construction, and keep legacy inline-base64 transcripts readable. Prefer the existing `media://inbound/<id>`/managed-media contract from https://github.com/openclaw/openclaw/pull/55513 where it fits. Coordinate with the chat-history projection path in https://github.com/openclaw/openclaw/pull/43982, but do not solve this only as UI/display redaction. Do not add a config option, do not perform startup/load-time migrations, and do not edit baseline or ignore files. Add focused regression coverage proving persisted user messages no longer contain the original image base64 while immediate model submission and retry/replay behavior still works. Add a concise `CHANGELOG.md` fix entry if the change is user-visible.
