# Coding Plan for openclaw/openclaw#1210: Images from Discord stored as base64 in session transcripts

Source report: [records/openclaw-openclaw/items/1210.md](records/openclaw-openclaw/items/1210.md)

## Summary

Keep open: current main still mitigates replay, display, and large Gateway attachment cases, but it does not persist inbound user image blocks as durable media references. The guarded session append path can still write `type: "image"` content with inline `data` into JSONL when images are passed to the agent prompt.

## Plan

Fix https://github.com/openclaw/openclaw/issues/1210 by adding a backward-compatible session transcript claim-check for inbound image content. Current main still lets user `content[].type === "image"` blocks with inline `data` reach `SessionManager.appendMessage`; do not solve this only through Gateway history display redaction or replay-time pruning. Implement the fix at the session persistence/replay boundary, using the existing media store and `media://inbound` / MediaPath patterns where practical. Persist compact media reference metadata instead of raw image `data` for new inbound user image blocks, hydrate only bounded current or recent references when building model context, and keep legacy inline-base64 transcripts readable. Coordinate behavior with the adjacent display-only work in https://github.com/openclaw/openclaw/pull/43982 and the merged Gateway claim-check pattern from https://github.com/openclaw/openclaw/pull/55513. Avoid adding a new config option or broad plugin/core policy change. Add focused regression coverage for transcript persistence size/shape, replay hydration, legacy inline transcript compatibility, Discord/media-note handling, and Gateway attachment interaction. Add a changelog entry if the final PR changes user-visible behavior.
