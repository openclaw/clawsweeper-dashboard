# Coding Plan for openclaw/openclaw#66512: [Feature] systemd sd_notify / Type=notify support for gateway readiness

Source report: [records/openclaw-openclaw/items/66512.md](records/openclaw-openclaw/items/66512.md)

## Summary

Keep open. Current `main` still does not implement the requested systemd readiness-notify contract: generated gateway units remain non-notify units, and the existing gateway ready boundary does not send `READY=1` through `NOTIFY_SOCKET`.

## Plan

Implement systemd readiness notification support for the gateway. Add a best-effort Linux/systemd notifier that sends `READY=1` only when `NOTIFY_SOCKET` is present, call it at the existing gateway post-attach ready boundary after sidecars/plugin services are ready and the `gateway ready` point is reached, and update `buildSystemdUnit` so generated gateway units use `Type=notify` plus `NotifyAccess=main`. Preserve current behavior outside systemd, keep notifier failures non-fatal, and do not bundle watchdog pings unless maintainers explicitly expand the scope. Check the prior closed sd-notify PR for useful lessons, but avoid reviving unrelated unit-name or watchdog churn. Coordinate with the open watchdog-only PR if it lands first. Add focused unit tests for the renderer and ready-boundary notifier behavior, update user-facing gateway/Linux/VPS docs if they describe systemd startup semantics, and add a changelog entry because generated service behavior changes.
