# Coding Plan for openclaw/openclaw#66512: [Feature] systemd sd_notify / Type=notify support for gateway readiness

Source report: [records/openclaw-openclaw/items/66512.md](records/openclaw-openclaw/items/66512.md)

## Summary

Keep open. Current main still lacks the requested systemd readiness-notify contract: generated Linux gateway units do not opt into notify mode, and the existing gateway-ready boundary does not send `READY=1`; the adjacent WatchdogSec work is related but does not supersede this readiness request.

## Plan

Implement systemd `sd_notify` readiness support for https://github.com/openclaw/openclaw/issues/66512. Add a small Linux/systemd notifier helper that is no-op when `NOTIFY_SOCKET` is unset, avoids a new dependency unless explicitly approved, and treats notification failures as nonfatal diagnostics. Call `READY=1` after the existing post-attach `gateway ready` boundary in `src/gateway/server-startup-post-attach.ts`, once the listener is already attached and startup-gated methods have been released. Update generated Linux gateway systemd units to include `Type=notify` and `NotifyAccess=main` only for gateway units that actually send readiness. Add regression coverage for unit rendering, helper no-op/notify behavior, and the ready hook. Coordinate with https://github.com/openclaw/openclaw/issues/11973 and closed https://github.com/openclaw/openclaw/pull/11974, but do not bundle `WatchdogSec`/`WATCHDOG=1` unless a maintainer explicitly expands the scope. Do not change launchd or Windows service behavior. Add docs and a changelog entry if the generated unit behavior changes.
