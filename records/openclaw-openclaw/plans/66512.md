# Coding Plan for openclaw/openclaw#66512: [Feature] systemd sd_notify / Type=notify support for gateway readiness

Source report: [records/openclaw-openclaw/items/66512.md](records/openclaw-openclaw/items/66512.md)

## Summary

Keep open. Current main still lacks the requested systemd readiness-notify contract: generated gateway units do not opt into notify mode, and the existing gateway-ready boundary does not send a `READY=1` notification.

## Plan

Implement systemd `sd_notify` readiness support for the gateway. Scope this to `READY=1` readiness: add a best-effort Linux notifier that is a no-op when `NOTIFY_SOCKET` is unset, handles systemd socket forms correctly, avoids shell execution, treats failures as nonfatal, and sends `READY=1` after the existing post-attach `gateway ready` boundary once HTTP listening is already established. Render matching generated gateway unit directives, likely `Type=notify` plus appropriate `NotifyAccess`, in `buildSystemdUnit`. Do not add watchdog pings unless maintainers explicitly expand scope; coordinate with the open watchdog PR if it lands first. Add focused regression tests for the unit renderer, notifier helper, and post-attach ready boundary; update Linux/gateway docs and changelog for the user-visible service behavior.
