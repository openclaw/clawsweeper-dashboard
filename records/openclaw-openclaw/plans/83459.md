## Summary

Keep open. Current main still has an offline `openclaw agents delete` path that calls a PATH `trash` binary, and the official runtime Dockerfile still does not install `trash-cli`, `gio`, or `trash-put`; the related Linux fallback reports and PRs were closed without a merged fix, and the only open adjacent work is the broader maintainer PR https://github.com/openclaw/openclaw/pull/68289.

## Plan

Fix the Docker/offline CLI deletion bug reported in https://github.com/openclaw/openclaw/issues/83459. Current main's `src/commands/onboard-helpers.ts` `moveToTrash` only runs `["trash", pathname]`, while the official Dockerfile does not install a trash provider; `openclaw agents delete` falls back to that helper when the Gateway is unavailable and can leave unique workspaces, agent dirs, or sessions on disk after config pruning. Prefer reusing the existing guarded `movePathToTrash` export from `src/plugin-sdk/browser-maintenance.ts` or an equivalent fs-safe wrapper rather than adding a Docker-only package dependency or an ad hoc `mv` fallback. Keep the shared-workspace guard in `src/commands/agents.commands.delete.ts` intact. Check the open adjacent maintainer PR https://github.com/openclaw/openclaw/pull/68289 before implementing to avoid duplicating its broad workspace-reset surface; a focused fix for this issue is acceptable if that PR remains broader or unsafe. Add focused unit coverage showing the CLI helper removes/moves a unique workspace without a PATH trash command and preserves the existing shared-workspace behavior. Add or update Docker/Testbox smoke coverage for a root-Dockerfile image unique-workspace `openclaw agents delete --force --json` flow with no `trash` binary available. Add a changelog entry because this is user-visible CLI/Docker behavior.
