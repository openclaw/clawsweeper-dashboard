# Coding Plan for openclaw/openclaw#78616: Discord multi-account startup should prioritize the configured primary/default account

Source report: [records/openclaw-openclaw/items/78616.md](records/openclaw-openclaw/items/78616.md)

## Summary

Keep open: current main still calculates Discord startup delay from the sorted Discord account list, so a configured default account such as `main` can be delayed behind earlier-sorted specialist accounts. The duplicate-token fix from https://github.com/openclaw/openclaw/issues/78118 is present, but it addresses a different same-token runtime hazard rather than distinct-account startup priority.

## Plan

Fix https://github.com/openclaw/openclaw/issues/78616 with a narrow Discord plugin change. Update `extensions/discord/src/channel.ts` so `resolveDiscordStartupDelayMs` orders startup candidates with the resolved/configured `channels.discord.defaultAccount` first when that account is enabled/configured and present, then keeps the remaining enabled/configured accounts in the existing stable `listDiscordAccountIds` order. Preserve `DISCORD_ACCOUNT_STARTUP_STAGGER_MS = 10_000`, do not add a new config option, do not hard-code operator-specific account ids, and do not relax duplicate-token suppression or Discord auth/allowlist behavior. Add a focused regression in `extensions/discord/src/channel.test.ts` where accounts such as `billy`, `farber`, and `main` are enabled with `defaultAccount: "main"`; assert `main` starts with no sleep and a specialist account receives the expected stagger. Add a user-facing changelog fix entry if repo policy requires it for this Discord startup behavior change.
