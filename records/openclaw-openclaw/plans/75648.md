# Coding Plan for openclaw/openclaw#75648: [Enhancement] Embedded-run upstream timeout is hard-coded at ~60s; provider 429/Retry-After is erased

Source report: [records/openclaw-openclaw/items/75648.md](records/openclaw-openclaw/items/75648.md)

## Summary

Keep open. Current main has the timeout configurability, progress-aware idle watchdog, and basic OpenAI-compatible status/error mapping, but the remaining Retry-After metadata still has no source field to travel through FailoverError and no endpoint path that sets a provider-derived Retry-After for `/v1/chat/completions` or `/v1/responses`.

## Plan

Fix the remaining Retry-After propagation for this issue. Current main already has `models.providers.<id>.timeoutSeconds`, progress-aware model idle timeout handling, SDK Retry-After caps, and OpenAI-compatible status/error mapping. Add an additive retry-timing field such as `retryAfterMs`/`retryAfterSeconds` to the failover metadata path, populate it from provider SDK/transport error shapes where a provider `retry-after-ms` or `retry-after` was received, and extend `resolveOpenAiCompatError` plus `/v1/chat/completions` and `/v1/responses` so non-streaming responses set `Retry-After` and streaming errors expose equivalent structured retry timing in the SSE error payload. Do not change timeout defaults, add provider-specific policy, or fold in durable prompt queueing from https://github.com/openclaw/openclaw/issues/49251. Add focused coverage for `FailoverError`/`describeFailoverError`, provider transport retry header extraction, non-streaming Chat Completions, streaming Chat Completions, non-streaming Responses, and streaming Responses. Add a changelog entry if the fix is user-facing.
