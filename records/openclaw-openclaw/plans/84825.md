# Coding Plan for openclaw/openclaw#84825: [Bug]: heartbeat.model override not honored by follow-up runs within a heartbeat session (2026.5.18)

Source report: [records/openclaw-openclaw/items/84825.md](records/openclaw-openclaw/items/84825.md)

## Summary

Keep open: current main honors `heartbeat.model` for the initial heartbeat turn and some immediate retry/fallback dispatch, but it still lacks a durable heartbeat-origin/model contract for separate follow-up entry points such as exec-approval agent followups and queued embedded followups. The related closed exec-timeout issue reduces the reported retry-storm amplifier, but it does not prove every heartbeat-originated follow-up stays on the heartbeat model.

## Plan

Fix the heartbeat follow-up model leak reported at https://github.com/openclaw/openclaw/issues/84825. Current main honors `agents.defaults.heartbeat.model` for the initial heartbeat turn, but follow-up entry points can lose heartbeat origin: `src/auto-reply/reply/followup-runner.ts` still sends queued embedded runs with `trigger: "user"`, and `src/agents/bash-tools.exec-approval-followup.ts` calls the gateway `agent` method without carrying heartbeat origin or the resolved heartbeat model, causing `src/gateway/server-methods/agent.ts` to resolve from the session/default model. Add a narrow contract so heartbeat-origin followups keep using the resolved heartbeat model for that heartbeat-originated continuation only. Do not make `heartbeat.model` sticky for later normal user turns, do not add a new config option, and do not mix in heartbeat timeout/no-route approval policy except as needed for regression setup. Add focused tests covering queued embedded heartbeat followups and exec-approval followup agent calls, plus existing opposite-direction heartbeat model bleed coverage. Add a single `CHANGELOG.md` Fixes entry if the repair changes user-visible behavior.
