# Coding Plan for openclaw/openclaw#64260: Include context window size and model max_tokens in llm_output event

Source report: [records/openclaw-openclaw/items/64260.md](records/openclaw-openclaw/items/64260.md)

## Summary

Keep open. Current main still emits `llm_output` with identifiers, assistant output, and usage counters only; the model context-window, effective context-budget, and max-output-token metadata exist elsewhere in the runtime but are not exposed to plugin hook consumers.

## Plan

Implement a narrow additive plugin-hook contract for https://github.com/openclaw/openclaw/issues/64260, preferably clustered with https://github.com/openclaw/openclaw/issues/64327 for naming consistency. Add optional fields to `PluginHookLlmOutputEvent` for native model context window, resolved effective context-token budget, and model max-output-token limit; use names that clearly distinguish native vs effective limits. Thread the values through the Pi embedded runner, CLI runner, and Codex app-server `llm_output` emitters without changing provider selection, prompt construction, model-call behavior, or existing hook fields. Prefer carrying already-resolved runtime facts forward (`ctxInfo`, `contextTokenBudget`, runtime model metadata) over plugin-side or request-time model lookup tables. Document the new hook payload fields in `docs/plugins/hooks.md`, add focused regression coverage, and add a changelog entry because this is a user-facing plugin hook surface. Do not attempt to solve per-call `llm_output`/`llm_call` semantics from https://github.com/openclaw/openclaw/issues/64268 unless the implementation explicitly touches that surface.
