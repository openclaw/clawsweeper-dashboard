## Summary

Keep open: current main still builds and delivers the subagent announce without a runtime-side parent-assistant transcript fingerprint check. The related open PRs only cover adjacent SILENT-token prompt parity and same-idempotency-key dispatch dedupe, not whether the parent already delivered the child result.

## Plan

Fix https://github.com/openclaw/openclaw/issues/78369 by adding a conservative runtime-side duplicate short-circuit in runSubagentAnnounceFlow after findings are known and the target requester session has been resolved, before buildAnnounceReplyInstruction/buildAnnounceSteerMessage/deliverSubagentAnnouncement. Read the requester parent session's latest assistant message from chat.history or an equivalent helper, compare only when the assistant message timestamp is at or after the child endedAt when timestamp data is available, and fail open if history/timing cannot be read safely. Implement a pure conservative fingerprint matcher using normalized lowercase collapsed whitespace and a 60+ character contiguous-window match in either direction; do not add cosine similarity unless it stays small and deterministic. Do not replace or block the adjacent SILENT-token prompt parity work in https://github.com/openclaw/openclaw/pull/78425 or the same-idempotency-key dispatch dedupe work in https://github.com/openclaw/openclaw/pull/81184. Add focused regression coverage for matching parent assistant text skipping delivery and returning delivered/cleanup success, unrelated parent text announcing normally, stale parent assistant text before endedAt announcing normally, and transcript-read failure announcing normally. Add a single CHANGELOG.md Fixes entry if the landing policy requires it, without bot/AI thanks lines.
