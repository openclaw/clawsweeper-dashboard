# Coding Plan for openclaw/openclaw#78369: [runtime] subagent-announce should fingerprint-match parent's last assistant message and short-circuit duplicates

Source report: [records/openclaw-openclaw/items/78369.md](records/openclaw-openclaw/items/78369.md)

## Summary

Keep open: current main still builds and delivers the subagent completion announce without a runtime-side check that the parent session already delivered matching content. The related open PRs cover adjacent SILENT-token prompt parity, same-idempotency-key dispatch dedupe, and tool-boundary hardening, but they do not implement this parent-transcript fingerprint short-circuit.

## Plan

Fix https://github.com/openclaw/openclaw/issues/78369 by adding conservative runtime-side duplicate suppression to subagent completion announce. In `runSubagentAnnounceFlow`, after the final `findings` and target requester session are known but before building/delivering the announce trigger, read the latest assistant message for `targetRequesterSessionKey` from the requester transcript. If `params.endedAt` is present, the latest assistant message timestamp is at or after that end time, and a conservative normalized fingerprint match shows that the assistant text already contains the child findings (or vice versa, using a contiguous 60+ character window or stricter equivalent), return `true` without delivering the announce. Fail open on missing transcript, missing timestamp, unreadable history, empty text, or no match. Do not add a config option, do not make core depend on plugin/channel specifics, and avoid broad fuzzy matching unless it remains clearly conservative. Add focused regression coverage for duplicate parent delivery, unrelated latest assistant text, stale assistant text before the child ended, and transcript-read failure. Reconcile nearby prompt/output/delivery edits if related PRs have landed, and add a single changelog fix entry if the change is user-visible.
