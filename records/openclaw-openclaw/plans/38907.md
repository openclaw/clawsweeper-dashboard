## Summary

Keep open: current main still routes metadata-free ACP bridge sessions into ACPX stale-session handling before normal Gateway generation, matching the reported `ACP_SESSION_INIT_FAILED` symptom.

## Plan

Fix the ACP bridge/runtime-boundary bug reported at https://github.com/openclaw/openclaw/issues/38907. On current main, `openclaw acp client` creates metadata-free bridge sessions, `AcpGatewayAgent.newSession` falls back to `acp:<uuid>`, and ACPX reply dispatch treats that ACP-shaped key as a stale ACP runtime session, yielding `ACP_SESSION_INIT_FAILED` before Gateway generation can run. Keep Gateway-backed ACP bridge sessions routed through normal `chat.send`/Gateway generation unless there is persisted ACP runtime metadata or an explicit ACPX binding. Preserve stale-session init failures for real ACPX sessions created by `/acp spawn` or configured/bound ACP sessions. Add focused regression tests around metadata-free bridge fallback and dispatch behavior; do not add a new config option or broaden ACP/native runtime semantics. Add a changelog fix entry because this is user-visible.
