## Summary

Keep this open: current main still has a source-visible collision between Gateway-backed ACP bridge sessions and ACP runtime sessions, so the first bridge prompt can be treated as a stale ACP runtime session instead of producing assistant chunks.

## Plan

Fix https://github.com/openclaw/openclaw/issues/38907. Current main's `openclaw acp client`/`openclaw acp` bridge creates bridge sessions with an `acp:<uuid>` fallback key in `src/acp/translator.ts`; Gateway canonicalizes that into `agent:<agent>:acp:<uuid>`, and `src/agents/agent-command.ts` then treats it as a stale ACP runtime session because `AcpSessionManager.resolveSession` requires persisted ACP metadata for ACP-shaped keys. Restore the documented Gateway-backed ACP bridge behavior so the first prompt reaches normal Gateway chat and streams `agent_message_chunk`, while keeping real ACP runtime sessions fail-closed when their metadata is missing.
