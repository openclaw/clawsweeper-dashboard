# Coding Plan for openclaw/openclaw#78820: Bug: update.run corrupts dist modules during in-place update

Source report: [records/openclaw-openclaw/items/78820.md](records/openclaw-openclaw/items/78820.md)

## Summary

Keep open: the fresh report matches a plausible current-main race where `update.run` awaits the package update and post-swap doctor work before scheduling the forced restart, leaving the old Gateway process able to lazy-load from a replaced `dist` tree.

## Plan

Fix https://github.com/openclaw/openclaw/issues/78820. The observable bug is that live control-plane `update.run` package-manager updates can replace the global OpenClaw package tree while the old Gateway process keeps running post-swap follow-up work, allowing old in-memory chunks to lazy-load missing/newly replaced `dist` files. Keep shell `openclaw update` behavior intact, keep failed install/verify/swap results from scheduling a restart, and do not add a config flag. Adjust the `update.run`/`runGatewayUpdate`/package-update boundary so live package swaps trigger the update restart or fresh-process handoff immediately after a successful swap and before old-process doctor/plugin follow-up work can import from the replaced tree; defer any required follow-up to the restarted process if needed. Add focused regression tests proving the live package-swap path does not run post-swap doctor before restart scheduling, while non-live shell/global update still performs its expected post-verify work. Add a user-facing `CHANGELOG.md` Fixes entry. Validate with targeted Vitest for the updater/restart files and targeted formatting.
