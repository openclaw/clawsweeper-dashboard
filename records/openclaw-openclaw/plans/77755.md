# Coding Plan for openclaw/openclaw#77755: WhatsApp group commands ignored when owner sender is reported as LID JID

Source report: [records/openclaw-openclaw/items/77755.md](records/openclaw-openclaw/items/77755.md)

## Summary

Keep open: current main and v2026.6.6 still omit `authDir` from the WhatsApp group command and owner-bypass identity call sites, so LID-only senders cannot use the existing account mapping before command authorization.

## Plan

Fix the WhatsApp plugin bug where group owner control commands such as `/new`, `/stop`, and `/status` are not authorized when the incoming sender is represented only as an `@lid` JID that maps back to the configured owner through the account auth directory. Keep the fix plugin-local: thread the existing `ResolvedWhatsAppAccount.authDir` or an equivalent resolved account authDir into `resolveWhatsAppCommandAuthorized()` and `applyGroupGating()` owner/sender identity resolution; do not add new config, change group-policy defaults, or broaden command authorization. Add focused regression tests where `allowFrom` contains the owner's E.164 identity, the group sender is a LID JID with no pre-filled `senderE164`, the authDir contains `lid-mapping-<lid>_reverse.json`, and owner commands are accepted while non-owner LID senders remain blocked. Changelog edits are not expected for this normal fix PR.
