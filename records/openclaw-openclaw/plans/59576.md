## Summary

Keep open. Current main and the latest release still feed the WebSocket post's `file_ids` directly into Mattermost media resolution, so an empty or missing array produces no inbound attachment; the direct fix PR was closed unmerged and the newer related PR is still open.

## Plan

Fix https://github.com/openclaw/openclaw/issues/59576 in the Mattermost plugin only. When an inbound Mattermost WebSocket `posted` payload has missing or empty `file_ids`, hydrate/re-fetch `/posts/{post_id}` through the existing authenticated Mattermost client with a short bounded retry/delay before calling the existing media resolver. Reuse the current SSRF-guarded media download path and do not add core channel APIs, new config, or broad debouncer refactors. First inspect https://github.com/openclaw/openclaw/pull/77619; if it is still open and suitable, extend or repair that approach instead of creating duplicate hydration logic, ensuring it actually covers delayed file IDs and not only thread files.
