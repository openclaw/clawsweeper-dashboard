## Summary

Keep open: current main still has a credible cron run-history chat routing defect. The UI links run history to the per-run session key, but the cron session store persists the transcript entry under the stable cron key, so chat history lookup for the per-run key can return no transcript or the wrong visible session.

## Plan

Fix https://github.com/openclaw/openclaw/issues/78944. Current main records cron run-history links with a per-run key like agent:<agent>:cron:<job>:run:<sessionId>, but createPersistCronSessionEntry only writes the session store under the stable agent:<agent>:cron:<job> key, and sessions.get cannot load the per-run chat. Implement the narrowest fix so the dashboard Open run chat link loads the actual historical run transcript. Preserve the run-scoped isolation added for context-engine execution; do not route future cron execution back through mutable base-session state. Prefer either a safe immutable per-run session-store snapshot or a session-history resolution path that maps cron run keys to their recorded sessionId/sessionFile. Add regression coverage for the UI link plus backend session/history resolution. Add a single-line CHANGELOG.md fix entry for the active version, crediting @kinlioncd if contributor attribution is used.
