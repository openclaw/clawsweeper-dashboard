Keep open. Current `main` still lacks a TUI display-name flag or option plumbing, and the Gateway TUI client still sends the hard-coded `openclaw-tui` display label even though the Gateway client contract already supports `clientDisplayName`.

## Plan

Add a custom display-name option for `openclaw tui` to resolve https://github.com/openclaw/openclaw/issues/41826. Implement a focused `--client-name <name>` or similarly clear TUI flag, thread it through `TuiOptions`, `runTui`, and `GatewayChatClient.connect`, and pass a sanitized/truncated label into the existing Gateway `clientDisplayName` option. Do not change the stable Gateway client id `GATEWAY_CLIENT_NAMES.TUI` / `openclaw-tui`, do not add a protocol redesign, and do not conflate this display label with usernames or auth identity. Add focused regression coverage for the CLI/options/connect path and default behavior, update the TUI docs, and add a user-facing `CHANGELOG.md` entry crediting @UULOVETEA if policy applies.
