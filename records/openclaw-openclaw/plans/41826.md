# Coding Plan for openclaw/openclaw#41826: Feature Request: Add custom client display name for TUI

Source report: [records/openclaw-openclaw/items/41826.md](records/openclaw-openclaw/items/41826.md)

## Summary

Keep open. Current main still lacks a TUI display-name option: the CLI exposes no client-name/display-name flag, `TuiOptions` cannot carry one, and the Gateway TUI path still sends the hard-coded `openclaw-tui` label even though the Gateway client already supports `clientDisplayName`.

## Plan

Implement custom TUI Gateway display labels for https://github.com/openclaw/openclaw/issues/41826. Add a focused `openclaw tui --client-name <name>` option, carry it through `TuiOptions`, TUI relaunch args, `runTui`, `GatewayChatClient.connect`, and `GatewayChatClient` into `GatewayClient.clientDisplayName`. Preserve `GATEWAY_CLIENT_NAMES.TUI` / `openclaw-tui` as the stable client id; do not rename protocol client ids or auth policy. Sanitize the user-provided label before sending it to the Gateway: trim whitespace, reject empty-after-trim values, strip ANSI/control sequences, and apply a reasonable length bound. Update CLI/web docs and add a user-facing changelog entry. Add focused tests proving the CLI accepts and propagates the flag, relaunch preserves it, and `GatewayChatClient` sends the custom display name while retaining the stable TUI client id.
