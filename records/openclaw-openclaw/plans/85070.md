## Summary

Keep open. This is a fresh, source-reproducible Telegram forum-topic message-loss bug: current main still treats a group/channel subagent completion as failed when the requester announce agent returns normal final text without message-tool delivery evidence, so the shipped user problem is not fully resolved by the earlier hardening work.

## Plan

Fix https://github.com/openclaw/openclaw/issues/85070. Preserve OpenClaw's mediated `message_tool_only` policy for group/channel subagent completions; do not raw-post private announce final text to Telegram. Make the subagent completion announce path deterministically deliver back to Telegram forum topics when the first requester-agent handoff returns normal final text without `message(action="send")`, for example by adding a specific forced message-tool retry/steer that carries the prior final text as private context and requires a message-tool send. Add focused regression coverage in `src/agents/subagent-announce-delivery.test.ts` and/or `src/agents/subagent-announce.format.e2e.test.ts` for a Telegram topic/group completion where the first handoff misses the message tool and the forced path succeeds, plus a failure case if the forced path still misses. Do not broaden this into the progress-only completion contract from https://github.com/openclaw/openclaw/issues/83863. Add a changelog entry because this is user-visible channel delivery behavior.
