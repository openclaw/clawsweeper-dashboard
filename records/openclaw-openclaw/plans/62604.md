# Coding Plan for openclaw/openclaw#62604: [Bug]: Control UI image paste flow can inject massive base64 text into the composer; send/draft handling is hard to patch safely in installed dist bundles

Source report: [records/openclaw-openclaw/items/62604.md](records/openclaw-openclaw/items/62604.md)

## Summary

Keep open. Current main and the latest release still let `handlePaste` fall through when the clipboard only contains `text/plain` or `text/html` data-image content, so the browser can insert the raw base64 string into the composer instead of the attachment path.

## Plan

Fix the Control UI composer paste bug reported in https://github.com/openclaw/openclaw/issues/62604. In `ui/src/ui/views/chat.ts`, harden `handlePaste` so clipboard payloads that provide `data:image/...;base64,...` via `text/plain` or `text/html` cannot be inserted into the textarea as raw draft text; convert supported pasted image data into the existing attachment payload path or reject it cleanly. Preserve existing file-backed image paste, file picker/drop behavior, side payload-store metadata-only state, and `sendChatMessageNow` draft/attachment restoration. Do not change provider routing, gateway attachment contracts, or installed-dist/source-map packaging policy. Add focused regression coverage in `ui/src/ui/views/chat.test.ts`; touch `ui/src/ui/controllers/chat.ts` or `ui/src/ui/app-chat.ts` only if the fix truly needs it. Add a user-facing changelog entry for the paste hardening unless the maintainer landing flow supplies it separately.
