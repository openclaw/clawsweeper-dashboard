# Coding Plan for openclaw/openclaw#62604: [Bug]: Control UI image paste flow can inject massive base64 text into the composer; send/draft handling is hard to patch safely in installed dist bundles

Source report: [records/openclaw-openclaw/items/62604.md](records/openclaw-openclaw/items/62604.md)

## Summary

Keep open. Current main still only intercepts clipboard items whose MIME starts with `image/`, so `text/plain` or `text/html` clipboard payloads containing `data:image/...;base64,...` fall through to the composer; the related wrong-image report is useful context but does not replace this distinct text-injection bug.

## Plan

Fix the Control UI composer paste bug reported at https://github.com/openclaw/openclaw/issues/62604. Harden `ui/src/ui/views/chat.ts` so clipboard text/html or text/plain payloads containing `data:image/...;base64,...` do not fall through into the textarea as raw base64; convert supported image data URLs into normal chat attachments or reject them cleanly without mutating the draft. Preserve existing file-backed image paste, file picker, drag/drop, attachment payload side-store behavior, and send/draft restoration in `ui/src/ui/app-chat.ts`. Add focused regression coverage in `ui/src/ui/views/chat.test.ts`; only touch controller/app-chat tests if the implementation changes serialization or restoration behavior. Keep the broader wrong-image byte-identity issue at https://github.com/openclaw/openclaw/issues/74960 as related context, not as the same bug. Add a user-facing changelog fix entry if runtime behavior changes. Do not add package hotpatch/source-map policy changes unless maintainers explicitly ask for that separate docs/support work.
