## Summary

Keep this open. Current main still has the core mismatch: main-session restart recovery only considers `running` sessions with `abortedLastRun: true`, while the config-reload timeout and restart drain-timeout paths can still force or proceed with a restart without marking affected main sessions; the secondary SIGUSR1 in-flight token issue appears addressed.

## Plan

Fix https://github.com/openclaw/openclaw/issues/82433 narrowly. Current main only recovers main sessions with `status: "running"` and `abortedLastRun === true`, while config reload deferral timeout and gateway restart drain timeout can proceed with active work without writing that marker. Add a bounded implementation that marks affected active main sessions as restart-aborted before forced restart/drain-timeout shutdown, reusing the existing `abortedLastRun` recovery contract. Do not implement a broad pending-inbound store, do not replace recovery with a new manifest/config surface, and do not rework the SIGUSR1 authorization token path unless a focused regression proves it still fails. Add regression coverage for config reload timeout with active work, drain timeout with active embedded/task work, recovery of marked sessions, and preservation of unmarked ordinary running-session behavior.
