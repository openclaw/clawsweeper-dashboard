# Coding Plan for openclaw/openclaw#76610: Matrix crypto-store: persistIdbToDisk writes snapshot non-atomically (corruption risk on unclean shutdown)

Source report: [records/openclaw-openclaw/items/76610.md](records/openclaw-openclaw/items/76610.md)

## Summary

Keep open. Current main and the latest release still write the Matrix crypto IndexedDB snapshot directly to the final JSON path, and the existing atomic JSON helper is available but not used on this persistence path.

## Plan

Fix https://github.com/openclaw/openclaw/issues/76610 by making Matrix crypto IndexedDB snapshot persistence atomic. In `extensions/matrix/src/matrix/sdk/idb-persistence.ts`, replace the direct `fs.writeFileSync(snapshotPath, JSON.stringify(snapshot))` plus `fs.chmodSync(snapshotPath, 0o600)` final-path write inside the existing `withFileLock()` callback with the existing `writeJsonFileAtomically()` helper from `openclaw/plugin-sdk/json-store`. Keep the existing lock boundary, snapshot format semantics, and zero-snapshot no-op behavior; do not fold in the separate shutdown/timer work from https://github.com/openclaw/openclaw/issues/76611 or https://github.com/openclaw/openclaw/pull/76709. Add focused regression coverage in `extensions/matrix/src/matrix/sdk/idb-persistence.test.ts` proving the written snapshot remains valid JSON with private permissions/trailing newline where platform-appropriate, and that a simulated atomic-write failure leaves the previous valid snapshot parseable. Add a concise `CHANGELOG.md` fix entry because this is user-visible Matrix E2EE persistence hardening.
