# Coding Plan for openclaw/openclaw#13870: Feature Request: Human-friendly device names in paired devices list

Source report: [records/openclaw-openclaw/items/13870.md](records/openclaw-openclaw/items/13870.md)

## Summary

Keep open: current main still lacks a durable operator-owned paired-device label and any rename path in storage, Gateway protocol, server handlers, CLI, or Control UI. The related metadata-display issue and open display-only PR cover useful identification details, but they do not supersede the user-settable naming workflow.

## Plan

Implement https://github.com/openclaw/openclaw/issues/13870 as a narrow paired-device naming feature. Add a durable operator-owned paired-device label/name separate from client-supplied `displayName`, with display precedence label > client displayName > deviceId. Add storage helpers and tests in `src/infra/device-pairing.ts`; add additive Gateway protocol/schema/validator support and a `device.pair.rename` server handler; register the method and scope it consistently with existing paired-device management so non-admin device-token callers can rename only their own device and admins can rename any paired device. Add `openclaw devices rename --device <id> --name <name>` support, preserving JSON behavior and sanitizing all terminal output with `sanitizeForLog`. Add a Control UI Devices/Nodes rename action and update docs plus `CHANGELOG.md`. Do not fold in broader client/OS/last-active metadata display from https://github.com/openclaw/openclaw/issues/23916 or the open display-only PRs.
