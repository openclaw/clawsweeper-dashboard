Keep open with high confidence. Current main and latest release still only expose client-supplied `displayName` or the raw device id for paired devices, while the requested durable operator-owned label, rename RPC, CLI command, and Control UI action are still absent; related work at https://github.com/openclaw/openclaw/issues/23916 and https://github.com/openclaw/openclaw/pull/57449 covers broader display metadata, not the rename workflow.

## Plan

Implement https://github.com/openclaw/openclaw/issues/13870 as a focused additive paired-device naming workflow. Add a durable operator-owned label/name field separate from client-supplied `displayName` and reconnect metadata, add an authorized Gateway method/schema/handler such as `device.pair.rename`, add `openclaw devices rename --device <id> --name <name>`, update the Control UI Devices section with a rename action, document label precedence in `docs/cli/devices.md` and `docs/gateway/protocol.md`, and add a user-facing changelog entry if this lands as a feature. Preserve `--json`, sanitize all displayed device-controlled or operator-controlled text, mirror existing self/admin paired-device authorization boundaries, and do not fold in the broader metadata/detail display work tracked at https://github.com/openclaw/openclaw/issues/23916 or the open CLI display PR https://github.com/openclaw/openclaw/pull/57449.
