## Summary

Keep open: current main still lacks a durable operator-owned paired-device label and has no paired-device rename workflow in storage, Gateway protocol, CLI, Control UI, or docs; related work covers broader metadata display or closed unmerged attempts, so this remains the canonical label feature.

## Plan

Implement https://github.com/openclaw/openclaw/issues/13870 as one focused paired-device label workflow. Add a durable operator-owned label field separate from client `displayName`, expose an additive Gateway protocol/schema/method such as `device.pair.rename` with self/admin authorization matching existing paired-device management rules, add `openclaw devices rename --device <id> --name <label>` and Control UI rename support, and document display precedence as operator label, then client displayName, then deviceId. Do not fold in the broader metadata-display work tracked by https://github.com/openclaw/openclaw/issues/23916, do not reuse `displayName` as the operator label, preserve token redaction and `--json`, sanitize all terminal-rendered device-controlled fields, and avoid CHANGELOG.md edits.
