## Summary

Keep open: current main still lacks a durable operator-owned paired-device label and there is no paired-device rename workflow in storage, Gateway protocol, CLI, Control UI, or docs. Related items cover broader metadata display or closed unmerged attempts, so this remains the canonical rename/label feature.

## Plan

Implement the paired-device naming workflow for https://github.com/openclaw/openclaw/issues/13870. Add a durable operator-owned label field separate from client `displayName`, define display precedence as operator label, then client displayName, then deviceId, and expose an authenticated rename operation through Gateway protocol/schema/handlers, `openclaw devices rename --device <id> --name <name>`, and the Control UI devices/nodes panel. Preserve existing JSON output compatibility except for additive fields, sanitize all terminal-rendered device-controlled or user-entered text, and mirror current paired-device self/admin authorization rules. Do not fold in the broader metadata-display/detail work from https://github.com/openclaw/openclaw/issues/23916, do not add a config option, and do not edit CHANGELOG.md. Update docs and focused regression tests for storage persistence, reconnect metadata not overwriting the operator label, Gateway authz, CLI behavior, and Control UI rendering/action wiring.
