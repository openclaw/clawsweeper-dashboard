# Coding Plan for openclaw/openclaw#13870: Feature Request: Human-friendly device names in paired devices list

Source report: [records/openclaw-openclaw/items/13870.md](records/openclaw-openclaw/items/13870.md)

## Summary

Keep open: current main still lacks a durable operator-owned paired-device label and rename workflow, while the related metadata issue covers broader device details and the only related implementation PR closed unmerged.

## Plan

Implement the paired-device naming feature requested by https://github.com/openclaw/openclaw/issues/13870. Add a durable operator-owned label to paired-device records that is separate from client-supplied `displayName`; reconnect metadata updates must never overwrite the operator label. Add an additive Gateway protocol method for setting or clearing the label, wire it through current schema exports, method descriptors, server dispatch, and paired-device management authorization using the existing self/admin boundaries. Add `openclaw devices rename --device <id> --name <name>` with an explicit way to clear the label, and add a Control UI action for the same operation. Display names with the precedence operator label, client `displayName`, then `deviceId`, while continuing to show the device ID where needed for destructive commands. Update the paired-device CLI and Gateway protocol docs. Add focused storage, protocol-handler/authz, CLI, and Control UI tests. Keep `node.rename` unchanged and do not fold in the broader metadata-display scope from https://github.com/openclaw/openclaw/issues/23916. Do not edit `CHANGELOG.md`; put user-visible release-note context in the eventual PR body or commit message. Stop and request maintainer input if the implementation would require changing the permanent authorization model or migrating paired-device state to a different storage owner.
