# Coding Plan for openclaw/openclaw#13870: Feature Request: Human-friendly device names in paired devices list

Source report: [records/openclaw-openclaw/items/13870.md](records/openclaw-openclaw/items/13870.md)

## Summary

Keep open: current main still lacks the requested durable, operator-owned paired-device naming workflow. The code only stores and displays client-supplied `displayName` metadata or the device id, and there is no paired-device label field, Gateway rename method, CLI rename command, or Control UI rename action; the related open metadata-display issue and PRs do not replace this rename workflow.

## Plan

Implement durable human-friendly paired-device labels for this issue. Add an operator-owned label field separate from client-supplied `displayName` in paired-device storage, expose an additive Gateway method/schema/handler such as `device.pair.rename` with the same self/admin authorization shape as existing paired-device management, add `openclaw devices rename --device <id> --name <label>` and Control UI rename support, and document display precedence. Do not reuse client reconnect `displayName` as the durable label, do not fold in the broader `openclaw devices list` metadata/detail work from https://github.com/openclaw/openclaw/issues/23916, and preserve JSON compatibility, token redaction, `approvedScopes` redaction, and terminal/UI sanitization of device-controlled fields. Add a CHANGELOG.md entry because this is user-facing.
