# Coding Plan for openclaw/openclaw#13870: Feature Request: Human-friendly device names in paired devices list

Source report: [records/openclaw-openclaw/items/13870.md](records/openclaw-openclaw/items/13870.md)

## Summary

Current main still lacks the requested durable, operator-owned paired-device naming workflow. Existing code stores and displays client-supplied `displayName` metadata, but reconnects can refresh that field and there is still no paired-device label field, Gateway rename method, CLI rename command, or Control UI rename action; related display-only work does not replace this request.

## Plan

Implement the paired-device naming workflow for https://github.com/openclaw/openclaw/issues/13870. Add a durable operator-owned paired-device label separate from client-supplied `displayName`, with clear display precedence of label > client displayName > deviceId. Add additive Gateway protocol schema/validators, a server method such as `device.pair.rename` with the same self-vs-admin authorization boundary used for remove/rotate/revoke, method registration/scopes, storage helper and regression tests. Add `openclaw devices rename --device <id> --name <name>` and update `openclaw devices list` display without changing `--json` compatibility or leaking token secrets/approvedScopes. Add a Control UI action/callback/button for paired devices and docs for the CLI and Gateway protocol. Keep broader metadata columns/detail mode tracked by https://github.com/openclaw/openclaw/issues/23916 and do not fold unrelated node.rename behavior into this PR. Add a user-facing changelog entry.
