# Coding Plan for openclaw/openclaw#13870: Feature Request: Human-friendly device names in paired devices list

Source report: [records/openclaw-openclaw/items/13870.md](records/openclaw-openclaw/items/13870.md)

## Summary

Keep open: current main still lacks a durable operator-owned paired-device label and any rename path in storage, Gateway protocol, server handlers, CLI, or Control UI. The related metadata-display issue and open display-only PRs do not supersede this user-settable naming workflow.

## Plan

Implement https://github.com/openclaw/openclaw/issues/13870 as a narrow paired-device rename workflow. Add a durable operator-owned paired-device label field separate from client-supplied displayName in src/infra/device-pairing.ts, with tests showing reconnect metadata cannot overwrite the operator label. Add an additive Gateway protocol/schema/method such as device.pair.rename, register it in the method list and method scopes, and preserve existing paired-device self/admin authorization boundaries. Expose the workflow in the CLI as openclaw devices rename --device <id> --name <label> with clear/set behavior and JSON output preserved, and add a Control UI action for paired devices. Display precedence should prefer operator label, then client displayName, then deviceId. Update docs/cli/devices.md, docs/gateway/protocol.md, and CHANGELOG.md for the user-facing behavior. Do not fold in broader metadata-display work from https://github.com/openclaw/openclaw/issues/23916 or the open display-only PRs except as coordination context; keep token secrets and approvedScopes redacted and sanitize device-controlled display fields.
