# Coding Plan for openclaw/openclaw#13870: Feature Request: Human-friendly device names in paired devices list

Source report: [records/openclaw-openclaw/items/13870.md](records/openclaw-openclaw/items/13870.md)

## Summary

Keep open: this issue is still necessary because current main and v2026.6.8 still lack a durable operator-owned paired-device label and any rename flow across storage, Gateway protocol, CLI, Control UI, or docs. The existing client `displayName` is not a safe substitute because reconnect metadata can refresh it.

## Plan

Add an additive operator-owned paired-device label workflow for https://github.com/openclaw/openclaw/issues/13870. Keep it separate from broader device metadata display in https://github.com/openclaw/openclaw/issues/23916. Implement one durable label field on paired devices, do not reuse client-controlled displayName, and make display precedence explicit as label, then client displayName, then deviceId. Add a Gateway protocol schema and handler, register the method with the existing device.pair methods, mirror existing self/admin device-management authorization boundaries, add `openclaw devices rename --device <id> --name <label>` or an equivalent narrow CLI command, add Control UI rename affordance, and update docs. Preserve existing `--json` behavior except for additive fields, sanitize terminal output, do not expose token secrets or approvedScopes, and do not edit CHANGELOG.md.
