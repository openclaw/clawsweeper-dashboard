## Summary

Keep open: current main still lacks the durable, operator-owned paired-device label and rename workflow requested here. Existing surfaces show client-supplied `displayName` or the raw `deviceId`, and the related metadata-display issue/PR do not implement an editable paired-device label.

## Plan

Implement the durable paired-device naming workflow for https://github.com/openclaw/openclaw/issues/13870. Add a stored operator-owned label field separate from reconnect/client `displayName`; expose an additive Gateway method such as `device.pair.rename` with schema/types/descriptors/server handler and the same self/admin authorization boundaries as paired-device management; add `openclaw devices rename --device <id> --name <label>`; add a Control UI rename action; document label/display precedence in CLI and protocol docs; add focused tests. Do not fold in the broader metadata-column/display work from https://github.com/openclaw/openclaw/issues/23916 or https://github.com/openclaw/openclaw/pull/57449, and do not change `node.rename` semantics. Preserve `--json`, sanitize every terminal-rendered label/metadata value, and add a user-facing changelog entry if the repair lane owns release notes.
