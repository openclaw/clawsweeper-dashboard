## Summary

Keep open: current main still has a source-visible iOS/shared chat event gap for live group-chat updates. The gateway already exposes subscribed `session.message` transcript updates, but the iOS transport does not subscribe to or decode that event and the shared view model currently ignores assistant `session.message` payloads.

## Plan

Fix https://github.com/openclaw/openclaw/issues/80231. On current main, iOS group-chat assistant replies can require leaving and reopening because `IOSGatewayChatTransport` does not subscribe to/decode `session.message`, while `OpenClawChatViewModel.handleSessionMessageEvent` only appends `user` role messages. Implement the narrow existing-contract fix: have the iOS transport subscribe the active session with `sessions.messages.subscribe`, decode `session.message` into `.sessionMessage`, and update shared chat handling so active-session assistant transcript messages become visible without a manual reload. Add focused regression coverage for iOS transport event mapping and shared ChatViewModel assistant `session.message` handling. Preserve the existing gateway protocol and do not add a new polling/config mode. Be careful to dedupe with existing `chat` final refresh behavior. Add a changelog entry because this is a user-visible iOS chat fix.
