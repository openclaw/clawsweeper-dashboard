## Summary

This looks like a valid current-main bug, not a cleanup-close candidate: OpenClaw has structured silent-delivery paths, but a fallback heartbeat/completion final reply ending in `notify=false` is still treated as ordinary visible text and sent without a silent flag.

## Plan

Fix https://github.com/openclaw/openclaw/issues/80569. Current main has structured silent delivery (`heartbeat_respond.notify=false`, message tool `silent`, Telegram `disable_notification`) but fallback heartbeat/completion final text can leak a trailing standalone `notify=false` into Telegram and send loudly. Keep structured metadata canonical and do not introduce a broad user-facing `notify=false` convention. Add a narrow parser/normalizer for OpenClaw-generated heartbeat/completion/final-reply fallback text that strips only a trailing standalone control marker, preserves ordinary prose containing `notify=false`, and either suppresses no-attention heartbeat output or passes `silent: true` through the existing outbound delivery path. Cover Telegram/group topic behavior by unit tests around the implicated normalizer/delivery path and existing silent transport tests. Do not change Telegram auth, topic routing, or global plugin policy.
