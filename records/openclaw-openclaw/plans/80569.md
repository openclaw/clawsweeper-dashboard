# Coding Plan for openclaw/openclaw#80569: Telegram replies ignore silent notification intent and leak `notify=false` into message body

Source report: [records/openclaw-openclaw/items/80569.md](records/openclaw-openclaw/items/80569.md)

## Summary

Keep open: current main has structured Telegram silent-delivery plumbing, but a plain trailing `notify=false` heartbeat/completion marker is still not stripped or converted before Telegram delivery, so the reported user-visible leak remains plausible from source inspection.

## Plan

Fix https://github.com/openclaw/openclaw/issues/80569 by preventing a standalone trailing `notify=false` control marker from leaking into Telegram heartbeat/completion replies. Preserve structured `heartbeat_respond` / message-tool metadata as the canonical contract. Add a narrow defensive path in the heartbeat/completion delivery boundary that strips only a standalone trailing `notify=false`; when remaining text is empty, suppress delivery, and when text remains, deliver it with silent notification metadata so Telegram receives `disable_notification=true`. Do not add a broad global parser that removes arbitrary `notify=false` mentions from normal user-visible text. Add focused regression coverage for marker stripping, non-control text preservation, and Telegram silent send behavior. Add a user-facing CHANGELOG.md fix entry if the repair changes shipped behavior.
