## Summary

Keep open. Current `origin/main` still has the reported source-level compaction-retry delivery race: the runner only flushes before the retry wait, while lifecycle retry resolution can unblock that wait before retry-generated block/channel flushing finishes.

## Plan

Fix https://github.com/openclaw/openclaw/issues/47335 on current main. The runner currently flushes block replies before `waitForCompactionRetryWithAggregateTimeout()`, but retry-generated replies can still be in the async block/channel flush when lifecycle resolves the compaction wait. Add a retry-specific delivery barrier, likely in `runEmbeddedAttempt` after the compaction retry wait resolves or an equivalent narrow seam, so retry-generated block replies drain before the attempt proceeds or returns. Preserve https://github.com/openclaw/openclaw/pull/59308: do not make ordinary slow end-of-run channel flushes block compaction completion beyond the retry delivery guarantee. Do not implement the broader final-payload dedupe changes from https://github.com/openclaw/openclaw/pull/62157 or the broader resume semantics from https://github.com/openclaw/openclaw/issues/59618. Add focused regression coverage for async `onBlockReplyFlush` during compaction retry and include a `CHANGELOG.md` fix entry if the runtime behavior change is user-visible.
