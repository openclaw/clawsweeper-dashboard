## Summary

Keep open. Current main still has the source-level compaction-retry delivery race: the runner drains before the retry wait, lifecycle code can resolve that wait before the async block/channel flush settles, and current tests still lock in the risky ordering.

## Plan

Fix https://github.com/openclaw/openclaw/issues/47335. Current main flushes `onBlockReplyFlush()` before `waitForCompactionRetryWithAggregateTimeout()`, but when a compaction retry generates a new reply, `handleAgentEnd()` can resolve the compaction wait before the retry's async block/channel flush settles, letting the attempt finish with streamed final text suppressed. Add a narrow retry-specific delivery barrier, likely a post-wait `onBlockReplyFlush()` or equivalent in `src/agents/pi-embedded-runner/run/attempt.ts`, and add focused regression coverage. Preserve the intentional behavior from https://github.com/openclaw/openclaw/pull/59308: ordinary slow end-of-run channel flushes should not block compaction wait resolution. Do not fold in the broader final-payload dedupe work from https://github.com/openclaw/openclaw/pull/62157 or the broader compaction-resume semantics from https://github.com/openclaw/openclaw/issues/59618. If this is user-visible, add a single-line active-version `CHANGELOG.md` fix entry without thanking bot handles.
