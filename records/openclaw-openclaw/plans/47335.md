# Coding Plan for openclaw/openclaw#47335: Reply generated but not delivered when compaction triggers session rollover

Source report: [records/openclaw-openclaw/items/47335.md](records/openclaw-openclaw/items/47335.md)

## Summary

Current main still has the source-level compaction-retry delivery race: the runner drains block replies before the compaction retry wait, but not after it, while lifecycle code resolves the retry wait before the async block/channel flush finishes. The closed unmerged fix PR and the related open block-streaming PR do not supersede this narrow ordering bug, so this should stay open as a focused repair candidate.

## Plan

Fix https://github.com/openclaw/openclaw/issues/47335 by adding a retry-specific delivery barrier so retry-generated block replies drain before `runEmbeddedAttempt` completes after `waitForCompactionRetryWithAggregateTimeout()`. Keep the fix scoped to the embedded runner/lifecycle ordering; do not broadly move all lifecycle/channel flushes before compaction wait resolution, because https://github.com/openclaw/openclaw/pull/59308 intentionally prevents slow final drains from blocking compaction completion. Add focused regression coverage that makes `onBlockReplyFlush` asynchronous, proves compaction retry wait can resolve before lifecycle flush completion, and proves the runner still awaits a post-wait drain before returning. Preserve existing media/error/fallback/direct-send safeguards in `src/auto-reply/reply/agent-runner-payloads.ts`; coordinate with but do not subsume https://github.com/openclaw/openclaw/pull/62157. Include a maintainer-owned `CHANGELOG.md` Fixes entry if the repair PR is created as an OpenClaw-maintained fix.
