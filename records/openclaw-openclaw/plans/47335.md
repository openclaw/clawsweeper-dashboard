## Summary

Keep open: current main still resolves the compaction retry wait before the retry-generated block-reply flush can finish, and the runner only flushes before waiting, so the reported lost-reply path remains source-reproducible.

## Plan

Fix https://github.com/openclaw/openclaw/issues/47335. Current main awaits `params.onBlockReplyFlush()` before `waitForCompactionRetryWithAggregateTimeout()`, but retry-generated block replies can still be draining after that wait resolves because `handleAgentEnd()` resolves compaction retry before awaiting `onBlockReplyFlush()`. Add a retry-specific post-wait delivery barrier in `src/agents/pi-embedded-runner/run/attempt.ts` or an equivalent narrow guarantee so the attempt cannot finish while retry-generated block replies are still draining. Preserve the slow-channel ordering from https://github.com/openclaw/openclaw/pull/59308; do not implement the broader auto-compaction task-resume behavior from https://github.com/openclaw/openclaw/issues/59618 or the general final-payload dedupe change from https://github.com/openclaw/openclaw/pull/62157 unless the narrow fix truly requires it. Add focused regression coverage for the pending-flush-after-retry-wait race, and add a changelog entry because this is user-visible reply delivery reliability.
