## Summary

Keep open. Current main still routes exec approval follow-ups by `sessionKey` only, while `/new` and `sessions.reset` replace the session instance under that key, so the reported stale follow-up path remains source-reproducible.

## Plan

Fix https://github.com/openclaw/openclaw/issues/59349. Current main builds `ExecApprovalFollowupTarget` with `sessionKey` only and `sendExecApprovalFollowup` resumes the gateway `agent` by that key, while `/new` and `sessions.reset` replace the `sessionId` under the same key. Thread the active runtime session id into exec tool defaults/follow-up target as an internal expected session id, then before agent-resume delivery or direct fallback delivery compare the current mapping for the session key to that expected id; if it differs, log and drop the stale follow-up. Do not change exec approval policy semantics, task runtime ownership, session reset behavior, or add a user-facing config option. Add focused regression coverage for target construction, stale follow-up drop after rebind, and preserved delivery when the session id still matches. Add a `CHANGELOG.md` Fixes entry because this is user-visible behavior.
