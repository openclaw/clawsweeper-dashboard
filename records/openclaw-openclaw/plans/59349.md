# Coding Plan for openclaw/openclaw#59349: [Bug] Exec approval follow-up can leak into a new session after /new because it rebinds by sessionKey instead of original sessionId

Source report: [records/openclaw-openclaw/items/59349.md](records/openclaw-openclaw/items/59349.md)

## Summary

Keep open. Current main still has the reported session-instance leak shape: exec approval follow-ups are created and dispatched by `sessionKey` only, while `/new` and `/reset` rotate `sessionId` under that same key, and no related open PR owns the original-session guard.

## Plan

Fix https://github.com/openclaw/openclaw/issues/59349 by binding exec approval follow-up targets to the original session instance. Capture the expected/current session id when gateway and node exec approval follow-up targets are created, carry it through the shared target and sender types, and before any `sendExecApprovalFollowup` agent resume or direct fallback, compare it with the latest `sessionKey -> sessionId` mapping; if it differs, log a concise stale-followup message and drop/return false. Preserve existing behavior when no expected session id is available, do not add a config option, and do not fold in unrelated retry/timeout/parser work from adjacent exec follow-up PRs. Add focused regression coverage for a simulated `/reset` or session-key rebind so stale follow-ups do not reach either the gateway `agent` call or direct outbound fallback. Add a single-line `CHANGELOG.md` Fixes entry if the branch is prepared for maintainer landing.
