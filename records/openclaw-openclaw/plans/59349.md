# Coding Plan for openclaw/openclaw#59349: [Bug] Exec approval follow-up can leak into a new session after /new because it rebinds by sessionKey instead of original sessionId

Source report: [records/openclaw-openclaw/items/59349.md](records/openclaw-openclaw/items/59349.md)

## Summary

Keep open. Current main still builds and dispatches exec approval follow-ups with only `sessionKey`, while `/new` and `sessions.reset` rotate the session instance under that key; the recent final-delivery refresh guard is adjacent but does not guard this follow-up dispatch path.

## Plan

Fix https://github.com/openclaw/openclaw/issues/59349. Capture the active runtime session id when creating exec approval follow-up targets in the gateway and node exec hosts, propagate it through `ExecApprovalFollowupTarget` and `sendExecApprovalFollowup`, and before resuming by `sessionKey` verify that the current session store mapping still matches the expected session id. If it differs because `/new` or `/reset` rebound the key, log a sanitized stale-follow-up drop and do not send the agent follow-up or direct fallback. Add focused regression coverage for both gateway and node target construction plus the dispatch/drop behavior; do not broaden this into the separate bundle-MCP disposal race or fallback-copy issue. Add a changelog entry for the user-visible bug fix if the PR lands.
