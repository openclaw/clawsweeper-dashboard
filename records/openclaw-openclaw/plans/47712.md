## Summary

Keep open: current main still has no public `messages.tts.filterEmoji` contract and no shared emoji filtering before TTS synthesis; the only emoji stripping I found is provider-local and does not cover the requested core TTS behavior.

## Plan

Implement https://github.com/openclaw/openclaw/issues/47712 as a narrow TTS config feature. Add `messages.tts.filterEmoji?: boolean` to the TTS config schema, public type, config help/labels, and TTS docs, with the documented default matching the issue request unless maintainer guidance says otherwise. Add a shared TTS text-preparation helper that strips markdown and, when enabled, removes emoji only from text passed to speech synthesis; do not mutate visible chat text, stored transcript text, reactions, or unrelated emoji behavior. Apply the helper consistently in auto-TTS (`maybeApplyTtsToPayload`), the agent `tts` tool, and gateway `tts.convert`; reconcile Local CLI and Discord voice provider-local sanitizers so behavior is not duplicated or contradictory. Add focused regression tests and a user-facing CHANGELOG entry. Stop and ask for maintainer guidance if the implementation would require broader non-TTS emoji policy changes.
