## Summary

Keep open. Current main has improved Docker PID-1 handling, but the browser plugin still supervises only the direct Chromium `ChildProcess` and has no browser-owned process-tree or descendant reaping path for the reported normal-exit Chromium/crashpad leak.

## Plan

Fix https://github.com/openclaw/openclaw/issues/64593. Current main still launches and stops only the direct managed Chromium ChildProcess in `extensions/browser/src/browser/chrome.ts`; browser-heavy PID-1/container deployments report normal-exit `chromium` and `chrome_crashpad` zombies accumulating under OpenClaw until process spawning fails with EAGAIN. Implement a narrow browser plugin lifecycle fix that supervises and reaps OpenClaw-owned Chromium across successful launches, explicit stop, restart after CDP mismatch, bootstrap/profile-decoration launch, startup retry cleanup, and crashpad/adopted descendants. Do not deep-import core internals from the browser plugin; if a shared process-tree helper is needed, expose it through an allowed plugin SDK process-runtime seam or keep the helper local to the plugin. Preserve Node and Bun paths. Be careful with Unix process groups: use positive-pid and ownership guards, avoid signaling PID 1 or the gateway process group, and keep Windows fallback behavior. Add focused regression coverage for invalid pid safety, Unix descendant/process-group cleanup, bootstrap cleanup, and direct-process fallback. Include a Linux/container proof path that repeatedly starts/stops managed Chromium and checks `ps` for no defunct `chromium` or `chrome_crashpad` descendants under the OpenClaw process. Add a changelog entry because this is user-visible reliability work.
