## Summary

Keep open. Current main and the latest release still supervise only the direct Chromium `ChildProcess`, while the report describes normal-exit `chromium` and `chrome_crashpad` zombies accumulating under the live OpenClaw gateway process in PID-1 containers.

## Plan

Fix https://github.com/openclaw/openclaw/issues/64593 as a narrow browser-plugin reliability bug. The observable problem is long-running PID-1 gateway containers accumulating defunct `chromium` and `chrome_crashpad` children after normal browser-tool use until process spawning fails with EAGAIN. Work in the current bundled browser plugin path, primarily `extensions/browser/src/browser/chrome.ts` and adjacent lifecycle tests. Preserve remote/attachOnly behavior and do not add broad startup migrations or unsafe process killing. A safe fix should supervise/reap OpenClaw-owned Chromium across normal exits, explicit stops, restarts, startup retry cleanup, and crashpad/adopted descendants; if Unix process groups are used, validate positive PIDs and only group-kill when the child was actually spawned into its own group. If a shared process-tree helper is needed, expose it through an appropriate plugin SDK process-runtime seam rather than importing core internals from the plugin. Add focused regression coverage for invalid PID handling, Unix/group or descendant cleanup behavior, Windows/direct-process fallback, bootstrap cleanup, and normal-exit cleanup. Add a `CHANGELOG.md` fix entry because this is user-visible reliability behavior. Stop if the design turns into a product/config decision or broad process-supervisor rewrite.
