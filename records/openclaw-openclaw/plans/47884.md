## Summary

Keep open. Current main still only partially handles custom memory embedding provider IDs: it can look up a configured API owner, but the reported `api: "openai-completions"` path does not reach the OpenAI memory adapter while preserving the custom provider's endpoint and auth.

## Plan

Fix https://github.com/openclaw/openclaw/issues/47884. Current main does not preserve a custom `models.providers.<id>` provider when `agents.defaults.memorySearch.provider` points at that ID and the provider uses `api: "openai-completions"`; lookup tries the API owner but the OpenAI embedding adapter/client then force `provider: "openai"`, losing the custom `baseUrl`, headers, and auth. Implement a generic OpenAI-compatible memory embedding path that preserves the requested custom provider ID for endpoint/auth/model-prefix resolution while selecting the OpenAI adapter/family. Do not add Bailian-specific policy, do not add a new config option, and do not solve this by requiring users to switch to a separate plugin. Add regression coverage for custom `openai-completions` provider lookup, OpenAI adapter option forwarding/cache data, remote bearer client endpoint/auth selection, and the gateway or memory_search path. Add a user-facing changelog fix entry if behavior changes are landed.
