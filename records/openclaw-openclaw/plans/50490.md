## Summary

Keep open. Current main and v2026.5.22 still have the reported Feishu gap: `/activation mention` persists session activation state, but Feishu group admission still decides dispatch from config-derived `requireMention` before that session state can hard-stop non-mention messages.

## Plan

Fix Feishu group activation so `/activation mention` is enforced before dispatch. Current source behavior: `src/auto-reply/reply/commands-session.ts` persists `sessionEntry.groupActivation`, but `extensions/feishu/src/bot.ts` calls `resolveFeishuReplyPolicy` and `resolveFeishuGroupSenderActivationIngressAccess` using only config-derived `requireMention`, so unmentioned messages can still dispatch after activation is switched to `mention`. Keep the fix inside the Feishu/plugin-owned boundary, preserve existing `requireMention` config defaults and pending-history recording, do not add a new config option, and do not move Feishu-specific policy into core. Add regression coverage for an open/`requireMention: false` Feishu group where stored `groupActivation: "mention"` blocks an unmentioned message, while `groupActivation: "always"` and directly mentioned messages still dispatch. Add a one-line `CHANGELOG.md` Fixes entry if this repair PR is maintainer-owned.
