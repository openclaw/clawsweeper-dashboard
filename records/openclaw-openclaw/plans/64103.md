# Coding Plan for openclaw/openclaw#64103: Session `status` field values (\"failed\", \"timeout\", \"done\") mislead agents into spawning duplicate sessions

Source report: [records/openclaw-openclaw/items/64103.md](records/openclaw-openclaw/items/64103.md)

## Summary

Keep open. Current main still exposes terminal-looking session statuses through persistence, gateway rows, and agent-facing `sessions_list` without explicit resumability/status-scope metadata, while `sessions.send` can still target and reactivate completed subagent sessions.

## Plan

Fix https://github.com/openclaw/openclaw/issues/64103 by preserving the existing `status` values for compatibility while adding explicit resumability/status-scope or targetability metadata to Gateway session rows and the agent-facing `sessions_list` output. Update the tool description/system prompt and public session-tool/subagent docs so agents understand `failed`, `timeout`, and `done` are last-turn/session-run states and should not by themselves cause a new `sessions_spawn`. Add focused regression coverage for `sessions_list` rows and completed/failed/timeout subagent follow-up behavior. Do not do a breaking rename of the existing `status` union, do not add a config option, and do not change unrelated session visibility or provider routing behavior. Add a changelog entry because this is user-visible agent/session behavior.
