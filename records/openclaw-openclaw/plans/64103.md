# Coding Plan for openclaw/openclaw#64103: Session `status` field values (\"failed\", \"timeout\", \"done\") mislead agents into spawning duplicate sessions

Source report: [records/openclaw-openclaw/items/64103.md](records/openclaw-openclaw/items/64103.md)

## Summary

Keep open. Current main and v2026.5.12 still expose terminal-looking legacy session statuses through persistence, gateway rows, and agent-facing `sessions_list`, while `sessions.send` can still target an existing `sessionId` and reactivate completed subagent sessions.

## Plan

Fix the ambiguous session status contract reported at https://github.com/openclaw/openclaw/issues/64103. Keep the existing `running | done | failed | killed | timeout` status values for backward compatibility; do not implement the breaking rename. Add explicit backward-compatible metadata that tells gateway clients and agent-facing `sessions_list` consumers whether a session is resumable/targetable by `sessions_send` and what scope the `status` value represents. Use current send constraints as the source of truth: a row with a usable `sessionId` and allowed send policy should not look permanently dead merely because the last/current run status is `done`, `failed`, or `timeout`; rows without a transcript/session id or with denied/deleted targets should not be marked resumable. Update `src/config/sessions/types.ts`, `src/gateway/session-utils.types.ts`, `src/gateway/session-utils.ts`, `src/agents/tools/sessions-helpers.ts`, `src/agents/tools/sessions-list-tool.ts`, tool descriptions/system prompt guidance, docs for session/subagent tools, focused tests, and a changelog entry. Use https://github.com/openclaw/openclaw/issues/69202 and https://github.com/openclaw/openclaw/issues/71756 as context only; do not implement compaction watchdogs, broad lifecycle finalization, or cleanup-policy changes in this PR.
