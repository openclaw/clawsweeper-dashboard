# Coding Plan for openclaw/openclaw#64103: Session `status` field values (\"failed\", \"timeout\", \"done\") mislead agents into spawning duplicate sessions

Source report: [records/openclaw-openclaw/items/64103.md](records/openclaw-openclaw/items/64103.md)

## Summary

Keep open. Current main still exposes terminal-looking `status` values through the persisted session contract, Gateway rows, and `sessions_list` without a companion resumability or status-scope field, while `sessions.send` can still target an existing session by `sessionId` and reactivate completed sub-agent sessions.

## Plan

Fix https://github.com/openclaw/openclaw/issues/64103 by making the existing session status contract explicit and resumable without renaming the current `running | done | failed | killed | timeout` values. Add backward-compatible metadata to Gateway session rows and the agent-facing `sessions_list` result, such as a boolean resumability signal and/or a status-scope field that clearly states the status is the latest run/turn state, not permanent session lifecycle. Update `sessions_send`/`sessions_list` descriptions, system prompt guidance, and public session/subagent docs so orchestrator agents prefer `sessions_send` for an existing resumable session before spawning a duplicate. Add focused regression coverage showing `done`, `failed`, and `timeout` rows carry the new clarity metadata and that `sessions_send` can still target an existing sessionId/reactivate completed sub-agent sessions. Do not implement the broader `sessions.activity.get/list` API from https://github.com/openclaw/openclaw/issues/39127, do not add a config option, and do not change or remove existing status values. Add a concise `CHANGELOG.md` fix entry because this changes agent-facing API/tool behavior.
