# Coding Plan for openclaw/openclaw#64103: Session `status` field values (\"failed\", \"timeout\", \"done\") mislead agents into spawning duplicate sessions

Source report: [records/openclaw-openclaw/items/64103.md](records/openclaw-openclaw/items/64103.md)

## Summary

Keep open. Current main still exposes terminal-looking legacy session statuses through persistence, gateway rows, and agent-facing `sessions_list` without explicit resumability or status-scope metadata, while `sessions.send` can still target and reactivate existing sessions.

## Plan

Fix the session status ambiguity reported at https://github.com/openclaw/openclaw/issues/64103. Preserve the existing `running | done | failed | killed | timeout` status values for compatibility; do not do a breaking rename. Add explicit resumability/status-scope or targetability metadata to gateway session rows and agent-facing `sessions_list` so agents can distinguish last-turn status from whether an existing session can receive `sessions_send`. Ensure `sessions_list` forwards the new metadata, update tool descriptions/system prompt guidance and docs to tell agents to prefer resuming existing sessions when targetable, and add regression coverage for `done`, `failed`, and `timeout` rows plus completed-subagent reactivation via `sessions.send`. Include a changelog entry because this changes user-facing agent/tool behavior. Do not alter unrelated session cleanup, compaction, provider, or Telegram behavior.
