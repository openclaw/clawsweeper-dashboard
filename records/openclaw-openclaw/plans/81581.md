## Summary

Keep open: current main still has the reported Telegram CLI bridge bug. The Telegram static action adapter contains the `thread-create` to `topic-create` remap, but the exported Telegram plugin action wrapper does not forward `resolveCliActionRequest`, so the CLI can still dispatch bare `thread-create` into Telegram handling.

## Plan

Fix https://github.com/openclaw/openclaw/issues/81581. Current main drops the Telegram `thread-create` to `topic-create` CLI remap because `extensions/telegram/src/channel.ts` exports a `telegramMessageActions` wrapper that does not forward `resolveCliActionRequest` from the optional runtime or `telegramMessageActionsImpl`. Add the missing forwarding in the Telegram wrapper while preserving existing fallback behavior. Add a focused regression test against the real exported `telegramPlugin.actions` surface proving `resolveCliActionRequest({ action: "thread-create", args: { channel: "telegram", target: "-100123", threadName: "Foo" } })` returns `action: "topic-create"`, maps `threadName` to `name`, and removes `threadName`. Keep non-Telegram/default `thread-create` behavior unchanged, do not add a broad `thread-create` alias to the Telegram runtime unless a maintainer explicitly asks, and preserve the existing `topicId` result field if you add any Telegram-shaped result alias. Contributor changelog is not required unless the maintainer landing flow asks for it.
