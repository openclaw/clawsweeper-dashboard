# Coding Plan for openclaw/openclaw#40838: [Feature]: requirement: Expand Telegram message field information for the agent.

Source report: [records/openclaw-openclaw/items/40838.md](records/openclaw-openclaw/items/40838.md)

## Summary

Keep open. Current main and latest v2026.5.22 still expose Telegram sender id/name/username to agent context but do not carry Telegram `msg.from.is_bot`; the existing shared `SenderFacts.isBot` field makes this a narrow additive context-plumbing task rather than a broad routing redesign.

## Plan

Fix https://github.com/openclaw/openclaw/issues/40838 by exposing the current Telegram sender's bot status to agent-visible inbound context. Populate `SenderFacts.isBot` from `msg.from?.is_bot` in `extensions/telegram/src/bot-message-context.session.ts`, carry it through `buildChannelInboundEventContext` into `MsgContext`/finalized context, and render it in the dynamic untrusted sender or conversation metadata. Do not expose raw `msg.from`, do not change requireMention, auth, allowlist, bot-loop, or bot-routing policy, and keep volatile sender data out of stable system metadata. Coordinate naming with https://github.com/openclaw/openclaw/issues/50482 and the bot-targeting fields in https://github.com/openclaw/openclaw/pull/72510 so the contracts do not diverge. Add focused regression tests for Telegram current sender `is_bot: true/false`, shared context mapping, and rendered inbound metadata; include a changelog entry if maintainers treat the prompt metadata as user-visible.
