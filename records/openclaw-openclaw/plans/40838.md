# Coding Plan for openclaw/openclaw#40838: [Feature]: requirement: Expand Telegram message field information for the agent.

Source report: [records/openclaw-openclaw/items/40838.md](records/openclaw-openclaw/items/40838.md)

## Summary

Keep open. Current main still has Telegram sender data and a shared `SenderFacts.isBot` contract, but the Telegram inbound payload, finalized `MsgContext`, and rendered sender metadata only expose sender id/name/username, so the agent still cannot classify bot-sourced Telegram messages from context.

## Plan

Fix the Telegram sender-bot metadata gap for this issue. Add a backward-compatible sender bot boolean by carrying `msg.from?.is_bot` from `extensions/telegram/src/bot-message-context.session.ts` into the existing `SenderFacts.isBot` contract, mapping it through `src/channels/inbound-event/context.ts` into `MsgContext`, and rendering it in `src/auto-reply/reply/inbound-meta.ts` as bounded sender metadata. Add/update focused regression tests for Telegram payload construction, shared inbound-event context, and rendered inbound metadata. Update relevant channel/plugin SDK docs and add a changelog entry if the new agent-visible field is user-facing. Do not expose raw Telegram `msg.from` objects, add Telegram bot-to-bot routing policy, add config options, or attempt the broader work tracked in https://github.com/openclaw/openclaw/issues/79077. Recheck interactions with https://github.com/openclaw/openclaw/pull/72510 if it lands first.
