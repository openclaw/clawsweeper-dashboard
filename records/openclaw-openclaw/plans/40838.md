## Summary

Keep open. Current main and v2026.5.19 still have Telegram sender data and the shared `SenderFacts.isBot` contract, but the Telegram context payload, `MsgContext`, and rendered inbound metadata do not expose bot classification to agents.

## Plan

Fix https://github.com/openclaw/openclaw/issues/40838 by exposing Telegram sender bot classification to agent-visible context without exposing raw Telegram message/user objects. Use the existing `SenderFacts.isBot` contract: populate it from `msg.from?.is_bot` in Telegram inbound context construction, add a bounded `MsgContext` field and rendered inbound metadata field, and update focused tests/docs. Keep the scope to sender classification; do not implement Telegram bot-to-bot routing, loop policy, senderTypePolicy, rate limits, or peer maps from https://github.com/openclaw/openclaw/issues/79077. Coordinate naming with adjacent bot-targeting work in https://github.com/openclaw/openclaw/pull/72510 so the field does not conflict with target-bot/reply-to-bot metadata. Add a user-facing changelog entry because this changes prompt/context behavior.
