# Coding Plan for openclaw/openclaw#40838: [Feature]: requirement: Expand Telegram message field information for the agent.

Source report: [records/openclaw-openclaw/items/40838.md](records/openclaw-openclaw/items/40838.md)

## Summary

Keep open. Current main still has the Telegram sender data and a shared `SenderFacts.isBot` contract, but the Telegram inbound context, finalized `MsgContext`, and rendered agent metadata only expose sender id/name/username/tag fields, so agents still cannot classify bot-sourced Telegram messages from context.

## Plan

Implement the additive sender-bot metadata requested in https://github.com/openclaw/openclaw/issues/40838. Keep the scope narrow: do not expose raw Telegram `msg.from`, do not add Telegram bot-to-bot routing/config policy, and do not implement the voice-origin request from https://github.com/openclaw/openclaw/issues/50482. Use the existing `SenderFacts.isBot` contract: pass `msg.from?.is_bot` from `extensions/telegram/src/bot-message-context.session.ts`, add a backward-compatible agent-visible `MsgContext` field, map it in `src/channels/inbound-event/context.ts`, render it in the sender or conversation inbound metadata, and update public docs plus focused regression tests. Recheck https://github.com/openclaw/openclaw/pull/72510 before editing in case it landed or changed field naming. Add a changelog entry because this changes user-visible prompt/context behavior.
