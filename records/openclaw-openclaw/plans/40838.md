## Summary

Keep open. Current main still has the Telegram sender source object and a shared `SenderFacts.isBot` contract, but the Telegram turn builder, finalized message context, and rendered inbound metadata expose only sender id/name/username without bot classification; the broader Telegram bot-to-bot design in https://github.com/openclaw/openclaw/issues/79077 and open bot-targeting PR https://github.com/openclaw/openclaw/pull/72510 are adjacent, not replacements for this focused metadata gap.

## Plan

Implement the focused metadata request from https://github.com/openclaw/openclaw/issues/40838. Source a trusted sender-bot boolean from Telegram `msg.from?.is_bot`, pass it as `SenderFacts.isBot` from `extensions/telegram/src/bot-message-context.session.ts` into `buildChannelTurnContext`, add an agent-visible context field such as `SenderIsBot`, and render it as a boolean in the inbound sender metadata block. Do not expose raw Telegram `msg.from`, add sender-type config, implement bot-to-bot routing, or change allowlist/loop policy; leave that broader design to https://github.com/openclaw/openclaw/issues/79077. Recheck overlap with https://github.com/openclaw/openclaw/pull/72510 if it lands first. Add focused regression coverage in the shared turn context, Telegram inbound payload, and inbound metadata tests, update the channel-turn/Telegram docs for the new field, and add a changelog entry if maintainer landing policy requires it.
