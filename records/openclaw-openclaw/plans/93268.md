## Summary

Keep open. The reported fast-path status bug is real, but this patch uses the generic `chunks` table as the readiness signal, which can over-report `Vector store: ready` for degraded indexes where sqlite-vec was unavailable and no `chunks_vec` rows were written.

## Plan

Repair PR https://github.com/openclaw/openclaw/pull/93268 without broad refactors. The central bug is valid: plain `openclaw memory status` can show `Vector store: unknown` after a successful vector index because status-only managers do not probe sqlite-vec. Do not use generic `hasIndexedChunks()` alone as the store-ready fallback because `chunks` rows also exist when vector writes were skipped. Change `extensions/memory-core/src/memory/manager.ts` to infer fast-path store readiness only from durable vector-store evidence, such as persisted vector dimensions plus indexed content, while preserving `semanticAvailable` as unprobed. Add or update colocated memory-core tests covering: an indexed vector DB reopened in status mode reports `storeAvailable: true` without provider initialization, and a vector-enabled degraded/FTS-only DB with chunks but no vector-store evidence does not report `storeAvailable: true`. Do not change config, migrations, CLI flags, docs, or package metadata.
