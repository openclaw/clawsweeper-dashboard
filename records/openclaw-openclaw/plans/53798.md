# Coding Plan for openclaw/openclaw#53798: [Bug]: Feishu voice message sent via message tool does not display audio duration

Source report: [records/openclaw-openclaw/items/53798.md](records/openclaw-openclaw/items/53798.md)

## Summary

Keep open. Current main still sends Feishu native audio through the upload helper without computing or passing duration, and the latest release has the same omission; the related duration PRs were closed unmerged, so this remains the canonical narrow bug.

## Plan

Fix https://github.com/openclaw/openclaw/issues/53798. In the Feishu plugin, compute a finite millisecond duration for the final bytes uploaded by `sendMediaFeishu` when the resolved Feishu upload is audio/video (`file_type: "opus"` or `"mp4"`) and pass it through the existing `uploadFileFeishu({ duration })` parameter. The duration must be based on the actual uploaded buffer after `prepareFeishuVoiceMedia` has either preserved native Ogg/Opus or transcoded voice-intent audio, and upload/send should continue gracefully if duration probing fails. Keep the fix inside the Feishu/plugin media boundary, preserve existing routing and fallback behavior, avoid shell-string ffprobe/ffmpeg calls, and do not turn ordinary MP3 attachments into voice messages. Add or update focused regression tests for native Opus/Ogg and voice-intent transcoded audio verifying the upload payload includes `duration`, plus non-audio/file fallback coverage proving no duration is sent. Add a user-facing changelog entry because this fixes Feishu voice-message display behavior. Validate with the listed Feishu tests and a changed check in Testbox/Crabbox if broader proof is needed.
