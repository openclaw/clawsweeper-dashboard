## Summary

Keep open. Current main and v2026.5.22 still send Feishu Ogg/Opus voice media through `uploadFileFeishu` without computing or passing `duration`, while the SDK contract and OpenClaw upload helper already support that field and the overlapping duration PRs were closed unmerged.

## Plan

Fix https://github.com/openclaw/openclaw/issues/53798 with a narrow Feishu plugin patch. The observable bug is that Feishu voice/audio messages sent through the message tool or voice-intent reply path upload as `file_type: "opus"`/`msg_type: "audio"` but do not pass `duration`, so Feishu does not show the audio duration. Preserve the current Feishu media routing policy, download metadata handling, `audioAsVoice` transcode behavior, local-root security checks, and ordinary MP3-as-file behavior. Derive a finite millisecond duration from the final buffer/name/content type after `prepareFeishuVoiceMedia`, pass it to `uploadFileFeishu` only for audio/video uploads, and gracefully omit it when probing fails. Prefer existing media-runtime helpers such as `runFfprobe` with argument arrays/stdin over shell strings or new runtime dependencies; if a new dependency is truly needed, update the Feishu package metadata and lockfile deliberately. Add focused tests in `extensions/feishu/src/media.test.ts` proving duration is included for native Ogg/Opus and post-transcode voice uploads, omitted on probe failure, and existing image/MP3 fallback routing still holds. Add a user-facing changelog Fixes entry for the Feishu duration behavior.
