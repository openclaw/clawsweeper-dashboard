## Summary

Keep open: current main still drops Feishu block payloads unless they can be consumed by the streaming-card path, and the latest shipped release contains the same behavior. The related open core PR targets final-payload suppression after block streaming, but it does not close this Feishu dispatcher bug.

## Plan

Fix the Feishu block-streaming delivery bug reported at https://github.com/openclaw/openclaw/issues/55027. Keep the existing Feishu config surface: `blockStreaming: true` should opt Feishu into core block payloads, and completed block payloads should be delivered via normal Feishu card/text send helpers without requiring Card Kit preview streaming. Preserve the earlier safety behavior that disables core block streaming unless Feishu explicitly opts in, preserve media delivery, duplicate-final suppression, streaming-card final safeguards, reply/thread options, and renderMode behavior. Do not attempt to solve the broader core final-payload suppression tracked by https://github.com/openclaw/openclaw/issues/78942 or https://github.com/openclaw/openclaw/pull/62157 except where a small local test fixture needs to avoid regressions. Add focused regression coverage in the Feishu dispatcher tests for `blockStreaming: true` with `streaming: false` and for `streaming: true` proving block payloads are not only consumed by streaming-card playback when the expected behavior is completed-block delivery. Add a user-facing changelog Fixes entry if the repair changes shipped behavior.
