# Coding Plan for openclaw/openclaw#55027: Feishu channel cannot do true block streaming; block chunks are dropped unless streaming card is enabled

Source report: [records/openclaw-openclaw/items/55027.md](records/openclaw-openclaw/items/55027.md)

## Summary

Keep open. Current main and v2026.5.18 still route Feishu block payloads through the streaming-card path: with preview streaming disabled they return before normal Feishu send helpers run, and with preview streaming enabled they are consumed by Card Kit playback instead of delivered as completed-block messages.

## Plan

Fix Feishu true block streaming for https://github.com/openclaw/openclaw/issues/55027. Keep the existing `channels.feishu.streaming` and `channels.feishu.blockStreaming` config surface; do not add a new flag and do not change the default opt-in semantics. In the Feishu plugin, make explicit `{ kind: "block" }` deliveries with `blockStreaming: true` send completed block text through the normal Feishu card/text send helpers even when preview `streaming` is false, instead of requiring `streamingEnabled && useCard`. Preserve existing preview streaming-card behavior, media handling, duplicate-final suppression, thread/root routing, and recent fixes for tool-call-heavy replies. Add focused regression coverage in the Feishu dispatcher tests for `blockStreaming: true`, `streaming: false`, and `renderMode: "card"` proving a block payload is delivered through normal Feishu sends. Include docs only if existing text becomes inaccurate, and include a changelog entry if the repair is user-facing.
