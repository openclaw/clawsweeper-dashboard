# Coding Plan for openclaw/openclaw#55027: Feishu channel cannot do true block streaming; block chunks are dropped unless streaming card is enabled

Source report: [records/openclaw-openclaw/items/55027.md](records/openclaw-openclaw/items/55027.md)

## Summary

Keep open: current main still drops Feishu block payloads when the streaming-card path cannot consume them, and when the card path is active it treats blocks as card deltas rather than normal completed-block messages.

## Plan

Fix Feishu completed-block delivery for https://github.com/openclaw/openclaw/issues/55027. Current main accepts `channels.feishu.blockStreaming: true` and passes block payloads to the Feishu dispatcher, but `extensions/feishu/src/reply-dispatcher.ts` returns from `{ kind: "block" }` delivery unless the streaming-card path is active, and the active card path consumes blocks as `queueStreamingUpdate(..., { mode: "delta" })` instead of normal completed-block messages. Implement the fix inside the Feishu plugin dispatcher so explicit `blockStreaming: true` block payloads deliver through the normal Feishu send helpers when block delivery is requested, without changing core streaming contracts, without adding a new config option, and without regressing streaming-card preview/final dedupe/media handling. Add focused regression coverage for `blockStreaming: true` with `streaming: false`, and for the card/streaming case if the behavior changes there. Preserve the legacy default where Feishu block streaming remains disabled unless explicitly opted in. Add a user-facing changelog fix entry. Validate with `node scripts/run-vitest.mjs extensions/feishu/src/reply-dispatcher.test.ts extensions/feishu/src/config-schema.test.ts`, an oxfmt check for touched files, and broad changed-check proof through Crabbox/Testbox before handoff.
