# Coding Plan for openclaw/openclaw#55027: Feishu channel cannot do true block streaming; block chunks are dropped unless streaming card is enabled

Source report: [records/openclaw-openclaw/items/55027.md](records/openclaw-openclaw/items/55027.md)

## Summary

Keep open. Current main and v2026.5.18 still route Feishu block payloads through the streaming-card path: with preview streaming disabled they return before normal Feishu send helpers run, and with preview streaming enabled they are consumed by Card Kit playback instead of delivered as completed block messages.

## Plan

Fix the Feishu block streaming delivery bug tracked by https://github.com/openclaw/openclaw/issues/55027. Preserve Feishu's explicit opt-in behavior: default block streaming must remain disabled unless `channels.feishu.blockStreaming` or the per-account equivalent is true. When Feishu `blockStreaming: true` and the runtime delivers `{ kind: "block" }` text, deliver the completed block through the normal Feishu send helpers instead of returning early or requiring `FeishuStreamingSession` Card Kit playback. Keep media handling, duplicate-final suppression, closed streaming-card safeguards, renderMode behavior, and existing streaming-card preview behavior intact. Add focused regression coverage in `extensions/feishu/src/reply-dispatcher.test.ts` for at least `blockStreaming: true` with `streaming: false` and a block payload, and update docs/changelog only if the user-visible behavior description needs it. Do not change core block streaming defaults or unrelated channels.
