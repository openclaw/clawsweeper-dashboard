# Coding Plan for openclaw/openclaw#55027: Feishu channel cannot do true block streaming; block chunks are dropped unless streaming card is enabled

Source report: [records/openclaw-openclaw/items/55027.md](records/openclaw-openclaw/items/55027.md)

## Summary

Keep open. Current main exposes Feishu `blockStreaming`, but block payload delivery is still tied to the Feishu streaming-card path: with preview streaming disabled, block chunks return before any normal Feishu send helper runs, and with preview streaming enabled they are consumed by Card Kit playback rather than true completed-block messages.

## Plan

Fix https://github.com/openclaw/openclaw/issues/55027. In the Feishu plugin, `channels.feishu.blockStreaming: true` should deliver `info.kind === "block"` payloads immediately as normal Feishu outbound messages/cards even when `channels.feishu.streaming` is false, and it should not require Card Kit preview streaming to avoid dropping the block. Keep Feishu `blockStreaming` default off; do not add a new config option or broad core/plugin API. Preserve duplicate-final, closed-streaming-final, media, voice fallback, chunking, threading, mention, and renderMode behavior. Add focused tests showing explicit block streaming sends block payloads through normal Feishu delivery when preview streaming is disabled, while unset/false blockStreaming still suppresses internal block payloads and streaming-card preview behavior still works. Consider related https://github.com/openclaw/openclaw/issues/78942 during design, but do not broaden into unrelated final-payload suppression unless the same testable code path is touched. Avoid using the broad subagent/character-streaming approach from https://github.com/openclaw/openclaw/pull/47007 as the fix shape. Add a changelog entry because this is user-visible Feishu behavior.
