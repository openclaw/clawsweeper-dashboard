# Coding Plan for openclaw/openclaw#14747: Feature request: configurable lane wait diagnostic threshold

Source report: [records/openclaw-openclaw/items/14747.md](records/openclaw-openclaw/items/14747.md)

## Summary

Keep open. Current main and the latest release still store a 2,000 ms default for queued lane wait diagnostics, diagnostics config/schema/docs expose no lane-wait threshold key, and the related implementation PR closed unmerged while the broader concurrency tracker remains separate.

## Plan

Implement a narrow diagnostics.laneWaitWarnMs config option for the lane wait diagnostic threshold. Add a positive integer millisecond field to the diagnostics config schema, exported types, labels/help, generated docs, and configuration reference; wire gateway config application so future enqueueCommandInLane calls without an explicit opts.warnAfterMs use the configured default while preserving the current 2,000 ms fallback when unset. Preserve explicit per-call overrides such as the manual cron run warnAfterMs: 5_000 path, document that already queued entries keep their stored threshold, add/update focused tests for default, configured, override, and reload/future-entry behavior, and add a CHANGELOG.md entry. Do not implement per-lane thresholds, adaptive timeouts, session concurrency, or auto-yield in this repair; those belong to the broader related tracker.
