## Summary

Keep open. Current main and v2026.5.18 still use the 2,000 ms queue-entry fallback, diagnostics config/schema/docs expose no lane-wait threshold, the earlier implementation PR closed unmerged, and the broader concurrency issue does not supersede this narrower logging-control request.

## Plan

Implement the narrow lane-wait diagnostics threshold request from https://github.com/openclaw/openclaw/issues/14747. Add a documented positive-integer `diagnostics.laneWaitWarnMs` config field that defaults to the existing 2,000 ms behavior when unset and applies to future queue entries that do not pass `opts.warnAfterMs`. Preserve explicit per-call overrides such as the manual cron-run `warnAfterMs: 5_000` path. Cover config schema/types/labels/help/generated docs, queue docs, reload behavior for future entries, and regression tests. Do not implement per-lane thresholds, adaptive timeouts, session concurrency, or auto-yield here; those remain part of https://github.com/openclaw/openclaw/issues/53023. Prefer an owner-boundary shape that avoids broad config imports in hot queue code if an existing gateway/config wiring or small queue setter can carry the prepared default. Add a changelog entry because this is a user-visible config improvement.
