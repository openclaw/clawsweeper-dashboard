# Coding Plan for openclaw/openclaw#14747: Feature request: configurable lane wait diagnostic threshold

Source report: [records/openclaw-openclaw/items/14747.md](records/openclaw-openclaw/items/14747.md)

## Summary

Keep open. Current main still stores a hardcoded 2,000 ms lane wait warning default, exposes no `diagnostics.laneWaitWarnMs` schema/type/help/docs surface, and the directly related implementation PR at https://github.com/openclaw/openclaw/pull/16366 is closed unmerged while https://github.com/openclaw/openclaw/issues/53023 is a broader concurrency/auto-yield umbrella, not a replacement for this focused log-noise control.

## Plan

Implement the focused lane wait diagnostic threshold requested by https://github.com/openclaw/openclaw/issues/14747. Add a documented positive integer millisecond config field, preferably `diagnostics.laneWaitWarnMs`, that defaults to the current 2,000 ms behavior when unset and supplies the default warning threshold for future `enqueueCommandInLane` entries. Preserve explicit per-call `opts.warnAfterMs` overrides, including the manual cron run override in `src/cron/service/ops.ts`. Wire the config through the existing gateway startup/reload path so future queue entries can pick up the new value without requiring a restart, and add/update tests for command queue defaults, gateway lane/config application, config reload planning, and cron override behavior. Update schema/types/labels/help/generated docs and user-facing docs, plus `CHANGELOG.md` because this is a user-facing config feature. Do not implement per-lane thresholds, adaptive timeouts, session concurrency, or auto-yield in this PR; those belong to broader concurrency work such as https://github.com/openclaw/openclaw/issues/53023.
