# Coding Plan for openclaw/openclaw#14747: Feature request: configurable lane wait diagnostic threshold

Source report: [records/openclaw-openclaw/items/14747.md](records/openclaw-openclaw/items/14747.md)

## Summary

Keep open. Current main and the latest release still hardcode the default lane wait warning threshold at 2,000 ms, the diagnostics config/docs expose no lane-wait threshold key, and the related implementation/workaround PRs are closed unmerged; the broader session-concurrency issue does not supersede this narrower logging-control request.

## Plan

Implement the narrow lane wait diagnostic threshold config requested by https://github.com/openclaw/openclaw/issues/14747. Add a documented positive-integer `diagnostics.laneWaitWarnMs` config field that defaults to the current 2,000 ms behavior when unset, applies to future `enqueueCommandInLane` entries without explicit `opts.warnAfterMs`, and preserves explicit per-call overrides such as the manual cron run `warnAfterMs: 5_000`. Update schema/types/help/labels/generated configuration docs and queue docs. Cover reload/no-restart behavior deliberately: config changes should affect future queued entries, while already-queued entries can retain their stored threshold. Do not implement per-lane thresholds, adaptive timeouts, session concurrency, or auto-yield; leave that broader work to https://github.com/openclaw/openclaw/issues/53023. Treat closed PRs https://github.com/openclaw/openclaw/pull/16366 and https://github.com/openclaw/openclaw/pull/14889 as context only, not branches to revive wholesale. Add focused regression coverage for default fallback, configured default, explicit override precedence, config reload, and cron override preservation. Add a changelog entry because this is user-visible diagnostics/config behavior.
