## Summary

Keep open. Current main and the latest shipped release still hardcode the lane-wait warning fallback at 2,000 ms, the diagnostics config/schema/docs expose stuck-session tuning but no lane-wait threshold, and the related implementation attempts closed unmerged while the broader concurrency tracker does not replace this narrow logging-control request.

## Plan

Implement the narrow lane-wait diagnostic threshold feature for https://github.com/openclaw/openclaw/issues/14747. Add a documented positive-integer `diagnostics.laneWaitWarnMs` config field that defaults to the existing 2,000 ms behavior when unset, applies to future queue entries that do not pass `opts.warnAfterMs`, and preserves explicit per-call overrides such as the manual cron-run 5,000 ms override. Update schema/types/help/labels/generated configuration docs, queue docs, reload-plan behavior/tests as needed, and add focused regression coverage. Do not implement per-lane thresholds, adaptive timeouts, session concurrency, or auto-yield; those belong to https://github.com/openclaw/openclaw/issues/53023. Include a user-facing changelog entry unless maintainers decide the landing process will add it.
