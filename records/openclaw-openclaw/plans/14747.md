# Coding Plan for openclaw/openclaw#14747: Feature request: configurable lane wait diagnostic threshold

Source report: [records/openclaw-openclaw/items/14747.md](records/openclaw-openclaw/items/14747.md)

## Summary

Keep open. Current main and the latest release still hardcode the default lane-wait warning threshold at 2,000 ms, diagnostics config/schema/docs expose no lane-wait threshold key, and the implementation PR that referenced this issue closed unmerged while the broader concurrency issue remains a separate product thread.

## Plan

Fix https://github.com/openclaw/openclaw/issues/14747 by adding a narrow, documented `diagnostics.laneWaitWarnMs` positive integer millisecond config field for the command queue lane-wait diagnostic threshold. Keep unset behavior at the current 2,000 ms default, preserve explicit per-call `warnAfterMs` overrides including cron manual runs, and make config reload semantics explicit so new future queue entries use the latest value without rewriting already queued entries. Update schema/types/help/labels/generated config docs and queue docs; add focused tests for default behavior, configured behavior, explicit override precedence, and reload/future-entry behavior. Do not implement per-lane thresholds, adaptive timeouts, session concurrency, or auto-yield; leave that broader work to https://github.com/openclaw/openclaw/issues/53023. Add a single-line changelog entry because this is user-visible config/diagnostics behavior.
