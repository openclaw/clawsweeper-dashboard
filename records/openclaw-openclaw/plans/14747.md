## Summary

Keep open. Current main and the latest tagged release still use a hardcoded queue-entry lane-wait warning default, the diagnostics config/schema/docs do not expose a lane wait threshold, and the only focused implementation PR is closed unmerged while the broader concurrency issue does not supersede this narrower request.

## Plan

Implement the narrow lane wait diagnostic threshold config requested in https://github.com/openclaw/openclaw/issues/14747. Add a positive millisecond `diagnostics.laneWaitWarnMs` field that defaults to the existing 2,000 ms behavior when unset and applies to future queued entries without an explicit `warnAfterMs`. Preserve explicit per-call overrides, including the manual cron run `warnAfterMs: 5_000` path, unless a maintainer explicitly asks to change that contract. Update config schema/types/help/labels/generated docs, queue docs, reload planning/tests so config reload behavior is explicit, and focused queue/gateway/cron tests. Do not expand into per-lane thresholds, adaptive timeouts, session concurrency, auto-yield, or diagnostic text changes already covered by https://github.com/openclaw/openclaw/pull/82792. Add a user-facing changelog entry if the repair is landed.
