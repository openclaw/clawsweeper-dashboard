# Coding Plan for openclaw/openclaw#14747: Feature request: configurable lane wait diagnostic threshold

Source report: [records/openclaw-openclaw/items/14747.md](records/openclaw-openclaw/items/14747.md)

## Summary

Keep open. Current main and the latest shipped release still hardcode the lane-wait warning fallback at 2,000 ms, diagnostics config/schema/docs do not expose a lane-wait threshold, the earlier implementation PR closed unmerged, and the broader concurrency issue does not supersede this narrow logging-control request.

## Plan

Implement the narrow lane-wait diagnostic threshold feature for https://github.com/openclaw/openclaw/issues/14747. Add a documented positive-integer `diagnostics.laneWaitWarnMs` config field that sets the default `warnAfterMs` for future queue entries when callers do not pass an explicit override, preserves the current 2,000 ms fallback when unset, preserves explicit per-call overrides such as cron manual runs, and updates startup/reload behavior so config changes apply to newly enqueued work without claiming already queued entries are retuned. Do not implement per-lane thresholds, adaptive timeouts, session concurrency, or auto-yield here; leave that to https://github.com/openclaw/openclaw/issues/53023. Update config schema/types/labels/help/generated docs, queue/config docs, focused tests, and CHANGELOG because this is user-visible config. Use https://github.com/openclaw/openclaw/pull/16366 only as prior context, not as a patch to copy wholesale because it bundled unrelated work.
