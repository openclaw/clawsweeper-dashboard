# Coding Plan for openclaw/openclaw#14747: Feature request: configurable lane wait diagnostic threshold

Source report: [records/openclaw-openclaw/items/14747.md](records/openclaw-openclaw/items/14747.md)

## Summary

Keep open. Current main and v2026.5.20 still use the hardcoded 2,000 ms queue-entry warning fallback, diagnostics config/schema/docs have no lane-wait threshold, and the related implementation attempts are closed unmerged.

## Plan

Implement the narrow lane-wait diagnostic threshold config tracked by https://github.com/openclaw/openclaw/issues/14747. Add a documented positive-integer diagnostics.laneWaitWarnMs field that controls the default warnAfterMs for future command queue entries when callers do not pass opts.warnAfterMs, preserving the current 2_000 ms fallback when unset. Preserve explicit per-call overrides, especially the manual cron warnAfterMs: 5_000 path. Update config schema/types/labels/help/generated docs, queue/config docs, and config reload planning/tests so future entries can pick up config changes without implying already-queued entries are retuned. Do not implement per-lane thresholds, adaptive timeouts, session concurrency, or auto-yield; those belong to https://github.com/openclaw/openclaw/issues/53023. Add focused regression tests and a changelog entry for the user-visible config option.
