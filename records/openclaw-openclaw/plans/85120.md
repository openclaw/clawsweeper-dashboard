Keep open: current main still blocks in-band package updates only when the managed gateway PID is present in `getSelfAndAncestorPidsSync()`, and that helper still stops at the immediate parent on macOS. The linked landed fixes addressed adjacent cwd and lazy-import restart failures, but the maintainer comment and current source both leave the nested LaunchAgent ownership case unresolved.

## Plan

Fix the in-band macOS LaunchAgent `openclaw update` bug from https://github.com/openclaw/openclaw/issues/85120. The expected boundary is the existing package-update safety contract: a chat/tool-launched updater must refuse before `maybeStopManagedServiceBeforePackageUpdate` calls `service.stop(...)` when the active LaunchAgent gateway is any ancestor of the updater, or it must hand off to launchd-owned supervision with durable status before stopping the gateway. Prefer a narrow fix around `getSelfAndAncestorPidsSync()` / macOS ancestor detection and the update CLI guard; do not weaken Linux/systemd ancestor handling, do not break managed update handoff helpers that intentionally survive gateway exit, and do not add a new config option.
