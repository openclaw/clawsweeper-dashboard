# Coding Plan for openclaw/openclaw#85120: [Bug]: in-band `openclaw update` on macOS LaunchAgent can stop the gateway supervising it

Source report: [records/openclaw-openclaw/items/85120.md](records/openclaw-openclaw/items/85120.md)

## Summary

Keep open: current main still has the ancestry-dependent refusal, but macOS only includes `process.ppid`, so a nested OpenClaw tool/chat-launched updater can miss the LaunchAgent gateway ancestor and enter the managed service stop path.

## Plan

Fix https://github.com/openclaw/openclaw/issues/85120. Preserve the existing safety contract: `openclaw update` invoked from a nested macOS LaunchAgent/gateway-owned OpenClaw tool/chat process must refuse before `maybeStopManagedServiceBeforePackageUpdate()` reaches `service.stop`, or must hand off to a launchd-owned durable updater before any gateway stop. Prefer a bounded repair around `getSelfAndAncestorPidsSync()` or the update CLI ancestry guard that detects transitive macOS ancestors using a platform-supported process lookup; do not add a config option, do not weaken the Linux `/proc` behavior, and do not break the control-plane `OPENCLAW_UPDATE_RUN_HANDOFF` managed-service handoff that intentionally runs after the gateway parent exits. Add focused regression coverage for nested macOS ancestry and the update CLI refusal, keep related systemd handoff behavior intact, and add a changelog entry if the PR is user-facing.
