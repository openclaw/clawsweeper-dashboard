## Summary

Keep open. Current main still allows a mixed Telegram network error to promote the transport to IPv4-only before that fallback has succeeded, and a following loopback `ECONNREFUSED` path can leave the sticky attempt on the dead IPv4 dispatcher.

## Plan

Fix the Telegram sticky IPv4 fallback dead-latch described in https://github.com/openclaw/openclaw/issues/55099. Keep the change inside the Telegram plugin transport surface, primarily `extensions/telegram/src/fetch.ts` and focused tests in `extensions/telegram/src/fetch.test.ts`. Preserve resolver-scoped dispatchers and existing IPv4 fallback behavior for genuine timeout/unreachable IPv6 cases, but prevent a mixed aggregate primary failure from permanently promoting to an IPv4-only dispatcher when the IPv4 candidate is a loopback/refused dead path. Add regression coverage for `dnsResultOrder: "verbatim"` with nested errors containing IPv4 loopback `ECONNREFUSED` plus IPv6 `ETIMEDOUT`/`EHOSTUNREACH`, and for sticky state after the IPv4-only attempt fails. Do not add a new config option, change global DNS defaults, or move this into core networking. Update docs/changelog only if the behavior or diagnostics exposed to users changes.
