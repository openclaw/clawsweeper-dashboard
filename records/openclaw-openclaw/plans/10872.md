# Coding Plan for openclaw/openclaw#10872: iMessage: Include reply/thread context in incoming messages

Source report: [records/openclaw-openclaw/items/10872.md](records/openclaw-openclaw/items/10872.md)

## Summary

Keep open: current main still only forwards iMessage reply context when the incoming payload already has resolved `reply_to_*` fields, and the active monitor still does not resolve GUID-only inline reply metadata from `chat.db`. The prior implementation PR was closed unmerged because it targeted the removed `src/imessage` path, so a focused replacement against the active plugin remains useful.

## Plan

Fix the open iMessage inbound reply-context gap tracked by https://github.com/openclaw/openclaw/issues/10872. Implement against the active `extensions/imessage/src/monitor/*` plugin path only; do not revive or edit removed `src/imessage` paths, do not add core hooks, and do not add new config keys. When an inbound iMessage notification lacks resolved `reply_to_*` context but can be tied to reliable inline-reply origin metadata, resolve the quoted message through the existing `channels.imessage.dbPath` / account `dbPath` surface and populate `ReplyToId`, `ReplyToBody`, and `ReplyToSender` before dispatch. Prefer the upstream imsg `thread_originator_guid` contract over unreliable `reply_to_guid`; if fallback DB lookup is needed, make it read-only, local-runtime-only, bounded, and fail closed for missing dbPath, permissions, database locks, old schema, missing quoted rows, and ID/ROWID mismatch. Preserve existing resolved-field behavior and coalescing behavior. Add focused regression coverage for GUID-only replies, already-resolved replies, missing quoted row, absent dbPath, schema/failing lookup, and no-overwrite behavior. Add a `CHANGELOG.md` fixes entry for the user-visible iMessage behavior if the implementation lands.
