## Summary

Keep open: current main and the latest release still only forward already-resolved iMessage `reply_to_*` fields and do not resolve inline-reply GUID metadata from `chat.db`. The previously linked implementation was closed unmerged because it targeted a removed path, so a focused replacement against the active plugin is still useful.

## Plan

Implement a focused replacement for inbound iMessage reply/thread context in the active plugin. Work only under `extensions/imessage/src/monitor/*` plus focused tests/docs if needed. Use the existing `channels.imessage.dbPath` or account `dbPath` surface; do not add a new config key, do not modify core channel contracts, and do not resurrect the removed `src/imessage` path. Accept/parse reliable GUID metadata from imsg, prefer `thread_originator_guid` over unreliable `reply_to_guid` when resolving the iMessage UI reply target, and map resolved quoted message id/body/sender into the existing `reply_to_id`, `reply_to_text`, and `reply_to_sender` path before dispatch. The lookup must fail closed on missing dbPath, remote wrapper setups, permissions, database locks, old schemas, and missing rows, and must not leak raw chat.db ROWIDs as public message ids beyond existing short-id behavior. Reuse lessons from https://github.com/openclaw/openclaw/pull/42683 but port to the active `extensions/imessage` files and add regression coverage for GUID-only replies and failure modes. Add a changelog entry only if maintainers decide this user-visible feature should land in the release notes.
