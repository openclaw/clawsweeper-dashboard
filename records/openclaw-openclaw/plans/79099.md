# Coding Plan for openclaw/openclaw#79099: Windows gateway probe still reports unreachable while gateway health is OK on 2026.5.6

Source report: [records/openclaw-openclaw/items/79099.md](records/openclaw-openclaw/items/79099.md)

## Summary

Keep open. Current main and the latest release still collapse pre-hello WebSocket closes, protocol mismatches, and generic `gateway closed` doctor failures into unreachable or not-running diagnostics, even though the docs define accepted WebSocket connect as reachable and the report has multiple real-world reproductions.

## Plan

Fix the gateway probe/doctor diagnostic contradiction reported in https://github.com/openclaw/openclaw/issues/79099. Current main records `connectLatencyMs` only after `hello-ok`, so accepted-but-closed handshakes such as protocol mismatch, auth/device-identity failure, or pre-hello close are classified as unreachable and doctor may print `Gateway not running.` while port diagnostics show a listener. Add a narrow tri-state or equivalent canonical diagnostic shape that distinguishes no listener from accepted-but-errored and fully healthy/reachable. Apply it through `gateway probe`, gateway status warning/output/JSON, and doctor gateway health/daemon flows. Preserve existing auth, pairing, origin, protocol rejection, and credential behavior; do not add config/env flags. Add focused regression tests for pre-hello close/protocol mismatch and doctor wording. Do not edit CHANGELOG.md.
