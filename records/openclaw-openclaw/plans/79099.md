# Coding Plan for openclaw/openclaw#79099: Windows gateway probe still reports unreachable while gateway health is OK on 2026.5.6

Source report: [records/openclaw-openclaw/items/79099.md](records/openclaw-openclaw/items/79099.md)

## Summary

Keep open. Current main still has source-visible paths where a WebSocket is accepted and then closed before `hello-ok`, but `gateway probe`, status output, and doctor still collapse that into unreachable or not running instead of reachable-but-errored diagnostics.

## Plan

Fix the Gateway probe/status/doctor diagnostic contradiction reported in https://github.com/openclaw/openclaw/issues/79099. Current main treats a WebSocket accepted by the Gateway but closed before `hello-ok` as unreachable because `probeGateway` only records `connectLatencyMs` after `hello-ok`; doctor also maps `gateway closed` to `Gateway not running.` even when the port scan proves a Gateway listener exists. Add a narrow accepted-but-errored connectivity state through `src/gateway/probe.ts`, `src/commands/gateway-status/helpers.ts`, `src/commands/gateway-status/output.ts`, and doctor rendering so true no-listener failures still say not running/unreachable, while protocol mismatch, auth failure, gateway-starting, and similar accepted-before-hello failures surface the specific reason as reachable-but-errored/degraded. Do not loosen auth, pairing, origin, TLS, or protocol validation. Add focused regression coverage for pre-hello close/protocol-mismatch classification, status text/JSON warning output, and doctor health/daemon wording. Update `docs/cli/gateway.md` only if the public output contract changes, and add a `CHANGELOG.md` Fixes entry because this is user-visible CLI diagnostics.
