## Summary

Keep open. Current main fixed known DeepSeek V4 proxy model IDs and already accepts `compat.thinkingFormat`, but the generic request wrapper still does not honor a model-level DeepSeek compat opt-in for arbitrary custom model IDs.

## Plan

In `src/agents/pi-embedded-runner/extra-params.ts`, update the generic post-plugin DeepSeek wrapper predicate so `openai-completions` models with `model.compat?.thinkingFormat === "deepseek"` get `createDeepSeekV4OpenAICompatibleThinkingWrapper` even when their model id does not end in `deepseek-v4-flash` or `deepseek-v4-pro`; keep the exact-id fallback and do not add a new config key. Add regression coverage for a custom provider/model id with `compat.thinkingFormat: "deepseek"`, thinking enabled, and assistant replay missing `reasoning_content`, proving the outgoing payload gets `thinking`, `reasoning_effort`, and blank assistant `reasoning_content`. Update user docs for the existing key and add a changelog entry if the behavior is user-visible. Do not import DeepSeek extension internals into core and do not broaden the wrapper to arbitrary models without the explicit compat opt-in.
