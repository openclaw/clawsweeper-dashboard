## Summary

Keep open: current main still has a source-reproducible Telegram streaming/TTS race where late TTS media turns a text-only final into a media final, causing the Telegram lane to clear the streamed preview and send a new voice/caption message instead of preserving the original text bubble.

## Plan

Fix https://github.com/openclaw/openclaw/issues/82570. In Telegram streaming delivery, when always-on/auto TTS adds a voice mediaUrl/audioAsVoice to a final payload after the answer text has already been streamed, preserve/finalize the existing text preview instead of clearing it and sending a new text+voice message. Send only the generated TTS audio as a follow-up voice message, keeping reply/thread/silent/account behavior correct. Keep the change scoped to late TTS audio on Telegram finalization; do not add a public messages.tts delivery config option and do not change normal photo/document/media finals. Add regression coverage for the exact path where a final payload has the same visible text plus audioAsVoice media while an answer stream already exists, and include live Telegram proof before merge.
