# Coding Plan for openclaw/openclaw#73619: Default `message` tool `accountId` to the calling agent's own Slack account when omitted

Source report: [records/openclaw-openclaw/items/73619.md](records/openclaw-openclaw/items/73619.md)

## Summary

Keep open. Current main still lets an omitted `message` tool `accountId` resolve from the inherited/default account before the calling-agent binding can run, and the focused closing PR was closed unmerged.

## Plan

Fix https://github.com/openclaw/openclaw/issues/73619 in a narrow replacement PR. Current main resolves omitted `message` tool account IDs from explicit params or inherited/default `defaultAccountId` before consulting the calling agent's route binding; this causes subagents and crons to post under the wrong Slack account. Preserve explicit `accountId` overrides. For omitted account IDs, prefer the calling agent's bound account for the target channel/peer via the existing route-binding resolver before falling back to inherited/default account context. Do not hard-code Slack or change broad account auto-selection policy tracked by other issues. Update `createMessageTool` so it does not write the inherited `agentAccountId` into `params.accountId` when the caller omitted it, and make account-scoped SecretRef resolution use the same selected bound account or defer scoping until selection is known. Add regression coverage for the combined `defaultAccountId` plus calling-agent binding case and for message-tool SecretRef scoping in that case. Add a user-facing `CHANGELOG.md` fix entry unless maintainers explicitly decide it is internal-only.
