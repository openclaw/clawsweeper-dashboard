# Coding Plan for openclaw/openclaw#53678: Feature request: Add Gateway received timestamp to inbound metadata for accurate latency tracking

Source report: [records/openclaw-openclaw/items/53678.md](records/openclaw-openclaw/items/53678.md)

## Summary

Keep open: current main has receive-context `receivedAt` groundwork for ack paths, but shared inbound metadata, hooks, direct-DM, and channel-turn surfaces still expose only the source timestamp, so the requested observability capability remains missing.

## Plan

Implement https://github.com/openclaw/openclaw/issues/53678 as an additive OpenClaw metadata feature. Add an optional Gateway/channel receipt timestamp, likely `ReceivedAt` on `MsgContext` and `receivedAt` in rendered/plugin-facing payloads, stamped once at ingress or carried from channel receive context. Do not repurpose `Timestamp`; it must remain the source-channel message time. Thread the value through `finalizeInboundContext`, inbound prompt metadata, hook event contracts, direct-DM dispatch, channel-turn context/types, and relevant Discord/Telegram inbound call sites. Document semantics for buffered/debounced messages and keep compatibility optional for third-party plugins. Add focused regression coverage and a `CHANGELOG.md` entry; do not add a config option or broader latency instrumentation in this PR.
