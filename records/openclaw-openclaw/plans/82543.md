## Summary

Keep open: current main and the latest release still have a plausible Zalo Personal startup-readiness gap after saved-session restore. The code starts the `zca-js` listener and handles later `error` or `closed` events, but it does not wait for or time out an initial `connected` event, so a listener that never opens or fails can leave the account task pending without a useful error.

## Plan

Fix the Zalo Personal startup hang reported in https://github.com/openclaw/openclaw/issues/82543. Add bounded listener readiness handling in the `zalouser` plugin around `zca-js` startup: subscribe to the upstream `connected`, `error`, and `closed` events, fail or reconnect with a clear error when startup never reaches `connected`, and keep the passive account monitor pending until abort only after readiness succeeds.
