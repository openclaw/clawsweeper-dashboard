# Coding Plan for openclaw/openclaw#39248: Bug: sandbox.mode: \"non-main\" silently breaks sessions_spawn subagent initialization

Source report: [records/openclaw-openclaw/items/39248.md](records/openclaw-openclaw/items/39248.md)

## Summary

Keep open. Current main still supports `agents.defaults.sandbox.mode: "non-main"`, still classifies generated native subagent sessions as sandboxed, and still dispatches the child run asynchronously after the accepted `sessions_spawn` response; the related Docker diagnostics PR was closed unmerged after being narrowed away from the `sessions_spawn` lifecycle path.

## Plan

Fix https://github.com/openclaw/openclaw/issues/39248. Current main supports `agents.defaults.sandbox.mode: "non-main"`, creates native child keys like `agent:<target>:subagent:<uuid>`, returns an accepted `sessions_spawn` response, then resolves Docker sandbox context in the asynchronous child run before prompt/transcript startup. Make sandbox startup failures for native child runs surface reliably: the child run/subagent session must transition to a terminal error or timeout state with actionable diagnostics instead of remaining running with zero messages/tokens, and gateway logs/session metadata should identify Docker create/start/setup failures. Preserve the documented non-blocking accepted response contract and do not disable `non-main` sandboxing. Treat https://github.com/openclaw/openclaw/issues/57992 as duplicate evidence that Docker create/start failures must be covered. Do not fold in the broader Docker env/setup inheritance policy from https://github.com/openclaw/openclaw/issues/69327 unless maintainers explicitly approve it. Add focused regression coverage around `sessions_spawn` lifecycle/error state and sandbox startup failure, plus Docker-backed proof or a forced Docker failure path. Changelog is expected if the user-visible diagnostic/session behavior changes.
