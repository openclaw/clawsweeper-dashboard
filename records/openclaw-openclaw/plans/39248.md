## Summary

Keep open. Current main still supports `non-main` sandboxing but the native `sessions_spawn` path can acknowledge a child before that child reaches Docker sandbox setup, and the only related diagnostics PR was closed unmerged as the wrong fix shape for this sessions_spawn bug.

## Plan

Fix https://github.com/openclaw/openclaw/issues/39248. Native `sessions_spawn` children that inherit `agents.defaults.sandbox.mode: "non-main"` can be accepted and registered while Docker sandbox setup fails or wedges before prompt/transcript progress, leaving the child shown as running with no messages or tokens. Keep `non-main` sandboxing supported and preserve the documented non-blocking accepted response. Do not disable sandboxing, change ACP runtime behavior, or add a new config option. Make sandbox startup/create/start/setupCommand failure for native child runs produce actionable gateway logging and a terminal child/subagent/session failure state instead of indefinite running. Add focused regression coverage with a forced sandbox startup failure and update any affected lifecycle tests. Include a changelog fix entry if the final PR is user-facing.
