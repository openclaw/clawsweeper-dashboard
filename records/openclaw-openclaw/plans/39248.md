# Coding Plan for openclaw/openclaw#39248: Bug: sandbox.mode: \"non-main\" silently breaks sessions_spawn subagent initialization

Source report: [records/openclaw-openclaw/items/39248.md](records/openclaw-openclaw/items/39248.md)

## Summary

Keep open. Current main still has the supported `non-main` sandboxed native subagent path, and the linked Docker-diagnostics PR was closed unmerged after being narrowed away from the `sessions_spawn` lifecycle failure this issue reports.

## Plan

Fix https://github.com/openclaw/openclaw/issues/39248. Current main supports `agents.defaults.sandbox.mode: "non-main"`; native `sessions_spawn` creates `agent:<target>:subagent:<uuid>`, classifies it as sandboxed, returns accepted before async child dispatch, and can leave a Docker-backed child run looking active with no transcript/messages/tokens when sandbox startup does not reach prompt execution. Implement a bounded lifecycle/error-handling repair so sandbox startup failure or timeout after the accepted ack transitions the child subagent/session/task to a terminal failed/timeout state with actionable diagnostics visible through subagent/session status and logs. Preserve the documented non-blocking `sessions_spawn` return shape and existing sandbox policy; do not disable `non-main`, add a new config option, or fold in the separate Docker env/setup inheritance policy from https://github.com/openclaw/openclaw/issues/69327. Add focused regression coverage for native `sessions_spawn` with `non-main` sandbox classification and forced Docker create/start/setupCommand failure or timeout. Add a changelog entry if the fix changes user-visible diagnostics or session state.
