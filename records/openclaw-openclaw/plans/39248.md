## Summary

Keep open. Current main still treats generated `sessions_spawn` child keys as sandboxed under `non-main`, returns the gateway `agent` accepted frame before the embedded runner reaches Docker sandbox setup, and the only related diagnostics PR was closed unmerged as the wrong fix shape for this sessions_spawn path.

## Plan

Fix the canonical sandboxed sessions_spawn startup bug tracked at https://github.com/openclaw/openclaw/issues/39248. Current main creates generated child session keys that are sandboxed under `agents.defaults.sandbox.mode: "non-main"`, returns the gateway `agent` accepted frame before `resolveSandboxContext()`/Docker startup, and can leave the subagent run showing as running with zero messages/tokens when startup fails before prompt progress. Keep the public `sessions_spawn` non-blocking contract, but ensure sandbox startup failures for native child runs are logged and reflected in terminal subagent/session/task state with actionable error text. Do not disable `non-main` sandboxing and do not move owner-specific behavior into core. Add focused regression coverage around the native child run lifecycle with a forced sandbox context/backend failure, plus Docker-backed proof for `non-main` sessions_spawn and a forced Docker create/start or setupCommand failure path. Add a changelog entry if the fix changes user-visible behavior.
