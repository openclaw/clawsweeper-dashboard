# Coding Plan for openclaw/openclaw#83511: TTS in final mode: visible text sent before audio causes Telegram delete-and-resend churn

Source report: [records/openclaw-openclaw/items/83511.md](records/openclaw-openclaw/items/83511.md)

## Summary

Current main and the latest release still settle visible ACP text before final-mode TTS is attempted, so Telegram can receive a text reply before the voice supplement. This should stay open as a focused Telegram/TTS delivery improvement; the related open PR deletes and resends after the fact, which does not fully address the no-churn goal.

## Plan

Fix the Telegram ACP final-mode TTS churn described in https://github.com/openclaw/openclaw/issues/83511. Implement a narrow captioned-voice capability or equivalent generic channel contract, advertise it for Telegram, and use it in `finalizeAcpTurnOutput()` so visible ACP text is not settled before a successful final-mode captioned voice TTS attempt. Preserve the duplicate-final guard from https://github.com/openclaw/openclaw/pull/55173 and the text fallback behavior from commit 7b8142997f79b9117b424ccb0aedf685d298dc5b: if TTS synthesis or media delivery fails, flush the original text exactly once. Do not solve the broader separate-delivery config request in https://github.com/openclaw/openclaw/issues/42539, and do not use the delete-and-resend approach from https://github.com/openclaw/openclaw/pull/81131 as the primary fix for this no-churn issue. Add focused regression coverage for success and fallback ordering, update docs/changelog only if the user-visible behavior contract changes, and keep non-Telegram or non-caption-capable channels on the existing path.
