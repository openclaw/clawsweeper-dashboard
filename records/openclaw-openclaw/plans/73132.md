# Coding Plan for openclaw/openclaw#73132: [Bug] openclaw gateway stop / launchctl bootout doesn't terminate underlying node process — wedged binary persists across stop

Source report: [records/openclaw-openclaw/items/73132.md](records/openclaw-openclaw/items/73132.md)

## Summary

Keep open. Current main still lacks a macOS LaunchAgent stop postcondition that verifies the actual OpenClaw gateway listener exited or that the configured port is free before reporting stop success; the adjacent bootout-by-default work in https://github.com/openclaw/openclaw/pull/79235 changed stop semantics but did not close this listener-cleanup gap.

## Plan

Fix https://github.com/openclaw/openclaw/issues/73132 on current main. macOS `openclaw gateway stop` must not report success while a verified OpenClaw gateway listener is still holding the configured port. Preserve the bootout-by-default and `--disable` behavior from https://github.com/openclaw/openclaw/pull/79235. Add a LaunchAgent stop postcondition after default bootout and `--disable` stop/bootout paths: resolve the effective gateway port from the LaunchAgent ProgramArguments/service env with an appropriate config fallback, wait briefly for verified OpenClaw listener PIDs to exit, send SIGTERM then SIGKILL only to verified gateway listener PIDs if needed, and return a non-zero error if the port remains busy or ownership cannot be safely attributed. Do not use broad `pgrep` or process-name kills, do not change Linux/systemd or Windows/schtasks behavior except for shared helper extraction, and add regression coverage for both default stop and `--disable` after a previous bootout. Changelog is expected because this changes user-visible gateway stop reliability.
