# Coding Plan for openclaw/openclaw#60799: cron: v2026.4.2 silently drops jobs.json (plain-array format) and first add clobbers all data

Source report: [records/openclaw-openclaw/items/60799.md](records/openclaw-openclaw/items/60799.md)

## Summary

Keep open. Current main and the latest release still load a valid top-level array cron store as an empty job list, and the first add path then persists only the newly-created in-memory job.

## Plan

Fix https://github.com/openclaw/openclaw/issues/60799 as a narrow cron-store compatibility bug. Current `loadCronStore()` and `loadCronStoreSync()` treat a valid top-level array `jobs.json` as empty, so the first `cron add` can overwrite legacy jobs. Add shared normalization in `src/cron/store.ts` that accepts both legacy `[{...}]` and current `{ "version": 1, "jobs": [...] }` shapes, keeps split-state behavior intact, and makes doctor/repair paths see legacy array jobs before returning empty. Add regression tests that load a top-level-array store and preserve existing jobs through a load-add-save round trip. Use https://github.com/openclaw/openclaw/pull/61113 as source context, but reimplement against current split-state `jobs-state.json` code. Do not broaden the PR into timeout defaults, auth cooldown clearing, or the separate partial in-memory overwrite race unless a preservation test proves the loader fix requires it. Add a changelog fix entry for user-visible data-loss prevention.
