# Coding Plan for openclaw/openclaw#41671: [Bug]: Telegram media download fails on IPv4-broken / IPv6-working hosts because runtime IPv4 fallback overrides config

Source report: [records/openclaw-openclaw/items/41671.md](records/openclaw-openclaw/items/41671.md)

## Summary

Keep open: current main and v2026.5.22 still build a forced IPv4-only Telegram fallback after broad fetch/network failures, even when explicit IPv6-friendly Telegram network config is recognized and passed into the transport.

## Plan

Fix https://github.com/openclaw/openclaw/issues/41671. Current main recognizes channels.telegram.network.autoSelectFamily/dnsResultOrder and passes it into resolveTelegramTransport, but retryable Telegram fetch/media errors can still promote direct or NO_PROXY-bypassed transports to forced IPv4-only dispatcher attempts. In extensions/telegram/src/fetch.ts, make sticky IPv4-only fallback respect explicit IPv6-friendly operator config, such as autoSelectFamily: true and/or dnsResultOrder: "verbatim", so runtime fallback does not override that config on IPv4-broken / IPv6-working hosts. Preserve default broken-IPv6 recovery, explicit IPv4 operator intent, explicit proxy behavior, pinned-IP SSRF checks, and the mirror IPv6-broken case in https://github.com/openclaw/openclaw/issues/75574. Add focused regression coverage in extensions/telegram/src/fetch.test.ts and the Telegram media retry surface proving explicit IPv6-friendly config does not create/promote IPv4-only fallback, while unconfigured/default and explicit IPv4 paths still do. Update CHANGELOG.md because Telegram media/API delivery behavior changes. Stop and ask for maintainer input if the fix would require a new public config option or broad transport rewrite.
