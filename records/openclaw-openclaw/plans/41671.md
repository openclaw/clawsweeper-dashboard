# Coding Plan for openclaw/openclaw#41671: [Bug]: Telegram media download fails on IPv4-broken / IPv6-working hosts because runtime IPv4 fallback overrides config

Source report: [records/openclaw-openclaw/items/41671.md](records/openclaw-openclaw/items/41671.md)

## Summary

Keep open: current main and the latest shipped tag still build Telegram fallback attempts that can promote retryable API and media fetches to a forced IPv4-only dispatcher even when the operator configured IPv6-friendly Telegram network settings.

## Plan

Fix https://github.com/openclaw/openclaw/issues/41671. In the Telegram plugin transport, make runtime fallback respect explicit IPv6-friendly `channels.telegram.network` settings so retryable API/media fetch failures do not promote to forced IPv4-only dispatchers when the operator configured `autoSelectFamily: true` and/or `dnsResultOrder: "verbatim"`. Keep default IPv4 fallback behavior for unconfigured IPv6-broken hosts and preserve explicit IPv4 operator intent such as `autoSelectFamily: false` or `dnsResultOrder: "ipv4first"`; do not add a new config option unless the existing policy cannot express the fix. Cover both Bot API fetch and inbound media download paths, including dispatcherAttempts through SSRF-guarded media fetches. Add focused regression tests in the Telegram fetch/media tests and update CHANGELOG.md with a user-facing fix entry. Do not weaken SSRF policy, proxy safety, or unrelated Telegram delivery behavior.
