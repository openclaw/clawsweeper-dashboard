## Summary

Keep open: current main and the latest release still build an IPv4-forced Telegram fallback after retryable fetch errors even when explicit IPv6-friendly Telegram network config is recognized and passed into the transport. The related duplicate at https://github.com/openclaw/openclaw/issues/55099 was closed in favor of this issue, while https://github.com/openclaw/openclaw/issues/75574 tracks the mirror IPv6-broken / IPv4-working case that the fix must preserve.

## Plan

Fix the Telegram transport fallback bug reported at https://github.com/openclaw/openclaw/issues/41671. In current main, explicit IPv6-friendly config such as channels.telegram.network.autoSelectFamily=true and dnsResultOrder=verbatim shapes the primary dispatcher, but retryable fetch errors can still promote the transport to an IPv4-only fallback. Change the Telegram fallback policy in extensions/telegram/src/fetch.ts so explicit IPv6-friendly operator config prevents forced IPv4-only fallback, while preserving default IPv4 fallback for unconfigured broken-IPv6 hosts and preserving explicit IPv4 operator intent tracked at https://github.com/openclaw/openclaw/issues/75574. Add focused regression coverage in extensions/telegram/src/fetch.test.ts and related network tests; do not add a new config option unless the existing contract cannot express the fix. Add a user-facing CHANGELOG.md fix entry unless maintainers decide this is release-internal.
