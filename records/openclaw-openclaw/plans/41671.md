## Summary

Keep open: current main and the latest release still recognize explicit Telegram network config for the primary transport but still build and promote a forced IPv4-only fallback path after broad retryable network errors. The closed batch PR that claimed this fix was not merged, and the newer open Telegram network issue covers the opposite IPv6-broken shape rather than superseding this report.

## Plan

Fix the Telegram network fallback bug reported at https://github.com/openclaw/openclaw/issues/41671: when `channels.telegram.network` is explicitly configured for IPv6-friendly behavior, retry/fallback handling must not force `autoSelectFamily: false`, `dnsResultOrder: "ipv4first"`, or a sticky IPv4-only dispatcher after generic fetch/network errors. Keep the existing default recovery for IPv6-broken/IPv4-working hosts and explicit IPv4 configuration. Focus on `extensions/telegram/src/fetch.ts`, `extensions/telegram/src/network-config.ts` if needed, and regression tests in `extensions/telegram/src/fetch.test.ts`; only touch media/SSRF files if the source review proves the media path needs the same explicit-config guard. Add tests for explicit `dnsResultOrder: "verbatim"` and/or `autoSelectFamily: true` preventing forced IPv4 fallback, plus tests that default fallback and explicit IPv4 settings still work. Do not add a new config option unless the existing policy cannot be repaired. Changelog is expected only if maintainers decide this user-facing fix should be credited during landing.
