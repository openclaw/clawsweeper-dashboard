# Coding Plan for openclaw/openclaw#41671: [Bug]: Telegram media download fails on IPv4-broken / IPv6-working hosts because runtime IPv4 fallback overrides config

## Summary

Keep open: current main and the latest release still honor explicit Telegram network config only for the primary dispatcher, then build an IPv4-only sticky fallback that can override the IPv6-capable path described here. The closed sibling at https://github.com/openclaw/openclaw/issues/55099 also points back to this issue as the canonical remaining fallback-policy fix.

## Plan

Fix https://github.com/openclaw/openclaw/issues/41671. Current main honors `channels.telegram.network` for the primary Telegram dispatcher, but `resolveTelegramTransport` still exports/promotes fallback attempts with `autoSelectFamily: false`, `dnsResultOrder: "ipv4first"`, and `forceIpv4: true`; Telegram media downloads pass those attempts into `saveRemoteMedia`. Update the Telegram transport policy so explicit IPv6-capable config, especially `dnsResultOrder: "verbatim"` and/or `autoSelectFamily: true`, does not promote API or media requests to IPv4-only fallback after generic fetch/network errors. Preserve default IPv4 fallback for no explicit config and IPv6-broken hosts, preserve explicit/env proxy behavior, preserve SSRF guarding and dispatcher close lifecycle, and do not add a new config/env option or edit CHANGELOG.md. Add focused regression coverage showing the explicit IPv6-friendly config path does not use IPv4-only fallback for Bot API/media attempts, while the default/no-config broken-IPv6 recovery path still falls back as before.
