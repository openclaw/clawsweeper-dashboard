Keep open: current main and the latest release still build forced IPv4-only Telegram fallback attempts even when explicit IPv6-friendly Telegram network config is recognized and passed into the transport, so the reported media/API failure remains source-reproducible.

In current main, explicit IPv6-friendly `channels.telegram.network` config is recognized and passed into `resolveTelegramTransport`, but retryable fetch/media errors can still promote to fallback dispatcher attempts that force `autoSelectFamily: false`, `dnsResultOrder: "ipv4first"`, and `forceIpv4: true`. Keep the fix inside the Telegram transport/media fallback boundary, likely `extensions/telegram/src/fetch.ts` plus focused tests. Do not add a new config knob first. Ensure explicit IPv6-friendly operator intent, especially `autoSelectFamily: true` and/or `dnsResultOrder: "verbatim"`, prevents forced IPv4-only fallback from API and media paths.
