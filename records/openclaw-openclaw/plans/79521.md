# Coding Plan for openclaw/openclaw#79521: [Bug]: voice-call waits on post-turn compaction before speaking response, causing 20s+ latency

Source report: [records/openclaw-openclaw/items/79521.md](records/openclaw-openclaw/items/79521.md)

## Summary

Keep open. Current main still makes the classic voice-call responder wait for a completed embedded-agent result, while the embedded runner waits for compaction/turn maintenance before forming returned payloads; the existing pre-compaction reply flush only helps callers that provide reply callbacks, and voice-call does not.

## Plan

Fix https://github.com/openclaw/openclaw/issues/79521 as a narrow classic voice-call latency bug. Current main has `extensions/voice-call/src/response-generator.ts` await `runEmbeddedAgent()` and then extract completed payloads; `src/agents/embedded-agent-runner/run/attempt.ts` flushes block replies before compaction only for callers that provide callbacks, then waits for compaction/maintenance before returning. Make the voice-call path hand the completed `{spoken}` text to TTS as soon as the assistant response is available, before nonessential post-turn compaction or turn maintenance completes. Prefer reusing the existing embedded-runner reply callback/flush seam or adding the smallest typed internal seam needed; keep post-turn cleanup running safely, preserve `toolsAllow`, session identity, timeout, and spoken-output filtering. Do not implement the broader sentence-level LLM-to-TTS streaming feature from https://github.com/openclaw/openclaw/issues/8355, do not add config/env flags, and do not change the shared speech-provider API unless the existing seam is insufficient and the PR explains why. Add focused tests that simulate a delayed compaction/maintenance wait and assert `generateVoiceResponse()` can resolve the spoken text before that delay, plus coverage that no early text is spoken for errors/reasoning/empty spoken payloads. No CHANGELOG.md edit.
