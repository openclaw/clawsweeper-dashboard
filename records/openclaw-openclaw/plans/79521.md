## Summary

Keep open: current main still routes classic voice-call replies through a fully awaited embedded run, so post-turn compaction can delay TTS even after the related `tools.allow` fix is present.

## Plan

Fix the voice-call compaction latency bug from https://github.com/openclaw/openclaw/issues/79521. Current main awaits `generateVoiceResponse()` in `extensions/voice-call/src/webhook.ts` before `manager.speak()`, and `generateVoiceResponse()` awaits the full `runEmbeddedPiAgent()` promise in `extensions/voice-call/src/response-generator.ts`; the embedded runner already has `onBlockReply`/`onBlockReplyFlush` support before compaction retry waits. Implement a low-latency voice handoff so the parsed spoken response is delivered to TTS as soon as the assistant payload is available and before post-turn compaction waits, while avoiding duplicate speech, keeping session persistence/trajectory behavior safe, preserving the routed `tools.allow` behavior, and not changing non-voice reply semantics. Do not add a new config option, disable compaction globally, or broaden plugin/core boundaries. Add focused regression coverage for the early handoff and any webhook callback plumbing.
