# Coding Plan for openclaw/openclaw#41750: Browser tool: zombie Chrome process blocks CDP port, causes gateway timeouts

## Summary

Keep open: current main and the latest shipped release still have the stale managed-Chrome/CDP-port failure path, and the only direct fix PR was closed unmerged after concrete safety review.

## Plan

Fix https://github.com/openclaw/openclaw/issues/41750 in the browser plugin. Current local managed OpenClaw profiles can get stuck when a stale/unresponsive Chrome process still holds the CDP port: `launchOpenClawChrome` checks `ensurePortAvailable(profile.cdpPort)` before any stale owned-process recovery, and `ensureBrowserAvailable` refuses untracked unhealthy local ports. Implement a narrow recovery path only for local managed OpenClaw profiles (`cdpIsLoopback`, not `attachOnly`, not remote). Verify OpenClaw ownership before terminating anything, using the managed user data dir, launch args, tracked process state, or another robust owner signal; do not kill by port alone. Avoid shell interpolation and avoid the unsafe patterns from https://github.com/openclaw/openclaw/pull/41758 (`fuser -k`, ps/grep shell pipelines, obsolete `src/browser/chrome.ts`). Terminate gracefully, escalate only when needed, wait for the port/CDP state to clear, relaunch, and preserve existing remote/attachOnly external-loopback errors. Add focused regression coverage for an unresponsive/busy managed CDP port and for preserving external loopback/attachOnly behavior. Add a `CHANGELOG.md` Fixes entry if this becomes the landing PR.
