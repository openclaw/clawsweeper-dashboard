# Coding Plan for openclaw/openclaw#70986: memory-wiki lint: wikilink extractor is not markdown-aware; resolver never matches frontmatter titles

## Summary

Keep open: current `main` and the latest release still expose both central Memory Wiki lint problems. The `.md` relative-target expansion is present, but the extractor still scans raw Markdown and the broken-link resolver still indexes only path-shaped targets, while title/normalization work remains in open resolver PRs.

## Plan

Fix the Memory Wiki wikilink extractor false positives for https://github.com/openclaw/openclaw/issues/70986 without implementing the resolver/title-normalization half in this job. In `extensions/memory-wiki/src/markdown.ts`, make `extractWikiLinks` skip wikilink and markdown-link patterns inside Markdown non-link regions: YAML frontmatter, fenced code blocks using backticks or tildes, inline code spans where feasible without breaking valid markdown link text, and HTML comments. Preserve real Obsidian `[[target|alias]]` and native `[title](relative.md)` links outside ignored regions. Add focused regression tests in `extensions/memory-wiki/src/markdown.test.ts` and, if needed, `extensions/memory-wiki/src/lint.test.ts` covering fenced bash `[[ ... ]]`, inline-code `[[reply_to_current]]`, HTML comments, frontmatter, and a real body wikilink that still extracts. Do not add tag-specific allowlists for `reply_to_*`, and do not add title/alias/fragment resolver normalization because that work is already represented by https://github.com/openclaw/openclaw/pull/73722, https://github.com/openclaw/openclaw/pull/80915, and https://github.com/openclaw/openclaw/pull/78556. Add a single user-facing changelog entry if this becomes a repair PR.
