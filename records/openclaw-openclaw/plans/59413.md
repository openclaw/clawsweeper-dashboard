# Coding Plan for openclaw/openclaw#59413: Feature Request: Per-candidate retry count for model fallback (support pool-based/proxy providers)

Source report: [records/openclaw-openclaw/items/59413.md](records/openclaw-openclaw/items/59413.md)

## Summary

Keep open. Current main and the latest shipped release still provide auth-profile rotation and cooldown controls, but not a retry-before-fallback budget scoped to a fallback candidate or provider/model route; the related retry PR at https://github.com/openclaw/openclaw/pull/55118 was closed unmerged and the discussion identifies this remaining gap.

## Plan

Implement the feature requested in https://github.com/openclaw/openclaw/issues/59413: add a backward-compatible retry-before-fallback budget scoped to a provider/model route or fallback candidate, following the provider/model-level direction discussed in the issue. Preserve current default behavior as one attempt per candidate unless configured. Apply retries before marking the candidate failed and before advancing to the next model, and keep permanent auth, billing, context-overflow, and explicit abort paths fail-fast. Update exported config types, zod schema/help/labels/generated schema as required by the repo, docs for model failover and gateway config, and add a user-facing changelog entry. Add regression tests proving timeout/502 retries can succeed on the same candidate, retry exhaustion advances to the next fallback, and unrelated auth-profile rotation controls still behave as before. Do not change provider defaults, bundled provider policy, or global fallback ordering beyond the new opt-in retry budget.
