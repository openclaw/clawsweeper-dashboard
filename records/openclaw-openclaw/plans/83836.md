# Coding Plan for openclaw/openclaw#83836: Codex replay can silently fail Telegram turns with invalid_encrypted_content from mirrored thinking blocks

Source report: [records/openclaw-openclaw/items/83836.md](records/openclaw-openclaw/items/83836.md)

## Summary

Keep open: current main still has a source-level path that can replay persisted OpenAI/Codex Responses `thinkingSignature.encrypted_content` into a later request, matching the reported `invalid_encrypted_content` and channel non-delivery symptoms.

## Plan

Implement a narrow fix for https://github.com/openclaw/openclaw/issues/83836. Current main can replay persisted OpenAI/Codex Responses `thinkingSignature` JSON containing stale `encrypted_content`, which can trigger `400 invalid_encrypted_content` and a non-deliverable Telegram turn. Fix the shared Codex/OpenAI replay path, preferably at request assembly or replay sanitization, so `openai-codex` does not resend stale encrypted reasoning payloads while preserving the reasoning item shape, IDs, and tool-call pairing needed for valid replay. Add a one-time sanitized retry for provider errors classified as `invalid_encrypted_content` if the first request used replayed encrypted reasoning. Ensure the final no-reply path surfaces a non-sensitive channel-visible failure if all attempts fail. Do not implement the broader hidden/silent-reasoning feature from https://github.com/openclaw/openclaw/issues/54496, and do not change GitHub Copilot behavior already tracked by https://github.com/openclaw/openclaw/issues/78867 unless a focused test proves the same shared helper should cover both. Add regression tests for native Codex replay with `encrypted_content`, paired reasoning plus tool calls, and the sanitized retry/error path; add a user-facing changelog entry because this affects channel delivery reliability.
