## Summary

Keep open: current main still has a source-level path that can replay same-model OpenAI/Codex Responses `thinkingSignature.encrypted_content` into later requests, matching the reported `invalid_encrypted_content` Telegram failures and lost normal replies.

## Plan

Fix https://github.com/openclaw/openclaw/issues/83836 by sanitizing stale OpenAI/Codex Responses encrypted reasoning replay below the channel layer. Current main preserves paired `thinkingSignature` JSON and `openai-transport-stream.ts` re-emits `encrypted_content`; implement a provider-aware fix that strips or removes stale `encrypted_content` for `openai-codex` replay while preserving valid same-turn reasoning/tool-call structure. Add a recover-once path for `invalid_encrypted_content` or equivalent invalid encrypted reasoning errors if the initial request fails, and make sure the user gets a non-sensitive channel-visible error if no reply can be delivered. Do not implement the broader silent-reasoning persistence feature from https://github.com/openclaw/openclaw/issues/54496, and do not make the Copilot-only PR https://github.com/openclaw/openclaw/pull/79176 the generic Codex fix.
