## Summary

Keep open: current main still routes Control UI typed `/new` through `sessions.create` without any `command:new` hook emission, and the focused fix PR #77376 is closed unmerged rather than shipped.

## Plan

Fix the Control UI typed `/new` regression from this issue. Current main routes `/new` through `host.onSlashAction("new-session")` -> `createSessionAndRefresh` -> `sessions.create`, which does not emit the documented `command:new` hook path and skips `session-memory`. Implement a narrow equivalent of #77376: add an explicit `emitCommandHooks?: boolean` opt-in to `SessionsCreateParams`, have the Control UI `/new` helper pass it only when creating from an existing parent session, and have `sessions.create` emit the `command:new`, `before_reset`, and session lifecycle hooks against the parent only when that flag is true. Do not use the older #77004 approach of firing hooks for every `parentSessionKey`; programmatic SDK child-session creates must remain hook-free by default. Keep `/reset` behavior unchanged. Update protocol schema/generated mirrors as required, add regression coverage for both `emitCommandHooks: true` and parent-without-flag, update the Control UI helper test expectation, and add a user-facing changelog fix entry crediting the human contributor if applicable.
