# Coding Plan for openclaw/openclaw#66631: [Bug] Feishu: Bot replies create new topics instead of replying within existing topic

Source report: [records/openclaw-openclaw/items/66631.md](records/openclaw-openclaw/items/66631.md)

## Summary

Keep open: current main still routes Feishu topic-group events that have `thread_id` but no `root_id` as topic sessions, then falls back to the triggering `message_id` for reply dispatch. The related open Feishu PRs cover adjacent reply-anchor problems, but I did not find an exact merged or open fix for resolving an `omt_*` topic container to its root `om_*` message before bot replies.

## Plan

Fix Feishu topic-group replies for events that include `thread_id` (`omt_*`) but omit `root_id`, `parent_id`, and `reply_target_message_id`. Resolve the topic container to the root `om_*` message after group/sender admission and before ACP failure replies, normal dispatcher creation, and broadcast dispatcher creation. Keep the change plugin-local under `extensions/feishu`, reuse or extend the existing Feishu message/thread lookup seam, and consider a small cache if repeated lookups are needed. Do not change normal `groupSessionScope: "group"` `root_id` behavior, P2P thread routing, `replyInThread: "disabled"` semantics, or unrelated topic-anchor behavior from the adjacent open PRs. Add focused regressions for topic-group `thread_id`-only events proving reply target/root metadata becomes the resolved `om_*` root for ACP failure and normal replies; update docs/changelog only if the behavior wording changes.
