## Summary

Keep open: current main still treats `thread_id`-only Feishu topic-group events as topic sessions, but reply dispatch falls back to `ctx.messageId` when `root_id` and `reply_target_message_id` are missing. I did not find a merged or linked PR that resolves the required `omt_*` topic container to the `om_*` root message before reply dispatch.

## Plan

Fix https://github.com/openclaw/openclaw/issues/66631 on current main. In the Feishu plugin, when an inbound group topic session or explicit `replyInThread` flow has a Feishu `threadId`/`omt_*` but no `rootId`, `parentId`, or `replyTargetMessageId`, resolve the topic root `om_*` before computing reply targets for ACP init failure replies, normal replies, and broadcast replies. Prefer a plugin-local helper using the existing Feishu message/thread API seam; cache or reuse results if that fits the local pattern, but preserve current fallback/logging if the lookup fails. Add focused regression coverage for a `topic_group` `thread_id`-only event proving `replyToMessageId`/`rootId` use the resolved root, and keep existing normal-group `root_id` behavior from https://github.com/openclaw/openclaw/issues/32980 unchanged.
