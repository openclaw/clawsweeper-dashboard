## Summary

Keep open: current main still routes Feishu topic-group events that have `thread_id` but no `root_id` as topic sessions, yet reply dispatch falls back to the triggering `message_id`, so the reported new-topic behavior remains source-reproducible.

## Plan

Fix the Feishu topic reply bug in https://github.com/openclaw/openclaw/issues/66631. Current main handles `thread_id`-only topic events as topic sessions but replies to `ctx.messageId`; resolve the `omt_*` topic thread to the root `om_*` message after disabled-group, group allowlist, and sender allowlist admission, then use that root for ACP failure replies, normal replies, and broadcast active-agent replies. Reuse or extend the existing Feishu message/thread helpers in `extensions/feishu/src/send.ts`; do not add a config option, do not move logic into core, and do not change normal-group `root_id` precedence or OpenClaw-created normal reply-thread session keys. Add regression tests for `group_topic`, `group_topic_sender`, and configured `replyInThread` when `thread_id` is present and `root_id`/`parent_id` are absent, plus a negative test that non-admitted events do not make the authenticated lookup. Add a user-facing `CHANGELOG.md` fix entry if the final patch changes runtime behavior.
