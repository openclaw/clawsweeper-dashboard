## Summary

Keep open: the bug and Telegram proof are strong, but the patch is not merge-ready because the serializer is acquired after durable queue work, uses a raw non-canonical destination key, and can turn a stalled platform send into a same-recipient delivery wedge.

## Plan

Repair PR https://github.com/openclaw/openclaw/pull/44143 without changing unrelated behavior. Preserve the useful goal of per-destination outbound FIFO delivery, but move lane acquisition before `enqueueDelivery(...)` and `withActiveDeliveryClaim(...)` so queue I/O cannot invert same-recipient send order. Replace the raw serializer key with a canonical destination identity that accounts for normalized account IDs, normalized recipient targets, and thread/topic routing; do not make adapters infer ordering from raw strings. Add abort/timeout handling so a never-settling platform send cannot hold a lane forever. Add focused regression tests for queue-write inversion, canonical key equivalence/distinct threaded destinations, and stalled-send behavior. Do not add config flags or changelog edits.
