## Summary

Keep open. Current main still does not synthesize a WSL2 `DISPLAY=:0` fallback for visible managed Chrome launches; explicit headed Linux starts with no inherited display still fail before spawn, and all target-repo PRs for this exact behavior are closed unmerged.

## Plan

Fix the WSL2 headed managed Chrome launch bug from https://github.com/openclaw/openclaw/issues/64464. In the browser plugin only, add a production-path spawn-env helper or equivalent so local managed Chrome launches set `DISPLAY=:0` only when all of these are true: platform is Linux/WSL2, the effective launch mode is headed, neither `DISPLAY` nor `WAYLAND_DISPLAY` is inherited, and the profile is local managed. Use the public `openclaw/plugin-sdk/runtime-env` WSL helper; do not import core internals, do not affect headless, remote CDP, attach-only, existing-session, or user-provided display values, and wire through the current `prepareOomScoreAdjustedSpawn(exe.path, args, { env })` path. Add regression coverage for the env passed to the real spawn path or a narrow exported helper plus launch integration, including preserving existing `DISPLAY`, empty/missing display behavior, headless no-op, non-WSL Linux no-op if that is the chosen policy, and current missing-display error behavior. Update browser Linux troubleshooting/browser docs only if behavior changes, and add a maintainer-owned changelog entry for the user-visible browser fix.
