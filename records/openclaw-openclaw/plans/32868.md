Keep open: current main and the latest shipped release still have the same same-channel block-reply delivery gap. The discussion has independent Telegram, Discord, and WeCom confirmations plus a community plugin workaround, but that workaround does not fix the core dispatcher barrier.

## Plan

Fix the same-channel block streaming delivery gap reported in https://github.com/openclaw/openclaw/issues/32868. Keep the change narrow: make accepted same-channel `dispatcher.sendBlockReply` delivery awaitable before `handleToolExecutionStart` continues, while preserving boolean false/drop semantics, abort behavior, channel ordering, and existing routed/cross-channel behavior. Do not implement a broad followup-idle lifecycle wait like https://github.com/openclaw/openclaw/pull/52026. Add focused regression coverage that proves a `text_end` block emitted before a tool waits for the real dispatcher delivery promise, not only the pipeline callback. Update `CHANGELOG.md` with a concise user-facing fix entry if this lands as a fix PR.
