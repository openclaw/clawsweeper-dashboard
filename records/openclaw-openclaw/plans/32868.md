## Summary

Keep open: current main still queues same-channel block replies through a non-awaitable dispatcher path, so the intended pre-tool `text_end` flush does not prove channel delivery completed. The discussion has independent WhatsApp, Telegram, Discord, and WeCom reports, plus an open but broader followup-idle PR that does not supersede this narrower bug.

## Plan

Fix https://github.com/openclaw/openclaw/issues/32868 on current main. The observable bug is that with `blockStreamingBreak: "text_end"`, same-channel block replies are queued through `dispatcher.sendBlockReply` and tool execution can continue before the channel delivery drains. Keep the fix narrow: make accepted same-channel block delivery awaitable or add a tightly scoped dispatcher barrier in the reply path; preserve dropped-payload behavior, abort/cancellation semantics, and cross-channel routing. Do not implement the broader followup-idle lifecycle approach from https://github.com/openclaw/openclaw/pull/52026.
