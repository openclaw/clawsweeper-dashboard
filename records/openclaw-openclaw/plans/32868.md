## Summary

Keep open. Current main still has the same-channel block-streaming await gap: `text_end` flushes wait for the block pipeline callback, but the same-channel callback queues dispatcher delivery and returns before channel delivery settles.

## Plan

Fix the same-channel block-streaming delivery ordering bug tracked at https://github.com/openclaw/openclaw/issues/32868. Current main queues same-channel `dispatcher.sendBlockReply(normalizedPayload)` from `dispatch-from-config.ts` without awaiting actual dispatcher delivery, so `onBlockReplyFlush` can resolve before channel delivery and tool execution continues too early. Implement a narrow compatible fix in the reply dispatcher/pipeline path: make accepted same-channel block delivery awaitable at the existing flush boundary or wait for dispatcher idle after a successful block enqueue. Preserve cross-channel `sendPayloadAsync` behavior, dropped/silent payload semantics, ACP direct delivery accounting, human-delay ordering, and plugin SDK compatibility for boolean-style dispatcher callers. Add focused regression coverage proving a same-channel `text_end` block is held until dispatcher delivery settles before subsequent tool execution or final/tool output can proceed. Add a user-facing changelog entry for the bug fix. Do not change unrelated streaming behavior, channel-specific plugins, provider routing, or broad registry/runtime loading.
