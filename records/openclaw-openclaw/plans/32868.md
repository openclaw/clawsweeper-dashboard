## Summary

Keep open: current main still has the same same-channel block-streaming delivery gap, with source evidence matching the reporter's root cause and multiple channel confirmations in the discussion.

## Plan

Fix https://github.com/openclaw/openclaw/issues/32868 on current main. Same-channel block streaming with `blockStreamingBreak: "text_end"` currently flushes the block pipeline before tool execution, but `dispatch-from-config.ts` only calls `dispatcher.sendBlockReply()` and returns while `ReplyDispatcher` delivers later on its internal `sendChain`. Add a narrow, abort-aware delivery barrier so the same-channel block path waits for the relevant dispatcher block delivery before tool execution continues. Prefer returning an awaitable result for the queued block or using a tightly scoped dispatcher idle wait; preserve boolean compatibility for skipped/legacy dispatchers, keep the existing cross-channel `sendPayloadAsync` path unchanged, and do not implement the broader followup-idle lifecycle approach from https://github.com/openclaw/openclaw/pull/52026. Add regression coverage that fails on current main by showing a `text_end` block flush does not proceed to tool start until dispatcher delivery drains, update affected dispatcher types/mocks, and add a user-facing changelog fix entry if this lands as a ClawSweeper fix PR. If feasible before landing, capture one redacted same-channel proof on WhatsApp, Telegram, Discord, or WeCom showing a block arrives before a following long tool call completes.
