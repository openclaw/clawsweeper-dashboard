# Coding Plan for openclaw/openclaw#32868: Block streaming: block replies not delivered before tool execution (same-channel)

Source report: [records/openclaw-openclaw/items/32868.md](records/openclaw-openclaw/items/32868.md)

## Summary

Keep open: current main still has the same same-channel block streaming delivery gap, and the canonical duplicate has already been closed back to this issue. The source shows block flushes are awaited before tool-start processing, but the same-channel callback only queues dispatcher delivery and returns before the channel send drains.

## Plan

Fix the same-channel block streaming delivery race tracked by https://github.com/openclaw/openclaw/issues/32868. Keep the change narrowly scoped to the reply dispatcher/block-reply path: when `blockStreamingBreak: "text_end"` flushes before tool execution, same-channel `onBlockReply` must wait until the queued dispatcher delivery for that block has drained before tool execution continues. Prefer returning an awaitable delivery result from `sendBlockReply()` while preserving existing boolean-style compatibility for callers/plugins, or an equivalently tight per-dispatcher barrier; do not implement a broad followup-runner wait like https://github.com/openclaw/openclaw/pull/52026. Cover abort/timeout cleanup, timer/listener cleanup, dropped-payload `false` behavior, and existing fire-and-forget callers such as ACP delivery. Add/update regression tests proving the same-channel path waits for dispatcher delivery before the next tool-start continuation, and keep cross-channel routed delivery behavior unchanged. Add a changelog entry because this is user-visible channel behavior.
