# Coding Plan for openclaw/openclaw#75657: fix: local GGUF embedding model warmup blocks Node.js event loop for minutes on startup (ARM64/Pi)

Source report: [records/openclaw-openclaw/items/75657.md](records/openclaw-openclaw/items/75657.md)

## Summary

Keep open. Current main still runs explicit local GGUF embedding initialization inside the Gateway Node.js process; startup deferral and long Promise.race timeouts do not isolate node-llama-cpp native work from the event loop once local embeddings are initialized or probed.

## Plan

Fix https://github.com/openclaw/openclaw/issues/75657 by isolating explicit local GGUF memory embedding initialization from the Gateway Node.js event loop. Keep the existing `memorySearch.provider: "local"` contract and defaults; do not add a config flag or change provider selection semantics. Move node-llama-cpp import, model resolution/download, getLlama, loadModel, createEmbeddingContext, and getEmbeddingFor calls behind a worker-thread or child-process boundary that works in packaged Node 22/24 and preserves Bun/source runtime behavior as far as current code supports it. Coordinate with https://github.com/openclaw/openclaw/pull/70596 so startup/prewarm paths do not move blocking native work earlier without isolation. Add focused regression coverage proving Gateway startup/channel start can proceed while local embedding init is slow, deep status/sync/first search call the isolated provider path, and failures/timeouts still surface as current memory embedding errors. Update CHANGELOG.md because this is user-visible runtime reliability. Stop and ask for maintainer guidance if the implementation requires a new public config option, dependency pin/override, or plugin API contract change.
