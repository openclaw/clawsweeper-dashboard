# Coding Plan for openclaw/openclaw#75657: fix: local GGUF embedding model warmup blocks Node.js event loop for minutes on startup (ARM64/Pi)

Source report: [records/openclaw-openclaw/items/75657.md](records/openclaw-openclaw/items/75657.md)

## Summary

Keep open. Current main still initializes the in-process `node-llama-cpp` local embedding provider inside the Gateway process; the startup deferral that exists today reduces some eager paths, but it does not isolate GGUF model resolution, model load, context creation, or embedding calls from the Node.js event loop. The related prewarm PR at https://github.com/openclaw/openclaw/pull/70596 overlaps the startup files but moves local probing earlier rather than adding the needed worker or child-process boundary.

## Plan

Fix the local GGUF memory embedding provider so `node-llama-cpp` model resolution, model loading, embedding context creation, and embedding calls cannot block the Gateway's main Node.js event loop. Preserve the existing `memorySearch.provider: "local"` user contract and avoid adding a new config option. Prefer a worker-thread or child-process boundary owned by the memory host SDK/provider layer; keep core plugin boundaries intact and do not move plugin-owned runtime policy into Gateway core. Coordinate with https://github.com/openclaw/openclaw/pull/70596: do not solve this by only prewarming builtin local embeddings at startup, because that can move blocking native work earlier without isolation. Add focused regression coverage around local provider initialization/call isolation, timeout behavior, deep status/probe, sync/first search, and startup scheduling. Include a changelog entry if user-visible behavior changes. Stop and ask for maintainer direction if the implementation would require a new user-facing config knob, a release packaging change beyond optional runtime support, or a broad memory provider API redesign.
