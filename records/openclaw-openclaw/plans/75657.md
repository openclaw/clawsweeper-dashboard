# Coding Plan for openclaw/openclaw#75657: fix: local GGUF embedding model warmup blocks Node.js event loop for minutes on startup (ARM64/Pi)

Source report: [records/openclaw-openclaw/items/75657.md](records/openclaw-openclaw/items/75657.md)

## Summary

Keep open. Current main avoids builtin/local startup work by default, but the local GGUF provider still resolves and loads `node-llama-cpp` in the Gateway process whenever local embeddings are initialized, so the core event-loop blocking risk remains. The related open PRs cover prewarm or diagnostic-budget changes, not the needed worker or child-process isolation.

## Plan

Fix https://github.com/openclaw/openclaw/issues/75657 by isolating the builtin `local` GGUF embedding provider from the Gateway event loop. Keep the existing memory provider contract and config surface; do not add a new user-facing config option. Move `node-llama-cpp` model resolution, model/context creation, and embedding calls behind a worker-thread or child-process boundary that works for packaged Node and Bun paths with the optional runtime dependency. Add focused regression coverage proving local provider initialization/embedding calls go through the isolated boundary and that startup/deep-status/sync paths do not rely on an in-process GGUF load. Coordinate with https://github.com/openclaw/openclaw/pull/70596 by avoiding any builtin/local startup prewarm until isolation is in place. Add a `CHANGELOG.md` fix entry unless maintainers decide the final patch is purely internal/test-only.
