# Coding Plan for openclaw/openclaw#85363: iMessage catch-up can replay live-handled messages after restart when cursor lags

Source report: [records/openclaw-openclaw/items/85363.md](records/openclaw-openclaw/items/85363.md)

## Summary

Keep open: current main still has the reported restart-boundary gap for opt-in iMessage catch-up. Catch-up persists its cursor after replay passes, but live `watch.subscribe` traffic still goes through the normal dispatch path and only relies on process-local inbound dedupe, so a gateway restart can forget already-handled live rows while the durable catch-up cursor lags.

## Plan

Fix the iMessage catch-up restart replay bug reported in https://github.com/openclaw/openclaw/issues/85363. Current main persists the catch-up cursor only from `performIMessageCatchup`, while live `watch.subscribe` traffic flows through `handleMessageNow` and relies on process-local inbound dedupe, so a restart can replay already-handled rows when the cursor lags. Implement a narrow restart-safe fix in the iMessage plugin: either advance the per-account catch-up cursor monotonically from the live path after rows are safely handled/skipped, or add an equivalent persistent provider-message dedupe keyed by stable iMessage identifiers and route scope. Preserve existing catch-up failure-hold, give-up, parser-watermark, and perRunLimit truncation semantics; do not advance past a held failed row or cap-truncated valid rows. Add focused regression coverage for the stale cursor plus reset inbound-dedupe scenario and update docs/changelog if user-visible behavior or configuration text changes.
