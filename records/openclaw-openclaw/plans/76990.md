## Summary

Keep open: current main still captures successful embedded-run `assistantTexts` and records them in trajectory/output data, but I did not find a success-path invariant that verifies or repairs an equivalent visible assistant turn on the active session branch. The related discussion keeps this as the canonical remaining transcript-invariant issue; https://github.com/openclaw/openclaw/issues/81286 was closed as a duplicate pointing back here.

## Plan

Fix this issue by adding a focused PI embedded-runner transcript invariant repair for successful attempts with non-empty visible `assistantTexts`. After a successful run, under the appropriate session manager/session-file lock, verify that an equivalent visible assistant message is reachable on the active transcript branch after the prompt leaf; append or repair exactly one normal visible assistant message only when it is missing, no-op when already persisted, ignore whitespace-equivalent duplicates, and no-op when the manager/session path is unavailable. Do not mirror normal WebChat live delivery into JSONL and do not change provider/channel policy. Coordinate with the adjacent orphaned-user persistence work in https://github.com/openclaw/openclaw/issues/76968 and https://github.com/openclaw/openclaw/pull/76969 so the repairs do not fight. Add focused regression coverage for already-persisted, missing, whitespace-equivalent duplicate, and unavailable-manager cases, plus a user-visible `CHANGELOG.md` fix entry.
