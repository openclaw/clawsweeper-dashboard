## Summary

Keep open. Current main still lacks the requested bridge between provider quota windows and the per-response `/usage` footer: the reply path formats token/cache/cost/session data only, while `/status` and the GitHub Copilot plugin already expose the Premium/Chat usage windows.

## Plan

Implement provider quota summaries in reply `/usage` footers for https://github.com/openclaw/openclaw/issues/7379. When `responseUsage` resolves to `tokens` or `full` and the active provider has provider usage windows, use the existing generic provider usage path (`resolveUsageProviderId`, `loadProviderUsageSummary`, `formatUsageWindowSummary`) to append a compact Premium/Chat-style summary to the existing usage footer. Keep provider-specific fetching inside plugins; do not hard-code GitHub Copilot behavior in core beyond generic provider-id resolution. Bound the request with a short timeout and swallow provider usage errors so reply delivery never fails or stalls because quota lookup failed. Coordinate with https://github.com/openclaw/openclaw/pull/69618 because it touches nearby footer composition, but do not implement unrelated `messages.responseFooter` or config surfaces. Update `docs/reference/token-use.md`, add a user-facing `CHANGELOG.md` entry, and add focused regression tests for the reply-runner footer path and Copilot Premium/Chat windows.
