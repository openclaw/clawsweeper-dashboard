Keep open. Current main still builds the per-response `/usage` footer from token/cache/cost/session data only, while `/status` and the GitHub Copilot plugin already expose the requested Premium/Chat quota windows; the direct implementation PR is closed unmerged and the broader open footer PRs do not currently own this specific provider-quota request.

## Plan

Implement the requested provider quota summary in the per-response `/usage` footer. On current main, when `responseUsage` resolves to `tokens` or `full` and a reply footer is already being emitted, load the active provider's usage summary through the existing generic provider-usage path, format a compact window summary with `formatUsageWindowSummary`, and append it to the existing footer. Keep provider-specific fetching inside plugins; do not add Copilot-only HTTP logic to core. Use the active provider/runtime facts already available in the reply path, pass workspace/agent context where needed, bound the usage lookup with a short timeout, and swallow usage-fetch failures so reply delivery is never blocked. Add regression coverage for a GitHub Copilot/Premium+Chat footer case and preserve existing token/cache/cost/session behavior. Update `docs/reference/token-use.md` and add a single-line user-facing changelog entry. Coordinate with nearby open footer PRs, especially https://github.com/openclaw/openclaw/pull/69618 and https://github.com/openclaw/openclaw/pull/56081, but keep this PR narrowly scoped to provider usage windows in `/usage` footers.
