## Summary

Keep open: current main still has a source-backed Discord message-tool routing bug where a bare allowlisted Discord user ID can be resolved to `channel:<id>` before delivery, bypassing the allowFrom-aware Discord DM resolver.

## Plan

Fix https://github.com/openclaw/openclaw/issues/78572 by wiring Discord's allowFrom-aware target resolution into the generic `message(action="send")` target resolver. Preserve explicit `channel:<id>` and non-allowlisted bare numeric channel behavior; only treat bare numeric IDs as `user:<id>` when they are present in the effective Discord DM `allowFrom` for the selected account, matching the existing Discord docs and PR https://github.com/openclaw/openclaw/pull/74303 intent. Prefer changing the Discord plugin resolver surface (`extensions/discord/src/channel.ts` / `extensions/discord/src/target-resolver.ts`) over adding core Discord-specific policy. Add regression coverage for a message-tool send with `target: "<snowflake>"` and Discord `allowFrom: ["<snowflake>"]` proving the send path receives `user:<snowflake>`, plus plugin-level resolver tests. Add a single-line `CHANGELOG.md` Fixes entry because this is user-facing. Stop if the fix would require a new config option, a new plugin SDK contract, or changing explicit channel target semantics.
