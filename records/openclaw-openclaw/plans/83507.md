## Summary

This should stay open: current main and the latest release still have the reported asymmetry. The `agent.send` rotation path persists a new session and emits `sessions.changed`, while the existing reset/create paths use the session lifecycle hook helpers; the related fix PR was closed unmerged, so there is no implementation on main to close against.

## Plan

Fix https://github.com/openclaw/openclaw/issues/83507 on current main. In `src/gateway/server-methods/agent.ts`, preserve the existing `sessions.changed` create/send emissions but also emit the existing session lifecycle hook helpers when `agent.send` auto-rotates a `sessionKey` to a new `sessionId`. Use the existing helpers from `src/gateway/session-reset-service.ts`; do not create a parallel hook path. A complete fix should pair `session_end` for the replaced session before `session_start` for the new session when an old session ID exists, pass `resumedFrom`, `nextSessionId`, `nextSessionKey`, store path, session file, cfg, and agent ID, and derive the `session_end.reason` from the actual rotation cause (`new`, `idle`, `daily`, `deleted`, or `unknown`). Add focused regression coverage for caller-supplied replacement session IDs and, where practical, freshness-based daily/idle and missing-transcript rotation. Preserve existing session store behavior, dedupe, `agent.send` response timing, and `sessions.changed` subscribers. Related closed reference: https://github.com/openclaw/openclaw/pull/83523. Add a changelog fix entry only if the maintainer repair workflow expects user-visible gateway/plugin lifecycle fixes to be recorded.
