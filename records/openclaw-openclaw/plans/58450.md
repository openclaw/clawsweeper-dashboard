## Summary

Keep open. Current main still has only narrow reminder/cron and planning-only mitigations; it does not enforce the broader invariant that a final reply promising later follow-up must be backed by a real same-turn follow-up mechanism.

## Plan

Fix the unbacked future-follow-up promise gap tracked by https://github.com/openclaw/openclaw/issues/58450. Keep the change narrow: extend the existing reply-pipeline guard so final user-visible text that concretely promises later follow-up, coming back, sending a follow-up, or continuing in the background is annotated or rewritten only when the same run has no real follow-up evidence. Reuse existing runResult/attempt evidence where possible, including successful cron adds, messaging delivery, sessions_spawn/background task evidence if available, and deterministic approval/message delivery evidence; do not add a new config option. Preserve the existing reminder guard behavior and coordinate with the separate false-positive work in https://github.com/openclaw/openclaw/issues/47586 and https://github.com/openclaw/openclaw/pull/47589: do not broaden bare 'remember' matching. Do not duplicate the narrower CJK incomplete-turn work in https://github.com/openclaw/openclaw/pull/79724, but account for its intended terminal-continuation boundary if touching the same helper. Add focused regression coverage for examples like 'I'll check and come back', 'I'll look into it and send a short follow-up', and at least one non-English example from the issue, plus negative cases for answer summaries and memory-retention statements. Add a user-facing changelog Fixes entry if the change is prepared for landing.
