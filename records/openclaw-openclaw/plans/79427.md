# Coding Plan for openclaw/openclaw#79427: [Bug]: Telegram single-account with explicit empty `accounts.<X>.groups: {}` silently replaces root `channels.telegram.groups` (allowlist drops every update)

Source report: [records/openclaw-openclaw/items/79427.md](records/openclaw-openclaw/items/79427.md)

## Summary

Keep open: current main still allows an explicit empty Telegram account-level `groups` map to shadow the root group allowlist, and the same nullish fallback pattern exists in the runtime group-policy resolver used before the `not-allowed` log.

## Plan

Fix https://github.com/openclaw/openclaw/issues/79427. In a single-account Telegram setup, an explicit empty `channels.telegram.accounts.<id>.groups: {}` should be treated as absent for group inheritance so root `channels.telegram.groups` still controls group routing. Preserve the multi-account behavior from https://github.com/openclaw/openclaw/issues/30673: root Telegram groups must not leak to accounts without their own non-empty groups when more than one Telegram account is configured. Audit both `extensions/telegram/src/account-config.ts` and the runtime group-policy path in `src/config/group-policy.ts` / Telegram callers; do not change unrelated channel semantics unless a typed Telegram-specific path or covered generic helper makes that explicit. Add regression coverage for `mergeTelegramAccountConfig` and for the group-policy/access path that would otherwise return `not-allowed` with an empty account groups map. Add a `CHANGELOG.md` fix entry crediting @nikolaykazakovvs-ux if repo policy expects issue-reporter credit. Do not add a new config option or change provider/plugin boundaries.
