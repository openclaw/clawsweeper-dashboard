# Coding Plan for openclaw/openclaw#79427: [Bug]: Telegram single-account with explicit empty `accounts.<X>.groups: {}` silently replaces root `channels.telegram.groups` (allowlist drops every update)

Source report: [records/openclaw-openclaw/items/79427.md](records/openclaw-openclaw/items/79427.md)

## Summary

Keep open: current main still treats an explicit empty Telegram account-level `groups` map as a real override, so the single-account root `channels.telegram.groups` allowlist is skipped and the runtime policy path can reject the configured group as not allowed. The two linked fix PRs are closed unmerged, and I did not find an open replacement.

## Plan

Fix https://github.com/openclaw/openclaw/issues/79427. In single-account Telegram configs, treat explicit empty `accounts.<id>.groups: {}` as absent so root `channels.telegram.groups` is inherited. Preserve the multi-account contract from https://github.com/openclaw/openclaw/issues/30673: root groups must not be inherited by accounts when more than one Telegram account is configured unless the account has non-empty groups. Update both `mergeTelegramAccountConfig` and the raw group-policy resolution path used by Telegram runtime authorization so they agree. Add regression tests for single-account empty account groups falling back, non-empty account groups taking priority, and multi-account empty account groups not falling back. Do not add a config option, do not change unrelated channels unless the shared resolver needs a channel-scoped guard, and include a user-facing changelog fix entry.
