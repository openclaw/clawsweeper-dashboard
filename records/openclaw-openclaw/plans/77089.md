# Coding Plan for openclaw/openclaw#77089: CLI backend: retry fresh session on any FailoverError, not just session_expired

Source report: [records/openclaw-openclaw/items/77089.md](records/openclaw-openclaw/items/77089.md)

## Summary

Keep open: current main still only clears and fresh-retries a persisted CLI session binding for `session_expired`, while `timeout` and `unknown` `FailoverError` paths can still rethrow from a resumed binding and leave the stale binding in place. The two linked contributor PRs were closed unmerged, and prior review context identified a hook-lifecycle concern that a replacement fix should avoid.

## Plan

Fix https://github.com/openclaw/openclaw/issues/77089 with a narrow CLI backend session recovery PR. Current main only clears persisted CLI session bindings and retries fresh for `session_expired`; make resumed-session `FailoverError` handling clear the stale persisted binding for every reason. Retry with a fresh CLI session exactly once only for recoverable stale-session reasons such as `session_expired`, `timeout`, and `unknown`; for non-recoverable reasons such as auth, billing, and rate limits, clear the stale binding and rethrow without retrying. Preserve `agent_end` hook lifecycle consistency: do not implement a successful outer retry after `runPreparedCliAgent` has already emitted a failed final hook event. Prefer moving the recoverable retry under the CLI runner lifecycle or adding an equivalent hook-aware store-clear seam. Add focused regression coverage for timeout and unknown stale-binding recovery, stale binding clearing before a fresh retry that then fails with a different reason, and non-recoverable clear-without-retry behavior. Reuse `clearCliSessionInStore`/session-store helpers and avoid unrelated provider, config, or watchdog-default changes. Add a user-facing changelog entry if this becomes a landed reliability fix.
