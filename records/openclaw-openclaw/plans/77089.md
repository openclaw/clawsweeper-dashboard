# Coding Plan for openclaw/openclaw#77089: CLI backend: retry fresh session on any FailoverError, not just session_expired

Source report: [records/openclaw-openclaw/items/77089.md](records/openclaw-openclaw/items/77089.md)

## Summary

Keep open: current main still only clears and fresh-retries a persisted CLI session binding for `session_expired`, while `timeout` and `unknown` `FailoverError` paths can still rethrow from a resumed binding and leave the stale session in place. The two linked contributor PRs are closed unmerged, so there is no active implementation candidate covering the remaining bug.

## Plan

Fix https://github.com/openclaw/openclaw/issues/77089 in a focused PR. Current main still clears/retries persisted CLI session bindings only for `FailoverError.reason === "session_expired"` in `src/agents/command/attempt-execution.ts`, while `src/agents/cli-runner/execute.ts` can throw `timeout` and `unknown` from the same resumed-session path. Implement narrow recovery for resumed CLI sessions with an active binding: clear the persisted binding for any `FailoverError`; retry fresh exactly once only for recoverable reasons (`session_expired`, `timeout`, `unknown`); for non-recoverable reasons such as `auth`, `billing`, and `rate_limit`, clear the stale binding and rethrow without retry. Preserve no behavior change for fresh-session failures with no active binding. Keep the hook lifecycle correct: do not return success after `runPreparedCliAgent` has already emitted a failed final hook event; prefer a hook-aware/store-aware seam or callback if retrying outside the runner would violate that contract. Add focused regression coverage for timeout and unknown stale-session recovery, non-recoverable clear-without-retry, and a fresh retry failure path that does not leave the original stale binding. Do not add config, schema migration, or broad fallback policy changes. Add a changelog Fixes entry if the repair is user-facing.
