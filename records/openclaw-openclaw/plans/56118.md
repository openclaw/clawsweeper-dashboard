## Summary

Keep open: current main still supports the reported node-host browser proxy path, but direct `openclaw nodes invoke --command browser.proxy` does not get the longer consent-safe timeout budget that the browser tool already uses for `profile="user"` through a browser node. The FSB extension-bridge comments are useful comparison context, but they do not provide an OpenClaw core fix for this direct proxy path.

## Plan

Fix the direct node browser proxy existing-session timeout mismatch reported in https://github.com/openclaw/openclaw/issues/56118. Current main gives the browser tool's `profile="user"` node route a 45000 ms proxy budget plus 50000 ms gateway budget, but direct `openclaw nodes invoke --command browser.proxy --params '{"path":"/tabs","query":{"profile":"user"},"method":"GET"}'` still goes through the generic 15000 ms CLI invoke budget, preserves that through gateway node.invoke, and lets `runBrowserProxyCommand` fall back to 20000 ms when command params omit `timeoutMs`. Implement a browser-owned normalization path so direct `browser.proxy` requests for existing-session profile manage routes such as `/`, `/profiles`, `/tabs`, and tab manage actions get the same consent-safe budget while preserving explicit caller `timeoutMs`, mutation guards, allowProfiles behavior, and generic node-command defaults for unrelated commands. Add focused regression tests for the direct CLI/gateway/node-host path and existing browser-tool route; add a single-line changelog entry because this is user-visible. Stop and ask if the implementation would require MCP session reuse, Chrome pre-authorization, an extension bridge, or a broad product/config decision.
