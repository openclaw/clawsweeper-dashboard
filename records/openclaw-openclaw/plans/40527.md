# Coding Plan for openclaw/openclaw#40527: Remote skill bin probe times out when node is co-located with gateway (macOS app as node on same host)

Source report: [records/openclaw-openclaw/items/40527.md](records/openclaw-openclaw/items/40527.md)

## Summary

Keep open: current main still starts the automatic macOS remote-bin probe during node-role connect before the Swift client starts its normal receive loop, and the latest release has the same ordering. The user follow-up confirms the automatic probe can time out while manual `system.which` succeeds, and the related open PR only targets probe churn/backoff rather than the first-connect readiness bug.

## Plan

Fix https://github.com/openclaw/openclaw/issues/40527 by preventing automatic remote macOS bin probes from being sent before an app-backed node can receive normal `node.invoke.request` events. Prefer a readiness contract after `hello-ok`/Swift listen starts, or safe buffering/dispatch of handshake-time events; do not add a global disable flag, do not suppress valid remote macOS node probing, and do not remove `system.run`/`system.which` support from legitimate macOS nodes. Coordinate with https://github.com/openclaw/openclaw/pull/79438 as churn/backoff work only; the fix here should address first-connect ordering. Add focused TypeScript and Swift regression coverage and update docs only if the node readiness contract changes user-visible behavior.
