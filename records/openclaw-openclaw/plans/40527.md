# Coding Plan for openclaw/openclaw#40527: Remote skill bin probe times out when node is co-located with gateway (macOS app as node on same host)

Source report: [records/openclaw-openclaw/items/40527.md](records/openclaw-openclaw/items/40527.md)

## Summary

Keep open. Current main still has a source-reproducible connect-ordering bug: the gateway can send the automatic remote bin probe as a node event before the Swift macOS node starts its normal receive loop, so the first probe can be dropped and later time out despite the node supporting `system.which`.

## Plan

Fix https://github.com/openclaw/openclaw/issues/40527. Current main registers node sessions and starts `refreshRemoteNodeBins` before sending `hello-ok`; `NodeRegistry.invoke` delivers the probe as `node.invoke.request`; Swift `GatewayChannel.waitForConnectResponse` can consume non-response events before `listen()` starts, so the first app-node bin probe can be lost and time out. Implement a narrow readiness/order fix: defer automatic remote bin refresh until the node is ready after `hello-ok`, or buffer and dispatch ordinary events observed during connect response handling. Preserve node command policy, macOS exec approvals, and existing `system.which`/`system.run` behavior. Do not add a config flag or loopback-only skip as the primary fix. Add regression coverage for the pre-`hello-ok` node invoke/readiness path plus existing remote skills behavior, and include a user-facing changelog entry for the fix.
