# Coding Plan for openclaw/openclaw#40527: Remote skill bin probe times out when node is co-located with gateway (macOS app as node on same host)

## Summary

Keep open. Current `main` and the latest release still start the automatic remote-bin probe during node connect before the Swift macOS client enters its normal receive loop, and the newer user comment confirms manual `system.which` can work while the automatic probe times out.

## Plan

Fix https://github.com/openclaw/openclaw/issues/40527 by preventing automatic remote skill bin probes from timing out during macOS app node connect. Current main registers a node and calls `refreshRemoteNodeBins` before `hello-ok`; the Swift app only starts its normal receive loop after the connect response, so the initial `node.invoke.request` can time out. Keep the fix narrow: gate/defer the automatic refresh until post-`hello-ok` readiness, or dispatch/buffer handshake-time events in the Swift channel. Do not add a global disable flag, do not remove legitimate `system.run`/`system.which` support, and do not fold in TTL/backoff/cache churn from https://github.com/openclaw/openclaw/pull/79438. Add focused TypeScript/Swift regression coverage and include a changelog entry if the user-visible warning behavior changes.
