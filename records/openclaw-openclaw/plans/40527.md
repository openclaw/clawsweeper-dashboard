## Summary

Keep open. Current main still schedules the automatic remote-bin refresh for a node-role connection before `hello-ok` is sent, while the Swift gateway channel starts its normal receive loop only after the connect response; the newer user comment also narrows this to connect/readiness timing rather than a missing `system.which` implementation.

## Plan

Fix the macOS app-backed node remote-bin probe readiness race for https://github.com/openclaw/openclaw/issues/40527. Current main registers node-role connections and schedules `refreshRemoteNodeBins` before sending `hello-ok`, while `GatewayChannel.connect()` starts the normal receive loop only after the connect response; this can drop or starve the automatic `node.invoke.request` and produce `remote bin probe timed out` even though manual `system.which` works later. Implement a narrow readiness/order fix: either gate the automatic remote-bin refresh until the node is confirmed ready after `hello-ok`, or buffer/dispatch pre-listen node events in the Swift channel. Preserve node command policy and macOS app `system.which`/`system.run` support. Do not add a global disable config, do not rely on loopback-only skipping as the sole fix, and do not duplicate the TTL/backoff/cache work tracked by https://github.com/openclaw/openclaw/pull/79438. Add focused regression coverage for the ordering path and update CHANGELOG.md if the fix is user-visible. Validate with the focused TS and Swift tests, and when feasible capture redacted macOS app plus gateway proof showing the automatic connect-time probe no longer times out.
