# Coding Plan for openclaw/openclaw#40527: Remote skill bin probe times out when node is co-located with gateway (macOS app as node on same host)

Source report: [records/openclaw-openclaw/items/40527.md](records/openclaw-openclaw/items/40527.md)

## Summary

Keep open. Current main still starts the automatic remote macOS bin probe before the node client has finished the hello-ok handshake and started its normal Swift receive loop, so the connect-time probe can still be dropped and later time out; the related open churn PR may reduce repeated probes but does not close this readiness bug.

## Plan

Fix the connect-time remote macOS bin probe readiness race for https://github.com/openclaw/openclaw/issues/40527. On current main, src/gateway/server/ws-connection/message-handler.ts registers a node and starts refreshRemoteNodeBins before hello-ok is sent, while the Swift GatewayChannel starts listen only after sendConnect completes; a node.invoke.request emitted in that window can be consumed by waitForConnectResponse and dropped. Implement the narrowest fix so automatic remote bin refresh runs only after the node client can receive ordinary events, or buffer/dispatch pre-hello events in OpenClawKit. Preserve node command policy, macOS system.which/system.run declarations, exec approval behavior, and the existing remote skill eligibility contract. Do not add a global disable flag, do not use denyCommands as the fix, and do not duplicate the TTL/backoff scope from https://github.com/openclaw/openclaw/pull/79438 except to stay compatible with it. Add focused regression coverage for event/probe ordering and update existing tests as needed; no changelog is required unless the final behavior change is user-visible release-note material.
