# Coding Plan for openclaw/openclaw#14079: Linux: No desktop notifications support

Source report: [records/openclaw-openclaw/items/14079.md](records/openclaw-openclaw/items/14079.md)

## Summary

Keep open. Current main and the latest shipped release still lack Linux desktop notification delivery for the core `system.notify` node surface: Linux policy allowlists the command, but the headless node host neither advertises nor dispatches it, docs/CLI still scope notification delivery to macOS, and plugins cannot claim the reserved command.

## Plan

Implement Linux desktop notification support for the existing core `system.notify` node surface requested in https://github.com/openclaw/openclaw/issues/14079. Keep the fix focused on the headless Linux node host: advertise `system.notify` only when the Linux backend can make a real attempt or return a structured unavailable result; dispatch it safely without shell interpolation; use argv-based process execution for `notify-send` if chosen; gracefully handle missing desktop session (`DISPLAY`/`WAYLAND_DISPLAY`) and missing backend; do not add broad D-Bus or `dunstify` fallback unless the implementation remains small and well-tested. Preserve existing macOS, iOS, Android, Windows, `system.run`, `system.which`, and exec approval behavior. Update CLI copy, public node docs, and `CHANGELOG.md` because this is user-visible. Add regression coverage for command advertisement, invocation success/unavailable paths, sanitized environment behavior, CLI coverage, and node-command policy behavior. Treat https://github.com/openclaw/openclaw/issues/24616 as adjacent diagnostic context only, not as the delivery fix.
