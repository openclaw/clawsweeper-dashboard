# Coding Plan for openclaw/openclaw#46899: [Bug]: Cron Job Delivery Channel Mismatch - Telegram Configured but Feishu Used

Source report: [records/openclaw-openclaw/items/46899.md](records/openclaw-openclaw/items/46899.md)

## Summary

Keep open. Current main and the latest release still parse cron `delivery.channel` as string-only and delete object-shaped persisted channel values before delivery resolution, so the reported Telegram-to-Feishu fallback path remains source-reproducible.

## Plan

Fix https://github.com/openclaw/openclaw/issues/46899 on current main. The source-proven bug is that persisted cron jobs can have `delivery.channel` shaped as `{ id: "telegram" }`; current `parseDeliveryInput()`/`coerceDelivery()`/store hydration erase that channel, then delivery planning and target resolution fall back to `last`, which can select Feishu or another configured channel for a Telegram target. Implement a generic cron-local fix: either accept `{ id: string }` as a channel value or fail closed before fallback routing, preserving valid string-channel behavior, provider-prefixed target validation, explicit `delivery.mode: "none"`, webhook/failure-destination behavior, and the implicit-`last` admission work in https://github.com/openclaw/openclaw/pull/80438. Do not add Telegram- or Feishu-specific branches. Add regression coverage for persisted hydration and the delivery plan/target path, and include a single CHANGELOG.md Fixes entry if the change is user-visible.
