# Coding Plan for openclaw/openclaw#44353: Fallback models not triggered on provider-level errors (e.g., AWS Bedrock model ID changes)

Source report: [records/openclaw-openclaw/items/44353.md](records/openclaw-openclaw/items/44353.md)

## Summary

Keep open: current main has generic fallback handling for unrecognized provider errors, but normal spawned subagent runs still lose the fallback list attached to the selected `subagents.model` config before `runWithModelFallback` builds candidates.

## Plan

Fix https://github.com/openclaw/openclaw/issues/44353. Current main already falls back on unrecognized provider errors when the candidate chain contains fallbacks, but normal `sessions_spawn` child sessions that select `agents.*.subagents.model` persist only the selected provider/model and `modelOverrideSource: "auto"`; `agent-command` then calls `resolveEffectiveModelFallbacks`, which reads regular `agents.*.model` fallback policy and ignores the selected `subagents.model.fallbacks`. Carry the selected subagent model fallback list into the normal spawned-subagent `runWithModelFallback` / `modelFallbacksOverride` path. Preserve explicit `sessions_spawn.model` as a strict user override, explicit empty fallback arrays, existing cron behavior from https://github.com/openclaw/openclaw/pull/82302, auto fallback primary-probe semantics, and avoid adding a new config option. Add focused regression coverage for default and per-agent `subagents.model: { primary, fallbacks }` in the normal spawned-subagent path, plus coverage that regular agent fallbacks and explicit user overrides still behave. Add a changelog entry because this is user-visible fallback behavior.
