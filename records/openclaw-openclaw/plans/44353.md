# Coding Plan for openclaw/openclaw#44353: Fallback models not triggered on provider-level errors (e.g., AWS Bedrock model ID changes)

Source report: [records/openclaw-openclaw/items/44353.md](records/openclaw-openclaw/items/44353.md)

## Summary

Keep open: current main has generic provider-error fallback behavior, but subagent model fallback lists are still dropped before spawned subagent runs reach the fallback runner.

## Plan

Fix the current issue by preserving configured subagent model fallbacks for spawned subagent runs. Current main already accepts `agents.defaults.subagents.model` and `agents.list[].subagents.model` as `{ primary, fallbacks }` and selects the primary, but the selected fallback list is dropped before `runWithModelFallback`. Add a subagent-aware fallback resolution path that feeds the selected subagent fallback list into `fallbacksOverride` for auto-selected spawned subagent runs, while preserving explicit user model overrides and explicit `fallbacks: []` semantics. Do not broaden this into cron preflight handling from https://github.com/openclaw/openclaw/issues/79329 unless the same helper can be reused cleanly. Add focused regression coverage for default and per-agent `subagents.model` configs and keep regular `agents.*.model.fallbacks` behavior unchanged. Because this is user-visible fallback behavior, include a changelog entry if a PR is prepared.
