## Summary

Keep open: current main and the latest tagged release still route Feishu `/status` and `/new` command-only text through the normal same-chat sequential key, so those commands can wait behind an active Feishu run. `/stop` now has a shipped control lane, but the broader reported fast-path regression is not fully fixed in OpenClaw main, and the target-repo fix PRs are closed unmerged while the only open closing PR is in a fork.

## Plan

Fix the Feishu text-command queue fast-path regression tracked by https://github.com/openclaw/openclaw/issues/42803. On current main, `createFeishuMessageReceiveHandler` always enqueues by `getFeishuSequentialKey`; that helper only routes abort-like text and `/btw` off the base chat lane, so `/status` and `/new` can wait behind an active same-chat run. Implement a narrow Feishu-owned sequential-key/message-handler fix so documented command-only immediate commands use an unblocked lane while ordinary Feishu messages keep same-chat FIFO serialization. Do not reintroduce the direct unbounded bypass or pre-authorization debounce-drop patterns from the closed target PRs; consider https://github.com/BingqingLyu/openclaw/pull/484 only as source context and audit it before reuse. Add/update regression tests for `/status`, `/new`, existing `/stop`, normal messages, and same-chat control-lane behavior. Include a single-line `CHANGELOG.md` fix entry because this is user-visible Feishu behavior.
