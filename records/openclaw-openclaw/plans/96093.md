# Coding Plan for openclaw/openclaw#96093: Diagnostics: no model.usage emitted for HTTP ingress (/v1/responses, /v1/chat/completions) — exporters blind to API traffic

Source report: [records/openclaw-openclaw/items/96093.md](records/openclaw-openclaw/items/96093.md)

## Summary

Keep open: current main still maps HTTP ingress usage into API responses and session state, but it does not emit the established `model.usage` diagnostic event for the shared `/v1/responses` and `/v1/chat/completions` ingress path, so exporters can miss token/cost telemetry for that traffic.

## Plan

Fix the HTTP-ingress diagnostics gap reported in https://github.com/openclaw/openclaw/issues/96093. Current main returns usage in `/v1/chat/completions` and `/v1/responses` responses from `result.meta.agentMeta`, but only channel replies and cron emit `model.usage`; add a shared, narrow core helper or equivalent small refactor so completed HTTP ingress turns emit the existing `model.usage` diagnostic once when diagnostics are enabled and usage is nonzero. Preserve existing response usage wire formats, do not add config/env options, do not add exporter-specific behavior, and avoid double-emitting for channel/cron paths. Add focused regression coverage for both OpenAI-compatible endpoints proving a trusted diagnostic subscriber receives `model.usage` with provider, model, channel, session id/key where available, usage buckets, and cost behavior aligned with existing reply/cron emissions.
