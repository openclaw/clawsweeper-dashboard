# Coding Plan for openclaw/openclaw#40991: Telegram inbound video can degrade to <media:video> placeholder with no attachment and no error when getFile() fails

Source report: [records/openclaw-openclaw/items/40991.md](records/openclaw-openclaw/items/40991.md)

## Summary

Keep open: current main and v2026.5.22 still have the source-reproducible Telegram path where `getFile()` failures return `null`, the single-message warning path only handles thrown errors, and body rendering can still emit video media metadata without an attached file. The direct target-repo fix PRs were closed unmerged, and the open document-focused PR does not fully own this video failure mode.

## Plan

Fix https://github.com/openclaw/openclaw/issues/40991 in the current `extensions/telegram` plugin layout. Current main returns `null` from `resolveMedia()` when Telegram `ctx.getFile()` fails or reports file-too-big, then the single-message handler continues with `allMedia = []` and body rendering can emit `<media:video>` or `<media:video> [file_id:...]` without an attachment or user-visible unavailable-media reason. Implement a narrow Telegram-owned fix that carries an explicit unavailable-media reason through `delivery.resolve-media.ts`, `bot-handlers.runtime.ts`, debounce/media-group/reply-context handling, and `bot-message-context.body.ts`. Preserve useful fallback metadata such as media type and file_id when no file path is available, but make clear that the attachment is unavailable; do not imply the agent received a real local file. Avoid unthrottled warning replies by adding bounded per-chat/sender/media-group suppression or otherwise proving warnings cannot spam allowed chats. Account for Telegram Bot API's public 20MB getFile download limit and self-hosted/local file paths already supported by existing code. Review the closed target PRs https://github.com/openclaw/openclaw/pull/41048 and https://github.com/openclaw/openclaw/pull/42059 for pitfalls, and do not treat the document-scoped https://github.com/openclaw/openclaw/pull/55970 as sufficient for the video issue. Add focused regression coverage for resolver null/failure reasons, single-message video body rendering, media groups, and any debounce/reply-context propagation touched. No core migration, config option, package/dependency change, or broad Telegram refactor should be included unless the existing plugin contract truly requires it.
