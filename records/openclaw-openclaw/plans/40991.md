# Coding Plan for openclaw/openclaw#40991: Telegram inbound video can degrade to <media:video> placeholder with no attachment and no error when getFile() fails

Source report: [records/openclaw-openclaw/items/40991.md](records/openclaw-openclaw/items/40991.md)

## Summary

Keep open: current main and v2026.5.22 still allow Telegram `getFile()` failures to return `null`, skip the single-message failure reply path, and render media metadata without an attached file. The earlier direct fix PRs were closed unmerged, while the open document-focused PR does not cover this video failure mode.

## Plan

Fix the Telegram inbound media null-resolution path for this issue. In current `extensions/telegram`, make failed `getFile()` resolution for video/media carry an explicit unavailable-media reason into single-message handling, debounce bursts, media groups, reply-context media, and `resolveTelegramInboundBody`; preserve useful placeholder/file_id metadata as metadata-unavailable rather than implying an attached file; add per-chat/sender or media-group warning suppression so repeated failures cannot spam chats; keep the fix inside the Telegram plugin/SDK boundary; do not change core channel contracts unless a missing generic seam is proven. Add or update focused regression tests for file-too-big and retry-exhausted `getFile()`, captioned video body rendering with empty `allMedia`, media groups, debounce propagation, and reply-context behavior. Add a user-facing changelog entry under the active fixes section. Before landing, collect redacted real Telegram proof for an unavailable or oversized video and a normal small video attachment.
