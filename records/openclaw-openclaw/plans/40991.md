## Summary

Keep open: current main and the latest release still allow Telegram getFile failures to become a null media result, skip the single-message warning path, and render video metadata without an attachment. The target-repo fix PRs are closed unmerged, while the remaining open related PR is document-scoped rather than a complete fix for this video path.

## Plan

Fix https://github.com/openclaw/openclaw/issues/40991 in the Telegram plugin. In extensions/telegram, carry getFile failure state from resolveTelegramFileWithRetry/resolveMedia into the single-message handler and body builder so failed inbound video/media does not look like an attached file. Preserve caption text and useful <media:video> [file_id:...] metadata as unavailable media context, add an accurate rate-limited user-visible warning or agent-visible unavailable reason, and do not add config or change the successful media attachment path. Cover file-too-big and exhausted getFile retries, captioned and no-caption video, debounce/media-group propagation if touched, and avoid unbounded warning spam. Coordinate with the document-only fallback work in https://github.com/openclaw/openclaw/pull/55970 if it is still open. Add a user-facing changelog entry unless maintainers decide to land it as internal-only.
