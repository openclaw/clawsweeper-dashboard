## Summary

Keep open: this is a source-backed Xiaomi MiMo compatibility bug on current main, not an already-implemented or duplicate cleanup item. The current Xiaomi OpenAI-compatible path advertises reasoning MiMo models but does not apply the existing reasoning-content replay backfill that only targets DeepSeek V4 model ids, and Xiaomi’s own contract says missing `reasoning_content` after tool-call history causes 400 responses.

## Plan

Fix Xiaomi MiMo OpenAI-compatible thinking tool-call replay so affected Xiaomi MiMo reasoning models include `reasoning_content` on replayed assistant messages when thinking is enabled, preventing follow-up `400 Param Incorrect` responses. Keep core plugin-agnostic: prefer a Xiaomi plugin `wrapStreamFn` using the existing shared DeepSeek-style payload patch helper or a small generic provider-family helper only if needed. Cover `xiaomi` provider refs for bundled `mimo-v2-pro`/`mimo-v2-omni` and configured/custom MiMo V2.5 ids without broad provider/baseUrl overmatching. Coordinate with https://github.com/openclaw/openclaw/pull/60304 because that PR normalizes reasoning-only final output but does not add request replay passback. Add focused tests for assistant tool-call history with enabled thinking and disabled thinking, and do not change unrelated Xiaomi TTS, auth, or model-selection behavior. Add a changelog entry because this is user-visible provider behavior.
