# Coding Plan for openclaw/openclaw#58765: Feature: Support output dimensionality truncation for local GGUF embedding models

Source report: [records/openclaw-openclaw/items/58765.md](records/openclaw-openclaw/items/58765.md)

## Summary

Keep open: current main and the latest v2026.5.18 release still accept and forward `memorySearch.outputDimensionality`, but the built-in local GGUF provider ignores it and returns full normalized vectors. The closed OpenAI-compatible dimensions issue does not cover this local provider path.

## Plan

Fix https://github.com/openclaw/openclaw/issues/58765 by making the built-in local GGUF memory embedding provider honor the existing `memorySearch.outputDimensionality` setting. Keep the boundary narrow: do not add a new config key unless the existing schema/types/docs make it unavoidable, and do not implement `queryInstruction` in this PR. In `packages/memory-host-sdk/src/host/embeddings.ts`, truncate local `embedding.vector` to the configured positive integer before calling `sanitizeAndNormalizeEmbedding`, for both `embedQuery` and `embedBatch`; preserve current full-vector behavior when unset and define sensible no-op behavior when the requested size is greater than or equal to the native vector length. In `extensions/memory-core/src/memory/provider-adapters.ts`, include `outputDimensionality` in the local runtime `cacheKeyData` so dimension changes separate cache/reindex identity. Add focused tests for query and batch truncation/normalization and local cache-key identity/reindex behavior. Update memory config docs and public type/help wording if needed to say local GGUF uses `outputDimensionality` for post-vector truncation. Add a changelog entry because this is user-visible. Do not change node-llama-cpp install policy, provider selection, or remote provider behavior.
