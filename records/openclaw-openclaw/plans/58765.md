# Coding Plan for openclaw/openclaw#58765: Feature: Support output dimensionality truncation for local GGUF embedding models

Source report: [records/openclaw-openclaw/items/58765.md](records/openclaw-openclaw/items/58765.md)

## Summary

Keep open: current `main` and the latest release still preserve `memorySearch.outputDimensionality` through generic memory-search config, but the built-in local GGUF provider never reads it and still returns full normalized vectors. The related closed dimension work covers OpenAI-compatible and generic provider plumbing, not this local provider truncation path.

## Plan

Fix https://github.com/openclaw/openclaw/issues/58765 by making the built-in local GGUF memory embedding provider honor the existing `memorySearch.outputDimensionality` field. Scope the change to local vector truncation: slice raw local embedding vectors to the configured positive dimension before normalization for both `embedQuery` and `embedBatch`, preserve full-vector behavior when unset, and stop if the implementation would require a new public config key or query-instruction product design. Include the configured dimension in the local provider runtime `cacheKeyData` so changing it changes providerKey/reindex identity. Add focused tests in `packages/memory-host-sdk/src/host/embeddings.test.ts` for query and batch truncation plus unset behavior, and in memory-core reindex/cache identity tests if needed. Update `docs/reference/memory-config.md` and config help/types wording to mention local truncation support only after the behavior exists. Add a user-facing changelog fix entry unless maintainers decide this is release-note-only.
