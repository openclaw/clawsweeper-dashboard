# Coding Plan for openclaw/openclaw#58765: Feature: Support output dimensionality truncation for local GGUF embedding models

Source report: [records/openclaw-openclaw/items/58765.md](records/openclaw-openclaw/items/58765.md)

## Summary

Keep open: current main still resolves `memorySearch.outputDimensionality`, but the built-in local GGUF provider ignores it and the local provider cache identity still omits the configured dimension. The related OpenAI-compatible dimensions work is shipped, but it does not cover this local GGUF path.

## Plan

Fix local GGUF memory embedding dimensionality for https://github.com/openclaw/openclaw/issues/58765. Keep the scope to the existing `memorySearch.outputDimensionality` contract; do not add `queryInstruction` or a new config key. In `packages/memory-host-sdk/src/host/embeddings.ts`, make `createLocalEmbeddingProvider` honor `options.outputDimensionality` for `embedQuery` and `embedBatch`, truncating the raw `embedding.vector` to the first N dimensions before `sanitizeAndNormalizeEmbedding` so stored vectors stay normalized. Define sane behavior for dimensions greater than the native vector length, preferably preserve the full vector or validate with a clear error consistent with existing config validation. In `extensions/memory-core/src/memory/provider-adapters.ts`, include the configured `outputDimensionality` in local provider `runtime.cacheKeyData` so provider-key changes trigger the existing full-reindex path. Add focused tests for query and batch truncation in `packages/memory-host-sdk/src/host/embeddings.test.ts`, and coverage that local provider cache identity changes when output dimensionality changes. Update `docs/reference/memory-config.md` and relevant config/type/help comments to document local support and reindex behavior. Add a changelog entry if this becomes a user-facing fix PR. Do not touch unrelated memory providers or broad query-instruction behavior.
