# Coding Plan for openclaw/openclaw#58765: Feature: Support output dimensionality truncation for local GGUF embedding models

Source report: [records/openclaw-openclaw/items/58765.md](records/openclaw-openclaw/items/58765.md)

## Summary

Keep open: current `main` and latest release still preserve `memorySearch.outputDimensionality` through generic memory-search config, but the built-in local GGUF provider never reads it and still returns full normalized vectors. The related closed dimension issues cover OpenAI-compatible and generic dimension plumbing, not this local provider path.

## Plan

Fix https://github.com/openclaw/openclaw/issues/58765 by making the built-in local GGUF memory embedding provider honor the existing `memorySearch.outputDimensionality` setting. Keep the scope narrow: in `packages/memory-host-sdk/src/host/embeddings.ts`, truncate the raw `embedding.vector` to the configured positive dimension before `sanitizeAndNormalizeEmbedding` for both `embedQuery` and `embedBatch`; if the requested dimension exceeds the returned vector length, fail clearly rather than storing malformed vectors. In `extensions/memory-core/src/memory/provider-adapters.ts`, include `outputDimensionality` in the local adapter `cacheKeyData` so dimension changes force cache/reindex separation. Add focused regression tests for query and batch truncation/normalization and for provider-key reindex identity, update local memory config docs/help/types wording, and add a user-facing changelog entry. Do not add a new `truncateDimensions` config key, do not implement the separate query-instruction feature, and do not change remote provider behavior.
