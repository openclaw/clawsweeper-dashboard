# Coding Plan for openclaw/openclaw#10467: [Feature Request]: Multi-lane concurrency support for sub-agents via sessions_spawn

Source report: [records/openclaw-openclaw/items/10467.md](records/openclaw-openclaw/items/10467.md)

## Summary

Keep open. Current main and the latest release still implement `sessions_spawn` with a fixed `subagent` queue lane, no caller-provided lane parameter, and only one scalar subagent concurrency cap.

## Plan

Fix https://github.com/openclaw/openclaw/issues/10467. Add an optional model-facing `sessions_spawn.lane` string parameter that defaults to the existing `subagent` lane, validate/normalize lane names with bounded rules and reserved-name handling, forward the resolved lane through both native `spawnSubagentDirect` and ACP `spawnAcpDirect`, and apply compatible per-lane subagent concurrency configuration while preserving `agents.defaults.subagents.maxConcurrent` as the default cap for the fallback `subagent` lane. Update strict config types/schema/help/docs, focused regression tests for schema/dispatch/config/gateway behavior, and a changelog entry. Do not change existing callers' default lane, do not introduce a broad orchestration framework, and coordinate naming with adjacent concurrency issues rather than adding overlapping knobs.
