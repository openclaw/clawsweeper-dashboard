# Coding Plan for openclaw/openclaw#10467: [Feature Request]: Multi-lane concurrency support for sub-agents via sessions_spawn

Source report: [records/openclaw-openclaw/items/10467.md](records/openclaw-openclaw/items/10467.md)

## Summary

Keep open. Current main still routes `sessions_spawn` through one fixed `subagent` queue lane: the schema has no caller-selected lane, native and ACP dispatch both hard-code `AGENT_LANE_SUBAGENT`, and config/docs/tests still describe scalar global subagent concurrency.

## Plan

Implement https://github.com/openclaw/openclaw/issues/10467 as a narrow backward-compatible feature. Add optional `sessions_spawn.lane` support that defaults to `subagent`, validates lane names with a bounded safe grammar, forwards the resolved lane through both native `spawnSubagentDirect` and ACP `spawnAcpDirect`, and applies compatible per-lane subagent concurrency while preserving `agents.defaults.subagents.maxConcurrent` as the scalar fallback for the default `subagent` lane. Update config types, strict zod schema/help/labels if needed, gateway startup and reload lane application, docs for subagents and ACP sessions, and focused tests. Do not change unrelated inbound per-agent lane behavior, do not solve session capacity reporting, and do not implement per-parent subagent caps in this PR. Add a user-facing changelog entry because this changes tool/config surface.
