# Coding Plan for openclaw/openclaw#10467: [Feature Request]: Multi-lane concurrency support for sub-agents via sessions_spawn

Source report: [records/openclaw-openclaw/items/10467.md](records/openclaw-openclaw/items/10467.md)

## Summary

Keep open. Current main still exposes `sessions_spawn` through one fixed `subagent` lane, with no caller-selectable lane field, no native/ACP forwarding, and no per-lane subagent concurrency config; the related open items cover adjacent scheduling scopes rather than superseding this exact request.

## Plan

Implement the feature requested in https://github.com/openclaw/openclaw/issues/10467. Add a backward-compatible optional `sessions_spawn.lane` parameter that defaults to the current `subagent` lane, validates and normalizes lane names, and forwards the resolved lane through both native `spawnSubagentDirect` and ACP `spawnAcpDirect` child dispatch. Add a compatible per-lane subagent concurrency config surface under the existing agent defaults/config schema, preserving `agents.defaults.subagents.maxConcurrent` as the default/fallback for the built-in `subagent` lane. Update strict config types, zod schema, generated schema/help artifacts if required by the repo, docs for `docs/tools/subagents.md` and `docs/tools/acp-agents.md`, and add a user-facing changelog entry. Add focused regression tests proving the tool schema accepts `lane`, native and ACP spawns dispatch on the selected lane, omitted lane preserves `subagent`, per-lane concurrency is applied at gateway startup/reload, and invalid lane names are rejected. Do not broaden the PR into per-parent spawn throttling, inbound per-agent main-lane routing, capacity-reporting tools, or CLI wait/lifecycle changes unless maintainers explicitly ask.
