# Coding Plan for openclaw/openclaw#10467: [Feature Request]: Multi-lane concurrency support for sub-agents via sessions_spawn

Source report: [records/openclaw-openclaw/items/10467.md](records/openclaw-openclaw/items/10467.md)

## Summary

Keep open. Current main still exposes `sessions_spawn` through one fixed `subagent` lane, and the recent maintainer-side comments are still accurate: the tool schema, native/ACP dispatch, config schema, docs, and tests have not gained caller-selected lanes or per-lane subagent limits. Related items are adjacent rather than superseding this request.

## Plan

Implement backward-compatible multi-lane support for `sessions_spawn` for https://github.com/openclaw/openclaw/issues/10467. Add an optional `lane` field to the tool schema, default it to `subagent`, validate lane names with bounded/low-cardinality rules, and forward the resolved lane through both native `spawnSubagentDirect` and ACP `spawnAcpDirect` gateway agent calls. Add compatible per-lane subagent concurrency config while preserving `agents.defaults.subagents.maxConcurrent` as the scalar fallback and preserving existing behavior when `lane` is omitted. Update schema/types/help/docs, gateway startup and hot-reload lane concurrency application, focused regression tests, and add a user-facing `CHANGELOG.md` entry. Do not change channel delivery semantics, thread binding behavior, subagent nesting policy, or the existing default `subagent` lane behavior beyond the new additive lane option.
