# Coding Plan for openclaw/openclaw#10467: [Feature Request]: Multi-lane concurrency support for sub-agents via sessions_spawn

Source report: [records/openclaw-openclaw/items/10467.md](records/openclaw-openclaw/items/10467.md)

## Summary

Keep open. Current main and the latest release still expose `sessions_spawn` through one fixed `subagent` queue lane, with no caller-selected `lane` parameter or per-lane subagent concurrency config; related open issues cover per-agent, per-parent, or output lifecycle concerns but do not supersede this exact request.

## Plan

Implement multi-lane `sessions_spawn` support for https://github.com/openclaw/openclaw/issues/10467. Add a validated optional `lane` parameter to the tool schema, default omitted or invalid-absent usage to the existing `subagent` lane, thread the resolved lane through both native `spawnSubagentDirect` and ACP `spawnAcpDirect`, and add compatible per-lane subagent concurrency config while preserving the existing scalar `agents.defaults.subagents.maxConcurrent` behavior for the default lane. Keep the change focused on caller-selected subagent lanes; do not fold in per-agent caps from https://github.com/openclaw/openclaw/issues/8061 or per-parent caps from https://github.com/openclaw/openclaw/issues/73870 unless a tiny shared helper is unavoidable. Update schema/types/help/docs and focused regression tests, and add a changelog entry because this is user-visible config/tool behavior.
