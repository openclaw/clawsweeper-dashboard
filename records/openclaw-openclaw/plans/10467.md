# Coding Plan for openclaw/openclaw#10467: [Feature Request]: Multi-lane concurrency support for sub-agents via sessions_spawn

Source report: [records/openclaw-openclaw/items/10467.md](records/openclaw-openclaw/items/10467.md)

## Summary

Keep open. Current main still exposes `sessions_spawn` through one fixed `subagent` queue lane: there is no caller-selected lane in the tool schema, no native or ACP lane forwarding, and only scalar subagent concurrency config. The closed per-agent-lane PR addressed inbound agent runs, not this subagent-lane feature.

## Plan

Implement multi-lane subagent concurrency for https://github.com/openclaw/openclaw/issues/10467. Add a backward-compatible `sessions_spawn.lane` parameter that defaults to `subagent`, validates lane names, forwards the resolved lane through both native `spawnSubagentDirect` and ACP `spawnAcpDirect`, and applies per-lane subagent concurrency while preserving `agents.defaults.subagents.maxConcurrent` as the default `subagent` lane cap. Update strict config types/schema/help/docs and focused tests for schema exposure, native dispatch, ACP dispatch, default fallback, per-lane concurrency setup, and existing callers. Do not change unrelated per-agent inbound lane semantics from the closed unmerged PR; keep core plugin-agnostic and avoid broad scheduler redesign.
