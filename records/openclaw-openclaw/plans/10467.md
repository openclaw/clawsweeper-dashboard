# Coding Plan for openclaw/openclaw#10467: [Feature Request]: Multi-lane concurrency support for sub-agents via sessions_spawn

Source report: [records/openclaw-openclaw/items/10467.md](records/openclaw-openclaw/items/10467.md)

## Summary

Keep open. Current main and the latest release still expose `sessions_spawn` through one fixed `subagent` lane, with no caller-selected `lane` field or per-lane subagent concurrency config; adjacent open issues cover related per-agent or per-parent caps but do not supersede this specific request.

## Plan

Implement additive multi-lane support for `sessions_spawn` without changing existing callers. Add an optional validated `lane` string to the tool schema that defaults to `subagent`, pass the resolved lane through both native `SpawnSubagentParams` and ACP `SpawnAcpParams`, use it in the gateway `agent` call instead of hard-coded `AGENT_LANE_SUBAGENT`, and add compatible per-lane subagent concurrency config while preserving `agents.defaults.subagents.maxConcurrent` as the default cap for the existing `subagent` lane. Update strict config schema/types/help/default behavior, docs, and focused tests. Do not implement unrelated per-agent maxConcurrent or per-parent spawn caps from adjacent issues unless the minimal lane plumbing requires a shared helper. Add a changelog entry because this is user-visible config/tool behavior.
