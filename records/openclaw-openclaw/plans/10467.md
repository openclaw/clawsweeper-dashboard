# Coding Plan for openclaw/openclaw#10467: [Feature Request]: Multi-lane concurrency support for sub-agents via sessions_spawn

Source report: [records/openclaw-openclaw/items/10467.md](records/openclaw-openclaw/items/10467.md)

## Summary

Keep open. Current main still exposes `sessions_spawn` through one fixed `subagent` queue lane, with no caller-selected lane, no native/ACP lane forwarding, and only scalar subagent concurrency config; the recent issue comments and the closed per-agent-lane PR do not supersede this requested subagent-lane feature.

## Plan

Implement the feature requested by https://github.com/openclaw/openclaw/issues/10467: add an additive `sessions_spawn.lane` parameter for native and ACP spawned sessions. Preserve existing behavior by defaulting omitted/blank lane to `subagent`; validate lane names and reserved names deliberately; forward the resolved lane through `createSessionsSpawnTool`, `SpawnSubagentParams`/`spawnSubagentDirect`, and `SpawnAcpParams`/`spawnAcpDirect`; apply compatible per-lane subagent concurrency configuration without changing existing `agents.defaults.subagents.maxConcurrent` semantics for the default lane. Update strict config types/schema/help/generated docs, user docs, changelog, and focused regression tests. Do not fold in the separate per-target cap from https://github.com/openclaw/openclaw/issues/8061 or the per-parent throttle from https://github.com/openclaw/openclaw/issues/73870 unless maintainers explicitly ask; keep this PR scoped to caller-selected subagent lanes and per-lane caps.
