# Coding Plan for openclaw/openclaw#85721: Browser automation: chrome-devtools-mcp processes accumulate and never terminate (memory leak)

Source report: [records/openclaw-openclaw/items/85721.md](records/openclaw-openclaw/items/85721.md)

## Summary

Keep open: current main still routes browser-plugin Chrome MCP sessions through `npx chrome-devtools-mcp@latest` and relies on MCP `client.close()` for teardown, while the pinned MCP SDK only owns the direct stdio child and Chrome DevTools MCP can create detached watchdog descendants. The related configured-MCP and Codex app-server cleanup work does not cover this browser-plugin session lifecycle path.

## Plan

Fix the browser plugin Chrome MCP subprocess leak reported in https://github.com/openclaw/openclaw/issues/85721. Keep the change scoped to browser-plugin Chrome MCP session lifecycle: ensure every path that closes a Chrome MCP session, including cached sessions, temporary/ephemeral probes, failed handshakes, aborts, timeouts, stale-session reconnects, profile stop, and browser runtime shutdown, also tears down the spawned `npx chrome-devtools-mcp@latest` wrapper and descendant `chrome-devtools-mcp`/watchdog processes. Respect the extensions boundary: do not deep-import core `src/process/**` from plugin production code; either add a valid plugin-local helper or promote a generic SDK/runtime seam with tests. Do not make global installation of Chrome MCP a requirement, do not remove existing `mcpCommand`/`mcpArgs` support, and do not change unrelated browser routing behavior. Add focused regression tests around process cleanup and existing-session close paths, then provide live Linux proof with repeated browser open/snapshot/screenshot operations showing bounded `npx`, `chrome-devtools-mcp`, and watchdog process counts after session close and gateway shutdown.
