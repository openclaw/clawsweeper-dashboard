## Summary

Keep open: current main still has a source-confirmed restart-token failure mode that matches the reported logs. An unauthorized SIGUSR1 handler return does not clear the emitted restart token, and later `gateway.restart.request` calls can still hit the `already in-flight` coalescing branch.

## Plan

Fix https://github.com/openclaw/openclaw/issues/79577. Current main can leave `src/infra/restart.ts` with `emittedRestartToken > consumedRestartToken` when an update-run SIGUSR1 is rejected by `src/cli/gateway-cli/run-loop.ts`, causing later `gateway.restart.request` calls to coalesce as already in-flight. Add a small restart-state API or equivalent path that rolls back/clears a rejected emitted SIGUSR1 cycle when the handler decides the signal is unauthorized or unusable, without breaking legitimate duplicate coalescing, active deferral, or update-run fresh-process replacement. Split the misleading unauthorized log so `not authorized`, `commands.restart=false`, and gateway-tool guidance are accurate. Add regression coverage showing: an update.run-style emitted SIGUSR1 whose authorization is not consumed does not leave future gateway.restart requests coalesced forever; valid authorized update.run restarts still carry reason `update.run`; external SIGUSR1 behavior remains gated by `commands.restart`; package updates still use delayMs=0, skipDeferral, and skipCooldown. Add a user-facing changelog fix entry. Do not change release/version metadata, dependency overrides, or broad restart policy.
