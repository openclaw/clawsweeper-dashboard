## Summary

Keep open. Current main still does not put an idle body-read timeout on the remote PDF load path; the newer stuck-session recovery is a last-resort run abort path, not a bounded PDF fetch timeout, and the PDF tool does not consume the abort signal passed to tool execution.

## Plan

Fix https://github.com/openclaw/openclaw/issues/68649 by adding bounded idle body-read timeout plumbing for remote PDFs only. Use the existing `readIdleTimeoutMs` contract in `src/media/fetch.ts`/`readResponseWithLimit`; add an optional `readIdleTimeoutMs` field to `WebMediaOptions`, forward it from the remote URL branch in `src/media/web-media.ts`, and have `src/agents/tools/pdf-tool.ts` pass a conservative default such as 30_000 ms when loading PDFs. Preserve existing `maxBytes`, local roots, sandbox, SSRF, and native-provider behavior; do not rework slash-command recovery or diagnostics policy. Add/update regression tests proving the PDF tool forwards the timeout and WebMedia forwards it to `readRemoteMediaBuffer`; include a user-facing changelog entry if repo landing policy requires it.
