# Coding Plan for openclaw/openclaw#49524: memory_search can stall a live session when the memory manager/tool-result path does not fail open

Source report: [records/openclaw-openclaw/items/49524.md](records/openclaw-openclaw/items/49524.md)

## Summary

Keep open: current main and `v2026.5.22` still await memory manager setup and `memory.manager.search()` without a tool-boundary deadline, so a never-settling memory path can still withhold the tool result and stall a live turn.

## Plan

Fix https://github.com/openclaw/openclaw/issues/49524. In `extensions/memory-core/src/tools.ts` and/or `extensions/memory-core/src/tools.shared.ts`, add a narrow hard timeout around the full `memory_search` tool execution path that can hang before returning a tool result, covering manager acquisition (`getMemoryManagerContext`) and `memory.manager.search()`. On timeout or setup/search failure, return the existing `buildMemorySearchUnavailableResult(...)` shape with `disabled: true` and `unavailable: true`; do not add a user config knob or rewrite memory manager, QMD watcher, sync, provider embedding timeout, or ranking behavior unless strictly necessary. Add focused Vitest coverage in `extensions/memory-core/src/tools.test.ts` using fake timers or an equivalent deterministic never-settling promise to prove the tool returns an unavailable result after the timeout and that ordinary thrown errors still use the existing fail-open payload. Keep the change scoped to memory-core tool liveness. Add a changelog entry because this fixes user-visible live session liveness.
