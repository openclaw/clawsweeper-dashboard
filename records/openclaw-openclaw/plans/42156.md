# Coding Plan for openclaw/openclaw#42156: Telegram callback queries time out when agent turn is queued behind sequentialize

Source report: [records/openclaw-openclaw/items/42156.md](records/openclaw-openclaw/items/42156.md)

## Summary

Keep open. Current main and the latest release still acknowledge ordinary Telegram callback queries only inside the queued callback handler, while the related closed PRs were never merged and the open callback-ack-text PR does not move the acknowledgement before sequentialize.

## Plan

Fix https://github.com/openclaw/openclaw/issues/42156. Current main accepts/dedupes Telegram updates in `extensions/telegram/src/bot-core.ts`, then installs `sequentialize(getTelegramSequentialKey)`, and only later answers callback queries inside `extensions/telegram/src/bot-handlers.runtime.ts`; ordinary callbacks still share the active chat/topic lane in `extensions/telegram/src/sequential-key.ts`. Add a middleware after accepted-update begin/dedupe and before `botRuntime.sequentialize(getTelegramSequentialKey)` that fire-and-forget acknowledges accepted `callbackQuery` updates quickly. Do not call `shouldSkipUpdate(ctx)` in the early middleware in a way that consumes `activeHandledUpdateKeys` and makes the later callback handler skip; add a non-consuming helper if needed. Keep callback action routing and authorization in the existing ordered handler. Reuse `withTelegramApiErrorLogging`/`runtime` so errors stay sanitized, preserve the existing Telegram API throttler behavior, and coordinate with https://github.com/openclaw/openclaw/pull/76975 if that PR or an equivalent callbackAck payload change has landed. Add regression coverage proving ACK happens while a same-chat/topic turn is blocked behind the sequentializer, while the callback action still waits for ordering; include non-callback and ACK failure pass-through coverage. Add a changelog entry because this is user-visible Telegram behavior.
