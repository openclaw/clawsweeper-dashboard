## Summary

Current main and the latest release still acknowledge regular Telegram callback queries only inside the queued callback handler, after the per-chat/topic sequentialize middleware, so the timeout path remains a valid narrow Telegram plugin bug.

## Plan

Fix https://github.com/openclaw/openclaw/issues/42156 with a narrow Telegram plugin change. Acknowledge accepted Telegram callback queries before `botRuntime.sequentialize(getTelegramSequentialKey)` so inline-button spinners clear even while an agent turn is active in the same chat/topic. Keep callback action routing and authorization in the existing ordered `callback_query` handler. Preserve the update tracker/dedupe behavior, pass runtime error logging through the existing Telegram API logging helper, and address the callback-spam/rate-limit concern raised on https://github.com/openclaw/openclaw/pull/42516 without adding a new config option or moving broader callback UX work from https://github.com/openclaw/openclaw/issues/76622. Add focused regression coverage proving the acknowledgement middleware runs before the sequentializer and that non-callback/duplicate/error paths continue safely. Add a user-facing changelog entry for the Telegram fix.
