Keep open: this is a focused secrets bug fix with strong proof, but the new env-ref allow-list can still let a provider-indexed non-auth target delete a live provider auth env line when that target uses a known provider env var as its ref.

## Plan

Fix PR https://github.com/openclaw/openclaw/pull/94106 without broad refactors. The bug is that `src/secrets/apply.ts` currently adds any known env ref id to `scrubbedProviderEnvKeys`; for provider-indexed non-auth targets such as `models.providers.*.headers.*`, `request.auth.*`, and TLS targets, this can still scrub `OPENAI_API_KEY`/provider auth env lines even though provider auth was not migrated. Keep plugin/config known env refs like `FIRECRAWL_API_KEY` scrubbing, keep non-provider refs like `OPENCLAW_GATEWAY_TOKEN` preserved, and keep provider auth targets/auth-profile targets scrubbing their own provider env vars. Add a focused regression in `src/secrets/apply.envscrub-collision.repro.test.ts` where migrating only `models.providers.openai.headers.x-shared-header` to `{source:"env", provider:"default", id:"OPENAI_API_KEY"}` preserves `OPENAI_API_KEY` in `.env` when the value matches the migrated header. Do not change config surfaces, docs, changelog, or unrelated secret/audit behavior.
