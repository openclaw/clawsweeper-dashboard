# Coding Plan for openclaw/openclaw#65329: bug(msteams): DM inline images and file attachments silently dropped

Source report: [records/openclaw-openclaw/items/65329.md](records/openclaw-openclaw/items/65329.md)

## Summary

Keep open. Current main still leaves the MS Teams personal-DM media gap: non-HTML DM media has no Bot Framework or Graph fallback after direct download returns no media, and the caller passes a Graph-style conversation ID into the resolver.

## Plan

Fix https://github.com/openclaw/openclaw/issues/65329 on current main. Scope the change to bundled MS Teams personal-DM inbound media. Current main first calls `downloadMSTeamsAttachments()`, then skips both Bot Framework and Graph fallbacks when non-HTML DM media has no `<attachment>` ids; `message-handler.ts` also passes `graphConversationId` into the resolver, hiding the original Bot Framework DM id. Preserve both ids in the handoff, add a narrowly scoped fallback for real non-HTML media/file candidates after direct download fails, and verify the Teams/Bot Framework attachment contract before deriving attachment ids or rewriting `asm.skype.com`/`msteams://media` references. Do not remove the mention-only HTML guard from PR #64645, and do not broaden this into the separate `file.download.info` SharePoint `downloadUrl` rewrite from https://github.com/openclaw/openclaw/issues/67177 unless one minimal patch naturally covers both. Add focused regression coverage for non-HTML personal-DM media, original-id preservation, and mention-only skip behavior. Add a user-facing changelog entry.
