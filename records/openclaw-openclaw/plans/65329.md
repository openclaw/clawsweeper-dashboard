## Summary

Keep open. Current main still leaves the MS Teams personal-DM non-HTML media path without a Bot Framework fallback after direct download returns no media, and related open work explicitly leaves clipboard-paste DM images as a separate follow-up.

## Plan

Fix the MS Teams personal-DM inbound media gap reported in https://github.com/openclaw/openclaw/issues/65329. In `extensions/msteams/src/monitor-handler/message-handler.ts` and `extensions/msteams/src/monitor-handler/inbound-media.ts`, preserve the original Bot Framework conversation id separately from the Graph chat id so the resolver can choose the Bot Framework v3 attachment path for personal DMs. Add a narrowly scoped fallback for real non-HTML media/file candidates after `downloadMSTeamsAttachments()` returns no media; do not remove the `<attachment id=...>` guard for mention-only HTML messages, do not broaden token host allowlists, and do not add a core/plugin API. Add regression tests for a personal DM with a non-HTML inline image/media candidate where direct download returns empty and Bot Framework fallback is attempted, plus a mention-only HTML case that still skips fallback. Coordinate with the narrower SharePoint `file.download.info` issue at https://github.com/openclaw/openclaw/issues/67177 but do not turn this into a broad Teams SDK migration. Add a user-facing changelog fix entry if the repair branch changes runtime behavior.
