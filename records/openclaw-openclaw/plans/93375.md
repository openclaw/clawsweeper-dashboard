## Summary

Keep open: current main still has a source-backed path where Telegram's internal temporary-unhealthy transport cooldown is not classified as recoverable, so isolated polling can exit or churn under health-monitor restarts without surfacing a useful user-visible error.

## Plan

Fix https://github.com/openclaw/openclaw/issues/93375. Current source lets `TelegramTransportAttemptUnhealthyError` (`telegram transport attempt temporarily unhealthy; retry after ...`) fall outside `isRecoverableTelegramNetworkError`, so Telegram isolated `getUpdates`/setup paths can throw, exit with code 1, and rely on gateway health restarts while status/logs do not surface the transient failure clearly. Keep the fix inside the Telegram plugin/gateway channel lifecycle; do not add config/env options or broad health-monitor policy changes. Make the internal temporary-unhealthy cooldown a recoverable bounded retry/status path, ensure isolated worker poll errors are logged or reflected in channel status before health-monitor restart, and add regression coverage for the exact error string/path.
