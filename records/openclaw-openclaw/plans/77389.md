# Coding Plan for openclaw/openclaw#77389: Bug: Telegram progress draft \"Sifting\" persists and triggers at every step after restart

Source report: [records/openclaw-openclaw/items/77389.md](records/openclaw-openclaw/items/77389.md)

## Summary

Keep open: current main still keeps Telegram draft message IDs only in per-process stream state, so a restart can lose the ID needed to delete a stale progress or tool-progress preview. Current main covers same-process cleanup, and the related closed PR at https://github.com/openclaw/openclaw/pull/81486 addressed final-delivery cleanup, but neither adds persisted pending-draft cleanup across process restart.

## Plan

Fix https://github.com/openclaw/openclaw/issues/77389 by making Telegram progress/tool-progress draft cleanup survive OpenClaw process restarts. In the Telegram plugin, persist only unfinished preview/progress draft message IDs with enough context to safely delete them later, clear stale pending drafts on startup or the next dispatch, and remove the pending record when the draft is finalized, deleted, or intentionally discarded. Preserve finalized replies and do not scan or delete arbitrary old Telegram messages. Cover both progress mode and the partial-mode preview tool-progress path if they share the same draft lifecycle. Add focused restart-style regression coverage that simulates a sent draft ID from a previous process and verifies the next process deletes only that stale pending draft before/around final delivery. Add a user-facing changelog fix entry. Do not change Telegram auth, allowlist policy, or unrelated room-event behavior.
