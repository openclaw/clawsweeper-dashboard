## Summary

Keep open: current main has same-process Telegram progress/tool-progress draft cleanup, but the restart case remains source-visible because active draft message IDs are still held in per-process stream state and are lost across process restart. The related closed PR addressed final-delivery cleanup, not persisted cleanup of an unfinished draft after restart.

## Plan

Fix https://github.com/openclaw/openclaw/issues/77389. Telegram progress/tool-progress drafts currently keep active message IDs only in the createTelegramDraftStream closure, so a process restart before clear/final cleanup loses the ID and cannot delete the stale draft. Add a narrowly scoped Telegram-owned pending progress-draft registry/persistence keyed by the minimal account/chat/thread context needed, record only unfinished progress/tool-progress preview message IDs, clear those stale drafts on startup or before the next dispatch, and remove records once the draft is finalized, cleared, or safely abandoned. Preserve finalized answer messages and retained transcript-backed previews; do not delete arbitrary recent Telegram messages and do not add a new user config option. Add regression coverage that simulates restart/lost in-memory stream state plus the existing same-process final cleanup behavior. Include a changelog entry if the fix is user-visible.
