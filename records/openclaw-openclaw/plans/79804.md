## Summary

Keep open: current main and the latest release still allow session verbose mode to bypass Telegram default progress suppression, and current tests explicitly codify that leak while the docs and reply-option contract promise suppression for final-only/off mode. The narrow fix direction was proposed in closed, unmerged PRs, and the still-open adjacent PR proposes the opposite policy, so this issue remains the canonical bug to fix.

## Plan

Fix https://github.com/openclaw/openclaw/issues/79804 narrowly. Current main lets session verbose mode bypass `suppressDefaultToolProgressMessages`, which leaks default text-only tool/status progress into Telegram when `streaming.mode: "off"` or tool progress is disabled. Update `src/auto-reply/reply/dispatch-from-config.ts` so the suppression flag is authoritative for default text progress even when verbose is on, while preserving existing delivery for media-only tool payloads, exec-approval payloads, and errors. Update `src/auto-reply/reply/dispatch-from-config.test.ts` to replace the verbose-override expectations with suppression expectations for tool summaries, plan updates, and working-status progress. Do not implement the opposite behavior from https://github.com/openclaw/openclaw/pull/77211, and do not disable all Telegram preview progress like https://github.com/openclaw/openclaw/pull/78020. Check whether Telegram wiring in `extensions/telegram/src/bot-message-dispatch.ts` needs only regression coverage or a small propagation change for explicit `toolProgress: false`; keep the fix scoped. Validation should include targeted auto-reply and Telegram dispatch tests plus live or Crabbox Telegram proof showing no internal tool/status cards with verbose on and progress disabled; redact private data in proof.
