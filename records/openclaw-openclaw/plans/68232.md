## Summary

Keep open: current main and the latest shipped release still have the Telegram-local state-loss path. The restart-drain and polling-restart fixes reduce adjacent disruption, but a `channels.telegram.*` hot reload still restarts the channel and creates a fresh `groupHistories` map.

## Plan

Fix the Telegram hot-reload state-loss bug reported in https://github.com/openclaw/openclaw/issues/68232. Preserve or transfer bot-local pending group history across restart-required Telegram config reloads when the Telegram account/token/chat identity has not changed; scope the retained state by account id, token identity, chat id, and topic/thread, and trim retained entries to the new effective historyLimit. Keep full restart behavior for token/account/webhook identity changes that genuinely require a clean bot. Prefer a Telegram-owned state handoff through the bot creation/start-stop path over a broad core gateway cache, and do not add a user-facing config option unless maintainer review explicitly asks for one. Check overlap with https://github.com/openclaw/openclaw/pull/46303 before implementation, but do not try to solve its generic SIGUSR1/debounce queue work in this PR. Add focused regression tests proving that a Telegram reload preserves pending group history for the same identity and does not leak it across account/token/chat/topic changes. Add a changelog entry because the fix is user-visible.
