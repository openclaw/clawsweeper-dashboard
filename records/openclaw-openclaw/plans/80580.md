## Summary

Keep open: current main and the latest release still have the source-backed Docker setup path that matches the reported failure. The root cause is Docker BuildKit pulling the external Dockerfile frontend from the `syntax` parser directive, not the setup script executing a comment, but the WSL Docker setup timeout remains actionable.

## Plan

Fix https://github.com/openclaw/openclaw/issues/80580 as a focused Docker setup bug. Current main and v2026.5.7 still have `Dockerfile:1` set to `# syntax=docker/dockerfile:1.7`, and `scripts/docker/setup.sh` builds that file through `scripts/lib/docker-build.sh` with `DOCKER_BUILDKIT=1`; on WSL the reporter hit a Docker Hub frontend token-fetch timeout before the build body. Do not remove all comments or broadly remove BuildKit/cache mounts. Find the narrowest maintainable way to avoid or gracefully handle the mandatory external Dockerfile frontend pull on supported Docker versions while preserving `RUN --mount` behavior, cache layout, and the prior BuildKit fix for https://github.com/openclaw/openclaw/issues/56681. Add focused regression coverage around the setup/build helper behavior and update the active changelog if the user-visible Docker setup behavior changes. Validate with `pnpm test src/docker-setup.e2e.test.ts src/dockerfile.test.ts src/docker-build-cache.test.ts test/scripts/docker-build-helper.test.ts`, then run a real Docker smoke on Testbox or another clean Docker host, such as `docker build -t openclaw:local -f Dockerfile .` or `OPENCLAW_SKIP_ONBOARDING=1 ./scripts/docker/setup.sh`, redacting private host details from proof.
