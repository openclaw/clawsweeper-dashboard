## Summary

Keep open: current main still logs pre-connect Gateway WebSocket timeout and close paths without a client identity label, and I found no duplicate issue or open PR that already tracks this exact observability gap.

## Plan

Implement https://github.com/openclaw/openclaw/issues/79601 as a focused Gateway WebSocket diagnostics change. Current main logs preauth `handshake timeout` and `closed before connect` records with only conn/peer/remote/header context; add a sanitized, non-secret claimed client identity label that owned clients can provide before the first connect frame. Prefer a WebSocket-upgrade mechanism compatible with browser clients; do not require arbitrary browser headers. Include the label in open/timeout/error/closed-before-connect diagnostics and reconcile it with `connectParams.client.id` once a valid connect frame arrives. Treat the preauth label as spoofable logging metadata only: do not use it for auth, pairing, scopes, rate limits, trusted-proxy decisions, or device policy. Update the TS GatewayClient and Control UI browser client at minimum, evaluate native OpenClawKit/Android gateway clients if the carrier needs cross-client consistency, add focused regression tests for idle preauth timeout and early-close logging, and add protocol docs plus a user-facing changelog entry if behavior changes. Keep the patch scoped to Gateway WS diagnostics and client identity propagation.
