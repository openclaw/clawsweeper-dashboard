# Coding Plan for openclaw/openclaw#76218: Allow cron payloads to specify a stable prompt_cache_key independent of per-run sessionId

Source report: [records/openclaw-openclaw/items/76218.md](records/openclaw-openclaw/items/76218.md)

## Summary

Keep open. Current main still has no validated cron or agent-turn `promptCacheKey` payload field, isolated cron still creates a fresh per-run session id, and both OpenAI request builders still derive `prompt_cache_key` from that session id; the optional model-override diagnostic concern is already addressed.

## Plan

Implement stable prompt-cache-key overrides for cron agent-turn runs. Add a validated `promptCacheKey` cron/agent-turn payload field, normalize and persist it with the existing cron payload fields, forward it through isolated cron into the embedded agent stream options, and teach OpenAI Responses and opted-in OpenAI-compatible completions transports to use the explicit key when present and `cacheRetention` is not `none`, otherwise fall back to `sessionId`. Preserve isolated transcript/session semantics; do not make isolated cron reuse persistent sessions. Keep existing provider compatibility stripping/`compat.supportsPromptCacheKey` behavior intact, especially for native OpenAI-Codex or other endpoints that should not receive the field. Update public docs and CHANGELOG for the cron usage pattern. Stop and ask if the implementation would require a broader cache-key policy beyond the explicit field.
