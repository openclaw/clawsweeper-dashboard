# Coding Plan for openclaw/openclaw#79847: qmd-manager leaks XDG_CONFIG_HOME / XDG_CACHE_HOME to all child spawns, breaking mcporter ≥ 0.10 integration

Source report: [records/openclaw-openclaw/items/79847.md](records/openclaw-openclaw/items/79847.md)

## Summary

Keep open: current main still passes qmd-scoped XDG variables into mcporter, while mcporter v0.10.2 intentionally uses XDG_CONFIG_HOME for config discovery, so this is a valid current-main integration bug rather than stale or external-only support.

## Plan

Fix https://github.com/openclaw/openclaw/issues/79847. Current qmd-manager builds one env containing agent-scoped XDG_CONFIG_HOME, QMD_CONFIG_DIR, and XDG_CACHE_HOME, then passes it to both qmd and mcporter. With mcporter v0.10.2, that makes mcporter look for mcporter.json under the agent qmd xdg-config directory and fail with Unknown MCP server 'qmd'. In extensions/memory-core/src/memory/qmd-manager.ts, separate the qmd process environment from the mcporter process environment so mcporter discovers its own config normally. Preserve qmd per-agent state for direct qmd commands and be careful not to break local qmd MCP keep-alive setups that still need qmd-specific env. Do not add a new user config option unless the existing mcporter/qmd env contracts make a code-only fix impossible; stop and report if that happens. Update extensions/memory-core/src/memory/qmd-manager.test.ts so mcporter no longer receives the qmd-scoped XDG_CONFIG_HOME, while qmd still receives the qmd state env. Add or update a changelog entry because this is user-visible. Use mcporter v0.10.2 XDG behavior as the dependency contract.
