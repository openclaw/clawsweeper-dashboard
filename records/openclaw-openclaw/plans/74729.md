## Summary

Keep open. Current main still advertises Feishu `read`, but the Feishu handler only accepts `messageId` and fetches one message; the broader formatter pagination issue at https://github.com/openclaw/openclaw/issues/71452 and its open PR do not cover this Feishu provider wrapper gap.

## Plan

Implement Feishu paginated chat-history reads for this issue. Preserve the existing single-message `messageId` behavior, then add a Feishu-owned `read` path for `chatId`/target chat history using the Feishu `im.message.list` API with `container_id_type: "chat"`; support bounded `limit`, `pageToken`/`page_token`, and before/after time filters after confirming the exact Feishu SDK/API parameter names and scope requirements. Return normalized `{ messages, hasMore, pageToken }` style details compatible with the shared message formatter, and update Feishu docs/tests. Do not duplicate the shared formatter-display work tracked by https://github.com/openclaw/openclaw/pull/79593; touch shared target alias/normalization only if Feishu `chatId` cannot otherwise satisfy `read` validation. Add a changelog entry if this lands as a user-visible plugin feature.
