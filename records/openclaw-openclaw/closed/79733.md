## Summary

Close: current main already drops WhatsApp inbound updates that lack user-visible content before access control, enqueueing, or group context assembly. The proof commit is on main only and is not contained in the latest release tag v2026.5.7.

## What This Changes

Filter WhatsApp group join, leave, and admin system events before they enter agent-visible group context.

## Best Possible Solution

Keep the current main content gate and carry it into the next WhatsApp plugin release; reopen only with a concrete system-event payload that still reaches `WebInboundMsg`.

## Reproduction Assessment

No failing current-main reproduction path remains. Source inspection shows WhatsApp group system events are Baileys stub metadata outside user message content, and current main drops updates without extractable user content before agent context assembly.

## Solution Assessment

Yes. Filtering at the WhatsApp platform ingress by positive user-content detection is narrower and more maintainable than enumerating every join/leave/admin stub type downstream.
