## Summary

Keep open: this PR targets a real Windows Gmail setup crash, but the current head still misses the documented CLI `gog` path, can direct-spawn resolved `.cmd` shims, executes bare `where.exe` during lookup, and adds an unrelated invalid SDK export.

## What This Changes

The PR adds a Windows-only resolver for `gcloud`, `gog`, and `tailscale` helper commands in Gmail setup/watcher code, with resolver tests and an SDK barrel export.

## Best Possible Solution

Land a focused Windows Gmail helper fix that uses a non-executing PATH/PATHEXT resolver, covers setup/run/gateway `gcloud`, `gog`, and `tailscale` paths, uses the shared Windows-safe process seam for `.cmd`/`.bat`, removes unrelated SDK exposure, and adds the required changelog entry.

## Reproduction Assessment

Yes. The linked bug gives concrete native Windows PowerShell steps, and current main still has bare helper invocations in the documented Gmail setup/run paths; I did not run a Windows live repro in this read-only review.

## Solution Assessment

No. The resolver direction is reasonable, but this PR is not the best fix until it covers the CLI `gmail-ops` path, avoids direct `.cmd` spawn and bare `where.exe`, removes the invalid SDK export, and records the user-facing fix in the changelog.
