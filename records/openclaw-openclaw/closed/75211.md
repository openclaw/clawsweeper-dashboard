## Summary

Keep open: current main still has the reported WhatsApp routing gap, with sender identity passed as `peer.id`, no WhatsApp configured-binding provider hooks, and no runtime warning for a phone-shaped `match.accountId` that is likely meant to be a sender. The related diagnostic/parity PR was closed unmerged, and the open ACP PR overlaps but is still blocked by security/proof concerns rather than superseding this issue.

## What This Changes

The issue asks WhatsApp to honor route bindings or warn when WhatsApp binding rules cannot be enforced, especially for phone-number-shaped `accountId` configs.

## Best Possible Solution

Land a maintainer-approved WhatsApp routing diagnostic and, if provider parity is desired, wire WhatsApp inbound through the configured-binding route contract with regression tests.

## Reproduction Assessment

Yes. Source inspection on current main shows the reporter's phone-number `match.accountId` is evaluated as a connected-account selector while WhatsApp inbound supplies the sender as `peer.id`, and there is no WhatsApp configured-binding provider or warning path.

## Solution Assessment

Unclear as a final product direction. A diagnostic is the narrowest low-risk fix for the reported `accountId` misuse, while configured-binding provider parity should only land with inbound routing and security regression coverage.
