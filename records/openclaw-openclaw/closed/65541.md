## Summary

Keep this PR open, but not merge-ready: current main still intentionally treats disabled block streaming as final-only for text, the PR branch is dirty against main, and the proposed fallback can suppress the final text after only queueing same-channel delivery.

## What This Changes

The PR changes disabled block-streaming reply delivery to send text and media block payloads immediately and updates reply-delivery tests to expect direct text delivery.

## Best Possible Solution

If maintainers want progressive delivery while `blockStreamingDefault` is off, update the handler, docs, changelog, final-payload dedupe, and same-channel dispatcher confirmation/fallback semantics together. If final-only is still the intended contract, keep the current implementation and close this PR with #32868 left as the canonical same-channel progressive-delivery issue.

## Reproduction Assessment

Yes for current-main behavior: call `createBlockReplyDeliveryHandler` with `blockStreamingEnabled: false`, no pipeline, and a text-only payload; current tests assert `onBlockReply` is not called. The live Discord claim was not re-run in this read-only review.

## Solution Assessment

No, not as-is. The PR is narrow, but it changes documented disabled-streaming semantics and can suppress the final fallback after same-channel delivery has only been queued; a safer fix needs maintainer-approved contract changes plus confirmed-delivery or fallback behavior.
