## Summary

Current `main` now treats local loopback CDP control as outside browser SSRF policy, retries/adopts pre-existing loopback Chrome sessions, and has regression tests covering the exact attach-only/manual-CDP symptoms from this report (`running: true`, tabs visible, no false "not running" state). That is strong enough to treat this issue as implemented on `main`, though I could not tie it to a shipped release from the available checkout history.

## Evidence

- **Loopback CDP health checks bypass browser SSRF for managed local profiles:** `resolveCdpReachabilityPolicy()` returns `undefined` for non-remote loopback `openclaw` profiles, with an inline comment explaining that OpenClaw's own local CDP control plane should not self-block on `127.0.0.1`.
