## Summary

Keep this issue open. Current main has not implemented the requested diagnostic alias handling: the read-tool start guard still checks only `path` and `file_path`, related implementation PRs were closed unmerged, and nearby tests do not cover `file` or `filePath`. There is also a broader contract mismatch to resolve because current runtime validation requires canonical `path`, while some display/fingerprint paths still recognize aliases.

## Best Possible Solution

Keep the issue as the canonical tracker. The best fix is to align the read-tool contract in one small agents patch: either intentionally support the aliases end to end and update the diagnostic guard plus tests for `path`, `file_path`, `file`, and `filePath`, or keep canonical `path` only and remove/adjust stale alias assumptions in diagnostics/display/tests with a maintainer product decision.
