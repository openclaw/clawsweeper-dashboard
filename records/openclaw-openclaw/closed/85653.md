## Summary

Keep open: current main and the latest release cover the earlier Feishu native-topic `thread_id` fixes, but this report describes a distinct reply-context shape where `chat_type` is reported as `group` and the canonical `omt_*` value appears outside the current session-routing inputs. Source inspection shows that shape can still fall onto the intentional normal-group `root_id` path, so this needs maintainer/live-payload follow-up rather than cleanup closure.

## What This Changes

The issue reports Feishu/Lark `group_topic` sessions splitting when reply-context messages expose an `om_*` reply target instead of consistently routing by the `omt_*` topic id.

## Best Possible Solution

Confirm a redacted raw Feishu `im.message.receive_v1` starter/reply pair on current release, then update the Feishu plugin to normalize native topic reply-context events to the canonical `omt_*` key before session routing while preserving normal group `om_*` reply-root behavior.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection gives a plausible reproduction for the reported field shape: a topic reply normalized as `chat_type: "group"` with an `om_*` root/reply target can still route by `root_id` instead of the canonical `omt_*` topic id.

## Solution Assessment

Unclear as a code change without live payload proof. The current `topic_group` `thread_id` hydration path is the right fix for prior reports, but expanding topic detection for `chat_type: "group"` needs maintainer confirmation so normal group reply-created threads keep their documented `om_*` behavior.
