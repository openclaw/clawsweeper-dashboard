## Summary

Keep open. Current main still does not implement parsed session-entry reuse: session sync can track deltas, but selected transcripts still go through the full parser. The PR remains useful source work, but it is not mergeable as-is because it targets stale memory paths and the proposed fast path has concrete correctness and transcript-retention blockers.

## What This Changes

The PR branch adds a previous-entry append-only fast path to session transcript parsing, stores parsed session entries during session sync, and adds an append-only reuse regression test.

## Best Possible Solution

Replace or rebase the optimization onto `packages/memory-host-sdk/src/host/session-files.ts` and `extensions/memory-core/src/memory/manager-sync-ops.ts`, with byte-level prefix validation, a one-read fallback to the existing full parser, bounded cache lifecycle, and regression coverage for ASCII append, non-ASCII append, rewritten-prefix fallback, line maps, timestamps, redaction, and transcript classification.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Verify the cached prefix before append reuse:** `src/memory/session-files.ts:125-130`
  - body: The fast path only checks that the file is at least as large as the cached entry before reusing `previous.content`. If an earlier JSONL record is rewritten or compacted and the file remains the same size or grows, the returned entry can keep stale historical content; validate the cached prefix byte-for-byte or fall back to full parsing.
  - confidence: 0.91
- **[P1] Slice append data by byte offset:** `src/memory/session-files.ts:127-129`
  - body: `previous.size` comes from `fs.stat` in bytes, but the patch uses it as a JavaScript string index in `raw.slice(...)`. Non-ASCII transcripts can put the append boundary at the wrong character, dropping or duplicating parsed messages and producing an incorrect `lineMap`.
  - confidence: 0.94
- **[P1] Bound the session entry cache:** `src/memory/manager-sync-ops.ts:149`
  - body: The new `sessionEntryCache` stores full `SessionFileEntry` objects, including transcript `content`, and there is no eviction or cleanup when sessions are pruned or indexes reset. A long-lived gateway that sees many transcripts can retain those strings indefinitely and grow heap usage over time.
  - confidence: 0.88
- **[P2] Reuse the first file read on fallback:** `src/memory/session-files.ts:123-150`
  - body: When the fast path reads the file and then fails its guard, the fallback reads the same transcript again. That makes warm-cache fallback slower than the original path; share the first read with the fallback or structure the byte-level parser so failed fast-path attempts do not double-read.
  - confidence: 0.82
