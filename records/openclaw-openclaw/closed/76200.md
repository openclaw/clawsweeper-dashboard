## Summary

Current main has removed the 2026.4.x hot-path bundled runtime-deps staging model that this report is about, replacing it with build/package-time bundled runtime surfaces plus explicit install/update/doctor dependency repair. The fix is on main only in the local evidence I could verify, not in v2026.4.29.

## What This Changes

The issue reports a 2026.4.29 regression where bundled plugin runtime-deps staging misses its warm cache on Pi-class hardware and blocks embedded-run spawns for seconds per plugin.

## Best Possible Solution

Keep the current main direction: dependency work belongs to install/update/doctor and package build surfaces, not a new per-embedded-run runtime cache layer.

## Reproduction Assessment

Yes for the affected release: the report gives concrete v2026.4.29 Pi 400 steps and `[plugins]` timing evidence. For current main, the reproduction no longer matches the code path because the old runtime-deps staging symbols are absent and the docs/tests assert no package-manager repair on startup or provider-policy loads.

## Solution Assessment

Yes. Removing hot-path runtime package-manager staging is narrower and more maintainable than adding another cache around the old staging loop.
