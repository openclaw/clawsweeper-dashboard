## Summary

Keep this open: current main still does not register Google Meet realtime provider API keys as plugin SecretRef inputs, while the realtime path reads those provider configs and strict secret readers still throw the reported unresolved SecretRef error.

## What This Changes

The issue reports that Google Meet realtime join/create and gateway tool paths fail when provider API keys are configured as exec:onepassword SecretRefs.

## Best Possible Solution

Add Google Meet realtime provider API-key paths to the existing plugin SecretRef contract and cover the CLI/gateway realtime path with focused regression tests.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: Google Meet realtime reads provider configs from plugins.entries.google-meet.config.realtime.providers, but the manifest does not declare those API-key paths as SecretRef inputs. I did not run a live Google Meet/1Password reproduction in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to use the existing plugin configContracts.secretInputs machinery for Google Meet and add regression coverage, rather than adding provider-local fallbacks or a new config option.
