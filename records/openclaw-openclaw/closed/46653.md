## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Verify tunnel ownership before signaling stored PIDs:** `src/cli/tunnel-cli.ts:291`
  - body: `tunnel down` trusts a persisted numeric PID and sends SIGTERM/SIGKILL without proving the process is still the SSH tunnel OpenClaw started. After the original tunnel exits and the OS reuses that PID, this can kill an unrelated user process and `status` will also report stale state as alive.
  - confidence: 0.93
- **[P1] Reject occupied ports before reporting tunnel readiness:** `src/cli/tunnel-cli.ts:222-224`
  - body: The startup loop treats any `EADDRINUSE` on the local port as success, so an existing gateway or other listener can make `tunnel up` write active state while the SSH child exits because `ExitOnForwardFailure=yes`. Check port availability before spawn or race a child-tied listener/connect probe against SSH exit.
  - confidence: 0.91
- **[P1] Share the existing SSH tunnel startup contract:** `src/cli/tunnel-cli.ts:169-200`
  - body: Current main already has `startSshPortForward` for target parsing, port availability, child-exit readiness, stderr capture, and stop behavior. This PR hand-rolls a parallel SSH startup path and already diverges in readiness and diagnostics, so future security and platform fixes will drift unless the persistent mode is extracted from the shared helper.
  - confidence: 0.86
- **[P2] Create the state directory before writing tunnel metadata:** `src/cli/tunnel-cli.ts:38-39`
  - body: `writeTunnelState` writes into `STATE_DIR` without ensuring it exists. On a fresh profile or new `OPENCLAW_STATE_DIR`, the SSH child has already been detached, so `ENOENT` leaves a running tunnel with no PID file for `down` or `status` to manage.
  - confidence: 0.9
- **[P2] Resolve the SSH executable through a shared platform-safe path:** `src/cli/tunnel-cli.ts:200`
  - body: The new command hard-codes `/usr/bin/ssh`, which is already tracked as broken for Windows OpenSSH and can also emit an unhandled spawn error when that path is missing. Reuse the planned shared SSH executable resolver instead of adding a second hard-coded SSH process surface.
  - confidence: 0.84
