## Label Justifications

- `P2`: The reported bug affects a bounded but user-visible background media completion workflow, and the fix is already shipped in the latest release.
- `impact:message-loss`: The reported failure mode is a generated media completion message not being delivered back to the requester session.

## Summary

Current main and the latest shipped tag already satisfy the reported scope requirement: omitted `syntheticScopes` now produce a synthetic operator client with `operator.write`, so the media completion announce path no longer loses the required scope.

## What This Changes

The issue reports that background media generation can save media files but fail to announce completion back to the requester session because in-process `agent` dispatch lacks `operator.write`.

## Best Possible Solution

Keep the centralized write-scoped synthetic operator default and ask users on 2026.5.20 to upgrade to v2026.5.22 or newer rather than adding a media-only special case.

## Reproduction Assessment

No current-main failure path remains: source inspection shows `runAnnounceAgentCall` omits `syntheticScopes`, but the current in-process dispatcher gives that synthetic client `operator.write`. The old-release log is coherent, but it no longer matches current main or v2026.5.22 source.

## Solution Assessment

Yes. The current fix at the gateway synthetic-client boundary is the narrow maintainable solution because it covers this announce path and other ordinary in-process write dispatches without adding a media-specific workaround.
