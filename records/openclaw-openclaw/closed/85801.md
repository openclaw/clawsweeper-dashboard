## Label Justifications

- `P2`: The reported failure affects agent turn completion, but current main and shipped releases already contain a bounded fix with limited blast radius.
- `impact:message-loss`: The report is about the assistant response being suppressed after a successful large tool result.

## Summary

Current main already covers the reported failure mode with a generic live tool-result cap, model-facing truncation, recovery logging, schema metadata, and docs; the implemented path differs from the suggested key name but solves the oversized-output problem at the right agent boundary.

## What This Changes

The issue asks for configurable truncation, a clear truncation marker, user-visible logging, and docs/schema coverage when tool output is too large for the next agent turn.

## Best Possible Solution

Keep the shipped generic `agents.defaults.contextLimits.toolResultMaxChars` implementation and ask for a fresh report only if the same failure reproduces on `v2026.5.20` or newer with truncation/context-overflow logs.

## Reproduction Assessment

No current-main failing reproduction was established. Source and focused tests show the reported path is now bounded before persistence and before the next model prompt, with truncation notices and recovery logs.

## Solution Assessment

Yes. The existing generic agent context-limit knob is a more maintainable boundary than a separate `tools.maxOutputChars` key, because it applies across live tool-result persistence, model-facing context, and overflow recovery.
