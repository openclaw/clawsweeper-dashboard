## Label Justifications

- `P1`: The proposed fix leaves a silent commitment-message loss workflow affecting real users reproducible after a suppressed delivery and the following heartbeat.
- `merge-risk: 🚨 message-delivery`: Merging the status-only guard can still prevent retry by recording a false successful-delivery dedupe marker.

## Summary

This PR targets the correct status transition, but it does not yet fix the full message-loss path: a suppressed send is still recorded as the last delivered heartbeat and reported as sent, so the following identical heartbeat can dismiss the pending commitment as a duplicate without retrying it. The submitted evidence is also test-only and does not directly exercise a suppressed result.

## What This Changes

The PR gates heartbeat commitment status updates so suppressed sends remain pending rather than becoming sent, while preserving dismissal for model-skipped responses.

## Best Possible Solution

Treat suppressed as non-delivery across the complete post-send transition: leave commitments pending, do not record delivered-heartbeat dedupe fields, emit an accurate non-sent outcome, and prove with a two-cycle test that the next heartbeat retries.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to return suppressed on the first send and produce the same heartbeat text on the next cycle; the stored dedupe marker then dismisses the still-pending commitment without another send attempt.

## Solution Assessment

No. Gating only markCommitmentsStatus misses the adjacent delivered-heartbeat dedupe and event updates, so it does not deliver the claimed retry behavior.
