## Summary

Close as implemented: current main and the shipped v2026.4.27 release strip stale trailing assistant replay turns before Anthropic provider submission, which addresses the reported assistant-tail systemEvent 400. The issue discussion’s cron-to-systemEvent diagnosis still matches the code path, but the provider-boundary sanitizer now covers that path and other systemEvent sources.

## What This Changes

Ensure main-session cron/systemEvent wakeups do not send Anthropic request histories that end with an assistant turn when the stored session transcript already ends on assistant output.

## Best Possible Solution

Keep the shipped provider-boundary replay sanitizer as the durable fix, since it handles cron systemEvents and other runtime-context short circuits without adding cron-specific placeholder turns.

## Reproduction Assessment

Yes. The report gives a deterministic cron/main-session assistant-tail path, and current regression coverage reproduces the provider-facing core shape by submitting Anthropic replay history ending in an assistant turn.

## Solution Assessment

Yes. Stripping stale assistant tails at the Anthropic replay boundary is narrower and more complete than a cron-only placeholder because it covers every caller that can reach the same provider request shape while preserving assistant turns that contain replay tool calls.
