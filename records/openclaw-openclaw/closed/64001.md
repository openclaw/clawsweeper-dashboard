## Summary

Current main already implements the QMD startup hardening needed for this v2026.4.8 ENOENT report: explicit `memory.qmd.command` paths are preserved, missing agent workspaces are created before QMD availability probes, and QMD subprocesses get a service-safe PATH containing the current Node binary directory. The changelog records this under 2026.4.12.

## Best Possible Solution

Close this issue as shipped. Keep the current QMD startup/probe behavior and ask users still seeing the failure on 2026.4.12 or newer to file a fresh report with current memory status, gateway logs, config, and affected service environment details.
