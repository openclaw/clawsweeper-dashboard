## Summary

Close: current main still has no Feishu inbound read-receipt send path, but the supported SDK/API surface I found only exposes read-status queries for bot-sent messages, not a mutation to mark inbound user DMs as read. The same request was also reviewed and closed as not planned in https://github.com/openclaw/openclaw/issues/79277 after checking the upstream contract.

## What This Changes

Request Feishu channel support for DM read receipts by marking inbound messages read automatically or adding `channels.feishu.sendReadReceipts`.

## Best Possible Solution

Keep the current Feishu typing-indicator feedback path and reconsider DM read receipts only if Feishu/Lark documents a supported mark-inbound-read API for bot messages.

## Reproduction Assessment

Yes at source level: current main’s Feishu receive path parses, dedupes, and enqueues inbound messages without any mark-read/read-receipt call or Feishu config option. I did not run a live Feishu tenant reproduction in this read-only review.

## Solution Assessment

No: adding a generic request to an undocumented mark-read endpoint is not the best maintainable source change. The safer direction is to avoid a Feishu read-receipt feature until the upstream platform documents the mutation, scopes, and rate limits.
