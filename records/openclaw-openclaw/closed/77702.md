- `impact:message-loss`: Telegram attachments can be suppressed or converted into literal text instead of delivered as media.

## Summary

Keep open: current main still has a source-backed Telegram dispatch gap where raw `MEDIA:` text can be classified and delivered as text before the shared outbound directive parser converts it to media, and this issue is the canonical open tracker for that symptom.

## What This Changes

Telegram local-path `MEDIA:` directives in assistant replies should be delivered as Telegram attachments instead of visible literal text.

## Best Possible Solution

Normalize Telegram reply payloads through the shared outbound plan before preview/text-only decisions, then add focused dispatch coverage for standalone local `MEDIA:` final replies.

## Reproduction Assessment

Yes at source level: current Telegram dispatch can route raw `MEDIA:` text through the text lane before the shared directive parser turns it into `mediaUrls`. I did not run a live Telegram bot in this read-only review.

## Solution Assessment

Yes. Reusing the existing shared outbound payload planner at the Telegram classification boundary is the narrowest maintainable fix; a Telegram-specific `MEDIA:` parser would duplicate policy and drift from core normalization.
