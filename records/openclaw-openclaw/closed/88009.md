## Summary

Keep open: current main and the latest release still batch memory embeddings inside each indexed file, so the reported cross-file batching inefficiency remains real and source-reproducible.

## What This Changes

The issue asks memory indexing to submit batch embedding jobs across dirty files instead of per file, with less aggressive provider batch polling.

## Best Possible Solution

Move provider batch submission to the source-sync level so dirty chunks from multiple files can fill provider batch jobs up to provider limits, while preserving existing config and per-file index writes.

## Reproduction Assessment

Yes, source-reproducible without live credentials: current sync calls indexFile per dirty file, and the batch provider only receives that file's chunks. The reporter's repeated one-file batch log matches that path.

## Solution Assessment

Yes, batching across dirty files at the memory sync layer is the maintainable direction. The poll concern can likely use the existing remote.batch.pollIntervalMs knob first, with adaptive backoff added only if the fix includes focused tests.
