## Summary

Keep open: the PR addresses a real Discord SecretRef crash path, but the current diff can make an explicit configured token look absent and fall through to `DISCORD_BOT_TOKEN`, weakening credential precedence and fail-closed behavior.

## What This Changes

The PR changes Discord token normalization and tests so unresolved SecretRef-backed Discord tokens return no token instead of throwing.

## Best Possible Solution

Preserve a distinct configured-unavailable token state for read-only status and health paths while keeping Discord startup/execution fail-closed and config-over-env precedence intact.

## Reproduction Assessment

Yes, by source inspection. Current main sends unresolved SecretRefs through strict `normalizeResolvedSecretInputString`, and the PR's fallback problem is visible because `undefined` config tokens continue into default-account env fallback.

## Solution Assessment

No. Returning `undefined` from the shared normalizer prevents one throw path but loses the configured-versus-missing distinction; a tagged or inspect-mode resolution path is the narrower maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve configured SecretRefs before env fallback:** `extensions/discord/src/token.ts:16-17`
  - body: Returning `undefined` for a SecretRef makes `channels.discord.token` look absent, so the default-account resolver can continue to `DISCORD_BOT_TOKEN`. That violates documented config-over-env precedence and can run the wrong Discord bot when the configured ref is unavailable; carry a configured-unavailable state instead.
