## Label Justifications

- `P2`: The report concerns memory recall correctness with limited blast radius, and current main/release already prevent the unsafe vector comparison.
- `impact:session-state`: The affected surface is memory index identity and recall state, where stale or mismatched embeddings could otherwise distort session/memory search results.
- `impact:auth-provider`: The trigger is embedding provider fallback/model selection, which is part of provider routing and configured embedding identity.

## Summary

Current main and the latest release already prevent the reported unsafe behavior: memory search checks the stored index identity against the active provider/model/provider settings before vector search, and the tool/CLI surface an explicit paused-index warning instead of returning cross-space vector rankings.

## What This Changes

The issue reports that query-time `memorySearch.fallback` can silently compare query embeddings from a fallback provider against an index built by a different embedding model, producing meaningless vector rankings instead of a safe degraded mode.

## Best Possible Solution

Keep the shipped identity guard and paused-index warning; broader model-aware multi-index failover can remain in the separate canonical work at https://github.com/openclaw/openclaw/issues/63990.

## Reproduction Assessment

No: the reported garbage vector-ranking path does not reproduce against current main from source inspection. Current search refreshes index identity after fallback and exits before vector search when the fallback provider/model does not match the stored index metadata.

## Solution Assessment

Yes for the central correctness issue: the current implementation refuses incompatible vector search and surfaces a paused-index warning rather than mixing vector spaces. Returning FTS-only results during fallback would be a separate recall-policy enhancement, not needed to close this unsafe-results report.
