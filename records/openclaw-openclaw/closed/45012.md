## Summary

Current `main` already addresses the reported 2026.3.12 pairing/UI failure well enough to close: local loopback `openclaw devices list` / `approve` now fall back to local pairing state when the gateway rejects with pairing-required, and the Control UI/docs now explicitly direct remote plain-HTTP access to HTTPS or localhost-secure flows.

## Evidence

## Risks / Open Questions

- The issue mixed two behaviors: a real local-loopback CLI regression and a remote plain-HTTP Control UI request that remains intentionally restricted for security. Closing assumes the current fix plus clarified docs are sufficient for the reported problem.
- I could not verify a released tag containing this change from the available shallow history, so this appears fixed on `main` but not yet attributable to a shipped release tag.
