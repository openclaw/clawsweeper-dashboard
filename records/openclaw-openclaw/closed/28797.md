## Summary

Current `main` already has tool lifecycle/progress event hooks and channel renderers that reuse the live preview message for Discord, Slack, Telegram, and Mattermost. The changelog records this as shipped in `v2026.4.21`.

## Evidence

- **Runtime emits tool lifecycle/progress events:** `src/agents/pi-embedded-subscribe.handlers.tools.ts:609` emits `stream:"tool"` with `phase:"start"`; `src/agents/pi-embedded-subscribe.handlers.tools.ts:716` emits `phase:"update"`; completion is tracked with `phase:"end"` item events and success/failure status in the same handler around `src/agents/pi-embedded-subscribe.handlers.tools.ts:901`.
  - file: [src/agents/pi-embedded-subscribe.handlers.tools.ts:609](https://github.com/openclaw/openclaw/blob/d3f6783b16ecc10c67b101bf019ca4294453206a/src/agents/pi-embedded-subscribe.handlers.tools.ts#L609)
- **Reply pipeline exposes hook callbacks for tool/item/progress state:** `src/auto-reply/reply/agent-runner-execution.ts:1095` forwards tool events into `onToolStart`, `onItemEvent`, `onPlanUpdate`, `onApprovalEvent`, `onCommandOutput`, and `onPatchSummary`, which provides the lifecycle extension surface requested by the issue.
  - file: [src/auto-reply/reply/agent-runner-execution.ts:1095](https://github.com/openclaw/openclaw/blob/d3f6783b16ecc10c67b101bf019ca4294453206a/src/auto-reply/reply/agent-runner-execution.ts#L1095)
- **Default channel path suppresses separate tool messages when preview progress is enabled:** `src/auto-reply/reply/dispatch-from-config.ts:894` honors `suppressDefaultToolProgressMessages`, so channels can replace standalone tool messages with edited preview/status UX.
  - file: [src/auto-reply/reply/dispatch-from-config.ts:894](https://github.com/openclaw/openclaw/blob/d3f6783b16ecc10c67b101bf019ca4294453206a/src/auto-reply/reply/dispatch-from-config.ts#L894)
- **Channels/docs implement same-message live tool progress:** `src/plugin-sdk/channel-streaming.ts:119` defines `streaming.preview.toolProgress`; `extensions/discord/src/monitor/message-handler.process.ts:977` uses it to push tool progress into the preview message; `docs/concepts/streaming.md:172` documents tool-progress preview updates for Discord, Slack, Telegram, and Mattermost.
  - file: [docs/concepts/streaming.md:172](https://github.com/openclaw/openclaw/blob/d3f6783b16ecc10c67b101bf019ca4294453206a/docs/concepts/streaming.md#L172)
- **Shipped release note:** `CHANGELOG.md:279` starts the `2026.4.21` section, and `CHANGELOG.md:288` records: channel preview streaming now streams tool-progress updates into live preview edits for Discord, Slack, and Telegram.
  - file: [CHANGELOG.md:279](https://github.com/openclaw/openclaw/blob/d3f6783b16ecc10c67b101bf019ca4294453206a/CHANGELOG.md#L279)

## Risks / Open Questions

- The implementation does not use the exact proposed event names `tool.start` / `tool.end` / `tool.progress`; instead it uses `stream:"tool"` with phases plus richer `item`/`command_output` events. Functionally this covers the request, but exact naming differs.

## Close Comment

Closing as implemented. Codex review against current `main` found the requested tool lifecycle/progress surface already in place: runtime tool events are emitted from `src/agents/pi-embedded-subscribe.handlers.tools.ts:609` and forwarded through reply hooks in `src/auto-reply/reply/agent-runner-execution.ts:1095`, while channels can render live tool progress by reusing the same preview message via `streaming.preview.toolProgress` (`src/plugin-sdk/channel-streaming.ts:119`). The current behavior is documented in `docs/concepts/streaming.md:172`, and the changelog records it as shipped in `v2026.4.21` at `CHANGELOG.md:288`. The current implementation uses a `stream:"tool"` + phase model and related item/command events rather than the exact proposed event names, but it covers the feature request well enough on `main`.
