## Summary

Keep open. Current `main` still has no code path that interprets a literal `env:MINIMAX_API_KEY` in `messages.tts.providers.minimax.apiKey` as an env-backed secret. The MiniMax TTS path reads that field through the generic secret-input helper, but the helper treats any non-empty string as already-resolved plaintext and only recognizes env refs when they are `${VAR}` templates or SecretRef objects. The current docs also describe TTS `apiKey` config as plaintext-or-SecretRef with env fallback, not `env:VAR` marker syntax. The issue discussion's PR references are useful context, but they do not match what is actually on `main` today.
