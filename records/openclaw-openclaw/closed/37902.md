## Summary

Close as implemented: current main and the latest release no longer expose or accept model-supplied `sessions_spawn` timeout overrides, so the reported timeout erosion path is gone while the operator config remains the timeout source.

## What This Changes

The issue requests a way to prevent model-supplied `sessions_spawn.runTimeoutSeconds` values from undercutting configured subagent timeout policy.

## Best Possible Solution

Keep the shipped behavior that removes per-call timeout overrides from `sessions_spawn`; it solves the undercut problem without adding another config key.

## Reproduction Assessment

No against current main: the model-visible schema omits timeout fields and stale timeout arguments are rejected before spawn execution. The old failure mode was credible for older releases that accepted per-call timeout values.

## Solution Assessment

Yes: removing model-facing per-call timeout overrides is a narrower and more maintainable fix than adding a separate minimum-timeout config surface.
