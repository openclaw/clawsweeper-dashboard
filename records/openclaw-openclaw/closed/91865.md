## Summary

Current main and v2026.6.5 intentionally keep OTLP log bodies behind broad content capture, and both already include the trusted trace-context bridge. The remaining problem is real but product/security-sensitive: logs outside an active trusted span remain uncorrelated, while changing the body default would weaken the documented privacy boundary.

## What This Changes

Make OTLP logs carry useful rendered messages and trace/span correlation instead of placeholder bodies and empty trace identifiers.

## Best Possible Solution

Preserve the secure default and existing `captureContent: true` opt-in for message bodies, then add focused trusted-context propagation for diagnostic operations expected to correlate with active agent turns.

## Reproduction Assessment

Yes. Source and focused tests deterministically show body `log` without broad content capture, and the exporter can only attach OTLP trace identifiers when the trusted diagnostic record already contains a valid span context.

## Solution Assessment

No, not as an unconditional body change. The maintainable path is to retain the documented privacy gate for bodies and separately repair trusted trace-context propagation at specific operations that should be correlated.
