## Summary

Current main already serves Google Chat webhook routes through the gateway’s plugin HTTP path, and the route-registry mismatch that could lead to 404s is covered by explicit regression tests. The same implementation is present in release `v2026.4.22`.

## Evidence

- **Google Chat still registers its webhook as a plugin HTTP route:** `registerGoogleChatWebhookTarget` installs a plugin route for the account webhook path, and `registerWebhookTargetWithPluginRoute` lazily registers that path with `registerPluginHttpRoute` on first target use.
  - file: [extensions/googlechat/src/monitor-routing.ts:27](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/extensions/googlechat/src/monitor-routing.ts#L27)
  - command: `nl -ba extensions/googlechat/src/monitor-routing.ts | sed -n '19,48p'`
  - sha: [56fe2aab9c87](https://github.com/openclaw/openclaw/commit/56fe2aab9c87596515f2280c8060dc740ea64a88)
- **Gateway pins the startup HTTP-route registry:** Gateway startup calls `pinActivePluginHttpRouteRegistry(params.pluginRegistry)` before building request handlers, which prevents later registry swaps from orphaning webhook routes.
  - file: [src/gateway/server-runtime-state.ts:108](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/src/gateway/server-runtime-state.ts#L108)

## Risks / Open Questions

- Conclusion is code-and-test based; I did not run a live Google Chat/Tailscale reproduction in this read-only review.
- `openclaw channels status --probe` for Google Chat still appears to be an API/auth probe rather than an end-to-end local webhook probe, so any remaining probe UX mismatch would be a separate follow-up from the 404 route bug.

## Close Comment

Closing as implemented on current `main`. This Codex review verified that Google Chat still registers `/googlechat` through the plugin HTTP route path (`extensions/googlechat/src/monitor-routing.ts:27`, `src/plugin-sdk/webhook-targets.ts:29`), and Gateway now pins and serves the startup HTTP-route registry when handling requests (`src/gateway/server-runtime-state.ts:108`, `src/gateway/server/plugins-http.ts:69`). The exact regression shape from this report is covered by tests that keep routes reachable after the active plugin registry changes (`src/gateway/server/plugins-http.test.ts:300`, `src/plugins/http-registry.test.ts:208`). I also confirmed the same implementation is present in release `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). If you still hit a 404 on `v2026.4.22` or newer, please open a fresh issue with the exact build, gateway bind/port config, and request/log sample.
