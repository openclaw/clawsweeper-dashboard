## Summary

Keep open. Current main already has managed-proxy routing and a strict-mode `OPENCLAW_PROXY_ACTIVE` path, but the requested first-class plugin/OAuth request policy and scoped private-network relaxation are not implemented and need security/product design.

## What This Changes

Request a first-class config or environment mechanism for plugin and OAuth guarded fetches to use trusted system proxies and optionally relax private-network blocking for scoped OAuth callbacks.

## Best Possible Solution

Design an explicit, auditable request-policy surface for plugin/OAuth fetches that composes with managed proxy config, preserves strict defaults, scopes private-network exceptions, and updates SDK/docs/tests together.

## Reproduction Assessment

No high-confidence runtime reproduction is available for the google-gemini-cli OAuth failure because the issue lacks exact commands, proxy/DNS setup, and logs. Source inspection does reproduce the requested surface gap: plugin OAuth fetches use strict guarded fetch by default and there is no first-class plugin/OAuth policy surface.

## Solution Assessment

Unclear. The existing `OPENCLAW_PROXY_ACTIVE` workaround path already exists for managed proxy routing, but a broader env/config bypass is not the safest final shape until maintainers define scope, precedence, auditability, and private-network exceptions.
