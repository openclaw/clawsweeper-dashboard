## Summary

Current main and shipped releases already provide the supported managed-proxy path for strict guarded plugin/OAuth fetches; plain ambient `HTTPS_PROXY` still stays ignored by default in `STRICT` mode by design, but `proxy.enabled` with `proxy.proxyUrl` or `OPENCLAW_PROXY_URL` activates the trusted operator-managed proxy path.

## What This Changes

Request a first-class global or env-driven way for strict guarded OAuth/plugin fetches, including google-gemini-cli auth, to route through trusted proxies and optionally handle trusted loopback/private destinations.

## Best Possible Solution

Keep the shipped managed proxy path as the supported solution: enable `proxy.enabled` with `proxy.proxyUrl` or `OPENCLAW_PROXY_URL`, and keep private/loopback allowances in the operator proxy policy instead of adding a second broad STRICT-mode bypass.

## Reproduction Assessment

Not applicable as a feature request. Source inspection and existing tests establish the supported current path rather than a failing current-main bug.

## Solution Assessment

Yes. The existing managed proxy design is the narrower maintainable solution because it requires explicit proxy enablement, preserves target hostname policy before proxy dispatch, and avoids turning plain ambient proxy env vars into a general SSRF bypass.
