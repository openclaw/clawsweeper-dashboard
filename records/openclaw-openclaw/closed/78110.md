## Summary

Close as implemented: current main and the shipped v2026.5.4 tree already guard the reported package-update/restart failure mode by blocking unsafe package swaps, forcing authorized immediate update restarts, and surfacing restart-unhealthy states to the UI.

## What This Changes

The issue requests that package-based update.run only report completion after restart scheduling/verification succeeds, and that failed restart handoff be surfaced to Control UI.

## Best Possible Solution

Keep the shipped update-restart safeguards and ask for a fresh report only if a v2026.5.4-or-newer gateway still logs an ignored update.run restart after package replacement.

## Reproduction Assessment

No live package update was run in this read-only pass. Source, docs, and regression tests show current main and v2026.5.4 no longer take the reported unsafe path: package swaps are blocked when restart is unavailable, and package update restarts use the authorized update handoff.

## Solution Assessment

Yes. The existing fix is the narrow maintainable path: preflight restart availability before mutation, force the update-owned restart without normal deferral/cooldown, respawn or hand off to a fresh process, and surface restart-unhealthy to Control UI.
