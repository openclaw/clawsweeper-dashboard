## Summary

Current main addresses the reported Telegram recovery failure mode: startup delivery recovery replays queued entries through `deliverOutboundPayloads`, the same outbound path that plans text chunks with the channel adapter, and Telegram advertises a 4000-character chunk limit clamped to Telegram's 4096 limit. The suggested exact permanent-error regex is not present, but the root oversized recovery send no longer reaches Telegram unchunked, and remaining recovery failures are bounded by `MAX_RETRIES`.

## Best Possible Solution

Keep the shipped recovery-through-outbound implementation. If maintainers want extra defense in depth later, add a narrow regression test and permanent-error pattern for Telegram's exact `message is too long` 400, but the reported recovery chunking bug is already fixed on main.
