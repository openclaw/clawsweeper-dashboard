## Summary

Current `main` already uses per-session command lanes for embedded agent runs, so different sessions are no longer serialized behind a single agent-wide lane. The shared main lane still honors `agents.defaults.maxConcurrent`, allowing parallel execution across sessions up to the configured limit.

## Close Comment

Closing from a Codex review: current `main` already routes embedded agent turns through a per-session lane rather than a single agent-wide lane. `src/agents/pi-embedded-runner/lanes.ts:3` resolves lanes as `session:<sessionKey>`, and `src/agents/pi-embedded-runner/run.ts:247` plus `src/agents/pi-embedded-runner/run.ts:281` enqueue each run on that session lane before entering the shared global lane. `src/auto-reply/reply/get-reply-run.ts:593` and `src/auto-reply/reply/get-reply-run.ts:618` also use the session key for queue control, while `src/gateway/server-lanes.ts:6` still applies `agents.defaults.maxConcurrent` to the global main lane. The queue itself honors concurrency limits (`src/process/command-queue.ts:169`, with coverage in `src/process/command-queue.test.ts:287`). I did not find a remaining agent-ID-wide serialization path matching this report on current `main`, so this is being closed as already implemented there.
