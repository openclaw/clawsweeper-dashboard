## Summary

Current main already implements custom-provider OpenAI Responses reasoning support. The active embedded/default Responses path uses OpenClaw’s boundary-aware transport instead of depending on built-in API registry replacement, and the Responses request builders map both `options.reasoning` and `options.reasoningEffort` into `reasoning.effort` when the model declares `reasoning: true`. The same implementation is present in release `v2026.4.24`.

## Best Possible Solution

Close this issue as implemented. Keep the shipped boundary-aware OpenAI Responses transport and the `options.reasoning` fallback; custom provider model entries still need `reasoning: true` for OpenClaw to emit Responses reasoning parameters.
