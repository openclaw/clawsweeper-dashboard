## Summary

Close as superseded. Current main still has the inline skill tool-policy gap, but this older PR is incomplete in the authorization path and the same remaining work is now tracked by the newer open PR #75125 plus bug report #75124.

## What This Changes

The PR changes inline skill `command-dispatch: tool` lookup to use `createOpenClawCodingTools` and adds regression tests for non-owner messaging-profile dispatch.

## Best Possible Solution

Close this stale branch as superseded and continue review of #75125, or an equivalent narrow replacement, while keeping #75124 open until the effective-policy fix lands.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: current main resolves inline skill `command-dispatch: tool` through raw `createOpenClawTools(...)` plus owner-only filtering, while a restricted profile or group `toolsBySender` case should be evaluated through `createOpenClawCodingTools(...)`.

## Solution Assessment

No. Routing through `createOpenClawCodingTools(...)` is the right seam, but this PR is not the best current solution because it omits gateway subagent binding and full sender identity; the newer #75125 branch is the better canonical implementation candidate.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve gateway subagent binding:** `src/auto-reply/reply/get-reply-inline-actions.ts:198`
  - body: The old inline dispatch path passed `allowGatewaySubagentBinding: true`. This `createOpenClawCodingTools(...)` call omits that option, and the factory only forwards plugin/runtime gateway binding when supplied, so plugin-backed inline tools that depend on gateway subagent binding can become unavailable.
  - confidence: 0.88
- **[P2] Pass the full sender policy context:** `src/auto-reply/reply/get-reply-inline-actions.ts:212-213`
  - body: This call now enforces group tool policy, but it only passes `senderId`. Group policy also matches `senderName`, `senderUsername`, and `senderE164`, so username/E.164/name-specific `toolsBySender` allow or deny rules can be skipped for inline skill dispatch.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The PR touches inline tool authorization and still omits sender identity inputs used by group-specific tool policy; the active security-hardening path should continue in #75125 or an equivalent fix.

Concerns:

- **[medium] Incomplete group policy context:** `src/auto-reply/reply/get-reply-inline-actions.ts:213`
  - body: The new policy-filtering call passes only `senderId`, while group tool policy can select rules using sender name, username, or E.164. Missing those inputs can bypass a sender-specific deny or fail to apply a sender-specific allow in inline skill dispatch.
  - confidence: 0.86
