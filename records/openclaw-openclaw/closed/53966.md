## Summary

Keep open: this PR targets a real Slack diagnostic gap and current main only covers the outbound delivery subset, but the PR head still replaces redacting error formatters with an unredacted Slack formatter, creating a security regression before merge.

## What This Changes

Adds a Slack-local `formatSlackError()` helper and tests, then routes selected Slack probe, setup, streaming, reaction, slash reply, and thread-resolution error logs through it for structured Slack API details.

## Best Possible Solution

Keep the Slack-local formatter boundary, but pass the final assembled string through the plugin SDK redaction seam, add regression coverage for token redaction plus structured Slack fields, and add the required changelog entry.

## Reproduction Assessment

Yes, by source inspection. A focused unit check can pass a token-shaped Slack/Web API error into the PR's `formatSlackError()` and observe the raw token, while current main's `formatErrorMessage()` redacts those token patterns.

## Solution Assessment

No as written. A Slack-local formatter is the right narrow boundary for richer API diagnostics, but it must preserve final-string redaction and include the required changelog entry before merge.
