## Summary

Close as implemented on main. Current main and v2026.4.29 already ship the model-picker filtering behavior through the Gateway `view` API, and the PR branch is now both obsolete and blocked by auth-sensitive review findings.

## What This Changes

The PR adds a Control UI model-selector filter config, extends `models.list` with filter metadata, adds `sessions.patch` credential preflight, updates Control UI status display, and adds related tests.

## Best Possible Solution

Close this PR and keep the shipped v2026.4.29 view-based model visibility implementation. If maintainers still want an operator-configurable picker policy beyond the current configured/full views, that should be a fresh narrow product/API discussion built on `resolveVisibleModelCatalog`.

## Reproduction Assessment

Not applicable as a bug reproduction for this PR. The shipped behavior is verified by current-main source, docs, changelog, and focused gateway/UI tests showing the Control UI uses the configured model view instead of the full catalog.

## Solution Assessment

No, this PR branch is not the best way to solve the issue now. Current main uses the existing additive `view` protocol and runtime auth checker, while the PR branch lets clients weaken server filtering and diverges from runtime auth behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep clients from overriding server filters:** `src/gateway/server-methods/models.ts:36-39`
  - body: The PR gives request `filter` priority over `gateway.controlUi.modelSelector.filter`, so an authenticated client can send `filter: "all"` and enumerate models even when the operator configured a narrower server-side policy.
  - confidence: 0.88
- **[P2] Reuse runtime auth availability for filtering:** `src/agents/model-selection.ts:814-820`
  - body: `hasAuthForProvider` checks only profiles, env keys, and custom API keys. It misses runtime-supported auth such as Bedrock AWS SDK availability and synthetic local providers, so authenticated filtering and preflight can hide or reject models OpenClaw can actually run.
  - confidence: 0.9
- **[P2] Keep configured mode from widening to the full catalog:** `src/agents/model-selection.ts:934-936`
  - body: When `filter: "configured"` is selected but no configured keys are found, the PR returns the full catalog. An explicit configured-only mode should not silently expose every built-in model when the configured set is empty.
  - confidence: 0.86
- **[P2] Enforce configured membership during model patches:** `src/gateway/sessions-patch.ts:410-411`
  - body: The `sessions.patch` preflight checks only provider auth for configured mode, so a client can still submit any authenticated catalog model that is not referenced in agent config, bypassing the mode's stated allowlist semantics.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff has no dependency or supply-chain changes, but it introduces security-sensitive model visibility controls with concrete bypass paths.

Concerns:

- **[medium] Client can weaken configured visibility:** `src/gateway/server-methods/models.ts:37`
  - body: `models.list` gives request `filter` priority over server config, allowing a client to request the full catalog despite an operator-configured authenticated/configured filter.
  - confidence: 0.88
- **[medium] Configured filter can expose every model:** `src/agents/model-selection.ts:936`
  - body: The new configured filter path falls back to the full catalog when the configured set is empty, defeating an explicit visibility restriction and disclosing providers/models the operator intended to hide.
  - confidence: 0.82
