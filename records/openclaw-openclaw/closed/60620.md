## Label Justifications

- `P2`: The original report described a real gateway/plugin startup bug with noisy duplicate registration and limited but meaningful risk for non-idempotent plugins.

## Summary

Close as implemented on current main. The gateway no longer activates runtime plugins during the pre-listen bootstrap phase; startup plugin runtime is loaded once in the post-attach path before sidecars, with regression coverage asserting a single loader call.

## What This Changes

The issue reports that gateway restarts activate configured plugins twice, causing duplicate registration logs and possible duplicate plugin side effects.

## Best Possible Solution

Keep the current main startup shape: bootstrap plugin metadata before listen, then activate startup plugin runtime once in post-attach before channel sidecars.

## Reproduction Assessment

No against current main. The earlier source reproduction path depended on one setup-runtime activation followed by a full-runtime reload; current main skips pre-listen runtime loading and invokes the startup plugin loader once post-attach.

## Solution Assessment

Yes. The current implementation solves this at the gateway startup boundary by making the early phase metadata-only and preserving a single runtime activation before sidecars, instead of adding plugin-specific idempotency guards.
