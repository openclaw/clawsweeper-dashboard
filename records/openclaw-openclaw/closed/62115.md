## Summary

Current `main` contains a targeted fix for the gateway `security=full` + `ask=off` exec regression, and the issueŌĆÖs follow-up narrowed the `1008 invalid request frame` symptom to a custom Python WebSocket client rather than the native OpenClaw CLI.

## Evidence

- **Reporter follow-up narrowed the WS symptom:** The follow-up comment states the `1008` handshake errors were produced by a custom Python `websockets/10.4` script on the host, not by the native `openclaw-cli`, and clarifies that the main remaining OpenClaw bug was the silent exec failure.
- **Gateway intentionally rejects non-protocol first frames:** Current gateway handshake code requires the first WebSocket message to be a normal request frame with `method === "connect"`; otherwise it closes with `1008` and `invalid request frame` / `invalid handshake`.
  - file: [src/gateway/server/ws-connection/message-handler.ts:362](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/src/gateway/server/ws-connection/message-handler.ts#L362)
  - command: `nl -ba src/gateway/server/ws-connection/message-handler.ts | sed -n '340,430p'`
  - sha: [51dd4f288f67](https://github.com/openclaw/openclaw/commit/51dd4f288f67ef4df4c7a121f078f508535df1fd)
- **Protocol contract for native clients:** The protocol schema defines `ConnectParams`, and the bundled gateway client sends them via a `{ type: "req", method: "connect", params }` frame. That matches the handshake requirement and supports the conclusion that the Python-client `1008` was not a native CLI regression proof.
  - file: [src/gateway/protocol/schema/frames.ts:20](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/src/gateway/protocol/schema/frames.ts#L20)
  - command: `nl -ba src/gateway/protocol/schema/frames.ts | sed -n '1,160p'`
  - sha: [51dd4f288f67](https://github.com/openclaw/openclaw/commit/51dd4f288f67ef4df4c7a121f078f508535df1fd)
- **Exec regression fixed on main:** `bash-tools.exec.ts` now explicitly skips script preflight for `host === "gateway" && security === "full" && ask === "off"`, and only runs preflight when that condition is false. This directly addresses the reported YOLO/full-profile regression path.
  - file: [src/agents/bash-tools.exec.ts:1046](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/src/agents/bash-tools.exec.ts#L1046)
  - command: `nl -ba src/agents/bash-tools.exec.ts | sed -n '930,1065p'`
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Regression coverage added:** The exec preflight test suite now includes a YOLO gateway-host heredoc case that executes `node <<'NODE' ...` and expects `status: "completed"` with output `ok`, which is the same class of failure described in the report.
  - file: [src/agents/bash-tools.exec.script-preflight.test.ts:382](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/src/agents/bash-tools.exec.script-preflight.test.ts#L382)
  - command: `nl -ba src/agents/bash-tools.exec.script-preflight.test.ts | sed -n '360,420p'`
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Changelog and docs match the fix:** The changelog records: `Exec/YOLO: stop rejecting gateway-host exec in security=full plus ask=off mode via the Python/Node script preflight hardening path`, and the docs now state that YOLO mode does not add an extra script-preflight rejection layer.
  - file: [CHANGELOG.md:414](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/CHANGELOG.md#L414)
  - command: `nl -ba CHANGELOG.md | sed -n '400,422p'`
