## Summary

Keep open: current main and the latest release still decode the session read-tool text path as UTF-8, and active linked PRs now target the same bug, so this issue should remain open until one canonical fix lands.

## What This Changes

The issue asks the built-in read tool to correctly display GBK/codepage-936 text files on Chinese Windows instead of UTF-8 mojibake.

## Best Possible Solution

Land one narrow PR that routes session read text buffers through the existing strict-UTF-8-then-Windows-codepage decoder without adding a model-facing encoding parameter unless maintainers explicitly approve that API.

## Reproduction Assessment

Yes, source-reproducible: current main reads text buffers with `buffer.toString("utf-8")`, while the report supplies GBK bytes that are not valid UTF-8. I did not run a live Chinese Windows filesystem repro in this read-only review.

## Solution Assessment

Yes, the best fix appears to be reusing the existing Windows codepage decoder in the read-tool path. Adding an `encoding` parameter is a broader API change and should not be the default repair unless maintainers choose it.
