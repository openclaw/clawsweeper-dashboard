## Summary

Close as implemented on current main. The Discord path no longer has the reported channel-owned inbound worker timeout/fallback reply: Discord now enqueues inbound work through the shared channel run queue, that queue explicitly does not impose run timeouts, focused tests assert long queued runs are not aborted or logged as timed out, provider setup ignores the legacy inbound worker timeout config, and docs now say Discord relies on session/tool/runtime lifecycle instead.

## What This Changes

The issue asks OpenClaw to avoid generic 1800-second Discord inbound worker timeout replies and provide clearer run/session diagnostics when Discord inbound work stalls.

## Best Possible Solution

Keep the current main behavior: Discord should rely on generic session/tool/runtime lifecycle for long-running turns and should not reintroduce a Discord-owned inbound worker timeout/fallback reply. Ship the current queue behavior in the next release.
