## Summary

Keep open: current `main` and the latest release still leave the isolated cron CLI branch without the cleanup flags that already exist on the CLI runner contract, while the embedded cron branch and docs expect isolated cron runtime teardown.

## What This Changes

The issue reports stale OpenClaw worker processes accumulating after cron or isolated agent turns, causing high host load, memory pressure, and slow or stalled responses.

## Best Possible Solution

Wire isolated cron CLI runs into the existing one-shot cleanup contract and add regression coverage, while preserving shared MCP loopback ownership so cleanup does not break active parent runs.

## Reproduction Assessment

Yes, at source level: current `main` still has the isolated cron CLI branch using `runCliAgent` without the available cleanup flags while adjacent paths and docs specify isolated cron teardown. I did not run the long Railway/macOS process-count stress reproduction in this read-only review.

## Solution Assessment

No, current `main` is not yet the best solution because cleanup is wired for embedded isolated cron runs but not for CLI isolated cron runs. The safest fix is to propagate the cleanup flags with shared-loopback ownership protection and focused regression tests.
