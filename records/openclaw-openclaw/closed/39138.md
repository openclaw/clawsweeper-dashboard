## Summary

Current main still lacks an outbound delivery recovery age cutoff. Gateway startup loads the full pending delivery queue, and old first-replay entries with `retryCount: 0` and no `lastAttemptAt` are still immediately eligible, so the reported restart flood remains valid and actionable.

## Best Possible Solution

Implement a conservative outbound recovery policy in `src/infra/outbound`: expire or move stale pending deliveries to `failed/` before replay, preserve recent crash recovery, and add focused tests for old `retryCount=0` entries without `lastAttemptAt`, `maxRecoveryMs` interaction, startup recovery, and reconnect drains. If operator control is added, wire it through config/schema/docs instead of hard-coding an undocumented knob.

## Clownfish Work Prompt

Fix the outbound delivery recovery stale replay bug reported in openclaw/openclaw#39138. Current `recoverPendingDeliveries` loads every pending `.json` entry and `isEntryEligibleForRecoveryRetry` immediately replays `retryCount=0` entries without `lastAttemptAt`, even when `enqueuedAt` is many hours old. Implement a conservative recovery policy in `src/infra/outbound` that expires or moves stale pending outbound deliveries to `failed/` before replay while preserving recent crash recovery.

Expected boundary:
- Keep the existing write-ahead queue and two-phase ACK behavior.
- Add a default max pending age for startup recovery, likely around 1 hour unless an existing config pattern suggests a safer name/default.
- Apply the stale cutoff to startup recovery and any reconnect drain path that can replay old first-replay entries.
- Do not drop fresh crash-recovery entries; recent `retryCount=0` entries without `lastAttemptAt` must still replay.
- If adding operator config, update exported config types/schema/help/docs consistently and keep the default conservative.
- Add a user-facing changelog fix entry with reporter attribution, not internal-only wording.

Add tests for:
- old `retryCount=0` entries without `lastAttemptAt` are expired/moved to `failed/` and not delivered;
- fresh `retryCount=0` crash entries still replay;
- stale retry entries with `lastAttemptAt`/`enqueuedAt` interact correctly with backoff and `maxRecoveryMs`;
- reconnect drain does not deliver stale entries;
- startup recovery summary/logging remains understandable.

Do not edit unrelated delivery formatting, channel adapters, or plugin APIs.
