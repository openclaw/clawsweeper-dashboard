## Summary

Keep open. Current main still has the Control UI ordering gap: the model picker awaits `sessions.patch` only inside its change handler, while the composer/send path and controls have no pending model-switch state to await before `chat.send` starts.

## What This Changes

Ensure Control UI model dropdown selections are persisted before the next affected `chat.send` starts.

## Best Possible Solution

Add a Control UI per-session model-switch barrier that disables the picker and send/queue controls while pending, awaits the `sessions.patch` before affected sends, rolls back optimistic selection on failure, preserves drafts/attachments, and leaves the Gateway protocol unchanged.

Yes. A UI-side barrier is the narrowest maintainable fix because the dropdown already owns session persistence and the current `chat.send` protocol has no model override field.

Fix issue https://github.com/openclaw/openclaw/issues/54240 by adding a Control UI-side per-session pending model-switch barrier. When the chat model dropdown calls `sessions.patch`, record the pending operation in shared UI state, disable the model picker and send/queue controls while it is pending, and have affected send paths await it before `chat.send`. If the patch fails, roll back the optimistic selected model, preserve the draft and attachments, surface the existing error style, and do not start the send. Keep the Gateway `chat.send` protocol unchanged unless a failing regression proves the UI barrier is insufficient. Add regression coverage with a deferred `sessions.patch` proving `chat.send` waits, plus control rendering coverage for disabled send/model controls. Add a user-facing `CHANGELOG.md` fix entry.
