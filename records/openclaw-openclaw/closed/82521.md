## Summary

Keep open: current main still has a source-level mismatch between the documented OpenAI/Codex auth-order contract and the explicit PI route, and no open PR appears to own this exact fix. The follow-up device-code report is also a real docs/CLI mismatch: the provider exposes `device-code` as an auth method, not as a top-level `--device-code` option.

## What This Changes

The issue asks OpenClaw to make documented `openai/gpt-*` plus `auth.order.openai` subscription-first routing engage `openai-codex:*` auth for explicit PI runs, and to fix the documented device-code login command.

## Best Possible Solution

Align runtime and docs: explicit PI OpenAI runs should honor a documented `openai-codex:*` profile selected through `auth.order.openai` before API-key fallback, and the docs should advertise the existing `--method device-code` login path or intentionally add a compatible alias.

## Reproduction Assessment

Yes, as a source-level reproduction: configure `openai/gpt-5.5` with `agentRuntime.id: "pi"` and `auth.order.openai` listing `openai-codex:*` before `openai:*`; current main resolves and filters auth for provider `openai` before the Codex PI guard can engage. I did not live-run the reporter's credentials.

## Solution Assessment

Yes: the narrow fix is to make the explicit PI auth-selection path honor the documented Codex profile before API-key fallback, plus correct the device-code docs to the existing method syntax. A broader auth rewrite or changing doctor back to legacy model refs would be riskier than necessary.
