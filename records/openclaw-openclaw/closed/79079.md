## Summary

Keep open. Current main still accepts the reported cron delivery shape during add/update validation, while the later delivery resolver can already prove that an explicit Telegram announce target with no resolvable `to` will fail with a missing-target error.

## What This Changes

The issue asks OpenClaw to reject or gracefully handle Telegram cron announce delivery without a configured target instead of accepting the job and failing each fire.

## Best Possible Solution

Add add/edit-time validation for announce delivery that is already known to be permanently unresolved, while preserving documented session/default-target fallback paths.

## Reproduction Assessment

Yes from source inspection: current validation accepts the reported delivery object, and the later resolver returns the missing-target error when no explicit, default, or session target exists. I did not run a live cron fire because this review was read-only.

## Solution Assessment

Yes: moving the existing delivery-target knowledge into add/edit validation is the narrowest maintainable fix. The implementation should preflight unresolved explicit announce routes instead of hard-requiring `to` for every announce job.
