## Summary

Current `main` now guards the exact failure mode described here: startup-triggered config writes are projected back onto the authored source config before persistence, destructive shrink/`gateway.mode` removal writes are rejected before commit, and suspicious clobbered reads are auto-restored from last-known-good with targeted regression coverage. I could not tie this to a released tag from the checked-out history, so the fix appears to be on `main` but not yet attributable to a specific shipped release.

## Evidence

- **Gateway startup still uses the reported write surfaces:** The same startup paths called out in the issue discussion are still the ones that persist changes: plugin auto-enable writes via `writeConfigFile(autoEnable.config)`, and generated gateway auth persists via `replaceConfigFile({ nextConfig: nextCfg, ... })`. That means the current hardened writer stack covers the exact startup code paths implicated in the report.

  - sha: [11804a484ded](https://github.com/openclaw/openclaw/commit/11804a484ded89374afa55020813cfa265aa143a)
- **Writes now preserve authored source config instead of round-tripping runtime shape:** `writeConfigFile` now derives a persist candidate with `resolvePersistCandidateForWrite({ runtimeConfig: snapshot.config, sourceConfig: snapshot.resolved, nextConfig: cfg, rootAuthoredConfig: snapshot.parsed })`. `resolvePersistCandidateForWrite` computes a merge patch from runtime to next config, projects the authored source shape back over runtime, and preserves include-owned/root-authored structure. That directly addresses the April 11 triage comment about runtime-shaped startup writes dropping authored fields.

  - sha: [11804a484ded](https://github.com/openclaw/openclaw/commit/11804a484ded89374afa55020813cfa265aa143a)
- **Destructive config writes are blocked before rename/copy:** Current `main` explicitly flags `size-drop:*` and `gateway-mode-removed` as blocking reasons, then rejects the write and saves a `.rejected.*` payload instead of replacing the active config when `allowDestructiveWrite` is not set.

- **Targeted regression tests cover both halves of the failure:** There is now a test that rejects destructive internal writes before the config is replaced, and separate recovery tests that restore from backup when metadata disappears, when `gateway.mode` disappears, and after a large size drop against last-known-good.
