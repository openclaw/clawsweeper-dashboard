## Summary

Keep open. The latest head still has a blocking wake-now intent regression: the explicit operator wake path can be flood-deferred under the new policy, and the protected `maintainer` label also rules out cleanup closure.

## What This Changes

This PR replaces raw heartbeat wake reasons with typed `source`/`intent` requests, adds centralized cooldown and flood gating, and updates cron, gateway, agent, plugin runtime, docs, changelog, and tests.

## Best Possible Solution

Keep the typed cooldown design, but classify explicit operator wake-now requests as `intent: "manual"` or give `source: "manual"` the same full exemption, with a focused regression test.

## Reproduction Assessment

Yes. A focused unit path can seed five recent run starts within the flood window, call the cron/system `wake(..., { mode: "now" })` path, and observe the current PR's `immediate` intent being skipped as `flood` instead of bypassing deferral like `manual`.

## Solution Assessment

No. Centralized typed wake intent is the right direction, but using `source: "manual"` with `intent: "immediate"` is not the narrowest compatible fix because only `intent: "manual"` is fully non-deferred.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Classify wake-now requests as manual intent:** `src/cron/service/timer.ts:1600`
  - body: `wake(..., { mode: "now" })` is the explicit operator-triggered wake path, but this call tags it as `intent: "immediate"`. `shouldDeferWake` only fully exempts `intent: "manual"`, while `immediate` still runs the flood guard, so repeated user wake-now commands can be skipped as `flood` during burst conditions. Use `intent: "manual"` here or give manual-source wakes the same exemption.
  - confidence: 0.88
