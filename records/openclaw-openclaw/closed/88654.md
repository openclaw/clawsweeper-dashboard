## Summary

Current main already fixes the reported cache churn: successful auth-profile usage updates can still persist state, but unchanged secrets no longer rewrite `auth-profiles.json`, so the `models.json` fingerprint is not invalidated by `lastUsed`-only updates. The fix is main-only and not in `v2026.5.28`.

## What This Changes

The issue asks OpenClaw to stop `markAuthProfileSuccess` from rewriting `auth-profiles.json` on every successful model call so plugin-backed provider model resolution can stay cached.

## Best Possible Solution

Keep the current main save-layer behavior and let the next release carry the fix; any separate auth-state invalidation concern should be tracked as a new, narrower issue.

## Reproduction Assessment

Yes: the `v2026.5.28` source path unconditionally rewrote `auth-profiles.json` after successful auth-profile updates, while current main no longer does. I did not run a live gateway reproduction because this review is read-only.

## Solution Assessment

Yes: fixing the save layer is the maintainable solution because it prevents no-op `auth-profiles.json` rewrites for all callers, not only `markAuthProfileSuccess`.
