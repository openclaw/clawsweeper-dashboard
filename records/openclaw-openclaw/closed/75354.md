## Summary

Close as implemented on current main for the linked Telegram status failure: main already splits Telegram read-only config accessors from runtime token resolution and has regression coverage for file-backed SecretRefs, while this PR would weaken the existing runtime strictness contract.

## What This Changes

The PR removes the strict re-resolution call for non-env SecretRefs in Telegram `resolveRuntimeTokenValue`, making those token values return `configured_unavailable` instead of throwing.

## Best Possible Solution

Keep the current main approach: read-only Telegram metadata should use the config accessor split, and runtime token resolution should continue to fail fast for unresolved non-env SecretRefs.

## Reproduction Assessment

Yes. The linked bug shape is covered by `extensions/telegram/src/shared.test.ts:170`, which exercises a Telegram file-backed SecretRef while reading allowlist/default-target metadata without token resolution. I did not execute tests because this review was constrained to read-only inspection.

## Solution Assessment

No. The PR diff is not the best fix because it changes the shared runtime token helper; the narrower main implementation fixes read-only status metadata without relaxing runtime SecretRef diagnostics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Preserve strict runtime SecretRef failures:** `extensions/telegram/src/token.ts:86-87`
  - body: `resolveRuntimeTokenValue` feeds `resolveTelegramToken`, which is used by runtime send, monitor, and action paths through `resolveTelegramAccount`, not only read-only status. After this change, an unresolved file or exec-backed `botToken` returns `source: "none"` instead of throwing the existing unresolved SecretRef error, so the current strictness regression test would fail and operators lose the correct remediation path.
  - confidence: 0.92
