## Summary

Keep open. Current main still has source-level paths for the reported reliability cluster: cache-inclusive prompt pressure can feed context-engine maintenance, the empty fallback boundary still has no large-token or memory-flush guard, and reply preflight compaction still runs before memory flush.

## What This Changes

The issue reports premature/destructive compaction caused by cached-token accounting, empty fallback summaries, and memory-flush ordering on long Anthropic cached sessions.

## Best Possible Solution

Land one coordinated reliability fix that uses a trusted active-context estimate, blocks destructive large empty fallbacks, and guarantees enabled memory flush cannot trail the compaction cycle it is meant to protect.

## Reproduction Assessment

Yes, at source level. The production trace is concrete, and current main still exposes the cache-inclusive currentTokenCount path, high-token empty fallback branch, and preflight-before-memoryFlush ordering; I did not run a live Opus/proxy replay.

## Solution Assessment

Partly. The requested direction is valid for the destructive fallback and flush ordering, but the cached-token fix should use trusted context snapshots or anomaly guards rather than blanket-dropping cacheRead.
