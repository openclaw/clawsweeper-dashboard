## Summary

Close as implemented on main. Current main resolves OpenAI Codex thinking profiles through a lightweight bundled provider policy artifact when the startup-lazy OpenAI plugin is not active, and the gateway/session paths that previously rejected or downgraded `xhigh` now use that shared resolver. The fix is present on current `main` but is not contained in the latest tagged release `v2026.4.29`.

## What This Changes

The issue requests that gateway/session thinking validation and metadata accept and advertise `xhigh` for `openai-codex/gpt-5.5` instead of falling back to the base `off|minimal|low|medium|high` profile when the OpenAI plugin is startup-lazy.

## Best Possible Solution

Keep the current lightweight provider-policy fallback on main and let the next release carry it to users; it preserves the plugin boundary and fixes the gateway/session validation surface without hardcoding OpenAI Codex model policy into core.

## Reproduction Assessment

Yes. The report and follow-up comments provide concrete 2026.4.27 gateway, CLI, session-list, and `sessions_spawn` failure paths, and current main has focused regression tests for the no-catalog startup-lazy provider case that previously reproduced the rejection.

## Solution Assessment

Yes. The current fix is the narrow maintainable direction: provider-owned thinking profiles remain in the OpenAI plugin policy artifact, while shared gateway/session validation can read that profile without requiring startup activation of the full plugin runtime.
