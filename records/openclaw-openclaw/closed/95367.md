## Summary

Close: the reporter confirmed the failure was caused by re-authenticating Claude through an SSH session that did not update the macOS Keychain; after re-authenticating from a logged-in GUI session, the LaunchAgent path worked. Current source and docs match that boundary: OpenClaw expects Claude CLI to be logged in on the gateway host and intentionally keeps managed Claude CLI runs isolated from inherited Anthropic/Claude override environment variables.

## What This Changes

The issue reports a macOS LaunchAgent-managed Gateway returning Claude CLI 401s while foreground `openclaw gateway run` works.

## Best Possible Solution

Keep OpenClaw's current Claude CLI credential isolation and direct users to reauthenticate Claude from a logged-in macOS GUI session on the gateway host before rerunning OpenClaw's Anthropic CLI auth setup.

## Reproduction Assessment

No. I did not run a macOS LaunchAgent live reproduction in this Linux read-only checkout; the reporter's follow-up provides the decisive before/after setup difference and says the managed path now works after GUI-session Claude re-authentication.

## Solution Assessment

Yes. Closing as externally resolved is the best current repository action; changing OpenClaw to pass inherited Anthropic or Claude Code env vars into managed Claude CLI runs would conflict with the existing isolation contract.
