## Summary

Keep open. Current main still has the reported disabled-fetch startup path, but this PR should not merge as-is because its resolver guard also suppresses the existing inactive SecretRef warning path for disabled fetch configs with plugin-scoped credentials; it also overlaps with #74921.

## What This Changes

The PR changes secrets runtime fast-path detection, runtime web-fetch provider resolution, and regression tests so `tools.web.fetch.enabled: false` skips web-fetch provider discovery.

## Best Possible Solution

Land one selected fix that makes credential-free `tools.web.fetch.enabled: false` skip web-fetch provider discovery during Gateway startup while preserving inactive SecretRef diagnostics for credential-bearing plugin-scoped fetch config; then close #74896 and supersede or close the overlapping implementation PR.

## Reproduction Assessment

Yes. #74896 provides a concrete Linux/Docker config, version, trace boundary, and removal workaround, and current main source confirms that a disabled fetch key still bypasses the secrets fast path and reaches provider discovery before Gateway bind.

## Solution Assessment

No for this exact patch as currently shaped. The fast-path change is directionally right, but the resolver-level guard should be narrower so plugin-scoped or credential-bearing disabled fetch config can still emit inactive SecretRef warnings without resolving the secret.
