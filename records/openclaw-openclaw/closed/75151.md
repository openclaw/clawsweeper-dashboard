## Label Justifications

- `P1`: The reported bug made long-lived agent sessions appear to lose continuity after overflow/compaction failure, which is an urgent user-facing session workflow break.
- `impact:data-loss`: The user-visible effect was lost access to recent conversation history even though the transcript still existed on disk.
- `impact:session-state`: The issue is about `sessions.json` mapping drift between the active session key, `sessionId`, and transcript file.

## Summary

Close: current main now solves the reported context-overflow/compaction-failure path by preserving the active session mapping instead of rotating it to a fresh missing transcript; the fix is current-main-only and not present in the latest v2026.6.6 release tag.

## What This Changes

The issue requests that context-overflow or compaction-failure recovery stop rotating `sessions.json` to a nonexistent `sessionFile` and orphaning the real transcript history.

## Best Possible Solution

Keep the current preserved-session overflow/compaction-failure behavior; separate manual `/reset` or `/new` missing-transcript cases remain tracked at https://github.com/openclaw/openclaw/issues/89245.

## Reproduction Assessment

Yes at source level. Current regression tests simulate both embedded overflow error payloads and thrown compaction failures and verify the session mapping is preserved; I did not run a live Telegram overflow replay.

## Solution Assessment

Yes. Preserving the existing mapping on failed overflow recovery is the narrowest maintainable fix for this issue because it satisfies the safe outcome of keeping the old transcript mapped instead of creating a new missing transcript.
