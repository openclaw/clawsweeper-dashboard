## Summary

Keep open: current main and the latest release still feed dreaming from daily memory notes, session transcripts, and accumulated recall traces, but not workspace file-creation events. This is a coherent new memory-source capability that needs a maintainer product decision on ownership, privacy filtering, and persistence rather than cleanup closure.

## What This Changes

Add workspace file-creation signals to dreaming’s light-phase inputs so newly created tools and artifacts can become durable memory-promotion candidates.

## Best Possible Solution

Define a typed, bounded memory-core artifact or capability event source with explicit privacy, allowlisting, deduplication, lifecycle, and SQLite persistence rules, then let light dreaming consume those prepared signals without polling arbitrary workspace paths.

## Reproduction Assessment

Yes, at high confidence from source inspection: the sweep explicitly ingests daily memory notes and session transcripts before consuming recall traces, with no workspace file-creation source. This demonstrates the limitation, though workspace watching is not an existing documented contract.

## Solution Assessment

No, the suggested generic configurable filesystem watcher is not yet the best solution. A typed memory-plugin event source is narrower, safer, avoids request-time discovery, and preserves the repository’s plugin and persistence boundaries.
