## What This Changes

The issue asks OpenClaw to handle ChatGPT/Codex OAuth backend rejections for non-working `openai-codex` GPT-5.x model ids such as `gpt-5.2-codex` and `gpt-5.3-codex`.

## Best Possible Solution

Keep this as OpenAI Codex provider-boundary work and add account-aware handling for the backend entitlement response so OpenClaw can mark the selected model unavailable for the current ChatGPT/Codex account, fall back cleanly when possible, and document the diagnostic path without globally deleting catalog rows.
