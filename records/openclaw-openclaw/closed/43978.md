## Summary

Close as implemented on current main: the WhatsApp reconnect-window delivery problem is now handled in the plugin-owned send path by waiting for a replacement socket and retrying disconnect-like send failures under the reconnect policy, with test and release proof in v2026.4.27. The PR should not merge as-is because it targets removed legacy files and its delivery-failure system event is broader than the stated final-reply behavior.

## What This Changes

The PR widens legacy WhatsApp Web auto-reply send retries from 3 attempts at 500ms linear backoff to 5 attempts at 1000ms and adds a delivery-failure system event in the old `src/web/auto-reply` path.

## Best Possible Solution

Keep the shipped plugin-owned reconnect-aware send path. Broader retry tunability can continue in #54183, and any future agent-visible delivery-failure signal should be implemented as a final-only, sanitized follow-up in `extensions/whatsapp`.

## Reproduction Assessment

Yes. The original reconnect-window behavior is covered by the current test that sets `socketRef.current = null`, installs a replacement socket during the wait, and verifies the reply is sent through that replacement socket; the PR-specific system-event bug is also reproducible from the dispatcher contract allowing `tool`, `block`, and `final` error kinds.

## Solution Assessment

No, this PR is not the best way to solve the issue now. Current main solves the retry-window at the socket/listener boundary in `extensions/whatsapp`, which is more robust than widening an obsolete outer retry wrapper; the PR's system event would need a final-only guard and current-path adaptation if still desired.
