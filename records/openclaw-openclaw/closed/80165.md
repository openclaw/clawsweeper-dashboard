## Summary

Current main routes Feishu DM completions back through the canonical requester session instead of deriving a new channel-specific session key, so the reported split-session behavior from 2026.4.15 is already addressed on main.

## What This Changes

The issue requests consistent session-key reuse for Feishu DM user turns and async completion notifications from subagents or exec results.

## Best Possible Solution

Keep the current main implementation that separates requester session identity from channel delivery metadata, and include it in the next shipped release if it has not shipped yet.

## Reproduction Assessment

Yes for the old reported behavior: the issue includes concrete Feishu session-store evidence from 2026.4.15, and the current source path is traceable from Feishu inbound routing through subagent completion delivery. I did not run a live Feishu websocket reproduction in this read-only review.

## Solution Assessment

Yes. The current implementation is the narrow maintainable fix because it keeps a single canonical requester session key and uses channel/account/to only as delivery hints for that session.
