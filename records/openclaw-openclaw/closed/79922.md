## Summary

Keep open: the startup-grace direction matches the linked bug, but the current PR head still skips the liveness sampler during grace, which can carry cold-start delay into the first post-grace sample and recreate the warning. The supplied real behavior proof is still narrative-only and not auditable.

## What This Changes

Adds an internal diagnostic heartbeat startup grace option, passes a 60 second grace window from gateway startup, and adds a focused liveness warning suppression test.

## Best Possible Solution

Keep the startup grace, but always run the liveness sampler on every heartbeat and suppress only liveness event/WARN emission while the grace window is active, with a test that counts sampling during grace.

## Reproduction Assessment

Yes. Current main source starts diagnostics at gateway startup without a grace window, and a queued-work heartbeat with a non-null liveness sample reaches the WARN path; I did not run tests because this review is read-only.

## Solution Assessment

No. The PR is aimed at the right bug, but the current implementation gates before sampling; the narrower maintainable fix is to sample every tick and gate only emission.
