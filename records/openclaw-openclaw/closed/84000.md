## Summary

Keep open: this targets a real tool-policy concern, but the patch should not merge as-is because it conflates additive `alsoAllow` with restrictive `allow` and would remove default `sessions_yield` from existing alsoAllow-only configs; the PR also lacks after-fix real behavior proof.

## What This Changes

The PR gates `sessions_yield` construction in `createOpenClawTools` with the existing tool-policy matcher and adds unit coverage for deny, narrow allow, and runtime deny cases.

## Best Possible Solution

Keep the construction-time gate, but make it honor only true restrictive allowlists and denylists while preserving default core tools for alsoAllow-only policies, then add regression coverage for both `tools.allow` and `tools.alsoAllow` cases.

## Reproduction Assessment

No high-confidence live reproduction is present. Source inspection confirms unconditional construction on current main and a source-level compatibility regression in the PR, but the related issue still lacks real runtime proof that filtered sessions can invoke the tool after policy removal.

## Solution Assessment

No, not as written. The narrow construction-time fix is reasonable, but it must preserve additive `alsoAllow` semantics instead of treating every factory allowlist entry as a restrictive core-tool allowlist.

- **[P1] Preserve alsoAllow-only core tools when gating sessions_yield:** `src/agents/openclaw-tools.ts:365-368`

Gate on truly restrictive allowlists, or preserve the implicit allow-all marker, and add an alsoAllow-only regression test.
