## Summary

Keep open. Current main still routes managed Deep Dreaming and manual `memory promote --apply` through the direct `MEMORY.md` append helper, and the newer review-queue duplicate was closed in favor of this issue as the canonical product/config discussion.

## What This Changes

Request to add a configurable review queue or final-promotion destination policy before Dreaming deep promotion writes candidates to `MEMORY.md`.

## Best Possible Solution

Define an additive, schema-backed final-promotion destination policy that preserves current `MEMORY.md` behavior by default while offering a structured review artifact and explicit approval/apply path.

## Reproduction Assessment

Not applicable as a feature/config request. Source, docs, and tests give high-confidence verification that current main still lacks a review-queue or final-promotion destination option.

## Solution Assessment

Unclear until maintainers choose the workflow contract. A separate final-promotion destination or approval flow appears safer than overloading `dreaming.storage.mode`, which current code and docs use for phase/report storage.
