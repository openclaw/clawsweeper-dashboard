## Summary

The issue is asking OpenClaw core to adopt or surface an external LanceDB memory plugin, but current main explicitly scopes optional capabilities like this to the plugin ecosystem: external repo ownership plus ClawHub/npm distribution, not core maintenance.

## Close Comment

Closing after a Codex review of current `main` (`c6684af682bcb59bd1ee85cb49f72d4074007e2c`). This looks like a good fit for the OpenClaw plugin ecosystem, not a core-repo cleanup item. The current project vision says optional capability should usually ship as plugins and be hosted/maintained in their own repository (`VISION.md:54-59`), community plugins are supposed to be published on ClawHub or npm (`docs/plugins/community.md:9-22`, `docs/plugins/community.md:129-149`), and the plugin author guide explicitly says external plugins do not need to be added to the OpenClaw repo (`docs/plugins/building-plugins.md:16-19`). OpenClaw also already exposes dedicated memory-plugin seams via `api.registerMemoryCapability(...)` / `registerMemoryEmbeddingProvider(...)` (`docs/plugins/sdk-overview.md:173-195`, `src/plugins/types.ts:2029-2060`). Given that scope, the enhanced LanceDB integration should live as a ClawHub/npm plugin maintained in its own repo rather than as an `openclaw/openclaw` core issue.
