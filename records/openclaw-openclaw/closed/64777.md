## Summary

Current main does not support the reported diagnosis. Unset `tools.profile` still means unrestricted `full`, `gateway` is not part of the `coding` profile at all, and the CLI-vs-channel difference is explained by owner-only tool filtering: `cron`, `gateway`, and `nodes` are removed for non-owner senders, while channel senders are only owners when `commands.ownerAllowFrom` (or internal admin scope) matches.

## Evidence

- **Unset profile still means full:** The tools docs state that `tools.profile` is a base allowlist, that local onboarding writes `tools.profile: "coding"` into new local configs when unset, and that `full` is 'No restriction (same as unset)'.
  - file: [docs/gateway/config-tools.md:18](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/docs/gateway/config-tools.md#L18)
  - command: `nl -ba docs/gateway/config-tools.md | sed -n '16,28p'`
  - sha: [fd65caf4b0ba](https://github.com/openclaw/openclaw/commit/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc)
- **Coding does not include gateway:** The core tool catalog includes `cron` in the `coding` profile, but `gateway` has an empty profile list. The profile table also defines `full` as an empty policy object, which resolves to no restriction.
  - file: [src/agents/tool-catalog.ts:225](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/src/agents/tool-catalog.ts#L225)
  - command: `nl -ba src/agents/tool-catalog.ts | sed -n '220,375p'`
  - sha: [fd65caf4b0ba](https://github.com/openclaw/openclaw/commit/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc)
- **Owner-only tools are filtered before profile policy:** Tool creation treats unknown sender ownership as unauthorized and applies `applyOwnerOnlyToolPolicy` before the remaining policy pipeline, so owner-only tools disappear from non-owner channel runs even when the base profile would otherwise allow them.
  - file: [src/agents/pi-tools.ts:671](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/src/agents/pi-tools.ts#L671)
  - command: `nl -ba src/agents/pi-tools.ts | sed -n '665,690p'`
  - sha: [fd65caf4b0ba](https://github.com/openclaw/openclaw/commit/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc)
- **Channel senders are not owners by default:** Command auth resolves owners from `commands.ownerAllowFrom`; if no owner allowlist matches, `senderIsOwner` stays false. The dedicated tests assert that direct-message and group-chat senders are not owners when no `ownerAllowFrom` is configured.
  - file: [src/auto-reply/command-auth.ts:277](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/src/auto-reply/command-auth.ts#L277)
  - command: `nl -ba src/auto-reply/command-auth.ts | sed -n '270,310p'; nl -ba src/auto-reply/command-auth.ts | sed -n '690,715p'; nl -ba src/auto-reply/command-auth.owner-default.test.ts | sed -n '1,90p'`
  - sha: [fd65caf4b0ba](https://github.com/openclaw/openclaw/commit/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc)
- **Cron and gateway are explicitly owner-only surfaces:** Docs call `gateway` an owner-only runtime tool, and channel config docs define `ownerAllowFrom` as the explicit owner allowlist for owner-only commands and tools. Tests also assert that non-owner senders lose `cron` and `gateway`, while owners keep them.
  - file: [docs/tools/index.md:88](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/docs/tools/index.md#L88)
