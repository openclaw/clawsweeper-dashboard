## Summary

Current main already solves the user-visible problem: Telegram file/media delivery no longer depends on agents curling Telegram Bot API endpoints. The prompt exposes `MEDIA:<path-or-url>` for attachment delivery, the message tool accepts media/path/filePath aliases, and the Telegram channel maps those internal payloads to `sendPhoto`, `sendDocument`, and related Bot API calls. The cross-linked #43280 correctly identified the intended internal route; although that PR closed unmerged, equivalent behavior is present on main and shipped in v2026.4.24.

## Best Possible Solution

Keep the shipped internal attachment path: current-chat replies should use `MEDIA:<path-or-url>`, and proactive Telegram sends should use the `message` tool with media/path/filePath-style payloads. OpenClaw should continue to discourage raw exec/curl provider messaging; any future model-specific refusal should be handled as a narrower prompt/test regression, not by authorizing direct Telegram Bot API curl from agents.
