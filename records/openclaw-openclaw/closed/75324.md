## Summary

Keep open. Current main still has the Discord action-discovery path that can strict-resolve a raw SecretRef token, and message-tool execution still skips command-time SecretRef resolution when constructed with an injected config snapshot. The issue is also paired with open closing PR #75424, so the right cleanup is to review and land or close that PR rather than closing this issue now.

## What This Changes

The report asks OpenClaw to make Discord SecretRef-backed tokens work consistently at runtime when `openclaw secrets audit` already reports them resolved, especially during Discord message-action discovery and message-tool sends.

## Best Possible Solution

Keep this issue open until #75424 or an equivalent fix lands. The end state should make Discord action discovery use config-only credential inspection for SecretRef-backed tokens, keep actual runtime token use strict, and resolve scoped command SecretRefs for message-tool execution even when a config snapshot is injected.

## Reproduction Assessment

Yes. The issue provides concrete config and steps, and current source gives a high-confidence static reproduction path: raw SecretRef config reaches `describeDiscordMessageTool` -> `resolveDiscordAccount` -> `resolveDiscordToken` -> `normalizeResolvedSecretInputString`.

## Solution Assessment

Yes, the linked PR direction is the narrow maintainable solution: avoid reading token values during discovery and resolve secrets at the scoped execution boundary. Closing the issue before that PR or an equivalent fix merges would be premature.
