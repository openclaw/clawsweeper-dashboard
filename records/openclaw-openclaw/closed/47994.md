## Summary

Current main already fixes the sticky automatic fallback behavior that this PR targets, using the newer selected-model/runtime-model split built around providerOverride/modelOverride plus modelOverrideSource:"auto" rather than this branch's modelIsFromFallback schema flag. The linked bug is closed, related #68991 was grouped as the same problem, and current code/tests/changelog show fallback selections are transient, user model choices are protected, stale runtime model fields are cleared when selection changes, and recovered primaries are retried on the next turn.

## Best Possible Solution

Close this PR as implemented on main. Keep the shipped selected/runtime split using providerOverride/modelOverride plus modelOverrideSource:"auto", where automatic fallback is transient, explicit user selections remain durable, runtime metadata can still record what served the turn, and recovered primaries are retried on subsequent turns.
