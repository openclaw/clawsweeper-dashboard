## Summary

Current and released code suppresses Telegram `NO_REPLY` lead fragments before partial delivery, resolving the reported visible flash. The proposed transport-level guard remains absent, but it would not prevent narration already sent to a preview and is unnecessary for the central observable bug.

## What This Changes

Prevent Telegram from briefly showing streamed text that is ultimately suppressed as a silent `NO_REPLY` response.

## Best Possible Solution

Keep shared pre-delivery prefix suppression as the canonical fix; open a new narrowly reproduced issue only if a token-only direct Telegram send can still reach the API outside that pipeline.

## Reproduction Assessment

No. Current and released code drops exact silent replies and streamed `NO`, `NO_`, and later `NO_REPLY` fragments before Telegram preview delivery, so the reported flash path no longer reproduces from source.

## Solution Assessment

Yes. Suppressing silent-token prefixes at the shared streaming boundary acts before visible delivery and avoids duplicating auto-reply policy inside the Telegram transport; a final send guard cannot undo narration already streamed.
