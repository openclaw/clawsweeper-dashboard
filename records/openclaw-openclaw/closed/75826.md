## Summary

Close as implemented on current main. The reported v2026.4.29 packaged-runtime dependency blow-up is addressed by the current startup path: gateway plugin loading is scoped through the startup plugin plan, empty scopes no longer fall back to loading everything, and current docs now treat bundled runtime-deps staging roots as legacy debris rather than new runtime output.

## What This Changes

The issue asks OpenClaw to stop staging runtime dependencies for every bundled plugin and instead scope dependency work to enabled/configured plugins without blocking gateway startup.

## Best Possible Solution

Keep the current main-branch direction: plugin dependencies are installed only through explicit install/update/doctor flows, gateway startup loads only the computed startup plugin scope, and legacy runtime-deps directories are cleaned by doctor rather than recreated at runtime. Ship this in the next release so v2026.4.29 users can upgrade off the bloated staging behavior.

## Reproduction Assessment

Yes. The issue gives a concrete v2026.4.29 reproduction with size, module count, spawned npm install, and event-loop-delay logs; current-main verification is source and regression-test inspection showing the unscoped startup plugin path has been removed.

## Solution Assessment

Yes. Removing the implicit broad startup/runtime-deps path is narrower and more maintainable than adding another filter to legacy staging, because it keeps package-manager work out of gateway startup and reload entirely.
