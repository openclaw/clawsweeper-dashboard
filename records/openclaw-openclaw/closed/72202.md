## Summary

Current main already handles the reported no-Codex-OAuth case: the OpenAI image provider only takes the Codex Responses route when an `openai-codex` auth profile is present, otherwise it resolves `openai` auth and posts to `/images/generations`. Config `env.vars.OPENAI_API_KEY` is applied during config reads before env substitution/runtime auth, and the v2026.4.26 changelog shows the OpenAI image/Codex auth-routing work shipped after the reporter's 2026.4.24 version.

## Best Possible Solution

Close as implemented by the shipped OpenAI image auth-routing behavior in v2026.4.26. If a user still reproduces this on v2026.4.26 or newer, the next useful report should include `image_generate action=list`, redacted auth-profile provider ids, and gateway logs around `image auth selected` so maintainers can tell whether an `openai-codex` profile is unexpectedly being resolved.
