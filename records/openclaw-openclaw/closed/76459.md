## Summary

Close as implemented. Current main no longer uses the hard-coded 2.5-second Telegram startup `getMe` probe; the startup path now uses the Telegram `getMe` request guard, honors higher `timeoutSeconds`, and the matching changelog entry shipped in `v2026.5.2`.

## What This Changes

The issue reports Telegram startup `getMe` aborting after a hard-coded 2500 ms timeout in Docker on Windows and requests a longer or configurable timeout.

## Best Possible Solution

Keep the shipped Telegram startup timeout behavior and direct users on older `2026.4.29` installs to upgrade to `v2026.5.2` or newer, with `channels.telegram.timeoutSeconds` available for unusually slow Bot API paths.

## Reproduction Assessment

No against current main: the reported 2500 ms startup `getMe` path is gone, and source/tests now show a 15-second default plus higher configured budgets. The original bug was source-reproducible in the earlier #75783 review, where `startAccount` still passed `2500` directly to the Telegram probe.

## Solution Assessment

Yes. Reusing the Telegram-owned `getMe` request guard and existing `timeoutSeconds` config is the narrow maintainable fix; adding a new core network policy or a separate Telegram-only config knob is unnecessary.
