## What This Changes

The issue requests a working non-interactive OpenAI Codex API-key auth path because `paste-token` stores pasted values as token credentials while Codex API-key auth requires an `api_key` profile with `key`.

## Best Possible Solution

Review and land the focused fix in https://github.com/openclaw/openclaw/pull/85533 or an equivalent narrow PR, then let the closing reference resolve this issue after merge.

## Reproduction Assessment

Yes. Source inspection confirms that current `paste-token` writes token-mode credentials, the API-key resolver expects `key` on `api_key` profiles, and current CLI registration has no non-interactive `paste-api-key` command.

## Solution Assessment

Yes. The best fix direction is a first-class non-interactive API-key paste path plus narrow Codex diagnostics, preserving generic token paste semantics instead of changing what `paste-token` means.
