## Summary

Keep open. Current main and the latest release still reload a bounded recent chat history window on every Control UI session switch and render the resulting chat items through the normal Lit repeat path; source supports the performance concern, but the reported 8-10s browser latency has not been reproduced in this review.

## What This Changes

Request to make Control UI session switching responsive by avoiding repeated recent-history reprocessing/rendering and adding a virtualized or paged chat history path.

## Best Possible Solution

Measure a large-transcript session switch in the Control UI, then land a focused performance change that keeps bounded-history safety while adding a maintainable paged or virtualized render path with regression coverage.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection does confirm the current switch path reloads chat.history and renders the bounded recent window without virtualization, which is enough to keep the report actionable.

## Solution Assessment

Unclear. Virtual scrolling or pagination is plausible, but the narrowest maintainable fix should be chosen after measuring the UI cost and deciding whether to extend the RPC surface or reuse the existing HTTP pagination path.
