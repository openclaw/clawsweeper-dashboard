Full review comments:

- **[P2] Keep registry failure counts on one record basis:** `src/tasks/task-registry.summary.ts:46-52`
  - body: `buildTaskStatusSnapshot` drops rows whose `cleanupAfter` has already passed, but this function still fills `total` and `byStatus` from every record. In the window before maintenance pruning, an expired failed/timed_out/lost row is still counted in `byStatus` and `total` while being omitted from `failures`, so JSON/status consumers get contradictory aggregates. Please compute all of these counters from the same filtered or unfiltered task set.
  - confidence: 0.89
- **[P2] Align task pressure with listed rows:** `src/commands/tasks.ts:122`
  - body: `tasksListCommand` renders every task in the filtered `tasks` array, but the new pressure summary is based on `buildTaskStatusSnapshot(tasks)`, which filters cleanup-expired terminal rows. That can print failed task rows while the pressure line says `0 recent failures` and omits historical failures, making operator triage misleading. Use the same record basis for the summary and table.
  - confidence: 0.87
- **[P2] Align flow summaries with linked task rows:** `src/commands/flows.ts:244`
  - body: The flow headline now reads failure counts from `taskSummary`, but the linked-task section below still prints every `listTasksForFlowId` row. Because `taskSummary` now comes through the cleanup-filtered snapshot path, cleanup-expired failed rows can appear under `Linked tasks` while the headline reports no recent or historical failures. Please make the headline and linked rows use the same filtered or unfiltered basis.
  - confidence: 0.86
- **[P3] Pad the Tasks header with the widened column:** `src/commands/flows.ts:91`
  - body: Rows now pad the Tasks column to `TASKS_PAD`, but the header still uses `"Tasks".padEnd(14)`. Once a flow row includes the longer recent/historical failure text, the Goal header starts far left of the row values, so the table is visibly misaligned. Use the same column width for the header.
