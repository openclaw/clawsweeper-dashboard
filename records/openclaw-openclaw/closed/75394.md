## Summary

Close as implemented on main. Current main now stages `qrcode` through the WhatsApp plugin runtime-deps path, and the package-level runtime-deps planner installs configured bundled plugin dependencies into the packaged runtime-deps root; the fix is main-only after `v2026.4.29`.

## What This Changes

The report asks OpenClaw to restore WhatsApp QR rendering in packaged `2026.4.29` installs where the shared QR helper cannot resolve the `qrcode` package.

## Best Possible Solution

Keep the owner-module staged-dependency fix, ship it in the next release, and treat the alternate linked fallback PR as superseded unless maintainers want a separate defensive resolution change.

## Reproduction Assessment

Yes. The report gives a concrete packaged `2026.4.29` QR-login path, and the static path is direct: WhatsApp QR rendering reaches the shared SDK QR helper, which imports `qrcode`, while the reported release did not stage that package for WhatsApp.

## Solution Assessment

Yes. Staging `qrcode` with the WhatsApp plugin dependency set fixes the missing package at the packaging boundary without adding a hardcoded `qrcode/lib/index.js` fallback or duplicating QR rendering inside the channel.
