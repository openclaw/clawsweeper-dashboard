## Label Justifications

- `impact:session-state`: The issue is about losing stable turn identity across compaction and replaying a prompt as a fresh turn.
- `impact:message-loss`: The reported loop leaves channels unresponsive and prevents the intended reply from completing.
- `impact:crash-loop`: The report describes repeated compaction/retry cycles that behave like an availability loop for the affected lanes.

## Summary

Keep open. Current main has a partial replay guard for persisted inbound messages, but isolated cron embedded runs still enter the overflow-compaction retry path without a durable current-turn identity, so they can still retry the raw prompt after compaction.

## What This Changes

The issue asks OpenClaw to stop compaction recovery from treating cron/system retry prompts as fresh user input and re-running the same tool cascade.

## Best Possible Solution

Add a durable turn/reply-obligation identity for cron, system, and inbound turns so post-compaction recovery resumes from the compacted transcript without duplicating side effects, with regression coverage around cron retries and existing inbound-message replay guards.

## Reproduction Assessment

Yes from source, though not from a live run in this review. Isolated cron calls runEmbeddedAgent without currentMessageId, and the overflow-compaction retry path falls back to params.prompt unless a continuation override is set.

## Solution Assessment

Unclear whether the reporter's full ledger proposal is the final shape, but the current currentMessageId-only guard is too narrow. The maintainable fix likely belongs in shared run/turn identity rather than a cron-only retry special case.
