## Summary

Keep open: the PR targets a real device-pairing token-scope bug, but the current patch has a blocking authorization regression around revoked role tokens and still only provides test-only real behavior proof.

## What This Changes

The PR changes device-pairing approval to mint upgraded role tokens from existing token scopes plus newly requested scope deltas, adds two regression tests, and adds a changelog fix entry.

## Best Possible Solution

Update the PR to treat revoked existing role tokens as having no reusable scope base during upgrade minting, add revoked-token upgrade regression coverage, and provide real gateway/device-pairing or MCP proof before merge.

## Reproduction Assessment

Yes, at source level: current main returns only pending role-scoped scopes for non-empty approvals while the approved baseline is unioned separately, matching the related scope replacement reports. I did not run a live Gateway/MCP loop in this read-only review.

## Solution Assessment

No. The PR direction is narrow and maintainable, but the current diff reuses `existingToken.scopes` even when that token is revoked; the safer fix should ignore revoked tokens as a merge base while preserving active narrowed tokens.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Ignore revoked token scopes when merging upgrade scopes:** `src/infra/device-pairing.ts:455`
  - body: When an existing role token has `revokedAtMs`, this new merge still feeds `existingToken.scopes` into the newly minted token. A later scope-bearing upgrade can therefore clear `revokedAtMs` and regrant scopes from the revoked token instead of starting from no reusable token base; treat revoked tokens as absent here and cover that path.
  - confidence: 0.87

## Security Review

Status: needs_attention

Summary: Needs attention: the patch narrows the reported bug but introduces a concrete revoked-token regrant path in token minting.

Concerns:

- **[medium] Revoked scopes can be regranted on upgrade:** `src/infra/device-pairing.ts:455`
  - body: Because the PR merges `existingToken.scopes` without checking `revokedAtMs`, approving a later scope-bearing request can mint a fresh unrevoked token that includes scopes from a revoked token record. That can undo a prior revocation decision in an authorization-sensitive path.
  - confidence: 0.86
