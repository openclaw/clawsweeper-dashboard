## Summary

Current `main` already contains a shared guarded-fetch fix that covers the reported Gemini memory-index `fetch failed` path. The report was against `2026.3.28`; the relevant transport fix shipped later in `v2026.4.7`, and current memory embedding requests flow through that corrected stack.

## Evidence

- **Release note for the transport fix:** `CHANGELOG.md` under `2026.4.7` explicitly calls out a fix for `TypeError: fetch failed` by forcing guarded fetch dispatchers to `allowH2: false`. That matches the reported error string and postdates the reported `2026.3.28` version.
  - file: [CHANGELOG.md:1153](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/CHANGELOG.md#L1153)
  - command: `nl -ba CHANGELOG.md | sed -n '1135,1165p'`
- **Memory embeddings use the shared guarded fetch path:** Remote memory HTTP calls go through `withRemoteHttpResponse(...)`, which directly calls `fetchWithSsrFGuard(...)`. This means memory indexing benefits from the same dispatcher fix as the other guarded fetch surfaces.
  - file: [src/memory-host-sdk/host/remote-http.ts:6](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/memory-host-sdk/host/remote-http.ts#L6)
  - command: `nl -ba src/memory-host-sdk/host/remote-http.ts | sed -n '1,40p'`
