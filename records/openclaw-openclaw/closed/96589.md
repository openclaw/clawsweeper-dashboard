## Label Justifications

- `P1`: A configured cron alert path can silently stop requesting notifications after restart, breaking a user-facing operator alert workflow.
- `impact:message-loss`: The bug can suppress configured cron failure notifications after reload, so operators may never receive expected alert messages.

## Summary

Keep open: current main still has the defaults-only cron failure-alert round-trip bug, and no linked PR or duplicate owns this exact SQLite sentinel collapse.

## What This Changes

The issue reports that `openclaw cron edit <job> --failure-alert` creates a defaults-only per-job alert config that works in memory but reloads as disabled after a gateway restart.

## Best Possible Solution

Preserve `failure_alert_disabled === 0` as an explicit enabled-with-defaults sentinel in the SQLite read codec and add a regression test that round-trips `failureAlert: {}` through the real cron store.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: the CLI creates `{}`, the store writes `failure_alert_disabled = 0`, the reader ignores that sentinel when other columns are null, and the resolver treats the resulting `undefined` as alerts disabled. I did not run a live gateway restart because this review is read-only.

## Solution Assessment

Yes, the best fix is the narrow store-codec repair plus a real store round-trip regression test. Changing the CLI or resolver would paper over the persistence bug instead of preserving the canonical stored state.
