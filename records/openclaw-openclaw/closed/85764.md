## Summary

Keep this PR open for maintainer review: source inspection confirms current main ignores `maxHoldMs` during acquisition-time stale inspection, and the branch is a focused fix with regression coverage, but merging it changes session-state locking semantics because a contender may remove a live holder's lock once the holder-recorded max-hold expires.

## What This Changes

The PR records `maxHoldMs` in session lock payloads, lets acquisition-time reclaim honor it for non-self-held locks via unchanged-snapshot removal, adds lock regression tests, refreshes Swift gateway models, clears one gateway test mock, and adds a changelog entry.

## Best Possible Solution

Land the narrow lock fix after maintainers accept holder-recorded `maxHoldMs` as the cross-process reclaim policy and update operator docs/help to describe acquisition-time reclaim.

## Reproduction Assessment

Yes at source level: current main's acquisition-time `shouldReclaim` inspection ignores `maxHoldMs`, while fs-safe calls `shouldReclaim` during lock contention. I did not run a live multi-process repro in this read-only review.

## Solution Assessment

Mostly yes, if maintainers accept the lock policy: the PR uses the holder-recorded max-hold value, skips self-held acquisition reclaim, and relies on unchanged-snapshot removal. The safer merge path is to pair it with docs/help updates and explicit maintainer acceptance of the cross-process semantics.
