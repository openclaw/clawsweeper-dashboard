## Summary

Close as duplicate/superseded. This PR is a one-file subset of the already-open #74979 path for #74859, and it still leaves the same abort-after-guard/before-`Agent.prompt()` preflight race, so the remaining work should be consolidated in the canonical PR or a maintainer replacement branch.

## What This Changes

This PR adds a pre-prompt abort guard and a cleanup-time `activeSession.abort()` call in the embedded runner to avoid zombie Agent loops for pre-aborted runs.

## Best Possible Solution

Consolidate the fix in #74979 or a maintainer replacement branch with an abort-aware prompt-start boundary and regression coverage for both pre-aborted signals and aborts during `AgentSession.prompt()` preflight. The linked bug should close only after that implementation lands.

## Reproduction Assessment

Yes. The linked #74859 report gives a deterministic pre-aborted `AbortSignal` reproduction, and the remaining PR-specific race can be reproduced by aborting after this guard but before `AgentSession.prompt()` reaches `Agent.prompt()` during async preflight.

## Solution Assessment

No. The one-shot guard helps the already-aborted case before this branch, but it is not the narrow complete fix because cleanup-time `activeSession.abort()` cannot cancel a Pi run before `Agent.activeRun` exists.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Close the abort window before prompt preflight:** `src/agents/pi-embedded-runner/run/attempt.ts:2735-2737`
  - body: The new guard runs once before the runtime-context branch, but the non-runtimeOnly path can still await `queueRuntimeContextForNextTurn()` and then call `activeSession.prompt()`. If the run aborts during that await, `activeSession.abort()` is a no-op until Pi creates `activeRun`, so the prompt can still start a floating Agent loop. Move the abort-aware boundary to immediately before each prompt start or pass cancellation into `AgentSession.prompt()` preflight.
  - confidence: 0.84
