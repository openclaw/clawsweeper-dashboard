## Summary

PR #41566 removes the root pnpm lockfile, but current main still treats `pnpm-lock.yaml` as required for pnpm-based installs, CI bootstrap, Docker builds/docs, local PR tooling, and audit/update logic. The PR premise that the lockfile is unused and can simply be regenerated does not match current repository behavior.

## Best Possible Solution

Close the PR and keep the root `pnpm-lock.yaml` tracked. Future dependency changes should update `package.json` and `pnpm-lock.yaml` together so frozen-install CI, Docker builds/docs, local PR tooling, and lockfile-backed audit/update paths remain deterministic.
