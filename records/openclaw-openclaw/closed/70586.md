## Summary

Keep this PR open, but not as-is: the Slack message tracking gap is real for accepted branches that return before the shared message handler, while the current patch also tracks normal handled messages before the handler that already records them.

## What This Changes

The PR threads Slack `trackEvent` into `registerSlackMessageEvents`, calls it from `message` and `app_mention` handlers, and adds registrar tests for tracked and mismatched events.

## Best Possible Solution

Keep the thread-through, but fire registrar-level tracking only for accepted early-return message/system-event paths that never reach `createSlackMessageHandler`, with focused tests and an Unreleased Slack fix entry.

## Reproduction Assessment

Yes. A static reproduction is enough: with this patch, a normal Slack `message` or channel `app_mention` calls `trackEvent` in the registrar and then reaches `createSlackMessageHandler`, which already calls the same tracker for accepted handled messages.

## Solution Assessment

No. The requested gap is narrow, but tracking immediately after the mismatch guard is broader than the current acceptance contract; the safer fix is to preserve handler-level tracking for normal handled events and add registrar tracking only where the handler is bypassed.
