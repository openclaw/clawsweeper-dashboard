## Summary

Current main already supports the requested Docker/headless-node workflow: run the node in the foreground with `openclaw node run`, then approve its device pairing with `openclaw devices approve`. `openclaw node install` remains intentionally service-manager-only.

## Close Comment

Closing this as implemented after Codex review.

Current `main` already supports the containerized/headless node flow. The supported command is `openclaw node run`; `openclaw node install` is the service-manager path, so it is not the right entrypoint for a foreground container process.

What I checked:
- `src/cli/node-cli/register.ts:41` registers `node run` as the foreground node-host command.
- `src/cli/node-cli/register.ts:75` registers `node install` separately for service installation.
- `src/cli/node-cli/daemon.ts:89` backs the service-manager install path.
- `src/node-host/runner.ts:179` connects the node host as a normal paired device with `role: "node"`.
- `docs/cli/node.md:109` and `docs/nodes/index.md:23` document the first-run pairing flow: list the pending request, then approve it with `openclaw devices approve <requestId>`.
- The same behavior is already present in `v2026.4.22`, so this is not an open core feature gap on current releases.

The confusing part is understandable: `pairing required` is expected on first connect until the node device is approved. For container/headless use, `openclaw node run` plus device approval is the supported path.

Codex evidence: reviewed against `cec34821754a`; fixed in `v2026.4.22`.
