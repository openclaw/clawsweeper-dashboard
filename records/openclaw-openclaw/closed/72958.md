## Summary

Keep open for maintainer review. Current main still has the fixed 16k/32k guard behavior, this PR is open and unmerged, and the item is protected by both MEMBER authorship and the `maintainer` label; the latest diff also still has a concrete P2 regression in custom-provider onboarding defaults.

## What This Changes

The PR changes agent context-window guard thresholds to dynamic 10%/20% values with 4k/8k floors, tracks reference tokens for capped runtime contexts, removes fixed embedded-runner guard overrides, and updates focused tests, local-model docs, and changelog text.

## Best Possible Solution

Keep the centralized dynamic guard resolver direction, but split custom-provider onboarding defaults from guard safety floors before merge. The final patch should preserve a usable custom-provider default context window, add or update onboarding regression coverage, and then proceed through normal maintainer CI/review for the protected PR.

## Reproduction Assessment

Yes. On current main, resolving an 8k model context and passing it to `evaluateContextWindowGuard` reproduces the original fixed-threshold block because the hard minimum is still 16k; the PR-introduced onboarding regression is reproducible by following `applyCustomApiConfig` for a new non-Azure custom provider after the exported guard floor is lowered to 4k.

## Solution Assessment

No, not as-is. The dynamic guard resolver and removal of fixed runner overrides are the right narrow direction, but coupling custom-provider onboarding defaults to the lowered guard floor creates unusable 4000/4096 model configs and should be decoupled before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86
