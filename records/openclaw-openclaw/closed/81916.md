## Summary

Keep open: the latest head addresses the earlier stale-output/tail review gap and supplies current-head runtime proof, but the new marker detector can misclassify ordinary transcript text as a compaction marker and drop fresh output/tail pressure.

## What This Changes

The PR changes the preflight compaction transcript estimator and focused tests to detect post-usage compaction markers, bound stale usage snapshots against active replay, drop marker-stale output/tail pressure, and preserve no-marker tail pressure.

## Best Possible Solution

Keep the marker-aware estimator, but trust only structured or otherwise internal compaction markers before dropping output/tail pressure, with a regression that ordinary user/tool text containing marker phrases stays conservative.

## Reproduction Assessment

Yes, by source inspection: a latest usage near the preflight threshold followed by ordinary user/tool text containing the refresh marker phrase will set hasPostUsageCompactionMarker and drop output/tail pressure. I did not execute it because this review is read-only.

## Solution Assessment

No. The marker-aware direction is right, but the implementation should not treat arbitrary transcript text as proof of compaction before lowering the conservative estimate.

- **[P2] Trust only structured compaction markers:** `src/auto-reply/reply/agent-runner-memory.ts:315-317`
