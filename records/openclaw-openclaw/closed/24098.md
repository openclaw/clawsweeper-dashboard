## Summary

Close as `clawhub`. The idea is coherent, but current `main` treats memory as an active plugin slot and exposes plugin SDK seams for alternative memory implementations. This PR adds a new global backend into the old core-style memory path, while current `main` only supports `memory.backend = "builtin" | "qmd"` and has moved the runtime into the `memory-core` plugin.

## Evidence

- **Vision scope prefers plugins for optional capability:** `VISION.md` says optional capability should usually ship as plugins and that memory is a dedicated plugin slot with multiple memory options.
  - file: [VISION.md:52](https://github.com/openclaw/openclaw/blob/d3f6783b16ecc10c67b101bf019ca4294453206a/VISION.md#L52)
- **Memory CLI is plugin-owned on current main:** The memory CLI docs state `openclaw memory` is provided by the active memory plugin, defaulting to `memory-core`.
  - file: [docs/cli/memory.md:12](https://github.com/openclaw/openclaw/blob/d3f6783b16ecc10c67b101bf019ca4294453206a/docs/cli/memory.md#L12)
- **Plugin SDK has explicit memory extension seams:** The SDK docs describe exclusive memory-plugin APIs and embedding-provider registration for custom memory behavior, pointing new memory capability toward plugins rather than new core backends.
  - file: [docs/plugins/sdk-overview.md:186](https://github.com/openclaw/openclaw/blob/d3f6783b16ecc10c67b101bf019ca4294453206a/docs/plugins/sdk-overview.md#L186)
- **Current config still only exposes builtin and QMD backends:** `MemoryBackend` on current `main` is still `"builtin" | "qmd"`, matching the global help text that only documents those two engines.
  - file: [src/config/types.memory.ts:3](https://github.com/openclaw/openclaw/blob/d3f6783b16ecc10c67b101bf019ca4294453206a/src/config/types.memory.ts#L3)
- **PR targets a stale layout, while current runtime lives in the memory-core plugin:** Local inspection shows `src/memory/` now only contains `root-memory-files.ts`, while the active search/runtime implementation lives under `extensions/memory-core/src/memory/`, including `search-manager.ts` handling builtin/QMD manager selection.
  - file: [extensions/memory-core/src/memory/search-manager.ts:79](https://github.com/openclaw/openclaw/blob/d3f6783b16ecc10c67b101bf019ca4294453206a/extensions/memory-core/src/memory/search-manager.ts#L79)

## Risks / Open Questions

- If maintainers want this capability bundled instead of external, the feature would need a plugin-first rework against the current memory plugin architecture rather than merging this PR as-is.
- Closing this PR does not provide the feature; it redirects the work to the current plugin/ClawHub path.

## Close Comment

Closing after a Codex review. Current `main` has a plugin-first memory architecture, and this feature fits that path better than another built-in backend branch. The scope anchor is `VISION.md:52`, which says optional capability should usually ship as plugins and that memory is a dedicated plugin slot; the user-facing memory surface is also explicitly plugin-owned in `docs/cli/memory.md:12`. The SDK already exposes memory-plugin seams for this kind of work in `docs/plugins/sdk-overview.md:186`.

This PR is also built on the older `src/memory/*` layout, but current `main` now keeps the active runtime under `extensions/memory-core/src/memory/` and still only exposes `memory.backend = "builtin" | "qmd"` in `src/config/types.memory.ts:3`. Given that architecture, the conservative maintainer path is to close this as core-scope mismatch and encourage a memory plugin / ClawHub implementation instead of merging it into OpenClaw core.
