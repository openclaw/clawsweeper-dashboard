## Summary

Keep open: the patch is too narrow to fix the linked deleted-cwd crash, because the guarded predicate still hands off to other startup paths that call `process.cwd()` directly. A broader open PR tracks the same bug, but its current proof blocker means this PR should be reviewed on its own rather than closed as safely superseded.

## What This Changes

The PR wraps the `shouldLoadCliDotEnv` cwd `.env` probe in a try/catch so deleted-cwd errors fall through to the state-dir `.env` check.

## Best Possible Solution

Use one safe cwd helper across the predicate, CLI/gateway/runtime dotenv loaders, PATH bootstrap, home-dir fallback, and TUI startup, then attach real deleted-cwd terminal proof before merging or closing the linked bug.

## Reproduction Assessment

Yes, source-level. Current main reaches unguarded `process.cwd()` calls in CLI dotenv and PATH/TUI startup paths, and an isolated Node probe confirms deleted cwd raises `ENOENT`/`uv_cwd`.

## Solution Assessment

No. Guarding only `shouldLoadCliDotEnv()` is not the best fix because the actual loader and sibling startup paths still call `process.cwd()` directly; a shared safe-cwd helper across the startup surface is the maintainable path.
