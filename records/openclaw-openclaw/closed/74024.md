## Summary

This PR is superseded by the existing strict skill-entry contract: current main intentionally keeps unknown top-level `skills.entries.<name>` keys invalid and supports custom per-skill data under `skills.entries.<name>.config`. That approach was explicitly chosen in #1225/#1226 and shipped in v2026.1.20, while this PR reopens the rejected top-level-field shape.

## Best Possible Solution

Close this PR as superseded by the shipped config-bag contract. Keep custom skill-owned settings under `skills.entries.<name>.config`, keep top-level skill entries strict for typo detection, and handle #73939 with docs/usage guidance unless a maintainer explicitly decides to reopen the public config contract.
