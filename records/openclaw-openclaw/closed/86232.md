## Summary

Keep open: current main still has a credible Discord/Codex completion-delivery gap because message-tool-only wakeups are rejected when the active Codex requester handle does not advertise the same source reply delivery mode.

## What This Changes

The issue reports that sub-agent completion auto-announcements and `sessions_yield` wakeups can be dropped in Discord guild chats when delivery is requested as `message_tool_only`.

## Best Possible Solution

Preserve `sourceReplyDeliveryMode` on Codex app-server active run handles like the PI runner does, keep the mismatch guard intact, and add focused regression coverage for message-tool-only completion wakeups.

## Reproduction Assessment

Yes by source inspection, but not by a live Discord run: message-tool-only Discord/Codex sessions are represented in fixtures, the queue rejects mismatched handles, and the Codex handle currently omits the required mode field.

## Solution Assessment

Yes, a narrow fix should propagate the existing `sourceReplyDeliveryMode` contract into the Codex active-run handle rather than loosening the shared mismatch guard or changing Discord visibility policy.
