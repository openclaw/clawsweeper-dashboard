## Summary

Keep open. Current main still forwards OpenAI-compatible Chat Completions messages from the upstream converter without a strict provider/model message-key serialization policy; the KeepMy Claw workaround in the discussion is useful external routing context, but it is not an OpenClaw source fix. Related PR #44086 addresses a different null-content compatibility failure and does not supersede this message-key issue.

## What This Changes

OpenAI-compatible providers should be able to strip provider-rejected extra message keys before sending Chat Completions payloads.

## Best Possible Solution

Add an explicit provider/model-level OpenAI-compatible Chat Completions serialization policy that runs after message conversion and before dispatch in both direct and embedded payload paths, with docs and regression coverage for prior assistant tool calls plus tool-result history.

## Reproduction Assessment

Yes. A high-confidence source reproduction path is an openai-completions model with prior assistant tool-call and tool-result history; current main shows those converted messages can reach provider payloads without strict key filtering.

## Solution Assessment

Partly. Sanitizing at the outbound payload boundary is the right fix shape, but automatically stripping every openai-completions payload would be too broad; an explicit compat policy is safer.
