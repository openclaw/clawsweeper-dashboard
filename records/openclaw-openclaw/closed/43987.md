## Summary

Current main already implements transcript-backed token fallback for `session_status`/`/status`, so sparse or zero cached session counters are replaced by transcript-derived usage instead of staying at `0/...`. The same behavior is present in the released `v2026.4.22` tree.

## Risks / Open Questions

- Issue comments about cumulative-vs-current context semantics and JSON `remainingTokens`/`percentUsed` are broader follow-up concerns, but they are distinct from the original `session_status` zero-display bug.
- If a provider/transcript emits no usage metadata at all, status can still be limited by missing upstream data; that is separate from the fixed sparse-snapshot fallback path.

## Close Comment

Closing from Codex review: current `main` already routes `session_status` through transcript-backed status rendering with `includeTranscriptUsage: true` (`src/agents/tools/session-status-tool.ts:547`, `src/agents/tools/session-status-tool.ts:598`). The regression coverage in `src/auto-reply/reply/commands-status.test.ts:440` writes transcript usage for an active session and expects `Context: 1.0k/32k`, which matches the reported `0/... despite active conversation` symptom. The transcript reader that powers this path aggregates assistant usage and marks it fresh in `src/gateway/session-utils.fs.ts:518` and `src/gateway/session-utils.fs.ts:568`, and the fallback is documented in `docs/reference/token-use.md:91`. This behavior is present in the shipped `v2026.4.22` release, so this issue is implemented on main/released code. If anyone can still reproduce `Context: 0/...` on `v2026.4.22` or newer, please open a new issue with the provider/model plus a transcript/log sample showing missing usage metadata.
