- **[P2] Validate file links before forwarding media:** `extensions/nextcloud-talk/src/monitor.ts:198`
  - body: `parseRichObjectContent` accepts any signed rich-object `http(s)` link and forwards it as `MediaUrls`, which later triggers media fetching. The core fetch path has SSRF guards, but this channel should still only promote URLs that match the configured Nextcloud backend or an explicit account policy so rich-object content cannot become arbitrary fetch input.
  - confidence: 0.84
- **[P2] Carry MIME metadata with promoted file links:** `extensions/nextcloud-talk/src/monitor.ts:237`
  - body: The rich-object file parameter includes `mimetype`, but the PR only stores `mediaUrls`. Current media normalization pads missing types to `application/octet-stream` and then falls back to URL extensions, so extensionless Nextcloud links such as `/f/<id>` can classify as unknown and never reach image/audio/video understanding.
  - confidence: 0.88
- **[P3] Return parsed text for rich objects without file placeholders:** `extensions/nextcloud-talk/src/monitor.ts:211-212`
  - body: When a valid rich-object body has no matching placeholders or file URLs, this helper returns `null`, so the caller falls back to the raw JSON string instead of the human-readable `message`. Live discussion says normal Nextcloud Talk messages can use that envelope with empty parameters, so this keeps leaking JSON into agent text.
  - confidence: 0.73
