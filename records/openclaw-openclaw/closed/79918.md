## Summary

Keep open: current main and v2026.6.6 still route realtime voice-call WebSocket upgrades with a raw prefix match, while open PRs already target the exact boundary fix.

## What This Changes

The issue asks for voice-call realtime WebSocket upgrade routing to accept only the configured stream path or slash-delimited child paths, not same-prefix siblings.

## Best Possible Solution

Land the clean route-boundary fix in https://github.com/openclaw/openclaw/pull/91784 or an equivalent exact-or-slash-child matcher with sibling/root regression coverage, then close this issue and supersede the older conflicting PR.

## Reproduction Assessment

Yes. By source inspection on current main, `pathname.startsWith(pattern)` accepts `/voice/stream/realtime-extra/token` for a configured `/voice/stream/realtime` path; no runtime test was run because this review is read-only.

## Solution Assessment

Yes. The narrow maintainable fix is in the webhook upgrade router: preserve exact and slash-delimited token child paths while rejecting same-prefix siblings before dispatching to `RealtimeCallHandler`.
