## Label Justifications

- `P1`: This PR changes an authorization boundary for browser proxy control and currently introduces a plugin runtime scope-elevation concern that should be resolved before merge.
- `merge-risk: 🚨 compatibility`: Existing non-admin direct `node.invoke(browser.proxy)` callers will fail after this change, even though the break may be intended security hardening.
- `merge-risk: 🚨 security-boundary`: The diff lets plugin runtime callers request scopes that can become a synthetic admin client for `node.invoke`, which affects authorization boundaries beyond normal CI coverage.

## Summary

Keep open: the admin-boundary fix is valid and the follow-up preserved the known first-party browser callers, but the branch now exposes a public plugin runtime path that can mint requested scopes for `node.invoke`; that needs a narrower trusted path or explicit maintainer API/security approval before merge.

## What This Changes

The PR requires `operator.admin` for direct `node.invoke(browser.proxy)` calls and updates Browser, Google Meet, plugin runtime, CLI runtime, and regression tests to request/pass admin-scoped node dispatch.

## Best Possible Solution

Keep the direct `browser.proxy` admin gate, preserve bundled Browser and Google Meet through a narrow trusted runtime/browser-proxy path, and document or explicitly approve any plugin API surface that can request elevated scopes.

## Reproduction Assessment

Yes. Source inspection shows current main has `browser.request` admin-scoped and `node.invoke` write-scoped while Browser and Google Meet use `node.invoke(browser.proxy)`; the PR body/comment also provide targeted before/after terminal output, though this read-only review did not rerun tests.

## Solution Assessment

No, not as-is. The direct admin gate and first-party caller updates are the right direction, but generic public runtime scope elevation should be narrowed or explicitly approved as an API/security decision.
