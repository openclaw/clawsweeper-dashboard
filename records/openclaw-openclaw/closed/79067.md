## Summary

This PR addresses a real cron false-failure path, but the current patch should stay open because the new fingerprint parser can over-match different file paths containing the fingerprint delimiter.

## What This Changes

The PR adds same-path `edit`/`write` recovery in `isSameToolMutationAction`, regression tests, and a changelog entry for the cron self-heal false-failure bug.

## Best Possible Solution

Keep the mutation-layer fix, but compare a structured or escaped file target so delimiter-bearing paths cannot over-match, then land it with a regression test.

## Reproduction Assessment

Yes. Current main source shows same-path `edit` to `write` recovery cannot clear `lastToolError`, and the PR body supplies live output from a production-helper driver showing the pre-fix fatal cron payload path and post-fix intended behavior.

## Solution Assessment

No, not as currently written. The mutation-layer location is the right narrow fix, but parsing unescaped fingerprint strings with `split("|")` breaks the fail-closed guarantee for legal paths containing that delimiter.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Avoid parsing unescaped path fingerprints:** `src/agents/tool-mutation.ts:239`
  - body: `buildToolActionFingerprint` stores raw path values in a `|`-delimited string, while write/edit path validation only checks for a non-empty string. Splitting the fingerprint here truncates paths containing `|`, so a failed edit on `/tmp/a|left` and a successful write to `/tmp/a|right` both extract as `path=/tmp/a` and incorrectly clear the prior failure. Please carry/escape a structured target or otherwise fail closed for ambiguous fingerprints, and add a delimiter-path regression test.
  - confidence: 0.87
