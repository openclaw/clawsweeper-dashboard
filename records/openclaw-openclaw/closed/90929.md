## Summary

Keep open: the PR targets a valid Dreams status undercount, but the patch only adds HHMM support to a duplicate doctor matcher while `memory-core` already accepts nested and slugged daily memory paths, so the reported counts can still drift. It also needs after-fix real behavior proof before merge.

## What This Changes

This PR broadens `doctor.memory.status`'s local short-term memory path regex to count top-level date files with a four-digit timestamp suffix.

## Best Possible Solution

Make the doctor status classifier mirror the current memory-core short-term path contract, including subdirectories, slug suffixes, and dreaming-path exclusion, with focused status regression coverage.

## Reproduction Assessment

Yes, from source: current `doctor.memory.status` filters recall-store entries through a strict date-only matcher, so a promoted entry with `memory/YYYY-MM-DD-HHMM.md` is skipped before the promoted counts are calculated. I did not run a full live gateway repro in this read-only review.

## Solution Assessment

No, this is the right surface but not the best fix yet: it partially widens the duplicate regex instead of matching the memory-core path contract that creates and consumes the same recall-store entries.
