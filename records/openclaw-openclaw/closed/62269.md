## Summary

Current OpenClaw already treats `openclaw gateway` as a core CLI command and hardens Windows bundled-plugin loading by normalizing absolute import paths and canonical `dist-runtime` wrapper paths. The reported `plugins.allow excludes "gateway"` follow-up is obsolete on current code, and the relevant behavior is already present in release `v2026.4.23`.

## Evidence

- **Gateway is a core CLI command on main:** `registerGatewayCli` unconditionally registers `program.command("gateway")` and `gateway run`, so Gateway is not a plugin-gated CLI surface on current main. That directly contradicts the follow-up comment claiming `plugins.allow` excludes `gateway`.
  - file: [src/cli/gateway-cli/register.ts:394](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/src/cli/gateway-cli/register.ts#L394)
  - sha: [51dd4f288f67](https://github.com/openclaw/openclaw/commit/51dd4f288f67ef4df4c7a121f078f508535df1fd)
- **User docs describe Gateway as first-class CLI:** The current public-facing docs instruct users to run the server with `openclaw gateway` or `openclaw gateway run`, matching the core CLI registration rather than a bundled-plugin command path.
  - file: [docs/cli/gateway.md:24](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/docs/cli/gateway.md#L24)
  - sha: [51dd4f288f67](https://github.com/openclaw/openclaw/commit/51dd4f288f67ef4df4c7a121f078f508535df1fd)
- **Windows absolute plugin paths are normalized before import:** `toSafeImportPath` explicitly rewrites Windows absolute plugin entry paths like `C:\...` into `file:///...` URLs before module loading. The issue’s failing paths were Windows absolute paths from a global npm install under `...\node_modules\openclaw\dist\extensions\...`.
  - file: [src/plugins/loader.ts:417](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/src/plugins/loader.ts#L417)
  - sha: [51dd4f288f67](https://github.com/openclaw/openclaw/commit/51dd4f288f67ef4df4c7a121f078f508535df1fd)
- **Bundled plugin load path now canonicalizes dist-runtime wrappers:** Real plugin loading resolves canonical `dist-runtime` wrapper sources and then imports the normalized path via Jiti, which is the exact hot path involved in bundled plugin startup failures.
  - file: [src/plugins/loader.ts:2404](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/src/plugins/loader.ts#L2404)
  - sha: [51dd4f288f67](https://github.com/openclaw/openclaw/commit/51dd4f288f67ef4df4c7a121f078f508535df1fd)
- **Regression test covers Windows plugin import rewriting:** Loader tests assert that Windows drive-letter and UNC plugin paths are converted to safe `file://` imports, so the Windows path handling is now guarded by test coverage.
  - file: [src/plugins/loader.test.ts:6600](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/src/plugins/loader.test.ts#L6600)
  - sha: [51dd4f288f67](https://github.com/openclaw/openclaw/commit/51dd4f288f67ef4df4c7a121f078f508535df1fd)
- **Shipped release contains the same implementation:** The latest shipped tag `v2026.4.23` already contains the inspected Gateway CLI registration and loader helpers (`registerGatewayCli`, `toSafeImportPath`, `resolveCanonicalDistRuntimeSource`), so this is not just unreleased main-branch behavior.
