## Summary

Keep this PR open, but do not merge it as written: it replaces an established Feishu session-continuity rule with a peer-id switch between starter and follow-up events, and the external contribution has no real Feishu behavior proof.

## What This Changes

The PR removes chat-type-sensitive Feishu topic-key selection and makes all topic-scoped group sessions prefer `thread_id` over `root_id`.

## Best Possible Solution

Preserve one canonical peer id throughout each event lifecycle: keep native `topic_group` sessions keyed by their stable thread identifier while retaining root/message-derived continuity for ordinary reply-created threads.

## Reproduction Assessment

Yes. The modified regression test routes the starter to `msg-topic-first` and the follow-up to `omt_topic_created`, directly demonstrating the proposed session split.

## Solution Assessment

No. Universal thread-id precedence deletes a deliberate distinction between native topics and ordinary reply trees; the narrower maintainable solution preserves stable identity for each event class.
