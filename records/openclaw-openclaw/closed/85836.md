## Summary

Close as implemented on current main: the observed old-process-after-package-swap failure is covered by the merged lifecycle-runtime SIGUSR1 hardening, and current `update.run` package-manager flow now uses a managed-service handoff instead of replacing the live Gateway package tree.

## What This Changes

The issue reports that Web UI `update.run` on a macOS LaunchAgent global-npm install leaves the old Gateway process running after disk has been updated.

## Best Possible Solution

Keep the current managed-service handoff plus lifecycle-runtime priming, and ship it in the next release so global-npm LaunchAgent users get the fresh-process update path.

## Reproduction Assessment

No current-main failing reproduction was established. The reporter's logs are high-signal for the pre-fix package-swap/SIGUSR1 failure, and current main now avoids the live package swap and eager-loads the SIGUSR1 lifecycle path.

## Solution Assessment

Yes. The current main solution is broader and safer than branching on config drift: package updates are handed outside the live Gateway process, and SIGUSR1 restart code is loaded before disk chunk rotation can break dynamic imports.
