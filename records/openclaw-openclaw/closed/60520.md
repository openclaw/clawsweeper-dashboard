## Summary

Current main already implements the PR's intended routed-reply fixes through the outbound delivery pipeline and channel-owned transport hooks. Reply tags are normalized in the generic outbound payload plan, Slack directives are compiled by the Slack outbound adapter, and Mattermost has an explicit channel-owned threadId-to-replyToId fallback. Current main also avoids the PR branch's reviewed Telegram regression because core does not apply a universal threadId-to-replyToId fallback.

## Best Possible Solution

Close this PR as already implemented on main and keep the shipped channel-owned transport resolver model. If more routed-reply regressions appear, add targeted tests around the production outbound pipeline rather than moving channel-specific thread fallback into core.
