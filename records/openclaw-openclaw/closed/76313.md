## Summary

Current `main` and the shipped `v2026.5.2` source already address the reported beta behavior: `gateway status --require-rpc` no longer depends on `sessions.list`, and managed gateway service generation/audit paths now keep gateway auth out of inline service env while pruning stale version-manager PATH entries.

## What This Changes

The issue reports that `2026.5.2-beta.2` gateway status times out on slow RPC/session listing and that systemd doctor/install repair keeps reporting embedded token and version-manager PATH drift.

## Best Possible Solution

Keep the shipped `v2026.5.2` implementation and ask for a fresh report only if the same repair loop or status timeout reproduces on `v2026.5.2` or newer with current command output.

## Reproduction Assessment

Yes for the beta report: the issue includes concrete command output, sanitized service lines, and journal timings. Against current source, the reported `sessions.list` health dependency is no longer present in the `--require-rpc` path.

## Solution Assessment

Yes. The current fix is generic: gateway health uses a cheap status RPC, and service repair is handled in shared systemd/env/audit code rather than adding Telegram-specific behavior.
