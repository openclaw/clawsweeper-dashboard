## Summary

Keep open: the PR exposes and documents cron `--webhook` delivery, but the current PR head still does not wire a runtime HTTP POST path, so users can create or edit jobs that appear accepted without delivering the promised webhook payload.

## What This Changes

This PR adds positional cron add/create parsing, public cron `--webhook` flags for add/create/edit, delivery cleanup, schedule identity handling, docs, and tests.

## Best Possible Solution

Land only after cron webhook delivery either performs a real HTTP POST for the supported job modes with live redacted proof, or the public flag/docs are removed or rejected until that contract exists.

## Reproduction Assessment

Yes from source inspection, though I did not run a live Gateway repro. `--webhook` stores webhook delivery, but webhook plans are marked not requested and the isolated/main execution paths never issue an HTTP POST.

## Solution Assessment

No. The ergonomic CLI shape is useful, but exposing and documenting `--webhook` before runtime dispatch exists is not the narrow maintainable solution; implement the dispatch path or reject/remove the flag until it is real.
