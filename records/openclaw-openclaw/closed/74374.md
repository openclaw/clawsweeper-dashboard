Keep open. Current main still has the reported DeepSeek disabled-thinking collision path, and two open closing PRs already target this bug, so the issue should remain open until one coordinated fix is reviewed and merged.

## What This Changes

The issue asks OpenClaw to prevent DeepSeek V4-pro disabled-thinking requests from replaying `reasoning_content` after the provider intermittently returns it.

## Best Possible Solution

Land one coordinated DeepSeek V4 payload-contract fix that normalizes disabled thinking aliases and cleans the final post-`extra_body` payload, with regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: per-model `params.thinking: false` or `"disabled"` falls through to DeepSeek V4's `high` default, then `extra_body.thinking.type=disabled` can overwrite the final payload after `reasoning_content` has been added.

## Solution Assessment

No. Current main is not the best solution yet; the narrow maintainable fix should cover both disabled-alias normalization and deterministic cleanup after final `extra_body` merging.
