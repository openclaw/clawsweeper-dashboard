## Summary

Keep open: this is a plausible agent-session improvement, but the current branch is not merge-ready because source review shows the new synthetic feedback path bypasses provider opt-out policy, can persist orphan toolResults, logs model-controlled names without a clear bound, and lacks contributor real behavior proof.

## What This Changes

This PR adds dropped-tool-call reporting, WARN logging for rejected assistant tool calls, synthetic error toolResults for fully dropped unknown or malformed calls, and related guard tests.

## Best Possible Solution

Keep bounded structured diagnostics, but redesign any model-visible unavailable-tool feedback so it respects allowSyntheticToolResults, remains paired and replay-visible to providers, and caps sanitized model-controlled log values.

## Reproduction Assessment

Yes for the blocking patch issues: PR head source shows full-drop synthetic toolResults written without an originating persisted assistant turn or allowSyntheticToolResults gate, while current main drops orphan toolResults during replay repair. The original repeated-call runtime problem still lacks contributor real behavior proof.

## Solution Assessment

No. Structured logging/reporting is a reasonable direction, but the proposed model-visible feedback should not bypass the existing synthetic-result policy or create orphan transcript entries.

  - body: Rejected tool names are model-controlled, and the missing-input or invalid-name paths can carry arbitrary-length strings. Cap and sanitize the display value before putting it in WARN text or structured log details to avoid log injection and log bloat.
  - confidence: 0.84
