## Summary

Current main implements the core interrupt behavior reported here: /stop is handled on a fast abort path before normal reply/model processing, clears queued session/follow-up work, aborts both reply and embedded Pi runs, and cascades to requester subagents. Command-only slash messages bypass the normal queue/model path, and the current pinned Pi runtime keeps the active-run streaming state true through tool execution so steering is delivered at the next tool/LLM boundary rather than being lost during tool gaps. Ordinary collect-mode messages still batch as follow-ups by design, with steer documented as the injection mode.

## Best Possible Solution

Keep the current fast-abort and command bypass behavior on main. If a future report finds a remaining specific gap, it should be narrowed to the affected control command or channel with fresh logs against a release that contains this current-main behavior; ordinary collect-mode follow-up batching should remain documented separately from steer injection.
