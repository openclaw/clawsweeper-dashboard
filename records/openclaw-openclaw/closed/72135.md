## Summary

Keep open: the reported cron false-positive path is still present on current main, but this PR is not safe to merge as written because its sentinel override trusts any payload text and the PR lacks inspectable after-fix real behavior proof.

## What This Changes

The branch adds whitespace-tolerant `===DONE_OK===` detection in `resolveCronPayloadOutcome` plus resolver tests so that sentinel text can make trailing error payloads non-fatal.

## Best Possible Solution

Accept only a trusted final or structured cron success signal that preserves run-level, failure-signal, and denial precedence, with focused resolver/runner coverage and redacted real-run proof.

## Reproduction Assessment

Yes, by source inspection. Current main has a resolver test where successful text followed by a later real error remains fatal, and `run.ts` maps that resolver flag directly to cron `status: "error"`.

## Solution Assessment

No. A success sentinel may be viable, but scanning every payload text is broader than the trusted final signal or structured metadata contract needed to avoid masking later real failures.
