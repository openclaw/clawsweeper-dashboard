## Summary

Current main already addresses the reported failure path for externally installed Brave-style runtime web-search providers: configured providers without current bundled metadata now keep runtime-provider discovery enabled, and regression coverage confirms `web_search` executes against an active runtime provider instead of returning provider unavailable.

## What This Changes

The issue asks for `web_search` to route to the configured Brave web-search provider when the Brave plugin is enabled and its API key is configured.

## Best Possible Solution

Keep the generic current-main runtime-provider discovery behavior and ship it in the next release rather than adding a Brave-specific routing special case.

## Reproduction Assessment

No current-main failing path is established. The release report is supported by its logs, but current main has source and regression-test coverage showing configured runtime providers remain discoverable when web-search metadata is missing.

## Solution Assessment

Yes. The current fix is at the generic web-search runtime-provider boundary, so it covers Brave and other external provider plugins without hard-coding one provider id.
