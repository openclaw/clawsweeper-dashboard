## Summary

Current main already addresses the reported Homebrew/nvm updater failure: package updates target the owning install prefix through staged npm installs, failed package-update exits restart the managed gateway, and macOS bootstrap recovery clears disabled launchd state.

## What This Changes

Report that `openclaw update` can leave the macOS gateway stopped and fail to update when a Homebrew-owned OpenClaw install is run with an nvm `npm` earlier on PATH.

## Best Possible Solution

Keep the current main updater behavior; the reverse nvm-only prefix mismatch remains better handled by the narrower thread at https://github.com/openclaw/openclaw/issues/78775 and its linked PR.

## Reproduction Assessment

No live high-confidence reproduction was run against current main. The old-version failure path is source-plausible from the report, but current main now routes the package update and restart recovery through code that avoids the described failure.

## Solution Assessment

Yes. The maintainable fix is in the updater and launchd recovery paths: target the owning install prefix, verify/swap staged package roots, and restart the managed gateway after package-update failure.
