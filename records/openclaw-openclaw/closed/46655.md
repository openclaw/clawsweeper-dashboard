## Summary

Current main does not show a reproducible path for stale inbound Telegram media to be re-attached to a later standalone text message. Telegram context media comes from the current message or an explicit reply target, and the shared keyed debouncer flushes pending buffered items before later non-debounced work instead of mixing them.

## Evidence

- **Telegram single-message flush uses only current message media:** When the debouncer flushes a single Telegram entry, it calls `processMessage` with `last.allMedia`, not previously buffered media from other turns.
  - file: [extensions/telegram/src/bot-handlers.runtime.ts:239](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/extensions/telegram/src/bot-handlers.runtime.ts#L239)
  - command: `nl -ba extensions/telegram/src/bot-handlers.runtime.ts | sed -n '234,250p'`
  - sha: [56fe2aab9c87](https://github.com/openclaw/openclaw/commit/56fe2aab9c87596515f2280c8060dc740ea64a88)
- **Extra inbound media only comes from explicit reply targets:** `resolveReplyMediaForMessage` returns media only when `msg.reply_to_message` exists and itself contains inbound media; otherwise it returns `[]`. There is no generic prior-turn media cache lookup here.
  - file: [extensions/telegram/src/bot-handlers.runtime.ts:489](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/extensions/telegram/src/bot-handlers.runtime.ts#L489)
  - command: `nl -ba extensions/telegram/src/bot-handlers.runtime.ts | sed -n '489,525p'`
  - sha: [56fe2aab9c87](https://github.com/openclaw/openclaw/commit/56fe2aab9c87596515f2280c8060dc740ea64a88)
- **Telegram context payload is built from current `allMedia` plus explicit `replyMedia` only:** `contextMedia` is `[...]currentMediaForContext, ...replyMedia]`, and those values become `MediaPath`/`MediaPaths` in the inbound context. No other previous-message media source is merged.
  - file: [extensions/telegram/src/bot-message-context.session.ts:320](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/extensions/telegram/src/bot-message-context.session.ts#L320)
  - command: `nl -ba extensions/telegram/src/bot-message-context.session.ts | sed -n '320,375p'`
  - sha: [56fe2aab9c87](https://github.com/openclaw/openclaw/commit/56fe2aab9c87596515f2280c8060dc740ea64a88)
- **Shared debouncer flushes pending buffered items before later non-debounced same-key work:** For non-debounced keyed items, the helper either flushes an existing buffer first or runs the item immediately when no buffer/key chain exists. That prevents a prior non-debounced photo turn from lingering in a buffer to be merged into a later text turn.
  - file: [src/auto-reply/inbound-debounce.ts:145](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/src/auto-reply/inbound-debounce.ts#L145)
  - command: `sed -n '145,178p' src/auto-reply/inbound-debounce.ts`
  - sha: [56fe2aab9c87](https://github.com/openclaw/openclaw/commit/56fe2aab9c87596515f2280c8060dc740ea64a88)
- **Debouncer behavior is covered by tests:** Core tests explicitly cover 'flushes buffered items before non-debounced item' and same-key ordering behind an active timer-backed flush, matching the path the issue comment proposed.
  - file: [src/auto-reply/inbound.test.ts:309](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/src/auto-reply/inbound.test.ts#L309)
