## Summary

Keep open: this is a focused fix for a credible embedded-runner timeout bug, but the current patch still lets thinking runs with explicit longer run/agent timeouts return through the 120s default idle watchdog, and the supplied helper-only proof does not show the live Gemini timeout fixed.

## What This Changes

The PR adds a `thinkingEnabled` parameter to embedded LLM idle-timeout resolution, passes non-off `thinkLevel` from `runEmbeddedAttempt`, and adds resolver tests for the new bypass.

## Best Possible Solution

Move the thinking bypass so it overrides only the implicit 120s idle watchdog while preserving explicit model/run/agent caps, add regression coverage for long explicit caps, and require redacted live Gemini/embedded proof before merge.

## Reproduction Assessment

Yes at source level: current main applies the 120s cloud idle watchdog without considering non-off thinking, and the linked issue provides matching 125-133s Gemini timeout logs. I did not establish a live Gemini reproduction in this read-only review.

## Solution Assessment

No as submitted. The direction is plausible, but the bypass is placed after branches that already clamp explicit longer run or agent timeouts back to the default idle watchdog.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Honor thinking before default timeout caps:** `src/agents/pi-embedded-runner/run/llm-idle-timeout.ts:187-193`
  - body: The new `thinkingEnabled` check runs only after `runTimeoutMs` and `agents.defaults.timeoutSeconds` have already returned through `clampImplicitTimeoutMs`, so a thinking run with a 300s run override or agent timeout still gets the 120s idle wrapper. That leaves the reported long pre-token thinking phase vulnerable to the same false timeout despite an explicit longer hard cap; handle `thinkingEnabled` before the implicit default clamp or add a dedicated thinking path that preserves explicit bounds.
  - confidence: 0.84
