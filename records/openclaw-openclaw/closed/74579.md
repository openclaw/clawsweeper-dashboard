## Summary

This should stay open. The report is fresh, concrete, release-specific, and current main still has the same native Ollama request shaping plus prompt-local runtime-context plumbing that could make a smaller local model attend to runtime/context metadata instead of the visible prompt; I found no source, docs, tests, or history evidence that the behavior has been fixed on main.

## What This Changes

The issue reports that native Ollama local models in clean TUI sessions answer OpenClaw runtime/context metadata instead of the user's prompt on OpenClaw 2026.4.26.

## Best Possible Solution

Reproduce with the reporter's diagnostics or a local Ollama daemon, then patch the owner boundary that actually causes runtime metadata to dominate the turn. The end state should keep the visible user prompt as the clear final task, keep hidden runtime context bounded and private for local models, verify compat.supportsTools:false resolves for the active model entry, and add focused coverage around native Ollama payload shaping plus context reporting.
