## Summary

Keep open: current main now clears same-process Telegram progress/tool-progress drafts before final delivery, but the restart case is still unresolved because active draft message IDs remain process-local and there is no persisted pending-draft cleanup path.

## What This Changes

The issue asks Telegram streaming to clear or replace progress draft messages across process restarts instead of leaving repeated or stacked "Sifting" drafts.

## Best Possible Solution

Telegram should persist only unfinished progress/tool-progress draft message IDs with chat/thread context, clear those stale pending drafts on startup or the next dispatch, and remove the record as soon as a draft is finalized or deleted.

## Reproduction Assessment

Yes, at source level: create a Telegram progress/tool-progress draft, restart before cleanup, and the new process has no streamMessageId for clear() to delete. I did not run a live Telegram restart in this read-only review.

## Solution Assessment

Yes: the narrow maintainable direction is Telegram-owned pending-draft persistence plus stale cleanup, not broad deletion of arbitrary old Telegram messages.
