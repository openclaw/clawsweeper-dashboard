## Summary

Current main and the shipped v2026.4.29 tree now solve the core plugin-observability problem: when an embedded run returns after a timeout/error path without a native terminal lifecycle event, `agent-command` emits a global lifecycle terminal backstop with `aborted: true`, and plugin agent-event subscriptions receive that through the existing event bus. The shipped payload uses `phase: "end"` plus `aborted: true` rather than the proposed `phase: "error"` and `errorType: "timeout"`, but existing session, task, and SDK code treats that structured terminal event as a timeout.

## What This Changes

Request an agent lifecycle event visible to plugins when an embedded run times out, preferably with structured timeout classification.

## Best Possible Solution

Keep the shipped terminal lifecycle backstop as the current timeout-observability contract, because it reuses the existing lifecycle terminal model and avoids duplicate terminal events. If maintainers want a more specific timeout taxonomy later, track that as a narrow additive payload-field enhancement rather than reopening the original silent-timeout gap.

## Reproduction Assessment

Yes. A run that times out without an embedded `agent_end` now follows the `abortRun(true)` path and then the `agent-command` fallback emits a global lifecycle terminal event with `aborted: true`; current tests and normalization code cover the timeout mapping from that terminal metadata.

## Solution Assessment

Yes. Emitting the backstop at the outer lifecycle boundary is safer than emitting directly inside the timeout timer because it only fires when the embedded loop did not already emit a terminal lifecycle event, reducing duplicate terminal-event risk while still giving plugins a structured timeout signal.
