## Summary

Keep this issue open. Current main still has the Telegram DM preflight echo gap: the DM voice note is transcribed for agent context, then marked as already transcribed, and the shared echo path only runs for newly produced audio transcription outputs. The issue is now the dedupe parent, and open PR #73866 is the active implementation candidate for the same regression.

## What This Changes

The issue requests that Telegram direct-chat voice-message transcripts be echoed back when `tools.media.audio.echoTranscript` is enabled, matching the working forum-group behavior.

## Best Possible Solution

Review and land PR #73866 or an equivalent small fix that threads the Telegram preflight transcript into a structured media-understanding echo path, sends the configured echo exactly once, preserves duplicate-STT avoidance, and adds regression coverage for the full Telegram DM voice-note handoff.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a Telegram private voice message with no text and `tools.media.audio.echoTranscript: true`: Telegram preflight transcribes it and marks the media index transcribed, then the shared media pass skips the audio and never enters the `sendTranscriptEcho` branch.

## Solution Assessment

Yes, the narrow fix direction is to repair the preflight-to-echo handoff rather than treating this as a Telegram address/routing failure. PR #73866 is the current candidate, but it still needs maintainer review for tight gating and Telegram-level coverage.
