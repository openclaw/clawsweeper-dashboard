## Summary

Current `main` already implements the requested behavior: explicit `cacheRetention` now flows through Anthropic-compatible custom providers such as Cloudflare AI Gateway, and explicit `cacheRetention: "long"` produces `cache_control.ttl: "1h"` on proxied/custom Anthropic hosts. The implementation landed after `v2026.4.22`, so it appears fixed on `main` but not yet in the latest tagged release.

## Evidence

- **Cloudflare gateway is routed through Anthropic Messages transport:** The bundled Cloudflare AI Gateway provider returns `api: "anthropic-messages"` with the Cloudflare gateway base URL, so it uses the Anthropic-compatible path that now receives the cache-retention fix.
  - file: [extensions/cloudflare-ai-gateway/catalog-provider.ts:69](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/extensions/cloudflare-ai-gateway/catalog-provider.ts#L69)

- **Explicit cacheRetention is now accepted for custom Anthropic-compatible providers:** `resolveCacheRetention` treats non-Bedrock `anthropic-messages` providers with explicit cache config as eligible, and passes through `short`/`long`/`none` instead of dropping the setting.
  - file: [src/agents/pi-embedded-runner/prompt-cache-retention.ts:17](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/src/agents/pi-embedded-runner/prompt-cache-retention.ts#L17)
