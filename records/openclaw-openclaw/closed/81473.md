## Summary

Current main already handles the central auth failure: URL tokens are applied before the first Control UI gateway connection, the gateway client sends `settings.token`, and token persistence is intentionally session-scoped rather than stored in localStorage.

## What This Changes

The issue reports that tokenized Control UI URLs may fail to persist the URL token before WebSocket authentication.

## Best Possible Solution

Keep the current main behavior: ingest URL tokens before connecting, send them through the gateway client, and keep auth tokens out of localStorage.

## Reproduction Assessment

No high-confidence failing reproduction exists for current main. Source and committed tests show the URL token is applied before the first gateway connection, while the reported localStorage check reflects an older storage contract.

## Solution Assessment

Yes. The current solution is the narrow maintainable path because it fixes the connection ordering while keeping gateway auth tokens out of persistent localStorage.
