Keep open: the PR targets a real Telegram warning bug, but the presence-only suppression can hide invalid defaultAccount fallback diagnostics and the supplied proof does not exercise the real CLI/resolver warning path after the patch.

## What This Changes

The branch changes Telegram account selection to suppress the missing accounts.default warning when channels.telegram.defaultAccount is set and adds a multi-account regression test.

## Best Possible Solution

Suppress the warning only when the normalized configured default actually resolves to a listed Telegram account, and preserve or replace the diagnostic for invalid defaultAccount fallback.

## Reproduction Assessment

Yes. Source inspection shows current main can warn when a valid defaultAccount is also the first sorted account id, and the PR diff shows the invalid-default fallback path would become silent.

## Solution Assessment

No, not as written. The intended suppression is right for valid explicit defaults, but presence-only gating should be replaced with listed-account validation or an invalid-default diagnostic.
