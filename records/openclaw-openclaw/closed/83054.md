## Summary

Keep open: current main still lets a channel restart rejection escape the hot-reload loop, so later channels are not attempted. The neighboring secrets.reload path already uses per-channel isolation, which makes this a focused gateway reliability fix rather than cleanup to close.

## What This Changes

The issue asks for gateway hot reload to keep restarting later channels when one channel stop/start fails during a multi-channel reload.

## Best Possible Solution

Add per-channel restart isolation in gateway hot reload, log or aggregate each failed channel restart, and cover stop/start failure cases with focused gateway reload tests.

## Reproduction Assessment

Yes, source-reproducible: a test harness can configure two `restartChannels`, make the first `stopChannel` or `startChannel` reject, and observe that current main exits the loop before the second channel is restarted. I did not run that test or a live gateway in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is to isolate channel restart failures inside the reload loop and add regression coverage that later channels are still attempted. A safer implementation should also preserve clear error logging so a failed channel restart is not reported as a fully clean reload.
