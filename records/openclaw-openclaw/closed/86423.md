## Label Justifications

- `P1`: Explicit cron allowlists can silently remove requested read/write tools while reporting status ok, breaking scheduled agent workflows for current users.
- `impact:data-loss`: The report describes scheduled memory consolidation writing nothing while runs complete ok, so expected persistent outputs can be silently lost.
- `impact:session-state`: The affected workload is memory consolidation, which can leave memory or session context stale when requested write tools vanish.

## Summary

Keep open: current main still removes native Codex/OpenClaw tool names from the dynamic tool list before applying an explicit allowlist, while the native Codex surface only turns on for an absent allowlist or a wildcard. That matches the reported failure mode for explicit cron `toolsAllow` lists that name `read`, `write`, `edit`, or `exec`.

## What This Changes

The issue requests that Codex app-server isolated cron `toolsAllow` entries such as `read`, `write`, `edit`, and `exec` expose those tools or fail loudly instead of completing successfully without them.

## Best Possible Solution

Keep broad Codex-native access behind the existing wildcard/full-surface path, but make explicit native-owned toolsAllow entries deterministic by exposing only enforceable requested tools through the OpenClaw dynamic surface or failing with a clear diagnostic when exact enforcement is impossible.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: pass an explicit non-wildcard toolsAllow containing native names and the native surface returns false while those same names have already been excluded from dynamic tools. I did not execute a live cron in this read-only review.

## Solution Assessment

No, enabling the whole native surface for any native tool name is not the safest fix because current source treats Codex native code mode as one broad app-server capability. The narrower solution is exact allowlist enforcement through dynamic OpenClaw tools, or a clear fail-closed error when exact enforcement cannot be honored.
