## Summary

Keep open: current main and v2026.6.6 still call the task registry summary and audit helper separately, and each path reconciles tasks before reaching the clone-and-sort task list. The related fix PRs are closed unmerged, so no canonical fix currently owns the work.

## What This Changes

The issue asks to remove duplicate `reconcileInspectableTasks()` work from `openclaw status`, ideally by reusing one in-memory snapshot for the registry summary and task audit.

## Best Possible Solution

Compute one reconciled task snapshot inside `getStatusSummary()` and pass it to the registry summary and task audit helpers, preserving zero-argument helper behavior for other callers.

## Reproduction Assessment

Yes by source inspection: current main calls two status task helper paths that each call `reconcileInspectableTasks()` before `listTaskRecords()` clones and sorts task records. I did not run a live large-ledger timing test in this read-only review.

## Solution Assessment

Yes: the per-status shared-snapshot direction is the narrowest maintainable fix. A module-level TTL cache is less safe unless it also owns task-registry mutation invalidation and freshness tests.
