## Summary

Keep open: current main still lacks a public per-job `acceptSilentStop` or equivalent cron policy, and recent production comments confirm the false-positive empty-stop failure still matters. The open related PR only covers the narrower `delivery.mode: "none"` handoff, so the broader flag needs maintainer judgment.

## What This Changes

Add a per-job cron payload flag that treats clean `stopReason="stop"` empty assistant turns as successful silent runs.

## Best Possible Solution

Make an explicit cron silent-success policy decision: either add a documented `acceptSilentStop`-style payload field and thread it into `allowEmptyAssistantReplyAsSilent`, or deliberately scope intentional empty stops to the existing no-delivery path.

## Reproduction Assessment

Yes, source-level. Current main has the embedded-runner opt-in but the cron payload/type/schema/executor do not expose or forward the requested policy, and commenters report matching production failures on recent releases.

## Solution Assessment

Unclear until maintainer signoff. Reusing `allowEmptyAssistantReplyAsSilent` is the right implementation seam, but adding `acceptSilentStop` changes public cron configuration and success semantics; https://github.com/openclaw/openclaw/pull/81374 covers only the narrower no-delivery path.
