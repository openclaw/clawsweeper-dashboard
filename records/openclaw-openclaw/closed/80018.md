## Summary

Current main already documents and implements the Codex runtime data-flow split: OpenClaw remains the owner of channels, session files, dynamic tools, approvals, media delivery, and transcript mirroring, while Codex owns the native model loop and canonical native thread internals.

## What This Changes

The issue asks whether Codex-related data should go directly to Codex in parallel or be handled through OpenClaw when the Codex runtime is selected.

## Best Possible Solution

Keep the documented split: OpenClaw is the integration and channel funnel for OpenClaw-owned surfaces, while Codex remains canonical only for its native model loop and thread internals.

## Reproduction Assessment

Not applicable: this is an architecture and documentation clarification request, not a runtime bug. The answer is source-reproducible from the current docs and Codex harness implementation.

## Solution Assessment

Yes: the current docs and code express the narrow maintainable contract without adding a second data path. A future change would only be needed for unsupported Codex-native internals such as editable native thread history or byte-for-byte model request capture.
