## Summary

Keep open: this is a protected maintainer-labeled, MEMBER-authored iOS PR and current main still has the missing-prompt behavior. The patch direction is useful, but I found a blocking Calendar add edge case that should be fixed before merge.

## What This Changes

The PR adds iOS privacy usage strings, a permission request bridge, Contacts/Calendar/Reminders first-use prompts, a Settings Privacy & Access section, tests, and a changelog entry.

## Best Possible Solution

Keep the privacy prompt restoration, but make Calendar add request full event access when a target calendar is specified and reserve write-only access for default-calendar adds.

## Reproduction Assessment

Yes from source inspection: current main returns false for `.notDetermined` Contacts/Calendar/Reminders access and lacks the added usage strings. The review finding is also source-reproducible because targeted `calendar.add` can request write-only access and then call calendar-reading resolution code.

## Solution Assessment

No, not yet. The overall direction is maintainable, but the Calendar add permission selection needs to distinguish default write-only adds from targeted-calendar adds that require full access.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Request full calendar access for targeted adds:** `apps/ios/Sources/Calendar/CalendarService.swift:46`
  - body: When `calendar.add` includes `calendarId` or `calendarTitle` on a fresh install, this branch grants only write-only calendar access and then immediately calls `resolveCalendar`, which reads existing calendars. Apple documents write-only access as unable to read events or other calendar information, so targeted adds can fail after the user accepts the prompt; request full access for targeted adds and keep write-only for default-calendar adds. ([developer.apple.com](https://developer.apple.com/documentation/eventkit/accessing-the-event-store?changes=_7&utm_source=openai))
  - confidence: 0.86
