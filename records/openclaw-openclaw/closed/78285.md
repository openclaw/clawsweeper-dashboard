## Summary

Keep open: source inspection confirms iMessage lacks the schema/type support and inbound `GroupSystemPrompt` wire-through needed for this request, and the protected `maintainer` label requires explicit maintainer handling.

## What This Changes

Requests iMessage per-group `systemPrompt` support by accepting `channels.imessage.groups.*.systemPrompt`, resolving it for inbound group chats, forwarding it as `GroupSystemPrompt`, and documenting the behavior.

## Best Possible Solution

Add an extension-owned iMessage config field and inbound resolver for trusted per-group prompts, with clear `chat_id`/`chat_guid`/wildcard semantics, regression tests, generated config docs, and an `imessage` docs section.

## Reproduction Assessment

Yes, source-level: current main has no iMessage `groups.*.systemPrompt` schema/type field and never sets `GroupSystemPrompt` in iMessage inbound context. I did not run a live iMessage message through the transport.

## Solution Assessment

Unclear as written: the owner-extension wire-through is the right boundary, but a one-line pass-through is incomplete without schema/types/docs and a decision on iMessage group identifier matching.
