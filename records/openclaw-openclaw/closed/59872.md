## Summary

Close as ClawHub/plugin scope. Current main still keeps `web_fetch` as a generic HTTP fetch tool and does not ship Twitter/X-specific routing, but the requested domain rewrite/blocking behavior can be implemented at the existing plugin hook layer without adding a core special case.

## What This Changes

The issue requests tool-level domain routing, blocking, or description overrides so Twitter/X URLs are not handled by the default `web_fetch` path.

## Best Possible Solution

Keep `web_fetch` generic in OpenClaw core and implement Twitter/X-specific routing as a ClawHub plugin using `before_tool_call`, with docs pointing users to `x_search` or browser-based alternatives when appropriate.

## Reproduction Assessment

Yes for the tool-layer gap: current source inspection shows `web_fetch` accepts any HTTP(S) URL and fetches the original URL without domain routing. The model-choice complaint itself is behavioral, but the missing enforcement layer is directly visible in current main.

## Solution Assessment

Yes for closing this core item: the narrow maintainable path is a ClawHub/plugin implementation because the existing hook API can rewrite or block `web_fetch` calls for specific domains without adding a Twitter/X special case to core.
