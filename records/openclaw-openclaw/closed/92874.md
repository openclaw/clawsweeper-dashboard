## Summary

Keep open: current main still creates atomic reindex temp SQLite files and only removes them through the in-process success/error path; no startup sweep for orphaned hard-restart leftovers is present. The report is narrow, source-backed, and fits the stability cleanup lane.

## What This Changes

The issue reports that builtin memory atomic reindex can leave `<store>/<name>.sqlite.tmp-<uuid>{,-wal,-shm}` files after a hard restart and asks for safe startup cleanup.

## Best Possible Solution

Add a bounded memory-core startup sweep for aged sibling `<db>.tmp-*` SQLite triplets before the live DB is opened or cached, reusing existing sidecar removal behavior and preserving young/live temp files.

## Reproduction Assessment

Yes, from source inspection: a hard kill during `runSafeReindex()` can leave the temp DB because cleanup is only in the success/error JavaScript path, and current startup opens the live DB without scanning sibling temp files. I did not run a destructive kill repro in this read-only review.

## Solution Assessment

Yes, a narrow startup or manager-creation cleanup in memory-core is the right owner boundary, as long as it only removes aged sibling temp triplets and avoids racing active reindexes. Adding a config flag or broad doctor migration would be heavier than the source-backed bug requires.
