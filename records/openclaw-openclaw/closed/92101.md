## Summary

This is a valid feature request for the bundled memory-wiki plugin, but it is not ready for autonomous implementation: the five-directory layout is a shipped cross-surface contract, and maintainers need to decide whether custom layouts are supported and whether vault-local configuration may coexist with the canonical plugin config.

## What This Changes

Add configurable memory-wiki page groups, optional vault-local layout overrides, and recursive Markdown discovery in configured directories.

## Best Possible Solution

Decide the supported vault-layout contract first; if custom layouts are accepted, implement one canonical plugin-owned resolved page-group model used by every reader, writer, index, report, linker, validator, and doc surface, and specify recursive discovery separately with focused path-safety and nested-link tests.

## Reproduction Assessment

Not applicable. Current main and v2026.6.5 consistently document and implement a fixed non-recursive vault layout, so the issue describes a missing capability rather than broken established behavior.

## Solution Assessment

No, not as proposed. Configurable groups may be useful, but dual configuration sources and a query-centered implementation are not the narrowest maintainable design because the layout is shared by every vault read/write surface.

## Risks / Open Questions

- The proposed `.wiki-page-groups.json` creates a second configuration authority beside `plugins.entries.memory-wiki.config`, with precedence and drift that repository policy does not currently endorse.
- The five directories are a shipped contract across initialization, creation routing, querying, compilation, indexes, generated dashboards, links, lint, docs, and tests; partial conversion would produce inconsistent vault behavior.
- Recursive discovery needs explicit rules for generated `index.md` files, nested relative links, duplicate or overlapping group roots, symlinks, and pages whose frontmatter kind conflicts with the configured group.
