## Summary

Current `main` already preserves Control UI pasted image attachments for text-only WebChat/Gateway sessions instead of dropping them, which matches the issue report.

## Evidence

- **Control UI still sends pasted images as chat attachments:** The Control UI paste flow reads clipboard images into `ChatAttachment` objects, and `chat.send` serializes them as base64 `attachments` payloads rather than discarding them.
  - file: [ui/src/ui/views/chat.ts:206](https://github.com/openclaw/openclaw/blob/350d3221800a18283915cd2f3b612ae04ddd490c/ui/src/ui/views/chat.ts#L206)
  - command: `nl -ba ui/src/ui/views/chat.ts | sed -n '206,239p'; nl -ba ui/src/ui/controllers/chat.ts | sed -n '213,242p'`
  - sha: [350d3221800a](https://github.com/openclaw/openclaw/commit/350d3221800a18283915cd2f3b612ae04ddd490c)
- **Current main no longer drops attachments for text-only models:** `parseMessageWithAttachments()` now treats `supportsImages === false` as a force-offload path, appending `media://inbound/*` refs instead of returning no images/offloaded refs. This directly addresses the reported 'image disappears and agent receives nothing' behavior.
  - file: [src/gateway/chat-attachments.ts:309](https://github.com/openclaw/openclaw/blob/350d3221800a18283915cd2f3b612ae04ddd490c/src/gateway/chat-attachments.ts#L309)

- **Gateway chat handler wires the text-only path through:** `chat.send` resolves whether the session model supports images and passes `supportsImages` into `parseMessageWithAttachments()`, then persists `offloadedRefs` for the run.
  - file: [src/gateway/server-methods/chat.ts:2204](https://github.com/openclaw/openclaw/blob/350d3221800a18283915cd2f3b612ae04ddd490c/src/gateway/server-methods/chat.ts#L2204)

  - file: [src/gateway/chat-attachments.test.ts:146](https://github.com/openclaw/openclaw/blob/350d3221800a18283915cd2f3b612ae04ddd490c/src/gateway/chat-attachments.test.ts#L146)
