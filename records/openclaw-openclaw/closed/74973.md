## Summary

Current main already fixes the reported LLM-context pollution path. `delivery-mirror` remains a persisted transcript artifact, but model replay now drops transcript-only OpenClaw assistant rows during `sanitizeSessionHistory()` and again immediately before prompt submission, covering the later session-context rebuild path described in the report. The same implementation is present in the latest release `v2026.4.27`, so the reported `2026.4.12` installation is behind the shipped fix.

## What This Changes

The issue asks for `delivery-mirror` assistant transcript entries to be excluded from LLM context so outbound message/file delivery cannot self-trigger repeated `message` tool calls.

## Best Possible Solution

Keep the current in-memory replay filtering: persisted transcript rows remain available for audit/user-facing history, while transcript-only OpenClaw assistant artifacts stay out of provider context. Any broader WebChat/API consumer-path cleanup can continue under the existing broader delivery-mirror track, but this LLM feedback-loop report no longer needs a separate core issue.

## Reproduction Assessment

Yes. The static reproduction path is to put a `provider=openclaw`, `model=delivery-mirror` assistant row in history and pass it through `sanitizeSessionHistory()` or `normalizeAssistantReplayContent()`; current source and regression coverage show it is dropped before model replay.

## Solution Assessment

Yes. Filtering at the replay/LLM boundary is the narrow maintainable fix because it preserves transcript bookkeeping without changing stored session roles or migrating old JSONL, and the final prompt-boundary guard covers later raw context rebuilds.
