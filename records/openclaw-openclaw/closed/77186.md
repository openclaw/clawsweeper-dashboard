## Summary

Current main already fixes the reported Control UI bootstrap-auth failure and keeps the WhatsApp QR login path wired; the implementation is also present in the shipped v2026.5.3 release, while the report is against 2026.5.2.

## What This Changes

The issue reports that WhatsApp channel linking cannot show a QR code because the Control UI request for `/openclaw/control-ui-config.json` returns `401 Unauthorized`.

## Best Possible Solution

Keep the v2026.5.3/current-main implementation and direct affected 2026.5.2 users to upgrade, with a new report if v2026.5.3 still returns 401 using a valid token.

## Reproduction Assessment

No, not on current main: source and tests show the bootstrap config request is now base-path-aware and authenticated with the dashboard bearer credential. The reviewed path covers the reported 401 condition without needing a new live WSL2/WhatsApp run.

## Solution Assessment

Yes. The narrow maintainable solution is already in place: Control UI supplies the existing gateway credential to the auth-gated bootstrap endpoint, and the existing WhatsApp QR provider continues to return the QR data URL.
