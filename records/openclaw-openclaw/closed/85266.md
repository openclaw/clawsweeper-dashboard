## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Parse retry-after-ms as milliseconds:** `src/agents/provider-rate-limiter.ts:159-161`
  - body: `retry-after-ms` is already treated as milliseconds in the current transport code, but this parser sends it through `parseRetryAfterMs()`, whose numeric branch multiplies by 1000 as seconds. A header like `retry-after-ms: 90000` would delay MiniMax traffic for about 25 hours instead of 90 seconds.
  - confidence: 0.96
- **[P1] Thread the real auth profile into limiter keys:** `src/agents/provider-transport-fetch.ts:586-588`
  - body: The limiter key reads `authProfileId` from `model`, but current runtime code keeps the selected auth profile separate and the OpenAI/Anthropic fetch callers pass only `model` into `buildGuardedModelFetch`. That collapses multiple MiniMax profiles into the `default` bucket, so one account's cooldown can throttle another account.
  - confidence: 0.92
- **[P2] Rollback limiter reservations on abort:** `src/agents/provider-rate-limiter.ts:340-355`
  - body: `reserveProviderLimiterSlot()` records minute/week usage before the sleep, and an abort during that sleep rejects without removing the reservation. Canceled requests can therefore consume RPM/TPM and weekly budget even though no provider call was made.
  - confidence: 0.9
- **[P2] Avoid counting stream-backed requests as one token:** `src/agents/provider-rate-limiter.ts:208-209`
  - body: When the fetch input is a `Request`, the body handed to the limiter is typically a `ReadableStream`; `estimateProviderRequestTokens()` returns `1` for every non-string body. That makes the TPM and weekly-token limiter ineffective for those SDK call sites unless the transport uses content length or handles common binary body types.
  - confidence: 0.82
