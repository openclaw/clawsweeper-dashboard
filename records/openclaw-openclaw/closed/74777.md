## Summary

Keep open. Current main still leaves the central local embeddings failure unresolved: `node-llama-cpp` is imported by bare specifier from the local memory runtime but is not retained by the managed runtime-deps plan, and an open linked PR specifically targets this issue with closing syntax.

## What This Changes

The issue asks OpenClaw to retain or reliably resolve local memorySearch runtime dependencies in `plugin-runtime-deps`, especially `node-llama-cpp`, `sqlite-vec`, and `chokidar`, or provide a durable repair path.

## Best Possible Solution

Land #74802 or an equivalent narrow fix that installs and retains `node-llama-cpp@3.18.1` through the managed runtime-deps plan when local memory search is selected, while keeping the existing release guardrail against normal root dependency inclusion and updating setup hints/docs to the durable repair path.

## Reproduction Assessment

Yes. The report gives a concrete v2026.4.27 setup with `agents.defaults.memorySearch.provider = "local"`, a readable GGUF model, and `openclaw memory status --deep`; static current-main inspection confirms the same bare import remains unmanaged.

## Solution Assessment

Yes, the best fix direction is a config-gated managed runtime-deps repair/selection path rather than relying on global npm resolution. The linked open PR appears to target that path, but it still needs normal maintainer review and merge before this issue can close.
