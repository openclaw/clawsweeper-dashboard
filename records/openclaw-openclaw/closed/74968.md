## Summary

Close as implemented on current main. The reported npm-global missing-`sharp` failure is now handled by the plugin runtime-dependency path: media image optimization loads the `media-understanding-core` public artifact, that plugin owns `sharp`, packaged public surfaces prepare staged runtime dependencies before loading, and doctor documents repair of missing bundled plugin runtime dependencies without writing into the global install root.

## What This Changes

The issue requests that macOS npm-global image attachment processing recover from or clearly diagnose a missing `sharp` native dependency after `npm install -g openclaw`.

## Best Possible Solution

Keep `sharp` behind the media-understanding plugin runtime-dependency staging and doctor repair flow, so packaged installs do not rely on mutating `/usr/local/lib/node_modules/openclaw` and missing native dependencies are repaired in OpenClaw-managed stage roots.

## Reproduction Assessment

Yes. The report gives a concrete affected-version path: macOS Apple Silicon global npm install with `node_modules/sharp` missing, any image attachment, and logs showing the optional `sharp` dependency error. I did not run a macOS global install in this read-only sweep; the current-main verification is source, docs, and regression-test based.

## Solution Assessment

Yes. The current implementation is the narrow maintainable fix because `sharp` is owned by `media-understanding-core` and is prepared through bundled plugin runtime-deps staging rather than by making the OpenClaw core package or global npm root own that native dependency.
