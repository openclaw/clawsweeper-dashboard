## Summary

Keep open: this maintainer-labeled PR needs review, and the diff has a blocking regression because it removes private QQBot admin commands from the framework auth path without preserving documented `commands.allowFrom.qqbot` authorization on the direct QQBot path.

## What This Changes

The PR filters QQBot `requireAuth` commands marked `c2cOnly` out of framework command registration, updates comments, and adds unit coverage for the new registry behavior.

## Best Possible Solution

Keep the private-command surface narrowing only after the QQBot direct command path preserves the shared command authorization contract, then add regression coverage and a changelog entry.

## Reproduction Assessment

Yes by source inspection: current main exposes `bot-approve` through `getFrameworkCommands()`, and the PR would remove that path for all `c2cOnly` auth commands. The remaining direct QQBot path can be traced to channel `allowFrom`/`groupAllowFrom` only.

## Solution Assessment

No. Filtering private commands is a reasonable hardening direction, but the patch should first preserve `commands.allowFrom.qqbot` semantics on the direct QQBot path or make an explicit maintainer-approved contract change.
