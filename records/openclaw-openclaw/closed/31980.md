## Summary

Current main no longer matches the issue’s reported startup path. Gateway single-instance enforcement is acquired before server startup via a dedicated gateway lock, with stale-owner validation by PID/start-time/port checks, and startup has regression coverage that the socket is not reachable until the gateway is fully ready. That removes the self-probe/self-detection race described for 2026.2.26.

## Close Comment

Closing from a Codex review because current `main` already removes the startup path described here. The gateway now acquires singleton ownership before server startup in `src/cli/gateway-cli/run-loop.ts:35`, with stale-owner recovery in `src/infra/gateway-lock.ts:246`; that is a lock-file/PID/start-time path, not a self-probe against the gateway’s own port. Startup ordering is also hardened so upgrade handlers are attached before any listen occurs (`src/gateway/server-runtime-state.ts:172`), and there is regression coverage that the port stays unreachable until startup is ready (`src/gateway/server.startup-websocket-race.test.ts:48`). This code is present in the shipped `v2026.4.22` release (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`), so the false positive reported against `2026.2.26` is implemented on main.
