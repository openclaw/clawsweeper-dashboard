## Summary

Current main already addresses the reported fallback stickiness: reset clears auto-owned fallback model state, auto-pinned fallback sessions probe the primary again, and external aborts do not trigger model fallback escalation.

## What This Changes

The issue asks for runtime fallback model selection to return to the configured primary after an abort or `/new`, instead of staying pinned to the fallback model.

## Best Possible Solution

Keep the current main implementation and let the next release carry the reset/probe fix forward.

## Reproduction Assessment

Yes, the reported release behavior is source-traceable from persisted fallback override/session reset paths and the issue includes concrete logs. I did not run a live Telegram/Claude repro in this read-only review; current main has regression coverage for the reset and abort decision paths.

## Solution Assessment

Yes. Clearing only auto-owned fallback state, preserving explicit user model choices, re-probing the primary, and suppressing fallback escalation on external aborts is narrower than adding a new TTL/config policy.
