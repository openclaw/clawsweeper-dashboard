## Summary

Keep open: current main and v2026.5.12 still dispose stdio MCP runtimes through the wrapper process tree only, with no Docker/container ownership or removal path for wrappers that leave containers running.

## What This Changes

Ensure stdio MCP cleanup for parallel `sessions_spawn` also terminates Docker containers launched by npm/npx wrapper MCP servers.

## Best Possible Solution

Add session-owned container tracking or labeling for Docker-wrapped stdio MCP servers, then remove only containers proven to belong to that OpenClaw MCP runtime while preserving process-tree signaling as the generic fallback.

## Reproduction Assessment

No high-confidence live reproduction was run against current main. The issue includes live Docker evidence, and source inspection shows current cleanup is still wrapper/process-tree-only, but I did not execute the UniFi Docker-wrapped MCP server through parallel `sessions_spawn`.

## Solution Assessment

No for the suggested immediate broad `docker rm` cleanup. The safer maintainable fix is to establish OpenClaw-owned container identity, such as labels or recorded container ids, before deleting containers during MCP runtime disposal.
