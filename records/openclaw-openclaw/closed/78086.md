## Summary

Keep open: this feature PR still has blocking source-level correctness issues in the new suspension/handoff paths, and the current proof section remains placeholder text rather than real after-fix runtime evidence.

## What This Changes

The branch adds quota-aware lane suspension, persisted session suspension state, fallback handoff injection, model.failover diagnostics/OTLP export, tests, and a changelog entry for agent failover.

## Best Possible Solution

Rework the branch so lane suspension only trips for the intended quota/billing/circuit cases, persists and consumes durable handoff state correctly, emits diagnostics for every suspension path, includes focused regression tests, and replaces placeholder proof with redacted runtime output.

## Reproduction Assessment

Yes for the PR defects: the problematic paths are directly visible in the latest PR head source. No current-main bug reproduction applies because this is a new feature PR, and the PR body does not provide real after-fix runtime proof.

## Solution Assessment

No; the current patch is not the best way to solve the issue because it can suspend lanes for unrelated fallback failures and does not yet satisfy the promised diagnostic and handoff contracts. A safer fix should narrow the suspension trigger and prove the runtime state machine end to end.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Gate lane suspension to suspend-worthy failures:** `src/agents/model-fallback.ts:412-421`
  - body: This summary path suspends any attributed session whenever multiple model candidates fail, regardless of the recorded reasons. Auth, format, model-not-found, or other non-quota fallback exhaustion can drop the main lane to concurrency 0 for the full TTL, so normal user turns can sit queued even though no quota/billing circuit was tripped.
  - confidence: 0.9
- **[P2] Emit failover diagnostics when suspending from runner paths:** `src/agents/pi-embedded-runner/run.ts:1882-1892`
  - body: The prompt and assistant suspension paths call `suspendSession` directly but never emit `model.failover`; only the cooldown skip path does. A real provider 429/402 suspension from these runner paths therefore will not show up in the diagnostic ring or OTLP exporter despite the feature contract.
  - confidence: 0.86
- **[P2] Capture the handoff summary before persisting suspension:** `src/agents/pi-embedded-runner/run.ts:1883-1892`
  - body: All new `suspendSession` call sites omit `summary`, and the new `summarizeForHandoff` helper is not called. The resume hook will inject `No recovery briefing was captured.` instead of the structured recovery briefing this PR promises.
  - confidence: 0.91
- **[P2] Preserve suspended records until handoff is consumed:** `src/config/sessions/store-maintenance.ts:241-247`
  - body: Maintenance clears any `quotaSuspension` once `now >= expectedResumeBy + ttl` before checking whether it is still `suspended`. If the process is down or the session is idle past that second TTL, the record is removed without ever entering `resuming`, so the next attempt gets no handoff.
  - confidence: 0.87
