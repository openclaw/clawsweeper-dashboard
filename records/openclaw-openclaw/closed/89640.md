- `P1`: The PR fixes a real channel workflow failure where sub-agent completion announcements can be delivered repeatedly after a post-send transcript mirror failure.

## Summary

Keep open for normal merge review: current main still lets a post-send transcript mirror throw after the channel delivery, while this PR confines that best-effort mirror failure to a warning and preserves the already-successful send.

## What This Changes

The PR wraps post-delivery transcript mirroring in `deliverOutboundPayloads` with warning-only handling for thrown and not-ok append failures and adds two regression tests.

## Best Possible Solution

Merge a narrow fix that treats post-send transcript mirrors as best-effort while preserving channel send results and idempotency, with regression coverage for thrown and not-ok mirror failures.

## Reproduction Assessment

Yes: the current-main source path sends first, then awaits an unguarded transcript mirror append, so an injected lock failure at that boundary reproduces the post-send throw described in the linked report. I did not run the live Telegram/Raspberry Pi scenario in this read-only review.

## Solution Assessment

Yes: guarding only the post-delivery mirror is the narrow owner-boundary fix, and it matches the existing cron sibling behavior without changing platform send semantics or adding a broader retry policy.
