## Summary

Current main and the latest release already handle the reported corrupt/partially written session JSONL failure class: runs repair malformed transcript lines before opening the session, and `SessionManager.open` recovers corrupt headers or starts a clean file instead of replaying a broken transcript.

## What This Changes

The issue asks for gateway restart/session startup to avoid loading a partially wiped or corrupt session JSONL after manual session file deletion.

## Best Possible Solution

Keep the shipped integrity-repair path and steer users to current release behavior rather than adding gateway waits for arbitrary external shell file operations.

## Reproduction Assessment

Yes, from source and regression tests for the failure class: malformed or corrupt session JSONL is the relevant input, and current code now repairs or recovers it. I did not run a live destructive `rm` race in this read-only review.

## Solution Assessment

Yes. Validating and repairing session integrity at run/session load is the safer durable fix than trying to coordinate gateway restart with arbitrary external shell deletions.
