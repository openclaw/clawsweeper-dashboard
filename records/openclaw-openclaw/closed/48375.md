## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Enforce controlUrl in tailnet mode:** `src/gateway/server-runtime-config.ts:160-162`
  - body: The advertised Headscale path is `controlUrl` with `tailscale.mode="off"` and `bind="tailnet"`, but this condition only reads daemon state when `controlUrl` is absent and mode is `serve`/`funnel`. With `controlUrl` set, a typo or stale value is accepted and the gateway binds to whichever tailnet the daemon is already on, making the new setting a no-op in its only supported mode.
  - confidence: 0.9
- **[P2] Use a contracted daemon status source:** `src/gateway/server-runtime-config.ts:41-44`
  - body: `readDaemonControlServerUrl()` probes `Self.ControlURL`, `CurrentTailnet.ControlURL`, `Prefs.ControlURL`, and top-level `ControlURL`, but Tailscale's upstream status type does not expose those fields. That means an existing Headscale daemon with `serve`/`funnel` and no config `controlUrl` can still fail open into the old opaque Serve/Funnel path instead of the clear fast-fail this PR intends.
  - confidence: 0.84
- **[P2] Preserve official control URLs at runtime:** `src/gateway/server-runtime-config.ts:153`
  - body: Config validation now allows official Tailscale control URLs with `serve`/`funnel`, but this runtime guard throws for any non-empty `tailscaleConfig.controlUrl` before consulting `isOfficialTailscaleControlServer()`. A config or CLI invocation that explicitly pins `https://controlplane.tailscale.com` is accepted by schema validation and then fails at startup.
  - confidence: 0.88
- **[P2] Redact daemon control URLs in errors:** `src/gateway/server-runtime-config.ts:166-167`
  - body: The startup error embeds `daemonControlUrl` verbatim. If the daemon/login-server source ever contains userinfo or query tokens, this writes sensitive control-plane material to normal CLI and gateway logs; strip userinfo, query, and fragment before including it in diagnostics.
  - confidence: 0.86
