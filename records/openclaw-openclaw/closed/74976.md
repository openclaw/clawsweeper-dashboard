## Summary

Close as implemented on current main. The report matches a Windows bundled-plugin loader path in the latest release: `v2026.4.27` disabled native Jiti loading on `win32`, while `@mariozechner/pi-ai@0.70.6` exposes `./oauth` only for ESM import. Current main includes a targeted post-release fix that enables the native compiled-plugin path on Windows and passes `allowWindows: true`, so built bundled plugin code no longer has to CJS-resolve `@mariozechner/pi-ai/oauth`. The fix is present on main only; no release tag contains it yet.

## What This Changes

The issue reports that OpenClaw 2026.4.27 can hit a CommonJS resolution path for `@mariozechner/pi-ai/oauth`, causing `ERR_PACKAGE_PATH_NOT_EXPORTED` during Telegram embedded-agent gateway sessions.

## Best Possible Solution

Keep the current main loader behavior and ship it in the next release; OpenClaw should avoid patching the third-party package export map unless a remaining runtime path still requires CJS resolution of the import-only OAuth subpath.

## Reproduction Assessment

Yes. The high-confidence reproduction is the `v2026.4.27` Windows loader path: built bundled plugin modules were not native-loaded on `win32`, and the Pi OAuth subpath has no CJS export.

## Solution Assessment

Yes. The current-main fix is the narrowest maintainable direction because it keeps compiled bundled plugins on the native ESM-capable path on Windows instead of modifying a third-party dependency or adding core-specific package export patches.
