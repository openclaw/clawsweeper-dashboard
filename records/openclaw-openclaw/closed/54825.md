## Summary

Current main and the latest stable release now cover the central user problem: WhatsApp drops error payloads before delivery, raw provider error pass-through is sanitized at the shared user-facing payload boundary, and Telegram has suppression controls for error replies.

## What This Changes

The issue requests suppression, customization, or owner-only routing for provider/runtime error replies sent to WhatsApp and Telegram users.

## Best Possible Solution

Keep the shipped WhatsApp suppression and shared raw-error sanitizer; handle any future custom-message or owner-only routing as a new narrow product proposal rather than keeping this broad leak report open.

## Reproduction Assessment

Not applicable as an open feature/config request now that the central WhatsApp leak is implemented. Source inspection shows current main suppresses WhatsApp error payload delivery and sanitizes raw provider error pass-through before channel-visible payloads.

## Solution Assessment

Yes for the central reported problem: unconditional WhatsApp isError suppression plus the shared safe formatter is narrower and safer than exposing a new default-on channel error-copy setting. Custom-message or owner-only variants would need a separate maintainer product decision.
