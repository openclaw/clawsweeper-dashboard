## Risks / Open Questions

- Same-store QMD contention changes from SQLITE_BUSY/retry noise into bounded waiting, so foreground memory search or index commands may wait behind an in-flight update or embed until the lock/write budget expires.
- The latest code acquires the global embed lock before the per-store write lock; if a same-store update is already running, other agents' embeds can wait behind that global lock even though the holder is waiting for the store lock.
- The supplied proof models the writer collision with real child processes, SQLite settings, and OpenClaw's real file-lock primitive, but it does not run a full gateway-plus-CLI scenario with the actual qmd binary.
