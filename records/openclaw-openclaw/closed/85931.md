## Summary

Keep open: this is a focused memory-core bug-fix PR for an open QMD SQLite lock-contention issue, and current main still only serializes QMD updates within one process while embeds use a separate cross-process lock. I found no blocking correctness finding in the diff, but maintainers still need to accept the bounded foreground wait behavior or run a real installed-QMD gateway-plus-CLI smoke before merge.

## What This Changes

The PR adds a per-agent QMD store write lock around both `qmd update` and `qmd embed` writes, while keeping the global embed lock and adding memory-core regression coverage for the shared lock path.

## Best Possible Solution

Land the narrow shared store-lock fix after maintainers accept or verify the bounded wait behavior with real QMD, with current CI green on the latest head.

## Reproduction Assessment

Yes, with high confidence from current-main source and supplied terminal proof, though I did not run the full gateway-plus-CLI reproduction locally. Current main has process-local update serialization and a separate embed file lock, while the PR proof demonstrates the underlying two-writer SQLite collision and shared-lock wait behavior.

## Solution Assessment

Yes, likely: using one per-store file lock for both QMD write phases is the narrow owner-boundary fix and preserves the existing global embed lock. A real-QMD concurrency smoke is the main remaining proof gap, not a code-shape blocker.
