## Summary

Current main already separates replay/reseed context from transcript writes, rotates compacted transcripts from a summary plus unsummarized tail instead of rewriting old history as fresh messages, and reloads WebChat history without duplicating optimistic tails. The implementation is present in the shipped v2026.4.27 tag, so this issue can close as implemented while adjacent duplicate-send and renderer-specific reports remain separate.

## What This Changes

The issue reports that after a session restart or migration, old session history is replayed into the new WebChat session transcript and shown as duplicate messages.

## Best Possible Solution

Keep the shipped implementation: compaction/restart continuity should flow through summaries, successor transcripts, and runtime-only reseed context, while WebChat should continue to render history from chat.history rather than treating replay context as new transcript messages.

## Reproduction Assessment

Yes. The current code and tests provide a high-confidence source-level reproduction path: a fresh session without compaction has no raw-history reseed, a compacted session reseeds from summary plus tail, transcript persistence stores the visible message body, and WebChat history reload does not duplicate optimistic tail messages.

## Solution Assessment

Yes. The current design is the narrow maintainable fix because it keeps old context available to the runtime without rewriting it as new transcript rows, and it leaves adjacent duplicate-send and renderer-specific fixes to their own surfaces.
