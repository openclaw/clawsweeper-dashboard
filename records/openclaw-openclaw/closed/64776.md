## Summary

Current `main` already wires sqlite-vec into the builtin memory runtime: memory DBs are opened with `allowExtension`, runtime sync loads the sqlite-vec extension before creating `chunks_vec`, vector rows are written during indexing, and `memory status` probes vector availability instead of leaving it at `unknown`. The same implementation is present in release `v2026.4.23` (`a9797214338ba31c52c796adbb75afb16e0684a9`).

## Evidence

- **Memory DB opens with extension loading enabled:** In release `v2026.4.23`, builtin memory opens SQLite with `new DatabaseSync(dbPath, { allowExtension })`, which is the Node requirement for extension loading and matches the failure mode discussed in linked issue `#65033`.
  - file: [extensions/memory-core/src/memory/manager-db.ts:9](https://github.com/openclaw/openclaw/blob/a9797214338ba31c52c796adbb75afb16e0684a9/extensions/memory-core/src/memory/manager-db.ts#L9)

  - sha: [a9797214338b](https://github.com/openclaw/openclaw/commit/a9797214338ba31c52c796adbb75afb16e0684a9)
