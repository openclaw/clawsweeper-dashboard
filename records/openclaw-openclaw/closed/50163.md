## What This Changes

The PR rewrites selected Discord listener, reaction, and voice-manager constructor parameter properties into explicit fields, adds a strip-only smoke test, marks Discord reaction system events untrusted, and adds a changelog entry.

## Best Possible Solution

Keep the shipped loader policy as the canonical fix; revisit constructor syntax only as a separate repo-wide TypeScript style decision if maintainers want that guardrail.

## Reproduction Assessment

Yes for the isolated syntax incompatibility: direct Node strip-only execution of the current Discord source files fails at parameter-property constructors. The user-facing git/dev crash is superseded because current main no longer routes source TypeScript plugin entries through native strip-only loading.

## Solution Assessment

No. The PR's constructor rewrite is mechanically safe, but the best current solution is already on main: keep source TypeScript on Jiti's transform path, which fixes the whole plugin/channel runtime boundary rather than this selected subset.
