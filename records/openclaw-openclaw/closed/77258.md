## Summary

Keep open. The duplicate-send bug is source-reproducible on current main, but this PR is not merge-ready: it is stale against the split queue files, leaves a post-provider-send crash window, and persists/logs raw idempotency material.

## What This Changes

The PR adds a persistent outbound dedupe ledger and deterministic idempotency keys for Discord-originated and tool-driven WhatsApp sends, threading keys through follow-up routing, message delivery, queue recovery, and tests.

## Best Possible Solution

Implement the fix against current main with opaque hashed keys, queue storage/recovery propagation, and completion committed at the smallest safe post-provider-send boundary, with regression coverage for crash recovery and Discord replay.

## Reproduction Assessment

Yes, source-level: current main writes a queue entry before delivery, acks only after delivery returns, and recovery replays pending entries with skipQueue. I did not run a live WhatsApp crash/restart test in this read-only review.

## Solution Assessment

No. The direction is useful, but this PR needs a privacy-safe key design, current-main queue split port, and safer completion semantics before it can solve the issue reliably.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Hash idempotency material before storing or logging it:** `src/infra/outbound/outbound-dedupe.ts:168-180`
  - body: The durable key includes the recipient, account/source IDs, message text, and media URLs, and duplicate suppression logs the full key. That leaks outbound content and identifiers into local state and logs; store an opaque hash and log only a short redacted identifier.
  - confidence: 0.94
- **[P2] Commit completion before the replay window reopens:** `src/infra/outbound/deliver.ts:815-816`
  - body: Recording the key after the delivery loop still allows the reported duplicate if the process dies after a provider send succeeds but before this record call. Commit completion at the smallest safe post-send boundary, or use provider-level idempotency where available.
  - confidence: 0.88
- **[P2] Port queue persistence through the split queue files:** `src/infra/outbound/delivery-queue.ts:46`
  - body: Current main moved queue storage and recovery reconstruction into delivery-queue-storage.ts and delivery-queue-recovery.ts. Adding idempotencyKey to this old monolithic file will not preserve the key through current-main persistence or replay.
  - confidence: 0.86
- **[P2] Include per-call identity in tool-send keys:** `src/infra/outbound/outbound-send-service.ts:125-132`
  - body: The tool-send key is based on source message plus payload content, so two intentional message-tool sends with the same text/media to the same target in one turn will share a key and the second will be suppressed for the TTL.
  - confidence: 0.78
