## Label Justifications

- `P1`: A stale per-session reply-run lock can block real inbound channel dispatch and produce no model call or outbound message until restart.
- `impact:message-loss`: The reported and source-supported failure mode accepts inbound messages but produces no reply or error for the affected session.
- `impact:session-state`: The root problem is stale in-memory per-session reply-run state outliving the persisted session status and blocking later turns.

## Summary

This issue should stay open because current main and v2026.6.1 still let visible reply-turn admission wait indefinitely behind an active reply operation, while the existing eventual busy path would be silent and non-delivering.

## What This Changes

The issue requests bounded, observable recovery for stale `replyRunRegistry` active-run locks that can make visible inbound message dispatch wait forever.

## Best Possible Solution

Make visible-turn admission bounded and observable, and route timeout recovery through a deliberate dispatch outcome that either safely reclaims stale reply work or tells the user without committing a silent drop.

## Reproduction Assessment

Yes for the core no-timeout path: current source and tests show visible admission can wait forever while an active reply operation remains. I did not run a live Octo reproduction for the intermittent lock-leak trigger.

## Solution Assessment

Unclear as a concrete patch direction: adding a timeout is necessary, but timeout handling also needs a non-silent delivery or recovery outcome. A safe fix should cover both the bounded wait and the stale-lock recovery/diagnostic path.
