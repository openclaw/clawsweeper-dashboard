## Summary

Keep this PR open: the linked OpenClaw Interaction Bridge use case is valid, but the patch is not merge-ready because the new `allowStateAccess` flag is not threaded through the normalized plugin config/startup contract and there is no after-fix real behavior proof.

## What This Changes

The PR adds an `allowStateAccess` plugin hook permission intended to let non-bundled plugins receive `agent_end` run-state metadata with `messages: []` instead of requiring full conversation access.

## Best Possible Solution

Complete the state-access contract across normalization, public config types, startup planning, status/docs/changelog, then prove it with a real non-bundled plugin run that shows `agent_end` fires with no message payload.

## Reproduction Assessment

Yes, from source inspection: current `main` blocks non-bundled `agent_end` registration unless the plugin has `allowConversationAccess=true`. I did not run a live repro because this pass is read-only.

## Solution Assessment

No, not as currently implemented: the direction is reasonable, but the proposed flag must be preserved through normalized config and startup policy before the registry gate can work reliably.

  - body: `registerTypedHook` now checks `policy?.allowStateAccess`, but the policy comes from `entry?.hooks` after `normalizePluginsConfig`, and that normalizer currently drops every hook key except `allowPromptInjection`, `allowConversationAccess`, and timeouts. Please add the field to the normalized config contract and its startup/status/public type plumbing before relying on it here.
