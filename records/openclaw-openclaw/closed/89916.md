## Summary

Current main and the latest shipped release already route configured custom-provider image models through shared model resolution instead of the old direct registry lookup that produced `Unknown model` on `2026.4.10`.

## What This Changes

The issue reports that a valid DashScope OpenAI-compatible vision model configured as `imageModel.primary` is rejected as `Unknown model` when used by the image tool.

## Best Possible Solution

Keep the shipped shared model-resolution path and ask users on `2026.4.10` to upgrade to `v2026.5.28` or newer before reopening with fresh logs.

## Reproduction Assessment

No high-confidence current-main reproduction remains: the reported `2026.4.10` direct lookup path is obsolete, and current main plus `v2026.5.28` resolve configured custom image models through shared provider metadata.

## Solution Assessment

Yes. The current solution is the narrow owner-boundary fix because it uses the shared model resolver and configured provider metadata instead of adding a DashScope-specific image-tool branch.
