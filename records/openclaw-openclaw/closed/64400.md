## Summary

Keep this issue open. Current main has a partial status/probe improvement for scope-limited `operator.read` failures, but the core doctor false-restart path is still present: non-interactive doctor still gates on the heavyweight `health` RPC with a 3000 ms timeout, and a failed health gate can still prompt/restart a running gateway service. The status timeout side is also not fully resolved because accepted connections whose follow-up detail RPC times out are still classified as unreachable rather than reachable/degraded.

## Best Possible Solution

Keep this issue open and fix the narrow core liveness path: have doctor Phase A use a lightweight gateway liveness/readiness signal such as `status` before service repair, prevent post-connect probe timeouts from being rendered as authoritative gateway unreachable when the listener/session was accepted, and add regression coverage proving slow `health` or detail RPCs do not drive `doctor` into `service.restart` for a running gateway. Handle any loopback unauth/device-identity change as a separate security-reviewed item.
