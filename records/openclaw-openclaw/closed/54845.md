## Summary

Current `main` already covers the reported outbound-registry regression well enough to close this PR. The failure site in outbound delivery now resolves adapters from the pinned channel registry, and current tests explicitly cover the post-registry-swap case that produced `Outbound not configured for channel` errors.

## Evidence

- **Outbound adapter loader is pinned:** `createChannelRegistryLoader()` reads `getActivePluginChannelRegistry()` and resolves channel entries from that pinned registry, not from the mutable active registry.
