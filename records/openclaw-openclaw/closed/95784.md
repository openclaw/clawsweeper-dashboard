## Summary

Keep open: current main and latest v2026.6.10 still route an isolated cron setup timeout into an immediate Gateway restart and cron restart-recovery pause, while https://github.com/openclaw/openclaw/pull/96396 is an open clean fixing PR with closing syntax for this issue. ## What This Changes

The issue asks OpenClaw to treat isolated cron setup timeout as a job-level failure/backoff instead of an unconditional full Gateway restart under event-loop saturation.

## Best Possible Solution

Land or revise https://github.com/openclaw/openclaw/pull/96396 so setup timeout records a cron error/backoff without restarting the Gateway, then close this issue after merge and track adaptive timeout/escalation work separately if maintainers still want it.

## Solution Assessment

Yes for the narrow first fix: the active PR removes the destructive restart path at the cron/gateway boundary while keeping timeout detection and error/backoff behavior. Maintainers still need to decide whether broader adaptive timeout or corruption-escalation rules should follow separately.
