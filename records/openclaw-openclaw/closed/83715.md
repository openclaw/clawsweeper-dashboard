## Summary

Keep open: the PR addresses a real doctor/config safety gap, but it is not merge-ready because the guard still runs after write-capable preflight recovery, interactive doctor can still use the original repair prompter on future-version skew, contributor real behavior proof is missing, and the live PR is currently conflicting.

## What This Changes

The PR attempts to block doctor repair, yes-mode, and gateway-token config writes against configs last touched by a newer OpenClaw version, suppressing repair hints and adding focused tests.

## Best Possible Solution

Move the future-version decision before any write-capable doctor preflight or health repair, make the read-only prompter apply to every detected skew unless the override is set, rebase, and add redacted real CLI proof.

## Solution Assessment

No: the direction is right, but the guard must run before any repair-capable preflight write and the prompter should become read-only for any detected future skew, not only explicit repair mode.
