## Summary

Current main treats tool-call interception as a plugin hook concern, not a Gateway internal-hook surface. The requested bridge would expand core in the opposite direction of the current docs, types, and regression tests, so this is better handled as a plugin/ClawHub integration using the existing `after_tool_call` plugin hook.

## Close Comment

Closing after a Codex review. Current `main` keeps tool-call interception on the plugin hook surface rather than the Gateway internal-hook surface: `docs/concepts/agent-loop.md:76` and `docs/automation/hooks.md:34` document internal hooks as command/lifecycle events, while `src/plugins/hooks.ts:817` exposes `after_tool_call` for plugins. Given `VISION.md:54`, this is better pursued as a plugin/ClawHub integration using the existing plugin hook API rather than expanding OpenClaw core internal hooks.
