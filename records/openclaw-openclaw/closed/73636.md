## Summary

Current main already implements the behavior this report asks for: TUI no longer asks for a broader default scope set than its initial operator-admin pairing, operator.admin satisfies later operator read/write/pairing checks, real broader reconnects become explicit scope-upgrade pending requests with requested/approved access details, and stale device-token reconnect loops are stopped in the Node and browser gateway clients.

## Best Possible Solution

Close this as implemented on main. Keep the current gateway/device-pairing behavior and ensure the next release carries the current main fixes; if a newer build still reproduces the loop, the follow-up should be a narrower regression with fresh gateway/client logs and the pending request details from openclaw devices list.
