## Summary

Keep open for maintainer review. Current main still has the reported BlueBubbles reply-context cache-key miss, and this PR is a narrow code/test fix that makes prefixed `p:<index>/<guid>` fallback fetches populate a matching cache key without broadening non-part slash aliases.

## What This Changes

The PR stores a BlueBubbles reply-context fallback cache alias for valid part-index reply ids and adds focused tests for prefixed and non-part slash reply ids.

## Best Possible Solution

Land this narrow BlueBubbles plugin fix after the normal changed-gate and CI checks pass; keep the behavior in the BlueBubbles fallback/cache layer rather than moving it into core.

## Reproduction Assessment

Yes. Current main has a source-level reproduction: fallback fetches strip `p:0/msg-guid` to `msg-guid` before caching, while subsequent monitor lookups exact-match the original prefixed id; the PR body also reports a focused helper probe showing bare hit and prefixed miss.

## Solution Assessment

Yes. Storing an alias only for validated `p:<index>/<guid>` inputs is a narrow maintainable fix, and the added negative test avoids turning arbitrary slash-containing strings into cache aliases.
