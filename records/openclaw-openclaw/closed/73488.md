## Summary

Keep open. Current main and the latest release still omit per-conversation identity from provider createStreamFn context, the paired OpenClaw PR was closed unmerged, and the merged Glueclaw-side fix remains a no-op until OpenClaw exposes these fields.

## What This Changes

Expose optional sessionKey and sessionId fields to provider plugin createStreamFn context so custom provider transports can scope per-conversation state.

## Best Possible Solution

Add a documented, optional provider createStreamFn identity contract, thread sessionKey/sessionId through session-bearing runner paths, leave one-off completions identity-less, and cover the passthrough with focused tests.

## Reproduction Assessment

Yes. Source inspection shows sessionKey/sessionId are available in session-bearing runner paths but are absent from ProviderCreateStreamFnContext and provider stream registration; the linked Glueclaw report gives a concrete concurrent-conversation transcript-bleed path.

## Solution Assessment

Yes directionally, but it needs maintainer API/privacy scope. Optional fields on createStreamFn are the narrow hook for fully custom provider transports, while the prepareExtraParams work at https://github.com/openclaw/openclaw/pull/54725 is adjacent and not a substitute.
