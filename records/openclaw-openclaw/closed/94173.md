## Summary

Keep this PR open, but it is not merge-ready: the clear-path pagination fix addresses a real Feishu large-document bug, but the SDK iterator used here can suppress list failures on a destructive write path and the supplied proof is still fake-client/local contract proof rather than live Feishu behavior.

## What This Changes

This PR changes Feishu document clearing to gather document blocks with `documentBlock.listWithIterator` before computing the delete range, and adds a two-page regression test for the `write` action.

## Best Possible Solution

Land a Feishu docx clear pagination fix that preserves checked list-page failures before any replacement insert; if maintainers want the whole docx block-list surface covered, factor a shared owner-module helper for clear, read, and list_blocks.

## Reproduction Assessment

Yes from source and dependency inspection, not live tenant replay: current main and `v2026.6.8` call `documentBlock.list` once before delete, while the pinned SDK declares the endpoint paginated and the PR test demonstrates the undercounted delete range for multi-page blocks.

## Solution Assessment

No as written. Paginating the clear path is necessary, but the safest fix preserves checked list-page failures before insert; a shared helper is cleaner if maintainers want sibling docx consumers fixed too.
