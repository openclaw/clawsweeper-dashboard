## Summary

Keep open: current main still intentionally makes the root dashboard and canvas auth paths differ, but the remaining question is an auth/security policy decision rather than conservative cleanup. The dashboard imports legacy query tokens at `/`, while canvas plugin routes require bearer auth or scoped capability URLs; making `?token=` work on canvas would broaden shared-secret handling onto untrusted canvas content.

## What This Changes

Request for `?token=` URL bootstrap to work on `/__openclaw__/canvas/` over an SSH tunnel the same way it works on the root dashboard URL.

## Best Possible Solution

Keep canvas on bearer headers and short-lived scoped capability URLs unless maintainers approve a deliberate broader auth mode; otherwise clarify operator-facing dashboard/startup guidance so users open the root Control UI URL.

## Reproduction Assessment

Yes, at source level. A request to `/__openclaw__/canvas/?token=...` reaches plugin-node capability auth, which ignores `token`, while the root Control UI bootstrap explicitly imports legacy query tokens.

## Solution Assessment

Unclear. Making canvas accept `?token=` is not clearly the best fix because canvas serves untrusted HTML/JS; maintainers should choose between clearer guidance and a deliberately designed canvas auth flow.
