## Summary

This newer iOS pairing report should close as a duplicate of the open canonical Gateway/CLI pairing recovery issue, which already tracks rotating pending request IDs, `unknown requestId` approval failures, and the remaining security-sensitive recovery UX.

## What This Changes

The issue reports that rapid iOS pairing retries replace pending request IDs before `openclaw devices approve <requestId>` can approve them.

## Best Possible Solution

Keep the remaining stale request-id approval recovery work consolidated in https://github.com/openclaw/openclaw/issues/74484 so one security-reviewed CLI/Gateway fix covers the shared behavior.

## Reproduction Assessment

Yes, at source level: a stale request id makes `approveDevicePairing` return null and the gateway maps that to `unknown requestId`. I did not run a live iOS pairing repro in this read-only review.

## Solution Assessment

No for the direct `--latest` auto-approve suggestion: current code intentionally requires approving an inspected exact request id. The better path is the canonical issue’s bounded same-device recovery design, with security tests for replacement matching and fail-closed cases.
