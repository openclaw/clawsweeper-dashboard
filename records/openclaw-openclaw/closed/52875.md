## Summary

Current main and the latest shipped release already implement the central requested behavior: `sessions_send` can target a configured agent by `agentId`, create that agent's main session when it is missing, and then send the message. The later discussion about richer received/active/waiting/completed message states is separate follow-up work, not a reason to keep this bug open.

## What This Changes

The issue reports that `sessions_send` failed with no session found for configured agents that had no active session after an upgrade.

## Best Possible Solution

Keep the shipped `sessions_send(agentId)` bootstrap behavior and track richer message-state/read-receipt semantics only in narrower follow-up issues if maintainers want that product shape.

## Reproduction Assessment

No. The central failing path no longer reproduces from source because current `sessions_send` creates a configured target agent's main session before sending when called with `agentId`, and the current regression test exercises the reporter-style `orion` target case.

## Solution Assessment

Yes. Bootstrapping a configured agent's main session inside `sessions_send(agentId)` is the narrow maintainable fix for the no-session-found bug; separate active/waiting/read states would be a new feature follow-up.
