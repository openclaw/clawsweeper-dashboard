## Summary

Keep open: the PR targets a real source-backed gateway signal bug, but the current patch leaves the SIGUSR1 restart cycle uncleared on lifecycle-loader failure and the supplied proof is a standalone simulation rather than patched OpenClaw gateway behavior.

## What This Changes

The PR wraps gateway SIGTERM and SIGUSR1 async signal-handler bodies in try/catch, logs lifecycle-loader failures, falls back to a plain SIGTERM stop, and skips failed SIGUSR1 restarts.

## Best Possible Solution

Land one canonical fix that keeps the SIGTERM fallback, makes SIGUSR1 consume or roll back the restart cycle on lifecycle-load failure, and proves the actual run-loop behavior with focused coverage or redacted gateway output.

## Reproduction Assessment

Yes, source-reproducible: current main awaits loadGatewayLifecycleRuntimeModule inside discarded async SIGTERM/SIGUSR1 handlers before request, so a loader rejection can escape and skip shutdown or restart handling. I did not run a live gateway repro because this review is read-only.

## Solution Assessment

No, not yet: the SIGTERM fallback is the right shape, but the SIGUSR1 failure path must also clear or roll back restart-cycle state instead of only logging. The safer solution is a narrow signal-handler failure path plus proof against the actual OpenClaw run-loop contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Clear the SIGUSR1 restart token on lifecycle-load failure:** `src/cli/gateway-cli/run-loop.ts:751`
  - body: This catch logs and returns after an authorized in-process restart can already set `emittedRestartToken` in `emitGatewayRestart`. Because the failure path never calls `markGatewaySigusr1RestartHandled` or rolls back the token, `hasUnconsumedRestartSignal()` stays true and later restart requests can coalesce or return false instead of retrying. Clear or roll back that cycle before skipping restart.
