## Summary

Keep open. Current main and the latest release still return after `process.ppid` on non-Linux platforms, so macOS nested callers can still leave a gateway grandparent outside the stale-process exclusion set; the reporter's follow-up narrows this to a latent correctness bug while the live SIGTERM loop is tracked separately.

## What This Changes

The issue asks OpenClaw to extend stale-gateway ancestor PID exclusion on macOS beyond the direct parent so grandchild cleanup callers do not signal their gateway ancestor.

## Best Possible Solution

Add a bounded Darwin/BSD parent-PID walk in `getSelfAndAncestorPidsSync()`, preserve the Linux `/proc` path, and add a focused macOS grandparent regression test.

## Reproduction Assessment

Yes, at source level. Stub `process.platform` to `darwin`, set `process.ppid` to a plugin host, and feed lsof output containing a gateway grandparent; current main returns before any transitive walk, leaving that gateway PID killable.

## Solution Assessment

Yes. Extending the existing ancestor-exclusion helper with a bounded macOS/BSD parent lookup is the narrowest maintainable fix and avoids changing stale-cleanup call-site policy.
