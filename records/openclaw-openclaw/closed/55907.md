## Summary

Current `main` already parses `channels.telegram.groups[-100…]` as a string-key path, not as an array index or doubly quoted key, and the same implementation is present in release `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`).

## Evidence

- **Bracket parser keeps unquoted negative IDs as raw string segments:** `parseBracketPathSegment` only JSON5-parses quoted bracket contents; otherwise it returns the trimmed bracket text unchanged. For `[-1003713572219]`, that yields the literal string `-1003713572219`, not a quoted JSON string.
  - file: [src/cli/config-cli.ts:111](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/src/cli/config-cli.ts#L111)
  - command: `nl -ba src/cli/config-cli.ts | sed -n '96,190p'`
  - sha: [7130e56f872a](https://github.com/openclaw/openclaw/commit/7130e56f872aea96577ac9fcbda0c7555a9ecc69)
- **Only unsigned digits are treated as array indices:** `isIndexSegment` matches `/^[0-9]+$/`, so negative IDs are not considered array indices. Downstream `setAtPath` therefore writes them as object keys under `groups` rather than array positions.
  - file: [src/cli/config-cli.ts:107](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/src/cli/config-cli.ts#L107)
