## Summary

Keep open: current main still lacks the requested Telegram reaction-directive behavior, and this PR is a useful direction but has blocking normalization/type issues plus required changelog work before merge.

## What This Changes

The PR adds parsing for Telegram reaction directives, stores reaction intent in outbound channel data, sends native Telegram reactions in Telegram delivery paths, and adds normalization plus adapter tests.

## Best Possible Solution

Land or replace this with a narrow fix that preserves reaction directives through both reply-normalization paths, sends native Telegram reactions only when a valid target exists, keeps media parsing intact, and adds the required changelog plus focused Telegram tests.

## Reproduction Assessment

Yes. Current main has no reaction parsing in `parseReplyDirectives`, while Telegram already has `reactMessageTelegram`; the linked report gives concrete `[[react_to_current:🔥]]` symptoms and live validation context.

## Solution Assessment

No, not as submitted. The product boundary is right, but the patch needs to preserve reaction data through the normal assistant reply path, fix the reaction option contract, avoid success on missing targets, and add changelog/test proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve reactions through reply normalization:** `src/auto-reply/reply/reply-directives.ts:29-35`
  - body: `parseReplyDirectives` now strips `[[react...]]` for every caller, but the normal assistant reply path uses `normalizeReplyPayloadDirectives`, which only copies text/media/reply/audio fields from the parse result. A reaction-only Telegram reply can therefore become empty text with no `channelData.telegram.reaction` before `deliverReplies` sees it, so the live assistant lane can still drop the directive instead of reacting.
  - confidence: 0.86
- **[P1] Fix the reaction options type before forwarding scopes:** `extensions/telegram/src/outbound-adapter.ts:121`
  - body: `params.react` is typed as `typeof reactMessageTelegram`, but `TelegramReactionOpts` does not declare `gatewayClientScopes`. Passing that property here, and in the bot delivery helper, should fail excess-property checking unless the option is added and honored or the extra property is removed.
  - confidence: 0.93
- **[P2] Do not acknowledge reaction-only payloads without a target:** `extensions/telegram/src/outbound-adapter.ts:126-130`
  - body: When text and media are empty, this returns a synthetic success even if no reaction was sent because `replyToMessageId` is missing. That can mark an outbound delivery as successful while Telegram receives nothing; return success only after a reaction is actually sent, or surface/drop the missing-target case.
  - confidence: 0.78
- **[P3] Add the required changelog entry:** `src/infra/outbound/payloads.ts:184`
  - body: This changes user-visible Telegram behavior by turning leaked reaction directives into native reactions. Repo policy requires a `CHANGELOG.md` entry for user-facing fixes, so this should add an active `### Fixes` bullet before merge.
  - confidence: 0.88
