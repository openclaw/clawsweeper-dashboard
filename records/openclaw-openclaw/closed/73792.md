## Summary

Keep open: current main still has the MCP stale-runtime bug, but this external PR is not mergeable yet because the close hook also fires for intentional startup cleanup, the added test imports non-exported helpers, and contributor real behavior proof is missing.

## What This Changes

The PR wires MCP transport close/error callbacks into session MCP runtime eviction and adds a regression test that kills a probe MCP server process.

## Best Possible Solution

Land a rebased narrow fix that evicts only unexpected transport closes, preserves multi-server startup cleanup, uses current test helper boundaries, and includes redacted real runtime proof.

## Reproduction Assessment

Yes, by source inspection: current main returns the cached runtime when fingerprint and workspace match, then callTool uses the cached client after the SDK close path clears _transport, so later requests reject with Not connected. I did not run the integration test because this review is read-only.

## Solution Assessment

No, not as submitted. Installing the close handler before connect is the right direction, but it must skip intentional cleanup closes and the test must be rebased onto current helper exports before the fix is maintainable.
