## Summary

Keep open: the PR targets the right gateway chat path, but the patch is incomplete against current main because provenance sender fields are not carried by the canonical `InputProvenance` type/normalizer, and the sender fallback conflicts with current Control UI sender-suppression behavior. It also lacks required after-fix real behavior proof for an external PR.

## What This Changes

The PR adds optional sender metadata to `systemInputProvenance` and uses trusted operator-provided provenance fields to populate chat sender and source-channel context.

## Best Possible Solution

Implement this as a small gateway protocol update only after the canonical provenance type/normalizer, chat context fallback semantics, focused tests, and protocol docs/artifacts are aligned with current main.

## Reproduction Assessment

Not applicable: this is an additive gateway protocol/metadata feature PR, not a reproduced bug report. The blocking issues are source-provable from the current normalization and Control UI sender tests.

## Solution Assessment

No: the affected path is appropriate, but the submitted patch is not the best solution because it adds schema fields without preserving them through the canonical provenance contract and risks undoing current Control UI fallback behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve sender fields through provenance normalization:** `src/gateway/server-methods/chat.ts:2272-2274`
  - body: `systemInputProvenance` is normalized before this context is built, but the canonical `InputProvenance` type and `normalizeInputProvenance` still only return `kind`, origin/source session, channel, and tool fields. As written, the new `senderId`/`senderName` reads either fail typechecking or never see the client-provided values, so the feature will not work until the shared type and normalizer are updated with regression coverage.
  - confidence: 0.93
- **[P2] Keep Control UI fallback sender fields suppressed:** `src/gateway/server-methods/chat.ts:2272-2274`
  - body: Current main intentionally omits `SenderId`, `SenderName`, and `SenderUsername` for Control UI/TUI clients, with a regression test covering that behavior. Falling back to `clientInfo` whenever trusted provenance sender fields are absent would reintroduce `openclaw-control-ui` as the sender for ordinary operator UI sends, so the fallback needs to preserve the existing `isOperatorUiClient` guard and only inject explicit trusted provenance sender metadata when present.
  - confidence: 0.87
