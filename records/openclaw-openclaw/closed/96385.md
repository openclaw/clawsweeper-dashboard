## Summary

Current main already implements the WebChat session-continuity behavior requested here: Control UI persists the active session selection and now resumes the backing Gateway session after reconnect/restart. The fix is current-main-only because the latest release tag does not include the reconnect-resume marker.

## What This Changes

The issue asks WebChat to reopen the user's previous conversation after Gateway or machine restart instead of showing a blank auto-created `main` session.

## Best Possible Solution

Keep the current-main WebChat session persistence implementation and let it ship in the next release; track distinct context-after-restart or workspace-layout failures separately.

## Reproduction Assessment

No live reproduction was run, but the issue has a high-signal source path: the latest release lacks the reconnect-resume state while current main stores and resumes the backing session id. The provided steps match earlier closed reconnect-reset reports.

## Solution Assessment

Yes. The current-main fix is the maintainable owner-boundary solution because it preserves the selected UI session and the Gateway session id rather than adding a new user setting or runtime fallback.
