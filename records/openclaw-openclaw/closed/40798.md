## Summary

Keep this PR open. Current main still has the stale `toolCallHistory` behavior: loop detection reads the full retained history, `recordToolCall` only appends/trims the sliding window, and the session state has no stale-cycle marker. The linked bug #40144 is still open, and although #40217 is a related open implementation attempt, it is not clearly a better or merged superseder. This PR is a focused candidate but needs maintainer review because its 60s staleness heuristic and `lastToolCallAt` updates can trade one false-positive class for missed slow/no-progress loops.

## Best Possible Solution

Keep this PR and #40144 open while maintainers converge the duplicate implementation attempts. The best fix should clear history across truly separate heartbeat/turn contexts without disabling loop detection for slow tools or long polling, cover the production detect-before-record ordering, late outcomes, and slow/no-progress loop cases in `src/agents/tool-loop-detection.test.ts`, then remove unrelated instruction-file churn and correct the changelog attribution if the fix lands.
