## Summary

Keep open. The report matches current main's QQ Bot chunked-upload path: large local/buffer media is documented and routed to `ChunkedMediaApi`, and the COS PUT still uses a Blob body with a manual `Content-Length`.

## What This Changes

QQ Bot large local media sent through `<qqmedia>` should succeed on the chunked upload path instead of failing during the presigned COS part PUT with an invalid `Content-Length` header.

## Best Possible Solution

Fix the QQ Bot plugin's chunked-upload PUT request body/header handling and add focused regression coverage, leaving the routing and core fetch guard unchanged unless another caller needs the same normalization seam.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only sweep because it needs QQ Bot credentials and a real presigned COS URL. Source inspection gives a clear path: send a >5 MiB local/buffer media item so `dispatchUpload` calls `ChunkedMediaApi.uploadChunked`, which currently PUTs a Blob with a manual `Content-Length`.

## Solution Assessment

Yes, the narrowest maintainable fix is in `putToPresignedUrl`: make the part body/header contract compatible with Node/undici/COS and cover it with a unit regression plus live QQ proof when credentials are available. A core fetch-guard change is not justified unless another caller needs the same behavior.
