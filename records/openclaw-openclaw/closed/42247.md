## Summary

Current main already exposes post-task lifecycle automation through the plugin SDK (`agent_end`, `session_end`, `subagent_ended`) plus runtime command execution helpers, while repo vision says optional capability should ship as plugins/ClawHub rather than new core hook surfaces.

## Evidence

## Risks / Open Questions

- This closes the request as out-of-scope for core, not because internal `session:completed` / `cron:completed` events exist exactly as proposed.
- If maintainers want a declarative no-code automation layer for non-plugin authors, this evidence does not rule out a future product decision to add one.

## Close Comment

Codex review: this looks like plugin/ClawHub scope rather than a core gap. Current `main` already exposes post-task lifecycle hooks through the plugin SDK: `agent_end` after a run completes (`src/agents/pi-embedded-runner/run/attempt.ts:2569`), `session_end` for session lifecycle (`src/auto-reply/reply/session-updates.ts:62`), and `subagent_ended` for subagent completion (`src/agents/subagent-registry-completion.ts:93`). Those hooks have typed run/session metadata in `src/plugins/hook-types.ts:154`, `src/plugins/hook-types.ts:400`, and `src/plugins/hook-types.ts:479`, and plugins can run shell commands via `api.runtime.system.runCommandWithTimeout` (`docs/plugins/sdk-runtime.md:307`, `src/plugin-sdk/run-command.ts:17`). `VISION.md:54` also states optional capability should usually ship as plugins and new skills should go to ClawHub first. On that basis, I’m closing this as better suited to a plugin/ClawHub package than new OpenClaw core hook/config surface.
