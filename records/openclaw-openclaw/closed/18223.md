## Summary

Current main and the latest release tag separate compaction from session-reset cleanup, so the reported regression no longer matches shipped behavior.

## Evidence

## Close Comment

Closing this as implemented after a Codex review. On current `main`, the active-run abort logic is scoped to the reset/delete cleanup service in `src/gateway/session-reset-service.ts:194`, and `sessions.reset` is routed there explicitly from `src/gateway/server-methods/sessions.ts:1349`. Compaction is on separate code paths: manual compaction uses the dedicated `sessions.compact` handler in `src/gateway/server-methods/sessions.ts:1489`, overflow compaction stays inside `contextEngine.compact(...)` plus `runContextEngineMaintenance(... reason: "compaction")` in `src/agents/pi-embedded-runner/run.ts:1092`, and direct compaction calls `activeSession.compact(...)` in `src/agents/pi-embedded-runner/compact.ts:1030` rather than `session.reset()`. I also verified the same separation in the released `v2026.4.22` tag (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). Given that structure, the specific regression described here no longer matches current shipped behavior.
