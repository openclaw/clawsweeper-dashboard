## Summary

Current main and the shipped v2026.4.22 release already satisfy the observable bug: omitted native `sessions_spawn.runTimeoutSeconds` is resolved from `agents.defaults.subagents.runTimeoutSeconds` before the run is registered, while explicit `0` still means no timeout. This PR is now redundant and its branch is dirty against current main.

## What This Changes

The PR removes `?? 0` fallback coercions from subagent steer, reactivation, and registry timeout propagation so undefined timeout values can flow into timeout resolution.

## Best Possible Solution

Keep the shipped spawn-boundary resolver, tests, and docs as the maintained implementation. Any future cleanup of lower-level registry timeout defaults should be a separate refactor only if a remaining user-visible path is identified.

## Reproduction Assessment

Yes. The focused reproduction is `sessions_spawn` with `runTimeoutSeconds` omitted and `agents.defaults.subagents.runTimeoutSeconds` set; current source and regression coverage show it resolves to the configured value instead of infinite timeout. I did not run tests because this review was constrained to read-only inspection.

## Solution Assessment
