## Summary

Keep open: the direction matches current Telegram account-group semantics, but the new doctor scanner misses a real multi-account shape where runtime still suppresses root groups.

## What This Changes

The PR updates Telegram group docs and adds a non-mutating doctor warning with tests for multi-account configs whose enabled allowlisted accounts lack account-local group rules.

## Best Possible Solution

Keep the docs and diagnostic approach, but make the doctor scanner use the same configured-account multi-account predicate as runtime and add a disabled-account regression test.

## Reproduction Assessment

Yes for the blocking finding. Source inspection shows runtime treats more than one configured Telegram account as multi-account, while the proposed doctor warning only treats two enabled accounts as multi-account.

## Solution Assessment

No, not as currently written. The warning should still skip disabled accounts in the reported hits, but the multi-account predicate itself needs to match runtime's configured-account count.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Match the runtime multi-account check:** `extensions/telegram/src/doctor.ts:233-234`
  - body: Runtime suppresses root Telegram `groups` as soon as more than one account is configured, even if one of those accounts is disabled. This new scanner returns early when fewer than two enabled accounts remain, so a config with root groups plus one enabled account and one disabled account still drops the enabled account's root group rules without the new doctor warning. Base the multi-account check on configured account keys, then keep filtering disabled accounts out of the reported hits.
  - confidence: 0.87
