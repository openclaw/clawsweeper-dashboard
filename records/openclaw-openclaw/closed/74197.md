## Summary

Current `main` already removes the default gateway-start QMD initialization path that caused the reported `qmd collection add` timeout/restart loop in v2026.4.26. QMD startup refresh is now opt-in via `memory.qmd.update.startup`, the default resolves to `off`, post-attach sidecars skip memory startup by default, and the opt-in path uses a one-shot CLI manager/sync path instead of arming the long-lived runtime manager. The fix is on `main` only; I found no release tag containing the proof commit yet.

## Best Possible Solution

Close as implemented on current `main`: keep QMD gateway-start refresh lazy by default, retain `idle`/`immediate` as explicit opt-ins, and let the next release carry the fix for users affected on v2026.4.26.
