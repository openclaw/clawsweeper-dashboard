## Summary

Keep open. The overlapping auth cooldown, OpenRouter tool metadata, and blank-message repair pieces are now covered on current main, but the PR still contains a distinct fallback-filtering change that introduces a source-level regression and lacks real behavior proof.

## What This Changes

The PR branch currently adds a `requiresTools` fallback-filtering option, passes it unconditionally from the embedded agent command path, and adds a `format` auth-profile cooldown exemption plus tests.

## Best Possible Solution

Keep the shipped auth, OpenRouter metadata, and blank-message repairs; if this PR continues, derive fallback filtering from the actual attempt tool-construction plan or leave explicit fallbacks runnable without tools.

## Reproduction Assessment

Yes, by source inspection: current main has raw `modelRun` and `promptMode: "none"` paths that disable tool construction, while the PR marks the whole embedded fallback chain as requiring tools before each attempt is planned.

## Solution Assessment

No. Reusing `supportsModelTools()` is the right local contract, but the PR should not pass `requiresTools: true` unconditionally from the command path; the safer alternative is to gate filtering on the actual tool-construction mode.
