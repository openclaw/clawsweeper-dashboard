## Summary

Keep open: current main still includes workspace-derived plugin roots, plugin scope, and activation/runtime flags in loader cache identity, while tests intentionally reject cross-workspace reuse. The issue has credible live timing data and related open work, but the safe fix needs maintainer design around cache compatibility rather than a mechanical key removal.

## What This Changes

The issue asks OpenClaw to avoid redundant plugin-loader cold loads across equivalent workspace/plugin-scope requests by narrowing or redesigning the plugin loader cache key.

## Best Possible Solution

Design a compatibility-aware registry reuse rule that shares loaded registries only when workspace roots, manifests, activation inputs, runtime mode, plugin scope, and config/trust inputs are proven equivalent, with regression tests for both reuse and workspace isolation.

## Reproduction Assessment

Yes, at source level: current main still hashes workspace-derived roots and `onlyPluginIds` into plugin-loader cache identity, and tests assert cross-workspace reuse is rejected. I did not run a live gateway timing benchmark for the reported 7s to 16s stalls.

## Solution Assessment

Unclear as proposed: reducing redundant cold loads is the right goal, but simply removing `workspaceDir` is unsafe because workspace-local plugin discovery is a current contract. The safer solution is a compatibility-aware reuse path with isolation tests.
