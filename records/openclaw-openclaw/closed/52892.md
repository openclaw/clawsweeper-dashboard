## Summary

Current main already uses batched SecretRef resolution for exec providers during gateway startup, including auth-profile refs, so the reported per-secret sequential startup bottleneck is already addressed and shipped in v2026.4.22.

## Evidence

- **Exec providers are batched by id in one request:** `resolveExecRefs` deduplicates all requested ref ids, builds one JSON stdin payload with `protocolVersion`, `provider`, and `ids`, then parses one response map back into resolved values.
  - file: [src/secrets/resolve.ts:655](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/secrets/resolve.ts#L655)

  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)
- **Gateway startup resolves config and auth-store refs together:** `prepareSecretsRuntimeSnapshot` includes auth-store refs by default, collects config plus auth-store assignments, and calls `resolveSecretRefValues(refs, ...)` once for the whole assignment set.
  - file: [src/secrets/runtime.ts:253](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/secrets/runtime.ts#L253)
