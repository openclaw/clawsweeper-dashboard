## Summary

Keep open: current main still enforces Telegram allowlists by dropping or logging unauthorized inbound events, but it does not provide the requested supported alert, retention, delivery, or recent-audit surface. The related unauthorized-hook issue is complementary, not a complete replacement for this Telegram-specific audit request.

## What This Changes

Request a Telegram security-observability surface for unauthorized DM, group, group-allowlist, and command attempts, with configurable logging/notification/audit behavior and privacy-safe metadata.

## Best Possible Solution

Design a Telegram-owned runtime security-event/audit path with minimized metadata, rate limiting, recent-history inspection, and optional owner/webhook delivery, coordinating any generic hook seam with https://github.com/openclaw/openclaw/issues/68214.

## Reproduction Assessment

Not applicable as a feature request. Source and docs show current allowlist enforcement and log/drop behavior, but there is no existing contract requiring blocked-attempt alerts or audit history.

## Solution Assessment

Unclear until maintainers choose the security/product shape. A privacy-minimized Telegram audit path is more maintainable than documenting log scraping, and a generic unauthorized hook should be coordinated with https://github.com/openclaw/openclaw/issues/68214 rather than assumed to replace this request.
