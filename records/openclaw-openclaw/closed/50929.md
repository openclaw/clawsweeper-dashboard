## Summary

Current main already solves the PR's user-visible bug by preventing non-webchat/TUI-originated run events from being routed into Control UI chat at all. The linked issue #50905 is closed as implemented, and v2026.4.24 contains the source-side visibility gating plus regression coverage, so this client-side filter PR is now obsolete.

## Best Possible Solution

Close #50929 as implemented on main. Keep the shipped source-side `isControlUiVisible` gating and regression tests, and avoid merging the stale client-side filter PR unless maintainers explicitly want a separate defense-in-depth sanitizer.

## Evidence

- **Only webchat is treated as the internal Control UI message channel:** `INTERNAL_MESSAGE_CHANNEL` is `webchat`, and `isInternalMessageChannel()` compares normalized channels to that value. TUI remains an operator UI client, but it is not the internal webchat channel used for Control UI chat surfacing.
  - file: [src/utils/message-channel.ts:61](https://github.com/openclaw/openclaw/blob/54f8e4145e43b35920d9b876bf1415eba21c2fd1/src/utils/message-channel.ts#L61)

For TUI-originated runs this evaluates false.
