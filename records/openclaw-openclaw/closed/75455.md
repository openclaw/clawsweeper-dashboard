## Summary

Keep open: the PR is aimed at a real BlueBubbles DM routing gap, but the current patch still bypasses service-aware handle ranking for bare phone/email chat identifiers and is missing the required user-facing changelog entry.

## What This Changes

The PR makes BlueBubbles DM inbound fallback prefer a non-empty `chatIdentifier` over `senderId` and updates the focused resolver tests.

## Best Possible Solution

Update DM fallback construction so bare phone/email identifiers use the handle resolver path while full or opaque BlueBubbles identifiers use `chat_identifier`, then add focused regression tests and a changelog entry.

## Reproduction Assessment

Yes. Source inspection shows current main ignores DM `chatIdentifier` in fallback resolution, and the PR regression is source-reproducible because bare identifiers would take the first-match `chat_identifier` path instead of the ranked handle path.

## Solution Assessment

No, not as currently proposed. Preferring precise inbound identifiers is the right direction, but handle-like DM identifiers should still resolve through the handle branch so the existing iMessage-over-SMS contract is preserved.
