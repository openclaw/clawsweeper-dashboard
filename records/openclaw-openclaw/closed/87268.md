## Summary

Keep open: current main and the latest release still route non-asset Control UI navigations through the service worker, and the paired open PR is the active fix path rather than a reason to close the issue.

## What This Changes

The issue reports that the Control UI service worker intercepts reverse-proxy HTTP-auth navigation 401 responses and prevents the browser's native credentials prompt from appearing.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/87077 or an equivalent service-worker navigation fix that lets browser-native HTTP auth challenges reach the network path while preserving the intended asset/API caching behavior.

## Reproduction Assessment

Yes, source-reproducible: the current production-registered service worker handles same-origin navigation-like GETs with event.respondWith(fetch(event.request)), matching the reported DevTools path. I did not run a live Caddy/Firefox repro in this read-only review.

## Solution Assessment

Unclear until maintainer review: the linked navigation bypass is a narrow likely fix, but it intentionally gives up the cached offline shell for top-level navigations, so maintainers should accept that trade-off or request a narrower variant.
