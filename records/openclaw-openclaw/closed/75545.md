## Summary

Keep open: this PR targets a source-reproducible gateway restart bug, but the current head still bypasses stale-service repair before returning `already-running`, CI shows the protected repair tests failing, and no real after-fix behavior proof is posted.

## What This Changes

Adds an `already-running` gateway-service start outcome, CLI output, tests, and a changelog entry so `openclaw gateway start` no-ops when the managed service is already running, plus a few formatting-only reflows.

## Best Possible Solution

Keep the idempotent `gateway start` behavior, but return `already-running` only after loaded-service repair checks pass, then add redacted real runtime proof for the profile/exec path.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: gateway exec invokes a login-shell PATH probe, current main's `gateway start` still reaches the service restart path, and Linux systemd restart maps to `systemctl --user restart`; I did not run a live systemd reproduction.

## Solution Assessment

No. The PR direction is right, but the no-op guard currently runs before stale-service repair checks, so the narrow maintainable fix is to move or condition that guard after repair collection.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Preserve repair before the running-service no-op:** `src/daemon/service.ts:201-205`
  - body: The new running-service guard returns before `collectGatewayServiceStartRepairIssues`, so a loaded service installed by an older CLI or pointing at a temporary/missing binary now reports `already-running` and skips the existing repair or fail path. Move the no-op after repair collection, or only return it when no repair issue exists.
  - confidence: 0.96
