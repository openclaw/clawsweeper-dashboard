## Summary

Keep open for maintainer review: the PR addresses a real Telegram message-tool preview mismatch with focused tests and sufficient real Telegram proof, but it adds public `reply-runtime` SDK exports that need an explicit API-contract decision before merge.

## What This Changes

The branch extracts shared source-visible-reply and message-tool availability helpers, exports them from `plugin-sdk/reply-runtime`, and uses them in Telegram dispatch/tests to suppress draft/reasoning previews when source replies resolve to `message_tool_only`.

## Best Possible Solution

Land the Telegram behavior fix after maintainers accept the additive `reply-runtime` contract, or revise the branch to keep the shared policy access behind a narrower approved runtime seam without changing the user-visible suppression behavior.

## Reproduction Assessment

## Solution Assessment

Yes, with an API-contract caveat. Pre-resolving the same shared source-reply policy before Telegram draft creation is the narrow behavior fix, but maintainers still need to decide whether these helpers belong in public `reply-runtime`.
