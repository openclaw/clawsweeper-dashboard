## Label Justifications

- `P1`: A post-update doctor prompt can trigger repeated gateway restarts on macOS, temporarily breaking the core gateway/channel workflow for real users.
- `impact:crash-loop`: The report and source path center on repeated supervisor restarts and stale-process cleanup around the gateway service.

## Summary

Keep open: current main and the latest release still have a source-backed path where an interactive doctor run can offer a default-Yes restart for a running gateway service after the gateway RPC health check fails, and existing tests only cover non-interactive update skipping rather than the reported interactive macOS restart loop.

## What This Changes

The issue asks doctor to stop defaulting to a running gateway service restart after update unless there is a concrete unhealthy or stale-process reason.

## Best Possible Solution

Tighten the doctor gateway daemon restart gate so a running service is not restarted by default after a transient post-update health miss; only offer a restart for a documented unhealthy/stale condition and default any manual running-service restart prompt to No.

## Reproduction Assessment

Yes from source, not live: set healthOk false with a loaded running service and an interactive prompter accepting defaults, and maybeRepairGatewayDaemon reaches the default-Yes restart path. A macOS LaunchAgent end-to-end repro was not run in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is in doctor daemon restart gating/defaults with focused regression tests. The repair should keep stopped/missing-service recovery intact while avoiding redundant running-service restarts after update.
