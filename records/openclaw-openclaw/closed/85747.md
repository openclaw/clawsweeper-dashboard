## What This Changes

Adds Discord REST timeout config, gateway force-drain restart phases, and an opt-in model worker-thread fetch pool wired through provider stream registration and gateway startup.

## Best Possible Solution

Use a provider/runtime-owned, abort-aware streaming transport if worker isolation is still desired, and keep any restart hard-kill behavior explicitly opt-in with upgrade tests and live gateway proof.

## Solution Assessment

No. The current approach buffers streaming responses, loses abort semantics, hardcodes provider URL policy in core, and makes force-drain behavior active without an explicit config decision.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Stream chunks instead of buffering worker responses:** `src/agents/model-worker.ts:50`
  - body: This awaits `response.arrayBuffer()` before posting back to the main thread, so an SSE/model stream is not delivered to the caller until the request finishes. That breaks the streaming/progress path the PR is trying to protect; pass chunks over `MessagePort` or keep streaming fetches on the main/provider transport.
  - confidence: 0.97
- **[P1] Gate force-drain behind explicit config:** `src/infra/restart.ts:521-523`
  - body: `opts.forceDrain` is optional, but `normalizeShutdownControllerConfig(undefined)` fills the default 10/20/30/40s thresholds and the poll calls `shutdownCtl.tick()` for every deferred restart. A normal config reload with active work will now hit `process.exit(0)` after 40s instead of honoring the existing deferral timeout or `0` indefinite behavior.
  - confidence: 0.95
- **[P1] Cancel the worker request before falling back:** `src/agents/model-worker-pool.ts:300-303`
  - body: On timeout the code deletes the pending entry and rejects, then `createModelWorkerPoolFetch` falls back to `baseFetch` while the worker fetch is still running. A late worker response is ignored, the worker can stay busy forever, and the same model request may be sent twice.
  - confidence: 0.9
