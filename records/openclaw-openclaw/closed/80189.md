## Summary

Keep open. The PR addresses the right source-level auth/scope bug, but the new `AUTH_SCOPE_MISMATCH` detail is not wired into the browser/shared GatewayClient reconnect suppression, so clients can still loop on the new terminal diagnostic instead of pausing for pairing guidance.

## What This Changes

The PR narrows bootstrap/cached device-token connect scopes, propagates device-token scope mismatches as `AUTH_SCOPE_MISMATCH`, and updates Swift, Control UI, gateway tests, docs, and changelog coverage.

## Best Possible Solution

Keep the PR direction, but add the new scope-mismatch detail to client reconnect/pause handling and tests so valid but under-scoped device tokens surface pairing guidance without clearing stored tokens or looping.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: a device token whose token value is valid but whose scopes do not cover the requested operator scopes returns `scope-mismatch`, and current main collapses that before protocol/client recovery. I did not run a live iOS or browser session in this read-only review.

## Solution Assessment

No, not yet. The PR is the right narrow approach for preserving full shared-token sessions while narrowing bootstrap/device-token reconnects, but it needs to treat the new `AUTH_SCOPE_MISMATCH` code as terminal in reconnect gates before merge.
