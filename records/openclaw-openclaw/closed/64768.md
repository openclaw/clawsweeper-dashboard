## Summary

Keep this PR open. Current main still does not implement the Discord bot-identity failure fix: `fetchUser('@me')` failures are swallowed into an undefined bot identity, startup continues with optional `botUserId`, and the require-mention drop remains guarded by `botId`. The PR is a plausible candidate for the linked security bug, but review comments flag policy and test issues that need maintainer resolution rather than cleanup closure.

## Best Possible Solution

Keep this PR open or supersede it with a focused Discord bot-identity startup fix for #42219. Maintainers should decide the intended policy for unresolved bot identity, implement cleanup/reconnect suppression through the existing Discord lifecycle/gateway helper shape, remove unsafe botId-dependent mention short-circuits where appropriate, and add deterministic regression tests for both `fetchUser('@me')` rejection and missing-id responses.

## Risks / Open Questions

- Current main may still run Discord with `botUserId` undefined after bot identity lookup failure, weakening require-mention handling, self-message filtering, and reply-to-bot detection.
- The PR changes identity policy beyond its title by falling back to `applicationId`; maintainers need to choose fail-fast versus validated fallback behavior.
- The PR's best-effort `gateway.disconnect()` may be insufficient if Carbon reconnect scheduling remains enabled; current lifecycle code already has more nuanced gateway readiness/restart handling.
- The PR's timeout-based test can leak background work and should be made deterministic before landing.
