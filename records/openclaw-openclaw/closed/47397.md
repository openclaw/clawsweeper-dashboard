## Summary

Current `main` already prevents one billing-disabled auth profile from poisoning sibling profiles for the same provider in embedded-agent runs. Healthy profiles are ordered ahead of disabled ones even with explicit `auth.order`, and the embedded runner skips active `disabledUntil` profiles during both initial selection and profile rotation.

## Evidence

- **Explicit auth order no longer pins disabled profile first:** `resolveAuthProfileOrder` partitions explicit-order profiles into available vs cooldown/disabled and returns healthy profiles first, so a billing-disabled `anthropic:default` no longer blocks `anthropic:manual` just because it appears first in `auth.order`.
  - file: [src/agents/auth-profiles/order.ts:123](https://github.com/openclaw/openclaw/blob/3d8e6a3aa3a26870c4ae61fd27853914fc66cf96/src/agents/auth-profiles/order.ts#L123)
  - command: `git blame -L 123,153 -- src/agents/auth-profiles/order.ts`
  - sha: [3d8e6a3aa3a2](https://github.com/openclaw/openclaw/commit/3d8e6a3aa3a26870c4ae61fd27853914fc66cf96)
- **Embedded runner skips disabled profiles on first selection:** `initializeAuthProfile` checks each candidate with `isProfileInCooldown(...)` and advances past active cooldown/disabled windows before applying auth, so embedded runs do not stop at a billing-disabled first profile when a later profile is healthy.
  - file: [src/agents/pi-embedded-runner/run/auth-controller.ts:489](https://github.com/openclaw/openclaw/blob/3d8e6a3aa3a26870c4ae61fd27853914fc66cf96/src/agents/pi-embedded-runner/run/auth-controller.ts#L489)

- **Embedded runner skips disabled profiles on rotation after failure:** After a failover-triggering assistant error, `advanceAuthProfile` increments through the provider's profile candidates and continues past any candidate still marked unusable, allowing retry on the next healthy profile.
  - file: [src/agents/pi-embedded-runner/run/auth-controller.ts:450](https://github.com/openclaw/openclaw/blob/3d8e6a3aa3a26870c4ae61fd27853914fc66cf96/src/agents/pi-embedded-runner/run/auth-controller.ts#L450)
  - command: `nl -ba src/agents/pi-embedded-runner/run/auth-controller.ts | sed -n '450,478p'`

- **Disabled state is intentionally treated as profile-unavailable:** `isProfileInCooldown` now returns true when `disabledUntil` is active, not just `cooldownUntil`, so the embedded-runner skip logic and auth-order reordering both consistently treat billing-disabled profiles as blocked profiles rather than poisoning the whole provider.
