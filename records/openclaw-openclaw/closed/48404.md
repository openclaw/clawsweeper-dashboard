## Summary

Keep open. The issue is recent and concrete, and current main does not prove the ImmortalWrt/OpenWrt packaged startup segfault is fixed. The wrapper now has a root-help fast path that may avoid one `openclaw --help` path, but direct `node dist/entry.js --help` still runs entry bootstrap and packaged compile-cache setup before root help. The non-bot comment links a third-party `neural-bridge` skill, which is context but not a fix or duplicate.

## Best Possible Solution

Keep this issue open as the canonical OpenWrt/ImmortalWrt startup crash report. Ask for a retest on the latest release or current main on the same host, with `dmesg` segfault lines, Node build/source details, and differential runs with `NODE_DISABLE_COMPILE_CACHE=1`, `OPENCLAW_NO_RESPAWN=1`, wrapper `openclaw --help`, direct `node dist/entry.js --help`, and normal gateway startup. If it still reproduces, the likely fix belongs in the launcher/entry compile-cache/bootstrap path or package/native-dependency startup diagnostics, plus an OpenWrt-specific smoke or troubleshooting note.
