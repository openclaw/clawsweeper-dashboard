## Summary

Keep open: the WhatsApp upload-file direction is plausible, but the current diff introduces a blocking scoped-allowlist regression and this is a member-authored PR with the protected `maintainer` label.

## What This Changes

The PR adds a channel action lowering hook and SDK helper so WhatsApp `upload-file` path/URL requests are advertised and executed through the existing send-media transport, with docs, changelog, and regression tests.

## Best Possible Solution

Keep the channel-owned lowering seam, but gate the caller's original action once and add a regression test for `tools.message.actions.allow: ["upload-file"]` so internal send-media dispatch does not require exposing generic sends.

## Reproduction Assessment

Yes for the blocking PR regression: source inspection shows the PR lowers `upload-file` to `send` and then applies the exact-name allowlist check again. A scoped allowlist containing only `upload-file` would reject the internal `send`.

## Solution Assessment

No as written: the lowering helper is a maintainable direction, but the second allowlist check makes the advertised action fail for a documented scoped-permission configuration. The safer fix is to preserve the original requested action for permission checks and use the lowered action only for dispatch.
