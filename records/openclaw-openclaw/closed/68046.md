## Summary

Keep open. This is a focused and useful gateway bug fix, but it is not merge-ready because the passthrough accepts malformed or unknown MCP content blocks, the branch is currently conflicting, and no real behavior proof is attached.

## What This Changes

The PR widens HTTP MCP tool-result normalization to pass through text, image, audio, resource, and resource_link blocks and adds helper-level tests for those shapes.

## Best Possible Solution

Revise this PR or a narrow replacement so the gateway preserves only SDK-valid MCP content blocks, falls back to text for malformed or unknown entries, covers the HTTP tools/call route, rebases cleanly, and includes redacted real MCP behavior proof before merge.

## Reproduction Assessment

Yes. Source inspection shows current main returns normalizeToolCallContent(result) for HTTP MCP tools/call, and that helper drops every non-type/text field from array content blocks; I did not run a live HTTP MCP client in this read-only review.

## Solution Assessment

No, not as submitted. Preserving valid MCP content at the gateway boundary is the right location, but the patch should validate known SDK block shapes and required fields before passthrough rather than casting any object with a string type.
