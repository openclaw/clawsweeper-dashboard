Support parsing Harmony-format plain-text tool call blocks so LM Studio gpt-oss assistant text can be promoted into structured tool calls.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/78326. Current main's `parseStandalonePlainTextToolCallBlocks` only accepts bracket-style blocks, so LM Studio gpt-oss Harmony text such as `commentary to=read code {"path":"/path/to/file","line_start":1,"line_end":400}` is not promoted into a structured tool call. Add a narrowly scoped parser path for standalone Harmony openings (`commentary`, `analysis`, `final`, and preserved `<|channel|>...<|message|>` forms) that still requires allowed tool names when provided, parses only JSON object arguments, respects the existing max payload limit, and does not weaken existing bracket behavior. Update LM Studio stream buffering/promotion if needed so Harmony text is not emitted as plain text before promotion. Add regression tests in the shared parser and LM Studio stream tests, including positive Harmony forms, preserved delimiter forms, allowed-tool rejection, malformed JSON, ordinary prose that must not promote, and existing bracket behavior.
