## Summary

Keep this PR open. The feature is still not implemented on current main, but the branch is not merge-ready as-is: current main has moved the active memory implementation into the memory host SDK and memory-core plugin, the PR still targets the old `src/memory` layout, and the latest patch has a concrete watcher parity bug plus a data-scope/security review concern.

## What This Changes

The PR adds code-file detection, declaration-boundary chunking, indexing dispatch, watcher patterns, and code-chunking tests under the legacy `src/memory` memory implementation.

## Best Possible Solution

Land a current-main implementation that treats source-code memory files as an explicit supported input type in the memory host SDK and memory-core plugin, with discovery, watcher/filter parity, chunk-output reindex metadata, focused tests, and user-facing docs aligned with the broader chunking direction in #68019.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Watch uppercase code extensions:** `src/memory/manager-sync-ops.ts:390-392`
  - body: `detectCodeLanguage()` lowercases extensions, so `memory/example.TS` is accepted and tagged as code, but these watcher globs are built only from lowercase `CODE_EXTENSIONS` keys such as `*.ts`. On case-sensitive filesystems, edits or deletes to uppercase-extension code files will not mark memory dirty, leaving indexed chunks stale after startup. Make watcher matching case-insensitive or route through the same directory-watch filter logic used by current main so discovery and watch behavior match.
  - confidence: 0.87

## Security Review

Status: needs_attention

Summary: No new dependency or CI execution surface is introduced, but the PR expands builtin memory indexing to source files without explicit data-scope guardrails.

Concerns:

- **[medium] Expanded indexed-data scope lacks guardrails:** `src/memory/internal.ts:126`
  - body: The PR accepts recognized code extensions in `isAllowedMemoryFilePath()` and sends resulting chunks through the existing embedding path. With a remote embedding provider configured, source contents placed under memory paths can be sent off-host, but the branch does not add docs, opt-in language, or migration guidance for that expanded data scope.
