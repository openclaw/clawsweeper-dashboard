## Summary

Keep open. Current main has diagnostics for event-loop starvation and shared HTTP/WebSocket gateway plumbing, but I did not find an implementation that keeps webhook HTTP responsive during the reported stall or self-heals the listener.

## What This Changes

Request that gateway webhook HTTP endpoints remain responsive, or recover automatically, when agent/cron/model work blocks the event loop while existing WebSocket clients stay connected.

## Best Possible Solution

Keep the issue open and first isolate the workload that starves the gateway event loop, then add focused coverage proving webhook HTTP probes remain reachable or fail into an intentional recovery path while WebSocket clients are connected.

## Reproduction Assessment

No high-confidence live reproduction was established in this review. Source inspection does confirm the relevant current-main path: BlueBubbles webhooks, gateway HTTP probes, and WebSocket upgrades share the same Node gateway server/event loop, while current event-loop health is diagnostic-only.

## Solution Assessment

Unclear. The current diagnostic/status path is not enough to solve the reported webhook loss, and adding automatic restart without identifying the blocking workload could hide the underlying failure mode.
