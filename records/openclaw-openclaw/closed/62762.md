## Summary

Keep open. The report has concrete, recent reproduction evidence, and current main still lacks the local status/probe fallback that the open maintainer PR says it is adding using this issue's evidence. The linked PR is an implementation candidate, not a merged fix.

## Best Possible Solution

Keep this issue open and use #73724, or a replacement, to fix the remaining local gateway status/probe classification: separate loopback reachability from detail/read probe failures, add a bounded status-RPC fallback where appropriate, and cover `status --all`, `gateway probe`, `health --verbose`, and gateway-backed agent turns. After a fix merges, close with the exact fixing commit and shipped release provenance.

## Evidence

- **Reporter context remains concrete:** The issue body and follow-up comments document the same loopback 1006 failure through OpenClaw 2026.4.12 after Node 24 normalization, stable reinstall, and ACP/Discord isolation; the reported failing surfaces include `gateway probe --json`, `status --all`, `health --verbose`, and a minimal `openclaw agent` turn.
- **Current status path still depends on probe result:** `resolveGatewayProbeSnapshot` calls `probeGateway` directly, then sets `gatewayReachable` from `isProbeReachable(gatewayProbe)`. There is no status-RPC fallback when the probe times out before connect.
