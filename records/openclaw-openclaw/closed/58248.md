## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Reject non-token auth modes before rotating:** `src/commands/auth-rotate.ts:41-43`
  - body: This writes `gateway.auth.mode` to `"token"` for every plaintext config. A deployment intentionally using `password`, `none`, or `trusted-proxy` can be silently switched to token auth on disk, so the command should fail unless the effective mode is already token or a maintainer-approved migration path is added.
  - confidence: 0.94
- **[P1] Preserve env-managed gateway tokens:** `src/commands/auth-rotate.ts:22-25`
  - body: The command only checks `gateway.auth.token` in the config object and never rejects env-only tokens; config reads also expose `${VAR}` references after substitution. Rotation can therefore persist a new plaintext token and make future config-first auth ignore the env-managed token.
  - confidence: 0.91
- **[P2] Emit expiry warnings from the shared HTTP auth flow:** `src/gateway/http-auth-helpers.ts:32-40`
  - body: The warning is attached only to `authorizeGatewayBearerRequestOrReply`, while normal gateway HTTP endpoints go through the shared auth request path. In those common routes, `gateway.auth.tokenExpiryHours` will not be consumed, so the new config often has no observable effect.
  - confidence: 0.86
- **[P2] Reset expiry state in the gateway process:** `src/commands/auth-rotate.ts:50`
  - body: This updates process-local expiry state inside the short-lived CLI process, not the running gateway process that evaluates warnings. After rotating a live gateway token, the warning clock can remain stale until the gateway itself resets or restarts.
  - confidence: 0.88
