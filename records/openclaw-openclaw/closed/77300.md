## Summary

Keep open: current main still routes `models.providers.<provider>.apiKey` targets into auth-profile scrubbing and deletes the matching profile ref field, while audit/docs treat auth-profile credentials as higher precedence. Two open fix PRs already target this issue, so it should remain open until one is reviewed and merged.

## What This Changes

The issue requests that `secrets apply` preserve `auth-profiles.<provider>:default.keyRef` when applying a `models.providers.<provider>.apiKey` SecretRef plan with auth-profile scrubbing enabled.

## Best Possible Solution

Land one narrow fix that preserves existing auth-profile SecretRef fields while still scrubbing plaintext credentials, update the current destructive test into regression coverage, and close the issue after merge.

## Reproduction Assessment

Yes. Source inspection shows a `models.providers.*.apiKey` target is added to provider scrub targets, and with the default scrub option the scrub loop deletes the matching auth profile's `keyRef`.

## Solution Assessment

Yes. The narrow maintainable fix is to preserve canonical auth-profile SecretRefs while removing plaintext fields; adding another option would leave the destructive default path intact.
