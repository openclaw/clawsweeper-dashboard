## Summary

Current main already supports the requested shadow-mode behavior without adding new `sendPolicy.match.accountId` or `peer` fields: multi-account DM isolation is handled by `session.dmScope: "per-account-channel-peer"`, which encodes both account and peer into the direct-message session key, and `sendPolicy` can already match those keys via `keyPrefix` or `rawKeyPrefix`. The inbound-processing half of shadow mode was also already shipped by the `#65461` send-policy fix.

## Evidence

- **Send-policy matching is prefix-based today:** The config schema allows `channel`, `chatType`, `keyPrefix`, and `rawKeyPrefix` matches for send-policy rules; there are no dedicated `accountId` or `peer` fields because policy matching is expected to use session-key prefixes.
  - file: [src/config/zod-schema.allowdeny.ts:25](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/src/config/zod-schema.allowdeny.ts#L25)

  - sha: [cc0f3e0e4036](https://github.com/openclaw/openclaw/commit/cc0f3e0e40364023cc3a39227a411898ee68ce3b)
- **Per-account DM scope encodes account and peer into the session key:** With `session.dmScope: "per-account-channel-peer"`, direct-message session keys are built as `agent:<agentId>:<channel>:<accountId>:direct:<peerId>`, which makes both per-account and per-peer send-policy targeting possible via key-prefix matching.
  - file: [src/routing/session-key.ts:157](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/src/routing/session-key.ts#L157)

  - file: [src/routing/resolve-route.test.ts:548](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/src/routing/resolve-route.test.ts#L548)

## Risks / Open Questions

- This close assumes the user can move DM routing off `session.dmScope: "main"`; if they intentionally keep the shared main-session mode, send-policy cannot distinguish accounts because those DMs collapse into one session key.
- Discoverability is still weak: the capability exists, but the current send-policy docs describe prefix matching generically rather than giving an explicit multi-account WhatsApp shadow-mode example.
