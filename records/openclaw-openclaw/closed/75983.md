## Summary

Keep open. The branch implements the requested agent-aware skills visibility surface and fixes the earlier changelog/docs blockers, but it introduces a merge-blocking CLI success-path exit after stdout writes.

## What This Changes

The branch adds agent-scoped skill status reporting, skills CLI output changes, a default-agent doctor readiness repair path, docs/changelog updates, and focused tests.

## Best Possible Solution

Keep the agent-aware status design, remove the success-path `process.exit(0)`, and let normal CLI completion handle successful skills output before maintainer review of the doctor repair behavior.

## Reproduction Assessment

Not applicable for the feature request itself. For the review finding, yes: the PR head statically shows `writeStdout` followed by `defaultRuntime.exit(0)`, and `defaultRuntime.exit` calls `process.exit`.

## Solution Assessment

Mostly yes: reusing the existing effective agent skill filter is the right narrow design for visibility reporting. The success path should return normally instead of forcing process exit after stdout writes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Return normally after writing skills output:** `src/cli/skills-cli.ts:77`
  - body: `defaultRuntime.writeStdout()` writes to `process.stdout` without waiting for a drain event, and `defaultRuntime.exit(0)` immediately calls `process.exit`. On successful `skills list/info/check`, especially `--json` piped output, this can truncate stdout and bypass the `runCli` cleanup path. Drop the success exit and let the async action return like the existing implementation did.
