## Summary

Keep open: current main still lacks GitHub CLI `GH_CONFIG_DIR` guidance or a doctor/skill diagnostic for this mismatch, while `gh` itself documents that config lookup falls back to the process environment.

## What This Changes

Report requests clearer docs or diagnostics for GitHub CLI auth when the agent or Gateway environment uses a different `HOME` from the existing `gh` config location and `GH_CONFIG_DIR` is unset.

## Best Possible Solution

Add GitHub skill setup/troubleshooting and, ideally, a focused doctor or skills-check warning that points users to `GH_CONFIG_DIR` when `gh` is installed but auth is hidden by a different runtime `HOME`.

## Reproduction Assessment

Yes, source-reproducible: `gh` documents `GH_CONFIG_DIR` and HOME-based config fallback, and current main has no OpenClaw guidance or diagnostic for that mismatch. I did not live-prove it against a root-owned systemd setup in this read-only pass.

## Solution Assessment

Yes, a narrow docs plus doctor/skills diagnostic is the maintainable fix; core should not copy or migrate GitHub CLI credentials between homes. The existing service env-file path is already a reasonable place to document the operator-supplied `GH_CONFIG_DIR`.
