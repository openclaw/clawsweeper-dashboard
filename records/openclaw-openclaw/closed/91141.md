## Summary

## What This Changes

Fix concurrent inbound session writes being misclassified as embedded-session takeover, which can abort the active reply and remove the newly accepted user turn from the transcript.

## Best Possible Solution

Complete the canonical single-writer transactional transcript migration, remove the file-fingerprint fence only after cutover, and add an acceptance test proving a concurrent inbound user message survives the active run’s model-I/O window and any subsequent failed-attempt repair.

## Reproduction Assessment

Yes at source level: current main still treats an unowned user append during prompt-lock release as takeover-significant and retains the reported orphan-user repair path. A live current-main Matrix timing reproduction was not performed.

## Solution Assessment

The reporter’s tactical ownership suggestion is plausible, but the maintainer-approved transactional SQLite migration is the better durable solution because it removes the file-fingerprint race rather than adding more benign-write exceptions; it must explicitly cover user-turn preservation.
