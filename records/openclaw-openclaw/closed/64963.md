## Summary

Keep open: current main still lacks the active reply-run heartbeat guard, and this PR is a narrow, source-backed fix with regression coverage and sufficient post-patch live output proof. I found no blocking correctness or security findings; normal maintainer review and merge gating remain the path forward.

## What This Changes

The PR imports the reply-run registry into `runHeartbeatOnce`, skips heartbeat execution when the resolved session still has an active reply run, and adds a regression test for an idle lane with an active reply operation.

## Best Possible Solution

Land this focused guard and regression test after normal maintainer review, while tracking broader heartbeat replay, default isolation, and cross-session contamination work separately.

## Reproduction Assessment

Yes, at source level: seed a Telegram-backed session, enqueue a system event, register a running reply operation for the same session key, force queue sizes to zero, and call `runHeartbeatOnce`. Current main lacks the active reply-run check, so that path reaches heartbeat execution instead of the retryable busy skip.

## Solution Assessment

Yes for the narrow race. Checking the existing session-keyed reply-run registry after heartbeat preflight and returning the existing retryable `requests-in-flight` skip is the smallest maintainable fix; broader stale replay or default-isolation behavior belongs in separate follow-up work.
