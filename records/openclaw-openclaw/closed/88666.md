## Summary

Keep open: the detector change is useful, but the status/deps path still maps npm installs to `package-lock.json` only, so a shrinkwrap-only npm global install would still report a missing lockfile after this PR. The supplied proof covers a detector simulation, not the full `openclaw status` path affected by the bug.

## What This Changes

The PR changes `detectPackageManager()` to prefer lock files over `package.json.packageManager`, adds `npm-shrinkwrap.json` as an npm marker, and updates detector tests.

## Best Possible Solution

Land one status-surface fix that treats `npm-shrinkwrap.json` as both npm install evidence and npm deps-lock evidence, with a `checkUpdateStatus` regression and redacted after-fix status output from an npm package/global install shape.

## Reproduction Assessment

Yes, at source level. Current main reads the pnpm `packageManager` field before lock files, and the PR would then send the reported shrinkwrap-only npm package root into a downstream npm deps check that still expects `package-lock.json`.

## Solution Assessment

No. Reordering detection is a useful part of the repair, but the narrow maintainable fix also needs the status/deps lockfile path to recognize `npm-shrinkwrap.json` and test the full `checkUpdateStatus` surface.
