## Summary

Keep open. Current main fixes the original Anthropic API-key header route, but the remaining diagnostic/docs gap is still present: manually stored Anthropic `token` profiles are accepted, resolved as runtime credentials, can sort ahead of `api_key`, and are not flagged by doctor/status as likely API-key misconfiguration.

## What This Changes

The issue requests safer handling, diagnostics, and documentation for Anthropic Enterprise/API-key credentials stored in `auth-profiles.json` as `type: "token"`.

## Best Possible Solution

Preserve Anthropic setup-token support, but add targeted doctor/status guidance and docs that flag API-key-shaped Anthropic credentials stored as `token` and direct API-only users to canonical `api_key` profiles.

## Reproduction Assessment

No for the exact Enterprise 529 loop without live Enterprise credentials. Yes for the remaining source path: current main still accepts and selects Anthropic `token` profiles without a targeted diagnostic when the credential is API-key-shaped.

## Solution Assessment

No, removing `token` support or trying to auto-detect Enterprise accounts is not the best fix because setup-token remains supported. The narrow maintainable solution is diagnostics/status/docs while keeping the current value-based header routing.
