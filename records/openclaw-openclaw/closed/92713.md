## Summary

Keep open: the patch is a useful Telegram transport fix in the right area, but it is not merge-ready because it only treats config-sourced DNS order as explicit and leaves the documented environment override on the same IPv4 fallback path; the PR also has only unit-test proof for a live Telegram/network behavior change.

## What This Changes

The PR changes Telegram transport fallback so a non-`ipv4first` `channels.telegram.network.dnsResultOrder` skips the sticky IPv4 and pinned-IP fallback attempts, with a focused fetch test.

## Best Possible Solution

Extend the fallback opt-out to documented env-sourced non-`ipv4first` DNS decisions while keeping implicit process/default behavior unchanged, add regression coverage, and provide live Telegram media-download proof for the affected network shape.

## Reproduction Assessment

Yes at source level: current main creates IPv4 and pinned-IP fallback attempts after honoring Telegram network config, and media downloads consume those attempts. I did not establish a live IPv4-broken, IPv6-working Telegram run in this read-only review.

## Solution Assessment

No. The fix is in the right module, but it is one-sided because it excludes the documented env DNS override; the narrow best fix is to handle explicit env/config overrides consistently and then prove the live media path.
