## Summary

Keep open. Current main still has a source-supported path where a Telegram DM can be accepted into the session queue behind active embedded compaction without a DM-specific queued acknowledgement, and the takeover/fairness parts are covered by open related work rather than a merged fix.

## What This Changes

Investigate a Telegram direct-message stall where inbound work appears silently queued behind embedded auto-compaction/session takeover before a delayed reply.

## Best Possible Solution

Coordinate the open Telegram queued-status PR with scheduler fairness and session-takeover fixes so Telegram DMs get a per-message queued or failure signal, followup work cannot starve fresh user/control work, and compaction takeover errors recover or fail visibly.

## Solution Assessment

Unclear as a single implementation path. The per-message queued signal in https://github.com/openclaw/openclaw/pull/46055 is the right Telegram UX direction, while https://github.com/openclaw/openclaw/issues/54488 and https://github.com/openclaw/openclaw/issues/83615 cover separate scheduler fairness and session-takeover recovery work.
