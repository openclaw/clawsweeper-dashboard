## Summary

Current main does not reproduce the reported Cloudflare Tunnel header-path failure: when `publicUrl` is configured, Twilio verification uses that configured public URL directly rather than depending on forwarded-host reconstruction, and existing regression tests cover both `publicUrl` verification and forwarded-host allowlisting.

## Evidence

## Close Comment

Closing from a Codex review. On current `main` (`0e7250f37bf347c72790abf2a6692d5ec84e5ea1`), Twilio webhook verification is not Cloudflare-header dependent when `publicUrl` is configured: the runtime keeps `config.publicUrl` as the canonical public URL (`extensions/voice-call/src/runtime.ts:272`, `extensions/voice-call/src/runtime.ts:297`), the Twilio provider passes that URL into verification (`extensions/voice-call/src/providers/twilio/webhook.ts:11`), and `buildTwilioVerificationUrl()` uses `publicUrl` before any forwarded-header reconstruction (`extensions/voice-call/src/webhook-security.ts:343`, `extensions/voice-call/src/webhook-security.ts:621`). There are also regression tests covering both `publicUrl` verification and forwarded-host allowlists (`extensions/voice-call/src/webhook-security.test.ts:439`, `extensions/voice-call/src/webhook-security.test.ts:594`). Your report also logged `Verification URL: https://oc1.example.com/voice/webhook`, which already matches the configured public URL, so the suspected Cloudflare Tunnel header-reconstruction bug does not line up with current code. If this still reproduces on `v2026.4.22`, please open a fresh issue with the exact inbound Twilio headers/body from a failed request (redacting secrets) so maintainers can compare the signed payload directly.
