## Summary

Keep open. Current main still exposes declared node commands before node-pair approval, and the PR moves in the right fail-closed direction, but it leaves already-connected nodes registered with an empty live command list after approval, so approved commands remain unavailable until reconnect.

## What This Changes

The PR changes node connection reconciliation to publish no effective commands for unpaired nodes and updates gateway tests to pre-approve invoke fixtures and assert pre-pairing node.invoke denial.

## Best Possible Solution

Land a fail-closed command gate for unapproved node-pair connections, then make approved commands available through an explicit post-approval path with tests for pre-approval blocking, same-connection post-approval availability, and existing approved-node invoke flows.

## Reproduction Assessment

Yes. Source and tests provide a high-confidence path: connect a device-paired node without node-pair approval, observe declared commands on `node.list`, then invoke an allowlisted command from an `operator.write` session.

## Solution Assessment

No, not as submitted. Returning an empty command list for unpaired connects is the right boundary, but the patch should also refresh the live session or authorize/list from approved node-pair state after `node.pair.approve`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Refresh live commands after node pairing approval:** `src/gateway/node-connect-reconcile.ts:84`

Approval only persists the paired record and broadcasts resolution, while `node.list` and `node.invoke` still read `NodeSession.commands`, so newly approved commands do not work until reconnect.
