Close as duplicate/superseded: the symptom matches the existing Discord/group-channel visible-reply cluster, and the provided config plus current source/docs point to documented tool-only reply behavior rather than a new Discord transport failure.

## What This Changes

The report asks for Discord guild-channel mentions to produce visible bot replies when the bot processes the message but does not post a final response.

## Best Possible Solution

Keep the current tool-only guild/channel default; users who want legacy automatic room replies should set messages.groupChat.visibleReplies to "automatic", while any default-policy change should continue through https://github.com/openclaw/openclaw/pull/75327.

## Reproduction Assessment

Yes for the expected no-auto-post path: current source, tests, docs, and the reporter's message_tool config show why a processed Discord channel turn can have typing/thinking but no visible final reply. No separate transport failure is reproduced from the provided logs.

## Solution Assessment

No, keeping this issue open as a new bug is not the best path because the behavior is already documented and covered by prior reports; review the existing product-change PR if maintainers want mentioned group/channel turns to bypass tool-only mode.
