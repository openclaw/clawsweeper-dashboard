## Summary

Keep open: current `main` still lets the shared CLI port parser accept values above 65535, even though the CLI error contract and comparable port helpers use the 1-65535 range.

## What This Changes

The issue asks for `parsePort` to reject numeric ports above 65535 instead of returning them as valid.

## Best Possible Solution

Add the 1-65535 bound in the shared `parsePort` helper and cover boundary values with a focused regression test so all CLI callers share the same contract.

## Reproduction Assessment

Yes. Source inspection shows `parsePort(65536)` and `parsePort('99999')` flow through a positive-integer-only helper, and a Node `net` probe confirms those ports are invalid at the runtime boundary.

## Solution Assessment

Yes. The narrow maintainable fix is to cap the shared parser at 65535 and add regression coverage rather than adding more caller-specific checks.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/83900. On current main, `src/cli/shared/parse-port.ts` delegates to `parseStrictPositiveInteger` and accepts ports above 65535 even though CLI messages and other port helpers use the 1-65535 range. Update the shared `parsePort` helper to return `null` unless the parsed integer is between 1 and 65535 inclusive. Add focused Vitest coverage, preferably `src/cli/shared/parse-port.test.ts`, for accepting `1` and `65535` and rejecting `0`, `65536`, `99999`, non-integers, junk strings, `null`, and `undefined`. Keep scope limited to parser/tests plus the required changelog fix entry; do not change config schemas, CLI wording, or service behavior beyond the shared parser contract.
