## Summary

Keep this PR open, but not merge-ready: it addresses a real Feishu message-delivery bug, yet the current suppression condition can discard a valid context reply target and the supplied proof explicitly is not from a live Feishu card click.

## What This Changes

The PR updates Feishu card-action synthetic message construction to suppress `reply_target_message_id` when the callback top-level `open_message_id` starts with `card-action-c-`.

## Best Possible Solution

Preserve real `context.open_message_id` targets, suppress only the selected temporary target, add a mixed-ID regression test, and require redacted live Feishu proof before merge.

## Reproduction Assessment

No high-confidence live reproduction is present. The source path is clear from synthetic callback construction through the Feishu reply dispatcher, but the PR body says the live Feishu tenant/card-click path was not tested.

## Solution Assessment

No, this is only a plausible mitigation. The safer fix is to preserve a valid `context.open_message_id`, suppress only the chosen temporary `card-action-c-*` target, and cover the mixed-ID case before merge.
