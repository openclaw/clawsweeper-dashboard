## Summary

Keep open: current main still uses the full-payload base64 regex, and this PR is a plausible narrow fix, but it is not merge-ready because contributor real behavior proof is missing and the branch currently fails lint on the new one-line padding check.

## What This Changes

This PR replaces gateway attachment base64 validation’s whole-string regex with a non-decoding character-code scan in `src/gateway/chat-attachments.ts`.

## Best Possible Solution

Land one non-regex, non-decoding gateway validator with the lint issue fixed, focused large-payload regression coverage, and redacted real WebChat/gateway proof, then use it to resolve the related attachment-crash bug.

## Reproduction Assessment

No high-confidence live reproduction was established in this review. The source path still uses the reported regex and the related gateway logs name `isValidBase64`, but prior focused large-string probes did not reproduce the RangeError outside the reporter’s runtime.

## Solution Assessment

Mostly yes in direction: a non-decoding linear validator is the narrow maintainable fix because it preserves the estimate-before-decode contract. As written, this PR still needs the lint fix, focused regression coverage, and contributor real behavior proof before merge.
