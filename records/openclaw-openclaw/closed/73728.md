## Summary

Current main removes the reported one-second plugin discovery/manifest snapshot TTL path entirely: the old cache-control module and env knobs are gone, docs now forbid wall-clock metadata caches, and provider/capability paths use explicit metadata snapshots or runtime registry reuse instead. The related #73853 discussion was useful and that PR is now merged for adjacent reload paths, but the specific default-TTL blocker no longer exists on main.

## What This Changes

Request to raise or gateway-scope the default plugin discovery and manifest cache TTLs because the previous 1s defaults caused conversation-paced gateway cache misses.

## Best Possible Solution

Keep the current main design: metadata freshness comes from explicit snapshot/registry ownership, while performance-sensitive gateway paths reuse active or compatible runtime registries and scoped runtime loads instead of restoring arbitrary wall-clock discovery or manifest TTLs.

## Reproduction Assessment

Yes. The report is reproducible by inspecting v2026.4.27, where the defaults are 1000 ms and snapshot TTL uses Math.min; current main verification shows those constants and cache controls are gone.

## Solution Assessment

Yes. Removing hidden wall-clock metadata caches and documenting explicit snapshot ownership is a narrower, more maintainable fix than choosing a longer gateway default while preserving stale metadata behavior.
