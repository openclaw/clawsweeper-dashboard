## Summary

Keep open: the latest branch fixes the managed-requirements downgrade issue, but it still preserves `approvalPolicy: "never"` when OpenClaw tool policy promotion is active, which can bypass the security policy added for Codex-native tools. The external PR also still needs real patched gateway/app-server proof before merge.

## What This Changes

The PR updates the Codex app-server path to read the bridged Codex home for disabled hooks, downgrade eligible non-never approval policies to `never`, emit warnings, and add focused tests plus native relay diagnostics.

## Best Possible Solution

Keep the bridged Codex-home detection, but fail closed or suppress the downgrade whenever OpenClaw tool policy or explicit approval requirements need hook-mediated approvals.

## Reproduction Assessment

Yes for the blocking finding: source inspection shows the branch returns before promotion when `shouldPromote` is true, and its new test pins that behavior. I did not run a live gateway because this review is read-only.

## Solution Assessment

No. The requirements gate is the right direction, but preserving `never` while a `before_tool_call` policy is active is not the narrowest safe fix; the policy path should stay fail-closed or report a blocked downgrade.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Fail closed when policy hooks would be bypassed:** `extensions/codex/src/app-server/run-attempt.ts:509-510`
  - body: `approvalPolicyDowngrade` now makes this helper return `never` even when `shouldPromote` is true, which is the path used when OpenClaw has a `before_tool_call` policy. The PR premise is that `features.hooks=false` makes the native hook relay unusable, so preserving `never` lets Codex-native commands run without the OpenClaw policy that current main promotes to `untrusted` to enforce. Suppress the downgrade, or keep the fail-closed/promotion behavior, when an OpenClaw tool policy is active.
  - confidence: 0.87
