## Summary

Issue #17072 is better scoped as an out-of-core governance/router plugin or ClawHub integration. Current main already provides plugin seams for pre-tool permission gating, post-tool-call telemetry, low-level harness routing, and ACP provenance receipts, while the repo vision explicitly keeps optional/heavy orchestration out of core.

## Close Comment

Codex review on `a59d1bd46d9fc0bd2d9194b3f17db2561efc51e3`: closing this as `clawhub`. This is a useful governance/router idea, but it fits OpenClaw’s plugin/ClawHub boundary better than core. `VISION.md:54` says optional capability should usually ship as plugins, and `VISION.md:101` plus `VISION.md:107` explicitly steer new optional skills to ClawHub and avoid heavy orchestration layers in core. Current main already exposes the needed seams for an out-of-core implementation: pre-tool gating via `before_tool_call` and approval requests (`docs/plugins/building-plugins.md:184`, `src/plugins/hook-types.ts:324`, `src/agents/pi-tools.before-tool-call.ts:219`), post-tool telemetry via `after_tool_call` (`src/plugins/hook-types.ts:339`), low-level execution routing via `api.registerAgentHarness(...)` (`docs/plugins/sdk-overview.md:70`), and provenance receipts on ACP ingress (`docs/cli/acp.md:296`, `src/acp/translator.ts:695`). Given that scope and the existing extension points, this should live as a plugin/ClawHub integration rather than new OpenClaw core surface.
