## Summary

Keep this draft PR open, but do not merge it as-is. The patch still sanitizes the shared `senderName` before tool-policy resolution, while current main treats that field as sender identity for `toolsBySender` `name:` matching; the claimed OpenAI user-message `name` root cause remains unproven in the checked serializers and upstream pi-ai source.

## What This Changes

The PR adds `sanitizeModelName`, applies it to `senderName` passed through embedded Pi agent/attempt paths, and adds unit coverage for the sanitizer.

## Best Possible Solution

Capture the actual failing outbound payload or add a failing serializer-level repro for #61192, then sanitize only the concrete provider payload field or introduce a separate model-safe name while preserving raw display identity for `toolsBySender` and plugin policy resolution.

## Reproduction Assessment

No for the claimed model 400: the report has logs and a workaround, but no captured outbound payload or failing test showing a sender display name serialized as OpenAI `messages[i].name`. The PR regression is reproducible from code by using a `toolsBySender` `name:John Doe` rule and observing that the patch changes the policy input to `John_Doe`.

## Solution Assessment

No. Sanitizing the shared `senderName` at runner/attempt boundaries is not the narrowest maintainable fix because that field is also policy identity; a safer fix would keep raw sender identity and sanitize only a transport-only model payload field after the actual offending serializer is proven.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep raw senderName for tool policy resolution:** `src/agents/pi-embedded-runner/run.ts:605`
  - body: This value is still shared with `runEmbeddedAttempt` and `createOpenClawCodingTools`, where `resolveGroupToolPolicy` forwards `senderName` into `toolsBySender` `name:` matching. Sanitizing it here changes `name:John Doe` into `John_Doe` and can collide with a real `John_Doe`, so existing group allow/deny rules can stop applying or apply to the wrong sender. Use a separate transport-only model name instead.
  - confidence: 0.91

## Security Review

Status: needs_attention

Summary: The diff introduces an access-control risk by mutating sender identity before group tool-policy matching.

Concerns:

- **[medium] Sanitized sender names can bypass or misapply tool policy:** `src/agents/pi-embedded-runner/run.ts:605`
  - body: `toolsBySender` supports display-name keyed policies, and this patch collapses different raw display names to the same sanitized token before policy resolution. In name-based group policies, that can deny the intended sender or grant tools to the wrong sender.
  - confidence: 0.88
