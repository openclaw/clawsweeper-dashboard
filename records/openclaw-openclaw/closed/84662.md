## Summary

Keep open: current main and the latest tagged release still prepend `OpenClaw runtime context for this turn:` into Codex `turn/start.input`, and the reporter supplied structured live evidence that those decorated user items accumulate in native `response.create` and compact inputs. The related fix PR was closed unmerged after prompt-boundary and real-proof concerns, so the narrow session-state bug remains unresolved.

## What This Changes

The issue asks Codex app-server to stop persisting per-turn OpenClaw runtime context as ordinary native Codex user history.

## Best Possible Solution

Use a Codex-supported non-persisted, lower-authority path for current-turn reference context, or a maintainer-approved split of trusted instructions from user-editable reference data, and add stale-history plus multi-turn/compaction regression coverage.

## Reproduction Assessment

Yes. Current main visibly decorates Codex `turn/start.input` as a text user item, and the reporter's sanitized JSONL/DB analysis shows the same marker accumulating in model-visible `role=user` `response.create` and compact inputs.

## Solution Assessment

Unclear. The fix belongs at the Codex app-server prompt/history boundary, but the closed related PR showed that the exact transient-context channel and prompt-authority split need maintainer approval before another implementation should land.
