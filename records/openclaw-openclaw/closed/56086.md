## Summary

Current `main` already addresses the report: stale-socket health decisions now key off provider-proven transport activity rather than inbound app-event freshness, and the health monitor restart path is channel-scoped instead of a full gateway restart.

## Evidence

- **Stale-socket now depends on transport activity, not inbound app events:** `evaluateChannelHealth` only performs stale-socket checks when `lastTransportActivityAt` is present, with an explicit comment that app-level events are not socket liveness.

- **Regression test covers the Discord false-positive case:** The test suite now asserts that a connected Discord-style channel with only `lastEventAt` and no `lastTransportActivityAt` remains healthy instead of being marked `stale-socket`.
  - file: [src/gateway/channel-health-policy.test.ts:139](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/src/gateway/channel-health-policy.test.ts#L139)
