## Summary

Keep open. Current `main` does not implement the proposed local CDP bridge, and the present browser contract still treats WSL2 split-host setups as raw remote CDP rather than `existing-session` via a local bridge.

## Evidence

- **WSL2 guidance still says to use raw remote CDP, not existing-session:** The current troubleshooting guide says `existing-session` / `user` should be used only when the Gateway runs on the same host as Chrome, and for WSL2 + Windows it explicitly says to prefer raw remote CDP.
  - file: [docs/tools/browser-wsl2-windows-remote-cdp-troubleshooting.md:34](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/docs/tools/browser-wsl2-windows-remote-cdp-troubleshooting.md#L34)
  - command: `nl -ba docs/tools/browser-wsl2-windows-remote-cdp-troubleshooting.md | sed -n '20,45p'`
  - sha: [56fe2aab9c87](https://github.com/openclaw/openclaw/commit/56fe2aab9c87596515f2280c8060dc740ea64a88)
- **Current browser docs forbid cdpUrl on existing-session profiles:** The browser docs state that `driver: "existing-session"` uses Chrome DevTools MCP instead of raw CDP and say not to set `cdpUrl` for that driver.
  - file: [docs/tools/browser.md:163](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/docs/tools/browser.md#L163)

- **Current config only auto-creates `user`, not `chrome-relay`:** `resolveBrowserConfig()` adds the built-in `user` existing-session profile, and the config tests assert that `chrome-relay` is no longer auto-created.
  - file: [extensions/browser/src/browser/config.ts:180](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/extensions/browser/src/browser/config.ts#L180)
