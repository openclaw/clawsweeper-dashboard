## Summary

Current `main` does not have a path where plain-text Discord sends become modal/form payloads unless the caller explicitly provides a valid Discord component modal. Empty/default interactive payloads are normalized away, and text-only sends still route through the normal text send path.

## Evidence

- **Empty/default interactive payloads are discarded:** `normalizeInteractiveReply` only returns a value when normalized blocks remain, and `hasInteractiveReplyBlocks` is just a truthiness check on that normalized result.
  - file: [src/interactive/payload.ts:194](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/src/interactive/payload.ts#L194)

  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
- **Discord interactive conversion cannot create modals from plain interactive replies:** `buildDiscordInteractiveComponents` only emits text/action rows for text, buttons, and selects, and returns `undefined` when no usable blocks remain.
  - file: [extensions/discord/src/shared-interactive.ts:24](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/extensions/discord/src/shared-interactive.ts#L24)

- **Discord outbound falls back to normal text/media delivery:** `discordOutbound.sendPayload` uses `sendTextMediaPayload` when no Discord component spec exists, so plain-text sends do not go down the component path.
  - file: [extensions/discord/src/outbound-adapter.ts:189](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/extensions/discord/src/outbound-adapter.ts#L189)

  - file: [extensions/discord/src/components.ts:434](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/extensions/discord/src/components.ts#L434)
