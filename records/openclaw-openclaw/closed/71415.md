## Summary

Current `main` resolves `--session-id` through explicit session-key lookup instead of silently collapsing to `agent:<id>:main`, and it has focused tests covering the exact regression path reported here. The fix appears landed on `main` after the latest release tag, so this is fixed on trunk but not yet in `v2026.4.23`.

## Evidence

- **Session resolver creates explicit keys for unknown `--session-id` values:** `resolveSessionKeyForRequest()` now keeps `--session-id` distinct from an agent main-session shortcut and, when no existing match is found, synthesizes a stable explicit key like `agent:<agent>:explicit:<sessionId>` instead of falling back to `agent:<agent>:main`.
  - file: [src/agents/command/session.ts:147](https://github.com/openclaw/openclaw/blob/5163a2fbf771dd90b1b5f51985bd4b5946fac038/src/agents/command/session.ts#L147)
  - command: `nl -ba src/agents/command/session.ts | sed -n '145,205p'`
  - sha: [5163a2fbf771](https://github.com/openclaw/openclaw/commit/5163a2fbf771dd90b1b5f51985bd4b5946fac038)
- **Regression test covers `--agent` + `--session-id` not remapping to main:** The command-session test explicitly asserts that `--agent` does not short-circuit `--session-id` back to the agent main session, and that a missing scoped session id resolves to `agent:mybot:explicit:target-session-id` rather than an existing main-session row.
  - file: [src/commands/agent/session.test.ts:106](https://github.com/openclaw/openclaw/blob/5163a2fbf771dd90b1b5f51985bd4b5946fac038/src/commands/agent/session.test.ts#L106)
  - command: `nl -ba src/commands/agent/session.test.ts | sed -n '106,169p'`
  - sha: [5163a2fbf771](https://github.com/openclaw/openclaw/commit/5163a2fbf771dd90b1b5f51985bd4b5946fac038)
- **End-to-end session resolution test for explicit session-id-only runs:** The agent session test asserts that `resolveSession({ sessionId: "explicit-session-123" })` produces `agent:main:explicit:explicit-session-123` and preserves the requested session id.
  - file: [src/commands/agent.session.test.ts:69](https://github.com/openclaw/openclaw/blob/5163a2fbf771dd90b1b5f51985bd4b5946fac038/src/commands/agent.session.test.ts#L69)
  - command: `nl -ba src/commands/agent.session.test.ts | sed -n '68,79p'`
  - sha: [5163a2fbf771](https://github.com/openclaw/openclaw/commit/5163a2fbf771dd90b1b5f51985bd4b5946fac038)
- **Session store persists and reloads explicit session-id runs:** The session-store test verifies that an explicit-session-id-only run stores backend session binding metadata under the explicit key and resolves the same key again on follow-up, which addresses the stale-history leakage reported in the issue.
  - file: [src/agents/command/session-store.test.ts:453](https://github.com/openclaw/openclaw/blob/5163a2fbf771dd90b1b5f51985bd4b5946fac038/src/agents/command/session-store.test.ts#L453)
  - command: `nl -ba src/agents/command/session-store.test.ts | sed -n '453,517p'`
  - sha: [5163a2fbf771](https://github.com/openclaw/openclaw/commit/5163a2fbf771dd90b1b5f51985bd4b5946fac038)
- **CLI gateway path uses the updated resolver before sending the run:** `openclaw agent` in gateway mode now calls `resolveSessionKeyForRequest({ agentId, to, sessionId })` and forwards that resolved key with the requested session id, so the user-facing CLI path uses the fixed behavior.
  - file: [src/commands/agent-via-gateway.ts:121](https://github.com/openclaw/openclaw/blob/ec8dbc4595583816f71af2568df61dbe7c090e1f/src/commands/agent-via-gateway.ts#L121)
