## Summary

Keep this PR open. The underlying bug is valid: current main still sends plugin approval requests without same-chat route metadata, while the gateway already supports routed plugin approvals. The proposed patch should not merge as-is because it reads `turnSource*` from `baseParams`, which is the tool-call parameter path, so approval routing would be influenced by model/plugin-provided input instead of trusted runtime context.

## What This Changes

The PR forwards `turnSourceChannel`, `turnSourceTo`, `turnSourceAccountId`, and `turnSourceThreadId` from before-tool-call `baseParams` into `plugin.approval.request`, adds two regression tests, and adds a changelog entry for #74003.

## Best Possible Solution

Thread trusted turn-source metadata from the gateway or embedded run context into `HookContext` or an equivalent trusted agent-side structure, forward only those trusted fields to `plugin.approval.request`, and keep focused coverage for routed and route-less plugin approval callers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Read approval routes from trusted context:** `src/agents/pi-tools.before-tool-call.ts:165-178`
  - body: This forwards `turnSource*` from `params.baseParams`, but `baseParams` is the original or policy-adjusted tool-call params passed through `runBeforeToolCallHook`. A model/plugin-influenced tool call can therefore fabricate route fields and affect where `plugin.approval.request` sends or keeps an approval. Route metadata needs to come from trusted runtime context, or be validated server-side, before it is forwarded.
