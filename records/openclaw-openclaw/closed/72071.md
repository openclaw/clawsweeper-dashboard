## Summary

Current main already addresses the reported Telegram leak path. The Ollama native stream adapter treats Kimi cloud tool calls as structured assistant content instead of visible text, has tests for tool-call-only and mixed text/tool responses, and the Telegram/direct-session output path extracts only text blocks before applying reasoning/tool-artifact sanitizers. The issue was reported against OpenClaw 2026.4.15; I could not determine a specific shipped release tag for the current implementation, so no fixed release is claimed.

## Best Possible Solution

Close this as implemented on current main. Keep the native Ollama/Kimi structured tool-call path and the Telegram text-only sanitizer path; if a fresh leak reproduces on a current build using `api: "ollama"` and a native Ollama base URL, track it as a new focused bug with transcript and config evidence.
