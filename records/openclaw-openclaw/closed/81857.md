## Summary

Keep this PR open: the current direction fixes the reported loaded-plugin/unloaded-sibling failure, but the new filter compares canonical runtime plugin ids with raw config ids and can drop explicitly configured plugins that rely on supported legacy aliases.

## What This Changes

The PR changes plugin tool resolution to filter loaded-registry lookup to loaded or explicitly configured plugin ids while keeping cold-load fallback scoped to the full manifest-resolved plugin set, and adds a regression test for a loaded plugin with an unloaded bundled sibling under `group:plugins`.

## Best Possible Solution

Normalize explicitly configured plugin ids before the loaded-registry filter, add a regression for legacy alias config, then land the narrow resolver fix once focused plugin-tool validation and normal CI are green.

## Reproduction Assessment

Yes. Current main's source path requires every manifest-selected plugin id to be present before using a loaded registry, and the supplied before/after Telegram proof shows the reported Active Memory path moving from unavailable to ok after the patch.

## Solution Assessment

No. The loaded-registry/cold-load split is the right shape, but the implementation should compare `runtimePluginIds` against normalized explicit plugin config, not raw `plugins.allow` and raw `plugins.entries` keys.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Normalize configured plugin ids before filtering:** `src/plugins/tools.ts:968-986`
  - body: `runtimePluginIds` has already gone through plugin-id normalization, but this new filter records raw `plugins.allow` values and raw `plugins.entries` keys. Legacy aliases such as `openai-codex` normalize to `openai`; when another loaded plugin survives `filteredPluginIds`, the aliased plugin is omitted from `resolvedPluginIds` and never reaches cold-load, so explicitly configured plugin tools can disappear. Populate this set from `normalizePluginsConfig(context.config?.plugins)` instead.
  - confidence: 0.82
