## Summary

Current `main` already clears stale `sessionFile` metadata on cron session rollover and repopulates it from the new `sessionId`, which addresses the reported `sessionId`/`sessionFile` mismatch for persistent `session:...` cron targets.

## Evidence

- **Rollover clears stale transcript pointer:** `resolveCronSession` preserves session overrides but, when a cron session rolls to a new session, explicitly clears `sessionFile` alongside other inherited routing state before returning the updated session entry.
  - file: [src/cron/isolated-agent/session.ts:68](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/src/cron/isolated-agent/session.ts#L68)
  - command: `nl -ba src/cron/isolated-agent/session.ts | sed -n '1,260p'`
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)
- **New session file is derived from new sessionId:** The cron isolated-agent run path calls `resolveCronSession(...)` for both isolated and persistent `session:...` jobs, and if `sessionFile` is absent it immediately assigns `resolveSessionTranscriptPath(runSessionId, agentId)`, keeping the stored transcript path aligned with the rolled `sessionId` before persistence.
  - file: [src/cron/isolated-agent/run.ts:481](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/src/cron/isolated-agent/run.ts#L481)
