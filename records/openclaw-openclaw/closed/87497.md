## Label Justifications

- `P0`: The report describes a replay-based security boundary bypass that can mint a fresh realtime voice stream credential from a previously valid signed webhook.
- `impact:security`: The affected behavior crosses webhook replay/auth boundaries and exposes a new realtime stream capability.

## Summary

Keep open: current `main` and the latest release still show the replay check after the realtime TwiML path that can issue a stream token, and the linked open fix PR should be reviewed and landed or closed before this issue is resolved.

## What This Changes

The issue reports that replayed Twilio voice-call webhooks can mint fresh realtime stream tokens before replay handling is enforced.

## Best Possible Solution

Land or revise the linked fix so replayed verified Twilio requests cannot reach any TwiML path that creates a fresh stream credential, then close this issue from the merged fix and release/backport decision.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: a replay-flagged verified Twilio request reaches `buildTwiMLPayload` before `verification.isReplay` is enforced. I did not execute a live Twilio replay in this read-only review.

## Solution Assessment

Yes, the linked PR's direction is the narrow fix shape: prevent replayed requests from entering credential-producing TwiML branches and cover both inbound and outbound realtime cases. Maintainers still need to review the security and release handling.
