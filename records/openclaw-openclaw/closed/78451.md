## Label Justifications

- `P2`: This is a valid provider/model-selection bug with limited blast radius to commitments extraction on affected MiniMax plans.
- `impact:auth-provider`: The failure is caused by provider/model routing selecting a MiniMax model variant the user's token plan does not support.
- `impact:session-state`: Commitments extraction failing prevents inferred follow-up state from being recorded for affected users.

## Summary

Keep open. Current main and the latest release still hardcode `fastMode: true` in hidden commitments extraction, which source and docs show rewrites `MiniMax-M2.7` to `MiniMax-M2.7-highspeed`; the linked closing PR is still open and conflicting, so this issue should remain open until a landable fix merges.

## What This Changes

The report asks OpenClaw commitments extraction to stop forcing MiniMax highspeed mode so non-highspeed MiniMax plans can use inferred commitments.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/78459 or an equivalent patch that removes the unconditional `true` and preserves the existing resolved fast-mode contract, with regression coverage for MiniMax default-false and configured-fast behavior.

## Reproduction Assessment

Yes. Source inspection on current main shows `defaultExtractBatch` always passes `fastMode: true`, and the MiniMax stream hook rewrites `MiniMax-M2.7` to highspeed when that extra param is true; the reporter supplied matching live HTTP 500 logs.

## Solution Assessment

Yes, the fix belongs in the commitments extraction path rather than a new commitments-only config knob. The narrowest maintainable solution is to remove the unconditional fast-mode override or pass the existing `resolveFastModeState(...).enabled` value for the resolved extractor model.
