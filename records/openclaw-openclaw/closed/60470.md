## Summary

Current main already implements the core UX requested in #60470: transient model fallback/rate-limit failures are classified, recorded as structured fallback attempts, logged through the model-fallback warning path, surfaced to WebChat as a non-red fallback status with selected/active provider-model state and attempt details, and converted to a warning-style cooldown message when all fallback candidates are exhausted.

## Best Possible Solution

Close #60470 as implemented on current main. Keep the existing structured model-fallback lifecycle, warning-colored WebChat fallback indicator, and exhausted-rate-limit cooldown copy. If maintainers want more polish, track it as a narrower follow-up for live per-second cooldown countdowns rather than keeping this broad UX issue open.
