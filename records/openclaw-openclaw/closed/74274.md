Keep open: this is a focused Control UI/Gateway bug-fix PR, and the latest head addressed Greptile's non-assistant guard concern, but it still has a blocking URL handling regression plus the external-contributor real behavior proof gate is unsatisfied.

## What This Changes

The branch renders assistant `mediaUrl`/`mediaUrls` as Control UI attachment cards, adds download links for non-image assistant attachments, and sets attachment disposition for local assistant-media downloads.

## Best Possible Solution

Land a rebased patch that keeps top-level assistant media rendering, forces downloads only for OpenClaw-owned local media routes, preserves external/provider/data URLs, and keeps current `mediaTicket` handling intact.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path for the blocking finding: current main preserves a signed URL like `https://example.com/files/report.pdf?sig=1`, while this PR appends `download=1` to every non-image attachment link.

## Solution Assessment

No. The local download/header direction is useful, but the PR should scope forced download query parameters to OpenClaw-owned media routes and rebase over the current ticketed assistant-media flow.
