## Summary

Keep open, but this PR is not merge-ready: the CLI restart notification idea is still useful, yet the patch can miss the startup sentinel window, writes through the invoking CLI environment, lacks routable session context, and has no real behavior proof.

## What This Changes

The PR modifies `src/cli/daemon-cli/lifecycle.ts` so `openclaw gateway restart` writes a best-effort restart sentinel after `runServiceRestart()` returns true.

## Best Possible Solution

Implement CLI restart notification as a daemon-env-aware pre-restart handoff with rollback and explicit routing, or document that routed post-restart notices are only supported through the Gateway tool and slash-command restart paths.

## Reproduction Assessment

Yes for the PR defects by source inspection: `runServiceRestart()` returns after post-restart checks, startup only checks for an existing sentinel during its sidecar pass, and the payload omits routable session fields. I did not run a live gateway restart in this read-only review.

## Solution Assessment

No. Moving the write after restart completion avoids the earlier no-op stale-sentinel bug, but the maintainable solution needs daemon environment ownership, earlier write/rollback timing, and explicit routing semantics.

## Review Findings

Overall confidence: 0.91

Full review comments:

- **[P2] Write the sentinel before startup can miss it:** `src/cli/daemon-cli/lifecycle.ts:284`
  - body: The new `writeRestartSentinel()` runs only after `runServiceRestart()` returns. That helper has already restarted the gateway and completed post-restart checks, while startup only schedules the sentinel wake if the file exists during its post-ready sidecar check, so direct restarts can miss this file and never send the intended notification.
  - confidence: 0.92
- **[P2] Write the sentinel using the daemon environment:** `src/cli/daemon-cli/lifecycle.ts:284`
  - body: `writeRestartSentinel()` defaults to the invoking CLI process environment. Installed gateways can run with persisted `OPENCLAW_STATE_DIR` or `OPENCLAW_CONFIG_PATH`, so this can write `restart-sentinel.json` outside the daemon's state directory and make the notification silently disappear.
  - confidence: 0.9
- **[P2] Carry a routable restart session in the sentinel:** `src/cli/daemon-cli/lifecycle.ts:271-282`
  - body: The payload only contains a generic message and stats. Without `sessionKey`, `deliveryContext`, `threadId`, or continuation data, the sentinel reader cannot reliably notify the user or active chat that initiated the restart and may fall back to the wrong session path.
  - confidence: 0.84
