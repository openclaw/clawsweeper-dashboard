## Summary

The reported root cause does not match OpenClaw’s code path. In the reporter’s cited version `2026.3.8` (`3caab9260cb0a0064e6a37b2de3bedc8a547e599`), Google audio transcription already routed through a shared Gemini inline-data helper that sets `contents[0].role = "user"`, and current `main` still does the same.

## Evidence

- **Reporter-cited version used the shared Gemini inline-data helper for audio STT:** `src/media-understanding/providers/google/audio.ts:8` calls `generateGeminiInlineDataText(...)` for `transcribeGeminiAudio`, so the outgoing `generateContent` payload comes from that shared helper.

  - command: `curl -L --silent https://raw.githubusercontent.com/openclaw/openclaw/3caab9260cb0a0064e6a37b2de3bedc8a547e599/src/media-understanding/providers/google/audio.ts | nl -ba | sed -n '1,40p'`
  - sha: [3caab9260cb0](https://github.com/openclaw/openclaw/commit/3caab9260cb0a0064e6a37b2de3bedc8a547e599)
- **Reporter-cited version already set `role: "user"` in the Gemini request body:** `src/media-understanding/providers/google/inline-data.ts:47` builds `contents: [{ role: "user", parts: [...] }]`, and `src/media-understanding/providers/shared.ts:56` sends that object via `JSON.stringify(params.body)` without stripping fields.
  - file: [src/media-understanding/providers/google/inline-data.ts:47](https://github.com/openclaw/openclaw/blob/3caab9260cb0a0064e6a37b2de3bedc8a547e599/src/media-understanding/providers/google/inline-data.ts#L47)

  - sha: [3caab9260cb0](https://github.com/openclaw/openclaw/commit/3caab9260cb0a0064e6a37b2de3bedc8a547e599)
