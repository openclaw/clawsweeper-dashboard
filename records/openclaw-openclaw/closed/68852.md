## Summary

Keep this PR open. Current main still has the reported false-positive path, and the PR is a narrow relevant fix, but the latest head still has a blocking fenced-code parser edge where a shorter internal backtick line can close a longer fence early and let code links reach lint.

## What This Changes

This PR strips fenced code blocks, HTML comments, and image markdown before memory-wiki link extraction, and adds focused `extractWikiLinks` coverage.

## Best Possible Solution

Revise the extractor so code-fence stripping tracks the opening backtick run and only closes on a same-type fence with at least that length. Keep the current outside-code/comment/image coverage and add a regression with a four-backtick fence containing a shorter triple-backtick line before markdown-like code text.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Honor the opening fence length:** `extensions/memory-wiki/src/markdown.ts:74`
  - body: The new regex can close a four-backtick block on an internal triple-backtick line once there is a preceding code line. That leaves later code such as `[fake](bad)` in `searchable`, so `wiki lint` can still report false broken-link targets from valid fenced code. Track the opener length, or use a small fence scanner, before extracting links.
  - confidence: 0.9
