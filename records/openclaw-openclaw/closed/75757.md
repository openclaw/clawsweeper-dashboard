## Summary

Current main and the shipped v2026.4.29 release already contain the Feishu streaming-card fixes for this two-card/late-final path. The dispatcher no longer closes a card from the active final-delivery branch, suppresses late final text after idle closeout, and has focused regression tests for one-card multi-final behavior; the earlier PR #42940 is useful context but has been superseded by the merged #68491/#72294 fixes.

## What This Changes

The issue reports Feishu/Lark streaming card mode creating duplicate cards for long or multi-final responses and requests updating the existing streaming card instead of closing and recreating it.

## Best Possible Solution

Keep the shipped Feishu dispatcher behavior and close this report; if a user is still seeing duplicated text inside a single card from overlapping block payloads, that narrower symptom should stay with the separate block/partial overlap track rather than reopening this two-card final-delivery issue.

## Reproduction Assessment

Yes. The former reproduction is now encoded in focused dispatcher tests: card-mode streaming finals coalesce into one streaming session, and a late final after idle close does not create a second card.

## Solution Assessment

Yes. The current fix stays inside the Feishu plugin dispatcher, preserves one active streaming card until idle closeout, and suppresses late final text without changing global streaming throttle or shared reply-dispatcher contracts.
