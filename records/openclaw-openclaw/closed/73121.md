## Summary

Close as implemented. Current main and the v2026.5.7 release now make run-backed CLI tasks depend on the live run context, so stale child-session or main-session rows no longer keep dead CLI tasks in `running`; a later matching production report was also verified and closed against the same fix.

## What This Changes

The issue asks for terminated CLI runtime tasks to be auto-transitioned out of `running` and for task cancellation lookup failures to be clearer.

## Best Possible Solution

Keep the shipped run-context-backed task maintenance contract; any desire for prefix lookup or wider full-ID table display should be handled as a separate CLI UX enhancement.

## Reproduction Assessment

No current-main failing reproduction remains. The previous source-level path with an inactive CLI `sourceId`/`runId` and a lingering session row now projects to `lost` after the grace period, and regression coverage locks that behavior.

## Solution Assessment

Yes. The shipped fix is the narrow maintainable path because it uses the existing live run-context authority and `lost` transition instead of adding PID polling, a new config knob, or broader process ownership policy.
