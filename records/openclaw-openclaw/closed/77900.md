## Summary

Keep open: current main still has Telegram transport fallback and polling restart backoff, but no host-level cooldown/circuit breaker that suppresses repeated requests to an unhealthy Telegram endpoint.

## What This Changes

The issue requests bounded Telegram Bot API transport failures during unreachable/offline windows so repeated ENETUNREACH retries do not starve the gateway event loop or trigger downstream TUI watchdog warnings.

## Best Possible Solution

Add a Telegram-owned host health cooldown around Bot API transport fallback attempts, with fixed conservative defaults first and regression tests for repeated unreachable fallback hosts and polling recovery.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection does show the current-main path: recoverable Telegram fetch errors promote to sticky IPv4/pinned-IP attempts, and no host-health state suppresses repeated unhealthy endpoint attempts.

## Solution Assessment

Yes for a narrow Telegram-local breaker/cooldown; unclear for the broader configurable shared framework proposed in the issue. The safer first fix is to bound the Telegram failure storm without introducing a new public config surface unless maintainers ask for it.
