## Summary

Current main already initializes bundled context engines before CLI compaction resolves the active engine, and focused regression coverage now locks that ordering. The fix appears current-main-only, not shipped in v2026.5.7.

## What This Changes

The issue reports that CLI turn compaction can resolve the default legacy context engine before the registry is initialized, preventing Claude CLI dreaming diary transcript persistence.

## Best Possible Solution

Keep the current main guard and regression test, then let the next release carry the CLI compaction lifecycle fix to users.

## Reproduction Assessment

Yes for the reported release path by source inspection: resolving the default engine before initialization can throw when the registry is empty. I did not run a live nightly dreaming sweep, and current main no longer has that ordering bug.

## Solution Assessment

Yes. The current implementation uses the same idempotent initialization boundary as the other context-engine resolution paths and adds focused ordering coverage without adding config or product behavior.
