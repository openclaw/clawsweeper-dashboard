## Summary

This issue should stay open: the source path supports the reported gateway/local mismatch, and current main does not appear to have a behavior fix beyond the latest release. Ollama dynamic discovery can learn `thinking` from `/api/show`, but allowlist-only gateway catalog rows can still be synthetic and off-only when no explicit provider metadata row exists.

## What This Changes

The issue reports that an Ollama Cloud model allowed only through `agents.defaults.models` loses live-discovered thinking capability metadata in the gateway catalog.

## Best Possible Solution

Keep the gateway and local Ollama paths using the same live-discovered capability metadata for an allowlisted dynamic model without requiring users to duplicate `reasoning` in `models.providers.ollama.models[]`.

## Reproduction Assessment

Yes from source, but not from a live run in this sweep. The reporter supplied concrete CLI steps, and the current source shows how allowlist-only synthetic gateway catalog rows can lack the Ollama `reasoning` capability that the local dynamic path discovers.

## Solution Assessment

Yes, the best fix is a narrow metadata propagation repair. The fix should preserve live-discovered Ollama capability metadata across the gateway catalog/allowed-model path rather than adding a new config option or hardcoding `minimax-m3`.
