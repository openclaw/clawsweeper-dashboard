## Label Justifications

## Summary

Current main and the latest release still return from the 15-second `memory_search` deadline without cancelling the underlying task. The issue remains necessary, but two open PRs already own the repair, including a clean member-authored candidate with live cancellation proof, so the right path is to review and land one rather than queue a third implementation.

## What This Changes

Propagate the `memory_search` tool deadline through the memory search and embedding stack so timed-out calls cancel provider requests, retry loops, and fallback activation.

## Best Possible Solution

Land one additive caller-signal path that aborts the active provider fetch, stops retry policy before message-pattern classification, and prevents fallback activation after cancellation while preserving the existing timed-out tool result; then close the overlapping PR.

## Reproduction Assessment

Yes. Current main deterministically leaves the raced task running, and https://github.com/openclaw/openclaw/pull/91742 provides a before-fix gateway reproduction where the tool returns at 15 seconds while the request remains open until the 60-second watchdog and then retries twice.

## Solution Assessment

Yes in direction. A tool-owned AbortSignal threaded through the existing search, retry, and provider-operation seams is the narrowest maintainable solution, but maintainers should choose between the two existing PRs rather than create another implementation.
