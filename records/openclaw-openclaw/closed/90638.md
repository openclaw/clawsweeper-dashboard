## Label Justifications

- `P1`: A documented recovery workflow is broken for Slack users when sessions wedge, with no useful in-channel acknowledgement or alternative.
- `impact:message-loss`: Whole-message recovery commands can be accepted at Slack ingress and then silently consumed without execution or reply.
- `impact:session-state`: The affected commands compact, reset, or stop session state and their failure can leave a session unusable.

## Summary

Keep open: current main and v2026.6.5 still silently discard whole-message Slack recovery commands when the accepted DM sender fails command authorization, so the reported recovery failure remains; maintainers must decide whether open-DM users gain control-command authority or receive an explicit denial.

## What This Changes

Make `/compact`, `/new`, and `/stop` execute as recovery commands in Slack DMs instead of silently disappearing or reaching the agent as prose.

## Best Possible Solution

Preserve separate inbound and privileged-command authorization, execute recovery commands for configured Slack owners, and return a clear authorization response for accepted non-owner DM senders instead of silently discarding the command.

## Reproduction Assessment

Yes at source level: an accepted Slack DM can carry a whole-message recovery command with failed command authorization, after which the shared reply runner returns no response. A live Slack workspace reproduction was not performed during this read-only review.

## Solution Assessment

No, broadly granting command access to all open-DM senders would be unsafe; the narrow maintainable fix is owner-authorized execution plus an explicit denial response for other accepted senders.
