## Summary

Keep open: the production hardening is directionally right, but the latest PR head still fails Windows CI because the unsafe-root test expects the default `C:\Windows` root while the resolver can fall back to the runner's real `WINDIR` as `C:\WINDOWS`.

## What This Changes

This PR routes Windows `.cmd`/`.bat` wrapper launches through `getWindowsInstallRoots().systemRoot` instead of `process.env.ComSpec`, adds Windows wrapper regression tests, and updates the changelog.

## Best Possible Solution

Keep the production hardening direction, but repair the new Windows test isolation so CI validates the intended resolver behavior without depending on runner root casing.

## Reproduction Assessment

Yes for the remaining PR blocker: `checks-windows-node-test` on head `04c287aae5e956f2f34531c72fe8a547e2a70d64` reports the `C:\Windows` versus `C:\WINDOWS` assertion failure at `src/process/exec.windows.test.ts:246`. The underlying hardening target is source-reproducible on current main because `src/process/exec.ts:110` still reads `process.env.ComSpec` for batch wrappers.

## Solution Assessment

No, not as submitted. The production change is the narrow maintainable fix, but the unsafe-root test should clear or stub all `WINDIR` variants, or assert against the resolver result, before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P2] Clear WINDIR before expecting the default root:** `src/process/exec.windows.test.ts:246-249`
  - body: In the unsafe-root loop, the test deletes only `process.env.windir`, but `getWindowsInstallRoots()` reads `WINDIR` case-insensitively after rejecting `SystemRoot`.
