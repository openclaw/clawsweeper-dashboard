## Label Justifications

- `P1`: The report describes a reproducible regression that blocks an automated cron workflow and misroutes generated media across isolated runs.
- `impact:message-loss`: The cron prompt can be displaced by a stale image callback, so the intended first message for the new run is not delivered.
- `impact:session-state`: A completion from a terminated run can attach to a later run of the same job, indicating session isolation/state drift.

## Summary

This looks like a valid current regression: isolated cron runs create fresh run-scoped sessions, but session-backed image generation completions are still keyed and delivered through the broader agent session path, so a late completion can plausibly land in a later run instead of being discarded with the terminated run.

## What This Changes

The issue requests that stale `image_generate` completions from terminated isolated cron runs be ignored so the next run starts with the configured cron prompt.

## Best Possible Solution

Preserve normal session-backed media completion delivery, but make run-scoped isolated cron completions no-op once their originating run is no longer active/current so later runs always start from the cron payload.

## Reproduction Assessment

Yes from source inspection, but not from a live run in this review. The current delivery path can store a non-run-scoped requester key for image generation and later fall back to direct delivery after the original isolated cron run is inactive.

## Solution Assessment

Yes, a narrow fix at the media-completion delivery boundary is the best direction. It should not add a user setting or change custom-session cron behavior; it should only prevent stale run-scoped isolated cron callbacks from starting or steering later runs.
