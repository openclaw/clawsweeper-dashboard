## Summary

Keep open: this is a focused Feishu bug fix, but the current PR is not merge-ready because the queue key is still resolved before native topic starter hydration and there is no contributor-supplied real Feishu proof.

## What This Changes

The PR threads Feishu account config into sequential-key selection, derives group queue keys from `resolveFeishuGroupSession`, adds Feishu sequential-key tests, and updates the changelog.

## Best Possible Solution

Revise Feishu queue-key selection so it matches routed peers for configured group sessions, especially hydrated native `topic_group` starters, then land after redacted real Feishu proof is added.

## Reproduction Assessment

Yes, from source inspection. Configure Feishu with `groupSessionScope: "group_topic"` or `"group_topic_sender"`; current main computes a chat-wide queue key before routing can split messages into scoped peers, and current tests show native topic starters can be hydrated only later.

## Solution Assessment

No, not as-is. Reusing `resolveFeishuGroupSession` is the right direction for populated topic ids, but this PR resolves the queue key before native topic hydration and can assign a starter and follow-up in the same topic to different lanes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep native topic starters on the hydrated lane:** `extensions/feishu/src/sequential-key.ts:34-43`
  - body: This call runs before `handleFeishuMessage` hydrates a missing `thread_id` for native `topic_group` starters. A starter without `root_id`/`thread_id` can be queued on a chat or message-derived lane while routing later uses the hydrated thread id, so its follow-up with `thread_id` may run on a different queue key despite sharing the same session. Move hydration before key selection or otherwise ensure starter and follow-up events use the same key, with `topic_group` starter coverage.
  - confidence: 0.9
