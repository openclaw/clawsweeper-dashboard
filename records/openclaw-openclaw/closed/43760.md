## Summary

Close as superseded: current main still models memorySearch fallback as a string with one shared remote config, and the remaining safe design is already tracked by the open #63990 multi-index embedding failover issue.

## What This Changes

Request separate provider, model, and remote endpoint configuration for memorySearch fallback embeddings so Gemini can remain primary while an OpenAI-compatible SiliconFlow endpoint acts as fallback.

## Best Possible Solution

Use #63990 as the canonical design path for provider/model-aware embedding failover with separate indexes, preserving separate endpoint configuration without extending the current single-index fallback in isolation.

## Reproduction Assessment

Not applicable as a feature request. Source inspection does confirm the reported current behavior: fallback is a string and the activated fallback receives the same resolved `memorySearch.remote` object.

## Solution Assessment

Yes for cleanup: closing in favor of #63990 is the better path because a narrow `fallbackRemote` addition would not address mixed vector spaces when providers or models differ.
