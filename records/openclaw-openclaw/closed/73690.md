## Summary

Current main already fixes the reported post-reply delay path by keeping Gateway-backed `openclaw agent` calls from triggering one-shot embedded MCP cleanup before the CLI receives the final response, while preserving cleanup for local/fallback embedded runs.

## Best Possible Solution

Close as implemented on main and keep the current ownership split: Gateway-backed agent calls ignore the historical one-shot cleanup flag, while local and embedded fallback CLI runs still retire local MCP/stdio resources after producing the reply.

## Evidence
