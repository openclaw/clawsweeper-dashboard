## Summary

Keep open. Current main still has the reported Telegram fail-open shape: `dmPolicy: "open"` returns before sender allowlist evaluation, account-level `allowFrom` overrides the visible top-level allowlist, and existing schema/docs/audit coverage makes public access explicit without resolving the conflicting effective-policy case. Related PR #38683 is open but targets auto-healing wildcard allowlists for cold-start validation, not this fail-closed security behavior.

## Best Possible Solution

Keep this open for Telegram-owned access-control work. Define effective-policy semantics that fail closed, or at least require explicit high-friction confirmation, when a restrictive top-level Telegram allowlist conflicts with account-level public DM settings. Add regression coverage for `mergeTelegramAccountConfig`, `enforceTelegramDmAccess`, direct event authorization, schema/setup/doctor/security/status surfaces, and coordinate with PR #38683 so wildcard auto-healing does not widen restrictive configs unintentionally.
