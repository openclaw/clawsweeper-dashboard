## Summary

Close as implemented on main: the Groq transcription path still sends `FormData` to `/audio/transcriptions`, and current main now normalizes runtime multipart requests by rebuilding the `FormData` and removing stale `content-type`/`content-length` headers; the shipped fix appears in `v2026.4.12`, while `v2026.4.5` lacked that normalization.

## Evidence

- **Current Groq/OpenAI-compatible transcription path uses multipart form upload:** `transcribeOpenAiCompatibleAudio(...)` builds a `FormData` body and posts it to `${baseUrl}/audio/transcriptions`, which is the path Groq uses for bundled audio transcription.
  - file: [src/media-understanding/openai-compatible-audio.ts:21](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/src/media-understanding/openai-compatible-audio.ts#L21)
  - command: `nl -ba src/media-understanding/openai-compatible-audio.ts | sed -n '20,80p'`
  - sha: [c997a9f9788e](https://github.com/openclaw/openclaw/commit/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a)
- **Current transport strips stale multipart headers before runtime dispatch:** `normalizeRuntimeRequestInit(...)` rebuilds non-runtime `FormData`, then deletes `content-length` and `content-type` before calling undici runtime fetch. That directly addresses Groq rejecting a stale/non-generated multipart header.
  - file: [src/infra/net/runtime-fetch.ts:49](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/src/infra/net/runtime-fetch.ts#L49)

  - sha: [c997a9f9788e](https://github.com/openclaw/openclaw/commit/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a)
- **Audio transcription requests flow through the runtime dispatcher path:** Guarded fetch uses `fetchWithRuntimeDispatcher(...)` whenever a per-request dispatcher is attached and the ambient global fetch path would not honor it, so provider audio uploads hit the normalized runtime-fetch path.
  - file: [src/infra/net/fetch-guard.ts:404](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/src/infra/net/fetch-guard.ts#L404)
