## Summary

Current main already contains the channel-presence and gateway-startup fixes for the reported upgrade regression: explicitly configured bundled channels are still treated as effective under restrictive plugin allowlists, status builds configured account details from that read-only channel surface, and QQBot has a credential backup path for the hot-upgrade config-loss variant. The proof is on main after v2026.4.29 and is not yet contained in a shipped tag.

## What This Changes

The issue requests that Feishu and QQBot account settings survive a v4.23 to v2026.4.29 upgrade, continue appearing in `openclaw status`, and continue replying to QQBot input.

## Best Possible Solution

Keep the current main behavior that treats explicit bundled channel config as authoritative for status and startup, and ship it in the next release rather than adding startup-time config migration.

## Reproduction Assessment

Yes. The high-confidence reproduction path is a configured bundled channel with a restrictive plugin allowlist, then checking read-only status presence and gateway startup inclusion; the exact CentOS/QQBot/Ollama live stack was not replayed in this read-only review.

## Solution Assessment

Yes. Preserving explicit bundled channel activation through the shared presence/startup policy is the narrowest maintainable fix, with QQBot credential backup covering the upgrade case where credentials disappear from config.
