## Summary

Keep open for maintainer review: current main and v2026.6.10 still have the web_fetch `fromCharCode` numeric-entity bug and escaped-ampersand double decode, while this PR fixes the shared decoder boundary with focused regression coverage and no blocking findings remain.

## What This Changes

The PR routes web_fetch entity decoding through the shared HTML entity helper, tightens that helper’s named and numeric entity handling, and adds regression tests for the affected extraction path.

## Best Possible Solution

Land this PR or an equivalent shared-decoder fix after normal maintainer review, because it removes the duplicated web_fetch decoder and preserves the prior entity contracts with focused tests.

## Reproduction Assessment

Yes. Current main shows the `htmlToMarkdown` path still performs `&amp;` replacement before numeric entities and uses `String.fromCharCode` for numeric entity code points; the PR body adds before/after output through that exported helper.

## Solution Assessment

Yes. Reusing the shared `decodeHtmlEntityAt` helper in one left-to-right pass is narrower and more maintainable than adding another local `fromCodePoint` decoder or only reordering the replacement chain.
