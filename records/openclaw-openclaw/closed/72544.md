## Summary

Current main already satisfies the PR's observable goal: profile/provider `alsoAllow` entries are merged into the effective global, agent, and provider policy objects before plugin tool allowlist collection, and both native agent and Gateway paths already pass those objects into `pluginToolAllowlist`. The proposed substitutions are harmless, but they duplicate behavior already present on main and in the latest checked release.

## What This Changes

The PR changes native agent and Gateway plugin tool construction to use `profilePolicyWithAlsoAllow` and `providerProfilePolicyWithAlsoAllow` when building `pluginToolAllowlist`.

## Best Possible Solution

Close this PR and keep the existing shipped effective-policy path; any remaining `tools.subagents.tools.alsoAllow` propagation gap should stay with the separate #51388 work instead of landing this duplicate profile/provider-profile substitution.

## Reproduction Assessment

Yes. The relevant reproduction can be checked statically: configure a restrictive profile plus `tools.alsoAllow` for an optional plugin tool, and current main already carries that entry through `pickSandboxToolPolicy()` into `pluginToolAllowlist` before `resolvePluginTools()` filters optional plugin tools.

## Solution Assessment

No for this PR as a separate merge. The patch is narrow and harmless, but current main already solves the profile/provider-profile path through the effective policy objects, so landing it would be redundant rather than the best cleanup.
