## Summary

Keep open: current source and the latest release already wire eligible live `memory_search` hits into recall tracking, but the reporter provides current-release evidence that their store is still populated only by session-corpus daily signals and cannot clear deep-promotion gates.

## What This Changes

The issue reports that dreaming deep promotion never writes durable memory because the short-term recall store contains only session-corpus entries with `recallCount: 0`, and asks whether live `memory_search` should feed recall signals.

## Best Possible Solution

Clarify and test the recall signal contract: eligible live `memory_search` hits should persist to short-term recall, while any broader support for `MEMORY.md`, non-dated files, or changed deep gates should go through the existing memory-core promotion semantics discussion.

## Reproduction Assessment

Partly yes from source: entries with `dailyCount: 1`, `recallCount: 0`, and one query/day cannot pass the reported deep-promotion gates. No high-confidence reproduction yet shows current main failing to record eligible dated memory hits from live `memory_search`.

## Solution Assessment

Unclear. Adding another tracking path would duplicate existing `memory_search` recall wiring; the safer path is to diagnose whether the reporter's hits are ineligible paths and then align any semantic change with the open promotion-signal issues.
