## Summary

Keep open: the latest PR head is useful and has real behavior proof, but it still introduces a merge-blocking security regression by fetching a token-derived Copilot catalog URL without the repository's guarded fetch path.

## What This Changes

The PR adds live GitHub Copilot `/models` catalog discovery, static `gpt-5.5` fallback metadata, docs, tests, and a changelog entry.

## Best Possible Solution

Keep the provider-owned live catalog design, route the Copilot `/models` request through guarded fetch with focused tests, then land this canonical PR and supersede https://github.com/openclaw/openclaw/pull/71924.

## Reproduction Assessment

Not applicable as a feature PR rather than a bug report. Source inspection does confirm the current-main baseline: the catalog hook returns an empty model list, the manifest is static, and `gpt-5.5` is absent from the static Copilot catalog.

## Solution Assessment

No, not as written: the provider boundary and fallback behavior are maintainable, but the new token-derived catalog fetch must use the existing guarded fetch pattern before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Route Copilot catalog fetches through the SSRF guard:** `extensions/github-copilot/models.ts:222-231`
  - body: The new `/models` request builds its URL from the Copilot token-derived `baseUrl` and calls `fetchImpl` directly. Existing Copilot embedding discovery uses `fetchWithSsrFGuard` for the same endpoint class; without that guard, catalog refresh can probe private or local hosts if the token-derived URL is unsafe.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff adds provider-local network discovery but bypasses the existing SSRF/private-network guarded fetch path.
