## Summary

Current main now implements bounded recovery for stalled active embedded runs: diagnostics still observes early active work, but after the extended no-progress window it escalates recovery with active abort enabled and abort-drains the embedded run. This resolves the reported indefinite `active_embedded_run action=observe_only` failure mode, with source, test, docs, and main-only provenance evidence.

## What This Changes

Stuck-session recovery should stop observing an active embedded run forever and abort/drain it after a bounded no-progress timeout.

## Best Possible Solution

Keep the current conservative main-branch policy: observe active work initially, then abort-drain embedded runs only after the documented extended no-progress window.

## Reproduction Assessment

No, not as a failing current-main bug. I did not live-reproduce the six-worker CPU incident, and current source plus regression coverage show stalled embedded runs now escalate to abort-drain instead of observing indefinitely.

## Solution Assessment

Yes. The current implementation is a maintainable fix because it preserves observe-only behavior for merely slow active work while adding a bounded escalation for no-progress embedded runs.
