## Summary

Keep open. Current `main` has fast paths for bare root help and selected non-help routed commands, but not for subcommand help. `openclaw config --help` still goes through primary-command registration, which imports the heavyweight `config-cli` module and its config/schema/secret dependencies before help can print. The repo’s startup benchmark also does not cover subcommand help, so there is no evidence this Raspberry Pi path has been fixed on `main`.

## Evidence

- **Issue discussion shows repeated ARM reproductions:** The issue body reports `openclaw config --help` at about 21.4s on Raspberry Pi 5 / Node v24.14.1, and two follow-up comments corroborate the problem, including `openclaw agent --help` at about 56.8s on 2026.4.21. That makes this more than a one-off local setup report.
- **Root help fast path is root-only:** `openclaw.mjs` only short-circuits help when argv is exactly `[node, script, --help|-h]`; `openclaw config --help` does not qualify for that optimization.
  - file: [openclaw.mjs:128](https://github.com/openclaw/openclaw/blob/938b53698ec17934a5d1a6cffb279c584f64d9f3/openclaw.mjs#L128)
  - command: `nl -ba openclaw.mjs | sed -n '108,170p'`
  - sha: [938b53698ec1](https://github.com/openclaw/openclaw/commit/938b53698ec17934a5d1a6cffb279c584f64d9f3)
- **Route-first fast paths explicitly skip help invocations:** `tryRouteCli(...)` returns early when `invocation.hasHelpOrVersion` is true, so the lightweight routed-command path does not apply to `config --help` or `agent --help`. The routed catalog and startup bench focus on commands like `status`, `gateway status`, and `config get`, not subcommand help.
  - file: [src/cli/route.ts:41](https://github.com/openclaw/openclaw/blob/938b53698ec17934a5d1a6cffb279c584f64d9f3/src/cli/route.ts#L41)
  - command: `nl -ba src/cli/route.ts | sed -n '1,120p' ; nl -ba scripts/bench-cli-startup.ts | sed -n '50,120p'`
  - sha: [938b53698ec1](https://github.com/openclaw/openclaw/commit/938b53698ec17934a5d1a6cffb279c584f64d9f3)
- **Help path still eagerly registers the primary command:** `shouldRegisterPrimaryCommandOnly(...)` returns true whenever a primary command exists, and `runCli(...)` then imports and registers that primary command before parsing help. For `config`, that means importing `./program/command-registry.js` and then `../config-cli.js` even for `openclaw config --help`.
  - file: [src/cli/run-main.ts:239](https://github.com/openclaw/openclaw/blob/938b53698ec17934a5d1a6cffb279c584f64d9f3/src/cli/run-main.ts#L239)
  - command: `nl -ba src/cli/command-registration-policy.ts | sed -n '1,32p' ; nl -ba src/cli/run-main.ts | sed -n '239,252p'`
  - sha: [938b53698ec1](https://github.com/openclaw/openclaw/commit/938b53698ec17934a5d1a6cffb279c584f64d9f3)
- **`config-cli` remains a heavyweight import surface:** `src/cli/config-cli.ts` pulls in config I/O, runtime schema, validation, zod schema, secret resolution, and runtime helpers at module top level before `registerConfigCli(...)` is even reached. That is consistent with slow help on low-power hosts because the help path pays the import cost up front.
  - file: [src/cli/config-cli.ts:1](https://github.com/openclaw/openclaw/blob/938b53698ec17934a5d1a6cffb279c584f64d9f3/src/cli/config-cli.ts#L1)
