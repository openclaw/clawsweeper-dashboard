## Summary

Current main addresses the two concrete causes named in the report: temp writes now use the shared fs-safe atomic replacement cleanup path, `node.list` has no per-call probe/wait, and Control UI node polling is scoped to the Nodes tab at a much lower interval. Broader gateway event-loop regressions remain represented by a separate open tracker, so this narrow issue can close as implemented on main.

## What This Changes

The issue asks OpenClaw to stop leaking atomic-write temp files and avoid continuous slow `node.list` polling/probing that saturated the gateway event loop.

## Best Possible Solution

Keep the current main implementation for atomic writes and scoped node polling, and route any remaining broad gateway-pressure symptoms to the open performance regression tracker.

## Solution Assessment
