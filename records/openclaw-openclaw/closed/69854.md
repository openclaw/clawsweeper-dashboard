## Summary

Current `main` already drops WhatsApp self-echo messages before they reach inbound auto-reply processing, and the same logic is present in release `v2026.4.23`, so the reported `2026.4.15` regression has been fixed and shipped.

## Evidence

- **Outbound sends are recorded for echo suppression:** `sendTrackedMessage()` records the returned WhatsApp message id with `rememberRecentOutboundMessage({ accountId, remoteJid, messageId })` before returning, which creates the exact cache entry needed to recognize a gateway-sent echo.
  - file: [extensions/whatsapp/src/inbound/monitor.ts:245](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/extensions/whatsapp/src/inbound/monitor.ts#L245)
  - command: `nl -ba extensions/whatsapp/src/inbound/monitor.ts | sed -n '245,274p'`
  - sha: [5fe333ada85a](https://github.com/openclaw/openclaw/commit/5fe333ada85afd0def9e85985ce57e36aa00108a)
- **Inbound echo is filtered before auto-reply:** `normalizeInboundMessage()` drops `fromMe` upserts whose id matches `isRecentOutboundMessage(...)`, with an inline comment stating this prevents self-chat mode from re-processing the bot's own replies as new inbound user messages.
  - file: [extensions/whatsapp/src/inbound/monitor.ts:365](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/extensions/whatsapp/src/inbound/monitor.ts#L365)
