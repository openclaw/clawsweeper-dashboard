## Label Justifications

- `P2`: This is a normal-priority performance bug fix with real dispatch latency impact but limited blast radius and no core outage.
- `merge-risk: 🚨 compatibility`: Merging the fallback as written can reuse an active registry across changed load options instead of honoring existing compatibility reload boundaries.

## Summary

Keep open: the performance problem is source-traceable and the PR has useful real-host proof, but the current patch widens registry reuse beyond the existing load-options compatibility boundary and also carries a supply-chain-sensitive dependency override on a now-conflicting branch.

## What This Changes

This PR adds a fallback in the standalone runtime plugin registry loader to reuse an active registry after a cache-key miss, adds focused tests, and bumps the root protobufjs override.

## Best Possible Solution

Keep the warm-registry fast path, but make it compatibility-aware for the known gateway-startup-rich versus dispatch-lean equivalence while preserving reload boundaries for config, trust, activation, runtime mode, workspace, plugin scope, and env-derived paths; handle the protobufjs audit bump separately or with explicit maintainer approval.

## Reproduction Assessment

Yes, source-level. Current main still has the lean dispatch load-options path and strict active lookup miss that falls through to loadOpenClawPlugins, while the PR body supplies live patched-container logs for the after-fix performance path.

## Solution Assessment

No, not as written. Reusing the warm registry is the right direction, but dropping loadOptions after the strict miss is broader than the current cache contract; the safer fix needs a compatibility helper plus stale-config/trust regression coverage.
