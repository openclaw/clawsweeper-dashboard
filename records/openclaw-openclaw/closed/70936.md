## Summary

Keep open. Current main and the latest release still omit `standardFontDataUrl`, so this PR targets a real PDF fallback bug, but the branch is not mergeable as written because it appends `path.sep` where PDF.js requires a literal trailing `/`, and it adds a stray changelog marker. The overlapping open PR #74355 is also an active implementation candidate, so maintainers should consolidate or revise rather than close this PR now.

## What This Changes

This PR updates the bundled `document-extract` PDF fallback to resolve `pdfjs-dist/standard_fonts` from the installed package root, pass it to PDF.js, add a local legacy type shim and mock-based regression test, and add a changelog entry.

## Best Possible Solution

Land one canonical plugin-owned fix that resolves the installed `pdfjs-dist/standard_fonts` directory, passes a plain filesystem path ending in literal `/` to PDF.js, preserves the type surface, includes real standard-font PDF coverage, keeps the changelog clean, and lets #51455 close through the merged PR.

## Reproduction Assessment

Yes. The underlying bug has a high-confidence path: current main calls PDF.js without `standardFontDataUrl`, and #51455/#74355 describe a standard-font PDF fallback case that warns or returns empty text. The PR-specific regression is also reproducible by evaluating the branch on Windows, where `path.sep` is `\` and PDF.js rejects it.

## Solution Assessment

No as written. Resolving from `pdfjs-dist/package.json` is the right direction, but the branch should append a literal `/`, remove the stray changelog marker, and ideally use the real-PDF integration coverage from #74355 or equivalent before merge.
