## Summary

Current main already has direct WebChat and Gateway safeguards for the reported silent-loss path: sent user messages are optimistically retained, stale or empty history refreshes preserve pending user bubbles, send failures and pending-run timeouts surface errors, and Gateway emits transcript updates for chat.send starts and failures. The implementation is present on main and is contained in the v2026.5.12-beta.1 tag, so this release-version report can close as implemented on current main rather than remain as a duplicate of the broader delivery/replay umbrella.

## What This Changes

The issue reports direct WebChat sends that sometimes produce no response, show no error, and disappear from rendered history during long-running sessions.

## Best Possible Solution

Keep the current main WebChat/Gateway behavior and ask for a fresh report with a current-main or v2026.5.12-beta.1 reproduction if direct WebChat can still lose sent messages.

## Reproduction Assessment

No, there is not a high-confidence failing reproduction on current main. The current source and focused tests instead cover the reported loss shape by preserving optimistic direct WebChat user messages across stale or empty history refreshes and surfacing send failures/timeouts.

## Solution Assessment

Yes, the current solution is the right boundary: direct WebChat keeps the local sent message visible, Gateway emits transcript updates for the durable history path, and errors are surfaced without adding a new config or product mode.
