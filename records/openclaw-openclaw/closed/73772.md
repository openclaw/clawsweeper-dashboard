## Summary

Keep open. Current main still lacks the proposed scoped `memory.*` Gateway RPCs, so the feature is not obsolete, but the PR branch is not mergeable as-is because it drops existing Gateway/protocol contracts and introduces unbounded retained state in the new memory RPC implementation; the existing Greptile and ClawSweeper comments correctly identify part of that risk.

## What This Changes

The PR branch adds scoped Gateway memory management RPC handlers, protocol schemas, generated Swift models, and stacked memory-core scoping/indexing changes from the vector-memory series.

## Best Possible Solution

Land the scoped memory management RPCs as an additive, current-main-based Gateway/protocol change after the prerequisite memory stack is resolved; preserve existing plugin/session/node/tool contracts, add bounded retention for jobs and source refs, document/changelog the new RPCs, regenerate TS/Swift artifacts from current schemas, and validate focused Gateway/protocol/memory lanes.

## Reproduction Assessment

Not applicable as a feature PR, but the blocking findings have a high-confidence static reproduction path: compare the PR diff against current main and observe that live `plugins.uiDescriptors` and `sessions.pluginPatch` contracts are removed while their current handlers/schemas still exist.

## Solution Assessment

No. The scoped memory RPC direction is plausible, but this branch is not the best mergeable form until it is rebased/regenerated from current main, made strictly additive, and given endpoint-independent retention bounds for new in-memory RPC state.
