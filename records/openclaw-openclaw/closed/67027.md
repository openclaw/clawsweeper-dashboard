## Summary

Current main already skips plugin loading for `cron` commands through the shared CLI command-path policy, so this PR's explicit `command-catalog` entry is redundant.

## Evidence

- **Cron is not opted into plugin loading on current main:** `cliCommandCatalog` only marks `agent`, `message`, `channels`, `directory`, and `agents` as `loadPlugins: "always"`; `cron` has no override, and `resolveCliCommandPathPolicy` starts from a default of `loadPlugins: "never"`. That means `cron add/list/...` already inherit no-plugin startup behavior on current main.
  - file: [src/cli/command-catalog.ts:35](https://github.com/openclaw/openclaw/blob/7875092f4ddc6f2dbe218bea8506c9ac1da87c39/src/cli/command-catalog.ts#L35)
  - command: `nl -ba src/cli/command-catalog.ts | sed -n '30,50p'; nl -ba src/cli/command-path-policy.ts | sed -n '1,30p'`
  - sha: [7875092f4ddc](https://github.com/openclaw/openclaw/commit/7875092f4ddc6f2dbe218bea8506c9ac1da87c39)
- **Current tests already lock the cron behavior:** `command-path-policy.test.ts` explicitly asserts that `resolveCliCommandPathPolicy(["cron", "list"])` returns `loadPlugins: "never"`, which addresses the PR's intended behavior via the shared policy layer instead of an explicit catalog row.
  - file: [src/cli/command-path-policy.test.ts:75](https://github.com/openclaw/openclaw/blob/7875092f4ddc6f2dbe218bea8506c9ac1da87c39/src/cli/command-path-policy.test.ts#L75)
