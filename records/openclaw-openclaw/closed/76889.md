## Summary

Keep open: current main still has Discord account-token resolution paths that throw on unresolved SecretRef objects when a source config reaches them, while docs and secret-runtime coverage say named Discord account tokens are supported SecretRef credentials.

## What This Changes

Discord named accounts should resolve SecretRef-backed tokens from the active runtime config snapshot before startup, status, allowlist, or account-resolution code reads them.

## Best Possible Solution

Add a shared Discord account-resolution path that selects the applicable active runtime config before credential reads, then cover named-account token SecretRefs in Discord startup and channels.status tests.

## Reproduction Assessment

Yes, at source level: set an active runtime snapshot whose source config contains a named Discord account token SecretRef and whose runtime config contains the resolved token, then call the Discord account resolver with the source config. Current code ignores selectApplicableRuntimeConfig and will read the unresolved SecretRef through the strict token helper.

## Solution Assessment

Unclear for the closed PR's approach, but the issue's runtime-snapshot direction is the maintainable fix. The narrow repair should route Discord account/token resolution through the active runtime snapshot rather than re-resolving env refs ad hoc inside Discord token code.
