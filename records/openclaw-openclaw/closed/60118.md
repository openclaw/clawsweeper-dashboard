## Summary

Current main already routes voice-call `responseModel` into the embedded runner, and the only remaining `LiveSessionModelSwitchError` path is now gated behind an explicit `liveModelSwitchPending` flag. The reported mismatch between plugin `responseModel` and the global default model no longer matches the current code path.

## Evidence

- **Voice-call still resolves and passes explicit responseModel:** `resolveVoiceResponseModel()` uses `voiceConfig.responseModel` when present, and `generateVoiceResponse()` forwards that resolved `provider` and `model` into `runEmbeddedPiAgent()`.
  - file: [extensions/voice-call/src/response-model.ts:12](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/extensions/voice-call/src/response-model.ts#L12)

- **Embedded runner only throws on explicit live-switch request:** In current main, `runEmbeddedPiAgent()` throws `LiveSessionModelSwitchError` only after `shouldSwitchToLiveModel()` returns a selection.
  - file: [src/agents/pi-embedded-runner/run.ts:1013](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/agents/pi-embedded-runner/run.ts#L1013)

  - file: [src/sessions/model-overrides.ts:116](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/sessions/model-overrides.ts#L116)

## Risks / Open Questions

- This was a read-only review; I did not run a live Twilio/Telnyx voice-call repro, so the decision is based on current code paths and targeted tests.
- A session that already has `liveModelSwitchPending: true` from an explicit user-triggered live model switch would still intentionally take the switch path; that is separate from the mismatch scenario reported here.
