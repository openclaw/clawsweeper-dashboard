## Summary

Current main and the latest stable tag already avoid the reported failing delivery route. The managed memory-core dreaming cron is created as a file-writing maintenance job with explicit no-delivery mode, reconciliation repairs drifted managed jobs, and cron delivery planning no longer resolves a bare `last` target for that shape.

## What This Changes

The issue asks for memory-core's auto-managed dreaming cron to avoid fatal delivery routing so DREAMS.md is written even when no active chat route exists.

## Best Possible Solution

Keep the shipped no-delivery managed dreaming cron shape and rely on reconciliation or doctor repair to normalize older managed job rows; users still on 2026.4.26 should upgrade to v2026.4.27 or repair the stale cron row.

## Reproduction Assessment

Yes. The historical failure path is source-reproducible from an isolated agentTurn job that falls back to announce/`last`, while current main no longer reproduces because memory-core writes `delivery.mode = "none"` and cron skips implicit target resolution for that mode.

## Solution Assessment

Yes. Disabling delivery for this maintenance sweep is narrower than best-effort announcement because dreaming writes filesystem artifacts directly and should not need a chat target at all; the repair path also covers stale persisted jobs.
