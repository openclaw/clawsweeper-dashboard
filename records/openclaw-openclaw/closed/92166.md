## Summary

Keep open: the PR addresses the earlier explicit `models.providers` override concern, but the new guard still misses documented `models.json` text-only rows and can promote them to image-capable; contributor proof is also only focused test output, not a real image/pdf tool run.

## What This Changes

The branch adds a media-understanding fallback that fills image input capability from the bundled static catalog when a resolved model lacks image input, plus a regression test for explicit text-only `models.providers` entries.

## Best Possible Solution

Move the capability merge to the resolver/planner or carry source metadata so catalog image input is filled only when input was omitted across both config and `models.json`, then prove a real image/pdf run.

## Reproduction Assessment

Yes, source-level. Current main can resolve a configured or registry model as text-only before static catalog metadata is applied, and the media runtime then rejects it before image execution; I did not run tests because this review is read-only.

## Solution Assessment

No. The updated PR is closer than the prior attempt, but checking only `cfg.models.providers` is still too narrow for the documented `models.json` provider path.
