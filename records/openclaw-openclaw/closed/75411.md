## Summary

Keep open. Current main still has the reported source-level gap: Copilot runtime auth stores the exchanged token in auth storage, but the embedded stream resolver only switches non-default session streams to boundary-aware transport when `resolvedApiKey` is nonempty, so an authStorage-only Copilot OpenAI Responses path can still remain `session-custom`.

## What This Changes

The issue asks OpenClaw to make GitHub Copilot OpenAI Responses HTTP 429 quota errors fail fast even when the embedded runner uses the session-custom stream path.

## Best Possible Solution

Make terminal 429 handling invariant for `openai-responses` by routing Copilot runtime-auth default PI streams through guarded OpenClaw transport, while preserving real custom session streams.

## Reproduction Assessment

Yes, at source level: current main can pass a Copilot runtime-auth credential through `authStorage` with no `resolvedApiKey`, and the resolver can then leave a non-default session stream as `session-custom`. I did not live-run an exhausted Copilot quota account.

## Solution Assessment

Yes. The narrow maintainable fix is to carry or recognize the PI default/prepared stream path so Copilot OpenAI Responses traffic reaches guarded transport; changing retry counts, idle timeout text, or user config would not close the routing gap.
