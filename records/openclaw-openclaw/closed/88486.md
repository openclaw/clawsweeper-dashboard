## Summary

This PR is still useful and should remain open for maintainer review: current main leaves interactive diagnostic lifecycle events without a session UUID, while the PR uses the existing session-store UUID contract with a guard for command-target mismatch. I found no blocking line-level defect in the diff, but maintainers should intentionally accept the plugin/exporter-visible session metadata change before merge.

## What This Changes

The PR hoists the interactive session-store lookup in `dispatchReplyFromConfig`, passes a guarded session UUID into `createDiagnosticMessageLifecycle`, and adds regression tests for normal interactive dispatch plus native command-target divergence.

## Best Possible Solution

Land the guarded lifecycle UUID population if maintainers accept the additive diagnostic metadata change, keeping the command-target mismatch guard and focused regression tests.

## Reproduction Assessment

## Solution Assessment

Yes, pending maintainer acceptance of the metadata change: reusing the existing session-store lookup and only carrying the UUID when its session key matches the lifecycle key is the narrowest maintainable fix I found.
