## Summary

Keep open: the hardening direction is useful, but the branch still relies on a reversible chmod path for non-rw copied skills, can break later skill refreshes, and lacks real Docker proof for the security and compatibility boundary.

## What This Changes

The PR adds Docker read-only overlays for workspace skill roots, hashes those overlays, chmods copied non-rw sandbox skills read-only, and adds workspace mount tests.

## Best Possible Solution

Keep the rw overlay and hash-state hardening, but replace the non-rw chmod copy path with a mount-backed or otherwise refresh-safe immutable skill-source contract before closing the linked sandbox issue.

## Reproduction Assessment

Yes, at source level: current main's non-rw skill sync removes and recopies `targetWorkspace/skills`, and the PR then chmods that tree read-only while the default Docker user is the workspace owner. I did not run a live Docker container in this read-only review.

## Solution Assessment

No; the rw read-only overlay direction is a good fit, but chmodding copied skills is reversible by the owning sandbox user and can make later sync refreshes fail. A mount-backed or explicitly scoped rw-only fix is safer.
