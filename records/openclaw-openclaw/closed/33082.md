## Summary

Keep open. Current `main` does not show a complete fix for cron-delivered TTS media. The TTS tool now emits structured `details.media` instead of inline `MEDIA:` text, but the cron final-payload path still builds from assistant text and an optional `attempt.toolMediaUrls` bridge that is referenced later yet not populated in the returned attempt state. The recent `NO_REPLY` media fix on `main` appears to cover assistant text that already contains media directives, not the structured TTS tool-result path described in #33082.

## Evidence

- **TTS tool returns structured media, not inline MEDIA text:** `createTtsTool()` stores audio delivery in `details.media` with `mediaUrl` and `audioAsVoice`, and the test asserts the tool-result content does not contain `MEDIA:`. That means the old 'assistant text contains MEDIA token' path is no longer the primary TTS delivery shape.
