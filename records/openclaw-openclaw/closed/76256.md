## Summary

Close as implemented: current main and the shipped v2026.4.20 recovery UX keep read-to-admin scope upgrades explicit, but no longer leave users with an unhandled pairing-required dead end; CLI and Control UI now show the pending upgrade, requested versus approved access, and the exact approval path.

## What This Changes

Report that after a gateway restart, admin-scoped operations such as `cron run` fail with a `scope-upgrade` pairing-required error while read operations still work.

## Best Possible Solution

Keep explicit approval for broader scopes, ask affected users to upgrade from the reported 2026.4.5 desktop app, and use the shipped devices CLI or Control UI approval flow instead of silently granting admin on loopback reconnects.

## Reproduction Assessment

No current failing reproduction remains from source inspection: the read-to-admin scope-upgrade path still exists, but current main renders actionable approval guidance and tests cover the pending-upgrade state. The report used OneClaw Desktop App 2026.4.5, which predates the shipped v2026.4.20 recovery UX.

## Solution Assessment

Yes. The implemented path is the narrow maintainable fix: preserve explicit admin-scope approval while surfacing the request ID and approval command, rather than silently widening a paired device to admin.
