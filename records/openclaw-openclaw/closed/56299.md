## Summary

Current main already implements the requested Control UI recovery path: the browser client reconnects after recoverable WebSocket closes with backoff, reuses the existing token/password or stored device token during the next connect handshake, and refreshes the active tab/chat state after hello so the selected session resumes without a manual refresh or login.

## Best Possible Solution

Close this as implemented on current main. Keep the shipped reconnect/session-resume behavior; if maintainers want a future UX polish that keeps the full dashboard visible during the short reconnect window instead of showing the login gate, that should be a narrower follow-up rather than this manual refresh/re-login request.
