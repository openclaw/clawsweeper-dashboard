## Summary

Current `main` already treats rate-limit fallback as transient instead of a sticky downgrade: it clears auto failover overrides before the next turn, retries the configured primary automatically, probes cooldowned primaries near expiry, and emits `fallback_cleared` once the preferred model is back in use.

## Close Comment

Closing from a Codex review because current `main` already implements the requested fallback auto-recovery well enough. Auto-created fallback overrides are now transient and are cleared before the next turn so OpenClaw retries the configured primary automatically (`src/auto-reply/reply/model-selection.ts:344`). The runtime also emits explicit `fallback` / `fallback_cleared` lifecycle events when it degrades and then returns to the preferred model (`src/auto-reply/reply/agent-runner.ts:1496`), with regression coverage for both the primary-retry path and fallback clearing (`src/auto-reply/reply/model-selection.test.ts:635`, `src/auto-reply/reply/agent-runner.runreplyagent.e2e.test.ts:880`). On the failover side, rate-limit handling now prefers profile rotation before model fallback and re-probes cooldowned primaries near expiry instead of leaving the session stuck on the downgraded model (`src/agents/pi-embedded-runner/run/failover-policy.test.ts:30`, `src/agents/model-fallback.ts:541`). I could not confirm a released tag containing this from the local checkout, so `fixedRelease` is left unset.
