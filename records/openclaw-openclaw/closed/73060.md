## Summary

Keep open: current main still drops explicit Mattermost `threadId` in the direct message-action send path, and this PR is a focused valid fix, but it still needs contributor-provided real Mattermost proof and a small comment cleanup before merge.

## What This Changes

The branch adds `params.threadId` as a normalized fallback between `replyToId` and legacy `replyTo` in Mattermost `handleAction` sends and adds two regression tests.

## Best Possible Solution

Land this focused Mattermost handler fix after updating the fallback comment and adding redacted real Mattermost proof, while leaving the broader media and implicit thread-context work to the canonical issue and companion PRs.

## Reproduction Assessment

Yes, at source level: a Mattermost `message` tool send with `threadId` and no `replyToId` reaches `handleAction`, which currently computes `replyToId` only from `replyToId` and legacy `replyTo`, so no Mattermost `root_id` is sent.

## Solution Assessment

Yes, with small cleanup. Adding normalized `threadId` between `replyToId` and legacy `replyTo` is the narrow maintainable fix for explicit thread targets and matches existing Mattermost outbound adapter precedence.
