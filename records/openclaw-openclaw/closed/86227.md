## Label Justifications

- `P1`: The report describes a real channel workflow outage where inbound Telegram messages failed for a configured Claude CLI agent.
- `impact:message-loss`: User messages during the outage received generic failures instead of being delivered to the configured agent turn.

## Summary

Current main already fixes the central `claude-cli` failure path: provider-owned CLI runtime aliases no longer go through missing embedded-harness selection, and startup planning includes the owning CLI backend plugin before channel sidecars. The fix is current-main-only; I could not prove it shipped in `v2026.5.22`.

## What This Changes

The issue reports that a configured `claude-cli` backend can be missing after gateway restart, causing Telegram inbound dispatches to fail with `MissingAgentHarnessError` while the gateway still reports healthy.

## Best Possible Solution

Keep the current main CLI-runtime alias passthrough and startup-owner behavior, and ask for a new report only if a build after this fix still produces `MissingAgentHarnessError` for a valid `claude-cli` deployment.

## Reproduction Assessment

No current-main failing reproduction remains: source inspection shows the valid `claude-cli` path now bypasses missing embedded-harness selection. The old behavior is supported by the reporter logs and by `v2026.5.22` source still showing the pre-fix throw path.

## Solution Assessment

Yes. Treating `claude-cli` as a provider-owned CLI runtime alias, not an embedded harness id, is the narrowest maintainable fix while preserving fail-closed behavior for real missing plugin harnesses.
