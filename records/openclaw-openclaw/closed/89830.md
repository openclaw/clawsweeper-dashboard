## Label Justifications

- `P1`: A valid inbound iMessage history response can block the iMessage receive workflow for affected users while repeatedly timing out pending RPC calls.
- `impact:message-loss`: The issue is about inbound iMessage receives being suppressed behind a single unparsed JSON-RPC response fragment.

## Summary

Keep open: current main and the latest release still frame iMessage RPC stdout with Node readline and parse each emitted line, so a valid JSON-RPC response containing raw U+2028 can still strand the pending request and block iMessage receive.

## What This Changes

The issue requests making the iMessage JSON-RPC stdout reader robust to valid `imsg rpc --json` responses containing raw U+2028/U+2029 so one message cannot block receives.

## Best Possible Solution

Replace the iMessage RPC stdout framing with an LF-only or complete-object parser that preserves JSON strings containing U+2028/U+2029, add focused regression coverage for pending id resolution, and keep the existing `imsg --json` and wrapper newline contract intact.

## Reproduction Assessment

Yes for a source-level reproduction: current `client.ts` uses Node readline and `JSON.parse` per emitted line, and a local Node proof shows a valid whole JSON payload with U+2028 is emitted as fragments that fail parse. No live macOS/imsg reproduction was run in this checkout.

## Solution Assessment

Yes: fixing the shared iMessage RPC client framing is the narrowest maintainable solution because monitor, probe, chat actions, and catchup all depend on the same request/response client. Encoder-side escaping in `imsg` would help, but OpenClaw should not depend on that for a valid JSON payload.
