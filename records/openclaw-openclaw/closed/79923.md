## Summary

Keep open: current main still has source-visible sibling-prefix route matching, but this issue's dynamic path-change expectation is not an established contract and the linked strict-route fix is still open; this needs maintainer review rather than cleanup closure.

## What This Changes

Request asks OpenClaw to cover or define behavior for active voice-call realtime sessions when `realtime.streamPath` changes, after the sibling path routing bug.

## Best Possible Solution

Land or reject the narrow strict path-boundary fix first, then explicitly document reconnect requirements or add a designed grace-period contract if maintainers want path changes to preserve active realtime sessions.

## Reproduction Assessment

Yes for the sibling-path portion by source inspection: `pathname.startsWith(pattern)` routes sibling prefixes on current main. No for the rolling-deployment path-change behavior; the report does not include a live reproduction and the repo does not define old-path token compatibility.

## Solution Assessment

Unclear: https://github.com/openclaw/openclaw/pull/79919 is the narrow fix for sibling route acceptance, but keeping old stream paths valid would require a new compatibility policy or config contract. A safer maintainable outcome may be documenting that active sessions reconnect after stream path changes unless maintainers choose to support a grace period.
