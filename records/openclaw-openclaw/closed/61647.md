## Summary

Current main and the v2026.4.24 release no longer rely only on an exact `=command:<hash>` entry for durable `allow-always` exec approvals. Gateway and node exec approval paths first persist analyzed executable/script allowlist patterns with `source: "allow-always"` and optional Windows `argPattern` metadata, then fall back to the exact command hash only when no safe broader pattern can be derived. The provided related context also shows duplicate issue #61653, filed by the same reporter with the same core report, has already been closed as implemented.

## Best Possible Solution

Close this duplicate bug as implemented and keep the shipped pattern-first allow-always behavior. Any remaining protocol/UI metadata alignment work should stay on the focused related PR #60064 rather than keeping this duplicate user-facing bug open.
