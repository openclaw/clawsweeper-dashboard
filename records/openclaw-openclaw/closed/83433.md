## Summary

Keep open: the PR addresses a real approval-runtime bug, but the current diff treats “no explicit gatewayUrl” as “local” even though current gateway resolution can still select an env/configured remote gateway. That would forward a process-local approval runtime token across the wrong trust boundary before merge.

## What This Changes

The branch forwards the operator approval runtime token through agent approval gateway calls and into the lower-level GatewayClient options, with focused regression tests.

## Best Possible Solution

Gate approval runtime token forwarding on the resolved final target: attach it only for the implicit local loopback gateway, and add negative tests for OPENCLAW_GATEWAY_URL plus gateway.mode=remote/gateway.remote.url.

## Reproduction Assessment

Yes, source-reproducible. Current main can make backend approval-scope request/wait calls without approvalRuntimeToken, while the server only grants approval-runtime visibility when that token validates; I did not run a live local/web UI repro in this read-only review.

## Solution Assessment

No, not as currently written. Forwarding the existing token is the right direction, but the patch should decide from the resolved gateway target rather than the caller’s raw gatewayUrl option.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Gate runtime token on the resolved local target:** `src/agents/tools/gateway.ts:163-166`
