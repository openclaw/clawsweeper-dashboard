## Summary

Current `main` has explicit protection for repeated unknown-tool retries and rewrites over-threshold missing-tool calls into plain assistant text, which matches the reported `Tool multi-search-engine not found` endless-loop failure.

## Evidence

- **Issue timeline already identified the failure mode:** The issue timeline cross-referenced PR #62972 (`fix(sandbox): enhance unknown tool error to prevent endless loops`), so the existing discussion already connected this report to unknown-tool retry handling.
  - command: `provided GitHub context`
- **Current loop detector blocks repeated unavailable tools:** `detectToolCallLoop` now checks the unknown-tool streak before generic repeat detection and returns a critical block message telling the model to stop retrying the missing tool.
  - file: [src/agents/tool-loop-detection.ts:440](https://github.com/openclaw/openclaw/blob/245451b6a9c529464e177afa65013adca0d63eb5/src/agents/tool-loop-detection.ts#L440)

  - sha: [245451b6a9c5](https://github.com/openclaw/openclaw/commit/245451b6a9c529464e177afa65013adca0d63eb5)
- **Regression test covers changing arguments:** There is a direct test asserting that repeated unknown-tool failures are blocked even when the tool arguments keep changing, which is the escape hatch that would otherwise allow an endless retry loop.
  - file: [src/agents/tool-loop-detection.test.ts:484](https://github.com/openclaw/openclaw/blob/245451b6a9c529464e177afa65013adca0d63eb5/src/agents/tool-loop-detection.test.ts#L484)
