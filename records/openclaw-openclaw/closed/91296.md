## Summary

The PR addresses a real supervised git-update race at the appropriate lifecycle boundary, and the earlier systemd run is useful evidence, but the current head substantially changes preflight selection, rollback, service ownership, UI polling, and restart failure handling after that proof was captured. Merge should remain blocked until maintainers accept the fail-closed availability contract and current-head real behavior proof covers the final lifecycle.

## What This Changes

The PR routes supervised git-source Gateway updates through detached managed-service handoff, adds git candidate preflight and managed-service stop/restart handling, tracks handoff completion in Control UI, and updates related tests and docs.

## Best Possible Solution

Keep the detached handoff and preflight-before-stop design, but land only after the stopped-service terminal policy is explicitly accepted and the exact current head proves successful update plus at least one post-stop failure/recovery path in a real managed service.

## Reproduction Assessment

Yes, at source level and through the earlier systemd run: current main can rebuild a supervised git checkout while the Gateway is live, and the supplied environment reproduced the corrected handoff lifecycle. The exact latest head has not been re-proven.

## Solution Assessment

Mostly yes: detached handoff with preflight before service stop is the cleanest owner-boundary fix for the live rebuild race. The unresolved question is the product contract for failures after the service has been stopped, not the central handoff architecture.
