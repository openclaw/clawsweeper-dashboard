## Summary

Current main already routes channel discovery for `status`, `status --deep`, and `channels list` through read-only/setup metadata, and Slack/Telegram both provide setup-entry channel config adapters on that path. That matches the reported symptom and removes the empty-table/no-accounts failure mode described for 2026.3.24.

## Evidence

- **Read-only channel discovery now promotes configured bundled channels:** `resolveReadOnlyChannelPluginsForConfig` collects configured channel ids and, for any configured bundled channel not already loaded, adds its setup plugin via `getBundledChannelSetupPlugin(channelId)`. That means configured Slack/Telegram channels are surfaced on read-only CLI paths instead of disappearing when full runtime plugins are not loaded.
  - file: [src/channels/plugins/read-only.ts:342](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/src/channels/plugins/read-only.ts#L342)
  - command: `nl -ba src/channels/plugins/read-only.ts | sed -n '335,430p'`
- **`channels list` consumes read-only channel metadata:** `channelsListCommand` now gets plugins from `listReadOnlyChannelPluginsForConfig(cfg)` and renders accounts from that set, so configured channels are listed without requiring full runtime channel loading.
  - file: [src/commands/channels/list.ts:115](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/src/commands/channels/list.ts#L115)
  - command: `nl -ba src/commands/channels/list.ts | sed -n '105,160p'`
