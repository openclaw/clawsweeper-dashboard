## Summary

Current main already provides the central deterministic enforcement capability through the public plugin API: a `before_tool_call` handler can rewrite arguments, block execution, or require user approval before the tool runs. The proposed skill-owned subprocess shape should not be added because executable policy belongs in a code plugin, not a declarative skill.

## What This Changes

Requests deterministic, skill-registered pre-tool hooks that can allow, block, modify, or prompt before selected tool invocations.

## Best Possible Solution

Keep the public code-plugin `before_tool_call` hook as the canonical enforcement boundary and direct policy authors to package their gate as a plugin rather than making skills execute privileged subprocesses.

Not applicable: this is a feature request rather than broken documented behavior, and current-main inspection verifies the requested enforcement outcomes through the plugin API.
