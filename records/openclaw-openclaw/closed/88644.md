## Summary

Keep open: current main intentionally rewrites legacy `openai-codex/*` refs to canonical `openai/*` with Codex runtime policy, but the fresh cron/session failover report still points at a plausible routing gap that is not fully closed by the existing doctor migration tests.

## What This Changes

The issue reports that `openclaw doctor --fix` rewrites legacy Codex model refs from `openai-codex/*` to `openai/*` and then routes Codex-only jobs through the wrong provider or fallback chain.

## Best Possible Solution

Keep canonical `openai/*` model refs, but fix any doctor/session/cron migration path where a repaired Codex ref can fall back to direct OpenAI/OpenClaw routing instead of preserving Codex runtime/auth semantics.

## Reproduction Assessment

Yes, partially from source: doctor cron payload migration rewrites `openai-codex/*` payload refs without a payload runtime-policy slot, and implicit Codex selection can fall back to OpenClaw if the Codex harness is unavailable. A live or focused harness repro is still needed for the exact failover observed in v2026.5.28.

## Solution Assessment

No, preserving `openai-codex/*` as the general fix is not the best direction because current docs make `openai/*` the canonical Codex-backed route. The maintainable fix is to ensure every repaired route keeps explicit Codex runtime/auth semantics or fails clearly when Codex is unavailable.
