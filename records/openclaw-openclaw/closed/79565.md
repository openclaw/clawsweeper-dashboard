## Summary

Keep open: the updated PR appears to fix the earlier queue-flush ordering problem and CI reports `android-test-play` passing on the current head, but it still needs real Android/Gateway behavior proof and a required changelog entry before merge.

## What This Changes

The PR delays Android `GatewaySession.currentConnection` publication until after the connect RPC succeeds, queues reconnect-window notification node events, flushes them after node reconnect, and adds a focused Android gateway regression test.

## Best Possible Solution

Merge a narrow Android runtime fix that enforces post-handshake node event sends, keeps reconnect-window notification replay bounded, records the user-facing fix in the changelog, and leaves broader notification reconciliation to a follow-up if maintainers confirm it is still needed.

Yes from source inspection: current main publishes `currentConnection` before `conn.connect()` completes, and the notification sink can call `sendNodeEvent` in that window. I did not live-run a Pixel/Gateway reconnect scenario in this read-only review, so this is source-reproducible rather than live reproduced.

## Solution Assessment

Yes for the code direction: returning ready metadata, publishing the connection, then invoking `onConnected` is the narrow fix for the earlier flush ordering race. The PR still needs a changelog entry and real behavior proof before it is merge-ready.
