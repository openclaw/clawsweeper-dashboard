## Summary

Keep this PR open. Current main still lacks Feishu recall/delete/history-list support, so the branch is not obsolete, but the proposed patch has blocking schema, SDK-shape, config, account-routing, and destructive-action review issues before it can merge.

## What This Changes

The PR adds a Feishu `feishu_message` tool with list/recall/delete actions, a `tools.message` toggle, message tests, plugin registration, and localized Feishu error-message changes.

## Best Possible Solution

Implement Feishu recall/delete/history listing through the shared channel message action surface where possible, or through a maintainer-approved Feishu tool with JSON Schema parameters, strict config/docs alignment, account-aware routing, correct Feishu API request shapes, and conservative destructive-action enablement.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Remove the unsupported delete query parameter:** `extensions/feishu/src/message.ts:85`
  - body: `client.im.message.delete` in the pinned Feishu SDK accepts `path.message_id`; this object literal also passes `params.chat_id`, which violates the SDK payload type and can fail typechecking or send an unsupported query. Drop the `params` field and keep `chat_id` only in local result metadata if needed.
  - confidence: 0.86
- **[P1] Use JSON Schema parameters for feishu_message:** `extensions/feishu/src/message.ts:152`
  - body: `parameters` is set to a Zod object, but OpenClaw's tool pipeline normalizes JSON-schema-shaped objects and does not convert Zod instances. Providers can receive an invalid function schema for `feishu_message`; register a TypeBox/JSON Schema schema or an explicit converted schema.
  - confidence: 0.9
- **[P1] Update merged Feishu tools config for message:** `extensions/feishu/src/types.ts:98`
  - body: Adding `message?: boolean` makes `Required<FeishuToolsConfig>` include `message`, but `resolveAnyEnabledFeishuToolsConfig` still builds an object without that field. Add `message` to the merged default and OR it across enabled accounts.
  - confidence: 0.91
- **[P2] Use the Feishu message-list query shape:** `extensions/feishu/src/message.ts:29-34`
  - body: The list call passes `chat_id` under `path`, while current Feishu message listing uses `params.container_id_type` and `params.container_id`. Without the query container shape, the new `list` action will not query the requested chat history correctly.
  - confidence: 0.86
- **[P2] Add the message toggle to the strict config schema:** `extensions/feishu/src/tools-config.ts:15`
  - body: The branch adds a `tools.message` type/default, but `FeishuToolsConfigSchema` is strict and still omits `message`. A user setting `channels.feishu.tools.message` would have config validation reject the key.
  - confidence: 0.89
- **[P2] Resolve message operations against the active account:** `extensions/feishu/src/message.ts:135-144`
  - body: The new tool binds enablement and the client to `accounts[0]`, so multi-account sessions can skip the tool or execute list/recall/delete against the wrong tenant. Follow the account-aware tool pattern using merged enablement and per-call account resolution.
  - confidence: 0.87
