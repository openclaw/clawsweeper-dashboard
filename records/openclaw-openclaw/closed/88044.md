## Label Justifications

- `P1`: The report describes a current channel workflow where agent replies are routed to webchat and disappear from the user's Feishu conversation.
- `impact:message-loss`: Replies are generated but delivered to the wrong channel, so the user does not receive them in Feishu.
- `impact:session-state`: The failure depends on session delivery metadata and inter-session provenance drifting away from the original external route.

## Summary

Keep open. Current main preserves stored external session route metadata in one helper, but `sessions_send` still injects inter-session turns as internal webchat, excludes `sessions_send` from the user-facing state-preservation allowlist, and the reply router intentionally refuses external routing for internal webchat turns without an explicit delivery route; that source path matches the reported lost Feishu replies.

## What This Changes

The issue reports that `sessions_send`-driven inter-session replies for a Feishu-originated session are emitted through webchat instead of the session's Feishu delivery route.

## Best Possible Solution

Preserve the established external delivery context for `sessions_send` inter-session turns that target an existing Feishu or other channel session, while keeping internal webchat-only sessions internal and adding focused regression coverage.

## Reproduction Assessment

Yes from source, but not from a live run in this read-only sweep. A focused repro can seed a Feishu `deliveryContext`, inject a `sessions_send` inter-session turn, and assert the final reply routes to Feishu rather than the internal webchat dispatcher.

## Solution Assessment

Yes, the best fix appears to be a narrow routing/session-state repair rather than a new A2A mode or config option. The repair should distinguish established external routes from genuinely internal webchat sessions.
