## Summary

Current main already addresses this class of idle hook-runner reinit by routing background snapshot/read-only plugin loads through non-activating loads, and the shipped tree includes regression coverage for those paths.

## Evidence

- **Snapshot provider loads are non-activating:** `resolveWebProviderLoadOptions` builds plugin load options with `cache: params.cache ?? false` and `activate: params.activate ?? false`, so snapshot web-provider resolution does not activate the global registry unless a caller explicitly opts in.
  - file: [src/plugins/web-provider-runtime-shared.ts:107](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/src/plugins/web-provider-runtime-shared.ts#L107)
  - command: `nl -ba src/plugins/web-provider-runtime-shared.ts | sed -n '80,130p'`
  - sha: [7ac35b4f6900](https://github.com/openclaw/openclaw/commit/7ac35b4f69009490180a35bf404638d4280a4865)
- **Read-only channel discovery is non-activating:** Read-only external channel discovery calls `loadOpenClawPlugins` with `cache: false` and `activate: false`, preventing background channel scans from reinitializing the global hook runner.
  - file: [src/channels/plugins/read-only.ts:405](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/src/channels/plugins/read-only.ts#L405)

## Close Comment

Closing this as implemented on current main after a Codex review. The idle/background snapshot paths that used to cause redundant plugin loads now use non-activating loads instead: `src/plugins/web-provider-runtime-shared.ts:107` builds snapshot loads with `activate: false`, and `src/channels/plugins/read-only.ts:405` calls `loadOpenClawPlugins` with `cache: false` and `activate: false` for read-only channel discovery. The regression is also covered in `src/plugins/loader.test.ts:3247`, which explicitly exercises the same read-only snapshot class. The shipped `v2026.4.22` tree already contains this behavior (release commit `00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). If you still see repeated `hook runner initialized...` logs on a newer build, please open a fresh issue with the exact code path and debug logs from that build.
