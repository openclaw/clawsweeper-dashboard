## Summary

Keep open: current main still treats both root and per-profile CDP URLs as raw strings, while the browser docs present credential-bearing remote CDP URLs and the SecretRef credential surface does not support or explicitly exclude those paths. The report is valid, recent, and security-sensitive, but the fix needs an owner-approved config/SecretRef contract rather than cleanup closure.

## What This Changes

Add or explicitly define SecretRef handling for `browser.profiles.*.cdpUrl` so remote CDP API keys do not have to be stored inline in `openclaw.json`.

## Best Possible Solution

Define an owner-approved CDP URL credential contract, then update browser config resolution, schema/types, SecretRef registry metadata, docs, redaction expectations, and regression coverage together.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure `browser.profiles.<name>.cdpUrl` with a SecretRef-like placeholder and `resolveProfile` will parse and return that literal URL string, while structured SecretRef objects are rejected by schema.

## Solution Assessment

Unclear. The gap is real, but ad hoc URL-substring interpolation is not obviously the safest contract; maintainers should choose between structured SecretInput support, a documented interpolation syntax, or explicitly unsupported behavior.
