## Summary

Close as duplicate/superseded. Current main still has the reported implicit daily 4:00 AM reset behavior, but the same remaining product work is already covered by open #61238, which explicitly clusters this issue for the silent daily reset/no-warning/no-opt-out problem; the narrower manual-only reset mode is tracked by open #57369.

## What This Changes

The issue asks OpenClaw to stop implicitly enabling daily session resets at 4:00 AM when `session.reset` is omitted, or to document and surface that default clearly.

## Best Possible Solution

Keep #61238 as the canonical tracker for the silent daily reset/default/discoverability UX, and use #57369 for the focused first-class manual-only reset mode. If maintainers choose unset-means-no-auto-reset, that should be implemented there with reset policy, schema, docs, and regression coverage updated together.

## Reproduction Assessment

Yes. The source-level path is deterministic: with no `session.reset` and no legacy `session.idleMinutes`, `resolveSessionResetPolicy` returns daily at 4:00 AM, and `initSessionState` rolls over the stale session on the next real message after that boundary.
