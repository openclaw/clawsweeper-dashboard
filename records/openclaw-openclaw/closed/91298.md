## Label Justifications

- `P2`: This is a bounded cron update/API bug with operator workarounds, but it can leave scheduled jobs pinned to stale model/provider choices.
- `impact:auth-provider`: The affected field controls the model/provider route used by scheduled agent jobs and can keep jobs on an unintended provider after defaults change.

## Summary

This should stay open: current main still drops `payload.model: null` during cron update normalization and only applies model patches when the value is a string, while nearby nullable clear paths already exist for fields like `toolsAllow`, `agentId`, `sessionKey`, and delivery fields.

## What This Changes

The issue asks for a supported cron update API path to clear a persisted `payload.model` override so the job can inherit the agent/default model again without deleting and recreating the job.

## Best Possible Solution

Add explicit nullable clear support for cron `payload.model` and likely `payload.fallbacks` through the protocol schema, MCP tool schema, normalization, merge logic, tests, and docs, while keeping omitted fields as merge-preserve.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: a cron job with an existing `payload.model` will keep that value because normalization deletes `model: null` and merge logic only applies string models. I did not run a live gateway repro in this read-only review.

## Solution Assessment

Yes, the narrowest maintainable fix is to align model/fallback patch clear semantics with the existing nullable clear pattern used by `toolsAllow` and delivery fields. Omitted `model` should continue to preserve the existing override so ordinary payload merges do not become destructive.
