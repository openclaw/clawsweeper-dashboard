## Summary

Close as a duplicate of the open canonical announce-delivery session conflict: current main keeps isolated cron execution on run-scoped cron sessions, while the remaining explicit Telegram sessionKey path is delivery/awareness writing to the main session, which is already tracked at https://github.com/openclaw/openclaw/issues/84583.

## What This Changes

The issue reports that isolated cron jobs configured with an explicit Telegram sessionKey reuse the main Telegram session file and repeatedly hit EmbeddedAttemptSessionTakeoverError.

## Best Possible Solution

Keep the remaining fix centralized in https://github.com/openclaw/openclaw/issues/84583 so cron announce delivery stops mutating an active user session while preserving isolated run-scoped execution.
