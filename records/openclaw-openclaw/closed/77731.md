## Summary

Keep open: the updated PR fixes the earlier auto-runtime mismatch, but it still has a blocking session-state repair bug. `doctor --fix` would repin any existing Codex-harness session to PI when Codex is not currently ready, even when that session has no legacy `openai-codex/*` route state.

## What This Changes

The PR adds doctor warnings and repairs for legacy `openai-codex/*` config/session routes, updates related docs and changelog text, and expands model-ref scanning and doctor tests.

## Best Possible Solution

Keep the config rewrite direction, but gate session-store repair on actual legacy `openai-codex/*` provider/model/auth state and preserve native Codex `agentHarnessId` pins unless owner-scoped cleanup proves they are stale.

## Reproduction Assessment

Yes, source-level: a session entry with `agentHarnessId: "codex"` and no legacy provider/model refs is reported stale when the repair runtime resolves to `pi`, then the repair writes the session harness fields to `pi`. I did not execute it because this review is read-only.

## Solution Assessment

No: the broad config/session cleanup is useful, but the current PR repairs unrelated native Codex session pins based only on current Codex readiness. The safer fix is to narrow session mutation to legacy route state and leave valid native Codex harness pins intact.
