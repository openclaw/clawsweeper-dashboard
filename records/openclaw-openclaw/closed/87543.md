## Summary

Keep open: current main still denies PreToolUse when the native hook relay is unavailable, and I found no callback-only fallback in the active CLI hook-runner path. The requested exception is plausible but security-sensitive, so it needs maintainer/security review before implementation.

## What This Changes

The issue requests a narrow fail-closed fallback that lets strictly validated trusted callback commands reach their ingress handler when the PreToolUse native hook relay is unavailable.

## Best Possible Solution

Add an owner-approved, narrowly tested fallback in the actual CLI hook runner path that permits only the canonical trusted callback ingress command and keeps every non-canonical PreToolUse request fail-closed.

## Reproduction Assessment

Yes for the denial path from source: the CLI catches relay bridge or gateway failures and renders a PreToolUse deny response, and the current test suite asserts that behavior. I did not run a live trusted-callback workflow in this read-only review.

## Solution Assessment
