## Summary

Current main already preserves configured bundled channels like Telegram during gateway startup even when `plugins.allow` only names an external plugin, which matches the reported `plugins.load.paths`/`plugins.installs` failure mode. The same guard and regression coverage are present in the latest release snapshot I could inspect.

## Evidence

- **Startup gating now bypasses restrictive allowlists for explicit bundled channel config:** `canStartConfiguredChannelPlugin(...)` treats explicitly configured bundled channels as eligible even when `plugins.allow` omits that channel plugin, so a config like `plugins.allow: ["fry"]` no longer suppresses Telegram startup.
  - file: [src/plugins/gateway-startup-plugin-ids.ts:103](https://github.com/openclaw/openclaw/blob/6b618f063516aa9ba85903aa4565c5a204c055b2/src/plugins/gateway-startup-plugin-ids.ts#L103)
- **Explicit channel config includes real per-channel settings, not only `enabled: true`:** `hasExplicitChannelConfig(...)` returns true for meaningful channel config objects unless `enabled === false`, so a normal Telegram bot config still counts as explicit channel enablement for the startup bypass.
  - file: [src/plugins/channel-presence-policy.ts:73](https://github.com/openclaw/openclaw/blob/6b618f063516aa9ba85903aa4565c5a204c055b2/src/plugins/channel-presence-policy.ts#L73)
- **Regression test keeps configured bundled channels in the startup scope under restrictive allowlists:** The startup plugin-id tests assert that a configured bundled channel remains in the gateway startup set even when `plugins.allow` only contains another plugin.
  - file: [src/plugins/channel-plugin-ids.test.ts:496](https://github.com/openclaw/openclaw/blob/6b618f063516aa9ba85903aa4565c5a204c055b2/src/plugins/channel-plugin-ids.test.ts#L496)
- **Regression test covers bundled Telegram loading with a restrictive allowlist:** Loader coverage explicitly verifies that bundled channel enablement bypasses restrictive allowlists and that Telegram still loads cleanly.
  - file: [src/plugins/loader.test.ts:1889](https://github.com/openclaw/openclaw/blob/6b618f063516aa9ba85903aa4565c5a204c055b2/src/plugins/loader.test.ts#L1889)
- **Plugin install flow matches the reported setup:** Docs state that when `plugins.allow` already exists, `openclaw plugins install` adds the installed external plugin id to that allowlist, which is the exact config shape this fix now tolerates.
  - file: [docs/tools/plugin.md:295](https://github.com/openclaw/openclaw/blob/6b618f063516aa9ba85903aa4565c5a204c055b2/docs/tools/plugin.md#L295)
- **Latest release snapshot contains the same startup allowlist bypass:** The `v2026.4.23` tree still contains the `explicitBundledChannelConfig` startup bypass logic in `src/plugins/gateway-startup-plugin-ids.ts`, so this is not only present on current `main`.
  - command: `git show v2026.4.23:src/plugins/gateway-startup-plugin-ids.ts | sed -n '88,130p'`
  - sha: [9b18ae877018](https://github.com/openclaw/openclaw/commit/9b18ae877018a34933e27bb7b2986cb97e9602aa)
