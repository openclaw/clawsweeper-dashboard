## Summary

Keep open. Current main still has the documented/schema-backed subagent thinking default and runtime consumption, but the agent-facing gateway config mutation guard still omits that leaf and will reject the reported patch before forwarding it.

## What This Changes

The issue asks for `gateway config.patch` to accept the documented `agents.defaults.subagents.thinking` default, or for the docs/schema to stop advertising that path.

## Best Possible Solution

Make a narrow guard change that allows only the documented thinking default leaves, add regression coverage proving those leaves pass while neighboring subagent policy fields remain protected, and align the exported per-agent config type if the per-agent path is intended to stay documented.

## Reproduction Assessment

Yes. A static reproduction is high-confidence: the reported merge patch changes `agents.defaults.subagents.thinking`, the current allowlist has no matching pattern, and `assertGatewayConfigMutationAllowed` throws before the gateway RPC write runs.

## Solution Assessment

Unclear pending maintainer security review. The likely maintainable fix is a narrow allowlist addition plus tests; broadening the whole `subagents` subtree would be unsafe because nearby fields change spawn authority and resource controls.
