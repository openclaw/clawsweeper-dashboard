## Summary

Keep open: current main and v2026.6.6 still classify stale active model calls as `session.stalled`, while this focused PR has sufficient classifier-level proof and should wait on maintainer acceptance of the operator-visible liveness semantics change.

## What This Changes

The PR updates session liveness diagnostics logic, tests, and docs so owned silent model calls report as `session.long_running` before `diagnostics.stuckSessionAbortMs` and as `session.stalled` at or after it.

## Best Possible Solution

Land the focused classifier, tests, and docs after maintainer signoff that pre-abort owned silent model calls should be long-running diagnostics and that any early alerting need should use a separate signal.

## Reproduction Assessment

Yes at source/proof level. Current main and v2026.6.6 classify stale active model calls as `session.stalled`, and the PR body provides a real logged session shape plus patched classifier output; I did not run a restarted gateway.

## Solution Assessment

Yes, if maintainers accept the diagnostic semantics. The patch is narrowly placed in the classifier/caller, preserves abort-threshold and ownerless-stale behavior, and documents the contract; if early stalled-style alerts must remain, a separate signal is safer than keeping the false stalled classification.
