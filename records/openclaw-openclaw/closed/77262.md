## Summary

Current main has Telegram-specific transport handling that directly targets the reported Windows/IPv6 long-poll stall mode, including HTTP/1.1-only undici dispatchers, sticky IPv4 and pinned-IP fallback, and polling transport rebuilds after stalls or recoverable getUpdates network errors.

## What This Changes

The issue asks for Telegram long-polling on Windows to reach api.telegram.org reliably when direct Node fetch succeeds but OpenClaw polling times out.

## Best Possible Solution

Keep the current main Telegram transport implementation and let the next release carry it; any remaining Windows-specific failure should be reported against that release with fresh logs.

## Reproduction Assessment

No live Windows reproduction was run. The report has concrete v2026.5.2 steps and logs, and current main contains targeted source and test coverage for the same Telegram long-poll transport failure mode.

## Solution Assessment

Yes. The current fix stays in the Telegram plugin-owned transport path, avoids a core policy change, and directly addresses the likely undici/Windows/IPv6 long-poll stall with HTTP/1.1 dispatchers and fallback rebuilds.
