## Summary

Keep open: the cache key still omits runtime env and PATH/binary eligibility inputs, so warm hydration can reuse stale `resolvedSkills`; the supplied proof is still test/check output rather than real warm gateway runtime proof on the current head.

## What This Changes

The PR adds a bounded in-memory `resolvedSkills` cache in `src/auto-reply/reply/session-updates.ts`, expands focused session snapshot tests, and adds a changelog entry.

## Best Possible Solution

Keep the cache only after reuse accounts for every skill eligibility input, or skips caching eligibility-sensitive snapshots, and require current-head redacted warm-turn gateway proof before merge.

## Reproduction Assessment

Not applicable as a user bug report; this is a performance PR. The introduced cache-key defect is source-reproducible by changing process env or PATH between warm hydrations for a skill gated by `requires.env` or `requires.bins`.

## Solution Assessment

No: the cache direction is reasonable, but the proposed key is not complete. The safer fix is to key or invalidate on env presence and local binary/PATH eligibility, or avoid caching snapshots whose eligibility depends on those runtime inputs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.85

Full review comments:

- **[P2] Key the cache by runtime eligibility inputs:** `src/auto-reply/reply/session-updates.ts:277-285`
  - body: `shouldIncludeSkill` also depends on `process.env` and local binary/PATH state through `hasEnv` and `hasBinary`. With this key, a warm gateway can reuse cached `resolvedSkills` after an API env var is added or removed, or after PATH changes, instead of rebuilding and re-evaluating eligibility. Include those runtime boundaries in the cache key, or skip caching for skills whose eligibility depends on them.
