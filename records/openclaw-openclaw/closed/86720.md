## Summary

Keep this PR open: the direction matches the reported live workaround, but the patch still misses the normal threaded final-reply case and drops the CLI result identifier needed for receipts and echo caches.

## What This Changes

This PR reroutes iMessage media-only sends to explicit chat targets through `imsg send-attachment --transport auto`, resolves numeric chat IDs to GUIDs, and adds unit coverage for chat GUID, chat ID, and DM media paths.

## Best Possible Solution

Keep the PR open and update the iMessage send path to use `send-attachment` for threaded explicit-chat media sends with `--reply-to` plus `messageGuid` receipt handling, then prove the group final-reply flow before merge.

## Reproduction Assessment

Yes, from source and PR discussion rather than my own live run: current main sends explicit-chat media through RPC `send` with `file`, while the PR discussion reports that path failing and `send-attachment --transport auto` succeeding on a live Mac.

## Solution Assessment

No; the proposed direction is plausible, but the implementation excludes threaded final replies and misses the dependency's `messageGuid` result shape. The safer solution is to cover the normal replyTo path and receipt contract directly.
