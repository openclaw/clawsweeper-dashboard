## Summary

Keep open. Current `origin/main` still lets positive out-of-range read offsets reach the upstream EOF throw path, and https://github.com/openclaw/openclaw/pull/75536 is an open closing PR that should be reviewed or replaced before this issue closes. The older related https://github.com/openclaw/openclaw/issues/52680 was closed for adapter-level containment, but this issue’s requested graceful read-level clamp is still not on main.

## What This Changes

The issue asks for the read tool to clamp or otherwise gracefully handle offsets beyond EOF instead of returning the upstream `Offset ... is beyond end of file` failure.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/75536 with a narrow wrapper-level EOF recovery that preserves sandboxing, path validation, image normalization, and existing truncation behavior.

## Reproduction Assessment

Yes. Source inspection of `origin/main` shows positive out-of-range offsets are forwarded unchanged to the pinned upstream read implementation, and the upstream 0.74.0 read tool throws when the computed start line is past EOF.

## Solution Assessment

Yes. A focused OpenClaw read-wrapper recovery is the narrowest maintainable fix because it catches only the dependency’s EOF-offset failure while leaving the dependency version, sandboxing, path validation, and image handling contracts intact.
