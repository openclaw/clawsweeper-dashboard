## Summary

Keep open. Current main still has a source-reproducible iMessage path where an anchorless group watch payload can be accepted as direct traffic, and the linked fix PR is still open, so this issue should remain the canonical tracker until that fix or a replacement lands.

## What This Changes

The issue requests iMessage monitor recovery or fail-closed handling for anchorless group link-preview watch payloads so replies cannot fall back to a sender DM.

## Best Possible Solution

Keep this issue open while reviewing or replacing https://github.com/openclaw/openclaw/pull/84503, with a final fix that repairs anchorless watch payloads before debounce/routing when GUID recovery succeeds and otherwise drops them safely.

## Reproduction Assessment

Yes. Source inspection shows current main accepts chat_id=0 with empty conversation identifiers and is_group=false, treats it as direct traffic, and builds To=imessage:<sender>; the reporter also supplied redacted live evidence and local replay proof.

## Solution Assessment

Yes for the requested direction. The maintainable fix is a pre-routing iMessage guard that recovers the real conversation via existing GUID/history RPCs when possible and otherwise drops before DM routing can occur.
