## Summary

Keep open. Current main still limits post-update work to plugin sync/convergence, while the requested user hook would add a new local-script execution surface that needs maintainer product and security design.

## What This Changes

The issue requests a user-facing post-update hook that can run local scripts after `openclaw update` and report their results.

## Best Possible Solution

Define or explicitly reject a supported update lifecycle hook with schema/types, docs, tests, and security semantics across CLI package-manager/git updates and Gateway auto-update/restart flows.

## Reproduction Assessment

Not applicable: this is a feature request rather than broken existing behavior. Source, docs, types, and live issue inspection give high confidence that current main has internal plugin post-update work but no user-facing post-update hook.

## Solution Assessment

Unclear for implementation: a config entry or directory convention is plausible, but maintainers need to decide lifecycle timing, failure handling, timeout, declaration authority, environment exposure, output redaction, and restart interactions before code is added.
