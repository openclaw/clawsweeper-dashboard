## Summary

Keep open. Current main still has the Codex app-server usage projection and generic prompt accounting shape that can double-count cached input: Codex `inputTokens` is passed through as `input`, `cachedInputTokens` is stored as `cacheRead`, and downstream session/context accounting still derives prompt/context tokens as `input + cacheRead + cacheWrite`. The current code has narrowed one related path by using `last` usage instead of cumulative-only totals, but it does not prove or enforce the cached-input normalization requested here.

## Best Possible Solution

Keep this issue open and fix it in the Codex app-server harness boundary unless upstream evidence proves `last.inputTokens` is already uncached. The most targeted path is to normalize Codex `inputTokens` to uncached input before storing `cacheRead`, then add regression coverage showing a Codex token usage snapshot where `totalTokens - outputTokens == inputTokens` and `cachedInputTokens` is a subset does not inflate `deriveSessionTotalTokens`, session store `totalTokens`, or `/status` context accounting. PR #70749-style checkpoint transcript aggregation should remain separate because it addresses duplicated transcript files, not this provider usage contract mismatch.
