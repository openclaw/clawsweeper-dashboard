## Summary

Keep open. The fallback direction is right and the PR now has real Telegram proof for the short final-send path, but the patch still has concrete message-delivery defects for long legacy preview finalization and long legacy edit fallback.

## What This Changes

The PR adds Telegram rich-method-unavailable fallbacks to legacy send/edit/preview paths, including 4096-character chunking for final legacy sends.

## Best Possible Solution

Keep the fallback in the Telegram plugin, but make preview and edit fallback paths honor the text actually accepted by legacy Telegram methods and the 4096-character contract before merge.

## Reproduction Assessment

Yes. From PR-head source inspection, a non-rich Bot API plus a >4096-character streaming preview sends only a capped legacy slice while lastDeliveredText records the full text; long rich edit fallback also calls legacy editMessageText without a limit check.

## Solution Assessment

No as-is. The fallback belongs in the Telegram plugin boundary, but the best fix must preserve only actually delivered preview text and enforce or safely split the legacy edit path before merge.
