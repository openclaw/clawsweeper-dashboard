## Summary

## What This Changes

The issue asks OpenClaw to accept an upgraded config containing `bundledDiscovery` or provide migration guidance.

## Best Possible Solution

Keep the current main behavior: support documented `plugins.bundledDiscovery`, migrate legacy allowlist configs through doctor, and direct users with a top-level key to move it under `plugins`.

## Reproduction Assessment

No high-confidence current-main failure path for the supported key: source inspection shows `plugins.bundledDiscovery` is accepted by the schema used for config validation. A top-level `bundledDiscovery` key would still be rejected, but current docs do not define that shape.

## Solution Assessment

Yes. Adding the key to the typed plugin config schema, using it in provider discovery, documenting it, and migrating legacy allowlist configs through doctor is the narrow maintainable fix; adding a root-level alias would need separate evidence that it was public contract.
