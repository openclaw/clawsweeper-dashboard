## Summary

This PR should close as superseded: the underlying per-call observability gap is now tracked by the open canonical issue and a newer additive hook PR, while this branch is stale, conflicting, lacks real behavior proof, and changes the existing public `llm_input`/`llm_output` contract in-place.

## What This Changes

The PR moves `llm_input` and `llm_output` plugin hook emission from embedded-attempt boundaries into assistant message stream handlers so plugins can observe each LLM round.

## Best Possible Solution

Keep the canonical issue and newer additive-hook PR as the design thread, and avoid merging this stale branch because it replaces existing aggregate hook semantics instead of preserving compatibility.

## Reproduction Assessment

Yes. Source inspection on current main shows per-assistant-message usage is recorded, while raw `llm_output` still dispatches once later with aggregate `getUsageTotals()` data; PR-head inspection shows the proposed per-round hook move and payload change.

## Solution Assessment

No. The PR addresses a real observability gap, but changing existing `llm_input`/`llm_output` semantics in-place is not the narrowest maintainable path when the remaining work needs an additive, documented, gated per-call contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Emit llm_input before prompt submission:** `src/agents/pi-embedded-subscribe.handlers.messages.ts:74`
  - body: Moving `llm_input` emission to `message_start` means rounds that fail before the first stream event, such as auth, rate-limit, network, timeout, or prompt precheck failures during prompt submission, no longer emit input telemetry. Current main emits before prompt execution, so tracing plugins would lose the failing round.
  - confidence: 0.9
- **[P2] Keep aggregate llm_output distinguishable:** `src/agents/pi-embedded-subscribe.ts:405-418`
  - body: The patch reuses `llm_output` for per-round payloads and removes the final aggregate event without a round index, call id, aggregate marker, or separate hook. Existing plugins cannot tell whether `assistantTexts` and `usage` describe one call or the whole turn, and aggregate consumers lose their current event.
  - confidence: 0.88
- **[P2] Normalize per-round usage before emitting llm_output:** `src/agents/pi-embedded-subscribe.ts:407-418`
  - body: The new per-round output path forwards `message.usage` through a narrow cast instead of the existing normalization path. Provider shapes using fields such as `totalTokens`, `prompt_tokens`, `completion_tokens`, cache-token fields, or timing fields can be dropped, so plugins may undercount or miss usage.
  - confidence: 0.88
- **[P2] Preserve current llm_output metadata:** `src/agents/pi-embedded-subscribe.ts:399-420`
  - body: Current main's `llm_output` contract includes context-window metadata, `resolvedRef`, `harnessId`, and prompt data. This subscription-level emitter sends only the older core fields, so rebasing this PR as-is would silently drop metadata for plugins.
  - confidence: 0.84
