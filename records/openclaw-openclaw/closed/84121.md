## Summary

The concrete Docker agent-delete residuals problem is already tracked by the open canonical report at https://github.com/openclaw/openclaw/issues/83459. This report adds useful Gateway/shared-helper context, but the source-backed CLI failure and Docker image gap overlap the existing issue, while the distinct Gateway bind-mount failure still needs concrete logs or a repro on that canonical thread.

## What This Changes

The issue asks for Docker agent deletion cleanup to use a robust shared trash path across CLI and Gateway, including `trash-cli` and fallback behavior for bind-mounted `.openclaw` directories.

## Best Possible Solution

Consolidate the fix under https://github.com/openclaw/openclaw/issues/83459 and make that fix cover both CLI and Gateway deletion without silently changing the documented Trash-not-hard-delete behavior.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partially: source inspection clearly supports the CLI Docker path because the image lacks `trash-cli` and the CLI only calls `trash`; the separate Gateway bind-mount failure is not reproduced beyond the reported scenario.

## Solution Assessment

Is this the best way to solve the issue? No: the useful fix direction should be handled in the canonical Docker deletion issue, and any hard-delete fallback needs an explicit maintainer choice because current docs promise Trash rather than `rm -rf`.
