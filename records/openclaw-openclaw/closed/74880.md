## Summary

Current main and the shipped v2026.4.27 tree already handle the reported DeepSeek thinking-mode replay failure: the DeepSeek provider wraps V4 requests, preserves or fills `reasoning_content` when thinking is enabled, strips it when thinking is disabled, and memory flush runs through that same embedded-agent/provider path.

## What This Changes

The issue reports that pre-compaction embedded memory flush fails when inherited session history contains DeepSeek thinking-mode `reasoning_content` messages.

## Best Possible Solution

Keep the provider-owned DeepSeek V4 replay handling rather than adding a memory-flush-only workaround; the shipped implementation fixes the general replay contract and the memory flush path inherits it.

## Reproduction Assessment

Yes. The issue gives a concrete DeepSeek thinking-mode session plus memory-flush reproduction path, and current regression tests directly exercise the `reasoning_content` replay payload cases that caused the 400.

## Solution Assessment

Yes. A provider-owned stream wrapper is the narrowest maintainable fix because DeepSeek's replay contract applies to all follow-up requests, while the memory flush path already uses the standard embedded-agent transport.
