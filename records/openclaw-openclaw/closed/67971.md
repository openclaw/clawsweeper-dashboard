## Summary

Current main solves the reported Discord startup failure: the provider now derives the application id from the bot token before attempting the Discord REST application lookup, so a transient non-OK `/oauth2/applications/@me` response no longer blocks startup for a normal valid bot token. The fix is present on current main but not in the latest tagged release.

## What This Changes

Discord startup should avoid hard-failing on transient non-OK application-id lookup responses by using the bot-token-derived application id while still surfacing real authentication failures.

## Best Possible Solution

Close this issue as fixed on current main and keep the token-first Discord startup behavior with its focused regression coverage; any desired direct helper status-classification change should be tracked separately as narrower API hardening.
