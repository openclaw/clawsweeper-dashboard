## Summary

Current main no longer lets LM Studio's provider normalizer rewrite unrelated provider configs during agent-scoped models.json generation. Z.AI Coding Plan base URLs remain the provider-owned /api/coding/paas/v4 endpoints, and native Ollama config preserves explicit no-/v1 baseUrl values. I could not identify a shipped release tag for this exact fix, so fixedRelease is null.

## Best Possible Solution

Close this issue as implemented on main. Keep the provider-scoped normalization path and current Z.AI/Ollama docs; any remaining generic HTTP 404 classification or failover wording should be tracked separately from the fixed baseUrl corruption.

## Evidence

- **models.json generation disables runtime plugin normalization:** The agent models.json normalization path calls normalizeProviderConfigWithPlugin with allowRuntimePluginLoad: false, so a provider with no own bundled policy surface, such as zai, falls back to the original provider config instead of loading LM Studio's hook.
  - file: [src/agents/models-config.providers.policy.runtime.ts:32](https://github.com/openclaw/openclaw/blob/7c0fdae9b95bfcc667af3ad341b63ecc194464e4/src/agents/models-config.providers.policy.runtime.ts#L32)
