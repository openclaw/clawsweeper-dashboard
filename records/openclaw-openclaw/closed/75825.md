## Summary

Keep open. The PR targets a plausible current-main session-store performance problem, but the branch should not merge as-is: its session write-back bypasses the existing locked atomic write contract, and its plugin change defers an old runtime-dependency installer outside the load/error boundary while current main has already moved plugin dependency work away from runtime loading.

## What This Changes

The PR writes normalized and maintained session-store data back to `sessions.json`, defers bundled plugin runtime-dependency installation with `setImmediate`, and carries a broad release-train diff across apps, docs, tests, config baselines, and plugin surfaces.

## Best Possible Solution

Salvage the session-store concern as a narrow current-main change that persists normalization or explicit maintenance through the existing lock and atomic-write path with stale-snapshot protection and regression tests; handle plugin startup latency only in the current install-time dependency architecture, not by backgrounding the old loader installer.

## Reproduction Assessment

Yes for the PR review findings: the changed source shows direct stale-snapshot writes from `loadSessionStore` and a backgrounded installer that can run after plugin loading continues. The production 60-120s dispatch delay is plausible from the PR body, but it was not live-reproduced here.

## Solution Assessment

No. Persisting compacted session state is a reasonable direction, but this PR is not the narrowest maintainable fix because it bypasses session-store locking and changes an obsolete, security-sensitive plugin dependency path instead of using current-main install-time dependency handling.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Use the session store lock for write-back:** `src/config/sessions/store-load.ts:138`
  - body: `loadSessionStore` is called by many readers outside `withSessionStoreLock`, so writing the parsed snapshot directly can overwrite a newer `updateSessionStore` save that lands after this read. Persisting normalization or maintenance needs to go through the locked/atomic path, or re-read/compare before replacing the file; the maintenance write-back below repeats the same issue.
  - confidence: 0.87
- **[P2] Keep runtime deps installed before plugin import:** `src/plugins/loader.ts:1511`
  - body: Deferring the installer to `setImmediate` lets `prepareBundledPluginRuntimeLoadRoot` return before missing deps are materialized, while the loader can still import the plugin module in the same turn. On a cold install, plugins that import those deps can fail before the background install runs, and install errors also escape the surrounding load error path.
  - confidence: 0.89

## Security Review

Status: needs_attention

Summary: The PR changes a package-manager runtime-dependency installation path and moves installation outside the intended lock and load-error boundary.

Concerns:
