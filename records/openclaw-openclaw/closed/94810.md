The PR changes Telegram sendChatAction 401 detection to trust structured Telegram `error_code` values before an unauthorized-text fallback and adds regression tests for false 401 suspension cases.

## Solution Assessment

Yes. This is the best narrow fix: grammY exposes Telegram `error_code` directly, so classifying structured 401s by that field and keeping only an unauthorized-text fallback avoids a new API surface or broader retry-policy change.
