## Summary

Keep open. Current `main` still sends provider-resolution snapshot loads through `loadOpenClawPlugins()` with `activate: false`, and the loader still unconditionally calls each plugin's `register()` for any non-`validateOnly` load. The current tests on `main` also codify that non-activating loads execute `register()` and only suppress global side effects, so the reported bug is not implemented away on `aa3a5f9ff7907c2a41d6259d24b529bda41f293a`.

## Evidence

- **Provider resolution still builds snapshot loads with `activate: false`:** `resolvePluginProviders()` passes `activate: params.activate ?? false` into both setup and runtime plugin-load options, so provider discovery still uses non-activating snapshot loads by default.
  - file: [src/plugins/providers.runtime.ts:167](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/src/plugins/providers.runtime.ts#L167)
  - command: `nl -ba src/plugins/providers.runtime.ts | sed -n '150,238p'`
  - sha: [aa3a5f9ff790](https://github.com/openclaw/openclaw/commit/aa3a5f9ff7907c2a41d6259d24b529bda41f293a)
- **Loader still calls `register()` during non-activating loads:** After resolving the plugin export, `loadOpenClawPlugins()` always runs `runPluginRegisterSync(register, api)` unless `validateOnly` short-circuits earlier. The `!shouldActivate` branch only restores global state after `register()` has already run.
  - file: [src/plugins/loader.ts:2705](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/src/plugins/loader.ts#L2705)

- **Tests on main still expect `register()` effects during `activate:false` loads:** The loader test for non-activating loads asserts that a plugin's `register(api)` can still add hook entries to the returned scoped registry during `activate:false`; only the global hook registry remains empty. That matches the issue's reported behavior rather than a fix.
