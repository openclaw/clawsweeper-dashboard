## Summary

Close as implemented. Current Synology Chat production webhook delivery no longer sends the channel-prefixed peer string to the webhook client for inbound replies; it routes through the inbound-turn path and sends to the numeric `chatUserId` or webhook `user_id`, with regression coverage and release-tag provenance showing the behavior shipped.

## What This Changes

The issue requests Synology Chat replies to work when `dmScope: "per-channel-peer"` records the reply peer as `synology-chat:<id>`.

## Best Possible Solution

Keep the shipped numeric inbound reply delivery path as the resolution for this issue. If maintainers want extra defense in depth, track a narrower follow-up to harden `parseNumericUserId` and add diagnostics when `sendMessage` returns `false`.

## Reproduction Assessment

Yes. The reported path maps to Synology webhook DM dispatch under `per-channel-peer`; current main can be traced from webhook registration to inbound-turn delivery, where the prefixed context value is no longer used as the `sendMessage` recipient.

## Solution Assessment

Yes. The current solution is the maintainable boundary: session/context peers can stay channel-prefixed for routing, while Synology Chat webhook delivery uses the stable numeric user ID required by the NAS API.
