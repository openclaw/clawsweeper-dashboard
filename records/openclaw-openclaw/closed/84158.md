## Label Justifications

- `P1`: A released Telegram isolated-ingress recovery path can still leave a real channel update stuck in processing state and delay user-visible Telegram handling.
- `impact:message-loss`: The affected update can remain claimed in the ingress spool instead of being delivered, retried, or failed cleanly.

## Summary

Keep open: current main still sends timeout recovery only the `blockedByLane` set, which is populated when a later pending same-lane update is blocked behind an active handler. A lone `.json.processing` handler has no pending backlog row after claim, so the reported edge case remains; the closed prior fix attempt at https://github.com/openclaw/openclaw/pull/84194 is useful context but was not merged.

## What This Changes

Recover Telegram isolated-ingress timeout handling for a single active spooled `.json.processing` update when no later same-lane update is queued.

## Best Possible Solution

Implement the narrow Telegram polling-session fix that evaluates active same-spool handler keys plus `blockedByLane`, and add a regression test for a lone timed-out spooled topic update with no later same-lane backlog.

## Reproduction Assessment

Yes, at source level: claim one spooled Telegram update, let `handleUpdate` hang, leave no later same-lane pending update, and advance past `spooledUpdateHandlerTimeoutMs`; current main then passes an empty `blockedByLane` set to timeout recovery. I did not execute a test because this review is read-only.

## Solution Assessment

Yes. The suggested union of active same-spool handler keys with `drain.blockedByLane` is the narrowest maintainable fix because it reuses the existing tombstone, reply-fence abort, stalled-lane, and isolated-ingress restart behavior.
