## Summary

Current main already includes the channel-registry pinning fix that prevents configured channels from disappearing from the send path after later plugin-registry churn. That matches this report’s `unsupported channel: whatsapp` symptom for a still-running channel, and the same implementation is present in release `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`).

## Evidence

- **Send path fails exactly when channel normalization loses the registered channel:** `send` returns `INVALID_REQUEST: unsupported channel: <id>` if `normalizeChannelId()` cannot resolve the requested channel from the registered channel registry.
  - file: [src/gateway/server-methods/send.ts:101](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/src/gateway/server-methods/send.ts#L101)
  - command: `nl -ba src/gateway/server-methods/send.ts | sed -n '101,143p;423,445p'`
  - sha: [34896839ba22](https://github.com/openclaw/openclaw/commit/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534)
- **Channel normalization depends on the active/pinned registered channel registry:** `normalizeAnyChannelId()` resolves only from registered channel plugin entries, so registry churn can surface as `unsupported channel` even when runtime status still exists elsewhere.
  - file: [src/channels/registry.ts:72](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/src/channels/registry.ts#L72)

- **Current main pins the startup channel registry to prevent later churn from evicting channels:** Gateway startup now calls `pinActivePluginChannelRegistry`, and the runtime comment explicitly says this prevents config-schema and other non-primary registry loads from evicting channel plugins.
  - file: [src/gateway/server-plugin-bootstrap.ts:98](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/src/gateway/server-plugin-bootstrap.ts#L98)

  - sha: [34896839ba22](https://github.com/openclaw/openclaw/commit/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534)
- **Runtime state prefers the pinned channel registry:** `getActivePluginChannelRegistryFromState()` returns the pinned channel registry when it has channels, instead of letting later active-registry swaps replace it.
  - file: [src/plugins/runtime-channel-state.ts:20](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/src/plugins/runtime-channel-state.ts#L20)

## Risks / Open Questions

- I did not reproduce this live against a real WhatsApp setup in this checkout; the close decision is based on code-path inspection plus targeted regression coverage.
- The report ties the symptom to a health-monitor restart window; the evidence strongly supports a now-fixed registry-desync class, but does not prove the restart itself was the only trigger.
