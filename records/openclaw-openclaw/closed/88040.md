## Summary

Keep open: current main still has a source-visible gap where the embedded runner can return a rotated session id/file, but the post-run session-store update is driven by the original session id. The latest release source shows the same call shape, so this is not already fixed or superseded.

## What This Changes

The issue requests that automatic transcript rotation after compaction update sessions.json with the runtime’s rotated session id/session file instead of the pre-run session id.

## Best Possible Solution

Adopt a single post-run active session identity from result.meta.agentMeta.sessionId/sessionFile for automatic compaction rotation, persist both fields into sessions.json, and add focused regression coverage for that handoff.

## Reproduction Assessment

Yes from source inspection: current main exposes the rotated session id/file in run metadata, but the post-run store update still writes the original session id. I did not run a live compaction repro because this review was read-only.

## Solution Assessment

No for the exact one-line proposal: sessionId adoption is necessary, but sessionFile also needs to be handled for entries that persist explicit transcript paths. The maintainable fix is to carry a closed active session identity through the post-run update boundary.
