## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Restrict the outbound bridge to subagent runs:** `src/commands/agent.ts:738`
  - body: The bridge is created for ordinary `--deliver` runs as well as subagent runs, but those runs still go through the normal final-delivery path. On Feishu this can stream a card and then send the same final answer again, so the bridge needs to be scoped to the subagent progress path or integrated with final-delivery suppression.
  - confidence: 0.92
- **[P1] Preserve the Feishu thread target:** `extensions/feishu/src/channel.ts:499`
  - body: The outbound dispatcher reduces `threadId` to a boolean and never passes the actual thread/root id into `createFeishuReplyDispatcher`. In topic-only contexts without `replyToId`, the streaming card has no root target and can be created at the top level instead of the originating topic.
  - confidence: 0.9
- **[P1] Keep block streaming opt-in:** `extensions/feishu/src/reply-dispatcher.ts:433-434`
  - body: `account.config?.blockStreaming !== false` makes block streaming enabled whenever the setting is unset, and `disableBlockStreaming: false` lets block chunks flow by default. Existing Feishu accounts should not start receiving partial block chunks unless they explicitly opt in.
  - confidence: 0.9
- **[P2] Throttle replacement card updates:** `extensions/feishu/src/streaming-card.ts:361-374`
  - body: `replaceContent()` bypasses the normal streaming update throttle, while the model-name animation queues replacements every 30 ms. That can burst Feishu CardKit update calls and trigger rate-limit or sequence-ordering failures during normal streaming.
  - confidence: 0.84
- **[P1] Avoid adding an unapproved channel dispatch API:** `src/channels/plugins/types.plugin.ts:85-91`
  - body: `createOutboundDispatcher` adds a new plugin API surface with Feishu-shaped streaming callbacks but no approved SDK contract, docs, cross-channel contract tests, or compatibility story. This should either use the existing message lifecycle/delivery seams or land with explicit maintainer approval and contract coverage.
  - confidence: 0.78
