## Label Justifications

- `P3`: The original issue affected missing usage metadata in an opted-in Telegram forum-topic reply mode, while ordinary message delivery continued to work.
- `impact:message-loss`: The missing footer was a suppressed portion of the channel-visible reply metadata in Telegram/message-tool-only delivery.

## Summary

Close as implemented. Current main now routes implicit Telegram/non-webchat message-tool-only current-source replies through the final source-reply payload path, so `/usage` decoration is preserved without scraping suppressed private final text.

## What This Changes

The issue asks for configured usage footers to appear on Telegram forum-topic visible replies sent through the message tool, or for the docs/config contract to clarify that they do not.

## Best Possible Solution

Keep the current metadata-preserving internal source-reply implementation and treat explicit routed message-tool sends as a separate behavior if a concrete new report shows they still need footer decoration.

## Reproduction Assessment

No current failing reproduction remains on main. The old source path is documented by https://github.com/openclaw/openclaw/pull/87425 and commit 8f941ea0acb7, and current source plus tests show the footer-bearing source-reply metadata now survives.

## Solution Assessment

Yes. The landed fix is the safer solution because it preserves trusted source-reply metadata through usage decoration instead of scraping suppressed private final text for a `Usage:` line.
