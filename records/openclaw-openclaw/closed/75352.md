## Summary

Keep open. Current main still launches bundled LSP stdio servers through a raw `spawn()` call, while the linked closing PR #75343 is open and unmerged, so this issue should stay open until that fix or a narrow equivalent lands.

## What This Changes

The issue reports that Windows LSP server startup fails because `spawnLspServerProcess()` launches npm CLI commands without resolving `.cmd` shims.

## Best Possible Solution

Review and land PR #75343 or a narrow equivalent that reuses the shared Windows spawn resolver in the LSP spawn path with regression coverage and a changelog entry.

## Reproduction Assessment

Yes. The issue gives a concrete Windows 11, Node 22.14.0, and `typescript-language-server` setup, and current main statically routes bundle LSP server commands into the raw `spawn()` path that matches the reported `.cmd` shim failure.

## Solution Assessment

Yes as a direction. Reusing `resolveWindowsSpawnProgram()`/`materializeWindowsSpawnProgram()` is the narrow maintainable fix; this issue should remain open until PR #75343 or an equivalent patch merges.
