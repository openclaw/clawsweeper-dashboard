## Summary

Keep open: the patch does not fix the runtime payload reported by the linked issue because the OpenRouter DeepSeek V4 wrapper mutates the payload after `buildOpenAICompletionsParams` and re-adds `reasoning_content: ""`.

## What This Changes

The branch changes the OpenAI completions reasoning replay sanitizer to delete empty `reasoning` and `reasoning_content` strings and adds a unit test for OpenRouter assistant replay.

## Best Possible Solution

Fix the OpenRouter DeepSeek V4 payload wrapper, or add a shared wrapper option, so OpenRouter does not send empty `reasoning_content` on assistant tool-call replay while preserving providers that require valid reasoning replay.

## Reproduction Assessment

Yes at source level: current main builds params, then the OpenRouter DeepSeek V4 wrapper re-adds `reasoning_content: ""` through `streamWithPayloadPatch`, and the existing OpenRouter test expects that field on an assistant `tool_calls` message. I did not perform a live OpenRouter request.

## Solution Assessment

No. The PR strips empty fields before the later OpenRouter wrapper runs, so the narrow maintainable fix needs to cover `ensureDeepSeekV4AssistantReasoningContent` or the OpenRouter wrapper path directly.
