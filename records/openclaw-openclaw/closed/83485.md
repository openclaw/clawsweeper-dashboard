## Label Justifications

- `P2`: The report describes a confusing but bounded CLI auth-status bug for Codex-backed OpenAI agent routes.
- `impact:auth-provider`: The affected surface is provider/runtime auth reporting for OpenAI and OpenAI Codex model routes.

## Summary

Current main already resolves the misleading missing-auth signal: `models status` now probes Codex synthetic auth for default OpenAI agent routes and the regression test asserts `openai/gpt-5.5` with no direct profiles/env no longer reports `missingProvidersInUse: ["openai"]`. This is current-main-only and not in `v2026.5.12` yet.

## What This Changes

The issue requests that `models status` distinguish direct OpenAI auth from Codex app-server runtime auth for `openai/gpt-*` agent models.

## Best Possible Solution

Keep the current main behavior that treats Codex synthetic auth as effective for default OpenAI agent routes, and ship it in the next release.

## Reproduction Assessment

Yes by source inspection: `v2026.5.12` lacks the current Codex synthetic-auth probe for canonical OpenAI routes, which matches the reported `missingProvidersInUse` output. Current main has focused regression coverage for the fixed status shape.

## Solution Assessment

Yes. The current fix is the narrow maintainable path because it reuses the plugin synthetic-auth contract instead of adding a separate Codex-home parser or a second auth source of truth.
