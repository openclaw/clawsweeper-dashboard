## Summary

Current `main` now resolves agent-session plugin tools through the gateway-owned active registry, so the `voice_call` tool reuses the gateway-started singleton runtime instead of constructing a second runtime that would re-bind `127.0.0.1:3334`.

## Evidence

- **Gateway startup loads plugins in gateway-bindable mode:** Gateway plugin startup passes `runtimeOptions.allowGatewaySubagentBinding: true` and marks the active registry as `gateway-bindable`, which is the prerequisite for sharing gateway-owned plugin state across tool calls.
  - file: [src/gateway/server-plugins.ts:493](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/src/gateway/server-plugins.ts#L493)
  - command: `nl -ba src/gateway/server-plugins.ts | sed -n '485,506p'`
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)
- **Agent-session runs opt into gateway binding:** Embedded agent execution now sets `allowGatewaySubagentBinding: true`, so tools resolved for normal agent sessions are asked to bind against the gateway-owned plugin runtime instead of a fresh standalone load.
  - file: [src/auto-reply/reply/agent-runner-execution.ts:1076](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/src/auto-reply/reply/agent-runner-execution.ts#L1076)
  - command: `nl -ba src/auto-reply/reply/agent-runner-execution.ts | sed -n '1070,1085p'`
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)
- **Tool resolution reuses the active gateway registry:** Plugin tool resolution explicitly returns the active registry when `allowGatewaySubagentBinding` is enabled, and the loader also treats a matching gateway-bindable cache key as compatible with default-mode resolution.
  - file: [src/plugins/tools.ts:61](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/src/plugins/tools.ts#L61)
  - command: `nl -ba src/plugins/tools.ts | sed -n '60,104p'`
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)
- **Loader compatibility keeps default tool loads on the same registry:** `getCompatibleActivePluginRegistry()` falls back to the active gateway-bindable registry when the cache key matches, preventing a second plugin module instance from being loaded for the same snapshot.
  - file: [src/plugins/loader.ts:1018](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/src/plugins/loader.ts#L1018)
  - command: `nl -ba src/plugins/loader.ts | sed -n '1018,1045p'`
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)
- **Voice-call tool and service share the same closure-held runtime when the registry is reused:** `voice_call` still uses the closure-local `runtime`/`runtimePromise` via `ensureRuntime()`, and `registerTool()` stores object tools as singleton objects. With active-registry reuse, the tool and service share that same closure instead of creating a new runtime that would hit `EADDRINUSE`.
  - file: [extensions/voice-call/index.ts:171](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/extensions/voice-call/index.ts#L171)
  - command: `nl -ba extensions/voice-call/index.ts | sed -n '160,190p'; nl -ba extensions/voice-call/index.ts | sed -n '420,570p'; nl -ba src/plugins/registry.ts | sed -n '332,345p'`
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)
- **Regression tests cover the gateway-bindable registry reuse path:** Current tests assert that gateway-bindable tool loads reuse the active registry and that the loader accepts the gateway-bindable active registry as compatible when cache inputs match.
  - file: [src/plugins/tools.optional.test.ts:356](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/src/plugins/tools.optional.test.ts#L356)
  - command: `nl -ba src/plugins/tools.optional.test.ts | sed -n '356,389p'; nl -ba src/plugins/loader.runtime-registry.test.ts | sed -n '27,70p'`

## Risks / Open Questions

- There is no voice-call-specific end-to-end test in the current tree that exercises `voice_call` against a live running gateway and Twilio; this close relies on the architectural fix plus registry reuse tests.
- The fix is present on `main` after the latest tagged release `v2026.4.23`, so users on that release may still reproduce the bug until the next release ships.
