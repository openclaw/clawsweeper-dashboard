## Summary

Current main and the latest release already use a reload-safe hook runner shape: hook availability is read directly from the active registry, and gateway plugin reloads replace the active registry and global runner instead of relying on a stale hook-availability cache.

## What This Changes

The issue reports that after plugin hot-reload, `message_sending` and related outbound hooks can stop firing because `hasHooks("message_sending")` allegedly returns a stale cached false value.

## Best Possible Solution

Close this as fixed and point affected 2026.4.x users to upgrade to `v2026.5.28` or newer; reopen with current-version logs only if hooks still stop after reload on the shipped path.

## Reproduction Assessment

No for current main: I did not reproduce the 2026.4.22 runtime, and source inspection of current main plus `v2026.5.28` shows no stale hook-availability cache on the reported path.

## Solution Assessment

Yes: the current implementation is the maintainable fix shape because plugin reload replaces the active registry/global runner, and `hasHooks` is a live registry query rather than a separately invalidated cache.
