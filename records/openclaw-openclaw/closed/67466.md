## Label Justifications

- `P1`: The PR touches a user-facing Feishu gateway startup path and its current implementation can break the channel lifecycle with restart loops and lost monitoring.
- `merge-risk: 🚨 compatibility`: The 3000ms hard cap changes the current configurable startup timeout behavior for existing Feishu installations.

## Summary

Close: current main and v2026.6.6 already address the central gateway-startup blocker by handing off long-lived channel account tasks instead of awaiting them, while the remaining Feishu timeout/status work is tracked elsewhere and this stale branch is dirty, unproven, and introduces a Feishu lifecycle regression.

## What This Changes

The PR changes Feishu monitor startup and setup-status probes to use 3000ms caps, parallelizes multi-account preflight, and detaches monitor startup into background tasks with adjusted tests.

## Best Possible Solution

Keep the shipped gateway handoff behavior, close this stale branch, and use the canonical open Feishu timeout reports for any remaining plugin-specific startup/status work.

## Reproduction Assessment

No high-confidence after-fix latency reproduction is present because the PR has no real Feishu proof.

## Solution Assessment

No. Bounding slow probes is a plausible mitigation, but this branch is not the best solution because current main already avoids waiting on long-lived channel tasks and any remaining Feishu timeout fix must preserve monitor lifetime, channelRuntime, and the startup timeout contract.
