## Label Justifications

- `P1`: The reported bug disables auth-profile rotation after rate-limit failures for users relying on multiple profiles.
- `impact:auth-provider`: The issue concerns auth-profile cooldowns and provider/profile routing after rate-limit failures.
- `impact:session-state`: The reported failure corrupts persisted profile usage state by clearing cooldown and failure-count metadata.

## Summary

Close as implemented: current main and the v2026.5.18 tag route classified assistant surface errors away from the normal embedded-run completion path, so the success marker no longer clears the just-recorded cooldown for the reported rate-limit failure path.

## What This Changes

The issue asks OpenClaw to stop marking an auth profile successful after an embedded assistant run ends with `stopReason: "error"`, preserving cooldown state for profile rotation.

## Best Possible Solution

Keep the shipped v2026.5.18 failover behavior: classified assistant errors should exit through failure handling, while the success marker remains reserved for genuine successful completions.

## Reproduction Assessment

Yes, source-level for the reported 2026.4.5 path: a classified assistant surface error could continue to the done branch after failure marking. Current main no longer reproduces that path because the surface-error branch returns a FailoverError before success marking.

## Solution Assessment

Yes. The shipped fix is the narrow maintainable boundary: failover-classified assistant errors exit through failure handling, so success marking does not need to double as an error classifier.
