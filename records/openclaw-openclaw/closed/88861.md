## Summary

Keep this PR open: the bug and direction are valid, but the patch duplicates existing session-delivery normalization and can still drop account/thread context for existing sessions.

## What This Changes

The PR hydrates OpenResponses callback turns from the addressed session entry and passes that context into ingress agent runs so the message tool can infer the current chat target.

## Best Possible Solution

Keep the OpenResponses runContext handoff, but build it from the canonical session delivery context helper and cover both deliveryContext-backed and legacy/route-backed session entries before merge.

## Reproduction Assessment

Yes. Current source shows OpenResponses callbacks resolve the session key but do not load session delivery context before invoking the agent, and the PR body supplies an after-fix local gateway e2e proof for the missing-target path.

## Solution Assessment

No. Passing runContext is the right fix boundary, but manually rebuilding session delivery context is not the best implementation when deliveryContextFromSession already owns that merge contract.
