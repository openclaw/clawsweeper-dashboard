## Summary

Keep open: current main still parses non-OK proxy error bodies with `Response.json()`, while this PR narrowly switches that branch to the existing bounded provider JSON reader, adds focused regression coverage, and now includes sufficient real local proxy proof. I found no blocking correctness or security findings; maintainers should sequence it with the broader open proxy stream-hardening PRs rather than close it as obsolete.

## What This Changes

The PR changes `streamProxy` to parse non-OK proxy error JSON through the shared bounded provider JSON reader and adds a regression test for oversized streamed error bodies.

## Best Possible Solution

Land this narrow bounded error-body fix, or explicitly sequence it with the broader proxy stream-hardening branch if maintainers prefer one combined proxy response guard.

## Reproduction Assessment

Yes. Current main reaches `Response.json()` for non-OK proxy responses before fallback, and the PR adds both source-level regression coverage and copied real local proxy output for the oversized-body path; this read-only review did not rerun tests locally.

## Solution Assessment

Yes. Reusing `readProviderJsonResponse` is the narrow maintainable fix for this branch because it moves the proxy error path onto the existing bounded response reader instead of adding a second proxy-specific body reader.
