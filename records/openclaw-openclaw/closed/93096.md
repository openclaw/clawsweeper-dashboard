## Summary

Keep open: this PR targets a real `openclaw status` performance bug that is still present on current main and v2026.6.6, but the submitted module-level TTL cache is broader than the status call that needs sharing and can return stale task inspection data after nearby task mutations.

## What This Changes

The PR adds a module-level 5 ms cache around `reconcileInspectableTasks()` and clears it from the task-maintenance test reset helper.

## Best Possible Solution

Share a single reconciled task snapshot inside `getStatusSummary()` and pass it to registry summary/audit helpers while preserving fresh zero-argument helper behavior elsewhere; any broader cache should be lifecycle-invalidated and tested.

## Reproduction Assessment

Yes by source inspection: current `getStatusSummary()` calls two task helpers that each call `reconcileInspectableTasks()`, and that reaches `listTaskRecords()` clone-and-sort work. I did not run a live 10K-task benchmark in this read-only review.

## Solution Assessment

No: the performance issue is real, but a module-level wall-clock cache is broader than the status-only duplicate work and lacks mutation/config invalidation. The narrower maintainable fix is to share one reconciled snapshot inside the status summary path or make a broader cache lifecycle-owned with tests.

- **[P2] Keep the cache tied to a fresh task snapshot:** `src/tasks/task-registry.maintenance.ts:885-886`
  - body: Returning the cached array here makes `reconcileInspectableTasks()` fresh only by wall-clock time. This helper is also used for task audit, maintenance, and active restart blockers, while the task registry emits `upserted`/`deleted` mutation events and `configureTaskRegistryMaintenance()` can change reconciliation inputs. Keep the optimization scoped to `getStatusSummary()` by sharing one snapshot there, or add lifecycle invalidation plus tests.
