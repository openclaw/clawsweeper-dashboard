## Summary

Keep open. Current main already documents and tests the LM Studio JIT opt-out as `models.providers.lmstudio.params.preload: false`, but this PR adds a parallel `sendLoadApi` config key and does not align the embedding preload path with the existing public contract.

## What This Changes

This PR adds a top-level `models.providers.*.sendLoadApi` boolean and uses it to skip LM Studio model-load warmups before inference and embeddings.

## Best Possible Solution

Use the already documented `models.providers.lmstudio.params.preload: false` contract everywhere LM Studio owns preload behavior, including memory embeddings, and avoid adding a second top-level option.

## Reproduction Assessment

Yes. Current main has a focused inference test for `params.preload: false`, and the remaining embedding case is reproducible by creating an LM Studio memory embedding provider with that documented flag and observing the unconditional `ensureLmstudioModelLoaded` call.

## Solution Assessment

No. The provider-owned opt-out direction is right, but the best current fix is to reuse `params.preload` from main and add embedding coverage instead of introducing `sendLoadApi`.
