## Summary

Keep open. Current `main` still uses the older line-by-line Windows task parser, so the `setlocal` / `endlocal` / chained-batch cases described in #53474 are not implemented on `main`, and the PR remains materially relevant.

## Evidence

- **Current parser still stops at the first non-`@echo`/`rem`/`set`/`cd /d` line:** `readScheduledTaskCommand()` on `main` only skips `@echo`, `rem `, parseable `set ` assignments, and `cd /d`. Anything else becomes `commandLine` immediately, so `setlocal`, `endlocal`, or same-line chained prologue statements would still be treated as the command instead of being skipped/unwrapped.
  - file: [src/daemon/schtasks.ts:120](https://github.com/openclaw/openclaw/blob/fe930b987e33f4e1acd93b5ef1ea76c8b7260047/src/daemon/schtasks.ts#L120)

  - sha: [fe930b987e33](https://github.com/openclaw/openclaw/commit/fe930b987e33f4e1acd93b5ef1ea76c8b7260047)
- **That parse result feeds runtime audit and port resolution:** `resolveScheduledTaskPort()` consumes `readScheduledTaskCommand()` output for `programArguments` and `OPENCLAW_GATEWAY_PORT`, so a misparsed `gateway.cmd` still affects scheduled-task status/doctor/stop behavior on current `main`.
  - file: [src/daemon/schtasks.ts:384](https://github.com/openclaw/openclaw/blob/fe930b987e33f4e1acd93b5ef1ea76c8b7260047/src/daemon/schtasks.ts#L384)
