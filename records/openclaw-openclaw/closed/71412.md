## What This Changes

The issue reports that a timed-out `stopChannel` leaves stale channel task bookkeeping behind, causing health-monitor restarts to no-op and inbound channels to go silent.

## Best Possible Solution

Keep https://github.com/openclaw/openclaw/issues/70024 as the canonical tracker and land or close https://github.com/openclaw/openclaw/pull/70056 after maintainer review, with this issue linked as production evidence.

Yes, at source level: a channel task that ignores abort can make `stopChannel` time out, leave `store.tasks` registered, and make the next health-monitor `startChannel` hit the stale-task guard. I did not rerun the live Telegram sleep/wake or Slack deployment scenarios.

## Solution Assessment

Yes for cleanup: the best tracker shape is one canonical issue plus its open implementation PR, not two open issues for the same remaining lifecycle bug. The code fix still needs guarded recovery semantics so old timed-out task callbacks cannot corrupt a replacement lifecycle.
