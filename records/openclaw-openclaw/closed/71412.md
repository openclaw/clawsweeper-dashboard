## Summary

Keep open: current main still has a source-reproducible stop-timeout path where a never-settling channel task remains registered and blocks the health-monitor replacement start, so this is not implemented or safely duplicate-closeable.

## What This Changes

The issue asks for channel stop-timeout recovery so stale channel tasks do not block health-monitor restarts after Telegram or Slack stop timeouts.

## Best Possible Solution

Ship a guarded gateway lifecycle recovery contract that can recover from never-settling timed-out channel tasks while fencing old callbacks and transport side effects from replacement lifecycles.

## Reproduction Assessment

Yes, at source level: a channel task that ignores abort can make `stopChannel` time out, leave `store.tasks` registered, and make the next `startChannel` return without starting a replacement. I did not rerun the live Telegram sleep/wake or Slack production scenario.

## Solution Assessment

Unclear for the proposed direct cleanup alone: clearing the maps fixes the no-op start, but the maintainable fix needs task identity/generation fencing or an accepted force-retire policy for old lifecycle side effects.
