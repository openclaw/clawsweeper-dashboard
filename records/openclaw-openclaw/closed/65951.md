## Summary

Current `main` does not support the reported failure mode: Telegram preflight transcript replacement and inbound-audio detection are separate, and the inbound context still carries `MediaType`/`MediaTypes` after `BodyForAgent` is replaced with transcript text.

## Evidence

- **Telegram body preflight replaces placeholder with transcript for DM voice notes:** `resolveTelegramBody(...)` computes `hasAudio`, runs preflight transcription with `MediaPaths`/`MediaTypes`, and replaces `bodyText` when the body was `<media:audio>`. The adjacent DM test covers this exact path and expects `bodyText` to become transcript text.
  - file: [extensions/telegram/src/bot-message-context.body.ts:193](https://github.com/openclaw/openclaw/blob/250acdd2a28c190ed7934e3ba99022f24123d3c9/extensions/telegram/src/bot-message-context.body.ts#L193)
  - command: `nl -ba extensions/telegram/src/bot-message-context.body.ts | sed -n '193,232p'; nl -ba extensions/telegram/src/bot-message-context.body.test.ts | sed -n '149,174p'`
  - sha: [250acdd2a28c](https://github.com/openclaw/openclaw/commit/250acdd2a28c190ed7934e3ba99022f24123d3c9)
- **Telegram inbound context preserves media metadata after transcript substitution:** `buildTelegramInboundContextPayload(...)` writes `BodyForAgent: bodyText` but independently keeps `MediaPath`, `MediaPaths`, `MediaType`, `MediaUrl`, `MediaUrls`, and `MediaTypes` from `contextMedia`. The transcript swap does not clear audio metadata before dispatch.
  - file: [extensions/telegram/src/bot-message-context.session.ts:320](https://github.com/openclaw/openclaw/blob/250acdd2a28c190ed7934e3ba99022f24123d3c9/extensions/telegram/src/bot-message-context.session.ts#L320)
  - command: `nl -ba extensions/telegram/src/bot-message-context.session.ts | sed -n '320,375p'`
  - sha: [250acdd2a28c](https://github.com/openclaw/openclaw/commit/250acdd2a28c190ed7934e3ba99022f24123d3c9)
- **Finalization guarantees media types remain present when media exists:** `finalizeInboundContext(...)` counts media entries and fills `MediaType`/`MediaTypes` whenever media is present, including padding arrays when needed. Tests assert that media-bearing contexts always retain these fields.
  - file: [src/auto-reply/reply/inbound-context.ts:98](https://github.com/openclaw/openclaw/blob/250acdd2a28c190ed7934e3ba99022f24123d3c9/src/auto-reply/reply/inbound-context.ts#L98)
  - command: `nl -ba src/auto-reply/reply/inbound-context.ts | sed -n '98,126p'; nl -ba src/auto-reply/inbound.test.ts | sed -n '166,200p'`
  - sha: [250acdd2a28c](https://github.com/openclaw/openclaw/commit/250acdd2a28c190ed7934e3ba99022f24123d3c9)
- **Inbound-audio gate checks media types before body placeholder fallback:** `isInboundAudioContext(...)` returns `true` as soon as any normalized `MediaType`/`MediaTypes` entry is audio. It only falls back to body-text placeholder/header checks if those media fields are absent or non-audio, so replacing `<media:audio>` with transcript text alone does not make `inboundAudio` false on current `main`.
  - file: [src/auto-reply/reply/dispatch-from-config.ts:131](https://github.com/openclaw/openclaw/blob/250acdd2a28c190ed7934e3ba99022f24123d3c9/src/auto-reply/reply/dispatch-from-config.ts#L131)

Current `main` still preserves audio metadata through the Telegram -> finalized-context -> dispatch path above, so that specific theory is obsolete against current code.
