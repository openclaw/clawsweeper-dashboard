## Summary

Current main already exposes a first-class filesystem listing tool (`ls`) in the agent toolset, which covers the reported need to check whether paths exist without forcing a noisy `read` failure. OpenClaw builds its base coding tools from `createCodingTools(...)`, keeps non-overridden built-ins such as `ls`, reserves `ls` in the embedded runtime allowlist, and advertises `ls` to the model as “List directory contents.” The latest release commit also contains the same tool surface.

## Evidence

  - file: [src/agents/pi-tools.ts:453](https://github.com/openclaw/openclaw/blob/a958b6e72328d1d922a6d133b1a70e866faf4013/src/agents/pi-tools.ts#L453)

## Risks / Open Questions

- OpenClaw does not appear to expose a separate `stat` tool with the exact metadata proposed in the issue, so the match is functional rather than one-to-one.
- `docs/tools/index.md` highlights `read`/`write`/`edit` but does not explicitly list `ls`, so discoverability may still be imperfect even though the capability exists.

## Close Comment

Closing this as implemented on current OpenClaw. A Codex review of `main` found that the agent runtime already exposes a first-class `ls` tool for filesystem inspection: `createOpenClawCodingTools()` keeps the built-in coding tools from `createCodingTools(...)` (`src/agents/pi-tools.ts:453`), the embedded runtime reserves `ls` in its built-in tool-name allowlist (`src/agents/pi-embedded-runner/tool-name-allowlist.ts:8`), and the system prompt explicitly advertises `ls` to the model as `List directory contents` (`src/agents/system-prompt.ts:485`). That gives agents a non-`exec` way to check paths without provoking a noisy `read`/`ENOENT` probe. The same tool surface is already present in the latest release, `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`).
