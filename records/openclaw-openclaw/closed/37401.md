## Summary

Current `main` does not appear to load a per-agent workspace `.env` at request/exec time. The only implemented dotenv loading path is startup-time loading from `process.cwd()/.env` plus the global state-dir `.env`, so the feature request is not implemented on `main` and should remain open.

## Evidence

- **Startup dotenv loader is cwd-only:** `loadDotEnv()` resolves only `path.join(process.cwd(), ".env")`, then loads the global runtime fallback. It takes no workspace or agent parameter.
  - file: [src/infra/dotenv.ts:281](https://github.com/openclaw/openclaw/blob/7d471837369885c12d540d6bafa319ff41129b89/src/infra/dotenv.ts#L281)
  - command: `nl -ba src/infra/dotenv.ts | sed -n '170,310p'`
  - sha: [7d4718373698](https://github.com/openclaw/openclaw/commit/7d471837369885c12d540d6bafa319ff41129b89)
- **Config loading hydrates dotenv once from startup env:** `maybeLoadDotEnvForConfig()` calls `loadDotEnv({ quiet: true })` only when operating on the real `process.env`, which is consistent with startup-time hydration rather than per-agent request-time loading.
  - file: [src/config/io.ts:915](https://github.com/openclaw/openclaw/blob/7d471837369885c12d540d6bafa319ff41129b89/src/config/io.ts#L915)

  - file: [src/cli/run-main.ts:178](https://github.com/openclaw/openclaw/blob/7d471837369885c12d540d6bafa319ff41129b89/src/cli/run-main.ts#L178)
