## Summary

Current main already covers the reported dreaming-narrative leak path: memory-core Dream Diary subagent runs use deterministic `dreaming-narrative-*` session keys, accepted attempts are deleted in a `finally` block, plugin-owned deletion removes the session-store row and archives transcripts, and a scrubber cleans stale dreaming entries/orphan transcripts. The proof commit is on `main` only in this checkout; no containing release tag was found.

## What This Changes

Request automatic cleanup or eviction of completed memory-core dreaming-narrative isolated sessions so `sessions_list` does not grow unbounded.

## Best Possible Solution

Keep the current owner-scoped memory-core cleanup path: Dream Diary runs should delete their own accepted subagent sessions immediately, while generic cron run aliases remain covered by cron retention and session-list filtering.

## Reproduction Assessment

Yes. The high-confidence verification path is source/test-backed: create a Dream Diary narrative run and confirm the generated `dreaming-narrative-*` session key is passed to `deleteSession`, with Gateway tests proving that owned deletion removes the session-store row.

## Solution Assessment

Yes. The current solution is narrower and safer than a broad TTL: memory-core deletes only sessions it created, preserves foreign/user sessions, and keeps a scrubber for stale artifacts left behind by older or failed cleanup paths.
