Keep open: current main still matches the latest-release regression path, and the source plus dependency contracts support the reported CLI STT failure. No duplicate issue or open fix PR was found for the remaining work.

## What This Changes

The issue reports that `whisper-cli` audio transcription fails for Telegram voice messages after upgrading from v2026.5.7 to v2026.5.12.

## Best Possible Solution

Keep the fs-safe staging hardening, force ffmpeg to emit WAV for the staged temp output, and cover that exact argument shape with a focused regression test.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level yes, live no: current main sends non-WAV `whisper-cli` audio through ffmpeg with a `.wav.part` temp output and no `-f wav`; live proof needs a host with ffmpeg and whisper-cli installed.

## Solution Assessment

Is this the best way to solve the issue? Yes: the narrow maintainable fix is to keep fs-safe staging and make the ffmpeg output format explicit, rather than reverting the broader output-safety change or changing Telegram authorization behavior.
