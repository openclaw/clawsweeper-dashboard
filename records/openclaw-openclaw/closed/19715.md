## Summary

Current main already supports explicit owner authorization and explicit notification routing for chat channels, including Telegram and WhatsApp. The requested behavior is covered by per-channel `allowFrom` allowlists, the dedicated `commands.ownerAllowFrom` owner allowlist, and heartbeat delivery settings that can pin notifications to a specific channel/contact instead of relying on the last active chat.

## Risks / Open Questions

- Users who leave heartbeat routing on `target: "last"` or rely only on pairing history can still observe last-chat delivery; the explicit owner/destination config exists, but it is opt-in configuration rather than a forced migration.

## Close Comment

Closing from a Codex review. Current OpenClaw already supports explicit owner identity and explicit chat notification routing for this use case. Owner-only command/tool access is controlled by `commands.ownerAllowFrom` (`docs/gateway/configuration-reference.md:841`, `src/auto-reply/command-auth.ts:270`), while Telegram and WhatsApp both support explicit per-channel sender allowlists via `channels.telegram.allowFrom` and `channels.whatsapp.allowFrom` (`docs/channels/telegram.md:114`, `docs/channels/whatsapp.md:152`). For notifications, heartbeat delivery can be pinned to a specific channel/contact with `agents.defaults.heartbeat.target` plus `to`/`accountId` instead of using the last active chat (`docs/gateway/heartbeat.md:100`, `src/infra/outbound/targets.ts:84`). WhatsApp also explicitly falls back to configured `allowFrom` recipients when the last session recipient is not authorized (`extensions/whatsapp/src/heartbeat-recipients.ts:50`, covered by `extensions/whatsapp/src/heartbeat-recipients.test.ts:76`). I also verified these paths are present in the shipped `v2026.4.22` release.
