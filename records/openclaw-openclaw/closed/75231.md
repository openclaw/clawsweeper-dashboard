## Summary

Close as implemented on main: current main no longer relies on the incomplete/stale runtime-deps mirror behavior reported from v2026.4.27, because package updates run the updated doctor repair before restart, gateway startup stages the full bundled runtime-deps plan before plugin activation, and runtime mirror files are materialized under lock with atomic temp-file replacement.

## What This Changes

The issue asks OpenClaw to prevent package-update restarts from launching the managed gateway before bundled plugin runtime dependencies and mirrored dist files are fully staged and verifiable.

## Best Possible Solution

Keep the current main behavior: package updates should repair staged bundled runtime dependencies before restarting, and gateway startup should continue atomically preparing the runtime mirror before plugin activation. Treat the status-probe disagreement mentioned in the report as a separate follow-up only if it still reproduces after this main-line fix ships.

## Reproduction Assessment

Yes for the release behavior: the report includes concrete macOS update steps, ENOENT paths, timing evidence, and a doctor repair workaround that map directly to the bundled runtime-deps staging and mirror code. I did not run a live macOS package update in this read-only review.

## Solution Assessment

Yes. The current implementation is a maintainable fix because it verifies/repairs through the updated package before restart and makes startup mirror preparation atomic and locked, rather than relying on the completion-cache warning as the runtime-deps verifier.
