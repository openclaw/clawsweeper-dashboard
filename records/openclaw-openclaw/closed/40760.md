## Summary

Keep this PR open. ## What This Changes

The PR adds fallback/timeout-aware typing TTL calculation, timeout-specific final failure copy, and unit/e2e coverage for auto-reply typing timeout behavior.

## Best Possible Solution

Land one consolidated auto-reply timing fix that derives the typing TTL from the same fallback candidate semantics used at runtime, accounts for the transient retry replay budget, uses a narrow timeout predicate for user-facing timeout copy, and includes regression tests for true timeouts versus bare network/DNS errors.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use a narrow timeout predicate for timeout copy:** `src/auto-reply/reply/agent-runner-execution.ts:613-615`
  - body: `resolveFailoverReasonFromError(err) === "timeout"` also matches socket/DNS/connectivity codes used for failover retry classification. A bare `ECONNREFUSED` or `EAI_AGAIN` failure would now tell the user the model run timed out instead of surfacing a connectivity failure, so this should use the narrower `isTimeoutError(err)` semantics and keep explicit timeout-message matching separate.
  - confidence: 0.91
- **[P2] Budget typing TTL from actual fallback candidates:** `src/auto-reply/reply/get-reply.ts:78-80`
  - body: The new TTL math counts `1 + fallbackCount`, but the runtime can append the configured primary when no override is supplied, and transient HTTP retry can replay the whole chain. In override/channel-model sessions this still under-budgets by at least one attempt, so typing can stop before the run finishes.
  - confidence: 0.86
