## Summary

Current `main` already routes explicit Telegram forum-topic cron targets through `threadId`/`messageThreadId`, and the same regression coverage is present in the shipped `v2026.5.6` tag, so the reported root-cause path does not match current outbound delivery behavior.

## What This Changes

Cron Telegram announce delivery should send `chatId:topic:threadId` targets to the specified forum topic instead of the default topic.

## Best Possible Solution

Keep the existing shipped Telegram cron threading implementation; if this still happens in a real deployment, a new report should include the exact OpenClaw version/SHA and redacted Telegram HTTP diagnostic payload showing the missing API parameter.

## Reproduction Assessment

No: current-main source inspection and shipped regression coverage show `-100...:topic:<id>` cron targets becoming `messageThreadId`, so the reported omission does not have a high-confidence current-main reproduction path.

## Solution Assessment

Yes: the current implementation is the narrowest maintainable path because explicit `:topic:` target parsing stays in the Telegram plugin and does not require an extra `getChat` call for user-specified forum topic targets.
