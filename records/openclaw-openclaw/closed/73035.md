## Summary

Current main and the shipped v2026.4.27 tree already address the reported runtime-context echo: the hidden custom entry no longer carries the human-readable preamble, the embedded runner strips `openclaw.runtime-context` custom messages before provider conversion, and regression tests cover both behaviors.

## What This Changes

The issue reports that OpenClaw v2026.4.25 exposed the `openclaw.runtime-context` custom message preamble in assistant replies, especially through Telegram with MiniMax.

## Best Possible Solution

Keep the current runtime-context/transcript separation: runtime context may be supplied as prompt-local system context, transcript custom entries stay hidden and bare, and provider conversion filters those entries before they can become model-visible user turns.

## Reproduction Assessment

Yes for the original regression path: the older v2026.4.25 behavior is represented by the prior matching #72386 report and release notes, while current-main source and tests show that the model-visible custom-message path has been removed. A live Telegram/MiniMax run is not needed to verify the fixed boundary on current main.

## Solution Assessment

Yes. The implemented fix is better than outbound-only post-processing because it prevents the runtime-context preamble from entering provider user-turn conversion while still preserving hidden transcript metadata.
