## Summary

Keep this PR open for maintainer review: current main and the latest release still run Active Memory recall without an explicit lane, while the PR uses the existing embedded-runner lane contract and includes after-fix gateway proof. The only material remaining question is the availability tradeoff from using one process-wide `active-memory` lane for all recall work.

## What This Changes

This PR adds a named `active-memory` lane to Active Memory's embedded recall run and a regression test asserting that lane is passed.

## Best Possible Solution

Land a bounded Active Memory fix that keeps recall off the parent main lane, with maintainer-approved lane scope and the focused regression coverage; broader nonblocking/default changes should remain in the related reliability issues.

## Reproduction Assessment

Yes. Current main omits `lane` in the Active Memory embedded recall call and the embedded runner defaults omitted lanes to `main`; the linked report includes matching timeout logs, though I did not run a live repro in this read-only review.

## Solution Assessment

Yes for the narrow self-deadlock fix: passing an explicit lane uses the existing embedded-runner contract and the PR adds focused coverage. The remaining solution-fit question is whether the lane should be global or scoped per agent/session.
