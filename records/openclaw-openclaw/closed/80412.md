## Summary

Keep open. Current main now includes the `qa suite --runtime-pair` surface, but the live QA gateway still only stages portable env API-key profiles plus the Anthropic setup token; the reported OpenAI Codex OAuth-only temp-agent path still lacks usable isolated-agent auth or an explicit fail-fast preflight.

## What This Changes

The issue asks QA-lab live runtime-pair runs to stage usable OpenAI/OpenAI Codex auth into isolated QA agents or fail fast with an actionable auth-preflight error.

## Best Possible Solution

Add a narrow live runtime-pair auth preflight or portable credential-staging contract for the selected providers, while keeping OAuth/token material out of preserved QA artifacts and coordinating with the runtime-parity PR and live proof tracker.

## Reproduction Assessment

No high-confidence live reproduction was run in this read-only review because the path requires maintainer OpenAI/Codex credentials. Source inspection does show the current runtime-pair path launches cells without a live-auth preflight and only stages env API-key/token profiles into isolated QA agents.

## Solution Assessment

Unclear whether staging OAuth is the best fix; copying full OAuth material into temp QA stores would be risky. The safer direction is a fail-fast preflight or a portable env API-key staging path unless maintainers explicitly approve a narrower OAuth handling contract.
