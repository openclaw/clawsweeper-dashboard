## Summary

Current `main` and the shipped `v2026.4.27` tag already carry the resolved-config Slack action path this report asks for. The Slack action runtime now threads `cfg` into send options, `sendMessageSlack` requires that runtime config instead of reloading raw config, and the message CLI resolves channel SecretRefs before dispatching the action.

## What This Changes

The issue reports that `openclaw message send --channel slack` drops resolved runtime config, causing SecretRef-backed Slack `botToken` values to be treated as missing.

## Best Possible Solution

Keep the current runtime-config propagation and resolved-config requirement in the Slack action/send helpers; users still on 2026.4.21 should update to `v2026.4.27` or newer.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. The report gives a concrete SecretRef-backed Slack config plus `openclaw message send --channel slack` path, and the current code maps directly to that path; this read-only review did not run a live Slack send.

## Solution Assessment

Is this the best way to solve the issue? Yes. Passing resolved `cfg` through the action boundary and requiring it in the send helper is the narrow maintainable fix because it preserves SecretRef resolution at command/gateway boundaries instead of re-reading raw config inside Slack send.
