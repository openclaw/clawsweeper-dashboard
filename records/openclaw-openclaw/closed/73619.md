## Summary

Keep open: current main still lets inherited/default message-tool account context win before the calling agent's bound Slack/channel account, so the reported attribution bug remains. The focused fix PR was closed unmerged, and any replacement needs to keep account selection and message-tool SecretRef scoping aligned.

## What This Changes

Default omitted `message` tool `accountId` values to the calling agent's configured Slack/channel account binding instead of inherited delivery/default account context.

## Best Possible Solution

Preserve explicit `accountId` overrides, select the calling agent's bound channel/target account before inherited/default fallback, and make message-tool SecretRef scoping use that same selected account.

## Reproduction Assessment

Yes. Source inspection on current main shows omitted `params.accountId` plus `defaultAccountId: "parent-account"`, `agentId: "agent-b"`, and an `agent-b` binding to another account selects the inherited default before binding lookup can run.

## Solution Assessment

Yes. The narrow maintainable fix is explicit account first, calling-agent bound account second, inherited/default account last, with SecretRef scoping aligned to that selected account.
