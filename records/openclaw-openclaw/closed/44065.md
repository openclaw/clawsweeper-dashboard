## Summary

Current `main` already gives TUI and Control UI distinct gateway client IDs and instance IDs, and the gateway auth test suite explicitly exercises a Control UI connection followed by a TUI connection against the same server without rejection. The same implementation is present in the released `v2026.4.22` tree, so the reported tui/control-ui identity collision is already fixed.

## Close Comment

Closing this as implemented on main after a Codex review. The current code already gives TUI and Control UI separate gateway client IDs (`src/gateway/protocol/client-info.ts:5-9`), the TUI connects as `openclaw-tui` with its own `instanceId` (`src/tui/gateway-chat.ts:116-128`), and Control UI connects as `openclaw-control-ui` with its own `instanceId` (`ui/src/ui/app-gateway.ts:279-287`). There is also explicit gateway coverage for a Control UI connection followed by a TUI connection succeeding against the same server (`src/gateway/server.auth.control-ui.suite.ts:317-346`). The same implementation is already present in the released `v2026.4.22` tree (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). If this still reproduces on `v2026.4.22` or newer, please open a new issue with exact repro steps, auth mode, and gateway logs.
