## Summary

Current `main` already prevents bundled channel plugins like Discord from being silently blocked by a restrictive `plugins.allow` list: bundled channel config is treated as explicit activation before the generic allowlist rejection, and config-driven auto-enable now materializes channel plugin ids back into the allowlist. Regression tests cover both paths.

## Evidence

- **Bundled channel config now bypasses restrictive allowlists:** `resolveExplicitPluginSelectionShared` marks bundled channels enabled by channel config as explicit, and `resolvePluginActivationDecisionShared` returns that explicit activation before the generic `not-in-allowlist` branch when `allowBundledChannelExplicitBypassesAllowlist` is enabled.
  - file: [src/plugins/config-activation-shared.ts:116](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/src/plugins/config-activation-shared.ts#L116)
  - command: `nl -ba src/plugins/config-activation-shared.ts | sed -n '103,245p'`
  - sha: [fd65caf4b0ba](https://github.com/openclaw/openclaw/commit/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc)
- **Bundled-channel bypass is wired into runtime activation:** `resolvePluginActivationState` passes `allowBundledChannelExplicitBypassesAllowlist: true`, so the bundled-channel bypass is active in current plugin enablement resolution.
  - file: [src/plugins/config-state.ts:182](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/src/plugins/config-state.ts#L182)
