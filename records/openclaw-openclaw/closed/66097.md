## Summary

Current `main` already handles attach-only loopback CDP profiles by probing `/json/version`, resolving the returned `webSocketDebuggerUrl`, and only surfacing the `not running` error when the HTTP probe itself fails. The relevant behavior is covered by regression tests on `main`, and the proving change is present in commit `db8f7478b43550d2949c04b2a9c9ca7fd474dad3` but not in the visible `v2026.4.23` tag.

## Evidence

- **Profile-level attachOnly is preserved for custom CDP profiles:** `resolveProfile()` normalizes the configured `cdpUrl` and keeps `profile.attachOnly` over the global browser default, matching the issue's `browser.attachOnly=false` + profile `attachOnly=true` setup.
  - file: [extensions/browser/src/browser/config.ts:400](https://github.com/openclaw/openclaw/blob/250acdd2a28c190ed7934e3ba99022f24123d3c9/extensions/browser/src/browser/config.ts#L400)
  - command: `nl -ba extensions/browser/src/browser/config.ts | sed -n '340,450p'`
  - sha: [250acdd2a28c](https://github.com/openclaw/openclaw/commit/250acdd2a28c190ed7934e3ba99022f24123d3c9)
- **Current availability path only throws "not running" when HTTP reachability fails:** `ensureBrowserAvailable()` first calls `isHttpReachable()`. If HTTP is reachable, it proceeds to `isReachable()` instead of throwing the `Browser attachOnly is enabled and profile ... is not running.` error.
  - file: [extensions/browser/src/browser/server-context.availability.ts:220](https://github.com/openclaw/openclaw/blob/250acdd2a28c190ed7934e3ba99022f24123d3c9/extensions/browser/src/browser/server-context.availability.ts#L220)

  - file: [extensions/browser/src/browser/chrome.ts:186](https://github.com/openclaw/openclaw/blob/250acdd2a28c190ed7934e3ba99022f24123d3c9/extensions/browser/src/browser/chrome.ts#L186)
