## Summary

Keep open; the PR is aimed at the right session path, but the constant default label conflicts with the existing unique-label contract and the CLI surface cited by the linked issue would still not show the stored label.

## What This Changes

The PR adds a gateway default label for auto-keyed dashboard sessions, two gateway regression tests, and a changelog entry.

## Best Possible Solution

Generate a readable per-session label that respects label uniqueness, then expose that friendly name in the session list surfaces named by the issue without weakening the existing duplicate-label validation.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: create two dashboard sessions through `sessions.create` without `key` or `label`; the PR would assign `WebUI chat` both times, and the second write hits the duplicate-label guard.

## Solution Assessment

No. Storing a constant label is not the best fix because labels are unique, and the local `openclaw sessions` text/json paths currently drop labels anyway.
