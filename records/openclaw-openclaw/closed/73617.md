## Summary

Current main already honors root `browser.executablePath` for the managed browser profile. The browser plugin now reads runtime config through its local SDK shim, `resolveBrowserConfig` keeps the top-level executable path, `resolveProfile` inherits it into the `openclaw` profile, status exposes it, and start passes it into executable resolution before auto-detection.

## Best Possible Solution

Close this as implemented on current main and keep the existing browser config/runtime path. The next useful maintainer action is release/upgrade verification; if a 2026.4.27-or-newer packaged install still shows `detectedPath: auto` with root `browser.executablePath` set, that should be reopened or filed as a new packaging/runtime-snapshot regression with `browser status --json` and gateway logs.
