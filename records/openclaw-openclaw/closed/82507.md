## Label Justifications

- `P2`: This is a real ACPX/Codex integration improvement with limited blast radius, but it needs product/security review rather than emergency handling.
- `impact:security`: The requested inheritance would change which user-installed plugin code can execute inside background ACPX Codex tasks.
- `impact:auth-provider`: The issue centers on Codex home selection and native Codex plugin/provider availability for ACPX sessions.

## Summary

Keep open for maintainer product/security review: current main still forces ACPX Codex through an isolated generated `CODEX_HOME`, and the requested plugin inheritance or marketplace path would add new background-plugin execution policy rather than fix a documented bug.

## What This Changes

Request ACPX Codex background tasks to inherit or explicitly configure user-installed Codex plugins such as Superpowers from the user’s normal Codex home.

## Best Possible Solution

Keep ACPX isolated by default, and if maintainers accept this capability, add an explicit allowlisted Codex home, plugin, or marketplace path with public docs, focused ACPX tests, and redacted real-runtime proof.

## Reproduction Assessment

No live end-to-end reproduction was run, but source, tests, and docs give a high-confidence source reproduction: current main always launches Codex ACP with the generated isolated `state/acpx/codex-home`. That path has no plugin-inheritance, marketplace, or `skills.load.extraDirs` handoff.

## Solution Assessment

No, automatic inheritance of the operator’s personal Codex plugins is not clearly the safest default. A narrower maintainable path would be explicit and allowlisted, such as a reviewed Codex-home or marketplace override with proof and docs.
