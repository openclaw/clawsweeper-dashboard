## Summary

Keep open. Current main still routes `openclaw update` restart preparation through the managed user-scope gateway service and generates a Linux restart script using `systemctl --user`; system-level gateway units are documented and discovered elsewhere but not handled by the updater. Open PR #74920 appears to address this path, but it is still open and unmerged.

## What This Changes

The issue asks for `openclaw update` to detect system-wide Linux systemd gateway units and restart them, or print accurate `sudo systemctl restart ...` guidance after package updates.

## Best Possible Solution

Land or split the update-restart portion of #74920 so package updates identify active system-scoped OpenClaw systemd units and either restart them when already authorized or emit precise sudo restart guidance, while preserving the existing managed user-service behavior.

## Reproduction Assessment

Yes. The issue body gives a concrete Debian/systemd system-unit setup, and current-main source inspection reproduces the missing updater path: update restart preparation and the Linux restart helper are user-service only while docs support system services.

## Solution Assessment

Yes. Extending the update restart path to reuse existing system-scope service discovery and emit explicit privileged restart guidance is the narrow maintainable fix; changing the normal gateway service adapter to control system units by default would be riskier.
