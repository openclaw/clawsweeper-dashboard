## Summary

Current main still matches the reported bug: the non-ACP `/new`/`/reset` path emits reset hooks and returns `null`, so a bare reset can continue through normal reply generation after session init has stripped the command body to an empty tail. The provided GitHub context also shows two open implementation PRs, #73371 and #73377, with closing syntax for this issue, so maintainer policy says to keep the issue open until one is reviewed and merged or closed.

## Best Possible Solution

Review and land the best of #73371 or #73377, or apply the same narrow fix directly: after non-ACP reset hooks fire, if `resetTail` is empty, return `shouldContinue: false` with `✅ Session reset.` or `✅ New session started.`, while preserving `/reset <tail>` and `/new <tail>` fallthrough. Add focused regression tests in `src/auto-reply/reply/commands-reset-hooks.test.ts` for bare `/reset`, bare `/new`, and tailed reset/new behavior.
