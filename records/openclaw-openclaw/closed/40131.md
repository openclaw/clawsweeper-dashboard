## Summary

Keep open. Current `main` still appears to drop Telnyx inbound webhooks after returning HTTP 200: the Telnyx parser does not populate `direction`/`from`/`to`, while the manager only auto-registers untracked webhook calls when `event.direction` is present. There is also still no merged Telnyx-specific answer path on `main`, and the linked fix PR #40188 remains open and unmerged.

## Evidence

- **Telnyx parser still omits inbound metadata:** `TelnyxProvider.normalizeEvent()` builds `baseEvent` with only `id`, `dedupeKey`, `callId`, `providerCallId`, and `timestamp`, then returns `call.initiated`/`call.ringing`/`call.answered` without `direction`, `from`, or `to`. Twilio/Plivo do set those fields.
  - file: [extensions/voice-call/src/providers/telnyx.ts:140](https://github.com/openclaw/openclaw/blob/bfa6708c03f24c7caa5d0a96a298c292be0d85c1/extensions/voice-call/src/providers/telnyx.ts#L140)
  - command: `nl -ba extensions/voice-call/src/providers/telnyx.ts | sed -n '120,220p'; nl -ba extensions/voice-call/src/providers/twilio.ts | sed -n '300,340p'; nl -ba extensions/voice-call/src/providers/plivo.ts | sed -n '240,270p'`
  - sha: [bfa6708c03f2](https://github.com/openclaw/openclaw/commit/bfa6708c03f24c7caa5d0a96a298c292be0d85c1)
- **Manager only auto-registers untracked webhook calls when direction exists:** `processEvent()` auto-creates a call only under `if (!call && providerCallId && eventDirection)`. If Telnyx sends an untracked inbound `call.initiated` without `event.direction`, execution falls through to `if (!call) return;`, which matches the reported silent drop.
  - file: [extensions/voice-call/src/manager/events.ts:113](https://github.com/openclaw/openclaw/blob/bfa6708c03f24c7caa5d0a96a298c292be0d85c1/extensions/voice-call/src/manager/events.ts#L113)

  - sha: [bfa6708c03f2](https://github.com/openclaw/openclaw/commit/bfa6708c03f24c7caa5d0a96a298c292be0d85c1)
- **Webhook path still returns 200 after processing attempt:** After verification, the webhook server parses the provider event, calls `processParsedEvents(parsed.events)`, and then always returns `normalizeWebhookResponse(parsed)`, which defaults to status 200/body `OK`. That explains a 200 response even when the manager drops the event.
  - file: [extensions/voice-call/src/webhook.ts:653](https://github.com/openclaw/openclaw/blob/bfa6708c03f24c7caa5d0a96a298c292be0d85c1/extensions/voice-call/src/webhook.ts#L653)
