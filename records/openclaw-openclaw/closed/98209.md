## Label Justifications

- `P1`: A real iOS Talk workflow can fail for SecretRef-backed ElevenLabs keys and may push users toward plaintext credential workarounds.
- `impact:security`: The affected behavior sits on the Talk provider secret-read boundary and the safe fix must avoid leaking resolved API keys to read-scope clients.
- `impact:auth-provider`: The bug is caused by SecretRef-backed provider credential resolution not reaching the native Talk config payload.
- `maturity:stable`: The issue affects the M4 Stable `gateway-runtime` feature area, specifically Gateway RPC APIs, roles and permissions, and security controls.

## Summary

Keep open: current main still has the `includeSecrets` early return that can send a source-shaped SecretRef to iOS, and the open linked fix candidate at https://github.com/openclaw/openclaw/pull/98210 closes this issue.

## What This Changes

Fix Gateway `talk.config` so SecretRef-backed ElevenLabs Talk API keys resolve to a string for iOS clients with `operator.talk.secrets` while remaining redacted for ordinary reads.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/98210, or an equivalent narrow Gateway fix, with tests proving resolved keys for scoped callers and redaction for unscoped callers.

## Reproduction Assessment

Yes: source inspection on current main shows `includeSecrets` returns before runtime SecretRef resolution, while iOS only accepts a string `apiKey`. I did not run tests because this review is read-only.

## Solution Assessment

Yes: the linked narrow Gateway fix is the right shape because it removes the early return and uses the runtime-resolved config only for already-authorized secret reads while preserving redaction for ordinary reads.
