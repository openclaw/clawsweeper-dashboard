## Label Justifications

- `impact:message-loss`: Final replies can fail to deliver for SIP-enabled iMessage users when OpenClaw attempts a native threaded reply despite reply actions being disabled.

## Summary

Keep this open. Current main and the latest release still forward iMessage final-reply metadata into outbound send without honoring `channels.imessage.actions.reply=false`, and the two linked closing PRs are still open implementation candidates rather than merged fixes.

## What This Changes

The issue asks OpenClaw to stop passing iMessage `reply_to`/`replyToId` into final outbound delivery when iMessage reply actions are disabled or private threaded replies are unavailable.

## Best Possible Solution

Land one narrow iMessage send-boundary fix that suppresses `replyToId` when `actions.reply` is false, with regression tests and redacted real macOS/iMessage proof; track the TTS override separately if maintainers want it.

## Reproduction Assessment

Yes for source-level reproduction: current main forwards `payload.replyToId` from final iMessage delivery into `sendMessageIMessage`, which unconditionally serializes it as `reply_to`. I did not run a live SIP-enabled macOS iMessage send in this read-only review.

## Solution Assessment

Yes, the best fix is to honor the existing `actions.reply` config at the iMessage send boundary shared by text and media delivery. Adding a new `replyToMode` option is not necessary for the reported bug because the existing reply-action knob already expresses the user's opt-out.
