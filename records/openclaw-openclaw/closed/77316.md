## Summary

Close as implemented: current main applies the effective denylist during optional media/PDF factory planning, so denied `pdf` prevents `createPdfTool` construction instead of relying on later filtering, with regression coverage and release proof in `v2026.5.3`.

## What This Changes

The issue asks for `tools.deny: ["pdf"]` to suppress built-in PDF tool initialization/prep rather than only filtering the tool after setup.

## Best Possible Solution

Keep the existing denylist-gated factory planning; operators on `2026.5.2` should upgrade to `v2026.5.3` or later rather than adding a separate PDF-specific config knob.

## Reproduction Assessment

No for current main: source inspection and regression coverage show `{ tools: { deny: ["pdf"] } }` makes the optional media plan return `pdf: false` before `createPdfTool` can run. The reported `2026.5.2` timing data is consistent with the pre-fix path, but the current-main path does not reproduce it.

## Solution Assessment

Yes. Applying the effective denylist inside optional media factory planning is the narrowest maintainable fix because it avoids initialization work before later tool filtering; a new `tools.media.pdf.enabled` switch is unnecessary for this case.
