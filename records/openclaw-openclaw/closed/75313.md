## Summary

Keep this open. Current main still destructively drops blank-only user session entries, and the open unmerged PRs #75368 and #75606 are implementation candidates rather than a reason to close the issue early.

## What This Changes

Rewrite blank-only persisted user session messages to non-empty placeholder content instead of dropping them during session-file repair, so strict OpenAI-compatible providers still receive a user turn.

## Best Possible Solution

Land one canonical source-level fix that rewrites blank-only user session entries to neutral non-empty content, preserves media-block cleanup behavior, and updates the focused session-file repair regression tests; use the broader PR only if maintainers also want to solve the adjacent trailing-assistant repair case in the same change.

## Reproduction Assessment

Yes. Current source reproduces the behavior: blank-only user content returns `drop`, the repair loop omits that entry, and the unit test asserts the blank user line disappears from the repaired session file.

## Solution Assessment

Yes. The narrowest maintainable solution belongs in `src/agents/session-file-repair.ts` with focused tests; a provider-specific transport guard should only be added if review proves file repair does not cover the failing replay path.
