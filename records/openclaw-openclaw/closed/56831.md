## What This Changes

The branch changes Discord component parsing and tests so empty LLM-default component specs return null instead of throwing on empty modal fields.

## Best Possible Solution

Keep https://github.com/openclaw/openclaw/pull/75652 as the canonical fix path against current main and close this older branch instead of merging its stale parser edit.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: Discord send parses object-shaped components before fallback text, and current main still throws on empty modal fields in readDiscordComponentSpec.

## Solution Assessment

No. The requested behavior is valid, but this branch is not the best solution because it edits the old parser location, relaxes malformed-field validation, and drops meaningful container-only component specs; the better path is the open canonical PR against components.parse.ts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Reject malformed modal fields:** `extensions/discord/src/components.ts:600`
  - body: The new guard only parses modal fields when fieldsRaw is a non-empty array, so a present malformed value like a string or object is silently treated as no modal. Keep empty arrays as absent, but still reject present non-array field values so bad tool calls do not fall through as plain messages.
  - confidence: 0.9
- **[P2] Preserve container-only component specs:** `extensions/discord/src/components.ts:623-624`
  - body: This null guard ignores components.container, but sendMessage adds top-level content as component text fallback only after parsing. A payload with container styling plus message content would now skip the component sender and lose the styling.
  - confidence: 0.87
