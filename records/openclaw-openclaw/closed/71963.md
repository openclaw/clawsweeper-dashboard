## Summary

Current main implements the requested audit-side recovery: active cron task ledger records are reconciled from durable cron run logs or persisted cron job state before `tasks audit`/maintenance can mark them `lost`. That directly covers the reported `sessionTarget: "isolated"`, `lastRunStatus: ok`, backing-session-missing false positive. The secondary `inconsistent_timestamps` warning is separate from the false-lost cron task issue.

## Best Possible Solution

Close #71963 as implemented on current main. Keep the durable audit/maintenance recovery path, and handle any remaining timestamp-ordering warning as a separate focused bug report. The linked #71968 discussion was useful for diagnosing the active-job path, but current main solves the user-visible false `lost` audit result via broader durable recovery.
