## Summary

Keep open: current `main` still separates selected model-provider tool policy, Control UI/WebChat client identity, and host exec allowlist enforcement. The reporter narrowed this away from a blanket Control UI exec block, but changing the remaining behavior would require explicit security/product direction rather than cleanup closure.

## What This Changes

The issue asks whether WebChat/Control UI host exec should honor configured elevated sender and `tools.byProvider.webchat` policy instead of ending in `exec denied: allowlist miss`.

## Best Possible Solution

Either clarify in docs/schema that `tools.byProvider` is model-provider policy and host exec allowlist remains separate, or add a reviewed browser-surface exec policy that preserves host allowlist and approval enforcement.

## Reproduction Assessment

Yes, source-level. Configuring `tools.byProvider.webchat` while the selected model provider is not `webchat`, then running gateway host exec in allowlist mode without a matching allowlist or durable approval, reaches the current fail-closed allowlist path.

## Solution Assessment

Unclear. Reinterpreting `tools.byProvider.webchat` as Control UI host exec authority is not the safest narrow fix; docs/schema clarification or an explicit reviewed browser-surface policy would be safer.
