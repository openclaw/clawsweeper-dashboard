## Summary

Keep open. Current main still has the reported budget mismatch in the non-context-engine tool-result guard, and the issue is already paired with open closing PRs that should be reviewed or reconciled before closure.

## What This Changes

The issue reports that the embedded runner's tool-result guard enforces limits from `model.contextWindow` instead of the resolved `contextTokens` runtime budget when those values diverge.

## Best Possible Solution

Land #75092, #74934, or an equivalent narrow patch that passes the already-resolved `contextTokenBudget` into the non-context-engine tool-result guard before the existing fallbacks, with regression coverage for a resolved budget that differs from `model.contextWindow`.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. A focused regression can run or unit-drive a non-context-engine embedded attempt with `contextTokenBudget` larger than `model.contextWindow` and assert `installToolResultContextGuard` currently receives the lower `model.contextWindow` value.

## Solution Assessment

Is this the best way to solve the issue? Yes for the current resolver contract. Threading `params.contextTokenBudget` into the existing guard fallback chain is the narrow maintainable fix; rejecting or normalizing inconsistent metadata would be a broader policy change.
