## Summary

Keep this open: current main still has a source-level path where Dreaming narrative sessions use timestamped keys and rely on final in-process cleanup, so a Gateway restart can still leave child sessions visible. The focused cleanup remains in the open PR at https://github.com/openclaw/openclaw/pull/70464, with broader residue evidence still tracked at https://github.com/openclaw/openclaw/issues/77723.

## What This Changes

The issue requests durable cleanup of Dreaming-spawned child sessions left visible after a Gateway restart interrupts the in-process cleanup path.

## Best Possible Solution

Land or replace the focused memory-core cleanup from https://github.com/openclaw/openclaw/pull/70464, then add regression coverage for restart-left `sessions.json` rows and related Dreaming sidecars before closing this issue.

## Reproduction Assessment

Yes, at source level. Current main still builds timestamped Dreaming narrative session keys, uses the same key for idempotency, and relies on a final `deleteSession` path that a Gateway restart can skip; I did not run a live restart during Dreaming.

## Solution Assessment

Yes, the memory-core stable-key plus pre-cleanup direction is the narrow maintainable fix. A broad Gateway startup scanner alone would be less owner-bounded and should not replace regression coverage for restart-left Dreaming rows.
