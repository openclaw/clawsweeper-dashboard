## Summary

Keep open. The latest branch is a useful narrow compatibility patch, but it still should not merge because it makes APPDATA-only Google Vertex ADC evidence resolve as authenticated while the request-time Google Vertex ADC reader on current main does not look in APPDATA, so those accepted setups fail when the transport tries to mint Bearer headers.

## What This Changes

The PR adds `${APPDATA}` auth-evidence path expansion and documentation, adds an APPDATA Google Vertex ADC fallback in the Google plugin manifest, and adds resolver tests while leaving `GOOGLE_CLOUD_PROJECT_ID` unsupported.

## Best Possible Solution

Align Google Vertex's provider-owned ADC lookup with the manifest evidence before merge: the runtime header path should find the same APPDATA/default Windows ADC file that `resolveEnvApiKey` accepts, with a transport/header regression test proving APPDATA-only authorized_user ADC resolves to Bearer auth. If maintainers still want `GOOGLE_CLOUD_PROJECT_ID`, expand the manifest and runtime project resolver together rather than only updating evidence.
