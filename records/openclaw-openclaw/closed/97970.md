## Summary

Keep open: the reported failure mode is credible because current main still refuses `gateway.bind=lan` with `auth.mode:none`, and the stable update/install path does refresh gateway service state during update; however, the exact updater step that allegedly writes `gateway.bind=lan` for a previously omitted bind is not yet proven from source, so this needs maintainer-side reproduction of the update/service-install path rather than closure.

## What This Changes

The issue reports that `openclaw update` on v2026.6.10 fills a missing `gateway.bind` with `lan`, causing existing `auth.mode:none` configurations to fail gateway startup with exit 78.

## Best Possible Solution

Preserve an omitted `gateway.bind` through update/service-install refreshes so the documented loopback default remains effective; if a path must set a non-loopback bind, it should require explicit auth or stop before restarting with an actionable diagnostic.

## Reproduction Assessment

Yes for the startup failure path: source and tests show `gateway.bind=lan` with `auth.mode:none` exits 78 with the reported message. No for the full update mutation path: source inspection did not yet identify where `openclaw update` writes `gateway.bind=lan` for an omitted bind.

## Solution Assessment

Unclear: preserving the omitted bind/default loopback is the narrowest maintainable fix if the updater is mutating config, while the proposed `config lock` is a separate product feature and should not block the bug fix.
