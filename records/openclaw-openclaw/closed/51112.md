## Summary

Keep open. Current main still has an active `openai-completions` transport path that dereferences streamed chunks before any null/object guard, so the reported null-chunk crash remains plausible and is not strongly proven fixed by the Pi package bump alone.

## Evidence

- **Unguarded chunk access in active OpenAI completions stream loop:** `processOpenAICompletionsStream` reads `chunk.id` and `chunk.usage` before validating that the streamed item is a non-null object. It only guards `chunk.choices` with `Array.isArray(...)`, so a `null` chunk would still throw before the guard runs.
  - file: [src/agents/openai-transport-stream.ts:1295](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/src/agents/openai-transport-stream.ts#L1295)
  - command: `nl -ba src/agents/openai-transport-stream.ts | sed -n '1268,1312p'`
  - sha: [6944d7025d9c](https://github.com/openclaw/openclaw/commit/6944d7025d9c0ba4891cbe646ab5dab039d504ca)
- **Default embedded-agent path uses boundary-aware transport instead of plain `streamSimple`:** When the session uses the default base stream, embedded runs switch to `createBoundaryAwareStreamFnForModel(params.model)`. That keeps OpenClaw on its own transport path for supported APIs during normal agent runs.
  - file: [src/agents/pi-embedded-runner/stream-resolution.ts:120](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/src/agents/pi-embedded-runner/stream-resolution.ts#L120)
  - command: `nl -ba src/agents/pi-embedded-runner/stream-resolution.ts | sed -n '112,126p'`
  - sha: [6944d7025d9c](https://github.com/openclaw/openclaw/commit/6944d7025d9c0ba4891cbe646ab5dab039d504ca)
- **Boundary-aware transport explicitly routes `openai-completions` to OpenClaw's local transport:** `createSupportedTransportStreamFn` returns `createOpenAICompletionsTransportStreamFn()` for `model.api === "openai-completions"`. That means the unguarded loop above is on the live transport path for this API.
  - file: [src/agents/provider-transport-stream.ts:83](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/src/agents/provider-transport-stream.ts#L83)
  - command: `nl -ba src/agents/provider-transport-stream.ts | sed -n '78,86p'`
  - sha: [6944d7025d9c](https://github.com/openclaw/openclaw/commit/6944d7025d9c0ba4891cbe646ab5dab039d504ca)
- **Pi packages are already bumped on main, but no corresponding local null-chunk guard is present:** Current main already pins `@mariozechner/pi-ai` to `0.70.0`, and the changelog records that bump in `Unreleased`. That reduces confidence in closing this as already fixed, because OpenClaw's own active transport code still shows the unsafe access pattern.

## Risks / Open Questions

- If maintainers close this as implemented, custom OpenAI-compatible providers that emit `null`/non-object SSE payloads may still crash agent runs in OpenClaw's boundary-aware `openai-completions` transport.
- The issue body points at an upstream Pi fix, but current main's local transport path means a dependency bump alone is not strong proof that the OpenClaw-facing bug is gone.
