## Summary

Close as implemented on current main. The current packaged runtime-deps path includes `chokidar` in the mirrored root dependency plan, computes missing deps from installed package satisfaction in stage roots rather than generated-manifest membership alone, and has regression coverage for already staged package-level deps not being reported missing; the proof commit is on `main` but not in `v2026.4.27`.

## What This Changes

The issue reports that `chokidar@^5.0.0` is reinstalled on every gateway restart because bundled runtime dependency missing detection treats a cached manifest that omits the package as missing even when the package is already present.

## Best Possible Solution

Keep the current main behavior: stage package-level mirrored runtime deps from the declared root dependency set, treat installed package satisfaction as authoritative for missing detection, and let the next release carry the main-line fix to users still on `v2026.4.27`.

## Reproduction Assessment

Yes. The reporter's log text matches the `v2026.4.27` startup logging path, and current main has source and regression-test coverage showing already staged package-level runtime deps are not reported missing; I did not run a live gateway restart because this review was read-only.

## Solution Assessment

Yes. The current approach is narrower and more maintainable than recording every transitive npm package: OpenClaw records the declared install plan, validates actual installed packages by name/version/entry files, and avoids binding cache correctness to npm's transitive graph layout.
