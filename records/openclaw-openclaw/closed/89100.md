## Label Justifications

- `P1`: The report describes real channel privacy leakage and group reply misrouting in a currently supported message-tool workflow.
- `impact:security`: The issue is about internal `chat_id`, `sender_id`, and sender metadata becoming visible in real channel conversations.
- `impact:message-loss`: The group-routing failure sends replies to a DM instead of the originating group, so the intended conversation receives no response.

## Summary

Keep open: current main has some outbound internal-context and boot-echo sanitization, but it still does not apply the existing inbound-metadata stripper to outbound `message.send` text, and explicit sender-id targets are still preserved instead of guarded against current-source group routing mistakes.

## What This Changes

The issue requests core runtime hardening for outbound `message.send` arguments so leaked inbound scaffolding is stripped and group-chat replies cannot be misrouted to the sender DM.

## Best Possible Solution

Add shared core hardening that reuses `stripInboundMetadata` on outbound visible text before dispatch, suppresses or errors when a text-only send becomes empty, and rejects or safely corrects an explicit sender-id target when the current source route is a different group conversation.

## Reproduction Assessment

Yes, at source level: a `message(action=send)` body containing the existing delivery and inbound metadata sentinels will pass through current outbound sanitization without `stripInboundMetadata`, and an explicit `to` value is preserved as the target instead of defaulting to current source. I did not run the live MiniMax/channel scenario.

## Solution Assessment

Yes: the narrowest maintainable fix is in the shared message-tool/outbound runtime boundary, not in individual channel adapters or provider-specific prompt guidance. Weak-model prompt hints can be follow-up defense in depth, but the core runtime should prevent the leak and obvious sender-id misroute.
