## Summary

Current main already fixes the cross-date Gateway logging problem this PR was trying to address. The exact duplicated `channels logs` resolver from the PR is not present, but long-lived Gateway and subsystem loggers now re-resolve the active rolling file on each write, so they move to the current-day log instead of continuing to write stale dated files. The normal `openclaw logs --follow`/Control UI path also uses the shared `logs.tail` resolver for missing daily files. This PR is now stale relative to current main and should close rather than merge its duplicated helper and brittle test.

## Best Possible Solution

Close this PR as implemented on current main. Keep the shipped writer-side rolling-file behavior and shared `logs.tail` fallback; if maintainers later want `openclaw channels logs` to read older channel history when today's file is absent, implement that as a small reuse of `readConfiguredLogTail()` or a shared logging utility, not by merging this stale duplicated resolver/test patch.
