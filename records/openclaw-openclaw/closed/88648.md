## Summary

Keep open: the `--json` startup-progress disable is useful, but the spinner guard still treats undefined `process.stdout.isTTY` as safe, so the piped-stdout half of the fix remains broken. The PR also still lacks inspectable after-fix terminal output or logs for the real JSON pipeline.

## What This Changes

The PR disables CLI startup progress when argv contains `--json`, adds a stdout TTY guard around Clack spinner creation, and updates CLI progress tests.

## Best Possible Solution

Require `process.stdout.isTTY === true` before creating the Clack spinner, cover the undefined/non-TTY stdout case in tests, and add redacted terminal proof for a real JSON pipeline before merge.

## Reproduction Assessment

Yes, for the patch defect: captured stdout reports `process.stdout.isTTY` as undefined, and the PR treats that as true before creating a Clack spinner. I did not run the full `openclaw sessions --json | jq` pipeline.

## Solution Assessment

No. Disabling startup progress for `--json` is the right direction, but the piped-stdout guard needs a strict true-TTY check and real terminal proof before this is the best fix.

- **[P1] Require true stdout TTY before starting the spinner:** `src/cli/progress.ts:109-110`
  - body: In a piped or captured process, Node can expose `process.stdout.isTTY` as `undefined`; the review environment does exactly that. Please require `process.stdout.isTTY === true` and cover the undefined case.
