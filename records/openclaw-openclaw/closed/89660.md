## Summary

Keep open. Current main still has a strict config-schema/runtime mismatch: the OpenAI completions runtime and shared compat types know about `requiresReasoningContentOnAssistantMessages`, but `ModelCompatSchema` and `ModelCompatConfig` do not accept it in user model config.

## What This Changes

The issue requests that custom provider model compat config accept `requiresReasoningContentOnAssistantMessages` so DeepSeek-compatible proxies can trigger the same assistant `reasoning_content` behavior as the native DeepSeek provider.

## Best Possible Solution

Align the public config schema, exported config type, docs, and focused regression tests with the already-existing runtime compat flag, without adding broader provider auto-detection policy.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: the strict model compat schema rejects unlisted compat keys, and the requested key is absent while runtime code reads it. I did not run the gateway because this review is read-only.

## Solution Assessment

Yes, the narrow best fix is to expose the already-existing runtime flag in the model compat config schema/type and add focused schema/runtime coverage; expanding provider auto-detection would be a separate product choice.
