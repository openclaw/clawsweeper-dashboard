## Summary

Current main fixes the reported Telegram reply-context projection path. Telegram still captures ReplyToId/ReplyToBody/ReplyToSender, the inbound user-context builder renders that reply body as a Replied message block, and the embedded runner now appends hidden runtime context to the prompt-local system context before calling the model instead of relying only on the hidden custom transcript entry. The fix is present on main but is not contained in the latest tagged release v2026.4.26.

## Best Possible Solution

Close this issue as fixed on current main and keep the prompt-local runtime-context path: Telegram reply metadata should stay in the runtime user-context block, be supplied to the model for the active turn via the temporary system prompt, and remain hidden from normal transcript replay.
