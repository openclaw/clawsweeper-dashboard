## Summary

Current main does not match the reported Feishu TTS failure path. Feishu outbound media is uploaded to Feishu first via `im.file.create`, then sent using the returned `file_key`; Feishu is also explicitly treated as a native voice-note TTS channel, with tests and docs covering Opus/audio delivery. The issue's stated root cause (passing a local file path directly to Feishu without upload) is not true on current main or the latest release snapshot inspected.

## Evidence

- **Feishu upload-before-send path on main:** `sendMediaFeishu` uploads non-image media with `uploadFileFeishu(...)`, which calls `client.im.file.create(...)`, then sends the returned `file_key` via `sendFileFeishu(...)`. This is the opposite of the issue's claimed root cause.
  - file: [extensions/feishu/src/media.ts:367](https://github.com/openclaw/openclaw/blob/350d3221800a18283915cd2f3b612ae04ddd490c/extensions/feishu/src/media.ts#L367)
  - sha: [350d3221800a](https://github.com/openclaw/openclaw/commit/350d3221800a18283915cd2f3b612ae04ddd490c)
- **Feishu TTS is routed as a voice-note target:** Feishu is in `OPUS_CHANNELS`; `textToSpeech` chooses target `voice-note` for Feishu channels, not a generic MP3 attachment path.
  - file: [extensions/speech-core/src/tts.ts:588](https://github.com/openclaw/openclaw/blob/350d3221800a18283915cd2f3b612ae04ddd490c/extensions/speech-core/src/tts.ts#L588)
  - sha: [350d3221800a](https://github.com/openclaw/openclaw/commit/350d3221800a18283915cd2f3b612ae04ddd490c)
- **TTS command preserves voice-note delivery intent:** `/tts audio` returns a reply payload with `mediaUrl: result.audioPath` and `audioAsVoice: result.voiceCompatible === true`, so Feishu-native voice-note delivery is preserved into outbound routing.
  - file: [src/auto-reply/reply/commands-tts.ts:167](https://github.com/openclaw/openclaw/blob/350d3221800a18283915cd2f3b612ae04ddd490c/src/auto-reply/reply/commands-tts.ts#L167)
  - sha: [350d3221800a](https://github.com/openclaw/openclaw/commit/350d3221800a18283915cd2f3b612ae04ddd490c)
- **Feishu media tests cover audio upload/send behavior:** Tests assert Opus uploads use `file_type: "opus"` and Feishu `msg_type: "audio"`; unsupported MP3 falls back to generic file upload/send instead of a broken voice send.
  - file: [extensions/feishu/src/media.test.ts:171](https://github.com/openclaw/openclaw/blob/350d3221800a18283915cd2f3b612ae04ddd490c/extensions/feishu/src/media.test.ts#L171)
  - sha: [350d3221800a](https://github.com/openclaw/openclaw/commit/350d3221800a18283915cd2f3b612ae04ddd490c)
- **Shipped release snapshot contains the same implementation:** Inspected release commit `00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5` (`v2026.4.22`) and found the same Feishu upload-before-send logic and Feishu voice-note TTS routing already present there.
