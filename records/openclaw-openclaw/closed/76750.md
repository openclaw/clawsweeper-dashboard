## Summary

Close as implemented on current main: the plugin-tool resolver now treats wildcard and plugin-group allowlists as selecting declared plugin tool contracts, the agent path passes global `tools.allow` into plugin materialization before applying deny policy, and `tools.catalog` loads plugin groups through the same resolver. The proof commit is on `main` only; it is not contained in `v2026.5.2`.

## What This Changes

The issue reports that `tools.allow: ["*"]` with `tools.deny` hides enabled browser and memory plugin tools from the agent and tool catalog.

## Best Possible Solution

Keep the current main implementation and let the next release carry the plugin-tool wildcard fix; no new config option or product policy is needed.

## Reproduction Assessment

No, not as a current-main failure. I did not run a live gateway smoke in this read-only sweep, but source inspection plus regression coverage show the reported wildcard path now includes browser and memory plugin contracts before deny filtering.

## Solution Assessment

Yes. The fix belongs at the generic plugin resolver and tool-policy boundary, which covers browser, memory-core, `tools.catalog`, and future plugin tools without hard-coding bundled plugin IDs.
