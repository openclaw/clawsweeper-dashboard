## Summary

This PR should close in favor of the broader active fix at https://github.com/openclaw/openclaw/pull/83068: it targets the same gateway hot-reload failure, but this patch would still report a partial channel restart as successfully applied.

## What This Changes

The PR wraps each final gateway hot-reload channel restart in a try/catch and adds a handler test for continuing to later channels after one stop failure.

## Best Possible Solution

Review and land the broader aggregate-failure fix from https://github.com/openclaw/openclaw/pull/83068 or an equivalent implementation, then close https://github.com/openclaw/openclaw/issues/83054 when it ships.

## Reproduction Assessment

Yes. A focused handler call with two `restartChannels` and a throwing `stopChannel`/`startChannel` reproduces the current-main abort path by source inspection; I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No. Continuing later restarts is right, but this PR swallows the failure; the maintainable fix is to attempt all restarts and then reject with an aggregate failure so reload status and snapshot promotion stay honest.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Preserve failure state after continuing channel restarts:** `src/gateway/server-reload-handlers.ts:433-441`
  - body: The config reloader treats `applyHotReload()` resolving as accepted and only logs `config reload failed` when it rejects. This catch only logs the channel error, so a failed channel restart still falls through to `config hot reload applied` and `setState`, which can promote a partial reload as successful. Collect failures and throw an aggregate error after attempting the remaining channels.
  - confidence: 0.88
