## Summary

Close: current main already fixes the reported main-chat `/new` routing by resetting the main session in place for the configured `session.dmScope: "main"` path, with source, docs, regression coverage, and merged PR provenance; the fix is not in `v2026.5.7` yet.

## What This Changes

The issue asks for Control UI `/new` from the main chat to keep a usable direct `agent:main:main` chat instead of switching to `agent:<agent>:dashboard:<guid>` child sessions.

## Best Possible Solution

Keep the merged current-main reset-in-place behavior and let it ship in the next release; broader `/new` versus `/reset` semantics can remain under https://github.com/openclaw/openclaw/issues/49517 if maintainers want a wider product change.

## Reproduction Assessment

No current-main failing path remains for the main-session case: source inspection and the regression test show `/new` now returns `agent:main:main` instead of a dashboard child when `session.dmScope` is `main`. I did not run tests because this review is read-only and existing checked-in coverage is sufficient for the cleanup decision.

## Solution Assessment

Yes for the reported main direct-chat path: the Gateway owns the no-key `sessions.create` fallback, so intercepting that path before `buildDashboardSessionKey` is the narrowest maintainable fix. Non-main `dmScope` or broader dashboard-session policy changes would be separate product behavior.
