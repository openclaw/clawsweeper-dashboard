## Summary

Keep open: current main still routes native Codex context-limit resolution through the selected `openai` model provider, so `models.providers.openai-codex.models[].contextTokens` can be ignored for the `openai/gpt-*` plus `agentRuntime.id: "codex"` path. The linked fix PR is still open with closing syntax, so this issue should remain open until that work or an equivalent change lands.

## What This Changes

The issue requests that `openai/gpt-*` models running through the native `codex` harness resolve Codex OAuth context-window and context-token overrides from `openai-codex` instead of `openai`.

## Best Possible Solution

Land PR #77861 or an equivalent narrow fix that carries an explicit `openai-codex` context-config provider for native Codex runs and uses it consistently in run, status, and session-persistence paths.

## Reproduction Assessment

Yes, as a source-level reproduction: configure `models.providers.openai-codex.models[].contextTokens` for `gpt-5.5`, then run `openai/gpt-5.5` through `agentRuntime.id: "codex"`; the current resolver still passes provider `openai` into provider-keyed context lookup. I did not live-run the native Codex harness in this read-only review.

## Solution Assessment

Yes, the best fix direction is the linked narrow implementation: introduce a context-config provider for native Codex and keep the model ref/auth boundary unchanged. That addresses the bug without adding a new config option or changing the documented route.
