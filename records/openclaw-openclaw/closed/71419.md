## Summary

Current `main` and the latest shipped release route CLI plugin warnings and gateway plugin loading through the same discovery/manifest pipeline. Unsafe ownership candidates are dropped before loading, and the gateway ready log only reports plugins whose status is actually `loaded`, so the reported warning/load mismatch does not match current behavior.

## Evidence

- **Unsafe plugin candidates are filtered before loading:** `addCandidate(...)` calls `isUnsafePluginCandidate(...)` and returns early, so a candidate flagged for suspicious ownership never reaches the loader candidate list.
  - file: [src/plugins/discovery.ts:481](https://github.com/openclaw/openclaw/blob/ec8dbc4595583816f71af2568df61dbe7c090e1f/src/plugins/discovery.ts#L481)
  - sha: [ec8dbc459558](https://github.com/openclaw/openclaw/commit/ec8dbc4595583816f71af2568df61dbe7c090e1f)
- **Ownership mismatch is explicitly covered by tests:** The discovery test asserts that a UID mismatch blocks the candidate and emits a `suspicious ownership` diagnostic.
  - file: [src/plugins/discovery.test.ts:1011](https://github.com/openclaw/openclaw/blob/ec8dbc4595583816f71af2568df61dbe7c090e1f/src/plugins/discovery.test.ts#L1011)
  - sha: [ec8dbc459558](https://github.com/openclaw/openclaw/commit/ec8dbc4595583816f71af2568df61dbe7c090e1f)
- **Gateway and CLI runtime loading share the same discovery inputs:** `loadOpenClawPlugins(...)` builds both runtime and CLI-metadata registries from `discoverOpenClawPlugins(...)` plus `loadPluginManifestRegistry(...)`, so they do not have independent ownership policies on current `main`.
  - file: [src/plugins/loader.ts:2127](https://github.com/openclaw/openclaw/blob/ec8dbc4595583816f71af2568df61dbe7c090e1f/src/plugins/loader.ts#L2127)
  - sha: [ec8dbc459558](https://github.com/openclaw/openclaw/commit/ec8dbc4595583816f71af2568df61dbe7c090e1f)
- **CLI config warnings come from the same manifest registry diagnostics:** Config validation converts `loadPluginManifestRegistry(...)` diagnostics into user-facing warnings/errors, including plugin-path findings, instead of using a separate ownership checker.
  - file: [src/config/validation.ts:841](https://github.com/openclaw/openclaw/blob/ec8dbc4595583816f71af2568df61dbe7c090e1f/src/config/validation.ts#L841)
  - sha: [ec8dbc459558](https://github.com/openclaw/openclaw/commit/ec8dbc4595583816f71af2568df61dbe7c090e1f)
- **Gateway ready log only reports actually loaded plugins:** Startup logging filters the registry to `plugin.status === "loaded"` before printing the ready line, so a plugin excluded by discovery would not appear in the reported loaded-plugin list.
  - file: [src/gateway/server-startup-post-attach.ts:448](https://github.com/openclaw/openclaw/blob/ec8dbc4595583816f71af2568df61dbe7c090e1f/src/gateway/server-startup-post-attach.ts#L448)
  - sha: [ec8dbc459558](https://github.com/openclaw/openclaw/commit/ec8dbc4595583816f71af2568df61dbe7c090e1f)
- **Latest release shows the same shared loader wiring:** The latest shipped release `v2026.4.23` already has `loadOpenClawPlugins(...)` building its registry from `discoverOpenClawPlugins(...)` and passing those candidates into `loadPluginManifestRegistry(...)`, matching current `main`.
  - file: [src/plugins/loader.ts:2005](https://github.com/openclaw/openclaw/blob/a9797214338ba31c52c796adbb75afb16e0684a9/src/plugins/loader.ts#L2005)

## Risks / Open Questions

- I did not run the reporter's exact Docker/container setup because this review was read-only and avoided artifact-producing repro steps; the conclusion is based on code, tests, and shipped-path inspection.
- A host CLI running as one UID and a gateway process running inside a container as another UID could still produce different observations across separate processes, but that would be environment drift rather than a current parser-vs-loader policy split in OpenClaw core.
