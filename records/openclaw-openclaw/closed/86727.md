## Summary

This should stay open: current `main` still classifies `exec` and `bash` as mutating before inspecting command arguments, so the reported read-only shell probe warning path remains possible. The linked broader warning-policy issue is related but does not replace this narrower classifier bug.

## What This Changes

The issue requests that clearly read-only shell probes be classified as non-mutating so their failures follow the non-mutating warning-suppression path.

## Best Possible Solution

Add a conservative shell-command read-only classifier in the tool mutation boundary, keep unknown shell commands mutating by default, and cover both classifier output and final-payload suppression behavior for representative read-only and mutating commands.

## Reproduction Assessment

Yes, from source inspection: `isMutatingToolCall("exec", { command: "ls -la" })` currently returns mutating because `exec` and `bash` return `true` before command inspection, and the payload policy surfaces mutating errors. I did not run tests because this review is read-only.

## Solution Assessment

Unclear without maintainer approval on the exact shell allowlist, but the proposed shape is directionally right: recognize only clearly read-only shell probes and keep every unknown or ambiguous command fail-closed as mutating.
