## Summary

Keep open: the PR addresses a real Android/Gateway provider-status mismatch and has sufficient emulator screenshot proof, but it changes shared `models.list` availability semantics and still has a small expiring-warning precedence gap in the command palette, so this needs maintainer review rather than cleanup close.

## What This Changes

The PR adds optional `available` metadata to `models.list`, threads it through generated Swift and Android model summaries, and updates Android provider readiness UI/tests to distinguish usable, expiring, setup-required, and catalog-only providers.

## Best Possible Solution

Land after maintainer approval of the optional `available` contract and after expiring-state labels are consistent across Home, Providers & Models, and the command palette.

## Reproduction Assessment

Yes, source-reproducible: current main computes Home provider readiness from `models.authStatus` status only and emits `No ready providers` when that count is zero, while Android separately loads usable model rows from `models.list`. I did not run a before-state emulator, but the current source and after-fix screenshots make the mismatch clear.

## Solution Assessment

Unclear as a final contract but plausible as a fix: adding optional availability metadata is a maintainable way to avoid Android guessing from auth warnings, but the shared API semantics need maintainer approval and the command-palette warning precedence should be made consistent.
