## What This Changes

WebChat should not show duplicate user or assistant messages after a WebSocket reconnect.

## Best Possible Solution

Keep the current main reconnect queue/idempotency behavior and ship it in the next release; new reports after a build containing 96635c7 should include WebChat logs and reconnect timing.

## Reproduction Assessment

Yes, from source and regression coverage: the v2026.5.27 path sent directly and did not preserve a pre-ACK reconnect retry, while current main queues the send and retries with the same idempotency key. I did not run a live browser repro in this read-only sweep.

## Solution Assessment

Yes. The current fix is the maintainable path because it preserves a single queued WebChat send across reconnect, reuses the same run id, and keeps server-side idempotency as the backstop.
