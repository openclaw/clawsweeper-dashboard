## Summary

Keep open: this PR addresses a real Slack DM thread-context bug and includes a changelog entry plus resolver-level proof, but the patch keys the new direct-thread exception off raw `message.channel_type`, so it misses D-prefix DMs that current main already normalizes as direct when Slack omits or misreports `channel_type`.

## What This Changes

The PR includes current-bot Slack root messages as starter context for IM/MPIM thread replies, adds a regression test, and records the Slack fix in `CHANGELOG.md`.

## Best Possible Solution

Keep the focused Slack fix, but derive the direct/MPIM decision from the normalized channel type already used by the prepare path and add coverage for D-prefix DMs with missing or wrong `channel_type`.

## Reproduction Assessment

Yes. Source inspection on current main shows current-bot Slack thread starters are always omitted, and the linked report gives concrete Slack DM thread steps; I did not run tests because this review is read-only.

## Solution Assessment

No. The requested direction is right, but the implementation should reuse the normalized/resolved Slack channel type instead of raw `message.channel_type` so it covers the direct-message cases current main already recognizes.
