## Best Possible Solution

Keep the string `"null"` workaround focused on `sessionKey`, or explicitly reserve `null` as an invalid agent id with documented compatibility coverage and real cron tool/gateway proof.

Yes by source inspection: current main preserves non-empty string `sessionKey` values through cron tool normalization and cron patch storage, so a tool path that serializes null as `"null"` reaches the gateway as a real key. I did not run a live gateway round trip in this read-only review.

## Solution Assessment

No as written: normalizing the model-produced `sessionKey: "null"` sentinel is the narrow fix, but applying the same sentinel to `agentId` conflicts with the current valid agent-id contract.
