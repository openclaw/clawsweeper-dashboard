## Summary

This request is better scoped as a context-engine plugin/ClawHub package, not OpenClaw core. Current `main` already provides a dedicated plugin surface for custom cross-session context assembly, compaction, bootstrap, and post-turn persistence, which matches the requested hibernation/restore behavior.

## Evidence

- **Core scope prefers plugins for optional capability:** `VISION.md` says core should stay lean, optional capability should usually ship as plugins, and plugin authors should host and maintain those plugins outside core.
  - file: [VISION.md:54](https://github.com/openclaw/openclaw/blob/4d7c4b3298920c0e6e2c5cc8bd7a1546e8122abe/VISION.md#L54)
  - sha: [4d7c4b329892](https://github.com/openclaw/openclaw/commit/4d7c4b3298920c0e6e2c5cc8bd7a1546e8122abe)
- **Docs route cross-session recall to context-engine plugins:** `docs/concepts/context-engine.md` explicitly says to install/select a plugin engine when you want different assembly, compaction, or cross-session recall behavior, and documents plugin installation/config for a `lossless-claw`-style engine.
  - file: [docs/concepts/context-engine.md:14](https://github.com/openclaw/openclaw/blob/4d7c4b3298920c0e6e2c5cc8bd7a1546e8122abe/docs/concepts/context-engine.md#L14)
  - sha: [4d7c4b329892](https://github.com/openclaw/openclaw/commit/4d7c4b3298920c0e6e2c5cc8bd7a1546e8122abe)
- **Public plugin API exists for this feature area:** `src/plugins/types.ts` exposes `registerContextEngine`, and the architecture docs state that context-engine plugins own session context orchestration for ingest, assembly, and compaction.
  - file: [src/plugins/types.ts:2010](https://github.com/openclaw/openclaw/blob/4d7c4b3298920c0e6e2c5cc8bd7a1546e8122abe/src/plugins/types.ts#L2010)
  - sha: [4d7c4b329892](https://github.com/openclaw/openclaw/commit/4d7c4b3298920c0e6e2c5cc8bd7a1546e8122abe)
- **Context-engine contract supports serialize/restore lifecycle:** `src/context-engine/types.ts` gives plugins `bootstrap(sessionFile)`, `afterTurn(sessionFile, messages, prePromptMessageCount, ...)`, `assemble(...)`, and `compact(sessionFile, ...)`, which are the hooks needed to persist session snapshots and restore them on later runs.
  - file: [src/context-engine/types.ts:171](https://github.com/openclaw/openclaw/blob/4d7c4b3298920c0e6e2c5cc8bd7a1546e8122abe/src/context-engine/types.ts#L171)
  - sha: [4d7c4b329892](https://github.com/openclaw/openclaw/commit/4d7c4b3298920c0e6e2c5cc8bd7a1546e8122abe)
- **Harness actually invokes those hooks:** `src/agents/harness/context-engine-lifecycle.ts` calls plugin `bootstrap`, `assemble`, and `afterTurn`/maintenance with `sessionFile` and message snapshots, confirming the plugin seam is live on current `main`.
  - file: [src/agents/harness/context-engine-lifecycle.ts:20](https://github.com/openclaw/openclaw/blob/4d7c4b3298920c0e6e2c5cc8bd7a1546e8122abe/src/agents/harness/context-engine-lifecycle.ts#L20)
  - sha: [4d7c4b329892](https://github.com/openclaw/openclaw/commit/4d7c4b3298920c0e6e2c5cc8bd7a1546e8122abe)
- **No built-in core flag by the requested name:** A repo search for `session.hibernation`, `restore-from`, and `hibernation` found no built-in config/CLI surface matching the requested core feature, so this is not already implemented in core.
  - command: `rg -n "session\.hibernation|restore-from|hibernation" src docs packages extensions -S`
  - sha: [4d7c4b329892](https://github.com/openclaw/openclaw/commit/4d7c4b3298920c0e6e2c5cc8bd7a1546e8122abe)

## Risks / Open Questions

- A plugin implementation would still need to handle backend-specific limits, especially where the Codex harness keeps its own native thread history/compactor (`docs/concepts/context-engine.md:80`).
- There is no proof in this repo of a shipped community plugin that already delivers this exact hibernation UX; the evidence supports scope, not an existing packaged solution.
