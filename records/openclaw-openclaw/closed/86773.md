## Label Justifications

## Summary

Close: current main and shipped releases now address the central failure by running provider-auth prewarm off the Gateway main thread, so the expensive warm path no longer monopolizes local RPC handling after restart.

## What This Changes

The issue asks for provider-auth startup prewarm to stop starving the Gateway event loop so local RPCs like sessions.list remain responsive after restart.

## Best Possible Solution

Keep the shipped off-main-thread provider-auth prewarm and route any broader startup or provider performance regressions to narrower current issues with fresh logs.

## Reproduction Assessment

No current-main failing reproduction remains. The original v2026.5.22 logs and related reports made the bug clear, and current source plus release history show the warm path moved off the main thread.

## Solution Assessment

Yes. Moving the expensive warm snapshot to a worker with cancellation and lifecycle coverage is the narrow maintainable fix for event-loop starvation; default-disable or operator-toggle PRs are optional mitigations, not required to close this bug.
