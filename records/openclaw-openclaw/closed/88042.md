## Label Justifications

- `P1`: The report describes a broken subagent-to-Telegram completion workflow on the latest release where users can miss required completion output after sessions_yield.
- `impact:message-loss`: The affected behavior is lost or undelivered subagent completion announcements.
- `impact:session-state`: The failure depends on yielded requester-session state and suspended completion-delivery state.

## Summary

Keep open: current main still does not provide a duplicate-safe recovery path that delivers a subagent completion after the requester/session delivery adapter is unavailable during the yield handoff. The central failure is not covered by the closed registry-mismatch issue; current code preserves some pending state after retry exhaustion, but it skips suspended deliveries rather than replaying them.

## What This Changes

The issue reports lost Telegram subagent completion announcements after a parent uses `sessions_yield` and the announce retry loop exhausts while the outbound adapter is unavailable.

## Best Possible Solution

Keep this open and design a durable, idempotent completion-delivery recovery path that either replays the preserved subagent result when the requester/channel adapter becomes available or exposes an explicit recovery path without risking duplicate sends.

## Reproduction Assessment

Yes. The issue gives a concrete Telegram polling scenario and logs, and current source shows the same short retry and suspended-delivery skip behavior; I did not run live Telegram proof in this read-only review.

## Solution Assessment

Unclear. Increasing retry count alone is probably not the best fix because current recovery code is intentionally duplicate-averse; the safer direction is a durable, idempotent replay or explicit recovery design.
