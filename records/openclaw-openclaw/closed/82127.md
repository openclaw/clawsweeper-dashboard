## Summary

Current main already addresses the reported lifecycle leak: silent `NO_REPLY` inter-session replies are non-deliverable, and embedded agent runs now emit a terminal lifecycle event even when the runtime returns without one, so downstream session/subagent cleanup is no longer left waiting.

## What This Changes

The issue reports that a `sessions_send` inter-session task can complete with `NO_REPLY` while the receiving session remains `running` until a much later auto-reset.

## Best Possible Solution

Ship the current main lifecycle backstop and keep `NO_REPLY` handling in the generic `sessions_send`/runner path rather than adding Telegram-specific cleanup.

## Reproduction Assessment

No current-main failing repro was established. The report has a concrete historical v2026.5.7 timeline, and source/history show the relevant `NO_REPLY` and lifecycle paths are now fixed on current main.

## Solution Assessment

Yes. The current implementation solves the root lifecycle problem at the shared runner and `sessions_send` boundaries, which is narrower and more maintainable than channel-specific cleanup.
