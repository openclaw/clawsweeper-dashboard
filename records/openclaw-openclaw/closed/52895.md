## Summary

Current main rejects non-string `mentionPatterns` at config validation time, so I could not verify a supported runtime path where this PR's crash case reaches mention gating. The PR also links an already-closed unrelated Slack issue, and its review thread identified concrete mention-gating regressions in the proposed fix.

## Evidence

- **Current mention code still expects strings:** `normalizeMentionPattern()` calls `pattern.includes(...)`, `normalizeMentionPatterns()` maps over `string[]`, and `buildMentionRegexes()` feeds it directly from `resolveMentionPatterns()`. That means the PR is defending an unvalidated input shape rather than changing current supported behavior.
  - file: [src/auto-reply/reply/mentions.ts:49](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/auto-reply/reply/mentions.ts#L49)
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Schema forbids non-string mention patterns:** `messages.groupChat.mentionPatterns` is defined as `z.array(z.string()).optional()`, so non-string entries are invalid config, not accepted runtime data.
  - file: [src/config/zod-schema.core.ts:383](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/config/zod-schema.core.ts#L383)
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Validation rejects invalid configs before runtime:** Config validation runs `OpenClawSchema.safeParse(...)`, and config loading throws on validation failure via `throwInvalidConfig(...)`. In supported config loads, malformed `mentionPatterns` do not reach inbound mention gating.
  - file: [src/config/validation.ts:646](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/config/validation.ts#L646)
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Load path is fail-fast on invalid config:** `loadConfig()` validates the resolved config and aborts when validation fails, rather than continuing with a malformed runtime config.
  - file: [src/config/io.ts:1265](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/config/io.ts#L1265)
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Proposed fix conflicts with current mention-gating semantics:** Current iMessage gating only enforces `requireMention` when `mentionRegexes.length > 0`, and the existing test suite explicitly allows group messages when `mentionPatterns` is empty. That matches the PR review comments warning that filtered-to-empty or rejected overrides would change behavior in unsafe ways.
  - file: [extensions/imessage/src/monitor/inbound-processing.ts:417](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/extensions/imessage/src/monitor/inbound-processing.ts#L417)
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Linked issue is unrelated and already closed:** The PR body says `Closes #50060`, but issue #50060 is a Slack undefined-text crash report and is already closed as implemented. That issue does not establish an open current-main bug in mention-pattern normalization.
  - command: `curl -fsSL https://api.github.com/repos/openclaw/openclaw/issues/50060`
