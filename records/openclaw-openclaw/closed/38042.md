## Summary

Keep open. Current `main` does not implement this feature: OpenClaw still only exposes built-in/custom provider flows for known transports and OpenAI/Anthropic-compatible endpoints, so this PR is not obsolete. There is also clear plugin-first architecture guidance for fully custom transports, which means the work likely needs maintainer/product direction on whether it belongs in core or should be reshaped as a provider plugin rather than closed as already handled.

## Evidence

- **No `local-server` transport on current main:** `MODEL_APIS` on `main` enumerates fixed transports including `openai-completions`, `openai-responses`, `anthropic-messages`, `google-generative-ai`, `ollama`, and `azure-openai-responses`; there is no `local-server` entry.
  - file: [src/config/types.models.ts:9](https://github.com/openclaw/openclaw/blob/627c19c5cbbe8856a11effe1871c442ac87d56c9/src/config/types.models.ts#L9)
  - sha: [627c19c5cbbe](https://github.com/openclaw/openclaw/commit/627c19c5cbbe8856a11effe1871c442ac87d56c9)
- **Current custom onboarding only handles OpenAI/Anthropic-compatible endpoints:** The custom-provider wizard offers only `openai`, `anthropic`, or `unknown` compatibility, and the detection path explicitly tells users the endpoint did not respond to OpenAI or Anthropic style requests when neither probe works.
  - file: [src/commands/onboard-custom.ts:43](https://github.com/openclaw/openclaw/blob/627c19c5cbbe8856a11effe1871c442ac87d56c9/src/commands/onboard-custom.ts#L43)
  - sha: [627c19c5cbbe](https://github.com/openclaw/openclaw/commit/627c19c5cbbe8856a11effe1871c442ac87d56c9)
- **User docs still describe custom/unlisted providers as compatibility-only:** Current onboarding docs say unlisted providers should use Custom Provider with an API compatibility mode of OpenAI-compatible, Anthropic-compatible, or auto-detect; the wizard docs say the same.
  - file: [docs/start/onboarding-overview.md:58](https://github.com/openclaw/openclaw/blob/627c19c5cbbe8856a11effe1871c442ac87d56c9/docs/start/onboarding-overview.md#L58)
  - sha: [627c19c5cbbe](https://github.com/openclaw/openclaw/commit/627c19c5cbbe8856a11effe1871c442ac87d56c9)
- **Architecture points toward plugin-owned custom transports:** Current docs say most provider-specific logic lives in provider plugins and that a provider needing a totally custom request executor is a deeper extension surface; the provider SDK also exposes `createStreamFn` for a fully custom transport, while `VISION.md` says optional capability should usually ship as plugins.
  - file: [docs/concepts/model-providers.md:30](https://github.com/openclaw/openclaw/blob/627c19c5cbbe8856a11effe1871c442ac87d56c9/docs/concepts/model-providers.md#L30)

## Risks / Open Questions

- Closing this PR would discard a feature that is still absent from current `main` and still requested by the gap in current custom-provider docs/onboarding.
- Merging as-is may still be the wrong shape: current architecture suggests a provider-plugin path for fully custom transports, so owner review is needed before deciding between core support and pluginization.
