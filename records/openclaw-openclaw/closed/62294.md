## Summary

Close as implemented on current main: the heartbeat runner now uses a shared cooldown/flood gate for event-driven wakes, generated macOS launchd plists set `ProcessType=Interactive`, and the final App Nap half landed after the latest release so the full fix is current-main-only.

## What This Changes

The issue asks OpenClaw to prevent heartbeat burst runs from non-interval wake reasons and macOS idle gaps from App Nap timer suspension.

## Best Possible Solution

Keep the shipped typed wake-intent cooldown design and the generated launchd `ProcessType=Interactive` key; future changes to explicit wake-now semantics should be tracked separately.

## Reproduction Assessment

Yes for the historical bug: related reports and tests reproduce repeated targeted `exec-event` wakes inside `nextDueMs`. On current main, that path is no longer a failing reproduction because both targeted and broadcast event wakes pass through the shared cooldown gate.

## Solution Assessment

Yes. The current implementation is narrower than the original blanket throttle proposal: loop-prone event wakes respect cooldown, while explicit manual/immediate wake-now paths keep their documented responsiveness with a flood backstop.
