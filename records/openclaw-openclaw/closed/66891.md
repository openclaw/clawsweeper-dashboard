## Summary

Current main already preserves Discord interactive/component reply payloads through the shared outbound delivery path. The old `deliverDiscordReply` no longer performs the plain text/media-only send directly; it delegates to shared outbound delivery, which keeps `interactive` and `channelData`, calls the Discord adapter's `sendPayload`, and the adapter resolves both `channelData.discord.components` and generic `interactive` into Discord component sends. The implementation is also present in release `v2026.4.24`.

## Best Possible Solution

Close this stale PR and keep the shipped shared outbound Discord implementation on main. Any narrower follow-up, such as changing behavior for forum-parent fallback or webhook-persona component sends, should be filed as a separate focused issue or PR against the current outbound adapter path.
