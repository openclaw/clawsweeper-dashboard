## Summary

Keep this PR open for maintainer compatibility acceptance. The patch is a focused Ollama provider fix with sufficient contributor terminal proof and no concrete correctness finding, but it intentionally changes hosted `models.providers.ollama` discovery behavior for existing configs that relied on automatic `/api/tags` catalog population.

## What This Changes

The PR changes the bundled Ollama plugin so hosted `*.ollama.com` configs without explicit provider model rows skip `/api/tags` discovery, adds focused discovery tests, widens the Ollama provider config input type, and updates Knip dependency metadata.

## Best Possible Solution

Land only after maintainers accept that hosted `models.providers.ollama` configs now require explicit models, while `ollama-cloud` remains the first-class hosted live-catalog path and local/self-hosted discovery stays preserved.

## Reproduction Assessment

Yes, source-level: current main reaches `buildProvider` for configured hosted Ollama base URLs, and the live hosted `/api/tags` endpoint is reachable. I did not run an authenticated OpenClaw model-list flow in this read-only review.

## Solution Assessment

Yes, with maintainer acceptance of the compatibility tradeoff: the guard is hosted-Ollama-specific, keeps local and remote self-hosted discovery, and preserves explicit model rows. The safer alternative is to require an upgrade note or design an explicit compatibility path before merge.
