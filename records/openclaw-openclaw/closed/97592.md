## What This Changes

Request to add a macOS Voice Chat mode that loops speech input, agent response, TTS playback, and automatic mic restart for hands-free conversation.

## Best Possible Solution

Keep the older macOS realtime Talk issue as the canonical tracker, preserve the shipped native STT/chat/TTS Talk fallback, and avoid adding a parallel voice_chat config/API surface unless maintainers sponsor that direction.

## Reproduction Assessment

Yes for the remaining feature gap by source inspection: macOS Talk uses local speech recognition plus chat.send plus TTS playback, not a native realtime Talk session. The narrower text-only premise is not reproduced on current main because TTS playback already exists.

## Solution Assessment

No for adding a separate voice_chat config and session stack as proposed. The maintainable path is to reuse the existing Talk surface and the older macOS realtime issue's direction to consume current realtime Talk contracts with the current TTS path as fallback.
