## Summary

Keep open: current main and the latest release still parse signed Nextcloud Talk webhooks through the strict chat-message Note schema before classifying valid non-message Activity Streams events, so the reported 400 behavior remains. The two direct fix PRs are closed unmerged, and the currently open Nextcloud Talk PRs are adjacent payload-parsing work rather than a viable canonical fix for this exact acknowledgement bug.

## What This Changes

The issue requests that signed non-message Nextcloud Talk webhooks, such as file-share events, be acknowledged without dispatch instead of returning HTTP 400 Invalid payload format.

## Best Possible Solution

Land a narrow monitor-level classifier that parses signed JSON before strict Note validation, returns 200 with no dispatch for valid non-message Talk events, and preserves 400 for malformed JSON or malformed message-shaped payloads.

## Reproduction Assessment

Yes, source-level: a signed JSON payload with a non-Note object fails `NextcloudTalkWebhookPayloadSchema`, and `decodeWebhookCreateMessage` writes HTTP 400 before any ignore branch can run. I did not run a live Nextcloud server in this read-only review.

## Solution Assessment

Yes, the narrowest maintainable fix is to classify signed JSON before strict message validation, acknowledge valid non-message Activity Streams events, and keep malformed JSON or partial message-shaped payloads on the existing 400 path.
