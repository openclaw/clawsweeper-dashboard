## Summary

Keep open: current main and the latest stable tag still return HTTP 400 for signed Nextcloud Talk webhook payloads that are valid Activity Streams events but not the strict chat-message shape. The focused fix PR for this exact contract was closed unmerged, while the still-open rich-object file-share PR covers a neighboring parser gap rather than this non-message 400 path.

## What This Changes

The issue asks the Nextcloud Talk webhook handler to acknowledge signed non-message Talk events instead of returning 400 Invalid payload format, while preserving 400 responses for malformed requests.

## Best Possible Solution

Update the Nextcloud Talk webhook classifier so signed JSON is parsed before strict message validation, malformed JSON and malformed message-shaped payloads still return 400, and signed non-message Activity Streams events return 200/empty without invoking the message handler.

## Reproduction Assessment

Yes, at source level. A signed JSON Activity Streams payload outside the strict Create/Person/Note/Collection schema reaches decodeWebhookCreateMessage and returns HTTP 400 Invalid payload format on current main; I did not run a live Nextcloud server in this read-only pass.

## Solution Assessment

Yes. The narrow maintainable fix is to distinguish malformed JSON and malformed supported message payloads from signed non-message Talk events after signature verification; the closed related PR followed that shape but did not merge.
