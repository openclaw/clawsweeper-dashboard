## Summary

Current `main` already implements the requested Bailian/DashScope token-usage handling for generic OpenAI-compatible provider ids. The report's proposed `prompt_tokens`/`completion_tokens` mapping root cause is not current-main reality; OpenClaw already normalizes those fields, and current routing/compat logic preserves streaming usage on DashScope endpoints even when configured under a non-`qwen` provider key like `bailian`.
