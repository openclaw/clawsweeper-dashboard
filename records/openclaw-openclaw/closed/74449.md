## Summary

Close as implemented on main. The reported v2026.4.23-v2026.4.26 Control UI token/device-token mismatch loop maps to the stale device-token reconnect bug fixed by current main and shipped in v2026.4.27, where browser and Node Gateway clients clear reused stored device tokens and stop reconnecting on AUTH_DEVICE_TOKEN_MISMATCH instead of rate-limit looping.

## What This Changes

The issue reports that the Control UI WebSocket connect is rejected with token_mismatch/device_token_mismatch after device approval on Ubuntu Kylin 20.04 with OpenClaw v2026.4.23 through v2026.4.26.

## Best Possible Solution

Keep the shipped v2026.4.27 stale device-token cleanup and no-reconnect behavior. If a user can still reproduce on v2026.4.27 or newer with fresh dashboard credentials, handle that as a new focused report with redacted auth-source comparison and separate any provider reply timeout from the WebSocket auth failure.

## Reproduction Assessment

Yes. The issue gives concrete versions and auth detail codes, and related #71609 provides the same high-confidence stale Control UI device-token mismatch loop; current tests now cover the AUTH_DEVICE_TOKEN_MISMATCH cleanup/no-reconnect path rather than requiring a live Kylin run for this cleanup decision.

## Solution Assessment

Yes. Clearing reused stale device tokens and treating AUTH_DEVICE_TOKEN_MISMATCH as non-recoverable in the clients is the narrow maintainable fix because it recovers from stale browser/client state without weakening Gateway authentication or accepting mismatched tokens.
