## Summary

Keep open. The report is a concrete current-release Gateway availability regression, and current main does not contain a source change after v2026.4.29 that would distinguish the relevant runtime-deps/Telegram paths from the reported build. The remaining failure is plausible in code because per-plugin bundled runtime-deps staging can still run a package-manager install on the Gateway process path, which matches the event-loop delay, WebSocket handshake timeout, and later high-CPU pnpm child evidence.

## What This Changes

The issue asks to fix a v2026.4.29 Linux npm-global Gateway hang where Telegram starts, then local RPC/status calls time out while a bundled runtime-deps package-manager child consumes CPU and memory.

## Best Possible Solution

Make bundled runtime-deps staging converge before the Gateway is considered ready, and prevent any late package-manager install from blocking the Gateway event loop after readiness. The fix should preserve external stage roots, root-owned npm global installs, and the current plugin boundary while adding regression coverage for a packaged Telegram-enabled Gateway path where installed deps are already materialized.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for a live reproduction: the report gives exact v2026.4.29 install steps, environment, logs, process snapshots, and the later pnpm child command. A deterministic local test still needs to be added around late runtime-deps staging/event-loop blocking.

## Solution Assessment

Is this the best way to solve the issue? Unclear because there is no proposed patch yet. The likely best direction is to fix runtime-deps staging/deduplication or move late installs off the Gateway event loop, not to mask the symptom by increasing RPC or WebSocket handshake timeouts.
