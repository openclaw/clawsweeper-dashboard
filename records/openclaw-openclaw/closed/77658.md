## Summary

Keep open: this is a protected maintainer/MEMBER PR, and the current patch still has source-backed correctness issues in the long-final preview path that should be addressed before merge.

## What This Changes

The PR updates Telegram final text delivery to split oversized final answers through the active preview, adds Telegram live QA/mock coverage and docs/changelog, and treats new diagnostics-otel recovery events as no-ops.

## Best Possible Solution

Keep the preview-reuse approach, but make it eligible only when the active preview may be finalized in place, and make remainder delivery success determine whether the full final is reported as delivered.

## Reproduction Assessment

Yes. Current main and the PR branch show a source-level path: oversized Telegram finals fall back to standard send on main, while the PR branch routes them through preview reuse even when the fresh-final guard would apply.

## Solution Assessment

No. Reusing the preview is the narrow direction for the transient-bubble bug, but this implementation needs to preserve the existing stale/overwritten-preview fallback and account for failed follow-up chunk delivery.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve fresh-final fallback for stale previews:** `extensions/telegram/src/lane-delivery-text-deliverer.ts:678-684`
  - body: This oversized-final branch runs before the normal send/cleanup fallback, but `tryDeliverLongFinalThroughPreview` never checks `shouldUseFreshFinalForLane`. For previews older than about a minute or previews followed by visible non-preview output, current behavior sends a fresh final and clears the preview; this path instead edits and retains the old preview as the first chunk, regressing the documented timestamp/order behavior. Bail out to the existing fresh-final path when that guard is true.
  - confidence: 0.88
- **[P2] Handle failed long-final remainder sends:** `extensions/telegram/src/lane-delivery-text-deliverer.ts:404-405`
  - body: After the preview is edited to `firstChunk`, the remainder delivery result is ignored. If `sendPayload` returns false because the dispatch was superseded or a hook cancels delivery, this still returns `preview-finalized` for the full text and emits success for content the user never received. Check the boolean and avoid reporting the full final as delivered when the follow-up chunks are not sent.
  - confidence: 0.78
