## What This Changes

The issue reports that chat.abort can return aborted:false when it is called immediately after an attachment-heavy chat.send with the same run id.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84306 if the early-registration and cleanup behavior still checks out; then close this issue as implemented after merge.

## Reproduction Assessment

Yes, at source level: current main awaits attachment/model preparation before inserting the run into chatAbortControllers, and chat.abort returns aborted:false when that entry is absent. I did not run a live Gateway repro in this read-only review.

## Solution Assessment

Yes: the narrow fix is to register the chat-send abort controller after validation/policy/dedupe checks but before awaited attachment preprocessing, with cleanup on pre-dispatch failure; the open closing PR already implements that direction with proof.
