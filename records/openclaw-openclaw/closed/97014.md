## Summary

Current main already fixes the reported Telegram `/status` rich-message email failure by disabling Telegram rich entity detection when rich payload text contains email-like tokens. The fix is main-only; v2026.6.10 does not contain it yet.

## What This Changes

The issue reports that Telegram `/status` drops its reply when rich messages are enabled and an OAuth profile email is rendered into invalid Telegram rich HTML.

## Best Possible Solution

Keep the central rich-message email-token guard on main and let the next release carry it; broader plain-text fallback hardening can be reviewed separately.

## Reproduction Assessment

Yes. The affected released path is source-reproducible from the report and code: `/status` can include OAuth profile labels, Telegram rich delivery used `sendRichMessage`, and v2026.6.10 lacked the email-token guard; current main now contains the guard.

## Solution Assessment

Yes. The shared rich-message builder guard is the narrow maintainable fix for this failure because direct sends, slash replies, and draft/progress sends all use that boundary instead of needing a `/status`-specific opt-out.
