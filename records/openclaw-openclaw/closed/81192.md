## Summary

Keep open: this is an active protected-label feature PR, and the diff has blocking behavior bugs around applying migration against stale onboarding config and re-offering migration on already-installed Codex repairs.

## What This Changes

The PR adds a generic post-install migration helper and wires Codex auth-choice onboarding paths to offer or hint at `openclaw migrate codex`, with Codex app-server subprocess cleanup changes.

## Best Possible Solution

Keep the feature on the PR branch, but make the onboarding caller persist and re-read config around accepted migrations and pass a true fresh-install signal before the migration helper sees a plugin id.

## Reproduction Assessment

Yes, from source inspection. The existing Codex install helper returns `installed: true` for already-on-disk repair, and the PR passes that value into a helper documented for just-installed plugin ids; the accepted migration path also writes config through the migration runtime while the caller keeps returning its earlier `nextConfig`.

## Solution Assessment

No. The direction is useful, but the implementation should flush and re-read config around `migrateDefaultCommand` and gate prompts on a newly-installed transition rather than the broad `installed` flag.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Sync config before applying the migration:** `src/plugins/provider-auth-choice.ts:175-180`
  - body: When the user accepts this new prompt, `offerPostInstallMigrations` can call `migrateDefaultCommand`, whose Codex migration path writes plugin config through the runtime config API. This caller then returns the pre-migration `nextConfig` to setup, so the later setup write can overwrite the migrated Codex plugin config. Flush the pending onboarding config before invoking migration and re-read or return the post-migration config.
  - confidence: 0.88
- **[P2] Gate the offer on a fresh install:** `src/plugins/provider-auth-choice.ts:167-180`
  - body: `installed` is also true when the Codex harness was already on disk and this run only repaired/enabled it, so rerunning auth-choice setup can resolve migration providers and show the prompt again. Add the promised fresh-install result and pass `installedPluginIds` only when Codex actually transitioned to installed; the non-interactive call site has the same pattern.
  - confidence: 0.91
