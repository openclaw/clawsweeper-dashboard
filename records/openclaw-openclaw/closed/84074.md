## Label Justifications

- `P2`: The report concerns plugin hook correlation for bundled MCP tools, which is important but limited to a specific runtime path.
- `impact:session-state`: The reported failure mode is missing session/run context in hook execution.

## Summary

Current main and the latest release already thread the shared HookContext through bundled MCP tool execution, including the session/run fields named in the report, so this is safe to close as already implemented.

## What This Changes

Issue reports that bundled MCP tools reach `before_tool_call` hooks without the same session/run context provided to ordinary OpenClaw tools.

## Best Possible Solution

Keep the shared HookContext forwarding path in the runner, tool-search catalog, and adapter rather than adding a separate bundled-MCP-only hook wrapper.

## Reproduction Assessment

No: the reported missing-context path does not match current main, where bundled MCP tools flow through `toToolDefinitions(..., hookContext)` or catalog wrapping. I did not run a live plugin repro because this read-only review cannot execute tests that may write artifacts, but the source path is explicit.

## Solution Assessment

Yes: the current shared adapter/catalog solution is the narrow maintainable fix because it covers bundled MCP and other unwrapped/cataloged tools without duplicating MCP-specific hook plumbing.
