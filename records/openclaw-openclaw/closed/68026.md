## Summary

Current `main` already handles the reported `read`/`web_fetch` parameter-validation failures as normal tool error results instead of letting them escape into an outer retry loop, and the same behavior is present in the shipped `v2026.4.23` release with regression coverage.

## Evidence

- **Tool adapter converts thrown tool failures into a structured error result:** `toToolDefinitions()` catches non-abort tool exceptions and returns `jsonResult({ status: "error", tool, error })` instead of rethrowing, which prevents ordinary tool validation/runtime errors from propagating as retryable outer-run failures.
  - file: [src/agents/pi-tool-definition-adapter.ts:137](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/src/agents/pi-tool-definition-adapter.ts#L137)
  - command: `nl -ba src/agents/pi-tool-definition-adapter.ts | sed -n '137,272p'`
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **`read` validates required params before execution:** `createOpenClawReadTool()` calls `assertRequiredParams(record, REQUIRED_PARAM_GROUPS.read, base.name)` before invoking the underlying read logic, so missing `path` is handled as parameter validation, not an untyped crash.
  - file: [src/agents/pi-tools.read.ts:680](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/src/agents/pi-tools.read.ts#L680)
