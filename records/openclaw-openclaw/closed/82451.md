## Summary

Current main addresses the reported Telegram multi-account `start-account` fan-out with an abort-aware startup `getMe` probe limiter and regression coverage; the related `getUpdates` liveness fix is in the v2026.5.12 changelog, while the more specific startup-probe limiter is current-main-only.

## What This Changes

The issue asks maintainers to investigate Gateway event-loop starvation during Telegram multi-account startup on `openclaw@2026.5.7` and clarify release or configuration mitigations.

## Best Possible Solution

Keep the current Telegram startup probe limiter and ship it in the next stable release; use `pollingStallThresholdMs` only for false-positive long-poll watchdog restarts.

## Reproduction Assessment

No, I do not have a high-confidence current-main runtime reproduction of the long-lived multi-bot host. The report’s logs are detailed, and current source/tests cover the implicated startup `getMe` fan-out path that was missing from the reported release.

## Solution Assessment

Yes. Bounding Telegram startup `getMe` fan-out inside the Telegram plugin’s `startAccount` path is the narrowest maintainable fix, while the existing polling watchdog remains scoped to `getUpdates` liveness.
