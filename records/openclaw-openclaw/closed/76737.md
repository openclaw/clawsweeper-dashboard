## Summary

Close: current main and the shipped v2026.5.2 tree already use the JSON short-term recall store and gate promotion on total short-term signals, so a missing SQLite `short_term_recalls` table and `recallCount: 0` are not the blocking condition described.

## What This Changes

The issue reports dreaming deep promotion selecting zero candidates because `recallCount` stays zero and a `short_term_recalls` SQLite table is missing.

## Best Possible Solution

Keep the existing JSON evidence store and total-signal promotion gate; workspace-specific stuck promotions should be reported with redacted `memory status --deep --json` and `.dreams` counter data.

## Reproduction Assessment

No high-confidence current-main failure was reproduced. Source, docs, and regression coverage show current promotion does not require a SQLite `short_term_recalls` table and can rank candidates with `recallCount: 0` when daily or grounded signals accumulate.

## Solution Assessment

Yes. The shipped/current design is the narrower maintainable solution because it keeps dreaming-owned machine state in `memory/.dreams/` and avoids adding a duplicate SQLite recall table.
