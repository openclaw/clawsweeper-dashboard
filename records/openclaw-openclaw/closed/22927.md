## Summary

Issue #22927 is a useful idea, but current main positions alternative memory backends as memory plugins rather than new built-in `memorySearch.store.driver` values in `memory-core`. The repo already exposes a dedicated memory plugin slot and SDK surfaces for custom memory implementations, while the built-in `memorySearch` schema and storage path remain explicitly SQLite-only.

## Risks / Open Questions

- A pgvector backend may still need additional SDK seams if a third-party memory plugin cannot reuse enough of `memory-core` today.
- Closing as scope may disappoint users who expected a small additive config change inside `memory-core` rather than a separate memory plugin.

## Close Comment

Thanks for the detailed writeup. I reviewed current `main` in Codex, and this looks better scoped as a separate memory plugin rather than a new built-in `memorySearch.store.driver` inside `memory-core`.

Concrete repo evidence:
- `VISION.md:55` says optional capability should usually ship as plugins, and `VISION.md:63` says memory is a special plugin slot.
- `docs/tools/plugin.md:109` documents alternative memory backends through `plugins.slots.memory` (`memory-core`, `memory-lancedb`).
- `docs/plugins/manifest.md:684` documents `kind: "memory"` as an exclusive plugin kind selected via `plugins.slots.memory`.
- `docs/plugins/sdk-subpaths.md:236` exposes memory host/storage SDK seams for plugin implementations.
- At the same time, current built-in config is still explicitly SQLite-only: `src/config/zod-schema.agent-runtime.ts:691` and `src/config/schema.base.generated.ts:6095` constrain `store.driver` to `"sqlite"`, and `extensions/memory-core/src/memory/manager-sync-ops.ts:265` uses SQLite-specific `vec0` tables.

Given that architecture, I’m closing this as a plugin-scope request rather than a core cleanup item. A pgvector/Postgres backend still seems useful, but the path on current main is a dedicated memory plugin or ClawHub-distributed plugin, not expanding `memory-core`’s built-in driver matrix.
