## Summary

Current main and release v2026.5.12 already ship the WhatsApp plugin on Baileys 7.0.0-rc11, whose upstream package contains the unified-session QR-auth behavior that superseded the old rc9 backport path. The previously paired fix PR was closed as obsolete after that dependency path landed.

## What This Changes

The issue reports WhatsApp QR pairing silently stalling after scan on OpenClaw 2026.3.x Docker images using Baileys rc9.

## Best Possible Solution

Keep the shipped Baileys rc11 WhatsApp plugin dependency path; any new QR failure should be filed as a fresh current-version report with logs and environment details.

## Reproduction Assessment

Yes for the reported 2026.3.13/rc9 path: the issue gives concrete Docker/dashboard QR scan steps and a side-by-side whatsapp-web.js comparison. I did not reproduce a live failure on current main because current main and v2026.5.12 no longer contain the rc9 dependency path.

## Solution Assessment

Yes. Using Baileys rc11 is the narrow maintainable fix because it carries the upstream unified-session handshake behavior directly, while the old rc9 patch PR is now obsolete.
