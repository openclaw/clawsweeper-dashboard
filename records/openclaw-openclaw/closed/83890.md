## Summary

Keep open: current main and the latest release still build a raw `ProcessStartInfo.Arguments` string in the generated Windows restart helper. The current task-name path is guarded, so this is bounded Windows restart hardening rather than a proven live exploit path, but the source-level quoting gap remains real enough for maintainer security follow-up.

## What This Changes

The issue requests hardening the Windows restart helper so `ProcessStartInfo` arguments cannot be malformed by embedded double quotes.

## Best Possible Solution

Keep the safe task-name gate and harden the generated PowerShell helper with either proven-compatible `ArgumentList` use or correct raw Windows argument escaping, plus focused regression coverage for embedded quotes and the existing guard.

## Reproduction Assessment

No high-confidence current-user exploit path is reproduced. Source inspection does show the helper would mishandle embedded double quotes if an unguarded future argument reached it, while today the only variable task-name argument is filtered first.

## Solution Assessment

Unclear until maintainers choose the compatibility target. Escaping embedded quotes in the existing raw string is likely the narrowest Windows PowerShell-compatible repair; `ArgumentList` is cleaner if runtime support is proven.
