## Summary

Keep open: the PR is pointed at the right bug, but the current diff can still leave a reset-triggered turn waiting when an active reply run exists without session-lane depth, and it is missing required landing polish.

## What This Changes

The PR adds reset-triggered active-run queue handling for `/new` and `/reset`, suppresses steer/followup handling for reset turns, expands interrupt cleanup, and adds queue-policy tests.

## Best Possible Solution

Land an updated version that carries reset-triggered interrupt intent through both active-run policy and prepared queue handling, independent of session-lane depth, while preserving heartbeat drops and normal non-reset queue behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a same-session active reply operation with `messages.queue.mode: steer`, `resetTriggered: true`, and zero session-lane depth; the PR can choose `run-now` but still wait because prepared queue handling sees the original queue mode.

## Solution Assessment

No. The direction is right, but reset-triggered turns need an effective interrupt path in prepared queue handling or equivalent abort intent before waiting, plus the missing regression coverage and changelog entry.
