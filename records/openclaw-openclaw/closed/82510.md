## Summary

Close as implemented on current main: native Ollama now has a supported `params.num_ctx` path, `openclaw doctor --fix` migrates older explicit native Ollama context budgets, and the docs explain why `lightContext` is not the control for Ollama's 4096 runtime default. The fix is not in stable `v2026.5.12`, but it is included in `v2026.5.14-beta.2` and current main.

## What This Changes

The issue reports that isolated cron runs on native Ollama fall back to a 4096-token runtime context after `v2026.5.12`, preventing tool use, and asks to restore or document a way to force the larger context.

## Best Possible Solution

Keep the explicit native Ollama `params.num_ctx` contract, have affected `v2026.5.12` users update to a build containing the migration and run `openclaw doctor --fix`, and avoid restoring the broad catalog/default fallback.

## Reproduction Assessment

Yes, for the root cause: current source and the related Crabbox proof show native Ollama omits `options.num_ctx` without explicit `params.num_ctx`, which matches Ollama falling back to a 4096 runtime context. I did not run the reporter's exact WSL2 cron job live.

## Solution Assessment

Yes. The merged doctor migration and docs are the narrowest maintainable fix because they restore larger contexts for upgraded explicit configs without reintroducing the broad native Ollama fallback that maintainers intentionally removed.
