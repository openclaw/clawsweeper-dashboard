## Summary

Keep open: the PR targets a real Telegram compaction-retry bug, but the current patch still records the inbound message id before the attempt proves a user message was persisted, so a pre-prompt compaction retry can drop the real inbound prompt.

## What This Changes

The PR adds run-loop current-message tracking, retry suppression, a focused overflow-compaction regression test, and a Telegram changelog entry.

## Best Possible Solution

The merged fix should make compaction retry idempotent only after the inbound channel user message is actually persisted, avoiding both duplicate transcript entries and lost prompts.

## Reproduction Assessment

Yes. The linked issue provides duplicate Telegram message_id logs, and current main source shows non-mid-turn overflow compaction retries can reuse params.prompt and currentMessageId; I verified the path by source inspection rather than a live Telegram run.

## Solution Assessment

No. The same-message continuation guard is the right narrow direction, but this patch records the id before persistence, so pre-prompt compaction can suppress and drop the first real inbound prompt.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Record submitted IDs only after persistence:** `src/agents/pi-embedded-runner/run.ts:1058`
  - body: This assignment runs before `runEmbeddedAttemptWithBackend` knows whether a user message was appended. If the attempt exits through the pre-prompt `promptErrorSource: "precheck"` path, the retry sees the same `currentMessageId`, switches to the continuation prompt, and sets `suppressNextUserMessagePersistence`, so the first real user message after compaction is dropped. Set this from the persisted-user callback or exclude no-submit prechecks from the same-id branch.
  - confidence: 0.93
