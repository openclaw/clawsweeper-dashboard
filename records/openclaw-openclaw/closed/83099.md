## Summary

Keep open: the dialog observability direction is useful, but the latest head still returns pending dialog state from `/act` without the current-URL allow gate, so dialog text can bypass the browser SSRF policy.

## What This Changes

The PR adds managed-browser modal dialog observation, pending/recent dialog state in action/snapshot/debug responses, `dialogId` handling for browser dialog actions, docs, tests, and a changelog entry.

## Best Possible Solution

Keep the dialog observability feature, but gate blocked `/act` dialog responses with the same current-tab URL policy before returning dialog state and add a denied-current-URL regression test for that path.

## Reproduction Assessment

Yes, from source inspection: PR head 8530d2f returns `browserState` from `/act` when `blockedByDialog` is set, while the route is not entered with `enforceCurrentUrlAllowed`. The PR adds snapshot-side denied-URL coverage but not action-side coverage.

## Solution Assessment

No. Surfacing observed dialogs is a reasonable feature, but the action response must fail closed or redact dialog state through current-URL policy before this is the maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Guard blocked-dialog act responses with URL policy:** `extensions/browser/src/browser/routes/agent.act.ts:657-661`
  - body: When `executeActViaPlaywright` reports `blockedByDialog`, this route returns `browserState.dialogs.pending` directly even though `/act` is not entered with `enforceCurrentUrlAllowed`. That state includes dialog messages/default values, so a tab currently on a URL denied by the browser SSRF policy can still leak page content through the blocked action response; snapshot/debug paths already gate this. Check the current target URL before including dialog state, or fail closed/redact it.
  - confidence: 0.86
