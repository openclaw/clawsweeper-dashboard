## Summary

Current `main` does not support the PR's root-cause claim. Telegram prompt assembly takes model-facing inbound text from `BodyForAgent`/`RawBody`, not from the legacy `Body` produced by `formatInboundEnvelope(...)`, so wrapping those envelope strings would not change the actual Telegram user text the model sees. The cited Discord precedent also does not match: Discord still passes raw message text as the primary inbound body and only uses `wrapExternalContent(...)` for guild `UntrustedContext` metadata. Combined with the published single-trust-boundary security model, this PR does not reproduce as a current Telegram-only security fix.

## Evidence

The PR only wraps the `formatInboundEnvelope(...)` body path, not the fields current prompt assembly actually prefers.
