## Summary

Keep open: this is a valid bug-fix area, but the PR does not fix the reported default `read({ path: "media://inbound/..." })` path because the new resolver runs only inside the workspace-only guard, and that guarded path still lacks the managed inbound-media allowance.

## What This Changes

The PR imports `resolveMediaReferenceLocalPath` and resolves `media://` path arguments inside `wrapToolWorkspaceRootGuardWithOptions` before sandbox validation.

## Best Possible Solution

Normalize supported `media://inbound/...` references in the actual OpenClaw read execution path, preserve managed inbound-media safety checks, add focused default and workspace-only regression coverage, and then provide redacted real behavior proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: OpenClaw can emit `media://inbound/...` prompt notes, and current main's default read tool sends that value to the upstream read implementation without media-reference normalization; I did not run a live channel attachment flow.

## Solution Assessment

No. The patch runs in the workspace-only guard instead of the actual read wrapper, and the guarded path still rejects the resolved media-store path; the narrower maintainable fix is to normalize supported media references before read execution and preserve the inbound-media allowance.
