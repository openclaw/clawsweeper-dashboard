## Summary

Current main now addresses the reported no-start `openai-codex/gpt-5.5` OAuth Responses path: the relevant post-report fix preserves the PI-wrapped Codex stream so OAuth bearer injection is not bypassed, strips unsupported native ChatGPT payload fields, and keeps the explicit timeout/error surfacing for no-payload turns. The fix is on main only; it is not contained in `v2026.4.27`.

## What This Changes

The issue reports that OpenClaw 2026.4.25 hangs without streamed tokens or useful logs when running `openai-codex/gpt-5.5` through the legacy Codex OAuth Responses adapter.

## Best Possible Solution

Keep the current-main transport fix and ship it in the next release; if a fresh install from a post-fix build still hangs on the same OAuth route, open a new focused report with the new version, trajectory, and provider request diagnostics.

## Reproduction Assessment

Yes. The issue gives a high-confidence original reproduction path: configure an OAuth-backed `openai-codex/gpt-5.5` primary model, send a first turn through webchat or `openclaw agent`, and inspect the no-start trajectory; this read-only review verified the current source and tests rather than running a live OAuth probe.

## Solution Assessment

Yes. Preserving the existing PI-wrapped Codex stream and sanitizing only the native ChatGPT Codex payload is the narrow provider-boundary fix; the existing timeout/error surfacing remains useful hardening for any future backend stall.
