## Summary

Keep open: current main still rejects this trusted-proxy loopback password path while public docs still describe it as supported, and an open linked fix PR exists but needs auth-boundary review before the issue can close.

## What This Changes

The issue requests that trusted-proxy `allowLoopback` same-host callers with no forwarded headers and a valid `gateway.auth.password` succeed via local-direct password fallback.

## Best Possible Solution

Gateway/security owners should settle the intended contract, then either land a reviewed hotfix for the documented fallback or update the docs while moving internal callers to the separate service-identity design.

## Reproduction Assessment

Yes, source-level. Current main returns the trusted-proxy failure before password auth can run, and current tests assert valid loopback password credentials are rejected; I did not run a live gateway in this read-only review.

## Solution Assessment

Unclear. The requested fallback matches public docs, but it conflicts with a recent fail-closed auth change and with the broader service-identity RFC, so the narrow fix needs gateway/security policy review first.
