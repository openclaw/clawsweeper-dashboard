## Summary

Current `main` has partial Telegram multi-account isolation safeguards, but I did not find the requested hard preflight write gate that verifies the intended bot identity plus target chat/topic/environment before applying token or config changes. The code enforces account scoping, rejects some fallback misrouting cases, and scopes pairing state per account, but that is narrower than this issue’s requested mandatory preflight confirmation and audit trail for each apply action.

## Evidence

- **Unknown-account token fallback is blocked, but only at account resolution time:** `resolveTelegramToken` refuses channel-level fallback for a non-default `accountId` that is missing from a configured multi-account section, which prevents one wrong-token path but is not a general preflight confirmation step.
  - file: [extensions/telegram/src/token.ts:122](https://github.com/openclaw/openclaw/blob/7d471837369885c12d540d6bafa319ff41129b89/extensions/telegram/src/token.ts#L122)
  - command: `nl -ba extensions/telegram/src/token.ts | sed -n '120,145p'`
- **Duplicate bot-token reuse is treated as misconfiguration:** Telegram config status marks a secondary account unconfigured when it shares a token with another account, enforcing one owner account per token. This helps token isolation, but it still does not verify a newly supplied token against an intended bot username/chat/topic/environment tuple before write.
  - file: [extensions/telegram/src/shared.ts:35](https://github.com/openclaw/openclaw/blob/7d471837369885c12d540d6bafa319ff41129b89/extensions/telegram/src/shared.ts#L35)
  - command: `nl -ba extensions/telegram/src/shared.ts | sed -n '30,90p'; nl -ba extensions/telegram/src/shared.ts | sed -n '160,195p'`
- **Named-account group traffic requires explicit binding:** Inbound Telegram group handling drops named-account fallback routes unless there is an explicit binding, which is another target-isolation safeguard at runtime rather than a config-write preflight.
  - file: [extensions/telegram/src/bot-message-context.ts:236](https://github.com/openclaw/openclaw/blob/7d471837369885c12d540d6bafa319ff41129b89/extensions/telegram/src/bot-message-context.ts#L236)
  - command: `nl -ba extensions/telegram/src/bot-message-context.ts | sed -n '232,258p'`
- **Pairing allowlist state is account-scoped:** Docs state that non-default accounts read and write only their scoped pairing allowlist files, so pairing approval state no longer leaks across Telegram accounts.
  - file: [docs/channels/pairing.md:41](https://github.com/openclaw/openclaw/blob/7d471837369885c12d540d6bafa319ff41129b89/docs/channels/pairing.md#L41)
  - command: `nl -ba docs/channels/pairing.md | sed -n '40,55p'`
- **No mandatory bot/target/environment preflight implementation found:** A targeted repo search did not find an implementation matching the issue’s proposed `preflight_guardrails`, `environment_id`, target-tuple confirmation, or audit-trail-on-apply semantics. Bot username appears in status/probe surfaces, but not as a write gate.
