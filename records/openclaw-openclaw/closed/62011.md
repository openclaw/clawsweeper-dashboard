## Review Findings

Overall confidence: 0.87

Full review comments:

- **[P1] Keep scoped exec from inheriting owner auth paths:** `src/agents/bash-tools.exec.ts:1396-1403`
  - body: When `agentEnvVars` is set, this still copies `HOME`, `SHELL`, and `PATH` from `process.env` into the child environment. A collaborator configured with `scope:"none"` can still run CLIs against the owner's home-directory credential discovery paths, so the isolation boundary promised by this config is bypassed unless the baseline is deliberately constrained and tested.
  - confidence: 0.86
- **[P2] Honor agent env.vars when scope is omitted:** `src/agents/pi-tools.ts:469-472`
  - body: `agentEnvVars` is only computed when `scope` is explicitly present and not `"all"`. That means `env: { vars: { ... } }` or `scope:"all"` silently falls back to raw `process.env` and never applies the agent-specific vars, contradicting the new contract that those vars always win.
  - confidence: 0.9
- **[P2] Allow own scope to read allowed runtime env:** `src/agents/agent-scope.ts:387-400`
  - body: `scope:"own"` allowlists only keys from `collectConfigRuntimeEnvVars(cfg)`, which is the OpenClaw config env map, not the inherited process environment. A key supplied by the shell or service environment, such as an API key, will not be passed even when listed in `allow`, so the advertised allowlisted global inheritance does not work.
  - confidence: 0.84
