## Summary

The truncation is best closed as outside the OpenClaw repo. Current main requests TTS bytes from the OpenAI-compatible endpoint, waits for the full response body, writes that complete buffer to disk, then passes the saved buffer to Telegram. The issue discussion’s Kokoro-FastAPI PR #448 link matches the remaining root cause: Kokoro’s Opus/OGG writer drops the final page before returning bytes.

## Best Possible Solution

Close this OpenClaw issue as external to the repository. The best current path is to follow the Kokoro-FastAPI Opus fix upstream, or configure OpenClaw’s OpenAI TTS provider with a non-Opus `responseFormat` such as `mp3`/`wav` when complete audio is more important than native Opus voice-note delivery.

## Risks / Open Questions

- Users on an unfixed Kokoro-FastAPI image will still see truncated native Opus output until Kokoro ships or they run a patched build; OpenClaw can only choose another response format or consume whatever bytes the endpoint returns.
