## Summary

Keep open. The Windows Claude CLI spawn bug is real and still source-reproducible on current main, but this PR is not merge-ready because it is conflicted, lacks inspectable after-fix real behavior proof, and adds a second Windows command-resolution policy in a security-sensitive spawn path.

## What This Changes

The PR adds a Windows `.cmd` shim resolver for supervisor child spawns, wires it into `createChildAdapter`, and adds resolver tests for Claude-style `.cmd` wrappers.

## Best Possible Solution

Land a rebased process-layer fix that routes supervisor child spawns through one shared Windows spawn resolver while preserving current child-adapter protections and adding focused Windows shim regression coverage.

## Reproduction Assessment

Yes, source-reproducible with high confidence: current main still spawns bare `claude` while the child adapter only shims npm/pnpm/yarn/npx, and Node's Windows process contract explains why raw `.cmd` wrappers fail without a shell. ## Solution Assessment

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Reuse the shared Windows spawn resolver:** `src/process/windows-command.ts:59-61`
  - body: This adds `resolveWindowsCmdShimArgv` as a second command-wrapper parser even though current main already has `resolveWindowsSpawnProgram` covering PATH/PATHEXT, `.cmd`/`.bat`, entrypoints, package metadata, and fallback policy. Carrying two policies in a command-execution path can drift into wrong-binary or inconsistent fallback behavior, so please move/reuse the shared resolver instead.
  - confidence: 0.9
- **[P1] Preserve current child spawn preparation:** `src/process/supervisor/adapters/child.ts:63-65`
  - body: The branch is behind current main and this spawn call passes `resolvedArgv` directly, while current main first runs `prepareOomScoreAdjustedSpawn` and uses its hardened environment. Resolving the conflict by keeping this shape would drop Linux child OOM protection and shell-init env stripping; thread the Windows-resolved argv through the current prepared-spawn flow.
  - confidence: 0.86
- **[P2] Resolve relative shims from the child cwd:** `src/process/windows-command.ts:99`
  - body: When `argv[0]` is a relative `.cmd` path with a separator, the resolver reads that path before `spawn` applies `options.cwd`. A configured command like `.\\node_modules\\.bin\\claude.cmd` can therefore inspect the gateway cwd's file or fall through to direct `.cmd` spawning instead of resolving the child cwd target.
  - confidence: 0.78
