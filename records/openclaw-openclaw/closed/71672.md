## Summary

Keep open. Current main still documents the browser local startup timeout knobs only as values to raise on slower hosts; it does not document the 120000 ms cap or fallback behavior that this PR adds. The PR is contributor-authored, has no protected labels, is docs-only, and the discussion supports that the added wording matches implementation.

## Evidence

- **Current docs lack proposed bounds:** The Browser section currently says local managed profiles use browser.localLaunchTimeoutMs and browser.localCdpReadyTimeoutMs and to raise them on slower hosts, but it does not mention the 120000 ms maximum or fallback for zero/negative/non-finite values.
  - file: [docs/gateway/configuration-reference.md:451](https://github.com/openclaw/openclaw/blob/d6ef1fcf24eea3a5985d653a3baf400e32aa1b75/docs/gateway/configuration-reference.md#L451)

  - sha: [d6ef1fcf24ee](https://github.com/openclaw/openclaw/commit/d6ef1fcf24eea3a5985d653a3baf400e32aa1b75)
- **Implementation clamps startup timeouts:** Current implementation defines MAX_BROWSER_STARTUP_TIMEOUT_MS as 120_000 and normalizeStartupTimeoutMs returns the fallback for non-finite or <= 0 values, otherwise Math.min(value, MAX_BROWSER_STARTUP_TIMEOUT_MS).
