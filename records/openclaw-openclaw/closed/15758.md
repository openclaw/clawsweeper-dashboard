## Summary

The request is a useful extension, but current main already provides the intended integration point as an external speech-provider plugin. A generic shell-backed TTS adapter fits the plugin/ClawHub path rather than new OpenClaw core config syntax.

## Risks / Open Questions

- There is no bundled generic shell-script TTS plugin today; closing as `clawhub` assumes the requested behavior should be delivered as an external plugin, not that an off-the-shelf package already exists.
- A script-returning-local-path wrapper would still need plugin-specific safety and portability decisions outside core.

## Close Comment

Closing as `clawhub` after Codex review. Current `main` already has the right extension seam for this: OpenClaw treats speech as a public plugin capability (`docs/plugins/building-plugins.md:145`, `docs/plugins/sdk-provider-plugins.md:482`, `src/plugins/types.ts:1613`), loads speech providers through the normal plugin registry (`src/plugins/captured-registration.ts:154`), and supports installing external plugins from a local directory/archive or package (`docs/tools/plugin.md:11`, `docs/tools/plugin.md:26`, `docs/tools/plugin.md:128`). The project vision also says optional capability should usually ship as plugins and keeps a high bar for adding more core surface (`VISION.md:52`). A generic shell-backed TTS adapter that shells out and turns a returned path/URL into audio is therefore better scoped as a ClawHub/community plugin than new built-in `messages.tts.provider: "script"` core config. For reference, the bundled Vydra speech provider already shows the same pattern of taking an upstream audio URL and adapting it into the normalized TTS result (`extensions/vydra/speech-provider.ts:131`).
