## Summary

Keep open. Current main does not already implement this SDK surface, and the PR is useful directionally, but the proposed raw stream wrapper needs maintainer review before merge because it can misroute per-call model changes, accepts invalid wrapper output, and broadens the security-sensitive stream path for all plugins.

## What This Changes

The PR adds a Plugin SDK `registerStreamFnWrapper` API, stores wrappers in `PluginRegistry`, wires no-op/test-helper defaults, and applies registered wrappers around the embedded runner `streamFn` before idle timeout handling.

## Best Possible Solution

Keep the PR open and route it to plugin SDK plus embedded-runner owners to decide the contract first: either expose a validated, documented same-transport wrapper with core/security wrappers preserved outside it, or design a request-scoped model-routing hook that runs before provider transport and auth resolution.

## Reproduction Assessment

Yes for the review findings. The failure paths are source-traceable: a wrapper that returns a non-function is assigned directly, and a wrapper that changes provider/API after line 1627 still uses stream/auth state selected from the original attempt model.

## Solution Assessment

No. A raw cross-plugin `StreamFn` wrapper applied after transport resolution is not the best current shape unless the API explicitly forbids provider/API changes and validates wrapper output; a narrower routing hook or guarded wrapper contract would be safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Bind routing before transport resolution or constrain the hook:** `src/agents/pi-embedded-runner/run/attempt.ts:1624`
  - body: `registerStreamFnWrapper` is advertised for per-call model routing, but this hook runs after the attempt has already selected the provider stream function and auth provider from the original model. If a wrapper changes provider/API families, the call still uses the original transport/auth path, so either move routing before resolution or explicitly reject/document same-transport-only wrappers.
  - confidence: 0.84
- **[P2] Validate wrapper output before swapping streamFn:** `src/agents/pi-embedded-runner/run/attempt.ts:1624`
  - body: The catch only isolates thrown exceptions. A JS plugin can return `undefined` or another non-function, which is then assigned to `activeSession.agent.streamFn` and will abort the next model call despite the isolation intent. Check the return value is callable before replacing the stream function.
  - confidence: 0.9
- **[P3] Sanitize plugin errors before logging:** `src/agents/pi-embedded-runner/run/attempt.ts:1627`
  - body: `String(err)` is interpolated directly into the warning. A plugin-thrown value can include newlines or secrets and will be written as-is to logs, so log a bounded single-line/redacted summary or structured error metadata instead.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The PR introduces a new security-sensitive stream replacement hook and logs plugin-thrown errors without sanitization.

Concerns:

- **[medium] Plugin wrappers can skip inner stream protections:** `src/agents/pi-embedded-runner/run/attempt.ts:1624`
  - body: The wrapper is installed outside provider/core wrappers and may return a function that never calls the inner `streamFn`, bypassing transcript sanitizers, refusal recovery, provider wrappers, and diagnostics. Native plugins are in-process, so this is partly a trust-model decision, but widening this to every full-mode plugin should be explicit or critical wrappers should remain outermost.
  - confidence: 0.76
- **[low] Raw plugin error text reaches logs:** `src/agents/pi-embedded-runner/run/attempt.ts:1627`
  - body: The catch block logs `String(err)` directly. Plugin error strings can inject extra log lines or include sensitive request/plugin data, so the logged value should be bounded, single-line, and redacted or structured.
  - confidence: 0.82
