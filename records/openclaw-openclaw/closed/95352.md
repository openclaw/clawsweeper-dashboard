## Label Justifications

- `P1`: The PR targets P1 completion-delivery loss and still has a route identity defect that can misroute detached-task completions across task scopes.
- `merge-risk: 🚨 session-state`: The durable lease stores requester/session routing state without using the task scope in the executor hook path.
- `merge-risk: 🚨 message-delivery`: A default-scoped route lease can be overwritten or read for a different task sharing the same raw runId, wrongly targeting completion messages.
- `merge-risk: 🚨 security-boundary`: Cross-scope reuse of requester_origin_json can expose or reuse another requester route for completion delivery.

## Summary

Keep this PR open: the scoped lease schema/API direction is useful, but current head still wires the production acquire and settle hooks through the default lease scope, so same-runId tasks in different task scopes can still mis-associate requester origins.

## What This Changes

The branch adds a scoped SQLite task_route_leases table/API, executor acquire/settle hooks, cleanup, docs, generated schema types, unit/integration tests, and a lifecycle repro script.

## Best Possible Solution

Land the storage foundation only after executor lifecycle hooks acquire and settle with the TaskRecord scope tuple, docs/comments match that invariant, and the companion cron wiring remains separately reviewed.

## Reproduction Assessment

Yes, source-level: current main already treats raw runId as scope-sensitive, while this PR head's production acquire and settle hooks call the scoped lease API without passing any scope. A focused executor test with two same-runId pending tasks in different scopes would reproduce the overwrite/default-scope behavior.

## Solution Assessment

No; the scoped schema/API is the right direction, but the current integration is not the best fix because the main lifecycle path still uses the default scope. The maintainable repair is to derive lease scope from the TaskRecord at acquire and from the updated TaskRecord rows at settle.
