## Summary

This PR should close as superseded: the same doctor migration regression is already covered by an earlier open PR with a narrower guard, regression test, and real behavior proof, while this one-line merge-order change would make stale legacy model entries override existing canonical config.

## What This Changes

The PR flips the `rewriteModelsMap` spread order so legacy `openai-codex/*` model-map entries override canonical `openai/*` entries during doctor route repair.

## Best Possible Solution

Close this branch in favor of reviewing or landing https://github.com/openclaw/openclaw/pull/84142, or an equivalent targeted guard that preserves explicit legacy runtime pins without making legacy records globally outrank canonical config.

## Reproduction Assessment

Yes at source level: a config with `model.primary` on `openai-codex/*` and a legacy model-map PI pin can hit `ensureCodexRuntimePolicy` before `rewriteModelsMap`, creating the canonical Codex entry that later wins. I did not execute tests in this read-only review, but the canonical PR provides before/after live output for that path.

## Solution Assessment

No. Flipping the merge order globally is broader than the bug because it can make stale legacy records clobber existing canonical records; the targeted synthesis guard in the canonical PR is the safer maintainable fix.
