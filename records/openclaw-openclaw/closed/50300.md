## Summary

Close: the linked bug was already closed after maintainer review, and current main stores session transcript timestamps as valid UTC instants while reader/display paths parse or localize them correctly. The PR changes that durable representation rather than fixing a reproducible current-main bug.

## What This Changes

The PR adds a local-offset timestamp formatter, wraps several SessionManager write paths, updates manual session header writers, and adds tests so new session JSONL timestamps use explicit local offsets instead of UTC `Z` strings.

## Best Possible Solution

Keep durable transcript timestamps as UTC ISO instants and rely on reader/display paths to localize them; if raw JSONL inspection remains confusing, document that storage contract rather than rewriting persisted timestamp semantics.

## Reproduction Assessment

Yes. The current-main reproduction check follows the write, read, and export paths: UTC `Z` strings are stored as true instants, parsed with `Date.parse`, and localized for display, so the reported shifted-time bug does not reproduce.

## Solution Assessment

No. Changing the persisted JSONL representation to local offsets is not the best fix for the closed report because current UTC storage is semantically correct; display or docs are the safer place to address raw-inspection confusion.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve malformed persisted timestamps:** `src/sessions/local-session-timestamps.ts:29`
  - body: This helper throws `RangeError` for unparsable persisted timestamp strings, and the wrapper applies it to existing entries. A user-edited or corrupted transcript could make wrapped session operations fail repeatedly instead of leaving the bad value alone.
  - confidence: 0.83
- **[P2] Normalize recovered headers after open-time repair:** `src/sessions/local-session-timestamps.ts:65-66`
  - body: `SessionManager.open()` can rewrite an empty or corrupted transcript and set `flushed = true` before this wrapper runs. The `flushed` guard then skips that freshly written header, so the recovery path still leaves a UTC `...Z` header even under the PR's intended behavior.
  - confidence: 0.86
