## Summary

Closing: current main already gets the required streamable-http POST `Accept` header from the pinned MCP SDK, so this PR changes the wrong seam and its tests do not verify the actual OpenClaw request path.

  - file: [src/agents/mcp-transport.ts:102](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/src/agents/mcp-transport.ts#L102)

  - file: [upstream:modelcontextprotocol/typescript-sdk/src/client/streamableHttp.ts:466](https://github.com/openclaw/openclaw/blob/v1.29.0/upstream%3Amodelcontextprotocol/typescript-sdk/src/client/streamableHttp.ts#L466)

That means the first OpenClaw streamable-http POST already goes through the SDK code that sets the Accept header.

So the added tests would validate the wrong seam and would not prove the reported runtime bug.
