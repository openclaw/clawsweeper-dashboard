## Summary

Keep open. Current `main` still contains code paths consistent with the report for existing-session browser behavior: when a snapshot/action omits `targetId`, tab selection can create a new tab if Chrome MCP temporarily reports zero tabs, and otherwise it chooses `lastTargetId` or the first page-like tab without filtering inspect/devtools pages. I did not find evidence on `main` that this issue was fixed.

## Evidence

- **Existing-session selection can open a new tab when tab listing is empty:** `ensureTabAvailable()` calls `listTabs()`, and if that returns zero tabs it unconditionally calls `openTab("about:blank")` before re-listing. That directly contradicts the reported expectation to avoid opening new tabs/windows when using an attached existing session.
  - file: [extensions/browser/src/browser/server-context.selection.ts:40](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/extensions/browser/src/browser/server-context.selection.ts#L40)
  - command: `nl -ba extensions/browser/src/browser/server-context.selection.ts | sed -n '35,95p'`
  - sha: [f03252aaf9cc](https://github.com/openclaw/openclaw/commit/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022)
- **For Chrome MCP profiles, opening a tab really creates a new page in the attached browser:** The Chrome MCP branch of `openTab()` calls `openChromeMcpTab(...)` and stores that new target as `lastTargetId`, so any transient empty-tab result on an existing-session profile can produce a real new tab in the user's browser session.
  - file: [extensions/browser/src/browser/server-context.tab-ops.ts:153](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/extensions/browser/src/browser/server-context.tab-ops.ts#L153)
  - command: `nl -ba extensions/browser/src/browser/server-context.tab-ops.ts | sed -n '150,175p'`
  - sha: [f03252aaf9cc](https://github.com/openclaw/openclaw/commit/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022)
- **Snapshots without explicit targetId rely on that default tab-selection path:** `GET /snapshot` resolves `targetId` from the query, then calls `profileCtx.ensureTabAvailable(targetId || undefined)`. So if the agent does not pass an explicit `targetId`, snapshot selection inherits the empty-tab new-page behavior and the default-tab heuristics.
  - file: [extensions/browser/src/browser/routes/agent.snapshot.ts:362](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/extensions/browser/src/browser/routes/agent.snapshot.ts#L362)
  - command: `nl -ba extensions/browser/src/browser/routes/agent.snapshot.ts | sed -n '358,390p'`
  - sha: [f03252aaf9cc](https://github.com/openclaw/openclaw/commit/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022)
- **Chrome MCP tab normalization does not distinguish inspect/devtools pages from real content tabs:** `toBrowserTabs()` drops titles and marks every Chrome MCP page as `type: "page"`. Combined with default selection choosing the first page tab, there is no current safeguard here to deprioritize `chrome://inspect` or similar non-content tabs.
  - file: [extensions/browser/src/browser/chrome-mcp.ts:79](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/extensions/browser/src/browser/chrome-mcp.ts#L79)
