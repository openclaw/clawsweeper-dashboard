## Summary

Current main already has a message-tool-only terminal guard for completion delivery, so a generated media completion run stops after the first successful implicit message send instead of letting session history drive repeated sends.

## What This Changes

The issue reports duplicate Discord image attachments from an image_generate completion wake that repeats message-tool sends for the same generated media.

## Best Possible Solution

Keep the current main terminal guard and ship it in the next release so generated-media completion wakes cannot post the same artifact repeatedly.

## Reproduction Assessment

Yes, source inspection matches the reported path: generated media completion wakes the requester session and relies on message-tool delivery. I did not run a live Discord reproduction in this read-only review, but current main contains the terminal guard for that path.

## Solution Assessment

Yes, the current main approach is a narrow maintainable fix: preserve message-tool delivery but terminate the completion turn after one successful implicit send. A broader deterministic direct-delivery rewrite is not needed for this duplicate-send symptom.
