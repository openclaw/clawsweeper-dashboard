## Summary

Keep open: the report has concrete macOS LaunchAgent evidence, current main still routes agent `update.run` package updates through the live Gateway process, and the linked fixing PR remains open with review blockers/proof gaps.

## What This Changes

Prevent agent-invoked self-update from stranding macOS LaunchAgent installs by handing package-managed `update.run` off to a safe external update path with durable health/completion reporting.

## Best Possible Solution

Review and land the linked PR or a replacement that routes package-managed `update.run` through a detached/supervised CLI update handoff, preserves service identity, delays success continuation until restart health passes, and proves the flow on a disposable macOS LaunchAgent install.

## Reproduction Assessment

Yes for a source-level reproduction path: on current main, an agent can invoke `update.run`, which reaches `runGatewayUpdate` in the live Gateway process for package installs. I did not run the destructive macOS LaunchAgent self-update scenario during this read-only review.

## Solution Assessment

Yes at the direction level: a detached/supervised handoff to the CLI managed-service update path is the narrow maintainable fix. The current linked PR is the active candidate but still needs its review blockers and real environment proof resolved.
