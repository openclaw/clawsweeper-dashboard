## Summary

Current main already has a gateway-wide async error boundary for request handlers, so thrown cron handler promises are converted into an `UNAVAILABLE` response instead of escaping as unhandled rejections. This PR's per-handler try/catch is redundant and would bypass the existing `formatForLog(...)` sanitization path.

## Evidence

- **Gateway request dispatcher awaits the selected handler:** `handleGatewayRequest(...)` resolves the method handler and `await`s it inside the request scope, so any rejection from `cronHandlers` propagates to the caller rather than becoming detached.
  - file: [src/gateway/server-methods.ts:106](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/src/gateway/server-methods.ts#L106)
  - command: `nl -ba src/gateway/server-methods.ts | sed -n '1,180p'`
  - sha: [f3c37a946cc4](https://github.com/openclaw/openclaw/commit/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4)
- **WebSocket message handler already catches failed request promises:** The gateway wraps `handleGatewayRequest(...)` in `(... )().catch(...)`, logs `request handler failed`, and responds with `errorShape(ErrorCodes.UNAVAILABLE, formatForLog(err))`. That is already the error boundary this PR is trying to add, and it preserves the existing sanitization path.
  - file: [src/gateway/server/ws-connection/message-handler.ts:1504](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/gateway/server/ws-connection/message-handler.ts#L1504)

- **Cron handlers still rely on the outer boundary on main:** Current `cron.list`, `cron.status`, `cron.add`, `cron.update`, `cron.remove`, and `cron.runs` still contain bare `await` calls. Since main ships that code alongside the outer `.catch(...)`, the reported crash path is already handled at the gateway boundary rather than in per-handler try/catch blocks.
