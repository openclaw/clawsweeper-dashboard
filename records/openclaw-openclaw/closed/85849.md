## Summary

Keep this PR open. The linear matcher is the right direction for the ReDoS fix, but the current patch introduces a blocking authorization semantic change: blank or whitespace-only allow entries are filtered out, which can turn an explicitly non-matching allowlist into allow-all behavior.

## What This Changes

This PR replaces regex-backed `tools.agentToAgent.allow` wildcard matching with a precompiled segment glob matcher, adds session-access coverage, and adds a changelog entry.

## Best Possible Solution

Land the linear glob matcher after preserving the existing deny behavior for blank allow entries and validating wildcard parity with focused session-access tests.

## Reproduction Assessment

Yes, source-reproducible. Current main still converts wildcard allow patterns into a `RegExp` inside `createAgentToAgentPolicy`, and the PR body includes terminal proof against the patched policy API.

## Solution Assessment

No, not as currently written. Replacing the regex with a segment matcher is the narrow maintainable fix, but it must preserve blank-entry allowlist semantics before merge.

Overall confidence: 0.93

Full review comments:

- **[P1] Preserve blank allow entries as non-matches:** `src/plugin-sdk/session-visibility.ts:164-166`
  - body: On main, `allow: [""]` is a non-empty allowlist and the blank entry returns false for every agent. This patch compiles that entry to `null`, filters it out, and then treats the compiled list as empty, so `matchesAllow()` returns true and `isAllowed()` can allow any cross-agent pair when enabled. Keep a separate raw-list-empty flag or compile blanks to a never-match pattern, and add blank/whitespace allowlist regression coverage.
