## Summary

Keep open for maintainer review. Current main does not implement this archive export/import workflow, and the PR is not ready to merge because it overlaps the newer provider-based migrate command and still raises restore/import security concerns that maintainers explicitly called out in discussion.

## What This Changes

The PR branch adds archive-based `openclaw migrate export` and `openclaw migrate import` commands for config, credentials, workspace, and session migration, with CLI registration, import/export infrastructure, and tests.

## Best Possible Solution

Pick the recovery model first. If core stays conservative, evolve `openclaw backup` with safer platform targeting, archive conversion or verification, and clearer manual restore docs. If OpenClaw should own automated import, integrate it with the existing provider-based migrate namespace and require preview-first UX, canonical target checks, archive validation, rollback/conflict handling, and security-focused tests before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Canonicalize targets before destructive writes:** `src/infra/migrate-import.ts:328-333`
  - body: `assertTargetWithinRoot` only compares `path.resolve` strings, so an existing symlinked parent such as the local `agents` directory can pass validation while the later `fs.rm`/copy operates outside the OpenClaw state tree. Resolve existing target ancestors or reject symlinked target parents before any destructive import write.
  - confidence: 0.89
- **[P2] Exclude unknown agents from default-workspace export:** `src/commands/migrate-shared.ts:174-180`
  - body: When `--agents` contains an ID that is not in `agents.list`, this branch still adds the default workspace because the requested agent lacks an explicit workspace. That makes `openclaw migrate export --agents typo --include workspace` include unrelated workspace data despite matching no agent.
  - confidence: 0.88
- **[P2] Preserve existing migrate provider commands:** `src/cli/program/register.migrate.ts:45-58`
  - body: Current main already uses `openclaw migrate` for plugin-owned provider imports with `list`, `plan`, and `apply`. This registration defines the command as archive `export`/`import` only, so after rebase it would conflict with or drop the existing Claude/Hermes migration surface.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff introduces a broad archive import/restore path and still lacks canonical target validation before destructive writes.

Concerns:

- **[high] Symlinked target parents can escape the state root:** `src/infra/migrate-import.ts:328`
  - body: Import target containment is checked lexically with `path.resolve`, but the write pass removes and copies paths afterward. If an existing target parent is a symlink, those writes can resolve outside the intended OpenClaw state tree.
  - confidence: 0.89
