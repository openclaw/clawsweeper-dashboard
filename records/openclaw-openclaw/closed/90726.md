## Summary

Current main and the v2026.6.1 tag already treat provider HTTP 500/503 failures as failover-worthy and route isolated cron agent turns through the configured fallback chain, so the reported fallback bypass is already implemented.

## What This Changes

Treat transient HTTP 500/503 model failures during cron agent turns as recoverable and advance through configured fallback models.

## Best Possible Solution

Keep the existing shipped failover path and ask for a new report only if a concrete job config and run log still show a supported fallback list being skipped.

## Reproduction Assessment

No. Current main and v2026.6.1 source show Google HTTP 500 errors carry status metadata, classify as timeout, and cron passes payload/configured fallbacks into runWithModelFallback; the issue does not include a config/run log that reproduces a remaining bypass.

## Solution Assessment

Yes. The existing shared provider error classification plus cron fallback policy is the narrow maintainable solution because it handles Google/NVIDIA-style 5xx failures without adding cron-specific special cases.
