## Summary

Current main now prevents the reported destructive cleanup path: direct cron delivery cleanup returns before `sessions.delete` unless the target key is cron-owned, and regression coverage locks that non-cron direct sessions are preserved.

## What This Changes

The issue reports that one-shot cron jobs with `deleteAfterRun` can delete and rebind a live direct-message session when `sessionTarget` resolves to the current or explicit live session key.

## Best Possible Solution

Keep the current cron-owned cleanup guard for this bug, and handle any broader policy to isolate all direct-message cron targets separately from the destructive delete/rebind fix.

## Reproduction Assessment

No current-main destructive reproduction remains. Source inspection shows the old composition still reaches direct delivery, but the current cleanup guard skips `sessions.delete` for non-cron live DM keys.

## Solution Assessment

Yes. Guarding the destructive cleanup call with the existing cron-owned session-key contract is the narrowest maintainable fix for the reported deletion and rebinding behavior.
