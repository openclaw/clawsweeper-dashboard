## Summary

Keep this open. Current main still lacks explicit handling for the reported uppercase `[TOOL_CALL]{tool => ..., args => ...}[/TOOL_CALL]` assistant text, and the WhatsApp reply path can still normalize and send that text as a visible reply.

## What This Changes

The issue requests that WhatsApp stop showing raw pseudo tool-call text and handle tool-call output in a user-visible way.

## Best Possible Solution

Add shared sanitizer and diagnostic coverage for this exact pseudo-call shape so it cannot leak through WhatsApp or history, while keeping actual execution limited to structured provider tool calls unless maintainers approve a safe parser.

## Reproduction Assessment

Yes. At source level, the reported text can flow through `normalizeWhatsAppPayloadTextPreservingIndentation` and then `deliverWebReply` without matching the existing sanitizer or diagnostic patterns before `msg.reply`.

## Solution Assessment

Partly. Suppression plus diagnostics is the best narrow repair; promoting this syntax into real tool execution is not the best automated fix because assistant-visible text is not a trusted tool-call contract.
