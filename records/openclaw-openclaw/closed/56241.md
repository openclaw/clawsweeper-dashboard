## Summary

Close. Current `main` already supports legacy `messages.tts.edge` via runtime compatibility migration on normal config reads, and docs/tests now describe that contract. This PR's schema-level test targets an outdated expectation that `TtsConfigSchema` should accept the legacy key directly.

## Close Comment

Closing from a Codex review. Current `main` already implements this compatibility on the runtime config-read path: `src/commands/doctor/shared/legacy-config-migrations.runtime.tts.ts:10` and `:71` include `edge` and migrate it to `messages.tts.providers.microsoft`, and `src/config/io.ts:1072` applies those legacy migrations on normal reads. The current docs also describe the intended contract in `docs/tools/tts.md:299` and `:305`: legacy `provider: "edge"` is normalized to `microsoft`, and legacy direct provider blocks are auto-migrated on load. There is already regression coverage for accepting legacy TTS provider keys via auto-migration in `src/config/config-misc.test.ts:692`. Given that, this PR's direct `TtsConfigSchema` test is no longer the right contract to merge.
