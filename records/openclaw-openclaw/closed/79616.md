## Summary

Current main and v2026.5.7 already remove the reported normal ACPX startup black box: ACPX registers a lazy backend without running the heavy runtime probe during Gateway startup. The exact subphase log format is not present, but the observable 2026.4.15 gap is superseded by the shipped lazy-startup design.

## What This Changes

The issue requests structured ACPX startup subphase timing logs to explain a long cold-start gap between sidecar startup and ACPX backend registration.

## Best Possible Solution

Keep the shipped lazy ACPX registration behavior and track the remaining Gateway-ready contract separately in https://github.com/openclaw/openclaw/issues/79596; add opt-in probe subphase logs only as a later diagnostics enhancement if maintainers still need them.

## Reproduction Assessment

No for current main: I did not establish a current-main failing path. Source and tag comparison show the old release path that entered ACPX service startup was replaced by lazy backend registration in v2026.5.7.

## Solution Assessment

Yes. Deferring real ACPX runtime startup out of the normal Gateway ready path is narrower than adding default phase logs around work that no longer runs during startup; opt-in probe instrumentation can be a separate enhancement if needed.
