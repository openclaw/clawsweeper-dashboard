## Summary

Current main still does not implement the requested early Telegram native typing handshake. Telegram builds a `sendTyping` callback after access checks, but the message processor does not call it when the accepted context is ready; it is only wired into the downstream reply typing pipeline, whose documented and implemented start point is the model/reply loop rather than inbound acceptance. The related #6946 remains a broader visible processing-message feature, not a replacement for this narrower typing timing bug.

## Best Possible Solution

Keep this issue open and implement the fix in the Telegram plugin path: after `buildTelegramMessageContext` has accepted and normalized an inbound message, but before slow dispatch, queueing, compaction, model, or tool work, start a best-effort native Telegram typing action through the existing typing lifecycle or an equivalent guarded one-shot. Add focused regression coverage proving accepted Telegram messages call `sendChatAction("typing")` before a delayed dispatch path, while ignored, unauthorized, disabled, unmentioned, suppressed, or binding-failed messages do not. Keep #6946 separate as the visible processing-placeholder feature.

## Risks / Open Questions

- An early Telegram typing fix must remain gated behind Telegram access checks, mention/body normalization, route/thread resolution, configured binding readiness, and typing suppression so ignored or unauthorized messages do not emit visible feedback.
- The fix should integrate with the existing typing lifecycle to avoid duplicate keepalive loops or lingering typing after dropped, silent, failed, or superseded turns.
