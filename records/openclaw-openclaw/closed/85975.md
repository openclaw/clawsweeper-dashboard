## Summary

Close: current main and the latest stable release no longer use the old fixed 70k Codex app-server startup fuse that caused large `thread_bootstrap` first turns to lose the warmed native thread; the broader long-session lifecycle work remains tracked separately.

## What This Changes

The issue asks OpenClaw to preserve compatible Codex app-server `thread_bootstrap` native thread bindings after large bootstrap turns instead of rotating them before context-engine compatibility checks can run.

## Best Possible Solution

Keep the shipped Codex app-server startup guard/reuse behavior and use https://github.com/openclaw/openclaw/issues/86023 for the broader long-running session lifecycle and diagnostics work.

## Reproduction Assessment

Yes for the original buggy release path: the old source cleared a saved `thread_bootstrap` binding at the fixed 70k guard before the compatibility gate could run. No current-main failing repro remains because the startup guard is now model-window-aware and covered by regression tests.

## Solution Assessment

Yes. The current implementation is the narrow maintainable fix for this issue: preserve compatible `thread_bootstrap` reuse through the old guard shape while still rotating legacy, stale, byte-oversized, or truly near-window native threads.
