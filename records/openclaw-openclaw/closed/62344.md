## Summary

Current `main` does not show a path where `sessionTarget: "isolated"` cron runs can persist model state into `agent:main:main`. Isolated jobs resolve to cron-scoped session keys, and the persist helper writes only the cron base key plus its per-run key. The prior maintainer trace on the issue reached the same conclusion and specifically asked for the persisted cron-store `sessionTarget` value; no follow-up evidence was provided.

## Evidence

- **Gateway cron key resolution:** `runIsolatedAgentJob` starts isolated jobs on `cron:<job.id>` and only switches away from that when `job.sessionTarget` starts with `session:`. There is no branch here that maps `sessionTarget: "isolated"` to the parent main session key.
  - file: [src/gateway/server-cron.ts:333](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/src/gateway/server-cron.ts#L333)
  - command: `nl -ba src/gateway/server-cron.ts | sed -n '300,360p'`
  - sha: [51dd4f288f67](https://github.com/openclaw/openclaw/commit/51dd4f288f67ef4df4c7a121f078f508535df1fd)
- **Cron session canonicalization:** `resolveCronAgentSessionKey` delegates to `toAgentStoreSessionKey`, so a raw isolated key like `cron:<job>` becomes `agent:<agentId>:cron:<job>`, not `agent:<agentId>:main`.
  - file: [src/cron/isolated-agent/session-key.ts:5](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/src/cron/isolated-agent/session-key.ts#L5)
  - command: `nl -ba src/cron/isolated-agent/session-key.ts | sed -n '1,220p'`
  - sha: [51dd4f288f67](https://github.com/openclaw/openclaw/commit/51dd4f288f67ef4df4c7a121f078f508535df1fd)
- **Persist helper only touches cron-scoped keys:** `createPersistCronSessionEntry` writes only `agentSessionKey` and `runSessionKey`. For isolated jobs those are the cron base key and `:run:` key; there is no write to `agent:main:main` unless the resolved `agentSessionKey` itself is that main key.
  - file: [src/cron/isolated-agent/run-session-state.ts:22](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/src/cron/isolated-agent/run-session-state.ts#L22)
  - command: `nl -ba src/cron/isolated-agent/run-session-state.ts | sed -n '1,220p'`
  - sha: [51dd4f288f67](https://github.com/openclaw/openclaw/commit/51dd4f288f67ef4df4c7a121f078f508535df1fd)
- **Current run path uses cron-scoped base and run keys:** `runCronIsolatedAgentTurn` resolves `baseSessionKey`, derives `agentSessionKey`, forces a fresh session when `sessionTarget === "isolated"`, and persists to `agentSessionKey` plus `runSessionKey`.
  - file: [src/cron/isolated-agent/run.ts:459](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/src/cron/isolated-agent/run.ts#L459)
  - command: `nl -ba src/cron/isolated-agent/run.ts | sed -n '430,590p'`
  - sha: [51dd4f288f67](https://github.com/openclaw/openclaw/commit/51dd4f288f67ef4df4c7a121f078f508535df1fd)
- **Concurrent-write guard and test coverage:** `updateSessionStore` re-reads inside the session-store lock before mutating, reducing same-process clobber risk, and tests assert isolated cron runs use fresh `agent:main:cron:*` keys rather than `agent:main:main`.
  - file: [src/config/sessions/store.ts:421](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/src/config/sessions/store.ts#L421)
  - command: `nl -ba src/config/sessions/store.ts | sed -n '380,438p'`
  - sha: [51dd4f288f67](https://github.com/openclaw/openclaw/commit/51dd4f288f67ef4df4c7a121f078f508535df1fd)
- **Prior issue discussion matches current code:** The April 12, 2026 maintainer comment on #62344 traced the same write path on `upstream/main`, concluded there was no direct contamination path, and asked for the persisted `.cron-store.json` `sessionTarget` value to distinguish likely config mismatch from an external concurrent writer. No follow-up evidence is present in the provided timeline.

## Risks / Open Questions

- This was a read-only review; I did not run a live reproduction because the task forbade write-producing commands and local `pnpm` was unavailable in the environment.
- An out-of-process writer mutating the same `sessions.json` outside OpenClaw's lock would still be a plausible explanation, but that would not be an isolated-cron key-bleed bug in current `main`.
