## Label Justifications

- `P1`: The bug breaks a real iOS Talk workflow for users with configured Gateway speech/TTS providers by starting an unconfigured realtime path and falling back to the system voice.
- `impact:auth-provider`: The failure is provider-routing specific: iOS chooses realtime/OpenAI handling instead of the configured Gateway Talk speech provider and its credentials/config.

## Summary

Keep open: current main still matches the report, and two open linked PRs are implementation candidates rather than a reason to close this issue. The iOS Talk manager still classifies any non-ElevenLabs active Gateway speech provider as realtime, then playback falls back to local ElevenLabs or iOS system speech instead of Gateway `talk.speak`.

## What This Changes

The issue requests that iOS Talk keep Gateway speech/TTS providers on native `talk.speak` playback instead of misclassifying them as realtime WebRTC.

## Best Possible Solution

Review and land one focused iOS Talk fix that derives realtime mode from parsed realtime config/catalog state, routes Gateway speech providers through `talk.speak`, and includes iOS regression coverage plus real-device or simulator proof.

## Reproduction Assessment

Yes, by source inspection and reporter evidence: current iOS still turns non-ElevenLabs Gateway speech providers into realtime mode and lacks a Gateway `talk.speak` playback branch. I did not run a live iPhone reproduction in this read-only review.

## Solution Assessment

Yes: the narrow fix belongs in iOS Talk provider-mode resolution and native playback routing, using the existing Gateway `talk.speak` RPC. A new config option or Gateway protocol change is not needed unless the linked PR review finds the existing RPC insufficient.
