## Summary

Current main already fixes the reported Feishu Windows failure by making the shared media-store fsync step best-effort on EPERM, and Feishu inbound image downloads use that shared save path.

## What This Changes

The issue reports Windows Feishu image downloads failing when the Gateway saves inbound media and the post-write fsync step returns EPERM.

## Best Possible Solution

Keep the shared media-store EPERM guard and let Feishu inherit it through the existing plugin runtime media API, rather than adding plugin-specific Windows filesystem logic.

## Reproduction Assessment

Yes. The source path is clear: Feishu image media saves through `saveMediaBuffer`, and an `EPERM` from the post-write `handle.sync()` would have failed before the current guard; I did not run a live Windows Feishu test.

## Solution Assessment

Yes. Handling EPERM in the shared post-write fsync helper is the narrowest maintainable fix because it covers Feishu and other channel uploads without weakening the actual write/rename path.
