## Summary

Current main still does not fully fix #69112: the embedded runner acquires the session write lock without forwarding the CLI/gateway turn timeout, so the lock helper keeps its 10s default wait. The same remaining work is already tracked by open PR #69159, which specifically adds timeoutMs forwarding and a machine-readable session_busy lock error for this issue.

## Best Possible Solution

Close #69112 as superseded by open PR #69159 and keep #69159 as the canonical implementation/review thread. The product fix should forward the caller's timeout into acquireSessionWriteLock and return a machine-readable session-busy or temporary-failure signal through the CLI/gateway JSON path.
