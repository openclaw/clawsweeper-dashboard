## Summary

Close as superseded: the remaining Feishu duration bug is already tracked by an open canonical issue, while this PR branch is not landable because it rewrites current Feishu metadata/routing behavior and adds synchronous ffprobe probing on the outbound send path.

## What This Changes

The PR modifies Feishu media sending to derive audio/video duration with ffprobe and pass it to Feishu file uploads while also refactoring media routing and download metadata handling.

## Best Possible Solution

Keep the canonical duration issue open and land a narrow Feishu patch that preserves current metadata and MIME-aware routing, computes a finite duration from the actual uploaded media bytes, and covers the duration field with focused tests.

## Reproduction Assessment

Yes. Source inspection shows current main routes native Feishu Ogg/Opus media through uploadFileFeishu without computing or passing duration; I did not run a live Feishu UI send in this read-only review.

## Solution Assessment

No. The PR targets the right bug, but the branch is broader than necessary, regresses current metadata/routing behavior, misses remote and buffer-backed duration paths, and blocks sends with synchronous probing.

Full review comments:

- **[P1] Preserve Feishu download metadata parsing:** `extensions/feishu/src/media.ts:181-191`
  - body: This replacement only reads lowercase response.headers and a simple filename= pattern, so Feishu SDK fields, header aliases, filename* names, and CJK filename recovery that current main preserves can be lost for downloaded resources.
  - confidence: 0.9
- **[P1] Keep the centralized MIME-aware media router:** `extensions/feishu/src/media.ts:483-489`
  - body: The inline routing maps every loaded video MIME to mp4 and downgrades audio MIME with a generic filename to stream, diverging from the current resolveFeishuOutboundMediaKind behavior and risking wrong Feishu message delivery.
  - confidence: 0.88
- **[P2] Avoid synchronous ffprobe during media sends:** `extensions/feishu/src/media.ts:501-510`
  - body: sendMediaFeishu now runs spawnSync("ffprobe") on the outbound send path, so a slow or malformed local media file can block the Node event loop for the timeout while other gateway work waits.
  - confidence: 0.84
- **[P2] Probe duration for the uploaded bytes, not only raw local paths:** `extensions/feishu/src/media.ts:494-499`
  - body: Duration is only populated for non-HTTP mediaUrl values that still exist as local files, so remote URLs and mediaBuffer sends can still upload opus/mp4 without duration even though those are valid sendMediaFeishu inputs.
  - confidence: 0.86
