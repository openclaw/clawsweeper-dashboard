## Summary

Keep open. Current main still lacks both requested capabilities, and the open adaptive-mode PR only covers the daily-plus-idle AND semantics; this PR also has concrete correctness defects and needs maintainer API direction before it is safe to repair or merge.

## What This Changes

The PR adds optional `session.reset.timezone` and `session.reset.idleGuard` config fields, wires user-timezone fallback through session reset policy resolution, updates schema/help/types, and adds reset policy tests.

## Best Possible Solution

Split the accepted behavior cleanly: review or land the adaptive-mode approach for AND semantics, and add validated explicit `session.reset.timezone` support separately if maintainers want timezone-aware daily boundaries.

## Reproduction Assessment

Yes. The current-main gap is reproducible by inspecting the strict schema, reset policy, and docs; the PR-specific failures are reproducible with an `Intl.DateTimeFormat` invalid-zone check and UTC+13/+14 offset calculation.

## Solution Assessment

No, not as-is. The timezone feature is useful, but the patch needs schema validation, correct offset handling, explicit fallback semantics, current-module porting, and reconciliation with #71400 before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Validate timezone IDs before evaluating resets:** `src/config/zod-schema.session.ts:21`
  - body: `timezone` accepts any string, but the reset helper later passes it to `Intl.DateTimeFormat`. A typo such as `UTC+8` throws `RangeError` during session freshness evaluation, so this needs schema or policy-boundary validation before runtime use.
  - confidence: 0.91
- **[P1] Fix offset wrapping for UTC+13 and UTC+14 zones:** `src/config/sessions/reset.ts:273-278`
  - body: The proposed offset normalization treats differences above 12 hours as day wrap, but valid IANA zones such as `Pacific/Tongatapu` and `Pacific/Kiritimati` exceed +12. Those users would get inverted offsets and reset boundaries on the wrong day.
  - confidence: 0.92
- **[P2] Reject idleGuard without an idle window:** `src/config/sessions/reset.ts:175-187`
  - body: `idleGuard` only switches to AND mode when `idleExpiresAt` exists. With `idleGuard: true` and no positive `idleMinutes`, the flag is silently skipped and the config behaves as daily-only, which is misleading for an idle guard.
  - confidence: 0.84
- **[P2] Keep omitted reset timezones host-local:** `src/config/sessions/reset.ts:132-133`
  - body: Using `params.userTimezone` as the fallback changes daily reset behavior for users who configured prompt timezone but did not set `session.reset.timezone`. That contradicts the new help text and the current host-local compatibility contract.
  - confidence: 0.8
