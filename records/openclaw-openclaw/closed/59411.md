## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Validate timezone IDs before evaluating resets:** `src/config/zod-schema.session.ts:21`
  - body: `timezone` accepts any string, but the reset helper later passes it to `Intl.DateTimeFormat`. A config typo such as `UTC+8` throws `RangeError` during session freshness evaluation instead of failing at the config boundary.
  - confidence: 0.91
- **[P1] Fix offset wrapping for UTC+13 and UTC+14 zones:** `src/config/sessions/reset.ts:273-278`
  - body: The offset normalization treats differences above 12 hours as day wrap, but valid IANA zones such as `Pacific/Tongatapu`, `Pacific/Kiritimati`, and `Pacific/Chatham` exceed +12. Those users would get inverted offsets and reset boundaries on the wrong day.
  - confidence: 0.92
- **[P2] Align omitted timezone behavior with the public contract:** `src/config/sessions/reset.ts:135-136`
  - body: The help text says an omitted reset timezone uses the host machine's local timezone, but the policy falls back to `agents.defaults.userTimezone`. Existing users who configured a prompt timezone could see daily reset boundaries move without setting `session.reset.timezone`.
  - confidence: 0.84
- **[P2] Reject idleGuard without an idle window:** `src/config/sessions/reset.ts:175-181`
  - body: `idleGuard` switches to AND mode only when `idleExpiresAt` exists. With `idleGuard: true` and no positive `idleMinutes`, the flag is silently ignored and the config behaves as daily-only, leaving operators no feedback on a useless guard.
  - confidence: 0.84
