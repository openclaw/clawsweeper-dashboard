## Summary

Keep open: the linked Linux/SSE2 image-optimization bug is still valid on current main, but this PR is not mergeable as-is because the new vipsthumbnail path has concrete correctness and security/parity gaps and lacks inspectable real-environment proof.

## What This Changes

Adds a Linux x64 `vipsthumbnail` fallback in `src/media/image-ops.ts` when Sharp-backed JPEG resizing fails, including a new `OPENCLAW_IMAGE_BACKEND=vips` preference path.

## Best Possible Solution

Rework the Linux fallback to use the current temp-workspace pattern, the documented libvips output contract, decoded-pixel and no-enlarge parity with Sharp, focused tests, and real proof from the affected environment.

## Reproduction Assessment

Yes for source-level review: forcing the Sharp-backed resize path to fail on Linux/non-sips still makes `optimizeImageToJpeg()` exhaust resize attempts and throw `Failed to optimize image`. The exact SSE2-only KVM failure remains reporter and PR-author verified rather than live hardware reproduced here.

## Solution Assessment

No. A Linux fallback is likely the right direction, but this patch uses the wrong vipsthumbnail quality contract, misses decoder/no-enlarge parity, and still needs backend policy approval.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Pass JPEG quality through the vips output path:** `src/media/image-ops.ts:486`
  - body: `vipsthumbnail` does not support `-Q` as a standalone option, so this command will fail before producing the fallback JPEG. Put the quality setting in the output saver spec, for example `out.jpg[Q=85]`, or use a supported libvips API.
  - confidence: 0.94
- **[P2] Keep vips decode under the pixel guard:** `src/media/image-ops.ts:486-487`
  - body: The Sharp backend enforces `limitInputPixels` at decode time, but this path can invoke `vipsthumbnail` after only the header preflight. Inputs whose dimensions are not positively bounded by that preflight need an equivalent decoded-pixel guard before the external decoder runs.
  - confidence: 0.84
- **[P2] Honor no-enlarge behavior in the vips backend:** `src/media/image-ops.ts:486`
  - body: `resizeToJpeg()` defaults `withoutEnlargement` to true through Sharp, while `vipsthumbnail -s N` normally upsizes or downsizes to fit the target. Use a downsize-only size restriction unless `params.withoutEnlargement === false` so direct media resize callers do not get enlarged images when Sharp fails.
  - confidence: 0.8
