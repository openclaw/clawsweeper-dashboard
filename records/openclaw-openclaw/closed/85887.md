## Summary

Current main and the v2026.5.22 release already fail closed with a clear diagnostic when an explicit tool allowlist leaves no callable tools, and the current code also preserves the explicitly allowed `message` tool in embedded runs.

## What This Changes

The issue reports that invalid or unavailable `agents.list.main.tools.allow` entries can make the agent unreachable without a clear error.

## Best Possible Solution

Keep the shipped v2026.5.22 behavior and ask affected 2026.5.19 users to upgrade; any new report should include the exact allowlist and gateway warning/error output.

## Reproduction Assessment

No high-confidence current-main failing reproduction remains. Source and shipped tests show explicit allowlists with no callable tools now produce a preflight error and warning instead of silently sending the run onward.

## Solution Assessment

Yes. The current implementation fails before prompt submission, names the offending allowlist source, and keeps explicitly allowed `message` available, which is the narrow maintainable fix for this report.
