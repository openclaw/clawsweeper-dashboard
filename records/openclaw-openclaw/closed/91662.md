## Summary

Close as implemented on current main: the reported v2026.6.1 failure is real in the release tag, but main now normalizes successful missing-content-type streamed OpenAI/Codex responses after sniffing the body and has targeted regression coverage; the fix has not shipped in a release yet.

## What This Changes

The issue asks OpenClaw to tolerate valid SSE from the native Codex Responses backend when the HTTP response is OK, has a readable body, and lacks a content-type header.

## Best Possible Solution

Keep the current main transport normalization and let the next release carry it; no separate OpenClaw source change remains for this issue.

## Reproduction Assessment

Yes from source against v2026.6.1: the release tag rejects OK streamed responses with missing content-type before parsing. Current main no longer has that failure path, so this is not a current-main failing repro.

## Solution Assessment

Yes. The main fix is at the shared guarded fetch boundary, normalizes only successful streamed OpenAI-compatible responses after body sniffing, and still rejects HTML or unknown bodies.
