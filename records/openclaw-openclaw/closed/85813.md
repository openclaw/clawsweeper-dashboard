## Label Justifications

- `P1`: The shipped behavior can duplicate a user-visible inbound request through Codex failover after a long-running turn timeout.
- `impact:message-loss`: The report is about duplicate/replayed channel-visible message handling after a Codex timeout.
- `impact:session-state`: The stale original user input can be injected into a fresh Codex turn, causing transcript/session state drift.

## Summary

Current main already treats Codex app-server turn-completion idle timeouts as app-server failures, stops before generic auth/model failover can replay the same user input, and maps the error to specific no-replay user-facing copy. The report matches v2026.5.20 shipped behavior, but the fix is present only on current main so far.

## What This Changes

The issue asks OpenClaw to stop replaying the original Codex user input when an app-server turn timeout triggers retry/profile rotation, and to clarify the timeout behavior.

## Best Possible Solution

Keep the current main app-server-failure boundary and let the next release ship it; if timeout-tuning confusion still reproduces on main, track that as a narrower docs/config issue.

## Reproduction Assessment

Yes, at source level: v2026.5.20 routes the timed-out app-server prompt error into assistant failover, while current main has a focused regression test showing that path no longer reaches fallback. I did not run a live 10-minute Telegram/Codex turn in this read-only review.

## Solution Assessment

Yes. The current fix sits at the Codex app-server failure boundary and prevents replay before generic auth/model failover, which is narrower and more maintainable than adding Telegram-specific suppression.
