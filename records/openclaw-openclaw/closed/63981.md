## Summary

Current main and the shipped release already solve the observable macOS sleep/wake no-listen failure that this PR targets: the Gateway binds HTTP before post-attach sidecar/network work and exposes pending sidecars through readiness instead of leaving port 18789 unopened. The PR's exact run-loop deadline is not on main, but the linked user problem is already covered, and the linked bug report was closed after the same source/release check.

## What This Changes

The PR adds a 60-second timeout around `params.start()` in the Gateway run loop and adds Vitest coverage for first-start timeout behavior plus late-resolving startup cleanup.

## Best Possible Solution

Keep the shipped listen-before-sidecars startup model and readiness signaling as the canonical fix for the closed sleep/wake report; if maintainers still want a generic Gateway startup deadline, track that as a separate product/lifecycle policy item.
