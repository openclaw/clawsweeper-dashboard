## Summary

Current main already solves PR #40253's central problem: Telegram inbound user turns flow through the shared session transcript path and are surfaced to Control UI/WebChat via `session.message` and `sessions.changed`, rather than the PR's proposed new `chat` event state. The linked issue #36640 is closed, and the PR's own review discussion identified protocol/UI risks in the `state: "inbound"` approach.

## Best Possible Solution

Close PR #40253 as implemented on main. Keep the shipped `session.message`/`sessions.changed` transcript event stream as the canonical WebChat/Control UI live-display path, and do not merge the PR's `onInboundMessage`/`chat state: "inbound"` approach.
