## Summary

Keep open: this is a maintainer-authored issue and the exact message-tool variant remains source-reproducible on current main. Current OpenClaw still lets `disableMessageTool` remove `message` from the Codex dynamic tool catalog, while the Codex app-server binding logic treats non-empty dynamic-tool fingerprint changes as thread-incompatible.

## What This Changes

The issue reports Codex native thread churn when a temporary subagent completion announcement turn disables the `message` tool and changes the Codex dynamic-tool catalog.

## Best Possible Solution

Keep this issue open and fix the Codex app-server path so temporary `message` delivery policy does not change thread schema identity while message sends remain guarded for internal turns.

## Reproduction Assessment

Yes, at source level: the Gateway accepts and forwards `disableMessageTool`, OpenClaw drops `message` from the tool list, and Codex thread lifecycle clears bindings on non-empty dynamic-tool fingerprint mismatches. I did not run the live Telegram/Codex three-turn repro in this read-only review.

## Solution Assessment

No, current main is not the best solved state because it still ties native thread compatibility to this temporary per-turn delivery-policy difference. The best fix is a stable thread-level Codex schema with per-turn message-tool execution/delivery enforcement, unless Codex adds a supported resume-time dynamic-tool update contract.
