- `impact:security`: The issue concerns an authorization boundary mismatch between Discord native command handling and shared built-in command auth.

## Summary

Keep open: current main still has a source-reproducible Discord native command auth mismatch where the argument-button UI can acknowledge a `/think` choice before the shared built-in command authorization result is known.

## What This Changes

The issue asks Discord native built-in slash commands and argument selections to use final shared command authorization before showing success-like acknowledgements or dispatching directive-only commands.

## Best Possible Solution

Resolve the shared command authorization before Discord native built-ins show success-like UI or dispatch, and make argument-button UI report the actual command result instead of an optimistic selection.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a Discord sender can pass the native channel/guild gate while shared command auth fails under owner-restricted command settings, then `/think` argument selection is acknowledged before that final result is known.

## Solution Assessment

Yes. The best fix is to make Discord native built-ins consult or carry the shared command authorization result before acknowledgement and to add regression coverage for directive-only commands such as `/think`.
