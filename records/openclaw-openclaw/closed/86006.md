## Summary

Close as not actionable in the OpenClaw source repository: the reported auto-close behavior is owned by the separate openclaw/clawsweeper apply workflow, while this repo only forwards OpenClaw issue and PR events to that automation. ## What This Changes

The issue reports that ClawSweeper auto-closes PRs even when its durable review marker says the PR needs human review.

## Best Possible Solution

Track the marker-versus-close-apply contract in openclaw/clawsweeper and keep OpenClaw’s workflow as a thin dispatcher.

## Reproduction Assessment

Yes for the external ClawSweeper path: the provided https://github.com/openclaw/openclaw/pull/85972 evidence shows the needs-human marker coexisting with a close-applied marker, and openclaw/clawsweeper source shows how that can happen. It is not reproducible as an OpenClaw source bug because this repo only dispatches events to ClawSweeper.

## Solution Assessment

No as an OpenClaw source change: editing this repo would not fix the apply lane that posts the close-applied marker. The maintainable fix is in openclaw/clawsweeper, either by changing the apply gate or clarifying that repair verdict markers are separate from close proposals.
