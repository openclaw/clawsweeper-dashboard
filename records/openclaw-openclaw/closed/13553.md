## Summary

Current main already implements the requested workflow: WhatsApp and BlueBubbles inbound attachments are saved into OpenClaw-managed inbound media storage and surfaced back as local paths, while legacy iMessage already passes validated attachment file paths through inbound context for later file-send/upload actions.

## Evidence

- **WhatsApp saves inbound attachments and returns a local mediaPath:** The WhatsApp inbound monitor downloads inbound media, writes it through the shared media store, and sets `mediaPath`/`mediaFileName` on the inbound message object.
  - file: [extensions/whatsapp/src/inbound/monitor.ts:484](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/extensions/whatsapp/src/inbound/monitor.ts#L484)
  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)
- **Shared media store writes inbound files under the managed media directory:** `saveMediaBuffer()` stores files under `path.join(resolveConfigDir(), "media", subdir)`, so inbound attachments are persisted in OpenClaw-managed storage rather than an ad hoc temp folder.
  - file: [src/media/store.ts:15](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/media/store.ts#L15)
  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)
- **Managed media roots are allowed for later local attachment sends:** Default local media roots include the preferred OpenClaw temp dir plus both config/state `media` directories, which is the path family used for later upload/send actions.
  - file: [src/media/local-roots.ts:30](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/media/local-roots.ts#L30)
  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)
- **Legacy iMessage already exposes attachment paths from validated roots:** The iMessage monitor filters attachments against configured/default attachment roots, keeps `mediaPath`/`mediaPaths`, and forwards them into inbound context. Default roots include `/Users/*/Library/Messages/Attachments`.
  - file: [extensions/imessage/src/monitor/monitor-provider.ts:249](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/extensions/imessage/src/monitor/monitor-provider.ts#L249)

  - file: [extensions/bluebubbles/src/monitor-processing.ts:1132](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/extensions/bluebubbles/src/monitor-processing.ts#L1132)
  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)

  - file: [src/infra/outbound/message-action-runner.media.test.ts:420](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/infra/outbound/message-action-runner.media.test.ts#L420)

## Risks / Open Questions

- The request named `~/clawd/temp`, but current OpenClaw uses managed media roots (`~/.openclaw/.../media` and validated iMessage attachment roots) instead of that exact legacy path.

## Close Comment

Closing from a Codex review because current main already implements this well enough. WhatsApp inbound media is downloaded, saved through the shared inbound media store, and surfaced back as `mediaPath`/`mediaFileName` (`extensions/whatsapp/src/inbound/monitor.ts:484`). BlueBubbles does the same and exposes `MediaPath`/`MediaPaths` in inbound context (`extensions/bluebubbles/src/monitor-processing.ts:1132`, `extensions/bluebubbles/src/monitor-processing.ts:1536`). Legacy iMessage already forwards validated attachment paths from configured/default attachment roots into inbound context (`extensions/imessage/src/monitor/monitor-provider.ts:249`, `extensions/imessage/src/monitor/monitor-provider.ts:376`, `extensions/imessage/src/media-contract.ts:5`). Those saved media locations are part of the default allowed local media roots (`src/media/local-roots.ts:30`), and `upload-file` is already the canonical file-send action (`src/infra/outbound/message-action-runner.media.test.ts:420`, `docs/channels/bluebubbles.md:376`). This is also present in release `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`).
