## Summary

Current `main` already covers the reported per-topic fixed-context use case via Telegram topic-level `agentId` routing to dedicated agents/workspaces, with workspace bootstrap context injected into sessions.

## Evidence

- **Telegram topics can route to a specific agent:** Topic config now exposes `agentId` with the documented behavior that it overrides group/binding routing for that topic.
  - file: [src/config/types.telegram.ts:239](https://github.com/openclaw/openclaw/blob/bcea5e75ebc7485d51655f43acdea9e5d368080f/src/config/types.telegram.ts#L239)
  - sha: [bcea5e75ebc7](https://github.com/openclaw/openclaw/commit/bcea5e75ebc7485d51655f43acdea9e5d368080f)
- **Runtime applies topic agent override to session routing:** When `topicAgentId` is set, Telegram routing rebuilds the session key and main session key for that agent, so the topic gets its own agent/session bucket.
  - file: [extensions/telegram/src/conversation-route.ts:59](https://github.com/openclaw/openclaw/blob/bcea5e75ebc7485d51655f43acdea9e5d368080f/extensions/telegram/src/conversation-route.ts#L59)

  - file: [docs/channels/telegram.md:492](https://github.com/openclaw/openclaw/blob/bcea5e75ebc7485d51655f43acdea9e5d368080f/docs/channels/telegram.md#L492)
