## Summary

Keep open: the reported Signal allowlist bug is source-reproducible, but this PR still updates the helper path rather than the current DM ingress access resolver, lacks after-fix real behavior proof, and appears to add live contact identifiers to a test fixture.

## What This Changes

The PR adds Signal sender alias fields, matches `allowFrom` entries against sender UUID/phone aliases in `isSignalSenderAllowed`, adds helper-level tests, and adds a changelog entry.

## Best Possible Solution

Carry both Signal identity forms into `resolveSignalAccessState` subject aliases, cover that access path with a regression test, replace apparent live identifiers with synthetic fixtures, and collect redacted after-fix proof from a patched Signal gateway.

## Reproduction Assessment

Yes, from source inspection rather than a live run here. Current main turns a both-fields Signal envelope into a phone-only sender and then builds an ingress subject without the UUID alias, so a UUID allow entry cannot match through the current DM access path.

## Solution Assessment

No. The helper-level alias match is useful but not sufficient because current DM pairing decisions run through `resolveSignalAccessState`; the narrow maintainable fix should update the ingress subject aliases and add access-path coverage.
