## Summary

Keep this PR open: the bug is real and the branch is pointed at the right heartbeat/user-visible surface, but the current diff still introduces sanitizer regressions and the real behavior proof is stale for the latest head.

## What This Changes

The branch adds Codex XML scaffolding stripping to assistant-visible/user-facing sanitizers and heartbeat delivery paths, requires `heartbeat_respond` in heartbeat-tool mode, and adds regression tests.

## Best Possible Solution

Land one focused heartbeat sanitizer fix that preserves full wrapped tool-call block stripping, handles `<file_contents>` close tags robustly, and includes latest-head real behavior proof for both legacy and heartbeat-tool paths.

## Reproduction Assessment

Yes, at source level. Current main sends heartbeat reply text through normalization paths that do not apply the canonical assistant-visible sanitizer, and https://github.com/openclaw/openclaw/issues/81369 supplies concrete leaked scaffolding examples.

## Solution Assessment

No, not as written. Reusing the canonical sanitizer is the right direction, but marker-only cleanup must not run before full wrapped-block stripping, `<file_contents>` close tags need safer handling, and latest-head proof needs to cover the added heartbeat-tool behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve full wrapped tool-call stripping:** `src/shared/text/assistant-visible-text.ts:760`
  - body: The new scaffolding stage runs before `stripToolCallXmlTags`. For a wrapped block with the tag on its own line, such as `<tool_calls>\n[{...}]\n</tool_calls>`, this line removes only the wrappers; the JSON then bypasses the existing block stripper and can be delivered as visible text. Keep full XML block stripping before marker-only cleanup, or make this pass remove the whole wrapper block.
  - confidence: 0.93
- **[P2] Honor file_contents close tags inside copied fences:** `src/shared/text/assistant-visible-text.ts:644-645`
  - body: When a `<file_contents>` block contains a copied Markdown fence, `findCodeRegions` can mark the real close tag as code. This `continue` skips the close, keeps `inFileContentsBlock` true, and drops visible text after the block. Once inside a file contents block, honor its closing tag even if the copied excerpt contains fences.
  - confidence: 0.84

## Security Review

Status: needs_attention
