## Summary

Current main already routes Ollama memory embeddings through direct config/env resolution, includes regression coverage for `OLLAMA_API_KEY` fallback, and the same implementation is present in the shipped `v2026.4.23` release. The issue's reported `buildApiKeyCredential`/auth-profile failure path does not match the current or shipped Ollama memory-embedding code.

## Evidence

- **Memory-core Ollama adapter uses the direct embedding provider:** The Ollama memory embedding adapter calls `createOllamaEmbeddingProvider(...)` directly for memory-core embeddings, so this path does not go through provider-auth profile setup.
  - file: [extensions/ollama/src/memory-embedding-adapter.ts:12](https://github.com/openclaw/openclaw/blob/2011de69d3dc8f1a11ebc774ba4af23f54892461/extensions/ollama/src/memory-embedding-adapter.ts#L12)
  - command: `nl -ba extensions/ollama/src/memory-embedding-adapter.ts | sed -n '1,40p'`
  - sha: [2011de69d3dc](https://github.com/openclaw/openclaw/commit/2011de69d3dc8f1a11ebc774ba4af23f54892461)
- **Ollama embedding auth resolves from remote config, provider config, then env:** `createOllamaEmbeddingProvider` resolves auth in-order from `agents.*.memorySearch.remote.apiKey`, then `models.providers.ollama.apiKey`, then `resolveEnvApiKey("ollama")`; there is no `buildApiKeyCredential(...)` or auth-profile lookup in this runtime path.
  - file: [extensions/ollama/src/embedding-provider.ts:101](https://github.com/openclaw/openclaw/blob/2011de69d3dc8f1a11ebc774ba4af23f54892461/extensions/ollama/src/embedding-provider.ts#L101)

  - file: [extensions/ollama/src/embedding-provider.test.ts:118](https://github.com/openclaw/openclaw/blob/2011de69d3dc8f1a11ebc774ba4af23f54892461/extensions/ollama/src/embedding-provider.test.ts#L118)
