## Summary

Keep open: the docs gap from the previous review is fixed, but the latest patch still passes the outer evaluate timeout as an ignored second argument to `callBrowserAct`, so the advertised long-running CLI behavior can still hit the old request timeout.

## What This Changes

The PR adds `--timeout-ms` to `openclaw browser evaluate`, forwards a timeout into the evaluate action body, documents the option, and adds a focused CLI parsing test.

## Best Possible Solution

Keep the docs and test coverage, but pass `timeoutMs` through the existing `callBrowserAct` params object so the helper applies request slack above the browser-side evaluate timeout.

## Reproduction Assessment

Yes for the PR defect: source inspection shows `callBrowserAct` reads `params.timeoutMs`, while the patch passes the evaluate timeout as a second positional argument. No full runtime repro was needed to see that the outer request timeout is ignored.

## Solution Assessment

No as currently patched. The maintainable fix is small: keep using the existing evaluate action `timeoutMs` contract, but put the outer timeout on the first `callBrowserAct` argument like the helper expects.
