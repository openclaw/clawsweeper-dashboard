## Summary

This is the same remaining browser Control UI/WebChat routing bug already tracked by open issue #45028: Control UI sends stay on the internal webchat route instead of returning to the externally backed channel session. The WeChat plugin detail is useful reproduction context, but the failing decision point is generic core Gateway routing, not WeChat-specific plugin code.

## Best Possible Solution

Close this issue as a duplicate of open issue #45028, keep the WeChat/openclaw-weixin reproduction as useful context there, and route implementation through the canonical Control UI-to-external-channel routing discussion. If maintainers choose to support this behavior, the fix should be a generic explicit external-delivery affordance in the Control UI/Gateway route boundary, with tests preserving the existing WebChat leakage safeguards.
