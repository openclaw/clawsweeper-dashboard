## Summary

Keep open. Current main has partial improvements, including lightweight daemon status probing and clearer read-probe wording, but it does not fully close this report: full `gateway probe` can still mark a post-connect read timeout as `ok: false`, and the reachability helper only treats missing-scope read failures as reachable. The newer #73535 report is clearly related and may become the canonical consolidation point, but it is not identical enough to close this Windows loopback probe report without maintainer judgment.

## Best Possible Solution

Keep this issue open, or have a maintainer explicitly consolidate it under #73535 if that becomes the canonical tracking issue. The likely product/code direction is to align `gateway probe` JSON, text, and exit-code semantics with the docs: once WebSocket hello succeeds, classify post-connect detail timeout/normal close as reachable-but-degraded or a probe-budget/read-probe failure unless there is a real connect/auth failure. Add regression tests for post-connect timeout/close and top-level probe exit behavior, while preserving `--require-rpc` for scripts that need read-scope proof.
