## Summary

Current OpenClaw already normalizes Telegram captioned-media text before it reaches the gateway message sanitizer, and the shipped Telegram tests cover captioned media/body extraction. The reported `[object Object]` behavior matches an older build, not current `main` or the latest release.

## Evidence

- **Telegram caption extraction is string-guarded:** `resolveTelegramTextContent()` only accepts `text`/`caption` when they are strings and otherwise falls back to `""`, so non-string caption payloads are not coerced into `[object Object]`. `getTelegramTextParts()` then uses that normalized string.

## Risks / Open Questions

- The report may have been valid on older builds such as 2026.4.2; this close relies on current `main` and the shipped `v2026.4.23` codepath rather than reproducing against that older revision.
- `sanitizeChatSendMessageInput()` is still string-only, so separate non-Telegram callers could still need a broader hardening change even though this Telegram-specific issue is already handled upstream in the plugin.
