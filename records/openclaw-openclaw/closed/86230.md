## Summary

Keep open: current main and v2026.5.22 still advertise `--at` as accepting `+duration`, while the shared `parseAt` path only accepts bare duration tokens, so the reported CLI mismatch remains real and narrowly fixable.

## What This Changes

Align `openclaw cron add --at` help and validation so documented `+duration` one-shot schedules work or the help stops advertising them.

## Best Possible Solution

Accept an optional leading `+` for `--at` relative durations, preserve bare-duration behavior, leave `--every` unchanged, and add a regression test plus changelog entry for the user-visible CLI fix.

## Reproduction Assessment

Yes, source-reproducible: current main advertises `+duration`, `parseAt` routes relative values through a digit-only duration regex, and the error path reports the same `duration like 20m` message from the issue. I did not run the mutating `cron add` command in this read-only review.

## Solution Assessment

Yes, accepting `+duration` in the `--at` parser is the best narrow fix because the help text already shipped that contract and the change is backward-compatible with bare durations. If maintainers prefer not to expand input syntax, the fallback is to remove `+duration` from help, but that would not repair scripts that followed the existing help.
