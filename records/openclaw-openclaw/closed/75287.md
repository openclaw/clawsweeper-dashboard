## Summary

Keep open. Current main still lets unowned CLI roots enter proxy startup and broad lazy plugin CLI loading, and #76379 is an open closing PR that should be reviewed or landed before this issue is closed.

## What This Changes

Report that unknown CLI commands such as `openclaw foo` initialize proxy/plugin loading, can leave a hot child process, and correlate with gateway/provider-load saturation.

## Best Possible Solution

Review and land #76379 or an equivalent narrow fix that rejects unowned CLI roots before proxy startup and plugin runtime loading, then track any remaining gateway saturation as narrower follow-up work.

## Reproduction Assessment

Yes, at source level. The reporter supplied process-group repro logs for `openclaw foo`, and current main still contains tests and source paths that route unowned roots through proxy startup and broad plugin registration; I did not run a live CLI smoke in this read-only review.

## Solution Assessment

Yes. The linked PR direction is the narrow maintainable fix for the central CLI failure: validate command ownership before managed proxy startup and plugin runtime loading, while preserving separate gateway/model-list mitigations already on main.
