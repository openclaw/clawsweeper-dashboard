## Summary

Keep open. Current main still has source-reproducible Feishu degraded approval text paths: document-comment payloads drop approval button command values, and shared fallback builders still render a literal `/approve <id>` placeholder; the open interpolation PR fixes only the generic placeholder piece, not Feishu degraded-surface guidance.

## What This Changes

The issue requests clearer Feishu approval fallback guidance when inline approval buttons are unavailable or approval content degrades to text.

## Best Possible Solution

Keep native Feishu cards for supported chats, but make degraded Feishu text fallbacks explain that buttons are unavailable on the current surface and include exact allowed `/approve <id> <decision>` commands, reusing PR https://github.com/openclaw/openclaw/pull/78793 if it lands first.

## Reproduction Assessment

Yes, at source level. The Feishu document-comment path converts an approval-style interactive payload to label-only text, and the generic fallback builders still emit a literal `/approve <id>` placeholder; I did not run tests or live Feishu because this was a read-only review.

## Solution Assessment

Yes. A focused fallback-rendering/copy repair is the narrowest maintainable path: reuse the existing approval button values and documented `/approve` contract without adding a new config knob or native approval transport.
