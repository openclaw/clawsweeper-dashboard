## Summary

Close as implemented on current main: the merged Telegram startup fix removes the reported pre-polling stall by reusing the startup bot identity and allowing polling to proceed after recoverable webhook-cleanup failures.

## What This Changes

Report that Telegram long polling in 2026.5.2 can stay silent on high-RTT hosts after pre-poll getMe/menu/webhook setup timeouts, unlike 2026.4.29.

## Best Possible Solution

Keep the merged #76735 behavior: reuse startup botInfo, avoid extra pre-poll Bot API confirmation, and let recoverable deleteWebhook failures fall through to getUpdates conflict handling.

## Reproduction Assessment

No current-main failure remains to reproduce. The prior 5.2 failure mode is represented by regression coverage that forces webhook-cleanup timeout and confirms polling starts.

## Solution Assessment

Yes. The current fix targets the startup gates directly without adding a new config knob, and keeps getUpdates as the authoritative check for remaining webhook conflicts.
