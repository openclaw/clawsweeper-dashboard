## Summary

Keep open: current main and v2026.5.28 still synthesize `lmstudio-local` into LM Studio provider config while config audit treats any non-empty `models.providers.*.apiKey` string as plaintext. The protected `security` label also requires maintainer/security handling rather than cleanup close.

## What This Changes

The issue asks to stop LM Studio’s keyless local default from writing a non-secret `apiKey` placeholder that security audit/startup checks report as plaintext.

## Best Possible Solution

Align LM Studio keyless setup and the config/security-audit path so bundled-declared non-secret markers do not count as plaintext, with focused tests for LM Studio setup, openclaw.json audit, and startup warning behavior.

## Reproduction Assessment

Yes, source-reproducible: current setup code writes `lmstudio-local` for keyless LM Studio provider config, and the config audit path treats non-empty `models.providers.*.apiKey` strings as plaintext. I did not run the CLI because this review is read-only.

## Solution Assessment

Unclear that simply removing `apiKey` is the safest fix because current runtime tests still accept the synthetic marker. The maintainable direction is to either stop persisting the marker with runtime proof, or make config audit honor bundled non-secret markers as narrowly as the models.json audit already does.
