## Summary

Keep this PR open, but do not merge it as-is: it fixes the reported npm shrinkwrap status path, yet the shared detector change also makes source/git checkouts with the repository's publish shrinkwrap prefer npm over the documented pnpm build manager.

## What This Changes

The PR changes package-manager detection and update dependency status so `npm-shrinkwrap.json` is treated as npm evidence and as the npm lock marker, with focused regression tests for shrinkwrap package roots.

## Best Possible Solution

Make shrinkwrap precedence apply only to published package-install status/deps detection while preserving pnpm `packageManager` precedence for source/git build-manager paths, then add real npm package-root proof.

## Reproduction Assessment

Yes, source-level: current main reads `packageManager` before lockfile markers, and the linked report shows a published npm package root with pnpm metadata plus `npm-shrinkwrap.json`. The PR-introduced regression is also source-reproducible because the repo root has both shrinkwrap and pnpm metadata while the changed detector returns npm first.

## Solution Assessment

No. The patch fixes the reported npm package shape in status, but the broad shared-detector precedence change is not the narrowest maintainable fix; the safer path is package-install-specific shrinkwrap handling or an explicit detector mode.

Full review comments:

- **[P1] Keep shrinkwrap from overriding source checkout managers:** `src/infra/detect-package-manager.ts:8-9`
  - body: This shared detector now returns `npm` for any root with `npm-shrinkwrap.json` before reading `packageManager`. The OpenClaw source root has both publish shrinkwrap and `packageManager: pnpm`, and `resolveUpdateBuildManager` uses this detector for git/source updates; after this change, git updates can run npm install/build commands in a pnpm workspace instead of requiring or bootstrapping pnpm. Please keep shrinkwrap precedence scoped to package-install status detection, or make this detector context-aware so source/build-manager callers still honor `packageManager`.
  - confidence: 0.94
