## Summary

Current `main` already has the outbound channel-resolution and WhatsApp media-path fixes that match this report, so this issue is implemented on `main` but not yet in the latest tagged release.

## Evidence

- **Outbound resolver now checks the active plugin registry directly:** `resolveOutboundChannelPlugin` first normalizes the requested channel, then returns a plugin from the active registry before falling back, which addresses the multi-channel case where the requested channel existed but was missing from the loaded registry path.
  - file: [src/infra/outbound/channel-resolution.ts:36](https://github.com/openclaw/openclaw/blob/6a61cb73c5622749d7940fd8d8b90c93adbb5081/src/infra/outbound/channel-resolution.ts#L36)

  - sha: [6a61cb73c562](https://github.com/openclaw/openclaw/commit/6a61cb73c5622749d7940fd8d8b90c93adbb5081)
- **Regression test covers the reported multi-channel failure mode:** There is an explicit test asserting the resolver bootstraps the requested channel when the active registry already contains a different channel, which is the same shape as a WhatsApp+Discord setup failing on explicit WhatsApp sends.
  - file: [src/infra/outbound/channel-resolution.test.ts:166](https://github.com/openclaw/openclaw/blob/6a61cb73c5622749d7940fd8d8b90c93adbb5081/src/infra/outbound/channel-resolution.test.ts#L166)

- **CLI/tool send path uses the fixed resolver:** `sendMessage` still throws `Unknown channel` only if `resolveOutboundChannelPlugin` returns nothing, so the new resolver behavior applies directly to CLI and shared outbound send flows.
