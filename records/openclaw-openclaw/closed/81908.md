## Summary

Keep open because the current main and latest release still have a source-plausible unbounded wait through the Telegram delete retry path, and an open linked PR with closing syntax should be reviewed rather than closing the issue early.

## What This Changes

The issue asks for Telegram message deletion to surface a timeout/error instead of hanging when the underlying Telegram Bot API call does not settle.

## Best Possible Solution

Land or revise the linked fix PR so Telegram/channel control requests are bounded without undermining existing Telegram timeout configuration or retry safety for non-idempotent sends.

## Reproduction Assessment

No high-confidence live reproduction was established in this review. Source inspection shows that if `api.deleteMessage` never settles, current main continues awaiting it through `requestWithDiag` and `retryAsync`, but a real Telegram DM delete was not run here.

## Solution Assessment

Unclear whether a generic `retryAsync` timeout is the best solution. The safer maintainable direction is a bounded Telegram/channel API timeout that respects configured Telegram request timeouts and avoids duplicate sends after ambiguous network timeouts.
