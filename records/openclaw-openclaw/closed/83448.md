## Label Justifications

- `P1`: The bug makes gateway `/reset` appear to clear context while the next CLI turn can silently resume the old provider-side session.
- `merge-risk: 🚨 compatibility`: Merging changes reset behavior for all CLI providers on non-subagent gateway sessions, which can affect existing setups that relied on resume after reset.
- `merge-risk: 🚨 session-state`: The diff deliberately changes persisted session binding state and provider-side resume continuity after reset.

## Summary

Keep open for maintainer review. The bug is source-reproducible on current main and the patch looks correct, but clearing every CLI provider binding on non-subagent gateway reset is an upgrade-visible session-state contract that maintainers should explicitly accept before merge.

## What This Changes

The PR clears CLI resume bindings for non-subagent gateway `sessions.reset`, preserves spawned-subagent bindings, adds reset regression tests, updates the changelog, and isolates one outbound test dependency.

## Best Possible Solution

Land the non-subagent gateway reset fix with regression coverage after maintainers accept provider-wide CLI reset semantics, while preserving spawned-subagent continuity and leaving hard-reset/write-side stale-binding work in their related threads.

## Reproduction Assessment

Yes at source level: current main copies the CLI binding fields during gateway reset, and the CLI runner later uses those fields to pass resume state. I did not run a live gateway reset in this read-only review, but the PR body supplies redacted before/after live output.

## Solution Assessment

Yes, with one maintainer choice: using the existing `clearAllCliSessions` helper behind the canonical subagent-key guard is a narrow fix for user-facing gateway resets. The remaining decision is whether provider-wide CLI clearing is the intended reset contract or whether this should be Claude-only first.
