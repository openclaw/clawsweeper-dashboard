## Summary

Current `main` already implements automatic handling for transient provider/API connection failures: model-provider calls get retry behavior, agent turns run through model/auth-profile fallback, connection/network errors are classified into that retry/failover path, lifecycle `error` events are emitted, and assistant error turns are exposed through session history tooling. The issue’s described `Connection error.`-only behavior no longer matches current `main`.

## Evidence

- **Retry behavior is documented for model providers and transient channel errors:** Docs say model providers use normal short SDK retries, long retry-after waits are surfaced so model failover can rotate to another auth profile or fallback model, and Telegram retries transient timeout/connect/reset/unavailable errors.
  - file: [docs/concepts/retry.md:28](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/docs/concepts/retry.md#L28)
  - sha: [cec34821754a](https://github.com/openclaw/openclaw/commit/cec34821754ad935c780805f3116f99449d46ac2)
- **Long retry-after responses are converted into immediate failover signals:** The guarded model fetch detects retryable 408/409/429/5xx responses over the configured cap and rewrites them with `x-should-retry: false`, so control returns to OpenClaw’s fallback logic instead of stalling.
  - file: [src/agents/provider-transport-fetch.ts:58](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/src/agents/provider-transport-fetch.ts#L58)
  - sha: [cec34821754a](https://github.com/openclaw/openclaw/commit/cec34821754ad935c780805f3116f99449d46ac2)
- **Agent turns are wrapped in model fallback handling:** `agent-command` runs each turn through `runWithModelFallback(...)`, passing the provider/model overrides into the actual attempt runner and treating later attempts as fallback retries.
  - file: [src/agents/agent-command.ts:870](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/src/agents/agent-command.ts#L870)
  - sha: [cec34821754a](https://github.com/openclaw/openclaw/commit/cec34821754ad935c780805f3116f99449d46ac2)
- **Connection/API failures are classified as retryable timeout/failover reasons:** Current tests assert that `ETIMEDOUT`, `ECONNREFUSED`, `ECONNRESET`, `EAI_AGAIN`, `Connection error.`, `fetch failed`, and similar network failures are treated as `timeout`, which is the failover/retry lane.
  - file: [src/agents/failover-error.test.ts:512](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/src/agents/failover-error.test.ts#L512)
  - sha: [cec34821754a](https://github.com/openclaw/openclaw/commit/cec34821754ad935c780805f3116f99449d46ac2)
- **Fallback succeeds after timeout-class failures:** `runWithModelFallback` is covered by a test where an `ETIMEDOUT` failure on the first model retries via fallback and succeeds on the next candidate.
  - file: [src/agents/model-fallback.test.ts:1271](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/src/agents/model-fallback.test.ts#L1271)
  - sha: [cec34821754a](https://github.com/openclaw/openclaw/commit/cec34821754ad935c780805f3116f99449d46ac2)
- **Embedded-run fallback covers service-unavailable failures without persisting a hard cooldown:** Current end-to-end fallback coverage treats `LLM error: service unavailable` as a timeout-lane failure and successfully switches to a fallback provider.
  - file: [src/agents/model-fallback.run-embedded.e2e.test.ts:599](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/src/agents/model-fallback.run-embedded.e2e.test.ts#L599)
  - sha: [cec34821754a](https://github.com/openclaw/openclaw/commit/cec34821754ad935c780805f3116f99449d46ac2)
- **Lifecycle error events already exist:** When the terminal assistant turn is an error, OpenClaw emits a lifecycle event with `phase: "error"` and the user-facing error text.
  - file: [src/agents/pi-embedded-subscribe.handlers.lifecycle.ts:113](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/src/agents/pi-embedded-subscribe.handlers.lifecycle.ts#L113)
  - sha: [cec34821754a](https://github.com/openclaw/openclaw/commit/cec34821754ad935c780805f3116f99449d46ac2)
- **Assistant error turns are exposed through history tooling:** History text extraction explicitly looks at assistant transcript entries, checks `stopReason === "error"`, and rewrites/sanitizes the stored assistant error text for history consumers; tests cover this behavior.
  - file: [src/agents/tools/chat-history-text.ts:58](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/src/agents/tools/chat-history-text.ts#L58)
  - sha: [cec34821754a](https://github.com/openclaw/openclaw/commit/cec34821754ad935c780805f3116f99449d46ac2)
