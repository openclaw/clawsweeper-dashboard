## Label Justifications

- `P2`: The original failure affected an advanced opt-in Codex app-server mode with a workaround and is now covered by shipped managed Codex behavior.
- `impact:crash-loop`: The reported symptom was the Codex app-server client closing before turn completion, making the command path unavailable for that run.
- `impact:message-loss`: The report explicitly noted Telegram turns needing command/tool execution could lose the visible reply when the app-server turn failed.

## Summary

Current main and shipped v2026.6.6 cover the reported managed Codex app-server path: the Codex plugin now ships @openai/codex 0.139.0, keeps OpenClaw tools registered as deferred dynamic tools for code-mode-only, documents the `item/tool/call` bridge, and has bounded recovery for replay-safe app-server client-close failures.

## What This Changes

The issue reports native Codex app-server command/tool execution ending before completion when `appServer.codeModeOnly` enables `features.code_mode_only`.

## Best Possible Solution

Keep the v2026.6.6-or-newer managed Codex implementation; users who deliberately override `appServer.command` to an older Codex 0.130-era binary should remove that override or upgrade Codex instead of carrying an OpenClaw wrapper.

Yes, at source/report level: the issue gives a focused beta.7 command smoke and before/after `code_mode_only` probe, and the old failure signature matches the app-server client-close path. I did not run a live managed Codex app-server repro during this read-only review.

## Solution Assessment

Yes. The best current solution is the shipped managed Codex path: Codex 0.139.0 supports code-mode-only nested tool calls, OpenClaw registers deferred dynamic tools for the app-server bridge, and app-server client-close recovery is bounded rather than wrapper-based.
