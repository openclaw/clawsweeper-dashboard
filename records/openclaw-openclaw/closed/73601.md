## Summary

Current main now implements the Google Live browser Talk fix: binary WebSocket JSON frames are decoded before parsing, setup completion can transition the UI out of connecting, queued browser audio is stopped on interruption/shutdown, and a separate stale-session recovery path covers the reported post-failure queue wedge. The fix is present on main but not in the latest verified release tag v2026.4.27.

## What This Changes

The issue reports that Google Live browser Talk on 2026.4.26 stays at Connecting Talk, does not play assistant audio, and can wedge later text chat dispatches until the Gateway restarts.

## Best Possible Solution

Keep the current main implementation and let the next release carry it to users; reopen or file a new focused report only if a live Google session on a build containing c399fb750b501f4b4ff559f74a4ce4314f6f3994 still hangs or wedges chat dispatch.

## Reproduction Assessment

Yes. The original report gives a concrete 2026.4.26 reproduction, and current tests now simulate the key failing browser frame shapes and cleanup paths; I did not run the live Google API flow in this read-only pass.

## Solution Assessment

Yes. Decoding the actual WebSocket frame data before JSON parsing is the narrowest maintainable fix for the root cause the reporter confirmed, and the added output/session cleanup covers the destructive secondary symptoms without changing provider contracts.
