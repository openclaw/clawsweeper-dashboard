## Summary

Current main already covers the Sarvam use case through generic custom-provider support: you can define a provider in `models.providers`, point it at `https://api.sarvam.ai/v1`, add Sarvam model IDs under `models.providers.<id>.models`, and use `request.auth` header mode for `api-subscription-key` instead of Bearer auth.

## Risks / Open Questions

- There is no bundled first-party `sarvam` provider ID or Sarvam-specific onboarding/doc page on current main; this close relies on the generic custom-provider mechanism satisfying the requested capability.

## Close Comment

Codex review: closing this because current `main` already supports the requested Sarvam setup without adding new core code. OpenClaw can define custom providers in `models.providers`, add explicit model IDs under `models.providers.<id>.models`, set the Sarvam base URL, and use `models.providers.<id>.request.auth = { mode: "header", headerName: "api-subscription-key", value: ... }` for the non-Bearer auth path. That behavior is documented in `docs/gateway/configuration-reference.md:2590` and `docs/gateway/configuration-reference.md:2601`, and the runtime implements it in `src/agents/provider-request-config.ts:201` and `src/agents/provider-request-config.ts:452`.

If a polished named Sarvam integration is still wanted later, the documented route is a provider plugin rather than new core special-casing; see `docs/plugins/sdk-provider-plugins.md:1`.
