## Summary

Keep open: current main still returns `not requested` for cron delivery previews on every `delivery.mode: "none"` job, even when explicit delivery fields remain in `jobs.json`. The run-history half appears largely implemented on main through delivery trace logging, so the useful remaining fix is narrow.

## What This Changes

The issue reports that cron delivery previews mismatch mode-none jobs with stored delivery fields and that run history can omit delivery `intended`/`resolved` details after `jobs.json` reload.

## Best Possible Solution

Keep the current run-trace behavior and fix the preview path so bare `mode: "none"` still shows not requested, while mode-none jobs with explicit routing fields preview the same resolved route used for message-tool context.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection shows the preview half is reproducible because `resolveCronDeliveryPreview` returns `not requested` for all mode-none plans before checking explicit delivery fields.

## Solution Assessment

Yes, a narrow fix in the preview path is the best direction. It should align previews with the existing delivery-plan/run-trace semantics without re-enabling runner fallback delivery or changing the stored job schema.
