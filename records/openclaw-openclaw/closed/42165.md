## Summary

Current main now fixes the reported least-privilege regression: configured `tools.fs` and `tools.exec` sections are no longer merged into the profile-stage allowlist, and regression coverage covers the broader restrictive-profile case. The fix is on main only; no containing release tag was found.

## What This Changes

The issue asks for restrictive tool profiles, especially `minimal` and `messaging`, to stop gaining filesystem/runtime tools merely because `tools.fs` or `tools.exec` sections are configured.

## Best Possible Solution

Keep the current main behavior: restrictive profiles should remain narrow, configured tool sections should only configure tools already granted by profile or explicit allowlist, and users who want filesystem/runtime access under those profiles should opt in with `tools.alsoAllow`.

## Reproduction Assessment

Yes. The reported path is statically checkable by configuring `tools.profile` as `minimal` or `messaging` with `tools.fs`/`tools.exec`; current main shows those sections no longer populate `profileAlsoAllow`, so the old reproduction should no longer expose those tools.

## Solution Assessment

Yes. The current fix removes the root implicit-grant behavior, preserves explicit `alsoAllow`, adds warnings for affected configs, and covers both runtime and filesystem policy paths without broadening core semantics.
