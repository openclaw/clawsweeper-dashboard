## Summary

Current main and shipped releases already fix the stale replay path: after overflow compaction, the runner resumes from the compacted transcript and suppresses re-persisting the already-written inbound user turn.

## What This Changes

The issue reports a duplicate user-message delivery after memory flush plus compaction caused the previous user turn to be treated as a new queued message.

## Best Possible Solution

Keep the shipped compaction retry guard and ask for a fresh report only if a current release still replays an already-persisted user turn with new logs.

## Reproduction Assessment

Yes. The issue log plus the later linked reproduction show an already-persisted user turn being replayed after compaction, and current tests exercise that exact persisted-current-message retry boundary.

## Solution Assessment

Yes. The current fix is the narrow owner-boundary solution: retry from the compacted transcript only after the inbound turn was actually persisted, while preserving normal prompt persistence when precheck overflow happened before persistence.
