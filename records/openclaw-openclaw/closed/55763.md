## Summary

Current main already addresses the reported behavior, and the relevant fixes were present in the tagged release `v2026.4.22`. Discord stale-socket recovery now keys off provider-proven transport activity instead of ordinary app-event freshness, which prevents quiet Discord channels from being restarted just because no user traffic arrived. Separately, embedded Pi runs now retry silent `stopReason=error` turns with no output once and then explicitly surface the failure instead of letting the turn go unanswered.

## Risks / Open Questions

- This review is based on static inspection only; I did not run a live Discord or Anthropic reproduction in this read-only pass.
- The issue combined two symptoms. The current code clearly fixes the quiet-channel stale-socket path and the silent no-output error-turn behavior, but it does not guarantee replay of messages missed during a real non-resumable Discord disconnect outside that original stale-socket scenario.
