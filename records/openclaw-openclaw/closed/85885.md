## Summary

Close: current main and shipped releases already solve the central fleet-burst problem with native stable per-agent heartbeat phases, and heartbeat disable is documented as `0m`; the remaining cron main-session relaxation would be a separate design request rather than the needed preventive fix.

## What This Changes

The issue requests a way to prevent simultaneous heartbeat ticks across multi-agent fleets, either by allowing non-default agents to target their main session from cron, adding a heartbeat offset, or documenting an explicit heartbeat disable value.

## Best Possible Solution

Keep the native stable per-agent heartbeat phase and `0m` disable contract; discuss non-default cron main-session targeting separately only if exact cron semantics are still needed.

## Reproduction Assessment

Yes for the reported workaround gap: source still shows the cron guard rejecting `sessionTarget="main"` for non-default agents. The simultaneous native heartbeat premise no longer reproduces from current source because interval heartbeats are phase-scheduled per agent.

## Solution Assessment

Yes. Native heartbeat staggering is the narrower maintainable solution because it fixes the gateway saturation cause without loosening cron main-session semantics or adding operator-managed offsets.
