The PR adds a Dockerfile cleanup step that removes `/app/node_modules/openclaw` and `/app/node_modules/.bin/openclaw` from the final runtime image after copying runtime assets.

## Best Possible Solution

Prune the stale OpenClaw package from `runtime-assets` before the final `node_modules` copy, including the top-level link, `.bin` shim, and pnpm virtual-store entry, with Docker proof that all three are absent.

## Reproduction Assessment

Yes. The linked issue gives a Docker build/run repro, and current source shows the final image copies `node_modules` while extension manifests can still pin the stale published OpenClaw package.

## Solution Assessment

No. Removing the top-level symlink and shim is a useful mitigation, but the maintainable fix should prune the pnpm virtual-store package before the final image copy.
