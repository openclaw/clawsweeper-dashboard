## Summary

Current `main` already contains targeted fixes for the reported `acpx openclaw` symptom where the ACP client reaches `end_turn` but no terminal reply is delivered: the ACPX runtime now routes OpenClaw bridge commands through a bridge-safe MCP-free path, and the gateway chat layer preserves/flashes buffered assistant text into the final event instead of ending silently.

## Evidence

- **Changelog records the exact `acpx openclaw` bridge fix:** `CHANGELOG.md` says documented `acpx openclaw` setups no longer fail on per-session MCP injection because OpenClaw ACP bridge commands now use the MCP-free runtime path.
  - file: [CHANGELOG.md:174](https://github.com/openclaw/openclaw/blob/8fade9df27a9a25cd017db1154409b3ad9bd3741/CHANGELOG.md#L174)

  - file: [extensions/acpx/src/runtime.ts:163](https://github.com/openclaw/openclaw/blob/8fade9df27a9a25cd017db1154409b3ad9bd3741/extensions/acpx/src/runtime.ts#L163)
