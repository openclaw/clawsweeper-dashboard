## Summary

Keep open. Current main still performs a single shared media-understanding fetch for Discord voice-note URLs, and the linked closing PR #74553 is open and unmerged.

## What This Changes

Request to add bounded retries for transient remote media fetch failures so Discord/audio attachments are not dropped before media-understanding transcription.

## Best Possible Solution

Keep the issue open until PR #74553, or an equivalent shared bounded retry with the same fail-fast carve-outs, merges.

## Reproduction Assessment

Yes. Source-level reproduction is high-confidence: send a Discord audio attachment through preflight/media-understanding and make the remote media fetch throw once or return a 5xx; current main has one default fetch attempt and rethrows before transcription providers run.

## Solution Assessment

Yes. The narrow maintainable fix is shared bounded retry at the `fetchRemoteMedia` and `MediaAttachmentCache` boundary, not Discord-specific retry logic.
