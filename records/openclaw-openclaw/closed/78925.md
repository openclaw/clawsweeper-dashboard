## Summary

Keep open: current main makes the reported refresh mismatch source-reproducible, but this PR is not ready because the restore helper also maps legacy or bare session keys to `main` and the external contributor has not provided after-fix browser proof.

## What This Changes

The PR changes `loadAgents()` so the Control UI can restore `agentsSelectedId` from the current `sessionKey` after loading the agent list.

## Best Possible Solution

Update the PR to restore only from explicit parsed `agent:<id>:...` session keys, preserve default fallback for legacy keys, add regression coverage plus a changelog entry, and then provide redacted refresh-flow proof.

## Reproduction Assessment

Yes, from source: current main initializes the selected agent as null on page load, restores `sessionKey` from settings, and `loadAgents` falls back to `res.defaultId` without checking an active `agent:<id>:...` session key. I did not run a browser refresh flow, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

No, not as written. Restoring from the active session key is the right boundary, but the implementation should gate on an explicitly parsed agent session key rather than a helper that maps unparseable keys to `main`.
