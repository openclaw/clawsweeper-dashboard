## Summary

Keep open. Current main still carries the reported macOS CLI credential-display path: the protocol and other clients understand `WizardStep.sensitive`, but `openclaw-mac wizard` text prompts ignore it and use an echoing reader that can also display bracketed initial values.

## What This Changes

Request that the macOS `openclaw-mac wizard` CLI mask sensitive credential text steps and suppress sensitive initial-value display.

## Best Possible Solution

Land a focused macOS CLI fix in `WizardCommand.swift` that branches on `step.sensitive == true`, uses a termios-based no-echo reader, restores terminal state reliably, and preserves long credential input without exposing initial values.

## Reproduction Assessment

Yes, at source level: a `WizardStep` with `type: "text"`, `sensitive: true`, and an `initialValue` reaches `promptAnswer(for:)`, which ignores `sensitive` and calls the echoing `readLineWithPrompt`. I did not run an interactive macOS TTY reproduction in this read-only review.

## Solution Assessment

Yes. The narrow maintainable solution is to fix only the macOS CLI renderer, because the protocol field already exists and the Node CLI plus SwiftUI client already treat it as a secure-input signal.
