## Summary

Current `main` already contains a targeted WebChat fix for this exact symptom: `session.message` no longer reloads chat history during an active run, which previously raced the optimistic user-message update and delayed the user bubble until assistant output arrived.

## Evidence

- **Optimistic user message append still happens immediately on send:** `sendChatMessage` appends a `role: "user"` entry to `state.chatMessages` before awaiting the gateway request, so the UI is designed to show the user bubble immediately.
  - file: [ui/src/ui/controllers/chat.ts:325](https://github.com/openclaw/openclaw/blob/6f72b74cecf8d1fec5c147ae932161bf0f37a68b/ui/src/ui/controllers/chat.ts#L325)
  - command: `nl -ba ui/src/ui/controllers/chat.ts | sed -n '300,360p'`
  - sha: [6f72b74cecf8](https://github.com/openclaw/openclaw/commit/6f72b74cecf8d1fec5c147ae932161bf0f37a68b)
- **Main now blocks the mid-run history reload that caused the visual race:** `handleSessionMessageGatewayEvent` now skips `loadChatHistory` while `host.chatRunId` is set, with an inline comment explicitly stating that reloading history mid-run reset `chatStream` and delayed the user message card until the first LLM delta arrived.
  - file: [ui/src/ui/app-gateway.ts:496](https://github.com/openclaw/openclaw/blob/e5babbb5e75cdc7adaf4bb626db3ce838b2430c5/ui/src/ui/app-gateway.ts#L496)
