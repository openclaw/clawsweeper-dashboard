## Summary

Close as implemented: current main and shipped releases have bounded/yielding session listing plus non-durable session-store writes, and persisted session rows surviving restart are documented behavior rather than live session recreation.

## What This Changes

Report that 80 persisted sessions keep reappearing after gateway restart and cleanup, causing slow session/node listing and event-loop saturation on OpenClaw 2026.4.29.

## Best Possible Solution

Keep the shipped session-store and sessions.list fixes; future reports after v2026.5.12+ should include fresh logs plus cleanup --fix-missing output if rows still regenerate unexpectedly.

## Reproduction Assessment

Partially. I did not run a live macOS LaunchAgent restart repro, but the old starvation path is source-reproducible from the session-store fsync/listing code and the linked production reports; the remaining persisted-row count after restart is documented behavior.

## Solution Assessment

Yes. The current implementation fixes the hot writer and listing paths without changing the persisted-session contract or weakening durable defaults for unrelated config/state writes.
