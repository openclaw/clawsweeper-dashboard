## Summary

Current main already fixes the reported Telegram `/new` empty-provider-call path. Bare `/new` and `/reset` now stop in the reset command handler with a local acknowledgement, and tests assert that `/new` does not invoke the model.

## Best Possible Solution

Keep the current main behavior: bare `/new` and `/reset` should acknowledge locally after reset hooks, while `/new <message>` and `/reset <message>` continue to seed the next model turn. No separate Telegram-specific core change is needed.
