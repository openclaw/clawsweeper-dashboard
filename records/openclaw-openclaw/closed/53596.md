## Summary

Current main already handles `heartbeat.every: "30m"` as a duration string, schedules the runner from that parsed interval, and includes regression coverage for the heartbeat drift path that can make 30-minute heartbeats appear to fire hours late under repeated requests-in-flight skips. The shipped v2026.4.24 changelog also contains the heartbeat cadence fix.

## Best Possible Solution

Close this as implemented on main. Keep the current duration-string parser and heartbeat scheduler/retry behavior; if a user still sees 6-7 hour gaps on v2026.4.24 or newer, the best follow-up is a new issue with heartbeat logs, skip reasons, active-hours config, queue/busy-session evidence, and the exact `agents.defaults.heartbeat` or `agents.list[].heartbeat` block.
