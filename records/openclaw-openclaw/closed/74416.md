## Summary

Close as implemented on current main. The reported DeepSeek V4 Flash post-first-turn fallback pattern matches the DeepSeek V4 replay gap that current main now covers by backfilling `reasoning_content` on all replayed assistant messages, while the OpenAI-compatible stream parser already treats null or absent `delta.content` as non-fatal and consumes `reasoning_content` deltas as thinking output. The fix is present on main but not yet proven shipped in a release tag after `v2026.4.26`.

## What This Changes

Report that `deepseek/deepseek-v4-flash` on the OpenAI-completions API falls back on every generation after the first turn when streaming emits `reasoning_content` chunks with `delta.content: null`.

## Best Possible Solution

Keep the provider-owned DeepSeek V4 replay normalization and OpenAI-compatible stream parsing now on main, then let the next release deliver it to users on `v2026.4.26`; reopen or file a narrower follow-up only if the same fallback loop reproduces on a build containing the main-branch fix.
