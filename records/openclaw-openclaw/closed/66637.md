## Summary

Current main handles the reported macOS remote app-node browser proxy failure by removing the false `browser.proxy` advertisement from native macOS app nodes in remote mode, preserving the richer non-browser app-node capabilities, and directing remote browser control to the CLI node-host path. The fixing commit also adds actionable missing-control-service messaging and regression coverage, and the changelog explicitly marks #66637 fixed.

## Best Possible Solution

Close #66637 as implemented on main. Keep the shipped design: native macOS app nodes in remote mode should expose native app capabilities such as canvas/camera/location/system commands, while browser automation from that Mac should be served by the CLI node-host (`openclaw node start` or `openclaw node run`). Related PR #69494 can remain scoped to SecretRef/auth behavior and should not be treated as the canonical fix for this no-listener bug.

## Evidence

- **Remote macOS app nodes no longer advertise browser proxy:** `resolvedCaps` only adds the `browser` capability when browser control is enabled and `connectionMode == .local`; `resolvedCommands` only appends `browser.proxy` when that capability is present. This prevents the gateway from routing remote browser requests to the native app node that lacks the local 18791 service.
