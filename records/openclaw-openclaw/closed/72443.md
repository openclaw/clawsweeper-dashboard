## Summary

Keep open. Current main still has `doctor.memory.status` on the live embedding probe path, and the built-in memory manager still calls `embedBatchWithRetry(["ping"])` without a probe-result cache. This PR is a focused implementation candidate for #71568 and is not superseded or already shipped.

## Best Possible Solution

Keep this PR open as the active implementation candidate for #71568. Maintainers should review the cache semantics and test strategy, preferably asking for a production-code or extracted-helper regression test if they want stronger coverage, then land it if the behavior is accepted.

## Risks / Open Questions

- The proposed test duplicates the production cache algorithm in a standalone harness, so future drift in `MemoryIndexManager.probeEmbeddingAvailability()` may not be caught as strongly as a direct production-code test or extracted-helper test.
- Caching failed probe results for 30 seconds is likely acceptable for status calls, but it can briefly hide provider recovery.
- The PR intentionally does not reduce first-call latency; it only makes repeated status calls fast within the TTL window.
