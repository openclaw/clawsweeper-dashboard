## Summary

Keep open. The issue is not a proven Codex runtime bug, but current main still shows the same QA-contract split: QA-lab parity runs use the coding profile, `sessions_send` is a coding-profile tool, and direct `message` is messaging-profile only. The linked Phase 2 harness PR is still the right place for maintainers to decide how this fixture should be classified.

## What This Changes

The issue asks QA-lab maintainers to stop treating direct `message` as required coding-profile coverage and instead make it report-only or cover it through a separate messaging/channel fixture.

## Best Possible Solution

Keep direct `message` out of required coding-profile defaults, give `sessions_send` its own fixture, and keep `message(action="send")` coverage in a messaging/channel scenario when maintainers want that surface covered.

## Reproduction Assessment

No. The exact runtime-tool fixture is not checked in on current main, but source inspection confirms the underlying split: coding-profile QA exposes `sessions_send` and excludes direct `message`, while channel coverage exercises `message(action="send")`.

## Solution Assessment

Unclear. Treating direct `message` as report-only under the coding profile and covering messaging separately looks narrower than broadening coding-profile tools, but maintainers need to settle that QA harness contract in the linked PR.
