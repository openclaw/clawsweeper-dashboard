## Summary

Keep open: this issue has the protected `maintainer` label, and current `main` still has a source-visible race where count-based session maintenance can remove subagent session rows that announce/result capture still expects to read.

## What This Changes

Make `session.maintenance.maxEntries` enforcement lifecycle-aware so pending subagent sessions cannot be evicted before result freeze or announce delivery completes.

## Best Possible Solution

Protect active and pending-delivery subagent session keys during maintenance capping, then add a focused regression for capped stores with `cleanup: "delete"` and completion delivery.

## Reproduction Assessment

Yes, source inspection gives a high-confidence regression path: force `enforce` mode with a low `maxEntries`, let a completed subagent row be capped before announce capture, and assert the parent does not receive `(no output)`, `session_id: unknown`, or zero token stats. I did not run a live Slack reproduction in this read-only sweep.

## Solution Assessment

Yes, the narrow maintainable fix is to make session maintenance preserve pending subagent delivery rows until result freeze/announce cleanup is complete. Archive fallback and instrumentation are useful follow-ups, but preserving lifecycle-critical rows is the core fix.
