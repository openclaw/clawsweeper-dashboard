## Summary

Keep open: the feature direction is useful, but the PR is not safely mergeable because it exposes live schema references, leaves public API/docs/test gaps, and lacks real behavior proof from a subscribing plugin.

## What This Changes

The PR adds optional tool descriptor fields to the `llm_input` plugin hook event, emits built-in and client tool schemas from the embedded runner, updates the hook runner test fixture, and refreshes the plugin SDK API hash.

## Best Possible Solution

Expose the final request-visible tool schema through a documented, backward-compatible hook payload that uses isolated normalized descriptors for both built-in and client tools, with focused tests and real plugin proof.

## Reproduction Assessment

Yes for the blocking patch defect: the PR diff passes live parameter objects into `runLlmInput`, and current request adapters keep those same parameter references. No live provider run is needed to see that a hook handler could mutate the shared schema object before request submission.

## Solution Assessment

No. The API direction is useful, but this patch should be rebased onto current main and use isolated normalized descriptors, document the new fields, test populated `clientTools`, and include real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Clone tool parameters before emitting llm_input:** `src/agents/pi-embedded-runner/run/attempt.ts:2048-2053`
  - body: The hook payload passes `tool.parameters` and `tool.function.parameters` by reference. Because request tool definitions keep those same parameter objects, an `llm_input` tracing or redaction hook can mutate the schema the model receives in the same turn, so the event needs an isolated copy or another non-mutating representation.
  - confidence: 0.92
- **[P3] Normalize descriptions in the hook payload:** `src/agents/pi-embedded-runner/run/attempt.ts:2047-2052`
  - body: The request adapters normalize missing tool descriptions to `""`, but this event emits raw optional descriptions. Plugins that diff or hash `llm_input` tools against the actual request payload can see false drift whenever a built-in or client tool omits its description.
  - confidence: 0.84
- **[P3] Document the new llm_input fields:** `src/plugins/hook-types.ts:176-177`
  - body: This changes the public plugin hook contract, but `docs/plugins/hooks.md` still describes `llm_input` as prompt and history only. Plugin authors need the `tools` and `clientTools` payload fields documented before relying on the SDK contract.
  - confidence: 0.82
