## Summary

Keep open. On current `main` (`c997a9f9788e0a96e8eab05e7ae94959e76cdb4a`), `resolveCronSession` still carries forward prior `cliSessionIds` / `cliSessionBindings` / `claudeCliSessionId` when `forceNew` creates a fresh isolated cron session. `run-executor.ts` still masks that state for the current CLI resume path, but the stored `sessionEntry` remains internally inconsistent, so the bug report still matches current code rather than being already fixed on `main`.

## Evidence

- **Fresh-session cleanup still omits CLI identity fields:** `resolveCronSession` spreads the old entry into the new `sessionEntry` and, when `isNewSession` is true, clears routing metadata plus `sessionFile` only. It does not clear `cliSessionIds`, `cliSessionBindings`, or `claudeCliSessionId`, so stale CLI identity can survive a forced isolated rollover.
