## Summary

Keep open: the protected security label requires maintainer handling, and current source plus the latest stable tag still support the reported Codex/OpenClaw sandbox boundary mismatch.

## What This Changes

The issue requests that Codex-native shell/code execution honor OpenClaw per-agent Docker sandboxing, or fail/warn when Codex execution remains gateway-local.

## Best Possible Solution

Define one security contract for Codex with OpenClaw Docker sandboxing: either run Codex app-server/native execution inside the active per-agent sandbox by default, or fail closed with a clear operator-facing warning when that boundary cannot be honored.

## Reproduction Assessment

Yes, at source level: current main spawns the Codex stdio app-server from the Gateway process and explicitly keeps Codex filesystem sandboxing while mirroring OpenClaw egress. A live Docker canary is still needed before landing a security fix.

## Solution Assessment

Unclear: running Codex inside the OpenClaw sandbox is plausible, but maintainers need to reconcile that with the related nested-sandbox failure mode and decide whether the default should be sandbox placement, fail-closed behavior, or a warning path.
