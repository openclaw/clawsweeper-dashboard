## Summary

Keep open: current main still catches and logs failed exec approval prompt/follow-up delivery without surfacing an agent-visible failure, while the related retry PR is closed unmerged and the open warning-suppression PR only handles benign expired-id noise.

## What This Changes

The issue asks OpenClaw to return an explicit agent-visible error when Discord exec approval prompt or follow-up delivery fails instead of dropping the command.

## Best Possible Solution

Surface failed approval prompt delivery and failed async follow-up delivery to the waiting agent/session using existing no-route, unavailable, or follow-up result contracts, with focused regression tests across forwarding, native delivery, and session fallback.

## Reproduction Assessment

Yes, at source level: injecting a rejected sendExecApprovalFollowup or a failed outbound delivery path exercises current code that logs and returns without an agent-visible failure. I did not run a live Windows/Discord/WebSocket reproduction in this read-only review.

## Solution Assessment

Yes: the narrow maintainable direction is to propagate delivery failure through existing approval-unavailable or follow-up result contracts with bounded retry for transient transport errors. A log-only suppression or retry-only patch would not fully cover failed prompt delivery and final delivery exhaustion.
