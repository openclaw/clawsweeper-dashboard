## Summary

Keep open. Current main still builds the announce rewrite handoff from child completion data plus a rewrite instruction, without an explicit bounded requester-context or freshness signal; the related routing/transcript fixes and the open ContextEngine RFC are adjacent, but not a shipped fix for this grounding bug.

## What This Changes

The issue asks for subagent and cron announce completions to preserve enough requester-session context so the parent reply is grounded in the active conversation.

## Best Possible Solution

Add a typed, sanitized, size-bounded requester-context and turn-freshness handoff to announce rewrite turns, with regression coverage for active, queued, direct, cron, and nested requester paths.

## Reproduction Assessment

Yes, source-level. Construct a subagent or cron task_completion announce on current main and inspect that triggerMessage/internalEvents contain child result fields plus the rewrite instruction, without a bounded requester-context or turn-freshness field; live channel proof was not run.

## Solution Assessment

Yes. The narrow maintainable fix is explicit bounded requester-context or typed session-context metadata in the announce handoff; raw transcript splicing or relying on unrelated Telegram/cron routing fixes would leave this bug class exposed.
