## Summary

Current main already exposes a plugin hook seam that can mechanically gate cron-triggered agent turns before the model run starts, so this belongs as a ClawHub/plugin implementation rather than a new core `cron.preflight` field.

## Close Comment

Closing from a Codex review. This is a useful idea, but it fits the existing plugin hook surface better than a new core cron field. Current main already exposes `before_agent_reply`, which can claim a turn before the LLM call and return synthetic output or silence the turn (`docs/concepts/agent-loop.md:96`, `docs/plugins/sdk-overview.md:202`). Cron-triggered turns now hit that hook before the CLI or embedded agent attempt starts (`src/agents/cli-runner.ts:64`, `src/agents/pi-embedded-runner/run.ts:334`), with regression coverage proving a claimed cron turn does not spawn the agent (`src/agents/cli-runner.before-agent-reply-cron.test.ts:96`, `src/agents/pi-embedded-runner/run.before-agent-reply-cron.test.ts:23`). Given the repo direction that optional capability should usually ship as plugins (`VISION.md:54`) and the documented ClawHub publish/install path for external plugins (`docs/plugins/building-plugins.md:125`), this is better implemented as a ClawHub plugin that runs your shell gate and rewrites/skips cron turns via `before_agent_reply`, rather than as new OpenClaw core cron schema.
