## Label Justifications

- `impact:message-loss`: If Discord cannot complete startup behind the configured proxy, inbound Discord messages are not delivered to the agent.

## Summary

Keep open. Current main and the latest release still keep Discord gateway metadata lookup on the guarded direct fetch when a Discord proxy is configured, and the current test suite explicitly locks in that behavior. The linked open fix PR, https://github.com/openclaw/openclaw/pull/80252, owns the implementation path; because a merged security hardening PR intentionally made metadata direct-only, this needs Discord/security maintainer review rather than stale cleanup.

## What This Changes

The issue reports that Discord gateway metadata REST fetches ignore the configured Discord proxy while the WebSocket path uses it.

## Best Possible Solution

Review the linked fix PR or a narrow replacement that routes only validated loopback Discord proxies through fetchWithSsrFGuard trusted_explicit_proxy, preserves direct fallback for invalid or remote proxies, and includes Discord plus SSRF-guard regression coverage.

## Reproduction Assessment

Yes, source-reproducible: current main still keeps /gateway/bot metadata on guarded direct fetch when a proxy is configured, and the current test asserts that behavior. I did not run a live Mainland China or ClashX proxy reproduction in this read-only review.

## Solution Assessment

Unclear without maintainer/security approval: the narrow technical fix is to use guarded explicit-proxy mode for validated loopback Discord proxies, but the current behavior was introduced by an intentional security hardening PR.
