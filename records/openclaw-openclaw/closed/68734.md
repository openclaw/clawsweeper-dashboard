## Summary

Keep open: this is a useful public plugin-hook feature with credible real-world proof, but the current patch is not merge-ready because source review still finds prompt-boundary and tool-surface correctness blockers plus an overlapping open hook-contract proposal.

## What This Changes

The PR extends `before_prompt_build` so plugins can receive available tool and skill names and return `toolsAllow` or `skillsAllow`, then wires attempt/session prompt logic and unit tests around that filtering.

## Best Possible Solution

Land one maintainer-approved hook contract that narrows tools and skills through the normal prompt/session pipeline while preserving provider transforms, hook context, empty-allowlist semantics, client-hosted tools, and compatibility tests.

## Reproduction Assessment

Yes for the review findings: source inspection of the PR head shows the empty `toolsAllow` guard, the direct `skillsAllow` prompt rebuild, and the `availableTools` source that excludes client-hosted tools. I did not run a live gateway in this read-only review.

## Solution Assessment

No, not as currently implemented. The narrow maintainable path is to pick one public hook contract and then route filtering through the existing prompt/session pipeline without losing transforms, hook context, empty-allowlist behavior, or hosted tools.

Full review comments:

- **[P1] Honor empty tool allowlists from hooks:** `src/agents/pi-embedded-runner/run/attempt.ts:1884`
  - body: This guard ignores `toolsAllow: []`, so a plugin that returns an explicit empty allowlist gets the full tool surface instead of no tools. Existing runtime `toolsAllow` treats an empty array as no tools, and the new hook type says only listed tools are sent, so this fails open at the tool-policy boundary.
  - confidence: 0.9
- **[P1] Preserve prompt mutations when filtering skills:** `src/agents/pi-embedded-runner/run/attempt.ts:1951-1952`
  - body: When `skillsAllow` is set, this rebuilds the system prompt from the original append params and applies it directly after hook `systemPrompt` / prepend / append context may already have changed `systemPromptText`. That drops those hook mutations and skips the normal provider transform path for the narrowed prompt.
  - confidence: 0.92
- **[P2] Expose client-hosted tools to hook classifiers:** `src/agents/pi-embedded-runner/run/attempt.ts:1839`
  - body: `availableTools` is populated from `effectiveTools`, but client-hosted tools are registered separately through `clientToolDefs` and `allCustomTools`. A plugin that returns `toolsAllow` from the hook-visible list cannot include those hosted tools, so `setActiveToolsByName(...)` can remove them for the turn.
  - confidence: 0.86
