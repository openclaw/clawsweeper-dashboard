## Summary

Current `main` already contains the Feishu reply-dispatcher changes and regression tests that address the duplicate block+final streaming-card path described in #40028. The fix is on `main` after `v2026.4.22`, so it appears implemented but not yet shipped in the latest tagged release.

## Evidence

- **Block delivery now stays on the streaming path when the card session is active:** `deliver()` starts/awaits streaming for `kind === "block"` and returns through the streaming branch when `streaming?.isActive()` is true, instead of falling through to a second card send.
  - file: [extensions/feishu/src/reply-dispatcher.ts:391](https://github.com/openclaw/openclaw/blob/86fa8eeb68c64eb0ae25e1eb8c6196061862a010/extensions/feishu/src/reply-dispatcher.ts#L391)

  - sha: [86fa8eeb68c6](https://github.com/openclaw/openclaw/commit/86fa8eeb68c64eb0ae25e1eb8c6196061862a010)
- **Final-text dedupe is implemented in the dispatcher:** The dispatcher tracks `deliveredFinalTexts` and skips exact duplicate final text before delivery; when streaming closes successfully it records the final text to suppress a second send.
  - file: [extensions/feishu/src/reply-dispatcher.ts:380](https://github.com/openclaw/openclaw/blob/86fa8eeb68c64eb0ae25e1eb8c6196061862a010/extensions/feishu/src/reply-dispatcher.ts#L380)
