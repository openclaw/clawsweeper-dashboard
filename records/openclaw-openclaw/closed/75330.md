## Summary

Keep open. Current main still has in-process agent preparation paths that can plausibly starve the Gateway event loop, and the newer related PR only appears to cover one transcript-I/O slice of the larger report. The issue has multiple corroborating user reports across platforms, so it is not stale or insufficiently supported.

## What This Changes

Investigate and fix Gateway event-loop starvation during agent preparation that causes Gateway RPCs and channel dispatchers to hang under OpenClaw 2026.4.29.

## Best Possible Solution

Treat this as an active Gateway performance incident shared with #75329 and informed by #75214/#75595: profile packaged 2026.4.29 and current main during an agent turn, then make prompt, skill, tool, transcript, and session-resource preparation non-starving through async I/O, bounded yielding, caching, or isolation, with regression proof that Gateway RPC latency stays bounded during prep.

## Reproduction Assessment

Unclear. The issue has strong multi-platform measurements and current-main code still contains plausible blocking paths, but a high-confidence reproduction needs a packaged/current-main run with event-loop diagnostics enabled while polling models.list, sessions.list, and node.list during agent preparation.

## Solution Assessment

Not applicable because this issue has no patch. The best fix is not to hide stuck-session or liveness warnings; it should remove or isolate the blocking prep work and add a regression check for Gateway responsiveness under agent prep load.
