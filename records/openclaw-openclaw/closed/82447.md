## Summary

Keep open: current main still sends the `items: {}` array-schema backfill that this Xiaomi Token Plan report identifies, and the Xiaomi provider does not yet apply a tool-schema compatibility rewrite for those payloads. The discussion also has fresh maintainer/user evidence that the standard endpoint and reasoning-content fixes do not fully prove the Token Plan tool-heavy path is fixed.

## What This Changes

The issue requests a Xiaomi MiMo/OpenAI-compatible provider compatibility fix for 2026.5.12-era full tool schemas that trigger `400 Param Incorrect`.

## Best Possible Solution

Add endpoint-aware Xiaomi tool-schema compatibility that preserves the global OpenAI fix but strips only synthetic empty array `items: {}` for Xiaomi-native routes, including Token Plan hosts.

## Reproduction Assessment

No, not as a live Token Plan run in this review. Source inspection shows the current outbound `items: {}` backfill and the issue has multiple matching `400 Param Incorrect` reports, but credentialed Token Plan validation is still missing.

## Solution Assessment

Yes, keeping this open for a narrow Xiaomi/provider compatibility fix is the best path. Reverting the global array-items normalizer would risk reopening the OpenAI-compatible schema bug that PR 81217 fixed.
