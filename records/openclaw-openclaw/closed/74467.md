## Summary

Current main and the latest release already prevent the reported WebChat reconnect path from creating empty chat turns: the browser client refuses empty sends, the Gateway independently rejects empty `chat.send` requests, and the auth-gated bootstrap config fetch is a tolerated no-op on 401/403 rather than a chat submission path.

## What This Changes

WebChat should silently reconnect after WebSocket drops without sending empty messages, even when `control-ui-config.json` returns 401 during reconnect.

## Best Possible Solution

Keep the current WebChat contract: reconnect only re-establishes the Gateway WebSocket, auth-gated bootstrap fetch failures stay non-chat side effects, and both client and Gateway reject empty chat turns. Users seeing this on 2026.4.25 should move to v2026.4.26 or newer.
