## Label Justifications

- `P1`: The report describes cross-chat Active Memory recall waits causing aborts and gateway unresponsiveness in concurrent real chat usage.
- `impact:session-state`: The affected path controls whether Active Memory recall context is available for independent sessions during prompt construction.
- `maturity:stable`: The affected gateway command-queue/runtime surface maps to `gateway-runtime` scored M4 Stable in the maturity shortlist.

## Summary

Close as duplicate: the requested Active Memory recall-lane concurrency knob is a narrower slice of the open canonical reliability work at https://github.com/openclaw/openclaw/issues/72015, which already tracks the single process-wide `active-memory` lane, default lane concurrency of one, and the maintainer decision around blocking recall, lane isolation, and defaults.

## What This Changes

Request to add an Active Memory config option such as `plugins.entries.active-memory.config.laneConcurrency` so operators can raise recall-lane concurrency above serialized execution.

## Best Possible Solution

Keep the broader Active Memory reliability decision centralized in the canonical issue and preserve this issue's operator workaround/concurrency evidence there.

## Reproduction Assessment

Yes. Source inspection shows the Active Memory recall uses one fixed `active-memory` lane, new lanes default to `maxConcurrent: 1`, and no current config path raises that lane.

## Solution Assessment

No. A raw lane-concurrency knob is one plausible mitigation, but the canonical issue should decide whether the best fix is per-agent/session lanes, nonblocking or single-shot recall, explicit config, or a combination.
