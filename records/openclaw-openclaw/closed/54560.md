## Summary

Current main already contains released Telegram polling fixes that match the reported Windows stall symptoms from 2026.3.23-2, including Windows/IPv6 long-poll hardening, transport rebuild/disposal on stalls, and liveness-based restart behavior.

## Evidence

- **Released fix note for the main polling hardening:** The 2026.4.18 changelog explicitly says Telegram polling now uses bounded transport lifecycle, publishes successful getUpdates liveness, avoids false stall restarts after recoverable getUpdates errors, and forces Telegram dispatchers to HTTP/1.1.
  - file: [CHANGELOG.md:499](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/CHANGELOG.md#L499)
  - command: `nl -ba CHANGELOG.md | sed -n '475,505p'`
- **Current code forces HTTP/1.1 for Telegram polling on Windows/IPv6-prone paths:** The dispatcher creation path documents and implements an HTTP/1.1-only pool because Undici HTTP/2 ALPN could stall Telegram long-polling on Windows/IPv6 networks.
  - file: [extensions/telegram/src/fetch.ts:298](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/extensions/telegram/src/fetch.ts#L298)
  - command: `nl -ba extensions/telegram/src/fetch.ts | sed -n '290,380p'`
- **Current polling loop rebuilds transport after stalls and recoverable polling errors:** The polling session marks the transport dirty on stall/error, logs the same stall signatures mentioned in the report, and restarts the polling cycle.
  - file: [extensions/telegram/src/polling-session.ts:243](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/extensions/telegram/src/polling-session.ts#L243)

  - file: [extensions/telegram/src/polling-transport-state.ts:22](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/extensions/telegram/src/polling-transport-state.ts#L22)
  - command: `nl -ba extensions/telegram/src/polling-transport-state.ts | sed -n '1,140p'`
