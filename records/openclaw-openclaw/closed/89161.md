## Summary

Keep open: current main still carries `sessions_send` handoffs as `role: "user"` with inter-session provenance, while the WebChat grouping/rendering path ignores that provenance and assigns the normal user-side class. The existing current-main special case only covers `subagent_announce`, so the related closed report does not fully cover this `sessions_send` case.

## What This Changes

The issue requests that WebChat render `sessions_send` inter-session messages as clearly agent-forwarded instead of user-authored.

## Best Possible Solution

Add a narrow WebChat display-layer projection for user-role messages with `provenance.kind === "inter_session"` and `sourceTool === "sessions_send"`, rendering them as clearly forwarded/agent-mediated with source attribution while preserving the underlying transcript role for provider compatibility.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: create a user-role message with `provenance.kind="inter_session"` and `sourceTool="sessions_send"`, then pass it through WebChat grouping; it remains a user group because the UI ignores that provenance. I did not run a live browser session in this read-only review.

## Solution Assessment

Yes, a WebChat display-layer fix is the best shape: docs explicitly keep `role: "user"` for provider compatibility, so the UI/projection should distinguish forwarded messages without changing the durable model transcript.
