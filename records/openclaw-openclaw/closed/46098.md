## Summary

Current `main` already supports Chromium-based `existing-session` attachment by accepting `browser.profiles.<name>.userDataDir` and forwarding it to Chrome DevTools MCP as `--userDataDir`. The latest release `v2026.4.22` also contains that implementation and user-facing docs for Chromium/Brave/Edge.

## Evidence

- **Existing-session profiles accept explicit Chromium user data dirs:** `profiles-service` validates `userDataDir` for `driver="existing-session"` and stores it on the profile instead of requiring CDP.
  - file: [extensions/browser/src/browser/profiles-service.ts:115](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/extensions/browser/src/browser/profiles-service.ts#L115)

## Risks / Open Questions

- The built-in `user` profile still defaults to Google Chrome auto-connect; Chromium users need a custom `existing-session` profile or explicit `userDataDir`.
- The shipped fix uses Chrome MCP `--userDataDir`, not the issue's proposed explicit `cdpUrl`/`cdpPort` handoff.
