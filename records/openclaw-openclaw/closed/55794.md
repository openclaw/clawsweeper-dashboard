## Summary

Current `main` contains a WhatsApp group session-routing/session-init fix that matches the reported regression pattern: group inbound now routes named-account group chats onto a scoped session key, legacy group activation is migrated onto that scoped key, and activation-only ghost entries now force creation of a real new session with `sessionId`/`updatedAt` instead of leaving a mapping with no transcript file.

## Evidence

- **Inbound WhatsApp group messages now route through an account-scoped group session key:** `resolveWhatsAppGroupSessionRoute(...)` rewrites named-account WhatsApp group routes onto `:thread:whatsapp-account-<account>` while leaving the default account unchanged. The group inbound handler applies that route before gating and reply processing.

- **Legacy group activation is read from the old key and backfilled onto the scoped key:** `resolveGroupActivationFor(...)` looks up both the scoped session key and the legacy group key, prefers real scoped state, and backfills `groupActivation` onto the scoped entry when only the legacy entry has it. This matches the issue/comment pattern where group routing state existed but the scoped session state was incomplete.
  - file: [extensions/whatsapp/src/auto-reply/monitor/group-activation.ts:31](https://github.com/openclaw/openclaw/blob/fc5920fb51346a682b269fdbeeec0971ae4ae723/extensions/whatsapp/src/auto-reply/monitor/group-activation.ts#L31)

- **Session init no longer reuses ghost entries without a real session id:** `initSessionState(...)` only reuses an existing session when the entry has both a valid `sessionId` and numeric `updatedAt`; otherwise it creates a fresh session. The current regression test explicitly covers a scoped WhatsApp group entry that contains only `groupActivation` and asserts that a new `sessionId` and `updatedAt` are created.
  - file: [src/auto-reply/reply/session.ts:432](https://github.com/openclaw/openclaw/blob/fc5920fb51346a682b269fdbeeec0971ae4ae723/src/auto-reply/reply/session.ts#L432)

That lines up with the follow-up report about a mapped session id whose `.jsonl` file never materialized.
