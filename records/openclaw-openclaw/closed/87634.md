## Summary

Current main and shipped releases no longer gate dreaming-only promotion on query-hash count alone; repeated dreaming days count through recallDays/contextDiversity, and the apply path repeats the same check.

## What This Changes

The issue asks for dreaming promotion to stop being blocked when entries have one stable internal dreaming query hash despite repeated daily dreaming signals.

## Best Possible Solution

Keep the shipped recallDays/contextDiversity promotion gate; any remaining zero-promotion report should be reopened as a new current-release issue with recallDays, score, and promote-explain JSON evidence.

## Reproduction Assessment

No high-confidence current-main failure reproduction is present. The production breakdown counts queryHashes only, while current source gates on max(queryHashes, recallDays) and shipped tests cover repeated dreaming-only signals.

## Solution Assessment

Yes, the current implementation is the narrow maintainable fix for this specific gate: it counts cross-day dreaming reinforcement without lowering defaults or adding a new config knob. If promotion is still zero, the next report needs recallDays and score evidence to identify a different blocker.
