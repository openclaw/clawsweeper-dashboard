## Summary

Current `main` already implements Telegram media-download fallback for the IPv6/dual-stack failure described in #44747. The media download path now reuses Telegram transport fallback policies, retries guarded fetches across those attempts, and prefers IPv4 in pinned SSRF DNS results.

## Evidence

- **Telegram media downloads now use transport fallback attempts:** `downloadAndSaveTelegramFile` passes `dispatcherAttempts`, `shouldRetryFetchError`, and proxy-DNS trust into `fetchRemoteMedia`, so `/file/bot...` downloads use the same Telegram transport fallback strategy instead of a single guarded fetch.
