## Summary

PR #58022 should close in favor of the canonical open bug #52511 and a targeted auth/permission recovery fix. The PR's broad reload-on-any-1000 approach remains unsafe because current server paths can emit code 1000 for non-auth failures, while current main already has reconnect and bounded auth-token recovery paths that should be extended if #52511 still reproduces.

## Best Possible Solution

Close PR #58022 as superseded by the canonical #52511 tracking path. Future work should either extend the current `GatewayBrowserClient` reconnect/device-token recovery flow or add a permission-specific recovery signal, rather than reloading the Control UI on every websocket code 1000 close.
