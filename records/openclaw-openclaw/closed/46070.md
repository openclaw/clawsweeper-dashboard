## Summary

This is a Paperclip adapter compatibility issue rather than an OpenClaw core bug to keep open. Current main intentionally treats normal WebSocket operator clients without device identity as unbound: self-declared scopes are cleared for shared token/password auth except for explicit trust paths such as Control UI break-glass or direct-loopback internal `gateway-client` backend RPCs. The documented migration path is for WebSocket clients to send challenge-signed device identity and use device/device-token auth, so Paperclip's `disableDeviceAuth: true` behavior needs to change outside this repository.

## Best Possible Solution

Close this repository item and have the Paperclip `openclaw_gateway` adapter implement the current Gateway WebSocket device-identity/device-token handshake, or move to a supported HTTP/trusted-proxy integration path if that better fits Paperclip's deployment model.
