## Summary

The request is a valid feature idea, but current main already treats per-command context routing as plugin scope: slash commands are plugin-extensible and context assembly is explicitly delegated to pluggable context engines. Under the repo vision, optional capability like this should ship as a plugin/ClawHub package rather than expanding OpenClaw core.

## Evidence

- **Repo vision keeps optional capability out of core:** The project vision says "Core stays lean; optional capability should usually ship as plugins" and that the bar for adding optional plugins to core is intentionally high.
  - file: [VISION.md:52](https://github.com/openclaw/openclaw/blob/d3f6783b16ecc10c67b101bf019ca4294453206a/VISION.md#L52)
  - sha: [d3f6783b16ec](https://github.com/openclaw/openclaw/commit/d3f6783b16ecc10c67b101bf019ca4294453206a)
- **Context assembly is already a plugin seam:** The context-engine docs define context engines as the component that controls which messages are included, how older history is summarized, and how context is managed per run, and they are installed/selected as plugins via `plugins.slots.contextEngine`.
  - file: [docs/concepts/context-engine.md:10](https://github.com/openclaw/openclaw/blob/d3f6783b16ecc10c67b101bf019ca4294453206a/docs/concepts/context-engine.md#L10)
  - sha: [d3f6783b16ec](https://github.com/openclaw/openclaw/commit/d3f6783b16ecc10c67b101bf019ca4294453206a)
- **SDK explicitly supports both pieces this feature needs:** The plugin SDK documents `api.registerContextEngine(id, factory)` for custom context assembly and `api.registerCommand(...)` for custom commands.
  - file: [docs/plugins/sdk-overview.md:168](https://github.com/openclaw/openclaw/blob/d3f6783b16ecc10c67b101bf019ca4294453206a/docs/plugins/sdk-overview.md#L168)
  - sha: [d3f6783b16ec](https://github.com/openclaw/openclaw/commit/d3f6783b16ecc10c67b101bf019ca4294453206a)
- **Runtime exposes slash-command and context-engine registration:** Plugin command handlers receive session-aware command context, and the registry wires both `registerCommand` and `registerContextEngine` into the public plugin API.
  - file: [src/plugins/types.ts:1692](https://github.com/openclaw/openclaw/blob/d3f6783b16ecc10c67b101bf019ca4294453206a/src/plugins/types.ts#L1692)
  - sha: [d3f6783b16ec](https://github.com/openclaw/openclaw/commit/d3f6783b16ecc10c67b101bf019ca4294453206a)
- **Core does not currently ship `/nc`, `/cc`, or `/fc`:** A targeted search for built-in aliases/names found no `/nc`, `/cc`, or `/fc` command definitions under `src`, `docs`, or `extensions`, so this is not already implemented on main.
  - command: `rg -n '"/nc"|"/cc"|"/fc"|textAlias: "/nc"|textAlias: "/cc"|textAlias: "/fc"|nativeName: "nc"|nativeName: "cc"|nativeName: "fc"' src docs extensions`
  - sha: [d3f6783b16ec](https://github.com/openclaw/openclaw/commit/d3f6783b16ecc10c67b101bf019ca4294453206a)

## Risks / Open Questions

- A plugin implementation may still discover a missing low-level seam for per-message tier state or context-engine coordination; if that happens, the follow-up should be a narrow core seam request rather than landing the whole feature in core.
- The current issue body mixes feature design, pricing anecdotes, and cross-links, so future contributors may need a tighter plugin-oriented spec to implement it cleanly.

## Close Comment

Closing this Codex review as `clawhub`, not as an OpenClaw core item. Current `main` already exposes the two extension points this idea needs: plugin slash commands via `api.registerCommand(...)` (`docs/plugins/building-plugins.md:165`, `src/plugins/types.ts:1692`) and pluggable context assembly via `api.registerContextEngine(...)` / `plugins.slots.contextEngine` (`docs/concepts/context-engine.md:10`, `docs/plugins/sdk-overview.md:170`, `src/plugins/registry.ts:1384`). `VISION.md:54` also sets the direction that optional capability should usually ship as plugins and that the bar for adding optional plugins to core is intentionally high. A ClawHub/community plugin can own `/nc` `/cc` `/fc` plus the tiered context policy without expanding OpenClaw core. If plugin prototyping turns up a missing generic seam, please open a narrower follow-up for that seam.
