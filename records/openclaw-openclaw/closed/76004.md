## Summary

Keep open. The latest force-push addressed part of the earlier bootstrap review by adding a standalone loader helper, but the PR still turns several runtime surfaces into loaded-registry-only probes and still has an env/plugin-root compatibility gap for active registry reuse.

## What This Changes

The PR memoizes prepared provider extra params and rewires provider, tool, channel, web, capability, memory, and migration runtime helpers to reuse loaded startup registries through new active/standalone registry helpers.

## Best Possible Solution

Keep the extra-param cache, but preserve loader-compatible cold fallbacks and require full env/config/workspace/runtimeOptions compatibility before reusing active or pinned registries.

## Reproduction Assessment

Yes. A source-level reproduction is to call `resolvePluginWebProviders`, `resolvePluginProviders`, or `getActiveMemorySearchManager` on the PR head with a configured plugin and no compatible active registry; the PR returns an empty/unavailable result where current main loads the scoped runtime registry.

## Solution Assessment

No. Memoizing prepared extra params per stable config is a plausible narrow optimization, but replacing resolver fallbacks with loaded-registry-only reads and env-blind reuse is not the safest implementation boundary.
