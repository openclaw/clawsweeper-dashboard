## Summary

Keep this PR open, but it is not mergeable as submitted: the external contributor proof is absent, the docs check is failing, and the diff has concrete PATH-wrapper and regression-test blockers.

## What This Changes

The PR threads `tools.exec.pathPrepend` into host/gateway exec runtime, wraps POSIX commands with an in-shell PATH export, adds a PATH removal helper, tests, and exec docs.

## Best Possible Solution

First prove the current-main failure in a real macOS or POSIX shell setup; if it still exists, land a wrapper that preserves PATH security invariants and has a startup-file regression test that actually runs before the target command.

## Reproduction Assessment

Unclear. The PR describes a macOS `~/.zshenv` repro, but it provides no inspectable after-fix output and current main already suppresses bash/zsh/fish startup files in `getShellConfig`.

## Solution Assessment

No, not as submitted. The in-shell prepend may be a narrow hardening direction if a real startup override remains, but this patch needs real proof, a no-empty-PATH export, meaningful startup-file coverage, and docs formatting before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Avoid creating an empty PATH segment:** `src/agents/bash-tools.exec-runtime.ts:602`
  - body: When `env.PATH` only contains configured prepends or is blank, `removePathPrepend` can leave PATH empty, but the wrapper still appends `:$PATH`. POSIX treats an empty PATH component as the current directory, so this can widen command lookup for exec runs that previously had only the configured directories.
  - confidence: 0.82
- **[P2] Fix the wrapper assertion:** `src/agents/bash-tools.exec-runtime.test.ts:721`
  - body: The wrapper is a JS template literal where `\${...}` produces `${...}` at runtime, so `spawnCall.argv` will not contain the backslash this assertion requires. As written, the new test fails instead of covering the wrapper.
  - confidence: 0.94
- **[P2] Simulate startup files before asserting PATH precedence:** `src/agents/bash-tools.test.ts:862`
  - body: This command modifies PATH inside the user command after the wrapper has run, so the expected `/evil/bin` first order is user-command behavior, not a shell startup override. It contradicts the stated guarantee and will not catch the regression this PR is meant to fix.
  - confidence: 0.9
