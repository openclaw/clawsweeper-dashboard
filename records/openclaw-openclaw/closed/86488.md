## Summary

Keep open. Current main has a partial durable delivery state for successful keep-mode subagent completions, but source still shows retry-limit handling can move results into a suspended state that is skipped by normal resume and later discarded, so the requested persistent retry/fallback contract remains necessary.

## What This Changes

The issue requests a persistent fallback for `sessions_yield` subagent completion announcements so completed child results are not lost after announce retries fail.

## Best Possible Solution

Define one core-owned durable completion delivery contract that persists successful subagent results, retries them through a lifecycle-owned delivery path, exposes blocked/discarded status deliberately, and covers Feishu plus channel-agnostic behavior in regression tests.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path for the core failure class: retry exhaustion enters suspended delivery, normal resume skips it, and the sweeper can later discard it. I did not run a live Feishu repro in this read-only review.

## Solution Assessment

Unclear. A durable retry/outbox direction fits the bug, but maintainers should choose the retention, retry cadence, and user-visible blocked/discarded semantics; exposing retry knobs alone would not preserve completed results.
