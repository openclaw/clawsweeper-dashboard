## Summary

The request is clear and useful, but on current `main` it fits the existing external context-engine plugin seam rather than OpenClaw core. Core already exposes a dedicated context-engine slot, plugin registration API, and docs for installing/selecting non-legacy engines, while `VISION.md` says optional capability should usually live as plugins maintained outside core.

## Evidence

- **Vision scope favors external plugins over new core optional capability:** `VISION.md` says OpenClaw has an extensive plugin API, core should stay lean, optional capability should usually ship as plugins, and plugin authors should host and maintain them outside core.
  - file: [VISION.md:54](https://github.com/openclaw/openclaw/blob/0e7250f37bf347c72790abf2a6692d5ec84e5ea1/VISION.md#L54)

- **Current docs already position context engines as installable plugins:** The context-engine docs tell users to install a context engine plugin from npm or a local path and select it with `plugins.slots.contextEngine`; they explicitly say to install a plugin engine when different assembly/compaction behavior is desired.
  - file: [docs/concepts/context-engine.md:14](https://github.com/openclaw/openclaw/blob/0e7250f37bf347c72790abf2a6692d5ec84e5ea1/docs/concepts/context-engine.md#L14)
  - sha: [0e7250f37bf3](https://github.com/openclaw/openclaw/commit/0e7250f37bf347c72790abf2a6692d5ec84e5ea1)
- **Core exposes a first-class context-engine plugin API:** The plugin API includes `registerContextEngine(id, factory)` as an exclusive slot registration method, which is the seam needed for a structured-context engine without adding new core capability.
  - file: [src/plugins/types.ts:2111](https://github.com/openclaw/openclaw/blob/0e7250f37bf347c72790abf2a6692d5ec84e5ea1/src/plugins/types.ts#L2111)
  - sha: [0e7250f37bf3](https://github.com/openclaw/openclaw/commit/0e7250f37bf347c72790abf2a6692d5ec84e5ea1)
- **Config and manifest docs already support selecting external context-engine plugins:** Docs and schema references show `plugins.slots.contextEngine` is the supported selector for `kind: "context-engine"` plugins, with `legacy` as the built-in default.
  - file: [docs/plugins/manifest.md:707](https://github.com/openclaw/openclaw/blob/0e7250f37bf347c72790abf2a6692d5ec84e5ea1/docs/plugins/manifest.md#L707)
  - sha: [0e7250f37bf3](https://github.com/openclaw/openclaw/commit/0e7250f37bf347c72790abf2a6692d5ec84e5ea1)

A search of shipped extensions/manifests found no bundled `structured-context` engine on current `main`, which reinforces that a specialized engine belongs as an external plugin rather than new core surface.

## Risks / Open Questions

- There may still be real product demand for a canonical structured-context engine; closing as out-of-scope does not address that demand.
- No referenced external/ClawHub package is identified here, so users would still need an actual plugin implementation to realize the idea.
