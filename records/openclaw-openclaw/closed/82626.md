## Summary

Keep open: the reported WebChat auto-TTS failure is plausible from current source, but this PR should not merge as-is because it corrupts source text/encoding, bypasses media normalization too broadly, and lacks after-fix real behavior proof.

## What This Changes

The PR updates speech-core auto-TTS to synthesize for block replies and mark TTS media trusted, skips reply-media normalization for trusted local media, and changes the plugin SDK reply payload file encoding.

## Best Possible Solution

Land one clean UTF-8 patch that preserves media safety boundaries, marks only core-generated TTS local audio as trusted for WebChat embedding, keeps non-WebChat delivery normalization intact, and adds focused regression coverage plus real WebChat proof.

## Reproduction Assessment

Unclear for live reproduction: current source shows a credible failure path because block replies skip final-mode auto-TTS and WebChat rejects untrusted local audio, but I did not run the WebChat/MiniMax flow.

## Solution Assessment

No: the intended direction is plausible, but this branch is not the best fix because it adds UTF-16/binary source churn, mojibake, and a global trustedLocalMedia normalizer bypass instead of a scoped WebChat-safe media path.
