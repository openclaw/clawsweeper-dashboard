## Summary

Current main and shipped releases already provide the operator-safe path requested here: `openclaw update --no-restart` is documented and leaves supervised services alone, and the current restart health check is PID-aware for runtime-owned listeners.

## What This Changes

Report that `openclaw update --yes` can complete a macOS launchd-managed package update but then fail non-zero during its internal restart, requesting a no-restart/service-management or structured restart outcome path.

## Best Possible Solution

Keep the shipped `--no-restart` operator path as the supported solution for externally supervised launchd update pipelines, with future work only if a new report shows the current flag or PID-aware health check failing.

## Reproduction Assessment

No; I did not run a macOS launchd package upgrade in this read-only review. The report includes concrete shipped-run logs, and source inspection shows current main already has the requested no-restart path plus PID-aware listener ownership checks.

## Solution Assessment

Yes; for externally supervised launchd pipelines, the shipped `openclaw update --yes --no-restart` path is narrower and safer than adding another service-management mode or exit-code contract for this report.
