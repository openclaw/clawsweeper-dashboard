## Summary

Close as implemented. The callback shown in the report is only the logging/status hook; the controller watchdog calls that hook and then immediately forces a 499 close, and current tests cover the post-408 app-silent recovery case.

## What This Changes

The issue asks for WhatsApp watchdog app-silent timeouts to tear down the Baileys socket and re-enter the reconnect loop instead of only logging and marking the monitor stale.

## Best Possible Solution

Keep the current controller-owned reconnect behavior and preserve the regression coverage that proves post-408 app-silent listeners are recovered.

## Reproduction Assessment

No. The reported source-level failure does not reproduce against current main or the `v2026.4.29` tag: the watchdog timer calls `forceClose()` immediately after the logging hook, and the post-408 app-silent scenario is covered by an e2e test.

## Solution Assessment

Yes. Keeping the reconnect trigger in `ConnectionController.startTimers()` is the narrowest design because the timer owns the timeout decision and can invoke the shared close/reconnect path after monitor-specific logging runs.
