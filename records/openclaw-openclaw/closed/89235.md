## Summary

Keep open: current main still has the reported iMessage cron target-normalization path, and the existing documented `chat_identifier:` support does not protect an unprefixed 32-character hex group identifier from being normalized as a phone number.

## What This Changes

The issue reports that iMessage cron `delivery.to` values containing unprefixed 32-character hex group chat identifiers are stripped into bogus phone-number targets while delivery reports success.

## Best Possible Solution

Fix the iMessage target parser/normalizer so unprefixed hex group identifiers are either safely canonicalized to `chat_identifier:<id>` with dependency proof or rejected before delivery, while preserving the documented explicit `chat_identifier:` path.

## Reproduction Assessment

Yes, at source level: an unprefixed hex target enters iMessage target normalization, falls through to `normalizeE164`, and can be converted into a digit-only `+...` target. I did not establish a live macOS cron repro in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix belongs at the iMessage target normalization/parsing boundary so cron, message CLI, and send paths share the same target semantics. If upstream `imsg` does not guarantee bare hex chat identifiers, the safer implementation is fail-closed validation with a clear error and docs/tests for the explicit `chat_identifier:` form.
