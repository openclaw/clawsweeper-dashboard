## Summary

Keep open: current main has useful but partial loop guardrails, while the requested platform-level rate-limit cooldowns, error classification, user retry confirmation, and cross-tool escalation policy remain unimplemented and require maintainer product/API decisions.

## What This Changes

Request platform-level tool-call circuit breakers that block repeated failing tool calls, cool down rate-limited tools, classify retryability, and escalate repeated failures to user confirmation.

## Best Possible Solution

Define a maintainer-approved retry safety policy on top of the existing loop-detection, diagnostic error metadata, before-tool-call, and approval seams, then split focused implementations for cooldowns, classifications, confirmation UX, config, tests, and docs.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature and product-policy request. Source inspection gives a high-confidence absence check for arbitrary-tool 429 cooldowns, retry confirmation, and cross-tool consecutive failure policy on current main.

## Solution Assessment

Unclear until maintainers choose the policy. Reusing the existing before-tool-call, diagnostic, loop-detection, and approval seams looks maintainable, but default enforcement, cooldown semantics, thresholds, and override UX need design first.
