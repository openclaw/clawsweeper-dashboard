## Summary

The reported reconnect/startup race is already handled in shipped OpenClaw: the TUI now retries retryable `chat.history` `UNAVAILABLE` responses during gateway startup instead of surfacing the transient error immediately.

## Evidence

- **Current TUI retries startup `chat.history` failures:** `loadChatHistory` wraps the `chat.history` request in a retry loop, treats retryable `UNAVAILABLE` errors for `chat.history` as transient startup failures, waits `retryAfterMs`, and retries for up to 60 seconds before surfacing an error.
