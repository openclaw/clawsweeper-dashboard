## Summary

Current `main` already ships the underlying fix through the existing `compat.requiresStringContent` compatibility path for OpenAI Completions, with runtime coverage in both the core transport and embedded runner plus docs/tests for the exact `expected a string` failure mode this PR targets.

## Risks / Open Questions

- The shipped fix is opt-in via `compat.requiresStringContent`, whereas this PR proposed automatic flattening for text-only models; if maintainers still want auto-detection rather than the existing compat flag, that would be separate follow-up work.
