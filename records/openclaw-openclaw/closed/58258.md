## Summary

Current `main` has already changed live model switching so automatic fallback rotations do not count as pending live switches, and stale pending flags are cleared once the requested model is already active. That removes the retry loop described in #58258, where a successful fallback reply could keep retriggering a live-switch restart and duplicate outbound messages.

## Evidence

- **Pending live switch is explicitly user-driven, not fallback-driven:** The session entry contract now documents `liveModelSwitchPending` as only for explicit user-driven session model changes; system-initiated fallbacks never set it, so fallback rotations are no longer mistaken for live switches.
