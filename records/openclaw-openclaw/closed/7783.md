## Summary

Close as `clawhub`: current main already emits the raw JSONL surfaces and plugin-based observability hooks needed for external ingestion, and repo scope says optional capability should usually live in plugins/community packages rather than new core infrastructure.

## Evidence

- **Vision keeps optional capability out of core:** The project vision says core should stay lean, optional capability should usually ship as plugins, and the bar for adding optional plugins to core is intentionally high.
  - file: [VISION.md:55](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/VISION.md#L55)
  - sha: [cec34821754a](https://github.com/openclaw/openclaw/commit/cec34821754ad935c780805f3116f99449d46ac2)
- **Main already exposes the raw JSONL inputs:** Gateway file logs are documented as JSONL, session transcripts are stored as `~/.openclaw/agents/<agentId>/sessions/<sessionId>.jsonl`, and cache trace already writes to `~/.openclaw/logs/cache-trace.jsonl`. That means an external ingester can consume the exact surfaces this issue targets without adding new core logging formats.
  - file: [docs/logging.md:14](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/docs/logging.md#L14)
  - sha: [cec34821754a](https://github.com/openclaw/openclaw/commit/cec34821754ad935c780805f3116f99449d46ac2)
- **Observability export is already plugin-scoped:** The logging docs say diagnostics exporters only attach when the exporter plugin is enabled, and they document the `diagnostics-otel` plugin as the supported export path for telemetry.
  - file: [docs/logging.md:180](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/docs/logging.md#L180)
  - sha: [cec34821754a](https://github.com/openclaw/openclaw/commit/cec34821754ad935c780805f3116f99449d46ac2)
- **Community/ClawHub is the supported home for extra capabilities:** Community plugins are the documented path for third-party OpenClaw capabilities, and ClawHub is the canonical discovery/install surface for skills and plugins.
  - file: [docs/plugins/community.md:9](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/docs/plugins/community.md#L9)
  - sha: [cec34821754a](https://github.com/openclaw/openclaw/commit/cec34821754ad935c780805f3116f99449d46ac2)
- **No first-party observability ingestion surface exists on main:** A repo search on current main found no `openclaw observability` CLI commands and no `src/observability` implementation, which matches the plugin/community scope rather than an in-progress core feature.
  - command: `rg -n "openclaw observability|observability watch|observability ingest|observability status|observability query|src/observability" src docs package.json`
  - sha: [cec34821754a](https://github.com/openclaw/openclaw/commit/cec34821754ad935c780805f3116f99449d46ac2)
- **Issue discussion already points to an external implementation:** The issue comment from `vivekchand` points to ClawMetry as a standalone package that reads OpenClaw JSONL sessions/logs and provides the observability/dashboard workflow outside core, reinforcing that this fits the external-plugin/tool lane.

## Risks / Open Questions

- If maintainers want first-party bundled local SQLite analytics despite the current scope guidance, this issue would need reopening as an explicit product exception.
- Closing this does not provide a maintained first-party replacement; users wanting this today still depend on a third-party plugin/tool path.
