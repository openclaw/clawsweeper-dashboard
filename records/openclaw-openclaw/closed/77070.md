## Summary

Keep open: current main includes a partial SSE-default fix for direct OpenAI provider params, but the reopened report still has a source-visible OpenAI Responses auto/WebSocket path and post-fix confirmations from multiple users.

## What This Changes

The issue reports a regression where OpenAI-backed gateway and embedded agent calls hang after updating to 2026.5.2 even though direct OpenAI API calls work.

## Best Possible Solution

Make direct OpenAI Responses embedded/subagent traffic resolve to SSE unless a verified supported WebSocket endpoint is intentionally configured, then align tests, docs, and changelog with that contract.

## Reproduction Assessment

Unclear for a live current-main run: source inspection gives a credible current path through explicit or legacy `transport: "auto"` to `wss://api.openai.com/v1/responses`, and the issue has post-fix confirmations, but I did not run a live OpenAI/VPS reproduction.

## Solution Assessment

No, closing as implemented is not the best solution because the SSE default only covers part of the path and current docs/tests still preserve WebSocket-first `auto`. The narrower maintainable fix is to make direct OpenAI Responses auto/default resolve to SSE or disable the Responses WebSocket path until it is backed by a real public endpoint.

Fix openclaw/openclaw#77070, carrying the related context from openclaw/openclaw#77570. On current main, direct OpenAI Responses embedded/subagent flows can still select WebSocket when `transport` is explicit or legacy `auto`; the WebSocket manager targets `wss://api.openai.com/v1/responses`, while direct Responses traffic should use SSE. Make direct OpenAI Responses auto/default resolve to SSE in embedded attempts and compaction, keep explicit WebSocket only if a verified supported endpoint/path is intentionally configured, and otherwise guard or remove Responses WebSocket selection. Update regression tests for attempt transport selection, stream resolution, and the OpenAI provider; update `docs/providers/openai.md` and `docs/concepts/model-providers.md` so they no longer advertise WebSocket-first as the OpenAI Responses default; add a user-facing `CHANGELOG.md` fix entry with human reporter credit if appropriate. Do not change unrelated providers or Codex OAuth behavior unless tests show they share the same broken public Responses WebSocket path. Stop and hand back if the implementation becomes a broader product decision about supporting OpenAI WebSocket Responses.
