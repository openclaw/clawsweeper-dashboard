## Label Justifications

- `P1`: The report describes a real released-version agent/channel workflow failure that drops replies and wedges gateway dispatch until restart.
- `impact:message-loss`: Telegram messages are acknowledged but no agent reply is delivered when the harness/runtime path fails.
- `impact:crash-loop`: The gateway degrades into repeated restarts and hanging child processes, affecting process-level availability.
- `impact:auth-provider`: The root symptom is provider/runtime routing treating claude-cli as an unregistered agent harness.

## Summary

Current main already separates provider-owned CLI runtimes from embedded agent harnesses and has Claude CLI session serialization/watchdog handling, so the reported v2026.5.22 idle-resume failure and spawn pile-up are addressed on main but not yet in the latest release.

## What This Changes

The issue reports that v2026.5.22 can throw MissingAgentHarnessError for the claude-cli runtime after idle, then pile up concurrent hanging Claude CLI resume processes for the same agent/session.

## Best Possible Solution

Keep the current main CLI-runtime/harness separation and Claude CLI serialization behavior, then ship it in the next release; reopen only with fresh logs from a build newer than 455d5e807cc5a4e3c558a0290f937864fe33becb if the same idle-resume path still fails.

## Reproduction Assessment

No current-main failing reproduction was established; source inspection shows current main routes claude-cli as a CLI runtime and serializes resume dispatch. The detailed v2026.5.22 report plus the release-to-main diff give a high-confidence old-release failure path.

## Solution Assessment

Yes. The current main fix is the narrow maintainable direction because it preserves the core boundary that claude-cli is a CLI backend, not an embedded harness, and controls same-session resume concurrency in the CLI runner.
