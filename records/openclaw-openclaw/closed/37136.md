## Summary

Current `main` already contains the Telegram self-message guard that prevents the reported heartbeat self-reply loop, and the same guard is present in release `v2026.4.22`.

## Evidence

- **Telegram message handler skips bot-authored echoes:** The inbound `message` handler returns early when `normalizedMsg.from.id === ctx.me?.id`, which directly blocks Telegram from re-ingesting the bot's own outbound messages.
  - file: [extensions/telegram/src/bot-handlers.runtime.ts:1955](https://github.com/openclaw/openclaw/blob/7d471837369885c12d540d6bafa319ff41129b89/extensions/telegram/src/bot-handlers.runtime.ts#L1955)

  - sha: [7d4718373698](https://github.com/openclaw/openclaw/commit/7d471837369885c12d540d6bafa319ff41129b89)
- **Legitimate channel-originated bot traffic still has a separate path:** A dedicated `channel_post` handler remains in place immediately after the self-echo guard, so the guard does not remove intended bot-to-bot channel handling.
  - file: [extensions/telegram/src/bot-handlers.runtime.ts:1978](https://github.com/openclaw/openclaw/blob/7d471837369885c12d540d6bafa319ff41129b89/extensions/telegram/src/bot-handlers.runtime.ts#L1978)

## Close Comment

Closing as implemented on main after Codex review. Current code already skips Telegram `message` updates authored by the bot itself in `extensions/telegram/src/bot-handlers.runtime.ts:1953-1956`, which is the self-reply path described here, and it preserves intended bot-to-bot channel handling via the separate `channel_post` handler at `extensions/telegram/src/bot-handlers.runtime.ts:1975-2025`. I also confirmed the same guard is present in release `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). If this still reproduces on a newer build, please open a fresh issue with updated logs and exact Telegram channel/account config.
