## Summary

Current main now handles the reported silent empty-error turn: replay-safe `stopReason="error"` turns with zero output and empty content are retried, and provider errors with messages are surfaced through the runner instead of disappearing.

## What This Changes

The issue reports that Baidu Qianfan API errors can produce an empty assistant response because the transcript records `stopReason: "error"`, no error text, and no content.

## Best Possible Solution

Keep the shared provider-error and empty-error runner handling; if Qianfan later exposes a distinct dropped `error_msg` payload after this ships, track that as a narrower provider payload-normalization bug with a redacted raw response sample.

## Reproduction Assessment

Yes, by source and regression-test shape rather than a live Qianfan call: current tests model `stopReason="error"`, zero output, and empty content, which matches the reported silent transcript pattern.

## Solution Assessment

Yes. The maintainable fix is in the shared OpenAI-compatible runner path because Qianfan uses that transport, with provider-specific parsing left only for a future raw payload shape that still escapes this handling.
