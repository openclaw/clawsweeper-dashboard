## Label Justifications

- `P1`: A selected Gemini CLI OAuth profile can be ignored during real agent execution, potentially breaking provider routing or using the wrong Google account for affected users.
- `impact:auth-provider`: The issue is about OAuth profile selection, provider runtime auth routing, and the external Gemini CLI credential source.
- `impact:security`: The report and fix path involve OAuth tokens and preventing selected credentials from being bypassed, leaked, or confused with ambient credentials.

## Summary

Keep open: current main has Gemini CLI OAuth refresh support and a generic CLI `prepareExecution` hook, but the bundled `google-gemini-cli` backend still launches `gemini` without an owner bridge that stages the selected OpenClaw OAuth profile for the subprocess.

## What This Changes

The issue asks for `google-gemini-cli` runs to honor the selected OpenClaw OAuth profile instead of ambient Gemini CLI credentials.

## Best Possible Solution

Add a Google-owned `prepareExecution` bridge that refreshes the selected OpenClaw profile, stages Gemini-compatible credentials under an isolated `GEMINI_CLI_HOME`, clears conflicting env as needed, cleans up safely, and proves the ambient-credential mismatch path.

## Reproduction Assessment

Yes, source-reproducible. Current main has the runner hook and refreshed OpenClaw profile data, but the Google CLI backend lacks the bridge while the upstream CLI reads its own `GEMINI_CLI_HOME` or `~/.gemini` state; I did not run a live Google account flow.

## Solution Assessment

Yes, the proposed owner-local `prepareExecution` bridge is the right layer because the plugin SDK already reserves that hook for auth/config bridges. The security details need owner review before automation writes OAuth material for an external CLI.
