## Summary

Keep open: current main still has the migration-order bug described by the report. The allowed-origin seed checks the raw legacy bind value before the later alias-normalization migration maps `0.0.0.0` to `lan`, so a legacy non-loopback config can be normalized without getting the required Control UI origins.

## What This Changes

Fix legacy doctor gateway migrations so host-style bind aliases that normalize to non-loopback modes also seed `gateway.controlUi.allowedOrigins`.

## Best Possible Solution

Normalize bind aliases before non-loopback origin seeding, or make the seeding step resolve aliases internally, while preserving existing-origin and Host-header fallback no-op behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: run legacy doctor migration on a config with `gateway.bind: "0.0.0.0"` and no `gateway.controlUi.allowedOrigins`; the seed checks the alias before normalization and returns, then the later migration maps the bind to `lan`.

## Solution Assessment

Yes. The narrow maintainable fix is to resolve or run the bind-alias normalization before the allowed-origin seed and add a focused regression test for non-loopback aliases plus loopback aliases.
