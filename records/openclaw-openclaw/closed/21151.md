## Summary

Current main already fixes the reported node-camera SSRF pivot: URL downloads are bound to the resolved node host, require HTTPS, use `fetchWithSsrFGuard`, re-check redirect targets, and fail closed when the node `remoteIp` is unavailable. The changelog records this as shipped in `v2026.3.2`.

## Close Comment

Closing from a Codex review: current `main` already addresses this report. `writeUrlToFile()` is no longer a bare `fetch()` path; it now requires HTTPS, binds the requested URL and any redirect target to the resolved node host, and routes the download through `fetchWithSsrFGuard` in `src/cli/nodes-camera.ts:77`. The callers pass `resolvedNode.remoteIp` / `node.remoteIp` as `expectedHost`, and the helper fails closed when `remoteIp` is unavailable (`src/agents/tools/nodes-tool-media.ts:123`, `src/cli/nodes-cli/register.camera.ts:174`, `src/cli/nodes-camera.ts:185`). Regression coverage is in `src/cli/nodes-camera.test.ts:143` and `src/cli/nodes-camera.test.ts:181`. The changelog records this fix in `v2026.3.2` at `CHANGELOG.md:3326`, so there is nothing left to track in this issue.
