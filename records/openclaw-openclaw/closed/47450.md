## Summary

Current main already implements the reported vLLM/OpenAI-style usage alias handling. OpenClaw now normalizes `prompt_tokens`/`completion_tokens`/`total_tokens`, applies that normalization when building assistant usage, and ships regression coverage for the exact alias mapping; the fix is also present in release `v2026.4.22`.

## Evidence

- **Shared usage normalization handles OpenAI-style aliases:** `normalizeUsage()` accepts `prompt_tokens`, `completion_tokens`, and `total_tokens` and maps them onto OpenClaw's normalized usage shape.

- **Response conversion uses normalized usage instead of raw field names:** The assistant-message conversion path calls `normalizeUsage(response.usage)` and derives `totalTokens` from normalized fields when needed, so alias-shaped provider payloads do not stay at zero.
  - file: [src/agents/openai-ws-message-conversion.ts:598](https://github.com/openclaw/openclaw/blob/0e7250f37bf347c72790abf2a6692d5ec84e5ea1/src/agents/openai-ws-message-conversion.ts#L598)
