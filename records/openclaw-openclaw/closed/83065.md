## Summary

Current main and the shipped v2026.6.6 release now resolve canonical inbound `media://inbound/<id>` references in the built-in `read` path for both host and sandbox reads, so the central Signal attachment access gap is fixed.

## What This Changes

Signal inbound attachments exposed as `media://inbound` refs should be accessible to agents through a built-in mechanism instead of requiring skill prompt workarounds.

## Best Possible Solution

Keep the shared media-reference resolution in the core read path and treat any future Signal-specific attachment failure as a new, narrower regression report with fresh trajectory evidence.

## Reproduction Assessment

Yes. The original failure is source-reproducible from Signal/media prompt notes handing `media://inbound` to a read path that previously treated it as a cwd-relative path; current main no longer follows that failing path.

## Solution Assessment

Yes. Reusing the shared media-reference resolver in the host and sandbox read paths is narrower and safer than adding a Signal-only env var, a new tool, or broad shell URI rewriting.
