## Summary

Current main now fixes the reported Discord self-message loop path: Discord monitor startup fails closed when the bot identity cannot be resolved, the own-bot ingress filters still run before debounce/preflight, and session rollover clears queued followups/lane state. The earlier unmerged PR #46338 is useful context, but current main resolves the observable bug by making a missing bot identity non-runnable instead of allowing contaminated messages into replay state.

## What This Changes

The issue asks OpenClaw to prevent Discord bot-authored messages from entering ingress, replay, or followup queues and surviving session resets.

## Best Possible Solution

Keep the current fail-closed Discord identity requirement and early own-bot ingress filters. If a future bypass creates replay contamination without going through Discord ingress, handle that as a narrower hardening follow-up by adding trusted channel author metadata rather than reopening this original startup failure path.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes: the mocked provider tests cover the original `fetchUser("@me")` rejection and missing-id cases and verify the monitor fails before message handling starts, so the original fail-open path no longer reproduces on current main.

## Solution Assessment

Is this the best way to solve the issue? Yes for the reported bug: failing closed before listener registration is a narrower and safer fix than accepting Discord input with unknown bot identity, and queue cleanup covers the `/new` persistence symptom.
