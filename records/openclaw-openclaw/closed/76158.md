## Summary

Current main already handles BlueBubbles tapback payloads with structural association fields as reaction events before message normalization and model dispatch, so the reported before_dispatch synthetic-text leak and resulting empty-turn fallback should not reproduce on main.

## What This Changes

The issue asks OpenClaw to preserve BlueBubbles tapback metadata through dispatch hooks and prevent reaction-only inbound events from producing generic empty-turn fallback replies.

## Best Possible Solution

Keep the current BlueBubbles reaction-event routing on main; broader no-visible-response fallback behavior should remain with its separate channel/runtime trackers.

## Reproduction Assessment

Yes. A BlueBubbles webhook payload with associatedMessageGuid plus associatedMessageType in the tapback ranges is enough to trace the old path, and current main routes it through normalizeWebhookReaction/processReaction before before_dispatch can run.

## Solution Assessment

Yes. For this BlueBubbles-specific leak, handling structural tapbacks as reaction events before model dispatch is narrower and safer than exposing channel-specific tapback fields to before_dispatch just so downstream plugins can block them.
