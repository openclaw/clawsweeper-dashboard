## Summary

Close as superseded. The same direct top-level custom-field request was already resolved by a maintainer decision in #1225/#1226 to keep `skills.entries.<name>` strict and support custom data under `skills.entries.<name>.config`; that contract shipped in `v2026.1.20`, and the later top-level catchall PR #74024 was closed unmerged for the same reason.

## What This Changes

The issue asks for custom fields such as `api_endpoint` and `user_api_key` to be accepted directly under `skills.entries.<skill-name>` during Gateway config validation.

## Best Possible Solution

Keep the shipped strict top-level skill-entry contract, use `skills.entries.<skill>.config` for custom skill-owned settings, and route Tavily credentials through `plugins.entries.tavily.config.webSearch.*` or `TAVILY_API_KEY`. A new issue is only needed if a concrete skill cannot consume the supported nested config path.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: `OpenClawSchema.safeParse` rejects an unknown direct field under `skills.entries.custom-skill`, the focused test asserts that rejection, and Gateway `config.patch` invokes the same validation path.

## Solution Assessment

Yes. Closing as superseded is the best cleanup path because the earlier maintainer decision intentionally preserves typo detection at the top level while providing a supported custom `config` bag for skill-specific data.
