## Summary

Close: merged dotenv hardening now supersedes this PR by preserving the fail-closed workspace dotenv trust boundary, while the PR branch still proposes a security-sensitive `OPENCLAW_CONFIG_PATH` exception that conflicts with current docs and tests.

## What This Changes

The PR removes the workspace-dotenv block for `OPENCLAW_CONFIG_PATH`, keeps `OPENCLAW_STATE_DIR` blocked, precomputes the global dotenv path before workspace loading, and updates dotenv tests.

## Best Possible Solution

Keep the fail-closed workspace dotenv policy on main and use trusted process or global `~/.openclaw/.env` for `OPENCLAW_CONFIG_PATH`; any project-local opt-in should be a fresh maintainer/security decision tracked in the linked issue.

## Reproduction Assessment

Yes, source-level: putting `OPENCLAW_CONFIG_PATH` in a workspace `.env` is filtered by `loadWorkspaceDotEnvFile` on current main before config path resolution can see it. I did not run tests because this review is read-only.

## Solution Assessment

No. The PR's direct exception is no longer the safest solution after the merged runtime-control hardening; a trusted global env surface or explicit maintainer-approved opt-in is the narrower maintainable path.
