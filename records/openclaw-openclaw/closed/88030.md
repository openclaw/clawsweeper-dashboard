## Summary

Keep open: current main and the latest release still build Feishu post replies as one `md` element containing the whole text, while the reported Feishu mention tags are generated as literal text before sending. The rendering-layer bug is source-reproducible, but the reported bot-cache collision path does not match current source names and should be verified before changing cache behavior.

## What This Changes

The issue reports that Feishu post-mode messages render OpenClaw `<at user_id="...">name</at>` mentions as plain text and asks to emit native Feishu `at` elements while avoiding bot-name cache conflicts.

## Best Possible Solution

Add a focused Feishu plugin fix that converts explicit `<at user_id="...">name</at>` text in post-mode outbound content into native Feishu `at` post elements, and change bot-name caching only if the current implementation still has a reproducible identity collision.

## Reproduction Assessment

No high-confidence live repro was run. The current source path is clear: OpenClaw can generate textual `<at user_id="...">...</at>` tags and then sends the whole post body as one `md` element.

## Solution Assessment

Yes for the rendering layer: splitting explicit at tags into native Feishu post elements is the narrow owner-boundary fix. The cache-layer suggestion is unclear on current main because the named cache/spec path was not found.
