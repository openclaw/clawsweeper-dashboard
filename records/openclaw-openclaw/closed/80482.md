## Summary

Keep open. The PR is a plausible fix for the linked inline API-key billing cooldown bug, but it still misses configured env SecretRef inline keys because ambient env auth can win before the new inline cooldown/provenance branch runs.

## What This Changes

The branch adds provider-level cooldown usage state for inline `models.providers.<id>.apiKey` credentials, skips disabled inline keys during auth checks, marks no-profile runner failures, and adds auth/runner regression tests.

## Best Possible Solution

Keep the provider-level cooldown approach, but preserve configured inline API-key provenance through env SecretRef resolution so literal keys and env SecretRefs are both marked and skipped without synthetic profiles or secret persistence.

## Reproduction Assessment

Yes, source-level. Current main resolves inline `models.providers.<id>.apiKey` credentials without a profile id, and the runner drops classified failures before recording cooldown state when no profile id exists; the PR's remaining gap is also source-reproducible through the env SecretRef branch.

## Solution Assessment

No. The provider-level cooldown identity is the right shape, but the current diff needs to handle configured env SecretRefs before ambient env auth wins, or otherwise tag configured SecretRefs distinctly from unrelated environment auth.
