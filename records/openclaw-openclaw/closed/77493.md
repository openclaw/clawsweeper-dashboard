## Summary

Keep open. Current main still has the source-backed gap: heartbeat runs can select a transient model override, but session_status only builds its model card from session/config state, and an open closing PR is already paired with this report.

## What This Changes

session_status should report the active heartbeat.model runtime override for sessionKey="current" during heartbeat runs instead of the stored session/default model.

## Best Possible Solution

Review and land PR #77553, or an equivalent narrow fix, so only semantic-current session_status calls receive the selected runtime model without changing other-session lookups or persisted session state.

## Reproduction Assessment

Yes, at source level: configure heartbeat.model and have the heartbeat call session_status({"sessionKey":"current"}); current main selects the heartbeat override for execution but never passes the selected provider/model into the status tool. I did not perform a live heartbeat run because this review is read-only.

## Solution Assessment

Yes, the narrow maintainable solution is to thread the already-selected runtime provider/model into session_status for semantic-current reads only. Updating SessionEntry persistence would be the wrong direction because related reports require heartbeat model state not to leak into normal session status.
