## Summary

Keep this issue open: current main and the latest release still contain the 60 second aggregate compaction retry wait, and five open closing PRs already own candidate fixes.

## What This Changes

The issue reports that in-turn compaction abandons valid long-running retry results after a hardcoded 60 second aggregate wait, causing overflow retry storms on large sessions.

## Best Possible Solution

Review and land one canonical fix, with https://github.com/openclaw/openclaw/pull/94421 currently owning the maintainer path for treating active retry streaming as active compaction retry work while preserving the idle fallback.

## Reproduction Assessment

Yes, at source level: current main still races a fixed 60 second aggregate wait against retry work, and the reporter supplied production trajectories where valid model results arrived after that window. I did not run a paid 200K-token provider reproduction in this read-only review.

## Solution Assessment

Yes for keeping this issue open: open closing PRs are the right implementation path, and closing the issue before a candidate merges would lose the canonical bug report. The best repair appears to preserve the idle deadlock guard while treating the retry model request as active work.
