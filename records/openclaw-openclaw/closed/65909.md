## Summary

Close as implemented: current main forwards the missing agent/workspace directory context into provider stream wrappers, Codex native search activation can now see persisted `openai-codex` auth before deciding `codex_auth_missing`, and the focused fix from #67843 shipped in v2026.4.27. The remaining planner/tool-call visibility question is better handled as a new focused report if someone can show the native tool is attached but not used.

## What This Changes

Ensure Codex native `web_search` activates for eligible Codex Responses runs when web search and `tools.web.search.openaiCodex` are enabled and valid Codex auth exists in the agent auth-profile store.

## Best Possible Solution

Keep the shipped generic provider-wrapper context forwarding and existing Codex native search activation contract; if a new run shows the native tool is present but the model does not call it, track that separately with provider payload and trajectory evidence rather than reopening the fixed auth-context issue.

## Reproduction Assessment

Yes for the concrete failure mode: before the fix, the provider wrapper context omitted `agentDir`, making persisted Codex auth invisible to native-search activation. Current source and regression coverage verify that the directory now reaches the activation path.

## Solution Assessment

Yes. Forwarding the already-available `agentDir` and `workspaceDir` through the generic provider wrapper context is the narrow maintainable fix and avoids adding Codex-specific startup migrations or core owner IDs.
