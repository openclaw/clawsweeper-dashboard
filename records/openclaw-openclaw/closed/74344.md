## Summary

Current main already implements the requested Telegram polling watchdog escape hatch: the hardcoded 90s threshold was replaced by a 120s default plus bounded top-level and per-account `pollingStallThresholdMs` config, with runtime wiring, schema validation, tests, docs, and release provenance. The reported v2026.4.15 build predates the fix, which shipped after that release.

## What This Changes

The issue reports Telegram polling watchdog livelock when the runner is rebuilt before the first `getUpdates`, and requests an operator-configurable stall threshold or documentation that the threshold is fixed.

## Best Possible Solution

Keep the shipped `pollingStallThresholdMs` implementation and direct affected operators to upgrade to at least `v2026.4.20`; if a post-upgrade deployment still livelocks with an appropriately raised threshold, it should be tracked as a fresh current-main reproduction.
