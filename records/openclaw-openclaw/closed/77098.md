## Summary

Keep this PR open, but do not merge it as-is: the helper-level first-wins change does not fix the loader snapshot/restore path described in the related report and it changes same-plugin replacement semantics without tests or docs.

## What This Changes

The PR changes `registerMemoryCorpusSupplement` to keep the first same-plugin corpus supplement registration and ignore later duplicate registrations.

## Best Possible Solution

Fix the loader or service registration boundary so non-activating lazy loads preserve already active corpus supplements while keeping intentional same-plugin replacement semantics, then add regression coverage and a changelog entry.

## Reproduction Assessment

No. The related issue has a detailed live trace and current source shows the plausible restore path, but this review did not establish a high-confidence executable current-main reproduction and the PR adds no regression test.

## Solution Assessment

No. The helper-level first-wins change does not address the restore that happens after registration and can mask legitimate supplement refreshes; the narrower fix belongs in loader state restoration or the service/runtime registration contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Fix the loader restore path instead:** `src/plugins/memory-state.ts:157-161`
  - body: The described loss happens when an `activate:false` load restores `previousMemoryCorpusSupplements` after `register()` runs. If that snapshot lacks the service-start supplement, this helper is either not called or its mutation is immediately overwritten, so the supplement can still be lost while later same-plugin refreshes are ignored. Please preserve or merge the active supplement at the loader restore boundary and add the lazy re-register regression test.
  - confidence: 0.86
