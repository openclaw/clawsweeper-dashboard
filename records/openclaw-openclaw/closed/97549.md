## Summary

This PR should stay open: current main and v2026.6.10 still have the unbounded no-output video URL buffer path, while the PR uses the shared bounded reader and adds focused regression coverage. I found no blocking code finding; the remaining blocker is maintainer acceptance of the intentional fail-closed behavior for oversized no-output downloads.

## What This Changes

The PR bounds `openclaw capability video generate` no-output provider URL downloads with `readResponseWithLimit`, redacts failed download URLs, and adds CLI regression tests for overflow, under-cap, configured-cap, empty-body, and HTTP-failure cases.

## Best Possible Solution

Land the bounded-reader change if maintainers accept the fail-closed cap, keeping `--output` as the large-video streaming path and keeping provider URL errors redacted.

## Reproduction Assessment

Yes. Source inspection shows current main and v2026.6.10 both fetch provider video URLs and buffer the no-output path with `Buffer.from(await response.arrayBuffer())`, and the PR tests drive that same CLI path with streamed oversized bodies.

## Solution Assessment

Yes, with maintainer sign-off. The CLI fallback is the right owner boundary and the shared bounded reader matches sibling generated-media code; the remaining question is accepting the fail-closed cap for upgrades.
