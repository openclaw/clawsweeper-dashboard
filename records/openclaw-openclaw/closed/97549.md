## Summary

Keep this PR open: the patch is a focused, well-supported fix for the CLI no-output video URL fallback, but it deliberately changes oversized downloads from best-effort buffering to a cap error, so maintainer compatibility sign-off is still needed before merge.

## What This Changes

This PR routes the CLI video generation no-output provider URL fallback through a bounded response reader, redacts failed download URLs, and adds focused CLI regression tests.

## Best Possible Solution

Land the bounded-reader change after maintainers accept the fail-closed cap, keeping `--output` as the large-video streaming path and keeping provider URL errors redacted.

## Reproduction Assessment

Yes. Source inspection shows current main and v2026.6.10 both buffer provider video URL responses with `arrayBuffer()` on the no-output path, and the PR proof demonstrates the old path drains a 64 MiB response while the bounded path stops near the cap.

## Solution Assessment

Yes, with maintainer compatibility sign-off. The CLI fallback is the right owner boundary and the shared bounded reader matches sibling generated-media code; the remaining decision is accepting fail-closed behavior for oversized no-output downloads.
