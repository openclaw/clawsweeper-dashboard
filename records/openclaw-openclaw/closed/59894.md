## Summary

Current `main` already wires doctor’s memory check through the active runtime plugin registry and only emits this warning when no memory runtime can actually be resolved. The selected memory-slot path is covered in tests and the release changelog shows the doctor memory status path was updated to respect the selected memory plugin.

## Evidence

- **Doctor warning only fires when no active memory runtime resolves:** `noteMemorySearchHealth()` now checks `resolveActiveMemoryBackendConfig({ cfg, agentId })` and only prints `No active memory plugin is registered for the current config.` when that call returns `null`.
  - file: [src/commands/doctor-memory-search.ts:332](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/commands/doctor-memory-search.ts#L332)

  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **Doctor bootstraps memory runtime from the provided config before checking backend:** `ensureMemoryRuntime()` loads the runtime plugin registry from `resolvePluginRuntimeLoadContext({ config: cfg })`, then resolves the active memory backend from that loaded runtime. This means doctor is no longer limited to a pre-registered builtin memory runtime.
