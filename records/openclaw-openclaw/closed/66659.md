## Summary

Keep open. Current `main` does mint a new `sessionId` for heartbeat isolated runs, but the saved `:heartbeat` entry still starts by spreading the previous `SessionEntry`, and reply-session init then deliberately reuses that freshly written heartbeat/system-event entry as a non-new session. Successful runs do overwrite some usage fields later, but there is not enough evidence that `heartbeat.isolatedSession: true` now guarantees a fully fresh per-tick session state, especially for carried metadata/session bindings. The latest tagged release in this checkout (`v2026.4.23`) contains the same code shape the report is pointing at, so this is not strong enough to auto-close as implemented.

## Risks / Open Questions

- No direct heartbeat-specific regression test proves that `isolatedSession: true` clears carried session metadata/bindings on every tick.
- Fresh-session semantics remain ambiguous for provider-owned/CLI-backed sessions because reused bindings can survive the heartbeat handoff path.
- The latest tagged release visible in this checkout (`v2026.4.23`) contains the same core implementation shape, so there is insufficient evidence that the report is obsolete.
