## Summary

Current `main` no longer treats `models.providers` as a required registry for fallback providers. Bundled/implicit providers are discovered separately, auth resolution prefers auth profiles and env vars before `models.providers.*`, and fallback tests on `main` exercise cross-provider fallback without any `models.providers` block. The issue's reported failure mode does not match current behavior.

## Evidence

- **`models.providers` is optional override config, not the required source of truth:** The provider docs say explicit `models.providers.<id>` entries are only for overriding bundled provider defaults, not for defining every fallback target.
  - file: [docs/concepts/model-providers.md:262](https://github.com/openclaw/openclaw/blob/a958b6e72328d1d922a6d133b1a70e866faf4013/docs/concepts/model-providers.md#L262)

  - sha: [a958b6e72328](https://github.com/openclaw/openclaw/commit/a958b6e72328d1d922a6d133b1a70e866faf4013)
- **Runtime model catalog merges implicit provider discovery with explicit providers:** `planOpenClawModelsJson` loads `explicitProviders = cfg.models?.providers ?? {}` and separately calls `resolveImplicitProviders(...)`, then merges both sets. `resolveImplicitProviders` discovers provider catalogs from provider plugins/auth context, so fallback providers do not need explicit `models.providers` entries to exist on current main.
  - file: [src/agents/models-config.plan.ts:47](https://github.com/openclaw/openclaw/blob/a958b6e72328d1d922a6d133b1a70e866faf4013/src/agents/models-config.plan.ts#L47)

- **Auth resolution does not require `models.providers` entries first:** `resolveApiKeyForProvider` checks auth profiles and env-backed auth before falling back to `models.providers.*.apiKey`, which means bundled fallback providers can be usable without explicit provider config.
