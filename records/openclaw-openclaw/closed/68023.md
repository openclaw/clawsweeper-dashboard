## Summary

Current `main` already enables HTTP(S) proxy handling for CLI/gateway startup and provider request paths, so the PR's core premise that providers silently ignore `HTTP_PROXY`/`HTTPS_PROXY` no longer matches shipped behavior.

## Evidence

- **CLI entry reaches `runCli()`:** The main CLI entry imports `./cli/run-main.js` and calls `runCli(argv)`, so gateway startup goes through the shared CLI bootstrap path.

That guarded fetch path builds no forced-direct dispatcher when `request.proxy` is absent, then falls back to the ambient/global fetch path when no per-request dispatcher is attached, so env-proxy bootstrap still applies without `models.providers.<name>.request.proxy.mode = "env-proxy"`.

## Risks / Open Questions

- I did not run a live gateway/chat reproduction in this review; the workspace lacks `pnpm`, so this conclusion is based on current `main` code paths, docs, and tests rather than an end-to-end runtime check.
- A provider-owned plugin that bypasses both the global Undici dispatcher and the guarded fetch helpers could still have bespoke proxy behavior, but this PR's startup heuristic would not reliably detect that class of issue anyway.
