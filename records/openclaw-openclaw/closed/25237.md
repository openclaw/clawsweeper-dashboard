## Summary

The request is a real capability gap, but current main and project scope point to an external plugin/MCP integration rather than an OpenClaw core feature. The repo vision explicitly keeps optional capabilities out of core, prefers MCP through `mcporter`, and uses external plugins/ClawHub for specialized integrations.

## Close Comment

Thanks for the write-up. Closing this as a `clawhub` scope item after a Codex review of current `main`.

The main reason is project scope, not usefulness. Current guidance says optional capability should usually ship as plugins rather than core (`VISION.md:54`), and OpenClaw explicitly prefers MCP through `mcporter` over a first-class core MCP/runtime path (`VISION.md:72`, `VISION.md:99`). ClawHub is also the supported distribution path for external skills/plugins (`docs/tools/clawhub.md:10`).

I also checked the current macOS node surface: the app does request/check Accessibility permission (`apps/macos/Sources/OpenClaw/PermissionManager.swift:85`), but current `main` only exposes that as permission metadata (`docs/nodes/index.md:376`) and advertises canvas/screen/system/camera/location/browser commands, not native AX automation commands (`apps/macos/Sources/OpenClaw/NodeMode/MacNodeModeCoordinator.swift:139`). That makes this a better fit for an external plugin or MCP server, similar to how other specialized integrations stay outside core (`docs/channels/wechat.md:27`).

If you want to pursue it, the recommended path is an external plugin and/or MCP server distributed outside core, with ClawHub as the natural packaging/discovery route.
