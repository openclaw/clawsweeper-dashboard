## Summary

Current main already resolves the reported Discord false stale-socket restart path through the broader channel-health change shipped in v2026.4.23: stale-socket recovery now keys off provider-reported transport activity, not `lastEventAt` app-message freshness, so quiet Discord channels are no longer restarted just because no inbound Discord messages arrived.

## Evidence

- **Health policy no longer uses app events as socket liveness:** `src/gateway/channel-health-policy.ts:115` explicitly says app-level events are not socket liveness and only evaluates stale-socket when `lastTransportActivityAt` is present.
  - file: [src/gateway/channel-health-policy.ts:115](https://github.com/openclaw/openclaw/blob/9b18ae877018a34933e27bb7b2986cb97e9602aa/src/gateway/channel-health-policy.ts#L115)

## Risks / Open Questions

- Current main fixes the reported behavior by changing the gateway health policy globally rather than merging this PR's Discord-specific raw-socket listener. If maintainers still want Discord to participate in positive stale-socket detection via heartbeat ACK traffic, that is follow-up work, not required to resolve this regression.
