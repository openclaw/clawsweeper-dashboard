## Summary

The request is a workflow-specific summary/checkpoint policy that fits OpenClaw's plugin/skill layer, not core. Current main already exposes compaction-provider and compaction-hook seams that can implement required-field checks, strict blocking, and suggestions outside core.

## Evidence

- **Vision scope: optional capability belongs in plugins/ClawHub:** The vision doc says core should stay lean, optional capability should usually ship as plugins, and new skills should be published to ClawHub first rather than added to core by default.
  - file: [VISION.md:54](https://github.com/openclaw/openclaw/blob/a59d1bd46d9fc0bd2d9194b3f17db2561efc51e3/VISION.md#L54)

  - file: [docs/concepts/compaction.md:75](https://github.com/openclaw/openclaw/blob/a59d1bd46d9fc0bd2d9194b3f17db2561efc51e3/docs/concepts/compaction.md#L75)

## Risks / Open Questions

- OpenClaw does not ship this policy out of the box today; closing here assumes the requested behavior is acceptable as an external ClawHub plugin/skill rather than a core guarantee.
- If someone wants to enforce checks against the produced summary text itself, they will likely need a custom compaction provider; the current before/after compaction hooks are observer seams, not a built-in validator pipeline.

## Close Comment

Closing as `clawhub` based on a Codex review of current `main`. This request is a workflow-specific checkpoint/summary policy rather than a core compaction invariant. The repo vision explicitly keeps optional capability in plugins and sends new skills to ClawHub first (`VISION.md:54-70`). OpenClaw already exposes the right seams for this: custom compaction providers (`docs/concepts/compaction.md:75-93`, `src/plugins/types.ts:2010-2013`) and compaction lifecycle hooks (`src/plugins/hook-types.ts:667-674`). That means a ClawHub plugin/skill can enforce required fields like deadline/owner/acceptance criteria, warn or block in strict mode, and add suggestions without hardcoding one workflow schema into OpenClaw core.
