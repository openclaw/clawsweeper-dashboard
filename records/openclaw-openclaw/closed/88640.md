## Label Justifications

- `P0`: The report is about a revoked Slack token crash-looping the whole gateway and making core runtime unavailable on the affected release.
- `impact:crash-loop`: The reported failure mode is a gateway process exit and systemd restart loop.
- `impact:auth-provider`: The trigger is a channel auth failure from revoked or invalid Slack credentials.

## Summary

Current main and the published v2026.5.28 OpenClaw/Slack packages contain the Slack fatal-auth handling and gateway channel-error containment needed for the reported crash-loop, so the 2026.5.20 behavior is behind current installable code.

## What This Changes

The issue asks OpenClaw to ship Slack `invalid_auth` containment to a release, keep the gateway alive when Slack credentials are revoked, and avoid supervisor crash-loops.

## Best Possible Solution

Keep the shipped channel-scoped degradation path: revoked Slack credentials should degrade Slack and leave the gateway running, rather than adding a supervisor-facing process exit for this channel-local condition.

## Reproduction Assessment

Yes for the affected old release path: the report has concrete revoked-token steps and logs, and current source/history shows the matching Slack auth and channel-containment fixes. I did not reproduce a failing current-main path because current main and the current published packages contain the containment.

## Solution Assessment

Yes. The maintainable fix is channel-scoped degradation with recorded Slack lastError while the gateway stays alive; a distinct gateway process exit would re-couple one Slack account failure to the supervisor.
