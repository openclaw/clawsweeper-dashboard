## Repair Work Prompt

Fix the memory indexing embedding batch-limit regression reported in https://github.com/openclaw/openclaw/issues/80226. Current main chunks individual text inputs through `enforceEmbeddingMaxInputTokens`, but unknown OpenAI-compatible models still default to the generic 8192/8000 budget, so providers such as Qianfan `bge-large-zh` that reject combined batches around 2000 tokens can still fail `openclaw memory index --force` before storing chunks. Keep the fix inside the memory/provider embedding boundary; prefer provider-declared limits or adaptive split/retry-on-low-limit behavior over broad startup migrations or unrelated provider policy. Add regression coverage for the reported `embeddings max length per batch size is 2000` failure path, including that reducible text batches are split or retried and that indexing does not lose all progress for a low-limit batch. Stop for maintainer guidance if the implementation needs a new public config option. Add a user-facing changelog entry if behavior changes.
