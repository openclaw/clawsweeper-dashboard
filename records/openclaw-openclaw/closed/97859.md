## Summary

Close as duplicate/superseded: the report is source-reproducible, but the same local CLI shared-token/device-identity root cause is already tracked by canonical issue #95997 with open fix PR #96002; broader recovery UX remains in #74484.

## What This Changes

`devices approve --token` on a loopback Gateway should authenticate through the shared Gateway credential without presenting the local CLI device identity and generating a replacement scope-upgrade request.

## Best Possible Solution

Track the remaining implementation through #95997 and land PR #96002 or an equivalent narrow successor, while keeping #74484 for the broader recovery UX contract.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.10 explicitly preserve device identity for explicit CLI loopback shared-token auth, and `devices approve` uses that CLI call path before the Gateway can issue a replacement scope-upgrade request.

## Solution Assessment

Yes for closure as a duplicate, not as fixed. The best path is the existing canonical issue and PR because #96002 changes the exact client/server boundary instead of adding a one-off `devices approve` workaround.
