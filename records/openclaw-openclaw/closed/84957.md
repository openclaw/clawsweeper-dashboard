## Summary

Keep open: current main still has a source-level path where an isolated cron run forwards `toolsAllow: ["exec"]` into the Codex app-server runtime, but the Codex narrow-allowlist path disables native code mode and filters OpenClaw `exec`/`process` out of dynamic tools, so the run can start with no usable shell tool. The related embedded-tool issue and PR do not cover this Codex app-server policy path.

## What This Changes

The issue reports isolated cron `agentTurn` jobs whose `payload.model` resolves to OpenAI Codex starting without a usable `exec` tool despite `payload.toolsAllow: ["exec"]`.

## Best Possible Solution

After cron resolves the model/runtime, validate whether the selected Codex runtime can honor the requested allowlist; expose a safe exec-capable native or sandbox-backed path only when policy allows it, otherwise fail before the model turn with an explicit cron/runtime error.

## Reproduction Assessment

Yes, at source level: cron forwards `toolsAllow: ["exec"]` into the Codex app-server path, narrow allowlists disable native code mode, and default dynamic-tool filtering removes OpenClaw `exec`/`process`. I did not run a live cron job in this read-only review.

## Solution Assessment

Unclear without maintainer policy input. The safest minimum is fail-fast validation when Codex cannot honor the allowlist; making `exec` work needs a narrow native or sandbox-backed design that does not widen permissions.
