## Summary

Keep open: current main only partially addresses the report. Generic `openai-responses` payloads already use array `input` and `stream: true`, but stable opt-in controls for fallback top-level `instructions` and omitting `max_output_tokens` on stricter proxy endpoints are still absent.

## What This Changes

Request stable configurable normalization for `api=openai-responses` proxy endpoints so stricter Responses-compatible relays can receive compatible request payloads without local patches.

## Best Possible Solution

Add narrowly scoped opt-in model-compat and payload-policy controls for proxy Responses payload shaping, with schema, catalog normalization, tests, and docs while preserving native defaults.

## Reproduction Assessment

No, not with a high-confidence live reproduction against a strict third-party relay. Source inspection does confirm the remaining current-main gap: generic `openai-responses` cannot opt into fallback top-level `instructions` and cannot omit `max_output_tokens`.

## Solution Assessment

Yes for the requested direction, but only after maintainer API approval. Model compat plus payload policy is the maintainable place for proxy-specific normalization, while defaults should stay unchanged for native OpenAI, Codex, and Azure routes.
