## Summary

Current main already contains the Slack HTTP webhook routing fix: Request URL routes dispatch through the same shared Slack handler registry that monitor startup registers into, and the fix was shipped in v2026.4.22.

## Close Comment

Closing as implemented on main after a Codex review. Current Slack HTTP routing now dispatches Request URL webhooks through the same shared registry used by monitor startup (`extensions/slack/src/http/registry.ts:18`, `extensions/slack/src/http/plugin-routes.ts:7`), the bundled Slack entry registers that route through the narrow `http-routes-api` split (`extensions/slack/index.ts:7`, `extensions/slack/http-routes-api.ts:1`), and there is regression coverage for the shared-registry dispatch path (`extensions/slack/src/http/plugin-routes.test.ts:65`). The changelog shows this shipped in `v2026.4.22` with the note that Slack HTTP events no longer 404 after successful route registration (`CHANGELOG.md:221`, release commit `00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). If you still hit 404s on a build at or newer than `v2026.4.22`, please open a new issue with the exact version, config, and gateway logs.
