## Label Justifications

- `P1`: The report describes a real provider rejection loop that can break long-running Anthropic agent sessions for users until automatic recovery is available.
- `impact:session-state`: The affected data is persisted session history containing stale signed thinking blocks that can poison future turns.

## Summary

Current main already resolves the reported Anthropic invalid thinking-signature loop by detecting pre-content terminal stream error events, retrying once with all thinking blocks stripped from the outbound replay copy, and avoiding duplicate output after content has begun streaming.

## What This Changes

The issue requests Anthropic transcript hygiene that prevents stale or invalid signed thinking blocks from being replayed and causing request rejection loops.

## Best Possible Solution

Keep the current main recovery path: preserve opaque Anthropic thinking signatures by default, then automatically recover from provider-reported invalid signatures by retrying the request without thinking blocks.

## Reproduction Assessment

Yes, at source level: the v2026.5.28 stream wrapper did not treat terminal assistant error events as recoverable before yielding them, while current main has a focused invalid-signature stream-error retry test. I did not run a live Anthropic session.

## Solution Assessment

Yes. Because Anthropic signatures are opaque and Sonnet 4.6 keeps prior thinking blocks by default, the maintainable fix is to preserve valid replay normally and recover by stripping thinking blocks only after the provider reports a signature failure.
