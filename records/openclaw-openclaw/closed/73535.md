## Summary

Keep this issue open. Current main still has the material split reported here: the default local status path relies on `probeGateway`, while `--require-rpc` first proves the Gateway `status` RPC separately. Current reachability classification only treats successful probes or `missing scope: operator.read` detail failures as reachable, so a timeout/unknown-capability default probe can still report failure even when a separate `status` RPC would succeed.

## Best Possible Solution

Keep the issue open. Reproduce the default local probe versus `--require-rpc` split on a local token-authenticated loopback setup, then add a focused regression test where `probeGateway` returns timeout/unknown while direct `status` RPC succeeds. The likely fix is either to separate handshake reachability from detail-probe uncertainty in status output or add a bounded local status-RPC fallback for default status paths. If maintainers choose #48360, #64400, or #73524 as the canonical thread, move this issue's regression-window and control evidence there before closing as duplicate/superseded.
