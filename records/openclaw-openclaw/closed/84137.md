## What This Changes

The issue asks whether Codex app-server post-tool raw assistant completions should keep fail-fast defaults or get a longer default post-tool synthesis budget.

## Best Possible Solution

Keep the compatible fail-fast default unless maintainers approve a semantic change backed by real Codex app-server workload evidence, then update the post-tool default with regression tests for pre-tool release, post-tool budget, and stuck-turn latency.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a semantic/defaults decision. Source and tests do give high-confidence proof of the current behavior: post-tool raw assistant progress uses the short assistant idle budget, while pre-tool raw assistant output can release final text.

## Solution Assessment

Unclear until maintainers choose the desired default. The related config-knob PR is the conservative short-term solution, while a default semantic change needs workload proof and explicit agreement on the stuck-turn latency tradeoff.
