## Summary

Keep open. Current main still gates QQBot debug logging with raw string truthiness, and the linked open fix PR at https://github.com/openclaw/openclaw/pull/82656 still needs review follow-through before this issue can close.

## What This Changes

QQBot debug logging should require explicit truthy `QQBOT_DEBUG` values instead of treating any non-empty string as enabled.

## Best Possible Solution

Tighten the QQBot-local debug parser to explicit truthy values, add regression coverage for false-like and unknown strings, then land the linked fix PR and close this issue.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: set `QQBOT_DEBUG=0` and the `!!process.env.QQBOT_DEBUG` guard passes before the exported debug loggers write to `console.*`.

## Solution Assessment

Yes for the issue direction, but the current linked PR is only a partial solution. The maintainable fix is an explicit truthy allowlist or equivalent local helper plus focused logger tests.
