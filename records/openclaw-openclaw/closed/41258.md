## Summary

Current `main` already guards `openclaw backup create` against the reported empty-header/leftover-temp-file failure mode by writing to a unique temp archive, publishing atomically only after `tar.c(...)` completes, and cleaning temp output in `finally`; regression tests cover the failure path.

## Evidence

- **Atomic temp-archive implementation:** `createBackupArchive` builds a unique `*.tmp` target, writes the tarball there, publishes it to the requested output only after success, and removes the temp file afterward.
  - file: [src/infra/backup-create.ts:134](https://github.com/openclaw/openclaw/blob/e94c0bf51542fe518e49ad4e2ee29882891b5603/src/infra/backup-create.ts#L134)

  - sha: [e94c0bf51542](https://github.com/openclaw/openclaw/commit/e94c0bf51542fe518e49ad4e2ee29882891b5603)
- **Create flow waits for tar completion before publish:** The archive write path awaits `tar.c(...)` against `tempArchivePath`, then calls `publishTempArchive`, with cleanup in `finally` so failed writes do not leave the temp artifact behind.
