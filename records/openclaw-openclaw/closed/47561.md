## Summary

Keep open: current main still derives the active DM session key directly from the current routing policy, so changing `session.dmScope` can move the same Telegram DM from the fresh isolated key back to `agent:main:main` without a rebind or warning.

## What This Changes

Ensure `/new` remains a durable fresh start across channel DM routing-mode changes, or explicitly migrates/warns when a routing change supersedes the fresh session.

## Best Possible Solution

Add a generic session-routing guard or migration path that detects when a reset-created DM session is superseded by a later routing-mode change, then preserves the fresh lane or surfaces an explicit diagnostic before older state is reused.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: create/reset a Telegram DM under `dmScope: "per-channel-peer"`, then remove that setting so the same peer resolves to `agent:main:main`. I did not run a live gateway/channel test, so this is not marked runtime-reproduced.

## Solution Assessment

Unclear. The current implementation is not sufficient because it silently follows the new route key, but choosing between rebind, migration, invalidation, or warning is a maintainer policy decision.
