## Summary

Keep open. Current main has partial mitigation for #56939, but it does not fully implement the requested behavior. The Ollama plugin can now be disabled for discovery and unreachable ambient discovery is quiet, but the default non-test path still calls `buildOllamaProvider` with no explicit Ollama config or real Ollama key, which probes the default `127.0.0.1:11434` endpoint. The docs also still frame implicit discovery as tied to `OLLAMA_API_KEY` or an auth profile, while tests assert no-key ambient local discovery when Ollama is reachable.

## Best Possible Solution

Keep the issue open and decide the intended product contract for Ollama discovery. The conservative fix would make local Ollama probing require an explicit opt-in signal, such as `OLLAMA_API_KEY`, an auth profile, `models.providers.ollama`, or `plugins.entries.ollama.config.discovery.enabled: true`; alternatively, document the no-key ambient probe clearly and add negative-result throttling so unavailable local Ollama cannot create repeated connection noise during gateway startup/model catalog refreshes.
