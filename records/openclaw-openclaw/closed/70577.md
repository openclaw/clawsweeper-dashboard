## Summary

#70577 should close as a duplicate of #61098. The remaining symptom is not the local node-llama-cpp install problem or the #71678 remote-indexing diagnostics work; it is the same live memory_search stale provider/tool-host state already tracked by the open maintainer-labeled #61098.

## Best Possible Solution

Close #70577 as a duplicate of the open canonical stale-provider issue #61098. The fix should stay focused there: make live memory_search re-resolve current agents.defaults.memorySearch config after config changes/restart boundaries, retire stale manager/provider state, and add a regression test that distinguishes the live tool path from CLI memory status/search.

## Evidence

- **Canonical tracker:** Provided GitHub context shows #61098 is open, maintainer-labeled, and tracks the same remaining behavior: live memory_search keeps using a stale embedding backend after agents.defaults.memorySearch.provider changes and restart/config-patch attempts. #70577 explicitly describes the same stale provider/manager state, only with the old Bedrock credentials error instead of an Ollama error.
