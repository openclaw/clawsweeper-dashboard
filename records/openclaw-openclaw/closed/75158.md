## Summary

Current main resolves the reported Active Memory timeout regression: typed hook registrations now carry an optional per-handler timeout, the hook runner prefers that handler timeout over the generic 15s `before_prompt_build` default, and Active Memory registers its prompt-build hook with `config.timeoutMs + setupGraceTimeoutMs`. The fix is covered by focused tests and appears main-only rather than proven shipped in `v2026.4.27`.

## What This Changes

The issue asks for `before_prompt_build` hook execution to respect Active Memory's configured `timeoutMs` instead of being capped by the generic 15s hook timeout.

## Best Possible Solution

Keep the current per-registration hook timeout design and ship it in the next release; it solves the Active Memory regression without adding a broader gateway hook-timeout configuration surface.

## Reproduction Assessment

Yes. The release-tag source for `v2026.4.27` reproduces the problem structurally because Active Memory registered no hook timeout and the hook runner only used the generic 15s modifying-hook budget; current main's source and tests verify that path no longer applies.

## Solution Assessment

Yes. Making Active Memory register its own bounded hook budget is the narrowest maintainable fix for this plugin-specific timeout mismatch; a top-level hook-timeout config can remain a separate product decision if other plugins need it.
