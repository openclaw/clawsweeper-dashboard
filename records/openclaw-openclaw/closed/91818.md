## Summary

Current main already enforces the requested mixed text/media invariant: when block streaming has already sent the visible text, final payload planning preserves any unsent media while clearing the duplicate final text, and the latest release source contains the same behavior.

## What This Changes

The issue reports duplicated user-visible final reply text when a visible-text plus MEDIA attachment reply is split across block/final channel delivery paths.

## Best Possible Solution

Keep the current final-payload planning behavior: text already sent by block streaming should not be repeated as the final media caption, while unsent media still delivers.

## Reproduction Assessment

No current-main failing reproduction is needed to keep this open: source and tests now cover the mixed block-stream text plus final media case and prevent the duplicated text path.

## Solution Assessment

Yes, the current solution is the narrow maintainable fix because final payload planning owns the block/final dedupe decision and preserves unsent media without pushing channel-specific policy into Bncr or other adapters.
