## Summary

Current main already replaced the old one-shot completion announce path with queue/direct/fallback dispatch plus deferred completion-delivery retries, so the reported "4 attempts over ~40s then permanently dropped" behavior no longer matches shipped code.

## Evidence

- **Transient WhatsApp listener errors are retried in the direct announce path:** The direct completion announce flow classifies `no active .* listener` as transient and retries with 5s/10s/20s delays instead of failing immediately.
  - file: [src/agents/subagent-announce-delivery.ts:126](https://github.com/openclaw/openclaw/blob/c1383680405879a3c3f43dceb20a694f54d3c13c/src/agents/subagent-announce-delivery.ts#L126)

  - file: [src/agents/subagent-announce-dispatch.ts:88](https://github.com/openclaw/openclaw/blob/c1383680405879a3c3f43dceb20a694f54d3c13c/src/agents/subagent-announce-dispatch.ts#L88)
