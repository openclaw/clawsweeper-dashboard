## Summary

Keep this PR open: it targets a real gateway reload bug, but the patch currently classifies `auth.cooldowns` as a no-op, which skips the hot-reload path for watcher-driven config changes, and the external PR still lacks real behavior proof.

## What This Changes

Adds an explicit `auth.cooldowns` no-op reload rule and two reload-plan unit cases for the top-level and subpath cooldown keys.

## Best Possible Solution

Classify `auth.cooldowns` as a hot reload-safe path, prove the reloader activates runtime config without restarting, and add redacted real gateway proof before merge.

## Reproduction Assessment

Yes, source-reproducible: current main lacks an `auth.cooldowns` reload rule, so `matchRule` falls through and `buildGatewayReloadPlan` treats the path as restart-required. No live run was performed because this is a read-only review.

## Solution Assessment

No: avoiding a restart is the right direction, but `kind: "none"` is not the best implementation because it bypasses the hot-reload activation path. A hot reload-safe rule with focused reloader coverage is the narrower maintainable fix.
