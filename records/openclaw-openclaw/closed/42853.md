## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Stop persisting ambient NODE_OPTIONS:** `src/daemon/service-env.ts:340`
  - body: `resolveNodeOptions` preserves or appends to the invoking shell's `NODE_OPTIONS`, and this line writes that value into generated service metadata. That bypasses the existing host-env blocklist: an install run with `NODE_OPTIONS="--require /tmp/hook.js --max-old-space-size=2048"` would freeze the `--require` startup hook into the managed daemon. Generate only the heap flag from typed config/default, or strictly tokenize and preserve only heap-size aliases.
  - confidence: 0.94
- **[P2] Wire the loaded config into node install:** `src/commands/node-daemon-install-helpers.ts:62`
  - body: This helper now reads `params.config?.gateway?.nodeMaxOldSpaceMb`, but `runNodeDaemonInstall` loads the config and still calls `buildNodeInstallPlan` without passing it. As a result, `openclaw node install` will always fall back to auto-detect instead of honoring the user's explicit heap cap for the independently installed node service.
  - confidence: 0.89
- **[P2] Detect underscore heap aliases before appending:** `src/daemon/service-env.ts:269`
  - body: The existing check only looks for `--max-old-space-size`, but Node accepts `--max_old_space_size` too. If a user already pinned `NODE_OPTIONS=--max_old_space_size=2048`, this appends a second heap flag and the later value can silently change their memory cap.
  - confidence: 0.87
