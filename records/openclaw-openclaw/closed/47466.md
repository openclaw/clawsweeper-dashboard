## Summary

Current main no longer treats Chrome stderr output as a fatal launch signal. The managed browser path waits for CDP HTTP/WS readiness and only includes stderr as diagnostic text if CDP never becomes reachable, which matches the requested fix for Docker/dbus noise.

## Evidence

- **Launch success is gated by CDP reachability, not stderr content:** `launchOpenClawChrome()` spawns Chrome, buffers stderr for later diagnostics, then polls `isChromeReachable(profile.cdpUrl)`. It throws only if CDP stays unreachable; stderr is appended as a hint after that failure path.
  - file: [extensions/browser/src/browser/chrome.ts:356](https://github.com/openclaw/openclaw/blob/0e7250f37bf347c72790abf2a6692d5ec84e5ea1/extensions/browser/src/browser/chrome.ts#L356)
  - command: `nl -ba extensions/browser/src/browser/chrome.ts | sed -n '356,401p'`
  - sha: [0e7250f37bf3](https://github.com/openclaw/openclaw/commit/0e7250f37bf347c72790abf2a6692d5ec84e5ea1)
- **Post-launch availability waits for CDP readiness:** `ensureBrowserAvailable()` calls `waitForCdpReadyAfterLaunch()`, which repeatedly checks `isReachable()` and errors with `not reachable after start` only when CDP readiness never arrives.
  - file: [extensions/browser/src/browser/server-context.availability.ts:166](https://github.com/openclaw/openclaw/blob/0e7250f37bf347c72790abf2a6692d5ec84e5ea1/extensions/browser/src/browser/server-context.availability.ts#L166)
