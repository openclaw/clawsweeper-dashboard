## Summary

Close as implemented on current main: the reported Docker restart hang matches the stale bundled runtime-deps lock/PID-reuse path, and main now records process start time in lock owners, expires recycled-PID locks, and has targeted Docker PID-reuse regression coverage. The proof commit is on main only; I did not find a containing release tag yet.

## What This Changes

Report that OpenClaw 2026.4.26 can hang at `[gateway] starting...` after restarting an existing Docker container, with Node threads blocked in `futex_wait_queue` and no health listener.

## Best Possible Solution

Keep the current runtime-deps lock incarnation check and startup diagnostics on main, and have affected operators retest with the next build containing 29d3b65a83913f326fdc5a14c043804bcf9b8a1e or newer; a new report should include startup trace output if a post-fix image still hangs.

## Reproduction Assessment

Yes, for the matched root cause. The targeted test in `src/plugins/bundled-runtime-deps.test.ts` models Docker PID reuse with a stale `.openclaw-runtime-deps.lock`, which maps closely to an existing-container restart stuck in the lock wait path.

## Solution Assessment

Yes. Fixing the runtime-deps lock ownership semantics is narrower and more maintainable than disabling plugin loading or adding a broad startup watchdog; the existing startup trace remains available for future diagnostics.
