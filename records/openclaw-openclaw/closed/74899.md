## Summary

Current main already fixes the reported browser-control AJV runtime-deps failure. The browser plugin still depends on the MCP SDK, current package metadata mirrors both `@modelcontextprotocol/sdk` and `ajv`, and the runtime-deps materialization now rejects an installed dependency whose declared entry file is missing, with a regression test for the incomplete `ajv/dist/ajv.js` case.

## What This Changes

The issue asks for the bundled browser plugin runtime-deps mirror to include and repair `ajv`/`ajv-formats` needed by `@modelcontextprotocol/sdk` after upgrading to OpenClaw 2026.4.27.

## Best Possible Solution

Keep the current main runtime-deps materialization behavior and ship it in the next OpenClaw package so packaged browser-control startup repairs incomplete MCP/AJV mirrors without browser-specific special cases.

## Reproduction Assessment

Yes. The report gives a concrete packaged v2026.4.27 Linux/npm-global path and exact `ERR_MODULE_NOT_FOUND` for `ajv`; current-main regression coverage models the same incomplete staged `ajv` entry-file condition.

## Solution Assessment

Yes. The current fix is at the right packaging boundary: runtime-deps reuse now validates declared package entry files before deciding a mirror is materialized, so any affected bundled plugin benefits without hard-coding browser-control behavior.
