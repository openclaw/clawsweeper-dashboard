## Summary

Keep open: current main still leaves native Ollama `options.num_ctx` unset unless `params.num_ctx` is explicit, still keeps the default non-cron LLM idle watchdog at 120 seconds, and the issue is paired with open implementation PRs rather than a merged fix.

## What This Changes

The issue reports a regression where local Ollama/qwen model runs after v2026.4.26/v2026.4.29 lose useful tool-following behavior and produce poor answers, with timeout and context handling suspected.

## Best Possible Solution

Review and land or close #76181 as the likely canonical fix, with #76122 treated as a narrower timeout-only alternative rather than the complete solution.

## Reproduction Assessment

Source-reproducible, but not a high-confidence live reproduction. Current main tests and code show the 120-second fallback and native Ollama no-`num_ctx` behavior; I did not verify the full Ollama/qwen3.6 gateway webchat failure live.

## Solution Assessment

Unclear for the issue as filed because there is no merged fix yet. A provider-aware Ollama context/watchdog repair like #76181 looks narrower and more complete than a global timeout bump, but it needs maintainer review for memory and timeout-policy tradeoffs.
