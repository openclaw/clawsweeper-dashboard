## Summary

Keep open: current main and the latest release still use a 30-minute Codex app-server terminal-idle default that is not derived from the effective run timeout, and terminal-idle timeouts still do not surface terminal-specific guidance. The open fix path at https://github.com/openclaw/openclaw/pull/85296 covers the early-abort half only and is currently conflicting, so this issue is still necessary.

## What This Changes

Make the Codex app-server terminal-idle watchdog follow the effective run timeout, or otherwise be configurable, and surface accurate terminal-idle timeout guidance.

## Best Possible Solution

Refresh or replace the existing focused PR with a narrow Codex app-server fix that derives the terminal-idle watchdog from the effective run timeout and emits terminal-idle-specific guidance when that watchdog fires.

## Reproduction Assessment

Yes, at source level: current main resolves the terminal-idle watchdog from the explicit option only and not params.timeoutMs, and terminal watch timeouts skip the Codex prompt-timeout outcome so generic agents.defaults guidance can still surface. I did not live-wait a 30-minute scheduled turn.

## Solution Assessment

Yes, the best fix is the narrow runtime-plus-diagnostics path: derive the existing terminal watchdog from the effective run budget and report terminal-idle-specific guidance, without adding a new public config knob by default.
