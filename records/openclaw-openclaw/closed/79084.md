## Summary

Current main now handles the reported empty/silent delivery path: normalized NO_REPLY payloads become an empty delivery plan, delivery is marked suppressed before any channel send, and the gateway run envelope remains an RPC response rather than Telegram message text.

## What This Changes

The issue reports that an exact NO_REPLY assistant result in Telegram was delivered as raw gateway run-envelope JSON instead of being suppressed.

## Best Possible Solution

Keep the current-main suppression behavior and ship it in the next release; successful empty or silent runs should remain terminal delivery success without channel text.

## Reproduction Assessment

No. I did not reproduce the old 2026.5.6 Telegram leak on current main; source and focused tests show exact NO_REPLY/empty payloads now suppress before durable channel delivery.

## Solution Assessment

Yes. The current-main fix is at the maintainable boundary: normalize outbound payloads, treat an empty delivery plan as suppressed success, and keep the gateway envelope on the RPC side only.
