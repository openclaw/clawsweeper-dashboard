## Summary

Keep this open: current main still has incompatible TS and Swift schemas for the same shared `identity/device.json` path, and the issue's reported token/pairing churn follows directly from those code paths.

## What This Changes

The issue reports that the macOS app and TypeScript CLI/gateway can use incompatible schemas for the same on-disk device identity file when `OPENCLAW_STATE_DIR` is shared.

## Best Possible Solution

Define and test one device identity persistence contract across TS and Swift so restarts preserve the same `deviceId` and cached device tokens without silently corrupting existing identities.

## Reproduction Assessment

Yes, by source inspection: put a TS PEM-format `identity/device.json` under a state dir and launch the Swift store with `OPENCLAW_STATE_DIR` pointing there; the Swift decode path cannot accept the PEM fields and regenerates a new identity. The reverse direction is also clear because the TS loader requires `version`, `publicKeyPem`, and `privateKeyPem`.

## Solution Assessment

Unclear as an implementation choice. The report's fix directions are valid, but the best maintainable path needs a deliberate auth/storage decision: compatible cross-runtime schema support or a non-colliding app identity path, with no silent regeneration for recognized alternate schemas.
