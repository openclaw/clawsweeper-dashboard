## Summary

Keep open: current main still has the cron state-sidecar persistence gap. `loadCronStore` repairs missing or invalid `updatedAtMs` in memory, but a clean Gateway restart with an already-future `nextRunAtMs` can avoid every startup persist path, leaving `jobs-state.json` unrepaired.

## What This Changes

The issue asks for cron restart/load repair to persist `updatedAtMs` into `jobs-state.json` instead of requiring the external health check to compensate repeatedly.

## Best Possible Solution

Persist corrected cron runtime-state sidecar data when load/startup repairs `updatedAtMs`, while keeping runtime fields out of `jobs.json` and avoiding a new config knob.

## Reproduction Assessment

Yes, source-level: use a future scheduled cron job whose `jobs-state.json` entry has a valid `nextRunAtMs` but missing or invalid `updatedAtMs`; startup repairs the timestamp in memory but has no persist path unless another scheduler change occurs.

## Solution Assessment

Yes. The narrow maintainable fix is to mark load-time sidecar repairs dirty or force a sidecar-only persist after startup repair, without rewriting runtime fields back into `jobs.json`.
