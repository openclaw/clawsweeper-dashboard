## Summary

Keep open: current main intentionally isolates `CODEX_HOME` and `HOME` for native Codex app-server launches and already documents that behavior, so this is not a strict bug; the remaining operator-home affordance is a real harness UX/security decision. The reporter’s #73910 link is useful context, but that open issue tracks ACP auth/timeout behavior, while this issue is about shell `~` semantics inside the native Codex harness.

## What This Changes

The issue asks OpenClaw to preserve operator-home `~` semantics in Codex harness sessions or, if isolation stays, document/warn and expose an explicit operator-home path.

## Best Possible Solution

Keep Codex `HOME` isolation as the default, then decide whether to add a carefully named operator-home affordance and stronger runbook guidance without redefining shell `~`.

## Reproduction Assessment

Yes, source inspection and existing tests give a high-confidence reproduction path: local Codex app-server start options set `HOME` to `<agentDir>/codex-home/home`. I did not run a live macOS Codex harness session in this read-only review.

## Solution Assessment

Unclear. Preserving the operator’s real `HOME` would conflict with the documented isolation goal, so the safer direction is a maintainer-approved explicit operator-home affordance or docs warning rather than changing `~` globally.
