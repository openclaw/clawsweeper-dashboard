Keep this PR open: the DST freeze is reproducible and the monotonic-boundary direction is useful, but the current patch is not merge-ready because it bypasses current main's precise token-bucket selected-hour filtering and lacks the required user-facing changelog entry.

## What This Changes

The PR changes Control UI Usage hour slicing and selected-hour filtering to use monotonic next-hour stepping across DST transitions, and adds node-side DST regression coverage.

## Best Possible Solution

Keep the monotonic hour-boundary algorithm, but wire it into `forEachSessionHourSlice` and the `sessionSpanTouchesSelectedHours` fallback while preserving `sessionTouchesSelectedHours` as the selected-hour entry point, then add focused DST tests and a changelog entry.

## Solution Assessment

No. Monotonic next-hour stepping is the right underlying fix, but this PR should preserve `sessionTouchesSelectedHours` token-bucket semantics instead of replacing selected-hour filtering with a span-only helper.
