## Summary

Keep open. Current `main` still does not show a Telnyx inbound-answer path in the voice-call plugin, and the CLI path still bootstraps its own runtime/listener instead of reusing an already-running gateway-owned listener.

## Evidence

- **Telnyx inbound events are parsed, but no answer command exists:** `TelnyxProvider.normalizeEvent()` maps `call.initiated`, `call.ringing`, `call.answered`, and related events, but the provider methods only implement outbound create, hangup, speak, and transcription start/stop. There is no Telnyx `/actions/answer` call or equivalent inbound-answer method in the provider.
  - file: [extensions/voice-call/src/providers/telnyx.ts:167](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/extensions/voice-call/src/providers/telnyx.ts#L167)
  - command: `nl -ba extensions/voice-call/src/providers/telnyx.ts | sed -n '140,330p'`
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)
- **Core provider contract has no inbound-answer primitive:** The shared `VoiceCallProvider` interface exposes webhook verification/parsing, outbound initiation, hangup, TTS, DTMF, listening, and status lookup, but no `answerCall` or similar hook. That leaves no generic path for current main to explicitly answer an inbound Telnyx call.
  - file: [extensions/voice-call/src/providers/base.ts:29](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/extensions/voice-call/src/providers/base.ts#L29)
  - command: `nl -ba extensions/voice-call/src/providers/base.ts | sed -n '1,160p'`
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)
- **CLI still creates its own runtime and bound webhook server:** `registerVoiceCallCli()` calls `ensureRuntime()` for `voicecall call`, `ensureRuntime()` constructs a fresh runtime with `createVoiceCallRuntime()`, and runtime startup immediately calls `webhookServer.start()`. The webhook server's EADDRINUSE guard only applies when the same server instance is already listening, not when another gateway process already owns the port.
  - file: [extensions/voice-call/src/cli.ts:127](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/extensions/voice-call/src/cli.ts#L127)
  - command: `nl -ba extensions/voice-call/src/cli.ts | sed -n '110,190p'; nl -ba extensions/voice-call/index.ts | sed -n '160,220p'; nl -ba extensions/voice-call/src/runtime.ts | sed -n '260,310p'; nl -ba extensions/voice-call/src/webhook.ts | sed -n '428,470p'`
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)
- **Docs still describe the plugin as gateway-resident:** The docs say the Voice Call plugin runs inside the Gateway process, which makes the current CLI behavior especially relevant to the reported `EADDRINUSE` symptom when a gateway listener is already active.
  - file: [docs/plugins/voice-call.md:30](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/docs/plugins/voice-call.md#L30)
