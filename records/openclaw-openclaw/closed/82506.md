## Summary

Keep open: the PR is directionally aimed at the linked exec approval delivery bug, but it introduces a blocking iOS approval-route regression and an unrelated cron payload validation regression. The supplied proof exercises an injected delivery failure rather than a real external approval route, so the contributor still needs real behavior proof before merge.

## What This Changes

The PR changes approval forwarding to report delivered/attempted delivery results, awaits failed target dispatches, adds approval regression tests, and includes additional channel-add and cron store changes.

## Best Possible Solution

Distinguish no-route from attempted delivery failure across forwarder and iOS delivery, keep unrelated cron validation out of this PR, and add focused regression coverage for turn-source fallback plus failed configured delivery.

## Reproduction Assessment

Yes, source-level. An exec approval with an enabled turn-source route and iOS delivery returning false for no visible targets would now skip the turn-source check because the PR marks that false result as attempted.

## Solution Assessment

No. Awaiting configured forwarder delivery is a reasonable direction, but the current patch misclassifies iOS no-route results, relaxes unrelated cron validation, and only partially covers the linked issue's follow-up failure scope.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve chat fallback when iOS has no route:** `src/gateway/server-methods/exec-approval.ts:401`
  - body: `handleRequested` returns `false` both when iOS has no visible approval target and when pushes fail. Wrapping every iOS result as `attempted: true` means a request from a chat with an enabled turn-source route will now skip `hasApprovalTurnSourceRoute` and expire with `no-approval-route` instead of falling back to the originating chat. Please distinguish no-target/no-visible-device from an actual failed push attempt before suppressing fallback.
  - confidence: 0.91
- **[P2] Keep rejecting blank cron agent-turn payloads:** `src/cron/persisted-shape.ts:67`
  - body: The cron service requires `payload.kind="agentTurn"` to include a message, and downstream normalization trims message text before use. Dropping the blank-string check lets whitespace-only persisted jobs load instead of being skipped and repaired, so an invalid cron job can run with an empty prompt. Please keep the trim check or split this unrelated behavior into a separately justified fix.
  - confidence: 0.84
