## Summary

Keep open: current main still has the reported fast-status placeholder behavior, but this PR is not merge-ready because it only enables the live `channels.status` call and does not update the missing configured-channel fallback path that emits the linked issue’s placeholder rows. ## What This Changes

The PR changes default text `openclaw status` to always request live `channels.status` after a reachable gateway probe and updates the focused status scan test.

## Best Possible Solution

Fix the missing configured-channel fallback so live `channelAccounts` can replace fast-mode placeholders, then merge only after maintainers accept the default RPC tradeoff and redacted real output proves the changed status behavior.

## Reproduction Assessment

Yes, source-reproducible: current main skips live `channels.status` for default text status, and the table builder still has tests and code that emit the reported fast-mode placeholder when setup fallback plugins are skipped. I did not run a live configured gateway in this read-only review.

## Solution Assessment

No: this PR is a plausible entry-point change but not the best complete fix, because it passes live data into a builder path that still emits placeholders for missing configured channel ids. The safer fix is to consume live `channelAccounts` before that fallback or choose a metadata-only wording change if maintainers reject the extra RPC.
