## Summary

Keep open: the PR targets a useful cleanup, but it is not merge-ready because it hard-codes provider-specific quarantine detection into core and leaves the task-list path unable to identify ordinary cron tasks whose provider metadata only exists in run logs.

## What This Changes

The PR adds default filtering for win-ollama-marked session, cron run, and task rows, plus includeQuarantined/--include-quarantined opt-ins, excludedQuarantinedCount fields, tests, changelog, and regenerated Swift protocol models.

## Best Possible Solution

Model quarantine as explicit generic session/run/task metadata exposed through additive includeQuarantined fields, let the win-ollama owner set that metadata outside core, and update the CLI/Gateway docs and tests around the new default filtering.

## Reproduction Assessment

Yes, from source inspection: current main has no quarantine filtering in the sessions, cron, or task list paths, and the PR proof exercises the added sessions/cron run helpers. The task-list behavior remains source-reproducibly incomplete because cron TaskRecord creation does not carry provider/model fields.

## Solution Assessment

No. The safer solution is a generic explicit quarantine metadata contract and owner-side win-ollama marking, with task filtering based on that metadata rather than substring matching provider labels in core.
