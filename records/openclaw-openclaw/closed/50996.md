## Summary

Close as superseded: current main and release v2026.4.29 fix the concrete `process.poll` cancellation freeze with a 30-second cap and abort-aware wait, while the broader framework-level tool interruption and steer semantics remain tracked by the open design issue #65223.

## What This Changes

Request to make long-running tool calls interruptible or otherwise responsive to inbound user messages, especially `process.poll` waits in Telegram/macOS gateway sessions.

## Best Possible Solution

Close this issue after linking #65223 as the canonical tracker for remaining all-tool interruption design, and keep the shipped `process.poll` cap plus abort-aware wait as the focused fix for the reported freeze.

## Reproduction Assessment

Yes. The high-confidence reproduction is to start a background process, call `process.poll(timeout=120000)`, then abort the run while the poll loop is sleeping; current main has focused unit coverage for the long-timeout clamp and abort path.

## Solution Assessment

Yes for this cleanup. The narrow `process.poll` fix is the maintainable owner-local repair, and the broader framework-level tool interruption/steer semantics belong in #65223 instead of being split across this older report.
