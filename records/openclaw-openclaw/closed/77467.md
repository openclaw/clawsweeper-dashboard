## Summary

Keep open: current main and the latest shipped release still lack a MiniMax-owned OAuth refresh hook, while the related implementation at https://github.com/openclaw/openclaw/pull/82056 is open and not merged. The issue remains valid, but it should close only after that PR or a replacement lands with refresh-path proof.

## What This Changes

The issue requests automatic refresh for expired `minimax-portal` OAuth credentials instead of falling through to `No credentials found`.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/82056, or a replacement, with a provider-owned MiniMax `refreshOAuth` implementation that preserves CN/Global region metadata, rotated refresh tokens, and refresh failure diagnostics.

## Reproduction Assessment

Yes, source-level: an expired `minimax-portal` OAuth credential reaches the provider refresh seam, finds no MiniMax `refreshOAuth`, misses the shared pi-ai fallback, and resolves as a missing profile. I did not run a live expired-token refresh with real MiniMax credentials.

## Solution Assessment

Yes: the right fix is a MiniMax provider-owned `refreshOAuth` hook through the existing plugin contract, not a core special case. The active PR is the current implementation path, but it still needs review and refresh proof before this issue can close.
