## Summary

Keep this PR open. Current main at 5828dcdb05aad587bff831467e7fa51f73312ff8 still does not expose or forward `channels.slack.logLevel`, and this open PR is the active implementation candidate for the closed operator request in #71531. The latest PR patch is focused and fixes the earlier lowercase/type-contract issues, but dependency source still makes the App-only wiring a functional question because OpenClaw constructs and passes its own `SocketModeReceiver`.

## Best Possible Solution

Keep this PR open as the active implementation path for the Slack log-noise request. Adjust or prove the runtime wiring so lowercase `channels.slack.logLevel` reaches the explicitly constructed `SocketModeReceiver` or its logger path while preserving current defaults when unset; keep the schema and exported `SlackConfig` type aligned; add tests proving the configured level reaches the receiver/socket warning path; and update Slack/config docs or generated metadata if this becomes a supported operator knob. Keep #72808 separate because it tracks stale/silent Slack disconnect behavior, not log-noise suppression.

## Risks / Open Questions

- Functional risk: forwarding `logLevel` only to Bolt `App` may not silence the reported socket-mode pong-timeout WARN because OpenClaw supplies an already-constructed `SocketModeReceiver`.

- Operational risk: `logLevel: "error"` intentionally suppresses all Slack/Bolt WARN output, not just the benign pong-timeout warning, so the supported behavior should be documented clearly.
