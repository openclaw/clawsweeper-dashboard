## Label Justifications

- `P1`: Cleartext provider credential exposure in gateway logs can affect real operators and deserves urgent security handling without making the core runtime unusable.
- `impact:security`: The report is about sensitive credentials being persisted to local logs after a denied tool call.
- `impact:auth-provider`: The exposed values are provider API keys and access tokens used for model/tool authentication.

## Summary

Keep open: current main still routes denied gateway exec failures through raw_params failure logging, and the default redaction pattern can miss JSON-escaped quoted env assignments for non-`sk-` token values, so the credential-log exposure remains actionable.

## What This Changes

The issue reports that denied `tools/exec` calls can persist cleartext inline environment credentials through `raw_params.command` in gateway error logs.

## Best Possible Solution

Prevent denied exec command strings from reaching persistent logs in cleartext, then add regression coverage for JSON-escaped quoted env assignments and the reported provider token prefixes.

## Reproduction Assessment

Yes, source-level: allowlist-missed gateway exec throws into the generic tool failure logger, which logs a JSON-stringified raw_params preview. The current env-assignment redaction pattern does not cover JSON-escaped quoted values for several non-`sk-` token shapes.

## Solution Assessment

Unclear which exact mitigation maintainers want, but the requested direction is sound: redact or suppress denied exec raw_params before persistent logging, with focused coverage for escaped shell assignments and provider token prefixes.
