## Summary

Current `main` already contains the managed-loopback CDP readiness/SSRF fixes that match this report’s symptom: Chrome is up on `127.0.0.1:18800`, but OpenClaw misclassifies its own local CDP endpoint as unavailable during managed browser startup.

## Evidence

- **Loopback CDP checks are explicitly allowed now:** `assertCdpEndpointAllowed()` and `fetchCdpChecked()` special-case loopback hosts so OpenClaw’s own local CDP endpoint is allowed through the browser SSRF guard instead of being treated like a blocked private-network target.
  - file: [extensions/browser/src/browser/cdp.helpers.ts:95](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/extensions/browser/src/browser/cdp.helpers.ts#L95)
  - command: `nl -ba extensions/browser/src/browser/cdp.helpers.ts | sed -n '95,115p;304,338p'`
- **Managed browser startup uses those loopback CDP probes:** The managed Chrome launcher still waits for CDP reachability during startup, and the availability layer uses the loopback-aware readiness checks before declaring the `openclaw` profile unavailable.
  - file: [extensions/browser/src/browser/chrome.ts:367](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/extensions/browser/src/browser/chrome.ts#L367)
  - command: `nl -ba extensions/browser/src/browser/chrome.ts | sed -n '367,397p'; nl -ba extensions/browser/src/browser/server-context.availability.ts | sed -n '220,256p'`
- **Release notes document this exact fix class:** `CHANGELOG.md` for `2026.4.14` includes Browser/CDP fixes stating that loopback CDP readiness checks were kept reachable under strict SSRF defaults and that managed local Chrome should no longer be misclassified as unavailable after start.
  - file: [CHANGELOG.md:663](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/CHANGELOG.md#L663)
  - command: `nl -ba CHANGELOG.md | sed -n '663,715p'`
