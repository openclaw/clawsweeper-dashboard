## Summary

Keep open. Current main still has a source-proven path where restart recovery makes a bounded synthetic `agent` resume call, records failure on timeout, and has no transcript-backed attach/import or user-visible recovery-failure path for the next active main session; the workspace-first workaround in the discussion is useful operational context but does not close the core recovery bug.

## What This Changes

The issue asks OpenClaw to preserve or recover main-session context after Gateway restart recovery times out instead of orphaning the existing transcript.

## Best Possible Solution

Add readiness-aware restart recovery that preserves the interrupted session binding until recovery is definitive and ends with either safe bounded transcript-backed attach/import or a visible recovery-failure notice.

## Reproduction Assessment

Yes, as a high-confidence source reproduction: a running main-session store row with `abortedLastRun: true`, a resumable transcript tail, and a Gateway `agent` resume timeout currently records failure without importing or reattaching the transcript. I did not run a live Windows/Telegram restart replay.

## Solution Assessment

Unclear for the exact transcript-import mechanism. Retrying the same 10-second synthetic resume may help transient readiness, but the maintainable fix needs a recovery policy for readiness, transcript preservation, and user-visible failure handling.
