## Label Justifications

- `P1`: The default daily reset path can split an active turn's transcript and disconnect normal session history from live output.
- `impact:session-state`: The session row can rotate while the active run continues writing through the old transcript path, creating session/transcript state drift.
- `impact:data-loss`: The output can remain on disk but become effectively lost from normal session access because it is split into archived and orphaned files.
- `impact:message-loss`: Post-boundary output can be written to an orphan transcript instead of the referenced session history users expect.

## Summary

Keep open: current main still evaluates the inbound daily rollover and archives the previous transcript before active-run admission, and the open fixing PR at https://github.com/openclaw/openclaw/pull/96715 is not merged yet.

## What This Changes

The issue asks OpenClaw to prevent implicit daily session rollover from renaming a live transcript while a reply run is still writing to it.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/96715 or an equivalent pre-archive active-run guard, then close this issue after the fix merges.

## Reproduction Assessment

Yes. Current main has a source-reproducible path: initSessionState marks the session stale and passes previousEntry for archiving before runPreparedReply reaches the active-run busy gate.

## Solution Assessment

Yes. The best fix is a narrow guard before previousSessionEntry drives archivePreviousSessionTranscript, and https://github.com/openclaw/openclaw/pull/96715 uses that owner boundary while preserving normal inactive stale rollover.
