## Summary

Current main already contains the direct-session runtime-policy fix that the report asks for, and the same implementation is present in release `v2026.4.22`. The code now derives a provider-scoped direct-session policy key from `agent:main:main`, carries that key through reply/subagent execution, and preserves `bashElevated` on the embedded run path, with targeted regression tests covering the direct-session and runtime-policy propagation behavior.

## Evidence

- **Release notes match the reported failure mode:** `CHANGELOG.md` under `## 2026.4.22` explicitly says: `Channels/sandbox: derive runtime policy keys for external direct messages that share the main conversation, so sandbox/tool policy no longer treats channel-originated DMs as local main-session runs.` That is the same class of bug described in #40034.
  - file: [CHANGELOG.md:178](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/CHANGELOG.md#L178)
  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)
- **Main-session DM policy key is derived instead of reusing `agent:main:main`:** `resolveRuntimePolicySessionKey()` rewrites direct-channel conversations on the main session into a provider/account/peer-specific session key, which is the core fix needed so elevated/sandbox policy resolves against the actual Telegram DM context.
  - file: [src/auto-reply/reply/runtime-policy-session-key.ts:86](https://github.com/openclaw/openclaw/blob/bfa6708c03f24c7caa5d0a96a298c292be0d85c1/src/auto-reply/reply/runtime-policy-session-key.ts#L86)
  - sha: [bfa6708c03f2](https://github.com/openclaw/openclaw/commit/bfa6708c03f24c7caa5d0a96a298c292be0d85c1)
- **Reply/subagent execution threads the derived runtime-policy key and `bashElevated` through the embedded run:** `get-reply-run.ts` computes `runtimePolicySessionKey`; `agent-runner-utils.ts` forwards both `sandboxSessionKey` and `bashElevated`; `agent-runner-execution.ts` passes `sandboxSessionKey` into `runEmbeddedPiAgent()`. That closes the propagation gap the issue describes for freshly spawned subagents.
  - file: [src/auto-reply/reply/get-reply-run.ts:237](https://github.com/openclaw/openclaw/blob/bfa6708c03f24c7caa5d0a96a298c292be0d85c1/src/auto-reply/reply/get-reply-run.ts#L237)

- **Regression tests cover direct-session runtime-policy behavior:** `runtime-policy-session-key.test.ts` asserts that a main-session direct chat derives `agent:main:<channel>:<account>:direct:<peer>` and becomes sandboxed under non-main policy, and `agent-runner-direct-runtime-config.test.ts` asserts that the derived runtime-policy key is passed into pre-run maintenance for direct reply runs.
