## Summary

Close as implemented. The report targets OpenClaw 2026.4.26, while current main routes Control UI `/new` through the shared reset-command handler and stops before model execution, so it no longer reaches the embedded runner as an empty provider request; the fix is documented in the v2026.4.29 release notes.

## What This Changes

The issue reports that starting a fresh Control UI session with `/new` fails because the embedded agent submits an empty provider `messages` array.

## Best Possible Solution

Keep the shared reset-command short-circuit as the canonical behavior and have affected 2026.4.26 users update to v2026.4.29 or newer; any new report should include current-version logs only if a specific surface still bypasses this command path.

## Reproduction Assessment

Yes. The old failure path is clear from the report, and current main can be traced from the Control UI `/new` send through Gateway command handling to a no-fallthrough reset acknowledgement.

## Solution Assessment

Yes. Handling bare `/new` once in the shared command layer is narrower and more maintainable than provider-specific placeholder injection, while `/new <message>` still seeds a real user turn.
