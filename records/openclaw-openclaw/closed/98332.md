## Summary

Keep open: this is a focused draft feature PR, but the current diff changes the embedded queue API in a way that conflicts with the closed agent-core queue-mode contract and does not implement the documented interrupt replacement-run lifecycle.

## What This Changes

Adds an `interrupt` steering mode to embedded-agent queue message options, calls `cancel("user_abort")` before queueing, and adds unit coverage for cancel/no-cancel handles.

## Best Possible Solution

Model interrupt as a separate queue action or intent, keep agent-core queue drain modes on the existing `QueueMode` contract, and prove abort-then-start-newest behavior on real long-running embedded sessions before merge.

## Reproduction Assessment

Yes for the review findings: source inspection shows the widened `steeringMode` can flow into the existing agent-core `QueueMode` assignment, and the cancel-before-queue path targets the run being aborted. I did not run tests because this was a read-only review.

## Solution Assessment

No. The current patch is a plausible sketch, but the maintainable solution is to keep interrupt intent separate from steering queue drain mode and use the existing abort/wait/replacement-run lifecycle rather than queueing through a cancelled handle.
