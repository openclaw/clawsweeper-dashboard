## Summary

Keep open; the route-inheritance idea is still relevant to the linked Feishu lost-reply bug, but the current head is not merge-ready because production handoffs still request message_tool_only suppression and the restart/pending-final recovery caller still does not preserve the inherited external route.

## What This Changes

The branch adds inherited external route/account/thread handling for sessions_send inter-session webchat handoffs and ACP hook dispatch, while also marking sessions_send and agent-step handoffs as message_tool_only.

## Best Possible Solution

Make sessions_send and agent-step handoffs use automatic source delivery, persist the inherited external route through restart/pending-final recovery, then refresh current-head redacted Feishu/Lark or equivalent real behavior proof before merge.

## Reproduction Assessment

Yes from source and linked issue logs, but not from a live run in this read-only review. The current head passes message_tool_only from sessions_send into the source-reply policy, which suppresses the final delivery path that should use the inherited external route.

## Solution Assessment

No; the current head is a plausible route-owner direction, but it is not the best fix while production handoffs suppress automatic delivery and recovery context still reads the live internal webchat route. The safer fix is one automatic delivery path with inherited route persistence in both live dispatch and recovery.
