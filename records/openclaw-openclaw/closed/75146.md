## Summary

Keep open for maintainer review: current main still has the duplicated-prompt retry path, and this PR is a narrow unmerged fix that applies cleanly on current main with focused regression coverage and no concrete security concerns.

## What This Changes

The PR tags mid-turn precheck recovery results, retries those recovery paths with a continuation prompt instead of the original user prompt, and adds regression coverage for truncation and compaction retries.

## Best Possible Solution

Land the narrow PR after the normal changed gate if maintainers accept the continuation-prompt approach, preserving turn-start preflight retries while preventing mid-turn recovery from resubmitting the original user request.

## Reproduction Assessment

Yes. Enable `agents.defaults.compaction.midTurnPrecheck.enabled`, trigger a post-tool-result precheck recovery, and current main reaches retry branches with `params.prompt` again; pi-coding-agent `prompt()` then creates and persists a new user message.

## Solution Assessment

Yes. Marking mid-turn recovery and substituting a continuation prompt is the narrowest maintainable fix against the existing runner interface; a native Pi continuation API integration could be explored later but is broader than this regression requires.
