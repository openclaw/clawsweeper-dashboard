## Summary

Keep open: the PR is aimed at a real memory/session pollution bug, but the branch deletes the boot-run session-store key that its new classifier needs, so BOOT.md transcripts can still be treated as ordinary conversation after the boot run. The PR also lacks inspectable after-fix real behavior proof.

## What This Changes

The PR routes BOOT.md startup turns through dedicated boot-run session keys, extends memory classification to skip boot-run transcripts, and adds reaping/tests for those entries.

## Best Possible Solution

Keep boot turns isolated from the main session, but preserve a durable boot-run classification signal or add reliable transcript-level boot detection, then prove memory ingestion skips a real boot transcript and aged boot-run entries are cleaned up.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main sends BOOT.md through the main session, and the PR branch deletes the boot-run key before later memory classification reads `sessions.json`. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

No; the peer boot-run category is a maintainable direction, but clearing the only durable classifier metadata before ingestion defeats the implementation. The safer fix is to keep or replace that durable marker and add a regression that covers post-run memory ingestion.
