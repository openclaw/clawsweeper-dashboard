## Summary

Current main does not fully cover the reported Telegram failure. The strict non-idempotent send retry path only retries pre-connect codes or Telegram 429s, while the broader recoverable-network helper only has the `failed after` grammY envelope. Also, the observed final-reply path still sends through direct `bot.api.sendMessage` wrappers rather than the standalone `sendMessageTelegram` safe-retry helper, so a classifier-only fix may not cover the logged final reply failures.

## Best Possible Solution

Keep this issue open and fix the narrow Telegram delivery gap with explicit safe-send semantics: cover grammY’s plain `HttpError` envelope only when it is demonstrably safe under the non-idempotent-send guardrails, and ensure the inbound final-reply path gets the same bounded retry behavior or an equivalent safe wrapper. Leave broader persistent queue/runtime drain work to #50040 unless the implementation must touch that design.
