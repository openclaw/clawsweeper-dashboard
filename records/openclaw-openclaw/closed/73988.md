## Summary

Close as superseded: the same symlink-bootstrap behavior is already tracked by the open maintainer-labeled #73401 path, and this PR also carries a concrete security regression that should not be resolved in a parallel bot PR.

## What This Changes

The branch adds a workspace bootstrap reader fallback that follows fixed top-level bootstrap-file symlinks to readable targets, adds regression tests, and adds a changelog entry.

## Best Possible Solution

Keep one maintainer/security-routed canonical implementation in #73401, preserve the useful reports and credits from #38622, #40210, #40230, and #52522 there, and require an explicit trust policy before external bootstrap symlink targets are loaded.

## Reproduction Assessment

Yes. The user bug can be reproduced by putting a top-level bootstrap-file symlink in a workspace that points outside the workspace and observing current main mark it missing; the PR security issue can be reproduced by pointing that symlink at a readable non-OpenClaw sensitive file.

## Solution Assessment

No. The PR fixes the missing-file symptom, but following arbitrary external targets with only a narrow OpenClaw-state denylist is not the safest or narrowest solution; the safer route is the maintainer-reviewed canonical path with containment, opt-in, or an audited allowlist.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep external bootstrap symlinks behind a trust boundary:** `src/agents/workspace.ts:215`
  - body: The new fallback lets a top-level `AGENTS.md`/`SOUL.md` symlink resolve to any readable local file, then only blocks known OpenClaw state paths.   - confidence: 0.91
