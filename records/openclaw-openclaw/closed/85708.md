## Summary

Keep open: this PR addresses a source-confirmed current-main restart-drain mismatch for embedded runs, and I found no blocking code-review finding. The remaining merge question is maintainer acceptance of the intentional compatibility and availability change around unbounded embedded-run waits.

## What This Changes

The PR removes the hidden 30s embedded-run restart cap, passes the configured restart drain budget to embedded runs, carries forced SIGUSR1 timeout intent through restart handling, adds focused tests, and adds a changelog entry.

## Best Possible Solution

Land the focused fix if maintainers accept the documented restart budget as the single embedded-run contract; otherwise revise toward an explicit, documented embedded-run watchdog with tests.

## Reproduction Assessment

Yes, source-backed: current main caps active embedded runs at 30s through resolveActiveRunDrainWaitMs even when the documented restart budget is longer or unbounded. I did not execute runtime proof because this review is read-only.

## Solution Assessment

Yes, if maintainers accept the documented drain-budget contract. A separate embedded-run watchdog is a valid alternative policy, but it should be explicit and tested rather than a hidden cap.
