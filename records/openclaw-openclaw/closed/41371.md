## Summary

Current `main` already has a configurable LLM first-response/idle timeout that covers the reported slow-local-model case, is documented, tested, and was shipped in the `2026.4.9` release notes.

## Risks / Open Questions

- The implemented control is global via `agents.defaults.llm.idleTimeoutSeconds`, not a per-provider `models.providers.<id>.connectTimeoutMs`; provider-specific tuning could still be a separate enhancement, but the core request to avoid premature first-token timeout is already satisfied.
