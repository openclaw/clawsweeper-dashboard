## Summary

Keep open: this is a focused Telegram bug fix candidate, but the patch is incomplete because it handles only `channels.telegram.network.dnsResultOrder` and leaves the documented env override path with the same sticky IPv4 fallback behavior. The submitted proof is also mock/test-only rather than real Telegram or affected-network proof.

## What This Changes

The PR changes Telegram fetch transport fallback to skip sticky IPv4 and pinned-IP fallback when `channels.telegram.network.dnsResultOrder` is explicitly set to a non-`ipv4first` value, and adds a regression test for that config path.

## Best Possible Solution

Treat any explicit non-`ipv4first` Telegram DNS decision, including the documented env override, as an opt-out from sticky IPv4 fallback while preserving default `ipv4first` recovery behavior.

## Reproduction Assessment

Yes at source level. The resolver returns `env:OPENCLAW_TELEGRAM_DNS_RESULT_ORDER` for `verbatim`, docs advertise that override, and the proposed guard only checks `source === "config"`; no live affected-network repro was run in this read-only review.

## Solution Assessment

No. The fix shape is narrow and plausible, but the best fix should key off an explicit non-`ipv4first` DNS decision rather than only the config source, with regression coverage for both documented override paths.
