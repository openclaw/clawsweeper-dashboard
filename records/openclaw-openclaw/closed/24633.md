## Summary

Current `main` already supports the reported Telegram bot-to-bot use case via normal sender allowlists and `channel_post` handling, so the blocker described in #24633 is implemented well enough without a separate `telegram.allowBots` key.

## Evidence

- **Inbound Telegram messages are not globally filtered by `is_bot`:** The main Telegram message handler only skips self-echoed bot messages (`from.id === ctx.me?.id`) and otherwise forwards inbound messages with `senderId`/`senderUsername` into the normal pipeline. A dedicated `channel_post` handler also normalizes bot-authored channel posts for processing.
  - file: [extensions/telegram/src/bot-handlers.runtime.ts:1954](https://github.com/openclaw/openclaw/blob/59a8afe6fa030f18af662e125e7dd826ce9a53ca/extensions/telegram/src/bot-handlers.runtime.ts#L1954)
  - command: `nl -ba extensions/telegram/src/bot-handlers.runtime.ts | sed -n '1948,2028p'`
  - sha: [59a8afe6fa03](https://github.com/openclaw/openclaw/commit/59a8afe6fa030f18af662e125e7dd826ce9a53ca)
- **Telegram group authorization is sender-ID based, not bot/human based:** Group allowlist enforcement calls `isSenderAllowed(...)` on the effective sender allowlist using `senderId` and `senderUsername`. There is no separate bot-message rejection in this path.
  - file: [extensions/telegram/src/group-access.ts:176](https://github.com/openclaw/openclaw/blob/d3f6783b16ecc10c67b101bf019ca4294453206a/extensions/telegram/src/group-access.ts#L176)
  - command: `nl -ba extensions/telegram/src/group-access.ts | sed -n '150,205p'`
  - sha: [d3f6783b16ec](https://github.com/openclaw/openclaw/commit/d3f6783b16ecc10c67b101bf019ca4294453206a)
- **Current Telegram config contract already exposes sender allowlists for groups:** The public Telegram config type documents `allowFrom` and `groupAllowFrom` as numeric sender allowlists and `groupPolicy: "allowlist"` as the group-message gate. There is no need for a separate `allowBots` contract to authorize another bot sender by numeric Telegram ID.
  - file: [src/config/types.telegram.ts:127](https://github.com/openclaw/openclaw/blob/d3f6783b16ecc10c67b101bf019ca4294453206a/src/config/types.telegram.ts#L127)

## Risks / Open Questions

- Discoverability is still a little poor because Telegram uses `allowFrom`/`groupAllowFrom` instead of a Slack-style `allowBots` key, so users may continue to search for the wrong config name.

## Close Comment

Closing from a Codex review because current `main` already covers the reported blocker. Telegram inbound handling no longer drops all bot-authored traffic: it only skips the bot's own echoed messages in `extensions/telegram/src/bot-handlers.runtime.ts:1954`, group authorization is driven by sender allowlists in `src/config/types.telegram.ts:127` and `extensions/telegram/src/group-access.ts:176`, and the live Telegram QA lane now explicitly allowlists one bot ID to talk to another in a private group in `extensions/qa-lab/src/live-transports/telegram/telegram-live.runtime.ts:491` with matching docs in `docs/concepts/qa-e2e-automation.md:74`. There is still no separate `telegram.allowBots` key, but the requested capability is already implemented on `main` via the existing Telegram sender-allowlist model.
