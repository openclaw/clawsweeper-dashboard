## Label Justifications

- `P1`: The reported fallback causes real opencode-go agent sessions to compact far too early, breaking a core long-context workflow for affected users.
- `impact:session-state`: The bug changes context and compaction state by using a 200K budget for models whose catalog advertises a 1M window.
- `impact:auth-provider`: The affected data is provider/model metadata used to route and size opencode-go model sessions correctly.

## Summary

Keep open: the report is a concrete model-context metadata bug with source-level support, and current main still leaves memory-flush/preflight threshold resolution dependent on warmed cache or manual config rather than reliably carrying the opencode-go catalog contextWindow into that path.

## What This Changes

The issue asks OpenClaw to load opencode-go plugin model contextWindow metadata into runtime context-window resolution so opencode-go models do not fall back to the 200K default for compaction thresholds.

## Best Possible Solution

Make the context-window resolver or its lifecycle-owned cache receive the active opencode-go provider catalog contextWindow before memory-flush and preflight compaction thresholds are computed, with focused regression coverage for opencode-go/deepseek-v4-pro or deepseek-v4-flash.

## Reproduction Assessment

Yes, source-reproducible: the memory-flush helper passes cfg plus allowAsyncLoad false and falls back to DEFAULT_CONTEXT_TOKENS when no provider/config/cache/modelContextWindow value is present. The issue also includes live gateway evidence showing compaction around the 200K threshold for a 1M opencode-go model.

## Solution Assessment

Yes, the requested direction is the right bug boundary, but the best fix should reuse lifecycle-owned provider/model metadata or pass prepared active-model context rather than adding request-time plugin discovery. A focused opencode-go regression should prove the chosen path before closure.
