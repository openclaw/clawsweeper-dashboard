## Summary

Keep open: current main now documents `allowConversationAccess` in the hooks reference and config type, and gateway startup logs plugin diagnostics, but warn-level hook denials still route through `info` and `openclaw plugins doctor` still hides warn diagnostics.

## What This Changes

The issue requests discoverable runtime and doctor signals when non-bundled plugins register conversation hooks without `plugins.entries.<id>.hooks.allowConversationAccess=true`.

## Best Possible Solution

Keep the default-deny security policy and existing diagnostics sink, but route warn-level plugin diagnostics as warnings and make `openclaw plugins doctor` surface warn diagnostics for blocked conversation hooks.

## Reproduction Assessment

Yes. Current source and existing loader coverage show a non-bundled plugin registering conversation hooks without `allowConversationAccess=true` gets zero typed hooks plus warn diagnostics, while doctor still filters those diagnostics out.

## Solution Assessment

Yes. The maintainable fix is a narrow diagnostics repair that keeps the conversation-content gate unchanged and improves warning-level visibility in gateway logs and plugin doctor.
