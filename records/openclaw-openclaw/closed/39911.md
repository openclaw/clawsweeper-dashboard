## Summary

Current `main` already covers this Feishu failure mode: Feishu config metadata is present in the generated bundled channel schema used by Settings/UI flows, Feishu setup now loads through a setup-only entry instead of importing the Lark SDK up front, and doctor has an explicit repair path for missing bundled Feishu runtime dependencies.

## Evidence

- **Reporter identified missing Feishu dependency as root cause:** The issue thread’s follow-up says the problem was that the Feishu extension had been uninstalled during upgrade and a dependency package was missing, matching the bundled-plugin runtime-dependency failure path now handled on main.
- **Feishu config is baked into generated bundled channel metadata:** `src/config/schema.ts:444` merges `GENERATED_BUNDLED_CHANNEL_CONFIG_METADATA` into the base config schema, and `src/config/bundled-channel-config-metadata.generated.ts:3240` contains a full `feishu` channel schema entry. That means Settings/config schema can expose Feishu without depending on full plugin runtime startup.
  - file: [src/config/schema.ts:444](https://github.com/openclaw/openclaw/blob/b98a6a46fb54145a33e753101c02f0d7d7e0bc3b/src/config/schema.ts#L444)

  - sha: [b98a6a46fb54](https://github.com/openclaw/openclaw/commit/b98a6a46fb54145a33e753101c02f0d7d7e0bc3b)
- **Feishu setup uses a setup-only entry path:** `extensions/feishu/setup-entry.ts:1` defines a bundled channel setup entry that loads `./setup-api.js`, and `extensions/feishu/setup-api.ts:1-3` exports the setup surfaces separately from the full runtime. This is the current main implementation for avoiding eager Lark SDK imports during setup/onboarding.
