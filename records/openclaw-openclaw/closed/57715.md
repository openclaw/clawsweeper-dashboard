## Summary

Current main now implements the narrow hot-path fix: sessions.list filters and limits before row construction, uses a request-scoped child-session index, and routes the RPC through the async listing path. The broader idea of a persistent summary/cache layer is still better tracked as separate follow-up work, not a reason to keep this specific report open.

## What This Changes

The issue requests a sessions.list performance fix for late row materialization, transcript fallback reads on discarded rows, and per-row child-session scans.

## Best Possible Solution

Keep the current main hot-path implementation, close this report, and leave broader persistent session-summary or direct-lookup indexing work to the newer broader follow-up such as #72165.

## Reproduction Assessment

Yes. The old failure mode is reproducible as a source-shape check: row construction used to happen before the requested limit and child lookup used to scan the store per built row; current main now shows the opposite structure and has targeted regression coverage for the limit/transcript case.

## Solution Assessment

Yes. The current implementation is the narrow maintainable fix for this report because it removes discarded-row transcript work and the O(N²) child-store scan without adding a broader TTL/cache layer whose invalidation policy belongs in separate product work.
