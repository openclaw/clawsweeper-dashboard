## Summary

This request is better owned by the external `openclaw-weixin` channel plugin (or a ClawHub-published follow-on plugin), not OpenClaw core. Current main documents WeChat as an external plugin surface, the plugin SDK already exposes setup/finalize hooks with full config access, and the project vision explicitly prefers optional capability in plugins rather than new core behavior.

## Evidence

- **WeChat is explicitly external plugin scope:** The WeChat docs say OpenClaw connects through Tencent's external `@tencent-weixin/openclaw-weixin` plugin, and that WeChat login, Tencent API calls, and account monitoring are owned by the external plugin rather than core.
  - file: [docs/channels/wechat.md:10](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/docs/channels/wechat.md#L10)

- **Plugin setup surface can mutate full OpenClaw config:** `ChannelSetupWizardFinalize` receives the full `OpenClawConfig`, the resolved `accountId`, and returns an updated config object. That is sufficient surface for a channel plugin to add agent records/bindings or prompt for account-specific setup without core special-casing.
  - file: [src/channels/plugins/setup-wizard-types.ts:235](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/src/channels/plugins/setup-wizard-types.ts#L235)

That makes this an ergonomic extension request rather than a missing core contract.

## Risks / Open Questions

- This assumes the `openclaw-weixin` maintainers are willing to own the UX and config mutation in their plugin.
- If maintainers later want a generic cross-channel 'create agent on new account' core workflow, that broader product decision would deserve a fresh core issue rather than this plugin-specific one.
