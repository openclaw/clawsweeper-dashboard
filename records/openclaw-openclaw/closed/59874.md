## Summary

Current `main` already covers the requested Slack progress-update work well enough: Slack send/edit actions accept Block Kit payloads, the Slack channel uses native Slack streaming transport under `channels.slack.streaming.nativeTransport`, and tool/plan/progress events are already routed into the live Slack preview path.

## Evidence

- **Slack message actions accept Block Kit payloads:** `handleSlackAction()` now parses a `blocks` parameter for both `sendMessage` and `editMessage`, allows empty text when blocks are present, and forwards the blocks into the Slack send/edit runtime.
  - file: [extensions/slack/src/action-runtime.ts:227](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/extensions/slack/src/action-runtime.ts#L227)
  - command: `nl -ba extensions/slack/src/action-runtime.ts | sed -n '150,280p'`
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **Blocks are passed through to Slack Web API:** `sendMessageSlack()` validates arbitrary block objects, then calls `chat.postMessage` with `blocks`; `editSlackMessage()` likewise calls `chat.update` with `blocks` and fallback text.
  - file: [extensions/slack/src/send.ts:310](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/extensions/slack/src/send.ts#L310)
  - command: `nl -ba extensions/slack/src/send.ts | sed -n '300,360p'`
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **Block validation is permissive enough for new Slack block types:** The block parser only requires an array of objects with a non-empty `type`, and the message-tool schema allows arbitrary additional properties, so newer Slack block payloads are not hard-coded out.
  - file: [extensions/slack/src/blocks-input.ts:13](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/extensions/slack/src/blocks-input.ts#L13)

- **Native Slack streaming transport is implemented:** The Slack plugin now has explicit `startSlackStream`, `appendSlackStream`, and `stopSlackStream` helpers built on Slack's `chatStream()` API, and the Slack reply dispatcher uses them for live replies when native transport is enabled.
  - file: [extensions/slack/src/streaming.ts:105](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/extensions/slack/src/streaming.ts#L105)

## Risks / Open Questions

- This appears to be implemented on `main` after the latest tagged release `v2026.4.23`, so release users may not have it yet.
- OpenClaw solves this at the Slack channel/runtime layer rather than exposing separate low-level public message-tool actions literally named `startStream` / `appendStream` / `stopStream`.
