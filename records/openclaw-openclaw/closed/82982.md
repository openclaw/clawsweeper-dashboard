## Summary

Keep open: current main has several Codex context-cap safeguards, but the preflight compaction path still resolves the budget from the selected provider/model without passing the effective Codex runtime provider, so the reported long-session failure remains plausible on current source.

## What This Changes

The issue reports that Codex/OAuth GPT-5.5 can be budgeted as a 1,000,000-token API-context model instead of the Codex-runtime budget, delaying lossless-claw compaction until context overflow.

## Best Possible Solution

Make preflight compaction and context-engine budgeting derive the same effective runtime context provider/cap as the actual Codex run, while preserving explicit lower user caps and repairing stale session values when the runtime cap is smaller.

## Reproduction Assessment

No high-confidence live reproduction was established here. The source path is reproducible by inspection: preflight compaction uses the selected provider/model budget before the runtime-aware Codex context provider is resolved for the actual run.

## Solution Assessment

No, the local workaround is not the best durable fix. The maintainable fix is to carry the effective Codex runtime context cap into preflight compaction, status/session persistence, and context-engine budgets consistently.
