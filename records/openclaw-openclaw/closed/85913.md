## Summary

Keep open: current main still has the same transcript-file contention window, and the added live gateway logs show this can become user-visible reply loss. The older fallback fix prevents wasting fallback candidates after takeover, but current code still allows duplicate embedded prompt holders to enter the same session file before the fence detects the race.

## What This Changes

Request to prevent concurrent embedded prompt holders and stuck-session recovery from racing the same session JSONL and causing EmbeddedAttemptSessionTakeoverError reply drops.

## Best Possible Solution

Add a transcript-file-scoped embedded prompt ownership guard before provider streaming, then make stuck-session recovery wait, steer, or fail cleanly instead of starting a second holder on the same JSONL file.

## Reproduction Assessment

Yes at source level, with supporting live logs: current tests and code show same-file mutation during releaseForPrompt causes EmbeddedAttemptSessionTakeoverError, and the reporter supplied real gateway traces showing reply loss. I did not run a live heartbeat/channel or stuck-recovery harness in this read-only review.

## Solution Assessment

Yes for the narrow repair: serialize or reject duplicate embedded prompt holders by resolved session file before provider streaming. Changing heartbeat isolatedSession defaults is optional product/config work and should not be required for the core reliability fix.
