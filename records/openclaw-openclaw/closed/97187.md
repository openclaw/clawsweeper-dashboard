## Summary

Keep open, but not as an automated bug repair: current main and v2026.6.10 intentionally expose caller-supplied HTTP tools as client-executed tool calls, so the remaining request is a Gateway product/security decision about adding an opt-in server-side memory or tool bridge.

## What This Changes

The issue asks OpenClaw to make the Gateway chat-completions endpoint execute or prefetch memory-backed tool calls server-side instead of returning `finish_reason: "tool_calls"` for the client to handle.

## Best Possible Solution

Keep the existing client-tool contract unless maintainers explicitly sponsor an opt-in server-side memory/RAG mode with a documented trust boundary and compatibility coverage.

## Reproduction Assessment

Yes for the observed response shape: current source, tests, docs, and v2026.6.10 all show `finish_reason: "tool_calls"` with client-executed follow-up semantics. No live OpenManager Gateway replay was run, and the remaining dispute is whether that documented behavior should change.

## Solution Assessment

No as a bug fix; server-side memory prefetch or tool execution is a new Gateway capability, not a repair to the current contract. The safer current path is for callers to execute returned `tool_calls` and send a `role: "tool"` follow-up unless maintainers approve an explicit opt-in server-side mode.
