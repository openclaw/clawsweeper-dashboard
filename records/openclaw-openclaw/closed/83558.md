- `impact:auth-provider`: The affected surface is OpenAI Codex auth method selection and provider routing for login.

## Summary

Close as superseded: the current and shipped OpenAI auth surface still exposes device-code login explicitly, while the no-prompt OAuth default was an intentional merged CLI behavior from https://github.com/openclaw/openclaw/pull/80790 rather than a missing `@openclaw/codex` registration.

## What This Changes

The issue asks to restore the Codex device-code auth option or method picker for `openclaw models auth login --provider openai-codex` after v2026.5.12.

## Best Possible Solution

Keep the current OAuth-default CLI contract and direct headless users to explicit `--device-code`; track any actual failure of that explicit method as a separate auth-provider bug.

## Reproduction Assessment

Yes. Source inspection shows the no-prompt observation: without an explicit method, `pickProviderAuthMethod` chooses OAuth before the method picker; source and package inspection also show `device-code` is still reachable explicitly.

## Solution Assessment

No. Re-registering device-code in `@openclaw/codex` is not the right fix because that package owns provider `codex`; `openai-codex` device-code lives in the OpenAI plugin and is already registered.
