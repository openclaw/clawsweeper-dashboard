## Label Justifications

- `P1`: A hot-looping Telegram spooled update can block real inbound messages while the gateway appears live.
- `impact:message-loss`: The reported retry loop leaves later Telegram messages unprocessed behind the poisoned update.
- `impact:crash-loop`: The process stays up but can enter a tight retry/log loop that makes the Telegram lane effectively unavailable.
- `impact:session-state`: The reported trigger is a reply-session initialization conflict for a channel session key.

## Summary

Current main already fixes this issue's central Telegram retry-storm symptom: `reply session initialization conflicted` spooled updates now use capped retry backoff and lane reservation instead of being reclaimed every drain tick. The requested dead-letter classifier was intentionally avoided in the merged fix because this conflict can be retryable; broader poison-update policy remains tracked separately.

## What This Changes

The issue reports that Telegram polling can retry a deterministic spooled update indefinitely without advancing the getUpdates offset.

## Best Possible Solution

Keep the current main Telegram backoff for reply-session init conflicts, and leave generic poison retry policy to https://github.com/openclaw/openclaw/issues/92980 plus upstream false-conflict cleanup to https://github.com/openclaw/openclaw/pull/96936.

## Reproduction Assessment

Yes at source level for the affected beta/release path: v2026.6.10 lacks the conflict-backoff helper and immediately releases failed spooled updates for retry. Current main has source and regression-test evidence that the immediate retry storm is blocked; no live Telegram replay was run in this read-only review.

## Solution Assessment

Yes for this issue's central retry-storm symptom. Telegram-owned backoff using existing ingress-queue metadata is the narrowest maintainable fix, while dead-lettering this conflict was deliberately avoided because it can drop legitimate retryable updates.
