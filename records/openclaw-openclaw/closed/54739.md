## Summary

Current main already threads explicit Slack `accountId` through `message read`, resolves that Slack account, and uses that account's read token (`userToken ?? botToken`) for `readMessages`. The same implementation is present in release `v2026.4.22`, so the reported `default`-token DM read path is already fixed in shipped code.

## Evidence

- **Message tool preserves explicit accountId:** `message` tool reads `accountId` from params, writes it back into the action params, and passes it as `defaultAccountId` into `runMessageActionForTool`.
  - file: [src/agents/tools/message-tool.ts:764](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/src/agents/tools/message-tool.ts#L764)
  - sha: [a6aa151653ab](https://github.com/openclaw/openclaw/commit/a6aa151653ab478e9aa23e95c6446c41adc4fce1)
- **Runner keeps accountId on plugin action context:** `runMessageAction` resolves `accountId` from params/defaults and writes it into `params.accountId` before plugin dispatch.
  - file: [src/infra/outbound/message-action-runner.ts:848](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/src/infra/outbound/message-action-runner.ts#L848)
  - sha: [a6aa151653ab](https://github.com/openclaw/openclaw/commit/a6aa151653ab478e9aa23e95c6446c41adc4fce1)
- **Slack read action forwards accountId and uses selected account token:** Slack message-action dispatch includes `accountId` on `readMessages`, and `handleSlackAction` resolves that account, then builds `readOpts` from that account's `userToken ?? botToken` before calling `readSlackMessages`.
  - file: [extensions/slack/src/message-action-dispatch.ts:107](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/extensions/slack/src/message-action-dispatch.ts#L107)
  - sha: [a6aa151653ab](https://github.com/openclaw/openclaw/commit/a6aa151653ab478e9aa23e95c6446c41adc4fce1)
- **Slack runtime proves account-scoped read-token selection:** `handleSlackAction` reads `accountId`, resolves the Slack account, and for reads chooses `userToken ?? botToken`, carrying both `accountId` and any token override into action opts.
  - file: [extensions/slack/src/action-runtime.ts:151](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/extensions/slack/src/action-runtime.ts#L151)
  - sha: [a6aa151653ab](https://github.com/openclaw/openclaw/commit/a6aa151653ab478e9aa23e95c6446c41adc4fce1)
- **Tests cover read-token behavior in shipped code:** Slack runtime tests assert that reads use the user token when available and fall back to the bot token otherwise; the same test/code exists in tag `v2026.4.22`.
