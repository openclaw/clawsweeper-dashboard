## Summary

Keep open for normal PR review/merge: the patch matches a real source-backed Telegram session-state bug, has sufficient redacted live-output proof, and I found no blocking correctness or security finding. The remaining maintainer decision is accepting that transient Telegram progress is chat-only instead of active transcript history.

## What This Changes

The PR gates Telegram direct-delivery transcript mirroring so non-final progress stays chat-only while durable/final replies still append to the session transcript, with a regression test and changelog entry.

## Best Possible Solution

Land the Telegram-boundary fix after required checks and maintainer acceptance of the transcript-history tradeoff; if durable progress history is needed later, add a separate diagnostic store instead of writing transient progress into active session transcripts.

## Reproduction Assessment

Yes: the current-main source path shows non-final direct Telegram deliveries can receive `transcriptMirror`, and the embedded lock rejects session-file changes after prompt-lock release. I did not rerun the live MiniMax Telegram flow, but the PR body provides redacted before/after live output for that path.

## Solution Assessment

Yes: limiting `transcriptMirror` at the Telegram direct-delivery boundary is the narrow maintainable fix, because it preserves final/durable mirroring while preventing transient progress from mutating the active transcript. A provider-specific workaround would miss the shared Telegram dispatch cause.
