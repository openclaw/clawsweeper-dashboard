## Summary

Keep open: current main and the latest release still document bare Lobster `openclaw.invoke --tool llm-task` examples, while the embedded runner only forwards ambient env and the Lobster shim requires URL/auth env or flags before it can call Gateway `/tools/invoke`.

## What This Changes

The issue requests that the documented Lobster-to-`llm-task` workflow either receive the required Gateway URL/auth context, document the full safe invocation setup, or point to a supported replacement.

## Best Possible Solution

Document the complete trusted-local setup for Lobster `openclaw.invoke` steps, including compatible URL/token env or flags, and state that the embedded runner does not implicitly inject Gateway credentials until a scoped auth boundary exists.

## Reproduction Assessment

Yes. I did not run a live Gateway workflow in this read-only sweep, but current main and `v2026.5.2` source reproduce the mismatch: docs show a bare command, the runner only forwards ambient env, the shim requires `OPENCLAW_URL`, and Gateway rejects unauthenticated `/tools/invoke`.

## Solution Assessment

Yes for the immediate repair if it is docs-first. The safer path is to clarify explicit URL/auth prerequisites and credential scope; automatic credential injection should remain a separate design/security decision.
