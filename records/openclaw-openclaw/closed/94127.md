## Summary

Keep this issue open: current main still prefers a caller-attached gateway method registry before rebuilding from the live plugin registry, and the linked open PR https://github.com/openclaw/openclaw/pull/94154 remains the canonical fix candidate rather than a merged implementation.

## What This Changes

The issue reports that plugin-registered gateway RPC methods appear in runtime inspection but are rejected by `openclaw gateway call` as `INVALID_REQUEST: unknown method`.

## Best Possible Solution

Review and land or reject https://github.com/openclaw/openclaw/pull/94154 as the canonical dispatcher fix, then close this issue after the fix merges with adequate plugin gateway-call proof.

## Reproduction Assessment

Yes, source-level: the WebSocket path passes an attached method registry into `handleGatewayRequest`, and current main still prefers that snapshot before the live plugin registry that `registerGatewayMethod` updates. I did not run the live plugin install/restart/CLI repro in this read-only review.

## Solution Assessment

Yes. The best fix belongs at the gateway dispatch chokepoint: preserve the attached registry when it owns the method, and fall back to the live plugin registry for stale misses.
