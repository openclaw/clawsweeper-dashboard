## Summary

Keep open. Current main still captures `startupConfigSnapshotRead` before the gateway loop and reuses it for every in-process restart start callback, and two open PRs with closing references already track the narrow fix.

## What This Changes

The issue requests fixing gateway in-process restarts so post-boot user config changes are not overwritten by a stale startup config snapshot.

## Best Possible Solution

Land one first-start-only snapshot consumption fix with a regression that calls the gateway start callback twice and verifies the second start rereads disk.

## Reproduction Assessment

Yes. Current main gives a high-confidence source-level reproduction path: write config after the initial gateway pre-read, trigger an in-process restart, and the next loop iteration still injects the original snapshot instead of rereading disk.

## Solution Assessment

Yes. The first-start-only `startupConfigSnapshotRead` approach is the narrow maintainable fix because it preserves the cold-start optimization while forcing restart iterations through the normal disk snapshot loader.
