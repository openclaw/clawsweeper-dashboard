## Summary

Keep open: the PR targets a real source-reproducible message-delivery bug, but the current parser can misroute supported per-account direct sessions and the supplied proof is a standalone copied parser probe rather than production-path behavior.

## What This Changes

The PR adds a guarded session-key-derived channel fallback in `createMessageTool` and threads the effective provider/id into schema, description, secret-scope, and runtime tool context.

## Best Possible Solution

Keep the narrow fallback direction, but parse canonical session-key shapes explicitly, add regression coverage for normal and per-account direct sessions, and require redacted production-path proof before merge.

## Reproduction Assessment

Yes, at source level: current main passes stale `currentChannelProvider: webchat` into `toolContext`, and the outbound runner can select the internal UI sink before normal channel resolution. I did not run a live Signal or Telegram gateway reproduction in this read-only review.

## Solution Assessment

No, not in the current PR form: the guarded session-key fallback is the right narrow direction, but the parser must skip account segments in supported per-account session keys before choosing the peer target.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Parse account-scoped session keys before choosing target:** `src/agents/tools/message-tool.ts:833`
  - body: For a supported key such as `agent:main:telegram:account-a:direct:peer-1`, this loop returns `account-a` as the effective `currentChannelId` because it takes the first non-marker segment after the channel. In the stale-webchat case, untargeted sends would route to the account id instead of the DM peer, so multi-account direct sessions remain broken or can be misrouted.
  - confidence: 0.92
