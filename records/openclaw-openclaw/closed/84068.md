## Label Justifications

- `P1`: The in-product update path is broken for recommended Linux systemd-supervised installs on a current shipped release, even though a manual shell upgrade workaround exists.

## Summary

Keep open: current `main` and `v2026.5.18` still start the managed update helper with a plain detached child process before scheduling the supervised restart, while systemd's default control-group kill behavior explains why that helper can die before writing `handoff.log`. The report is concrete, maps to the documented control-plane update path, and is a narrow updater bug rather than cleanup to close.

## What This Changes

Clicking `Update now` on systemd-supervised package-manager installs should hand the update helper outside the service cgroup and report a useful failure if the helper never runs.

## Best Possible Solution

Keep the existing managed-service update contract, but make the systemd path launch the helper in a transient user scope or service outside `openclaw-gateway.service`, then add a diagnostic sentinel state for helpers that never create `handoff.log`.

## Reproduction Assessment

Yes, there is a high-confidence reproduction path from the reporter's concrete Linux user-systemd upgrade and the current source path. I did not run a live systemd upgrade in this read-only review, so the structured status is source-reproducible rather than locally reproduced.

## Solution Assessment

Yes, the suggested direction is the maintainable fix: the systemd helper should escape the service cgroup before the gateway restart, while preserving the existing managed-service contract and adding a clearer never-ran diagnostic.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/84068. The observable bug is that `Update now` / `update.run` on Linux user-systemd global installs writes the managed-service handoff files and sentinel, then restarts `openclaw-gateway.service` before the helper can run, leaving no `handoff.log` and no version change. Keep the control-plane contract that package-manager updates do not run inside the live Gateway. Add a systemd-specific branch in `src/gateway/server-methods/update-managed-service-handoff.ts` that launches the helper outside the current unit cgroup, preferably with `systemd-run --user --scope --unit=openclaw-update-<safe id> --collect` or an equivalent transient user unit, and falls back to the existing unavailable/failed behavior when that handoff cannot be started. Preserve launchd, Windows task, and non-supervised behavior. Add or update focused Vitest coverage for the systemd argv/env/sentinel behavior, the existing plain detached path where still appropriate, and a diagnostic `managed-service-handoff-helper-never-ran` or equivalent if the started sentinel survives without a helper log. Update user-facing docs only if response reasons or diagnostics change. Do not add a new config option, do not run npm install inside the live Gateway, and do not weaken cleanup of helper scripts or sentinel metadata.
