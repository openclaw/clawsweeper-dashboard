## Label Justifications

- `P1`: The in-product update workflow is broken for recommended Linux user-systemd installs, leaving users stuck on the old gateway version unless they manually upgrade from a shell.

## Summary

Keep open: current main and v2026.5.20 still launch the managed update helper as a plain detached child while generated systemd units use control-group kill semantics. Two open PRs already target this issue with closing references, so the issue should stay open until one lands.

## What This Changes

Fix systemd-supervised Control UI/package updates so the handoff helper escapes the gateway service cgroup and Update Now can complete instead of staying at `managed-service-handoff-started`.

## Best Possible Solution

Review and land one canonical scoped-systemd handoff PR, then close this issue with the merged commit and release provenance while preserving non-systemd paths.

## Reproduction Assessment

Yes. The reporter gives a concrete Linux user-systemd upgrade failure, and current source still starts the helper as a detached child before scheduling the supervised restart; I did not run a live systemd upgrade in this read-only review.

## Solution Assessment

Yes. A transient user systemd scope is the narrow maintainable direction for a helper that must survive the gateway service cgroup restart, but this issue should stay open until a linked PR lands.
