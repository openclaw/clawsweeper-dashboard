## Summary

Keep open. The PR fixes the direct cron wake adapter, but current main still leaves queued cron fallback wakes unsanitized, so the linked cross-context routing bug can still reproduce when a direct wake skips as busy or cron-in-progress.

## What This Changes

The branch drops inherited heartbeat `to` and `accountId` fields for direct cron `target: "last"` wakes, with a gateway regression test and changelog entry.

## Best Possible Solution

Move destination-field omission to a shared cron forced-target-last wake boundary used by both direct and queued wakes, with regression coverage for busy and cron-in-progress fallback cases.

## Reproduction Assessment

Yes. A main-session `systemEvent` cron job with `wakeMode: "now"` and global heartbeat `to`/`accountId` can reproduce when direct `runHeartbeatOnce` skips as busy or `cron-in-progress`, because the queued fallback later merges `{ target: "last" }` with the configured heartbeat defaults.

## Solution Assessment

No. Direct adapter sanitization is directionally correct, but the narrow maintainable fix must also cover queued `requestHeartbeat` cron wakes that use the same forced target-last semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9
