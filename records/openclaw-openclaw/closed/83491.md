## Label Justifications

- `P1`: The reported release regression made a real WhatsApp agent workflow unusable until the current-main fix.
- `impact:message-loss`: The reported WhatsApp channel returned errors instead of agent replies.
- `impact:session-state`: The report centers on existing channel session/runtime binding changing across upgrade.
- `impact:auth-provider`: The failure replaced Claude CLI OAuth intent with an Anthropic API-key path.

## Summary

Current main already fixes the central runtime rebinding: legacy Claude CLI OAuth configs are backfilled into model-scoped `agentRuntime: { id: "claude-cli" }`, which is the runtime contract used by WhatsApp gateway turns. The fix is on main after v2026.5.12 and is not proven shipped in a release yet.

## What This Changes

The issue reports that upgrading to v2026.5.12 makes WhatsApp sessions ignore a legacy `claude-cli` runtime, fall back to Pi, lose tools, and show misleading billing/auth errors.

## Best Possible Solution

Keep the merged model-scoped Claude CLI backfill, ship it in the next release, and track downgrade/plugin repair or broader OAuth error-copy work in narrower follow-up items.

## Reproduction Assessment

Yes, for the original regression at source level: the merged fix PR documents the pre-fix resolver gap, and current main has regression coverage for legacy Claude CLI OAuth config backfill. I did not run a live WhatsApp upgrade reproduction in this read-only review.

## Solution Assessment

Yes. Writing provider/model-scoped `claude-cli` runtime policy matches the documented runtime contract and avoids re-enabling legacy whole-agent runtime selection.
