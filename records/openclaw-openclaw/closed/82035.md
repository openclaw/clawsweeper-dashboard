## Summary

Keep open: the reporter’s release evidence matches a current-main source path where Discord marks a partial draft as final-delivery-handled before preview finalization or fallback delivery has definitely produced the final visible reply. The broader related report remains open, but this issue describes a distinct final-delivery mismatch that is narrow enough to fix directly.

## What This Changes

The issue reports that Discord partial streaming can leave a truncated live draft visible even though the full assistant reply is stored in the OpenClaw transcript.

## Best Possible Solution

Make Discord partial-preview finalization state reflect successful visible final delivery, or make cleanup clear any pending partial preview when finalization/fallback is skipped or fails after a final payload is available, with a focused regression test for the failure path.

## Reproduction Assessment

No, not as a live end-to-end reproduction from this review. The issue is source-reproducible on current main because the Discord partial draft can be marked handled before the final edit or fallback send has actually succeeded, and cleanup then skips clearing that draft.

## Solution Assessment

Yes, a narrow code fix in the Discord draft-preview/final-delivery path is the best direction. Changing the default streaming mode or transcript storage would not address the broken final visible reply contract.
