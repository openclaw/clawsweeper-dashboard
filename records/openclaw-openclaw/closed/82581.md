## Label Justifications

- `P2`: The issue describes a downgrade/state-safety gap with limited but important Docker and gateway startup scope, and it needs policy review before implementation.
- `impact:data-loss`: The report centers on state fragmentation and potential SQLite/state corruption across downgrade-created split directories.
- `impact:session-state`: The reported failure mode would mis-associate or fragment agent workspace, session, and checkpoint state.

## Summary

Keep open: current main has a future-version guard, but Docker launches the foreground gateway path without the service marker that triggers the hard startup block, so the reported downgrade path is still not clearly covered.

## What This Changes

The issue requests startup/layout version gating to prevent older Docker or foreground runtimes from operating against state/config last written by a newer OpenClaw layout.

## Best Possible Solution

Define the supported downgrade/recovery contract, then extend the existing `meta.lastTouchedVersion` guard or add a state-layout marker only where it reliably covers Docker and foreground gateway startup without blocking documented recovery.

## Reproduction Assessment

No for the exact reported Docker downgrade: I did not establish a live 2026.5.7-to-newer volume reproduction. Source inspection does show the relevant current-main gap, because Docker uses foreground gateway startup while the hard future-config startup halt only applies when `OPENCLAW_SERVICE_MARKER` is set.

## Solution Assessment

Unclear: the requested `.openclaw_version` marker may not be the best path because OpenClaw already has `meta.lastTouchedVersion` and a future-config guard. The maintainable fix is to decide whether that guard should apply to Docker/foreground gateway startup, and what recovery override remains supported.
