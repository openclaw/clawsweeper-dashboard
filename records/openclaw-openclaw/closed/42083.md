## Summary

Keep this PR open. Current main still scans the full `memory-lancedb` `agent_end` message history and applies `toCapture.slice(0, 3)` after filtering, so the inefficiency/starvation behavior this PR targets is still present. The PR is not ready to land as-is because it is stale against current live-config hook semantics and needs focused cursor/reset/compaction/retry tests, but it remains a useful implementation candidate, especially with related open issue #71349 active.

## Best Possible Solution

Keep this PR open as an implementation candidate. Rebase it onto current main, preserve live runtime config semantics, simplify or prove the cursor invalidation model, and add focused `memory-lancedb` regression tests for duplicate skipping across repeated `agent_end` calls, retry-after-failure, reset/session cleanup, and compaction or history rewrite handling. Coordinate the final fix with #71349.

## Risks / Open Questions
