## Summary

Keep open: this is a plausible focused gateway diagnostics bug fix, but the reply-run activity mirroring still has a blocking lifecycle gap and the external draft PR lacks after-fix real behavior proof.

## What This Changes

The PR mirrors reply operations into diagnostic active-work state, treats `noop/no_active_work` recovery as queue-clearing, and adds diagnostic regression tests.

## Best Possible Solution

Track reply operations as their own diagnostic active-work source, or make the embedded-run marker reference-counted, while keeping the noop queue-clear fix and focused regressions.

## Reproduction Assessment

Yes, from source inspection: current main does not mirror reply-run registry activity into diagnostic activity, and `noop/no_active_work` is non-mutating. I did not run the tests in this read-only review, but the PR's focused tests encode the two reported paths.

## Solution Assessment

No. The noop cleanup is narrow, but the reply-run mirroring uses a shared embedded-run boolean that existing embedded-run cleanup can clear before the reply operation ends; use a separate or ref-counted activity marker instead.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep reply activity alive after embedded cleanup:** `src/auto-reply/reply/reply-run-registry.ts:382`
  - body: This marker shares the existing `activeEmbeddedRun` boolean with the embedded runner, but `clearActiveEmbeddedRun()` already calls `markDiagnosticEmbeddedRunEnded()` before terminal cleanup, while the outer `ReplyOperation` remains active until the `agent-runner` finally block. That means the exact window where recovery still sees `active_reply_work` can again have no diagnostic `activeWorkKind`, so the detector can still emit `queued_work_without_active_run`. Track reply-run activity separately or ref-count the marker so embedded-run cleanup cannot erase it before `clearReplyRunState()`.
  - confidence: 0.86
