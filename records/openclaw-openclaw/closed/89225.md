## Summary

Keep open: current main and the latest release still eagerly start fs-safe root resolution during workspace-scoped host write/edit tool construction, and the open closing PR is the active fix path.

## What This Changes

`openclaw doctor` should not emit an unhandled `FsSafeError: root dir not found` rejection when an agent workspace directory is missing.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/89226, or an equivalent lazy-root fix, then close this issue once the fix reaches main.

## Reproduction Assessment

Yes. The source path is high-confidence: doctor schema projection constructs coding tools, current main eagerly starts `fsRoot(root)` for workspace-only write/edit operations, and the pinned fs-safe dependency rejects missing roots with the reported error; I did not run the CLI because this review is read-only.

## Solution Assessment

Yes. Lazy, memoized root resolution at the host workspace operation boundary is the narrow maintainable fix because schema-only construction stops touching the filesystem while real read/write/access calls still use fs-safe enforcement.
