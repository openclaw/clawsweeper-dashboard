- `P1`: The report describes a real Telegram channel workflow becoming silent while queued user messages pile up behind one poison update.

## Summary

Keep open: current main and v2026.5.20 still requeue all Telegram spooled handler errors, and the linked closing PR is open and unmerged, so this issue should not be closed before a fix lands.

## What This Changes

The issue reports that a Telegram slash-command update can poison the ingress spool when `MissingAgentHarnessError` is retried indefinitely instead of being dead-lettered.

## Best Possible Solution

Land a focused fix, likely through the linked PR or a replacement, that classifies missing-harness spool failures as non-retryable and uses the existing failed-tombstone path while keeping transient failures retryable.

## Reproduction Assessment

Yes, source-reproducible: a thrown `bot.handleUpdate` error enters `#releaseFailedSpooledUpdate`, releases the claim, and requeues the same update; the current test suite also expects that retry behavior. I did not run a live Telegram repro in this read-only pass.

## Solution Assessment

Yes for the narrow spool-classification fix: dead-lettering nonrecoverable missing-harness errors through the existing tombstone path is the maintainable path, while slash fallback or a spool-status CLI can remain follow-up work.
