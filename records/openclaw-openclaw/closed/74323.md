## Summary

Keep open. Current main has an unshipped Gateway client readiness change that is directly relevant to loopback handshake timeouts, plus earlier loopback proxy-bypass work, but I could not prove from source/history alone that it fixes this exact active-memory plus Lark/Feishu streaming failure class. The issue is recent, concrete, against the latest shipped release, and overlaps still-open Gateway starvation reports, so closing now would be premature.

## What This Changes

The issue reports loopback Gateway WebSocket pre-auth handshake timeouts in OpenClaw 2026.4.26 that break active-memory embedded subagent recall, Lark/Feishu streaming card output, and embedded runs.

## Best Possible Solution

Keep this open through a reporter or maintainer retest on a build containing 6bbacd14 or newer. If the issue still reproduces, fold the remaining work into the Gateway starvation track from #73655 and add a focused regression around loopback GatewayClient construction/readiness under active-memory embedded recall and the relevant streaming client path.
