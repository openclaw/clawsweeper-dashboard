## Summary

Current `main` already covers the reported WhatsApp reconnect-drop path: inbound/reply sends follow socket replacement across reconnects, transient disconnect errors are retried, outbound deliveries are persisted before send, and the WhatsApp monitor drains pending deliveries for the reconnecting account. The changelog records this as shipped in `v2026.4.10`.

## Evidence

- **Reply/send path survives reconnect gaps:** WhatsApp inbound send closures use a shared `socketRef` so sends can follow a replaced socket, and `sendTrackedMessage` retries retryable disconnect / no-active-socket failures with reconnect backoff instead of dropping immediately.
  - file: [extensions/whatsapp/src/inbound/monitor.ts:85](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/extensions/whatsapp/src/inbound/monitor.ts#L85)
  - command: `nl -ba extensions/whatsapp/src/inbound/monitor.ts | sed -n '35,120p'`
- **Reconnect retry loop for send failures:** `sendTrackedMessage` retries on `closed|reset|timed out|disconnect|no active socket` errors, clears the stale socket ref, waits for reconnect, then re-sends on the current socket.
  - file: [extensions/whatsapp/src/inbound/monitor.ts:260](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/extensions/whatsapp/src/inbound/monitor.ts#L260)
  - command: `nl -ba extensions/whatsapp/src/inbound/monitor.ts | sed -n '230,320p'`
- **Pending deliveries drained on WhatsApp reconnect:** When WhatsApp reconnects, the monitor calls `drainPendingDeliveries` for that WhatsApp account and bypasses normal backoff for `No active WhatsApp Web listener` failures so queued sends are replayed immediately after reconnect.
  - file: [extensions/whatsapp/src/auto-reply/monitor.ts:393](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/extensions/whatsapp/src/auto-reply/monitor.ts#L393)
  - command: `nl -ba extensions/whatsapp/src/auto-reply/monitor.ts | sed -n '380,420p'`
- **Outbound messages are write-ahead queued:** Core outbound delivery persists a queue entry before sending and only acks it after success, so reconnect recovery has something concrete to replay instead of silently losing the message.
  - file: [src/infra/outbound/deliver.ts:649](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/src/infra/outbound/deliver.ts#L649)
  - command: `nl -ba src/infra/outbound/deliver.ts | sed -n '620,760p'`
- **Reconnect drain behavior is covered by tests:** Recovery tests verify reconnect drains fresh pending entries, immediately retries `no listener` failures, and skips duplicate re-delivery while another live delivery already holds the queue claim.
  - file: [src/infra/outbound/delivery-queue.reconnect-drain.test.ts:96](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/src/infra/outbound/delivery-queue.reconnect-drain.test.ts#L96)
