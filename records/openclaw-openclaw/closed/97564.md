## Summary

Keep open: current main and v2026.6.10 still make the retry-limit path return a local error for `model_not_found` instead of throwing a `FailoverError` that lets the configured fallback chain continue.

## What This Changes

The issue asks OpenClaw to use configured model fallbacks when a provider reports `model_not_found` for a decommissioned primary model, while still surfacing the error when no distinct fallback remains.

## Best Possible Solution

Update the existing retry-limit failover decision so `model_not_found` can advance to a remaining distinct configured fallback, while preserving the current terminal error when no fallback is configured or the chain is exhausted.

## Reproduction Assessment

Yes at source level: current main returns `return_error_payload` for retry-limit exhaustion with `failoverReason: "model_not_found"` even when `fallbackConfigured` is true. I did not run a live provider-decommission repro.

## Solution Assessment

Yes: the narrow maintainable fix is to update the existing retry-limit failover policy and tests, not add a new config flag or duplicate fallback machinery.
