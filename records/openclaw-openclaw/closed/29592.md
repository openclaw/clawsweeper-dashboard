## Summary

Current main already prevents the Feishu typing reaction from being re-posted on every keepalive tick, and that same logic is present in the latest shipped release.

## Evidence

- **Feishu typing add is now idempotent:** The Feishu reply dispatcher short-circuits repeated keepalive starts once a reaction already exists: `if (typingState?.reactionId) return;`. That prevents repeated `POST /open-apis/im/v1/messages/:message_id/reactions` calls after the first successful add.
  - file: [extensions/feishu/src/reply-dispatcher.ts:144](https://github.com/openclaw/openclaw/blob/c6684af682bcb59bd1ee85cb49f72d4074007e2c/extensions/feishu/src/reply-dispatcher.ts#L144)
  - command: `nl -ba extensions/feishu/src/reply-dispatcher.ts | sed -n '120,165p'`
  - sha: [c6684af682bc](https://github.com/openclaw/openclaw/commit/c6684af682bcb59bd1ee85cb49f72d4074007e2c)
- **Cleanup seals later keepalive ticks:** `fireStop()` sets `closed = true`, and the typing start guard refuses to run `start()` when sealed. Even if the async start/cleanup race occurs, later keepalive ticks are skipped instead of continuing to call the Feishu reaction API every 3 seconds.
  - file: [src/channels/typing.ts:87](https://github.com/openclaw/openclaw/blob/c6684af682bcb59bd1ee85cb49f72d4074007e2c/src/channels/typing.ts#L87)
  - command: `nl -ba src/channels/typing.ts | sed -n '1,120p'`
  - sha: [c6684af682bc](https://github.com/openclaw/openclaw/commit/c6684af682bcb59bd1ee85cb49f72d4074007e2c)
- **Sealed-state check is enforced in the shared guard:** `createTypingStartGuard()` returns `skipped` whenever `isSealed()` is true, which is the mechanism that blocks restarted keepalive ticks after cleanup.
  - file: [src/channels/typing-start-guard.ts:20](https://github.com/openclaw/openclaw/blob/c6684af682bcb59bd1ee85cb49f72d4074007e2c/src/channels/typing-start-guard.ts#L20)
  - command: `sed -n '1,220p' src/channels/typing-start-guard.ts`
  - sha: [c6684af682bc](https://github.com/openclaw/openclaw/commit/c6684af682bcb59bd1ee85cb49f72d4074007e2c)
- **The fix is already shipped:** The latest release tag contains the same Feishu idempotency guard in `extensions/feishu/src/reply-dispatcher.ts`, so this is not only fixed on HEAD but also present in the released tree.
  - file: [extensions/feishu/src/reply-dispatcher.ts:144](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/extensions/feishu/src/reply-dispatcher.ts#L144)

## Risks / Open Questions

- `src/channels/typing.ts` still does not re-check `closed` immediately after `await fireStart()`, so a narrower race may still leave a stale Feishu reaction behind even though it no longer causes the reported repeated POST loop/quota exhaustion.
