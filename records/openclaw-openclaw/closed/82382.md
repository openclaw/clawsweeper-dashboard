## Summary

Current main already terminalizes the Codex `<turn_aborted>` path and persists a terminal session lifecycle state, so the reported `status="running", hasActiveRun=false` state should not remain after the abort cleanup on main. The linked Telegram ingress-init report is a separate closed issue and does not leave unique work here.

## What This Changes

The issue asks for embedded Codex turn-abort cleanup to release a session from `running` when no active run remains.

## Best Possible Solution

Keep the current-main terminal lifecycle behavior and let the next release carry it to users who are still on `v2026.5.12`.

## Reproduction Assessment

No high-confidence current-main reproduction remains from source inspection: the abort marker now resolves as an aborted run and the gateway lifecycle path maps aborted terminal events away from `running`. The reported shipped-release behavior is credible, but I did not run a live Telegram/Codex abort scenario in this read-only review.

## Solution Assessment

Yes. The current-main solution uses the existing lifecycle/session contract instead of adding a new config or watchdog path, which is the narrowest maintainable fix for a stale `running` row after an aborted embedded turn.
