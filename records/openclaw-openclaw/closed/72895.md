## Summary

Keep open. Current main removes the older bundled runtime-deps staging direction and has source-checkout Baileys hotfixes, but the official WhatsApp npm/plugin install path can still install its own unverified `@whiskeysockets/baileys` copy via npm with scripts disabled, so the reported outbound media crash is not proven fixed for packaged/plugin installs.

## What This Changes

The issue requests hardening WhatsApp outbound media uploads so Baileys encrypted temp-file `ENOENT` failures cannot crash the Gateway, while preserving dispatcher handling, improving upload timeout control, and cleaning up stale runtime-deps selection behavior.

## Best Possible Solution

Make the WhatsApp plugin package/install path deterministically carry or verify the Baileys media-upload fix before WhatsApp runtime activation, preferably in a WhatsApp-owned package boundary or explicit install/update verifier, while keeping legacy runtime-deps roots as cleanup-only and deciding the upload-timeout config as a separate user-facing follow-up if needed.

## Reproduction Assessment

Yes. The issue has concrete CLI reproduction steps and crash logs, and current source gives a high-confidence package-path reproduction: an `@openclaw/whatsapp` npm install can materialize Baileys `7.0.0-rc.9` through the script-disabled plugin installer without applying the pnpm patch shown in the source checkout.

## Solution Assessment

No. The current source-checkout Baileys patch and legacy runtime-deps cleanup are useful, but they are not sufficient for the externalized WhatsApp plugin npm install path that now owns the Baileys dependency.
