## Summary

Keep open: the PR addresses a real memory CLI initialization gap that is not implemented on current main, but the submitted guard can skip the required bootstrap when capability fallback discovery returns a provider from manifests while the process-global registry is still cold. The PR also lacks the required changelog entry for a user-facing memory CLI fix, and #72937 overlaps the same root problem.

## What This Changes

The PR re-exports memory embedding provider registry helpers through the memory CLI host runtime, calls a new built-in provider registration guard before memory manager creation, and adds CLI tests for cold and already-populated registration paths.

## Best Possible Solution

Use one canonical cold-registry strategy for memory embeddings: seed built-in memory-core adapters before memory manager/provider initialization using registered-only state, or call the built-in registration helper unconditionally and rely on its existing registered-only filter. The final patch should include a capability-fallback regression test and an active changelog entry, while coordinating with #72937 so duplicate initialization logic does not land twice.

## Reproduction Assessment

Yes. A focused unit reproduction is a cold registered-provider set where capability fallback makes `listMemoryEmbeddingProviders()` return `local`; the PR guard exits and `registerMemoryEmbeddingProvider` is never called.

## Solution Assessment

No as submitted. The safer narrow fix is to call `registerBuiltInMemoryEmbeddingProviders()` unconditionally, or skip only after checking registered-only adapters, because capability fallback providers are not the same as registered adapters.
