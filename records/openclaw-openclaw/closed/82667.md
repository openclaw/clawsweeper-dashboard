## Summary

Keep open: the sanitizer gap is real and the PR has sufficient real-behavior proof, but current PR head still has a blocking chat_window parser false-positive that can delete a blank-separated user message beginning with a `#<digits>` token.

## What This Changes

The PR exports the Delivery hint, extends `stripInboundMetadata` to strip Delivery and chat_window envelope blocks, adds sanitizer regression tests, and updates an inbound-meta partial mock.

## Best Possible Solution

Land the sanitizer fix after tightening the chat_window row boundary so generated metadata is removed without deleting blank-separated user bodies, with a combined regression test for that case.

## Reproduction Assessment

Yes. Source inspection shows current main emits Delivery and chat_window blocks and routes historical user messages through `stripInboundMetadata`; the remaining PR defect is reproducible from the parser loop when a chat_window block is followed by a blank and user text beginning `#1234`.

## Solution Assessment

No, not as written. The central sanitizer is the right place to solve the issue, but the chat_window parser must stop at the body separator or match a stricter generated row shape before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve # messages after chat_window context:** `src/auto-reply/reply/strip-inbound-meta.ts:275-277`

The generated prefix is followed by a blank separator before the actual user body, so a real message beginning `#1234 ...` is dropped. Stop at the body separator or require the full generated row shape, and add that combined regression.
