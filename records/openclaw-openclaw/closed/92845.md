## What This Changes

The PR updates `docs/tools/browser-control.md` to state that the standalone loopback browser-control HTTP API requires `OPENCLAW_EAGER_BROWSER_CONTROL_SERVER=1` in the Gateway environment before endpoints listen.

## Best Possible Solution

Land the docs clarification as the narrow fix; any change to auto-start the standalone HTTP listener should be a separate product/security decision.

## Reproduction Assessment

Yes. Source and tests on current main show the standalone HTTP listener is lazy unless `OPENCLAW_EAGER_BROWSER_CONTROL_SERVER` is truthy, while the current reference docs omit that requirement.

## Solution Assessment

Yes. Updating the browser-control reference is the narrowest maintainable fix because runtime lazy startup is intentional in current tests and auto-starting the listener would be a separate behavior change.
