## Summary

Close as ClawHub scope. Current main has a bundled Xiaomi plugin for the standard endpoint, and the requested Token Plan onboarding, SGP endpoints, key prefix check, and V2.5 catalog are a provider-specific variant that the existing plugin manifest/provider SDK surfaces can express outside OpenClaw core.

## What This Changes

Request to add Xiaomi Token Plan as a dedicated onboarding/provider option with SGP endpoints, `tp-` key validation, and a V2.5 model catalog.

## Best Possible Solution

Keep OpenClaw core as-is and let Xiaomi Token Plan be implemented as a ClawHub provider plugin using the current manifest/provider SDK surfaces; if that work uncovers a missing generic plugin API, track that API gap separately.

## Reproduction Assessment

Not applicable as a feature request. The source/docs/test check has high confidence that current main lacks the requested Token Plan onboarding option, while the existing custom provider fallback and plugin-provider surfaces are present.

## Solution Assessment

Yes for this cleanup decision: a ClawHub provider plugin is the narrowest maintainable path because the requested behavior is a service-specific provider variant and current plugin APIs already cover provider ids, auth choices, base URLs, catalogs, and onboarding metadata.
