## Summary

Keep open: the bug and supplied proof are credible, but the WhatsApp suppression remains a compatibility-sensitive UX change because it disables composing for every unmentioned message-tool-only group turn before the run knows whether a visible message-tool reply will be sent.

## What This Changes

The PR changes shared typing-mode tool-start behavior and WhatsApp inbound dispatch so unmentioned message-tool-only group turns pass `suppressTyping`, with focused regression tests.

## Best Possible Solution

Merge a version that suppresses composing for silent background WhatsApp runs while preserving existing typing behavior for visible automatic, command, and message-tool replies unless channel owners explicitly accept the no-typing tradeoff.

## Reproduction Assessment

Yes, source plus supplied proof gives a high-confidence reproduction path: current main starts composing on tool start before renderable text, and WhatsApp wires composing to `sendPresenceUpdate("composing")`. I did not run a live WhatsApp session myself, but the PR includes before/after logs and screenshots for the same path.

## Solution Assessment

Unclear as submitted: the shared message-mode guard is narrow, but the WhatsApp `suppressTyping` gate is chosen before knowing whether a visible message-tool reply will be sent. The maintainable fix either needs a narrower visible-reply-aware design or explicit maintainer acceptance of the no-typing tradeoff.
