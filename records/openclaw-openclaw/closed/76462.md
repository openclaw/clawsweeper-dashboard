## Summary

Current main already fixes the reported release symptom: the `.jsonl.bak-*` files match the session-file repair path, and that path now preserves valid delivered transcript turns instead of rewriting the active file down to a shortened history. The fix is on `main` but I could not prove it is included in a shipped release after `v2026.5.2`.

## What This Changes

The issue reports that WebChat on `v2026.5.2` keeps the same session row but rewrites the active transcript so only recent turns remain while `.bak-*` backups still contain earlier history.

## Best Possible Solution

Keep the current repair-preservation implementation and ship it in the next release so `v2026.5.2` users get the fixed session-file repair behavior.

## Reproduction Assessment

No current-main failure path remains: source and tests show the repair path that creates `.bak-*` backups now preserves valid earlier transcript turns. The release-only mechanism is identifiable from the backup naming and repair call site rather than from a failing current-main run.

## Solution Assessment

Yes. Preserving valid transcript entries inside the existing session-file repair pass is the narrowest maintainable fix because that is the active path found that backs up and rewrites `.jsonl` transcripts without changing the session row.
