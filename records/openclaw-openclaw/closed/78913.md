## Summary

Current main still emits the reported post-spawn guidance for ordinary run-mode subagent spawns, while the docs and child-orchestrator prompt already describe push-based completion without recommending `sessions_yield` as the default wait path.

## What This Changes

The issue asks to stop the `sessions_spawn` accepted note from recommending `sessions_yield` as the default wait mechanism in push-based CLI/chat subagent flows.

## Best Possible Solution

Align the accepted-note text with the existing push-based subagent contract, leaving `sessions_yield` runtime behavior and the separate MCP failure report to their own issue.

## Reproduction Assessment

Yes, from source inspection: ordinary non-cron run-mode `sessions_spawn` resolves `SUBAGENT_SPAWN_ACCEPTED_NOTE` and returns the literal `sessions_yield` recommendation in `result.note`. I did not run a live gateway spawn in this read-only review.

## Solution Assessment

Yes, the narrowest maintainable fix is to change or context-gate the accepted-note guidance and update the locked test expectations. It should not change `sessions_yield` semantics or fold in the separate MCP/claude-cli failure tracked at https://github.com/openclaw/openclaw/issues/77426.
