## Summary

Keep open. Current main and v2026.5.12 still allow unmatched exec approval follow-up text to pass through the direct external fallback after session resume failure, and the linked open issues cover prefix/retry or broader late-output behavior rather than this source-like payload guard.

## What This Changes

The issue reports that exec approval follow-up recovery can send raw internal handler/source-like text to Telegram after session resume fails.

## Best Possible Solution

Add a fail-closed guard at the direct approval follow-up delivery boundary for unmatched/source-like text, cover the reported snippets with regression tests, and separately trace the producer of unsafe `resultText`.

## Reproduction Assessment

Yes, source-level: call `sendExecApprovalFollowup` with a deliverable Telegram route, a session key, a rejected `callGatewayTool("agent")`, and source-like unmatched `resultText`; current main parses it as `other` and can send sanitized raw text through direct fallback. I did not establish a live Telegram reproduction in the read-only checkout.

## Solution Assessment

Yes for containment: the narrow maintainable boundary is `formatDirectExecApprovalFollowupText` or `sendDirectFollowupFallback`, where unverified `other` text can be dropped or replaced before external delivery. The remaining producer root cause should be traced separately.
