Full review comments:

- **[P1] Keep groupScope inside the binding session contract:** `src/config/types.agents.ts:41-43`
  - body: Current main already has a strict `bindings[].session` object for per-binding session overrides, including `dmScope`. Adding `bindings[].groupScope` as a top-level field creates a second public config location and conflicts with current schema/help, so this should be rebased into `bindings[].session` or another maintainer-approved session target contract before merge.
  - confidence: 0.92
- **[P1] Propagate groupScope to outbound session routes:** `src/routing/resolve-route.ts:673-674`
  - body: The new scope is only threaded through inbound route resolution. Outbound paths that omit an explicit `sessionKey` still call `buildOutboundBaseSessionKey()` with only global `cfg.session.dmScope`, so messages mirrored from a merged group can fork back into a per-channel session instead of the configured main session.
  - confidence: 0.87
- **[P2] Preserve groupScope when Telegram topics override agents:** `extensions/telegram/src/conversation-route.ts:65-72`
  - body: Telegram topic-agent overrides rebuild the route's `sessionKey` with `buildAgentSessionKey()` and only pass `dmScope` and `identityLinks`. A matched Telegram group/topic binding configured for main would still be rewritten to a per-topic session in that path.
  - confidence: 0.8
- **[P2] Route proactive IRC sends through joined gateway state:** `extensions/irc/src/send.ts:72-74`
  - body: The bundled IRC registry only sees monitor clients in the current Node process, while ordinary CLI-originated sends can run elsewhere and still fall back to a transient unjoined connection. That delivery behavior should be handled in the dedicated IRC proactive-send PR or reworked here with proof.
  - confidence: 0.82
