Keep this PR open: the scoped labeled screenshot direction is useful and maintainer-supported, but the current patch still has a source-reproducible overlay placement bug that can hide top-edge badge text in scrolled viewport and element-clipped captures.

## What This Changes

The PR extends the browser plugin labeled screenshot path so Playwright-backed full-page, ref, and element captures draw labels and return optional per-ref annotation boxes.

## Best Possible Solution

Keep the scoped-label semantics and additive annotations, but repair badge headroom to use captured-image coordinates before merging.

## Reproduction Assessment

Yes. Source inspection shows the old shipped labels path ignores fullPage/ref/element, and the remaining review finding is source-reproducible because badge flipping uses document-space y while viewport/element captures can start below document y=0.

## Solution Assessment

No, not yet. The PR is in the right browser plugin boundary and fixes the shipped silent fallback, but the overlay helper should calculate badge headroom in capture-space before this becomes the best implementation.

- **[P2] Flip badges using capture-space y:** `extensions/browser/src/browser/screenshot-annotate.ts:224-225`
  - body: `overlayItems` are document-space, but this headroom check also uses raw document y.
