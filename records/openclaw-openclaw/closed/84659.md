## Summary

Keep open: current main still has an independent 8,000-character live tool-result event sanitizer that can clip an 8.7KB read result even though the read wrapper and model-context cap are higher or configurable, and the linked PR does not prove this read/live-display path is fixed.

## What This Changes

The issue reports that medium-sized `read` tool output started appearing truncated after the 2026.5.18 update.

## Best Possible Solution

Pin the failing surface with a live 2026.5.18-or-newer repro; if it is the live/display sanitizer, align that cap with `toolResultMaxChars` or route clipped output through explicit full-output retrieval metadata.

## Reproduction Assessment

No, not for the exact reporter claim. Source inspection gives a medium-confidence path for live/display truncation because `sanitizeToolResult` clips text over 8,000 characters before tool-result events are emitted.

## Solution Assessment

Unclear. Aligning the live sanitizer with `toolResultMaxChars` is the narrow fix if live events are the failing surface, but maintainers should decide whether the broader full-message reader flow should own recovery for clipped UI/history content.
