## Summary

Current main still does not actively detect the Mattermost connected-but-deaf WebSocket condition. The Mattermost plugin marks the socket connected and polls the bot `update_at` timestamp for the disable/enable stale-session case, but the WebSocket seam and tests still lack ping/pong keepalive or provider-proven `lastTransportActivityAt` reporting. The open PR #57621 remains the focused implementation candidate for this issue and the related open reports #44160 and #51104.

## Best Possible Solution

Keep this issue open and review or land #57621, or an equivalent Mattermost-owned fix. The fix should extend `extensions/mattermost/src/mattermost/monitor-websocket.ts` with WebSocket ping/pong keepalive that terminates deaf sockets so `runWithReconnect` can reconnect, preserve the existing `getBotUpdateAt` stale-session check, and add regression coverage in `extensions/mattermost/src/mattermost/monitor-websocket.test.ts`. If maintainers choose the generic health-monitor route instead, Mattermost should publish `lastTransportActivityAt` only from real transport liveness, not ordinary app-message silence.
