## Review Findings

Overall confidence: 0.88

Full review comments:

- **[P1] Reuse the existing channel prompt contract:** `src/config/types.channels.ts:85`
  - body: Current main already supports per-channel `systemPrompt` on Slack, Discord, Telegram, and other channel configs, then passes that through `GroupSystemPrompt` into `extraSystemPrompt`. Adding a new top-level `systemPromptByChannel` means the same channel can now receive two standing prompts with different precedence and ID semantics. Please extend or document the existing per-channel prompt surface instead, or make file references part of that contract with explicit precedence.
  - confidence: 0.89
- **[P1] Constrain prompt file reads before injection:** `src/agents/channel-system-prompt.runtime.ts:30`
  - body: This reads every configured path synchronously and fully before the run is queued, including absolute and home paths. That bypasses existing bootstrap prompt caps and can inject secrets or very large files into provider-bound system prompts, or stall the event loop on slow filesystems. Add a trusted-root policy, size cap, and secret/path guard before merging a file-backed prompt surface.
  - confidence: 0.87
- **[P2] Match the documented path expansion to the code:** `docs/channels/channel-system-prompts.md:63`
  - body: The docs promise that `$HOME` expands before resolution, but `expandHome()` only handles `~` and `~/`. A config value like `$HOME/prompts/x.md` will be treated as a workspace-relative literal path, so either implement and test `$HOME` expansion or remove that claim.
  - confidence: 0.96
- **[P3] Remove the release-owned changelog edit:** `CHANGELOG.md:48`
  - body: Root policy keeps `CHANGELOG.md` release-owned for generated releases. The PR body already carries the release-note context, so remove this normal feature PR's changelog line and leave release generation to the release workflow.
  - confidence: 0.95
