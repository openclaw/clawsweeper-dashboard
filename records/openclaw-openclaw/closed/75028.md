## Summary

Current `main` already implements the reported container PID-reuse fix in the shared bundled runtime-deps filesystem lock helper, including the `.openclaw-runtime-mirror.lock` path named in the report. The fix is covered by a Docker PID-reuse regression test and is documented in the current changelog, but it is not proven shipped in the latest `v2026.4.27` tag.

## What This Changes

The issue requests reclaiming stale bundled plugin runtime-deps locks in containers by validating lock ownership with more than PID alone.

## Best Possible Solution

Keep the current shared lock-owner starttime validation and carry it into the next release; operators already stuck on a pre-fix lock can remove that stale lock directory once before restarting on a fixed build.

## Reproduction Assessment

Yes. The issue includes concrete container restart steps and current-main regression coverage models the same same-PID/different-starttime condition; tests were inspected rather than run because this review is read-only.

## Solution Assessment

Yes. Recording `/proc` starttime in the shared lock helper and requiring it to match the live PID is the narrow maintainable fix for Linux container PID reuse, and it covers both install and mirror locks that use the helper.
