## Summary

Keep open. Current main still does not implement per-agent suspension, and this PR remains a relevant implementation candidate, but the branch is not merge-ready because the promised suspension semantics still miss cron/heartbeat execution, the all-disabled routing fallback case, and parseable JSON output for the new commands.

## What This Changes

The PR adds an optional `agents.list[].enabled` flag, `openclaw agents disable/enable` commands, disabled status in agent listing/JSON output, schema/docs/generated baseline/changelog updates, and routing/config regression tests.

## Best Possible Solution

Land a refreshed implementation that defines a typed, schema-validated `agents.list[].enabled?: boolean` contract, keeps disabled agents visible to management operations, excludes them consistently from routing and background execution, preserves machine-readable CLI output, updates generated docs, and covers config preservation plus scheduler suspension with regression tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Honor disabled agents in cron and heartbeat:** `src/config/types.agents.ts:62-64`
  - body: The new contract documents `enabled: false` as stopping cron and heartbeat, but the PR does not update `src/gateway/server-cron.ts` or `src/infra/heartbeat-runner.ts`; both current paths derive work from raw `cfg.agents.list`, so background jobs for a disabled agent can keep running.
  - confidence: 0.9
- **[P2] Route empty enabled sets to the fallback:** `src/routing/resolve-route.ts:140-142`
  - body: After disabled agents are skipped from the lookup, a config where every listed agent is disabled leaves the lookup map empty. `pickFirstExistingAgentId` then returns the requested id verbatim, so a binding can still route to the disabled agent instead of the fallback.
  - confidence: 0.87
- **[P2] Keep disable and enable JSON output parseable:** `src/commands/agents.commands.disable.ts:58-60`
  - body: `agents disable --json` calls `logConfigUpdated(runtime)` before `writeRuntimeJson`, which writes a human line to stdout ahead of the JSON payload. Guard that log in JSON mode, and apply the same fix to the enable command.
  - confidence: 0.88
