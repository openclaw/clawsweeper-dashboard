## Summary

Keep open. Current main still has a source-level Telegram typing gap: `thinking` mode is documented to start typing on the first reasoning delta, but `signalReasoningDelta` still returns before any renderable text exists, so long pre-first-output reasoning can leave Telegram without a keepalive. Related broad and transport issues are either separate or point back to this narrower follow-up rather than superseding it.

## What This Changes

The issue reports that Telegram typing keepalives no longer remain visible during active long-running agent work.

## Best Possible Solution

Fix the shared typing signaler so reasoning deltas in `thinking` mode start and refresh the existing typing loop before the first text delta, keep the current tool-start/update refresh behavior, and prove the visible Telegram path after the patch.

## Reproduction Assessment

Yes, source-level: create a `thinking` mode typing signaler and call `signalReasoningDelta` before any text delta; current main returns without starting the typing loop despite the documented contract. No live Telegram DM reproduction was run in this read-only review.

## Solution Assessment

Yes for the narrow repair direction: remove the pre-text `hasRenderableText` gate from `signalReasoningDelta`, keep tool-start/update TTL refreshes intact, and update focused regression coverage. Broader receipt or heartbeat features should wait unless live proof shows a separate remaining gap.
