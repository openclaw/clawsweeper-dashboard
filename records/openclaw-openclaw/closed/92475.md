## Label Justifications

- `P1`: The report describes a real-user regression that continuously ran full-context heartbeat turns, exhausted provider windows, and blocked interactive work.
- `impact:session-state`: The heartbeat storm repeatedly executed large heartbeat session contexts outside the configured cadence.
- `impact:auth-provider`: The issue concerns Codex subscription-limit handling, auth-profile cooldown state, and provider fallback behavior.

## Summary

Current main and the latest stable release now cover the reported heartbeat wake feedback loop and Codex hard-quota retry storm: event wakes are gated by a shared cooldown path before `runOnce`, and Codex usage-limit failures can mark the OpenAI auth profile blocked until the provider reset.

## What This Changes

The issue requests that heartbeat event wakes respect configured cadence and that Codex subscription-limit failures stop being retried continuously during automated heartbeat runs.

## Best Possible Solution

Keep the shipped `v2026.6.6` scheduler and Codex reset-blocking implementation; route broader cron/provider key-pool policy to the existing canonical issues.

## Reproduction Assessment

Yes at source level: current regression tests encode the repeated exec-event wake loop, and the Codex quota path is covered by reset-enrichment and auth-profile blocked-until tests; the exact overnight production sequence was not live-replayed.

## Solution Assessment

Yes for the central issue: heartbeat cadence is fixed at the runner cooldown boundary, while Codex reset handling lives in the Codex plugin plus shared auth-profile usage state; broader quota/circuit-breaker expansion remains tracked by https://github.com/openclaw/openclaw/issues/14376 and https://github.com/openclaw/openclaw/issues/28847.
