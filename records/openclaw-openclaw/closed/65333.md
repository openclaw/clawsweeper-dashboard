## Summary

Keep open; the PR is not merge-ready because it changes a generated-token path into no-auth mode even though that path is reached when install policy requires token auth. Current main, tests, and docs consistently keep generated gateway credentials in token mode and describe `auth.mode="none"` as an explicit trusted-loopback opt-out.

## What This Changes

The PR changes gateway install-token persistence to default missing `gateway.auth.mode` to `"none"` instead of `"token"`, and adds a unit test expecting that persisted mode.

## Best Possible Solution

Keep generated-token persistence in token mode. If maintainers want a lower-friction local no-auth setup, add a separate exposure-aware policy that proves loopback-only/private-ingress safety, refuses or warns for Tailscale/non-loopback/daemon paths, and updates docs plus regression coverage across bind and Tailscale modes.

## Reproduction Assessment

Yes for the introduced behavior: call `resolveGatewayInstallToken()` with empty `gateway.auth`, `autoGenerateWhenMissing: true`, and `persistGeneratedToken: true`, then inspect the mocked `replaceConfigFile` payload. I did not run tests because this review is read-only, but the current unit seam and PR diff make the behavior directly checkable.

## Solution Assessment

No. Persisting `mode: "none"` in the generated-token branch is not the narrowest maintainable fix because this branch runs after install policy has decided a token is needed; a safer solution would keep token mode unless a separate exposure-aware local-only policy explicitly opts into no-auth.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve token auth for generated install tokens:** `src/commands/gateway-install-token.ts:62`
  - body: This path still saves a generated token, but changing the fallback mode to `"none"` makes that token unused by later gateway auth. The branch is reached after install policy decides token auth is needed, so a daemon install can end up with a config that looks credentialed while runtime authorization short-circuits as no-auth. Keep the fallback as `"token"` unless a separate exposure-aware policy has explicitly opted into no-auth.
  - confidence: 0.92
