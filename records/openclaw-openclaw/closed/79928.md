## Summary

Keep open: the linked bug is valid, but this PR’s `git checkout --detach -- <ref>` form breaks ordinary valid refs and would regress git plugin installs before it fixes the flag-parsing issue.

## What This Changes

This PR inserts `--` before the parsed git plugin ref in the checkout argv and updates `src/plugins/git-install.test.ts` for option-like refs.

## Best Possible Solution

Use a Git command form that treats the user value as a ref while preserving normal tag and branch installs, then cover both valid refs and option-like invalid refs.

## Reproduction Assessment

Yes. A disposable real Git repo reproduces the current-main option-like-ref bug and also shows the PR command rejects a normal valid tag ref.

## Solution Assessment

No. Adding `--` to `git checkout --detach` uses checkout’s pathspec separator; a safer fix is a ref-aware command such as the verified `git switch --detach -- <ref>` shape or another live-proven equivalent.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.97

Full review comments:

- **[P1] Preserve valid ref checkouts:** `src/plugins/git-install.ts:290`
  - body: Git treats arguments after `--` in `git checkout` as pathspecs, not the branch/ref for `--detach`. In a real repo, `git checkout --detach -- v1.2.3` exits 128 with `--detach does not take a path argument`, so every git plugin install that pins a normal tag or branch would fail; use a ref-aware form such as the verified `git switch --detach -- <ref>` shape instead.
  - confidence: 0.98
