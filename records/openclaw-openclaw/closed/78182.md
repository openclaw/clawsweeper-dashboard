## Summary

Keep open: current main still routes `/model default` through normal model-ref and fuzzy model resolution, so it never reaches the existing session-reset path as an `isDefault` selection.

## What This Changes

Request `/model default` and possibly `/model primary` to clear the session model override and return to the configured default model.

## Best Possible Solution

Add an explicit `/model default` reset path in model directive resolution that reuses the existing `isDefault` session override clearing, documents the command, and leaves parent-session inheritance for a separate follow-up.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: `/model default` is parsed as raw model text and then resolved as a normal model/fuzzy fragment. I did not run a live chat session because this review was read-only, but the current-main path is clear.

## Solution Assessment

Mostly yes: handling `default` as a reserved reset token before normal model resolution is the narrowest maintainable fix. The patch should add regression coverage and define whether `primary` is also supported, without changing parent-session inheritance in the same pass.

In the model directive resolver, recognize a reserved reset token for `default` and return a `ModelDirectiveSelection` for the configured `defaultProvider/defaultModel` with `isDefault: true` so the existing `applyModelOverrideToSessionEntry` logic clears `providerOverride` and `modelOverride`.
