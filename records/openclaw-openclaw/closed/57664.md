## Summary

Current `main` already suppresses the noisy `tools.profile` startup warning for built-in profile entries that are known core tools but unavailable in the active runtime, while still warning on explicit extra entries and real unknown tool names.

## Evidence

- **Default profile warnings are suppressed for built-in profile entries:** `buildDefaultToolPolicyPipelineSteps` accepts `profileUnavailableCoreWarningAllowlist` / `providerProfileUnavailableCoreWarningAllowlist`, and `applyToolPolicyPipeline` filters known core-tool unknowns against that allowlist before emitting a warning.
  - file: [src/agents/tool-policy-pipeline.ts:39](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/src/agents/tool-policy-pipeline.ts#L39)
- **Main tool-resolution paths wire the profile allowlist into suppression:** Both the main Pi tool path and embedded runner effective policy pass `profilePolicy?.allow` into `profileUnavailableCoreWarningAllowlist`, so baseline `tools.profile` entries do not warn just because a given runtime/provider/model does not expose them.
  - file: [src/agents/pi-tools.ts:679](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/src/agents/pi-tools.ts#L679)
- **Regression tests cover the reported startup-warning case:** `suppresses built-in profile warnings for unavailable gated core tools` and `default profile steps suppress unavailable baseline profile entries` assert no warning for default `tools.profile (coding)`, while nearby tests still require warnings for explicit extra allowlist entries like `browser` or `apply_patch` in `tools.allow`.
  - file: [src/agents/tool-policy-pipeline.test.ts:79](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/src/agents/tool-policy-pipeline.test.ts#L79)
- **Bundled-pass false positives are also blocked:** The embedded-runner bundled-tool pass now force-suppresses unavailable-core warnings, and regression tests verify valid core tool names like `read` / `exec` do not log as unknown there while genuine typos still do.
  - file: [src/agents/pi-embedded-runner/effective-tool-policy.ts:156](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/src/agents/pi-embedded-runner/effective-tool-policy.ts#L156)
- **Changelog records the warning clarification:** The changelog explicitly notes: `Agents/tool warnings: distinguish gated core tools like apply_patch from plugin-only unknown entries in tools.profile warnings`.
  - file: [CHANGELOG.md:2589](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/CHANGELOG.md#L2589)

## Close Comment

Closing this as implemented after Codex review.

Current `main` already suppresses the noisy `tools.profile` startup warning for built-in profile entries that are known core tools but unavailable in the active runtime, while still warning on explicit extra entries and real unknown tool names.

What I checked:

- **Default profile warnings are suppressed for built-in profile entries:** `buildDefaultToolPolicyPipelineSteps` accepts `profileUnavailableCoreWarningAllowlist` / `providerProfileUnavailableCoreWarningAllowlist`, and `applyToolPolicyPipeline` filters known core-tool unknowns against that allowlist before emitting a warning. (`src/agents/tool-policy-pipeline.ts:39`)
- **Main tool-resolution paths wire the profile allowlist into suppression:** Both the main Pi tool path and embedded runner effective policy pass `profilePolicy?.allow` into `profileUnavailableCoreWarningAllowlist`, so baseline `tools.profile` entries do not warn just because a given runtime/provider/model does not expose them. (`src/agents/pi-tools.ts:679`)
- **Regression tests cover the reported startup-warning case:** `suppresses built-in profile warnings for unavailable gated core tools` and `default profile steps suppress unavailable baseline profile entries` assert no warning for default `tools.profile (coding)`, while nearby tests still require warnings for explicit extra allowlist entries like `browser` or `apply_patch` in `tools.allow`. (`src/agents/tool-policy-pipeline.test.ts:79`)
- **Bundled-pass false positives are also blocked:** The embedded-runner bundled-tool pass now force-suppresses unavailable-core warnings, and regression tests verify valid core tool names like `read` / `exec` do not log as unknown there while genuine typos still do. (`src/agents/pi-embedded-runner/effective-tool-policy.ts:156`)
- **Changelog records the warning clarification:** The changelog explicitly notes: `Agents/tool warnings: distinguish gated core tools like apply_patch from plugin-only unknown entries in tools.profile warnings`. (`CHANGELOG.md:2589`)

So I’m closing this as already implemented rather than keeping a duplicate issue open.
