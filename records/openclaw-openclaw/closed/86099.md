## Summary

Keep open: current main has the lower-level vLLM Qwen payload mapper, but it does not yet expose a vLLM-owned binary thinking profile for Qwen-style models, so the central UI/slash-command mismatch remains.

## What This Changes

Request model-aware vLLM and Pi thinking controls that expose Qwen-style off/on choices, persist the durable thinking state, map it to vLLM Qwen payload fields, and keep `/reasoning` separate from `/think`.

## Best Possible Solution

Add a vLLM-owned thinking profile that exposes `off` and labeled `on` for explicitly configured Qwen-style models, reuses the existing stored thinking level/default machinery, preserves the current vLLM payload mapper, and covers Pi/session/UI/docs behavior with focused tests.

## Reproduction Assessment

Yes, at source level: current vLLM has Qwen request shaping but no vLLM provider thinking profile, so session/control surfaces still derive choices from the generic profile path. A live vLLM run was not performed.

## Solution Assessment

Unclear until maintainers choose the profile-selection contract. The requested direction matches the existing provider-profile architecture, but the safest implementation should avoid brittle Qwen model-id guesses when explicit model params already carry the stronger fact.
