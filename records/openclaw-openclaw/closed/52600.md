## Summary

Current main already solves the PR's central `sessions_send` serialization problem by routing session sends onto per-target-session nested lanes instead of the single shared `CommandLane.Nested` lane. The command-queue drain/restart helpers discussed in the PR are also present on main, while the PR's lazy resolver API is not needed for the current `sessions_send` path. Remaining cron-specific nested-lane work is tracked separately by #54314.

## Best Possible Solution

Close this PR and keep the current per-session nested lane implementation on main for `sessions_send`. Any remaining cron nested-lane concurrency concern should continue under the narrower #54314 thread rather than reviving this lazy-loading command-queue PR.
