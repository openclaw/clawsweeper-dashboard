## Summary

Close as a duplicate of the existing Feishu P2P direct-thread tracking path. Current main still has the reported P2P Topic reply gap, but the same user-visible problem is already tracked by open issue #38806 and active repair PRs #73972/#74001; this issue’s useful extra detail is that webhook delivery may require a Feishu message-detail lookup to recover `thread_id`.

## What This Changes

Feishu P2P Topic webhook handling should recover thread context, using `GET /im/v1/messages/{message_id}` when needed, so bot replies stay inside the originating Topic thread.

## Best Possible Solution

Keep #38806 as the canonical bug, consolidate on one of the active Feishu PRs, and make the final Feishu-owned fix recover `thread_id` from message details when webhook events lack it while preserving the existing group/topic and root-id-only quote behavior.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: a Feishu `p2p` event without usable thread metadata is parsed as a direct message, no current-message detail lookup runs, and outbound dispatch strips reply metadata before sending.

## Solution Assessment

Yes for maintainer cleanup: closing this duplicate is the best tracking outcome because #38806 already owns the remaining P2P thread reply problem. The code solution itself should be a narrow Feishu plugin change coordinated with #73972/#74001, not a new core routing change.
