## Summary

Close as duplicate/superseded by the open canonical shared-session tracker. The Slack-specific problem is still valid on current main, but #19929 now tracks the remaining product/security design for shared sessions across selected group/channel/thread chats, and the linked implementation PRs were closed unmerged after threat-model concerns.

## What This Changes

Request to add opt-in session routing so Slack thread replies can share a parent or main session instead of always using thread-suffixed session keys.

## Best Possible Solution

Use #19929 as the single design and implementation tracker, keeping isolated defaults while designing an explicit opt-in shared-session model with channel/thread membership, trust-boundary documentation, history keys independent of shared session keys, first-seen thread tracking, reply routing back to the originating Slack surface, and focused routing/config/docs tests.

## Reproduction Assessment

Yes. A static reproduction is high confidence: Slack thread replies flow through resolveSlackRoutingContext into resolveThreadSessionKeys without useSuffix=false, and current tests assert the resulting :thread:<thread_ts> session key.

## Solution Assessment

No for the original broad boolean as a standalone fix. The safer path is the canonical #19929 design: explicit selected channel/thread grouping with documented trust boundaries rather than a Slack-only switch that can accidentally merge contexts across users or channels.
