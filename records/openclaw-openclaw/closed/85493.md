## Label Justifications

- `P2`: This is a focused command-menu bug fix with limited blast radius, but it affects real Telegram native command registration behavior.
- `merge-risk: 🚨 compatibility`: The PR changes the shared native-command registry output and can remove alias specs from existing non-Telegram native-command consumers.
- `merge-risk: 🚨 message-delivery`: Dropping aliases from the list used for native command handlers can stop alias commands such as `/side` from reaching the intended handler on native surfaces or text-disabled configs.

## Summary

Keep open: the reported Telegram menu-slot bug is real, but this branch suppresses aliases in the shared native-command registry instead of only in Telegram's menu payload, so it can remove native aliases from other providers and from Telegram handler registration; it also still lacks live Telegram proof.

## What This Changes

The PR changes native command spec generation so `nativeAliases` remain lookup aliases but are not emitted as separate native command menu specs.

## Best Possible Solution

Keep native aliases in the shared registry and handler list, then filter alias specs only when building Telegram's `setMyCommands` menu payload, with sibling-provider tests and live Telegram proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main expands `nativeAliases` into native specs, Telegram sends those specs to `setMyCommands`, and Telegram caps that payload at 100 commands. I did not run a live Telegram bot in this read-only review.

## Solution Assessment

No, this is not the best shape yet. The maintainable fix is Telegram-menu-local alias filtering while preserving shared native aliases for handlers and sibling providers.
