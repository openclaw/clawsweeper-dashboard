## Label Justifications

- `P1`: The reported symptom is an urgent cron/agent workflow stall with timeouts and failed deliveries, even though the maintained code path is already fixed.
- `impact:crash-loop`: The report describes gateway unresponsiveness after cron jobs pile up and require restart-style recovery.
- `impact:message-loss`: The report describes cron agent messages failing or starving in queues rather than reaching their destinations.

## Summary

Current main and the latest shipped release already route isolated cron agent turns through cron-owned session keys and the dedicated cron nested lane, so the reported shared `agent:main:main` gridlock path is not present in the maintained implementation.

## What This Changes

The issue requests that isolated cron `agentTurn` jobs stop sharing `agent:main:main` and instead use unique sessions or lanes so cron runs are not blocked by active user dialog.

## Best Possible Solution

Keep the shipped cron-owned session and `cron-nested` lane behavior; any new report should include versioned logs showing a current build still emitting `agent:main:main` for isolated cron runs.

## Reproduction Assessment

No: current main source and regression tests show isolated cron agent turns use `agent:<id>:cron:<jobId>:run:<sessionId>` plus `cron-nested`, not `agent:main:main`. The report does not include a versioned current-main reproduction log that contradicts that path.

## Solution Assessment

Yes: the maintained solution is the existing cron-owned run session plus dedicated `cron-nested` execution lane, which addresses both session sharing and inner-lane concurrency without adding a new agent id or config mode.
