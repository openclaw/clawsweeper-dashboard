## Summary

Keep open. Current main still has the reported cron update atomicity gap, and an open PR (#74068) explicitly closes this issue with a targeted gateway validation patch, so the right path is to review and land or close that PR rather than closing this issue early.

## What This Changes

Invalid `--cron` expressions should be rejected before mutating disabled cron jobs, and a failed enable should not leave an invalid schedule enabled.

## Best Possible Solution

Cron add/update should validate the effective cron schedule before committing user-requested mutations, ideally with service-level staging or equivalent gateway pre-validation, plus regression coverage for disabled invalid edits and failed enable rollback.
