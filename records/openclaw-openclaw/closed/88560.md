## Summary

Keep open: current main still has a provider/model normalization path that can carry `anthropic/...` as the model id for provider `anthropic`, while the docs show `provider/model` allowlist keys are supported and the related reports are still open rather than a merged canonical fix.

## What This Changes

The issue reports a v2026.5.28 regression where provider-prefixed Anthropic model keys in `agents.defaults.models` remain prefixed during provider lookup, causing doubled-prefix model errors and broken fallback attempts.

## Best Possible Solution

Normalize own-provider-prefixed model ids before provider lookup and error-hint generation, then add regression coverage that documented Anthropic `provider/model` allowlist keys and each fallback candidate resolve independently.

## Reproduction Assessment

Yes from source inspection, though I did not execute a live turn: current main can keep an already-prefixed Anthropic id under provider `anthropic` and later formats `provider/modelId`. The cross-candidate leak is log-supported and should be locked down with a focused regression test.

## Solution Assessment

Yes, the narrow maintainable fix is in model-id normalization/resolution plus error-hint formatting, not a new config workaround. The repair should also prove fallback candidates do not reuse a prior candidate's model id.
