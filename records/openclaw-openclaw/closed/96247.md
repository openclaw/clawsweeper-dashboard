## Summary

Keep open: current head addresses the earlier replay finding and is a focused candidate fix for the queued mid-batch duplicate-send path, but it still needs maintainer acceptance of the fail-closed post-send recovery semantics before landing.

## What This Changes

The PR changes outbound queue cleanup and recovery so mid-batch failures with send evidence become `unknown_after_send`, and `unknown_after_send` entries are not fully replayed after a `not_sent` reconciliation verdict.

## Best Possible Solution

Land this focused Path 2 fix after maintainers accept fail-closed post-send recovery semantics, while keeping the broader duplicate Telegram work under https://github.com/openclaw/openclaw/issues/96242 and https://github.com/openclaw/openclaw/pull/92274.

## Reproduction Assessment

Yes, at source level. Current main can leave a mid-batch post-send failure in the wrong queue state and the PR body supplies production gateway logs plus SQLite evidence; no live Telegram reproduction was run in this read-only review.

## Solution Assessment

Yes, with a maintainer-owned semantics choice. Marking only failures with send evidence as `unknown_after_send` and refusing full replay for post-send evidence is the narrow Path 2 mitigation; per-payload selective replay would be a larger delivery-ledger design.
