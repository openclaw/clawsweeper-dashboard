## Summary

Current main and the latest release already cover the Kimi Anthropic stream-order failure through the boundary-aware Anthropic Messages transport and safe invalid-stream error handling; the linked canonical bug was also closed as covered by that path. The PR branch is useful context but should not be merged because its subscribe-level synthetic close still has race and stale-end correlation bugs.

## What This Changes

The PR adds assistant-message lifecycle state to the embedded Pi subscription handler, synthesizes a `message_end` when a new assistant `message_start` arrives mid-message, resets that state on compaction retry, and adds Kimi Coding regression tests.

## Best Possible Solution

Keep the released boundary-aware transport and invalid-stream guard path, and close this PR as superseded; any future Kimi regression should be reported against the current transport path with fresh logs rather than merging subscribe-level synthetic stream surgery.

## Reproduction Assessment

Yes for the original failure class: #57753 and #57523 include concrete Kimi `anthropic-messages` reproduction steps and logs for the SDK stream-order error on v2026.3.28-era builds. I did not run live Kimi credentials in this read-only review, but current source and release checks cover the code path that supersedes the PR.

## Solution Assessment

No, the PR branch is not the best way to solve it now. Current main’s boundary-aware per-request transport addresses the lower-level stream boundary problem more directly, while the PR’s synthetic close can race async finalization and correlates stale ends by visible text.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Await synthetic close before resetting state:** `src/agents/pi-embedded-subscribe.handlers.messages.ts:224`
  - body: `handleMessageEnd()` can return a promise when `blockReplyBreak === "message_end"` and block-reply flush callbacks are async. Dispatching it with `void Promise.resolve(...)` lets the synthetic close finalize after the next message has started, which can clear the new `deltaBuffer` and flip `inAssistantMessage` false mid-stream.
  - confidence: 0.88
- **[P2] Do not match stale ends by visible text alone:** `src/agents/pi-embedded-subscribe.handlers.messages.ts:503`
  - body: The stale-end filter stores normalized visible text and skips the first later end with that same text. If overlapping messages share text, or the newer message ends before the delayed stale one, a valid current `message_end` can be dropped and the stale one processed as current.
  - confidence: 0.82
- **[P2] Track synthetic closes without visible text:** `src/agents/pi-embedded-subscribe.handlers.messages.ts:202`
  - body: The PR only records a stale synthetic close when `staleVisibleText` is non-empty. Media-only, directive-only, or otherwise empty visible assistant messages can still receive a delayed real `message_end` that is treated as live and closes a newer message.
  - confidence: 0.76
