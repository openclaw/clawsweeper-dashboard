## Summary

Current `main` already addresses this failure mode by resolving the requested subagent model, storing it on the child session as a provider/model override before the first `agent` call, and then having the agent runtime select stored session overrides before defaulting to the agent primary model. The linked closed PR #74730 identified a real forwarding concern, but the maintained implementation now solves the user-visible problem through the session override path instead of passing model params on the gateway `agent` call.

## What This Changes

The issue reports that `sessions_spawn` ignores its `model` parameter and runs all subagents on the configured primary model.

## Best Possible Solution

Keep the current session-override based implementation: it routes `sessions_spawn.model` through the existing per-session model contract, keeps status consistent, and avoids needing to widen the gateway `agent` call's model-override authorization path.

## Reproduction Assessment

Yes. The source-level reproduction is to configure a primary model, call `sessions_spawn` with a different `provider/model`, and inspect the child session/runtime model; current tests encode the key guard by asserting the child-session model is persisted before `gateway:agent`.

## Solution Assessment

Yes. Persisting the resolved model as the child session override before launch is the narrowest maintainable fix because `agent-command` already resolves stored session overrides and the gateway `agent` method can remain least-privilege.
