## Summary

Current main still has the active-run-gated TUI terminal transition reported here. The shipped streaming watchdog mitigates active-run lost-final cases, but it does not cover unbound/local BTW runs that set the footer to streaming without owning activeChatRunId, nor inactive finals where no run remains in flight.

## Best Possible Solution

Keep this issue open and implement a narrow TUI state-machine fix in src/tui/tui-event-handlers.ts: after terminal cleanup, set the terminal activity status when the finishing run was active or when there is no activeChatRunId and no tracked session run remains. Route the local BTW empty-final branch through the same recovery helper, and add focused tests for inactive final clearing, local BTW delta plus empty final, concurrent-run preservation, and terminal error/aborted preservation.

## Risks / Open Questions

- A broad fallback could incorrectly clear streaming while another run is legitimately active; the fix should only transition after terminal cleanup when there is no activeChatRunId and no tracked sessionRuns entry remains.
- Terminal error and aborted states need ownership-aware handling too, so the recovery path preserves error/aborted status instead of blindly converting everything to idle.
