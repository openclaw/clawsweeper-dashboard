## Label Justifications

- `impact:other`: The issue affects agent reliability when managing many cron jobs because oversized tool results can hide list entries from the model.

## Summary

Keep open: current main and latest release still lack an opt-in compact cron list response, and the related implementation PR was closed unmerged, so the focused core tool UX/API request remains real but needs maintainer approval before repair work.

## What This Changes

Add an optional `compact: true` mode to the agent-facing `cron(action="list")` path so large cron lists can return lightweight job summaries instead of full CronJob objects.

## Best Possible Solution

If maintainers accept the UX direction, add an opt-in compact projection at the protocol, service, gateway, and agent-tool boundary with focused tests, preserving the default full list and using cron.get for details.

## Reproduction Assessment

Partially. Source inspection confirms current main and v2026.6.6 return full CronJob objects and expose no compact option, but a seeded 20-30 job runtime proof would still be needed to quantify the truncation threshold.

## Solution Assessment

Unclear until maintainers approve the new core API shape. If approved, the proposed opt-in projection is narrower and safer than changing the default response or adding a config knob.
