## Summary

Current main and release v2026.6.6 already implement the central bundled-MCP request-timeout fix: configured per-server request timeout now reaches SDK `Client.callTool`, so this cleanup can close as implemented.

## What This Changes

The issue asks OpenClaw to pass a configurable per-server MCP request timeout into bundled MCP `Client.callTool` so tools running longer than the SDK 60s default do not fail.

## Best Possible Solution

Keep the shipped `requestTimeoutMs` / `timeout` implementation; if maintainers want a longer default or `resetTimeoutOnProgress`, track that as a separate product-policy follow-up with fresh evidence.

## Reproduction Assessment

Yes, at source level for affected builds: the old bundled MCP runtime called SDK `Client.callTool` without `RequestOptions`, and SDK 1.29.0 defaults omitted request timeouts to 60 seconds. I did not run a live slow-tool wall-clock reproduction during this read-only cleanup pass.

## Solution Assessment

Yes for the central bug: the shipped implementation adds an explicit per-server request-timeout config and passes it into SDK `callTool` at the owner runtime boundary. The proposed 5-10 minute default and progress-reset behavior are separate policy choices rather than necessary remaining work for this issue.
