## Summary

Keep open. The PR targets a real current-main sync stat-error bug, but the branch is stale against the shared allowlist helper split and its read-file rethrow would still leave sync and async allowlist reads inconsistent; related PR #63324 is already the narrower active repair path but is not merged.

## What This Changes

The PR changes `src/pairing/pairing-store.ts` so the old sync pairing allowlist reader rethrows non-ENOENT `statSync` and `readFileSync` errors instead of treating the allowlist as missing.

## Best Possible Solution

Refresh or land the narrower current-main stat-error fix in `src/pairing/allow-from-store-file.ts` with focused regression tests, then decide separately whether read-file failures should surface consistently across both sync and async readers.

## Reproduction Assessment

Yes: a focused test can stub `fs.statSync` to throw `EACCES` and observe current main returning an empty allowlist while the async stat path rethrows. A second stub where stat succeeds and `readFileSync` throws `EACCES` reproduces the PR's sync-only read behavior against the current async fallback contract.

## Solution Assessment

No: rethrowing unexpected sync stat errors is the right direction, but this PR also rethrows sync read failures without changing the async reader and targets an old inline helper. The safer path is to update the current shared helper and test the stat contract separately from any read-fallback policy change.
