## Summary

Current main already runs a pre-restart doctor repair during package updates and includes a core doctor migration that removes stale bindings with missing agent IDs, so the reported upgrade crash-loop path is covered on main but not in the latest release.

## What This Changes

The issue reports that UpgradeOpenClaw can finish reconfiguration without validating the resulting config, leaving Gateway in a restart crash loop when stale bindings reference missing agents.

## Best Possible Solution

Keep the current-main update and doctor behavior; users on v2026.5.28 still need the next release or a manual `openclaw doctor --fix` plus restart to receive it.

## Reproduction Assessment

Yes from source: stale `bindings[].agentId` values are schema-invalid when `agents.list` is present, and current main has a doctor migration that removes those stale bindings. I did not run the full systemd upgrade scenario because this read-only review cannot execute mutating install/update flows.

## Solution Assessment

Yes: running the existing core doctor repair during the update flow before restart is the right owner boundary because runtime stays strict while doctor owns migration/repair.
