## Summary

Keep open: current main and the latest release still hard-code Control UI anti-framing headers, and there is no config/schema/docs opt-in for same-origin framing. The requested direction is plausible, but it relaxes clickjacking defenses and needs an explicit maintainer/security decision before implementation.

## What This Changes

Add a deny-by-default `gateway.controlUi` frame-ancestor policy so operators can opt into same-origin or trusted-origin Control UI framing.

## Best Possible Solution

If maintainers accept the use case, add a narrowly validated deny-by-default frame policy with schema/help/docs/security-audit coverage and focused tests for deny, same-origin, and multi-origin CSP-only modes.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: every Control UI response receives `X-Frame-Options: DENY` plus `frame-ancestors 'none'`, and the strict config schema has no override field.

## Solution Assessment

Unclear until maintainers decide the security policy. The proposed `frameAncestors` shape is plausible, but it should be reviewed as an intentional clickjacking-policy relaxation rather than a mechanical bug fix.
