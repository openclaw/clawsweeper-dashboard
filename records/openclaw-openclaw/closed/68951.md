## Summary

Current `main` routes `memory rem-backfill --agent <id>` through the selected agent workspace for both builtin and QMD memory managers, so the reported main-workspace write path no longer matches the present code.

## Evidence

- **`rem-backfill` writes to the manager-reported workspace:** The command resolves `opts.agent`, fetches that agent's memory manager, reads `status.workspaceDir`, and uses that same `workspaceDir` for diary writes, short-term staging, JSON output, and the printed `dreamsPath`/`workspace` lines.
  - file: [extensions/memory-core/src/cli.runtime.ts:1725](https://github.com/openclaw/openclaw/blob/938b53698ec17934a5d1a6cffb279c584f64d9f3/extensions/memory-core/src/cli.runtime.ts#L1725)
  - command: `nl -ba extensions/memory-core/src/cli.runtime.ts | sed -n '1723,1922p'`
  - sha: [938b53698ec1](https://github.com/openclaw/openclaw/commit/938b53698ec17934a5d1a6cffb279c584f64d9f3)
- **Builtin memory manager is agent-scoped:** `MemoryIndexManager.get()` resolves `workspaceDir` from `resolveAgentWorkspaceDir(cfg, agentId)`, and `status()` returns that exact `workspaceDir`.
  - file: [extensions/memory-core/src/memory/manager.ts:165](https://github.com/openclaw/openclaw/blob/938b53698ec17934a5d1a6cffb279c584f64d9f3/extensions/memory-core/src/memory/manager.ts#L165)
  - command: `nl -ba extensions/memory-core/src/memory/manager.ts | sed -n '155,175p;709,742p'`
  - sha: [938b53698ec1](https://github.com/openclaw/openclaw/commit/938b53698ec17934a5d1a6cffb279c584f64d9f3)
- **QMD memory manager is also agent-scoped:** The QMD runtime config uses `resolveAgentWorkspaceDir(cfg, agentId)`, and QMD `status()` also reports `this.workspaceDir`, so the active backend does not change the workspace target on current `main`.
  - file: [extensions/memory-core/src/memory/qmd-manager.ts:1320](https://github.com/openclaw/openclaw/blob/938b53698ec17934a5d1a6cffb279c584f64d9f3/extensions/memory-core/src/memory/qmd-manager.ts#L1320)
  - command: `nl -ba extensions/memory-core/src/memory/qmd-manager.ts | sed -n '1310,1335p;3029,3038p'`
  - sha: [938b53698ec1](https://github.com/openclaw/openclaw/commit/938b53698ec17934a5d1a6cffb279c584f64d9f3)
- **Non-default agents do not resolve to `main` workspace by default:** `resolveAgentWorkspaceDir()` keeps non-default agents on their own workspace path, and the regression test expects a non-default agent to resolve under `agents.defaults.workspace/<id>` rather than the default agent workspace.
  - file: [src/agents/agent-scope-config.ts:149](https://github.com/openclaw/openclaw/blob/938b53698ec17934a5d1a6cffb279c584f64d9f3/src/agents/agent-scope-config.ts#L149)
  - command: `nl -ba src/agents/agent-scope-config.ts | sed -n '149,171p'; nl -ba src/agents/agent-scope.test.ts | sed -n '481,514p'`
  - sha: [938b53698ec1](https://github.com/openclaw/openclaw/commit/938b53698ec17934a5d1a6cffb279c584f64d9f3)

## Risks / Open Questions

- There is no command-level regression test here for `memory rem-backfill --agent <id>`, so this conclusion rests on current code-path inspection rather than a fresh end-to-end repro run.
- A user could still point two agents at the same workspace intentionally via config; that would be configuration overlap, not the old implicit main-workspace routing described in this issue.
