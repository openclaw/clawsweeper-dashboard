## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Route DOCX through the document-extractor boundary:** `src/media/docx-extract.ts:1`
  - body: PDF/input_file document parsing currently goes through `extractDocumentContent` and the bundled `document-extract` plugin contract. This new core `JSZip` parser is imported directly by `input_file` and `read`, which splits document parsing ownership and bypasses existing plugin availability/failure behavior. Either extend the document-extractor contract for DOCX or get an explicit maintainer decision before landing a core-only parser.
  - confidence: 0.86
- **[P1] Bound DOCX inflation before allocating XML:** `src/media/docx-extract.ts:71`
  - body: `documentXmlFile.async("uint8array")` inflates the full `word/document.xml` entry before checking `maxXmlBytes`. JSZip accumulates chunks and concatenates them before resolving, so a small compressed DOCX can allocate beyond the configured cap before this guard runs. The extractor needs a pre-allocation or streaming limit before it is exposed to user-provided files.
  - confidence: 0.94
- **[P1] Keep MIME sniffing before accepting DOCX files:** `src/media/input-files.ts:400`
  - body: Adding DOCX to the default allowlist makes the new parser reachable from `input_file`, but this branch's file path still trusts the declared media type or response `Content-Type` before dispatch. Current main has a `detectMime` pass before the allowlist; preserve that gate so arbitrary bytes labeled as DOCX do not reach archive parsing.
  - confidence: 0.88
- **[P2] Update the input_file MIME documentation:** `src/media/input-files.ts:106`
  - body: The public OpenResponses docs list the current allowed `input_file` MIME types and PDF-only extraction behavior. If DOCX becomes a default-supported type, those docs need to mention the new MIME, limits, and extraction behavior so the API contract matches the implementation.
  - confidence: 0.82
