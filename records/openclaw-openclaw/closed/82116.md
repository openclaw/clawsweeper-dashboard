## Summary

Current main already implements the narrow fix for this report: configured custom/local model provider base URLs are trusted by exact origin through the guarded fetch path, so a Tailscale/private-DNS llamacpp endpoint at the configured origin should no longer hit the private-address block. The fix is current-main-only in this checkout; no tag contains it yet.

## What This Changes

The issue reports that OpenClaw 2026.5.12 blocks a Tailscale Magic DNS HTTPS endpoint for a locally hosted llamacpp provider because URL fetch resolves it to a private/internal/special-use IP address.

## Best Possible Solution

Keep the current exact-origin trust behavior and ship it in the next release; users on 2026.5.12 need a build containing 44840007d42d89155d7d9a323dc81438618e26b6 or a broader explicit provider request.allowPrivateNetwork opt-in if appropriate.

## Reproduction Assessment

Yes for the reported release path: a configured custom/local provider baseUrl resolving to a private address matches the exact SSRF guard log and the fixed code path. I did not reproduce a current-main failure because current main now has source and test coverage for the allowed exact-origin path.

## Solution Assessment

Yes; exact-origin trust for an operator-configured custom/local provider baseUrl is the narrow maintainable fix because it allows the intended Tailscale/private-DNS origin without broadly trusting other private hosts, ports, redirects, or metadata endpoints.
