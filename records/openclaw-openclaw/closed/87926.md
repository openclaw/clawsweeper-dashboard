## Summary

Keep open for maintainer product and security design: current main has deterministic exec approval policy, allowlists, safeBins, and approval forwarding, but no YOLT-style classifier hook or durable approval-resume contract for expired/off-box approval cards.

## What This Changes

Add an OpenClaw approval integration path for YOLT-style command-risk classification with auditable decisions and recoverable approval intent.

## Best Possible Solution

Design one opt-in exec approval classifier contract that can consume external classifiers such as YOLT while preserving OpenClaw’s canonical host exec checks, audit trail, and approval identity semantics.

## Reproduction Assessment

Not applicable as a bug reproduction. Source and docs inspection show the requested classifier hook and recoverable approval-intent flow are absent from current main.

## Solution Assessment

Unclear as proposed. YOLT may be a useful classifier provider, but the maintainable OpenClaw shape is an opt-in approval-policy hook/API that keeps the parser-backed exec boundary authoritative.

## Risks / Open Questions

- Auto-approving shell commands from semantic labels can weaken the host exec security boundary unless the classifier is opt-in, parser-backed, auditable, and ordered with allowlist, safeBins, strict inline-eval, and command binding.
- Approval resumption after expiry needs a stable command identity and replay boundary so stale, cross-session, or cross-device approvals cannot authorize a different command.
- The generic classifier request overlaps https://github.com/openclaw/openclaw/issues/63734, so a YOLT-specific path should not create a competing policy surface.
