## Summary

This is a useful project-context workflow, but current OpenClaw architecture points it to a ClawHub skill/plugin rather than new core behavior. Current main already exposes the hooks and SDK seams needed to track per-project state, inject active project context, and extend memory/context behavior outside core.

## Best Possible Solution

Close this core issue and implement the idea as a ClawHub project-context skill or plugin. The package should track explicit project declarations/switches, store per-project context in plugin-owned session/workspace state, inject the active project context through before_prompt_build or a context engine, and optionally add project-specific memory corpora. If that work finds a missing generic SDK seam, open a narrow plugin API issue.
