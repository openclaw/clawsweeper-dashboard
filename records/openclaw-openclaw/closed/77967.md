## Summary

Keep open: current main still has a shared accounting gap where Feishu completion logs are based on dispatcher final-reply counts, while committed message-tool delivery evidence lives elsewhere and is not exposed as a side-path-satisfied turn result.

## What This Changes

Distinguish Feishu/Lark turns where a same-turn side-path message send succeeded from true no-reply turns instead of collapsing both into `replies=0`.

## Best Possible Solution

Add a typed, versioned channel-turn delivery signal for committed message-tool or side-path sends that Feishu can log without changing dispatcher final-reply counts.

## Reproduction Assessment

Yes, at source level: current main can process a message-tool-only group/channel turn with `queuedFinal=false` and `counts.final=0` while committed messaging-tool delivery evidence is tracked separately in the agent runner. I did not run a live Feishu WebSocket scenario.

## Solution Assessment

Unclear: distinguishing the states is the right direction, but the maintainable fix should be a shared channel-turn result contract rather than a Feishu-only log tweak.
