## Summary

Close as superseded: the linked loop-detection policy decision kept this surface opt-in/default-off to avoid extra warning or log noise unless operators explicitly enable it, while this PR adds disabled-mode INFO logging and is stale against current main's generic no-progress blocking behavior.

## What This Changes

The branch computes loop-detection verdicts when `tools.loopDetection.enabled` is false, emits deduplicated `[loop-detection-diag]` INFO logs, and updates tests, docs, and changelog.

## Best Possible Solution

Keep the default-off/opt-in policy as the canonical behavior; any future disabled-mode diagnostic should be proposed as an explicitly approved opt-in design that preserves current main's generic no-progress block.

## Reproduction Assessment

Yes for the code-path behavior: current main returns before detector analysis when loop detection is disabled, and PR head would run the verdict and log in that disabled path. I did not run a live OpenClaw setup, and the PR still lacks real behavior proof.

## Solution Assessment

No. The current PR conflicts with the linked opt-in/default-off policy, regresses current main's generic no-progress critical block, and would need an explicit product/config decision before a disabled-mode diagnostic could be accepted.
