## Summary

Current `main` contains targeted Telegram polling changes and regression coverage that match this report’s silent fresh-deploy/post-restart failure mode: the startup `getUpdates` offset preflight was removed, stalled polling is now force-restarted with a rebuilt transport, and duplicate-poller/conflict cases now emit actionable diagnostics. The issue was kept open on March 29 pending a concrete fix; that evidence now exists on `main`, although it does not appear to be part of the latest tagged release yet.

## Evidence

- **Changelog entries match the reported symptom:** `main`'s changelog explicitly lists two unreleased Telegram fixes: removing the startup persisted-offset `getUpdates` preflight to avoid self-conflicts before the runner starts, and keeping the polling stall watchdog active so silent `getUpdates` failures no longer require a manual gateway restart.

- **Startup path no longer performs offset-confirmation polling:** The current Telegram monitor reads the persisted update offset, skips any startup confirmation `getUpdates` call, acquires a polling lease, and immediately starts `TelegramPollingSession`, which is the code-path change called out in the changelog.
  - file: [extensions/telegram/src/monitor.ts:193](https://github.com/openclaw/openclaw/blob/c65aa1d2a6e573d49619c4dd0698a76010476e7d/extensions/telegram/src/monitor.ts#L193)
