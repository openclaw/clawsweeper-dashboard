## Label Justifications

- `P2`: The reported behavior is a bounded macOS update/restart bug that can break automation exit-code handling without making the gateway unhealthy.

## Summary

Current main and the v2026.5.20 tag already contain the version-change guard that recognizes a refreshed LaunchAgent serving the new package version and skips the redundant restart path. The reported failed run was launched from the pre-fix 2026.5.7 updater, while the reporter's same-version rerun on v2026.5.20 exercised the fixed path successfully.

## What This Changes

The issue asks OpenClaw to avoid failing `openclaw update` when a macOS LaunchAgent has already respawned a healthy new-version gateway on the configured port.

## Best Possible Solution

Keep the shipped version-aware post-refresh guard, and only reopen this path if a newer-to-newer package update still races with `gateway-restart.log` and `openclaw gateway status --deep` evidence.

## Reproduction Assessment

No high-confidence current-main reproduction is available. The concrete log was from an update launched by v2026.5.7, while current main and v2026.5.20 source include a version-changing regression test and guard for the reported race.

## Solution Assessment

Yes. The current implementation checks the updated gateway version immediately after service refresh and avoids a redundant restart script, which is the narrow fix for the reported healthy-new-gateway port race.
