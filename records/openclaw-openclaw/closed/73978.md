## Summary

Keep open. Current main still has a high-confidence source-level path where Discord can route an application command that has no local wrapper and OpenClaw returns without acknowledging it, while plugin native-command specs still come from the runtime command registry rather than a static Discord-ready contract.

## What This Changes

The issue reports Discord plugin-registered slash commands missing Discord's 3-second interaction ACK deadline and producing "Unknown interaction" or duplicate-ACK errors.

## Best Possible Solution

Define a Discord native plugin-command readiness and reconciliation contract so every Discord-routable plugin command has a local defer-first wrapper before interactions arrive, with stale routed commands acknowledged or cleaned up and event-loop saturation handled in the related session/load work.

## Reproduction Assessment

Yes, at source level: current main can receive an application command with no matching local wrapper and return without acknowledging it. I did not run a live Discord end-to-end repro in this review.

## Solution Assessment

No, the issue's blanket eager-loading suggestion is probably not the narrowest owner-boundary fix. A readiness/reconciliation contract plus explicit stale-command acknowledgement is safer than forcing broad plugin eager loading.
