## Summary

Close as implemented on current main: the code now dedupes provider message ids before dispatch and avoids replaying an already-persisted user turn after overflow compaction retry; I could not prove the fix is in v2026.5.3.

## What This Changes

Request to prevent Telegram/native agent sessions from replaying duplicate inbound user messages or prior assistant output during retry and context-overflow compaction.

## Best Possible Solution

Keep the current main implementation and ship it in the next release so Telegram/native retries use provider-message idempotency and transcript-based compaction continuation.

## Reproduction Assessment

Yes for the original bug: the report includes a concrete Telegram session where the same user instruction was appended multiple times during overflow compaction, and current-main regression tests cover the compaction retry replay case. I did not perform a live Telegram replay in this read-only review.

## Solution Assessment

Yes. The current implementation addresses the narrow failure at both ingress idempotency and compaction retry persistence, without adding a new config or product policy.
