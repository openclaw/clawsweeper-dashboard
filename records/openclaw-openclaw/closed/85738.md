## Label Justifications

- `P1`: The reported release behavior breaks real agent/channel replies once CLI-backed sessions cross the memory-flush threshold, even though main already contains the fix.
- `impact:message-loss`: The visible assistant reply can be replaced by a `MissingAgentHarnessError` payload instead of the intended channel response.
- `impact:session-state`: The failure is triggered by persisted session runtime override state interacting with compaction and memory-flush handling.

## Summary

Current main already contains the CLI-runtime pass-through that prevents `selectAgentHarnessDecision` from throwing for provider-owned aliases like `claude-cli`, with focused regression coverage and merged PR provenance.

## What This Changes

The issue reports that memory flush forwards a persisted CLI runtime override such as `claude-cli` into embedded harness selection, causing `MissingAgentHarnessError` instead of a normal reply.

## Best Possible Solution

Keep the main-branch selector pass-through and ship it in the next release so CLI-backed memory flushes stop surfacing harness-selection errors to users.

## Reproduction Assessment

No current-main failure was reproduced: source inspection shows the CLI-alias guard now runs before the throw, and regression coverage asserts `claude-cli` returns PI. The reported v2026.5.20 failure path is still credible from the issue details and the merged fix diff.

## Solution Assessment

Yes: the main-branch fix puts the CLI-alias exception in the shared harness selector reached by memory flush and other in-process callers, while preserving fail-closed behavior for unknown or wrong-provider runtime ids.
