## Summary

Current main fixes the reported default deadlock path by making omitted `gateway.reload.deferralTimeoutMs` use a bounded 300000ms restart wait, with regression coverage that emits `SIGUSR1` after the default timeout even while active task counts remain nonzero. The inbound path can still start work during the deferral window, but the issue explicitly listed a hard timeout as an acceptable outcome, and the unbounded stuck-processing/force-kill behavior is no longer the default behavior on main.

## What This Changes

Prevent a new inbound WebChat message during a deferred restart-required `config.apply` from leaving the main session indefinitely stuck in processing.

## Best Possible Solution

Keep the bounded default restart deferral as the shipped deadlock guard, and treat any future desire to reject or queue inbound messages during the deferral window as a separate latency/UX follow-up rather than the blocker for this deadlock report.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes against the reported/default-at-the-time behavior: the `v2026.4.27` tree passed omitted `deferralTimeoutMs` through without a default and documented that omitted values waited indefinitely. Against current main's default config, the unbounded deadlock no longer reproduces because the restart emits after 300000ms.

## Solution Assessment

Is this the best way to solve the issue? Yes for this report: the bounded default is the narrowest low-risk fix for the force-kill deadlock and was one of the reporter's suggested acceptable directions. A direct inbound reject/queue gate could still improve user feedback, but it is not required to eliminate the indefinite restart deferral.
