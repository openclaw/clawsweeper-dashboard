## Summary

Keep open: current main and the latest shipped tag still write the Matrix crypto IDB snapshot directly to the final JSON path, so the reported unclean-shutdown corruption window remains. The open Matrix stop-helper PR is related to a sibling persistence issue, but it does not touch this atomic replacement gap.

## What This Changes

The issue asks to replace Matrix crypto IDB snapshot direct overwrites with atomic JSON replacement and regression coverage.

## Best Possible Solution

Use writeJsonFileAtomically inside the existing Matrix IDB file-lock boundary, keep the snapshot format and callers unchanged, and add regression coverage that failed replacement leaves the previous snapshot valid.

## Reproduction Assessment

Yes, at source level: current main still writes directly to the final snapshot path, and a focused test can simulate write or replacement failure without depending on crash timing. I did not run a live unclean-shutdown reproduction.

## Solution Assessment

Yes. Reusing the existing atomic JSON helper under the current withFileLock scope is the narrowest maintainable fix; the sibling event-driven persistence and awaited-shutdown work should stay separate.
