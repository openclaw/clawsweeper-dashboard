## Summary

Keep open for maintainer product and security direction. Current main and v2026.6.9 still export placeholder OTLP log bodies unless broad boolean `diagnostics.otel.captureContent: true` is enabled, and the related trace-correlation fix explicitly left log-body export as a separate privacy decision.

## What This Changes

The issue requests that diagnostics-otel OTLP log record bodies contain rendered log message text instead of placeholder category strings.

## Best Possible Solution

Keep the safe default unless maintainers approve either the existing broad content-capture policy as sufficient or a narrow documented log-body opt-in with redaction and regression proof.

## Reproduction Assessment

Yes for source-level reproduction: current main and v2026.6.9 explicitly emit `"log"` for normal OTLP log bodies unless broad boolean content capture is enabled. I did not run a live Loki collector path in this read-only review.

## Solution Assessment

No as a default behavior change. The safer solution needs maintainer privacy/security approval, likely by retaining broad `captureContent: true` as the documented path or adding a narrower explicit log-body opt-in.
