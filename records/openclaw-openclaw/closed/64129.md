## Summary

Keep open: current main still has the setup/onboard provider-auth path that can apply a provider default over an existing primary model, and the exact narrow fix PR was closed unmerged. The broader paid-preview consent policy remains separately tracked, but the setup preservation bug is still actionable here.

## What This Changes

The issue asks OpenClaw to prevent paid Google provider setup from silently changing existing heartbeat/background model usage into paid Gemini traffic.

## Best Possible Solution

Pass the existing default-preservation option through setup/onboard provider auth, add a setup regression test, preserve first-run and explicit default-setting behavior, and leave broader pricing-consent policy to the related open issue.

## Reproduction Assessment

Yes, source-level: setup forwards `setDefaultModel` without `preserveExistingDefaultModel`, the provider-auth helper preserves only when that flag is true, and heartbeat can inherit the resulting default. I did not run a live paid Google billing reproduction.

## Solution Assessment

Yes for the narrow bug: reusing the existing preservation seam in setup is the smallest maintainable fix and avoids Google-specific or heartbeat-specific policy changes. Broader price-tier consent and warning behavior belongs in the related product-policy issue.
