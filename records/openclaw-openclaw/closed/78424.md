## What This Changes

The issue reports that `sessions.resolve` treats `sessionKey="current"` as a literal key even though agent prompts and wrapper behavior recommend it as the caller's current-session alias.

## Best Possible Solution

Fix the gateway/current-session alias path, or an equivalent shared resolver path, so semantic `current` no longer produces failed `sessions.resolve` probes while preserving literal `current` key/sessionId behavior.

## Reproduction Assessment

Yes, from source inspection: current main still routes raw `current` through `sessions.resolve` before falling back, and the direct RPC has no requester-current binding. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

Yes: the best fix is a bounded shared resolver or gateway-context change, not another wrapper-only shortcut, because the observable failure is the gateway RPC rejecting/logging raw `current` before wrappers recover.
