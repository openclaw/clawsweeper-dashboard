## Summary

Close as cannot reproduce. On current main, the gateway health monitor only treats a channel as a stale socket when it reports both `connected === true` and a `lastTransportActivityAt` timestamp. Feishu currently does not publish those transport-liveness fields, so the default health-check loop described in the issue no longer triggers the restart path that would wipe the in-memory pending mention history.

## Evidence

- **Health monitor now gates stale-socket restarts on transport telemetry:** `evaluateChannelHealth` only enters stale-socket logic when `snapshot.connected === true && lastTransportActivityAt != null`; otherwise it returns healthy.
  - file: [src/gateway/channel-health-policy.ts:117](https://github.com/openclaw/openclaw/blob/cb2c36b04975e34ab7014a11575d6589aa96ae32/src/gateway/channel-health-policy.ts#L117)
- **Regression tests cover non-telemetry channels as healthy:** Tests assert that running channels without a `connected` field stay healthy, and connected channels without transport tracking are not restarted by the monitor.
  - file: [src/gateway/channel-health-monitor.test.ts:485](https://github.com/openclaw/openclaw/blob/cb2c36b04975e34ab7014a11575d6589aa96ae32/src/gateway/channel-health-monitor.test.ts#L485)
- **Health policy test confirms no stale-socket failure without transport tracking:** `does not flag stale sockets for channels without transport tracking` expects `{ healthy: true, reason: "healthy" }` when `lastTransportActivityAt` is null.
  - file: [src/gateway/channel-health-policy.test.ts:214](https://github.com/openclaw/openclaw/blob/cb2c36b04975e34ab7014a11575d6589aa96ae32/src/gateway/channel-health-policy.test.ts#L214)
- **Feishu pending history is still per-lifecycle in-memory state:** `monitorSingleAccount` recreates `chatHistories` with `new Map<string, HistoryEntry[]>()` on each monitor lifecycle, so an actual restart would still clear pending mention history.
  - file: [extensions/feishu/src/monitor.account.ts:654](https://github.com/openclaw/openclaw/blob/cb2c36b04975e34ab7014a11575d6589aa96ae32/extensions/feishu/src/monitor.account.ts#L654)
- **Feishu does not currently publish transport-liveness patches:** A repo search found no Feishu use of `createConnectedChannelStatusPatch`, `createTransportActivityStatusPatch`, or any `lastTransportActivityAt` updates; the Feishu gateway start path only sets `port` status.
  - file: [extensions/feishu/src/channel.ts:1210](https://github.com/openclaw/openclaw/blob/cb2c36b04975e34ab7014a11575d6589aa96ae32/extensions/feishu/src/channel.ts#L1210)
  - command: `rg -n "createTransportActivityStatusPatch|lastTransportActivityAt|createConnectedChannelStatusPatch|setStatus\(.*connected|setStatus\(.*lastTransport|setStatus\(.*lastEvent" extensions/feishu/src src/channels src/gateway`

## Risks / Open Questions

- This only resolves the reported health-check restart reproduction path. Pending Feishu mention history is still in-memory and would be lost on a real restart, crash, or manual stop/start.
- `gh` inspection was unavailable in this checkout because `GH_TOKEN` is unset, so the review is based on current local `main` code and tests only.
- If Feishu later starts reporting transport-liveness status again, this path should get a Feishu-specific regression test so restart-driven history loss cannot regress silently.
