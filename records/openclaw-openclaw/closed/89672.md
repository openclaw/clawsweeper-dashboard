## Summary

Keep this PR open for normal review: the runtime fix is the right owner-layer change for the reported Skills tab checkbox state leak, but the external PR still needs after-fix real Control UI proof and the added regression test should be tightened because it does not currently dirty the checkbox DOM state before the list mutation.

## What This Changes

The branch keys the Skills tab list rendering with Lit `repeat(skill.skillKey)` and adds a jsdom regression test for checkbox state after a skill leaves the filtered tab.

## Best Possible Solution

Land the keyed `repeat` fix after the contributor adds real after-fix UI proof and tightens the regression test so it fails without keyed row identity.

## Reproduction Assessment

Yes, source-backed: current main reuses unkeyed Lit child parts by list position while the checkbox property can be changed by the user before `skills.update` reloads the filtered report. I did not run a browser repro in this read-only review.

## Solution Assessment

Yes, keyed `repeat` by `skill.skillKey` is the narrow maintainable fix for row identity; manually forcing checkbox state or remounting every row would be broader. The test should still simulate the actual checked-state transfer.
