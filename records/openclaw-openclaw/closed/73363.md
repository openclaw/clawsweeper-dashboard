## Summary

Keep open. The PR targets a valid privacy bug, but after preserving `isReasoning` it still lets reasoning-only payloads flow through fallback delivery and recovery paths, and the contributor has not supplied after-fix Matrix or Feishu proof.

## What This Changes

The PR propagates `isReasoning` from embedded-runner payloads, filters cron summary/text/delivery pickers in the primary pass, adds focused tests, and updates the changelog.

## Best Possible Solution

Land one canonical fix that preserves `isReasoning` through embedded payloads and suppresses it across cron summary, output, delivery, synthesized fallback, and post-error recovery, with redacted Matrix or Feishu proof.

## Reproduction Assessment

Yes, at source level. Current main creates reasoning payload items, drops or ignores the marker in returned cron payloads, and cron selectors plus direct delivery can choose and send nonempty reasoning text.

## Solution Assessment

No as-is. Preserving `isReasoning` and filtering primary selectors is the right direction, but reasoning should remain non-user-visible in fallback and post-error recovery paths rather than being returned as last-resort content.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Suppress reasoning-only cron fallbacks:** `src/cron/isolated-agent/helpers.test.ts:169-173`
  - body: The new fallback expectation keeps a lone `{ isReasoning: true }` payload selectable. `resolveCronPayloadOutcome()` can pass that through `deliveryPayloads`, and direct cron delivery sends selected payloads via `sendDurableMessageBatch()`, so a thinking-only run can still expose internal reasoning text. Please keep reasoning out of fallback passes and return no delivery when only reasoning exists.
  - confidence: 0.88
- **[P2] Exclude reasoning from post-error recovery:** `src/cron/isolated-agent/helpers.ts:291-296`
  - body: After this PR preserves `isReasoning`, these success checks still count any later non-error text. An error followed only by a reasoning payload can clear the fatal error path and report the cron run as ok, so the checks should also require `isReasoning !== true`.
  - confidence: 0.84
