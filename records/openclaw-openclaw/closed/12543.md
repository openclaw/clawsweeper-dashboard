## Summary

The reported hardcoded Google OAuth secret issue is no longer present on current `main`, and the latest shipped tag already uses an externalized Google OAuth flow instead of embedding Google OAuth client secrets in OpenClaw source. The specific bundled plugins named in the report are also no longer shipped.

## Evidence

- **Reported plugin IDs are retired:** Current config treats `google-antigravity-auth` and `google-gemini-cli-auth` as legacy removed plugin IDs, which matches the report's cited components no longer being shipped.
  - file: [src/config/validation.ts:43](https://github.com/openclaw/openclaw/blob/3361593442e295e9bb1f55cbfb38861ea9413e45/src/config/validation.ts#L43)

  - sha: [3361593442e2](https://github.com/openclaw/openclaw/commit/3361593442e295e9bb1f55cbfb38861ea9413e45)
- **Current Google OAuth flow does not embed a client secret:** `resolveOAuthClientConfig()` first reads `OPENCLAW_GEMINI_OAUTH_CLIENT_ID` / `_SECRET` (or `GEMINI_CLI_*`) from env, then falls back to extracting the locally installed Gemini CLI credentials; there is no hardcoded Google OAuth client secret literal in OpenClaw production code here.
  - file: [extensions/google/oauth.credentials.ts:258](https://github.com/openclaw/openclaw/blob/3361593442e295e9bb1f55cbfb38861ea9413e45/extensions/google/oauth.credentials.ts#L258)

## Risks / Open Questions

- The current `google-gemini-cli` integration still reuses Gemini CLI OAuth credentials by reading the locally installed Gemini CLI package. If maintainers want to revisit that integration design, it should be tracked separately from this specific hardcoded-secret report.
