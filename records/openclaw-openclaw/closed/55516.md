## Summary

#55516 is a real, still-unimplemented request, but the same remaining nested-lane concurrency work is now tracked more completely by open #57952, which explicitly covers hook-dispatched agent runs plus the related cron and sessions_send serialization/deadlock cases. Current main still only configures cron/main/subagent lane concurrency and leaves CommandLane.Nested at the command-queue default of 1, so this should close as superseded, not implemented.

## Best Possible Solution

Close #55516 as superseded by #57952 and use #57952 as the canonical implementation tracker. The eventual fix should add one documented, schema-backed nested-lane concurrency control, apply it during gateway startup and hot reload, and cover hook, cron, and sessions_send nested-lane behavior with focused tests.
