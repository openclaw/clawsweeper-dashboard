## Summary

Keep open: the PR targets a real Codex context-cap regression, but the submitted runtime fallback can bypass canonical OpenAI provider-level context caps and conflicts with the current doctor-migration boundary for legacy `openai-codex` config.

## What This Changes

The PR changes `resolveContextConfigProviderForRuntime` to return `openai-codex` for Codex runtime when legacy `models.providers["openai-codex"].models` exists and canonical `models.providers.openai.models` is empty.

## Best Possible Solution

Fix the regression at the canonical migration/repair boundary, or make any fallback narrowly prove that canonical OpenAI model-level and provider-level caps always win before legacy data is consulted.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main returns `openai` for Codex runtime context lookup, so legacy-only `models.providers.openai-codex.models[].contextTokens` is not read by the OpenAI context resolver. I did not run a live upgraded-config repro in this read-only review.

## Solution Assessment

No. The PR is a plausible narrow mitigation for one legacy model-only shape, but it is not the best fix because it can bypass canonical provider-level caps and sidesteps the existing doctor migration contract.
