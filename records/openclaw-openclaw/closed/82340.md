## Summary

Current main already fixes the reported outgoing replay shape at the Chat Completions request-builder boundary: `reasoning_details` strings are no longer sent as `reasoning_details` on assistant messages, and the behavior is covered by focused transport tests. The fix is on current main only; I could not prove it is included in a shipped tag.

## What This Changes

The issue asks OpenClaw to stop replaying top-level `reasoning_details` strings on assistant messages sent to OpenRouter Chat Completions providers.

## Best Possible Solution

Keep the current request-builder sanitizer and let any broader OpenRouter preserved-reasoning semantics continue in narrower follow-up issues.

## Reproduction Assessment

Yes for the original bug: the issue and merged PR evidence include a concrete curl A/B where `reasoning_details` caused OpenRouter 500s and removing it returned 200. I did not rerun live OpenRouter in this read-only pass, and current main now has the sanitizer and regression tests.

## Solution Assessment

Yes. Fixing this in `buildOpenAICompletionsParams` after `convertMessages` is the right boundary because it covers the outgoing JSON payload immediately before Chat Completions requests are built, instead of relying on history cleanup helpers that may not run on every path.
