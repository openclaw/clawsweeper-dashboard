## Summary

Current main already solves the Discord 1005/1006 stale-resume loop through the pinned @buape/carbon 0.16.0 gateway implementation. Carbon bounds failed RESUME attempts, clears cached resume state, and falls back to fresh IDENTIFY; OpenClaw pins that dependency, uses the Carbon gateway path, and promotes the fresh-IDENTIFY debug event in Discord logs. The PR discussion and local VPN reproduction describe a real failure mode, but this PR branch is obsolete against the current Discord plugin layout and shipped dependency behavior.

## Best Possible Solution

Close this PR as implemented on main and keep the current Carbon-backed Discord gateway recovery. If a 1005/1006 resume-close loop still reproduces on @buape/carbon 0.16.0 or later, track it as a fresh issue against the current extensions/discord layout with logs showing the Carbon reconnect/fresh-IDENTIFY sequence.
