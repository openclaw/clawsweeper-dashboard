## Summary

Keep open. Current main still contains the reported disagreement: managed Chrome launch failure is decided by a final `isChromeReachable` result, while the thrown error is formatted from `diagnoseChromeCdp`, so the contradictory “failed but diagnostic says ready” path remains source-backed and narrow enough to repair.

## What This Changes

The issue asks for managed Chrome launch readiness to use the same stronger CDP diagnostic for the final success/failure decision and diagnostic message.

## Best Possible Solution

Make the final managed-launch readiness decision from one captured `diagnoseChromeCdp` result, while preserving singleton-lock recovery, stderr redaction/hints, and existing managed-browser startup contracts.

## Reproduction Assessment

Yes, at source level: current main can enter the failure branch when the final `isChromeReachable` returns false, then format an immediately successful `diagnoseChromeCdp`. I did not run the intermittent macOS cold-start path live.

## Solution Assessment

Yes. The narrow maintainable fix is to use a single captured diagnostic for the final decision and message, rather than adding broader retry or cooldown behavior around an inconsistent final probe.
