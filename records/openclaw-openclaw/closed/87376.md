## Label Justifications

- `P0`: The report concerns a claimed auth-boundary collapse from hook-only ingress into full Gateway operator access.
- `impact:security`: The affected surface is Gateway shared-secret authentication and hook ingress secret separation.

## Summary

Keep open. Current main and the latest release still allow `hooks.token` to match Gateway password or trusted-proxy local password auth at startup; existing audit/docs flag this as critical but intentionally keep password-mode reuse startup-compatible, so this needs a maintainer security and upgrade decision.

## What This Changes

The issue asks OpenClaw to reject `hooks.token` reuse against Gateway password and trusted-proxy local password auth, not only Gateway token auth.

## Best Possible Solution

Make a maintainer security decision on whether reused hook/Gateway shared secrets should fail startup; if yes, update the startup guard, tests, docs, and doctor/audit migration guidance together.

## Reproduction Assessment

Yes from source inspection, not from a live run. Current main and `v2026.5.26` only compare `hooks.token` to token-mode Gateway auth, while tests assert password/trusted-proxy password reuse is allowed and HTTP bearer auth is replayed as a password candidate.

## Solution Assessment

Unclear pending maintainer decision. Startup rejection is the strongest safe-default fix, but it would replace documented audit-only compatibility and needs an upgrade/migration plan.
