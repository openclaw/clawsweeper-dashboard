## Summary

Current `main` contains targeted compaction-retry fixes that address the reported post-compaction hang: pending retries are counted until all retries finish, compaction-end events with `willRetry` keep the run in retry state instead of resolving early, and the runner now has an aggregate timeout fallback so a lost retry resolution cannot hold the session indefinitely.

## Evidence

- **Pending compaction retries are tracked until fully drained:** `pendingCompactionRetry` is incremented on retry, decremented on each retry completion, and the compaction wait only resolves when both `pendingCompactionRetry === 0` and `compactionInFlight === false`.
  - file: [src/agents/pi-embedded-subscribe.ts:333](https://github.com/openclaw/openclaw/blob/3361593442e295e9bb1f55cbfb38861ea9413e45/src/agents/pi-embedded-subscribe.ts#L333)
  - command: `nl -ba src/agents/pi-embedded-subscribe.ts | sed -n '332,360p'`
  - sha: [3361593442e2](https://github.com/openclaw/openclaw/commit/3361593442e295e9bb1f55cbfb38861ea9413e45)
- **Compaction wait no longer resolves early:** `waitForCompactionRetry()` now waits on the shared compaction promise whenever compaction is active or retries are pending, including the microtask race window.
  - file: [src/agents/pi-embedded-subscribe.ts:884](https://github.com/openclaw/openclaw/blob/3361593442e295e9bb1f55cbfb38861ea9413e45/src/agents/pi-embedded-subscribe.ts#L884)

- **`willRetry` compaction endings explicitly re-arm retry state:** On `compaction_end` with `willRetry`, the handler records another pending retry and resets subscriber state instead of treating the compaction as terminal.
  - file: [src/agents/pi-embedded-subscribe.handlers.compaction.ts:46](https://github.com/openclaw/openclaw/blob/3361593442e295e9bb1f55cbfb38861ea9413e45/src/agents/pi-embedded-subscribe.handlers.compaction.ts#L46)
