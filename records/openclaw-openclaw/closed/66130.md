## Summary

Current main fixes the observable bug from #66130: config-triggered restarts no longer have a default 300s force-restart drain, and restart emission is deferred until active gateway work has drained unless an explicit positive `gateway.reload.deferralTimeoutMs` is configured. The command-queue-wide `GatewayDrainingError` still exists for the actual restart shutdown path, but current config and gateway-tool restarts wait before emitting SIGUSR1, so they do not enter the old 5-minute global rejection window while work is active. The same change landed in `ec1f72b6c58f05432e6d5578fef4884b6adc2fee`, which also closed related issue #65485.

## Best Possible Solution

Close #66130 as implemented on current main. Keep the shipped idle-window restart behavior; track any future per-channel partial drain or agent-warning UX as a narrower follow-up rather than keeping this bug open for the old 5-minute global rejection behavior.
