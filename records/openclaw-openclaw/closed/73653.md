## Summary

Close as duplicate/superseded. Current main still has the reported Gateway protocol gap, but the same remaining Paperclip `paperclip` metadata rejection is already tracked by the earlier open #62102, and open #69139 is the active implementation candidate for the schema/test change. This issue adds useful detail about a more general adapter metadata namespace, but that product choice should be consolidated into the canonical thread rather than tracked as a second open issue.

## What This Changes

The issue asks for Gateway `agent` request params to accept third-party adapter metadata, preferably through a typed `adapterMeta` or `adapterContext` namespace, so Paperclip-style metadata no longer causes `INVALID_REQUEST`.

## Best Possible Solution

Keep one canonical Gateway protocol fix in #62102 and #69139. The preferred end state is an additive, documented metadata contract for `agent` params with validator and server regression coverage; maintainers can still choose a narrower `paperclip` allowlist there if they decide compatibility should remain service-specific.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: send or validate an `agent` params object with a required `message` and `idempotencyKey` plus any unknown top-level key such as `paperclip`; current main rejects it before runtime handling.

## Solution Assessment

Yes, closing this duplicate is the best cleanup path because the same remaining bug is already tracked in #62102 with #69139 as the open implementation candidate. The code fix itself should be decided in that canonical thread, preferably without broad `additionalProperties: true` on the whole `agent` payload unless maintainers intentionally relax typo protection.
