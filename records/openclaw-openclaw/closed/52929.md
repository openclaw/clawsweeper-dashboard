## Summary

Current `main` already implements stale session-lock cleanup for `.jsonl.lock` files during gateway startup and in the shared session-lock utility, with regression coverage. The fix appears to have landed after the latest tagged release, so the issue is fixed on `main` but not yet tied to a shipped release from the evidence available here.

## Evidence

- **Startup now sweeps session lock directories:** `startGatewaySidecars()` resolves agent session directories and calls `cleanStaleLockFiles({ removeStale: true })` before other sidecars continue, which directly addresses stale `.jsonl.lock` files left after crashes/restarts.
  - file: [src/gateway/server-startup-post-attach.ts:149](https://github.com/openclaw/openclaw/blob/38caa6832d4e1d344736d635b0c5fdad37e4430b/src/gateway/server-startup-post-attach.ts#L149)

- **Shared lock cleaner removes stale `.jsonl.lock` files:** `cleanStaleLockFiles()` scans `*.jsonl.lock`, inspects PID/age/starttime, and removes stale locks when `removeStale` is enabled. `acquireSessionWriteLock()` also writes PID metadata needed for stale detection.
