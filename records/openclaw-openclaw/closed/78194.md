## Summary

Keep open: current main still has a Claude CLI telemetry gap where cache/input counters can be present while the context numerator remains unknown, and the existing guard against cumulative CLI totals means this needs a narrow runtime/status fix rather than cleanup closure.

## What This Changes

The issue reports that `/status` and `session_status` show `Context: ?/1.0m` for Claude CLI runtime sessions despite available token/cache telemetry and a detected 1M context limit.

## Best Possible Solution

Persist a trusted Claude CLI prompt or last-call context snapshot and let `/status` use it, while preserving the guard that rejects cumulative CLI totals as context usage.

## Reproduction Assessment

Yes, at source level: a Claude CLI session entry with nonzero input/cache counters and `totalTokensFresh: false` follows the current `buildStatusMessage` path to `Context: ?/1.0m` while still rendering the cache line. I did not run a live Claude CLI OAuth session in this read-only pass.

## Solution Assessment

Yes, the best fix is narrow but should be in the telemetry handoff, not a display-only guess. Carry a trusted Claude CLI prompt/last-call token snapshot into `totalTokens` or transcript fallback, and keep raw cumulative CLI totals out of context accounting.

## Repair Work Prompt

Fix the Claude CLI `/status` and `session_status` context numerator gap reported in https://github.com/openclaw/openclaw/issues/78194. Current main persists Claude CLI input/output/cache counters but deliberately leaves `totalTokensFresh=false` because raw CLI usage can be cumulative; as a result status shows `Context: ?/1.0m` while cache stats show live-looking prompt data. Implement a narrow fix that carries a trusted Claude CLI prompt/last-call context snapshot into session/transcript usage, or derives it only from fields that are safe for context accounting. Add regression coverage for a Claude CLI run with small uncached input plus large cacheRead showing a real context numerator, and keep the existing cumulative-usage guard green. Add a single-line CHANGELOG.md user-facing fix entry.
