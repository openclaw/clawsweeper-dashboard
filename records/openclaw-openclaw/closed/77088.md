## Summary

Keep open: current main still has the monotonic Telegram sticky transport fallback, and open PR #77157 is the right implementation vehicle before this issue closes.

## What This Changes

The issue asks Telegram fetch fallback to recover from transient transport promotion instead of staying latched to IPv4 or pinned-IP dispatchers until process restart.

## Best Possible Solution

Land a narrow Telegram plugin recovery fix like #77157 that probes or demotes back to primary after bounded sticky successes, with focused regression coverage and no new config knobs in the first pass.

## Reproduction Assessment

Yes at source level: a recoverable primary failure promotes the sticky index, and later successful requests start from that promoted dispatcher. I did not live-reproduce the overnight flaky-network saturation path.

## Solution Assessment

Yes for the repair direction: the narrowest maintainable fix belongs in `extensions/telegram/src/fetch.ts` with tests replacing the current permanent-sticky assertions. Dispatcher pool tuning and config knobs should remain separate unless maintainers choose that policy.
