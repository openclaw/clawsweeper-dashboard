## Summary

Keep open. The report is current, detailed, and still maps to real hot paths in current main: embedded runs synchronously ensure runtime plugins before the first attempt and still run the normal pre-prompt token estimate before provider submission. Current main contains an unreleased runtime-deps hot-path mitigation, but it does not prove the reported first-reply event-loop starvation is fixed or that the remaining broad runtime-plugin fallback load is harmless.

## What This Changes

Investigate and fix first-LLM-call event-loop stalls in v2026.4.29, initially suspected as synchronous token estimation and later refined toward runtime plugin loading and bundled dependency staging.

## Best Possible Solution

Keep this open until a maintainer captures a focused current-main Docker trace for the first post-boot LLM call, split enough to separate runtime plugin loading, bundled dependency/mirror verification, provider/catalog work, prompt assembly, and pre-prompt token estimation. Then fix only the proven owner path, likely by keeping dependency repair off reply hot paths and narrowing any remaining runtime plugin fallback load without weakening plugin hook/tool availability.
