## Summary

Current main already fixes the linked Talk startup error by making the chat error banner dismissible and clearing stale realtime Talk error state; the PR's extra disable-button heuristic is unnecessary and would miss the configured-but-unconfigured provider error that the report used.

## What This Changes

The PR adds a dismiss button for Talk errors and tracks realtime provider availability to disable the Talk button after a missing-provider failure.

## Best Possible Solution

Keep the current-main dismiss-and-clear implementation; any future Talk availability UX should be provider-state driven rather than keyed to one error string.

## Reproduction Assessment

Yes. The linked report gives clear steps for the original missing-provider Talk startup path, and current main now has source-level dismissal wiring plus regression tests for that exact error state.

## Solution Assessment

No for this PR as written. Current main is the narrower maintainable fix, while the PR's disable-button path depends on one error substring and misses the configured-but-unconfigured provider message emitted by the resolver.
