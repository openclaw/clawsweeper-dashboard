## Summary

Current main already contains the WhatsApp active-listener fix for outbound sends missing the registered session, and the changelog records it as shipped in 2026.3.24 with an additional default-account hardening in 2026.4.11.

## Evidence

- **Changelog shipped fix:** The 2026.3.24 changelog explicitly says WhatsApp outbound sends now keep the active Web listener on a direct process-global symbol so split runtime chunks share the connected Baileys session and `openclaw message send --channel whatsapp` stops failing after connect.
  - file: [CHANGELOG.md:2203](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/CHANGELOG.md#L2203)
  - command: `nl -ba CHANGELOG.md | sed -n '2198,2206p'`
- **Process-global listener registry:** Current main stores WhatsApp connection controllers in a `globalThis` singleton keyed by `Symbol.for("openclaw.whatsapp.connectionControllerRegistry")`, which is the implementation that prevents split chunks from seeing different listener maps.
  - file: [extensions/whatsapp/src/connection-controller-registry.ts:11](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/extensions/whatsapp/src/connection-controller-registry.ts#L11)
  - command: `nl -ba extensions/whatsapp/src/connection-controller-registry.ts | sed -n '1,80p'`
- **Outbound send path uses shared registry:** Outbound sends resolve the account id and then read the active listener from the shared registered connection controller before sending, instead of relying on a chunk-local listener reference.
  - file: [extensions/whatsapp/src/send.ts:39](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/extensions/whatsapp/src/send.ts#L39)
  - command: `nl -ba extensions/whatsapp/src/send.ts | sed -n '28,60p'`
- **Regression coverage for replacement listeners:** The controller tests now cover keeping the previous registered controller alive until a replacement listener is ready, which directly guards the missing-listener failure mode during reconnect/replacement.
  - file: [extensions/whatsapp/src/connection-controller.test.ts:96](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/extensions/whatsapp/src/connection-controller.test.ts#L96)
  - command: `nl -ba extensions/whatsapp/src/connection-controller.test.ts | sed -n '70,130p'`
- **Later default-account hardening:** The 2026.4.11 changelog adds another WhatsApp active-listener fix so omitted account ids honor the configured default account instead of falling back to `default`.
  - file: [CHANGELOG.md:847](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/CHANGELOG.md#L847)
  - command: `nl -ba CHANGELOG.md | sed -n '842,850p'`

## Close Comment

Closing as implemented on `main` after Codex review. The current WhatsApp outbound path now resolves sends through a process-global active-listener registry (`extensions/whatsapp/src/connection-controller-registry.ts:11`, `extensions/whatsapp/src/send.ts:39`), which is the exact fix for the "linked/connected but no active listener" failure mode. This was also recorded in the changelog as shipped in `2026.3.24` (`CHANGELOG.md:2203`), with an additional active-listener default-account hardening in `2026.4.11` (`CHANGELOG.md:847`). If you still hit outbound delivery problems on a current build, please open a fresh issue with current version, exact send path, gateway logs, and whether the failure is the explicit "No active WhatsApp Web listener" error versus a silent Baileys delivery failure.
