## Summary

Current main and the shipped v2026.5.6 tag already classify the reported `content_filter` provider finish reason as a failed cron run rather than an `ok` cron success, with delivery status tracked separately.

## What This Changes

The issue requests cron announce runs to surface `finish_reason: content_filter` as an error or filtered delivery state instead of reporting `ok` when no user message is delivered.

## Best Possible Solution

Close this as implemented and keep the existing shipped error classification; use https://github.com/openclaw/openclaw/issues/72985 for the separate fatal-error-delivery accounting follow-up.

## Reproduction Assessment

No. The reported `status: ok` path does not match current main or the v2026.5.6 source path: `content_filter` becomes a provider error, cron classifies the fatal payload as `status: error`, and delivery status is separate.

## Solution Assessment

Yes. The current implementation is the narrow maintainable solution for this issue: provider finish reasons map into the existing error-payload path, and cron records execution outcome separately from delivery outcome.
