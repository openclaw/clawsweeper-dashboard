## Summary

Keep open: current main intentionally protects durable external conversation entries from prune/cap/disk-budget eviction, the reported growth mechanism is source-reproducible, and resolving it needs a new retention/config design under the protected maintainer label.

## What This Changes

Add an explicit lifecycle policy for protected external conversation session entries so Slack channel/thread sessions can preserve routing continuity without retaining full artifacts indefinitely.

## Best Possible Solution

Design a conservative protected-conversation retention tier that trims heavy references into minimal routing records, updates cleanup reporting/docs/schema, and preserves Slack/thread delivery continuity.

## Reproduction Assessment

Yes for the storage-growth mechanism from source inspection: protected Slack channel/thread-style entries are skipped by stale pruning, entry capping, and disk-budget entry eviction. I did not run a live Slack-heavy reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers choose the retention contract. A second-tier protected-session policy is a plausible direction, but the exact config name, default behavior, tombstone shape, and artifact reclamation rules need product/API review before a narrow implementation is safe.
