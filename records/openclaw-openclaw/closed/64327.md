## Summary

Keep open. Current main still has the resolved effective context-token cap internally, but the public plugin hook context, `llm_output` payload, and `model_call_*` payload/context surfaces do not expose that cap; the related context-window telemetry request remains open and adjacent rather than a substitute.

## What This Changes

Expose the agent-resolved effective context-token limit from `contextTokens` config in LLM hook event metadata alongside native model context limits.

## Best Possible Solution

Add backward-compatible optional hook metadata for native context window and resolved effective context-token budget, including source/reference semantics, and wire it through Pi, CLI, Codex harness, and sanitized model-call hook paths with docs and focused tests.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: configure a 200k model with `agents.defaults.contextTokens` at 20k; current tests prove the resolver returns 20k, while the public hook types and emitters expose no field carrying that value.

## Solution Assessment

Yes. The maintainable path is an additive hook-contract update fed from existing `ContextWindowInfo` and `contextTokenBudget` data; plugin-side lookup tables cannot reliably recover the agent-level cap after runtime resolution.
