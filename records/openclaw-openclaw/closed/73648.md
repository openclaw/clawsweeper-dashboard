## Summary

Keep open: current main still has the zombie-task restart bug and this PR targets the right gateway lifecycle path, but the latest head still has a concrete replacement-race correctness bug in the timeout status write, so it is not merge-ready.

## What This Changes

This PR changes gateway channel task lifecycle handling to clear stop-timeout task/abort entries, guard task-owned runtime writes with a current-task promise check, update regression tests, and add a changelog fix entry.

## Best Possible Solution

Land this PR or an equivalent gateway fix after adding the missing guard around timeout status writes, rebasing cleanly, and clearing the affected gateway/channel validation; keep deeper per-channel transport cancellation as a separate follow-up.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Guard timeout status writes after replacement:** `src/gateway/server-channels.ts:689-694`
  - body: The timeout branch only identity-checks before deleting the old task/abort entries, then writes `running: false` and the timeout error unconditionally. If two `stopChannel` calls overlap on the same hung task, the first timeout can clear the maps and a fresh `startChannel` can register; the later timeout will not delete the new task, but it will still overwrite the fresh runtime. Gate this status write on the same task identity or skip it when `store.tasks` now points at another task.
  - confidence: 0.84
