- `P2`: The bug can make WebChat refresh appear to lose recent conversation context, but the underlying transcript remains on disk and the blast radius is limited to WebChat refresh/preview state.
- `impact:session-state`: The issue is about refreshed WebChat state diverging from the stored session transcript and in-memory draft/queue state.

## Summary

Keep open: current main still has a source-backed WebChat refresh failure class because `chat.history` reads a bounded raw transcript tail before display projection, while session preview metadata also remains tail-bounded. The issue also overlaps broader queue/draft refresh persistence choices that need maintainer scoping rather than cleanup closure.

## What This Changes

The issue asks WebChat refresh to restore recent human-visible conversation history and preview metadata even when the transcript tail is dominated by internal tool, tool-result, or silent traffic.

## Best Possible Solution

Backfill enough display-visible WebChat history and preview rows before applying UI limits, while handling queue and draft persistence through a reviewed storage/lifecycle decision or the linked replacement PRs.

## Reproduction Assessment

Yes, source-level. Build a transcript where the last raw history rows are mostly tool, tool-result, or silent assistant records, then refresh WebChat; current main reads a bounded raw tail before projection and can return too few visible rows.

## Solution Assessment

Unclear for the whole issue. The history-tail and preview repair is a narrow maintainable bug fix, but durable queue/draft restoration needs a maintainer decision about storage lifetime, privacy, attachment handling, and replay timing.
