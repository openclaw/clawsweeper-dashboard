## Label Justifications

- `P1`: The report describes real scheduled agent runs stalling until recovery aborts them, which is an urgent provider/cron workflow breakage.
- `impact:session-state`: The failure mode leaves isolated cron sessions stuck in model-call progress until recovery intervenes.
- `impact:auth-provider`: The affected behavior is tied to DeepSeek provider model routing and request/stream timeout handling.

## Summary

Current main already fixes the observable long-stall path by applying the 120s cloud-provider idle watchdog to cron model streams instead of waiting for stuck-session recovery; the remaining DeepSeek-specific `requestTimeoutMs` proposal is not needed to solve the reported stall.

## What This Changes

The issue reports DeepSeek V4 Flash cron streaming stalls in isolated sessions, attributing them to the thinking wrapper ignoring `model.reasoning` and missing DeepSeek request timeouts.

## Best Possible Solution

Keep the current main behavior that applies the cloud-provider idle watchdog to isolated cron model streams; a separate issue should be opened only if DeepSeek still stalls past 120s on a current-main build.

## Reproduction Assessment

Yes at source level: v2026.6.10 disabled the default idle watchdog for cron without an explicit timeout, while current main no longer does. I did not run a live credentialed DeepSeek cron job in this read-only review.

## Solution Assessment

Yes for cleanup: current main fixes the observable long stall by bounding cloud-provider cron stream silence at the idle-timeout layer, which is narrower than adding a DeepSeek-only default timeout. The remaining `model.reasoning=false` idea is not needed for the bundled DeepSeek V4 entries because current catalog marks them `reasoning: true`.
