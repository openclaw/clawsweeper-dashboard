## Summary

Close as implemented on current main: the xAI plugin now declares and registers `x_search`, and plugin tool resolution loads it when the xAI web-search/API-key signal is present.

## What This Changes

The issue requests that agents can see and use the xAI `x_search` tool when `plugins.entries.xai.config.xSearch` is enabled with xAI credentials configured.

## Best Possible Solution

Keep the plugin-owned implementation on main and let the next release carry the fix; users still affected on newer builds should provide fresh runtime status and redacted config evidence.

## Reproduction Assessment

No current-main failure path is evident. Source inspection shows the reported configuration now satisfies the xAI plugin tool availability path, and focused regression tests cover `x_search` visibility from xAI auth/config signals.

## Solution Assessment

Yes. The current fix keeps `x_search` owned by the xAI plugin and uses generic manifest metadata/tool-resolution seams, which is the maintainable boundary for this provider-specific tool.
