## Summary

Keep open: current main still has a source-visible path where stale-session recovery can classify queued work as recoverable, then skip lane cleanup because stale reply state is still reported as active work. The related open PR targets one likely session-reset root cause, but it is not merged and does not yet prove the Telegram group crash/cancellation case is fully fixed.

## What This Changes

The issue reports Telegram group session lanes staying stuck in processing when stale active reply state makes recovery keep the lane and queue inbound messages indefinitely.

## Best Possible Solution

Keep this issue open while maintainers land or expand https://github.com/openclaw/openclaw/pull/80427, then add regression or live proof that stale active_reply_work cannot block queued Telegram group lanes past the stuck-session abort threshold.

## Reproduction Assessment

Yes, at source level: current main can classify queued work without active diagnostic work as stale_session_state, then recoverStuckDiagnosticSession can skip with active_reply_work when the reply registry still reports that session active. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

Yes directionally: recovery should not trust stale active reply state forever. The linked reset-cleanup PR is a likely narrow part of the fix, but the final solution should prove the Telegram group lane recovers or add broader stale reply liveness handling.
