## Summary

Keep open: the PR fixes the normal xAI auth-profile path, but it misses plugin-only tool construction, so an xAI-only allowlist can still fail to register `x_search` or `code_execution` when the only credential is an auth profile.

## What This Changes

The PR threads auth-profile availability and API-key resolution helpers into plugin tool context, updates bundled xAI tool registration/execution to use them, and adds docs, tests, and a changelog entry.

## Best Possible Solution

Forward the auth profile store through the plugin-only construction branch, cover an xAI-only allowlist regression, and then land the focused auth-profile fix after CI stays green.

## Reproduction Assessment

Yes, source-level. With an auth-profile-only xAI credential and a narrow allowlist such as `x_search`, the construction plan uses the plugin-only branch, but that branch does not pass `authProfileStore`, so `hasAuthForProvider` remains unavailable.

## Solution Assessment

No, not yet. The PR is the right shape for normal plugin tool construction, but it should also wire the same auth helpers through the plugin-only path before merge.
