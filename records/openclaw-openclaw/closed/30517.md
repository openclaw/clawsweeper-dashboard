## Summary

Current `main` already supports proactive post-startup agent actions through the bundled `boot-md` gateway startup hook and `BOOT.md`, which can send outbound status messages using the message tool after channels are up.

## Evidence

- **Gateway startup event is dispatched after channel startup:** `startGatewaySidecars` triggers the internal `gateway:startup` hook after channels finish starting.
  - file: [src/gateway/server-startup-post-attach.ts:268](https://github.com/openclaw/openclaw/blob/c6684af682bcb59bd1ee85cb49f72d4074007e2c/src/gateway/server-startup-post-attach.ts#L268)
  - sha: [c6684af682bc](https://github.com/openclaw/openclaw/commit/c6684af682bcb59bd1ee85cb49f72d4074007e2c)
- **Bundled startup hook is documented:** Docs list `boot-md` as a bundled hook for `gateway:startup` and state that it runs `BOOT.md` when the gateway starts.
  - file: [docs/automation/hooks.md:152](https://github.com/openclaw/openclaw/blob/c6684af682bcb59bd1ee85cb49f72d4074007e2c/docs/automation/hooks.md#L152)
  - sha: [c6684af682bc](https://github.com/openclaw/openclaw/commit/c6684af682bcb59bd1ee85cb49f72d4074007e2c)
- **BOOT flow can proactively send messages:** `runBootOnce` builds a startup prompt that explicitly tells the agent to use the `message` tool to send outbound messages, and it runs against the agent main session.
  - file: [src/gateway/boot.ts:43](https://github.com/openclaw/openclaw/blob/c6684af682bcb59bd1ee85cb49f72d4074007e2c/src/gateway/boot.ts#L43)
  - sha: [c6684af682bc](https://github.com/openclaw/openclaw/commit/c6684af682bcb59bd1ee85cb49f72d4074007e2c)
- **Per-agent startup execution is covered by tests:** Integration test verifies `gateway:startup` runs BOOT for each configured agent scope.
  - file: [src/hooks/bundled/boot-md/handler.gateway-startup.integration.test.ts:35](https://github.com/openclaw/openclaw/blob/c6684af682bcb59bd1ee85cb49f72d4074007e2c/src/hooks/bundled/boot-md/handler.gateway-startup.integration.test.ts#L35)
  - sha: [c6684af682bc](https://github.com/openclaw/openclaw/commit/c6684af682bcb59bd1ee85cb49f72d4074007e2c)
- **Feature shipped in latest release:** The same `boot-md` docs and `src/gateway/boot.ts` implementation are present at release commit `00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5` (`v2026.4.22`).
  - command: `git show 00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5:src/gateway/boot.ts | sed -n '1,220p'`
  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)

## Risks / Open Questions

- This is implemented via the existing startup hook and `BOOT.md` workflow, not the exact `agents.defaults.startup` config shape proposed in the issue.

## Close Comment

Codex review against current `main` found this is already supported through the existing Gateway startup hook flow. `src/gateway/server-startup-post-attach.ts:268` dispatches `gateway:startup` after channels start, `docs/automation/hooks.md:152` documents the bundled `boot-md` hook for that event, and `src/gateway/boot.ts:43` explicitly allows `BOOT.md` startup runs to use the `message` tool for proactive outbound sends while running against the agent’s main session context. The behavior is also covered by `src/hooks/bundled/boot-md/handler.gateway-startup.integration.test.ts:35`. This path is present in release `v2026.4.22` at `00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`, so I’m closing this as already implemented on main, via the existing `boot-md`/`BOOT.md` startup workflow rather than a new `agents.defaults.startup` config block.
