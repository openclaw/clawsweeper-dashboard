## Summary

Close as implemented. The Teams installed-app symptom now matches the diagnosed bundled runtime-dependency JWT regression: current main and the v2026.4.27 release unwrap `jsonwebtoken`'s CommonJS default export before Bot Framework token validation, and the release changelog calls out inbound Teams 401s after the runtime-deps move as fixed.

## What This Changes

Messages sent from an installed Microsoft Teams app should reach the configured OpenClaw `/api/messages` webhook and receive replies when Azure Bot Web Chat already works against the same endpoint.

## Best Possible Solution

Keep the v2026.4.27 Teams JWT runtime fix as the shipped answer for this regression. Users still seeing installed Teams DMs fail on v2026.4.27 or newer should file a fresh report with redacted reverse-proxy status, OpenClaw debug logs, and JWT header/claim shape so maintainers can separate setup issues from webhook validation defects.

## Reproduction Assessment

Yes. The original report gives a concrete Azure-backed Teams CLI setup and Web Chat versus installed Teams DM comparison, and #73754 adds a matching valid-token diagnostic for the same webhook path. I did not reproduce live locally because this requires Azure/Teams access.

## Solution Assessment

Yes for the diagnosed regression. Unwrapping the staged CommonJS JWT module before validation is the narrowest maintainable fix because it restores `verify` while preserving signature, audience, issuer, expiration, and configured-app binding checks.
