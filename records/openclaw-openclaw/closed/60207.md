## Summary

Keep open. Current main still has heartbeat-specific duplicate suppression and several inbound/message-id dedupe paths, but I do not see the requested shared cross-source outbound suppression between heartbeat-generated replies and normal assistant replies. The related items in the provided timeline look adjacent, not a canonical replacement for this remaining core behavior gap.

## Evidence

- **Heartbeat dedupe is heartbeat-local:** Heartbeat suppression compares the current heartbeat text only against `entry.lastHeartbeatText` and `lastHeartbeatSentAt` within a 24h window, then records only the last delivered heartbeat payload. It does not consult normal assistant delivery history.
  - file: [src/infra/heartbeat-runner.ts:1147](https://github.com/openclaw/openclaw/blob/b64bfc5d9abf078217a0a63c25d4c2bcbf4e6fab/src/infra/heartbeat-runner.ts#L1147)
  - command: `nl -ba src/infra/heartbeat-runner.ts | sed -n '1080,1360p'`
  - sha: [b64bfc5d9abf](https://github.com/openclaw/openclaw/commit/b64bfc5d9abf078217a0a63c25d4c2bcbf4e6fab)
- **Normal replies use a separate outbound route:** The normal reply path claims inbound dedupe at message intake, then later sends final replies via `sendFinalPayload`/`routeReplyToOriginating` or the dispatcher. That dedupe is about duplicate inbound provider messages, not equivalent outbound content recently sent by heartbeat.
  - file: [src/auto-reply/reply/dispatch-from-config.ts:331](https://github.com/openclaw/openclaw/blob/b64bfc5d9abf078217a0a63c25d4c2bcbf4e6fab/src/auto-reply/reply/dispatch-from-config.ts#L331)
  - command: `nl -ba src/auto-reply/reply/dispatch-from-config.ts | sed -n '274,1280p'`
  - sha: [b64bfc5d9abf](https://github.com/openclaw/openclaw/commit/b64bfc5d9abf078217a0a63c25d4c2bcbf4e6fab)
- **Routed replies directly enter shared delivery:** `routeReply` normalizes the reply and calls `deliverOutboundPayloads` with the payload and routing context; there is no cross-source dedupe key or recent-content suppression policy in that call.
  - file: [src/auto-reply/reply/route-reply.ts:211](https://github.com/openclaw/openclaw/blob/b64bfc5d9abf078217a0a63c25d4c2bcbf4e6fab/src/auto-reply/reply/route-reply.ts#L211)
  - command: `nl -ba src/auto-reply/reply/route-reply.ts | sed -n '1,280p'`
  - sha: [b64bfc5d9abf](https://github.com/openclaw/openclaw/commit/b64bfc5d9abf078217a0a63c25d4c2bcbf4e6fab)
- **Shared delivery is transport/queue oriented:** `deliverOutboundPayloads` persists a write-ahead queue entry and sends normalized payloads through channel handlers and optional hooks. The queue entry is per send, and the delivery core does not maintain a shared recent-content cache across heartbeat and normal reply sources.
  - file: [src/infra/outbound/deliver.ts:757](https://github.com/openclaw/openclaw/blob/b64bfc5d9abf078217a0a63c25d4c2bcbf4e6fab/src/infra/outbound/deliver.ts#L757)
  - command: `nl -ba src/infra/outbound/deliver.ts | sed -n '757,1280p'`
  - sha: [b64bfc5d9abf](https://github.com/openclaw/openclaw/commit/b64bfc5d9abf078217a0a63c25d4c2bcbf4e6fab)
- **Queue ids are unique per send:** `enqueueDelivery` generates a fresh secure UUID for each queued delivery, which supports crash recovery but does not provide stable content idempotency across independent producers.
  - file: [src/infra/outbound/delivery-queue-storage.ts:133](https://github.com/openclaw/openclaw/blob/b64bfc5d9abf078217a0a63c25d4c2bcbf4e6fab/src/infra/outbound/delivery-queue-storage.ts#L133)
