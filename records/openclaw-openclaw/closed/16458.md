## Summary

The reported brute-force path assumes an untrusted/public pairing-code validation surface, but current `main` and the latest shipped release only redeem DM pairing codes through trusted owner/operator approval paths. Unapproved DMs create/send pairing challenges and are ignored; I could not find a remotely reachable production path that lets an unknown sender submit repeated code guesses.

## Close Comment

Closing after a Codex review because I could not reproduce a remotely reachable brute-force path on current `main` or the latest shipped release `v2026.4.22`. DM pairing is documented as an explicit owner approval step (`docs/channels/pairing.md:10`), and unknown senders only create/send a pairing challenge via `issuePairingChallenge(...)` (`src/pairing/pairing-challenge.ts:24`); their message is not processed until approval. On current `main`, the only production caller of `approveChannelPairingCode(...)` is the trusted/local `openclaw pairing approve` CLI flow (`src/cli/pairing-cli.ts:107`, `src/pairing/pairing-store.ts:630`), and a repo search for `approveChannelPairingCode` did not turn up any channel, webhook, or gateway path that would let an unknown sender submit repeated code guesses. I also compared these paths to `v2026.4.22`; the code is unchanged there. If there is a concrete unauthenticated or otherwise untrusted channel/plugin flow that redeems DM pairing codes remotely, please open a new issue with that exact path and reproduction steps.
