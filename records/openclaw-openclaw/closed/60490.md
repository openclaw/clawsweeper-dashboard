## Summary

Current main and the v2026.4.26 tag implement the Windows gateway service recovery paths behind this report: Scheduled Task registration is no longer treated as sufficient, task-run no-ops fall back to launching the staged gateway script, restart waits for an actual healthy listener, and Windows docs describe the Scheduled Task plus Startup-folder fallback. The later workaround comment about a registered/ready task with no 127.0.0.1:18789 listener matches the now-covered upstream path.

## Best Possible Solution

Close this as implemented on main and keep the current Windows Scheduled Task/Startup-folder fallback plus health-check behavior. If a user can still reproduce a login/reboot no-listener state on v2026.4.26 or newer, the best follow-up is a fresh issue with `openclaw gateway status --deep`, Task Scheduler state, port listener output, and gateway logs from that version.
