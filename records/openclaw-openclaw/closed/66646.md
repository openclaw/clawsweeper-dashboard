## Summary

Keep open. Current main no longer classifies typed session write-lock timeouts as provider timeouts, but the fallback runner still advances unclassified errors to later candidates, so direct lock contention can still cascade through the model fallback chain.

## What This Changes

The issue asks OpenClaw to treat session write-lock contention as infrastructure/session-busy instead of advancing through model fallback candidates.

## Best Possible Solution

Add a narrow session write-lock/infrastructure fail-fast branch at the model fallback or agent-attempt boundary, while preserving generic unknown-error fallback and leaving stale-lock reclaim plus queue/backoff policy unchanged unless separately accepted.

## Reproduction Assessment

Yes. A source-level reproduction is clear: configure multiple fallback candidates and make the first candidate callback throw `SessionWriteLockTimeoutError`; current main suppresses timeout classification but still advances because unclassified errors continue while candidates remain.

## Solution Assessment

No. The current classifier-only fix is useful but incomplete because returning null still flows through the generic unknown-error fallback path; the safer repair is an explicit session-lock/infrastructure escape before recording or advancing a model candidate.
