## Summary

Keep open: current main and latest release still have a source-visible gate where `active_work_without_progress` on an embedded run logs `recovery=none` until processing-session age reaches `diagnostics.stuckSessionAbortMs`, even when the run's own `lastProgressAgeMs` has already crossed that threshold. The reported webchat log matches that gap (`age=121s`, `lastProgressAge=364s`, queue depth present), so the old zombie-run and active-reply-work fixes do not fully cover this case.

## What This Changes

The issue reports webchat sessions remaining stuck in `processing` after a network interruption, with gateway reconnect succeeding but embedded-run session recovery not firing until manual gateway restart.

## Best Possible Solution

Update the diagnostics active-abort eligibility so stalled embedded-run recovery can fire from stale progress age under the existing `stuckSessionAbortMs` contract, then add a regression test for the reported `age < abort threshold` and `lastProgressAge >= abort threshold` shape.

## Reproduction Assessment

Yes at source level: current main can produce the reported `recovery=none` shape because embedded-run recovery checks processing-session age while the report's stale-progress age is already past the default abort threshold. I did not run a live WSL/webchat network cutover reproduction.

## Solution Assessment

Yes, the best fix is narrow: the recovery runtime already knows how to abort, drain, and force-clear stale embedded work once called, so the missing piece is the heartbeat eligibility condition for stale embedded-run progress.

## Repair Work Prompt

Fix the webchat/network-interruption stalled-session recovery gap reported in https://github.com/openclaw/openclaw/issues/92635. Current main and v2026.6.6 can log `activeWorkKind=embedded_run lastProgressAge>=diagnostics.stuckSessionAbortMs recovery=none` when processing session age is still below the abort threshold, especially because queued messages reset diagnostic session `lastActivity`; this can leave queued work behind a stale embedded run. Keep the fix in the diagnostics/session recovery boundary. Do not add config options, change defaults, reset sessions automatically, or alter unrelated channel/provider behavior. Likely update `src/logging/diagnostic.ts` so active embedded-run abort eligibility uses stale no-progress age where appropriate, consistent with the docs for `diagnostics.stuckSessionAbortMs`, while preserving the existing safeguards for long-running embedded work that is still making progress. Add or update focused regression coverage in `src/logging/diagnostic.test.ts` for a processing embedded run with `queueDepth > 0`, session age below the abort threshold, and `lastProgressAgeMs` at or above the abort threshold, asserting recovery is requested with `allowActiveAbort: true`. Also run the stuck-session recovery runtime tests to guard abort/drain behavior. No `CHANGELOG.md` edit.
