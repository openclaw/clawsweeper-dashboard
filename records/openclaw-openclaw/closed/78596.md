## Summary

Current main now blocks the reported Telegram heartbeat fallback path: tool-capable heartbeats use the structured heartbeat response tool with `message_tool_only` delivery, and Telegram suppresses its no-visible-response silent fallback for that mode. The fix is on main only; it is not in `v2026.5.7`.

## What This Changes

The issue asks OpenClaw to stop surfacing the synthetic `No further note from me.` silent-reply fallback as a regular Telegram `claw-bridge` message after heartbeat or async completion.

## Best Possible Solution

Keep the current main implementation and let the next release carry the structured heartbeat response and Telegram fallback-suppression behavior.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No live Telegram reproduction was run; the old source path was clear from the prior review, and current main now has source and regression coverage showing that path is suppressed for tool-capable heartbeat runs.

## Solution Assessment

Is this the best way to solve the issue? Yes; suppressing Telegram's silent fallback only for `message_tool_only` heartbeat delivery fixes the system-noise leak without removing the intentional direct-chat silent-reply rewrite policy globally.
