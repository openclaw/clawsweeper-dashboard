## Summary

Keep open. The underlying regression is real on current main, but this PR still has a blocking correctness issue: its context-free suppression check treats some conditional manifest suppressions as unconditional, which can reject valid configured inline model rows for third-party plugins.

## What This Changes

The PR adds an inline-model suppression check in `resolveExplicitModelWithRegistry`, a regression test for configured `openai-codex/gpt-5.4-mini` rows with `api`, and a changelog entry.

## Best Possible Solution

Keep the inline OpenAI Codex rejection, but implement it through an explicit unconditional-suppression path or corrected condition semantics where missing config fails `providerConfigApiIn`; add coverage for both the stale OpenAI inline row and an API-only conditional suppression case.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Use an unconditional-only suppression check:** `src/agents/pi-embedded-runner/model.ts:595-598`
  - body: The new no-context call assumes missing `config` means only unconditional suppressions match. In the manifest matcher, `providerConfigApiIn` is skipped when no configured API is present, so an API-only conditional suppression can now reject a valid configured inline row. Please add an explicit unconditional-suppression path, or make missing config fail `providerConfigApiIn`, with coverage for that case.
  - confidence: 0.89
