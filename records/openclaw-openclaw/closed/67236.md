## Summary

Current main already carries QQ message ids into the shared auto-reply inbound dedupe claim, which skips duplicate or inflight repeats before the agent/reply resolver can produce another visible reply. The earlier ClawSweeper comment correctly identified that the QQ queue itself does not dedupe, but the reported duplicate `deliver`/`onMessageSent` behavior is covered downstream by the dispatch-level claim.

## Best Possible Solution

Keep the shared dispatch-level inbound dedupe as the canonical reply-once guard for QQ Bot. Add an earlier QQ-owner short-TTL cache only if maintainers want to suppress duplicate typing or attachment preprocessing; no core issue remains for duplicate visible agent replies.

## Risks / Open Questions

- The dedupe claim runs after QQ inbound context construction, so duplicate QQ pushes may still repeat earlier non-reply work such as typing indicators or attachment preprocessing before the shared dispatcher skips the duplicate.
- There is generic `MessageSid` + origin regression coverage, but not a QQ-specific WebSocket replay test that sends the same QQ event two or three times end to end.
