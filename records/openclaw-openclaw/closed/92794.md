## Summary

Current main and the latest release already have a Workboard execution writer: dispatcher-started runs build `card.execution`, the store derives attempt metadata from it, and the SQLite store persists it into the `execution_*` columns. A database can still show zero populated rows if no Workboard dispatcher-started execution has occurred, but the reported missing application write path is present.

## What This Changes

The issue reports that denormalized Workboard card `execution_*` SQLite columns exist but are never populated by application code.

## Best Possible Solution

Keep the existing dispatcher-owned `card.execution` writer and SQLite persistence path; do not add attempt-to-card SQL triggers that would duplicate or invert the current source of truth.

## Reproduction Assessment

No. Current main has a concrete dispatcher-to-store-to-SQLite write path for `execution_status`, so the reported missing application writer does not reproduce from source inspection.

## Solution Assessment

Yes. The current implementation is the maintainable direction because the dispatcher owns the run id/session/model details and the store derives attempts from `card.execution`, rather than trying to reconstruct incomplete execution rows from attempt-table triggers.
