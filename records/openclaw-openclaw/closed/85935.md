## Summary

Keep open: current main still wires the Sessions view Refresh button to a non-append `sessions.list` reload that replaces the in-memory result. The related pagination work addressed bounded responses and Load More, but it did not add a manual delta refresh path, and the right fix needs a maintainer choice on Gateway/API and UI semantics.

## What This Changes

The issue asks for the Control UI sessions refresh path to avoid full list replacement and instead use incremental, delta, or state-preserving refresh behavior.

## Best Possible Solution

Keep the issue open for a focused maintainer decision: either add a well-specified bounded delta refresh contract, or implement a smaller state-preserving full-refresh fallback while retaining the existing default limit and pagination behavior.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: click Refresh in `/sessions`, which calls `loadSessions(state)`, requests `sessions.list`, and replaces `state.sessionsResult` for non-append loads. I did not run a browser repro because this was a read-only cleanup review.

## Solution Assessment

Unclear as written: the report lists several possible fixes, and the durable delta approach needs a Gateway contract decision. The safest narrow fallback would preserve UI state across the already-bounded reload if maintainers do not want a new delta API yet.
