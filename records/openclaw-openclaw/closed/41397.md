## Summary

Current main already avoids the reported Twilio `<Say>` greeting fallback by deferring Twilio conversation-mode initial playback until the media-stream start/connect path is active, registering the stream before speaking, and only allowing TwiML fallback when no active stream exists.

## Evidence

- **Conversation-mode greeting is deferred for Twilio streaming:** `maybeSpeakInitialMessageOnAnswered` returns early in conversation mode when the Twilio stream-connect path is enabled, so the initial greeting is not spoken on `call.answered`.
  - file: [extensions/voice-call/src/manager.ts:305](https://github.com/openclaw/openclaw/blob/e94c0bf51542fe518e49ad4e2ee29882891b5603/extensions/voice-call/src/manager.ts#L305)

  - sha: [e94c0bf51542](https://github.com/openclaw/openclaw/commit/e94c0bf51542fe518e49ad4e2ee29882891b5603)
- **Greeting is triggered only after accepted stream start is registered:** The media-stream handler calls `onConnect` only after the Twilio `start` frame is accepted and the session is stored; the webhook then registers the stream before invoking `speakInitialMessage`.
  - file: [extensions/voice-call/src/media-stream.ts:317](https://github.com/openclaw/openclaw/blob/e94c0bf51542fe518e49ad4e2ee29882891b5603/extensions/voice-call/src/media-stream.ts#L317)
