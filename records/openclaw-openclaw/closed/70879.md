## Label Justifications

- `P2`: This is a normal-priority Feishu delivery reliability bug with a clear plugin-owned surface and limited channel-specific blast radius.
- `impact:message-loss`: The reported rate-limit failures can cause outbound Feishu replies or media messages to be dropped after a terminal send failure.

## Summary

Keep open: current main still treats Feishu outbound rate-limit delivery failures as terminal on the main text/card, media, and initial streaming-card send paths, while the existing backoff classifier is scoped to typing reactions.

## What This Changes

The issue asks for Feishu outbound message delivery to retry retryable rate-limit failures such as flow-control and HTTP 429 responses.

## Best Possible Solution

Add a Feishu-owned bounded retry/backoff helper for explicit retryable delivery failures, apply it consistently to create/reply sends, and keep proactive pacing plus delivery-failure notification work on their separate trackers unless maintainers intentionally merge the scopes.

## Reproduction Assessment

Yes, at source level: mock Feishu client.im.message.create or reply, or the initial streaming-card send, to throw or return a retryable nonzero code; current main makes one delivery attempt and then throws. I did not run a live Feishu rate-limit scenario in this read-only review.

## Solution Assessment

Yes. A narrow Feishu-plugin retry wrapper around outbound delivery sends is the maintainable fix; it should reuse or factor retryable classification and avoid a core retry layer or new public config option.
