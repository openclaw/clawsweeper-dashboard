## Summary

Keep open. Current `main` has absorbed some adjacent fixes around voice-call runtime teardown and OAuth `refresh_token_reused` recovery, but it does not implement this PR's fail-closed external-webhook guard or the OAuth duplicate-profile cleanup described in the PR.

## Evidence

- **Voice-call guard is not on main:** `createVoiceCallRuntime` still computes `const webhookUrl = publicUrl ?? localUrl` and proceeds to initialize the manager without checking whether Twilio/Telnyx/Plivo are being given a loopback-only callback URL.
  - file: [extensions/voice-call/src/runtime.ts:295](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/extensions/voice-call/src/runtime.ts#L295)
  - sha: [7ac35b4f6900](https://github.com/openclaw/openclaw/commit/7ac35b4f69009490180a35bf404638d4280a4865)
- **Current tests do not cover the PR's webhook fail-closed behavior:** The current runtime test file covers teardown-on-failure, idempotent stop, and `fullConfig` wiring, but there is no test for rejecting Twilio/Telnyx/Plivo when the webhook URL resolves to localhost/loopback.
  - file: [extensions/voice-call/src/runtime.test.ts:72](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/extensions/voice-call/src/runtime.test.ts#L72)
  - sha: [7ac35b4f6900](https://github.com/openclaw/openclaw/commit/7ac35b4f69009490180a35bf404638d4280a4865)
- **OAuth upsert still does no duplicate cleanup:** Both `upsertAuthProfile` and `upsertAuthProfileWithLock` simply write `store.profiles[profileId]` and save/update the store. There is no cleanup pass that removes duplicate OAuth profiles for the same provider/email or repairs related `order`/`lastGood`/`usageStats` references.
  - file: [src/agents/auth-profiles/profiles.ts:62](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/src/agents/auth-profiles/profiles.ts#L62)
  - sha: [7ac35b4f6900](https://github.com/openclaw/openclaw/commit/7ac35b4f69009490180a35bf404638d4280a4865)
- **Part of the PR's auth motivation is now handled differently on main:** `oauth-manager.ts` already has explicit `refresh_token_reused` recovery logic: after a refresh failure it reloads the store, attempts to recover a changed stored credential, and retries the refresh path. That means the PR's auth piece is not fully obsolete, but it would need re-evaluation against newer mainline behavior rather than a straight close as already implemented.
  - file: [src/agents/auth-profiles/oauth-manager.ts:551](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/src/agents/auth-profiles/oauth-manager.ts#L551)
  - sha: [7ac35b4f6900](https://github.com/openclaw/openclaw/commit/7ac35b4f69009490180a35bf404638d4280a4865)
- **Adjacent lifecycle cleanup already landed separately:** Main now wraps post-start voice-call initialization in a `try/catch` and calls `lifecycle.stop({ suppressErrors: true })` on failure, covering the leaked-resource review concern, but this is distinct from the PR's missing webhook reachability guard.
  - file: [extensions/voice-call/src/runtime.ts:267](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/extensions/voice-call/src/runtime.ts#L267)
  - sha: [7ac35b4f6900](https://github.com/openclaw/openclaw/commit/7ac35b4f69009490180a35bf404638d4280a4865)
