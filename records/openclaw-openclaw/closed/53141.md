## Summary

Current `main` no longer matches the reported behavior: oversized auto-TTS replies are kept on the speech path by summarize-then-speak or truncate-then-speak fallback, and Telegram + ElevenLabs still route successful speech output as voice notes rather than silently dropping to text-only.

## Evidence

- **Long replies stay on the TTS path:** `maybeApplyTtsToPayload` no longer skips long replies when they exceed the soft TTS limit. If summarization is disabled it truncates and continues; if summarization is enabled it summarizes; if summarization fails it truncates and continues.
  - file: [extensions/speech-core/src/tts.ts:1116](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/extensions/speech-core/src/tts.ts#L1116)
  - command: `nl -ba extensions/speech-core/src/tts.ts | sed -n '1116,1143p'`
- **Successful auto-TTS still attaches audio:** After synthesis succeeds, the reply payload is returned with `mediaUrl`, and native voice-note channels set `audioAsVoice` when the provider says the output is voice-compatible.
  - file: [extensions/speech-core/src/tts.ts:1161](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/extensions/speech-core/src/tts.ts#L1161)
  - command: `nl -ba extensions/speech-core/src/tts.ts | sed -n '1152,1180p'`
