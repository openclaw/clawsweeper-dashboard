## Summary

## What This Changes

The issue reports that Docker sandbox sessions with `workspaceAccess: "none"` mount the isolated `/workspace` read-only, causing sandbox read/write tool failures.

## Best Possible Solution

Use #37634 as the canonical product fix: decide the `workspaceAccess: "none"` contract there, preserve this issue's Docker/Colima evidence by linking it, and implement the sandbox permission change with focused mount, bridge, docs, and regression coverage.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: Docker setup selects `workspaceAccess: "none"`, current sandbox resolution chooses an isolated workspace, the Docker mount is read-only, and the bridge rejects writes unless access is `"rw"`.
