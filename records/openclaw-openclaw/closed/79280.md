## Summary

Current main already avoids the reported CLI transcript data-loss path by writing CLI turn transcripts directly before any context-engine compaction work, so the missing legacy-engine registration no longer drops the session file.

## What This Changes

The issue requests that gateway/CLI-runner sessions persist transcripts reliably even when the legacy context engine has not been initialized by an embedded or subagent path first.

## Best Possible Solution

Keep the direct CLI transcript persistence on main; track any remaining legacy-engine compaction warning separately only if it still affects current builds after the transcript file is written.

## Reproduction Assessment

No for current main: the reported data-loss path does not match the current CLI transcript code, which writes the session file before any context-engine compaction resolution. The old release logs are plausible, but current source and test coverage show the transcript file creation path is independent of the legacy-engine registry.

## Solution Assessment

Yes: decoupling CLI transcript writes from context-engine resolution is a narrower maintainable fix for the observed data loss than forcing gateway boot to eagerly load more runtime solely for persistence. If maintainers want to eliminate a later compaction warning too, that should be a focused follow-up around `runCliTurnCompactionLifecycle`.
