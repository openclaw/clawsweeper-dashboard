## Summary

Close as duplicate/superseded. The release behavior the reporter saw is real, but current main has already adopted the narrower #63654/#63987 outcome: keep the built-in Qwen Coding Plan catalog conservative while allowing explicitly configured `qwen/qwen3.6-plus` entries to resolve on Coding Plan base URLs. The broader request to remove the guards entirely matches the already-reviewed #66367 direction, not a new untracked core bug.

## What This Changes

The issue asks OpenClaw to stop suppressing `qwen3.6-plus` for Qwen Coding Plan endpoints because the reporter's direct API call to the Coding Plan endpoint succeeds.

## Best Possible Solution

Keep the current main policy unless maintainers explicitly re-evaluate provider availability: do not advertise `qwen3.6-plus` by default on Qwen Coding Plan, but allow users whose subscription supports it to opt in through explicit provider model config. If Alibaba's Global/Intl Coding Plan docs and behavior become consistently supported, a focused Qwen plugin change can remove the catalog filter, manifest suppression, and matching docs/tests together.

## Reproduction Assessment

Yes. The release/default-catalog behavior is reproducible from source: Coding Plan base URLs still filter and suppress the built-in `qwen3.6-plus` row, while current main also has a tested explicit-config path that resolves the model.

## Solution Assessment

Yes for the current cleanup decision. The existing opt-in policy is the narrowest maintainable resolution given provider-doc inconsistency and account-specific availability; removing all suppressions should be a separate maintainer product decision, not a fresh duplicate issue.
