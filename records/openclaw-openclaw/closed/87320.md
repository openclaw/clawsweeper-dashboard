## Summary

Keep open. Current main still treats an explicit semver `--tag` as a resolved target without confirming the npm registry has published it, then dry-run exits before the package-manager install step that would fail on `openclaw@<version>`.

## What This Changes

Request that `openclaw update --dry-run` warn or fail when a requested npm version is not yet available from the npm registry instead of green-lighting an apply that later fails.

## Best Possible Solution

Add a scoped registry availability preflight for exact registry version tags in the package-update dry-run/apply planning path, with a clear propagation-lag warning or error and a short docs note.

## Reproduction Assessment

Yes from source inspection and the production observation: exact semver tags bypass npm lookup in dry-run while apply installs `openclaw@<version>`. The same `2026.5.26` registry gap has passed, so a high-confidence regression test should mock npm returning no exact version.

## Solution Assessment

Yes, the narrowest fix is to make dry-run/apply planning surface missing exact registry targets before reporting a clean plan. Automatic fallback to a GitHub package spec would change update source semantics and should not be the default repair.

In the package-manager `openclaw update` path, make `--dry-run --tag <exact semver>` check or report npm registry availability for registry targets instead of treating a parsed semver as fully resolved when npm has not published it yet.
