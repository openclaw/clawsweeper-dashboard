## Summary

Close as implemented on current main: the reported v2026.4.27 Discord restart failure is covered by the current Discord-owned gateway implementation, which replaced the Carbon gateway path, tracks READY/RESUMED directly, retries startup READY stalls with a fresh identify, and has regression coverage for the reported 1006/reconnect/not-ready lifecycle states. I could not prove the proof commit is in a shipped tag, so this is main-only evidence rather than a released fix.

## What This Changes

The issue asks for the bundled Discord channel to resume responding after a macOS gateway restart instead of staying at "awaiting gateway readiness" with 1006 reconnects and no outbound replies.

## Best Possible Solution

Ship the current Discord gateway implementation and ask for a fresh report only if the same restart behavior reproduces on a build containing 4aa08e9d or a later release, with gateway logs around READY/RESUMED, close codes, and channel status probes.

## Reproduction Assessment

Yes. The issue gives a concrete v2026.4.27 path: onboard Discord, restart the macOS LaunchAgent, then observe the awaiting-readiness log and 1006 reconnect loop; current-main source and tests exercise the matching startup and runtime READY-stall paths.

## Solution Assessment

Yes. Keeping the fix inside the Discord plugin/gateway lifecycle is the narrow owner-boundary solution; no core migration or generic channel change is needed for a Discord-specific gateway READY regression.
