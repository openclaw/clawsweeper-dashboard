## Summary

This PR should close as superseded: it targets the real inline tool-dispatch policy bug from #75124, but the branch is still incorrect and the active #75125 already tracks the same fix using the safer shared tool-builder direction.

## What This Changes

The PR modifies `src/auto-reply/reply/get-reply-inline-actions.ts` so `command-dispatch: tool` skill slash commands run through a manually assembled tool-policy pipeline before selecting the requested tool.

## Best Possible Solution

Close this PR in favor of the active shared-builder fix path in #75125, or replace it with an equivalent branch that reuses the canonical tool construction context.

## Reproduction Assessment

Yes. A static reproduction exists with a `toolsBySender` group policy and a tool-dispatch skill for a denied tool; this PR does not pass sender/account fields into group policy resolution.

## Solution Assessment

No. The direction is right, but manually duplicating the policy pipeline is not the safest solution; using `createOpenClawCodingTools(...)` or a shared effective-tool helper avoids the missing context and active-context gating bugs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Pass sender context into group policy:** `src/auto-reply/reply/get-reply-inline-actions.ts:323-327`
  - body: The new inline policy path calls `resolveGroupToolPolicy` with only config, session key, and group id. Sender/account metadata is required for `toolsBySender` matching, while the normal tool builder forwards those fields, so a sender-scoped group deny can still be bypassed by a tool-dispatch skill command.
  - confidence: 0.86
- **[P2] Gate sandbox and subagent policy to active contexts:** `src/auto-reply/reply/get-reply-inline-actions.ts:337-338`
  - body: These helpers are not neutral fallbacks: sandbox policy returns default sandbox denies, and subagent policy returns main-role denies even for ordinary sessions. Calling both unconditionally makes host/non-subagent skill commands lose tools that normal agent turns would still expose.
  - confidence: 0.91

## Security Review

Status: needs_attention

Summary: The PR still leaves a concrete sender-scoped group tool-policy bypass on a user-invocable inline slash-command surface.

Concerns:

- **[high] Sender-scoped group policy remains bypassable:** `src/auto-reply/reply/get-reply-inline-actions.ts:323`
  - body: The PR does not pass sender/account metadata into `resolveGroupToolPolicy`, so `channels.*.groups.*.toolsBySender` rules can fail to match and a tool-dispatch skill can see a tool that the normal turn policy would remove.
  - confidence: 0.86
