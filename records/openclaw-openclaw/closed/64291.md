## Summary

Current main does not support the issue’s central premise: `session_status`/`sessions.json.totalTokens` are intentionally a latest prompt-context snapshot, not the full JSONL transcript size, and replay trimming on main is driven by configured history limits rather than parentId-chain or compaction state alone.

## Evidence

- **`totalTokens` is a prompt snapshot, not transcript length:** `deriveSessionTotalTokens` documents that `SessionEntry.totalTokens` is used as a prompt/context snapshot and intentionally excludes completion/output tokens.
  - file: [src/agents/usage.ts:242](https://github.com/openclaw/openclaw/blob/11804a484ded89374afa55020813cfa265aa143a/src/agents/usage.ts#L242)
  - command: `nl -ba src/agents/usage.ts | sed -n '240,260p'`
  - sha: [11804a484ded](https://github.com/openclaw/openclaw/commit/11804a484ded89374afa55020813cfa265aa143a)
- **Session store persists latest-call context, not cumulative transcript size:** `persistSessionUsageUpdate` explicitly uses `lastCallUsage` for `totalTokens` so context-window utilization reflects the final API call’s current context instead of cumulative per-run usage.
  - file: [src/auto-reply/reply/session-usage.ts:115](https://github.com/openclaw/openclaw/blob/11804a484ded89374afa55020813cfa265aa143a/src/auto-reply/reply/session-usage.ts#L115)

- **Tests expect transcript aggregation plus latest context snapshot:** `readLatestSessionUsageFromTranscript` tests assert that input/output/cache counters aggregate across the transcript while `totalTokens` stays equal to the latest context snapshot; another test confirms earlier assistant usage is read from outside the old tail window.
  - file: [src/gateway/session-utils.fs.test.ts:731](https://github.com/openclaw/openclaw/blob/11804a484ded89374afa55020813cfa265aa143a/src/gateway/session-utils.fs.test.ts#L731)

DEFAULT_GROUP_HISTORY_LIMIT`, with `DEFAULT_GROUP_HISTORY_LIMIT = 50`.
