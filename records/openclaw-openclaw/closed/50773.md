## Summary

Keep open. Current main still skips heartbeat runs only for the main command lane and the resolved session lane; it does not check the cron lane, active cron job state, or an opt-in all-lanes busy policy. The related implementation PR #50774 is closed unmerged, and the adjacent open PR #64963 addresses active reply runs rather than cron collisions.

## Best Possible Solution

Keep this issue open and land a focused core fix: always defer heartbeat runs while cron work is active, add the requested opt-in `heartbeat.skipWhenBusy` policy if maintainers accept that config surface, route all busy skip reasons through the same wake retry/no-schedule-advance behavior, and cover manual cron lane, timed active cron jobs, all-lane busy mode, docs, schema/types, and scheduler retry tests.

Implement heartbeat cron-collision avoidance for this issue. Current main only skips heartbeats for the main command lane and resolved session lane. Add an always-on guard so `runHeartbeatOnce` defers while cron work is active, covering both manual cron lane work (`CommandLane.Cron` and, if appropriate, `CommandLane.CronNested`) and timed cron jobs tracked by `src/cron/active-jobs.ts`. Add opt-in `agents.defaults.heartbeat.skipWhenBusy` / per-agent heartbeat `skipWhenBusy` if maintainers accept the requested config surface; when true, defer heartbeats while any relevant command lane is busy, including subagent/nested lanes. Keep the fix generic to runtime lanes/cron state; do not add local-LLM-provider special cases. Coordinate with adjacent PR #64963 semantics so active-reply and lane/cron busy guards do not duplicate or mask each other. Update config schema/types/help/docs and focused tests.
