## Summary

Keep open: this is useful security hardening, but the patch is not ready as-is because the new bootstrap-token gate depends on the optional normal auth limiter and inherits the default loopback exemption. Current main still has the unthrottled mutex-backed verifier, and the PR discussion already confirms the no-config and default-loopback gaps.

## What This Changes

The PR adds a bootstrap-token auth rate-limit scope, gates pre-auth bootstrap-token verification, adds unit/integration coverage, and updates the changelog.

## Best Possible Solution

Use a dedicated bootstrap-token limiter before verifyDeviceBootstrapToken that is always constructed and loopback-non-exempt, or an explicitly approved equivalent, with coverage for no-config, default-loopback, browser-origin, and configured remote cases.

## Reproduction Assessment

Yes. Source inspection shows a signed device handshake with auth.bootstrapToken reaches verifyBootstrapToken before any bootstrap-token limiter on current main, and verifyDeviceBootstrapToken serializes under withLock; the PR body also supplies configured-limiter runtime proof.

## Solution Assessment

No. The pre-verifier gate is the right shape, but tying it to the optional normal auth limiter leaves no-config and default-loopback paths able to enter the mutex unless gateway/security owners explicitly accept that policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use an always-on bootstrap-token limiter:** `src/gateway/server/ws-connection/auth-context.ts:167`
  - body: Because this block only runs when `params.rateLimiter` is supplied, gateways without `gateway.auth.rateLimit` still call `verifyBootstrapToken`. Configured deployments also keep loopback exempt by default, so local attempts can still reach the mutex. Use a dedicated always-created bootstrap-token limiter, or an explicitly approved equivalent, and cover no-config/default-loopback cases.
  - confidence: 0.9
