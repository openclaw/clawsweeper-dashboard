## Label Justifications

- `P1`: The PR addresses a pre-auth gateway DoS path that can stall legitimate onboarding through a mutex-backed verifier.
- `merge-risk: 🚨 compatibility`: The new bootstrap-token scope can change existing onboarding behavior after repeated bad tokens from the same client IP.
- `merge-risk: 🚨 availability`: The patch intentionally short-circuits bootstrap-token verification during lockout, which can deny pairing from that IP until the lockout expires.

## Summary

Keep this PR open: current main and the latest release still call the mutex-backed bootstrap verifier without a bootstrap-token rate-limit gate, while the PR adds the missing gate, deferred failure accounting, serialization, and focused coverage. I did not find a concrete patch bug, but the stricter pre-auth lockout envelope is security/availability-sensitive and still needs gateway/security owner acceptance before merge.

## What This Changes

The PR adds a `bootstrap-token` auth rate-limit scope, gates and serializes WebSocket bootstrap-token verification before the mutex-backed verifier, defers failure accounting until final auth failure, and adds focused unit plus gateway tests.

## Best Possible Solution

Land the focused gateway fix after gateway/security owner review accepts the bootstrap-token lockout envelope and the latest head has the focused gateway tests plus changed-check proof recorded.

## Reproduction Assessment

Yes. Source inspection shows current main and `v2026.5.28` enter `verifyDeviceBootstrapToken` without a bootstrap-token limiter, and the PR body supplies a real-module runtime probe for forged concurrent attempts; I did not rerun the live probe in this read-only review.

## Solution Assessment

Yes, this is an acceptable focused mitigation: gate and serialize the bootstrap-token decision before the mutex-backed verifier while preserving device-token fallback. The remaining question is owner acceptance of the lockout and loopback envelope, not a concrete code defect in the latest diff.
