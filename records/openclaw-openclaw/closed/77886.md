## Summary

Keep open: current main still hard-blocks agent-initiated protected config changes, and the requested owner-approved path is a security-sensitive feature/product design rather than cleanup that can be closed.

## What This Changes

Add a first-class owner approval flow for protected Gateway config mutations such as enabling `/mcp` and registering MCP servers.

## Best Possible Solution

Design a core Gateway protected-config proposal and approval contract that binds approval to the exact patch, base hash, requesting session, owner identity, expiry, and risk class while keeping protected paths default-denied.

## Reproduction Assessment

Yes for the current limitation: source inspection shows agent-facing `gateway config.patch` and `config.apply` reject `commands.mcp` and `mcp.servers` because they are outside the allowlist. I did not run a live Gateway, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

No implemented solution exists yet, and the best fix is not a simple allowlist expansion. A maintainable solution needs a first-class protected-config approval flow that preserves the model/operator boundary.
