## Label Justifications

- `P1`: The issue reported a real scheduled-automation regression that duplicated outbound side effects.
- `impact:message-loss`: The affected cron run could send duplicate Telegram messages for one scheduled fire.
- `impact:session-state`: The report involved one run identity producing multiple session starts and conflicting cleanup/success terminal states.
- `impact:auth-provider`: The replay moved execution from the configured non-Codex payload model to the Codex primary/fallback model.

## Summary

Current main and the shipped v2026.6.6 release now treat embedded-agent results with visible output or committed outbound/tool side effects as terminal for model fallback, so the reported scheduled cron replay is already fixed while documented pre-side-effect cron fallback remains intact.

## What This Changes

Issue requests that scheduled cron agentTurn runs with a non-Codex payload model avoid replaying completed side-effecting work on the agent's Codex primary model.

## Best Possible Solution

Keep the current side-effect-aware fallback boundary: allow configured cron model fallback before committed work, but treat delivered messages, tool side effects, cron additions, and durable progress as terminal for that cron fire.

## Reproduction Assessment

Yes: the issue includes detailed live scheduled-run evidence, and source inspection shows the old cron fallback shape plus the current side-effect terminal guards that address it.

## Solution Assessment

Yes: the implemented boundary is better than disabling cron fallbacks globally because it preserves documented fallback for pre-side-effect failures while preventing replay after visible or committed work.
