## Label Justifications

- `P1`: The report describes a fresh regression that duplicates scheduled automation side effects and visible Telegram deliveries for real cron users.
- `impact:message-loss`: The affected cron run can send duplicate Telegram messages for one scheduled fire.
- `impact:session-state`: The report shows one scheduled run identity producing multiple session starts and conflicting cleanup/success terminal states.
- `impact:auth-provider`: The effective route changes from the configured non-Codex payload model to the Codex primary/fallback model during replay.

## Summary

Keep open: the reporter supplied credible scheduled-run evidence, and current main still has the cron payload-model fallback path that can re-enter a configured Codex fallback after a failed/cleanup-ended first attempt. The unresolved part is the maintainer policy boundary for stopping replay after committed side effects without removing documented pre-side-effect model fallback.

## What This Changes

The issue requests that scheduled cron runs with a non-Codex `payload.model` execute once on that configured model instead of replaying on the agent's Codex primary and duplicating side effects.

## Best Possible Solution

Add a scheduled-cron regression fix that treats a clean stop or committed side effects as terminal for that cron fire, while preserving documented fallback only for pre-side-effect provider/model failures.

## Reproduction Assessment

Yes at source level: current main still honors `payload.model` first and then permits configured fallbacks, and the reporter supplied live scheduled-run evidence for the cleanup then Codex replay shape. I did not run a live current-main MiniMax/Telegram scheduled cron in this read-only review.

## Solution Assessment

Unclear until maintainers choose the fallback boundary. Disabling all fallbacks for `payload.model` is not the best narrow fix because docs/tests preserve that contract; a side-effect-aware terminal gate is safer.
