Current main and release v2026.5.28 already implement the requested dampening: unauthenticated/no-device `probeGateway()` calls now short-circuit after repeated `device identity required` closes, with regression coverage for the important recovery paths.

## What This Changes

The issue reports repeated unauthenticated CLI gateway WebSocket probes after device-required rejections and asks for dampening/backoff rather than endless probe noise.

## Best Possible Solution

Keep the shipped `probeGateway()` dampening as the source-level fix; treat any unauthenticated `/health` endpoint as a separate product and security discussion if demand remains.

## Reproduction Assessment

Yes from source and prior issue evidence: repeated fresh unauthenticated `probeGateway()` calls without a cached device identity receive the gateway's 1008 `device identity required` close. I did not run a live reproduction in this read-only cleanup pass because current main now contains the fix.

## Solution Assessment

Yes. The best bounded fix is at `src/gateway/probe.ts`, where all CLI/status/restart probe callers converge; the broader `/health` idea would be a separate gateway product/security surface.
