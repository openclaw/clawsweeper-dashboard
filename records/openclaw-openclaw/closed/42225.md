## Summary

Current main no longer has the stale `openai-codex/gpt-5.4` metadata problem described in #42225. The provider now publishes `contextWindow: 1_050_000` for GPT-5.4, model resolution prefers runtime-resolved Codex metadata for explicit `openai-codex/gpt-5.4`, and session/status paths treat `272000` as the intentional default `contextTokens` cap rather than a stale catalog `contextWindow`.

## Close Comment

Closing based on a Codex review of current `main`. The stale GPT-5.4 metadata path described here is implemented on main: `extensions/openai/openai-codex-provider.ts:57` and `extensions/openai/openai-codex-provider.ts:549` now publish `openai-codex/gpt-5.4` with native `contextWindow: 1_050_000`, and `src/agents/pi-embedded-runner/model.ts:671` plus `extensions/openai/openai-codex-provider.ts:490` make explicit `openai-codex/gpt-5.4` resolution prefer the provider runtime metadata instead of a stale registry row. The regression coverage in `src/agents/pi-embedded-runner/model.test.ts:1024` and `src/commands/models/list.list-command.forward-compat.test.ts:3` matches that behavior. The remaining `272000` on current main is the intentional default `contextTokens` cap documented in `CHANGELOG.md:1332`, not the stale catalog/root-cause reported here. If there is still a request to raise the default effective cap itself, that should be tracked separately.
