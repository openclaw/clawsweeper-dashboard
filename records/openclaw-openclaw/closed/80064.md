## Summary

Keep open: current main and the latest release still generate raw `voice:*` keys while the startup migration canonicalizes raw session keys into `agent:<agentId>:...`; the linked closing pull request is open and not a merged fix.

## What This Changes

The issue asks Voice Call per-phone and per-call generated session keys to use the canonical agent-scoped format so caller continuity survives Gateway restarts.

## Best Possible Solution

Land a Voice Call-owned fix that emits canonical agent-scoped generated session keys from the configured `agentId`, preserves explicit keys, updates resolver/call-path coverage, and validates restart continuity before closing the issue.

## Reproduction Assessment

Yes, source-reproducible: current main emits raw `voice:*` keys and its tests assert that format, while Gateway startup migration rewrites raw keys into agent-scoped keys. I did not run a live Gateway restart or telephony scenario in this read-only review.

## Solution Assessment

Yes, the issue’s proposed direction is the narrow owner-side fix: make Voice Call generate canonical agent-scoped keys instead of adding special migration exceptions. The linked pull request still needs repair or replacement before it can be the implementation.
