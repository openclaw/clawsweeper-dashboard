## Summary

Keep open: the config-validation fix is useful and source-reproducible, but the patch currently rejects a runtime alias that current selection accepts and still lacks real after-fix behavior proof.

## What This Changes

The branch adds plugin-aware validation for provider/model-scoped `agentRuntime.id` values and regression coverage for provider ids such as `anthropic`.

## Best Possible Solution

Keep the early plugin-aware config validation, but share the runtime-id normalization used by harness selection and cover accepted aliases such as `codex-app-server` before merge.

## Reproduction Assessment

Yes. Current main accepts arbitrary provider/model `agentRuntime.id` strings in schema and later fails closed for unknown runtimes; the PR regression is also source-reproducible because `codex-app-server` normalizes to `codex` at runtime but is absent from the raw known-runtime set.

## Solution Assessment

No as submitted. The config-validation boundary is the right place, but it must normalize runtime ids with the same helper used by selection before rejecting unknown ids.
