## Summary

Keep open: the patch is in the right area, but it can still miss Kiro stale-resume failures when the resume-required signal is carried as structured `detailCode` instead of rendered message text, and the real behavior proof is mock-only.

## What This Changes

## Best Possible Solution

Carry `SESSION_RESUME_REQUIRED` through the manager retry predicate as structured data, cover both `startTurn` and legacy event paths, then merge only with redacted live Kiro proof or an explicit maintainer proof override.

## Reproduction Assessment

Yes at source level: the linked issue provides exact Kiro stale-resume logs, and current main shows the retry path depends on the manager matcher. I did not run a live Kiro backend in this read-only review.

## Solution Assessment

No: this PR is a useful direction but not the best fix because it matches rendered text while supported `detailCode` can be dropped before the retry decision. The safer fix is structured signal propagation plus a `startTurn` regression.
