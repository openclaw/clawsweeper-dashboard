## Summary

Close this PR as superseded by the earlier active replacement #73972. Current main still needs the Feishu P2P thread fix, but this PR is the weaker duplicate: it does not target the P2P thread root and it adds an unchecked `ctx.threadId?.trim()` path that the already-open replacement avoids.

## What This Changes

The PR updates Feishu P2P direct-message handling to set thread-reply flags when `thread_id` is present, relax DM reply skipping for that path, and adds focused tests plus a changelog entry.

## Best Possible Solution

Close this duplicate and consolidate review on #73972 or an equivalent single Feishu-owned patch that targets P2P thread roots, normalizes optional IDs safely, preserves root_id-only quote replies and group behavior, then closes #38806 after merge.

## Reproduction Assessment

Yes. A P2P Feishu event with both `root_id` and `thread_id` can be traced through current main: the handler stores both IDs but keeps all non-group reply-thread routing disabled and skips reply targets for DMs.

## Solution Assessment

No. This PR is not the best remaining fix because it only flips the thread flags and reply skipping; #73972 covers the same bug with root targeting, safer ID normalization, and an additional ACP failure-path case.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Target the thread root for P2P thread replies:** `extensions/feishu/src/bot.ts:701-702`
  - body: The new `directThreadReply` flag changes the thread flags, but the reply target selection still falls through to the non-group default `ctx.replyTargetMessageId ?? ctx.messageId`; the added test also locks in the child message as the target. For P2P thread events with `root_id`, this misses the source bug's required root target and can keep replies outside the originating thread.
  - confidence: 0.84
- **[P3] Validate thread_id before trimming it:** `extensions/feishu/src/bot.ts:701`
  - body: Feishu webhook parsing only validates required fields before casting the payload, and optional `thread_id` is stored as received. Calling `.trim()` here can throw on a malformed non-string `thread_id`, dropping that inbound message instead of treating it as a non-thread DM.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff has no supply-chain changes, but it introduces a low-severity malformed-payload DoS risk in Feishu DM thread detection.

Concerns:

- **[low] Unchecked thread_id trim can drop malformed events:** `extensions/feishu/src/bot.ts:701`
  - body: `ctx.threadId` comes from an external Feishu payload and optional fields are not runtime-normalized before `parseFeishuMessageEvent` stores them. The new `.trim()` call can throw on a non-string `thread_id`, causing that inbound message to fail dispatch.
  - confidence: 0.86
