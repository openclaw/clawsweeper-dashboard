## Summary

Keep open: current main still rejects the reported Windows drive-letter Docker bind case, and the latest PR branch fixes the earlier parsing/allowlist issues, but it introduces a blocking Windows path normalization hole in sandbox policy/audit handling.

## What This Changes

The PR updates sandbox Docker bind parsing and validation so Windows drive-letter host sources are accepted across config validation, runtime policy checks, audit coverage, tests, and changelog.

## Best Possible Solution

Land Windows drive-letter bind support only after host-path normalization preserves the drive root for `..` traversal, blocked-path/audit coverage includes that case, and the related Windows sandbox-security PR is reconciled.

## Reproduction Assessment

Yes, source-level. Current main still mis-parses `D:/...` in config validation and rejects non-`/` sources at runtime, matching the PR body's Windows gateway failure; the proposed patch bug is also source-reproducible via `path.posix.normalize` on drive-root traversal.

## Solution Assessment

No, not yet. The shared-parser direction is maintainable, but the PR should use Windows-aware normalization that clamps at the drive root before policy/audit comparisons and should add regression tests for that path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Preserve the drive root when normalizing Windows binds:** `src/agents/sandbox/host-paths.ts:43`
  - body: After this PR accepts drive-letter bind sources, `normalizeSandboxHostPath` still runs them through `path.posix.normalize`. That drops the `C:` prefix when `..` climbs above the drive root, so a bind like `C:/../Users/Alice/.ssh:/mnt:ro` is treated as `Users/Alice/.ssh` instead of the Windows path it resolves to, which can miss blocked credential-path and audit classification. Normalize drive-letter paths with Windows semantics and add coverage for drive-root traversal.
  - confidence: 0.84
