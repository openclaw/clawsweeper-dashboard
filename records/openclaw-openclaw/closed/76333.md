## Summary

Keep open. Current main and v2026.5.12 still use the fixed 60s Dream Diary narrative wait, timeout handling still leaves `DREAMS.md` unwritten, and the linked closing PR is still open rather than merged.

## What This Changes

The issue reports Dream Diary narrative generation timing out because the hardcoded narrative subagent wait is too short for current embedded-agent setup cost.

## Best Possible Solution

Review and land or replace a single memory-core fix that gives Dream Diary narrative runs enough bounded time, chooses the supported timeout contract, and updates tests/docs/changelog as needed.

## Reproduction Assessment

Yes, source-backed: current main and v2026.5.12 still pass a fixed 60s timeout into the Dream Diary subagent wait, and the timeout test shows `DREAMS.md` remains absent when that wait returns `timeout`. I did not run the full 15-workspace gateway scenario in this read-only review.

## Solution Assessment

Unclear. The implicated owner path is memory-core, but maintainers still need to choose whether the correct contract is a larger fixed default, an env/config override, late-output handling, or some combination.
