## Summary

Keep open: current main still has the reported required `turnSourceTo` match in the trusted backend chat replay bridge, and the latest shipped tag has the same code. The related Control UI and chat-channel replay PRs are useful context, but they do not cover the webchat null-target case described here.

## What This Changes

The issue asks OpenClaw to let webchat node exec approval replay succeed when `turnSourceTo` is null while preserving the existing approval binding checks.

## Best Possible Solution

Patch only the chat replay bridge so `turnSourceTo` is optional when absent, add a webchat/null-target regression case, and keep all existing binding rejection tests green.

## Reproduction Assessment

Yes, from source inspection and reporter logs: current main and `v2026.5.12` require a non-null `turnSourceTo` in the chat bridge, while webchat can replay without that target. I did not execute the live webchat flow in this read-only review.

## Solution Assessment

Yes, the narrowest maintainable fix is to use the existing optional string matcher for `turnSourceTo` only, while preserving the required channel/session checks and optional account/thread checks. A regression test should prove null-target webchat succeeds and changed non-null targets still fail.
