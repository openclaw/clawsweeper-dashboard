## Summary

Keep open: the branch addressed the earlier unrelated dependency and changelog problems, but the latest raw group-policy fallback can still weaken multi-account group isolation, and the supplied proof does not exercise the after-fix Telegram gateway authorization path.

## What This Changes

The PR changes Telegram account config and shared group-policy resolution so empty account-level `groups` can fall back to channel-level groups, adds merge-helper tests, and adds a changelog entry.

## Best Possible Solution

Mirror Telegram merge semantics in the raw policy resolver: treat empty account groups as unset only for single-account configs, preserve multi-account isolation, add resolver/gateway regression coverage, and provide redacted runtime proof.

## Reproduction Assessment

Yes, via source inspection. The current raw resolver and Telegram gateway call path show why `accounts.default.groups: {}` suppresses root groups, and the latest diff shows a remaining multi-account fallback edge without needing a live Telegram run.

## Solution Assessment

No. The single-account empty-as-absent direction is right, but the shared raw resolver should gate that fallback by account count so it does not weaken multi-account isolation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve multi-account isolation in raw group fallback:** `src/config/group-policy.ts:309`
  - body: The new raw resolver treats empty account `groups` as absent for every account-aware channel, so a Telegram multi-account config with root `groups` and `accounts.work.groups: {}` now falls through to `channelConfig.groups`. That conflicts with the Telegram merge contract and tests that multi-account bots must not inherit channel-level groups unless explicitly set; gate this fallback to single-account configs or pass the already-merged account groups into the policy path.
  - confidence: 0.86
