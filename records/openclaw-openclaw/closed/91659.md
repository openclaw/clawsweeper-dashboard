## Summary

Current main and the latest release no longer have the reported process-lifetime cooldown poison: the cooldown probe Set is scoped inside each fallback run, same-provider cooldown probes are explicitly per-run, and timeout rotation has coverage that avoids cooling down the timed-out profile.

## What This Changes

The issue reports that client-side LLM idle timeouts can populate provider cooldown state that survives `/model`, new sessions, and gateway restarts, leaving the fallback chain stuck for the gateway lifetime.

## Best Possible Solution

Keep the shipped per-run cooldown probe implementation and use the separate linked issue for any remaining `/model` fallback product decision.

## Reproduction Assessment

No current-main reproduction remains for the process-lifetime cooldown poison: current source scopes the Set per fallback run and shipped tests cover timeout rotation without profile cooldown. The broader `/model` strict-fallback behavior is tracked separately.

## Solution Assessment

Yes for this issue's central cooldown-poisoning bug: the current implementation removes the process-lifetime state and documents per-run probing. Additional reset/status UI would be a separate product decision rather than required to fix the reported root cause.
