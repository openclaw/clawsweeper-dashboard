Keep open: the report describes cron data loss against the documented persistence contract, and current main still does not provide an implemented, duplicate, or cannot-reproduce close path for the removal-before-reload symptom. ## What This Changes

Investigate and fix cron jobs being silently removed around config reload after a failed then successful `cron.add` sequence.

## Best Possible Solution

Add jobId-scoped audit around cron removals, identify the actual remover, then fix that confirmed path with guarded recovery that cannot undo intentional deletes.

## Reproduction Assessment

No. The incident report has detailed timestamps and current source maps the observed log to `cron.remove`, but there is not yet a high-confidence current-main path that makes config reload remove a successfully created job.

## Solution Assessment

Unclear. Backup restore alone is not the best fix until maintainers know whether the remover was a tool call, plugin cleanup, operator action, or a reload-adjacent race; the safer path is first to attribute the removal caller.
