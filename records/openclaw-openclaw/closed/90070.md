## Label Justifications

- `P1`: The report describes a real channel workflow where incoming qqbot messages were blocked for hours until the gateway process was killed.
- `impact:message-loss`: Incoming channel messages were reportedly accepted by the system but silently queued behind the stuck embedded run.
- `impact:session-state`: The failure mode is a stale running/BLOCKED session lifecycle state after compaction completion failed.

## Summary

Current main and the latest release already contain bounded compaction-wait handling, willRetry=false compaction-end resolution, and active-run cleanup that address the reported permanent qqbot session block.

## What This Changes

The issue reports qqbot sessions becoming permanently blocked when failed auto-compaction emits an incomplete compaction_end with willRetry=false.

## Best Possible Solution

Keep the current shipped compaction lifecycle and ask for fresh v2026.6.1-or-newer logs only if the same BLOCKED/running state still reproduces after upgrade.

## Reproduction Assessment

No live current-main qqbot reproduction was run, but the reported log shape maps directly to the embedded compaction_end willRetry=false path. Current source and the latest release show that path now resolves the compaction wait and bounds lost retry waits.

## Solution Assessment

Yes. The current implementation addresses the maintainable boundary: compaction-end handling resolves non-retry waits, the runner adds an aggregate timeout, and active embedded runs clear during cleanup instead of requiring channel-specific qqbot logic.
