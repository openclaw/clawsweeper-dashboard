## Summary

Keep open. Current main has partial Codex/OpenAI-Codex hardening, but source still supports the core stale-route path: stored direct or parent model overrides can override changed defaults unless an allowlist rejects them, and reset logic deliberately preserves legacy/user overrides. The issue discussion also includes a later concrete reproduction on `openclaw@2026.4.25`, so this is not a stale one-shot report.

## What This Changes

The issue asks OpenClaw to stop message-channel and embedded runs from selecting stale `codex` or `openai-codex` model/auth routes after Codex is disabled or the configured default route changes.

## Best Possible Solution

Keep this as the umbrella unless maintainers split it, and land a policy-backed cleanup plus regression suite that prevents stale automatic or legacy Codex state from winning after a default-route change while preserving explicit supported Codex routes.

## Reproduction Assessment

Yes. A high-confidence source reproduction can seed a session or parent session with `providerOverride: "openai-codex"` and `modelOverride: "gpt-5.4"` while the configured default points elsewhere; current selection code will still choose the stored override when no allowlist rejects it.

## Solution Assessment

Unclear. There is no patch under review, and the narrowest maintainable fix needs maintainer-approved stale-state policy rather than wholesale deletion or normalization of every `codex`/`openai-codex` entry.
