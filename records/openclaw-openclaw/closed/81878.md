## Summary

Keep open: current main still has the explicit-only native Ollama `num_ctx` behavior, the regression is source-reproducible, and https://github.com/openclaw/openclaw/pull/81900 is an open paired fix PR that references this issue with closing syntax. The reporter's follow-up strengthens the need for owner review of the fix direction rather than changing the close decision.

## What This Changes

The issue reports that native Ollama requests no longer derive `options.num_ctx` from configured `contextWindow` or `maxTokens` budgets after the v2026.5.10-era change.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/81900 or an equivalent narrow compatibility fix that preserves explicitly configured Ollama budgets without reintroducing unwanted implicit runtime context for unconfigured models.

## Reproduction Assessment

Yes, at source level. Current main only forwards explicit `params.num_ctx`, while the reported config has `contextWindow`/`maxTokens` without `params.num_ctx`, and adjacent tests assert that no native `options.num_ctx` is sent.

## Solution Assessment

Unclear. The issue is valid, but the best fix should be owner-reviewed to distinguish explicit user budgets from discovered/default catalog windows; a migration to `params.num_ctx` may be safer than broad runtime fallback.
