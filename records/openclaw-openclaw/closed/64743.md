## Evidence

- **Reporter follow-up config contradicts the issue summary:** The issue follow-up comment shows `agents.defaults.model.primary` set to `yh-llama-cpp/Qwen3.5-9B-Q4_K_M.gguf`, while `Qwen3.5-35B-A3B-UD-IQ4_XS.gguf` appears in `fallbacks`. That does not match the report's claim that the active model was the MoE model itself.
- **Reporter log shows timeout-driven fallback, not an unrelated model swap:** The same follow-up log shows repeated `FailoverError: LLM request timed out.` and then `All models failed (4)` listing the configured primary plus fallbacks. That points to all configured candidates timing out, not OpenClaw choosing an unrelated model.
- **Current main explicitly classifies timeouts into failover messaging:** `resolveAssistantFailoverErrorMessage` returns `LLM request timed out.` when the assistant run timed out, which is the exact string shown in the reporter's logs.
  - file: [src/agents/pi-embedded-runner/run/assistant-failover.ts:252](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/src/agents/pi-embedded-runner/run/assistant-failover.ts#L252)
  - sha: [fd65caf4b0ba](https://github.com/openclaw/openclaw/commit/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc)
- **Current main builds the candidate chain from requested model plus configured fallbacks:** `resolveFallbackCandidates` adds the requested model first, then configured `agents.defaults.model.fallbacks`, and finally the configured primary when applicable. This matches the behavior shown in the issue log.
  - file: [src/agents/model-fallback.ts:460](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/src/agents/model-fallback.ts#L460)
  - sha: [fd65caf4b0ba](https://github.com/openclaw/openclaw/commit/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc)
- **Docs and tests on main describe timeout-based fallback as intended behavior:** The docs say normal runtime order is current model, configured fallbacks, then configured primary for override runs, and list `timeout-shaped failover errors` as advancing fallback. The embedded e2e test also explicitly covers a timeout on the primary leading to fallback success.
  - file: [docs/concepts/model-failover.md:19](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/docs/concepts/model-failover.md#L19)
  - sha: [fd65caf4b0ba](https://github.com/openclaw/openclaw/commit/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc)
- **Timeout-to-fallback behavior is covered by an explicit embedded e2e test:** The `falls back on timeout errors using defaults-only model fallbacks` test expects the primary timeout to advance to the next configured fallback candidate.
  - file: [src/agents/model-fallback.run-embedded.e2e.test.ts:451](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/src/agents/model-fallback.run-embedded.e2e.test.ts#L451)
