## Summary

Keep open: current main still treats `replyToCurrent: false` as a veto before implicit current-message `replyToId` assignment, while directive parsing still emits that false value by default and the linked fix PRs are closed unmerged.

## What This Changes

The issue requests that parser-default `replyToCurrent: false` stop suppressing implicit `replyToId` assignment for queued or followup replies.

## Best Possible Solution

Separate parser-default false from an intentional non-threading signal, then apply implicit `replyToId` consistently in shared final, block, and followup payload paths while preserving command replies as non-threaded.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is `applyReplyThreading({ payloads: [{ text: "hello", replyToCurrent: false }], replyToMode: "all", currentMessageId: "42" })`, which current main leaves unthreaded because line 41 short-circuits before assigning the implicit id.

## Solution Assessment

Yes. The narrow maintainable fix is in the shared reply-threading pipeline, but it should distinguish parser defaults from explicit non-threading instead of blindly removing all false handling.
