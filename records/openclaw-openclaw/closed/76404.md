## Summary

Close as superseded by the already-open #62164. Current main still lacks a `sessions.list` response cache, but #62164 tracks the same cache work with broader invalidation inputs, docs/changelog/tests, while this smaller patch has a stale in-flight invalidation bug.

## What This Changes

The branch adds an in-memory 15-second `sessions.list` response cache with in-flight request coalescing and invalidation from `emitSessionsChanged()`.

## Best Possible Solution

Consolidate on #62164 as the canonical `sessions.list` cache implementation and ensure it handles config, session-store, transcript, and subagent invalidation with regression coverage.

## Reproduction Assessment

Yes for the patch defect by source inspection: start a slow `sessions.list` cache miss, mutate a session through an `emitSessionsChanged` path before it settles, then call the same list params. The old in-flight promise can still resolve and cache pre-mutation rows because invalidation does not clear or version in-flight work.

## Solution Assessment

No. The narrower TTL cache is not the best path because it can repopulate stale rows after invalidation and omits the broader invalidation/test/protocol work already centralized in #62164.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Invalidate in-flight list promises:** `src/gateway/server-methods/sessions.ts:148`
  - body: A session mutation can arrive while a slow `sessions.list` miss is still running. `emitSessionsChanged()` clears only `sessionsListResponseCache`, but callers with the same params still await the old `sessionsListInFlight` promise, and that promise writes its pre-mutation result back into the cache, so stale rows can be served for the 15s TTL even though invalidation fired.
  - confidence: 0.91
