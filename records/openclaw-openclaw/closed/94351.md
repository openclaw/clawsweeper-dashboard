## What This Changes

The PR rewrites gateway-served Control UI `index.html` public asset hrefs to the `__openclaw__` namespace, preserves those hrefs during client startup link sync, and adds Gateway HTTP regression tests.

## Best Possible Solution

Close this PR as superseded by https://github.com/openclaw/openclaw/pull/94204.

## Reproduction Assessment

Yes, source-backed for the PR defect: with `basePath: "/openclaw"`, the PR emits `/openclaw/__openclaw__/manifest.webmanifest`, while the manifest and bootstrap code lead to an unsupported `/openclaw/__openclaw__/control-ui-config.json` fetch. I did not run a live browser/proxy deployment in this read-only review.

## Solution Assessment

No, not as written. Server-side rewriting is a reasonable layer for pre-JS asset requests, but configured-basePath deployments should keep manifest-relative URLs on a supported base path or add the matching namespace bootstrap/start-url contract.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P1] Keep base-path manifest URLs on the configured base path:** `src/gateway/control-ui.ts:752-754`
  - body: For configured-basePath requests this branch rewrites `href="/manifest.webmanifest"` to `/openclaw/__openclaw__/manifest.webmanifest`. Because the manifest uses `start_url: "./"`, browsers can resolve an installed app launch to `/openclaw/__openclaw__/`; the UI then infers `/openclaw/__openclaw__` as its base path and fetches `/openclaw/__openclaw__/control-ui-config.json`, but the gateway only serves `/openclaw/control-ui-config.json` and the legacy single-underscore alias. Please rewrite configured-basePath assets to `/<basePath>/<asset>` or add the matching namespace bootstrap/start-url contract and tests.
