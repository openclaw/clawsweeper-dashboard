## Summary

Close as implemented: current main already ships the workspace-inheritance/subagent bootstrap behavior that this PR was trying to add, and the PR branch still has blocker findings that should not be revived as-is.

## What This Changes

The PR adds `contextFiles` and `inheritContext` options to `sessions_spawn`, injects selected workspace files into child prompts, and extends `PluginLogger` with an optional log transport type.

## Best Possible Solution

Keep the shipped workspace-inheritance implementation and handle any future role-specific instruction-file API as a separate product/security design, such as the narrower work tracked in #54504.

## Reproduction Assessment

Yes for the PR-branch defect: the diff and review comments show `contextFiles` resolving from `targetAgentId` instead of the requester workspace. For the original user problem, current main is source-verified through workspace inheritance and bootstrap injection tests; I did not run tests because this review was read-only.

## Solution Assessment

No, this PR is not the best way to solve it now. Current main shipped a narrower internal workspace-inheritance path, while this PR adds an extra public file-injection API with the wrong workspace lookup and insufficient bounds.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Read context files from the requester workspace:** `src/agents/subagent-spawn.ts:525`
  - body: The new context-file loader resolves `requesterWorkspaceDir` with `targetAgentId`, so `sessions_spawn({ agentId: ... })` reads the child agent workspace instead of the caller's workspace. That silently drops or injects the wrong `AGENTS.md`/`TOOLS.md` context in the cross-agent case the feature is meant to support.
  - confidence: 0.94
- **[P2] Constrain contextFiles before loading workspace files:** `src/agents/tools/sessions-spawn-tool.ts:61-66`
  - body: `contextFiles` accepts an unbounded list of arbitrary basenames and the implementation reads each file into the child system prompt. A caller with `sessions_spawn` could inflate prompts or load unexpected files such as `.env`; reuse the bounded bootstrap-file loader or add an explicit allowlist, size/count limits, and tests.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The PR adds an LLM-facing workspace-file read path without a file allowlist or count/size cap.

Concerns:

- **[medium] Arbitrary basename files can enter child prompts:** `src/agents/subagent-spawn.ts:525`
  - body: The proposed `contextFiles` path validation blocks traversal but still allows any simple filename in the requester workspace to be read into the child system prompt, which can bypass the narrower bootstrap-file contract and expose sensitive workspace files.
  - confidence: 0.82
- **[low] Unbounded contextFiles can inflate prompt and I/O work:** `src/agents/tools/sessions-spawn-tool.ts:61`
  - body: Unlike `attachments`, the proposed schema has no `maxItems`, so callers can request many file reads and large prompt expansion before the child run starts.
  - confidence: 0.8
