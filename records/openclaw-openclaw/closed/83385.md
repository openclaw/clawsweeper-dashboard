## Summary

Keep this PR open: it targets the right memory indexing path and includes usable real behavior proof, but the new frontmatter scanner can terminate on an indented `---` inside a valid YAML block scalar and index the rest of that frontmatter as chunk text.

## What This Changes

The PR strips leading Markdown YAML frontmatter from memory retrieval chunks and embedding input, exposes parsed frontmatter on markdown file entries, and adds host SDK plus memory-core regression tests.

## Best Possible Solution

Tighten the frontmatter fence detection so only a real unindented closing delimiter strips metadata, add a regression for block scalars containing an indented `---`, and keep the current chunk line-number behavior.

## Reproduction Assessment

Yes, from source inspection: current main chunks and embeds the full markdown content, including leading frontmatter. The PR finding is also source-reproducible because line.trim() === "---" treats an indented block-scalar line as the closing fence.

## Solution Assessment

No: the central approach is maintainable, but the fence scanner needs to distinguish an unindented frontmatter delimiter from valid indented YAML scalar content before this is safe to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Require a real frontmatter closing fence:** `packages/memory-host-sdk/src/host/internal.ts:176-182`
  - body: The scanner closes the frontmatter block on any line whose trimmed value is `---`. In valid YAML frontmatter, a literal block scalar may contain an indented `---` line; this code would stop there, parse only the prefix, and index the remaining frontmatter plus the real closing fence as retrieval text. Please only accept an unindented closing delimiter, and add a regression for that block-scalar case.
