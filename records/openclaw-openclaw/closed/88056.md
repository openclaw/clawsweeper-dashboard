## Summary

Keep open: current main still has a source-reproducible path where the generic `/skill` wrapper can reject a valid skill command, and the maintainer label plus MEMBER-authored report make this ineligible for conservative cleanup closure.

## What This Changes

The issue asks for generic `/skill <name> [input]` in live chat to resolve the same eligible skill command that appears in `commands.list` and works through its direct slash command.

## Best Possible Solution

Fix command handling so `/skill` loads the current resolved skill command list when the precomputed list is only a directive-filter placeholder, then add regression coverage for text and native command paths.

## Reproduction Assessment

Yes, from source inspection rather than a live chat run: `/skill ...` can be evaluated with `params.skillCommands` set to an empty array from directive resolution, causing `handleSkillCommandUsage` to return `Unknown skill` instead of loading workspace skill commands.

## Solution Assessment

Yes, the narrow maintainable fix is to keep the existing skill-command path and correct the skill-command loading/fallback semantics; no new command surface, config option, or product policy is needed.
