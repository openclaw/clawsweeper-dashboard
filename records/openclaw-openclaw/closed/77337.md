## Summary

Keep open: current `main` still has an all-corpus merge path where wiki hits can fill the entire truncated result set, so the reported zero-memory result is plausible and not implemented away.

## What This Changes

The issue asks for `memory_search` and `wiki_search` with `corpus=all` to return merged memory and wiki results instead of only visible wiki hits when memory hits exist.

## Best Possible Solution

Keep the shared search behavior but make the `corpus=all` merge preserve results from each nonempty requested corpus before filling remaining slots by rank, with regression tests for both tools.

## Reproduction Assessment

Yes, by source inspection: configure wiki to return at least `maxResults` higher-scored hits and memory to return a lower-scored hit, then `corpus=all` will sort and slice the memory hit away. I did not execute the test locally because dependency tooling is unavailable and the review is read-only.

## Solution Assessment

Yes, the narrow maintainable fix is a shared or duplicated all-corpus merge helper plus focused regression coverage in the two plugin-owned search paths. Changing core plugin APIs or documenting the manual two-call workaround would leave the existing contract weak.
