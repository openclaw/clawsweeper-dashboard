## Summary

Keep open: current main still does not implement a per-call `onComplete` exec option, but this PR needs revision before merge because the proposed API overlaps with open PR #46926 and the branch introduces concrete completion-wake contract bugs.

## What This Changes

The PR branch adds an `onComplete` exec option that treats `notify` as background execution, forces notify-on-exit behavior, and threads an explicit completion flag through process sessions.

## Best Possible Solution

Settle one exec completion-wake contract: either a revised per-call `onComplete` API or the default-notification direction from #46926, with literal validation, routed and unrouted notification semantics, explicit node-host support or rejection, docs, and focused tests for quiet success plus wake delivery.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Gate the no-poll hint on routable notifications:** `src/agents/bash-tools.exec.ts:1581-1583`
  - body: `onCompleteHint` is appended whenever `onComplete="notify"` is used, but `maybeNotifyOnExit` still drops the event when `session.sessionKey` is empty. Tool instances without a session key can tell callers not to poll even though no wake can be delivered.
  - confidence: 0.92
- **[P2] Reject or implement onComplete for node-host exec:** `src/agents/bash-tools.exec.ts:1248`
  - body: The patch treats `onComplete="notify"` as a background request, but `host=node` returns through `executeNodeHostCommand` before the local background/session path. Node executions therefore do not get the promised auto-backgrounding or completion wake semantics.
  - confidence: 0.9
- **[P2] Constrain onComplete to supported values:** `src/agents/bash-tools.exec-runtime.ts:168-173`
  - body: The schema accepts any string while the handler only recognizes `"notify"`, so typos such as `"notfiy"` silently no-op. Expose a literal/enum schema or return a warning/error for unsupported values.
  - confidence: 0.86
