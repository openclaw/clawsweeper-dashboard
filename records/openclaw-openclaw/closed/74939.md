## Summary

Keep open for maintainer review. Current main does not contain the proposed root Railway config, and the PR as written would not start the Gateway on Railway because its start command invokes the bare CLI path rather than `gateway`; it also changes Railway deployments away from the reviewed Dockerfile path without enough product or security context.

## What This Changes

The PR adds a root `railway.json` that forces Railway's `RAILPACK` builder, clears `dockerfilePath`, sets `node openclaw.mjs` as the start command, and configures `/health` as the deployment health check.

## Best Possible Solution

If OpenClaw wants repository-owned Railway config, it should be designed as an explicit deployment surface: start `openclaw gateway` directly, bind/listen on the Railway port model, preserve the required token/state/volume settings, document the template-to-config relationship, and make an intentional security decision about replacing the Dockerfile path with Railpack.

## Reproduction Assessment

Yes for the blocking PR regression. Static inspection is enough: the proposed start command is the bare CLI, and current main routes bare root invocations to Crestodian, which exits on non-TTY deploys instead of binding the Gateway.

## Solution Assessment

No. Disabling Docker with this root config is not the best solution as written; the narrow maintainable path would either keep the existing template/Docker deployment or add a Railway config that explicitly runs the Gateway and preserves equivalent deployment hardening.
