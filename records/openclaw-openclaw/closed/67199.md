## Summary

Current main already resolves and forwards per-agent `thinkingDefault` into reply runs, and the same implementation is present in release `v2026.4.23`, so the reported ingress fallback path is already fixed/shipped.

## Evidence

- **Directive resolution returns the defaulted thinking level:** `getReplyDirectives` computes `resolvedThinkLevelWithDefault` from the explicit/session value, then `modelState.resolveDefaultThinkingLevel()`, then `agentCfg?.thinkingDefault`, and returns that value as `resolvedThinkLevel`. That means downstream code receives the defaulted level, not the pre-default variable from the issue report.
  - file: [src/auto-reply/reply/get-reply-directives.ts:494](https://github.com/openclaw/openclaw/blob/6f72b74cecf8d1fec5c147ae932161bf0f37a68b/src/auto-reply/reply/get-reply-directives.ts#L494)

  - sha: [6f72b74cecf8](https://github.com/openclaw/openclaw/commit/6f72b74cecf8d1fec5c147ae932161bf0f37a68b)
- **Runner handoff uses the resolved value:** `runPreparedReply` accepts `resolvedThinkLevel`, only falls back to `modelState.resolveDefaultThinkingLevel()` when it is still unset, and passes that same variable into `followupRun.run.thinkLevel`. With the current directive pipeline, the embedded runner is not seeing an undefined value from this path.
