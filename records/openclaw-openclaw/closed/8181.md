## Summary

Close as implemented on current main. The remaining setup gap called out in the earlier discussion was addressed after the latest release: explicit `provider: "local"` now gets the default downloadable EmbeddingGemma GGUF and packaged installs retain/repair the `node-llama-cpp` runtime through managed plugin runtime deps. The fix is on main only, not in `v2026.4.27`.

## What This Changes

The issue requests turnkey local memory embeddings with a default GGUF download, automatic native runtime handling, a simple `memorySearch.provider: "local"` toggle, and clear fallback messaging.

## Best Possible Solution

Keep the current main implementation for packaged installs: explicit local memory search should stage the native runtime through managed plugin deps and download the default GGUF on first use. Track GPU backend/stability and any stale doc wording in narrower follow-ups rather than keeping this broad setup request open.

## Reproduction Assessment

Not applicable as a feature request. The verification path is source inspection for `memorySearch.provider: "local"`: current main selects the default model URI, resolves it through `node-llama-cpp`, and includes `node-llama-cpp@3.18.1` in bundled runtime deps for local memory configs.

## Solution Assessment

Yes. Managed plugin runtime deps are the narrowest maintainable way to make packaged local embeddings turnkey without restoring `node-llama-cpp` as a root dependency; explicit GPU/backend controls and crash isolation remain separate issues.
