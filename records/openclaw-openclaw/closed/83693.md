## Label Justifications

- `P1`: The reported behavior makes the user-facing stop command fail to interrupt active agent/channel work in a shipped release.
- `impact:session-state`: The issue is about active run/session lifecycle state not being interrupted when the user sends a stop command.

## Summary

Current main already handles this by routing stop text through the immediate abort path, resolving the active embedded run from the session key, clearing queued work, and covering Telegram control-lane bypass with regression tests; the fix is current-main-only and is not in v2026.5.12.

## What This Changes

The issue asks for `/stop` to interrupt an active agent run immediately instead of waiting for the current tool/action to finish.

## Best Possible Solution

Keep the current-main fast-abort implementation and ship a release containing c32878d1b7fce248a7bfaaae317f6688aef2fab5 so v2026.5.12 users receive the fix.

## Solution Assessment
