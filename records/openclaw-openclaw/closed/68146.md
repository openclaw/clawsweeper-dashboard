- `merge-risk: 🚨 compatibility`: The PR changes startup-gated RPC behavior from immediate retryable failures to potentially long waits, which can affect existing retry-aware clients during upgrades.
- `merge-risk: 🚨 availability`: Holding requests during sidecar startup can increase perceived stalls and in-flight request lifetime if sidecars are slow or fail to open the barrier.

## Summary

Keep open: current main still has the immediate method-level startup gate, and this PR is a plausible targeted fix, but it needs conflict resolution, current-head real behavior proof, and an explicit Gateway startup-strategy decision before merge.

## What This Changes

The PR adds a Gateway startup barrier that holds startup-gated requests until post-attach sidecars finish, then falls back to the existing retryable startup `UNAVAILABLE` response on timeout.

## Best Possible Solution

Rebase onto current main and either land a Gateway-owned bounded wait with fresh restart proof, or close this as superseded if maintainers choose fail-fast retryable startup responses as the supported design.

## Reproduction Assessment

No high-confidence end-to-end reproduction was run in this read-only review. Source inspection does confirm current main can return immediate retryable `UNAVAILABLE` for startup-gated methods while the set is populated, but the visible cached-browser symptom needs fresh restart proof because current main also rejects startup-pending connects.

## Solution Assessment

Unclear: a bounded server-side barrier is a plausible durable fix for old or cached clients, but it now competes with current main's retryable startup-connect strategy. Maintainers should decide the desired startup contract before merge.
