## Summary

Keep this PR open: the source-proven bug is real, but the patch changes public plugin SDK file-lock semantics in a way that can let another process reclaim a lock from a still-running owner once `stale` elapses.

## What This Changes

The PR changes stale lock reclamation to evaluate expired or unparseable `createdAt` before PID liveness and updates stale-lock and plugin SDK file-lock tests.

## Best Possible Solution

Land a fix that recovers PID-reused stale auth/profile locks without making generic public file locks stealable from live owners, ideally with a stronger owner identity or a narrow compatibility-preserving path plus focused upgrade proof.

## Reproduction Assessment

Yes. Current main and the latest release both check `payload.pid` before `createdAt`, so a payload with an expired `createdAt` and a reused live PID is kept from source inspection.

## Solution Assessment

No. The patch fixes the reported PID-reuse case, but it does so by changing the public SDK default for all file-lock callers; the safer path is to preserve live-owner protection or explicitly approve and document the new hard-stale contract.
