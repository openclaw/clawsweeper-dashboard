## Summary

Current `main` already implements persistent outbound delivery recovery for gateway restarts: outbound sends are written to a state-dir delivery queue before send, acknowledged or failed after send, and drained again on gateway startup. The changelog shows this shipped in release `v2026.2.13`.

## Evidence

- **Shipped release note:** The `2026.2.13` changelog explicitly says OpenClaw added 'a write-ahead delivery queue with crash-recovery retries to prevent lost outbound messages after gateway restarts.'
  - file: [CHANGELOG.md:5042](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/CHANGELOG.md#L5042)
  - command: `nl -ba CHANGELOG.md | sed -n '5018,5045p'`
- **Write-ahead enqueue before send:** `deliverOutboundPayloads()` persists outbound work before delivery by calling `enqueueDelivery(...)`, then acknowledges or marks failure after the send attempt completes.
  - file: [src/infra/outbound/deliver.ts:649](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/infra/outbound/deliver.ts#L649)
  - command: `nl -ba src/infra/outbound/deliver.ts | sed -n '644,721p'`
- **Persistent on-disk queue storage:** Queue entries are stored under the OpenClaw state dir in `delivery-queue/*.json`; `enqueueDelivery()` writes entries to disk and `ackDelivery()` uses a two-phase rename/unlink flow to avoid replay after a crash between send and cleanup.
  - file: [src/infra/outbound/delivery-queue-storage.ts:44](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/infra/outbound/delivery-queue-storage.ts#L44)
  - command: `nl -ba src/infra/outbound/delivery-queue-storage.ts | sed -n '44,183p'`
- **Gateway startup recovery:** Gateway scheduled service activation calls `recoverPendingDeliveries(...)` on startup, and the recovery module explicitly scans pending queue entries and retries them on startup.
  - file: [src/gateway/server-runtime-services.ts:116](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/gateway/server-runtime-services.ts#L116)
  - command: `nl -ba src/gateway/server-runtime-services.ts | sed -n '55,120p'; nl -ba src/infra/outbound/delivery-queue-recovery.ts | sed -n '340,434p'`
- **Regression coverage:** Tests cover simulated crash recovery, retry/failure handling, replaying stored delivery options, and gateway startup wiring that triggers recovery after sidecars are ready.
  - file: [src/infra/outbound/delivery-queue.recovery.test.ts:51](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/infra/outbound/delivery-queue.recovery.test.ts#L51)

## Risks / Open Questions

- The current implementation is a persisted `delivery-queue/*.json` store with retry/backoff/max-retry handling; it does not appear to implement the exact TTL-based drop policy proposed in the issue body.
