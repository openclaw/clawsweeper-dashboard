## Summary

Keep this open: current main still has a plausible Responses replay failure for custom `openai-responses` providers because generic Responses payloads can be unstored while replaying server-owned reasoning item IDs. Direct OpenAI routes have a newer store-aware wrapper, but the reported custom proxy route is not covered by that direct-OpenAI-only path.

## What This Changes

The issue reports that `openai-responses` sessions can get stuck on stale server-side Responses IDs when requests are not stored.

## Best Possible Solution

Make Responses replay store-aware: keep stored continuation for native OpenAI-compatible routes that support it, but strip or recover stale server-owned replay IDs for unstored/custom proxy turns without adding a new config surface.

## Reproduction Assessment

No high-confidence live repro was run. Source inspection shows the risky shape on current main: generic `openai-responses` sends unstored payloads while replaying persisted Responses reasoning item IDs.

## Solution Assessment

Yes, fixing the replay/payload boundary is the best direction. Broadly forcing `store: true` for all custom proxies would be less safe than making replay depend on whether the previous response was actually stored or recoverable.
