## Summary

Keep open: current main still has channel reachability diagnostics after onboarding, but `openclaw onboard` does not expose an interactive per-channel connection preflight or `--test-connection`-style flag before writing config. This remains new onboarding UX/product work, not an implemented fix or a broken existing contract.

## What This Changes

Add an onboarding connection preflight that checks configured channel reachability before finalizing the OpenClaw config.

## Best Possible Solution

Design a generic onboarding channel preflight that reuses plugin `probeAccount`/`auditAccount` and gateway reachability plumbing, with clear skip behavior and no provider-specific core handshakes.

## Reproduction Assessment

Not applicable; this is a feature request, not a failing current contract. Source and docs show live diagnostics through `openclaw channels status --probe`, `openclaw status --deep`, and `openclaw health --verbose`, but not an onboarding preflight.

## Solution Assessment

Unclear; the requested direction is plausible, but maintainers need to decide prompt timing, opt-in/skip semantics, and credential/gateway lifecycle first. The narrow maintainable path would reuse the existing plugin probe contracts rather than add Telegram or Discord-specific logic to core onboarding.
