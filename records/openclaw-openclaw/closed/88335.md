## Label Justifications

- `P1`: The original bug caused a real Feishu channel workflow to visibly lose streamed assistant reply content by showing only the latest fragment.
- `impact:message-loss`: The affected behavior was channel delivery content being truncated or overwritten during Feishu streaming-card updates.

## Summary

Current main and the shipped v2026.6.5 release already send cumulative Feishu streaming-card text to the CardKit content endpoint; the suffix-diff helper reported here was removed by the merged Feishu fix, with regression tests and release notes covering the behavior.

## What This Changes

The issue asks Feishu streaming cards to send full accumulated text to CardKit instead of only the incremental suffix diff.

## Best Possible Solution

Keep the shipped cumulative/full-text Feishu CardKit behavior for this bug and handle tenant-specific long-reply append-mode tradeoffs in the separate Feishu latency thread.

## Reproduction Assessment

Yes for the shipped regression: the merged fix diff and tests show the previous `hello` to `hello!` update sent only `!`. Current main and v2026.6.5 send the cumulative text, so the failing path no longer reproduces there.

## Solution Assessment

Yes. Removing suffix extraction at the Feishu streaming-card update boundary is the narrow owner-local fix and matches the official CardKit full-text streaming contract; append-mode latency tradeoffs belong in the separate follow-up.
