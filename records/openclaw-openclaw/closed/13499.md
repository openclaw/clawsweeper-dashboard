## Summary

Issue #13499 is a useful orchestration idea, but current main positions it as ClawHub/plugin territory rather than an OpenClaw core feature. Core already provides the building blocks for threshold-driven policies (`session_status`, `sessions_spawn`, heartbeat isolated sessions) and already owns the lower-level threshold behavior for compaction/memory flush.

## Close Comment

Closing after a Codex review. This looks like a good ClawHub skill/plugin, not an OpenClaw core cleanup item. Current main already exposes the primitives needed for a threshold policy outside core: `session_status` reports session usage/state (`docs/concepts/session-tool.md:81`), `sessions_spawn` creates isolated child sessions (`docs/concepts/session-tool.md:101`), and heartbeat can already run in fresh isolated sessions (`docs/gateway/heartbeat.md:23`, `docs/gateway/heartbeat.md:227`). Core also already has threshold-based compaction support through pre-compaction memory flush (`docs/reference/session-management-compaction.md:330`, `docs/gateway/configuration-reference.md:1419`). `VISION.md:54`, `VISION.md:69`, and `VISION.md:106` explicitly keep optional orchestration / new skills out of core by default and point that work toward ClawHub. If this is still wanted, the right next step looks like a ClawHub skill or plugin that watches `session_status` and applies a local policy with `sessions_spawn` or reset flows.
