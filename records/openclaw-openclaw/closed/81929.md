## What This Changes

## Best Possible Solution

Fix the wait-loop timeout bound, then land after maintainer approval and green CI while preserving the non-blocking `cron.run` RPC and using run-id-filtered history for completion.

## Reproduction Assessment

Not applicable: this is an additive feature PR, not a broken existing behavior report. Source inspection confirms current main lacks the wait/run-id filter surfaces, and the PR body supplies live after-fix Gateway output for the intended behavior.

## Solution Assessment

No: polling `cron.runs` by the returned `runId` is the right narrow design, but the current wait loop must cap sleep to the remaining timeout so `--wait-timeout` is a true maximum.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Cap each sleep to the remaining wait timeout:** `src/cli/cron-cli/register.cron-simple.ts:77`
  - body: The loop checks the timeout before sleeping, then always sleeps for the full `pollIntervalMs`. If a run never records a terminal entry and a caller uses `--wait-timeout 1s --poll-interval 10s`, the command can block for roughly 10 seconds before the next timeout check, which breaks the documented maximum wait and matters for shutdown/pre-stop hooks. Clamp the sleep to the remaining timeout or use a timeout-aware timer.
  - confidence: 0.91
