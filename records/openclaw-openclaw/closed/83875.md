- `merge-risk: 🚨 compatibility`: Changing managed updates from package-root cwd to home cwd can change dotenv-derived config and behavior for existing installs.

## Summary

Keep open: the PR is pointed at a real managed-update crash and the submitted terminal proof is useful, but the current implementation starts managed updates from the user home whenever it exists, which can silently load a different `.env` and alter update config or secrets during the handoff.

## What This Changes

The PR changes managed update handoff cwd selection, adds early `process.cwd()` fallback handling, updates regression tests, and adds a changelog entry for the macOS LaunchAgent `uv_cwd` handoff crash.

## Best Possible Solution

Use an OpenClaw-controlled stable directory, preferably the newly created handoff temp directory, as the managed helper and updater cwd, then keep the `uv_cwd` startup fallback and prove the handoff without introducing cwd-based dotenv drift.

## Reproduction Assessment

Yes for the source-level failure path: the linked production log shows `uv_cwd`, and current main starts the handoff from the package root while early CLI startup can call `process.cwd()`. I did not run a live macOS LaunchAgent reproduction in this read-only review.

## Solution Assessment

No, not as currently implemented: durable cwd hardening is the right direction, but preferring the user home creates cwd-based dotenv drift. The narrower maintainable fix is to use the controlled handoff temp directory as cwd and reserve user/root paths for fallback only.
