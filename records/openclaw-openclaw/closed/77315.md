## Summary

Current main and the shipped v2026.5.3 tag already implement the requested cache on the manifest metadata/model-id normalization hot path, with regression coverage showing repeated metadata calls do not re-read the same manifest.

## What This Changes

The issue requests caching `listOpenClawPluginManifestMetadata` results used under provider model-id normalization to avoid repeated synchronous plugin manifest scans during startup.

## Best Possible Solution

Keep the shipped fingerprinted metadata cache and policy cache; any remaining startup slowness should be reported as a new issue with a fresh profile against v2026.5.3 or newer.

## Reproduction Assessment

No current-main failing reproduction remains. The issue profile identifies the old hot path, and current source plus regression coverage show repeated calls now use cached manifest metadata instead of re-reading the same manifest.

## Solution Assessment

Yes. The current implementation is the narrow maintainable fix because it caches manifest metadata by file fingerprint and caches normalization policies by config fingerprint/current snapshot while preserving invalidation when manifests or config change.
