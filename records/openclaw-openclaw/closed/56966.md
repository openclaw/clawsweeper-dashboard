## Summary

Current main addresses the fresh first-owner Telegram pairing failure without merging the DM pairing credentials store into group authorization. Pairing approval now bootstraps `commands.ownerAllowFrom` when no command owner exists, Telegram native group command auth accepts that owner identity, and docs/tests preserve the deliberate boundary that raw DM pairing-store entries do not authorize every paired sender in groups.

## What This Changes

The issue requests that Telegram DM pairing approval authorize the same user for group slash commands such as `/activation` without manually adding the user to config-level `allowFrom` or `groupAllowFrom`.

## Best Possible Solution

Keep the current security model: first-time pairing should bootstrap a command owner for one-owner setups, while group sender authorization remains explicit through Telegram `allowFrom`, `groupAllowFrom`, `groups`, or per-group/per-topic overrides instead of broadening all DM-paired users into group command authority.

## Reproduction Assessment

Yes. The reported path is concrete, and current-main source/tests show the old mismatch is addressed for first-owner setup by writing `commands.ownerAllowFrom` during pairing approval while keeping DM pairing-store entries alone insufficient for group auth.

## Solution Assessment

Yes. The current-main approach is safer than the linked PR’s pairing-store merge because it fixes the fresh operator setup without turning every paired DM sender into a group command sender.
