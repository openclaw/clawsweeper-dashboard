## Summary

Keep open: current main and the latest stable tag still have the successful block-stream path that drops text-only final payloads without proving the final text was delivered. An open PR targets the same root behavior, but it is still open and has correctness/proof blockers, so this issue should remain open as the user-facing report.

## What This Changes

The issue reports that WhatsApp block streaming can suppress the complete final assistant reply after a partial streamed block was sent.

## Best Possible Solution

Fix `buildReplyPayloads` so final text is suppressed only when current delivery evidence proves it was already sent, while preserving coalesced-stream dedupe, unsent media follow-up, error/fallback payloads, direct block keys, and message-tool dedupe.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: call `buildReplyPayloads` with block streaming enabled, `didStream()=true`, `isAborted()=false`, `hasSentPayload()=false`, and a text final payload. Current main has a focused test that expects this to return no reply payloads.

## Solution Assessment

Yes for the issue direction: the maintainable fix is proof-based final suppression in the existing reply-payload path. The open PR direction is useful but needs additional correctness work so coalesced streams and existing media/error/direct-send safeguards are preserved.
