## Summary

Keep open: current main still leaves Copilot Claude/Anthropic model definitions without the pi-ai compat opt-out, and the linked closing PR remains open rather than merged.

## What This Changes

The issue asks OpenClaw to stop sending pi-ai's rejected `tools[].eager_input_streaming` field for GitHub Copilot Claude/Anthropic-compatible models.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/75393 or a replacement that applies the provider-local Anthropic compat override across static manifest/default rows, live catalog rows, dynamic fallback rows, normalization/schema, and focused regression coverage.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: tool-using `github-copilot/claude-*` models still reach pi-ai with unset eager-streaming compat, and pi-ai 0.74.0 defaults to sending the rejected field. I did not run a credentialed Copilot request.

## Solution Assessment

Yes, the provider-local compat flag is the right mechanism, but it should be applied through shared Copilot model metadata and normalization so static, live, and dynamic model paths all receive it.
