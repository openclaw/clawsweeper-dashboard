## Summary

Keep this issue open. Current main has native Ollama tool-call support and canonical TUI session keys, but I still do not see source or test evidence that the combined `openclaw tui --deliver` + local Ollama + session-aware tool path prevents `openclaw-tui` from being treated as a session reference. The prior ClawSweeper comment identified the same gap, and the relevant current paths still leave that failure class plausible.

## Best Possible Solution

Keep the issue open and add a focused regression for `openclaw tui --deliver` using the TUI UI client identity `openclaw-tui`, canonical session key `agent:main:main` or `current`, and a native Ollama-style tool-capable response that exercises a session-aware tool such as `session_status`. If the model is being shown or encouraged to pass `openclaw-tui` as a session reference, fix the prompt/session metadata or resolver guidance so tools use `current`, `main`, or the canonical agent session key instead.

## Evidence

- **Issue discussion is concrete and recent:** The provided issue body reports OpenClaw 2026.4.12 with local Ollama qwen3:8b, direct Ollama tool calls succeeding, and only the OpenClaw TUI tool path returning a session-id failure for `openclaw-tui`. The existing ClawSweeper comment also kept it open after reading current TUI/session/Ollama paths.
