## Summary

Current `main` already provides a Telegram-native/mobile exec-approval flow: same-chat `/approve`, native Telegram approval delivery/buttons with approver DMs and topic preservation, and agent-routed exec followups back into the originating session. The config split called out in the issue is also now documented explicitly as forwarding vs native-client layers.

## Evidence

- **Agent-routed exec followups:** Approved async exec completions are resumed as an `agent` turn in the same session/channel via `buildAgentFollowupArgs(...)` and `callGatewayTool("agent", ...)`, rather than only dumping raw terminal output straight to a target.
  - file: [src/agents/bash-tools.exec-approval-followup.ts:127](https://github.com/openclaw/openclaw/blob/11cffb230013c2d8bd1e0e18da04b0b985b8abd5/src/agents/bash-tools.exec-approval-followup.ts#L127)
- **Same-chat Telegram approvals:** `/approve` is handled in shared chat command flow and resolves exec/plugin approvals through the gateway; Telegram approvers are explicitly authorized in the command path.
  - file: [src/auto-reply/reply/commands-approve.ts:126](https://github.com/openclaw/openclaw/blob/11cffb230013c2d8bd1e0e18da04b0b985b8abd5/src/auto-reply/reply/commands-approve.ts#L126)
- **Telegram native approval client:** Telegram native approval capability resolves approver DMs, preserves origin chat/topic targets, notifies origin chats when DM-only delivery is used, and loads a Telegram-native approval runtime.
  - file: [extensions/telegram/src/approval-native.ts:67](https://github.com/openclaw/openclaw/blob/11cffb230013c2d8bd1e0e18da04b0b985b8abd5/extensions/telegram/src/approval-native.ts#L67)
- **Telegram approval UX and topic support:** Telegram exec approvals are documented as approver-DM or originating-chat/topic flows with inline approval buttons and topic-preserving followups.
  - file: [docs/channels/telegram.md:741](https://github.com/openclaw/openclaw/blob/11cffb230013c2d8bd1e0e18da04b0b985b8abd5/docs/channels/telegram.md#L741)
- **Config model clarified:** The advanced exec-approval docs now distinguish `approvals.exec` forwarding from `channels.<channel>.execApprovals` native approval clients, document same-chat `/approve`, and state that native approval UI is the primary path when available.
  - file: [docs/tools/exec-approvals-advanced.md:237](https://github.com/openclaw/openclaw/blob/11cffb230013c2d8bd1e0e18da04b0b985b8abd5/docs/tools/exec-approvals-advanced.md#L237)
- **Shipped release evidence:** Release `2026.4.2` added DM-first native chat approvals plus clarified forwarding-vs-native config, and `2026.4.1` fixed Telegram topic-aware exec-approval followups.
  - file: [CHANGELOG.md:1546](https://github.com/openclaw/openclaw/blob/11cffb230013c2d8bd1e0e18da04b0b985b8abd5/CHANGELOG.md#L1546)

## Risks / Open Questions

- Users who only configure `approvals.exec.mode: "targets"` and do not enable `channels.telegram.execApprovals.enabled` (`true` or `"auto"`) can still stay on the forwarded `/approve` path instead of the richer Telegram-native approval UX.
- The Control UI mobile ergonomics called out in the report may still matter as a fallback surface, but Telegram/mobile approvals no longer depend on that path when the native Telegram approval client is enabled.
