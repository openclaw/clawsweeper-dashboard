## Summary

Close as `clawhub`. Current `main` already has plugin-owned webhook ingress primitives and a bundled generic webhooks plugin, while the project vision explicitly keeps optional integrations out of core and prefers plugin/ClawHub delivery for this kind of platform-specific automation.

## Evidence

- **Vision keeps optional capability out of core:** `VISION.md` says core should stay lean, optional capability should usually ship as plugins, and new skills should go to ClawHub first. It also lists new core skills and heavy orchestration layers as things the project will not merge for now.
  - file: [VISION.md:54](https://github.com/openclaw/openclaw/blob/c1383680405879a3c3f43dceb20a694f54d3c13c/VISION.md#L54)

  - sha: [c13836804058](https://github.com/openclaw/openclaw/commit/c1383680405879a3c3f43dceb20a694f54d3c13c)
- **Plugin SDK already supports webhook-specific HTTP routes:** Plugin architecture docs expose `api.registerHttpRoute(...)` with `auth: "plugin"` specifically for plugin-managed webhook verification/signature handling, which matches a GitHub-webhook bridge better than a new core gateway feature.
  - file: [docs/plugins/architecture-internals.md:529](https://github.com/openclaw/openclaw/blob/c1383680405879a3c3f43dceb20a694f54d3c13c/docs/plugins/architecture-internals.md#L529)

- **Bundled webhooks plugin already covers trusted external automation:** `docs/plugins/webhooks.md` documents a plugin that adds authenticated HTTP routes for trusted systems such as CI jobs, with shared-secret auth, request guards, rate limiting, and TaskFlow binding.
  - file: [docs/plugins/webhooks.md:11](https://github.com/openclaw/openclaw/blob/c1383680405879a3c3f43dceb20a694f54d3c13c/docs/plugins/webhooks.md#L11)

  - file: [extensions/webhooks/index.ts:19](https://github.com/openclaw/openclaw/blob/c1383680405879a3c3f43dceb20a694f54d3c13c/extensions/webhooks/index.ts#L19)

## Risks / Open Questions

- There is no bundled GitHub webhook plugin on `main` today, so closing this in core scope assumes maintainers want the feature to live as a plugin/ClawHub package instead of as a first-party bundled integration.
- If maintainers later decide GitHub issue-comment automation is a strategic default workflow, this scope call could change.
