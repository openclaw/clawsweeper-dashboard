## Summary

Keep open. Current main still evaluates retained `toolCallHistory` across time gaps, records new calls by append-and-size-trim only, and runs loop detection before recording the next call. The provided GitHub context also shows open, unmerged PRs with closing syntax for this issue, so those should be reviewed or closed before this issue is resolved.

## Best Possible Solution

Keep this issue open. Review and either land one focused fix from #40217/#40798 or implement a maintainer-owned fix around `src/agents/tool-loop-detection.ts` and `src/agents/pi-tools.before-tool-call.ts` that prunes or ignores stale history before detection, keeps outcome recording consistent, and adds regression coverage for separated heartbeat cycles plus a continuous-loop control case.

## Risks / Open Questions

- The stale-gap threshold needs maintainer judgment so slow but continuous in-turn loops do not bypass protection.
- A record-only stale reset is likely incomplete because the before-tool hook currently runs detection before recording the new call.
- Impact is limited to users who enable `tools.loopDetection`, since the guard is documented and implemented as disabled by default.
