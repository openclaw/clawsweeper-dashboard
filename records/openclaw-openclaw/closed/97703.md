## Summary

Current main already solves the OOM risk this PR targets: all Google OAuth token responses now pass through the shared `fetchWithTimeout` 16 MiB bounded reader before `requestTokenGrant` can call `response.text()` or `response.json()`. The remaining parser-side shape is redundant for the central network-buffering problem and broader parser hardening is separately covered by https://github.com/openclaw/openclaw/pull/97587.

## What This Changes

The PR changes `extensions/google/oauth.token.ts` to replace token-exchange `response.text()` and `response.json()` reads with bounded OpenClaw provider response readers.

## Best Possible Solution

Keep the current-main shared Google OAuth fetch cap from https://github.com/openclaw/openclaw/pull/97628, and review the broader parser-side hardening in https://github.com/openclaw/openclaw/pull/97587 separately if maintainers still want that defense-in-depth layer.

## Reproduction Assessment

No for current main. The old unbounded helper behavior is visible in `v2026.6.10`, but current main now caps the shared Google OAuth response body before token parsing.

## Solution Assessment

No, this PR is no longer the best way to solve the issue. The shared OAuth helper cap on current main is the better owner-boundary fix for the OOM path, with parser-side hardening left to the broader related PR if needed.
