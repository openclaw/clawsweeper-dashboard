## Summary

Current main and the shipped v2026.4.27 release handle the linked anthropic-messages failure on the provider-owned Anthropic transport path, while this PR changes the OpenAI WebSocket response converter and is incomplete even for malformed OpenAI Responses payloads. The branch should not be merged as-is, but the reported user-facing bug is covered by the current transport routing and parser behavior.

## What This Changes

The branch adds `Array.isArray` guards around `response.output` and nested `item.content` iteration in `src/agents/openai-ws-message-conversion.ts`.

## Best Possible Solution

Close this stale PR and keep the provider-owned Anthropic transport as the shipped fix for the linked relay-provider failure. Any future OpenAI Responses hardening should be a separate focused change that normalizes `response.output` and message `content` before all array operations and adds malformed-shape coverage.

## Reproduction Assessment

Yes. The related issue provides concrete `anthropic-messages` provider setup, model, message, and logs, and current-main code-path review maps that setup away from the PR-touched OpenAI WebSocket converter and into the Anthropic transport.

## Solution Assessment

No. The proposed patch is not the best way to solve the linked issue because it edits the wrong transport family for the reported `anthropic-messages` route, and its new guards are applied too late to protect the OpenAI pre-scan it tries to harden.
