## Summary

Keep this PR open: the direction matches the cron timeout/token-bloat regression, but the current classifier is both under-inclusive for common exact commands and over-inclusive for natural-language cron prompts that start with command-like verbs.

## What This Changes

The PR defaults command-style cron agent turns to lightweight bootstrap context, threads cron context mode through CLI and embedded execution, revises Codex cron instructions, and adds focused tests plus a changelog entry.

## Best Possible Solution

Keep the lightweight cron propagation, but replace or tighten the classifier so exact shell-command payloads get lightweight context while natural-language cron jobs keep the default context unless explicitly overridden.

## Reproduction Assessment

No high-confidence live cron timeout reproduction was established. Source inspection does reproduce the PR defects: the new regex misses common exact commands and matches verb-led prose on the current PR head.

## Solution Assessment

No. The propagation approach is reasonable, but the command-style detector needs to be fixed before this is the narrow maintainable solution.

  - body: The new detector only recognizes paths, cd, and a short allowlist of binaries, so exact cron payloads such as `ls -la`, `echo ok`, or `sed -n '1p' file` still run with full bootstrap context.   - confidence: 0.88

That drops workspace/bootstrap context for legitimate prose cron turns that current main leaves on the default context path.
  - confidence: 0.9
