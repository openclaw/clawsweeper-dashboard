## Summary

Keep open. Current main includes useful mitigations for bounded/lightweight session lists and trajectory writer yielding, but the source still shows broad `sessions.list` loading/merging full stores before row construction and `pi-trajectory-flush` still inherits the fixed 10000 ms cleanup timeout.

## What This Changes

The issue asks OpenClaw to keep session listing responsive under moderate session-store load and make trajectory flush cleanup less brittle than a fixed 10s timeout.

## Best Possible Solution

Land or replace the targeted session-list resolver-cache and trajectory-flush cleanup fixes, then close this issue only after both large-store listing and cleanup timeout behavior are proven on current main.

## Reproduction Assessment

Partially yes. The fixed 10000 ms pi-trajectory-flush timeout is source-reproducible from the cleanup helper and call site, while the reported sessions.list latency is supported by multiple production logs and matching full-store hot-path code but not by a synthetic current-main timing run here.

## Solution Assessment

Yes for keeping this open. The maintainable path is to review/land the focused open PRs, or replace them with equivalent scoped fixes, rather than closing this issue while either reported path remains only partially addressed.
