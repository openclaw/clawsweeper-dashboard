## Summary

Close as duplicate of the open canonical retention-policy issue. Current main still only has global session maintenance controls plus transient active-key preservation, while #50900 tracks the same remaining need in a broader per-pattern/per-type form.

## What This Changes

The issue requests a per-session or per-topic pin/protection mechanism so selected session keys are excluded from `pruneAfter` and `maxEntries` cleanup.

## Best Possible Solution

Keep the design and implementation discussion centralized in #50900, where maintainers can choose a durable public config contract for retention rules or overrides instead of splitting pinning and pattern-retention into separate cleanup threads.
