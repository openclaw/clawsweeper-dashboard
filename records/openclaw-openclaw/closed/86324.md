## Label Justifications

- `P1`: A runaway heartbeat tool loop can burn very large token budgets and stall a real background agent workflow.
- `impact:session-state`: The report involves main-session heartbeat context and repeated heartbeat tool state being replayed in the same long-lived session.

## Summary

The issue remains necessary: current main exposes `heartbeat_respond` as a normal heartbeat tool, records the response only after the model/tool loop finishes, and has no default rolling loop breaker for repeated identical heartbeat response calls.

## What This Changes

The issue requests guardrails so main-session heartbeat turns cannot repeatedly call `heartbeat_respond` and burn large token budgets.

## Best Possible Solution

Add a runtime-level heartbeat guard that accepts at most one `heartbeat_respond` per heartbeat turn and treats a valid response as terminal, without changing default heartbeat session semantics or adding a new config knob.

## Reproduction Assessment

No high-confidence live reproduction was established. The reporter gives concrete incident details, and current source shows `heartbeat_respond` is non-terminal with default loop detection off, but I did not reproduce the exact repeated-call sequence against current main.

## Solution Assessment

Yes for the narrow guard: making `heartbeat_respond` terminal or once-per-turn is the maintainable first fix. Changing heartbeat to isolated/light context by default would be a broader product choice and should not be required to stop this loop.
