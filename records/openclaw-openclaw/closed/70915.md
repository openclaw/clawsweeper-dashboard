## Summary

Keep this PR open. It targets a real ZAI image-tool routing bug, but it is not merge-ready because the branch lacks after-fix real behavior proof and the endpoint override is fed through the shared models.json preparation path, which can persist or race the wrong endpoint beyond the single vision request.

## What This Changes

The PR wraps the ZAI media-understanding provider to rewrite known ZAI vision models from `/api/coding/paas/vN` base URLs to `/api/paas/vN` and adds helper unit tests.

## Best Possible Solution

Keep the provider-owned ZAI vision endpoint fix, but implement the endpoint override without writing a patched provider config through shared models.json, then require redacted real behavior proof before merge.

## Reproduction Assessment

Yes. Configure `agents.defaults.imageModel` as `zai/glm-4.6v` with the ZAI provider base URL set to a coding endpoint and call the image tool; current main keeps the generic image request on that configured transport while the linked issue reports 404 and direct success against the standard endpoint.

## Solution Assessment

No, not yet. The provider-local boundary is right, but the implementation should avoid request-scoped cfg rewrites that flow into shared models.json, and the PR still needs after-fix real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Keep the vision endpoint override out of models.json:** `extensions/zai/media-understanding-provider.ts:77`
  - body: Passing the patched cfg into `describeImageWithModel` sends it through `ensureOpenClawModelsJson(params.cfg, params.agentDir)` before model discovery. That can write the standard ZAI endpoint into the agent's shared models.json, so concurrent or subsequent non-vision ZAI traffic can flip between the configured coding endpoint and the vision override; keep this routing request-scoped after model resolution or in provider transport instead.
  - confidence: 0.86
