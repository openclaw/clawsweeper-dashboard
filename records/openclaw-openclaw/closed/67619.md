## Summary

Current `main` and the latest shipped release both reload gateway-backed chat history on session switch, guard against stale cross-session history responses, and include regression coverage for switching sessions. The report was filed against `2026.2.17`, and the described data-loss behavior does not match the current shipped Control UI path.

## Evidence

- **Session switch always reloads history:** `switchChatSession()` resets the local chat view, syncs the session key, and immediately calls `loadChatHistory(state)` for the selected session instead of relying on unsaved browser-local state.
  - file: [ui/src/ui/app-render.helpers.ts:509](https://github.com/openclaw/openclaw/blob/7e52223d327555c611e8c91ee2500897e9eab0bf/ui/src/ui/app-render.helpers.ts#L509)
  - command: `nl -ba ui/src/ui/app-render.helpers.ts | sed -n '509,522p'`
  - sha: [7e52223d3275](https://github.com/openclaw/openclaw/commit/7e52223d327555c611e8c91ee2500897e9eab0bf)
- **History application is pinned to the active session:** `loadChatHistory()` versions requests with `chatHistoryRequestVersions` and only applies a response when it is still the latest request and `state.sessionKey` still matches, preventing an older session response from wiping the currently selected session.
  - file: [ui/src/ui/controllers/chat.ts:19](https://github.com/openclaw/openclaw/blob/7e52223d327555c611e8c91ee2500897e9eab0bf/ui/src/ui/controllers/chat.ts#L19)

- **Regression test covers the exact switch-session race:** `ignores stale history responses after switching sessions` asserts that after starting a history load for `main`, switching to `other`, and resolving the old request, the stale `main` response does not replace the active session's messages; only the `other` history is shown.
  - file: [ui/src/ui/controllers/chat.test.ts:779](https://github.com/openclaw/openclaw/blob/7e52223d327555c611e8c91ee2500897e9eab0bf/ui/src/ui/controllers/chat.test.ts#L779)

- **Docs say session state is gateway-owned and transcript-backed:** The session docs state that all session state is owned by the gateway, and the context docs state that normal history persists in the session transcript until compacted or pruned. That means there is no separate explicit UI-side save step required to preserve chat history across session switches.
