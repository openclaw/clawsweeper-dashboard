## Summary

Current main already routes Discord final replies containing `[[reply_to_current]]` through normal Discord delivery instead of finalizing a draft preview path, and focused regression coverage exists for that behavior; the fix is current-main-only and not in the latest tagged release yet.

## What This Changes

The issue reports Discord channel replies being dropped when assistant output starts with the `[[reply_to_current]]` directive.

## Best Possible Solution

Keep the current-main Discord reply-delivery implementation and let the next release carry it to users who are still on 2026.5.7.

## Reproduction Assessment

No high-confidence current-main failing reproduction remains: the reported version was 2026.5.7, while current main has source and regression-test coverage showing directive-prefixed Discord finals are delivered through the normal send path. I did not run a live Discord session in this read-only sweep.

## Solution Assessment

Yes. The current implementation is the narrow maintainable fix because it keeps draft-preview finalization for ordinary finals but forces directive-bearing finals through the established Discord delivery pipeline.
