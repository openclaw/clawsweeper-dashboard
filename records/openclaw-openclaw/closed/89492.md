## Summary

Keep open: current main still closes normalized App SDK streams as a clean end-of-stream when the event pump throws, and the public SDK docs describe these streams as ready external-app surfaces. The issue is narrow, source-backed, and suitable for a focused SDK fix rather than cleanup closure.

## What This Changes

The issue reports that App SDK normalized event iterators hide event-pump failures by returning a normal closed iterator state.

## Best Possible Solution

Preserve the event-pump failure cause in the SDK stream layer and make active and future oc.events()/Run.events() iterators surface that failure instead of reporting a clean end-of-stream.

## Reproduction Assessment

Yes, from source inspection: a throwing transport iterator or normalizeGatewayEvent failure reaches startEventPump's close/catch path, and EventHub.stream().next() then returns done:true. I did not run the provided one-off command because the review was required to keep the checkout read-only.

## Solution Assessment

Yes, a focused SDK fix is the right layer: EventHub or startEventPump should carry the failure to normalized SDK iterators, with tests for before-yield and after-yield pump failures. Adding config flags or moving this to plugin/transport ownership would be broader than needed.
