## Summary

Keep open: current main still uses requestId-based Control UI device pairing, and WhatsApp `allowFrom` remains a channel DM sender allowlist rather than a browser OTP auth factor. This is an unresolved security-sensitive auth feature that needs maintainer design, not a cleanup close.

## What This Changes

Request to add a WhatsApp or messaging-delivered one-time-code handshake for approving unrecognized Control UI browser devices without SSH or local CLI approval.

## Best Possible Solution

Design an opt-in, channel-agnostic device-pair approval or notification handshake with WhatsApp as one transport, explicit target config, short-lived/replay-safe codes, rate limits, audit logs, scope-preserving approval, and docs.

## Reproduction Assessment

Not applicable as a bug reproduction. Source and docs give a high-confidence current-main behavior check: unpaired Control UI browsers create requestId-based device-pair requests, while WhatsApp pairing is channel DM access, not a browser OTP handshake.

## Solution Assessment

Unclear. WhatsApp OTP may solve the single-operator use case, but the safer maintainable direction is an opt-in, channel-agnostic pairing approval design rather than hardcoding WhatsApp into gateway auth.
