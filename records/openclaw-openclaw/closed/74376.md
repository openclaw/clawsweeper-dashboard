## Summary

Current main now fixes the central reported failure mode: browser CLI status/start resolves browser profile settings from the authored source config instead of a stale gateway runtime snapshot, and managed Chrome still passes `--no-sandbox` when the resolved config has `browser.noSandbox=true`. The fix is present on main after `v2026.4.27` and is not in a tagged release yet.

## What This Changes

The issue reports Chrome browser automation failing with namespace/sandbox errors after configuring `executablePath` and `noSandbox` on a Windows/npm and Ubuntu/container setup.

## Best Possible Solution

Keep the source-config refresh and shared browser control runtime behavior, then ship it in the next release. Split-host Windows/WSL setups should continue to use the documented remote CDP profile path when Chrome is not running on the same host as the Gateway.

## Reproduction Assessment

Yes for the fixed code path: the regression test models a stale runtime config after gateway startup and proves browser status/start now preserves source-configured `executablePath` and `noSandbox`. I did not run a live Windows/WSL Chrome sandbox repro in this read-only review.

## Solution Assessment

Yes. Refreshing browser control from the source config and sharing one control runtime is the narrow maintainable fix for configured `noSandbox`/`executablePath` being ignored; it avoids broad Chrome sandbox recovery while preserving the existing opt-in sandbox behavior.
