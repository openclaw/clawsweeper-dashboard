## Summary

Keep this PR open. The security-hardening goal is still relevant for Signal, but the submitted patch is not merge-ready because it wraps the legacy formatted envelope while current prompt assembly still prefers raw `BodyForAgent`; the history changes also introduce nondeterministic wrapper IDs during rendering.

## What This Changes

This PR imports `wrapExternalContent` and wraps Signal's formatted inbound envelope body and rendered pending group-history body in `extensions/signal/src/monitor/event-handler.ts`.

## Best Possible Solution

Rework the Signal hardening around the active prompt contract: wrap or attach untrusted user text on the `BodyForAgent`/structured `UntrustedContext` path, preserve clean command fields for command parsing, keep generated metadata outside untrusted blocks, make pending-history wrappers stable, and add focused Signal coverage for DMs, groups, debounced messages, quote-only context, and pending group history.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Wrap the model-facing Signal body:** `extensions/signal/src/monitor/event-handler.ts:159-162`
  - body: Wrapping only the `body` passed to `formatInboundEnvelope` does not protect Signal replies, because this handler still finalizes `BodyForAgent` and `RawBody` from raw `entry.bodyText`. Core prompt assembly prefers `BodyForAgent`, so inbound Signal messages still reach the model unwrapped.
  - confidence: 0.95
- **[P2] Avoid regenerating history wrappers during render:** `extensions/signal/src/monitor/event-handler.ts:181-184`
  - body: `wrapExternalContent` creates a fresh random marker ID on every call. Calling it inside pending-history `formatEntry` makes unchanged history byte-different across turns and undermines deterministic prompt caching; wrap at ingest or use stable IDs instead.
  - confidence: 0.91
- **[P3] Keep generated history IDs outside the untrusted body:** `extensions/signal/src/monitor/event-handler.ts:181-184`
  - body: The proposed template places the application-generated `[id:...]` suffix inside the untrusted wrapper with sender text. Append trusted routing metadata outside the external-content block so it is not presented as user-controlled content.
  - confidence: 0.86
