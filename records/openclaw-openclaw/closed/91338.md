## Summary

Keep open. The PR now handles null clears in cron normalization and service merging, but public `cron.update` still validates the normalized patch against a protocol schema that only allows null for `toolsAllow`, so the new null values would be rejected before reaching `mergeCronPayload`. The external proof is still test-suite output rather than a real cron update clear path.

## What This Changes

The branch changes cron payload normalization, merge behavior, and `CronPayloadPatch` types so `null` clears several optional `agentTurn` payload fields.

## Best Possible Solution

Align the gateway protocol schema and validator tests with the nullable clear contract for each intended field, then add real cron.update proof showing a stored override cleared back to default inheritance.

## Reproduction Assessment

Yes from source inspection, though not by a live run: `cron.update` normalizes the patch and validates it before service update, while the current protocol schema rejects the newly preserved nulls except for `toolsAllow`.

## Solution Assessment

No. The service-layer null clear direction is plausible, but the PR is not the best complete fix until the gateway protocol schema and validation tests are aligned with the same public API contract.
