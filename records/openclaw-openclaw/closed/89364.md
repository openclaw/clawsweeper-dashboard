## What This Changes

The PR adds bounded local-loopback retry delays to Crestodian's startup gateway health probe and adds regression coverage for a delayed local gateway becoming reachable.

## Best Possible Solution

Land this focused local-loopback retry if maintainers accept the bounded startup latency; otherwise tune the retry budget while preserving the startup-race regression test.

## Reproduction Assessment

Yes from source inspection: current main performs one 900ms `/healthz` probe in the Crestodian startup path, so a local gateway that becomes reachable after that first miss is reported unreachable. I did not run a live SSH/systemd reproduction in this read-only review.

## Solution Assessment

Yes, with a maintainer tradeoff: changing only the Crestodian startup caller avoids raising the shared gateway probe timeout for other commands, while the added latency for genuinely down local gateways needs explicit acceptance or tuning.
