## Summary

Keep this PR open: the cleanup direction is useful, but the patch still has two correctness blockers in the runtime alias path and Telegram reasoning-preview lifecycle.

## What This Changes

The PR adds provider-dispatcher runtime packaging aliases, changes Telegram reasoning-stream cleanup/docs/tests, strips formatted reasoning blocks in message-tool sends, adjusts progress backtick cleanup, and isolates one module-loader test path.

## Best Possible Solution

Revise the PR so provider-dispatcher legacy aliases resolve to the actual stable dist entry and every `/reasoning stream` finalization path clears reasoning previews, with targeted regression tests for both cases.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: the alias writer skips missing targets, and final reasoning lane delivery still marks previews retained before cleanup.

## Solution Assessment

No. The approach is directionally right, but the PR needs to align the compatibility alias target with the emitted dist path and keep reasoning lanes transient even when the final payload contains a reasoning segment.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Point legacy aliases at the real provider dispatcher entry:** `scripts/runtime-postbuild.mjs:44-46`
  - body: The new legacy aliases only get written when `dist/provider-dispatcher.js` exists, but the PR adds the stable entry as `auto-reply/reply/provider-dispatcher`, which emits under that nested path. As written, `writeLegacyRootRuntimeCompatAliases` will skip these legacy provider-dispatcher chunks in the real package, so the update-time lazy chunk failure remains. Point the aliases at the nested stable entry, or add and test an actual root stable entry.
  - confidence: 0.89
- **[P2] Keep final reasoning segments transient:** `extensions/telegram/src/bot-message-dispatch.ts:1028`
  - body: This removes retention when an answer-only final follows streamed reasoning, but a final payload containing `<think>...</think>answer` still delivers a final reasoning lane first. `createLaneTextDeliverer` then marks that reasoning preview complete and retained, so cleanup will not clear it, despite the new docs saying `/reasoning stream` previews are transient. The fix needs to keep reasoning finalization transient in stream mode and add coverage for the tagged-final split path.
  - confidence: 0.82
