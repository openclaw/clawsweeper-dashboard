## Summary

Current main preserves a same-baseUrl custom provider through `/models`, Telegram callback selection, runtime model resolution, and custom-provider auth; the proof commit is on `main` only and is not in a tagged release yet.

## What This Changes

The issue requests that a custom Z.AI-compatible provider such as `zai-light` remain selectable and runnable through Telegram `/models` even when it shares the built-in `zai` baseUrl.

## Best Possible Solution

Keep the current main behavior that preserves custom provider ids across catalog, selection, runtime, and auth, and retain or add focused regression coverage before the next release.

## Reproduction Assessment

No, not as a current-main failure. The failing release behavior was not live-reproduced in this read-only review, but the current-main source path for `/models` through runtime/auth was traced end to end and no baseUrl-based provider collapse remains on that path.

## Solution Assessment

Yes. Preserving the configured provider id at the catalog, session, runtime, and auth boundaries is the narrow maintainable fix and matches the documented custom-provider contract.
