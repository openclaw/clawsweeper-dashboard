## Label Justifications

- `P1`: The report described production agents repeating prior external mutations after reset, which is an urgent broken agent workflow even though current main has now addressed it.
- `impact:session-state`: The affected path is session-memory and startup context, where prior-session state could be re-associated with a new/reset session.
- `impact:security`: The reported failure mode was prompt-injection-like replay of historical user instructions into tool-enabled agents with external API authority.

## Summary

Current main addresses the reported automatic replay path: recent date-only and slugged daily memory files are loaded for bare `/new` and `/reset` only through a runtime startup prelude that marks them as untrusted quoted notes and tells the model not to follow instructions inside. The slugged session-memory preload fix shipped in `v2026.4.24` and is still present on current main.

## What This Changes

The issue asks OpenClaw to prevent session-memory transcripts saved on `/reset` or `/new` from being reintroduced as current executable instructions.

## Best Possible Solution

Keep the runtime-owned untrusted startup-context implementation for reset/new memory replay, and leave distinct malformed-token or memory-search sanitization concerns to their narrower open items.

## Reproduction Assessment

No, not against current main. Source and tests show bare `/new` and `/reset` startup memory now uses an untrusted quoted prelude for slugged artifacts, so the raw-current-input replay path described for v2026.4.15 is no longer present.

## Solution Assessment

Yes. Runtime-owned startup framing covers both date-only and slugged daily memory on `/new` and `/reset` without requiring agents to infer trust from raw memory file contents.
