- `impact:session-state`: The task ledger and agent-visible status can diverge from the user’s multi-image workflow when a second request is reported as the first task’s status.

## Summary

Keep open: current main still treats any active session image-generation task as a duplicate before reading the new prompt or filename, and the linked open PR is the active implementation path rather than a reason to close the issue before merge.

## What This Changes

The issue asks `image_generate` to stop suppressing a distinct second image request in the same session as a duplicate status response for the first active task.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/83614 if prompt-aware duplicate matching is accepted; otherwise implement a structured fail-fast rejection that clearly says no new generation was started while preserving same-request retry protection.

## Reproduction Assessment

Yes by source inspection: with any queued or running session-backed image task, a second generate action returns duplicate-guard status before the new prompt or filename is represented. I did not run a live provider repro in this read-only review.

## Solution Assessment

Unclear until the linked PR is reviewed: prompt-aware duplicate matching is a plausible fix, while structured active-task rejection is the narrower choice if maintainers want to keep the one-active-task guardrail.
