## Summary

Keep open: current main still has separate live tool-progress rendering and final `chat.history` projection/reset paths, so the reported WebChat size jump remains source-plausible and does not meet a close reason. Current rendering collapses tool rows by default, which mitigates raw output bloat, but it does not prove the live-to-final visual footprint is stable.

## What This Changes

WebChat should avoid jarring bubble size changes when tool-heavy assistant turns transition from live streaming to final history rendering.

## Best Possible Solution

Define the Control UI contract for live tool progress, then make live tool rendering and final history projection preserve a stable visual footprint through completion.

## Reproduction Assessment

No high-confidence browser reproduction was run. Source inspection shows the live tool-message path, final history reload, and projection/reset boundary that can plausibly produce the reported reflow.

## Solution Assessment

Unclear until maintainers choose the desired UX. Reusing the existing collapsed tool-call rendering during streaming or stabilizing the final history swap is likely narrower than adding another setting by default.
