## Summary

Keep open: current main and v2026.6.10 still show a plausible cross-workspace LanceDB memory path because the LanceDB memory table has no agent/workspace owner field and recall searches the whole table. Some reported vectors are already mitigated or inaccurate in current source, so this needs maintainer/security design rather than conservative cleanup closure.

## What This Changes

The issue asks OpenClaw to prevent memory-core, active-memory, and LanceDB-backed memory from leaking remembered content between workspaces in multi-user deployments.

## Best Possible Solution

Define and test an owner-scoped memory storage contract for LanceDB-backed memory, including migration/doctor handling for existing rows and docs for any intentional shared-memory opt-in, while preserving memory-core's current workspace-filtered wrappers.

## Reproduction Assessment

Yes, source-level only: current memory-lancedb store/search lacks agent or workspace identity and active-memory routes to memory_recall when memory-lancedb is selected. I did not run the reported 52-workspace Feishu deployment, and one claimed session-memory vector does not match current source.

## Solution Assessment

Unclear as written: the requested isolation direction is valid, but a narrow workspace_key column on generic plugin_state_entries would not solve the unpartitioned LanceDB recall path by itself. The best fix needs a maintainer-approved owner/scope contract plus stored-data migration and regression coverage.
