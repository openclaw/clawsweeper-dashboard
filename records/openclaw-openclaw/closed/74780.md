## Summary

Keep open. Current main does have a core `message` tool and the HTTP invoke path can build it, but `/tools/invoke` still filters tools through the selected session's tool policy before returning 404. That leaves a plausible, current failure mode for default/onboarded `tools.profile: "coding"` or restrictive workflow-agent configs, while Lobster docs still show `openclaw.invoke --tool message` as an intended workflow pattern. The reporter also provided a concrete Telegram setup and a recipes-extension fallback, so this is actionable context rather than a stale or unsupported request.

## What This Changes

Investigate a regression where `POST /tools/invoke` reports `Tool not available: message` for Telegram approval notifications despite a healthy configured Telegram channel.

## Best Possible Solution

Define the intended contract for workflow approval notifications, then make it explicit in code and docs: either route `human_approval` notifications through a trusted internal delivery path that does not depend on the public HTTP tool allowlist, or document and validate the required `message` tool policy with a clearer error and focused `/tools/invoke` regression coverage.
