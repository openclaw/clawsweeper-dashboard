## Summary

Current main and the latest shipped release already avoid Gateway restarts for skill enable/disable changes: the UI sends skill toggles through `skills.update`, the handler writes `skills.entries.<skillKey>.enabled` with automatic reload intent, `skills.*` paths are classified as no-restart reload paths, and the reloader invalidates skill snapshots so the next agent turn sees the change. The remaining Save-button wording is a separate UX preference, not needed to solve the reported restart churn.

## What This Changes

Allow users to change multiple skill enablement toggles and apply them without restarting the Gateway after each toggle.

## Best Possible Solution

Keep the shipped skill hot-reload and snapshot-invalidation behavior as the solution for restart avoidance; treat any future batch-save control as optional UX polish rather than a Gateway restart fix.

## Reproduction Assessment

Yes. The source-level reproduction path is high confidence: a skill toggle calls `skills.update`, writes `skills.entries.<skillKey>.enabled`, and flows through reload planning where `skills.*` is a no-restart path that only invalidates skill snapshots. I did not run a live Gateway because this review was required to stay read-only.

## Solution Assessment

Yes for the reported restart problem. The current implementation is narrower and more maintainable than adding a Save button solely to avoid restarts because it removes the restart cause while keeping skill changes immediately persisted and reflected on the next agent turn.
