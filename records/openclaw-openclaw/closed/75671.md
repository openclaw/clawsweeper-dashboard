## Summary

Keep open: this active maintainer-labeled, MEMBER-authored PR still has a functional blocker on the latest head. The policy-refresh fast path treats omitted or empty `policyPluginIds` as safe, while the chat `/plugins enable` and `/plugins disable` path still calls the refresh without target IDs, so it can skip the intended source rebuild and leave the persisted plugin registry stale.

## What This Changes

This PR adds a persisted installed-plugin-index policy-refresh fast path for plugin enable/disable, passes target plugin IDs from terminal CLI toggles, extends enable results with resolved IDs, updates tests, and adds a changelog entry.

## Best Possible Solution

Keep the policy-refresh optimization, but make the persisted-only path target-aware for every policy mutation path. Omitted or empty `policyPluginIds` should fall back to a full rebuild, or every caller including chat `/plugins enable` and `/plugins disable` should pass the resolved plugin ID, with regression coverage for both paths.

## Reproduction Assessment

Yes. On PR head, call `refreshPersistedInstalledPluginIndex` with `reason: "policy-changed"` and no `policyPluginIds` after a target plugin is absent from the persisted index; `hasPolicyRefreshTargets` returns `true`, so the source rebuild is skipped. The chat toggle caller currently provides that no-id path.

## Solution Assessment

No. The fast path is the right shape for terminal CLI toggles that pass explicit targets, but treating missing target IDs as safe is not the narrow maintainable solution; require target coverage or pass IDs from all policy mutation callers.
