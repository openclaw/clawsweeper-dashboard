Full review comments:

- **[P2] Handle path-based upload-file inputs:** `extensions/imessage/src/actions.ts:607-615`
  - body: This adapter advertises `upload-file`, but the shared message action contract allows `filePath`, `path`, or `media` and passes file access through `mediaAccess`/`mediaReadFile`. A normal `message({ action: "upload-file", filePath: ... })` call will still fail before reaching imsg because this path requires a base64 `buffer` plus `filename`.
  - confidence: 0.89
- **[P2] Keep formatting off the RPC send path:** `extensions/imessage/src/send.ts:216-237`
  - body: `sendMessageIMessage` strips Markdown and sends `params.formatting` through regular JSON-RPC `send`, but upstream imsg documents formatting on `send-rich`, not RPC `send`. Basic sends like `**bold**` can therefore become unstyled `bold` instead of preserving the original text or using the supported rich-send path.
  - confidence: 0.87
- **[P3] Add labeler coverage for the migration page:** `.github/labeler.yml:55-59`
  - body: This PR adds `docs/channels/imessage-from-bluebubbles.md`, but the iMessage label rule still only matches the existing iMessage doc. Add the migration page to the same labeler globs so future docs edits route to the channel owners.
  - confidence: 0.82
- **[P3] Narrow the owner-only cache threat model:** `extensions/imessage/src/monitor-reply-cache.ts:132-138`
  - body: The comment says `0600`/`0700` closes a same-UID read/injection vector, but owner-only modes do not stop another process running as the same Unix user. Narrow the claim to other local users, or add an actual integrity boundary if same-UID tampering remains in scope.
  - confidence: 0.86
