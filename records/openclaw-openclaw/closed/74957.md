## Summary

Close as implemented. Current main's Control UI chat header uses the Lit session/model controls rather than the reported `SessionHeader.tsx`/`modelName` path, and the visible model picker resolves its label from the active session row's `model` and `modelProvider`. The Gateway also exposes those fields through `sessions.list`/`sessions.patch`, with regression coverage, and the verified implementation is present in the shipped `v2026.4.27` tree.

## What This Changes

The issue requests that the Control UI session header show the active model name by reading the backend session object's `model` field instead of an outdated or mismatched `modelName` field.

## Best Possible Solution

Keep the shipped Control UI model picker/header behavior bound to the public `sessions.list` and `sessions.patch` `model`/`modelProvider` contract. If maintainers want an additional non-selector text label in the header, that should be tracked as a narrow UI enhancement rather than this backend-field mismatch bug.

## Reproduction Assessment

Yes. The reported mismatch can be checked against current main by following the chat header render path: it resolves the displayed model from `sessionsResult.sessions[].model` and `modelProvider`, and the backend tests verify those fields are returned after session model changes.

## Solution Assessment

Yes. The current implementation is the maintainable fix because it uses the Gateway's public session row contract instead of adding a compatibility alias for a stale `modelName` property or introducing the non-existent React `SessionHeader.tsx` path.
