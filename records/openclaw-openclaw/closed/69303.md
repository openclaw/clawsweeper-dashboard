## Summary

Current main implements the requested fix: replay-unsafe dispatch progress now causes the inbound dedupe key to be committed on the catch path instead of released, while failures before any replay-unsafe progress still release the in-flight key for safe retry. The implementation landed in commit 303cde8f6087f0b4859b26313df5cdc5cd0e4952 via merged PR #72291 and is recorded under Unreleased, with no release tag containing the fix yet.

## Best Possible Solution

Close #69303 as implemented on main by keeping the merged narrow auto-reply fix: safe failures before replay-unsafe progress still retry, while failures after visible output, tool/progress callbacks, or session progress commit the inbound dedupe key so the same external `message_id` is skipped. Broader duplicate-delivery hardening can stay in the related duplicate-message follow-up threads if maintainers want more layers later.
