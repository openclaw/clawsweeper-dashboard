## Summary

Keep open. Current `main` does not implement the proposed Feishu synchronous re-entry guard, and mainline tests still codify multiple streaming sessions from repeated `final` deliveries within one dispatcher lifecycle.

## Evidence

- **No synchronous re-entry guard on main:** `extensions/feishu/src/reply-dispatcher.ts:268-319` still guards `startStreaming()` only with `streamingStartPromise` and `streaming`, then clears both in `closeStreaming()`. There is no `streamingStarted`-style flag on current `main`.
  - file: [extensions/feishu/src/reply-dispatcher.ts:268](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/extensions/feishu/src/reply-dispatcher.ts#L268)

- **Current tests still expect multiple streaming sessions:** `extensions/feishu/src/reply-dispatcher.test.ts:329-350` explicitly expects two `FeishuStreamingSession` instances after two `final` payloads on the same dispatcher/options object, which matches the duplicate-card behavior this PR is trying to change rather than proving it is already fixed on `main`.
  - file: [extensions/feishu/src/reply-dispatcher.test.ts:329](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/extensions/feishu/src/reply-dispatcher.test.ts#L329)
