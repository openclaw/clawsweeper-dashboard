## Label Justifications

- `P1`: The report described a repeated watchdog hang that blocked a real agent/channel lane until a gateway bounce.
- `impact:session-state`: The root problem was a poisoned persisted Claude CLI session id being reused across turns.
- `impact:crash-loop`: The failure mode was a repeated timeout/kill/restart loop that made the lane unavailable for multiple watchdog windows.

## Summary

Current main already fixes the poisoned Claude CLI resume-id cascade, and the shipped v2026.5.28 release contains the same clear-after-failed-reuse path plus regression coverage.

## What This Changes

Discard the stored `claude-cli` resume session id after a reused turn fails with `AbortError` or a non-`session_expired` `FailoverError`, preventing repeated watchdog retries of poisoned state.

## Best Possible Solution

Keep the shipped clear-only path for reused `claude-cli` bindings; leave the separate MCP diagnostics and timeout-policy ideas to their own open follow-ups.

## Reproduction Assessment

No current-main failing reproduction remains. Source inspection and regression tests show reused Claude CLI bindings are cleared after `AbortError` or timeout `FailoverError`, so the next turn cannot reuse that poisoned id.

## Solution Assessment

Yes. Clearing the stored binding after a failed reused `claude-cli` turn is the narrow core runtime fix; timeout-cap and MCP visibility changes remain separate concerns.
