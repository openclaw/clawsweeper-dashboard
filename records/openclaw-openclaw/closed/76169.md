## Summary

Current main now implements the reported proxy-required provider path: `openai-completions` uses the guarded model fetch, and that fetch upgrades provider requests to trusted env-proxy mode when `HTTPS_PROXY` applies and no explicit dispatcher policy is configured.

## What This Changes

The issue asks OpenClaw to route `openai-completions` `/v1/chat/completions` requests through `HTTPS_PROXY` in proxy-required sandboxes.

## Best Possible Solution

Keep the targeted provider env-proxy routing now on main and let the next release carry it, while preserving strict guarded-fetch defaults for non-provider or explicitly pinned transport paths.

## Reproduction Assessment

No current-main failing reproduction path remains. The old failure path is source-understood, and current main now routes the same provider request shape through trusted env-proxy mode when `HTTPS_PROXY` applies.

## Solution Assessment

Yes. The current implementation is the narrow maintainable fix because it upgrades configured provider calls without globally weakening strict SSRF-guarded fetch behavior or overriding explicit provider dispatcher policy.
