## Summary

Do not cleanup-close this maintainer-labeled PR. It fixes a real WhatsApp recovery path and includes sufficient CLI proof, but the new same-id install gate can still send an already installed disabled/not-activated channel plugin through remote install before local enable/reuse; live GitHub now shows the PR merged, so the remaining action is a narrow follow-up repair rather than closing an open branch.

## What This Changes

The PR routes `channels add --channel <id>` through the trusted channel catalog for same-id externalized channels and adjusts doctor repair sequencing to preserve failed install ids while cleaning unrelated stale plugin config.

## Best Possible Solution

Keep the same-id externalized recovery behavior, but make `channels add` detect and enable/reuse already installed same-id channel plugins before attempting any remote install, with a regression test for the offline installed-but-disabled WhatsApp case.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: configure or index an installed same-id WhatsApp plugin that is disabled/not activated, make remote install unavailable, then run non-interactive `channels add --channel whatsapp`; the new gate reaches the install helper before local enable/reuse.

## Solution Assessment

No. Installing when the plugin is truly absent is the right direction, but the maintainable fix should split absent from installed-but-disabled/not-activated and only download for the absent case.
