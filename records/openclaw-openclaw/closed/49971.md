## Summary

Current `main` already exposes the public plugin-hook surfaces this RFC asks for: install-time gating, runtime tool-call gating, inbound communication interception, gateway-start hooks, and plugin-owned RPC/command surfaces. The proposed trust provider can be implemented on top of existing SDK contracts without a new core `onAgentVerify` hook.

## Risks / Open Questions

- There is no dedicated core `onAgentVerify` symbol or standardized `TrustVerificationResult`; plugins must map trust decisions onto existing hook results plus their own RPC/command surfaces.
- OpenClaw still does not ship a built-in DID/VC trust provider in core; this close only covers the requested extensibility seam, not a bundled provider or standard trust-score UX.
