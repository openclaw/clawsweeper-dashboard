## Summary

Keep open. Current main does not implement `channels.feishu.customEventHandlers` or any equivalent raw Feishu event subscription seam, and this PR remains a relevant implementation candidate. It should not be swept as a ClawHub-only item because the current plugin hook surface observes normalized messages, not arbitrary Feishu WebSocket event types on the existing connection.

## Best Possible Solution

Keep this PR open for maintainer review. If the capability is accepted, implement it as an explicit trusted Feishu event extension seam: add Feishu config schema/manifest/docs, normalize filesystem handler paths, honor fire-and-forget/webhook acknowledgment behavior, preserve existing queue/dedup semantics, and cover the reachable config path with focused tests. If maintainers decide this should be external, first provide a documented plugin API that lets a ClawHub plugin subscribe to selected Feishu event types on the existing connection.

## Risks / Open Questions

- Security review: the PR introduces config-driven dynamic import of operator-specified module paths inside the Gateway process. That is a real code-execution surface and needs explicit trusted-config semantics, schema/docs wording, path-to-file-URL normalization, and preferably reuse of the existing hook/module loader pattern before merge.
- Functional review: the PR diff also touches Feishu message, comment-notice, reaction, and bot-menu dispatch/dedup behavior around the custom handler work. Review comments already identified possible regressions in control-command queue lanes, comment serialization, non-retryable bot-menu replay handling, and webhook acknowledgment latency.
- Supply-chain review: the fetched PR file list shows no workflow, package, dependency, or lockfile changes; the main security-sensitive change is runtime module loading from config, not third-party dependency resolution.
