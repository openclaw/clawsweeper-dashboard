## Summary

Current main and the latest release already include the requested channel-plan participation in `autoMigrateLegacyState`: the early-return now checks `detected.channelPlans.hasLegacy`, and the auto path runs channel migration plans and merges their changes and warnings. The prior concern about doctor-only channel migration is superseded by shipped source/docs behavior that now treats bundled channel legacy state as a startup doctor-flow trigger and ships the auto path in `v2026.6.6`.

## What This Changes

The issue requests that channel legacy migration plans detected by `autoMigrateLegacyState` prevent the no-op early return and be applied with their changes and warnings in the returned result.

## Best Possible Solution

Keep the shipped startup migration behavior and close this report as fixed by the released auto-migration path.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not on current main: the reported early-return path is fixed because channel plans now participate in the no-legacy decision and are run through the existing migration runner.

## Solution Assessment

Is this the best way to solve the issue? Yes; the shipped implementation changes the decision point and uses the existing plan runner/result merge path rather than adding a parallel channel-specific migration path.
