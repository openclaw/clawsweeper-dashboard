## Summary

Keep open: current main and the latest tagged release still have unsynchronized commitments-store whole-file mutations, and an open closing PR already targets the fix path with real behavior proof.

## What This Changes

The issue requests serialization for commitments-store status, attempt, inferred-upsert, and expiry writes so concurrent writers cannot silently overwrite each other.

## Best Possible Solution

Review and land the existing serialization PR, or an equivalent narrow fix, that routes every commitments-store read-modify-save mutation through a per-store queue plus cross-process file lock with regression coverage.

## Reproduction Assessment

Yes. The issue provides a deterministic `Promise.all` repro against exported store functions, and current source inspection confirms both writers can read the same stale snapshot and save whole-file state afterward; I did not execute it in this read-only pass.

## Solution Assessment

Yes. Keeping the issue open while https://github.com/openclaw/openclaw/pull/81153 is reviewed is the right path; the narrow maintainable fix is a commitments-specific mutation gate with both same-process queueing and cross-process locking.
