## Summary

Keep open. The PR targets the right config validation mapper for the linked operator-diagnostics bug, but the submitted helper would add misleading bound hints for non-numeric Zod issues and exclusive numeric bounds, and it is missing the required changelog entry.

## What This Changes

The PR adds numeric min/max hint formatting to mapped Zod config validation errors and adds a regression test for `session.agentToAgent.maxPingPongTurns`.

## Best Possible Solution

Land a narrow mapper change that enriches only numeric-origin bounds, handles inclusive and exclusive constraints accurately, preserves non-numeric length/size messaging, covers those cases in focused tests, and records the user-facing diagnostics fix in `CHANGELOG.md`.

## Reproduction Assessment

Yes. Source inspection shows `session.agentToAgent.maxPingPongTurns: 10` reaches a `.max(5)` Zod schema while current main returns the raw Zod message through `mapZodIssueToConfigIssue` without a maximum-bound hint.

## Solution Assessment

No, not as submitted. `mapZodIssueToConfigIssue` is the right surface, but the helper needs numeric-origin gating, exclusive-bound wording, focused regression coverage, and a changelog line before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Gate bound hints to numeric-origin issues:** `src/config/validation.ts:492-509`
  - body: Zod emits `too_big` and `too_small` for string length, arrays, sets, and files too. Since this helper only checks the issue code plus numeric `maximum`/`minimum`, config fields such as `z.string().min(1)` would gain wording like `(minimum allowed value is 1)`, which misstates a length constraint as a numeric value bound; check `record.origin` before appending the hint.
  - confidence: 0.9
- **[P2] Honor exclusive numeric bounds:** `src/config/validation.ts:492-509`
  - body: Current config schemas use `z.number().positive()`, and Zod implements that as `gt(0)` with `inclusive: false`. As written, rejected `0` values would be described as `(minimum allowed value is 0)`, making the failing value look allowed; branch on `inclusive === false` and cover that path in tests.
  - confidence: 0.9
- **[P3] Add the changelog entry:** `src/config/validation.ts:518`
  - body: This changes user-facing config validation diagnostics, so repo policy requires a single-line Unreleased Fixes entry before merge. Add the entry with appropriate contributor credit instead of landing the diagnostics change silently.
  - confidence: 0.87
