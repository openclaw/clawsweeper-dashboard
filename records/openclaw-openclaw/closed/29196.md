## Close Comment

Closing this as `clawhub` after a Codex review of current `main`. OpenClaw already has the supported building blocks for this idea without adding a new core `RULES.md` bootstrap format: persistent operating constraints live in `AGENTS.md` standing orders (`docs/automation/standing-orders.md:32`), plugin hooks can inject prompt/system policy (`docs/concepts/agent-loop.md:94`), and tool calls can be hard-blocked or sent through approval with `before_tool_call` (`src/plugins/hook-types.ts:324`). The repo vision also says optional capability should usually ship as plugins and new skills should go to ClawHub first (`VISION.md:54`). Given that direction, a rules/constraints feature fits better as a ClawHub plugin that reads workspace policy and enforces it through existing hook seams. If that implementation hits a missing SDK seam, a focused follow-up issue for the missing hook/contract would be the right next step.
