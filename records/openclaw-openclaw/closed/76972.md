## Summary

Current main resolves the reported footgun by treating `plugin-runtime-deps` as legacy debris: packaged postinstall and doctor cleanup remove stale versioned roots and global symlinks, while docs now state plugin dependency convergence belongs to explicit install/update/doctor flows rather than gateway startup.

## What This Changes

The issue asks OpenClaw to avoid stale `plugin-runtime-deps/openclaw-<old-version>-...` trees after gateway upgrades for non-bundled linked plugins, either by creating a matching current tree or warning clearly.

## Best Possible Solution

Keep the current cleanup model: `plugin-runtime-deps` remains retired legacy state that package postinstall and `openclaw doctor --fix` remove, while external plugin dependencies stay owned by install/update flows and local plugin directories.

## Reproduction Assessment

No; against current main, source inspection shows packaged postinstall and doctor cleanup remove the stale roots and symlinks instead of leaving a version-mismatched dependency tree. I did not run the mutating upgrade fixture in this read-only review.

## Solution Assessment

Yes; removing the retired runtime-deps tree is narrower and more maintainable than creating a new startup-managed dependency cache, and the docs now align plugin dependency ownership with install/update/doctor flows.
