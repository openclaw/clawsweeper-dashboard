## Summary

Current main and the latest release already route stop requests through the active-run abort path, pass abort signals into agent/tool execution, and document/test subagent cascade stop, so this report can close as already implemented.

## What This Changes

The issue requests responsive cancellation so a user stop command can interrupt long-running tool execution or agent waits instead of being queued behind blocking calls.

## Best Possible Solution

Keep the shipped abort implementation; open a new narrow bug only if a current-main repro identifies a specific tool or provider path that ignores the established stop/AbortSignal contract.

## Reproduction Assessment

No; this read-only review did not establish a failing current-main reproduction path. Source, docs, and tests instead show that `/stop` fast-aborts active runs, foreground exec observes abort signals, and subagent stop cascades are already covered.

## Solution Assessment

Yes; the current implementation reuses the central active-run abort controller and task/subagent cancellation paths instead of adding a parallel cancellation system. Background/yielded exec staying alive is an intentional pollable-process behavior, not evidence that `/stop` is queued behind foreground tool execution.
