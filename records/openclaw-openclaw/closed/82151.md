## Summary

Keep open: current main intentionally treats cron `payload.model` as a per-job primary, but there is still no implemented diagnostic or migration policy that warns when stored cron pins diverge from the user's intended default/auth route. Current OpenAI docs also make `openai/*` the canonical Codex agent route, so the remaining work is maintainer policy rather than a clean implemented-on-main close.

## What This Changes

The issue asks OpenClaw to migrate, flag, or surface stale cron `payload.model` pins when model/auth routing changes so scheduled jobs do not silently diverge from the intended OpenAI/Codex auth route.

## Best Possible Solution

Add an owner-approved diagnostic, likely in `openclaw doctor` plus cron list/status output, that reports cron model overrides and their effective runtime/auth route while preserving intentional per-job model selection unless a migration policy is explicitly chosen.

## Reproduction Assessment

No for the full API-key billing bypass claim on current main; current source does give a high-confidence path for `payload.model` overriding default cron model selection. The auth/billing side needs a real OAuth/API-key setup or an owner-approved diagnostic contract to verify precisely.

## Solution Assessment

Unclear: removing or rewriting cron `payload.model` pins could break intentional job primaries. A doctor/status warning that shows effective route and auth is the safer direction, but maintainers need to choose the migration and UX policy.
