## Summary

Keep open. Current main still treats `plugins.slots.*` as scalar plugin-id selectors, while this RFC asks for a new public config/plugin contract with owner provenance and overwrite policy that needs maintainer API review.

## What This Changes

The issue proposes evolving `plugins.slots.<slot>` from a bare plugin-id string into a backward-compatible owner record with provenance fields.

## Best Possible Solution

Have maintainers choose the minimal public slot ownership contract first, then stage accepted work through schema/types, read normalization, docs/tests, canonical writes, and explicit override semantics.

## Reproduction Assessment

Not applicable: this is an RFC for a public config/plugin contract, not a bug report. Source, docs, tests, and release inspection give high-confidence proof that current behavior remains scalar slot selection.

## Solution Assessment

Unclear until maintainer design review. The staged rollout is plausible, but field names, provenance guarantees, migration behavior, and overwrite policy are public API choices rather than a safe automated cleanup fix.
