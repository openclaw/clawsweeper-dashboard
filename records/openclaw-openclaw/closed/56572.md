## Summary

Current main already mitigates the reported isolated-cron zombie-session problem by storing isolated runs under dedicated `...:run:<sessionId>` keys and pruning those run records by key+age, independent of `status: "done"`. The cron timer invokes that reaper from a `finally` block, so expired run sessions are cleaned up even when job execution fails.

## Risks / Open Questions

- Current code solves the leak/cost-retention problem by pruning expired `...:run:` records, not by persisting explicit `status: "done"` / `endedAt` on those records. If someone needs those fields for a separate UI/API contract, that would be a different follow-up.

## Close Comment

Closing from a Codex review because current `main` already covers the zombie isolated-cron session problem reported here. Isolated cron fires now persist as dedicated `agent:...:cron:<jobId>:run:<sessionId>` records (`src/cron/isolated-agent/run.ts:492`), and OpenClaw prunes expired cron-run records by key and age in `src/cron/session-reaper.ts:54` using `cron.sessionRetention`, without depending on `status: "done"`. That sweep is wired into the cron timer `finally` block (`src/cron/service/timer.ts:815`), and there are regression tests for both pruning and failure-path execution (`src/cron/session-reaper.test.ts:73`, `src/cron/service.session-reaper-in-finally.test.ts:49`). This is present in the `v2026.4.23` tree, so the accumulation behavior described against OpenClaw 2026.3.24 is no longer current. If you can still reproduce stale isolated cron run entries on a current build past the retention window, please open a new issue with fresh reproduction data, including the relevant `sessions.json` snippet and matching `cron/runs/<jobId>.jsonl` lines.
