## Summary

Current main and released builds already resolve the built-in `local` memory embedding provider through the manifest-backed provider lookup, so the reported unknown-provider failure is fixed; the former closing PR is now closed, and the remaining open related work is distinct local-server/provider feature work.

## What This Changes

The issue requests that `agents.defaults.memorySearch.provider: "local"` work in the memory CLI/runtime path instead of failing with `Unknown memory embedding provider: local`.

## Best Possible Solution

Keep the shipped manifest-backed provider contract for the built-in in-process `local` adapter, and track separate HTTP/self-hosted embedding-provider requests in their own issues or PRs.

## Reproduction Assessment

No for current main: I did not reproduce the failure live, and current source plus released trees resolve `local` through manifest-backed provider lookup. The historical failure remains well supported by the report and the merged fix PR's before/after evidence.

## Solution Assessment

Yes: the manifest-backed provider contract is the narrowest maintainable solution because it keeps provider ownership in `memory-core` and avoids adding manager-specific eager registration for one caller.
