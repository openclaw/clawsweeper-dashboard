## Summary

Keep open: current main still returns `skipped` during isolated cron local-provider preflight before the configured fallback chain is constructed, and the related retry PR only addresses slow local providers rather than fallback bypass.

## What This Changes

The issue asks cron isolated-agent model preflight to honor configured cloud/local fallback chains before skipping a run when the selected local primary endpoint is unreachable.

## Best Possible Solution

Make cron preflight evaluate the effective fallback candidate chain, or defer the skip until the normal fallback path is exhausted, while preserving `fallbacks: []`, alias resolution, subagent fallback policy, and the local endpoint probe cache.

## Reproduction Assessment

Yes, at source level. An unavailable local primary makes `src/cron/isolated-agent/run.ts` return `skipped` before `src/cron/isolated-agent/run-executor.ts` constructs `runWithModelFallback`, and the reporter supplied matching standalone and production traces.

## Solution Assessment

Yes. The narrow maintainable fix is to apply the existing cron fallback policy to the preflight decision before skipping; adding retries alone helps cold starts but does not solve configured cloud fallback bypass.
