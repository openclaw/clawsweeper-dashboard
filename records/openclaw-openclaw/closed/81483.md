## Summary

Current main already fixes the reported OpenAI Codex migration failure: live harness selection no longer lets stale session `agentHarnessId` pins override provider/model runtime policy, and regression coverage now verifies stale OpenAI PI pins are ignored. The fix is present on current main after https://github.com/openclaw/openclaw/pull/79238, but it is not in the latest stable release tag `v2026.5.7`.

## What This Changes

The issue asks for existing agents with stale PI session harness pins to bind to the Codex harness after OpenAI runtime migration, or to receive a clear repair path.

## Best Possible Solution

Keep the current main behavior: provider/model runtime policy drives harness selection, stale session pins are ignored or cleaned by doctor, and affected 2026.5.6 users should update to a build containing `02fe0d8978dbf5b7acc4529d505b31f044fdf481` or later.

## Reproduction Assessment

No for current main: the v2026.5.6 reproduction no longer matches source, because stale OpenAI `agentHarnessId: "pi"` is ignored and covered by regression tests. I did not run tests due the read-only review constraint, but the current code path and tests are explicit.

## Solution Assessment

Yes: resolving harness selection from provider/model runtime policy and using doctor to clean stale session state is the narrow maintainable fix. Reintroducing session or whole-agent pins would conflict with the documented runtime contract.
