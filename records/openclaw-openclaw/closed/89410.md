## Summary

Keep open: the PR addresses a real chat diagnostic hygiene problem, but the current patch is not merge-ready because it hides sanitized failure signals too broadly, hard-codes an incomplete chat-provider list, and still lacks real Telegram/chat proof.

## What This Changes

The branch passes message channel/provider context into embedded payload generation, suppresses exec-like and middleware tool-error warning payloads for selected chat providers/session keys, adds Telegram-focused regression tests, and updates Telegram docs.

## Best Possible Solution

Keep this as a redaction fix: derive chat-surface coverage from existing channel metadata, suppress raw command/stderr/stack details, preserve a generic failure notification for chat and cron, and require redacted real Telegram proof before merge.

## Reproduction Assessment

Yes, source-level: the PR head returns `showWarning: false` for middleware errors before cron/session exemptions, and its own updated test expects only assistant text after a middleware exec failure. I did not execute tests in this read-only review.

## Solution Assessment

No. The safer fix is to redact raw command/stderr details while keeping a generic failure signal, and to use channel metadata or a shared channel classifier instead of a partial provider list.
