## What This Changes

The issue reports that Discord audioAsVoice follow-up text and extra media sends lose reply threading when an implicit single-use reply target is consumed by the first internal voice send.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/95978 or an equivalent narrow Discord outbound fix that captures the reply target once inside the audioAsVoice logical-response branch while preserving ordinary single-use fanout behavior.

## Reproduction Assessment

Yes, at source level: current main resolves the single-use reply supplier separately for the voice, text, and media sends, and current tests expect the later audioAsVoice sends to have no replyTo. I did not run tests because this review is read-only, but the issue's failing expectation matches the inspected code.

## Solution Assessment

Yes: capturing the resolved reply target once for the audioAsVoice logical response is the narrow owner-boundary fix, because it avoids changing the shared single-use policy used by ordinary multi-send payloads.
