## What This Changes

The issue asks OpenClaw to expose Telegram sender fields, especially `msg.from.is_bot`, in agent-visible context so agents can distinguish bot-authored messages from human senders.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/96810, or an equivalent narrow projection through `SenderFacts`, `MsgContext`, and inbound sender metadata; keep broader Telegram provenance fields in separate work.

## Reproduction Assessment

Yes. Source inspection on current main shows Telegram has access to `msg.from.is_bot`, but the Telegram builder, shared context projection, and prompt metadata do not expose that value to the agent.

## Solution Assessment

Yes. The best fix is the narrow existing-seam projection used by the open linked PR; exposing raw Telegram payloads or waiting on the broader bot-targeting PR would be less focused.
