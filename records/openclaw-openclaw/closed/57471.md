## Summary

Current main does not implement the requested direct live-session redirect inside `runWithModelFallback()`. It still normalizes `LiveSessionModelSwitchError` as a failed fallback candidate and continues the chain; tests and docs now codify that behavior to avoid outer retry loops, but that leaves #57471's latency/telemetry request unresolved.

## Best Possible Solution

Keep this open. The best fix is a targeted `runWithModelFallback()` change that detects `LiveSessionModelSwitchError`, moves or inserts the error's provider/model as the next candidate, emits distinct redirect telemetry, and adds regression tests for target-in-list and target-not-in-list cases while preserving the existing infinite-loop protections.

## Evidence

- **Issue context:** #57471 specifically asks for `LiveSessionModelSwitchError` to jump/insert the locked target model and emit `live_session_redirect` instead of walking the full fallback chain as `candidate_failed`. The only provided related item, #57472, is a closed infinite-loop report with a similar live-session-switch/fallback interaction, not a canonical open follow-up for the remaining latency issue.
- **Current implementation continues the fallback chain:** `runWithModelFallback()` catches `LiveSessionModelSwitchError`, wraps it in `FailoverError` with reason `overloaded`, records it through `recordFailedCandidateAttempt()`, and `continue`s to the next candidate. That is the behavior reported by the issue, not the requested jump to the target provider/model.
