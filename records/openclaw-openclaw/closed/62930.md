## Summary

Current main already routes memory-core remote embedding HTTP through the env-proxy-aware guarded fetch path, and config `env` values are applied into the runtime environment before config substitution. This looks fixed on `main`, though I could not tie it to a released tag newer than `v2026.4.23`.

## Evidence

- **Memory remote HTTP now upgrades to trusted env-proxy mode:** `withRemoteHttpResponse()` passes `GUARDED_FETCH_MODE.TRUSTED_ENV_PROXY` when `shouldUseEnvHttpProxyForUrl(params.url)` is true. This is the shared remote HTTP path used by memory embedding requests.
  - file: [src/memory-host-sdk/host/remote-http.ts:15](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/memory-host-sdk/host/remote-http.ts#L15)
  - command: `nl -ba src/memory-host-sdk/host/remote-http.ts | sed -n '1,80p'`
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Proxy env detection explicitly covers HTTP_PROXY and HTTPS_PROXY:** `shouldUseEnvHttpProxyForUrl()` resolves `http_proxy`/`https_proxy` with upper-case fallbacks and only bypasses when `NO_PROXY` matches, so ambient shell proxy vars are part of the decision.
  - file: [src/infra/net/proxy-env.ts:34](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/infra/net/proxy-env.ts#L34)
