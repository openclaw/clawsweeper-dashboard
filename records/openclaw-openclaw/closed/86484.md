## Summary

Keep open: the PR is aimed at a real Discord voice wake-name bug, but the current fuzzy matcher can still accept ambient phrases with prepended words as wake names, and the PR body only provides mocked/local test proof rather than live Discord voice proof.

## What This Changes

The PR adds leading fuzzy wake-name matching for Discord OpenAI realtime voice, adds Discord voice E2E coverage, and updates the changelog.

## Best Possible Solution

Tighten the Discord wake-name matcher so fuzzy matches reject extra leading tokens/prefixes, add regression coverage for article-prefixed false wakes, and include redacted live Discord voice proof before landing.

## Reproduction Assessment

Yes, from source inspection: with `requireWakeName` enabled, the PR's candidate builder can compact `a Molty, ...` to `amolty`, and the new fuzzy rule accepts distance 1 against `molty`. I did not run the Vitest suite in this read-only review.

## Solution Assessment

No; the direction is right, but the current implementation is too permissive for a wake-name gate. The safer fix is to distinguish ASR misspellings from extra leading tokens before applying edit-distance acceptance.
