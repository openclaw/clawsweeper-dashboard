## Summary

Current `main` already replaces the Windows Scheduled Task launch command with a staged `gateway.cmd` wrapper, keeps `schtasks` quoting separate from the inner `cmd.exe` command parsing, and falls back to launching the wrapper directly when Task Scheduler accepts `/Run` but never actually starts the gateway. That matches the reported failure mode for non-ASCII Windows profile paths and is covered by dedicated regression tests. The implementation is also present in release `v2026.4.22`.

## Close Comment

Closing from a Codex review: this looks implemented on current `main` and shipped in `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). The Windows gateway service path now stages a `gateway.cmd` wrapper and points Scheduled Task `/TR` at that script instead of handing the raw Node + `dist/index.js` command line directly to `schtasks` (`src/daemon/schtasks.ts:87`, `src/daemon/schtasks.ts:242`, `src/daemon/schtasks.ts:820`). There is dedicated regression coverage for that exact no-op start path in `src/daemon/schtasks.startup-fallback.test.ts:247`. Given the current implementation and shipped release, I’m closing this as already fixed on main.
