## Label Justifications

- `P1`: The original report affected live agent replies and provider routing for Anthropic/OpenRouter configurations, making it an urgent channel/provider workflow regression.
- `impact:session-state`: The issue concerned how inbound metadata and user turns were represented in session/model prompt state.
- `impact:message-loss`: The reported provider rejection and fallback behavior could suppress or duplicate expected channel replies.
- `impact:auth-provider`: The failure depended on provider/model routing behavior for Anthropic models through OpenRouter or direct Anthropic routes.

## Summary

Current main and v2026.6.6 already solve the reported consecutive user-role metadata injection: inbound metadata is carried as current-turn context and joined with the active prompt as one user submission, while the related fallback retry transcript duplication was also shipped.

## What This Changes

The issue asks OpenClaw to stop injecting group/channel metadata as separate user-role messages before the actual user message, which broke Anthropic/OpenRouter role compatibility and caused fallback behavior.

## Best Possible Solution

Keep the shipped role-alternation-safe prompt boundary for this issue, and let the separate runtime-context PRs handle any further product decision about hiding metadata from prompt text entirely.

## Reproduction Assessment

No current-main failing reproduction remains from source inspection: current metadata flows through `currentInboundContext` and is joined with the active prompt instead of becoming separate user-role turns. I did not run a live Discord/OpenRouter capture in this read-only review.

## Solution Assessment

Yes. The shared prompt-boundary implementation is the right maintainable fix for this report because it covers Discord, Telegram, Feishu, WebChat, and sibling inbound paths without adding provider-specific user-turn squashing for the current turn.
