## Summary

Current main implements the observable fix for #69790. Telegram DM voice/audio now enters the normal agent path through preflight transcription, same-session embedded runs are serialized before transcript lock acquisition, persistent old live-PID lock files are reclaimed, and in-process held locks have watchdog cleanup. The reported model-fallback masking of `session file locked` is also covered in the shipped v2026.4.24 changelog.

## Best Possible Solution

Keep the current main behavior: normal Telegram audio should flow through the bundled Telegram preflight transcription path, same-session agent turns should serialize through the session lane, and lock recovery should stay in the generic session write-lock layer. If a public synthetic inbound API is still desired, track that as a separate plugin/channel SDK feature rather than reopening this lock-wedge bug.

## Risks / Open Questions

- Directly importing a built `dist/extensions/telegram/...` bot module from another process is still an internal path, not a stable inbound injection API.
- A legitimate fresh active run can still hold the transcript write lock until it finishes or times out; current main fixes the persistent stale/live lock wedge rather than allowing concurrent transcript writers.
