## Summary

Keep open. Current main still ties mutating-tool error clearing to an action fingerprint that includes message `path`/`filePath`, so a `message` retry that succeeds after changing the attachment path will not clear the earlier failure. The same behavior is present in `v2026.4.22`.

## Evidence

- **Error only clears when the succeeding tool call matches the stored mutating-action fingerprint:** `handleToolExecutionEnd` preserves `lastToolError` for mutating tools unless `isSameToolMutationAction(...)` returns true for the later successful call.
  - file: [src/agents/pi-embedded-subscribe.handlers.tools.ts:818](https://github.com/openclaw/openclaw/blob/7d471837369885c12d540d6bafa319ff41129b89/src/agents/pi-embedded-subscribe.handlers.tools.ts#L818)
  - command: `nl -ba src/agents/pi-embedded-subscribe.handlers.tools.ts | sed -n '800,835p'`
  - sha: [7d4718373698](https://github.com/openclaw/openclaw/commit/7d471837369885c12d540d6bafa319ff41129b89)
- **Message fingerprints currently include attachment path aliases:** `buildToolActionFingerprint` appends `path`, `file_path`, `filePath`, `filepath`, and `file` for all mutating tools before also appending `to`/`target`. A retry that changes only `filePath` therefore gets a different fingerprint even if the message destination is the same.
  - file: [src/agents/tool-mutation.ts:170](https://github.com/openclaw/openclaw/blob/7d471837369885c12d540d6bafa319ff41129b89/src/agents/tool-mutation.ts#L170)

There is no message-tool test covering a corrected attachment path retry, and the generic fingerprint tests explicitly treat path aliases as the stable target key.
