## Summary

Current main no longer reproduces the reported 30s restart delay: gateway startup always supplies the gateway port to the lock code, and `acquireGatewayLock` now treats a free configured port as a dead owner immediately, with a darwin regression test covering the recycled/alive-PID case.

## Evidence

- **Gateway startup passes the real port into the lock path:** `run.ts` passes `lockPort: port` into `runGatewayLoop`, and `run-loop.ts` acquires the gateway lock with that port.
  - file: [src/cli/gateway-cli/run.ts:656](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/cli/gateway-cli/run.ts#L656)
  - command: `nl -ba src/cli/gateway-cli/run.ts | sed -n '650,665p'`
  - sha: [3a64aa49a901](https://github.com/openclaw/openclaw/commit/3a64aa49a9014518fa1977fbc023f3e0b978ae5d)
- **Free port short-circuits stale lock handling:** `resolveGatewayOwnerStatus` checks the configured port first and returns `"dead"` immediately when the port is free, before PID/start-time fallback logic.
  - file: [src/infra/gateway-lock.ts:186](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/infra/gateway-lock.ts#L186)
  - command: `nl -ba src/infra/gateway-lock.ts | sed -n '186,199p'`
  - sha: [3a64aa49a901](https://github.com/openclaw/openclaw/commit/3a64aa49a9014518fa1977fbc023f3e0b978ae5d)
- **Regression test covers the reported macOS-style case:** There is a test asserting the lock is reclaimed when the owner PID appears alive but the configured port is free on `darwin`, which is the exact stale-restart scenario from the issue.
  - file: [src/infra/gateway-lock.test.ts:249](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/infra/gateway-lock.test.ts#L249)
  - command: `nl -ba src/infra/gateway-lock.test.ts | sed -n '228,270p'`
  - sha: [3a64aa49a901](https://github.com/openclaw/openclaw/commit/3a64aa49a9014518fa1977fbc023f3e0b978ae5d)
- **Runtime singleton enforcement is bind-based:** Gateway startup also treats socket bind as the authoritative single-instance guard and throws `GatewayLockError` on `EADDRINUSE`; the docs describe this as the intended mechanism.
  - file: [src/gateway/server/http-listen.ts:18](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/gateway/server/http-listen.ts#L18)
  - command: `nl -ba src/gateway/server/http-listen.ts | sed -n '1,90p'`
  - sha: [3a64aa49a901](https://github.com/openclaw/openclaw/commit/3a64aa49a9014518fa1977fbc023f3e0b978ae5d)

## Risks / Open Questions

- `src/infra/gateway-lock.ts` still does not register an immediate process-exit cleanup handler, so the lock file itself can still persist after an early crash; the reason to close here is that the reported 30s restart delay no longer reproduces on current main because the free-port check reclaims it immediately in the real gateway startup path.
