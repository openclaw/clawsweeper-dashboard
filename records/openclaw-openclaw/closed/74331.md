## Summary

This PR should close as superseded by the active canonical proxy work in #58034. Current main still lacks a `web_fetch` env-proxy opt-in, but this branch unconditionally changes the default security boundary while #58034 tracks the narrower default-off config/docs/test design.

## What This Changes

The PR adds `useEnvProxy: true` to the `web_fetch` guarded-fetch call and adds a regression test asserting the flag is forwarded.

## Best Possible Solution

Keep the remaining work centralized in #58034: preserve strict DNS pinning by default, add a documented explicit trusted-env-proxy opt-in if maintainers accept that trust model, and keep the proxy applicability, `NO_PROXY`, hostname-policy, config-schema, docs, and regression-test coverage together.

## Reproduction Assessment

Yes. The code-level path is high confidence: current main has a test proving `web_fetch` keeps pinned DNS with `HTTP_PROXY`, while the PR body describes the proxy-only DNS-blocked environment where that strict path fails before the proxy can resolve the host.

## Solution Assessment

No. The one-line unconditional `useEnvProxy: true` is not the best fix because it changes default `web_fetch` SSRF/DNS behavior; the safer path is the explicit opt-in and hardening tracked in #58034.
