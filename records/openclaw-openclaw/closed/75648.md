## Summary

Keep open: current main has configurable provider timeouts, SDK Retry-After sleep caps, and a progress-aware idle watchdog, but it still does not carry provider Retry-After metadata through failover into OpenAI-compatible HTTP/SSE errors.

## What This Changes

The issue asks OpenClaw to make embedded-run upstream timeouts tunable and preserve provider 429/Retry-After details instead of surfacing only generic timeout errors.

## Best Possible Solution

Keep the shipped timeout controls and add generic provider retry-timing propagation through failover metadata plus OpenAI-compatible HTTP and SSE error mapping.

## Reproduction Assessment

Yes, at source level: construct or coerce a provider 429 with Retry-After headers, then inspect FailoverError, describeFailoverError, resolveOpenAiCompatError, and the OpenAI-compatible endpoints; current main has no field or header path to carry the retry timing. The live production throttling sequence was not rerun.

## Solution Assessment

Yes for the narrowed repair: extend the existing failover metadata and OpenAI-compatible error mapper to carry retry timing, set Retry-After for non-streaming responses, and include streaming-safe retry metadata. Changing timeout defaults or implementing durable queue/backoff policy would be broader than this issue’s remaining gap.
