## Summary

Keep open: this is useful Gateway security hardening, but the branch is not merge-ready against current main because it does not account for the newer `auth.approvalRuntimeToken` field and still needs protocol docs for the new public limits.

## What This Changes

The PR adds Gateway connect-auth length primitives, applies them to `auth.token`, `auth.bootstrapToken`, `auth.deviceToken`, and `auth.password`, adds validator tests, and records the change in the changelog.

## Best Possible Solution

Land a rebased schema-bound fix that covers every current connect auth field, documents the public wire limits, and preserves operator approval runtime auth.

## Reproduction Assessment

Yes. Source inspection on current main shows sub-64 KiB `auth.*` strings pass `ConnectParamsSchema`; the connect handler validates that schema before auth work, and bootstrap tokens then flow into locked stored-token scans.

## Solution Assessment

Mostly yes, but not merge-ready. Schema-level `maxLength` rejection is the right boundary before deeper auth work; the branch must be rebased to preserve and bound `approvalRuntimeToken` and document the new wire caps.
