## Summary

Current main already fixes the reported Telegram Web unreadable-message regression: rich Telegram messages are now opt-in, and default text delivery uses standard `sendMessage`/HTML fallback. The fix is main-only after `v2026.6.8`, so affected release users need the next patch release or a current-main build.

## What This Changes

The issue reports that Telegram Web cannot render OpenClaw messages after v2026.6.8 switched outbound Telegram delivery to `sendRichMessage`, and asks for standard `sendMessage` fallback or a disabling config option.

## Best Possible Solution

Keep the current main behavior: standard Telegram `sendMessage`/HTML delivery by default, with Bot API rich messages available only through explicit `richMessages` opt-in and the next patch release carrying that fix.

## Reproduction Assessment

Yes for the reported released behavior from source and related report evidence: v2026.6.8 routes Telegram text through rich-message sends, producing the unsupported-message symptom reported by multiple users. Current main no longer follows that default path; I did not run a live Telegram Web repro in this read-only review.

## Solution Assessment

Yes. PR 93279's default-compatible `sendMessage` path with explicit `richMessages` opt-in is the narrow owner-boundary fix and avoids adding a separate disable flag for the same behavior.
