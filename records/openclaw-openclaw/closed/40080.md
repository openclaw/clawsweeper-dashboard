## Summary

The PR's reported restart-loop root cause does not match current `main`: IRC startup now stays pending until abort through the passive monitor lifecycle, and there is a focused regression test covering that contract.

## Evidence

- **IRC startup no longer returns immediately:** `startIrcGatewayAccount` awaits `runStoppablePassiveMonitor(...)` and passes the account abort signal through to the monitor start path, so the gateway task does not resolve right after connect.
  - file: [extensions/irc/src/gateway.ts:17](https://github.com/openclaw/openclaw/blob/bfa6708c03f24c7caa5d0a96a298c292be0d85c1/extensions/irc/src/gateway.ts#L17)
  - command: `nl -ba extensions/irc/src/gateway.ts | sed -n '17,55p'`
  - sha: [bfa6708c03f2](https://github.com/openclaw/openclaw/commit/bfa6708c03f24c7caa5d0a96a298c292be0d85c1)
- **Passive monitor lifecycle explicitly waits for abort:** `runStoppablePassiveMonitor` delegates to `runPassiveAccountLifecycle`, which awaits `waitUntilAbort(params.abortSignal)` before calling `stop(handle)`. That is the opposite of the PR's claimed 'startAccount returned immediately' failure mode.
