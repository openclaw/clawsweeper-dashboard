## Summary

Current main and the shipped release already satisfy the safe delivery invariant: generic wake system events are drained into the model prompt by the normal reply runner, while only heartbeat-owned exec and tagged cron events are held for dedicated heartbeat prompts. The reported transcript grep is not proof that the model missed the event because heartbeat transcript persistence intentionally records the generic heartbeat placeholder instead of the full command body.

## What This Changes

The issue reports that `/hooks/wake` with `mode:"now"` deletes generic wake system events during the heartbeat without rendering their text to the agent prompt.

## Best Possible Solution

Keep the shipped split: generic wake text drains as normal `System:` prompt context, while exec and tagged cron events remain owned by dedicated heartbeat prompts.

## Reproduction Assessment

No. Source and release-tag inspection show generic wake events are prepended to the model command body; the reported transcript grep checks the persisted heartbeat placeholder, not the full prompt sent to the model.

## Solution Assessment

Yes. The current split is the maintainable boundary: untrusted generic wake text is normal system-event context, not a heartbeat-owned prompt, while exec and tagged cron events keep their dedicated prompt owners.
