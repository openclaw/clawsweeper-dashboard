## Label Justifications

- `P2`: This is a normal-priority runtime/session improvement with bounded Copilot and QQBot blast radius, but it has merge blockers before landing.
- `merge-risk: 🚨 session-state`: Persisting Copilot SDK session IDs changes session lifetime across process restarts and can mis-resume after reset when durable delete fails.
- `merge-risk: 🚨 auth-provider`: The QQBot app access-token POST is a secret-bearing auth path now routed through a new guarded fetch path without matching real runtime proof.

## Summary

Keep open. The Copilot persistence direction is useful and current main does not already have it, but the PR still has a session-state blocker: reset delete failures are only masked by process-local state, so a restart can reuse a stale SQLite binding. The QQBot guard also still needs redacted runtime proof or an explicit maintainer override before merge.

## What This Changes

The branch persists Copilot SDK session IDs in plugin-state SQLite with compatibility fingerprints and wraps QQBot app-token POSTs in the plugin SDK SSRF guard with capture disabled.

## Best Possible Solution

Land after reset invalidation is durable or fail-closed across process restarts, and after the QQBot token guard has redacted real runtime proof or an explicit maintainer proof override.

## Reproduction Assessment

Yes from source inspection: force a plugin-state delete failure during reset, keep the stale binding row, then construct a fresh harness; the new process no longer has resetBlockedStoredSessions and can look up the old row. I did not run tests because this review is read-only.

## Solution Assessment

No. Persisting Copilot bindings is the right direction, but reset invalidation needs to survive restart or fail closed, and the QQBot auth fetch change needs real runtime proof or maintainer override before this is merge-ready.
