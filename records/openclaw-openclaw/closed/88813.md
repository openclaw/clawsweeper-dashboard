## Summary

Keep open: current main still routes an unknown config schema lookup through the gateway RPC error path and agent gateway tool failure path, so the requested in-band `schema_path_not_found` tool result is not implemented.

## What This Changes

Issue requests that unknown `gateway` tool `config.schema.lookup` paths return a structured non-throwing tool result instead of surfacing a raw gateway/tool failure in channels.

## Best Possible Solution

Handle schema lookup misses at the agent gateway-tool/RPC translation boundary by returning a non-error tool result such as `{ ok: false, code: "schema_path_not_found", path }`, while preserving direct RPC semantics unless maintainers intentionally change the protocol.

## Reproduction Assessment

Yes, source-reproducible: unknown lookup paths return `null`, the gateway RPC turns that into a failed response, and the agent gateway tool does not catch it before the tool-error warning path can surface it. I did not run a live Slack reproduction in this read-only pass.

## Solution Assessment

Yes, the best fix is to convert this specific lookup miss before it becomes a tool execution error; suppressing channel chips alone would leave the model-facing tool result and other channels inconsistent.
