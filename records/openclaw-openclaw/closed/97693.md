## Summary

Keep open: current main still has the unbounded Discord `requestDiscord` success-path read, while this focused PR adds a bounded reader and real loopback proof without a concrete correctness defect. The remaining decision is maintainer acceptance of the new exported 4 MiB fail-closed response cap.

## What This Changes

The PR replaces Discord `requestDiscord` successful-response `res.text()` with a 4 MiB bounded reader and adds loopback HTTP regression tests for normal and oversized responses.

## Best Possible Solution

Land the bounded success-body reader after maintainers accept the 4 MiB exported API cap, or adjust the cap to match the sibling Discord REST boundary before merge.

## Reproduction Assessment

Yes. Current main visibly uses unbounded `res.text()` on the success path, and the PR body plus added loopback tests provide a high-confidence path for normal and oversized response behavior; I did not rerun tests because this cleanup review is read-only.

## Solution Assessment

Yes, with maintainer cap acceptance. The patch is in the owning Discord plugin, reuses the shared bounded response reader, preserves normal JSON parsing, and turns oversized success bodies into a labeled overflow error instead of buffering them unboundedly.
