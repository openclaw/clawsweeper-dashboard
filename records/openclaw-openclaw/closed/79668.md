## Summary

Keep open; this is a plausible provider-owned fix, but the current retry path can still miss a header-only/no-SSE stall and return an empty successful attempt instead of retrying.

## What This Changes

The branch adds a Gemini 3 direct Google first-response retry, moves Gemini tool-schema normalization into shared Google hooks, updates Google tests, and adds a changelog entry.

## Best Possible Solution

Keep the provider-local retry and shared hook direction, but make a post-header/no-SSE first-attempt timeout retryable and add regression coverage for that stream shape before merge.

## Reproduction Assessment

Yes. The linked issue and PR body give concrete direct Google Gemini main-session commands, timeout logs, and copied live before/after output; I did not run credentials during this read-only pass.

## Solution Assessment

No. The solution direction is narrow and maintainable, but the current implementation can classify a timeout-cancelled body as a clean empty stream instead of retrying.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Retry after header-only first-attempt stalls:** `extensions/google/transport-stream.ts:828-832`
  - body: When Google returns response headers but sends no SSE event, the retry deadline aborts the child signal and `parseGoogleSseChunks` cancels the reader. A cancelled stream can resolve the pending `iterator.next()` as `done`, so this branch returns `ready` with no first chunk instead of retrying, missing the reported no-first-SSE stall and possibly finalizing an empty response. Check the timeout state before treating `first.done` as success and add a test where fetch resolves to a stream that never emits.
  - confidence: 0.87
