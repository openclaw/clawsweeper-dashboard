## Summary

Close as superseded: a better open PR already owns the same cron duplicate-text bug, and this PR's prompt-layer approach can hide the actual cron payload in isolated heartbeat runs where the referenced system message is not in the run session.

## What This Changes

This PR changes heartbeat cron prompt construction so cron events tagged with `contextKey: "cron:*"` are referenced as already-delivered system messages instead of embedding the event text, with updated heartbeat tests.

## Best Possible Solution

Use https://github.com/openclaw/openclaw/pull/91287 as the canonical landing path because it removes the duplicate generic `System:` render while keeping heartbeat as the single owner that includes cron event text.

## Reproduction Assessment

Yes, source-level. Current main generically renders cron-tagged system events while heartbeat also wraps those events, and the linked issue plus the superseding PR's proof cover the duplicate raw-plus-wrapper path.

## Solution Assessment

No. Referencing a prior system message in the heartbeat prompt is too narrow because isolated heartbeat runs can use a different run session; filtering cron-tagged entries out of generic `System:` rendering is the safer owner-boundary fix.

- **[P1] Keep cron payload text when no system line is in the run session:** `src/infra/heartbeat-runner.ts:1250`
