## Summary

Keep this open: current main and the latest shipped release still allow timeout/no-output-timeout subprocess results to reach plugin callers with a nullable `code`, so the cache-poisoning path remains source-reproducible even though the macOS launchd fan-out was not live-reproduced here.

## What This Changes

The issue reports slow OpenClaw CLI subprocess startup under macOS launchd fan-out plus plugin-facing timeout results that can poison ClawKitchen's subprocess cache.

## Best Possible Solution

Harden plugin-facing timeout and no-output-timeout subprocess results first, then separately profile the macOS recipes/Kitchen launchd startup path before changing worker scheduling or CLI architecture.

## Reproduction Assessment

Yes at source level: current `src/process/exec.ts` can still return a nullable code on timeout paths, and `src/plugins/runtime/runtime-system.ts` exposes that raw helper to plugins. I did not run the 21-plist macOS launchd setup or the external recipes/Kitchen plugins.

## Solution Assessment

Yes for a narrow first repair: normalizing timeout/no-output-timeout results before plugin callers treat partial output as success is the smallest maintainable fix. Broader startup profiling, launchd jitter, or worker RPC redesign should be separate measured follow-up work.
