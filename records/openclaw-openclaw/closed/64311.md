## Summary

This is a valid feature request, but current OpenClaw scope puts cross-channel shared memory on the plugin path rather than in core. Main intentionally keeps room/channel sessions isolated, while the existing Honcho plugin already provides cross-session and cross-channel memory with message search and prompt injection.

## Evidence

- **Vision: optional memory/runtime features belong in plugins:** OpenClaw's vision says core should stay lean, optional capability should usually ship as plugins, memory is a dedicated plugin slot, and plugin promotion belongs in ClawHub rather than core.
  - file: [VISION.md:52](https://github.com/openclaw/openclaw/blob/11804a484ded89374afa55020813cfa265aa143a/VISION.md#L52)
  - sha: [11804a484ded](https://github.com/openclaw/openclaw/commit/11804a484ded89374afa55020813cfa265aa143a)
- **Plugin distribution path is the documented default:** The plugin guide says new capabilities should be published to ClawHub or npm instead of being added to the OpenClaw repository, and specifically calls out tool/hook plugins as the right surface for custom runtime behavior.
  - file: [docs/plugins/building-plugins.md:11](https://github.com/openclaw/openclaw/blob/11804a484ded89374afa55020813cfa265aa143a/docs/plugins/building-plugins.md#L11)
  - sha: [11804a484ded](https://github.com/openclaw/openclaw/commit/11804a484ded89374afa55020813cfa265aa143a)
- **SDK already exposes the needed extension seams:** The SDK overview documents plugin registration for tools, hooks, services, gateway methods, memory prompt supplements, and memory corpus supplements, which is enough surface area for a shared-memory plugin rather than a core-only feature.
  - file: [docs/plugins/sdk-overview.md:61](https://github.com/openclaw/openclaw/blob/11804a484ded89374afa55020813cfa265aa143a/docs/plugins/sdk-overview.md#L61)
  - sha: [11804a484ded](https://github.com/openclaw/openclaw/commit/11804a484ded89374afa55020813cfa265aa143a)
- **Existing plugin already targets cross-session/channel memory:** The Honcho docs explicitly say it is for persistent memory that works across sessions and channels, persists conversations after every turn, carries context across channel switches, and exposes tools such as `honcho_context` and `honcho_search_messages`. It is installed as `@honcho-ai/openclaw-honcho`.
  - file: [docs/concepts/memory-honcho.md:1](https://github.com/openclaw/openclaw/blob/11804a484ded89374afa55020813cfa265aa143a/docs/concepts/memory-honcho.md#L1)
  - sha: [11804a484ded](https://github.com/openclaw/openclaw/commit/11804a484ded89374afa55020813cfa265aa143a)
- **Core session behavior remains intentionally isolated by room:** Session docs say group chats are isolated per group and rooms/channels are isolated per room; the session-key implementation only applies `dmScope` and `identityLinks` to direct-message keys, while non-DM group/channel keys remain room-specific.
  - file: [src/routing/session-key.ts:130](https://github.com/openclaw/openclaw/blob/11804a484ded89374afa55020813cfa265aa143a/src/routing/session-key.ts#L130)
  - sha: [11804a484ded](https://github.com/openclaw/openclaw/commit/11804a484ded89374afa55020813cfa265aa143a)

That is useful product feedback for memory-plugin design, but it does not change the current core/plugin boundary.
  - command: `GitHub issue #64311 body and comments`

## Risks / Open Questions

- Operators who want a bundled local-only shared-memory solution still need a plugin or custom hook today; closing this does not add a built-in default.
- Honcho is an external plugin path, so teams that need a different storage model may still want a separate plugin implementation.
