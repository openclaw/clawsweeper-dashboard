## Summary

Current main already covers the requested workflow with active-run queue steering, explicit `/steer`, configurable `messages.queue` modes including `collect` and `interrupt`, and runtime-specific batching/merge behavior documented for Pi and Codex.

## What This Changes

The issue requests interrupting or steering an active message run so later user input can be merged into the active context instead of waiting for the original run to finish.

## Best Possible Solution

Keep the current queue-steering design: use default `/queue steer` for same-run context injection, `/queue collect` for quiet-window aggregation, and `/queue interrupt` only when the newest prompt should replace the active run.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken existing behavior report. Current-main docs, source, and tests provide a high-confidence behavior check for the requested mid-run context-merging workflow.

## Solution Assessment

Yes: current main solves the user workflow through runtime-aware steering and queue modes instead of arbitrary tool-call interruption, which preserves tool/result consistency while still giving users explicit collect and interrupt controls.
