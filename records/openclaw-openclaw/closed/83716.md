## Label Justifications

- `P3`: This is a feature/ergonomics request that is already covered by current hook and policy surfaces rather than an urgent regression.
- `impact:security`: The request is about pre-execution tool policy enforcement for built-in file and runtime tools, which is a security-boundary concern.

## Summary

Current main and the v2026.5.12 release already provide the requested enforcement path: plugins can use `before_tool_call` to block, require approval for, or rewrite OpenClaw-owned built-in tool calls, and per-agent tool policy has allow/deny controls.

## What This Changes

Request a per-agent way to disable built-in tools or a blocking pre-tool hook so plugins can enforce tool-call policy before execution.

## Best Possible Solution

Keep the shipped `before_tool_call` hook and existing tool policy controls as the supported path; a separate `disabledTools` key is not needed for the stated enforcement use case.

## Reproduction Assessment

Not applicable: this is a feature request, not a bug report. The relevant verification is source/docs inspection showing current main already has the requested blocking pre-tool hook and per-agent tool policy fields.

## Solution Assessment

Yes. The implemented `before_tool_call` hook is the more flexible requested option and avoids relying on MCP name replacement, while existing `tools.deny`/per-agent policy handles static disabling needs.
