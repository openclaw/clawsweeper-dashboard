## Summary

Current main already routes M4A/M4B/F4A/F4B ISO-BMFF ftyp brands as audio through the shared MIME sniffing path because OpenClaw pins file-type 22.0.1, whose ftyp switch returns audio MIME types for those brands. The LINE-specific and Slack-specific discussion remains useful context, but the shared core path now resolves the observable transcription skip for M4A/M4B-style audio-only containers.
