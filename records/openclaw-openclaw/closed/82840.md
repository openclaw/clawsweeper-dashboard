## Summary

Keep this PR open. It fixes several real Signal group ID lowercasing sites and the added proof is useful, but source review shows the normal channel inbound wrapper and group metadata resolver can still lowercase the mixed-case Signal group ID, so the patch is incomplete; the protected `maintainer` label also requires explicit maintainer handling.

## What This Changes

The PR preserves Signal group ID casing in routing/session-key builders, explicit session-key normalization, session-store key normalization, and focused regression tests.

## Best Possible Solution

Land one complete fix that preserves Signal group IDs at every session ingress point, including the channel wrapper and group metadata, while keeping structural tokens and non-Signal keys canonicalized.

## Reproduction Assessment

Yes for the source-level corruption: current main still lowercases mixed-case Signal group IDs in route/session/store-adjacent paths. I did not run a live Signal daemon reproduction in this read-only review.

## Solution Assessment

No. The direction is right, but the patch is not the best complete fix until the same opaque-ID preservation is applied to `recordInboundSession` and `resolveGroupSessionKey`, with regressions covering those public paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve channel session keys before recording:** `src/routing/resolve-route.ts:664`
  - body: The route key is preserved here, but the normal inbound turn path still calls `recordInboundSession`, which lowercases both `sessionKey` and `update.sessionKey` before the store sees them (`src/channels/session.ts:42` and `src/channels/session.ts:63`). A Signal group reply can therefore still persist or update a lowercased `agent:main:signal:group:<id>` key, so this wrapper needs the same opaque-ID-preserving normalization.
  - confidence: 0.93
- **[P2] Preserve the stored groupId metadata too:** `src/config/sessions/store.session-key-normalization.test.ts:85`
  - body: This new regression checks the store key and origin target, but session metadata still derives `groupId` through `resolveGroupSessionKey()`, which lowercases the resolved id at `src/config/sessions/group.ts:152`. That leaves the reported inconsistency intact: preserved `origin.to` with a corrupted `groupId`, and recovery/policy paths that trust `entry.groupId` can still miss the Signal group.
  - confidence: 0.9
