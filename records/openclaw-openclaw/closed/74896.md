## Summary

Keep open. Current main still has a pre-bind Gateway startup path where any present `tools.web.fetch` object disables the secrets fast path and reaches web-fetch provider discovery before the HTTP listener binds; two open PRs also explicitly claim to close this issue, so maintainers should review, land, replace, or close the canonical fix before this issue is resolved.

## What This Changes

Gateway startup should bind its HTTP port when `openclaw.json` contains `tools.web.fetch`, and a disabled fetch config should not force blocking provider discovery before bind.

## Best Possible Solution

Keep the issue open and land one canonical fix that prevents optional or disabled web-fetch configuration from blocking Gateway bind. The end state should either skip credential-free disabled fetch surfaces before provider discovery or defer nonessential fetch-provider discovery until after the listener is live, while preserving fail-fast behavior for active SecretRefs.

## Reproduction Assessment

Yes. The report gives a concrete config, command, trace boundary, version range, and removal workaround, and current source inspection shows the matching pre-bind path through runtime web-tool discovery.

## Solution Assessment

Unclear. The linked PR direction is plausible for the `enabled:false` escape hatch, but the best maintainable fix should be validated against both disabled and enabled present-key configs and should avoid weakening active credential/SecretRef checks.
