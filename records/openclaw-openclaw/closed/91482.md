## Summary

Keep this open: current main still ships the same non-optional file-transfer tool exposure, and `dir_list` still requires a paired node after the model has already selected the tool. The shipped v2026.6.1 source has the same behavior, and I found no merged or open canonical fix in the provided context or local searches.

## What This Changes

Make bundled file-transfer `dir_list` unavailable or self-explanatory when no paired nodes exist, so agents do not mistake it for local directory listing or guess node names.

## Best Possible Solution

Land a plugin-owned repair that gives `dir_list` a clear no-paired-nodes path and stronger node-parameter wording, hiding it dynamically only if the existing plugin tool resolver can do that without adding a new config surface.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction path: with no paired nodes, the default file-transfer plugin can still expose `dir_list`, and execution then returns `node required` or `unknown node` for guessed values. I did not run a live Windows/Ollama session.

## Solution Assessment

Yes, a narrow plugin-owned fix is the best layer: improve the file-transfer descriptor and no-node execution behavior without changing local filesystem tools or adding config. Dynamic hiding is acceptable only if it fits the existing plugin tool discovery contract.
