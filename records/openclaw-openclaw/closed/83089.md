## Summary

Keep this open for maintainer product/API review rather than treating it as a narrow bug fix. Current main and the latest shipped release intentionally skip plugin tools whose names collide with already-available tools, and the public plugin docs tell authors to choose unique names; supporting override, merge, or automatic namespacing for `memory_search` would change the plugin tool-name contract and intersects with the broader third-party memory discussion at https://github.com/openclaw/openclaw/issues/82977.

## What This Changes

Requests that OpenClaw allow or otherwise handle an OpenViking plugin registering `memory_search` despite an existing memory-core tool with the same name.

## Best Possible Solution

Keep the global unique tool-name contract unless maintainers intentionally design a generic memory-provider or tool-shadowing API; OpenViking should use unique/prefixed tool names or the memory slot instead of overriding memory-core by name.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: with `memory-core` exposing `memory_search`, a plugin tool registering the same normalized name is skipped by `resolvePluginTools`. I did not run a live OpenViking install.

## Solution Assessment

No for an automated bug fix: changing this to override, merge, or namespace tool names would alter the public plugin API and memory ownership model. A maintainer should first decide whether OpenClaw wants a generic memory-provider contract or whether external plugins must keep unique tool names.
