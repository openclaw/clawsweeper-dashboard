## Label Justifications

- `P2`: The issue described bounded user-visible channel reply loss after gateway restart, with no security, crash-loop, or runtime-wide outage signal.
- `impact:message-loss`: The original failure was a final assistant reply being written to session output without delivery to the originating channel.
- `impact:session-state`: The behavior depends on restart-aborted main-session state and recovered delivery context metadata.

## Summary

Current main and the shipped `v2026.6.1` release implement the requested scoped restart-recovery delivery path, and the reporter’s follow-up independently confirmed the fixed baseline in that release.

## What This Changes

The issue requests optional delivery of resumed final replies after gateway restart recovery when a channel session still has a valid delivery target.

## Best Possible Solution

Keep the shipped scoped delivery-context implementation from https://github.com/openclaw/openclaw/pull/86089; broader final fallback delivery semantics remain tracked separately at https://github.com/openclaw/openclaw/issues/87561.

## Reproduction Assessment

Yes. The pre-fix `v2026.5.28` source path hard-coded `deliver: false`, and the reporter supplied local bncr logs showing delivery resumed when that was temporarily flipped; current main no longer has that path.

## Solution Assessment

Yes. The shipped fix is the safer scoped approach requested in the thread: deliver only when a normalized deliverable context exists, respect send policy, and use best-effort delivery instead of globally forcing `deliver: true`.
