## Risks / Open Questions

- The PR's current callback shape can let the inbound `groupId` participate in the DM allowlist calculation for group-message events. Greptile notes this is not a practical bypass because the event handler consumes `dmAccess` only for non-group messages, but it is an access-control semantic leak worth fixing before merge.
- The linked #53308 report includes a separate `requireMention` default concern that this PR does not address. Merging this PR as-is should not close the broader report unless that part is split or separately resolved.
- Current docs describe Signal `groupAllowFrom` as a group sender allowlist while `groups` is keyed by Signal group id. Maintainers should decide whether to make `groupAllowFrom` accept group ids, route group-id admission through `groups`, or update docs/tests to remove the ambiguity.
