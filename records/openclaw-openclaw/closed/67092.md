## Summary

Current main still preserves text before an orphan closing reasoning tag, so the reported `reasoning </think> final` shape can still leak through shared assistant-visible/history sanitization and the streaming block-reply sanitizer. The two linked fix PRs were closed unmerged, and existing tests still assert the opposite behavior for unpaired close tags.

## Best Possible Solution

Keep this issue open. Land a focused core sanitization fix that treats an orphan closing reasoning tag outside an active reasoning block as a privacy boundary when answer text follows it, while preserving literal/code examples. Apply the behavior consistently in `src/shared/text/reasoning-tags.ts` and the embedded streaming `stripBlockTags` path, then add regression tests for shared text, assistant-visible/history extraction, embedded streaming/block replies, and UI extraction.

## Risks / Open Questions

- A broad rule that drops everything before any literal closing reasoning tag could erase legitimate documentation or code examples; keep the existing inline/fenced code protections and add regression coverage around examples.
- There are at least two sanitizer paths to update: the shared assistant-visible/history sanitizer and the stateful embedded streaming `stripBlockTags` path. Fixing only one would leave inconsistent preview, delivery, or history behavior.
