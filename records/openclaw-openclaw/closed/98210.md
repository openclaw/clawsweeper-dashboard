## Summary

Keep open: this PR is the active fix candidate for the linked iOS Talk SecretRef bug, and current main still returns the source-shaped payload before runtime resolution for authorized `includeSecrets` calls. The patch looks technically correct from source review, but it intentionally changes a credential-bearing Gateway response and still needs explicit maintainer acceptance of the `operator.talk.secrets` trust boundary before merge.

## What This Changes

The PR changes Gateway Talk config handling so authorized callers receive the active runtime-resolved Talk provider API key at `talk.resolved.config.apiKey` while source provider maps remain SecretRef-shaped or redacted.

## Best Possible Solution

Land this narrow Gateway response fix after maintainers accept `operator.talk.secrets` as the native Talk credential-read boundary; if that boundary is rejected, replace it with an explicitly narrower native credential handoff contract.

## Reproduction Assessment

Yes. Source inspection on current main shows `includeSecrets` returns before runtime SecretRef resolution, while iOS requests `includeSecrets: true` and only treats a string `apiKey` as configured; I did not run a physical iOS session in this read-only review.

## Solution Assessment

Yes for the code layer. Gateway owns runtime SecretRef materialization and iOS cannot resolve SecretRef objects; the remaining merge decision is whether maintainers accept the existing `operator.talk.secrets` boundary for returning the active provider credential.
