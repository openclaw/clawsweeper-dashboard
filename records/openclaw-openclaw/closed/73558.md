## Summary

Close as outside this repository: the reported hardcoding is in the external `kimi-claw` connector, while current OpenClaw main has the generic routing/session machinery and plugin SDK surfaces needed for channel plugins to honor `bindings` and `session.dmScope`. The bundled Kimi code in this repo is a model provider plugin, not the bridge/channel named in the report.

## What This Changes

The issue asks for `kimi-claw` bridge-token inbound messages to use OpenClaw `bindings` and `session.dmScope` routing instead of hardcoded `agent:main:main` dispatch.

## Best Possible Solution

The best path is to fix the upstream `kimi-claw` connector so bridge-token inbound events enter OpenClaw as a proper channel/plugin turn, or at least call the public routing/inbound SDK helpers before dispatch. If that upstream work exposes a missing generic SDK capability, the follow-up should be a narrow OpenClaw plugin-API issue or PR rather than a core special case for one connector.

## Reproduction Assessment

No high-confidence in-repo reproduction exists because current main does not contain the `kimi-claw` bridge/channel implementation. The report gives credible external reproduction steps and source snippets, but those reproduce in the third-party connector package, not this checkout.

## Solution Assessment

No, a core-side special case for `kimi-claw` would not be the best fix. OpenClaw already has generic `bindings`/`dmScope` routing and public plugin SDK hooks; the maintainable solution is for the connector to use those surfaces instead of hardcoded `agentId` and `sessionKey` values.
