## Summary

Current main already addresses the QQ Bot upgrade failure: the externalized QQ Bot target is now the official `@openclaw/qqbot`, the current QQ Bot source no longer contains the child-process scanner trigger described in the v2026.5.2 logs, and post-update plugin sync failures now fail loudly before restart rather than silently leaving users with an inactive gateway.

## What This Changes

The issue requests that upgrades from bundled QQ Bot to an external plugin either restore QQ Bot automatically without false security-scan blocking or show a clear recovery path.

## Best Possible Solution

Keep the current scanner-clean official QQ Bot external package path and let the next release carry it; leave Brave-specific externalization fallout tracked separately in #76373.

## Reproduction Assessment

No, not against current main. The report’s v2026.5.2 scanner triggers no longer match current QQ Bot source, and source inspection found no QQ Bot child_process execution trigger to reproduce the same install block.

## Solution Assessment

Yes. Removing the false-positive subprocess path and failing post-update plugin sync loudly is narrower and safer than adding a broad auto-accept security-warning flag for upgraded plugins.
