## Summary

Current main and the shipped v2026.5.2 source already gate the macOS LaunchAgent restart path on post-restart health, so the reported stopped/free Gateway state now exits as a restart failure instead of a successful restart.

## What This Changes

Report that `openclaw gateway restart` can print a successful LaunchAgent restart while launchd is not running the Gateway and the RPC endpoint is unreachable.

## Best Possible Solution

Keep the shipped restart health gate; if a current release still exits zero in this stopped/free state, reopen with `openclaw gateway restart --json`, `openclaw gateway status --deep --json`, and the restart log so the remaining edge can be isolated.

## Reproduction Assessment

No. Source and shipped regression coverage show the reported stopped/free post-restart state now follows the failure path rather than the final success path.

## Solution Assessment

Yes. The current implementation is the narrow maintainable fix: restart through the service manager, then verify launchd runtime and Gateway port/reachability before reporting final success.
