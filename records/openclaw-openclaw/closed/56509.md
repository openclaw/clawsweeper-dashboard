Full review comments:

- **[P1] Rewrite the matched toolResult ID before pairing:** `src/agents/session-transcript-repair.ts:449`
  - body: When `rawId` only matches through the new sanitized-ID fallback, the PR stores the result under the original tool-call ID but keeps the `toolResult` object unchanged. That can emit an assistant tool call like `call_1` paired with a result that still carries `call1`, which strict replay paths can reject instead of repairing.
  - confidence: 0.88
- **[P2] Keep stale append read receipts on the safe path:** `extensions/whatsapp/src/inbound/monitor.ts:469-483`
  - body: This early `append` skip runs before `normalizeInboundMessage`, so stale append messages no longer build a read-receipt target or call `maybeMarkInboundAsRead`. Existing WhatsApp behavior covers stale append history by marking it read while skipping delivery; dropping that path can leave replayed messages unread and eligible to resurface.
  - confidence: 0.79
- **[P3] Fix the startup retry attempt denominator:** `src/gateway/server.impl.ts:470-471`
  - body: `STARTUP_MAX_RETRIES` is the retry count, but the warning logs attempts as `1/3`, `2/3`, `3/3` while a fourth total attempt still follows. Label the field as retry count or use `STARTUP_MAX_RETRIES + 1` for total attempts so operators can interpret the startup logs correctly.
  - confidence: 0.76
