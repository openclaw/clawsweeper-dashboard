## Summary

Keep open. Current main still has static TTS provider/model/voice resolution and STT language hints, but no language-routing config, no detected-language metadata on transcription outputs, and no route resolver that can switch TTS synthesis by input or reply language.

## What This Changes

The issue requests optional language-aware TTS routing so inbound voice replies can select provider, model, and voice from detected input or reply language while preserving current fixed-voice behavior as fallback.

## Best Possible Solution

Define a backward-compatible generic contract that carries optional STT/input/reply language metadata into TTS route resolution, documents fallback precedence, and composes with existing global, agent, channel, account, preference, persona, and directive overrides.

## Reproduction Assessment

Yes, at source level. With auto-TTS in inbound mode, current main gates on inboundAudio and then resolves provider/model/voice from static config and overrides without an input-language, reply-language, or detected STT language signal.

## Solution Assessment

Unclear for the exact proposed JSON shape. The maintainable direction is a generic STT metadata plus TTS route-resolution contract, not a provider-specific workaround or prompt-only convention.
