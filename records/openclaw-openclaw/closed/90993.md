## Summary

Keep open: current `main` still lacks an OpenClaw-owned hard lifetime bound for native hook relay CLI processes, and the report provides credible field evidence of leaked `openclaw-hooks` processes causing OOM. The exact stdin hypothesis is not fully proven against current upstream Codex, but that uncertainty argues for a focused fix and validation, not cleanup closure.

## What This Changes

The issue reports that `openclaw hooks relay` child processes can remain alive indefinitely and accumulate enough resident memory to trigger host OOM.

## Best Possible Solution

Add an OpenClaw-owned hard lifetime bound for each relay CLI invocation so process cleanup does not depend solely on the spawning harness.

## Reproduction Assessment

No high-confidence live reproduction in this review. Source inspection shows a reproducible failure shape when relay stdin never reaches EOF and confirms the relay command lacks an OpenClaw-owned hard process deadline.

## Solution Assessment

Yes, a narrow process-lifetime guard is the best fix direction. Preserve relay policy and fallback output semantics while bounding stdin read, bridge/gateway execution, output flushing, and final shutdown.
