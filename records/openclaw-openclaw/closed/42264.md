## Summary

Current main already decouples the probe run timeout from post-run idle-flush cleanup. The run timeout is cleared before teardown starts, and the later idle wait/flush path is best-effort cleanup rather than part of the probe timeout budget. The same structure is present in the shipped `v2026.4.22` tag, so this request is already implemented well enough to close.

## Close Comment

Closing after Codex review. Current `main` already clears the run timeout before the post-run idle-flush teardown starts: `src/agents/pi-embedded-runner/run/attempt.ts:2597` clears `abortTimer`, and the later cleanup at `src/agents/pi-embedded-runner/run/attempt.ts:2817` only does best-effort teardown through `src/agents/pi-embedded-runner/run/attempt.subscription-cleanup.ts:39` and `src/agents/pi-embedded-runner/wait-for-idle-before-flush.ts:49`. That means cleanup can still add wall-clock latency, but it does not consume the probe timeout budget or convert an already-successful reply into `request timed out`. I also verified the same structure is present in the shipped `v2026.4.22` release (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). If you still see probe timeouts on a newer build, please open a fresh issue with exact provider/model, command, logs, and transcript timing, because the March 10, 2026 diagnosis here no longer matches current main.
