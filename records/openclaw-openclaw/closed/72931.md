## Summary

Current main and the shipped v2026.4.27 release already address the reported bootstrap path: the bundled Control UI is Vite/Lit rather than Angular/Zone.js, the gateway CSP permits the required font endpoints and hashes inline index scripts without enabling unsafe inline script execution, and invalid config reload/startup recovery is covered by current source and tests. The remaining lifecycle guidance is documented as using `openclaw gateway restart` rather than chaining stop/start as a restart substitute.

## What This Changes

The issue asks for the bundled Control UI to boot cleanly under the gateway CSP and for gateway service restart/start-stop behavior to avoid terminal-flashing loops after Windows npm installs.

## Best Possible Solution

Keep the shipped v2026.4.27 behavior: the Control UI should remain Zone-free, CSP changes should stay explicit and covered by HTTP tests, and operators should use the documented managed-service restart path for lifecycle recovery.

## Reproduction Assessment

Yes. The issue gives concrete Windows/npm steps and browser errors; current-main inspection targets that same UI bootstrap/CSP path plus the invalid-config recovery and gateway lifecycle surfaces implicated by the logs.

## Solution Assessment

Yes. The current approach is the maintainable fix: remove the Angular/Zone expectation, hash any inline index scripts instead of enabling broad `unsafe-inline`, keep font allowances narrow, and recover invalid config through last-known-good rather than exposing a user-configurable CSP escape hatch.
