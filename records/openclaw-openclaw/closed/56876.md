## Summary

Close as duplicate. The same remaining read-tool problem is already tracked by the older open #35406, and current main still lacks a general binary-document guard or extraction path for `read`; the open PRs #57600 and #57612 are implementation candidates against that canonical thread.

## What This Changes

Request file type detection and preprocessing so binary documents read through `read` are extracted or replaced with a safe placeholder instead of raw binary text.

## Best Possible Solution

Keep #35406 as the canonical thread for the remaining read-tool binary document behavior, and review #57600/#57612 or an equivalent maintainer-approved patch there. The desired end state is a small, documented read-path guard first, with parser-backed extraction added only where the file format and security boundary are clear.

## Reproduction Assessment

Yes. The code path is high-confidence: current `read` delegates to the upstream read tool, and that dependency converts any non-image file buffer to UTF-8 text, so a `.docx`/`.xlsx`/`.pptx` read can still produce binary-looking text.

## Solution Assessment

Yes. Closing this duplicate is the best cleanup action because #35406 predates it, describes the same `read` behavior in more exact terms, and already has open implementation candidates; closing this issue should not imply the feature is implemented.
