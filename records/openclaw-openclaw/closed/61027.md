## Summary

Current `main` already fixes the reported Gemini 2.5 Pro failure by sanitizing Google thinking payloads before transport, so omitting `thinking` in `llm-task` no longer needs to auto-default to avoid `thinkingBudget: 0`.

## Evidence

- **`llm-task` still forwards omitted thinking as undefined:** `llm-task` leaves `thinking` unset when omitted and passes `thinkLevel: resolvedThinkLevel` into the embedded runner, so the fix path is downstream rather than a plugin-level default.
  - file: [extensions/llm-task/src/llm-task-tool.ts:138](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/extensions/llm-task/src/llm-task-tool.ts#L138)

  - sha: [59523e66da03](https://github.com/openclaw/openclaw/commit/59523e66da03591bc7bcc183a11f690fa0aae641)
- **Embedded runner always applies Google payload sanitization:** Post-plugin stream wrapping adds `createGoogleThinkingPayloadWrapper(...)` when no provider-specific wrapper already handled the request, covering `llm-task` embedded runs before the Google request is sent.
  - file: [src/agents/pi-embedded-runner/extra-params.ts:439](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/src/agents/pi-embedded-runner/extra-params.ts#L439)

## Risks / Open Questions

- `llm-task` still does not auto-select a thinking preset when `thinking` is omitted; the shipped fix is a Google/Gemini transport compatibility guard. This closes the reported Gemini 2.5 Pro failure mode, not a broader design request for all reasoning-capable models.
