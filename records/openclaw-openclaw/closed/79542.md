## Summary

Keep open: the branch still applies Copilot state-directory diagnostics to every ACP session even though current ACP is multi-harness, and the external PR still lacks observed after-fix real behavior proof.

## What This Changes

The PR adds an `openclaw sessions link` CLI command, tests, ACPx docs, and a changelog entry for listing ACP session keys, `acpxSessionId` values, Copilot state paths, and status flags.

## Best Possible Solution

Make the command scope explicit: filter to Copilot ACP sessions or add backend-aware state-path/status resolution for all ACP harnesses, with non-Copilot regression coverage and redacted after-fix runtime proof.

## Reproduction Assessment

Yes for the review finding: source inspection shows every `entry.acp` row gets a Copilot state path, while current docs list non-Copilot ACP harnesses. The original feature need has before-only evidence, not a patched command run.

## Solution Assessment

No. A generic `sessions link` command should not report Copilot state paths for Claude, Codex, Gemini, Kimi, or other ACP rows; Copilot-scoped filtering or backend-aware resolution is safer.
