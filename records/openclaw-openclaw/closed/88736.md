## Label Justifications

- `P1`: The report describes a current update path that can repeatedly stop the managed gateway and break real channel/agent workflows after update.
- `impact:crash-loop`: The affected launchd updater loop repeatedly stops the gateway, making the managed runtime unable to stay available.
- `impact:message-loss`: When the gateway repeatedly drops, connected channels such as Discord, Telegram, and cron can miss or interrupt message handling.

## Summary

Keep open: the reported macOS runaway updater loop is still plausible against current main because launchd updater cleanup only recognizes `ai.openclaw.update.*`, while the report shows `ai.openclaw.manual-update.*`; a repeated package update still stops the managed gateway before replacement.

## What This Changes

The issue asks for macOS update handoffs to run once instead of relaunching under launchd and repeatedly stopping the managed gateway after an update.

## Best Possible Solution

Recognize and disarm observed one-shot updater launchd labels such as `ai.openclaw.manual-update.*` before stopping the gateway, and surface them in doctor/status cleanup while preserving the existing package-refresh behavior unless maintainers choose a broader policy change.

## Solution Assessment

Yes directionally: the narrow maintainable fix is to disarm the one-shot updater job before the package-update stop step and cover the manual-update label in tests. Changing the already-tested package refresh behavior when no newer version is available should be treated as separate policy work.
