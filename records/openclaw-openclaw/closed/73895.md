## Summary

Keep open: current main still exposes only the existing usage/resource fields on the OpenAI-compatible, OpenResponses, and WS chat surfaces; the requested routing/config lineage and timing metadata is not implemented and would expand a public protocol surface with security impact.

## What This Changes

Request Gateway v2 response metadata for routing/config lineage, fallback/policy decisions, timings, gateway version, decision trace, and reasoning-token usage so a benchmark smoke harness can validate live output.

## Best Possible Solution

Define an additive, typed, redacted Gateway metadata envelope at the protocol and HTTP boundaries, document its compatibility and privacy contract, then wire it through OpenAI-compatible, OpenResponses, and WS chat response paths with focused regression coverage.

## Reproduction Assessment

Not applicable as a runtime bug reproduction; this is an instrumentation/protocol feature request. Source inspection of current main confirms the relevant HTTP and WS response schemas do not expose the requested fields.

## Solution Assessment

Unclear because no patch exists yet. Emitting from the routing layer is plausible, but the maintainable path needs Gateway/protocol owner approval for schema shape, redaction, versioning, and reasoning-token semantics before implementation.
