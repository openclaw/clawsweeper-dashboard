## Summary

This PR adds a generic local CLI TTS wrapper as a new bundled plugin, but current OpenClaw scope docs and plugin docs explicitly steer optional capabilities like this to external plugins/ClawHub instead of expanding the core bundle set. Main already exposes the speech-provider SDK and install/discovery paths needed to ship this outside the repo.

## Close Comment

Closing as `clawhub` after a Codex review of current `main`. The main repo now documents this kind of optional capability as an external plugin path rather than a new bundled core plugin: `VISION.md:54-61` says optional capability should usually ship as plugins, the preferred path is an npm/local extension in the author’s own repo, and the bar for adding optional plugins to core is intentionally high. The speech-provider SDK surface already exists for exactly this (`docs/plugins/sdk-provider-plugins.md:483-507`, `docs/plugins/manifest.md:388-418`, `src/plugin-sdk/speech-core.ts:1-40`), and OpenClaw already supports installing plugins from npm or a local directory/archive (`docs/tools/plugin.md:11-33`). Current main keeps bundled speech providers curated (`docs/tools/plugin.md:98-116`, `docs/tools/tts.md:10-20`) and does not include `tts-local-cli` on main. Because this PR is a generic local CLI wrapper around user-supplied command/args/cwd/env, it fits the community plugin / ClawHub route better than expanding the core bundled set. If you want to keep pursuing it, the recommended path is to publish it as an external OpenClaw plugin on ClawHub or npm.
