## Summary

Keep open: current `main` still has the source-before-built package-state probe path when a bundled channel catalog entry is rooted at the source plugin, and the only direct fix PR was closed unmerged after review found it did not carry the correct loader boundary.

## What This Changes

Bundled channel package-state probes should avoid loading source `auth-presence.ts` before built auth-presence artifacts in dev checkouts when that produces noisy startup warnings.

## Best Possible Solution

Teach package-state probes to use the matching built bundled artifact with a matching built root/boundary when it exists, while preserving a source-loader fallback for missing or stale dist output.

## Reproduction Assessment

Yes, source-backed: current `main` still resolves extensionless package-state probes under the entry root with `.ts` before `.js`, and the issue includes Matrix and WhatsApp logs from that path. I did not run the live gateway startup path in this checkout.

## Solution Assessment

No, current `main` is not the best end state because it can still enter the source-first path for bundled auth probes. The prior PR's direction was close, but the fix must carry the matching built root or boundary so `loadChannelPluginModule` can accept the built artifact.
