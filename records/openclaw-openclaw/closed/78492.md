## Summary

Keep open: current main and the latest release still spawn the auto-update child with only `OPENCLAW_AUTO_UPDATE`, while the child env resolver inherits the gateway service markers that the update guard treats as an in-service package update. The linked fix PR is closed unmerged, and the remaining repair is narrow but still needed.

## What This Changes

The issue reports that the gateway auto-update child inherits gateway service markers and is blocked from running package updates.

## Best Possible Solution

Scrub the gateway service marker, service kind, and gateway runtime PID from the auto-update child environment before spawning the CLI update path, while preserving the guard for user-run updates inside a live gateway process.

## Reproduction Assessment

Yes for a source-level reproduction: a managed gateway sets the service markers, `runAutoUpdateCommand` does not remove them, and the update command still treats that env as a gateway-service package update. I did not run a live launchd auto-update in this read-only review.

## Solution Assessment

The requested direction is right, but the issue body's suggested `undefined` override is not sufficient against current `mergeChildEnv` semantics because `undefined` does not delete a copied base env key. The better repair is an explicit scrubbed base env or a tested deletion contract in the spawn env resolver.

## Repair Work Prompt

Fix the managed-gateway auto-update env leak reported in https://github.com/openclaw/openclaw/issues/78492. In `src/infra/update-startup.ts`, ensure the child spawned by `runAutoUpdateCommand` cannot inherit `OPENCLAW_SERVICE_MARKER`, `OPENCLAW_SERVICE_KIND`, or `OPENCLAW_GATEWAY_SERVICE_PID` from the gateway service process while preserving `OPENCLAW_AUTO_UPDATE=1`. Do not weaken the package-update guard for operator-run updates inside a live gateway service. Be careful: in current `src/process/exec.ts`, setting an override key to `undefined` does not delete a value already copied from `process.env`, so either pass a scrubbed base env or deliberately change and test the env deletion contract. Add focused regression coverage that sets the gateway service markers in `process.env` and proves the actual spawned/resolved child env omits them; keep changelog untouched.
