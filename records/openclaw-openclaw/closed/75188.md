## Summary

Current main and the v2026.4.27 release source already build the expected Codex Responses image-generation request: the body includes a `tools` array with an `image_generation` tool whenever it sets `tool_choice: { type: "image_generation" }`. The reported malformed body does not match the shipped provider code, regression tests, JSON transport helper, or the upstream OpenAI Responses tool contract.

## What This Changes

The issue reports that `openai/gpt-image-2` image generation through an `openai-codex` OAuth profile sends `tool_choice: "image_generation"` without a matching `image_generation` tool definition.

## Best Possible Solution

Keep the shipped plugin-owned OpenAI image provider behavior. If someone still sees this after v2026.4.27, the useful follow-up is a new report with a redacted captured request body or gateway logs around `image auth selected`, because the source path already sends the expected `tools` entry.

## Reproduction Assessment

No high-confidence live reproduction is available here: this read-only review found no OpenAI/Codex credentials in the environment or `~/.profile`. The source-level request-body path and regression tests show the reported missing-tools request is not produced by current main or the v2026.4.27 release source.

## Solution Assessment

Yes. The narrow maintainable solution is the current OpenAI plugin route that constructs a Codex Responses request with the image-generation tool definition and forced tool choice; no core migration or new plugin API is needed.
