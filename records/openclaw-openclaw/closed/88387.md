## Summary

Keep open: the Matrix state refactor is useful, but the patch still removes the shipped startup-verification.json deviceId fallback before preserving that metadata for old token roots, so upgraded Matrix installs can stay on a fresh root and lose existing crypto/thread-binding state. The PR also still has tests-only proof and explicitly has not been exercised against live Matrix homeserver traffic.

## What This Changes

The PR moves Matrix inbound dedupe, startup verification, and thread binding persistence from plugin-local JSON files into Matrix plugin-state SQLite with one-shot legacy imports and contract-test runtime support.

## Best Possible Solution

Preserve or migrate legacy startup-verification device metadata before storage-root selection, add focused upgrade regression coverage, then provide redacted live Matrix startup/token-rotation proof.

## Reproduction Assessment

Yes for the patch-level bug by source inspection: seed an old Matrix token root whose only device metadata is startup-verification.json, then resolve paths after token rotation with the same deviceId. Current main and v2026.5.27 use that file as a fallback; the PR removes it before a pre-routing migration.

## Solution Assessment

No. Moving ephemeral Matrix state into plugin-state SQLite is a reasonable direction, but the patch is not the safest solution until legacy startup device metadata is preserved before storage-root routing.
