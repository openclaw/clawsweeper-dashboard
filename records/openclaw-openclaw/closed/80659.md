## Summary

Keep open: the reported timeout ghost-turn bug is plausible from source inspection, but this PR is not mergeable as written because it removes existing timeout-delivery safeguards, can replay partial assistant fragments, carries broad unrelated release/workflow/app/plugin/UI changes, and still lacks attached after-fix real behavior proof.

## What This Changes

The branch changes the embedded Pi runner to always append a timeout error on prompt-phase timeout and, in the current PR diff, also carries broad unrelated release, workflow, docs, app, plugin, and UI changes.

## Best Possible Solution

Narrow the branch to the embedded runner and focused tests, append a timeout error after successful non-messaging tool/media payloads, preserve messaging-tool delivery and partial-fragment safeguards, and attach redacted after-fix proof.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main can skip the timeout error when timedOutDuringPrompt is true and non-messaging tool/media payloads make payloadsWithToolMedia non-empty. I did not run a live 120-second provider timeout.

## Solution Assessment

No, this patch is not the best way to solve the issue as written. The maintainable fix should preserve the messaging-tool delivery guard, suppress partial assistant fragments, and add focused regression coverage for successful tool/media payloads followed by prompt idle timeout.
