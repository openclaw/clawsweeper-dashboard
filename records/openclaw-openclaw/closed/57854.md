## Summary

Current `main` already treats transient `fetch failed` unhandled rejections as non-fatal, and the latest release `v2026.4.23` contains the same code paths for the gateway/Telegram/Discord handling relevant to this report.

## Evidence

- **Global unhandled-rejection handler no longer exits on transient fetch/network failures:** `src/infra/unhandled-rejections.ts:123` recognizes bare and wrapped `fetch failed` messages, and `src/infra/unhandled-rejections.ts:373` logs `Non-fatal unhandled rejection (continuing)` instead of calling `process.exit(1)` for transient network errors.
  - file: [src/infra/unhandled-rejections.ts:123](https://github.com/openclaw/openclaw/blob/9b18ae877018a34933e27bb7b2986cb97e9602aa/src/infra/unhandled-rejections.ts#L123)
  - sha: [9b18ae877018](https://github.com/openclaw/openclaw/commit/9b18ae877018a34933e27bb7b2986cb97e9602aa)
- **Regression tests cover both plain and wrapped fetch-failed rejections:** `src/infra/unhandled-rejections.fatal-detection.test.ts:117` asserts no exit for `TypeError("fetch failed")` and the wrapped Discord error `Failed to get gateway information from Discord: fetch failed`.
  - file: [src/infra/unhandled-rejections.fatal-detection.test.ts:117](https://github.com/openclaw/openclaw/blob/9b18ae877018a34933e27bb7b2986cb97e9602aa/src/infra/unhandled-rejections.fatal-detection.test.ts#L117)
  - sha: [9b18ae877018](https://github.com/openclaw/openclaw/commit/9b18ae877018a34933e27bb7b2986cb97e9602aa)
- **Telegram polling path suppresses and restarts after unhandled network rejections:** `extensions/telegram/src/monitor.ts:91` registers a handler that marks the polling transport dirty, aborts the active fetch, and restarts polling for recoverable Telegram polling errors. `extensions/telegram/src/monitor.test.ts:601` verifies the transport is rebuilt after an unhandled polling network rejection.
  - file: [extensions/telegram/src/monitor.ts:91](https://github.com/openclaw/openclaw/blob/9b18ae877018a34933e27bb7b2986cb97e9602aa/extensions/telegram/src/monitor.ts#L91)
  - sha: [9b18ae877018](https://github.com/openclaw/openclaw/commit/9b18ae877018a34933e27bb7b2986cb97e9602aa)
- **Discord gateway metadata fetch failures fall back instead of crashing:** `extensions/discord/src/monitor/gateway-plugin.ts:76` wraps transient `/gateway/bot` failures as `fetch failed`; `extensions/discord/src/monitor/gateway-plugin.ts:215` falls back to the default gateway URL; `extensions/discord/src/monitor/gateway-plugin.ts:274` uses that fallback during client registration. Tests at `extensions/discord/src/monitor/provider.proxy.test.ts:297` and `:416` cover transient 503 and body-read failures.
  - file: [extensions/discord/src/monitor/gateway-plugin.ts:76](https://github.com/openclaw/openclaw/blob/9b18ae877018a34933e27bb7b2986cb97e9602aa/extensions/discord/src/monitor/gateway-plugin.ts#L76)

## Risks / Open Questions

- This closes the crash report only; session recovery after a real process restart is a separate concern from #57851.
- Logging for generic transient fetch failures is still concise, so exact endpoint attribution may remain limited outside channel-specific handlers.
