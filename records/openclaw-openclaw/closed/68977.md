## Summary

Keep open. Current main still exposes `enqueueSystemEvent` as an ephemeral in-memory boolean-returning queue with only consecutive-text duplicate suppression; the durable idempotency primitives nearby do not provide the requested scoped public system-event contract.

## What This Changes

Request scoped durable idempotency keys for `enqueueSystemEvent` so plugin resume flows can retry without enqueueing duplicate system events.

## Best Possible Solution

Define an additive plugin/core contract for durable scoped system-event idempotency, including persistence, cleanup, duplicate result shape, payload mismatch behavior, and tests across sessions and restarts.

## Reproduction Assessment

Yes. Source inspection on current main is enough to reproduce the missing API and persistence contract; the crash window follows from the documented in-memory, non-persistent queue rather than from a live crash test.

## Solution Assessment

Unclear. Scoped durable idempotency is a plausible direction, but the maintainable fix needs maintainer API design to avoid breaking existing boolean callers and to choose between extending `enqueueSystemEvent` or adding a separate durable helper.
