## Summary

Keep this open: current main still has the Docker host-path leak by source inspection, and open PR #77446 already references this issue with closing syntax, so the issue should stay open until that PR lands or is closed.

## What This Changes

The issue reports a Docker setup regression after 2026.04.29 where a fresh macOS Docker install starts the gateway but the first agent reply fails with `EACCES: permission denied, mkdir '/Users'`.

## Best Possible Solution

Review and land or close PR #77446, which should keep host bind-mount sources as host paths while pinning container-visible runtime config and workspace paths to `/home/node/.openclaw`.

## Reproduction Assessment

Yes. By source inspection, macOS Docker setup can write `/Users/...` paths into `.env`, the container imports those paths during onboarding, and the first reply path can try to create that host path inside Linux Docker.

## Solution Assessment

Yes. The narrowest maintainable fix is at the Docker Compose/setup boundary: keep host paths for volume interpolation, but override in-container `OPENCLAW_CONFIG_DIR` and `OPENCLAW_WORKSPACE_DIR` so runtime code uses `/home/node` paths.
