## Summary

Current main already covers the session-state loss problem: reset paths emit a transcript-backed `before_reset` hook, `/new` and `/reset` can persist recent context through the bundled session-memory hook, and token-pressure cases use default-on pre-compaction memory flush rather than a silent reset.

## What This Changes

Requested a Gateway/session hook that lets an agent summarize or persist important context before an automatic reset clears conversation state.

## Best Possible Solution

Keep the shipped reset hooks and default memory-flush path; a future request should only reopen this area if it shows a concrete gap that existing hooks, session-memory, or compaction memory flush cannot cover.

## Reproduction Assessment

Not applicable; this is a feature request, not a broken existing contract. The current-main proof is source/docs/tests showing reset and compaction memory-preservation paths.

## Solution Assessment

Yes; the existing hook and memory-flush design is the narrow maintainable solution, and it avoids adding a second `PRE_RESET` special-message API unless a future report proves the current surfaces are insufficient.
