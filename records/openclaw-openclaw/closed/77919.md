## Summary

Keep open: current main already forwards `providerOptions.deepgram.utterances`, but the issue’s stated value is native utterance data with timestamps/confidence, and the current audio transcription contract only returns text/model.

## What This Changes

Add Deepgram `utterances` support for audio transcription alongside the existing Deepgram query option handling.

## Best Possible Solution

Clarify the intended native surface: keep using `providerOptions.deepgram.utterances` for the query flag, and add an additive audio transcription metadata contract only if OpenClaw should expose returned utterance/word timing data.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that the query flag already forwards, while structured utterance metadata is not represented in the current audio transcription result path.

## Solution Assessment

No, the proposed two-line query change is not the best fix because the generic pass-through already handles `utterances`; a real native feature would need docs and likely an additive media-output metadata design.
