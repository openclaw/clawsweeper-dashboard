## Label Justifications

- `P1`: The report describes a fresh production-style outage where stale OpenAI fallback credentials stopped Slack and Discord agent responses across all models.
- `impact:auth-provider`: The core problem is stale OpenAI API-key routing and Codex auth source precedence during key rotation.
- `impact:message-loss`: The described failure mode suppresses user-visible agent replies in Slack and Discord when all model attempts exhaust auth fallback.

## Summary

Keep open: current main and the latest release still let an existing per-agent Codex app-server account take precedence over the current `OPENAI_API_KEY` fallback, so the reported key-rotation failure mode is not fixed. OpenClaw auth profiles already provide a better central credential path, but there is no implemented doctor/migration or precedence change for stale per-agent Codex API-key accounts.

## What This Changes

The issue requests centralized OpenAI key rotation for Codex-backed subagents so per-agent `codex-home/auth.json` files do not retain stale API keys and doctor can detect or repair stale copies.

## Best Possible Solution

Add a maintainer-approved Codex auth migration/doctor path that moves API-key fallback into OpenClaw-managed auth profiles or safely refreshes only API-key app-server accounts from the current credential source, while preserving intentional ChatGPT account isolation.

## Reproduction Assessment

Yes by source inspection. With no explicit auth profile, OpenClaw can read `CODEX_API_KEY` or `OPENAI_API_KEY`, but it skips app-server login whenever `account/read` returns any account, and Codex persists API-key accounts in `CODEX_HOME/auth.json`.

## Solution Assessment

Unclear. The proposed `$OPENAI_API_KEY` or `op://` support inside Codex `auth.json` would change the Codex storage contract, while an OpenClaw-owned auth-profile/doctor migration may be a narrower and safer fix.
