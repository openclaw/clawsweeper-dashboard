## Summary

Current main already implements the requested durable fix: `chokidar` is declared in the root runtime dependency set, mirrored into `openclaw.bundle.mirroredRootRuntimeDependencies`, folded into bundled runtime-deps planning, and written into the generated runtime-deps install manifest. This overlaps with the earlier related #74688 cleanup; the fix is on current main only, because the `v2026.4.27` tag still lacks the mirror metadata.

## What This Changes

The issue asks OpenClaw to declare `chokidar` in generated plugin-runtime-deps for releases that ship qmd-manager so gateway startup does not depend on repeated auto-mitigation.

## Best Possible Solution

Keep the current explicit runtime-deps mirror metadata and carry it into the next release so fresh generated plugin-runtime-deps installs include `chokidar` deterministically.

## Reproduction Assessment

Yes. The reported reproduction is high-confidence for `v2026.4.27`: qmd-manager imports `chokidar`, while that tag's root package lacks `openclaw.bundle.mirroredRootRuntimeDependencies`; current main no longer matches that broken metadata state.

## Solution Assessment

Yes. The current-main fix is the narrow maintainable solution because bundled package-level runtime deps are intentionally explicit metadata, and the planner/materializer already consumes that metadata into the generated runtime-deps package.
