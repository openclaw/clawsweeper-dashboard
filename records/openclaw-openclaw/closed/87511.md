## Label Justifications

- `P1`: The issue blocks multi-turn Codex MCP delegation for affected users because the required thread identifier is dropped from model-visible tool output.
- `impact:session-state`: The lost `threadId` is the continuation state needed to associate later `codex-reply` calls with the original Codex thread.

## Summary

Keep open. Current main and the latest release still serialize inbound MCP `structuredContent` only when `content[]` is empty, so a Codex `threadId` returned alongside normal text remains hidden from model-facing tool content.

## What This Changes

The issue asks OpenClaw to expose inbound bundle-MCP `structuredContent` to the agent even when the MCP result also includes `content[]`, so Codex `codex-reply` can continue a thread.

## Best Possible Solution

Fix the bundle-MCP to `AgentToolResult` boundary so present `structuredContent` is visible in model-facing text content, with focused materialization coverage and a real Codex MCP smoke when feasible.

## Reproduction Assessment

Yes at source level. A `CallToolResult` with both non-empty `content` and `structuredContent` follows the current content branch and never serializes the structured fields into model-facing content; I did not rerun a live Codex MCP session.

## Solution Assessment

Yes. The narrowest maintainable fix is at the bundle-MCP materialization boundary, before provider transports and agent loop consumers see an `AgentToolResult`.
