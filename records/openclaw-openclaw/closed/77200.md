## Summary

Keep open: the PR targets a valid plugin SDK bug, but the current diff has a blocking rollback regression because the catch path would retain memory supplements from a failed plugin registration.

## What This Changes

The PR changes the plugin loader to merge previous/current memory corpus and prompt supplements during restore and adds a changelog entry for #77039.

## Best Possible Solution

Preserve active supplement state on successful non-activating lazy re-registration while keeping failed registration rollback as an exact snapshot restore, with focused tests for both paths.

## Reproduction Assessment

Yes. Source inspection of the PR diff plus the existing failing-register test gives a high-confidence path: a plugin can register memory supplements, throw during register, and the patched catch path would append those failed entries back into global memory state.

## Solution Assessment

No. The preservation merge is appropriate for the successful non-activating restore path, but applying it to the error rollback path breaks the existing rollback contract; the safer fix is success-path merge only plus regression coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Keep failed memory registrations rolled back:** `src/plugins/loader.ts:2382-2395`

If register() adds a memory prompt or corpus supplement and then throws, memoryPluginState still contains that failed plugin's entries, so the filter appends them back instead of restoring the pre-register snapshot. That leaves failed plugins contributing memory state after load reports an error.
