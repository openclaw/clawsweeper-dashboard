## Summary

Close as duplicate/superseded by the open Telegram abort-scoping PR #48537. The new Windows/Docker report adds useful reproduction detail, but the remaining source-level problem is already tracked there: polling restarts abort the shared Telegram fetch signal, which can cancel `sendChatAction` and `sendMessage` even when direct Bot API calls work.

## What This Changes

Make Telegram typing and final reply delivery survive OpenClaw polling/restart network failures when direct Telegram Bot API calls still succeed.

## Best Possible Solution

Use #48537 as the canonical fix path: rebase or replace it against current `bot-core.ts` so polling restarts still abort `getUpdates` but do not abort `sendChatAction` or `sendMessage`, then validate Windows/npm and Docker Telegram delivery.

## Reproduction Assessment

Yes. The issue body gives a concrete live reproduction, and the static path is high-confidence: polling restart aborts the shared controller, current bot-core attaches that signal to every Bot API fetch, and the existing test still expects broad cancellation.

## Solution Assessment

Yes. Closing this duplicate and keeping the current-main repair on #48537 is the narrowest maintainable path; broader durable delivery/retry-queue work belongs in the existing reliability threads, not a second copy of this abort-scoping bug.
