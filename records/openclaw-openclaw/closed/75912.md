## Summary

Close as implemented on current main. The 2026.4.29 release still forced uncached provider hook resolution, but current main adds a provider-runtime cache and routes the cited prep consumers through `resolveProviderRuntimePlugin`, which addresses the reported repeated OpenRouter/provider-runtime lookup latency.

## What This Changes

The issue asks OpenClaw to cache or reuse provider-runtime/plugin hook resolution across embedded-run prep phases so Telegram/OpenRouter runs do not wait 80-120s before `stream-ready`.

## Best Possible Solution

Keep the generic provider-runtime cache on main, ship it in the next release, and leave any remaining plugin-tool factory amortization to the narrower #75290 tracker.

## Reproduction Assessment

Yes. The issue provides concrete 2026.4.29 timing logs and a provider-runtime cache proof of concept, and the release source independently shows the uncached `cache: false` hook path that current main removed.

## Solution Assessment

Yes. The current main solution is the narrow generic fix: cache provider runtime plugin resolution by provider/config/workspace inputs and reuse runtime hooks across prep phases, rather than adding OpenRouter or Kimi-specific fast paths.
