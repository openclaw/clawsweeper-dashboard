## Summary

Current OpenClaw already supports Windows managed Gateway startup without keeping the launching PowerShell open, and the shipped docs now explicitly cover this exact symptom. Foreground `openclaw gateway run` is expected to stop when its shell closes; the supported fix is `openclaw gateway install` plus `openclaw gateway status/restart`.

## Close Comment

Closing as implemented on current OpenClaw. Codex review of `main` found that native Windows already supports running the Gateway without keeping the launching PowerShell open: `openclaw gateway install` installs a managed Windows startup path, using Scheduled Tasks first and falling back to a per-user Startup-folder login item when task creation is denied (`src/daemon/schtasks.ts:846`). That fallback is covered by regression tests in `src/daemon/schtasks.startup-fallback.test.ts:194`, and the user docs now describe the expected flow in `docs/platforms/windows.md:40` and `docs/help/faq.md:2935`. The same implementation is present in the shipped release `v2026.4.22`. In other words, closing the shell is only expected to break a foreground `openclaw gateway run`; the supported background path is already in product.
Codex evidence: Reviewed against main [d3f6783b16ec](https://github.com/openclaw/openclaw/commit/d3f6783b16ecc10c67b101bf019ca4294453206a). Fix evidence: release [v2026.4.22](https://github.com/openclaw/openclaw/releases/tag/v2026.4.22).
