## Summary

Current main already has the enforceable guard surfaces requested here: plugin hooks can intercept outbound delivery, dispatch, agent-run phases, and tool calls, including approval-gated sensitive operations. The remaining declarative `guards` DSL idea would be an optional wrapper over shipped plugin APIs rather than missing core capability.

## What This Changes

Request an outbound/pre-delivery guard mechanism to inspect, block, modify, or require confirmation for agent responses and sensitive tool actions before channel delivery.

## Best Possible Solution

Keep the shipped plugin-hook path and build domain-specific memory or Feishu confirmation policy as a code plugin unless a concrete missing SDK field appears.

## Reproduction Assessment

Not applicable: this is a feature request, not a current-main bug report. Source and release inspection show the requested enforceable hook surfaces already exist.

## Solution Assessment

Yes: plugin hooks are the narrow maintainable solution because they enforce runtime policy without adding a separate core `guards` DSL; a declarative wrapper can live in a plugin if desired.
