## Summary

Keep open: current main and the latest tagged release still flush buffered block-streamed text before a media payload, so Discord receives the caption candidate and attachment as separate deliveries even though the Discord media sender supports caption text with files.

## What This Changes

The issue asks for Discord assistant replies containing text plus a `MEDIA:` attachment in one turn to send as one Discord message with the text as the attachment caption.

## Best Possible Solution

Update the block reply coalescer so compatible buffered visible text is folded into the next media payload as caption text, while preserving status/reasoning boundaries, duplicate-media suppression, and adapter-specific media exceptions.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: enqueueing text chunks followed by a media payload takes the current `hasMedia` branch that flushes buffered text before media. I did not perform a live Discord send in this read-only review.

## Solution Assessment

Yes, the maintainable fix is in the shared block coalescer with Discord adapter regression coverage, not a Discord-only workaround. The repair should only combine buffered text with media when reply metadata boundaries are compatible.
