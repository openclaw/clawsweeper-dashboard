## Summary

Current main already has layered protection for the reported Telegram duplicate injection path: Telegram update/message_id dedupe runs before inbound handling, Telegram message_id is preserved as MessageSid, shared dispatch dedupe rejects duplicate or in-flight MessageSid scopes before the agent turn, and queued follow-ups dedupe by the same message id. The related checkpoint-growth report is closed, and this review found the same-message_id injection problem covered by source and regression tests in the shipped v2026.4.27 tree.

## What This Changes

The issue asks OpenClaw to deliver each Telegram inbound message to an agent session only once by deduplicating repeated Telegram messages with the same message_id.

## Best Possible Solution

Keep the shipped layered dedupe implementation and close this broad same-message_id Telegram injection report; any future recurrence should be opened with a fresh session excerpt and Telegram update/log details so maintainers can identify a distinct path not covered by update tracking or MessageSid dedupe.

## Reproduction Assessment

Yes. The high-confidence reproduction path is to replay the same Telegram update/message_id through the handler or dispatch the same finalized context twice; current tests and source show the second event is skipped before another model turn can be started.

## Solution Assessment

Yes. The current solution is the narrow maintainable shape: Telegram-owned update dedupe handles platform replays, shared dispatch dedupe handles cross-route/in-flight repeats, and queue dedupe handles busy-session follow-ups without adding Telegram-specific policy to core.
