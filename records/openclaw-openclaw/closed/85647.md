## Label Justifications

- `P1`: The report describes an active provider/model regression that breaks real github-copilot Claude tool-using requests and can trigger paid-provider fallback behavior.
- `impact:auth-provider`: The affected behavior is provider/model compatibility metadata for the github-copilot model route and downstream provider payload construction.

## Summary

Close as a duplicate: the same github-copilot Claude eager tool input streaming failure is already tracked by the older open canonical issue, and an open fix PR is attached there while current main still matches the same source-level root cause.

## What This Changes

The issue reports that github-copilot Anthropic/Claude models still fail because Copilot model definitions omit the `supportsEagerToolInputStreaming: false` compat flag, causing pi-ai to send rejected `eager_input_streaming` tool payload fields.

## Best Possible Solution

Keep https://github.com/openclaw/openclaw/issues/75348 as the canonical tracker and review or replace https://github.com/openclaw/openclaw/pull/75393 with a fix that covers static manifest/default rows, live catalog rows, dynamic fallback rows, normalization, and focused regression coverage.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: current Copilot Claude model definitions can reach pi-ai with unset eager-streaming compat, and pi-ai 0.75.4 defaults that to emitting the rejected `eager_input_streaming` field; I did not run a credentialed live Copilot request.

## Solution Assessment

Yes, the provider-local compat opt-out is the right direction, but this specific report should close as a duplicate so the implementation discussion stays on the canonical issue and its open fix PR.
