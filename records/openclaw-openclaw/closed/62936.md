## Summary

Current `main` still does not implement the managed-browser doctor warnings from PR #62936, and the proposed patch has unresolved correctness issues already called out in review: an unguarded `resolveBrowserExecutableForPlatform` throw can crash `openclaw doctor`, and the new managed-profile detection would warn on `attachOnly` profiles that are documented to never launch a local browser.

## Evidence

- **Current main only checks existing-session Chrome MCP readiness:** `noteChromeMcpBrowserReadiness` still collects only Chrome MCP `existing-session` profiles and returns early when none are configured, so the managed-browser readiness warnings proposed by this PR are not present on `main`.
  - file: [extensions/browser/src/doctor-browser.ts:23](https://github.com/openclaw/openclaw/blob/245451b6a9c529464e177afa65013adca0d63eb5/extensions/browser/src/doctor-browser.ts#L23)
  - command: `nl -ba extensions/browser/src/doctor-browser.ts | sed -n '1,220p'`
  - sha: [245451b6a9c5](https://github.com/openclaw/openclaw/commit/245451b6a9c529464e177afa65013adca0d63eb5)
- **Managed executable resolver throws on stale browser.executablePath:** `resolveBrowserExecutableForPlatform` throws when `resolved.executablePath` is set but missing on disk. The PR patch calls this helper for managed profiles without a try/catch, which would turn a doctor diagnostic into a hard failure for a stale path misconfiguration.
  - file: [extensions/browser/src/browser/chrome.executables.ts:703](https://github.com/openclaw/openclaw/blob/245451b6a9c529464e177afa65013adca0d63eb5/extensions/browser/src/browser/chrome.executables.ts#L703)
  - command: `nl -ba extensions/browser/src/browser/chrome.executables.ts | sed -n '340,470p'`
  - sha: [245451b6a9c5](https://github.com/openclaw/openclaw/commit/245451b6a9c529464e177afa65013adca0d63eb5)
- **attachOnly profiles are explicitly non-launching:** OpenClaw's browser docs say `attachOnly: true` means never launch a local browser; only attach to one that is already running. The PR patch collects every non-`existing-session` profile as a managed launch candidate, so it would emit false-positive managed-launch warnings for attach-only profiles.
  - file: [docs/tools/browser.md:159](https://github.com/openclaw/openclaw/blob/245451b6a9c529464e177afa65013adca0d63eb5/docs/tools/browser.md#L159)
  - command: `nl -ba docs/tools/browser.md | sed -n '145,170p'`
  - sha: [245451b6a9c5](https://github.com/openclaw/openclaw/commit/245451b6a9c529464e177afa65013adca0d63eb5)
- **OpenClaw config supports attachOnly on non-existing-session profiles:** For `openclaw` driver profiles, `resolveProfile` carries through `attachOnly: profile.attachOnly ?? resolved.attachOnly`, so attach-only is not limited to `existing-session` profiles. That makes the PR's `profile.driver !== "existing-session"` filter too broad for managed-launch readiness warnings.
  - file: [extensions/browser/src/browser/config.ts:359](https://github.com/openclaw/openclaw/blob/245451b6a9c529464e177afa65013adca0d63eb5/extensions/browser/src/browser/config.ts#L359)

That discussion aligns with the current code and patch inspection, so there is not strong evidence to close this PR as obsolete or already implemented.
