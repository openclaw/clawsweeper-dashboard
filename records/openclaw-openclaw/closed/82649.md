## Summary

This PR should stay open, but it is not ready to merge: it targets a real SecretRef/gateway auth failure path, yet the patch broadens file-provider secret reads so world-readable secret payloads are accepted by default and still lacks real CLI/TUI proof from the reported `/etc/openclaw/secrets.json` setup.

## What This Changes

The PR changes file-backed SecretRef reads to pass `allowReadableByOthers`, includes lower-level resolver errors in gateway SecretRef unresolved messages, adds a POSIX regression test, and adds a changelog entry.

## Best Possible Solution

Define a narrow shared-system-secret permission policy that preserves strict defaults, then update resolver coverage, docs if needed, and redacted real CLI/TUI proof for the accepted `/etc` setup.

## Reproduction Assessment

Yes, source-reproducible: current main calls `readSecureFile` for file-provider payloads without `allowReadableByOthers`, and CLI/TUI gateway auth resolves `gateway.auth.token` SecretRefs before connecting. I did not establish a live cross-uid `/etc` reproduction.

## Solution Assessment

No. Passing `allowReadableByOthers` unconditionally is broader than the reported shared `/etc` case because it accepts world-readable secret payloads and still does not solve files owned by another uid.

## Review Findings

Overall correctness: patch is incorrect
