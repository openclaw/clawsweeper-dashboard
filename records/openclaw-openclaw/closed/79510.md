## Summary

Keep open: this PR addresses a source-reproducible gateway restart race that current main still appears to have, and the proposed cancellation/drain approach is focused. The patch has sufficient real behavior proof and no blocking code finding from this review, but GitHub currently reports the branch as conflicting, so it needs a refresh and normal merge gates rather than cleanup closure.

## What This Changes

The PR adds abort/drain context for gateway config hot reloads, stops the config reloader before channel teardown, skips canceled channel restarts, and updates gateway plus rebase-drift tests and the changelog.

## Best Possible Solution

Refresh the branch onto current main, keep the gateway-owned cancellation/drain fix, drop or revalidate any rebase-drift test-only edits, then land through normal checks with the linked issue closing on merge.

## Reproduction Assessment

Yes. The linked logs plus current-main source show a high-confidence path: defer a channel hot reload behind active work, trigger SIGUSR1/gateway-tool restart during the deferral, and the old hot reload can still call channel stop/start while restart creates a new channel lifecycle.

## Solution Assessment

Yes. Canceling and draining the config reloader at shutdown/restart is the narrow maintainable fix for stale deferred reload work; a process-wide per-channel single-flight would be broader and would mask old lifecycle work instead of stopping it.
