## Summary

Keep open. Current main has moved browser control away from the legacy `chrome-relay` path to `existing-session` / `profile="user"`, but I did not find strong evidence that the reported remote macOS node gap is fixed. The macOS app node still advertises `browser.proxy`, yet its Swift implementation only forwards to a local loopback browser-control port and does not start the local browser-control service first; the CLI node-host path does start that service before proxying. That makes the underlying remote macOS browser-proxy failure still plausible on current main, so this is not safe to close as fixed or obsolete.
