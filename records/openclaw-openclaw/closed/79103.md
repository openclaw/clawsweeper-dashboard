## Summary

Keep open: current main matches the documented contract, but the request would change host exec safety semantics by letting an approval-delivery toggle bypass host-local allowlists.

## What This Changes

The issue asks for `approvals.exec.enabled: false` and Discord native exec approvals disabled to make host exec ignore the per-agent allowlist and run commands unconditionally.

## Best Possible Solution

Preserve the separation between approval delivery and host execution authorization unless maintainers explicitly approve a new break-glass policy; otherwise improve diagnostics/docs to point operators to `tools.exec.security=full`, `tools.exec.ask=off`, and matching host approvals defaults for no-prompt mode.

## Reproduction Assessment

Yes, from current source inspection: disabling forwarding/native clients suppresses approval delivery, while host-local approval policy still enforces allowlist misses and can time out as `approval-timeout (allowlist-miss)`. I did not run a live macOS/Discord reproduction in this read-only review.

## Solution Assessment

No, not as stated: making `approvals.exec.enabled: false` bypass host allowlists would conflate delivery routing with execution authorization. A safer path is an explicit host policy change or clearer UX around the existing YOLO settings.
