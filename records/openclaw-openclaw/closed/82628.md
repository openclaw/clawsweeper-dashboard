## Summary

Keep open: current main still matches the reported WebChat auto-TTS dispatch/type gap, and this is a valid user-facing bug rather than stale cleanup.

## What This Changes

Auto-TTS in WebChat should attach and render generated local audio for `messages.tts.auto: "always"`, including block-streamed replies, instead of silently dropping it.

## Best Possible Solution

Fix the WebChat auto-TTS path so generated local TTS audio is rendered while preserving the internal local-media trust boundary and the existing SDK stripping of untrusted plugin payloads.

## Reproduction Assessment

Yes from source, with high confidence: final-mode TTS returns early for `kind: "block"`, synthesized TTS lacks `trustedLocalMedia`, and WebChat refuses local audio without that marker. I did not run a live browser/MiniMax reproduction in this read-only pass.

## Solution Assessment

No for the literal suggested SDK `Omit` removal: exposing `trustedLocalMedia` through the public plugin payload type conflicts with current tests and the internal trust boundary. The maintainable fix should keep the trust marker core-owned while allowing the speech-core/WebChat auto-TTS path to mark its own generated local audio as trusted.
