## Summary

Keep open: current main still uses a 2 second Chrome `--version` probe and reports timeout/null as an unknown Chrome version, and the open closing PR should resolve this issue only after it lands or is replaced.

## What This Changes

The issue requests a more reliable Chrome version probe for `openclaw doctor --deep` on macOS cold-cache runs where Gatekeeper can make the first Chrome spawn exceed the current timeout.

## Best Possible Solution

Review and land the open plist-first browser plugin fix with real macOS doctor proof, then close this issue as implemented.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes by source inspection: a Chrome `--version` call that exceeds 2000 ms makes `execFileSync` throw, `execText` return null, and doctor emit the reported warning; I did not reproduce the macOS Gatekeeper delay live.

## Solution Assessment

Is this the best way to solve the issue? Yes, the narrow maintainable path is to fix the browser plugin version probe, preferably with macOS plist-first detection and a bounded fallback timeout rather than changing unrelated core policy.
