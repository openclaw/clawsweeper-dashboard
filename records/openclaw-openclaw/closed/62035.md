## Summary

Current main implements the requested fix. Issue #62035 reported false-positive `inconsistent_timestamps` warnings when `startedAt` was a few milliseconds earlier than `createdAt`; merged PR #71871 normalized task lifecycle timestamps at create, update, and restore boundaries, while keeping the audit rule strict. Regression tests cover earlier `startedAt` on create/update/restore and assert zero `inconsistent_timestamps` findings. No fixed release tag was identified, so `fixedRelease` is null.

## Best Possible Solution

Close #62035 as implemented on main and keep the registry-boundary timestamp normalization from PR #71871. Users on 2026.4.5 should update to a build that includes #71871 once available; any future timestamp-audit noise should be filed as a new, narrower reproduction if it survives this normalization path.
