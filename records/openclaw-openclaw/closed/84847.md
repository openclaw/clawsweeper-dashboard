## Summary

Current main and the latest shipped release still process all resolved memory-core dreaming workspaces in one cron/heartbeat-triggered run without a memory-pressure guard. The issue also has a protected `maintainer` label and an open linked fixing PR, so it should remain open for explicit maintainer review rather than cleanup close.

## What This Changes

The issue asks for managed memory-core dreaming cron work to be bounded per tick and deferred under RSS or heap pressure instead of sweeping all configured workspaces in one gateway tick.

## Best Possible Solution

Review and either land or reject https://github.com/openclaw/openclaw/pull/84848, with the accepted fix keeping direct non-cron dreaming intact while bounding cron-triggered work and documenting any cadence change.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: current main resolves all configured dreaming workspaces and sweeps them in one cron/heartbeat-triggered path, with a test asserting multi-workspace fan-out; I did not run a live pressure scenario.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainer review accepts the cadence policy; a memory-core-owned bounded cron and pressure-deferral fix is the right direction, but the specific batch and threshold behavior should be reviewed in the linked PR.
