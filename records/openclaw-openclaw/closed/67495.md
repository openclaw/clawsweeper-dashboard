## Summary

Close as implemented on main. A newer merged implementation, #74742, shipped the bidirectional node file-transfer workflow as the bundled file-transfer plugin with dedicated `file_fetch`/`file_write` tools, node-host handlers, command gating, path policy, audit logging, and integrity checks, so this PR is now superseded by current main. https://github.com/openclaw/openclaw/pull/67495

## What This Changes

The PR branch adds `file_pull`/`file_push` nodes-tool actions, `file.read`/`file.write` node commands and node-host handlers, command-policy wiring, and unit coverage for the new transfer helpers.

## Best Possible Solution

Keep the shipped file-transfer plugin as the canonical implementation and close this PR rather than trying to retrofit the older `file_pull`/`file_push` branch.

## Reproduction Assessment

Not applicable as a feature PR rather than a bug report. The implementation state is directly verifiable from current origin main and the `v2026.4.29-beta.1` release artifact, which expose the replacement file-transfer tools and node-host commands.

## Solution Assessment

Yes. The shipped plugin is the better solution because it solves the same bidirectional transfer workflow while adding default-deny path policy, dangerous-command opt-in, approval/audit hooks, symlink defenses, and sha256 integrity checks that this branch does not fully provide.

## Review Findings

- **[P1] Use the path-scoped file-transfer policy:** `src/gateway/node-command-policy.ts:80-81`
  - body: This branch gates `file.read`/`file.write` at the command level, but the handlers accept broad absolute or home-relative paths once enabled. The shipped replacement adds per-node allow/deny path policy before dispatch; without an equivalent gate, enabling the command grants much broader node filesystem access.
  - confidence: 0.89

  - confidence: 0.86
