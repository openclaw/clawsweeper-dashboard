## Summary

Current main and the shipped `v2026.4.23` release already implement the managed loopback-CDP readiness fix for the `"not reachable after start"` false-negative described here.

## Evidence

- **Shipped release note matches the reported symptom:** `v2026.4.23` changelog explicitly says managed local Chrome readiness/status probes now bypass browser SSRF for their own loopback control plane, so OpenClaw no longer misclassifies a healthy child browser as `"not reachable after start"` (`#65695`, `#66043`).
  - file: [CHANGELOG.md:684](https://github.com/openclaw/openclaw/blob/a9797214338ba31c52c796adbb75afb16e0684a9/CHANGELOG.md#L684)
  - command: `git show v2026.4.23:CHANGELOG.md | nl -ba | sed -n '678,688p'`
  - sha: [a9797214338b](https://github.com/openclaw/openclaw/commit/a9797214338ba31c52c796adbb75afb16e0684a9)
- **Current reachability policy exempts the local managed profile:** `resolveCdpReachabilityPolicy(...)` now returns `undefined` for non-remote loopback profiles with `driver === "openclaw"`, so browser SSRF policy no longer self-blocks OpenClaw's own local CDP health/control checks.
  - file: [extensions/browser/src/browser/cdp-reachability-policy.ts:19](https://github.com/openclaw/openclaw/blob/250acdd2a28c190ed7934e3ba99022f24123d3c9/extensions/browser/src/browser/cdp-reachability-policy.ts#L19)
  - command: `nl -ba extensions/browser/src/browser/cdp-reachability-policy.ts | sed -n '19,33p'`
  - sha: [250acdd2a28c](https://github.com/openclaw/openclaw/commit/250acdd2a28c190ed7934e3ba99022f24123d3c9)
- **Managed-browser availability flow uses that policy for startup readiness:** The managed browser availability path resolves CDP reachability policy and passes it into both HTTP reachability and websocket readiness checks, so the loopback exemption applies directly to `start`/attach readiness for the `openclaw` profile.
  - file: [extensions/browser/src/browser/server-context.availability.ts:72](https://github.com/openclaw/openclaw/blob/250acdd2a28c190ed7934e3ba99022f24123d3c9/extensions/browser/src/browser/server-context.availability.ts#L72)
  - command: `nl -ba extensions/browser/src/browser/server-context.availability.ts | sed -n '72,95p'`
  - sha: [250acdd2a28c](https://github.com/openclaw/openclaw/commit/250acdd2a28c190ed7934e3ba99022f24123d3c9)
- **Regression tests lock the loopback false-negative fix:** Tests assert that local managed loopback CDP control stays outside browser SSRF policy, and that a pre-existing loopback browser is reused successfully under strict default policy with `undefined` passed to the reachability probes.
  - file: [extensions/browser/src/browser/cdp.helpers.test.ts:257](https://github.com/openclaw/openclaw/blob/250acdd2a28c190ed7934e3ba99022f24123d3c9/extensions/browser/src/browser/cdp.helpers.test.ts#L257)
  - command: `nl -ba extensions/browser/src/browser/cdp.helpers.test.ts | sed -n '257,266p'`
  - sha: [250acdd2a28c](https://github.com/openclaw/openclaw/commit/250acdd2a28c190ed7934e3ba99022f24123d3c9)
- **User-facing docs now describe this exact failure class:** The browser docs call out `Chrome CDP websocket for profile "openclaw" is not reachable after start` as a CDP readiness failure and explicitly document that local loopback `openclaw` profile checks intentionally skip SSRF reachability enforcement for OpenClaw's own control plane.
  - file: [docs/tools/browser.md:575](https://github.com/openclaw/openclaw/blob/250acdd2a28c190ed7934e3ba99022f24123d3c9/docs/tools/browser.md#L575)
