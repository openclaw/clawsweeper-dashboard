## Summary

Keep open: current main and the latest release still use a 30s default for non-media Codex dynamic-tool calls, and session_status has no timeout parameter, so the reported timeout path remains actionable.

## What This Changes

The issue asks to raise or make configurable the Codex app-server dynamic-tool RPC timeout so enumeration-heavy session_status calls do not fail at 30 seconds.

## Best Possible Solution

Raise the generic Codex dynamic-tool default in both main and side app-server paths, keep the existing 600s clamp and media/message special cases, and update focused tests and docs.

## Reproduction Assessment

Partly: source inspection gives a high-confidence path for any generic Codex dynamic tool that runs longer than 30s, and session_status cannot pass timeoutMs today. This review did not live-reproduce the reporter's large-session-store latency.

## Solution Assessment

Yes: raising the shared generic Codex dynamic-tool default in both main and side paths is the narrowest maintainable fix for the reported blocker. A config override can be considered later, but it is not required to unblock the described failure.
