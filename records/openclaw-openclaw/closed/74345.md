## Summary

Current main now covers the reported failure class: task maintenance clears stale/orphaned parent-owned ACP session metadata, ACPX has ownership-checked process cleanup and startup orphan reaping, and hot session-store writes no longer hold the writer lock across durable fsync work. The exact original live fingerprint was not rerun here, but the source, tests, changelog, related closure discussion, and fix provenance are strong enough to close as implemented on main.

## What This Changes

The issue reports that OpenClaw 2026.4.27 can saturate the Gateway event loop and repeatedly fail ACP session cleanup at idle, causing ACP session accumulation, model timeouts, and Telegram/Discord transport stalls.

## Best Possible Solution

Keep the current mainline ACP cleanup, ACPX process ownership reaping, and session-store non-durable hot-write behavior as the resolution for this 2026.4.27 regression, with narrower remaining saturation reports tracked separately.

## Reproduction Assessment

Yes for the originally reported bad build: the issue gives exact checkout/build/restart steps and a strong five-signal log fingerprint. I did not reproduce a current-main failure; current source and tests now cover the implicated ACP cleanup, process reaper, and session-write-lock paths.

## Solution Assessment

Yes. The implemented path is maintainable because it fixes the ownership and cleanup contracts directly and removes durable fsync from the hot session-store writer lock, rather than adding a watchdog or broad process-name reaper.
