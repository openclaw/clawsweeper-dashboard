## Summary

Current main already covers this reply path. Gateway `chat.send` still goes through the normal inbound reply runner, and that runner now persists fallback-selected provider/model state plus provider-scoped auth handling so later injected turns do not stay pinned to the original provider auth lookup.

## Evidence

- **Gateway path still uses the shared reply runner:** `chat.send` dispatches through `dispatchInboundMessage`, so gateway-injected sends exercise the same reply/fallback path as other inbound turns.
