## Summary

Current main has a focused WhatsApp reconnect/liveness implementation for the reported post-408 app-silent failure path, with regression coverage and docs/changelog notes tying it to the related #63855/#66920 reports. The fix is on main but I could not verify a containing release tag, so users on v2026.4.27 or older still need a newer build.

## What This Changes

The issue reports that WhatsApp inbound handling stays dead after a 408/app-silent reconnect cycle in 2026.4.26, especially in a dual-account setup, and asks for reconnects to restore the inbound message pipeline.

## Best Possible Solution

Keep the current plugin-owned reconnect and inbound fallback implementation on main, and ship it in the next release so 2026.4.26/v2026.4.27 users can move off the affected behavior without adding core-specific WhatsApp policy.

## Reproduction Assessment

Yes. The issue maps to an existing focused e2e reproduction that simulates a 408 reconnect followed by continued transport frames with no app-message delivery and verifies that the monitor creates another listener; the original nondeterministic live dual-account soak was not rerun here.

## Solution Assessment

Yes. The current fix stays inside the WhatsApp plugin boundary, separates transport liveness from app-message silence, preserves quiet healthy sessions, adds a shorter recovery window for recently active reconnects, and adds a group metadata fallback without moving WhatsApp-specific policy into core.
