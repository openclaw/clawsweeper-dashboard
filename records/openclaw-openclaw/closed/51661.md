## Summary

Current main already covers both reported problems: `openclaw browser` goes through the shared gateway credential resolver, which in local mode falls back to `gateway.remote.token` when local auth is unset, and the Chrome MCP existing-session `open`/`navigate_page` path now has timeout-and-reset handling that shipped in `v2026.4.22`.

## Evidence

- **Browser CLI uses shared gateway RPC path:** `openclaw browser` subcommands call `callBrowserRequest`, which routes through `callGatewayFromCli("browser.request", ...)`, so browser CLI auth comes from the normal gateway credential resolution path.
  - file: [extensions/browser/src/cli/browser-cli-shared.ts:30](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/extensions/browser/src/cli/browser-cli-shared.ts#L30)

  - file: [src/gateway/credentials.ts:102](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/src/gateway/credentials.ts#L102)
