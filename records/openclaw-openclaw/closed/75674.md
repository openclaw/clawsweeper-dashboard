## Summary

Keep open. Current main still preserves MCP `resource` content at the bundle-MCP boundary and provider transports still only consume Pi text/image blocks, while the linked closing PR remains open and unmerged.

## What This Changes

The issue reports that MCP tool results containing `type: "resource"` are passed to the model as `(see attached image)` instead of the resource text.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/75714 so MCP resource text is normalized at the bundle-MCP to Pi boundary before provider transports.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: return `content: [{ type: "resource", resource: { text: "..." } }]` from a bundle MCP tool, and current main preserves that raw block until provider paths that only read top-level text/image content.

## Solution Assessment

Yes. Boundary normalization at bundle-MCP materialization is the narrowest maintainable fix because provider transports already operate on Pi text/image content and should not each learn MCP-specific block shapes.
