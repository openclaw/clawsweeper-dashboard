## Label Justifications

- `P1`: The issue describes a real agent/channel workflow where a delivered or generated final reply can strand session state and block later inbound messages.
- `impact:session-state`: The central failure is persisted `pendingFinalDelivery` state remaining set after final delivery should have completed cleanup.

## Summary

Current main and the latest release still have the reported abort-before-clear ordering, so the stuck pending-final state bug remains real. Keep this issue open because https://github.com/openclaw/openclaw/pull/93201 is now the open closing PR for the exact fix and the issue should close only after that PR lands.

## What This Changes

Fix the user-message dispatch race where a late abort after successful final delivery can skip `pendingFinalDelivery` cleanup and block later inbound messages.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/93201 if its focused cleanup-before-abort fix remains acceptable, then close this issue as implemented.

## Reproduction Assessment

Yes, at source level: current main sends or queues the final payload, then checks the abort signal before clearing pending-final state. I did not run a live gateway timing reproduction in this read-only review.

## Solution Assessment

Yes; clearing durable pending-final bookkeeping after accepted final delivery and only then surfacing a late abort is the narrow owner-path fix. The open linked PR implements that shape with focused regression coverage.
