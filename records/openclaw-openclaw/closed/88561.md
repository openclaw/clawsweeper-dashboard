## Summary

Keep open: current main repairs ordinary transcript/truncation tool-result pairing, but a context-engine `assemble()` result is applied after those repair passes, and OpenAI chat-completions conversion can still serialize a remaining orphan `toolResult` as a `tool` message. The reported lossless-claw/WeChat path was not live-reproduced, but the source path matches the reported 499 failure closely enough for a narrow core repair candidate.

## What This Changes

The issue reports that lossless-claw compaction can orphan tool-role messages after a model switch, causing OpenAI-compatible provider requests to fail.

## Best Possible Solution

Add a generic post-context-engine-assemble replay repair or validation step that uses the existing tool-use/tool-result pairing helper before any assembled messages can reach provider transport, with focused regression coverage and no lossless-claw or WeChat special case.

## Reproduction Assessment

Yes from source inspection: if a context engine returns assembled messages containing a `toolResult` without its assistant tool call, current main can pass that orphan through to OpenAI chat-completions serialization. The exact WeChat/lossless-claw live sequence was not run.

## Solution Assessment

Yes: the narrow maintainable fix is to reuse the existing pairing repair after context-engine assembly, before provider conversion. A plugin-specific workaround would leave the same core boundary open for other context engines.

Do not hardcode lossless-claw, WeChat, provider names, or user examples in production code.
