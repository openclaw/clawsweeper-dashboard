## Summary

Close: current main and v2026.5.12 now make `openai/gpt-5.4-nano` the canonical nano route while `openclaw doctor --fix` repairs stale `openai-codex/*` config and session refs, so the 2026.4.5 failure mode is addressed without adding a separate Codex-provider nano row.

## What This Changes

The issue asks OpenClaw to stop `openai-codex/gpt-5.4-nano` from silently normalizing or falling through and to clarify whether nano is supported or unsupported on the Codex route.

## Best Possible Solution

Keep `openai/gpt-5.4-nano` as the supported nano model route and use doctor repair for stale `openai-codex/*` config/session state instead of reintroducing a legacy Codex-provider nano route.

## Reproduction Assessment

No current-main live reproduction was run. The old 2026.4.5 path is addressed in source by canonical `openai/gpt-5.4-nano` support plus doctor repair for stale `openai-codex/*` config and session refs.

## Solution Assessment

Yes. Canonicalizing legacy `openai-codex/*` refs to `openai/*` while preserving Codex auth/runtime is narrower and more maintainable than adding another legacy provider row for nano.
