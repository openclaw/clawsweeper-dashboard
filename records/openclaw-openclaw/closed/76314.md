## What This Changes

The PR adds a per-call dmScope override, uses per-channel-peer scoping for WhatsApp inbound direct messages, updates last-route recording, and adjusts WhatsApp docs, changelog, and tests.

## Best Possible Solution

Close this branch and continue review of the account-aware fix at https://github.com/openclaw/openclaw/pull/76730; close the linked issue only after that fix lands.

## Reproduction Assessment

Yes. Source inspection on current main shows WhatsApp direct inbound messages pass a peer to resolveAgentRoute, the resolver falls back to dmScope "main", and direct peer session-key generation returns the agent main key under that scope.

## Solution Assessment

No. This PR uses per-channel-peer, which fixes separate contacts on one WhatsApp account but omits accountId; the account-aware per-account-channel-peer approach in the canonical PR is the safer fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use account-aware WhatsApp DM scope:** `extensions/whatsapp/src/auto-reply/monitor/on-message.ts:94`
  - body: The changed override uses `per-channel-peer`, which drops `accountId` from the session key. In WhatsApp multi-account setups, the same sender on two accounts routed to the same agent can still share `agent:<agentId>:whatsapp:direct:<peerId>`, so this should use the existing `per-account-channel-peer` contract instead.
  - confidence: 0.9
