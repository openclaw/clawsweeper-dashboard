## Summary

Keep open. The PR targets a real current-main gap by adding effective exec defaults to the reply prompt hint, but the proposed headless warning is wired to `systemSent`, which current main uses as persisted session-system-prompt state rather than as a cron/system-turn signal. That would put headless guidance on ordinary later user turns and can miss first-turn system events, so the patch should not merge as-is.

## What This Changes

The PR changes reply prompt construction to include resolved effective exec defaults, adds a headless/system-turn exec guidance line, and extends the exec-hint unit test.

## Best Possible Solution

Keep the effective-defaults part, but derive headless guidance from an explicit system-turn signal such as heartbeat/provider run kind instead of `systemSent`, and add coverage proving cron/system turns get the warning while ordinary later user turns do not.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use the actual system-event signal for headless guidance:** `src/auto-reply/reply/get-reply-run.ts:520`
  - body: The new warning is gated on `systemSent`, but that flag only means the session system prompt has already been persisted. `ensureSkillSnapshot` sets it true on the first turn of normal chat sessions, so later user messages would be told this is a headless/system-driven turn and to avoid interactive approvals, while a first cron/system event can still have `systemSent=false` before the snapshot update. Gate this on the actual heartbeat/cron/exec event signal or a dedicated run-kind boolean instead.
  - confidence: 0.9
