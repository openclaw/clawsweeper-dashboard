## Summary

Current main already solves the stale gateway-lock restart delay in a broader way than PR #57919: lock recovery probes the configured gateway port before PID/cmdline checks, treats a free port as a dead owner, removes the stale lock, and retries. Gateway startup passes the resolved port into lock acquisition for initial start and restart reacquire, and the behavior shipped in 2026.2.23 via commit e6383a2c13e59c14fccd56a04417f869dac88bd6 from the #23760 follow-up.

## Best Possible Solution

Close PR #57919 and keep the shipped port-liveness stale-lock recovery from e6383a2c13e59c14fccd56a04417f869dac88bd6/#23760 rather than adding another per-lock process exit handler. If a future report shows a remaining gap, it should add a focused regression around the port-free EEXIST recovery path or a concrete startup path that calls `acquireGatewayLock()` without a port.

## Evidence

- **Current lock recovery uses port liveness before PID checks:** `checkPortFree()` probes the configured port with a timeout; `resolveGatewayOwnerStatus()` returns `dead` when the port is free, and the EEXIST path removes the stale lock and retries. This covers the early/unclean-exit case because a crashed gateway has no listener.
