## Summary

Current main, and the shipped v2026.5.20 release, already separate cron turn completion from job success for the reported blocker class: structured cron tool-denial signals, fatal error payloads, and run-level agent errors now record the run as `error` instead of `ok`.

## What This Changes

The issue asks cron status recording to treat blocked or incomplete agent runs with no expected output as non-success even when the agent ends its turn cleanly.

## Best Possible Solution

Keep the structured cron outcome classifier shipped in v2026.5.20; any new case should come in as a narrower report with run-log and tool/runtime metadata showing a blocker that is not represented as a fatal payload, failure signal, or run-level error.

## Reproduction Assessment

No current-main failing reproduction is present: source and regression tests show structured cron denials, fatal error payloads, and run-level errors now resolve to `status: "error"`. The original v2026.5.19 repro is credible, but it is covered by the v2026.5.20 path.

## Solution Assessment

Yes. The shipped approach is the maintainable direction because it uses structured tool/run metadata and fatal payload state, while avoiding the inverse bug of substring-scanning free-form assistant summaries.
