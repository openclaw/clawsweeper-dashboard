## Summary

Keep open: current main still contains the state-only uninstall path that can remove a workspace nested under `~/.openclaw`, and the closing-syntax repair PR #75061 is open but unmerged.

## What This Changes

The issue asks for uninstall state/config cleanup to preserve workspaces under `.openclaw` when the workspace deletion option is not selected.

## Best Possible Solution

Review and land PR #75061 or an equivalent narrow fix that preserves configured nested workspace directories during state cleanup unless `--workspace` or `--all` is selected.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: default workspace under `~/.openclaw`, recursive state removal of `~/.openclaw`, and workspace cleanup skipped when that scope is unchecked.

## Solution Assessment

Yes. The narrow maintainable solution is to pass configured workspace directories as a preserve list to state cleanup when workspace deletion is not selected, while keeping explicit workspace deletion destructive.
