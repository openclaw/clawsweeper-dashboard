## Summary

Current main resolves the reported user-visible failure. Kilocode discovery can still fall back to the one-entry static `kilo/auto` catalog, but configured-provider model misses no longer stop at that catalog: the embedded runner now synthesizes a runtime model from the configured provider and requested model id, so the request can reach Kilo Gateway. The Kilocode plugin also contributes configured Kilo model rows back into catalog/listing surfaces. The related Kilo discovery report #65578 is already closed.

## Best Possible Solution

Close this as implemented on current main. Keep the generic configured-provider passthrough and Kilocode catalog augmentation; if maintainers want retry or retain-last-discovered-catalog behavior later, track that separately as a resilience improvement rather than this client-side `model_not_found` bug.
