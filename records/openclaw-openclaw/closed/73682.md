## Summary

Keep open. Current main still has an actionable `anthropic:claude-cli` runtime failure path: Claude CLI credential caching is keyed only by credentials path, while the read result depends on whether Keychain access is allowed. A no-Keychain caller with a TTL can cache `null`, and the later external CLI sync path can reuse that `null` instead of reaching fresh macOS Keychain credentials. The reporter's follow-up live timing evidence strengthens the existing source-level diagnosis, and #71026 does not supersede this narrower cron/runtime failure.

## Best Possible Solution

Keep this issue open. The narrowest likely fix is to include Keychain access intent or credential source in the Claude CLI credential cache key, then add a regression that seeds a no-Keychain cached miss and verifies a later `anthropic:claude-cli` external-sync/cron auth resolution can still adopt fresh macOS Keychain credentials. It would also be useful to promote diagnostics around skipped or refused external CLI bootstrap when the outcome is an explicit-profile auth failure.
