## Summary

Keep open: current main still lacks the requested server-side WebChat dictation path, and open PR #76021 explicitly fixes this issue but has not merged. The issue should wait for that implementation review and landing rather than being closed early.

## What This Changes

Request server-side WebChat voice input using browser audio capture and the configured `tools.media.audio` transcription backend instead of browser `SpeechRecognition`.

## Best Possible Solution

Ship an authenticated, size-bounded WebChat dictation RPC/UI path that uses `tools.media.audio` and returns transcript text to the composer, then let the linked PR close this issue after merge.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Source inspection gives high confidence that current main has Talk and attachment flows but no MediaRecorder-style dictation control or Gateway transcription RPC.

## Solution Assessment

Yes, the active PR direction is the narrow maintainable solution: a dedicated draft-transcription RPC avoids overloading `chat.send`, keeps provider credentials server-side, and can enforce upload limits before returning text to the composer.
