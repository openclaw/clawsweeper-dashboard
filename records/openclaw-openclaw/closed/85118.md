## Summary

Close as implemented. Current main and v2026.6.6 moved auth-profile persistence and runtime state off `auth-profiles.json` into the per-agent SQLite store, with doctor importing legacy JSON, so the reported external lock contention on `auth-profiles.json` no longer blocks the gateway OAuth refresh path.

## What This Changes

The issue asks OpenClaw to prevent or document gateway OAuth refresh failures when external scripts use incompatible locks around `auth-profiles.json`.

## Best Possible Solution

Keep SQLite as the canonical auth-profile store, use doctor for legacy JSON imports, and treat direct writes to old `auth-profiles.json` files as obsolete external workflow debt.

## Reproduction Assessment

No. The reported current-main path depends on gateway runtime reads/writes to `auth-profiles.json`, and current source plus regression tests show runtime auth profiles are stored in SQLite and legacy JSON is not read.

## Solution Assessment

Yes. Moving canonical auth-profile state into SQLite with doctor-owned JSON import is a better long-term fix than teaching the gateway to interoperate with arbitrary external file-lock mechanisms around a legacy JSON store.
