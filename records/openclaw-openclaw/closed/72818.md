## Summary

Close: current main and release v2026.4.29 already solve the configured-channel startup problem through the policy-checked startup/deferred plugin plan. The PR’s raw configured-channel fallback is now redundant and would regress plugin disablement policy if merged.

## What This Changes

The PR merges loaded channel plugins with configured channel IDs resolved through bundled fallback lookup, adds a context diagnostic source field, and adds QA-channel thread roundtrip coverage plus small test formatting cleanup.

## Best Possible Solution

Keep the current manifest/lookup-table driven startup path and do not merge the raw fallback. If the diagnostic source field or extra QA thread coverage is still useful, it should land as a separate narrow PR that does not alter channel plugin selection.

## Reproduction Assessment

Yes. The intended startup scenario is traceable on current main through the lookup-table deferred channel plan and reload-before-start path; the PR regression is statically reproducible with a configured bundled channel plus plugins.enabled=false, a denylist entry, or plugins.entries.<id>.enabled=false.

## Solution Assessment

No. The PR’s direct cfg.channels fallback is not the best fix because it bypasses the policy-approved registry plan; the shipped startup/deferred plugin plan is the safer maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Gate fallback channel startup through plugin policy:** `src/gateway/server-channels.ts:692-696`
  - body: This loop starts raw configured channel ids through getChannelPlugin(), whose bundled fallback is not policy-aware. If the registry omitted a channel because plugins are globally disabled, denied, or disabled by entry, this can still resolve and start bundled plugin code; derive candidates from the policy-approved startup/deferred channel owner plan instead.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The PR broadens plugin execution in a way that can bypass configured plugin disablement policy.

Concerns:

- **[high] Plugin disablement can be bypassed:** `src/gateway/server-channels.ts:692`
  - body: Configured channel keys are resolved through bundled fallback lookup without checking global plugin disablement, denylists, or disabled plugin entries, so plugin code can start even when the registry correctly omitted it for policy reasons.
  - confidence: 0.88
