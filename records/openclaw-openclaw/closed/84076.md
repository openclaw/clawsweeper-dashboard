## Summary

Keep open: current main and v2026.5.18 have the prior fail-fast Codex app-server watchdogs, but the reported visible recovery/status and retry-safe state after a post-tool `turn/completed` loss is still not implemented or superseded.

## What This Changes

The issue requests visible recovery/status and retry-safe state preservation when a Codex app-server turn times out after `item/completed` without `turn/completed`.

## Best Possible Solution

Add a bounded recovery path for completion-idle timeouts after completed current-turn items: release the lane, emit an accurate user-visible failure/status, preserve replay metadata, and only offer or perform retry when side-effect safety is known.

## Reproduction Assessment

Yes. Source inspection plus the reporter's v2026.5.18 live logs give a high-confidence reproduction shape: Codex app-server emits `item/completed`, omits `turn/completed`, and OpenClaw times out via `turn_completion_idle_timeout`; I did not rerun the live provider/channel repro.

## Solution Assessment

Unclear as a full fix. Visible timeout status is a straightforward bug fix, but retry/resume after completed tools needs a maintainer decision around replay safety and side effects.
