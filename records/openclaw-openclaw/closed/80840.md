- `P2`: The report is a valid bounded Voice Call runtime gap affecting voicemail/call completion flows, but it is not a core runtime outage or security emergency.

## Summary

Keep open: current main and the latest stable tag still advertise configured Voice Call realtime tools to providers while runtime startup only binds `openclaw_agent_consult`, so the reported `end_call` gap remains valid and is not a duplicate of the earlier consult-tool fix.

## What This Changes

The issue asks for Voice Call realtime tools such as `end_call` to have a runtime handler binding path instead of being advertised to the model without dispatch support.

## Best Possible Solution

Add an owner-reviewed Voice Call realtime action contract: bind an explicitly configured built-in `end_call` tool to `manager.endCall(callId)` with tests and docs, and defer any generic `realtime.tools[]` action-mapping API to a deliberate follow-up.

## Reproduction Assessment

Yes. Source-level reproduction is high-confidence: configure a custom realtime `end_call` tool, the runtime advertises it, and a provider tool call for that name reaches `executeToolCall` with no registered handler and returns `Tool "end_call" not available`.

## Solution Assessment

Unclear as written. The suggested default auto-registration would fix the reported voicemail hangup path, but the safer maintainable direction is an explicit built-in realtime action contract before any broader custom-tool mapping.
