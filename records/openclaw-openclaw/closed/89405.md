## Label Justifications

- `P1`: The reported release behavior can accumulate hundreds of node.exe processes, make the gateway unresponsive, and break active channel/cron workflows.

## Summary

Current main addresses the reported runaway cron child-process accumulation by retiring isolated-cron bundled MCP runtimes in the run-finally cleanup path; the latest release still lacks that cleanup, so this is a current-main-only fix rather than a shipped one.

## What This Changes

The issue reports Windows node.exe process accumulation during WhatsApp cron run delivery and asks to make skipQueue configurable or fix completed run cleanup.

## Best Possible Solution

Keep the current main cleanup that retires isolated-cron bundled MCP runtimes at run end; avoid adding a skipQueue config knob unless new evidence shows a separate delivery-queue leak after this fix ships.

## Reproduction Assessment

No live Windows/WhatsApp reproduction was run, but the shipped/current source diff is clear: v2026.5.28 did not retire the isolated cron MCP runtime at run cleanup, while current main does and has targeted coverage.

## Solution Assessment

Yes. Fixing the cleanup path is the better solution because skipQueue is an internal duplicate-prevention guard for direct cron delivery, while the reported process accumulation is addressed by retiring the runtime that owns stdio child processes.
