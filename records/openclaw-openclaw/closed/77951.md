## Summary

Keep open: the stale ambient-key warning path is source-reproducible on current main, but this patch is too broad, still lacks contributor real behavior proof, and misses the required changelog entry.

## What This Changes

The PR changes Ollama discovery to set the provider-builder `quiet` flag from `!hasMeaningfulExplicitConfig` and adds a mocked stale-`OLLAMA_API_KEY` warning regression test.

## Best Possible Solution

Keep the stale ambient-key fix direction, but narrow the quiet logic so only truly ambient no-config discovery is silent while configured alternate loopback Ollama providers stay loud; add the changelog entry and contributor CLI proof before merge.

## Reproduction Assessment

Yes, at source level. Current main makes a real `OLLAMA_API_KEY` with no explicit `models.providers.ollama` produce `quiet: false`, and the provider builder warns when the localhost probe is unreachable.

## Solution Assessment

No. Removing `hasRealOllamaKey` from the quiet condition is too broad because existing tests define an explicit alternate-loopback-provider case that should remain loud.
