## Summary

Keep open. Current `main` still serializes both stages described in the issue: the voice-call response generator waits for one completed Pi-agent result before returning spoken text, and the Twilio telephony path still awaits one completed telephony TTS `Buffer` before chunking audio to the media stream. OpenClaw now also has a separate realtime voice mode, but that path is explicitly distinct from and mutually exclusive with the classic `streaming` telephony path, so it does not resolve this provider-generic streaming/sentence-pipelining request.

## Evidence

- **Voice response generation still returns one final text result:** `generateVoiceResponse` awaits `agentRuntime.runEmbeddedPiAgent(...)`, then extracts spoken text from the completed payload list and returns a single `{ text }` object. There is no incremental callback, iterator, or sentence-by-sentence handoff to TTS in this path.
