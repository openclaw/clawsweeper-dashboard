## Summary

Current main still broadcasts chat events broadly on the server, but the shipped Control UI and native chat clients all gate chat rendering by session before mutating UI state. The reported visible cross-talk does not match current behavior, so this PR looks obsolete rather than already implemented.

## Evidence

- **Gateway broadcast scope is still global for chat-class events:** `EVENT_SCOPE_GUARDS` treats `chat` and `chat.side_result` as generic operator-read broadcasts, and the broadcaster iterates every connected client unless a targeted connId set is supplied.
  - file: [src/gateway/server-broadcast.ts:22](https://github.com/openclaw/openclaw/blob/dc7c7034254971fa6e9769e99a18da88f432f8c7/src/gateway/server-broadcast.ts#L22)

  - sha: [dc7c70342549](https://github.com/openclaw/openclaw/commit/dc7c7034254971fa6e9769e99a18da88f432f8c7)
- **Chat event producers still use untargeted broadcast calls:** Current main still emits `broadcast("chat", ...)` from agent-event streaming/finalization paths, so this PR's server-side session-targeting is not already present under another commit.
  - file: [src/gateway/server-chat.ts:743](https://github.com/openclaw/openclaw/blob/dc7c7034254971fa6e9769e99a18da88f432f8c7/src/gateway/server-chat.ts#L743)

## Risks / Open Questions

- Server-side chat fanout is still broader than necessary, so there may still be passive over-delivery or efficiency/privacy concerns even though current clients do not render other-session chat events.
- This checkout appears shallow, so I could not audit exactly when the client-side session guards landed; the decision is based on current `main` behavior.
