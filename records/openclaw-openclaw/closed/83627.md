## Label Justifications

- `P2`: This is a normal-priority auth-provider enhancement with meaningful impact for power users but no current core outage.
- `impact:data-loss`: The report describes an OAuth credential profile being overwritten when another plan uses the same provider and email key.
- `impact:auth-provider`: The affected surface is provider OAuth profile identity, selection, and fallback rotation for OpenAI Codex auth.

## Summary

Close as duplicate/superseded: the same remaining auth-profile capability is already tracked by the open named OAuth login profile issue and its active implementation PR, while this report adds a concrete same-email Business + Pro use case rather than a separate source-level fix.

## What This Changes

The issue requests support for multiple OAuth profiles for the same provider and email, distinguished by plan/workspace/account identity so OpenAI Codex auth can rotate across subscriptions.

## Best Possible Solution

Keep the remaining work consolidated on https://github.com/openclaw/openclaw/issues/40402 and https://github.com/openclaw/openclaw/pull/49315, ensuring that the final profile-id design covers same-email plan/workspace cases without breaking existing OAuth profiles.

## Reproduction Assessment

Yes, by source inspection: current OpenAI Codex identity returns the email as the profile name even when account/plan metadata exists, and auth-profile persistence upserts by that id. I did not run a live OAuth flow with two OpenAI plans, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

Yes for cleanup: consolidating this on the existing named OAuth login/profile-id tracker is the best path. The implementation still needs maintainer agreement on profile-id semantics and compatibility in the canonical issue/PR.
