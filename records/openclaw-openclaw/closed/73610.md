## Summary

Keep open. Current main still gates Telegram long-polling startup on a successful `deleteWebhook` cleanup; recoverable cleanup failures back off and retry before the grammY runner starts. The related implementation PR #47384 remains open and unmerged, so the reported behavior is not yet fixed on main.

## Best Possible Solution

Keep this issue open and use it to drive the Telegram plugin fix. The best path is to review/rebase #47384 or land an equivalent owner-module change in `extensions/telegram/src/polling-session.ts`: after a recoverable `deleteWebhook` cleanup failure, call `getWebhookInfo`; if the returned webhook URL is empty, mark cleanup satisfied and start polling; otherwise preserve retry behavior. Add or update Telegram regression tests, and update docs only if a user-facing option is introduced.

## Risks / Open Questions

- The sticky IPv4 and transport fallback work may reduce some Telegram route-timeout failures, but it does not remove the startup dependency on successful `deleteWebhook`, so closing this as already fixed would hide the remaining empty-webhook case.
- A fix should not blindly ignore all cleanup failures; it should continue polling only after confirming `getWebhookInfo` reports an empty webhook URL, while preserving retries or hard failures for active webhooks, auth errors, and confirmation failures.
