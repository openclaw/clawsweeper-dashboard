## Summary

Current `main` already implements configurable embedded-runner LLM idle-timeout handling for the first-response/bootstrap phase, including `agents.defaults.llm.idleTimeoutSeconds`, `0` to disable, fallback to `agents.defaults.timeoutSeconds`, updated user-facing guidance, and regression coverage for slow/silent timeout paths.

## Evidence

- **Runtime timeout resolution:** The embedded runner resolves the first-token idle watchdog from config and only wraps the model stream when the resolved timeout is greater than zero. `resolveLlmIdleTimeoutMs()` honors `agents.defaults.llm.idleTimeoutSeconds`, treats `0` as disabled, falls back to `agents.defaults.timeoutSeconds`, disables the unconfigured watchdog for cron, and otherwise uses a 120s default.
  - file: [src/agents/pi-embedded-runner/run/llm-idle-timeout.ts:22](https://github.com/openclaw/openclaw/blob/6b618f063516aa9ba85903aa4565c5a204c055b2/src/agents/pi-embedded-runner/run/llm-idle-timeout.ts#L22)
  - command: `nl -ba src/agents/pi-embedded-runner/run/llm-idle-timeout.ts | sed -n '1,144p'`
- **Bootstrap path uses that resolver:** The embedded attempt path computes `idleTimeoutMs` before streaming the model response and applies `streamWithIdleTimeout(...)` only when the resolved value is positive, so `0` disables the watchdog on the first bootstrap response too.
  - file: [src/agents/pi-embedded-runner/run/attempt.ts:1621](https://github.com/openclaw/openclaw/blob/6b618f063516aa9ba85903aa4565c5a204c055b2/src/agents/pi-embedded-runner/run/attempt.ts#L1621)
