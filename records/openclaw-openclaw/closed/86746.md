## Summary

Keep open: current main and the latest release still use a 16K default live tool-result cap and the agent-facing gateway config tool still blocks edits to this path, while docs only partially expose the existing knob. The remaining work is real, but changing defaults and agent-writable config paths needs maintainer product/security judgment.

## What This Changes

Request to make `agents.defaults.contextLimits.toolResultMaxChars` more discoverable and better suited to large-context models, including default behavior, docs, `config.patch`, truncation notices, and a possible alias.

## Best Possible Solution

Keep the canonical `agents.defaults.contextLimits.toolResultMaxChars` path, improve discoverability and actionable truncation hints, and make any default or `config.patch` allowlist change explicit with focused compatibility/security tests; avoid a top-level alias unless maintainers choose a config-policy exception.

## Reproduction Assessment

Yes at source level: current main falls back to a 16K default, tests assert that default, the schema allows larger values, and the agent gateway config allowlist omits this path. I did not run a live gateway session because this review was read-only.

## Solution Assessment

Unclear as a single package: documentation and truncation-hint improvements are narrow, but dynamic defaults, agent-writable config changes, and an alias require maintainer decisions. A safer path is to improve the canonical knob first, then separately decide whether the default and protected-path policy should change.
