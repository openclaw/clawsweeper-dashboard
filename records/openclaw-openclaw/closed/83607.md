## Label Justifications

- `merge-risk: 🚨 auth-provider`: The diff changes provider credential resolution for cron preflight and can choose credentials that differ from the later model run.
- `merge-risk: 🚨 security-boundary`: The diff starts sending Authorization headers on a preflight path, so wrong-agent credential selection is a sensitive-data routing concern.

## Summary

This PR should stay open because the patch adds a compile-blocking unused parameter and the new auth lookup does not use the cron run's actual agent/profile context, so it can still probe with no token or the wrong token for non-default agents.

## What This Changes

The PR changes cron model-provider preflight to resolve an API key and send it as an Authorization bearer header on local provider probe requests.

## Best Possible Solution

Resolve preflight auth from the same cron agentDir/profile context used by the actual run, keep authless local providers headerless, and cover both authenticated and authless local provider probes with focused tests.

## Reproduction Assessment

Yes, by source inspection: current main sends the local provider probe as GET without headers, and fetchWithSsrFGuard returns any non-redirect HTTP response to the caller. I did not run a real LiteLLM/vLLM instance, so the noisy 401 log is source-reproducible rather than live-reproduced here.

## Solution Assessment

No. Adding auth to the probe is the right direction, but this implementation must use the same agent/profile context as the cron run and must compile under the repo's no-unused-parameter settings.
