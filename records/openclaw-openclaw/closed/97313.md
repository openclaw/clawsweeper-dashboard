## Summary

Keep open: current main still has a credible Z.AI path where a saved `zai:default` API-key profile can mask the current env/config key, and doctor auth health currently treats API-key profiles as static rather than detecting bogus instruction-string credentials.

## What This Changes

The issue reports Z.AI embedded/sub-agent runs failing with HTTP 401 because a stale or bogus saved auth profile is used after key rotation and `doctor --fix`.

## Best Possible Solution

Add a narrow Z.AI/auth-profile repair that rejects or repairs instruction-string API-key profiles, prefers the current configured credential only when safe, and gives `doctor --fix` a focused static-profile check.

## Reproduction Assessment

No high-confidence live reproduction was run, but the source path is clear: seed `zai:default` with an instruction-string API key and the current auth selection can try that profile before env/config fallback. The reporter's logs and follow-up give credible real-world evidence, but the upstream 401 was not rerun here.

## Solution Assessment

Yes, the best fix is a narrow auth-profile/onboarding/doctor repair rather than a broad sub-agent fallback change. The fix should validate or repair bad static API-key profile values and avoid changing intentional profile precedence for valid user-selected profiles.
