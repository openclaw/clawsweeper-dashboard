## Summary

Current main already preserves group-chat sender attribution as structured runtime context rather than inline body prefixes, so the reported loss of `SenderName` and `InboundHistory.sender` before the model does not hold for the checked-out code.

## What This Changes

Request to prefix Telegram group current and history messages with sender labels before they reach the LLM, optionally controlled per group.

## Best Possible Solution

Keep the structured sender/history metadata path on main; if literal `Alice: message` inline prefixes are still wanted, track that separately as a product/config request.

## Reproduction Assessment

No. Current main carries `SenderName` and `InboundHistory[].sender` into structured runtime context before the model turn, so the reported discard path does not reproduce by source inspection.

## Solution Assessment

Yes. The current structured metadata approach is the narrower maintainable solution because it preserves attribution while keeping untrusted sender names and history out of the plain user body.
