## Summary

Keep open. Current `main` still routes built-in `github-copilot` OpenAI Responses models through the generic HTTP transport without a Copilot-specific continuation hook comparable to the OpenAI WebSocket `previous_response_id` path, and the docs do not yet explain any premium-request continuation limitation. The bundled `copilot-proxy` plugin is a relevant workaround path, but it is a separate VS Code bridge and does not satisfy the built-in provider request in #60167.

## Evidence

- **GitHub Copilot plugin does not register a continuation hook:** The built-in `github-copilot` provider currently registers `wrapStreamFn` and `buildReplayPolicy`, but no provider-owned transport-turn-state or WebSocket-session hook that could carry provider continuation state across turns.
  - file: [extensions/github-copilot/index.ts:151](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/extensions/github-copilot/index.ts#L151)

  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **OpenAI WebSocket continuation is gated to provider=openai:** `shouldUseOpenAIWebSocketTransport` returns `false` unless `provider === "openai"` and `modelApi === "openai-responses"`. That excludes `github-copilot`, so Copilot models do not use the existing WebSocket `previous_response_id` continuation path.
  - file: [src/agents/pi-embedded-runner/run/attempt.thread-helpers.ts:41](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/agents/pi-embedded-runner/run/attempt.thread-helpers.ts#L41)

  - file: [src/agents/openai-transport-stream.ts:850](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/agents/openai-transport-stream.ts#L850)
