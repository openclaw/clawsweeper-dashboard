## Summary

Current main includes a shared memory-transport fix for env-proxied remote embedding calls, and Gemini memory indexing uses that path.

## Evidence

- **Gemini memory embeddings use the shared remote HTTP helper:** The Gemini embedding provider sends its POST requests through `withRemoteHttpResponse`, so proxy and fetch behavior is inherited from the shared memory-host transport layer rather than Gemini-specific fetch code.
  - file: [extensions/google/embedding-provider.ts:183](https://github.com/openclaw/openclaw/blob/dc7c7034254971fa6e9769e99a18da88f432f8c7/extensions/google/embedding-provider.ts#L183)
  - command: `nl -ba extensions/google/embedding-provider.ts | sed -n '166,200p'`
  - sha: [dc7c70342549](https://github.com/openclaw/openclaw/commit/dc7c7034254971fa6e9769e99a18da88f432f8c7)
- **Shared memory transport now opts proxied targets into trusted env-proxy mode:** `withRemoteHttpResponse` calls `fetchWithSsrFGuard` and switches to `GUARDED_FETCH_MODE.TRUSTED_ENV_PROXY` when `shouldUseEnvHttpProxyForUrl(url)` is true, which is the missing behavior for proxy-required remote embedding calls.
  - file: [src/memory-host-sdk/host/remote-http.ts:15](https://github.com/openclaw/openclaw/blob/dc7c7034254971fa6e9769e99a18da88f432f8c7/src/memory-host-sdk/host/remote-http.ts#L15)

- **Proxy detection covers HTTPS_PROXY/HTTP_PROXY and respects NO_PROXY:** The proxy helper mirrors Undici env-proxy selection: HTTPS requests prefer `https_proxy`/`HTTPS_PROXY`, then `http_proxy`/`HTTP_PROXY`, and `shouldUseEnvHttpProxyForUrl` suppresses proxying for `NO_PROXY` matches.
  - file: [src/infra/net/proxy-env.ts:34](https://github.com/openclaw/openclaw/blob/dc7c7034254971fa6e9769e99a18da88f432f8c7/src/infra/net/proxy-env.ts#L34)
