## Summary

Current main already implements the PR’s central behavior: live channel status probes use bounded concurrent account/channel work with stable result ordering, and the route-first CLI path uses a 30000ms default timeout for `channels status --probe`. The implementation is present in the v2026.4.24 release, so this PR is now obsolete as an implementation candidate.

## Best Possible Solution

Close this PR as already implemented and shipped. Keep the current bounded-concurrency route-first implementation on main; if maintainers want more hardening, open a smaller follow-up for the Commander fallback timeout default or a focused `channels.status` concurrency regression test.

- The Commander fallback in `src/cli/channels-cli.ts` still declares a 10000ms default, so `OPENCLAW_DISABLE_ROUTE_FIRST=1 openclaw channels status --probe` would not receive the 30000ms omitted-timeout behavior. The normal routed CLI path used by users avoids that fallback, so this is optional hardening rather than a reason to keep this PR open.
- Current main does not appear to carry the PR’s exact channel-specific fake-timer concurrency regression test, but the implemented behavior is source-visible and backed by the shared order-preserving concurrency helper tests.
