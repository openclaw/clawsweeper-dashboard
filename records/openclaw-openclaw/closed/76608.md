## Summary

Current main already tracks managed npm-root dependencies before invoking npm, so OpenClaw-managed plugin installs no longer prune prior managed plugins; the literal manual unpacking path remains outside the documented install surface. The fix is present on main after v2026.5.2 and is not in a release tag yet.

## What This Changes

Installing a new npm plugin should not cause npm workspace cleanup to delete previously installed plugins from the managed npm root.

## Best Possible Solution

Keep the managed npm-root manifest tracking on main and route manually unpacked or custom plugins through the documented `openclaw plugins install` or custom load-path flows.

## Reproduction Assessment

Yes for the released v2026.5.2 failure mode: npm can prune siblings when the managed root manifest omits prior plugin dependencies. I did not run a live repro under the read-only review constraint, but current-main source and tests show the installer now writes the cumulative dependency set before npm reconciliation.

## Solution Assessment

Yes. Tracking managed npm dependencies in the root `package.json` before install is the narrow maintainable fix because update and uninstall already rely on npm-managed package metadata; using `--no-save` would weaken that managed state.
