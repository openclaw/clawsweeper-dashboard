## Summary

Keep open: the UX is useful, but the patch needs rework before merge because it crosses the exec-approval trust boundary, misses current BlueBubbles reaction routing, and broadens the public gateway RPC surface.

## What This Changes

The branch adds BlueBubbles tapback approval handling that parses replied-to prompt text to call `exec.approval.resolve`, adds monitor tests, and exports `callGateway` from the plugin SDK gateway runtime.

## Best Possible Solution

Implement tapback approvals through BlueBubbles/native approval runtime metadata: track delivered prompt GUIDs, resolve only matching reaction targets, verify approval-scoped identity, use the approval gateway resolver, cover native webhook and negative cases, and add a changelog entry.

## Reproduction Assessment

Yes. The review issues are source-reproducible from the PR diff and current main: approval ids come from `replyToBody`, native reaction payloads route to `processReaction`, and pending approval state is not bound to delivered BlueBubbles prompt GUIDs.

## Solution Assessment

No. The requested UX is worth keeping, but the maintainable fix should be GUID-bound and approval-scoped instead of parsing free-form reply text or exporting generic gateway RPC access.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Bind approval tapbacks to tracked prompt GUIDs:** `extensions/bluebubbles/src/monitor-processing.ts:884-888`
  - body: This derives the approval id from `replyToBody`, which can be user-controlled or quoted text rather than proof that the reaction targeted an OpenClaw approval prompt. Store the delivered BlueBubbles approval message GUID and resolve only reactions whose target GUID matches pending approval metadata.
  - confidence: 0.92
- **[P2] Handle native BlueBubbles reaction webhooks:** `extensions/bluebubbles/src/monitor-processing.ts:882`
  - body: The resolver only runs from `processMessage`, but webhook handling normalizes `message-reaction` and `reaction` payloads and sends them to `processReaction`. Installations that emit native reaction webhooks will still enqueue a normal reaction event instead of resolving approvals.
  - confidence: 0.88
- **[P2] Use approval-scoped authorization for tapbacks:** `extensions/bluebubbles/src/monitor-processing.ts:895`
  - body: `commandAuthorized` is the control-command/channel gate and can disagree with the normal `/approve` authorization path. Tapback approval should verify the pending approval requester/approver or the channel approval capability before calling resolve.
  - confidence: 0.86
- **[P2] Keep gateway calls approval scoped:** `src/plugin-sdk/gateway-runtime.ts:4`
  - body: This feature only needs approval resolution, but exporting `callGateway` promotes a generic host RPC helper to the plugin SDK. Use `openclaw/plugin-sdk/approval-gateway-runtime` or a dedicated approval API so plugins do not get a broad gateway method seam.
  - confidence: 0.86
- **[P3] Add the user-facing changelog entry:** `extensions/bluebubbles/src/monitor-processing.ts:882`
  - body: This is a new BlueBubbles approval UX, but the PR does not update `CHANGELOG.md`. OpenClaw policy requires a user-facing feature entry before landing.
  - confidence: 0.8

## Security Review

Status: needs_attention

Summary: The patch changes exec approval resolution and public plugin SDK gateway access, and it still needs security hardening before merge.

Concerns:

- **[high] Approval resolution trusts spoofable reply text:** `extensions/bluebubbles/src/monitor-processing.ts:884`
  - body: The handler derives the approval id from `replyToBody` without proving the reacted-to message is a tracked OpenClaw approval prompt, so an allowed actor could craft or quote approval-shaped text and resolve a pending exec approval.
