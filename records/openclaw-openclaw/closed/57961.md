## Summary

Current `main` already fixes this: live model switching is limited to explicit pending user switches, and `LiveSessionModelSwitchError` is normalized into the fallback path so rate-limit failover can continue to the next candidate instead of looping the original model. The changelog says this shipped in `v2026.4.5`.

## Evidence

- **Changelog shipped fix:** `CHANGELOG.md` under `## 2026.4.5` says: only treat explicit user-driven model changes as pending live switches, so fallback rotation, heartbeat overrides, and compaction no longer trip `LiveSessionModelSwitchError` before making an API call.
  - file: [CHANGELOG.md:1194](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/CHANGELOG.md#L1194)

- **Live-switch detector narrowed to explicit pending flag:** `shouldSwitchToLiveModel` now returns a switch only when `liveModelSwitchPending` is set, and its comment explicitly says this replaced the earlier approach that could not distinguish user `/model` switches from system-initiated fallback rotations.
  - file: [src/agents/live-model-switch.ts:123](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/src/agents/live-model-switch.ts#L123)
