## Summary

Current main already preserves main-session cron `systemEvent` payload text through the transient runtime handoff: cron queues the text with a cron context key, heartbeat classifies that queued event as cron content, and the generated heartbeat prompt body embeds the payload text. The proof is present in the shipped `v2026.4.27` tag and remains on current main.

## What This Changes

Ensure `sessionTarget: "main"` cron `systemEvent` payload text reaches the main-session heartbeat/runtime context instead of producing only a bare heartbeat poll.

## Best Possible Solution

Keep the shipped transient-runtime-context implementation for main-session cron events, and handle separate ghost-run warning or routing-leak work in the narrower related reports that already track those distinct behaviors.

## Reproduction Assessment

Yes. The issue gives a concrete cron configuration and trajectory symptom, and current main has focused regression coverage that exercises the key path by enqueueing cron system events and asserting the heartbeat prompt body contains the payload text.

## Solution Assessment

Yes. The current implementation keeps synthetic cron wakeups out of visible chat while still feeding queued cron text into the runtime prompt, which is narrower and more maintainable than restoring visible user prompts. The open ghost-run warning work is adjacent diagnostics, not the missing-payload handoff itself.
