## Summary

Keep this PR open: the intended glyph change is useful, but the current diff would hide real archive failures by overriding archive rows before status-based result formatting can show error or conflict glyphs.

## What This Changes

The PR changes migrate output rendering so manual-review and archive migration items use kind-specific glyphs instead of status-derived glyphs.

## Best Possible Solution

Keep the kind-specific glyphs for non-failure manual/archive rows, but let archive `error` and `conflict` result rows keep the existing status glyphs and add focused output coverage for those cases.

## Reproduction Assessment

Yes. Source inspection shows archive items can validly become `error` or `conflict`, and the PR’s unconditional archive override would render those result rows with the book glyph before status-based failure glyphs are checked.

## Solution Assessment

No. The patch should preserve the useful manual/archive visual distinction without masking archive failure status in apply results.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Preserve failure glyphs for archive rows:** `src/commands/migrate/output.ts:184-186`
  - body: Archive items are not always successful: `archiveMigrationItem` returns `error` for missing/symlinked sources and filesystem failures, and `conflict` when the archive target already exists. Because this override runs before result-mode status formatting, those failed archive rows would display `📖` instead of `❌` or `⚠️`, making a failed apply look filed away rather than failed. Restrict the book glyph to preview or successful archive rows and let failures use the status glyphs.
  - confidence: 0.91
