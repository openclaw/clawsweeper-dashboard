## What This Changes

The issue reports a local-managed browser tab-selection race where `ensureTabAvailable()` discards an opened tab and filters tabs lacking `wsUrl`, leading to `BrowserTabNotFoundError`.

## Best Possible Solution

Patch `createProfileSelectionOps()` to preserve the opened or last non-empty local tab snapshot, tolerate delayed `wsUrl` discovery and transient `listTabs()` failures, and add focused regression coverage without changing browser config or public CLI/API surface.

## Reproduction Assessment

Yes, at source level. Current main still discards the opened tab, immediately re-lists, filters local-managed candidates by `wsUrl`, and throws when no candidate remains.

## Solution Assessment

Yes, the best fix belongs in `createProfileSelectionOps()`, where the known target is lost and delayed `wsUrl` metadata becomes `tab not found`; the prior PR review adds that transient `listTabs()` failures should be handled too.

## Repair Work Prompt

Fix the local-managed browser tab-selection race reported at https://github.com/openclaw/openclaw/issues/63343. In `extensions/browser/src/browser/server-context.selection.ts`, update `createProfileSelectionOps().ensureTabAvailable()` so an opened tab or last non-empty unfiltered local tab snapshot is not lost while waiting for delayed per-tab `wsUrl` discovery. Preserve existing per-tab `wsUrl` candidates as the preferred path, tolerate transient `listTabs()` failures inside the discovery polling loop, and fall back only to an already observed selectable tab when no `wsUrl` candidate appears. Add focused regression coverage in `extensions/browser/src/browser/server-context.tab-selection-state.test.ts` for: opened tab result/list lag, tab first observed without `webSocketDebuggerUrl` then later empty lists, and transient `/json/list` rejection during polling. Use the closed PRs only as context; produce a clean narrow patch, do not change public CLI flags/config/schema, and do not edit `CHANGELOG.md`.
