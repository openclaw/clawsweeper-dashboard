## Summary

Current `main` already prevents the reported duplicate Discord ACP final reply. ACP final fallback now depends on whether visible text was already delivered, routed/direct deliveries consult a channel visibility hook, and the Discord plugin explicitly marks non-empty `block` deliveries as visible text. The shipped `v2026.4.23` tag contains the same behavior.

## Evidence

- **Final fallback is suppressed once visible text was delivered:** `finalizeAcpTurnOutput` initializes `queuedFinal` from `hasDeliveredVisibleText()` and only sends the assembled final text fallback when visible text was not delivered or failed, which is the guard the issue said was misfiring.
