## Summary

Keep this PR open. Current main still has the stale gateway health-cache behavior, but the latest PR head is not ready because the new comparator can falsely treat fresh multi-account health snapshots as stale by comparing a default runtime account to a preferred top-level cached health summary.

## What This Changes

The PR refreshes gateway health cache entries when runtime channel state diverges and also carries agent fast-mode/model metadata/SSE transport, guarded extension fetch, CLI runtime-deps, and WhatsApp qrcode runtime mirror fixes.

## Best Possible Solution

Keep the runtime-snapshot freshness approach, but make the comparison account-aware so the cache is bypassed only when the same runtime account's lifecycle differs or an account is genuinely missing.

## Reproduction Assessment

Yes. A source-level reproduction is a channel with different default and preferred accounts whose lifecycle values differ: runtime.channels uses the default account, while cached health channels can use the preferred account.

## Solution Assessment

No, not as-is. The direction is maintainable, but the comparator should select cachedChannel.accounts[runtimeSnapshot.accountId] or avoid the top-level comparison when account identities differ.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Compare runtime channels to their cached account:** `src/gateway/server-methods/health.ts:50-52`
  - body: `runtime.channels[channelId]` is the default account, but `cached.channels[channelId]` can be the preferred/bound account. In a multi-account setup where those accounts have different lifecycle values, every fresh cache entry looks stale and bypasses the TTL; select `cachedChannel.accounts[runtimeSnapshot.accountId]` or skip the top-level comparison when account ids differ.
  - confidence: 0.91
