## Label Justifications

- `P2`: The reported older behavior could mutate user config and restart the gateway, but current main and stable releases already address the concrete startup/background write path.
- `impact:data-loss`: The issue is about unauthorized mutation of user-authored `openclaw.json` config data.
- `impact:session-state`: The reported restart could interrupt active gateway sessions and in-flight work.

## Summary

Current main and stable releases since `v2026.5.12` no longer match the reported older-release behavior: startup-derived gateway config changes are runtime-only, runtime defaults are not persisted back into `openclaw.json`, and OpenClaw-owned writes now have a persistent config audit path.

## What This Changes

The issue asks OpenClaw to stop background gateway logic from writing `memory.qmd.update` defaults into the user config and restarting the gateway without explicit user authorization.

## Best Possible Solution

Keep the shipped runtime-only startup implementation, leave actual config-edit restart policy under `gateway.reload.mode`, and keep the broader update-time overwrite pattern tracked at https://github.com/openclaw/openclaw/issues/85080.

## Reproduction Assessment

No, not on current main: source inspection shows the startup path no longer writes the derived config, and stable release `v2026.5.12` contains the runtime-only implementation. The older `2026.4.23` report is credible, but it no longer describes current behavior.

## Solution Assessment

Yes: making startup-derived changes runtime-only and keeping runtime defaults out of persisted config is the narrow maintainable fix for the unauthorized write that caused the observed restart. Changing the documented `gateway.reload.mode="hybrid"` restart policy for actual config edits would be separate product work.
