## Summary

Close as superseded: the same bundled MCP callTool timeout problem is now tracked by open #73536/#60967 with a config-backed implementation and focused tests, while this branch's progress-reset claim is incomplete under the MCP SDK contract.

## What This Changes

The PR changes bundled MCP tool execution to pass a five-minute `RequestOptions.timeout` and `resetTimeoutOnProgress` to `session.client.callTool` in `src/agents/pi-bundle-mcp-runtime.ts`.

## Best Possible Solution

Land one config-backed bundled MCP timeout fix that passes a per-call `RequestOptions.timeout` through the existing transport config/runtime seams; if indefinite progress-based extension is desired, register an `onprogress` handler/progress token or avoid claiming progress reset semantics.

## Reproduction Assessment

Yes. Current main calls `session.client.callTool` without request options, and MCP SDK v1.29 defaults requests to 60,000ms, so a bundled MCP tool that responds after more than 60 seconds reproduces the timeout. The progress-reset flaw in this PR is also source-reproducible because the SDK only resets on progress when an `onprogress` handler is installed.

## Solution Assessment

No. A fixed 300,000ms timeout reduces the reported symptom, but the newer #73536 approach better matches the existing config/transport boundary, and this PR's progress-reset option is ineffective without `onprogress`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Request progress notifications before relying on reset:** `src/agents/pi-bundle-mcp-runtime.ts:284`
