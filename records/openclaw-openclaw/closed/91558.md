## Summary

Current main already has the safe Kimi Code replay path: the OpenAI-compatible transport now honors the explicit reasoning-content compat flag for custom providers, and the bundled Kimi provider backfills `reasoning_content` on assistant tool-call replay when thinking is enabled. This is current-main-only for the custom-provider fix, so the reported v2026.6.1 behavior is explained by the release lag.

## What This Changes

The issue requests safe Kimi Code thinking/tool replay for OpenAI-compatible provider configurations that otherwise omit `reasoning_content` on assistant tool-call history.

## Best Possible Solution

Keep the current-main compat behavior and direct custom Kimi OpenAI-compatible users to either the bundled `kimi/kimi-for-coding` provider or `compat.requiresReasoningContentOnAssistantMessages: true` until the next release ships it.

## Reproduction Assessment

Yes for the shipped failure path at source level: v2026.6.1 ignored the explicit custom-provider compat flag in the agent transport, while current main honors it. I did not run the live Kimi request without credentials.

## Solution Assessment

Yes. The maintainable fix is to keep Kimi-specific behavior in the bundled provider and expose the generic compat flag for custom OpenAI-compatible routes instead of hard-coding every custom provider id.
