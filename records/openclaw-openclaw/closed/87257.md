## Summary

Keep open. Current main and the latest release still format inbound conversation metadata timestamps without seconds, while the shared timestamp formatters already support seconds; this is a small, source-reproducible observability improvement rather than a stale or duplicate item.

## What This Changes

Request to include seconds in the existing inbound conversation-info metadata timestamp shown to agents.

## Best Possible Solution

Add seconds to the existing inbound conversation-info timestamp while preserving envelope timezone settings and leaving the separate gateway-received timestamp request to its canonical issue.

## Reproduction Assessment

Yes, by source inspection: current main forwards inbound timestamps through `formatEnvelopeTimestamp` without `displaySeconds`, and the existing tests assert minute-only output. I did not run a live channel flow in this read-only review.

## Solution Assessment

Yes, the narrowest maintainable fix is to enable second precision only for the existing conversation-info timestamp path and update focused tests. Adding a new `receivedAt` or gateway timing field would solve a different, already tracked product request.
