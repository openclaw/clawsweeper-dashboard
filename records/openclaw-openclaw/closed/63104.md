## Summary

Current main already prevents the user-visible duplicate Discord reply/token-spend path through the shared auto-reply inbound dedupe. The PR discussion correctly identified cross-handler duplicate delivery and the deactivated-handler pitfall, but the shipped/current solution now claims inbound work in the generic dispatcher using the finalized Discord MessageSid context, so a second duplicate delivery returns before a second reply run is queued.

## Best Possible Solution

Close this PR as implemented on main by the shared inbound dispatcher dedupe. Keep the generic dispatcher guard as the canonical fix; if duplicate preflight work later becomes a measurable Discord monitor problem, track it as a narrower follow-up with fresh repro evidence.
