## Summary

Close as implemented: current main and the v2026.4.27 release contain the embedded Pi tool-registration path that keeps configured MCP tools in the effective tool set, converts them into Pi custom tools, and passes their exact names into the session allowlist before provider serialization.

## What This Changes

Configured `mcp.servers` tools should appear as provider-visible `server__tool` entries during normal embedded Pi/main-agent turns instead of being absent from outbound LLM requests.

## Best Possible Solution

Keep the shipped v2026.4.27 embedded-runner registration path. If a fresh capture on 2026.4.27 or newer still omits configured `server__tool` names, track that as a new regression against the provider/session boundary with the captured request body.

## Reproduction Assessment

Yes. The report gives a high-confidence reproduction path using working stdio MCP servers plus the debug proxy to inspect the outbound provider request; current source inspection verifies the previously missing registration boundary is now covered.

## Solution Assessment

Yes. Registering the configured MCP tools through the normal embedded-runner custom-tool and session-allowlist path is the narrow maintainable fix, and it preserves existing profile and `tools.deny: ["bundle-mcp"]` semantics.
