## Summary

Keep open: the fix direction is useful for the linked regression, but the current PR still breaks the runtime-plan payload key contract, misses native `/status` fast-path replies, and has only test/mock proof rather than a real TTS-enabled run.

## What This Changes

The PR adds an `isCommandResponse` marker to slash-command replies, tags normalized command-handler results, skips Auto-TTS for marked payloads, and updates command send-policy coverage.

## Best Possible Solution

Add the marker to all reply payload contracts and every slash-command fast path that can return status text, keep explicit `/tts audio` and `/tts latest` media replies working, and attach redacted real TTS-enabled proof before merge.

## Reproduction Assessment

Yes, source-reproducible: `/active-memory status` returns a plain text payload, command normalization does not mark it, and final dispatch applies Auto-TTS when `ttsAuto` resolves to `always`. I did not run a live TTS server or channel bot in this read-only review.

## Solution Assessment

No, not as submitted: the command marker plus central dispatch guard is the right shape, but this head must mirror the runtime-plan payload type, cover the native `/status` fast path, and provide real TTS-enabled proof.
