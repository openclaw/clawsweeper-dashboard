## Summary

Keep open: the help-text scope is useful, but the current diff incorrectly claims UTC is the default for cron expressions and still lacks rendered-help proof from the contributor.

## What This Changes

The branch changes `cron add` and `cron edit` CLI help so `--tz` mentions `--cron` and offset-less `--at` datetimes, and makes `cron edit --at` mention `--tz`.

## Best Possible Solution

Revise the help strings to mention `--cron` and offset-less one-shot datetimes while preserving separate defaults, then include redacted rendered help output.

## Reproduction Assessment

Yes. Source inspection shows the PR adds `default: UTC` to combined `--tz` help, while `resolveCronTimezone` falls back to the Gateway host timezone and tests assert that wording.

## Solution Assessment

No, not as written. The scope is narrow, but the implementation collapses recurring cron and one-shot datetime defaults into one UTC claim; the safer fix keeps those defaults distinct.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Remove the UTC default from cron add help:** `src/cli/cron-cli/register.cron-add.ts:89`
  - body: The new `--tz` help says the default is UTC for `--cron` as well as offset-less `--at`, but recurring cron schedules without `tz` fall back to the Gateway host local timezone. Users could omit `--tz` expecting UTC recurring execution and get local-time scheduling instead.
  - confidence: 0.94

  - confidence: 0.94
