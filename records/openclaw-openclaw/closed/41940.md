## Summary

Keep this PR open for maintainer follow-up. Current main still does not wire Feishu connection or transport activity into the gateway health monitor, and this PR head is incomplete against the current runtime path because the status callback is not passed from channel startup through the account monitor. Related open PRs #39487 and #39489 cover the same Feishu health-signal area and should be consolidated or one selected as canonical.

## Best Possible Solution

Keep this PR open for maintainer follow-up unless maintainers explicitly consolidate it into #39489 or another canonical PR. The best fix is Feishu-owned: pass ctx.setStatus/statusSink through channel.ts, monitor.ts, monitor.account.ts, and monitor.transport.ts; publish connected, mode, lastConnectedAt, lastEventAt, and validated lastTransportActivityAt; clear connected on abort/error/shutdown; and add focused tests proving the runtime snapshot drives the current gateway health policy.
