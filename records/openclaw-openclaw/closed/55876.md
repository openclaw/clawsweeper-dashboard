## Summary

Current `main` already routes Telegram forum-supergroup messages into topic-scoped sessions, including the case where inbound updates omit `chat.is_forum`. The Telegram message handler now resolves forum state via `getChat`, normalizes the message, and feeds the shared message-context/session pipeline; both webhook and polling modes use that same bot path.

## Evidence

- **Inbound forum detection fallback now handles missing `is_forum`:** `extensions/telegram/src/bot-handlers.runtime.ts` resolves forum state for every inbound `message` via `resolveTelegramForumFlag({ chatId, chatType, isGroup, isForum: msg.chat.is_forum, getChat })` and then normalizes the message before processing.
  - file: [extensions/telegram/src/bot-handlers.runtime.ts:1945](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/extensions/telegram/src/bot-handlers.runtime.ts#L1945)
  - command: `nl -ba extensions/telegram/src/bot-handlers.runtime.ts | sed -n '1939,1972p'`
- **Session creation uses resolved forum state to create topic session keys:** `extensions/telegram/src/bot-message-context.ts` calls `resolveTelegramForumFlag(...)`, then `resolveTelegramThreadSpec(...)`, and for forum groups sets `resolvedThreadId`, which drives topic-scoped routing/session creation.
  - file: [extensions/telegram/src/bot-message-context.ts:142](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/extensions/telegram/src/bot-message-context.ts#L142)
  - command: `nl -ba extensions/telegram/src/bot-message-context.ts | sed -n '142,154p'`

  - file: [extensions/telegram/src/bot/helpers.ts:85](https://github.com/openclaw/openclaw/blob/7130e56f872aea96577ac9fcbda0c7555a9ecc69/extensions/telegram/src/bot/helpers.ts#L85)
