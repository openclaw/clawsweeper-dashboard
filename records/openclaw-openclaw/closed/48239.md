## Summary

Current main already has the routing guard this report asks for: Telegram native approval handling is gated by `agentFilter`, resolves the request’s bound Telegram account from turn-source/session metadata, rejects ambiguous multi-account foreign-channel cases, and routes origin notices through the same turn-source-aware path. The same behavior is present in the latest release tag, so this can close as already implemented.

## What This Changes

The issue asks for Telegram exec approval prompts and `/approve` handling to stay bound to the requesting `main` agent/chat instead of surfacing through another Telegram-routed agent/chat.

## Best Possible Solution

Keep the shared turn-source/account-bound native approval routing in place. If a new wrong-chat case appears on `v2026.4.29` or newer, it should be filed as a fresh, narrower bug with the bot account id, session key, turn-source fields, and approval-route notice/log lines.

## Reproduction Assessment

Yes for a source-level regression path: current tests cover stale session delivery, stale account binding, multi-account ambiguity, and `agentFilter` request filtering. I did not run a live Telegram bot reproduction in this read-only cleanup pass.

## Solution Assessment

Yes. The current solution is the maintainable direction because it uses shared approval request metadata and account binding instead of a Telegram-local chat-to-agent mapping table, while still allowing DM-first native approval delivery.
