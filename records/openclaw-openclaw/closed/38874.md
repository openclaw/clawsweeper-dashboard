## Summary

Current `main` already supports supplementary memory-adjacent plugins alongside the single primary memory slot through non-exclusive memory prompt/corpus supplements plus normal lifecycle hooks, so the core feature request is implemented well enough even though `kind: "memory"` remains exclusive.

## Evidence

- **Exclusive `kind` is slot-only:** The SDK now documents `kind` as only for exclusive slots (`"memory"` or `"context-engine"`), which means additive companions should not claim the memory slot.
  - file: [docs/plugins/sdk-entrypoints.md:79](https://github.com/openclaw/openclaw/blob/d3dd61f2c5bd6cea175544f3a76a68ec11fa17c2/docs/plugins/sdk-entrypoints.md#L79)
- **Non-exclusive supplement APIs are public:** The plugin SDK documents `registerMemoryPromptSupplement` and `registerMemoryCorpusSupplement` as additive memory-adjacent surfaces, and the runtime API types mark both as non-exclusive.
  - file: [docs/plugins/sdk-overview.md:101](https://github.com/openclaw/openclaw/blob/d3dd61f2c5bd6cea175544f3a76a68ec11fa17c2/docs/plugins/sdk-overview.md#L101)
- **Runtime merges supplements beside the primary memory plugin:** Memory plugin state keeps one primary capability plus separate prompt/corpus supplement lists; `buildMemoryPromptSection(...)` returns primary output followed by sorted supplements.
  - file: [src/plugins/memory-state.ts:139](https://github.com/openclaw/openclaw/blob/d3dd61f2c5bd6cea175544f3a76a68ec11fa17c2/src/plugins/memory-state.ts#L139)
- **Supplements are not slot-gated:** Registry wiring only gates exclusive memory registrations; additive `registerMemoryPromptSupplement` and `registerMemoryCorpusSupplement` are registered directly for the plugin id.
  - file: [src/plugins/registry.ts:1487](https://github.com/openclaw/openclaw/blob/d3dd61f2c5bd6cea175544f3a76a68ec11fa17c2/src/plugins/registry.ts#L1487)
- **Recall injection and capture hooks already exist:** Plugins can mutate prompt context with `before_prompt_build` and receive end-of-run data with `agent_end`; the Pi runner fires `agent_end` after each run.
  - file: [src/plugins/hook-before-agent-start.types.ts:21](https://github.com/openclaw/openclaw/blob/d3dd61f2c5bd6cea175544f3a76a68ec11fa17c2/src/plugins/hook-before-agent-start.types.ts#L21)
- **Bundled example ships this pattern:** `memory-wiki` explicitly says it does not replace the active memory plugin, registers both additive memory supplements, and documents that shared memory search/get can reach it as a non-exclusive supplement.
  - file: [docs/plugins/memory-wiki.md:13](https://github.com/openclaw/openclaw/blob/d3dd61f2c5bd6cea175544f3a76a68ec11fa17c2/docs/plugins/memory-wiki.md#L13)
- **Shipped release evidence:** The changelog for `2026.4.7` records the restored `memory-wiki` stack plus Plugin SDK memory-artifact/memory-prompt seams for companion plugins and context engines.
  - file: [CHANGELOG.md:1055](https://github.com/openclaw/openclaw/blob/d3dd61f2c5bd6cea175544f3a76a68ec11fa17c2/CHANGELOG.md#L1055)

## Risks / Open Questions

- This does not allow two plugins to both declare `kind: "memory"`; third-party plugins like Cognee still need to migrate to the supported companion-plugin pattern to coexist with the primary memory slot.
- The public docs show the pieces, but there may still be some documentation gap for plugin authors expecting a dedicated `memory-augment` kind.
