## Summary

Keep open: current main and v2026.5.12 still reject approval-backed symlink cwd paths, and the requested escape hatch is a new security-sensitive config/product decision. The related implementation PR is open but unmerged and already has review, security, and real-behavior-proof blockers, so this issue should remain the tracking discussion.

## What This Changes

Request an opt-in setting that lets approval-bound `system.run` accept symlinked cwd paths while preserving canonical approval binding.

## Best Possible Solution

If maintainers accept the threat model, land a narrow default-off opt-in that displays requested and canonical cwd, records the symlink context in the approval plan, preserves canonical identity revalidation, and updates schema, docs, and regression tests without unrelated dependency changes.

## Reproduction Assessment

Yes for the existing behavior: current source and tests show approval-backed `system.run` denies direct symlink cwd paths and mutable symlink path components. This is not a broken existing contract; it demonstrates that the requested opt-in is absent.

## Solution Assessment

Unclear pending maintainer security/product review. The direction is plausible, but any implementation should preserve the approval UX binding for requested versus canonical cwd and avoid unrelated dependency policy changes.
