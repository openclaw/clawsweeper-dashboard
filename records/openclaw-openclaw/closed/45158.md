## Label Justifications

- `P1`: The original report described a user-facing Feishu routing regression that could send multiple users or accounts into one agent workflow.
- `impact:session-state`: Wrong routing can mix transcript, memory, and per-agent session state across users configured for isolation.
- `impact:security`: Misrouting can collapse intended user, account, or tool-policy boundaries into the wrong agent context.
- `impact:message-loss`: Messages can be delivered to the wrong configured agent rather than the intended per-user agent.

## Summary

Close as implemented: current main now covers the account-scope binding pitfall behind this Feishu multi-agent routing report, with explicit wildcard/default semantics in code and docs and a shipped beta tag containing the fix.

## What This Changes

The issue reports Feishu multi-agent bindings with seven configured users/accounts all dispatching to one agent session instead of the intended per-user agent sessions.

## Best Possible Solution

Keep the landed account-scope semantics: stored bindings remain explicit, new multi-account CLI bindings use wildcard scope when appropriate, and users configure `accountId: "*"` or a specific account for manual Feishu bindings.

## Reproduction Assessment

No current-main failing reproduction remains for the account-scope setup path: Feishu passes account and peer into routing, and new multi-account CLI bindings resolve to `accountId: "*"`. The exact seven-account setup was not live-reproduced because the full config and credentials are unavailable.

## Solution Assessment

Yes. The landed solution preserves explicit stored-binding semantics, adds wildcard/default documentation, and fixes the new bind/add setup path without silently broadening existing persisted configs at runtime.
