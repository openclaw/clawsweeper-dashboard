## Summary

Close as superseded by the maintainer-owned durable delivery semantics thread: current source and shipped docs intentionally keep `message_tool_only` final text private unless the message tool sends it, while broader delivery-status changes are already tracked elsewhere.

## What This Changes

The issue requests a Telegram/source-channel guard so normal assistant final text cannot remain internal when no `message(action="send")` delivery occurred.

## Best Possible Solution

Keep the current private-final contract, and route any durable sent/failed/suppressed delivery-state changes through https://github.com/openclaw/openclaw/issues/87561.

## Reproduction Assessment

Yes, source inspection reproduces the behavior: `message_tool_only` suppresses automatic source delivery, and tests prove those finals are not persisted for replay. I did not run a live Telegram turn.

## Solution Assessment

No, not as a separate Telegram auto-send fix. Automatically mirroring private finals conflicts with the current contract; durable delivery outcome semantics should be decided in the canonical cross-channel issue.
