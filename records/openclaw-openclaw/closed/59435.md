## Summary

Current main already preserves the installed macOS LaunchAgent service environment during package-update service refreshes. The update path reads the existing service command/environment, carries that captured environment into the updated `gateway install --force` and `gateway restart` processes, and launchd label/config-path resolution uses `OPENCLAW_LAUNCHD_LABEL`, `OPENCLAW_PROFILE`, and `OPENCLAW_CONFIG_PATH` from that service environment rather than rebuilding from an unrelated shell profile. The behavior is present in the latest release tag v2026.4.26 as well as current main.

## Best Possible Solution

Close as implemented on main and shipped by v2026.4.26. Keep the current update behavior that refreshes the LaunchAgent from the installed service environment; if a custom hand-authored plist without OpenClaw service env still fails, that should be a new narrower report with the exact plist/env details.
