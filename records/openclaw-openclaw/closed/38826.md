## Summary

Keep open. Current `main` still has a likely unfixed Telegram path for automatic TTS replies: generic outbound delivery preserves `audioAsVoice`, but the Telegram outbound adapter drops that flag before calling `sendMessageTelegram`, so routed auto-TTS/`[[tts:]]` replies can skip `sendVoice` and fall into text fallback / silent-token handling. That matches the report that `/tts audio` works while automatic Telegram TTS does not.

## Evidence

- **Generic outbound preserves the voice-note flag:** `deliverOutboundPayloads` carries `audioAsVoice` into `sendOverrides` and passes it through media delivery calls, so upstream code expects channel adapters to honor it.
  - file: [src/infra/outbound/deliver.ts:869](https://github.com/openclaw/openclaw/blob/4d7c4b3298920c0e6e2c5cc8bd7a1546e8122abe/src/infra/outbound/deliver.ts#L869)
  - command: `nl -ba src/infra/outbound/deliver.ts | sed -n '868,972p'`
- **Telegram outbound adapter drops the flag:** `telegramOutbound.sendMedia` and `telegramOutbound.sendPayload` forward `mediaUrl`, roots, threading, etc., but never map `audioAsVoice` to Telegram's `asVoice` option before calling `sendMessageTelegram`.
  - file: [extensions/telegram/src/outbound-adapter.ts:176](https://github.com/openclaw/openclaw/blob/4d7c4b3298920c0e6e2c5cc8bd7a1546e8122abe/extensions/telegram/src/outbound-adapter.ts#L176)
  - command: `nl -ba extensions/telegram/src/outbound-adapter.ts | sed -n '1,320p'`
- **Telegram only sends voice notes when `asVoice` is set:** `sendMessageTelegram` calls `api.sendVoice` only when `opts.asVoice === true`; otherwise audio goes through the normal audio path.
  - file: [extensions/telegram/src/send.ts:925](https://github.com/openclaw/openclaw/blob/4d7c4b3298920c0e6e2c5cc8bd7a1546e8122abe/extensions/telegram/src/send.ts#L925)
  - command: `nl -ba extensions/telegram/src/send.ts | sed -n '920,950p'`
- **Direct Telegram bot delivery does honor `audioAsVoice`:** The bot delivery path checks `reply.audioAsVoice` and calls `bot.api.sendVoice(...)`, which explains why native `/tts audio` can work even if routed automatic TTS does not.
  - file: [extensions/telegram/src/bot/delivery.replies.ts:350](https://github.com/openclaw/openclaw/blob/4d7c4b3298920c0e6e2c5cc8bd7a1546e8122abe/extensions/telegram/src/bot/delivery.replies.ts#L350)
  - command: `nl -ba extensions/telegram/src/bot/delivery.replies.ts | sed -n '350,410p'`
- **Routed-reply layer expects the flag to survive:** `routeReply` tests explicitly preserve `audioAsVoice` on outbound payloads, so the loss happens downstream in the Telegram adapter rather than earlier normalization.
  - file: [src/auto-reply/reply/route-reply.test.ts:403](https://github.com/openclaw/openclaw/blob/4d7c4b3298920c0e6e2c5cc8bd7a1546e8122abe/src/auto-reply/reply/route-reply.test.ts#L403)
  - command: `nl -ba src/auto-reply/reply/route-reply.test.ts | sed -n '400,420p'`
