## Label Justifications

- `P1`: The issue reports a current-release regression that prevents Telegram private-chat messages from creating agent turns while the channel otherwise appears connected.
- `impact:message-loss`: Allowed private Telegram text updates are visible from Telegram but are not counted, dispatched, or turned into OpenClaw agent messages in the affected release.

## Summary

Current main already fixes the Telegram isolated-polling path that could acknowledge/delete spooled updates before debounced normal-message processing finished, which matches the report where `/status` worked but normal private text was never counted. The fix is only on main after v2026.6.6, so closing as implemented on main is appropriate rather than leaving a duplicate release-regression report open.

## What This Changes

A normal allowed or paired Telegram private message in polling mode should be received and create an agent turn instead of being ignored while native `/status` still works.

## Best Possible Solution

Keep the current-main deferred spooled replay handling and ship it in the next release; no user config or product decision is needed for this issue.

## Reproduction Assessment

Yes from source inspection: v2026.6.6 could delete isolated spooled updates immediately after `bot.handleUpdate`, while current main adds deferred replay settlement for the normal-message/debounce path that the issue exercises.

## Solution Assessment

Yes: the main fix addresses the correct owner boundary by making Telegram isolated replay wait for deferred normal-message processing instead of adding config workarounds or changing allowlist semantics.
