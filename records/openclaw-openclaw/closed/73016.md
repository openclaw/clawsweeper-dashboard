## Summary

This should close as ClawHub/plugin work rather than core work. The request is a useful optional WhatsApp transport adapter, and current OpenClaw already has the channel-plugin, HTTP-route, inbound-dispatch, config, status, and outbound seams needed to build a WAHA adapter outside core; the issue does not identify a missing core API that blocks that path.

## What This Changes

Request a WAHA-backed WhatsApp channel plugin or `whatsapp.engine: "waha"` mode that uses WAHA REST send endpoints, webhooks for inbound events, lifecycle/status mapping, and existing WhatsApp access-control semantics.

## Best Possible Solution

Build WAHA as an external channel plugin published through ClawHub, using `registerHttpRoute` for WAHA webhooks, the channel plugin contract for config/status/outbound behavior, and shared inbound reply dispatch helpers; if implementation exposes a specific missing SDK seam, open a narrow plugin API issue or PR for that seam.

## Reproduction Assessment

Not applicable; this is a feature request, not a bug report. The verification path is scope/API feasibility, and current main has no WAHA adapter while the plugin API already covers the requested adapter shape.

## Solution Assessment

Yes, ClawHub/plugin routing is the narrowest maintainable solution. Adding a first-class core WAHA channel would expand core for an optional wrapper transport when current plugin contracts can host the integration.
