## Summary

Keep open. The PR targets a valid onboarding hatch stall, but the current patch scopes the 5-minute timeout too broadly because the existing TUI option is reused for every chat sent from the setup-launched TUI, including no-bootstrap sessions and later interactive messages.

## What This Changes

The PR adds a 5-minute timeout to setup's Terminal hatch relaunch, tests propagation through TUI launch and the embedded backend, and adds a changelog entry.

## Best Possible Solution

Scope the 5-minute timeout to the automatic setup wake run only, while preserving default timeout behavior for no-bootstrap setup hatches and later manual TUI messages.

## Reproduction Assessment

Yes. A source-level reproduction is enough here: pass timeoutMs from setup, keep the setup-launched TUI open, then send any later local message; current TUI code forwards opts.timeoutMs on every sendChat call.

## Solution Assessment

No. Passing the existing global TUI timeout option from setup is not the narrowest maintainable fix; a one-shot initial-message timeout or separate internal hatch option would match the intended contract better.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87
