## Summary

Keep open. The PR targets a real Active Memory timeout problem and the tool-construction filtering direction is useful, but the latest head still introduces a P2 regression by routing memory-only runs through a separate embedded-run setup that skips normal provider extra params and transcript compatibility wrappers.

## What This Changes

The PR adds an Active Memory memory-only embedded-run mode, forwards `embeddedMode` and `toolsAllow` through the runner, filters core/plugin tool construction by runtime allowlists, and tightens Active Memory hook/watchdog timeout handling and tests.

## Best Possible Solution

Keep the PR open and preserve the narrow `toolsAllow` construction optimization plus the tighter blocking deadline, but factor memory-only Active Memory runs through the shared embedded-run provider setup so extra params, provider wrappers, transcript repair, and tool-call compatibility stay consistent with normal embedded attempts.

## Reproduction Assessment

Yes. The linked bug has direct-chat Active Memory steps and timeout logs, and the PR-specific regression is reproducible by source inspection: Active Memory now passes `embeddedMode: "memory-only"`, and that branch returns before the normal setup that applies provider extra params and transcript wrappers.

## Solution Assessment

No. The requested direction is narrow, but this implementation should not fork a reduced embedded runner path; the safer solution is to share the normal setup and only avoid constructing tools that cannot survive the final runtime allowlist.
