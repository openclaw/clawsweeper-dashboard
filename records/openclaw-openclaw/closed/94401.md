## Summary

Keep open for maintainer merge review: the duplicate session-memory bug is still present on current main and the latest release, and this PR is a viable candidate fix with sufficient terminal proof, but it changes which persisted assistant transcript rows enter memory artifacts.

## What This Changes

The PR imports the delivery-mirror helper, tracks the last emitted assistant text in `getRecentSessionContent`, skips duplicate delivery-mirror rows, resets on user and command turns, and adds session-memory regression tests.

## Best Possible Solution

Land one canonical consumer-layer session-memory filter from this branch if maintainers accept the session-state tradeoff, then close the linked issue and supersede broader duplicate PRs.

## Reproduction Assessment

Yes, at source level: current main extracts and pushes text from every assistant message row, so a raw thinking row followed by a delivery-mirror copy with the same text yields duplicate assistant lines. I did not run a live provider `/new` or `/reset` repro in this read-only review.

## Solution Assessment

Yes, this is an acceptable narrow fix. A consumer-layer delivery-mirror duplicate filter is safer than deleting persisted transcript rows because replay and history surfaces intentionally keep those rows, though maintainers may still choose a stricter lineage-only predicate.
