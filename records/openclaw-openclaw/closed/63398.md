## Summary

Keep open. Current main still has the reported failure class: bundled MCP schemas are materialized raw, appended after the normal provider-schema normalization pass, and LiteLLM does not register a tool-schema normalization hook for OpenAI/Gemini-compatible upstreams.

## What This Changes

The issue reports provider validator failures when LiteLLM-routed OpenAI/Gemini models receive bundled MCP tool schemas with unsupported JSON Schema shapes.

## Best Possible Solution

Normalize or diagnose the final effective tool list after bundled MCP/LSP tools are appended, with provider-family handling for LiteLLM/OpenAI-compatible and Gemini-compatible routes and focused regression coverage for MS365/workspace-mcp-like schemas.

## Reproduction Assessment

Yes for the source-level failure class: current main still lets raw bundled MCP schemas reach LiteLLM/OpenAI-compatible serialization without recursive cleanup for nested type arrays or Gemini array-items gaps. The exact live MS365 tool that produced the original 400 remains unidentified.

## Solution Assessment

Yes in direction, but not as an existing patch: the maintainable fix is a provider-aware compatibility pass over final effective tools, not an MS365-only workaround or the narrower required-array change in #66415.
