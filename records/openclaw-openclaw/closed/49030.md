## Summary

Keep PR #49030 open. Current main does not yet implement the PR's central behavior for a custom provider id like `ollama-remote` with `api: "ollama"` and `apiKey: "ollama-local"`; however, the PR is not merge-ready as-is because the provided review context identifies concrete regressions in the proposed patch.

## Best Possible Solution

Keep the PR open but require a narrowed, production-ready rework: handle `models.providers.<customId>` entries with `api: "ollama"` and the `ollama-local` marker in both `resolveApiKeyForProvider` and `hasAvailableAuthForProvider` without noisy privacy-gate logging, add focused tests using localhost/neutral fixtures, and coordinate any broader custom-provider runtime ownership gaps with the existing multi-Ollama tracking items.
