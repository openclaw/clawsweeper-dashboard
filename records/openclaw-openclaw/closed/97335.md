## Summary

Keep open. Current main does not clearly fix the reported cron fallback abort; instead it has a recently merged 60 second cloud cron idle cap that conflicts with the reporter's expected 600 second progressing-fallback behavior and still needs maintainer policy review.

## What This Changes

The issue reports cron agent turns aborting after a provider fallback succeeds and makes progress, despite an explicit longer job timeout.

## Best Possible Solution

Keep the issue open until maintainers decide the cron fallback idle-timeout policy and verify a real fallback run that is making progress no longer aborts with a generic request failure.

## Reproduction Assessment

Yes at source level, not live. Current main routes the cron fallback through runWithModelFallback and caps remote/cloud cron model-call idle waits at 60 seconds even when the job timeout is 600 seconds, which is consistent with a progressing fallback run later aborting before the outer timeout.

## Solution Assessment

Unclear until maintainers pick the timeout policy. A code-only patch that simply removes or keeps the 60 second cap would be premature because the cap was recently merged to solve stream-stall fallback loss, and this report asks for different behavior once the fallback is actively progressing.
