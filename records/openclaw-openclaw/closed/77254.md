## Summary

Keep open: the report is fresh, corroborated by multiple deployments, and current main has related registry-pinning and external-plugin fixes but no high-confidence proof that the Discord adapter-loss path is fully fixed.

## What This Changes

The issue reports Discord outbound delivery losing its external plugin adapter after registry reload or upgrade, causing cron announcements and final replies to fail despite Discord remaining connected.

## Best Possible Solution

Fix generic external channel plugin resolution so the pinned or loaded channel registry remains authoritative for outbound target resolution and adapter loading across registry reloads and plugin updates, with no Discord-specific hard-code.

## Reproduction Assessment

No high-confidence live reproduction was run. The source path is clear enough for triage: a missing resolved channel plugin produces `Unsupported channel`, and a missing outbound adapter produces `Outbound not configured`, matching the reported logs.

## Solution Assessment

Yes, the narrow maintainable direction is a generic resolver/loader fix around the pinned or loaded channel registry plus regression coverage. A Discord-only workaround would cross the wrong ownership boundary.
