## Label Justifications

- `P1`: The shipped release could silently remove user runtime routing and break Anthropic MAX OAuth-backed agent runs.
- `impact:data-loss`: The reported failure is `doctor --fix` removing a user configuration mapping needed for runtime routing.
- `impact:auth-provider`: The removed mapping changes Anthropic OAuth routing from the Claude CLI backend to the direct API path.

## Summary

Current main fixes the reported doctor regression by moving recoverable whole-agent `claude-cli` runtime intent into provider/model-scoped Anthropic model entries before removing the legacy key. The fix is not in the latest tagged release, so v2026.5.18 users remain affected until the next release ships.

## What This Changes

The issue requests that `openclaw doctor --fix` preserve or migrate Claude CLI runtime mappings so Anthropic MAX OAuth routes continue using `claude-cli` instead of the direct Anthropic API path.

## Best Possible Solution

Keep the current migration fix and ship it in the next release; treat any separate cooldown-clear command as a follow-up only if users still need it after the route-preservation bug is gone.

## Reproduction Assessment

Yes. The v2026.5.18 source path deletes whole-agent `agentRuntime` without preserving it, and current main shows the repair path that migrates it into model-scoped policy; I did not run a live MAX OAuth setup.

## Solution Assessment

Yes. Migrating recoverable legacy whole-agent runtime intent into the provider/model-scoped policy is the narrow maintainable fix because the current resolver intentionally reads runtime policy from model/provider entries, not legacy agent-level pins.
