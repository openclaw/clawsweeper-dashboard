## Summary

Keep open: current `main` still has a source-level gap where `tools.alsoAllow` without `tools.allow` is additive for normal policy filtering, but the media generation factory receives a reduced nonempty allowlist that can filter out `image_generate`, `video_generate`, and `music_generate` before they become live.

## What This Changes

The issue reports that media generation tools are catalog-visible but not live on v2026.5.4 when global `tools.alsoAllow` is set without `tools.allow`.

## Best Possible Solution

Preserve implicit allow-all semantics for built-in media factory availability when `tools.alsoAllow` is used without `tools.allow`, while keeping the optional-plugin discovery guard that prevents loading every optional plugin tool.

## Reproduction Assessment

Yes by source inspection: `alsoAllow` without `allow` becomes an implicit allow-all policy, but the extracted factory allowlist drops the synthetic wildcard before media tool construction. I did not run a live gateway in this read-only sweep, but the current-main path matches the reported catalog-visible/not-effective symptom.

## Solution Assessment

Yes: the narrow maintainable fix is to carry or reconstruct the implicit allow-all semantics at the media factory boundary. Steering users to replace valid `alsoAllow` config with `tools.allow: ["*", ...]` would leave a silent valid-config trap.
