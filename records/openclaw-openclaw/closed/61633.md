## Summary

Current main covers the reported cron isolated-run accumulation/OOM chain: cron jobs are guarded by running-state markers with stale cleanup, isolated runs return terminal results on failure/abort paths, old cron run-session rows are pruned regardless of status, and session-store maintenance is enforced and capped by default, including load-time pruning for oversized stores. The exact lazy/streaming sessions.json loader proposed in the report is not the mechanism, but the observable unbounded backlog that made the gateway unbootable is now bounded and shipped in v2026.4.26.

## Best Possible Solution

Close this as implemented on main and keep the shipped cron/session maintenance path. If maintainers still want a true streaming or lazy `sessions.json` parser, that should be tracked as narrower storage architecture work rather than keeping this cron accumulation/OOM report open.
