## Summary

The request is a useful reset-orchestration workflow, but current main already exposes it as plugin/hook material rather than something that needs another core slash command. OpenClaw core has `/reset` and `/reset soft`, pre-reset hooks, startup-context reload, and bundled session-memory persistence; VISION says optional capability and heavy orchestration should stay out of core and go through plugins/ClawHub.

## Close Comment

Closing from a Codex review. Current `main` already provides the needed reset primitives through extension surfaces rather than needing another core slash command: `/reset` and `/reset soft` are built in (`docs/tools/slash-commands.md:91-93`), reset paths emit `before_reset` hooks with the prior transcript before context is lost (`src/auto-reply/reply/commands-reset.ts:48-118`, `src/auto-reply/reply/commands-reset-hooks.ts:140-157`), bare resets reload startup context including recent daily memory (`src/auto-reply/reply/get-reply-run.ts:347-404`), and the bundled `session-memory` hook already persists reset context into `memory/YYYY-MM-DD-slug.md` (`docs/automation/hooks.md:156-171`, `src/hooks/bundled/session-memory/handler.ts:53-170`).

Given that, this reads as plugin/skill orchestration on top of existing core reset hooks, not a core gap. That matches current project scope: `VISION.md:52-70` says optional capability should usually ship as plugins and new skills should go to ClawHub first, and `VISION.md:101-107` explicitly avoids heavy orchestration layers in core. A `/reboot` workflow can be implemented as a plugin/skill using `api.registerCommand(...)` plus reset hooks, so I’m closing this as better suited for ClawHub/plugin space rather than OpenClaw core.
