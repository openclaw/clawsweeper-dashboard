## Summary

Keep open. Current main still contains the reported resolver behavior, and the open closing PR remains the implementation path rather than a reason to close the issue early.

## What This Changes

The issue reports that isolated cron agent-turn summaries can surface an intermediate tool error instead of the agent's final recovered assistant reply.

## Best Possible Solution

Repair and land https://github.com/openclaw/openclaw/pull/74840 or a replacement resolver fix that lets trusted recovered final assistant text win for this all-error payload shape while preserving run-level failures, denials, structured delivery, and genuinely fatal tool errors.

## Reproduction Assessment

Yes, by source inspection. Calling `resolveCronPayloadOutcome` with only error payloads, `finalAssistantVisibleText`, and `preferFinalAssistantVisibleText: true` still resolves to the last error, and the current helper test asserts that behavior.

## Solution Assessment

No, not as a blanket final-assistant-wins rule. The fix belongs in `resolveCronPayloadOutcome`, but it should classify recovered versus unresolved fatal payloads before the fatal delivery override selects the error text.
