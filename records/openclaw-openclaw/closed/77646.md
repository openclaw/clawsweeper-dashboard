## Summary

This should stay open: current main still treats OTLP/HTTP protobuf as the supported gateway OpenTelemetry protocol, while JSON-only collector support needs a deliberate config/protocol decision. The related default-only PR was closed after the author agreed that flipping the Docker default alone is the wrong scope.

## What This Changes

Request durable gateway OpenTelemetry protocol handling so JSON-only OTLP/HTTP collectors do not require manual service-env edits that can be overwritten.

## Best Possible Solution

Choose one durable protocol contract and implement it end-to-end across diagnostics-otel, config/schema, service-env regeneration, Docker/docs, and focused regression coverage.

## Reproduction Assessment

Yes, source-level: current main still defaults Docker to http/protobuf, rejects http/json in diagnostics-otel, and documents only http/protobuf as honored today. I did not live-run the exact JSON-only collector rejection.

## Solution Assessment

No for the default-only approach: it would feed an unsupported protocol into current OpenClaw paths. The maintainable solution is either durable protocol override/preservation semantics or full http/json exporter support with schema, docs, and tests.
