## Summary

The requested Open WebUI Channels integration fits OpenClaw's documented external channel-plugin path and does not meet the current bar for adding a new bundled core extension.

## Evidence

- **Vision says optional capability should ship as plugins, not core:** Current main states that optional capability should usually ship as plugins, the preferred path is npm/local plugin distribution, plugins should be hosted and maintained in their own repository, and the bar for adding optional plugins to core is intentionally high.

  - file: [docs/plugins/building-plugins.md:11](https://github.com/openclaw/openclaw/blob/c6684af682bcb59bd1ee85cb49f72d4074007e2c/docs/plugins/building-plugins.md#L11)

## Risks / Open Questions

- Maintainers could still choose to bundle an Open WebUI channel later if they decide it is strategic enough to override the current plugin-first policy.
- Closing for scope does not provide the requested functionality; it only reflects that the supported path is an external ClawHub/npm channel plugin.

## Close Comment

Closing after a Codex review of current `main` at `c6684af682bcb59bd1ee85cb49f72d4074007e2c`. The request is reasonable, but it fits OpenClaw's external plugin path rather than a new bundled core extension. `VISION.md:54` says optional capability should usually ship as plugins, be hosted in their own repository, and that the bar for adding optional plugins to core is intentionally high. The current plugin docs also say you do not need to add a plugin to this repo and should publish channel plugins to ClawHub or npm instead (`docs/plugins/building-plugins.md:11`, `docs/plugins/sdk-channel-plugins.md:11`, `docs/plugins/community.md:9`). Given that current main already documents a supported community-channel plugin route, this is better tracked as a ClawHub/community plugin effort than as an OpenClaw core issue.
