## Summary

Keep open: current main still routes localized Spanish `schtasks /Create` denial through the throw path instead of the existing Startup-folder fallback, and the node install command uses that same Windows service backend.

## What This Changes

The issue requests that Windows node install fall back to the per-user Startup-folder launcher when `schtasks` returns localized Spanish access-denied output.

## Best Possible Solution

Extend the shared Windows `schtasks` fallback predicate to recognize localized access-denied failures and add focused regression coverage for the shared node/gateway service path.

## Reproduction Assessment

Yes by source inspection: a create failure with detail `Error: Acceso denegado.` reaches `shouldFallbackToStartupEntry` and currently returns false, so `activateScheduledTask` throws instead of writing the Startup launcher. I did not live-run the Spanish Windows host path in this read-only review.

## Solution Assessment

Yes: broadening the shared `schtasks` fallback predicate and adding a regression test is the narrowest maintainable fix because node install already uses the shared Windows service backend with node-specific task metadata.
