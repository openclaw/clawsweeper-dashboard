## Summary

Keep open. Current main still appears to contain the reported write-time clobber path: the auth-choice code returns a provider default as `agentModelOverride` under `setDefaultModel: false`, and `agents add` writes that override into the new agent config. The production direction in this PR is useful, but the branch is not merge-ready as-is because it weakens an auth-profile persistence assertion and is stale against current main.

## What This Changes

This PR updates provider/default auth-choice logic and tests so adding an agent with an existing `agents.defaults.model` primary inherits that default instead of writing the provider default as a per-agent model override.

## Best Possible Solution

Rebase the fix onto current main, suppress `agentModelOverride` only when the original config already has an inherited primary model, keep the no-default fallback behavior, restore unconditional auth-profile persistence assertions, and add focused coverage for both `applyAuthChoice` and the `agents add` write path.

## Reproduction Assessment

Yes. The source-level path is high-confidence: current main returns `agentModelOverride` under `setDefaultModel: false`, current `agents add` writes that override into the new agent entry, and existing tests still expect the override even with an existing default primary.

## Solution Assessment

No, not as-is. The production approach is the right narrow seam, but the branch should be rebased and the weakened auth-profile assertion fixed before it is a mergeable solution.
