## Summary

Keep open: current main and the latest release still reap non-terminal voice-call records solely because `answeredAt` is unset, so the reported active `speaking` Twilio call can still be ended at the 120-second default threshold. The focused fix PR that addressed this path was closed unmerged.

## What This Changes

Active Twilio voice-call conversations should not be ended by stale-call cleanup around the 120-second threshold.

## Best Possible Solution

Land a narrow owner-local voice-call fix that treats genuinely active conversation states as non-stale while preserving cleanup for stuck pre-answer records.

## Reproduction Assessment

Yes. A focused source repro is to pass the reaper a `speaking` or `listening` call older than `staleCallReaperSeconds` with no `answeredAt`; current main will call `endCall` on that record. I did not run a live Twilio call or tests in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to update the voice-call stale reaper or active-call bookkeeping to exempt active conversation states, with regression coverage; the closed unmerged https://github.com/openclaw/openclaw/pull/79124 demonstrates the likely shape.
