## Summary

Keep open. Current main still assumes the legacy cron crontab output and terminal note message are strings, and the doctor contribution loop does not isolate a thrown legacy-cron check, so the reported Termux/partial-Linux crash remains a valid narrow bug.

## What This Changes

The issue asks for `openclaw doctor` to skip or warn when Linux cron prerequisites return unexpected values instead of crashing.

## Best Possible Solution

Harden the legacy cron probe and note rendering against nullish or non-string values, add focused regression tests, and keep unsupported Linux environments as skip/warn cases rather than adding Termux-specific support policy.

## Reproduction Assessment

Yes, at source level: a Linux crontab reader resolving with `stdout: undefined` flows into `crontab.split(...)`, and the doctor contribution loop lets that throw abort the command. I did not live-run Termux, so this is source-reproducible rather than live reproduced.

## Solution Assessment

Yes, the narrow maintainable fix is to coerce or guard the crontab and note inputs at the affected boundaries, then test the degraded path. A broader Termux support feature or new config option is unnecessary.
