## Summary

Current main already reclaims stale session transcript lock files when the recorded PID is dead, zombie, recycled, too old, malformed, or a legacy same-PID orphan; it also cleans stale `.jsonl.lock` files during gateway startup and via doctor. The latest release tag available in this checkout, v2026.4.26, contains the same core session-lock implementation, while the report was against an older `2026.4.24` container image.

## Best Possible Solution

Keep the shipped stale session-lock reclaim and startup/doctor cleanup paths. If the reporter still sees this on a current v2026.4.26-or-newer image, the next best issue should include the live lock payload, `ps` state for the recorded PID, and whether the fallback model list is explicit config or generated by OpenClaw.
