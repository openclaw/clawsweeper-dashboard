## Summary

Current main and the v2026.6.6 release already cover the central failure: ordered OpenAI OAuth profiles are forwarded into Codex harness runs instead of leaving the inference path with `profile=-`; the separate migrated empty/default auth-store concern is tracked elsewhere.

## What This Changes

The issue reports OpenAI/Codex OAuth profiles present in auth state but not attached to inference requests, causing `profile=-` and 401 responses.

## Best Possible Solution

Keep the shipped v2026.6.6/current auth-forwarding path for this inference handoff, and use https://github.com/openclaw/openclaw/issues/91352 for the distinct migrated empty/default auth-store problem.

## Reproduction Assessment

No current-main failure path remains from source inspection: the reported 2026.6.1 logs are credible, but current main and v2026.6.6 now forward the selected OpenAI OAuth profile into Codex harness attempts. I did not run live OAuth inference in this read-only review.

## Solution Assessment

Yes. The current runtime-plan and scoped-store path is a better durable solution than only widening the bootstrap predicate because it forwards the Codex-owned OpenAI profile while preserving plugin-harness credential isolation.
