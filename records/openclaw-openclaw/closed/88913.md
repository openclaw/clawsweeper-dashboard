## What This Changes

The PR lets enabled installed plugin manifests contribute `nonSecretAuthMarkers`, filters disabled installed plugin roots from fallback metadata scanning, clears marker caches through plugin metadata lifecycle hooks, and updates the manifest docs/type comment.

## Best Possible Solution

Merge only after maintainer approval of the external-plugin marker trust boundary; if accepted, this manifest-metadata plus lifecycle-cache shape is the maintainable owner-boundary direction.

## Reproduction Assessment

Yes, at source level: current main filters manifest `nonSecretAuthMarkers` to bundled origins in `src/agents/model-auth-markers.ts`, so a global installed plugin marker is ignored. I did not run tests because this was a read-only review.

Conditionally yes: if maintainers accept the external plugin trust boundary, using manifest metadata with lifecycle-cleared caches is the clean owner-boundary fix. Without that decision, the safer alternative is to keep bundled-only markers or gate external markers behind a narrower trusted contract.
