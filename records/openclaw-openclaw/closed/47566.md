## Summary

Close: the PR is a useful plugin-authoring DX idea, but it adds optional scaffolding as a new OpenClaw core CLI surface even though current policy and a related closed request route this work to ClawHub/npm ecosystem tooling. The branch also lacks real behavior proof and is behind current plugin package, ID, activation, entrypoint, and docs contracts.

## What This Changes

The PR adds an `openclaw plugins init [directory]` CLI command plus tests to generate `openclaw.plugin.json`, `index.ts`, and `package.json` scaffolding for plugin projects.

## Best Possible Solution

Keep OpenClaw core focused on the plugin SDK, install, docs, and validation contracts; publish any scaffold generator as ClawHub/npm ecosystem tooling that consumes the canonical docs and snippets.

## Reproduction Assessment

Yes for the PR defects. Static inspection of the branch against current-main plugin contracts shows valid scoped IDs are rejected and generated scaffolds omit required package metadata, explicit activation, and the canonical entry helper; no live command was run because this was a read-only review.

## Solution Assessment

No. The feature direction is understandable, but adding this as a core CLI command is not the best current repository outcome; if maintainers later want core scaffolding, it should be redesigned from current package, manifest, and docs contracts.

Full review comments:

- **[P2] Include ClawHub-required package metadata:** `src/cli/plugins-init.ts:82-88`
  - body: The generated `package.json` only writes `openclaw.extensions`, but current external plugin validation requires `openclaw.compat.pluginApi` and `openclaw.build.openclawVersion`. A freshly scaffolded external plugin would still fail the package contract before publishing.
  - confidence: 0.94

  - body: `plugins init` rejects any ID containing `/`, so valid scoped IDs like `@team/my-plugin` fail even though the install path accepts `@scope/name`. Reuse the canonical plugin ID validator or keep the scaffold validator aligned with it.
  - confidence: 0.95
- **[P2] Emit explicit startup activation metadata:** `src/cli/plugins-init.ts:72-80`
  - body: The generated manifest omits `activation`, while current plugin docs say every plugin should choose `activation.onStartup` intentionally and omission no longer startup-loads implicitly. New scaffolded plugins should start from the current manifest contract.
  - confidence: 0.88
- **[P3] Generate the canonical plugin entry helper:** `src/cli/plugins-init.ts:36-40`
  - body: The scaffold emits a bare default register function, while current authoring docs and checklist use `definePluginEntry`. New projects should start from the canonical helper instead of a compatibility-shaped function export.
  - confidence: 0.8
- **[P3] Document the public init command:** `src/cli/plugins-cli.ts:828-836`
  - body: This adds a new public `openclaw plugins init` subcommand, but the CLI and plugin authoring docs are unchanged. Add docs for the command, options, generated files, and intended install/publish flow if this surface proceeds.
  - confidence: 0.84
