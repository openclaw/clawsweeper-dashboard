## Summary

Current `main` already prevents the two concrete Telegram failure modes in the report, and delivery recovery is bounded rather than indefinite. Reply-targeted Telegram sends now use `allow_sending_without_reply`, long outbound Telegram text is chunked before send, and recovery moves entries to `failed/` after a bounded retry budget.

## Evidence

- **Reply-target sends no longer fail on missing parent:** Telegram send builds reply params with `allow_sending_without_reply: true` for both `reply_to_message_id` and `reply_parameters`, preventing deleted/missing parent replies from failing the send path.
  - file: [extensions/telegram/src/send.ts:424](https://github.com/openclaw/openclaw/blob/7d471837369885c12d540d6bafa319ff41129b89/extensions/telegram/src/send.ts#L424)
- **Regression coverage for reply-target handling:** Tests assert threaded reply sends include `allow_sending_without_reply: true`.
  - file: [extensions/telegram/src/send.test.ts:510](https://github.com/openclaw/openclaw/blob/7d471837369885c12d540d6bafa319ff41129b89/extensions/telegram/src/send.test.ts#L510)
- **Long Telegram text is chunked before send:** Telegram send splits outbound text into 4000-char chunks before calling the API.
  - file: [extensions/telegram/src/send.ts:751](https://github.com/openclaw/openclaw/blob/7d471837369885c12d540d6bafa319ff41129b89/extensions/telegram/src/send.ts#L751)
- **Regression coverage for long-message chunking:** Tests assert both outbound Telegram chunks stay at or below 4000 characters.
  - file: [extensions/telegram/src/send.test.ts:1792](https://github.com/openclaw/openclaw/blob/7d471837369885c12d540d6bafa319ff41129b89/extensions/telegram/src/send.test.ts#L1792)
- **Recovery is bounded and prunes exhausted entries:** Delivery recovery uses `MAX_RETRIES = 5` and moves exhausted entries to `failed/` instead of retrying forever; tests cover both max-retry pruning and persisted retry-count increments.
  - file: [src/infra/outbound/delivery-queue-recovery.ts:43](https://github.com/openclaw/openclaw/blob/7d471837369885c12d540d6bafa319ff41129b89/src/infra/outbound/delivery-queue-recovery.ts#L43)
- **Shipped fixes called out in changelog:** Changelog records the missing-parent reply fix in `2026.3.22` and the cold-start outbound chunking fix in `2026.4.2`, which together cover the reported Telegram cases.
  - file: [CHANGELOG.md:2295](https://github.com/openclaw/openclaw/blob/7d471837369885c12d540d6bafa319ff41129b89/CHANGELOG.md#L2295)

## Risks / Open Questions

- The generic permanent-error matcher in `src/infra/outbound/delivery-queue-recovery.ts` does not explicitly list every possible Telegram 400 string, so unrelated permanent Telegram errors could still consume the bounded retry budget before moving aside.
