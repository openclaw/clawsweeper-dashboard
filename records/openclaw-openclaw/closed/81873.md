## Label Justifications

- `P2`: This was a normal-priority browser profile routing bug with a limited but real impact on existing-session browser users.
- `impact:session-state`: The issue concerns attaching OpenClaw to a user's existing signed-in browser session and preserving the configured session endpoint.

## Summary

Current main and the latest shipped release now implement the reported Chrome MCP endpoint forwarding: existing-session profile config is preserved, `cdpUrl` maps to Chrome MCP endpoint args instead of `--autoConnect`, and the WSL2 docs now direct split-host Windows Chrome setups to raw remote CDP.

## What This Changes

The issue asks OpenClaw to honor `browser.profiles.user.cdpUrl` and related Chrome MCP profile options for existing-session browser profiles in WSL2/Windows Chrome setups.

## Best Possible Solution

Keep the shipped profile-object forwarding and documented WSL2 split-host path: use raw remote CDP for Windows Chrome from WSL2, and use Chrome MCP existing-session only for host-local signed-in sessions.

## Reproduction Assessment

No; on current main, source and regression tests show the configured `cdpUrl` path maps to Chrome MCP endpoint args instead of `--autoConnect`. I did not run a live WSL2 Windows Chrome scenario.

## Solution Assessment

Yes; carrying the resolved profile object into Chrome MCP and documenting raw remote CDP for WSL2 split-host setups is the narrow maintainable solution for this report.
