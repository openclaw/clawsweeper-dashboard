## Summary

Keep open: current main has the low-level Chrome MCP `cdpUrl` forwarding path, but the product contract is still inconsistent across WSL guidance, profile creation, and browser docs, so this is not a safe cleanup-close.

## What This Changes

The issue reports that an existing-session browser profile in WSL2 ignores configured `cdpUrl`/`mcpArgs` and launches Chrome MCP with `--autoConnect` instead of attaching to a Windows Chrome CDP endpoint.

## Best Possible Solution

Align the browser profile contract: either support existing-session `cdpUrl` consistently across manual config, profile creation, tests, and docs, or document and diagnose WSL2 plus Windows Chrome as raw remote CDP with `attachOnly`.

## Reproduction Assessment

Unclear. Source inspection shows resolved existing-session profiles can forward `cdpUrl`, but I did not establish a live WSL2 plus Windows Chrome failure path and the issue's sample config, profile service, and docs do not agree on the intended mode.

## Solution Assessment

Unclear. A narrow code fix may be possible, but maintainers should first decide whether split-host WSL should use Chrome MCP endpoint forwarding or the documented raw remote CDP profile path.
