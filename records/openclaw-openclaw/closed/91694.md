Keep open: current main still has a source-visible gap for synthetic Codex app-server auth, so this is not safe maintainer cleanup. ## What This Changes

The issue reports that Codex/OpenAI quota usage disappeared from Telegram and CLI status after upgrading from 2026.6.1 to 2026.6.5-beta.6.

## Best Possible Solution

Bridge Codex app-server rate-limit data into the existing provider-usage hook so `/status` and `openclaw status --usage` can show OpenAI/Codex usage when runtime auth is synthetic and usable.

## Reproduction Assessment

Yes from source, but not from a live VPS run: current main gates OpenAI usage loading on an OAuth/token status label and then resolves provider usage auth from plugin hooks or auth profiles, while the Codex provider only exposes synthetic app-server auth and no provider-usage hook in the inspected registration.

## Solution Assessment

Yes: the narrowest maintainable solution is to use the existing provider plugin usage hook for Codex app-server rate limits rather than adding a new status-only fallback path.
