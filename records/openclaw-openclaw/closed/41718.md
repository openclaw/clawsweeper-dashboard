## Summary

Keep open. Current main still clears the Gateway Token when the Overview WebSocket URL input changes, so this PR is not implemented on main. The PR should not land as-is because its UI hunk preserves the existing token across endpoint edits, which can send or persist a gateway-scoped token under a different gateway URL. The unrelated memory-lancedb retry hunk is already covered more completely on current main.

## Best Possible Solution

Keep this PR open for maintainer rework or replacement in the #41545 cluster. The best fix is a narrow UI change that improves in-progress WebSocket URL editing without reusing or persisting an existing token for a different normalized gateway. A likely direction is to track the draft URL separately, clear or reload the token only when the normalized endpoint actually changes, or require explicit confirmation before reusing a token across gateways. Drop the already-superseded memory-lancedb hunk and add focused regression coverage for both edit ergonomics and cross-gateway token isolation.
