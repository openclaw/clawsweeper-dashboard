## Summary

Current main already routes Discord component interactions through an immediate matched-interaction ACK path before plugin handling, and the Discord component registry is shared across duplicate module instances. The shipped v2026.4.22 release contains both the implementation and regression tests, so the reported timeout/expired-button behavior is already addressed on supported builds.

## Close Comment

Closing based on a Codex review of current `main` and the shipped `v2026.4.22` release. Discord component interactions now ACK on the matched-interaction path before the plugin handler runs (`src/plugins/interactive.ts:34`, `extensions/discord/src/monitor/agent-components.ts:295`), and there is regression coverage that locks the ordering to `ack` before `handler` (`src/plugins/interactive.test.ts:676`). The Discord component registry is also global rather than module-local (`extensions/discord/src/components-registry.ts:1`), with a test proving state is shared across duplicate module instances (`extensions/discord/src/components.test.ts:116`). That addresses the two failure modes described here: late initial ACKs and lost custom-component state. This is present in shipped release `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). If you still hit this on a newer build, please open a fresh issue with exact version, component payload, and gateway/channel logs so maintainers can chase a new regression.
