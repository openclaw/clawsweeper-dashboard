## Summary

This should close because the same claude-max proxy serialization root cause was already triaged in #20566, where maintainers confirmed it was upstream proxy behavior rather than an OpenClaw core serializer defect. Current OpenClaw also has a generic string-content compatibility path for text-only OpenAI-compatible backends, so the remaining fix belongs in the external proxy or a patched fork, not a second OpenClaw issue.

## What This Changes

The issue reports that the external claude-max-api-proxy adapter coerces content-block arrays to `[object Object]` when OpenClaw users route Claude Max requests through that proxy.

## Best Possible Solution

Keep OpenClaw core generic, avoid a claude-max-specific serializer special case, and route the content-block serialization fix to the upstream proxy or a maintained patched fork; users needing a text-only workaround can use OpenClaw's existing `compat.requiresStringContent` model setting where appropriate.
