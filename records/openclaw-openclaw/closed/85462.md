## Best Possible Solution

Replace the global pool with a scoped gateway lifecycle fix that preserves full connection/security context, per-request callbacks, abort handling, and teardown, then split unrelated work and add focused tests plus redacted before/after `doctor --fix` proof.

## Solution Assessment

No; a global `GatewayClient` pool keyed by method/token/password is not the best fix because it breaks per-call lifecycle and omits auth/target context. A scoped transport reuse or caller-level throttling fix with tests and real upgrade proof is safer.

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Do not reuse a settled one-shot GatewayClient:** `src/gateway/call.ts:62-64`
  - body: `GatewayClient` stores the `onHelloOk` callback from its constructor options, so returning an existing live client reuses the first call's already-settled closure. A later same-key `callGateway` may start the old instance but never send the new method or settle the new promise correctly.
  - confidence: 0.94
- **[P1] Include the full gateway context in the pool key:** `src/gateway/call.ts:58-59`
  - body: The key only includes method/token/password, but the client is constructed with URL, TLS fingerprint, scopes, device identity, approval runtime token, client mode/name, instance id, and protocol bounds. Reusing across any of those differences can target the wrong gateway or authorization context.
  - confidence: 0.93
- **[P1] Preserve abort and accepted callbacks:** `src/gateway/call.ts:711-714`
  - body: Current main passes `signal` and `onAccepted` into `GatewayClient.request`, and `agent-via-gateway` relies on that to track accepted runs and abort them on signals. This patch drops those request options, so interrupted accepted gateway runs can keep running.
  - confidence: 0.91
- **[P1] Do not leave per-call CLI clients open indefinitely:** `src/gateway/call.ts:78-83`
  - body: `releasePooledClient` no longer stops the client after the wrapper promise settles. Current main waits for teardown before resolving; this no-op release can leave WebSockets, tick timers, and reconnect loops alive after a one-shot CLI command should be done.
  - confidence: 0.9
- **[P2] Actually enforce the pool size cap:** `src/gateway/call.ts:67-76`
  - body: When the map is at capacity and all entries are still live, this loop deletes nothing and then adds another key anyway. Diverse method/auth combinations can therefore grow past the advertised cap.
  - confidence: 0.88
