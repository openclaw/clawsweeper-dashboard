## Summary

## What This Changes

The issue requests that `writeConfigFile` skip disk I/O when the newly serialized config hash matches the existing snapshot hash.

## Best Possible Solution

Land a corrected version of https://github.com/openclaw/openclaw/pull/75543, or a narrow replacement, that adds the writer-boundary no-op guard with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: make `createConfigIO().writeConfigFile` process an existing snapshot whose serialized output hashes to the same value, and current main still proceeds toward `replaceFileAtomic` instead of returning early.

## Solution Assessment

Yes. The writer boundary is the narrowest maintainable place to suppress byte-identical config rewrites after a caller has already requested persistence; any semantic extension should preserve the writer return contract and existing JSON5-aware config conventions.
