## Summary

Keep open: current main still leaves owned chat abort controllers alone on WebSocket close, and the linked open PR is the active fix path. The remaining blocker is not evidence quality; it is deciding disconnect semantics so cleanup does not break reconnect/page-refresh continuity that current code and docs also support.

## What This Changes

Request WebSocket close cleanup so in-flight chat or agent runs owned by the closing connection are aborted instead of continuing until maintenance expiry.

## Best Possible Solution

Define disconnect ownership semantics, then land a narrow gateway cleanup that cancels truly abandoned runs while preserving supported reconnect and page-refresh continuity.

## Reproduction Assessment

Yes for the reported current-main behavior: source inspection shows the close handler does not inspect chatAbortControllers while chat and agent runs store ownerConnId, and the reporter supplied a five-trial wire-level reproduction. I did not run a live gateway E2E in this read-only pass.

## Solution Assessment

No for unconditional abort-on-close as the final answer without maintainer judgment. The existing abort helper is the right primitive, but the maintainable fix should account for reconnect and page-refresh continuity.
