## Summary

Keep this PR open: the current main path can silently persist a literal undefined JSONL line, and the proposed guard fixes that central serializer boundary with focused regression coverage and adequate real behavior proof.

## What This Changes

The PR makes `serializeJsonlLine` throw when `JSON.stringify` returns `undefined` and adds Vitest coverage for non-serializable root values plus normal JSONL serialization.

## Best Possible Solution

Land this narrow serializer guard with the added regression tests once normal CI and maintainer review pass, while keeping readers tolerant of already-corrupt transcript lines.

## Reproduction Assessment

Yes, at source level: current `serializeJsonlEntry` interpolates the raw `JSON.stringify` result, and Node confirms non-serializable root values stringify to `undefined`. The reader then catches parse errors and skips malformed JSONL lines.

## Solution Assessment

Yes. Guarding `serializeJsonlLine` is the narrow central fix because all JSONL write helpers use it, and the patch leaves normal object/primitives and existing reader tolerance unchanged.
