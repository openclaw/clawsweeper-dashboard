## Label Justifications

- `P2`: The reported bug can make semantic memory search unavailable for affected OpenAI-compatible embedding setups, but the fix is already present in the latest release/current main.
- `impact:session-state`: The failure mode disables or invalidates runtime memory recall despite a freshly rebuilt index.

## Summary

Current main and release v2026.6.10 already use the embedding runtime cache identity for memory index provider keys, so the reported deterministic CLI/runtime mismatch for OpenAI-compatible embeddings is fixed.

## What This Changes

The issue reports that `openclaw memory index --force` writes a different provider key than runtime `memory_search` expects for `openai-compatible` embedding providers with custom remote metadata.

## Best Possible Solution

Keep the v2026.6.10/current-main implementation that derives memory index identity from provider runtime cache data; broader missing-metadata race reports should stay on their separate trackers.

## Reproduction Assessment

No current-main failure reproduction is expected: source inspection shows CLI sync initializes the embedding provider and writes the same cacheKeyData-derived provider key that runtime status/search compare. The issue's OpenClaw 2026.6.8 steps are plausible for older builds but are covered by v2026.6.10/current main.

## Solution Assessment

Yes. The current implementation is the maintainable fix because provider identity is centralized in `resolveMemoryIndexProviderIdentities()` and shared by runtime identity checks and metadata writing rather than duplicating provider/model hashing in the CLI path.
