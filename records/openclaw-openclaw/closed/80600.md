## Summary

Keep open: this is an active protected feature PR, and the latest head now has live behavior proof plus green checks, but source review still finds a resource-boundary blocker in the code-execution worker and a module-guard false positive that should block merge.

## What This Changes

The branch adds an opt-in generic OpenClaw code mode with `exec`/`wait` controls, a QuickJS-WASI worker, hidden tool catalog bridging, config/schema/docs/tests, build wiring, and a `quickjs-wasi` dependency.

## Best Possible Solution

Keep the feature opt-in, enforce output limits before worker messages, fix template-literal handling in the module scanner, then have maintainers review the new code-execution and dependency surface before merge.

## Reproduction Assessment

Yes. Source inspection of the latest head shows the worker returns completed `value`/`output` without a worker-side `maxOutputBytes`, and the module guard still scans unmasked template literal text.

## Solution Assessment

No, not as-is. The feature direction is plausible and the latest proof/check state is much better, but the narrow maintainable path is to enforce the output cap inside the worker and make the module scanner distinguish template string text from executable template expressions.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Enforce output caps inside the worker:** `src/agents/code-mode.worker.ts:385-389`
  - body: The worker returns completed `value` and `output` before `maxOutputBytes` is enforced, and its config does not include that limit. A hostile code-mode cell can therefore transfer data above the configured cap across `worker_threads` before the parent rejects it, weakening the resource boundary this feature documents.
  - confidence: 0.9
- **[P2] Mask template literal text in the module guard:** `src/agents/code-mode.ts:369`
  - body: The module-access pre-scan masks comments and single/double-quoted strings, but leaves template literal text intact. Safe code like ``const msg = `import docs later`;`` still matches the `import` regex and fails before QuickJS runs, so ordinary string construction can break `exec`.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The new code-execution worker still lets oversized guest results cross the worker boundary before the configured output cap is enforced.

Concerns:

- **[medium] Output cap is enforced after worker transfer:** `src/agents/code-mode.worker.ts:385`
  - body: Because `maxOutputBytes` is absent from the worker config and completed results are posted back before parent-side checks, model-authored code can push larger serialized data through `worker_threads` than operators configured.
  - confidence: 0.9
