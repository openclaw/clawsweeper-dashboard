## Summary

Keep open. Current main still resolves stored auth-profile SecretRefs through the same strict startup preparation batch, and the open draft PR that closes this issue has not merged.

## What This Changes

The issue reports gateway startup aborting when a stale SecretRef remains in a non-default stored model auth profile.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/77213 or an equivalent targeted patch that separates strict config SecretRef resolution from startup-only stored auth-profile degradation with regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows stored api_key.keyRef and token.tokenRef values are collected into the startup-wide strict resolver, so a stale provider reference can still abort startup.

## Solution Assessment

Yes. The narrow maintainable fix is to keep config and reload SecretRefs strict while degrading only stale stored auth-profile refs during startup preflight.
