## Summary

Current main addresses the reported startup Telegram message drop by dispatching gateway startup hooks only after channel-start handoff and by routing Telegram message-tool sends through the Gateway action path to the Bot API, where success is based on the real Telegram response rather than an internal counter. The current send path surfaces missing Bot API message IDs as errors and has targeted retry coverage for transient pre-connect/rate-limit cases, matching the issue’s requested retry/failure-surfacing behavior even though it does not require waiting for the polling adapter to be fully connected.

## Best Possible Solution

Keep the current Gateway-routed Telegram message action path and startup-hook ordering on main. If a future user can still reproduce a real Telegram drop on a newer build, the follow-up should include Gateway logs plus the actual Bot API response fields so maintainers can debug a narrower Telegram delivery/runtime issue.
