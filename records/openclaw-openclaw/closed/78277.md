## Summary

Keep this PR open. It targets a real bootstrap-token lost-update bug, but the proposed merge result still leaves one bootstrap-token state writer on the process-local lock, the added regression test uses the same reentrant lock helper instead of a non-reentrant process boundary, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR adds a cross-process file-lock wrapper around several bootstrap-token state operations and adds an in-process lock-contention test for token issuance.

## Best Possible Solution

Serialize every mutating bootstrap-token state path with the cross-process file lock, add non-reentrant multi-process or stale-snapshot regression coverage, and require redacted after-fix setup-code proof before merge.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: one process can load and later replace `devices/bootstrap.json` while another process verifies/redeems a bootstrap token under only a module-local lock; I did not run a live two-process Gateway repro in this read-only review.

## Solution Assessment

No. Cross-process locking is the right narrow approach, but this patch is incomplete because one mutating cleanup path remains process-local and the regression test does not prove the cross-process lost-update behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Lock every bootstrap-state writer:** `src/infra/device-bootstrap.ts:316`
  - body: `revokeDeviceBootstrapTokensForDevice()` still deletes records and persists `devices/bootstrap.json` under the old process-local lock. A rejected pending device can therefore race with issuance or verification and overwrite newer bootstrap-token state, so route this path through the same cross-process bootstrap state lock.
  - confidence: 0.9
- **[P2] Replace the reentrant lock-contention test:** `src/infra/device-bootstrap.test.ts:97-99`
  - body: The new contention test acquires `acquireFileLock()` from the same process and helper used by the implementation, but that helper is reentrant. This does not prove the separate-process lost-update case, so use a child process or an explicit stale-snapshot harness that cannot re-enter the lock.
  - confidence: 0.88
