## Summary

Keep open. Current main still has a source-level path that can persist a text-only `:assistant-media` assistant transcript when media cannot be embedded, and open PRs #73962 and #75481 already target the narrow fix.

## What This Changes

The issue asks Control UI/WebChat to stop storing duplicate assistant replies from the gateway assistant-media/TTS supplement path after media delivery has been disabled or cannot resolve.

## Best Possible Solution

Land one canonical gateway fix that only writes supplemental assistant-media transcript entries after resolved audio or image content exists, with a stale-media regression test and changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an agent-run final/block payload with visible text plus a stale or unembeddable media URL: `isMediaBearingPayload` passes, media building returns `null`, and the append path can persist fallback text with a `:assistant-media` key.

## Solution Assessment

Yes. Gating the supplemental append on resolved non-text media content is the narrowest maintainable fix because the normal model transcript already owns the visible assistant text while command/non-agent TTS fallback behavior can stay unchanged.
