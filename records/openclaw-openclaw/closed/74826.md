## Summary

Keep open. Current main still contains the unconditional capability-provider fallback runtime-dependency opt-out, and this PR appears to make the narrow intended repair by restoring the loader default for user-facing capability discovery while preserving an explicit opt-out for snapshot-style callers. I found no blocking correctness or security findings in the two-file diff, but the PR still needs normal maintainer review and changed-gate/package-path validation before merge.

## What This Changes

The PR adds an optional `installBundledRuntimeDeps` parameter to capability-provider fallback resolution, omits the loader opt-out by default, and updates capability-provider runtime tests to cover both default install behavior and explicit opt-out behavior.

## Best Possible Solution

Land this narrow PR or an equivalent patch after maintainer review and changed-gate/package-path validation, so user-facing capability discovery uses the existing bundled runtime-dependency staging by default and reserves the opt-out for callers that explicitly need side-effect-free snapshots.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a packaged install without root workspace dependencies running local capability discovery such as `openclaw infer tts providers`, which reaches `listSpeechProviders(cfg)` and then the fallback registry load that currently skips bundled runtime-deps staging before loading Microsoft speech.

## Solution Assessment

Yes. The PR is the narrowest maintainable fix I found: it restores the loader default on ordinary fallback loads and keeps an explicit `installBundledRuntimeDeps: false` escape hatch for snapshot callers instead of broadening CLI preloading or adding provider-specific policy.
