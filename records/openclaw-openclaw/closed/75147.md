## Summary

Current main already has the requested non-preemptive behavior for normal inbound messages: active session runs use the queue system, default to steering instead of interrupting, and fall back to follow-up queueing rather than dropping the in-flight tool result. The reported release, v2026.4.27, predates that current-main behavior, so this should close as fixed on main but not yet as a shipped release fix.

## What This Changes

The issue reports that a new user message can preempt a long-running tool execution and cause the original tool result to be lost.

## Best Possible Solution

Keep the current main queue semantics: default active-run steering with follow-up fallback, while leaving `interrupt` as an explicit legacy opt-in for users who intentionally want abort-and-replace behavior.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the reported scenario shape: send a second message while a session run is active and executing tools; on current main the source path routes that message through steering/follow-up queueing instead of aborting the original run.

## Solution Assessment

Is this the best way to solve the issue? Yes. Session-level queueing is narrower and more maintainable than globally blocking all new user input, because it preserves ordering and delivery without dropping the current tool result.
