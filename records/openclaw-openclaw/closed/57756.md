## Summary

The reported session-key access pattern still matches current code, but it is documented as intentional under OpenClaw's one-trusted-operator Gateway model. Current security policy says `sessionKey` is routing/context selection, not per-user authorization, and deployments requiring mutually untrusted operator isolation should use separate gateways, hosts, and credentials rather than expecting per-device session tenancy inside one gateway.

## Best Possible Solution

Close this as outside the supported OpenClaw core trust model. Keep the current method-scope Gateway behavior and security docs; operators who need mixed-trust or per-user isolation should run separate gateways, hosts, and credentials. If maintainers later want per-device session tenancy, it should be opened as a deliberate product/security design proposal for a new Gateway authorization model.

## Risks / Open Questions

- A future maintainer product decision could add per-device or per-operator session tenancy, but that would be a new Gateway authorization model rather than a bug fix under the current documented trust model.
- Deployments that put mutually untrusted operators on one Gateway remain unsafe by design; the supported mitigation is isolation by gateway, host/OS user, and credentials.
