## Summary

Close as implemented on current main. Since the earlier keep-open review, Telegram partial streaming now uses durable message/edit/delete previews for DM lanes, sends a fresh final message for long-lived Working previews, records delivery failures/skips, and falls back when a turn ends without visible delivery; the #75073 change is in Unreleased and not in the latest release yet.

## What This Changes

The issue requests reliable Telegram DM delivery for a natural assistant final reply after long audio transcription, tool work, and subagent spawning with partial streaming enabled.

## Best Possible Solution

Keep the current main Telegram preview-delivery design: message-based preview edits for all DM streaming lanes, fresh final sends for long-lived previews, and explicit fallback/logging on delivery failure. If a recurrence appears on a build after this main snapshot, it should be handled as a new report with current gateway logs around Telegram send/edit/fallback diagnostics.

## Reproduction Assessment

No, there is not a high-confidence reproduction of the original incident because the production logs are gone and the synthetic test in the report used explicit message sends. Current main does have focused regression coverage for the relevant delivery mechanisms: DM preview transport, long-lived Working preview finalization, error-plus-final delivery, and fallback on skipped or failed finals.

## Solution Assessment

Yes. The current implementation is the narrow maintainable fix for this class: keep Telegram partial streaming on the normal message/edit/delete path and fall back to ordinary final delivery when old previews, edit failures, or no-visible-response states would otherwise risk silence.
