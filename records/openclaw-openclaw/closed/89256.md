## Summary

Close as implemented: current main and shipped v2026.5.28 already contain the same-lane owned transcript write publication path that covers the reported `sessionKey`-bound isolated cron takeover class.

## What This Changes

The issue reports `EmbeddedAttemptSessionTakeoverError` in isolated cron jobs that bind an explicit Telegram-derived `sessionKey`, asking whether that binding maps to the remaining same-lane embedded-session race.

## Best Possible Solution

Keep the shipped v2026.5.28 same-lane owned-write publication behavior; new reports should include fresh failures from v2026.5.28 or newer if this cluster persists.

## Reproduction Assessment

Yes, at source level: the reported class is the embedded prompt-lock fence firing when a session file changes while released without a trusted owned-write publication. I did not run a live six-agent cron cluster, but current tests cover the relevant same-process and external-interleave cases.

## Solution Assessment

Yes: recording trusted same-process transcript writes at the session-lock boundary is the maintainable fix because it preserves explicit cron `sessionKey` routing while still failing closed on external file edits.
