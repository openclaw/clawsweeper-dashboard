## Summary

Current `main` no longer starts Telegram's native approval/gateway client during plugin registration. The Telegram plugin exposes native approvals through a lazy runtime adapter, the gateway now bootstraps approval handlers separately from channel startup, and approval-handler startup retries instead of crashing the process. OpenClaw also shipped an explicit startup fix for native approval handlers in `v2026.4.22`.

## Evidence

- **Telegram approval runtime is lazy-loaded, not eagerly connected during plugin registration:** `extensions/telegram/src/approval-native.ts` wires `nativeRuntime` through `createLazyChannelApprovalNativeRuntimeAdapter(...)` with a deferred `import("./approval-handler.runtime.js")`, so the runtime is not constructed at plain plugin registration time.
  - file: [extensions/telegram/src/approval-native.ts:91](https://github.com/openclaw/openclaw/blob/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd/extensions/telegram/src/approval-native.ts#L91)

  - sha: [01bc49c88c0c](https://github.com/openclaw/openclaw/commit/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd)
- **Gateway starts approval bootstrap separately from channel account startup:** `startChannelApprovalHandlerBootstrap(...)` is invoked before `startAccount(...)`, and the actual channel runtime context is passed into the account task separately. This no longer matches the issue's claimed `plugin.register() -> immediate WebSocket connect` path.
  - file: [src/gateway/server-channels.ts:401](https://github.com/openclaw/openclaw/blob/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd/src/gateway/server-channels.ts#L401)

  - file: [src/infra/approval-handler-bootstrap.ts:19](https://github.com/openclaw/openclaw/blob/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd/src/infra/approval-handler-bootstrap.ts#L19)
