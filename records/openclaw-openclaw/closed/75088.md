## Summary

Keep open. Current main still has a hard-coded assistant-facing tooling summary that names Brave even though the managed web_search implementation already resolves provider-specific definitions and DuckDuckGo is a documented key-free fallback.

## What This Changes

The issue asks for the assistant-facing web_search description to reflect the active provider or use provider-neutral wording instead of always saying Brave API.

## Best Possible Solution

Make the Tooling section stop naming Brave for the generic web_search tool: either pass the resolved tool description into buildAgentSystemPrompt for web_search or change that built-in summary to provider-neutral wording, then add a focused prompt test proving DuckDuckGo/custom provider sessions do not mention Brave incorrectly.

## Reproduction Assessment

Yes. A static current-main check reproduces it: buildAgentSystemPrompt renders the hard-coded web_search Brave summary whenever web_search is in toolNames, while the actual tool definition path resolves DuckDuckGo-specific description, parameters, and provider output.

## Solution Assessment

Yes, the requested direction is the right fix boundary. The narrow maintainable solution is in prompt composition, not provider execution, because the provider resolution path already carries correct metadata.
