## Summary

This should stay open: current main still relies on the sqlite-vec loader path that can degrade vector writes when the bundled package is not resolved, and the report is concrete enough to treat as a narrow latest-release regression candidate.

## What This Changes

After upgrading to 2026.5.4, bundled sqlite-vec should still auto-load for gateway memory search even when the gateway cwd is outside the OpenClaw install directory.

## Best Possible Solution

Keep the issue open and fix the memory-core sqlite-vec resolver so packaged/global installs can find the bundled sqlite-vec loadable extension without requiring users to set extensionPath manually.

## Reproduction Assessment

No high-confidence live reproduction was established. The report includes a concrete environment, log, and workaround, and current source makes the failure path plausible because the bundled fallback depends on resolving sqlite-vec when extensionPath is unset.

## Solution Assessment

Yes, a narrow resolver/packaging fix is the right direction. Preserve explicit extensionPath precedence, then make the bundled sqlite-vec path resolve from the installed OpenClaw package/runtime location rather than requiring a user config workaround.
