## Summary

Keep this PR open, but it still needs changes before merge: the directReply path can abort after a callback that does not prove source-visible delivery, and the supplied proof is placeholder text rather than real runtime evidence.

## What This Changes

The PR adds a directReply tool-result flag with subscriber/runner abort handling, helper APIs, tests, changelog text, and several unrelated QA/test stability edits.

## Best Possible Solution

Define a typed, source-policy-aware direct-reply contract that aborts only after confirmed visible delivery or preserves a final fallback, with real gateway/channel proof for automatic and suppressed delivery modes.

## Reproduction Assessment

Yes, at source level: a directReply text result calls emitBlockReply and then aborts, while current main can suppress delivery or buffer text-only blocks. I did not run live proof because this was a read-only review.

## Solution Assessment

No. Callback invocation is not a reliable delivery boundary; the safer solution is a policy-aware delivery path that only aborts after visible delivery is confirmed or a fallback is preserved.
