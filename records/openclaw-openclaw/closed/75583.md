## Summary

Keep open. Current main still has only process-local Slack thread participation, and this PR is a plausible narrow implementation of restart-safe persistence through the existing SDK state API, but it is still open, protected by the maintainer label, has exact-head CI failures, and overlaps older Slack persistence PRs that need canonical maintainer handling.

## What This Changes

The PR adds best-effort SDK keyed-store persistence for Slack bot-participated thread markers, wires persisted lookup into Slack mention preparation, records route agent id on delivered replies, and updates Slack docs, tests, and the changelog.

## Best Possible Solution

Land one canonical Slack plugin implementation that keeps the in-memory cache as the hot path, uses the SDK keyed store for restart durability, preserves process-local fallback on persistent-store failure, documents the behavior, and then closes or supersedes the older overlapping Slack persistence PRs.

## Reproduction Assessment

Yes. On current main, record Slack thread participation, restart the Gateway so the process-local dedupe cache is lost, then send an inbound thread reply without a new mention; source inspection shows no persisted marker can restore bot_thread_participant.

## Solution Assessment

Yes for the code direction. Keeping the fix inside the Slack plugin and using the existing bundled-plugin SDK state API is narrower than adding core Slack behavior or a separate JSON file store; the remaining blockers are exact-head validation and canonical branch selection.
