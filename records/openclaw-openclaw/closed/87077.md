## Summary

Keep this PR open for maintainer review: the patch is a narrow and source-aligned fix for the reverse-proxy HTTP-auth handoff, but it intentionally changes Control UI top-level navigation from service-worker fallback behavior to browser-native handling.

## What This Changes

The PR adds an `event.request.mode === "navigate"` early return in `ui/public/sw.js` so top-level Control UI navigations bypass the service worker.

## Best Possible Solution

Merge this shape only if maintainers accept losing top-level offline shell fallback as the right tradeoff for reverse-proxy HTTP-auth compatibility; otherwise request a narrower design that preserves fallback behavior for non-auth navigation failures.

## Reproduction Assessment

Yes for source-level reproduction: current `ui/public/sw.js` sends navigations through `event.respondWith(fetch(...))`, matching the reported 401 path. I did not run a live reverse-proxy browser scenario in this read-only review, but the PR body supplies after-fix Firefox/Caddy evidence.

## Solution Assessment

Unclear without maintainer product judgment: the bypass is the narrowest code change for native HTTP-auth prompts, but it trades away the existing top-level offline shell fallback. A conditional auth-only path would be more complex and may not be worth it.
