## Label Justifications

- `P1`: The issue reports a broken CLI/Gateway workflow where completed agent responses are delayed or dropped for real users under Gateway event-loop starvation.
- `impact:message-loss`: The reported symptom is zero CLI stdout, delayed final responses, and sometimes dropped agent results despite successful Gateway completion logs.
- `impact:crash-loop`: The Gateway process stays alive but becomes effectively unavailable through severe event-loop delay, handshake timeouts, and response stalls.
- `impact:session-state`: The report describes turn responses and file artifacts arriving one turn late or not at all, which is session/turn state drift from the user's perspective.

## Summary

Close as a duplicate of the open canonical Gateway event-loop starvation tracker; this report adds useful CLI final-response delivery evidence, but the remaining root work is the same single-threaded Gateway starvation and WebSocket responsiveness problem already tracked by https://github.com/openclaw/openclaw/issues/78861.

## What This Changes

The issue reports that Gateway event-loop starvation under sustained agent runs delays or drops final WebSocket responses to `openclaw agent` CLI clients after the Gateway has logged successful completion.

## Best Possible Solution

Close this satellite after linking its CLI evidence to the canonical tracker, then solve the remaining work under https://github.com/openclaw/openclaw/issues/78861 with profiled scheduler, offload, and back-pressure changes that prove WS and CLI responsiveness.
