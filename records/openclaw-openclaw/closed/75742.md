## Summary

Close as duplicate/superseded. The new report is useful because it narrows the trigger to a normal over-1200px JPEG and the `imageMaxDimensionPx` workaround, but the remaining repository work is the same shared image optimizer failure already tracked by #73424 and its open implementation lane #74243.

## What This Changes

The issue asks the `image` tool to successfully downsample a valid JPEG above the default `imageMaxDimensionPx` value, or return an actionable error instead of `Failed to optimize image`.

## Best Possible Solution

Consolidate this evidence onto #73424 and finish #74243 or an equivalent `src/media/web-media.ts` fallback that keeps valid under-cap originals sendable when optimization fails, while preserving MIME metadata, HEIC conversion behavior, decoded-pixel guards, local path guards, and over-cap failures.

## Reproduction Assessment

Yes. The issue provides concrete steps, environment, dimensions, and logs, and current main still has a source-visible path where `loadWebMedia()` can propagate `Failed to optimize image` before the provider call.

## Solution Assessment

Yes. Closing this duplicate is the best maintainer cleanup path because #73424 is the canonical open tracker and #74243 is the existing implementation lane; the actual code fix should remain at the shared media loader boundary.
