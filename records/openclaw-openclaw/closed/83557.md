## Summary

Keep open. The narrowed `sessions_spawn` report is still not proven fixed on current `main`: source inspection shows the tool forwards `thinking`, but the gateway returns an accepted child run before the embedded agent startup/thinking validation can fail, so the reported accepted-with-no-work failure remains plausible and needs a targeted fix or live proof.

## What This Changes

The issue reports that model-facing ad-hoc `sessions_spawn` on OpenAI/OpenAI-Codex GPT models silently accepts but fails to run when `thinking` is not `off`.

## Best Possible Solution

Make native ad-hoc `sessions_spawn` either start a real child run with the requested supported OpenAI/Codex thinking level or return a visible structured startup/unsupported-level error before the parent treats the spawn as accepted.

## Reproduction Assessment

No high-confidence live reproduction was run in this review. The issue provides concrete live steps, and current source shows an accepted native spawn can return before embedded agent startup/thinking validation has succeeded, so the failure shape is source-reproducible but the exact OpenAI rejection branch still needs live or targeted simulated proof.

## Solution Assessment

Yes, the best direction is a narrow runtime fix around native `sessions_spawn`, `spawnSubagentDirect`, and gateway child startup so accepted spawns cannot hide immediate child-run failures. Keep it separate from the inherited-thinking-defaults work tracked by https://github.com/openclaw/openclaw/issues/55790.
