## Summary

Keep open: current main still exposes native plugin config from the baked `plugins.entries.<id>.config` path, while the requested env override and writable overlay behavior would add a new config contract and sandbox trust policy that needs maintainer direction.

## What This Changes

The issue requests an environment-variable override and/or writable overlay path for `plugins.entries.<id>.config` so plugin configuration can change in policy-locked sandboxes without rebuilding images.

## Best Possible Solution

Define one generic, schema-validated plugin config resolver that merges baked config with an approved writable overlay and explicit plugin-declared env mappings, then document precedence and locked-sandbox behavior.

## Reproduction Assessment

Yes for the OpenClaw side: current source inspection shows plugin config is resolved, validated, and exposed from `plugins.entries.<id>.config` without a declared env override or writable overlay path. The external sandbox filesystem denial was not live-run here.

## Solution Assessment

Unclear for the exact implementation: env-only support may not cover runtime reconfiguration when sandbox env is fixed, while overlay-only misses standard ecosystem env conventions. A generic resolver with documented precedence looks maintainable, but it needs maintainer approval first.
