## Summary

Close as implemented on current main. The later #71120 fix path now treats 429 responses without usable retry timing as terminal for the SDK, caps implicit idle watchdogs below long run timeouts, and shipped in v2026.4.29; the reported affected version was 2026.4.27.

## What This Changes

The issue asks OpenClaw to surface GitHub Copilot `/responses` HTTP 429 quota errors promptly as provider rate-limit failures instead of classifying them as two 120-second idle-timeout retries.

## Best Possible Solution

Keep the shipped v2026.4.29+ generic transport behavior: terminal Copilot 429s should bypass long SDK retries and enter normal provider failover/error handling, while any fresh reproduction on a newer version should be opened with updated trajectory and installed-version evidence.

## Reproduction Assessment

Yes. The issue provides a concrete exhausted-Copilot-quota path and direct curl proof for the upstream 429; I did not live-run it because it requires an actively rate-limited Copilot token, but current source and focused tests cover the reported failure class.

## Solution Assessment

Yes. The implemented fix is the maintainable boundary: generic provider transport handles SDK retry suppression for terminal 429s, while the embedded runner owns idle-timeout caps rather than adding Copilot-specific timeout policy in core.
