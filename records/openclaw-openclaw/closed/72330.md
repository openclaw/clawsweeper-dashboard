## Summary

Current main already handles the reported Matrix bundled-plugin staging failure. The runtime mirror code now deletes the existing destination before renaming the staged plugin directory, under the bundled runtime-deps filesystem lock, so a stale non-empty `dist/extensions/matrix` mirror should no longer produce the Linux `ENOTEMPTY` rename loop. The adjacent Matrix native crypto bootstrap and Windows drive-letter import issue also have current source and test coverage. No newer release tag than `v2026.4.24` is present in this checkout, so the fix is implemented on main but no fixed release can be cited here.

## Best Possible Solution

Close as implemented on current main. Keep the shipped main behavior: external bundled-plugin runtime mirrors should replace stale destinations under a lock, Matrix should bootstrap the native crypto binding when ignored lifecycle scripts leave it missing, and Windows absolute module paths should continue to be converted to file URLs before loading. If a user can still reproduce this after installing a build that includes `edf40ab6c9440604fe6a74aeaa8d8bc81b5ee16b`, the right follow-up would be a fresh issue with the new gateway log, package version, stage directory, and whether the failure is from `rm`, `rename`, or module load.

## Evidence

- **Runtime mirror removes stale destination before rename:** Bundled plugin loading now copies into a private `.plugin-<id>-*/plugin` staging directory, removes the existing `mirrorRoot` recursively, then renames the staged root into place. This directly addresses the reported `rename(.../.plugin-matrix-*/plugin -> .../extensions/matrix)` ENOTEMPTY condition.
