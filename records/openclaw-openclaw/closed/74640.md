## Summary

Current main already guards the reported WebChat session loss path: config rewrites are handled through config-only write helpers, gateway config restarts do not create an agent continuation from a generic sessionKey, and explicit session rotation is admin-gated with regression coverage that preserves the existing main sessionId for write-scoped callers.

## What This Changes

Preserve the active WebChat `agent:main:main` session pointer and transcript history across `openclaw.json` rewrites from onboarding, daemon installation, and config commands.

## Best Possible Solution

Keep config persistence, restart notices, and explicit session resets separate. The current main behavior is the right end state: config rewrites can wake or notify the existing session, but they should not rotate `agent:main:main` or archive its transcript unless an authorized reset path is invoked.

## Reproduction Assessment

Yes, at the source/test level. The issue provides a concrete WebChat plus config rewrite sequence, and current main has focused regression tests for the dangerous reset paths: config.patch with a sessionKey does not create a continuation, write-scoped `/reset` through agent is rejected, and chat.send `/reset` preserves the existing main sessionId.

## Solution Assessment

Yes. The current implementation solves this at the maintainable boundary by keeping session rotation behind the explicit admin reset path while leaving config writes and restart sentinel wakes session-neutral.
