## Summary

This older PR should close because current main and v2026.6.6 already cover the central send-attachment problem with a different, better-owned implementation, while the remaining `preloadSendMedia` API is unproven, compatibility-sensitive, and not a useful stale branch to keep open.

## What This Changes

The PR adds an opt-in plugin action adapter flag so `send` actions with media hints can preload media into `buffer`, `contentType`, and `filename`, plus regression tests for opted-in and non-opted-in plugin send handlers.

## Best Possible Solution

Keep the shipped buffer-to-media implementation and current mediaAccess/prepareSendPayload plugin model; only revisit a send-media preload API through a fresh, documented PR with a real plugin need and proof.
