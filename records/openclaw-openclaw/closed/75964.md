## Summary

Keep open. Current `main` still records a recurring `wakeMode: "now"` main-session cron job as `ok` after one retryable busy heartbeat skip and a fire-and-forget wake, so the reported ghost-run path remains valid.

## What This Changes

Recurring main-session cron jobs should not record a successful run before their queued `systemEvent` is actually processed when the initial heartbeat is skipped for temporary lane pressure.

## Best Possible Solution

Retry recurring wake-now main cron jobs for temporary `requests-in-flight` and `lanes-busy` skips within the existing bounded wait, keep immediate deferral for `cron-in-progress`, and update regression coverage plus changelog.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists by running a recurring main-session `wakeMode: "now"` systemEvent cron with `runHeartbeatOnce` mocked to `requests-in-flight`; current coverage already asserts the short `ok` ghost-shaped outcome.

## Solution Assessment

Yes, with the per-reason boundary preserved. The narrow maintainable fix is to remove the recurring fast exit for temporary lane pressure while retaining the existing `cron-in-progress` defer path and max-wait fallback.
