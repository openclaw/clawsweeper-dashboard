## Label Justifications

- `P2`: This is a bounded but real TUI workflow gap for long-running agent turns rather than an emergency or release blocker.
- `impact:message-loss`: The TUI suppresses follow-up prompts before they can enter OpenClaw's message queue path.

## Summary

Keep open: current main and the latest release still block normal TUI submits before they reach OpenClaw queue handling, and the related closed queue/lane issues do not cover this pre-submit TUI policy gap.

## What This Changes

The issue asks whether the TUI should allow prompts to be submitted while an agent run is active so existing queue modes can steer, defer, collect, or interrupt them.

## Best Possible Solution

Define the intended TUI busy-submit policy, then route normal mid-run TUI submits into the existing Gateway queue modes with visible queued state while preserving Esc abort semantics.

## Reproduction Assessment

Yes from source: the current TUI submit and command-handler paths reject normal messages while `activeChatRunId` is set, and tests assert the busy message. I did not run a live TUI session because this review is read-only.

## Solution Assessment

Unclear until maintainers choose the policy: queue docs support active-run message handling after ingress, but TUI tests intentionally preserve the busy lock. The maintainable path is to decide the TUI default and queued-state UX before changing code.
