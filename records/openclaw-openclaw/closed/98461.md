## Summary

Current main and the latest release already expose live tool activity to Control-UI/WebChat before the final transcript message: Gateway emits run-scoped `agent` tool events and session-scoped `session.tool` events, and the UI turns those frames into live pending tool cards.

## What This Changes

The issue requests an additive Gateway WebSocket progress surface so Control-UI/WebChat users see tool-call activity before the completed assistant turn arrives.

## Best Possible Solution

Keep the shipped additive `tool-events`/`session.tool` WebSocket path for Control-UI/WebChat; handle the separate OpenResponses HTTP/SSE early-progress discussion in its existing related issue and PR.

## Solution Assessment
