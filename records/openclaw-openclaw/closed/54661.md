## Summary

Current main already blocks heartbeat/cron/exec system events from updating session-origin metadata, which is the path that would have changed the webchat main-session display name to values like `heartbeat`. The same guard is present in the shipped `v2026.4.22` release.

## Evidence

- **System-event origin metadata is explicitly skipped:** `deriveSessionOrigin()` returns early for `heartbeat`, `cron-event`, and `exec-event` when `skipSystemEventOrigin` is set, so those events do not contribute a new origin label that could become the session display name.
  - file: [src/config/sessions/metadata.ts:54](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/config/sessions/metadata.ts#L54)
  - command: `git show v2026.4.22:src/config/sessions/metadata.ts | sed -n '54,63p'`
  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)
- **Main reply/session path enables that guard for system events:** The session persistence path calls `deriveSessionMetaPatch(..., skipSystemEventOrigin: isSystemEvent)`, so heartbeat-triggered main-session turns do not rewrite session origin/display metadata.
  - file: [src/auto-reply/reply/session.ts:672](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/auto-reply/reply/session.ts#L672)
