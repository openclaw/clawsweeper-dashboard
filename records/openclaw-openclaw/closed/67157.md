## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve the existing CLI bootstrap:** `scripts/openclaw-runcli-launcher.mjs:15-21`
  - body: The new npm bin imports `runCli` directly, bypassing the existing `openclaw.mjs`/`src/entry.ts` bootstrap that handles respawn planning, env normalization, warning filtering, compile cache setup, profile/container parsing, and fast paths before loading the CLI. Installed commands would lose those startup contracts.
  - confidence: 0.91
- **[P1] Load the packaged run-main entry:** `scripts/openclaw-runcli-launcher.mjs:9`
  - body: The launcher scans the root `dist` directory for `run-main-*.js`, but current build metadata and guards use `dist/cli/run-main.js`. In a normal package layout this can throw `missing run-main entry` before the CLI starts.
  - confidence: 0.86
- **[P1] Align release validation with the bin change:** `package.json:17`
  - body: Changing `bin.openclaw` here disagrees with `collectReleasePackageMetadataErrors`, which still requires `openclaw.mjs`. The npm release preflight will fail until the validator and matching tests are updated for any final entrypoint change.
  - confidence: 0.96
- **[P2] Recognize the new launcher in gateway argv checks:** `package.json:17`
  - body: Switching npm shims to this script can expose `scripts/openclaw-runcli-launcher.mjs` in process command lines, but `isGatewayArgv` does not recognize that path. Gateway lock/liveness checks can then misclassify a running gateway.
  - confidence: 0.84
- **[P2] Run checks through the validated bin path:** `scripts/local-upgrade-from-tarball.sh:129-134`
  - body: The script verifies `$BIN_PATH`, but then runs `openclaw --version` and the live regression through the default PATH command. If another OpenClaw binary appears earlier on PATH, this can validate the wrong install.
  - confidence: 0.9
- **[P2] Await the live smoke promise:** `scripts/test-live-memory-pro-cli-host.mjs:270-271`
  - body: `main()` is async, but the top-level `try/catch` calls it without `await`, so async failures bypass the structured JSON error handler and surface as unhandled rejections.
  - confidence: 0.95
- **[P2] Prefer stdout for stats JSON parsing:** `scripts/test-live-memory-pro-cli-host.mjs:84`
  - body: `combined` puts stderr before stdout, and `extractFirstJsonObject` parses the first `{` it sees. A diagnostic object on stderr can be parsed instead of the `stats --json` stdout payload, making the smoke validate the wrong JSON.
  - confidence: 0.82
- **[P3] Use npm pack output instead of parsing ls:** `scripts/local-upgrade-from-tarball.sh:76-77`
  - body: `ls -t ... | head -n 1` is fragile for unusual filenames and produces a poor failure mode when no tarball matches. `npm pack --json` or the command output should provide the exact created tarball path.
  - confidence: 0.78
