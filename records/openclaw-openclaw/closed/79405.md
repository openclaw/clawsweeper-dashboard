## Summary

Keep open: the PR targets a real subagent completion-delivery failure class, but the current branch still introduces blocking regressions in normal completion delivery and does not wire its fallback flag through the actual direct-delivery helper.

## What This Changes

The branch adds taskId-aware subagent completion context, direct completion fallback delivery/statuses, lifecycle retry/give-up handling, and related tests.

## Best Possible Solution

Preserve the requester-agent completion contract, keep fallback behavior scoped to terminal give-up paths, forward fallback enablement through the actual delivery boundary, and align this patch with the active durable-delivery work.

## Reproduction Assessment

Yes, at source level. PR head shows task-aware lookup can stop existing taskId-free completion delivery, and the fallback flag is accepted by the public delivery function but not forwarded to the helper that gates fallback sends.

## Solution Assessment

No. The intended hardening is reasonable, but this branch is not the narrow maintainable fix until it preserves normal completion delivery and wires the fallback path through the real send boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Keep task lookup out of normal completion delivery:** `src/agents/subagent-announce.ts:556-570`
  - body: This lookup now runs for every non-nested completion and returns false when taskId or task delivery context is missing. Existing direct completion callers already pass a requester origin without a task id, so they would stop before the normal announce path instead of only using task context for terminal give-up fallback.
  - confidence: 0.92
- **[P1] Forward the fallback flag to direct delivery:** `src/agents/subagent-announce-delivery.ts:1177-1194`
  - body: deliverSubagentAnnouncement accepts allowCompletionFallback, but the direct delivery call never passes it to sendSubagentAnnounceDirectly. That helper gates fallback sends on params.allowCompletionFallback === true, so the terminal give-up retry path still cannot send the fallback this PR is meant to add.
  - confidence: 0.95
