## Label Justifications

- `P1`: The report describes a latest-release gateway provider workflow that makes Azure Foundry DeepSeek V4 deployments unusable for affected users.
- `impact:auth-provider`: The failure is in provider/model routing and request payload compatibility for an Azure Foundry model provider configuration.

## Summary

Keep open: current main and the latest release still have a generic DeepSeek V4 post-plugin wrapper that can match an OpenAI-compatible model by id and inject `thinking` without consulting explicit `reasoning` or `compat.thinkingFormat` overrides. The reported Azure Foundry/custom-provider path is a real source-level bug, not a core-scope close candidate.

## What This Changes

The issue asks the gateway DeepSeek V4 OpenAI-compatible thinking fallback to respect per-model `reasoning: false` and `compat.thinkingFormat` overrides for Azure Foundry/custom provider deployments.

## Best Possible Solution

Narrow the generic DeepSeek V4 fallback so explicit model compatibility wins: skip DeepSeek-style `thinking` injection when `reasoning` is false or `compat.thinkingFormat` is explicitly non-DeepSeek, while leaving provider-owned DeepSeek/OpenRouter/proxy replay support intact.

## Reproduction Assessment

Yes, there is a high-confidence source reproduction: an OpenAI-compatible model whose normalized id is `deepseek-v4-flash` or `deepseek-v4-pro` reaches the generic post-plugin wrapper, which then writes `thinking` without checking `reasoning` or `compat.thinkingFormat`. I did not run a live Azure Foundry call.

## Solution Assessment

Yes, the best fix is a narrow compatibility gate in the generic fallback plus a regression test for the reported Azure Foundry/custom-provider config. A new `suppressThinking` config option is not needed unless existing `reasoning` and `thinkingFormat` metadata cannot express the provider contract.
