## Summary

Current `main` already releases session-lane work on error. Embedded runs are queued per session, and the queue removes the active task and pumps the lane in both success and error paths; the same implementation is present in release `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`).

## Evidence

- **Embedded runs are serialized through a per-session lane:** `runEmbeddedPiAgent()` resolves a session lane and enqueues the run through `enqueueCommandInLane(sessionLane, ...)`, so the reported stuck state would come from queue bookkeeping rather than a separate lock.
  - file: [src/agents/pi-embedded-runner/run.ts:249](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/agents/pi-embedded-runner/run.ts#L249)
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Lane error path clears the active task and resumes draining:** In `enqueueCommandInLane()`, the async worker calls `completeTask(...)` inside both the success path and the `catch` path, and the error path also calls `pump()` before rejecting. That means an API error cannot leave the session lane permanently occupied once the task unwinds.
  - file: [src/process/command-queue.ts:188](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/process/command-queue.ts#L188)
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Embedded-run state is cleared before terminal lifecycle emission and again in final cleanup:** The runner clears active embedded-run state in `onBeforeLifecycleTerminal` and again in the surrounding `finally`, covering error endings and downstream notification failures.
  - file: [src/agents/pi-embedded-runner/run/attempt.ts:1848](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/agents/pi-embedded-runner/run/attempt.ts#L1848)
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Latest release already contains the same queue error cleanup:** `git show 00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5:src/process/command-queue.ts` shows the same error-path `completeTask(...); ... pump(); entry.reject(err);` implementation in release `v2026.4.22`.
  - file: [src/process/command-queue.ts:199](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/process/command-queue.ts#L199)
