## Summary

Current main addresses the reported hot path: bundled runtime-deps staging is now legacy cleanup-only, Gateway startup publishes a reusable plugin metadata snapshot, live tool/capability lookups can use that snapshot instead of rebuilding manifest registries, and manifest parsing has a file-signature cache.

## What This Changes

Avoid severe Gateway stalls by preventing repeated plugin-runtime-deps scans and repeated openclaw.plugin.json manifest reads on the main thread.

## Best Possible Solution

Keep the current manifest-snapshot and file-signature-cache approach, and leave remaining legacy plugin-runtime-deps cleanup in explicit doctor/update repair paths rather than Gateway startup.

## Reproduction Assessment

Yes. The report gives a concrete v2026.4.29 Pi 4 reproduction with Gateway logs, event-loop metrics, strace counts, and repeated manifest reread observations; this read-only review verified the current-main source paths that remove those hot-path causes.

## Solution Assessment

Yes. The current implementation is the maintainable fix because it passes explicit metadata snapshots through Gateway/runtime flows, caches only unchanged manifest file parses, and treats old runtime-deps trees as repairable legacy state instead of adding hidden broad discovery caches.
