## Summary

Keep open: current main still has the Assistant `message_changed` no-sender early return, and DM thread replies still route through thread-scoped Slack sessions. The report is credible, but the safest fix needs Slack routing and Assistant App ownership decisions before changing documented session behavior.

## What This Changes

The issue reports that Slack DM thread replies and Assistant App `assistant_app_thread` `message_changed` events can arrive over Socket Mode without waking the agent session.

## Best Possible Solution

Split the work into an owner-approved Slack DM thread routing decision and a contract-safe Assistant App `message_changed` handler that dispatches valid human edits and logs rejected events clearly.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partly: the Assistant no-metadata drop is reproducible from current source and tests, and DM thread replies are provably thread-scoped, but the exact Case 1 silent drop still needs live Slack logs or a focused fixture beyond routing inspection.

## Solution Assessment

Is this the best way to solve the issue? Unclear: the Slack plugin is the right surface, but flattening DM threads or dispatching to both sessions changes current documented behavior; the Assistant fallback should be designed against Slack's event contract and sender authorization rules.
