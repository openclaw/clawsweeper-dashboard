## Summary

Keep open. Current main and the latest shipped release still lack a Discord-owned IPv4/DNS fallback in the REST and gateway WebSocket paths, and open PR #77449 is already the implementation candidate for this issue.

## What This Changes

Request that Discord inbound REST and gateway networking continue to work on IPv4-only Windows networks after the 2026.5.x undici transport change.

## Best Possible Solution

Land a Discord plugin-owned transport fix that applies IPv4-preferred DNS lookup to REST metadata and gateway WebSocket paths, then let the linked fixing PR close the issue.

## Reproduction Assessment

Yes. The issue gives two Windows IPv4-only reproductions plus a working `--dns-result-order=ipv4first` workaround, and current-main source inspection shows the default Discord REST and gateway WebSocket paths still lack owner-local DNS fallback handling.

## Solution Assessment

Yes. The narrowest maintainable direction is the owner-local Discord transport approach in PR #77449; setting process-wide `NODE_OPTIONS` from the launcher would be broader than this Discord-specific regression needs.
