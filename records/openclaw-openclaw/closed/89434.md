## Summary

This should stay open: current main and the latest release still have the reported WebRTC Talk startup/teardown race, and the requested fix is a narrow Control UI stability repair.

## What This Changes

The issue reports a Control UI Realtime Talk WebRTC race where startup can call `addTrack` after `stop()` has cleared the peer connection.

## Best Possible Solution

Make WebRTC Talk startup cancellation-safe across awaited setup steps, clean up resources once, and add a focused regression test for `stop()` while startup is pending.

## Reproduction Assessment

Yes, by source inspection: `stop()` can null `this.peer` while `start()` is suspended at `getUserMedia`, then `start()` resumes and calls `this.peer.addTrack(...)`. I did not execute a browser repro in this read-only pass.

## Solution Assessment

No, the proposed one-line `if (this.peer)` guard is only a partial fix. The safer solution is to guard the whole async startup sequence after awaits and avoid continuing setup after cancellation.
