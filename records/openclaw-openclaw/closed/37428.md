## Summary

Keep open. Current `main` mitigates the startup race by draining queued WhatsApp deliveries again after the listener connects, but it does not implement the core requested behavior: startup recovery still records `No active WhatsApp Web listener` as a failed attempt and increments `retryCount`, and there is still no delayed second startup recovery pass in gateway startup.

## Evidence

- **Startup recovery still counts transient readiness failures:** `drainQueuedEntry` sends failed non-permanent recovery attempts through `failDelivery(...)`, so a `No active WhatsApp Web listener` error still becomes a counted retry during startup recovery.
  - file: [src/infra/outbound/delivery-queue-recovery.ts:221](https://github.com/openclaw/openclaw/blob/7d471837369885c12d540d6bafa319ff41129b89/src/infra/outbound/delivery-queue-recovery.ts#L221)

- **`failDelivery` unconditionally increments retryCount:** The queue storage path increments `entry.retryCount += 1` for every failed delivery attempt; there is no readiness-specific deferral path here.
