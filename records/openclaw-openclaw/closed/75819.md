## Summary

Keep open. Current main still lacks the no-marker PPID ancestry guard, the related merged service-marker fix is distinct, and maintainer discussion explicitly says this PR should remain open after rebase for the active-Gateway ancestry path.

## What This Changes

The PR exports the existing PID ancestry walker, uses it in `openclaw update` to refuse restartful package-manager updates when the active Gateway PID is in the caller ancestry, adds a CLI regression test, and updates the changelog.

## Best Possible Solution

Land the narrow ancestry guard after normal maintainer review and exact-head CI, preserving the merged service-marker guard and the changelog entry.

## Reproduction Assessment

Yes. #75691 and the PR comments give a concrete systemd plus Gateway-owned exec path, and current main source still stops/updates based on service state without checking whether the active Gateway PID is in the caller ancestry.

## Solution Assessment

Yes. Reusing the existing bounded ancestry walker and refusing before service stop/package replacement is the narrow maintainable fix for this failure mode; a detached restart helper would be a larger separate design.
