## Summary

Keep open. Current main has meaningful App SDK progress, but it still does not provide the requested public, platform-agnostic reusable gateway WebSocket client package; the in-tree SDK is private and Node-built, and the Control UI still owns a separate browser client implementation.

## What This Changes

The issue requests a reusable `@openclaw/gateway-client`-style package with shared protocol types, request/event handling, reconnection, authentication, and pluggable Node/browser/RN transports for third-party consumers and in-tree clients.

## Best Possible Solution

Define and document a supported public gateway client contract, either by publishing `@openclaw/sdk` as the official surface or splitting a lower-level adapterized gateway client package with Node/browser/RN support and parity coverage for existing clients.

## Reproduction Assessment

Not applicable as a feature/API request. Source inspection is enough to verify the current state: related SDK work exists, but the requested public platform-agnostic gateway-client package is not complete on current main.

## Solution Assessment

Unclear. The current App SDK direction is a strong partial answer for Node/TypeScript consumers, but closing this as solved would skip the package publication, low-level generated client, platform adapter, and Control UI drift concerns raised in the request.
