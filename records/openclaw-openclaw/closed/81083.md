## Summary

Keep open: current main still has the unknown-command help/version asymmetry, and this PR contains the right central fix, but the new guard reads config before cheap command-root gating and can make unrelated help/version paths config-dependent.

## What This Changes

The PR changes CLI unknown-root detection so unknown commands followed by `--help` or `--version` throw the existing unknown-command error and adds regression tests.

## Best Possible Solution

Keep the unknown-root rejection, but make the config lookup lazy so root version and known help/version paths stay config-independent, with regression tests for both behaviors.

## Reproduction Assessment

Yes. Current-main source shows the unknown-primary check explicitly bails on help/version and its only existing caller is skipped for help/version, matching the linked issue and PR terminal proof; I did not rerun commands because this review is read-only.

## Solution Assessment

No, not as submitted. The core unknown-root fix is narrow, but the implementation should avoid loading config for root version and known help/version paths that do not need plugin-root ownership checks.
