## Summary

Keep open. Current `main` does route a received transcript into the call manager and auto-response path, but the OpenAI realtime transcription provider still only emits a final transcript on `conversation.item.input_audio_transcription.completed`, while the media-stream handler closes the STT session immediately on Twilio stream stop. That leaves the reported `speech_stopped` -> stream stop ordering plausibly unfixed, and there is no regression test covering that timing.

## Evidence

- **Transcript delivery path exists only after a transcript callback fires:** `VoiceCallWebhookServer` logs the transcript, creates a `call.speech` event, calls `manager.processEvent(event)`, and then invokes `handleInboundResponse(...)` for inbound/conversation calls. This shows the manager/session path is present once `onTranscript` is reached.
