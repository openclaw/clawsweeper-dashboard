## Summary

Current main already contains a targeted fix for channel-registry eviction during plugin loading, and the shipped release `v2026.4.22` includes it. The code now pins the startup channel registry so later plugin-registry replacements cannot silently drop Telegram, with regression tests covering config-schema/plugin-status registry swaps and startup-registry churn.

## Evidence

- **Channel registry is explicitly pinned at startup:** `pinActivePluginChannelRegistry` was added with a comment stating it prevents later `setActivePluginRegistry` calls, config-schema reads, and other non-primary registry loads from evicting channel plugins.
  - file: [src/plugins/runtime.ts:151](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/plugins/runtime.ts#L151)
  - command: `nl -ba src/plugins/runtime.ts | sed -n '145,175p'`
  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)
- **Gateway startup and deferred reload both preserve the pinned channel registry:** `loadGatewayStartupPlugins(...)` and `reloadDeferredGatewayPlugins(...)` both pass `beforePrimeRegistry: pinActivePluginChannelRegistry`, so startup channel plugins stay stable even if the active registry changes later.
  - file: [src/gateway/server-plugin-bootstrap.ts:98](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/gateway/server-plugin-bootstrap.ts#L98)
