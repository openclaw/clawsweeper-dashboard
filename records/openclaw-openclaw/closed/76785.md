## Summary

Current main already fixes the managed npm plugin-root pruning path reported here: installs now preserve existing managed plugin dependencies and regression coverage asserts both sibling plugins remain after a second install. The reporter's follow-up diagnosis matches the old failure mode, but the fix is present on main and is not yet proven shipped in a release tag.

## What This Changes

The issue reports that installing a second npm plugin in v2026.5.2 prunes the previously installed npm plugin from the shared managed root.

## Best Possible Solution

Keep the managed npm-root manifest implementation and ship it in the next release so stable users get the sibling-plugin preservation fix.

## Reproduction Assessment

Yes for the reported v2026.5.2 behavior from the issue logs and installer diagnosis; for current main, source inspection and regression coverage show the second install preserves both managed npm plugin packages.

## Solution Assessment

Yes. Preserving npm's top-level dependency set through a managed package.json is the narrow maintainable fix for npm pruning extraneous sibling plugins.
