## Summary

The reported Moonshot duplicate tool-call ID failure is already fixed on current main and is present in the v2026.5.19 release tree; the stream wrapper now rewrites duplicate IDs within a single assistant message before tool execution/replay.

## What This Changes

Issue reports deterministic Moonshot 400s when multi-tool turns send duplicate `tool_call.id` values such as `exec:2`.

## Best Possible Solution

Keep the shipped stream-level duplicate-ID repair that preserves native Kimi IDs, and ask users on 2026.5.18 to upgrade to v2026.5.19 or newer.

## Reproduction Assessment

No current-main failing repro remains: the historical 2026.5.18 CLI prompt and log are credible, but current source and v2026.5.19 regression coverage show duplicate IDs are rewritten before follow-up requests.

## Solution Assessment

Yes: normalizing duplicate live-output IDs at the stream boundary fixes the malformed tool turn without enabling broad Moonshot ID sanitization that would break native Kimi IDs.
