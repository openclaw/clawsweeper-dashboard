## Summary

Keep open: current main and v2026.5.12 still send Ollama native thinking control as top-level `think`, and the issue is now paired with an open same-author PR that attempts a guard for non-reasoning models. The suggested blanket move to `options.think` conflicts with current OpenClaw docs/tests and the upstream Ollama chat contract, so this needs maintainer resolution rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to stop sending Ollama `think` as a top-level chat request field for non-thinking models and route it through `options` instead.

## Best Possible Solution

Keep top-level Ollama `think` for reasoning-capable native models, then land a narrow maintainer-approved guard, warning, or validation path so non-reasoning models do not receive truthy native thinking control.

## Reproduction Assessment

Yes, for a narrowed source path: current main can put a truthy configured `params.think`/`params.thinking` or runtime thinking value on the top-level Ollama chat payload, and upstream Ollama rejects truthy `req.Think` for non-thinking models. The exact reported gateway default route still needs live confirmation because current OpenClaw also validates supported thinking levels.

## Solution Assessment

No, the blanket `options.think` fix is not the best supported direction because it conflicts with current OpenClaw tests/docs and upstream Ollama’s top-level chat contract. A capability-aware guard or explicit validation is the safer product path, and the open paired PR should be reviewed with that boundary in mind.
