## Summary

Close as implemented on current main: the DeepSeek lightweight provider-policy surface now exposes the same V4 thinking profile used by the runtime provider hook, so fallback/session-row/UI paths can see `xhigh` and `max`.

## What This Changes

The issue reports that the Control UI `/think` picker for native DeepSeek V4 sessions omits `xhigh` and `max` despite docs and changelog promising those levels.

## Best Possible Solution

Keep the current plugin-owned implementation, where DeepSeek V4 thinking levels are defined once and exposed through both runtime and lightweight provider-policy paths.

## Reproduction Assessment

No for current main: the reported 2026.5.2 fallback gap is source-explained, but current main now returns the DeepSeek V4 profile through the provider-policy artifact. I did not run a live browser session because this was a read-only review and local `pnpm` is unavailable.

## Solution Assessment

Yes: the implemented path is the narrow maintainable fix because the Control UI already consumes gateway/provider thinking levels, and DeepSeek-specific behavior belongs in the DeepSeek plugin policy artifact rather than UI regex logic.
