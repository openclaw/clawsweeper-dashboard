## Summary

Keep open: the Discord button delivery goal is useful and the contributor supplied real proof, but the branch still adds a Discord-native `components` field to the shared Gateway `send` API, which conflicts with the current presentation/channel-owned payload boundary and needs maintainer API direction before merge.

## What This Changes

The PR adds top-level Gateway `send.components`, forwards it through Discord raw component delivery/runtime send deps, re-exports `ChannelOutboundContext`, regenerates Swift models, and adds a changelog entry.

## Best Possible Solution

Keep the Discord button delivery goal, but expose it through a maintainer-approved channel-owned Gateway input or the semantic `presentation` contract rather than a Discord-native top-level `send` field.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the source-level gap: current main rejects a top-level `components` field because `SendParamsSchema` is closed and Ajv does not remove additional properties; the contributor also supplied live after-fix Discord proof for the desired behavior.

## Solution Assessment

Is this the best way to solve the issue? No. The end-to-end Discord behavior is useful, but the proposed top-level shared `send.components` field is not the narrowest maintainable API boundary given the existing `channelData.discord` and semantic `presentation` contracts.
