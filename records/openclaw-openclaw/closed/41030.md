## Summary

PR #41030 is superseded by the merged/shipped channel-health redesign in #69833. Current main solves the quiet-channel false-positive problem by separating provider-proven transport liveness from app-level inbound events, while still treating stale active-run heartbeats as a real stuck-run recovery signal. The PR's fresh-lastEventAt guard would now conflict with that design and with the availability concern raised in its discussion.

## Best Possible Solution

Close PR #41030 as superseded by #69833/v2026.4.22. Keep the current design: provider-proven transport activity handles quiet connected sockets, and active-run heartbeats determine whether a busy run is still making progress. Any future tuning of stale active-run restarts should be a new focused issue using a non-user-controlled progress signal or an explicit hard-timeout policy.
