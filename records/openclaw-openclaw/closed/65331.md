## Summary

Keep open. The bug is still present on current main, but this PR is now stale against the refactored runtime lookup path: current `resolveRuntimeTextTransforms()` delegates to `getActiveRuntimePluginRegistry()` in `active-runtime-registry.ts`, while the PR edits the older in-file loader layout from its base branch.

## What This Changes

The PR imports `getActivePluginRegistry()` in `src/plugins/text-transforms.runtime.ts` and uses it as the primary registry lookup before the existing require-based fallback for plugin text transforms.

## Best Possible Solution

Adapt this PR or open a narrow replacement that updates the current active-runtime registry seam so bundled main-process code can read `getActivePluginRegistry()` while retaining any needed external-plugin fallback, then add focused regression coverage for runtime text-transform resolution through CLI and provider consumers.

## Reproduction Assessment

Yes. The high-confidence source path is a plugin that registers `textTransforms`, followed by CLI/provider prompt resolution on current main where `resolveRuntimeTextTransforms()` reads through a require-probing helper that can return null before consumers apply replacements.

## Solution Assessment

No, not as currently proposed. The direct active-registry lookup is likely the right fix direction, but the patch is against the old file layout and should be rebased onto `active-runtime-registry.ts` or the current resolver with tests.
