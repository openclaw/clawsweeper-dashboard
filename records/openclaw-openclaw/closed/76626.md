## Summary

Keep open. The report matches a current source-level split: CLI web search uses runtime provider discovery, while the agent `web_search` path can force bundled-only resolution when runtime web-search metadata is absent or unhelpful, which can hide an installed external Brave provider even though CLI discovery reports it configured.

## What This Changes

The issue requests that agent `web_search` use the same configured Brave provider that works through the CLI capability web-search path.

## Best Possible Solution

Align agent `web_search` provider resolution with CLI/runtime discovery for configured installed or external web-search providers, while preserving the bundled fast path only when a bundled provider is definitively selected.

## Reproduction Assessment

Yes, at source level. With an installed external Brave provider and absent or `none` runtime web-search metadata, `createWebSearchTool` can pass `preferRuntimeProviders: false`, causing `runWebSearch` to resolve only bundled candidates even though the CLI path defaults to runtime provider discovery.

## Solution Assessment

No current-main solution is present. The narrow maintainable fix is to make the agent path use runtime/installed provider discovery whenever the configured provider is external or plugin-scoped web-search config requires it, without adding a new config knob or hardcoding Brave in core.
