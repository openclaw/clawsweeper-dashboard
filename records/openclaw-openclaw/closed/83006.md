## Summary

Keep open: this is a protected, active feature PR and current main does not have the proposed authoring surface; the latest head still leaves concrete authoring-command validation and security defects to repair before merge.

## What This Changes

The PR adds a `defineToolPlugin` SDK helper, `openclaw plugins init/build/validate` commands, docs/tests/API exports, and migrates the bundled `llm-task` plugin to the helper.

## Best Possible Solution

Land a repaired version that keeps the SDK/CLI direction but hardens root-boundary checks, uses the normal manifest parser contract, generates publishable scaffold metadata, and validates the exact single runtime entry.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a current-main bug, but yes for the review blockers: the source paths are present in the latest PR head, and the Windows cross-drive escape is shown by a read-only `path.win32.relative` probe.

## Solution Assessment

Is this the best way to solve the issue? No, not as currently written. The feature direction is plausible, but the implementation should share existing manifest and package-boundary contracts before importing or publishing generated metadata.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Reject cross-drive entries outside the plugin root:** `src/cli/plugins-authoring-command.ts:57-64`
  - body: `normalizeRelativePath` only rejects `..` escapes. On Windows, an entry on another drive normalizes to a drive-qualified relative path like `D:/outside/index.js`, passes this guard, and can then be imported and persisted as if it lived under the plugin root.
  - confidence: 0.9
- **[P2] Parse plugin manifests with the normal loader:** `src/cli/plugins-authoring-command.ts:45-47`
  - body: `readJsonFile` uses strict `JSON.parse` for `openclaw.plugin.json`, even though the supported manifest loader accepts JSON5 syntax. A manifest with comments or trailing commas can pass normal plugin loading and then fail `plugins validate` or `plugins build --check`.
  - confidence: 0.9
- **[P2] Generate publishable names for scoped ids:** `src/cli/plugins-authoring-command.ts:347-349`
  - body: `plugins init` interpolates the raw plugin id into `package.json.name`, so an id such as `@acme/quotes` becomes `openclaw-plugin-@acme/quotes`, which is not a publishable package name. Normalize or validate the id before writing scaffold metadata.
  - confidence: 0.84
- **[P2] Require one generated runtime entry during validation:** `src/cli/plugins-authoring-command.ts:253-255`
  - body: `validateToolPluginProject` only checks whether `openclaw.extensions` includes the selected entry. A package with extra stale entries still passes validation even though `plugins build` would rewrite it to a single selected runtime entry, allowing unintended runtime entrypoints to ship.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff introduces authoring-time runtime imports and generated package metadata with concrete path-boundary and stale-entry concerns.

- **[low] Harden authoring entry root checks:** `src/cli/plugins-authoring-command.ts:57`
  - body: Because `plugins build/validate` imports the selected entry and writes it into package metadata, cross-drive paths that bypass the intended root guard can create an out-of-package runtime reference on Windows.
  - confidence: 0.82
