## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Reject unsupported image counts before posting:** `extensions/huggingface/image-generation-provider.ts:73`
  - body: This provider advertises `maxCount: 1`, but `generateImage` never checks `req.count`; CLI callers can pass `--count 2` and get a single image back without an error because the CLI path forwards count straight to the runtime. Add an early `count > 1` rejection and cover it in the provider test.
  - confidence: 0.88
- **[P2] Forward the runtime SSRF policy to the request helper:** `extensions/huggingface/image-generation-provider.ts:123-131`
  - body: The image-generation runtime passes `ssrfPolicy` to providers so transports honor the same hostname allowlists, fake-IP, and private-network policy as other image providers. This new Hugging Face request omits `ssrfPolicy: req.ssrfPolicy` when calling `postJsonRequest`, leaving this provider path inconsistent with the guarded provider contract.
  - confidence: 0.9
- **[P2] Allow valid single-segment Hugging Face repo IDs:** `extensions/huggingface/image-generation-provider.ts:27`
  - body: Hugging Face's repo-id validator accepts either `repo_name` or `namespace/repo_name`, and OpenClaw can pass a single-segment model as `huggingface/<repo_name>`. The slash-required regex rejects those valid repo IDs before the router sees them; align the validator with the upstream repo-id rules while still blocking `..`, leading slashes, and multi-segment paths.
  - confidence: 0.78
