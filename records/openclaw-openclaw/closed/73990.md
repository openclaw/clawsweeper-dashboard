## Summary

Keep open. Current main still drops contentful transcript lines when provider usage telemetry is absent, and the later OpenAI-compatible streaming fixes only help endpoints that return usage after `stream_options.include_usage` is sent; they do not provide the requested transcript-derived fallback when telemetry is missing.

## What This Changes

The issue requests a conservative transcript-derived context estimate for local/OpenAI-compatible sessions when provider usage telemetry is missing or zero.

## Best Possible Solution

Add a bounded transcript-content estimate only when no positive provider usage snapshot exists, while preserving provider-reported usage, preserved stale totals, delivery-mirror filtering, and recent-tail performance limits.

## Reproduction Assessment

Yes, source-reproducible: current code filters out assistant content without usage/model identity, aggregate usage returns null without an accepted snapshot, and the current test explicitly asserts that a contentful no-usage transcript returns null. I did not run a live local LLM endpoint because this was a read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is in transcript usage extraction and its status/session-list consumers: estimate from bounded transcript content only when telemetry is absent or zero, and keep positive provider telemetry authoritative.
