## Summary

Keep open: the PR targets a real Ollama regression and includes sufficient request-body proof, but the patch restores a broad runtime-window fallback that can also apply to discovered/default model metadata, so it needs maintainer policy review before merge.

## What This Changes

The PR changes Ollama native request construction, runtime tests, and provider docs so `options.num_ctx` falls back from `contextWindow` or `maxTokens` when `params.num_ctx` is absent.

## Best Possible Solution

Preserve explicit configured Ollama budgets while preventing discovered/default model metadata from driving native `options.num_ctx`, either with budget provenance or a migration to `params.num_ctx`.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main only forwards explicit `params.num_ctx`, while the linked configuration relies on `contextWindow` without `params.num_ctx`. I did not run a live Ollama daemon.

## Solution Assessment

No, the current patch is broader than the reported explicit-config regression because it falls back from the final runtime model window after discovery/config merging. A narrower solution should distinguish explicit user budget from discovered/default metadata or migrate existing configs to `params.num_ctx`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

- **[P2] Avoid deriving native num_ctx from discovered windows:** `extensions/ollama/src/stream.ts:312-316`

That means unconfigured models can start sending `options.num_ctx` again, contradicting the docs' no-budget behavior and reopening the resource/latency behavior current main intentionally fixed; only explicit or migrated budgets should drive native `num_ctx`.
