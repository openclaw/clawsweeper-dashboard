## Summary

Current main now implements the central stale-final suppression behavior with a foreground reply freshness fence, and regression coverage exercises the same timing gap described here. The fix is present on main after the latest stable release, so this should close as current-main-only implemented.

## What This Changes

Request delivery-time suppression of an older final reply when a newer same-session/channel/target inbound message supersedes it.

## Best Possible Solution

Keep the current foreground freshness fence for stale visible finals; track content-aware routing or async child-branch cancellation separately if maintainers want those broader extensions.

## Reproduction Assessment

Not applicable as a feature request. Current-main regression coverage exercises the relevant race by starting an older dispatch, then a newer inbound for the same session target, and asserting only the newer final is delivered.

## Solution Assessment

Yes. The implemented fence is narrower than queue-size polling: it cancels only visible delivery for superseded same-target foreground replies at beforeDeliver while leaving unrelated targets isolated.
