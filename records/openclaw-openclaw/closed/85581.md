## Summary

Close as a duplicate: the same bundled-MCP request-timeout problem is already tracked in the older canonical issue and has active implementation PRs covering the competing timeout-contract choices.

## What This Changes

This issue asks for per-server MCP tool-call timeout config to reach SDK `Client.callTool()` so long-running bundled MCP tools are not capped by the SDK 60s default.

## Best Possible Solution

Keep the canonical timeout discussion and one reconciled implementation path open, then close duplicates into https://github.com/openclaw/openclaw/issues/60967.

## Reproduction Assessment

Yes, at source level. Current main reaches `Client.callTool()` without SDK request options, and SDK 1.29.0 defaults omitted request timeouts to 60000 ms; I did not run a live >60s MCP tool.

## Solution Assessment

No, this should not be solved as a separate issue. The narrow runtime fix is to pass SDK `RequestOptions.timeout`, but the public field name, default duration, and progress-reset semantics should be settled in the canonical threads.
