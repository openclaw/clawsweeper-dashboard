## Summary

Current main already surfaces model fallback switches: the reply runner records the selected-to-active fallback pair, emits fallback/fallback_cleared lifecycle events, adds inline Model Fallback notices when verbose output is enabled, shows active fallback state in /status with a reason, and the Control UI renders fallback active/cleared indicators from those lifecycle events. The changelog records this as shipped in 2026.2.21, and related ClawSweeper context for #33654/#39818 reached the same implemented-on-main conclusion.

## Best Possible Solution

Close this issue as already implemented on main. Keep the shipped visibility model: lifecycle events for clients, Control UI fallback indicators, /status fallback context, and verbose inline notices. If maintainers want a stricter always-notify or require-approval policy later, that should be tracked as a narrower product change rather than this silent-fallback bug report.
