## Summary

Current main already solves the reported user-visible failure: model fallback now continues to the next candidate even when the provider error is unrecognized, so an HTTP 412 Fireworks account-suspension response no longer has to be classified as `billing` to advance the fallback chain. The same behavior is present in release `v2026.4.24`. The 412-specific billing classification gap remains, but it is no longer the blocker described in the issue.

## Best Possible Solution

Close this issue as fixed by the shipped generic fallback-loop behavior. Keep the current implementation because it addresses the reported bot-stops-responding symptom across more provider-specific status codes than a one-off 412 mapping; open a narrower follow-up only if billing-specific classification, cooldown, or user-facing copy for suspended-account messages is desired.

## Evidence
