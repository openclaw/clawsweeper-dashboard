## Summary

Keep open: current main and the shipped 2026.5.19 release still classify `host.orb.internal` outside the local Ollama no-auth path, so the reported missing-key regression remains source-reproducible and is not superseded by another canonical item.

## What This Changes

The issue reports that local Ollama models fail after upgrading to 2026.5.19 when OpenClaw runs inside an Orb VM and reaches host Ollama through `host.orb.internal`.

## Best Possible Solution

Add a narrow local-host classification for the Orb host alias, keep public/cloud Ollama hosts credential-required, and cover runtime auth plus setup discovery with regression tests.

## Reproduction Assessment

Yes from source inspection: with `baseUrl: "http://host.orb.internal:11434"`, `api: "ollama"`, and `apiKey: "ollama-local"`, both the core custom-provider marker path and the Ollama synthetic-auth classifier leave the provider without usable auth. I did not run a live Orb VM path in this read-only review.

## Solution Assessment

No, current main is not the best end state because it treats a known VM host alias as remote for local Ollama auth. The maintainable fix is a bounded alias/local-host policy update with tests, not a broad relaxation for all dotted or `.internal` hosts.
