## Summary

Keep open: current main and the latest release still skip QMD embedding maintenance when omitted `searchMode` resolves to lexical `search`, and an open closing PR remains the implementation path rather than a close condition.

## What This Changes

The issue asks OpenClaw to honor `memory.qmd.update.embedInterval` for QMD embedding after memory document changes when `searchMode` is omitted/defaulted.

## Best Possible Solution

Carry explicit `embedInterval` intent through resolved QMD config and update the manager, tests, and docs if maintainers want lexical `search` to opt into embedding; otherwise document that users must choose `query` or `vsearch` for automatic embedding maintenance.

## Reproduction Assessment

Yes, source-reproducible: omitted `memory.qmd.searchMode` resolves to `search`, and current main returns from both embed execution and timer scheduling before a positive `embedIntervalMs` can matter. I did not run a live QMD/WSL repro.

## Solution Assessment

Unclear. The linked PR targets the scheduler gap, but the maintainable fix should distinguish explicit `embedInterval` from the 60-minute default and align the BM25-only docs/test contract.
