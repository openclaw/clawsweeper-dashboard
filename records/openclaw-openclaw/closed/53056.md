## Summary

Keep open. Current `main` and `v2026.5.3` still encode Slack ampersands before normal sends and preview text, the two linked fix PRs are closed unmerged, and Slack's docs make this parsing-sensitive enough to need live rendering proof rather than cleanup closure. ([docs.slack.dev](https://docs.slack.dev/messaging/formatting-message-text/?utm_source=openai))

## What This Changes

The issue requests that Slack messages containing literal ampersands render as `&` instead of visible `&#38;`.

## Best Possible Solution

Keep this open until Slack rendering is verified directly, then update the Slack-owned formatter, preview helpers, send paths, and regression tests to enforce exactly one safe escaping layer while preserving intentional Slack tokens.

## Reproduction Assessment

No. Current source shows OpenClaw emits `&amp;` for raw ampersands and tests expect that payload, but Slack docs say that is the literal-text escape form; the reported visible `&#38;` still needs a live Slack smoke with raw and pre-escaped samples.

## Solution Assessment

Unclear. The closed PR direction of removing ampersand escaping is not proven best because Slack documents `&` as a control character; the safer fix is a verified exactly-one-escaping policy with coverage for literal entities and active Slack tokens.
