## Summary

Keep open: the Bedrock context-window bug is still source-reproducible on current main, and this PR is a useful fix path, but the branch still has a blocking Claude 4 fallback matcher bug and lacks after-fix real Bedrock proof.

## What This Changes

The PR backfills Bedrock Claude 3/3.5/Opus 4 context-window entries, adds an Anthropic Claude family fallback, adds mocked discovery tests, and adds a changelog entry.

## Best Possible Solution

Repair the Bedrock plugin fallback to cover documented Claude 4 ID shapes with regression tests, or narrow the fallback to Claude 3.x, then add redacted real Bedrock proof before landing it as the canonical fix for https://github.com/openclaw/openclaw/issues/73328.

## Reproduction Assessment

Yes, at source level. Current main lacks the reported known-window entries and any family/override fallback, and the discovery path falls back to defaultContextWindow for missing foundation models and inference profiles; I did not run a live Bedrock gateway.

## Solution Assessment

No, not as-is. The explicit table backfill is a narrow fit, but the advertised Claude 4 family fallback must match real Bedrock ID shapes or be removed from the claimed scope.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Match real Claude 4 Bedrock IDs in the fallback:** `extensions/amazon-bedrock/discovery.ts:134`
  - body: The new fallback claims to cover Claude 4 variants, but `^anthropic\.claude-(?:3|4)` only matches IDs beginning `anthropic.claude-4`. Bedrock Claude 4 IDs use `anthropic.claude-sonnet-4...`, `anthropic.claude-opus-4...`, and `anthropic.claude-haiku-4...`, so an unlisted Claude 4 profile would still inherit the default context window. Widen the matcher and add a Claude 4 fallback test, or scope the fallback to Claude 3.x only.
  - confidence: 0.9
