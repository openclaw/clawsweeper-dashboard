## Summary

This is the same ACP runtime-control bug already tracked in the older open canonical issue: current main still maps per-spawn ACP timeouts into `session/set_config_option`, and the open fix PRs target that shared unsupported-config path rather than a Gemini-only failure.

## What This Changes

The issue asks OpenClaw to let ACP `sessions_spawn` use an outer run timeout without requiring the target ACP adapter to support `session/set_config_option` for `timeout`.

## Best Possible Solution

Keep https://github.com/openclaw/openclaw/issues/73875 as the canonical tracker, land or replace the focused ACP runtime-control fix, and preserve this issue's Gemini timeout evidence by linking it there.

## Reproduction Assessment

Yes from source inspection: ACP `sessions_spawn` stores `runTimeoutSeconds` as `runtimeOptions.timeoutSeconds`, `buildRuntimeConfigOptionPairs` emits `timeout`, and runtime-control application sends it through `session/set_config_option` before the turn. I did not rerun a live Gemini ACP harness in this read-only sweep.

## Solution Assessment

Yes: duplicate/superseded close is the narrow cleanup path because the older open canonical issue and open fix PRs already track the same unsupported ACP config-option behavior. The implementation should be reviewed on those PRs rather than splitting another tracker for the same root cause.
