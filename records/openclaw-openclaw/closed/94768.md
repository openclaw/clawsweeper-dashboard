## Label Justifications

- `P1`: The reported behavior is a user-facing cross-agent approval prompt leak in a chat channel workflow.
- `impact:security`: The issue concerns approval metadata crossing to an unrelated agent/channel boundary.
- `impact:message-loss`: The failure mode is misrouted or duplicated approval messages through the wrong Telegram bot connection.

## Summary

Current main now solves the reported user-visible leak for before_tool_call/Skill Workshop plugin approvals by forwarding trusted turn-source channel, target, account, and thread metadata into the existing plugin approval route, so Telegram native approval handling can stay scoped to the originating bot/account instead of unbound delivery.

## What This Changes

The issue reports that plugin approval prompts can be delivered to unrelated connected channel clients in multi-agent Telegram setups instead of only the requesting agent's route.

## Best Possible Solution

Keep the current-main route-forwarding implementation and close this report as fixed on main; if a new two-bot leak reproduces after a release containing the fix, track that as a fresh, narrower Telegram approval-routing bug.

## Reproduction Assessment

Yes at source level for the reported release: v2026.6.8 omitted turn-source fields from before_tool_call plugin approvals while approval runtimes could receive plugin approval events. Current main no longer matches that failure path because the route fields are forwarded and covered by focused tests.

## Solution Assessment

Yes. The current fix uses the existing gateway contract and trusted HookContext route metadata, which is narrower and safer than adding a new agent-id visibility rule or documenting the broad fallback as safe.
