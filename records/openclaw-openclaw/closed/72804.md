## Summary

Current main already solves the reported browser-proxy problem through the shipped `browser.extraArgs` surface: config accepts Chromium proxy flags, the browser resolver preserves them, Chrome launch omits `--no-proxy-server` when a proxy-control flag is present, and docs now tell users to use `browser.extraArgs` for `--proxy-server=...`. The exact `browser.proxy` alias is not present, but the user-visible need to avoid editing compiled JS after updates is covered.

## What This Changes

Request to add a config-based way to proxy OpenClaw-managed browser launches instead of manually removing the default `--no-proxy-server` flag after each update.

## Best Possible Solution

Keep the shipped `browser.extraArgs` path as the supported browser-proxy mechanism and point users at the browser docs. A separate `browser.proxy` alias would be ergonomics sugar, not required to solve the manual compiled-JS edit problem.

## Reproduction Assessment

Not applicable as a feature request, but the current behavior has a high-confidence static verification path: set `browser.extraArgs` to `--proxy-server=...`, and the launch builder omits `--no-proxy-server` while appending that flag.

## Solution Assessment

Yes. The existing `browser.extraArgs` solution is the narrowest maintainable fix because it reuses Chromium's own proxy flags, preserves direct mode by default, and avoids a new config contract for every Chrome proxy variant.
