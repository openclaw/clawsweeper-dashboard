## Summary

Current main already prevents the reported cron-spawned subagent notification spam: cron task records are silent, successful subagent task records are not auto-delivered through the task ledger, and subagent completion announcements for cron run sessions are routed internally with `deliver: false`. The broader configurable task notification default remains tracked separately by #64081.

## What This Changes

Requested configurable notification policy defaults for subagents spawned from cron jobs so silent cron runs do not emit per-subagent completion notifications.

## Best Possible Solution

Keep the current cron/subagent delivery behavior: child completion updates from cron run sessions should stay internal and the cron delivery path should consolidate final descendant output. Handle any broader user-configurable task notification policy through #64081 so the public config shape is designed once across runtimes.

## Reproduction Assessment

No. Current main has a high-confidence static non-reproduction path: cron requester sessions are treated as internal announcement targets with `deliver: false`, and successful subagent task records are not auto-delivered by the task ledger.

## Solution Assessment

Yes for the cron spam behavior. The current implementation solves it at the delivery boundary without adding a subagent-only config key, while the broader configurable default belongs in #64081.
