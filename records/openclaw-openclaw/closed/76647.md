## Summary

Keep open. The PR targets a real source-reproducible group/channel delivery gap, but the patch currently has blocking delivery-policy regressions in ACP sendPolicy denial and plugin-bound same-provider replies; it is also draft and protected by maintainer handling.

## What This Changes

The PR routes group/channel `message_tool` final, block, tool, abort, before-dispatch, TTS, and ACP replies through `routeReply`, with WhatsApp, Discord, and ACP regression coverage.

## Best Possible Solution

Revise the PR so only message-tool suppression re-enables routed delivery, sendPolicy denial remains fully silent, plugin-bound same-provider replies either stay suppressed or use the same `routeReply` path, and focused regressions cover those cases.

## Reproduction Assessment

Yes, from source inspection: current main defaults group/channel rooms to `message_tool_only`, only loads `routeReply` for cross-provider routes, and then skips normal final delivery under `suppressDelivery`. I did not execute tests because this was a read-only review.

## Solution Assessment

No. Routing same-provider group/channel replies through `routeReply` is the right direction, but this patch must first preserve sendPolicy denial in ACP delivery and avoid automatic dispatcher fallback for plugin-bound same-provider notices/replies.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve sendPolicy denial for ACP routed replies:** `src/auto-reply/reply/dispatch-acp-delivery.ts:202-205`
  - body: When `sendPolicy` denies delivery on a group/channel turn, `sourceReplyDeliveryMode` can still be `message_tool_only` and `suppressUserDelivery` is true. This new gate treats that combination plus an originating target as permission to send via `routeReply`, even though the hook deliberately relies on the ACP delivery layer to keep deny-mode output silent. Pass an explicit reason/flag so only message-tool suppression, not sendPolicy denial, enables routed delivery.
  - confidence: 0.92
- **[P2] Route plugin-bound notices through the message tool:** `src/auto-reply/reply/dispatch-from-config.ts:809`
  - body: For same-provider group/channel `message_tool` mode this condition lets plugin-bound claims run, but `sendBindingNotice` only routes through `routeReplyToOriginating`; when `shouldRouteToOriginating` is false it falls back to `dispatcher.sendFinalReply` or `sendToolResult`. A handled plugin reply or missing-plugin notice is then visibly delivered through automatic dispatcher delivery instead of the unified message sender. Keep the old suppression for this path or teach `sendBindingNotice` to use `routeReplyToSourceViaMessageTool`.
  - confidence: 0.84
