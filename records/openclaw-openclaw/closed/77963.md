## Summary

Keep open: the bug is valid and the PR is directionally right, but the current patch still risks dropping valid iterable/foreign `HeadersInit` values and needs real OpenClaw plugin/global-fetch proof before merge.

## What This Changes

The PR adds a shared header-sanitization helper, applies it to proxy/runtime fetch plus redirect/debug-capture paths, and adds a changelog fix entry.

## Best Possible Solution

Preserve the boundary-sanitization approach, keep valid iterable/Headers-like inputs intact, add focused regression coverage, and merge only after real OpenClaw plugin/global-fetch proof on Node 22.

## Reproduction Assessment

Yes. The native `Headers` failure is reproducible with a Symbol-keyed record, and current main clearly forwards `init.headers` through proxy/runtime/redirect paths without normalization; I did not run a full Discord plugin reproduction in this checkout.

## Solution Assessment

No, not yet. Boundary sanitization is the right narrow fix, but this implementation should preserve iterable or foreign `Headers` objects and still needs after-fix proof from the actual OpenClaw runtime path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.78

Full review comments:

- **[P2] Preserve iterable header inputs while sanitizing:** `src/infra/net/redirect-headers.ts:40-44`
  - body: The new sanitizer treats every non-array, non-global `Headers` object as a plain record and rebuilds it with `Object.entries()`. Valid iterable header inputs, including foreign-realm or Headers-like objects, can have no enumerable string properties, so this sends empty headers through the proxy/runtime/redirect paths and can drop auth or content headers. Return iterable/Headers-like inputs unchanged or clone them safely while only filtering plain records.
  - confidence: 0.78
