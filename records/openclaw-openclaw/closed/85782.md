## What This Changes

The issue asks for terminal TUI lifecycle/provider errors on an active chat run to render a visible run error, clear stale pending state, and unblock input.

## Best Possible Solution

Review and land or close https://github.com/openclaw/openclaw/pull/85996; the intended end state is one TUI handler path where terminal lifecycle errors for the active run produce the same visible terminal cleanup as chat errors.

## Reproduction Assessment

Yes from source inspection: send a lifecycle `phase: "error"` event for the active run without a terminal chat error, and the handler sets status without using the transcript/error cleanup path. I did not run a live TUI/provider repro in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is to route terminal lifecycle errors for the active TUI run into the existing visible terminal error cleanup semantics, while preserving retryable/fallback lifecycle behavior.
