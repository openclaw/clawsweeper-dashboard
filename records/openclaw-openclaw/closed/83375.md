## Summary

Close as ClawHub-scoped: current main and the latest shipped release no longer contain a supported BlueBubbles channel, the supported iMessage path is the bundled `imessage` plugin via `imsg`, and the existing plugin/channel SDK docs cover an external BlueBubbles plugin without showing a missing core API.

## What This Changes

The issue reports BlueBubbles catchup polling constructing `iMessage;-;` chat GUIDs on macOS Tahoe where BlueBubbles exposes `any;-;` chat GUIDs instead.

## Best Possible Solution

Keep BlueBubbles out of OpenClaw core, guide users to `channels.imessage` for bundled iMessage, and publish any maintained BlueBubbles Tahoe GUID handling as an external ClawHub channel plugin unless a generic SDK gap is found.

## Reproduction Assessment

No for current main: the reported 2026.4.24 BlueBubbles runtime is absent from HEAD and v2026.5.12, so the exact BlueBubbles REST polling failure cannot be reproduced against supported OpenClaw core. The supported iMessage catchup path is source-checkable and uses `imsg` JSON-RPC instead.

## Solution Assessment

No for core: querying BlueBubbles for a Tahoe-safe chat GUID may be the right repair for an external BlueBubbles plugin, but reviving BlueBubbles-specific REST behavior in core conflicts with the documented migration and plugin-scope direction.
