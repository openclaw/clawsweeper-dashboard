## Summary

Closing as cannot reproduce. On current `main`, the production WhatsApp inbound path drops messages before enqueue when no direct-chat `from` can be resolved, so `resolvePeerId()` is not reached with `msg.from === undefined` on the shipped path.

## Evidence

- **Inbound contract requires `from`:** `WebInboundMessage` declares `from: string` and `conversationId: string`, so the auto-reply monitor consumes a normalized message shape, not arbitrary Baileys payloads.
  - file: [extensions/whatsapp/src/inbound/types.ts:54](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/extensions/whatsapp/src/inbound/types.ts#L54)
  - command: `sed -n '1,220p' extensions/whatsapp/src/inbound/types.ts`
  - sha: [f3c37a946cc4](https://github.com/openclaw/openclaw/commit/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4)
- **Normalizer drops unresolved direct peers:** `normalizeInboundMessage()` returns `null` when `remoteJid` is missing, when the chat is `@status`/`@broadcast`, or when `resolveInboundJid(remoteJid)` cannot produce `from`; this prevents an undefined direct peer id from entering the queue.
  - file: [extensions/whatsapp/src/inbound/monitor.ts:352](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/extensions/whatsapp/src/inbound/monitor.ts#L352)
  - command: `nl -ba extensions/whatsapp/src/inbound/monitor.ts | sed -n '340,390p'`
  - sha: [f3c37a946cc4](https://github.com/openclaw/openclaw/commit/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4)
- **Queued messages always copy normalized `from`:** The queued inbound message passed downstream sets both `from` and `conversationId` from `inbound.from`, which only exists after the normalizer's guard succeeds.
  - file: [extensions/whatsapp/src/inbound/monitor.ts:564](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/extensions/whatsapp/src/inbound/monitor.ts#L564)
  - command: `nl -ba extensions/whatsapp/src/inbound/monitor.ts | sed -n '540,585p'`
  - sha: [f3c37a946cc4](https://github.com/openclaw/openclaw/commit/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4)
- **Auto-reply handler only receives listener output:** `createWebOnMessageHandler()` is wired through `attachWebInboxToSocket`/the listener path, so the production call site consumes the normalized inbox messages above.
  - file: [extensions/whatsapp/src/auto-reply/monitor.ts:233](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/extensions/whatsapp/src/auto-reply/monitor.ts#L233)
  - command: `nl -ba extensions/whatsapp/src/auto-reply/monitor.ts | sed -n '228,270p'`
  - sha: [f3c37a946cc4](https://github.com/openclaw/openclaw/commit/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4)
- **The old crash site still assumes the invariant:** `resolvePeerId()` still calls `msg.from.includes("@")`, but on current `main` that call sits behind the inbound normalization invariant rather than accepting raw undefined `from` values.
  - file: [extensions/whatsapp/src/auto-reply/monitor/peer.ts:5](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/extensions/whatsapp/src/auto-reply/monitor/peer.ts#L5)
  - command: `nl -ba extensions/whatsapp/src/auto-reply/monitor/peer.ts | sed -n '1,80p'`
  - sha: [f3c37a946cc4](https://github.com/openclaw/openclaw/commit/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4)

## Risks / Open Questions

- If a nonstandard caller bypasses `extensions/whatsapp/src/inbound/monitor.ts` and constructs a malformed `WebInboundMessage` manually, `resolvePeerId()` still has no runtime null guard.
