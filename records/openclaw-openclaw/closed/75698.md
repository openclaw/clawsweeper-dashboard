## Summary

Keep open. The PR addresses a real user-visible error leakage path, but the patch is not merge-ready because it regresses rate-limit messaging, strips legitimate file paths from normal non-error replies, and is stale against the current shared error-parser split.

## What This Changes

The PR sanitizes embedded assistant/tool error text before it reaches messaging payloads, adding tests for Brave Search error JSON and local filesystem path redaction.

## Best Possible Solution

Rebase or replace the PR with a narrow current-main fix: recognize Brave-style `ErrorResponse` payloads in the shared formatter, sanitize embedded tool/assistant error surfaces, and keep filesystem path redaction limited to error contexts so normal replies can still mention useful local paths.

## Reproduction Assessment

Yes. Static inspection shows current main can surface raw `lastToolError.error` in embedded warning payloads, and the PR regressions are reproducible from the patch plus existing sanitizer and rate-limit test contracts; live tests were intentionally not run during this read-only review.

## Solution Assessment

No. Sanitizing user-facing tool errors is the right direction, but this patch is broader than necessary and stale; the safer solution is an error-context-only redaction path plus shared parser support for `ErrorResponse` payloads.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve rate-limit-specific error copy:** `src/agents/pi-embedded-helpers/errors.ts:381-382`
  - body: This branch now treats every rate-limit error as an overloaded service response. That bypasses the existing rate-limit formatter, so messages like `429 rate limit reached`, provider reset hints, or quota retry text lose their useful rate-limit details and would conflict with the existing tests that assert those details are preserved.
  - confidence: 0.9
- **[P2] Limit path redaction to error contexts:** `src/agents/pi-embedded-helpers/errors.ts:445`
  - body: `sanitizeUserFacingText` is also used for ordinary assistant replies, with `errorContext: false` for non-error payloads. With this unconditional final `stripFilesystemPaths(...)`, a normal answer that needs to tell the user about `/Users/alice/project/file.ts` or `C:\Users\bob\...` will be rewritten to `<path>`, breaking legitimate messaging content. Keep filesystem path redaction behind an error-context path or apply it only when formatting tool/assistant errors.
  - confidence: 0.87
