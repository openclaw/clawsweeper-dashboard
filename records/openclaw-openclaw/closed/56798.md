## What This Changes

The PR raises Node's default autoSelectFamily attempt timeout from 250 ms to 300 ms during `ensureGlobalUndiciStreamTimeouts()` when the getter reports 250, plus test coverage.

## Best Possible Solution

Keep the shipped guarded-fetch family-policy implementation and leave broader channel-specific IPv6 fallout to the canonical follow-up tracker rather than merging a process-wide Node timeout mutation.

## Reproduction Assessment

Yes for the PR-introduced policy regression: local Node reports 250 ms for the untouched default, the explicit CLI flag, and the same `NODE_OPTIONS` setting, matching the PR's value-only guard. I did not live-reproduce the original broken-IPv6 provider outage in this read-only review.

## Solution Assessment

No for this PR branch. Current main's shared per-dispatcher Undici policy is the safer fix because it covers guarded provider requests without mutating process-wide Node defaults or clobbering explicit connect overrides.
