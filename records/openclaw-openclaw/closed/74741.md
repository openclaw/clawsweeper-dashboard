## Summary

Keep open. Current main still matches the reported Signal SSE timeout path: the Signal event stream has a 10s default deadline, the monitor does not override it for the long-lived receive stream, and the affected Signal files have no diff from v2026.4.27. Open PR #74744 directly closes this issue and #73086 overlaps, so the issue should remain open until maintainers choose and land a fix.

## What This Changes

Signal's `/api/v1/events` SSE consumer should keep the receive stream open long enough to consume inbound Signal events and update channel activity instead of timing out after 10 seconds.

## Best Possible Solution

Land one narrow Signal-plugin fix that disables or overrides the short deadline only for the long-lived receive SSE stream, preserves bounded RPC/probe/send deadlines and existing reconnect/abort/parse limits, and adds coverage proving both an idle receive stream and inbound activity/status behavior.

## Reproduction Assessment

Yes. A local `/api/v1/events` endpoint that accepts the connection but delays SSE response headers or events beyond the current 10s deadline exercises the same monitor path and produces the reported timeout; the reporter's live logs show the same failure against signal-cli.

## Solution Assessment

No. Current main is not the best solution because the monitor receive stream still uses the generic bounded deadline. The maintainable fix is Signal-local: disable or override that deadline for the monitor SSE stream while keeping bounded deadlines for finite Signal operations and verifying status activity.
