## Summary

Current main implements the core remote node file-I/O problem through the enabled-by-default File Transfer plugin rather than new `system.read`/`system.write` commands. It provides structured node-backed read/list/fetch/write tools, keeps the generic file tools on gateway/sandbox files, and adds explicit node command/path policy so the high-risk filesystem boundary is not opened implicitly.

## What This Changes

Request node-protocol-backed remote file I/O so agents can read, write, and edit files on the node host when execution is routed to a node.

## Best Possible Solution

Keep the File Transfer plugin as the canonical, policy-gated node filesystem surface. If maintainers later want editor-style partial patching or exact `read`/`write`/`edit` proxying, that should be opened as a narrower follow-up on top of the shipped file-transfer boundary.

## Reproduction Assessment

Not applicable as a bug reproduction; the current-main check is source-backed. The original gap is no longer present at the product level because node file reads/writes are available through structured `file_fetch`/`file_write` tools instead of shell exec workarounds.

## Solution Assessment

Yes for the broad remote file-I/O problem: a dedicated, deny-by-default plugin/tool surface is safer than implicitly routing standard filesystem tools to another host based on `tools.exec.host`. It deliberately avoids making gateway workspace files disappear behind node routing.
