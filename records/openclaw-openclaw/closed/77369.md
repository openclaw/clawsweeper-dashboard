## Summary

Keep open: current `main` only knows how to delete Telegram drafts whose message id is still held by the current in-memory draft stream, and a process restart loses that state before cleanup can run.

## What This Changes

The issue asks Telegram progress draft messages such as `Sifting...` to be cleared or replaced after an OpenClaw process restart so stale progress labels do not remain visible in chat.

## Best Possible Solution

Keep the issue open and add a Telegram-plugin-local transient draft ledger that records active progress draft IDs, clears or marks them before finalization, and deletes safe leftovers on restart within a bounded TTL.

## Reproduction Assessment

Yes at source level: create a Telegram progress draft, terminate the process before final cleanup, then restart; the new process has no prior `streamMessageId`, and current cleanup only deletes the in-memory id. I did not run a live Telegram restart in this read-only sweep.

## Solution Assessment

Yes: the narrow maintainable fix is in the Telegram plugin's draft lifecycle and persisted state, not a core migration or a new user option. The implementation should explicitly protect finalized previews before deleting persisted leftovers.

## Repair Work Prompt

Fix openclaw/openclaw#77369. Telegram progress drafts in `streaming.mode: "progress"` can leave stale labels such as `Sifting...` visible after a process restart because the active draft message id is only in memory. Implement a Telegram-plugin-local fix: persist active transient progress draft message IDs as soon as the preview send returns, remove or mark them before successful finalization/retention, and on Telegram bot startup/account initialization attempt to delete only safe leftover transient drafts within a bounded TTL. Do not add a user config option, do not move owner-specific behavior into core, and do not delete finalized assistant messages or unrelated bot-sent messages if a crash happens during final delivery. Add regression coverage that simulates restart by resetting module state/loading persisted draft state, plus same-process cleanup coverage. Add a user-facing changelog fix entry.
