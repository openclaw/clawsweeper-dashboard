## Summary

Keep open. The issue title's original inheritance hypothesis does not match current main: exec config resolution already threads global/agent `safeBins` and merged `safeBinProfiles` into agent sessions. But the updated diagnosis remains valid: built-in profiles still cover only the original small set, `isSafeBinUsage()` still rejects matched safe bins when no profile exists, and current tests still prove unprofiled `echo` is denied. Latest release `v2026.4.23` shows the same profile set, so there is no shipped fix for the underlying behavior yet.

## Evidence

That makes the original "not inherited" hypothesis hard to support on current main.

- Closing as `cannot_reproduce` would hide a still-real bug: common safe-bin entries remain unusable without custom profiles, especially in isolated cron/heartbeat contexts where there is no accumulated allow-always state to mask the problem.
- The issue title is stale. The actionable problem on current main is missing built-in profiles plus approval fallback for unprofiled bins, not config inheritance.
- History is shallow in this checkout, so I could not independently verify the status of PR #60282 from git history alone; however current main and latest release both still lack the claimed profile additions.
