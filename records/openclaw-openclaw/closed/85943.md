## Label Justifications

- `P1`: The issue reports a recent regression causing real Telegram agent workflows to appear silent or time out under concurrent Codex app-server load.
- `impact:message-loss`: The reported user-visible failure is Telegram typing followed by silence, late replies, and gateway timeouts.
- `impact:session-state`: The request centers on Codex app-server thread, compaction, recovery, and session coupling across concurrent agents.

## Summary

Keep open: current main and the latest release still intentionally route legacy Codex CLI refs toward the Codex app-server path and mark the bundled `codex-cli` backend unsupported. The report is meaningful, but restoring a first-class CLI-isolated Codex runtime is a compatibility and runtime-product decision rather than a safe autonomous cleanup close.

## What This Changes

Request restoring `codex-cli` as an opt-in first-class Codex runtime, preserving explicit `codex-cli/*` refs, documenting both Codex runtime modes, and improving app-server backpressure for high-concurrency Telegram deployments.

## Best Possible Solution

Make an explicit maintainer runtime policy decision: either restore a plugin-owned `codex-cli` opt-in path with migration, docs, tests, and live high-concurrency proof, or document app-server-only Codex support and address isolation/backpressure through targeted app-server fixes.

## Reproduction Assessment

Partly. Current main source and docs clearly reproduce the missing first-class `codex-cli` route, but the high-concurrency Telegram stall/performance symptoms were not live-reproduced in this read-only review.

## Solution Assessment

Unclear. Restoring `codex-cli` may be the right operator escape hatch, but maintainers need to choose between reintroducing the CLI runtime contract and finishing narrower app-server isolation/backpressure work.
