## Summary

Keep open: current main still has the reported target-channel announce route, but this PR should not merge as-is because its requester path uses strict delivery that the gateway can reject for webchat/internal or unresolved targets, and the external branch still lacks real behavior proof.

## What This Changes

The PR changes `sessions_send` A2A announce handling to call gateway `agent` on the requester session before falling back to target-channel `send`, and updates related agent/tool tests.

## Best Possible Solution

Route A2A announce replies through the requester session using gateway best-effort/session-only delivery semantics, with focused coverage for webchat/internal requester sessions and externally routed requester sessions.

## Reproduction Assessment

Yes, from source inspection. Current main still resolves and sends the final A2A announce using the target/display session, and the PR's own new webchat requester case demonstrates the intended route that must be handled safely.

## Solution Assessment

No, not as currently proposed. The requester-routing direction is right, but strict `deliver: true` without `bestEffortDeliver` conflicts with the gateway delivery contract for webchat/internal or unresolved requester targets.
