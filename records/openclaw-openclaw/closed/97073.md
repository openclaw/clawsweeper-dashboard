## Summary

Keep open. Current main and the latest release still have the Slack reset-history mismatch, and this issue is protected by the `maintainer` label with an open linked fix PR awaiting maintainer review.

## What This Changes

Slack replies in an existing thread after a reset should seed bounded Slack thread history instead of starting the reset turn without prior thread context.

## Best Possible Solution

Review and land, or revise, the linked PR so Slack prepare treats stale thread rows as initial context while preserving fresh-session skip behavior, `initialHistoryLimit: 0`, `historyScope`, and `inheritParent` semantics.

## Reproduction Assessment

Yes, at source level: seed an existing Slack thread session row that is stale under daily or idle reset policy, then process a reply in that thread. Current main's Slack prepare sees a timestamp and skips history while session initialization evaluates the row as stale and starts a new session.

## Solution Assessment

Yes, the best fix is to make Slack's initial thread-history gate reset-freshness-aware rather than replaying old OpenClaw transcripts or adding another config option. The open PR appears to be the right candidate, pending maintainer acceptance of its SDK seam.
