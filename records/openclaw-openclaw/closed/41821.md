## Summary

Current main already implements Telegram media-download retry/fallback handling for `fetch failed` errors and preserves multi-photo media-group processing, with regression coverage and a shipped release containing the implementation.

## Evidence

- **Media downloads now retry through Telegram transport fallback:** Telegram media file downloads call `fetchRemoteMedia` with `dispatcherAttempts` and `shouldRetryFetchError`, so file downloads participate in the transport fallback path instead of failing on the first network error.
  - file: [extensions/telegram/src/bot/delivery.resolve-media.ts:235](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/extensions/telegram/src/bot/delivery.resolve-media.ts#L235)
- **`fetch failed` triggers sticky IPv4 and pinned-IP fallback:** The Telegram transport explicitly matches `fetch failed` plus known network codes, then retries with fallback dispatchers, escalating from the default path to IPv4-only and then a pinned Telegram API IP.
  - file: [extensions/telegram/src/fetch.ts:114](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/extensions/telegram/src/fetch.ts#L114)
- **Media groups are buffered and all photos are collected into one turn:** The media-group handler sorts the group, resolves each item, appends every successful media file to `allMedia`, and then processes the combined message once instead of dropping the whole album after the first item.
  - file: [extensions/telegram/src/bot-handlers.runtime.ts:389](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/extensions/telegram/src/bot-handlers.runtime.ts#L389)
- **Regression tests cover the transport fallback:** Tests assert the exported fallback dispatcher attempts for Telegram media downloads and cover sticky IPv4 plus pinned-IP fallback behavior for `fetch failed` network paths.
  - file: [extensions/telegram/src/fetch.test.ts:478](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/extensions/telegram/src/fetch.test.ts#L478)
- **Regression tests cover multi-photo media groups:** The e2e-style Telegram media tests verify same-group buffering and assert that a two-photo album produces one reply with `MediaPaths` length 2.
  - file: [extensions/telegram/src/bot.media.downloads-media-file-path-no-file-download.e2e.test.ts:316](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/extensions/telegram/src/bot.media.downloads-media-file-path-no-file-download.e2e.test.ts#L316)
- **Docs match the implemented behavior and the shipped release contains it:** The Telegram docs now state that `TypeError: fetch failed` is retried as a recoverable network error, and the same implementation is present in release tag `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`).
  - file: [docs/channels/telegram.md:822](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/docs/channels/telegram.md#L822)

## Close Comment

Closing from a Codex review because current OpenClaw main already implements this well enough. Telegram media downloads now flow through the retryable transport fallback in `extensions/telegram/src/bot/delivery.resolve-media.ts:235`, and the transport explicitly treats `fetch failed` as a recoverable Telegram network path with sticky IPv4 and pinned-IP fallback in `extensions/telegram/src/fetch.ts:114`. Media groups are also buffered and merged in `extensions/telegram/src/bot-handlers.runtime.ts:389`, with regression coverage in `extensions/telegram/src/fetch.test.ts:478` and `extensions/telegram/src/bot.media.downloads-media-file-path-no-file-download.e2e.test.ts:316`. This implementation is present in the shipped `v2026.4.22` release (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). If you still see this on `v2026.4.22` or newer, please open a fresh issue with current gateway logs and your Telegram network/proxy config.
