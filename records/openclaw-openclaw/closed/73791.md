## Summary

This should stay open: the missing-python fs-safe bug is real and the fallback direction is useful, but the proposed patch is not merge-ready because the copy fallback reopens the source after the verified handle and size check, and the PR omits the required user-facing changelog entry.

## What This Changes

The PR changes `writeFileWithinRoot` and `copyFileWithinRoot` to fall back to legacy write/copy behavior when the pinned Python write helper cannot spawn, and adds fs-safe tests for fallback and non-spawn error behavior.

## Best Possible Solution

Land the missing-python fallback for #72362, but keep the copy path tied to the originally verified source identity and size decision, add the required `CHANGELOG.md` entry, and then let the merged fix supersede the narrower error-message-only #72531 while leaving the broader Docker image python3 availability question in #75041 separate.

## Reproduction Assessment

Yes. #72362 gives a concrete `node:*bookworm-slim` reproduction without `python3`, and current main confirms the helper falls back to spawning `python3` while write/copy normalize helper failures instead of falling back.

## Solution Assessment

No, not as-is. The fallback is the narrow maintainable direction, but the copy fallback must preserve or revalidate the original source identity and size decision, and the PR needs the required release note before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Revalidate the reopened copy source:** `src/infra/fs-safe.ts:820-824`
  - body: In the spawn-error path, the code copies a newly opened `params.sourcePath` after the original verified handle and `maxBytes` check. If that path is swapped before the reopen, the fallback can copy a different or larger file than the one originally validated. Compare the reopened identity and size against `source.stat`, or keep copying from the verified source handle, before calling the legacy copier.
  - confidence: 0.87

  - confidence: 0.8

## Security Review

Status: needs_attention

Summary: The PR touches fs-safe boundary code and introduces a concrete source revalidation gap in the copy fallback.

Concerns:

- **[medium] Copy fallback can reopen a different source:** `src/infra/fs-safe.ts:820`
  - body: After the original source handle is verified and size-checked, the fallback reopens `params.sourcePath` and copies that new handle without checking it is the same file or still within the original size decision. In a python3-less runtime, a path replacement between those steps can weaken the current verified-handle behavior.
  - confidence: 0.87
