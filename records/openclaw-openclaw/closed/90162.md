Make final delivery resolve Discord outbound through a stable send-capable channel registry or loaded runtime plugin, invalidate failed non-send bootstrap attempts, and add a regression for the Discord-style setup/runtime registry split.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/90162. Current main and v2026.6.1 still let Discord final delivery miss the send-capable outbound adapter when `src/channels/plugins/registry-loader.ts` falls back from the pinned channel registry to a mutable active registry, and `src/infra/outbound/channel-bootstrap.runtime.ts` can mark a bootstrap attempt before the delivery path proves the resulting registry can send. Keep the fix in the channel/outbound registry boundary; do not add config, do not edit CHANGELOG.md, and do not hardcode Discord-specific behavior in core unless it is test-only. Preserve the legitimate setup-shell bootstrap case currently covered by `src/infra/outbound/deliver.test.ts`, but prevent stale or non-send-capable active registries from causing `Outbound not configured for channel: discord` after inbound has already loaded the Discord plugin. Add focused regression coverage for a pinned/setup channel registry, a replacement active registry, and a later send-capable Discord-style runtime plugin; include delivery recovery/final reply coverage if the existing harness supports it.
