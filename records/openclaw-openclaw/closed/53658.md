## Summary

Current main already fixes both reported Google OpenAI-compatible fallback failures: OpenAI-completions payloads only send `store: false` when compat says the endpoint supports it, and Google-compatible Gemini tool calls now capture and replay `thought_signature` for same-route history. Regression coverage exists for the Google OpenAI-compatible signature round trip and for omitting `store` on proxy-like completions routes.

## Best Possible Solution

Close as implemented on current main. Keep the compatibility gating and Gemini signature round-trip tests; reopen or file a narrower issue only if a fresh repro against current main shows a remaining cross-model fallback replay case.

## Evidence

- **Issue context reviewed:** Issue #53658 reports OpenClaw 2026.3.23-2 sending top-level `store: false` to Google's OpenAI-compatible endpoint and dropping Gemini `thought_signature` during fallback/replay. The follow-up comment adds proxy-captured proof for both failure modes, and related PR #66949 explicitly linked this issue as fixed for the OpenAI-compatible path.
- **Store is gated by OpenAI-completions compat:** `buildOpenAICompletionsParams` adds `params.store = false` only when `compat.supportsStore` is true, so endpoints detected as unsupported do not receive the top-level `store` field.
