## Summary

Keep this PR open. It is paired with same-author open bug #55532, and current main still does not prove the PR's intended Feishu WebSocket reconnect/PingInterval fix is implemented. The merged #72411 direction is relevant and credited this PR's investigation, but current main still awaits an SDK `start()` promise that the locked SDK resolves immediately after starting its internal reconnect loop, and the current `wsConfig` heartbeat defaults are passed through an option the SDK constructor does not read. This PR also should not merge as-is because its patch mutates `Lark.WSClient.prototype` process-wide.

## Best Possible Solution

Keep this PR open for maintainer reconciliation with #55532 and the merged #72411 direction. The best path is to verify the real `@larksuiteoapi/node-sdk` 1.62.0 startup, reconnect, callback, and pong/heartbeat behavior, then either close this PR and the paired issue together as implemented/superseded or land a safer app-layer wrapper that avoids prototype mutation and has regression coverage matching the actual SDK contract.
