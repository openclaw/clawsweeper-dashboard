## What This Changes

The branch omits `store: false` from the OpenAI Responses WebSocket request builder for proxy-like endpoints and also changes active-memory channel filtering for slash-containing IDs.

## Best Possible Solution

Apply one documented Responses persistence policy to the actual LiteLLM HTTP/Pi path and any supported WebSocket path, with native no-regression and proxy persistence coverage.

## Reproduction Assessment

No live reproduction was established, but the blocker is source-reproducible: LiteLLM/custom-base-url Responses sessions are excluded from the WebSocket path this PR changes. Existing tests also show the current WebSocket proxy expectation that the branch must update.

## Solution Assessment

No. A WebSocket-only condition is not the best fix for the linked LiteLLM report because the reported route stays on the HTTP/Pi payload path; a shared provider/model-aware store policy is safer.
