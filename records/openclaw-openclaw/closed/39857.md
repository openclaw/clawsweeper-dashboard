## Summary

Keep open: current main and v2026.6.6 still let a positive context-token override bypass the selected model's known window in the shared resolver and session-store persistence path. The central repair is already owned by the open, mergeable PR at https://github.com/openclaw/openclaw/pull/90889, so this issue should wait for that PR to land or be replaced.

## What This Changes

The issue asks OpenClaw to cap effective session context tokens by the selected model's known context window before reporting or persisting status/session metadata.

## Best Possible Solution

Land or replace the narrow resolver/session-store correction in https://github.com/openclaw/openclaw/pull/90889, then close this issue from the merged fix; handle first-status cache warmup separately in https://github.com/openclaw/openclaw/pull/92775 if still needed.

## Reproduction Assessment

Yes, source-reproducible. On current main, a positive contextTokensOverride returns before provider/model lookup, and the session-store path persists that override before asking the resolver for the selected model window.

## Solution Assessment

Yes, the best fix shape is the narrow shared resolver plus session-store delegation in the open replacement PR. A status-only patch would leave persisted session metadata and other resolver callers inconsistent.
