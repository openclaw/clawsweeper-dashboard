- `P1`: The reported upgrade regression silently removed raw Telegram updates and media from an active downstream ingestion path.

## Summary

Current main addresses the reported post-update failure mode: isolated Telegram ingress is enabled by default, a competing `getUpdates` poller now causes a recoverable restart instead of leaving the account path dead, and channel status exposes both polling errors and stalled spool backlog. Running simultaneous pollers is not the right fix because Telegram offset ownership would conflict; the later reply-metadata request is a separate plugin-hook enhancement rather than a reason to keep this regression report open.

## What This Changes

Restore Telegram isolated-polling spool operation after an in-place update, expose spool-writer health, and consider reply context in the ingest hook.

## Best Possible Solution

Keep one isolated Telegram poll owner with durable spool handoff, conflict-aware restart, and status-based liveness reporting; evaluate reply metadata separately as a typed plugin hook contract.

## Reproduction Assessment

Yes at source level: an overlapping old/new gateway or other poller produces Telegram 409 conflicts, and the pre-fix isolated worker path lacked the current conflict-specific restart behavior. The exact reporter installation was not rerun, but the source path, diagnostic, and focused regression test closely match the update-window symptom.

## Solution Assessment

Yes for the central regression: conflict-aware restart plus status reporting preserves a single poll owner and avoids permanent dormancy. No for simultaneous pollers; that would create competing Telegram offset ownership, while reply metadata belongs in a separate typed hook proposal.
