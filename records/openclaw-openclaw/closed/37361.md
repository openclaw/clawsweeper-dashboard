## Summary

Keep open. Current `main` still has the reported failure mode: local memory embeddings call `resolveModelFile(...)` during first-use initialization with no transient `ENOENT`/finalize retry, and `memory_search` converts any resulting exception into a disabled/unavailable response instead of recovering within the same call.

## Evidence

- **Local GGUF resolution still has no retry/backoff:** `createLocalEmbeddingProvider()` calls `resolveModelFile(modelPath, modelCacheDir)` inside `ensureContext()`. If that throws, the catch only clears `initPromise` and rethrows; there is no `.ipull`/`.gguf` finalize handling, retry loop, or fallback at this layer.
  - file: [src/memory-host-sdk/host/embeddings.ts:38](https://github.com/openclaw/openclaw/blob/7d471837369885c12d540d6bafa319ff41129b89/src/memory-host-sdk/host/embeddings.ts#L38)
  - command: `nl -ba src/memory-host-sdk/host/embeddings.ts | sed -n '1,120p'`
  - sha: [7d4718373698](https://github.com/openclaw/openclaw/commit/7d471837369885c12d540d6bafa319ff41129b89)
- **Existing embedding retry policy does not cover this error:** `isRetryableMemoryEmbeddingError()` only treats rate-limit / quota / 5xx-style failures as retryable. A rename `ENOENT` during local model finalization would not match, so the built-in retry loop would not retry it.
  - file: [extensions/memory-core/src/memory/manager-embedding-policy.ts:83](https://github.com/openclaw/openclaw/blob/7d471837369885c12d540d6bafa319ff41129b89/extensions/memory-core/src/memory/manager-embedding-policy.ts#L83)

- **memory_search still surfaces the failure as unavailable:** `memory_search` returns `buildMemorySearchUnavailableResult(...)` both when manager setup returns an error and when search execution throws. The unavailable payload is generic embedding/provider error handling, not a transient finalize-state recovery path.
