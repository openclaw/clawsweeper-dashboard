## Summary

The PR is a focused, source-backed fix and the patch appears correct: it keeps resolved secrets out of generated state, uses the repository’s existing SecretRef parser at the runtime boundary, and limits legacy audit suppression to the matching provider and current env ref. It should remain open for normal maintainer review because the linked issue is still owned by this open PR, and the contributor has not supplied after-fix real behavior proof beyond focused tests.

## What This Changes

The PR stores env-backed provider API keys in generated models.json as secretref-env markers, resolves those markers at model-registry use time, recognizes matching legacy bare markers during security audit, and updates related tests and docs.

## Best Possible Solution

Keep one canonical SecretRef marker format across generated models.json, resolve it only at credential use, retain the narrowly provider-matched audit allowance for stale generated files, and require a redacted real generation-to-audit-to-provider proof before merge.

## Reproduction Assessment

Yes, at source level: current generation can persist a bare env-ref name and the audit classifies that non-marker string as plaintext. The linked issue also gives a concrete generated-file and deep-audit path, although this review did not execute it.

## Solution Assessment

Yes. Reusing the established SecretRef marker/parser contract and resolving at the model-registry boundary is narrower and safer than persisting resolved secrets or adding a second credential representation; the remaining gap is real upgrade/runtime proof, not a confirmed code defect.
