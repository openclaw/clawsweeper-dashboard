## Summary

Close as implemented on current main: the reported v2026.6.8 Telegram Web unsupported-message regression is fixed by restoring standard Telegram sendMessage/HTML delivery as the default and making rich messages explicit opt-in. The fix is current-main-only and is not included in the latest v2026.6.8 release.

## What This Changes

The issue reports that Telegram Web renders OpenClaw Telegram bridge replies as unsupported messages after OpenClaw 2026.6.8.

## Best Possible Solution

Keep the current main behavior: standard Telegram sendMessage/HTML delivery by default, Bot API rich messages only through explicit richMessages opt-in, and ship that in the next patch release.

## Reproduction Assessment

Yes at source/report level for v2026.6.8: the shipped tag sends default Telegram text through sendRichMessage, and multiple user reports describe Telegram Web replacing accepted messages with unsupported placeholders. I did not run a live Telegram Web repro in this read-only review.

## Solution Assessment

Yes. Current main restores the broadly compatible sendMessage/HTML path as the default and keeps Bot API rich messages behind an explicit richMessages opt-in, which is the narrow owner-boundary fix for this regression.
