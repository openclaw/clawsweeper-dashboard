## Summary

The null timeout code is reproducible from current source, and the focused change correctly centralizes failure normalization without altering non-timeout signal exits. This PR should remain open for maintainer handling because it is member-authored and changes a plugin-visible result contract.

## What This Changes

The PR makes timeout and no-output-timeout results return code 124 when process closure yields null or zero and updates focused generic and Windows tests to assert that exact code.

## Best Possible Solution

Adopt code 124 as the canonical numeric status for both timeout modes at the shared helper boundary, retain null for non-timeout signal exits, and land after maintainers accept the plugin-facing compatibility change and required CI is green.

## Reproduction Assessment

Yes. Current main leaves a timeout code null when the killed child closes with `close(null, "SIGKILL")`, and the existing Windows test provides a high-confidence source reproduction path.

## Solution Assessment

Yes. Normalizing only timeout terminations in the shared helper is narrower and more reliable than repairing individual callers, while preserving null for genuine non-timeout signal exits.
