## Summary

Close as implemented on current main. The reported v2026.5.2 failure path is addressed by the current plugin-index design: legacy `plugins.installs` records are migrated into the state-managed plugin index, config writes are blocked or rolled back if that migration cannot preserve records, and plugin discovery uses persisted install records instead of requiring `openclaw.json` to retain `plugins.installs`.

## What This Changes

The issue asks `update.run` to preserve global plugin install metadata so npm-installed plugins survive the post-update gateway restart.

## Best Possible Solution

Keep the state-managed plugin index as the durable source of install metadata and let the next shipped build carry this main-branch fix rather than restoring `plugins.installs` as user config.

## Reproduction Assessment

No for current main: I did not run a live `update.run`, but source and regression tests show current main no longer requires `openclaw.json` to retain `plugins.installs`. The issue is source-supported against v2026.5.2, where the reporter’s logs and state match the pre-main-release failure mode.

## Solution Assessment

Yes. The current implementation preserves install records in the machine-managed plugin index and uses that index for discovery, which is narrower and more maintainable than reintroducing `plugins.installs` as durable user config.
