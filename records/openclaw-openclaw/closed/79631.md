## Summary

This PR should stay open: the linked Telegram tool-only-turn bug is source-reproducible on current main, but this patch is too broad and can add an extra `Done.` after a message tool already delivered visible content. The PR also still lacks after-fix real Telegram/Gateway proof, which is required for this external Telegram-visible change.

## What This Changes

The PR adds a `toolSummary.calls > 0` check in `runReplyAgent` and returns `Done.` when a run produces no payloads.

## Best Possible Solution

A maintainable fix should acknowledge direct/private local tool-only side-effect turns only when no committed message-tool, block, cron, approval, queued-followup, or source-reply delivery already produced visible output.

## Reproduction Assessment

Yes, source-level: current main still returns `undefined` for zero payloads and empty no-content payloads are normalized away before delivery. I did not establish a live Telegram/Gateway reproduction in this read-only review.

## Solution Assessment

No: using `toolSummary.calls > 0` is too broad because message tools are tool calls too. The safer solution is to key the acknowledgment off the absence of committed outbound delivery evidence and the direct/private delivery contract rather than generic tool-call count.
