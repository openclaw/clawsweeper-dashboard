## Label Justifications

- `P2`: This is a concrete gateway/systemd lifecycle bug with real service downtime impact but a limited platform and service-manager surface.
- `impact:crash-loop`: The report describes a self-sustaining supervised gateway restart loop where each process is killed before becoming healthy.
- `impact:message-loss`: The restart cascade repeatedly interrupts messaging-platform operators and can drop in-flight gateway work.

## Summary

Keep open: current main and v2026.5.12 still ship the 5s systemd restart delay and run service-mode stale process cleanup before the supervised `/healthz` arbitration. The open implementation at https://github.com/openclaw/openclaw/pull/83489 is a credible fix candidate, but it has not merged and may not cover the RestartSec/audit/docs part of this report.

## What This Changes

The issue asks OpenClaw to prevent systemd-supervised gateway restart loops when `RestartSec=5` fires before a warming gateway reaches `/healthz` and exit 78 can stop `Restart=always`.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/83489 if maintainers accept the healthy-listener ownership behavior, then close this issue only after deciding whether the systemd `RestartSec` default, audit, and docs also need to move above normal warmup time.

## Reproduction Assessment

No live high-confidence reproduction was run in this read-only review. Source inspection and reporter logs show the path: service-mode cleanup can run before `/healthz` arbitration, the default unit restarts after 5s, and exit 78 only fires after a healthy probe.

## Solution Assessment

Unclear for final fix depth. The active PR's early `/healthz` check matches the documented supervisor contract, but maintainers still need to decide whether the shipped systemd timing/audit/docs should also change.
