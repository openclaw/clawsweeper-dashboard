- `merge-risk: 🚨 compatibility`: Existing split-provider setups can change behavior on approved followup runs because the delivery channel becomes the fallback provider.

## Summary

Keep open: current main still has the exec approval routing gap, but this PR is not merge-ready because it makes the approved followup run inherit the delivery channel as the tool-policy provider and the supplied proof is test/mock-only.

## What This Changes

The PR adds `messageChannel` to exec defaults, forwards it from agent tool construction, and uses it for exec approval turn-source routing before falling back to `messageProvider`.

## Best Possible Solution

Preserve the delivery channel and the tool-policy provider as separate values through exec approval request and followup, with split-provider regression coverage and redacted real behavior proof.

## Reproduction Assessment

Yes at source level: current main sends exec approval `turnSourceChannel` from `messageProvider` even though adjacent agent/plugin approval routing already uses `messageChannel ?? messageProvider`. I did not run tests because this review is read-only.

## Solution Assessment

No: using `messageChannel` for approval delivery is the right direction, but this patch reuses that same value where the followup run later derives `messageProvider`. The safer fix carries delivery channel and tool-policy provider separately.
