## Summary

Current main already solves the observable avatar problem through the newer Control UI avatar pipeline: remote avatar URLs are rejected before rendering and replaced by the built-in logo/badge, local authenticated avatars are fetched into blob URLs, and failed local avatar fetches clear back to fallback rendering. The PR’s older on-error handler approach is therefore obsolete, and its root-absolute favicon subchange is not the direction current main took because the current tests retain route-relative bootstrap behavior.

## What This Changes

The PR adds Lit image-error fallbacks and keyed remounts for welcome and grouped assistant avatars, changes the empty-basePath logo URL to `/favicon.svg`, adds fallback CSS, and expands UI regression tests.

## Best Possible Solution

Keep the shipped Control UI avatar policy: block remote image fetches at the helper/CSP layer, render same-origin or blob avatars only after the authenticated avatar pipeline succeeds, and fall back to the bundled assistant logo/badge when no safe avatar is available.

## Reproduction Assessment

Yes. The deterministic check is the current `renderChatAvatar` test that renders an assistant avatar set to an `https://` URL and expects the bundled fallback image, plus the `refreshChatAvatar` path that clears failed local avatar fetches.

## Solution Assessment

Yes for current main, no for merging this old branch as-is. The shipped helper/CSP and authenticated blob pipeline is safer than adding remote image `error` handlers, and current tests intentionally keep no-basePath bootstrap paths route-relative rather than adopting the PR’s `/favicon.svg` change.
