## Label Justifications

- `P1`: The original failure mode could silently lose WhatsApp inbound messages after a gateway crash.
- `impact:message-loss`: The item is specifically about inbound channel messages being acknowledged before OpenClaw could safely process or replay them.
- `impact:crash-loop`: The reported loss happens when the gateway crashes or restarts during the acknowledgment-to-processing window.

## Summary

Current main now implements the requested WhatsApp durable inbound handoff: messages are persisted before dispatch, pending records replay on attach, retryable failures remain pending, and read receipts are delayed until finalization. The fix landed after v2026.5.22, so this is current-main-only until the next release.

## What This Changes

The issue reports WhatsApp inbound message loss when the gateway marks messages read before durable processing and then crashes.

## Best Possible Solution

Keep the merged plugin-state durable receive implementation and close this issue as fixed on current main, with the release gap resolved by the next OpenClaw release.

## Reproduction Assessment

No on current main: source inspection shows accepted WhatsApp inbound records now enter a durable journal before dispatch and read receipts happen after handler finalization. The old ack-before-process path is still visible in v2026.5.22, not in current main.

## Solution Assessment

Yes. The merged replacement fix uses plugin-state pending/completed journals rather than a WhatsApp-local JSON cache, which addresses durable acceptance, replay, retryable failures, and delayed read receipts in the right channel/plugin boundary.
