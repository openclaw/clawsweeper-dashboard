## Summary

Keep open. Current main still builds the gateway runtime from the slim Node image and the default runtime package list still omits `python3`; the optional apt hook only helps local image builds. The related open PRs are adjacent symlink/fs-safe work and do not restore user-authored workspace `.py` script execution in pulled official images.

## What This Changes

Restore official gateway Docker image support for `python3` so workspace and skill `.py` scripts that worked before the v2026.4.x slim-image switch do not fail at runtime.

## Best Possible Solution

Ship baseline `python3` in the published gateway runtime image again, add a Dockerfile regression assertion and image smoke proof, and align Docker docs plus the changelog with that runtime contract. If maintainers intentionally keep the gateway image interpreter-free, the missing-`python3` failure path and docs should make that breaking change explicit and actionable.

## Reproduction Assessment

Yes. The issue gives direct `docker run` comparisons between affected and last-known-good release images, and current main source still predicts the missing-`python3` result because the runtime base is slim and the default apt list omits `python3`.

## Solution Assessment

Yes. Re-adding baseline `python3` to the official gateway runtime is the narrowest compatibility restoration for pulled images; a docs-only apt-package recipe helps custom builds but would not fix users consuming release tags.
