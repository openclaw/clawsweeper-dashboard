## Summary

Close as implemented on main: active-memory now treats colon-bearing channel IDs as conversation IDs and falls back to the runnable provider channel, so QQ `c2c:...` values should no longer reach bundled plugin `dirName` validation.

## What This Changes

The issue requests that active-memory handle QQ direct-session keys like `c2c:<openid>` without failing bundled plugin directory validation.

## Best Possible Solution

Keep the current main fix that sanitizes active-memory channel selection while preserving strict bundled public-surface path validation, then ship it in the next release.

## Reproduction Assessment

Yes, by source inspection: QQ direct messages carry `qqbot:c2c:...` addresses, hook context can expose the scoped suffix as `channelId`, and the public-surface guard rejects colons in directory names. Current main prevents that colon-bearing value from being used as the embedded recall run channel.

## Solution Assessment

Yes, the current fix is the narrow maintainable path because it keeps the security/path guard intact and fixes the owner boundary in active-memory channel resolution instead of allowing colons in plugin directory names.
