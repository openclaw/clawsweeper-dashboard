## Summary

Keep open. Current main still has the reported split: TUI and devices CLI resolve gateway auth in the caller process before connecting, while the gateway daemon can resolve the same SecretRef in its own process context; changing that boundary is security-sensitive and needs maintainer review.

## What This Changes

The issue requests that TUI and CLI gateway management commands resolve `/etc/openclaw/secrets.json` file-backed `gateway.auth.token` SecretRefs consistently with the running gateway daemon.

## Best Possible Solution

Define the intended trust model first, then either document the required system-file provider setup and improve diagnostics, or add a deliberately scoped daemon-mediated/local-operator auth path with regression coverage.

## Reproduction Assessment

Yes at source level, but not as a live `/etc` run: TUI and devices CLI resolve the gateway token locally before connection, and the file provider rejects non-current-user or broadly readable files unless insecure mode is allowed. The reporter's exact service-user and file-permission setup still needs live confirmation.

## Solution Assessment

Unclear. The requested daemon-context resolution may be the right product direction, but it changes the credential boundary; a safer near-term path may be clearer diagnostics and documented `/etc` provider requirements unless maintainers approve a secure daemon-mediated design.
