## Summary

Keep open: this PR addresses a real `openclaw status` cold-path performance bug, but the current branch still has a P2 shared-session-store aggregation regression and carries an extra task-health surface that needs scope handling before merge.

## What This Changes

The branch disables manifest normalization in status model parsing, caches CLI-provider checks during status rendering, reuses per-agent session rows for global status sessions, and adds stale task health to CLI, gateway, SDK, and docs surfaces.

## Best Possible Solution

Land the status cold-path fix after rebuilding global session rows once per unique store without `agentIdOverride`, and split or explicitly accept the task-health surface separately.

## Reproduction Assessment

Yes, source-backed: the PR body and follow-up comment provide before/after timing and microcheck output, and current main still has the manifest-normalization and per-row CLI-provider call paths. I did not rerun the live 20s timing in this read-only checkout.

## Solution Assessment

No, not as submitted: the manifest-normalization opt-out is a good narrow direction, but the shared-store reuse change breaks global session aggregation. The safer solution is to reuse loaded stores while building global rows once per unique store without per-agent overrides.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Deduplicate shared session stores:** `src/commands/status.summary.ts:264`
  - body: When `session.store` lacks `{agentId}`, every configured agent resolves to the same store file. Appending rows already built with `agentIdOverride` makes global `sessions.count` and `sessions.recent` include each stored session once per agent, with shared rows stamped as the current agent. Build the global rows once per unique store without the override, or dedupe by store/key.
