## Summary

Keep open: the patch is technically sound and well-proven, but it intentionally narrows shipped voice-call status payloads under a protected `maintainer` label, so maintainer acceptance of the compatibility/API direction is still needed.

## What This Changes

This PR adds a voice-call status DTO projection and applies it to gateway and tool status responses, with tests asserting that sensitive `CallRecord` fields are omitted.

## Best Possible Solution

Merge the redacted status DTO only after maintainers explicitly accept the read-scope response contract; if raw call records remain necessary, keep them behind a separate privileged or history-specific surface.

## Reproduction Assessment

Yes. Current main and `v2026.6.10` source return raw `CallRecord` objects from read-scoped voice-call status paths, and the PR discussion provides exact-head live Gateway proof showing the redacted result after the patch.

## Solution Assessment

Yes technically: projecting a local DTO at the status egress points is the narrowest maintainable fix for the read-scope exposure. The unresolved part is maintainer acceptance of the shipped compatibility break and whether raw details need a separate privileged/history surface.
