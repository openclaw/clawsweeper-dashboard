## Summary

Current `main` already has builtin-memory fixes and regression coverage for the reported failure pattern: it bootstraps empty indexes on search, preserves the live DB across failed/safe reindexes, waits for in-flight sync before closing SQLite, and has follow-up recovery for stale SQLite handles and FTS-only reindex wipes.

## Evidence

- **Empty index now bootstraps on first search:** `extensions/memory-core/src/memory/manager.ts:289` forces `sync({ reason: "search", force: true })` when no indexed content exists, so a restart/empty-index state no longer stays permanently empty. Regression coverage is in `extensions/memory-core/src/memory/index.test.ts:481`.
  - file: [extensions/memory-core/src/memory/manager.ts:289](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/extensions/memory-core/src/memory/manager.ts#L289)
- **Shutdown race around `database is not open` is already addressed:** `extensions/memory-core/src/memory/manager.ts:561` returns early when the manager is closed, and `extensions/memory-core/src/memory/manager.ts:811` waits for pending sync/provider init before `this.db.close()`. Regression coverage is in `extensions/memory-core/src/memory/manager.async-search.test.ts:29`. The changelog also records this exact symptom as fixed in `CHANGELOG.md:4420`.
  - file: [extensions/memory-core/src/memory/manager.ts:561](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/extensions/memory-core/src/memory/manager.ts#L561)
- **Safe reindex keeps the live DB and reopens cleanly:** `extensions/memory-core/src/memory/manager-sync-ops.ts:1121` builds into a temp SQLite path, swaps only after success, and restores/reopens state on failure/success. Regression coverage in `extensions/memory-core/src/memory/manager.atomic-reindex.test.ts:29` verifies failed reindex keeps the prior index.
  - file: [extensions/memory-core/src/memory/manager-sync-ops.ts:1121](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/extensions/memory-core/src/memory/manager-sync-ops.ts#L1121)
- **Readonly/stale-handle recovery exists on current main:** `extensions/memory-core/src/memory/manager-sync-control.ts:82` detects readonly-handle failures, reopens the SQLite connection, restores schema/vector metadata, and retries once. Regression coverage is in `extensions/memory-core/src/memory/manager.readonly-recovery.test.ts:156`. The changelog records stale-handle auto-reopen in `CHANGELOG.md:3874`.
  - file: [extensions/memory-core/src/memory/manager-sync-control.ts:82](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/extensions/memory-core/src/memory/manager-sync-control.ts#L82)
- **FTS-only reindex no longer wipes chunks:** `extensions/memory-core/src/memory/manager.fts-only-reindex.test.ts:99` verifies forced reindex preserves indexed chunks when no embedding provider is available, matching the empty-rows symptom from the report. The changelog notes this fix in `CHANGELOG.md:1880`.
  - file: [extensions/memory-core/src/memory/manager.fts-only-reindex.test.ts:99](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/extensions/memory-core/src/memory/manager.fts-only-reindex.test.ts#L99)

## Risks / Open Questions

- I did not run a live repro in this checkout because the review was read-only and the local environment does not have `pnpm`, so the conclusion is based on current source, tests, and changelog evidence rather than an end-to-end runtime verification.
- The original report may have combined more than one memory failure mode, but the current mainline contains explicit fixes for each reported symptom cluster: closed-handle races, stale SQLite handles, empty-index bootstrap, and FTS-only reindex wipes.
