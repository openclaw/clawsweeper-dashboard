## Summary

Keep open: current main still hard-caps serialized inbound history at 20 despite configured channel/group history windows, but this PR should not merge as-is because its fallback can apply the group cap to histories produced with separate Slack DM or channel-specific limits.

## What This Changes

The branch adds a prompt-serialization history limit option, threads Discord's resolved history limit into inbound context, and adds a regression test for serializing more than 20 history entries.

## Best Possible Solution

Carry each producer-resolved history cap next to `InboundHistory`, have prompt serialization use only that explicit cap, and keep the safe 20-entry default when no producer cap is known.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: pass more than 20 structured `InboundHistory` entries while using a documented history limit above 20, and current main still serializes only 20. The PR regression is also source-reproducible through Slack DM history, where the collected DM cap can differ from `messages.groupChat.historyLimit`.

## Solution Assessment

No, not as written. Adding a serializer cap is the right direction, but falling back to the global group cap in `runPreparedReply` is not the narrow maintainable fix because history producers already resolve their own limits.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88
