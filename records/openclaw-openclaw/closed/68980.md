## Summary

Close as implemented on main: the Windows stale-gateway cleanup path now invokes `%SystemRoot%\\System32\\taskkill.exe` directly and treats taskkill failures as non-crashing cleanup outcomes, with matching regression coverage and a shipped changelog entry in `v2026.4.23`.

## Evidence

- **Windows stale-pid cleanup uses absolute taskkill path and swallows failures:** `terminateStaleProcessesWindows` builds `taskkill.exe` from `process.env.SystemRoot` + `System32`, then checks `spawnSync(...).error` / exit status instead of throwing through the cleanup path.
  - file: [src/infra/restart-stale-pids.ts:426](https://github.com/openclaw/openclaw/blob/a9797214338ba31c52c796adbb75afb16e0684a9/src/infra/restart-stale-pids.ts#L426)
  - command: `nl -ba src/infra/restart-stale-pids.ts | sed -n '420,470p'`
  - sha: [a9797214338b](https://github.com/openclaw/openclaw/commit/a9797214338ba31c52c796adbb75afb16e0684a9)
- **Gateway startup path calls the hardened stale-pid cleanup before bind:** Service-mode gateway startup calls `cleanStaleGatewayProcessesSync(port)` before binding the port, which is the path relevant to stale-listener restart failures.
  - file: [src/cli/gateway-cli/run.ts:441](https://github.com/openclaw/openclaw/blob/938b53698ec17934a5d1a6cffb279c584f64d9f3/src/cli/gateway-cli/run.ts#L441)
