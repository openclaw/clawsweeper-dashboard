## Summary

Keep open. Current `main` and the latest release still normalize every tool and then unconditionally apply `wrapToolWithBeforeToolCallHook`; the issue body is partly stale because the marker Symbol is now enumerable, but source inspection still shows a real double-wrap risk for already wrapped tools.

## What This Changes

The issue reports duplicate `before_tool_call` hook execution when already wrapped tools pass through `normalizeToolParameters` and `createOpenClawCodingTools`.

## Best Possible Solution

Fix the shared tool construction path so normalization preserves before_tool_call wrapper metadata and `createOpenClawCodingTools` cannot nest before_tool_call wrappers, with focused regression coverage.

## Reproduction Assessment

Yes from source: an already wrapped executable tool that reaches `normalizeToolParameters` and then the unguarded `createOpenClawCodingTools` hook map can be wrapped again, producing nested hook runners. I did not execute a failing test because this review was read-only.

## Solution Assessment

Unclear on the exact patch shape, but the best direction is narrow: preserve wrapper marker/source/context metadata across normalization and dedupe or intentionally rewrap at the `createOpenClawCodingTools` hook boundary.
