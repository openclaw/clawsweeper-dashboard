## Summary

Keep open for maintainer review: this PR has strong OpenShift proof and addresses the missing approval-store case, but the branch still falls back to the chmod-writing path when a readable empty or invalid `exec-approvals.json` already exists. The protected maintainer label and the overlapping maintainer PR at https://github.com/openclaw/openclaw/pull/86964 mean maintainers should choose the canonical landing path rather than closing this automatically.

## What This Changes

The PR adds a no-write fast path for missing default no-prompt exec approvals, preserves Mac exec-host socket persistence with an internal override, updates Kubernetes manifests to parent-mount the PVC with migration, and documents the storage shape.

## Best Possible Solution

Merge one canonical fix that resolves existing readable approval state without rewriting when the effective policy is default `full/off`, preserves fail-closed persisted/socket paths, and carries the Kubernetes parent-mount migration only if maintainers want that upgrade behavior in the same change.

## Reproduction Assessment

Yes. Current main source shows `resolveExecApprovals()` always calls `ensureExecApprovals()`, whose directory hardening rethrows non-Windows chmod errors; the PR body adds OpenShift fsGroup PVC proof for the missing-file case. I did not rerun the cluster smoke in this read-only review.

## Solution Assessment

No. The missing-file fast path is directionally narrow, but it should also handle existing readable empty or invalid default-equivalent stores before persisting, or maintainers should choose the sibling branch that already covers that case.
