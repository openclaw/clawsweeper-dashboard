## Summary

Close this PR because it bakes a downstream Telegram/Gemini/Outlook workspace and startup policy into the upstream root Docker image, while current OpenClaw keeps Docker generic and performs onboarding/configuration through documented setup paths.

## What This Changes

The PR adds a custom Docker entrypoint, template-generated OpenClaw config/auth files, a baked agent workspace gitlink, Python skill dependencies, and ignore rules for that workspace.

## Best Possible Solution

Keep the root Docker image generic; reusable seeding should be a reviewed opt-in mechanism or downstream image with tests, docs, pinned dependencies, and no embedded personal workspace or provider defaults.

## Reproduction Assessment

Yes for PR-introduced failures: source inspection shows the entrypoint exits unless custom Telegram, LLM, timezone, and user-name variables are set, while the current Docker path starts with --allow-unconfigured. The patch also calls envsubst without installing gettext-base by default.

## Solution Assessment

No. Baking a downstream Telegram/Gemini/Outlook workspace and startup policy into the upstream Dockerfile is not the narrow maintainable solution; a downstream image or generic opt-in seed hook is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.96

Full review comments:

- **[P2] Install venv dependencies before creating the venv:** `Dockerfile:262-264`
  - body: A plain documented Docker build reaches this unconditional RUN with only python3 installed; Debian's python3 package does not provide the venv/ensurepip pieces needed here, so the stock root image can fail before users ever get to onboarding.
  - confidence: 0.92
- **[P1] Preserve unconfigured Docker startup:** `scripts/openclaw-entrypoint.sh:16-19`
  - body: The new entrypoint exits unless Telegram, LLM, timezone, and user-name env vars are set, so the existing CMD with --allow-unconfigured no longer allows the generic container to start for onboarding or non-Telegram setups.
  - confidence: 0.96
- **[P2] Install envsubst before using it:** `scripts/openclaw-entrypoint.sh:35-36`
  - body: The first-run config path invokes envsubst, but the Dockerfile only mentions gettext-base in a comment and does not install it by default. Builds that get past the venv step can still fail on first startup when openclaw.json is missing.
  - confidence: 0.9
- **[P1] Do not copy an unreviewed gitlink into the image:** `Dockerfile:269`
  - body: The Dockerfile copies openclaw-team-workspace, but the PR adds that path only as a subproject commit. The image therefore depends on workspace contents that are not present in the reviewed diff and may be missing or unreviewed in normal checkouts.
  - confidence: 0.94

## Security Review

Status: needs_attention

Summary: The diff introduces Docker supply-chain, secret-handling, and startup-policy concerns in addition to functional build failures.

Concerns:

- **[high] Unreviewed gitlink is copied into the image:** `Dockerfile:269`
  - body: The PR adds openclaw-team-workspace as a subproject commit and copies it into /app/agent-workspace, making the image depend on external workspace contents that are not present in the reviewed diff.
  - confidence: 0.96
- **[medium] Unpinned PyPI package is installed at build time:** `Dockerfile:262`
  - body: The Dockerfile installs websockets from PyPI without a version, hash, lockfile, or repository dependency policy, weakening reproducibility and supply-chain posture for the root image.
  - confidence: 0.92
- **[medium] Raw envsubst writes secret-bearing config JSON:** `scripts/openclaw-entrypoint.sh:35`
  - body: The entrypoint substitutes environment variables directly into OpenClaw config and auth profile JSON, which can corrupt JSON structure and stores provider keys as inline plaintext instead of using existing config and SecretRef paths.
  - confidence: 0.86
- **[low] Insecure Control UI flag becomes a baked default:** `templates/openclaw.template.json:47`
  - body: The seeded config sets gateway.controlUi.allowInsecureAuth to true even though current audit code warns when that compatibility toggle is enabled.
  - confidence: 0.9
