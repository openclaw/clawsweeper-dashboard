## Summary

This PR is pointed at a valid CLI stale-session bug, but the current patch still lets a stale stored binding survive when the fresh retry fails with a different FailoverError reason.

## What This Changes

The PR widens CLI resumed-session recovery for timeout and unknown FailoverErrors, adds regression tests, and adds a changelog entry for #77089.

## Best Possible Solution

Clear the persisted CLI binding before starting the fresh retry for a reused-session failure, then keep the existing one-retry fallback behavior and persist any replacement binding after a successful fresh run.

## Reproduction Assessment

Yes. Source inspection on current main shows timeout and unknown CLI FailoverError paths can bypass stale-binding clear, and the PR's remaining blocker follows from the fresh retry rethrowing a different FailoverError reason before the caller clears the original stale binding.

## Solution Assessment

No, not as-is. Widening stale-session recovery is the right direction, but the binding clear needs to happen before the fresh retry or be driven by the original reused-session failure rather than the fresh retry's final error.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Clear the binding before retrying fresh:** `src/agents/cli-runner.ts:330-335`
  - body: This branch retries a resumed CLI session before the store-aware caller clears the persisted binding. If the original stale session fails with timeout or unknown and the fresh retry then throws a different FailoverError such as auth, rate_limit, or billing, the caller only sees that fresh reason and skips the timeout/unknown clear path, so the next turn can resume the same dead stored session again. Clear the stored binding before starting the fresh retry, or pass enough original-failure context to the caller to clear it reliably.
