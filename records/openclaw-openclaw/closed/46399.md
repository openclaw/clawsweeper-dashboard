## Summary

Current `main` already contains the voice-call runtime lifecycle changes that prevent a second webhook bind on the configured port: the plugin shares one memoized runtime across service startup and tool calls, startup failures clean up the webhook/tunnel state before retry, and the webhook server `start()` path is explicitly idempotent with regression coverage. The changelog records this fix under `v2026.3.2` as `Voice-call/runtime lifecycle`.

## Evidence

- **Shared runtime across startup and tool paths:** The plugin memoizes `runtimePromise`/`runtime` in `ensureRuntime()`, so gateway service startup and later `voice_call` tool invocations reuse the same runtime instead of creating a second webhook server.
  - file: [extensions/voice-call/index.ts:171](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/extensions/voice-call/index.ts#L171)

  - file: [extensions/voice-call/index.ts:559](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/extensions/voice-call/index.ts#L559)
