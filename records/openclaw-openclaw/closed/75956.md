## Summary

Keep open. Current main has diagnostics and adjacent plugin-tool discovery mitigations, but `resolvePluginTools` still invokes selected plugin tool factories during each tool-resolution pass, and open PR #75974 already targets this issue with a closing reference.

## What This Changes

The issue reports native-Windows Gateway turns spending about 78s in `core-plugin-tools` because plugin tool factories are rebuilt on every request, and asks for cached or lazy tool materialization plus instrumentation.

## Best Possible Solution

Land a context-safe plugin tool preparation change, likely an explicit cacheability contract or static-definition/per-run-binding split, then cover the hot path with regression tests and a changelog entry.

## Reproduction Assessment

Yes. The issue gives concrete native-Windows steps and timing logs, and current main still shows the reported factory-resolution mechanism; I did not perform a live Windows run in this read-only review.

## Solution Assessment

Unclear. A direct global cache like the open PR describes is not obviously the best fix because factories receive per-run context; a safer solution should define which factory outputs are stable before reusing them.
