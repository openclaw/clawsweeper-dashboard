## Summary

This is a valid Kubernetes fsGroup chmod regression, but it is already covered by the older canonical issue and an open fix PR that targets the same `exec-approvals` chmod path, so keeping this separate report open would split one root cause.

## What This Changes

The issue reports that exec approvals fail on Kubernetes CSI-backed PVCs because `ensureDir` rethrows `EPERM` from chmod on a root-owned `~/.openclaw` directory.

## Best Possible Solution

Track and land the vetted chmod-tolerance fix through https://github.com/openclaw/openclaw/pull/83695, while linking this report’s vSphere CSI/operator details as duplicate context for the canonical issue.

## Reproduction Assessment

Yes, at source level. Current main and `v2026.5.19` still rethrow non-Windows `chmodSync(dir, 0o700)` failures in `ensureDir`, and the issue provides a concrete Kubernetes fsGroup PVC state that makes that chmod return `EPERM`.

## Solution Assessment

Yes for the bug direction, but this report should not be tracked separately. The maintainable path is the existing canonical issue plus the open chmod-tolerance PR, with this issue linked for its additional CSI/operator reproduction detail.
