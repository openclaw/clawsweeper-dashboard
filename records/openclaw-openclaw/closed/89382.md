## Summary

Current main already handles this timeout/Copilot fallback problem: timeout cooldowns are model-scoped when the failed model is known, and transient timeout cooldowns can be probed so same-provider fallback models are not blocked by the shared auth profile.

## What This Changes

The issue reports that a GitHub Copilot request timeout marks the shared Copilot auth profile unavailable, causing same-provider fallback models to fail without being attempted.

## Best Possible Solution

Keep the current main behavior: transient timeout/rate-limit cooldowns stay model-aware or probeable, while billing/auth failures remain profile-wide, and ship it in the next release.

## Reproduction Assessment

No high-confidence failing reproduction remains on current main. The issue provides concrete logs for the older/released behavior, and current source plus regression tests now cover timeout-scoped cooldowns and same-provider fallback attempts.

## Solution Assessment

Yes. The current implementation fixes the right owner boundary by making auth-profile cooldown checks model-aware in the shared auth/fallback paths instead of adding a Copilot-specific workaround.
