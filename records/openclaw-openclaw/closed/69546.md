## Summary

Keep open. Current main still feeds QQBot inbound routing from the gateway startup config snapshot while bindings are classified as no-restart config, and open PR #73567 is the active closing implementation path.

## What This Changes

QQBot should apply newly added peer-specific bindings on the next inbound message instead of requiring a gateway restart.

## Best Possible Solution

The desired end state is for QQBot to resolve the active runtime config for each inbound route decision, with focused regression coverage for account-to-peer binding changes between messages.

## Reproduction Assessment

Yes. The issue provides concrete QQBot C2C steps and before/after routing logs, and current main still passes the startup `ctx.cfg` object into the route decision; I did not run a live QQBot account in this read-only review.

## Solution Assessment

Yes. Fetching the active runtime config at the QQBot inbound boundary is the narrowest maintainable fix; making binding edits restart-required or changing generic route-cache policy would be broader than necessary.
