- `impact:message-loss`: Inbound replies to affected delivery-only lanes errored instead of producing a delivered agent turn.

## Summary

Current main and the latest inspected release already handle the central bounce loop: transcript-only delivery mirrors are removed from replay, no-real-message compaction returns a non-compacted success, and preflight treats that as a skip rather than throwing.

## What This Changes

The issue asks OpenClaw to stop Telegram delivery-only lanes with only delivery-mirror transcript entries from repeatedly failing replyResolver preflight compaction.

## Best Possible Solution

Keep the shipped replay-filter and preflight skip behavior; if any v2026.5.27-or-newer lane still bounces, track that as a fresh, narrower session-registry hardening report with current logs.

## Reproduction Assessment

No current-main failure path remains from source inspection: mirror-only replay entries are filtered, no-real compaction returns a non-compacted success, and preflight skips that result. The reported v2026.5.22 logs are coherent, but they no longer match current main or the inspected v2026.5.27 release behavior.

## Solution Assessment

Yes. The current implementation uses the narrow maintainable path: keep transcript-only records visible in JSONL, remove them from provider replay, and let preflight continue when compaction has nothing real to summarize.
