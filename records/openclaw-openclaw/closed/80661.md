## Summary

Keep open: the PR is focused and current checks are green, but it still has a blocking effective-config validation gap where merged `runRetries` can violate `max >= min` after individually valid defaults and per-agent overrides are combined.

## What This Changes

This PR adds `agents.defaults.runRetries` and `agents.list[].runRetries` config/schema/docs/tests and wires the resolved value into the embedded Pi runner loop limit.

## Best Possible Solution

Validate or reject the effective merged `runRetries` before runtime clamping, add a focused regression for inherited `min` with per-agent `max`, then land only if maintainers want this public embedded-runner config surface.

## Reproduction Assessment

Not applicable: this is a feature/config PR, not a bug report. Current main source confirms the embedded run-loop retry boundary is hardcoded today.

## Solution Assessment

No: the typed config surface and embedded-runtime scope are reasonable, but the PR should validate the effective merged `min`/`max` relationship instead of silently clamping a per-agent cap below an inherited minimum.
