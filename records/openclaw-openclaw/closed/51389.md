Full review comments:

- **[P1] Extend strict model schema for fallbackOnErrors:** `src/config/types.agents-shared.ts:39`
  - body: `fallbackOnErrors` is added to `AgentModelConfig`, but `AgentModelSchema` is still strict and accepts only `primary` and `fallbacks`. A user following the new config example would get an unknown-key validation error before the feature can run.
  - confidence: 0.92
- **[P1] Honor defaults-level fallbackOnErrors:** `src/agents/agent-scope.ts:259-263`
  - body: The PR body documents `agents.defaults.model.fallbackOnErrors`, but this resolver reads only the selected agent's `model` object. Agents without a per-agent model object will ignore the documented defaults-level setting and keep the legacy fallback policy.
  - confidence: 0.88
- **[P2] Thread fallbackOnErrors through queued follow-ups:** `src/auto-reply/reply/agent-runner-utils.ts:171-175`
  - body: Adding the option to `resolveModelFallbackOptions()` covers the direct helper path, but current main's queued follow-up runner builds `runWithModelFallback()` arguments separately. Queued follow-up replies can therefore use different fallback behavior from normal turns for the same config.
  - confidence: 0.79
- **[P2] Stop image fallback on disallowed statuses:** `src/agents/model-fallback.ts:834-839`
  - body: `runWithImageModelFallback()` passes `fallbackOnErrors` into each attempt, but after a failed attempt the loop always records the error and advances to the next image candidate. A disallowed status such as 400 with `fallbackOnErrors: [429]` still burns through image fallbacks instead of failing fast.
  - confidence: 0.82
