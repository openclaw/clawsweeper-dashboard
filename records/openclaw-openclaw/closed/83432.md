## Summary

Current main already supports the requested skill scoping through per-agent skill allowlists, channel/topic skill filters, and cron agent selection, so the remaining `cron --skills` flag shape is not needed to solve the reported token-load problem.

## What This Changes

The issue requests per-agent, per-channel, or per-cron skill scoping so interactive sessions do not load digest/pipeline skills intended only for scheduled jobs.

## Best Possible Solution

Keep the current-main scoped-skill implementation: use `agents.defaults.skills` and `agents.list[].skills` for agent-level scoping, channel/topic `skills` where available, and `openclaw cron add --agent <id>` for cron jobs that need a different skill set.

## Reproduction Assessment

Not applicable as a feature request. Source inspection shows current main already has the requested scoped skill-loading paths for agents, supported channel/topic configs, and cron jobs through agent selection.

## Solution Assessment

Yes. The current implementation solves the reported prompt-bloat problem with one skill-filter contract shared by interactive sessions and cron, while avoiding a second per-job `--skills` source of truth.
