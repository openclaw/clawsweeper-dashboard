## Summary

Keep open: current main still waits for startup plugin runtime work before the channel sidecar phase, and heartbeat/cron remain tied to post-sidecar readiness. There is also an open linked implementation PR, https://github.com/openclaw/openclaw/pull/79756, so this should stay open while that design is reviewed or replaced.

## What This Changes

The issue requests dependency-driven gateway startup so Discord/channel sidecars, heartbeat, and cron are not blocked behind ACPX runtime readiness.

## Best Possible Solution

Land a dependency-aware startup path that prepares or pins channel registry state early, starts channel sidecars independently of slow ACPX runtime work, and gates only ACPX-dependent consumers on ACPX readiness.

## Reproduction Assessment

Yes for source-level reproduction: current main still awaits `plugins.runtime-post-bind` before entering channel sidecar startup, and scheduled services remain tied to post-sidecar readiness. I did not live-reproduce the reporter's Windows cold-start trace.

## Solution Assessment

Not as the one-file early-start approach in the linked PR: channel startup depends on a populated runtime channel registry. The maintainable solution needs an explicit dependency-ready startup seam or split plugin loading before channels are started.
