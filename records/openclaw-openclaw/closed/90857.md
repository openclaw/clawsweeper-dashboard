## Summary

Keep open: current main and the latest release only support time-window inbound debounce, while the requested count-based immediate trigger would add new public config and plugin-SDK behavior that needs maintainer product/API direction.

## What This Changes

Add a `messages.inbound.triggerCount` option that flushes a debounced inbound message batch immediately once enough same-window messages arrive.

## Best Possible Solution

Design one generic inbound debounce policy at the core/SDK boundary, keep time-only behavior as the default, and add count-trigger config, schema, docs, and focused tests only after maintainers approve the API shape.
