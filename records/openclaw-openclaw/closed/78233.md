## Summary

Keep open: current main still creates manual cron task records without marking the cron job active, while task maintenance relies on the active-job set before falling back to lost/recovery handling. The existing docs and tests define live cron runtime ownership as the expected guard, so this is a narrow valid bug rather than a stale duplicate.

## What This Changes

Long-running manual cron runs should remain backed by cron runtime ownership so task maintenance does not emit a transient lost status before final result reconciliation.

## Best Possible Solution

Mirror the existing active-job ownership contract in the manual-run producer path, add focused regression coverage, and leave broader durable-recovery policy to #68191.

## Reproduction Assessment

Yes, source-level: a long manual cron run creates a `runtime: "cron"` task with `lastEventAt` at start, but the manual path never marks the job active, so after the 5-minute grace `hasBackingSession` can return false while execution is still in progress. I did not run a live long-duration gateway scenario in this read-only review.

## Solution Assessment

Yes: reusing the existing `markCronJobActive`/`clearCronJobActive` contract around manual execution is the narrowest maintainable fix and matches the scheduled-run paths. The safer boundary is to avoid changing durable recovery or lost-task policy in the same patch.
