## Label Justifications

- `P1`: The reported bug caused Telegram users to receive incomplete assistant replies even when OpenClaw had the complete final answer.
- `impact:message-loss`: The affected Telegram delivery path dropped the tail of the assistant's final user-visible message.

## Summary

Current main and the latest release already cover the reported stale-prefix failure: final reply payload construction now prefers canonical final assistant text over streamed/intermediate text, and Telegram finalization has an additional transcript-backed guard for truncated preview finals.

## What This Changes

The issue reports that Telegram can receive a strict prefix of the complete final assistant response even though OpenClaw metadata has the full final text.

## Best Possible Solution

Keep the shipped canonical-final payload selection and ask reporters still seeing this on v2026.5.18 or newer to open a fresh issue with a new transcript/log sample showing the final payload and run metadata disagreeing.

## Reproduction Assessment

Yes for source-level reproduction: the current regression test models a streamed prefix and verifies the complete final assistant answer is selected. I did not live-replay the old Telegram incident in this read-only cleanup review.

## Solution Assessment

Yes. The maintainable fix is upstream of Telegram delivery, where final reply payloads are built from canonical assistant final content; the Telegram transcript-backed guard is useful defense-in-depth for preview finalization.
