## Summary

Keep open. Current main still treats `openclaw logs --follow` Gateway fetch failures as fatal, and open PR #75059 is already linked as the closing implementation candidate while PR #75372 overlaps the same retry surface.

## What This Changes

Request that `openclaw logs --follow` keep tailing across transient Gateway disconnects with warning/backoff/retry behavior instead of exiting immediately.

## Best Possible Solution

Review and land or adapt one focused follow-mode retry implementation, leaving broader streaming RPC architecture to a separate tracked change.

## Reproduction Assessment

Yes. Source inspection on current main shows the follow loop catches a `fetchLogs` transport failure, emits the Gateway error, and exits; the existing explicit-url test also asserts that fatal behavior.

## Solution Assessment

Yes in direction. The narrow maintainable fix belongs in `src/cli/logs-cli.ts` with targeted tests/docs, retrying only transient transport failures rather than changing all one-shot Gateway RPC calls.
