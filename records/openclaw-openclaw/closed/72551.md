## Summary

The reported false-success path does not appear fixable in the OpenClaw source tree: current core cron delivery only marks success after the channel outbound adapter returns delivery results, and adapter exceptions become error/not-delivered outcomes. The suspected WeChat session-guard/send behavior lives in the external `@tencent-weixin/openclaw-weixin` package, whose current npm source owns `session-guard.ts`, `channel.ts`, and the Tencent send API handling.

## What This Changes

Cron announce delivery to `openclaw-weixin` reports `delivered: true` even though the WeChat message does not arrive after the WeChat session token has expired.

## Best Possible Solution

Keep OpenClaw core channel-agnostic and fix the external WeChat plugin to throw when Tencent send responses indicate `ret`/`errcode` failure or session expiry. Once the plugin throws or returns no delivery result, current OpenClaw cron delivery will record the run as failed or not delivered instead of a successful announcement.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No live reproduction is available here because it requires an expiring WeChat session token and the external Tencent plugin/service, but code inspection gives a high-confidence ownership path: core handles adapter throws correctly, while the suspected session/API handling is external.

## Solution Assessment

Is this the best way to solve the issue? Yes for this repository: the maintainable fix is plugin-side API error validation in `@tencent-weixin/openclaw-weixin`, not a WeChat-specific core special case that would break the generic channel adapter boundary.
