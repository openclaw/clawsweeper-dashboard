Keep this PR open. Current main still keeps WebChat queued-message state in memory only, while the linked bug remains open and this PR is a focused implementation candidate for queue persistence. ## Risks / Open Questions

- The PR still appears to lack focused regression tests for restore, TTL expiry, dedupe, empty-queue cleanup, reconnect drain, and session-switch restore.
- The `handleConnected` restore/drain design should be reviewed carefully because lifecycle connection setup occurs before `connectGateway` finishes establishing the gateway client; a drain gated on `host.connected` may not fire on initial reconnect.
- Persisting queued message text and attachment data URLs in localStorage is a real privacy/storage tradeoff for the Control UI surface and should be explicitly accepted or adjusted before merge.
