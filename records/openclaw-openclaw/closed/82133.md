## Summary

Keep open: the PR addresses a valid Slack/plugin modal gap, but the current diff can route existing `openclaw:*` modal callbacks to plugin handlers by namespace collision and expose full modal state unintentionally; it also lacks live Slack behavior proof and carries the protected `maintainer` label.

## What This Changes

The PR routes Slack modal `view_submission` and `view_closed` events to plugin interactive handlers, preserves compact agent system events, updates Slack docs, tests, and the changelog.

## Best Possible Solution

Keep the compact system-event design, but require explicit plugin modal routing through `pluginInteractiveData` or a truly namespaced `openclaw:<namespace>:<payload>` callback format before exposing full modal state.

## Reproduction Assessment

Yes, source-reproducible: current main routes Slack modal submissions into compact system events while plugin interactive dispatch is wired for block actions. I did not run a live Slack workspace reproduction in this read-only review.

## Solution Assessment

No, not as written: the intended direction is reasonable, but the callback fallback is too broad. The safer fix is to require an explicit plugin marker or strict namespaced callback form while preserving existing `openclaw:*` modal system events.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Require explicit plugin routing for modal callbacks:** `extensions/slack/src/monitor/events/interactions.modal.ts:71-72`
  - body: This fallback turns every `openclaw:*` modal callback into plugin interactive data. Existing modal lifecycle tests use callback IDs like `openclaw:deploy_form` for core system events, so a plugin registered as `deploy_form` would now receive those submissions and full `stateValues` even though the modal never opted into plugin routing. Require `pluginInteractiveData` or a stricter `openclaw:<namespace>:<payload>` form before dispatching.
  - confidence: 0.9
