## Best Possible Solution

Land this focused lifecycle-owned cache if maintainers accept the cache lifetime; otherwise narrow the cache lifetime in this branch rather than reviving the older stale PR.

## Reproduction Assessment

Yes from source inspection: current main allocates a fresh realpath map per fingerprint build, so repeated mutable index fingerprinting cannot reuse realpath calls across builds. The linked issue supplies Windows profile context, but this read-only review did not rerun a Windows profile.

## Solution Assessment

Yes, this is the best current branch shape I found: it moves the cache to the installed manifest registry lifecycle boundary, bounds it, clears it through the existing lifecycle hook, and adds targeted tests. A narrower pass-scoped cache would avoid the merge risk but would not fit the existing process-stable plugin metadata cache pattern as cleanly.
