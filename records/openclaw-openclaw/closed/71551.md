## Summary

Current `main` already fixes the reported Control UI avatar 401 regression: the UI now fetches local `/avatar/:agentId` assets with Authorization headers and blob-wraps them instead of leaving the browser to request the protected route unauthenticated.

## Evidence

- **Regression path in released v2026.4.23:** The released Control UI code returned `avatarUrl` from `/avatar/:agentId?meta=1` and, when it decided not to do an authenticated follow-up fetch, set the local `/avatar/main` URL directly on the UI state. That leaves the browser to hit the protected avatar route itself, which matches the reported 401 symptom.
  - file: [ui/src/ui/app-chat.ts:610](https://github.com/openclaw/openclaw/blob/a9797214338ba31c52c796adbb75afb16e0684a9/ui/src/ui/app-chat.ts#L610)
  - command: `git show a9797214338ba31c52c796adbb75afb16e0684a9:ui/src/ui/app-chat.ts | nl -ba | sed -n '590,625p'`
  - sha: [a9797214338b](https://github.com/openclaw/openclaw/commit/a9797214338ba31c52c796adbb75afb16e0684a9)
- **Fix on main:** On `main`, `refreshChatAvatar` no longer bypasses the authenticated fetch for local avatar URLs. It always performs a JS `fetch()` against the local `/avatar/:agentId` endpoint with the resolved Authorization header and then stores a blob URL, which removes the unauthenticated browser request path.
  - file: [ui/src/ui/app-chat.ts:698](https://github.com/openclaw/openclaw/blob/fc5920fb51346a682b269fdbeeec0971ae4ae723/ui/src/ui/app-chat.ts#L698)

- **Targeted regression tests added:** UI tests now explicitly assert that both the metadata request and the local `/avatar/main` fetch carry the Authorization header for shared-token auth, covering the exact failure mode described in the issue. Adjacent coverage also checks device-token auth.
  - file: [ui/src/ui/app-chat.test.ts:213](https://github.com/openclaw/openclaw/blob/fc5920fb51346a682b269fdbeeec0971ae4ae723/ui/src/ui/app-chat.test.ts#L213)
