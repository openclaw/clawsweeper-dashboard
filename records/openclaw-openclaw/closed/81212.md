## Summary

Keep open: current main still filters CLI runtime providers out of `/models`, and https://github.com/openclaw/openclaw/pull/81239 is the open closing implementation candidate that should be reviewed or rejected before this issue is closed.

## What This Changes

The issue asks for `/models` provider pickers to make CLI runtime providers such as `claude-cli`, `codex-cli`, and `google-gemini-cli` discoverable again.

## Best Possible Solution

Review and either land or explicitly reject https://github.com/openclaw/openclaw/pull/81239, preserving runtime/auth validation and keeping https://github.com/openclaw/openclaw/issues/81216 as the separate catalog-refresh follow-up.

## Reproduction Assessment

Yes, at source level: current main classifies CLI runtime ids as legacy aliases, filters them before building `/models` providers, and Telegram renders that filtered provider array directly. I did not run a live Telegram bot in this read-only review.

## Solution Assessment

Unclear until maintainer review of the linked PR completes. Restoring CLI runtime providers as sibling buttons is the narrow active fix, but the longer-term runtime-selector UX and auth boundary should remain explicit maintainer decisions.
