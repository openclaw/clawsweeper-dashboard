## Summary

Current main already fixes the observable Telegram bug: inbound Telegram message IDs are carried as the current message ID, the shared reply-threading pipeline fills missing replyToId values for replyToMode="all", and Telegram delivery sends that resolved reply target. PR #50338's delivery-level defaultReplyToId fallback is now redundant, and the current shared implementation also avoids the edge cases raised in review around replyToCurrent=false, replyToMode="first", and explicit invalid reply IDs.

## Best Possible Solution

Close PR #50338 as redundant with current main. Keep the shared reply-threading implementation as the canonical fix because it handles final replies and block-streamed replies before Telegram delivery, preserves replyToMode semantics, and respects opt-out/explicit reply cases. If a future direct deliverReplies-only caller needs implicit default threading, that should be filed as a narrower follow-up with a concrete caller and test.
