## Summary

Current main and the shipped v2026.5.2 release already preserve nested OpenRouter model IDs through the Telegram `/model` path, so the reported `not allowed` behavior from 2026.4.27 is covered by maintained parser and regression-test behavior.

## What This Changes

The issue asks for Telegram `/model openrouter/...` selections on FreeBSD to accept configured OpenRouter models instead of returning `not allowed`.

## Best Possible Solution

Keep the shipped parser and Telegram native-command behavior, and route any remaining reports to update to v2026.5.2 or verify the full OpenRouter ref is present in `agents.defaults.models`.

## Reproduction Assessment

No failing current-main reproduction is present. Source inspection and shipped regression coverage show full OpenRouter refs are accepted when allowlisted; `not allowed` now points to a missing allowlist entry rather than Telegram parsing.

## Solution Assessment

Yes. Preserving first-slash provider parsing and validating against full allowed model keys is the narrow maintainable fix for OpenRouter model IDs containing slashes.
