## Summary

Current `main` already applies `gateway.controlUi.dangerouslyDisableDeviceAuth` in the Control UI WebSocket validation path, and the behavior is covered by regression tests. The changelog shows this bypass was restored in release `2026.3.13`, with later Control UI auth fixes also landing afterward.

## Evidence

- **Control UI auth policy enables bypass:** `resolveControlUiAuthPolicy()` sets `allowBypass` from `dangerouslyDisableDeviceAuth` and drops the device payload for Control UI sessions.
  - file: [src/gateway/server/ws-connection/connect-policy.ts:23](https://github.com/openclaw/openclaw/blob/350d3221800a18283915cd2f3b612ae04ddd490c/src/gateway/server/ws-connection/connect-policy.ts#L23)

  - file: [src/gateway/server/ws-connection/connect-policy.ts:121](https://github.com/openclaw/openclaw/blob/350d3221800a18283915cd2f3b612ae04ddd490c/src/gateway/server/ws-connection/connect-policy.ts#L121)

## Risks / Open Questions

- Older builds before the `2026.3.13` fix can still reproduce this behavior.
- The bypass is intentionally limited to operator Control UI sessions; node-role sessions still require device identity.
