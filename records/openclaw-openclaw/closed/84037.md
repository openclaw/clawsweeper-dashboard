## Summary

Keep open: this is a recent, concrete Codex runtime performance and diagnostics request. Current main includes related mitigations for keyed app-server clients and no-op post-tool/finalize relay commands, but the default Codex path still launches a local app-server, keeps native hook relay enabled, can still install PreToolUse helper relay work, and lacks separate app-server/helper/host-contention CPU attribution.

## What This Changes

The issue asks to reduce Codex app-server steady-state CPU, avoid expensive OpenClaw hook/helper child processes in normal interactive gateway use, add CPU attribution diagnostics, and document supported production settings for self-hosted Codex runtime deployments.

## Best Possible Solution

Define a measured Codex runtime performance plan: capture current-main idle and simple-turn profiles, set acceptable app-server/helper CPU targets, land focused relay optimizations such as the PreToolUse guard when safe, and document the supported self-hosted production profile.

## Reproduction Assessment

No high-confidence live reproduction of the reported 15-40% idle CPU on current main was established in this read-only review. Source inspection does confirm the relevant app-server, native hook helper, and diagnostics gaps, so a current Linux gateway/Codex profile is the next proof step.

## Solution Assessment

Not applicable; there is no proposed patch on this issue. The best direction is measured attribution first, then focused hot-path optimization and docs rather than closing this as solved by the related shared-client and post-tool relay mitigations.
