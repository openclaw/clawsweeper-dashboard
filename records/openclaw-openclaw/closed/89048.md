## Summary

Keep this issue open: current `main` still has the reported lexical-path check in the stale OAuth shadow scanner, and the auth JSON writer follows symlink targets before saving, so the described shared-store mutation path remains plausible and narrow enough to fix.

## What This Changes

The issue reports that `openclaw doctor --fix` can prune the shared main Codex OAuth profile when a non-main agent's `auth-profiles.json` is a symlink to the main auth store.

## Best Possible Solution

Add realpath-equivalence checks in both the scan and locked repair paths so a child auth store that resolves to the main auth store is treated as inherited, then cover that with a symlink regression test.

## Reproduction Assessment

Yes, source inspection gives a high-confidence repro path: create a main auth store, symlink a child `auth-profiles.json` to it, then run the stale-shadow scan/repair path. I did not execute it because the review is read-only.

## Solution Assessment

Yes, the requested realpath guard at scan time plus a recheck inside the locked repair write path is the narrow maintainable fix. It preserves the existing stale-shadow cleanup behavior for real local child stores while avoiding shared-store mutation.
