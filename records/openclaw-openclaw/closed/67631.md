## Summary

Keep open: current main still has source-backed macOS doctor and gateway service paths that can install, start, restart, or rewrite the default `ai.openclaw.gateway` LaunchAgent without detecting app-owned local mode. A linked open PR, https://github.com/openclaw/openclaw/pull/81725, explicitly owns the fix path, so this issue should remain open until that PR or a corrected successor lands.

## What This Changes

The issue asks OpenClaw to prevent Mac App and CLI doctor/gateway repair flows from creating or repairing competing macOS gateway LaunchAgent ownership that desynchronizes Mac App and Telegram sessions.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/81725 or a corrected successor that detects genuine OpenClaw.app launchd ownership and skips only default CLI gateway repair paths while preserving explicit profile services and adding regression plus macOS launchd proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level yes: current main still lets CLI doctor/gateway repair manage `ai.openclaw.gateway` without checking app-owned local mode, though I did not run a live macOS launchd repro from this Linux checkout.

## Solution Assessment

Is this the best way to solve the issue? Yes, the linked PR direction is the right boundary: detect real OpenClaw.app launchd ownership in the CLI repair paths and skip only the default service repair, while leaving named profiles and explicit operator-owned services alone.
