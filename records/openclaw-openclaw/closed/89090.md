## Label Justifications

- `P1`: The issue reports a real channel workflow producing 30+ duplicate visible messages and exhausting run retries despite loopDetection warnings.
- `impact:message-loss`: The affected behavior is duplicated user-visible channel messages from repeated identical `message.send` calls.

## Summary

Keep open: current main and the latest release still hash message-tool outcomes with volatile delivery identifiers, while the critical loop-detection paths require stable result hashes before blocking. The report is source-reproducible and has no merged canonical fix yet.

## What This Changes

The issue reports that repeated `message(action="send")` calls only warn because unique `messageId` values make each outcome hash look different.

## Best Possible Solution

Normalize loop-detection hashing for message send outcomes so volatile delivery ids do not hide repeated identical sends, then add focused regression coverage for direct, gateway, and plugin-handled message results.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: record repeated `message` sends with identical args but changing `messageId` in the result, then `detectToolCallLoop` cannot reach a critical branch because each `resultHash` differs.

## Solution Assessment

Yes, the best first fix is targeted message-send outcome normalization in loop detection. A broader input-only critical detector would change generic detector semantics and has more false-positive risk.
