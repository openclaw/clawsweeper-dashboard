Full review comments:

- **[P1] Do not reintroduce owner-only tool descriptors:** `src/agents/tools/owner-only-tools.ts:1-7`
  - body: Current main removed the old `ownerOnly` descriptor path and centralizes gateway owner-only behavior in the security/gateway policy layer. This PR adds a second `OPENCLAW_OWNER_ONLY_CORE_TOOL_NAMES` list under agent tools, which will drift from `GATEWAY_OWNER_ONLY_CORE_TOOLS` and does not match the current owner-boundary architecture after rebase.
  - confidence: 0.86
- **[P1] Gate task operations through owner-scoped helpers:** `src/agents/tools/task-tool.ts:129-140`
  - body: The new owner-only `task` tool still accepts `ownerKey` and uses global registry APIs for create/get/list/update/output. Owner-only gateway access only proves the caller is an owner; it does not prove they own every task record. Current main has `task-owner-access` helpers for this boundary, so this tool can expose or mutate tasks outside the caller's session owner scope unless every lookup/update goes through those helpers and ignores caller-supplied owner keys.
  - confidence: 0.88
- **[P2] Use flat enum schemas for new tool actions:** `src/agents/tools/task-tool.ts:31-39`
  - body: `Type.Union([...Type.Literal])` emits `anyOf` for the new action/runtime fields, while the current tool-schema contract provides `stringEnum`/`optionalStringEnum` specifically because some providers reject those unions. As written, the new `task` and `team` tools can be unavailable or invalid for providers that reject nested `anyOf` schemas.
  - confidence: 0.86
- **[P2] Classify allowed-mount escapes as workspace boundaries:** `src/agents/fs-permission-denied.ts:22-27`
  - body: Several new sandbox call sites wrap `Sandbox path escapes allowed mounts` as a permission-denied cause, but the classifier only checks sandbox-root and outside-root text. Those errors will be reported with `reason=unknown`, so callers looking for `workspace_boundary` will miss a real workspace-boundary denial.
  - confidence: 0.86
- **[P2] Return ToolInputError for missing team updates:** `src/agents/tools/team-tool.ts:87-88`
  - body: `updateTeam` throws a plain `Error` when the team does not exist, and `team` action `update` lets that exception escape. The gateway maps plain tool exceptions to HTTP 500, so a client typo like an unknown `teamId` is surfaced as an internal failure instead of a correct 400-style tool input error.
  - confidence: 0.82
