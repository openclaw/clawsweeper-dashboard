## Label Justifications

- `P3`: The item is a low-risk delivery ergonomics feature request and the central workflow already has a supported configuration path.
- `impact:message-loss`: The reported problem is duplicated channel messages during multi-turn tool/media delivery.

## Summary

Current main and the shipped v2026.5.20 release already cover the central request through message-tool-only visible replies: normal assistant text is suppressed while explicit message-tool payloads can deliver the final text/media once.

## What This Changes

Requests a delivery-hold mode that suppresses intermediate assistant text during multi-turn tool/media chains so chat channels only receive the final combined result.

## Best Possible Solution

Use the shipped message-tool visible reply mode for channels that need held delivery, and only reopen a narrower request if that supported mode cannot cover a concrete transcript/configuration.

## Reproduction Assessment

Not applicable: this is a feature request rather than a broken documented behavior report. Source, tests, and docs show the current delivery contract already suppresses ordinary assistant text in message-tool-only mode.

## Solution Assessment

Yes for the central problem: the shipped message-tool visible reply contract suppresses automatic text and lets the agent send the final text/media once. A parallel per-turn directive would duplicate that policy unless a new report shows a gap the existing mode cannot cover.
