## Summary

Keep open. The PR targets a real source-reproducible footer gap, but the current patch bypasses the message-tool-only privacy boundary by scraping suppressed private text for a `Usage:` line; the author has acknowledged the concern and needs to rework it plus add live proof.

## What This Changes

This PR extracts a `Usage:` line from suppressed message-tool-only final replies, sends it as a standalone final reply, and adds a regression test for that path.

## Best Possible Solution

Mark the actual response-usage footer with explicit metadata or emit it as a dedicated suppression-allowed payload, preserve sendPolicy and room-event privacy checks, and prove the Telegram forum-topic behavior live.

## Reproduction Assessment

Yes, by source inspection: `message_tool_only` suppresses final payloads while the usage footer is appended later as ordinary payload text. I did not run a live Telegram forum-topic repro in this read-only review.

## Solution Assessment

No. The PR is aimed at the right gap, but text scraping is not a safe provenance boundary; the footer needs explicit metadata or a dedicated payload before it can bypass suppression.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Avoid scraping suppressed replies for usage footers:** `src/auto-reply/reply/dispatch-from-config.ts:1689-1694`
  - body: In message-tool-only mode, normal final answers are intentionally private unless the model uses the message tool. This block publishes any suppressed line beginning with `Usage: `, so user- or model-produced private text can leak into a group/channel, and it also skips the existing room-event exclusion. Please carry explicit usage-footer provenance before allowing delivery despite source suppression.
  - confidence: 0.9
