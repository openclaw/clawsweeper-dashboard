## Summary

Close as implemented on current main: the runtime-deps repair path now prunes stale legacy version-scoped roots, Gateway startup runs the same shared cleanup before plugin loading, and package updates use the updated install restart/verification path rather than relying on the old process. The fix is present on main but not yet in a tagged release.

## What This Changes

The issue asks for update and Gateway runtime-deps cleanup so stale plugin-runtime-deps roots from prior OpenClaw versions cannot crash-loop Gateway after an OpenClaw update.

## Best Possible Solution

Keep the current shared cleanup and restart behavior on main, ship it in the next release, and track broader cache inspection/pruning UX separately from this crash-loop report.

## Reproduction Assessment

Yes. The current-main cleanup gap is now covered by a focused regression test that creates stale runtime-deps roots and asserts the intended removals; the original macOS launchd crash-loop was not replayed during this read-only review.

## Solution Assessment

Yes. The current solution is broader and safer than an update-only deletion hook because the policy runs through shared Gateway startup and repair paths, preserves active locks, and keeps compatible previous roots limited to dependency node_modules reuse instead of old plugin dist imports.
