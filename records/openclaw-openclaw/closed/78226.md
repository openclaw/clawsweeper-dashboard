## Summary

Keep open: this PR targets a real current-main exec-approval race, but the head still has correctness and security blockers, is currently conflicting, and lacks after-fix real node/gateway behavior proof.

## What This Changes

The PR adds locked, hash-guarded exec-approval writeback through node-host, gateway, and local CLI paths with regression coverage for stale approval metadata saves.

## Best Possible Solution

Keep the locked CAS direction, but finish it by guarding initialization/default writes, adding or replacing the process-scoped queue helper, updating CLI mocks, and attaching redacted real node/gateway race proof.

## Reproduction Assessment

Yes, at source level. Current main loads approvals before an async skill-bin refresh and later saves that stale object through recordAllowlistUse, while admin writes use a baseHash check; I did not run a live two-client node/gateway race.

## Solution Assessment

No as submitted. The hash/CAS writeback approach is the narrow maintainable shape, but this branch is still non-buildable, leaves one stale write path unguarded, and has stale focused CLI tests.
