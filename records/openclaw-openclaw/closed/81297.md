## Summary

Current main already routes agent turns, Gateway session model resolution, and `models status --agent` through agent-aware model selection, so the reported default-vs-agent mismatch no longer matches the code path. The fallback expectation in the report is handled by the existing per-agent `fallbacks` contract rather than by implicit fallback to `agents.defaults.model`.

## What This Changes

The issue asks for runtime LLM routing and diagnostics to honor `agents.list[].model.primary` before falling back to `agents.defaults.model.primary`.

## Best Possible Solution

Keep the centralized agent-aware model resolver on main, and use `agents.list[].model.fallbacks` when a per-agent primary should fail over to another provider.

## Reproduction Assessment

No live qwen/deepseek gateway turn was run, but current-main source and regression coverage give a high-confidence non-reproduction path: agent-aware resolution now selects the per-agent primary before the global default. The reported fallback-to-default behavior is not the current documented contract unless the agent entry defines `fallbacks`.

## Solution Assessment

Yes. Resolving the default model through `resolveDefaultModelForAgent({ agentId })` is the narrow maintainable fix because it centralizes per-agent precedence for CLI, Gateway, and runtime call sites without adding another config mode.
