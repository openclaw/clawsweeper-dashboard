## Summary

Keep this issue open. Current main still reads the heartbeat responsePrefix as a raw config string, does not pass the model-selection callback into the heartbeat getReplyFromConfig call, and then prepends that raw string during heartbeat reply normalization. The provided discussion also shows open same-author PR #43065 with closing syntax, so this issue should remain open until that PR or a successor fix is merged or closed on its own merits.

## Best Possible Solution

Keep this issue open and review #43065 or a successor patch. The fix should route heartbeat runs through the shared reply-prefix context, pass onModelSelected into getReplyFromConfig, resolve responsePrefix after model selection and before normalizeHeartbeatReply, and add focused regression coverage for templated heartbeat alert prefixes plus prefixed HEARTBEAT_OK suppression.
