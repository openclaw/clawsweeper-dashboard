## Summary

Keep open. Current main has `chat.history` response-size and Slack pong-timeout mitigations, but the stale-token Control UI retry loop remains source-reproducible because an `AUTH_TOKEN_MISMATCH` with no cached device-token retry still schedules reconnect.

## What This Changes

The issue asks OpenClaw to stop Control UI auth retry loops on stale gateway tokens and prevent `chat.history` work from starving Slack Socket Mode connectivity.

## Best Possible Solution

Make `AUTH_TOKEN_MISMATCH` terminal unless a cached device-token retry is actually queued, preserve the one-shot trusted retry path, and keep the `chat.history` timeout UX scoped to #44897 or a separate maintainer-approved patch.

## Reproduction Assessment

Yes for the remaining auth-loop behavior: clear stored device auth, connect with a stale shared token, receive `AUTH_TOKEN_MISMATCH`, and current source/tests expect a reconnect. The live Slack Socket Mode timing was not reproduced here.

## Solution Assessment

Yes. The narrow maintainable fix is reconnect gating in the browser and Node gateway clients, while the existing `chat.history` and Slack timeout mitigations remain separate supporting work.
