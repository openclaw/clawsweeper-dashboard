## Summary

Current `main` already handles this case: the shared poll-param detector only treats meaningful poll values as poll intent, the send-path guard still uses that detector, and regression coverage proves a media send with empty/default `poll*` fields is routed as a normal send. The fix was shipped in `v2026.4.22`.

## Evidence

- **Poll detection ignores empty/default values:** `hasPollCreationParams()` now returns true only for non-empty strings, non-empty string arrays, non-zero numeric values, or boolean `true`, so `pollQuestion=""`, `pollOption=[]`, `pollDurationHours=0`, and `pollMulti=false` are treated as unset.
  - file: [src/poll-params.ts:49](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/poll-params.ts#L49)
  - command: `nl -ba src/poll-params.ts | sed -n '1,260p'`
  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)
- **Send-path validation uses the narrowed detector:** `runMessageAction()` still rejects real poll misuse on `action === "send"`, but only when `hasPollCreationParams(params)` is true, so empty/default poll fields no longer misclassify a send as a poll.
