## Summary

Keep open. Current main still starts the voice-call initial greeting from the Twilio media-stream connect callback before the OpenAI realtime transcription session is connected, which matches the reported timeout/deaf-call behavior. Open PR #75257 already references this issue with closing syntax, so the issue should stay open until that PR lands or is replaced.

## What This Changes

Twilio inbound voice-call media streams using OpenAI realtime transcription time out during live calls while direct OpenAI WebSocket probes succeed, leaving caller speech untranscribed.

## Best Possible Solution

Review and either land PR #75257 or replace it with an equivalent narrow fix that establishes STT readiness before the initial greeting/TTS path runs, handles STT connect failure as a stream/call failure instead of a silent half-connected call, and adds focused regression coverage for media-stream startup ordering.

## Reproduction Assessment

Yes for triage: the issue includes concrete live steps, redacted config, exact logs, persisted call state, and a direct OpenAI WebSocket comparison. Current main still shows the matching startup order and timeout path, though I did not run a live carrier call locally.

## Solution Assessment

Unclear until PR #75257 is reviewed. Its stated direction, awaiting STT readiness before firing onConnect and the initial greeting, matches the current source-level race and appears narrower than increasing timeout values or changing provider defaults.
