## Summary

Current `main` no longer has the reported semantics. Bundled chat channels explicitly enabled under `channels.<id>.enabled` bypass `plugins.allow`, restrictive allowlists are auto-augmented for configured built-in channels, and the config help text now documents that behavior. The latest release tag `v2026.4.23` already contains the same implementation.

## Evidence

- **Bundled channel enablement bypasses plugin allowlist in current code:** `src/plugins/config-activation-shared.ts` marks bundled channels enabled via channel config as an explicit selection before the allowlist path, and only treats `plugins.allow` as an explicit selector for non-bundled plugins.
  - file: [src/plugins/config-activation-shared.ts:116](https://github.com/openclaw/openclaw/blob/6b618f063516aa9ba85903aa4565c5a204c055b2/src/plugins/config-activation-shared.ts#L116)

  - sha: [6b618f063516](https://github.com/openclaw/openclaw/commit/6b618f063516aa9ba85903aa4565c5a204c055b2)
- **Activation state resolver turns on bundled-channel allowlist bypass:** `src/plugins/config-state.ts` calls the shared activation resolver with `allowBundledChannelExplicitBypassesAllowlist: true`.
  - file: [src/plugins/config-state.ts:181](https://github.com/openclaw/openclaw/blob/6b618f063516aa9ba85903aa4565c5a204c055b2/src/plugins/config-state.ts#L181)

- **Tests cover the exact reported case:** `src/plugins/config-state.test.ts` asserts that `channels.telegram.enabled=true` keeps the bundled Telegram plugin active even when `plugins.allow` only contains `browser`; `src/config/plugin-auto-enable.core.test.ts` asserts configured built-in channels are preserved in restrictive allowlists.
  - file: [src/plugins/config-state.test.ts:335](https://github.com/openclaw/openclaw/blob/6b618f063516aa9ba85903aa4565c5a204c055b2/src/plugins/config-state.test.ts#L335)
