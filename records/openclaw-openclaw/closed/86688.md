## Label Justifications

- `P1`: A transient network flap can terminate the gateway and break a real Telegram channel workflow until an operator notices or restarts it.
- `impact:crash-loop`: The report is specifically about an uncaught exception that exits the gateway and can trigger supervisor crash-throttle behavior.
- `impact:message-loss`: While the gateway is parked, Telegram polling and sends stop, so channel messages can be missed or delayed until manual recovery.

## Summary

Keep open: current main handles several Telegram and gateway network failures, but ENETDOWN is still missing from both the Telegram recoverable/pre-connect taxonomy and the global benign uncaught network classifier, so the reported crash family is not fully covered.

## What This Changes

The issue reports gateway termination when transient ENETDOWN or related network errors escape Telegram/SSRF guarded fetch, leaving macOS LaunchAgent-supervised gateways parked until restart.

## Best Possible Solution

Extend the existing transient/pre-connect network error taxonomy to include ENETDOWN across Telegram transport and global uncaught handling, add focused regression tests, and leave operator watchdogs as workarounds rather than the product fix.

## Reproduction Assessment

No live, high-confidence reproduction was established in this read-only review because the trigger depends on a network interface flap during connect. Source inspection does show the reported ENETDOWN code is not currently classified by the Telegram or global non-fatal network handlers.

## Solution Assessment

Yes, the narrow maintainable fix is to add ENETDOWN to the existing network-error classification and regression tests. Adding macOS watchdog behavior would only mask the crash instead of preventing the transient connect error from reaching the fatal handler.
