## Summary

Current main already addresses the reported unregistered Codex harness failure by loading the Codex runtime owner before harness selection and by avoiding the unregistered-harness throw for implicit Codex runtime selection. The fix is current-main-only from the available tags, so it is not shipped in v2026.5.12 yet.

## What This Changes

The issue reports a regression where Codex-primary agents on 2026.5.12 dispatch `openai-codex/*` requests to an unregistered `codex` harness and fall back or fail.

## Best Possible Solution

Keep the current-main runtime availability fallback and Codex plugin preloading behavior, then ship it in the next release while continuing to steer legacy `openai-codex/*` config through doctor repair.

## Reproduction Assessment

No current-main failing reproduction remains from source inspection: the current selection path either loads the `codex` plugin before selection or avoids throwing for implicit missing Codex runtime. The v2026.5.12-to-main diff shows the exact availability fallback added after the shipped release.

## Solution Assessment

Yes. The current approach preserves fail-closed behavior for explicit `agentRuntime.id: "codex"` while preventing implicit OpenAI/Codex default routing from surfacing an unregistered-harness error.
