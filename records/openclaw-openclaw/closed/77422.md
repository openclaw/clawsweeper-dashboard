## Summary

Close as implemented on current main: session-only subagent announce calls now resolve to an internal WebChat run at the gateway agent boundary, so the run-level Control UI visibility gate receives `webchat` and `server-chat` does not suppress the live websocket broadcast for the reported path.

## What This Changes

The issue requests live WebChat websocket delivery for subagent completion announcements to idle WebChat-bound parent sessions, instead of requiring a UI refresh to see the durable history update.

## Best Possible Solution

Keep the current gateway-agent behavior that maps session-only internal WebChat announce runs to WebChat visibility, and add a narrower regression test if this path changes again.

## Reproduction Assessment

No live reproduction was established in this read-only pass. Against current main, the reported path no longer reproduces by source inspection because the queue-drained announce enters `agent` as session-only and is converted to a WebChat-visible run before the visibility gate.

## Solution Assessment

Yes for the observable issue: resolving the session-only announce as an internal WebChat run at the gateway boundary is narrower than adding special-case broadcast behavior in `server-chat`. A future cleanup could still centralize the visibility helper for clarity.
