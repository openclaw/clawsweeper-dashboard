## Summary

Current main and the shipped v2026.4.27 release cover the actionable service-lifecycle causes from the WSL2 report: doctor now blocks the duplicate system/user systemd install path, audits and repairs stale pinned service ports, and documents the exact recovery signatures. The report's kernel-level ghost-socket theory was not separately proven, but the observable failure modes and maintainer-facing recovery path are implemented well enough to close this cleanup item.

## What This Changes

The issue reports a WSL2 gateway crash loop after v2026.4.24, tied to duplicate system/user systemd services, stale hardcoded service ports, strict config validation, and a suspected gateway listener/watchdog race.

## Best Possible Solution

Keep the v2026.4.27 service-lifecycle behavior: use doctor/status to detect duplicate supervisors and stale pinned ports, and ask for a fresh issue only if a clean v2026.4.27+ WSL2 install with no duplicate supervisors still shows a listener hang.

## Reproduction Assessment

Yes for the deterministic service-lifecycle parts: the report gives a WSL2 setup with system/user systemd overlap and a changed gateway.port ignored by the installed service, and current tests cover those repair points. A live WSL2 ghost-socket reproduction was not run in this read-only review.

## Solution Assessment

Yes. The shipped fix is the narrow maintainable path: repair doctor/service audit and startup bind handling rather than adding startup-time core migrations or extension-specific policy.
