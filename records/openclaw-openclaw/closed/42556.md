## Summary

Current `main` already handles the Windows plugin-install npm path without enabling `shell: true`: plugin dependency installs still go through `runCommandWithTimeout(...)`, and the Windows exec layer rewrites `npm`/`npx` to `node + npm-cli.js` when available or wraps `.cmd` fallbacks through `cmd.exe /d /s /c`. The issue’s stated root cause (`shouldSpawnWithShell` being false) is outdated in current code and in the latest release.

## Evidence

- **Plugin installs still use the shared exec path for `npm install`:** `install-package-dir` runs plugin dependency installation via `runCommandWithTimeout(["npm", "install", "--omit=dev", "--silent", "--ignore-scripts"], ...)`, so Windows plugin installs flow through the current process execution shim.
  - file: [src/infra/install-package-dir.ts:245](https://github.com/openclaw/openclaw/blob/7d9172d5e093435d37ed3e00e476a4eaf84a252b/src/infra/install-package-dir.ts#L245)
  - sha: [7d9172d5e093](https://github.com/openclaw/openclaw/commit/7d9172d5e093435d37ed3e00e476a4eaf84a252b)
- **Windows exec path avoids direct `.cmd` spawn EINVAL:** `src/process/exec.ts` documents the Windows `EINVAL` problem, rewrites `npm`/`npx` to `node + npm-cli.js` when possible, and otherwise falls back to `.cmd` plus an explicit `cmd.exe /d /s /c` wrapper. `shouldSpawnWithShell(...)` remains `false` intentionally for argv-injection hardening, not because Windows compatibility is missing.
  - file: [src/process/exec.ts:45](https://github.com/openclaw/openclaw/blob/7d9172d5e093435d37ed3e00e476a4eaf84a252b/src/process/exec.ts#L45)

## Close Comment

Closing from a Codex review because current OpenClaw already implements this Windows plugin-install path. `src/infra/install-package-dir.ts:245` still installs plugin dependencies through `runCommandWithTimeout(["npm", "install", ...])`, and `src/process/exec.ts:45` now handles Windows by rewriting `npm`/`npx` to `node + npm-cli.js` when available, or falling back to `.cmd` wrapped through `cmd.exe /d /s /c` (`src/process/exec.ts:63` and `src/process/exec.ts:105`). `shouldSpawnWithShell(...)` intentionally stays `false` for injection hardening (`src/process/exec.ts:117`, `src/process/exec.test.ts:94`), so the original root-cause analysis no longer matches current code. There are direct regression tests for both the `node + npm-cli.js` path and the `npm.cmd` fallback in `src/process/exec.windows.test.ts:184` and `src/process/exec.windows.test.ts:212`, and the same implementation is present in release `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). If someone still hits a Windows plugin install failure on a current build, please open a new issue with the exact OpenClaw version, runtime (Node/Bun), full install command, and full error output.
