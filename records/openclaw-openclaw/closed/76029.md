## Summary

Close as implemented: current main and the shipped v2026.4.29 startup path already keeps provider warmup scoped, metadata-only, and bounded, and channel startup no longer waits for a hung primary-model warmup.

## What This Changes

The request asks for bounded, nonblocking provider auth/discovery during Gateway startup so an unreachable Codex token endpoint cannot hang the Gateway.

## Best Possible Solution

Keep the shipped scoped startup warmup and timeout behavior; any remaining Windows/Node 24 hang should be reopened with startup trace logs that identify a different sidecar or request-time auth path.

## Reproduction Assessment

No, not from the current code path described in the report. The claimed every-provider startup token exchange does not match current startup source: only the configured primary provider is warmed, discovery is metadata-only and bounded, and tests cover hung warmup not blocking channel startup.

## Solution Assessment

Yes. The existing scoped, nonblocking startup warmup is narrower than moving provider initialization to workers, and runtime OAuth refresh already has a hard upper bound for actual token exchanges.
