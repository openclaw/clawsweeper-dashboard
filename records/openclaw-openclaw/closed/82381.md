## Summary

Current main already implements the central fix: isolated Telegram polling no longer awaits each spooled handler inside the drain loop, and the current lane-scoped background handling is covered by tests. The fix is on main only and is not yet in the v2026.5.12 release the reporter tested.

## What This Changes

The issue asks for isolated Telegram polling to avoid blocking the whole spooled-update drain when one `bot.handleUpdate` call is slow or hung.

## Best Possible Solution

Keep the current main lane-scoped implementation and ship it in the next release, rather than adding a blind per-update timeout that could race duplicate/background handlers on the same Telegram lane.

## Reproduction Assessment

No. The v2026.5.12 shape described by the report is valid, but current main no longer has a drain loop that awaits one handler before considering later spooled updates.

## Solution Assessment

Yes. The current fix is the safer maintainable solution: independent lanes and control updates can proceed, while same-lane ordering remains intact and a wedged backlog is surfaced as unhealthy.
