## Summary

Close as plugin-owned: the report describes valid Lossless Claw/LCM state semantics, but OpenClaw core already routes native `/new` and `/reset` to the target session and exposes reset/session lifecycle hooks that a context-engine plugin can use. Hardcoding LCM SQL into Discord or Telegram handlers would cross the core/plugin boundary that the repo explicitly keeps lean.

## What This Changes

Have Discord and Telegram `/new` and `/reset` archive the active Lossless Claw/LCM conversation in addition to rotating the OpenClaw session JSONL.

## Best Possible Solution

Keep OpenClaw core generic and implement the LCM archive/active-row reset inside the Lossless Claw plugin using `before_reset` or `session_end`; only add a new generic plugin API if those existing hooks prove insufficient.

## Reproduction Assessment

Not applicable as an OpenClaw-core reproduction: the concrete failure depends on external Lossless Claw database state. Source inspection confirms the core boundary: reset rotates OpenClaw sessions and exposes plugin lifecycle hooks rather than owning LCM conversation rows.

## Solution Assessment

No, the suggested core SQL in Discord/Telegram handlers is not the best fix because it hardcodes an external plugin schema into core channel code. The maintainable solution is plugin-owned reset handling through the existing reset/session lifecycle hooks.
