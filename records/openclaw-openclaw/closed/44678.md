## Summary

Close #44678 as superseded. The PR targets the same #44550 Ollama `params.num_ctx` bug, but current main has since moved the Ollama stream into the bundled plugin and a newer open PR, #69464, tracks the same remaining work against that current architecture with schema/runtime model plumbing, native and OpenAI-compatible Ollama payload changes, regression tests, and live Ollama verification.

## Best Possible Solution

Close #44678 as superseded by #69464. Keep the remaining work focused on the newer current-architecture PR or an equivalent patch that adds model `params` schema support, threads params onto the runtime model, and resolves Ollama `num_ctx` from explicit config before falling back to discovered `contextWindow`.

## Evidence

- **Current native Ollama behavior:** Current main still builds native Ollama `options.num_ctx` from `model.contextWindow ?? 65536` and then calls `options?.onPayload?.(body, model)`. This means the hook portion of #44678 is already present in the current plugin stream, but configured `params.num_ctx` is not the native source of truth on main.
