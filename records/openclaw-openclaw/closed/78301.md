## Summary

Keep open: adjacent Weixin/core failures have since been fixed, but current main still lacks plugin-aware diagnostics for invalid plugin `Content-Length` failures and split runtime module state. The useful remaining work is a maintainer-owned diagnostics/security-contract decision, not conservative cleanup.

## What This Changes

Request clearer plugin loader/runtime diagnostics for invalid legacy plugin contracts, especially plugin-supplied `Content-Length`, split runtime module state, and migration guidance.

## Best Possible Solution

Preserve hard fetch protections, add plugin-named diagnostics that surface the underlying undici cause, and document or warn about runtime-store-safe patterns for split runtime modules.

## Reproduction Assessment

Yes, for the source-level diagnostic gap: current main still passes ordinary manual `Content-Length` through to runtime fetch and does not translate the resulting undici cause with plugin context. The corrected missing-runtime case is not reproducible because `api.runtime` is still injected.

## Solution Assessment

Unclear as proposed: better diagnostics are the right direction, but converting hard `Content-Length` rejection into strip-and-warn is a security policy change. The safer maintainable path is enriched diagnostics while keeping rejection behavior unless maintainers decide otherwise.
