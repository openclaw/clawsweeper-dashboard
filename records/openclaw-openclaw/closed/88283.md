## Summary

Keep open. The npm-installed shrinkwrap fix is useful, but the branch is not merge-ready because its early shrinkwrap fallback can still misclassify supported pnpm global package-root layouts as npm.

## What This Changes

The PR changes package-manager detection to prefer `npm-shrinkwrap.json` over `package.json.packageManager` when no same-root `pnpm-lock.yaml` exists, adds an npm shrinkwrap dependency marker fallback, and adds focused tests.

## Best Possible Solution

Preserve package-root ownership first, then apply the npm shrinkwrap fallback only when npm ownership is the best-supported interpretation, with regression coverage for npm, pnpm direct global, and pnpm virtual-store layouts.

## Reproduction Assessment

Yes, at source level. Current main reads `packageManager` before shrinkwrap markers, and the PR's remaining compatibility problem follows from existing pnpm global layout code where ownership markers live outside the package root.

## Solution Assessment

No. The proposed npm fallback is the right direction, but same-root `pnpm-lock.yaml` is too narrow; the detector needs to preserve current pnpm-owned package roots before treating shrinkwrap as npm ownership.
