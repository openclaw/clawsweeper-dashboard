## Label Justifications

- `P1`: The report describes a recent production recovery failure that can leave real Telegram agents stuck after an apparent session reset.
- `impact:session-state`: The affected behavior is divergent session stores and stale running session records under an isolated HOME.

## Summary

Keep open: current main still has process-env-derived session store resolution that can point at a Codex-isolated HOME, and I did not find a guard or doctor detector for nested Codex shadow session stores. The closed related reports cover adjacent stall/reset bugs, while the remaining recovery split-brain behavior is still distinct.

## What This Changes

The issue asks OpenClaw to prevent session recovery commands from reading or mutating a Codex-isolated shadow state tree instead of the canonical gateway session store.

## Best Possible Solution

Add a shared Codex-shadow state guard and doctor detector; online recovery should use the running gateway's canonical state, while offline writes should require explicit canonical env or --store and print the target store path before mutation.

## Reproduction Assessment

Yes from source inspection: without canonical OPENCLAW_STATE_DIR or OPENCLAW_CONFIG_PATH, CLI session-store resolution follows the process HOME and can select a nested Codex shadow store. I did not run a live mutating repro in this read-only review.

## Solution Assessment

Unclear as a product choice: the report lists warning, refusal, gateway-derived state, explicit store flags, and doctor repair as viable directions. The narrowest maintainable path is shared shadow detection plus gateway-first online recovery semantics.
