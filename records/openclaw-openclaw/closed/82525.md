## Summary

Keep open. Current main caps Telegram startup getMe fanout and has newer startup handoff tracing, but it still permits two concurrent startup probes with a 15s minimum timeout and three in-budget attempts, so the reported post-limiter slow-Bot-API stalls still need maintainer profiling and policy judgment.

## What This Changes

The issue reports Telegram bot startup getMe retries causing gateway event-loop stalls during Bot API slow periods and requests tighter timeout, backoff/degrade behavior, and account startup staggering.

## Best Possible Solution

Profile current-main on an affected multi-account Telegram setup, then add a Telegram-local fail-fast/degrade policy only if live proof shows the remaining two-probe startup path still starves the gateway.

## Reproduction Assessment

No high-confidence current-main reproduction was established. Source confirms the startup getMe path, two-probe limiter, 15s minimum timeout, and retry loop, but the reported event-loop starvation still needs live or induced Bot API slowness proof.

## Solution Assessment

Unclear. A tighter timeout or fail-open backoff may be useful, but it conflicts with the current slow-Telegram timeout contract and should be chosen after profiling rather than applied mechanically.
