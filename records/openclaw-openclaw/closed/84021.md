## Summary

This is a real DingTalk connector behavior request, but current main has no in-tree DingTalk transport and the OpenClaw contract puts platform-native outbound shaping in the channel plugin, so it should be handled in the DingTalk connector/ClawHub path rather than kept open as a core issue.

## What This Changes

The report asks DingTalk announce delivery to convert `<@数字>` mention text into DingTalk API `atDingTalkUserIdList` entries so notifications fire.

## Best Possible Solution

Have the DingTalk connector implement `<@id>` parsing and `atDingTalkUserIdList` population in its outbound send path, publish/update it through ClawHub or its official external catalog path, and bring a new OpenClaw issue only if a generic SDK seam is missing.

## Solution Assessment

Yes; routing the fix to the DingTalk connector/ClawHub path is the maintainable solution because OpenClaw core should not hard-code DingTalk mention payload fields.
