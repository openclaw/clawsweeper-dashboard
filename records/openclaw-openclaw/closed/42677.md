## Summary

Close as superseded. Current main still has the automatic daily/idle rollover memory-persistence gap, but that remaining work is now consolidated under open tracker #11960 plus newer active implementation PRs #71792 and #61675; this older branch uses a parallel `session:rollover` internal event and still carries the duplicate-reset concern from review.

## What This Changes

This PR adds a new internal `session:rollover` hook for automatic stale-session rollover, wires bundled `session-memory` to persist the previous session on that event, and updates related hook metadata, docs, and tests.

## Best Possible Solution

Consolidate on #11960 and the newer active implementation path, especially #71792 or #61675. The final fix should make lazy daily/idle rollover persist the prior session exactly once through the selected reset or lifecycle hook contract, preserve existing manual `/new` and `/reset` behavior, and keep focused daily, idle, fresh-session, and ACP-bound reset regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: make a session stale by daily or idle policy, send the next non-command inbound message, and current main creates a new session with `resetTriggered` false while `session-memory` remains gated to `command:new`/`command:reset`.

## Solution Assessment

No. This PR’s direction is useful, but the best current solution is not to merge this older parallel `session:rollover` contract as-is; maintainers should land or explicitly supersede it with the newer current-base reset-hook implementation and duplicate-write coverage.
