## Summary

Current `main` does not implement this QQ Bot TTS fix yet, and the PR still has a plausible unresolved correctness issue around mixed remote-image-plus-voice payloads. Conservative cleanup should keep the PR open for maintainer review rather than auto-close it.

## Evidence

- **Current main still drops voice/audio MIME from the finalized inbound context:** `buildCtxPayload()` only copies `MediaTypes`/`MediaType` from `inbound.localMediaTypes` when `localMediaPaths` exist. Voice attachments are exposed separately as `QQVoiceAttachmentPaths`/`QQVoiceAttachmentUrls`, and the remote-media branch adds URLs without any MIME types, so voice-only inbound QQ Bot messages still do not mark the context as audio here.
  - file: [extensions/qqbot/src/engine/gateway/outbound-dispatch.ts:352](https://github.com/openclaw/openclaw/blob/11804a484ded89374afa55020813cfa265aa143a/extensions/qqbot/src/engine/gateway/outbound-dispatch.ts#L352)
  - command: `nl -ba extensions/qqbot/src/engine/gateway/outbound-dispatch.ts | sed -n '350,430p'`
  - sha: [11804a484ded](https://github.com/openclaw/openclaw/commit/11804a484ded89374afa55020813cfa265aa143a)
- **Inbound pipeline still classifies only image URLs into media-type arrays:** `buildInboundContext()` populates `localMediaTypes`/`remoteMediaTypes` by iterating `imageUrls` only, while voice attachments are kept in `uniqueVoicePaths`/`uniqueVoiceUrls`. That matches the PR’s stated root cause and shows the bug is still present after the gateway refactor.
  - file: [extensions/qqbot/src/engine/gateway/inbound-pipeline.ts:241](https://github.com/openclaw/openclaw/blob/11804a484ded89374afa55020813cfa265aa143a/extensions/qqbot/src/engine/gateway/inbound-pipeline.ts#L241)
  - command: `nl -ba extensions/qqbot/src/engine/gateway/inbound-pipeline.ts | sed -n '220,380p'`
  - sha: [11804a484ded](https://github.com/openclaw/openclaw/commit/11804a484ded89374afa55020813cfa265aa143a)
- **Framework TTS inbound detection still depends on audio MIME or explicit audio body markers:** `isInboundAudioContext()` first checks `ctx.MediaType`/`ctx.MediaTypes` for `audio`/`audio/*`, then falls back to `<media:audio>` or `[Audio` body markers. QQ Bot’s current inbound body is built from the formatted envelope plus `- Voice:` dynamic text, not those audio placeholders, so missing audio MIME remains material to behavior.
  - file: [src/auto-reply/reply/dispatch-from-config.ts:125](https://github.com/openclaw/openclaw/blob/11804a484ded89374afa55020813cfa265aa143a/src/auto-reply/reply/dispatch-from-config.ts#L125)
  - command: `nl -ba src/auto-reply/reply/dispatch-from-config.ts | sed -n '118,155p'`
  - sha: [11804a484ded](https://github.com/openclaw/openclaw/commit/11804a484ded89374afa55020813cfa265aa143a)
- **The existing Codex review concern is grounded in current attachment normalization:** `normalizeAttachments()` assigns URL attachment MIME from `MediaTypes[index]`, and `finalizeInboundContext()` pads `MediaTypes` to match the media count. That means a fix that appends or overwrites audio without preserving remote URL MIME types can misclassify a remote image URL as audio in mixed image+voice messages, matching the PR review comment’s warning.
  - file: [src/media-understanding/attachments.normalize.ts:27](https://github.com/openclaw/openclaw/blob/11804a484ded89374afa55020813cfa265aa143a/src/media-understanding/attachments.normalize.ts#L27)
  - command: `nl -ba src/media-understanding/attachments.normalize.ts | sed -n '1,120p'`
  - sha: [11804a484ded](https://github.com/openclaw/openclaw/commit/11804a484ded89374afa55020813cfa265aa143a)
- **QQ Bot still lacks the gateway regression test the PR itself says is missing:** The extension currently has tests for plugin version, command auth, config, and manifest schema only; there is no QQ Bot gateway/inbound-context seam test covering voice-only or mixed voice+image media typing.
  - command: `find extensions/qqbot -maxdepth 3 \( -name '*test.ts' -o -name '*spec.ts' \) | sort`
