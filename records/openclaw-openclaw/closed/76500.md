## Summary

Close as implemented on current main: commit 928c70fb6beef9d6ddb7aadb9c1e131a987bdece keeps `pdfjs-dist` behind the actual document extraction path instead of paying it during PDF tool staging. No release tag contains that commit yet, so this is a main-only fix after v2026.5.2.

## What This Changes

Avoid per-run PDF tool staging overhead from eager `pdfjs-dist` loading when no PDF is being processed.

## Best Possible Solution

Keep the current lazy document-extractor artifact design; if PDF staging latency remains on a newer main build, open a fresh profile with current-main timings around model resolution rather than adding a disable knob first.

## Reproduction Assessment

No high-confidence current-main failure remains: source inspection shows `pdfjs-dist` is no longer imported during PDF tool staging. The original v2026.5.2 timing data was concrete, but this read-only pass did not rerun a live gateway profile.

## Solution Assessment

Yes. For the reported eager `pdfjs-dist` root cause, the current narrow artifact plus execute-time extraction path is the narrow maintainable fix; new `tools.deny` or PDF-enabled configuration is not needed to solve this bug.
