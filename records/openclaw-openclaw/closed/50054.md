## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve queued ACP turns when the lock is held:** `src/auto-reply/reply/dispatch-acp.ts:240-247`
  - body: `AcpSessionManager.runTurn()` already serializes same-session turns and still executes both requests, but this branch returns `acp_execution_locked` before that queue. Because the default no-Redis path uses `LocalSessionLockManager`, a second in-process ACP prompt can now be marked skipped and lost instead of waiting behind the first turn.
  - confidence: 0.92
- **[P2] Lease the shared ACP turn path:** `src/auto-reply/reply/dispatch-acp.ts:233-239`
  - body: The new lock is acquired only inside `tryDispatchAcpReply`, while `/acp steer` still calls `acpManager.runTurn()` directly. In Redis mode, a steer command can overlap a normal prompt for the same ACP session, leaving the cross-worker session race this feature is meant to prevent.
  - confidence: 0.88
- **[P2] Stop the turn when Redis renewal is lost:** `src/auto-reply/reply/dispatch-acp.ts:265-284`
  - body: Renewal failure is logged but the original turn keeps running. If Redis loses the key or a renewal fails past the TTL, another worker can acquire the session while the first worker is still mutating ACP state, so the distributed lock no longer protects long-running turns.
  - confidence: 0.86
- **[P2] Close Redis sockets on setup failure:** `src/auto-reply/reply/session-lock-manager.ts:279-287`
  - body: `connectRedis()` sends `AUTH` and `SELECT` before returning the connection to the caller that closes it. If either setup command rejects, the just-opened socket is left alive, so a misconfigured Redis deployment can leak a TCP connection on every skipped ACP turn.
  - confidence: 0.84
