## Summary

Current `main` already prevents this `[object Object]` prompt-serialization path: direct `agent`/`chat.send` RPC ingress requires string `message` fields, and the gateway prompt builder now converts content-array message bodies into extracted text instead of interpolating raw objects.

## Evidence

- **Gateway prompt builder now de-objects message bodies:** `safeBody()` explicitly handles content-array bodies so they do not serialize as `[object Object]`, using `extractTextFromChatContent()` before formatting the prompt; this is the path used by both OpenAI chat and OpenResponses gateway adapters.
  - file: [src/gateway/agent-prompt.ts:9](https://github.com/openclaw/openclaw/blob/564f820efa2bb81ae49b8f55a93e41492ad31917/src/gateway/agent-prompt.ts#L9)
  - command: `nl -ba src/gateway/agent-prompt.ts | sed -n '1,220p'`
  - sha: [564f820efa2b](https://github.com/openclaw/openclaw/commit/564f820efa2bb81ae49b8f55a93e41492ad31917)
- **Current gateway adapters route conversation entries through the fixed prompt builder:** Both adapters build `conversationEntries` with `body: content` / `body: messageContent` and then call `buildAgentMessageFromConversationEntries()`, so array/object-form user content is normalized before agent prompt construction.
  - file: [src/gateway/openai-http.ts:428](https://github.com/openclaw/openclaw/blob/c1383680405879a3c3f43dceb20a694f54d3c13c/src/gateway/openai-http.ts#L428)
