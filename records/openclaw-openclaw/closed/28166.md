## Summary

Keep open. Current `main` does not contain the PR’s two core behaviors: diagnostic trace-context propagation is absent, and log transport registration still only attaches to the currently cached logger instead of a cross-bundle/global registry.

## Evidence

- **No trace-context types on diagnostic events:** `DiagnosticBaseEvent` only has `ts` and `seq`, and the event variants shown for webhook/message flow do not carry `traceId`, `parentSpanId`, or a trace context object.
  - file: [src/infra/diagnostic-events.ts:5](https://github.com/openclaw/openclaw/blob/d3f6783b16ecc10c67b101bf019ca4294453206a/src/infra/diagnostic-events.ts#L5)
  - command: `nl -ba src/infra/diagnostic-events.ts | sed -n '1,90p'`
  - sha: [d3f6783b16ec](https://github.com/openclaw/openclaw/commit/d3f6783b16ecc10c67b101bf019ca4294453206a)
- **Current diagnostic emitters do not populate trace metadata:** `logWebhookReceived`, `logMessageQueued`, and `logMessageProcessed` emit events without any trace fields, so there is no production path on `main` to correlate webhook/message/model spans the way the PR describes.
  - file: [src/logging/diagnostic.ts:102](https://github.com/openclaw/openclaw/blob/d3f6783b16ecc10c67b101bf019ca4294453206a/src/logging/diagnostic.ts#L102)

  - sha: [d3f6783b16ec](https://github.com/openclaw/openclaw/commit/d3f6783b16ecc10c67b101bf019ca4294453206a)
- **OTEL service still starts spans without parent context:** `spanWithDuration` calls `tracer.startSpan(...)` with attributes/startTime only. There is no `context.active()`, `trace.setSpanContext(...)`, or parent trace/span handling before recording `message.processed` spans.
  - file: [extensions/diagnostics-otel/src/service.ts:361](https://github.com/openclaw/openclaw/blob/d3f6783b16ecc10c67b101bf019ca4294453206a/extensions/diagnostics-otel/src/service.ts#L361)
