## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Reopen flows before spawning retry children:** `src/tasks/task-executor.ts:569`
  - body: `resolveRetryableManagedChildTaskFlow` allows retries from `failed` and `lost`, but this branch passes `parentFlowId` into the child spawn while the parent flow is still terminal. The registry rejects linking tasks to terminal flows, so failed/lost retries can fail or start detached work instead of moving the managed flow back to waiting.
  - confidence: 0.91
- **[P1] Honor cancellation before completion finalization:** `src/tasks/task-registry.ts:948`
  - body: A cancelled managed flow can still be finalized by the child completion path first, because completion sync does not skip `cancelRequestedAt` flows and runs before the cancellation sync. Once that writes a terminal child outcome, the cancellation path returns early, so user cancellation is never reflected.
  - confidence: 0.9
- **[P2] Require wait identifiers to match terminal tasks:** `src/tasks/task-registry.ts:815-825`
  - body: This matcher only rejects mismatches when both the wait state and task provide an identifier. If a flow waits on a specific run/session and another linked terminal task lacks that field, the flow can close from the wrong child outcome; require any identifier present in `waitJson` to also be present and equal on the task.
  - confidence: 0.86
- **[P2] Discard stale task-flow load responses:** `ui/src/ui/controllers/task-flows.ts:23-33`
  - body: When a user switches sessions during `loadLatestTaskFlow`, the second load is skipped while `taskFlowLoading` is true, and the first response can still write `taskFlowDetail` into the new session. Capture the requested session key or use a request token so stale responses cannot populate actions for the wrong chat.
  - confidence: 0.88
- **[P2] Align guidance with retry eligibility:** `src/tasks/task-flow-guidance.ts:41-47`
  - body: `getTaskFlowGuidance` labels every managed `failed` or `lost` flow as retryable, but the retry resolver can mark the same flow non-retryable when launch metadata disables replay. That produces contradictory CLI/runtime/UI guidance, so guidance should derive from the same eligibility result used by the retry command.
  - confidence: 0.83
