## Summary

Keep open: this PR still targets a real Gateway WebSocket handshake race, and the linked source contribution at https://github.com/openclaw/openclaw/pull/49751 provides useful context, but the current head has a blocking resettable connect-auth watchdog defect in the unauthenticated preauth path.

## What This Changes

This PR moves Gateway WebSocket preauth timer clearing to a valid `connect` frame, adds a bounded connect-auth watchdog, and updates Gateway docs, changelog, and focused tests.

## Best Possible Solution

Repair the branch with a one-shot connect-auth deadline or explicit connecting state, add duplicate-connect regression coverage, and preserve the original contributor credit when landing.

## Reproduction Assessment

Yes. PR-head source shows a valid connect whose auth stalls can be followed by another valid preauth connect before timeout; because no client is registered yet, the second path clears and re-arms the watchdog.

## Solution Assessment

No. The two-phase timer direction is the right fix boundary, but this implementation is not the best solution until duplicate preauth connects cannot extend the auth deadline.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Make the connect-auth watchdog one-shot:** `src/gateway/server/ws-connection.ts:408`

Because the socket has no registered client until after async auth completes, a second valid preauth connect can re-enter this path and extend the deadline indefinitely; preserve the first deadline or reject duplicate connects before rearming.
