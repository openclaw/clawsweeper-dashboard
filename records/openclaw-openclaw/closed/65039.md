## Summary

Keep open. The shipped timeout/reconnect mitigation fixed the broader slow-Ready cluster, but current main and v2026.5.18 still expose `daveEncryption=false`, pass it through to `@discordjs/voice`, and return the same `/vc join` failure if the dependency never reaches Ready; the remaining question is the intended non-DAVE join contract.

## What This Changes

The issue asks OpenClaw to fix Discord `/vc join` aborting before `Ready` when `channels.discord.voice.daveEncryption=false`, after the voice WebSocket Hello and before UDP handshaking.

## Best Possible Solution

Make the Discord voice join contract explicit: either support DAVE-disabled joins through Ready with live coverage, or reject `daveEncryption=false` before join with a clear diagnostic and docs.

## Reproduction Assessment

Yes, with high source-level confidence. The issue includes live repro steps and a detailed trace, and current main still has the same DAVE-disabled pass-through plus pre-Ready `entersState()` rejection path that returns the reported `/vc join` failure.

## Solution Assessment

Unclear. A narrow code change is possible, but the maintainable fix depends on whether maintainers want non-DAVE Discord voice joins to be supported or to fail early as unsupported because Discord may require DAVE.
