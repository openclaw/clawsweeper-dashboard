## Summary

Issue #13329 proposes an optional external memory integration. Current main explicitly routes optional memory backends through the plugin system and ClawHub/community plugin path, with existing public memory-plugin and memory-event SDK seams that fit a `mem` integration better than adding more core surface.

## Close Comment

Closing from a Codex review. This looks like a useful idea, but it fits OpenClaw's external plugin path rather than core. Current main is explicit that optional capability should usually ship as plugins and that memory is an exclusive plugin slot (`VISION.md:54-64`). The plugin contract already supports dedicated memory plugins (`docs/plugins/manifest.md:141`) and the community distribution path is ClawHub/npm, with ClawHub called out as the canonical discovery surface (`docs/plugins/community.md:9-22`). Main also already exposes public memory event-journal helpers (`src/memory-host-sdk/events.ts:5-60`, surfaced through the SDK in `docs/plugins/sdk-subpaths.md:235-250`), which is the kind of seam a deterministic `mem` sidecar can build on. If you want to pursue this, the project direction here is an external memory plugin or companion plugin published through ClawHub, then a follow-up issue only if a specific SDK seam is missing.
