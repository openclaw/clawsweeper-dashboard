## Summary

Current main already implements the missing bearer injection for the observed OpenAI Responses path. The embedded runner now wraps boundary-aware OpenAI/Codex Responses transports and passes the resolved run credential or authStorage bearer into `options.apiKey`, which matches the reported `api.openai.com/v1/responses` 401 Missing bearer failures. The proof is on current main only; the latest tag checked, `v2026.4.27`, does not contain this wrapper.

## What This Changes

The issue requests that OpenAI and Codex Responses requests attach the bearer token resolved from the selected OpenAI/OpenAI Codex auth profile so fallback chains do not exhaust with 401 Missing bearer errors.

## Best Possible Solution

Keep the current main bearer-injection fix and ship it in the next release; any fresh reproduction after that build should be filed as a narrower report with the exact runtime route, auth profile id/provider, and whether it is PI Responses or native Codex app-server.
