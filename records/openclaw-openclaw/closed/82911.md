## Summary

Keep open: current main still lets stale ambient `webchat` context flow through the message tool, and the outbound runner can turn that into an internal UI source-reply sink before normal channel fallback can use the non-webchat session context.

## What This Changes

The issue asks for the `message` tool to prefer a routable non-webchat session-key delivery context over stale `currentChannelProvider=webchat` ambient state.

## Best Possible Solution

Add a narrow source-level resolver in `createMessageTool` that uses the session-key-derived non-webchat channel and target only for the contaminated stale-webchat shape, with regression coverage for the message tool and outbound runner.

## Reproduction Assessment

Yes, at source level: current main still passes stale `currentChannelProvider: webchat` into `toolContext`, and the outbound runner maps that shape to `sourceReplySink: internal-ui` before channel resolution. I did not run a live Signal reproduction.

## Solution Assessment

Yes, the guarded session-key preference proposed in the issue is the narrowest maintainable direction, but it should be implemented in source with tests rather than as a generated `dist` patch.
