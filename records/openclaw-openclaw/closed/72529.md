## Summary

Current main already supports the requested cron-to-persistent-session routing through the generic `session:<id>` cron target, which covers subagent session keys without needing a subagent-specific target spelling. The report’s observed `--session-key` behavior is expected because `--session-key` is not the routing knob for existing sessions.

## What This Changes

The issue requests cron agent-turn jobs be able to route directly into existing persistent subagent sessions rather than only main or isolated sessions.

## Best Possible Solution

Keep the generic persistent-session cron target as the supported API: users should schedule with `--session "session:<full-session-key>"` and, for non-default agents, pin `--agent <agentId>` rather than adding a subagent-only alias.

## Reproduction Assessment

Yes. The current-main verification path is to create a cron agent-turn job with `sessionTarget: "session:<key>"` and inspect runner params; existing gateway and runner tests cover custom session-key routing, and subagent keys have the same supported colon-rich shape.

## Solution Assessment

Yes. The existing generic `session:<id>` target is the narrowest maintainable solution because cron stays session-key agnostic and works for subagent, ACP, channel, and named persistent sessions without hard-coding a subagent-only spelling.
