## Summary

Current main already scopes generic inbound dedupe by provider, route target, account, thread, and message id, so the collision this PR is trying to prevent is already handled without changing Telegram's provider message id. The PR should not merge as-is because replacing `MessageSid` with a composite Telegram route key would break current-message reply threading that expects Telegram's numeric `message_id`.

## What This Changes

The PR adds a Telegram helper that composes `MessageSid` from account, chat, thread, and message id, applies it to normal and native-command inbound contexts, and adds helper unit tests.

## Best Possible Solution

Keep the current route-scoped dedupe implementation on main. If a future Telegram path needs an explicit composite provider key, add it as a separate full/private dedupe field while preserving numeric `MessageSid` for prompt metadata and Telegram reply APIs.

## Reproduction Assessment

Yes. A static reproduction is available from current main: `buildInboundDedupeKey` scopes by route metadata already, while the PR's composite `MessageSid` would fail Telegram's numeric `replyToId` parsing path.

## Solution Assessment

No. The PR is not the best fix because the generic dedupe layer is already route-scoped, and overloading `MessageSid` breaks a separate provider-id contract; the safer solution is to leave `MessageSid` numeric and keep route scoping in the dedupe key.

Full review comments:

- **[P1] Preserve the numeric Telegram message id:** `extensions/telegram/src/bot-message-context.session.ts:355-356`
  - body: `MessageSid` is also exposed as conversation `message_id` and used as the current `replyToId`. Telegram delivery parses reply ids as numeric Telegram `message_id` values, so replacing `MessageSid` with `telegram:<account>:<chat>:...` makes current-message replies and native quote lookup stop threading correctly. Keep the numeric short id and put any composite key in a separate field.
  - confidence: 0.91
- **[P2] Use the DM topic id in the scoped key:** `extensions/telegram/src/bot-message-context.session.ts:359`
  - body: `resolvedThreadId` is only set for forum groups; DM topics are carried as `dmThreadId` or `threadSpec.id`. The new call therefore still produces the empty-thread form for DM topics even though the PR says DM topics are part of the route scope, and the native-command path makes the same omission. Use `threadSpec.id` or `resolvedThreadId ?? dmThreadId` consistently if this key remains.
  - confidence: 0.87
