## Summary

Keep open: current main and the latest release still validate browser upload paths only against the temp uploads root, while WebChat staged attachments can reach the agent as managed inbound media. The linked open fix PR owns the implementation path, so this issue should stay open until that PR, or an equivalent maintainer-approved boundary fix, lands.

## What This Changes

The issue asks for `browser.upload` to accept OpenClaw-managed WebChat inbound media paths instead of only the temp uploads directory.

## Best Possible Solution

Land one canonical, owner-approved browser upload resolver that accepts only temp uploads plus verified managed inbound media refs or first-level inbound media files, then close this issue after https://github.com/openclaw/openclaw/pull/83660 or an equivalent fix merges.

## Reproduction Assessment

Yes, source-level: current main validates `browser.upload` paths only against `DEFAULT_UPLOAD_DIR`, while WebChat attachment staging can deliver `media/inbound/*` paths with `MediaWorkspaceDir`. I did not run the full live WebChat/browser flow in this read-only review.

## Solution Assessment

Yes, a shared browser-owned resolver that covers the tool, hook route, and Playwright use-time validation is the maintainable repair shape. The linked PR appears aligned, but it needs maintainer approval of the upload security boundary before the issue can close.
