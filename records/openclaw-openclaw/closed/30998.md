## Summary

Current main already treats tool-output/context optimization as plugin territory, not OpenClaw core: VISION directs optional capability to plugins, OpenClaw exposes a pluggable context-engine slot plus tool-result transform hooks, and it already ships the optional bundled `tokenjuice` plugin for compressing noisy tool results.

## Close Comment

Closing as `clawhub` after Codex review. Current `main` already points this problem to the plugin ecosystem rather than new core middleware: `VISION.md:54` says optional capability should usually ship as plugins, `docs/concepts/context-engine.md:10` documents plugin-owned context assembly/compaction, and `docs/concepts/agent-loop.md:98` exposes `tool_result_persist` specifically for transforming tool results before they are persisted. There is also already a shipped example of this pattern in `docs/tools/tokenjuice.md:10` and `extensions/tokenjuice/openclaw.plugin.json:1`, where the optional bundled `tokenjuice` plugin compacts noisy tool results without adding more core runtime policy. Given that scope and the existing seams, this request is better pursued as a ClawHub/plugin-context-engine implementation instead of adding native OpenClaw core middleware.
