## Summary

Current `main` already carries the cron timezone fix path and explicit Asia/Shanghai regression coverage, and the changelog records that fix as shipped in release `2026.2.27`. The reported `nextRunAtMs` value (`2026-03-15T03:00:00Z`, i.e. 11:00 Asia/Shanghai) matches the pre-fix failure mode, but that is not how current `main` computes cron schedules.

## Evidence

- **Cron scheduler uses `schedule.tz` as the Croner timezone:** `computeNextRunAtMs(...)` resolves `schedule.tz` and passes it into `new Cron(expr, { timezone, catch: false })`, so cron evaluation is timezone-aware rather than defaulting to gateway local/UTC time.
  - file: [src/cron/schedule.ts:9](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/src/cron/schedule.ts#L9)
  - sha: [68315792677f](https://github.com/openclaw/openclaw/commit/68315792677f9628d1370d6869366442f1d3ecad)
- **Asia/Shanghai regression test exists on current main:** The scheduler test suite has explicit coverage for daily `Asia/Shanghai` cron schedules and asserts the next run advances to the next valid local occurrence instead of rolling back or producing an invalid past result.
  - file: [src/cron/schedule.test.ts:26](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/src/cron/schedule.test.ts#L26)
  - sha: [68315792677f](https://github.com/openclaw/openclaw/commit/68315792677f9628d1370d6869366442f1d3ecad)
- **Public cron contract expects `tz`:** The gateway protocol schema for cron schedules exposes `tz` on `schedule.kind = "cron"`, matching the scheduler implementation path that consumes timezone-aware schedules.
  - file: [src/gateway/protocol/schema/cron.ts:119](https://github.com/openclaw/openclaw/blob/68315792677f9628d1370d6869366442f1d3ecad/src/gateway/protocol/schema/cron.ts#L119)
  - sha: [68315792677f](https://github.com/openclaw/openclaw/commit/68315792677f9628d1370d6869366442f1d3ecad)

## Risks / Open Questions

- The issue body includes both `schedule.timezone` and `schedule.tz`. Core OpenClaw only contracts `tz`; if some external caller persists only `timezone` before entering core, that upstream caller could still mis-schedule jobs even though OpenClaw main is fixed.
