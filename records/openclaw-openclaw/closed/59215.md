## Summary

The reported v2026.4.1 loopback exec pairing failure matches the already-fixed local role-upgrade regression tracked in #59045 and fixed by #59092, shipped in v2026.4.2.

## Evidence

- **Original report matches fixed regression:** Issue #59215 reports exec approval registration failing on ws://127.0.0.1:18789 with close 1008 "pairing required" on OpenClaw 2026.4.1. Related closed issue #59045 reports the same local loopback exec failure after v2026.3.31 and is linked as fixed by PR #59092.
  - command: `GitHub issue context plus https://github.com/openclaw/openclaw/issues/59045`
- **Fix PR explains root cause and scope:** PR #59092 states that local clients connecting with a new role were rejected with 1008 "pairing required" because empty device token maps lost legacy roles and local role-upgrade pairing was not silently allowed. The PR says the patch lets local loopback exec/node clients acquire roles silently while keeping remote clients unchanged.
  - file: [src/infra/device-pairing.ts:2609](https://github.com/openclaw/openclaw/blob/51edd30/src/infra/device-pairing.ts#L2609)

  - sha: [51edd30](https://github.com/openclaw/openclaw/commit/51edd30)
- **Current main allows local role-upgrade pairing:** Current main's shouldAllowSilentLocalPairing accepts reason "role-upgrade" for non-remote local pairing paths, which addresses the loopback role-upgrade rejection described in the issue.
  - file: [src/gateway/server/ws-connection/handshake-auth-helpers.ts:1185](https://github.com/openclaw/openclaw/blob/afd6b5d6fce1dac98539d8b45bd0f9db9ff25386/src/gateway/server/ws-connection/handshake-auth-helpers.ts#L1185)
