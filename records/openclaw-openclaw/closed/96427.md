## Summary

Keep open: current main still leaves the old run-id reverse lookup behind when an ACP session is rebound, and this PR is a narrow owner-boundary fix with focused regression coverage and sufficient terminal proof.

## What This Changes

The PR deletes a previous active-run reverse lookup when an ACP in-memory session is rebound to a new run id and adds a regression test.

## Best Possible Solution

Land the session-store cleanup after ordinary maintainer review, keeping the invariant inside createInMemorySessionStore rather than filtering stale lookups at callers.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: create one ACP session, call setActiveRun with an old run id and then a new run id, and the old run id can still resolve through runIdToSessionId because it is not deleted.

## Solution Assessment

Yes. The in-memory session store owns both activeRunId and the reverse lookup, so deleting the previous mapping during setActiveRun is narrower and cleaner than making callers distrust getSessionByRunId.
