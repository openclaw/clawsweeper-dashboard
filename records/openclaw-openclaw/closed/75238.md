## Summary

Close as superseded. The reported post-2026.4.27 Telegram topic symptom matches the already-closed shared group/channel visible-replies upgrade behavior: ordinary group/channel final replies are private by default unless the agent uses the message tool or config restores automatic visible replies, while current Telegram topic delivery still preserves `message_thread_id` when a visible send is actually attempted.

## What This Changes

The issue asks for Telegram forum-topic assistant replies after upgrading to 2026.4.27 to be delivered back to the originating group topic with the correct `message_thread_id`.

## Best Possible Solution

Keep the #74876/#75099 resolution as canonical: preserve the safer private-by-default group/channel behavior, use `message(action=send)` for visible room output or set `messages.groupChat.visibleReplies` to `automatic` to restore legacy auto-posting, and open a new Telegram-specific bug only if topic replies still fail after that config is set.

## Reproduction Assessment

Yes. There is a high-confidence code-level reproduction for the reported missing `sendMessage`: a group/channel turn with empty visible-replies config resolves to `message_tool_only` and suppresses automatic final delivery. Current tests also show the supported restore path by setting `messages.groupChat.visibleReplies` to `automatic`.

## Solution Assessment

Yes. Closing as superseded is the best cleanup path because the remaining behavior is the shared 2026.4.27 group/channel visible-replies contract, not a separate Telegram topic addressing defect; the Telegram send path still carries `message_thread_id` for explicit or automatic visible sends.
