## Summary

Current main implements the Telegram polling stall recovery stack identified in the issue discussion: bounded HTTP/1-only Telegram dispatchers, explicit transport lifecycle/disposal, dirty transport rebuilds, getUpdates liveness/status publishing, and the later watchdog follow-up that keeps stall recovery active even when grammY reports the runner stopped while its task is still pending. The issue has no protected labels and no fresh current-main reproduction after those fixes landed.

## Best Possible Solution

Close this broad active-stall report as implemented on current main. Keep the shipped polling recovery stack in place; if a current-main user can still reproduce delayed or missing Telegram replies, track that as a fresh bug with `OPENCLAW_DIAG=telegram` logs spanning the stall, rebuild, and next attempted poll.
