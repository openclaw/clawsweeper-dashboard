## Summary

Close as implemented on current main: DeepSeek V4 now exposes a max-capable thinking profile through both the runtime provider and the cold provider-policy artifact, cron validates against that generic profile, and the DeepSeek wrapper sends `reasoning_effort: "max"` for `max`/`xhigh`.

## What This Changes

The issue asks for DeepSeek V4 isolated cron jobs with `thinking: "max"` to preserve `reasoning_effort: "max"` instead of downgrading to `high`.

## Best Possible Solution

Keep the plugin-owned policy artifact/cold-profile path and ship it in the next release; no core DeepSeek-specific regex fallback is needed.

## Reproduction Assessment

No high-confidence failing reproduction remains on current main. Source inspection traces isolated cron thinking validation to the DeepSeek policy artifact that advertises `max`, so the reported downgrade path is no longer present.

## Solution Assessment

Yes. The current fix is the narrow maintainable boundary because DeepSeek-specific thinking levels stay in the DeepSeek plugin/public policy artifact while core cron uses generic provider thinking resolution.
