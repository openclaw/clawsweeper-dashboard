- `impact:other`: The issue concretely blocks managed ClawHub skill updates on affected Windows setups, which is maintainer-visible but outside the more specific impact labels.

## Summary

Keep open: the reported Windows skills-update failure is still plausible on current main and latest release because OpenClaw routes ClawHub skill updates through the pinned fs-safe move helper, and that helper only falls back from rename on EXDEV, not EPERM.

## What This Changes

The issue asks OpenClaw to make Windows EPERM rename failures during `openclaw skills update --all` use the existing copy fallback instead of aborting the update.

## Best Possible Solution

Fix `movePathWithCopyFallback` in `@openclaw/fs-safe`, release and bump the exact OpenClaw dependency, then add OpenClaw regression coverage that simulates Windows EPERM during skill update replacement.

## Reproduction Assessment

Yes for a source-level reproduction: simulate `fs.rename` returning EPERM while the skill update path moves an existing target directory and the pinned helper will rethrow before copy fallback. A live Windows run was not performed here.

## Solution Assessment

Unclear as an in-repo-only patch: the narrow maintainable fix belongs in `@openclaw/fs-safe`'s move fallback, followed by an OpenClaw dependency bump and focused EPERM regression coverage.
