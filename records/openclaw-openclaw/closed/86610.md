## Summary

This should close in OpenClaw because the reported repeated `history.list` and `messages.get` retries are implemented inside the external `gogcli` push handler; current OpenClaw only configures, starts, renews, and supervises that process.

## What This Changes

The issue asks for Gmail Pub/Sub push handling to honor Gmail 429 retry-after per account and avoid repeated Gmail history/message fetches while rate-limited.

## Best Possible Solution

Track and fix the per-account Gmail 429 retry-after circuit in `openclaw/gogcli`; keep OpenClaw as a thin launcher unless gogcli needs a new CLI flag or output contract for OpenClaw to consume.

## Reproduction Assessment

Yes, partially: the issue includes runtime 429 logs, and source inspection shows the external `gogcli` push handler makes Gmail history/message fetches without a persisted retry-after circuit. I did not live-trigger Gmail 429s against current main.

## Solution Assessment

No for this repository: adding a per-account retry-after circuit to OpenClaw would be the wrong boundary because OpenClaw does not handle Pub/Sub pushes or Gmail API fetches. The maintainable fix belongs in `gogcli`'s watch server state and handler code.
