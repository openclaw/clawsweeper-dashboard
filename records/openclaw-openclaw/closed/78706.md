## Summary

Keep open for maintainer judgment: the reported behavior is real on current main, but it is also explicitly documented and covered by tests, so changing it would alter existing heartbeat selection semantics rather than fix a clearly broken contract.

## What This Changes

The issue requests that agents without their own heartbeat block inherit `agents.defaults.heartbeat` even when another agent has explicit heartbeat config.

## Best Possible Solution

Have the heartbeat owner decide whether explicit per-agent heartbeat blocks are an allowlist or only overrides; if the model changes, update both heartbeat enablement paths, tests, docs, and a changelog/migration note together.

## Reproduction Assessment

Yes. Source inspection and the existing `isHeartbeatEnabledForAgent` test show that a config with global defaults plus one explicit agent heartbeat disables other listed agents for heartbeat purposes on current main.

## Solution Assessment

No, not as an automatic bug fix. The suggested fallback semantics may be reasonable, but they conflict with current docs and tests, so the safest solution starts with a maintainer decision about the intended config contract.
