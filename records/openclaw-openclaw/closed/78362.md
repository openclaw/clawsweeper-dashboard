## Summary

Keep open: current main still serves a strict Control UI CSP, still constructs browser-side Zod object schemas on the startup/settings path, and has no Control UI jitless initialization. The linked unsafe-eval PR was closed as the wrong security tradeoff, while the config-fetch 401 path is only partly mitigated by current auth candidate retry logic.

## What This Changes

The issue requests a fix for Control UI CSP eval violations from Zod's Function constructor path and an initial authenticated bootstrap config fetch returning 401.

## Best Possible Solution

Initialize Control UI Zod in jitless mode before any browser schemas are constructed or parsed, keep the strict CSP without `unsafe-eval`, and only adjust the config fetch flow if current runtime proof shows the 401 still affects valid auth.

## Reproduction Assessment

Yes, at source level: current main serves `script-src 'self'`, uses Zod object schemas in the Control UI startup/settings path, and has no jitless initialization. I did not perform a live browser reproduction in this read-only pass.

## Solution Assessment

Yes for the jitless direction, not for relaxing CSP: Zod 4.4.3 provides `z.config({ jitless: true })`, which is narrower than allowing eval-equivalent code execution across the authenticated Control UI.
