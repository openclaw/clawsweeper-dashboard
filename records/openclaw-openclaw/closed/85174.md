## Summary

Close as duplicate: current main still shows the reported `api.registerHook(...)` path registering internal event keys while runtime lifecycle dispatch uses `api.on(...)` typed hooks, and the same remaining SDK compatibility decision is already tracked by the open canonical issue at https://github.com/openclaw/openclaw/issues/77285.

## What This Changes

The issue asks OpenClaw to make user-plugin hook handlers registered through `api.registerHook` for typed lifecycle names dispatch at runtime or stop reporting them as ready.

## Best Possible Solution

Use https://github.com/openclaw/openclaw/issues/77285 as the canonical tracker, and resolve the SDK contract there by choosing either explicit diagnostics for typed names passed to `registerHook` or a tested compatibility bridge to typed `api.on` hooks.

## Reproduction Assessment

Yes. Source inspection on current main shows `api.registerHook("gateway_start")` and `api.registerHook("before_agent_run")` are stored as internal hook events, while runtime dispatch checks `registry.typedHooks` populated by `api.on(...)`; I did not run the live macOS/Telegram flow.

## Solution Assessment

Yes for cleanup: closing this as a duplicate keeps one canonical SDK compatibility discussion open. The code solution itself still needs maintainer choice between warning/rejecting typed hook names in `registerHook` or adding a compatibility bridge.
