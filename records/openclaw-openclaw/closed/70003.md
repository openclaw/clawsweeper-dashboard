## Summary

Current main covers the user-facing request through interrupt/steer/background paths rather than true multi-active same-session transcript writers: queue modes include `steer` and `interrupt`, `sessions.steer` aborts and restarts an active session, `chat.abort` and stop phrases are out-of-band, foreground exec receives abort signals, and `sessions_spawn` returns an accepted run id without waiting for child completion.

## Best Possible Solution

Keep the shipped per-session serialization model and use the existing intervention paths: `/stop` or `chat.abort` to cancel, `sessions.steer` or `/redirect` to abort and restart, queue mode `steer`/`interrupt` for inbound corrections, and nonblocking `sessions_spawn` or backgrounded `exec` for long work. Open a narrower design issue only if maintainers want true concurrent same-session transcript writers beyond the current safety model.
