## Summary

Close as `clawhub`: current `main` already treats `search` as a shared message action owned by channel plugins, and the repo scope explicitly pushes optional integrations like email search into plugins/ClawHub rather than new core behavior.

## Evidence

- **Vision scopes optional capability to plugins/ClawHub:** The project vision says core should stay lean, optional capability should usually ship as plugins, and plugin promotion belongs in ClawHub.
  - file: [VISION.md:54](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/VISION.md#L54)
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Shared message tool delegates to channel plugins:** The plugin architecture docs say core owns the shared `message` tool host, while channel plugins own action discovery, schema, and final execution.
  - file: [docs/plugins/architecture.md:181](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/docs/plugins/architecture.md#L181)
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Message-tool schema/actions are plugin-owned:** `ChannelMessageToolDiscovery` is explicitly plugin-owned for actions, capabilities, and schema fragments, which matches adding channel-specific email search via a plugin rather than core branching.
  - file: [src/channels/plugins/types.core.ts:54](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/channels/plugins/types.core.ts#L54)
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Core already has a generic `search` message action:** The shared channel action name list already includes `search`, so the missing piece is an email-channel implementation, not a new core action kind.
  - file: [src/channels/plugins/message-action-names.ts:27](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/channels/plugins/message-action-names.ts#L27)
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Bundled channels already implement message search through the plugin seam:** The Discord channel plugin advertises `search` from `describeMessageTool(...)`, showing the intended pattern for channel-owned search behavior behind the shared message tool.
  - file: [extensions/discord/src/channel-actions.ts:106](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/extensions/discord/src/channel-actions.ts#L106)
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Email is explicitly called out as an external channel-plugin use case:** The channel adapter types document external channel plugins such as email/SMS/custom integrations and show an `emailGatewayAdapter` example using plugin runtime helpers.
  - file: [src/channels/plugins/types.adapters.ts:267](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/channels/plugins/types.adapters.ts#L267)
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Docs already point email search workflows to external tooling:** The Lobster docs show `gog.gmail.search` and `email.triage` examples, which is existing evidence that email search automation is expected to compose through plugins/workflows instead of expanding core message-tool logic.
  - file: [docs/tools/lobster.md:73](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/docs/tools/lobster.md#L73)
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)

## Risks / Open Questions

- Closing this in core does not provide a maintained Gmail/Hotmail/IMAP plugin today; there may still be real user demand for one.
- If an email plugin hits a genuine missing SDK seam later, that narrower SDK-gap issue should be reopened or filed separately.
