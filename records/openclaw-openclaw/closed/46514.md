## Summary

Current main intentionally keeps Discord and multi-agent session history isolated per agent/session key, and the remaining request is an opt-in shared transcript/orchestration policy for trusted multi-bot rooms. That fits the ClawHub/plugin boundary rather than OpenClaw core: plugin hooks and session tools provide the right extension surface, while related shared-context requests have already been closed on that same boundary.

## Best Possible Solution

Keep OpenClaw core's per-agent session isolation as the default. Build this as an opt-in ClawHub/community plugin that records or relays shared transcript entries for trusted Discord multi-bot rooms using message/session hooks and existing session tooling. If that plugin exposes a concrete missing hook or runtime helper, open a narrow plugin API issue for that seam rather than changing core Discord routing semantics.

## Risks / Open Questions

- Cross-agent shared Discord history is product- and privacy-sensitive: a core implementation could bypass the documented per-agent workspace/session/auth isolation and expose channel or account context across agents without a clear trust policy.
- A plugin should define explicit opt-in policy for which bot accounts, agents, guilds, channels, and transcript fields may be shared before any shared-history behavior is enabled.
