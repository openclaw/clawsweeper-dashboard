## Summary

Keep open. Current `main` still routes `sessionTarget:"main"` cron wakeups through heartbeat code paths that treat `heartbeat.every: "0m"` as disabled, so the reported skip/disabled behavior remains plausible on `c1383680405879a3c3f43dceb20a694f54d3c13c`.

## Evidence

- **Cron wake-now path still inherits the agent heartbeat config:** `buildGatewayCronService` merges `agents.defaults.heartbeat` and per-agent heartbeat into `baseHeartbeat`, then merges the cron override on top before calling `runHeartbeatOnce`. That means `every: "0m"` is still present unless explicitly overridden.
  - file: [src/gateway/server-cron.ts:303](https://github.com/openclaw/openclaw/blob/c1383680405879a3c3f43dceb20a694f54d3c13c/src/gateway/server-cron.ts#L303)

  - sha: [c13836804058](https://github.com/openclaw/openclaw/commit/c1383680405879a3c3f43dceb20a694f54d3c13c)
- **`runHeartbeatOnce` still skips when interval resolves to zero:** `runHeartbeatOnce` returns `{ status: "skipped", reason: "disabled" }` when `resolveHeartbeatIntervalMs(...)` is falsy. A merged heartbeat config carrying `every: "0m"` hits this branch.
  - file: [src/infra/heartbeat-runner.ts:743](https://github.com/openclaw/openclaw/blob/c1383680405879a3c3f43dceb20a694f54d3c13c/src/infra/heartbeat-runner.ts#L743)

That keeps the `next-heartbeat` / queued wake path coupled to heartbeat enablement.
