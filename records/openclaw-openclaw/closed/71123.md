## Summary

Keep open. Current `main` still short-circuits Tailscale hostname detection when `gateway.tailscale.mode` is `"off"`, so `openclaw status` cannot surface connected-client DNS/runtime details in the overview even though the formatter and tests already support showing `off · <tailnet host>`. The issue is recent, actionable, and already has an open PR reference in discussion.

## Evidence

- **Current main still gates hostname lookup on config mode:** `createStatusScanCoreBootstrap` sets `tailscaleDnsPromise` to `null` whenever `cfg.gateway?.tailscale?.mode ?? "off"` is `"off"`, instead of attempting `getTailnetHostname(...)`. That means an unconfigured but connected local Tailscale client will never contribute DNS/runtime info to status output on current `main`.
  - file: [src/commands/status.scan.bootstrap-shared.ts:93](https://github.com/openclaw/openclaw/blob/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2/src/commands/status.scan.bootstrap-shared.ts#L93)

That makes the bootstrap short-circuit the blocking behavior, not the formatter.
