## Summary

Keep open. Current main still has the reported telephony TTS gap: the voice-call Twilio stream path sends raw spoken text into telephony synthesis, while the existing directive parser and override plumbing are only used by the message-channel TTS path.

## What This Changes

Add support for `[[tts:voiceId=...]]` and related TTS directives in the voice-call telephony TTS path so mid-call voice/model overrides are parsed, applied, and not spoken aloud.

## Best Possible Solution

Implement this as a small owner-path change: parse and strip TTS directives before voice-call telephony synthesis, carry the existing directive override data through an additive telephony TTS request contract, and lock the behavior with focused voice-call and speech-core tests for cleaned text plus ElevenLabs/OpenAI override propagation.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: with an active Twilio stream, pass text containing `[[tts:voiceId=...]]` into `playTts`; current main forwards that exact text to `synthesizeForTelephony` and the telephony runtime has no override parameter to apply the directive.

## Solution Assessment

Yes. Reusing `parseTtsDirectives` and extending the telephony TTS boundary additively is the narrowest maintainable fix; adding a second parser inside the voice-call plugin would risk drift from the supported message-channel directive behavior.
