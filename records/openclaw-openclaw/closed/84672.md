## Summary

Keep open: current main and v2026.5.20 still contain the heartbeat pending-final replay, target-none skip, duplicate skip, and heartbeat send-success cleanup gaps described by the report. The issue also has an open closing PR at https://github.com/openclaw/openclaw/pull/84679, and the broader pending-final heartbeat cleanup is still tracked by https://github.com/openclaw/openclaw/issues/83184 and https://github.com/openclaw/openclaw/pull/83187.

## What This Changes

The issue reports that scheduled heartbeats can stop making fresh model calls after pending final-delivery state or duplicate suppression leaves them skipped.

## Best Possible Solution

Review the linked heartbeat fixes together and land a narrow state cleanup that lets stuck heartbeats make fresh model calls again without weakening durable final-delivery recovery or intentional duplicate-notification suppression.

## Reproduction Assessment

Yes at source level: seeded pendingFinalDeliveryText on a heartbeat reaches the get-reply replay path, then target-none or duplicate handling can record a skipped heartbeat without a fresh model call. I did not run the live Docker/Telegram scenario in this read-only review.

## Solution Assessment

Yes in direction: the fix belongs in the existing heartbeat and final-delivery state paths, not in a new config option. The target-none PR and broader pending-final cleanup should be reconciled so both the internal-only and delivered-heartbeat stuck states are covered.
