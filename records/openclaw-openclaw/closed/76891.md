## Summary

Current main already avoids the reported first-touch stall: the TUI missing-session path remains read-only, `sessions.list` uses lightweight rows, and the release notes identify shipped sessions-list responsiveness work after the reported 2026.4.29 build.

## What This Changes

The issue asks for `sessions.list` on a brand-new TUI session key to return quickly when the session store has only a single existing row.

## Best Possible Solution

Keep the current lightweight `sessions.list` and bounded startup-memory behavior; users on 2026.4.29 should update to a current build rather than adding a new core option.

## Reproduction Assessment

No. I did not run a live TUI/gateway with a throwaway state, and current main's source plus focused regression tests show the brand-new-key path stays read-only and returns defaults instead of synchronously creating or indexing a memory-backed session.

## Solution Assessment

Yes. The current implementation is the narrow maintainable solution: keep `sessions.list` lightweight and bounded, with memory startup/session sync outside the synchronous list RPC path.
