## Summary

Close as a duplicate of #45469. Current main has some structured auth-failure reconnect pauses, but the TUI 1006 symptom still maps to the same unbounded GatewayClient reconnect policy already tracked by the open canonical issue.

## Best Possible Solution

Use #45469 as the canonical thread. The best fix is a deliberate GatewayClient reconnect policy with finite retry/terminal callback semantics, plus TUI handling that surfaces the final close code/detail clearly instead of continuing an opaque reconnect loop.

## Evidence

- **TUI uses GatewayClient without a TUI-specific retry cap:** GatewayChatClient constructs GatewayClient for Gateway-backed TUI mode and its onClose handler only resets readiness and forwards the close reason string to the TUI; it does not add a max retry policy or surface the close code.

  - file: [src/gateway/client.ts:900](https://github.com/openclaw/openclaw/blob/bdfb408ce6eeb724a638c106104dbbcae50b612c/src/gateway/client.ts#L900)
