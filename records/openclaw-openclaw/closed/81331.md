## Summary

Keep open: the patch fixes one config helper, but it does not update the gateway startup channel-presence path that selects configured external channel plugins, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR treats `{ enabled: true }` as meaningful channel config in `src/config/channel-configured-shared.ts` and adds an `isChannelConfigured` regression test.

## Best Possible Solution

Make enabled-only channel config semantics consistent across channel-configured and gateway startup presence discovery, then prove the installed external Weixin plugin is selected at gateway startup.

## Reproduction Assessment

Yes, at source level: current main still treats `{ enabled: true }` as not meaningful in `src/channels/config-presence.ts`, while gateway startup uses that path to build configured channel ids. I did not live-run the packaged Weixin plugin in this read-only review.

## Solution Assessment

No. Updating only `src/config/channel-configured-shared.ts` is not the narrow complete fix because the reported gateway startup path uses `src/channels/config-presence.ts`; the patch should update or unify that helper and add startup/plugin-plan coverage.
