## Label Justifications

- `P1`: The reported 2026.4.26 behavior broke live gateway control-plane and channel workflows under real orchestration load.
- `impact:crash-loop`: The gateway process stayed alive but effectively hung for `/readyz` and local control paths, which is an availability failure.
- `impact:message-loss`: The report includes dropped typing indicators, delayed Discord status output, and stale channel replies.
- `impact:session-state`: The reload blocker was stale active reply, embedded-run, and task-run accounting that kept config changes from applying.

## Summary

Current main and v2026.6.6 cover the reported 2026.4.26 failure cluster with bounded restart deferral, task-blocker diagnostics/recovery, readiness event-loop health, and `logs --follow` reconnect behavior; broader control-plane separation remains tracked separately.

## What This Changes

The issue asks OpenClaw to keep gateway readiness, control-plane, log-follow, and reload paths responsive under 2026.4.26 orchestration load and to bound stale active-run deferral.

## Best Possible Solution

Keep the shipped v2026.6.6 gateway stability fixes, use https://github.com/openclaw/openclaw/issues/42026 for future control-plane/runtime separation, and ask for a fresh current-version diagnostic export if the same symptom recurs.

## Reproduction Assessment

Yes for the affected 2026.4.26 release: the report gives concrete loopback `/readyz` timeouts, CPU-hot main-thread samples, reload-deferral logs, and `logs --follow` disconnect output. I did not reproduce the old release live during this read-only review, and current main now has targeted source fixes for those paths.

## Solution Assessment

Yes. The implemented path is the maintainable fix for this report: bound reload deferral, expose event-loop health in readiness, recover interrupted sessions, and make log following reconnect, while leaving the larger distributed-runtime architecture to https://github.com/openclaw/openclaw/issues/42026.
