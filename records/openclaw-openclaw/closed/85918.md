## Summary

Keep open: current main still filters DeepSeek DSML tool-call text on the OpenAI completions stream instead of promoting complete, valid DSML invocations into executable tool calls, and no open or merged OpenClaw PR appears to own that remaining DSML-ingestion work.

## What This Changes

The issue requests recovering complete DeepSeek/Foundry DSML tool-call markup emitted as assistant text into executable toolCall blocks on the OpenAI completions stream.

## Best Possible Solution

Add a narrowly gated DeepSeek/Foundry DSML recovery path in the OpenAI completions stream that preserves native `tool_calls`, promotes only complete valid invocations, and leaves malformed DSML non-executable.

## Reproduction Assessment

Yes, at source level: the reporter supplied a concrete live CLI repro, and current main shows the completions stream only promotes native `choiceDelta.tool_calls` while DSML text is filtered or diagnosed as text. I did not re-run live Foundry credentials in this read-only pass.

## Solution Assessment

Yes, if implemented narrowly: a conservative recovery layer in `processOpenAICompletionsStream` or an equivalent provider-specific seam is the maintainable fix for existing completions configs. The fix should not broaden into general XML/prose tool execution.
