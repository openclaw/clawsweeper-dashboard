## Summary

Current `main` already provides a native restart path for agents and config writes that avoids the reported self-termination pattern. The supported model-facing `gateway` tool is exposed by default, its `restart` action schedules an in-process SIGUSR1 restart instead of shelling out, and `config.patch`/`config.apply` use the same scheduler when a restart is required. The issue describes agents bypassing that supported surface with raw `pkill`, not a missing core restart mechanism on current `main`.

## Close Comment

Closing as `implemented_on_main` after a Codex review. Current OpenClaw already has a native restart mechanism for this case: agents get the model-facing `gateway` tool by default (`src/agents/openclaw-tools.ts:255`), its `restart` action schedules an in-process SIGUSR1 restart instead of using `pkill` (`src/agents/tools/gateway-tool.ts:357`, `src/infra/restart.ts:466`), and restart-required `config.patch` / `config.apply` calls use that same scheduler after the write (`src/gateway/server-methods/config.ts:581`, `src/gateway/server-methods/config.ts:654`). The CLI path is also safe here: `openclaw gateway restart` signals the verified gateway listener with SIGUSR1 rather than kill-and-relaunch (`src/cli/daemon-cli/lifecycle.ts:124`). This implementation is present in the shipped `v2026.4.22` release (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). If you can still reproduce raw `pkill` behavior on a current build, please open a fresh issue with the exact prompt/tool transcript, because that would point to an agent instruction/tool-selection regression rather than a missing restart facility.
