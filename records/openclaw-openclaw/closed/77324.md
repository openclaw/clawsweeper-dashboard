## Summary

Keep open: the report describes a real transcript durability contract violation, and current main still has separate live-delivery and durable-transcript paths with tests that intentionally avoid mirroring normal agent-run assistant finals.

## What This Changes

WebChat/Control UI should persist every displayed streamed assistant message into the session transcript so later turns can see it.

## Best Possible Solution

Add guarded post-run reconciliation for WebChat agent-run finals: verify the durable transcript contains the Pi assistant turn, then idempotently append a fallback or surface an unsaved warning only when it is still missing.

## Reproduction Assessment

No, not with high confidence from an end-to-end current-main run. Source inspection does show the divergence path: live assistant finals can render from gateway chat events while durable history depends on a separate Pi transcript write.

## Solution Assessment

No, blocking UI render on persistence ack is not the best primary fix because streaming and transcript persistence are intentionally separate. The safer fix is a post-run reconciliation/fallback that preserves duplicate prevention when Pi already wrote the assistant turn.

Fix https://github.com/openclaw/openclaw/issues/77324. WebChat/Control UI can display agent-run streamed/final assistant text from live gateway events while the durable session transcript relies on Pi `message_end`; if that Pi write is missed, later turns lose visible assistant context. Implement a narrow gateway-side reconciliation/fallback for normal WebChat agent-run finals: after the run completes, verify the durable transcript for the session contains the corresponding assistant turn before idempotently appending a gateway fallback or surfacing an unsaved warning. Do not reintroduce blind mirroring that duplicates normal Pi-managed assistant turns, and do not change unrelated channel delivery behavior. Add regression coverage for the missing-Pi-assistant case and for the already-persisted case. Add a user-facing changelog entry.
