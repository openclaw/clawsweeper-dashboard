## Summary

Close as superseded: current main still lacks this multi-workspace config, but the remaining safe multi-root filesystem work is now better represented by the active `tools.fs.roots` proposal, while this stale branch is conflicting, lacks real behavior proof, and still has source-reproducible runtime and security-boundary blockers.

## What This Changes

The branch adds `agents.list[].multipleWorkspaces`, resolves those entries into a `workspace-composite-<agentId>` directory, populates it with symlinks, and passes that composite workspace into the reply path.

## Best Possible Solution

Keep the remaining work on the explicit filesystem-root contract in https://github.com/openclaw/openclaw/pull/52951; if maintainers reject that direction, open a fresh narrow design rather than revive this symlink-composite branch.

## Reproduction Assessment

Not applicable for the central request as a current-main bug because this is an additive feature. Yes for PR blockers: source inspection shows non-reply workspace setup paths omit `multipleWorkspaces`, and root-like workspace paths can derive an empty symlink name.

## Solution Assessment

No. The symlink-composite implementation changes the workspace and filesystem boundary at once; the cleaner current direction is an explicit, maintainer-approved multi-root access contract with tests and proof.
