## Summary

Keep open. Current `main` still does not implement Feishu-side handling for `audioAsVoice`/MP3 TTS output: speech/TTS code marks Feishu as a native voice-note channel, but the Feishu outbound path only sends native audio for `.ogg`/`.opus` and its tests explicitly assert that MP3 is sent as a generic file attachment.

## Evidence

- **Speech core treats Feishu as a native voice-note channel:** `supportsNativeVoiceNoteTts()` includes `feishu`, TTS requests target `voice-note` for that channel, and successful TTS sets `audioAsVoice` when the provider says the audio is voice-compatible.
  - file: [extensions/speech-core/src/tts.ts:594](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/extensions/speech-core/src/tts.ts#L594)

  - sha: [59523e66da03](https://github.com/openclaw/openclaw/commit/59523e66da03591bc7bcc183a11f690fa0aae641)
- **`/tts audio` already forwards the voice-note intent:** The slash-command path builds a reply payload with `audioAsVoice: result.voiceCompatible === true`, so the Feishu channel is expected to consume that hint.
  - file: [src/auto-reply/reply/commands-tts.ts:167](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/src/auto-reply/reply/commands-tts.ts#L167)

  - file: [src/infra/outbound/deliver.ts:100](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/src/infra/outbound/deliver.ts#L100)
