## Summary

Keep this issue open: current main and the latest stable release still use the fixed 512-byte recent-session header probe, and the linked open PR is the active fix candidate for the same session-resume bug.

## What This Changes

The issue reports that `continueRecent()` can start a fresh session when `findMostRecentSession()` rejects a valid JSONL transcript whose first header line exceeds 512 bytes.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/94578, or an equivalent bounded first-line scanner fix with regression coverage, then close this issue after merge.

## Reproduction Assessment

Yes. Source inspection shows `loadEntriesFromFile()` can read the full valid JSONL transcript while `findMostRecentSession()` validates only the first 512 bytes before `continueRecent()` decides whether to resume; I did not run a live repro because this review is read-only.

## Solution Assessment

Yes in shape. The maintainable fix belongs in the recent-session header validation path as a bounded first-line reader with regression coverage, rather than loading whole transcripts during directory scans.
