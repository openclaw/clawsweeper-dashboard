## Summary

Keep open. The PR fixes a real Codex app-server projection gap, but the current head still publishes new global tool-stream payloads without the existing Pi-style sanitizer and is missing the required user-facing changelog entry.

## What This Changes

The PR adds Codex app-server normalized `stream: "tool"` start, update, and result events plus projector tests for command, turn-completed, and bulky-output redaction cases.

## Best Possible Solution

Keep the Codex tool-stream bridge, but route start args, update partials, final results, and errors through the shared tool redaction contract, preserve the full-output gates, add secret-bearing regression tests, and add a changelog entry.

## Reproduction Assessment

Yes. On the PR head, instantiate `CodexAppServerEventProjector`, send MCP or dynamic tool items with secret-like `arguments`, `partialResult`, or `error.message`, and inspect `onAgentEvent`; the new tool payloads are emitted directly from the projector.

## Solution Assessment

No. Bridging Codex app-server items into normalized tool events is the right narrow direction, but this implementation is not the best safe version until it matches the existing Pi producer sanitizer and release-note policy.
