## Summary

Current main already has a built-in persistent memory stack that covers the issue's functional request: file-backed cross-session memory, proactive pre-reply recall, automatic pre-compaction memory flush, background consolidation, and a compiled durable knowledge layer.

## Evidence

- **Vision scope already defines memory as a plugin-owned builtin surface:** The repo vision says core stays lean, optional capability should usually ship as plugins, and memory is a dedicated plugin slot with one active memory plugin at a time.
  - file: [VISION.md:54](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/VISION.md#L54)

  - file: [docs/concepts/memory.md:9](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/docs/concepts/memory.md#L9)

## Risks / Open Questions

- The shipped design is not the exact proposed hot/warm/cold compaction-tree architecture, so a future issue would still be needed if maintainers want that specific implementation rather than the current memory-core + active-memory + memory-wiki approach.
