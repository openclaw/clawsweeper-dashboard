## What This Changes

The branch adds an opt-in `skipIfLoadInFlight` parameter to `resolvePluginProviders` and has provider hook-list resolution use it to avoid repeating provider load-state/in-flight probes.

## Best Possible Solution

Land the narrow hot-path cleanup after maintainer acceptance of the resolver option surface, keeping direct `resolvePluginProviders` callers on their default load behavior.

## Reproduction Assessment

Yes for the source-level behavior: current main computes provider load state once in `isPluginProvidersLoadInFlight` and again in `resolvePluginProviders` for hook resolution. I did not run a profiler in this read-only pass.

## Solution Assessment

Yes. The branch moves the opt-in in-flight skip into the resolver used by the hook path, which removes duplicated load-state work while leaving ordinary resolver calls on their existing behavior; the remaining maintainer call is whether the small optional flag is acceptable on that runtime seam.
