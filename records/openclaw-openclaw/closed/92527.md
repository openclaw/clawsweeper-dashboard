## Label Justifications

- `P1`: The PR targets a shipped credential exposure, but its current implementation would break the Gmail watcher workflow for deployed users.
- `merge-risk: 🚨 compatibility`: The patch requires two CLI flags absent from the latest official `gogcli` command schema.

- `merge-risk: 🚨 security-boundary`: The patch changes authentication-secret transport and temporary storage, including cleanup behavior on failures.

## Summary

The security problem is real and shipped, but this patch cannot work with the current official `gogcli`: its generated `gmail settings watch serve` schema exposes only `--token` and `--hook-token`, not the proposed file flags. The PR should stay open for a coordinated dependency-first fix, and its prose-only production claim is not sufficient to override the dependency contract.

## What This Changes

The PR writes Gmail watcher push and hook tokens to private temporary files, passes them through `--token-file` and `--hook-token-file`, cleans the files on child error or exit, and adds unit coverage for the new arguments and file permissions.

## Best Possible Solution

Add and release the two file-based secret flags in `openclaw/gogcli` first, define the minimum compatible version or capability check in OpenClaw, then make temporary-file creation/spawn exception-safe and land this caller change with fresh-install, upgrade, and redacted live process proof.

## Reproduction Assessment

Yes. OpenClaw v2026.6.6 passes the raw push and hook tokens in the long-lived `gog` child argv, and the proposed branch deterministically invokes flags absent from the current dependency schema.

## Solution Assessment

No, not as a caller-only patch. File-based secret input is a good end state, but the dependency must implement and release the contract first, followed by explicit compatibility and cleanup handling in OpenClaw.
