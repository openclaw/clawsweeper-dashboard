## Summary

Current main already prevents Discord bots from remaining indefinitely in the "awaiting gateway readiness" state by timing out READY after both startup and post-connect websocket-open events, marking the account disconnected, and force-stopping/restarting the lifecycle before the health monitor would need to notice it.

## Evidence

- **Startup READY timeout added:** `waitForGatewayReady()` now treats a startup READY stall as a failure after 15s, pushes `connected: false` with `startup-not-ready`, restarts the gateway once, and throws if READY still never arrives.

## Close Comment

Closing as implemented on `main` after Codex review. The reported stuck state is now handled directly in the Discord lifecycle rather than waiting for the generic health monitor: `extensions/discord/src/monitor/provider.lifecycle.ts:180` starts a READY watchdog when the websocket opens and force-stops with `connected: false` / `runtime-not-ready` if READY never arrives, and `extensions/discord/src/monitor/provider.lifecycle.ts:276` does the same for startup with a restart path. The regression coverage in `extensions/discord/src/monitor/provider.lifecycle.test.ts:218` and `extensions/discord/src/monitor/provider.lifecycle.test.ts:504` exercises both "startup never reaches READY" and "runtime reconnect opens but never becomes ready". I was not able to tie this to a released tag from the available shallow history, so this is confirmed on current `main` (`a6aa151653ab478e9aa23e95c6446c41adc4fce1`) rather than a specific shipped release.
