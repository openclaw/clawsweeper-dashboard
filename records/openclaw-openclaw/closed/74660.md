## Summary

Current main already covers the reported echo path: runtime-context custom messages are stripped before provider conversion, the queued custom message no longer carries the human-readable runtime-context preface, and user-facing sanitization removes copied internal-context delimiters and runtime-context prefaces. Direct tag inspection shows the same behavior is present in v2026.4.27.

## What This Changes

The issue requests wrapping runtime context custom messages in OpenClaw internal-context delimiters so echoed runtime context is stripped from user-facing responses.

## Best Possible Solution

Keep the current runtime-context separation model: hidden runtime-context custom messages stay out of LLM-facing history, prompt-local runtime context remains transient, copied runtime-context envelopes stay sanitized, and broader persisted-history work remains tracked separately under #62306.
