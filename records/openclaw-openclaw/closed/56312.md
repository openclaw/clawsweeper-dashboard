## Summary

Keep open. Current main can download ordinary small Telegram `video` media, but it still does not fully solve this report: the linked animation/GIF-video PR was closed unmerged, the current Telegram inbound media pipeline still omits `msg.animation`, and the reported 60 MB video case can still fall through the `getFile` too-large/null path with no saved inbound file.

## Best Possible Solution

Keep this issue open for a Telegram plugin fix or maintainer split. The likely path is to add `animation` to the same inbound detection, file-id, download, and placeholder paths as `video`, add regression coverage for inbound animation/video downloads, and separately document or surface the Telegram `getFile` large-file limitation versus OpenClaw's `mediaMaxMb` cap, including the self-hosted Bot API/trusted local roots path where appropriate.
