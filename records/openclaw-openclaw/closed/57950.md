## Summary

Current `main` still wires `browser.request` whenever the bundled browser plugin is actually activated. The report is better explained by config/runtime state mismatch than by a reproducible browser-plugin registration bug on `main`.

## Evidence

- **Browser plugin registers method and service together:** The bundled browser plugin registers the `openclaw browser` CLI, the `browser.request` gateway method, and the `browser-control` service in one place.
  - file: [extensions/browser/plugin-registration.ts:28](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/extensions/browser/plugin-registration.ts#L28)
  - command: `nl -ba extensions/browser/plugin-registration.ts | sed -n '20,45p'`
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)
- **Gateway bootstrap test proves method appears when plugin loads:** Current integration coverage asserts that loading the bundled browser plugin adds `browser.request` and the browser control service.
  - file: [src/gateway/server-plugin-bootstrap.browser-plugin.integration.test.ts:42](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/src/gateway/server-plugin-bootstrap.browser-plugin.integration.test.ts#L42)
  - command: `nl -ba src/gateway/server-plugin-bootstrap.browser-plugin.integration.test.ts | sed -n '35,65p'`
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)
- **Inspect/list are not live runtime probes:** The docs now explicitly say `plugins list` is not a live probe of an already-running Gateway and `plugins inspect` is a module-loaded inspection pass, so seeing `browser.request` there does not prove the running Gateway process has that method.
  - file: [docs/cli/plugins.md:203](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/docs/cli/plugins.md#L203)
  - command: `nl -ba docs/cli/plugins.md | sed -n '190,220p'`
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)
- **Docs match the reported symptom to plugin activation state:** Current browser troubleshooting docs say that missing `browser.request` / unavailable browser tool usually means the `browser` plugin was not activated in the running Gateway, most commonly because of `plugins.allow` or plugin enablement state.
  - file: [docs/tools/browser.md:68](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/docs/tools/browser.md#L68)
  - command: `nl -ba docs/tools/browser.md | sed -n '56,82p'`
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)
- **Reporter confirmed resolution:** In the issue thread, the reporter replied on 2026-04-02 that setting the browser plugin enabled in config resolved the problem.

## Risks / Open Questions

- Repo inspection cannot prove which local Gateway process/profile the reporter restarted; a local service/config mismatch outside the repo is still possible.
- The clearer browser troubleshooting/docs appear to be newer than the affected March 2026 builds, so older installs may still have produced confusing signals.
