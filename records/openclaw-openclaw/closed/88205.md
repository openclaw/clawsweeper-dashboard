- `impact:crash-loop`: The issue is about gateway restart/update being blocked by stale active-work state, a process availability problem.

## Summary

Keep open. Current main has stale-task audit, maintenance, and restart timeout/force mitigations, but the central ACP zombie cleanup is still incomplete: ACP task reconciliation treats a persisted ACP session entry as live backing, so a dead run with surviving metadata can remain `running` and continue to appear as a restart blocker.

## What This Changes

The issue asks OpenClaw to detect and terminate stale ACP background task runs so dead `running` rows cannot block gateway restart or update.

## Best Possible Solution

Teach task maintenance and restart-blocker reconciliation to recover or mark stale parent-owned ACP task runs terminal/lost when their ACP runtime backing is known dead, while preserving healthy bound ACP sessions.

## Reproduction Assessment

Yes, with medium confidence from source: create an old `running` ACP task with a surviving ACP child session entry and current maintenance keeps it active because metadata exists. I did not run a live ACP process crash/restart scenario in this read-only review.

## Solution Assessment

No for the literal configurable TTL as the primary fix; the safer repair is to extend existing task maintenance so ACP rows are reconciled from authoritative runtime/process state and only dead backing runs are marked terminal or lost.
