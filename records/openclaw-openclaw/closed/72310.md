## Summary

Current main now addresses the reported default plugin-scanning churn. The former 1-second discovery/manifest snapshot TTL knobs are gone, gateway startup builds a plugin metadata snapshot from the persisted local plugin registry, startup planning and activation reuse that manifest registry, and runtime loading can skip fresh discovery when a manifest registry is supplied. The proof commit is on current main after the latest release tag, so this can close as implemented on main but not yet shipped in a tagged release.

## What This Changes

The issue asks OpenClaw to make default plugin scanning viable on mid-range or low-core hardware by reducing scan/cache churn, adding smarter caching or hysteresis, and possibly offering a constrained-host performance mode.

## Best Possible Solution

Keep the current persisted-registry and metadata-snapshot startup path, close this broad report as fixed on main, and handle any remaining low-core tuning as a new measured performance regression if operators can still reproduce excessive scanning after this commit.
