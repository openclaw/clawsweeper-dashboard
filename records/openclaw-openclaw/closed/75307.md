## Summary

Current `main` and the shipped `v2026.4.29` release already cover the reported failure mode: interrupted cron runs are marked failed on startup, stale task records are reconciled or projected out of active counts, and the cron timer has a regression guard against the stuck-`runningAtMs` hot loop that matches the CPU saturation symptom.

## What This Changes

Issue requests that in-flight cron, subagent, and CLI task records be reconciled to a terminal lost/interrupted state after a gateway crash or restart so channel reloads are not blocked indefinitely.

## Best Possible Solution

Keep the shipped `v2026.4.29` startup reconciliation, task maintenance, and cron timer guard behavior; users still on `2026.4.27` should upgrade, and any new post-upgrade case should be opened with fresh task registry, cron store, and reload logs.

## Reproduction Assessment

Yes. The codebase has focused regression paths for this class: restart with persisted `runningAtMs` marks cron jobs interrupted, and a stuck `runningAtMs` plus past `nextRunAtMs` no longer arms a zero-delay timer loop.

## Solution Assessment

Yes. The current approach is the narrow maintainable fix: reconcile durable task state at startup/maintenance, project stale records out of active reload counts, and prevent the scheduler hot loop without forcing broad channel reload behavior changes.
