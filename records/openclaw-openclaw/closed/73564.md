## Summary

Keep open. Current main still has the reported Signal attachment bug, the latest shipped release still contains the same hardcoded 1 MiB RPC response cap, and the open closing PR #73572 has not merged, so this issue should remain open for review/landing or an equivalent fix.

## What This Changes

Signal inbound attachment fetch should let `channels.signal.mediaMaxMb` control the effective HTTP RPC response cap instead of using an unconfigurable 1 MiB limit.

## Best Possible Solution

Implement the fix in the Signal plugin by adding a bounded per-call RPC response cap for `getAttachment`, deriving it from the effective per-account media limit with base64 and JSON headroom, and keeping the existing conservative default for ordinary RPC/check paths.

## Reproduction Assessment

Yes. A high-confidence reproduction exists from current main alone: make `signalRpcRequest` receive a response larger than `1_048_576` bytes and it rejects before parsing, while the attachment path reaches that same client call before the decoded `mediaMaxMb` check can run.

## Solution Assessment

Yes. The narrowest maintainable solution is the shape already proposed in #73572: thread `maxResponseBytes` through the Signal RPC client and use it only for attachment fetches based on the effective media cap, instead of weakening the shared client guard.
