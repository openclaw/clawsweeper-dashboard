## Summary

Keep this issue open. Current main still has independent WhatsApp reaction, direct text, and auto-reply/quoted-reply delivery paths, and it still does not reconcile model/session text with provider dispatch acceptance or expose a clear operator warning when text exists but WhatsApp delivery is not confirmed.

## Best Possible Solution

Keep this issue open and implement a WhatsApp-owned delivery-state snapshot around direct sends, reactions, and auto-reply/quoted-reply delivery. The fix should record whether model text was produced, whether a provider dispatch was invoked, mode (`message`, `reply`, `reaction`, media variants), provider message id or error when available, and whether no outbound provider send occurred; then surface an operator/debug warning for text produced but not delivered and add regression coverage for reaction-success/text-or-reply-failure split paths.

## Clownfish Work Prompt

Fix the WhatsApp delivery-state visibility gap reported here. In the WhatsApp plugin, add mode-aware delivery-state observability around direct `sendMessageWhatsApp`, `sendReactionWhatsApp`, auto-reply `deliverWebReply`, and `dispatchWhatsAppBufferedReply`. The goal is to distinguish model/session text from provider transport delivery: record whether text/media was produced, whether provider dispatch was invoked, send mode (`message`, `reply`, `reaction`, and media where useful), provider message id when available, provider error when delivery fails, and whether no outbound provider send occurred. Surface a clear operator/debug warning when visible text is produced but WhatsApp provider delivery is not invoked or accepted. Keep the implementation WhatsApp-owned; do not move WhatsApp-specific policy into core. Be careful not to treat reaction void results as failed delivery merely because they have no message id. Add regression tests where ack/reaction succeeds but text or quoted reply delivery fails, and ensure transcript/echo tracking is not treated as proof of WhatsApp delivery. Update WhatsApp docs if new warning/log fields or troubleshooting guidance are user-visible. Changelog: add a single-line Fixes entry with reporter attribution if this becomes user-facing behavior; no changelog for test-only plumbing.
