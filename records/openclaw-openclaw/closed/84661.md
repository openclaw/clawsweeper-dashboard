## Best Possible Solution

Merge a narrow path-aware parse error fix after the contributor adds redacted real CLI output and the wrapper reports the parse reason once.

## Reproduction Assessment

Yes. Source inspection shows current main parses the plan JSON directly in `readPlanFile`, then the outer apply handler formats the resulting syntax error without the plan path; I did not run the CLI path because this review is read-only.

## Solution Assessment

Mostly yes. `readPlanFile` is the narrow fix point because it has both the plan path and parse operation, but the wrapper should avoid combining an embedded formatted cause with `{ cause: err }`.

## Review Findings
