## Summary

Keep #57811 open. Current main has fixed the separate cron `--model` precedence/forwarding issue, and the Ollama plugin can route a custom provider base URL once its provider hook is invoked. But the core reported failure for custom provider ids such as `ollama-mini`/`ollama-spark` is not fully proven fixed: provider runtime lookup still resolves by provider id/aliases/hookAliases, while the bundled Ollama manifest and registration only claim `ollama`, not arbitrary `models.providers.<id>.api === "ollama"` entries.

## Best Possible Solution

Keep the issue open and fix the remaining provider ownership gap. The best path is a plugin/runtime contract that lets the Ollama provider own configured `models.providers.<id>` entries with `api: "ollama"`, or an explicit hook-alias/activation mapping derived from configured provider ids, then add regression coverage for isolated cron and `openclaw capability model run --gateway` using a non-canonical provider id such as `ollama-mini`. Preserve the already-implemented cron `--model` override behavior.
