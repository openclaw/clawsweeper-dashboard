## Summary

Current main does not support the reported root cause. The dedicated `cron` tool is registered as `cron`, flows through the shared tool-policy pipeline, and deny rules take precedence. The issue thread itself later narrowed the observed behavior to `exec` running `openclaw cron ...`, which is a separate capability-parity concern rather than a direct `cron`-tool deny-list bypass.

## Evidence

- **Deny rules win for tool-name matching:** `isToolAllowedByPolicyName` compiles deny/allow globs and returns `false` immediately on a deny match before checking allow rules.
  - file: [src/agents/tool-policy-match.ts:5](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/src/agents/tool-policy-match.ts#L5)

  - sha: [56fe2aab9c87](https://github.com/openclaw/openclaw/commit/56fe2aab9c87596515f2280c8060dc740ea64a88)
- **Persistent agents use the shared policy pipeline:** `createOpenClawCodingTools` resolves effective agent/global policy from the session key, then applies owner-only filtering and `applyToolPolicyPipeline` to the constructed tool list. There is no cron-specific bypass in this path.
  - file: [src/agents/pi-tools.ts:353](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/src/agents/pi-tools.ts#L353)
