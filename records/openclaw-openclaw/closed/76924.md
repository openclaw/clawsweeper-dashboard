## What This Changes

The issue asks OpenClaw to surface Anthropic content-filter invalid_request_error responses, notify the user, and stop replaying the same blocked tool-call response indefinitely.

## Best Possible Solution

Keep the current main behavior and ship it in the next release; reopen only if the same Anthropic content-filter response can still loop after this main-only fix.

## Reproduction Assessment

No, not as a current-main failure. Source inspection shows the reported Anthropic invalid_request_error path now becomes a visible provider-error payload, and replay-safe silent-error retries are capped.

## Solution Assessment

Yes. Handling this in the central embedded-run and provider-error payload path is the narrow maintainable fix because the failure is a runtime liveness/error-surfacing problem, not a new Anthropic-specific user feature.
