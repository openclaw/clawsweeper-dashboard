## Summary

Keep open: this is the active candidate fix for the linked Slack message.send bug, but current head still drops the safe receipt identifiers the report asks to preserve, exposes an internal reconciler as public SDK API, and has only synthetic test proof rather than real behavior proof.

## What This Changes

The PR adds a delivered-message result reconciler to Codex and embedded agent tool-result middleware paths, exports it through the agent harness SDK, and adds regression tests for middleware failures after message sends.

## Best Possible Solution

Land a narrow internal runtime reconciler that snapshots privacy-safe delivery IDs before middleware mutation, keeps non-send/internal/deferred paths fail-closed, resolves or documents any SDK surface deliberately, and adds current-head real behavior proof.

## Reproduction Assessment

Yes from source inspection: current main executes message.send before tool-result middleware and returns buildMiddlewareFailureResult() when middleware throws or returns invalid output, while Slack send results are nested under details.result. I did not run tests because this review is read-only.

## Solution Assessment

No: the runtime layer is right, but the current PR is not the best final fix until it snapshots safe receipt identifiers and avoids or deliberately documents the public SDK export. The safer path keeps the helper internal unless maintainers choose a plugin-facing contract.
