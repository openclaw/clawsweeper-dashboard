## Summary

Keep open: current main and the latest release still have the webchat-only internal source-reply gate, while the documented `/usage` behavior promises a per-response footer. The report is narrow, source-backed, and not superseded by the related issues in the provided context.

## What This Changes

Requests that Telegram `message_tool_only` replies use the normal decorated reply path so `/usage tokens` and `/usage full` footers appear on visible Telegram replies.

## Best Possible Solution

Route implicit current-source `message_tool_only` sends for Telegram and other channels through the internal source-reply payload path, preserve explicit target sends as normal outbound delivery, copy reply payload metadata when appending usage, and cover Telegram DM/topic plus explicit-route exceptions in tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a Telegram `message_tool_only` send with no explicit route does not satisfy the webchat-only internal sink condition, so it can be delivered before `appendUsageLine` decorates final payloads; I did not run a live Telegram probe in this read-only pass.

## Solution Assessment

Yes. The narrow maintainable direction is to generalize the implicit current-source sink beyond webchat, keep explicit-route sends on the outbound path, and preserve reply payload metadata when appending the usage footer.
