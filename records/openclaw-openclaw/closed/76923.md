## Summary

Keep open: the coordinator/RPC direction is plausible, but the new safe CLI path bypasses the installed service target resolution used by normal restarts, so custom-port/profile managed gateways can be missed. I also found a smaller docs reference omission.

## What This Changes

The PR adds a safe Gateway restart coordinator, restart preflight/request RPC methods, explicit `--safe` gateway/daemon restart handling, aligned task restart blockers, docs, changelog, and tests.

## Best Possible Solution

Land a revised PR that keeps the explicit safe restart design, targets the same installed Gateway service/port/env as normal restart, updates all Gateway CLI option docs, and adds focused regression coverage.

## Reproduction Assessment

Yes, by source inspection. The new safe path returns before `resolveGatewayLifecyclePort`, while current install/restart code stores and uses custom service ports in program arguments and `OPENCLAW_GATEWAY_PORT`.

## Solution Assessment

No, not yet. The RPC/coordinator approach can be the right shape, but the CLI needs to resolve the installed service target consistently with normal restart before this is merge-ready.

Preserve the explicit `--safe`/RPC design, but make safe restart target the running installed Gateway consistently with normal restart, preferably by resolving the service command/environment/port before the RPC and passing an explicit loopback URL or shared lifecycle RPC target into `callGatewayCli`. Do not change default restart, `--force`, or `--wait` semantics; do not add a new config option.
