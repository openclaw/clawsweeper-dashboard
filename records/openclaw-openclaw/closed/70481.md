## Summary

Keep open. Current `main` still contains a cleanup path that can add a 30s wait after the configured active-memory timeout, so a `timeoutMs: 5000` run can still block `before_prompt_build` for roughly 35s. That matches the report closely, and the docs still describe `config.timeoutMs` as a hard timeout for the blocking memory sub-agent.

## Evidence

- **`before_prompt_build` blocks on active-memory recall:** The plugin's `before_prompt_build` hook awaits `maybeResolveActiveRecall(...)` before returning prompt context, so any overrun directly delays the main reply path.
  - file: [extensions/active-memory/index.ts:2024](https://github.com/openclaw/openclaw/blob/0e9156d2053022d208c3eea1d0fa1e377ed0a154/extensions/active-memory/index.ts#L2024)

  - sha: [0e9156d20530](https://github.com/openclaw/openclaw/commit/0e9156d2053022d208c3eea1d0fa1e377ed0a154)
- **Configured timeout is forwarded and a local abort timer is armed:** `timeoutMs` is passed into `runEmbeddedPiAgent(...)`, and the plugin also starts its own `AbortController` timer at `params.config.timeoutMs` before awaiting the subagent.
  - file: [extensions/active-memory/index.ts:1678](https://github.com/openclaw/openclaw/blob/0e9156d2053022d208c3eea1d0fa1e377ed0a154/extensions/active-memory/index.ts#L1678)

- **Runner cleanup waits for idle with no shorter timeout override:** After the attempt finishes, cleanup calls `flushPendingToolResultsAfterIdle(...)` with `clearPendingOnTimeout: true` but without a `timeoutMs` override, so cleanup inherits the helper default even for aborted/timed-out runs.
