## Summary

Keep open: current main still treats channel/plugin sidecars as part of Gateway readiness and WebSocket/TUI attach gating, so the requested core-ready boundary is not implemented. Related open items cover narrower slices, but they do not supersede this broader startup contract decision.

## What This Changes

Request an explicit Gateway/TUI core-ready milestone that comes up before channel, plugin, bundled-dependency, and other sidecar startup work can block attachment.

## Best Possible Solution

Define an additive core-ready readiness model that exposes core Gateway control surfaces early while reporting channels, plugin services, bundled dependency diagnostics, and hooks as pending or degraded sidecars.

## Reproduction Assessment

No for the exact downstream wedge: I did not run the macOS/Node 24 recovery scenario. Yes at source-contract level: current main still gates /readyz, WebSocket connect, and startup-gated methods on sidecar completion.

## Solution Assessment

Unclear: the requested direction is plausible, but the best fix needs a maintainer decision on which Gateway/TUI surfaces are core-ready-safe before changing readiness and protocol semantics.
