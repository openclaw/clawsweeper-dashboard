## Summary

Current main fixes the reported update-check state-dir permission regression by moving update-check persistence out of `update-check.json` and into the shared SQLite state database, whose open/write path hardens managed state directories and database files. The fix is present on current main but not in the latest release tag inspected, so this can close as implemented on main only.

## What This Changes

The issue reports that startup update-check JSON writes can chmod the OpenClaw state directory back to an umask-derived group-writable mode after it was hardened to `0700`.

## Best Possible Solution

Keep the current SQLite-backed update-check state path and let the broader managed-directory hardening continue under its separate related issue and PR.

## Reproduction Assessment

No current-main repro remains: the release-era source path wrote `update-check.json` through `writeJson`, but current main reads and writes update-check state through shared SQLite instead. The old behavior is still source-proven in `v2026.6.10`, which confirms why the report was valid before the main-only fix.

## Solution Assessment

Yes, the current main solution is the better fix for this issue because it removes the steady-state JSON file path entirely and migrates legacy JSON state through doctor. Broader `mkdir` hardening is separate and remains tracked outside this issue.
