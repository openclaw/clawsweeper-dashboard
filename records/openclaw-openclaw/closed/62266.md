## Summary

Current `main` does not contain this PR's runtime-fallback wiring. `createImageTool()` still returns `null` whenever config-driven image-model resolution fails, even though the runner already knows `modelProvider`, `modelId`, and `modelHasVision` when it builds tools. There is also a newer linked bug (`#70482`) on the same opencode-go/image-tool area, so there is no high-confidence basis to close this PR.

## Evidence

- **Image tool still hard-disables without config-derived image model:** On current `main`, `createImageTool()` only accepts `modelHasVision`; it does not accept runtime `modelProvider`/`modelId`, and it returns `null` immediately when `resolveImageModelConfigForTool(...)` returns `null`. That is the exact gap this PR is trying to fill.

- **Tool assembly still drops runtime provider/model before calling image tool:** `createOpenClawTools()` has `modelProvider` and `modelId` in its options type, but the current `image` tool call only forwards `config`, `agentDir`, workspace/sandbox state, and `modelHasVision`. The PR's wire-through is not on `main`.
