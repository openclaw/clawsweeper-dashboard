## Summary

Keep this PR open. It changes the generic no-timeout gateway fallback, but the reported `openclaw message send --media` path still passes an explicit `10000` ms timeout before reaching that helper, and the PR has no after-fix real large-file send proof.

## What This Changes

The PR changes `src/gateway/call.ts` so gateway calls without an explicit timeout resolve the shared preauth handshake timeout helper and default to 15000 ms.

## Best Possible Solution

Align gateway-executed message actions with the shared timeout policy while preserving explicit caller overrides, update focused gateway/message-action regression coverage, and require redacted real-send proof before merge.

## Reproduction Assessment

Yes by source inspection, though not by a live WeCom send. The reported CLI command reaches `messageCommand` without a timeout override, `resolveGatewayActionOptions` fills `10_000`, and the gateway message-action RPC passes that explicit value.

## Solution Assessment

No. The PR raises the generic fallback for calls that omit `timeoutMs`, but the reported path already supplies `timeoutMs: 10_000`; the narrower maintainable fix needs to cover the message-action resolver or stop passing that hard-coded default.
