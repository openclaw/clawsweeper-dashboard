## Summary

Current main already fixes the reported Homebrew PATH failure for macOS skill binary detection by explicitly adding `/opt/homebrew/bin` and `/usr/local/bin` to the app’s preferred PATH and propagating that PATH into the macOS app launch/service flows that back skill checks.

## Risks / Open Questions

- The implementation hardcodes the common Homebrew/default macOS bin directories instead of literally parsing `/etc/paths.d`, so unusual custom-only PATH entries would be a separate follow-up from the reported Homebrew case.

## Close Comment

Closing from a Codex review because current `main` already covers the reported macOS/Homebrew failure mode. The app now resolves binaries through `CommandResolver.preferredPaths()`, which explicitly includes `/opt/homebrew/bin` and `/usr/local/bin` (`apps/macos/Sources/OpenClaw/CommandResolver.swift:78`), and that PATH is propagated into the app launch agent (`apps/macos/Sources/OpenClaw/LaunchAgentManager.swift:49`) plus the local node/gateway command paths used by the macOS app (`apps/macos/Sources/OpenClaw/NodeServiceManager.swift:58`, `apps/macos/Sources/OpenClaw/GatewayLaunchAgentManager.swift:147`). Skill binary checks ultimately read `process.env.PATH` (`src/shared/config-eval.ts:152`), so Homebrew-installed binaries like `peekaboo` are visible on current main. This is already present in release `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). If someone still sees a missing-binary case after `v2026.4.22`, that would need a fresh issue with the exact binary path and PATH contents because it would be a narrower follow-up than the original Homebrew-path bug.
