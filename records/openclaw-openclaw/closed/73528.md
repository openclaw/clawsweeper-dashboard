## Summary

Current main already addresses the reported v2026.4.26 startup bottleneck. The released code path awaited primary-model warmup inside `sidecars.channels` and ran `ensureOpenClawModelsJson()` plus `resolveModel()`/`resolveModelAsync()` before starting channels; current main schedules warmup in the background, bounds it, scopes provider discovery to the configured provider, and no longer imports or calls the PI model resolver from the startup warmup path. Existing tests cover both the non-blocking channel startup behavior and the removal of the heavy PI model module load from warmup. The related #60027 escape-hatch PR remains a separate opt-in proposal, but the observable startup-blocking problem reported here is implemented on main.

## Best Possible Solution

Keep the current main behavior: primary-model warmup should remain metadata-only, provider-scoped, bounded, and off the channel-start critical path. Review #60027 independently only if maintainers still want an explicit operator escape hatch in addition to the implemented non-blocking startup path.
