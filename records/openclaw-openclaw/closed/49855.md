## Summary

Keep this issue open. Current main still downloads Feishu/Lark file-typed video resources through the existing `type=file` path with no `type=media` fallback or Feishu 502/503/504 retry, and open PRs #49967 and #50164 are explicit implementation candidates with closing syntax for this same report.

## Best Possible Solution

Keep this issue open and review the focused Feishu plugin fix. The best path is to add a conservative `extensions/feishu` fallback from `type=file` to Feishu's media resource type on vetted 502/503/504 failures, optionally use video-like metadata only where safe, preserve normal file/audio behavior, and add targeted regression tests before closing after a PR lands on main.
