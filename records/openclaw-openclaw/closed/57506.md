## Summary

Current `main` does not hardcode the agent `tts` tool to Edge TTS. The tool delegates to the shared speech runtime, which resolves `messages.tts` provider config, supports legacy `messages.tts.openai` blocks, honors OpenAI `baseUrl`, and lets local `~/.openclaw/settings/tts.json` override provider selection. The follow-up comments match that prefs override behavior.

## Close Comment

Closing this as `cannot_reproduce` based on current `main`.

- Codex review shows the agent `tts` tool is not hardcoded to Edge anymore; it delegates to the shared `textToSpeech()` runtime in `src/agents/tools/tts-tool.ts:72`.
- That shared runtime resolves provider selection from `messages.tts` plus local prefs, and it prefers `~/.openclaw/settings/tts.json` over host config when a prefs provider is set (`extensions/speech-core/src/tts.ts:465`).
- Legacy direct config like `messages.tts.openai` is still read on current `main` (`extensions/speech-core/src/tts.ts:228`), and the OpenAI speech provider still honors `baseUrl` and posts to `${baseUrl}/audio/speech` (`extensions/openai/speech-provider.ts:99`, `extensions/openai/tts.ts:109`).
- The docs explicitly say local TTS prefs override `messages.tts.*` (`docs/tools/tts.md:413`), which matches the follow-up comments here that `~/.openclaw/settings/tts.json` was the actual cause.

If this still reproduces on a current build after clearing or verifying `~/.openclaw/settings/tts.json`, please open a fresh issue with the exact `messages.tts` config, the prefs file contents, and `/tts status` output.
