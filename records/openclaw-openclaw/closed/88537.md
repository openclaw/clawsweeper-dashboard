## Summary

Keep open: current main wires Tokenjuice into OpenClaw embedded and Codex dynamic-tool middleware, but the native Codex `commandExecution` transcript path still stores raw `bash` text without exec-style details, matching the reporter's live evidence.

## What This Changes

The issue asks for Tokenjuice to compact real main-agent Codex `bash` tool results instead of storing and replaying raw output without exec-style details.

## Best Possible Solution

Normalize native Codex `commandExecution` results into the existing exec-style tool-result middleware contract, run the `codex` middleware before persistence, and add regression coverage proving Tokenjuice markers/details survive in the stored transcript.

## Reproduction Assessment

Yes, there is a high-confidence source reproduction path: native Codex `commandExecution` results are recorded as raw `toolResult` text without `details`, while the Tokenjuice adapter requires completed/failed exec-style details. I did not rerun the live `openclaw agent` command during this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is to adapt native Codex bash results into the existing tool-result middleware path rather than adding a separate Tokenjuice-only fallback. A broad fallback inside Tokenjuice would be less clean because the missing data is produced at the Codex transcript boundary.
