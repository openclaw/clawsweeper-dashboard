## Summary

Current main already prevents this false rate-limit symptom: disconnected-node calls short-circuit as `node not connected`, cooldown windows without a recorded failure reason no longer default to `rate_limit`, and reply fallback only shows rate-limit copy for pure rate-limit/overload exhaustion.

## Evidence

- **Disconnected nodes short-circuit before invoke timeout handling:** The gateway returns `UNAVAILABLE` with message `node not connected` and details code `NOT_CONNECTED` when the node is still disconnected after the wake/connect check, instead of proceeding as a normal successful invoke path.
  - file: [src/gateway/server-methods/nodes.ts:981](https://github.com/openclaw/openclaw/blob/11cffb230013c2d8bd1e0e18da04b0b985b8abd5/src/gateway/server-methods/nodes.ts#L981)
  - sha: [11cffb230013](https://github.com/openclaw/openclaw/commit/11cffb230013c2d8bd1e0e18da04b0b985b8abd5)
- **Cooldowns without reason history no longer become fake rate limits:** `resolveProfilesUnavailableReason` explicitly falls back to `unknown` when an active cooldown has no recorded failure counts, with an inline note that the old `rate_limit` default caused false `rate limit reached` warnings.
  - file: [src/agents/auth-profiles/usage.ts:281](https://github.com/openclaw/openclaw/blob/11cffb230013c2d8bd1e0e18da04b0b985b8abd5/src/agents/auth-profiles/usage.ts#L281)
  - sha: [11cffb230013](https://github.com/openclaw/openclaw/commit/11cffb230013c2d8bd1e0e18da04b0b985b8abd5)
- **Regression test covers the cooldown fallback change:** Unit coverage asserts that an active cooldown with no reason history resolves to `unknown`, not `rate_limit`.
  - file: [src/agents/auth-profiles/usage.test.ts:280](https://github.com/openclaw/openclaw/blob/11cffb230013c2d8bd1e0e18da04b0b985b8abd5/src/agents/auth-profiles/usage.test.ts#L280)
  - sha: [11cffb230013](https://github.com/openclaw/openclaw/commit/11cffb230013c2d8bd1e0e18da04b0b985b8abd5)
- **Reply fallback no longer rewrites mixed-cause failures as rate-limit:** The agent-runner only treats `FallbackSummaryError` as rate-limit when every attempt is `rate_limit` or `overloaded`; mixed-cause exhaustion is intentionally kept off the rate-limit path.
  - file: [src/auto-reply/reply/agent-runner-execution.ts:1575](https://github.com/openclaw/openclaw/blob/11cffb230013c2d8bd1e0e18da04b0b985b8abd5/src/auto-reply/reply/agent-runner-execution.ts#L1575)
  - sha: [11cffb230013](https://github.com/openclaw/openclaw/commit/11cffb230013c2d8bd1e0e18da04b0b985b8abd5)
- **Regression test locks the mixed-cause fallback behavior:** Test coverage verifies that mixed-cause fallback exhaustion does not show the rate-limit countdown/copy.
  - file: [src/auto-reply/reply/agent-runner-execution.test.ts:1620](https://github.com/openclaw/openclaw/blob/11cffb230013c2d8bd1e0e18da04b0b985b8abd5/src/auto-reply/reply/agent-runner-execution.test.ts#L1620)
  - sha: [11cffb230013](https://github.com/openclaw/openclaw/commit/11cffb230013c2d8bd1e0e18da04b0b985b8abd5)
- **Shipped changelog entry matches the implemented cooldown fix:** The changelog records the shipped behavior change: cooldown windows with no recorded failure history now default to `unknown` instead of `rate_limit`, specifically to avoid false API rate-limit warnings.
  - file: [CHANGELOG.md:2757](https://github.com/openclaw/openclaw/blob/11cffb230013c2d8bd1e0e18da04b0b985b8abd5/CHANGELOG.md#L2757)
  - sha: [11cffb230013](https://github.com/openclaw/openclaw/commit/11cffb230013c2d8bd1e0e18da04b0b985b8abd5)

- Current main fixes the false rate-limit classification, but it does not appear to implement the exact node-specific user-facing copy proposed in #58903.
