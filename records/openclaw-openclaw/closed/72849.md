## Summary

Keep open: current main still has the missing-ID cron state collision path, and an open linked PR already references this issue with a proposed fix, so the issue should remain open until that PR or a successor lands.

## What This Changes

The issue asks OpenClaw to backfill stable IDs for persisted cron jobs missing both `id` and legacy `jobId`, and to guard cron runtime state updates from undefined job IDs.

## Best Possible Solution

Land a cron fix that canonicalizes missing job IDs before sidecar state merge, rejects invalid outcome IDs, covers polluted sidecar cases, and then lets the linked PR close this issue.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: load multiple persisted cron jobs missing both `id` and `jobId`, then sidecar state merge or timer outcome lookup uses the missing job ID and can collapse state onto the wrong row.

## Solution Assessment

Yes directionally, but the implementation should repair IDs before state merge or ignore invalid sidecar keys. The open linked PR targets the right bug, but review should ensure it covers already-polluted sidecar state and includes real behavior proof.
