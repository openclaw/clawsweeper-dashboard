## Summary

Close as implemented. Current main already has managed Chromium profile recovery for stale `Singleton*` locks from another host or dead PID, retries the launch once after Chromium reports the profile-in-use failure, and the same behavior is present in the latest `v2026.4.29` release tree.

## What This Changes

Request automatic recovery for stale Linux Chrome profile locks after a crash or machine migration so browser startup does not fail with `profile in use` errors.

## Best Possible Solution

Keep the shipped managed-profile recovery path: clear only stale `Singleton*` locks after Chromium reports the profile-in-use failure, preserve live current-host locks, and keep personal existing-session profile cleanup as manual/user-owned recovery.

## Reproduction Assessment

Yes. The current test path reproduces the reported failure class by creating a `SingletonLock` for another host plus `SingletonCookie`/`SingletonSocket`, and the launch test simulates Chromium's profile-in-use stderr before asserting cleanup and retry.

## Solution Assessment

Yes. The current implementation is the narrow maintainable fix because it only removes locks after Chromium identifies the profile as in use and only when the lock cannot represent a live current-host Chrome process; unconditional startup deletion would be riskier.
