## Summary

Close: the runtime-auth missing-alias problem is already handled by the current supported build/postbuild path, and the earlier canonical report was closed with fresh-build proof; this PR adds a fallback for an unsupported no-postbuild state and lacks real behavior proof.

## What This Changes

The PR adds a provider-auth-runtime resolver fallback that scans for hashed runtime-model-auth chunks and adds a unit test for the no-stable-alias case.

## Best Possible Solution

Keep the runtime-postbuild stable alias path as the canonical fix and reopen only if a supported `pnpm build` or packaged install again fails to emit or import the alias.

## Reproduction Assessment

No high-confidence current-main reproduction is present for the supported build path. Source shows the old resolver would fail if the alias is absent, but current main’s supported postbuild path is designed and tested to create that alias.

## Solution Assessment

No. Scanning hashed chunks in the resolver is weaker than the existing build/postbuild contract because it bypasses the wrapper-selection logic and only helps an unsupported no-postbuild state.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.83

Full review comments:

- **[P2] Preserve stable alias selection for colliding chunks:** `src/plugin-sdk/provider-auth-runtime.ts:212-219`
  - body: When the stable alias is absent, this fallback imports the first `runtime-model-auth.runtime-*` filename returned by `readdirSync()`. Current postbuild logic deliberately resolves colliding runtime chunks by selecting the wrapper chunk, so this can import the wrong hashed chunk depending on directory order; reuse the alias-selection logic instead of adding a first-match scan.
  - confidence: 0.83
