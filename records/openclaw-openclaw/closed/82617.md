## Summary

Keep open: the reported WebChat auto-TTS gap is source-plausible, but this PR is not mergeable because it changes final-mode TTS to run on streaming blocks, weakens the internal trusted-media boundary, and contains encoding corruption.

## What This Changes

The PR changes speech-core auto-TTS payloads and generic reply dispatch normalization so WebChat can receive trusted local TTS audio, and changes the plugin SDK reply payload type.

## Best Possible Solution

Keep trustedLocalMedia internal, set it only on OpenClaw-generated local auto-TTS audio, carry it through reconstructed TTS-only final payloads, and cover WebChat plus dispatch tests without enabling per-block final-mode synthesis.

## Reproduction Assessment

No live high-confidence reproduction was run. Source inspection shows auto-TTS local audio can miss the trustedLocalMedia flag while WebChat requires it, but the exact WebChat after-fix flow still needs real proof.

## Solution Assessment

No. The narrow maintainable fix is to preserve the internal trust flag through the existing final TTS-only path, not to expose the flag through the public SDK or synthesize TTS for every streaming block.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Keep trusted media from bypassing normalization:** `src/auto-reply/reply/dispatch-from-config.ts:595-596`
  - body: This early return trusts any payload carrying `trustedLocalMedia`, and the branch also exposes that flag through the public plugin SDK. That lets plugin or hook payloads skip the normal media staging/read-policy path, even though current main marks this trust bit as internal-only.
  - confidence: 0.9
- **[P2] Do not synthesize final-mode TTS for each block:** `extensions/speech-core/src/tts.ts:1762`
  - body: Allowing `kind: "block"` through final-mode TTS makes the block callback synthesize audio for streamed chunks, while dispatch already synthesizes one final TTS-only payload after block streaming completes. This can send partial or duplicate audio and extra provider calls.
  - confidence: 0.9
- **[P2] Carry trust through the final TTS-only payload:** `src/auto-reply/reply/dispatch-from-config.ts:1660-1664`
  - body: Adding the flag to `maybeApplyTtsToPayload` is not enough because the block-stream fallback rebuilds `ttsOnlyPayload` from `mediaUrl` and omits `trustedLocalMedia`. WebChat can still drop that local audio path unless the reconstructed final payload carries the internal trust signal.
  - confidence: 0.88
- **[P2] Restore the corrupted TTS prompt text:** `extensions/speech-core/src/tts.ts:559`
  - body: The branch changes the prompt hint to `鈮?{maxLength}`, which both corrupts the `≤` character and drops the `${maxLength}` interpolation. TTS instructions would no longer tell the model the actual configured spoken-text limit.
  - confidence: 0.95
- **[P2] Restore UTF-8 text for the SDK source:** `src/plugin-sdk/reply-payload.ts:1`
  - body: `src/plugin-sdk/reply-payload.ts` is encoded as a binary/UTF-16-like file in the PR head. Keep TypeScript sources as normal UTF-8 text so diffs, linting, and runtime loaders can process the public SDK file reliably.
  - confidence: 0.93

## Security Review

Status: needs_attention
