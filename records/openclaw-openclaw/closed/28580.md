## Close Comment

Thanks for the contribution work here. After a Codex review of current `main`, this looks better scoped as a community plugin rather than something to merge into OpenClaw core. Current project policy says optional capability should usually ship as plugins, that plugin authors should host and maintain them in their own repo, and that the bar for adding optional plugins to core is intentionally high (`VISION.md:54`, `VISION.md:57`, `VISION.md:59`). The plugin docs also explicitly say you do not need to add a plugin to the OpenClaw repo and should publish to ClawHub or npm instead (`docs/plugins/building-plugins.md:16`), with ClawHub documented as the canonical discovery surface for community plugins (`docs/plugins/community.md:9`, `docs/plugins/community.md:14`).

There is also already QQ support on current `main` via the bundled `qqbot` channel (`docs/channels/qqbot.md:10`, `README.md:26`), and `VISION.md:104` specifically calls out wrapper channels around already supported channels as out of merge scope unless there is a clear capability or security gap. On that basis, I’m closing this as a community-plugin/ClawHub path rather than a core-repo item. If you publish and maintain the NapCat plugin externally, that fits the current OpenClaw plugin model.
