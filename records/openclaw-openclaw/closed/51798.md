## Summary

Current main already fixes the reported false-negative by reading Ollama `/api/show` capability metadata, marking `vision` models as `input: ["text", "image"]`, and routing Ollama as an image-capable media-understanding provider. That implementation is also present in the shipped release `v2026.4.22`.

## Evidence

- **Image error comes from model metadata gate:** The current image runtime still throws `Model does not support images` only when the resolved registry entry lacks `input: "image"`, so this report reduces to model capability metadata, not the Ollama request path.
  - file: [src/media-understanding/image.ts:116](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/src/media-understanding/image.ts#L116)

  - file: [extensions/ollama/src/provider-models.ts:98](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/extensions/ollama/src/provider-models.ts#L98)
