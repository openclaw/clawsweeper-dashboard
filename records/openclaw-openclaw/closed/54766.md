## Summary

Current `main` already converts `openai-completions` image inputs to base64 `data:` URLs instead of gateway-local HTTP image URLs, so the requested force-base64 behavior is already implemented by default.

## Evidence

- **OpenAI-completions explicitly uses image_url parts backed by data URIs:** `usesOpenAICompletionsImageParts()` returns true for `api === "openai-completions"`, and `toImageUrlFromBase64()` builds `data:${mediaType};base64,...` URLs.
  - file: [src/agents/openai-ws-message-conversion.ts:51](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/src/agents/openai-ws-message-conversion.ts#L51)
  - command: `nl -ba src/agents/openai-ws-message-conversion.ts | sed -n '1,95p'`
- **Both raw image blocks and input_image sources are serialized as base64 data URLs for openai-completions:** `contentToOpenAIParts()` maps `type: "image"` and `type: "input_image"` to `type: "image_url"` with `url: data:...` when the model API is `openai-completions`.
  - file: [src/agents/openai-ws-message-conversion.ts:88](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/src/agents/openai-ws-message-conversion.ts#L88)
  - command: `nl -ba src/agents/openai-ws-message-conversion.ts | sed -n '90,170p'`
- **Current transport stream also inlines image data as data URLs:** `buildOpenAIResponsesInput()` in the transport path emits `image_url: \`data:${item.mimeType};base64,${item.data}\`` for user images and tool-result images, with no local `http://127.0.0.1:18789/media/inbound/...` conversion in this path.
  - file: [src/agents/openai-transport-stream.ts:264](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/src/agents/openai-transport-stream.ts#L264)
  - command: `nl -ba src/agents/openai-transport-stream.ts | sed -n '220,380p'`

## Risks / Open Questions

- I did not find a dedicated regression test in `src/agents/openai-transport-stream.test.ts` that asserts this exact image serialization path, so future regressions could be possible without additional test coverage.
- The latest tagged release provided in the task is `v2026.4.22`, while the inspected `main` SHA is newer; this appears implemented on `main`, but I could not tie it to a shipped release tag.
