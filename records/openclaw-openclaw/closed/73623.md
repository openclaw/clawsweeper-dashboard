Close as superseded by #73554. Current main still needs the missing-plugin validation fix, but #73554 already tracks the same remaining work for #73551 and this PR’s patch has a blocking canonical-id bug when the validation matches a display name.

## What This Changes

This PR adds registry lookup validation and tests to `plugins enable` and `plugins disable` so nonexistent plugin IDs fail before config is written.

## Best Possible Solution

Use #73554 as the canonical implementation, land exact discovered plugin-id validation before enable/disable config writes, keep cleanup of already-stale entries in a separate doctor/fix path, and close duplicate implementation branches after the chosen patch merges.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Pass the resolved plugin ID to config writes:** `src/cli/plugins-cli.ts:510`
  - body: The new guard accepts display-name matches via `entry.name === id`, but both handlers still call the config mutation helpers with the original argument. A display-name invocation passes validation and writes `plugins.entries["My Plugin"]` instead of the canonical plugin id, leaving the same stale-config class this PR is trying to prevent. Resolve to `targetPlugin.id` everywhere downstream or keep validation id-only.
  - confidence: 0.94
