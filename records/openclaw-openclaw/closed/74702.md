## Summary

Keep open. Current main still does not export `openclaw/plugin-sdk/zalouser`, while `openclaw@2026.4.22` shipped that subpath, `openclaw@2026.4.27` removed it, and both published Lark plugin versions named by the reporter still import it at load time.

## What This Changes

The issue asks OpenClaw to restore or provide a migration path for the missing `openclaw/plugin-sdk/zalouser` subpath so `@larksuite/openclaw-lark` can load after OpenClaw 2026.4.27.

## Best Possible Solution

Restore the shipped compatibility surface as a deprecated, narrow `openclaw/plugin-sdk/zalouser` facade that forwards the existing generic helpers needed by published packages, then align SDK exports, API baseline, docs, tests, and changelog so new plugins use generic subpaths.

## Repair Work Prompt

Fix the `openclaw/plugin-sdk/zalouser` compatibility regression reported for `@larksuite/openclaw-lark` after OpenClaw 2026.4.27.

Observable bug: `@larksuite/openclaw-lark@2026.4.7` and `2026.4.8` import `openclaw/plugin-sdk/zalouser`; `openclaw@2026.4.22` exported that subpath, but `openclaw@2026.4.27` and current main do not, so plugin loading fails with `Cannot find module .../plugin-sdk/root-alias.cjs/zalouser`.

Expected fix boundary: restore a deprecated, compatibility-only `openclaw/plugin-sdk/zalouser` subpath or an equivalent host-side compatibility path. Do not change unrelated plugin loader behavior or external package metadata.
