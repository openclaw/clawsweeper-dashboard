## Summary

Current main and the latest release now handle the reported Discord READY startup hang: the Discord lifecycle waits for gateway readiness, restarts and retries when READY does not arrive, waits for async gateway plugin registration before supervision, and avoids the normal `/users/@me` startup fetch when the bot id can be parsed from the token.

## What This Changes

The issue reports a Discord gateway regression where the bot appears online but never reaches READY, so guild channel inbound messages are not received after upgrading to OpenClaw 2026.5.7.

## Best Possible Solution

Keep the shipped v2026.5.28+ Discord gateway lifecycle behavior; any remaining READY hang on v2026.5.28 or newer should be reported as a fresh issue with new logs and environment details.

## Reproduction Assessment

No current-main failure was reproduced in this read-only review. The historical 2026.5.7 and 2026.5.12 reports are concrete, and current source plus shipped regression tests now cover the READY-timeout and reconnect path.

## Solution Assessment

Yes. The shipped solution addresses the maintainable owner boundary: Discord gateway startup now waits for plugin registration, detects missing READY, reconnects with backoff, avoids the normal startup identity fetch timeout path, and preserves socket-generation safety for IDENTIFY.
