- `impact:session-state`: The issue is about reasoning content being stripped from replayed conversation history, which changes later-turn context state.

## Summary

Keep open: current main still drops replayed reasoning history for unowned custom `openai-completions` models unless the model id is whitelisted, and the provider config schema has no accepted override for that policy.

## What This Changes

The issue requests a supported config path to prevent custom `openai-completions` providers from dropping replayed reasoning content.

## Best Possible Solution

Add one canonical, documented provider/model replay-policy override or extend an existing model-compat surface so custom OpenAI-compatible providers can opt into preserving reasoning history without changing the safe default for every backend.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction: a custom non-whitelisted `openai-completions` model resolves `dropReasoningFromHistory: true`, while strict provider schemas reject the proposed override key. I did not run a live llama.cpp backend in this read-only review.

## Solution Assessment

Unclear whether the proposed provider-level, direct-key, or per-model shape is the best permanent API. The maintainable fix should choose one canonical config surface, document it, and cover both custom-provider preservation and existing default stripping behavior in tests.
