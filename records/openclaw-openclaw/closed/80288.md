## Summary

Close: the same Codex HOME isolation and real-home escape-hatch cluster was superseded by the later not-planned decision in https://github.com/openclaw/openclaw/issues/78202 and https://github.com/openclaw/openclaw/pull/78213, while current main still intentionally reserves `HOME` and `CODEX_HOME` for per-agent isolation.

## What This Changes

Request an explicit Codex app-server `homeDir` or env override so spawned Codex keeps isolated `CODEX_HOME` while using the operator's real `HOME` for local CLI auth.

## Best Possible Solution

Keep the current isolated `HOME`/`CODEX_HOME` contract and route any future trusted-local compatibility mode through a new product/security proposal with tests and docs.

## Reproduction Assessment

Yes. Source and tests on current main show stdio Codex app-server launches set `HOME` to the per-agent native home unless an internal start-option env override is already present; I did not run a live `gh auth status` reproduction.

## Solution Assessment

Yes, closing as superseded is the best cleanup path. Implementing the requested `homeDir` config would reopen the same isolation and real-home product/security boundary that the related HOME/REAL_HOME cluster rejected.
