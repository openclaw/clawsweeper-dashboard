## Summary

Close as implemented on current main. The reported bottleneck is addressed by the current plugin-owned memory stack: structured LanceDB memory tools, prompt-time active recall, opt-in session transcript search, dreaming-based consolidation, and workflow/session persistence hooks are present in source, docs, and tests; the proof commit is on main but not yet in a release tag.

## What This Changes

Request a structured, context-activated cross-conversation memory architecture with store/query APIs, startup memory budgeting, recall activation, consolidation, optional cross-agent sharing, and workflow state persistence.

## Best Possible Solution

Keep the current plugin-owned memory architecture: use `memory-lancedb` or `memory-core` for active memory, `active-memory` for prompt-time recall, dreaming for consolidation, and plugin host hooks for workflow state. Any remaining multi-slot role-composition design should stay in the narrower #60572 architecture thread.

## Reproduction Assessment

Not applicable as a feature request rather than a runtime bug. The high-confidence verification path is source/docs/test inspection on current main showing the requested store, recall, consolidation, transcript search, and workflow persistence surfaces exist.

## Solution Assessment

Yes. The current implementation is the narrower maintainable direction because it solves the user-facing memory bottleneck through memory plugins and SDK seams instead of adding a new core-owned memory architecture; separate multi-slot composition remains a distinct architecture RFC.
