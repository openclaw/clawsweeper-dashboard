## Summary

Keep open. The revised PR now uses the safer character-loop validator and no longer has the earlier pre-guard decode concern, but it still needs real gateway/webchat proof and focused regression coverage before merge, and it overlaps with the open fix cluster at https://github.com/openclaw/openclaw/issues/80679 and https://github.com/openclaw/openclaw/pull/80708.

## What This Changes

The PR changes `src/gateway/chat-attachments.ts` so `isValidBase64` validates base64 characters and padding with a loop instead of a full-string regex.

## Best Possible Solution

Land one stack-safe gateway attachment validator that preserves estimate-before-decode limits, includes focused large-payload regression coverage, and attaches redacted real webchat or gateway proof; choose between this PR and https://github.com/openclaw/openclaw/pull/80708 rather than merging both.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No live current-main reproduction was run in this read-only review, but the linked v2026.5.7 logs and current source give a clear source path through `chat.send` -> `parseMessageWithAttachments` -> `isValidBase64` on a multi-MB base64 image.

## Solution Assessment

Is this the best way to solve the issue? The revised character-loop direction is the safer narrow fix because it removes the stack-prone regex without decoding before size guards. It still needs real gateway/webchat proof and focused regression tests before merge.
