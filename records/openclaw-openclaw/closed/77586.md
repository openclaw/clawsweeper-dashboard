## Summary

Current main already handles the reported path: disabled-tool embedded runs ignore inherited config allowlists, and the same behavior is covered by a regression test and shipped in v2026.5.3.

## What This Changes

The issue asks for commitment extraction runs with `disableTools: true` to ignore inherited `tools.alsoAllow` entries instead of failing on an empty callable tool set.

## Best Possible Solution

Keep the shipped disabled-tools allowlist guard and have affected 2026.5.2 users upgrade to v2026.5.3 or later.

## Reproduction Assessment

No for current main: source inspection shows inherited config allowlists are ignored when `disableTools` is true, and the regression test covers the previously failing shape. The original 2026.5.2 report is source-plausible but no longer reproduces on current code.

## Solution Assessment

Yes. The implemented fix is narrow: it skips inherited config allowlists only for disabled-tool runs while preserving fail-closed behavior for explicit runtime `toolsAllow`.
