## Summary

Keep open: current main still has a source-proven WebChat long-history path where the browser requests only the latest 100 messages, the Gateway returns that bounded tail without a truncation signal, and the renderer cannot show its hidden-history notice for server-side truncation.

## What This Changes

The issue requests WebChat to render or clearly account for assistant messages preserved in transcript history so the UI does not appear to lose replies compared with TUI.

## Best Possible Solution

Define an explicit WebChat long-history contract, preferably by adding additive chat.history truncation metadata or pagination and wiring Control UI so bounded history never looks like silent assistant-message loss.

## Reproduction Assessment

Yes, at source level: Control UI requests 100 messages, the Gateway returns only that bounded tail, and the renderer only warns about messages hidden inside the payload it received. The exact Safari/DeepSeek transcript was not live-reproduced because the raw transcript was not provided.

## Solution Assessment

Unclear. A renderer-only truncation notice is a useful sketch, but maintainers still need to choose between limit-plus-one probing, additive hasMore/cursor metadata, or full load-older pagination; https://github.com/openclaw/openclaw/pull/72178 is closest but remains open with review concerns.
