## Summary

Keep open: current main still accepts `threadName` in the message-tool surface but the Discord `send` bridge and runtime send path do not read, forward, rename, or warn on it. The existing Discord edit-channel path can rename channels/threads, and related search did not find an exact canonical fix PR.

## What This Changes

The issue asks for Discord message sends with `threadName` targeting an existing thread to either rename that thread or surface a warning instead of silently ignoring the parameter.

## Best Possible Solution

Fix the Discord-owned send pipeline so `threadName` on a send to an existing Discord thread is honored through the existing rename helper or returned as an explicit warning, with focused regression coverage.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction: call `message` with `action:"send"`, `channel:"discord"`, an existing thread target, and `threadName`; current main sends the message without reading or forwarding `threadName`. I did not run a live Discord rename because this review is read-only.

## Solution Assessment

Yes, the best fix is plugin-local: consume `threadName` in the Discord send path for resolved thread targets using the existing edit-channel primitive, or return a structured warning when it cannot be applied. Core should remain plugin-agnostic.
