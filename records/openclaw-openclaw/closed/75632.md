## Summary

Current main already addresses the reported startup stall shape after v2026.4.29: the gateway no longer lets default/configured model and session views wait indefinitely on catalog discovery, chat history reads are bounded to the requested recent window, TUI startup history retries are covered, and session-store maintenance no longer runs on normal reads. The proof is main-only so far, not in the latest release tag.

## What This Changes

Report requests fixing a WSL2 TUI/Gateway startup regression in 2026.4.27-2026.4.29 where first connect and `chat.history`/`models.list`/`sessions.list` could stall or time out compared with 2026.4.23.

## Best Possible Solution

Keep the current main implementation and let the next release carry it; if startup still stalls after `e6c1a6637a`, collect `OPENCLAW_GATEWAY_STARTUP_TRACE=1`, diagnostics event-loop output, and fresh TUI/Gateway RPC timings against that build.

## Reproduction Assessment

Yes for the affected releases: the report gives repeatable WSL2 `npm install openclaw@2026.4.29` / `--dev` steps plus concrete slow `chat.history`, `sessions.list`, and `models.list` timings. I did not run a live WSL2 reproduction in this read-only review; current-main verification is source and regression-test based.

## Solution Assessment

Yes. The main-line fix set is the maintainable path: bound expensive reads/discovery, yield during large session listing, retry startup-only history unavailability in the TUI, and keep plugin runtime-deps behavior out of hot request paths rather than masking the problem with longer client timeouts or WSL2 special cases.
