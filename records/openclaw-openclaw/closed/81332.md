## Summary

Current main already fixes the reported timeout mismatch: the Codex dynamic-tool bridge now derives `image_generate` watchdog timeouts from the per-call argument, then `agents.defaults.imageGenerationModel.timeoutMs`, before falling back to the 30000ms default, with the documented 600000ms cap. This appears to be current-main-only rather than shipped in `v2026.5.7`, so users on that release can still see the reported behavior until a newer release is installed.

## What This Changes

The issue reports that Codex dynamic-tool execution times out `image_generate` after 30000ms instead of honoring per-call or configured image-generation timeout budgets.

## Best Possible Solution

Keep the current-main bridge behavior and let the next release carry it; reopen only if a build after this proof commit still times out `image_generate` at 30000ms despite a larger per-call or configured timeout.

## Reproduction Assessment

No current-main failure remains from source inspection: the timeout resolver now returns the per-call value or configured image-generation timeout before the 30000ms default. The reported release symptom is credible, but it matches behavior that current main has already changed.

## Solution Assessment

Yes. The current implementation fixes the narrow contract mismatch in the bridge without changing provider behavior or adding a new config surface; background delivery for long image jobs would be a separate product enhancement.
