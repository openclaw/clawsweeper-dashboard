## Summary

Keep open: the report is fresh, concrete, and current source still supports a failure path where configured `openai/...` image understanding resolves through the direct OpenAI provider and requires OpenAI API-key auth even though separate Codex media-understanding providers exist. The remaining question is product/API contract, not stale cleanup: maintainers need to decide whether `openai/*` vision should fall back to Codex OAuth like image generation, or whether Codex-only installs must select the explicit Codex media route.

## What This Changes

The issue requests that image/media understanding use Codex OAuth routing instead of failing with `OPENAI_API_KEY required` in Codex-OAuth-only setups.

## Best Possible Solution

Codify the intended media-understanding route and make code, tests, and docs match: either support Codex OAuth fallback for `openai/*` image understanding, or improve diagnostics/docs so Codex-only users select the explicit `codex/*` or `openai-codex/*` route.

## Reproduction Assessment

No high-confidence live reproduction yet; current source supports the failure when image understanding is configured for `openai/...` and only Codex OAuth is available. The missing piece is the reporter's exact media/image model config and a live Codex-OAuth run with no `OPENAI_API_KEY`.

## Solution Assessment

Unclear; routing through the Codex media provider may be the right fix, but current docs and provider ids distinguish direct OpenAI, `openai-codex`, and bounded `codex` routes. Maintainers should choose that contract before an automated fix changes fallback behavior.
