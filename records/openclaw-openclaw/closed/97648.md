## Summary

Keep this PR open for maintainer review: the patch is a focused Mistral provider hardening change with good source/dependency evidence and sufficient loopback proof, but the new 16 MiB fail-closed response-body boundary is compatibility-sensitive and needs maintainer acceptance before merge.

## What This Changes

The PR adds a bounded Mistral SDK fetcher with a 16 MiB response-body cap, injects it through the SDK HTTPClient, and adds loopback plus synthetic bounded-stream tests.

## Best Possible Solution

Land the bounded fetcher after maintainers accept or tune the 16 MiB Mistral boundary, keeping the guard core-internal and covered by the focused loopback and synthetic tests.

## Reproduction Assessment

Yes. Source inspection shows current main constructs the Mistral SDK without a bounded fetcher while the SDK stream parser reads and buffers `responseBody.getReader()` chunks until SSE boundaries; the PR body provides after-fix loopback output for the capped path.

## Solution Assessment

Yes, with a compatibility caveat. Injecting the SDK-supported HTTPClient fetcher is the narrowest maintainable interception point for Mistral SDK responses; a config knob or broader SDK surface would be extra product surface unless maintainers reject the fixed threshold.
