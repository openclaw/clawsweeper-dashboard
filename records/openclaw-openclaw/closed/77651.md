## Summary

Keep open: current main still has the reported generic gateway channel-lifecycle path, and the related issues cover adjacent event-loop starvation rather than this manual-stop reconnect poison state.

## What This Changes

Reports that Slack Socket Mode can remain permanently dead after a stop timeout leaves the account manually stopped and marked as still running.

## Best Possible Solution

Add a narrow lifecycle distinction between explicit user stops and recovery stops so timeout recovery clears manual suppression or schedules reconnect without launching duplicate provider tasks.

## Reproduction Assessment

Yes, from source inspection: an abort-ignoring channel task can drive `stopChannel()` into the 5000ms timeout branch, leaving `manuallyStopped` and `store.tasks` in the state that suppresses restart. I did not run a live Slack socket scenario in this read-only review.

## Solution Assessment

No, the current timeout behavior is not the best way to solve the issue. The safer repair is to scope manual-stop persistence to user-initiated stops and let health-monitor recovery recover after timeout while keeping duplicate task protection.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/77651. Current main leaves a channel/account in `manuallyStopped` and `running: true` when `stopChannel()` times out waiting for an abort-ignoring task; the health monitor then skips the account and `startChannel()` cannot recover while the old task remains tracked. Implement a narrow generic gateway lifecycle fix in `src/gateway/server-channels.ts` and, if needed, `src/gateway/channel-health-monitor.ts`: distinguish explicit user-initiated manual stops from health-monitor/recovery stops, or otherwise clear manual-stop suppression on timeout recovery, while preserving the duplicate-task guard for the still-running old task. Add focused regression coverage in `src/gateway/server-channels.test.ts` and `src/gateway/channel-health-monitor.test.ts` for an abort-ignoring task that times out during recovery and does not remain permanently ghost-running/manually stopped. Update `CHANGELOG.md` under the active Fixes section because this is user-facing. Do not add a new config option, do not add Slack-specific core IDs, and do not weaken the existing no-duplicate-task behavior.
