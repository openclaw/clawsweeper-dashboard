## Summary

Close as implemented on current main: the linked Telegram status failure is already addressed by a config-only accessor path, while the PR branch would relax strict runtime SecretRef behavior that current tests intentionally preserve.

## What This Changes

The PR removes Telegram's strict non-env SecretRef resolution call from `resolveRuntimeTokenValue`, causing unresolved file/exec-style bot token refs to return `configured_unavailable` instead of throwing during runtime token resolution.

## Best Possible Solution

Keep the current-main split: read-only Telegram metadata should use the config accessor/inspection path, and runtime Telegram token resolution should remain strict for unresolved non-env SecretRefs.

## Reproduction Assessment

Yes. The PR regression is reproducible by the existing `extensions/telegram/src/token.test.ts` strict SecretRef case, and the linked status shape is covered by the current `extensions/telegram/src/shared.test.ts` read-only accessor test.

## Solution Assessment

No. The PR changes the runtime credential resolver to solve a read-only inspection symptom; current main already has the narrower maintainable fix in the config-accessor path.
