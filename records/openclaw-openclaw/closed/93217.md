## Summary

Current main and the shipped v2026.6.6 release already cover the reported mixed-version update chain: systemd service activation reloads and retries missing-unit failures, package updates restart through the updated install when refresh fails, and restart health fails the update when the old Gateway remains active.

## What This Changes

The issue asks OpenClaw update to avoid leaving a stale, RPC-broken Gateway process holding the port after CLI/config/systemd surfaces move to the target package version.

## Best Possible Solution

Keep the shipped v2026.6.6 update hardening; treat any post-v2026.6.6 Linux systemd reproduction as a fresh, narrower report with current diagnostics.

## Reproduction Assessment

## Solution Assessment

Yes. The maintainable solution is the current shipped path: reload/retry systemd activation, restart through the updated install when refresh fails, verify the target Gateway version, and fail the update instead of reporting success when the old runtime remains.
