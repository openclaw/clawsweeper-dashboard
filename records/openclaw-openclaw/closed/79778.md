## Summary

Keep open: current main still has the WhatsApp audio-preflight echo routing bug, and an open closing PR already targets the narrow fix path, so this issue should close only after that PR or an equivalent patch lands.

## What This Changes

The issue requests restoring WhatsApp voice-note transcript echoes when `tools.media.audio.echoTranscript` is enabled after the WhatsApp plugin externalization.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/79788 or an equivalent WhatsApp-owner patch that supplies routing context at both preflight call sites and keeps focused regression coverage.

## Reproduction Assessment

Yes, by source inspection: both WhatsApp preflight paths can transcribe audio while passing no channel/target metadata, and the shared echo sender returns early without those fields. I did not run a live WhatsApp account reproduction in this read-only review.

## Solution Assessment

Yes. Passing the existing WhatsApp routing fields into the existing `transcribeFirstAudio`/`sendTranscriptEcho` path is the narrow maintainable fix; plugin-local echo config or moving echo emission back into core would be broader than needed.
