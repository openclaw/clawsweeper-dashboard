## Summary

Keep open. The performance target is reasonable and distinct from the related discovery-cache work, but the patch has a cwd-sensitive trust-cache bug: it memoizes by raw `modulePath` even though current logic resolves relative paths against `process.cwd()`, so identical relative paths from different plugin roots can reuse the wrong Codex trust result. The `maintainer` label also means this PR needs normal maintainer handling rather than cleanup closure.

## What This Changes

Memoizes `isOfficialInstalledCodexPluginModulePath` results by `modulePath` to avoid repeated upward `package.json` walks during plugin loader trust checks.

## Best Possible Solution

Keep the perf cache, but key it by the resolved module path or another cwd-aware key and add a regression test covering the same relative module path under trusted and untrusted cwd contexts.

## Reproduction Assessment

No high-confidence executed repro was run in this read-only review. Source inspection shows a clear focused repro path: call the alias map with the same relative module path while switching cwd between an installed Codex package and a non-Codex package.

## Solution Assessment

No. The memoization layer is a reasonable narrow perf fix, but the key must match the normalized/cwd-aware path semantics before this is safe to merge.
