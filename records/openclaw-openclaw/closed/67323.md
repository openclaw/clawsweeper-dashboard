## Summary

Keep open. Current main has useful generic inbound and followup dedupe, plus the newer #71761 runtime-context fix, but the Teams DM path still enqueues the inbound user text as a system event before dispatch-level dedupe runs. Since system events are later prepended to prompts, this leaves a concrete duplicate-context path for the exact MSTeams symptom, and the maintainer discussion already says to keep this issue as a concrete repro under umbrella #69208 unless proven strictly duplicate or fixed.

## Best Possible Solution

Keep #67323 open. The likely fix is to remove or guard the MSTeams inbound-preview `enqueueSystemEvent` path, especially for DMs, or move any needed logging to non-prompt logs. Dispatch-level inbound dedupe should happen before prompt-affecting side effects, and a focused regression test should cover a Teams DM arriving while the session is busy and assert that the activity id/text appears only once in the assembled prompt/transcript. Continue linking the broader transcript/replay work to #69208.
