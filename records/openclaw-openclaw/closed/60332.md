## Summary

Close as a duplicate of #49380. Current main still has the same underlying accounting gap, and #49380 is open, older, and more complete as the canonical tracker for plugin context engines not being able to increment the status compaction counter.

## What This Changes

Request that `/status` and `session_status` include compactions performed by third-party `ContextEngine` plugins with `ownsCompaction: true`, including proactive or background compactions.

## Best Possible Solution

Keep the remaining implementation discussion in #49380 and add one additive, plugin-facing accounting mechanism there: a callback, return value, or per-session stats API that can persist successful engine-owned compaction deltas into the existing session store for both `/status` and `session_status`.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: an `ownsCompaction` engine can compact inside `afterTurn`, but that method returns `Promise<void>` and the status surfaces only read persisted `SessionEntry.compactionCount`.

## Solution Assessment

Yes for cleanup: closing this as a duplicate is the best repository action because #49380 already tracks the same remaining work with fuller root-cause detail. The code fix itself should use an explicit successful-compaction signal or idempotent stats delta, not count every `afterTurn` invocation.
