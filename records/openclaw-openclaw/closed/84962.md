## What This Changes

Adds a Railway config with one Control UI origin and `trustedProxies`, copies `deploy/` into the Docker runtime image, changes the Docker workspace-deps copy step, and updates Railway docs to set `OPENCLAW_CONFIG_PATH`.

## Best Possible Solution

Use the existing config keys, but seed or document a per-deployment `/data` config whose `allowedOrigins` matches the actual Railway domain and whose `trustedProxies` is minimal, proven, or operator-supplied, then prove the Railway `/openclaw` path in a real deployment.

No. A hardcoded bundled config is not the best fix; the maintainable path is per-deployment persistent config plus a proven minimal proxy trust boundary.
