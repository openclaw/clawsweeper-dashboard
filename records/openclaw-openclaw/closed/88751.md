## Summary

This should stay open: current main already detects and repairs the disabled Codex plugin route mismatch in the doctor repair/preview helper, but `openclaw doctor --lint` only runs registered health checks and does not register that Codex route check, so the issue's lint-specific failure remains valid.

## What This Changes

The issue requests that `openclaw doctor --lint` flag configurations where an OpenAI model forces the native `codex` agent runtime while the `codex` plugin is disabled or unavailable.

## Best Possible Solution

Register or adapt the existing Codex route warning as a structured doctor lint health check, keeping repair in `doctor --fix` and preserving the explicit-runtime fail-closed behavior.

## Reproduction Assessment

Yes from source inspection: lint runs registered health checks, the registered core list omits the Codex route check, and the existing Codex route helper would warn for the reported disabled-plugin configuration. I did not run a live lint command because the review had to keep the checkout read-only.

## Solution Assessment

Yes, the best fix is to wire the existing Codex route detector into doctor lint/health output rather than adding another runtime fallback or new config option; `doctor --fix` already owns the repair path.
