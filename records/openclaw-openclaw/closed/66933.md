## Summary

Keep open: current main does not implement global/per-agent sender tool tiers, and the PR still has source-reproducible blockers in a security-sensitive tool-capability boundary.

## What This Changes

The PR adds global and per-agent `toolsBySender` config, channel-scoped sender keys, docs/schema updates, and sender-policy filtering for core, plugin, and bundled tools.

## Best Possible Solution

Keep the PR open, preserve legacy sender-id matching for malformed `channel:` keys, thread canonical channel context through plugin group policy resolvers, add focused regression tests for all filtering paths, then do maintainer security/product review before merge.

## Reproduction Assessment

Yes for the review findings via source inspection. Current main falls back to legacy `id` matching, while PR head classifies `channel:alice` as typed and drops it, and plugin group resolvers still lack provider context for channel-bucket matches.

## Solution Assessment

No. The direction is plausible, but the implementation should first preserve compatibility and carry canonical channel identity through plugin-owned group policy contexts so the advertised sender tiers apply consistently.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve legacy channel-prefixed sender IDs:** `src/config/types.tools.ts:230`
  - body: Adding `channel` to the typed prefixes makes an existing legacy key like `channel:alice` enter channel parsing. Because the value has no inner `channelId:senderId` separator, parsing returns no key before the legacy id fallback, so that sender-specific allow/deny entry silently stops applying after upgrade.
  - confidence: 0.9

  - body: This only passes `messageProvider: channel` to the generic config resolver. Plugin-owned group resolvers are still called above without provider context, and `ChannelGroupContext` has no provider field, so `channel:<provider>:<senderId>` rules for Slack/Discord/MSTeams group `toolsBySender` entries cannot hit the new channel bucket and fall through to less-specific policy.
  - confidence: 0.86
