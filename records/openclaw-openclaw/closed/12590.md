## Summary

Keep open. Current main and the latest release still contain the reported counter-synchronization path, and open PR https://github.com/openclaw/openclaw/pull/51421 targets the same issue with closing syntax.

## What This Changes

The report asks for memoryFlush to fire on every eligible auto-compaction cycle instead of being skipped after flush bookkeeping synchronizes with compactionCount.

## Best Possible Solution

Land PR https://github.com/openclaw/openclaw/pull/51421 or an equivalent focused patch that preserves the pre-increment flush marker after flush-triggered compaction and adds consecutive-cycle regression coverage.

## Reproduction Assessment

Yes, at source level. Current main can persist both counters to the same post-increment value after a flush-triggered compaction, and the next shouldRunMemoryFlush call skips because the counters match.

## Solution Assessment

Yes. The narrowest maintainable fix is to stop recording the post-increment compaction count as the flush marker, then update the runner/gate expectations and consecutive-cycle coverage.
