## Summary

Current main already solves the central Discord/Slack user problem: final assistant MEDIA directives are parsed into mediaUrls, local paths are staged through the outbound attachment pipeline, and the Discord/Slack adapters upload those media entries instead of sending raw path text.

## What This Changes

The issue asks Discord and Slack to turn local-path MEDIA directives from agent replies into viewable attachments or URLs instead of plain filesystem text.

## Best Possible Solution

Keep the current shared structured-media pipeline; if a future raw-path leak appears from a non-final or streaming/message-tool path, track it as a narrower repro against that source path.

## Reproduction Assessment

No. I did not run a live Discord/Slack session in this read-only review, and current source plus tests show the covered final-reply path strips MEDIA text and routes mediaUrls into attachment upload paths.

## Solution Assessment

Yes. The current implementation solves this at the shared final-reply/media staging boundary and then lets Discord/Slack use their normal upload paths, which is narrower than adding duplicate channel-specific text parsers.
