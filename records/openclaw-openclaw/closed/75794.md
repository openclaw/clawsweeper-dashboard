## Summary

Keep open: current main still sends the affected `openclaw cron list`, `add`, and `rm` commands through Gateway RPC. The CLI timeout bounds the wait, but the operator path still depends on a responsive Gateway instead of reading the local cron store directly.

## What This Changes

The issue requests that `openclaw cron list`, `add`, and `rm` avoid hanging behind a busy or degraded Gateway by using the local cron job store directly, with conservative mutation fallback behavior.

## Best Possible Solution

Add a local cron-store inspection path for `list` that preserves current JSON/table output, then implement mutations with deterministic coordination: prefer Gateway RPC when healthy, and only use direct local writes when the Gateway is unreachable or an explicit local mode is selected, with a safe reload/re-arm signal for the running scheduler.

## Reproduction Assessment

Yes for the underlying dependency: stall or mock `callGatewayFromCli` for `cron.list`, `cron.add`, or `cron.remove`, and the CLI waits on Gateway RPC rather than reading `jobs.json`. The exact reported duration still needs a live degraded-Gateway setup because current main also has a default 30s CLI timeout.

## Solution Assessment

Unclear for the full proposed mutation behavior. Local reads are the narrowest maintainable fix for `list`, while `add` and `rm` need a guarded fallback or explicit local mode with reload/re-arm handling rather than silent writes behind a live Gateway.

## Repair Work Prompt

Fix openclaw/openclaw#75794. Current `openclaw cron list`, `add`, and `rm` depend on Gateway RPC and become unusable when the Gateway is busy/degraded. Implement a conservative local cron-store path. Scope: make `cron list` able to read the local store directly using existing cron store helpers (`resolveCronStorePath`, `loadCronStore`/`loadCronStoreSync`) while preserving `--all`, `--json`, delivery preview behavior where safely available, and existing table output. For `add` and `rm`, keep Gateway RPC as the preferred healthy path; add a guarded fallback or explicit local mode only for unreachable/timed-out local Gateway, using existing store normalization/persistence helpers and ensuring the running Gateway will reload/re-arm safely. Do not silently write behind a responsive Gateway, do not invent SQLite support, and do not bypass existing cron validation. Add focused CLI tests that prove list does not call `callGatewayFromCli` on the local path and that mutation fallback behavior is deterministic. Add a user-facing CHANGELOG.md fix entry with proper attribution to the reporter, and update cron CLI/docs copy if flags or fallback behavior change.
