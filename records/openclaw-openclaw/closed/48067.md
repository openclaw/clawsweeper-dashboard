## Summary

Current main resolves the active session row's model through the owning agent before returning `sessions.list`, and the CLI TUI footer renders that resolved row model from `sessionInfo`. The related unmerged #48110 proposal pointed at this same surface, but the behavior is already present on current main and verified in the shipped `v2026.4.26` tag.

## Best Possible Solution

Keep the existing agent-aware session-row resolution and TUI row-consumer path. If a future report shows the footer is wrong before any session row exists, track that as a narrower defaults fallback issue rather than reopening this active-session model bug.
