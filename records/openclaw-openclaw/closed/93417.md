## Summary

Keep open: this is a real Gateway diagnostics trust bug, but this PR is not merge-ready because the new runtime install records are passed to a snapshot API that still neither types nor consumes them, and after-fix real Gateway/OTEL proof is absent.

## What This Changes

The branch threads plugin install records through runtime plugin loading and manifest registry trust checks for diagnostics-otel, with regression coverage for trusted persisted installs and spoofed or untrusted records.

## Best Possible Solution

Add a snapshot-level trusted-install-record path that separates discovery records from trusted persisted/runtime records, prove raw config remains untrusted, and include redacted Gateway OTEL proof before merge.

## Reproduction Assessment

Yes, from source inspection: service startup only injects `internalDiagnostics` when the manifest record has `trustedOfficialInstall`, and diagnostics-otel returns early when that capability is absent. I did not run a live Gateway/OTEL setup in this read-only review.

## Solution Assessment

No. The proposed direction is plausible, but this patch does not wire the new install-record input into the snapshot implementation and needs a clearer trusted-record source boundary.
