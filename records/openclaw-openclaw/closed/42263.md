## Close Comment

Closing from a Codex review because current `main` already implements this path. OpenClaw’s iMessage monitor schema accepts `reply_to_id`, `reply_to_text`, and `reply_to_sender` in `extensions/imessage/src/monitor/types.ts:10`, the parser validates and normalizes them in `extensions/imessage/src/monitor/parse-notification.ts:60`, and inbound processing forwards them as `ReplyToId`, `ReplyToBody`, and `ReplyToSender` in `extensions/imessage/src/monitor/inbound-processing.ts:381` and `extensions/imessage/src/monitor/inbound-processing.ts:633`. The behavior is also covered by `extensions/imessage/src/monitor.gating.test.ts:162`. I also compared the latest release commit to current `main` on these files and found no diff, so this implementation is already present in released code as well. If you still hit a missing-reply-context case, please open a new issue with a full raw inbound payload or gateway logs from the affected `imsg` setup so maintainers can check whether the upstream CLI is omitting the `reply_to_*` fields.
