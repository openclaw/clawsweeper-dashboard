## Summary

Keep open. Current main still passes Chrome MCP a bare temporary screenshot path and then reads that same bare path, while upstream Chrome MCP saves to the extension-suffixed filename; the open #74685 fix candidate should be reviewed or landed instead of closing this early.

## What This Changes

The issue reports that Chrome DevTools MCP `profile="user"` screenshots fail with `ENOENT` because OpenClaw reads a temporary screenshot path that Chrome MCP did not create.

## Best Possible Solution

Review and land #74685, or an equivalent browser plugin fix that reads the Chrome MCP extension-suffixed screenshot file and adds focused regression coverage.

## Reproduction Assessment

Yes, at source level: a fake Chrome MCP session that writes only `<filePath>.<format>` after a successful `take_screenshot` call would make current main read the unwritten bare path and hit `ENOENT`. I did not perform a live Chrome MCP run.

## Solution Assessment

Yes. The narrowest maintainable fix belongs in `extensions/browser/src/browser/chrome-mcp.ts`, matching #74685's direction of reading the extension-suffixed file and covering that contract in a Chrome MCP unit test.
