## Summary

Current main already satisfies the request: the Venice provider uses a live provider-catalog discovery path, has fallback behavior, documentation, and the same implementation is present in the shipped v2026.5.3 release.

## What This Changes

Request dynamic fetching for the Venice AI provider model list instead of relying only on a static bundled catalog.

## Best Possible Solution

Keep the shipped Venice provider-local discovery implementation and direct users to `openclaw models list --all --provider venice` for the current catalog.

## Reproduction Assessment

Not applicable: this is a feature request, not a bug reproduction. Source inspection and a live endpoint shape check show the requested dynamic catalog path already exists on current main and in v2026.5.3.

## Solution Assessment

Yes: the current solution keeps Venice-specific catalog fetching inside the Venice provider plugin, reuses the provider `buildProvider` catalog hook, and preserves the manifest-backed fallback when the API is unavailable.
