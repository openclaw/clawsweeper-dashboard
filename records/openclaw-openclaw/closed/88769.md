## Summary

Keep this PR open for normal maintainer review and landing; current main still has the reported raw-content leak path, and the PR routes visible OpenAI-compatible content through a shared streaming reasoning-tag filter with focused parser/filter coverage and sufficient before/after runtime proof. I found no blocking patch defect, but this remains merge-risky because it changes parser-level message/session content routing for security-sensitive text.

## What This Changes

The PR adds a shared streaming reasoning-tag filter and wires it into both OpenAI-compatible stream parsers so inline reasoning tags are removed from visible text while hidden thinking content and literal code examples are preserved.

## Best Possible Solution

Land the parser-boundary fix after maintainers accept the residual text-routing tradeoffs and the final merge SHA has focused parser/filter proof.

## Reproduction Assessment

Yes for a source-level reproduction: current main appends raw `delta.content` into text while also routing mirrored `reasoning_content` into thinking, so a stream containing both fields duplicates reasoning into visible and hidden parts. I did not run a local test in this read-only review, but the PR supplies before/after runtime output for the same stream shape.

## Solution Assessment

Yes. Filtering at the OpenAI-compatible parser boundary is the narrowest maintainable fix because it prevents bad visible text from being emitted or persisted while keeping the existing delivery-time sanitizer as defense in depth.
