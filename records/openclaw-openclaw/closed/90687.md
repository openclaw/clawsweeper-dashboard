The PR makes `GatewayChannelActor.connect()` no-op when the actor is already connected and adds a Swift regression test for stale `URLSession` task state.

## Best Possible Solution

Land the narrow actor-owned connection-state fix with the regression test after a redacted live macOS node-mode run shows healthy sessions no longer recreate/log repeated gateway connects.

## Reproduction Assessment

No live current-main reproduction was established in this review. The source path is reproducible: repeated `connect()` calls on current main recreate the WebSocket when `connected` remains true but `task.state` is non-running, matching https://github.com/openclaw/openclaw/issues/90668.

## Solution Assessment

Yes, this appears to be the best narrow fix: `GatewayChannelActor` owns connection truth and already clears `connected` on receive, tick, send, and shutdown failures. Caller-side log suppression or macOS-loop throttling would hide the duplicate task creation instead of fixing it.
