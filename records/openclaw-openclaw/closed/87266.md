## Summary

Keep open: the fix is useful and has real local proof, but current head still rewrites every HTTP 403 Realtime client-secret failure into the direct-key diagnostic, which can hide valid-key permission or model-access errors.

## What This Changes

The PR changes OpenAI Realtime Talk auth handling to detect Azure/proxy-looking API keys, emit a direct OpenAI Realtime auth diagnostic, add regression tests, and update Talk/OpenAI docs.

## Best Possible Solution

Keep the direct-key diagnostic, but only convert proven invalid-key or 401-style auth failures and preserve non-auth 403 provider errors with a focused regression test.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction for the review finding: a ProviderHttpError with status 403 reaches isOpenAIRealtimeAuthFailure and is converted because the predicate checks status === 403 without requiring an invalid-key code or message. I did not run the PR tests in this read-only review.

## Solution Assessment

No; the current approach is close, but the bare 403 branch is too broad. The safer solution is to keep the proxy/Azure key diagnostic while preserving non-auth 403 provider errors and covering that with a regression test.
