## Summary

Keep this issue open: current main and the latest release still reach an unguarded shared-memory `.search(...)` call on the reported supported path, and the linked closing PR is still open rather than landed.

## What This Changes

Make `wiki_search` tolerate malformed or throwing shared memory search managers when `search.backend="shared"` and the requested corpus includes `memory` or `all`.

## Best Possible Solution

Land or replace a focused memory-wiki fix that treats shared search as optional augmentation while preserving wiki-only results and current session-visibility filtering for raw shared hits.

## Reproduction Assessment

Yes. Source inspection on current main shows `search.backend="shared"` plus per-call `corpus="memory"` or `"all"` reaches the direct `.search(...)` call; I did not run the external memory plugin setup, so this is source-level reproduction rather than live execution.

## Solution Assessment

Yes. The narrow maintainable fix is a guard/catch around the raw shared-memory search call in `memory-wiki`, before existing session visibility filtering and result mapping; changing memory-core or config schema is not needed.
