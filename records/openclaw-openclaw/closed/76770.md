## Summary

Close: current main now addresses the reported context-cost problem with deferred/searchable OpenClaw dynamic tools for Codex and an opt-in Tool Search catalog for PI runs. The exact requested `tools.defer_loading` name was not adopted, but the observable lazy-schema behavior is implemented and documented.

## What This Changes

Request a user-editable way to defer or lazy-load first-party OpenClaw MCP/dynamic tool schemas so long-running agents do not pay every schema on every turn.

## Best Possible Solution

Keep the current provider-appropriate implementation: Codex uses searchable deferred dynamic tools by default, and PI large catalogs use `tools.toolSearch`; open narrower follow-ups only for runtimes that still lack a safe deferred/searchable surface.

## Reproduction Assessment

Not applicable; this is a feature request, not a broken existing behavior report. Source, docs, tests, and merged PR provenance confirm current main now has the requested lazy/deferred tool-schema behavior for the relevant OpenClaw tool surfaces.

## Solution Assessment

Yes; the current implementation is the maintainable shape because it uses Codex-native searchable dynamic tools where supported and a separate PI Tool Search catalog where that runtime owns the tool loop, instead of adding one generic flag with unclear provider fallback semantics.
