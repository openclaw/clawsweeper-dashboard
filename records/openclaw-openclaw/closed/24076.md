## Summary

Current `main` already implements mobile contact-book access on both iOS and Android nodes, and Android also exposes call-log search. The request in #24076 is satisfied well enough by the shipped node command surface.

## Risks / Open Questions

- The iOS platform docs currently do not spell out the contacts commands even though the code exposes them, so discoverability is weaker than the implementation.

## Close Comment

Closing from a Codex review because current `main` already implements this request. iOS nodes advertise `contacts` and expose `contacts.search` / `contacts.add` in `apps/ios/Sources/Gateway/GatewayConnectionController.swift:825`, route those commands in `apps/ios/Sources/Model/NodeAppModel.swift:1364`, and implement the Contacts store access in `apps/ios/Sources/Contacts/ContactsService.swift:17`. Android nodes expose the same contact commands plus `callLog.search` in `apps/android/app/src/main/java/ai/openclaw/app/node/InvokeCommandRegistry.kt:89`, with the handlers in `apps/android/app/src/main/java/ai/openclaw/app/node/ContactsHandler.kt:305` and `apps/android/app/src/main/java/ai/openclaw/app/node/CallLogHandler.kt:151`. The gateway test coverage in `src/gateway/gateway-misc.test.ts:617` also expects these commands on current main. If there is still a narrower missing mobile contact/call-log workflow beyond these shipped commands, please open a new issue with the exact command or UX gap.
