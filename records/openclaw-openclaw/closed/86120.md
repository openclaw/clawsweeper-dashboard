## Label Justifications

- `P1`: The report describes a real channel workflow becoming unusable until gateway restart after long-running Claude CLI turns.
- `impact:message-loss`: Subsequent inbound Telegram messages fail immediately instead of reaching the agent workflow.
- `impact:session-state`: The failure is tied to stalled session recovery and persisted `sessions.json` state.
- `impact:auth-provider`: The thrown error comes from agent runtime/provider routing for the `claude-cli` backend.

## Summary

Current main already addresses the reported failure mode: `claude-cli` is treated as a provider-owned CLI runtime alias instead of a missing agent harness, stalled active-work recovery is delayed behind a configurable abort threshold, and session-store maintenance now has bounded cleanup paths.

## What This Changes

The issue asks OpenClaw to stop long-running Claude CLI sessions from leaving later Telegram messages failing with `MissingAgentHarnessError`, add stall-timeout configurability, and keep bloated `sessions.json` state from blocking recovery.

## Best Possible Solution

Keep the current-main runtime-alias and configurable stall-recovery behavior, and include it in the next release so users on 2026.5.20 can upgrade without manual gateway restarts.

## Reproduction Assessment

No live reproduction was run against current main. The reported failure path is source-backed by the old `claude-cli` runtime being interpreted as a harness id, and current tests cover that alias no longer throwing `MissingAgentHarnessError`.

## Solution Assessment

Yes. The maintainable fix is to keep `claude-cli` owned by CLI-runtime routing, not the agent-harness registry, while stall recovery remains configurable and delayed for slow legitimate work.
