## Summary

Keep open. Current main still performs a single SearXNG category request and treats a successful empty result set as terminal, while core web-search fallback only advances on provider unavailability, thrown errors, or structured missing-key errors.

## What This Changes

Add a fallback so category-specific web searches that return zero results retry once as general search before returning empty results.

## Best Possible Solution

Implement the retry inside the SearXNG provider/client: after an empty successful response for a non-general categorized request, retry once with general while preserving cache, count, language, timeout, endpoint guard, and existing error handling.

## Reproduction Assessment

Yes. A deterministic mocked SearXNG response where a weather-category request returns {"results":[]} should show current main issuing one categorized request and returning count 0 without a second general request.

## Solution Assessment

Yes. A SearXNG-local retry is the narrowest maintainable fix because generic runWebSearch should keep treating successful zero-result payloads as valid for other providers.
