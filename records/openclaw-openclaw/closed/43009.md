## Label Justifications

- `P2`: This is a persistent user-visible TUI reporting bug with limited functional blast radius and no security or availability impact.
- `impact:session-state`: The TUI footer inaccurately represents the active session's context-usage state when usable session metadata exists elsewhere.

## Summary

Keep open: current main and the latest release still render the TUI footer from `totalTokens` plus `contextTokens` only, while the merged gateway/status context-budget work is not carried through the TUI session model. The related zero-token PRs are closed unmerged, so no open or merged fix currently owns the remaining TUI repair.

## What This Changes

The issue asks the TUI footer to show actual context token usage instead of `tokens ?/200k` when the gateway/status layer has usable context data.

## Best Possible Solution

Carry `totalTokensFresh` and `contextBudgetStatus` through the TUI session metadata, render fresh exact totals when available, render estimated usage with explicit estimated notation when exact usage is unavailable, and keep `?` only for genuinely unknown context usage.

## Reproduction Assessment

Yes, source-reproducible. Current main can hold a known `contextTokens` value without a fresh exact `totalTokens`, and the TUI footer deterministically renders that state through `formatTokens` as `tokens ?/<context>`.

## Solution Assessment

Yes, the narrow repair is freshness-aware TUI projection and rendering that reuses the existing gateway/status semantics. The broader display-preference persistence comment is useful but should be tracked separately rather than bundled into this token-rendering fix.
