## Summary

Keep open: this is a focused but still-blocked PR. Current main has generic stuck-session diagnostics/recovery, but the branch's Discord-specific breaker remains a product-policy decision and the latest diff still has source-reproducible stale-timer and abort-settlement defects plus no real Discord proof.

## What This Changes

The PR adds a Discord-only stuck-session breaker for queued follow-ups, passes active-run callbacks into reply execution, exports a new queue helper, and adds focused Vitest coverage.

## Best Possible Solution

Either reuse current diagnostic recovery/abort-and-drain behavior or land a corrected Discord-specific recovery path only after owner approval, with timer replacement or cleanup, settled abort before drain, and redacted live Discord proof.

## Reproduction Assessment

Yes for the PR defects: the diff's per-key timer guard plus current drain cleanup make stale same-key timers source-reproducible, and immediate drain after signal-only abort can race the reply-run registry. I did not run tests because this review is read-only.

## Solution Assessment

No, not as written. The safer implementation should reuse current abortAndDrainEmbeddedPiRun/diagnostic recovery boundaries or explicitly clear/replace timers and wait for abort settlement before resuming follow-up drain.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Refresh breaker timers for each queued session:** `src/auto-reply/reply/queue/circuit-breaker.ts:73-74`
  - body: The new timer map is keyed only by queueKey and returns when a timer already exists. A normal queue drain can empty and delete the follow-up queue without clearing this timer, so a later Discord session on the same key can skip arming its own breaker and remain unprotected after the stale timer no-ops.
  - confidence: 0.88
- **[P2] Wait for abort settlement before draining recovery work:** `src/auto-reply/reply/queue/circuit-breaker.ts:113`
  - body: abortActiveRun only requests cancellation, but the PR immediately prepends the recovery notice, marks the queue idle, and calls scheduleDrain. While the prior reply operation is still registered, the recovery drain can hit ReplyRunAlreadyActiveError and leave queued work blocked behind the run that is still shutting down.
  - confidence: 0.9
