## Summary

Issue #67658 is superseded by open PR #67711, which directly tracks the same memory-wiki bridge prune bug and proposed guard. Current main has a related loader-side fix for losing the memory capability, but the remaining empty-artifacts prune guard is still best handled in the canonical PR.

## Best Possible Solution

Use PR #67711 as the canonical maintainer review path: land or revise its guard so bridge imports do not prune existing source pages when public memory artifacts are transiently unavailable while source files remain intact, while preserving the existing prune behavior for genuine source deletion.

## Evidence

- **Issue discussion and canonical PR:** The issue comment identifies `syncMemoryWikiBridgeSources` building an empty active key set when public artifacts are unavailable, then pruning all bridge entries; the timeline cross-references PR #67711, titled `fix(memory-wiki): skip bridge prune when capability unavailable but sources intact [AI-assisted]`, whose body says it fixes #67658.
- **Current bridge import still prunes registered empty-artifact capability results:** `syncMemoryWikiBridgeSources` reads public artifacts, builds `activeKeys` from the collected artifacts, and only skips pruning when `getMemoryCapabilityRegistration()` is absent. If a registered memory capability returns zero artifacts, `activeKeys` remains empty and prune still runs.
