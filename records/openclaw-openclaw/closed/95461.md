## Summary

Close as implemented: current main and the latest release already route standard Google Gemini API-key traffic through native Gemini transport and pass raw API keys as `x-goog-api-key`, so the `AQ.` prefix is no longer blocked by OpenClaw's standard Google provider path. The broader wrong-endpoint credential safety boundary remains tracked separately in https://github.com/openclaw/openclaw/issues/85042.

## What This Changes

Requested support for newer Gemini API keys with the `AQ.` prefix instead of assuming only older `AIza...` keys work.

## Best Possible Solution

Keep the released native Gemini transport behavior for the standard Google provider and continue broader credential-to-transport hardening in the canonical open issue.

## Reproduction Assessment

No current-main failing reproduction remains for the standard Google provider path. The historical same-root failure is source-reproducible from the related reports: Google keys were routed through an OpenAI-compatible endpoint before the native Gemini transport default and raw `x-goog-api-key` path were restored.

## Solution Assessment

Yes. The best narrow solution is the one now on main and in `v2026.6.9`: standard Google provider traffic uses native Gemini transport and raw API-key headers; the broader non-standard provider/transport mismatch stays with the canonical safety issue.
