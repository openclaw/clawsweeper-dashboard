## Summary

Current main does not fix the reported vLLM/Qwen tool-call failure. OpenClaw still sends `tool_choice: "auto"` by default for local/proxy OpenAI-compatible completions with tools, only emits executable tool calls from non-empty structured `tool_calls`, and the vLLM plugin/docs remain generic rather than providing a Qwen/vLLM-scoped `required` policy or documented workaround.

## Best Possible Solution

Keep this open for maintainer work. The best path is a vLLM/Qwen-scoped compatibility option, first-class per-model forwarding for a `toolChoice`-style setting, or explicit docs for an `agents.defaults.models["vllm/<model>"].params.extra_body: { tool_choice: "required" }` workaround, with regression coverage proving the override reaches the local vLLM OpenAI-compatible payload without changing native OpenAI behavior globally.
