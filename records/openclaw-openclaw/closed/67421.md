## Summary

Keep open. Current main does not have per-agent web_fetch SSRF overrides, but this draft/protected PR is security-sensitive and the current head would regress SSRF policy cache/schema semantics after the recent IPv6 ULA work.

## What This Changes

The PR adds agents.list[].tools.web.fetch.ssrfPolicy parsing, help, audit coverage, and runtime wiring so the active agent can override web_fetch SSRF policy.

## Best Possible Solution

Rebase onto current main, preserve the full current SSRF policy contract in schema/runtime/cache, get explicit maintainer/security signoff on agent-selection semantics, add the required changelog entry, then re-review the draft.

## Reproduction Assessment

Yes for the PR findings: source inspection shows current main has allowIpv6UniqueLocalRange while the PR serializer omits it, and the alias bug follows from merging global allowPrivateNetwork=true with agent dangerouslyAllowPrivateNetwork=false before calling isPrivateNetworkAllowedByPolicy.

## Solution Assessment

No as-is. The direction fits the existing per-agent tool policy surface, but the PR needs safer SSRF policy merge/serialization semantics and owner review before it is the narrowest maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Include the IPv6 ULA flag in SSRF cache keys:** `src/infra/net/ssrf.ts:86-94`
  - body: Current main supports `allowIpv6UniqueLocalRange`, but this new serializer leaves that field out of the effective-policy cache key. After rebase, a fetch performed with the IPv6 ULA exemption can share a cache key with a stricter policy and leak the broader-policy cached response into strict `web_fetch` calls; include the flag and cover the cache boundary in tests.
  - confidence: 0.92
- **[P1] Canonicalize private-network aliases when merging policies:** `src/infra/net/ssrf.ts:77-84`
  - body: The shallow spread lets legacy and new private-network aliases conflict across scopes. For example, global `allowPrivateNetwork: true` plus agent `dangerouslyAllowPrivateNetwork: false` still allows private-network fetches because `isPrivateNetworkAllowedByPolicy` treats either true flag as access; normalize the aliases so an agent override can actually tighten the baseline.
  - confidence: 0.9
- **[P2] Report only actual agent-level SSRF widening:** `src/security/audit.ts:1145-1154`
  - body: The agent audit warning fires whenever the effective policy allows private-network access, even when the global baseline already allows it and the agent only set an unrelated exception. That makes the audit say the agent widened access when it did not; compare global and agent/effective policy before choosing the warning wording.
  - confidence: 0.84
- **[P2] Add the required changelog entry:** `src/config/zod-schema.agent-runtime.ts:610`
  - body: This is a user-facing `feat` that adds new config keys and runtime behavior, but the PR file list has no `CHANGELOG.md` entry. Repo policy requires a changelog entry before landing this kind of change.
