## Summary

Keep open. This is the active candidate fix for the canonical QMD embedInterval issue, but the branch still changes the documented lexical-mode contract, can opt malformed nonblank intervals into background embeds, and lacks real OpenClaw/QMD runtime proof.

## What This Changes

The PR changes QMD memory scheduling so an explicitly configured `memory.qmd.update.embedInterval` can trigger `qmd embed` even when QMD search mode is lexical `search`.

## Best Possible Solution

Have memory/QMD owners decide the lexical-mode contract; if accepted, gate the override on a successfully parsed positive interval, update QMD docs, and require redacted runtime proof before merge.

## Reproduction Assessment

Yes. Source inspection shows current main resolves `embedInterval`, but lexical `search` mode returns before the interval can schedule or run embedding; no live QMD process was run in this read-only review.

## Solution Assessment

No as submitted. Carrying explicit interval intent through resolved config is a reasonable boundary if owners accept the override, but the flag must require a parsed positive interval and docs/runtime proof need to align.
