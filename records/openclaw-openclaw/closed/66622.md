## Summary

Keep open. The underlying bug still appears real on current `main`, but this PR is not strong enough for conservative maintainer cleanup: the proposed switch to `resolveModelWithRegistry` likely changes MiniMax miss handling in a user-visible way, and the added test mocking strategy appears incompatible with the runtime normalization path. The prior review discussion was useful and directionally correct about the missing dynamic-model preparation, but it also surfaced unresolved regressions.

## Evidence

- **Current image resolver still bypasses dynamic-model preparation:** On `main`, `resolveImageRuntime` normalizes the provider/model and then calls `modelRegistry.find(...)` directly. It never calls `prepareProviderDynamicModel` or `resolveModelWithRegistry`, so dynamic-provider image models can still miss exactly as described in the PR/issue.

This matches the unresolved Codex review comments on the PR.

- The bug this PR targets still appears to exist on current `main`, so closing it would likely hide a real image-model resolution gap for dynamic providers.

- The PR's added tests do not provide high-confidence proof because the proposed provider-runtime mock appears incomplete for the normalization path.
