## Label Justifications

- `P1`: The current release can abort valid slow local-model turns at 120s despite documented provider timeout configuration, breaking agent replies for affected Docker/Ollama users.
- `impact:message-loss`: Affected channel or UI messages can time out without an agent reply when the local model is silent beyond 120 seconds.
- `impact:auth-provider`: The issue is centered on model-provider configuration and runtime provider timeout routing.

## Summary

Keep open: current `main` still caps the documented Docker-local Ollama provider timeout to the 120s idle watchdog because `host.docker.internal` is not treated as local. The related open PR https://github.com/openclaw/openclaw/pull/83979 targets the same resolver branch, but it is unmerged and still proof-gated.

## What This Changes

Make `models.providers.<id>.timeoutSeconds` extend the LLM idle watchdog for Docker/local Ollama so slow local models are not aborted at 120s.

## Best Possible Solution

Honor the documented Docker-local provider endpoint or explicit provider timeout in the idle-timeout resolver, add regression coverage for `host.docker.internal`, and preserve the default cloud silent-stream guard unless maintainers explicitly accept broader timeout semantics.

## Reproduction Assessment

Yes, at source level: with `baseUrl: "http://host.docker.internal:11434"` and `modelRequestTimeoutMs: 1200000`, current `main` does not classify the endpoint as local and caps the idle watchdog to 120000ms. I did not run a live Docker/Ollama reproduction in this read-only pass.

## Solution Assessment

Yes for the issue's narrow path: the maintainable fix is in idle-timeout locality classification or explicit provider-timeout handling, with focused tests for `host.docker.internal`. The related open PR may be the right vehicle if maintainers accept its broader explicit-timeout semantics and add usable proof.
