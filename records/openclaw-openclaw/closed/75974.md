## Summary

Keep open. The PR targets a real plugin-tool slowdown and includes a changelog/test, but the patch is not mergeable because the cache-hit timing value is type-wrong and the `pluginId`-only cache reuses request-scoped tool objects across sessions, channels, sandbox state, and browser policy.

## What This Changes

The PR adds a global plugin tool factory result cache keyed by `pluginId`, a reset helper/test, and a changelog entry for the Windows plugin-tool slowdown.

## Best Possible Solution

Keep the PR open but redesign caching around a context-safe opt-in contract or a complete registry/config/request cache key, with regression tests for context-sensitive factories before merge.

## Reproduction Assessment

Yes for the PR regression: call `resolvePluginTools()` twice for the same plugin id with different `sandboxed`, browser, session, or delivery context values and the PR cache reuses the first tool. Existing Lobster and Zalo Personal tests already show factory output depends on those context fields; the linked Windows slowdown itself was not live-reproduced in this read-only pass.

## Solution Assessment

No. Caching may be the right performance direction, but this PR’s global `pluginId` cache violates the current plugin tool factory contract and also appears to fail type checking in the cache-hit timing branch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Unpack the factory timing result:** `src/plugins/tools.ts:465`
  - body: `describePluginToolFactoryResult()` returns `{ result, resultCount }`, but this cache-hit branch assigns that whole object to `PluginToolFactoryTiming.result`, which is typed as the string union used by `formatPluginToolFactoryTiming()`. This should fail type checking and would also format the timing result incorrectly.
  - confidence: 0.97
- **[P1] Preserve request-scoped factory context:** `src/plugins/tools.ts:457`
  - body: Several current factories intentionally close over `OpenClawPluginToolContext`: browser captures sandbox/host policy and session key, Lobster gates on `ctx.sandboxed` and binds TaskFlow to `ctx.sessionKey`, and Zalo Personal captures ambient `deliveryContext`. A cache keyed only by `pluginId` reuses the first tool object for later requests, so later sessions/channels can inherit stale policy or message routing.
  - confidence: 0.93
