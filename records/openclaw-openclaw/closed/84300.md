## Summary

Keep this PR open: it now covers the main callback plumbing and Codex dynamic-tool path, but the supplied proof is still mock/terminal-only and the delivered signal may fire too late to prevent the in-flight keepalive race described by the bug.

## What This Changes

The PR adds a source-reply-delivered typing-controller signal, invokes it from guarded `message(action="send")` results in `message_tool_only` mode across Pi/Codex tool paths, and updates tests plus changelog.

## Best Possible Solution

Suppress or seal source typing early enough that a source-conversation `message_tool_only` send cannot leave a keepalive packet racing the delivered reply, then back it with focused unit coverage and redacted live Discord proof.

## Reproduction Assessment

No high-confidence live reproduction was run in this read-only review. The linked issue gives concrete Discord DM `message_tool_only` steps, and current main source shows the idle-gated typing cleanup path that can leave the reported TTL behavior plausible.

## Solution Assessment

No. The callback plumbing is the right area, but firing only after the send resolves may be too late for the in-flight keepalive packet named in the root cause; the safer fix should prevent that race and prove it in Discord.
