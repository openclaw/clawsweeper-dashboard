## Summary

Keep open: the report describes a current, user-visible Feishu WebSocket dispatch regression on the latest release, and the current source still has a runtime-injection-sensitive path where Feishu inbound handling depends on `getFeishuRuntime()` before calling `core.channel.inbound.run`. I did not find a merged fix or canonical replacement item that safely owns this exact failure.

## What This Changes

The issue requests a fix for Feishu WebSocket inbound messages failing to dispatch because the Feishu plugin runtime is not initialized before monitor handling calls the core inbound runner.

## Best Possible Solution

Ensure the Feishu channel runtime setter is applied before any WebSocket or webhook account monitor can register inbound handlers, then cover the packaged gateway startup path with a focused regression test.

## Reproduction Assessment

No live high-confidence reproduction was run in this read-only review. The reporter provided concrete versioned steps and logs, and current source shows the same runtime-dependent Feishu inbound dispatch path, so it is source-reproducible with medium confidence.

## Solution Assessment

Unclear until the startup path is isolated. The best fix should preserve the plugin/runtime boundary and prove the Feishu `PluginRuntime` is injected before monitor startup, rather than passing the lightweight CLI `RuntimeEnv` into the Feishu runtime store.
