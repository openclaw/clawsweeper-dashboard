## Summary

Keep open. Current `main` still matches the report: `repairLaunchAgentBootstrap` treats a successful fresh bootstrap as `repaired` but immediately runs `launchctl kickstart -k`, and existing tests assert that sequence; #74361 mitigates recycled-PID lock fallout but does not remove this LaunchAgent SIGTERM source.

## What This Changes

The issue asks to skip `launchctl kickstart -k` after `repairLaunchAgentBootstrap` successfully bootstraps a fresh macOS LaunchAgent, while preserving the bounce for already-loaded jobs.

## Best Possible Solution

Gate `kickstart -k` to the already-loaded bootstrap branch, return `repaired` immediately after a fresh bootstrap, and cover both branches with regression tests plus a changelog entry.

## Reproduction Assessment

Yes, source-reproducible with high confidence: the current fresh-bootstrap path returns `repaired` and then calls `kickstart -k`, and `src/daemon/launchd.test.ts` asserts that ordering. I did not perform a live macOS launchd run in this read-only review.

## Solution Assessment

Yes. The requested `repairStatus === "already-loaded"` guard is the narrowest maintainable fix because it preserves the needed bounce for an existing job while avoiding a needless SIGTERM of a newly bootstrapped RunAtLoad agent.
