## Summary

Keep this PR open. Current main still rejects Claude Code-style flat/alias parameters for the read/write/edit tools, so the intended behavior is not implemented. The PR is also not ready to merge as-is: its proposed `prepareArguments` wrapper is not carried through the current embedded `customTools` adapter path before validation, matching the P1 review concern. Security review found no dependency/CI/supply-chain changes, but the diff changes file-tool argument acceptance and needs tighter tests and alias scope review.

## Best Possible Solution

Keep the PR open, but request a revised implementation: add or use a core pre-validation argument-normalization seam for OpenClaw-owned read/write/edit tools, or explicitly preserve/invoke `prepareArguments` in the production custom-tool adapter before validation. The revision should avoid or justify the broad `file` alias, add focused tests around `createOpenClawCodingTools` and the embedded adapter path, and prove sandbox/workspace guards still run after normalization. If maintainers prefer a plugin route, first add a provider-agnostic plugin API because the current normal hook/provider-schema hooks do not cleanly solve this across providers.
