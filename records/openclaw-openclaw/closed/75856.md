## Summary

Close as implemented on main. Current code treats provider-side abort text such as `This operation was aborted` as a failover-worthy timeout, normalizes abort-wrapped provider failures before deciding whether to rethrow, and throws prompt-side failover errors from inside the `runWithModelFallback` attempt so the fallback chain can continue. The same implementation is present in the shipped `v2026.4.29` tag, which is newer than the reported `2026.4.26` version.

## What This Changes

The issue requests that a provider 429 followed by an AbortController AbortError update auth failure state and enter model fallback instead of ending the session with `openclaw:prompt-error`.

## Best Possible Solution

Keep the generic provider/failover implementation rather than adding Cerebras-specific handling. If this still reproduces on `v2026.4.29` or newer, the next useful report should include the raw thrown error shape, nested `cause` fields, and the session JSONL around the prompt-error entry so the classifier can be extended precisely.

## Reproduction Assessment

Yes, at the code-path level: an AbortError message matching `This operation was aborted` is covered by the failover timeout matcher, and prompt-side classified failures are thrown back through `runWithModelFallback`. I did not run a live Cerebras reproduction because this was a read-only review.

## Solution Assessment

Yes. The current solution is the maintainable generic path: classify provider abort/stream aborts as failover-worthy when they are not explicit user aborts, preserve true user abort behavior, and keep 429 retry-bypass handling in the shared provider transport.
