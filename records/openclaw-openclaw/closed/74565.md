## Summary

Keep this PR open. Current main still treats a non-empty malformed `gateway.remote.url` as absent in both the core setup-code resolver and the device-pair plugin path, so it can fall through to another URL source before issuing a bootstrap token. The proposed patch is narrow, adds the missing invalid-vs-absent distinction, and I found no concrete correctness or security regression in the diff.

## What This Changes

The PR adds `gateway.remote.url` invalid-state checks in core pairing setup and device-pair `/pair` URL resolution, plus regression coverage for setup-code, QR CLI, and device-pair command paths.

## Best Possible Solution

Land this narrow validation change, or an equivalent replacement, so both core setup-code generation and the device-pair plugin reject non-empty malformed `gateway.remote.url` values before any Tailscale/bind fallback or bootstrap-token issuance.
