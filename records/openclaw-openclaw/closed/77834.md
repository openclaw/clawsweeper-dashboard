## Summary

Current main and the v2026.5.4 release already cover the reported 2026.5.3 local-model timeout path: local/private model endpoints no longer get the default idle watchdog, and provider-scoped request timeouts still apply when configured.

## What This Changes

The issue asks OpenClaw to stop slow or timed-out skill invocations with a local Qwen/llama.cpp model on 2026.5.3, where runs frequently surface `LLM request timed out` despite fast local generation.

## Best Possible Solution

Keep the v2026.5.4 local-provider timeout behavior and handle any remaining post-upgrade reports with provider baseUrl, timeoutSeconds, and fresh logs.

## Reproduction Assessment

No live Qwen/llama.cpp setup was run, so there is not a high-confidence current-main failing reproduction. The reported `LLM request timed out` path is clear from source, and current main’s source/tests show the local-provider watchdog behavior that prevents the 2026.5.3 failure mode.

## Solution Assessment

Yes. The current implementation is the narrow maintainable fix because it keeps the cloud-provider idle watchdog, exempts local/private model endpoints that can legitimately stay silent during prompt evaluation, and still honors provider-scoped `timeoutSeconds`.
