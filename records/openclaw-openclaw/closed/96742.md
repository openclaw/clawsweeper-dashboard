## Summary

Current main and the latest inspected release already preserve custom Ollama provider IDs through memory embedding adapter creation; the reported `Provider: ollama (requested: ollama-proxy)` status text is the resolved adapter plus requested custom ID, not proof that the custom `baseUrl` was dropped.

## What This Changes

The issue reports that `memorySearch.provider` loses a custom Ollama provider's configured `baseUrl` when the provider is an alias under `models.providers` with `api: "ollama"`.

## Best Possible Solution

Keep the shipped custom-provider-ID implementation; if the reporter still sees a 404, the next useful evidence would be redacted debug logs showing the actual request URL rather than the status adapter label.

## Reproduction Assessment

No high-confidence current-main reproduction remains from source inspection. The checked status/search path preserves the custom provider ID into the Ollama adapter, and the status text showing `ollama` as provider is expected adapter display alongside the requested custom ID.

## Solution Assessment

Yes. The maintainable solution is the existing documented provider-owner lookup plus custom-ID config preservation; adding a second runtime path or config knob is not needed for the reported config shape.
