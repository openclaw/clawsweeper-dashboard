## Summary

Current `main` already avoids the reported `job must be object` failure for `openclaw cron add`: the CLI builds a cron job payload directly for `cron.add`, and the cron tool itself now recovers flat add params when `job` is missing.

## Evidence

- **CLI `cron add` sends a normalized job payload directly:** `registerCronAddCommand` parses `--name/--every/--message` into `schedule`, `payload`, and related fields, then calls `callGatewayFromCli("cron.add", opts, params)` with that job-shaped object rather than a flat tool payload.
  - file: [src/cli/cron-cli/register.cron-add.ts:66](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/cli/cron-cli/register.cron-add.ts#L66)

- **CLI tests cover current `cron add` parameter shaping:** Tests assert that `cron add` generates the expected `cron.add` params, including isolated/main session inference and delivery defaults, which exercises the current add path instead of a missing `job` wrapper path.
