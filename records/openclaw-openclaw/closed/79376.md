## Label Justifications

- `P1`: The original bug duplicated user-visible WhatsApp group posts after a failed channel send, affecting a real outbound channel workflow.
- `impact:message-loss`: The report is about duplicated channel messages from delivery recovery replaying sends the caller had already treated as failed.

## Summary

Close as implemented: current main and the v2026.5.28 release mark queued platform send attempts before plugin I/O, and recovery refuses to replay started or unknown entries unless a channel reconciler proves they were not sent; WhatsApp does not declare that reconciler, so listener-down UNAVAILABLE sends no longer silently recover into duplicate posts.

## What This Changes

The issue asks OpenClaw to stop delivery-recovery from later posting WhatsApp sends that already returned UNAVAILABLE, or to report them as deferred instead of failed.

## Best Possible Solution

Keep the shipped recovery-state behavior, where ambiguous started sends are not replayed unless a channel explicitly proves they were not sent; the broader WhatsApp 408/load instability remains separate from this duplicate-replay bug.

## Reproduction Assessment

No current-main reproduction remains: source inspection shows started WhatsApp send failures are now marked and recovery refuses blind replay without a reconciler. The original 2026.5.7 reproduction is credible from the provided logs, but the current shipped path no longer matches it.

## Solution Assessment

Yes: the current generic recovery-state guard is a maintainable fix because it preserves crash recovery only when replay is safe or explicitly reconciled, without adding WhatsApp-specific policy to core.
