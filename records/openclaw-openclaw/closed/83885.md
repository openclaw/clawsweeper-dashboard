## Label Justifications

- `P2`: The item is a scoped Twitch plugin reliability bug with limited blast radius.
- `impact:message-loss`: The reported broken cached client could prevent Twitch chat send or receive paths from working until reset.
- `impact:auth-provider`: The report centers on Twitch authentication failure handling and retry behavior after bad credentials.

## Summary

Current main and the shipped `v2026.6.6` release now keep Twitch clients in a pending connection path and only put them in the reusable client map after the event-based connection helper resolves, so the reported broken-client cache path is implemented.

## What This Changes

The issue reports that Twitch `ChatClient.connect()` was started without confirmed connection handling, allowing a failed initial Twitch connection to be stored and reused from the clients map.

## Best Possible Solution

Keep the shipped event-based Twitch connection lifecycle; if a new credential-failure path still breaks in the field, open a fresh report with live logs against `v2026.6.6` or newer.

## Reproduction Assessment

Yes for the original source-level failure: the initial Twitch plugin code cached the client immediately after `client.connect()`. Current main and `v2026.6.6` no longer reproduce that source shape because cache insertion now happens after `connectClient` resolves; I did not run a live invalid-credential Twitch test.

## Solution Assessment

Yes: the shipped fix follows Twurple's event-driven `connect()` contract by waiting on auth/connect/disconnect events and keeping failed setup out of the reusable clients map.
