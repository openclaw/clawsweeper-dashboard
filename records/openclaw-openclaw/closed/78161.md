## Summary

Keep open: current main still trusts a stored user/legacy session override in allow-any mode, so a corrupt OpenRouter override can route as `anthropic/claude-haiku-4.5` before the configured `openrouter/...` default is used.

## What This Changes

Repair corrupt persisted session model overrides where an OpenRouter model id is stored as a native provider/model pair.

## Best Possible Solution

Add a read-side cleanup for stored overrides whose provider/model cannot resolve to a registered allowed model, and add write-side validation so `sessions.patch` and the Control UI cannot persist that corrupt shape again.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a session entry with `providerOverride: "anthropic"` and `modelOverride: "claude-haiku-4.5"` is accepted in allow-any mode and passed to model resolution as `anthropic/claude-haiku-4.5`.

## Solution Assessment

Yes. The narrow maintainable fix is to reject or normalize invalid session model writes and self-heal already-stored invalid overrides before dispatch, using the existing model catalog/provider resolution contracts.

## Repair Work Prompt

Fix the session model override corruption reported in https://github.com/openclaw/openclaw/issues/78161, preserving the related context from https://github.com/openclaw/openclaw/issues/70572. On current main, a stored `providerOverride: "anthropic"` plus `modelOverride: "claude-haiku-4.5"` can be used in allow-any mode even when the configured default is `openrouter/anthropic/claude-haiku-4.5`, causing `Unknown model: anthropic/claude-haiku-4.5`. Add a read-side guard in the agent dispatch/model-selection path that clears or ignores stored session model overrides whose provider/model cannot resolve to a registered/allowed model, regardless of allow-any mode. Add a write-side guard around `sessions.patch`/`applyModelOverrideToSessionEntry` so corrupt provider/model pairs are rejected or normalized before persistence; use existing model catalog/provider resolution helpers rather than checking only `cfg.models.providers`, because plugin/catalog providers exist. Add focused regression coverage for a corrupt OpenRouter wrapper id stored as a native provider pair, valid `openrouter/anthropic/...` writes, and no regression for legitimate configured/plugin providers. Include a user-facing changelog fix entry; do not add a config option or change fallback policy for valid user-selected session models.
