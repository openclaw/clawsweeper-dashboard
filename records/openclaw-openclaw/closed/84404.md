## Summary

Close as not reproducible: current main and v2026.5.18 route gateway restart through `systemctl restart` or the RPC/SIGUSR1 restart path; the only `systemctl stop` path is the explicit stop command, which docs already say not to use as a restart substitute.

## What This Changes

The issue asks OpenClaw to stop using `systemctl stop` for Linux gateway restart flows because systemd treats explicit stop as a manual stop and leaves `Restart=always` units inactive.

## Best Possible Solution

Keep explicit stop semantics and continue routing restart requests through `systemctl restart` or the existing `gateway.restart.request`/SIGUSR1 path; reopen only with fresh logs showing a current restart command actually issuing stop.

## Reproduction Assessment

No. Source and release inspection show restart paths use `systemctl restart` or SIGUSR1 restart handling, while `systemctl stop` is only used for the explicit stop command.

## Solution Assessment

Not applicable as a fix. Changing `gateway stop` to restart would break explicit stop semantics; the current supported solution is to use `openclaw gateway restart` for restarts.
