## Summary

Keep open: current `main` and the latest release still have a source path that can rewrite the session’s top-level runtime provider from canonical `openai` back to `openai-codex`, which matches the reported recurring doctor warning.

## What This Changes

The issue reports that canonical OpenAI routes executed through Codex OAuth persist `openai-codex` into session `modelProvider`, making `openclaw doctor` repeatedly warn about legacy Codex session route state.

## Best Possible Solution

Keep the top-level session model route as the user-facing canonical provider for `openai/*` runs, store Codex OAuth selection only in auth-profile metadata, and add a regression for doctor-fix followed by a successful Codex-auth canonical OpenAI run.

## Reproduction Assessment

Yes from source, but not from a live run in this review. The current code can route canonical `openai` through `openai-codex`, then persist that runtime provider into `modelProvider`, which doctor immediately classifies as legacy session route state.

## Solution Assessment

Yes, a narrow bug fix is the right path. The maintainable fix is to separate user-facing route persistence from auth/runtime provider metadata, not to change doctor’s legacy-route scanner or add a new config knob.
