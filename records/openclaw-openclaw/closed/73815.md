## Summary

Current main implements the reported two-path `OPENCLAW_PLUGIN_STAGE_DIR` behavior for bundled memory runtime imports. The memory-core runtime facade now routes built public-surface loads through bundled runtime-deps staging before import, and the runtime-deps layer links or copies dependencies found in earlier read-only search roots into the final writable install root, so a package present in the first path is available when the runtime mirror imports from the second path. This is fixed on current `main` only; I could not prove it has shipped in a release after `v2026.4.26`.

## Best Possible Solution

Keep the current main implementation: staged bundled public-surface loads should prepare the runtime mirror before import, and layered stage roots should reuse packages from earlier roots by linking/copying them into the final writable root. The remaining product action is to ship this in the next release rather than keep a separate core issue open.
