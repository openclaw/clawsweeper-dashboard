## Summary

Keep open. Current main still has a concrete MSTeams source path for `...:thread:old:thread:new` keys: the handler routes by the normalized channel id, mutates the returned route after lookup, and the thread helper appends to whatever base key it receives.

## What This Changes

The issue requests preventing MSTeams channel-thread session keys from mixing an old thread-qualified session key with a newly appended thread suffix.

## Best Possible Solution

Fix MSTeams thread routing so thread-qualified session keys are derived from an immutable clean base route, and add regression coverage for old-thread-plus-new-thread keys and route-cache mutation.

## Reproduction Assessment

Yes for a high-confidence source-level reproduction: `resolveMSTeamsRouteSessionKey` given a base like `agent:main:msteams:channel:19:x@thread.tacv2:thread:old` and a new channel thread id deterministically produces a double suffix. A full live Teams fresh-room recipe is still missing.

## Solution Assessment

Yes. The narrow maintainable fix is MSTeams-owned: clean or preserve an immutable base session key before suffixing, and avoid mutating cached route objects returned by shared routing.
