## Summary

Keep open. Current `main` still has the unsafe `cron` CLI output path: `src/cli/cron-cli/shared.ts` dereferences `job.schedule.kind` and `job.payload.kind` without null guards, while cron store load still uses `normalizeCronJobInput(raw)` without applying defaults, so legacy/corrupt jobs can still reach the CLI with missing `schedule` or `payload`. That means #37299 is not clearly fixed on `main` or in `v2026.4.22`.

## Evidence

- **Current CLI still dereferences missing schedule/payload:** `printCronList()` calls `formatSchedule(job.schedule)` and `formatSchedule()` immediately reads `schedule.kind`; it also reads `job.payload.kind` at two call sites. `printCronShow()` also reads `job.payload.kind` directly.
  - file: [src/cli/cron-cli/shared.ts:203](https://github.com/openclaw/openclaw/blob/7d471837369885c12d540d6bafa319ff41129b89/src/cli/cron-cli/shared.ts#L203)
  - command: `nl -ba src/cli/cron-cli/shared.ts | sed -n '190,380p'`
  - sha: [7d4718373698](https://github.com/openclaw/openclaw/commit/7d471837369885c12d540d6bafa319ff41129b89)
- **Store load does not guarantee defaults for legacy jobs:** `ensureLoaded()` normalizes persisted jobs with `normalizeCronJobInput(raw)`, but does not use the create-path defaults. A job that cannot infer `payload`/`schedule` keeps the partial shape and is placed back into memory.
  - file: [src/cron/service/store.ts:39](https://github.com/openclaw/openclaw/blob/7d471837369885c12d540d6bafa319ff41129b89/src/cron/service/store.ts#L39)

The store-load path uses `applyDefaults: false`, so malformed persisted jobs can remain without a usable `payload` or `schedule`.
