## Summary

Keep open: the report is fresh, externally reported, and the current heartbeat wake layer still has a plausible global stuck-dispatch mode if an active heartbeat handler never settles; the supplied related items overlap but do not cleanly cover this compaction/recreation + no-op config-write symptom.

## What This Changes

The issue reports that heartbeat polling can silently stop after session compaction/recreation while the gateway and direct session remain responsive, resuming only after a heartbeat config write.

## Best Possible Solution

Add scheduler self-healing and diagnostics for a heartbeat wake handler that remains running indefinitely, then prove interval, retry, targeted cron/system-event, and config hot-reload paths keep dispatching after compaction-like stalled runs.

## Reproduction Assessment

No, not for the exact production compaction sequence. Source inspection does show a plausible stuck scheduler path where a non-settling heartbeat handler leaves the global wake layer in running state and blocks later heartbeat dispatches.

## Solution Assessment

Unclear but bounded. The best fix is likely in the heartbeat wake/scheduler state machine, not in session compaction itself, unless a focused repro proves compaction leaves a specific runner promise or lane stuck.
