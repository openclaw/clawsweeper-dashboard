## Summary

Current main directly addresses the reported stale dist chunk failure by forcing package-manager `update.run` restarts to be immediate, non-deferred, and no-cooldown after a package swap, while shell package updates stop/restart managed services around the install and verify the restarted version.

## What This Changes

The issue asks for update/restart semantics that prevent an already-running gateway from lazy-importing removed hashed dist chunks after a package update.

## Best Possible Solution

Keep the current main behavior: package swaps must force a clean immediate restart, supervised shell updates should stop/restart the managed service, and `--no-restart` remains an explicit operator opt-out.

## Reproduction Assessment

No high-confidence failing reproduction remains against current main. Source and regression tests show the package-manager update path now avoids the stale-chunk window by forcing an immediate restart after package replacement.

## Solution Assessment

Yes. The current fix is narrower than changing Telegram outbound delivery because it hardens the update/restart boundary that can leave any channel importing from a replaced dist tree.
