## Summary

Close as implemented on current main. The heartbeat resolver can still produce an exec-event turn without `MessageThreadId` for explicit heartbeat targets, but the final reply dispatch path now recovers the thread id from the thread-scoped session key and Slack route delivery maps that id to `thread_ts`, which satisfies the reported observable problem.

## What This Changes

The issue requests that Slack async exec completion replies from a channel thread stay in the originating Slack thread when heartbeat delivery uses an explicit Slack target.

## Best Possible Solution

Keep the current final-reply routing behavior that recovers thread context from thread-scoped session keys for exec-event turns. If maintainers want the assistant-visible exec-event envelope itself to include Slack thread metadata, that can remain a narrower follow-up separate from this reported channel-root leak.

## Reproduction Assessment

Yes. The report provides a concrete Slack repro with marker output, timestamps, config, and a thread-scoped session key; current main has a code-level verification path showing that same session-key thread id is restored before Slack final delivery.

## Solution Assessment

Yes. Recovering the final reply thread from the active thread-scoped session key is the narrowest safe fix for the observable leak because it avoids broadly inheriting stale heartbeat thread ids while still preserving the originating Slack thread for this exec-event session.
