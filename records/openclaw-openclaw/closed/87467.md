## Summary

This issue should stay open: current main has forward handling for auto fallback pins with origin metadata, but source still allows a direct `modelOverrideSource: "auto"` pin without fallback-origin fields to be reused as the session model and never become eligible for the primary-recovery probe path.

## What This Changes

The issue reports that an automatically persisted rate-limit fallback model override can keep a Discord session on the fallback model after the primary model recovers.

## Best Possible Solution

Add a narrow normal-selection or probe repair for explicit auto fallback pins missing origin metadata so they clear or retry the configured primary, while preserving user overrides, legacy missing-source overrides, parent-session behavior, and the documented five-minute probe cadence.

## Reproduction Assessment

Yes, source-reproducible: current main cannot create a primary-recovery probe for an explicit auto pin that lacks fallback-origin metadata, and the model-selection tests preserve generic auto pins across turns until reset. I did not run the live Discord/Windows path.

## Solution Assessment

Yes, the best fix is a narrow source repair around model selection or primary-probe resolution for explicit auto pins without origin metadata. It should not broaden user override behavior or change the documented periodic probe policy.
