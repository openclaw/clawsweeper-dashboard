- `P2`: The report is a focused outbound attachment delivery bug with limited blast radius and no evidence of a core-wide outage.
- `impact:message-loss`: The failure prevents generated attachments from leaving the runtime through the message send path.

## Summary

Keep open: current main and the latest release still drive normal `message.send` media delivery from `mediaUrl`/`mediaUrls`, while buffer-only attachment data is not materialized for ordinary send.

## What This Changes

The issue requests support for `message.send` attachments supplied only as an in-memory buffer with filename and content type, without a media path or URL.

## Best Possible Solution

Materialize buffer-only `message.send` attachments into an OpenClaw-owned temporary media artifact before normal delivery, preserving the existing channel `mediaUrl`/`mediaUrls` contract and media access limits.

## Reproduction Assessment

Yes by source inspection: `message.send` builds normal delivery payloads from `mediaUrl`/`mediaUrls`, while buffer-only attachments never become media URLs for the durable send path. I did not run a live reproduction because this review is read-only.

## Solution Assessment

Yes, the requested direction is the right boundary if implemented as buffer-to-temp-media normalization before channel dispatch. The safer alternative is not to add per-channel buffer contracts, because current channel adapters already consume media paths/URLs.
