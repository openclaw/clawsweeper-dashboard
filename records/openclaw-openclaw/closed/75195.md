## Summary

Keep open. The PR targets a real `chat.history` display-normalization gap for delivery-mirror rows, but the current diff also filters `gateway-injected` assistant rows that current main uses for UI-visible `chat.inject` notes and persisted abort partials, so the patch would make those entries disappear after a history reload.

## What This Changes

The PR exports a shared predicate for OpenClaw transcript-only assistant rows, reuses it in pi-embedded subscriber handling, filters matching rows from `chat.history`, and adds shared predicate unit coverage.

## Best Possible Solution

Keep the gateway-side history filter for redundant delivery mirrors, but do not hide all `gateway-injected` rows from `chat.history`; split the predicates or filter only delivery mirrors for display history, and add regression coverage proving delivery mirrors are hidden while `chat.inject` notes and abort partials survive reload.

## Reproduction Assessment

Yes. A high-confidence source-level check is to write a session transcript row with `role: "assistant"`, `provider: "openclaw"`, and `model: "gateway-injected"` as produced by `chat.inject` or abort persistence, then call `chat.history`; the PR filter would remove that row even though current docs and handlers treat it as UI-visible.

## Solution Assessment

No. Filtering at the gateway boundary is the right fix point for delivery mirrors, but reusing the subscriber/replay predicate for `chat.history` is too broad because `gateway-injected` has current UI-visible uses.
