## Summary

Current `main` already reclaims stale gateway lock files before restart/start by validating the recorded owner PID and listener state, removing dead/stale locks, and releasing the lock before respawn. The changelog shows this shipped in `v2026.4.5`.

## Evidence

- **Stale lock owners are detected and removed:** `acquireGatewayLock()` reads the existing lock payload, checks whether the recorded owner is dead via port probe / PID liveness / argv checks, and deletes the lock when the owner is dead or stale instead of failing restart/start.
  - file: [src/infra/gateway-lock.ts:193](https://github.com/openclaw/openclaw/blob/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd/src/infra/gateway-lock.ts#L193)
  - command: `nl -ba src/infra/gateway-lock.ts | sed -n '180,343p'`
  - sha: [01bc49c88c0c](https://github.com/openclaw/openclaw/commit/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd)
- **Restart path releases the lock before spawning the replacement gateway:** `runGatewayLoop()` explicitly releases the held gateway lock before respawning so the child can acquire it immediately, addressing restart handoff races.
  - file: [src/cli/gateway-cli/run-loop.ts:63](https://github.com/openclaw/openclaw/blob/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd/src/cli/gateway-cli/run-loop.ts#L63)

- **Regression coverage exists for stale-lock recovery:** Tests cover reclaiming a stale lock when the configured port is free and when Windows/macOS cmdline checks show the lock owner is not actually a gateway.
  - file: [src/infra/gateway-lock.test.ts:249](https://github.com/openclaw/openclaw/blob/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd/src/infra/gateway-lock.test.ts#L249)
