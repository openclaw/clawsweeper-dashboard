- `impact:auth-provider`: The issue is about routing `image_generate` through configured provider/model credentials and provider ids.

## Summary

Keep open: current main already supports the LiteLLM subcase, but arbitrary custom `models.providers.<id>` entries still do not become `image_generate` providers unless a plugin registers that image-generation capability.

## What This Changes

The issue requests that `image_generate` support custom or third-party OpenAI-compatible providers configured under `models.providers`, including LiteLLM-style provider paths.

## Best Possible Solution

Keep this open for a provider/API design decision: either add a generic OpenAI-compatible image-provider path for custom model providers, or document that users must use LiteLLM or a plugin-owned `registerImageGenerationProvider` implementation.

## Reproduction Assessment

Yes from source inspection: a custom provider/model ref becomes a candidate, then `generateImage` calls the image-provider registry and records `No image-generation provider registered for <provider>` when that provider id is not registered by a plugin capability. I did not run a live custom endpoint repro.

## Solution Assessment

Unclear; LiteLLM is already solved through a bundled plugin, while the custom provider path needs a maintainer decision between auto-detecting OpenAI-compatible image endpoints, adding explicit config, or requiring a plugin-owned image provider.
