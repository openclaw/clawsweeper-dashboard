## Summary

This is a valid, current behavior bug: the reported `spaces/<id>` value is a legitimate Google Chat target/session component, while active-memory currently has a colon-only guard before passing channel context into the embedded recall run, where slash-bearing values can reach bundled plugin public-surface dirName validation.

## What This Changes

Active-memory should not fail its recall sub-agent for Google Chat sessions whose conversation id contains `spaces/<spaceId>`.

## Best Possible Solution

Keep the public-surface dirName guard intact and fix active-memory to pass only a runnable channel/plugin id, such as `googlechat`, into embedded recall runs while treating `spaces/<id>` only as conversation identity.

## Reproduction Assessment

Yes, source-backed rather than live: a Google Chat session with `spaces/<id>` can feed a slash-bearing channel/conversation value into active-memory’s embedded recall context, and the public-surface loader throws the reported dirName error on slashes.

## Solution Assessment

Yes: the narrow maintainable fix is in active-memory channel-context resolution, not in relaxing bundled plugin path validation or changing Google Chat’s documented `spaces/<id>` identifiers.
