## Summary

Keep open: the reported Codex text-verbosity bug is real, but this patch still makes canonical `openai/*` params override explicit `openai-codex/*` PI params when both are configured, and the supplied proof does not exercise the live `/status` or runtime-param route.

## What This Changes

This PR changes `resolveExtraParams` to consult canonical `openai/*` model params for `openai-codex` runtime sessions and adds resolver/status tests for text verbosity.

## Best Possible Solution

Use canonical `openai/*` params as the Codex-runtime fallback only when explicit `openai-codex/*` PI params are absent, add both-configured regression coverage, and require redacted real-route proof before merge.

## Reproduction Assessment

Yes, as a source-level reproduction. Current main resolves params by active provider/model, so an `openai-codex` runtime session misses canonical `openai/gpt-5.5` params and then applies the GPT-5 low verbosity default.

## Solution Assessment

No, not as written. Falling back to canonical `openai/*` matches the documented route, but making it win over explicit `openai-codex/*` PI params is not the safest compatibility-preserving fix.
