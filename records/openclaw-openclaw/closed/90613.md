## Summary

Keep open: this targets a real Google Vertex ADC regression, but the new generic fallback is too broad and can persist raw setup-provider API key results into generated models.json instead of only recognized non-secret auth markers.

## What This Changes

The PR changes missing provider API-key synthesis to fall back from env-var-name resolution to the raw resolved env/auth value, and adds a Google Vertex ADC models.json planning regression test.

## Best Possible Solution

Keep the targeted ADC repair, but only persist recognized non-secret auth markers on the fallback path and add a regression test proving plain setup-provider secrets are not serialized.

## Reproduction Assessment

Yes, source-level: current main drops ADC auth evidence because resolveEnvApiKeyVarName ignores the gcloud adc source, and filterWritableProviders then removes model providers without apiKey. I did not run a live Vertex ADC turn in this read-only review.

## Solution Assessment

No: the helper is the right neighborhood, but the fallback should be narrowed to recognized non-secret markers instead of accepting any raw resolveEnvApiKey result. The guarded direction described in https://github.com/openclaw/openclaw/pull/90609 is closer if validated.
