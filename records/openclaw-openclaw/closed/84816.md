## Label Justifications

- `P1`: The PR targets an urgent real user-facing Discord `/models` timeout path, but the current branch still leaves that timeout reachable.
- `merge-risk: 🚨 compatibility`: The branch changes provider visibility freshness from per-call computation to process-level prepared state, which can affect existing model-listing behavior unless upgrade and invalidation semantics are proven.
- `merge-risk: 🚨 auth-provider`: Provider auth availability is the cached fact, so stale or incorrectly scoped answers can misreport which model providers are usable.
- `merge-risk: 🚨 availability`: The remaining TTL and warm-window paths can still block the gateway event loop long enough for Discord interactions to fail.

## Summary

Keep open: the prewarm direction is credible and the post-warm proof is useful, but the branch still lets user-facing model listing fall back to synchronous provider discovery after the 10s prepared-state TTL and during startup/reload warm windows, so Discord `/models` can still time out.

## What This Changes

The PR adds a warmed provider-auth snapshot for model-listing auth checks, invalidates/rewarms it on startup, hot reload, and logout, and adds focused tests plus a changelog entry.

## Best Possible Solution

Keep the prewarm optimization, but make user-facing `/models` use a prepared or nonblocking pending/stale-while-refresh auth state consistently, with explicit invalidation for auth/config/plugin mutations and regression tests for expiry, startup, reload, logout, workspace, and agent scope.

## Reproduction Assessment

Yes for the PR blockers by source inspection: the branch test advances to 10,001 ms and expects fallback to compute, while startup and hot reload schedule the warmer after user-facing readiness or after clearing state. I did not run tests because this was a read-only review.

## Solution Assessment

No. The prepared-auth direction is the right shape, but this implementation still reintroduces request-time provider discovery on expiry and warm windows; a nonblocking refresh/invalidation lifecycle is the narrower maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep expired auth snapshots off the synchronous path:** `src/agents/model-provider-auth.ts:75-77`
  - body: The prepared auth state is only used for 10 seconds. After that, `matchesWarmedScope` is false and each `/models` build falls back to the same synchronous provider auth discovery that the PR measured at about 20 seconds, so a long-running Discord picker can still miss the 3s ack window. Refresh in the background or keep a safe stale snapshot instead of returning to request-time discovery.
  - confidence: 0.96
- **[P2] Avoid accepting picker traffic with no auth snapshot:** `src/gateway/server-startup-post-attach.ts:1021-1028`
  - body: This warmer starts only after `gateway ready` and runs fire-and-forget. During the PR's reported ~49s warm window, `currentProviderAuthState` is still null, so the first `/models` interaction after startup can still take the old slow path and fail in Discord. Either readiness or the request path needs a usable prepared/pending state before user-facing pickers can depend on this fix.
  - confidence: 0.9
- **[P2] Do not clear hot reload state before a replacement is usable:** `src/gateway/server-reload-handlers.ts:315`
  - body: Every hot reload now clears the prepared auth state up front and only schedules the rewarm near the end. Requests that arrive during that gap fall back to synchronous auth discovery, and unrelated hot reloads also create the gap because the clear is unconditional. Keep the previous matching snapshot while refreshing, or install a nonblocking pending state, then swap when the new answer is ready.
  - confidence: 0.88
