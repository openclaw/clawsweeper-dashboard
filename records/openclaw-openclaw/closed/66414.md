## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Use live state for attachment read counts:** `ui/src/ui/views/chat.ts:597-601`
  - body: `updateAttachmentReadPending()` derives `next` from render-time `props.attachmentReadPending`. With multiple FileReader callbacks, updates can apply the same stale base, so the send gate can reopen while reads are still pending or stay blocked incorrectly, leaving the image-drop race in place.
  - confidence: 0.91
- **[P1] Merge loaded attachments against current state:** `ui/src/ui/views/chat.ts:669`
  - body: The FileReader load handlers still append to `props.attachments` captured when the handler was created. If a user pastes, selects, drops, or removes attachments while reads are pending, the later callback can overwrite newer state and drop images before send.
  - confidence: 0.87
- **[P2] Let override commands bypass the attachment gate:** `ui/src/ui/app-chat.ts:276`
  - body: The new early return runs before command handling and does not check `messageOverride`, so UI actions such as `handleSendChat("/compact")` become silent no-ops while a file read is pending even though they do not send attachments. Limit the gate to composer sends that would include pending attachments.
  - confidence: 0.82
- **[P3] Expose pending processing in the send button label:** `ui/src/ui/views/chat.ts:1693`
  - body: When the send button is disabled because an attachment is processing, the title changes but the accessible name still says `Queue message` or `Send message`. Screen-reader users get a disabled send control without the reason, so the `aria-label` should reflect the pending attachment state too.
  - confidence: 0.78
