## Summary

Keep open: the PR is still necessary because current main and v2026.6.9 still linkify email-like rich Markdown unless `skipEntityDetection` is already set, while Telegram documents `skip_entity_detection` for rich-message email auto-detection. I found no blocking code or security findings; required CI should be refreshed because the visible failures point outside this Telegram diff.

## What This Changes

Adds a shared Telegram rich-message email-token guard that disables rich entity detection for email-containing rich payloads and adds send, draft, and reply-delivery regression tests.

## Best Possible Solution

Land the centralized rich-message guard after required checks are green on current main, preserving explicit Markdown links and existing link-preview opt-out behavior.

## Reproduction Assessment

Yes for source-level reproduction: current main and v2026.6.9 linkify rich Markdown unless `skipEntityDetection` is already true, and the affected Telegram rich send paths all route through that builder. I did not rerun the live Telegram failure in this read-only review.

## Solution Assessment

Yes, this is the best fix shape: the skip decision is placed in the shared rich-message builder used by durable sends, edits, drafts, and reply delivery instead of patching only one caller. The main tradeoff is the intentional loss of automatic email linkification for email-containing rich payloads.
