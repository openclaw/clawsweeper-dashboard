## Summary

Current main now contains a targeted fix for the reported Windows Telegram startup failure class: channel and account startup fanout is capped, Bonjour self-probe races are recovered instead of crashing, and the Unreleased changelog explicitly ties that fix to this issue. The fix is on main only and is not in the latest provided release v2026.4.29.

## What This Changes

Issue reports Windows Telegram gateway unusability in v2026.4.24 through v2026.4.29, with event-loop saturation, Telegram polling stalls, Bonjour crashes, model fallback latency, and a Windows ESM loader regression.

## Best Possible Solution

Keep the generic main-branch fix: cap startup pressure in the gateway and recover Bonjour self-probe races without adding Telegram-specific startup policy to core.

## Reproduction Assessment

Yes for the shipped regression: the issue provides a concrete v2026.4.29 Windows 11, Node 22, multi-Telegram setup, and two later Windows confirmations. I did not need a live Windows run to verify the main-branch implementation because the fix is directly present in source, tests, and changelog provenance.

## Solution Assessment

Yes. The current fix is the narrowest maintainable direction because it addresses generic gateway startup fanout and the exact Bonjour ciao crash signature instead of hard-coding a Telegram-only delay policy.
