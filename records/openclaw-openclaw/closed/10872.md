## Summary

Current main already consumes iMessage reply context when the imsg payload provides resolved reply fields, and the current upstream imsg message schema now exposes the quoted body and sender fields the monitor needs.

## What This Changes

The issue requests inbound iMessage inline replies to include quoted message context in the agent-visible message.

## Best Possible Solution

Keep the existing current-main reply-context forwarding path and rely on a current imsg build for resolved quote body/sender fields; track any remaining GUID-to-ReplyToId mapping separately if it proves necessary.

## Reproduction Assessment

No high-confidence current-main failure path remains for the supported setup: current imsg now emits resolved quote body/sender fields, and current OpenClaw forwards those fields into agent-visible reply context. I did not run a live macOS iMessage flow.

## Solution Assessment

Yes. The lowest-maintenance solution is to keep OpenClaw's existing resolved-field handling and use a current imsg build; a direct chat.db fallback is no longer the best first step unless fresh evidence shows the current imsg contract is insufficient.
