## Summary

Current main already threads Telegram inbound media downloads through the resolved runtime token and shared Telegram transport, with matching regression coverage; the same relevant implementation is present in released v2026.4.23.

## Evidence

- **Media path uses resolved bot token from runtime:** `registerTelegramHandlers` builds Telegram media runtime options from `opts.token` and the already-resolved `telegramTransport`, so inbound media handling inherits the bot instance token instead of rereading raw channel config.
  - file: [extensions/telegram/src/bot-handlers.runtime.ts:126](https://github.com/openclaw/openclaw/blob/7875092f4ddc6f2dbe218bea8506c9ac1da87c39/extensions/telegram/src/bot-handlers.runtime.ts#L126)
  - sha: [7875092f4ddc](https://github.com/openclaw/openclaw/commit/7875092f4ddc6f2dbe218bea8506c9ac1da87c39)
- **Media runtime keeps caller token, not config token text:** `resolveTelegramMediaRuntimeOptions` returns `token: params.token` and only reads account config for `apiRoot`, trusted local roots, and network flags.
  - file: [extensions/telegram/src/accounts.ts:107](https://github.com/openclaw/openclaw/blob/7875092f4ddc6f2dbe218bea8506c9ac1da87c39/extensions/telegram/src/accounts.ts#L107)
  - sha: [7875092f4ddc](https://github.com/openclaw/openclaw/commit/7875092f4ddc6f2dbe218bea8506c9ac1da87c39)
- **File downloads reuse Telegram transport policy:** `downloadAndSaveTelegramFile` builds the download URL from `params.token` and fetches with `transport.sourceFetch` plus `dispatcherAttempts`, preserving the same direct/env-proxy/explicit-proxy transport behavior used for Telegram API calls.
  - file: [extensions/telegram/src/bot/delivery.resolve-media.ts:232](https://github.com/openclaw/openclaw/blob/7875092f4ddc6f2dbe218bea8506c9ac1da87c39/extensions/telegram/src/bot/delivery.resolve-media.ts#L232)
  - sha: [7875092f4ddc](https://github.com/openclaw/openclaw/commit/7875092f4ddc6f2dbe218bea8506c9ac1da87c39)
- **Regression test covers resolved URL construction:** The inbound media e2e test asserts the download path uses `https://api.telegram.org/file/bottok/...`, proving the media URL is built from the resolved runtime token used by the test harness rather than a raw config placeholder.
  - file: [extensions/telegram/src/bot.media.downloads-media-file-path-no-file-download.e2e.test.ts:43](https://github.com/openclaw/openclaw/blob/7875092f4ddc6f2dbe218bea8506c9ac1da87c39/extensions/telegram/src/bot.media.downloads-media-file-path-no-file-download.e2e.test.ts#L43)
  - sha: [7875092f4ddc](https://github.com/openclaw/openclaw/commit/7875092f4ddc6f2dbe218bea8506c9ac1da87c39)
- **Regression test covers proxy fetch reuse:** A dedicated inbound media test asserts `proxyFetch` is called instead of `globalThis.fetch` for the Telegram file download path.
  - file: [extensions/telegram/src/bot.media.downloads-media-file-path-no-file-download.e2e.test.ts:134](https://github.com/openclaw/openclaw/blob/7875092f4ddc6f2dbe218bea8506c9ac1da87c39/extensions/telegram/src/bot.media.downloads-media-file-path-no-file-download.e2e.test.ts#L134)

## Risks / Open Questions

- This was a read-only code/history review; I did not run a live Telegram repro against a proxy-only environment.
- The reported `openshell:resolve:env:...` marker is not a documented repo-native Telegram token format; if an external layer still injects that literal after startup, that would be outside the supported config contract rather than a remaining gap in the current Telegram media path.
