## Summary

Current main implements the root-cause fix for #69229. The task registry now normalizes task lifecycle timestamps on create, update, and restore, so caller-supplied startedAt values captured before registry insertion no longer produce startedAt < createdAt records. Regression tests cover create, update, and restore cases and assert zero inconsistent_timestamps findings. The changelog records this as #71871 under 2026.4.25 Unreleased, so no shipped release tag was identified.

## Best Possible Solution

Close #69229 as implemented on current main. Keep the registry-boundary timestamp normalization from #71871, and handle still-open duplicate PRs such as #60428, #69356, #69880, and #70293 as superseded separately.
