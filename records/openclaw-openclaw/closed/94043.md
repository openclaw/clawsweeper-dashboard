## Summary

Keep open: the resolver-side change is narrow and useful, but the PR still leaves the documented Bitwarden config without `BWS_SERVER_URL` in `passEnv`, so the self-hosted path can remain broken when users follow the docs.

## What This Changes

The PR forwards `BWS_SERVER_URL` from the bundled Bitwarden resolver to the spawned `bws` CLI, updates the Bitwarden secrets docs, and adds resolver subprocess env tests.

## Best Possible Solution

Keep the narrow resolver forwarding, add `BWS_SERVER_URL` to the documented Bitwarden `passEnv` path, and verify the provider-to-resolver-to-`bws` env handoff before merge.

## Reproduction Assessment

Yes, by source inspection: current `main` only passes `passEnv` variables to exec resolvers and the current Bitwarden resolver only forwards `BWS_ACCESS_TOKEN` and `PATH` to `bws`. I did not run a live self-hosted Bitwarden repro in this read-only review.

## Solution Assessment

No. The resolver forwarding is the right layer for the second hop, but the PR is incomplete until `BWS_SERVER_URL` is also present in the supported `passEnv` configuration path.
