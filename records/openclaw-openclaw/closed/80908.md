## Summary

This should close as a duplicate of the open reset-continuity feature request because current main still treats daily/idle rollover as a fresh session by design, while the requested carry-forward/replay behavior is already tracked more broadly in the canonical issue.

## What This Changes

Request to preserve or replay prior conversation state when a stable channel session key rolls over to a cold runtime after idle or daily expiry.

## Best Possible Solution

Keep the reset-continuity design centralized in the open canonical issue, and fold this report’s stable-key, cliSessionBindings, and archived-transcript evidence into that implementation discussion.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: create a stale direct-session entry that does not qualify for the provider-owned CLI implicit-daily exception, then initSessionState mints a new sessionId and archives the old transcript. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No separate fix should be tracked here; the best path is the canonical reset-continuity design in https://github.com/openclaw/openclaw/issues/58818. Any implementation should explicitly decide how much history to carry, how to handle CLI bindings, and how to preserve explicit clean-reset semantics.
