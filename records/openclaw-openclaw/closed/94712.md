Full review comments:

- **[P1] Gate probe fallback on the original error:** `extensions/telegram/src/fetch.ts:482-483`
  - body: This guard only stops fallback inside `resolveTelegramTransport.fetch`; in the reported probe path, `probeTelegram` still catches that thrown error and calls `transport.forceFallback("probe timeout/network error")`, which can produce the same codes=none pinned-IP fallback log. Pass the original error into that decision or let the probe ask the same local-socket classifier before promotion.
  - confidence: 0.91
- **[P1] Preserve retryable numeric errno fallback:** `extensions/telegram/src/fetch.ts:434-436`
  - body: `collectErrorCodes` now adds every numeric `errno` to `ctx.codes`, but the fallback predicate still recognizes symbolic retry codes. Numeric-only retryable failures such as ENETUNREACH-shaped errors will make `ctx.codes.size > 0`, miss `FALLBACK_RETRY_ERROR_CODES`, and skip the fallback current main would take; normalize numbers to symbols or restrict numeric handling to local socket failures.
  - confidence: 0.9
- **[P2] Cover actual EADDRNOTAVAIL errno variants:** `extensions/telegram/src/fetch.ts:482-483`
  - body: The new guard only checks raw numeric string `99`, but the linked report also calls out macOS `49`, and a local Node 24 EADDRNOTAVAIL socket error reports `errno: -99`. If the original failure path lacks `.code`, numeric-only local socket failures can still fall through to the existing code-less `fetch failed` fallback branch.
  - confidence: 0.82
