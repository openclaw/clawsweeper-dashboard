## What This Changes

The PR changes gateway command scope gating so privileged commands enforce `GatewayClientScopes` from caller context instead of relying on resolved command channel metadata.

## Best Possible Solution

Land the caller-context scope gate after maintainer/security review, contributor-provided real gateway behavior proof, and the minor `/acp` denial-text cleanup.

## Reproduction Assessment

Yes from source inspection: create a gateway command context with `GatewayClientScopes` lacking `operator.admin` and `command.channel` set to an external channel, and current main's shared gate returns `null` before checking scopes. I did not run a live gateway flow in this read-only review.

## Solution Assessment

Yes, enforcing on the presence of `GatewayClientScopes` is the narrowest maintainable fix for this authorization gap. The implementation still needs the `/acp` denial copy aligned with the new condition and real behavior proof before merge.
