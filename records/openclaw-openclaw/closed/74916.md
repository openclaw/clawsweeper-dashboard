## Summary

Keep open. Current main still allows the disabled-config capability fallback to build bundled compat config and call the runtime loader, while this PR applies the narrow short-circuit and matching regression coverage; I found no discrete blocking bug in the proposed patch, but it still needs maintainer review and the normal changed gate before merge.

## What This Changes

The PR adds a `plugins.enabled: false` short-circuit to plugin capability provider resolution and updates fallback tests to assert disabled configs do not run bundled compat or runtime loader work.

## Best Possible Solution

Land the narrow capability-runtime guard that treats `plugins.enabled: false` as a hard stop before bundled capability compat or runtime snapshot loading, with regression coverage for both list and targeted provider fallback paths.

## Reproduction Assessment

Yes. The high-confidence reproduction is the current-main unit-level control flow where a disabled config still goes through bundled compat and a mocked fallback registry can return a bundled provider; the existing tests at the disabled cases currently only prove dependency repair is disabled, not that loading is skipped.

## Solution Assessment

Yes. The proposed short-circuit is the narrowest maintainable solution because it enforces the documented master-disable contract before active or fallback capability lookup, instead of relying on a compat helper that intentionally re-enables bundled plugin ids.
