## Summary

Keep open. Current `main` fixed the specific `openclaw gateway restart` LaunchAgent path in `src/daemon/launchd.ts`, but the broader macOS self-restart surface described by the issue is not fully closed out: `src/infra/restart.ts` still performs an in-process Darwin `launchctl kickstart -k` / `bootstrap` sequence with no detached handoff, and `src/auto-reply/reply/commands-session.ts` still routes one restart path through that function.

## Evidence

- **CLI LaunchAgent restart path is now kickstart-first with detached handoff:** `src/daemon/launchd.ts:625` now schedules `scheduleDetachedLaunchdRestartHandoff(...)` when the current process is the managed LaunchAgent, otherwise runs `launchctl enable`, `launchctl kickstart -k`, and only falls back to `bootstrap` when `kickstart` reports not-loaded. This matches the issue's requested behavior for `openclaw gateway restart`.
  - file: [src/daemon/launchd.ts:625](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/src/daemon/launchd.ts#L625)
  - command: `nl -ba src/daemon/launchd.ts | sed -n '625,685p'`
  - sha: [7ac35b4f6900](https://github.com/openclaw/openclaw/commit/7ac35b4f69009490180a35bf404638d4280a4865)
- **Tests cover the fixed launchd.ts behavior:** `src/daemon/launchd.test.ts:650` asserts restart avoids `bootout`, prefers `kickstart -k`, bootstraps only on not-loaded, and `src/daemon/launchd.test.ts:753` asserts detached helper handoff when invoked from the current LaunchAgent. `src/daemon/launchd.integration.e2e.test.ts:177` also checks restart keeps the service running with a new PID.
  - file: [src/daemon/launchd.test.ts:650](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/src/daemon/launchd.test.ts#L650)
