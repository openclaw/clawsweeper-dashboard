## Summary

Keep open. Current main still does not expose a shared normalized delivery-failure reason taxonomy: cron surfaces coarse delivery states plus free-form error text, while CLI/message delivery has a separate structured reason model. The related cron-warning work at https://github.com/openclaw/openclaw/pull/80139 is still open and only addresses the narrower execution-vs-delivery status split tracked by https://github.com/openclaw/openclaw/issues/49190, while https://github.com/openclaw/openclaw/issues/74095 is broader task taxonomy work rather than a complete replacement for this delivery-specific issue.

## What This Changes

The issue requests structured, normalized delivery failure reason codes across cron runs, messaging/channel delivery, status cards, and JSON output.

## Best Possible Solution

Define an additive canonical delivery outcome taxonomy shared by cron, Gateway/API output, Control UI, plugin hooks, and message delivery, while preserving existing coarse status fields for compatibility and mapping each reason to one operator next step.

## Reproduction Assessment

Yes at source level: current cron types, Gateway schemas, UI types, and plugin hooks still expose coarse delivery statuses plus error strings, while message delivery has a separate reason-coded JSON model. I did not run a live cron/channel scenario in this read-only review.

## Solution Assessment

Unclear as a concrete implementation: the requested direction is appropriate, but the best fix is a maintainer-chosen additive taxonomy and compatibility plan rather than a one-off cron or UI string mapping.
