## Summary

Keep open. Current main still has raw synthetic missing-tool-result text paths and heartbeat filtering preserves helper tool turns when a heartbeat produces visible output, so a heartbeat-context pollution bug remains plausible and is not fully covered by the broader durable transcript-repair trackers.

## What This Changes

The issue asks OpenClaw to record heartbeat/tool-result transcript repairs as structured diagnostics and keep raw synthetic repair text out of later heartbeat context.

## Best Possible Solution

Add a heartbeat-aware repair path that records bounded structured diagnostics for missing tool results and keeps raw synthetic repair text out of heartbeat-visible context without disabling strict-provider transcript repair globally.

## Reproduction Assessment

Yes, at source level. A heartbeat span with an assistant helper tool call, a synthetic missing-result toolResult, and later visible heartbeat output is retained by the heartbeat filter, while current repair paths can still generate the raw synthetic text.

## Solution Assessment

Unclear pending maintainer policy. The narrow fix should not disable synthetic repair globally; it should mark or diagnose repair artifacts and make heartbeat context assembly filter or fail them consistently.
