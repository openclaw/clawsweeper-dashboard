## Summary

Do not close. Current `main` has partially improved the March 2026 symptom cluster, but there is not strong evidence that the full report is resolved. `gateway status` is now connect-only by default and `gateway probe` now treats missing `operator.read` as degraded reachability, yet `openclaw status` still treats only `probe.ok === true` as reachable and post-connect detail-RPC timeouts still resolve as `ok: false`.

## Evidence

- **`gateway status` semantics changed on main:** Docs and the daemon probe path now say default `gateway status` proves service/connect/auth-at-handshake only, while `--require-rpc` upgrades to a read probe. That addresses part of the original contradiction, but it is a behavior change rather than proof the underlying bug is gone.
  - file: [docs/cli/gateway.md:185](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/docs/cli/gateway.md#L185)

  - sha: [ebd7f19a3b20](https://github.com/openclaw/openclaw/commit/ebd7f19a3b20001960474ec9ba9cc424e45f7b23)
- **Missing `operator.read` is now modeled as degraded-but-reachable for `gateway probe`:** The current gateway-status helpers treat a probe with connect latency plus `missing scope: ...` as reachable, and the regression test explicitly expects `ok: true`, `degraded: true`, and `scopeLimited: true` for that case.
  - file: [src/commands/gateway-status/helpers.ts:272](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/src/commands/gateway-status/helpers.ts#L272)

- **`openclaw status` still gates reachability on `probe.ok === true`:** The status scan snapshot still sets `gatewayReachable` only when `gatewayProbe?.ok === true`; scope-limited or other post-connect probe failures are not treated as reachable here.
