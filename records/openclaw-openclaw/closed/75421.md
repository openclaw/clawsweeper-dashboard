## Summary

Current main and the v2026.4.29 release already contain the shared embedded-runner guard for this bug: blank visible user prompts are skipped before provider submission, while runtime-only and media-only turns remain allowed. The reported 2026.4.26 behavior is covered by the merged #74168 fix plus the existing replay/session-file repair safety nets.

## What This Changes

The issue reports that model fallback after context overflow inserts empty user messages, causing ZAI/GLM providers to reject retries.

## Best Possible Solution

Keep the shared embedded-runner guard shipped in v2026.4.29, with replay and session-file repair as layered recovery. If this reappears on v2026.4.29 or newer, the next actionable report should include the session trajectory or sanitized JSONL around the fallback retry.

## Reproduction Assessment

Yes. The high-confidence source-level path is a blank non-runtime visible prompt with replay history and no images; current tests verify that this now records `blank_user_prompt` and does not call the provider prompt function.

## Solution Assessment

Yes. The current shared embedded-runner guard is the narrowest maintainable fix because it catches blank visible prompts across channels and providers before submission, without blocking runtime-only maintenance turns or media-only prompts.
