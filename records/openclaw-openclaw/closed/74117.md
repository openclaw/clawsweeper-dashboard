## Summary

Keep open. Current main still has the reported inbound-dispatch runtime plugin ensure path, and the provided GitHub context shows PR #74118 is open, unmerged, and linked with closing syntax, so this issue should wait for that implementation to be reviewed, landed, replaced, or closed.

## What This Changes

The issue reports that `ensureRuntimePluginsLoaded` rebuilds the runtime plugin registry on each inbound dispatch because its reduced load options do not match the gateway boot registry cache key.

## Best Possible Solution

The shipped end state should make inbound dispatch reuse the already-populated runtime plugin registry for this ensure-only path, with focused regression coverage proving that an active registry avoids a rebuild while preserving real reloads when no active registry exists.

## Reproduction Assessment

Yes. The current source gives a high-confidence reproduction path: inbound dispatch calls the ensure helper, the helper passes reduced explicit load options, and the resolver reloads when those options are not compatible with the gateway startup registry cache key.

## Solution Assessment

Unclear. The helper-level fast path proposed by PR #74118 is the narrowest fix for the hot path, but maintainers should decide whether the compatibility rule belongs in `ensureRuntimePluginsLoaded` or in the shared loader resolver.
