## Summary

Keep open: current main and shipped v2026.5.20 still reject per-session Talk realtime instructions/context, and the request is an additive protocol capability that needs maintainer agreement on prompt-policy semantics.

## What This Changes

The issue requests a bounded per-session instructions or context field for realtime `talk.client.create` and gateway-relay `talk.session.create` sessions.

## Best Possible Solution

Add an additive, bounded per-session context field if maintainers approve the trusted-caller model, append it after OpenClaw's base/configured realtime prompt, and update protocol docs plus schema/handler tests for both browser and gateway-relay paths.

## Reproduction Assessment

Yes, from source inspection: current schemas disallow extra properties and tests explicitly pin request-time instruction override rejection, so the reported request shape fails validation on current main. I did not run a live Gateway RPC because the read-only review has direct validator and runtime evidence.

## Solution Assessment

Unclear until maintainers decide the prompt-policy boundary. The safer implementation is a bounded additive context field appended after OpenClaw's base/configured realtime prompt, not replacing configured policy text.
