## Summary

Keep open. The report is fresh, concrete, and against the latest shipped release; current main contains related Telegram transport lifecycle hardening, bounded Undici pools, request timeouts, and polling rebuild logic, but there is still no source or test proof that normal Telegram polling reuses TCP connections or keeps TIME_WAIT stable under the reporter's scenario.

## What This Changes

The issue asks OpenClaw to stop Telegram long polling from creating new HTTPS/TIME_WAIT sockets every roughly 10 seconds and to reuse bounded keep-alive connections per bot account.

## Best Possible Solution

Keep the issue open and validate current main with live Telegram bots while correlating socket churn to Bot API methods. If client-side churn is confirmed, keep the fix inside the Telegram plugin transport or polling runner, preserve the existing dispatcher cleanup and fallback semantics, and add focused instrumentation or tests around dispatcher reuse rather than moving Telegram-specific policy into core.
