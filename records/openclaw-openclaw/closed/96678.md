## Summary

Close: current main and the latest release already handle Firecrawl non-ok HTTP responses in `postFirecrawlJson` before JSON parsing, so this PR adds a redundant second guard rather than fixing a reachable production path.

## What This Changes

The PR adds a `response.ok` check inside `readFirecrawlJsonResponse` and throws with the HTTP status and body before delegating to the shared JSON parser.

## Best Possible Solution

Keep the shipped `postFirecrawlJson`-level guard as the single Firecrawl HTTP-error path and close this redundant branch; future response-ok work should target providers whose current wrappers still pass non-ok responses into JSON parsing.

## Reproduction Assessment

No. Current main and `v2026.6.10` already throw from `postFirecrawlJson` on non-ok Firecrawl responses before `readFirecrawlJsonResponse` runs, and the existing test suite covers a 400 JSON error body through that wrapper.

## Solution Assessment

No. The best solution is the existing wrapper-level guard because it covers both search and scrape, preserves one error-formatting path, and keeps upstream details bounded and wrapped.
