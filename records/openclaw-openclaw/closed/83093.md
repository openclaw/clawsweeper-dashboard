## Summary

Keep open: current main still has the reported Codex app-server gap, and two open closing PRs already target this issue. The issue should stay open until one fix PR is reviewed and merged.

## What This Changes

The issue requests that declined or blocked Codex app-server native tools populate attempt-level `lastToolError` so external channels send a visible error reply instead of silently dropping the turn.

## Best Possible Solution

Review and land one narrow fix PR that records non-success Codex-native tool completions as `lastToolError` and proves the projector-to-payload behavior with focused regressions.

## Reproduction Assessment

Yes, by source inspection. A declined `commandExecution` is normalized to a blocked native tool, `buildResult` does not carry it into `lastToolError`, and an aborted empty turn skips the incomplete-turn fallback so the payload builder can return no visible reply.

## Solution Assessment

Yes. The narrow maintainable fix is to bridge blocked or failed native Codex tool completions into the existing `lastToolError` payload path, without adding Telegram-specific behavior or new policy.
