## Summary

Keep open: the underlying channel-local startup degradation idea is still not on current main, but this PR is not mergeable as written because recovery bypasses the current reload restart/rollback path, recoveryConfig can replay stale channel state, and the external PR still lacks real behavior proof.

## What This Changes

This PR adds gateway startup degradation for missing env-backed channel SecretRefs, carries a recoveryConfig through secrets.reload, adds merge-patch support, and extends gateway reload tests.

## Best Possible Solution

Rework on current main by adding an explicit startup channel-degradation path in the current startup secret activator, preserving fail-fast behavior for non-availability errors, recovering through the existing secrets.reload restart/rollback path, and documenting the revised startup SecretRef contract.

## Reproduction Assessment

Yes, at source level for the current all-or-nothing behavior: current main prepares the full secrets snapshot and throws the normalized fatal startup error on startup activation failure, and the focused startup secret test asserts that contract. I did not run tests in this read-only review.

## Solution Assessment

No. The direction is useful, but this PR is not the best implementation because its recovery path bypasses the current channel restart/rollback seam and can replay stale recoveryConfig over explicit channel-disable writes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Route recovery through the channel restart path:** `src/gateway/server.impl.ts:1351-1354`
  - body: After degraded startup, the affected channel was never started because the active snapshot has it disabled. This reload branch only activates recovered secrets/config and returns, so config can say the channel is enabled while the runtime remains offline until a later monitor or restart; route recovery through the existing reload plan.
  - confidence: 0.88
- **[P2] Preserve explicit disabled-channel recovery edits:** `src/secrets/runtime.ts:310-311`
  - body: The recoveryConfig forwarding diffs from the degraded sourceConfig, so an operator write of `channels.<id>.enabled=false` while the secret is missing can produce no delta. The old recoveryConfig can keep `enabled=true`, letting a later `secrets.reload` re-enable a channel the operator explicitly disabled.
  - confidence: 0.82
