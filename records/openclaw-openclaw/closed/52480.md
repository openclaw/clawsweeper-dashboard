## Summary

Keep open. Current main still retries Discord REST calls only for the local `RateLimitError`, so the central transient-error behavior has not landed. The PR remains a useful implementation candidate for open #52396, but the current patch has blocking review findings around an undeclared test import and retrying ambiguous transport failures through a shared runner used by non-idempotent Discord send/thread POSTs.

## What This Changes

The PR adds a Discord transient-error retry predicate to `createDiscordRetryRunner` and a new `extensions/discord/src/retry.test.ts` covering rate-limit, transient, permanent, and exhausted retry cases.

## Best Possible Solution

Update this PR or replace it with a focused Discord retry policy that uses the current local `RateLimitError`, preserves Discord `retry_after` handling, retries transient failures only on idempotent lookup/edit/delete/read operations, and uses a separate send-safe predicate or runner for message, component, poll, sticker, voice, reply, and thread-creation POSTs. Tests should cover both retry and no-retry cases for those boundaries.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Use the current Discord RateLimitError in the test:** `extensions/discord/src/retry.test.ts:1`
  - body: The new test imports `RateLimitError` from `@buape/carbon`, but that package is not declared by the Discord plugin or current workspace lockfile, and production retry code checks the local class from `./internal/discord.js`. This can fail module resolution and would not validate the same class the retry predicate uses.
  - confidence: 0.93
- **[P2] Scope transient retries away from send POSTs:** `extensions/discord/src/retry.ts:25-27`
  - body: This widens the shared `createDiscordRetryRunner` used by `createDiscordClient().request`, which current send paths use for `createChannelMessage` and forum thread creation. A timeout or reset after Discord accepts the first write can replay the POST and create duplicate visible messages or threads, so transient retries need idempotent call-site scoping or a send-safe runner.
  - confidence: 0.9
