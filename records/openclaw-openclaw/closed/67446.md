## Summary

Keep open. Current main still classifies tool-result errors only from normalized `details.status`, and bundled MCP materialization still only sets `details.status` for `isError === true`, so status-less MCP content errors are not implemented on main. The PR is still a plausible focused fix, but the latest PR behavior needs maintainer judgment because it trusts explicit `isError: false` and only leaves the content-prefix fallback reachable when `isError` is omitted.

## Best Possible Solution

Keep this PR open for maintainer review. Decide whether OpenClaw should strictly trust MCP `isError` or provide a compatibility fallback for non-conformant servers; then either land a revised agent-scoped implementation with tests for `isError: true`, `isError: false`, omitted `isError`, `Error:` false positives, and `after_tool_call.error`, or close the PR after explicitly rejecting content-prefix error inference.
