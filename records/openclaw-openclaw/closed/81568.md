## Summary

Close as duplicate: the same JSON stdout corruption bug is already tracked by the earlier open repair PR and canonical issue, and this PR adds no unique implementation, tests, or real behavior proof beyond that path.

## What This Changes

The branch adds an early `routeLogsToStderr()` call in `runCli` before lazy plugin CLI registration whenever `hasJsonOutputFlag(parseArgv)` is true.

## Best Possible Solution

Keep the canonical issue and earlier PR as the repair path, then scope stderr routing to the plugin-registration window or true JSON-output mode with focused regression proof.

## Reproduction Assessment

Yes. Source inspection shows plugin CLI registration runs before `parseAsync`/preAction, while plugin loader debug lines go to stdout until `forceConsoleToStderr` is set; the linked issue gives the concrete `openclaw memory search ... --json` repro.

## Solution Assessment

No. The intended direction is right, but this duplicate PR uses broad argv-only detection and lacks unique proof; the safer fix should be finished in the canonical open PR with scoped routing and tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Scope early stderr routing to real JSON output:** `src/cli/run-main.ts:732-735`
  - body: This flips `routeLogsToStderr()` for any argv token that looks like `--json` before Commander has parsed the selected plugin command, and leaves the flag on for the rest of the process. Existing JSON-mode detection respects command semantics and the `--` terminator, so a text-mode plugin invocation such as `openclaw memory search -- --json` would have its normal `defaultRuntime.log` output forced to stderr. Scope the early flip to the plugin-registration window or to true JSON-output mode.
  - confidence: 0.87
