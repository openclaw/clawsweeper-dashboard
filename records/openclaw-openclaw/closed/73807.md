## Summary

Keep open: the PR targets a valid source-level OpenAI Codex OAuth stale-`expires` path, but the patch still bypasses current forced-refresh/capped-recheck contracts, skips identity enrichment on the shortcut, and lacks real behavior proof. The merged OAuth fallback work mitigates related failures, but it does not supersede this PR's pre-refresh JWT-exp shortcut.

## What This Changes

The PR adds an OpenAI Codex OAuth access-token JWT `exp` shortcut plus provider tests so stale stored `expires` values can avoid an HTTP refresh when the cached JWT is still valid.

## Best Possible Solution

Rebase or replace this with a provider-local fix that only shortcuts non-forced stale-expiry reads, enforces an authoritative capped recheck, preserves Codex identity metadata, and includes real OAuth runtime proof.

## Reproduction Assessment

Yes, at source level. Seed an OpenAI Codex OAuth credential with a stale stored `expires` value and a long-lived JWT access token: current main reaches provider refresh because the shared usability gate reads only stored `expires`.

## Solution Assessment

No. A provider-local JWT check is plausible, but this PR's current early return cannot honor forced refreshes or its capped recheck promise and skips current identity enrichment.
