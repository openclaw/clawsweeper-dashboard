## Summary

The request is coherent and useful, but current main treats memory behavior and custom compaction policy as plugin-owned extension points rather than OpenClaw core work, so this belongs in a ClawHub memory/compaction plugin.

## Close Comment

Closing this as a Codex review because it fits OpenClaw’s plugin boundary rather than core scope. Current main explicitly says core should stay lean and that memory is a special plugin slot (`VISION.md:52`), the SDK already exposes memory-specific extension seams like `registerMemoryCapability()` and `registerMemoryFlushPlan()` (`docs/plugins/sdk-overview.md:195`), and compaction summarization is already pluggable through `registerCompactionProvider()` (`docs/concepts/compaction.md:75`). OpenClaw also already has the core persistence primitives this proposal builds on: `MEMORY.md`, daily `memory/YYYY-MM-DD.md` notes, automatic pre-compaction memory flush, and dreaming (`docs/concepts/memory.md:15`, `docs/concepts/memory.md:103`, `docs/concepts/memory.md:115`). Given that, a tiered daily/weekly/monthly/yearly summarization system is better pursued as a memory/compaction plugin published through ClawHub instead of expanding OpenClaw core (`docs/plugins/building-plugins.md:16`).
