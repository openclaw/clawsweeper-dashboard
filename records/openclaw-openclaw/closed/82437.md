## Summary

Keep open: current main still has a source-backed path where official OpenAI agent model selection resolves to the Codex harness, an unavailable forced harness throws the reported error, `/v1/chat/completions` maps that ordinary error to a generic 500, and readiness does not validate harness availability.

## What This Changes

The issue reports that v2026.5.12 can pass gateway readiness but return a generic 500 from `/v1/chat/completions` when the selected `codex` agent harness is not registered.

## Best Possible Solution

Align release/package Codex plugin availability, runtime selection, readiness, and OpenAI-compatible errors so the same config either serves through Codex/PI or fails before ready with actionable diagnostics.

## Reproduction Assessment

Yes, source-reproducible: current source still routes official `openai/*` agent models to Codex by default, throws the reported unregistered-harness error when the plugin is absent, maps that error to a generic 500, and readiness does not validate the harness. I did not run the live image.

## Solution Assessment

Unclear: the report lists several acceptable outcomes, and the narrowest maintainable fix depends on whether maintainers want release images to include/repair Codex, readiness to fail on missing harnesses, or this deployment path to use provider/model-scoped PI.
