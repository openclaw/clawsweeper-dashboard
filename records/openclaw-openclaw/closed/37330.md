## Summary

Current `main` does not implement this Slack/group-chat feature. Routing still picks a single agent per inbound conversation by default, and the existing multi-agent `broadcast` path is documented and coded as channel-specific rather than a generic per-agent reply-stagger for Slack.

## Evidence

- **Single-agent routing remains the default:** The multi-agent docs still describe inbound routing as deterministic single-agent selection: shared groups should be bound to one agent or use broadcast groups, and if multiple bindings match the first one wins.
  - file: [docs/concepts/multi-agent.md:225](https://github.com/openclaw/openclaw/blob/7d471837369885c12d540d6bafa319ff41129b89/docs/concepts/multi-agent.md#L225)
  - command: `nl -ba docs/concepts/multi-agent.md | sed -n '220,242p'`
  - sha: [7d4718373698](https://github.com/openclaw/openclaw/commit/7d471837369885c12d540d6bafa319ff41129b89)
- **Core route resolver returns one chosen agent:** `resolveAgentRoute()` builds and returns one `{ agentId, ... }` route via `choose(...)`; there is no generic fan-out or per-agent delay path in the shared routing layer.
  - file: [src/routing/resolve-route.ts:657](https://github.com/openclaw/openclaw/blob/7d471837369885c12d540d6bafa319ff41129b89/src/routing/resolve-route.ts#L657)
