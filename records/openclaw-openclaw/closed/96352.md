## Summary

Keep open: the both-shape preservation is the right direction for the cron session-prune bug, but this head also uses the expanded preserve-segment set as the reported `runningCronJobs` count, so non-slug running jobs can be overreported in maintenance output.

## What This Changes

The branch preserves both raw lowercased and slugified cron job segments during `tasks maintenance` session-registry pruning, with a regression test for both persisted session-key shapes.

## Best Possible Solution

Keep the both-shape preservation, but split the preserve-segment set from the reported running cron job count and assert both behaviors in the regression test.

## Reproduction Assessment

Yes. Source inspection shows current main lowercases running job ids while main-session cron-run keys slugify the job segment, so a live stale `agent:main:cron:daily-report:run:*` row for job id `Daily Report` can be pruned.

## Solution Assessment

No, not yet. Preserving both raw and slug segments is the right fix shape, but the implementation must keep the public running-job count separate from the expanded preserve keys.
