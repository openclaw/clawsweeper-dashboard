## Summary

Keep this PR open. Current main still drops Discord upload MIME types in the standard send, component send, and native interaction reply paths, and WebP still goes through the shared image optimizer. The PR remains useful, but it needs a rebase and a narrower WebP policy before merge because the proposed shared bypass can regress WhatsApp/shared media behavior and weakens the current validation/re-encode path.

## Best Possible Solution

Keep this PR open for maintainer review or supersede it only after a canonical fix lands. The best path is to rebase it, preserve `media.contentType` in all Discord Blob upload paths including standard sends, component sends, and native interaction replies, and handle WebP preservation deliberately through a Discord-specific option or a shared media API with signature validation, pixel-limit enforcement, metadata policy, and regression tests. If shared media semantics intentionally change, update the WhatsApp expectations and consolidate the overlap with #41265 and #41419 after the chosen fix lands.

## Risks / Open Questions

- Merging the PR as-is would change shared `loadWebMedia` behavior for all callers, not only Discord; WhatsApp WebP auto-replies that currently compress to JPEG under the channel cap could start failing with over-cap errors.
- Raw WebP preservation needs an explicit validation and privacy policy: signature-confirmed WebP detection, pixel/dimension limits, malformed input rejection, and metadata stripping or deliberate metadata preservation.
- The PR is outdated against current main because the media implementation moved from the WhatsApp plugin facade to `src/media/web-media.ts`; it needs a rebase before code review or CI can be meaningful.
- There are multiple open candidate PRs for the same linked bug, so maintainers should choose or compose a canonical fix rather than close this PR as a duplicate prematurely.
