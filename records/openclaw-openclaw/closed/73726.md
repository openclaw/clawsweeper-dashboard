## Summary

Keep open. Current main still sends benign Telegram delete-message 400s through the shared log-and-rethrow path, while nearby Telegram helpers already have fail-soft precedents and the known fix PRs are closed unmerged.

## What This Changes

The issue asks the Telegram plugin to treat already-gone or not-deletable `deleteMessage` 400s as soft no-op results with a warning.

## Best Possible Solution

Patch only the Telegram plugin delete action path: classify benign delete no-op errors, suppress their error-level log, return a structured warning with `deleted: false`, keep non-benign failures hard, and add regression tests plus a changelog entry.

## Reproduction Assessment

Yes. A high-confidence reproduction path exists by injecting an `api.deleteMessage` mock that rejects with a grammY-style 400 message and observing the current `requestWithDiag` log/rethrow path; the reported CLI command reaches the same action branch.

## Solution Assessment

Yes. A narrow Telegram helper/action-runtime change is the best maintainable fix; changing shared retry or API logging semantics would risk hiding unrelated Telegram failures.

## Repair Work Prompt

Fix the Telegram delete-message fail-soft bug reported at https://github.com/openclaw/openclaw/issues/73726. In `extensions/telegram/src/send.ts`, add a narrow helper for benign Telegram delete no-op errors such as `message to delete not found`, `message can't be deleted`, `MESSAGE_ID_INVALID`, and `MESSAGE_DELETE_FORBIDDEN`; make `deleteMessageTelegram` catch only those cases, suppress the shared error-level API log for them, return `{ ok: false, warning }`, and continue throwing non-benign failures. In `extensions/telegram/src/action-runtime.ts`, propagate the soft result as a structured delete result with `deleted: false` and the warning, mirroring the existing reaction warning pattern. Add focused regression tests for all known benign strings, non-benign rethrow, and action-runtime warning propagation. Add a single-line `CHANGELOG.md` Fixes entry under Unreleased. Do not change shared retry/logging semantics globally, do not soften auth/network failures, and do not move this into core.
