## Label Justifications

- `impact:crash-loop`: The issue is explicitly about SIGABRT killing the Gateway process during shutdown.

## Summary

Current main and the latest shipped release already address the OpenClaw-side failure mode: local node-llama-cpp embeddings are isolated in a worker, Gateway startup no longer initializes QMD by default, and QMD/native aborts after valid output are handled without taking down the Gateway.

## What This Changes

The issue reports a macOS Apple Silicon Gateway shutdown SIGABRT from node-llama-cpp Metal cleanup and asks OpenClaw to avoid loading the native addon at startup.

## Best Possible Solution

Keep the shipped worker-isolation, lazy-load, and QMD salvage behavior; any remaining crash on v2026.5.27 or newer needs a new report with exact config and logs showing a remaining in-Gateway node-llama-cpp load path.

## Reproduction Assessment

No; I did not run an Apple Silicon Metal shutdown repro. The reported stack is plausible, but current source and release evidence show the Gateway no longer owns the node-llama-cpp native lifecycle for local embeddings and no longer initializes QMD on normal startup.

## Solution Assessment

Yes; the best OpenClaw-side fix is to avoid loading the native addon in the Gateway process and to tolerate known QMD/native abort-after-output paths. The underlying ggml/node-llama-cpp destructor issue still belongs upstream, but OpenClaw has already reduced its blast radius.
