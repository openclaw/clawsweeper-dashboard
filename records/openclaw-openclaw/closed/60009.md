## Summary

Current `main` does not show a core path that truncates `~/.openclaw/identity/device.json`. OpenClaw generates its own full Ed25519 device identity, and the reported `IDENTITY_SIGNATURE_INVALID` symptom matches a stale or incorrect WebSocket handshake that skipped the current `connect.challenge` nonce/payload requirements.

## Evidence

- **Device identity generation is internal and writes full PEM fields:** `loadOrCreateDeviceIdentity()` resolves `~/.openclaw/identity/device.json`, generates an `ed25519` keypair, exports full `spki`/`pkcs8` PEM strings, and stores them as `publicKeyPem` / `privateKeyPem`. There is no truncation logic in this path.
  - file: [src/infra/device-identity.ts:57](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/infra/device-identity.ts#L57)
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **Official client waits for challenge and signs the nonce-bound payload:** The gateway client refuses to send `connect` until it has a nonce, then builds the v3 device-auth payload and signs it with the stored private key before sending `device.nonce` and the raw public key.
  - file: [src/gateway/client.ts:440](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/gateway/client.ts#L440)
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **Server and docs map this exact failure to handshake/payload mismatch:** The server rejects missing or mismatched nonces and invalid signatures during connect. The protocol docs and troubleshooting guide explicitly say `device signature invalid` / `DEVICE_AUTH_SIGNATURE_INVALID` means the client signed the wrong payload for the current handshake, and `device nonce required` / `device nonce mismatch` mean the client is not following `connect.challenge` + `device.nonce`.
  - file: [src/gateway/server/ws-connection/message-handler.ts:715](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/gateway/server/ws-connection/message-handler.ts#L715)
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **Tests cover successful signing and expected failure modes:** Device-identity tests verify that generated identities sign and verify correctly, and gateway auth tests require reading `connect.challenge` first and assert `device signature invalid` / nonce-specific errors for malformed handshakes.
  - file: [src/infra/device-identity.test.ts:22](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/infra/device-identity.test.ts#L22)
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **No relevant code fix exists between latest release and current main:** Comparing `v2026.4.23` (`a9797214338ba31c52c796adbb75afb16e0684a9`) to current `main` shows only docs changes in the relevant surfaces; the device-identity and gateway auth code is unchanged.
  - command: `git diff --stat a9797214338ba31c52c796adbb75afb16e0684a9..a155b84cdadef068de50274318952032cd867788 -- src/infra/device-identity.ts src/infra/device-identity.test.ts docs/gateway/protocol.md docs/gateway/troubleshooting.md src/gateway/server/ws-connection/message-handler.ts src/gateway/client.ts`
