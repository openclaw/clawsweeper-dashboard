## Summary

Keep open: current main still has a real observability gap between embedded attempt dispatch and Codex session start, and the related latency reports are broader rather than a duplicate owner for this specific lifecycle timing request.

## What This Changes

The issue requests trace-level Codex app-server thread lifecycle timing between embedded attempt dispatch and session.started so startup latency can be attributed to binding, plugin config, RPC, or write stages.

## Best Possible Solution

Add a Codex app-server lifecycle stage tracker around startOrResumeThread, mirror the embedded summary shape through a plugin-local or proper SDK helper, cover resume/start/binding/plugin-config stages, and keep any native hook relay cleanup narrowly diagnostic.

## Reproduction Assessment

Yes for the observability gap: source inspection shows session.started is recorded only after startOrResumeThread completes, and there is no per-stage Codex lifecycle summary. I did not live-reproduce the reported Docker latency sample.

## Solution Assessment

Yes, focused lifecycle timing is the narrowest maintainable solution for this report. The implementation should not optimize or change thread semantics until the new diagnostics identify the expensive stage.
