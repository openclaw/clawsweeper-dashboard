## Summary

Close as implemented on current main. The strict-provider failure mode is covered by the current embedded-runner guard against blank visible user prompts, and the WebChat bare `/new`/`/reset` path now builds a non-empty startup prompt plus a non-empty transcript marker; the `prompt.submitted.messages=[]` trajectory field is pre-prompt session history, not proof that the provider payload had no user message.

## What This Changes

The issue asks OpenClaw to avoid sending a system-only or zero-conversation-message first request to strict ZAI/GLM models during WebChat session bootstrap.

## Best Possible Solution

Keep the generic embedded-runner guard and WebChat startup prompt behavior on main. If a ZAI/GLM 400 still happens after this main commit, the next actionable report should include the actual provider payload or a strict mocked OpenAI-compatible payload test, because the trajectory `messages` count alone is not the provider payload.

## Reproduction Assessment

Yes. The source-level reproduction path is the reported WebChat bare `/new`/`/reset` or blank first-turn path: current main now builds a non-empty startup prompt for WebChat reset starts and skips blank visible prompts before provider submission.

## Solution Assessment

Yes. The maintainable fix is generic at the WebChat/embedded-runner boundary, not a ZAI-only workaround; durable transcript fanout is useful for UI/history, but the provider payload is protected by the prompt handoff and blank-prompt guard.
