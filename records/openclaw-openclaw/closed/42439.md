## Close Comment

Closing this as `cannot_reproduce` based on a Codex review of the issue thread and current `main`.

The strongest evidence is the reporter’s own follow-up on 2026-03-11: after repeated Telegram and Discord voice-message tests with debug logging enabled, they said Whisper succeeded, no file-block fallback appeared, and they could not reproduce the corruption. They also noted the original failure may have come from an already-corrupted session history rather than a stable Telegram-specific voice path.

Current `main` still routes Telegram voice notes through audio preflight transcription in `extensions/telegram/src/bot-message-context.body.ts:193`, with regression coverage in `extensions/telegram/src/bot-message-context.audio-transcript.test-support.ts:83`, so I could not find a current code path that matches the original theory of Telegram voice notes corrupting the message through a text/file fallback.

If this still happens on a current release, please open a fresh issue with exact reproduction steps, debug logs, the Telegram payload type, model/provider, thinking setting, and whether the session was brand new or resumed.
