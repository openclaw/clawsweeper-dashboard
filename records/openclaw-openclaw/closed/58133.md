## Summary

Keep open. The underlying `sessions_send` timeout problem is still present on current main, but this branch is not safe to merge because it turns terminal wait timeouts into accepted delivery and can announce stale assistant history; maintainers should choose or consolidate the overlapping #68196 contract before this proceeds.

## What This Changes

This branch changes `sessions_send` to return accepted/pending delivery and start delayed A2A announce delivery after an initial wait timeout, with bounded-follow-up comments and regression tests.

## Best Possible Solution

Land one consolidated fix that distinguishes soft wait-window expiry from terminal run timeout, preserves the baseline reply guard for delayed reads, keeps announce delivery alive only for proven in-flight accepted runs, and adds the required changelog entry.

## Reproduction Assessment

Yes. Current main can be reproduced with a mocked `sessions_send` path where `agent` returns a run id and the first nonzero-timeout `agent.wait` returns `status: "timeout"`; the code returns a hard timeout before any delayed announce path. The PR regressions are reproducible with mocked terminal-aborted timeout and stale-history/no-new-reply cases.

## Solution Assessment

No. The direction is valid for soft wait-window expiry, but this implementation is not the safest fix because it treats all timeout results as accepted and reads delayed replies without the pre-run baseline; #68196 should be considered during consolidation.

- **[P1] Do not mark terminal timeouts as accepted:** `src/agents/tools/sessions-send-tool.ts:380-383`
  - body: `agent.wait` can return `status: "timeout"` for both caller wait-window expiry and already-ended aborted runs. This branch converts both cases into accepted pending delivery, so terminal failures lose their failure signal and callers can wait for an announce that will never arrive.

  - body: The normal wait path captures `baselineReply` so old assistant history is not reused as the new run reply, but the timeout branch calls `startA2AFlow(undefined, runId)`. The delayed A2A flow then reads the latest assistant reply without that baseline and can announce stale history.
  - confidence: 0.89
