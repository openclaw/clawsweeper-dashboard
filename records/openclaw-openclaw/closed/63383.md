## Summary

Current `main` already covers the reported persisted-cron case by normalizing stored jobs before execution, backfilling a missing `sessionTarget` from `payload.kind`, and guarding the delivery-plan `.startsWith(...)` path.

## Evidence

- **Persisted cron jobs are normalized on load before runtime use:** `ensureLoaded(...)` reads the store and runs every persisted job through `normalizeCronJobInput(raw)` before putting it into the in-memory scheduler state, so legacy/malformed stored jobs are repaired in memory before run paths inspect `sessionTarget`.
  - file: [src/cron/service/store.ts:37](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/cron/service/store.ts#L37)
  - command: `nl -ba src/cron/service/store.ts | sed -n '30,80p'`
  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **Normalization backfills missing sessionTarget from payload kind:** When `sessionTarget` is absent, cron normalization defaults `systemEvent` jobs to `main` and `agentTurn` jobs to `isolated`, matching the issue's requested compatibility behavior for the reported `agentTurn` case.
  - file: [src/cron/normalize.ts:527](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/cron/normalize.ts#L527)
  - command: `nl -ba src/cron/normalize.ts | sed -n '527,537p'`
  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **Defaulting behavior is covered by tests:** The cron normalization tests assert that a job without an explicit `sessionTarget` is inferred as an `agentTurn` and gets `sessionTarget: "isolated"`.
  - file: [src/cron/normalize.test.ts:394](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/cron/normalize.test.ts#L394)
  - command: `nl -ba src/cron/normalize.test.ts | sed -n '392,406p'`
  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **Delivery-plan path is now null-safe before startsWith:** `resolveCronDeliveryPlan(...)` checks `typeof job.sessionTarget === "string"` before calling `.startsWith("session:")`, eliminating the raw TypeError in that path when older data is incomplete.
  - file: [src/cron/delivery-plan.ts:71](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/cron/delivery-plan.ts#L71)
  - command: `nl -ba src/cron/delivery-plan.ts | sed -n '70,77p'`
  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)

## Risks / Open Questions

- `assertSupportedJobSpec(...)` still assumes `job.sessionTarget` is present, so the proof here depends on the current load/normalization path being used before execution rather than on every call site being individually null-safe.
