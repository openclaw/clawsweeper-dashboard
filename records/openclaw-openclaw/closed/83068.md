## Summary

Keep open: the PR fixes the main hot-reload restart isolation direction, but the force-pushed head still introduces a rollback-state regression for plugin pre-replace failures by starting every attempted stop candidate without preserving whether it was actually running or manually stopped.

## What This Changes

This PR changes gateway hot reload to aggregate per-channel restart failures, roll back plugin pre-replace stops, add focused reload-handler regression tests, and add a changelog entry for the linked hot-reload issue.

## Best Possible Solution

Keep the aggregate per-channel restart behavior, but make plugin pre-replace rollback restore only the channels/accounts that were actually running before the attempted stop or preserve manual-stop state during rollback.

## Reproduction Assessment

Yes. Current main is source-reproducible from the plain awaited restart loop; the remaining PR regression is source-reproducible from default manual `stopChannel` plus default `startChannel` clearing manual-stop state.

## Solution Assessment

No. The aggregate restart isolation is the right fix direction, but starting every recorded rollback candidate is not the narrowest maintainable solution because it can change pre-reload channel state after an aborted plugin swap.
