## Summary

Keep open. Current main fixed the polling half of the false-positive report, but the webhook half remains: webhook startup does not publish runtime status, and Telegram status diagnostics explicitly skip webhook accounts, so `openclaw doctor` and `channels status --probe` cannot warn on a stuck webhook account.

## What This Changes

Request improved Telegram diagnostics so logs/status/doctor distinguish token initialization from actual polling or webhook transport activity.

## Best Possible Solution

Implement webhook-mode runtime status for Telegram alongside the existing polling status: publish `mode: "webhook"`, startup/disconnect state, successful `setWebhook` state, and transport activity from authenticated inbound webhook handling, then add webhook-specific status issues that doctor and channel status can surface.
