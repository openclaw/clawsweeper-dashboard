## Summary

Keep open for normal maintainer review: current main and v2026.6.6 still disable gateway config hot-reload after native watcher retry exhaustion, while this PR adds a narrow polling fallback with no blocking code-review findings and sufficient after-fix live-output proof.

## What This Changes

The PR changes the gateway config reloader to recreate chokidar in polling mode after native watcher retries are exhausted, and updates the watcher error recovery test for the native-to-polling lifecycle.

## Best Possible Solution

Merge the narrow gateway fallback after maintainer review and normal gates, keeping native watching as the default and disabling hot-reload only if polling-mode recovery also fails.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source and supplied logs: current main and v2026.6.6 retry native chokidar watchers and then disable hot-reload; I did not run a live ENOSPC gateway in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? Yes; a bounded switch to chokidar polling after native watcher retry exhaustion is narrower than adding a config knob and avoids requiring global CHOKIDAR_USEPOLLING or sysctl workarounds.
