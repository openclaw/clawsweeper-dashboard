## Summary

Close as external plugin packaging/ClawHub scope: the branch adds a one-off OpenClawBrain package/id exception in core, while current main already has manifest-first plugin identity and repository policy routes owner-specific plugin behavior through the plugin or a documented generic seam.

## What This Changes

The PR adds OpenClawBrain-specific package-name aliases in plugin discovery/install and tests suppression of the manifest/package mismatch log.

## Best Possible Solution

Keep core plugin identity generic; OpenClawBrain should align or publish its package metadata through ClawHub/npm, or propose a documented generic package-to-plugin-id equivalence seam if the manifest contract is insufficient.

## Reproduction Assessment

Yes, by source inspection: current main will install under `openclawbrain` when the manifest declares that id, but `@openclawbrain/openclaw` does not pass the generic package/id equivalence check and would still produce the informational mismatch log. The manifestless discovery branch would derive `openclaw`, but native plugin docs require a manifest for plugin identity.

## Solution Assessment

No. A hard-coded OpenClawBrain alias in core is not the best solution because it makes OpenClaw carry owner-specific package policy; the maintainable path is aligned external plugin metadata or a generic documented compatibility seam.
