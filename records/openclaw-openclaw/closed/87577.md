## Summary

Keep open: current main still routes Linux gateway service status and restart through the systemd user unit path, while docs and doctor/update code already recognize system-scoped gateway units as a real deployment shape.

## What This Changes

The issue requests detecting system-level systemd gateway services in `openclaw status` and `openclaw gateway restart` instead of treating them as missing user services or unmanaged processes.

## Best Possible Solution

Teach gateway status and restart to recognize system-scoped OpenClaw gateway units, report system-managed status/guidance, and avoid unmanaged process signaling for known systemd-managed gateway processes.

Yes from source inspection and the provided concrete reproduction: with only a system-scoped unit, the current Linux service path checks the user unit and then restart can fall back to unmanaged SIGUSR1. I did not run a live systemd service in this read-only review.
