## Summary

Close as ClawHub/core-scope cleanup. Current OpenClaw deliberately keeps paired-node shell execution on the `exec host=node` path, blocks the removed `nodes run` style through the core `nodes` CLI, and already exposes plugin CLI plus node runtime APIs that can support an opt-in direct node shell workflow outside core.

## What This Changes

The issue requests restoring a direct CLI way to execute shell commands on paired nodes, similar to the legacy `openclaw nodes run`, for batch remote command execution.

## Best Possible Solution

Keep core focused on explicit node capabilities and `exec host=node`; a trusted direct node shell CLI should be published as an opt-in ClawHub/native plugin that preserves Gateway node pairing, node command policies, and exec approval behavior.

## Reproduction Assessment

Yes for the observable current behavior: the code and coverage test show `openclaw nodes run` is absent and `nodes invoke --command system.run` is blocked. The desired batch CLI workflow is a feature request, so no live node reproduction is required to decide scope.

## Solution Assessment

Yes. Given the existing plugin CLI registration and `api.runtime.nodes.invoke` surface, moving this optional high-risk convenience wrapper to ClawHub is the narrower maintainable direction than restoring it as a default core command.
