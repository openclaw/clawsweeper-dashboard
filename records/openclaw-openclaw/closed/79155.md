Keep open: current main and latest release still truncate multiline slash-command payloads in shared command normalization, and the open closing PR at https://github.com/openclaw/openclaw/pull/81305 is the active fix path.

## What This Changes

Multiline slash skill and slash command payloads should remain available to downstream handlers instead of being cut after the first newline.

## Best Possible Solution

Preserve multiline tails at the shared command-normalization boundary, keep command-head normalization intact, and cover the normalizer, command context, and skill invocation path before merging the linked fix or equivalent.

## Reproduction Assessment

Yes, source-reproducible: current main slices slash command input at the first newline in `normalizeCommandBody()`, while the downstream skill parser can consume multiline args if the tail survives. I did not run tests because this review is read-only.

## Solution Assessment

Yes: the narrow maintainable fix is to preserve the newline tail in shared command normalization and prove the command-context and skill-dispatch paths. The open closing PR should be reviewed, refreshed, or replaced only if maintainers reject that branch.
