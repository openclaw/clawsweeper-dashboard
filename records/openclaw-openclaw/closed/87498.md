## Label Justifications

- `P0`: The linked behavior crosses a webhook security boundary by letting a replayed signed Twilio request mint a fresh realtime stream credential.
- `merge-risk: 🚨 compatibility`: The fix intentionally changes replayed Twilio realtime TwiML retry behavior, which existing deployments may observe during provider retries or reconnects.

## Summary

Keep open: current main still allows replayed Twilio realtime TwiML handling before replay enforcement, and this PR is a focused security fix with supplied real behavior proof and no blocking code findings. The main remaining maintainer decision is accepting the changed Twilio retry semantics for replayed realtime TwiML fetches.

## What This Changes

The PR gates Twilio initial and realtime TwiML shortcuts behind non-replay verification and updates replay tests so replayed inbound and outbound realtime TwiML requests no longer mint stream TwiML.

## Best Possible Solution

Land a focused security fix that enforces replay state before any TwiML path creates a realtime stream capability, with regression tests and explicit maintainer acceptance of replay retry semantics.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main handles `consumeInitialTwiML` and realtime TwiML before checking `verification.isReplay`, and current tests assert replayed inbound Twilio webhooks return stream TwiML. I did not run tests because this review is read-only.

## Solution Assessment

Yes, the PR takes the narrow maintainable path for the reported security issue by making replay state authoritative before realtime stream TwiML creation. If Twilio retry compatibility needs to be preserved, the safer follow-up would be an idempotent non-token replay response rather than minting a new stream token.
