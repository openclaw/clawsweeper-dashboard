## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Coerce the stored limit before sessions.list:** `ui/src/ui/controllers/sessions.ts:159`
  - body: This branch changes `SessionsState.sessionsFilterLimit` to `number`, but the real app still initializes `sessionsFilterLimit` as the string `"120"` and the Sessions tab calls `loadSessions(app)` before any user edit. Forwarding `state.sessionsFilterLimit` here can send `limit: "120"` to `sessions.list`, which violates the integer schema and makes the Sessions page fail on initial load. Keep the current parser/coercion here or complete the default/state migration before building params.
  - confidence: 0.92
- **[P2] Preserve blank limits without a zero sentinel:** `ui/src/ui/views/sessions.ts:293`
  - body: Clearing or entering an invalid value writes `0` into filter state even though the PR describes blank/invalid values as undefined. That makes blank, invalid, and explicitly unbounded state indistinguishable and pushes sentinel handling onto later callers. Use string editing state or `number | undefined`, then normalize at the request boundary.
  - confidence: 0.82
