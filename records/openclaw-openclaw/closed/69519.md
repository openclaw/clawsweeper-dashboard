## Summary

Keep open: this is a plausible narrow UI bug fix, but the current PR head still has blocking correctness issues and lacks after-fix real behavior proof.

## What This Changes

The branch converts the Control UI Sessions Limit filter to a numeric input, changes selected limit state types/tests, and includes unrelated CSS formatting churn.

## Best Possible Solution

Finish the narrow UI fix with one consistent state model and only send positive integer limits to `sessions.list`, then add after-fix real behavior proof.

## Reproduction Assessment

Yes, by source inspection. Current main renders the Sessions Limit control as a plain string input while the Gateway `sessions.list` contract requires a positive integer limit.

## Solution Assessment

No. A numeric input is a reasonable direction, but this implementation removes request-boundary coercion before fully migrating stored defaults/state and uses `0` as a blank sentinel.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Coerce stored limit before sessions.list:** `ui/src/ui/controllers/sessions.ts:159`
  - body: `loadSessions` now forwards `state.sessionsFilterLimit` directly when no override is passed, but the same branch still initializes that property from a string default and refresh/open paths call `loadSessions` without overrides. That can send `limit: "120"` to `sessions.list`, violating the integer schema until the user edits the field.
  - confidence: 0.92
- **[P2] Preserve blank limits without a zero sentinel:** `ui/src/ui/views/sessions.ts:293`
  - body: The input handler maps empty, invalid, or non-positive values to `0`, so clearing the field stores an invalid sentinel and turns later refreshes into an omitted-limit path. Keep editable blank state as blank/undefined, or preserve string state and only send positive integers at the request boundary.
  - confidence: 0.84
