## Summary

Keep open. Current main still binds the Gateway HTTP/WebSocket server before startup sidecars finish and before `startupSidecarsReady` flips; the WebSocket upgrade/connect path does not consult readiness or return a retryable startup-not-ready response. Existing code only gates selected post-connect RPC methods, so the reported startup-window handshake problem remains plausible and is not covered by the related closed startup fixes.

## Best Possible Solution

Keep this issue open and fix the Gateway WebSocket startup boundary. Either delay WebSocket upgrades/connects until the readiness contract is satisfied, or return a prompt retryable not-ready response while `startup-sidecars` is pending. Update Control UI, TUI, and CLI clients to retry that response cleanly, and add a regression test that holds sidecars pending, connects during the window, and verifies no long handshake timeout.
