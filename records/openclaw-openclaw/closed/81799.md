## Summary

This should close as superseded because the same Telegram HTML-to-plain fallback problem is already covered by an open follow-up PR, while current main only contains the earlier formatting preservation fix.

## What This Changes

Telegram should never deliver rendered HTML tags as plain text when an HTML chunk falls back after a parse error.

## Best Possible Solution

Keep the remaining work consolidated in the existing Telegram fallback PR or an equivalent narrow replacement that forwards a real plain-text fallback and derives safe plain text from HTML-mode retries.

## Reproduction Assessment

Yes, source-level: an HTML-mode Telegram send that hits withTelegramHtmlParseFallback can retry without parse_mode using fallback text derived from rendered HTML when no plainText is supplied. I did not run live Telegram, but the source path and adjacent fake-API tests make the failure mode clear.

## Solution Assessment

Yes, consolidating this with the open PR that forwards plainText and derives HTML-mode plain fallback is the best maintainable path. A narrow strip-only fallback would not cover all pre-rendered HTML chunks already used as fallback text.
