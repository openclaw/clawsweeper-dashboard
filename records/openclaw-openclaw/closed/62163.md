## Summary

Current `main` no longer broad-loads the plugin registry for `openclaw pairing approve ...`; pairing stays on the non-plugin-preload startup path, plugin CLI loading is narrowed to manifest-declared command owners, and the changelog says that behavior shipped in `v2026.4.14`.

## Evidence

- **Pairing inherits the default no-plugin-preload CLI policy:** The CLI command catalog only opts specific roots such as `agent`, `message`, `channels`, `directory`, and `agents` into `loadPlugins: "always"`. `pairing` is absent, so it falls back to the default `loadPlugins: "never"` policy.
  - file: [src/cli/command-catalog.ts:34](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/src/cli/command-catalog.ts#L34)
  - command: `nl -ba src/cli/command-catalog.ts | sed -n '1,160p'`
  - sha: [51dd4f288f67](https://github.com/openclaw/openclaw/commit/51dd4f288f67ef4df4c7a121f078f508535df1fd)
- **CLI bootstrap now skips registry loading when plugin preload is off:** `ensureCliCommandBootstrap()` returns immediately when `loadPlugins` is false, and only calls `ensureCliPluginRegistryLoaded(...)` when a command explicitly asked for plugin loading.
  - file: [src/cli/command-bootstrap.ts:31](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/src/cli/command-bootstrap.ts#L31)
  - command: `nl -ba src/cli/command-bootstrap.ts | sed -n '1,120p'`
  - sha: [51dd4f288f67](https://github.com/openclaw/openclaw/commit/51dd4f288f67ef4df4c7a121f078f508535df1fd)
- **Startup policy resolves plugin preload strictly from command-path policy:** `shouldLoadPluginsForCommandPath()` converts the catalog policy into the runtime boolean. With `pairing` not opted in, startup resolves `loadPlugins` to `false` for that command path.
  - file: [src/cli/command-startup-policy.ts:19](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/src/cli/command-startup-policy.ts#L19)
  - command: `nl -ba src/cli/command-startup-policy.ts | sed -n '1,120p'`
  - sha: [51dd4f288f67](https://github.com/openclaw/openclaw/commit/51dd4f288f67ef4df4c7a121f078f508535df1fd)
- **Plugin CLI discovery/registration is narrowed to manifest-declared command owners:** CLI plugin loading now derives `onlyPluginIds` from `resolveManifestActivationPluginIds(...)` for the active primary command and forwards that narrowed scope into both metadata loading and command registration, avoiding unrelated plugin runtime fanout.
  - file: [src/plugins/cli-registry-loader.ts:50](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/src/plugins/cli-registry-loader.ts#L50)
  - command: `nl -ba src/plugins/cli-registry-loader.ts | sed -n '1,180p'`
  - sha: [51dd4f288f67](https://github.com/openclaw/openclaw/commit/51dd4f288f67ef4df4c7a121f078f508535df1fd)
- **Pairing approval itself is a direct pairing-store mutation path:** `registerPairingCli()` calls `approveChannelPairingCode(...)` directly and `parseChannel()` explicitly allows extension channel ids without requiring a runtime registry-backed lookup path.
  - file: [src/cli/pairing-cli.ts:18](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/src/cli/pairing-cli.ts#L18)
  - command: `nl -ba src/cli/pairing-cli.ts | sed -n '1,220p'`
  - sha: [51dd4f288f67](https://github.com/openclaw/openclaw/commit/51dd4f288f67ef4df4c7a121f078f508535df1fd)
- **Release notes say the activation narrowing shipped in 2026.4.14:** The `2026.4.14` changelog includes: `Plugins/loading: narrow CLI, provider, and channel activation to manifest-declared needs ... avoid loading unrelated plugin runtime.` That matches the issue's reported root cause and indicates the fix shipped after `v2026.4.5`.
