## Summary

Current main already handles the heartbeat plugin-target problem through startup-time plugin loading and pinned channel-registry state before the heartbeat runner starts, so this stale branch is obsolete even though its lazy-normalization approach is not the best patch.

## What This Changes

The branch adds a cfg-aware retry in outbound channel normalization so heartbeat target resolution can bootstrap plugin channels before falling back.

## Best Possible Solution

Keep the current main startup-load and pinned-channel-registry implementation, and close this stale branch instead of repairing the older lazy-bootstrap approach.

## Reproduction Assessment

No. The original failing path depends on plugin channel ids being absent when heartbeat target normalization runs; current main loads and pins the plugin channel registry before the heartbeat runner starts, so I do not have a current-main failing path for `heartbeat.target: "telegram"`.

## Solution Assessment

Yes for current main; no for this branch. Startup-time plugin loading with a pinned channel registry is narrower and more maintainable than per-call lazy bootstrapping, while the PR diff can still validate through a stale channel-registry surface.
