## Summary

Current main already has the subagent completion announce recovery path this issue asks for: completion-mode announce uses explicit direct/queue fallback behavior, active requester sessions are woken or queued instead of silently dropping completion delivery, and there is regression coverage for the failure mode described here. The same announce implementation is already present in release v2026.4.22.

## Evidence

- **Completion announce state machine:** Completion-mode dispatch tries direct delivery first, then falls back to queue delivery when direct delivery fails, instead of stopping at a channel-only post.
  - file: [src/agents/subagent-announce-dispatch.ts:42](https://github.com/openclaw/openclaw/blob/40be5ad58187fd20b8defe5fc30e649190c131a8/src/agents/subagent-announce-dispatch.ts#L42)
  - command: `nl -ba src/agents/subagent-announce-dispatch.ts | sed -n '40,110p'`
- **Active requester wake / fallback handling:** When the requester session is active, completion delivery first tries to wake that session; if wake injection cannot happen, the failure is surfaced back to the dispatcher so queue fallback can take over rather than losing the parent continuation.
  - file: [src/agents/subagent-announce-delivery.ts:491](https://github.com/openclaw/openclaw/blob/40be5ad58187fd20b8defe5fc30e649190c131a8/src/agents/subagent-announce-delivery.ts#L491)
  - command: `nl -ba src/agents/subagent-announce-delivery.ts | sed -n '430,620p'`
- **Regression coverage for the reported failure mode:** There is a dedicated regression test proving that when an active completion wake cannot be injected, announce falls back to queued follow-up delivery and still targets the parent session.
