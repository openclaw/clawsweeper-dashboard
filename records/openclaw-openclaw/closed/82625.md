## Summary

Keep open: the PR targets a real Telegram duplicate-final regression, but the current patch applies partial-preview dedupe in the generic reply-payload builder and can suppress final payloads that Slack needs to commit live previews. The contributor proof is useful source-level evidence, but Telegram streaming changes still need live Telegram proof before merge.

## What This Changes

The PR records the latest partial-preview text in `runReplyAgent()`, passes it into `buildReplyPayloads()`, and filters matching final text or captions with focused regression tests.

## Best Possible Solution

Carry an explicit channel or finalizable-preview signal into payload building so Telegram suppresses duplicate final text without dropping final payloads required by other preview finalizers.

## Reproduction Assessment

Yes for source-level reproducibility: current main leaves final payloads intact when partial preview streaming runs with block streaming off, and the contributor proof shows duplicate final text/caption payloads before the patch. I did not establish live Telegram reproduction in this read-only review.

## Solution Assessment

No, not as currently implemented: the dedupe behavior should be scoped to Telegram or an explicit finalizable-preview contract rather than every `onPartialReply` user.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

- **[P2] Scope preview dedupe to finalizable preview channels:** `src/auto-reply/reply/agent-runner-payloads.ts:432-434`
