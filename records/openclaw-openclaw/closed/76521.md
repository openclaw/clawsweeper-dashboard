## Summary

Close as implemented on current main. Active Memory is now handled as its own startup plugin, and current source gates both plugin-entry disablement and live `config.enabled:false` before any embedded recall run can start.

## What This Changes

The issue asks for `plugins.entries.active-memory.enabled:false` and `config.enabled:false` to skip Active Memory entirely without blocking replies while `memory-core` remains enabled.

## Best Possible Solution

Keep the current main disabled-path behavior and let remaining Active Memory latency while enabled continue in the broader timeout tracker.

## Reproduction Assessment

No for current main: source inspection shows the disabled entry prevents startup, and `config.enabled:false` exits the live hook before embedded recall. Regression tests cover the no-call path, but no live Telegram gateway run was performed in this read-only review.

## Solution Assessment

Yes for the disabled path: the combination of startup activation gating and the live config early return is the narrow maintainable fix. The separate problem of Active Memory blocking while enabled should stay with the existing Active Memory timeout work.
