## What This Changes

The branch adds a new `apps/HarmonyOS` ArkTS/DevEco app, registers a HarmonyOS gateway client ID, and updates gateway client-id validation coverage.

## Best Possible Solution

Decide whether HarmonyOS is first-party OpenClaw app scope; if yes, land a hardened implementation aligned with bootstrap-token setup codes, secure startup, deterministic dependency inputs, labeler/docs coverage, and redacted real-device validation.

## Reproduction Assessment

Yes for the source-level PR defects: current setup codes carry `bootstrapToken`, gateway auth reads `auth.bootstrapToken`, and this branch parses/sends different fields. I did not establish a full runtime reproduction on a physical HarmonyOS device.

## Solution Assessment

No. The HarmonyOS direction may be useful, but this diff is not the best mergeable form while core pairing contracts, startup persistence, deterministic build inputs, security review, and real behavior proof remain unresolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Decode setup payloads before host fallback:** `apps/HarmonyOS/entry/src/main/ets/viewmodel/MainViewModel.ets:829-831`
  - body: `openclaw qr --setup-code-only` emits a base64url JSON payload, but this parser accepts a bare alphanumeric string as a host before decoding. Normal setup codes can therefore become `<setup-code>:18789` instead of extracting the embedded gateway URL and credential.
  - confidence: 0.92
- **[P1] Send setup credentials through bootstrapToken:** `apps/HarmonyOS/entry/src/main/ets/api/GatewaySession.ets:263-267`
  - body: Current setup-code payloads carry `bootstrapToken`, and the gateway bootstrap path reads `auth.bootstrapToken`. This serializer only sends shared `token` or `password`, so setup-code pairing is rejected on bootstrap-token gateways even if decoding succeeds.
  - confidence: 0.94
- **[P1] Initialize secure prefs before loading saved state:** `apps/HarmonyOS/entry/src/main/ets/viewmodel/MainViewModel.ets:139-140`
  - body: `mainViewModel` starts initialization from its constructor, but `securePrefs.init(this.context)` starts later in `EntryAbility`. Reads before an init promise exists fall back to defaults for host, credentials, and onboarding state, so persisted connection state can be lost for the session.
  - confidence: 0.88
- **[P2] Run hvigor without a shell command string:** `apps/HarmonyOS/hvigorw.js:28-32`
  - body: The wrapper joins forwarded argv into `hvigor ${args}` and runs it with `execSync`, so shell metacharacters in build arguments are interpreted and PATH decides which `hvigor` binary runs. Use a non-shell invocation with an args array and the local tool entrypoint.
  - confidence: 0.91
- **[P2] Commit deterministic HarmonyOS dependency inputs:** `apps/HarmonyOS/.gitignore:26-27`
  - body: The app declares a ranged third-party dependency while `.gitignore` excludes OH lockfiles. First-party app builds need a pinned and reviewable dependency graph so DevEco/local builds resolve the same package contents.
  - confidence: 0.82
- **[P2] Restore the saved gateway password in onboarding:** `apps/HarmonyOS/entry/src/main/ets/pages/OnboardingPage.ets:63`
  - body: Onboarding restores `gatewayToken` from the view model but leaves `gatewayPassword` empty. A manual onboarding retry can write that empty value back before connecting, clearing a saved password and breaking password-mode gateways until the user re-enters it.
  - confidence: 0.86
- **[P2] Keep notifications separate from speaker output:** `apps/HarmonyOS/entry/src/main/ets/pages/OnboardingPage.ets:791`
  - body: The Notifications toggle is persisted through `setSpeakerEnabled`, so disabling notifications during onboarding also disables speaker/TTS output. Persist notifications independently instead of mutating an unrelated audio setting.
  - confidence: 0.82
- **[P2] React to onboarding state changes after mount:** `apps/HarmonyOS/entry/src/main/ets/pages/Index.ets:21-24`
  - body: The root page computes `showOnboarding` only once in `aboutToAppear`, while post-onboarding UI can call `setOnboardingCompleted(false)`. Tapping Run onboarding again therefore does not route back to onboarding until the root page is recreated.
  - confidence: 0.8
- **[P2] Gate requests until authentication completes:** `apps/HarmonyOS/entry/src/main/ets/api/GatewaySession.ets:464-468`
  - body: `connected` becomes true immediately after the WebSocket opens, before `performAuthentication()` finishes. Public callers using `isConnected()` or request APIs can race the handshake and send frames before the gateway has authenticated this client.
  - confidence: 0.84
- **[P3] Fail instead of truncating oversized key material:** `apps/HarmonyOS/entry/src/main/ets/api/DeviceAuthStore.ets:247-249`
  - body: If the HUKS scalar value is larger than the target Ed25519 key size, this silently slices off the most-significant bytes. A malformed key should raise a clear error instead of producing corrupted device identity material.
  - confidence: 0.78
- **[P3] Normalize URL-safe base64 before decoding keys:** `apps/HarmonyOS/entry/src/main/ets/api/DeviceAuthStore.ets:289-303`
  - body: The decoder only recognizes the standard `+/` alphabet and skips unknown characters, so URL-safe `-` and `_` input can decode to shorter bytes and trigger silent device-key regeneration. Normalize URL-safe base64 before decoding or reject invalid characters clearly.
  - confidence: 0.74
