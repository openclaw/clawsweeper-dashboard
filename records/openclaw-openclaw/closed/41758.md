## Summary

PR #41758 should close as superseded by the still-open canonical issue #41750. Current main still lacks safe automatic recovery for an unresponsive managed Chrome process holding the CDP port, and this PR is stale, targets an obsolete source path, and has unresolved blocking safety/ordering review findings.

## Best Possible Solution

Close this stale PR as superseded and keep #41750 as the canonical tracker. A replacement fix should target extensions/browser/src/browser/chrome.ts and the availability path, safely identify OpenClaw-managed Chrome processes across macOS/Linux/Windows, avoid shell-command interpolation, avoid killing unrelated port owners, and add regression coverage for an unresponsive managed Chrome process holding the CDP port.

## Evidence

- **Canonical tracker remains open:** Provided GitHub context shows #41750 is the original open issue for the same browser tool failure: an unresponsive OpenClaw-managed Chrome process keeps --remote-debugging-port=18800 occupied and causes subsequent browser tool launches/timeouts. PR #41758 was explicitly titled and described as a fix for #41750.
- **Current main still checks port availability before launch cleanup:** launchOpenClawChrome on current main calls ensurePortAvailable(profile.cdpPort) before resolving the executable, user data directory, or spawning Chrome.
