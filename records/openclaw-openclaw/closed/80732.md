## Summary

Keep open. Current main still keeps non-loopback private generic OpenAI-compatible provider endpoints behind explicit `request.allowPrivateNetwork`, and the active implementation path is the open exact-origin SSRF policy PR that still needs maintainer/security approval.

## What This Changes

Request automatic, narrowly scoped SSRF trust for configured generic OpenAI-compatible provider `baseUrl` private endpoints, plus consistent private-network handling for related Responses paths.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/80751 or an equivalent exact-origin SSRF policy after maintainer/security approval, then track the Codex provider-id collision separately if it still reproduces.

## Reproduction Assessment

Yes. Source inspection on current main shows a non-loopback private provider `baseUrl` without `request.allowPrivateNetwork` resolves to a policy without `allowPrivateNetwork` or `allowedHostnames`, so the SSRF guard rejects the literal private host; I did not run a live LM Studio or vLLM server.

## Solution Assessment

Unclear until maintainer/security review signs off. The open exact-origin PR is a safer shape than the issue's one-line hostname allowlist because it scopes trust by origin and covers redirect/metadata cases, but the trust boundary is still a product/security decision.
