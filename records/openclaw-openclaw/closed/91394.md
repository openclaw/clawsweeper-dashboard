## Summary

Keep open: current main still lets the `openai-completions` stream finish a `stop` response with `content: []` and no tool calls, while downstream recovery is conditional and does not prove this reported StepFun post-tool silent reply is handled for all channel paths.

## What This Changes

The issue asks OpenClaw to guard OpenAI-compatible completion responses that end with `stop` but have no assistant-visible content or tool calls.

## Best Possible Solution

Normalize an empty `stop` with no text, thinking, or tool calls into the existing retry/error path, warn with provider/model context, and keep explicit silent-reply policy unchanged.

## Reproduction Assessment

Yes, source inspection gives a high-confidence adapter-level reproduction: a streamed completion that only supplies `finish_reason: stop` leaves `output.content` empty and reaches the `done` event. I did not run a live StepFun session in this read-only review.

## Solution Assessment

Yes, the narrowest maintainable fix is at the provider normalization or existing empty-turn recovery boundary, not in every channel sender. The implementation should reuse the current retry/error semantics instead of inventing channel-specific fallback text.
