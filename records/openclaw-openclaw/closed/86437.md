Full review comments:

- **[P1] Preserve the sharp backend alias:** `src/media/image-ops.ts:115`
  - body: Current main accepts `OPENCLAW_IMAGE_BACKEND=sharp`, but this switch now sends that existing value to `auto`. Operators who pinned the bundled backend can silently regain external tool fallbacks and different failure modes; keep `sharp` as a compatibility alias for Photon or fail with an explicit migration path and test.
  - confidence: 0.91
- **[P2] Keep HEIC conversion on external fallbacks:** `src/media/image-ops.ts:1288`
  - body: When `OPENCLAW_IMAGE_BACKEND=photon` is set, `imageBackendsForOperation()` returns only Photon and this branch throws a hard error, so `sips`, ImageMagick, GraphicsMagick, or ffmpeg are never tried. That contradicts the PR's stated HEIC/HEIF system-converter path and breaks HEIC inputs under the new explicit backend value.
  - confidence: 0.88
- **[P2] Include BMP and TIFF in Photon gates:** `src/media/image-ops.ts:630-636`
  - body: The core metadata parser now recognizes BMP/TIFF, but the Photon decodable gate still only allows PNG/GIF/WebP/JPEG before resize paths enter Photon. Photon documents BMP/TIFF support, so hosts without external image tools can still fail these formats through the bundled backend path.
  - confidence: 0.86
- **[P2] Keep the Sharp guard for pinned installs:** `scripts/install-cli.sh:968-975`
  - body: The installer still supports explicit version installs, including older releases that depend on Sharp, but these npm install calls no longer export the libvips guard. That can reintroduce the global-libvips build failure for rollback or pinned-version installs even though latest no longer ships Sharp.
  - confidence: 0.82
