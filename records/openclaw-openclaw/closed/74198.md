## Summary

This is a duplicate of the already-open memory-flush additional-write-target work in #66612, with the `memory/` subdirectory variant also tracked by #68995. Current main still exposes and forwards only one flush write target, and the existing tests still assert rejection of other paths, so this should not be closed as implemented. The best cleanup is to close this issue as duplicate and keep the fix design on the canonical open threads.

## Best Possible Solution

Close this issue as a duplicate and track the remaining work in #66612, coordinating with #68995 for the `memory/` subdirectory case. The eventual implementation should add one explicit validated additional-write-target contract through the memory flush plan/config or plugin API, thread it through the embedded runner and write wrapper, keep the daily file append-only, and document/test the allowed path semantics.
