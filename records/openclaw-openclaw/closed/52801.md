## Summary

Keep this PR open. Current main still lacks outbound view-once media support, and this PR is the active implementation attempt, but the latest head still has blocking privacy-sensitive validation issues around Signal media kind support, preflight media downloads, and WhatsApp auto-reply text paths.

## What This Changes

This PR adds a `viewOnce` send option across the CLI, agent tool schema, gateway send protocol, outbound delivery and queue recovery, WhatsApp Baileys media/session handling, Signal JSON-RPC sends, and auto-reply forwarding paths.

## Best Possible Solution

Land a rebased, maintainer-approved implementation that treats `viewOnce` as an additive media-only contract, supports only media kinds the channel dependency actually honors, validates unsupported cases before any send starts without consuming one-shot media URLs, preserves queue/recovery semantics, and documents the CLI/tool/gateway behavior for WhatsApp and Signal.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Reject Signal view-once videos before RPC send:** `extensions/signal/src/send.ts:179-180`
  - body: `signal-cli` documents view-once as image-attachment-only, but this branch treats `attachmentKind === "video"` as supported and sends `params["view-once"] = true`. A Signal video send can therefore reach the daemon as an unsupported privacy request instead of failing predictably.
  - confidence: 0.91
- **[P2] Avoid preflight media downloads for view-once validation:** `src/infra/outbound/deliver.ts:696-698`
  - body: This validation downloads every media URL, but the WhatsApp and Signal senders load the same URL again for the real send. One-shot or short-lived URLs can be consumed by validation, so an otherwise valid view-once send fails before delivery.
  - confidence: 0.86
- **[P2] Reject view-once replies that still send permanent text:** `extensions/whatsapp/src/auto-reply/deliver-reply.ts:139`
  - body: Adding `viewOnce` to WhatsApp auto-reply media sends leaves the text-only branch and later `remainingText` chunks as ordinary `msg.reply` sends. A view-once reply with no media, or a long caption split into later chunks, can still send permanent WhatsApp text.
  - confidence: 0.79

## Security Review

Status: needs_attention

Summary: No supply-chain changes were found, but the diff introduces privacy-sensitive disappearing-media behavior with validation gaps that can misrepresent delivery semantics.

Concerns:

- **[medium] Permanent text can bypass view-once expectation:** `extensions/whatsapp/src/auto-reply/deliver-reply.ts:139`
  - body: WhatsApp auto-replies only attach `viewOnce` to media payloads; text-only replies and split caption text still use permanent `msg.reply` sends, which can leak content a caller expected to be disappearing.
  - confidence: 0.79
- **[low] Signal view-once support exceeds dependency contract:** `extensions/signal/src/send.ts:179`
  - body: The PR advertises and sends Signal view-once for video attachments even though signal-cli documents view-once support for image attachments, creating an unreliable privacy surface.
  - confidence: 0.86
