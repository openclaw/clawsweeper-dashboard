## Summary

Current `main` already supports Codex/Claude/Cursor-style plugin bundle manifests and loads their skill roots through the normal OpenClaw plugin path. The report’s failed manual copy into `~/.openclaw/skills` matches unsupported usage, not a missing core feature.

## Evidence

- **Bundle manifests are first-class supported formats:** `src/plugins/bundle-manifest.ts` explicitly recognizes `.codex-plugin/plugin.json`, `.claude-plugin/plugin.json`, and `.cursor-plugin/plugin.json`, then loads bundle skill roots/capabilities for those formats.
  - file: [src/plugins/bundle-manifest.ts:13](https://github.com/openclaw/openclaw/blob/a53fea39050beba21ac5468d876c37323cfdd2e6/src/plugins/bundle-manifest.ts#L13)
  - command: `nl -ba src/plugins/bundle-manifest.ts | sed -n '1,220p'`
  - sha: [a53fea39050b](https://github.com/openclaw/openclaw/commit/a53fea39050beba21ac5468d876c37323cfdd2e6)
- **Bundle loading includes skill roots:** `loadBundleManifest()` maps Codex, Claude, and Cursor bundles into OpenClaw bundle manifests with `skills` entries, and `detectBundleManifestFormat()` auto-detects those bundle layouts plus manifestless Claude layouts.
  - file: [src/plugins/bundle-manifest.ts:328](https://github.com/openclaw/openclaw/blob/a53fea39050beba21ac5468d876c37323cfdd2e6/src/plugins/bundle-manifest.ts#L328)

  - file: [docs/plugins/bundles.md:10](https://github.com/openclaw/openclaw/blob/a53fea39050beba21ac5468d876c37323cfdd2e6/docs/plugins/bundles.md#L10)

## Risks / Open Questions

- The specific external `dotnet/skills` repository was not fetched in this environment, so this review proves OpenClaw-side support generically rather than validating that exact repo layout end-to-end.
- There is still a discoverability gap: copying a bundle into raw skill directories like `~/.openclaw/skills` is not the supported path, so users can still be confused even though bundle support exists.
