## Label Justifications

- `P2`: This is a normal-priority persistent session metadata bug with visible user impact and bounded storage/session-list blast radius.
- `impact:data-loss`: Replacement upserts can remove stored human-facing `label` and `displayName` values from persistent session records.
- `impact:session-state`: The affected fields are persistent session identity metadata used by session lists and channel-bound UI.

## Summary

Keep open: current main and the latest release still replace stored session entries without preserving existing human title fields, and https://github.com/openclaw/openclaw/pull/93072 remains the open fix candidate rather than a reason to close this issue before merge.

## What This Changes

The issue reports that replacement upserts for persistent session entries can erase existing human-facing `label` and `displayName` fields when the incoming entry lacks those title fields.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/93072 with the narrow store-boundary title preservation and regression coverage, then close this issue after the fix reaches main.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: store an entry with non-empty `label` and `displayName`, call `upsertSessionEntry` for the same key with replacement metadata that omits those fields, and read the stored entry back.

## Solution Assessment

Yes, subject to maintainer/API-owner compatibility acceptance. The store upsert boundary is the narrow durable fix because replacement writes pass through it, while explicit clear behavior can remain on documented empty-string or patch/replace paths.
