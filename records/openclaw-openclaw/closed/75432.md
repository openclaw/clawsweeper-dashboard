## Summary

Close as duplicate/superseded. Current main shows the OpenAI Codex provider already allowlists bare `gpt-5.4` for `xhigh`, while the remaining failure mode is the same provider-profile visibility gap already tracked by #74796 with #75295 as the open implementation candidate.

## What This Changes

The issue reports that `openai-codex/gpt-5.4` rejects explicit or default `xhigh` thinking even though the OpenAI Codex provider allowlist includes `gpt-5.4`.

## Best Possible Solution

Keep #74796 as the canonical tracker and review or land #75295, or an equivalent generic provider-runtime fallback, so startup-lazy provider thinking profiles are visible to agent command validation, gateway session rows, and `sessions.patch` without adding OpenAI Codex-specific rules to shared core.

## Reproduction Assessment

Yes. The issue gives concrete 2026.4.27 steps, and current-main source shows the same source-level failure path when the provider profile is not visible: validation falls back to base thinking levels and rejects explicit `xhigh`.

## Solution Assessment

Yes. The best fix is the canonical generic provider-profile visibility fix, not a `gpt-5.4` allowlist change, because current main already allowlists `gpt-5.4` inside the OpenAI Codex provider.
