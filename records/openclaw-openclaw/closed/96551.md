## Summary

Keep open: the PR fixes the core document filename/header bug, but it still treats audio and video assistant media as attachments even though current WebChat renders those MIME families inline through the same route.

## What This Changes

The PR adds `Content-Disposition` generation to Control UI assistant media responses, keeps images inline, serves non-image media as attachments, and adds focused gateway tests for image, document, and Unicode filenames.

## Best Possible Solution

Keep the document/Unicode filename fix, but classify previewable media MIME families as inline and reserve attachment disposition for document, text/application, and unknown binary responses.

## Reproduction Assessment

Yes, at source level: current main and `v2026.6.10` stream assistant-media documents without `Content-Disposition`, and WebChat document links use that route. I did not run a live browser save-dialog reproduction in this read-only review.

## Solution Assessment

No as submitted. The header builder is the right layer for filenames, but the best fix should preserve inline disposition for image, audio, and video while making document-like media downloadable.
