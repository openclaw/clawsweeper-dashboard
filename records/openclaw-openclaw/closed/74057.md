## Summary

Keep open because the branch still has a merge-blocking router side effect: model-scoped llama.cpp `/props` discovery is added without disabling router autoload, so read-only discovery can load unloaded models.

## What This Changes

The PR adds llama.cpp `/props` runtime context discovery for self-hosted OpenAI-compatible providers, maps runtime caps to `contextTokens`, updates focused tests, and adds a changelog entry.

## Best Possible Solution

Keep the runtime-cap discovery, but make model-scoped llama.cpp `/props` lookups side-effect-free with `autoload=false` and a regression assertion while preserving explicit `contextWindow` precedence.

## Reproduction Assessment

Yes. Source inspection of PR head plus the upstream llama.cpp router docs gives a high-confidence path: multi-model discovery calls `/props?model=<id>`, and upstream autoloads by default unless `autoload=false` is set.

## Solution Assessment

No. The PR direction is the narrow maintainable fix for https://github.com/openclaw/openclaw/issues/73664, but the model-scoped router probe must be made side-effect-free before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Avoid autoloading router models during props discovery:** `src/plugins/provider-self-hosted-setup.ts:91`
  - body: llama.cpp router mode autoloads a model by default when a GET endpoint is scoped with `model`; its docs say callers can pass `autoload=false` per request to avoid that side effect. This helper now probes `/props?model=...` for every discovered model, so read-only discovery can start loading unloaded router models. Add `autoload=false` to model-scoped props probes or only probe models known to be loaded.
  - confidence: 0.88
