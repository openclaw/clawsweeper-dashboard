## Close Comment

Closing from a Codex review because current main already repairs this failure mode on replay. The replay sanitizer now treats `openai-codex-responses` the same as the OpenAI Responses path and runs `sanitizeToolUseResultPairing(...)` before replay submission (`src/agents/pi-embedded-runner/replay-history.ts:460`). That repair routine explicitly moves matching `toolResult` entries back next to the assistant tool-call turn, drops orphaned/duplicate results, and synthesizes a missing `toolResult` when needed (`src/agents/session-transcript-repair.ts:416`). There is also regression coverage asserting that an OpenAI replay history containing only an assistant tool call is repaired into `assistant` + `toolResult` instead of being replayed as-invalid (`src/agents/pi-embedded-runner.sanitize-session-history.test.ts:691`). I also checked the shipped `v2026.4.22` tag and it already contains this replay-repair branch, so this report no longer matches current supported code. If you can still reproduce it on `v2026.4.22` or newer, please open a fresh issue with the exact poisoned session shape and current provider/model path.
