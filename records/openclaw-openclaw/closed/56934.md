## Summary

Current main and the latest shipped release treat `sk-ant-oat*` as Anthropic token auth on purpose, not API-key auth, and now have dedicated setup-token onboarding plus test/QA coverage showing Opus 4.6 works on that path. The report's claim that Bearer routing itself breaks current Anthropic setup-token usage no longer matches the codebase or shipped docs/tests.

## Evidence

- **Current transport intentionally routes `sk-ant-oat*` through Bearer auth:** `isAnthropicOAuthToken()` matches `sk-ant-oat` tokens, and the Anthropic transport sends those via `authToken`/`Authorization: Bearer` instead of `x-api-key`.
  - file: [src/agents/anthropic-transport-stream.ts:190](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/agents/anthropic-transport-stream.ts#L190)
  - sha: [3a64aa49a901](https://github.com/openclaw/openclaw/commit/3a64aa49a9014518fa1977fbc023f3e0b978ae5d)
- **Setup-token is now a first-class token flow, not API-key flow:** The bundled Anthropic provider exposes `setup-token` as `kind: "token"` with the hint `Manual bearer token path`, separate from `api-key`.
  - file: [extensions/anthropic/provider-contract-api.ts:29](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/extensions/anthropic/provider-contract-api.ts#L29)
  - sha: [3a64aa49a901](https://github.com/openclaw/openclaw/commit/3a64aa49a9014518fa1977fbc023f3e0b978ae5d)
- **Onboarding stores setup-tokens as token credentials:** Anthropic setup-token auth writes credentials as `type: "token"` and configures the profile with `mode: "token"`, so current onboarding no longer models this as `api_key`.
  - file: [extensions/anthropic/register.runtime.ts:121](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/extensions/anthropic/register.runtime.ts#L121)
  - sha: [3a64aa49a901](https://github.com/openclaw/openclaw/commit/3a64aa49a9014518fa1977fbc023f3e0b978ae5d)
- **Current tests cover Bearer routing for `sk-ant-oat*`:** The Anthropic transport test explicitly expects `authorization: "Bearer sk-ant-oat-example"` plus Claude CLI headers for `sk-ant-oat*` tokens.
  - file: [src/agents/anthropic-transport-stream.test.ts:282](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/agents/anthropic-transport-stream.test.ts#L282)
  - sha: [3a64aa49a901](https://github.com/openclaw/openclaw/commit/3a64aa49a9014518fa1977fbc023f3e0b978ae5d)
- **Current live/QA coverage treats setup-token auth as working for Opus 4.6:** There is a live test that completes a request using a setup-token profile, and a QA smoke scenario that verifies `anthropic/claude-opus-4-6` works with a token auth profile through the regular Anthropic provider.
  - file: [src/agents/anthropic.setup-token.live.test.ts:181](https://github.com/openclaw/openclaw/blob/9b18ae877018a34933e27bb7b2986cb97e9602aa/src/agents/anthropic.setup-token.live.test.ts#L181)
