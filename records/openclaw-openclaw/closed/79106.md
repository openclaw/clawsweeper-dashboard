## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Gate or document exec on SDK tools.invoke:** `src/gateway/tool-resolution.ts:237-249`
  - body: This registers `exec` in the shared `surface === "http"` resolver. `invokeGatewayTool` uses that same resolver for both `POST /tools/invoke` and SDK `tools.invoke`, so an admin SDK client can also get host shell when `gateway.tools.allow` includes `exec`. Either keep this registration HTTP-only or update protocol docs, tests, and live proof for the SDK auth boundary before merge.
  - confidence: 0.9
- **[P2] Forward the full exec policy defaults:** `src/gateway/tool-resolution.ts:53-67`
  - body: `resolveExecConfig` returns `mode`, `reviewer`, and `commandHighlighting`, and the agent path forwards those into `createExecTool`, but the new HTTP defaults builder drops them. That means configs such as `tools.exec.mode: "auto"` can behave differently on this new surface instead of preserving the audited agent exec policy.
  - confidence: 0.86
- **[P2] Make the exec regression fail without this registration:** `src/gateway/tools-invoke-http.test.ts:1029-1041`
  - body: The added test can pass even if the new lazy exec registration is removed, because this suite's `createOpenClawTools` mock already returns an `exec` tool. For this shell-exposure change, add coverage that proves the resolver-added exec path itself, such as a resolver test where the core tool factory does not already provide `exec`.
  - confidence: 0.82
