## Summary

Close as implemented on main. Current main and the v2026.4.29 tag already avoid the reported single cached system block: the prompt builder places dynamic channel/session context after an internal cache boundary, Anthropic payload shaping splits that boundary so only the stable prefix receives cache_control, and volatile Conversation info is built into the user prompt body rather than the cached system prefix.

## What This Changes

The issue requests keeping volatile Anthropic per-message metadata such as message_id and timestamp out of the cached system block so prompt-cache hits survive after the first turn.

## Best Possible Solution

Keep the existing split-cache implementation: stable system prompt content remains cache-marked, volatile runtime and channel/session context remains outside that cached block, and per-message Conversation info stays in user context. A new issue would only be needed if a fresh raw payload from the current package shows a runtime path that bypasses this boundary-aware Anthropic transport.

## Reproduction Assessment

Yes. The report's claimed payload shape is reproducible as a deterministic source-level check: current main and the v2026.4.29 tag build a boundary-split Anthropic system payload and put Conversation info in user context, so the reported single cached system string does not match the shipped request builder.

## Solution Assessment

Yes. The current implementation is the narrow maintainable fix: it combines the prior user-context relocation from #20597 with the #59054 system cache-boundary split, avoiding a broader prompt restructure while preserving Anthropic cache semantics.
