## Summary

The PR addresses a real current-main regression in the default isolated Telegram polling path with a focused structured-error fix that mirrors established classic polling recovery; no blocking code defect was found, but the protected maintainer label and missing real Bot API proof keep it in human review.

## What This Changes

The PR carries Telegram Bot API error codes across the isolated ingress worker boundary, restarts isolated polling on 409 conflicts with webhook cleanup and transport rotation, and exposes polling conflicts and stalls through channel status.

## Best Possible Solution

Keep the structured error code in the worker contract and the shared in-session recovery policy, then verify one redacted real Bot API 409 run shows webhook cleanup, transport rotation, continued polling, and accurate channel status before landing.

## Reproduction Assessment

Yes. Current main’s getUpdates-only worker drops the numeric Bot API code, and the parent rethrows a 409 because it is not classified as a recoverable network error, which gives a high-confidence source reproduction path.

## Solution Assessment

Yes. Preserving the structured API code at the existing worker boundary and applying the already-established classic polling recovery invariant is narrower and safer than parsing error text or changing gateway-wide restart policy.
