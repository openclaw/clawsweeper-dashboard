## Summary

Current `main` does not have a live Windows update path that spawns `bunx`. The updater uses `bun`, `pnpm`, `corepack`, and `npm` commands instead, so this PR's `bunx.cmd` shim change is not fixing an active repro on `main`.

## Evidence

- **Windows exec shim list on main:** `resolveCommand()` in `src/process/exec.ts` currently wraps `corepack`, `pnpm`, and `yarn` on Windows, but not `bunx`.
  - file: [src/process/exec.ts:80](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/src/process/exec.ts#L80)

  - sha: [d38ed0831d73](https://github.com/openclaw/openclaw/commit/d38ed0831d7372c8505dc944aea69b5b83da1435)
- **Global bun updates use `bun add -g`:** The current global-update path builds bun installs as `bun add -g <spec>`, not `bunx ...`.
  - file: [src/infra/update-global.ts:561](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/src/infra/update-global.ts#L561)

## Risks / Open Questions

- If a future or out-of-tree Windows flow starts spawning `bunx`, `src/process/exec.ts` still will not wrap it.
- This PR only targeted `src/process/exec.ts`; analogous Windows shim lists such as `src/process/supervisor/adapters/child.ts` also do not include `bunx`.
