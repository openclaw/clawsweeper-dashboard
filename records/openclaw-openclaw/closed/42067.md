## Summary

Current OpenClaw already covers the reported startup-exit path: Discord gateway metadata proxy failures that look like the reporter's `upstream connect error ...` response are treated as transient, the Discord plugin falls back to the default gateway URL instead of aborting startup, and wrapped `Failed to get gateway information from Discord: fetch failed` unhandled rejections are now non-fatal. The shipped `v2026.4.22` tree also contains this behavior.

## Close Comment

Closing from a Codex review because the reported startup-exit path is already implemented in current OpenClaw and is present in the shipped `v2026.4.22` tree. `extensions/discord/src/monitor/gateway-plugin.ts` now treats transient `/gateway/bot` failures, including plain-text proxy bodies like `upstream connect error or disconnect/reset before headers. reset reason: overflow`, as fallback conditions and continues with the default Discord gateway URL instead of aborting startup. `src/infra/unhandled-rejections.ts` also classifies wrapped `Failed to get gateway information from Discord: fetch failed` errors as transient, so they no longer terminate the gateway process. The matching regression coverage is in `extensions/discord/src/monitor/provider.proxy.test.ts` and `src/infra/unhandled-rejections.fatal-detection.test.ts`. Bun is still not the recommended Gateway runtime, but the specific crash-on-startup path from this report is already fixed.
