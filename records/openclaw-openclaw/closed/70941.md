## Summary

Close as duplicate/superseded. Current main still has the `openclaw models set` false-positive path, but the remaining root cause is the shared config writer comparing canonical JSON output against the previous raw byte size; open issue #71865 now tracks that same size-drop guard failure with a stronger reproduction and the correct writer-level fix direction.

## What This Changes

The issue reports that `openclaw models set` can fail to persist a valid config change because the config writer rejects a large raw byte-size drop as `CONFIG_WRITE_REJECTED`.

## Best Possible Solution

Use #71865 as the canonical tracker and fix the shared config writer so formatting-only shrink is judged against a canonicalized previous config or semantic diff, while preserving fail-closed protection for real clobbers; include a `models set` regression alongside the PowerShell/BOM auth-login case.

## Reproduction Assessment

Yes. A valid existing `openclaw.json` whose raw formatting makes it more than twice the canonical JSON size can reproduce the source-level failure: `openclaw models set <provider/model>` reaches `updateConfig` and then the shared writer rejects the write as `CONFIG_WRITE_REJECTED`.

## Solution Assessment

Yes. Closing this issue in favor of #71865 is the best cleanup path because a writer-level normalized or semantic size-drop policy covers `models set`, auth login, and similar config mutations without adding uneven command-specific destructive-write bypasses.
