## Summary

Current `main` shows the reported behavior comes from two different paths: the actual `pdf` tool does honor `agents.defaults.pdfModel`, while inbound PDF attachments use a separate media-understanding path with a default 4-page extraction cap. With the `pdf` tool explicitly enabled under tool policy, the reported `pdfModel`-ignored behavior does not match current `main`.

## Evidence

- **`pdfModel` is explicitly preferred for the `pdf` tool:** The resolver returns `agents.defaults.pdfModel` ahead of image/default-model fallbacks, and the unit test asserts that explicit `pdfModel` wins.
  - file: [src/agents/tools/pdf-tool.model-config.ts:39](https://github.com/openclaw/openclaw/blob/11cffb230013c2d8bd1e0e18da04b0b985b8abd5/src/agents/tools/pdf-tool.model-config.ts#L39)
  - command: `nl -ba src/agents/tools/pdf-tool.model-config.ts | sed -n '1,220p'`
  - sha: [11cffb230013](https://github.com/openclaw/openclaw/commit/11cffb230013c2d8bd1e0e18da04b0b985b8abd5)
- **Current tests prove the `pdf` tool uses the configured model:** `createPdfTool` tests verify native PDF execution returns `details.model` equal to the configured `anthropic/claude-opus-4-6`, and fallback execution returns `details.model` equal to the configured `openai/gpt-5.4-mini`.
  - file: [src/agents/tools/pdf-tool.test.ts:197](https://github.com/openclaw/openclaw/blob/11cffb230013c2d8bd1e0e18da04b0b985b8abd5/src/agents/tools/pdf-tool.test.ts#L197)
  - command: `nl -ba src/agents/tools/pdf-tool.test.ts | sed -n '1,260p'`
  - sha: [11cffb230013](https://github.com/openclaw/openclaw/commit/11cffb230013c2d8bd1e0e18da04b0b985b8abd5)
- **The 4-page limit exists on attachment ingestion, not the `pdf` tool:** Inbound file/PDF extraction defaults to `DEFAULT_INPUT_PDF_MAX_PAGES = 4`, which matches the reporter's observation when a PDF is processed as an attachment/media input instead of through the dedicated `pdf` tool.
  - file: [src/media/input-files.ts:115](https://github.com/openclaw/openclaw/blob/11cffb230013c2d8bd1e0e18da04b0b985b8abd5/src/media/input-files.ts#L115)
  - command: `nl -ba src/media/input-files.ts | sed -n '1,220p'`
  - sha: [11cffb230013](https://github.com/openclaw/openclaw/commit/11cffb230013c2d8bd1e0e18da04b0b985b8abd5)
- **Default tool policy excludes `pdf` under the common onboarding profile:** Docs say local onboarding defaults new configs to `tools.profile: "coding"`, and the published tool-profile tables for `coding` do not include `pdf`. That matches the discussion pointing out `tools.alsoAllow: ["pdf"]` is required if a restrictive profile is active.
  - file: [docs/gateway/config-tools.md:20](https://github.com/openclaw/openclaw/blob/11cffb230013c2d8bd1e0e18da04b0b985b8abd5/docs/gateway/config-tools.md#L20)
  - command: `nl -ba docs/gateway/config-tools.md | sed -n '16,30p'`

- **Issue discussion already identified the missing tool enablement:** Commenter `tanys123` noted that the `pdf` tool is not enabled by default and suggested `openclaw config set tools.alsoAllow '["pdf"]'`, which aligns with the current tool-profile docs and explains the 'no access to pdf read tool' part of the report.

## Risks / Open Questions

- The report may have reflected real confusion in `v2026.3.31`, but the current `main` evidence points to configuration/tool-policy and attachment-path behavior rather than an active core regression.
- Docs are still somewhat easy to misread because the dedicated `pdf` tool and the separate PDF attachment/media-understanding path have different limits and routing behavior.
