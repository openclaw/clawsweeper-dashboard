## Summary

Keep #69800 open. Current main has generic GatewayClient retry behavior and node-host stderr logging for socket errors/closes, but I did not find a source, test, docs, or changelog-backed fix for the reported managed node-host silent-zombie after a 1012/ECONNREFUSED burst. The report is recent, has concrete production evidence, and a related comment agrees this failure family still needs a lifecycle fix.

## Best Possible Solution

Keep this issue open and implement a targeted node-host lifecycle hardening path: emit a startup/reconnect-attempt log before each connection attempt, add a watchdog for stuck pre-connect/auth/disconnected states, and either keep retrying observably forever or exit after a bounded silent/disconnected window so launchd KeepAlive can respawn. Add a GatewayClient/node-host regression that simulates 1012 service-restart closes plus ECONNREFUSED bursts, and document or expose a reliable node liveness signal for operators.

## Evidence

- **Issue evidence is concrete and current:** The issue reports OpenClaw 2026.4.15 on macOS launchd with a live-but-disconnected node process for about 25h, stale `Last Connect`, no stderr writes from the zombie instance, and recovery only after `openclaw node` service kickstart. A follow-up comment explicitly treats the 1012 burst as a known rough spot and recommends bounded retry plus process exit under KeepAlive.
- **Node-host only logs socket callbacks, then keeps process alive forever:** `runNodeHost` wires `onConnectError` and `onClose` to stderr, starts the GatewayClient, then awaits a never-resolving promise. There is no bounded failure counter, startup heartbeat, disconnected-duration watchdog, or managed-service exit path here.
