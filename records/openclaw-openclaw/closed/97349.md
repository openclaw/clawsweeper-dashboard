## Summary

Keep open: current main still constructs Azure OpenAI Responses SDK clients without guarded fetch and still returns non-OK bodies from the shared sanitizer unchanged, while this PR provides a focused fix with sufficient real local HTTP-server proof. The remaining concern is maintainer acceptance of the Azure guarded-transport compatibility change, not a concrete patch defect.

## What This Changes

The PR injects `buildGuardedModelFetch` into both Azure OpenAI Responses SDK client paths and adds a lazy shared cap for non-OK response bodies.

## Best Possible Solution

Land the resolved-base-url guarded-fetch injection and shared lazy non-OK cap after maintainers explicitly accept the Azure guarded-transport compatibility risk.

## Reproduction Assessment

Yes. Source and dependency inspection show current main returns non-OK bodies unchanged and OpenAI SDK v6.39.1 reads final non-OK bodies with `response.text()`; the PR body adds pasted local HTTP-server proof for the capped and cancel paths.

## Solution Assessment

Yes. The shared lazy `TransformStream` cap fixes the unbounded error-body path without the earlier eager-read retry/cancel regression, and resolved-base-url fetch injection is the right Azure provider boundary.
