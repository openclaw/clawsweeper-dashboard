## Label Justifications

- `P0`: The report describes a security-boundary bypass where authenticated admin RPC for one gateway can execute against another gateway's control-plane context.
- `impact:security`: The affected path crosses the documented separate-gateway trust boundary for administrative RPC methods.
- `impact:session-state`: The root cause is request-context misassociation between live gateway instances in the same process.

## Summary

Keep this issue open: current main still has the reported gateway/plugin context path, and the same-author linked fix PR is open with closing syntax. This should close only after that PR lands or is replaced by an equivalent server-local context binding.

## What This Changes

The issue requests binding admin HTTP RPC gateway-method dispatch to the current gateway instance instead of allowing it to fall back to process-global gateway context.

## Best Possible Solution

Land or replace the linked fix so plugin HTTP gateway-method dispatch always carries the server-local `GatewayRequestContext` and no longer relies on the process-global fallback for authenticated admin RPC.

## Reproduction Assessment

Yes, from source inspection: plugin HTTP routes create a runtime scope without `context`, while gateway method dispatch falls back to the process-global resolver installed by `startGatewayServer`. I did not run a live two-gateway PoC because this review is read-only.

## Solution Assessment

Yes, the narrow maintainable fix direction is to bind plugin HTTP runtime scopes to the server-local gateway context. The issue should stay open until https://github.com/openclaw/openclaw/pull/83487 is merged, revised, or superseded by an equivalent fix.
