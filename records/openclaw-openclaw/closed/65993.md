## Summary

Keep open. Current main still preserves the Feishu behavior that can split one logical reply into multiple card deliveries: the Feishu dispatcher hard-disables block streaming, the streaming card still uses the 100ms throttle, and the regression tests explicitly expect a second streaming card when a later final payload extends an earlier one after the first stream has closed.

## Evidence

- **Feishu still forces block streaming off:** The Feishu reply dispatcher returns `disableBlockStreaming: true`, so Feishu ignores the channel/user block-streaming setting and always resolves block streaming to off for this surface.
  - file: [extensions/feishu/src/reply-dispatcher.ts:494](https://github.com/openclaw/openclaw/blob/250acdd2a28c190ed7934e3ba99022f24123d3c9/extensions/feishu/src/reply-dispatcher.ts#L494)

  - sha: [250acdd2a28c](https://github.com/openclaw/openclaw/commit/250acdd2a28c190ed7934e3ba99022f24123d3c9)

A later non-identical final payload falls through to a new delivery path instead of being merged into the existing card.
