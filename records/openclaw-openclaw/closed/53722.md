## Summary

Current `main` already carries TTS audio through the outbound media path, preserves `audioAsVoice`, and Feishu maps compatible voice-note audio to native `audio` messages instead of leaking a filesystem path as plain text.

## Evidence

- **TTS replies emit media payloads, not plain text paths:** `/tts` returns a reply payload with `mediaUrl: result.audioPath`, `audioAsVoice: result.voiceCompatible === true`, and `trustedLocalMedia: true`.
  - file: [src/auto-reply/reply/commands-tts.ts:167](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/auto-reply/reply/commands-tts.ts#L167)

  - file: [extensions/speech-core/src/tts.ts:588](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/extensions/speech-core/src/tts.ts#L588)

  - file: [src/infra/outbound/deliver.ts:190](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/infra/outbound/deliver.ts#L190)

  - file: [src/infra/outbound/deliver.test.ts:603](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/infra/outbound/deliver.test.ts#L603)

## Risks / Open Questions

- Feishu currently sends native voice messages for Opus/Ogg voice-note payloads; unsupported audio formats such as generic MP3 still fall back to regular file delivery, so any remaining report would need a concrete repro showing a current provider/channel combination emitting non-voice-compatible audio.
