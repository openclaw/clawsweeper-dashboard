## Summary

Keep open: current main still matches the reported doctor false positive because the daemon install plan uses the host-env override blocklist when validating exec SecretRef `passEnv` keys.

## What This Changes

Allow exec SecretRef providers to include `HOME` in `passEnv` without `openclaw doctor` emitting a host-env security warning.

## Best Possible Solution

Ship a focused doctor/daemon plan fix that treats exec `passEnv` as inherited env rather than override env, with regression coverage and a changelog entry.

Yes from source inspection: an active exec SecretRef provider with `passEnv: ["HOME"]` reaches the doctor service-audit install plan, and the current OR check flags `HOME` through the override-only policy. I did not run a live doctor command in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is to separate passEnv inheritance validation from override validation in the daemon install/doctor plan, while keeping truly dangerous inherited keys blocked.
