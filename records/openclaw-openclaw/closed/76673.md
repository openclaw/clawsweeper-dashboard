## Summary

Close as duplicate/superseded: this PR targets the same health-monitor stop-timeout recovery tracked by #71412, while open maintainer-labeled PR #72912 is the active and more complete implementation path.

## What This Changes

The PR changes channel stop-timeout handling to forget stale task bookkeeping, mark the account stopped, guard the old final stopped write, and add server-channel regression tests for replacement after a timed-out stop.

## Best Possible Solution

Use #72912 as the canonical fix because it scopes forced lifecycle retirement to health-monitor recovery, gates stale lifecycle callbacks, cleans task-scoped runtime state, and includes tests plus changelog coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: the health monitor calls stopChannel then startChannel, and current main's stop-timeout branch retains store.tasks so the start call no-ops; I did not run tests because this review is read-only.

## Solution Assessment

No. This PR is a narrower partial patch with remaining lifecycle races; #72912 is the safer maintainable solution because it adds an explicit health-monitor force-retire mode and guards stale callbacks throughout the lifecycle.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Gate retired lifecycles before restart scheduling:** `src/gateway/server-channels.ts:544`
  - body: This guard only protects the final stopped write. After a timed-out stop is forgotten and the replacement start clears `manuallyStopped`, the old `trackedPromise` still enters the following auto-restart continuation, writes `restartPending: true`/increments attempts on the replacement, then aborts its sleep. Gate that continuation on the same current-lifecycle check or preserve retirement state for the old task.
  - confidence: 0.87
- **[P2] Keep forced retirement scoped to health restarts:** `src/gateway/server-channels.ts:689-694`
  - body: The timeout branch now deletes task/abort bookkeeping for every `stopChannel` call, including manual stops and shutdown/control-UI stops. If the provider ignored abort, OpenClaw can report it stopped and allow a second lifecycle while the old one is still alive. Keep the conservative manual-stop behavior and opt into forced retirement only from health-monitor recovery.
  - confidence: 0.82
