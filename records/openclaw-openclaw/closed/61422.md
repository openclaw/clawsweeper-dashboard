## Summary

Keep open: the cache-invalidation problem is real, but this PR's current diff is too broad because it moves all Project Context files out of the cached prefix and leaves the prompt-order guard contract stale.

## What This Changes

The PR moves the `# Project Context` block in `buildAgentSystemPrompt` from before to after `SYSTEM_PROMPT_CACHE_BOUNDARY`.

## Best Possible Solution

Implement a stable-vs-volatile context split: keep AGENTS/SOUL/IDENTITY/USER/TOOLS/BOOTSTRAP cacheable, move only volatile or auto-written context below the boundary, and update tests, docs, and changelog together.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists by building a prompt with injected context files, comparing their positions to `SYSTEM_PROMPT_CACHE_BOUNDARY`, and checking Anthropic payload shaping where only the prefix receives `cache_control`.

## Solution Assessment

No. Moving every Project Context file below the boundary avoids one invalidation case but forfeits caching for stable workspace context; a narrower volatility classification is the maintainable fix.
