## Summary

Keep open. Current `main` has partial mitigations for this area, but not strong evidence that the cross-process stale-`runningAtMs` race described in #59056 is fully resolved. Runtime state was moved out of `jobs.json` into `jobs-state.json`, and restart cleanup/replay paths were tightened, but the persistence path still rewrites the caller's full in-memory runtime snapshot without a compare-and-swap or reload/merge guard before write.

## Evidence

- **State split shipped, but only as a storage/layout mitigation:** `2026.4.20` records the cron runtime-state split into `jobs-state.json`, and current docs describe `jobs.json` as definitions plus separate runtime state. That addresses the old `jobs.json` corruption path from older releases, but it does not by itself prove cross-process runtime-state writes are conflict-safe.
