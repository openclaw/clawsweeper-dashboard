## Label Justifications

- `P2`: The report describes a real WSL-specific gateway availability failure, but the actionable fix is a limited setup/docs path already tracked by an open PR.
- `impact:crash-loop`: The gateway repeatedly receives SIGTERM and restarts or stops when the WSL terminal closes.
- `impact:message-loss`: The reporter observes Telegram messages briefly accepted and then no further replies after the gateway is terminated.

## Summary

Close as superseded: the WSL2 idle-termination/gateway shutdown problem is already covered by the open Windows WSL autostart documentation fix at https://github.com/openclaw/openclaw/pull/90992, while current gateway code only handles an external SIGTERM and does not show a safe OpenClaw-side ignore-SIGTERM flag.

## What This Changes

The issue asks for guidance or a fix when closing the Ubuntu terminal in WSL2 sends SIGTERM to a lingering systemd user gateway and leaves Telegram unreachable.

## Best Possible Solution

Land the canonical WSL autostart docs fix and keep normal graceful SIGTERM handling; do not add an OpenClaw flag that ignores supervisor shutdown signals.

## Reproduction Assessment

Yes for source/provided evidence, though I did not run a live WSL repro. Current docs still contain the old headless WSL recipe, upstream WSL tracking matches terminal-close idle termination despite systemd, and the canonical PR supplies after-fix WSL proof.

## Solution Assessment

Yes. The narrow maintainable solution is to fix the WSL keepalive/autostart recipe already tracked in https://github.com/openclaw/openclaw/pull/90992, not to make the gateway ignore SIGTERM under a supervisor.
