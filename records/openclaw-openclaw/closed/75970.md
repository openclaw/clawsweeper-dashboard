## Summary

Keep open for maintainer review: the source and regression coverage support the cron quarantine fix, and I found no discrete line-level correctness blocker, but the PR intentionally changes persisted cron storage behavior and needs explicit maintainer acceptance plus current-head check/rebase handling before merge.

## What This Changes

The PR quarantines malformed persisted cron jobs into a `jobs-quarantine.json` sidecar, sanitizes active `jobs.json` after successful quarantine, reports quarantine state in doctor/docs, and adds cron/UI regression coverage.

## Best Possible Solution

Land the sidecar-plus-sanitize design only if maintainers want quarantine to become the cron storage contract; otherwise keep invalid rows out of runtime but defer active-file cleanup to doctor/manual repair.

## Reproduction Assessment

Yes. Source inspection shows current main can admit malformed schedule-bearing cron rows into UI filtering paths while the service skips invalid persisted jobs without a quarantine sidecar; the linked bug and PR tests/proof cover the malformed-store cron list/dashboard failure path, though I did not run a live repro in this read-only review.

## Solution Assessment

Yes, conditional on maintainer acceptance. The PR fixes the runtime data-loss/crash path by preserving raw malformed rows before sanitizing the active store, but load-time persisted-file cleanup is a product/compatibility decision rather than a purely mechanical bug fix.
