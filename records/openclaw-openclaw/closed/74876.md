## Summary

Keep open. Current main still defaults every group/channel source turn to message-tool-only delivery, so normal final replies remain private unless configured otherwise; docs partially explain the behavior, but the changelog wording, agent-facing config patch access, warning/migration posture, and default-policy question are not fully resolved on main.

## What This Changes

The issue asks OpenClaw to address the v2026.4.27 group/channel visible-reply default change that makes final replies private across all group chat channels, plus the related changelog, warning, config access, reload guidance, and migration concerns.

## Best Possible Solution

Keep the issue open until the remaining pieces land or are explicitly rejected: all-channel release-note wording, an operator-accessible way to set visibleReplies where appropriate, accurate reload/restart guidance, and a deliberate warning or migration/default decision for existing group deployments.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists on current main: with empty config and ChatType group or channel, source delivery resolves to message_tool_only and dispatch suppresses sendFinalReply; setting messages.groupChat.visibleReplies to automatic restores delivery in the current regression tests.

## Solution Assessment

Unclear. The current implementation is a coherent safety default, but it is not a complete fix for the upgrade-surprise report; the maintainable solution depends on maintainer policy across warning, docs/changelog, config access, and migration/default behavior.
