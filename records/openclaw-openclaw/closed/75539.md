## Summary

Keep open: current main has the merged Telegram getMe probe fallback, but the QQBot REST, token, channel API, and gateway WebSocket paths still lack a visible channel-owned IPv4/family fallback for the reported broken-IPv6 dual-stack failure mode.

## What This Changes

The issue reports Telegram and QQBot connection failures and event-loop stalls on dual-stack hosts without usable global IPv6 after the plugin runtime moved onto bundled Undici.

## Best Possible Solution

Keep the merged Telegram probe behavior and add a QQBot-owned transport helper/policy for REST, token refresh, channel API, and gateway WebSocket paths, then tighten obsolete QQBot raw-fetch allowlist entries.

## Reproduction Assessment

No, not end to end. The reporter supplied concrete affected environments and current main still shows QQBot raw/default transport paths, but I did not run QQBot with credentials on a dual-stack host lacking usable IPv6.

## Solution Assessment

Yes for the direction: keep Telegram fallback in the Telegram transport/probe path and fix QQBot inside QQBot-owned transport helpers rather than mutating process-wide DNS policy.
