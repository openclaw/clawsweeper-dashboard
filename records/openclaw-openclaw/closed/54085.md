## Summary

Keep open. Current main does not implement the broad short-lived-command respawn skip, and the proposed patch still conflicts with current CLI startup contracts: it lets newly skipped commands bypass startup TLS reexec and it reverses the shipped foreground `gateway run` no-respawn policy.

## What This Changes

The PR changes CLI respawn policy so only `gateway`, `daemon`, `agent`, and `memory` commands reexec, moves respawn-policy assertions into a dedicated test file, and removes the old assertions from `cli-utils.test.ts`.

## Best Possible Solution

Split the CLI respawn planner so command policy can skip only ExperimentalWarning suppression for verified short-lived commands while mandatory startup-environment reexec still runs when needed, and preserve the current foreground `gateway run` no-respawn behavior unless a maintainer explicitly changes that product policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Preserve startup TLS reexec when skipping warnings:** `src/cli/respawn-policy.ts:44`
  - body: The new allowlist makes `shouldSkipRespawnForArgv` true for commands like `status`, `config`, and `models`, but `buildCliRespawnPlan` returns before it can inject `NODE_EXTRA_CA_CERTS`. On Linux/nvm installs those commands would no longer get the documented startup trust-root fix, so the warning-suppression policy needs to be separated from mandatory TLS reexec.
  - confidence: 0.91
- **[P2] Keep foreground gateway runs on the no-respawn path:** `src/cli/respawn-policy.ts:21`
  - body: Putting `gateway` in `COMMANDS_NEEDING_RESPAWN` makes bare `openclaw gateway` and `openclaw gateway run` reexec again. Current main intentionally skips that path and the changelog records it as the fix for #72720, where low-memory Linux/Node 24 hosts could hang before logs, so this PR should not restore the startup respawn for foreground gateway runs.
  - confidence: 0.82
