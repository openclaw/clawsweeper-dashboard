## Label Justifications

- `P2`: The issue reported a bounded but real multi-agent and channel wake-routing defect.
- `impact:message-loss`: Wake-triggered output could fail to reach the conversation that requested it.
- `impact:session-state`: The wake event and heartbeat could be associated with the default agent instead of the originating agent/session.

## Summary

Current main fully resolves the reported wake-routing defect: cron wake now carries the originating or explicit session and agent through the tool, protocol, gateway, queued event, and heartbeat paths, with focused coverage for agent-only targeting and conversation-lane delivery. The fix merged through https://github.com/openclaw/openclaw/pull/83738 after live Telegram before/after proof; it is newer than v2026.6.5 and therefore remains current-main-only.

## What This Changes

Allow cron wake events to target the originating or explicitly selected agent/session instead of always routing to the default agent.

## Best Possible Solution

Keep the landed origin-aware wake implementation and its focused regression coverage; users should upgrade to the first release after v2026.6.5 that contains this current-main fix.

## Reproduction Assessment

Yes. The pre-fix omission is clear from the v2026.6.5 source, focused regression tests pin the failing routing boundary, and the merged fix supplied live before/after Telegram evidence.

## Solution Assessment

Yes. Propagating origin identity through the existing cron tool, protocol, gateway, event queue, and heartbeat boundaries is the canonical fix; it also preserves the existing untargeted-wake behavior.
