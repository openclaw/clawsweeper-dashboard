## Summary

Current `main` already addresses this: the per-agent `models.json` generator now normalizes resolved secrets back to non-secret markers before serialization, enforces source-managed markers from the source config snapshot, and the OpenAI Codex catalog path no longer emits an `apiKey` at all for OAuth/device-code auth. The implementing change is on `main` in `306c0f73bff1338482dd6533f61a86905bf541b1` (PR `#71368`).

## Evidence

- **Models.json planner now normalizes and re-enforces managed secrets before write:** `planOpenClawModelsJsonWithDeps()` constructs `secretRefManagedProviders`, runs `normalizeProviders(...)`, then `enforceSourceManagedProviderSecrets(...)` before serializing the final provider map. That means persisted `models.json` data comes from marker-normalized provider configs rather than raw runtime-resolved secrets.
  - file: [src/agents/models-config.plan.ts:105](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/src/agents/models-config.plan.ts#L105)
  - command: `nl -ba src/agents/models-config.plan.ts | sed -n '80,120p'`
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **Resolved env secrets are converted back to marker names:** `normalizeProviders()` explicitly reverse-maps a resolved `apiKey` value back to the provider's canonical env-var marker to prevent plaintext persistence in `models.json`.
  - file: [src/agents/models-config.providers.normalize.ts:77](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/src/agents/models-config.providers.normalize.ts#L77)
  - command: `nl -ba src/agents/models-config.providers.normalize.ts | sed -n '70,100p'`
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **Source snapshot forces SecretRef-managed apiKey/header markers:** `enforceSourceManagedProviderSecrets()` reads the source config SecretRef, then rewrites provider `apiKey`/headers to env markers or `secretref-managed` before the file is written, preventing runtime plaintext from being serialized back out.
  - file: [src/agents/models-config.providers.source-managed.ts:30](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/src/agents/models-config.providers.source-managed.ts#L30)
  - command: `nl -ba src/agents/models-config.providers.source-managed.ts | sed -n '20,110p'`
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **OpenAI Codex catalog no longer writes an apiKey for OAuth/device-code auth:** The OpenAI Codex provider catalog now only checks whether an auth profile exists and returns `buildOpenAICodexProvider()`. That builder contains `baseUrl`, `api`, and `models`, but no `apiKey`, so device-code/OAuth credentials are not embedded into generated per-agent catalogs.
  - file: [extensions/openai/openai-codex-provider.ts:454](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/extensions/openai/openai-codex-provider.ts#L454)

  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **Regression tests cover marker persistence instead of plaintext:** New tests assert that auth-profile env/file/exec refs persist as env markers or `secretref-managed`, and that generated `models.json` keeps `OPENAI_API_KEY`/`secretref-managed` markers under runtime-source projection rather than runtime plaintext.
  - file: [src/agents/models-config.providers.auth-provenance.test.ts:58](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/src/agents/models-config.providers.auth-provenance.test.ts#L58)
