## Summary

Close as implemented on current main: the shared install-root churn depended on legacy startup/runtime dependency staging, and main now makes plugin dependency work install/update/doctor-only, treats `plugin-runtime-deps` as legacy debris, and blocks those artifacts from package output. The proof is on `main` only; no release tag contains it yet.

## What This Changes

Prevent bundled runtime dependency plans that share one install root from repeatedly pruning and reinstalling each other's packages during startup, Control UI, or channel/plugin loading.

## Best Possible Solution

Keep the current install/update/doctor-only dependency model and avoid reintroducing startup-time shared install roots; the next release should carry this current-main behavior.

## Reproduction Assessment

Yes for the affected release line: the issue and Linux follow-up give concrete repeated npm add/remove and missing-module traces. Current-main reproduction is not expected because the code and docs no longer have runtime startup paths that generate or install shared bundled dependency plans.

## Solution Assessment

Yes. Removing runtime package-manager mutation and treating old generated roots as cleanup-only is safer than a superset manifest because it eliminates the shared install-root pruning class instead of trying to make competing plans converge.
