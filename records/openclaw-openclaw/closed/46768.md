## Summary

Current main already implements targeted parent-session wakeups for parent-owned ACP child completions and progress events, which addresses the reported idle-orchestrator behavior for `sessions_spawn({ runtime: "acp", mode: "run" })` workflows.

## Evidence

- **ACP child spawns are tracked as background tasks owned by the parent session:** `createRunningTaskRun(...)` is called for ACP spawns with the parent session as `ownerKey`/session scope, so completions are routed back through task delivery instead of waiting for a manual ping.
  - file: [src/agents/acp-spawn.ts:1315](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/src/agents/acp-spawn.ts#L1315)
  - sha: [56fe2aab9c87](https://github.com/openclaw/openclaw/commit/56fe2aab9c87596515f2280c8060dc740ea64a88)
- **Task completion fallback explicitly wakes the parent session:** When background-task delivery falls back to the session, OpenClaw enqueues a system event and immediately calls `requestHeartbeatNow({ reason: "background-task", sessionKey: ownerKey })`. Blocked follow-ups do the same with `background-task-blocked`.
  - file: [src/tasks/task-registry.ts:978](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/src/tasks/task-registry.ts#L978)
  - sha: [56fe2aab9c87](https://github.com/openclaw/openclaw/commit/56fe2aab9c87596515f2280c8060dc740ea64a88)
- **State-change updates use the same session-queue path:** For `notifyPolicy: "state_changes"`, `maybeDeliverTaskStateChangeUpdate()` queues the update back to the owner session when direct channel delivery is unavailable, using the same wake-triggering `queueTaskSystemEvent(...)` path.
  - file: [src/tasks/task-registry.ts:1129](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/src/tasks/task-registry.ts#L1129)
  - sha: [56fe2aab9c87](https://github.com/openclaw/openclaw/commit/56fe2aab9c87596515f2280c8060dc740ea64a88)
- **Docs now describe the parent-owned ACP completion channel:** The ACP docs explicitly say one-shot ACP children spawned via `sessions_spawn({ runtime: "acp", mode: "run" })` report completion back through the internal task-completion announce path and that this path is responsible for waking/reporting to the parent.
  - file: [docs/tools/acp-agents.md:342](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/docs/tools/acp-agents.md#L342)
  - sha: [56fe2aab9c87](https://github.com/openclaw/openclaw/commit/56fe2aab9c87596515f2280c8060dc740ea64a88)
- **Regression tests cover session-queued completion delivery and wake behavior:** Tests assert that ACP completion fallback marks delivery as `session_queued`, queues a `Background task done` system event for the parent session, and sets a pending heartbeat wake for blocked follow-ups.
  - file: [src/tasks/task-registry.test.ts:683](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/src/tasks/task-registry.test.ts#L683)
  - sha: [56fe2aab9c87](https://github.com/openclaw/openclaw/commit/56fe2aab9c87596515f2280c8060dc740ea64a88)
