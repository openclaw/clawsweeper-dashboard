## Summary

Keep open: the linked failed-exec loop bug is valid and this PR is close, but the patch drops stable failed-exec discriminators while removing volatile output from the hash, which can make different failure modes count as the same no-progress streak.

## What This Changes

The PR changes the agent tool-loop detector so failed `exec` outcomes ignore volatile output in result hashing, with tests for critical and global breaker escalation.

## Best Possible Solution

Keep the output-stripping fix, but hash stable failed-exec metadata such as `failureKind` and `exitSignal` along with status, exit code, and timeout, with regression coverage for both volatile output and distinct failure modes.

## Reproduction Assessment

Yes, by source inspection: current main hashes failed exec output into `resultHash`, while critical/global blockers require identical hashes, so varying failed output prevents escalation. I did not run a local reproducer in this read-only review.

## Solution Assessment

No. The proposed direction is the right narrow fix, but the best implementation should preserve stable failed-exec fields instead of hashing only status, exit code, and timeout.
