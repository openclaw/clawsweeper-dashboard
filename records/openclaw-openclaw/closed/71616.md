## Summary

Current main already fixes the Control UI race reported in #71616: final assistant chat events update the Lit state immediately, and the final `chat.history` refresh now preserves local user/assistant messages when the gateway returns either a stale snapshot or an empty first-turn snapshot.

## Best Possible Solution

Keep the current main implementation. No new core API or product work is needed for this issue; if the behavior is still reproducible on a build containing 06d409dc2738034f6005ede426726e830ea48679 or newer, the best follow-up is a fresh bug with gateway logs, browser console output, and the observed `chat` / `chat.history` event sequence.

## Evidence

- **Issue context:** #71616 reports OpenClaw 2026.4.23 on macOS where Control UI assistant replies do not render until a later user send. The author association is NONE and there are no protected labels, so it is eligible for maintainer cleanup if current main resolves it.
- **Final assistant event updates UI state immediately:** `handleChatEvent` handles `state === "final"` by normalizing the assistant payload and appending it to `state.chatMessages`, with a fallback that appends the accumulated stream text when the final payload lacks a message. `chatMessages` is a Lit `@state()` field, so this assignment is the render trigger.
