## Summary

Close as implemented. Current main now handles the two observed duplicate paths: Gateway history reads only the active transcript branch for forked user entries, and the Control UI replaces stale optimistic/streamed tails once final history catches up; the fix is also shipped in v2026.4.27.

## What This Changes

The issue reports that Control UI/WebChat renders both user and assistant chat messages twice in sequence after sending a message or refreshing history.

## Best Possible Solution

Keep the shipped split fix: Gateway history should continue exposing only canonical active-branch transcript messages, while the Control UI should keep optimistic tails only for genuinely stale history snapshots and otherwise trust the refreshed canonical history.

## Reproduction Assessment

Yes. The issue includes concrete Control UI send and refresh steps, and current main has focused regression fixtures for the two reproducible causes: forked user transcript entries and stale streamed assistant tails after `chat.history` refresh.

## Solution Assessment

Yes. The current implementation addresses the problem at the narrow ownership points: transcript branch selection in Gateway history and optimistic-tail replacement in the Control UI merge path, rather than papering over duplicates in the final renderer.
