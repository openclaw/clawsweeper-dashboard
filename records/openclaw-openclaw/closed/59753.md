## Summary

#59753 is the same remaining Feishu WebSocket recovery bug already tracked by open issue #52618, with open PR #57978 as the focused supervisor-loop implementation path. Current main still has the single-shot WSClient path and HTTP-only Feishu status probe, so this should be deduplicated rather than marked implemented.

## Best Possible Solution

Close #59753 as a duplicate of #52618 and keep the remaining implementation focused on PR #57978 or an equivalent Feishu-owned fix in `extensions/feishu/src/monitor.transport.ts`. The canonical work should include both pieces surfaced here: a bounded/stall-aware WSClient supervisor loop and a status/doctor liveness signal that does not treat an HTTP bot ping as proof that the persistent WebSocket is functional.
