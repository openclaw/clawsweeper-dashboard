## Summary

Keep open: current main has plugin command registration, argument parsing, help/list exposure, and `continueAgent`, but it still lacks the requested typed pass-through/body-replacement result that sends the remaining command text to the LLM as the same-turn user message. This remains a public Plugin SDK feature gap, with recent user confirmation and no canonical merged fix or duplicate that owns the same pass-through contract.

## What This Changes

Request an additive Plugin SDK command API/result shape so registered slash commands can run plugin logic and pass the remaining command text to the LLM as the same-turn user message.

## Best Possible Solution

Add an additive typed plugin command continuation result for body replacement, wire it through text and native plugin command paths, document it in the Plugin SDK/slash-command docs, and keep existing `continueAgent` behavior backward compatible.

## Reproduction Assessment

Not applicable: this is a new public Plugin SDK capability request, not a broken documented behavior. Source inspection confirms the adjacent registration, args, help/listing, and `continueAgent` pieces exist, while typed pass-through/body replacement does not.

## Solution Assessment

Unclear until maintainers choose the public SDK contract. The maintainable direction is an additive typed body-replacement continuation integrated with existing command authorization, help/listing, and native/text command paths, rather than plugin authors parsing hooks manually.
