## Label Justifications

- `P2`: This is a focused Feishu channel delivery bug with limited blast radius but real user-visible answer loss.
- `impact:message-loss`: The reported failure can silently drop assistant text produced before a tool call from the final Feishu-visible reply.

## Summary

Keep open: current main and the latest release still have the Feishu dispatcher path that can replace accumulated streamed text with the later final payload. The linked closing PRs target ACP `final_only`, while this issue has been narrowed to Feishu streaming-card reply delivery.

## What This Changes

The issue requests preserving assistant text emitted before tool calls in Feishu streaming card replies when the same turn continues after tool execution.

## Best Possible Solution

Fix the Feishu reply dispatcher to preserve or merge accumulated streamed text across assistant invocations before final card close, add a regression for partial pre-tool text plus a later final payload on the same card, and then reconcile the ACP-only PRs against that scope.

## Reproduction Assessment

Yes, at source level: Feishu partial callbacks accumulate text into `streamText`, but a later active-card final delivery assigns `streamText = text` and card close uses that overwritten value. I did not run a live Feishu credentialed reproduction.

## Solution Assessment

Yes for the Feishu-dispatcher fix direction, but no for treating the linked ACP-only PRs as sufficient. The narrow maintainable fix belongs in `extensions/feishu/src/reply-dispatcher.ts` with regression coverage for the partial-plus-final tool-call turn.
