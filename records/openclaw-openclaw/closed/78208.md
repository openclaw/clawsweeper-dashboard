## Summary

Keep open: current main still rejects `cron.status` and `cron.list` before dispatch when the cron tool is in self-remove-only scope, matching the reported v2026.5.4 failure. The remaining fix is valid but changes a cron owner-only authorization boundary, so it needs maintainer policy review before implementation.

## What This Changes

Request self-scoped cron `status`/`list` introspection for isolated cron runs so heartbeat-runner STATUS updates do not abort under v2026.5.4.

## Best Possible Solution

Add a scoped read-only cron introspection path that permits `status` and tightly filtered self/current-agent `list` while preserving the existing mutation guard for add/update/run/remove.

## Reproduction Assessment

Yes. Source inspection and existing tests show a self-remove-scoped cron tool rejects `list` before dispatch, and the same pre-switch guard rejects `status`; I did not run a live gateway in this read-only review.

## Solution Assessment

Unclear pending maintainer policy review. The narrow maintainable direction is read-only self-introspection only, with `list` forced to the current job or current agent scope and all mutation paths still denied.
