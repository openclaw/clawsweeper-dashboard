## Summary

Keep open: current main still source-reproducibly accepts duplicate cron job names, and https://github.com/openclaw/openclaw/pull/76445 is the active open fixing PR with closing syntax for this issue.

## What This Changes

Treat cron job names as unique during add/edit so duplicate rows cannot silently coexist, with optional replace/listing UX proposed for legacy duplicates.

## Best Possible Solution

Land or replace the open fixing PR with a service-layer uniqueness guard that compares normalized names and returns the existing gateway validation error shape; treat `--replace` and duplicate-list warnings as follow-ups if maintainers want that UX.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. Source inspection shows duplicate add reaches the unconditional `jobs.push`, and duplicate rename reaches row replacement without any sibling-name check; the issue body also gives concrete CLI steps.

## Solution Assessment

Is this the best way to solve the issue? The service-layer validation direction is right because CLI, gateway RPC, and agent-tool paths funnel through cron service operations. The full `--replace` and warning-marker UX needs maintainer product judgment and can be separated from the core duplicate-rejection fix.
