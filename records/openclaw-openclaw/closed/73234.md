## Summary

Current main implements the root-cause fix described in the issue discussion: the auto-TTS payload path now carries `spokenText` when it appends audio, and the WebChat assistant-media transcript path strips visible text from TTS supplement payloads before appending the media mirror. Existing regression coverage asserts the assistant-media transcript entry no longer contains the original assistant text.

## Best Possible Solution

Keep the current main implementation: auto-TTS should continue to populate `spokenText` at the speech-core source, while WebChat keeps stripping visible text from TTS supplement media mirrors. If follow-up work is needed, it should be a release note or narrower regression issue for any remaining surface that still receives a media payload without `spokenText`.
