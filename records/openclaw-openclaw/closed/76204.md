## Summary

Current main already prevents the reported empty-prompt provider call: blank prompt submissions are skipped before the embedded session calls the model, and runtime-only events with hidden context are converted to a non-empty marker prompt.

## What This Changes

The issue requests gateway-side validation so empty user prompts are dropped or rejected without poisoning provider auth-profile cooldowns.

## Best Possible Solution

Keep the current embedded-runner boundary guard and carry it into the next release; reopen only if logs show a distinct corrupt-session recovery path that still reaches providers with no prompt or media.

## Reproduction Assessment

Yes. The focused reproduction is a whitespace-only prompt with existing replay history or a runtime-only event with an empty transcript prompt; current tests prove the former is skipped and the latter gets a non-empty marker prompt.

## Solution Assessment

Yes. Handling this before `activeSession.prompt(...)` is the narrowest maintainable fix because it prevents malformed provider calls and therefore avoids auth-profile cooldown poisoning.
