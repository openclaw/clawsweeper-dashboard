## Summary

Current main already fails closed after a package-manager success that does not actually materialize every requested bundled runtime dependency, so the false-success path reported here is implemented on main rather than remaining open work. The linked runtime-deps reports cover adjacent stale-root, lock, and hash-race cases, but they do not supersede this specific post-install verification path.

## What This Changes

The issue reports that bundled plugin runtime dependency installation can log success after a partial npm install, leaving packages such as json5 missing and bundled channels crash-looping.

## Best Possible Solution

Keep the current post-install verification as the authoritative guard: every requested bundled runtime dependency must be present and version-satisfied before doctor or startup reports a successful repair. Timing heuristics are not needed as the primary fix because missing packages now fail by package identity.

## Reproduction Assessment

Yes. The current regression test reproduces the relevant failure mode by mocking a clean package-manager exit that installs nothing, and current main throws a missing-spec error instead of reporting success.

## Solution Assessment

Yes. Verifying actual package sentinels and versions after install is narrower and more reliable than treating fast installs as suspicious; it directly protects doctor, startup repair, and isolated staging copy-back paths.
