## Summary

Keep open. Current main still has a commitments-specific model routing gap: the hidden extractor calls the embedded runner without a provider/model, and that runner falls back to built-in direct OpenAI defaults, matching the reporter's observed `openai` auth failures on an `openai-codex` OAuth-only install.

## What This Changes

The issue asks for inferred commitment extraction to inherit the configured agent/default provider-qualified model, or provide an explicit commitments model/circuit breaker, instead of silently falling back to direct OpenAI.

## Best Possible Solution

Make the commitments extractor resolve the owning agent's effective/default model through the existing provider-qualified model-selection path, pass that provider/model/auth profile/agent dir into the embedded extractor run, and add focused regression coverage for `openai-codex/gpt-5.5` OAuth-only setups. A small commitments-local terminal auth/config cooldown is reasonable, while broad background retry policy should stay with #14376.

## Reproduction Assessment

Yes. The issue has concrete logs and current source gives a high-confidence static reproduction: enable commitments with an `openai-codex/gpt-5.5` default and no `OPENAI_API_KEY`, then the extractor omits model routing and the embedded runner selects `openai/gpt-5.5`.

## Solution Assessment

Yes for the model-inheritance direction: using the existing agent/default model resolver is the narrowest maintainable fix and preserves provider-qualified IDs, aliases, and auth profiles. An explicit commitments model setting can be added later if product wants a separate low-cost extractor model.
