## Summary

Issue #59099 is an obsolete report against a removed CLI path. Current main no longer supports `openclaw browser extension install/path`; the legacy Chrome extension relay was removed in the 2026.3.22 breaking browser migration, and the supported paths are now `openclaw browser` plus Chrome MCP or raw remote CDP depending on host layout.

## Evidence

- **Changelog documents the removal:** The 2026.3.22 changelog explicitly says OpenClaw removed the legacy Chrome extension relay path, bundled extension assets, `driver: "extension"`, and `browser.relayBindHost`, and tells users to migrate with `openclaw doctor --fix`. It references PR #47893, matching the useful explanation already posted in the issue thread.
  - file: [CHANGELOG.md:2224](https://github.com/openclaw/openclaw/blob/660cea680ae5d3d8b7a95c0e2fa7cf0da2f52a27/CHANGELOG.md#L2224)
  - command: `nl -ba CHANGELOG.md | sed -n '2214,2232p'`
- **Current browser CLI has no `extension` subtree:** On current main, the browser plugin registers `openclaw browser` and its supported subcommand groups only. There is no `browser extension ...` command family in the current CLI surface.
  - file: [extensions/browser/src/cli/browser-cli.ts:20](https://github.com/openclaw/openclaw/blob/660cea680ae5d3d8b7a95c0e2fa7cf0da2f52a27/extensions/browser/src/cli/browser-cli.ts#L20)
  - command: `nl -ba extensions/browser/src/cli/browser-cli.ts | sed -n '1,220p'`
- **Legacy config is migrated, not supported:** Current doctor normalization still rewrites legacy browser-extension config: it removes `browser.relayBindHost` and converts `browser.profiles.*.driver: "extension"` to `"existing-session"`, confirming the old extension path is retired rather than a still-supported feature.
  - file: [src/commands/doctor/shared/legacy-config-core-normalizers.ts:27](https://github.com/openclaw/openclaw/blob/660cea680ae5d3d8b7a95c0e2fa7cf0da2f52a27/src/commands/doctor/shared/legacy-config-core-normalizers.ts#L27)
  - command: `nl -ba src/commands/doctor/shared/legacy-config-core-normalizers.ts | sed -n '1,180p'`
- **Runtime keeps the old relay hook as a removal stub:** The runtime helper for the extension relay is now an intentional no-op because 'the Chrome extension relay path has been removed', and the test suite asserts that no-op behavior.
  - file: [extensions/browser/src/browser/server-lifecycle.ts:9](https://github.com/openclaw/openclaw/blob/660cea680ae5d3d8b7a95c0e2fa7cf0da2f52a27/extensions/browser/src/browser/server-lifecycle.ts#L9)
  - command: `nl -ba extensions/browser/src/browser/server-lifecycle.ts | sed -n '1,120p'`
- **Docs point users to the replacement flows:** Current docs describe `openclaw browser` and Chrome MCP as the supported local-browser path, and the WSL2 + Windows troubleshooting guide explicitly says to prefer raw remote CDP because Chrome MCP is host-local, which directly matches the reporter's WSL environment.
  - file: [docs/tools/browser-wsl2-windows-remote-cdp-troubleshooting.md:44](https://github.com/openclaw/openclaw/blob/660cea680ae5d3d8b7a95c0e2fa7cf0da2f52a27/docs/tools/browser-wsl2-windows-remote-cdp-troubleshooting.md#L44)
  - command: `nl -ba docs/tools/browser-wsl2-windows-remote-cdp-troubleshooting.md | sed -n '1,120p'`
- **Issue follow-up mixes in a separate warning:** The reporter's follow-up comment is about the 'Config was last written by a newer OpenClaw' warning after upgrade. That is a separate update/config-version concern and does not change the fact that `openclaw browser extension install/path` was intentionally removed before this issue was filed.

## Risks / Open Questions

- Current CLI parsing likely still falls through to Commander’s generic 'too many arguments' message for removed `browser extension ...` invocations, so there may still be a small UX papercut even though the underlying command path is intentionally gone.
