## Summary

Current main supersedes this PR with a maintainer-authored Telegram fix for the linked bug that deliberately keeps the plain grammY `Network request for '<method>' failed!` plus uncoded `TypeError("fetch failed")` envelope single-shot. The PR’s only functional change now conflicts with that explicit safety boundary, even though the branch itself is narrow and tested.

## What This Changes

The PR changes Telegram safe-send retry classification and tests so a plain grammY `Network request for '<method>' failed!` HttpError wrapping `TypeError("fetch failed")` is treated as retryable for non-idempotent sends.

## Best Possible Solution

Close this PR as superseded by the current-main maintainer fix for #74203. Keep the stricter safe-send boundary unless maintainers explicitly revisit the duplicate-delivery risk for uncoded plain grammY envelopes.
