## Summary

Keep open. The report is credible for the existing SecretRef-backed provider-auth contract, but current source still leaves the failing boundary split between native subagent dispatch, the active gateway secrets snapshot, ACP env stripping, and the reporter's custom provider mapping, so it is not a safe cleanup close.

## What This Changes

The issue asks for subagent or ACP sessions spawned through `sessions_spawn` to preserve env-backed provider API-key resolution from the gateway process.

## Best Possible Solution

Prove native subagent and ACP `sessions_spawn` auth separately, then preserve provider SecretRef resolution through the active gateway secrets snapshot or a narrow explicit handoff instead of forwarding the full host environment.

## Reproduction Assessment

Unclear. The issue has detailed reported steps and a 401, but I did not run a live systemd/gateway/Alibaba setup and source inspection leaves the failing branch split between native subagent config resolution, ACP env filtering, and custom provider auth.

## Solution Assessment

Unclear. Broad parent-environment inheritance is not the best default because it risks credential leakage; the safer direction is to keep SecretRef semantics flowing through the active runtime snapshot or add a constrained credential handoff only where reproduction proves a gap.
