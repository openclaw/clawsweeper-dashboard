## Summary

Current main and shipped releases already cover the reported external dynamic-peer/direct-message sandboxing path. The raw `non-main` helper remains per-session, but channel-originated direct messages that share an agent main conversation now receive a derived runtime-policy key, so sandbox/tool policy treats them as external non-main runtime work instead of local main-session work.

## What This Changes

The issue asks for a way to keep the trusted main agent unsandboxed while dynamic peer/direct-message agents remain sandboxed under `agents.defaults.sandbox.mode="non-main"`.

## Best Possible Solution

Keep the shipped runtime-policy-key behavior and documented per-session `non-main` semantics; use per-agent overrides only for local non-channel agent main sessions that must be sandboxed.

## Reproduction Assessment

No, not for the reported channel-originated dynamic-peer path on current main. Source and tests show external direct messages sharing a main conversation now resolve to a derived non-main runtime-policy key before sandbox resolution.

## Solution Assessment

Yes. The shipped fix is narrower and more compatible than redefining `non-main`: it preserves main-session conversation sharing while making sandbox/tool policy see external DMs as external runtime work.
