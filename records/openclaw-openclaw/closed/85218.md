## Label Justifications

- `P1`: The reported symptom is an auth regression that makes the MiniMax CN model provider unusable for affected users.
- `impact:auth-provider`: The issue is specifically about provider credential routing and outbound auth headers.

## Summary

Current main and the latest shipped release already cover the reported MiniMax CN missing-header failure: Anthropic-compatible API-key requests now include `x-api-key`, and `authHeader` is honored as a separate Authorization-header override. The linked PR targets Bearer-only custom gateways, while this issue’s MiniMax CN error asks for the API key header that v2026.5.20 already sends.

## What This Changes

The issue reports that MiniMax CN custom Anthropic endpoint calls fail with 401 because runtime auth sends the wrong header despite `authHeader` configuration.

## Best Possible Solution

Keep the shipped v2026.5.20 Anthropic API-key transport; any remaining MiniMax CN failure should be handled as a new MiniMax-specific header-precedence report with current-version evidence.

## Reproduction Assessment

No high-confidence live reproduction was run because that requires a MiniMax CN credential. Source and released-test inspection show the reported missing `X-Api-Key` runtime path no longer matches v2026.5.20/current main.

## Solution Assessment

Yes. The narrow shipped fix is to send `x-api-key` on Anthropic-compatible API-key requests while preserving the separate `authHeader` override path; Bearer-only gateway behavior belongs to the linked custom-provider PR, not this MiniMax CN issue.
