## Label Justifications

- `P1`: The report describes a real ACP `sessions_spawn` workflow failing every time for OpenCode when automatic model passthrough reaches a harness without model-control capability.
- `impact:auth-provider`: The failure is in model-selection/provider routing into ACP session startup, where an OpenClaw default model is sent to a harness that cannot accept model control.

## Summary

Keep open: current main still passes configured ACP spawn model options into the acpx runtime before the harness has proven model-control support, and acpx 0.10.0 throws the reporter's exact unsupported-model error when no ACP model list is advertised.

## What This Changes

The issue asks OpenClaw to skip automatic ACP model passthrough for harnesses that do not advertise model support, while letting the harness use its own configured model.

## Best Possible Solution

ACP spawns should treat automatic/default model choices as best-effort for ACP harnesses: apply them only when the adapter supports model control, while explicit model requests still fail clearly when unsupported.

## Reproduction Assessment

Yes, by source inspection: configure an ACP agent whose session creation returns no ACP `models`, let `sessions_spawn(runtime: "acp")` resolve an automatic configured model, and acpx 0.10.0 throws the reported unsupported-model error. I did not run a live OpenCode process in this read-only review.

## Solution Assessment

Yes, the maintainable fix is to distinguish explicit model requests from automatic/default model selection and only make the automatic choice fatal when the ACP adapter can actually honor it. A broad config flag or unconditional removal of ACP model passthrough would be too coarse.
