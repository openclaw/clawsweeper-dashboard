## Summary

Keep open: this PR targets a real Barnacle/ClawSweeper proof-gate problem, but the patch currently lets PR commenters spoof a pass marker and may fetch comments with a token that lacks issue-comment read permission. The protected `maintainer` label also means this needs explicit maintainer handling, not cleanup closure.

## What This Changes

The PR adds exact-head ClawSweeper proof-verdict marker handling to the real-behavior-proof check, Barnacle proof-label synchronization, and related tests.

## Best Possible Solution

Keep the exact-head verdict idea, but only trust markers from the ClawSweeper GitHub App or bot identity, use a token that can read issue comments with a safe fallback, and add forged-comment plus token-path regression coverage before merge.

## Reproduction Assessment

Yes by source inspection and related PR timeline. Current main does not read ClawSweeper verdict comments in the required proof check, and the related PR at https://github.com/openclaw/openclaw/pull/83581 shows an exact-head pass marker while the Real behavior proof check remained failing.

## Solution Assessment

No. The direction is maintainable, but this implementation must authenticate the marker source and use a token with issue-comment read permission before it is safe to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Authenticate ClawSweeper verdict comments:** `scripts/github/real-behavior-proof-policy.mjs:252-259`
  - body: `hasClawSweeperExactHeadProof` scans every issue comment body and accepts a matching PR number plus head SHA, but it never verifies that the comment was posted by the ClawSweeper GitHub App or bot. An external contributor can post that hidden marker themselves and make the required Real behavior proof check pass without accepted proof, so the marker should be accepted only from a trusted ClawSweeper comment and covered with a forged-comment regression test.
  - confidence: 0.96
- **[P2] Use an issue-readable token for comment fetching:** `scripts/github/real-behavior-proof-check.mjs:54`
  - body: The new comment fetch prefers `appToken` whenever it exists, but the workflow creates that app token with only `permission-members: read`; the added job-level `issues: read` applies to `GITHUB_TOKEN`, which this code will skip. In the intended exact-head fallback path, a members-only app token can make the check throw on the comments API instead of reading the marker, so request `permission-issues: read` for the app token or retry with `GITHUB_TOKEN` on permission failure.
  - confidence: 0.88
