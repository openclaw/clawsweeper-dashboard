## Summary

Current main already implements the requested messaging-surface interrupt/steer behavior: queue mode `steer` injects inbound messages into an active run at the next model boundary, `/stop` aborts and clears queued work, and Telegram abort-like messages bypass the normal session lane via a dedicated control lane. The same behavior is present in release `v2026.4.22`.

## Evidence

## Risks / Open Questions

- The inspected implementation is soft-boundary steering/abort, not a guaranteed hard kill in the middle of an individual tool call.
- The exact `/kill` alias proposed in the issue was not verified; the implemented control path uses `/stop` and recognized stop-like phrases instead.

## Close Comment

Closing as `implemented_on_main` after Codex review. Current main already covers the core request from messaging surfaces: queue mode `steer` injects inbound messages into a running turn at the next model boundary (`docs/concepts/queue.md:29`, `docs/concepts/agent.md:84`, `src/auto-reply/reply/agent-runner.ts:969`), `/stop` aborts the active run and clears queued followups/sub-agent work (`src/auto-reply/reply/commands-session-abort.ts:129`), and Telegram stop-like messages are routed on a dedicated `:control` sequential lane so they do not sit behind the busy chat lane (`extensions/telegram/src/sequential-key.ts:66`, covered by `extensions/telegram/src/sequential-key.test.ts:59`). I also verified these paths are already present in release `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). If you meant a stricter hard-kill semantic inside a single in-flight tool call, that would need a narrower follow-up issue, but the feature requested here is already implemented.
