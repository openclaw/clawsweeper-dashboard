## Summary

Close as duplicate/superseded. #65526 is a feature-form duplicate of the same MCP stdio child-process env inheritance problem tracked in #65525, and open PR #65574 explicitly targets and fixes both #65525 and #65526. Current main has not implemented the requested propagation, so the right cleanup is to keep the canonical bug/PR open rather than keep two parallel reports.

## Best Possible Solution

Close #65526 as the duplicate feature-form report, and keep #65525 plus PR #65574 as the canonical active tracking path. The eventual fix should be resolved there with a narrow stdio MCP environment policy for proxy vars and an explicit decision for `NODE_OPTIONS` that stays consistent with the current stdio startup safety filter and docs.

## Evidence

- **Reporter linked canonical bug:** The only comment on #65526 says: "Bug report with full root cause trace: #65525". The #65525 body describes the same MCP stdio child-process failure: `NODE_OPTIONS` and proxy environment variables from the gateway are not propagated, breaking proxy-only sandbox MCP servers.
- **Open PR supersedes this issue:** The provided GitHub context for PR #65574 shows it is open, titled `fix(mcp-stdio): propagate proxy env vars to child processes`, changes `src/agents/mcp-stdio.ts` and `src/agents/mcp-stdio.test.ts`, and explicitly lists `Fixes #65525` and `Fixes #65526`.
- **Current stdio launch config still lacks ambient proxy defaults:** Current main returns `env: toMcpEnvRecord(raw.env, ...)`, which means only user-configured stdio MCP env is forwarded after filtering; there is no merge of ambient `HTTP_PROXY`, `HTTPS_PROXY`, `NO_PROXY`, lowercase variants, or `NODE_OPTIONS`.
