## Label Justifications

- `P1`: The report is a recent regression that blocks all agent turns for users whose primary and fallback models are Google Vertex ADC-backed models.
- `impact:auth-provider`: The failure is in Google Vertex provider auth/model availability, where ADC credentials are recognized in one path but not preserved into model config generation.
- `impact:message-loss`: Affected users can send messages but the agent fails before producing a usable model response when every Google Vertex candidate resolves as missing.

## Summary

Keep open: this still looks actionable on current main because Google Vertex ADC auth evidence resolves to a non-secret marker for runtime auth, but the models.json generation path only fills missing provider apiKey values from env-var names or profiles, so ADC-backed static catalog rows can still be dropped before embedded runtime model resolution.

## What This Changes

The issue reports a regression where `google-vertex/*` models using authorized_user ADC credentials fail at runtime with `model_not_found` after upgrading to 2026.5.28 or 2026.6.1, despite direct Vertex API calls succeeding.

## Best Possible Solution

Propagate resolved Google Vertex ADC auth evidence into generated provider config as `gcp-vertex-credentials`, then prove embedded runtime resolves `google-vertex/*` catalog models and the transport uses ADC bearer auth without requiring a manual auth profile or custom `models.providers` entry.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: set ADC auth evidence without a `google-vertex` auth profile and the current models.json writer has no path to fill the static catalog provider apiKey with `gcp-vertex-credentials`. A live Vertex turn was not run in this read-only review.

## Solution Assessment

Yes, the best fix is to repair the existing auth-evidence-to-model-config path so ADC-backed Google Vertex providers stay registered. Adding a new config option or telling users to manually add `models.providers.google-vertex.models[]` would be a workaround, not the right product fix.
