## Label Justifications

- `P1`: The PR targets a Gateway/MCP shutdown leak that can accumulate child processes across restarts and degrade real agent workflows.
- `merge-risk: 🚨 compatibility`: The patch changes MCP stdio shutdown behavior by force-killing surviving children before `close()` returns, which can affect existing slow-cleanup MCP servers.

## Summary

Keep open for normal merge review: the PR addresses a real current-main MCP stdio shutdown race by delivering the existing process-tree SIGKILL before `close()` returns, and I found no blocking code or security findings. Maintainers should consciously accept the bounded hard-kill behavior because it changes MCP stdio shutdown semantics.

## What This Changes

The PR adds a synchronous SIGKILL fallback to `OpenClawStdioClientTransport.close()` and a regression test for SIGTERM-ignoring MCP stdio children.

## Best Possible Solution

Land the focused transport-level fix once maintainers accept the shutdown grace tradeoff, and keep remote SSH process cleanup as separate design work.

## Reproduction Assessment

Yes, source-reproducible: current main returns from `close()` after scheduling the unref'd kill-tree SIGKILL rather than delivering it, and the PR body provides before/after harness output for a SIGTERM-ignoring child. I did not run the live harness in this read-only review.

## Solution Assessment

Yes, with maintainer acceptance of the hard-kill timing: the transport owns the child pid and bundle cleanup already delegates to `transport.close()`. Adding a kill in `disposeSession()` would duplicate local transport cleanup and still would not solve remote SSH child ownership.
