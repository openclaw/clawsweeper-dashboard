## Summary

Keep open: the PR is a focused diagnostic fix and now includes terminal proof for the default `/workspace` resolver case, but the patch still hard-codes the hint in the active escape throw so configured Docker workdirs can receive the wrong recovery path.

## What This Changes

The PR extends sandbox path-escape error messages in the shared path guard, sandbox fs mount resolver, OpenShell fs bridge, and changelog to include a container workspace hint.

## Best Possible Solution

Keep the diagnostic-only scope, but route path-escape errors through code that can use the configured container workdir and cover both default and custom workdir cases.

## Reproduction Assessment

Yes, source-reproducible: current main rejects escaping paths through `resolveSandboxPath` and `resolveSandboxFsPathWithMounts` without a container hint. The PR body also shows a live Node/tsx invocation proving the default hint after the patch.

## Solution Assessment

No. The diagnostic-only direction is the right narrow fix, but this implementation puts the configured-root hint after an earlier throw and hard-codes `/workspace` in the active guard.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Pass the configured workdir into the escape hint:** `src/agents/sandbox-paths.ts:72`
  - body: `resolveSandboxPath` only receives the host root, so this new hint hard-codes `/workspace` for every caller. In `resolveSandboxFsPathWithMounts`, escaping host paths hit this guard before the later `params.defaultContainerRoot` throw can run, so a sandbox configured with `docker.workdir: "/sandbox-root"` still tells the agent to use `/workspace/`, which is the wrong recovery path. Thread the configured container root to the active throw and cover custom workdirs.
  - confidence: 0.9
