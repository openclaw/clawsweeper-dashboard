## Label Justifications

- `P2`: The reported failure removes Telegram's native command menu but leaves the gateway and bot otherwise running, making it a normal-priority channel UX regression.
- `impact:other`: The concrete impact is loss of Telegram bot menu entries from stale sync state, which is maintainer-visible but outside the narrower owned impact labels.

## Summary

Current main and shipped releases already remove the stale file-backed Telegram command-hash cache that made restarts skip `setMyCommands`; the remaining same-process/custom-command concerns are separate from this stale-local-file failure mode.

## What This Changes

The issue asks OpenClaw to avoid leaving Telegram bot commands empty when a stale local command-hash cache says the menu is already synchronized.

## Best Possible Solution

Keep the shipped process-local command-sync cache for this stale-file restart failure, and track any separately reproduced same-process remote reconciliation or custom-command race under the existing Telegram command-menu follow-up.

## Reproduction Assessment

No high-confidence current-main reproduction path remains. The old v2026.4.14 path is source-reproducible through `command-hash-*.txt`, but current main and `v2026.4.15` use only process-local state, so a restart cannot trust the stale file.

## Solution Assessment

Yes for this failure mode. Replacing the persisted hash with a process-local cache is the narrowest maintainable fix because it preserves rapid-restart rate-limit protection within one process while forcing re-registration after restart.
