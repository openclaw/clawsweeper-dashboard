## Summary

Keep open: current main still advertises and creates OpenAI OAuth-backed Realtime Talk sessions without proving the `/v1/realtime/calls` media handshake works, and the related docs work only clarified Platform billing rather than resolving this HTTP 500 failure mode.

## What This Changes

The issue reports that OpenAI OAuth-backed Realtime Talk appears configured without a direct API key but every WebRTC or relay media handshake fails before audio becomes ready.

## Best Possible Solution

Keep the issue open until maintainers decide the supported OpenAI Realtime auth contract, then either prove OAuth-backed `/v1/realtime/calls` works in a live audit or gate/error the no-key OAuth path with clear user guidance and regression coverage.

## Reproduction Assessment

Yes for the OpenClaw readiness/reporting side: current source shows OAuth can mark OpenAI Realtime configured and create a WebRTC session without a media-handshake audit. No for the actual OpenAI HTTP 500, because this read-only review did not run a live OAuth Realtime call.

## Solution Assessment

Unclear; the current behavior is not enough to close because it can advertise and mint sessions for a path reported to fail. The best fix depends on whether maintainers want OAuth-backed Realtime to remain supported with live proof or fail closed toward a direct funded Platform API key.
