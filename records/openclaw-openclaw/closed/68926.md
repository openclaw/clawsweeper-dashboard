## Best Possible Solution

Keep the shipped shared chat-controls implementation from current main and close this stale PR rather than merging its older duplicated selector logic.

## Solution Assessment

Yes for current main, not for the PR branch. The shipped shared helper is the narrower maintainable solution because it scopes sessions by agent while preserving the existing model/thinking metadata contract.

Overall confidence: 0.92

Full review comments:

- **[P2] Derive agent targets from a plain main key:** `ui/src/ui/app-render.helpers.ts:178`
  - body: When the active session is a legacy/non-agent key, this branch passes resolveSidebarChatSessionKey(state) into buildAgentMainSessionKey as a main key. That value can already be a full key such as agent:main:main, so selecting another agent can route to an invalid nested key like agent:beta:agent:main:main.
  - confidence: 0.91
- **[P2] Preserve session-provided thinking options:** `ui/src/ui/app-render.helpers.ts:722-751`

Sessions advertising levels such as adaptive, xhigh, max, or provider-specific labels can lose valid choices and show the wrong inherited default.
  - confidence: 0.93
