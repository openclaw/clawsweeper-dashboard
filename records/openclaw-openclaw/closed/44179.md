## Summary

Keep this open: current main still has source-level paths where model special tokens can reach visible assistant output, and the TUI-only fix PR was closed unmerged. The later `openclaw agent --json` evidence usefully broadens the needed fix beyond the original TUI resolver.

## What This Changes

The issue asks OpenClaw to apply existing model-special-token sanitization to primary visible assistant output paths so tokens like `<|assistant|>` do not leak into chat, JSON results, or persisted transcripts.

## Best Possible Solution

Centralize the existing assistant-visible sanitizer at final display/result/transcript boundaries, with regression coverage for TUI final/fallback text, `openclaw agent --json` payloads, and CLI transcript persistence.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: token-bearing `finalText` or fallback `streamedText` passed to `resolveFinalAssistantText` is returned unchanged, and internal agent command payload text can be projected to JSON or transcripts without `sanitizeAssistantVisibleText`.

## Solution Assessment

No to the original TUI-only patch boundary; yes to reusing the shared `sanitizeAssistantVisibleText` pipeline at the shared assistant-visible output boundaries. That avoids duplicating regexes and covers the later JSON/transcript evidence without adding provider-specific logic.
