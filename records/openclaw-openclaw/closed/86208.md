## Label Justifications

- `P2`: The report describes a workflow-blocking cron model-call timeout with a workaround and a fixed newer release.
- `impact:crash-loop`: The issue is about repeated isolated cron job timeouts that make the affected automation unavailable.
- `impact:auth-provider`: The affected surface is provider/model-call timeout handling for an OpenAI-compatible DeepSeek provider.

## Summary

Current main and the latest release already cover the reported timeout boundary: isolated cron payload timeouts are preserved into embedded runs, provider timeoutSeconds feeds OpenAI-compatible request timeouts and the model idle watchdog, and docs describe the provider-timeout versus whole-run-timeout split. The report was against OpenClaw 2026.5.20, while the fixed surface is present in v2026.5.22 and current main.

## What This Changes

Isolated cron model calls using slow DeepSeek/OpenAI-compatible providers should honor timeout settings instead of timing out at model-call-started.

## Best Possible Solution

Keep the shipped v2026.5.22 behavior; affected 2026.5.20 users should upgrade and set models.providers.<id>.timeoutSeconds for slow DeepSeek/OpenAI-compatible providers while keeping cron timeoutSeconds at or above that ceiling.

## Reproduction Assessment

No live DeepSeek V4 Pro reproduction was run, but the source path is clear: isolated cron payload timeouts flow into embedded model-call timeout handling, and current tests cover the reported timeout boundary. The supplied issue body gives concrete environment and symptom data against OpenClaw 2026.5.20.

## Solution Assessment

Yes; the current implementation is the maintainable split between whole-run cron timeout and provider HTTP/model-call timeout. It fixes the timeout path without adding another config surface or making cron timeouts silently override provider transport policy.
