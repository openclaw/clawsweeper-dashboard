## Summary

Keep this PR open. Current main still lacks Discord duplicate-token account suppression, and the PR remains a plausible implementation candidate for the duplicate DM processing bug. However, the supplied diff appears incomplete for the actual monitor startup path because Discord still exposes raw account ids to the gateway, and gateway startup starts each configured account independently after resolving `isConfigured`. The earlier ClawSweeper review concern still holds against current main SHA 230f7122dd5cd8a37a55e671254bd9be8278f6e8.

## Best Possible Solution

Keep this PR open for maintainer review, but request changes before landing. The fix should make duplicate-token ownership visible to the gateway startup path, likely in Discord plugin config `isConfigured`/`unconfiguredReason` or an equivalent gateway-consumed startup account surface, prefer explicit config over env fallback for the kept account, preserve status diagnostics for the dropped account, and add a regression test proving only one Discord monitor starts when `default` resolves `DISCORD_BOT_TOKEN` and a named account has the same configured token.
