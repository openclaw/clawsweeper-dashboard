## Summary

Keep open. Current `main` still uses one cron-owned `AbortController` for the whole timed job, passes that same aborted signal into isolated cron execution, and forwards it unchanged into each `runWithModelFallback` attempt. I found no shipped fix on `main` or in `v2026.4.22`; the relevant timeout code is unchanged there.
