## Summary

Keep open. Current main still lets Telegram no-caption photo-shaped messages seed `bodyText` from the message-shape `<media:image>` placeholder before saved media MIME is considered, and the related broad PR is closed unmerged.

## What This Changes

The issue asks for Telegram inbound photo placeholders to reflect the downloaded media `contentType` instead of always telling the model `<media:image>`.

## Best Possible Solution

Add a Telegram-local media placeholder classifier that uses saved `allMedia[].contentType` when media exists, preserves message-shape placeholders for missing downloads, and covers document/unknown, video, mixed attachment, audio, and multi-image cases in focused tests.

## Reproduction Assessment

Yes. A focused unit reproduction is to call `resolveTelegramInboundBody()` with no user text, `msg.photo`, and `allMedia: [{ path, contentType: "application/octet-stream" }]`; current source returns `<media:image>`. I did not execute it because this review was required to keep the checkout read-only.

## Solution Assessment

Yes. Deriving the no-user-text placeholder from saved media MIME in the Telegram body path is the narrow maintainable fix; the hardcoded `image/jpeg` fallback discussed in the issue is not the right solution.
