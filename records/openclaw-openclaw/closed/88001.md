## Label Justifications

- `P1`: The report describes a reproduced real-user cron workflow that returns ok while silently skipping remaining automation steps after `image_generate`.
- `impact:session-state`: The affected behavior is run/session lifecycle drift: the isolated cron turn closes before the callback can update the intended run state.
- `impact:message-loss`: The generated-media completion is delivered to the external channel but not delivered to the cron agent turn that needs it to continue the workflow.

## Summary

Keep open: the related stale-callback fix does not cover this report. Current main still has a source-backed path where `image_generate` starts an async media task, the turn can be treated as terminal progress, and an inactive run-scoped cron completion is delivered directly to the channel instead of resuming the cron workflow.

## What This Changes

Requests that isolated cron `agentTurn` jobs using `image_generate` receive the media completion callback and continue the remaining workflow instead of finishing early with a direct fallback message.

## Best Possible Solution

Fix the run-scoped isolated cron media completion path so the cron run remains resumable or pending until the generated-media callback is processed by the agent turn, and only falls back to direct channel delivery when the run is truly stale or abandoned.

## Reproduction Assessment

Yes for source-level reproduction: current code starts async `image_generate`, treats async-started tool activity as terminal progress, and direct-delivers inactive run-scoped cron media completions. A live provider cron run was not executed in this read-only pass.

## Solution Assessment

No, the current related fix is not the best complete solution because it handles stale callback routing but still leaves a valid isolated cron workflow unable to continue after the image callback. The better fix is a narrow cron/media lifecycle repair with regression coverage for callback continuation.
