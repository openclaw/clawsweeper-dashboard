## Summary

Current `main` serializes Telegram inbound processing per chat/topic before message handling, and the handler awaits full context/session setup plus reply dispatch. The reported same-chat duplicate-session initialization race does not match the current implementation, and there is an explicit regression test that holds the first turn open and verifies the second same-topic message does not begin until the first completes.

## Evidence

- **Same-chat/topic updates are serialized before Telegram handlers run:** `bot.use(botRuntime.sequentialize(getTelegramSequentialKey))` installs a per-chat/per-topic lane ahead of inbound handlers, so a second ordinary message for the same Telegram chat cannot enter the handler pipeline while the first is still running.
  - file: [extensions/telegram/src/bot-core.ts:412](https://github.com/openclaw/openclaw/blob/a8edf29bd0f1bd3c869e5e3fab91043d01e6ac33/extensions/telegram/src/bot-core.ts#L412)
  - command: `nl -ba extensions/telegram/src/bot-core.ts | sed -n '368,416p'`
  - sha: [a8edf29bd0f1](https://github.com/openclaw/openclaw/commit/a8edf29bd0f1bd3c869e5e3fab91043d01e6ac33)
- **The per-message processor holds the lane until context build and dispatch finish:** `createTelegramMessageProcessor` awaits `buildTelegramMessageContext(...)` and then awaits `dispatchTelegramMessage(...)`, so the sequentialized lane is not released during session/context initialization or reply delivery.
  - file: [extensions/telegram/src/bot-message.ts:71](https://github.com/openclaw/openclaw/blob/a8edf29bd0f1bd3c869e5e3fab91043d01e6ac33/extensions/telegram/src/bot-message.ts#L71)
