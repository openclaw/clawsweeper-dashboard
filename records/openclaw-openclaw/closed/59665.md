## Summary

Close: the request to mirror the login-shell PATH is superseded by the current shipped macOS Gateway policy, which intentionally writes a canonical LaunchAgent PATH and avoids copying user/version-manager shell paths.

## What This Changes

The issue requests that `openclaw gateway install` capture the current macOS login-shell PATH into the generated Gateway LaunchAgent plist.

## Best Possible Solution

Keep the shipped canonical macOS LaunchAgent PATH policy and route user-specific tool availability through explicit command paths, managed env, or a separately approved opt-in service-PATH design.

## Reproduction Assessment

Yes at source level: current main still does not make the LaunchAgent PATH match a login shell, because `buildServiceEnvironment(..., platform: "darwin")` returns the canonical service PATH and tests assert that behavior. I did not run a live macOS launchd install in this read-only review.

## Solution Assessment

No: copying the full login-shell PATH is not the best default for the shipped service environment policy. The safer current solution is canonical PATH plus explicit command paths or a maintainer-approved opt-in for additional user dirs.
