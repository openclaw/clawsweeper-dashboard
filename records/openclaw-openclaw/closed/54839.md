## Summary

Current `main` already enforces loopback-by-default gateway binding, rejects non-loopback binds without auth, and applies browser-origin checks. The report’s claim that a default npm install exposes a globally reachable unauthenticated port does not match the current implementation.

## Evidence

- **Gateway bind default is loopback:** `GatewayConfig.bind` documents `loopback` (`127.0.0.1`) as the default for the Gateway WS + Control UI server.
  - file: [src/config/types.gateway.ts:401](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/src/config/types.gateway.ts#L401)

  - file: [src/gateway/net.ts:298](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/src/gateway/net.ts#L298)

## Risks / Open Questions

- I could not inspect GitHub-side follow-up discussion or older release history because `gh` was unauthenticated in this environment, so I did not determine the earliest shipping release for these protections.
- Container installs are a special case: unset `gateway.bind` can resolve to `auto`/`0.0.0.0` for port-forwarding compatibility, but startup still refuses a non-loopback bind without auth and the issue specifically reports an Ubuntu npm install rather than a containerized deployment.
