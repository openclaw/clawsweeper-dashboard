## Summary

Current `main` and the shipped `v2026.4.23` release already implement the shared-session cron delivery behavior that matches the reporter's diagnosed root cause: `current` binds to a persisted `session:<key>`, isolated-like agent turns default to announce/shared delivery, and message-tool sends to the current target suppress fallback delivery instead of losing Discord delivery capability.

## Evidence

- **Reporter follow-up root cause:** The provided issue comment says the local live fix was changing shared-session scheduled wakes so `sessionTarget === "current"` and `sessionTarget.startsWith("session:")` use the shared delivery contract instead of a cron-owned contract, and that a live Discord cron canary then delivered successfully.
  - command: `Provided GitHub context comment 4297888984`
- **Current-session binding and default announce delivery:** Cron normalization resolves `sessionTarget: "current"` into `session:<sessionKey>` when session context exists, falls back to `isolated` otherwise, and auto-adds `delivery: { mode: "announce" }` for isolated-like agent turns with no explicit delivery config.
