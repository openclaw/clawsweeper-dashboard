## Summary

Keep open. Current main still resolves media audio provider auth through `resolveApiKeyForProvider` and `requireApiKey` before calling the plugin's `transcribeAudio`, and the linked closing implementation at https://github.com/openclaw/openclaw/pull/75005 is still open and unmerged.

## What This Changes

The issue asks OpenClaw to support explicitly local or no-auth media-understanding audio providers without failing early on missing API-key credentials.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/75005 or an equivalent provider-owned media synthetic-auth hook that preserves normal auth precedence and fail-closed behavior for unmarked providers.

## Reproduction Assessment

Yes, as a source-level reproduction. Configure a plugin-only media audio provider without credentials and trigger transcription; current main resolves provider auth and can throw before `transcribeAudio` is called.

## Solution Assessment

Yes in direction, but it is not on main yet. A provider-owned synthetic/no-auth hook is narrower than bypassing the media auth gate globally because configured keys, env auth, profiles, and explicit profile errors can still take precedence.
