## Summary

Current `main` already preserves cron/subagent delivery context and routes cron descendant results back through the cron job's configured delivery path, with regression coverage for the reported failure mode.

## Evidence

- **Cron requester sessions are treated as internal announce targets:** `isInternalAnnounceRequesterSession()` now returns true for cron session keys, preventing child completion announces from being sent directly to an unrelated active main-session channel.
  - file: [src/agents/subagent-announce-delivery.ts:167](https://github.com/openclaw/openclaw/blob/250acdd2a28c190ed7934e3ba99022f24123d3c9/src/agents/subagent-announce-delivery.ts#L167)

  - sha: [250acdd2a28c](https://github.com/openclaw/openclaw/commit/250acdd2a28c190ed7934e3ba99022f24123d3c9)
- **Regression test covers cron-run child announces staying on the cron session:** The timeout regression test asserts a child announce for `agent:main:cron:...:run:...` targets the cron run session with `deliver=false` and no outward `channel`/`to`/`accountId`, avoiding the misroute described in the issue.
  - file: [src/agents/subagent-announce.timeout.test.ts:375](https://github.com/openclaw/openclaw/blob/250acdd2a28c190ed7934e3ba99022f24123d3c9/src/agents/subagent-announce.timeout.test.ts#L375)
