## Summary

Keep open. Current main still has the Telegram per-chat/per-topic sequencer that can block immediate queue modes behind a busy topic lane, so the PR targets a real gap, but the current patch is not mergeable as-is: it diverges from core queue defaults, ignores per-instance bot config, and checks only reply-run activity instead of the full embedded-run busy state.

## What This Changes

The PR wraps Telegram sequential-key selection with a busy-session control lane for immediate queue modes, adds Telegram tests for the routing behavior, and exports reply-run active lookup through the plugin SDK.

## Best Possible Solution

Land an equivalent Telegram-owned busy-session bypass after rebasing onto current main, using the same queue-mode resolution semantics as core, the bot instance config, and a generic busy-state seam that covers both reply operations and embedded-only active runs, with focused Telegram/core tests and the required changelog entry.

## Reproduction Assessment

Yes. A focused test can create a busy Telegram session with no explicit queue mode and show current main keeps ordinary messages on the topic key while core resolves the queue mode to steer; PR-specific regression tests can cover default config, opts.config override, and embedded-only active runs.

## Solution Assessment

No. The design direction is narrow and appropriate for Telegram, but this patch is not the best implementation until it reuses or mirrors core queue settings, honors createTelegramBotCore's instance config, and routes busy-state through the same embedded-run-aware path that core uses.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Match the core queue default:** `extensions/telegram/src/sequential-key.runtime.ts:95`
  - body: Core resolveQueueSettings defaults an unset queue mode to steer, but this wrapper falls back to collect. With the default Telegram config, a busy session stays on the ordinary topic lane even though core would treat the turn as immediate steering, so the main path remains blocked.
  - confidence: 0.92
- **[P2] Treat embedded-only runs as busy:** `extensions/telegram/src/sequential-key.runtime.ts:173`
  - body: This gate only checks isReplyRunActiveForSessionKey, while core busy-state resolution also covers embedded runs that can exist without a reply operation. In that state, immediate Telegram turns still fall back to the topic lane and cannot reach core interrupt/steer handling promptly.
  - confidence: 0.88
- **[P2] Use the bot instance config for lane policy:** `extensions/telegram/src/bot-core.ts:551`
  - body: createTelegramBotCore can be constructed with opts.config, but the new sequential-key wrapper reloads global runtime config instead. A webhook, polling session, or test using an instance override can therefore choose the wrong queue mode or command policy for the bypass decision.
  - confidence: 0.86
