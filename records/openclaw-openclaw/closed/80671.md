## Summary

I found no blocking code or security findings in the merged patch.

## What This Changes

The PR adds Telegram bot-token fingerprints to long-poll offset state, resets stale or legacy offsets during polling startup, logs and deletes stale state, and adds regression coverage plus a changelog entry.

## Best Possible Solution

Keep the merged fingerprinted offset-state implementation on main and ship it in the next release; no separate repair PR is needed.

## Reproduction Assessment

Yes, at source level: the released/pre-merge code stored only botId plus lastUpdateId and seeded the update tracker, which skips update_id values at or below that watermark. The merged main no longer reproduces because it discards fingerprintless or mismatched token state before polling.

## Solution Assessment

Yes: persisting a short token fingerprint and invalidating fingerprintless or mismatched offset state is the narrow maintainable fix. It also avoids reintroducing the pre-poll getUpdates probe that was removed to prevent Telegram 409 self-conflicts.
