## Summary

Current main implements the supported Tailscale Serve WebSocket path: Serve exposes `wss://<magicdns>`, the WS handshake uses the Tailscale-aware auth surface, Tailscale identity headers are verified with `tailscale whois`, and tests cover the WS wrapper. Mobile pairing/docs now steer Tailscale/public clients away from raw `ws://<tailscale-ip>:18789` and toward Serve/Funnel `wss://` URLs. The latest release tag `v2026.4.24` contains the same implementation, and the provided related-item context shows the later matching Tailscale Serve WebSocket report #54008 was closed as implemented.

## Best Possible Solution

Close this broad issue as implemented. Keep the supported core path as Tailscale Serve/Funnel over `wss://<magicdns>` with Tailscale identity verification for WebSocket handshakes; leave alternative direct-tailnet/dual-bind behavior for a new, narrower issue only if a current supported client still needs it after using the documented `wss://` Serve path.
