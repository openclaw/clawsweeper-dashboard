## Summary

Current main contains a targeted fix for the reported async media-generation failure path: failed image generation completions now surface directly when the requester-session handoff fails, avoiding the stalled recovery path that could lead to a later phantom media send. The fix is on main only and is newer than the latest tagged release.

## What This Changes

The issue reports that an OpenAI `image_generate` safety rejection can stall session recovery and leave a planned-but-unwritten media path that Telegram later tries to send.

## Best Possible Solution

Keep the current-main async failure delivery behavior and let the next release carry it; broader dropped-media visibility can continue separately in https://github.com/openclaw/openclaw/pull/69310.

## Reproduction Assessment

No current-main failure run was executed, but the report’s logs match the async media-generation failure handoff that commit 9a36e897be8ac907bef273c08d097d27562d9235 now covers with source and regression-test evidence.

## Solution Assessment

Yes. Directly delivering failed async media-generation completions is the narrowest maintainable fix for the recovery stall path, while successful image media paths remain tied to completed `saveMediaBuffer` writes.
