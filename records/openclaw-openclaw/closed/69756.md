## Summary

Current main fixes the 2026.4.15 webchat history placeholder regression by redacting persisted base64 audio payloads before `chat.history` applies its oversized-message budget. The related discussion and prior #65238 context point to the same placeholder symptom, and the fix shipped in v2026.4.22.

## Best Possible Solution

Keep the shipped bounded `chat.history` behavior: redact persisted base64 audio before response sizing, retain the placeholder only as a safety fallback for truly oversized entries, and ask for a fresh report with transcript shape/logs if the placeholder still appears on v2026.4.22 or newer for non-audio content.
