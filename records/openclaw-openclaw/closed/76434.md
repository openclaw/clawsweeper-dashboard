## Summary

This PR should stay open because the warning is a reasonable narrow diagnostic, but the branch also advertises WhatsApp configured-binding provider parity while current WhatsApp inbound routing still does not call the configured/runtime binding route helpers.

## What This Changes

This PR adds WhatsApp configured-binding provider helpers and tests, plus a one-time runtime warning for WhatsApp route bindings that appear to use `match.accountId` as a sender phone instead of `match.peer.id`.

## Best Possible Solution

Keep the diagnostic if maintainers want it, and either split/remove the configured-binding provider claim or wire WhatsApp inbound through the configured/runtime binding route helpers with regression coverage.

## Reproduction Assessment

Yes for the review finding. Source inspection shows the PR can make WhatsApp configured bindings compilable, while current WhatsApp inbound routing still goes through `resolveAgentRoute` without the configured/runtime binding route helpers.

## Solution Assessment

No. The diagnostic is narrow, but preserving the configured-binding provider and changelog parity claim is not the best fix unless inbound routing also enforces those bindings or that scope is split for maintainer approval.
