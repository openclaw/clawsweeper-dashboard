## Summary

Current `main` already implements Mattermost auto-threading for channel/group replies via `channels.mattermost.replyToMode`, and the same thread root is reused for preview streaming, block chunks, and follow-up payload delivery.

## Evidence

- **Config surface exists:** Mattermost config now exposes `replyToMode` (`off|first|all|batched`) as the supported knob for threaded replies.
  - file: [extensions/mattermost/src/config-schema-core.ts:102](https://github.com/openclaw/openclaw/blob/7d9172d5e093435d37ed3e00e476a4eaf84a252b/extensions/mattermost/src/config-schema-core.ts#L102)
  - sha: [7d9172d5e093](https://github.com/openclaw/openclaw/commit/7d9172d5e093435d37ed3e00e476a4eaf84a252b)
- **Per-account override is wired:** Mattermost account config is merged from channel + account config, and `resolveMattermostReplyToMode` reads `account.config.replyToMode` for channel/group messages while keeping DMs non-threaded.
  - file: [extensions/mattermost/src/mattermost/accounts.ts:51](https://github.com/openclaw/openclaw/blob/7d9172d5e093435d37ed3e00e476a4eaf84a252b/extensions/mattermost/src/mattermost/accounts.ts#L51)
  - sha: [7d9172d5e093](https://github.com/openclaw/openclaw/commit/7d9172d5e093435d37ed3e00e476a4eaf84a252b)
- **Top-level posts become thread roots:** `resolveMattermostEffectiveReplyToId` returns the triggering post id for channel/group messages when `replyToMode` is `first`, `all`, or `batched`, and `resolveMattermostThreadSessionContext` forks the session onto that thread root.
  - file: [extensions/mattermost/src/mattermost/monitor.ts:342](https://github.com/openclaw/openclaw/blob/7d9172d5e093435d37ed3e00e476a4eaf84a252b/extensions/mattermost/src/mattermost/monitor.ts#L342)
  - sha: [7d9172d5e093](https://github.com/openclaw/openclaw/commit/7d9172d5e093435d37ed3e00e476a4eaf84a252b)
- **Streaming and final delivery reuse the same thread:** The draft preview is created with `rootId: effectiveReplyToId`, and final/block payload delivery resolves `replyToId` from that same thread root, so follow-up chunks stay under the same Mattermost thread.
  - file: [extensions/mattermost/src/mattermost/monitor.ts:1622](https://github.com/openclaw/openclaw/blob/7d9172d5e093435d37ed3e00e476a4eaf84a252b/extensions/mattermost/src/mattermost/monitor.ts#L1622)
  - sha: [7d9172d5e093](https://github.com/openclaw/openclaw/commit/7d9172d5e093435d37ed3e00e476a4eaf84a252b)
- **Docs and tests match the requested behavior:** Docs say `replyToMode: "all"` starts a thread under the triggering post and that follow-up chunks/media continue in that same thread; tests cover top-level channel/group threading and thread-scoped session keys.
  - file: [docs/channels/mattermost.md:140](https://github.com/openclaw/openclaw/blob/7d9172d5e093435d37ed3e00e476a4eaf84a252b/docs/channels/mattermost.md#L140)
  - sha: [7d9172d5e093](https://github.com/openclaw/openclaw/commit/7d9172d5e093435d37ed3e00e476a4eaf84a252b)
- **Not shipped in latest release:** Commit `7d9172d5e093435d37ed3e00e476a4eaf84a252b` is not an ancestor of release commit `00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5` (`v2026.4.22`), so this is on `main` but not in the latest tagged release.

## Risks / Open Questions

- This behavior is present on current `main` only; it has not shipped in `v2026.4.22`.
