## Summary

Keep open: the latest head fixes the main `.env` value-collision cases and has strong CLI proof, but a supported legacy web-search credential path still under-scrubs when moved to a file/exec SecretRef, so this needs one more narrow repair before merge.

## What This Changes

The PR changes `secrets apply` `.env` scrubbing from global migrated-value matching to per-target/provider keyed scrubbing and adds collision regression coverage under `src/secrets`.

## Best Possible Solution

Keep the keyed scrub design, but derive non-provider owned env keys from explicit target/compat ownership metadata that includes legacy global web search, with a regression that does not reintroduce global value matching.

## Reproduction Assessment

Yes by source inspection: the PR-head map never adds `BRAVE_API_KEY` for `tools.web.search.apiKey` file/exec migrations because the supported legacy target has no provider-like path segment. I did not run the repro locally in this read-only review.

## Solution Assessment

No. The per-key scrub map is the right boundary, but the current non-provider ownership inference is incomplete for a supported legacy web-search target.
