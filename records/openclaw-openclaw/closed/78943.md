## Summary

Close as implemented: current main now initializes built-in context engines before CLI compaction resolves the default legacy engine, and the merged canonical repair has focused regression coverage; the fix is current-main-only and not in v2026.5.7.

## What This Changes

The issue reports that the main CLI runtime resolves the default legacy context engine before registering built-in context engines, causing transcript persistence failures.

## Best Possible Solution

Keep the current-main initializer ordering and let it ship in the next release; users on v2026.5.7 need a post-fix build for this repair.

## Reproduction Assessment

Yes for the original bug: the pre-fix source path resolved the default legacy engine before the built-in initializer, and the resolver throws when that registry entry is absent. Current main no longer reproduces by source inspection because the lifecycle now initializes before resolving.

## Solution Assessment

Yes: the current implementation uses the existing idempotent initializer at the narrow CLI compaction boundary, matching adjacent runtime ordering without adding config or product policy.
