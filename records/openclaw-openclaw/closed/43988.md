## Summary

Current main already resolves short PATH-based commands to absolute executable paths before persisting an `allow-always` approval, and it also has a durable exact-command fallback when no stable binary path can be derived. The reported "Always Allow behaves like Allow once" path is implemented/fixed on main and shipped in the latest release.

## Evidence

- **Basename commands resolve through PATH:** `resolveExecutablePath()` searches PATH entries and returns the concrete executable path, and `buildExecutableResolution()` stores that as `resolvedPath`, so a basename like `jq`/`id` is canonicalized before approval persistence.
  - file: [src/infra/executable-path.ts:93](https://github.com/openclaw/openclaw/blob/50e36983bb2d827adc617282c090c0a0359c6eee/src/infra/executable-path.ts#L93)
  - sha: [50e36983bb2d](https://github.com/openclaw/openclaw/commit/50e36983bb2d827adc617282c090c0a0359c6eee)
- **PATH resolution is regression-tested:** The command-resolution test explicitly covers a PATH executable (`rg -n foo`) and expects `resolveCommandResolution()` to produce the absolute executable path from the PATH fixture.
  - file: [src/infra/exec-command-resolution.test.ts:101](https://github.com/openclaw/openclaw/blob/50e36983bb2d827adc617282c090c0a0359c6eee/src/infra/exec-command-resolution.test.ts#L101)
  - sha: [50e36983bb2d](https://github.com/openclaw/openclaw/commit/50e36983bb2d827adc617282c090c0a0359c6eee)
- **Allow Always persists resolved binary paths:** `collectAllowAlwaysPatterns()` takes the resolved execution target candidate path and adds that concrete path to the persisted allow-always pattern set; `persistAllowAlwaysPatterns()` then writes those patterns into the approvals allowlist.
  - file: [src/infra/exec-approvals-allowlist.ts:961](https://github.com/openclaw/openclaw/blob/50e36983bb2d827adc617282c090c0a0359c6eee/src/infra/exec-approvals-allowlist.ts#L961)
  - sha: [50e36983bb2d](https://github.com/openclaw/openclaw/commit/50e36983bb2d827adc617282c090c0a0359c6eee)
- **Gateway approval flow no longer degrades to allow-once:** On an `allow-always` decision, the gateway exec path persists resolved allowlist patterns, and if none can be derived it falls back to a durable exact-command approval hash instead of silently acting like `allow-once`.
  - file: [src/agents/bash-tools.exec-host-gateway.ts:324](https://github.com/openclaw/openclaw/blob/50e36983bb2d827adc617282c090c0a0359c6eee/src/agents/bash-tools.exec-host-gateway.ts#L324)

## Close Comment

Closing from a Codex review because current `main` already implements this behavior. Basename commands are resolved through PATH before persistence (`src/infra/executable-path.ts:93`, `src/infra/exec-command-resolution.test.ts:101`), `allow-always` stores the resolved binary path in the exec approvals allowlist (`src/infra/exec-approvals-allowlist.ts:961`, `src/infra/exec-approvals.ts:971`), and the gateway approval flow now falls back to a durable exact-command approval if no stable binary path can be derived instead of behaving like `allow-once` (`src/agents/bash-tools.exec-host-gateway.ts:324`, `src/infra/exec-approvals.ts:810`). The relevant code is also present in the released `v2026.4.22` tree, so this looks implemented on main and shipped.
