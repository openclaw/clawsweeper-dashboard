## Summary

Close as a duplicate. The same documented OTEL `captureContent` gap is already tracked in an older open issue with an active linked PR, and this issue’s useful privacy notes belong on that canonical thread rather than creating a second tracker.

## What This Changes

The issue requests that diagnostic model and tool completion events include opt-in content fields so the diagnostics OTEL plugin can export `captureContent` attributes.

## Best Possible Solution

Keep the remaining work on https://github.com/openclaw/openclaw/issues/77391 and https://github.com/openclaw/openclaw/pull/79691, with content attached to diagnostic events only behind explicit capture-content policy gates.

## Reproduction Assessment

Yes. Current main documents and accepts the `captureContent` settings, and diagnostics-otel reads the content fields, but the model and tool completion event types/emitters do not provide those fields; I did not run a live OTLP collector in this read-only review.

## Solution Assessment

No, a separate tracker is not the best way to solve it. The maintainable path is to continue review on the older canonical issue and active PR, and to reject unconditional raw-content broadcast in favor of explicit capture-content gating.
