## Summary

Keep this PR open. Current main does not yet implement the heartbeat directive-preservation or defaults-only multi-agent enablement change, and the latest PR head has addressed the earlier test drift. The remaining blocker is in the new task-block stripping heuristic: it still treats any top-level Markdown bullet after `tasks:` as task YAML, so a normal non-task directive can be removed even though the heartbeat docs promise that non-task `HEARTBEAT.md` content is preserved.

## What This Changes

The branch preserves `HEARTBEAT.md` context around structured `tasks:` blocks, applies `agents.defaults.heartbeat` to all agents unless per-agent heartbeat entries restrict scope, adds focused heartbeat tests, migrates three infra/security warning sites to subsystem logging, and adds a changelog entry.

## Best Possible Solution

Centralize or align task-block stripping with the heartbeat task parser so only recognized task entries are removed and all top-level non-task Markdown before or after `tasks:` is preserved, including bullet lists; keep the defaults-only multi-agent behavior covered by tests and leave the logger migration scoped to warning output.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve top-level bullets after heartbeat tasks:** `src/infra/heartbeat-runner.ts:718-724`
  - body: While `inTasksBlock` is true, the new stripper treats every `trimmed.startsWith("-")` line as task YAML. A `HEARTBEAT.md` that puts a normal Markdown bullet immediately after the `tasks:` block will have that directive removed from Additional context, even though the docs promise non-task content is preserved. End the block based on indentation or recognized task-entry shape instead of every top-level `-` line, and cover that case.
  - confidence: 0.86
