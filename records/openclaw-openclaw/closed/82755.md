## What This Changes

The issue reports Active Memory timing out on every turn, leaving recall counts at zero and preventing dreaming from promoting memory candidates.

## Best Possible Solution

Keep the fix path centered on the canonical lane-deadlock issue and its open PR, then use the existing provider-specific reliability trackers for any remaining slow-model or stream-cleanup symptoms after that lands.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: Active Memory awaits recall inside before_prompt_build, the embedded recall omits lane, missing embedded lanes resolve to main, and command lanes are serial by default.

## Solution Assessment

Yes. The narrow maintainable fix is to run Active Memory recall on a dedicated non-main lane; changing dreaming thresholds would only mask the downstream symptom while recall remains broken.
