## Summary

Close as implemented on current main. The packaged runtime-deps mirror now includes `sqlite-vec` for the default memory plugin, regression coverage locks that packaged plan, sqlite-vec load failures are surfaced in warning text, and the supported vector config path is documented under `agents.defaults.memorySearch.store.vector.*`; the related #74692/#74711 runtime-deps gap is no longer present on main. No release tag in this checkout contains the proof SHA yet, so this is main-only after `v2026.4.27`.

## What This Changes

Fix the memory-core sqlite-vec path so packaged installs can load the vector extension, populate `chunks_vec`, expose load errors, and use the supported vector-store configuration surface.

## Best Possible Solution

Keep the current main behavior: stage `sqlite-vec` through the packaged runtime-deps mirror, keep explicit load-error logging, and direct operators to configure vector storage through `agents.defaults.memorySearch.store.vector.*` or per-agent memory-search overrides rather than broadening the memory-core plugin config schema.

## Reproduction Assessment

Yes for the historical failure: the 2026.4.10 Docker report gives a concrete warning loop and manual sqlite-vec load check, and the related #74692/#74711 context identifies the packaged runtime-deps mirror gap. Against current main, that specific packaged dependency gap no longer reproduces because `sqlite-vec` is mirrored and covered by a regression test.

## Solution Assessment

Yes. Mirroring the already-pinned `sqlite-vec` dependency into packaged bundled-plugin runtime deps is the narrowest maintainable fix for the runtime resolution failure, and keeping vector knobs under the existing memory-search config contract avoids adding a second plugin-local config path.
