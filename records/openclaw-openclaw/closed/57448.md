## Summary

Current main implements the central Telegram DM ACP handoff problem. ACP session spawns can bind the current Telegram DM conversation, follow-up messages route directly to the bound ACP session, and `/acp close` removes the binding so the chat returns to normal routing. The related #57910 PR was closed unmerged, but the same user-visible capability is now present on main through the current-conversation binding path.

## Best Possible Solution

Close this as implemented on main. Keep the current current-conversation ACP binding behavior, and track any future request for a `bind: "here"` tool alias or different `/new` revert semantics as a narrower follow-up.

## Risks / Open Questions

- Current `sessions_spawn` uses `thread: true` for this ACP binding path rather than the proposed `bind: "here"` field; the slash-command surface uses `--bind here`.
- Current docs/tests make `/new` and `/reset` reset the bound ACP session in place; `/acp close` is the explicit unbind/return path. A different `/new` revert behavior would be a narrower follow-up product choice.
