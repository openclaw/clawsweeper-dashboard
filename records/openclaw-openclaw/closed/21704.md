## Summary

Current main already implements native OpenAI TTS voice-note handling for Telegram-class channels: voice-note targets use OpenAI `opus`, keep the provider-derived file extension, and Telegram sends compatible audio via `sendVoice` instead of forcing `.mp3`. This behavior is also present in released tag `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`).

## Close Comment

Codex review: closing as implemented on `main`.

The underlying Telegram voice-note path is already in place. On current `main`, the OpenAI speech provider does not hardcode `.mp3` for voice-note delivery: `extensions/openai/speech-provider.ts:72` selects `opus` for `voice-note` targets, and `extensions/openai/speech-provider.ts:86` maps that to `.opus`. The speech core then routes Telegram through the native voice-note path (`extensions/speech-core/src/tts.ts:588`, `extensions/speech-core/src/tts.ts:810`) and writes the temp output using the provider-selected extension (`extensions/speech-core/src/tts.ts:771`). Telegram delivery sends compatible audio with `sendVoice` (`extensions/telegram/src/bot/delivery.replies.ts:349`), and the shared media compatibility helper accepts `.ogg` / `.opus` and `audio/ogg` / `audio/opus` (`src/media/audio.ts:4`).

This behavior is also present in released tag `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`), and the docs already describe Telegram OpenAI TTS output as `opus` voice messages in `docs/tools/tts.md:428`. If you still see `.mp3` attachments on `v2026.4.22` or newer, please open a fresh bug with exact `messages.tts` config, channel/plugin details, provider `baseUrl`, and any logs showing the generated filename/content type.
