## Summary

Keep open. The patch is not safe to merge because it deletes `message`/`text` before send validation, while raw `card` is not currently treated as send content, so the reported raw-card path would fail before plugin dispatch; the external PR also lacks after-fix real behavior proof.

## What This Changes

The PR adds card-aware message action normalization that moves `message`/`text` into `_cardNotificationHint` when a send action includes `card`, plus focused normalization tests.

## Best Possible Solution

Treat card sends through the existing `presentation` contract, or explicitly update the send schema, validation, and plugin contract so raw `card` is first-class content before stripping visible text, then prove the real Feishu/Lark behavior.

## Reproduction Assessment

No live Feishu/Lark reproduction is available. Source inspection gives a high-confidence reproduction of the PR regression: after the PR deletes `message`/`text`, raw `card` is not counted as send content and the request fails before plugin dispatch.

## Solution Assessment

No. Deleting text in normalization is too early for the current send pipeline; the maintainable fix should either use `presentation` or make raw `card` a validated send-content contract before removing fallback text.
