## Summary

Keep open. The PR is a plausible gateway/API change, but it lacks after-fix real behavior proof, is conflicting with current main, and the numeric nudge budget implementation does not actually honor positive budgets beyond 1.

## What This Changes

The PR adds wake-nudge budget fields to `node.list`/`node.describe`, introduces `OPENCLAW_NODE_WAKE_NUDGE_BUDGET=0` unlimited behavior, extends wake tests, and carries `sessions_spawn` failure-role cleanup.

## Best Possible Solution

Rebase onto current main, define and document the wake-nudge budget contract, either implement real positive-budget accounting or restrict the knob to the intended unlimited/default modes, then prove the gateway/node behavior with focused tests and real output.

## Reproduction Assessment

Not applicable as a strict bug reproduction. Source inspection shows current main lacks the requested budget field and env handling, but the PR is adding a new API/config surface rather than repairing documented broken behavior.

## Solution Assessment

No. The gateway direction is plausible, but the current implementation needs budget semantics fixed or narrowed, branch conflicts resolved, and real behavior proof before it is the best maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

- **[P2] Honor positive wake nudge budgets:** `src/gateway/server-methods/nodes.ts:134`
  - body: This parses any positive `OPENCLAW_NODE_WAKE_NUDGE_BUDGET` value and exposes it as `limit`, but the only state kept is `lastNudgeAtMs`. After one successful nudge, `remaining` becomes `0` until the cooldown expires, so a budget like `2` is reported but never usable. Track a per-window count or make the knob explicitly boolean/unlimited-only.
