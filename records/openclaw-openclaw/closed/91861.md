## Summary

Current main still preserves the reported asymmetric MEDIA directive handling: block replies keep the directive as text, final replies extract media, and caption removal only occurs when the block pipeline matches a constructed text-only payload. The supplied v2026.6.5 logs show two acknowledged text sends, current tests do not exercise this exact directive-driven block/final split, and the broader design issue remains separate rather than superseding this focused bug.

## What This Changes

Ensure a final reply containing visible text plus a MEDIA directive delivers the text once while still delivering the attachment across block and final channel paths.

## Best Possible Solution

Add an exact directive-driven block/final regression and update the canonical same-route sent-payload evidence so the later media payload retains the attachment but drops only text already delivered for that logical reply.

## Reproduction Assessment

Yes at source level: current main contains the exact asymmetric parsing and conditional dedupe path, and the report includes repeated v2026.6.5 live logs showing two successfully acknowledged text sends. This read-only review did not run a live Bncr channel reproduction.

## Solution Assessment

Yes, the best fix is a narrow exactly-once delivery repair with a regression for `hello\n\nMEDIA:/tmp/a.txt`; the broader channel.message payload-preservation design remains separately tracked at https://github.com/openclaw/openclaw/issues/91867.
