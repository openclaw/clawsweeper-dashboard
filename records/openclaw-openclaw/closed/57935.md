## Summary

Current `main` already removes the Telegram startup `getUpdates` offset-confirmation preflight and adds regression coverage for the stalled post-restart polling path described in #57935, but that fix does not appear in the latest tagged release `v2026.4.23` yet.

## Evidence

- **Current main no longer does startup offset-confirmation polling:** `TelegramPollingSession.#runPollingCycle` now starts polling without the old persisted-offset `bot.api.getUpdates(...)` preflight, removing the extra startup `getUpdates` call that could interfere with post-restart inbound polling.
