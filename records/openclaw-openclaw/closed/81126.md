## Summary

Keep open: current main still drops `data.details` for numeric ACP/JSON-RPC `RequestError` values, and https://github.com/openclaw/openclaw/pull/81188 is an open closing PR for the same bug rather than a merged fix.

## What This Changes

Preserve ACP SDK numeric `RequestError.data.details` when `toAcpRuntimeError` normalizes ACP runtime failures so visible error text includes the real cause.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/81188 or an equivalent narrow ACP normalization fix with real behavior proof, then close this issue after the fix merges.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to pass an `Error` shaped like ACP SDK `RequestError` with numeric `code: -32603`, message `Internal error`, and `data.details` into `toAcpRuntimeError`, then render it with `formatAcpRuntimeErrorText`; current main keeps only the generic message.

## Solution Assessment

Yes. The narrowest maintainable fix is to preserve a redacted diagnostic from numeric RequestError `data.details` during ACP runtime error normalization while leaving ACP option semantics and known string-code behavior unchanged.
