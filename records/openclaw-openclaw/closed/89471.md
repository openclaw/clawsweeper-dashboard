## Summary

Current main and shipped release content already use line-based `openclaw logs --limit`; byte bounding is handled separately by `--max-bytes`, so the reported semantics mismatch is not present in the maintained code path.

## What This Changes

The issue asks for `openclaw logs --limit N` to return the last N log lines, or for byte-based truncation to be exposed and documented separately.

## Best Possible Solution

Keep the current split contract: `--limit` controls returned log lines and `--max-bytes` controls the bounded read window; ask for a fresh report only if a current install still returns byte-counted output.

## Reproduction Assessment

No against current main: source inspection shows `--limit` is applied after splitting the bounded log text into lines. The reporter may have hit an older or stale installed binary, but the current maintained source and latest release tag do not reproduce byte-counted `--limit` semantics.

## Solution Assessment

Yes: the current implementation is the clean contract because it keeps line count and byte safety as separate controls. No additional core change is needed unless a new report proves a distinct packaging or installed-binary path still behaves differently.
