## Summary

Keep open. Current `main` still performs a single normalized registry lookup in `resolveImageRuntime`, so a request like `provider=bailian` with `model=bailian/qwen3.5-plus` can still miss when the registry stores the bare model id. The PR's fallback logic is not present on `main`, and current tests only cover related config pairing behavior, not this runtime lookup path.

## Evidence

- **Current image runtime still does one lookup only:** `resolveImageRuntime` normalizes `provider` and `model`, calls `modelRegistry.find` once, and throws `Unknown model` if that single lookup misses. There is no same-provider fallback from `bailian/qwen3.5-plus` to `qwen3.5-plus` on current `main`.
