## Summary

Keep open: current main still has global/per-run timeout handling, but no public per-channel agent-turn timeout map or channel-aware timeout resolver. This is valid feature/config work, but it needs maintainer decisions on config shape, default values, and channel-id semantics before implementation.

## What This Changes

Request a per-channel agent-turn timeout configuration map for Telegram, Discord, webchat, and cron while preserving existing global and per-cron-job timeout behavior.

## Best Possible Solution

Add a backward-compatible typed channel timeout map to agent defaults and entries, pass normalized channel identity into the centralized resolver, keep cron `payload.timeoutSeconds` first, and document/test the fallback order.

## Reproduction Assessment

Yes for the repository-level capability gap: current main has no `channelTimeoutSeconds` key and no channel-aware timeout resolver. The live OAuth soft-throttle hang scenario was not reproduced.

## Solution Assessment

Unclear until maintainers decide the public config shape and defaults; the central resolver direction is the maintainable boundary, but the exact channel-id/default policy needs review.
