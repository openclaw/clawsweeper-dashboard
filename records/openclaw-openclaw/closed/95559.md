## Summary

Keep open: the linked stale Control UI service-worker problem is real and current main/latest release still lack automatic browser recovery, but this PR is not merge-ready because the service-worker broadcast uses the wrong `Client.postMessage` shape, can race activation/claiming, and lacks real stale-browser proof.

## What This Changes

The PR adds a service-worker activation broadcast in `ui/public/sw.js` and a `navigator.serviceWorker` message listener in `ui/src/main.ts` that reloads the Control UI when a new worker reports `sw-updated`.

## Best Possible Solution

Repair the activation broadcast with valid Service Worker messaging, sequence it inside activation lifetime work after claiming or deliberately include uncontrolled clients, add focused coverage, and provide redacted browser proof of one reload and successful reconnect.

## Reproduction Assessment

Yes at source level: stale Control UI protocol ranges are rejected by the Gateway, and current main plus v2026.6.9 lack an activation reload path. I did not run a live Zeabur or Chrome stale-service-worker upgrade reproduction in this read-only review.

## Solution Assessment

No for the current patch. The activation-message approach is plausible, but the service-worker API call, activation lifetime sequencing, and real stale-browser proof need to be fixed before this is the best mergeable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Use the service-worker Client.postMessage signature:** `ui/public/sw.js:48`
  - body: `Client.postMessage` in a service worker accepts the message plus an optional transfer list or options object, not a Window-style target origin. Passing `"*"` here can throw before stale tabs receive `sw-updated`, so use the one-argument call or a valid options/transfer argument.
  - confidence: 0.95
- **[P1] Wait for claimed clients before broadcasting:** `ui/public/sw.js:46-48`
  - body: The new `matchAll` and `postMessage` work is not part of `event.waitUntil` and can run before `clients.claim()` resolves; because uncontrolled clients are excluded by default, the stale windows this fix is meant to reload can be missed or the worker can be terminated before the broadcast completes.
  - confidence: 0.91
