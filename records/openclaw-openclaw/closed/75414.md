## Summary

Close as implemented on main: the merged async transcript I/O work removes the synchronous checkpoint copy from the compaction paths implicated by the report, and current tests cover the no-sync-copy behavior. The fix is on `main` but is not contained in the latest `v2026.4.29` tag yet.

## What This Changes

The issue asks OpenClaw to prevent large session transcript compaction from blocking the Gateway event loop long enough to disconnect QQBot WebSockets and miss cron heartbeats.

## Best Possible Solution

Keep the focused async checkpoint-copy fix on main and ship it in the next release; if new liveness logs show remaining long event-loop stalls after this commit, track those as a narrower follow-up with fresh stall signatures and affected hot paths.

## Reproduction Assessment

Yes. The issue provides a concrete large-transcript compaction signature, and current source/tests give a high-confidence verification path: the implicated compaction checkpoint copy now goes through async I/O and the old sync helper has no production compaction callers.

## Solution Assessment

Yes. The merged main change is the narrow maintainable fix for the reported stall source: move large checkpoint snapshot copy and metadata validation off synchronous filesystem operations without adding QQBot-specific timeout policy or broader cron behavior changes.
