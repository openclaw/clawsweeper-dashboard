## Summary

Current `main` no longer matches the reported reproduction paths. `openclaw gateway restart` is a service-manager command on Linux/systemd, and the `openclaw agent --local` / gateway-fallback paths go straight to embedded agent execution rather than spawning a gateway process.

## Evidence

- **`gateway restart` is explicitly the service command:** The CLI registers `openclaw gateway restart` as "Restart the Gateway service (launchd/systemd/schtasks)", not a foreground gateway wrapper.
  - file: [src/cli/daemon-cli/register-service-commands.ts:115](https://github.com/openclaw/openclaw/blob/ed6094b3014c56882da15a7942130b00ea410c6e/src/cli/daemon-cli/register-service-commands.ts#L115)
  - command: `nl -ba src/cli/daemon-cli/register-service-commands.ts | sed -n '70,125p'`
  - sha: [ed6094b3014c](https://github.com/openclaw/openclaw/commit/ed6094b3014c56882da15a7942130b00ea410c6e)
- **Loaded services restart via service manager:** `runServiceRestart()` only uses the unmanaged SIGUSR1 fallback when the service is not loaded; when loaded, it calls `params.service.restart(...)`. `runDaemonRestart()` wires Linux restart through that path.
  - file: [src/cli/daemon-cli/lifecycle-core.ts:398](https://github.com/openclaw/openclaw/blob/ed6094b3014c56882da15a7942130b00ea410c6e/src/cli/daemon-cli/lifecycle-core.ts#L398)
  - command: `nl -ba src/cli/daemon-cli/lifecycle-core.ts | sed -n '345,490p'`
  - sha: [ed6094b3014c](https://github.com/openclaw/openclaw/commit/ed6094b3014c56882da15a7942130b00ea410c6e)
- **Linux systemd restart is `systemctl restart`:** The systemd backend runs `systemctl ... restart <unit>` in `restartSystemdService()`, which means the CLI restart path delegates to systemd instead of spawning a detached gateway child.
  - file: [src/daemon/systemd.ts:633](https://github.com/openclaw/openclaw/blob/ed6094b3014c56882da15a7942130b00ea410c6e/src/daemon/systemd.ts#L633)
  - command: `nl -ba src/daemon/systemd.ts | sed -n '604,648p'`
  - sha: [ed6094b3014c](https://github.com/openclaw/openclaw/commit/ed6094b3014c56882da15a7942130b00ea410c6e)
- **`agent --local` and gateway fallback are embedded-only:** `agentCliCommand()` sends `--local` straight to `agentCommand(...)`, and the non-local fallback after gateway failure also returns `agentCommand(...)`; neither path starts a gateway process.
  - file: [src/commands/agent-via-gateway.ts:180](https://github.com/openclaw/openclaw/blob/ed6094b3014c56882da15a7942130b00ea410c6e/src/commands/agent-via-gateway.ts#L180)
  - command: `nl -ba src/commands/agent-via-gateway.ts | sed -n '175,205p'`
  - sha: [ed6094b3014c](https://github.com/openclaw/openclaw/commit/ed6094b3014c56882da15a7942130b00ea410c6e)
- **Foreground respawn only happens for unsupervised gateway runs:** `restartGatewayProcessWithFreshPid()` returns `supervised` when Linux systemd markers are present, and the test suite covers that behavior. The detached-child respawn path is only the unsupervised fallback.
  - file: [src/infra/process-respawn.ts:26](https://github.com/openclaw/openclaw/blob/ed6094b3014c56882da15a7942130b00ea410c6e/src/infra/process-respawn.ts#L26)

  - sha: [ed6094b3014c](https://github.com/openclaw/openclaw/commit/ed6094b3014c56882da15a7942130b00ea410c6e)
- **Docs now distinguish service restart from foreground gateway runs:** The FAQ explicitly says `openclaw gateway restart` restarts the background service, while `openclaw gateway` is the foreground terminal run.
  - file: [docs/help/faq.md:1720](https://github.com/openclaw/openclaw/blob/ed6094b3014c56882da15a7942130b00ea410c6e/docs/help/faq.md#L1720)
