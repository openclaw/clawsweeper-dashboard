## Summary

Current main implements the practical fix path proposed in the issue: mutable state can be isolated with `OPENCLAW_STATE_DIR` or profiles, and the default cron store is derived from that state directory before falling back to `OPENCLAW_CONFIG_PATH`'s parent. The multi-gateway docs now explicitly require unique state dirs, config paths, ports, and workspaces, with examples. This is shipped in `v2026.4.24`.

## Best Possible Solution

Keep the current state-dir/profile isolation contract. Operators running multiple gateways should use `--profile` or set unique `OPENCLAW_STATE_DIR`, `OPENCLAW_CONFIG_PATH`, port, and workspace values; `cron.store` remains available for explicit custom cron layout. A config-stem cron migration is not needed in core unless maintainers want to support shared-state multi-gateway operation as a new product mode.

Distinct state dirs therefore produce distinct `cron/jobs.json` paths even when config files live under the same parent.

## Risks / Open Questions

- The implementation does not namespace cron by `OPENCLAW_CONFIG_PATH` stem. If two manually launched gateways intentionally share the same `OPENCLAW_STATE_DIR` and do not set distinct `cron.store` paths, they will still share mutable state by design.
