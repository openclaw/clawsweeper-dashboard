## Summary

Keep open. Current main still matches the reported Discord `/gateway/bot` metadata timeout/log-flood behavior: the metadata lookup has a hardcoded 10s timeout, falls back with the same transient warning, and exposes no user-facing timeout knob or dispatcher policy change for this request. The prior ClawSweeper comment remains accurate for current main.

## Best Possible Solution

Keep this open for a Discord plugin-owned fix. A conservative implementation would add a bounded configurable metadata timeout, rate-limit or demote repeated fallback warnings, and add focused tests for the timeout/log policy. If maintainers confirm connection reuse is the root cause, the dispatcher policy should stay in the Discord metadata fetch path or a generic plugin SDK HTTP seam, without moving Discord-specific policy into core.
