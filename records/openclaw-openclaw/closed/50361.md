## Summary

Close as implemented on main. Current main already prevents the user-visible Feishu streaming-card duplicate this PR targets: multiple finals before idle coalesce into one streaming card, and any later card final after a successful streaming close is suppressed regardless of raw text formatting while media attachments still deliver.

## What This Changes

The PR adds a Feishu final-text fingerprint normalizer and uses it for duplicate-final suppression, with a regression test for newline, zero-width-character, and trailing-whitespace variants.

## Best Possible Solution

Keep the current state-based Feishu streaming-card close guard on main. If users later report formatting-equivalent duplicate plain-text finals outside streaming-card delivery, handle that as a separate narrow Feishu dispatcher issue instead of reviving this stale streaming-card PR.

## Reproduction Assessment

Yes. The existing Feishu dispatcher tests model the relevant streaming final, idle close, and late-final sequences; static inspection shows the current guard suppresses post-close card finals independent of raw formatting. I did not execute tests because this review was required to remain read-only.

## Solution Assessment

Yes. Current main’s state-based suppression is the better fit for the reported streaming-card behavior because it prevents any successful post-close final text from reopening a duplicate card while preserving media delivery and error recovery.
