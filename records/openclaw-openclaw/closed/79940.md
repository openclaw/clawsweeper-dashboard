## Summary

Keep open: the PR targets a real current-main metadata leak, but it should not merge as-is because the new enrichment scope is caller-selected through `agentId`, leaving cross-agent job names available to read-scoped callers who know another agent id.

## What This Changes

The PR switches all-scope `cron.runs` job-name enrichment from `cron.list()` to paginated `cron.listPage()` with optional `agentId`, adds regression/protocol coverage, refreshes Swift protocol models, and updates the changelog.

## Best Possible Solution

Keep all-scope run rows compatible while building name/search metadata only from an authorized caller/session-visible job set, with explicit admin behavior and regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows the current leak through unfiltered `context.cron.list()`, and the PR’s residual issue is reproducible from the diff because `agentId` is accepted from request params and used for name lookup.

## Solution Assessment

No. `listPage()` and pagination are useful primitives, but making the filter a public request field is not the best access-control fix; the scope should come from trusted caller/session context or an explicitly authorized admin path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Derive the cron run-name scope from the caller:** `src/gateway/server-methods/cron.ts:531`
  - body: The new `agentId` is trusted directly when building the all-scope `jobNameById` map. Because `cron.runs` is readable with `operator.read` and the schema accepts any non-empty string, a caller can request another known agent id and still get that agent's job names attached and searchable in run history. Derive this scope from authenticated/session context, or require an admin-checked override, instead of taking it from the request.
  - confidence: 0.88

## Security Review

Status: needs_attention

Concerns:
