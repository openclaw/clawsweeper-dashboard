## Summary

Keep open. Current main still drops raw/canonical session-key mismatches before comparing the active run id, while this PR applies a narrow active-run fallback with focused delta/final/negative coverage and no blocking correctness or security findings.

## What This Changes

The PR updates Control UI chat event handling to accept events for the active run id when the event sessionKey differs, adds regression tests, and adds a changelog entry.

## Best Possible Solution

Land this focused PR after normal maintainer approval and branch-protection checks so the linked bug closes through merge, while preserving strict session-key filtering for non-active-run events.

## Reproduction Assessment

Yes. A focused current-main reproduction is state.sessionKey "main" with state.chatRunId "run-1" and a chat payload carrying sessionKey "agent:main:main", runId "run-1", and state "delta" or "final"; the strict sessionKey guard returns null before delivery.

## Solution Assessment

Yes. Matching the event to the active client-generated run id is narrower than changing gateway broadcast semantics or broadly normalizing every UI session key, and the PR keeps mismatched non-active events filtered out.
