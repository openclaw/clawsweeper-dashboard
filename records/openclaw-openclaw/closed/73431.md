## Summary

Keep open. The report is recent and concrete, and current main does not give enough evidence to close it as fixed: Discord read/search still await Carbon REST calls without an action-level timeout, CLI message actions do not pass an abort signal, and Discord plugin actions default to local execution rather than the gateway RPC timeout path. The inbound hook path is partially present through shared dispatch, but the prior #33038 session-key fallback is still absent and there is no end-to-end Discord guild hook regression proof in the inspected code.

## Best Possible Solution

Keep this issue open as a current Discord regression. Review and either land or replace #61572 for the read/search hardening, adding a bounded action/REST timeout and structured errors for Discord read/search. Separately add an end-to-end Discord guild inbound regression test that registers `message_received` and internal `message:received` observers and verifies `processDiscordMessage` reaches the shared dispatch hook path; if the missing-session-key case reproduces, either restore the `CommandTargetSessionKey` fallback or prove Discord always supplies a valid `SessionKey` before dispatch.
