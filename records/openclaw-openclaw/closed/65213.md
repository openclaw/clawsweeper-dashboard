## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Resolve explain keys in the selected store context:** `src/commands/sessions.ts:119-123`
  - body: The explain path bypasses the selected session-store target and canonicalizes `opts.explain` without the parent `--agent` context. In a multi-agent setup, `openclaw sessions --agent work explain main` can explain the default agent's main session instead of the requested agent; build the inspected session from the normalized target/store context before resolving the key.
  - confidence: 0.9
- **[P2] Select the freshest matching session entry:** `src/commands/sessions.ts:126-127`
  - body: `target.storeKeys.find((key) => store[key])` returns the first candidate in canonical/input order, not the newest persisted match. When canonical and legacy or case-variant keys coexist, explain can report stale model/workspace/tool state; use the same freshest-match helper used by current session lookup paths.
  - confidence: 0.88
- **[P2] Keep capability and tool resolution on the inspected store:** `src/commands/sessions.ts:145-165`
  - body: After the command picks `entry` from the loaded store, it recomputes subagent capabilities and effective tools from `target.canonicalKey` plus config only. For `--store <custom>` or a legacy `matchedKey`, that can describe a different configured-store session than the one being explained; pass the inspected store/matched key through the capability/tool path or centralize this in one session-explain helper.
  - confidence: 0.86
- **[P2] Honor parent JSON for the explain subcommand:** `src/cli/program/register.status-health-sessions.ts:184`
  - body: The parent `sessions` command already accepts `--json`, and sibling subcommands merge parent and child options. This action only passes `Boolean(opts.json)`, so `openclaw sessions --json explain <key>` falls back to text output and breaks scripts that place shared flags before the subcommand.
  - confidence: 0.9
- **[P2] Report reason flags only for the winning model source:** `src/commands/sessions.ts:217-223`
  - body: The diagnostic flags and reason chain mark provider overrides and runtime model fields as active whenever they are present, but `resolveSessionModelRef` only uses an explicit override when a model override is present and prefers overrides over runtime fields. Sessions with provider-only overrides or both override and runtime metadata will get misleading reason output.
  - confidence: 0.82
