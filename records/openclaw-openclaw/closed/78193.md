## Summary

Current main and the v2026.5.4 release already route `--tz` into one-shot `--at` parsing and cover the behavior with regression tests, so the reported silent-ignore path is no longer present.

## What This Changes

The issue asks for `openclaw cron add --at ... --tz ...` to honor the IANA timezone instead of storing the offset-less one-shot time as UTC.

## Best Possible Solution

Keep the shipped timezone-aware one-shot cron implementation and ask for a fresh report only if a v2026.5.4-or-newer install still stores the wrong UTC instant.

## Reproduction Assessment

No. Against current main, source shows `cron add/edit` schedule resolution forwards `tz` into `parseAt`, and regression tests assert timezone-adjusted UTC output for offset-less `--at` values.

## Solution Assessment

Yes. The current implementation is the narrow maintainable fix: normalize schedule options once, pass `tz` to `parseAt` for one-shot schedules, and leave explicit-offset `--at` values unchanged.
