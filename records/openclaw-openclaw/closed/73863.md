## Summary

Current main already covers the core request: Gateway file logs are structured JSONL with context fields and trace correlation, logging levels and console format are configurable, logs rotate daily and by `logging.maxFileBytes` into bounded numbered archives without suppressing later diagnostics, and diagnostics/status/health/export/OTEL surfaces provide the requested troubleshooting tools. The exact suggested knobs for gzip compression and configurable archive-retention days are not present, but they are narrower follow-ups rather than a reason to keep this broad duplicate observability request open.

## Best Possible Solution

Close this broad request as implemented by the shipped logging and diagnostics work in v2026.4.26. Keep the current core implementation, and open a focused follow-up only if maintainers explicitly want configurable archive-retention days or compressed log archives.
