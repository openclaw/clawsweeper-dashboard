## Summary

Keep this PR open: it targets a real Feishu envelope-send bug, but the current patch still has two blocking payload-preservation regressions and lacks external real behavior proof.

## What This Changes

The PR adds a Feishu serializer for OpenClaw payload envelopes, routes Feishu send/thread-reply envelope actions through it, and adds mocked serialization/action tests.

## Best Possible Solution

Keep the Feishu-owned envelope fix, but preserve or reject multi-payload envelopes, delegate media-bearing payloads to the existing media flow, and land only with focused regression tests plus redacted live Feishu proof.

## Reproduction Assessment

Yes, at source level. Current main rejects payload-envelope-only Feishu sends before the create-message helper because the action handler has no payloads branch.

## Solution Assessment

No as proposed. The Feishu plugin action/send boundary is the right place to fix this, but the patch must stop truncating payload arrays and must preserve media payload semantics before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Process every payload entry before sending:** `extensions/feishu/src/send.ts:109-110`
  - body: The serializer only reads `payloads[0]`, so additional visible payload entries are silently ignored while the action can still report success. Please either send every payload entry or fail fast when more than one payload is present.
  - confidence: 0.91
- **[P1] Preserve media payloads in envelope send path:** `extensions/feishu/src/channel.ts:711-713`
  - body: The new `payloads` branch catches the envelope before the existing media path, but the serializer only maps text and interactive cards. Media-bearing payloads should delegate to the media flow or fail explicitly instead of dropping attachments.
  - confidence: 0.9
