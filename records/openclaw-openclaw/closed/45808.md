## Summary

Keep this PR open. Current main still lacks Discord account-level `agentId` routing, so the feature gap is real, but the proposed patch has blocking routing/status issues before it is safe to merge.

## What This Changes

The PR adds Discord account-level `agentId` config/schema/types, synthesizes Discord route bindings at routing time, broadens route cache invalidation, exposes account `agentId` in Discord account snapshots, and adds focused binding tests.

## Best Possible Solution

Keep the PR as an implementation candidate, but merge only after synthesized Discord account routes respect runtime-eligible accounts, explicit binding precedence remains intact, effective account routing is exposed through the primary status API, Discord/multi-agent docs are updated, and focused routing/outbound/status tests cover the new contract.

## Reproduction Assessment

Yes. Current main can be checked by reading `listBindings`, the Discord config type/schema, and the routing docs; the requested account-level route is absent. The PR bugs are also reproducible by inspection: `synthesizeDiscordAccountBindings` pushes bindings for any account with `agentId`, and outbound auto-selection consumes `buildChannelAccountBindings`.

## Solution Assessment

No. The direction is plausible, but the current patch is not the safest merge shape until it filters disabled/unconfigured Discord accounts for synthesized bindings and propagates the effective binding through the primary status surface.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Skip disabled Discord accounts when synthesizing routes:** `src/config/bindings.ts:109-112`
  - body: `synthesizeDiscordAccountBindings` creates a route for every account entry with an `agentId`, even if that account is disabled or has no usable token. Those synthesized bindings feed `buildChannelAccountBindings`, which outbound message actions use to auto-select an account for an agent, so a disabled or unconfigured Discord account can be picked for sends and fail later or bypass the operator's disabled account intent. Filter synthesized bindings to runtime-eligible Discord accounts, or keep these synthetic routes out of outbound account selection.
  - confidence: 0.86
- **[P2] Propagate agentId through the primary status builder:** `extensions/discord/src/shared.ts:79-81`
  - body: Adding `agentId` only to Discord `describeAccount` does not make it reach `channels.status`/Mission Control: the primary status builder constructs snapshots from safe inspected/runtime fields and never merges `plugin.config.describeAccount`. As a result, the main status API can still omit the account binding even though this PR claims Mission Control visibility.
  - confidence: 0.81
