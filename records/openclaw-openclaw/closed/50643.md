## Summary

Keep open. The PR addresses a real current-main macOS gateway TLS TOFU vulnerability, but the patch is incomplete against the documented explicit pin contract and current structured TLS failure path, and the contributor has not provided real macOS behavior proof.

## What This Changes

The PR adds OSLog-backed `SecTrustEvaluateWithError` checking before accepting and storing a first-use macOS gateway TLS pin in `GatewayTLSPinningSession`.

## Best Possible Solution

Complete the hardening so macOS direct `wss://` first use fails closed unless system trust succeeds or an explicit configured/confirmed fingerprint matches, records structured TLS failures, and has focused Swift plus redacted runtime proof.

## Reproduction Assessment

Yes, by source inspection. A fresh macOS direct `wss://` endpoint with no stored Keychain pin still sets `allowTOFU: true`, then saves and accepts the presented certificate before `systemTrustOk` can reject it.

## Solution Assessment

No. The trust-evaluation guard is the right direction, but this PR should also wire the documented explicit fingerprint path into macOS direct mode and preserve current structured TLS failure reporting before merge.

- **[P1] Wire the explicit private TLS pin path:** `apps/shared/OpenClawKit/Sources/OpenClawKit/GatewayTLSPinning.swift:110-112`

Users with self-signed or private-CA gateways cannot reach the explicit-pin branch, so carry the configured or confirmed pin into the session before failing closed.
  - confidence: 0.9

A direct cancel in this new TOFU guard would surface as a generic WebSocket cancellation after rebase instead of `tlsCertificateUntrusted`, so record the failure or fall through to the existing structured failure path.
  - confidence: 0.82
