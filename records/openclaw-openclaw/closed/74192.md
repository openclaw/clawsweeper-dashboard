## Summary

Current main contains a targeted OpenAI Codex transport fix that addresses the reported post-2026.4.23 latency path: the native ChatGPT/Codex Responses backend no longer receives the large `max_output_tokens`/metadata/service-tier payload fields, and the PI OAuth-wrapped stream is preserved instead of being replaced. The fix landed after v2026.4.27, so the later v2026.4.27 benchmark comment is consistent with the issue still reproducing before the main-branch fix.

## What This Changes

The issue reports a regression where `openai-codex/gpt-5.5` local infer, gateway infer, and agent runs became much slower after 2026.4.23 compared with Hermes and the earlier OpenClaw release.

## Best Possible Solution

Keep the current main fix and ship it in the next release; handle any remaining non-OpenAI gateway latency under the narrower related reports rather than reopening this OpenAI Codex provider/transport item.

## Reproduction Assessment

Yes. The issue includes concrete timing commands for local infer, gateway infer, and `openclaw agent`; I did not run live provider timing in this read-only cleanup, but current-main source and regression tests cover the exact provider payload and stream-wrapper path implicated by those commands.

## Solution Assessment

Yes. The implemented fix is the narrowest maintainable path: it changes the shared OpenAI Codex transport boundary so local infer, gateway raw model runs, and agent runs all benefit, while preserving custom Codex-compatible endpoints and avoiding a global model context/token-cap change.
