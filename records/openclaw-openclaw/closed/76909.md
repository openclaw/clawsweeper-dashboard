## Summary

Keep open for maintainer review, but do not merge as-is: this draft changes the plugin trust model and still has concrete security/correctness blockers plus missing real behavior proof.

## What This Changes

This PR adds a `system` plugin origin from machine-wide directories, bypasses normal plugin activation controls, grants implicit prompt/conversation hook permissions, floors hook priority, updates origin ranks, and adds tests.

## Best Possible Solution

Define the admin trust model first, then implement a hardened core policy or system-plugin model with enforced OS ownership/ACL checks, reserved ordering, CLI/status visibility, docs, and focused regression coverage.

## Reproduction Assessment

Yes. Source inspection of the PR diff and current main is enough: the branch bypasses plugin controls, trusts a Windows ProgramData root without ACL validation, uses raw numeric priority, and does not reserve system origins in duplicate precedence.

## Solution Assessment

No. The proposed path makes the privileged tier unconditional before the trust boundary is defined; the safer solution is a maintainer-approved system-policy design with platform permission enforcement and origin-tier ordering.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Define admin trust before bypassing plugin controls:** `src/plugins/config-activation-shared.ts:141-151`
  - body: This always enables system-origin plugins ahead of the global toggle, denylist, allowlist, and per-entry disables, while the same branch grants implicit prompt/conversation access. Because native plugins run in-process, this needs an approved and enforced machine-wide ownership/ACL trust boundary before user controls are bypassed.
  - confidence: 0.93
- **[P1] Validate Windows system roots before trusting them:** `src/plugins/roots.ts:35-37`
  - body: The new ProgramData root is privileged, but current discovery permission validation returns without checking ownership or ACLs on Windows. Add platform-specific admin ACL validation or keep the system tier disabled on Windows until that exists.
  - confidence: 0.87
- **[P2] Reserve ordering by origin, not only numeric priority:** `src/plugins/registry.ts:2078-2083`
  - body: Clamping system hooks to a priority floor does not guarantee they run first because hook execution sorts only by raw priority and non-system hooks can register a larger number. Use an origin tier or cap non-system priorities for protected policy hooks.
  - confidence: 0.91
- **[P2] Give system origins top duplicate precedence:** `src/plugins/manifest-registry.ts:97`
  - body: The branch adds `system` to one origin rank while duplicate-id precedence still falls through to the default rank, so config/global/bundled/workspace duplicates can override the system record despite the PR's non-overridable claim. System needs explicit highest precedence in both same-root canonicalization and duplicate resolution.
  - confidence: 0.9
