## Summary

Current main removes the reported hot-path plugin metadata re-resolution by reusing process-local/current plugin metadata snapshots and dropping the per-lookup watched-file hashing that was present in the v2026.5.22 tag.

## What This Changes

The issue reports a 2026.5.22 gateway regression where every turn stalls for roughly 28-30 seconds while plugin metadata is re-resolved before the model call.

## Best Possible Solution

Keep the current-main snapshot reuse behavior and ship it through the normal stable release path; no further core issue is needed for the already-removed hot scan path.

## Reproduction Assessment

Yes for the released regression by source inspection: v2026.5.22 still re-hashed watched plugin files on memo lookup, matching the reported filesystem-stat storm. I did not run a live macOS fs_usage reproduction, and current main no longer has that hot path.

## Solution Assessment

Yes. Reusing lifecycle-owned metadata snapshots and requiring reload for plugin file changes matches the repository hot-path policy better than adding cache-duration knobs or continuing per-turn filesystem freshness checks.
