## Summary

Current main already fixes the reported user-visible WARN spam: idle and transient max-delay liveness samples are still recorded as telemetry, but they no longer go to the default Gateway warning log unless there is queued/waiting work or sustained active-work delay.

## What This Changes

Apply startup-grace-equivalent suppression so diagnostic liveness warnings do not appear during normal cold-start `models.list` and `sessions.list` latency.

## Best Possible Solution

Keep the current targeted diagnostic gating, which preserves queued/stalled-work warnings while removing the nonblocking cold-start WARN noise, and ship it in the next release.

## Reproduction Assessment

No for current main: the reported active=0/waiting=0/queued=0 transient max-delay sample is now routed away from `diagnosticLogger.warn` by source logic and regression tests.

## Solution Assessment

Yes. The current-main fix is narrower than blanket startup suppression because it keeps real queued, waiting, and sustained active-work warnings visible while quieting the reported cold-start false positive.
