## Summary

Keep open. Current main still runs plain security audits with plugin security collector loading enabled by default, while this PR narrowly changes that boundary and is active under the protected maintainer/security/automerge path rather than a cleanup-close candidate.

## What This Changes

The PR changes security audit so plugin-owned security collectors load only for `--deep` or explicit opt-in, adds a plain-audit regression test, updates CLI help/docs, and adds a changelog entry.

## Best Possible Solution

Land this narrow audit-boundary fix through the existing automerge lane after exact-head checks pass, and keep #75456 open for the remaining hidden-directory scanning and process-cleanup work.

## Reproduction Assessment

Yes. The current-main reproduction path is static: `openclaw security audit` passes `deep: false` without a collector-loading override, and the audit context default still enables plugin collector discovery that can import runtime registry/metadata loader paths.

## Solution Assessment

Yes. Defaulting collector loading to `opts.loadPluginSecurityCollectors ?? deep` is the narrow maintainable fix because it preserves deep audits and explicit internal callers while moving routine audits back to the cold config/filesystem path.
