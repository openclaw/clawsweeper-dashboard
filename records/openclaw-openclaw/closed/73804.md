## Summary

Close as implemented on current main: the Bonjour failure modes reported here are now handled as bounded, non-fatal plugin behavior, and local discovery can no longer hold Gateway startup indefinitely. The exact reverse-proxy auto-disable heuristic was not added, but the current implementation solves the observable double-boot/startup-delay and re-advertise-loop problem with narrower gateway/plugin safeguards.

## What This Changes

The issue asks OpenClaw to prevent the bundled Bonjour/mDNS plugin from causing managed gateway restarts or long startup delays on macOS and similar reverse-proxy/server deployments.

## Best Possible Solution

Keep the current bounded Bonjour plugin behavior: startup should continue quickly, known ciao failures should stay non-fatal, repeated advertiser churn should disable local discovery for the process, and operators who do not need LAN multicast should use the documented env or plugin disable path. A separate product discussion would be appropriate only if maintainers still want automatic reverse-proxy detection.

## Reproduction Assessment

Yes for the failure class, though not for the exact Mac Mini launchd environment. Current tests simulate ciao cancellation/assertion handling, stuck probing/announcing recovery, flapping advertiser disablement, and a never-settling local discovery service that previously could delay startup.

## Solution Assessment

Yes. The current fix is narrower than guessing from reverse-proxy or gateway.bind signals: it preserves Bonjour on normal LAN hosts while making failures bounded, non-fatal, and explicitly disableable for deployments where multicast is harmful.
