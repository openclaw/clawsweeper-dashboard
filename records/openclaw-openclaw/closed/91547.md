## Summary

Keep open. The Docker store-seed fix is aimed at the right runtime-assets failure, but the new selector helper does not implement npm's `any` platform selector semantics, which can under-seed a valid production package before offline prune.

## What This Changes

The PR updates `scripts/list-prod-store-packages.mjs` and its tests so Docker runtime-assets store seeding follows target os/cpu/libc selectors and includes snapshot optional dependency closure.

## Best Possible Solution

Keep the target-aware Docker store seed, add npm-compatible `any` selector handling with focused tests, then let maintainers decide whether the targeted runtime-assets proof is enough or a final image smoke is needed.

## Reproduction Assessment

Yes. Source inspection on current main and v2026.6.1 shows Docker runtime-assets seeds every lockfile package before target-scoped offline prune, while the lockfile contains non-target ACPX optional packages matching the linked failure logs; I did not run Docker in this read-only pass.

## Solution Assessment

No, not as-is. Target-aware filtering is the right layer, but the helper must mirror npm's `any` selector semantics before merge while preserving the new optional snapshot closure.
