## Summary

Current main still has a source-visible elevated-channel resolution gap, and the exact runtime fix PR was closed without merging. The issue should stay open as a narrow bug-fix candidate, ideally reconciled with the open sandbox-explain diagnostic PRs.

## What This Changes

The issue reports that elevated exec requests from Telegram resolve as an unknown or synthetic channel, causing `tools.elevated.allowFrom.telegram` to miss.

## Best Possible Solution

Land one focused fix that recovers the real channel for exec-event elevated checks and makes `sandbox explain` resolve both raw and agent-scoped Telegram session keys without broadening elevated defaults.

## Reproduction Assessment

Yes, source-reproducible: current main builds exec-event contexts with `OriginatingChannel` set to the delivery channel but evaluates elevated allowlists from `Provider`, and the diagnostic path can still leave the channel unresolved. I did not run the live Windows/Telegram flow in this read-only review.

## Solution Assessment

Yes for the repair direction: use the existing system-event/origin-channel pattern for runtime elevation and use bundled channel normalization for CLI diagnostics. The partial open PRs are not sufficient alone because they only address the sandbox-explain half.
