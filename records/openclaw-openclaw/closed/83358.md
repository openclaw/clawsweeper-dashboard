## Summary

Keep open. Current main already prevents stored device tokens and local fallback from silently replacing explicit `--url --token`, but source inspection still shows explicit-token CLI device-management calls carry CLI device identity and method-selected scopes into gateway pairing/scope-upgrade checks, so the reported recovery/auth isolation problem remains actionable.

## What This Changes

The issue reports that explicit shared-token auth for `openclaw devices` commands can still be affected by paired-device state during recovery flows.

## Best Possible Solution

Define one canonical recovery contract for explicit shared-token `devices` commands: either suppress CLI device-identity pairing state for that path or intentionally request the needed admin scope, then update regression coverage and docs for approve, remove, rotate, and revoke.

## Reproduction Assessment

Yes from source inspection, but not from a live run: explicit token auth wins over stored device tokens, while the CLI still sends device identity and requested scopes, so gateway pairing/scope-upgrade checks can interrupt the management RPC before it runs. I did not execute the macOS loopback repro.

## Solution Assessment

Unclear: the issue should not close, but changing this path affects authz and pairing semantics. The safest direction is a maintainer-approved contract plus focused tests, not an opportunistic scope broadening.
