## Summary

Close as implemented on current main: npm plugin installs now use a managed npm-root manifest/lock and post-update recovery rechecks missing payloads before reporting success, so the reported sibling-pruning path no longer applies.

## What This Changes

The issue asks for plugin recovery during `openclaw update` to stop losing earlier npm plugins when multiple missing payloads are recovered.

## Best Possible Solution

Keep the managed npm-root install implementation on main and ship it in the next release; users on `v2026.5.2` can reinstall affected plugins until then.

## Reproduction Assessment

Yes: the report gives a concrete two-plugin recovery path, and source inspection traces it through `repairMissingPayloads` to `updateNpmInstalledPlugins` and `installPluginFromNpmSpec`. I did not run npm installs because this review was read-only, and current main has the manifest-backed fix.

## Solution Assessment

Yes: maintaining a real managed `package.json` and lockfile in the npm root is the narrowest maintainable fix because it preserves sibling dependencies while keeping the existing per-plugin update flow.
