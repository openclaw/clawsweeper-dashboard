## Summary

Close as `clawhub`: current main already exposes plugin and gateway seams for inbound-message observation, live session subscriptions, and cross-session delivery, and the repo vision explicitly prefers optional channel workflow automation like this to ship as plugins/ClawHub rather than as new core routing behavior.

## Evidence

- **Scope anchor prefers plugins/ClawHub for optional capability:** `VISION.md` says core should stay lean, optional capability should usually ship as plugins, and if something cannot be built as a plugin the preferred path is extending the plugin API rather than adding one-off core behavior. It also says plugin discovery/promotion belongs in ClawHub.

That is the exact surface a relay/notification plugin would need to detect replies from isolated DM sessions.

## Risks / Open Questions

- There is no already-linked relay plugin in the issue discussion, so this close depends on maintainers accepting that existing plugin/gateway seams are sufficient even without a packaged example.
- OpenClaw does not ship a built-in per-channel relay policy today; operators who want this behavior still need a plugin, hook pack, or external bridge implementation.
