## Summary

Current main already implements the requested capability through the existing provider-plugin transport hooks. `ProviderPlugin.createStreamFn` is the public seam for a fully custom `StreamFn`, the runtime dispatches it today, the docs already direct plugin authors to use it, and bundled providers on main already rely on it. This makes a separate `api.registerStreamProvider()` registry redundant for the stated use case.

## Evidence

## Risks / Open Questions

- This review treats the existing provider-scoped `createStreamFn` hook as sufficient for the PR's requested custom transport capability; if maintainers specifically want a second, non-provider registry API for DX reasons, that would be a new ergonomic enhancement rather than missing core functionality.
- `ProviderCreateStreamFnContext` is not identical to the PR's proposed `registerStreamProvider` context, but current main already pairs custom transports with provider-owned auth and transport hooks (`prepareRuntimeAuth`, `resolveTransportTurnState`, `resolveWebSocketSessionPolicy`), which covers the stated custom-stream-provider use case.
