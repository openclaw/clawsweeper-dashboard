## Summary

Current main already fixes the Discord 1005/1006 stale-resume loop through the pinned @buape/carbon 0.16.0 gateway behavior. Carbon now resets reconnect counters only after READY/RESUMED, bounds failed RESUME attempts, clears stale session state, and falls back to fresh IDENTIFY; OpenClaw pins that dependency, promotes the fresh-IDENTIFY debug event, and handles reconnect exhaustion in the Discord lifecycle. The PR identified a real failure mode, but its OpenClaw-side patch is obsolete against the current extensions/discord layout and shipped dependency behavior.

## Best Possible Solution

Close this PR as implemented on main and keep the current Carbon-backed Discord gateway behavior. If a 1005/1006 stale-resume loop still reproduces on @buape/carbon 0.16.0 or later, track it as a fresh issue with current Discord plugin logs showing the Carbon reconnect/fresh-IDENTIFY sequence and any remaining lifecycle symptoms.
