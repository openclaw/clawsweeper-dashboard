## Summary

Current main has moved the gateway startup internal-hook dispatch out of the awaited startup path, so boot-md can still run BOOT.md work but no longer holds Gateway sidecar readiness while the agent turn completes.

## What This Changes

The issue reports that the bundled boot-md `gateway:startup` hook made Gateway restart availability wait on slow BOOT.md agent work.

## Best Possible Solution

Keep the current post-readiness fire-and-forget startup hook dispatch, with any future follow-up limited to CPU-bound BOOT.md starvation if that is separately reproduced after readiness.

## Reproduction Assessment

No current-main failing repro: source inspection shows `gateway:startup` is now scheduled after 250ms and `triggerInternalHook` is not awaited. The old reported path remains understandable from source as boot-md awaiting `runStartupTasks -> runBootOnce -> agentCommand`.

## Solution Assessment

Yes. Moving internal hook dispatch out of the awaited startup sidecar path is the narrow maintainable fix for restart availability while preserving boot-md behavior; a global hook timeout would be broader and risk unrelated hooks.
