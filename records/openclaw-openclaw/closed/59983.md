## Summary

Current `main` already hardens the timeout/abort teardown path so session locks are released even when cleanup steps throw, and it also adds stale session-lock reclamation plus regression coverage for the leaked-lock scenario described in #59983.

## Evidence

- **Timeout/abort cleanup now always releases the session lock:** `cleanupEmbeddedAttemptResources()` wraps each teardown step in best-effort `try/catch` blocks and still calls `await params.sessionLock.release()` in a `finally`, which directly addresses the reported leaked `.jsonl.lock` after abort/timeout cleanup failures.
