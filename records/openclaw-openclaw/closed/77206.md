## Summary

Keep open: current main still denies `internalDiagnostics` to global-origin diagnostics plugins, and open PR #77210 is already linked as the implementation path for this issue.

## What This Changes

The issue reports that externally installed `diagnostics-otel` loses OpenTelemetry export after 2026.5.2 because the service context capability gate only trusts bundled plugin origins.

## Best Possible Solution

Land a narrow fix, likely through #77210 after trust-boundary review, that lets official external diagnostics packages receive `internalDiagnostics` while still denying arbitrary workspace/config plugins.

## Reproduction Assessment

Yes. Source inspection on current main shows global-origin services cannot receive the capability, and `diagnostics-otel` returns after logging the reported error when the capability is absent; I did not run a live gateway in this read-only review.

## Solution Assessment

Unclear until maintainer trust-boundary review. #77210 is a narrow implementation candidate, but the final fix should avoid granting internal diagnostics to spoofed non-official packages that only match the plugin id.
