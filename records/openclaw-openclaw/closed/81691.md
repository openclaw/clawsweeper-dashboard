## Summary

Keep this open: current main and v2026.6.8 still contain the reported `runAtMs + 1` cron-slot probe, and https://github.com/openclaw/openclaw/pull/81731 is the open unmerged closing PR for the exact fix.

## What This Changes

The issue reports that cron stale-future repair can rebase valid exact-second cron slots because slot validation probes the previous run at `runAtMs + 1`.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/81731, or an equivalent narrow probe-past-the-candidate-second fix with focused regression coverage, then close this issue.

## Reproduction Assessment

Yes. Source inspection shows current main still probes `runAtMs + 1`, the maintenance repair path still depends on that classification, and Croner 10.0.1 operates at second precision by decrementing to an exact second.

## Solution Assessment

Yes. The best fix is the narrow helper change to probe past the candidate second plus regression coverage for the exact-slot case; the linked open PR follows that shape.
