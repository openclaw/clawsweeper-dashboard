## Summary

The reported log matches credential drift or an env/config precedence issue, not a source-level `channels.status` bug. Current main sends `channels.status` through the normal Gateway credential resolver, `token_mismatch` is only emitted when the provided token differs from the Gateway auth token, and the `scopes=0` field is expected after an unauthorized device-less attempt clears unbound scopes.

## What This Changes

The issue reports recurring `token_mismatch` failures from the `gateway:channels.status` WebSocket client when token auth is enabled.

## Best Possible Solution

Keep the existing Gateway auth behavior and resolve the local credential precedence or service environment drift; open a new issue only if redacted diagnostics prove the client and Gateway resolve the same token and still fail.

## Reproduction Assessment

No: the issue includes logs but not the effective client env, service env, and resolved auth surfaces needed to reproduce a repo bug. Current code provides a high-confidence explanation for the observed `token_mismatch` as a mismatched provided token.

## Solution Assessment

Not applicable as a source patch: the narrow maintainable path is local credential cleanup with `openclaw doctor` and env/service-manager checks, not a core auth change.
