## Summary

Current main and the latest release already cover the central concern: Codex app-server OAuth selection and API-key scrubbing are independent from the sandbox exec-server egress policy, so enabling supported OpenClaw sandbox egress does not require switching to API-key execution.

## What This Changes

The issue asks OpenClaw to document or verify that Codex app-server sandbox egress preserves ChatGPT/Codex OAuth auth profiles instead of falling back to API-key execution.

## Best Possible Solution

Keep the shipped Codex OAuth-preserving auth bridge and OpenClaw-owned sandbox egress path, and direct users to the existing Codex harness and sandbox config docs for verification signals.

## Reproduction Assessment

Yes. Source inspection and focused tests show the OAuth env-scrubbing path and the sandboxExecServer egress path, and the shipped docs give the runtime/status signals to verify the OAuth path instead of API-key fallback.

## Solution Assessment

Yes. The current design is the right owner-boundary solution because OpenClaw owns sandbox egress while Codex app-server auth remains selected through explicit OpenAI/Codex OAuth profiles, existing app-server accounts, or API-key fallback only when no profile/account is available.
