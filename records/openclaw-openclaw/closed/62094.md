## Summary

Current main already fixes the reported reset/new transcript reuse by rotating the persisted session transcript whenever the session ID changes for the same session key, and it includes a regression test for that case. The PR's proposed `forceFreshSessionFile` flag is therefore obsolete on main.

## Evidence

- **Current resolver rotates transcript on sessionId change:** `resolveAndPersistSessionFile()` now checks whether the persisted entry's `sessionId` matches the incoming `sessionId`; when it does not, it stops reusing the old `sessionFile` pointer and resolves/persists a new transcript path for the new session.
  - file: [src/config/sessions/session-file.ts:21](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/src/config/sessions/session-file.ts#L21)
  - command: `nl -ba src/config/sessions/session-file.ts | sed -n '1,80p'`
  - sha: [51dd4f288f67](https://github.com/openclaw/openclaw/commit/51dd4f288f67ef4df4c7a121f078f508535df1fd)
- **Reset/new path already generates a fresh sessionId before persistence:** `initSessionState` assigns a new UUID when the session is not being reused, then calls `resolveAndPersistSessionFile()` with that new `sessionId`. Inference: `/new` and `/reset` now get a fresh backing transcript through the generalized rotation logic rather than a special flag.
