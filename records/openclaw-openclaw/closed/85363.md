## What This Changes

Ensure iMessage restart catch-up cannot redispatch rows already handled by the live watcher when the persisted cursor lags.

## Best Possible Solution

Land or rework the linked fix PR so live-handled iMessage rows update a durable monotonic high-water mark, or use equivalent persistent message dedupe, while preserving catch-up retry holds.

Yes, from source inspection: put the persisted iMessage catch-up cursor behind a live-handled row, restart the gateway, and catch-up queries from the stale cursor after the in-memory dedupe has reset.

## Solution Assessment

Yes: updating a durable monotonic iMessage high-water mark from the live path, or adding equivalent persistent provider-message dedupe, is the narrow maintainable fix; the linked open PR is the current implementation candidate.
