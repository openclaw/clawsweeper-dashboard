## Summary

This PR should stay open for author or maintainer revision: current main does not already provide the task-events SDK seam or memory-core task outcome recorder, and the proposed patch still has lifecycle/config bugs plus unresolved retention/privacy review for a new persistent outcome log.

## What This Changes

The PR adds a public task registry event listener SDK subpath and wires memory-core to append terminal task outcome JSONL records for later dreaming feedback.

## Best Possible Solution

Merge a revised version that ties the listener to plugin lifecycle cleanup, reads current config in the callback, documents the new SDK/log contract, and settles retention for outcome data.

## Reproduction Assessment

Yes for the blocking patch bugs: source inspection gives a high-confidence path by registering or reloading memory-core twice, then emitting a running-to-terminal task transition. The old listener remains active, and later writes can use stale registration-time config.

## Solution Assessment

No, not as currently proposed. The additive SDK seam is plausible, but the implementation needs lifecycle cleanup, live config access, public docs, and a retention/privacy decision before it is the maintainable solution.

- **[P2] Register the recorder cleanup:** `extensions/memory-core/index.ts:34`
  - body: registerTaskOutcomeRecorder(api) returns an unsubscribe for the process-global task-registry listener, but this call discards it.   - body: This listener is long-lived, but it resolves the default agent and workspace from registration-time api.config. After a config reload changes the default agent or workspace, later task outcomes can be written to the wrong workspace or skipped; read api.runtime.config.current() inside the callback before resolving paths.
  - confidence: 0.86

The plugin SDK boundary needs docs/plugins coverage for the supported contract and lifecycle expectations, so third-party plugin authors know how to use the seam safely.
  - confidence: 0.82
