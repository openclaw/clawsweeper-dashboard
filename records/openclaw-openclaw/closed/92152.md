## Summary

Current main already fixes the reported v2026.6.5 memory index metadata failure: the live SQLite index is no longer removed during POSIX swaps, and post-swap reopen refuses to auto-create an empty no-meta database. The fix is main-only and not included in v2026.6.5.

## What This Changes

The issue asks to fix memory_search metadata handling so boot/session reindexing preserves index meta, the gateway accepts CLI-built metadata, and the suggested recovery action does not worsen the index state.

## Best Possible Solution

Keep the main implementation and let the next release carry the memory index swap fix; reopen or file a new issue only if the same symptom reproduces on a build containing 865fdab075.

## Reproduction Assessment

Yes, by source inspection: v2026.6.5 could temporarily remove the SQLite index during reindex and reopen the absent path as an empty database, matching the missing meta symptom. I did not run a live Docker repro, but the current tests cover the fixed swap and no-create behavior.

## Solution Assessment

Yes, the fix is at the right layer: it preserves the SQLite index across atomic swaps and prevents empty database creation instead of adding a runtime shim around missing metadata.
