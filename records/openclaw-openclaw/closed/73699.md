## Label Justifications

- `P3`: This is a useful Discord voice ergonomics feature, and the reporter explicitly framed it as nice-to-have rather than a blocker.
- `impact:session-state`: The original issue was about Discord voice using an isolated session without the text-channel session's memory and context.
- `impact:message-loss`: The requested bridge also covered voice transcripts and replies not appearing in the paired text channel.

## Summary

Close as implemented: current main and the shipped v2026.5.28 tree support Discord voice turns routing through a configured existing Discord text-channel session via `voice.agentSession`, which solves the central voice-as-I/O/session-isolation problem. Automatic visible transcript/reply mirroring is still a narrower follow-up, already represented by https://github.com/openclaw/openclaw/issues/53562.

## What This Changes

The issue requests a Discord voice bridge mode where voice transcripts and spoken replies share the existing text-channel agent session instead of using an isolated voice-channel session.

## Best Possible Solution

Keep the shipped `voice.agentSession` target path as the same-session voice-as-I/O solution, and use https://github.com/openclaw/openclaw/issues/53562 if maintainers still want automatic visible transcript/reply mirroring in Discord text channels.

## Reproduction Assessment

Not applicable as a feature request rather than a broken existing contract. Source, docs, and tests now confirm current main supports same-session Discord voice routing through `voice.agentSession`, and no live Discord voice run was attempted in this read-only review.

## Solution Assessment

Yes for the central session-isolation problem: `voice.agentSession` is a narrow config surface that reuses normal Discord routing instead of adding a parallel `bridgeToText` mode. It is not the full automatic text transcript/reply mirror requested in the body, so that remaining behavior is better handled by the narrower related thread.
