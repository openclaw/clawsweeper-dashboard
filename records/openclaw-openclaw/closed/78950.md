## Summary

Keep open: the PR addresses one HTTP/SSE policy path, but it changes Responses persistence defaults for every proxy-like endpoint, leaves the related opt-in policy and shared transport behavior unresolved, and lacks real LiteLLM behavior proof.

## What This Changes

The PR changes the Responses payload policy so `storeMode:"disable"` omits `store:false` for configured non-native Responses endpoints and adds a unit test for proxy versus native OpenAI behavior.

## Best Possible Solution

Add a documented provider/model Responses persistence policy that both HTTP/SSE and WebSocket payload builders honor, with LiteLLM opt-in coverage and native OpenAI/Azure no-regression coverage.

## Reproduction Assessment

No high-confidence live reproduction is available from this PR. Source inspection and the linked issue logs show the current OpenClaw side can send or preserve `store:false` for proxy-like Responses routes, but the PR does not include a real LiteLLM multi-turn run.

## Solution Assessment

No. Omitting `store:false` globally for proxy-like endpoints is not the best maintainable fix because it changes retention defaults and leaves the WebSocket builder outside the policy; an explicit shared persistence opt-in is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Make proxy response persistence opt-in:** `src/agents/openai-responses-payload-policy.ts:328-330`
  - body: This change omits `store:false` for every configured non-native Responses endpoint. For LiteLLM/OpenAI-compatible proxies that forward omitted fields to OpenAI, the documented default is `store:true`, so this changes retention semantics for all proxy routes despite current OpenClaw docs and the linked issue's explicit opt-in proposal. Please gate persistence behind a provider/model policy instead of changing every proxy by default.
  - confidence: 0.88
- **[P2] Share the store policy with the WebSocket builder:** `src/agents/openai-responses-payload-policy.ts:328-330`
  - body: The affected policy is not used by `buildOpenAIWebSocketResponseCreatePayload`; that builder still emits `store:false` whenever `supportsResponsesStoreField` is true, and current coverage asserts this for proxy routes. If proxy persistence is the intended fix, the same policy needs to be applied to the WebSocket payload path or the behavior remains transport-dependent.
  - confidence: 0.82
