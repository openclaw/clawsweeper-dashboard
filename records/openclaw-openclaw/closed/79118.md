## Summary

Keep open: current main still has the provider-webhook speech gap, and the issue is paired with an open closing PR that should be refreshed, reviewed, and landed rather than closing the issue early.

## What This Changes

Telnyx and other provider-webhook final speech events should trigger the same Voice Call auto-response loop as Twilio media-stream transcripts after the greeting.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/79336 or an equivalent Voice Call plugin patch that routes final provider-webhook speech into a guarded auto-response handoff.

## Reproduction Assessment

Yes. A source-level reproduction is clear: Telnyx `call.transcription` normalizes to `call.speech`, current main forwards it to the manager, and the manager records the transcript without invoking the response loop.

## Solution Assessment

Yes. The narrow maintainable fix is to add the response handoff at the Voice Call webhook/runtime layer with waiter, dedupe, and transcript-persistence guards, matching the open PR direction.
