## Summary

Keep open: current main still has a source-reproducible duplicate reasoning-field path for OpenRouter DeepSeek V4, and https://github.com/openclaw/openclaw/pull/79961 is already open with closing syntax for this issue.

## What This Changes

The issue reports that OpenRouter DeepSeek V4 Flash requests with thinking enabled include both top-level `reasoning_effort` and nested `reasoning.effort`, causing OpenRouter to return HTTP 400.

## Best Possible Solution

Review and land or close https://github.com/openclaw/openclaw/pull/79961; the desired end state is that OpenRouter DeepSeek V4 emits exactly one supported reasoning-effort representation with regression coverage for the composed wrapper path.

## Reproduction Assessment

Yes, at source level: OpenRouter thinking can create nested `reasoning.effort`, the OpenRouter provider composes that path before the DeepSeek V4 wrapper, and the DeepSeek V4 enabled branch adds top-level `reasoning_effort` without clearing nested `reasoning`. I did not establish a live OpenRouter 400 in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to make the final OpenRouter DeepSeek V4 payload choose one reasoning-effort shape, with focused coverage that asserts the competing field is absent.
