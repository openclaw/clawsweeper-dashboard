## Summary

Keep open. Current main still preserves provider identity from the related merged fix, but the approval-follow-up `agent` continuation has no path to carry elevated availability/default state into the embedded exec tool, so the reported second elevated command failure remains a valid security-sensitive bug.

## What This Changes

Elevated exec approval follow-up runs should keep elevated mode available for subsequent approved commands in the same assistant turn.

## Best Possible Solution

Add a narrow trusted approval-follow-up path that reconstructs or propagates only elevated availability and default level for the continuation, with regression coverage proving every `allow-once` command still requires a separate approval.

## Reproduction Assessment

Yes. I did not run a live Telegram/Docker repro in this read-only review, but the issue gives deterministic steps and current source shows the follow-up run drops elevated defaults before the embedded exec gate.

## Solution Assessment

No. The merged provider-identity fix was useful but incomplete; the narrow maintainable fix needs scoped elevated-default propagation or reconstruction for approval follow-up continuations.
