## Summary

Keep open for explicit security and product design. Current main has cryptographic device pairing, exact-command exec approvals, and plugin approvals, but no separate-device, fresh-presence gate covering high-risk chat-originated actions. The identical report at https://github.com/openclaw/openclaw/issues/91689 was closed by its opener, while the other linked issues address different trust boundaries.

Likely related people: @steipete for product/security defaults, @obviyus for the threat model, and @vincentkoc for gateway/auth review; confidence is medium because local feature-history attribution is limited.

## What This Changes

Add a gateway-level device-bound step-up authorization requirement for high-risk actions initiated through chat channels.

## Best Possible Solution

Approve a security design for typed cross-owner sensitive-action descriptors and a fresh-presence credential, then extend the existing gateway pending-approval and device systems with a staged critical-action gate instead of creating a parallel auth store or inferring risk from command text.

## Reproduction Assessment

Not applicable: current documentation expressly does not define exec approval as a per-user authorization boundary, so device-bound step-up is a new security capability rather than a reproducible defect in promised behavior.

## Solution Assessment

Yes in broad architecture, but not yet as a complete implementation plan: the central gateway boundary and reuse of pending approvals are appropriate, while existing device pairing must not be mistaken for fresh user presence and the taxonomy and defaults need security-owner approval.

## Risks / Open Questions

- A paired-device signature can prove key possession without proving fresh human presence unless the approval credential requires a defined local interaction such as biometric, PIN, or explicit device confirmation.
- A central risk taxonomy can miss plugin-owned, indirect, or newly added actions unless typed action descriptors are mandatory at the canonical execution boundary.
- Default enforcement could stop established chat-controlled workflows; enrollment, recovery, rollout, configuration, and migration behavior need an explicit compatibility plan.
- Allowing policy changes, device enrollment, or revocation through the initiating chat surface would undermine the intended trust separation.
