## Summary

Keep open: current main still suppresses plain-text group/channel final replies in the default tool-only mode when the message tool is available, and changing that fallback would alter the documented private-by-default channel policy.

## What This Changes

The issue asks OpenClaw to restore or diagnose visible Discord/Telegram replies for local LLMs that generate plain text instead of calling `message(action=send)` under the default group/channel reply mode.

## Best Possible Solution

Maintainers should choose the product policy for tool-available-but-not-called local-provider turns, then implement either a narrow compatibility path or stronger diagnostics without weakening group/channel privacy unintentionally.

## Reproduction Assessment

Yes. Source inspection shows an unset group/channel config resolves to `message_tool_only` when the message tool is available, and the dispatch test covers a plain final reply being generated but not delivered.

## Solution Assessment

Unclear. A plain-text fallback would restore legacy local-LLM behavior, but it conflicts with the documented private-by-default group/channel policy; explicit config, provider-aware diagnostics, or an opt-in heuristic need maintainer judgment.
