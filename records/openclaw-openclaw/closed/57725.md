## Summary

Issue #57725 describes repeated embedded OpenAI ws-stream upgrade failures with HTTP 500s on v2026.3.28. Current main already contains the relevant fixes, and the shipped changelog shows they went out in v2026.4.22: native OpenAI websocket session/header decisions now use shared endpoint classification, non-public/custom OpenAI base URLs stay on HTTP instead of drifting onto the hardcoded websocket path, and failed websocket sessions enter a cooldown instead of retry-spamming every turn.

## Evidence

- **Reporter corroboration:** The issue body reports repeated `[ws-stream] WebSocket connect failed ... falling back to HTTP` warnings on v2026.3.28, and @danilovmy confirmed the same behavior immediately after upgrading to v2026.3.28.
- **Shipped changelog entry: endpoint-classified websocket/session headers:** The v2026.4.23 release tree includes a 2026.4.22 changelog entry stating that native OpenAI websocket metadata and session-header decisions were routed through the shared endpoint classifier so local mocks and custom `models.providers.openai.baseUrl` endpoints stay out of the native OpenAI path.
