## Summary

Current main already supports the reported workflow through existing shared-skill loading and sandbox mount controls: shared/global skill roots are first-class, and operators can keep the workspace writable while exposing additional host paths as read-only bind mounts.

## Evidence

- **Shared/global skill roots are documented:** Skills config docs say built-in roots always include `~/.openclaw/skills`, `~/.agents/skills`, `<workspace>/.agents/skills`, and `<workspace>/skills`, and also support `skills.load.extraDirs` for extra shared roots.
  - file: [docs/tools/skills-config.md:85](https://github.com/openclaw/openclaw/blob/11cffb230013c2d8bd1e0e18da04b0b985b8abd5/docs/tools/skills-config.md#L85)
  - command: `nl -ba docs/tools/skills-config.md | sed -n '1,110p'`
  - sha: [11cffb230013](https://github.com/openclaw/openclaw/commit/11cffb230013c2d8bd1e0e18da04b0b985b8abd5)
- **Skill loader already reads those roots directly:** `loadWorkspaceSkillEntries` loads extra dirs, managed skills, personal `~/.agents/skills`, project `.agents/skills`, and workspace `skills/`, so shared/global skills do not require symlink tricks inside each workspace.
  - file: [src/agents/skills/workspace.ts:533](https://github.com/openclaw/openclaw/blob/11cffb230013c2d8bd1e0e18da04b0b985b8abd5/src/agents/skills/workspace.ts#L533)

  - file: [docs/gateway/sandboxing.md:274](https://github.com/openclaw/openclaw/blob/11cffb230013c2d8bd1e0e18da04b0b985b8abd5/docs/gateway/sandboxing.md#L274)

## Risks / Open Questions

- The exact proposed `tools.fs.workspaceOnlyRead` / `workspaceOnlyWrite` knobs do not exist; this close relies on the supported sandbox-plus-bind model and existing skill-root loading instead.
- If someone needs the same split specifically for unsandboxed host FS tools, that narrower design request would need a new issue with a concrete non-sandbox use case.
