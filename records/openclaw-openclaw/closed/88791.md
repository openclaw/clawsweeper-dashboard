## Summary

Keep open: current `main` still strips accumulated structured tool-call blocks when an OpenAI-compatible stream ends with `finish_reason: stop`, so the reported silent terminal turn remains actionable. The repair should be narrower than the issue’s unconditional guard because existing tests intentionally protect text replies from spurious tool-call execution.

## What This Changes

The issue reports that streamed structured tool-call blocks are dropped when an OpenAI-compatible provider reports `finish_reason: stop`, causing a terminal non-deliverable turn instead of tool execution.

## Best Possible Solution

Preserve the existing spurious text-reply protection, but promote `finish_reason: stop` to `toolUse` when finalized structured tool-call blocks are present and there is no non-empty visible assistant text; keep `length` and error paths from executing partial tool calls.

## Reproduction Assessment

Yes from source inspection: a mocked stream with structured `tool_calls`, no visible assistant text, and final `finish_reason: stop` follows the current filter path and loses the tool-call block. I did not execute the test in this read-only review.

## Solution Assessment

No for the issue’s exact unconditional patch: it is directionally right but too broad because existing tests protect visible text replies from spurious tool calls. The best fix is a conditional structured-tool promotion that leaves text replies, `length`, and error stops unchanged.
