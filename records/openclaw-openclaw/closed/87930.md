## Summary

Keep open: current main still has a source-backed Mattermost message-tool delivery gap where file-shaped send params can report success without guaranteed Mattermost upload/file_ids delivery.

## What This Changes

The issue asks for Mattermost `message(action="send")` attachment-shaped inputs to either upload through Mattermost files/posts or fail with an explicit unsupported attachment error.

## Best Possible Solution

Route Mattermost tool-send media through the existing Mattermost upload path with the computed media access context, and reject any still-unsupported attachment shapes with a clear error instead of returning ok.

## Solution Assessment

Yes: the narrow maintainable fix is to connect the existing send normalization and media access context to Mattermost's existing upload/post path, plus explicit errors for unsupported shapes. A broader new API or config option is not needed for the reported bug.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/87930. Mattermost `message(action="send")` currently normalizes `filePath`/`path`/`attachments[]` into send media, then the Mattermost plugin `handleAction` path handles `send` before the core adapter and only passes `params.media` to `sendMessageMattermost`, dropping `mediaLocalRoots`/`mediaReadFile`, `mediaUrls`, buffer/filename, and structured attachment context. Implement the narrow plugin/outbound fix so Mattermost send uploads the supported canonical attachment shape through the existing `/files` then `/posts` `file_ids` path, while preserving host/local media root checks and returning explicit errors for unsupported attachment shapes instead of ok-without-upload. Do not add a config option, do not broaden file-read permissions, and do not edit CHANGELOG.md. Add focused regression coverage for Mattermost message-tool send with local filePath/structured attachment context and any explicit unsupported-shape behavior.
