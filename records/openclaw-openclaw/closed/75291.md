## Summary

Current main addresses the observable startup/event-loop starvation path well enough to close. The report’s high-signal `core-plugin-tools`/runtime-deps reinstall symptom matches the already-fixed #75283 convergence bug: main now treats generated package-level runtime-deps supersets as materialized, so later lazy plugin activation should not relaunch package-manager installs after gateway pre-staging. Telegram fetches in current main are async and abortable, so the source evidence does not support a separate synchronous-fetch core fix from this report.

## What This Changes

The issue reports OpenClaw 2026.4.29 `openclaw tui` freezing at startup with gateway event-loop liveness warnings, Telegram Bot API timeouts, and repeated plugin/runtime dependency loading during `core-plugin-tools` preparation.

## Best Possible Solution

Keep the current runtime-deps convergence fix, ship it in the next release, and avoid a Telegram worker-thread rewrite unless a post-fix reproduction shows Telegram-specific async fetch handling still blocks the event loop.

## Reproduction Assessment

Yes. The issue gives a high-confidence release reproduction against v2026.4.29: start `openclaw tui` against a gateway with Telegram and multiple plugins, then observe liveness warnings plus repeated `core-plugin-tools`/runtime-deps preparation delays. I did not run the live host scenario in this read-only review, but the source and history map the reported symptom to the fixed runtime-deps convergence path.

## Solution Assessment

Yes. The current fix is the narrow maintainable solution for the observed gateway starvation: stop repeated package-manager runtime-deps installs after a package-level install root is already materialized. The Telegram path is already async and abortable in current main, so moving Telegram fetches to workers is not the best first fix without new evidence.
