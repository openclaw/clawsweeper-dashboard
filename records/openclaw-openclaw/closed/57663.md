## Summary

Current `main` already persists a successful fallback as the session's active model and surfaces fallback state/model details through lifecycle events, Control UI handling, verbose notices, and `/status`, which addresses the reported repeated full-chain timeout churn and missing active-model visibility.

## Evidence

- **Successful fallback is persisted into session selection state:** `applyFallbackCandidateSelectionToEntry` writes fallback provider/model/auth selection into the session entry, and `persistFallbackCandidateSelection` stores it before the retry continues so later turns reuse the working fallback instead of restarting from the failing primary.
  - file: [src/auto-reply/reply/agent-runner-execution.ts:215](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/src/auto-reply/reply/agent-runner-execution.ts#L215)
  - sha: [7e49cc87f9e9](https://github.com/openclaw/openclaw/commit/7e49cc87f9e9cdad47d0572331d7a7a47281a83e)
- **Persistence is explicitly guarded and written to the session store:** The runtime snapshots prior selection, applies the fallback selection, writes it to the active session store, and only rolls it back if that fallback candidate later fails.
  - file: [src/auto-reply/reply/agent-runner-execution.ts:720](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/src/auto-reply/reply/agent-runner-execution.ts#L720)
