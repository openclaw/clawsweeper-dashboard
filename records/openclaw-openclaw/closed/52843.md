## Summary

Current main no longer feeds Discord inbound images into model input at raw attachment size. Discord still persists the downloaded attachment buffer, but model-side image loading goes through the shared `loadWebMedia` path, which resizes/recompresses images and clamps them to the shared 6 MB image budget before base64 injection into a vision-capable run.

## Evidence

- **Discord inbound attachments are saved, then exposed as media paths:** Discord downloads each attachment and saves it with `saveMediaBuffer(...)`, then later builds the inbound media payload from those saved paths.
  - file: [extensions/discord/src/monitor/message-utils.ts:395](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/extensions/discord/src/monitor/message-utils.ts#L395)
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Discord turns include saved media paths in the agent payload:** `processDiscordMessage` turns the resolved attachment list into `MediaPath`/`MediaPaths` via `buildDiscordMediaPayload(mediaList)`.
  - file: [extensions/discord/src/monitor/message-handler.process.ts:430](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/extensions/discord/src/monitor/message-handler.process.ts#L430)
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Prompt image loading for saved inbound media uses the shared loader:** When the prompt references a saved inbound media entry, `loadImageFromRef` resolves it and calls `loadWebMedia(...)` before converting the image buffer to base64 for model input.
  - file: [src/agents/pi-embedded-runner/run/images.ts:373](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/agents/pi-embedded-runner/run/images.ts#L373)
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Shared loader recompresses/resizes images before model injection:** `loadWebMedia` routes image inputs through `optimizeAndClampImage(...)`; `optimizeImageToJpeg(...)` tries 2048/1536/1280/1024/800 px resize steps and quality sweeps until the image fits the cap.
  - file: [src/media/web-media.ts:446](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/media/web-media.ts#L446)
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Model-side image cap is 6 MB on current main:** The embedded runner passes `MAX_IMAGE_BYTES` into prompt-image loading, and that constant is `6 * 1024 * 1024`.
  - file: [src/media/constants.ts:1](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/media/constants.ts#L1)
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)

## Risks / Open Questions

- Channel adapters still store the original inbound file on disk; this review only establishes that the model-input path is normalized, not that channel-level storage is recompressed.
- Current main uses the shared 6 MB image-input cap rather than `channels.<name>.mediaMaxMb` for model-side normalization, so a separate config-unification request would still be distinct.
