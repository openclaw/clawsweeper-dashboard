## Summary

Keep open: current main still has a source-reproducible path to the reported Google Vertex model_not_found failure, and I did not find a merged fix or shipped release that restores API-key/profile-backed google-vertex model resolution.

## What This Changes

The issue reports a v2026.5.28 regression where Google Vertex Express API-key auth no longer resolves google-vertex model refs unless matching models.providers.google-vertex.models[] rows exist.

## Best Possible Solution

Restore built-in google-vertex API-key/profile-backed model resolution for supported Gemini refs, with regression coverage for configs that have no explicit models.providers.google-vertex rows and no changelog edit.

## Reproduction Assessment

Yes, source-reproducible: current main contains the exact error-builder path reported when agents.defaults.models has a google-vertex model but models.providers.google-vertex.models[] is absent. I did not run a live Vertex Express API-key call in this read-only review.

## Solution Assessment

Yes: the narrowest maintainable fix is to repair built-in Google Vertex provider catalog/model resolution for API-key or auth-profile setups, not to require users to add manual models.providers rows for a bundled provider.
