## Summary

Keep open: current main still routes custom `anthropic-messages` provider IDs through the base thinking profile unless a provider-specific hook exists, and Claude 4.6 profiles still omit `max` despite the current Anthropic effort contract. The report is source-reproducible, but the safest fix needs maintainer judgment on the provider-policy boundary rather than a generic core fallback.

## What This Changes

Request custom Anthropic Messages-compatible providers to receive Claude adaptive/max thinking profiles and align Claude 4.6 thinking levels with provider docs.

## Best Possible Solution

Add a provider-boundary-safe profile path for configured Anthropic-compatible custom providers, align Claude 4.6 `max` support with the upstream effort contract, and cover profile validation, transport behavior, session defaults, and docs with focused tests.

## Reproduction Assessment

Yes. Source inspection shows a custom provider id with no `resolveThinkingProfile` hook reaches the base thinking profile, so `adaptive` and `max` are unavailable even when the configured model uses the Anthropic Messages transport.

## Solution Assessment

No. The suggested direct Claude fallback in generic thinking resolution is not the best maintainable fix because it bypasses the provider-policy boundary; a safer solution should carry the right transport/model-family or declared capability fact into provider-owned thinking policy.
