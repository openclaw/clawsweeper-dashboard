## Summary

Close: current main addresses the reported gateway event-loop starvation by delaying provider auth pre-warm until after sidecars are ready and running the warm path through a worker-backed off-main-thread implementation. The fix is present on main only; I could not prove it is in a shipped release tag.

## What This Changes

The issue reports Windows gateway startup event-loop starvation while provider auth state pre-warm runs, delaying health/RPC/Telegram polling for minutes.

## Best Possible Solution

Keep the current main implementation that defers and offloads provider auth pre-warm, ship it in the next release, and leave broader CLI/config-reload follow-up work to the related open reports such as https://github.com/openclaw/openclaw/issues/86044 and https://github.com/openclaw/openclaw/issues/87285.

## Reproduction Assessment

Yes for the affected shipped path: the issue provides concrete timing logs, and v2026.5.27 source still shows gateway pre-warm calling the main-thread warmCurrentProviderAuthState implementation. I did not establish a failing current-main repro because current main now uses a deferred off-main-thread worker path.

## Solution Assessment

Yes: moving provider auth pre-warm into a delayed worker-backed sidecar is the narrow core fix for the event-loop starvation described here. A new config-disable flag is not required to solve this specific gateway responsiveness bug, though provider diagnostics can remain separate follow-up work.
