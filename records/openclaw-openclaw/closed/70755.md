## Summary

Keep this issue open. Current main still uses the BlueBubbles server/API password as the effective inbound webhook credential, the docs still instruct operators to put that password in the public webhook URL, and the linked open PR #70757 is the paired implementation candidate rather than a reason to close this before merge.

## What This Changes

The issue asks OpenClaw to stop reusing `channels.bluebubbles.password` as the public BlueBubbles webhook secret and add a separate inbound webhook credential.

## Best Possible Solution

Land #70757 or an equivalent maintainer-approved fix that uses a dedicated BlueBubbles webhook secret for inbound authentication while keeping `channels.bluebubbles.password` reserved for BlueBubbles API access.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure `channels.bluebubbles.password`, send `POST /bluebubbles-webhook?password=<that value>` with a caller-controlled `handle.address`, and current main authenticates before using that sender in allowlist, pairing, command, and agent-routing logic.

## Solution Assessment

Yes. A distinct inbound webhook credential is the narrow maintainable solution; moving the same BlueBubbles API password from a query parameter to a header would reduce URL exposure but would not separate the reusable server credential from the public webhook auth boundary.
