## Label Justifications

- `P1`: The reported shipped behavior can add a 50-60s tail to every Discord reply after auth-profile lock PID reuse, breaking a real channel workflow.
- `impact:auth-provider`: The stale lock is on `auth-profiles.json.lock` and blocks auth-profile success bookkeeping for provider profile state.

## Summary

Keep open. Current main and the latest shipped release still check PID liveness before `createdAt` expiry in the shared stale-lock helper, and auth-profile file locks use that helper, so the reported reused-PID stale-lock case remains valid. The related open PR mitigates reply blocking but does not fix lock reclamation itself.

## What This Changes

Requests stale lock reclamation to honor `createdAt` expiry before trusting PID liveness when reclaiming auth/plugin file locks.

## Best Possible Solution

Update the shared stale-lock helper so invalid or expired `createdAt` makes the lock reclaimable before PID liveness is trusted, while preserving non-expired live-PID behavior and adding regression coverage for helper and plugin file-lock use.

## Reproduction Assessment

Yes, there is a high-confidence source reproduction: call `shouldRemoveDeadOwnerOrExpiredLock` with a live `pid`, expired `createdAt`, and `isPidDefinitelyDead: () => false`; current source returns false before considering expiry. I did not run tests because this review is read-only.

## Solution Assessment

Yes, the suggested direction is the narrowest maintainable fix: make explicit lock expiry win before PID liveness in the shared helper. A stronger owner identity could be a later hardening step, but it is not required to make `staleMs` work as documented by the current lock payload.
