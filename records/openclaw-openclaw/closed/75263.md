## Summary

Keep open. Current main has several Telegram polling recovery protections, but I did not find a high-confidence fix for this fresh startup-specific report: the transport still has process-local sticky IPv4 fallback state, the monitor intentionally preserves that transport across clean/conflict polling restarts, and there is no regression test covering the reported sequence where startup logs the provider name and then never reaches an observable first polling attempt.

## What This Changes

Fix a Telegram long-poll startup regression where an immediate sticky IPv4 fallback after restart can leave inbound polling silent while outbound Telegram sends still work.

## Best Possible Solution

Keep this in the Telegram plugin owner boundary: add a focused transport/session fix that makes sticky fallback recoverable after successful connectivity or bounded time, emits an observable first polling lifecycle log, and covers the startup-after-fallback path with unit tests before considering the issue resolved.
