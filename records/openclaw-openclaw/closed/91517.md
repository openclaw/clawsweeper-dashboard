## Summary

Keep open: current main and the latest release still read the full persisted subagent run table from the production snapshot path, so the reported CPU hotspot remains actionable and source-supported.

## What This Changes

The issue reports that getSubagentRunsSnapshotForRead repeatedly performs full SQLite reads at roughly 22 Hz, causing high gateway CPU usage, and requests snapshot caching or write-driven invalidation.

## Best Possible Solution

Add a bounded process-local cache for persisted subagent registry snapshots with local write invalidation, short cross-process staleness bounds, and focused tests for read reuse and freshness.

## Reproduction Assessment

Yes from source inspection: current main calls loadSubagentRegistryFromSqlite() on each production getSubagentRunsSnapshotForRead call, and that loader synchronously reads all subagent_runs. I did not run a live perf reproduction in this read-only review.

## Solution Assessment

Yes: a small process-local snapshot cache with local write invalidation and a short staleness bound is the narrowest maintainable fix for the repeated-read driver. A pure event-invalidation cache would be too narrow because other processes can write the SQLite rows.
