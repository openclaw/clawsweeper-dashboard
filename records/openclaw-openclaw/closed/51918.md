## Summary

Current main already implements the central request through shipped active-run steering: default inbound queue mode is `steer`, active streaming runs inject accepted inbound messages at the next runtime boundary instead of waiting for a later followup turn, and docs/tests cover the behavior. The implementation is broader than the exact owner-only config names proposed in the issue, but it solves the owner stop/correction/addendum delay without aborting the active run.

## What This Changes

The issue requests owner or priority messages to preempt FIFO followup delivery during active agent turns and be injected at the next tool boundary without aborting pending work.

## Best Possible Solution

Keep the shipped `steer`-mode active-run steering as the supported answer for same-turn corrections and addenda. If maintainers later want an owner-only or explicit priority-flag policy, that should be a narrower follow-up request on top of the existing steering contract.

## Reproduction Assessment

Yes. The high-confidence current-main path is source-backed: an inbound message arriving while a session run is active resolves to default `steer`, then `runReplyAgent` queues it into the active streaming runtime instead of waiting for a later collected followup.

## Solution Assessment

Yes. Default active-run steering is the narrowest maintainable solution for the reported stop/correction/addendum delay because it preserves tool-call boundaries, avoids aborting the run, and keeps `collect`, `followup`, `steer-backlog`, and `interrupt` as explicit alternate modes.
