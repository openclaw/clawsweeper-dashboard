## Summary

Current `main` already covers both parts of this report: BlueBubbles inbound-gap recovery shipped via dedupe/catchup in `v2026.4.14`/`v2026.4.15`, and the stale-socket restart classification was changed in `v2026.4.22` so it now depends on provider-published transport activity rather than inbound app-message freshness. BlueBubbles does not currently publish `lastTransportActivityAt`, so the false `stale-socket` restart path described here no longer matches `main`.
