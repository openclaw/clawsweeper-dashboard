## What This Changes

The issue requests a unified ingress authorization gate so channels provide platform facts while core evaluates DM/group allowlists, command/event gates, route gates, activation, and admission consistently.

## Best Possible Solution

Keep the merged channel ingress runtime as the core authorization boundary and use `docs/refactor/ingress-core.md` for any follow-up wrapper deletion or repack work.

## Reproduction Assessment

Not applicable. This was an architecture RFC rather than a single failing runtime path, and source inspection on current main confirms the requested shared authorization boundary now exists.

## Solution Assessment

Yes. The shipped name is `resolveChannelMessageIngress` rather than `authorizeIngress`, but it implements the maintainable shape requested here: plugins provide platform facts, and core owns generic policy, event gates, command gates, route gates, redacted diagnostics, and admission.
