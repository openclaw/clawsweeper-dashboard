## Summary

Keep open. Current main still does not pass the inbound Signal group id into the Signal access callback, so `groupAllowFrom` entries are evaluated only against the sender identity; the shared mention resolver also still falls through to `requireMention: true` when no explicit group/default value is configured. Open PR #73538 is the linked implementation candidate and is unmerged, so this issue should stay open until that PR is reviewed or replaced and the mention-default policy is decided.

## What This Changes

The issue asks for Signal group messages to be accepted when `groupAllowFrom` contains the inbound Signal group id and for explicitly configured groups to avoid silently requiring mentions by default.

## Best Possible Solution

Keep the issue open until maintainers review and either land PR #73538 or replace it with an equivalent Signal-local group-id compatibility fix. Separately, make an explicit product decision for Signal mention defaults, then align focused regression tests and the public Signal/group docs with the chosen behavior.

## Reproduction Assessment

Yes. Static current-main reproduction is high-confidence: with `channels.signal.groupPolicy = "allowlist"` and `groupAllowFrom` containing a Signal group id, the handler extracts `groupId` but group access only checks the sender identity; for the mention half, a group config without `requireMention` reaches the final `return true`.

## Solution Assessment

Unclear for the full issue. A Signal-local group-id match is the narrowest maintainable fix if maintainers want `groupAllowFrom` to accept Signal group ids, but the `requireMention` fallback changes shared group-policy semantics and should be decided with tests and docs rather than folded in implicitly.
