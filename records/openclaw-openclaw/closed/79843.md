## Summary

Current `main` and the reported `v2026.5.7` source do not match the reported silent-empty behavior: `sessions.get` already accepts `key` or `sessionKey`, and `chat.history` rejects an unrecognized `key` parameter via schema validation before reading history.

## What This Changes

The issue requests consistent or aliased RPC parameter handling between `sessions.get` and `chat.history` for the same session key value.

## Best Possible Solution

Keep the current contract: `sessions.get` accepts both names, while `chat.history` fails closed on unrecognized params unless maintainers later choose an explicit compatibility alias.

## Reproduction Assessment

No. Current source and the reported release source show `sessions.get` accepting `sessionKey`, while `chat.history` rejects `{ "key": ... }` via schema validation instead of silently returning an empty payload.

## Solution Assessment

Yes. Closing is the best cleanup outcome because the observable silent-empty failure does not match current or reported-release source; accepting `key` on `chat.history` would be a separate optional API ergonomics change.
