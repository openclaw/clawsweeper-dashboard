## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Gate the marker on the selected Claude CLI runtime:** `extensions/anthropic/register.runtime.ts:695-696`
  - body: This branch runs for every `provider: "anthropic"` auth lookup whenever the host env flag is set, but direct Anthropic API-key/OAuth routes also use provider id `anthropic` and the synthetic-auth context does not include the selected model runtime. That can make non-Claude-CLI Anthropic runs look authenticated with the fake marker; the fix needs to prove `agentRuntime.id === "claude-cli"` or use a maintainer-approved config/API seam, with a direct-Anthropic regression test.
  - confidence: 0.9
- **[P1] Pass the Bedrock signal to the CLI before trusting it:** `extensions/anthropic/cli-auth-seam.ts:22`
  - body: The new helper trusts `CLAUDE_CODE_USE_BEDROCK` from `process.env`, but the bundled Claude CLI backend deliberately includes that variable in `CLAUDE_CLI_CLEAR_ENV`, and the runner deletes clearEnv keys from the spawned child env. In the reported host-env setup, auth resolution can pass while the Claude process no longer receives Bedrock mode, so this needs explicit re-injection/preservation for the same run or a marker source tied to backend env, plus a spawn-env regression test.
  - confidence: 0.88
