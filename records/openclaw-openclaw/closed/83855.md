- `impact:crash-loop`: A cached rejected config snapshot can make guarded CLI bootstrap calls fail repeatedly in the same process after a transient failure.

## Summary

Keep open: current main still caches the config snapshot promise without clearing it on rejection, and the focused tests do not cover the production cache path because Vitest bypasses the cache. The exact transient ENOENT example is not proven from source because config IO converts many read errors into invalid snapshots, but the sticky rejected-promise behavior is real if the snapshot read ever rejects.

## What This Changes

Retry CLI config snapshot reads after a rejected readConfigFileSnapshot promise instead of caching the rejection for the process lifetime.

## Best Possible Solution

Clear the cached snapshot promise only on rejection while preserving in-flight sharing for successful reads, then add a focused regression for retry-after-rejection.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows a mocked or actual rejected readConfigFileSnapshot promise would be returned again, but the specific filesystem ENOENT path is not proven because config IO catches many read failures.

## Solution Assessment

Yes. The narrowest maintainable fix is to keep the successful promise cache but reset configSnapshotPromise when that promise rejects, with a regression test that exercises the production cache branch.
