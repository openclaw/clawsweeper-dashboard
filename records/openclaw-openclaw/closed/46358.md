## Summary

Current main contains a targeted loopback-auth fix and regression coverage for the reported `missing scope: operator.read` failure on local gateway/control-ui paths, so this issue is implemented on main.

## Evidence

- **Shared-token baseline that caused the regression:** The default-token auth suite shows device-less shared-token connects that request `operator.read` are granted empty scopes, which matches the reported post-upgrade failure mode.
  - file: [src/gateway/server.auth.default-token.suite.ts:247](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/src/gateway/server.auth.default-token.suite.ts#L247)

  - sha: [56fe2aab9c87](https://github.com/openclaw/openclaw/commit/56fe2aab9c87596515f2280c8060dc740ea64a88)
- **Current main fixes authenticated loopback probes:** `probeGateway` now explicitly keeps authenticated loopback probes device-bound so local token/password probes do not get scope-limited by shared-auth scope stripping, and it passes that device identity into `GatewayClient`.
  - file: [src/gateway/probe.ts:155](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/src/gateway/probe.ts#L155)

- **Regression test covers the reported scope loss:** The probe test asserts that an authenticated loopback probe sends device identity, requests `operator.read`, performs the read RPCs, and reports `read_only` auth on success.
  - file: [src/gateway/probe.test.ts:130](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/src/gateway/probe.test.ts#L130)
