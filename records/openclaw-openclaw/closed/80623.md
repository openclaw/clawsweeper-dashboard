## Summary

Keep open: the PR changes the public plugin manifest contract from explicit tool-name ownership to a boolean/wildcard mode, while current main, docs, tests, and shipped release notes define `contracts.tools` as an explicit `string[]`; it also still lacks after-fix real behavior proof.

## What This Changes

The PR maps boolean `contracts.tools: true` to `['*']`, treats `'*'` as allowing any plugin tool name, and adds helper tests for wildcard tool-contract behavior.

## Best Possible Solution

Keep explicit string-array tool ownership and fix the linked report with clearer invalid-manifest diagnostics and plugin-author docs unless maintainers intentionally approve a documented wildcard mode.

## Reproduction Assessment

Yes. Source inspection shows boolean `contracts.tools` normalizes to no declared names on current main, and `registerTool` then emits the reported missing-contract diagnostic.

## Solution Assessment

No. The PR's boolean-to-wildcard mapping is a new public manifest mode; the narrower maintainable fix is clearer invalid-shape diagnostics and docs unless maintainers approve wildcard semantics.
