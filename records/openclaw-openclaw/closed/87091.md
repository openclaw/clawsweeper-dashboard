## Summary

Current main already fixes the reported fatal preflight no-op: successful contextEngine compaction results with `ok=true` and `compacted=false` now continue instead of aborting, with regression coverage. The latest release still had the old fatal branch, so this is current-main-only until the next release.

## What This Changes

The issue asks OpenClaw preflight compaction to treat a contextEngine `ok=true`, `compacted=false`, `below_threshold` result as a successful no-op instead of aborting Telegram/shared GUI replies.

## Best Possible Solution

Keep the current main behavior: successful no-op compaction results continue the reply, while deferred or real failure results remain fatal, and ship this in the next release.

## Reproduction Assessment

No current-main failure path remains. The old behavior is source-reproducible in v2026.5.22, but current main returns for `ok=true`, `compacted=false` no-op results instead of throwing.

## Solution Assessment

Yes. The current implementation uses the existing ContextEngine `ok`/`compacted` contract and keeps fatal handling for deferred or real failures without adding plugin-specific logic or a new config surface.
