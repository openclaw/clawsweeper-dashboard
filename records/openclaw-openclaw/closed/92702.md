## Summary

Keep open: this is a focused member-authored bug-fix PR, and current-main source inspection supports the reported orphaned runtime path; the patch appears to fix the manager-level sessionKey rebind case without introducing a blocking correctness or security finding.

## What This Changes

The PR disposes the previous registered, unleased session MCP runtime when a sessionKey is rebound to a new sessionId, and adds regression coverage for rebind disposal and leased-runtime preservation.

## Best Possible Solution

Land the manager-level rebind cleanup if maintainer review agrees, while leaving the adjacent session-reset timeout cleanup to its separate PR.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main can persist a new sessionId for the same sessionKey after compaction, then getOrCreate remaps that sessionKey while leaving the prior registered runtime alive until idle sweep. I did not execute the live WSL2/LadybugDB scenario in this read-only review.

## Solution Assessment

Yes, this is the best narrow fix for the manager-level bug: the session MCP runtime manager owns the sessionKey -> sessionId invariant, and disposing only the prior registered, unleased runtime is safer than adding per-server restart config or pushing stale-PID cleanup into MCP servers. The related gateway reset PR covers a different cleanup path rather than replacing this change.
