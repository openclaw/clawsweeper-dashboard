## Summary

Keep open: current main still has a plausible Windows Scheduled Task start-success false positive. The reported wrapper recursion is tracked separately at https://github.com/openclaw/openclaw/issues/86007, but this issue’s refused Scheduled Task launch path remains distinct and source-reproducible.

## What This Changes

The issue reports that Windows `openclaw gateway start --json` can return success even when Task Scheduler refuses the `OpenClaw Gateway` task and the gateway port never opens.

## Best Possible Solution

Make Windows Scheduled Task start verify a real launch/listener and surface refused `Last Run Result` failures like `0x800710E0`; install/force repair should clear or replace task settings that block launch, with focused tests and Windows proof.

## Reproduction Assessment

Yes, at source level: simulate `schtasks /Run` exit 0 followed by `Last Result: -2147020576` / `0x800710E0` and no listener. Current main classifies the runtime as stopped but still emits `ok: true` because the start path ignores `state.running`; I did not run a live Windows host.

## Solution Assessment

Yes for keeping it open; the narrow maintainable fix is in the Windows service start/repair path, not in the user’s Startup-folder workaround. The implementation should preserve existing Startup fallback behavior while making refused task launches fail loudly or repair safely.
