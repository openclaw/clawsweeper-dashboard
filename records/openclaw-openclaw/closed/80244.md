## Summary

The reported duplicate-send behavior belongs to the external openclaw-lark/native-card plugin path, not the OpenClaw core message contract; current core docs and code use `presentation` or plugin-owned `channelData` for rich cards, and the reporter’s follow-up says the plugin-level fix was applied and verified.

## What This Changes

The issue asks OpenClaw core to suppress `message`/`text` when a send action also includes a provider-native `card` parameter.

## Best Possible Solution

Leave raw provider-native card behavior owned by the channel plugin, and use OpenClaw’s existing `presentation` or plugin-owned `channelData` contract for any in-repo rich-card work.

## Reproduction Assessment

No high-confidence in-repo reproduction is available because the reported duplicate send is in the external openclaw-lark plugin path. Current source confirms raw `card` is not a shared send-content contract in OpenClaw core.

## Solution Assessment

Yes for closing this repository item: the plugin-owned suppression described by the reporter is the narrowest maintainable fix. A core-level raw-card normalization would require a new message-tool contract and maintainer product design.
