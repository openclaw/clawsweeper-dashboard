## Summary

Keep open: current main still lets temporary owner-only authorization changes alter the Codex app-server dynamic-tool fingerprint, so the reported native-thread churn remains source-reproducible. The prior ClawSweeper review and reporter follow-up are still aligned with the current code path, and the repair needs explicit security-conscious design review.

## What This Changes

The issue asks OpenClaw to preserve Codex app-server native thread continuity across temporary owner-only tool availability flips while still rejecting unauthorized owner-only tool execution.

## Best Possible Solution

Separate Codex thread schema compatibility from per-turn owner authorization so permission flips do not rotate the native thread and the bridge rejects unauthorized owner-only calls fail-closed.

## Reproduction Assessment

Yes, at source level: alternate the same Codex app-server session between owner and downgraded turns so `toolBridge.specs` changes by `cron`, `gateway`, and `nodes`; current main then treats the fingerprint as incompatible and starts `thread/start`. I did not run the live Feishu setup in this read-only review.

## Solution Assessment

Unclear for the exact implementation. The issue's direction is right, but the best repair needs maintainer/security choice between updating tools on resume/turn start or keeping a stable schema with per-turn bridge authorization.
