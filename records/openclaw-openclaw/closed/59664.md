## Summary

Current main and the published docs now document Slack native exec approval configuration, the approver fallback behavior, and the distinction between shared `approvals.exec` forwarding and `channels.slack.execApprovals.*` native-client routing. Release notes indicate the behavior/docs cleanup shipped in v2026.4.2.

## Evidence

- **Slack docs document the missing config path:** `docs/channels/slack.md` now has an `Exec approvals in Slack` section listing `channels.slack.execApprovals.enabled`, `channels.slack.execApprovals.approvers`, `channels.slack.execApprovals.target`, `agentFilter`, and `sessionFilter`; it also states that approvers fall back to `commands.ownerAllowFrom` where possible.
  - file: [docs/channels/slack.md:807](https://github.com/openclaw/openclaw/blob/afd6b5d6fce1dac98539d8b45bd0f9db9ff25386/docs/channels/slack.md#L807)
  - sha: [afd6b5d6fce1](https://github.com/openclaw/openclaw/commit/afd6b5d6fce1dac98539d8b45bd0f9db9ff25386)
- **Slack docs clarify explicit config is optional for default owner fallback:** The same Slack docs state that Slack auto-enables native exec approvals when `enabled` is unset or `"auto"` and at least one approver resolves, show `commands.ownerAllowFrom: ["slack:U12345678"]` as the no-explicit-config default, and show the explicit `channels.slack.execApprovals` block for override/origin-chat delivery cases.
  - file: [docs/channels/slack.md:820](https://github.com/openclaw/openclaw/blob/afd6b5d6fce1dac98539d8b45bd0f9db9ff25386/docs/channels/slack.md#L820)
  - sha: [afd6b5d6fce1](https://github.com/openclaw/openclaw/commit/afd6b5d6fce1dac98539d8b45bd0f9db9ff25386)
- **Exec approval docs explain the two config layers:** `docs/tools/exec-approvals.md` now distinguishes shared `approvals.exec` forwarding from native approval clients, states that `channels.<channel>.execApprovals` controls native clients, lists Slack under native clients, and documents that Slack approvers can be explicit or inferred from `commands.ownerAllowFrom`.
  - file: [docs/tools/exec-approvals.md:45](https://github.com/openclaw/openclaw/blob/afd6b5d6fce1dac98539d8b45bd0f9db9ff25386/docs/tools/exec-approvals.md#L45)
  - sha: [afd6b5d6fce1](https://github.com/openclaw/openclaw/commit/afd6b5d6fce1dac98539d8b45bd0f9db9ff25386)
- **Release notes show this shipped after the reported v2026.4.1 behavior:** The v2026.4.2 release notes include: `Exec approvals/channels: auto-enable DM-first native chat approvals when supported channels can infer approvers from existing owner config, while keeping channel fanout explicit and clarifying forwarding versus native approval client config.`
  - command: `GitHub releases page for v2026.4.2`
  - sha: [d74a122](https://github.com/openclaw/openclaw/commit/d74a122)
- **Issue discussion was incorporated into the review:** The issue timeline and comments linked several docs PR attempts, including qkal’s PR #59921 and earlier PRs #59675/#59698, which correctly identified the docs gap and the `commands.ownerAllowFrom` fallback. Current main has the clarified form rather than the earlier misleading wording that native Slack approvals always required both configs.
