## Summary

This should stay open because current main still has the reported false-positive class: `test_speech` verifies fresh bridge output bytes, not BlackHole loopback level, Meet sender audio, or participant audibility. The remaining work is a Google Meet plugin diagnostics/product decision, not conservative cleanup.

## What This Changes

The issue requests stronger Google Meet Chrome talk-back health semantics or diagnostics when BlackHole/SoX byte output succeeds but participants still hear silence.

## Best Possible Solution

Keep this in the Google Meet plugin by splitting TTS generation, output-command bytes, loopback or sender evidence, and audibility limits in status/docs, then evaluate WebAudio/MediaStream talk-back as a separate supported backend only after live validation.

## Reproduction Assessment

Yes for the source-level false-positive class: current main can set `speechOutputVerified` from fresh `lastOutputBytes` without loopback, sender-level, or participant-audibility proof. No for the exact live macOS meeting silence incident because this read-only review did not run Google Meet/BlackHole validation.

## Solution Assessment

Unclear. The narrowest maintainable first step is to stop overclaiming byte output as audibility through Google Meet plugin diagnostics and docs; a browser-level audio backend may be useful but needs product and live-validation review.
