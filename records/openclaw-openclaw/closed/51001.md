- **[P1] Keep protocol v4 when rebasing:** `src/gateway/protocol/version.ts:1`
  - body: This branch adds `src/gateway/protocol/version.ts` with only `PROTOCOL_VERSION = 3`, but current main is already on protocol v4 with min-client/probe constants and native guard coverage. Resolving the conflict by taking this file would downgrade or desynchronize the public Gateway protocol for existing clients.
  - confidence: 0.9
- **[P2] Reject half-configured retry policies:** `src/gateway/protocol/schema/cron.ts:290-291`
  - body: `retryCount` and `retryDelayMs` are accepted independently in the cron add/update schemas, but the timer only retries when both are positive. A client can therefore create or patch a job successfully with only one field and get no retry behavior at runtime, so the API needs all-or-nothing validation or defaults.
  - confidence: 0.91
- **[P2] Do not accept retry flags for one-shot jobs unless they work:** `src/cli/cron-cli/register.cron-add.ts:88-89`
  - body: The new `--retry-count`/`--retry-delay` flags are available on every `cron add`, including `--at` jobs, but the one-shot error path still uses only the existing global retry config and never reads the per-job fields. That makes the command appear to configure per-job retries while runtime behavior is unchanged.
  - confidence: 0.88
- **[P2] Recompute failed jobs when retry policy changes:** `src/cron/service/jobs.ts:491-495`
  - body: `applyJobPatch` persists new retry fields, but `update()` only recomputes `nextRunAtMs` for schedule or enabled changes. Editing a failed recurring job to add retries can leave the old natural next run in place, so the incident-response edit path is a no-op until some later scheduling event.
  - confidence: 0.84
