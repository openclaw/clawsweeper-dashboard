## Summary

Keep open: current main still has the reported lazy import shape for `web_fetch` and `sessions_send`, and the build guard/stable-entry list does not cover the plain hashed `runtime-*` and `manager-*` chunks named in the report.

## What This Changes

After an npm global install/update, `sessions_send` and `web_fetch` should not fail from stale hashed internal module filenames.

## Best Possible Solution

Stabilize or compatibility-alias the affected lazy package boundaries and add build/package regression coverage so long-running npm-global Gateways survive package replacement for these tools.

## Reproduction Assessment

Yes at source/package level, but not as a live Windows run. The reported stale chunk names line up with current lazy imports that are not stable dist entries and with the existing issue comment's package-artifact check across 2026.5.4 and 2026.5.6.

## Solution Assessment

No, current main is not yet the best solution because its stable alias and guard logic do not cover the reported plain hashed chunks. The narrow maintainable fix is to stabilize or alias those lazy boundaries and prevent similar non-`.runtime` lazy imports from escaping build checks.
