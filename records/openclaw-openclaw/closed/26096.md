## Summary

Current main already gives plugins the two building blocks this PR is trying to add: plugin command handlers receive `sessionKey`, and plugins already have read/write session-store access through `api.runtime.agent.session.*`. The PR's proposed `api.sessions` surface is redundant ergonomics rather than missing core capability.

## Risks / Open Questions

- Current main does not add the exact `api.sessions.getEntry()/updateEntry()/init()` alias proposed by this PR; maintainers could still choose that as a follow-up ergonomics change, but the underlying capability is already present.
- `SessionEntry` is not re-exported from the root `openclaw/plugin-sdk` barrel exactly as proposed here; plugin authors can still work against the existing runtime session-store surface.

## Close Comment

Closing after Codex review because current `main` already provides this capability through the existing plugin runtime seam. Plugin command handlers already receive `sessionKey` in `src/plugins/types.ts:1703` and `src/plugins/commands.ts:215`, with coverage in `src/plugins/commands.test.ts:582`. Plugins also already have session-store access via `api.runtime.agent.session.resolveStorePath/loadSessionStore/saveSessionStore/resolveSessionFilePath` in `src/plugins/runtime/types-core.ts:67`, documented in `docs/plugins/sdk-runtime.md:69`, and used by bundled plugin code in `extensions/active-memory/index.ts:326`. I also checked the `v2026.4.22` release tag (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`), and these surfaces were already shipped there. Since the requested functionality is already available on released OpenClaw, I’m closing this as implemented on main rather than landing a second API shape for the same capability.
