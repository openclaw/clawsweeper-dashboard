## Summary

Current main fixes the reported Control UI `process is not defined` path by keeping browser fallback slash-command metadata on a browser-safe registry path; the reported `v2026.4.29` image predates that fix, so the release can still reproduce it until a newer image ships.

## What This Changes

The issue asks for the Docker-served Control UI to stop blanking on startup with `Uncaught ReferenceError: process is not defined` in the released `2026.4.29` image.

## Best Possible Solution

Keep the current browser-safe registry split and let the next Docker/npm release carry the fixed Control UI bundle.

## Reproduction Assessment

Yes. The report gives a concrete `v2026.4.29` Docker setup and console stack, and the release-to-main diff shows that the released shared registry still imported server thinking runtime while current main removes that browser import.

## Solution Assessment

Yes. Moving provider-aware thinking choices into the server-only registry while leaving static fallback choices for the browser is the narrow maintainable fix, preferable to adding a browser `process` shim.
