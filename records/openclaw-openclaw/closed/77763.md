## Summary

Keep open: the underlying per-agent voice routing bug is real, but this PR still has a gateway trust-boundary blocker, drops the existing `voicecall.start` timeout, and its real behavior proof predates the current head.

## What This Changes

The PR threads per-call agent IDs through Google Meet and Voice Call call creation, stores `CallRecord.agentId`, adds a plugin-runtime gateway dispatch helper, tests, a proof script, and a changelog entry.

## Best Possible Solution

Keep the per-call `CallRecord.agentId` resolver design, but derive gateway owner identity from trusted runtime scope or a narrow internal allowlist, preserve bounded request timeouts, and refresh current-head real behavior proof.

## Reproduction Assessment

Yes, by source inspection. Current main has `toolContext.agentId`, but Google Meet and Voice Call drop it before call creation and downstream voice response code falls back to the configured/default agent.

## Solution Assessment

No for the current PR head. The per-call routing model is the right fix shape, but the gateway helper should not accept caller-supplied owner identity and the replacement dispatch path must preserve the configured start timeout.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Derive gateway owner identity from runtime scope:** `src/plugins/runtime/gateway-dispatch.runtime.ts:35-36`
  - body: `dispatchPluginGatewayRequest` is exported through the public plugin runtime SDK but forwards caller-supplied `pluginRuntimeOwnerId` into the synthetic gateway client. Since `voicecall.start` honors per-call `agentId` and `sessionKey` whenever that marker is present, another plugin can mint the owner marker; derive it from the trusted runtime scope or keep this handoff internal/allowlisted.
  - confidence: 0.88
- **[P2] Preserve the voicecall.start timeout:** `extensions/google-meet/src/voice-call-gateway.ts:126`
  - body: This call used to go through `client.request(..., { timeoutMs: requestTimeoutMs })`, but the new in-process dispatch only passes `pluginRuntimeOwnerId`. If `voicecall.start` or the provider start path stalls, the Meet join can now hang instead of failing on the configured timeout; thread a timeout through the helper or keep a bounded request wrapper.
  - confidence: 0.78
