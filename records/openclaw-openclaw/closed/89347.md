## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Backfill cron runtime state too:** `src/state/openclaw-state-db.ts:260-306`
  - body: When opening a legacy `cron_jobs` table that only has `job_json`, this backfill extracts config fields but never writes `state_json` or the split runtime columns such as `next_run_at_ms`, `last_run_at_ms`, and delivery/error state. `loadCronRows` reconstructs job state from those columns/`state_json`, not from `job_json`, so existing scheduled jobs can lose their persisted `nextRunAtMs` and run history on upgrade.
  - confidence: 0.96
- **[P2] Backfill cron descriptions:** `src/state/openclaw-state-db.ts:260-306`
  - body: The legacy `job_json` backfill copies `name` and other config fields but never sets the new `description` column. `rowToCronJob` only exposes `row.description`, so existing job descriptions stored in `job_json` disappear from list/detail/update flows after the DB is opened.
  - confidence: 0.91
- **[P2] Preserve numeric cron thread ids:** `src/state/openclaw-state-db.ts:377-381`
  - body: `CronDelivery` allows numeric `threadId` values and normal persistence stringifies them, but this backfill reads `delivery.threadId` with `textField`. Legacy jobs with numeric thread/topic IDs are migrated with `delivery_thread_id = NULL`, so delivery no longer targets the configured thread.
  - confidence: 0.91
- **[P2] Use a valid cron wake-mode default:** `src/state/openclaw-state-db.ts:524-526`
  - body: Adding `wake_mode` with `DEFAULT 'auto'` can leave malformed or partially unbackfilled legacy rows with a value outside `CronWakeMode`, which only accepts `now` or `next-heartbeat`. Default this column to a canonical value or normalize invalid values after the backfill so startup does not create non-canonical cron rows.
  - confidence: 0.87
