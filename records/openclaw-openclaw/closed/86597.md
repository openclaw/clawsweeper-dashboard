## Summary

Keep open. This is a concrete Control UI display bug: current source still resolves the chat model picker from a volatile per-session cache or the active `sessionsResult` row, and the adjacent related issues do not provide a merged/canonical fix for the reported A-to-B-to-A persisted-override display path.

## What This Changes

The issue reports that WebChat shows the configured default in the model picker after returning to a session even though the session's persisted model override is still active and used for messages.

## Best Possible Solution

Add a focused WebChat regression test for switching from session A to B and back after a per-session model override, then make the picker derive the visible model from authoritative session state or a durable per-session override cache without changing Gateway persistence or routing.
