## Summary

Current `main` already preserves Telegram forum-topic routing across restart-style delivery paths and no longer falls back to threadless forum sends, so the reported silent-loss behavior is already addressed.

## Evidence

- **Session delivery keeps Telegram topic/thread context:** `resolveSessionDeliveryTarget` parses stored delivery context/targets, carries `lastThreadId`, and returns the resolved `threadId` for later delivery instead of dropping it.
  - file: [src/infra/outbound/targets-session.ts:67](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/src/infra/outbound/targets-session.ts#L67)
  - sha: [34896839ba22](https://github.com/openclaw/openclaw/commit/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534)
- **Heartbeat delivery preserves group-route thread IDs for Telegram:** `resolveHeartbeatDeliveryTarget` merges explicit turn-source delivery context and, for plugins that opt in, reuses the stored group-route thread id instead of losing topic routing after restart/wake handling.
  - file: [src/infra/outbound/targets.ts:112](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/src/infra/outbound/targets.ts#L112)
  - sha: [34896839ba22](https://github.com/openclaw/openclaw/commit/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534)
- **Telegram plugin opts into preserved heartbeat topic routing and maps threadId to messageThreadId:** The Telegram channel advertises `preserveHeartbeatThreadIdForGroupRoute: true`, and outbound sends convert the resolved `threadId` into Telegram `messageThreadId` on the actual send call.
  - file: [extensions/telegram/src/channel.ts:693](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/extensions/telegram/src/channel.ts#L693)
  - sha: [34896839ba22](https://github.com/openclaw/openclaw/commit/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534)
- **Regression tests cover restart-style Telegram topic delivery:** Regression tests explicitly verify that an exec-event stays pinned to the original Telegram topic when the stored session route drifts, and that isolated scheduled heartbeats still send with `messageThreadId` after reading persisted session state.
  - file: [src/infra/heartbeat-runner.ghost-reminder.test.ts:536](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/src/infra/heartbeat-runner.ghost-reminder.test.ts#L536)

- **Forum sends fail instead of silently retrying without message_thread_id:** The Telegram send path only allows threadless retry for DM topics. Forum-topic sends throw on `message thread not found`, and the regression test asserts there is no retry without `message_thread_id`.
  - file: [extensions/telegram/src/bot/delivery.send.ts:48](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/extensions/telegram/src/bot/delivery.send.ts#L48)
