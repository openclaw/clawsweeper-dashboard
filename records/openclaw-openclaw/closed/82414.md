## Summary

Keep open. Current main still has a source-visible path where startup ACP identity reconciliation checks every pending identity with a stable runtime id, calls the runtime ensure/status path, and only increments `failed` on errors; the related older ACP startup report is closed but does not supersede this specific stale-session cleanup request.

## What This Changes

Prune or freshen ACP sessions whose backing sidecar is confirmed gone instead of retrying them and growing startup reconcile failures indefinitely.

## Best Possible Solution

Add a narrow ACP/ACPX stale-backing cleanup path that only clears or freshens persisted ACP state after the runtime can prove the backing process/session record is gone, with regression coverage for both dead and still-live persistent sessions.

## Reproduction Assessment

No, not as a live runtime repro in this read-only review. Source inspection does show the reported failure shape: startup reconciliation retries pending stable ACP identities and records failures without a prune/freshen path.

## Solution Assessment

Unclear whether the exact suggested PID check is the best fix because current core session identity has no PID field. The safer direction is an ACPX-owned or optional runtime-contract check that tells the manager when persisted backing state is definitely dead.
