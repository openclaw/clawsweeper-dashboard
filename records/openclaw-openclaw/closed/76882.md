## Summary

Close as cannot_reproduce: current main documents and implements `denyCommands` as exact command-ID filtering, and inspected `canvas.eval` paths do not forward JavaScript payloads to host shell execution or bypass `system.run` approvals.

## What This Changes

The issue asks for `gateway.nodes.denyCommands` to block shell-equivalent behavior embedded inside permitted command payloads such as `canvas.eval`, not only exact node command IDs.

## Best Possible Solution

Keep `denyCommands` as the documented command-ID filter; consider payload-level execution restrictions only as a separate policy design if maintainers want that capability.

## Reproduction Assessment

No for the claimed bug: source inspection on current main shows `canvas.eval` is a separate node command carrying JavaScript to canvas/WebView code, not a `system.run` command. The report does not include an exact payload that reaches host shell execution through this path.

## Solution Assessment

No; changing `denyCommands` to semantically scan arbitrary command payloads would expand a documented exact command-ID policy and likely needs a separate product/API decision. Current guidance already points users to remove risky command IDs and tighten exec/tool policy.
