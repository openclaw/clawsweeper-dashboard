## Label Justifications

- `P1`: The issue affects real macOS node-mode gateway sessions and can churn a connected node every few seconds without an explicit disconnect.
- `impact:session-state`: The reported behavior is repeated connection-state reset/reconnect churn in a node session that should remain stable.

## Summary

Keep open: current main and v2026.6.1 still have a source-level macOS direct `wss://` node-mode reconnect loop where each pass can create a new TLS-pinning session identity and force a fresh gateway channel despite an otherwise healthy connection.

## What This Changes

The issue asks OpenClaw to stop macOS node mode from repeatedly reconnecting and logging successful gateway connections during a healthy remote direct `wss://` session.

## Best Possible Solution

Reuse a stable TLS-pinning session box for unchanged macOS direct `wss://` endpoint and TLS parameters, reconnect only when the real connection inputs change, and add a regression proving repeated healthy loop passes produce one connection notification.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: unchanged `wss://` macOS node-mode loop passes create a fresh TLS session identity, and `GatewayNodeSession` treats that identity change as a reconnect. I did not run a live macOS app session in this read-only review.

## Solution Assessment

Yes, the best fix is narrow: stabilize or cache the macOS coordinator's TLS session box for unchanged connection inputs, while leaving `GatewayNodeSession` free to rebuild when a caller intentionally supplies a different session.
