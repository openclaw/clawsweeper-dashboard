## Summary

Close as duplicate/superseded: the same Control UI remote-image rendering policy is already tracked in #40945 with open PR #41025, and this issue has no unique reproduction data that needs separate tracking.

## What This Changes

Requests that chat messages containing image URLs such as .jpg, .png, or .gif render the image inline instead of showing a plain text link.

## Best Possible Solution

Keep the remaining remote-image work centralized in #40945/#41025, and only implement bare-URL previews there after maintainers approve the Control UI remote-media policy.

## Reproduction Assessment

Not applicable as a feature request rather than a bug. Source inspection confirms current main intentionally linkifies plain URLs and blocks remote image rendering through the markdown renderer, CSP, tests, and docs.

## Solution Assessment

Yes, closing this as duplicate/superseded is the best cleanup action. A separate implementation would bypass the canonical #40945/#41025 privacy and CSP decision that must cover both markdown image URLs and any bare-URL auto-preview variant.

## Security Review

Status: needs_attention

Summary: Automatic remote image rendering would make operator browsers fetch arbitrary third-party URLs from chat content.

Concerns:

- **[medium] Remote image previews leak viewer metadata:** `ui/src/ui/markdown.ts:417`
  - body: Auto-rendering chat-provided http(s) image URLs could expose IP address, user agent, and read timing to the image host; current renderer, CSP, and docs intentionally prevent that fetch class.
  - confidence: 0.9
