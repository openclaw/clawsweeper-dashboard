## Summary

Keep open. Current main gates off the reported reconcile GET when `native=false`, but explicit disable still awaits Discord command clearing by design and docs, so the requested zero-REST disabled mode needs maintainer product judgment.

## What This Changes

Request that `channels.discord.commands.native=false` skip the Discord slash-command lifecycle entirely at startup, including application-command GET and clear/delete REST calls.

## Best Possible Solution

Define disabled-native semantics explicitly, then preserve stale-command cleanup as an explicit or background operation while offering a documented no-REST startup policy if maintainers want that mode.

## Reproduction Assessment

Partial yes. Source inspection gives high confidence that explicit `native=false` still awaits a Discord application-command clear REST call, but the exact reported reconcile GET is not evident on current main because deploy/reconcile is gated off.

## Solution Assessment

Unclear. Skipping all REST calls would satisfy the low-power startup request, but it conflicts with the documented clear-on-disable contract and #63075; a separate deploy/clear policy like the direction in #75961 is likely safer but needs maintainer review.
