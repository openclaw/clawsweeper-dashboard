## Summary

Current `main` does not match the PR's reported undefined-`.replace()` paths: the cited text-normalization and error-formatting call chains already coerce or guard inputs, several touched helpers have since been centralized/refactored, and one proposed null-to-empty change conflicts with current test expectations.

## Evidence

- **`extractTextFromChatContent` normalizes from coerced strings:** The default `normalizeText` still calls `.replace()`, but both sanitize and normalize paths feed it through `coerceChatContentText(...)`, so nullish/non-string values become `""` before normalization.
  - file: [src/shared/chat-content.ts:34](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/shared/chat-content.ts#L34)
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **`formatAssistantErrorText` is only called behind `lastAssistant` guards:** Current failover/payload paths invoke `formatAssistantErrorText(...)` only when an assistant message exists, so the PR's claimed undefined `msg` path is not present in these mainline callers.
  - file: [src/agents/pi-embedded-runner/run/assistant-failover.ts:263](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/agents/pi-embedded-runner/run/assistant-failover.ts#L263)
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **Anthropic base URL normalization already has a truthy caller guard:** `normalizeAnthropicBaseUrl(baseUrl)` is only reached from `if (isAnthropicMessagesModel(model) && baseUrl)`, so `baseUrl.replace(...)` is not fed `undefined` on current `main`.
  - file: [src/plugins/provider-model-compat.ts:80](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/plugins/provider-model-compat.ts#L80)
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **Messaging dedupe already null-coerces before `.replace()`:** `normalizeTextForComparison()` starts from `normalizeLowercaseStringOrEmpty(text)`, which returns `""` for nullish input before the chained `.replace()` calls.
  - file: [src/agents/pi-embedded-helpers/messaging-dedupe.ts:12](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/agents/pi-embedded-helpers/messaging-dedupe.ts#L12)
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **Touched sanitizer paths were refactored into the shared visible-text pipeline:** `sanitizeTextContent()` now delegates to the shared assistant-visible-text sanitizer, and that pipeline exits early on falsy text before calling the strip helpers the PR tried to patch individually.
  - file: [src/shared/text/assistant-visible-text.ts:563](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/shared/text/assistant-visible-text.ts#L563)
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **One proposed behavior change contradicts current tests:** Current tests intentionally cast null into `stripReasoningTagsFromText(...)` and expect null passthrough, so the PR's blanket null-to-empty rewrite would not be a conservative cleanup on current `main`.
  - file: [src/shared/text/reasoning-tags.test.ts:5](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/shared/text/reasoning-tags.test.ts#L5)
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **Optional-chain claim on `meta?.trim().replace(...)` does not reproduce:** A direct Node check shows `meta?.trim().replace(/\s+/g, ' ')` evaluates to `undefined` when `meta` is absent rather than throwing, so that cited path is not an undefined-`.replace()` crash on its own.
  - command: `node -e "const meta=undefined; console.log(meta?.trim().replace(/\\s+/g,' '))"`
