## Summary

Current main already prevents the live gateway runtime PID from being emitted as a stale restart cleanup candidate, so this PR's central user-visible fix is redundant with the restart-health path that feeds daemon/gateway restart.

## What This Changes

The PR filters stale restart cleanup candidates in `runDaemonRestart` and adds mixed/self-only regression coverage for stale lists that include the live runtime PID.

## Best Possible Solution

Keep the restart-health ownership classifier as the canonical guard for stale PID cleanup; add lifecycle-level defense later only if maintainers want redundant hardening beyond the current production path.

## Reproduction Assessment

No. On current main, the production restart-health path excludes the running runtime PID before `runDaemonRestart` receives stale cleanup candidates, so the PR's self-in-stale-list condition is not reachable through the current waiter.

## Solution Assessment

Not applicable for merge. The proposed lifecycle filter is defensible, but current main already solves the user-visible stale self-kill problem at the restart-health ownership boundary, which is the narrower canonical source of the stale PID list.
