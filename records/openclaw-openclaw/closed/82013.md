## Summary

Keep open. Current main and the latest release still rotate stale idle/daily sessions and only run the automatic agentic memory flush from the pre-compaction/preflight path, so the requested significant-work session-end nudge, continuity prelude, or opt-in stitching behavior remains unresolved and needs product/security design.

## What This Changes

Request to add a session-end or next-session continuity mechanism that prompts agents to persist memory after significant work before idle session rotation severs context.

## Best Possible Solution

Define a guarded continuity mechanism that reuses memory-flush machinery and recorded tool-effect metadata, coordinates with the pre-reset flush and lifecycle-hook trackers, and leaves sticky sessionId stitching behind an explicit maintainer-approved contract if it is added at all.

## Reproduction Assessment

Not applicable as a strict bug reproduction; this is a feature request for new continuity behavior. Source inspection gives high confidence that current main still rotates stale sessions and only performs agentic memory flush from the pre-compaction/preflight path.

## Solution Assessment

Unclear pending maintainer decision. A session-end nudge is plausible and narrower than sticky sessionId reuse, but maintainers still need to choose trigger policy, cost limits, ordering, sandbox/tool behavior, and whether any opt-in config belongs in the public contract.
