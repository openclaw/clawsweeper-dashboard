## Summary

Keep open: current main still has the omitted-account precedence bug, but this PR is not ready because it can select the bound account after message-tool SecretRefs were scoped to the inherited account, and it also lacks the required changelog entry.

## What This Changes

The branch changes the message tool and outbound runner so omitted `accountId` resolves as explicit parameter, then calling-agent account binding, then inherited/default account, with runner regression cases.

## Best Possible Solution

Land a rebased version that uses one effective account-selection path for both SecretRef scoping and dispatch, preserves explicit overrides and fallback behavior, and includes focused regression coverage plus a changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `createMessageTool` with inherited `agentAccountId: "parent-account"`, a calling-agent binding to `account-b`, account-scoped SecretRefs for both accounts, and omitted `params.accountId`; the patched branch can scope secrets for the parent while dispatching as `account-b`.

## Solution Assessment

No, not as currently patched. The runner precedence change is the right direction, but the same selected account must be used before or during SecretRef resolution, and the PR needs the required changelog entry.
