## Summary

Keep open: current main and latest v2026.5.7 still leave Discord final status-reaction cleanup on default hold timers, so the narrowed `doneHoldMs`/`errorHoldMs` parity bug remains valid even though Discord lifecycle transitions are otherwise wired.

## What This Changes

The issue asks Discord to honor configured `messages.statusReactions.timing.doneHoldMs` and `errorHoldMs` for final status reaction cleanup.

## Best Possible Solution

Apply the configured merged status-reaction timing in the Discord finalizer, keep current restore behavior when `removeAckAfterReply` is false, and add focused Discord regression coverage.

## Reproduction Assessment

Yes, at source level: on current main, set `messages.removeAckAfterReply=true` and a custom `messages.statusReactions.timing.doneHoldMs`, then the Discord finalizer still sleeps with `DEFAULT_TIMING.doneHoldMs`. I did not run a live Discord gateway in this read-only review.

## Solution Assessment

Yes for the narrowed bug: the best fix is to mirror Telegram and Slack by computing merged timing once and using it for Discord done/error cleanup. Changing final persistence when `removeAckAfterReply` is false should stay out of this bug fix unless maintainers choose that policy separately.
