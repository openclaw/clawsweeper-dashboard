## Summary

Close as superseded: current main intentionally enforces `allowFrom` for direct WhatsApp outbound targets, this PR would reopen that security boundary, and the remaining outbound-only use case is now tracked by the opt-in `allowTo` design in #49549.

## What This Changes

The PR changes WhatsApp outbound target resolution and tests so `mode: "explicit"` direct E.164 targets outside `allowFrom` resolve successfully while implicit and heartbeat-style routes remain gated.

## Best Possible Solution

Keep the shipped fail-closed `allowFrom` behavior and handle outbound-only WhatsApp targeting through #49549 or an equivalent opt-in `allowTo` config with schema, docs, and tests.

## Reproduction Assessment

Yes. The source-level reproduction is `resolveWhatsAppOutboundTarget({ to: <valid E.164>, allowFrom: [<different E.164>], mode: "explicit" })`: current main returns a WhatsApp `allowFrom` policy error, while this PR changes that case to succeed.

## Solution Assessment

No. An unconditional explicit-mode bypass is not the best solution because normal outbound sends use explicit mode; an opt-in outbound allowlist such as `allowTo` is the narrower maintainable policy path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Do not bypass WhatsApp allowFrom for explicit sends:** `src/whatsapp/resolve-outbound-target.ts:34`
  - body: Adding `params.mode === "explicit"` here allows normal message-tool sends to reach valid WhatsApp numbers outside a restrictive `allowFrom` list. The main outbound send path passes explicit mode, so this reopens the explicit-send authorization bypass fixed by #20921 for #20108.
  - confidence: 0.9
