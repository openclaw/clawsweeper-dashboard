## Summary

Current main already bounds the reported sessions.json growth by pruning isolated cron run entries out of the session store. The shipped fix is retention-based rather than skillsSnapshot deduplication, but it directly addresses the unbounded-growth failure mode described in the issue.

- Current main still writes the same session entry under both the cron base key and run key in src/cron/isolated-agent/run-session-state.ts:33-40, so this is a retention-based fix, not structural skillsSnapshot deduplication.

## Close Comment

Closing from a Codex review: current main already addresses the reported unbounded sessions.json growth by pruning isolated cron run session rows after a retention window. The implementation is in src/cron/session-reaper.ts:1-149, the sweep is wired into the cron timer finally path in src/cron/service/timer.ts:820-849, and the behavior is covered by src/cron/session-reaper.test.ts:73-136 plus src/cron/service.session-reaper-in-finally.test.ts:131-170. This shipped in v2026.4.22 (00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5). Current main still stores duplicate run entries internally, but the specific unbounded-growth path from repeated cron `:run:` entries is now bounded by retention and existing session-store maintenance.
