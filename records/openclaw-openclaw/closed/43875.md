## Summary

Current main already includes shipped fixes for stale packaged `dist` imports after global npm upgrades, plus update/restart hardening that cleans stale gateway processes and verifies the installed package inventory.

## Risks / Open Questions

- This was a read-only review; I did not run a live npm-upgrade reproduction, so the conclusion relies on current code, docs, and shipped changelog evidence rather than a fresh end-to-end runtime trace.

## Close Comment

Closing as implemented on main after a Codex review. OpenClaw now ships the stale-upgrade hardening described in `CHANGELOG.md:598` for v2026.4.15 (`CLI/update: prune stale packaged dist chunks after npm upgrades ... so global upgrades stop failing on stale chunk imports`), and current main verifies the installed package inventory after global updates in `src/infra/update-global.ts:92` and `src/infra/update-runner.ts:1294`. The update path also restarts the gateway by default and cleans stale gateway PIDs after restart in `src/cli/update-cli/update-command.ts:729`, which matches the double-restart symptom from this report. Docs now point package installs at `openclaw update`, which runs doctor and restarts the gateway automatically (`docs/install/updating.md:11`). If someone can still reproduce stale hashed-module imports on v2026.4.22+ with the current update/restart flow, please open a new issue with exact commands, expected vs actual behavior, and `openclaw gateway status --deep` output.
