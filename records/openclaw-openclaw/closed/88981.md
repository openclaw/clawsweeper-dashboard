- **[P1] Decode reactions from the Activity content field:** `extensions/nextcloud-talk/src/monitor.ts:230`
  - body: Nextcloud Talk's bot webhook docs put the reaction emoji in the Like activity's top-level `content`; `object.content` is the reacted-to message body. Reading `payload.object.content` here means real Like/Undo reaction webhooks will not produce the tapped emoji, so approval reaction bindings will not match.
  - confidence: 0.96
- **[P1] Require explicit approvers before dispatching reactions:** `extensions/nextcloud-talk/src/inbound.ts:470-477`
  - body: `authorizeActorAction` returns an authorized result for empty approver sets as an implicit same-chat fallback, and this branch checks only `auth.authorized`. With `allowFrom: []`, any room-allowed sender can resolve a stashed approval by reaction; the PR's own test currently expects that behavior.
  - confidence: 0.95
- **[P2] Persist approval bindings outside process memory:** `extensions/nextcloud-talk/src/approval-stash.ts:39`
  - body: The new approval stash is a module-level `Map`, but the approval prompt is visible in Talk after delivery. If the gateway restarts before a user reacts, the message remains but the binding is gone; use the shared persistent approval-reaction store pattern instead.
  - confidence: 0.9
- **[P1] Port this onto the current message-action adapter:** `extensions/nextcloud-talk/src/channel.ts:129-134`
  - body: Current main already wires `nextcloudTalkMessageAdapter` and `nextcloudTalkMessageActions`, including focused reaction handling and explicit reaction-removal rejection. This PR defines a local older adapter and omits current main's message adapter wiring, so it needs to be rebuilt on the current channel structure before review can continue.
