## Summary

Keep open. Current `main` has cross-session and broadcast primitives, but not the requested conference-style shared multi-agent session. The shipped behavior is still 1:1 `sessions_send` plus isolated multi-agent broadcast handling, so there is no strong basis to close this as implemented, obsolete, or incoherent.

## Evidence

- **Session tools are still 1:1:** The docs describe `sessions_send` as sending to another session, with an optional reply-back loop where agents alternate messages. That is bilateral messaging, not a shared multi-participant session.
  - file: [docs/concepts/session-tool.md:68](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/docs/concepts/session-tool.md#L68)
  - command: `nl -ba docs/concepts/session-tool.md | sed -n '1,120p'`
  - sha: [aa3a5f9ff790](https://github.com/openclaw/openclaw/commit/aa3a5f9ff7907c2a41d6259d24b529bda41f293a)
- **Tool schema accepts a single target:** `sessions_send` only accepts one `sessionKey` or `label`/`agentId` target plus one `message`; there is no participant list, room id, moderator, or conference lifecycle field.
  - file: [src/agents/tools/sessions-send-tool.ts:37](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/src/agents/tools/sessions-send-tool.ts#L37)

- **A2A flow is explicit ping-pong between two sessions:** The implementation swaps `currentSessionKey` and `nextSessionKey` in a loop and then performs a single announce send. This is a two-party exchange, not a shared session with multiple participants.
  - file: [src/agents/tools/sessions-send-tool.a2a.ts:70](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/src/agents/tools/sessions-send-tool.a2a.ts#L70)
