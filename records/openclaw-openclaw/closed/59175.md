## Summary

Current main already ships the Synology Chat secure-by-default TLS behavior that this PR is trying to add. The client helpers, account default, tests, docs, security warning surface, and release changelog all now show that certificate validation is on unless `allowInsecureSsl: true` is explicitly configured, so the PR is redundant rather than a remaining source change to merge.

## What This Changes

The PR changes Synology Chat client, account resolution, setup, webhook handling, config schema, and tests so `allowInsecureSsl` is secure by default while explicit insecure mode remains available with warnings.

## Best Possible Solution

Keep the shipped secure-by-default implementation on main and close this stale duplicate PR; if maintainers still want extra setup-prompt polish, that should be a separate narrow follow-up rather than merging this redundant branch.

## Reproduction Assessment

Yes. The behavior is reproducible through the existing mocked Synology Chat client tests that inspect `https.request` and `https.get` options for default and explicit insecure modes.

## Solution Assessment

Yes. The current main implementation is the narrow maintainable fix for the security bug: default TLS verification stays enabled, explicit insecure mode remains available, and user-facing docs/warnings cover the opt-in risk.
