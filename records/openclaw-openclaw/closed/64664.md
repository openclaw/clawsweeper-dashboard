## Summary

Keep open. Current main still drops pending approval IDs on gateway restart and the Telegram inline callback plus generic `/approve` reply paths still surface or rethrow the raw not-found failure instead of clear expiry/restart guidance; the related implementation PRs in the provided context are closed unmerged.

## What This Changes

The issue asks for stale approval interactions after a gateway restart to stop showing a confusing raw error, with persistence suggested as a possible broader fix.

## Best Possible Solution

Translate approval-not-found failures into actionable expiry/restart guidance at the generic `/approve` and Telegram inline callback surfaces, while leaving durable pending-approval persistence for a separate design decision.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: restart creates fresh in-memory approval managers, and existing `/approve` plus Telegram tests already simulate `unknown or expired approval id` paths that still expect raw failure behavior.

## Solution Assessment

Yes for the immediate bug. Mapping the existing approval-not-found classifier at the user-facing reply/callback handlers is the narrowest maintainable fix; persisting approvals across restarts is broader than needed for the confusing stale-button error.
