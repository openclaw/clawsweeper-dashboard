## Summary

Keep open. Current main still has the reported uncaught transient-network gap, so the PR is not obsolete, but the proposed patch has a blocking correctness issue: it makes the shared benign uncaught-exception classifier accept all transient network errors before any fatal/config precedence check, which can suppress fatal wrappers that contain a transient network cause. The related cleanup context also points to #60515 and #63780 as the canonical issue/implementation thread for this crash class.

## What This Changes

The PR changes the shared uncaught-exception classifier so transient network errors are logged as non-fatal, and updates infra/CLI tests for raw Telegram EHOSTUNREACH handling.

## Best Possible Solution

Implement uncaught transient-network handling in one canonical PR with fatal/config precedence preserved, and add coverage for both raw EHOSTUNREACH continuing and fatal wrappers with nested network causes still exiting.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve fatal precedence before suppressing network exceptions:** `src/infra/unhandled-rejections.ts:345-347`
  - body: The new early `isTransientNetworkError(err)` check reuses a classifier that walks nested causes, but the uncaught-exception path has no preceding fatal/config guard. A fatal wrapper such as `ERR_WORKER_UNCAUGHT_EXCEPTION` with an `EHOSTUNREACH` cause would now be logged as non-fatal and skip fatal hooks/exit; mirror the unhandled-rejection fatal-before-transient ordering or explicitly exclude fatal wrappers before returning true.
  - confidence: 0.9
