## Summary

Keep open: current main and v2026.5.28 still accept per-entry hook config keys that do not declare or load handler modules, while the supported surfaces are discovery-based hook directories, `hooks.internal.load.extraDirs`, hook installs, and legacy `hooks.internal.handlers`.

## What This Changes

The issue reports that `hooks.internal.entries` accepts TypeScript handlers, `extraDirs`, and `installs` without effect or diagnostics while `bootstrap-extra-files` works.

## Best Possible Solution

Keep the discovery-based hook contract and add focused validation, doctor, or hook-status diagnostics for unknown configured hook names and misnested loader keys, without adding a parallel `entries.<id>.handler` API.

## Reproduction Assessment

Yes at source level: configure `hooks.internal.entries.<id>.handler` without a matching discovered hook and current code will accept the key but load zero handlers. I did not run a live Windows gateway because this review is read-only.

## Solution Assessment

Yes for the diagnostic half: the narrow maintainable fix is to warn or reject unsupported per-entry loader shapes and point users to `HOOK.md` discovery, `hooks.internal.load.extraDirs`, hook installs, or legacy handlers. Adding a new `entries.<id>.handler` declaration API would be a larger product/API choice and is not needed to fix the silent failure.
