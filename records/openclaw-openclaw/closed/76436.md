## Summary

Current main already prevents the reported failure: bare `/new`/`/reset` turns get a non-empty internal transcript marker, and memory session indexing drops blank content before any OpenAI-compatible embedding batch is sent. The fix is present on current main but I could not prove it is included in `v2026.5.2`.

## What This Changes

The issue asks OpenClaw to handle bare `/new` or `/reset` messages without sending empty text to a ZhipuAI OpenAI-compatible embedding endpoint during memory search.

## Best Possible Solution

Keep the current main behavior and ship it in the next release; avoid provider-specific placeholder hacks for ZhipuAI when the generic reset and memory-indexing boundaries already prevent blank embedding calls.

## Reproduction Assessment

No current-main failing reproduction path remains. Source inspection shows Feishu `/new` reaches the shared reset path, the reset transcript marker is non-empty, and blank session chunks are filtered before embedding.

## Solution Assessment

Yes. The current solution is the narrow maintainable fix because it preserves transcript semantics and blocks empty embedding batches generically instead of adding a ZhipuAI-only workaround.
