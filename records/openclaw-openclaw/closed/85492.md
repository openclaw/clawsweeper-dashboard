## Summary

Keep open: this is a plausible focused bug fix for cron announce control-token leaks, but it should not merge as-is because the patch creates a second hard-coded control-reply classifier and still lacks after-fix real behavior proof.

## What This Changes

The branch adds cron-delivery suppression for exact `REPLY_SKIP` and `ANNOUNCE_SKIP` responses and regression coverage in core-channel direct announce tests.

## Best Possible Solution

Move the suppressed control-reply predicate into a shared gateway-neutral helper, have both chat and cron delivery call it, keep the focused regression tests, and attach redacted real cron delivery proof before merge.

## Reproduction Assessment

Yes from source inspection: current main suppresses only `NO_REPLY` in the cron delivery normalizer, while the documented control-reply contract and linked issue expect exact `REPLY_SKIP` and `ANNOUNCE_SKIP` to be suppressed too. I did not run a live channel repro in this read-only review.

## Solution Assessment

No: wiring suppression into cron delivery is the right narrow surface, but the patch should share the existing control-reply classifier instead of adding a second hard-coded token list.
