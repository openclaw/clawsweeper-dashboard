## Summary

This request is better scoped as a plugin/ClawHub feature than an OpenClaw core change. Current main already exposes the needed `before_tool_call` interception, plugin-owned config, and Codex bridge behavior, while the project vision explicitly says optional capabilities and new skills should ship outside core when plugin paths exist.

## Evidence

- **Project scope prefers plugins over new core optional behavior:** The vision doc says optional capability should usually ship as plugins, to prefer bundle-style plugins when they can express the capability, and that plugin promotion belongs in ClawHub; new skills should go through ClawHub first.

  - sha: [5fe333ada85a](https://github.com/openclaw/openclaw/commit/5fe333ada85afd0def9e85985ce57e36aa00108a)

## Risks / Open Questions

- No published ClawHub/plugin implementation was linked on this issue, so closing the core item does not itself deliver an `xread` router for users.
- Codex-native bridging can block with a reason, but the docs note it does not yet rewrite native tool arguments, so a plugin solution would be block-and-instruct rather than transparent argument redirection.
