## Summary

Keep open: the performance problem and contributor proof are credible, but this PR’s unkeyed process-global TTL cache can reuse plugin metadata across incompatible config, workspace, env, or installed-index scopes and is missing the required changelog entry.

## What This Changes

The PR adds a process-global five-second TTL cache around `loadPluginMetadataSnapshot()` in `src/plugins/plugin-metadata-snapshot.ts`.

## Best Possible Solution

Keep the performance fix direction, but implement reuse through explicit compatible `PluginMetadataSnapshot` or `PluginLookUpTable` ownership, or a fully compatibility-keyed cache with invalidation coverage and a changelog entry.

## Reproduction Assessment

Yes for the PR defect: applying the branch and calling `loadPluginMetadataSnapshot()` with different config, workspace, env, or index inputs inside five seconds would hit the unkeyed return path. I did not live-reproduce the original current-main UI freeze, but the PR body includes after-fix live output from the reporter's setup.

## Solution Assessment

No. The performance goal is valid, but the current unkeyed wall-clock cache bypasses the existing compatibility contract; explicit snapshot reuse or a fully scoped cache is safer.
