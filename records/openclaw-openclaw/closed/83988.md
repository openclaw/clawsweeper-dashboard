## Summary

This PR should stay open: the Telegram root cause is source-reproducible, but the branch is not merge-ready because it changes non-deferred TTS media semantics, currently has red CI, and still lacks real Telegram behavior proof.

## What This Changes

The PR defers ACP visible-text settlement for final-mode Telegram TTS, sends accumulated text with the generated voice media payload, and adds dispatch tests for ordering and fallback behavior.

## Best Possible Solution

Keep the Telegram-only settlement deferral, but build final TTS payloads conditionally: caption text only for the deferred Telegram path and the existing audio-only `visibleTextAlreadyDelivered` supplement for non-deferred channels.

## Reproduction Assessment

Yes, source-level: current main settles ACP visible text before final-mode TTS synthesis, so Telegram can display text before a later voice payload. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No: the direction is right for Telegram, but this branch still attaches visible text to non-deferred final TTS media and does not typecheck. The safer solution is conditional payload construction plus real Telegram proof.
