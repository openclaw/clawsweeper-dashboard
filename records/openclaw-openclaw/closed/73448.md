## Summary

Keep open. Current main still exposes `asVoice` on the shared message tool but drops it before the shared send, direct delivery, and gateway send paths; open PR #73483 already uses closing syntax for this issue and open PR #74920 overlaps the same media/send path, so the issue should remain open until one implementation lands or is superseded.

## What This Changes

The issue requests preserving the shared message-tool `asVoice` flag through outbound send handling so audio can reach downstream delivery as `audioAsVoice`.

## Best Possible Solution

Land one narrow shared pass-through that maps message-tool and CLI `asVoice` to downstream `audioAsVoice` through the send runner, core send service, direct/gateway send contracts, generated protocol clients where needed, docs, and focused regression tests, while leaving plugin-owned transcoding behavior in channel plugins.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: `asVoice` is accepted by the shared message tool schema, but the inspected runner, core send service, direct send type, and gateway send contract have no field that can carry it to delivery.

## Solution Assessment

Yes. The requested direction is the narrow maintainable fix because lower-level outbound delivery and Feishu media handling already honor `audioAsVoice` once it reaches them; the remaining work is preserving that intent across the shared surfaces.
