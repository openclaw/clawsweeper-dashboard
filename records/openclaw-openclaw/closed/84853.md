## Summary

Keep this PR open: the code change is focused and the source path supports the bugfix, but the protected maintainer label, missing after-fix real gateway proof, and deliberate suppression of generic exec progress events need maintainer signoff before merge.

## What This Changes

This PR returns early for exec `tool_execution_update` events suppressed by the live-output throttle and adds regression coverage that the throttled update emits no global or local agent events.

## Best Possible Solution

Land the focused early-return fix only after maintainer acceptance of the exec progress-event contract and redacted after-fix gateway/tool-run proof; if clients need a heartbeat, preserve a lightweight explicit signal instead of the redundant generic update flood.

## Reproduction Assessment

Partly: current main source clearly shows throttled exec updates still emit generic item/tool traffic, and the PR test exercises that seam. I do not have high-confidence live reproduction of the reported CPU saturation after the patch.

## Solution Assessment

Likely yes: the early return is the narrowest handler-level fix after the throttle state check and preserves start/final/non-exec paths. The open question is whether dropping generic command/item progress events is an accepted compatibility tradeoff.
