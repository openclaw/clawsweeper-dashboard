## Label Justifications

- `P1`: A never-settling plugin-owned compaction can leave an interactive agent workflow unresponsive during a turn.
- `impact:crash-loop`: The report is about process-level availability and agent hangs during compaction, even if the process does not literally crash.
- `impact:session-state`: The failure occurs inside session compaction and retry handling, leaving the active session lane blocked behind in-flight state work.

## Summary

Keep open: current `main` still directly awaits plugin-owned `ContextEngine.compact()` in the reported host and Codex harness lanes, and the issue is paired with open fixing pull request https://github.com/openclaw/openclaw/pull/84083. The related hook-timeout gap is separately represented by https://github.com/openclaw/openclaw/pull/84153, so this issue should close only after the compact-timeout fix merges.

## What This Changes

Bound plugin-owned context-engine compaction so an `ownsCompaction` plugin cannot hang an agent turn indefinitely when its `compact()` promise is slow or never settles.

## Best Possible Solution

Land a reviewed fix that applies the existing compaction safety-timeout pattern to every plugin-owned `compact()` path and adds cooperative abort support without forcing existing context-engine plugins to change.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure an `ownsCompaction` engine whose `compact()` never resolves, then trigger queued/manual, timeout-recovery, overflow-recovery, or Codex-owned compaction; current main awaits that promise directly.

## Solution Assessment

Yes. Reusing the existing compaction safety-timeout pattern for plugin-owned `compact()` calls, plus an optional abort signal for cooperative engines, is the narrow maintainable fix path.
