## Label Justifications

- `P2`: The report affected memory sync for a specific remote embedding configuration, not core runtime availability.
- `impact:session-state`: The failure surface is post-compaction memory/session indexing, which can leave semantic memory out of sync.
- `impact:auth-provider`: The reported breakage centered on embedding provider routing and API-key fallback semantics.

## Summary

Close: current main and the shipped v2026.5.27 release provide the supported `openai-compatible` memory embedding path for self-hosted OpenAI-style `/v1/embeddings` endpoints without requiring an API key, while keeping that path intentionally separate from `provider: "openai"` credential semantics.

## What This Changes

The issue requests that post-compaction memory embedding sync use a configured self-hosted OpenAI-compatible `remote.baseUrl` without falling back to OpenAI API credentials.

## Best Possible Solution

Keep `openai-compatible` as the canonical self-hosted embedding route and treat any future `provider: "openai"` no-auth override as a separate product decision.

## Reproduction Assessment

No live reproduction was run for the original host. Current source and tests establish that the supported `openai-compatible` path routes to `remote.baseUrl` and omits Authorization when no API key is configured, so the central self-hosted endpoint failure is not reproducible on current main with the supported provider.

## Solution Assessment

Yes. The separate `openai-compatible` provider is the maintainable fix because it supports local/self-hosted OpenAI-style embedding endpoints without inheriting global OpenAI chat credentials or changing `provider: "openai"` semantics.
