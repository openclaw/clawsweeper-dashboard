## Summary

Close as duplicate/superseded: current main and v2026.6.5 still contain the Synology Chat 120s local delivery timeout, but the same root cause is already tracked by https://github.com/openclaw/openclaw/issues/59966 and has an open, proof-positive fix candidate at https://github.com/openclaw/openclaw/pull/90592.

## What This Changes

The issue reports Synology Chat discarding long-running agent replies because its webhook handler races delivery against a hardcoded 120s timer.

## Best Possible Solution

Keep the canonical timeout work centered on https://github.com/openclaw/openclaw/issues/59966 and land or replace https://github.com/openclaw/openclaw/pull/90592, preserving this report as linked real-world evidence.

## Reproduction Assessment

Yes. Source inspection on current main shows every authorized Synology webhook delivery is raced against setTimeout(..., 120_000), and the reporter's logs align with that 120s boundary; I did not run a live NAS reproduction.

## Solution Assessment

Yes for closing as a duplicate. The best fix is to remove the Synology-specific local timeout and rely on the shared session/runtime timeout path, which is exactly what the existing open fix PR targets.
