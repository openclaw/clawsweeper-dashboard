## Summary

This should close because the reported threshold behavior belongs to the external `lossless-claw` context-engine plugin, while current OpenClaw main already supplies active-attempt model budget/model data through the context-engine contract.

## What This Changes

The issue asks for `lossless-claw` compression thresholds to adapt to the currently active fallback model context window instead of the originally configured primary model window.

## Best Possible Solution

Keep the OpenClaw context-engine contract as-is and fix `lossless-claw` to consume the active `tokenBudget` or active model data; open a new OpenClaw issue only if the plugin can show a missing SDK field or a desired fallback-before-compaction policy change.
