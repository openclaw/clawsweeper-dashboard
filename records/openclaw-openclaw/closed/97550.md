## Summary

Keep open: current main and the latest release still use an unbounded APNs relay `response.json()` read, while this PR is focused, proof-backed, and has no concrete code defect found; the remaining merge decision is maintainer acceptance of the intentional over-16 MiB fail-closed behavior.

## What This Changes

The PR replaces APNs relay response parsing with a 16 MiB bounded read via `readResponseWithLimit` and adds regression tests for malformed, empty, BOM-prefixed, under-cap, and oversized relay replies.

## Best Possible Solution

Land the bounded relay read after maintainers accept the over-cap fail-closed behavior and exact-head gates finish cleanly.

## Reproduction Assessment

Yes from source inspection and PR proof: current main and the latest release use `response.json()` on the relay response, while the PR body provides loopback live output showing the bounded path stops before the full oversized body.

## Solution Assessment

Yes, with maintainer acceptance: using the existing bounded response helper at the APNs relay boundary is the narrowest maintainable fix, and a new configurable cap would add config surface without evidence that normal relays need it.
