## Summary

Current main no longer has a control-UI codepath that reads local skill markdown for the skills panel, and the remaining workspace-skill status path preserves the workspace file path/source. The reported Homebrew-path read does not match the current main implementation.

## Evidence

- **Control UI only loads skill status:** The current skills controller requests only `skills.status` for the local skills view.
  - file: [ui/src/ui/controllers/skills.ts:105](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/ui/src/ui/controllers/skills.ts#L105)
  - sha: [7ac35b4f6900](https://github.com/openclaw/openclaw/commit/7ac35b4f69009490180a35bf404638d4280a4865)
- **Skills view renders metadata, not SKILL.md contents:** The skills dialog shows `skill.source` and `skill.filePath`; there is no local file-read call in the current view.
  - file: [ui/src/ui/views/skills.ts:551](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/ui/src/ui/views/skills.ts#L551)
  - sha: [7ac35b4f6900](https://github.com/openclaw/openclaw/commit/7ac35b4f69009490180a35bf404638d4280a4865)
- **No skills markdown-read RPC exists:** The gateway method list for this surface is `skills.status`, `skills.search`, `skills.detail`, `skills.bins`, `skills.install`, and `skills.update`; there is no current `skills.read`/docs-open endpoint for local SKILL files.
  - file: [src/gateway/server-methods-list.ts:66](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/src/gateway/server-methods-list.ts#L66)
  - sha: [7ac35b4f6900](https://github.com/openclaw/openclaw/commit/7ac35b4f69009490180a35bf404638d4280a4865)

  - file: [src/agents/skills/workspace.ts:533](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/src/agents/skills/workspace.ts#L533)

## Close Comment

Closing from a Codex review because I can’t confirm this on current `main` (`7ac35b4f69009490180a35bf404638d4280a4865`). The current Control UI skills panel only requests `skills.status` (`ui/src/ui/controllers/skills.ts:105`) and renders the returned `skill.source` / `skill.filePath` (`ui/src/ui/views/skills.ts:551`); the current gateway surface for skills is only `skills.status/search/detail/bins/install/update` (`src/gateway/server-methods-list.ts:66`, `src/gateway/server-methods/skills.ts:69`), so there is no current control-UI local-SKILL.md read path that would try `/opt/homebrew/.../skills/.../SKILL.md`. On the backend, workspace skills are still loaded from `<workspaceDir>/skills` as `openclaw-workspace` (`src/agents/skills/workspace.ts:533`), and `buildWorkspaceSkillStatus()` passes through the actual `filePath` unchanged (`src/agents/skills-status.ts:206`). Current main also has regression coverage for a workspace skill overriding a bundled name (`src/agents/skills.buildworkspaceskillstatus.test.ts:133`). If this is still reproducible on a current-main build, please open a fresh issue with the exact UI action and a current log snippet so it can be narrowed down against the present codepath.
