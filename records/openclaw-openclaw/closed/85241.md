## Summary

Keep this PR open: the reported regression is source-reproducible on current main and the patch addresses the parent-session case, but it still needs maintainer handling for real A2A proof or proof override and for the shared loopback lifecycle gap before merge.

## What This Changes

The PR changes the MCP loopback singleton to report whether the caller created it, stores that fact in CLI run preparation, gates run-end cleanup on it, and adds focused lifecycle tests.

## Best Possible Solution

Use an active-run lease or refcount for the shared MCP loopback server so every run releases its claim and only the last release closes it, then land with focused regression coverage plus live A2A proof or an explicit maintainer proof override.

## Reproduction Assessment

Yes, by source inspection: current main reuses a process-wide MCP loopback singleton and cleanup-enabled runs close it unconditionally at run end. I did not run a live A2A gateway repro in this read-only review.

## Solution Assessment

No, not yet. The creator bit fixes the reported parent-created server ordering, but a lease/refcount is the safer lifecycle fix for any overlapping run that shares the singleton.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Track active loopback users before closing:** `src/gateway/mcp-http.ts:248`
  - body: The new `created` bit only protects a later reuser from closing the server. If two cleanup-enabled runs overlap and the creator finishes first, this still calls `closeMcpLoopbackServer()` while the reuser is active, recreating the same `fetch failed` class for that run. Please model the shared loopback as a lease/refcount and close only on last release.
  - confidence: 0.87

Please set the ownership fact for the owned cleanup case and add a false case for reused servers.
  - confidence: 0.9
