## Summary

Keep this PR open but block merge: the startup logging idea is useful, and the PR body includes real startup proof, but the current diff swallows Teams listener startup failures and turns them into a null-app success result. That violates current main's fail-fast startup contract and existing lifecycle coverage.

## What This Changes

The PR changes Microsoft Teams webhook startup to create a Node HTTP server, bind it on 0.0.0.0, add startup error logging, and make small handler callback edits.

## Best Possible Solution

Preserve the Teams monitor fail-fast startup contract, keep improved startup logging only if startup errors still reject or rethrow, and leave configurable bind-host behavior to the dedicated host-binding path.

## Reproduction Assessment

Yes for the review finding: source inspection of the PR diff shows startup errors become a null-app success result, while current main and lifecycle coverage require rejection on listen failure. No high-confidence reproduction was established for the PR's claimed original `expressApp.listen` failure.

## Solution Assessment

No: explicit HTTP server construction and clearer logging could be acceptable, but swallowing startup failures is not the best fix. The safer repair is to log and still reject or rethrow, with bind-host configurability handled by https://github.com/openclaw/openclaw/pull/63347.
