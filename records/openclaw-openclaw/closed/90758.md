## Summary

Current main already solves the escape-hatch routing need with canonical top-level route bindings, including channel-wide, account-scoped, direct-peer, and group-peer matches across Telegram, WhatsApp, and iMessage.

## What This Changes

The issue asks for optional channel-level agentId routing for iMessage, WhatsApp, Telegram DMs, and Telegram groups so each channel can target a different agent/provider as a failover path.

## Best Possible Solution

Keep using the documented top-level bindings[] route surface for agent routing instead of adding duplicate channels.*.agentId config keys.

## Reproduction Assessment

Not applicable; this is a requested config/routing capability rather than a broken current behavior. Source inspection shows the central routing path is already available through bindings[].

## Solution Assessment

Yes for the central problem: top-level bindings are the canonical and more general routing solution. Adding channel-local agentId keys would duplicate a compatibility-sensitive config surface without proving the existing route binding surface is insufficient.
