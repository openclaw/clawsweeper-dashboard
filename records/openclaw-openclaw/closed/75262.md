## Summary

Keep open. The PR is a useful start, but it should not merge as-is: it adds CLI/Gateway/service flag plumbing, then implements the scheduled cwd piece with `process.chdir` during an awaited isolated run even though current main already has run-scoped `workspaceDir` seams. It also leaves the user-facing docs/changelog/tests absent and does not address the linked report's broader env/context parity request.

## What This Changes

The PR adds an `--as-scheduled` option to `openclaw cron run`, threads `asScheduled` through CLI, Gateway schema/handler, cron service, timer, and isolated runner APIs, and changes flagged isolated manual runs to chdir into the agent workspace.

## Best Possible Solution

Implement scheduled-environment parity as explicit run-scoped cron execution policy: pass workspace, environment, and context behavior through the existing runner seams, avoid process-global cwd/env mutation, document the CLI/Gateway protocol option, add an active changelog entry, and cover manual-versus-scheduled parity with focused cron tests.

## Reproduction Assessment

Yes for the PR regression. The patch visibly calls `process.chdir` before awaiting cron execution, and current main already passes `workspaceDir` into the CLI and embedded cron runners as run-scoped state.

## Solution Assessment

No. The `--as-scheduled` direction may be useful, but mutating process cwd is not the best way to solve it; the safer path is a typed run-scoped execution option that applies cwd/env/context at the runner boundary.
