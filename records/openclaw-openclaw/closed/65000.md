## Summary

Close #65000 as superseded by open issue #67777. Current main still does not prove durable parent-facing subagent/ACP completion delivery across reconnect/retry gaps, but the same remaining failure class is now tracked more concretely in #67777, with hardening work represented by open PR #68464.

## Best Possible Solution

Close #65000 as superseded by #67777, keep #67777 as the canonical bug for durable subagent/ACP completion redelivery, and continue any implementation review in #68464 or a successor PR. The likely product fix belongs in the subagent/ACP completion delivery path as a persisted parent-owned delivery obligation rather than transient direct delivery or ephemeral system-event fallback.

## Evidence

- **Original issue scope:** #65000 reports OpenClaw 2026.4.9 reconnect-gap behavior where the child/subagent run completes, but parent-facing final delivery is unreliable or delayed, and asks for durable pending-delivery context across retry/restart/reconnect paths.
- **Canonical related tracker:** The provided related context identifies open issue #67777, "Subagent completion delivery can be lost on direct-announce timeout, drain, or orphan prune," as kept open because current main still lacks durable completion redelivery. That is the same remaining product problem as #65000, expressed with narrower current failure modes.
- **Completion dispatch still has a non-durable failure path:** Completion announce dispatch remains direct-first, only tries queue fallback after direct failure, and returns the original direct failure when fallback queueing cannot deliver. That confirms the durable-retry concern is not separately fixed in #65000.
