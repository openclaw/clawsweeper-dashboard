## Summary

Keep open. Current main still starts outbound delivery recovery on gateway startup and replays pending delivery-queue entries in enqueue order with only retry/backoff/max-retry gates. I found no age threshold, stale-drop policy, or queue metadata that distinguishes ephemeral transcript/confirmation notices from durable outbound replies. The linked Discord restart issues explain one source of frequent restarts, but their closure does not resolve this delivery-recovery behavior.

## Best Possible Solution

Keep this issue open and fix it in the outbound delivery queue/recovery owner path. The conservative direction is to add explicit queued-delivery metadata for recovery policy or expiry, make ephemeral transcript/restart/confirmation notices non-recoverable or short-lived, and add regression tests proving old pending entries are logged/moved aside instead of batch-delivered on startup while durable recent replies still recover.
