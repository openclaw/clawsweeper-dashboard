## Summary

Current main and the shipped v2026.5.28 Feishu plugin already resolve Feishu send/thread-reply text with `text` before `message`, so the reported v2026.5.20 behavior is fixed by updating.

## What This Changes

The issue asks Feishu message-tool sends to honor `text` when both `text` and legacy `message` are supplied.

## Best Possible Solution

Keep the shipped Feishu handler behavior and have affected users update OpenClaw and `@openclaw/feishu` to v2026.5.28 or newer.

## Reproduction Assessment

No current-main failure path was found. The provided steps are plausible for the reported v2026.5.20 plugin, but current main and v2026.5.28 source choose `text` first; I did not run a live Feishu API call in this read-only review.

## Solution Assessment

Yes. The current implementation is the narrow maintainable fix: Feishu resolves one outbound body by checking `text` before the legacy `message` alias and sends that value through the existing outbound paths.
