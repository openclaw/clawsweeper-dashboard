## Label Justifications

- `P2`: This is a real user-visible reply-delivery compatibility issue, but current main treats it as a documented policy/default tradeoff with an explicit config opt-out.

## Summary

Close: the reported Feishu/Lark DM behavior is superseded by the maintainer policy merged in https://github.com/openclaw/openclaw/pull/83571, which intentionally keeps Codex direct/source visible replies tool-gated by default and documents `messages.visibleReplies: "automatic"` as the old-mode opt-out.

## What This Changes

Restore automatic/streaming visible replies for Codex/GPT-backed Feishu/Lark direct messages unless the user explicitly selects message-tool-only/private output.

## Best Possible Solution

Keep the maintainer-approved Codex message-tool default, use `messages.visibleReplies: "automatic"` when operators want the old Feishu/Lark DM streaming path, and open a new issue only for current-main failures with that explicit config.

## Reproduction Assessment

Yes, at source level. Unset Codex direct/source chats resolve through the harness `message_tool` default to `message_tool_only`, and current tests assert the no-final-queued shape; I did not run a live Feishu tenant reproduction.

## Solution Assessment

Yes for cleanup, but not by changing the default. The maintainable current outcome is the merged policy PR plus documented `messages.visibleReplies: "automatic"` opt-out; changing Feishu/Codex direct defaults again would reopen a product decision.
