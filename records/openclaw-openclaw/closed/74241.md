## Summary

Close as superseded: current main still has the timezone gap, but the same remaining work is already tracked by open PR #39610 and its canonical issue #39579. The reporter's production Berlin/London drift context is useful evidence for that PR, not a separate source-level fix path.

## What This Changes

The issue requests user-local timezone injection and per-agent/profile timezone configuration so agent turns and heartbeat scheduling do not use the container or host timezone for user-facing time reasoning.

## Best Possible Solution

Use #39610 as the canonical repair path, update it or its replacement to explicitly preserve the Berlin-container/London-user scenario from this report, and land the shared per-agent timezone resolver through the config, prompt, gateway timestamp, envelope, heartbeat, and current-time helper surfaces.

## Reproduction Assessment

Yes. The source-level reproduction is clear on current main: run with host/container timezone different from the desired user timezone and no global `agents.defaults.userTimezone`, then use gateway `agent`/`chat.send` timestamp injection or heartbeat `activeHours.timezone: "user"`; those paths resolve from defaults or host timezone rather than a per-agent/profile timezone.

## Solution Assessment

Yes. The best direction is the one already represented by #39610: add a shared agent-aware timezone resolver with default/host fallback and thread it through existing timestamp and scheduling seams, rather than injecting an uncached clock into the system prompt or changing cron timezone semantics globally.
