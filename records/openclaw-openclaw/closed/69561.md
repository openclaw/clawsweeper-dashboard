## Summary

Current main already documents and implements the safer macOS permission model: OpenClaw.app owns TCC-facing permissions, Peekaboo automation is hosted through the app bundle, and the latest release already ships that docs state.

## Evidence

- **macOS docs say the app owns TCC prompts:** The main macOS docs state that OpenClaw.app owns Notifications/Accessibility/Screen Recording/Microphone/Speech/AppleScript permissions, and that `system.run` executes in the macOS app over local IPC in UI/TCC context.
  - file: [docs/platforms/macos.md:9](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/docs/platforms/macos.md#L9)
  - command: `nl -ba docs/platforms/macos.md | sed -n '1,220p'`
  - sha: [5fe333ada85a](https://github.com/openclaw/openclaw/commit/5fe333ada85afd0def9e85985ce57e36aa00108a)
- **IPC docs explicitly scope prompts to the GUI app bundle:** The IPC docs say `system.run` is forwarded from the node host service to the macOS app, and that TCC prompts originate only from the GUI app bundle.
  - file: [docs/platforms/mac/xpc.md:60](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/docs/platforms/mac/xpc.md#L60)
  - command: `nl -ba docs/platforms/mac/xpc.md | sed -n '1,140p'`
  - sha: [5fe333ada85a](https://github.com/openclaw/openclaw/commit/5fe333ada85afd0def9e85985ce57e36aa00108a)
- **Peekaboo docs route UI automation through OpenClaw.app permissions:** The Peekaboo docs say the `peekaboo` CLI reuses the macOS app's TCC permissions, rather than instructing users to grant Accessibility directly to a Node runtime.
  - file: [docs/platforms/mac/peekaboo.md:10](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/docs/platforms/mac/peekaboo.md#L10)
  - command: `nl -ba docs/platforms/mac/peekaboo.md | sed -n '1,140p'`
  - sha: [5fe333ada85a](https://github.com/openclaw/openclaw/commit/5fe333ada85afd0def9e85985ce57e36aa00108a)
- **Runtime code requests Accessibility from the macOS app process:** `PermissionManager.ensureAccessibility` calls `AXIsProcessTrusted` / `AXIsProcessTrustedWithOptions` inside the macOS app codepath, which matches the app-scoped permission model described in the docs.
  - file: [apps/macos/Sources/OpenClaw/PermissionManager.swift:85](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/apps/macos/Sources/OpenClaw/PermissionManager.swift#L85)
  - command: `nl -ba apps/macos/Sources/OpenClaw/PermissionManager.swift | sed -n '1,120p'`
  - sha: [5fe333ada85a](https://github.com/openclaw/openclaw/commit/5fe333ada85afd0def9e85985ce57e36aa00108a)
- **Latest release already ships the app-scoped docs:** The latest release commit contains the same app-scoped Peekaboo/XPC docs, so this is not just on unreleased main.
  - file: [docs/platforms/mac/xpc.md:60](https://github.com/openclaw/openclaw/blob/a9797214338ba31c52c796adbb75afb16e0684a9/docs/platforms/mac/xpc.md#L60)
  - command: `git show a9797214338ba31c52c796adbb75afb16e0684a9:docs/platforms/mac/xpc.md | sed -n '1,120p'`
  - sha: [a9797214338b](https://github.com/openclaw/openclaw/commit/a9797214338ba31c52c796adbb75afb16e0684a9)
- **Issue discussion correctly identified the launcher-scoping risk:** The issue comment from `m13v` explained that macOS TCC keys Accessibility to the launching binary. Current docs/implementation address that by routing the documented privileged path through the signed app bundle instead of telling users to grant Accessibility to `node`. The issue timeline also cross-referenced PR #69654 (`docs: warn about node Accessibility grants`).

## Risks / Open Questions

- The docs resolve this by documenting the supported app-scoped flow, not by adding a literal sentence that says "granting Accessibility to node grants it to all npm packages."
