## Summary

Current main and the shipped v2026.4.27 release already implement the Mattermost-specific private-network opt-in. The reported logs show the browser SSRF flag was enabled, but Mattermost uses its own account config path, and current code forwards that Mattermost opt-in to both the status probe and API fetch paths that produced the reported `mattermost-probe` and `mattermost-api` blocks.

## What This Changes

The issue asks for Mattermost deployments on private/internal IP addresses to bypass OpenClaw's SSRF private-network block through configuration so the gateway channel can connect.

## Best Possible Solution

Keep the scoped Mattermost opt-in as the runtime behavior: trusted private/internal Mattermost servers should use `channels.mattermost.network.dangerouslyAllowPrivateNetwork`, while browser SSRF flags remain limited to browser automation.

## Reproduction Assessment

Yes. The provided logs are enough to reproduce the blocked path when only `browser.ssrfPolicy.dangerouslyAllowPrivateNetwork` is set; current-main code shows the Mattermost probe/API paths unblock only when the Mattermost account config opts in.

## Solution Assessment

Yes. A per-Mattermost account flag is the narrowest maintainable solution because it relaxes SSRF checks only for the configured Mattermost channel instead of making a browser or global network policy affect unrelated fetch surfaces.
