## Summary

Keep open: the patch is aimed at the right skills snapshot invalidation path, but the new bump happens after the session update code has already captured the snapshot version, so the first existing-session run after a watch-root change can still reuse the stale `skillsSnapshot`. The supplied proof is a local source probe/replay, not a live Gateway/Claude plugin-dir mount check.

## What This Changes

The PR adds a `watch-targets` skills snapshot change event when `ensureSkillsWatcher` replaces a watcher because resolved watch roots changed, and adds a focused watcher regression test.

## Best Possible Solution

Make watcher-root invalidation visible to the same session snapshot decision, then add coverage through `session-updates` and redacted live proof that the existing Claude plugin-dir mount includes the newly added skill.

## Reproduction Assessment

Yes, source-level only: current `main` rebuilds the watcher without a version bump, and the PR's bump is not observed by the same `session-updates` call because the version is read first. I did not run a live Gateway/Claude mount repro in this read-only review.

## Solution Assessment

No. Bumping on watch-root changes is the right direction, but the version must be re-read or produced before `shouldRefreshSnapshotForVersion` runs, and the regression should cover the actual session snapshot/mount path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Re-read the version before deciding snapshot reuse:** `src/agents/skills/refresh.ts:183-188`
  - body: The new bump fires inside `ensureSkillsWatcher`, but `session-updates` reads `snapshotVersion` before calling that function and then compares the existing snapshot against the old value. Move/recompute the version in the session update path and cover that flow, not only the watcher helper.
