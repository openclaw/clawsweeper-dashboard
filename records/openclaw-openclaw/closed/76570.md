## Summary

Keep open for maintainer review: the PR is aimed at a real config round-trip concern, but this one-line root `.passthrough()` change conflicts with OpenClaw's documented strict config contract and the open canonical policy discussion in #70578.

## What This Changes

The PR changes the root `OpenClawSchema` from `.strict()` to `.passthrough()` and adds a regression test expecting unknown top-level config keys to survive parsing.

## Best Possible Solution

Decide the root config contract in #70578, then land either a strict fail-closed preservation fix or an explicit opaque-root-key design with matching docs, doctor behavior, tests, and security review.

## Reproduction Assessment

Yes. Source and tests give a high-confidence path: current `validateConfigObject` rejects a root `customUnknownField`, matching the documented strict-validation contract rather than the PR's desired passthrough behavior.

## Solution Assessment

No. Changing only the root schema to `.passthrough()` is not the best fix because it changes a documented config validation policy without updating doctor, docs, schema consumers, or the broader #70578 decision.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep root config validation fail-closed:** `src/config/zod-schema.ts:1090`
  - body: Changing the root schema to `.passthrough()` makes OpenClaw accept arbitrary top-level config keys, but the current docs and tests explicitly require unknown keys to fail validation except `$schema`. This would turn typos and removed legacy keys into inert persisted data without the matching doctor/docs/schema policy work, so this should not merge as a one-line schema change.
