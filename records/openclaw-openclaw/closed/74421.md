## Summary

Current main now fixes the likely remaining failure mode behind this report: Telegram already sends audio-only voice notes through preflight and shared media understanding, and a newer main-only provider-resolution fix keeps providers named in `tools.media.audio.models` available even when a partial active capability registry exists. That closes the observable problem on main, although the proof commit is not in a shipped tag yet.

## What This Changes

The issue requests restoring Telegram voice-note auto-transcription so audio-only inbound voice notes are transcribed and included as machine-generated text in the agent context.

## Best Possible Solution

Keep the generic provider-resolution fix on main and let the next release carry it; Telegram should continue using the existing preflight plus shared media-understanding path rather than adding a Telegram-specific STT bypass.

## Reproduction Assessment

Yes. The issue gives a concrete pre-fix path: configure `tools.media.audio.models` for OpenAI, send an audio-only Telegram voice note, and observe whether the transcript reaches `BodyForAgent`; current tests model the Telegram preflight and the missing configured-provider merge that would block STT.

## Solution Assessment

Yes. The maintainable solution is the generic capability-provider merge plus the existing Telegram/media-understanding pipeline, because the reported configuration uses the shared `tools.media.audio` surface rather than a Telegram-only STT API.
