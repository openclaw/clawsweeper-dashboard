## Summary

Keep open. Current main still awaits embedded-run auth initialization before marking the startup `auth` stage, and API-key credentials still pass through provider runtime auth preparation; the newer auth-performance reports are adjacent but do not supersede this API-key custom-gateway case.

## What This Changes

The issue reports a 10-15 second embedded-run `auth` startup stage on API-key-only model auth profiles and asks for diagnosis or a safe optimization path.

## Best Possible Solution

Add fine-grained auth substage timing for this API-key custom-gateway path, then optimize the proven hot subpath while preserving provider `prepareRuntimeAuth` semantics and credential refresh behavior.

## Reproduction Assessment

No high-confidence live reproduction of the 10-15 second timing was run. Source inspection does confirm that the reported `auth` bucket still wraps profile initialization and provider runtime auth preparation on current main.

## Solution Assessment

No for the withdrawn blanket skip/config direction. The safer solution is targeted instrumentation followed by a measured cache, fast path, or parallelization only where provider hook contracts show it is safe.
