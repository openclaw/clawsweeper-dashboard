## Summary

Keep open. Current main does not implement `executeTool`, and the PR targets a real plugin-author gap, but the proposed patch has merge-blocking behavior and security issues: it cannot resolve valid factory-registered tools without explicit names, and it creates a direct tool execution path that skips existing optional-tool policy and hook/audit behavior.

## What This Changes

The PR adds a public `api.executeTool(toolName, params)` method to `OpenClawPluginApi`, wires it through the plugin API builder and registry, and adds test-helper no-op stubs.

## Best Possible Solution

Keep the PR open and rework the feature around a shared tool-resolution/execution helper, or explicitly narrow and document it as a non-policy-gated internal helper. The maintainable end state should resolve actual factory-produced tool names, preserve user opt-in for optional tools, define hook/diagnostic semantics, pass an intentional context shape, and add targeted contract tests plus SDK docs.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is available: register a valid factory tool without `opts.name` like the bundled browser plugin, then call `executeTool("browser", ...)`; the PR lookup checks an empty `entry.names` list and throws `Tool not found` before the factory can return the tool.

## Solution Assessment

No. Adding a public helper may be useful, but this implementation is not the best path because it bypasses existing policy/hook execution boundaries and does not match current factory-based tool registration semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Resolve factory tool names before rejecting lookup:** `src/plugins/registry.ts:1642-1644`
  - body: `registerTool` leaves `entry.names` empty for factory registrations unless the plugin passed `opts.name`/`opts.names`, while normal tool resolution executes the factory and reads the returned tool's `name`. With this lookup, valid bundled patterns such as the browser and Lobster factories will always throw `Tool not found`; resolve the owned factory output before failing, or persist names at registration.
  - confidence: 0.93
- **[P1] Honor optional-tool allowlists before executing:** `src/plugins/registry.ts:1658`
  - body: This path executes an owned tool directly from `registry.tools` without the optional-tool allowlist check that `resolvePluginTools` applies. Optional tools are the documented opt-in surface for side-effectful plugin tools, so `executeTool` can run an optional tool even when the user has not enabled it; reject disallowed optional entries or route through the same policy-aware resolver.
  - confidence: 0.88
- **[P1] Preserve tool-call hooks for direct plugin execution:** `src/plugins/registry.ts:1658`
  - body: Calling `tool.execute` here skips the existing `before_tool_call` wrapper, fail-closed hook behavior, diagnostic events, and loop-outcome recording used by the agent, MCP, and HTTP tool paths. That means policy plugins cannot block or audit this new public execution path; route through a guarded execution helper or document and restrict the API so it is not treated as a normal tool call.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The PR adds a new public plugin-triggered tool execution path that directly invokes tools without the existing policy, hook, audit, and context boundaries.

Concerns:

- **[medium] Direct tool execution bypasses policy hooks:** `src/plugins/registry.ts:1658`
  - body: `executeTool` resolves an owned registry entry and calls `tool.execute` directly. That bypasses optional-tool allowlisting, `before_tool_call` hooks and approvals, diagnostic events, loop outcome recording, and the fuller tool context used by existing execution paths, which is a concrete security and audit regression for side-effectful plugin tools.
