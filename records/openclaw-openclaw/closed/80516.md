## Summary

Keep open: current main still treats unknown `plugins.deny` ids as hard validation issues while `plugins.allow` and `plugins.entries` are warnings, and this PR is a focused fix with matching regression coverage and sufficient real behavior proof. I found no blocking correctness issue in the diff; the remaining maintainer work is choosing this narrower PR versus the overlapping open PR and handling the required-check state.

## What This Changes

The PR changes config plugin validation so stale `plugins.deny` ids warn instead of failing, suppresses misleading install hints for denied official-external ids, adds focused validation tests, and updates the changelog.

## Best Possible Solution

Land this validation-layer fix or the equivalent canonical PR, then close the duplicate PR path and keep broader doctor atomic-write behavior as a separate design decision.

## Reproduction Assessment

Yes. Current main source shows a high-confidence reproduction path: a config with only a stale `plugins.deny` id becomes invalid because the deny call lacks `warnOnly`, while the same shape in allow/entries is accepted as a warning.

## Solution Assessment

Yes. Changing the validation source is the narrowest maintainable fix because it aligns doctor and gateway behavior for the same stale config drift, while `suppressInstallHint` avoids telling users to install a plugin they intentionally denied.
