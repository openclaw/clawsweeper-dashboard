## Summary

Keep open. Current main still has the reported event gap: `/goal` mutates session metadata through the chat-command pipeline, while `sessions.changed` is only emitted from gateway RPC/lifecycle paths; the related `/name` PR at https://github.com/openclaw/openclaw/pull/88581 explicitly leaves this as follow-up work.

## What This Changes

Emit `sessions.changed` for in-chat commands that mutate session metadata, currently `/goal` on main and `/name` if the linked command PR lands.

## Best Possible Solution

Add one internal command-mutation signal that bubbles from store-mutating chat commands to gateway chat dispatch, reuse the existing `sessions.changed` payload shape, and cover `/goal` plus future command handlers with regression tests and docs.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: run a store-mutating `/goal` command while subscribed to session events and the command path persists state without any `sessions.changed` signal. I did not run a live Gateway repro because this review was read-only.

## Solution Assessment

Yes, the best fix is a narrow internal signal from command handlers to a gateway context that can call the existing session-change broadcaster. Emitting directly from config/session helpers or adding polling would create a second source of event semantics.
