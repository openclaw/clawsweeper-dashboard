## Summary

Keep open: this is a focused bug-fix PR, but the patch remains incorrect. Current env merging ignores `undefined` overrides, and the update CLI also blocks package updates when the updater is still under the active gateway process tree.

## What This Changes

The PR adds undefined gateway-service marker overrides to `runAutoUpdateCommand`, a regression test for that override shape, and a changelog entry for gateway auto-update.

## Best Possible Solution

Replace this with a service-safe auto-update handoff that actually removes inherited service state and runs outside the live gateway ancestry while preserving the manual in-service update block.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: service env injects the markers, the current child env merge preserves inherited values, and update-cli tests/source show both marker and gateway-ancestry package-update blocks.

## Solution Assessment

No. The proposed env override is not the best fix because `undefined` values are ignored by the current merge contract and the updater child remains inside the gateway process tree.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.96

Full review comments:

- **[P1] Delete inherited markers in the resolved child env:** `src/infra/update-startup.ts:285-286`
  - body: Setting these keys to `undefined` does not remove values inherited from `process.env`: `runCommandWithTimeout` resolves env by copying the base env first, and `assignChildEnvValue` returns on undefined overrides. A gateway-launched child still sees both markers and hits the package-update guard.
  - confidence: 0.97

  - confidence: 0.92
