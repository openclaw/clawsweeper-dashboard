## Summary

Keep open: current main still renders managed inbound media-store paths as model-visible `media://inbound/<id>` prompt refs, while only images are inlined into the current turn. Built-in `read`, `pdf`, and media-understanding paths can resolve managed refs in some integrated consumers, but that does not solve the reported shell/custom document-reader workflow, and the open related PR only covers unsupported binary documents while preserving PDF URI behavior.

## What This Changes

The issue reports that non-image inbound channel attachments are exposed to agents as `media://inbound/...` refs that shell and custom document-reader tools cannot open.

## Best Possible Solution

Keep this as the canonical media prompt/tool-boundary issue, then choose a scoped fix that makes non-image attachments usable by shell/custom document readers without broadly leaking host paths; likely options are a guarded resolver/tool handoff or a narrowly accepted prompt-visible path policy with regression tests.

## Reproduction Assessment

Yes, source-level: current `main` still maps managed inbound paths to `media://inbound/<id>` in prompt notes and only image attachments are inlined into the current turn. I did not run a live Telegram bot reproduction.

## Solution Assessment

No to a blanket absolute-path prompt fix: current URI privacy behavior is intentional and built-in `read`/`pdf` consumers already resolve managed refs in some paths. The best fix needs a maintainer-approved resolver or scoped path-visibility contract.
