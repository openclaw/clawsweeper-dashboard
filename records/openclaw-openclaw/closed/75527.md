## Summary

Keep open. This is a draft member PR with the protected `maintainer` label, so conservative cleanup should not close it; source review also found a blocking gap where the new bounded chat-history path still full-loads parentId/tree transcripts, which are an active Gateway/Pi transcript format. The sessions/config caps look directionally useful and the PR body includes targeted/Testbox proof, but the tree-transcript path needs maintainer or author handling before merge.

## What This Changes

The PR bounds Gateway chat-history and sessions-list transcript reads, caps sessions-list title/last-message hydration and child-link lookup work, caps oversized extension config schemas, and adds focused tests plus changelog entries.

## Best Possible Solution

Keep the PR open and revise the chat-history tail reader so parentId/tree transcripts preserve active-branch semantics without materializing the whole transcript; then recheck the sessions-list and config-schema caps with the existing targeted Gateway/config tests and changed gate.

## Reproduction Assessment

Yes. A source-level reproduction is to request `chat.history` with a small limit against a large parentId/tree transcript; the proposed tail reader detects `parentId` and calls the existing unbounded `readSessionMessages` path before slicing.

## Solution Assessment

No. Tail reads are the right direction for append-only transcripts, but the maintainable fix also needs a bounded branch-aware path for current parentId transcripts instead of falling back to `readSessionMessages`/`SessionManager.open`.
