## What This Changes

The PR removes MS Teams legacy JSON imports from runtime stores, adds doctor migrations for conversations, polls, and SSO tokens, leaves pending-upload JSON unmigrated, and updates docs/tests.

## Best Possible Solution

Keep runtime free of legacy JSON reads, but make the doctor imports idempotent and prove failed-archive/rerun behavior for conversations, polls, and SSO tokens before merge.

## Reproduction Assessment

Source-reproducible: if archiveLegacySource leaves a legacy JSON file in place, a later doctor run will detect it again and register missing rows with registerIfAbsent. I did not execute the PR branch, but the path is visible in the proposed doctor migration code.

## Solution Assessment

No. Moving legacy reads out of runtime is the right layer, but the proposed doctor path needs idempotent migration tracking or atomic source retirement before it is the best fix.

Full review comments:

- **[P1] Make failed archive imports idempotent:** `extensions/msteams/doctor-contract-api.ts:491`
  - body: These doctor migrations register durable rows before calling `archiveLegacySource`, but that helper can leave the JSON file in place when `<file>.migrated` already exists or rename fails. Since the old runtime migration markers/content hashes are removed, the next `openclaw doctor --fix` will detect the same file again and `registerIfAbsent` can recreate an SSO token the user already removed from SQLite. Keep a per-source/content migration marker or make source retirement atomic before inserting durable rows.
