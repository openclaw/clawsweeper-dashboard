## Summary

Current main already contains the auth-store changes that prevent a live gateway from writing stale in-memory OAuth state back over fresher on-disk credentials, and the current Anthropic runtime path reads Claude CLI auth directly instead of depending on a copied persisted refresh token. The same implementation is present in release `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`).

## Evidence

- **Locked auth-store writes reload from disk:** `updateAuthProfileStoreWithLock` reloads the auth store from disk while holding the file lock, with an inline note that this avoids overwriting fresher CLI/config-auth writes with stale in-memory auth state during usage/cooldown updates.
  - file: [src/agents/auth-profiles/store.ts:135](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/agents/auth-profiles/store.ts#L135)
  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)
- **Saving rotates the runtime snapshot too:** `saveAuthProfileStore` writes the persisted store, updates the cached store, and refreshes the runtime auth snapshot for that agent, so later reads do not keep serving stale OAuth credentials after rotation.
  - file: [src/agents/auth-profiles/store.ts:375](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/agents/auth-profiles/store.ts#L375)
  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)
- **Regression coverage for rotated OAuth tokens:** The test `refreshes the runtime snapshot when a saved store rotates oauth tokens` verifies that after saving refreshed OAuth credentials, both `ensureAuthProfileStore(...)` and `auth-profiles.json` return the new access/refresh pair instead of the stale one.
  - file: [src/agents/auth-profiles.store.save.test.ts:75](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/agents/auth-profiles.store.save.test.ts#L75)
  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)
- **Current Anthropic runtime auth is synthetic from Claude CLI:** The current Anthropic plugin resolves Claude CLI runtime auth from `readClaudeCliCredentialsForRuntime()` and returns synthetic auth from the live CLI credential, rather than requiring a copied persisted refresh token as the runtime source.
  - file: [extensions/anthropic/register.runtime.ts:396](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/extensions/anthropic/register.runtime.ts#L396)
  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)
- **Current Anthropic auth surface no longer exposes direct OAuth setup:** The Anthropic provider contract now exposes `Claude CLI`, `setup-token`, and `API key` auth choices; there is no separate direct Anthropic OAuth login path matching the March 2026 report.
  - file: [extensions/anthropic/provider-contract-api.ts:12](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/extensions/anthropic/provider-contract-api.ts#L12)
  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)

## Risks / Open Questions

- I did not run a live gateway reproduction because this review was explicitly read-only and had to keep the checkout byte-for-byte clean.
- If someone can still reproduce stale overwrites on `v2026.4.22` or newer, it is more likely a new regression on a different auth path than the original March 2026 flow described here.
