## Summary

This report matches the already-active missing-index-metadata regression tracked in https://github.com/openclaw/openclaw/issues/90361, while its proposed provider-initialization explanation is contradicted by both v2026.6.5 and current source. The canonical issue already contains stronger concurrent-search evidence, a live mitigation, and the more plausible shared-manager reindex path, so keeping a second root-cause-specific report open would fragment the investigation.

## What This Changes

The issue reports that `memory_search` becomes unavailable after gateway startup because the manager allegedly calculates index identity before initializing its embedding provider.

## Best Possible Solution

Continue the investigation in https://github.com/openclaw/openclaw/issues/90361 and isolate reindex construction from the live manager database handle, or otherwise serialize identity reads, rather than eagerly initializing providers during manager creation.

## Reproduction Assessment

No for this issue's claimed provider-initialization mechanism: v2026.6.5 and current source already defer the unknown provider key and use the configured provider/model. The broader missing-metadata failure has strong reproduction evidence in the canonical issue.

## Solution Assessment

No; eager provider initialization adds startup dependency work but does not explain a genuinely absent metadata row or the shared temporary-database reindex race documented in the canonical issue.
