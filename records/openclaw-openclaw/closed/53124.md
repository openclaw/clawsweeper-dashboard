## Summary

Current `main` already sequences Telegram preview cleanup after queued reply delivery completes, and the same implementation is present in release `v2026.4.22`, so this issue is already fixed on shipped code.

## Evidence

- **Dispatcher waits for all queued deliveries before returning:** `withReplyDispatcher` always calls `dispatcher.markComplete()` and then awaits `dispatcher.waitForIdle()` in its `finally`, so upstream callers do not exit while reply deliveries are still in flight.
  - file: [src/auto-reply/dispatch-dispatcher.ts:3](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/auto-reply/dispatch-dispatcher.ts#L3)

  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Reply dispatcher tracks and serializes pending sends:** `createReplyDispatcher` chains each delivery onto `sendChain`, increments `pending`, awaits `options.deliver(...)`, and exposes `waitForIdle()` over that chain. This is the mechanism that prevents fire-and-forget reply callbacks from racing cleanup.
  - file: [src/auto-reply/reply/reply-dispatcher.ts:151](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/auto-reply/reply/reply-dispatcher.ts#L151)
