## Summary

Current main and the shipped v2026.6.6 release handle the reported WebChat 4000-character cap by removing the Control UI maxChars override, adding a full-message reader for truncated history rows, and retiring the old persisted WebChat config surface.

## What This Changes

The issue reports that finalized WebChat assistant messages truncate to 4000 characters despite the configured chat history cap.

## Best Possible Solution

Keep the shipped bounded chat.history plus full-message-reader design; if current releases still lose access to full content, open a new report against the current WebChat behavior rather than the retired gateway.webchat config path.

## Reproduction Assessment

No, the exact current-main 4000-character WebChat history override does not reproduce by source inspection: Control UI no longer sends maxChars in chat.history. I did not run a live WebChat scenario in this read-only review; the conclusion is based on current source, tests, maintainer discussion, and release provenance.

## Solution Assessment

Yes, for the reported 4000-cap regression: the best current solution is the maintainer-approved bounded chat.history path with chat.message.get for full truncated rows, plus removal of the retired gateway.webchat config surface. A request for every long reply to remain fully inline would be a new product-policy change, not the same fixed regression.
