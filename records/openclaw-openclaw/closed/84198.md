## Summary

The report is well described, but the remaining behavior is owned by Tencent's external WeChat plugin and the Tencent iLink send surface, while current OpenClaw core already exposes and forwards the generic voice-note intent to channel plugins.

## What This Changes

The issue asks OpenClaw to send WeChat TTS replies from the external openclaw-weixin channel as native WeChat voice messages instead of WAV file attachments.

## Best Possible Solution

Keep OpenClaw core channel-agnostic and track native WeChat voice-bubble support in Tencent/openclaw-weixin, reopening an OpenClaw issue only if that work exposes a missing plugin API.

## Reproduction Assessment

No high-confidence in-repo reproduction path exists because the WeChat runtime is external. Source and external tracker evidence still corroborate the observed behavior: OpenClaw passes voice-note hints to plugins, while Tencent/openclaw-weixin currently falls back to file attachments because iLink drops bot VOICE messages.

## Solution Assessment

No for an OpenClaw core change. The maintainable solution is for the external WeChat plugin/platform to implement native voice sending if the Tencent iLink behavior allows it, while core keeps the existing generic audioAsVoice and tts.voice seams.
