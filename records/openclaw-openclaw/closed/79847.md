## Summary

This should stay open: current main still passes the qmd-scoped XDG environment into mcporter, and the overlapping fix is an open PR that needs normal maintainer review/landing rather than issue cleanup.

## What This Changes

qmd-manager should stop leaking qmd-scoped XDG_CONFIG_HOME/XDG_CACHE_HOME into mcporter child processes so mcporter 0.10+ can find its configured qmd server.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/79745 or an equivalent qmd-manager fix that keeps qmd-only XDG state out of mcporter process env while preserving configured mcporter servers, then close this issue as implemented.

## Reproduction Assessment

Yes, this is high-confidence source-reproducible: current main sends qmd-scoped XDG variables to runMcporter, and mcporter 0.10.2 uses XDG_CONFIG_HOME for config discovery when no explicit config is passed. I did not execute the live daemon scenario in this read-only review.

## Solution Assessment

Yes, the narrow maintainable direction is to split qmd-only environment from mcporter process environment or use the active PR's explicit per-agent mcporter config plus clean mcporter env. The existing open PR is the current best vehicle rather than a duplicate repair branch.
