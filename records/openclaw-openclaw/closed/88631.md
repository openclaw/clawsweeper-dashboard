## Summary

Keep open: current main still routes Feishu final text replies through the streaming CardKit path when streaming is enabled, and that path does not apply the normal text chunking limit or a long-card fallback before updating/closing the card.

## What This Changes

The issue reports silent truncation of long Feishu assistant replies when default streaming card mode is used instead of chunked text delivery.

## Best Possible Solution

Keep the issue open and add a Feishu-owned streaming-card boundary that preserves complete reply text, preferably by splitting or falling back to chunked text before any CardKit markdown limit can drop content.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only review. Source inspection does show the default Feishu streaming path can send unbounded accumulated reply text to CardKit without applying the existing text chunk limit.

## Solution Assessment

Yes, the best direction is a narrow Feishu delivery fix, not a new config mode. The implementation should preserve complete text while keeping the current default streaming UX when replies are within the safe card size.
