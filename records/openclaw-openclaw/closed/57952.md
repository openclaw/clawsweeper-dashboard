## Summary

Current main solves the reported gateway-wide serialization bottleneck without adding a broad `CommandLane.Nested` concurrency knob. Cron and hook-dispatched isolated agent work now uses a dedicated `cron-nested` lane governed by `cron.maxConcurrentRuns`, and `sessions_send` work is scoped to per-target-session nested lanes so unrelated sessions no longer share one global nested lane.

## Best Possible Solution

Close as implemented and keep the current narrower routing design: `cron.maxConcurrentRuns` governs cron/hook isolated execution through `cron-nested`, while `sessions_send` uses per-target-session nested lanes. A future webhook lane-override request should be tracked separately only if maintainers want that explicit product surface.
