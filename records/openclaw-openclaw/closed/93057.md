## Summary

Keep open: the PR targets the right process-runtime bug, but the patch flips the helper default without preserving group cleanup for existing detached callers, so it can leave descendants running in MCP, LSP, shell, and harness timeout paths.

## What This Changes

The PR changes Unix process-tree termination so omitted `detached` uses direct-pid kill, keeps group-kill only for explicit `detached: true`, and updates kill-tree unit tests.

## Best Possible Solution

Land the safe non-detached behavior only after the PR either threads actual detachment through existing detached callers or safely infers process-group leadership, with tests for both non-detached gateway safety and detached descendant cleanup.

## Reproduction Assessment

Yes at source level: current main has POSIX detached callers that omit `detached`, and the PR changes those paths from group signaling to direct-pid signaling. I did not run the live macOS launchd scenario in this read-only review.

## Solution Assessment

No: the shared helper is the right place to make non-detached kills safe, but this diff is too one-sided because it drops existing detached group cleanup unless every caller opts back in. A caller audit or safe group-leader check is the narrower maintainable fix.
