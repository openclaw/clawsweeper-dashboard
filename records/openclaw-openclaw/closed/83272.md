## Label Justifications

- `P1`: A wedged Telegram isolated-ingress handler can stop real inbound Telegram delivery for an account or lane until manual recovery.
- `impact:message-loss`: The issue is about spooled Telegram updates not being dispatched after the offset has advanced, so Telegram will not redeliver them.
- `impact:session-state`: The stuck active handler and spool claim state can leave the channel/session view stale until the handler settles or the gateway is restarted.

## Summary

Keep open: current main improves Telegram isolated-ingress lane concurrency and stale-backlog health, but a never-settling spooled `handleUpdate` still remains active without a deadline or recovery path and can keep later same-lane inbound updates blocked after Telegram will not redeliver them.

## What This Changes

The issue reports that Telegram isolated ingress can advance the update offset after spooling an update, then hang indefinitely before dispatching that spooled update into the session.

## Best Possible Solution

Add an owner-reviewed bounded recovery path for stuck spooled Telegram handlers that preserves same-lane ordering, avoids duplicate dispatch, and keeps offset/spool durability semantics explicit.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: current tests can model a `handleUpdate` promise that never settles, and current main keeps that lane active with only stale-health reporting. I did not run a live Telegram setup in this read-only pass.

## Solution Assessment

No, current main is only a partial fix: lane concurrency and stale-health diagnostics are useful, but they do not bound or recover a never-settling spooled handler. The best next step is an owner-approved recovery policy before implementation.
