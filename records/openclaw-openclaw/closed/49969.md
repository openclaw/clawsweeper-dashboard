## Summary

Close: current main already covers this well enough. The stale-socket restart path now only applies to channels with provider-proven transport activity, which WhatsApp does not report here, and WhatsApp status now surfaces disconnected/stale/reconnecting web-listener state explicitly instead of looking generically connected.

## Evidence

- **Stale-socket recovery now requires transport-activity tracking:** `evaluateChannelHealth()` only returns `stale-socket` when `connected === true` and `lastTransportActivityAt != null`; otherwise the channel stays healthy instead of being restarted for idle app-event timestamps.

  - file: [src/gateway/channel-health-policy.test.ts:214](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/src/gateway/channel-health-policy.test.ts#L214)
