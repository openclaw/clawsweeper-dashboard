## Summary

Keep open. The reporter’s sanitized config now gives a source-reproducible current-main failure: Discord per-channel `allow` is still documented and doctor-migrated, but bundled-channel validation checks the strict Discord schema before that alias is normalized.

## What This Changes

The issue asks OpenClaw to preserve the 2026.4.23 Discord per-guild channel config shape using `allow` plus `requireMention` when upgrading to 2026.5.12.

## Best Possible Solution

Choose one compatibility boundary and make it consistent: either normalize `allow` to `enabled` before bundled Discord schema validation with regression coverage, or update docs/startup diagnostics so affected upgrades clearly run `openclaw doctor --fix` before gateway start.

## Reproduction Assessment

Yes. The source path is high-confidence: the supplied config contains `channels.discord.guilds.*.channels.*.allow`, current main validates Discord channel config against a strict schema that lacks `allow`, and doctor migration runs outside that startup validation path.

## Solution Assessment

Unclear. The closed related PR's pre-validation normalization is a narrow functional fix, but adopting it would intentionally move a Discord legacy alias into startup validation despite the repo's doctor-only legacy repair policy.
