## Summary

Keep this open. Current main still has only a phase-based assistant visibility contract, so `audience` metadata in assistant text signatures is ignored at shared extraction, UI projection, provider replay, Gateway schema, and embedded delivery boundaries; the related issues cover adjacent sanitizer/commentary/pick-one problems but do not supersede this audience-boundary contract gap.

## What This Changes

Add a public/internal assistant audience contract across text signatures, provider replay, schemas, visible extraction, and delivery so internal assistant text and exact `NO_REPLY` are not selected for user-visible output.

## Best Possible Solution

Define one versioned `public | internal` assistant audience contract, propagate it through signatures/OpenResponses/replay/history/UI, enforce public-only selection at shared visible delivery boundaries, and keep legacy phase-only fallback narrowly scoped for old transcripts.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a text block signed as `{ "v": 1, "id": "x", "phase": "final_answer", "audience": "internal" }` has its audience ignored and remains eligible for final-answer extraction.

## Solution Assessment

Yes for the direction, but it should be solved as a maintainer-owned shared contract rather than per-channel filtering. The maintainable path is a versioned audience field enforced at shared extraction and delivery boundaries with explicit legacy fallback rules.
