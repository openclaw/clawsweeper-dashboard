## Label Justifications

## Summary

Keep open: current `main` still has the LINE trailing-slash bucket mismatch, and the PR fixes it narrowly inside the LINE plugin using the already exported webhook path canonicalizer. I found no blocking correctness or security findings in the read-only review.

## What This Changes

The PR canonicalizes LINE webhook paths so a configured trailing slash registers and dispatches through the same route bucket as the no-trailing-slash path.

## Best Possible Solution

Merge the narrow LINE plugin fix after normal CI and maintainer gates, keeping the shared webhook SDK canonicalization as the single route-key rule.

## Reproduction Assessment

Yes from source: configure LINE with `webhookPath: "/line/webhook/"`; current main registers the bucket as `/line/webhook` but the route handler reads `/line/webhook/`, leaving no target for signature validation.

## Solution Assessment

Yes. Canonicalizing the plugin-local path before both registration and route lookup is narrower than refactoring LINE onto the full shared request pipeline, and it reuses the exported SDK helper instead of adding LINE-specific path logic.
