## Label Justifications

- `P1`: This fixes a current broken cron workflow where one-shot main-session scheduled jobs can leave queued work recorded as skipped and fail delete-after-run cleanup.

## Summary

Keep open: current main still records this path as `skipped: disabled` after the cron system event has already been queued, while this PR provides a narrow status fix, a focused regression test, and now sufficient real temp-gateway proof. I found no blocking correctness or security findings.

## What This Changes

The PR changes main-session wake-now cron runs to report success when the immediate heartbeat is skipped because heartbeats are disabled, and adds a regression test for one-shot `at` job cleanup.

## Best Possible Solution

Land the narrow cron status fix with its regression test after normal merge gates, and let the linked issue close when this PR merges.

## Reproduction Assessment

Yes. Source inspection shows current main enqueues the system event, receives `skipped: disabled` from the disabled heartbeat path, and then avoids `deleteAfterRun` cleanup because only `ok` results delete one-shot jobs; I did not run the repro in this read-only review.

## Solution Assessment

Yes. Treating the disabled immediate heartbeat as a successful queued handoff after the system event is already enqueued is narrower than changing cron storage or heartbeat configuration, and it preserves the existing busy-retry and other skip behavior.
