## Summary

Close as implemented on current main. The WebSocket handshake now returns a retryable startup-unavailable response while startup sidecars are pending, and sidecars only clear that pending state after plugin/channel startup completes, so pre-ready WS clients cannot send agent RPCs into an empty tool registry. This fix is present on main but is not contained in the latest release tag checked here.

## What This Changes

Prevent WS-origin agent runs from being accepted before plugin sidecars and tool registration are ready, avoiding misleading empty-tool-registry failures during gateway startup.

## Best Possible Solution

Keep the current main startup-unavailable handshake behavior and ship it in the next release; a buffered replay UX can remain a separate enhancement if maintainers want better latency than client retry.

## Reproduction Assessment

Yes. The report gives a concrete pre-ready startup window, and current main has a focused WS startup test that reproduces the pending-sidecars condition and verifies retryable rejection before any post-connect request can run.

## Solution Assessment

Yes. Gating the WebSocket connect handshake until sidecars are ready is the narrowest maintainable fix for the reported WS pre-ready window because it prevents agent dispatch from reaching an unpopulated registry without adding a new queue/replay subsystem.
