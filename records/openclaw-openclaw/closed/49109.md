Keep this PR open, but it should not merge as-is: the cron helper fix is still relevant on current main, while the PR’s gateway retry is method-agnostic and the branch carries unrelated security-sensitive SSRF/network-boundary changes.

## What This Changes

The PR changes cron CLI helper status checks, adds CLI gateway transport retry tests/logic, and currently includes broad unrelated status fast-path, plugin registry/generated baseline, channel/plugin test, build-entrypoint, and SSRF changes.

## Best Possible Solution

Split the work into a small cron helper patch plus #54475 or an equivalent structured pre-hello gateway fix, and remove unrelated SSRF/status/plugin/channel changes from this PR.

## Reproduction Assessment

Yes. Static current-main inspection reproduces the cron JSON/helper path, and the gateway close path is clear from `executeGatewayRequestWithScopes`; the flaky transport case itself is covered more directly by related #54475’s described unit reproduction.

## Solution Assessment

No. The cron helper direction is narrow, but the blanket CLI retry and unrelated security-sensitive diff are not the safest maintainable solution; a structured pre-hello gateway fix plus a small cron probe patch is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Do not retry mutating gateway RPCs:** `src/cli/gateway-rpc.ts:50-68`
  - body: The retry loop retries any method when `callGateway` rejects with matching close/timeout text. Because `callGateway` sends the RPC after `hello-ok`, a close before the response can still occur after the gateway processed a mutating command like `cron.add`, so this can duplicate jobs or repeat side effects. Keep retry scoped to a structured pre-request/pre-hello signal or use the focused gateway-layer fix instead.
  - confidence: 0.82
