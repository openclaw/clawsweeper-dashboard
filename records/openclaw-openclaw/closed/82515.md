## Summary

Keep open: current main and the latest shipped Matrix plugin still let matrix-js-sdk request MSC4222 `state_after` without a Matrix compatibility guard, and the report includes concrete E2EE evidence that outbound encryption can remain unconfigured on affected homeservers.

## What This Changes

The issue reports that Matrix E2EE outbound replies can fail when incomplete MSC4222 `state_after` sync data omits `m.room.encryption`.

## Best Possible Solution

Add a Matrix-plugin compatibility guard that prevents incomplete MSC4222 `state_after` sync data from bypassing `m.room.encryption` processing, with focused regression coverage for `/sync` request shaping and encrypted outbound readiness.

## Reproduction Assessment

No, not with high confidence from this review alone: reproducing requires a Matrix homeserver that returns incomplete MSC4222 `state_after`. The reporter's logs plus matrix-js-sdk source make the failure path source-reproducible enough to keep open and queue a focused fix.

## Solution Assessment

Yes, the best direction is a narrow Matrix-plugin compatibility mitigation around SDK sync request handling, not a core change. It should preserve the existing Matrix transport safety wrappers and add a regression test for the reported homeserver response shape.
