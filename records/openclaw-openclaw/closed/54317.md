## Summary

Current OpenClaw already supports explicit WebSocket session routing by passing `sessionKey` in `chat.send`/`chat.history`, not in the WS handshake, and it can resolve an existing historical session from `sessionId` via `sessions.resolve`. There is no separate transcript-only recovery path when the gateway store has no matching session entry.

## Evidence

- **WS chat RPCs already take `sessionKey`:** The WebSocket protocol schema requires `sessionKey` on both `chat.history` and `chat.send`, which is the explicit routing mechanism for WS clients.
  - file: [src/gateway/protocol/schema/logs-chat.ts:26](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/gateway/protocol/schema/logs-chat.ts#L26)
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **`chat.send` and `chat.history` route through the requested session key:** `chat.history` loads the requested `sessionKey` and reads that session's transcript; `chat.send` takes `p.sessionKey` and canonicalizes it through `loadSessionEntry(...)` before running the turn.
  - file: [src/gateway/server-methods/chat.ts:1886](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/gateway/server-methods/chat.ts#L1886)
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **WS handshake has no session selector:** The `connect` schema includes protocol/client/auth fields only and does not define `sessionKey` or `sessionId`, so routing is not done during the WS handshake.
  - file: [src/gateway/protocol/schema/frames.ts:20](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/gateway/protocol/schema/frames.ts#L20)
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Historical sessions can be resolved from `sessionId`:** `sessions.resolve` accepts either `key`, `sessionId`, or `label`; when `sessionId` is provided it searches the session index and returns the matching session key.
  - file: [src/gateway/sessions-resolve.ts:73](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/gateway/sessions-resolve.ts#L73)
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Gateway-owned store is authoritative, not bare transcript files:** Docs state all session state is owned by the gateway, with `sessions.json` as the store and transcript files alongside it. `loadSessionEntry(...)` resolves from the session store, so a transcript file by itself is not a separate resumable WS routing surface.
  - file: [docs/concepts/session.md:76](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/docs/concepts/session.md#L76)
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Behavior was already in the latest release:** The release commit for `v2026.4.22` already contains the same `chat.send` `sessionKey`, `sessions.resolve(sessionId)`, handshake schema, and `x-openclaw-session-key` docs.
  - file: [src/gateway/protocol/schema/logs-chat.ts:35](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/gateway/protocol/schema/logs-chat.ts#L35)
