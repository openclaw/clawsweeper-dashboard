## Summary

Keep open: current main and the latest release still derive the cron isolated-agent session context snapshot from accumulated run usage, while sibling session writers already use the last-call snapshot to avoid this exact overcount.

## What This Changes

The issue asks for cron isolated-agent finalization to persist the final API call's usage as `SessionEntry.totalTokens` instead of accumulated loop usage.

## Best Possible Solution

Update `finalizeCronRun` to use `agentMeta.lastCallUsage` or `promptTokens` for the context snapshot while keeping accumulated usage for billing, telemetry, and input/output counters.

## Reproduction Assessment

Yes, there is a high-confidence source reproduction path: feed a cron run result with accumulated `usage.input` larger than `lastCallUsage.input`, and current `finalizeCronRun` passes only the accumulated usage into `deriveSessionTotalTokens`. I did not run the supplied Vitest in this read-only pass.

## Solution Assessment

Yes, the suggested fix is the best narrow shape: mirror the existing user-turn last-call snapshot pattern in the cron writer and leave cumulative usage accounting intact for cost and telemetry.
