## Summary

Keep this draft PR open. Current main still does not implement the proposed per-file-signature cache inside `loadPluginManifest`; the existing reuse is metadata-snapshot, registry TTL, or installed-index fallback caching. The PR is a focused implementation candidate for the still-open CPU regression in #73647, and its diff is limited to manifest loading plus a regression test rather than an obsolete or unrelated change.

## Best Possible Solution

Keep the draft PR open and review it as the focused manifest-load caching candidate for #73647. The best path is to verify the CPU root cause against current main, confirm cache key and invalidation behavior for unchanged and changed manifest files, keep the boundary-file checks intact, add any missing regression coverage, then run the targeted manifest loader test and the appropriate changed gate/Testbox validation before merge.

## Risks / Open Questions

- The proposed cache invalidation needs maintainer review on the reported Docker/Synology filesystem; `dev`, `ino`, `size`, and `mtimeMs` is plausible, but the PR currently demonstrates unchanged-file reuse more directly than changed-file invalidation.
- The PR should preserve boundary-file safety checks on every call. The patch opens through `openBoundaryFileSync` before cache lookup, which looks directionally correct, but this should remain part of normal review.
