## Summary

Close as implemented: current main and the v2026.5.12 release now rewrite the reported generic provider internal-error/request-ID shape into chat-safe copy, and the existing pre-output transient retry path is still present.

## What This Changes

The issue asks Feishu-visible provider `server_error` payloads to become generic user-facing failure text while keeping raw provider request IDs in logs and retrying only before visible output is delivered.

## Best Possible Solution

Keep the shared formatter/sanitizer implementation from v2026.5.12, since it fixes the Feishu-visible payload through the common reply path without adding Feishu-specific filtering.

## Reproduction Assessment

No on current main: the reported provider text now matches `isGenericProviderInternalError` and is rewritten before raw API formatting or Feishu delivery. The old leak path was source-reproducible and is covered by the merged PR's formatter/payload proof.

## Solution Assessment

Yes. The shared formatter, sanitizer, and failover seam is the narrow maintainable fix here; Feishu sends the prepared reply text, so a Feishu-only fallback is no longer needed for the reported payload shape.
