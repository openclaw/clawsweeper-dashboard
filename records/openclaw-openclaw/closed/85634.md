Full review comments:

- **[P1] Preserve cleanup-root safety guards:** `src/commands/doctor/shared/plugin-dependency-cleanup.ts:117-121`
  - body: This branch replaces current main's guarded target collection with an older unguarded collector. Current main rejects parent segments, arbitrary explicit stage roots, symlinked extension roots, and cleanup roots that resolve outside OpenClaw-owned directories before any recursive removal; the PR head collects targets without those checks and later removes them, so an unsafe stage path or symlink can delete data outside the cleanup area. Rebase this file and keep the existing safety filters/tests while adding any parallelism.
  - confidence: 0.94
- **[P1] Do not pool one-shot gateway clients:** `src/gateway/call.ts:681-683`
  - body: `acquirePooledClient` can return an existing `GatewayClient`, but that client still has the `onHelloOk` closure, URL, TLS, device identity, and scopes from the first request. Calling `startGatewayClientWhenEventLoopReady` on that reused object starts the old client again rather than installing this request's callback, so later calls can time out or run with the wrong connection/auth context. Keep per-call clients or redesign the client API to support explicit connected request reuse.
  - confidence: 0.93
- **[P2] Keep stale OAuth cleanup after sidecar migration:** `src/commands/doctor/repair-sequencing.ts:137-150`
  - body: `repairStaleOAuthProfileShadows` intentionally skips profiles that still contain a legacy `oauthRef`, while `maybeRepairLegacyOAuthSidecarProfiles` removes that ref when it migrates credentials inline. Running both in `Promise.all` means the shadow cleanup can scan before migration and miss the profile, requiring another doctor run to finish the repair. Keep the OAuth sidecar migration sequenced before stale shadow cleanup and parallelize only independent work.
  - confidence: 0.86
