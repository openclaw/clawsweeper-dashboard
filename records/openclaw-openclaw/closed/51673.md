## Summary

Current main already fixes the observable vLLM/OpenAI-compatible stale/zero token-accounting problem through the shipped streaming-usage and transcript fallback path, so this older consolidation PR is superseded by main rather than mergeable as-is.

## Best Possible Solution

Keep the shipped main implementation: local OpenAI-compatible providers such as vLLM should use streaming usage accounting plus transcript-derived display fallback. Do not merge this stale consolidation branch; any remaining edge case should be filed as a focused regression against the current streaming-usage/session-display path.
