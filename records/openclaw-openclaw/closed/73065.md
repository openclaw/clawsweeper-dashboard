## Summary

Current main already fixes the reported Matrix self-verification completion path. The merged Matrix SAS follow-up makes the shared verification manager wait for the rust-crypto verifier promise and cross-sign the own device before the CLI reports completion, which directly covers the report’s “other client succeeded, OpenClaw failed afterward” failure mode.

## What This Changes

The issue reports that `openclaw matrix verify self` fails after Matrix emoji verification with a missing signing-key error even though the other Matrix client reports success.

## Best Possible Solution

Keep the current manager-level SAS completion behavior and make sure it ships in the next release. If the same error is reported on a build after `86956f71e62a6561b4a6412dec26dde992960dbf`, ask for `openclaw matrix verify self --verbose`, `verify status --verbose`, account type, and homeserver details.

## Reproduction Assessment

Yes, for the addressed failure class: the report gives concrete `openclaw matrix verify self` steps, and the test suite now models the race where SAS confirmation returns before the verifier promise settles. I did not run a live Matrix verification in this read-only review.

## Solution Assessment

Yes. Fixing the shared Matrix verification manager is the narrowest maintainable path because both `verify self` and lower-level SAS commands flow through it, avoiding a CLI-only workaround or core changes.
