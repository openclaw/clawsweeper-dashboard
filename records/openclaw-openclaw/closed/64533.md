## Summary

Current `main` and the shipped `v2026.4.23` release already use the newer reply-media staging path that allows OpenClaw-managed temp media via agent-scoped outbound media access. The reported `normalizeReplyMediaPaths` regression path from `v2026.4.9` is no longer present in current code or the latest release.

## Evidence

- **Reply media now stages local paths instead of blocking absolute host paths in the old helper flow:** `createReplyMediaPathNormalizer()` resolves local media through `resolveOutboundAttachmentFromUrl()` with agent-scoped media access, and when sandbox mode is off it falls through to `persistLocalReplyMedia(media)` rather than a sandbox-only tmp allowlist branch.
  - file: [src/auto-reply/reply/reply-media-paths.ts:127](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/src/auto-reply/reply/reply-media-paths.ts#L127)
  - command: `nl -ba src/auto-reply/reply/reply-media-paths.ts | sed -n '100,245p'`
  - sha: [fd65caf4b0ba](https://github.com/openclaw/openclaw/commit/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc)
- **Agent-scoped outbound media access widens local roots for concrete media sources:** `resolveAgentScopedOutboundMediaAccess()` uses `getAgentScopedMediaLocalRootsForSources()` when host media reads are allowed, so concrete local files are staged under explicit allowed roots rather than rejected by a fixed absolute-path allowlist.
  - file: [src/media/read-capability.ts:83](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/src/media/read-capability.ts#L83)
