## Summary

Keep open: current main and the latest release have the full opt-out path via `messages.suppressToolErrors`, but the requested `final`/normal-chat delivery policy remains unresolved and is still a product/safety decision supported by multiple operator comments.

## What This Changes

Request a configuration policy for user-visible transient tool-error warnings, especially an `all | final | off` style mode for recovered tool failures after a useful final reply.

## Best Possible Solution

Define one canonical messages-level tool-error delivery policy that preserves the conservative default, keeps or migrates the existing full-suppression switch, and tests any final-only mode across core payload assembly plus affected channels.

## Reproduction Assessment

Yes, at source level. Current payload tests construct a success-looking final reply plus a failed mutating tool and assert that a separate warning remains visible; I did not run tests because this review is read-only.

## Solution Assessment

No, not exactly as proposed. The full `off` outcome already exists through `messages.suppressToolErrors`; the remaining best fix is a maintainer-approved canonical messages-level policy with explicit safety semantics for final-only suppression.
