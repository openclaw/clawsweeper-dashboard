## Summary

Keep this issue open. Current main still has the exact tool-call gate the report identifies: the DeepSeek V4 payload patch only backfills `reasoning_content` on assistant messages whose outgoing OpenAI-compatible payload already has a `tool_calls` array. Existing tests and changelog evidence cover the older #71372 tool-call replay fix, but I did not find coverage for a final/ordinary assistant message without `tool_calls`. DeepSeek's thinking-mode docs also distinguish plain no-tool turns from tool-call turns and say tool-call turns must keep returning `reasoning_content`, including the final assistant message in the documented tool-call flow. ([api-docs.deepseek.com](https://api-docs.deepseek.com/zh-cn/guides/thinking_mode))

## Best Possible Solution

Keep the issue open and fix it in the DeepSeek provider/shared OpenAI-compatible thinking wrapper. Add a focused regression test for a DeepSeek V4 thinking replay that includes an assistant message without `tool_calls` after a tool-using turn, then adjust the payload patch to preserve or synthesize `reasoning_content` for that replay shape without weakening the disabled-thinking path.
