The issue reports that Microsoft Foundry DeepSeek V4 alias provider ids still receive unsupported `thinking` request fields even though the canonical `microsoft-foundry` provider was fixed.

## Best Possible Solution

Ship a narrow provider-equivalence fix so Foundry alias/configured provider ids skip top-level `thinking` like canonical `microsoft-foundry`, while unrelated OpenAI-compatible DeepSeek V4 proxies keep the wrapper.

## Solution Assessment

No current fix is present. The maintainable fix is to recognize Foundry-equivalent configured provider ids or honor the resolved compat contract before applying the generic DeepSeek V4 fallback.
