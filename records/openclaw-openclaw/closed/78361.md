## Summary

Current main and the latest release already solve the central user problem with the existing provider timeout knob: explicit `models.providers.<id>.timeoutSeconds` now raises the model idle/stream watchdog above the implicit ~120s cap for cloud and self-hosted providers.

## What This Changes

Request to add user-configurable model stream idle-timeout knobs for global, per-agent, and cron runs so slow or silent provider streams can exceed the default ~120s watchdog.

## Best Possible Solution

Keep `models.providers.<id>.timeoutSeconds` as the canonical supported knob for slow or silently buffered model streams; discuss any future per-agent or per-cron idle-only granularity as a separate product/config-surface decision.

## Reproduction Assessment

No high-confidence current-main failure remains. Source and regression tests show the supported provider timeout path now raises the idle watchdog above the old ~120s ceiling.

## Solution Assessment

Yes for the central problem. The existing provider timeout knob is the narrower maintainable solution because it avoids adding global, per-agent, and per-cron idle-only config surfaces while still giving Gemini-preview and slow-provider users an escape hatch.
