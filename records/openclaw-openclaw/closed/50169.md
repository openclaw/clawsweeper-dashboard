## Summary

Current main already covers this report well enough: `openclaw cron list/add` are RPC-only on the current CLI/server path, and the Windows Scheduled Task restart path now explicitly cleans lingering gateway listeners, waits for port release before `/Run`, and has regression coverage for the stale-listener restart race that matches the reported `gateway already running ... lock timeout` failure mode.

## Evidence

- **Cron CLI path is RPC-only:** `openclaw cron status/list/add` call `cron.status`, `cron.list`, and `cron.add` over Gateway RPC; there is no restart call in these command handlers.
  - file: [src/cli/cron-cli/register.cron-add.ts:23](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/src/cli/cron-cli/register.cron-add.ts#L23)
  - command: `nl -ba src/cli/cron-cli/register.cron-add.ts | sed -n '1,120p'`
- **Gateway cron handlers do not request restart:** Current Gateway handlers for `cron.list`, `cron.status`, and `cron.add` only validate params, list status/jobs, and add the job via `context.cron.add(...)`; they do not schedule or trigger a restart.
  - file: [src/gateway/server-methods/cron.ts:137](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/src/gateway/server-methods/cron.ts#L137)
  - command: `nl -ba src/gateway/server-methods/cron.ts | sed -n '130,260p'`
- **Windows Scheduled Task restart now drains lingering listeners before relaunch:** `restartScheduledTask` ends the task, resolves the gateway port, terminates lingering listener PIDs, waits for port release, force-kills remaining busy listeners if needed, and only then calls `/Run` again.
  - file: [src/daemon/schtasks.ts:968](https://github.com/openclaw/openclaw/blob/512d3d22/src/daemon/schtasks.ts#L968)
  - command: `nl -ba src/daemon/schtasks.ts | sed -n '968,1007p'`
  - sha: [512d3d22](https://github.com/openclaw/openclaw/commit/512d3d22)
- **Regression tests cover the stale-listener restart race:** Tests assert that Scheduled Task stop/restart cleanup kills lingering verified gateway listeners and waits for port release before `/Run`, which is the reported lock-timeout symptom class.
  - file: [src/daemon/schtasks.stop.test.ts:110](https://github.com/openclaw/openclaw/blob/512d3d22/src/daemon/schtasks.stop.test.ts#L110)
  - command: `sed -n '100,220p' src/daemon/schtasks.stop.test.ts`
  - sha: [512d3d22](https://github.com/openclaw/openclaw/commit/512d3d22)
- **Windows self-restart path now cleans stale gateway processes before supervisor relaunch:** `triggerOpenClawRestart()` calls `cleanStaleGatewayProcessesSync()` before the Windows Scheduled Task relaunch path, preventing stale listener races during self-restart.
  - file: [src/infra/restart.ts:353](https://github.com/openclaw/openclaw/blob/512d3d22/src/infra/restart.ts#L353)
  - command: `git blame -L 353,392 -- src/infra/restart.ts`
  - sha: [512d3d22](https://github.com/openclaw/openclaw/commit/512d3d22)
- **Detached Scheduled Task handoff and fallback are implemented and tested:** Windows self-restart now uses a detached helper script that retries `schtasks /Run` and falls back to the installed startup script when needed; tests cover the helper contents and fallback behavior.
  - file: [src/infra/windows-task-restart.ts:24](https://github.com/openclaw/openclaw/blob/512d3d22/src/infra/windows-task-restart.ts#L24)
  - command: `git blame -L 1,140 -- src/infra/windows-task-restart.ts`
