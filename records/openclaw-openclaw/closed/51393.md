## Summary

Keep open: current main and the latest release still parse managed browser evaluate input as a parenthesized function/expression, so declaration statement bodies such as `var`, `let`, and `const` remain unsupported. The related broad browser-evaluate discussions do not fully cover this narrower function-body ergonomics gap, and the FSB comment is useful prior art for a focused fix.

## What This Changes

Request browser `act`/CLI evaluate to accept JavaScript statement bodies, including `var`/`let`/`const`, instead of only function or expression input.

## Best Possible Solution

Add a single browser-plugin evaluate input normalizer that preserves existing function/expression behavior while accepting statement/function-body snippets, then cover managed and supported existing-session paths with docs and focused tests.

## Reproduction Assessment

Yes at source level: current main wraps evaluate input in `eval("(" + fnBody + ")")`, and the read-only probe reproduces the reported `var` syntax error. I did not run a full browser server scenario because this review is read-only.

## Solution Assessment

Yes: the narrowest maintainable fix is inside the browser plugin evaluate normalization/execution path, not a new command or core API. The fix should preserve the current function/expression contract and add statement-body support with tests and docs.
