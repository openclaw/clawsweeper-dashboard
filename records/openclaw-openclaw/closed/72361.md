## Label Justifications

- `P0`: The PR targets an access-control boundary bypass for persistent allowlist writes that affect future sender authorization.
- `merge-risk: 🚨 security-boundary`: The current patch can leave a disabled command-origin channel able to mutate another channel's pairing allowlist store.

## Summary

Keep open: this is a valid security bugfix PR, but the current patch is not merge-ready because its new store-only guard authorizes the selected allowlist target as the origin and the PR still lacks after-fix real behavior proof.

## What This Changes

The PR adds a `configWrites` denial check and focused tests for store-targeted `/allowlist add|remove --store` pairing-store mutations.

## Best Possible Solution

Authorize the actual command origin separately from the selected allowlist store target, add a disabled-origin cross-channel regression, and land only with redacted real channel/store proof.

## Reproduction Assessment

Yes, by source inspection. Current main skips the only `configWrites` guard on the store-only path, and the PR still leaves a source-reproducible disabled-origin cross-channel case because it uses the selected target as the origin.

## Solution Assessment

No. Reusing the existing `configWrites` boundary is the right direction, but this patch should preserve the true command origin separately from the selected store target before merge.
