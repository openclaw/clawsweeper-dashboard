## Summary

Close as mostly implemented: current main already solves the linked Telegram media-access problem by downloading audio/document media through the guarded media path and exposing local media paths, shipped in v2026.4.24; the remaining exact `file_id` placeholder plus `download-file` action is a stale, conflicting, security-sensitive follow-up rather than a safe branch to merge.

## What This Changes

The PR adds Telegram media-placeholder `file_id` attributes, a Telegram `download-file` message action, and helper/test updates for media and audio placeholder handling.

## Best Possible Solution

Keep the shipped guarded auto-download and media-path behavior; any future exact `download-file` UX should be redesigned on the current Telegram plugin surface using the guarded media downloader.

## Reproduction Assessment

Yes for the PR review findings by source inspection: the diff targets removed legacy paths and direct Bot API fetches, while current main owns Telegram actions under `extensions/telegram` and already has a guarded media-download path. This is not a current-main user bug reproduction.

## Solution Assessment

No. The useful media-access problem is already solved by the shipped guarded media-store path, and the PR's remaining exact action/placeholder UX is not the narrowest maintainable implementation on current main.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Port download-file onto the Telegram plugin surface:** `src/channels/plugins/actions/telegram.ts:107-108`
  - body: The added discovery and dispatch live in the legacy channel-action shim, but current main owns Telegram message actions in `extensions/telegram/src/channel-actions.ts`; as written, the action is not advertised or handled after rebasing.
  - confidence: 0.93
- **[P1] Use the guarded Telegram media downloader:** `src/agents/tools/telegram-actions.ts:478-493`
  - body: This path calls Telegram URLs directly, buffers the whole response, and writes to `/tmp` instead of using the existing transport-aware, size-limited, SSRF-checked media downloader and media store, so proxy/apiRoot deployments and media protections can be bypassed.
  - confidence: 0.91
- **[P2] Keep audio detection compatible with attributed placeholders:** `src/telegram/bot-message-context.ts:470`
  - body: Emitting `<media:audio file_id="...">` requires updating the shared inbound-audio parser too; it still matches only bare `<media:audio>`, so inbound-audio-conditioned behavior can miss messages when media type facts are absent.
  - confidence: 0.87
- **[P3] Avoid deterministic temporary-path overwrites:** `src/agents/tools/telegram-actions.ts:500`
  - body: The output path uses only the first 16 characters of `fileId`, so different files sharing that prefix can overwrite each other; use the media store or a collision-resistant sanitized name.
  - confidence: 0.79
