## Summary

Keep open: the shared resolver delegation is useful, but the current patch still uses the synchronous status result while warming the cache in the background, so a one-shot status read can keep showing the fallback. The PR also lacks after-fix real behavior proof for `openclaw status` or Gateway `/status`.

## What This Changes

The PR changes CLI/Gateway status context-window resolution to use the shared resolver, starts background context-window cache warming from status and Gateway startup paths, and updates status/context tests.

## Best Possible Solution

Keep the shared resolver delegation, but prepare or await the needed read-only context metadata before returning status output and leave any startup warmup lifecycle-owned, non-blocking, and proven with real status/startup output.

## Reproduction Assessment

Yes. Source inspection and the PR's own added test show the first status-style resolver result remains `200_000` until async warmup finishes, while the changed status code still returns the synchronous value immediately.

## Solution Assessment

No. Delegating the duplicate CLI runtime resolver is a good direction, but fire-and-forget warmup is not the best fix for a one-shot status command because it does not guarantee the returned status output is corrected.
