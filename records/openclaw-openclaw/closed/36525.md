## What This Changes

The issue requests a semantic refactor of `agents.defaults.sandbox.workspaceAccess` so sandbox mode uses only the session sandbox at `/workspace` and avoids exposing the main workspace except through explicit binds.

## Best Possible Solution

Keep the remaining design and implementation work on https://github.com/openclaw/openclaw/issues/37634 and https://github.com/openclaw/openclaw/pull/46300, with the explicit-bind authorization discussion left on https://github.com/openclaw/openclaw/issues/59718.

## Reproduction Assessment

Yes. Source inspection on current main shows `workspaceAccess: "none"` resolves to the sandbox workspace but marks Docker/local/remote filesystem write paths writable only for `rw`, so writes to `/workspace` are rejected before or at the bridge layer.

## Solution Assessment

No, keeping a second broad tracker is not the best way to solve it.
