## Summary

Close #69482 as a duplicate of the already-open #60000 tracker. Current main still has the same underlying gateway schema mismatch: allow-always persistence can write `source: "allow-always"`, but `exec.approvals.set` validates against an allowlist-entry schema that omits `source` and rejects additional properties before normalization. The related fix PRs in the supplied context are not merged/shipped, so this should close as duplicate, not as fixed.

## Best Possible Solution

Use #60000 as the canonical open tracker and fix the shared gateway approval-file schema mismatch there: accept the durable allowlist metadata that current runtime code writes, add a regression test for `exec.approvals.set`/`--gateway` with `source: "allow-always"`, and preserve #69482's Telegram-specific reproduction as supporting evidence.

## Evidence

- **Canonical open tracker:** Provided related-item context and the local ClawSweeper report show #60000 is open and already tracks the same remaining gateway allowlist schema bug: runtime allowlist entries can contain `source`, while the gateway `exec.approvals.set` schema rejects that metadata. #60000 is the better canonical tracker because it covers the shared schema mismatch rather than only the Telegram producer path.
- **Gateway schema still omits source:** `ExecApprovalsAllowlistEntrySchema` currently allows `id`, `pattern`, `argPattern`, `lastUsedAt`, `lastUsedCommand`, and `lastResolvedPath`, then sets `additionalProperties: false`; it does not include `source`.
