## Summary

Close as a duplicate/satellite: current main still has the reported device-less token-auth scope clearing behavior, but the remaining product decision is already tracked by the open canonical gateway auth scope items.

## What This Changes

The issue requests a way for token-authenticated gateway WebSocket clients to receive effective operator scopes so observer daemons can subscribe to session and channel message events non-interactively.

## Best Possible Solution

Keep https://github.com/openclaw/openclaw/issues/80836 as the canonical design thread for bounded token scope grants and preserve this report as concrete observability evidence there.

## Reproduction Assessment

Yes. Source and existing tests show that device-less shared token WS auth currently returns auth.scopes as [] and then read-scoped methods such as sessions.messages.subscribe fail with missing operator.read; I did not run a live gateway because this review is read-only.

## Solution Assessment

No for a separate new tracker. The maintainable path is to resolve the bounded token scope contract in the canonical gateway.auth.tokenScopes discussion, with tests and docs for the WS auth semantics.
