## Summary

Keep open: current main still routes context-overflow recovery through in-process compaction/summarization work, and I did not find a current-main or shipped change that addresses the reported compaction-specific event-loop starvation and delayed Telegram fetch timer.

## What This Changes

The issue reports that context-overflow auto-compaction can block the Node event loop long enough to delay Telegram fetch timers and downstream MCP calls.

## Best Possible Solution

Profile a live compaction-overflow repro, then move or split the CPU-heavy compaction preparation/summarization path so channel timers, Gateway RPCs, and pending fetches keep making progress during compaction.

## Reproduction Assessment

No, not with high confidence from this review alone. The issue provides precise production logs and current source still matches the in-process compaction and delayed fetch-timeout path, but I did not run a live current-main compaction starvation repro.

## Solution Assessment

Unclear. Worker-thread isolation or cooperative yields are plausible, but maintainers should first profile the compaction phases and choose the ownership boundary across compaction, provider calls, and channel/Gateway responsiveness.
