## Summary

Close as implemented on current main. Feishu now resolves a media-aware inbound dedupe key before the early processing claim, debounce checks, suppressed-message recording, and final persistent dedupe, and regression tests cover same-`message_id` audio events with different `file_key` values. This is main-only proof: the latest release tag inspected, `v2026.5.2`, does not contain this implementation yet.

## What This Changes

The issue reports that Feishu audio messages with the same `message_id` but different audio content are deduplicated as duplicates and dropped.

## Best Possible Solution

Keep the Feishu-local media-aware dedupe implementation and let the next release carry it; avoid broad shared-core dedupe changes for this channel-specific upstream ID behavior.

## Reproduction Assessment

No current-main failure is reproducible from source: same-id audio events now resolve to different media-aware keys before the early claim and final persistent dedupe. Regression tests cover distinct `file_key` values processing independently while a true repeat stays suppressed.

## Solution Assessment

Yes. The current Feishu-owned key resolver at the receive, debounce, suppression, and finalize call sites is the narrowest maintainable fix; changing shared core dedupe would be broader and risk unrelated channels.
