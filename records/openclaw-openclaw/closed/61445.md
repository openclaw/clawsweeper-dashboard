## Summary

This should close as duplicate/superseded because the remaining restart replay/cursor work is already tracked by open #62761. Current main still lacks a persisted iMessage replay cursor and the self-chat echo path can dispatch when the echo cache is empty, so this is not implemented; consolidating on the broader open cursor issue is the cleaner maintainer path.

## What This Changes

Request to prevent iMessage gateway restarts from replaying self-sent Messages DB rows as inbound messages when the in-memory echo cache starts empty.

## Best Possible Solution

Keep the implementation work on #62761: add an iMessage-owned per-account rowid/high-water cursor, pass the resume point to `watch.subscribe` when available, update it only after safe handling, and cover restart/self-chat/iCloud replay cases with focused plugin tests.

## Reproduction Assessment

Yes. A focused reproduction is an `is_from_me=true` self-chat payload with matching sender/chat/destination and an empty echo cache; full live replay depends on `imsg`/Messages DB timing, which is why the durable cursor work belongs with #62761.

## Solution Assessment

No separate fix here is not the best path. A one-off echo-cache warmup or grace window would only cover this subcase, while #62761’s persisted cursor/resume fix addresses the restart replay class that causes it.
