## Summary

Close as superseded. The same Runtime session-identity problem is now tracked by open #40044 on a narrower `sessionId` path, while this PR exposes raw `sessionKey` values that can embed channel and peer routing identifiers in provider-visible prompt metadata.

## What This Changes

The PR adds optional `sessionKey` runtime metadata, passes `params.sessionKey` from the embedded runner, renders `sessionKey=...` in the Runtime line, and adds unit coverage for present and omitted values.

## Best Possible Solution

Keep the canonical work on #40044 or an equivalent maintainer-approved path: expose a stable prompt-visible session UUID, document how `sessions_history` should use it, and keep raw route/session keys out of provider prompts unless that contract is deliberately approved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Avoid exposing raw session keys in Runtime:** `src/agents/system-prompt.ts:947`
  - body: `sessionKey` values are route keys: core builders can include channel, account, peer, group, or channel IDs in them. Rendering the raw key in the provider-visible system prompt leaks routing metadata and duplicates the safer #40044 `sessionId` path, so this should use the stable session UUID/current-session contract instead.
  - confidence: 0.85
