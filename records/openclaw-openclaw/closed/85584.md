- `merge-risk: 🚨 compatibility`: Changing forced message-tool-only behavior can alter existing cron prompts that relied on the model being pushed to call `message`.

## Summary

Keep this PR open for maintainer review: the patch looks coherent and I found no blocking line-level defect, but it deliberately changes cron delivery ownership and should be merged only after the compatibility/message-delivery tradeoff is accepted.

## What This Changes

The PR changes isolated cron `none` and `announce` delivery so the message tool remains available but is no longer forced, makes announce delivery direct-fallback-owned, and updates focused cron delivery tests.

## Best Possible Solution

Cron routine final delivery should have one owner: direct cron fallback for announce output, with the message tool optional and same-target verified sends suppressing fallback.

## Reproduction Assessment

Yes. Current main source shows `cron_none` and `cron_announce` force the message tool, and the PR body includes after-fix cron output plus focused tests, though I did not run tests in this read-only review.

## Solution Assessment

Yes, if maintainers accept the ownership contract. Changing `SourceDeliveryPlan` to direct fallback while keeping verified same-target message-tool suppression is the narrowest maintainable fix I found.
