## Summary

Keep open: this draft, maintainer-labeled PR addresses a real diagnostic recovery gap, but the proposed scheduler can drop the new structured recovery event after a successful abort/drain.

## What This Changes

Adds experimental `diagnostics.stuckSessionAbortMs`, structured `session.recovery` diagnostics, stale bookkeeping cleanup, schema/docs/reload updates, tests, and a changelog entry for stalled/stuck session recovery.

## Best Possible Solution

Keep the PR open, fix the recovery scheduler so successful abort/release paths always emit `session.recovery` without clobbering newer session state, then let a maintainer decide the experimental default and scope against #71127 and #76038.

## Reproduction Assessment

Yes. Code inspection gives a high-confidence path: a real successful abort/drain can update diagnostic state to `idle` before the PR-head `.then` block runs, causing the guard at `src/logging/diagnostic.ts:466` to skip the promised recovery event.

## Solution Assessment

No. The second-stage threshold is a plausible narrow direction, but the implementation should decouple recovery-result logging from the stale `processing` state guard and add coverage using the real abort/drain cleanup path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Emit recovery events after successful drains:** `src/logging/diagnostic.ts:468-469`
  - body: When recovery aborts an embedded run successfully, `abortAndDrainEmbeddedPiRun` waits for the run to end, and the normal cleanup path updates the diagnostic session to `idle` and changes `lastActivity` before this `.then` runs. This new guard then returns before `logSessionRecovery`, so real successful aborts skip the promised `session.recovery` event even though the injected fake in the test still passes. Emit the recovery result before rejecting the expected idle transition, or explicitly allow that transition.
  - confidence: 0.84
