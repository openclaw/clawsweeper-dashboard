## Summary

Current main still does not add `session:spawn` / `session:complete` to the internal `hooks.internal` surface, but the supported Plugin SDK already exposes equivalent subagent lifecycle and delivery-routing hooks. That makes this plugin/ClawHub scope rather than a new OpenClaw core internal-hook feature.

## Risks / Open Questions

- Plugin docs currently point to the plugin hook surface in general, but they do not obviously enumerate every subagent hook by name, so discoverability is weaker than the issue author likely expected.
- Workspace-local internal hooks (`HOOK.md` packs) still do not have feature parity with the plugin hook surface; this close only makes sense because the repo now has a supported plugin extension path for the same use case.

## Close Comment

Closing this as a Codex review under `clawhub`. Current `main` does not add `session:spawn` / `session:complete` to the internal `hooks.internal` event list (`docs/automation/hooks.md:34`), but the supported Plugin SDK already exposes the equivalent subagent lifecycle seams: `subagent_spawning`, `subagent_delivery_target`, `subagent_spawned`, and `subagent_ended` (`src/plugins/hook-types.ts:75`, `src/plugins/hook-types.ts:451`). Those hooks are wired into the live spawn/completion flow (`src/agents/subagent-spawn.ts:495`, `src/agents/subagent-spawn.ts:1114`, `src/agents/subagent-announce-delivery.ts:266`, `src/agents/subagent-registry-completion.ts:93`) and already let a plugin observe subagent start/end and reroute completion delivery via `{ channel, accountId, to, threadId }`, which covers the multi-bot routing/logging/orchestration use case from this issue. Given the repo scope in `VISION.md:52` (`Core stays lean; optional capability should usually ship as plugins.`), the right path here is a ClawHub/OpenClaw plugin built on `api.registerHook(...)`, not expanding the core internal-hook surface.
