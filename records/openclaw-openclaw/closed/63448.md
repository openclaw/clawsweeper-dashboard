## Summary

Keep open. Current `main` does not contain this PR's change, and the current code/tests support the review concern that reusing `job.sessionKey` for isolated cron runs would regress isolation by routing many jobs onto caller session keys that the cron tool auto-populates.

## Evidence

- **Current main still hardcodes dedicated cron keys for isolated runs:** `runIsolatedAgentJob` initializes `sessionKey` to `cron:${job.id}` and only overrides it for explicit `session:` targets. There is no `job.sessionKey` branch on `main`.
  - file: [src/gateway/server-cron.ts:333](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/gateway/server-cron.ts#L333)
  - command: `nl -ba src/gateway/server-cron.ts | sed -n '330,355p'`
  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **Main test suite asserts isolated jobs must stay on dedicated cron keys:** The gateway cron test explicitly expects isolated jobs with model overrides to call `runCronIsolatedAgentTurn` with `sessionKey: \`cron:${job.id}\`` and not `main`, and expects cleanup on that cron key.
  - file: [src/gateway/server-cron.test.ts:318](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/gateway/server-cron.test.ts#L318)
  - command: `nl -ba src/gateway/server-cron.test.ts | sed -n '318,360p'`
  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **Cron tool auto-stamps sessionKey from the caller session:** When a cron job object lacks `sessionKey`, the cron tool writes the resolved caller session key into the job. That means many jobs already carry a caller-derived `job.sessionKey` even when the user did not explicitly request custom isolated-session routing.
  - file: [src/agents/tools/cron-tool.ts:530](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/agents/tools/cron-tool.ts#L530)

  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **Agent turns default to isolated sessionTarget:** Cron normalization keeps backward-compatible defaults by assigning `sessionTarget = "isolated"` for `agentTurn` payloads unless the user explicitly chooses `current` or `session:...`.
  - file: [src/cron/normalize.ts:527](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/cron/normalize.ts#L527)
  - command: `nl -ba src/cron/normalize.ts | sed -n '527,537p'`
  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **Isolated runner uses the provided key as the base session and still forces a new session:** `runCronIsolatedAgentTurn` derives `baseSessionKey` from the passed `input.sessionKey` and then calls `resolveCronSession(... forceNew: input.job.sessionTarget === "isolated")`. Replacing the dedicated cron key with a caller session key would therefore rotate/persist isolated runs under that caller key.
  - file: [src/cron/isolated-agent/run.ts:459](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/cron/isolated-agent/run.ts#L459)

- **PR discussion already identified the regression mode:** The review thread on this PR flagged a P1: because cron-tool auto-stamps `sessionKey` and `agentTurn` defaults to `sessionTarget: "isolated"`, routing isolated jobs through `job.sessionKey` would reuse the caller's live session key instead of the dedicated `cron:<jobId>` key.
  - sha: [f5141383f493](https://github.com/openclaw/openclaw/commit/f5141383f4935c0bc8f0713a689ae643c632855a)

## Risks / Open Questions

- The underlying bug report about honoring an explicitly requested `--session-key` for isolated jobs may still need a narrower fix, but this PR's approach is not safe on current `main`.
- Merging this PR as written would likely regress existing isolated cron jobs by rebinding them to caller session keys that are auto-populated by the cron tool.
