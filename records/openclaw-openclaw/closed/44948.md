## Summary

Current main already implements the websocket continuation behavior described in #44948, and the same fix is present in release v2026.4.22.

## Close Comment

Closing from a Codex review because this behavior is already implemented on `main` and is present in released `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). The current websocket continuation path only sends `previous_response_id` for true incremental tool-result follow-ups (`src/agents/openai-ws-message-conversion.ts:299`, `src/agents/openai-ws-request.ts:113`), falls back to a full-context replay without `previous_response_id` when there are no new tool results (`src/agents/openai-ws-stream.ts:823`), and tracks the response parent per websocket manager/session from that stream turn’s own `response.completed` event (`src/agents/openai-ws-connection.ts:315`, `src/agents/openai-ws-connection.ts:595`). Regression coverage in `src/agents/openai-ws-stream.test.ts:2711` and `src/agents/openai-ws-stream.test.ts:2786` matches the stale-branch scenario described here.
