## Label Justifications

- `P2`: The original report described cron job aborts with a limited cron/watchdog blast radius rather than a system-wide outage.
- `impact:crash-loop`: The affected behavior is premature abortion of otherwise healthy isolated cron agent runs.

## Summary

Current main and the latest stable release now clear the cron pre-execution watchdog once CLI process, Codex app-server tool, or embedded tool execution starts, so the reported silent long-running tool path is bounded by the job timeout instead of the half-timeout watchdog.

## What This Changes

The issue reports that cron's isolated-agent pre-execution watchdog can abort silent long-running shell/tool work before the configured job timeout.

## Best Possible Solution

Keep the shipped execution-milestone split: true pre-execution stalls can still fail early, while started CLI/process/tool work remains governed by the configured job timeout.

## Reproduction Assessment

No. The current source path emits an execution milestone before silent CLI or tool work and clears the pre-execution watchdog, and regression coverage keeps execution-stage runs alive until the job timeout.

## Solution Assessment

Yes. The current solution is narrower than removing the divisor because it preserves early detection for genuine pre-execution stalls while allowing started work to use the configured timeout.
