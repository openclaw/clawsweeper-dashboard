## Summary

Keep open. Current main still has a source-visible path where a successful `sessions_send` target run followed by an `agent.wait` WebSocket close returns `status: "error"`, while only non-terminal wait timeouts enter the delayed A2A recovery path; the open compensation work covers a narrower timeout-classification problem, not the reported normal-close/stale-routing failure class.

## What This Changes

The issue asks OpenClaw to fix intermittent `sessions_send` timeouts and `gateway closed (1000 normal closure)` failures in long-running Windows multi-agent deployments where Gateway health checks still pass.

## Best Possible Solution

Keep this as the broader tracker until maintainers either reproduce the Windows multi-agent normal-close failure or add targeted recovery and instrumentation for accepted `sessions_send` runs whose `agent.wait` transport is interrupted, while preserving terminal timeout behavior.

## Reproduction Assessment

No high-confidence live reproduction exists for the full Windows stale-routing report. Source inspection does show a focused path where a successful target run followed by `agent.wait` closing with `gateway closed (1000 normal closure)` becomes a `sessions_send` error.

## Solution Assessment

Unclear because this is not a patch. A maintainable fix should handle proven recoverable wait-transport interruptions separately from terminal timeouts and coordinate with the narrower post-timeout compensation work.
