## Summary

Close as superseded by the narrower open tracker for argument-scoped exec allowlist matching. Current main intentionally treats `safeBins` as a stdin-only executable/profile fast path, not a compound-command trust list, and the attempted exact safeBins fix was closed unmerged; the remaining legitimate capability is tracked in #60427.

## What This Changes

The issue asks for `tools.exec.safeBins` to allow compound command strings such as `gh pr list`, `curl -s`, and `python3 -c` instead of denying them as allowlist misses.

## Best Possible Solution

Keep `safeBins` constrained to profiled stdin-only filters, and use #60427 as the canonical product/design thread for any future binary-plus-argument allowlist support.

## Reproduction Assessment

Yes. The current-main matcher and regression test provide a high-confidence static reproduction: unprofiled interpreter or compound safeBins entries still deny, matching the reported symptom while reflecting the current documented boundary.

## Solution Assessment

No. Matching compound commands through `safeBins` is not the best fix because it would turn the stdin-only fast path into a broad command trust list; the safer path is the argument-scoped allowlist design tracked in #60427.
