## Summary

Close as `clawhub`. Current main keeps internal workspace hooks limited to command/message/session/gateway events, while the existing plugin hook surface already exposes tool lifecycle interception and approval gating. The request is useful, but it fits a plugin/ClawHub implementation rather than expanding OpenClaw core internal hooks.

## Evidence

- **Internal hooks are intentionally narrower:** The internal hooks docs describe them as small gateway scripts for command and lifecycle events, and the published event list does not include tool-call/tool-result events.
  - file: [docs/automation/hooks.md:11](https://github.com/openclaw/openclaw/blob/754acc4478fe361db1e5303fd60f589d52beea7a/docs/automation/hooks.md#L11)

  - sha: [754acc4478fe](https://github.com/openclaw/openclaw/commit/754acc4478fe361db1e5303fd60f589d52beea7a)
- **Plugin hooks already expose tool lifecycle data:** The plugin hook contract includes `before_tool_call` with tool name and params, `after_tool_call` with tool name, params, result, error, and `durationMs`, plus `tool_result_persist` for persisted tool-result handling.
  - file: [src/plugins/hook-types.ts:306](https://github.com/openclaw/openclaw/blob/754acc4478fe361db1e5303fd60f589d52beea7a/src/plugins/hook-types.ts#L306)

- **Core runtime executes those plugin tool hooks:** The hook runner has dedicated `runBeforeToolCall`, `runAfterToolCall`, and `runToolResultPersist` paths.
  - file: [src/plugins/hooks.ts:777](https://github.com/openclaw/openclaw/blob/754acc4478fe361db1e5303fd60f589d52beea7a/src/plugins/hooks.ts#L777)

Tests also cover blocking tool execution via `before_tool_call` on HTTP and MCP tool surfaces.

## Risks / Open Questions

- Users who specifically want file-based internal hook scripts, not plugins, still do not have a first-class tool lifecycle event surface there.
- If maintainers later decide internal hooks should gain parity with plugin hooks for operator scripting, this issue may need to be reopened or superseded.
