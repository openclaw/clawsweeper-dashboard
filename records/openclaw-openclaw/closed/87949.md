## Best Possible Solution

Update the PR so the real message-tool path either safely routes runner-normalized media through Mattermost's existing upload path with the media access context or rejects the original unsupported attachment inputs before posting, remove the release-owned changelog edit, and add redacted live Mattermost proof or maintainer validation before merge.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: send through the message tool with `filePath` or `attachments[]`, the runner maps that media into `params.media`, and Mattermost's direct action path does not receive the media-read context needed for safe local upload. ## Solution Assessment

No, the proposed direction is close but not the best implementation yet because the guard is placed after runner normalization and checks `!mediaUrl`, so it misses the real reported path. The safer fix is to detect the original unsupported attachment inputs in the runner-dispatched path or wire the existing upload path with the proper media access context.
