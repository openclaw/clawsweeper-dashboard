## Summary

#66500 is the same Feishu inbound Chinese filename mojibake bug already tracked by existing open Feishu filename reports, especially #48388 and #59409. Current main still has the same header-derived filename path, so this should close as duplicate rather than as implemented.

## Best Possible Solution

Close #66500 as a duplicate of the existing Feishu inbound filename tracking, with #48388 as the older canonical issue and #59409/#59431 as related active fix work. The eventual fix should land on the canonical thread with a targeted inbound regression test for Chinese filenames plus care not to rewrite legitimate non-UTF-8 filenames incorrectly.
