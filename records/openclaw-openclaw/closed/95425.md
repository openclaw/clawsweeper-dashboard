## Summary

Keep open; the root-problem grouping idea is useful, but the new skill instructions are not merge-ready because the checkout recipe still fetches from `origin` for fork-head PRs and the resolved-thread rule lacks a concrete data source. The external PR also has only static inspection proof, not a real `gh-issues --reviews-only` run.

## What This Changes

This PR rewrites the `gh-issues` skill's reviews-only follow-up guidance to group review comments by root problem, process groups sequentially, and add deterministic checkout/reply instructions.

## Best Possible Solution

Keep the sequential root-problem design, but make the checkout instructions use the actual PR head repo/ref and add a concrete resolved-thread metadata fetch path before asking for redacted live-output proof.

## Review Findings

Overall confidence: 0.89

Full review comments:

- **[P2] Fetch the PR head remote in the checkout recipe:** `skills/gh-issues/SKILL.md:206-207`
  - body: This recipe always fetches `origin/$BRANCH`, but the skill already supports fork-created PRs through `--fork`, `PUSH_REPO`, and `PUSH_REMOTE`. For a `fix/issue-*` PR whose head lives in a fork, `origin` may not have that ref, or it may have a different branch with the same name, so reviews-only follow-up can fail before editing or push replies for the wrong branch. Make the example use the PR head repository/remote, or use an equivalent exact-head checkout flow.
  - confidence: 0.91
- **[P2] Fetch review thread state before skipping resolved threads:** `skills/gh-issues/SKILL.md:189`
  - body: The new text tells workers to ignore resolved threads, but the listed fetches only read PR comments, reviews, and issue comments. Those sources do not give workers the thread resolution state needed to make that decision reliably; add a `reviewThreads`/resolved-state fetch path or remove the resolved-thread rule from this flow.
  - confidence: 0.87
