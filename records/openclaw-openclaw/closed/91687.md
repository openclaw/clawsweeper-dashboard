## Summary

Current `main` and shipped `v2026.6.1` already restrict model fallback candidates to the requested/effective primary, explicit fallback entries, and the configured primary, while exact configured provider/model refs are protected from plugin-catalog normalization. That addresses the reported merge-mode fall-through to a provider outside `agents.defaults.model.fallbacks`.

## What This Changes

The issue reports that `models.mode: "merge"` can add a plugin-catalog provider candidate to the runtime fallback chain even when `agents.defaults.model.fallbacks` only names a different provider/model.

## Best Possible Solution

Keep the shipped explicit fallback-chain behavior and ask for a fresh report only if the same fall-through reproduces on `v2026.6.1` or newer.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? The report includes detailed old-version logs and config, but I did not run a live current-main repro; source and release-tag inspection show the current chain no longer has the hidden catalog-provider path.

## Solution Assessment

Is this the best way to solve the issue? Yes: the fix lives at the model fallback and model-ref normalization boundary, where it preserves exact configured provider refs and limits fallback attempts to explicit operator-controlled candidates.
