Full review comments:

- **[P1] Refresh continuation config on reload:** `src/auto-reply/continuation/config.ts:5-6`
  - body: `config.ts` promises continuation values are hot-reloadable, but `agents.defaults.continuation.*` still matches the catch-all `agents` no-op reload rule and several enforcement paths call `resolveContinuationRuntimeConfig()` instead of the live snapshot helper. Operators can flip `crossSessionTargeting`, caps, or delays and keep running with stale security/cost settings until restart, contrary to the PR contract.
  - confidence: 0.88
- **[P1] Redact continuation reasons before tracing:** `src/infra/continuation-tracer.ts:462`
  - body: Both delegate and work spans copy the first 80 characters of `args.reason` into `reason.preview`. That value is model/user-supplied continuation text and can contain private task details, so exporting it as an OTel attribute leaks content to telemetry backends; drop it, hash it, or pass it through the existing redaction policy before export.
  - confidence: 0.86
- **[P2] Strip stale continuation markers from every payload:** `src/auto-reply/continuation/signal.ts:93-98`
  - body: The backward scan stops after stripping the payload that supplies the winning signal. If an earlier text payload also contains a bracket continuation marker, that stale marker remains in the outbound reply and can be shown to users even though the runtime consumed a later marker; keep the last signal but strip markers from all text payloads before delivery.
  - confidence: 0.83
