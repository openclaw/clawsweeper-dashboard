## Summary

Current main still allows the misleading dual-error path for normal npm plugin install failures: non-forced security failures can fall through to hook-pack fallback, and the formatter still appends `Also not a valid hook pack: ...`. This issue should remain the canonical tracker; the related #69181 was already closed as a duplicate into it.

## Best Possible Solution

Keep this issue open and fix the CLI install routing. Treat security-policy failures and valid-plugin validation failures as terminal plugin-install errors, and attempt hook-pack fallback only for explicit hook-pack-looking inputs or clear non-plugin/missing-plugin classifications. Add a regression in `src/cli/plugins-cli.install.test.ts` for a normal npm plugin security-block failure without `--dangerously-force-unsafe-install`, asserting `installHooksFromNpmSpec()` is not called and `Also not a valid hook pack` is absent.
