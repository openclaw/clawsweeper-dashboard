## Summary

Keep this PR open. Current main contains the uncaught-exception handler registry and SDK exports, but it still does not implement the PR’s central behavior: transient network uncaught exceptions such as ENETUNREACH/ENOTFOUND still fall through to fatal CLI handling, and Telegram polling recovery is still wired only to unhandled rejections. The discussion also has fresh production corroboration for the same class of Telegram and Discord voice failures, while the linked bug #60515 remains open.

## Best Possible Solution

Keep this PR open for maintainer review, rebase, or a direct current-main port. The best path is to preserve fatal hooks, add narrowly reviewed ENETUNREACH/ENOTFOUND-class uncaught suppression where appropriate, wire Telegram polling recovery through registerUncaughtExceptionHandler as well as registerUnhandledRejectionHandler, add regression tests for Telegram uncaught polling HttpError recovery and CLI ENETUNREACH/ENOTFOUND uncaught suppression, and resolve duplicate legacy-handler logging before landing.
