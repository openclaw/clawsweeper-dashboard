## Summary

Current main already routes reply execution back through config unless the session has an explicit `providerOverride`/`modelOverride`; stale runtime `model`/`modelProvider` fields in `sessions.json` are no longer the authority for selecting the next run's model. The same behavior is present in the shipped `v2026.4.22` release.

## Evidence

- **Reply path only reloads explicit session overrides:** `get-reply.ts` preloads stored model selection from `resolveStoredModelOverride(...)`, which reads only `providerOverride`/`modelOverride`, not the runtime `model`/`modelProvider` fields from `sessions.json`. That means stale runtime model metadata is not used to choose the next reply model after restart.
  - file: [src/auto-reply/reply/get-reply.ts:334](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/auto-reply/reply/get-reply.ts#L334)
  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)
- **Model selection falls back to config when no override exists:** `createModelSelectionState(...)` derives `directStoredOverride` only from persisted override fields and otherwise keeps the configured `defaultProvider`/`defaultModel`. There is no runtime-model fallback in this reply-selection path.
  - file: [src/auto-reply/reply/model-selection.ts:302](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/auto-reply/reply/model-selection.ts#L302)
  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)
- **Regression test covers default re-resolution:** The test `falls back to default when no modelOverride is set` asserts that session model selection uses the configured default when there is no persisted override, which is the behavior this issue requested on restart.
  - file: [src/auto-reply/reply/model-selection.test.ts:444](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/auto-reply/reply/model-selection.test.ts#L444)
  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)
- **Docs explicitly describe stale runtime fields as non-authoritative:** The failover docs now state that live-session reconciliation prefers persisted session overrides over stale runtime model fields, matching the implemented behavior in the reply path.
  - file: [docs/concepts/model-failover.md:301](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/docs/concepts/model-failover.md#L301)
  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)

## Risks / Open Questions

- Some non-reply Gateway/status helpers still use `resolveSessionModelRef(...)` and can surface stale runtime metadata until a later run refreshes the session entry, even though reply execution itself now re-resolves from config.
- The checkout history is shallow/grafted, so I could not recover the original patch commit; the release SHA below is the best local proof that the behavior shipped.
