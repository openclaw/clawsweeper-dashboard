## Summary

Current main fixes the reported rapid-Enter duplicate backend submit through the shared Control UI send path, not by adding the exact keyboard guard proposed in the issue. `handleSendChat` now keys submissions by session/message/attachments and suppresses a matching in-flight submit before a second `chat.send` id is generated, with regression coverage and WebChat docs. Related PR #72512 was closed unmerged after the same conclusion. The fix is on current main from f6b2ba4a10afe7bedf154f1765bc1943748d3c81, but I could not prove it is in the latest release tag v2026.4.26.

## Best Possible Solution

Keep the shared `handleSendChat` submit coalescing on main, because it covers both Enter and button submits and preserves the intended active-run queue workflow. Close this issue as implemented on main, and only reopen or file a new issue if a build containing f6b2ba4a10afe7bedf154f1765bc1943748d3c81 can still produce duplicate backend `chat.send` requests from rapid Enter presses.
