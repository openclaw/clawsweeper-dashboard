## Summary

Keep open: current main still treats literal cron `payload.model: "default"` as an explicit model ref and can produce the reported allowlist rejection, while the normal null/blank create/load paths are partly normalized away and need focused coverage for edit/clear behavior.

## What This Changes

Allow isolated cron jobs with `payload.model` set to `default`, null, blank, or omitted to inherit the stored cron-session or agent/default model instead of failing preflight.

## Best Possible Solution

Normalize cron inheritance sentinels before preflight, add an explicit clear path for persisted payload model overrides, and preserve strict allowlist failures for real explicit model refs.

## Reproduction Assessment

Yes for source-level reproduction of the literal `default` failure: current main validates every nonempty payload model as an explicit ref before preflight. Null/blank inheritance is not fully reproduced because normal create/load paths delete those values, but the edit merge path can preserve stale models.

## Solution Assessment

Yes, a narrow fix is appropriate if it normalizes only unset/default sentinels and adds a clear path for persisted overrides. Loosening the allowlist globally or silently falling back for unknown explicit models would not be the best solution.
