## What This Changes

The issue reports that after upgrading to OpenClaw 2026.5.18, embedded channel runs fail with `EmbeddedAttemptSessionTakeoverError` when the session JSONL changes while the prompt lock is released.

## Best Possible Solution

Keep the takeover fence, make it ownership-aware for legitimate OpenClaw-managed or lock-coordinated session writes during the released prompt window, and land focused regression coverage that still rejects true external takeovers.

## Reproduction Assessment

Yes, at source level: create an embedded session-lock controller, call `releaseForPrompt()`, mutate the same session file, then call `withSessionWriteLock()` and current main throws `EmbeddedAttemptSessionTakeoverError`. I did not run the live Feishu/APFS path.

## Solution Assessment

No merged solution exists on current main yet. The append-only ownership-aware direction in the open PR is plausible for runner-owned writes, but co-tenant shared-session writes need the separately tracked ownership/generation treatment.
