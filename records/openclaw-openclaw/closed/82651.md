## Summary

Current main still has the reported `maxPoints` boundary bug, and the issue should stay open. The cause is narrow and source-proven, so this is a good focused repair candidate rather than a cleanup close.

## What This Changes

The issue requests that `loadSessionUsageTimeSeries` return an empty `points` array for zero, negative, or non-finite `maxPoints` values while preserving positive downsampling behavior.

## Best Possible Solution

Add a small validation guard in the time-series loader before step calculation, then cover zero, negative, `NaN`, and infinity in the existing infra test file.

## Reproduction Assessment

Yes. A small transcript with two assistant usage entries and `maxPoints: 0` reaches the current downsampling branch, computes an infinite step, and returns one aggregate bucket; this was confirmed by source inspection and an isolated loop check, not by running the full test suite.

## Solution Assessment

Yes. The narrow maintainable fix is to validate `maxPoints` inside `loadSessionUsageTimeSeries` before computing the downsampling step, without changing the gateway protocol, UI, or aggregation model.
