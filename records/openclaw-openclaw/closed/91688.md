## Summary

The patch is technically sound and well proven, but it intentionally tightens the released cron CRUD contract: schedules with no future match now fail immediately instead of being accepted as dormant jobs. That product and compatibility choice needs explicit cron-owner acceptance, so this PR should remain open for maintainer review.

## What This Changes

The PR adds shared service-layer validation that rejects cron schedules with no future match on creation, schedule changes, and re-enabling, maps the failure to gateway INVALID_REQUEST, and adds regression tests including staggered schedules and legacy dormant-job repairs.

## Best Possible Solution

Keep the dependency-backed validation at the shared cron service boundary and preserve repair and disable operations for existing dormant jobs, then merge only after a cron owner explicitly accepts fail-fast rejection for both enabled and disabled new schedules.

## Reproduction Assessment

Yes. Current main converts Croner’s null next run to undefined and accepts the job without an armed timestamp; the PR also supplies matching before-and-after production-path output for Feb 30 and Apr 31 expressions.

## Solution Assessment

Yes for the implementation boundary and dependency semantics: shared service validation covers RPC and direct callers, uses the injected clock, and preserves legacy repair operations. The unresolved question is whether maintainers want this stricter contract to include disabled job creation.
