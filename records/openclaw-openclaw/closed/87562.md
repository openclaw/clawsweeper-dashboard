## Summary

Keep open: the PR targets a real OpenRouter billed-cost undercount, but the proposed wrapper misses valid direct `result()` stream consumers and the new authenticated OpenRouter lookup still lacks live vendor proof.

## What This Changes

This PR adds an OpenRouter post-stream cost reconciliation wrapper that fetches generation metadata and overwrites `usage.cost.total` when `/generation` reports a larger billed `total_cost`.

## Best Possible Solution

Keep the OpenRouter-owned reconciliation, make final-message reconciliation idempotent across iterator and direct `result()` paths, add direct-result regression coverage, and provide live or redacted OpenRouter proof for the authenticated lookup.

## Reproduction Assessment

Yes at source level: current main records streamed `usage.cost.total` without an OpenRouter `/generation` lookup, and linked issue #68066 provides concrete OpenRouter generation totals showing a 4x undercount.

## Solution Assessment

No as written: provider-local post-stream reconciliation is the right boundary, but it must cover both async iteration and direct `result()` consumers with one idempotent reconciliation path.
