## Summary

Keep open. Current `main` still injects the caller session key into `cron.add` requests when `job.sessionKey` is absent, and isolated execution still prefers any stored `sessionKey` over the default `cron:<jobId>` base session. That matches the reported contamination path and conflicts with the docs that define `sessionTarget: "isolated"` as a dedicated `cron:<jobId>` session, while `current` is the mode that binds at creation time.

## Evidence

- **Cron tool still stamps missing sessionKey from caller context:** `createCronTool()` normalizes the job, then unconditionally sets `job.sessionKey = resolvedSessionKey` whenever the caller has an `agentSessionKey` and the job omitted `sessionKey`. There is no guard for `sessionTarget === "isolated"`.

  - command: `nl -ba src/agents/tools/cron-tool.ts | sed -n '528,556p'`
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)
- **Current test suite codifies the sessionKey stamping behavior:** The test `stamps cron.add with caller sessionKey when missing` asserts that a newly added cron job inherits the caller session key, so this behavior is intentional in current `main`, not an already-fixed regression.
  - file: [src/agents/tools/cron-tool.test.ts:276](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/src/agents/tools/cron-tool.test.ts#L276)

- **Isolated runner still uses stored sessionKey before cron:<jobId>:** `runCronIsolatedAgentTurn()` computes `baseSessionKey` as `input.sessionKey?.trim() || \`cron:${input.job.id}\``, so any persisted session key prevents the documented `cron:<jobId>` isolation path from being used.
  - file: [src/cron/isolated-agent/run.ts:459](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/src/cron/isolated-agent/run.ts#L459)

- **Docs say isolated and current are distinct modes:** The cron docs define `isolated` as running in a dedicated `cron:<jobId>` session with a fresh session, while `current` is the mode that is bound at creation time. That documented contract does not match the current tool behavior for isolated jobs created from chat.
