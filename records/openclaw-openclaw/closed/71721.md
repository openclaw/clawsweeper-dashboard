## Label Justifications

- `P1`: The original report was a gateway startup crash in sandboxed deployments that made the TUI unable to connect.
- `impact:crash-loop`: The issue is about a process-level gateway crash/disconnect caused by denied network-interface enumeration.

## Summary

The reported sandbox crash path through @homebridge/ciao `NetworkManager.assumeNetworkInterfaceNames` is now handled on current main and shipped; the remaining raw LAN/tailnet interface helpers are separate optional paths, not the stack shown in this issue.

## What This Changes

The issue asks for gateway startup to survive `os.networkInterfaces()`/`uv_interface_addresses` failures in restricted NemoClaw/OpenShell sandboxes instead of disconnecting the TUI.

## Best Possible Solution

Keep the shipped Bonjour suppression for restricted sandboxes; file a separate narrow issue if optional `bind=tailnet` or wide-area interface discovery still crashes a gateway.

## Reproduction Assessment

Yes, source-level for the original failure: the issue includes a concrete NemoClaw/OpenShell command path and ciao stack, and current tests reconstruct the exact `uv_interface_addresses` SystemError shape. Current main is expected not to crash on that path because the classifier now suppresses it.

## Solution Assessment

Yes. The shipped fix is the narrow maintainable solution for the reported stack: classify ciao's interface-enumeration failure, warn once, and disable mDNS instead of letting the unhandled rejection kill the gateway.
