## Summary

Keep open. Current main still has the Gateway no-auth backend handshake gap: `auth.mode: "none"` resolves successfully, but shared-auth state and local backend self-pairing still do not admit method `none`, so device-less backend clients can reach `1008: device identity required`. The open fix PR at https://github.com/openclaw/openclaw/pull/75781 references this issue with closing syntax, and the later Windows stable reproduction in the discussion reinforces that this should stay open until that PR or a corrected replacement lands.

## What This Changes

Allow internal Gateway backend RPC/self-pairing connections to work under documented trusted-loopback `gateway.auth.mode: "none"` without requiring device identity.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/75781 or a corrected replacement that carries a local-backend auth-none allowance through the full missing-device handshake path with regression coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes: source inspection on current main shows `auth.mode: "none"` produces method `none`, leaves `sharedAuthOk` false, misses the local backend skip, and reaches the `1008` device-identity rejection; a later Windows 2026.5.6 comment also reports the same runtime failure.

## Solution Assessment

Is this the best way to solve the issue? Yes, keeping this issue open while reviewing the linked PR is the right current path; the maintainable fix should be a narrow local-backend no-auth allowance in the full handshake/missing-device policy, not a broad pairing bypass.
