## Summary

Current main and the shipped v2026.4.27 release already cover the reported leak class: commentary-phase assistant text is dropped before user-visible partial/block/final delivery, and known OpenClaw runtime-context scaffolding is stripped from user-facing text. The remaining Telegram tool-progress preview path is intentional and configurable, and it does not match the leaked assistant narration examples in this report.

## What This Changes

The issue asks OpenClaw to prevent tool-call intermediate narration and internal runtime-context text from bypassing SOUL.md output rules in user-facing Telegram replies.

## Best Possible Solution

Close this as shipped in v2026.4.27 and keep the deterministic delivery-boundary filters in place. Operators who want no live tool-progress preview on Telegram should use the documented `streaming.preview.toolProgress: false` setting, while SOUL.md remains model guidance rather than a general-purpose deterministic language filter.

## Reproduction Assessment

Yes. The issue gives concrete leaked strings, and the current regression tests reproduce the same class by injecting commentary-phase assistant messages and copied runtime-context prefaces, then asserting they do not reach user-visible output.

## Solution Assessment

Yes. The current implementation fixes this at the delivery and sanitizer boundaries, which is narrower and more reliable than trying to reinterpret arbitrary SOUL.md prose as a post-hoc output filter.
