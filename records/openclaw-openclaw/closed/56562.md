## Summary

Keep this issue open. Current main partially mitigates the examples with hard-coded runtime validation for the built-in `read`, `write`, and `edit` filesystem tools, but it does not implement the requested `tools.validate_before_call` or `tools.retry_on_missing_params` config, does not validate all tools from their JSON schemas before dispatch, and does not automatically re-prompt/retry missing-parameter calls.

## Best Possible Solution

Keep the issue open and implement this as a core tool-dispatch/runtime feature: add typed and zod-backed `tools.*` config, validate missing required parameters against the normalized tool schema before handler invocation, return a low-noise structured tool result for validation failures, and only add bounded automatic retry when the failure is known to be pre-dispatch and side-effect free. The existing read/write/edit guard can remain as a narrow safety net or be folded into the generic validator once equivalent coverage exists.

## Risks / Open Questions

- A generic schema validator needs to preserve provider-specific schema normalization, plugin hooks, and tools with intentionally flexible parameter shapes.
- Automatic retry must be bounded and side-effect aware so it does not replay a partially executed or approval-gated tool call.
