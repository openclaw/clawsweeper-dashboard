## Summary

Current main already removes the reported reply-side GPT brevity mutation: successful final payloads are returned unchanged, and regression coverage now asserts GPT prose is preserved for an external-channel-style run. The old guard is still present in v2026.5.12, so this is fixed on current main but not yet proven shipped in a release.

## What This Changes

The issue asks OpenClaw to stop GPT-5 reply-side brevity post-processing from truncating final reply payload text before external channel delivery, especially Signal.

## Best Possible Solution

Keep the current-main behavior that does not rewrite successful final reply payloads, and let channel plugins handle platform size limits through delivery, chunking, or explicit errors.

## Reproduction Assessment

No for current main: the truncating guard is absent from current source and regression tests now assert the relevant GPT final text stays intact. The v2026.5.12 source still contains the old guard, which explains the reported runtime observation.

## Solution Assessment

Yes: removing the reply-layer mutation path is the narrowest maintainable fix, because delivery payloads should not be silently rewritten after model generation. A new opt-in config is unnecessary for the fixed default behavior.
