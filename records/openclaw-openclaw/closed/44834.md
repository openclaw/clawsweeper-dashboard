## Summary

Current main already prevents the reported `openclaw secrets configure` prompt corruption by avoiding plugin runtime/CLI registration for the built-in `secrets` command path, and `secrets configure` now relies on config/manifest/secret-contract surfaces rather than loading Feishu-style plugin runtime modules. The PR’s filter-based patch is obsolete, and the existing review comments correctly identified that the patch was only partial because it restored the filter before later confirmation prompts.

## Best Possible Solution

Close this PR as obsolete and keep the shipped startup/lazy-registration implementation. If prompt corruption returns, the next best change would be a targeted regression test proving `openclaw secrets configure` does not load plugin runtimes before prompts, rather than merging this partial filter wrapper.

## Evidence

- **Built-in secrets command skips plugin CLI registration:** `run-main` registers the primary built-in command first, then computes `hasBuiltinPrimary`; when true, `shouldSkipPluginCommandRegistration` prevents plugin CLI registration before parsing. This covers `secrets` because it is a built-in sub-CLI.
