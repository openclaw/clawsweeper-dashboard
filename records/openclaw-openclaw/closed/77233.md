## Summary

Close as implemented on current main: the web provider resolver now falls back from an otherwise compatible active runtime registry with zero web-search providers to a scoped plugin provider load, and regression coverage names the exact empty-registry case. The fix is present on main after v2026.5.3 and does not appear to be in a shipped release tag yet.

## What This Changes

Make agent-facing web_search discover an installed external Brave web-search plugin when the active runtime registry has no web-search providers.

## Best Possible Solution

Keep the current main fallback behavior and ship it in the next release so external web-search plugins are visible to the managed web_search tool even after unrelated runtime registries are active.

## Reproduction Assessment

Yes for the reported release path: the issue gives concrete 2026.5.3 logs and current tests/source identify the empty active-registry failure mode. Current main no longer reproduces that path because the resolver falls through to plugin provider discovery.

## Solution Assessment

Yes; the current implementation is the narrowest maintainable fix because it preserves explicit empty scopes while treating a zero-provider active registry as non-authoritative for web provider discovery.
