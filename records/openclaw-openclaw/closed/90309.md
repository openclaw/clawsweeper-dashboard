## Summary

Current main and the latest release already implement the central fix: manual/non-timeout aborts now call the same session-lock release helper instead of leaving that cleanup behind the timeout-only abandonment branch.

## What This Changes

The issue reports that pressing stop during a run can leave the embedded attempt session write lock held until maxHoldMs, causing later turns in the same session to fail with SessionWriteLockTimeoutError.

## Best Possible Solution

Keep the current abort-lock cleanup behavior in v2026.6.1/current main; any new lock-timeout report should include a reproduction against that version or a newer main build.

## Reproduction Assessment

No current-main failure reproduction remains from source inspection: the described timeout-only release gap has been replaced by an unconditional abort-lock release helper. I did not run the live stop-button scenario in this read-only review.

## Solution Assessment

Yes: the current implementation is the narrow maintainable fix because it makes manual and timeout aborts share the same lock-release invariant without changing lock reentrancy or adding configuration.
