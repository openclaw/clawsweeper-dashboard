## Summary

Keep open. Current main still has no `maxExecutionMs` or task-level command-lane timeout API, so the PR targets a real missing hardening path. It should not be closed as implemented or stale, but it needs rework before merge because the provided review discussion identifies material issues: the timeout wrapper can free the lane while the original task keeps running, and no production caller wires the new option into agent or cron lane enqueues.

## Best Possible Solution

Keep this PR open for maintainer review, but require a revised implementation before merge: add a queue-level execution budget that has a clear cancellation/accounting contract, propagate an `AbortSignal` or otherwise prevent timed-out work from continuing side effects after the lane is freed, wire agent and cron callers to their existing timeout policies, and coordinate with the related #48690 timeout work and dependent #68785 circuit-breaker PR.

## Risks / Open Questions

- Merging this PR as-is could free the lane slot while the original task continues side effects concurrently, weakening lane serialization and restart drain accounting.
- Without production wiring, the proposed `maxExecutionMs` path would be effectively test-only and would not mitigate the stated stalled-lane scenario.
- A fixed or poorly chosen lane timeout could conflict with existing long or disabled agent and cron timeout policies.
