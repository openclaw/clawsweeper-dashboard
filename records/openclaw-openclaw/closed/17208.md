## Summary

The request is coherent, but on current `main` it fits the documented plugin/context-engine extension model rather than OpenClaw core. Core already exposes generic context diagnostics, while opinionated context scoring and warning policy is explicitly the kind of optional capability the project asks to ship as plugins/ClawHub.

## Risks / Open Questions

- A plugin cannot currently extend the built-in `/status` card directly, so an external implementation would likely use its own slash command or context-engine prompt additions instead of native `/status` wiring.
- If maintainers want one canonical scoring heuristic across all users, this scope call could change, but the present repo docs point the other way.

## Close Comment

Closing this as a Codex review because the requested context-quality scoring is plugin-shaped work, not a core cleanup item on current `main`. The project vision explicitly says optional capability should usually ship as plugins and that new skills should go to ClawHub first (`VISION.md:52-70`). OpenClaw already has a dedicated context-engine extension point for custom context assembly/compaction policy, token estimates, prompt additions, and post-turn maintenance (`docs/concepts/context-engine.md:10-16`, `67-96`, `116-149`, `182-209`), and the public plugin API can register custom commands before built-in commands and before agent invocation (`src/plugins/types.ts:2000-2008`, `docs/plugins/sdk-overview.md:83-89`). Core already covers the generic diagnostics side with `/status`, `/context list`, and `/context detail` (`docs/concepts/context.md:20-26`, `113-130`). Given that shape, this belongs as a ClawHub/context-engine plugin rather than additional OpenClaw core behavior.
