The fix is present on main only in this checkout; no release tag contains it yet.

## What This Changes

The issue asks for Telegram `/stop` to abort an in-flight prompt again after the reported 2025.5.12 upgrade.

## Best Possible Solution

Keep the current main Telegram control-lane and isolated-spool draining implementation, and let the next release carry the fix to users still on the affected release.

## Reproduction Assessment

No; current main no longer has a high-confidence failing reproduction because `/stop` is routed onto Telegram's control lane and tests cover control updates bypassing an active spooled turn. The old failure mode is source-traceable to the isolated-ingress spool lane drain that is now fixed on main.

## Solution Assessment

Yes; the current fix is the narrow maintainable solution because it lets abort/status control updates bypass blocked normal Telegram lanes without changing `/stop` command semantics or adding configuration.
