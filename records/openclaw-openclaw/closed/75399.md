## Summary

Keep open. Current main still has the media-generation tool-registration regression described by this PR, and the proposed patch is a narrow alignment between the cheap availability gate and the existing provider readiness/model-resolution path. I found no blocking review findings in the provided diff, but the PR is still open and should be reviewed, validated, and landed rather than closed by cleanup.

## What This Changes

The PR passes runtime media-generation providers into the shared availability gate and adds regression coverage so configured local or non-auth providers can expose image, music, and video generation tools without an explicit default model config or API-key auth.

## Best Possible Solution

Land a narrow generic fix that makes tool registration no stricter than runtime model resolution: explicit model config should remain a fast path, and otherwise the gate should use the same provider readiness semantics that generation already uses, including plugin-owned `isConfigured` callbacks for local providers like Comfy.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a config with `plugins.entries.comfy.config.<capability>.workflow` or `workflowPath` plus `promptNodeId`, `mode: "local"`, no `agents.defaults.*GenerationModel`, and no COMFY auth; current main's gate returns false while Comfy `isConfigured` returns true.

## Solution Assessment

Yes. The proposed direction is the narrowest maintainable fix because it keeps the gate generic, preserves the explicit model-config short circuit, and delegates readiness to the same runtime provider list and `isConfigured` contract already used by model resolution.
