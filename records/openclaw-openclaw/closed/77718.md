## Summary

This should stay open: current main and v2026.6.6 already handle an omitted per-agent workspace by falling back to the default workspace, but a literal null workspace still fails schema validation and the doctor compatibility pipeline does not appear to repair that malformed field.

## What This Changes

The issue asks OpenClaw to recover from agents.list entries whose workspace field is missing or null instead of entering a gateway startup crash loop.

## Best Possible Solution

Add a narrow doctor --fix compatibility repair that removes or normalizes null agents.list[].workspace values so runtime config remains strict and workspace resolution continues through the existing fallback path.

## Reproduction Assessment

Yes for the null case from source inspection: workspace is optional but must be a string, invalid snapshots are preserved as invalid, and gateway startup throws on invalid config. Missing workspace alone no longer reproduces because the current resolver falls back to configured/default workspace paths.

## Solution Assessment

Yes, the best fix is a doctor-owned repair for null agents.list[].workspace, not a startup guard or runtime shim. That matches the repository policy that runtime reads canonical config while doctor migrates malformed shipped/user config.
