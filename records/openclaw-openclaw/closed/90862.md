## Label Justifications

- `P2`: The report describes loss of cron run-history observability on a limited storage configuration, but the affected cron path is already fixed in current and stable code.
- `impact:data-loss`: The reported failure silently drops per-run cron history entries while keeping only the latest job state.

## Summary

Current main and the shipped stable release no longer write cron run history through the failing JSONL append path, so the reported cron data-loss mode is already fixed for OpenClaw core.

## What This Changes

The issue reports cron run-history files being created or truncated to zero bytes when appendRegularFile fchmod fails on root-owned mounts.

## Best Possible Solution

Keep the shipped SQLite cron run-log implementation; if cron history is still empty on a fresh 2026.6.1+ install, track that as a new SQLite persistence bug with concrete logs.

## Reproduction Assessment

No for current main: the old 2026.5.28 JSONL path is source-reproducible, but current main and v2026.6.1 write cron history to SQLite instead of appendRegularFile. I did not run a live Azure Files mount because the current source and shipped package no longer exercise the reported fchmod path for cron run logs.

## Solution Assessment

Yes: for this cron data-loss report, the SQLite persistence move is the best core fix because it removes the fragile JSONL chmod append path and keeps legacy file import in doctor migration code.
