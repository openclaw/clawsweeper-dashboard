## Summary

Keep open: current main still has the two Slack safety edges described here, and the report is distinct from the linked threading feature request because it is about idempotency and avoiding thread-to-channel leakage.

## What This Changes

The issue requests safer Slack inbound idempotency and fail-closed same-channel thread send handling to prevent duplicate agent turns and thread-to-channel leakage.

## Best Possible Solution

Keep this open and fix Slack with account-scoped durable/longer inbound idempotency plus a same-channel threaded-send guard that errors unless a thread is resolved or topLevel/threadId:null is explicit.

## Reproduction Assessment

Yes at source level: replaying the same Slack channel/timestamp after the 60s in-memory dedupe window or sending to the same Slack channel without currentThreadTs/currentChannelId reaches the reported fallback paths. I did not run a live Slack replay in this read-only review.

## Solution Assessment

Unclear without maintainer compatibility input: durable account-scoped dedupe is the right direction, but fail-closed message-tool routing should be accepted explicitly because it changes the behavior of contextless same-channel sends.
