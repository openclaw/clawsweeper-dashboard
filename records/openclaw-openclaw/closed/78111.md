## Summary

Keep this PR open: it targets a valid source-reproducible agent workspace regression, but the external contribution still needs real behavior proof and the proposed fallback formula has a path derivation concern before merge.

## What This Changes

The PR changes non-default agent workspace fallback resolution to derive `workspace-${id}` from the configured default workspace parent and updates agent-scope regression tests.

## Best Possible Solution

Land a narrow resolver fix that keeps the default agent honoring `agents.defaults.workspace` while giving non-default agents unambiguous sibling-style workspaces, backed by regression tests and a real CLI/routing transcript.

## Reproduction Assessment

Yes, from source inspection: current main derives a non-default fallback workspace as a child of the configured default workspace, and routing then treats paths inside that child as also inside the default workspace. I did not run a live CLI scenario because this review is read-only.

## Solution Assessment

No: the PR fixes the nested default-workspace case, but its `dirname(fallback)/workspace-${id}` derivation can move custom defaults to an unintended root-level sibling. A safer fix should either restore the state-dir `workspace-${id}` fallback or preserve the configured fallback basename when deriving the sibling.
