## Summary

The reported Telegram multi-account stale-socket restart problem is already fixed in current main and the fix was shipped after `2026.3.2`; the issue’s release-request is now obsolete.

## Evidence

- **Health policy no longer keys stale-socket on app events:** Current gateway health evaluation only considers `lastTransportActivityAt` for stale-socket detection, not `lastEventAt`, so idle channels are not restarted just because no user-visible events arrived.
  - file: [src/gateway/channel-health-policy.ts:115](https://github.com/openclaw/openclaw/blob/4d7c4b3298920c0e6e2c5cc8bd7a1546e8122abe/src/gateway/channel-health-policy.ts#L115)
  - sha: [4d7c4b329892](https://github.com/openclaw/openclaw/commit/4d7c4b3298920c0e6e2c5cc8bd7a1546e8122abe)
- **Telegram polling publishes heartbeat activity on empty polls:** Telegram polling status marks every successful `getUpdates` call as transport activity, with an inline comment stating that a successful long-poll proves the transport is alive even when there are no updates.
  - file: [extensions/telegram/src/polling-status.ts:20](https://github.com/openclaw/openclaw/blob/4d7c4b3298920c0e6e2c5cc8bd7a1546e8122abe/extensions/telegram/src/polling-status.ts#L20)
