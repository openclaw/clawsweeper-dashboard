## Summary

Keep this PR open. Current main still lacks LINE open-DM wildcard validation, while the LINE runtime treats `dmPolicy: "open"` as unconditional DM acceptance. The PR is a focused LINE-owned schema hardening change with targeted tests and no observed CI, dependency, package-resolution, lifecycle-script, or secret-handling changes. It still needs maintainer review for account-level inheritance semantics and migration/docs wording before landing.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is a LINE-owned config-schema hardening change that reuses `requireOpenAllowFrom`, adds focused top-level and account-level regression tests, aligns account-level validation with `resolveLineAccount` inheritance unless maintainers intentionally want stricter account-local explicitness, and adds clear migration/docs or changelog wording for the user-visible validation change before running the appropriate LINE extension checks.

## Risks / Open Questions

- The PR intentionally tightens validation and can reject existing hand-written LINE configs that used `dmPolicy: "open"` without `allowFrom: ["*"]`; migration wording and likely docs/changelog coverage should be added or confirmed.
- The current PR validates each account object in isolation, while runtime account resolution inherits top-level `allowFrom`; maintainers should decide whether parent `allowFrom: ["*"]` should satisfy account-level open mode, as it does for several other channels.
- Leaving this unmerged preserves a security-footgun on current main: an operator can write `dmPolicy: "open"` with a specific LINE `allowFrom` entry and still expose DMs to all senders.
