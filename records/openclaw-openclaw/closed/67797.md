## Summary

Current `main` already resolves default thinking from the runtime model catalog when the selected model is missing from `models.json` or lacks a `reasoning` flag, so OpenRouter models are no longer forced to `Think: off` just because only the cache knows they are reasoning-capable.

## Evidence

- **Runtime catalog hydration for thinking defaults:** `createModelSelectionState()` now reloads the runtime model catalog when the selected model is absent from the current catalog view or its `reasoning` field is missing, then passes that hydrated catalog into `resolveThinkingDefault()`. That directly covers the reported OpenRouter cache-vs-`models.json` gap.
  - file: [src/auto-reply/reply/model-selection.ts:289](https://github.com/openclaw/openclaw/blob/306c0f73bff1338482dd6533f61a86905bf541b1/src/auto-reply/reply/model-selection.ts#L289)
  - command: `nl -ba src/auto-reply/reply/model-selection.ts | sed -n '278,330p'`
  - sha: [306c0f73bff1](https://github.com/openclaw/openclaw/commit/306c0f73bff1338482dd6533f61a86905bf541b1)
- **Reasoning-capable models no longer fall back to off:** `resolveThinkingDefaultForModel()` now uses provider thinking profiles and, when a catalog entry is reasoning-capable, resolves to `medium` or the nearest supported non-`off` level instead of leaving the model at `off`.
  - file: [src/auto-reply/thinking.ts:230](https://github.com/openclaw/openclaw/blob/306c0f73bff1338482dd6533f61a86905bf541b1/src/auto-reply/thinking.ts#L230)

- **Regression coverage for reasoning-capable catalog defaults:** Unit tests on `main` assert that reasoning-capable catalog models default to `medium`, and binary/profile-specific providers remap that to the strongest supported non-`off` level such as `low`.
  - file: [src/auto-reply/thinking.test.ts:277](https://github.com/openclaw/openclaw/blob/306c0f73bff1338482dd6533f61a86905bf541b1/src/auto-reply/thinking.test.ts#L277)

## Risks / Open Questions

- The current mainline behavior and docs now use `medium` as the generic fallback for reasoning-capable models, so the exact expected value in the original report (`low`) is outdated even though the reported `off` behavior is fixed.
- The fix is on `main` only; users on release `v2026.4.23` may still see the old behavior until a newer release ships.
