## Summary

Close as implemented: current main now makes the macOS package-update restart path idempotent when a refreshed LaunchAgent is already serving the expected version, slows launchd respawn back to the platform default, and widens the Gateway bind retry window around supervisor-owned restarts.

## What This Changes

The issue requests hardening for macOS LaunchAgent update restarts so launchd KeepAlive respawn does not race the update restart and cause a port conflict on 18789.

## Best Possible Solution

Keep the merged lifecycle hardening on main and let the next release carry it; no additional core issue needs to remain open for the same update-cycle race.

## Reproduction Assessment

Yes at source level: the original race maps to the package-update service refresh followed by a redundant restart attempt, plus the LaunchAgent throttle and Gateway bind retry timing. I did not live-reproduce it under macOS launchd in this review.

## Solution Assessment

Yes. The current implementation uses the narrow maintainable path: detect the already-healthy updated Gateway before restarting again, slow launchd respawn to the documented default, and tolerate transient EADDRINUSE while the supervisor settles.
