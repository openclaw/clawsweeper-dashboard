## Summary

Keep this PR open. The direction matches the reported Control UI reconnect/Stop bug, but the latest diff still has a blocking `sessions.list` regression in the active-run plumbing, and the PR also has maintainer-protected handling signals.

## What This Changes

The PR adds Gateway WebSocket pings, exposes session active-run state, and uses that state so Control UI can keep Stop and session-scoped aborts available after reconnect or reload.

## Best Possible Solution

Keep the PR direction, but repair the active-run lookup to use the request context registry and stored session keys, then land with focused gateway/UI tests and the existing changelog entry.

## Reproduction Assessment

Yes. The PR diff itself gives a high-confidence reproduction: invoke `sessions.list` after this patch and it reaches `chatHandlers.chatAbortControllers`, which is not the active-run registry and will fail before returning rows.

## Solution Assessment

No. The backend/UI model is the right narrow approach, but this implementation uses the wrong object for active abort state; a context-scoped lookup is the safer maintainable fix.
