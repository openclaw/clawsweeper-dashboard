## Summary

Keep open: current main still launches the bundled Claude CLI with a strict generated MCP config and an OpenClaw-only `--allowedTools` pattern, while the requested user/project MCP access would relax a deliberate security boundary and needs maintainer design review.

## What This Changes

The issue asks for opt-in Claude CLI access to user/project MCP servers beyond the OpenClaw loopback when running `openclaw agent --agent claude`.

## Best Possible Solution

Define an explicit opt-in contract that preserves loopback-only isolation by default, uses OpenClaw `mcp.servers` or approved imported Claude MCP entries as the supported registry, and derives matching Claude allowlist patterns only for approved servers.

## Reproduction Assessment

Yes, at source level: resolving the bundled `claude-cli` backend produces `--allowedTools mcp__openclaw__*`, and the bundle-MCP injector appends a strict generated MCP config. I did not run a live Claude CLI session in this read-only review.

## Solution Assessment

Unclear: the proposed knobs are plausible, but the best contract should be chosen by maintainers because this changes a deliberately strict Claude MCP trust boundary. A safer direction is explicit opt-in allowlisting with regression tests for default isolation and enabled non-OpenClaw MCP access.
