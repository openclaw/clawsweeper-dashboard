## Summary

Current OpenClaw already treats the reported provider names as canonical bundled plugin ids, auto-enable intentionally writes those plugin ids into `plugins.entries`/`plugins.allow`, and validation resolves them from the manifest registry instead of rejecting them. The reported hard `plugin not found` failure does not match current `main` or the current release snapshot inspected here.

## Evidence

- **Bundled manifests define the reported ids as plugin ids:** A direct manifest scan finds `anthropic`, `minimax`, `moonshot`, `nvidia`, `openrouter`, `ollama`, `openai`, and `zai` as bundled plugin `id` values under `extensions/*/openclaw.plugin.json`.
  - command: `rg -n '"id"\s*:\s*"(openai|anthropic|ollama|openrouter|minimax|moonshot|nvidia|zai)"' extensions/*/openclaw.plugin.json`
  - sha: [250acdd2a28c](https://github.com/openclaw/openclaw/commit/250acdd2a28c190ed7934e3ba99022f24123d3c9)
- **Manifest contract uses plugin id for plugins.entries:** The manifest docs define `id` as the canonical plugin id used in `plugins.entries.<id>`, and document `autoEnableWhenConfiguredProviders` / `providers` as the provider-to-plugin ownership metadata.
  - file: [docs/plugins/manifest.md:136](https://github.com/openclaw/openclaw/blob/250acdd2a28c190ed7934e3ba99022f24123d3c9/docs/plugins/manifest.md#L136)
  - sha: [250acdd2a28c](https://github.com/openclaw/openclaw/commit/250acdd2a28c190ed7934e3ba99022f24123d3c9)
- **Auto-enable materializes owning plugin ids:** Provider auto-enable resolves configured providers/models to owning plugin ids, then writes `plugins.entries[pluginId].enabled = true` and allowlists that same `pluginId`. This is the intended control-plane behavior, not an invalid alias write.
  - file: [src/config/plugin-auto-enable.shared.ts:470](https://github.com/openclaw/openclaw/blob/250acdd2a28c190ed7934e3ba99022f24123d3c9/src/config/plugin-auto-enable.shared.ts#L470)
  - sha: [250acdd2a28c](https://github.com/openclaw/openclaw/commit/250acdd2a28c190ed7934e3ba99022f24123d3c9)
- **Validation accepts discoverable plugin ids from the manifest registry:** Config validation loads the plugin manifest registry, builds `knownIds` from `registry.plugins.map(record => record.id)`, and uses that set to validate plugin references.
  - file: [src/config/validation.ts:825](https://github.com/openclaw/openclaw/blob/250acdd2a28c190ed7934e3ba99022f24123d3c9/src/config/validation.ts#L825)
  - sha: [250acdd2a28c](https://github.com/openclaw/openclaw/commit/250acdd2a28c190ed7934e3ba99022f24123d3c9)
- **Unknown plugins.entries are stale-entry warnings, not the reported hard failure:** For `plugins.entries.*`, missing ids are now downgraded to warning-only stale entries instead of startup-blocking errors, which directly contradicts the reported current-state failure mode.
  - file: [src/config/validation.ts:1088](https://github.com/openclaw/openclaw/blob/250acdd2a28c190ed7934e3ba99022f24123d3c9/src/config/validation.ts#L1088)
  - sha: [250acdd2a28c](https://github.com/openclaw/openclaw/commit/250acdd2a28c190ed7934e3ba99022f24123d3c9)
- **The current release snapshot already contains the same validation behavior:** The latest release commit inspected for `v2026.4.23` contains the same `plugins.allow`/`plugins.entries` manifest-id validation path, so this is not just unreleased main-branch behavior.
  - file: [src/config/validation.ts:1056](https://github.com/openclaw/openclaw/blob/a9797214338ba31c52c796adbb75afb16e0684a9/src/config/validation.ts#L1056)
  - sha: [a9797214338b](https://github.com/openclaw/openclaw/commit/a9797214338ba31c52c796adbb75afb16e0684a9)
