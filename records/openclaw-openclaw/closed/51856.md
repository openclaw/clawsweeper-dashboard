## Summary

Current `main` already fixes the reported `message.send` misclassification by treating empty/default poll-creation fields as unset, and the same code is present in release `v2026.4.22`, so this PR is obsolete.

## Evidence

- **Current send validation already ignores empty/default poll creation values:** `hasPollCreationParams()` only treats non-empty strings, string arrays with at least one non-empty entry, non-zero finite numbers, and boolean `true` as poll intent. That means `pollQuestion=""`, `pollOption=[]`, `pollDurationHours=0`, and explicit false booleans do not trigger poll validation anymore.
  - file: [src/poll-params.ts:49](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/src/poll-params.ts#L49)
  - sha: [aa3a5f9ff790](https://github.com/openclaw/openclaw/commit/aa3a5f9ff7907c2a41d6259d24b529bda41f293a)
- **`message.send` uses that helper directly on the live path:** `runMessageAction()` only throws the poll-only error when `action === "send"` and `hasPollCreationParams(params)` returns true, so the current main code path already avoids the false positive described in the PR body.
  - file: [src/infra/outbound/message-action-runner.ts:927](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/src/infra/outbound/message-action-runner.ts#L927)

- **Regression coverage exists for the default-value case:** The current test suite explicitly asserts that zero-valued numeric poll params and false booleans are not treated as poll creation intent.
  - file: [src/poll-params.test.ts:37](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/src/poll-params.test.ts#L37)
