The branch replaces Nextcloud Talk webhook signature comparison with padded `crypto.timingSafeEqual`, adds focused signature/header regression coverage, and adds a changelog entry.

## Best Possible Solution

Land the PR after the remaining checks complete and a maintainer is comfortable with the security hardening. A later cleanup can centralize this on `safeEqualSecret`, but the current patch is a narrow, maintainable fix for the reported behavior.

## Reproduction Assessment

Yes. On current main, a non-empty supplied signature whose length differs from the SHA-256 hex digest exits at `extensions/nextcloud-talk/src/signature.ts:28`; the PR adds a test that verifies the compare path still runs for a truncated signature.

## Solution Assessment

Yes. Padding both buffers before `timingSafeEqual` and checking the original lengths afterward is the narrow fix, matches the existing LINE/shared-secret pattern, and preserves boolean behavior for invalid signatures.
