## Summary

Current main already handles `[[reply_to_current]]` without a Telegram-specific null-drop path, and the observed group no-send behavior is explained by the current group/channel visible-reply policy: ordinary group final replies are private unless `messages.groupChat.visibleReplies` is set to `automatic`. Main also logs and documents that policy/config path, so the report does not identify remaining source work for the default group behavior.

## What This Changes

The issue reports Telegram group inbound agent replies being dropped when final output starts with `[[reply_to_current]]`, while CLI `--deliver` still sends to the same group.

## Best Possible Solution

Keep the current explicit group visibility model: default group final replies stay private, `message(action=send)` is used for room output, and operators who want legacy behavior set `messages.groupChat.visibleReplies` to `automatic`. A future report should focus on configured-automatic Telegram groups if they still fail to emit `sendMessage`.

## Reproduction Assessment

Yes for the observable no-send symptom as an expected policy path: current tests reproduce default group/channel final-reply suppression and the automatic opt-in. No high-confidence reproduction remains for the hypothesized `[[reply_to_current]]` null-resolution drop in current main because the source maps or strips the directive before delivery.

## Solution Assessment

Yes; the best current solution is the existing config-controlled visibility policy rather than silently sending group final replies as plain messages and bypassing the room-output safety model. If automatic mode is configured and still drops, the safer fix would belong in the Telegram delivery path with a focused regression test.
