## Summary

Keep this PR open for maintainer and security review. Current main does not implement the requested override, and the proposed patch likely does not yet deliver the advertised `exec` loopback-MCP use case because the loopback resolver filters a catalog that does not include core native coding tools.

## What This Changes

The PR filters loopback MCP `excludeToolNames` through `gateway.tools.allow` and adds a unit test for allowing `exec` while preserving explicit `deny` precedence.

## Best Possible Solution

Implement an explicit loopback MCP native-tool opt-in that actually materializes the sandbox-aware native tools only when configured, preserves explicit `deny` precedence, and updates the public docs, schema/help text, and security audit coverage for the new exposure model.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: current main’s loopback path passes `excludeToolNames`, but the resolver starts from `createOpenClawTools`, while native `exec` is created in `createOpenClawCodingTools`; the PR’s added test does not exercise that path.

## Solution Assessment

No. Filtering the exclusion set is a necessary piece, but it is not sufficient to make core native tools available over loopback MCP and it widens a security-sensitive config contract without matching docs/audit changes.

## Review Findings
