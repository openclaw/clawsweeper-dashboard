## Summary

Current main has a harness-aware auth path for native Codex runs, so the reported `openai:manual` forwarding mismatch should no longer occur. The fix is present on main after `v2026.5.2`, with targeted regression coverage, but I could not prove it has shipped in a release yet.

## What This Changes

Ensure native Codex runtime runs with `openai/gpt-*` select a compatible `openai-codex:*` auth profile instead of forwarding generic `openai:*` API/manual auth.

## Best Possible Solution

Keep the current main harness-aware auth selection and ship it in the next release; no separate open core issue is needed for this exact mismatch.

## Reproduction Assessment

No. Current main should not reproduce the failure because the source path now auto-selects `openai-codex` auth for `openai/gpt-*` Codex harness runs, and the focused test asserts that exact case.

## Solution Assessment

Yes. The current fix is the narrow maintainable path: make auth forwarding aware of the selected Codex harness while preserving normal `openai/*` API-key behavior outside that harness.
