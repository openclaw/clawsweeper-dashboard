- `P2`: This is a normal-priority channel lifecycle bug with a clear source path and limited blast radius to WhatsApp hot reload behavior.
- `impact:session-state`: A disabled WhatsApp account can keep its live session running after reload, leaving runtime state out of sync with config.
- `impact:other`: Operators currently need a full gateway restart to soft-disable a running WhatsApp account, causing avoidable channel downtime.

## Summary

This issue should stay open: current main still treats all `channels.whatsapp.*` changes as dynamic-read no-ops, so flipping an account `enabled` flag does not put WhatsApp in the channel-restart path that would stop the running socket.

## What This Changes

Hot config reload should stop an already-running WhatsApp account when `channels.whatsapp.accounts.<name>.enabled` changes from true to false.

## Best Possible Solution

Add a focused hot-reload lifecycle path for WhatsApp account enablement changes that stops disabled running accounts, keeps credentials on disk, and preserves dynamic reads for non-lifecycle WhatsApp config.

## Reproduction Assessment

Yes, by source inspection: `channels.whatsapp.accounts.<id>.enabled` matches the WhatsApp no-op reload prefix, and no channel restart/stop is scheduled. I did not live-send a WhatsApp DM in this read-only review.

## Solution Assessment

Yes, a focused account-lifecycle hot reload is the maintainable fix. A broad gateway restart or making all WhatsApp config hot-restart would solve the symptom but risks unnecessary churn for config that is intentionally read dynamically.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/87951. Current main classifies `channels.whatsapp.accounts.<id>.enabled` changes as no-op dynamic reads because the WhatsApp plugin declares `noopPrefixes: ["channels.whatsapp"]`, so a running account is not stopped when enabled flips to false. Add a focused reload/lifecycle path that stops affected running WhatsApp accounts on true-to-false enablement changes without deleting auth credentials, and does not broadly restart WhatsApp for unrelated dynamic-read config. Prefer a narrow generic channel/account reload rule only if it preserves existing plugin boundaries; otherwise keep the behavior plugin-owned. Add regression coverage for the reload plan/handler and channel manager behavior. Do not edit CHANGELOG.md.
