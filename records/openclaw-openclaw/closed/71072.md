## Summary

Keep this PR open. Current main still preserves the final empty-registry result for `onlyPluginIds: []`, but it still enters cache/reentry setup, discovery, and manifest-registry loading before filtering candidates, so it does not implement the requested O(1) empty-scope fast path. The latest PR head is still the active implementation candidate, but its fast path returns before `pluginLoaderCacheState.beginLoad(cacheKey)`, so it should be revised to preserve the existing same-key reentry contract before merge. Security review found no workflow, dependency, lockfile, package-script, secret-handling, or artifact-download changes.

## Best Possible Solution

Keep this PR open and revise it in place: add the explicit empty-scope fast path while preserving the existing cache reentry contract, activation-clearing side effects, and the distinction between `undefined` and `[]`. The focused test should prove discovery and manifest registry loading are skipped, and a small regression should cover the activating/reentrant empty-scope path. No docs or changelog entry appears necessary unless maintainers want release-note visibility for startup performance.
