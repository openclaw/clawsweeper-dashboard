## Summary

Keep open. Current main now handles the WhatsApp voice-note delivery path for auto-TTS and TTS tool output, including Ogg/Opus transcoding and Baileys PTT send behavior, but the issue explicitly asks for product UX that is still missing: read-this/read-latest actions, per-chat voice toggle, selected-reply idempotency, and optional confirmation. A maintainer comment on the issue also explicitly says to keep this open for that remaining UX.

## Best Possible Solution

Keep #66032 open as the product UX follow-up. The best fix is a first-class on-demand TTS request flow for messaging channels, likely built on the existing TTS provider and WhatsApp voice-note delivery path, with explicit selection of latest/current assistant reply, a stable idempotency key for that selected reply, per-chat voice preferences, and tests covering duplicate request suppression and WhatsApp delivery.
