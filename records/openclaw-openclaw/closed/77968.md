## Summary

Current main still has the reported cache lifetime in the core context-engine loop hook, so this should stay open as an actionable narrow bug rather than be closed as stale or external.

## What This Changes

The issue asks OpenClaw core to invalidate `installContextEngineLoopHook`'s cached assembled context when live messages shrink or context assembly fails.

## Best Possible Solution

Invalidate the cached assembled view inside the core hook when the source array shrinks or engine processing fails, and add focused regression coverage without adding extension-specific reset policy or new config.

## Reproduction Assessment

Yes, at the helper/source level: a cached assembled view can be returned after `sourceMessages.length` no longer exceeds the fence, and the cache is not cleared after assembly failure. I did not run a live gateway reset scenario in this read-only review.

## Solution Assessment

Yes, the proposed direction is the narrowest maintainable fix because `installContextEngineLoopHook` owns the cache and already has the source-message fallback boundary.

Fix stale `lastAssembledView` caching in `installContextEngineLoopHook` for this issue. In `src/agents/pi-embedded-runner/tool-result-context-guard.ts`, track the source message length used to build the cached assembled view, clear the cached view and marker when the current source shrinks before the no-new-messages short-circuit, and clear both when `afterTurn`/ingest/`assemble` processing throws or when the engine returns the original source array. Preserve the existing fast path that reuses an assembled view for the same unchanged source. Add focused regression tests in `src/agents/pi-embedded-runner/tool-result-context-guard.test.ts` for source shrink after a cached assembled view and for an `assemble()` failure after a cached view followed by an unchanged call. Add a user-facing `CHANGELOG.md` fix entry for this issue. Do not add lossless-claw-specific hooks, plugin IDs, config options, or broader session reset policy changes.
