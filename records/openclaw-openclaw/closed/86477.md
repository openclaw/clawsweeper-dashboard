## Summary

Current main already handles the central reported case: `process poll` resolves the active and finished session maps before waiting, returns `No session found for <id>` when neither exists, and the same behavior is present in the latest release.

## What This Changes

The issue requests that `process poll` return an immediate error for a non-existent process session ID instead of waiting for the requested timeout.

## Best Possible Solution

Keep the current process-tool contract: missing IDs fail immediately, finished IDs report completion, and any separate stale-active-session hang should be filed with `process list`, `process log`, and gateway exit evidence.

## Reproduction Assessment

No. Current-main source gives a high-confidence negative check: a truly missing `sessionId` returns before the poll wait loop, and a finished backgrounded session is read from the finished-session map.

## Solution Assessment

Yes. The current implementation is the narrow maintainable behavior for this report; a distinct stale-active-session problem would need a separate reproduction and targeted fix.
