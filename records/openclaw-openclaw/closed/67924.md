## Summary

Current main and the latest release already avoid the reported no-approval exec false positive by skipping script-file preflight for gateway `security=full` plus `ask=off`, and docs/tests now state and cover that behavior. This PR is therefore obsolete for the linked reproduction, and its custom string scanner has unresolved security-sensitive parsing concerns for JS comments/regex literals.

## Best Possible Solution

Close this PR as obsolete because current main and `v2026.4.24` already solve the linked `security=full` plus `ask=off` user-visible false positive by skipping script-file preflight in that mode. Keep the shipped behavior, and open or use a separate maintainer-scoped issue if approval-mode preflight needs a safer language-aware implementation.

## Risks / Open Questions

- If maintainers want to improve approval-mode script preflight, current main can still flag `$VAR` inside Python/Node string literals when preflight is active; that should be tracked as a narrower parser-backed follow-up rather than landing this stale PR.
- The PR's proposed JS/Python quote-state scanner is security-sensitive because misclassifying comments or regex literals as open strings can suppress a real shell-bleed token outside a string.
