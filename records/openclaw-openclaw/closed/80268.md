## Summary

Keep open: current main and the latest release still expose Chrome MCP existing-session status as transport readiness only, and the open fix PR at https://github.com/openclaw/openclaw/pull/80280 is the right vehicle to review or land before this issue closes. The broader page-tool timeout root cause remains tracked separately at https://github.com/openclaw/openclaw/issues/80036.

## What This Changes

Request that Chrome MCP existing-session browser status expose a separate page-tool readiness signal instead of only transport readiness fields.

## Best Possible Solution

Land or replace the focused additive status-surface fix from https://github.com/openclaw/openclaw/pull/80280, while leaving the broader Chrome MCP page-tool timeout investigation to https://github.com/openclaw/openclaw/issues/80036.

## Reproduction Assessment

Yes, at source level: the current test exercises transport-up/page-reachability-down and current main still returns `running: true` and `cdpReady: true` without any page-readiness field. I did not live-run a macOS Chrome existing-session profile.

## Solution Assessment

Yes, the narrow direction is a separate additive readiness field backed by a bounded page-tool probe, rather than overloading the existing transport-oriented `cdpReady` field. The linked PR is the current implementation candidate and should be reviewed or landed before this issue closes.
