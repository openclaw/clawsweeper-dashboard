## Summary

Keep open for maintainer review: the fix is focused and supported by source/tests/proof, but it intentionally changes malformed config paths from permissive normalization to fail-closed errors, so the compatibility tradeoff should be accepted before merge.

## What This Changes

The PR tightens `src/cli/config-cli.ts` path parsing to reject empty or whitespace-only config path segments and adds regression coverage for malformed paths plus preserved bracket and escaped-dot forms.

## Best Possible Solution

Land the parser hardening and regression tests once a maintainer accepts the fail-closed compatibility tradeoff for malformed CLI config paths.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current `parsePath` drops empty pieces and `normalizeStringEntries` filters them before `config set/get/unset` acts. I did not run the CLI because this review was read-only.

## Solution Assessment

Yes, with maintainer acceptance of the compatibility tradeoff: rejecting malformed segments before reads or writes matches the sibling parser and prevents wrong-key mutation. The safer alternative would be a compatibility window only if maintainers consider typo-tolerant paths a shipped CLI contract.
