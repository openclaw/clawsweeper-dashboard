## Summary

Current main has moved Gateway RPC clients onto an event-loop-readiness gate before opening the WebSocket, so heavy module evaluation or package-resolution stalls no longer occur inside the connect.challenge signing window. The reported cron path also no longer preloads plugins, and the nonce-signing path is synchronous once the challenge is received.

## What This Changes

The issue reports that upgraded CLI commands can hang for about 30 seconds after receiving connect.challenge, apparently looping over package.json resolution before signing and sending the nonce.

## Best Possible Solution

Keep the current main behavior: Gateway RPC clients should only open the WebSocket after event-loop readiness, CLI read-only commands should stay off plugin-runtime preload paths, and plugin-registry repair should remain in doctor/plugins registry paths for bad local state.

## Reproduction Assessment

Yes. The report gives a concrete strace-backed reproduction around openclaw cron list, and current source/test coverage directly exercises the challenge-to-signed-connect path plus the cron no-plugin-preload policy; I did not run a live poisoned-state replay in this read-only review.

## Solution Assessment

Yes. The current fix is the narrow maintainable direction because it prevents local module-resolution stalls from consuming the pre-auth challenge window without adding core startup migrations or special-casing cron.
