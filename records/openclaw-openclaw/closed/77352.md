## Summary

Current main does not match the reported root cause: active plugin loading binds `api.registerCommand()` through the full plugin registry API, while the unbound `loader.ts` path is the non-activating CLI metadata path only.

## What This Changes

The issue asks OpenClaw to bind `registerCommand` in the plugin loader so `api.registerCommand()` registrations populate the runtime command registry.

## Best Possible Solution

Keep the current split: active runtime loads bind `registerCommand` for dispatch, while `cli-metadata` remains limited to root CLI descriptor capture.

## Reproduction Assessment

No. Current-main source inspection does not reproduce the asserted failure: active loads use the full registry API with `registerCommand`, and checked-in tests assert active `api.registerCommand()` entries appear in command specs.

## Solution Assessment

Not applicable as a code fix for the reported root cause. If a separate CLI metadata feature is desired, it should be scoped to CLI descriptor capture rather than runtime command dispatch.
