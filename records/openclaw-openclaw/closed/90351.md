## Best Possible Solution

Land this with the focused tests and Mantis proof once maintainers accept the autogenerated key-shape and unresolved-retry dedupe semantics; otherwise pause for a narrower delivery policy decision.

## Reproduction Assessment

## Solution Assessment

Yes, if maintainers accept the semantics. The message-tool layer is the narrow owner because it generates keys before both gateway `message.action` and core send consume them; channel-specific Telegram dedupe would be narrower and would not address the shared message-tool idempotency gap.
