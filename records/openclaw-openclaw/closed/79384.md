## Summary

Keeping this open: current main still documents and accepts `tools.web.search.maxResults`, but the agent-facing gateway mutation guard excludes that path and rejects non-allowlisted config changes. The related fix PR at https://github.com/openclaw/openclaw/pull/79391 closed unmerged, and the remaining change touches a deliberate control-plane trust boundary.

## What This Changes

Requests making `tools.web.search.maxResults` adjustable through the agent-facing `gateway config.patch` path for native `web_search` tuning beyond the current default of 5.

## Best Possible Solution

Have a maintainer decide whether web-search result tuning belongs in the agent-tunable `config.patch` allowlist; if approved, add the narrow allowlist entry, regression coverage, and provider-default coverage for SearXNG.

## Reproduction Assessment

Yes from source inspection: a patch changing `tools.web.search.maxResults` is outside the current gateway tool allowlist and would be rejected before the gateway call is forwarded. I did not run a live gateway command in this read-only review.

## Solution Assessment

Unclear: the closed related PR is directionally narrow for `config.patch`, but the full user problem may also require SearXNG to honor the configured default. The safer path is policy approval plus focused allowlist and provider regression tests.
