## Summary

Current `main` now exposes session lifecycle state through the new session index APIs/events, which gives operators and heartbeat/healing agents a concrete way to detect stale sessions without relying on `last_active` alone.

## Close Comment

Closing as implemented on `main` after Codex review. Current `main` now exposes session lifecycle state in the session index instead of relying on `last_active` alone: `GatewaySessionRow` includes `status`, `startedAt`, `endedAt`, and `runtimeMs` (`src/gateway/session-utils.types.ts:16`), lifecycle events map sessions to `running` / terminal states including `timeout` (`src/gateway/session-lifecycle-state.ts:39`), and the gateway persists plus broadcasts those updates through `sessions.changed` on run start/end (`src/gateway/server-chat.ts:505`, `src/gateway/server-chat.ts:649`, `src/gateway/server-chat.ts:1004`). The public session control/docs are also present on `main` (`docs/gateway/protocol.md:338`, `docs/gateway/protocol.md:393`). That gives heartbeat/healing flows the missing stale-session signal this issue was asking for, even though the shipped path is session lifecycle observability rather than a separate `gateway.responseTimeout` notifier config.
