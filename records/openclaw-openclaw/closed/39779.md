## Summary

Current `main` already fixes the reported localhost-vs-127.0.0.1 failure by binding loopback gateways on both `127.0.0.1` and `::1` when IPv6 loopback is available, so `ws://localhost:...` can reach the same gateway even if `localhost` resolves to IPv6.

## Evidence

- **Loopback bind now expands to IPv4 + IPv6:** `resolveGatewayListenHosts()` returns `["127.0.0.1", "::1"]` when the configured bind host is `127.0.0.1` and IPv6 loopback is available.
  - file: [src/gateway/net.ts:398](https://github.com/openclaw/openclaw/blob/3a5e535beddd67a2adb3dca02a83e27c239b0c7f/src/gateway/net.ts#L398)

## Risks / Open Questions

- The fix appears to be on `main` after the latest tagged release `v2026.4.22`, so users on older releases may still hit the bug until a later release ships this change.
- The UI still derives its default URL from `location.host`; the issue is resolved by the gateway listening on `::1` as well, not by a UI fallback to `127.0.0.1`.
