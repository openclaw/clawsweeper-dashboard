## Summary

Keep open: this is a focused candidate fix for the linked doctor auth data-loss bug, and current main still has the destructive repair path. I found no blocking correctness or security finding, but the main merge-readiness gap is the lack of in-tree regression coverage for the two-profile collision case.

## What This Changes

The PR adds a guard in `repairOAuthProfileIdMismatch` to skip legacy OAuth profile-id repair when the suggested destination profile already exists, and passes config into destination metadata resolution.

## Best Possible Solution

Land the focused guard with a regression test or accepted upgrade proof that the collision case preserves both profiles while the existing single-profile migration still works.

## Reproduction Assessment

Yes, at source level: current main selects the sole non-legacy OAuth profile and then overwrites `nextProfiles[toProfileId]`; https://github.com/openclaw/openclaw/issues/97522 also includes direct repair-function proof. I did not run the CLI in this read-only review.

## Solution Assessment

Yes, the guard is the narrowest safe owner-boundary fix because doctor auth repair already owns this migration and skipping avoids destructive account merging. A non-destructive merge policy would be broader and needs more product judgment.
