## Label Justifications

- `P2`: The report describes a bounded Telegram multi-account delivery/session routing failure, not a global gateway outage.
- `impact:message-loss`: The observable failure is generated Telegram responses not reaching the user.
- `impact:session-state`: The claimed root cause is ambiguous persisted Telegram session identity and account metadata.

## Summary

Current main and the latest tagged release already cover the central multi-account Telegram DM session identity problem: named non-default Telegram accounts get account-qualified direct-session keys, and inbound session metadata carries the account id into delivery routing.

## What This Changes

The issue requests that Telegram session identity include the account id so outbound delivery can disambiguate multiple Telegram accounts sharing the same chat id.

## Best Possible Solution

Keep the shipped account-qualified named-account DM routing and have remaining reports test against `v2026.5.18` or use the documented `per-account-channel-peer` scope for explicit multi-account DM isolation.

## Reproduction Assessment

No, not as a current-main failure: the source and regression fixtures now show distinct account-qualified session keys for named Telegram accounts sharing a DM chat. I did not run a live Telegram gateway in this read-only review.

## Solution Assessment

Yes; the current implementation fixes the maintainable boundary by preserving account identity in the Telegram routing/session layer and by documenting `per-account-channel-peer` for explicit multi-account DM scopes.
