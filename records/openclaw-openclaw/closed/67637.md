## Summary

Keep open. Current main still does not expose actual agent tool-call traces on `QaBusMessage`; the QA bus contract, persistence/query paths, wait helpers, and qa-channel delivery path still carry text/metadata/attachments/reactions only, while mock-provider `/debug/requests` remains an adjacent workaround rather than the requested bus-level trace surface.

## What This Changes

The issue requests an optional `QaBusMessage.toolCalls`-style trace surface so qa-lab scenarios can assert that agents actually invoked expected tools.

## Best Possible Solution

Define an additive, sanitized QA trace surface sourced from actual tool-start events, then carry it through QA bus create, clone, snapshot, read, search, poll, wait, and qa-channel delivery paths with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main is enough to reproduce the gap: sanitized tool-start events exist internally, but `QaBusMessage` and the QA bus delivery/query paths do not expose them.

## Solution Assessment

Unclear. The proposed optional `QaBusMessage.toolCalls` field is plausible, but maintainers should first decide the public SDK shape versus a narrower qa-lab-only API and set the redaction policy.
