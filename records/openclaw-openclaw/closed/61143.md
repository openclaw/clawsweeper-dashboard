## Summary

Keep open. Current `main` still lacks the Signal-side MIME fallback this PR proposes, and the existing Signal tests on `main` still encode the old `application/octet-stream` behavior when `contentType` is missing. A related downstream AAC upload fix has landed, but this inbound Signal classification gap is still present in core code.

## Evidence

- **Signal monitor still does not pre-resolve MIME or pass filename:** `fetchAttachment` on current `main` still imports only `saveMediaBuffer` from the media runtime and calls it with `attachment.contentType ?? undefined`; it does not call `detectMime`, and it does not forward `attachment.filename`/`originalFilename`. That means the PR's core change is not present on `main`.
  - file: [extensions/signal/src/monitor.ts:10](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/extensions/signal/src/monitor.ts#L10)
  - command: `nl -ba extensions/signal/src/monitor.ts | sed -n '1,40p' && nl -ba extensions/signal/src/monitor.ts | sed -n '270,330p'`
  - sha: [59523e66da03](https://github.com/openclaw/openclaw/commit/59523e66da03591bc7bcc183a11f690fa0aae641)
- **`saveMediaBuffer` still cannot use attachment filename to detect MIME:** The current `saveMediaBuffer` implementation accepts `originalFilename`, but MIME detection still only uses `{ buffer, headerMime: contentType }`. Without the outer Signal fix, a missing Signal `contentType` still leaves no `filePath`-based fallback for `.aac` voice notes.
  - file: [src/media/store.ts:407](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/src/media/store.ts#L407)
