## Summary

Keep open. Current main still preserves the stricter #47487 tool-profile behavior, but the reported silent cron UX gap remains source-backed: the mismatch only logs at runtime, config validation and doctor preview do not surface it, and the cron denial classifier still misses the unavailable-tool wording from this report.

## What This Changes

The issue asks OpenClaw to surface restrictive `messaging` profile plus configured `exec`/`process` expectations so isolated cron runs do not report tool-policy failures as successful deliveries.

## Best Possible Solution

Preserve strict tool-profile authorization, add shared diagnostics for configured tool sections that are not granted by the active restrictive profile, and classify unavailable-tool cron outputs as failed runs when no successful result follows.

## Reproduction Assessment

Yes, source-level. `tools.profile: "messaging"` plus `tools.exec` leaves `profileAlsoAllow` undefined, `config validate` and doctor preview do not surface that mismatch, and the cron classifier lacks the unavailable-tool wording shown in the report.

## Solution Assessment

Yes. The narrow maintainable fix is louder diagnostics plus targeted cron fatal classification, not restoring the implicit `exec`/`process` grants removed for #47487.

Fix openclaw/openclaw#75763. Preserve the strict #47487 behavior: do not implicitly add `exec`, `process`, `read`, `write`, or `edit` from configured `tools.exec`/`tools.fs`. Add a shared diagnostic for restrictive `tools.profile` or `agents.list[].tools.profile` values that have configured `tools.exec` or `tools.fs` sections but no matching explicit `alsoAllow`; surface it through `openclaw config validate` and doctor preview. Update isolated cron result classification so unavailable-tool/tool-policy outputs like "doesn't have `exec` available", tool-not-available loop-guard text, or the reported not-runnable tool-policy failure mark the run as an error when no later successful output exists. Add focused regression tests and a user-facing changelog entry. Keep the patch scoped to diagnostics/status classification, not authorization widening.
