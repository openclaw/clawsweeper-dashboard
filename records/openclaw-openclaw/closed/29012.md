## Summary

Current `main` already covers this use case: slash-command authorization is configurable per provider via `commands.allowFrom`, native command registration can already be overridden per channel where supported, and unauthorized whole-message slash commands are ignored instead of executed.

## Evidence

  - file: [src/config/types.messages.ts:132](https://github.com/openclaw/openclaw/blob/59a8afe6fa030f18af662e125e7dd826ce9a53ca/src/config/types.messages.ts#L132)

  - file: [src/auto-reply/command-auth.ts:313](https://github.com/openclaw/openclaw/blob/59a8afe6fa030f18af662e125e7dd826ce9a53ca/src/auto-reply/command-auth.ts#L313)

## Close Comment

Closing as implemented on `main` based on Codex review. Current `main` already supports the requested behavior: `commands.allowFrom` is now a per-provider slash-command allowlist (`src/config/types.messages.ts:132`), command authorization resolves provider-specific entries like `telegram` vs `whatsapp` before falling back (`src/auto-reply/command-auth.ts:313`), and unauthorized standalone control commands are ignored instead of being executed (`src/auto-reply/reply/get-reply-run.ts:338`). The config reference also documents both per-channel native-command overrides and per-provider command authorization (`docs/gateway/configuration-reference.md:860`). This means you can keep owner command access on one channel while blocking command use on customer-facing WhatsApp conversations without adding a new per-agent config shape. This appears to be on `main` via `59a8afe6fa030f18af662e125e7dd826ce9a53ca`; I could not verify a released tag containing it yet, so leaving `fixedRelease` unset.
