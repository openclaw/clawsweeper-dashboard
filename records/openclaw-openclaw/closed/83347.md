## Summary

Keep open: current main still narrows OpenClaw-sandboxed Codex app-server turns from `danger-full-access` to `workspace-write`, and the resulting Codex turn policy sends `networkAccess: false`, matching the reported loss of outbound access. https://github.com/openclaw/openclaw/pull/82452 is a relevant open fix path, but it is not merged and still needs proof plus maintainer review of the sandbox egress boundary.

## What This Changes

The issue asks for sandboxed Codex agents to retain outbound network access when OpenClaw sandbox egress permits it, without disabling the outer OpenClaw sandbox.

## Best Possible Solution

Land or revise a focused fix that carries OpenClaw sandbox egress intent into Codex turn policy, keeps default `workspace-write` network-denied, and documents the trust boundary for outer-sandbox-controlled network access.

## Reproduction Assessment

Yes, source-reproducible: current main clamps an active OpenClaw-sandboxed Codex `danger-full-access` turn to `workspace-write`, then builds a Codex `workspaceWrite` policy with `networkAccess: false`. I did not run the reporter's live DNS or Google API flow.

## Solution Assessment

Unclear until maintainer review: https://github.com/openclaw/openclaw/pull/82452 is a narrow code direction, but the sandbox egress policy needs accepted real-behavior proof and owner approval before merge.
