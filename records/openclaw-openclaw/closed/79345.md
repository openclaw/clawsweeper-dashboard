## Summary

Keep open: current main and the latest release still route Telegram named-account token resolution through the caller-supplied config instead of selecting the active runtime snapshot, so a SecretRef-backed account can look configured in read/status paths while runtime token resolution returns no token or throws. The issue also remains security-sensitive because the original report exposed a Telegram Bot Token and the body is now redacted.

## What This Changes

The issue requests a fix for Telegram multi-account SecretRef token resolution when status/read paths show accounts but runtime token lookup uses unresolved source config.

## Best Possible Solution

Rotate or revoke the exposed Telegram credential, then add a Telegram-owned runtime-config selector before token/account credential reads with regression coverage for SecretRef-backed named accounts against source plus runtime snapshots.

## Reproduction Assessment

Yes, at source level: pass a source config with channels.telegram.accounts.<id>.botToken as a SecretRef while the active runtime snapshot contains the resolved token, then call resolveTelegramToken or resolveTelegramAccount with the source config. Current main reads the source config directly instead of selecting the runtime snapshot.

## Solution Assessment

Yes for the repair direction: mirror the Discord selectApplicableRuntimeConfig pattern inside the Telegram plugin before credential reads. A new config option, broad core migration, or relaxed runtime SecretRef policy is not the narrowest fix.
