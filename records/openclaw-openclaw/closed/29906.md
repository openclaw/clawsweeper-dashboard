## Summary

Current main already implements configurable proactive compaction via `agents.defaults.compaction.reserveTokens` (plus `keepRecentTokens`/`reserveTokensFloor`), and the runtime now pre-checks prompt size to compact before submission when the reserve-based budget would overflow. The changelog shows this config surface shipped in release v2026.2.21, which predates this issue.

## Risks / Open Questions

- Discoverability is imperfect: `docs/gateway/configuration-reference.md` does not surface `reserveTokens` as clearly as the type/schema/docs deep-dive, so users may still miss the existing control.

## Close Comment

Closing as implemented after Codex review. Current `main` already has the requested control under `agents.defaults.compaction.reserveTokens` rather than a separate `triggerTokens` key. The runtime applies that setting in `src/agents/pi-settings.ts:65`, the threshold semantics are documented as `contextTokens > contextWindow - reserveTokens` in `docs/reference/session-management-compaction.md:228`, and current embedded-agent code also does a pre-submit overflow precheck in `src/agents/pi-embedded-runner/run/attempt.ts:2190` so compaction can happen before the prompt is sent. This shipped in `v2026.2.21` per `CHANGELOG.md:4454`. If you still see a provider-specific limit mismatch on a current build, please open a fresh bug with the exact provider/model id, config, `/status` output, and overflow logs.
