## Summary

Current main and the latest stable release already address the reported stuck-`announcing` Bonjour loop: the advertiser now tracks stuck non-`announced` states, retries only a bounded number of times, then disables Bonjour for the current Gateway process instead of restarting forever. The release changelog for `v2026.4.27` explicitly calls out the same stuck `probing`/`announcing` failure tracked by related issue #69011.

## What This Changes

The issue reports that OpenClaw's internal Bonjour/mDNS advertiser on Windows can remain in `announcing`, restart about every 40 seconds, and spam logs with no clear disable path.

## Best Possible Solution

Keep the shipped bounded-restart behavior and point affected users to upgrade to `v2026.4.27` or newer; if the same loop persists after upgrading, it should be handled as a fresh post-fix report with new logs from that version.

## Reproduction Assessment

Yes. The report's log string maps directly to the current watchdog branch, and the unit test reproduces a service stuck in `announcing` long enough to verify that the advertiser disables itself and stops further restart spam.

## Solution Assessment

Yes. Bounding the ciao/mDNS recovery loop and disabling only Bonjour for the current Gateway process is the narrow maintainable fix because the Gateway remains running and users still have documented `discovery.mdns.mode="off"` and `OPENCLAW_DISABLE_BONJOUR=1` controls for hostile multicast environments.
