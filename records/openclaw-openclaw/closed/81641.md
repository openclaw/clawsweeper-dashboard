## Summary

This PR is a focused and useful CLI bug fix, but it still misses the current defaulting path where model keys are normalized before built-in aliases are materialized. Merged as-is, a config using a retired Gemini key can still list a built-in alias and then get the old misleading removal error.

## What This Changes

The PR exports the default model alias table, adds a built-in-alias-specific `models aliases remove` error, and adds focused alias command tests.

## Best Possible Solution

Normalize the source models map for the built-in-alias check so `remove` matches `applyModelDefaults`, with regression coverage for a retired Gemini source key.

## Reproduction Assessment

Yes. Source inspection shows `list` reads the normalized/resolved config while the PR's new `remove` guard still checks the unnormalized source model map.

## Solution Assessment

No, not yet. The approach is right, but the guard needs to compare against the same normalized model map used by `applyModelDefaults`.
