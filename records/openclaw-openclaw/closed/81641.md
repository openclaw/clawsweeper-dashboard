## Summary

Keep this PR open: it fixes a real CLI inconsistency and includes sufficient terminal proof, but the proposed built-in-alias check still diverges from the existing default-materialization contract in two reachable config shapes.

## What This Changes

The PR exports the default model alias table, adds a built-in-alias-specific error path to `modelsAliasesRemoveCommand`, and adds focused alias command tests.

## Best Possible Solution

Keep the focused CLI error-message fix, but make built-in detection reuse the same materialization contract as `applyModelDefaults`: normalize source model keys and require `alias === undefined`, with regressions for empty aliases and retired Gemini keys.

## Reproduction Assessment

Yes. Source inspection shows `list` consumes resolved config while `remove` mutates source config, and the PR body includes terminal proof for the current mismatch and intended after-fix behavior.

## Solution Assessment

No, not yet. The fix direction is narrow and maintainable, but the PR should align its condition with `applyModelDefaults` rather than using a falsy, unnormalized source-map lookup.

- **[P2] Preserve explicit empty alias disables:** `src/commands/models/aliases.ts:106`
  - body: This treats `alias: ""` as a materialized built-in alias because the check is based on falsiness. Current defaults intentionally skip default alias materialization whenever `entry.alias !== undefined`, and existing coverage proves empty strings disable defaults; with that config `models aliases list` will not show `gemini`, but `remove gemini` now reports a built-in alias instead of the correct not-found path.

Existing defaults normalize retired Gemini keys before materializing aliases, so a source config containing `google/gemini-3-pro-preview` still makes `list` show `gemini`; this branch misses that source key and falls back to the misleading `Alias not found` message.
  - confidence: 0.86
