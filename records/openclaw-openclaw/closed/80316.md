## Summary

Current main already removes the affected BlueBubbles runtime and the supported iMessage replacement persists and hydrates short-id mappings before action resolution, so the observable one-shot CLI failure is fixed on main without changing the generic conversation_info field order.

## What This Changes

The issue requests preferring `MessageSidFull` over `MessageSid` in the agent conversation_info envelope so BlueBubbles one-shot CLI actions can resolve message ids.

## Best Possible Solution

Keep the current iMessage replacement path as the maintained behavior and direct BlueBubbles users to migrate; if maintainers later want both ids in the prompt envelope, track that as a separate envelope-contract change.

## Reproduction Assessment

No for current main: the exact BlueBubbles runtime is gone, and the supported iMessage resolver now reloads persisted short-id mappings before a fresh-process lookup. Source and regression-test coverage show the reported in-memory-only failure path no longer matches main.

## Solution Assessment

Yes for the observable failure: the maintained iMessage path resolves short ids across process restarts without changing generic prompt metadata semantics. A separate `message_id_full` envelope field would be a broader contract decision, not needed to close this current-main cleanup.
