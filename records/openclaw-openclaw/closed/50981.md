## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Send numeric agent timeouts:** `src/agents/tools/sessions-send-tool.ts:251`
  - body: `agent` RPC validation defines `timeout` as an integer, so forwarding `String(targetTimeoutSeconds)` makes real gateway calls reject these nested agent runs as invalid even though the mocked tests pass. Keep the forwarded timeout numeric in both nested callers.
  - confidence: 0.93
- **[P1] Preserve fire-and-forget run budgets:** `src/agents/tools/sessions-send-tool.ts:220-251`
  - body: For `timeoutSeconds: 0`, this now forces the target run to 30 seconds instead of omitting `timeout` and inheriting the normal agent budget. Existing fire-and-forget subagent work that legitimately runs longer can be killed early, so this path needs to preserve the previous run budget while still returning immediately.
  - confidence: 0.88
- **[P2] Keep lock wait inside the effective caller budget:** `src/agents/pi-embedded-runner/run/attempt.ts:1705-1708`
  - body: Passing the full `params.timeoutMs` to the session write lock lets a target spend the whole requested timeout waiting for the lock and then continue into model execution, while the caller-side `agent.wait` has already expired. The lock wait should be bounded by the same effective deadline that governs the caller-visible run result.
  - confidence: 0.82
- **[P2] Align helper target timeout with its wait cap:** `src/agents/tools/agent-step.ts:45-56`
  - body: `runAgentStep` grants the child run the full `params.timeoutMs` but still caps `agent.wait` at 60 seconds, so callers above 60 seconds can return `undefined` while the child run continues in the background. Either cap the target timeout to the wait budget or make the wait budget match the propagated target timeout.
  - confidence: 0.86
