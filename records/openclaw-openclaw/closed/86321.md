## Summary

Keep open. The patch targets a real, source-reproducible Telegram progress leak and the code path is focused, but it still needs live Telegram proof and maintainer judgment on exposing a new shared reply option through the plugin SDK type.

## What This Changes

The PR adds a forced default tool/progress suppression reply option, wires Telegram to set it for streaming-off or tool-progress-disabled configurations, and adds focused shared-dispatch plus Telegram wiring tests.

## Best Possible Solution

Land one narrow Telegram suppression fix after redacted live Telegram proof and a maintainer decision on whether the forced suppression control belongs in the public reply API, while preserving final reply delivery and explicitly enabled progress behavior.

## Reproduction Assessment

Yes, source-level reproduction is clear: current main suppresses default progress only while verbose progress is off, so Telegram's disabled progress config can be bypassed when verbose becomes enabled. I did not run live Telegram proof in this read-only review.

## Solution Assessment

Mostly yes: the forced bit is a narrow way to preserve existing verbose behavior unless a channel explicitly asks for authoritative suppression. The unresolved part is whether that bit should be public plugin API or an internal dispatcher control.
