## Summary

Current main already implements the trace-correlation behavior reported here, and the fix is present in the shipped v2026.4.27 release. The diagnostics-otel service now tracks trusted diagnostic lifecycle spans and parents run/model/tool/context spans from validated internal trace ancestry instead of relying on ambient OpenTelemetry context; tests cover trusted parenting, post-completion model-usage parenting, and untrusted trace injection rejection.

## What This Changes

The issue reports that bundled diagnostics-otel spans are created from the active OpenTelemetry context instead of the diagnostic event trace context, fragmenting run traces and breaking log-to-trace correlation.

## Best Possible Solution

Keep the shipped trusted diagnostic span-parenting implementation, and have affected operators upgrade to v2026.4.27 or newer so the external listener-wrapping workaround can be removed.

## Reproduction Assessment

Yes. The original failure path is source-reproducible from the old behavior described in the report, and current main has focused tests that exercise the fixed trusted run/model/tool lifecycle parenting without depending on ambient context.active().

## Solution Assessment

Yes. The current implementation is safer than blindly parenting every span from evt.trace because it only uses internally trusted diagnostic events, tracks started spans, and explicitly rejects untrusted plugin-emitted trace injection.
