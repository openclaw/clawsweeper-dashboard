## Summary

Current main already addresses the reported Active Memory recall-tool failure: the hidden recall run now uses the target agent config while allowlisting the memory recall tools, and memory-core still declares and registers `memory_search`/`memory_get`.

## What This Changes

Active Memory should keep `memory_search` and `memory_get` available to its lightweight before-prompt recall sub-agent after upgrading to 2026.5.2.

## Best Possible Solution

Keep the current main implementation and ship it in the next release so 2026.5.2 users receive the Active Memory memory-tool availability fix.

## Reproduction Assessment

No for current main: I did not run a live gateway reproduction, but source inspection shows the reported empty explicit allowlist path is now covered by the current embedded Active Memory config and memory-tool allowlist. The original report remains plausible for the tagged 2026.5.2 release.

## Solution Assessment

Yes: preserving the target agent runtime config for the embedded recall run is the narrow maintainable fix because it lets the existing plugin-tool contract materialize `memory_search` and `memory_get` without hard-coding memory-core behavior elsewhere.
