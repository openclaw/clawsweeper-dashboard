## Summary

Keep this PR open, but it is not merge-ready. The requested PID observability is valid on current main, yet this patch reads PID fields that the pinned ACPX public runtime API does not expose, so real ACPX sessions would still omit the PID.

## What This Changes

Adds optional PID fields to ACP runtime/session metadata and attempts to propagate delegate-provided PIDs into ACP `sessions_spawn` and session status results.

## Best Possible Solution

Expose PID through a typed ACPX runtime contract first, then propagate it through OpenClaw's ACP runtime types, manager metadata, `sessions_spawn`, `session_status`, docs, changelog, and integration-style coverage with clear visibility and staleness rules.

## Reproduction Assessment

Yes. Source and dependency inspection are enough: current main omits PID from ACP spawn/session metadata, and the pinned ACPX public runtime contract does not expose the PID shape this PR reads.

## Solution Assessment

No. The direction is useful, but the patch is not the best fix because it forwards mocked, non-contract fields instead of consuming a real ACPX process-metadata source and documenting the exposure contract.
