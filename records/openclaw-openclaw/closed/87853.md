## Summary

Keep open: the useful Discord reconnect jitter is still blocked because the delayed startup connect timer is not lifecycle-owned and can fire after explicit gateway cleanup.

## What This Changes

The PR threads account IDs into Discord gateway plugin creation, applies deterministic one-shot startup jitter before the first non-resume connect, and adds jitter hash tests.

## Best Possible Solution

Keep the startup jitter behavior, but store the pending timer handle, cancel it from the gateway disconnect path, and cover delayed startup plus disconnect-before-fire with fake-timer tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with nonzero startup jitter, call `connect(false)`, then `disconnect()` before the timer fires; the queued callback later calls `super.connect(false)`.

## Solution Assessment

No, not as submitted. The one-shot jitter is a plausible narrow fix, but it needs lifecycle-owned cancellation so gateway cleanup remains authoritative.
