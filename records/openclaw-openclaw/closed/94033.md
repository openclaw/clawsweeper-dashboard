## Label Justifications

- `P1`: The report describes a real scheduled isolated-agent workflow failing during long-running tool execution before the intended timeout behavior.
- `impact:crash-loop`: The affected isolated cron run can be aborted by a process-level lane timeout, disrupting scheduled agent availability.

## Summary

Keep open: current main still has the reported source-level timeout path, and the issue is paired with multiple open closing PRs rather than a merged fix.

## What This Changes

The issue reports isolated cron agent `CommandLaneTaskTimeoutError` during long tool execution because lane progress is not refreshed while the embedded attempt is in flight.

## Best Possible Solution

Land one focused fix that refreshes lane progress while `runEmbeddedAttemptWithBackend` is in flight, with regression proof for the command-lane sliding timeout and embedded-runner cleanup semantics.

## Reproduction Assessment

Yes. Source inspection shows the command lane uses a sliding progress timestamp, while the embedded runner refreshes that timestamp only on startup/progress callbacks and not during the whole awaited embedded attempt.

## Solution Assessment

Yes on the root fix direction, but the issue itself should not close until one linked PR lands. The maintainable solution is a scoped heartbeat or equivalent in-flight progress signal that preserves the existing absolute timeout and cleanup contracts.
