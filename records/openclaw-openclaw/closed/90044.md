## What This Changes

The issue asks for Slack messaging-tool send telemetry and reply dedupe to stay bound to the active thread so a thread-scoped send cannot suppress later automatic replies outside that same thread.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/90045 or an equivalent focused fix that carries the active Slack thread into send telemetry and compares dedupe routes with the originating thread included.

## Reproduction Assessment

Yes, source-reproducible: current main can compare a thread-scoped Slack tool send against an origin route that lacks the origin thread, and implicit message sends only record explicit args.threadId. I did not run a live Slack workspace reproduction in this read-only review.

## Solution Assessment

Yes, the linked PR's direction is the narrow owner-boundary fix: preserve the active thread in message-send evidence and include originatingThreadId in final/followup dedupe. The issue should stay open until that implementation or an equivalent reviewed fix merges.
