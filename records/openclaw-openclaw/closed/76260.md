## Summary

Keep open. Current main still shows the reported boundary mismatch: `web_fetch` uses the SSRF guarded fetch path, while gateway host `exec` reaches the shell runner without destination classification; the linked fix PR was closed unmerged after review found bypasses and exec-behavior regressions.

## What This Changes

The issue asks whether host `exec` should enforce the same private/internal destination blocking that `web_fetch` applies, after a model fell back from blocked `web_fetch` localhost URLs to `curl` via `exec`.

## Best Possible Solution

Settle and document the host-exec egress contract, then implement robust isolation/proxy enforcement with tests or explicitly state that host exec is outside `web_fetch` SSRF guarantees.

## Reproduction Assessment

Yes, at source level. The issue provides a localhost transcript, and current main shows `web_fetch` using the SSRF guard while gateway host `exec` runs shell commands without a destination classifier.

## Solution Assessment

Unclear. The issue identifies a real boundary gap, but the prior parser-based PR showed bypass and overblocking risks, so the best fix needs maintainer/security policy direction before implementation.
