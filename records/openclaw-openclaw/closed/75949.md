## Summary

Close as duplicate of the still-open canonical session-write-lock tracking issue #13744. Current main still shows the same 10s lock-contention root cause, while the remaining queue/config/single-writer work is already tracked there and partly overlapped by open PR #69159.

## What This Changes

Queue or retry peer-agent and system-event deliveries that contend with a long-running turn's session transcript write lock instead of dropping them after `SessionWriteLockTimeoutError`.

## Best Possible Solution

Keep #13744 as the canonical place for the fix: a transcript-file-aware serialization/retry path plus any public lock timeout configuration should be designed once and applied consistently across embedded runs, CLI/gateway agent calls, heartbeat, cron, and transcript rewrite paths.

## Reproduction Assessment

Yes. The source and unit test give a high-confidence local reproduction for the root failure: hold a session write lock longer than the 10s default, then attempt a second non-reentrant acquisition for the same transcript; the full Wren-to-Telegram incident route was not live-replayed in this read-only review.
