## Summary

Keep open. Current main still lacks supported custom cron IDs, and this PR remains useful but is not merge-ready because real behavior proof is missing, two P2 correctness issues remain, and overlapping custom-ID work needs convergence.

## What This Changes

The branch adds optional custom cron job IDs across the CLI, Gateway protocol, agent tool normalization, cron service/types, docs, generated Swift models, and tests, with duplicate-ID rejection.

## Best Possible Solution

Land one canonical custom cron-ID implementation that validates path-safe stable IDs end-to-end, rejects duplicates with structured Gateway/CLI errors, preserves UUID defaults and Swift client source compatibility, and closes the canonical feature issue.

## Reproduction Assessment

Yes. Source inspection on current main shows `cron add` has no `--id`, `CronAddParamsSchema` has no `id` and rejects extra properties, and `createJob` still assigns `crypto.randomUUID()` unconditionally.

## Solution Assessment

No. The direction is maintainable, and this PR handles case-variant duplicate IDs, but it is not the best merge shape until Gateway duplicate errors, Swift initializer compatibility, branch freshness, and real behavior proof are handled.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve Swift initializer compatibility:** `apps/shared/OpenClawKit/Sources/OpenClawProtocol/GatewayModels.swift:4142`
  - body: Adding `id: String?` as a required initializer argument changes the public `CronAddParams(...)` call signature even though the wire field is optional. Default generated optional initializer parameters to `nil` or add a compatibility overload, then regenerate the current Swift model output so existing callers keep compiling.
  - confidence: 0.9
- **[P2] Return duplicate IDs as invalid cron.add requests:** `src/cron/service/ops.ts:326`
  - body: The duplicate guard throws a plain `Error`, but Gateway `cron.add` only converts `TypeError` and `RangeError` from `context.cron.add` into `INVALID_REQUEST`. A duplicate ID through Gateway or CLI can therefore surface as an internal failure instead of the promised validation response.
  - confidence: 0.86
