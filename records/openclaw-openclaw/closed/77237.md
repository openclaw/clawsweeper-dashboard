Keep this PR open for maintainer review: it addresses a real managed npm rollback bug, includes current-head terminal proof and focused regression coverage, and I found no blocking correctness or security finding. The remaining decision is whether maintainers accept the full managed-root snapshot cost on failed install/update paths.

## What This Changes

The PR snapshots and restores the managed npm plugin root, including package metadata and node_modules, when npm-backed install validation rejects a candidate after npm has mutated the root.

## Best Possible Solution

Merge the exact snapshot rollback design if maintainers accept the failure-path disk and I/O cost, because preserving the previous local root is safer than trying to reconstruct npm side effects after validation rejects a candidate.

## Reproduction Assessment

Yes. The current-main source path mutates the managed npm root before late validation and then rolls back with uninstall/remove rather than an exact previous-root restore; I did not run a local repro because this review is read-only.

## Solution Assessment

Yes, with maintainer acceptance of the cost. The snapshot approach is the narrowest reliable fix for exact rollback because npm can mutate package metadata, lockfiles, the target package, and hoisted dependencies in one install.
