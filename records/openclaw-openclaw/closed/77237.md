## Summary

The PR appears to address a real managed npm rollback bug on current main, includes sufficient terminal proof and focused regression coverage, and I found no blocking correctness or security finding; the remaining merge question is maintainer acceptance of the full-root snapshot I/O tradeoff.

## What This Changes

The PR changes managed npm plugin installs to snapshot and restore the npm root when post-install validation rejects a candidate, adds installer regression tests, and records a changelog entry.

## Best Possible Solution

Merge the exact snapshot rollback if maintainers accept the failure-path disk and I/O cost; otherwise require a lighter rollback design that still atomically restores package metadata, lockfile, plugin files, and hoisted dependencies.

## Reproduction Assessment

Yes by source inspection, though I did not run a local repro in this read-only review. Current main mutates the managed npm root before post-install validation and then rolls back by uninstalling/removing the target rather than restoring the previous root.

## Solution Assessment

Yes, with a maintainer tradeoff. Exact snapshot restore is the most reliable narrow fix for npm mutations across package metadata, lockfile, plugin files, and hoisted dependencies; a lighter design is only preferable if maintainers reject the I/O cost.
