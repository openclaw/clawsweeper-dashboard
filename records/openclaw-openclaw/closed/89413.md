## Summary

Keep open. The quarantine direction is useful, but the amended patch still treats OpenAI `allowed_tools` tool-choice objects as compatible whenever any projected tool survives, so a filtered-out tool can still leave an invalid provider payload.

## What This Changes

The PR snapshots OpenAI tool descriptors before strict-schema and payload conversion, filters unreadable descriptors, omits empty tool lists, and drops stale forced or pinned tool choices in OpenAI payload builders.

## Best Possible Solution

Reconcile every function-constraining `tool_choice` variant against the projected tool set, including Responses and Chat `allowed_tools`, with mixed and all-filtered regression coverage.

## Reproduction Assessment

Yes for the review finding: source inspection shows non-`function` object tool choices are accepted whenever any projected tool survives, while OpenAI `allowed_tools` choices name specific allowed tools. A mixed healthy/unreadable tool set can therefore leave a stale allowed-tools constraint.

## Solution Assessment

No. The patch is close, but the best fix is to make tool-choice reconciliation understand `allowed_tools` instead of only string and named-function choices.
