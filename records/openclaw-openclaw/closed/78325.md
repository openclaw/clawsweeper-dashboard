## Summary

Close as duplicate/superseded: the remaining context-loss concern is already tracked by #76629, which covers Feishu/Lark provider-bound replay pollution from hidden runtime-context rows, `NO_REPLY`, failed empty assistants, and copied runtime scaffolding. The specific request to move runtime context into a system turn conflicts with the current cache-stability boundary from #77521, so the canonical fix should stay with replay hygiene rather than a separate role-shape issue.

## What This Changes

The issue asks OpenClaw to inject `openclaw.runtime-context` as a system turn instead of a hidden `custom_message` so mid-turn user messages are not separated by runtime metadata.

## Best Possible Solution

Keep #76629 as the canonical replay-hygiene fix: prune OpenClaw-owned runtime/no-op artifacts at provider replay and add doctor cleanup for polluted transcripts, without moving per-turn context into the ordinary system prompt.

## Reproduction Assessment

No, not for the full wrong-directory behavior from this report alone. Source inspection does confirm the reported `custom_message` storage shape, and #76629 has the high-confidence source path for the broader provider-bound replay pollution.

## Solution Assessment

No. Treating these rows as system turns is not the narrowest maintainable fix because #77521 intentionally kept dynamic runtime context out of the system prompt; #76629's replay-boundary cleanup is safer.
