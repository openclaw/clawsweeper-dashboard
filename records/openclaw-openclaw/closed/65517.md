## Summary

Current main resolves the permanent Telegram unresponsiveness path reported in #65517. Active Memory now races the recall sub-agent against its own hard timeout and returns a timeout result instead of awaiting a hung embedded run indefinitely; regression coverage simulates a sub-agent that never observes abort and verifies the hook returns near timeoutMs. Core embedded-run cleanup also remains abort-aware and clears active handles. The hard-deadline Active Memory change is on current main but I could not tie it to an existing release tag, so fixedRelease is null.

## Best Possible Solution

Close #65517 as implemented on main. Keep the shipped hard-deadline/abort-cleanup behavior for Active Memory, and use #65309 or a focused follow-up for any remaining request to make Active Memory recall background/non-blocking rather than preserving this Telegram starvation report.
