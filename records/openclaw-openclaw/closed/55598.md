## Summary

Close as implemented_on_main. Current main already supports arbitrary JavaScript execution in the browser tool through `action:"act"` with `kind:"evaluate"`, plus the CLI surface `openclaw browser evaluate`. The capability is documented, wired through both Playwright and Chrome MCP/CDP paths, and present in the released `v2026.4.22` tree.

## Risks / Open Questions

- The issue asked for a literal `browser.exec` command name; current OpenClaw ships the capability under `browser act kind=evaluate` / `openclaw browser evaluate` instead of that exact alias.

## Close Comment

Closing from a Codex review because the requested capability is already implemented on current `main`, and it is also present in the released `v2026.4.22` tree (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). The browser tool schema already exposes `action: "act"` with `kind: "evaluate"` plus an `fn` string (`extensions/browser/src/browser-tool.schema.ts:4`, `extensions/browser/src/browser-tool.schema.ts:81`). The browser control route handles that action and returns the result (`extensions/browser/src/browser/routes/agent.act.ts:360`, `extensions/browser/src/browser/routes/agent.act.ts:552`). Both runtime backends execute the supplied JavaScript: Chrome MCP via `evaluate_script` (`extensions/browser/src/browser/chrome-mcp.ts:694`) and Playwright via page/element evaluation (`extensions/browser/src/browser/pw-tools-core.interactions.ts:784`). The CLI and docs already surface this as `openclaw browser evaluate --fn ...` and explicitly describe it as arbitrary JavaScript execution in the page context (`extensions/browser/src/cli/browser-cli-actions-input/register.form-wait-eval.ts:95`, `docs/cli/browser.md:158`, `docs/tools/browser-control.md:317`). This means the underlying request is already satisfied, even though the public name is `evaluate` rather than a new `browser.exec` alias.
