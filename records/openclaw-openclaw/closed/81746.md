## Summary

Keep this PR open for maintainer handling: the linked Telegram stall is a real bug class, but the PR has the protected `maintainer` label, the submitted proof is synthetic-only for a Telegram transport change, and the diff introduces blocking polling-regression risks.

## What This Changes

The PR moves Telegram Bot API polling into a worker-backed isolated ingress path with a durable spool, adds Telegram ingress tests, updates stuck-session recovery logging behavior, and adds a changelog entry.

## Best Possible Solution

Keep the isolated-ingress direction, but land it only after preserving current polling error semantics, scoping durable spool state to bot identity, and adding live Telegram proof for the transport change.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partly: the linked bug and PR body describe a plausible event-loop saturation path, and the current source still couples getUpdates to the main polling cycle, but I did not rerun a live Telegram saturation repro in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? Not as submitted. Moving ingress out of the main event loop is the right boundary, but the patch needs parity with current getUpdates conflict handling, bot-identity scoping for the spool, and live Telegram proof before it is merge-ready.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve getUpdates conflict recovery in the worker:** `extensions/telegram/src/telegram-ingress-worker.runtime.ts:73-79`
  - body: The worker turns non-OK Bot API responses into a plain `Error` with only the description, and then rethrows anything `isRecoverableTelegramNetworkError` does not recognize. A 409 `getUpdates` conflict is currently handled by the polling session as a recoverable duplicate-poller case with transport dirtying and backoff, but under the new default isolated ingress it will instead make the worker exit and stop the Telegram monitor path. Preserve the Bot API error code/method and route 409 conflicts through the same restart behavior.
  - confidence: 0.86
- **[P2] Scope the ingress spool to the current bot identity:** `extensions/telegram/src/telegram-ingress-spool.ts:41`
  - body: The new spool path is keyed only by account id. The existing offset store intentionally discards persisted watermarks when the bot id or token fingerprint changes, but undrained spooled updates would survive in the same directory and be handled by the replacement bot after a token rotation, potentially blocking all newer updates behind an old file. Include bot identity in the spool path or payload and skip or retire mismatches on rotation.
  - confidence: 0.8
