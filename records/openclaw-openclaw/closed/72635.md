## Summary

Keep open. The linked Docker missing-python bug is valid, but this PR is not merge-ready because its fallback runs inside the low-level pinned write helper and can replace the helper's no-follow parent walk with a path-based writer.

## What This Changes

The PR adds spawn-error detection to `src/infra/fs-pinned-write-helper.ts` and a new spawn-fallback test so missing or unexecutable `python3` falls back to the local writer for buffer and stream inputs.

## Best Possible Solution

Handle missing `python3` while preserving pinned-directory guarantees: implement a Node fallback with equivalent no-follow descriptor semantics, or move the fallback to the higher fs-safe layer through hardened legacy write/copy paths with symlink-parent, retarget-race, buffer, and stream-copy coverage.

## Reproduction Assessment

Yes. #72362 has a concrete Docker slim reproduction for missing `python3`, and the PR regression follows from combining the existing symlink-parent rejection test with the PR's mocked spawn-ENOENT fallback path.

## Solution Assessment

No. Falling back from `runPinnedWriteHelper()` to `runPinnedWriteFallback()` is not the safest fix because it bypasses the helper's pinned parent-directory semantics; the fallback should preserve those semantics or occur at a higher layer with equivalent boundary checks.
