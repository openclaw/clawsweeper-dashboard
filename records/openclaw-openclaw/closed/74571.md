## Summary

Keep open. The low-level IPv6 ULA classifier change is plausible, but the PR does not wire the new opt-in into the actual `web_fetch` user config/runtime path: current `main` still accepts and forwards only `allowRfc2544BenchmarkRange`, so the reported fake-IP proxy failure would remain for `tools.web.fetch` as submitted.

## What This Changes

Adds an optional IPv6 unique-local SSRF-policy exemption to shared IP/SSRF helpers and tests so `fc00::/7` can be allowed independently of the RFC2544 IPv4 fake-IP flag.

## Best Possible Solution

Revise the PR so the IPv6 ULA opt-in is a complete, documented `tools.web.fetch.ssrfPolicy` surface: keep defaults strict, update config schema/types/help/generated schema and docs, thread the bit through `runWebFetch` including cache partitioning and guarded fetch policy, preserve it in policy helpers, and add regression coverage for fake-IP DNS answers containing both IPv4 RFC2544 and IPv6 ULA records.

Any caller that composes a hostname policy with the new ULA opt-in would silently lose the exemption before calling the guard, so the helper and its tests should be updated with the new field.
  - confidence: 0.82
