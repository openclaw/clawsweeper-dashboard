## Summary

Keep open. Current main still keeps malformed tool-call argument repair disabled for Responses-family model APIs, and #75281 is the active open closing PR that should be reviewed before this issue is closed.

## What This Changes

The issue asks whether malformed tool-call argument repair should be broadened from Kimi/openai-completions to Codex, Azure, or OpenAI Responses transports.

## Best Possible Solution

Review and either land or reject #75281's narrow transport-specific gate change after confirming compatible `toolcall_delta` behavior for Codex and Azure Responses, while keeping generic `openai-responses` guarded unless its WebSocket path is proven safe.

## Reproduction Assessment

Unclear. The current exclusion is reproducible from the gate and guard tests on current main, but the reporter explicitly says they have not observed malformed Codex arguments, so there is no high-confidence runtime failure repro yet.

## Solution Assessment

Unclear. A narrow expansion for verified HTTP/SSE Responses transports looks like the maintainable direction, but the open PR should settle the exact transport list and keep the WebSocket-capable generic OpenAI Responses path separate.
