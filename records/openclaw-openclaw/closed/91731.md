## Summary

Keep open: the linked leak is valid, but this patch is too broad because it would prune live `processing` diagnostic entries with active work before the heartbeat can classify or recover them.

## What This Changes

The PR changes diagnostic session-state pruning so non-idle entries older than the 30-minute TTL with no queued work are deleted from the process-local Map.

## Best Possible Solution

Make non-idle pruning activity-aware, with regression coverage for both a ghost non-idle entry and a live long-running processing entry.

## Reproduction Assessment

Yes, source inspection shows the reported leak path: current pruning only deletes stale idle entries, while failed or stale recovery outcomes can leave non-idle state behind. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

No. The requested cleanup direction is valid, but the PR uses `queueDepth <= 0` as the only liveness guard even though current code treats active work with no queue as a valid long-running processing session.
