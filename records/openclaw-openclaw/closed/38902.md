## Summary

Current `main` now has explicit compatibility support for strict OpenAI-compatible local backends that reject structured `messages[].content` arrays, which matches the root cause identified in the issue thread. The transport can flatten text-only content to plain strings via `compat.requiresStringContent`, the config schema accepts that compat flag plus `compat.supportsTools`, and the docs now call out this exact failure mode for local `/v1/chat/completions` backends.

## Risks / Open Questions

- This appears fixed on `main` but not yet shipped in the latest tagged release `v2026.4.22`, so users on released builds may still hit the old behavior.
- The fix is opt-in via model compat config; some stricter backends may still need `compat.supportsTools: false` or may still have upstream limitations around streaming/tool calling.
