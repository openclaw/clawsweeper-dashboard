## Summary

Current main already contains the Slack health-monitor recovery change for this failure mode: non-manual stop timeouts are tracked as recovery state, the follow-up restart backoff is no longer tied to the already-aborted stop signal, and regression coverage verifies a restart after a timed-out recovery stop.

## What This Changes

Report that Slack Socket Mode health-monitor restarts can time out during shutdown and leave inbound Socket Mode disconnected without a later provider restart.

## Best Possible Solution

Keep the current main recovery behavior and ship it in the next release; any new failure after 2026.5.8 should be reported with fresh main-version logs showing whether the old task never settles at all.

## Reproduction Assessment

Yes. The report has a concrete v2026.5.7 log sequence, and current source/tests exercise the same non-manual health-monitor stop-timeout path that previously prevented a restart after the aborted stop signal.

## Solution Assessment

Yes. The current fix is narrowly placed in the generic channel lifecycle manager, preserves the duplicate-start guard, and only special-cases recovery stop timeouts enough to let the existing auto-restart path run.
