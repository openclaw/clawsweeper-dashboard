## What This Changes

The PR extends memory-core dream diary prompt construction with date, day-count, and previous-entry context, reads existing `DREAMS.md` before narrative generation, and adds prompt unit coverage.

## Best Possible Solution

Keep the temporal-context fix in memory-core, but read `DREAMS.md` through safe path checks, extract actual narrative openings instead of diary metadata, add coverage for `generateAndAppendDreamNarrative()` with an existing diary file, and require redacted real run proof.

## Solution Assessment

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Guard diary reads before prompting the model:** `extensions/memory-core/src/dreaming-narrative.ts:952`
  - body: This read follows whatever `DREAMS.md` resolves to and then includes extracted lines in the narrative subagent prompt. The existing write path rejects symlinked or non-file diaries before writing, but this new pre-prompt read bypasses that guard, so a workspace can make diary context come from an unsafe target. Check `lstat`/file type, or reuse an equivalent safe read path, before adding previous entries to the prompt.
  - confidence: 0.86
- **[P2] Skip diary metadata when collecting previous openings:** `extensions/memory-core/src/dreaming-narrative.ts:968`
  - body: Real diary blocks start with the italic timestamp written by `buildDiaryEntry`, and this predicate accepts that as the first non-empty non-comment line. That means the prompt will usually list dates instead of the previous prose openings, so the anti-repeat context does not reach the model. Skip the date/metadata line and cover `generateAndAppendDreamNarrative()` against an existing diary file.
  - confidence: 0.9
