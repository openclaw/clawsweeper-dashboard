## Summary

Keep open. The issue is concrete and still has an open implementation candidate, #63780, with closing syntax in the provided GitHub context. Current main has transient-network classification for ENETUNREACH, but the top-level uncaughtException handlers still only honor registered exception handlers and the narrow benign EPIPE/EIO path before exiting, while Telegram currently registers only an unhandled-rejection handler for polling network errors.

## Best Possible Solution

Keep this issue open and review #63780 as the canonical implementation candidate. The likely fix belongs either in the top-level uncaughtException handling with fatal/config precedence preserved, or in Telegram via the existing plugin-facing registerUncaughtExceptionHandler seam so recoverable polling transport failures can restart without crashing the gateway.
