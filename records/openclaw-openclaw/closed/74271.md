## Summary

Keep open. Current main still has the local gateway env/config token precedence split and no redacted state-dir `.env` mismatch warning; open PR #74433 references this issue with closing syntax and remains the implementation candidate, so this should stay open until that PR lands or is replaced.

## What This Changes

Request a redacted doctor/status diagnostic when a non-empty `OPENCLAW_GATEWAY_TOKEN` in the state-dir `.env` can override a configured `gateway.auth.token` in local gateway mode.

## Best Possible Solution

Keep this issue as the canonical report and resolve it by landing #74433 or a replacement that adds one shared redacted warning for differing non-empty state-dir `OPENCLAW_GATEWAY_TOKEN` and configured `gateway.auth.token`.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: local gateway mode with `gateway.auth.token` configured and a different non-empty `OPENCLAW_GATEWAY_TOKEN` loaded for client/probe paths exercises env-first credentials against config-first gateway auth; this read-only pass did not run a live gateway.

## Solution Assessment

Yes for the requested direction. A redacted warning is the narrow maintainable fix because it preserves intentional env-based auth and avoids deleting secrets; the final patch should parse the state-dir `.env`, compare differing non-empty values, and avoid printing either token.
