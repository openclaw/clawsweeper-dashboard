## Summary

Keep open: current main and v2026.5.12 still do not provide the requested identity-anchored cross-surface routing. The lower-level agent/Gateway plumbing can already carry an explicit session key, and an open PR covers the CLI flag subpart, but the broader voice/webhook/programmatic identity-anchor config remains a new privacy-sensitive routing feature.

## What This Changes

Request to route one user identity into a canonical session across Telegram, voice, WhatsApp, webhooks, and programmatic `openclaw agent` inputs, including a CLI session-key override and identity-anchor config.

## Best Possible Solution

Design an additive, opt-in identity/session-target contract that extends `dmScope` and `identityLinks`, review the CLI `--session-key` subpart under that contract, and preserve isolated defaults with explicit docs and regression coverage.

## Reproduction Assessment

Not applicable: this is a feature request, not a bug against an existing contract. Source, docs, and current tests give high confidence that the requested CLI selector and identity-anchor config are absent on current main.

## Solution Assessment

Unclear as a concrete implementation plan: the CLI flag matches existing lower-level plumbing, but the identity-anchor map needs maintainer-approved privacy, validation, and routing semantics before a core config change.
