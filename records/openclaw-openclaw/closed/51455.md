## Summary

Keep open. Current main still omits `standardFontDataUrl` in the bundled `document-extract` PDF.js call, the page-render fallback remains reachable, `v2026.4.27` has the same omission, and open PR #70936 already uses closing syntax while #74355 is another active implementation candidate.

## What This Changes

Configure the `document-extract` PDF.js fallback renderer to pass a standard-font asset path through `standardFontDataUrl` and add regression coverage for standard-font PDFs.

## Best Possible Solution

Land one narrow plugin-owned fix that resolves the installed `pdfjs-dist/standard_fonts/` directory as a slash-terminated filesystem path, passes it through the local `PdfJsModule.getDocument` type and call site, and adds focused regression coverage that would fail without the option.

## Reproduction Assessment

Yes. The issue body gives a concrete fallback-render reproduction with standard-font PDFs, and current main statically preserves both the missing option and reachable render path that make the warning plausible.

## Solution Assessment

Yes. The best boundary is the bundled `document-extract` plugin, not core; the maintainable path is to review and land one of the existing narrow PRs with real regression coverage.
