## Summary

Current main already implements atomic backup archive publication and cleanup for `openclaw backup create`, including `--verify`, and the same backup implementation is already present in the latest shipped tag `v2026.4.22`. The reported leftover 10-byte `.tmp` behavior is directly contradicted by the current code and regression tests.

## Evidence

- **Atomic publish path:** `createBackupArchive()` writes the tarball to a unique `<output>.UUID.tmp` path, then publishes it with `publishTempArchive()`. That publish step hard-links to the final path and falls back to `fs.copyFile(..., COPYFILE_EXCL)` when hard links are unsupported, preventing an unfinished temp file from being treated as the final archive.
  - file: [src/infra/backup-create.ts:134](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/src/infra/backup-create.ts#L134)

  - sha: [7ac35b4f6900](https://github.com/openclaw/openclaw/commit/7ac35b4f69009490180a35bf404638d4280a4865)
- **Temp cleanup on exit:** After tar creation and publish, a `finally` block removes both the temp archive path and the temporary manifest directory, so current main is explicitly written to clean up the kind of stranded `.tmp` file described in the issue.
  - file: [src/infra/backup-create.ts:334](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/src/infra/backup-create.ts#L334)

## Close Comment

Closing as implemented on main after a Codex review. The current backup path already writes to a unique `*.tmp` archive, publishes it atomically with `publishTempArchive()` using hard-link plus `COPYFILE_EXCL` fallback, and then cleans up temp files in `finally` (`src/infra/backup-create.ts:142`, `src/infra/backup-create.ts:334`). The regression is covered by `src/commands/backup.atomic.test.ts:58`, which verifies tar failures leave no final archive and no leftover temp entries, and by `src/commands/backup.atomic.test.ts:107`, which covers the unsupported-hard-link fallback. `--verify` also runs only after `createBackupArchive()` returns the finished archive path (`src/commands/backup.ts:23`). The latest shipped tag `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`) already contains the same backup implementation as current `main`, so this issue appears fixed in release.
