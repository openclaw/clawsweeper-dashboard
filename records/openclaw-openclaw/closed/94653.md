## Summary

Keep open: this is a useful cron CLI surface, but the patch is not merge-ready because `--clear-fallbacks` does not clear the current cron patch contract and empty fallback lists cannot be expressed from the new CLI flag. The older overlapping https://github.com/openclaw/openclaw/pull/93369 covers the same issue more completely, but it is still open and needs normal review/refresh before this PR can be closed as superseded.

## What This Changes

This PR adds `--fallbacks` to cron add/create and edit, plus `--clear-fallbacks` to edit, mapping values into agent-turn cron payloads.

## Best Possible Solution

Land one complete cron fallback CLI implementation that preserves absent, empty, non-empty, and clear semantics across CLI parsing, protocol validation, normalization, service patching, docs, and focused tests.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path for the review findings: `--clear-fallbacks` sends `fallbacks: null`, current normalization drops that field, and the service has no fallback-null clear branch. Current docs/tests also show `fallbacks: []` is the strict no-fallback state, while this PR's parser returns `undefined` for an empty CLI value.

## Solution Assessment

No, this is not the best fix as written. The maintainable fix is to use a fallback-specific parser and either wire nullable clear semantics through the cron patch contract end-to-end or leave the clear flag out until that contract is changed.
