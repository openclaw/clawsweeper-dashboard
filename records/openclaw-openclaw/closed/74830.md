## Summary

Keep open. Current main still has the abort/wait regression this PR targets, but the proposed patch should not merge as-is because it can leave chat abort markers active far longer than the dedupe snapshot they protect, creating a stale-marker path for later chat sends that reuse the same idempotency key.

## What This Changes

The PR preserves Gateway abort/cancel wait snapshots across late chat and agent completion by keeping aborted chat markers, making RPC timeout snapshots non-overwritable, mapping aborted agent completions to timeout/rpc, and adding Gateway regression tests.

## Best Possible Solution

Keep the cancel snapshot authoritative until the underlying run settles, then clear or partition the chat abort marker so it cannot suppress future terminal snapshots after the normal dedupe window. The final patch should cover second-wait-after-settle for both resolve and reject paths, plus marker cleanup or idempotency-key reuse behavior.

## Reproduction Assessment

Yes. A high-confidence code reproduction is: start a chat or agent run, call `sessions.abort`, observe the timeout/rpc snapshot, let the underlying send or agent promise settle, then call `agent.wait` again; current main can allow late ok/error completion to win because the abort marker or agent dedupe snapshot is not durable enough.

## Solution Assessment

No, not yet. Preserving cancel snapshots is the right direction, but this patch keeps the chat aborted marker until the 60-minute maintenance cleanup instead of only through underlying completion, while the dedupe cache expires after five minutes.
