## Summary

Close as superseded: https://github.com/openclaw/openclaw/pull/79041 already tracks the same Webhooks child-session boundary bug with a more complete session-visibility guard, docs, registered-route coverage, and real behavior proof; this smaller patch lacks real proof and has correctness gaps.

## What This Changes

The PR adds a Webhooks `run_task.childSessionKey` string-prefix guard and tests rejection of a non-descendant key.

## Best Possible Solution

Close this PR in favor of reviewing and landing or superseding https://github.com/openclaw/openclaw/pull/79041, then close https://github.com/openclaw/openclaw/issues/79038 after a merged fix.

## Reproduction Assessment

Yes. Source inspection on current main shows `run_task` accepts and forwards `childSessionKey`, and the cancellation path later acts on the stored child session key.

## Solution Assessment

No. This patch is not the best fix because it uses a lexical prefix check, rejects valid recorded child sessions, misses exact route-session allowance from the linked issue, and returns a failure shape the existing status mapper treats as 409.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Map the new rejection before returning it:** `extensions/webhooks/src/http.ts:644-647`
  - body: This new `created: false` result is still passed through `mapRunTaskStatus`, which maps unknown `run_task` failures to HTTP 409. The added test expects HTTP 200, so it will fail and clients will not receive the claimed response.
  - confidence: 0.92
- **[P2] Use session ownership records instead of prefixes:** `extensions/webhooks/src/http.ts:638-639`
  - body: The linked bug asks for the route session itself or sessions already recorded as spawned/parented by that route session. A case-insensitive string prefix both rejects valid recorded child sessions with other agent/session-key shapes and accepts unrecorded keys that merely share the prefix.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff touches a Webhooks authorization boundary, and the string-prefix guard is not equivalent to the recorded session-tree authority model.

Concerns:

- **[medium] Do not use lexical prefixes as session authority:** `extensions/webhooks/src/http.ts:638`
  - body: The guard allows any `childSessionKey` starting with the route key plus `:` and rejects recorded child sessions outside that lexical shape; this can leave the Webhooks boundary inconsistent with the actual session-tree records used elsewhere.
  - confidence: 0.86
