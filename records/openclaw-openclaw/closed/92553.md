## Label Justifications

- `P1`: A stale generated model catalog can make all configured models unavailable for affected users, breaking model-dependent runtime workflows.
- `impact:auth-provider`: The issue affects provider/model registry loading and model resolution for configured providers.

## Summary

Keep open: current `main` and the latest release still have the plugin-catalog early return that can discard already loaded custom models, and two open closing PRs now cover the repair but neither has merged yet.

## What This Changes

The issue reports that one invalid generated plugin model catalog can abort the entire custom model load, leaving model resolution with zero available models and hiding the underlying validation error.

## Best Possible Solution

Land one reviewed fix PR that isolates invalid generated plugin catalog shards while preserving a visible registry warning/error, then close this issue from the merged commit.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: put a generated plugin catalog with a provider/model missing `api` next to otherwise valid root or plugin models, then `loadCustomModels()` returns the shard error and leaves the registry empty. I did not run the probe because this review is read-only.

## Solution Assessment

Yes, the best current direction is a narrow runtime isolation fix in `ModelRegistry` with regression coverage; generator cleanup for stale shards is useful but should not be required before stopping one bad shard from taking down all models.
