## Label Justifications

- `P1`: The reported behavior can flood a real messaging channel and expose raw tool/session output from a normal failed tool path.
- `impact:security`: The issue is about raw failed tool output and historical session/search content becoming user-visible instead of staying bounded or summarized.
- `impact:message-loss`: One failed tool result produced hundreds of channel messages from a single turn, disrupting channel delivery semantics.

## Summary

Close as implemented on current main: the embedded runner now caps tool-error detail extraction to a first-line 400-character summary, caps exec aggregated output before result/command-output events, and builds user-visible fallback warnings from that bounded `lastToolError.error`. The proof is current-main-only; the latest release tag does not contain these embedded-runner files.

## What This Changes

The issue asks OpenClaw to cap and summarize user-visible tool-error fallback replies so raw failed tool stdout/stderr cannot flood messaging channels.

## Best Possible Solution

Keep the core pre-dispatch caps on main, and let the Feishu/Lark plugin separately add a channel-local chunk ceiling if desired.

## Reproduction Assessment

No, I did not establish a failing current-main repro. The report has concrete logs from an older release, and source inspection shows the current path now bounds the tool-error and exec-output surfaces that caused the flood.

## Solution Assessment

Yes. The current-main fix is at the right core layer: sanitize and cap tool result/error data before fallback payload construction, with channel-specific caps left as defense in depth.
