## Summary

Keep open: current main still builds Responses input messages for system/developer/user turns without the explicit `type: "message"` discriminator, and Microsoft’s Foundry/OpenAI schema describes message input items with `type` set to `message`. This is a narrow provider payload-shape bug, not stale cleanup. ([learn.microsoft.com](https://learn.microsoft.com/en-us/azure/ai-foundry/openai/reference-preview?view=foundry-classic&utm_source=openai))

## What This Changes

The issue requests Azure AI Foundry Responses API compatibility by adding explicit message input item types and clarifying the Azure SDK/project-endpoint workaround.

## Best Possible Solution

Update the shared Responses converter to emit explicit `type: "message"` and array `input_text` content for system/developer/user messages, then cover Foundry/OpenAI-compatible payload shape with focused regression tests.

## Reproduction Assessment

Yes for source-level reproduction: current main builds affected Responses input items without `type: "message"`, and the upstream Microsoft schema documents the message item discriminator. I did not run a live Foundry request because no Azure credentials were available.

## Solution Assessment

Yes for the central bug: adding the discriminator in shared Responses payload conversion is the narrowest maintainable fix and avoids a new provider config flag. The Azure SDK `api-version` behavior should be handled separately if maintainers want `azure-openai-responses` itself to support project endpoints.
