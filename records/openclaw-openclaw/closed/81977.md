## Summary

Close: current main already implements the observable Gateway client-tool loop for `/v1/chat/completions`; request `tools` are converted into client tools, pending client calls are returned as OpenAI-compatible `tool_calls`, and the docs/tests cover the follow-up `role: "tool"` turn. This appears current-main-only, not proven shipped in `v2026.5.12`.

## What This Changes

Request a Gateway OpenAI Chat Completions mode or header that proxies client-provided tools through and returns model `tool_calls` for local client execution.

## Best Possible Solution

Keep the current agent-first client-tool contract; if a released build still fails for a specific client/model, ask for a new bug with redacted request/response logs and the selected OpenClaw version.

## Reproduction Assessment

No. I did not establish a failing current-main path; source, docs, and regression tests show the endpoint already accepts client tools and returns pending calls as `tool_calls`.

## Solution Assessment

Yes. The current implementation is the narrow maintainable answer to the central problem because it preserves Gateway agent routing while delegating client-provided tools back to the OpenAI-compatible client.
