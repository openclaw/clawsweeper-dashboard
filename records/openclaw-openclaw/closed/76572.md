## Summary

Keep open. Current main still has the strict context-window model-id match, and this PR is the right narrow fix path, but the new helper should also preserve established provider-alias normalization before merge.

## What This Changes

This PR adds provider-prefix-aware matching in `resolveContextWindowInfo`, two bare/scoped id regression tests, and a changelog fix entry.

## Best Possible Solution

Land this PR after making the new matching helper compare scoped prefixes through normalized provider ids and adding an alias regression case.

## Reproduction Assessment

Yes. Source inspection shows current main strict-matches configured ids against the runtime `modelId`, and the runner forwards that runtime id into the guard, so a `provider/model` runtime id misses a bare configured entry.

## Solution Assessment

Not quite. Fixing `resolveContextWindowInfo` is the right boundary, but the helper should mirror the existing normalized provider-prefix matching so alias providers keep working for scoped configured ids.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Normalize provider aliases before stripping scoped model IDs:** `src/agents/context-window-guard.ts:40`
  - body: The new helper builds a literal `${provider}/` prefix. This misses the reverse case when the config stores a scoped id with an alias spelling, such as `z.ai/glm-5`, while runtime passes provider `z-ai` and model `glm-5`; the guard then still falls back instead of using configured `contextTokens`. Reuse normalized provider-prefix parsing here and cover the alias case.
