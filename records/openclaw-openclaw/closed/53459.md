## Summary

Close PR #53459. On current `main`, Docker installs already support `EXA_API_KEY` and `FIRECRAWL_API_KEY` via the mounted `~/.openclaw/.env` global fallback, and the repo docs explicitly say users should not add API keys to `docker-compose.yml`. The PR premise that there are existing hardcoded plaintext keys in `docker-compose.yml` does not match current `main`.

## Evidence

- **Compose file has no EXA/Firecrawl keys to replace:** The current `openclaw-gateway` `environment:` block only includes infra/runtime variables such as `OPENCLAW_GATEWAY_TOKEN`, session keys, and `TZ`; there are no `EXA_API_KEY` or `FIRECRAWL_API_KEY` entries present on `main`.
  - file: [docker-compose.yml:4](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/docker-compose.yml#L4)

- **Docker docs say not to put API keys in docker-compose:** ClawDock docs state that `~/.openclaw/.env` is mounted into the container, OpenClaw loads it automatically, and there is 'No need to add API keys to `docker-compose.yml`'. That directly contradicts the PR's proposed fix path.
  - file: [scripts/clawdock/README.md:199](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/scripts/clawdock/README.md#L199)

## Risks / Open Questions

- There is still a discoverability gap: `.env.example` documents `FIRECRAWL_API_KEY` but not `EXA_API_KEY`, so Docker users may miss the preferred `~/.openclaw/.env` setup path for Exa.
- The split between project-root `.env` for Docker Compose infra vars and `~/.openclaw/.env` for gateway secrets is easy to misunderstand, but that is a docs/usability issue rather than evidence that this PR's reported bug exists on current `main`.
