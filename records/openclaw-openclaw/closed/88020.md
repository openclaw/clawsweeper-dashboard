## Summary

Keep open: the report is still actionable because current recovery handles thrown/rejected thinking errors, but Anthropic and Bedrock transports can surface the same provider rejection as a terminal stream error event that the recovery wrapper forwards instead of retrying.

## What This Changes

The issue asks OpenClaw to recover from Anthropic/Bedrock `Invalid signature in thinking block` failures by stripping stale thinking blocks and retrying instead of hard-failing the session.

## Best Possible Solution

Handle matching Anthropic/Bedrock thinking-signature terminal error events inside the existing recovery wrapper before emitting them, retry once with thinking blocks stripped, and treat any classifier update as supplemental logging/UI coverage rather than the recovery mechanism.

## Reproduction Assessment

Yes by source inspection, with strong report evidence: provider transports can emit the matching rejection as a terminal error event, while the current recovery wrapper only retries thrown/rejected errors. I did not live-reproduce the long-session provider expiry in this read-only pass.

## Solution Assessment

No, the proposed `REPLAY_INVALID_RE`-only fix is not the best complete fix because that classifier is used for formatting/observation, not for the existing retry path. The recovery wrapper should also detect matching terminal stream error events and retry before the hard-fail is surfaced.
