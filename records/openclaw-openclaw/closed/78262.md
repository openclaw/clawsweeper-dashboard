## Summary

Keep open: current `main` has source paths that can still split Feishu topic sessions when the first topic turn lacks a stable topic id and later turns include `thread_id`, and the related merged PRs do not cover that exact payload shape.

## What This Changes

The issue requests stable Feishu `group_topic` session keys so the first message and later replies in the same Feishu topic resolve to one session.

## Best Possible Solution

Resolve or learn a stable Feishu topic id before session routing for first-turn topic messages, add a regression for the missing-topic-id starter event, and preserve the documented normal-group root-id behavior.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only review. The source-level path is clear if the first Feishu topic event lacks `thread_id`/topic `root_id` and a later reply includes `thread_id`, because current routing can choose different topic keys for those two events.

## Solution Assessment

No, current `main` is not a complete solution for the reported payload shape. The narrow maintainable fix is Feishu-owned topic-id normalization before `resolveAgentRoute`, with focused tests and no core routing contract change.
