## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Scope ACP permission to the calling plugin:** `src/config/types.plugins.ts:48`
  - body: The new `plugins.allowAcpSpawn` flag is global, so enabling ACP for one trusted plugin gives every loaded plugin access to `api.runtime.acp.spawn()` and `prompt()`. Mirror the existing `plugins.entries.<id>` plus plugin runtime identity model before exposing host-side agent dispatch.
  - confidence: 0.9
- **[P2] Mark plugin ACP prompts as explicit turns:** `src/plugins/runtime/runtime-acp.runtime.ts:52-67`
  - body: `promptPluginAcp()` calls the gateway `agent` method without `acpTurnSource: "manual_spawn"`, so follow-up prompts into ACP sessions are treated as dispatch rather than explicit plugin/operator turns. Native ACP spawn sets this marker, and plugin prompts should follow the same policy path.
  - confidence: 0.88
- **[P2] Move ACP runtime types to a leaf contract:** `src/plugins/runtime/types-core.ts:1`
  - body: Importing `SpawnAcp*` types from `../../agents/acp-spawn.js` pulls the public plugin runtime type surface into the agents implementation graph. Define plugin-facing DTOs or a leaf contract so the SDK/runtime boundary does not depend on the ACP spawn implementation module.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff introduces privileged plugin-to-ACP dispatch with a global opt-in instead of caller-scoped authorization.
