## Summary

Current main already includes the WhatsApp active-listener registry fix for split runtime chunks, plus regression coverage for the light/heavy runtime boundary that matches this issue's failure mode. The changelog shows it shipped in v2026.3.22.

## Evidence

- **WhatsApp listener registry is now global singleton:** The registry is stored on globalThis behind Symbol.for("openclaw.whatsapp.connectionControllerRegistry"), so split WhatsApp runtime chunks share one controller map instead of each holding isolated state.
  - file: [extensions/whatsapp/src/connection-controller-registry.ts:11](https://github.com/openclaw/openclaw/blob/512d3d2287d509f75a32a2344f1b71b8fbfdb0c1/extensions/whatsapp/src/connection-controller-registry.ts#L11)
  - command: `nl -ba extensions/whatsapp/src/connection-controller-registry.ts | sed -n '1,120p'`
  - sha: [512d3d2287d5](https://github.com/openclaw/openclaw/commit/512d3d2287d509f75a32a2344f1b71b8fbfdb0c1)
- **Outbound send path still resolves through shared registry:** WhatsApp outbound send throws the reported "No active WhatsApp Web listener" error only if the shared registry lookup returns null, so the singleton registry change directly addresses this failure path.
  - file: [extensions/whatsapp/src/send.ts:39](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/extensions/whatsapp/src/send.ts#L39)

- **Regression test covers split light/heavy runtime listener sharing:** The runtime-boundary test stages separate light and heavy WhatsApp runtime modules, registers a listener in one, and asserts the other can read it. That is the same cross-chunk listener-resolution problem described in the issue.
