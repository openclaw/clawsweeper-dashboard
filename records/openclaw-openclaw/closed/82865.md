## Summary

Keep open: current main still contains the reported Docker `--allow-unconfigured` startup path, and the open linked contributor PR is the right implementation path to review before resolving this issue. Source inspection supports the Docker/auth-boundary concern, but the exact live exposure still needs Docker validation because runtime token generation and non-loopback auth guards also run.

## What This Changes

The issue asks OpenClaw to stop official Docker startup from bypassing gateway configuration/auth checks via `--allow-unconfigured`.

## Best Possible Solution

Docker startup should fail closed for exposed or unconfigured gateway auth while preserving explicit token, password, trusted-proxy, and safe loopback onboarding flows.

## Reproduction Assessment

No high-confidence live reproduction was established here. Source inspection confirms the Docker `--allow-unconfigured` command, the guard bypass, and container non-loopback bind path, but a built Docker image should confirm the exact external-access behavior.

## Solution Assessment

Unclear pending linked PR review. Removing or constraining the Docker `--allow-unconfigured` default looks like the likely direction, but it must be checked against Docker onboarding and the existing runtime token/non-loopback guard behavior.
