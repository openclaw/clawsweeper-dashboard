## Summary

Current main already solves the observable offline-node problem behind #55716/#55625. `node.list` and `node.describe` now build a known-node view from pairing stores plus in-memory connected sessions, so offline paired nodes are returned as `connected: false` without invoking or polling the node. The PR's proposed 5s gateway-tool timeout/cache is no longer the necessary fix for this path, and the shipped code preserves the 30s `node.invoke` default that reviewers correctly called out for long-running node commands.

## Best Possible Solution

Close this PR as implemented on main by the shipped known-node catalog/listing behavior. Keep the current separation: `node.list`/`node.describe` read known and connected state without touching offline nodes, while `node.invoke` retains its longer default for real node commands. Track any future per-method client timeout policy or TTL caching as narrower follow-up work only if there is fresh evidence.

## Evidence

This path does not call `nodeRegistry.invoke`, LAN discovery, or any per-node network wait that could block on an offline node for 30s.
