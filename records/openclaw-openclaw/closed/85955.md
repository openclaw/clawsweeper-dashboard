## Summary

Keep open: the PR is a focused fix for the linked session-file contention bug, but the new prompt-holder wait can block later same-file attempts indefinitely when the prior holder is hung or force-cleared before it reaches `reacquireAfterPrompt`.

## What This Changes

The PR serializes embedded runner provider-prompt lock releases per resolved session JSONL file, separates compaction idle waits from prompt ownership, and adds same-file prompt contention regressions.

## Best Possible Solution

Keep the per-session-file prompt serialization, but bind holder waits to abort/recovery lifecycle or a bounded timeout so stale holders fail or clear cleanly instead of blocking the session lane.

## Reproduction Assessment

Yes at source level: a holder that never calls `reacquireAfterPrompt()` leaves `waitForPromptSessionFileTurn()` awaiting `holder.released`, and stuck-run force-clear does not clear that new holder map. I did not execute a live hung-provider harness in this read-only review.

## Solution Assessment

No for the current patch as written: per-file prompt serialization is the right direction, but the holder wait needs a bounded or recovery-owned stale-holder path before merge.
