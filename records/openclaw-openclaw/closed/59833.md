## Summary

Current `main` already contains a shipped Telegram startup fix that matches this symptom: startup control-plane calls (`deleteWebhook`, `getMe`, `setWebhook`) are now hard-bounded instead of hanging indefinitely before polling begins. The changelog records that fix in `v2026.4.5`, and the current Telegram manifest still declares `grammy` runtime dependencies, so the thread's missing-dependency theory is not the best match for the code on `main`.
