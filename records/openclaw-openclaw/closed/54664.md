## Summary

Current `main` does not implement the requested stale-`list_pages` recovery, and the PR's latest revision still has an unresolved concurrent-session eviction risk, so this is not a safe conservative close.

## Evidence

- **Main still throws tool-level `list_pages` errors without recovery:** In `callTool`, current `main` only tears down the session on thrown transport/timeout/abort failures. Any MCP tool error (`result.isError`) immediately throws `extractToolErrorMessage(...)` and returns no retry path.

  - command: `nl -ba extensions/browser/src/browser/chrome-mcp.ts | sed -n '292,381p'`
  - sha: [a6aa151653ab](https://github.com/openclaw/openclaw/commit/a6aa151653ab478e9aa23e95c6446c41adc4fce1)
- **`list_pages` has no stale-session special case on main:** `listChromeMcpPages` directly calls `callTool(profileName, userDataDir, "list_pages")`, so the reported stale selected-page failure would still surface unchanged on current `main`.
  - file: [extensions/browser/src/browser/chrome-mcp.ts:434](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/extensions/browser/src/browser/chrome-mcp.ts#L434)

Under concurrent `list_pages` calls, that can evict a freshly created replacement session and leak its MCP process handle. There is no follow-up commit or response in the provided timeline after that review.
