## Summary

Keep open: the underlying final-media bug is real and the PR includes sufficient after-fix logs, but the current head can replay media-only `text_end` replies at `message_end` because the new empty final-flush path emits media without marking that reply delivered.

## What This Changes

The PR changes embedded Pi reply-stream handling and tests to preserve final `MEDIA:` directives and queued structured tool media through outbound replies.

## Best Possible Solution

Land a rebased patch that final-consumes trailing directives exactly once, records delivery state for media-only final `text_end` replies, and keeps the related dispatcher-normalization issue linked at https://github.com/openclaw/openclaw/issues/78697.

## Reproduction Assessment

Yes. Source inspection shows current main can leave a trailing `MEDIA:` line in `pendingTail` until final consume, and the PR's focused test models that `text_end` path; I did not run tests in this read-only review.

## Solution Assessment

No. The direction is right, but the final empty-flush delivery must mark or dedupe media-only `text_end` output before `message_end` can consume the same final text again.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86
