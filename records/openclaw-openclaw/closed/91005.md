## Summary

Close: current main already addresses the silent Telegram polling freeze path. Startup `getMe` is bounded and cached, successful bot info is handed to grammY to avoid a second startup `getMe`, and polling liveness now forces restart/status updates instead of leaving the worker wedged.

## What This Changes

The issue reports that Telegram long-polling can silently stop after a `getMe` fetch timeout without a channel-exit/reconnect signal, leaving health monitoring unaware.

## Best Possible Solution

Keep the shipped Telegram startup and polling-liveness implementation; ask for a new report only if a build at or after `v2026.6.6` still shows the silent `getMe` freeze.

## Reproduction Assessment

No high-confidence current-main failing reproduction remains. The reporter's production logs are credible for older builds, while current source and regression coverage now exercise bounded startup and polling-stall recovery paths.

## Solution Assessment

Yes. The current implementation fixes the owner-owned failure points directly: Telegram startup bounds/reuses `getMe`, and Telegram polling liveness restarts stalled workers without adding an external watchdog or new config surface.
