## Summary

Keep open: current main still implements daily reset as a lazy freshness check on later non-system turns, and there is still no gateway-owned wall-clock sweep that would rotate an idle session exactly at the configured 4am boundary with no incoming message.

## What This Changes

The issue asks for daily `session.reset` with `atHour: 4` to rotate sessions at wall-clock time even when no messages arrive after the boundary.

## Best Possible Solution

Add a gateway-owned daily reset sweep that rolls over eligible configured sessions at the wall-clock boundary while preserving lazy reset fallback and the existing system-event guard.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: create a daily-reset session before 4am, allow the boundary to pass with no non-system turn, and current main has no scheduler that would rotate it. I did not run tests because this review is read-only.

## Solution Assessment

Yes: the maintainable fix is a dedicated gateway lifecycle sweep, not changing heartbeat or cron message handling to reset sessions. That preserves the current system-event safety boundary while satisfying the documented daily reset contract.
