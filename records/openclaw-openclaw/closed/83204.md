## Summary

Current main already fixes the Telegram isolated-ingress control-command blockage: `/status` and `/stop` are classified onto a Telegram control lane, and spooled updates now drain per lane instead of awaiting one wedged topic update before later control updates. The broader stuck-session and Codex-runtime causes remain tracked separately.

## What This Changes

The issue asks for Telegram `/stop` and `/status` to bypass a wedged per-topic isolated ingress/spool lane so users can inspect or cancel stuck runs from chat.

## Best Possible Solution

Keep the current main control-lane and per-lane isolated-ingress spool implementation, and use the linked canonical issues for any remaining stuck-run root-cause recovery work.

## Reproduction Assessment

Yes. Source inspection of v2026.5.12 shows isolated ingress awaited each spooled update in one FIFO loop, so a wedged normal topic update could block later `/status` or `/stop`; current-main tests model the same path without a live Telegram run.

## Solution Assessment

Yes. The current main fix is the narrow maintainable shape: classify Telegram control commands onto a control lane and drain isolated spool work per lane without adding a new config knob or changing normal per-topic ordering.
