## Summary

Keep open. Current main still has the startup-aborting batch resolution behavior, and this PR is the active focused fix for the linked stale auth-profile SecretRef bug; I did not find a blocking correctness issue in the diff.

## What This Changes

The PR separates config SecretRef resolution from auth-store SecretRef resolution, downgrading provider-scoped auth-profile resolution failures to a warning while keeping config SecretRefs fail-fast.

## Best Possible Solution

Land this PR or an equivalent narrow fix after CI confirms the secrets runtime tests, preserving fail-fast config SecretRefs while isolating stale generated auth-profile provider refs to the affected profile.

## Reproduction Assessment

Yes. A current-main auth-store profile whose keyRef points at a missing exec provider reproduces through resolveConfiguredProvider throwing, resolveSecretRefValues stopping, and startup activation wrapping the preparation error as a required-secrets startup failure.

## Solution Assessment

Yes. Splitting config assignments from auth-store assignments is the narrow maintainable boundary: required config secrets still fail startup, while generated auth profiles can degrade to normal unavailable-credential handling.
