## Summary

Current main now implements the requested Windows gateway launcher behavior: generated gateway service installs merge the generated service environment before choosing the Scheduled Task action, so the hidden-launcher marker creates and targets `gateway.vbs`. The latest shipped tag inspected here still shows the old `env` lookup, so this is a current-main-only fix rather than a proven shipped release fix.

## What This Changes

The issue reports that Windows `openclaw gateway install --force` on v2026.6.10 does not create `gateway.vbs`, while the Scheduled Task behavior is inconsistent across reinstall.

## Best Possible Solution

Keep the current-main hidden-launcher implementation and ask for a fresh report only if a clean build from a commit after f5148aff2505020cb01bfffc313aa7cfd768fd59 still shows Task Scheduler targeting `gateway.cmd`.

## Reproduction Assessment

Yes. Source comparison shows the released path can miss the generated hidden-launcher marker, while current main resolves the launcher from the merged task environment and has regression coverage for `gateway.vbs`.

## Solution Assessment

Yes. Reusing the existing hidden-launcher path at the generated service-env boundary is the narrow maintainable fix and avoids adding another launcher mechanism or config option.
