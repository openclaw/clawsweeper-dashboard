## Summary

Keep open: current main still has the coalesced in-flight restart path that returns success before the restart tool's continuation-writing hook can run, and the related attempted fixes are closed unmerged rather than superseding this narrow bug.

## What This Changes

The issue reports that `gateway.restart` can accept `continuationMessage` but silently drop it when the request coalesces with an already in-flight gateway restart.

## Best Possible Solution

Fix the restart tool and scheduler contract so an accepted continuation is queued exactly once for the correct pending restart, or the response explicitly reports that no continuation was queued.

## Reproduction Assessment

Yes, source-reproducible. With an emitted but unconsumed restart signal, a `gateway.restart` call with `continuationMessage` reaches the early coalesced return before `beforeEmit` can write the sentinel.

## Solution Assessment

Yes, the best path is a narrow gateway restart contract fix with session-safe continuation handling. The broader checkpoint and auto-resume RFC is useful separately, but it is not the minimal fix for this accepted one-shot continuation.
