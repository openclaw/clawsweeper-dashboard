## Summary

Keep open for maintainer review: current main still accepts reversed explicit usage date ranges, and the PR is a focused, well-proven fix, but changing two gateway RPCs from successful responses to `INVALID_REQUEST` is a compatibility-sensitive contract choice.

## What This Changes

This PR replaces shared usage date parsing with `resolveDateRange`, rejects invalid or inverted explicit date ranges before the `usage.cost` and `sessions.usage` loaders, and updates focused gateway tests.

## Best Possible Solution

Land the centralized validation after maintainers accept the stricter gateway RPC contract; otherwise ask for a compatibility-preserving design for reversed explicit ranges.

## Reproduction Assessment

Yes. Source inspection shows current main builds an inverted `parseDateRange` window for valid reversed explicit dates and passes it into both usage loaders; the PR body also includes parent negative-control terminal proof and a forward-range control.

## Solution Assessment

Yes, pending maintainer compatibility acceptance. Centralizing invalid-date and inverted-range checks in `resolveDateRange` is cleaner than duplicating handler guards, and schema- or UI-only validation would not cover both gateway RPC methods and direct clients.
