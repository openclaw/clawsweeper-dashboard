## Summary

Current main already solves the user-visible leak class by keeping heartbeat/runtime-only prompt text out of visible transcript prompts and by projecting chat/session history through shared filters for runtime envelopes, heartbeat prompts, and heartbeat acknowledgements. The PR branch should not be merged as-is because its provenance-based hiding gives caller-supplied `internal_system` provenance display-hiding authority without a trusted-ingress gate.

## What This Changes

The PR branch tags heartbeat, cron, and exec-event turns with `InputProvenance.kind = "internal_system"`, filters those provenance-tagged user messages from Gateway and Control UI history, adds focused regressions, and includes unrelated formatting/import churn.

## Best Possible Solution

Close this PR and keep the current main runtime-only transcript/display-projection implementation. If provenance-based hiding is revisited later, `internal_system` should be set only by trusted internal origins and stripped or rejected from untrusted gateway ingress before any history filter treats it as authority.

## Reproduction Assessment

Yes. The original leak class is now covered by focused current-main tests for heartbeat transcript placeholders, hidden internal runtime context, session-history projection, and Control UI reload filtering; the PR branch trust regression is reproducible by sending an `agent` request with `inputProvenance.kind = "internal_system"` and then reading history under the branch.

## Solution Assessment

Yes for current main, but no for the PR branch as-is. Runtime-only transcript prompts plus a shared display projection are the narrower maintainable fix; provenance-based suppression needs a trusted-origin gate before it can be safe.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Gate hidden provenance on trusted ingress:** `src/gateway/chat-display-projection.ts:396`
  - body: The new history filter drops user messages solely because `provenance.kind === "internal_system"`, but the `agent` RPC accepts caller-supplied `inputProvenance` and forwards it into persistence without the admin-scope gate used by `chat.send` system provenance. A write-scope caller could persist a user turn with this kind and make it disappear from chat/session history, so strip or reject `internal_system` at untrusted ingress and only set it from internal code.
  - confidence: 0.86

Concerns:

The branch should gate or strip that kind unless trusted internal code set it.
