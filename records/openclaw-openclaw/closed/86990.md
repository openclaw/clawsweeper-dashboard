## Summary

Keep open: the new `/skill` usage handler can reject valid native slash skill invocations before the real workspace skill list is loaded, and the PR still needs real command behavior proof before merge.

## What This Changes

The branch adds a `/skill` usage/unknown handler, wires it into command handlers, updates command-info tests, and skips full session-store maintenance for command-only session writes.

## Best Possible Solution

Keep the early bare `/skill` help path, but let named `/skill` invocations continue unless a loaded skill list proves the target is unknown, then add focused native-fast-path coverage and real command proof.

## Reproduction Assessment

Yes from source inspection: the native slash fast path passes `skillCommands: []` into `handleCommands`, and the new handler treats that empty list as enough to reject named `/skill` invocations. I did not run a test because this review was read-only.

## Solution Assessment

No as submitted: the idea of handling bare `/skill` early is reasonable, but the implementation must distinguish an unloaded skill list from a known empty list so valid skill invocations can still dispatch.
