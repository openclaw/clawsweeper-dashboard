## Summary

Current main stores session header timestamps as true UTC ISO 8601 strings, and the in-repo readers/display paths parse and localize them correctly. The reported 8-hour shift appears to come from reading the raw `Z` string as if it should already be local wall time, which does not match the current implementation or JavaScript date semantics.

## Evidence

- **Session header write path uses UTC ISO string:** `ensureSessionHeader()` writes `timestamp: new Date().toISOString()` into the JSONL session header. `toISOString()` serializes the absolute instant in UTC and is not a local-time string.
  - file: [src/config/sessions/transcript.ts:30](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/src/config/sessions/transcript.ts#L30)
  - command: `nl -ba src/config/sessions/transcript.ts | sed -n '18,36p'`
  - sha: [45146913007d](https://github.com/openclaw/openclaw/commit/45146913007da723b4f06aba6a66cf4a695974c0)
- **Transcript readers parse string timestamps as instants:** Session-file ingestion treats string timestamps via `Date.parse(value)` and converts them to epoch milliseconds, so a stored `...Z` timestamp is interpreted as UTC and preserved as the same instant.
  - file: [src/memory-host-sdk/host/session-files.ts:409](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/src/memory-host-sdk/host/session-files.ts#L409)
  - command: `nl -ba src/memory-host-sdk/host/session-files.ts | sed -n '409,428p'`
  - sha: [45146913007d](https://github.com/openclaw/openclaw/commit/45146913007da723b4f06aba6a66cf4a695974c0)
- **User-facing export localizes header timestamp:** The HTML session export renders `new Date(header.timestamp).toLocaleString()`, which converts the stored UTC instant to the viewer's local timezone for display.
  - file: [src/auto-reply/reply/export-html/template.js:1556](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/src/auto-reply/reply/export-html/template.js#L1556)
  - command: `nl -ba src/auto-reply/reply/export-html/template.js | sed -n '1554,1557p'`
  - sha: [45146913007d](https://github.com/openclaw/openclaw/commit/45146913007da723b4f06aba6a66cf4a695974c0)
- **Timezone reproduction in Asia/Shanghai matches expected behavior:** A read-only Node repro with `TZ=Asia/Shanghai` showed that `2026-03-19T12:00:00+08:00` serializes to `2026-03-19T04:00:00.000Z` and parses back to local noon, proving the `Z` form is the correct UTC representation of the same moment, not a mislabeled local timestamp.
  - command: `TZ=Asia/Shanghai node -e 'const d=new Date("2026-03-19T12:00:00+08:00"); console.log(JSON.stringify({iso:d.toISOString(), parsed:new Date(d.toISOString()).toString(), local:d.toString(), epoch:d.getTime()}))'`

## Risks / Open Questions

- Users inspecting raw JSONL files may still find UTC `Z` timestamps confusing if they expect stored values to already be formatted in local wall time.
- This review did not use `gh` because GitHub CLI auth was unavailable in the environment; the decision is based on current-main code inspection and a local read-only date-semantics repro.
