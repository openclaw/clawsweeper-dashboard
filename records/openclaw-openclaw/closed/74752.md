## Summary

Keep open: current main still has the TTS local-media filter gap, and open PR #74901 explicitly closes this issue. Telegram voice delivery exists once the media/audioAsVoice payload survives queueing, so the issue should wait for that PR to land or be replaced.

## What This Changes

The issue asks for the `tts` tool to automatically deliver generated Microsoft TTS audio to Telegram as a voice message instead of only producing `(spoken)` text.

## Best Possible Solution

Review, land, or replace PR #74901 with a narrow filter-boundary fix that lets structured TTS `trustedLocalMedia` survive when `builtinToolNames` is present but missing raw `tts`, plus regression coverage for that case and Telegram voice payload delivery.

## Reproduction Assessment

Yes. ## Solution Assessment

Yes for the linked repair direction, no for current main. The narrow maintainable solution is to honor trusted structured TTS media at the filter boundary while preserving the existing raw-name and MCP protections for untrusted local paths.
