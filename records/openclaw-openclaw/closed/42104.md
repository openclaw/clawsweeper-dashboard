## Summary

Current `main` and the latest release do not implement a plugin-side mandatory keyword filter for Synology Chat. The webhook handler forwards any inbound payload with `token`, `user_id`, and `text`, and only strips an upstream `trigger_word` when Synology itself includes that field. The report appears to describe Synology outgoing-webhook trigger configuration rather than an OpenClaw bug.

## Risks / Open Questions

- If Synology Chat is configured upstream to emit webhooks only for a trigger word, users can still experience keyword-gated behavior before requests reach OpenClaw.
- The docs do not explicitly call out that keyword-triggered routing is a Synology-side integration choice, so similar reports may recur.

## Close Comment

Thanks for the report. I reviewed this with Codex against current `main` (`7ac35b4f69009490180a35bf404638d4280a4865`) and the latest release `v2026.4.22`, and I could not confirm an OpenClaw-side bug here.

The Synology Chat plugin does not have a keyword-filter setting in its config surface (`extensions/synology-chat/src/config-schema.ts:4`, `extensions/synology-chat/src/types.ts:5`). The webhook handler accepts any inbound payload that includes `token`, `user_id`, and `text` (`extensions/synology-chat/src/webhook-handler.ts:294`), and it only strips a prefix when Synology itself sends a `trigger_word` field (`extensions/synology-chat/src/webhook-handler.ts:461`). The tests reflect that behavior as well (`extensions/synology-chat/src/webhook-handler.test.ts:498`).

That means the mandatory-keyword behavior described here looks like Synology outgoing-webhook trigger configuration happening before OpenClaw receives the message, rather than a plugin parser requirement inside OpenClaw. On that basis, I’m closing this as not reproducible in current OpenClaw. If you can reproduce a case where Synology sends OpenClaw a normal webhook payload without `trigger_word` and OpenClaw still drops or empties the message, please open a new issue with the exact raw payload shape or gateway logs.
