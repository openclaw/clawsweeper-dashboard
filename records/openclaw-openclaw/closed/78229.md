## Summary

Keep open: the PR targets a real plugin startup regression, but the current patch is not merge-ready because it broadens startup intent to any explicitly enabled non-bundled plugin, including plugins whose manifest explicitly opts out of startup, and it still lacks after-fix real gateway proof.

## What This Changes

The PR treats explicitly enabled non-bundled plugins as legacy hook startup candidates and adds positive and negative startup planner tests.

## Best Possible Solution

Restore legacy external hook startup for the reported enabled-only hook plugins while preserving manifest startup opt-outs and later disable, deny, allowlist, and activation-state gates, then prove it in a real gateway/container run.

## Reproduction Assessment

Yes, source-level. Current main filters enabled-only legacy external hook plugins before Gateway onlyPluginIds, and the linked bug report supplies Docker logs plus hook side-effect checks.

## Solution Assessment

No, not in this form. The startup-planner direction is narrow and appropriate, but this patch also bypasses activation.onStartup false for non-bundled plugins and still needs real gateway proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Preserve explicit startup opt-out for non-bundled plugins:** `src/plugins/gateway-startup-plugin-ids.ts:442-443`
  - body: This shortcut returns true for every non-bundled plugin with `entries.<id>.enabled=true` before the later startup planner can honor a manifest `activation.onStartup: false`. The existing `demo-global-startup-opt-out` fixture is a global plugin with that opt-out, and its test still expects an enabled+allowed config to produce no startup plugin IDs; with this branch it would now be loaded into the gateway.
  - confidence: 0.95

## Security Review

Status: needs_attention

Summary: The diff can unintentionally start native non-bundled plugin code in the Gateway despite an explicit startup opt-out.

Concerns:

- **[medium] Avoid widening native plugin startup execution:** `src/plugins/gateway-startup-plugin-ids.ts:442`
  - body: The new enabled-only shortcut makes any explicitly enabled non-bundled plugin a Gateway startup candidate, even when its manifest says `activation.onStartup: false`. Native plugins run in-process with Gateway privileges, so preserving explicit startup opt-outs is part of the security boundary for when plugin code executes.
  - confidence: 0.9
