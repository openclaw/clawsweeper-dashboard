## Label Justifications

- `P1`: The reported behavior can leave completed subagent work invisible to the requester during provider-load failures, which breaks an active agent workflow.
- `impact:message-loss`: The user-visible failure is a completed subagent wake/completion message not reaching the parent/requester.
- `impact:session-state`: The parent session, subagent lifecycle, and task ledger can disagree about whether completion delivery actually happened.

## Summary

Close as duplicate/superseded: the original fallback-cooldown classifier gap is fixed on current main, and the remaining cooldown-aware retry and durable delivery-status semantics are already covered by the open canonical subagent delivery trackers.

## What This Changes

The issue asks for sub-agent announce delivery to treat FallbackSummaryError/provider-cooldown failures as transient, retry visibly with cooldown-aware delays, and keep task delivery status truthful after exhaustion.

## Best Possible Solution

Keep the current classifier and task-status improvements on main, and consolidate the remaining cooldown-aware and durable completion-delivery work under https://github.com/openclaw/openclaw/issues/44925 and https://github.com/openclaw/openclaw/issues/67777.

## Reproduction Assessment

Yes. Source inspection shows the original classifier failure is fixed, while the remaining cooldown-expiry delay gap still reproduces because announce retry uses fixed delays and does not read `soonestCooldownExpiry`.

## Solution Assessment

Yes for cleanup. The best triage outcome is to close this duplicate tracker while keeping the remaining product/code work on the canonical subagent delivery issues rather than splitting the same reliability problem across several open reports.
