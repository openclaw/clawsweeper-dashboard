## Summary

Keep open: current main still has the Telegram plugin-approval mismatch where fallback suppression rejects a missing turnSourceChannel before consulting native-delivery or session-binding facts, while Telegram native handling can still route the same request from the session key.

## What This Changes

The issue reports duplicate Telegram plugin-approval prompts when a request has a session key but no turnSourceChannel, allowing both fallback forwarding and native Telegram delivery.

## Best Possible Solution

Align shared fallback suppression with the same trusted session-binding and native-delivery route facts used by Telegram native handling, and add regression coverage for null-channel Telegram plugin approvals plus explicit non-Telegram guard cases.

## Reproduction Assessment

Yes, at source level. A Telegram session-store entry plus a plugin approval request with sessionKey and no turnSourceChannel lets native Telegram resolve a session-bound route while fallback suppression returns false before checking native delivery.

## Solution Assessment

Yes. The narrow maintainable fix is in the shared suppression/session-binding path; fixing only the related metadata propagation issue at https://github.com/openclaw/openclaw/issues/74003 would not cover other null-channel approval callers.
