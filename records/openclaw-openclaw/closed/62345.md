## Summary

Current main and the shipped v2026.4.27 release cover the reported Discord proxy/undici failure class: Discord REST clients now receive the configured proxy-aware fetch, that fetch uses undici with a per-request ProxyAgent, JSON request bodies still set Content-Type explicitly, and the slash-command path defers then uses follow-up response semantics instead of blocking the event queue. The linked #71383 fix also landed the Discord undici multipart side of the same production class.

## What This Changes

The issue reports Discord final-reply edits and slash commands failing behind a configured HTTP proxy after the 2026.4.5 undici v8 upgrade, especially when a user preload installs an EnvHttpProxyAgent global dispatcher.

## Best Possible Solution

Keep the shipped Discord REST proxy implementation and documented managed proxy path, with any future regressions narrowed to a focused test around global dispatcher interaction rather than reopening the broad undici v8 upgrade report.

## Reproduction Assessment

Yes. The report gives a concrete proxy-only Discord setup and specific final-reply/slash-command failures; current-main inspection shows the configured Discord REST path now uses injected proxy-aware undici fetch and updated slash handling instead of the failing global-fetch behavior.

## Solution Assessment

Yes. The current direction is the narrow maintainable fix: route Discord REST through the configured per-channel/per-account proxy fetch and keep slash responses on Discord's defer/follow-up path, rather than pinning undici globally or depending on an operator preload script.
