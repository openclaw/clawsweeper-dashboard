## Summary

Close as ClawHub scope: this is an optional new multi-agent memory-governance plugin proposal, and current OpenClaw policy/docs route optional plugins to external plugin distribution. Current main exposes generic plugin SDK seams for tools, commands, hooks/services, gateway methods, memory prompt/corpus supplements, and public memory artifacts, while the named memory-graph store is not present in this repo.

## What This Changes

The issue proposes a new `@openclaw/plugin-fleet-bridge` plugin with typed prior/posterior/dissent records, append-only SQLite invariants, compaction rules, and guidance on memory plugin home/migration strategy.

## Best Possible Solution

Keep OpenClaw core lean and build this as an external code plugin distributed through ClawHub, using the existing plugin SDK seams. If an external prototype exposes a concrete missing memory/plugin API, open a narrower SDK issue or PR for that specific seam.

## Reproduction Assessment

Not applicable; this is an RFC for a new optional plugin, not a bug report. The relevant verification path is source/docs inspection of current plugin and memory extension points, which supports external implementation.

## Solution Assessment

Yes; routing this to ClawHub/external plugin development is the narrowest maintainable path. Adding a bundled `@openclaw/*` plugin to core would conflict with the current optional-plugin policy unless a concrete missing SDK seam is first demonstrated.
