- `impact:session-state`: The wrong agent session key can persist conversation state under the default agent instead of the requested agent.

## Summary

This is the same unresolved Gateway OpenAI-compatible session-routing bug already tracked by the older open report, so the safest cleanup is to close this duplicate and keep the canonical issue as the fix target.

## What This Changes

Make OpenAI-compatible chat completions preserve the selected agent when `x-openclaw-session-key` is provided as an unprefixed session key.

## Best Possible Solution

Use https://github.com/openclaw/openclaw/issues/74679 as the canonical fix target and make the shared Gateway OpenAI-compatible request context scope unprefixed explicit session keys to the selected agent while preserving already agent-prefixed keys.

## Reproduction Assessment

Yes from source inspection: `resolveSessionKey` returns the header unchanged, and the ingress command path then scopes unprefixed explicit keys without the model-resolved agent override. I did not run the live Gateway flow because this was a read-only review.

## Solution Assessment

Yes, closing this duplicate is the best cleanup path; the code fix should land against the older canonical issue with focused coverage for unprefixed header keys plus `model: "openclaw/<agentId>"`.
