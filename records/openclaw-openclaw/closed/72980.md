## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Override inherited session markers:** `src/agents/bash-tools.exec.ts:1615-1617`
  - body: The guard checks the already-merged environment, so an ambient `OPENCLAW_SESSION_KEY` from the gateway process or sandbox defaults prevents `defaults.sessionKey` from being written. That breaks the promised runtime guarantee unless the caller explicitly overrides via `params.env`, so the resolver needs to distinguish explicit request env from inherited/default env.
  - confidence: 0.9
- **[P2] Keep session markers out of node host env:** `src/agents/bash-tools.exec.ts:1615-1617`
  - body: This block runs before the `host === "node"` branch even though the PR says node execution is out of scope. Current node execution sends only `requestedEnv` to `system.run` while approval analysis reads the full request env, so a node command can be analyzed with the marker and run remotely without it.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff has no supply-chain change, but it exposes session routing metadata through subprocess environments without a settled trusted-marker contract.

Concerns:

- **[medium] Session key becomes subprocess environment data:** `src/agents/bash-tools.exec.ts:1615`
  - body: `OPENCLAW_SESSION_KEY` would be inherited by child processes and descendants, where it can be logged, printed, captured in diagnostics, or inspected by tools that read process environments; maintainers should accept and document that exposure in the canonical implementation.
  - confidence: 0.86
- **[medium] Ambient env can spoof the runtime marker:** `src/agents/bash-tools.exec.ts:1615`
  - body: Because the proposed guard preserves an inherited `OPENCLAW_SESSION_KEY`, stale or operator-injected ambient env can be passed to commands as if it were the current runtime session marker.
  - confidence: 0.86
