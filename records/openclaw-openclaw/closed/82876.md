## Summary

Keep open. The PR targets a plausible linked macOS rollback/gateway loop, but the patch introduces blocking issues: rate-limited protocol mismatches no longer send the detail code the client needs to stop reconnecting, and doctor --fix can remove update launchd jobs during updater-driven doctor runs. The PR also has only test/stub proof, not real macOS rollback or launchd behavior proof.

## What This Changes

The PR adds a PROTOCOL_MISMATCH detail code, pauses GatewayClient reconnects for that code, removes stale macOS updater launchd jobs during doctor --fix, and rate-limits repeated protocol mismatch handling by remote address.

## Best Possible Solution

Keep the protocol mismatch error detail available whenever the server closes for a mismatch, gate updater-job removal out of update-in-progress runs unless the job is proven inactive, and add focused tests plus redacted macOS rollback or launchd proof.

## Reproduction Assessment

No high-confidence end-to-end reproduction is available. Source and the linked issue logs support the protocol-mismatch reconnect mechanics, but no live macOS rollback proof identifies the stale process on current main.

## Solution Assessment

No. The direction is plausible, but the current patch suppresses the detail needed to pause reconnects and auto-removes update jobs too broadly during updater-driven doctor runs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep sending mismatch details when rate limiting:** `src/gateway/server/ws-connection/message-handler.ts:602-604`
  - body: After the fifth mismatch from an address, this branch closes before sending the error frame with `details.code = PROTOCOL_MISMATCH`. `GatewayClient` only pauses reconnect after reading that detail from the failed connect response, so a stale local client can consume the quota and leave later incompatible clients from the same loopback address retrying without the new stop signal. Rate-limit the log, or otherwise keep a machine-readable mismatch signal on the close path.
  - confidence: 0.88
- **[P2] Do not remove updater jobs during update doctor runs:** `src/flows/doctor-health-contributions.ts:390-392`
  - body: This calls the remover for every `doctor --fix` run. The updater itself runs `doctor --non-interactive --fix` with `OPENCLAW_UPDATE_IN_PROGRESS=1`, and the stale-updater scanner currently treats any `ai.openclaw.update.*` launchd row as removable, even one with a live pid. That can remove the active updater job mid-update; gate this repair during update-in-progress runs or require proof the job is inactive before removing it.
  - confidence: 0.84
