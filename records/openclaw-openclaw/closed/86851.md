## Summary

Keep open: current `main` still drops Zalo reply-quote id/body metadata before agent context is built, while the common inbound context path already supports reply metadata for other channels.

## What This Changes

The issue requests forwarding Zalo `data.quote` metadata into the agent context as standard reply/quote metadata.

## Best Possible Solution

Add Zalo quote extraction in the plugin-owned inbound path and feed it into the existing standard reply/quote context fields, with focused regression coverage and no config or core behavior change.

## Reproduction Assessment

Yes from source inspection: a Zalo message with `data.quote.globalMsgId`/`msg` reaches `toInboundMessage()`, but those fields are not stored or passed into `buildContext()`. I did not run a live Zalo credential repro in this read-only review.

## Solution Assessment

Yes, the narrowest maintainable fix is in the Zalo plugin inbound parser and monitor context assembly. Prefer the existing `supplemental.quote`/`ReplyTo*` path over an ad hoc core change, and keep the existing `implicitMention` owner check intact.
