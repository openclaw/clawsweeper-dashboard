## Summary

Issue #41505 is already implemented on current `main` and in release `v2026.4.22`: WhatsApp inbound runtime resolves the active account first, injects that account's `ackReaction`/`allowFrom` into the working config, and group gating consumes the resolved account-scoped allowlist rather than the root-level list.

## Evidence

- **Inbound runtime uses resolved account overrides:** `startWebAutoReplyMonitor` resolves the active WhatsApp account and rewrites the working WhatsApp config with `account.ackReaction`, `account.allowFrom`, `account.groupAllowFrom`, and related fields before message processing.
  - file: [extensions/whatsapp/src/auto-reply/monitor.ts:119](https://github.com/openclaw/openclaw/blob/bae7b54a85b17d180e4bfc28973a7e4fd1646a93/extensions/whatsapp/src/auto-reply/monitor.ts#L119)
  - command: `nl -ba extensions/whatsapp/src/auto-reply/monitor.ts | sed -n '108,160p'`
  - sha: [bae7b54a85b1](https://github.com/openclaw/openclaw/commit/bae7b54a85b17d180e4bfc28973a7e4fd1646a93)
- **Inbound policy derives allowFrom from resolveWhatsAppAccount:** `resolveWhatsAppInboundPolicy` calls `resolveWhatsAppAccount(...)`, sets `configuredAllowFrom = account.allowFrom ?? []`, and derives self-chat/group fallback behavior from that resolved account-scoped list.
  - file: [extensions/whatsapp/src/inbound-policy.ts:84](https://github.com/openclaw/openclaw/blob/bae7b54a85b17d180e4bfc28973a7e4fd1646a93/extensions/whatsapp/src/inbound-policy.ts#L84)

Current `main` does that, and the only production caller is on that path.
