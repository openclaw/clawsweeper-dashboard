## Summary

Keep this PR open: current main can leak heartbeat ACK text into WebChat when `showOk` is true, and the production change is narrow, but the PR is not merge-ready because contributor-supplied real WebChat proof is still insufficient and the new regression test would pass against the old implementation.

## What This Changes

The PR removes WebChat heartbeat suppression's dependency on `resolveHeartbeatVisibility(..., channel: "webchat")` so heartbeat chat output is hidden for heartbeat runs regardless of `showOk`, and adds a server-chat regression test.

## Best Possible Solution

Land the narrow Gateway fix after the test truly covers `showOk: true` and the contributor supplies real WebChat behavior proof, while keeping external-channel `showOk` delivery in the heartbeat runner.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main reads `channels.defaults.heartbeat.showOk` for WebChat and returns `!showOk`, so `showOk: true` can bypass heartbeat chat suppression. I did not run a live Gateway/WebChat reproduction in this read-only review.

## Solution Assessment

Yes, the production direction is the narrow maintainable fix for the WebChat `HEARTBEAT_OK` leak because it separates internal chat-stream suppression from external channel delivery. The PR still needs a stronger regression test and real behavior proof before merge.
