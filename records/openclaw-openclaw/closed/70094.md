## Summary

Current main already handles the stated supervisor/session-lock problem at the process-tree and session-lock layers. The PR's SIGTERM-only change is superseded by the existing adapter path: supervisor cancellation still calls the hard-kill adapter API, but that API routes child/PTY processes through graceful process-tree termination, sending SIGTERM first and escalating to SIGKILL after a grace period. Session write locks also have termination-signal cleanup, stale-lock reclamation, and startup cleanup coverage.

## Best Possible Solution

Close this PR and keep the current main behavior: supervisor cancellation should continue using the adapter force-termination API because the adapter already performs SIGTERM-first process-tree shutdown with SIGKILL fallback and wait settlement. If a current release still leaves agent session locks behind, it should be tracked as a fresh focused bug with logs/reproduction against the current process-tree and session-lock cleanup paths.
