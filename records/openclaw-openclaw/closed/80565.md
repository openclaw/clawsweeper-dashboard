## Summary

Current main already addresses the Docker pnpm/A2UI resync problem: the Dockerfile constrains install/prune to the target Linux CPU/libc and disables pnpm dependency verification before the post-copy build scripts, including the A2UI bundle step, with regression coverage in place.

## What This Changes

The issue requests that arm64 Linux Docker builds stop downloading optional native binaries for unrelated platforms during the A2UI bundling step.

## Best Possible Solution

Keep the landed Docker pnpm 11 build policy, and reopen only with fresh current-main arm64 logs if Docker still fetches non-target optional binaries during the A2UI step.

## Reproduction Assessment

No high-confidence current-main failing repro was run in this read-only review. The old failure path is source-proven by the issue logs and the merged Docker fix, and current main now has explicit settings plus regression tests for that path.

## Solution Assessment

Yes. The current fix is the narrow maintainable solution because it keeps the existing A2UI fallback and prevents pnpm from re-verifying or resyncing dependencies during post-copy Docker script runs, instead of adding a new user-facing build flag.
