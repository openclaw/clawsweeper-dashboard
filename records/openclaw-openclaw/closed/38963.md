## Summary

Current `main` already carries resolved direct-message sender names through session metadata into Gateway session rows and renders that human-readable name alongside the raw session key in the Dashboard session list.

## Evidence

- **Feishu still resolves sender names:** When `resolveSenderNames` is enabled, Feishu looks up the sender display name and stores it on the inbound context as `senderName`, which is the source data the issue asked to surface.
  - file: [extensions/feishu/src/bot.ts:342](https://github.com/openclaw/openclaw/blob/4d7c4b3298920c0e6e2c5cc8bd7a1546e8122abe/extensions/feishu/src/bot.ts#L342)

  - sha: [4d7c4b329892](https://github.com/openclaw/openclaw/commit/4d7c4b3298920c0e6e2c5cc8bd7a1546e8122abe)
- **Direct sessions derive a human-readable conversation label from SenderName:** For direct chats, `resolveConversationLabel` prefers `SenderName`, and session metadata persists that label into `origin.label`. That gives session rows a readable name even when the stored key is a raw peer ID.
  - file: [src/channels/conversation-label.ts:38](https://github.com/openclaw/openclaw/blob/60956ba6ac96372285cd2b82f6203eaa205b341e/src/channels/conversation-label.ts#L38)

- **Gateway session rows expose displayName from persisted/session-derived metadata:** `buildGatewaySessionRow` sets `displayName` from `entry.displayName`, then group naming, then `entry.label`, then `origin.label`. For a direct Feishu session with a resolved sender name, that yields the readable sender name in the row returned to the Dashboard.
  - file: [src/gateway/session-utils.ts:1256](https://github.com/openclaw/openclaw/blob/60956ba6ac96372285cd2b82f6203eaa205b341e/src/gateway/session-utils.ts#L1256)
