## Summary

Close as implemented on current main: the reported failing workaround was forcing OpenAI Responses to SSE, and current main now honors explicit SSE so it goes through the HTTP/SSE fallback path with focused regression coverage. The fix is on main after v2026.4.29 and does not appear to be in a shipped release tag yet.

## What This Changes

The issue reports OpenClaw 2026.4.29 embedded Slack-triggered runs timing out against direct OpenAI GPT-5.2/GPT-5.4 Responses streaming even though raw OpenAI Responses/SSE calls succeed.

## Best Possible Solution

Keep the current narrow transport fix on main and ship it in the next release so direct OpenAI API-key users can force SSE when WebSocket-first transport is unhealthy, without broadening model timeouts or changing Slack delivery behavior.

## Reproduction Assessment

Yes: the issue gives a concrete release/environment path, and current main has a focused source-level regression test for the failing forced-SSE behavior. I did not run a live Slack/OpenAI EC2 reproduction, but the fixed path matches the reporter’s raw-SSE workaround exactly.

## Solution Assessment

Yes: honoring explicit SSE is the narrow maintainable fix because raw Responses SSE already worked for the reporter, while a generic timeout increase would only hide a transport-selection bug. Current main preserves auto/WebSocket behavior while making the explicit SSE escape hatch deterministic.
