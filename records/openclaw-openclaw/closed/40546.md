## Summary

Current `main` already implements LaunchAgent recovery for installed-but-not-loaded gateway services, and the changelog shows this behavior shipped in 2026.4.5. The code no longer matches the issue's described "hint-only" path.

## Evidence

- **Gateway start now recovers installed-but-unloaded services instead of only hinting:** `startGatewayService()` reads full service state, distinguishes `missing-install` from `installed but not loaded`, and calls `service.restart(...)` when install artifacts still exist.
  - file: [src/daemon/service.ts:114](https://github.com/openclaw/openclaw/blob/d4e93e791bc5093ea14666bf4d8202b45a5132b4/src/daemon/service.ts#L114)
- **launchd restart path re-bootstrap/retries when `kickstart` hits a not-loaded LaunchAgent:** `restartLaunchAgent()` enables the service, retries with `bootstrap` when `launchctl kickstart -k` reports not-loaded, and only falls back to an error after that repair path fails.
  - file: [src/daemon/launchd.ts:625](https://github.com/openclaw/openclaw/blob/d4e93e791bc5093ea14666bf4d8202b45a5132b4/src/daemon/launchd.ts#L625)
- **CLI start/restart explicitly wire macOS LaunchAgent recovery:** `runDaemonStart()` and `runDaemonRestart()` both pass `recoverInstalledLaunchAgent(...)` for the not-loaded path, so `openclaw gateway start` and `restart` no longer stay on hint-only behavior for installed LaunchAgents.
  - file: [src/cli/daemon-cli/lifecycle.ts:152](https://github.com/openclaw/openclaw/blob/d4e93e791bc5093ea14666bf4d8202b45a5132b4/src/cli/daemon-cli/lifecycle.ts#L152)
- **Dedicated recovery helper repairs installed LaunchAgent plists:** `recoverInstalledLaunchAgent()` checks whether the plist exists and then calls `repairLaunchAgentBootstrap()`; on success it returns a loaded `started`/`restarted` result with the message `re-bootstrapped launchd service.`
  - file: [src/cli/daemon-cli/launchd-recovery.ts:14](https://github.com/openclaw/openclaw/blob/d4e93e791bc5093ea14666bf4d8202b45a5132b4/src/cli/daemon-cli/launchd-recovery.ts#L14)
- **Tests cover the repaired not-loaded start/restart flows:** `lifecycle-core.test.ts` asserts loaded `started`/`restarted` results for the repaired launchd path, and `launchd-recovery.test.ts` plus `launchd.test.ts` cover bootstrap repair success/failure handling.
  - file: [src/cli/daemon-cli/lifecycle-core.test.ts:208](https://github.com/openclaw/openclaw/blob/d4e93e791bc5093ea14666bf4d8202b45a5132b4/src/cli/daemon-cli/lifecycle-core.test.ts#L208)
- **Changelog records this as shipped behavior:** `CHANGELOG.md` under `2026.4.5` explicitly says macOS now re-bootstrap/kickstarts unloaded LaunchAgents and recovers installed-but-unloaded LaunchAgents during `openclaw gateway start` and `restart`.
  - file: [CHANGELOG.md:1247](https://github.com/openclaw/openclaw/blob/d4e93e791bc5093ea14666bf4d8202b45a5132b4/CHANGELOG.md#L1247)

Current `main` already implements LaunchAgent recovery for installed-but-not-loaded gateway services, and the changelog shows this behavior shipped in 2026.4.5. The code no longer matches the issue's described "hint-only" path.

What I checked:
