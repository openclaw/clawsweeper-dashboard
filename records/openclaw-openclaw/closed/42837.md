- `P2`: This is a concrete Feishu channel routing regression with limited blast radius, not a core runtime outage.
- `impact:session-state`: Named-account Feishu DM users can be routed into shared `agent:main` state instead of their intended per-user dynamic agent session.

## Summary

Keep open: current main and the latest release still create Feishu dynamic-agent bindings without `match.accountId`, while Feishu named-account DM routing resolves with an account id and core routing treats account-less bindings as default-account-only. The closed source PR is unmerged, and the merged docs PR documents the limitation rather than fixing it.

## What This Changes

The issue reports that Feishu `dynamicAgentCreation` creates agents and bindings but still routes DM messages to `agent:main` instead of independent per-user agents.

## Best Possible Solution

Add a focused Feishu plugin fix that passes the active account id into dynamic binding lookup and creation, writes `match.accountId` for named-account dynamic bindings, preserves default-account legacy behavior, and adds regression coverage.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: Feishu named-account DMs resolve with `accountId`, generated dynamic bindings omit `match.accountId`, and core routing ignores account-less bindings for named accounts. I did not run a live Feishu tenant reproduction in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the Feishu dynamic-agent path: pass account id into binding detection and writes, with regression coverage for both default and named-account behavior.
