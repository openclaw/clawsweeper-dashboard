## Label Justifications

- `P1`: The released bug can brick every message dispatch for an affected long session until the user starts over.
- `impact:message-loss`: The reported failure produces no reply for every subsequent inbound message on the wedged session.
- `impact:session-state`: The failure is driven by preflight compaction/session state disagreement and leaves the existing session unusable until reset.

## Summary

Current main already handles the reported successful no-op preflight compaction result as a skip instead of throwing, with focused regression coverage; the fix is main-only and not present in the latest v2026.5.22 release.

## What This Changes

Treat successful no-op preflight compaction results such as `ok:true`, `compacted:false`, `reason:"already under target"` as benign so long sessions can continue dispatching.

## Best Possible Solution

Keep the current main preflight behavior and let the next release carry it; no additional core issue is needed for the already-under-target no-op case.

## Reproduction Assessment

Yes for the shipped v2026.5.22 source: that tag throws whenever `result.compacted` is false, so the reported `ok:true, compacted:false` result follows the failing path. Current main no longer reproduces because the same result returns as a skipped preflight compaction.

## Solution Assessment

Yes. The current main implementation is a narrow fix that preserves real failure throws while treating successful no-op compaction as a non-fatal skip, with a regression test for `already under target`.
