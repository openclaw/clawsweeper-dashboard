## Summary

Close as superseded by the shipped session-scoped Control UI token design and the related maintainer discussion. Current main still reproduces the reported browser-close loss, but docs, tests, and the merged storage work intentionally keep gateway tokens out of localStorage, so the requested behavior is a new admin-token retention policy rather than a bug fix.

## What This Changes

The issue requests persisting the Control UI gateway token in localStorage so it survives browser close and reopen.

## Best Possible Solution

Keep the shipped sessionStorage-only default; any future persistent-token mode should be a separate opt-in product/security design with clear logout, clear-token, and docs semantics.

## Reproduction Assessment

Yes. Source, tests, and docs on current main show the token is stored in sessionStorage and omitted from localStorage, so closing the browser session loses it by design.

## Solution Assessment

No. The suggested helper swap to localStorage conflicts with the shipped admin-token storage boundary; the narrow maintainable solution is to keep sessionStorage unless maintainers approve an explicit persistent-token mode.
