- `impact:message-loss`: The issue is about silently dropping the tail of a prior bot message from the next turn's reply context.
- `impact:session-state`: The reported failure is worst after idle reset, where the truncated reply context is the only state linking the new session to the user's short response.

## Summary

Keep open: current main and the latest release still apply the 500-character metadata-field sanitizer to reply-target chat-window bodies, so the reported context loss remains a real core reply-context bug.

## What This Changes

The issue requests preserving longer replied-message bodies in inbound reply context instead of truncating them with the 500-character short-field cap.

## Best Possible Solution

Split short metadata-field sanitization from long-form reply/body sanitization, use a bounded body cap that preserves multi-paragraph reply targets, and add regression coverage for a >500-character reply target whose actionable tail survives.

## Reproduction Assessment

Yes at source level: current main routes chat-window body through the 500-character transcript-field sanitizer. I did not run the full live channel/session-reset repro in this read-only review.

## Solution Assessment

Yes: separating long-form reply body limits from short metadata limits is the narrow maintainable fix. A new config knob should be avoided unless maintainers explicitly want tunable prompt-budget policy.
