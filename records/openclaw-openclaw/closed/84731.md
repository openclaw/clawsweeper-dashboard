- `P2`: The issue is a concrete performance bug in an existing user-facing model-listing command with limited blast radius.
- `impact:auth-provider`: The affected behavior is provider/model catalog discovery and model-choice visibility for configured providers.

## Summary

Keep open: current main still routes the auto-reply `/models` command through the non-read-only catalog load, and the open fix PR at https://github.com/openclaw/openclaw/pull/84735 has closing syntax but has not merged and has review comments pointing to remaining timeout/fallback work.

## What This Changes

The issue reports that Telegram `/models` is slow for large LiteLLM catalogs because the command appears to use the full model-catalog refresh path instead of the fast read-only listing path.

## Best Possible Solution

Mirror or share Gateway `models.list` behavior for auto-reply `/models`: keep explicit `all` and provider wildcards on the full catalog path, use a read-only catalog with a short timeout/fallback for ordinary browse, and prove the LiteLLM-sized command latency with focused tests and redacted runtime evidence.

## Reproduction Assessment

Yes for the source-level path: current main's `/models` command calls `loadModelCatalog({ config })` without `readOnly`, and that non-read-only path performs the full catalog refresh work. I did not run the reporter's live LiteLLM setup, so the wall-clock timing remains external evidence.

## Solution Assessment

Yes on direction, but not complete on current main: reusing the Gateway read-only/timeout contract is the narrow maintainable fix. The linked PR should be reviewed or updated, and this issue should close only after that implementation lands or is superseded by an equivalent merged fix.
