## Summary

Current main already solves the Telegram polling-offset problem this PR was trying to address, but with the newer accepted-offset policy shipped in v2026.4.24 rather than the PR's media-group hold/release tracker. The PR is also against the old src/telegram layout, while Telegram now lives under extensions/telegram.

## Best Possible Solution

Close this PR and keep the shipped v2026.4.24 accepted-offset implementation. If maintainers want stronger at-least-once delivery guarantees for buffered media groups, that should be opened as a fresh issue against current extensions/telegram code with a current-main reproduction and an explicit product decision about replay versus duplicate-processing tradeoffs.

## Evidence

That means buffered media-group flush completion no longer controls restart offset persistence.
