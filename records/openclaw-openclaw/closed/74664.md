## Summary

Keep open: current main still has the OpenAI Completions transport builder emitting tool payload fields without checking the existing `compat.supportsTools` contract, and #74738 is already the paired open fix PR.

## What This Changes

OpenAI-compatible chat-completions payloads should omit `tools`, `tool_choice`, and the tool-history `tools: []` fallback when `model.compat.supportsTools` is `false`.

## Best Possible Solution

Land #74738 or an equivalent narrow patch that gates OpenAI Completions tool emission and the history fallback with `supportsModelTools(model)`, with regression coverage and a changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a deterministic current-main path: call `buildOpenAICompletionsParams()` with `compat.supportsTools: false` and non-empty `context.tools`, or with tool-call history and no current tools, and the builder reaches unguarded `params.tools` assignments.

## Solution Assessment

Yes. Applying the existing `supportsModelTools()` helper inside `buildOpenAICompletionsParams()` is the narrow maintainable fix; a dependency bump alone does not cover OpenClaw's local transport builder.
