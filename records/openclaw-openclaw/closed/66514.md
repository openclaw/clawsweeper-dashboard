## Summary

Current `main` already hardens the browser health/recovery path against the stale-CDP state described here: status now keys `running` off real CDP readiness, readiness requires a successful `Browser.getVersion` round-trip, `browser start` waits for CDP readiness after launch/restart, and snapshot/screenshot timeout errors now report detailed browser status.

## Evidence

- **Status no longer treats HTTP-only reachability as healthy:** The browser status route now probes both HTTP and CDP, then reports `running: cdpReady` instead of treating partial reachability as healthy.
