## Summary

Keep open. This is a real config persistence bug and the preservation-by-default direction is plausible, but the current PR still has a P1 deletion-contract regression and lacks required after-fix real behavior proof.

## What This Changes

The PR changes config merge-patch writes to preserve absent target keys by default, applies explicit `unsetPaths` during write preparation, forwards pruned gateway auth paths, and adds config persistence tests.

## Best Possible Solution

Keep the preservation-by-default direction, but make intentional deletion reliable across config mutation APIs, especially channel/account removal, with persisted-file regressions and redacted Docker or CLI restart proof.

## Reproduction Assessment

Yes, at source level: current main emits `null` for omitted keys and applies that patch during runtime writeback, matching the reported config clobber. The PR regression is also source-reproducible because channel removal still writes an omitted-key config without `unsetPaths`.

## Solution Assessment

No as currently patched. Explicit `unsetPaths` is a maintainable direction, but the PR migrates only one gateway auth deletion path and leaves channel/account deletion-by-omission unreliable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve delete intent for channel removal:** `src/config/io.write-prepare.ts:31`
  - body: At PR head, `createMergePatch` treats omitted target keys as preserve-only, but `channels remove --delete` still calls `deleteAccount` and then `replaceConfigFile` without `writeOptions.unsetPaths`. Delete adapters such as the shared channel helper and Feishu omit channel/account fields, so this can report a deletion while restoring the old channel config or credentials on disk.
