## Summary

Close PR #41273 as cannot reproduce. Current `main` has a newer channel health monitor implementation, and the shipped reload/shutdown call paths do not pass an `abortSignal`, so the specific listener-retention leak described in the PR is not exercised on current `main`.

## Evidence

- **Only current non-test caller omits abortSignal:** `rg -n "startChannelHealthMonitor\(" -g '*.ts'` shows only three callsites: the implementation, tests, and `startGatewayChannelHealthMonitor`. The current runtime caller constructs the monitor with `channelManager`, `checkIntervalMs`, `staleEventThresholdMs`, and `maxRestartsPerHour`, but no `abortSignal`.
  - file: [src/gateway/server-runtime-services.ts:38](https://github.com/openclaw/openclaw/blob/d068cb960de47d5587c5942f6beb86e0f124b449/src/gateway/server-runtime-services.ts#L38)

  - sha: [d068cb960de4](https://github.com/openclaw/openclaw/commit/d068cb960de47d5587c5942f6beb86e0f124b449)
- **Reported direct-stop paths exist, but without a registered abort listener:** The hot-reload path calls `state.channelHealthMonitor?.stop()` and the gateway close prelude calls `runtimeState?.channelHealthMonitor?.stop()`. On current `main`, both operate on the monitor instance created above, which does not have an `abortSignal` attached.
  - file: [src/gateway/server-reload-handlers.ts:150](https://github.com/openclaw/openclaw/blob/d068cb960de47d5587c5942f6beb86e0f124b449/src/gateway/server-reload-handlers.ts#L150)

- **Abort listener is conditional on abortSignal:** `stop()` currently clears the interval only, and the abort listener is registered only inside the `else` branch when `abortSignal` is provided. Without an `abortSignal` caller, there is no listener to retain the closure on the reload/shutdown paths cited in the PR.
