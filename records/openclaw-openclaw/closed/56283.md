- **[P1] Preserve WhatsApp formatting before auto stripping:** `src/infra/outbound/sanitize-text.ts:86`
  - body: With the new auto path, `whatsapp` is treated as not Markdown-capable, so the strip pass runs after channel sanitization. WhatsApp currently converts `**bold**` to `*bold*` and preserves `_italic_`; stripping after that removes intentional formatting and changes delivered messages.
  - confidence: 0.92
- **[P2] Resolve strip config with account scope:** `src/infra/outbound/deliver.ts:371`
  - body: `resolveChannelMarkdownConfig()` only reads `cfg.channels[channel].markdown`, but channel Markdown config already has account-scoped resolution patterns. Multi-account operators would not be able to override stripping per account even though the surrounding config shape supports account entries.
  - confidence: 0.86
- **[P2] Consume the whole fenced code fence:** `src/shared/text/strip-markdown.ts:9`
  - body: The new fenced-code regex only accepts alphabetic info strings and leaves surrounding fence newlines in common mid-message cases. Fences such as ```c++ and adjacent blocks can leak label remnants or add unexpected blank lines in outbound plain text.
  - confidence: 0.88
- **[P1] Avoid broad stripping with the old underscore matcher:** `src/infra/outbound/deliver.ts:349`
  - body: Applying this branch's `stripMarkdown()` to outbound text by default can mutate legitimate identifiers such as `foo_bar_baz` or URL path segments before users copy them. Current main has underscore-safety tests, so the PR must rebase and preserve that behavior before broad delivery use.
  - confidence: 0.82
