## Label Justifications

- `P1`: This regression can make every affected OpenRouter agent turn fail immediately after upgrade.
- `impact:session-state`: The overflow loop can repeatedly compact or reset sessions without resolving the oversized default output request.
- `impact:auth-provider`: The failure is rooted in provider/model metadata resolution for OpenRouter and the selected model endpoint.

## Summary

Keep open: current main has a generic proxy-like output-cap clamp, but OpenRouter capability parsing still uses top-level `context_length` instead of the endpoint-specific `top_provider.context_length`, so the reported model can still default to a 262144-token output cap against a 262144-token endpoint.

## What This Changes

Fix the OpenRouter regression where discovered model metadata can make the default output-token budget equal the endpoint context limit, causing every agent turn to overflow after upgrade.

## Best Possible Solution

Use OpenRouter's endpoint-specific context limit when available for effective context budgeting, keep explicit `maxTokens`/`contextWindow` overrides intact, and leave the generic proxy-like transport clamp as a backstop.

## Reproduction Assessment

Yes. Source plus live catalog inspection show that the reported OpenRouter model exposes a 1M top-level context, a 262144-token endpoint context, and a 262144-token endpoint max completion cap; current main parses that into a 1M context budget and therefore does not clamp the 262144 output request.

## Solution Assessment

No. The current generic proxy-like clamp is a useful partial fix, but the best fix for this issue is to normalize or carry OpenRouter `top_provider.context_length` into the effective runtime context budget before transport clamping.
