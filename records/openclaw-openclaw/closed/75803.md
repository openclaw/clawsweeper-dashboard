## Summary

Close as implemented on current main. The reported 2026.4.29 embedded-runtime behavior is addressed by routing embedded runner runtime-plugin loads through verify-only bundled dependency handling, plus current materialization logic that treats already-installed runtime-dependency supersets as converged; this is on main but not proven shipped in the latest release tag.

## What This Changes

The issue asks OpenClaw to avoid repeated per-plugin bundled runtime dependency resolution during embedded gateway fallback or `openclaw agent --local` runs.

## Best Possible Solution

Keep the current-main design: dependency repair happens in explicit startup/prestage or repair paths, while embedded runner runtime-plugin loading remains verify-only and accepts materialized package-level supersets. A new run-scoped spec cache is unnecessary unless future profiling shows meaningful remaining per-plugin verification cost after this change.

## Reproduction Assessment

Yes. The issue gives a concrete reproduction path with `openclaw agent --local` and log-line counting, and the current-main source comparison shows the reported v2026.4.29 hot path no longer repairs bundled runtime deps during embedded runtime plugin loading.

## Solution Assessment

Yes. The current-main fix is narrower and safer than adding an install callback cache because it removes package-manager repair from the embedded hot path while preserving explicit startup/prestage repair and dependency verification.
