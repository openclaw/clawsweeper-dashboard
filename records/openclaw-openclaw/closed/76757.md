## Summary

Keep open: this draft, maintainer-labeled PR still has a P2 correctness issue where model fallback can reuse an auth order prepared for the primary provider when running a different fallback provider.

## What This Changes

Threads prepared model/auth/provider/plugin/tool/media/TTS/Gateway runtime state through reply execution and adds scoped runtime helpers, tests, AGENTS guidance, and a changelog entry.

## Best Possible Solution

Keep the prepared-runtime reuse, but make prepared auth state provider-scoped and add cross-provider fallback auth regression coverage before normal maintainer review.

## Reproduction Assessment

Yes. Source inspection of the PR head shows authProfileOrder is prepared for the initial provider and then reused during model fallback attempts for other providers without recomputing provider-specific order.

## Solution Assessment

No. The performance direction is reasonable, but the current implementation should filter or recompute prepared auth order for the fallback provider before merge.
