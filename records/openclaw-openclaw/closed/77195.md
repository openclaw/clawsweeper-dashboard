## Summary

Current `main` and the `v2026.5.3` tag already contain the CLI approval-scope fix: `openclaw devices approve <requestId>` now inspects the pending request and connects with `operator.admin` when the approval would mint or inherit admin operator access.

## What This Changes

The issue reports that fresh CLI device pairing can leave the CLI with only `operator.pairing`, causing `openclaw devices approve` to fail when admin-scoped approval is needed.

## Best Possible Solution

Keep the shipped CLI-side approval-scope selection and Gateway-side approval guard: the CLI asks for admin only when the pending device request needs it, while lower-scope pairing approvals remain least-privilege.

## Reproduction Assessment

No, not against current `main`: the source path now computes `operator.admin` for admin-scoped pending approvals before calling `device.pair.approve`, and the regression test covers that case. The original report remains credible for `2026.5.2`, but the current code path no longer matches the failing behavior.

## Solution Assessment

Yes. The current fix is narrow and maintainable because it adjusts the CLI request scopes for the specific approval being performed while leaving the Gateway enforcement in `approveDevicePairing` intact.
