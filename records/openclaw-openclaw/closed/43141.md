## Summary

Current main resolves the reported cron timeout-before-model-invocation failure through the shipped cron execution-lane changes rather than the originally proposed generic queue-abort API. Manual cron.run calls now enqueue asynchronously and return an enqueued acknowledgement, isolated agent-turn timeouts are deferred until the runner enters its effective execution lane, and cron inner agent work is mapped away from the outer cron lane to cron-nested. The generic command queue still lacks per-entry AbortSignal removal, but the concrete stale caller-timeout reproduction for queued isolated cron runs no longer matches current main.

## Best Possible Solution

Close this issue as implemented by the current cron execution-lane behavior. Keep the shipped timeout deferral, async manual cron.run queueing, and cron-nested lane mapping; open a narrower follow-up only if maintainers still want generic AbortSignal support in enqueueCommandInLane for other callers.
