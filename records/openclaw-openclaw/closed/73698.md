## Summary

Keep open. Current main explains the most likely cause: the reporter's original Telegram config intentionally admitted public DMs and default DM routing can share the main session, but the report is recent, security-sensitive, and still lacks sanitized raw Telegram updates/session records needed to distinguish expected forwarded-context presentation from an actual routing bug.

## What This Changes

The issue reports that a Telegram bot configured with `dmPolicy: "open"` and `allowFrom: ["*"]` accepted messages from an unknown Telegram user, leading to execution of attacker-supplied host commands.

## Best Possible Solution

Keep this in maintainer/security triage until the reporter provides sanitized Telegram updates, relevant gateway logs/session keys, and effective tool/approval configuration. The desired end state is either clear expected-behavior guidance for public DMs plus shared-session context, or a narrow Telegram routing/session isolation fix if the evidence shows sender isolation failed.

No. Current main supports reproducing unknown Telegram users reaching the bot under `dmPolicy: "open"` with wildcard `allowFrom`, but not the specific claim that another user's forwarded context appeared in the reporter's session without raw Telegram update and session evidence.

Is this the best way to solve the issue? No, not as close-only cleanup. The maintainable path is evidence-first security triage, then either confirm expected public-DM/shared-session behavior with prevention guidance or patch Telegram routing if sender/session isolation is shown to fail.
