## Summary

Keep open: the Telegram abort-bleed bug is still present on current `main`, but this PR is dirty and edits the old `bot.ts` implementation shape instead of the current `bot-core.ts` path.

## What This Changes

The PR scopes Telegram `fetchAbortSignal` handling to `getUpdates` via a grammY API middleware and updates fetch-abort tests for `getUpdates` versus `sendMessage`.

## Best Possible Solution

Land a rebased or replacement current-main Telegram fix in `bot-core.ts` or an equivalent current seam, keeping shared restart aborts on `getUpdates` only while outbound methods rely on their own request signals and timeouts, with regression coverage and a changelog entry.

## Reproduction Assessment

Yes. Source inspection is high-confidence: polling restart aborts the shared controller, `bot-core.ts` forwards that signal to every Telegram fetch, and the current test still asserts broad abort behavior; I did not run live Telegram in this read-only review.

## Solution Assessment

No. The concept is the narrow maintainable fix, but this branch implements it in the old `bot.ts` shape and needs to move to current `bot-core.ts` or an equivalent current-main seam before merge.
