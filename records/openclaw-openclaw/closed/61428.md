## Summary

Close as superseded: the hook-session continuity problem is now tracked by the canonical open issue and newer implementation PR, while this branch still has policy, replay-scope, generated-config, and proof blockers.

## What This Changes

The PR adds mapping-level `sessionTarget: "main" | "isolated"` for hook-agent mappings and threads it through config, mapping resolution, dispatch payloads, and cron job creation.

## Best Possible Solution

Close this superseded branch and concentrate review on https://github.com/openclaw/openclaw/pull/75918 or an equivalent explicit persistent-session mode that preserves default isolation and existing session-key gates.

## Reproduction Assessment

Yes for the PR blockers: source inspection shows a `sessionTarget: "main"` mapping would be validated and replay-scoped using one key, then dispatched with a different agent main-session key. The original feature request is a new opt-in behavior, not a broken existing contract.

## Solution Assessment

No. The mapping-level `sessionTarget` idea is plausible, but the safer maintained direction is the canonical `sessionMode` track because it handles direct/mapped/transform paths, default isolation, replay scope, docs, and policy gates together.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Validate the effective main session key before dispatch:** `src/gateway/server-http.ts:793`
  - body: When a mapped action sets `sessionTarget: "main"`, this patch lets dispatch replace the already-validated hook session key with the agent main-session key. That makes `hooks.allowedSessionKeyPrefixes` apply to a key that may never be used, which can reject valid main-target mappings or allow a disallowed `agent:*` execution after a `hook:*` key passed validation.
  - confidence: 0.9
- **[P2] Include sessionTarget in hook replay scope:** `src/gateway/server-http.ts:793`
  - body: Forwarding `sessionTarget` changes where the hook run executes, but the replay fingerprint still omits that field. Two otherwise identical mapped deliveries with the same idempotency key can return the cached run id from an isolated run when the second request should run in the main session, or vice versa.
  - confidence: 0.84
- **[P2] Regenerate config metadata for the new mapping key:** `src/config/zod-schema.hooks.ts:47`
  - body: Adding `sessionTarget` to the strict hook mapping schema makes this a public config key, but generated schema/help/label surfaces are not updated. Schema-driven config tooling will not expose or preserve `hooks.mappings[].sessionTarget`, creating config contract drift.
  - confidence: 0.87

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete hook session-prefix policy concern because dispatch can execute a different key than the one validated by the request handler.

Concerns:

- **[medium] Session-prefix policy checks the wrong key:** `src/gateway/server-http.ts:793`
  - body: For mapped `sessionTarget: "main"` runs, the request handler validates the hook/default session key, but dispatch rewrites execution to the agent main-session key. On exposed hook endpoints this can bypass an operator prefix allowlist or wrongly reject allowed main-session configurations.
  - confidence: 0.89
