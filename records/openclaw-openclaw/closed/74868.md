## Summary

Keep this issue open. Checked-out main still matches the reported failure mode, and an open closing PR already tracks the fix, so the issue should remain open until that PR is reviewed and merged or closed.

## What This Changes

The issue requests that group/channel visible replies fall back to automatic delivery when `messages.groupChat.visibleReplies` is `message_tool` but the effective `message` tool is unavailable.

## Best Possible Solution

Make source reply visibility policy message-tool-aware at the dispatch/run boundary, keep automatic delivery when tool-only visible replies cannot work, and align prompt guidance plus focused regression tests with that behavior.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: a group/channel turn with default `message_tool` visible replies and `message` removed by `disableMessageTool` or a restrictive allowlist still resolves to `message_tool_only` and suppresses automatic delivery on checked-out main.

## Solution Assessment

Yes. The requested fallback is the narrow maintainable direction, as long as it is driven by the effective tool availability for the turn rather than config text alone.
