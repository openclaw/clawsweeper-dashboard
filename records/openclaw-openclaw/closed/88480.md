## Summary

Keep open: current main still has a source-visible path where a configured `google` model row without an explicit `api` can resolve to `openai-responses`, matching the reporter's 401 symptom. The provided related items are adjacent Google/provider reports but do not own this native Google transport-selection bug.

## What This Changes

The issue reports that Google Gemini chat models can be resolved as `google/*` but dispatched through OpenAI Responses instead of the native Google transport.

## Best Possible Solution

Make `google` configured provider/model rows without an explicit API resolve to `google-generative-ai`, preserve explicit API overrides, and add focused regression coverage for runtime resolution and models-list parity.

## Reproduction Assessment

Yes, source inspection gives a high-confidence repro path: configure `models.providers.google.models[]` with a Gemini model but no explicit `api`, then resolve that model with dynamic discovery absent or bypassed. The fallback code can select `openai-responses` instead of `google-generative-ai`.

## Solution Assessment

Yes, the best fix is a narrow provider-owned defaulting/normalization repair for Google model rows, not key-format detection. Explicit user-selected APIs and generic custom-provider defaults should remain unchanged.
