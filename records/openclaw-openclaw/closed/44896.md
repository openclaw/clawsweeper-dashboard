## Summary

Keep open. Current main still downgrades enabled Moonshot/Kimi thinking to `thinking.type: "disabled"` for pinned `tool_choice`, while the transport can emit `reasoning_effort` independently and the docs say incompatible Moonshot tool choices should normalize to `auto`.

## What This Changes

Fix the Moonshot/Kimi request payload so reasoning-enabled runs with pinned tool choices do not produce an invalid `reasoning_effort` plus disabled-thinking combination.

## Best Possible Solution

Normalize pinned or otherwise incompatible Moonshot `tool_choice` values to `auto` whenever effective thinking is enabled, while preserving explicit disabled thinking and Kimi K2.6 `thinking.keep` gating.

## Reproduction Assessment

Yes. A high-confidence source-level path exists on current main: non-off Moonshot thinking plus pinned `tool_choice` becomes `thinking.type: "disabled"`, while the OpenAI-compatible transport can still send `reasoning_effort`.

## Solution Assessment

Yes for the docs-aligned `tool_choice: "auto"` fallback, and no for the literal suggested `"disabled"` to `"enabled"` replacement because that would leave a Moonshot-incompatible pinned tool choice in place.
