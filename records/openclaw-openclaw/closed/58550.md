## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Avoid JSON fallback writes after SQLite save failures:** `src/config/sessions/store-facade.ts:149-154`
  - body: In sqlite mode this catch writes the updated store to `sessions.json` when SQLite throws, but later reads still prefer SQLite as soon as it is readable again. A transient SQLite error can therefore acknowledge a save that disappears on the next load because the SQLite database remains stale.
  - confidence: 0.92
- **[P1] Propagate JSON-to-SQLite migration failures:** `src/config/sessions/store-sqlite.ts:603-605`
  - body: `migrateJsonToSqlite` converts parse, open, and write failures into `0`, which callers treat like an ordinary empty store. On first access this can leave sqlite mode reading an empty or stale hot store instead of failing the migration or falling back to JSON.
  - confidence: 0.9
- **[P2] Normalize legacy rows before SQLite import:** `src/config/sessions/store-sqlite.ts:574-589`
  - body: The JSON loader runs session-store migrations and runtime normalization before exposing rows, but this migration inserts raw parsed JSON. Older rows with legacy fields such as `provider` or `lastProvider` can be imported with missing indexed channel columns until each session is rewritten.
  - confidence: 0.84
- **[P2] Memoize migration attempts only after success:** `src/config/sessions/store-facade.ts:83-93`
  - body: `migrationAttempted` is marked before migration work runs, so a transient first-run failure suppresses retries for the rest of the process. Mark the key only after a successful or genuinely unnecessary migration so later loads can recover without restart.
  - confidence: 0.86
