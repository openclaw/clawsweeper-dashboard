## Summary

Keep this PR open: it targets a real unbounded success-response read class that still exists on current main, but the branch is not merge-ready because it duplicates the shared provider JSON reader, applies a risky 1 MiB cap to image/OAuth/provider paths, leaves existing json-only test mocks unhandled, and has no branch-specific real behavior proof.

## What This Changes

The PR replaces selected Vydra, xAI, Runway, Pixverse, DeepInfra, Together, ElevenLabs, Microsoft Foundry, and Microsoft success-response JSON reads with `readResponseWithLimit` plus `JSON.parse` under 1 MiB caps.

## Best Possible Solution

Rework the batch through the shared provider JSON reader or a small consistent wrapper, use payload-appropriate caps, update focused tests for real Response bodies plus overflow cancellation, and add redacted runtime proof before merge.

## Reproduction Assessment

Yes, source-reproducible: current main still uses unbounded `response.json()` on the targeted external success paths, and merged sibling work proves the same oversized streamed JSON class can be bounded. I did not run a live oversized-response harness for this branch because this review is read-only.

## Solution Assessment

No. Bounded reads are the right class of fix, but this branch should reuse the shared provider JSON reader where possible, repair the helper/test contracts, size caps to payload contracts, and prove the changed runtime behavior.
