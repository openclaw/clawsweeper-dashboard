## Summary

Issue #43922 is a broad proposal for a richer agent architecture, but current `main` treats that class of functionality as plugin/skill territory rather than OpenClaw core. The project vision explicitly keeps core lean, sends new skills to ClawHub first, and says default agent-hierarchy/heavy orchestration layers are out of scope for core.

## Close Comment

Thanks for the detailed write-up. Codex review on current `main` concludes this is better tracked as a ClawHub/plugin effort rather than an OpenClaw core issue, so closing as `clawhub`.

Concrete scope evidence: `VISION.md:54-70` says core should stay lean, optional capability should usually ship as plugins, and new skills should go to ClawHub first. `VISION.md:101-107` also explicitly says default agent-hierarchy frameworks and heavy orchestration layers that duplicate existing agent/tool infrastructure are not in scope for core right now. At the same time, current `main` already exposes the relevant extension seams through pluggable memory/dreaming (`docs/cli/memory.md:12-14`, `docs/cli/memory.md:128-145`), isolated agents/workspaces (`docs/cli/agents.md:8-40`), and ClawHub-backed skill/plugin installs (`docs/cli/skills.md:10-45`, `docs/cli/plugins.md:54-69`).

If you want to pursue this idea, the best next step is a focused ClawHub plugin/skill or external memory/runtime package proposal with a concrete user-facing slice instead of a new default core architecture.
