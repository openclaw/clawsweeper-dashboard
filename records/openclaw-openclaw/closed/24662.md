## Summary

Issue #24662 is already fixed on current `main` and documented as shipped in release `v2026.4.20`: webchat pasted/uploaded images are sent as `attachments`, forwarded through `chat.send` as `replyOptions.images`, and no longer get rejected as empty text-only turns.

## Close Comment

Closing this as fixed after a Codex review. Current `main` already handles webchat pasted/uploaded images end-to-end: the Control UI sends them as `attachments` (`ui/src/ui/controllers/chat.ts:232`), `chat.send` parses and forwards them as `replyOptions.images` (`src/gateway/server-methods/chat.ts:2194`, `src/gateway/server-methods/chat.ts:2474`), and the empty-turn guard now treats those inline images as real media instead of rejecting the turn as textless (`src/auto-reply/reply/get-reply-run.ts:412`). There is also a regression test covering inline image forwarding (`src/auto-reply/reply/dispatch-acp.test.ts:553`). The shipped fix is called out in `CHANGELOG.md:347` under `2026.4.20` (`Webchat/images: treat inline image attachments as media for empty-turn gating...`). If anyone still hits this on `v2026.4.20` or newer, please open a fresh issue with the exact OpenClaw version, browser, whether the image was paste vs file picker, and current gateway logs.
