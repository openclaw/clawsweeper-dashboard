## Summary

Keep open. Current main has the partial runtime-backed health work, but the Telegram credential-source mismatch is still plausible: health builds from a projected runtime snapshot, while channels.status uses the richer per-account snapshot builder. There is also an open implementation candidate with closing syntax for this issue, so the issue should remain open until that PR is reviewed/imported or superseded.

## Best Possible Solution

Keep this issue open and review/import the open implementation candidate, or land a focused replacement that makes health use the same safe resolved-account snapshot contract as channels.status while preserving the public health redaction boundary. Add regression coverage comparing Telegram health and channels status for running, lastStartAt, mode, and tokenSource, including cache/startup behavior for default non-probe health.
