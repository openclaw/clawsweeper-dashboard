## Summary

Close as duplicate: this issue reports the same completed native subagent `sessions.json` retention gap already tracked by the open canonical #51767 thread. Current `main` still confirms the same source-level behavior, so the remaining code/product work should stay centralized on that tracker rather than split across another issue.

## What This Changes

Request automatic cleanup or retention for completed native subagent session-store entries so `sessions.json` does not keep stale `agent:<id>:subagent:<uuid>` rows indefinitely.

## Best Possible Solution

Keep #51767 as the canonical tracker and implement one lifecycle-aware, restart-safe retention path there for completed subagent and ACP session rows, preserving active/thread-bound sessions and using the existing session deletion/transcript archival helpers.

## Reproduction Assessment

Yes. The source-level path is high confidence: omitted cleanup becomes `keep`, keep-mode registration leaves `archiveAtMs` unset, and the sweeper only calls `sessions.delete` for archive-timed delete-mode runs.

## Solution Assessment

Yes for the triage outcome. Closing this duplicate is the narrowest cleanup because #51767 already tracks the remaining design and implementation work; the eventual fix should be a lifecycle-aware retention policy rather than ad hoc deletion of all subagent keys.
