## Summary

Current main does not show a WhatsApp groupPolicy bypass here. The reported config still authorizes group senders via the documented `groupAllowFrom <- allowFrom` fallback, and current debounce code with `debounceMs: 0` does not batch messages into a later flush.

## Evidence

- **WhatsApp sender allowlist fallback:** `resolveWhatsAppInboundPolicy` falls back from `groupAllowFrom` to `allowFrom`, so the issue config's `allowFrom` entries are treated as the group sender allowlist when `groupAllowFrom` is unset.
  - file: [extensions/whatsapp/src/inbound-policy.ts:88](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/extensions/whatsapp/src/inbound-policy.ts#L88)
  - command: `nl -ba extensions/whatsapp/src/inbound-policy.ts | sed -n '70,138p'`
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Group allowlist semantics on current main:** `resolveChannelGroupPolicy` explicitly allows a group through under `groupPolicy="allowlist"` when sender-level filtering is configured upstream (`hasGroupAllowFrom`), even if no explicit `groups` map is present.
  - file: [src/config/group-policy.ts:352](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/config/group-policy.ts#L352)
  - command: `nl -ba src/config/group-policy.ts | sed -n '332,365p'`
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Docs match current behavior:** Current WhatsApp docs say omitted `channels.whatsapp.groups` means all groups are eligible, and unset `groupAllowFrom` falls back to `allowFrom`.
  - file: [docs/channels/whatsapp.md:181](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/docs/channels/whatsapp.md#L181)
  - command: `nl -ba docs/channels/whatsapp.md | sed -n '180,198p'`
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Fail-closed case is separately covered:** Current test coverage blocks group messages only when `groupPolicy="allowlist"` has no group sender allowlist at all; that is not the configuration reported here because `allowFrom` was populated.
  - file: [extensions/whatsapp/src/monitor-inbox.blocks-messages-from-unauthorized-senders-not-allowfrom.test-support.ts:330](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/extensions/whatsapp/src/monitor-inbox.blocks-messages-from-unauthorized-senders-not-allowfrom.test-support.ts#L330)
  - command: `nl -ba extensions/whatsapp/src/monitor-inbox.blocks-messages-from-unauthorized-senders-not-allowfrom.test-support.ts | sed -n '328,360p'`
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Current debounce path does not batch at 0ms:** `createInboundDebouncer` only batches when `debounceMs > 0`; with `debounceMs: 0`, items flush immediately, so the report's delayed 622-char batched flush does not match the current main code path.
  - file: [src/auto-reply/inbound-debounce.ts:166](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/auto-reply/inbound-debounce.ts#L166)

## Risks / Open Questions

- There is still no separate outbound group allowlist guard in `extensions/whatsapp/src/auto-reply/deliver-reply.ts`, so any future inbound-routing bug would rely on inbound policy correctness alone.
- The report may have come from older behavior on `2026.3.13`, but the locally available history is shallow, so I could not pinpoint a specific fixing commit or release for that older path.
