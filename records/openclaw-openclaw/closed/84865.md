## Summary

Keep open: current main still treats user-sourced model overrides as strict and disables configured fallbacks, while an open linked implementation PR is already the right place to review whether that policy should change.

## What This Changes

The issue asks OpenClaw to recover from provider outages after a user manually switches models by allowing a per-call fallback chain without persisting that fallback over the user's selected model.

## Best Possible Solution

Review and either land or reject the linked PR with an explicit policy decision: preserve strict user model pins with visible failure, or intentionally allow non-persistent per-call fallback with updated tests, docs, and runtime proof.

## Reproduction Assessment

Partly: current source and tests show user-sourced model overrides resolve to an empty fallback list despite configured fallbacks. I did not run the full live provider-outage deadlock path in this read-only review.

## Solution Assessment

Unclear: the linked PR's fallback approach is plausible for availability and preserves the stored user model, but it conflicts with current strict `/model` documentation and needs maintainer policy approval or a narrower visible-failure fix.
