## Summary

Current main and shipped releases already use the canonical OpenRouter `/api/v1` base URL and self-heal stale `/v1` OpenRouter and Arcee provider metadata, so the reported generated `models.json` failure is fixed.

## What This Changes

The issue asks for generated OpenRouter and Arcee provider config to use `https://openrouter.ai/api/v1` instead of the stale `https://openrouter.ai/v1` route.

## Best Possible Solution

Keep the shipped provider-owned canonicalization and ask for a new report with image digest/version proof only if a fixed release still regenerates `/v1`.

## Reproduction Assessment

No, current main does not reproduce the wrong generated route: the provider catalogs emit `/api/v1` and normalization rewrites stale `/v1` metadata. The upstream probe still confirms why the old `/v1` route failed, but that is no longer the current-main generated behavior.

## Solution Assessment

Yes. The existing fix is the narrow provider-owned solution: correct the catalog defaults and normalize stale provider config, resolved-model metadata, and transport metadata without adding startup migrations or core policy.
