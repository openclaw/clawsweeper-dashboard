## Summary

Current main and the shipped v2026.4.29 tag already cover the reported MiniMax/pi-ai crash path: OpenClaw resolves MiniMax through its own Anthropic-compatible boundary transport, guards missing usage fields there, and normalizes missing model input metadata to a text-only array before transport code can read it.

## What This Changes

The issue requests defensive handling for MiniMax/manifest-auto runs when provider responses omit usage fields or model metadata omits input capabilities.

## Best Possible Solution

Keep the shipped OpenClaw boundary-aware transport and model-normalization behavior, and have affected 2026.4.26 users upgrade to v2026.4.29 or newer. Any remaining crash should be filed as a new report with the exact model ref, provider id, and stack trace from the upgraded build.

## Reproduction Assessment

Yes. The reported failure maps directly to upstream pi-ai's unsafe `message.usage` and `model.input` reads, and current OpenClaw/v2026.4.29 can be checked at the same boundary to confirm those reads are guarded or normalized away.

## Solution Assessment

Yes. The current approach is the narrow maintainable fix: normalize model metadata once during resolution and route MiniMax Anthropic-compatible streams through OpenClaw's guarded transport instead of adding MiniMax-specific checks throughout core call sites.
