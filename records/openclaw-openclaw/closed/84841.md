## Label Justifications

- `P1`: The report describes a real wake-now path that leaves queued voice/system input unprocessed until another user message or scheduled heartbeat arrives.
- `impact:message-loss`: The affected system event is queued but effectively suppressed from delivery until a later unrelated trigger wakes the agent.

## Summary

Current main already implements immediate heartbeat wake semantics: `intent: "immediate"` bypasses the not-due/min-spacing gate, targeted wakes keep their session and heartbeat override, and retryable busy skips are retried by the wake layer.

## What This Changes

The issue asks for plugin runtime `requestHeartbeat({ intent: "immediate" })` and `/hooks/wake` mode `now` to wake an idle agent promptly instead of waiting for the next scheduled heartbeat.

## Best Possible Solution

Keep the current main immediate-intent scheduler behavior and ship it in the next release so release users get prompt plugin and hook wake delivery.

## Reproduction Assessment

No high-confidence failing reproduction on current main; I did not replay the Raspberry Pi/Ollama setup from v2026.5.18. Current-main source and regression tests show immediate wake-now paths dispatch instead of waiting for the scheduled heartbeat.

## Solution Assessment

Yes; the current main solution is the maintainable path because it centralizes wake deferral by declared intent while preserving flood protection and retryable busy handling.
