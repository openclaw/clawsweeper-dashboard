## Summary

Current `main` already strips inline reply directives from outbound iMessage text, preserves reply metadata separately via `reply_to`, and includes shared tests for both `[[reply_to_current]]` and `[[reply_to:<id>]]`. The PR is redundant against shipped code and targets a stale path/layout.

## Evidence

- **iMessage send path strips delivery tags before RPC send:** `sendMessageIMessage` runs `stripInlineDirectiveTagsForDelivery(message).text` before building RPC params, then sends native reply metadata separately in `params.reply_to` instead of leaving directive text in the message body.
