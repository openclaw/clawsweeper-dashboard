## Label Justifications

- `P1`: The report describes a startup regression that can make channel replies, MCP startup, and gateway RPCs time out for real users on v2026.5.22.
- `impact:message-loss`: The reported cascade includes delayed or undeliverable Feishu and WeCom messages while the event loop is starved.

## Summary

Current main covers the reported all-provider startup starvation: provider auth warming now runs after gateway readiness and uses the read-only catalog path that avoids blocking provider discovery; the proof commit is main-only and not in a shipped tag I could verify.

## What This Changes

The issue reports provider auth pre-warming blocking the Node.js event loop for 60-90s during gateway startup, causing channel, MCP, Control UI, and message timeouts.

## Best Possible Solution

Keep the current main implementation and let the next release carry the read-only, delayed auth prewarm behavior; keep Discord-specific probe fallout tracked separately.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for v2026.5.22 from the detailed logs and source comparison: the released warm path used the full catalog load, while current main moved that path to read-only catalog loading; I did not run the Windows live scenario.

## Solution Assessment

Is this the best way to solve the issue? Yes for the central all-provider stall: current main removes provider-runtime catalog discovery from the warm path and keeps prewarm after readiness without adding a compatibility-sensitive config knob.
