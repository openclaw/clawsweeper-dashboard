## Summary

Keep open: the latest PR head still opts auth/profile file locks into a stale-removal primitive that rechecks the observed sidecar and then deletes by pathname, which can remove a fresh lock under contention. The `maintainer` and dependency-change labels also make this a human review item rather than cleanup-close material.

## What This Changes

The branch bumps `@openclaw/fs-safe` to 0.2.4, adds dead-owner stale-lock policy/helpers/tests, and opts plugin SDK file locks into stale auth/profile lock recovery.

## Best Possible Solution

Keep stale auth sidecar recovery fail-closed until fs-safe deletion is identity-bound, or land an equivalent recovery path with multi-process contention proof.

## Reproduction Assessment

Yes, source-reproducible. Current main routes auth-store and OAuth refresh writes through `withFileLock`, and the PR’s opt-in stale recovery reaches the fs-safe removal path that rechecks and then deletes by pathname.

## Solution Assessment

No, not at the current head. Moving stale-removal mechanics into fs-safe is the right boundary, but `@openclaw/fs-safe@0.2.4` does not make the final delete identity-bound.
