## Summary

Keep this PR open for maintainer handling: the latest head still opts auth/profile locks into `@openclaw/fs-safe@0.2.4` stale deletion, and that dependency path still verifies the snapshot before deleting by pathname. Under concurrent stale reclaimers, one process can remove another process's freshly acquired auth lock, so the patch can weaken credential-store mutual exclusion.

## What This Changes

The branch bumps `@openclaw/fs-safe` to 0.2.4, enables stale lock removal in the plugin SDK file-lock wrapper, adds PID-owner liveness tests/helpers, updates the changelog, and includes small plugin packaging/wizard test adjustments.

## Best Possible Solution

Keep the auth lock wrapper fail-closed until fs-safe provides an identity-bound stale-deletion primitive, or land equivalent dependency-source proof and multi-process contention proof before enabling removal in OpenClaw.

## Reproduction Assessment

Yes, source-reproducible. Current main routes auth-store and OAuth refresh writes through `withFileLock`, and the PR head enables fs-safe stale recovery on that wrapper while fs-safe 0.2.4 still rechecks then deletes by pathname.

## Solution Assessment

No, not at the current head. Moving stale-removal mechanics into fs-safe is the right boundary, but the consumed dependency version still lacks identity-bound deletion or contention proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep stale recovery fail-closed until deletion is identity-bound:** `src/plugin-sdk/file-lock.ts:87`
  - body: Enabling `remove-if-unchanged` here still relies on fs-safe code that rechecks the observed sidecar and then deletes `lockPath` by pathname.
