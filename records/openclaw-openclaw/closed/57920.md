## Summary

Keep open. Current `main` still rebuilds `childSessions` from ended/stale child links, so the reported ghost-child-session behavior is not clearly fixed. There is now TTL-based pruning for completed subagent runs, which weakens the `runs.json grows unbounded` part, but stale `spawnedBy` / `parentSessionKey` session-store rows can still be reattached on read.

## Evidence

- **Read-path still attaches ended run children:** `resolveChildSessionKeys()` iterates controller runs and adds `childSessionKey` when the latest controller matches, but it does not exclude runs with `endedAt` or `cleanupCompletedAt` set.
  - file: [src/gateway/session-utils.ts:296](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/src/gateway/session-utils.ts#L296)
  - sha: [7e49cc87f9e9](https://github.com/openclaw/openclaw/commit/7e49cc87f9e9cdad47d0572331d7a7a47281a83e)
- **Stale child-side session rows still reconstruct parent linkage:** The same function also scans every session-store row and reattaches any row whose `spawnedBy` or `parentSessionKey` matches the controller. If no newer subagent-run record exists, terminal/stale store rows are still added to `childSessions`.
  - file: [src/gateway/session-utils.ts:310](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/src/gateway/session-utils.ts#L310)

- **Current tests still expect ended children on parent rows:** `listSessionsFromStore` test on current `main` asserts that the main session keeps both a running child and an ended/failed child in `childSessions`, which matches the issue rather than disproving it.
  - file: [src/gateway/session-utils.subagent.test.ts:109](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/src/gateway/session-utils.subagent.test.ts#L109)
