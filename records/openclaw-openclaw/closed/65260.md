## Summary

Current main contains the OpenAI Codex transport fix that addresses this latency class: native ChatGPT/Codex Responses requests now strip the oversized/unsupported payload fields before send, and the PI OAuth-wrapped stream is preserved instead of being replaced. The fix is on main after v2026.4.27, so this can close as implemented-on-main rather than remain open for general latency triage.

## What This Changes

Investigate whether the `openai-codex` OAuth Responses path has a configuration, transport, cache, or provider bottleneck causing consistent 12-13s server-side latency independent of cache hits, model choice, prompt size, and session history.

## Best Possible Solution

Keep the current main fix, ship it in the next release, and treat any remaining Codex latency after that build as a fresh diagnostic report with model-call timing evidence rather than keeping this pre-fix report open.

## Reproduction Assessment

Yes. The issue includes a concrete live reproduction path and timing data; I did not run live Codex OAuth timing here, but current-main source and regression tests cover the exact provider payload and OAuth stream-wrapper path implicated by the related provider/transport delay report.

## Solution Assessment

Yes. The implemented fix is the narrowest maintainable solution for this latency class: sanitize only native ChatGPT/Codex Responses payloads and preserve custom compatible endpoints and existing OAuth-wrapped streams.
