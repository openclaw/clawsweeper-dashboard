## Summary

Current `main` already routes Telegram/channel sessions through the gateway-bindable plugin registry, and the shipped `v2026.4.23` tag contains the same wiring. The reported gap between webchat and Telegram tool availability is no longer present in the maintained code paths.

## Evidence

- **Gateway startup pins a gateway-bindable plugin registry:** Gateway plugin startup initializes the active registry in `gateway-bindable` mode and loads plugins with `runtimeOptions.allowGatewaySubagentBinding: true`, which is the registry mode channel sessions need to see plugin-owned tools.
  - file: [src/gateway/server-plugins.ts:487](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/src/gateway/server-plugins.ts#L487)
  - command: `nl -ba src/gateway/server-plugins.ts | sed -n '470,515p'`
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
- **Channel-session agent runs opt into gateway subagent binding:** Inbound auto-reply execution passes `allowGatewaySubagentBinding: true` into `runEmbeddedPiAgent(...)`, so Telegram/channel turns build their tool set against the gateway-bindable runtime instead of a webchat-only path.
  - file: [src/auto-reply/reply/agent-runner-execution.ts:1076](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/src/auto-reply/reply/agent-runner-execution.ts#L1076)
