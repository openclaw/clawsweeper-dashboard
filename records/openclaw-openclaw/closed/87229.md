## Summary

Keep open: current main still derives WhatsApp/media metadata into the canonical inbound context, but the generic `message_received` plugin event mapper drops those fields before plugin hooks receive the event. The related opt-in issue remains adjacent rather than canonical for this payload-loss bug.

## What This Changes

The issue asks OpenClaw to preserve inbound WhatsApp media metadata in plugin `message_received` event payloads.

## Best Possible Solution

Forward the already-derived media fields through `toPluginMessageReceivedEvent` metadata and add focused regression coverage for both the generic mapper and WhatsApp hook emission.

## Reproduction Assessment

Yes from source inspection: WhatsApp passes a canonical inbound context into `toPluginMessageReceivedEvent`, and that mapper omits the canonical media fields from metadata. I did not run a live WhatsApp reproduction because this review had to keep the checkout read-only.

## Solution Assessment

Yes; adding the existing canonical media fields to the existing plugin event metadata is the narrowest maintainable fix. The separate hook opt-in config mismatch should stay with https://github.com/openclaw/openclaw/issues/86390.
