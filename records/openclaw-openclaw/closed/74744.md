## Summary

Keep open. Current main still applies the default 10s deadline to the Signal SSE receive stream, and this PR is an active, plausible fix with focused regression coverage; the remaining decision is maintainer selection between this PR and overlapping #73086, not cleanup closure.

## What This Changes

The PR changes the Signal client and tests so SSE event streams no longer inherit the 10s default RPC/check timeout unless a caller passes `timeoutMs`.

## Best Possible Solution

Land one narrow Signal SSE timeout fix that keeps quiet-but-healthy `/api/v1/events` streams open while preserving bounded RPC/check deadlines, explicit stream header deadlines, parsing limits, abort cleanup, and reconnect behavior; then close the linked bug and the unselected overlapping PR.

## Reproduction Assessment

Yes. The focused reproduction is an SSE endpoint that sends response headers and then stays idle; current main's `streamSignalEvents` path still arms the 10s default deadline, matching the linked issue logs and the PR's regression test target.

## Solution Assessment

Unclear. This PR is a small maintainable fix, but #73086 solves the same bug with an explicit monitor-level opt-out, so the best final path depends on whether maintainers want no default deadline for all Signal event-stream callers or only for the long-lived monitor.
