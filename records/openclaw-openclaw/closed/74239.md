## Summary

Current main and v2026.4.26 already prevent the reported bodyless `400 status code (no body)` from entering context-overflow recovery: overflow recovery now requires explicit overflow signatures, and no-body 400/422 wrappers classify as unknown/null rather than format/context failures. The remaining unsupported-metadata root cause mentioned in the report is separate and still tracked by #73963.

## Best Possible Solution

Close this issue as shipped in v2026.4.26/current main. Keep the current behavior that bodyless 400/422 provider responses surface as hard/unknown provider failures instead of context-overflow recovery, and leave the separate openai-codex metadata regression to #73963.
