## Summary

Current main still stores sessions as local JSON metadata plus JSONL transcripts, and there is no PostgreSQL/MySQL or backend-selector implementation. The remaining external/session-storage abstraction work is already covered by the broader open tracker #63492, while the related SQLite PR #58550 is only performance/local-store context and is not a merged PostgreSQL implementation.

## Best Possible Solution

Close this issue as superseded by #63492 and keep the design centralized there: add a core session storage abstraction under the session/gateway storage path, preserve JSON file storage as the default, and make SQL backends such as PostgreSQL/MySQL possible through that design and migration tooling. Treat #58550 as useful local performance context unless it is reworked or merged into the canonical abstraction.
