## Label Justifications

## Summary

Current main and the latest release already implement the requested Telegram fallback suppression: tool-only source delivery is carried through the dispatch result and prevents the empty-response fallback from firing.

## What This Changes

The issue requests suppressing Telegram's generic empty-response fallback when a turn has already sent a visible reply through the message tool and ends with an intentional NO_REPLY sentinel.

## Best Possible Solution

Keep the shipped message-tool-only suppression behavior; any future report should include sanitized logs showing this fallback still firing on v2026.6.1 or newer.

## Reproduction Assessment

No: current main and v2026.6.1 already suppress the Telegram empty-response fallback when the dispatch result is message-tool-only, including the zero-final/zero-delivery test shape. I did not run a live Telegram repro because this read-only review could prove the current source behavior directly.

## Solution Assessment

Yes: the existing implementation is broader and cleaner than checking exact NO_REPLY after a successful send because it preserves the message_tool_only contract that final assistant text stays private and silence means no message-tool send.
