## Summary

Keep open: this is a plausible `openclaw health` bug fix, but the patch needs revision because it would report intentionally disabled configured channels as runtime failures, and the external contributor has not provided after-fix real behavior proof.

## What This Changes

The PR changes health channel formatting to print `failed (runtime)` when a channel summary has `running: false` and a non-empty `lastError`, and adds a unit regression for an iMessage Messages DB access error.

## Best Possible Solution

Surface stopped runtime failures only for enabled participating accounts, preserve disabled and unconfigured states as non-failures, and cover both the iMessage error case and the disabled-account edge case before merge.

## Reproduction Assessment

Yes, from source: current main falls through to `configured` for configured channel summaries without status/probe output even when runtime state has `running: false` and `lastError`. I did not run a live iMessage setup in this read-only review.

## Solution Assessment

No. The direction is narrow, but the current condition is too broad because gateway runtime state uses `running: false` plus a `lastError` for disabled configured accounts too.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Skip disabled accounts when surfacing runtime errors:** `src/commands/health-format.ts:209-210`
  - body: This new branch treats any `running === false` summary with a `lastError` as a runtime failure. Gateway runtime state also uses `enabled: false`, `configured: true`, `running: false`, and `lastError: "disabled"` for intentionally disabled channels, while status issue collection skips disabled accounts. With this PR, `openclaw health` would report an intentional disabled state as `failed (runtime) - disabled`; gate this on enabled accounts and add coverage for the disabled case.
  - confidence: 0.88
