## Summary

Close as superseded by the open canonical tracker #57447. Current main still has the reported cross-agent `sessions_send` denial under non-`all` session visibility, but #57447 tracks that same remaining authorization bug and open PR #57483 is the implementation candidate; the adjacent parent-only sub-agent request #18967 remains open for the separate safer parent-scoped API design.

## What This Changes

The issue requests migration/UX handling for existing isolated cron workflows whose cross-agent `sessions_send` calls started returning forbidden after `tools.sessions.visibility` began defaulting to `tree`.

## Best Possible Solution

Consolidate the remaining runtime behavior under #57447 and review or replace PR #57483 on current main, while leaving #18967 open for a separate parent-only API. The end state should let explicitly A2A-authorized cross-agent sends work without broad transcript visibility, with docs and errors aligned to that model.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: with default `tools.sessions.visibility="tree"`, a cross-agent target hits the visibility rejection in `createSessionVisibilityChecker` before `tools.agentToAgent` policy can authorize `sessions_send`.

## Solution Assessment
