## Summary

Keep open: the reported cron delivery-channel problem is valid enough to fix, but this PR regresses existing persisted delivery writeback semantics by restoring the whole delivery object after reload.

## What This Changes

The PR snapshots `CronJob.delivery` across manual, timer-driven, and startup catch-up cron runs and adds Telegram delivery preservation regression coverage.

## Best Possible Solution

Protect the explicit delivery channel by isolating execution-time mutation or reconciling only immutable channel identity, while preserving persisted delivery target/account/thread writebacks after reload.

## Reproduction Assessment

Yes for the PR regression: current main has a focused manual-run source test that expects persisted Telegram target writebacks to survive, and the PR flips that expectation. The original timer/startup channel revert remains source-plausible from the stored-object execution path, but I did not live-run it in this read-only review.

## Solution Assessment

No. Restoring the entire `delivery` object is broader than necessary; the narrower maintainable fix is to prevent or reconcile only the channel mutation without discarding valid persisted delivery writebacks.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve manual-run delivery writebacks:** `src/cron/service/ops.ts:120-123`
  - body: Manual runs force-reload the cron store before the final persist so delivery target normalization or writebacks made during execution survive. Restoring the pre-run delivery snapshot here discards persisted `delivery.to`, `accountId`, or `threadId` updates and the PR changes the existing regression to expect that stale target.
  - confidence: 0.93
