## Summary

The PR addresses a source-reproducible TUI abort bug covered by the docs, and the patch is narrowly scoped with focused regression coverage; it should stay open because it is an unmerged protected PR that needs normal maintainer review rather than cleanup closure.

## What This Changes

The PR adds TUI pending run-id state so Esc and `/abort` can abort a chat send after `chat.send` resolves but before the first gateway event sets `activeChatRunId`.

## Best Possible Solution

Land this TUI-local state fix after maintainer review, preserving the existing strict `chat.abort { sessionKey, runId }` protocol and the added regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows `sendChat` can resolve with no `activeChatRunId`, and `abortActive` then takes the `no active run` branch until the first matching chat event binds the run id.

## Solution Assessment

Yes. Tracking the acknowledged run id in TUI state is the narrowest maintainable fix for the pre-event abort gap and avoids broadening the gateway abort protocol.
