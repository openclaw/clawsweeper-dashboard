## Summary

Keep open: current main still routes the top-level managed proxy through the same `global-agent` Node core hook, the docs/tests still preserve that contract, and the linked draft PR is the active fix path rather than a merged implementation.

## What This Changes

The issue reports that `proxy.enabled=true` with a local HTTP forward proxy makes Discord TLS verification validate against `localhost` instead of the Discord gateway host.

## Best Possible Solution

Land a maintainer-reviewed fix like #76442 that preserves the original target host/SNI for managed HTTPS CONNECT sockets, adds deterministic TLS validation coverage, and keeps the documented proxy coverage intact.

## Reproduction Assessment

Yes, at source level. The report gives exact config, logs, and a minimal `global-agent` reproduction, and current main still activates that same managed proxy bootstrap path; I did not run a live gateway repro in this read-only review.

## Solution Assessment

Yes for keeping this issue open. The narrow maintainable path is to review and land the linked draft PR or an equivalent SNI-safe managed proxy fix, not to close the issue or rely on owner-specific proxy workarounds.
