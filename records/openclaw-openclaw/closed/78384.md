## Summary

Keep open: current main still appends generic reasoning-only and empty-response visible-answer retry instructions without a latest-inbound user-message anchor. The Telegram supersession fixes cover stale delivery ordering, but the confirmed semantic retry mismatch inside the embedded runner remains a focused open bug.

## What This Changes

The issue asks OpenClaw to prevent reasoning-only, tool-use, and empty-response retry continuations from producing a final reply for stale nearby user intent instead of the latest inbound message.

## Best Possible Solution

Add a generic embedded-runner latest-inbound/run-intent anchor to retry prompt construction, with regression coverage proving a retry continuation cannot answer an older nearby user turn.

## Reproduction Assessment

No high-confidence live replay was established. The production confirmation and current source show the retry path clearly enough for source-level reproduction, but the exact Telegram stale-intent final answer was not replayed on current main.

## Solution Assessment

Yes. The narrow maintainable solution is a core embedded-runner latest-turn anchor with focused regression tests; a Telegram-only dispatch change is insufficient because the reported mismatch happens inside retry continuation semantics.
