## Summary

Current main already handles generated plugin-skill symlinks without treating valid plugin-owned targets as path escapes, and there is focused browser-plugin coverage for the reported symlink shape. The proof commit is on main but is not contained in a shipped tag in this checkout.

## What This Changes

The issue reports a misleading startup warning when a generated plugin-skill symlink points from the OpenClaw plugin-skills directory to a bundled plugin skill outside that directory.

## Best Possible Solution

Keep the current main implementation and let the next release carry this startup-warning fix to stable users.

## Reproduction Assessment

No for current main: source inspection shows the generated plugin-skill symlink path is accepted when it resolves inside an enabled plugin skill root, and the browser-automation symlink shape is covered by the current test fixture. The reported warning matches v2026.5.5-era behavior rather than the checked-out main behavior.

## Solution Assessment

Yes. The current fix is narrow because it preserves escape warnings for arbitrary symlink targets while allowing only OpenClaw-generated plugin-skill symlinks whose realpath remains under the enabled plugin's declared skill roots.
