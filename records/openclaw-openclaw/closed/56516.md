## Summary

Keep open. Current main still matches the reported behavior: one-shot `sessions_spawn` defaults to keeping child sessions, keep-mode subagent runs are not scheduled for archive/delete, `sessions.list` still returns completed subagent rows from the session store, and the Control UI renders that list. Related open issue #47975 remains the broader stale subagent/session responsiveness tracker, while this issue is still useful as the focused Control UI session-list cleanup report.

## Best Possible Solution

Keep this issue open as the focused Control UI/session-list tracker. Maintainers should decide the intended retention contract for one-shot subagents, then implement it consistently across `sessions_spawn`, subagent registry cleanup/archive behavior, `sessions.list`, and the Control UI Sessions page, with focused regression tests for completed run-mode subagents disappearing or being intentionally retained with explicit filtering/docs.
