## Summary

Keep open: the PR is a useful Telegram topic-context fix, but the cache write is one-sided and still misses sibling OpenClaw forum-topic mutation paths, and the PR still lacks live Telegram proof for a topic/session-context change.

## What This Changes

The branch adds Telegram topic-name cache reads in native command context and cache writes after forum-topic create/edit actions, with action-runtime regression tests.

## Best Possible Solution

Centralize or share account-scoped topic-name cache updates wherever OpenClaw creates or renames Telegram forum topics, keep the native command cache read, and verify with live Telegram proof that agent context includes the topic name.

## Reproduction Assessment

Yes. Source inspection of current main shows normal inbound messages can populate `TopicName`, while native command context does not read it and forum-topic create/edit actions do not write it back to the cache.

## Solution Assessment

No, not as submitted. The native-command read path is appropriate, but the cache writes should cover all OpenClaw topic mutation paths or be centralized so thread bindings and auto-topic renames do not drift.
