- **[P1] Keep reasoning stripping code-aware:** `extensions/whatsapp/src/outbound-media-contract.ts:73-77`
  - body: This raw fallback runs before the shared sanitizer and treats any `<final>...</final>` or `<think>...</think>` substring as control markup. A normal answer that teaches these tags or includes them inline/fenced can be truncated or emptied before the shared code-aware parser preserves it, so WhatsApp can silently drop legitimate reply text. Please move the unclosed-tag repair into the shared sanitizer/parser or add equivalent code-aware handling here with literal inline and fenced regression cases.
  - confidence: 0.9
- **[P1] Do not suppress chunks from literal tag examples:** `extensions/whatsapp/src/auto-reply/monitor/inbound-dispatch.ts:685-688`
  - body: The streaming flag is driven by raw regex matches on each WhatsApp chunk, so a visible message chunk that mentions `<think>` without a matching close in the same chunk can set suppression and hide later chunks until another raw marker appears. Because this runs before any shared reasoning parser state, the channel can drop valid user-visible WhatsApp reply chunks. Please use a shared reasoning stream parser or add code-aware state with tests for literal tag examples across chunks.
  - confidence: 0.87
