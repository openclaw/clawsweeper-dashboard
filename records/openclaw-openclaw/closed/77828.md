## Summary

Keep open: this recent contributor PR still contains merge-blocking correctness and security design issues, and current main does not already implement the requested Hermes arbiter metadata contract.

## What This Changes

The PR adds Hermes arbiter metadata plumbing across Gateway send, outbound hook delivery, task and exec/media completion paths, generated Swift models, and targeted tests.

## Best Possible Solution

Define a generic, validated metadata contract with reviewed caller policy, keep async media completions requester-session mediated, and move Hermes routing policy into plugin/config-owned code with live proof.

## Reproduction Assessment

Yes for the review blockers by source inspection: current main lacks Gateway send metadata and preserves requester-session-mediated async media completion, while the PR adds arbitrary send metadata and an early direct-send path. I did not run a live Hermes setup.

## Solution Assessment

No. The proposed path mixes a service-specific Hermes contract, arbitrary client-controlled metadata, and a media delivery policy change into core; the safer direction is a validated generic contract plus plugin/config-owned routing.
