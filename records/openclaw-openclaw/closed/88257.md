Keep open: current `origin/main` and the latest release still miss the exact `ws` pre-handshake close message in the benign uncaught-exception classifier, while the global handlers exit on non-benign uncaught exceptions. This is a narrow gateway/channel availability bug with no canonical duplicate found.

## What This Changes

The issue reports that a pre-handshake WebSocket close can surface as an uncaught exception that is not classified as benign, causing the gateway process to exit.

## Best Possible Solution

Add a narrow central benign-classifier case for the exact `ws` pre-handshake close error with focused regression coverage, then verify the Feishu/gateway path no longer exits the process.

## Reproduction Assessment

Source-level reproduction is clear for the classifier gap: `isBenignUncaughtExceptionError(new Error("WebSocket was closed before the connection was established"))` follows the false path on current `origin/main`. I did not run a live Feishu gateway crash reproduction.

Yes: the maintainable fix is to centralize the exact `ws` transient close classification in `src/infra/unhandled-rejections.ts` with tests, rather than duplicating guards in `src/index.ts` and `src/cli/run-main.ts`.
