## Repair Work Prompt

Fix the core runtime bug reported at https://github.com/openclaw/openclaw/issues/86586: when one assistant message emits multiple parallel `sessions_send` calls, a faster callee's pending A2A/pingback delivery must not be orphaned behind a slower sibling tool call or lost when the caller run ends. Start by adding a focused regression that simulates a cron/run-scoped caller issuing two `sessions_send` calls, with one callee completing and attempting requester delivery while the sibling remains blocked. Then repair the narrow runtime/session delivery path so the caller receives the queued inter-session message before it resumes or the message is durably recoverable after run termination. Preserve provider tool-call pairing semantics, existing A2A skip behavior for parent-owned child sessions, and current docs/API shape; do not add a config option unless maintainers explicitly ask.
