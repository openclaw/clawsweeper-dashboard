## Summary

Close as duplicate: the remaining auto-reply heartbeat model persistence bug is already tracked by the older open canonical issue at https://github.com/openclaw/openclaw/issues/62954, and current source inspection shows the same root cause rather than a distinct WhatsApp-only failure.

## What This Changes

The issue reports that heartbeat/background runs can stamp their heartbeat provider/model into a normal user session, causing status surfaces to show the heartbeat model as if the user session switched models.

## Best Possible Solution

Keep the remaining implementation discussion on https://github.com/openclaw/openclaw/issues/62954 and fix the auto-reply persistence boundary there so heartbeat runs preserve the prior shared-session runtime model identity.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: a heartbeat with a different heartbeat.model reaches auto-reply with that provider/model, and current session usage persistence still writes those run values into the shared session entry that status/display resolvers prefer. I did not run a live WhatsApp heartbeat in this read-only pass.

## Solution Assessment

Yes, closing this as duplicate is the best cleanup path because the older open canonical issue tracks the same root cause and direct fix boundary. The code fix should be a narrow heartbeat-preserve mode in auto-reply session usage persistence, not a status-only display mask.
