## Summary

Keep open: current main and v2026.6.9 still only recover the older concatenated cron-key corruption, while this issue's whitespace-padded `job` keys remain unrecovered and are already paired with open closing PRs.

## What This Changes

The issue reports that `cron.add` rejects jobs from affected local-model tool calls because top-level `job` keys such as `schedule`, `sessionTarget`, `payload`, and `enabled` arrive with trailing spaces.

## Best Possible Solution

Land one narrow cron canonicalization fix that normalizes recognized whitespace-padded cron job keys while preserving strict gateway validation for unknown or conflicting fields.

## Reproduction Assessment

Yes, source-level. Current main sends `cron.add` jobs through `canonicalizeCronToolObject`, which does not recover whitespace-padded keys, and strict `CronAddParamsSchema` rejects unexpected padded properties.

## Solution Assessment

Yes. The best fix boundary is the cron canonicalizer that already owns model-shaped cron argument repair, not a global parser change or a gateway schema loosening.
