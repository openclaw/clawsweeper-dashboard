## Summary

Keep open. Current main still has unbounded GatewayClient reconnect scheduling, and this active PR is not merge-ready because it adds only an opt-in API, does not wire any existing caller, lacks real behavior proof, and still has a timer-cleanup bug in the cap path.

## What This Changes

The PR adds an optional `maxReconnectAttempts` GatewayClient option, counts and resets reconnect attempts, pauses reconnect after the cap, and adds reconnect unit tests.

## Best Possible Solution

Settle a caller-aware retry policy, wire any intended cap to the short-lived client surface, clear lifecycle timers before pausing, document or validate option semantics, and keep the linked issue open until that behavior is actually covered.

## Reproduction Assessment

Yes for the source-level loop: ordinary close handling reaches `scheduleReconnect()`, which schedules another `start()` without an attempt counter on current main. The long-running zombie-process and CPU claims still lack live reproduction in this review.

## Solution Assessment

No. An opt-in cap may be a useful API, but this patch does not apply it to any affected caller and returns before timer cleanup on the cap path.
