## Summary

Current main already addresses the reported document-context blowout: inbound attachments go through the media-understanding file path, extracted file text is hard-capped, and binary Office MIME types like `.xlsx` are now skipped instead of being decoded into prompt text.

## Evidence

- **Inbound channel attachments now flow through media-understanding before agent execution:** `getReplyFromConfig` calls `applyMediaUnderstanding` whenever inbound media is present, so Discord/Telegram attachments hit the shared file-handling path before the agent sees the prompt.
  - file: [src/auto-reply/reply/get-reply.ts:131](https://github.com/openclaw/openclaw/blob/84fc00afdacbbb67e39fc1341242eedad09243df/src/auto-reply/reply/get-reply.ts#L131)
  - command: `nl -ba src/auto-reply/reply/get-reply.ts | sed -n '120,170p'`
  - sha: [84fc00afdacb](https://github.com/openclaw/openclaw/commit/84fc00afdacbbb67e39fc1341242eedad09243df)
- **Binary vendor Office MIME types are treated as binary, not text:** `isBinaryMediaMime` returns true for `application/vnd.*` payloads except `+json`/`+xml`, which covers Excel MIME types such as `.xls`/`.xlsx` and prevents them from being coerced into file-text extraction.
  - file: [src/media-understanding/apply.ts:332](https://github.com/openclaw/openclaw/blob/8d1f98ef08cda2809ce3c5b3491a67c10a8afa88/src/media-understanding/apply.ts#L332)
  - command: `nl -ba src/media-understanding/apply.ts | sed -n '325,350p'`
  - sha: [8d1f98ef08cd](https://github.com/openclaw/openclaw/commit/8d1f98ef08cda2809ce3c5b3491a67c10a8afa88)
- **Binary attachments are skipped before extraction/injection:** During file-block extraction, attachments with binary MIME and no forced text mapping are immediately skipped instead of being appended into `ctx.Body`.
  - file: [src/media-understanding/apply.ts:389](https://github.com/openclaw/openclaw/blob/8d1f98ef08cda2809ce3c5b3491a67c10a8afa88/src/media-understanding/apply.ts#L389)
  - command: `nl -ba src/media-understanding/apply.ts | sed -n '380,472p'`
  - sha: [8d1f98ef08cd](https://github.com/openclaw/openclaw/commit/8d1f98ef08cda2809ce3c5b3491a67c10a8afa88)
- **Extracted file text now has a hard character cap:** `DEFAULT_INPUT_FILE_MAX_CHARS` is `60_000`, and `extractFileContentFromSource` clamps extracted PDF text and decoded text files to `limits.maxChars` before returning content for prompt injection.
  - file: [src/media/input-files.ts:112](https://github.com/openclaw/openclaw/blob/8d1f98ef08cda2809ce3c5b3491a67c10a8afa88/src/media/input-files.ts#L112)
  - command: `nl -ba src/media/input-files.ts | sed -n '102,117p'; nl -ba src/media/input-files.ts | sed -n '375,394p'`
  - sha: [8d1f98ef08cd](https://github.com/openclaw/openclaw/commit/8d1f98ef08cda2809ce3c5b3491a67c10a8afa88)
- **The cap is wired to configurable gateway file limits:** `applyMediaUnderstanding` resolves file limits from `cfg.gateway.http.endpoints.responses.files`, so the truncation behavior is configurable rather than hard-coded at the call site.
  - file: [src/media-understanding/apply.ts:86](https://github.com/openclaw/openclaw/blob/8d1f98ef08cda2809ce3c5b3491a67c10a8afa88/src/media-understanding/apply.ts#L86)

  - file: [src/media-understanding/apply.test.ts:1388](https://github.com/openclaw/openclaw/blob/8d1f98ef08cda2809ce3c5b3491a67c10a8afa88/src/media-understanding/apply.test.ts#L1388)
