## Label Justifications

- `P1`: The reported symptom breaks real Slack reply delivery for CLI-runtime users on the latest shipped release.
- `impact:message-loss`: The issue is about assistant reply content being replaced before Slack delivery.
- `impact:auth-provider`: The failure is caused by routing a CLI runtime/auth backend id through the embedded harness registry.

## Summary

Close: current main already routes configured CLI runtime ids away from harness validation on this path; the fix is on main with regression coverage but is not included in v2026.5.20.

## What This Changes

The issue asks OpenClaw to stop treating a model-scoped claude-cli CLI runtime id as an embedded agent harness during Slack reply delivery and transcript mirroring.

## Best Possible Solution

Keep the shared CLI-runtime dispatch guard on main, and leave the separate compaction preflight MissingAgentHarnessError path tracked by https://github.com/openclaw/openclaw/issues/84857.

## Reproduction Assessment

No: I did not reproduce a failing current-main path; current main now skips harness validation for CLI runtime ids and has focused regression coverage for that dispatch boundary.

## Solution Assessment

Yes: solving this at the shared model-fallback and CLI dispatch boundary is narrower and safer than adding a Slack-specific workaround; the remaining sibling compaction path is already tracked separately.
