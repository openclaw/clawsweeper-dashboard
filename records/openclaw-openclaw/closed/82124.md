## Summary

Keep open. Current main still has no public `config unset` flag that forwards the existing narrow `allowConfigSizeDrop` write option, so the requested CLI affordance remains unimplemented and needs maintainer policy approval before changing destructive-write behavior.

## What This Changes

Add a `config unset` opt-in flag that bypasses only the size-drop guard for intentional large config deletes.

## Best Possible Solution

Expose a narrowly named `config unset` opt-in that sets `allowConfigSizeDrop` only for existing-path explicit deletes, with regression coverage for default rejection, allowed size drop, missing paths, schema validation, and hard guards staying blocked.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: `config unset` can remove a large node and records `unsetPaths`, but it never forwards `allowConfigSizeDrop`, while the write layer rejects drops below half the original byte size unless that internal option is true.

## Solution Assessment

Unclear pending maintainer product decision. The proposed narrow flag matches the existing lower-level write option, but maintainers should choose the public flag name and confirm the destructive-write escape hatch belongs on `config unset`.
