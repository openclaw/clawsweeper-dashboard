## Summary

Keep open: the patch is focused and I found no blocking code-level defect, but the contributor proof is only mocked and the automatic restart behavior is availability-sensitive enough to need real plugin-channel proof and maintainer acceptance before merge.

## What This Changes

The branch adds a post-login CLI fallback that requests `gateway.restart.request` when local `channels.start` reports a missing runtime channel, plus a mocked regression test.

## Best Possible Solution

Land the narrow fallback only after redacted real external-plugin proof shows saved auth triggers a safe restart and the channel runs afterward; pair it with the startup-loader fix if restart alone still cannot load the plugin.

## Reproduction Assessment

No live high-confidence reproduction was run in this read-only review. Source inspection reproduces the failure path: channel login calls `channels.start`, and the gateway rejects channel ids absent from the active registry with invalid/unknown-channel errors.

## Solution Assessment

Acceptable mitigation, not yet proven best. Restart uses an existing safe lifecycle for process-stable plugin metadata, but live proof and maintainer acceptance should decide whether this beats a dedicated reload or startup-planner fix.
