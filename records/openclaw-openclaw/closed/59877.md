## Summary

Current `main` already narrows `/status` task rows through a task-status snapshot, limits completed-task visibility to recent terminal items, truncates task detail to 120 chars, and has regression coverage that stale completed cron output does not appear in `/status`. The shipped `v2026.4.23` tag contains the same logic/tests, so the reported flood is already fixed well enough on main.

## Evidence

- **`/status` now formats from a filtered task snapshot:** `formatSessionTaskLine` no longer reads the latest task directly. It builds a snapshot and renders counts/headline from `buildTaskStatusSnapshot(...)`, which is the path that filters what can appear on the status card.
  - file: [src/status/status-text.ts:85](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/status/status-text.ts#L85)
  - command: `nl -ba src/status/status-text.ts | sed -n '85,100p'`
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **Completed-task visibility is time-bounded and details are truncated:** `TASK_STATUS_RECENT_WINDOW_MS` is 5 minutes, `TASK_STATUS_DETAIL_MAX_CHARS` is 120, `formatTaskStatusDetail` truncates terminal summaries, and `buildTaskStatusSnapshot` only keeps active tasks plus recent terminal tasks. That removes the old unbounded wall-of-text behavior from completed cron output.
