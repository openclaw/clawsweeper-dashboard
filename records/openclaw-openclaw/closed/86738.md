## Summary

Current main and the latest shipped release already use a generic non-image attachment path for Control UI/WebChat uploads, so the reported audio-specific scope gap from 2026.5.20 no longer needs a separate core issue.

## What This Changes

The issue asks OpenClaw to handle Control UI/WebChat audio attachments without keeping large payloads in Lit state or optimistic chat messages, matching the PDF/image attachment offload behavior.

## Best Possible Solution

Keep the v2026.5.22/current-main generic attachment offload behavior; any remaining current-release stack overflow specifically in base64 validation should stay with https://github.com/openclaw/openclaw/issues/80679.

## Reproduction Assessment

No: I did not reproduce the old 2026.5.20 failure on current main. Source and v2026.5.22 release inspection show audio/non-image attachments now use the generic payload-store, send-time serialization, and media offload path.

## Solution Assessment

Yes: the current generic non-image offload path is the maintainable fix because it covers audio, PDF, text, and other non-video files without adding an audio-only special case.
