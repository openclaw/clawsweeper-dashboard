- **[P1] Preserve poll vote data before enqueueing updates:** `extensions/whatsapp/src/inbound/monitor.ts:472`
  - body: `messages.update` gives this handler each voter's `pollUpdate`, including selected options, but the synthetic message drops it and always sends `pollUpdateMessage: {}`. Agents and hooks would only see a generic placeholder and still could not react to actual poll results.
  - confidence: 0.9
- **[P1] Handle event response updates from Baileys:** `extensions/whatsapp/src/inbound/monitor.ts:444-446`
  - body: This branch skips every `messages.update` that lacks `pollUpdates`, but Baileys also carries RSVP updates in `eventResponses`. Those RSVP events will never reach the new `<event-rsvp:...>` extraction path when delivered as updates.
  - confidence: 0.87
- **[P1] Keep reactions out of normal reply turns by default:** `extensions/whatsapp/src/inbound/monitor.ts:547`
  - body: The reaction listener enqueues every reaction through the normal inbound pipeline, which can create an agent turn or reply from an emoji-only signal. Current related design points toward passive, opt-in reaction hooks, so this needs an explicit maintainer decision and tests before changing default WhatsApp behavior.
  - confidence: 0.82
- **[P1] Do not bypass the WhatsApp hook opt-in for gated groups:** `extensions/whatsapp/src/auto-reply/monitor/on-message.ts:145-169`
  - body: This early-return path fires `message:received` directly for unmentioned group messages without checking `channels.whatsapp.pluginHooks.messageReceived`. Current main gates WhatsApp message hooks behind explicit opt-in because payloads include private content and identifiers.
  - confidence: 0.88
- **[P2] Recognize the pinned poll creation variants:** `extensions/whatsapp/src/inbound/extract.ts:147-153`
  - body: The pinned Baileys proto includes `pollCreationMessageV5`, but this extractor only checks V1/V2/V3. Newer WhatsApp poll creation messages can still be treated as contentless and dropped.
  - confidence: 0.84
- **[P2] Distinguish unknown RSVP states from maybe:** `extensions/whatsapp/src/inbound/extract.ts:187`
  - body: Baileys defines `UNKNOWN = 0` and `MAYBE = 3`, but this mapping reports every non-going/non-not-going value as `maybe`. Empty or unknown RSVP updates would be counted as tentative attendees instead of remaining unknown.
  - confidence: 0.86
- **[P2] Preserve non-zero poll validation when rebasing:** `src/poll-params.ts:57`
  - body: The unrelated poll-param hunk treats only positive numbers as poll intent, while current main intentionally treats any non-zero finite number as intent so invalid negative durations still hit poll validation instead of silently becoming normal sends.
  - confidence: 0.78
