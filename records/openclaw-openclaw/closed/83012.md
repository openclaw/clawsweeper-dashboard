## Summary

Current main and shipped releases already use a Telegram sequential lane key that includes forum topic thread IDs, so separate topics in the same group do not share the bare chat queue.

## What This Changes

The issue asks for Telegram group topic updates to be processed on `chatId:message_thread_id` lanes instead of one lane per group chat.

## Best Possible Solution

Keep the current topic-aware Telegram lane behavior; reopen only with a redacted raw update/log showing a specific topic update shape that bypasses `message_thread_id` or `is_topic_message`.

## Reproduction Assessment

No. Current main and the shipped release source map forum topic updates to `telegram:<chatId>:topic:<threadId>`, and regression tests cover separate topic lanes in the same group chat.

## Solution Assessment

Yes. The existing shared Telegram sequential-key helper is the narrow maintainable solution because both direct bot middleware and isolated polling spool reuse the same per-topic lane key.
