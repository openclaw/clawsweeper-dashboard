## Summary

Keep open. Current `main` still has the reported architecture: the voice-call CLI and `voice_call` agent tool both call the plugin-local `ensureRuntime()`, which creates a standalone runtime and starts its own webhook server instead of routing to the already-running gateway plugin instance.

## Evidence

- **CLI still creates local runtime:** The plugin registers the `voicecall` CLI with `ensureRuntime`, and the `call`/`start`/other subcommands immediately await that local runtime before acting. There is no gateway RPC path in this CLI code.
  - file: [extensions/voice-call/index.ts:548](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/extensions/voice-call/index.ts#L548)
  - command: `nl -ba extensions/voice-call/index.ts | sed -n '150,620p'`
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
- **Agent tool still creates local runtime:** The `voice_call` tool implementation calls `ensureRuntime()` at execution start, then invokes the manager directly. No `callGatewayTool` usage or equivalent gateway delegation exists in the voice-call tool path.
  - file: [extensions/voice-call/index.ts:422](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/extensions/voice-call/index.ts#L422)
  - command: `nl -ba extensions/voice-call/index.ts | sed -n '150,620p'`
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
- **ensureRuntime still constructs standalone runtime:** `ensureRuntime()` caches and returns `createVoiceCallRuntime(...)` directly from plugin registration state. That is the same local-runtime pattern described in the report.
  - file: [extensions/voice-call/index.ts:174](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/extensions/voice-call/index.ts#L174)
  - command: `nl -ba extensions/voice-call/index.ts | sed -n '150,620p'`
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
- **Runtime always starts webhook server:** `createVoiceCallRuntime()` constructs `VoiceCallWebhookServer` and unconditionally calls `webhookServer.start()`. This means a second local runtime still attempts to bind the configured voice-call port.
  - file: [extensions/voice-call/src/runtime.ts:242](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/extensions/voice-call/src/runtime.ts#L242)

  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
- **Webhook server binds configured port:** `VoiceCallWebhookServer.start()` calls `this.server.listen(port, bind, ...)` after only guarding duplicate starts on the same instance. It does not detect or reuse a gateway-owned server/runtime from another process.
  - file: [extensions/voice-call/src/webhook.ts:431](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/extensions/voice-call/src/webhook.ts#L431)

- **No shipped CLI/SDK gateway helper for voice-call:** The voice-call extension exposes gateway methods in the plugin and documents them in the README, but its SDK/runtime barrel is only a thin re-export and does not provide a CLI or tool RPC helper analogous to patterns used by other plugins such as browser.
