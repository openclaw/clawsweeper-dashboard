## Summary

Current main and the shipped v2026.5.7 release already fix the reported Telegram startup blocker: recoverable deleteWebhook failures no longer keep the polling provider in a pre-poll retry loop, and regression coverage verifies polling starts after a cleanup timeout.

## What This Changes

Gateway restart should not block boot when Telegram deleteWebhook cleanup repeatedly fails during polling startup.

## Best Possible Solution

Keep the shipped v2026.5.7 Telegram startup behavior and handle any remaining local update failure as a separate install/update report.

## Reproduction Assessment

No current-main failure remains to reproduce. The old 2026.4.24 source shows the reported retry loop, while current main has regression coverage that reaches polling after a deleteWebhook timeout.

## Solution Assessment

Yes. The shipped fix is narrower than adding a skip knob: it keeps deleteWebhook cleanup but lets recoverable failures fall through to getUpdates conflict handling instead of blocking startup.
