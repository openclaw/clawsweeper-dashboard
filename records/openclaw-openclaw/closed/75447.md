## Summary

Keep this issue open. Current main still leaves same-id ClawHub plugin records on the external update path even when a bundled plugin with the same id exists, and the linked fix PR is open but not merged, so this should not be closed before that implementation lands or is replaced.

## What This Changes

The issue requests that `openclaw update` avoid automatically replacing a newer bundled WhatsApp plugin with an older same-id ClawHub plugin.

## Best Possible Solution

Land PR #75497 or an equivalent generic fix that compares same-id bundled and ClawHub plugin versions during update-channel sync, switches older ClawHub records back to the bundled path before external refresh, preserves explicit newer external installs, and keeps npm-installed overrides covered by tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to seed `plugins.installs.whatsapp` as a ClawHub record with an older version while a bundled WhatsApp source exists; current main leaves that record untouched in bundled sync and then updates it through the ClawHub installer path.

## Solution Assessment

Yes, the best direction is a generic bundled-plugin freshness/preference guard in the plugin update flow, not a WhatsApp-specific special case. The open linked PR is the right review target; the issue should remain open until that PR or an equivalent patch lands.
