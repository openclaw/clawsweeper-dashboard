## Summary

Current main now implements the shared Gateway client/probe fix needed for the reported timeout class. The nodes CLI still uses the shared Gateway RPC wrapper for `node.pair.list` and best-effort `node.list`, and that wrapper now waits for local event-loop readiness before opening the WebSocket; `gateway probe` uses the same readiness gate. The related probe/connect-timeout regression #74279 is recorded as addressed on main, but the proof commit is newer than the latest tagged release in the review context.

## What This Changes

Make `openclaw nodes list` return reliably against a healthy loopback Gateway and avoid leaving an immediate follow-up `gateway probe` in a connect-timeout window.

## Best Possible Solution

Keep the shared Gateway client readiness implementation on main and let it ship in the next release. If a future report shows a true server-side `node.pair.list` or pairing-file stall after this fix, it should be tracked with `/readyz` event-loop diagnostics, Gateway RPC timings, and pairing-state evidence as a narrower follow-up.

## Reproduction Assessment

Yes for the affected release: the report includes concrete repeated `openclaw nodes list` commands on v2026.4.25 and an immediate `openclaw gateway probe` after a timeout. I did not run a live loopback repro in this read-only pass, but the supplied logs and current source map to the shared RPC/probe connection path.

## Solution Assessment

Yes. A shared readiness gate in the Gateway client/probe path is narrower and more maintainable than special-casing `nodes list`, because the observed failure includes both `nodes list` and `gateway probe` loopback timeouts.
