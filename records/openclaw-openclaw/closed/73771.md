## Summary

Keep open. The PR is aimed at a valid memory isolation gap on current main, but the proposed branch is not merge-ready: it is stacked over two still-open prerequisite PRs and introduces blocking migration, concurrency, and large-index regressions that were also flagged in the existing Greptile and ClawSweeper comments.

## What This Changes

The PR branch adds agent namespace scoping across memory index schema/query/state paths, memory_search scope/provenance handling, QMD mcporter daemon state, CLI embedding provider registration, and related memory-core/package tests.

## Best Possible Solution

Land the memory isolation series in order after rebasing and narrowing this branch, then fix the package FTS migration, preserve existing indexing concurrency and streaming/yielding behavior, add any required changelog entry for user-facing memory isolation changes, and validate shared store.path isolation across schema migration, writes/deletes, keyword search, vector search, status, session state, and QMD mcporter daemon state.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main has no agent_id schema/filtering for shared memory DB rows, and the PR head's Set.length check will never rebuild an existing FTS table that lacks agent_id, causing later inserts with agent_id to fail.

## Solution Assessment

No, not as-is. The owner-area direction is appropriate for agent namespace isolation, but this branch must fix the migration and preserve the existing non-batch concurrency, streaming vector search, and cache-seeding yield contracts before it is the narrowest maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Use Set.size for the FTS rebuild check:** `packages/memory-host-sdk/src/host/memory-schema.ts:46-47`
  - body: `readColumns` returns a `Set`, so `existingFtsColumns.length` is always `undefined` and this condition never rebuilds old FTS tables that lack `agent_id`. Existing package-backed databases will keep the old virtual table, and later inserts that include `agent_id` will fail after upgrade.
  - confidence: 0.98
- **[P2] Preserve non-batch index concurrency:** `extensions/memory-core/src/memory/manager-embedding-ops.ts:94-96`
  - body: This drops `remote.nonBatchConcurrency` and the Ollama non-batch default by replacing the old `configuredNonBatchConcurrency` and `providerId` inputs with `configuredConcurrency`. Current tests require inline indexing to honor that setting and default Ollama to one worker when batch mode is disabled.
  - confidence: 0.93
- **[P2] Keep fallback vector scoring bounded:** `extensions/memory-core/src/memory/manager-search.ts:248-254`
  - body: The fallback vector path now loads every matching chunk with `.all()`, scores all rows, sorts the full array, and then slices. Current main uses an iterator with a bounded top-K accumulator, so this regresses large indexes by materializing all chunks and embeddings in memory.
  - confidence: 0.91
- **[P2] Preserve yielding during cache seeding:** `extensions/memory-core/src/memory/manager-sync-ops.ts:315-340`
  - body: Changing cache seeding to `.all()` plus a fully synchronous loop removes the existing `setImmediate` yield during large rebuilds. Large embedding caches can now block the event loop long enough to delay gateway health probes and other async work.
  - confidence: 0.87
