## Summary

Keep open. Current main does not implement MCP trace-context propagation or MCP OTEL spans, and this PR is the active implementation candidate, but it still has concrete correctness findings plus a telemetry trust-boundary review item before it should merge.

## What This Changes

The PR adds an opt-in `diagnostics.otel.mcp` canary with config/schema/help updates, MCP JSON-RPC diagnostic events, loopback trace-scope handling, bundled MCP `traceparent` injection, OTEL MCP span export, and targeted tests.

## Best Possible Solution

Keep this PR as the implementation candidate, but require revisions before merge: implement full `traceparent`/`tracestate` behavior or narrow the public surface to traceparent-only, classify top-level JSON-RPC `error` responses as MCP errors, mirror trusted-event guards across MCP span handlers, document the MCP canary trust model in the OpenTelemetry docs, and validate with the targeted tests plus the changed gate.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Propagate tracestate or narrow the canary contract:** `src/gateway/mcp-http.trace.ts:64-65`
  - body: The PR body and config/help text promise `traceparent`/`tracestate` propagation through `params._meta`, but inbound extraction reads only `_meta.traceparent` and bundled outbound calls write only `traceparent`. With the canary enabled, caller `tracestate` vendor state is silently dropped, so either implement it or narrow the public contract to traceparent-only.
  - confidence: 0.88
- **[P2] Classify JSON-RPC error responses as MCP errors:** `src/gateway/mcp-http.trace.ts:141-142`
  - body: `completeMcpTraceScope` only treats `response.result.isError === true` as an error. Responses returned by `jsonRpcError` use a top-level `error` object, so unsupported MCP methods currently export `mcp.request.completed` instead of `mcp.request.error`.
  - confidence: 0.9
- **[P2] Require trusted metadata before closing MCP spans:** `extensions/diagnostics-otel/src/service.ts:2100-2118`
  - body: `recordMcpRequestStarted` rejects untrusted diagnostic events, but the completed/error handlers only check `tracesEnabled`. If an untrusted MCP diagnostic event path is introduced, these handlers can create detached exported spans; mirror the trusted guard before taking or creating spans.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff adds no dependency, CI, install, or secret-handling change, but it introduces a remote MCP trace-context trust boundary that needs attention before merge.

Concerns:

- **[medium] Document remote MCP trace-context trust:** `src/gateway/mcp-http.trace.ts:64`
  - body: When enabled, the canary parses caller-supplied `params._meta.traceparent` and uses it as the parent context for exported MCP spans. That is the intended feature, but it allows trace-correlation spoofing into an operator collector unless the opt-in risk and trust model are documented and accepted.
  - confidence: 0.78
- **[low] Guard completed and error MCP telemetry events:** `extensions/diagnostics-otel/src/service.ts:2100`
  - body: The completed/error span handlers lack the `metadata.trusted` guard used by the started handler, so a future untrusted MCP diagnostic event path could produce exported spans despite no trusted start event being accepted.
  - confidence: 0.82
