## Summary

Keep open: current main and latest release still have the heartbeat send-success write without pending-final cleanup, and the linked closing PR remains open rather than merged.

## What This Changes

This issue reports that heartbeat-driven substantive replies leave pendingFinalDelivery state set after successful delivery, causing later heartbeat ticks to redeliver stale text or skip fresh agent work.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/83187 or an equivalent heartbeat cleanup with full pending-final field parity and regression coverage, while leaving broader retry-bound work to https://github.com/openclaw/openclaw/issues/85743.

## Reproduction Assessment

Yes, at source level: current main stores substantive heartbeat output as pendingFinalDelivery, then records visible heartbeat send success without clearing those fields. I did not run a writable fixture because this review is read-only.

## Solution Assessment

Yes, clearing pending-final state in the heartbeat send-success owner path is the narrow maintainable fix for this issue. The linked PR is the implementation lane, while stale orphan replay caps belong to the adjacent follow-up.
