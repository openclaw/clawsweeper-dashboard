## Label Justifications

- `P1`: The bug can stop heartbeat-driven agent work after a successful substantive heartbeat and silently suppress later notifications or model calls.
- `impact:session-state`: The failure is stale `pendingFinalDelivery*` session state that blocks subsequent heartbeat runs.
- `impact:message-loss`: Affected deployments can miss heartbeat notifications without a visible delivery error.

## Summary

Keep open: current main still has the source-reproducible heartbeat send-success cleanup gap, and the issue already has an open linked fix PR that should be reviewed or landed before the report closes.

## What This Changes

The issue reports that substantive heartbeat-driven replies are sent but leave `pendingFinalDelivery*` session state set, causing later heartbeat ticks to short-circuit instead of invoking the agent.

## Best Possible Solution

Land or otherwise resolve the linked heartbeat cleanup PR so successful heartbeat deliveries clear pending-final state without changing failed-send retry semantics.

## Reproduction Assessment

Yes from source inspection: real heartbeat content can be persisted as pending final delivery, the heartbeat success path sends and records only heartbeat metadata, and the next get-reply call can return the stale pending text before normal agent work. I did not run a live repro in this read-only review.

## Solution Assessment

Yes: clearing the existing pending-final fields only after confirmed heartbeat send success mirrors the user-message delivery contract and preserves retry state for failed sends.
