## Label Justifications

- `P1`: The report describes a scheduled Codex cron regression that prevents shell-dependent jobs from running while returning normal assistant text instead of command execution.
- `impact:security`: The remaining fix concerns when scheduled runs may receive native shell/file execution capability, which is a sandbox and authorization boundary.
- `impact:auth-provider`: The reported failure is tied to Codex runtime/provider/harness selection and the resulting tool surface for scheduled runs.

## Summary

Keep open: current main does not provide enough proof that the manual-main versus isolated-cron Codex shell mismatch is fixed, and the open related work covers finite `toolsAllow` fail-closed behavior rather than this no-allowlist main-agent parity case.

## What This Changes

The issue reports that isolated cron `agentTurn` runs for the `main` Codex runtime expose only `message` while equivalent manual main-agent Codex turns expose native shell access.

## Best Possible Solution

Make isolated cron use the same provider/model Codex runtime policy and native tool-surface decision as manual agent turns when no restrictive per-run allowlist is set, while keeping explicit deny, sandbox, and finite-allowlist fail-closed behavior separate.

## Reproduction Assessment

No live current-main reproduction was run. Source and docs show the relevant cron, fresh-session, provider/model runtime, and Codex native-tool-surface paths, but the exact no-allowlist `main` mismatch still needs a real isolated cron proof.

## Solution Assessment

Unclear as an implementation path: #84709 is useful for finite required-tool fail-closed behavior, but this issue needs a narrower maintainer decision on manual-vs-isolated Codex runtime parity without changing explicit `toolsAllow` security semantics.
