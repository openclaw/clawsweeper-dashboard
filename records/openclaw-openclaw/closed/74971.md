## Summary

Keep open. Current main still lets config loading materialize model defaults, resolve bundled provider policy artifacts, and prepare bundled public-surface runtime deps unless the caller opts out; the provider-policy loader still does not pass that opt-out, so the reported sealed Docker EACCES path remains plausible and actionable.

## What This Changes

The issue asks OpenClaw to stop failing config load on v2026.4.27 when a sealed Docker gateway cannot write to `/var/lib/openclaw/plugin-runtime-deps`.

## Best Possible Solution

Patch the provider-policy public-artifact control-plane path so config validation and default materialization can load lightweight provider policy artifacts without preparing bundled runtime-dependency roots, while leaving startup, plugin activation, doctor, and explicit dependency repair paths able to stage runtime deps when they actually need them.

## Reproduction Assessment

Yes. The issue provides a concrete sealed Docker reproduction and exact stack trace, and current main source still follows the same config-load path into provider-policy public-artifact loading without the runtime-deps opt-out.

## Solution Assessment

Yes. The narrow maintainable solution is to pass `installRuntimeDeps: false` for provider-policy public artifact loading used by config/default control-plane work and add regression coverage; globally disabling runtime-deps staging would be broader and risk runtime plugin activation behavior.
