## Summary

Close: current main now handles the reported one-shot session-target cron stall class by keeping `runningAtMs` as the run reservation while adding bounded setup/pre-execution watchdogs that abort, record an error run, and clear the running marker. This is current-main-only; it is not in the latest tagged release.

## What This Changes

The issue reports intermittent one-shot at-cron drops where `runningAtMs` is set for a `session:` agent turn but no session, run, log, or on-disk evidence appears.

## Best Possible Solution

Keep the current watchdog-and-diagnostics implementation; broader cron health/logging enhancements can continue under https://github.com/openclaw/openclaw/issues/13593.

## Reproduction Assessment

No. I did not establish a live current-main reproduction; current source and regression tests show the old pre-start/pre-execution stall path now records an error and clears `runningAtMs` instead of remaining silently stuck.

## Solution Assessment

Yes. Preserving `runningAtMs` as the duplicate-run reservation while adding bounded setup/pre-execution watchdogs is narrower and safer than moving the marker until after spawn completion.
