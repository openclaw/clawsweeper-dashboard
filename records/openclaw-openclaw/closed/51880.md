## Summary

Current main already supports this outcome through account-scoped routing: cross-agent spawned replies inherit the target agent's bound channel account, and Telegram outbound delivery sends through that resolved account. The requested behavior exists, but via explicit agent/channel account bindings rather than a new `replyAccount` config knob.

## Evidence

- **Cross-agent reply origin resolves the target agent's bound account:** `resolveRequesterOriginForChild()` rebinds `accountId` from the target agent's channel/peer binding when the target agent differs from the requester, instead of keeping the inbound account.
  - file: [src/agents/spawn-requester-origin.ts:113](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/src/agents/spawn-requester-origin.ts#L113)
- **Subagent spawns use that resolved origin for delivery:** `spawnSubagentDirect()` calls `resolveRequesterOriginForChild()` before creating the child session origin, so cross-agent replies use the resolved bound account.
  - file: [src/agents/subagent-spawn.ts:653](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/src/agents/subagent-spawn.ts#L653)
- **Telegram outbound sends honor the resolved account id:** Telegram outbound send context carries `accountId` into `sendMessageTelegram`, so once the target account is resolved, replies go out through that Telegram bot account.
  - file: [extensions/telegram/src/outbound-adapter.ts:39](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/extensions/telegram/src/outbound-adapter.ts#L39)
- **Regression coverage asserts target-agent account routing:** The subagent spawn test sets requester account `bot-beta`, target agent binding account `bot-alpha`, then asserts the spawned delivery request uses `accountId: "bot-alpha"` for the reply path.
  - file: [src/agents/subagent-spawn.thread-binding.test.ts:73](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/src/agents/subagent-spawn.thread-binding.test.ts#L73)
- **Telegram named-account routing is documented for operators:** Docs describe explicit account-scoped bindings such as `openclaw agents bind --agent work --bind telegram:ops`, and note that `telegram:ops` scopes routing to that account.
  - file: [docs/cli/agents.md:60](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/docs/cli/agents.md#L60)
- **The same spawn-account logic is present in the latest shipped tag:** Verified `v2026.4.22` contains the same `resolveRequesterOriginForChild()` logic and the same bound-account subagent test via `git show v2026.4.22:src/agents/spawn-requester-origin.ts` and `git show v2026.4.22:src/agents/subagent-spawn.thread-binding.test.ts`.
  - command: `git show v2026.4.22:src/agents/spawn-requester-origin.ts | nl -ba | sed -n '104,131p' ; git show v2026.4.22:src/agents/subagent-spawn.thread-binding.test.ts | nl -ba | sed -n '73,154p'`

## Risks / Open Questions

- This is implemented through explicit per-agent account bindings, not a separate `replyAccount: self|inbound` config field. If operators have not bound the target agent to its own Telegram account for that conversation, replies will still follow the existing inbound-account route.
