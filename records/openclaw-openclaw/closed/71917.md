## Summary

PR #71917 can close as implemented on current main for the linked #71800 user problem. Current main no longer depends on the old `loadConfig()` path for logger startup settings: it reads `logging` directly from the active config file and applies `logging.maxFileBytes` during logger settings resolution, with tests and docs covering the config-backed behavior. The proposed `OPENCLAW_LOG_MAX_FILE_BYTES` env var is not present on main, but it was a workaround for the config-load race rather than a remaining required fix.

## Best Possible Solution

Close this PR and keep the current main implementation: logger startup should continue using the direct active-config read plus `logging.maxFileBytes` rotation. If maintainers still want a new documented env override, it should be opened as a separate product/API decision with explicit validation, warning, floor, and ceiling semantics rather than landing this workaround for an already-fixed race.
