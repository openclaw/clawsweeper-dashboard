## Close Comment

Thanks for the report. A Codex review against current `main` shows this is already fixed in shipped OpenClaw releases, so I'm closing it.

- `CHANGELOG.md:1493` lists the shipped browser/plugin fix in `v2026.4.5`: OpenClaw now auto-enables the bundled browser plugin when browser config or browser tool policy references it, and it gives clearer guidance when `plugins.allow` excludes `browser`.
- `extensions/browser/plugin-registration.ts:28` wires the browser CLI, `browser.request`, and the browser control service through the same plugin registration path.
- `src/gateway/server-plugin-bootstrap.browser-plugin.integration.test.ts:42` now asserts that loading the bundled browser plugin adds both `browser.request` and the browser control service together, covering the mismatch reported here.
- `extensions/browser/src/gateway/browser-request.ts:236` now returns an explicit `browser control is disabled` error when browser control is off, instead of relying on a missing gateway method.
- `docs/tools/browser.md:62` and `docs/gateway/troubleshooting.md:469` document the `plugins.allow` / bundled-browser-plugin loading path for this symptom.

If someone still hits the same error on a current release, the follow-up should be a new issue with the exact OpenClaw version, effective `plugins.allow` / `plugins.entries.browser.enabled` / `browser.enabled` settings, and the matching gateway logs from that run.
