## Label Justifications

- `P1`: The PR targets a high-impact gateway lifecycle regression that can leave user sessions running and block follow-up messages after provider timeouts.
- `merge-risk: 🚨 session-state`: The changed cancellation predicate can clear run context and persist a failed terminal state while a supported fallback continuation is still active.
- `merge-risk: 🚨 message-delivery`: Premature terminal cleanup during fallback can misorder or suppress the fallback response path that should continue delivering chat output.

## Summary

Keep this PR open for fixes: the root bug and proof are credible, but the patch changes deferred lifecycle-error cancellation to `start` only, which can prematurely finalize an active provider fallback path that current tests treat as a continuation.

## What This Changes

The PR stores deferred gateway lifecycle errors as timer/event/options records, changes pending-error cancellation to lifecycle `start` only, and adds regression tests for error-to-tool cleanup and error-to-start retry ordering.

## Best Possible Solution

Keep the structured pending-error state, but use a cancellation predicate that preserves supported lifecycle continuation signals such as fallback while preventing assistant/tool/item events from clearing terminal cleanup.

## Reproduction Assessment

Yes. Current main source shows lifecycle errors schedule deferred cleanup and any later non-error event clears the timer; the PR adds focused tests and pasted real Gateway output for the `error -> tool -> grace` path, though I did not run tests in this read-only checkout.

## Solution Assessment

No. The structured pending-error storage is the right narrow direction, but the start-only cancellation rule should preserve existing lifecycle fallback continuation behavior or be backed by an explicit maintainer contract change.
