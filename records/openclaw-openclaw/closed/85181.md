## Summary

Current main already fixes the reported same-run delivery-mirror takeover path by running matching transcript appends through the embedded attempt's owned session write-lock context and publishing the resulting fingerprint; the fix is main-only and not in v2026.5.20.

## What This Changes

Ensure message-tool outbound delivery-mirror transcript writes with media do not bypass embedded runner session write-lock fencing and fail the same run.

## Best Possible Solution

Keep the current owned transcript write-publication implementation on main and let the next release carry it; the separate cron-trigger discussion can remain under https://github.com/openclaw/openclaw/issues/84583 if cron-specific behavior still needs tracking.

## Reproduction Assessment

Yes at source level for the affected release path: delivery-mirror append changed the session file while the embedded prompt fence was active. I did not establish a failing current-main run because current main now wraps and publishes matching owned transcript writes.

## Solution Assessment

Yes. The main implementation fixes the generic transcript-write ownership boundary instead of special-casing deliver.ts, which is the maintainable solution for same-run message tool mirrors.
