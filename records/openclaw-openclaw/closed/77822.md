## Summary

Keep open: current main and the latest release still contain the cached descriptor-backed tool path implicated by the report, and https://github.com/openclaw/openclaw/pull/78075 is the open closing PR for the fix.

## What This Changes

The issue reports that repeated plugin `register()` materialization can split closure state between hooks and tools, breaking stateful plugins such as `@mem0/openclaw-mem0`.

## Best Possible Solution

Land or supersede the narrow resolver-side fix in https://github.com/openclaw/openclaw/pull/78075 so hook handlers and tool factories use the same live plugin registry closure when a stateful hook+tool plugin is active.

## Reproduction Assessment

Yes, at source level: current main can choose cached descriptor-backed tools before live registry tools, matching the reported split between a live `before_prompt_build` hook and a later tool factory. I did not run a live mem0 gateway reproduction in this read-only review.

## Solution Assessment

Yes, the open resolver-side fix direction is the narrowest maintainable solution because it preserves live hook/tool closure state without adding a broad shared-state SDK contract. This issue should close only after that PR or a superseding fix lands with adequate proof.
