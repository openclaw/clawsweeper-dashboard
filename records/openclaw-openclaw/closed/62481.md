## Summary

Keep open. Current main still lacks a phantom-provider filter, so the linked OpenRouter/google catalog problem is not implemented there; this draft PR is useful source work for https://github.com/openclaw/openclaw/pull/62481, but it needs rebase/rework before merge because the added filter performs a second provider-resolution pass on catalog load and the branch predates newer current-main model-catalog behavior.

## What This Changes

The PR adds a model-catalog post-filter for unconfigured native-provider entries shadowed by aggregator-qualified model IDs, adds OpenRouter regression coverage, refactors the model-catalog test mocks, and adjusts one subagent cleanup race assertion.

## Best Possible Solution

Land one current-main implementation that filters only proven duplicate aggregator-prefix artifacts in the model catalog, preserves explicit/implicit/merged native providers and provider-owned catalog rows, reuses the provider state or generated `models.json` from the existing models-json preparation path, and closes #62317 while superseding duplicate PR attempts.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to stub the registry with `google` + `gemini-3-flash-preview` and `openrouter` + `google/gemini-3-flash-preview`; current main returns both entries and user-facing model browsing/status code groups by each entry's provider.

## Solution Assessment

No, not as-is. Filtering in the model catalog is the right boundary, but this PR should be rebased onto current main and changed to reuse existing provider-resolution/models-json state instead of invoking provider discovery a second time during catalog loading.
