## Summary

Current main and the shipped v2026.4.29 release contain the Windows exec wait-settlement fix that matches the reported foreground exec hang: when a Windows child process exits and stdout/stderr drain but Node never emits `close`, the supervisor now resolves the wait instead of leaving the exec tool without a result. The exact WorkClaw + LM Studio + PDF command was not live-run here, but the current source, regression coverage, and release changelog cover the failure mode that would prevent stdout/stderr from being delivered back to the agent.

## What This Changes

The issue reports that WorkClaw/OpenClaw exec hangs indefinitely on Windows 11 for `uv run python "read_file.py" "some-file.pdf"` through LM Studio instead of returning stdout/stderr within the timeout.

## Best Possible Solution

Keep the shipped supervisor-level Windows wait-settlement behavior as the fix; if the same command still hangs on v2026.4.29 or newer, the right follow-up is a new report with current logs and a minimal reproduction so maintainers can distinguish packaging, provider/tool-call, and PDF extraction issues from the now-fixed child-process wait path.

## Reproduction Assessment

Yes for the matched source-level failure mode: `src/process/supervisor/adapters/child.test.ts` reproduces Windows exit plus drained stdout/stderr without a `close` event and verifies that the wait settles. The exact WorkClaw/LM Studio/PDF environment was not live-reproduced here.

## Solution Assessment

Yes. Fixing the child-process supervisor is the narrow maintainable solution because foreground `exec` already resolves from `run.promise`; ensuring that Windows waits settle restores result delivery for all affected commands without adding LM Studio- or PDF-specific handling.
