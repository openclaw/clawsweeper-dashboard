## Summary

Keep open: current main and the latest release still propagate generic iMessage RPC send timeouts without a one-shot send-rich fallback, and the docs do not cover the stale receive condition where phone-sent messages never reach `chat.db`. The request is actionable, but retrying after an ambiguous send timeout can duplicate messages unless maintainers approve that tradeoff or the fix first proves no row was sent.

## What This Changes

The issue requests an iMessage text-send fallback from long-lived `imsg rpc send` to one-shot `imsg send-rich`, plus troubleshooting docs for stale receive-side Messages delivery.

## Best Possible Solution

Keep this open until a narrow iMessage plugin patch can safely recover text sends from wedged RPC and document receive diagnostics without changing config defaults.

## Reproduction Assessment

Partially: current-main source and tests reproduce the outbound behavior because generic text RPC timeouts are deliberately thrown without CLI fallback. I did not reproduce the live long-lived RPC wedge or APNs receive failure on macOS in this read-only review.

## Solution Assessment

Unclear as written: the direction is plausible, but the one-shot resend is safest if it first rules out an already-sent message or gets maintainer approval for duplicate risk. The receive-side portion is best handled as troubleshooting docs rather than runtime polling or periodic relaunches.
