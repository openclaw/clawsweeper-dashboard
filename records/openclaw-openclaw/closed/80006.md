## Summary

Keep open. The PR targets a real current-main ClawHub hang path, but the proposed timeout races `response.arrayBuffer()` while trying to cancel `Response.body` after that body is locked, so the caller can receive a timeout while the stalled body read remains active.

## What This Changes

This PR adds a ClawHub archive body-read timeout helper and stalled-body regression tests for skill, package, and ClawPack downloads.

## Best Possible Solution

Read ClawHub archive bodies with a reader- or abort-owned timeout shared across skill, package, and ClawPack downloads, with regression coverage using a realistic stalled `ReadableStream`.

## Reproduction Assessment

Yes. Current main clears the fetch timeout before archive body reads and then awaits `response.arrayBuffer()` directly; a Node Web Streams check also confirms the PR's cancellation approach hits a locked body.

## Solution Assessment

No. The direction is right, but the helper should consume through a reader or carry an abort signal through body consumption instead of cancelling `Response.body` after `arrayBuffer()` has locked it.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Abort the active body read:** `src/infra/clawhub.ts:652`
  - body: After `response.arrayBuffer()` starts, the response body is locked; `response.body.cancel(...)` rejects with `TypeError: Invalid state: ReadableStream is locked` and the rejection is ignored. The caller sees the timeout, but the stalled read can keep running, so use a reader-owned read or abort signal that can cancel the in-flight body read.
  - confidence: 0.91
