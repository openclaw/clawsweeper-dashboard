## Summary

Close as superseded. The PR targets #43551, but that linked bug was already closed after maintainer review because current main intentionally supports plugin-owned settings under `plugins.entries.<id>.config`, not arbitrary top-level plugin-entry keys. The proposed passthrough change would make top-level keys validate even though validation, normalization, and runtime delivery still consume nested `config`, so it is not a mergeable fix.

## What This Changes

The PR changes `PluginEntrySchema` from `.strict()` to `.passthrough()` and adds a regression test asserting top-level plugin-entry keys such as `mode`, `userId`, `autoRecall`, and `oss` validate and are preserved.

## Best Possible Solution

Keep the plugin-entry envelope explicit and route plugin-owned settings through `plugins.entries.<id>.config`. If maintainers want compatibility for legacy top-level keys, add a deliberate migration or doctor/fix path with warnings and tests rather than raw passthrough.

That turns a clear schema error into settings that validate but are not applied.
