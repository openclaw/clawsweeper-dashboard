## Label Justifications

- `P1`: The affected older release could silently stop all scheduled heartbeats for an agent until manual intervention.
- `impact:message-loss`: The reported defect suppresses scheduled heartbeat messages and warning delivery.
- `impact:session-state`: Persisted pending-final state survives restarts and previously influenced subsequent heartbeat scheduling.

## Summary

The reported v2026.5.26 failure mode is resolved in v2026.6.5 and current main: pending finals no longer retry indefinitely, scheduled heartbeats defer for at most 30 seconds, heartbeat runs carry the resolved configured destination into delivery context, and session reset/restart recovery clears or replays the persisted state.

## What This Changes

Investigate and fix a persistent heartbeat outage caused by undeliverable pending final deliveries, inconsistent heartbeat target routing, and the lack of an operator recovery command.

## Best Possible Solution

Keep the shipped bounded-defer, unified delivery-context, reset, and restart-recovery behavior; operators on v2026.5.26 should upgrade to v2026.6.5 or newer.

## Reproduction Assessment

No on v2026.6.5/current main: source and regression tests show a 30-second maximum scheduler defer, no active retry-counter loop, and safe handling of pending heartbeat finals.

## Solution Assessment

Yes; bounding scheduler deferral, retaining the resolved delivery context, and using existing reset/restart recovery is narrower and safer than adding another permanent retry loop or a heartbeat-specific CLI state mutation.
