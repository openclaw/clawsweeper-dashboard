## Summary

Current main already fixes the reported Telegram native-command regression: Telegram `auto` native-command resolution is restored, the command-sync hash cache is process-local instead of on-disk, and the fix shipped in v2026.4.15.

## Evidence

- **Shipped release note:** `CHANGELOG.md` under `## 2026.4.15` records two Telegram native-command fixes matching this report: restoring plugin-registry-backed auto defaults for `commands.native` / `commands.nativeSkills`, and keeping the Telegram command-sync cache process-local so restarts re-register commands instead of trusting stale on-disk state.
  - file: [CHANGELOG.md:745](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/CHANGELOG.md#L745)
  - command: `nl -ba CHANGELOG.md | sed -n '740,840p'`
  - sha: [cc0f3e0e4036](https://github.com/openclaw/openclaw/commit/cc0f3e0e40364023cc3a39227a411898ee68ce3b)
- **Auto resolution now uses plugin metadata:** `resolveAutoDefault()` now looks up the channel plugin and returns `plugin.commands.nativeCommandsAutoEnabled` / `nativeSkillsAutoEnabled`, which is the exact path the report said was resolving incorrectly for Telegram `auto`.
  - file: [src/config/commands.ts:6](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/src/config/commands.ts#L6)
  - command: `nl -ba src/config/commands.ts | sed -n '1,220p'`
  - sha: [cc0f3e0e4036](https://github.com/openclaw/openclaw/commit/cc0f3e0e40364023cc3a39227a411898ee68ce3b)
- **Regression test locks Telegram auto=true:** Current tests assert that `resolveNativeCommandsEnabled({ providerId: "telegram", globalSetting: "auto" })` returns `true`, covering the failure mode described by the reporter and commenters.
  - file: [src/config/commands.test.ts:130](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/src/config/commands.test.ts#L130)
  - command: `nl -ba src/config/commands.test.ts | sed -n '120,210p'`
  - sha: [cc0f3e0e4036](https://github.com/openclaw/openclaw/commit/cc0f3e0e40364023cc3a39227a411898ee68ce3b)
- **Telegram startup uses the fixed resolver:** Telegram bot startup computes `nativeEnabled` and `nativeSkillsEnabled` through `resolveNativeCommandsEnabled()` / `resolveNativeSkillsEnabled()` with `providerId: "telegram"`, so the restored auto-default logic is on the live startup path.
  - file: [extensions/telegram/src/bot-core.ts:441](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/extensions/telegram/src/bot-core.ts#L441)
  - command: `nl -ba extensions/telegram/src/bot-core.ts | sed -n '430,490p'`
  - sha: [cc0f3e0e4036](https://github.com/openclaw/openclaw/commit/cc0f3e0e40364023cc3a39227a411898ee68ce3b)
- **Command hash cache is no longer persisted on disk:** `bot-native-command-menu.ts` now keeps `syncedCommandHashes` in a process-local `Map` and reads/writes that in-memory cache, which removes the stale on-disk empty-hash behavior described in the issue.
  - file: [extensions/telegram/src/bot-native-command-menu.ts:214](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/extensions/telegram/src/bot-native-command-menu.ts#L214)
  - command: `nl -ba extensions/telegram/src/bot-native-command-menu.ts | sed -n '140,235p'`
  - sha: [cc0f3e0e4036](https://github.com/openclaw/openclaw/commit/cc0f3e0e40364023cc3a39227a411898ee68ce3b)
- **Empty-menu caching is guarded and explicitly tested:** The sync path only writes the empty-menu hash after a successful `deleteMyCommands()`, and there is a regression test that verifies an empty-menu hash is not cached when `deleteMyCommands` fails.
  - file: [extensions/telegram/src/bot-native-command-menu.ts:267](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/extensions/telegram/src/bot-native-command-menu.ts#L267)

- **User-facing docs now match the fixed behavior:** Current docs say `commands.native: "auto"` enables native commands for Telegram, consistent with the restored runtime behavior on main.
  - file: [docs/channels/telegram.md:308](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/docs/channels/telegram.md#L308)
