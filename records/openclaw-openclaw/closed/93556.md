## Label Justifications

- `P1`: A configured external Weixin channel account can fail before `startAccount`, breaking a real messaging-channel workflow.
- `impact:message-loss`: The startup failure can prevent the Weixin account from handling inbound or outbound channel messages.

## Summary

Keep open: current main and v2026.6.9 still list manifest channel account keys after route-safe normalization but resolve by exact normalized key, so raw Weixin IDs like `59000514e8ad@im.bot` can still miss their configured account before `startAccount`. The open candidate fix at https://github.com/openclaw/openclaw/pull/93686 is not merged yet.

## What This Changes

Fix Gateway/openclaw-weixin startup so raw provider account IDs containing characters such as `@` or `.` resolve to the configured channel account and preserve session routing.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/93686, or an equivalent generic normalized manifest account resolver fix, then close this issue as implemented.

## Reproduction Assessment

Yes. At source level, current main lists manifest accounts with route-safe normalization but resolves the normalized account ID by direct raw-object lookup before `startAccount`.

## Solution Assessment

Yes. The maintainable fix is to align manifest account listing and resolution through the same normalized lookup invariant, not add Weixin-specific routing behavior.
