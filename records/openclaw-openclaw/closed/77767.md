## Summary

Close: current main and v2026.5.4 intentionally keep normal group/channel final replies private unless the agent uses the message tool or `messages.groupChat.visibleReplies` is set to `automatic`; the cron error is a separate explicit-channel requirement for ambiguous outbound jobs.

## What This Changes

The issue requests automatic visible replies back into Discord, Slack, and Telegram group/channel conversations when multiple channels are configured.

## Best Possible Solution

Keep the documented message-tool-only group/channel default; users who want legacy automatic room replies should set `messages.groupChat.visibleReplies: "automatic"` or reopen with delivery logs that show the message tool actually sent and the provider rejected delivery.

## Reproduction Assessment

Yes for the reported symptom by source inspection: group/channel turns default to `message_tool_only`, so typing and model work can occur while normal final delivery is suppressed. I did not live-test external providers.

## Solution Assessment

Yes; the narrow maintainable answer is the existing config/documentation path: use a tool-capable model/message tool for visible room output or set `messages.groupChat.visibleReplies` to `automatic`, and specify a channel for ambiguous cron/outbound jobs.
