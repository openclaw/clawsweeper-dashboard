## Summary

Current `main` already guards heartbeat and main-session cron handling against busy conversation lanes, retries deferred heartbeats, and documents/tests the behavior; the implementation is also present in the shipped `v2026.4.23` release.

## Evidence

- **Heartbeat skips busy main and session lanes:** `runHeartbeatOnce()` returns `requests-in-flight` when either the main command lane is busy or the resolved session lane is busy, with an inline comment stating this avoids interrupting an active streaming turn.
  - file: [src/infra/heartbeat-runner.ts:759](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/src/infra/heartbeat-runner.ts#L759)
  - command: `nl -ba src/infra/heartbeat-runner.ts | sed -n '755,795p'`
  - sha: [9a0b26cafca7](https://github.com/openclaw/openclaw/commit/9a0b26cafca76d267dd231b5eaceadb071ec749d)
- **Wake layer retries deferred heartbeats:** When a heartbeat wake gets `status: "skipped", reason: "requests-in-flight"`, the wake queue re-enqueues the same target and schedules a retry after the default retry delay instead of dropping it.
  - file: [src/infra/heartbeat-wake.ts:177](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/src/infra/heartbeat-wake.ts#L177)
  - command: `nl -ba src/infra/heartbeat-wake.ts | sed -n '165,190p'`
  - sha: [9a0b26cafca7](https://github.com/openclaw/openclaw/commit/9a0b26cafca76d267dd231b5eaceadb071ec749d)
- **Busy-session regression test exists:** Regression coverage asserts that a heartbeat run is skipped with `requests-in-flight` and does not call the reply path when the target session lane already has queued work.
  - file: [src/infra/heartbeat-runner.skips-busy-session-lane.test.ts:64](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/src/infra/heartbeat-runner.skips-busy-session-lane.test.ts#L64)
  - command: `nl -ba src/infra/heartbeat-runner.skips-busy-session-lane.test.ts | sed -n '60,110p'`

- **Main-session cron jobs defer instead of losing delivery:** Main-session cron execution enqueues the system event first, then if `runHeartbeatOnce()` reports `requests-in-flight`, it falls back to `requestHeartbeatNow()` with the same `cron:<job>` reason and `heartbeat.target = "last"`, preserving the scheduled wake for later delivery.
