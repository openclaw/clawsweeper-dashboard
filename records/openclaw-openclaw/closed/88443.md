## Summary

Keep open: current main still treats `auth.cooldowns` changes as restart-required because the gateway reload planner has no `auth` or `auth.cooldowns` rule, so those paths fall through to a full gateway restart. The linked billing-misclassification issue remains useful context, but this issue is a distinct reload-planning bug with a narrow source-backed fix path.

## What This Changes

Change gateway reload behavior so `auth.cooldowns` config changes do not force a full restart that can abort in-flight CLI/channel work.

## Best Possible Solution

Add a narrow reload-planning fix so `auth.cooldowns` changes are handled by an auth-safe hot path or dynamic-read no-op, with regression coverage proving those edits no longer request a full gateway restart.

## Reproduction Assessment

Yes for a source-level reproduction: current main has no reload rule for `auth.cooldowns`, so those changed paths fall through to restart-required behavior in `buildGatewayReloadPlan`. I did not run a live systemd/gateway reproduction in this read-only review.

## Solution Assessment

Yes: hot-reloading or explicitly no-oping `auth.cooldowns` in the reload planner is the narrowest first fix; moving cooldowns out of config entirely is broader and should not block the restart-loss repair.
