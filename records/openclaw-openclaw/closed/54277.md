## Summary

Current `main` does not reproduce the reported allowlist bypass. In Discord guild routing, an unmatched channel under a configured `channels` map resolves to `{ allowed: false }`, not `undefined`, so the downstream allowlist policy blocks it.

## Evidence

- **Unmatched Discord channels resolve as disallowed:** `resolveDiscordChannelConfigWithFallback` returns `resolveChannelMatchConfig(...) ?? { allowed: false }`, so a configured guild channel allowlist produces an explicit disallow object for non-matching channels.
  - file: [extensions/discord/src/monitor/allow-list.ts:476](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/extensions/discord/src/monitor/allow-list.ts#L476)
  - command: `nl -ba extensions/discord/src/monitor/allow-list.ts | sed -n '435,476p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Preflight uses resolved channel config before policy check:** `preflightDiscordMessage` gets `channelConfig` from `resolveDiscordChannelConfigWithFallback(...)` and then computes `channelAllowed` from that resolved object before calling `isDiscordGroupAllowedByPolicy`. With an unmatched configured channel, `channelConfig.allowed` is `false`.
  - file: [extensions/discord/src/monitor/message-handler.preflight.ts:774](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/extensions/discord/src/monitor/message-handler.preflight.ts#L774)
  - command: `nl -ba extensions/discord/src/monitor/message-handler.preflight.ts | sed -n '774,824p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Allowlist policy blocks unmatched routes:** `evaluateGroupRouteAccessForPolicy` returns `allowed: false` with reason `route_not_allowlisted` when `groupPolicy === "allowlist"`, `routeAllowlistConfigured === true`, and `routeMatched === false`.
  - file: [src/plugin-sdk/group-access.ts:80](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/plugin-sdk/group-access.ts#L80)
  - command: `nl -ba src/plugin-sdk/group-access.ts | sed -n '58,92p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Tests cover unmatched Discord channel resolution:** Discord monitor tests assert that unresolved thread/channel lookups under a configured `channels` map produce `allowed === false`.
  - file: [extensions/discord/src/monitor.test.ts:367](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/extensions/discord/src/monitor.test.ts#L367)
  - command: `nl -ba extensions/discord/src/monitor.test.ts | sed -n '367,405p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Tests cover allowlist blocking when channelAllowed is false:** Discord groupPolicy tests assert that `allowlist` blocks when `channelAllowlistConfigured` is true and `channelAllowed` is false.
  - file: [extensions/discord/src/monitor.test.ts:609](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/extensions/discord/src/monitor.test.ts#L609)

## Risks / Open Questions

- `resolveDiscordChannelPolicyCommandAuthorizer` and a few helper paths still use `channelConfig?.allowed !== false`; if a future caller passes `undefined` while also bypassing `resolveDiscordChannelConfigWithFallback`, this could become a real bug. I did not find such a caller on current `main`. 
- `gh` inspection was unavailable in this environment because `GH_TOKEN` was not set, so this review is based on local `main` code and tests only.
