## Summary

Current main and the latest release now route Telegram conversations through stable channel peer session keys, recreate missing Telegram session metadata on the next inbound DM, and carry regression coverage for that recovery path, so the reported 2026.4.9 auth-epoch pairing loss is already fixed.

## What This Changes

The issue reports Telegram messages being silently dropped after a claude-cli auth-epoch reset rotates the underlying CLI session while the chat remains paired to the old session id.

## Best Possible Solution

Keep the shipped stable Telegram routing and session recreation path; users on 2026.4.9 should upgrade to v2026.6.10 or newer.

## Reproduction Assessment

No live current-main failing reproduction was run because the code now appears fixed. Source inspection confirms the reported auth-epoch reset trigger, and current shipped regression coverage exercises Telegram session recreation after stale metadata is removed.

## Solution Assessment

Yes. Stable Telegram peer routing plus create-if-missing session metadata is the narrow maintainable fix for the pairing loss, while current auth-boundary history code correctly avoids replaying raw old credential-context messages across auth-profile or auth-epoch resets.
