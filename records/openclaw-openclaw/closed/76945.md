## Summary

Keep open: current main still has a source-level reproduction for the confusing no-delivery preview, and status still classifies cron session-store rows as ordinary direct/group sessions instead of distinguishing cron history.

## What This Changes

The issue asks for clearer `cron list` and `openclaw status` output for cron jobs configured with `delivery.mode: none`.

## Best Possible Solution

Short-circuit explicit no-delivery previews to clear no-delivery text, then label cron session rows as cron/history in status while preserving existing runtime delivery semantics and session retention.

## Reproduction Assessment

Yes, from source inspection. Explicit `delivery: { mode: "none" }` produces a non-requested plan but does not hit the preview `not requested` branch, and status classification has no cron/history kind.

## Solution Assessment

Yes, a narrow presentation fix is the best path. Keep `resolveCronDeliveryPlan` and message-tool runtime behavior intact, but make preview and status rendering accurately reflect no fallback delivery and retained cron session history.

## Repair Work Prompt

Fix the current-main cron no-delivery presentation bug from https://github.com/openclaw/openclaw/issues/76945. In `src/cron/delivery-preview.ts`, make explicit `delivery: { mode: "none" }` preview as no delivery requested instead of resolving `last` or fail-closed fallback routes, while preserving runtime/message-tool delivery-plan semantics. In the status session summary/rendering path, distinguish agent-scoped cron session-store rows as cron/history rather than ordinary direct/group sessions; do not delete or migrate session history. Add focused regression coverage for the preview and status table behavior, and add a user-facing `CHANGELOG.md` fix entry. Related context: #69285 fixed persisted no-delivery status, and #69587 added delivery previews, so avoid regressing those behaviors.
