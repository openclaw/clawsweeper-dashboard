## Summary

Current main already supports the requested Buzzster-style integration through OpenClaw's existing generic OpenAI-compatible endpoint path, so a dedicated core provider entry is not needed.

## Close Comment

Closing as implemented on main. This Codex review checked current `main` and the shipped `v2026.4.22` release, and OpenClaw already supports third-party OpenAI-compatible endpoints through the existing generic `openai` route rather than requiring a dedicated core provider entry. The current docs show this explicitly for a Claude proxy in `docs/providers/claude-max-api-proxy.md:79`, the custom-provider onboarding path accepts arbitrary OpenAI-compatible base URLs in `src/commands/onboard-custom-config.ts:445`, and runtime handling keeps non-OpenAI hosts on the proxy-style transport in `extensions/openai/openai-provider.ts:82`. On that basis, Buzzster can already be configured through the existing OpenAI-compatible endpoint support, so there is no separate core cleanup item to keep open.
