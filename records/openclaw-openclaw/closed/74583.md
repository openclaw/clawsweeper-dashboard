## Summary

Current main covers the reported local-loopback `openclaw logs --follow` failure: gateway close/timeout errors now fall back to the configured Gateway log file instead of exiting through the “Gateway not reachable” path, with tests and docs for that behavior. The broader streamed/persistent websocket approach remains open in #56475, but the observable bug in this report is implemented on main.

## What This Changes

The issue asks for `openclaw logs --follow` to keep live log monitoring usable after local gateway websocket close or short hiccup instead of exiting repeatedly.

## Best Possible Solution

Keep the current local-loopback fallback on main for the reported monitoring failure, and let #56475 or the broader gateway stability trackers handle any remaining transport-level streaming improvements.

## Reproduction Assessment

Yes. The report gives a concrete command and `gateway closed (1000)` output, and current tests reproduce equivalent close failures by making `logs.tail` reject, then verifying the CLI falls back to the configured log file rather than exiting.

## Solution Assessment

Yes for this report’s implicit local loopback case. The fallback is a narrow maintainable fix for local live monitoring while preserving explicit remote `--url` behavior; the persistent websocket design in #56475 is broader follow-up work rather than required to close this bug.
