## Summary

Keep open. Current main still renders presentation-derived Slack controls and separately supplied interactive controls with independent counters, so duplicate Slack block/action IDs remain possible; the PR's offset approach covers the relevant send surfaces, and I did not find a blocking correctness or security issue in the proposed diff.

## What This Changes

The PR adds offset-aware Slack interactive block rendering, applies it when merging presentation/native and interactive Slack controls, and adds a regression test for mixed presentation plus interactive buttons.

## Best Possible Solution

Land or otherwise apply the narrow Slack ID-offset fix for the message-action and outbound rendering paths, keeping the change inside the Slack plugin and preserving existing generated IDs when controls are not merged.

## Reproduction Assessment

Yes. A Slack send payload with presentation buttons plus separate interactive buttons follows two fresh `buildSlackInteractiveBlocks` calls on current main and can produce duplicate `openclaw_reply_buttons_1` and `openclaw:reply_button:1:1` IDs without needing a live Slack workspace.

## Solution Assessment

Yes. Offsetting generated interactive IDs against already-rendered Slack blocks is a narrow maintainable fix for both affected send paths; a shared render state would also work, but this patch avoids a larger renderer refactor.
