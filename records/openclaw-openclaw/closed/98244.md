## Label Justifications

- `P1`: The reported provider workflow can keep an agent turn waiting roughly two minutes after semantic completion, making OpenAI Responses streaming effectively unusable when reproduced.
- `impact:auth-provider`: The affected surface is OpenAI provider/model transport behavior for Responses API streaming.
- `impact:crash-loop`: The user-visible failure mode is a runtime hang or stall until timeout rather than prompt stream completion.

## Summary

Keep open: current main and the latest release still process `response.completed` without terminating either Responses stream loop, and the linked candidate PR is open with a blocking review finding rather than a merged fix.

## What This Changes

Report that OpenAI Responses streaming can stall for roughly 120 seconds after `response.completed` instead of returning promptly.

## Best Possible Solution

Land a cleanup-aware terminal-event fix for both Responses stream processors, with a stalled-after-`response.completed` regression and either corrected work in https://github.com/openclaw/openclaw/pull/98253 or a narrower replacement PR.

## Reproduction Assessment

Yes at source level: a Responses async iterable that yields `response.completed` and then never resolves would keep current main awaiting the next SDK event. I did not run a credentialed live OpenAI request in this read-only review.

## Solution Assessment

No, not as currently proposed in the linked PR: adding termination controls without wiring them does not solve the stall, and a one-sided transport-only change misses the shared Responses processor. The best fix is terminal-event handling with iterator cleanup and focused regressions across both processors.
