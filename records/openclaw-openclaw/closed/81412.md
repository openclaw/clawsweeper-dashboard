## Summary

Close as duplicate/superseded: exact and trailing cron `NO_REPLY` handling already exists, while the remaining suppress-if-present/config policy is the same silent-token semantics question already tracked in the broader open NO_REPLY thread.

## What This Changes

The issue requests a cron flag or config option that suppresses announce delivery whenever `NO_REPLY` appears anywhere in the final response.

## Best Possible Solution

Keep the remaining suppress-if-present and cron escape-hatch decision in https://github.com/openclaw/openclaw/issues/40165 so OpenClaw gets one consistent silent-token policy instead of a one-off cron regex flag.

## Reproduction Assessment

Yes, source-level. Current tests explicitly deliver non-trailing `NO_REPLY` mentions, and the cron normalizer only suppresses exact or trailing-token forms, so a sentence like `Per contract: NO_REPLY.` remains outside current suppression semantics.

## Solution Assessment

No, not as a separate cleanup item. A cron-specific flag could be one possible product choice, but the maintainable path is to settle the shared NO_REPLY policy in the canonical open thread and then apply it consistently across cron and other delivery paths.
