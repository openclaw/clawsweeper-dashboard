## Summary

Keep this issue open. Current main partially addresses the report by routing `qmd embed` through `memory.qmd.update.embedTimeoutMs` and documenting the update knobs, but the reported ARM/WSL timeout loop is not resolved because OpenClaw still intentionally runs `qmd embed` during boot, periodic maintenance, and forced sync even when `searchMode: "search"` is configured.

## Best Possible Solution

Keep this issue open for a maintainer decision on QMD's BM25-only/no-embed contract. The likely implementation belongs in `extensions/memory-core/src/memory/qmd-manager.ts`, `src/memory-host-sdk/host/backend-config.ts`, config schema/help, QMD docs, and regression tests: add or document an explicit supported way for constrained hosts to disable vector embedding maintenance, decide whether `searchMode: "search"` should imply that behavior, and coordinate the narrower boot/vector-probe aspect with #59234.

## Risks / Open Questions

- Changing `searchMode: "search"` to suppress all embedding work could break users who expect vectors to stay warm for later `vsearch`, `query`, or hybrid retrieval mode changes.
- The current public/config help split is easy to misread: `memory.qmd.limits.timeoutMs` controls search queries, while `memory.qmd.update.embedTimeoutMs` controls `qmd embed`.
- A durable fix may need both an OpenClaw-side no-embed/lexical-only maintenance contract and coordination with upstream QMD/node-llama-cpp behavior on ARM or other no-Vulkan hosts.
