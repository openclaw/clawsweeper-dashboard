## Label Justifications

- `P1`: The report describes a latest-release gateway update path interrupting an active user conversation hours after the requested update.
- `impact:message-loss`: The reported delayed restart/recovery message interrupted an active Discord thread conversation.
- `impact:session-state`: The failure centers on stale restart sentinel and continuation state surviving past the intended restart window.

## Summary

Close: current main already moves control-plane package updates to a managed-service handoff and keeps the update restart sentinel pending until post-restart health verification, which addresses the delayed stale sentinel behavior reported from v2026.5.12.

## What This Changes

The issue requests that `openclaw gateway update.run` either restart the Gateway immediately after an update or cancel stale restart sentinel work so it cannot interrupt conversations hours later.

## Best Possible Solution

Keep the current main managed-service handoff and pending-sentinel behavior, and let it ship in the next release rather than adding another runtime sentinel workaround.

## Reproduction Assessment

No high-confidence current-main failure reproduction is needed because current main has changed the implicated path. The v2026.5.12 source path is source-reproducible, but I did not run a live Linux/systemd update flow in this read-only sweep.

## Solution Assessment

Yes. Moving package-manager `update.run` work outside the live Gateway and delaying sentinel continuation delivery until final health verification is the narrowest maintainable fix for this failure mode.
