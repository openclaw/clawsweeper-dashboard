## Summary

Keep open. Current main still limits the attachment path to media-only explicit chat targets, while direct-handle media and captioned media can still fall through to legacy RPC `send` with a `file` field; the related merged work at https://github.com/openclaw/openclaw/pull/86770 intentionally left those paths unchanged.

## What This Changes

The issue requests routing iMessage direct-handle and captioned media sends through the attachment path instead of legacy RPC `send` with a `file` field.

## Best Possible Solution

Keep this open and fix the bundled iMessage send path so supported media sends use `send-attachment --transport auto`, resolving direct handles to an attachment-capable chat target and sending captions separately when the bridge cannot send rich captioned attachments atomically.

## Reproduction Assessment

Yes, at source level: a direct-handle media send or any captioned media send can reach RPC `send` with `params.file`, while the attachment path is gated to media-only explicit chat targets. I did not run a live iMessage bridge repro in this read-only pass.

## Solution Assessment

Yes: the best fix is a narrow iMessage plugin send-path change that reuses existing target parsing/chat-resolution patterns and adds regression coverage for direct-handle and captioned media cases.
