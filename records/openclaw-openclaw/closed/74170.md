## Summary

Close as superseded: current main still has the optimizer failure path, but the same remaining work is now tracked by the newer canonical PR #74243, which explicitly supersedes this PR and addresses the MIME, cap, guard, and diagnostic boundaries that this branch misses. The proposed patch is not safe to merge as-is because it returns original bytes from the JPEG helper after all resize failures, including guard failures, and the fallback would be mislabeled as JPEG by the caller.

## What This Changes

The PR changes image optimization to return the original buffer when all JPEG resize attempts fail and adds regression coverage for optimizer backend failures.

## Best Possible Solution

Close this branch as superseded and keep the remaining work centered on #74243 and #73424. The accepted fix should live in the active `src/media/web-media.ts` path, fall back only for sendable under-cap originals, preserve the original MIME when returning original bytes, keep pixel/dimension/HEIC guard failures fail-closed, and retain optimizer diagnostics.

## Reproduction Assessment

Yes. Current main has a source-level reproduction path through `loadWebMedia()` into `optimizeImageToJpeg()`, and `src/media/web-media.test.ts` still asserts the `Failed to optimize image` rejection when JPEG optimization cannot produce output.

## Solution Assessment

No. Returning original bytes from `optimizeImageToJpeg()` is too low in the stack because the caller has already committed to JPEG output; the safer solution is the #74243-style caller-level fallback where MIME, byte cap, and guard policy are all available.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep pixel guard failures fail-closed:** `src/web/media.ts:496`
  - body: `resizeToJpeg()` enforces the image pixel limit before resize work. Because this patch catches every resize error and then returns the original bytes, a byte-small but pixel-oversized image that currently rejects could pass through if it is under the byte cap. Distinguish backend optimization failures from guard failures and continue throwing the guard errors.
  - confidence: 0.89
- **[P2] Preserve fallback content type:** `src/web/media.ts:496`
  - body: The caller wraps every `optimizeImageToJpeg()` result as JPEG and returns `image/jpeg`. Returning the original PNG/WebP bytes from this helper will mislabel fallback media, which can break downstream providers or channels. Move the fallback to the caller or return explicit format/content-type metadata.
  - confidence: 0.92
- **[P2] Point the test at the edited media module:** `src/media/web-media.optimize.test.ts:17-21`
  - body: As presented, the production edit is in `src/web/media.ts` while the new regression test imports `./web-media.js` from `src/media`. That does not reliably exercise the changed module on this branch, so the coverage can fail before testing the fallback or cover a different file after the media path move.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff has no supply-chain changes, but it can bypass the existing image pixel-limit resource guard by treating guard failures as optimizer fallback failures.

Concerns:

- **[medium] Image pixel guard can be bypassed:** `src/web/media.ts:496`
  - body: `resizeToJpeg()` rejects images over `MAX_IMAGE_INPUT_PIXELS` before resize work. The PR catches that rejection and returns the original buffer, so a small-byte, huge-pixel image could pass media preprocessing if the byte cap accepts it, weakening the existing DoS/resource guard.
  - confidence: 0.88
