## Label Justifications

- `P2`: This is a valid Linux gateway lifecycle bug with clear source ownership, but it is limited to root-owned systemd user-service installs with stale sudo environment variables.
- `impact:crash-loop`: The report concerns gateway process lifecycle availability: stop can target the wrong user manager and start/status can misreport or fail while the root service is still running.

## Summary

Keep open: current main and v2026.5.18 still route effective-root plus a non-root `SUDO_USER` to the sudo caller's machine-scoped systemd user manager before root's direct `--user` manager, so the reported root-owned user-service lifecycle mismatch remains actionable.

## What This Changes

The issue asks gateway stop/start/status to target root's own systemd user service when an effective-root shell contains stale `SUDO_*` variables.

## Best Possible Solution

Land a focused daemon systemd resolver fix that chooses root's direct `--user` manager when the effective service context is root's user unit, with regression coverage that preserves the existing sudo-to-root and sudo-u paths.

## Reproduction Assessment

Yes, at source level: current main deterministically routes effective uid 0 plus non-root `SUDO_USER` to `--machine <sudo-user>@ --user` before direct root `--user` handling. I did not run a live root/systemd reproduction in this read-only review.

## Solution Assessment

Yes. A focused resolver fix with tests is the best direction; it should not add a config option or broaden into system-scoped unit support, but it must preserve real sudo-to-root and sudo-u behavior.
