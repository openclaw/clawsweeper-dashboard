## Summary

Keep open: current main and the latest release still preserve the effective-root plus non-root `SUDO_USER` path that targets the sudo caller's user manager, while the report is specifically about a root-owned user unit under `/root`. The June 15 follow-up comment adds useful before/after live evidence, and the linked fix PRs for this issue are closed unmerged.

## What This Changes

The issue reports that root-shell gateway lifecycle and status commands can target a stale sudo user's systemd user manager instead of root's own user service.

## Best Possible Solution

Fix the shared systemd user-scope resolution so effective-root with root home/runtime and a root-owned user unit targets root's user manager, while preserving valid `sudo openclaw ...` flows where the sudo user's unit is the intended target.

## Reproduction Assessment

Yes for source reproduction: current `execSystemctlUser` still prefers `SUDO_USER` when effective uid is root, and the adjacent regression test preserves that behavior. I did not establish a fresh live Linux systemd reproduction in this read-only review.

## Solution Assessment

No, closing this issue is not the best way to solve it because current main and the latest release still contain the central stale-sudo user-scope behavior. A narrow resolver fix plus focused regression coverage is the maintainable path.
