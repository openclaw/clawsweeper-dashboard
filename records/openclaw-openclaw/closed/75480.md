## Summary

Close as a duplicate of the generic channel stop-timeout lifecycle bug already tracked by open issue #71412, with open PR #72912 as a current implementation candidate. Current main still shows the same root cause described here, but the Slack trace is evidence for the shared manager-level failure rather than a separate tracker.

## What This Changes

Slack Socket Mode should recover after a health-monitor stop timeout instead of leaving the adapter half-stopped and blocking a replacement adapter.

## Best Possible Solution

Keep one canonical manager-level fix path for #71412/#72912 that can retire or quarantine wedged timed-out channel tasks safely, with regression coverage using this Slack trace as an additional affected-channel example.

## Reproduction Assessment

Yes. The issue includes timestamped live Slack logs, and current main also has a focused unit-test reproduction showing that after the 5s stop timeout a subsequent start does not create a second account task.

## Solution Assessment

Yes for closing this duplicate: the narrow maintainable solution is to fix the shared channel manager lifecycle path once in the canonical tracker, then add Slack-specific cleanup only if late Slack retry callbacks still break the manager-level repair.
