## Label Justifications

- `P1`: The PR targets a real interactive web-chat blocking bug where scheduled cron work can delay human main-session replies.
- `merge-risk: 🚨 compatibility`: The patch intentionally changes where main-session cron work runs, which can affect workflows that relied on main-session transcript reuse.
- `merge-risk: 🚨 session-state`: The remaining startup catch-up mismatch can make task/session history disagree with the actual cron event and heartbeat lane.

## Summary

Keep this PR open: it fixes the main cron lane bug in the right area and has credible terminal proof, but the startup catch-up path can still record a different cron run lane than the one execution uses. The protected maintainer label also means this needs explicit maintainer handling rather than cleanup closure.

## What This Changes

The PR moves main-session cron system events and heartbeat wakes onto cron-owned run session keys, forwards delivery context, and updates cron/gateway tests plus cron documentation.

## Best Possible Solution

Land a version that computes one main cron run lane per execution and uses it consistently for timer, manual, startup catch-up, task records, events, and heartbeat wakes, with the global-scope exception documented or deliberately accepted.

## Reproduction Assessment

Yes, at source level. Current main routes main-session cron events, heartbeat wakes, and task records through the target main session key, and the linked issue includes repeated timing logs showing human web-chat turns queued behind cron on that lane.

## Solution Assessment

No, not quite yet. The PR chooses the right boundary, but startup catch-up still needs the same resolved cron run key as execution, and the docs should make the global-scope exception explicit.

## Repair Work Prompt

Fix PR https://github.com/openclaw/openclaw/pull/82767 by making main-session cron run-lane resolution use one execution timestamp or resolved run key across timer due runs, forced/manual `CronService.run`, and startup catch-up. The observable bug is that startup catch-up can create a task record with `childSessionKey=agent:<agent>:cron:<job>:run:<startedAt>` while `executeMainSessionCronJob` queues/wakes `agent:<agent>:cron:<job>:run:<job.state.runningAtMs>`. Keep isolated/custom cron execution and scheduling semantics unchanged, preserve delivery context forwarding, and keep the intentional `session.scope="global"` behavior explicit. Add focused regression coverage for startup catch-up main cron task childSessionKey matching the queued system event and heartbeat sessionKey; update cron docs to mention the global-scope exception if it remains. The existing changelog entry is sufficient unless the repair changes user-visible scope beyond this PR.
