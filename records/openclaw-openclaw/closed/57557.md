## Summary

Close as implemented on current main. The reported hot path still reads the subagent registry snapshot for session/subagent listing, but the disk loader now caches parsed `subagents/runs.json` by file signature and has regression coverage proving unchanged loads do not reparse the file.

## What This Changes

Avoid blocking gateway RPC hot paths by caching persisted subagent run registry reads instead of synchronously reparsing a growing `subagents/runs.json` on every read.

## Best Possible Solution

Keep the current file-signature cache and write-side invalidation on main, then let the next stable release carry the fix to users; no additional core issue is needed for the central O(n) parse report.

## Reproduction Assessment

Yes. The old release path is reproducible by repeatedly calling the subagent registry read path with an unchanged large `subagents/runs.json`; current main’s focused persistence test verifies the regression by counting `fs.readFileSync` calls.

## Solution Assessment

Yes. The current implementation is the narrow maintainable fix: it preserves cross-process freshness with a cheap file signature check, avoids repeated JSON parsing when unchanged, and keeps existing cleanup/TTL behavior separate.
