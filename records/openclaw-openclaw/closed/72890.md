## Summary

Keep open. Current main still lacks a config-backed way for opaque Anthropic-compatible custom provider model IDs to opt into adaptive thinking, and adding that public capability needs maintainer API/product judgment rather than cleanup closure.

## What This Changes

The issue requests a model/provider config capability that lets custom Anthropic-compatible provider IDs declare adaptive-thinking support.

## Best Possible Solution

Add a typed Anthropic adaptive-thinking opt-in and wire it through config schema, catalog normalization, thinking policy/session validation, Anthropic payload construction, docs, and focused regression tests while preserving existing Claude ID heuristics.

## Reproduction Assessment

Yes. Source inspection is enough: a custom api "anthropic-messages" model with reasoning enabled and an opaque ID routes through the Anthropic transport, but the adaptive branch only runs when supportsAdaptiveThinking(model.id) matches known Claude ID substrings.

## Solution Assessment

Yes in direction, but not as an automatic cleanup fix. A typed opt-in is more maintainable than hardcoding private proxy prefixes or enabling adaptive thinking for every opaque Anthropic-compatible model, but maintainers need to choose the public API surface first.
