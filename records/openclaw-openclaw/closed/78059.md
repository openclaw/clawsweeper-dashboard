## Summary

Keep open: current main still has the Slack thread idle rollover plus timestamp-only thread-context gate, and no merged or closing PR addresses the inbound triggering-message loss; the related open Slack PR covers outbound stream delivery for a separate issue.

## What This Changes

The issue reports that the first Slack thread message after an idle session reset is dropped instead of being delivered to the old or newly created thread session.

## Best Possible Solution

Preserve the inbound Slack thread event through stale session rollover and make Slack thread-history seeding use the same freshness decision as session initialization, with focused regression coverage and live Slack proof if feasible.

## Reproduction Assessment

No high-confidence live reproduction was established in this review. The reporter supplied concrete v2026.5.4 evidence, and current source still shows the stale rollover plus timestamp-only Slack thread-history gate that can be exercised with a targeted regression test.

## Solution Assessment

Yes for the direction: preserving the triggering inbound event and aligning Slack prefetch freshness with initSessionState is narrower than disabling idle resets or changing default reset policy. The existing outbound-stream PR is not the best fix for this inbound loss because it targets generated replies after dispatch.
