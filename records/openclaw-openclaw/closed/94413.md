## What This Changes

The PR replaces the embedded runner’s hardcoded 60s compaction retry aggregate wait with a helper derived from the existing compaction timeout plus a 30s margin, and adds focused helper/regression tests.

## Best Possible Solution

Keep the config-derived timeout-plus-margin approach if maintainers accept the longer wait, then require redacted real behavior proof before landing.

## Reproduction Assessment

Yes, from source plus the linked report: current main and v2026.6.8 use a 60s aggregate wait while the documented compaction timeout can be 180s or higher. I did not rerun a live 200K-token compaction locally.

## Solution Assessment

Yes, the code approach looks like the best inspected fix because it reuses the existing compaction timeout surface and adds a margin so the inner timeout wins. A fixed 300s constant or exact timeout reuse would be less maintainable.
