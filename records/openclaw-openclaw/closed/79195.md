## Summary

Keep open: current main still has timeout helper call sites that can emit bare `fetch-timeout` warnings because `operation` and `url` are only logged when callers provide them.

## What This Changes

The issue requests fetch timeout warnings to consistently include operation and URL context, with future callers prevented from omitting the operation.

## Best Possible Solution

Add operation and available URL context at the remaining in-repo timeout helper call sites, cover the regression with focused fake-timer tests, and enforce future context without an unversioned SDK break.

## Reproduction Assessment

Yes, by source inspection: current main has direct `buildTimeoutAbortSignal` callers that omit `operation` and `url`, while the helper only logs those fields when supplied. A focused fake-timer test can make this an executable reproduction without live network access.

## Solution Assessment

Partly: auditing callers is the right narrow fix, but making `operation` required on the exported SDK helper needs a compatible enforcement strategy. The phase-context idea should only be included if it can reuse existing runtime context without new broad lookup behavior.
