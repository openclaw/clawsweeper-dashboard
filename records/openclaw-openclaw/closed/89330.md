## Label Justifications

- `impact:message-loss`: The reported stale replay ids can turn embedded replies or follow-up/tool-turn conversations into user-visible failed responses.
- `impact:session-state`: The issue is about stale prior Responses output item ids being replayed into later turns when the backend does not persist them.
- `impact:auth-provider`: The failure depends on OpenAI-compatible provider routing and provider compatibility metadata around Responses `store` support.

## Summary

Keep open: current main still separates stripped-store payload policy from Responses item-id replay gating, and the linked open fix PR owns the focused repair.

## What This Changes

The issue asks OpenClaw to stop replaying prior OpenAI Responses item ids when the effective request is non-persistent through `store=false` or provider `supportsStore=false`.

## Best Possible Solution

Review and land the linked focused fix PR if its store-enabled replay preservation still holds, then close this issue as implemented.

## Reproduction Assessment

Yes. Source inspection shows `supportsStore=false` can strip `store` without also disabling replay ids, and the linked PR provides a focused regression path plus redacted backend proof.

## Solution Assessment

Yes. The narrow fix is to make effective non-persistence, including `shouldStripStore`, disable Responses item-id replay while preserving store-enabled OpenAI replay behavior.
