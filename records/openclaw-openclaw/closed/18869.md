## Summary

This is a useful idea, but current main points it toward a ClawHub/community plugin rather than additional OpenClaw core behavior. Core already exposes pre-dispatch and message hooks that can implement turn-taking and pre-LLM gating, supported channels already have bot-visibility controls where the transport allows it, and Telegram group bot-to-bot visibility is a platform limitation.

- A future maintainer could still choose to add a bundled/reference plugin for this pattern, but that would not change the current core-vs-plugin scope decision.

## Close Comment

Closing this as a Codex maintainer review. The idea is valid, but it fits ClawHub/community-plugin scope rather than OpenClaw core. Current main already exposes the right plugin seams for pre-LLM inbound gating and turn-taking (`src/plugins/hook-message.types.ts:1`, `src/plugins/hooks.ts:649`, `src/auto-reply/reply/dispatch-from-config.ts:575`, `src/auto-reply/reply/dispatch-from-config.ts:675`), and supported channels already have transport-level bot-visibility controls where that is possible (`docs/channels/discord.md:1114`, `docs/channels/matrix.md:266`). Telegram group bot-to-bot visibility is a Telegram platform limitation, not a missing core switch (`extensions/telegram/src/bot-handlers.runtime.ts:1976`), and WhatsApp is not a separate bot-user surface (`docs/channels/groups.md:12`). That matches the repo direction in `VISION.md:52`, which keeps optional capability in plugins/ClawHub instead of expanding core.
