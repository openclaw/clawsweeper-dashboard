## Summary

Keep open for maintainer review: the PR is a focused gateway availability fix with a source-supported failure mode and no blocking correctness finding, but it intentionally changes startup/import timing and overlaps with the open restart-token fix in https://github.com/openclaw/openclaw/pull/84334.

## What This Changes

The PR eagerly loads the gateway lifecycle runtime at run-loop startup, catches SIGUSR1 handler rejections to mark the restart token handled, and adds a changelog entry.

## Best Possible Solution

Land or fold the lifecycle-runtime priming fix only after maintainers decide the startup fail-fast tradeoff and sequence it cleanly with the token-reset work in https://github.com/openclaw/openclaw/pull/84334.

## Reproduction Assessment

Yes, at source level: current main and `v2026.5.20` lazily import the lifecycle runtime inside the SIGUSR1 handler before the restart token is guaranteed consumed, and the scheduler coalesces while that token is unconsumed. I did not run the full package-swap repro locally.

## Solution Assessment

Mostly yes: eager-loading the existing lifecycle runtime hub is a narrow way to remove the lazy-import-after-disk-rotation window, and the catch prevents listener rejections from sticking the token. It should still be sequenced with https://github.com/openclaw/openclaw/pull/84334 because that PR handles the primary restartIntent token leak.
