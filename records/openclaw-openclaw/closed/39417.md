## Summary

Current `main` already signs the Control UI device-auth payload with the same effective token it sends in `auth.token`, and the same auth logic is present in shipped release `v2026.4.22`, so the reported mismatch is already implemented/fixed.

## Evidence

- **Control UI signs with the sent auth token:** `buildGatewayConnectAuth` sends `selectedAuth.authToken` as `auth.token`, and `buildGatewayConnectDevice` signs the payload with that same `authToken`, preventing the shared-token vs signed-token mismatch described in the issue.
  - file: [ui/src/ui/gateway.ts:231](https://github.com/openclaw/openclaw/blob/cb2c36b04975e34ab7014a11575d6589aa96ae32/ui/src/ui/gateway.ts#L231)
  - command: `nl -ba ui/src/ui/gateway.ts | sed -n '231,277p'`
  - sha: [cb2c36b04975](https://github.com/openclaw/openclaw/commit/cb2c36b04975e34ab7014a11575d6589aa96ae32)
- **Shared-token and cached-device-token selection is consistent:** `selectConnectAuth` resolves `authToken` from the explicit shared token first, only falls back to the stored device token when no shared token/password is present, and uses `authDeviceToken` only for the bounded retry path.
  - file: [ui/src/ui/gateway.ts:588](https://github.com/openclaw/openclaw/blob/cb2c36b04975e34ab7014a11575d6589aa96ae32/ui/src/ui/gateway.ts#L588)
  - command: `nl -ba ui/src/ui/gateway.ts | sed -n '588,618p'`
  - sha: [cb2c36b04975](https://github.com/openclaw/openclaw/commit/cb2c36b04975e34ab7014a11575d6589aa96ae32)
- **Server verifies against `auth.token` before `auth.deviceToken`:** Gateway signature verification resolves the signature token as `auth.token ?? auth.deviceToken ?? auth.bootstrapToken ?? null`, so the current Control UI retry path remains consistent when it sends both a shared token and a device token.
  - file: [src/gateway/server/ws-connection/handshake-auth-helpers.ts:271](https://github.com/openclaw/openclaw/blob/cb2c36b04975e34ab7014a11575d6589aa96ae32/src/gateway/server/ws-connection/handshake-auth-helpers.ts#L271)
  - command: `nl -ba src/gateway/server/ws-connection/handshake-auth-helpers.ts | sed -n '271,277p'`
  - sha: [cb2c36b04975](https://github.com/openclaw/openclaw/commit/cb2c36b04975e34ab7014a11575d6589aa96ae32)
- **Regression test covers the reported mismatch:** The browser client test asserts that with an explicit shared token and a cached device token, the signed payload contains `shared-auth-token` and does not contain `stored-device-token`.
  - file: [ui/src/ui/gateway.node.test.ts:237](https://github.com/openclaw/openclaw/blob/cb2c36b04975e34ab7014a11575d6589aa96ae32/ui/src/ui/gateway.node.test.ts#L237)
