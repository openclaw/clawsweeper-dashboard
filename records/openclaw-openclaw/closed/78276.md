## Summary

Keep open: current main and the latest shipped tag still use process-local serialization for bootstrap-token state, and the linked open PR remains the active fix path.

## What This Changes

The issue reports a cross-process lost-update race that can revive a consumed setup-code bootstrap token in `devices/bootstrap.json`.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/78277 or an equivalent security-reviewed fix that serializes every bootstrap-token state mutation across processes and proves the race with non-reentrant coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: separate processes can each load `devices/bootstrap.json` and later replace it with stale whole-file state because bootstrap state uses a module-local async lock instead of a cross-process file lock.

## Solution Assessment

Yes on direction. The narrow maintainable solution is to put all bootstrap-token read-modify-write operations behind one cross-process serialization boundary, with regression proof that cannot re-enter the same process lock.
