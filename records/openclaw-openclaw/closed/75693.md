## Summary

Close as a duplicate of the already-open heartbeat busy-retry tracker #40611. Current main does defer heartbeat runs when the main/session lane is busy, but retryable busy skips are still retried by the wake layer after 1 second, which matches this report's repeated 1-5 minute heartbeat interruptions better than a distinct WebChat-only skipWhenBusy failure.

## What This Changes

The issue reports that heartbeat polling with `skipWhenBusy: true` still interrupts active WebChat sessions and fires far more often than the configured 30-minute cadence.

## Best Possible Solution

Keep #40611 as the canonical open tracker and solve the shared heartbeat busy-session retry behavior there, with a longer/configurable retry delay or backoff that preserves the existing no-drift scheduling semantics.

## Reproduction Assessment

Yes for the root behavior: current tests and source reproduce `requests-in-flight` skips being retried after 1 second. A live WSL2 WebChat run was not performed in this read-only review, but the WebChat agent path uses the same main/session lane machinery.

## Solution Assessment

Yes. Closing this as a duplicate is the best cleanup path because the remaining fix is not WebChat-specific; it belongs in the canonical heartbeat busy-retry work tracked by #40611.
