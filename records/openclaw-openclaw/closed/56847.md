## Summary

Keep open. The branch only suppresses the read-start warning for `file` and `filePath`, but current main and the pinned upstream read contract still execute `read` through canonical `path`, so the patch can hide diagnostics for calls that still fail validation; the supplied proof also does not exercise a real OpenClaw run.

## What This Changes

The PR adds `file` and `filePath` to the embedded read-tool start-warning alias list and adds handler tests for those aliases.

## Best Possible Solution

Define one coherent read parameter contract across schema, validation, warning diagnostics, display/fingerprint helpers, docs, and tests; support aliases end to end only if maintainers want that compatibility.

## Reproduction Assessment

Mixed: yes for the narrow handler branch, because source inspection shows `file` or `filePath` without `path` reaches the current warning guard. No for the full successful-read false-positive claim, because current execution validation rejects alias-only reads.

## Solution Assessment

No. Warning-only alias suppression is not the best standalone fix because it can hide diagnostics for reads that execution still rejects; the safer direction is maintainer-approved contract alignment.
