## Summary

Keep open: current main and the latest release still return realtime TwiML before creating the provider bridge, then create and connect the OpenAI/Google realtime bridge only after the carrier media-stream `start` frame. The report is concrete and source-reproducible, but the fix needs maintainer design because pre-answer provider sessions affect quota, cleanup, and default behavior.

## What This Changes

The issue requests prewarming Voice Call realtime provider WebSockets during TwiML webhook handling to reduce first-turn latency for Twilio realtime calls.

## Best Possible Solution

Add a maintainer-approved Voice Call realtime prewarm path in the plugin-owned webhook/realtime handler that starts provider setup during TwiML handling, preserves the current lazy fallback, buffers early media, and closes unused prewarmed sessions on terminal or abandoned calls.

## Reproduction Assessment

Yes for the source-level timing path: current main returns realtime TwiML first, then constructs and connects the provider bridge only after the carrier WebSocket `start` frame. I did not run a live Twilio carrier reproduction in this read-only review.

## Solution Assessment

Unclear as written: prewarming is the right performance direction, but maintainers need to choose default versus opt-in behavior, provider quota exposure, cleanup semantics, and live proof expectations for OpenAI and Google.
