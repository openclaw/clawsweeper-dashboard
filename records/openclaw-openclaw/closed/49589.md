## Summary

The proposed pre-generation classifier is useful, but it fits OpenClaw's documented external-plugin/ClawHub path rather than a bundled core PR. Current main already documents the required plugin hooks (`before_model_resolve`, `before_prompt_build`, `after_tool_call`) and explicitly says optional plugins should be hosted outside the repo and distributed via ClawHub or npm.

## Evidence

- **Repo vision prefers external plugins for optional capability:** `VISION.md` says optional capability should usually ship as plugins, the preferred path is npm plus local extension loading, plugin authors should host and maintain plugins in their own repository, and plugin promotion belongs in ClawHub.
  - file: [VISION.md:54](https://github.com/openclaw/openclaw/blob/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5/VISION.md#L54)

- **User-facing plugin docs say repo inclusion is unnecessary:** `docs/plugins/building-plugins.md` states that plugins extend OpenClaw with new capabilities and do not need to be added to the OpenClaw repository; instead they should be published to ClawHub or npm and installed with `openclaw plugins install ...`.
  - file: [docs/plugins/building-plugins.md:11](https://github.com/openclaw/openclaw/blob/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5/docs/plugins/building-plugins.md#L11)

That matches the PR's stated architecture without needing a bundled core feature.
  - file: [docs/plugins/hooks.md:63](https://github.com/openclaw/openclaw/blob/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5/docs/plugins/hooks.md#L63)
