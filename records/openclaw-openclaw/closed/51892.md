## Summary

Keep this open. Current main still sends iMessage reply metadata to the legacy imsg RPC as `reply_to`, while the requested native-threading field needs maintainer/upstream confirmation because the public imsg send contract I checked does not show send-side `reply_to_guid` support.

## What This Changes

The issue requests native iMessage threaded replies by extracting `replyToId`/`[[reply_to:...]]` metadata and passing it to imsg as `reply_to_guid`.

## Best Possible Solution

Keep the issue open until the imsg send contract is pinned; then either wire OpenClaw to the documented native reply field with focused fake-client coverage, or coordinate upstream imsg support and direct new deployments to BlueBubbles where native reply threading is documented.

## Reproduction Assessment

Yes for the OpenClaw plumbing gap: source inspection shows a `replyToId` send currently produces `reply_to`, not `reply_to_guid`. Native Messages UI threading still needs dependency-contract or live macOS confirmation.

## Solution Assessment

Unclear. Passing `reply_to_guid` is only the best fix if imsg documents and consumes that field for `send`; otherwise the maintainable fix is upstream support or a documented legacy-channel limitation.
