## Summary

Current main already covers the reported gateway stdio MCP leak path: stdio MCP transports close with process-tree termination, gateway reset/delete retires the old session MCP runtime, and shipped idle eviction disposes session runtimes that miss explicit cleanup. The issue was filed against v2026.4.9, while the relevant MCP lifecycle fixes are present in the v2026.4.27 tree and documented in release notes after that version.

## What This Changes

The issue requests cleanup of `vibe-trading-mcp` stdio child processes when gateway sessions end so MCP processes do not accumulate under `openclaw-gateway`.

## Best Possible Solution

Keep the shipped MCP lifecycle design: close stdio transports with process-tree termination, retire runtimes on explicit session reset/delete and run-end paths, and use the idle TTL sweep as the backstop for long-lived gateway sessions. A new issue should only be opened if the reporter can reproduce surviving MCP processes on v2026.4.27 or newer after the configured idle TTL.

## Reproduction Assessment

Yes. The report gives a concrete old-version operational path, OpenClaw 2026.4.9 with Feishu-driven stdio MCP usage and `ps` output showing children under `openclaw-gateway`; I did not run a live Feishu/vibe-trading reproduction because this read-only review verified the corresponding current-main cleanup code and regression tests.

## Solution Assessment

Yes. The current implementation is broader and safer than only adding a hook call: it handles graceful and forced stdio process-tree shutdown, explicit session reset/delete cleanup, run-end cleanup paths, and missed cleanup through TTL-based idle eviction.
