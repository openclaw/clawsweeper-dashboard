## Summary

Close as duplicate: the remaining product/API work is already tracked by open #42039, which covers the same cron restart catch-up behavior and the same need for a public way to prevent irrelevant replay. Current main still has the behavior, so this is not implemented; the conservative cleanup is to keep the older broader tracker open and preserve this report's A-share after-hours example there.

## What This Changes

Add a global, per-cron, or freshness-window policy to disable stale missed-job catch-up when the Gateway restarts.

## Best Possible Solution

Keep #42039 as the canonical open tracker for the missed-run policy and carry this issue's A-share after-hours notification example into that discussion. The eventual fix should add a backward-compatible cron-owned missed-run policy, with config/schema/protocol/docs/tests updated together, rather than splitting the same API decision across duplicate issues.

## Reproduction Assessment

Yes. Current main can reproduce the behavior by persisting a recurring cron job with a past nextRunAtMs and starting CronService; existing restart catch-up tests already assert immediate replay or later deferral rather than stale skipping.

## Solution Assessment

Yes for triage: closing this duplicate is the best cleanup because #42039 is older, open, and tracks the same remaining public cron policy. The implementation itself should be decided there, preferably as an explicit skip or freshness policy rather than only more stagger/deferral knobs.
