## Summary

Keep open: current main and the latest release still have the top-level Skills page fetch `skills.status` without an `agentId`, so it remains scoped to the default agent even though the backend and the Agents page can already load selected-agent skill inventories.

## What This Changes

The issue requests an agent selector for the Web UI Skills page so users can view workspace skills from non-default agents.

## Best Possible Solution

Add an explicit agent scope selector to the top-level Skills page using the existing agents list and `skills.status.agentId`, while preserving the separate `/agents` allowlist controls and making default/all-agent scope unambiguous.

## Reproduction Assessment

Yes. Source inspection shows the top-level Skills page calls `skills.status` with no `agentId`, and the gateway falls back to the default agent when `agentId` is omitted.

## Solution Assessment

Yes, the narrow maintainable path is UI wiring: reuse the existing agent selection and `skills.status.agentId` contract rather than adding config or new backend semantics.
