## Summary

Keep open. Current main has partial post-release mitigations for gateway handshakes and the Anthropic/LM Studio provider startup path, but it does not prove the reported Windows `gateway probe` false-negative is fixed. The probe path still starts `GatewayClient` directly without the event-loop readiness guard proposed by the related open probe-timeout PR.

## What This Changes

The issue requests a fix for a Windows regression where `openclaw gateway probe` times out against an otherwise healthy local gateway on v2026.4.26, with disabled Anthropic/LM Studio provider load failures contributing to startup or handshake stalls.

## Best Possible Solution

Keep this issue open as current-release regression evidence, land or supersede a focused probe/connect fix that covers Windows event-loop starvation and startup/provider stalls, preserve the provider startup/package mitigations already on main, and close only after regression coverage plus Windows manual proof show `gateway probe` no longer false-negatives against a healthy loopback gateway.
