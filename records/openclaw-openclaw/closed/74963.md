## Summary

Keep open. Current main still derives the packaged bundled runtime-deps stage key from the lexical `path.resolve(packageRoot)`, while adjacent loader paths can arrive with realpath-normalized package roots; the linked open fix PR with closing syntax is still unmerged, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue reports OpenClaw 2026.4.27 multi-instance deployments deriving divergent bundled runtime-deps hash directories, leading bundled channel loaders to look for dist chunks in the wrong staged directory.

## Best Possible Solution

Review and land #75048, or an equivalent narrow replacement, so packaged bundled runtime-deps hash a canonical package-root filesystem path and include regression coverage for equivalent package-root views. After that lands, reassess whether disabled or unconfigured bundled channel entrypoints still produce noisy status or health imports.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to expose the same packaged OpenClaw root through a lexical symlink/junction-style path and its canonical realpath, then compare the resolved external bundled runtime-deps install roots; current main still has the lexical hash seam that can split those roots.

## Solution Assessment

Yes. Canonicalizing `packageRoot` before hashing the runtime-deps stage key is the narrowest maintainable fix because it converges the staged directory without changing channel activation policy or broadening hot bundled channel loading.
