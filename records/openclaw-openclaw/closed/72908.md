## Summary

Current main implements the updater lifecycle hardening requested here: package-manager updates now stop a running managed Gateway before replacing the package tree, restart through the updated install, verify the restarted Gateway reports the expected updated version, and fail with diagnostics when the old runtime is still observed. The fix shipped in v2026.4.27.

## What This Changes

The issue reports that `openclaw update --tag 2026.4.25` can replace the global package while leaving the old managed Gateway process running, producing mixed-version dynamic import failures and empty update diagnostics.

## Best Possible Solution

Keep the shipped package-update lifecycle: stop managed services before package replacement, run post-update repair/sync from the updated install, restart through that updated install, and fail with explicit diagnostics when the observed Gateway version or health does not match the completed package update.

## Reproduction Assessment

Yes. The issue provides a concrete 2026.4.24 to 2026.4.25 package-update path, and current tests model the key stale-runtime case by probing a Gateway that still reports the old version after restart.

## Solution Assessment

Yes. The current implementation addresses the narrow lifecycle fault directly rather than adding a compatibility shim for missing hashed chunks: it prevents the old process from running across package replacement and verifies the post-update runtime before success.
