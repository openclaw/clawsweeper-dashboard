## Summary

Keep open. The PR targets a real current-main hot path: the session-store writer still reaches temp-file and parent-directory fsync while the per-store writer slot is held, and v2026.5.3 still contains that behavior; however, the patch is not merge-ready because its durability rationale overclaims transcript fsync, the default-durability test does not actually assert sync calls, and the required changelog entry is missing.

## What This Changes

Adds a default-on `durable` option to atomic JSON/text writes, opts the session-store registry write out of fsync, and adds json-files tests for durable modes.

## Best Possible Solution

Land a narrow version that preserves default durable atomic writes, limits `durable: false` to the session-store registry, documents the actual crash-loss tradeoff, adds direct regression coverage, and records the user-facing gateway fix in the changelog.

## Reproduction Assessment

Yes, from source inspection and related field reports: current main and v2026.5.3 route session-store saves through `writeTextAtomic`, which awaits temp-file and parent-directory fsync while the session-store writer slot is held. I did not run a live heavy-cron reproduction in this read-only pass.

## Solution Assessment

Mostly yes: a default-on internal `durable` option with a session-registry-only opt-out is a narrow maintainable fix for the reported lock-hold path. It should not merge until the durability wording, direct default fsync test, and changelog entry are fixed.
