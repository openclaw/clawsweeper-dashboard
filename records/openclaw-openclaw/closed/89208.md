## Summary

Keep open: current main and the latest release still skip the lane reset when embedded-run abort and drain both succeed, so queued session-lane work can remain unpumped after recovery.

## What This Changes

The issue reports that stuck-session recovery aborts a ghost embedded run but leaves queued Slack user messages stuck in the command lane.

## Best Possible Solution

Use the existing recovery lane reset/drain path whenever a recovered session lane still has queued work, while preserving safeguards that avoid resetting unrelated active lane work.

## Reproduction Assessment

Yes from source inspection: current recovery reads the lane's queued count but skips `resetCommandLane` when `activeSessionId`, `aborted`, and `drained` are all truthy, so queued work is not pumped by the recovery path.

## Solution Assessment

Yes: extending the reset decision to include queued lane work is the narrowest maintainable fix because it uses the existing lane recovery primitive and does not add new policy or configuration.
