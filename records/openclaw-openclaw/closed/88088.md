## Summary

Keep open: this maintainer-labeled PR is a focused MiniMax OAuth security-boundary fix, but the patch would drop the shipped HTTP_PROXY/HTTPS_PROXY behavior for MiniMax OAuth because the new guarded fetches use default strict mode instead of trusted env-proxy mode.

## What This Changes

The PR replaces two MiniMax OAuth raw fetch calls with `fetchWithSsrFGuard`, adds explicit selected-region hostname allowlists, and extracts token response parsing.

## Best Possible Solution

Keep the SSRF guard, selected-region hostname allowlist, and delayed release, but preserve the shipped env-proxy OAuth behavior and add focused coverage before merge.

## Reproduction Assessment

Yes from source inspection: current and shipped MiniMax OAuth initializes env-proxy dispatch before outbound requests, while the PR head guarded calls omit trusted env-proxy mode. I did not run live MiniMax OAuth, but the code-path mismatch is direct.

## Solution Assessment

No. Moving to `fetchWithSsrFGuard` is the right boundary, but this patch is not the best form until it preserves the shipped HTTP_PROXY/HTTPS_PROXY OAuth path and proves that behavior.
