## Summary

Keep open. Current main still routes the reported OpenCode Zen DeepSeek V4 Flash Free case through a generic OpenAI-compatible replay path that can rebuild `reasoning_content` from the prior assistant turn and then strip it before the follow-up request, which matches the reported 400 and is still present in the latest release.

## What This Changes

Preserve DeepSeek V4 Flash Free thinking-mode `reasoning_content` on OpenCode Zen follow-up tool-call requests so sessions do not stall on provider 400s.

## Best Possible Solution

Add narrow provider/model-family handling that preserves or backfills `reasoning_content` for DeepSeek V4 thinking-mode tool-call replay through OpenCode without changing unrelated OpenCode Zen models.

## Reproduction Assessment

Yes at source level: a follow-up OpenAI-completions payload for `opencode/deepseek-v4-flash-free` with a prior assistant `thinkingSignature: "reasoning_content"` will not meet the current preservation checks. I did not establish a live-provider repro in this read-only review.

## Solution Assessment

Yes, a narrow replay-policy or wrapper fix is the maintainable path. The fix should not classify all `opencode-native` traffic as DeepSeek, because Zen also proxies non-DeepSeek models.
