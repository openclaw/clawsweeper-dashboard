## What This Changes

Dashboard `sessions.patch` model switching should preserve a compatible `authProfileOverride` instead of clearing it and causing API auth failure or fallback.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/81886 so server-side `sessions.patch` preserves compatible auth overrides with focused regression coverage.

## Reproduction Assessment

Yes. A model-only Dashboard request reaches `applyModelOverrideToSessionEntry` without profile or preservation data, and the current gateway test asserts that same-provider model patches clear the auth override fields.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the server-side `sessions.patch` model path so Dashboard and SDK callers share the same auth-preservation behavior; the linked PR is the active implementation candidate.
