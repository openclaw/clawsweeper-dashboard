## Summary

Current main now carries the resolved agent scope through the whole `models status --probe` path, and the latest local release tag also contains the fixed agent-directory threading. The prior closed PR remains useful context, but the observable bug is no longer present in the shipped/current code path.

## What This Changes

The issue asks for `openclaw models status --probe --agent <id>` to read the target agent auth store and auth order instead of the default agent store.

## Best Possible Solution

Keep the current scoped probe implementation and regression coverage. Future changes to `models status --probe` should preserve the invariant that the resolved agent directory is used for both probe target planning and probe execution.

## Reproduction Assessment

Yes for the original bug, via the source-level path: dropping `agentDir` from `buildProbeTargets()` makes auth-order filtering read the default store. Current main no longer reproduces it, and the regression test covers the store-selection behavior without requiring live provider credentials.

## Solution Assessment

Yes. Passing the already-resolved `agentId`, `agentDir`, and `workspaceDir` through the probe pipeline is the narrowest maintainable fix because it reuses the existing agent-scoped auth-store contract instead of adding a parallel lookup path.
