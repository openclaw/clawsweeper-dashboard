## Summary

The requested WeChat TTS SILK/native voice-message behavior is owned by the existing external `@tencent-weixin/openclaw-weixin` channel plugin, not by OpenClaw core. Current `main` already keeps WeChat transport/media behavior out of this repo and exposes plugin seams for outbound audio and voice-intent handling.

## Evidence

- **WeChat is external plugin-owned:** The WeChat docs state that OpenClaw connects through the external `@tencent-weixin/openclaw-weixin` plugin, that the WeChat code does not live in the core repo, and that media upload/download and outbound sending are owned by the external plugin.
  - file: [docs/channels/wechat.md:10](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/docs/channels/wechat.md#L10)
  - sha: [59523e66da03](https://github.com/openclaw/openclaw/commit/59523e66da03591bc7bcc183a11f690fa0aae641)
- **Project vision prefers plugin ownership:** VISION says core should stay lean, optional capability should usually ship as plugins, and plugins should be hosted and maintained in their own repositories, with plugin promotion/discovery living in ClawHub.
  - file: [VISION.md:54](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/VISION.md#L54)
  - sha: [59523e66da03](https://github.com/openclaw/openclaw/commit/59523e66da03591bc7bcc183a11f690fa0aae641)
- **Core already exposes outbound audio/voice plugin seams:** The channel outbound contract already passes `mediaUrl` and `audioAsVoice` to plugin outbound adapters, which is the seam a channel plugin would use for native voice-note delivery logic.
  - file: [src/channels/plugins/outbound.types.ts:14](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/src/channels/plugins/outbound.types.ts#L14)
  - sha: [59523e66da03](https://github.com/openclaw/openclaw/commit/59523e66da03591bc7bcc183a11f690fa0aae641)
- **TTS payloads already carry voice intent:** `/tts audio` currently returns a reply payload with `mediaUrl` plus `audioAsVoice` when the synthesized result is voice-compatible, so plugin-owned outbound code can distinguish voice-note intent from plain file sends.
  - file: [src/auto-reply/reply/commands-tts.ts:154](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/src/auto-reply/reply/commands-tts.ts#L154)
  - sha: [59523e66da03](https://github.com/openclaw/openclaw/commit/59523e66da03591bc7bcc183a11f690fa0aae641)
- **Native voice-note support is channel-specific, not generic core behavior:** The shared TTS layer only treats a fixed set of Opus-style channels as native voice-note targets (`telegram`, `feishu`, `whatsapp`, `matrix`, `discord`). WeChat is not in that generic list, and SILK transport would require plugin-specific encoding/send behavior.
  - file: [extensions/speech-core/src/tts.ts:588](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/extensions/speech-core/src/tts.ts#L588)
  - sha: [59523e66da03](https://github.com/openclaw/openclaw/commit/59523e66da03591bc7bcc183a11f690fa0aae641)

## Risks / Open Questions

- The external WeChat plugin may not currently expose a public issue tracker or ClawHub listing for this exact feature, so closing in core could feel dismissive unless the comment clearly redirects it to the plugin layer.
- If maintainers later decide WeChat native voice-note capability should be standardized across plugins, the current plugin-only boundary could change.
