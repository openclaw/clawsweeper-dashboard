Full review comments:

- **[P1] Do not clear pending run state on a timer:** `src/tui/tui.ts:582`
  - body: `checkStaleBusy()` runs from the status timers, so a slow provider start or long `waiting` phase with no events for two minutes clears `pendingOptimisticUserMessage` before a run is claimed. The first chat event then cannot be marked local, which can misclassify the turn and trigger the history-reload paths this TUI code uses for external runs.
  - confidence: 0.88
- **[P1] Do not reload history while the same run is active:** `src/tui/tui.ts:1050`
  - body: This recovery callback calls `loadHistory()` while `state.activeChatRunId` is still the same run. `loadHistory()` clears and rebuilds the chat log, so a sequence gap during a healthy sparse run can replace in-flight output without a terminal event proving the run finished.
  - confidence: 0.86
