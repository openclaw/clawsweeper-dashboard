## Label Justifications

## Summary

Keep open: current main still does not add a doctor preflight guard for CLI-versus-service config path drift, and the issue is already paired with an open unmerged fix PR that uses closing syntax for this report.

## What This Changes

The issue asks `openclaw doctor --fix` to warn or use the active service config path before creating a minimal root config that can mask the real gateway config and disable elevated tools.

## Best Possible Solution

Land or replace the linked PR with a bounded doctor preflight that compares the CLI repair target against the running service or gateway config path and warns before writing a divergent config.

## Reproduction Assessment

Yes, source-reproducible: current main resolves the doctor config path from the CLI process environment and home, while the doctor preflight has no guard that compares this path with the daemon/service config path before repair writes. I did not perform a live root/systemd run in this read-only review.

## Solution Assessment

Yes, the reported direction is the right fix surface, but the safest implementation should use the existing service/gateway config context where possible instead of only hard-coded alternate home paths.
