## Summary

PR #44815 is obsolete on current main. The Feishu parser now replaces raw Feishu media JSON for image, file, audio, video/media, and sticker messages with OpenClaw media placeholders, and the saved media path is still passed separately through the inbound media payload. This covers the image-path mismatch from #44792 and the broader file/audio/video/sticker concern raised in the PR discussion.

## Best Possible Solution

Close #44815 as implemented on main and keep the broader shipped parser/media-payload behavior. If follow-up is needed, it should be a narrower regression against the current `<media:...>` placeholder flow, not this older one-line `[Image]` PR.

## Evidence

- **Current parser covers the PR fix and broader media types:** `parseMessageContent` now handles `image`, `file`, `audio`, `video`, `media`, and `sticker` message types by returning `inferPlaceholder(messageType)` plus an optional filename instead of returning raw Feishu JSON with `image_key` or `file_key`. For images this is `<media:image>`, which solves the same user-visible problem as the PR's proposed `[Image]` placeholder.
