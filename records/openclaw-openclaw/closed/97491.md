## What This Changes

The PR changes model row normalization so loopback/local-baseUrl rows report `available: true` even when registry availability lacks the exact model key, and adds a regression test.

## Best Possible Solution

Land the narrow row-level fix if maintainers accept the documented provider-level Auth semantics; use a separate live-readiness field or probe if exact endpoint/model execution status is desired.

## Reproduction Assessment

Yes, at source level: current main computes `local` but still derives `available` from registry availability or provider-auth fallback, so a local row omitted from `availableKeys` can render unavailable. The PR body also supplies after-fix CLI output against a real Dockerized Ollama daemon.

## Solution Assessment

Yes, if maintainers accept provider-level local-marker availability: changing the shared row normalization helper is the narrowest fix for configured, catalog, and discovered model-list rows. A live readiness probe would be a separate product/API change, not a smaller repair for this display bug.
