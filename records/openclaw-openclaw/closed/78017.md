## Summary

Keep open: current main and latest release still pin `@earendil-works/pi-tui` 0.74.0, whose render path clears scrollback with `\x1b[3J` when `firstChanged < prevViewportTop`; OpenClaw still streams assistant text into that markdown path. The upstream fix PR is closed unmerged and npm has no newer pi-tui release, so the remaining work needs maintainer/upstream dependency handling rather than cleanup closure.

## What This Changes

The issue reports that long `openclaw tui` assistant messages disappear from terminal scrollback because pi-tui content-driven full redraws emit `\x1b[3J`.

## Best Possible Solution

Land or approve a pi-tui fix that preserves scrollback for content-driven full redraws, then update OpenClaw to that release or approved patch with live TUI proof.

## Reproduction Assessment

Yes, source-reproducible with high confidence: current main pins pi-tui 0.74.0, that tarball clears scrollback on `firstChanged < prevViewportTop`, and OpenClaw streams assistant updates through `ChatLog.updateAssistant` into pi-tui Markdown. I did not run a live terminal recording in this read-only pass.

## Solution Assessment

Yes, the narrow maintainable solution is to fix pi-tui's redraw behavior and then consume that approved dependency in OpenClaw. An OpenClaw-only markdown workaround would be more fragile because the destructive escape sequence is emitted by the renderer branch itself.
