## Summary

Close: the passwordless Microsoft Teams auth problem is superseded by the merged and shipped federated-auth implementation, while this branch targets the old `@microsoft/agents-hosting` path and still has correctness blockers for the proposed DAC mode.

## What This Changes

The PR adds a `defaultAzureCredential` MS Teams auth type, wires a DAC token provider through old Teams token call sites, extends config/schema/tests, resets setup auth fields, and adds `@azure/identity` to the plugin dependency set.

## Best Possible Solution

Keep the shipped `authType: "federated"` certificate and managed-identity implementation as the supported passwordless Teams path; any future DAC-style convenience should be a fresh, current-main design rather than this stale branch.

## Reproduction Assessment

Not applicable. This is a feature PR, and the relevant check is source/history comparison: current main already supports the enterprise passwordless Teams auth path through `authType: "federated"`.

## Solution Assessment

No. The branch is no longer the best solution because current main has a different Teams SDK architecture and a shipped federated-auth design, while the PR still carries DAC identity-pinning and env-backed setup issues.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Stop pinning DAC identity IDs to the bot app id:** `extensions/msteams/src/sdk.ts:35-38`
  - body: `DefaultAzureCredential` treats these options as the user-assigned managed/workload identity client IDs. Pinning both to the Teams bot app id breaks valid deployments where the app registration and managed identity are separate, which the shipped current-main config supports via a separate `managedIdentityClientId`.
  - confidence: 0.88
- **[P2] Mirror env fallbacks in the configured check:** `extensions/msteams/src/token.ts:69-71`
  - body: `resolveMSTeamsCredentials` accepts `MSTEAMS_APP_ID` and `MSTEAMS_TENANT_ID`, but `hasConfiguredMSTeamsCredentials` only checks the config object. Env-only passwordless setups can therefore look unconfigured and fall into the setup path that prompts for client-secret credentials.
  - confidence: 0.84
