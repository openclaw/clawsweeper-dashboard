## Summary

Keep open: this PR addresses a valid exec boundary bug, but the current patch still leaves a concrete direct-script bypass and the PR is dirty against the base branch.

## What This Changes

The PR adds exec preflight parsing, tests, docs, and a changelog entry to reject OpenClaw message delivery commands from agent `exec` while preserving non-delivery message commands.

## Best Possible Solution

Keep the exec preflight boundary, inspect direct path-like shell script targets regardless of arbitrary dotted names or by shebang, add focused regression coverage, rebase, and rerun targeted exec checks with real behavior proof.

## Reproduction Assessment

Yes. Source inspection shows current main lets `openclaw message send` reach the message CLI delivery path from `exec`, and PR head still lets a direct `./send.v1` shell script skip shell-file inspection.

## Solution Assessment

No. The exec preflight boundary is the right approach, but this implementation remains incomplete until dotted path-like direct shell scripts are inspected before execution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Inspect dotted direct shell scripts:** `src/agents/bash-tools.exec.ts:543-546`
  - body: The direct-script guard only inspects commands with a known shell suffix or path-like names with no dot. An executable `./send.v1` with a shell shebang skips `validateScriptFileForShellBleed`, so an `openclaw message send` inside can still reach exec; treat path-like direct script targets as inspectable regardless of suffix or check the shebang first.
  - confidence: 0.88
