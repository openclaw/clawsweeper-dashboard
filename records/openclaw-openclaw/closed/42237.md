## Summary

Current `main` does not match the reported routing behavior: with `session.dmScope: "main"`, Telegram default-account DMs and WebChat both resolve to the agent main session, and Telegram topic IDs are handled as thread suffixes rather than as a cross-channel merge key.

## Close Comment

Closing from a Codex review because I can’t verify this on current `main` (`7d9172d5e093435d37ed3e00e476a4eaf84a252b`). On current OpenClaw, `session.dmScope: "main"` still means direct messages share the agent main session (`docs/channels/channel-routing.md:26`, `src/routing/session-key.ts:143`), WebChat defaults to that main session (`docs/channels/channel-routing.md:132`), and the Telegram DM test on `main` expects `agent:main:main` for a default-account private chat (`extensions/telegram/src/bot-message-context.named-account-dm.test-support.ts:143`). Topic IDs are only used for Telegram thread suffixing (`extensions/telegram/src/session-route.test.ts:5`), not as a cross-channel merge key; if you want the same person’s DMs across channels to collapse intentionally, the documented mechanism is `session.identityLinks` (`docs/gateway/security/index.md:536`). Because the reported `agent:main:telegram:direct:...` result does not match current main for the provided `dmScope: main` setup, I’m closing this as not reproducible on current OpenClaw. If this still reproduces on `v2026.4.22` or newer, please open a fresh issue with the exact Telegram account setup, whether the Telegram side is a DM topic, and the `sessions --json` output for both turns.
