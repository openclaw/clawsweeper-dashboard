## Summary

Current main and the latest release already address the central silent Telegram delivery-loss problem: dispatcher-side delivery failures are counted, logged through the Telegram onError path, and trigger a visible fallback reply for non-room events when no reply was delivered.

## What This Changes

The issue requests that Telegram delivery/enqueue failures stop being silently swallowed and produce visible user feedback for retryable or permanent send failures.

## Best Possible Solution

Keep the shipped generic visible-fallback and retry behavior; any class-specific Telegram error wording should be a narrower enhancement, not this silent-loss bug.

## Reproduction Assessment

No failing current-main reproduction remains from source inspection. The historical failure mode is source-reproducible, but current main records dispatcher failedCounts/non-silent failures and sends a visible fallback for direct non-room events.

## Solution Assessment

Yes for the central bug. The current implementation fixes the silent-loss path with shared dispatcher failed-count accounting plus Telegram-owned fallback delivery, which is narrower than adding a new class-specific error subsystem for this report.
