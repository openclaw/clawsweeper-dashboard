## What This Changes

The issue asks OpenClaw to propagate `before_tool_call` hook blocks to Codex as tool errors so sub-agent or cron sessions do not remain in `processing`.

## Best Possible Solution

Keep the current main implementation and let the next release carry the main-only fix; no plugin-author janitor workaround should be needed for this path.

## Reproduction Assessment

No. I did not run a live gateway scenario in this read-only review, but current main source and regression tests show blocked Codex dynamic tools return a failed tool response instead of leaving the session waiting.

## Solution Assessment

Yes. The current implementation is the narrow maintainable fix: preserve terminal hook-block semantics, synthesize a blocked tool result with the plugin reason, and return an error response through the Codex bridge.
