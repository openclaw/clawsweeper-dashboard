## Summary

Keep this issue open. Current main still rejects provider-only `models.providers.openai` entries before the memory embedding runtime can use the configured `baseUrl`, and open PR #39766 already targets this issue with closing syntax.

## What This Changes

The issue requests that memory embeddings use a custom `models.providers.openai.baseUrl` even when that provider entry omits a `models` array.

## Best Possible Solution

Review and land PR #39766 or an equivalent focused fix that defaults provider `models` to `[]` across Zod, generated schema, TypeScript types, and regression tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: the reported config omits `models`, current main requires that field, and validation rejects it before the embedding runtime can use `models.providers.openai.baseUrl`.

## Solution Assessment

Yes. Defaulting `models` at the config boundary is the narrow maintainable fix because downstream code can continue receiving an array while credential/baseUrl-only provider entries become valid.
