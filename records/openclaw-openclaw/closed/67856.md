## Summary

Closing as cannot_reproduce. Current `main` and the latest release already propagate configured dreaming timezone into the managed cron job, missing `tz` falls back to the gateway host timezone rather than UTC, and the issue’s sample `runAtMs` value converts to `2026-04-17 03:00:00` in `Asia/Shanghai`, not 11:00.

## Evidence

- **Dreaming config resolves timezone from plugin config or userTimezone fallback:** `resolveMemoryDreamingConfig()` reads `dreaming.timezone` first, then falls back to `cfg.agents.defaults.userTimezone`, so managed dreaming does have a timezone source when either is configured.
  - file: [src/memory-host-sdk/dreaming.ts:355](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/src/memory-host-sdk/dreaming.ts#L355)
  - command: `nl -ba src/memory-host-sdk/dreaming.ts | sed -n '348,360p'`
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **Managed dreaming cron writes `schedule.tz` when timezone is resolved:** `buildManagedDreamingCronJob()` includes `...(config.timezone ? { tz: config.timezone } : {})` in the persisted cron schedule.
  - file: [extensions/memory-core/src/dreaming.ts:160](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/extensions/memory-core/src/dreaming.ts#L160)

## Risks / Open Questions

- External API consumers cannot infer the gateway host timezone from an omitted `schedule.tz`; if maintainers want that surfaced explicitly, that is a separate API/design request rather than evidence of incorrect scheduling.
