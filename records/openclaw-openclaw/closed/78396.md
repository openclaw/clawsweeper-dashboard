## Summary

Keep open: current main and the latest release still contain the source-level Matrix forced-reset repair path that can recreate secret storage and immediately call setup-new cross-signing again; the related Matrix PR is still open and is not a merged fix.

## What This Changes

The issue reports that `openclaw matrix verify bootstrap --force-reset-cross-signing` can run a second destructive Matrix cross-signing reset after stale secret-storage repair, leaving recovery material and E2EE state inconsistent.

## Best Possible Solution

Patch the Matrix bootstrapper so one forced bootstrap can perform at most one destructive cross-signing reset while keeping recovery-key staging, server secret storage, room-key backup creation, and UIA no-partial-mutation guards coherent.

## Reproduction Assessment

Yes, at source level: current main skips initial secret-storage bootstrap during forced reset, allows repair-time SSSS recreation after a repairable failure, and then calls setup-new cross-signing again. I did not establish a live homeserver reproduction in this read-only review.

## Solution Assessment

Unclear whether the exact two-line patch in the issue is the safest final shape. The maintainable fix should prevent the repeated destructive reset while preserving UIA handling, recovery-key staging, and backup consistency.
