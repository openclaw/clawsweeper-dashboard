## Summary

Keep open. The socket-teardown direction is plausible, but this PR still introduces a blocking retry runId correctness regression and the supplied logs are not sufficient real behavior proof of the no-zombie/fallback path.

## What This Changes

The PR adds abort-triggered dispatcher destruction to guarded fetch, exports dispatcher destruction, changes embedded retry run IDs, and adds a gateway client-disconnect test.

## Best Possible Solution

Keep the fix at the guarded-fetch dispatcher teardown boundary, remove the retry runId mutation, add focused dispatcher-destruction or fallback-survival coverage, and require stronger redacted runtime proof before merge.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only review. Source inspection and the supplied before logs make the reported zombie-socket/fallback failure plausible because current main uses graceful dispatcher close while Undici distinguishes graceful close from abrupt destroy.

## Solution Assessment

No. The guarded-fetch teardown boundary is the narrow maintainable direction, but changing retry runId is not part of that fix and the added test should prove dispatcher teardown or fallback survival rather than another client-disconnect abort signal.
