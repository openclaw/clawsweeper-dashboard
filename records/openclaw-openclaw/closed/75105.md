## Summary

Keep open. Current main fixed known DeepSeek V4 proxy model IDs and already accepts `compat.thinkingFormat`, but the generic request wrapper still does not honor a model-level DeepSeek compat opt-in for arbitrary custom model IDs.

## What This Changes

Allow a model-level compatibility setting to trigger DeepSeek-style `reasoning_content` replay handling for custom OpenAI-compatible DeepSeek/proxy models.

## Best Possible Solution

Reuse the existing `compat.thinkingFormat: "deepseek"` model-level contract to trigger the generic DeepSeek V4 replay wrapper, with focused tests and docs, instead of adding `reasoningFormat` or another overlapping key.

## Reproduction Assessment

Yes. A high-confidence source path is a custom `openai-completions` model with `compat.thinkingFormat: "deepseek"`, thinking enabled, replayed assistant history missing `reasoning_content`, and a model id whose final segment is not `deepseek-v4-flash` or `deepseek-v4-pro`.

## Solution Assessment

Yes. The narrow maintainable fix is to consume the existing model compat key at the wrapper-install predicate and keep the current exact model-id fallback; adding a new public config key would duplicate the existing contract.
