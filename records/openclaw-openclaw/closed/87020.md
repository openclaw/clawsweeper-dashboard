## Summary

Keep open: current main and the latest release still have a real source-visible gap where the gateway reports a `config.openFile` failure as a successful RPC payload and the Control UI does not inspect that payload, so headless users can get no actionable feedback.

## What This Changes

The issue asks the Control UI to show an actionable message, including the config path, when the config-file Open action cannot launch an external opener on a headless VPS.

## Best Possible Solution

Fix the existing Control UI config-open flow so a resolved `{ ok: false, path, error }` response becomes an inline or toast error with the file path and headless/external-opener guidance, while preserving successful desktop opener behavior.

## Reproduction Assessment

Yes, source-reproducible: mock or trigger `config.openFile` returning `{ ok: false, path, error }`; the browser client resolves the request and `openConfigFile` does not surface the payload. I did not run a live headless VPS session in this read-only review.

## Solution Assessment

Yes, the narrow maintainable solution is to handle the existing failure payload in the Control UI and add focused regression coverage; changing or removing Linux `xdg-open` is not needed.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/87020. The Control UI config Open action currently awaits `config.openFile` but only handles thrown RPC failures; the gateway returns a resolved payload `{ ok: false, path, error }` when the external opener fails, so headless Linux/VPS users get no actionable UI feedback. Preserve successful desktop opener behavior and do not remove Linux `xdg-open`. Add a small UI state/message path that surfaces the config path and clear external-opener/headless guidance when the payload is `ok: false`; avoid leaking raw shell stderr into the UI unless deliberately sanitized. Add focused controller and/or render tests for the failure payload and keep gateway tests aligned. Do not edit CHANGELOG.md. Follow `ui/AGENTS.md`: if adding translatable copy through locale files, update English source and generated locale output through the normal generator rather than hand-editing non-English bundles.
