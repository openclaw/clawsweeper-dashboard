## Label Justifications

- `P1`: The issue describes a broken subagent orchestration workflow where parent sessions remain parked and recovery requires a gateway restart.
- `impact:session-state`: The reported failure leaves parent and child run state stuck, with active subagent records not settling after the declared timeout.
- `impact:message-loss`: The parent session does not receive the timeout/completion event needed for `sessions_yield` to resume.

## Summary

Keep open: current main still treats a non-terminal `agent.wait` timeout as a reason to keep the subagent active and retry, while the docs say `runTimeoutSeconds` aborts the sub-agent run after the configured deadline.

## What This Changes

The issue requests gateway-enforced `sessions_spawn.runTimeoutSeconds` so timed-out child runs fail, notify the parent waiting via `sessions_yield`, and stop counting as active.

## Best Possible Solution

Make `runTimeoutSeconds` a hard parent-visible deadline: after the deadline, synthesize a timeout completion, abort or detach the child run safely, run normal subagent cleanup/delivery, and keep transient wait retries only before that deadline.

## Reproduction Assessment

Yes for a source-level reproduction: current code and tests show a non-terminal `agent.wait` timeout keeps the subagent active instead of producing the documented timeout completion. I did not run a live gateway hang scenario in this read-only pass.

## Solution Assessment

Yes, enforcing the existing `runTimeoutSeconds` contract in the gateway/subagent completion path is the maintainable direction. The repair should distinguish transient wait timeouts from the explicit run deadline rather than removing the retry grace wholesale.
