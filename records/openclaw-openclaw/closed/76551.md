## Summary

Keep open; current main still has a slot-blind stale-plugin repair path that can remove `plugins.allow` and `plugins.entries` while leaving an explicit `plugins.slots.memory` reference that config validation treats as fatal. The open sibling #76550 covers the same recovery gap from the slot-cleanup side, so both should be fixed together rather than closed pre-merge.

## What This Changes

The issue asks `openclaw doctor --fix` to handle stale plugin cleanup atomically when `plugins.slots.*` still references the plugin being removed from `plugins.allow` or `plugins.entries`.

## Best Possible Solution

Make stale plugin repair slot-aware and atomic so the post-repair config either keeps the slot-selected plugin consistently configured or removes the stale slot reference together with the stale allow/entry refs.

## Reproduction Assessment

Yes, source-reproducible. A config with a missing plugin in `plugins.allow`, `plugins.entries`, and explicit `plugins.slots.memory` follows code that removes allow/entry refs but leaves the slot, after which validation reports `plugins.slots.memory: plugin not found`.

## Solution Assessment

Yes, the requested repair boundary is appropriate. The narrow maintainable fix is to extend the existing stale-plugin doctor repair to handle slot references consistently, not to add a new config mode or change plugin installation mechanics.
