## Summary

Current main already fixes the reported Brave provider boundary: agent-managed `web_search` now late-binds runtime web-search metadata/config and can resolve runtime-only providers, while config validation accepts installed plugin-manifest provider ids. The proof commit is on `main` but is not contained in a shipped tag in this checkout.

## What This Changes

Brave web search users on 2026.5.2 reported that update/doctor could leave Brave installed and capability search working, while agent-managed `web_search` failed to resolve the provider.

## Best Possible Solution

Keep the current main runtime-provider resolution and let the next release carry it to users on 2026.5.2.

## Reproduction Assessment

No current-main failing reproduction remains from source inspection: the agent tool now resolves the active runtime-selected provider path that the reporter showed was missing in 2026.5.2. I did not run the macOS/npm live gateway scenario, but the relevant source and regression tests cover the reported boundary.

## Solution Assessment

Yes. Resolving agent `web_search` through the same active runtime provider surface used by capability search is the narrow maintainable fix, and it avoids hard-coding Brave-specific behavior in core.
