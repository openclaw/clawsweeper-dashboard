## Summary

## What This Changes

The issue requests aligning `wiki_apply` operation values with documented wiki apply flows, either by accepting CLI-style aliases or by documenting the internal tool op names.

## Best Possible Solution

Make `wiki_apply` explicitly accept and normalize `synthesis` and `metadata` to the canonical mutation ops, using a flat string enum schema and focused docs/tests so agents and CLI guidance agree.

## Repair Work Prompt

Fix the memory-wiki `wiki_apply` op mismatch reported in https://github.com/openclaw/openclaw/issues/90303. Current main exposes only `create_synthesis` and `update_metadata` in `extensions/memory-wiki/src/tool.ts`, while CLI/docs use `synthesis` and `metadata`; a model/tool call with the CLI-style op fails schema validation before execution. Keep the fix plugin-owned. Prefer importing/reusing the plugin-sdk flat `stringEnum` helper instead of `Type.Union([Type.Literal(...)])`, explicitly list accepted ops, normalize `synthesis -> create_synthesis` and `metadata -> update_metadata` inside `normalizeMemoryWikiMutationInput`, and keep returned mutation/result operation values canonical. Add focused tests for the tool schema and normalizer/gateway behavior accepting aliases; leave existing canonical ops and CLI behavior working. Update the relevant memory-wiki docs only if needed to make tool op values/aliases clear. Do not add core tool-validation fallback, config options, broad compatibility shims, or CHANGELOG edits.
