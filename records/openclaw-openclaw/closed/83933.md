## Summary

This PR addresses a real current-main cron state-loss path and is not obsolete. The patch centralizes the manual-run distinction in `applyJobResult`, and I found no discrete blocking code defect, but the merge intentionally changes manual `cron run` semantics and needs maintainer acceptance of that compatibility tradeoff.

## What This Changes

The PR adds an `isManual` option through manual cron completion paths so manual runs preserve `deleteAfterRun` jobs and leave scheduled retry, alert, and backoff counters untouched.

## Best Possible Solution

Land the central `isManual` boundary if maintainers accept that manual cron runs are verification/debug runs that must not consume scheduled lifecycle state.

## Reproduction Assessment

Yes, from source inspection: current `finishPreparedManualRun` reaches `applyJobResult` without a manual discriminator, and `applyJobResult` deletes successful `deleteAfterRun` at-jobs and mutates counters unconditionally. I did not run the failing test path in this read-only review.

## Solution Assessment

Yes, the proposed central discriminator is the narrow maintainable fix because scheduled runs continue through the existing default path while only manual callers opt into preserved scheduler state. The remaining decision is policy/compatibility, not a concrete code defect I can point to in the diff.
