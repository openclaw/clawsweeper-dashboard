## Summary

Current main already fixes the reported runtime-deps retention problem: bundled runtime-deps selection now treats the normalized memory slot as active even when `memory-core` is absent from `plugins.entries`, and regression coverage expects `memory-core` with `config: {}` to stage `chokidar@^5.0.0`. The proof commit is on main only and is not contained in the latest shipped tag.

## What This Changes

The issue asks for bundled runtime-dependency staging to retain dependencies from auto-loaded bundled plugins such as `memory-core` when later plugins refresh the shared runtime-deps install root.

## Best Possible Solution

Keep the current main behavior: retain runtime deps through the policy-aware bundled plugin selector rather than scanning every bundled plugin, then ship this main-only fix in the next release.

## Reproduction Assessment

Yes. The high-confidence reproduction path is the packaged shared runtime-deps ensure flow with `memory-core` absent from `plugins.entries` and `config: {}`; current source and regression coverage show that path now includes `chokidar@^5.0.0`. I did not execute tests because this review was explicitly read-only.

## Solution Assessment

Yes. The current main approach is narrower than the suggested scan-all change because it keeps inactive or disabled bundled plugin deps out of the plan while still including the default/selected memory slot.
