## Summary

Current main already solves the observable continuity problem through explicit persistent cron session targets: `sessionTarget: "current"` and `session:<id>` let recurring agent-turn cron jobs bind to a stable session and reuse it until the normal reset policy expires. The proposed `reuseSession` flag is not present on main, but the central user workflow is implemented without overloading the documented fresh-by-default `isolated` semantics.

## Best Possible Solution

Close this PR and keep the shipped explicit session-target model: use `sessionTarget: "current"` or `session:<id>` / `openclaw cron add --session session:<id>` for recurring cron context continuity, while preserving literal `isolated` jobs as fresh-by-default. A future proposal for a convenience flag should be opened as a focused API/product discussion if maintainers still want that extra surface.

## Risks / Open Questions

- Closing this leaves out the proposed convenience flag (`--reuse-session`) and requires users to choose `--session session:<id>` or create from `current` when they want recurring context continuity.
- The PR branch includes useful bot-review discussion and patch-safety work, but its exact API addition would need renewed maintainer product judgment if someone still wants a convenience alias for persistent cron sessions.
