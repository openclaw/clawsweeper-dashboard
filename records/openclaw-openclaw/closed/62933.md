## Summary

Current `main` already contains the Control UI-side normalization and regression coverage needed for provider-qualified custom-provider model switches. The gateway patch path accepts qualified nested model refs, and the Control UI now preserves/qualifies those refs instead of tripping the stale-provider path that matched the reported `model not allowed` error.

## Evidence

- **Control UI model normalization now handles stale providers and nested ids:** `resolvePreferredServerChatModelValue(...)` prefers catalog-backed provider matches for plain ids, preserves slash-containing nested ids, and avoids stale-provider double-prefixing such as `zai/openai/gpt-5-mini`. That directly addresses the Control UI-side qualification failure described in the issue.
