## Summary

Keep this PR open. Current main does not implement transparent session transcript/session-store encryption, and the PR is not merge-ready because the diff is security-sensitive, incomplete at the central persistence boundary, and introduces static ESM/sync API blockers plus plaintext fallback/backup risks.

## What This Changes

The PR adds local AES-GCM encryption helpers, an EncryptedSessionManager wrapper, transcript and sessions.json migration/read helpers, and gateway/session-store call-site changes intended to transparently encrypt session JSONL transcripts and sessions.json metadata.

## Best Possible Solution

Design encryption at the central session persistence boundary with explicit key sourcing, recovery and rollback semantics, fail-closed migration, complete coverage of SessionManager readers/writers, docs, changelog, focused tests, and maintainer/security-owner approval before landing.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists from the PR diff and current main: the PR adds await inside a synchronous function, CommonJS require calls in a type: module package, and current main's normal embedded run still writes through SessionManager.open(params.sessionFile) directly.

## Solution Assessment

No. This patch is not the best way to solve the issue because it patches selected readers and helper writes instead of the central persistence seam, while storing key material and plaintext backups beside the data it intends to protect.

## Review Findings

Overall confidence: 0.94

Full review comments:

- **[P1] Keep async imports out of the sync preview reader:** `src/gateway/session-utils.fs.ts:474`
  - body: readLastMessagePreviewFromOpenTranscript is still synchronous, but this adds await import(...) inside it. That is a syntax/build blocker before runtime; use a static import or change the exported call chain consistently.
  - confidence: 0.97
- **[P2] Use ESM imports for encryption helpers:** `src/config/sessions/encrypted-session-manager.ts:242-252`
  - body: This package is loaded as ESM, but EncryptedSessionManager.open() and forkFrom() call require(...). Any path that reaches these helpers can fail with require undefined; use static ESM imports or an existing lazy runtime boundary.
  - confidence: 0.92
- **[P2] Encrypt the central SessionManager persistence path:** `src/config/sessions/transcript.ts:281`
  - body: This only routes the delivery-mirror helper through the encrypted wrapper. Normal embedded runs and other SessionManager.open/forkFrom paths can still write plaintext JSONL, so the claimed transparent transcript encryption is incomplete.
  - confidence: 0.91
- **[P2] Fail migration instead of preserving plaintext lines:** `src/config/sessions/migrate-transcripts.ts:36-37`
  - body: When a line fails to encrypt, migration appends the original plaintext line and can still report success. That leaves sensitive content unencrypted while callers believe migration completed.
  - confidence: 0.9
- **[P2] Do not leave plaintext backups beside encrypted sessions:** `src/config/sessions/migrate-transcripts.ts:40-42`
  - body: The migration copies the original plaintext transcript to a .backup-* file next to the encrypted file. After migration, sensitive transcript content still remains on disk, defeating the at-rest encryption goal for filesystem readers.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff introduces at-rest encryption and migration behavior with concrete key co-location, plaintext fallback, and plaintext backup concerns.

Concerns:

- **[high] Session keys are stored beside encrypted state:** `src/config/sessions/encryption.ts:45`
  - body: The new key wrapper stores both the passphrase file and wrapped master key under the OpenClaw state directory. A same-user filesystem reader who can copy session files can likely copy these key files too, so this does not satisfy the stated filesystem-compromise defense boundary without a different key source such as a user secret or platform keystore.
  - confidence: 0.88
- **[high] Migration leaves plaintext residues on disk:** `src/config/sessions/migrate-transcripts.ts:40`
  - body: The migration path copies plaintext transcripts and sessions.json to backup files before encryption, and failed per-line encryption can preserve plaintext in the migrated file. That means sensitive session content can remain readable after a reported migration.
  - confidence: 0.9
