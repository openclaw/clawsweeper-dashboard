## Label Justifications

## Summary

Current main addresses the reported stuck-recovery gap: stalled embedded-run recovery now gates on stale run progress rather than refreshed session age, keeps recovery keyed by sessionKey when sessionId is missing, and has regression coverage for queued work refreshing session activity.

## What This Changes

The issue reports a WhatsApp embedded run whose stable sessionKey remains known but sessionId becomes unknown, leaving stuck-session recovery at recovery=none while queued messages block until gateway restart.

## Best Possible Solution

Keep the current main implementation and ship it in the next release; reopen only if a fresh log on current main still shows recovery=none after lastProgressAge exceeds diagnostics.stuckSessionAbortMs.

## Reproduction Assessment

Yes for the core failure condition: current tests and source model the case where queued/session activity refreshes while embedded-run progress is stale. I did not run a live WhatsApp reconnect reproduction, but the reported recovery=none predicate is covered by source inspection.

## Solution Assessment

Yes. Using lastProgressAgeMs for active embedded-run recovery is narrower than broad lane reclamation by sessionKey and preserves the existing safety guard against clearing active unregistered lane work.
