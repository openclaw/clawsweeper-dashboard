## Summary

Keep open: current main still uses a fresh WebChat send id as the agent run id for each message, while the existing prompt-cache affinity hook is not populated on the chat.send path. The provider hit-rate numbers still need live confirmation, but the source path supports the central report and there is no merged fix or canonical replacement item that closes it.

## What This Changes

The issue reports that WebChat starts a new embedded agent run for each message in one session, destabilizing provider prompt-cache reuse and lowering cache hit rates.

## Best Possible Solution

Add a stable, privacy-safe promptCacheKey for session-bound WebChat/chat.send turns while preserving per-send idempotency run ids, then cover it with a two-message same-session regression test.

## Reproduction Assessment

No live provider reproduction was run, but the source path is reproducible: WebChat generates a fresh idempotency/run id per send and the normal reply-run base params do not set the existing promptCacheKey field. The provider hit-rate claim still needs live or harness proof after a fix.

## Solution Assessment

No, reusing one runId across messages is not the best fix because runId is also the WebChat send idempotency and UI lifecycle key. The maintainable path is to keep per-message run ids and add stable session-level prompt-cache affinity through the existing promptCacheKey contract.
