## Summary

Keep open: this maintainer-labeled PR should not be cleanup-closed, and the patch has a correctness blocker in the new manifest-built tool catalog path.

## What This Changes

The PR moves plugin `tools.catalog` entries toward manifest-based construction, adds Tavily/Firecrawl tool availability metadata, introduces an opt-in static plugin tool factory cache flag, and updates tests, docs, and changelog.

## Best Possible Solution

Keep the manifest-first catalog direction, but filter manifest tools through the same effective plugin activation/base policy as runtime loading before replacing `resolvePluginTools` in `tools.catalog`.

## Reproduction Assessment

Yes. A static reproduction path is clear: with the PR loop, a disabled/default-disabled bundled plugin with `contracts.tools` and no `toolMetadata` is included because bundled manifests pass control-plane availability and missing tool metadata returns available.

## Solution Assessment

No, not yet. Avoiding runtime tool materialization is the right direction, but the PR needs an activation-policy gate so disabled bundled plugin tools do not appear in the catalog.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Filter catalog tools through plugin activation:** `src/gateway/server-methods/tools-catalog.ts:131-136`
  - body: The new manifest loop treats bundled plugins as available via `isManifestPluginAvailableForControlPlane`, and tools without `toolMetadata` pass availability by default. Gate each manifest plugin through the same activation/base policy before adding its tools.

## Repair Work Prompt

Repair PR #76014 by preserving the manifest-based `tools.catalog` optimization while filtering plugin manifest tools through effective plugin activation/base policy. The observable bug is that disabled/default-disabled bundled plugin tools with no `toolMetadata` would be listed because `isManifestPluginAvailableForControlPlane` returns true for bundled plugins and `hasManifestToolAvailability` returns true without metadata. Update `src/gateway/server-methods/tools-catalog.ts` to honor global disablement, per-plugin `enabled: false`, deny/allow policy, default enablement, and auto-enabled config without materializing runtime tool factories. Add focused regression coverage in `src/gateway/server-methods/tools-catalog.test.ts` for an explicitly disabled bundled plugin and a disabled-by-default no-metadata bundled tool. Keep the changelog entry if the user-facing behavior remains; do not touch unrelated plugin manifests, generated baselines, or dependency metadata.
