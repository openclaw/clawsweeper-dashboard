## Summary

Current main already prevents the reported starvation path for silent/stalled model calls. Each embedded candidate gets an LLM idle watchdog, timeout aborts are preserved as timeout-shaped AbortErrors, and runWithModelFallback classifies them as failover errors and proceeds to the next configured fallback. The related PR #46193 had the same diagnosis but was closed unmerged; the shipped main/release implementation covers the user-visible failure mode through the idle-timeout/failover path.

## Best Possible Solution

Close as implemented. Keep the shipped LLM idle-timeout plus timeout-failover classification path rather than adding a second fallbackAttemptTimeoutSeconds knob; if a new report shows a provider hanging before a stream object exists or an already-aborted cron signal preventing fallback, track that as a narrower reproduction against the remaining cron/provider boundary.
