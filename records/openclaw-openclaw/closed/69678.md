## Summary

Keep open: current main and v2026.5.12 still hard-code the cron run-log summary cap at 2000 UTF-16 code units, and no duplicate or closing PR owns the requested configurable or larger cap. The remaining choice is a user-facing config/default/audit policy decision, so this should not be auto-closed or queued as a narrow bug fix yet.

## What This Changes

The issue requests making cron run-log summaries configurable or increasing the hard-coded 2000-character cap so long delivered agent outputs remain auditable.

## Best Possible Solution

Define a bounded cron run-log summary cap contract, likely under cron.runLog, then align schema/help/docs, helper plumbing, persistence behavior, and focused regression coverage.

## Reproduction Assessment

Yes, at source level. Any cleaned cron output longer than 2000 UTF-16 code units passes through pickSummaryFromOutput, receives a 2000-character prefix plus ellipsis, and is then persisted as the run-log summary.

## Solution Assessment

Unclear until maintainers choose policy. Adding cron.runLog.maxSummaryChars is a narrow implementation path, but the default value and whether summary should serve as full audit text are user-facing product decisions.
