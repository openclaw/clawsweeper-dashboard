## Summary

Current main already addresses this Slack cron announce failure by treating `delivery.channel` as the channel provider, validating it at cron add/update time, and documenting the Slack target form. The exact reported misuse now fails early instead of persisting a broken cron job.

## Best Possible Solution

Keep the shipped gateway validation, CLI/docs guidance, and Slack direct-delivery behavior. Users with older invalid cron rows should repair them with `openclaw cron edit <id> --channel slack --to "channel:<id>"`; a bulk doctor migration can be a separate enhancement if maintainers want it.

## Evidence

- **Issue discussion identified the config mistake:** The linked commenter explained that `--channel` expects the delivery provider such as `slack`, while the Slack channel ID belongs in `--to`; their workaround was `openclaw cron edit <id> --channel slack --to C0AT2Q238MQ`. This matches the current implementation boundary.
- **Gateway validates announce delivery provider at add/update time:** `assertConfiguredAnnounceChannel` compares `delivery.channel` against configured announce channel IDs and throws `delivery.channel must be one of: ...` for invalid target-id-as-provider values; both `cron.add` and `cron.update` call this before persisting changes.
