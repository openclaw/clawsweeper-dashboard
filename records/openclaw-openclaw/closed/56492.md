## Summary

Keep open. Current main has a startup READY-timeout/restart mitigation for Discord gateway stalls, but the issue’s reported root cause is still present in OpenClaw: `SafeGatewayPlugin.registerClient()` still performs an awaited async gateway-info fetch before `super.registerClient(client)`, so the constructor fire-and-forget registration window has not been removed or directly regression-tested.

## Evidence

- **Root-cause race still present in gateway plugin registration:** `SafeGatewayPlugin.registerClient()` still awaits `fetchDiscordGatewayInfoWithTimeout(...)` before calling `super.registerClient(client)`. That means OpenClaw still relies on async plugin registration completing later, rather than making registration synchronous or awaiting it at construction time.
  - file: [extensions/discord/src/monitor/gateway-plugin.ts:274](https://github.com/openclaw/openclaw/blob/6b618f063516aa9ba85903aa4565c5a204c055b2/extensions/discord/src/monitor/gateway-plugin.ts#L274)
  - command: `nl -ba extensions/discord/src/monitor/gateway-plugin.ts | sed -n '170,320p'`
  - sha: [6b618f063516](https://github.com/openclaw/openclaw/commit/6b618f063516aa9ba85903aa4565c5a204c055b2)
- **Current code mitigates with restart logic instead of awaited registration:** `waitForGatewayReady()` waits 15s for READY, records `startup-not-ready`, then disconnects/reconnects the gateway and only throws after a second timeout. This addresses the symptom of hanging startup, but it is not the awaited-registration fix requested in the issue body/title.
