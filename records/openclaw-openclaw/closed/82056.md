## Summary

Keep open: this PR addresses a real MiniMax OAuth refresh bug at the right provider seam, but the current branch still loses refresh failure reason strings, defaults legacy CN credentials to the global refresh host, leaves the user docs stale, and lacks real proof of the refresh callback path.

## What This Changes

The PR changes the MiniMax bundled plugin to use account.* /oauth2 device-code endpoints, adds a MiniMax refreshOAuth path and tests, and updates the raw-fetch allowlist.

## Best Possible Solution

Land a MiniMax-local refreshOAuth implementation after it preserves server failure reasons, supports both regions for new and legacy credentials, updates the MiniMax provider docs, and includes redacted real refresh-path proof.

## Reproduction Assessment

Yes, as source-level reproduction: current main stores MiniMax OAuth refresh tokens and advertises auto-refresh, but minimax-portal registers no refreshOAuth hook, so expired credentials fall through the generic refresh path. I did not run a live expired-token repro in this read-only review.

## Solution Assessment

No, not yet: provider-owned refreshOAuth on MiniMax account/oauth2 is the right seam, but this branch still needs correctness fixes, docs alignment, and real refresh-path proof before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve MiniMax refresh failure details:** `extensions/minimax/oauth.ts:263-264`
  - body: When `/oauth2/token` returns HTTP 200 with `status: "error"` and `base_resp.status_msg` like the branch test's `refresh_token_reused`, this throws a generic message. OAuthManager and doctor classify re-auth cases from strings such as `refresh_token_reused` or `invalid_grant`, so permanent refresh failures would not get the intended recovery path. Include the server status message/error text in the thrown error and update the test.
  - confidence: 0.9
- **[P2] Support legacy CN refresh region detection:** `extensions/minimax/provider-registration.ts:208-211`
  - body: New logins store `enterpriseUrl`, but existing MiniMax OAuth credentials created by current main only store `access`/`refresh`/`expires`, and CN/global share the same profile id. With no `enterpriseUrl`, this branch always refreshes against the global account host, so existing CN users can still fail after merge. Add a safe legacy path, such as region metadata repair or regional fallback behavior covered by tests.
  - confidence: 0.84
- **[P3] Update the MiniMax OAuth endpoint docs:** `extensions/minimax/oauth.ts:9-13`
  - body: The code now authenticates OAuth users against `account.minimax.io` and `account.minimaxi.com`, but the provider docs in this PR still say OAuth onboarding authenticates against the `api.*` hosts. Update the MiniMax provider page to distinguish the account OAuth host from the API inference base URL so docs match the shipped behavior.
  - confidence: 0.86
