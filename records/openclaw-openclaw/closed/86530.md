## Label Justifications

- `P2`: The bug blocks later requests to one affected session but is already fixed on current main and has bounded blast radius.
- `impact:session-state`: The report is about stale per-session lock state after an agent session ends.
- `impact:message-loss`: The stale lock causes subsequent channel requests to time out rather than reach the agent normally.

## Summary

Current `main` already includes the embedded-attempt lock teardown fix that matches this report: `session.ended` is recorded before cleanup, cleanup can reacquire the `.jsonl.lock`, and the newer outer teardown now disposes any retained lock so a post-prompt or cleanup exception cannot leave later requests behind `SessionWriteLockTimeoutError`. The fix is on `main` only and is not in `v2026.5.22`, which matches the reporter's release.

## What This Changes

The issue reports that a completed agent session can leave its session `.jsonl.lock` file behind after `session.ended`, causing later requests for that session to wait and fail with a 60 second lock timeout.

## Best Possible Solution

Keep the current-main lock teardown fix and let the next release carry it to users on `v2026.5.22`; reopen only if the same symptom reproduces on a build containing commit `32ddfc22f5e432ef245541960a67949cb9f8d4ea`.

## Reproduction Assessment

Yes from source and history: in `v2026.5.22`, `session.ended` is recorded before `acquireForCleanup`, but there is no outer `releaseRetainedSessionLock` fallback, so an exception after post-prompt cleanup can leave the live-process lock until watchdog reclaim. I did not run a live Feishu/macOS reproduction in this read-only review.

## Solution Assessment

Yes; the current-main fix is the narrow maintainable solution because it releases the retained lock in the run's outer teardown while keeping normal cleanup ownership and double-release safety intact.
