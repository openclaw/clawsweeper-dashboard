## Summary

Keep open. The PR targets the real #74423 catalog-noise bug, but it has the protected `maintainer` label and a blocking regression in the proposed `/models` path: auth-based visibility drops the target agent directory, so non-main agent credentials can be filtered against the wrong store.

## What This Changes

The PR adds shared model-catalog visibility/auth helpers, wires them into Gateway `models.list`, chat `/models`, and model setup pickers, and updates docs, tests, and changelog so unauthenticated catalog entries are hidden by default while explicit `all` views remain available.

## Best Possible Solution

Keep the shared visibility policy, but thread `agentDir` through `buildModelsProviderData` into `resolveVisibleModelCatalog` and add a regression test for a secondary-agent `/models` session whose provider auth exists only in that agent store, then let a maintainer review and land the protected PR.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Pass agent-scoped auth context into /models visibility:** `src/auto-reply/reply/commands-models.ts:345`
  - body: `/models` visibility now depends on provider auth, but this call only passes `agentId` and drops the already-computed `agentDir`. `buildModelsProviderData` then checks auth against the default store, so a secondary agent with its own provider profiles can have valid providers hidden, or providers from the wrong store shown. Thread `params.agentDir` through the helper and into `resolveVisibleModelCatalog`.
