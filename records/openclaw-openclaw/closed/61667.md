## Summary

Keep open. Current main has added durable path/pattern allowlist support and preserves approval state, but it still does not fully solve the reported Windows dynamic-command workflow: allow-always can still fall back to exact command hashes for interpreter-like commands, and Windows auto-persisted allow-always patterns bind the full argv exactly, so the reported `Day_007.md` -> `Day_008.md` case would still prompt unless the user manually configures a broader allowlist/trust policy.

## Best Possible Solution

Keep the issue open and implement a deliberate trust-scope UX/API for allow-always, likely offering explicit choices such as exact command, executable path, script path, or path glob. The implementation should preserve strictInlineEval protections for inline interpreter eval, define Windows argument-pattern behavior clearly, and add regression tests for the reported Python script with changing data-file arguments plus persistence across gateway/node-host restart.
