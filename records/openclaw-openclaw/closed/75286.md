## Summary

Current main already covers the reported bare `/reset` failure path: bare resets now terminate in command handling instead of falling through to model dispatch, and the OpenAI Codex Responses transport has a second guard that prevents system-only Codex calls from sending an empty `input` payload. The fix is present on main but not proven shipped in a release tag.

## What This Changes

The issue reports that a bare Telegram `/reset` on an `openai-codex` agent can trigger an empty OpenAI Responses request and forward the provider HTTP 400 string back to Telegram.

## Best Possible Solution

Keep the current main behavior: bare `/reset` should be handled as a command acknowledgement without dispatching an empty model turn, while the Codex Responses transport should retain its system-only payload guard as defense in depth.

## Reproduction Assessment

Yes. The report provides a concrete v2026.4.26 reproduction, and current-main source plus regression tests cover the same failing shapes: bare `/reset` does not continue to model execution, and system-only Codex Responses payloads get non-empty input.

## Solution Assessment

Yes. The current solution fixes the problem at the shared command and transport boundaries, avoiding Telegram-specific special casing while preventing malformed Codex Responses requests if another system-only path reaches the transport.
