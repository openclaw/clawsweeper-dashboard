## Summary

Current main and the shipped v2026.4.27 release already cover the reported stale Claude CLI binding loop. The code verifies a stored claude-cli session against the Claude project transcript before attempting resume, clears missing or empty bindings through the session-store merge path, and tests cover both session_expired clearing and missing-transcript no-resume behavior.

## What This Changes

Fix Telegram DM claude-cli session binding recovery so an expired or missing Claude CLI session id is cleared or replaced instead of causing repeated duplicate CLI exec retries and lost conversation continuity.

## Best Possible Solution

Keep the shipped transcript-missing recovery path. If paired cli exec lines still occur on v2026.4.27 or newer, the remaining best path would be a new, narrower report with redacted stream-json output and the post-run agentMeta shape, because that would point to a separate session-id extraction issue rather than the stale-binding fossilization described here.

## Reproduction Assessment

Yes. The report's manual deletion path maps to covered tests that create a phantom claude-cli binding, verify no --resume is passed when the Claude transcript is missing, and verify the stale binding is removed from persisted session state.

## Solution Assessment

Yes. Verifying the provider-owned transcript before resume and clearing the stale binding through the session-store merge path is narrower and safer than a polling sidecar or changing output mode; the backend already defaults to jsonl/stream-json and carries session ids from Claude output when present.
