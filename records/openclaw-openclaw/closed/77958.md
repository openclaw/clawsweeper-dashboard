## Summary

This should stay open: current main still creates plugin-skill directory links with Windows directory symlink semantics, and the report describes a narrow, source-backed Windows standard-user failure.

## What This Changes

Plugin-skill registration should work on Windows standard accounts without Developer Mode by using directory junctions or an equivalent fallback for plugin-shipped skill directories.

## Best Possible Solution

Use Windows directory junctions for generated plugin-skill directory links, preserve the existing containment and cleanup guarantees, and add regression coverage plus Windows real-behavior proof when feasible.

## Reproduction Assessment

Yes from source inspection, but not by live execution here: current main hardcodes `fs.symlinkSync(..., "dir")` for directory targets and only warns on failure, which matches the reported Windows EPERM path. Live reproduction should be run on a standard Windows account without Developer Mode before landing.

## Solution Assessment

Yes, the narrowest maintainable fix is to use `junction` for Windows directory plugin-skill links or retry EPERM with a junction while keeping the existing security checks and cleanup behavior. This should not broaden into the related stderr-noise report #77942 beyond avoiding this specific warning after a successful fallback.

## Repair Work Prompt

Fix the Windows plugin-skill registration bug reported at https://github.com/openclaw/openclaw/issues/77958. In `src/agents/skills/plugin-skills.ts`, generated plugin skill entries under `~/.openclaw/plugin-skills` currently use `fs.symlinkSync(target, linkPath, "dir")`; on Windows standard accounts without Developer Mode this can fail with EPERM. For directory skill targets, use a Windows junction (`fs.symlinkSync(target, linkPath, "junction")` on `process.platform === "win32"`) or retry EPERM with a junction while preserving the existing path containment checks, basename collision handling, idempotence, stale cleanup, and plugin-agnostic core boundary. Add focused regression coverage in `src/agents/skills/plugin-skills.test.ts` for the Windows EPERM/junction path, preferably by isolating link creation or injecting/mocking the symlink operation rather than requiring actual Windows privileges. Ensure cleanup still handles generated junction/symlink entries safely. Add a single-line `CHANGELOG.md` entry under Unreleased Fixes crediting @jvpflum. Do not add a config option and do not broaden the change into the unrelated Ollama stderr issue #77942, except that this specific plugin-skill warning should disappear when the junction fallback succeeds.
