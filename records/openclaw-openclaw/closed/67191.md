## Summary

Keep open. Current main still runs BOOT.md through a silent boot session and the startup hook does not provide a channel, target, thread, or reply context, so the remaining work is to define the BOOT.md startup delivery contract rather than close this as fixed.

## What This Changes

Request BOOT.md startup runs to deliver user-visible status messages instead of running silently.

## Best Possible Solution

Define one explicit BOOT.md startup delivery contract, then align boot execution, docs, and regression tests around that contract.

## Reproduction Assessment

Yes, source-reproducible. The restart scenario maps to current main: boot-md calls runBootOnce without routing context, runBootOnce uses deliver:false, and the test suite encodes that silent command shape.

## Solution Assessment

No. The suggested one-line deliver:true change is not proven best because the boot path still lacks a startup delivery destination; maintainers should choose the delivery contract first.
