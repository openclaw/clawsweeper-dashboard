## Summary

Keep this PR open for maintainer review. Current main does not implement the PR's new `persist` or `skills.trustedSources` surfaces, and the linked feature request remains broader than the PR because it asks for commercial verified/signed or encrypted skill wrapping and security-review policy. The PR is also security-sensitive because it changes code-safety audit behavior for skills.

## Best Possible Solution

Keep this PR open only as an implementation candidate for maintainer review. The best path is to split the work: first decide the product/security model for commercial signed or encrypted skill bundles and ClawHub verification, then either document and test a narrow compatibility alias for the existing `always` behavior or design a reviewed trust/provenance API for audit bypasses. The audit-scan bypass should not land without an explicit threat model, user-facing docs, and status/config/schema alignment.

## Risks / Open Questions

- `skills.trustedSources` weakens the current audit model by letting broad skill source classes bypass code-safety scanning; even with an info finding, this needs explicit maintainer security review and docs.
- The `persist` field overlaps with the existing `metadata.openclaw.always` gate override but adds a new public config/metadata surface without current docs or generated schema/help updates in the provided PR file list.
- The linked feature request asks for commercial verified, signed, or encrypted skill wrapping; this PR implements neither wrapping nor a ClawHub provenance/trust model, so merging it could prematurely close a broader product request.
- The PR diff does not update status/reporting surfaces such as `src/agents/skills-status.ts`, which currently derives `always` only from metadata; config-level `persist` could be included at runtime but still appear ineligible or misleading in status output.
