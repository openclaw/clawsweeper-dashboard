## Summary

Current main already addresses the reported Telegram sleep/wake half-open polling stall by hard-timing out stuck `getUpdates` calls after 45s instead of waiting for the old watchdog-only recovery path, and the fix is documented in the changelog and current troubleshooting docs.

## Close Comment

Closing this as implemented on main after a Codex review. The reported Telegram sleep/wake stall is no longer handled only by the old ~100s watchdog path: `extensions/telegram/src/request-timeouts.ts:6` now gives `getUpdates` a hard 45s timeout, `extensions/telegram/src/bot-core.ts:204` wires that timeout into Telegram fetches, and `extensions/telegram/src/bot.fetch-abort.test.ts:72` covers the abort behavior. The changelog shipped this as `Telegram/polling: hard-timeout stuck getUpdates requests so wedged network paths fail over sooner instead of waiting for the polling stall watchdog` in the `2026.3.22` release (`CHANGELOG.md:2309`). Current docs also treat slow Telegram polling stalls as an upgrade/tuning path now (`docs/channels/troubleshooting.md:51`, `docs/channels/telegram.md:260`). The macOS-specific reachability and LaunchAgent `ProcessType` ideas from this report are not all on main, but the concrete bug in the title, polling stalling for ~100s after sleep/wake or network change, is already addressed by the shipped timeout-based recovery path.
