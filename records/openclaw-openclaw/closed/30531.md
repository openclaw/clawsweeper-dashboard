## Summary

Current `main` already implements Discord-specific proxy routing for gateway WebSocket traffic and startup REST lookups, which is the OpenClaw-side integration needed for Clash/Sing-box `url-test` or fallback handover. The Discord lifecycle also reacts immediately to gateway close/reconnect events, so the reported Discord proxy-failure path is covered on `main` even though the external proxy failover logic itself remains in the proxy tool.

## Risks / Open Questions

- The issue asks for built-in proxy failover semantics, while the implemented design delegates failover to an external local proxy such as Clash/Sing-box and provides the OpenClaw routing hook needed to use it.
- This appears unreleased as of `v2026.4.22`, so users on the latest tagged release would still need a newer main build to get it.

## Close Comment

Closing as implemented on `main` after Codex review. Current `main` now has Discord-specific proxy routing via `channels.discord.proxy`, and the docs explicitly state that it covers Discord gateway WebSocket traffic plus startup REST lookups for application ID and allowlist resolution (`docs/channels/discord.md:749`). The Discord lifecycle also reacts immediately to gateway close/reconnect events (`extensions/discord/src/monitor/provider.lifecycle.ts:223`). This landed in commit `c0a7b6a510758645c5267734b984f4885a605b67` and is on current `main`, but not in release `v2026.4.22` yet.
