## Summary

Keep open. Current `main` and the latest release still record cron run usage into cron telemetry/session state but do not emit the trusted `model.usage` event that OTel and Prometheus diagnostics plugins consume; the linked fix PR is closed unmerged.

## What This Changes

The issue requests `model.usage` diagnostic events for cron-isolated agent runs so scheduled model spend appears in diagnostics/OTel metrics.

## Best Possible Solution

Add a cron finalization `model.usage` emission that matches the established auto-reply diagnostic event contract and covers the billed aggregate in regression tests.

## Reproduction Assessment

Yes at source level: current `main` records nonzero cron usage into telemetry/session state without emitting `model.usage`, while auto-reply emits that diagnostic event and diagnostics plugins subscribe to it. I did not run a live cron plus OTel reproduction in this read-only review.

## Solution Assessment

Yes. Emitting from cron finalization is the narrowest maintainable repair, provided it uses the same diagnostic contract as auto-reply and keeps billed total separate from context/session occupancy.
