## Summary

Current main already implements the central Discord/channel-backed export behavior: native command turns carry the real target session into command handling, and the export handler now writes the HTML file or returns a visible error instead of silently doing nothing. The fix is current-main-only in this checkout; no release tag contains the proof commit yet.

## What This Changes

The issue reports that `/export-session` and `/export` silently no-op in Discord/channel-backed sessions after v2026.5.12 instead of writing an HTML export and confirming it.

## Best Possible Solution

Keep the current generic native command target-session handling on main and let the next release carry it; the separate CLI/ACP transcript completeness work remains tracked at https://github.com/openclaw/openclaw/issues/90844.

## Reproduction Assessment

No current-main failing repro was established. Source and regression coverage show current main targets the channel session for native commands and returns export success/error text, while the proof commit is not in the latest release tag inspected.

## Solution Assessment

Yes. The current generic command-target session routing is the maintainable fix because it keeps `/export-session` channel-agnostic instead of adding Discord-specific export logic.
