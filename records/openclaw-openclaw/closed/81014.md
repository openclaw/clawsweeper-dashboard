## Summary

Close: current main and shipped docs already define `cacheWrite` as provider-reported telemetry. OpenAI-compatible paths can report `cacheRead` while intentionally leaving `cacheWrite` at zero unless upstream emits a write-side counter, and Anthropic paths preserve upstream cache-creation tokens.

## What This Changes

The issue asks whether session telemetry `cacheWrite=0` with nonzero `cacheRead` is intended and requests documentation or a pointer to the population path.

## Best Possible Solution

Keep the provider-reported counter semantics and direct users to the prompt-caching docs rather than deriving cache writes from cache hits.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: OpenAI-compatible usage maps cached tokens to `cacheRead` and leaves `cacheWrite` at zero, while the session store persists zero unless normalized usage includes a write counter.

## Solution Assessment

Yes. The shipped docs and current code are the maintainable solution; synthesizing write tokens from read tokens would make telemetry less accurate for providers that do not expose write-side accounting.
