## Summary

Keep open: the linked performance problem is real on current main, but this PR is not merge-ready because the module-level 5 ms cache can both miss the reported slow-path cache hit and return stale task snapshots after registry mutations.

## What This Changes

The PR adds a short-lived in-process cache around `reconcileInspectableTasks()` and clears it in the task registry test reset helper.

## Best Possible Solution

Use a per-`getStatusSummary()` shared reconciled task snapshot, preserving zero-argument helper behavior for other callers; only keep a shared cache if it is lifecycle-owned and invalidated by task-registry mutation events.

## Reproduction Assessment

Yes by source inspection: current main still calls the registry summary and task audit helper paths separately, and each reaches `reconcileInspectableTasks()` before `listTaskRecords()` clones and sorts the task list. I did not run a live 10K-task benchmark in this read-only review.

## Solution Assessment

No. A scoped shared snapshot inside `getStatusSummary()` is narrower and safer than a module-level TTL cache over mutable task registry state.

Full review comments:

- **[P2] Stamp the cache after reconciliation:** `src/tasks/task-registry.maintenance.ts:901`
  - body: This stores `time: now` from before `listTaskRecords().map(...)` runs. On the reported large-task workload, the first reconciliation takes far longer than 5 ms, so the cached result can already be outside the TTL before the second status helper runs and the duplicate work remains. Scope the reuse to the status call or timestamp after the result is computed.
  - confidence: 0.92
- **[P2] Invalidate cached snapshots on task mutations:** `src/tasks/task-registry.maintenance.ts:886-887`
  - body: `reconcileInspectableTasks()` is shared by status, task audit, maintenance, and restart-blocker checks, but this cache is not invalidated when task records are restored, created, updated, or deleted. A caller can reconcile, mutate the registry, then ask for another inspectable snapshot within the TTL and read the old array. Keep the cache scoped to one status call or wire it to the task-registry observer events with tests.
  - confidence: 0.9
