## Summary

Current `main` already handles this class of failure: it distinguishes LLM idle timeouts from normal run timeouts, retries one silent idle-timeout on the same model, surfaces a targeted user-facing/config message, and documents the supported `agents.defaults.llm.idleTimeoutSeconds` workaround. The changelog shows these timeout fixes shipped in `v2026.4.9` and `v2026.4.10`.

## Evidence

## Risks / Open Questions

- The top-level subagent announce status is still a generic `timed out`; the more specific idle-timeout reason is surfaced through the run payload/error text and runtime metadata rather than a distinct public status enum.
- If a provider has its own hard upstream idle disconnect independent of OpenClaw's watchdog, the inspected code does not add provider-specific heartbeat token injection; the shipped fix is the longer/configurable watchdog plus retry/error handling.
