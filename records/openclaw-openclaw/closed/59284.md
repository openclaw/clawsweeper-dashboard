## Summary

Keep open. Current main still has no owner-presence guard for Slack rooms: bot-authored messages are dropped only when `allowBots` is false, and after `allowBots` is enabled the normal room path relies on channel policy and optional `users` allowlists while the owner `allowFrom` match is used for control-command gating, not general message routing.

## What This Changes

The issue requests a Slack `requireOwnerPresent` guard so bot-authored messages in group channels with `allowBots: true` only route when an authorized owner is a channel member.

## Best Possible Solution

Add the guard in the Slack plugin-owned inbound authorization path, expose it through Slack config types/schema/docs/UI hints, cache owner membership by channel with a short TTL, and add focused tests for bot and human room messages with owner present, owner absent, lookup failure, and existing `allowBots` attachment behavior.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: configure an allowed Slack room with `allowBots: true`, omit a `users` allowlist, and send a bot-authored message; current main has no owner membership check before the message can be prepared for routing.

## Solution Assessment

Unclear pending maintainer/security review. The proposed `requireOwnerPresent` guard is a plausible narrow Slack-owned fix, but the default behavior and lookup failure semantics should be decided explicitly because this is a security-sensitive compatibility boundary.
