## Summary

Current `main` already implements a targeted workaround for this pnpm 10 partial-workspace prune bug. The Docker build now rewrites `pnpm-workspace.yaml` to only the root, `ui`, and opted-in bundled plugins before running `pnpm prune --prod`, so prune no longer reconciles against the later full workspace copy that caused installed package dependencies to be wiped.

## Evidence

- **Current Dockerfile narrows the workspace before prune:** `runtime-assets` writes `/tmp/pnpm-workspace.runtime.yaml`, copies it over `pnpm-workspace.yaml`, and only then runs `CI=true NPM_CONFIG_FROZEN_LOCKFILE=false pnpm prune --prod`. The adjacent comment explicitly says prune must not rediscover unrelated workspaces from the later full source copy.
