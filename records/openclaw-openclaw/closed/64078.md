## Summary

Keep open. Current main still preserves the reported Slack routing split for ordinary top-level channel messages under `replyToMode="all"`, and an open paired fix PR with closing syntax is already tracking the implementation.

## What This Changes

The issue requests that Slack top-level channel messages with `replyToMode="all"` use the same per-message thread id for inbound `SessionKey` routing and reply delivery instead of also using the shared channel session.

## Best Possible Solution

Review and land #64080, or an equivalent Slack-owned patch, so room/channel routing derives one conversation thread id for both inbound session routing and reply delivery while preserving `replyToMode=off`, `first`, `batched`, true thread replies, seeded mentions, and current DM behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is available by calling `resolveSlackRoutingContext` for a Slack channel message with `isRoomish: true`, no `thread_ts`, no seeded mention, and `replyToMode="all"`; the session key falls back to the base channel while `MessageThreadId` is the message timestamp, and the issue has two live reports matching that path.

## Solution Assessment

Yes for the proposed direction. Unifying the routed thread id with the reply-delivery thread id is the narrowest maintainable fix, provided the existing mode-specific behavior and seeded mention cases remain covered by tests.
