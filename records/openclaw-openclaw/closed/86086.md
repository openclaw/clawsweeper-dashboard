## Label Justifications

- `P2`: A broken WebChat canvas embed would be a normal user-facing bug with limited blast radius, not a core outage.
- `impact:security`: The report concerns Gateway auth and scoped authorization for untrusted canvas iframe content.

## Summary

Current main and the latest release already route WebChat canvas embeds through scoped plugin capability URLs; the bare canvas URL 401 path is intentionally unauthorized, while the WebChat-rendered scoped path is authorized.

## What This Changes

The issue requests that WebChat canvas `[embed ...]` iframes include Gateway authorization when loading `/__openclaw__/canvas/*` in token-auth mode.

## Best Possible Solution

Keep the shipped scoped capability URL path as the supported WebChat canvas auth mechanism; reopen or file a new report only with the actual iframe `src` and hello payload showing `pluginSurfaceUrls.canvas` missing or ignored.

## Reproduction Assessment

No high-confidence failing current-main reproduction is present. Source and regression tests show WebChat canvas iframes are rewritten to scoped capability URLs, and the only 401 path is the intentionally unauthenticated bare canvas URL.

## Solution Assessment

Yes. The scoped capability URL is the safer existing solution because it authorizes the connected WebChat session without putting the shared Gateway token into iframe markup.
