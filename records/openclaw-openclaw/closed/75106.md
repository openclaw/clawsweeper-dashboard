## Summary

Keep open. The underlying bug is still valid on current main, but this PR's outer `Promise.race` does not cancel or neutralize the already-started embedding batch probe, and the external PR still has only test/check evidence rather than real after-fix CLI proof.

## What This Changes

The PR adds an 8-second wrapper around `memory status --deep` embedding probes, a `timedOut` probe result, timeout rendering, non-TTY progress fallback, focused tests, docs, and changelog text.

## Best Possible Solution

Put the diagnostic budget into a cancellable or event-loop-safe manager/provider probe path, keep the split vector-store/embedding status output, add non-TTY progress, and require real patched CLI output showing prompt completion.

## Reproduction Assessment

Yes for the source-level failure path: current main awaits `probeEmbeddingAvailability()`, which can enter `embedBatchWithRetry(["ping"])` under the 600-second local batch timeout. I did not live-run the reporter's Windows/local-provider setup.

## Solution Assessment

No. The UX direction is right, but the PR wraps the probe from outside instead of cancelling or neutralizing the inner batch probe/timer, so it does not reliably bound the process yet.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Cancel the inner embedding probe:** `extensions/memory-core/src/cli.runtime.ts:122`
  - body: This returns from the wrapper after 8 seconds, but `manager.probeEmbeddingAvailability()` has already started `embedBatchWithRetry(["ping"])`; that helper starts its own local 600-second timeout and provider work. The CLI can still stay alive until that inner batch path settles, so put the budget inside an abortable/event-loop-safe probe path or otherwise prevent the inner work from retaining the process.
  - confidence: 0.88
