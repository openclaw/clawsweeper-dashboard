## Summary

Keep this PR open: current main still does not fully cover the requested native ACP binding/routing behavior, but the proposed patch has blocking regressions in Feishu actions, current-binding bootstrap delivery, ACP rebind authorization, and internal announce classification.

## What This Changes

The PR adds shared ACP spawn-origin, placement, conversation-canonicalization, and bound-delivery helpers, then updates Discord, Telegram, Feishu, session announce/send routing, and generated API/config baselines.

## Best Possible Solution

Revise and rebase the PR before merge, keeping the shared origin-aware placement direction while preserving Feishu actions, sender ownership checks, correct bootstrap/bound delivery routing, store-backed internal announce detection, current generated baselines, and focused regression coverage.

## Reproduction Assessment

Yes. The PR body gives concrete Telegram/Discord repro flows, and current main has source/test evidence that native Discord thread-context ACP spawning still binds as child while /acp current binding has a sender ownership gate.

## Solution Assessment

No. The architectural direction is reasonable, but this patch is not mergeable until it fixes the Feishu action regression, bootstrap target selection, native current-binding authorization, internal announce fallback, and stale generated artifacts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Reinstate Feishu advertised action handlers:** `extensions/feishu/src/channel.ts:585`
  - body: The diff removes the Feishu handlers for pins, channel/member lookup, channel-list, and reactions after the edit branch, but the plugin still advertises those actions. Users can discover and invoke them, then every call falls through to unsupported-action behavior.
  - confidence: 0.93
- **[P1] Preserve bootstrap routing for current ACP bindings:** `src/agents/acp-spawn.ts:589`
  - body: This change allows placement: "current", but the bootstrap delivery plan still reconstructs a target from the bound conversation id through the generic delivery helper. For current Discord/Telegram surfaces that can prefer malformed targets like channel:user:<id> or channel:<chatId> over the requester origin, so the initial ACP task can be misrouted even after binding succeeds.
  - confidence: 0.86
- **[P1] Enforce ownership before current ACP rebinds:** `src/agents/acp-spawn.ts:589`
  - body: When placement resolves to current, this path binds immediately and records boundBy: "system"; unlike /acp spawn, it does not check an existing conversation binding's boundBy sender first. A different participant can therefore rebind an already-bound ACP DM/thread and clear the previous ownership metadata.
  - confidence: 0.88
- **[P2] Keep store-backed internal announce detection:** `src/agents/subagent-announce.ts:93-101`
  - body: Replacing getSubagentDepthFromSessionStore with a key-prefix check loses the stored spawnDepth/spawnedBy fallback used for legacy or alias requester sessions. Those internal sessions can be treated as external, causing orchestration completion messages to be delivered visibly to user channels.
  - confidence: 0.72
