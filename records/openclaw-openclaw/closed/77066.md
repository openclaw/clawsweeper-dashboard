## Label Justifications

- `P1`: The original report describes a recurring gateway startup failure that prevents a configured WhatsApp account from receiving messages.
- `impact:crash-loop`: The issue is about repeated health-monitor restarts after a secondary WhatsApp account crashes on boot.
- `impact:message-loss`: The failed secondary listener meant inbound WhatsApp messages for that account were not received.
- `impact:auth-provider`: The failure is tied to loading account-scoped WhatsApp auth state from persisted credentials.

## Summary

Current main and the latest release no longer have the undefined WhatsApp runtime path behind the reported `readWebAuthState` crash, and the WhatsApp auth/startup flow now models unstable auth state and restarts the running gateway after local CLI login.

## What This Changes

Investigate a WhatsApp multi-account boot crash where a secondary account with an explicit `authDir` fails to start after gateway restart with a `readWebAuthState` TypeError.

## Best Possible Solution

Keep the shipped plugin-owned WhatsApp auth/runtime implementation and ask for a fresh issue with a `v2026.6.6` diagnostics export if the crash still reproduces after upgrade.

## Reproduction Assessment

No live current-main failure was reproduced. The reported v2026.5.2 crash maps to a WhatsApp auth/runtime boot path that current main and `v2026.6.6` now handle through lazy runtime exports and explicit auth-state decisions.

## Solution Assessment

Yes. The implemented path keeps auth loading and Baileys state handling inside the WhatsApp plugin boundary, preserves account-scoped `authDir`, and adds retry-safe auth-state handling instead of adding a new config workaround.
