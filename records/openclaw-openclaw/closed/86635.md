## Label Justifications

- `P1`: The report describes a real gateway hot-path stall that blocks timers and inbound channel progress for normal messages.
- `impact:crash-loop`: The affected synchronous work can hang the Node event loop long enough to trigger gateway health and timeout failures.
- `impact:message-loss`: Inbound replies can sit without visible agent progress while channel timers and fetches are delayed.
- `impact:auth-provider`: The affected code is the provider/model default-resolution and model-choice path.

## Summary

Keep open. Current main still builds the default-model alias index on the inbound reply path before aliases or model directives are known to be needed, and related work in https://github.com/openclaw/openclaw/pull/86552 only covers the repeated manifest metadata part, not the eager alias-index construction reported here.

## What This Changes

The issue reports that provider-qualified default model resolution eagerly builds and normalizes the configured-model alias index on inbound replies, causing long gateway event-loop stalls.

## Best Possible Solution

Make provider-qualified default resolution cheap by short-circuiting the provider/model path before alias-index construction, and build alias data lazily only for directive or override paths that actually need alias lookup.

## Reproduction Assessment

No for a high-confidence end-to-end reproduction of the 80-85s stall in this checkout; yes for source-reproducible eager work. Current main still constructs the alias index before the provider-qualified default path can return.

## Solution Assessment

Yes. The narrow maintainable fix is to preserve alias behavior while deferring alias-index construction and skipping model-key normalization for aliasless configured entries.
