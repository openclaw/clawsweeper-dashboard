## Summary

Keep open: current `origin/main` and shipped `v2026.5.22` still show the Web UI calling `sessions.usage` without `agentId`, while the gateway defaults an omitted `agentId` to the default agent, so non-main usage never reaches the client-side agent filter.

## What This Changes

The issue requests fixing the Control UI Usage page so non-main agent usage remains discoverable after `sessions.usage` became agent-scoped.

## Best Possible Solution

Keep gateway agent scoping and make the Control UI agent filter a server-side scope selector that passes `agentId` to `sessions.usage`, with regression coverage for non-main agents.

## Reproduction Assessment

Yes. The issue gives concrete multi-agent steps, and source inspection shows the current Web UI omits `agentId` while the current gateway scopes omitted requests to the default agent; I did not run a live UI reproduction in this read-only review.

## Solution Assessment

Yes. Passing an explicit agent scope from the existing Usage page agent filter is the narrow maintainable fix; changing the backend to return all agents on omitted `agentId` would undo the scoped lookup hardening.
