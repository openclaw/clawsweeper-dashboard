## Summary

Keep open. Current `main` still constructs assistant error turns with `content: []`, replay normalization only fixes string assistant content, and session-file repair only drops malformed JSONL lines. The Bedrock replay-poison path described in #71572 remains plausible on `bc73141e8284c219933970dbd4f6aeacb05bbb77`.

## Evidence

- **Empty assistant error turns are still produced:** `buildStreamErrorAssistantMessage` still returns an assistant message with `content: []`. That builder is used in the stop-reason recovery path for provider/stream failures, so current main still has a first-party path that can create the reported transcript shape.

## Risks / Open Questions

- Bedrock/Anthropic-class replay can still fail hard if a persisted assistant error turn has `content: []`.
- Gateway restart likely does not clear a poisoned session because the on-load repair pass ignores syntactically valid but semantically invalid transcript entries.
- `delivery-mirror` and `gateway-injected` assistant turns appear to be handled as transcript-only for UI delivery, but there is no matching core replay filter in the inspected path.
