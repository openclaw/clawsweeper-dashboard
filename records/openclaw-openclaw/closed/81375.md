## Label Justifications

- `P1`: The report describes a real channel workflow where inbound WhatsApp replies can be delayed behind cron work after restart.
- `impact:message-loss`: The user-visible failure is an inbound chat message that is ingested but receives no visible reply until the lane clears.
- `impact:session-state`: The root path binds cron systemEvent work to the live chat session lane via sessionKey state.

## Summary

Keep open. Current main and v2026.5.18 still bind missing cron job sessionKey values from the caller session before main/systemEvent cron execution uses that key, and the closing implementation remains open at https://github.com/openclaw/openclaw/pull/81376.

## What This Changes

The issue asks for cron.add to avoid binding implicit main/systemEvent user-facing cron jobs to the caller's live chat session lane while preserving explicit session keys and isolated agentTurn context.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/81376 with the scoped cron-tool normalization guard and regression coverage, then close this issue after the fix merges.

## Reproduction Assessment

Yes, at source level: current main defaults implicit systemEvent cron jobs to main, stamps a missing job.sessionKey from the caller session, and later uses that key for main cron enqueue/wake. I did not run the full live WhatsApp restart scenario.

## Solution Assessment

Yes, the narrow maintainable fix is to guard implicit main/systemEvent jobs from caller sessionKey stamping while preserving explicit job.sessionKey and isolated agentTurn context.
