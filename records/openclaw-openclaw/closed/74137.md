## Summary

Keep open. Current main has useful defenses for this cluster, including Telegram empty-body filtering, non-empty bare reset transcript markers, replay/session-file blank-user repair, and an OpenAI Codex empty-input provider guard, but it still lacks the requested shared last-chance guard for blank visible user prompts before provider submission. The issue is also paired with open PR #74168 using closing syntax, so maintainers should resolve that PR before closing this issue.

## What This Changes

The issue asks OpenClaw to drop blank Telegram user messages before they are persisted or submitted to provider APIs, preventing raw empty-input provider errors from surfacing back into Telegram chats.

## Best Possible Solution

Land #74168 or an equivalent narrow embedded-runner guard that skips blank non-runtime visible user prompts before persistence/provider submission, while preserving media-only turns and runtime-only internal events. Keep the existing Telegram ingress, replay sanitization, provider-side OpenAI Codex guard, and session-file repair as layered defenses with focused regression coverage.

## Reproduction Assessment

Yes. The issue comments provide a high-confidence 2026.4.26 reproduction through Telegram `/new` in a group across two bots; current-main review also shows why the broader blank visible prompt path still needs a shared guard even though the bare reset transcript path has been mitigated.

## Solution Assessment

Yes, the best fix direction is the narrow pre-provider embedded-runner guard proposed by #74168 or an equivalent patch. Telegram-only filtering and provider-specific validation are useful defenses, but they do not cover blank visible prompts that reach the shared runner through other channel or envelope paths.
