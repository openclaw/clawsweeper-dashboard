## Summary

Keep open. Current main still emits the legacy sandbox escape diagnostic without the container-visible workspace mapping, while the resolver and bridge already carry the host root and container workdir needed for a focused recovery hint.

## What This Changes

Request that `Path escapes sandbox root` errors include the sandbox host root to container mount mapping so agents can recover by using `/workspace/...`.

## Best Possible Solution

Keep sandbox enforcement unchanged and add a focused diagnostic that includes the rejected path, sandbox host root, container mount target, and a concise hint to use the container path.

## Reproduction Assessment

Yes, at source level. Calling `resolveSandboxFsPathWithMounts` with an outside path such as `/tmp/healthcheck-alert/config.json`, a sandbox workspace root, `defaultContainerRoot: "/workspace"`, and normal mounts reaches the legacy escape branch without using the mapping.

## Solution Assessment

Yes. The narrow maintainable fix is diagnostic-only in the sandbox fs resolver or bridge; it should not auto-translate arbitrary outside paths, weaken sandbox checks, or add a config knob.
