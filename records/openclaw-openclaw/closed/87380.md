## Summary

Keep this PR open: current main still lacks a durable `gateway.tls.extraCaCerts` config path, and the branch appears to implement the requested install-plan behavior without a discrete code defect. The remaining blocker is maintainer/security acceptance of the new process-wide CA trust config contract.

## What This Changes

Adds `gateway.tls.extraCaCerts` to config/schema/docs and injects its trimmed value into `NODE_EXTRA_CA_CERTS` for generated managed gateway service environments.

## Best Possible Solution

Accept `gateway.tls.extraCaCerts` as the explicit config-owned trust path only with maintainer/security sign-off, then merge it as the canonical upgrade-safe alternative to editing generated service env files.

## Reproduction Assessment

Yes from source, though I did not run a local service reinstall: current main filters dangerous existing `NODE_EXTRA_CA_CERTS` service-env edits and has no config-owned extra CA path. The linked report gives a concrete macOS `gateway install --force` path that matches this source behavior.

## Solution Assessment

Yes, if maintainers accept the product/security contract: a config-owned CA bundle path is cleaner than preserving arbitrary generated env edits. The safer alternative, if maintainers reject process-wide trust in gateway config, is a narrower trust design rather than env preservation.
