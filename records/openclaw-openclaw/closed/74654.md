## Summary

Close as implemented on main. The cron update atomicity bug is already fixed by the merged #74720 path: current main applies updates to a cloned job, forces cron parsing for disabled cron schedule edits before replacing stored state, covers the disabled-edit and failed-enable regressions, and shipped in v2026.4.29-beta.1. This PR correctly identified the service-layer problem, but it is no longer the canonical merge path.

## What This Changes

The PR adds a `validateCronScheduleExpr` helper, calls it before `applyJobPatch` in `cron.update` for cron schedule changes and enable attempts, adds service and gateway regression tests, and adds a changelog entry.

## Best Possible Solution

Keep the shipped clone-and-replace cron update implementation from #74720 as the canonical fix, and close this PR so future follow-up work can focus on narrow test-only coverage such as the separate draft #74068 if maintainers still want it.

## Reproduction Assessment

Yes. The original bug has a deterministic service-level path: update a disabled cron job to an invalid cron expression, then enable an already-invalid disabled cron job; current main covers both paths in `src/cron/service.issue-regressions.test.ts`. This read-only review inspected the regression source and did not run tests.

## Solution Assessment

No, this PR is no longer the best merge path. Its service-layer pre-validation direction was valid, but current main's clone-and-replace update is safer because it prevents any throwing update path from partially mutating stored cron state while still validating disabled cron edits.
