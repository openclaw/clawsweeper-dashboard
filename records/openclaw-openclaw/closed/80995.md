## Label Justifications

- `P1`: The issue describes a real Google Chat channel workflow where agent replies can be delivered outside the user's active thread.
- `impact:message-loss`: The reported behavior misroutes visible messages to the wrong Google Chat thread context.

## Summary

Keep open: current main still has the Google Chat thread resource available on inbound events, but the Google Chat threading adapter does not map that resource into the message-tool current reply target, and the explicitly linked fix PR is still open/unmerged.

## What This Changes

The issue reports Google Chat `message(action=send)` replies using `[[reply_to_current]]` escaping the inbound thread instead of targeting `message.thread.name`.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/80996 or an equivalent Google Chat threading fix, then close this issue after the fix reaches main.

## Reproduction Assessment

Yes, at source level. Current main carries `message.thread.name` into `ReplyToId`, but Google Chat has no `buildToolContext`, so `[[reply_to_current]]` can inherit the inbound message resource instead of the thread resource.

## Solution Assessment

Yes. A Google Chat-owned `buildToolContext` adapter is the narrow maintainable fix because it uses the existing plugin threading seam without changing shared core reply semantics.
