## Summary

Current main and the shipped v2026.4.27 release already fix the startup-prewarm stall. The startup path keeps primary-model prewarm metadata-only, bounds it to 5 seconds, and schedules it without blocking channel startup, so the old `resolveModelAsync` provider-hook fallback cannot hold gateway readiness for 30 seconds.

## What This Changes

The issue reports that `prewarmConfiguredPrimaryModel` can block gateway startup for 30 seconds when an `openai-codex/*` primary model misses the registry and falls into async provider runtime discovery.

## Best Possible Solution

Keep the shipped metadata-only, bounded, background startup prewarm path. Broader provider catalog latency or first-message tool factory stalls should stay in their narrower related reports rather than reopening this startup-prewarm item.

## Reproduction Assessment

Yes. The reported old path is concrete from the issue steps, and current source plus regression tests give a high-confidence check that startup prewarm no longer calls the model resolver fallback or blocks channel startup.

## Solution Assessment

Yes. The current implementation is the narrowest maintainable fix: it removes runtime-hook model resolution from startup prewarm and bounds/detaches warmup, while preserving the documented distinction between `openai-codex/*` PI routing and the native `codex` plugin.
