## Summary

Current main and the v2026.5.12 release already guard the reported duplicate Telegram processing path: Telegram message IDs feed the shared inbound dedupe key, duplicate or in-flight inbound turns are skipped before dispatch, and follow-up queues dedupe repeated message IDs while preventing double drains.

## What This Changes

The issue reports intermittent duplicate Telegram direct-chat replies when one session appears both active and queued under gateway load.

## Best Possible Solution

Keep the v2026.5.12+ inbound and follow-up dedupe behavior; reopen with 2026.5.12-or-newer logs only if the same Telegram MessageSid is still processed twice.

## Reproduction Assessment

No, I did not recreate the high-load race on current main. The reported path is source-identifiable, and current main plus v2026.5.12 contain guards that skip duplicate or in-flight inbound Telegram turns before they can run twice.

## Solution Assessment

Yes. The shipped approach is the narrow maintainable fix because it dedupes at the generic inbound dispatch and follow-up queue layers instead of adding Telegram-only load heuristics.
