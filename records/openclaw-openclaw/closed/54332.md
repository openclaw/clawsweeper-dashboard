## Summary

Current main already gives each WhatsApp reconnect a fresh watchdog window, and the same implementation is present in release `v2026.4.22`, so the reported stale-idle flap loop is already fixed in shipped code.

## Evidence

## Close Comment

Closing after Codex review: current `main` already implements the watchdog-reset behavior this report asks for, and the same code is shipped in `v2026.4.22`. In `extensions/whatsapp/src/connection-controller.ts:103` a reconnect creates a fresh connection with `startedAt: Date.now()` and `lastInboundAt: null`, and in `extensions/whatsapp/src/connection-controller.ts:563` the watchdog baseline is `connection.lastInboundAt ?? connection.startedAt`, so a reconnect gets a new timeout window instead of inheriting stale idle time. There is also a regression test for this exact case at `extensions/whatsapp/src/auto-reply.web-auto-reply.connection-and-logging.e2e.test.ts:258` (`gives a reconnected listener a fresh watchdog window`). I also checked that these files are unchanged between `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`) and current `main`. If you can still reproduce this on `v2026.4.22` or newer, please open a new issue with fresh logs from that build.
