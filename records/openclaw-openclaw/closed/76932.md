## Summary

Close as duplicate/superseded: current main still has the reported source-level guard mismatch, but open fix PRs #75525 and #69033 already track the same remaining work and regression coverage.

## What This Changes

The issue asks for the embedded-runner tool-result context guard to stop counting hidden `toolResult.details` payloads when estimating model-visible context pressure.

## Best Possible Solution

Keep one canonical fix PR, exclude hidden details from guard budgeting, and add focused regression coverage for aggregate hidden-detail pressure without exposing details to the model.

## Reproduction Assessment

Yes, source-reproducible: several tool-result messages with small visible `content` and bounded-but-nontrivial `details` can stay below the per-result truncation threshold while their aggregate hidden details push `estimateContextChars` over the guard threshold. I did not run a live embedded Pi session in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is to budget the guard from the same model-visible payload that provider conversion and compaction use, with a regression test. The existing persistence cap is useful but not sufficient because many capped detail payloads can still inflate the aggregate estimator.
