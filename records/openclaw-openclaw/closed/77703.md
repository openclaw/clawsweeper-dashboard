## Summary

Keep open. Current main still has a plausible bug path where an isolated cron agent-turn timeout can surface as a `CommandLaneTaskTimeoutError` from `cron-nested`, even though the cron service and docs expect a cron-owned timeout result after cleanup.

## What This Changes

The issue reports that a Windows npm-global OpenClaw 2026.5.4-beta.2 isolated cron run over WhatsApp stays active until `cron-nested` hits its 330000 ms command-lane timeout.

## Best Possible Solution

Make cron's timeout handling authoritative for isolated agent-turn jobs so `timeoutSeconds` records a cron timeout outcome and the `cron-nested` lane guard only releases stuck work without becoming the user-visible failure.

## Reproduction Assessment

No live current-main reproduction was run, but the source path is credible: isolated cron uses a cron-owned timeout while the embedded runner also races the same work with `timeoutMs + 30000` on `cron-nested`. The reporter's 330000 ms log matches that source-derived path.

## Solution Assessment

No, the current implementation is not the best way to solve the issue because the generic lane timeout can preempt the documented cron timeout cleanup path. A narrow fix should coordinate those timers or normalize `CommandLaneTaskTimeoutError` from cron-owned nested work into the cron timeout outcome.
