## Label Justifications

- `P1`: The report describes gateway restart killing active Windows workflows and losing session state for real users.
- `impact:data-loss`: Abrupt restart can discard task results or persisted transcript state during active work.
- `impact:message-loss`: The mid-turn restart path can drop or clear user/assistant message progress before recovery.
- `impact:session-state`: The core failure is session state not being preserved or restored across gateway restart.

## Summary

Keep open: current main has an opt-in safe restart path and restart-intent drain work, but plain Windows Scheduled Task restart is still documented as service-manager behavior and the session-manager path can still clear a restart-mid-turn file with user messages but no assistant reply.

## What This Changes

The issue requests Windows Scheduled Task gateway restart to wait for active work, write restart recovery state, and preserve sessions instead of abruptly ending tasks.

## Best Possible Solution

Settle the Windows restart contract, land the narrow session-preservation fix, and require native Windows proof that the chosen restart modes preserve active work or clearly document when they do not.

## Reproduction Assessment

No for the full native Windows scenario: I did not run a Windows Scheduled Task restart with active work. Source inspection does give a clear reproduction path for the session-reset half and a plausible force-cleanup path for plain Scheduled Task restart.

## Solution Assessment

Unclear. The opt-in --safe path is a maintainable partial solution, but making plain Windows restart drain-safe by default conflicts with the current documented compatibility behavior and needs a maintainer decision.
