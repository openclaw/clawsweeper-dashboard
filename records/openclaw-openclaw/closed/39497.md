## Summary

Current main and the latest release no longer match the filed repro: with only `workspaceAccess: "ro"`, `/agent/...` reads go through the sandbox filesystem bridge, while rejection happens only when the optional `tools.fs.workspaceOnly` guard is enabled and is now documented and tested as expected isolation.

## What This Changes

The issue asks for sandboxed read-only sessions to accept valid `/agent/...` read paths instead of rejecting them as escaping the sandbox root.

## Best Possible Solution

Keep the current split: `/agent` reads work through the sandbox filesystem bridge by default, while `tools.fs.workspaceOnly` continues to reject non-workspace mounts unless maintainers approve a separate security/product change.

## Reproduction Assessment

No. For the config shown in the issue, current main and v2026.5.7 do not enable the workspace-root guard and existing coverage expects `read /agent/secret.txt` to succeed; the rejection path is only covered when `tools.fs.workspaceOnly` is explicitly enabled.

## Solution Assessment

No, not as the suggested guard-only `/agent` remap. The safer current behavior is to let the sandbox bridge handle `/agent` by default and keep workspace-only mode restricted to the sandbox workspace directory.
