## Summary

Current main addresses the reported edit-tool failure mode well enough: exact-match edit failures now include current file contents so the agent can retry without a separate read, post-write throws are recovered when readback proves the edit applied, host and sandbox edit paths both use that wrapper, and brittle multi-hunk edits have a first-class `apply_patch` tool documented as the intended escape hatch. The memory-specific impact is also mitigated because memory-triggered flush turns are restricted to append-only `write`, not destructive `edit`/`write` replacement behavior.

## Best Possible Solution

Close this as implemented on current main. Keep the shipped design: preserve strict `edit` semantics for precise replacements, rely on `apply_patch` for brittle multi-hunk edits, and keep memory-flush writes append-only. Any future request for fuzzy/regex edit modes should be a new, narrowly scoped product/API discussion rather than another duplicate exact-match failure report.

## Risks / Open Questions

- The `edit` tool intentionally remains exact-match by default; the robust hunk workflow is `apply_patch`, and non-OpenAI providers may need explicit configuration if they do not receive that tool by default.
- Users running older releases before the shipped recovery/apply_patch behavior can still encounter the original errors until they update.
