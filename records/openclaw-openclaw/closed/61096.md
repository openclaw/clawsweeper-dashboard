## Summary

Current main already contains the fallback/session-model persistence fix this report asks for, and the changelog shows it shipped in v2026.4.10. The remaining Discord picker "Tried to set..." warning discussed on the issue is a narrower UX/path concern, not the broader override-never-persists bug as reported.

## Evidence

- **Changelog shipped fix:** The 2026.4.10 changelog explicitly says: "Models/fallback: preserve `/models` selection across transient primary-model failures and config reloads..." which matches the reported failure mode of user-selected model overrides being lost when the primary model/provider fails.
  - file: [CHANGELOG.md:933](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/CHANGELOG.md#L933)
  - sha: [59523e66da03](https://github.com/openclaw/openclaw/commit/59523e66da03591bc7bcc183a11f690fa0aae641)
- **Current main preserves user-selected overrides:** Current session-model resolution clears only transient auto-failover overrides before allowlist validation, while user-selected stored overrides continue through normal resolution; the reset path is gated behind disallowed stored overrides rather than primary-provider failures.
  - file: [src/auto-reply/reply/model-selection.ts:173](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/src/auto-reply/reply/model-selection.ts#L173)
  - sha: [59523e66da03](https://github.com/openclaw/openclaw/commit/59523e66da03591bc7bcc183a11f690fa0aae641)
- **Regression tests for persisted user override:** Current tests assert that a user-selected override survives across turns, and that legacy persisted overrides without `modelOverrideSource` are still treated as user selections rather than being clobbered.
  - file: [src/auto-reply/reply/model-selection.test.ts:723](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/src/auto-reply/reply/model-selection.test.ts#L723)
  - sha: [59523e66da03](https://github.com/openclaw/openclaw/commit/59523e66da03591bc7bcc183a11f690fa0aae641)
- **Regression tests for fallback path:** Current fallback execution tests explicitly cover the exact regression class here: fallback must not persist over a user-selected or legacy session override when the primary model fails and fallback runs.
  - file: [src/auto-reply/reply/agent-runner-execution.test.ts:2492](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/src/auto-reply/reply/agent-runner-execution.test.ts#L2492)
  - sha: [59523e66da03](https://github.com/openclaw/openclaw/commit/59523e66da03591bc7bcc183a11f690fa0aae641)
- **Current code returns explicit not-allowed errors:** If a requested model is outside the allowed set, current directive resolution returns an explicit "Model ... is not allowed" error instead of silently reporting success while not persisting the override.
  - file: [src/auto-reply/reply/model-selection-directive.ts:348](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/src/auto-reply/reply/model-selection-directive.ts#L348)
  - sha: [59523e66da03](https://github.com/openclaw/openclaw/commit/59523e66da03591bc7bcc183a11f690fa0aae641)
- **Prior maintainer verification on issue:** The issue discussion already includes a maintainer verification against then-current main stating the root cause could not be reproduced there, while calling out that the Discord picker warning might still indicate a separate UX/error-surfacing issue.
