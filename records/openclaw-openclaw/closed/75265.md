## Summary

Keep open. Current main still short-circuits both capability-provider lookup paths before consulting the active runtime registry when `plugins.enabled === false`, and this PR is a focused attempt to restore access to already-active providers while still skipping manifest/compat fallback loading. I found no discrete blocking bug in the diff, but maintainer review should confirm the intended boundary against the documented global-disable behavior and run the changed gate.

## What This Changes

The PR moves the global plugin-disable guard after active runtime registry lookup in capability provider resolution and updates two unit tests to expect already-active providers without manifest or compat loading.

## Best Possible Solution

Settle the intended contract for disabled configs, then land a narrow capability-runtime fix that either permits only already-active providers or keeps global disable as a hard stop, with regression coverage for both list and targeted provider resolution.

## Reproduction Assessment

Yes. The current-main unit-level path is clear: pass `plugins.enabled: false` with a mocked active registry and the resolver returns no providers before calling `resolveRuntimePluginRegistry()`. The reported TTS symptom follows from `resolveTtsProviderOrder` receiving an empty provider list and trying only the invalid primary provider.

## Solution Assessment

Unclear pending maintainer confirmation. The PR is the narrowest code change for restoring active-registry access without reintroducing compat/fallback loads, but the intended behavior under `plugins.enabled: false` conflicts with the documented hard-disable semantics enough to require human review.
