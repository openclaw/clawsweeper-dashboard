## Summary

Current `main` makes clear that flow controls live under `openclaw tasks flow`, not a top-level `openclaw flows` command. The original report was valid for `v2026.3.31`, but the mismatch has since been corrected in docs/changelog and reinforced by CLI tests; the fix shipped in `v2026.4.23`.

## Evidence

- **CLI wiring uses `tasks flow`, not top-level `flows`:** The CLI registers `tasks` as the parent command and nests `flow list|show|cancel` under it; there is no standalone `flows` registrar in the current implementation.
