## Summary

Current main already reconciles the reported Control UI duplicate: user sends are rendered optimistically, same-session `session.message` gateway echoes defer history reload while the run is active, and the final `chat.history` refresh replaces matching optimistic messages by role plus visible text instead of appending a second `openclaw-control-ui` user row. Existing tests cover the no-duplicate history catch-up path, and the public Control UI docs describe the canonical-transcript replacement behavior.

## Best Possible Solution

Keep the current Control UI reconciliation model: render local optimistic user messages immediately, defer same-session `session.message` history reloads while a chat run is active, and let canonical `chat.history` replace matching optimistic messages by visible role/text. Reopen only with a fresh current-main reproduction showing a new mismatch in signature, sender-label normalization, stale built assets, or browser-extension interference.
