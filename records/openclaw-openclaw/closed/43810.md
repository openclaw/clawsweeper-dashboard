## Summary

The UX goal is understandable and the screenshot proves the repeated Telegram response, but this one-line change is not safe across the shared pairing surface: it repeatedly exposes an active code and converts an existing request into a reported creation event. Maintainer approval of a bounded reminder policy is needed before implementation.

## What This Changes

The PR changes the shared pairing challenge guard so pending users receive the existing pairing-code reply again on later unauthorized messages.

## Best Possible Solution

Keep request creation one-shot, and—if maintainers approve reminders—add an explicit bounded reminder result/path that preserves `created: false`, does not invoke `onCreated`, limits redisclosure by trigger and cooldown, and has shared plus representative channel tests and docs.

## Reproduction Assessment

Yes. The source path is deterministic: an existing request returns a code with `created: false`, and the proposed guard proceeds through callback, reply delivery, and a `{ created: true }` result; the screenshot also demonstrates this in Telegram.

## Solution Assessment

No. A one-line guard change is not the best fix because it conflates creation with reminders and establishes an unconditional cross-channel credential-redisclosure policy; a separate bounded reminder path is safer and more maintainable.
