## Summary

Keep open: the slow-filesystem performance problem is real and the PR has useful live proof, but the patch can reuse cached plugin registry results across caller-supplied plugin scopes because the cache key omits candidates and install-record contents.

## What This Changes

The branch adds a 2-second in-memory coalescing cache around `loadPluginRegistrySnapshotWithMetadata` and a changelog entry for slow-filesystem gateway latency.

## Best Possible Solution

Keep the short coalescing mitigation only after every behavior-affecting input is isolated in the key, or bypass caching for caller-supplied registry scopes and leave the deeper event-loop fix to the companion async work.

## Reproduction Assessment

Yes. The PR body supplies concrete WSL2 WhatsApp and `channels.status` before/after timings, and current-main source shows `channels.status` still reaches the synchronous registry snapshot path; I did not rerun the live WSL2 scenario.

## Solution Assessment

No, not as written. A short TTL can be a reasonable narrow mitigation, but the cache must fingerprint or bypass supplied registry inputs before it is safe to merge.
