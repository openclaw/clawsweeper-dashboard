## Summary

Current `main` already handles same-provider fallback after an overloaded provider/profile cooldown by explicitly probing one cooldowned candidate instead of failing with `No available auth profile`. The reported March 2026 behavior is obsolete on `main`.

## Evidence

- **Fallback loop now probes cooldowned same-provider candidates:** `runWithModelFallback` checks whether every auth profile for a candidate provider is in cooldown and, for transient reasons like `overloaded`, passes `allowTransientCooldownProbe: true` instead of immediately skipping the candidate.
