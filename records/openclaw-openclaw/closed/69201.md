## Summary

Keep open: the current PR still has a blocking correctness issue because it compares raw `to` strings before channel-specific target parsing can prove whether two forms share the same conversation, and it also lacks required real behavior proof from the external contributor.

## What This Changes

The PR changes `mergeDeliveryContext` and unit tests so fallback `threadId` is not inherited when same-channel `to` differs or fallback has a `threadId` without `to`.

## Best Possible Solution

Preserve the stale-thread protection, but decide target mismatch using parsed/canonical channel routes or keep the guard in callers that already have loaded channel parsing, then cover both equivalent-target and true-mismatch cases.

## Reproduction Assessment

Yes, source-reproducible. Current main inherits fallback `threadId` for same-channel merges, and the PR regression is also source-reproducible because existing target-parsing tests show different raw strings can share the same conversation after parsing.

## Solution Assessment

No, not as written. The helper is a reasonable place to block true stale-thread leakage, but raw `to` inequality is too early for plugin-specific target grammars; canonical route comparison or a caller-layer guard is safer.
