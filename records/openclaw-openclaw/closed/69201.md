## Summary

Keep open. The PR targets a real source-level stale-thread risk, but the helper-level raw `to` comparison can drop valid fallback thread IDs for channel targets that parse to the same conversation, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR changes `mergeDeliveryContext` to suppress fallback `threadId` inheritance on same-channel `to` mismatch or fallback thread IDs without a `to`, with unit tests for those merge shapes.

## Best Possible Solution

Keep the stale-thread protection, but base it on parsed or canonical channel route conversation identity, or apply it only in callers that already carry route facts, with regression cases for equivalent raw targets and true mismatches.

## Reproduction Assessment

Yes, at source level. Current main inherits fallback `threadId` after only a channel-conflict check, so same-channel mismatched-target and orphan-thread fallback shapes can be traced without running a live transport.

## Solution Assessment

No, not yet. The fix belongs near delivery-context merging, but comparing raw `to` strings is less safe than comparing parsed/canonical route identity or using caller-held route facts.
