## Summary

Keep open. Current main still has the implicated SSRF policy boundary, and an open linked PR uses closing syntax for this issue; closure should wait for that PR to be reviewed, landed, or replaced, especially because the proposed fix changes SSRF allowlist semantics.

## What This Changes

The issue asks OpenClaw to allow Docker/local model-provider requests to `host.docker.internal` and other `.internal` hostnames when the provider SSRF policy is explicitly trusted.

## Best Possible Solution

Land or replace the linked PR with a narrow SSRF fix that makes the documented `request.allowPrivateNetwork: true` path work for `host.docker.internal` while preserving metadata and DNS-rebinding protections, or explicitly update the allowlist contract if maintainers choose broader trust.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source inspection reproduces the no-private-opt-in path where an auto-generated `hostnameAllowlist` does not bypass `.internal` or resolved private IP checks; the exact `allowPrivateNetwork: true` claim is still unclear because the shown config omits that nested option.

## Solution Assessment

Is this the best way to solve the issue? Unclear. The open PR fixes the demonstrated Docker flow by broadening hostname allowlist trust, but the safer maintainer path is to keep private-network access gated by explicit opt-in unless the SSRF contract is deliberately changed.
