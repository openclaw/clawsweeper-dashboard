## Summary

Current `main` already contains a generic final-reply flush fix for message-boundary/tool-heavy turns, and the changelog shows it shipped in `v2026.4.1`, which is newer than the reporter's `2026.3.28` environment.

## Evidence

- **Shipped fix in changelog:** `CHANGELOG.md` for release `2026.4.1` includes: 'Pi/TUI: flush message-boundary replies at `message_end` so turns stop looking stuck until the next nudge when the final reply was already ready.' That matches this report's missing-final-reply boundary after tool-heavy turns.
  - file: [CHANGELOG.md:1751](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/CHANGELOG.md#L1751)
  - command: `nl -ba CHANGELOG.md | sed -n '1634,1752p'`
- **Message-end flush now awaited:** Current `main` flushes buffered block replies and awaits `onBlockReplyFlush()` for `blockReplyBreak === "message_end"` before finalizing the assistant message, which directly covers short final replies that only exist at the message boundary.
  - file: [src/agents/pi-embedded-subscribe.handlers.messages.ts:836](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/src/agents/pi-embedded-subscribe.handlers.messages.ts#L836)

## Risks / Open Questions

- The WeChat runtime itself lives in the external `@tencent-weixin/openclaw-weixin` plugin, so a separate plugin-only regression could still exist even though the OpenClaw mainline delivery boundary is fixed.
- This review was code/history-based only; no live WeChat reproduction was run in this read-only environment.
