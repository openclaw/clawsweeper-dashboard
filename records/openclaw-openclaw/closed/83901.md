## Summary

This looks like a valid current-main CLI bug: the documented ACP flag is registered, but the Commander action reads the wrong option property, so the flag cannot disable the working-directory prompt prefix.

## What This Changes

The issue asks OpenClaw to make `openclaw acp --no-prefix-cwd` actually pass `prefixCwd: false` into the ACP gateway.

## Best Possible Solution

Fix the ACP CLI option registration and forwarding together: let Commander default the negated option to true, pass false only when `--no-prefix-cwd` is present, and add a focused CLI regression test.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to parse `['acp', '--no-prefix-cwd']` through `registerAcpCli` with `serveAcpGateway` mocked and observe that current source forwards `prefixCwd: true`; Commander 14.0.3’s contract explains why `opts.noPrefixCwd` is never set.

## Solution Assessment

Yes, with one correction to the report’s suggested fix. The maintainable fix should both remove the explicit `false` default from the negated Commander option and read Commander’s `prefixCwd` attribute, otherwise the default path can be inverted.
