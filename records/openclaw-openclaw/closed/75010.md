## Summary

Current main already fixes the packaged loader fallback by resolving built read-only fallback candidates from the installed package `dist/` root, with regression coverage for the exact `node_modules/openclaw/dist/read-only-*.js` case. The reported `v2026.4.27` failure is credible because that tag still used source-relative loader candidates, but the proof commit is main-only and not contained in a shipped tag yet.

## What This Changes

The issue asks for the packaged read-only plugin/channel fallback loader used by `openclaw status` and plugin checks to resolve inside the installed `openclaw/dist/plugins` tree instead of a missing sibling `plugins/loader.js` path, or to ship a compatible fallback.

## Best Possible Solution

Keep the current dist-root resolver and regression test, then ship the fix in the next stable package rather than relying on a global sibling-directory shim.

## Reproduction Assessment

Yes. The high-confidence reproduction path is the source/tag comparison: `v2026.4.27` resolves loader candidates source-relatively from a built chunk, while current main has an explicit installed-package regression test for the bad sibling `plugins/loader.js` path.

## Solution Assessment

Yes. Resolving from the built importer's `dist/` root to package-owned loader entries is the narrow maintainable fix; creating or documenting a global shim outside the package would preserve the packaging bug instead of fixing it.
