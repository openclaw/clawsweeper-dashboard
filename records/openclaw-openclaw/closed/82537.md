## Label Justifications

- `P2`: This is a concrete multi-device WebChat workflow gap with limited blast radius, but it is an additive product feature rather than an urgent regression.
- `impact:session-state`: The request is about how WebChat devices share visible transcript and active session state.
- `impact:message-loss`: The user-visible symptom is that messages and live updates from one WebChat device are not visible on another device.
- `impact:security`: Any token/operator-linked session sharing changes session visibility across authenticated browser device identities.

## Summary

Keep open. Current main and v2026.5.18 still use explicit WebChat `sessionKey` routing with browser-local session selection and per-browser device identity; the related issues cover external-channel mirroring, refresh persistence, or TUI observation, not this operator-linked cross-device WebChat session design.

## What This Changes

Request an opt-in WebChat mode where paired browsers under the same gateway token or operator can share session history and live updates across devices.

## Best Possible Solution

Define an explicit default-off shared WebChat session or device-linking contract that keeps `sessionKey` as the canonical route while specifying discovery scope, consent UI, permissions, privacy, live fanout, and upgrade behavior.

## Reproduction Assessment

Not applicable as a strict bug reproduction: this is an opt-in feature request, not a broken documented contract. Source inspection confirms current main remains explicit-session and browser-local for selection, but I did not run a live two-device scenario.

## Solution Assessment

Unclear. Token/operator-based automatic merging is not yet proven to be the safest design; the maintainable path is an explicit shared-session or device-linking contract with UI and trust-boundary rules before code changes.
