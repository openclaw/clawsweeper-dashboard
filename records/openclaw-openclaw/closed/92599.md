## Summary

Close as a duplicate/satellite: the same remaining cron stream-stall fallback problem is already tracked by the open canonical issue at https://github.com/openclaw/openclaw/issues/85900, while current main only partially addresses the timeout/fallback mechanics.

## What This Changes

The issue reports stalled model-provider connections that can outlive provider timeouts, suppress `model_call_ended`, and prevent configured fallback models from running before a cron/job timeout.

## Best Possible Solution

Keep https://github.com/openclaw/openclaw/issues/85900 as the canonical issue and fold this report's provider-specific timeout evidence into that timeout/fallback fix.

## Reproduction Assessment

Yes at source level: a cron with `payload.model`, configured fallbacks, and a provider stream that opens but never yields should exercise the same timeout/fallback race. I did not run a live stalled-provider replay in this read-only review.

## Solution Assessment

No separate issue is the best path; the maintainable solution is to resolve the timeout hierarchy and fallback behavior in the canonical open issue. Current main has partial fixes, but the remaining policy/validation work should stay consolidated.
