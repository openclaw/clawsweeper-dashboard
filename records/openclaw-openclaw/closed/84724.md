## Summary

Keep open: current main still rejects the requested self-disable path, and the remaining work is a narrow but security-sensitive expansion of the restricted cron self-scope rather than an implemented fix or duplicate cleanup target.

## What This Changes

Allow isolated cron runs with the existing restricted cron self-cleanup grant to disable their own current job while preserving the job record.

## Best Possible Solution

Add a dedicated self-disable action or a tightly constrained self-scoped update path that only applies `enabled:false` to the active cron job, with regression tests for allowed self-disable and denied cross-job or multi-field updates.

## Reproduction Assessment

Yes. Source inspection and existing tests show that a self-scoped cron tool call with `action: "update"`, the current `jobId`, and `patch: { enabled: false }` is rejected before the gateway update path runs.

## Solution Assessment

Yes, the requested direction is reasonable if maintainers approve the boundary change. The maintainable version is a narrow self-disable capability, not general self-scoped cron update access.
