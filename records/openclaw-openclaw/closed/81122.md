## Label Justifications

- `P1`: A source-proven stale Twilio provider state bug can break a real inbound voice-call workflow by sending later callers to hold instead of the media stream.
- `impact:session-state`: The issue is about stale in-memory call stream state (`activeStreamCalls`) causing the provider to make the wrong routing decision.
- `impact:message-loss`: When the later call is queued to hold music, its caller audio and transcript are not delivered to the voice-call media stream path.

## Summary

Keep open: current main and the latest release still mark a Twilio inbound call active when stream TwiML is generated, but there is no short pending-stream expiry if the media WebSocket never reaches `start`, so the next inbound call can still be sent to the hold queue.

## What This Changes

The issue requests cleanup for stale Twilio inbound stream state after a failed or no-stream call so later calls do not hear hold music.

## Best Possible Solution

Add a provider-owned pending stream activation expiry that clears only unregistered Twilio stream activations, while preserving the existing hold queue for genuinely connected active streams.

## Reproduction Assessment

Yes, source-reproducible: one inbound Twilio request can add `activeStreamCalls` when stream TwiML is returned, and if no media `start`, terminal status, or unregister callback follows, the next inbound request sees `hasActiveStreams` and returns queue TwiML. This review did not run a live Twilio call.

## Solution Assessment

Yes, the suggested pending-stream cleanup is the narrow owner-boundary fix; making queueing configurable would be a separate product choice and is not needed to fix stale no-stream state.

## Repair Work Prompt

Fix the Twilio voice-call stale active stream issue reported at https://github.com/openclaw/openclaw/issues/81122. Current main adds a call SID to `activeStreamCalls` when returning inbound stream TwiML, but if Twilio never sends a media `start` event, the call remains active until terminal status or manual cleanup and the next inbound call gets hold queue TwiML. Keep the fix inside the voice-call plugin/Twilio provider boundary. Add a bounded pending-stream activation cleanup, cancel it when `registerCallStream` runs, clear it on unregister, terminal callbacks, and hangup, and preserve queue behavior for a genuinely registered active stream. Avoid adding a new config option unless existing behavior cannot solve it. Add focused regression coverage for no-start expiry allowing the next inbound call to stream, plus coverage that true active registered streams still queue and existing terminal/unregister cleanup still works.
