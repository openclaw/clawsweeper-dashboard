## Summary

Close as implemented on current main. The remaining `core-plugin-tools` prep work now has source-backed fixes: expensive built-in media factories are gated before construction, plugin tool prompt descriptors are cached with context-sensitive keys while execution reloads the live runtime, and per-plugin timing diagnostics are present and documented; the proof commit is not in the latest tagged release yet.

## What This Changes

Reduce embedded-run `core-plugin-tools` prep latency by avoiding repeated plugin/core tool factory work, adding context-safe caching or opt-out behavior, and exposing per-plugin timing.

## Best Possible Solution

Ship the current main implementation in the next release and handle any newly isolated media-factory memoization as a narrower follow-up, such as #73039 if it still applies after rebasing.

## Reproduction Assessment

Yes. The issue has repeated v2026.4.27/v2026.4.29 traces plus controlled `disablePluginTools` comparisons, and the pre-fix request-time factory path is visible in the discussion; this review verified the current-main fix by source, docs, tests, and history rather than a live gateway trace.

## Solution Assessment

Yes. The current approach is safer than caching whole tool objects: it avoids constructing unavailable built-in factories, caches only descriptors with context-sensitive keys, and reloads the live plugin runtime at execution time.
