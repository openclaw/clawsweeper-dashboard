## Summary

Keep open: current main has adjacent prompt-mode, tool-profile, queue, and session-setting primitives, but it still lacks a user-facing per-session chat/agent mode. This is an additive core runtime/product feature rather than an implemented cleanup candidate.

## What This Changes

Request adds an opt-in session-level `/mode chat|agent` switch for lightweight conversations with reduced prompt, history, and tool exposure while preserving full agent behavior.

## Best Possible Solution

Define a core session-mode contract that consistently drives command UX, durable session state, prompt assembly, history limiting, tool policy, docs, and tests across TUI, WebChat, and messaging channels.

## Reproduction Assessment

Not applicable. This is an additive feature request, and current source/docs show no existing `/mode chat|agent` behavior to reproduce as a bug.

## Solution Assessment

Unclear until maintainer design. The requested slash-command UX is plausible, but the durable solution should be a core session/runtime contract rather than a command-only alias.
