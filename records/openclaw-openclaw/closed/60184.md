## Summary

Current main already solves the observable bug: console capture forwards through the tslog logger, tslog filters by `minLevel` before attached transports, and OpenClaw corrected the level-to-minLevel mapping in #44646. The PR's explicit `isFileLogLevelEnabled` guard is harmless but redundant against the shipped centralized logging-level fix.

## What This Changes

The PR wraps `enableConsoleCapture()` file-log forwarding in `isFileLogLevelEnabled(level)` and adds two mock exports to a cron delivery-dispatch test.

## Best Possible Solution

Keep the centralized shipped logging threshold behavior from #44646 rather than adding a console-only duplicate gate; if maintainers want follow-up cleanup, the remaining low-risk improvement would be to update the stale `enableConsoleCapture()` comment to mention that file capture still respects `logging.level`.

## Reproduction Assessment

Yes. The reported path is `logging.level: warn` plus `console.log`; current source and the tslog v4.10.2 contract show `logger.info()` is filtered below the warn `minLevel` before the attached file transport runs.

## Solution Assessment

Yes. The shipped fix is the better maintainable solution because it corrects the shared severity mapping and logger threshold behavior for all file-log call paths, while this PR only adds a redundant guard in console capture.
