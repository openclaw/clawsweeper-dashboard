## Label Justifications

- `P1`: Stale iMessage backlog can trigger real agent turns and replies after bridge recovery, which is a broken channel workflow for affected users.
- `impact:message-loss`: The issue is about stale/duplicated inbound channel messages being dispatched as fresh user requests.

## Summary

Keep open: current main added a local startup rowid watermark, but the reported remote `cliPath` bridge path still has no watermark or stale-inbound guard before dispatch.

## What This Changes

The issue reports stale iMessage live-monitor notifications being treated as fresh agent requests after Messages.app/imsg bridge recovery, with catchup disabled and a remote `cliPath` wrapper setup.

## Best Possible Solution

Add one canonical iMessage monitor recovery fence that covers remote `cliPath`/bridge-reconnect live notifications before inbound dispatch, while keeping catchup semantics and docs aligned.

## Reproduction Assessment

Yes, source-reproducible: with a remote or non-local iMessage monitor where `watchStartupRowidWatermark` is null, a valid old `watch.subscribe` notification passes the current guard and is enqueued. I did not run a live imsg bridge reproduction in this read-only review.

## Solution Assessment

No, current main is only a partial solution: the local readable-DB path has a startup `since_rowid` guard, but the reported remote wrapper path still needs a maintainer-approved stale/live recovery policy.
