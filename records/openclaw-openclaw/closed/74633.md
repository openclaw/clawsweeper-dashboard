## Summary

Current main already addresses the reported gateway hang paths: `/v1/models` is now a lightweight agent-id response, the reported `/v1/chat/completions` request no longer loads the model catalog unless an explicit `x-openclaw-model` override is supplied, stale model catalog refreshes serve the last good catalog in the background, local discovery advertisement is startup-bounded, and Bonjour/Telegram stall recovery is covered in code and tests. The changelog and merged related PR explicitly reference this report; the proof is on main and not in the latest provided release tag.

## What This Changes

The issue reports that a healthy local gateway can hang on `/v1/*` requests while Bonjour, Telegram polling, and model-pricing/model-catalog work time out.

## Best Possible Solution

Keep the current main implementation: `/v1/models` should remain a lightweight OpenClaw agent catalog, `/v1/chat/completions` should avoid model-catalog discovery unless an explicit provider/model override requires it, discovery plugins should stay startup-bounded, and channel/discovery background loops should recover or disable themselves without blocking gateway HTTP responsiveness.

## Reproduction Assessment

Yes. The issue provides a concrete cURL/log reproduction for 2026.4.26, and current-main inspection maps that reproduction to the `/v1/models`, `/v1/chat/completions`, discovery, Bonjour, Telegram, and model-pricing paths; I did not run the live macOS reproduction in this read-only pass.

## Solution Assessment

Yes. The current fix is a maintainable narrow split: keep `/v1` control-plane paths off expensive discovery work, serve cached model catalogs during stale refreshes, and bound or recover the optional background subsystems instead of adding a broad gateway-wide workaround.
