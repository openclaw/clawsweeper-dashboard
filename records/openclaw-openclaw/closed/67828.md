## Summary

Current `main` already routes `openclaw cron edit` through the cron service update path, recomputes `state.nextRunAtMs` when `schedule` changes, and has an explicit regression test covering rescheduled `at` jobs.

## Evidence

- **CLI edit path reaches cron service update:** `cron.update` in the gateway validates the patch and then calls `context.cron.update(jobId, patch)`, so `openclaw cron edit` uses the service update logic rather than mutating persisted state directly.
  - file: [src/gateway/server-methods/cron.ts:250](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/src/gateway/server-methods/cron.ts#L250)
  - command: `nl -ba src/gateway/server-methods/cron.ts | sed -n '250,327p'`
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **Schedule edits recompute cached next-run state:** `update()` sets `job.state.nextRunAtMs = computeJobNextRunAtMs(job, now)` whenever `patch.schedule` or `patch.enabled` is present, which directly addresses the stale cached `nextRunAtMs` failure described in the issue.
  - file: [src/cron/service/ops.ts:306](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/src/cron/service/ops.ts#L306)
  - command: `nl -ba src/cron/service/ops.ts | sed -n '283,320p'`
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **At-type jobs are explicitly re-armed after reschedule:** `computeJobNextRunAtMs()` returns the new `at` timestamp for a completed one-shot job when the edited `schedule.at` is later than `state.lastRunAtMs`, instead of treating the old run as final.
  - file: [src/cron/service/jobs.ts:277](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/src/cron/service/jobs.ts#L277)
  - command: `nl -ba src/cron/service/jobs.ts | sed -n '277,305p'`
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **Regression test matches this bug class:** `service.issue-19676-at-reschedule.test.ts` asserts that a completed `at` job returns the new timestamp after rescheduling and does not stay tied to the stale cached time.
  - file: [src/cron/service.issue-19676-at-reschedule.test.ts:27](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/src/cron/service.issue-19676-at-reschedule.test.ts#L27)
  - command: `nl -ba src/cron/service.issue-19676-at-reschedule.test.ts | sed -n '1,89p'`
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **Runtime state now lives in a sidecar file:** The user docs say cron definitions stay in `jobs.json` while runtime execution state, including `nextRunAtMs`, persists in `jobs-state.json`, which matches the current storage model and helps explain why direct `jobs.json` edits are no longer the supported state-mutation path.
