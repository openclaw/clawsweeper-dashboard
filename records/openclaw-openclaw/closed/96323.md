## Summary

Keep this PR open, but it is not merge-ready: the hardening direction is useful, yet the current patch can reject valid OpenAI image and embedding-batch payloads, leaves the OAuth first body read unbounded, and still has only unit-test proof.

## What This Changes

The PR routes several OpenAI plugin success response reads through `readResponseWithLimit` with 1 MiB caps and adds OAuth/realtime bounded-read unit tests plus Response mock updates.

## Best Possible Solution

Move bounded reads to the first body read in each path, use payload-specific caps that preserve valid image and batch outputs, and add real redacted proof for both overflow rejection and normal success.

## Reproduction Assessment

Yes. Source inspection shows the PR head applies 1 MiB caps to image and batch file-content success bodies while the current code carries large base64 image and batch output payloads, and OAuth still prebuffers before the changed parse.

## Solution Assessment

No. `readResponseWithLimit` is the right primitive, but this patch applies it at the wrong OAuth layer and uses one-size-fits-all caps that are too small for valid payload classes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Raise the image response cap for real payloads:** `extensions/openai/image-generation-provider.ts:84`
  - body: This adds a 1 MiB cap to `/images/generations` and `/images/edits` success JSON before parsing, but this path can return base64 image data and supports large or multiple images. Valid generated image responses can exceed 1 MiB, so use an image-payload-sized cap or another bounded parsing strategy that preserves valid outputs.
  - confidence: 0.94
- **[P1] Do not cap batch output files at one MiB:** `extensions/openai/embedding-batch.ts:117-121`
  - body: `fetchOpenAiFileContent` reads completed batch output and error files, not only small metadata. Existing batch settings allow large request groups, so a 1 MiB cap on file content can reject valid embedding batch results; use a separate larger output-file cap from the small status JSON cap.
  - confidence: 0.93
- **[P2] Bound OAuth bodies before postTokenForm buffers them:** `extensions/openai/openai-chatgpt-oauth-flow.runtime.ts:229-233`
  - body: `postTokenForm` already reads the token endpoint body with `response.arrayBuffer()` before this bounded parse runs, so the untrusted body is fully buffered first. Move the byte cap into the first read in `postTokenForm` and cover exchange, refresh, and non-OK token responses consistently.
  - confidence: 0.91
