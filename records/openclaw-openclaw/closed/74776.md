## Summary

Current main and the shipped v2026.4.27 release already address the renamed gateway-startup problem: primary-model prewarm is bounded and detached from channel startup, and provider catalog discovery now has startup timeout handling. The reporter was on 2026.4.26, so the best cleanup outcome is to close this as fixed by the newer release while treating any remaining first-message memory-search stall as a separate, narrower report.

## What This Changes

The issue asks for gateway channel startup to stop waiting several minutes on `prewarmConfiguredPrimaryModel` / `ensureOpenClawModelsJson` provider discovery after restart.

## Best Possible Solution

Keep the v2026.4.27/current-main startup behavior. If the reporter still sees first-message `memory_search` or local embedding event-loop stalls after upgrading, that should be tracked as a separate memory/provider cold-start issue with `openclaw memory status --deep --json` and channel logs.

## Reproduction Assessment

Yes for the reported prewarm/channel-startup block: the issue gives a concrete restart timeline, and current tests reproduce the failure mode with a hanging primary-model prewarm while asserting channels still start. A live Windows/Copilot-enterprise reproduction was not run for this read-only sweep.

## Solution Assessment

Yes. Detaching startup prewarm from channel startup and bounding provider discovery is the narrowest maintainable fix for the renamed issue because it preserves warmup benefits without making channel readiness depend on provider catalog latency.
