## Summary

Close as superseded. The linked VPS/LAN report is already closed after manual approval guidance, and current main has a narrower merged fix for the native approvals stale-baseline case, while this PR’s broader change would weaken the intended scope-upgrade approval boundary.

## What This Changes

The PR removes the `scope-upgrade` special case that forces device pairing requests to be non-silent in `src/gateway/server/ws-connection/message-handler.ts`.

## Best Possible Solution

Keep the merged narrow backend/native-approval path and the explicit scope-upgrade approval boundary. Any still-reproducible VPS/LAN failure should come back as a focused report or PR that proves a trusted device-less backend path is blocked without allowing explicit paired devices, browsers, nodes, or remote clients to widen scopes silently.

## Reproduction Assessment

Yes. The source issue had concrete 2026.4.25 LAN-bind steps, and current main has focused regression tests for the sensitive path: read-scoped paired shared-auth and backend clients reconnect requesting `operator.admin`, receive pairing-required failures, and keep the stored approved scope baseline unchanged.

## Solution Assessment

No. This PR is not the best way to solve the report because it removes the global scope-upgrade guard; the safer solution is the already-merged narrow local shared-auth/native-approval handling plus explicit approval for real paired-device scope upgrades.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Preserve explicit approval for scope upgrades:** `src/gateway/server/ws-connection/message-handler.ts:980-982`
  - body: With this change, local/shared-secret `scope-upgrade` requests become silent because `shouldAllowSilentLocalPairing` returns true for that reason.   - confidence: 0.93
