## Summary

Close as implemented on main: current active-memory no longer feeds colon-containing scoped conversation IDs into bundled plugin public-surface dirName resolution, and the guard is present in the v2026.5.12 release after the reported 2026.5.2 build.

## What This Changes

The issue reports that Slack thread session keys with colon-delimited thread components crash memory-core or active-memory when used as bundled plugin directory names.

## Best Possible Solution

Keep the generic active-memory provider/channel separation and direct users on 2026.5.2 to upgrade to v2026.5.12 or newer; no Slack-specific dirName sanitizer is needed.

## Reproduction Assessment

No for current main; the old-version failure path is credible from the logs, but current source and the v2026.5.12 release route colon-containing scoped channel IDs through the provider instead of bundled dirName resolution.

## Solution Assessment

Yes; the current generic scoped-channel guard is the maintainable fix because it preserves strict bundled dirName validation while keeping provider identity separate from conversation IDs.
