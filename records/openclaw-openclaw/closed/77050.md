## Summary

Keep open: current main still leaves `file:` config env values literal, and this active PR is the linked implementation candidate, but it is not mergeable as-is because it adds a new config-secret shorthand with unsafe direct file reads and lacks real behavior proof.

## What This Changes

The PR adds `file:<relpath>` handling to config env var application and a new test file covering file content, newline trimming, missing files, traversal, and existing-env precedence.

## Best Possible Solution

If maintainers approve the shorthand, implement it through SecretRef-grade guarded file reading and document the config contract; otherwise direct users to structured SecretRefs or process env.

## Reproduction Assessment

Yes, at source level: current main copies config env strings literally after existing-env and `${VAR}` checks, so `file:secrets/xai-api-key.txt` remains literal. A live xAI call is not needed to prove that config-path behavior.

## Solution Assessment

No: the PR demonstrates the requested shorthand, but the direct synchronous file read is not the safest maintainable implementation for secret-bearing config. The safer path is to reuse or mirror the existing SecretRef file-provider guardrails after maintainers decide the shorthand should exist.
