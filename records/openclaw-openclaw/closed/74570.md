## Summary

Close as implemented on current main. The managed package-update path now refreshes service metadata from the updated install, restarts through the updated entrypoint, verifies the running Gateway reports the expected post-update version, and fails the update instead of reporting success when an old Gateway remains active. Current main also includes runtime-deps mirroring for root bare imports and targeted coverage for the chokidar/memory-core class of failure.

## What This Changes

The issue reports a v2026.4.23 to v2026.4.26 package update where the CLI updates but the managed Gateway stays on v2026.4.23, causing plugin compatibility rejection and missing runtime-deps failures.

## Best Possible Solution

Keep the current main behavior and ship it in the next release: package updates should fail loudly on Gateway version skew, and packaged runtime-deps mirrors should stay covered by metadata plus tests so memory/plugin runtime imports resolve after updates.
