## Summary

Close as implemented on current main. The requested user-visible behavior is now covered by the existing `silentReply` policy surface: operators can opt direct/DM conversations into silence globally or per surface, and current main routes clean empty or reasoning-only direct turns to silent `NO_REPLY` instead of the visible incomplete-turn fallback.

## What This Changes

Request a global or per-channel config option that silently suppresses empty-turn fallback delivery instead of sending `Agent couldn't generate a response` to the channel.

## Best Possible Solution

Keep the current `silentReply`-based implementation as the supported configuration path, documenting `agents.defaults.silentReply.direct = "allow"` for global direct silence and `surfaces.signal.silentReply.direct = "allow"` for Signal-specific direct/TTS deployments.

## Reproduction Assessment

Yes. The issue can be reproduced as a direct/DM auto-reply run with `silentReply.direct = "allow"` and a clean empty or reasoning-only assistant turn; current tests now cover the flag propagation and resulting `NO_REPLY` terminal payload.

## Solution Assessment

Yes. Reusing `silentReply` is the narrowest maintainable solution because it preserves conservative direct-chat defaults, supports global and per-surface overrides, and avoids adding a parallel incomplete-turn-only config surface.
