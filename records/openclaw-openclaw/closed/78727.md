## What This Changes

The issue reports that CLI/MCP device-pairing scope upgrades can rotate an operator token down to only the pending scopes, causing repeated scope-upgrade approval requests.

## Best Possible Solution

Keep the generic pairing/token issuance fix on main; handle any remaining scope-mismatch diagnostic wording through the narrower diagnostics thread rather than reopening the MCP-specific loop report.

Yes, at source/history level: the issue includes detailed runtime logs, and the affected approval/token path was identifiable and is now covered by focused regression tests. I did not establish a failing current-main reproduction because current main contains the fix.
