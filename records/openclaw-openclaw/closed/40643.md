## Summary

Current main already prevents the assistant-error replay poisoning scenario this PR targets. Instead of only collapsing consecutive assistant `stopReason: "error"` entries, main drops failed assistant turns (`error` or `aborted`) from transport replay for OpenAI/Anthropic paths and retries empty zero-output error turns before they can accumulate. The extra routed-Mistral hunk in the stale PR targets an older file that no longer exists and should be re-filed separately if still reproducible against the current provider replay-policy architecture.

## Best Possible Solution

Close this stale PR and keep the shipped main implementation: failed assistant transport turns are dropped before strict-provider replay, and empty zero-output error turns are retried before they accumulate. Any remaining routed-Mistral strict9 behavior should be handled as a separate focused issue or PR against the current plugin replay-policy files, not by reviving this older mixed-scope branch.
