## Summary

Close as implemented on current main: the pre-listen `plugins.bootstrap` path no longer loads plugin runtime/dependency artifacts, and the reporter's current-main rebuild confirmation matches the source and fix history.

## What This Changes

The issue reports a v2026.4.26 Docker gateway startup hang at `plugins.bootstrap` before HTTP binding and readiness.

## Best Possible Solution

Keep the current main implementation and let the next release carry it to published Docker users.

## Reproduction Assessment

Yes for the original report: the issue provides a deterministic v2026.4.26 Docker force-recreate path with startup trace ending after `control-ui.seed` before `plugins.bootstrap` completes. I did not need a local Docker rerun because current source and the reporter's main rebuild both show the pre-listen path has changed.

## Solution Assessment

Yes. Moving heavy plugin runtime/dependency work out of pre-listen `plugins.bootstrap` and into the post-bind startup path is the narrowest maintainable fix for the observed no-listener hang; generic timeout hardening could be tracked separately if it becomes a distinct report.
