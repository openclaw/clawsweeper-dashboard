## Summary

Current main already forces the WebChat thread to scroll when a pending send is enqueued and has an E2E check for the delayed-ACK case that matches the reported banner-only behavior; the fix is current-main-only and not in v2026.5.28.

## What This Changes

The issue reports that Windows WebChat does not auto-scroll to the newest message after sending and instead shows the New messages banner until clicked.

## Best Possible Solution

Keep the current-main pending-send scroll behavior and let the next release carry it; use the related open PR only for its separate manual-scroll interrupt hardening.

## Reproduction Assessment

No; I did not establish a failing current-main Windows reproduction. The stable-release report is consistent with the older enqueue path, but current main has source and E2E coverage for the after-send scroll behavior.

## Solution Assessment

Yes; the current-main path fixes this at the shared WebChat send boundary by scrolling the pending local message immediately and preserving the post-ACK scroll, without adding Windows-specific branching or config.
