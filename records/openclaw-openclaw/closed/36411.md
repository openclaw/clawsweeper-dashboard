## Summary

Current `main` already prevents the reported gateway restart storm from invalid config, including an unsupported `agents.defaults.memorySearch.provider`. Startup now restores the last-known-good config when the active config is invalid, and unrecoverable config errors exit with code 78 so the systemd unit does not keep restarting the gateway. The repo also now documents `openclaw config validate` as the preflight path and lists the supported memory-search providers.

## Risks / Open Questions

- Auto-recovery depends on a usable last-known-good backup; if none exists, startup still fails, but current main exits with code 78 so systemd should not loop.
- The supported-provider list is documented in schema help/docs rather than enforced as a zod enum at the `memorySearch.provider` field itself.

## Close Comment

Closing from a Codex review because current `main` already addresses the reported failure mode well enough. Invalid gateway config is now restored from the last-known-good backup during startup (`src/gateway/server-startup-config.ts:72` and `src/cli/gateway-cli/run.ts:261`), and unrecoverable config failures exit with code 78, which the systemd unit marks as `RestartPreventExitStatus=78` (`src/cli/gateway-cli/run.ts:110`, `src/daemon/systemd-unit.ts:79`) so the service should not keep restarting in a loop. There is also regression coverage for the startup recovery path (`src/gateway/server-startup-config.recovery.test.ts:57`), and the docs now point operators to `openclaw config validate` plus last-known-good recovery guidance (`docs/cli/config.md:383`, `docs/gateway/troubleshooting.md:278`). Supported `memorySearch.provider` values are documented in `src/config/schema.help.ts:937`. This appears fixed on `main` at `3361593442e295e9bb1f55cbfb38861ea9413e45`; I could not tie it to a released tag yet.
