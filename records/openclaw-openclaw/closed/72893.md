## Summary

Keep open. Current main still matches the reported failure path: the DeepSeek provider wraps V4 thinking payloads but does not expose a provider `resolveThinkingProfile`, so `xhigh` and `max` are rejected by thinking-level validation before the DeepSeek stream wrapper can map them. Open PR #73008 references this issue with closing syntax and remains the active implementation path.

## What This Changes

Allow native DeepSeek V4 Flash/Pro models to accept `/think xhigh` and `/think max` by exposing a provider-owned thinking profile instead of falling back to the base thinking levels.

## Best Possible Solution

Review and land PR #73008, or an equivalent DeepSeek-local patch, so only `deepseek-v4-flash` and `deepseek-v4-pro` expose `off|minimal|low|medium|high|xhigh|max` while older DeepSeek models keep their existing profile.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main lacks the DeepSeek provider profile hook, so `listThinkingLevels("deepseek", "deepseek-v4-pro")` follows the base profile path before request construction.

## Solution Assessment

Yes. A provider-local `resolveThinkingProfile` plus focused DeepSeek regression coverage is the narrowest maintainable fix; changing the core thinking resolver would broaden provider-specific model policy unnecessarily.
