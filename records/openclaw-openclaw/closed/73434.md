## Summary

Close: current main has a shipped bounded-drain fix for the production deadlock path this PR targets, and the changelog explicitly marks this PR as superseded. The remaining timeout layer in this branch should not be merged as-is because it introduces a 60-second adapter-level default that conflicts with existing long-running tool contracts.

## What This Changes

The PR changes transcript replay defaults, adds a configurable adapter-level per-tool-call timeout, threads that setting through embedded runner tool splitting, and adds config schema plus agent-runtime tests.

## Best Possible Solution

Keep the shipped bounded pending-tool-task drain as the accepted fix for the stuck-session path, close this superseded branch, and handle any future per-tool timeout work as a narrower opt-in/tool-aware change with aligned docs and tests.

## Reproduction Assessment

Partly: the linked lost-tool-result reports provide concrete stuck-session traces, and current main can be checked by reading the bounded pendingToolTasks drain path. The PR-introduced timeout regression is reproducible with a foreground adapted tool that legitimately runs longer than 60 seconds while agents.defaults.toolCallTimeoutSeconds is unset.

## Solution Assessment

No for this PR branch: the shipped progress-aware pending-task drain is the narrower accepted solution for the deadlock path, while the proposed global adapter timeout changes existing tool behavior. A safer timeout solution would preserve tool-specific contracts and make any new cap explicit or tool-aware.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve existing long-running tool timeouts:** `src/agents/pi-tool-definition-adapter.ts:295-299`
  - body: This default applies a 60s timeout to every adapted tool when users have not configured `agents.defaults.toolCallTimeoutSeconds`. Existing tools such as `exec` already expose their own timeout contract (`tools.exec.timeoutSec` defaults to 1800s and per-call `timeout: 0` disables it), so foreground commands and other long-running tools can now be aborted long before their documented timeout. Make the adapter timeout opt-in or tool-aware instead of changing the default for every tool.
  - confidence: 0.86
