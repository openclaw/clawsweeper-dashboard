## Summary

Keep open: current main still omits the implicit Telegram default account from account enumeration once a named account is present, and gateway startup/status depend on that enumeration. Earlier matching reports were closed stale rather than actively tracking the remaining current-main bug.

## What This Changes

The issue asks OpenClaw to preserve or loudly warn about the implicit Telegram default account when a named Telegram account is added.

## Best Possible Solution

Keep the implicit default account in Telegram account enumeration whenever top-level/default Telegram credentials or config are present, and add focused regression coverage for account listing, gateway startup, and status output with a named account.

## Reproduction Assessment

Yes, as a source-level reproduction: with top-level `channels.telegram.botToken` plus `channels.telegram.accounts.fusion`, `listTelegramAccountIds()` returns only the named account unless `default` is explicitly configured or bound. Gateway startup and status then iterate that list, so the default account is not started or shown.

## Solution Assessment

Yes. The narrow maintainable fix is to correct Telegram account selection or default promotion so existing default-account semantics are preserved; warning-only documentation would be weaker because the CLI already preserves the default when adding a non-default account.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/82780. Current Telegram account enumeration drops the implicit `default` account when `channels.telegram.accounts.<name>` exists, even if top-level `channels.telegram.botToken`/default config is present; gateway startup and status use that list, so the default bot silently stops polling. Preserve existing CLI/default-account semantics: adding a named account must not remove the default polling account, and explicit account ids/defaultAccount routing must remain deterministic. Likely fix in `extensions/telegram/src/account-selection.ts` and possibly shared account helpers only if the same contract is intentionally shared; avoid adding a config option or weakening explicit unknown-account fail-closed behavior. Add focused regression tests for `listTelegramAccountIds`, enabled-account listing, and gateway/status behavior with top-level Telegram default config plus a named account. Related context: https://github.com/openclaw/openclaw/issues/26681 and https://github.com/openclaw/openclaw/issues/16652 were prior stale/closed reports of the same symptom; https://github.com/openclaw/openclaw/issues/61012 is adjacent default-account routing work but not the same tracker. Add a changelog entry because this is user-visible Telegram behavior.
