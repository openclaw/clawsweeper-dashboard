## Summary

Keep open. Current main still matches the reported authorization bug: configured `tools.exec` is converted into implicit `exec`/`process` profile allowances and then merged with explicit profile `alsoAllow`, so a restricted messaging agent can receive runtime tools unless a separate deny rule is configured. The focused fix PR #47991 is closed unmerged, while #42165 is an adjacent filesystem analogue and #69190 is broader rather than a canonical replacement.

## What This Changes

Ensure restricted agent tool profiles with an explicit `alsoAllow` list that excludes `exec` do not receive `exec` or `process` just because `tools.exec` is configured globally.

## Best Possible Solution

Revise the tool-policy contract so global `tools.exec` configuration does not widen restricted agent profiles, preserve deny precedence, add focused regression coverage for `messaging` plus explicit `alsoAllow`, document global-versus-agent semantics, and handle the `tools.fs` analogue consistently.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure global `tools.exec`, set an agent to `tools.profile: "messaging"` with `alsoAllow` excluding `exec`, and `resolveEffectiveToolPolicy()` still returns `profileAlsoAllow` containing `exec` and `process`.

## Solution Assessment

Unclear until maintainers settle the global-versus-agent semantics, but current main is not the best solution because it conflates exec safety configuration with exec access. The narrow maintainable direction is to require explicit access opt-in for restricted profiles while keeping deny precedence and documenting any intentional agent-level exception.
