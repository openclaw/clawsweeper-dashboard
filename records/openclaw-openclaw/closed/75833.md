## Summary

Close as implemented on current main. The reported failing path is the old `plugin-runtime-deps` layout; current source only retains that name in cleanup and guardrail paths, while WhatsApp QR rendering now goes through the core media runtime where `qrcode` is a declared OpenClaw runtime dependency. The proof is on `main`, but I did not find a release tag containing it yet.

## What This Changes

The issue reports a regression where WhatsApp QR pairing fails on 2026.4.29 because QR rendering imports `qrcode` from a generated `plugin-runtime-deps` path that cannot resolve the package.

## Best Possible Solution

Keep the current main approach: QR rendering should stay in the shared media runtime with `qrcode` supplied by the OpenClaw package, and old generated plugin dependency roots should be cleaned as legacy state rather than revived as a runtime loading path.

## Reproduction Assessment

Yes. The issue includes exact onboarding/UI steps and a concrete Node module-resolution error; source inspection maps that error to the WhatsApp QR rendering path and the legacy runtime-deps package layout.

## Solution Assessment

Yes. The current solution is the narrowest maintainable direction because it removes reliance on generated `plugin-runtime-deps` roots and keeps QR dependencies with the core media runtime instead of duplicating SDK-owned dependencies in the WhatsApp plugin.
