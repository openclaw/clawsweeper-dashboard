## Summary

Close as implemented. Current main now stops bare `/reset` and `/new` after reset hooks acknowledge the command, so the non-ACP command path used by Telegram cannot continue into an empty-message provider call; the regression is covered by tests and documented in `v2026.4.29`.

## What This Changes

Prevent bare `/reset` and `/new` on an already-fresh session from reaching an empty-message provider call after reset lifecycle hooks.

## Best Possible Solution

Keep the shipped command-stop behavior and leave deeper lifecycle contract changes to a separate issue only if a model-backed plugin can still reproduce an empty-message hook call after `v2026.4.29`.

## Reproduction Assessment

Yes. The issue gives concrete Telegram steps and provider logs, and current-main regression tests now exercise the relevant bare `/reset` and `/new` command path that previously could continue to model execution.

## Solution Assessment

Yes. Returning `shouldContinue: false` for bare reset commands is the narrowest maintainable fix because it preserves hook emission and `/reset <message>` seeding while blocking the empty provider call.
