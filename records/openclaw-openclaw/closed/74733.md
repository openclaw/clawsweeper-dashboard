## Summary

Keep open. Current main still has a source-reproducible WebChat race where an older `chat.history` response can apply against the request-start message snapshot and clear active stream state after a local send; this PR’s code/test direction is narrowly aimed at that bug, but it still needs convincing real WebChat race proof and maintainer sequencing with overlapping WebChat reliability PRs.

## What This Changes

The PR updates Control UI/WebChat history reconciliation to preserve optimistic user sends and active stream/tool state across stale `chat.history` responses, adds reset cleanup coverage, and touches a temp filename expression plus changelog text.

## Best Possible Solution

Keep the focused stale-history reconciliation fix alive, rework it onto the current run-lifecycle helpers, require real WebChat race proof, then land it or explicitly supersede it with the accepted broader WebChat reliability path.

## Reproduction Assessment

Yes. Source inspection on current main shows an older `loadChatHistory` request can snapshot pre-send messages, then a local send appends an optimistic message and starts a stream before the stale history response applies and clears stream state.

## Solution Assessment

Yes for the stated stale-history race: tracking local mutations and reconciling against the latest visible chat state is a narrow maintainable fix. The reset cleanup part should be adapted to current main’s shared `reconcileChatRunLifecycle` path rather than kept as an older direct reset pattern.
