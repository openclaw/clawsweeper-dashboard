## Summary

Close as superseded. The same stuck-session abort work is already tracked by the open canonical bug and earlier active implementation PR, while this PR’s smaller heartbeat-side queue-clear approach has blocking correctness issues and should not become the parallel implementation path.

## What This Changes

The PR adds `diagnostics.stuckSessionAbortMs` config/schema/reload handling plus heartbeat logic and tests that clear a session command-lane queue and reset diagnostic state after a processing session exceeds the configured threshold.

## Best Possible Solution

Close this duplicate PR and keep implementation focused on #71127 plus #73243 or a maintainer-approved replacement that adds a conservative abort threshold through the established recovery runtime, releases lanes through the reset/drain path, and aligns telemetry, config docs, changelog, and tests.

## Reproduction Assessment

Yes. The current-main gap is reproducible by source inspection and existing tests: active work remains observe-only by default, and `clearCommandLane` is proven to reject only pending entries while `resetCommandLane` releases the active lane slot.

## Solution Assessment

No. Directly clearing a session lane queue from the heartbeat is not the best fix because it bypasses the recovery runtime and leaves active lane work held; the safer path is to extend `recoverStuckDiagnosticSession` with explicit abort-threshold semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Route aborts through the lane reset path:** `src/logging/diagnostic.ts:678-680`
  - body: `clearCommandLane` only rejects queued entries and leaves the active task slot occupied. With a session lane concurrency of 1, diagnostics can mark the session idle while the stuck active task still blocks later work, so recovery is reported without actually freeing the lane.
  - confidence: 0.91
- **[P2] Evaluate the abort threshold independently:** `src/logging/diagnostic.ts:676`
  - body: The abort check is nested under the warning-threshold branch, so a valid abort threshold below `stuckSessionWarnMs` cannot fire at the configured time. Either enforce the ordering during config resolution or check abort timing separately from warning emission.
  - confidence: 0.87
