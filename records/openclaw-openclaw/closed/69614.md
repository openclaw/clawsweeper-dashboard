## Summary

Close as `clawhub`: the report targets the external DingTalk channel plugin's inbound stream path, not OpenClaw core on current `main`. Current docs classify DingTalk as a community plugin, channel inbound handling is explicitly plugin-owned, and the repo has no bundled DingTalk implementation to patch in core.

## Evidence

- **DingTalk is a community plugin:** Current docs list DingTalk under community plugins, with npm package `@largezhou/ddingtalk` and repo `github.com/largezhou/openclaw-dingtalk`.
  - file: [docs/plugins/community.md:52](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/docs/plugins/community.md#L52)
  - sha: [5fe333ada85a](https://github.com/openclaw/openclaw/commit/5fe333ada85afd0def9e85985ce57e36aa00108a)
- **Inbound handling is plugin-owned:** The channel-plugin guide says inbound message handling is channel-specific and each channel plugin owns its own inbound pipeline. That matches this report's claimed failure mode: dropped DingTalk stream/image callbacks in rapid succession.
  - file: [docs/plugins/sdk-channel-plugins.md:560](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/docs/plugins/sdk-channel-plugins.md#L560)
  - sha: [5fe333ada85a](https://github.com/openclaw/openclaw/commit/5fe333ada85afd0def9e85985ce57e36aa00108a)
- **Project scope points optional channels to plugins/ClawHub:** VISION says core should stay lean, optional capability should usually ship as plugins, plugin packages should be hosted in their own repositories, and plugin promotion belongs in ClawHub.
  - file: [VISION.md:52](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/VISION.md#L52)
  - sha: [5fe333ada85a](https://github.com/openclaw/openclaw/commit/5fe333ada85afd0def9e85985ce57e36aa00108a)
- **No bundled DingTalk implementation on main:** A tracked-file search for `dingtalk` / `ddingtalk` returned no bundled implementation in the OpenClaw checkout, which is consistent with DingTalk being external rather than a core-maintained channel in this repo.
  - command: `git ls-files | rg "dingtalk|ddingtalk"`
  - sha: [5fe333ada85a](https://github.com/openclaw/openclaw/commit/5fe333ada85afd0def9e85985ce57e36aa00108a)

## Risks / Open Questions

- If the loss is ultimately caused by a shared core ingress/media path rather than the DingTalk plugin's own inbound stream handler, closing to the plugin would be premature. I did not find evidence of a cross-channel core regression on current `main`.
