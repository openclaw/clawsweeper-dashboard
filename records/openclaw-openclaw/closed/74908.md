## Summary

Current main implements the reported Active Memory outer hook timeout problem: the plugin now registers `before_prompt_build` with a timeout derived from its own configurable `timeoutMs`, and the hook runner prefers that handler timeout over the default 15s cap. The fix is present on main after the latest release, so this can close as implemented-on-main rather than shipped.

## What This Changes

Request to make the outer `before_prompt_build` modifying-hook timeout configurable so Active Memory can exceed the default 15s hook cap.

## Best Possible Solution

Keep the plugin-owned implementation: Active Memory should use its own timeout config to set both the recall budget and the registered prompt-hook budget, while the generic hook runner keeps a conservative default for plugins that do not opt into a longer handler budget.

## Reproduction Assessment

Yes. The reported cap can be checked by comparing v2026.4.27, where Active Memory registered `before_prompt_build` without hook options, against current main, where a configured 90000ms Active Memory timeout registers a 120000ms hook cap and tests assert that behavior.

## Solution Assessment

Yes. For the Active Memory problem described here, deriving the hook-runner budget from the Active Memory plugin config is the narrowest maintainable fix and avoids adding a broader core config API before multiple plugins need it.
