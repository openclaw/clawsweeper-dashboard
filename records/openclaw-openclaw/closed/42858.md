## Summary

Keep open. Current main still does not map or alias `bailian` to Qwen/Model Studio auth candidates, so env-backed auth for a `bailian` provider can still miss `BAILIAN_API_KEY`; the linked open fix PR should be resolved before cleanup.

## What This Changes

The issue asks OpenClaw to resolve Bailian provider API keys from `BAILIAN_API_KEY` or `MODELSTUDIO_API_KEY` when users configure env-backed auth.

## Best Possible Solution

Resolve the linked implementation path with a provider-owned manifest alias/env metadata change and regression coverage, or explicitly document migration away from `bailian`.

## Reproduction Assessment

Yes. Source inspection on current main shows `resolveEnvApiKey("bailian")` normalizes to `bailian`, finds no candidate or alias entry, and no current parser supports the reported `__env__:` marker; I did not run a live Bailian request.

## Solution Assessment

No. The one-line core-map change proposed in the issue is not the best current shape; a provider-owned manifest auth alias/env declaration with a regression test is safer, unless maintainers choose explicit migration docs instead.
