## Summary

Current main already implements the requested behavior: plugin install metadata is committed to the managed plugin index, and isolated single-file `plugins` includes are written through to the include target with regression coverage.

## What This Changes

Make `openclaw plugins install` persist plugin install metadata when the `plugins` config section is loaded via a single-file `$include`.

## Best Possible Solution

Keep the managed plugin-index design plus single-file include write-through; any remaining report should focus on a stale or mismatched installed package artifact rather than changing the current source path.

## Reproduction Assessment

No for current main: source inspection shows the isolated `plugins` include path is intercepted before the root writer, and the regression test covers the reported config shape. I did not establish a failing current-main path.

## Solution Assessment

Yes. The current implementation is the narrow maintainable fix because install records live in machine-managed plugin index state while user-authored single-file `plugins` includes receive only the config entry changes.
