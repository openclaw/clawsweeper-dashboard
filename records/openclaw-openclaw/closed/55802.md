## Summary

Current `main` already resolves the reported long-command approval problem by removing the heuristic host-exec obfuscation gate entirely, rather than adding a configurable `tools.exec.maxCommandChars` threshold.

## Evidence

- **Gateway host exec no longer applies an obfuscation-length prefilter:** `processGatewayAllowlist()` decides approval from allowlist analysis, heredoc handling in allowlist mode, and optional strict inline-eval only; there is no max-command-length heuristic or obfuscation detector in this path.
  - file: [src/agents/bash-tools.exec-host-gateway.ts:104](https://github.com/openclaw/openclaw/blob/4630ce3d9e2f089062ccd386b0ae241325961d1a/src/agents/bash-tools.exec-host-gateway.ts#L104)
  - sha: [4630ce3d9e2f](https://github.com/openclaw/openclaw/commit/4630ce3d9e2f089062ccd386b0ae241325961d1a)
- **Node host exec likewise has no long-command obfuscation gate:** `executeNodeHostCommand()` evaluates allowlist/approval state and optional strict inline-eval only; no configurable or hardcoded command-length threshold remains in the node host path.
  - file: [src/agents/bash-tools.exec-host-node.ts:127](https://github.com/openclaw/openclaw/blob/4630ce3d9e2f089062ccd386b0ae241325961d1a/src/agents/bash-tools.exec-host-node.ts#L127)
