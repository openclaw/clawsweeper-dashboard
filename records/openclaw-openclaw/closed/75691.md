## Summary

Keep open. Current main still blocks only inherited Gateway service-marker environments and does not compare the active Gateway runtime PID against the caller ancestry; #75819 is the open closing PR for that missing guard.

## What This Changes

Request `openclaw update` to detect when it is running under the active Gateway process ancestry and fail with explicit restart guidance instead of silently leaving the live Gateway on old code.

## Best Possible Solution

Add a conservative updater preflight that compares the active managed Gateway runtime PID with the caller self/ancestor PID set before service stop or package mutation, then aborts with external-shell or manual-restart guidance.

## Reproduction Assessment

Yes. The issue gives concrete systemd and agent-exec steps, and current main still lacks the PID-ancestry check that would block a gateway-owned child without inherited service markers.

## Solution Assessment

Yes. A defensive ancestry preflight is the narrowest maintainable solution; landing #75819 or an equivalent reconciled patch is safer than adding a detached sudo/systemctl helper.
