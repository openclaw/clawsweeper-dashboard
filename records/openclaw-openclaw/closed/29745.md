## Summary

Current `main` no longer matches the reported behavior: invalid configs are auto-recovered from last-known-good during startup and hot reload when possible, invalid-config logging is deduped per config path, and the new tests/docs cover the recovery flow instead of allowing an infinite in-process retry/log loop.

## Evidence

## Close Comment

Closing as implemented on current `main` after a Codex review. The current code restores invalid configs from the last-known-good backup during startup (`src/gateway/server-startup-config.ts:71`) and during config reload (`src/gateway/config-reload.ts:209`) instead of staying in the old invalid-config churn path. Main also now dedupes invalid-config logging per config path (`src/config/io.invalid-config.ts:21`), with regression coverage in `src/config/io.invalid-config.test.ts:46`, `src/gateway/server-startup-config.recovery.test.ts:57`, and `src/gateway/config-reload.test.ts:651`. The current unreleased changelog also records the config/gateway recovery work in `CHANGELOG.md:171`. I could not prove a shipped release tag for this from the checkout, so I’m marking it implemented on `main` rather than released. If you still hit repeated log growth on a released build, please open a fresh issue with the exact version, supervisor (`launchd`/`systemd`), and whether a last-known-good backup existed.
