## Summary

Keep open. Current main still treats normal Gateway startup without `gateway.mode=local` as an intentional fail-closed guard, while the Web UI no-provider and Raw-mode crash claims need narrower reproduction before a safe implementation can be scoped.

## What This Changes

The issue asks OpenClaw to let the Gateway and Web UI start without requiring complete gateway, model, or provider configuration.

## Best Possible Solution

Define a reviewed bootstrap contract that keeps a loopback-only Control UI reachable before provider setup while preserving fail-closed behavior for unsafe configs, then fix any separately reproduced UI crash or Raw-mode defect with targeted regression coverage.

## Reproduction Assessment

Partial. The Gateway policy portion is source-reproducible from current source, tests, and docs: ordinary startup blocks missing `gateway.mode`, while `--allow-unconfigured` starts the bootstrap path. The no-provider Web UI crash and Raw-mode failure do not have a high-confidence reproduction path because the report lacks concrete UI steps, config, logs, and version details.

## Solution Assessment

Unclear. Broadly relaxing the startup guard is not proven to be the narrowest maintainable fix because it changes an intentional Gateway safety contract; a safer path is a maintainer-approved loopback bootstrap state plus separate fixes for reproduced UI failures.
