## Summary

Current `main` already preserves operator scopes for Control UI when `gateway.controlUi.dangerouslyDisableDeviceAuth=true`, and the same gateway logic is present in release `v2026.4.23`.

## Evidence

- **Bypass explicitly allows operator Control UI without device identity:** `evaluateMissingDeviceIdentity()` returns `allow` for Control UI operator sessions when `controlUiAuthPolicy.allowBypass` is active, which is the `dangerouslyDisableDeviceAuth` path described in the issue.
  - file: [src/gateway/server/ws-connection/connect-policy.ts:121](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/src/gateway/server/ws-connection/connect-policy.ts#L121)
  - command: `nl -ba src/gateway/server/ws-connection/connect-policy.ts | sed -n '1,180p'`
  - sha: [7e49cc87f9e9](https://github.com/openclaw/openclaw/commit/7e49cc87f9e9cdad47d0572331d7a7a47281a83e)
- **Scope-clearing guard excludes the bypass case:** `shouldClearUnboundScopesForMissingDeviceIdentity()` only clears unbound scopes when the decision is not `allow`, or when `allowBypass` is false. That means the explicit Control UI bypass does not wipe requested operator scopes.
  - file: [src/gateway/server/ws-connection/connect-policy.ts:84](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/src/gateway/server/ws-connection/connect-policy.ts#L84)
