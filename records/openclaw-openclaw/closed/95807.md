## Summary

Keep this PR open: current main does not have the every-specific unresolved-next-run guard, the diff is focused, and I found no concrete patch defect after reading the cron scheduling path. The remaining concern is a maintainer compatibility choice because malformed every retries will stop being re-armed on a backoff-only timestamp.

## What This Changes

The PR adds every-specific unresolved next-run handling in cron result application, with warnings and regression tests for malformed every schedule state.

## Best Possible Solution

Land the narrow every-only guard after maintainers accept the malformed-state clearing behavior as the intended sibling policy to the existing cron unresolved guard.

## Reproduction Assessment

Yes. Source inspection shows current main falls back to `backoffNext` when an `every` retry has `normalNext === undefined`, and invalid `everyMs` is the concrete non-mock way to make that happen; I did not run tests in this read-only review.

## Solution Assessment

Yes, if maintainers accept the compatibility tradeoff. The every-only helper matches the existing cron unresolved guard without changing `at` schedules or valid `every` retries; the safer alternative is to keep the current fallback until maintainers decide malformed `every` state should stop retrying.
