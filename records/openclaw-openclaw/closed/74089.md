## Summary

Keep open: the PR addresses a real current-main OpenAI TTS directive gap, but it still introduces a blocking fatal parse path for out-of-range speed directives.

## What This Changes

The PR adds OpenAI-owned `[[tts:speed]]` directive parsing, focused tests, telephony override coverage, and a changelog entry.

## Best Possible Solution

Merge the provider-owned OpenAI speed directive fix after out-of-range directive values are converted to non-fatal warnings and the focused test expectation is updated.

## Reproduction Assessment

Yes, source-level. Current main routes `[[tts:speed=1.5]]` to the selected provider parser, but OpenAI returns unhandled for `speed`, and the PR blocker is also source-reproducible because `requireInRange` throws through an uncaught provider parser call.

## Solution Assessment

No, not as-is. Provider-local OpenAI parsing with the OpenAI `0.25..4.0` range is the right shape, but invalid model-emitted speed directives should return warnings instead of aborting directive parsing.

## Review Findings

Return `handled: true` with a warning for out-of-range values and update the focused test expectation.
