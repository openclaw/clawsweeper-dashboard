## Summary

Current `main` no longer matches the empty-registry startup path described in #67104/#67163: the plugin/channel registry is activated before Telegram provider startup reaches `resolveNativeCommandsEnabled`, so this PR targets an obsolete startup race rather than a current reproducible bug.

## Evidence

- **Gateway load activates the registry before returning:** `loadGatewayPlugins(...)` builds `pluginRegistry` via `loadOpenClawPlugins(...)`, and the loader activates that registry with `activatePluginRegistry(...)` before it returns the loaded registry.
  - file: [src/gateway/server-plugins.ts:493](https://github.com/openclaw/openclaw/blob/7875092f4ddc6f2dbe218bea8506c9ac1da87c39/src/gateway/server-plugins.ts#L493)
  - command: `nl -ba src/gateway/server-plugins.ts | sed -n '434,510p'; nl -ba src/plugins/loader.ts | sed -n '2968,2984p'`
  - sha: [7875092f4ddc](https://github.com/openclaw/openclaw/commit/7875092f4ddc6f2dbe218bea8506c9ac1da87c39)
- **Channel startup happens after plugin bootstrap:** `startGatewayServer` consumes the bootstrapped `pluginRegistry`, then immediately enumerates `listChannelPlugins()` from the active state before later handing control to `createChannelManager`, which invokes each channel plugin's `startAccount(...)` asynchronously.
  - file: [src/gateway/server.impl.ts:363](https://github.com/openclaw/openclaw/blob/7875092f4ddc6f2dbe218bea8506c9ac1da87c39/src/gateway/server.impl.ts#L363)
  - command: `nl -ba src/gateway/server.impl.ts | sed -n '360,376p'; nl -ba src/gateway/server-channels.ts | sed -n '418,429p'`
  - sha: [7875092f4ddc](https://github.com/openclaw/openclaw/commit/7875092f4ddc6f2dbe218bea8506c9ac1da87c39)
- **Telegram reaches command auto-resolution only inside startAccount:** The Telegram plugin calls `monitorTelegramProvider(...)` from `gateway.startAccount`, and that path creates the bot before evaluating `resolveNativeCommandsEnabled(...)` / `resolveNativeSkillsEnabled(...)`. On current main, that happens after registry activation, not during plugin loading.
  - file: [extensions/telegram/src/channel.ts:898](https://github.com/openclaw/openclaw/blob/7875092f4ddc6f2dbe218bea8506c9ac1da87c39/extensions/telegram/src/channel.ts#L898)
  - command: `nl -ba extensions/telegram/src/channel.ts | sed -n '898,918p'; nl -ba extensions/telegram/src/polling-session.ts | sed -n '180,197p'; nl -ba extensions/telegram/src/bot-core.ts | sed -n '441,449p'`
  - sha: [7875092f4ddc](https://github.com/openclaw/openclaw/commit/7875092f4ddc6f2dbe218bea8506c9ac1da87c39)
- **Runtime channel-state fallback prefers a populated active registry:** `getActivePluginChannelRegistryFromState()` returns the pinned registry only when it has channels; otherwise it falls back to a populated active registry. The reported failure mode now requires both registries to be empty at the same time.
  - file: [src/plugins/runtime-channel-state.ts:20](https://github.com/openclaw/openclaw/blob/7875092f4ddc6f2dbe218bea8506c9ac1da87c39/src/plugins/runtime-channel-state.ts#L20)
  - command: `nl -ba src/plugins/runtime-channel-state.ts | sed -n '20,30p'`
  - sha: [7875092f4ddc](https://github.com/openclaw/openclaw/commit/7875092f4ddc6f2dbe218bea8506c9ac1da87c39)

But current `main` introduced the new `src/config/commands.ts` in commit `9e5d09c962628ca9aa8dc998d33e1f02f83d11a5` after this PR was opened, so the proposed fix now targets an older code path than the one shipping on `main`.
  - file: [src/config/commands.ts:6](https://github.com/openclaw/openclaw/blob/9e5d09c962628ca9aa8dc998d33e1f02f83d11a5/src/config/commands.ts#L6)
