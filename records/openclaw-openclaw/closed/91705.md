## Summary

Keep this PR open, but the current patch is not merge-ready: it fixes a real diagnostic-state leak path, yet treats every `failed/action:none` recovery as cleanup-safe even though that outcome can come from exceptions before active work or queued work is proved stopped.

## What This Changes

This PR adds `failed && action === "none"` to the diagnostic recovery predicates that allow outcomes to mark a session idle and clear queued diagnostic state.

## Best Possible Solution

Split failed recovery into cleanup-eligible and cleanup-unsafe outcomes, and only mark idle or clear queued state after the runtime has proved there is no active owner or queued lane work left to preserve.

## Reproduction Assessment

Yes, from source inspection: current main leaves failed recovery outcomes non-mutating while pruning only deletes idle, no-queue entries. I did not run a live gateway TTL repro in this read-only review.

## Solution Assessment

No; the patch is a plausible cleanup path but not the best fix because `failed/action:none` is a generic exception outcome. The safer fix is to distinguish cleanup-safe failures from failures that leave active or queued work unresolved.
