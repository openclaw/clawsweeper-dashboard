## Summary

Close as implemented on current main: the base config schema now accepts arbitrary `plugins.entries.*.config` payloads, and plugin-aware validation delegates those payloads to the loaded plugin manifest `configSchema`. The OpenViking tenant-header behavior remains plugin-side, but the OpenClaw core validator bottleneck reported here is addressed on `main`.

## What This Changes

Allow plugin-specific fields under `plugins.entries.*.config` by validating them against each plugin's declared `configSchema` instead of rejecting them in the base Gateway schema.

## Best Possible Solution

Keep the current plugin-schema validation behavior on `main`; any remaining OpenViking tenant identity fields should be handled by the OpenViking plugin schema and docs rather than by hardcoding OpenViking keys in OpenClaw core.

## Reproduction Assessment

No current-main failure path remains: source inspection shows the base schema accepts arbitrary `config` keys and the plugin-aware validator delegates to plugin schemas. The reporter's release log is credible for older builds, but the current `main` code no longer has that global additional-properties rejection.

## Solution Assessment

Yes. The implemented owner-boundary fix is the maintainable one: core accepts an opaque plugin config payload, then validates it against the loaded plugin's declared schema instead of adding service-specific fields to the global schema.
