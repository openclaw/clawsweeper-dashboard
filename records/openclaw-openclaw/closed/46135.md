## Summary

Keep open. Current `main` still has the reported Feishu streaming state shape, and the existing Feishu dispatcher tests explicitly allow a second streaming session after an earlier close, so there is not strong evidence that issue #46135 is already fixed on `main` or in `v2026.4.22`.

## Evidence

- **`startStreaming()` still lacks a synchronous re-entry flag:** `startStreaming()` only guards on `!streamingEnabled || streamingStartPromise || streaming` before creating a new `FeishuStreamingSession`; there is no separate `starting`/turn-scoped guard as described in the issue.
  - file: [extensions/feishu/src/reply-dispatcher.ts:268](https://github.com/openclaw/openclaw/blob/c1383680405879a3c3f43dceb20a694f54d3c13c/extensions/feishu/src/reply-dispatcher.ts#L268)
  - sha: [c13836804058](https://github.com/openclaw/openclaw/commit/c1383680405879a3c3f43dceb20a694f54d3c13c)
- **`closeStreaming()` clears both guards back to `null`:** `closeStreaming()` awaits startup, then resets `streaming = null` and `streamingStartPromise = null`, which re-opens the `startStreaming()` path for later callbacks in the same dispatcher lifecycle.
  - file: [extensions/feishu/src/reply-dispatcher.ts:302](https://github.com/openclaw/openclaw/blob/c1383680405879a3c3f43dceb20a694f54d3c13c/extensions/feishu/src/reply-dispatcher.ts#L302)
