## Summary

Close as implemented: the sync writer no longer has the reported unguarded chmod path on current main, and the first verified release artifact with the fix is v2026.5.9-beta.1. The reporter’s v2026.5.7 update was correct for the stable release, but later fs-safe work superseded this issue.

## What This Changes

The issue asks for synchronous `writeTextFileAtomic` writes to tolerate `chmod` EPERM on Windows-hosted Docker mounts when writing auth/config files.

## Best Possible Solution

Keep the fs-safe-backed implementation and treat broader Windows EPERM auth-profile cascades as separate follow-up work under their own issues.

## Reproduction Assessment

No for current main. The failing path is visible in v2026.5.7 source, but current main and v2026.5.9-beta.1 route the active writer through fs-safe wrappers instead of the unguarded chmod call.

## Solution Assessment

Yes. The fs-safe-backed writer is a more maintainable fix than a local one-line try/catch because it centralizes private file writes and preserves permission hardening as best-effort on unsupported filesystems.
