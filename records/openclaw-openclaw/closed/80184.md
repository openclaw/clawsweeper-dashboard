## Summary

Keep open. Current main still lets a selected-route env API-key label win before the runtime-alias OAuth fallback can apply, and the related follow-up fix was closed unmerged rather than landing on main.

## What This Changes

The issue asks `/status` to show the active Claude CLI OAuth auth path instead of an unrelated `ANTHROPIC_API_KEY` label when an Anthropic model is routed through the `claude-cli` runtime alias.

## Best Possible Solution

Land a narrow status-rendering fix that prefers the active OAuth label, and suppresses unused API-key cost display, only when the selected and active models are equivalent runtime aliases; keep auth resolution and provider routing unchanged.

## Reproduction Assessment

Yes at source level: pass an equivalent `anthropic/*` selected model and `claude-cli/*` active model with selected auth `api-key` and active auth `oauth`; current `status-message.ts` keeps the raw selected auth label before considering the runtime alias. I did not run a live Claude CLI OAuth gateway session in this read-only review.

## Solution Assessment

Yes, the best fix is the narrow renderer-level path proposed by the issue: prefer active OAuth only for equivalent runtime aliases where selected auth is API-key and active auth is OAuth, plus regression coverage for direct API-key behavior. Changing provider routing or auth resolution would be broader than needed.
