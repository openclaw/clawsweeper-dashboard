## Summary

Keep open: current main and the latest release still allow MEMORY.md to reach normal Discord channel sessions through bootstrap context and built-in memory search, while the Discord and workspace docs describe shared/guild sessions as not auto-loading MEMORY.md. The remaining decision is security/product policy, not stale cleanup.

## What This Changes

The issue reports that Discord guild/channel sessions can access MEMORY.md despite documentation saying guild channels do not auto-load long-term memory.

## Best Possible Solution

Set one explicit privacy contract for long-term memory in shared channels, then enforce it consistently in bootstrap injection, built-in memory_search/memory_get, QMD scope, Discord docs, and regression coverage with an explicit opt-in path if guild memory recall remains supported.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: normal Discord channel sessions are not filtered out of MEMORY.md bootstrap context, and built-in memory_search has no chat-type guard before searching indexed memory.

## Solution Assessment

Unclear until maintainers choose the intended security contract. A safe fix should make the DM/shared-channel policy explicit and enforce it in both prompt bootstrap and memory tools, with an opt-in or migration story for existing guild workflows.
