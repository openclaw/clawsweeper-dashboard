## Summary

Keeping this open: it is a protected beta-blocker with a concrete Mattermost direct-message regression report, and current source still shows `/new` and `/reset` acknowledgements can be produced as ordinary final source replies that are blocked by `message_tool_only` delivery mode.

## What This Changes

Mattermost direct-chat `/new` or `/reset` commands can reset session state without a visible acknowledgement when source replies are suppressed.

## Best Possible Solution

Route reset acknowledgements as explicit source-visible control replies that survive `message_tool_only` source suppression for Mattermost and sibling channel surfaces, with focused regression coverage for `/new` and `/reset` in suppressed-delivery contexts.

## Reproduction Assessment

Yes from source, with strong reporter evidence. Current reset commands return ordinary final replies, and current message-tool-only delivery suppresses automatic source replies; the missing piece is a live Mattermost rerun in this read-only review.

## Solution Assessment

Yes, the likely best fix is narrow: treat session reset acknowledgements as explicit source-visible control replies rather than changing Mattermost routing or broad delivery defaults.
