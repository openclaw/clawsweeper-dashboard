## Summary

Current `main` now treats the reported POSIX `exec host=node` absolute-path native-binary case as allowed under the node `system.run.prepare` planner, with explicit regression coverage, so the original Linux/macOS bug in #66136 is already implemented on `main`.

## Evidence

- **Node-host exec still routes through prepare, but uses the prepared plan at runtime:** `executeNodeHostCommand()` calls `system.run.prepare`, parses the returned plan, and forwards that plan into `system.run`. This is the path the issue called out, and the fix therefore lives in the planner rather than in a host=node bypass.
  - file: [src/agents/bash-tools.exec-host-node.ts:99](https://github.com/openclaw/openclaw/blob/250acdd2a28c190ed7934e3ba99022f24123d3c9/src/agents/bash-tools.exec-host-node.ts#L99)

  - sha: [250acdd2a28c](https://github.com/openclaw/openclaw/commit/250acdd2a28c190ed7934e3ba99022f24123d3c9)
- **Current planner allows absolute-path native binaries in shell payloads:** `shellPayloadNeedsStableBinding()` now only fail-closes shell payloads when the first token is relative, mutable/writable, or script-like. `resolveMutableFileOperandSnapshotSync()` returns success with no mutable operand when that stable-binding check passes, instead of emitting `approval cannot safely bind this interpreter/runtime command`.

Adjacent tests still deny relative and writable absolute copies, preserving the hardening the reporter expected for mutable/script-like paths.
