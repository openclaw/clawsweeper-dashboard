## Summary

Current main already implements the useful part of this PR: channel startup is bounded-parallel, per-channel startup failures are caught and logged without aborting later channels, and there is regression coverage for that behavior. The remaining detached-IIFE part of the PR would change the documented `gateway:startup` ordering that current main intentionally preserves.

## Best Possible Solution

Close this PR and keep the shipped main implementation. If maintainers still want fully detached sidecar readiness later, it should be a narrower follow-up that explicitly designs `/readyz`, Control UI availability, `gateway:startup` ordering, and startup-gated method behavior with tests.
