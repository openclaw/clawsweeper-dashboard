## Summary

Keep open: current main and v2026.5.26 still have a source-visible path where a timeout can be persisted as auth-profile-wide cooldown while same-provider fallback probing is bounded, so the reported chain exhaustion remains plausible and is not implemented away.

## What This Changes

The issue reports that a transient Google timeout can put the shared auth profile into cooldown and exhaust same-provider model fallbacks without a corresponding Google API error.

## Best Possible Solution

Land a narrow auth/fallback regression fix that prevents internal or provider-start-uncertain timeouts from poisoning a provider-wide auth profile, or scopes timeout cooldown to the attempted model while preserving billing/auth disables as profile-wide.

## Reproduction Assessment

No live high-confidence reproduction was established. Source inspection shows a credible path: provider-started timeout maps to an auth profile failure, timeout cooldowns are profile-wide, and same-provider fallback probing is capped after one provider probe.

## Solution Assessment

Yes, the best fix is a focused auth/fallback correction rather than a new config option. The repair should add regression coverage for same-provider Google-style fallback chains and stop if it would require changing provider policy beyond timeout scoping or provider-start detection.
