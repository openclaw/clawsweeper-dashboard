## Summary

Current `main` already routes Telegram conversations into agent-scoped session keys, resolves the default model from that routed agent, and the status surfaces now intentionally show selected-vs-active fallback model state. The same evidence is present in release `v2026.4.22`, so this issue is already implemented.

## Evidence

- **Telegram routing keeps the bound agent on the session key:** `resolveTelegramConversationRoute` builds/overrides the Telegram route with the routed `agentId` and an agent-scoped `sessionKey`, so follow-up execution is attached to the routed agent rather than the global default agent.
  - file: [extensions/telegram/src/conversation-route.ts:48](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/extensions/telegram/src/conversation-route.ts#L48)
  - sha: [a6aa151653ab](https://github.com/openclaw/openclaw/commit/a6aa151653ab478e9aa23e95c6446c41adc4fce1)
- **Reply execution resolves model selection from the routed session agent:** `getReplyFromConfig` derives `agentId` from `ctx.SessionKey`/`CommandTargetSessionKey` and immediately resolves the default model for that agent before running the reply path.
  - file: [src/auto-reply/reply/get-reply.ts:176](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/src/auto-reply/reply/get-reply.ts#L176)
  - sha: [a6aa151653ab](https://github.com/openclaw/openclaw/commit/a6aa151653ab478e9aa23e95c6446c41adc4fce1)
- **Per-agent model primary is applied by the shared resolver:** `resolveDefaultModelForAgent` swaps the effective primary model to the routed agent’s `agents.list[].model.primary` before resolving the configured model ref.
  - file: [src/agents/model-selection.ts:195](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/src/agents/model-selection.ts#L195)
  - sha: [a6aa151653ab](https://github.com/openclaw/openclaw/commit/a6aa151653ab478e9aa23e95c6446c41adc4fce1)
- **`session_status`/status card distinguishes selected model from active fallback model:** `buildStatusMessage` renders `Model:` from the selected override/configured model and a separate `Fallback:` line from the active runtime model when they differ.
  - file: [src/status/status-message.ts:484](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/src/status/status-message.ts#L484)
  - sha: [a6aa151653ab](https://github.com/openclaw/openclaw/commit/a6aa151653ab478e9aa23e95c6446c41adc4fce1)
- **Gateway session listing shows the selected override model, not the fallback runtime model:** `buildGatewaySessionRow` prefers `selectedModel` for the row display, and the regression test asserts that a fallback runtime `gpt-5.4` does not replace the selected override in session listings.
  - file: [src/gateway/session-utils.ts:1281](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/src/gateway/session-utils.ts#L1281)
  - sha: [a6aa151653ab](https://github.com/openclaw/openclaw/commit/a6aa151653ab478e9aa23e95c6446c41adc4fce1)
- **Released tests cover the reported status mismatch behavior in `v2026.4.22`:** Verified in the release tag that `src/auto-reply/status.test.ts` expects `Model: openai/gpt-4.1-mini` plus `Fallback: anthropic/claude-haiku-4-5`, and `src/gateway/session-utils.test.ts` expects session listings to show the selected override model even when runtime fell back.
  - command: `git show v2026.4.22:src/auto-reply/status.test.ts | nl -ba | sed -n '1028,1056p' ; git show v2026.4.22:src/gateway/session-utils.test.ts | nl -ba | sed -n '833,855p'`
  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)
