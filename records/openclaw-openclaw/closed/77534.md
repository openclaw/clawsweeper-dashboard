## Summary

Close as implemented on current main: the shared media store now treats `EPERM` from the post-write fsync as best-effort, and regression coverage confirms buffer saves still complete for the affected path.

## What This Changes

Report asks to stop Windows media uploads from failing when `writeSavedMediaBuffer` fsyncs a temp file opened read-only.

## Best Possible Solution

Keep the current main fix and let the next release carry it; reopen only if Windows uploads still fail after this guarded media-store path ships.

## Reproduction Assessment

No current-main failing path is present: source inspection shows `EPERM` from the fsync step is caught, and the colocated regression test covers that exact failure mode. The affected release reproduction in the report is credible, but current main no longer propagates the error.

## Solution Assessment

Yes. The current implementation is a narrow maintainable fix because it preserves the fsync attempt where it succeeds, treats only `EPERM` as best-effort, and still throws other write, open, sync, or rename failures.
