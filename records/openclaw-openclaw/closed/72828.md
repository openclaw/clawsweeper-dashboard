## Summary

Close as duplicate/superseded. Current main still lacks post-start Feishu SDK retry-exhaustion recovery, but the same failure mode is already tracked by open canonical issue #52618 and open maintainer PR #73739, which adds the constructor-level SDK callbacks and monitor-owned recovery loop this report needs.

## What This Changes

Report asks the Feishu WebSocket monitor to recover automatically when the Lark SDK exhausts reconnect attempts and otherwise leaves the gateway frozen until restart.

## Best Possible Solution

Keep #52618 as the canonical user problem and land #73739 or an equivalent current-main Feishu supervisor that uses the Lark SDK lifecycle callback contract, recreates dead WS clients with bounded backoff, preserves existing redacted logging, and then closes duplicate Feishu reconnect reports after the fix ships.

## Reproduction Assessment

Yes. The issue provides concrete logs, current main source confirms the monitor parks after `wsClient.start()` with no post-start recovery signal, and the locked SDK source shows reconnect exhaustion is reported through a constructor `onError` callback that OpenClaw does not currently subscribe to.

## Solution Assessment

No for the literal snippet, because `@larksuiteoapi/node-sdk@1.62.0` exposes `onError` on the WSClient constructor rather than `start()` options. The narrow maintainable fix is the #73739-style wrapper/supervisor path in the Feishu plugin, not a core migration or another duplicate issue.
