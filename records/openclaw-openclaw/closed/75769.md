## Summary

Close as implemented on current main. The report targets v2026.4.29 and matches the same post-release event-loop starvation hot path as #75577: request-time plugin/tool prep could still trigger bundled runtime dependency work and registry reload cost. Current main keeps runtime-deps repair out of agent/plugin tool construction, reuses the gateway-bindable active registry for Discord-triggered runs, and adds bounded/yielding session-list behavior for another synchronous gateway hot path.

## What This Changes

The issue asks OpenClaw to prevent Discord heartbeat ACK timeouts caused by agent startup/prep work blocking the Node.js event loop during Discord-triggered agent runs.

## Best Possible Solution

Keep the current main changes that remove runtime-deps repair and full plugin activation from request-time tool discovery, ship them in the next release, and only revisit Discord heartbeat isolation if current-main traces still show event-loop starvation after the plugin/tool hot path is gone.

## Reproduction Assessment

Yes for the v2026.4.29 failure signature: the report gives concrete prep-stage timings and Discord heartbeat logs, and current-main source differs exactly in the per-turn plugin/runtime-deps paths those timings implicate. I did not live-reproduce the Discord bot on OCI ARM64.

## Solution Assessment

Yes. The narrow maintainable fix is to keep agent startup/tool discovery from doing heavyweight plugin runtime dependency and registry work on the gateway event loop; moving Discord heartbeats to a worker thread would be a broader transport redesign that is not needed for the identified v2026.4.29 hot path.
