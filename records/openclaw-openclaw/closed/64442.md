## Summary

Current main no longer uses Discord reaction/app-event listener activity as the stale-socket health signal. It now tracks stale-socket from transport activity, and Discord gateway websocket frames feed that transport-liveness path with regression coverage, so the reported reaction-timeout -> stale-socket restart loop is already implemented away on main.

## Evidence

- **Reporter follow-up narrows the root cause:** The issue reporter later said `ackReactionScope: "off"` did not stop the ~35 minute restart pattern and concluded the reaction handler timeouts were a symptom rather than the cause.
- **Discord listener events stay app-level only:** `trackInboundEvent` updates `lastEventAt`/`lastInboundAt` and explicitly documents that Carbon heartbeats are internal, so Discord app events are not used as socket-liveness proof.
