## Summary

Close as implemented: current main and the v2026.6.1 release already persist `memory.recall.recorded` from `memory_search` hits into the short-term recall store and event log when dreaming is enabled, with regression coverage for that exact path.

## What This Changes

The issue reports that `memory.recall.recorded` persistence stopped after upgrading to v2026.5.26, leaving `openclaw memory promote` with no short-term recall candidates despite successful `memory_search` calls.

## Best Possible Solution

Keep the current memory-search recall tracking path and ask for a new report only if the same failure reproduces on v2026.6.1 or later with dreaming enabled and fresh `memory status --deep` evidence.

## Reproduction Assessment

No current-main failure path is reproduced. Current main and v2026.6.1 source plus regression tests show `memory_search` hits writing the short-term recall store and `memory.recall.recorded` event when dreaming is enabled.

## Solution Assessment

Yes. The current implementation fixes the reported persistence gap at the memory tool boundary, which is the narrowest owner boundary for `memory_search` recall recording.
