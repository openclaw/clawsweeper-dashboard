## Summary

Keep open: the fix direction and supplied package-upgrade proof are strong, but the latest branch still has two correctness blockers in the snapshot layer that can leave managed-service configs unsnapshotted or recovery snapshots stale.

## What This Changes

The branch adds update-time config preservation with a pre-update snapshot, update-aware unknown-key stripping, `plugins.installs` protection, update-time size-drop blocking, focused tests, changelog text, and a small SDK docs update.

## Best Possible Solution

Land this after the snapshot uses the exact config path that the update doctor or restart flow will mutate, and after pre-update snapshots are refreshed or bounded per update while preserving the strip and size-drop guards.

## Reproduction Assessment

Yes at source level: current main strips unknown keys before the write migration that preserves `plugins.installs`, and update-triggered doctor writes can bypass the legacy skip guard with size-drop allowance. I did not run the destructive packaged upgrade locally in this read-only review.

## Solution Assessment

No, not as submitted. The root-cause fixes are in the right area, but the snapshot path and retention behavior need a narrow repair before this is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Snapshot the resolved update config path:** `src/cli/update-cli/update-command.ts:146`
  - body: `createUpdateConfigSnapshot` always snapshots module `CONFIG_PATH`, but package-update doctor env is resolved from managed service env and can override `OPENCLAW_CONFIG_PATH`. Managed installs can therefore save the caller config while the doctor mutates the service config; derive the snapshot path from the same resolved env used for the write.
  - confidence: 0.91
- **[P2] Refresh or bound pre-update snapshots:** `src/config/backup-rotation.ts:128-138`
  - body: The fixed `<config>.pre-update` path is written with exclusive create and `EEXIST` is ignored, so the first snapshot for that config wins forever. Later updates cannot capture their own pre-update state, and stale config data can linger; refresh it per update or use bounded timestamped retention.
  - confidence: 0.88

## Security Review
