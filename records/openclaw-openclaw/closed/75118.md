## Summary

Keep open: current main still lacks the requested opt-in, and this PR is the active implementation path, but the patch still has correctness, proof, and security-review blockers before merge.

## What This Changes

The PR adds a default-off per-server `mcp.servers.<name>.injectCallerContext` option that injects OpenClaw caller headers for owner-trusted remote bundle MCP servers across CLI and embedded Pi paths, with schema, changelog, and tests.

## Best Possible Solution

Keep the owner-bound exact-URL opt-in shape, preserve all operator headers/placeholders, add source config metadata with regenerated artifacts, and require redacted real CLI or remote MCP proof plus maintainer/security signoff.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a bug, but yes: source inspection of current main shows the config type, Zod schema, bundle merge path, and CLI env preparation do not implement the requested opt-in.

## Solution Assessment

Is this the best way to solve the issue? No, not as-is. The per-server exact-URL opt-in is a plausible maintainable shape, but this branch still corrupts some embedded Pi headers, lacks source-backed schema metadata, and needs real proof plus security approval.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Restrict Pi expansion to caller-context headers:** `src/agents/pi-bundle-mcp-runtime.ts:211-214`
  - body: This replacement runs over every string header on an opted-in embedded Pi MCP server, so an existing operator header such as `Authorization: Bearer ${MY_INTERNAL_TOKEN}` becomes `Bearer ` because `MY_INTERNAL_TOKEN` is absent from the OpenClaw caller-context map. Limit expansion to the injected OpenClaw caller headers or preserve unknown placeholders verbatim.
  - confidence: 0.91
- **[P2] Add schema hint source metadata:** `src/config/schema.base.generated.ts:27685-27690`
  - body: The branch adds the `mcp.servers.*.injectCallerContext` hint directly to generated schema output, but `FIELD_LABELS` and `FIELD_HELP` do not define that source key. The next schema generation will drop the UI/docs hint, so add the source metadata and regenerate instead.
  - confidence: 0.85

## Security Review
