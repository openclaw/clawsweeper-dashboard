## Summary

Current `main` contains explicit bundled-plugin SDK aliasing and regression coverage for the exact Matrix startup resolution failure reported here, and the fix shipped in `v2026.4.23`.

## Evidence

- **Matrix bundled runtime now aliases `openclaw` / `openclaw/plugin-sdk/*` before Jiti loads the runtime:** `extensions/matrix/src/plugin-entry.runtime.js` builds a trusted alias map for both unscoped and scoped plugin SDK package names, then uses that Jiti instance to load the bundled Matrix runtime entry. That directly addresses the reported `Cannot find package 'openclaw'` import path from bundled Matrix startup.
  - file: [extensions/matrix/src/plugin-entry.runtime.js:87](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/extensions/matrix/src/plugin-entry.runtime.js#L87)
  - command: `nl -ba extensions/matrix/src/plugin-entry.runtime.js | sed -n '1,260p'`
  - sha: [51dd4f288f67](https://github.com/openclaw/openclaw/commit/51dd4f288f67ef4df4c7a121f078f508535df1fd)
- **Regression tests cover the Matrix wrapper alias path:** `extensions/matrix/src/plugin-entry.runtime.test.ts` has dedicated cases asserting the wrapper builds scoped and unscoped plugin-sdk aliases, keeps them deterministic, and ignores unsafe subpaths. That is the exact packaged/Jiti resolution surface implicated by the report.
  - file: [extensions/matrix/src/plugin-entry.runtime.test.ts:235](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/extensions/matrix/src/plugin-entry.runtime.test.ts#L235)

That is the same startup symptom from this issue.
  - file: [scripts/e2e/bundled-channel-runtime-deps-docker.sh:597](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/scripts/e2e/bundled-channel-runtime-deps-docker.sh#L597)
