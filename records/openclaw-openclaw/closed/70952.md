## Summary

The current OpenClaw core SecretRef contract does not include a native 1Password Connect provider; it supports env/file/exec SecretRefs, and the gateway runtime already collects active SecretRefs into one startup resolution pass, dedupes exact SecretRef keys, and batches exec-provider ids per provider. The reported duplicate OPC vault/item/fetch paths are provider-internal 1Password lookup work, which can be solved by a ClawHub/npm 1Password Connect exec resolver or plugin using the existing batched exec protocol without a missing core API.

## Best Possible Solution

Keep OpenClaw core's current SecretRef contract unchanged and move the 1Password Connect optimization into a ClawHub/npm exec resolver or plugin that accepts the existing batched ids request and memoizes vault/container lookup, item lookup, and fetched item payloads in memory for that invocation. Reopen only if that implementation exposes a concrete missing core API.

## Risks / Open Questions

- If maintainers later want a first-party native SecretRef provider-source API beyond env/file/exec, this could be reopened as explicit product/API design work rather than a cleanup issue.
- A poorly implemented external exec resolver can still make duplicate backend calls even though OpenClaw batches ids; the cache needs to live in that resolver/plugin.
