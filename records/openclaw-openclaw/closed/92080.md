## Summary

Close as a duplicate of the canonical generic plugin-approval routing bug: the Feishu-specific symptom maps to the same missing turn-source metadata path tracked in https://github.com/openclaw/openclaw/issues/74003, with https://github.com/openclaw/openclaw/pull/90918 already carrying the focused repair path.

## What This Changes

The issue reports that Feishu `skill_workshop action=apply` returns `Plugin approval unavailable (no approval route)` on OpenClaw 2026.6.1 even though Feishu registers an approval capability.

## Best Possible Solution

Use https://github.com/openclaw/openclaw/issues/74003 as the canonical route-metadata bug, review or replace https://github.com/openclaw/openclaw/pull/90918, and keep broader chat approval-evidence design in https://github.com/openclaw/openclaw/issues/93048.

## Reproduction Assessment

Yes. Source inspection shows the current before_tool_call plugin approval request omits the turn-source fields, and the gateway returns `decision: null` when no approval client, forwarder delivery, or turn-source route exists; I did not run a live Feishu bot flow in this read-only review.

## Solution Assessment

Yes. Forwarding existing turn-source metadata into the existing `plugin.approval.request` contract is the narrow maintainable fix; Feishu-specific routing or a missing-config approval bypass would be the wrong layer.
