## Summary

Current `main` already implements interruptible queue handling for long-running session work: inbound messages can be configured to `steer`, `interrupt`, `followup`, or `collect`, the Gateway exposes `sessions.steer` as an interrupt-and-steer control, and the Control UI shows queued work with a Steer action and stop/abort controls.

## Evidence

- **Queue modes include interrupt and steer:** The queue design doc says inbound messages can `steer` the current run, `followup`, `collect`, or `interrupt` the active run so the newest message runs next; it also documents overflow/drop handling for stale queued items.
  - file: [docs/concepts/queue.md:25](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/docs/concepts/queue.md#L25)

- **Runtime aborts active session and clears queued work:** `interruptSessionRunIfActive` aborts tracked and embedded runs, clears session queues, waits for the active run to end, and `handleSessionSend(... interruptIfActive: true)` reports `interruptedActiveRun` when restarting the session.
  - file: [src/gateway/server-methods/sessions.ts:368](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/src/gateway/server-methods/sessions.ts#L368)
