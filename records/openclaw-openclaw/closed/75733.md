## Summary

Keep open. The report matches the current code path closely enough to treat it as an unresolved loader-context bug: Telegram declares `grammy` as a staged bundled plugin dependency, but the bundled channel fallback loader still loads the full channel plugin with runtime-deps installation explicitly disabled and logs the same warning string on failure. Current startup staging reduces the active-runtime failure path, but I did not find source or regression coverage proving the noisy bundled-channel fallback warning is fixed on main.

## What This Changes

The issue asks OpenClaw to stop warning that Telegram is missing `grammy` when the package is present in the staged bundled plugin runtime-deps root after updating to 2026.4.29.

## Best Possible Solution

Add a narrow loader fix and regression test so full Telegram runtime loads that are expected to use staged bundled dependencies resolve through the prepared runtime-deps context, while metadata/discovery paths continue to avoid importing heavy runtime modules. The end state should make live Telegram startup and status/log paths agree: either the fallback uses the active/staged plugin registry or it stays quiet unless Telegram is genuinely unloadable.

## Reproduction Assessment

Yes, at source level. The reporter provided a concrete packaged-install log, and current main still has a path where `getBundledChannelPlugin()` imports the full Telegram plugin with `installRuntimeDeps: false`, which can reproduce a `grammy` resolution warning when the global package lacks `grammy`.

## Solution Assessment

No, current main is not yet the best solution. Startup pre-staging is useful, but the fallback bundled channel loader still has a context mismatch; the safer fix is to scope staged resolution to full runtime fallback loads without weakening the existing discovery guardrails.
