## Summary

Current `main` intentionally treats `runLevelError` as a hard-failure signal in cron outcome resolution. The PR's one-line change would mask genuine embedded-runner failures as `status: "ok"`, and the underlying report does not match the current runner contract because `meta.error` is now reserved for hard failure kinds rather than messaging-tool warnings.

## Evidence

- **Cron helper still treats run-level errors as fatal:** `resolveCronPayloadOutcome` only considers a later success payload as recovery when `runLevelError` is absent; otherwise any error payload remains fatal.
  - file: [src/cron/isolated-agent/helpers.ts:154](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/cron/isolated-agent/helpers.ts#L154)

- **Current main has an explicit regression test for this behavior:** `src/cron/isolated-agent.helpers.test.ts` asserts that an error payload plus later assistant text is still fatal when `runLevelError` is a `context_overflow`.
  - file: [src/cron/isolated-agent.helpers.test.ts:42](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/cron/isolated-agent.helpers.test.ts#L42)

- **Cron status is derived directly from fatal-payload resolution:** `runCronIsolatedAgentTurn` sets `status: "error"` vs `"ok"` from `hasFatalErrorPayload`, so removing the `runLevelError` guard would flip real runner failures to success when later text exists.

- **Embedded runner error metadata is a hard-failure contract:** `EmbeddedPiRunMeta.error.kind` is limited to `context_overflow`, `compaction_failure`, `role_ordering`, `image_size`, and `retry_limit`; it does not represent generic messaging-tool warnings.
  - file: [src/agents/pi-embedded-runner/types.ts:111](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/agents/pi-embedded-runner/types.ts#L111)
