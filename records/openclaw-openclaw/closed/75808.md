## Summary

Current main and the `v2026.4.29` tag already implement the requested per-agent filesystem policy: an agent-level `tools.fs.workspaceOnly` value overrides the global setting, and that effective value is passed into host `write` and `edit` operations. The earlier April reports are addressed for write/edit; for truly unrestricted `apply_patch`, the separate `tools.exec.applyPatch.workspaceOnly: false` guard is still intentionally required.

## What This Changes

The issue asks whether per-agent `tools.fs.workspaceOnly` overrides in OpenClaw v2026.4.29 can let one main agent write outside its workspace while keeping sub-agents workspace-restricted.

## Best Possible Solution

Keep the shipped per-agent fs policy behavior. For a trusted main agent, set that agent's `tools.fs.workspaceOnly` to `false`; keep restricted agents at `true` and preferably combine that with sandbox mode and tool allow/deny lists. If the trusted agent also needs `apply_patch` outside the workspace, configure the separate `tools.exec.applyPatch.workspaceOnly: false` only for that trusted agent because it is intentionally guarded more tightly.

## Reproduction Assessment

Yes. The source-level reproduction is covered by unit tests for agent override precedence and host write/edit outside the workspace when `workspaceOnly=false`; the same implementation is present in the `v2026.4.29` tag.

## Solution Assessment

Yes. The current implementation is the narrow maintainable solution: config resolution stays per-agent, the effective boolean is threaded into the existing host write/edit factories, and the stronger `apply_patch` guard remains separate for explicit opt-in.
