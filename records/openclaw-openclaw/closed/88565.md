## Summary

Keep open: current main still attempts an `openclaw.json` rewrite during `models auth login`, and the current auth-profile parser rejects `type: "apiKey"`; the reported valid-profile wipe needs a focused repro because the current upsert path appears merge-shaped.

## What This Changes

The issue reports that `models auth login` can reject or truncate `openclaw.json`, drop existing `auth-profiles.json` entries, and silently ignore camelCase API-key profile types.

## Best Possible Solution

Define a non-destructive login persistence contract: auth-profile writes must preserve existing profiles, main config writes must preserve all source config or be skipped unless an explicit config/default change is required, and invalid profile types should surface a clear diagnostic or doctor repair path.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not yet end-to-end: source inspection confirms the main-config write and `apiKey` type rejection, but the reported wipe of other valid profiles still needs a focused current-main repro.

## Solution Assessment

Is this the best way to solve the issue? Unclear: preventing destructive writes is right, but “never write openclaw.json” needs maintainer confirmation because current code uses `auth.profiles` metadata there.
