- `P1`: The reported release behavior breaks a real Telegram channel workflow until manual operator recovery.

## Summary

Current main fixes the reported Telegram isolated-ingress stuck-spool failure: spooled updates are no longer handled before the grammY bot is initialized, and recoverable init failures retry instead of entering the endless `Bot not initialized` loop seen on v2026.5.12.

## What This Changes

The issue reports a Windows Scheduled Task cold-start race where a Telegram `getMe` timeout leaves isolated polling ingress spooling updates that repeatedly fail with `Bot not initialized`.

## Best Possible Solution

Keep the current init-before-drain implementation and route any remaining Windows cold-start starvation after a fixed build to a fresh Telegram startup diagnostics issue.

## Reproduction Assessment

Yes for the shipped v2026.5.12 failure: the report has detailed logs and the dependency/source path explains why `handleUpdate` fails without bot initialization. No current-main failure path remains because current main initializes the bot before isolated spool drain.

## Solution Assessment

Yes. Initializing at the isolated-ingress boundary is the narrow maintainable fix because it satisfies grammY's `handleUpdate` contract and lets existing recoverable setup backoff retry before any spooled update is dispatched.
