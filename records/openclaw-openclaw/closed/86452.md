## Summary

Current main already handles the reported session-lock leak: embedded abort cleanup is bounded by an abort-settle timeout and releases the session write lock in a finally block, with a regression test for the non-settling timeout case. The same source and test are present in the v2026.5.22 release tag.

## What This Changes

The issue reports a session write lock remaining held after a chat.abort embedded abort-settle timeout in v2026.5.22.

## Best Possible Solution

Keep the shipped bounded abort-cleanup release path; if operators still see this symptom, track it as a new distinct lock-holder path with full logs.

## Reproduction Assessment

Yes for the central lock-release condition: the current regression test keeps the abort-settle promise pending past the timeout and verifies that the lock release still happens. I did not run tests because this review was read-only, but the source path is directly auditable.

## Solution Assessment

Yes. The current solution is narrow: bound abort settlement, skip long idle waiting for aborted cleanup, and always release the session lock in finally without adding new config or channel-specific policy.
