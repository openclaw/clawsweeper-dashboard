## Summary

Current main already forwards configured MCP `headers` for `transport: "streamable-http"`. OpenClaw preserves the header map from `mcp.servers`, selects the streamable HTTP transport, constructs the MCP SDK `StreamableHTTPClientTransport` with `requestInit.headers`, and the pinned SDK merges those headers into the POST/GET/DELETE request path used for initialize, tool listing, and tool calls. The same implementation is present in `v2026.4.24`.

## Best Possible Solution

Close this issue as implemented on main. Keep the current `StreamableHTTPClientTransport` `requestInit.headers` forwarding path; if maintainers touch this area again, the useful follow-up would be a narrow regression test that captures the actual streamable-http transport options or fetch headers.
