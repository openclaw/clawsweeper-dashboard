Current main already has the Codex/app-server and diagnostic recovery paths needed to prevent this reported `blocked_tool_call` stall from staying at `recovery=none`; the fix appears current-main-only and not included in the latest tagged release.

## What This Changes

The issue reports native Codex harness turns getting stuck in `processing` after a bash command exits, with stalled diagnostics showing `blocked_tool_call` and `recovery=none`.

## Best Possible Solution

Keep the current-main recovery design: Codex app-server idle watchdogs handle quiet turns, and diagnostic abort-drain recovery handles stale active tool or embedded-run state after the configured threshold.

## Reproduction Assessment

No high-confidence live reproduction was run for the reporter's exact host setup. The reported failure mode is source-checkable, and current main has tests for the stale native `bash` tool-call recovery path that would have produced `recovery=none` before the active-abort gate existed.

## Solution Assessment

Yes for the central stuck-session problem: current main uses watchdogs plus abort-drain recovery rather than brittle OS child-process probing. If a future report still reproduces after this code ships, it should include fresh diagnostics from a build containing `21c5f8dc6df1db60ca481b2c2bbcf5473a8d657a`.
