## Label Justifications

- `impact:message-loss`: Thread replies can be silently ignored when `requireMention` is enabled, which is a message-delivery impact.

## Summary

Keep open. Current main still has a single Mattermost `requireMention` gate for non-DM messages, so thread replies with `requireMention: true` are still dropped unless mentioned; the open related implementation PR is a candidate but has not merged and this issue already carries a maintainer-review/product-decision signal.

## What This Changes

Request separate Mattermost mention-gating behavior so top-level channel messages can require an @mention while thread replies can continue without one after the bot has joined the thread.

## Best Possible Solution

Keep this issue as the canonical Mattermost thread mention-policy decision, then land a reviewed Mattermost-owned fix that preserves top-level channel mention gating while allowing explicit thread follow-up behavior with upgrade-safe config/defaults.

## Reproduction Assessment

Yes, from source inspection: with Mattermost `requireMention: true`, any non-direct post that is not mentioned is dropped before the thread context is dispatched, and the current gate does not branch on `threadRootId`. I did not run a live Mattermost CRT instance, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

Unclear. The requested `requireMentionInThread` option is plausible, but the best product shape may be thread-participation wakeup semantics or a broader Mattermost threading policy rather than a new config key; maintainers should decide before implementation.
