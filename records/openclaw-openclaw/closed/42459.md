## Summary

Current main already routes Mattermost draft previews through a finalizable draft lifecycle that waits for in-flight preview writes before either finalizing in place or falling back to normal delivery, so the 2-second race described in #42459 no longer matches the code on main.

## Evidence

- **Draft loop serializes preview writes and exposes in-flight waiting:** `flush()` waits on any existing `inFlightPromise` before sending the next preview update, and `waitForInFlight()` awaits the outstanding network write directly.
  - file: [src/channels/draft-stream-loop.ts:20](https://github.com/openclaw/openclaw/blob/d29eaeafc1fc1ca86f0ccb5c6b2b3d3668502132/src/channels/draft-stream-loop.ts#L20)
  - sha: [d29eaeafc1fc](https://github.com/openclaw/openclaw/commit/d29eaeafc1fc1ca86f0ccb5c6b2b3d3668502132)
- **Finalization path removes pending work and waits instead of using a timeout:** `stopForClear()` and `seal()` mark the stream stopped/final, cancel pending timer work, and then `await loop.waitForInFlight()`. That prevents a late preview create/patch from landing after final delivery.
  - file: [src/channels/draft-stream-controls.ts:58](https://github.com/openclaw/openclaw/blob/d29eaeafc1fc1ca86f0ccb5c6b2b3d3668502132/src/channels/draft-stream-controls.ts#L58)

  - file: [src/channels/draft-preview-finalizer.ts:34](https://github.com/openclaw/openclaw/blob/d29eaeafc1fc1ca86f0ccb5c6b2b3d3668502132/src/channels/draft-preview-finalizer.ts#L34)

  - file: [extensions/mattermost/src/mattermost/draft-stream.ts:100](https://github.com/openclaw/openclaw/blob/d29eaeafc1fc1ca86f0ccb5c6b2b3d3668502132/extensions/mattermost/src/mattermost/draft-stream.ts#L100)

## Close Comment

Closing from a Codex review because current `main` already implements the behavior this issue asked for. Mattermost draft previews now use the shared finalizable draft lifecycle: `src/channels/draft-stream-controls.ts:58`-`67` cancels pending timer work and waits for any in-flight preview write, and `src/channels/draft-preview-finalizer.ts:34`-`54` flushes/seals before an in-place final edit or waits via `discardPending()` before fallback delivery. Mattermost is wired to that path in `extensions/mattermost/src/mattermost/draft-stream.ts:100`-`114` and `extensions/mattermost/src/mattermost/monitor.ts:297`-`330`, so the old 2-second busy-wait race described here no longer matches the code. The stale-overwrite/late-update cases are also covered in `extensions/mattermost/src/mattermost/draft-stream.test.ts:123`-`164` and `src/channels/draft-stream-controls.test.ts:123`-`151`. This implementation is already present in the shipped `v2026.4.22` tree (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`).
