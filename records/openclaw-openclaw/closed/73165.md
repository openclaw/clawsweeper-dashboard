## Summary

Keep open: the MCP advertised-skill readiness gate is not on current main and remains potentially useful, but the PR head is not mergeable because an explicit skill-level denial can fall through to a server-level grant, the smoke script runs the global mcporter check from the wrong cwd, external real behavior proof is missing, and the branch is currently conflicting.

## What This Changes

The PR adds MCP metadata parsing for skills, gates prompt-advertised MCP-backed skills on mcp-readiness.json records, adds prompt coverage, and introduces an mcporter readiness smoke script while touching gateway and qa-channel setup plumbing.

## Best Possible Solution

Land a rebased readiness-gate design only after skill-level records are authoritative, the global mcporter smoke check runs from the user home context, and freshness/owner semantics are documented or intentionally specified.

## Reproduction Assessment

Yes, for the PR-head defect from source inspection: a readiness file with skills[name].callable false and servers[server].callable true reaches the server fallback in isMcpSkillRuntimeReady. Current main does not have the readiness feature, so this is source-reproducible on the proposed patch rather than a current-main bug.

## Solution Assessment

No. The direction is plausible, but the current patch is not the best mergeable solution until the precedence and smoke cwd defects are fixed, the branch is rebased, and after-fix real behavior proof is added.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Short-circuit skill readiness denials:** `src/agents/skills/workspace.ts:175-180`
  - body: When a readiness file has a skill-level record that is not callable for this agent, the new code still checks the server record and can return true. That lets a server-level grant advertise a skill whose own entry denies callability; treat an existing skill record as authoritative before falling back to the server.
  - confidence: 0.94
- **[P2] Use home cwd for global mcporter config:** `scripts/mcp-readiness-smoke.mjs:59-61`
  - body: The global mcporter config case falls back to the OpenClaw home as cwd before running mcporter config list. That verifies global mcporter from the state directory instead of the user's home context, so use os.homedir() for the non-workspace branch.
  - confidence: 0.86
