## What This Changes

The PR changes Codex app-server tool-surface gating so any explicit `toolsAllow` entry matching a Codex-native tool enables native code mode, and adds unit coverage for native and non-native names.

## Best Possible Solution

Keep explicit `toolsAllow` restrictions authoritative: either pass and enforce a per-native-tool Codex restriction if supported, or require an explicit whole-native-surface opt-in and fail loudly for unsupported partial native lists.

Yes, from source: current main filters native names out of dynamic tools and only enables native code mode for undefined or wildcard `toolsAllow`, so an explicit native list loses both surfaces. I did not run a live cron because this review is read-only.

## Solution Assessment

No. The PR targets the right predicate, but enabling the indivisible native code-mode surface for any single native-owned name is not the safest fix for a restricted allowlist.
