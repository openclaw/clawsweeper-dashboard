## Summary

Current `main` already handles Slack stream teardown failures for unflushed buffers, including `missing_recipient_user_id`, without routing them through the noisy generic stop-stream error path.

## Evidence

## Close Comment

Codex review: this looks already fixed on `main`, and the same implementation is present in the shipped `v2026.4.22` release. In `extensions/slack/src/streaming.ts:236`, `missing_recipient_user_id` is treated as a benign finalize error and an unflushed-buffer teardown is converted into `SlackStreamNotDeliveredError` instead of surfacing the raw Slack API error. In `extensions/slack/src/monitor/message-handler/dispatch.ts:944`, that error is caught and routed through normal `chat.postMessage` fallback, while the noisy `slack-stream: failed to stop stream` log at `:947` is reserved for other errors. There is direct coverage for this in `extensions/slack/src/streaming.test.ts:135` and `extensions/slack/src/monitor/message-handler/dispatch.preview-fallback.test.ts:412`. Closing as implemented on main.
