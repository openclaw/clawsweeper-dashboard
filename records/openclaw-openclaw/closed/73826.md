## Summary

Current main already implements the reported WebChat/Control UI audio attachment path: `MEDIA:` audio URLs normalize to `kind: "audio"`, assistant audio attachments render an inline `<audio controls>` player instead of a generic download card, and the Control UI has an authenticated assistant-media route for local media paths. The latest local release tag `v2026.4.26` contains the same source implementation; changelog history also records earlier WebChat audio controls and TTS history support in the 2026.4.9 and 2026.4.11 sections.

## Best Possible Solution

Close this issue as implemented. Keep the current Control UI source and tests for audio attachment normalization/rendering; any remaining desire for guaranteed autoplay should be handled as a narrower follow-up with browser autoplay policy constraints called out explicitly.
