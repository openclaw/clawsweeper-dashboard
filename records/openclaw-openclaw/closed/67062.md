## Summary

Keep open. Current main covers the reporter's concrete `multimedia.nt.qq.com.cn` example through the default `*.qq.com.cn` allowlist, but QQ Bot still has no supported `channels.qqbot.mediaAllowHosts` config or runtime merge point, and the requested change is a security-sensitive plugin feature rather than cleanup-close material.

## What This Changes

Request to add a configurable `channels.qqbot.mediaAllowHosts` allowlist that merges with QQ Bot's default media SSRF host policy.

## Best Possible Solution

Implement this inside the bundled QQ Bot plugin by adding typed config, schema, manifest metadata, generated config metadata, docs, and tests, then merge normalized configured hosts with the conservative defaults in both inbound and outbound media download paths while preserving HTTPS, redirect, DNS, and private-network protections.

## Reproduction Assessment

Yes for the remaining config gap: current source exposes no QQ Bot `mediaAllowHosts` schema, type, manifest, generated metadata, or docs surface, and both media download paths still call the fixed `downloadFile` policy. The original `multimedia.nt.qq.com.cn` example is no longer the strongest reproduction because `*.qq.com.cn` is now default-allowed.

## Solution Assessment

Unclear until maintainers approve the SSRF semantics. The narrow maintainable direction is a QQ Bot-owned config merge through the existing SSRF policy helpers, not a core migration or broad shared media-fetch rewrite.
