## Summary

Keep this PR open for maintainer handling. It is protected by MEMBER author context and the `maintainer` label, current `main` does not already implement the proposed Gateway abort/wait behavior, and the latest branch still has a concrete correctness gap for session-wide aborts that return more than one run id.

## What This Changes

The branch emits Gateway lifecycle cancellation events for aborted chat runs, avoids normal chat completion dedupe writes after abort, writes `sessions.abort` wait snapshots with run-kind-aware dedupe prefixes, and adds targeted gateway abort/wait test coverage plus one cleanup retry change.

## Best Possible Solution

Keep the Gateway-only scope, but update the existing branch so every run id returned by a session-wide abort receives a terminal wait snapshot under the correct `chat:` or `agent:` prefix; leave SDK chat-projection behavior to #74750 and keep #74704 as the umbrella until all happy-path semantics are stable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Record abort snapshots for every aborted run:** `src/gateway/server-methods/sessions.ts:1351-1353`
  - body: `sessions.abort` can be called with only a session key, and `chat.abort` can return multiple authorized run ids for that session. This patch writes the terminal wait snapshot only for `runIds[0]`, so an immediate `agent.wait` for any other aborted run can still miss the cancellation snapshot and report a wait-only timeout instead of the cancelled terminal state. Iterate all returned run ids and choose the dedupe prefix per run.
  - confidence: 0.84
