## Summary

Current `main` already covers the reported cron-store/gateway mismatch: persisted jobs are loaded from `jobs.json`, legacy `jobId` entries are normalized to stable `id` values for scheduler lookups, startup catch-up runs before the timer is armed, and regression tests now assert that `cron list`/`cron status` stay responsive during startup catch-up instead of hanging.

## Evidence

- **Legacy cron ids normalize on load:** `ensureLoaded()` normalizes persisted cron jobs through `normalizeCronJobIdentityFields()` / `normalizeCronJobInput()` and warns when a legacy `jobId` field was converted to an in-memory `id`, which directly addresses lookup failures like `unknown cron job id` for legacy stores.
