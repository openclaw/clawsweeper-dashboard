## Summary

Current main and the latest release now prevent generated stale `sessionFile` paths from following a new `sessionId`, so the reported heartbeat/cron transcript reuse problem is implemented; the later archived-artifact/dreaming concern is a distinct follow-up, not the original old-file reuse bug.

## What This Changes

The issue asks for heartbeat isolated sessions to use a fresh transcript when `sessionId` rotates instead of appending to the previous transcript file.

## Best Possible Solution

Keep the shipped session-file rollover behavior; handle archived transcript or dreaming-ingestion cleanup under a separate focused tracker if it still reproduces.

## Reproduction Assessment

Yes. The historical failure path is source-reproducible from the old forced isolated cron rollover carrying `sessionFile`, and current tests now cover clearing and rewriting generated transcript paths; I did not run a live heartbeat because this was a read-only review.

## Solution Assessment

Yes. The best fix is at the session identity/store boundary: clear stale cron metadata on fresh isolated runs and rewrite generated `sessionFile` paths whenever the persisted `sessionId` changes.
