## Summary

Keep this PR open. Current main still has the unvalidated bare-host fallback in both setup-code URL normalization paths, and the PR is a narrow, relevant fix with regression coverage for malformed public setup URLs before bootstrap token issuance. I did not find a concrete correctness blocker in the proposed patch from the provided diff.

## What This Changes

The PR tightens device-pair and QR public setup URL normalization in `extensions/device-pair/index.ts` and `src/pairing/setup-code.ts`, and adds regression tests for valid bare host ports, malformed scheme-like inputs, invalid ports, unsupported schemes, and URL userinfo.

## Best Possible Solution

Land a narrow fix that validates synthesized public setup URLs in both the core QR setup-code resolver and the bundled `device-pair` plugin, preserving documented bare host support while failing before bootstrap token issuance for malformed or credential-bearing URLs.
