Keep open. Current main still does not implement this PR's direct non-vector `openclaw ltm query` command, `--cols`, `--filter`, `--order-by`, or `MemoryDB.getTable()` accessor. ## Risks / Open Questions

- Merging this as-is would add a second memory inspection command while #67952 is still open for overlapping `ltm list` row-inspection behavior.
- `--filter` forwards a user-supplied SQL-style predicate to LanceDB; the current allowlist lowers accidental misuse but still deserves maintainer review as a local data-access surface.
- With `--order-by`, the PR materializes all filtered rows via `toArray()` before sorting and slicing, which can be memory-heavy on large memory tables.
- The PR adds production behavior without targeted tests for invalid limits, filter validation, column projection, omitted vectors, sort-before-limit semantics, or omitted sort-column behavior.
- The PR does not update user docs for the new command or the ordered-query full-scan tradeoff.
