## Summary

Keep this PR open. Current main and the latest release still lack an active-poll guard for background exec completion events, so the PR is not obsolete, but the latest refcount version can still drop a configured `notifyOnExit` event when an active `process poll` aborts after the exec exits and before poll consumes the exit.

## What This Changes

The PR adds a `ProcessSession.pollActiveCount` refcount, suppresses `notifyOnExit` while `process poll` is active, marks poll/kill-consumed exits as notified, and adds focused poll ghost regression tests.

## Best Possible Solution

Amend the PR so active-poll suppression only replaces the notification when a poll actually consumes the exit, or hands the completion back to the normal notification path when poll aborts before consumption. Keep the refcount design, add direct abort-race coverage, and preserve normal `notifyOnExit` behavior when no poll is active.

  - body: The new guard suppresses `maybeNotifyOnExit()` whenever `pollActiveCount > 0`, but `process poll` only sets `exitNotified` after the wait loop reaches the exited branch.
