## Summary

Current `main` already covers this locale-refresh bug. The overview page now prefers persisted `settings.locale` during startup instead of relying solely on async i18n hydration, the selected option is rendered from `currentLocale`, and there is a regression test that asserts the dropdown stays on `zh-CN` while `i18n.getLocale()` is still `en`. The changelog also records this as already fixed on `main` under `#48019`.

## Evidence

- **Overview now uses persisted locale first:** `renderOverview()` computes `currentLocale` from `props.settings.locale` when it is supported, only falling back to `i18n.getLocale()` otherwise. The `<option>` rendering also marks the matching locale selected from that computed value.
