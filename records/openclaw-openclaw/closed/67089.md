Full review comments:

- **[P1] Restore the writeConfigFile options brace:** `src/config/io.ts:1909-1912`
  - body: The wrapper call to `io.writeConfigFile(nextCfg, { ... })` now closes with `);` instead of `});`, leaving the options object unterminated. This makes the branch fail TypeScript parsing before any of the security changes can run.
  - confidence: 0.99
- **[P1] Bootstrap existing configs before enforcing signatures:** `src/config/io.ts:1112-1114`
  - body: The new verifier treats a missing `.sig` on any established config as suspicious, and `loadConfig()` rejects that by returning `{}`. Existing installations will not have a sidecar signature on upgrade, so a valid config can be dropped on first boot with no migration path.
  - confidence: 0.96
- **[P1] Keep config authorization in the scope registry:** `src/gateway/server-methods/config.ts:442-446`
  - body: The PR adds inline `operator.admin` checks in the config handlers even though method authorization is already resolved centrally through the gateway method scope registry. Duplicating that policy creates a second source of truth and can drift from the advertised least-privilege contract.
  - confidence: 0.9
- **[P2] Surface signature persistence failures:** `src/config/io.hmac-integrity.ts:77-79`
  - body: Both signature write helpers swallow every write error, but the read path can later reject missing signatures. A successful config write followed by a silent `.sig` write failure can therefore create a later lockout without telling the caller persistence was incomplete.
  - confidence: 0.9
- **[P2] Thread config IO dependencies into HMAC key lookup:** `src/config/io.hmac-integrity.ts:23-39`
  - body: `readGatewayToken()` reads `os.homedir()` and `process.env` directly instead of using the `createConfigIO` dependencies that the config path uses. Tests and alternate state dirs can then read the config from one home but verify HMACs against another, so the new integrity path is not reliably exercised.
  - confidence: 0.88
- **[P2] Remove the broken CodexBar onboarding branch:** `src/wizard/setup.ts:510-522`
  - body: The new `codexbar` branch only prints a note, then falls through to the normal auth/model selection flow because it does not break or return. That does not implement the claimed “skip LM config” behavior and adds an unrelated external manager choice to a security PR.
  - confidence: 0.83
