## Summary

Keep open: the linked StepFun regression is valid, but this PR still estimates the wrong request surface and lacks after-fix real provider proof, so it is not merge-ready.

## What This Changes

The PR adds per-call max_tokens/max_completion_tokens clamping in the OpenAI-completions request builder and adds a changelog entry for the linked max_tokens regression.

## Best Possible Solution

Compute the remaining output budget from the finalized chat-completions payload, preserve existing smaller caps and no-cap behavior, then add focused regression coverage plus redacted real provider proof.

## Reproduction Assessment

Yes. A source-level reproduction can build an OpenAI-completions request with a 262144-token context and a large prompt; current main forwards the full cap without subtracting prompt tokens, matching the linked StepFun rejection shape.

## Solution Assessment

No. Dynamic per-call clamping is the right direction, but this implementation estimates the pre-conversion context and changes no-cap behavior instead of budgeting from the final request payload.

Overall correctness: patch is incorrect

- **[P1] Compute the cap from the final request payload:** `src/agents/openai-transport-stream.ts:2977-2987`
  - body: This estimates `completionsContext.messages` before the request payload is built, but the provider sees converted messages, optional flattened content, replay fields, and tool payloads.   - confidence: 0.9

That changes existing provider defaults on upgrade and should stay omitted unless a configured/model cap is present or maintainers explicitly choose a migration.
  - confidence: 0.84
