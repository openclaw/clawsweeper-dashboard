## Summary

This PR is superseded by current main. It targets the v2026.4.24 `openclaw status` QQBot `PlatformAdapter not registered` crash from #72465, but current main fixes that status path generically: status only inspects configured/effective channels and disables setup-runtime fallback, so unconfigured QQBot is no longer loaded through `resolveQQBotAccount`. The PR also is not the best follow-up hardening as written because its no-adapter path skips config-backed `clientSecret` values and the changelog entry lacks the required contributor attribution.

## Best Possible Solution

Close this PR as superseded by the current generic status/read-only discovery fix. If QQBot resolver hardening is still desired, open a new focused PR that preserves config > file > env semantics without requiring the platform adapter for plaintext/allowed setup snapshots, includes an isolated no-adapter regression test, and fixes changelog attribution if a user-facing entry is needed.
