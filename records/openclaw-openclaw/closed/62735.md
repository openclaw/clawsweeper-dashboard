## Summary

Keep #62735 open. Current main still does not provide the requested custom `api: "ollama"` path: the Ollama plugin is only resolved for registered provider ids/aliases, the bundled Ollama manifest only owns `ollama`, and the generic transport registry still explicitly treats `ollama` as unsupported. The `injectNumCtxForOpenAICompat` path also remains gated by Ollama identity/port heuristics rather than acting as an explicit override for arbitrary local proxies. Related PR #69703 proposed part of this fix but was closed unmerged, and its behavior is not present in current main.

## Best Possible Solution

Keep this issue open and fix it in core/provider-runtime terms. A complete maintainer path should either route custom provider entries with `api: "ollama"` through the native Ollama stream when the operator opts in, or add a documented explicit override for OpenAI-compatible `num_ctx` injection that bypasses the current provider-name/port heuristic. Regression tests should cover an arbitrary custom provider id and non-11434 proxy port, not just the built-in `ollama` provider.
