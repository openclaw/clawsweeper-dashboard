This is the same status-summary regression now tracked by the canonical open issue #73525 and open implementation PR #73662. Current main still has the split described here, but the remaining repair should be consolidated through that canonical thread rather than tracked separately.

## What This Changes

The issue requests that `openclaw status --deep` and `status --json --deep` show configured/running channels when gateway health or `openclaw channels status --probe` proves the channel is healthy.

## Best Possible Solution

Use #73525 and #73662 as the canonical tracking and implementation path, and make the final fix synthesize or merge gateway `channels.status` account snapshots into the status Channels table while preserving the local config-only fallback for unreachable gateways.

Yes. The code path itself gives a high-confidence reproduction: run `status --deep` against a reachable gateway whose `channels.status` has channel accounts while local read-only channel discovery returns no rows; the health section can be populated while the Channels table stays empty.
