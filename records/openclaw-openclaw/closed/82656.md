## Summary

Keep open. The bug is valid, but this PR is not merge-ready because the denylist still enables unknown non-empty `QQBOT_DEBUG` values, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR replaces the QQBot debug flag truthiness check with a false-like denylist and adds table tests for disabled and enabled values.

## Best Possible Solution

Make QQBot debug logging opt in only for explicit truthy values, keep the parser local to the engine boundary, cover unknown-string plus false-like/truthy cases, and attach redacted real behavior proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with `QQBOT_DEBUG=0` or `false`, the current `!!process.env.QQBOT_DEBUG` guard is truthy and the exported QQBot debug helpers emit output.

## Solution Assessment

No. The false-like denylist is a useful partial fix, but the narrower maintainable solution is an explicit truthy allowlist that matches the existing env helper contract and treats unknown strings as disabled.
