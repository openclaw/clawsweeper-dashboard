## Summary

Keep open: current main still has the source-level `store:false` path for proxy-like OpenAI Responses routes, and the open linked implementation PR has not landed. The remaining decision is provider payload policy plus real LiteLLM/proxy proof, not cleanup closure.

## What This Changes

The issue asks OpenClaw to let OpenAI Responses-compatible LiteLLM/proxy deployments persist Responses items for multi-turn continuations instead of falling back after `rs_*` item failures.

## Best Possible Solution

Review the linked provider payload-policy PR, decide whether persistence should be an explicit compat option or an approved documented default, validate a real LiteLLM/proxy multi-turn run, and close this issue only after the accepted fix lands.

## Reproduction Assessment

No live high-confidence reproduction is available because the failing LiteLLM proxy is private. Source inspection does show the current-main path where Responses item ids are replayed while `store:false` is applied to proxy-like Responses routes.

## Solution Assessment

Unclear until maintainer review settles the persistence contract. A hardcoded LiteLLM model special case is not the best solution; the maintainable path is an explicit provider/model persistence opt-in or a deliberately approved default policy for proxy-like Responses endpoints.
