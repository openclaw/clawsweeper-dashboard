## Summary

Current main now fixes the reported false-success attachment path by materializing buffer-only `send` attachments into outbound media, forwarding the resulting `mediaUrl` through channel delivery, and using Discord's existing multipart upload branch; the fixed tree is shipped in v2026.6.6.

## What This Changes

The issue reports that Discord `message.send` accepts file/path/buffer attachment inputs but returns success while delivering a Discord message with no uploaded attachment.

## Best Possible Solution

Keep the shipped shared buffer-to-media hydration path and treat the older overlapping candidate PRs as obsolete unless maintainers want separate cleanup beyond this fixed bug.

## Reproduction Assessment

Yes, source-level. The reported false-success path maps to ordinary `send` accepting `buffer` while Discord upload requires a `mediaUrl`, and current main now includes tests proving buffer-only `send` materializes into media paths.

## Solution Assessment

Yes. The current implementation stages send buffers into the shared outbound media path so Discord uses its existing multipart upload route, which is narrower and cleaner than a Discord-only special case.
