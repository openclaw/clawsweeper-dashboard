## Summary

Close as duplicate: the observable 128k status/session metadata problem is already tracked by the older open #76532, whose reproduction and suggested fix match the current source path more closely than the cache-overwrite diagnosis in this report.

## What This Changes

The issue requests that `/status` and session status report the configured 1M context window instead of a stale persisted 128k value for custom provider models.

## Best Possible Solution

Keep #76532 as the canonical fix target and resolve the provider-scoped versus bare custom model matching in `context-window-guard`, with regression coverage proving runtime persistence and status display use the configured effective context window.

## Reproduction Assessment

Yes, from source inspection and the older canonical report: a configured bare custom model can miss the strict model-id lookup, fall back to a 128k runtime default, persist that value, and have status display it. I did not run a live gateway scenario in this read-only cleanup pass.

## Solution Assessment

Yes. The narrow maintainable solution is the one tracked in #76532: fix `context-window-guard` model matching for provider-scoped and bare IDs, then cover persistence/status so stale 128k values are not produced for the configured custom model.
