## Label Justifications

- `P1`: The report describes a current-release regression that makes the Discord channel and gateway workflow unusable for an enabled user setup.
- `impact:crash-loop`: The reported gateway restart and heartbeat-failure loop is a process availability problem tied to Discord startup.
- `impact:message-loss`: The failure prevents the agent from replying through Discord and cascades into missed channel responses.

## Summary

Keep open: current main still starts the Discord startup enrichment probe that calls `/users/@me` with a 2.5s timeout, and the diff since v2026.5.22 does not remove or defer that probe. The existing tests only prove the probe is fire-and-forget, not that it cannot starve startup under the reported Windows/gateway load.

## What This Changes

The issue requests a Discord startup fix so the `/users/@me` bot/application probe cannot starve the gateway and make Discord unusable on Windows.

## Best Possible Solution

Land a focused Discord plugin fix that makes bot/application enrichment non-critical during startup, with bounded defer/backoff or circuit breaking and regression coverage that prevents tight startup probe storms.

## Reproduction Assessment

Yes for a maintainer with a Discord token and the reported Windows startup shape: restart the gateway with `channels.discord.enabled=true` and watch the `/users/@me` probe path. I did not run that live setup here, but current source still contains the implicated startup call.

## Solution Assessment

No, current main is not enough: making the probe fire-and-forget avoids direct blocking, but still starts the 2.5s `/users/@me` fetch during account startup. The narrower maintainable fix is to keep startup functional even when enrichment times out or is deferred.
