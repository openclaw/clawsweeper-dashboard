## Summary

Keep open. Current `main` still routes WhatsApp auto-replies through a delivery path that never runs `message_sending` or `message_sent` hooks. The issue discussion also points to a local-only patch / open PR for these exact files, which matches the gap still visible in the checkout.

## Evidence

- **WhatsApp auto-reply delivery bypasses outbound hooks:** `deliverWebReply()` sends text with `msg.reply()` and media with `msg.sendMedia()` after payload normalization, but this function never looks up a hook runner and never invokes `message_sending` or `message_sent`. That means cancel/modify hooks cannot affect WhatsApp auto-replies on the current path.
  - file: [extensions/whatsapp/src/auto-reply/deliver-reply.ts:25](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/extensions/whatsapp/src/auto-reply/deliver-reply.ts#L25)
  - command: `nl -ba extensions/whatsapp/src/auto-reply/deliver-reply.ts | sed -n '1,280p'`
  - sha: [9a0b26cafca7](https://github.com/openclaw/openclaw/commit/9a0b26cafca76d267dd231b5eaceadb071ec749d)
- **Buffered WhatsApp replies still flow into the hookless delivery path:** `dispatchWhatsAppBufferedReply()` hands every visible buffered payload to `params.deliverReply(...)`; once the reply is produced, WhatsApp auto-replies go through `deliverWebReply()` rather than the generic outbound delivery helper that applies outbound hooks.
  - file: [extensions/whatsapp/src/auto-reply/monitor/inbound-dispatch.ts:296](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/extensions/whatsapp/src/auto-reply/monitor/inbound-dispatch.ts#L296)

- **Core outbound delivery already applies `message_sending`:** `deliverOutboundPayloads()` explicitly calls `applyMessageSendingHook(...)` before sending each payload, so the hook contract exists on `main`; the WhatsApp auto-reply path is missing that integration rather than the platform lacking the feature.
