## Summary

Keep this PR open. Current main still has a real `web_search` schema/runtime mismatch for common providers, but this PR is not mergeable because lowering the generic schema to `MAX_SEARCH_COUNT` would under-advertise provider-specific Tavily and Exa contracts that intentionally allow more than 10 results.

## What This Changes

The PR changes the generic `web_search` count schema maximum from `20` to `MAX_SEARCH_COUNT` and adds a regression test asserting `createWebSearchTool()` exposes that shared limit.

## Best Possible Solution

Make the agent-facing `web_search` schema reflect the selected provider through lightweight provider metadata or another contract-safe seam, while preserving the common 10-result providers and the higher Tavily/Exa limits. Add regression coverage for at least one common 10-result provider, Tavily at 20, and Exa at 100.

## Reproduction Assessment

Yes. Static inspection is enough: compare the generic `createWebSearchTool()` schema, shared `resolveSearchCount()` clamp, and Tavily/Exa provider schemas/runtimes on current main; no live provider calls are needed to reproduce the contract mismatch or the PR's regression path.

## Solution Assessment

No. The PR's `MAX_SEARCH_COUNT` change fixes the common-provider over-promise by globally under-advertising providers that intentionally allow more results, so a provider-aware schema strategy is the safer maintainable fix.
