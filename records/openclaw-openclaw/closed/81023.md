## Summary

Close as superseded: the same linked sessions cleanup bug is already covered by https://github.com/openclaw/openclaw/pull/81012 with regression coverage and real behavior proof, while this PR's skip-only guard would leave the malformed cleanup row unpruned.

## What This Changes

The PR adds a colon check in `pruneMissingTranscriptEntries` so `sessions cleanup --fix-missing` skips legacy entries whose stored `sessionId` looks like a session key.

## Best Possible Solution

Close this PR and route the bug through https://github.com/openclaw/openclaw/pull/81012, which preserves strict session-id validation while pruning malformed cleanup rows with coverage and proof.

## Reproduction Assessment

Yes, at source level. A store row whose stored `sessionId` is `agent:main:main` reaches `resolveSessionFilePath` during `--fix-missing`, then the strict session-id validator rejects the colon-containing value before cleanup can continue.

## Solution Assessment

No. Skipping key-shaped values prevents the immediate exception but fails the documented `--fix-missing` behavior by leaving the malformed row in the store; the narrower maintainable fix is to treat resolver failure in this cleanup path as a missing entry, as https://github.com/openclaw/openclaw/pull/81012 does.

## Review Findings

- **[P2] Prune invalid entries instead of skipping them:** `src/config/sessions/cleanup-service.ts:212`
