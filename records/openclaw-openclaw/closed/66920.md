## Summary

Keep this issue open. The report is recent, concrete, and current main still has generic WhatsApp reconnect/watchdog handling without a group-specific recovery path for the state where DMs continue but group upserts stop after repeated 408 reconnects. The related open PR #72621 explicitly says it does not attempt the group-reliability work tracked here and in #7433.

## Best Possible Solution

Keep this open as a WhatsApp plugin reliability bug. Add a focused regression harness for repeated 408 close/reopen cycles with both DM and group traffic, then fix the WhatsApp plugin with group-aware liveness/recovery and validated Baileys group reliability hooks such as socket-connected group metadata, retry message lookup, and group metadata event handling.
