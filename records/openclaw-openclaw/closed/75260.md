## Summary

Close as ClawHub/plugin scope. The PR adds an optional Vesicle iMessage channel as a bundled OpenClaw plugin, while current main documents external ClawHub/npm plugins as the right path for optional channel integrations and already exposes the channel, HTTP route, webhook, secret, SSRF, and runtime seams this branch uses. The author's verification and prior replacement context are useful, but they do not establish a missing generic core API; the webhook ambiguity should be fixed before external publication.

## What This Changes

The PR adds a bundled `@openclaw/vesicle` channel plugin with Vesicle REST send/probe/webhook handling, docs, secret metadata, generated channel metadata, install smoke coverage, package/lockfile wiring, and channel index updates.

## Best Possible Solution

Keep OpenClaw core unchanged for this optional channel and carry Vesicle as an external ClawHub/npm channel plugin using the existing SDK seams. Reopen core work only for a concrete missing generic plugin API or an explicit maintainer decision to make Vesicle an official bundled plugin.

## Reproduction Assessment

Not applicable for the additive channel request; the scope decision is based on current main docs/source and the PR diff. For the review finding, yes: configure two enabled Vesicle accounts with the same `webhookPath` and `webhookSecret`, then send a valid signed POST and the handler selects the first matching target.

## Solution Assessment

No. Adding Vesicle as a bundled core channel is not the narrowest maintainable solution under the current plugin/ClawHub direction; the better path is external plugin publication, with ambiguous webhook-target handling fixed before release.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Reject ambiguous Vesicle webhook target matches:** `extensions/vesicle/src/webhook.ts:206-212`
  - body: The webhook handler authenticates by taking the first target whose HMAC matches. If two accounts reuse the same path and secret, an inbound request is accepted for whichever target was registered first instead of failing closed, which can misroute messages and replies across accounts. Collect matches and reject when the match count is not exactly one.
  - confidence: 0.86
