## Summary

Keep this issue open. Current main still resolves `web_fetch` provider candidates through the bundled-origin path before explicit/runtime provider matching, while the plugin docs expose `webFetchProviders` and `registerWebFetchProvider` as plugin-facing contracts; the focused fix PR #74962 is still open and unmerged, so this issue should wait for that PR or an equivalent landed fix.

## What This Changes

External plugins that declare and register web fetch providers should be usable by `web_fetch` when explicitly configured, instead of being filtered out as non-bundled providers.

## Best Possible Solution

Align `web_fetch` with the existing `web_search` runtime-provider split: use active runtime web-fetch providers across plugin origins when non-sandboxed resolution explicitly prefers runtime providers, keep sandboxed web_fetch bundled-only, and add regression coverage for an explicitly configured external provider with readability disabled.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: register a runtime-only third-party web-fetch provider, configure it explicitly with `tools.web.fetch.readability: false`, and current main still builds the candidate list from bundled providers before reaching the reported no-provider error path.

## Solution Assessment

Yes. The narrow maintainable solution is in `src/web-fetch/runtime.ts`, with tests in `src/web-fetch/runtime.test.ts` and likely `src/agents/tools/web-tools.fetch.test.ts`; documenting the contract as bundled-only would conflict with the current plugin manifest and SDK docs.
