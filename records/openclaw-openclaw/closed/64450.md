## Summary

Current `main` already contains TUI-side run-state recovery and session/model reconciliation logic that matches this report: stale/finalized runs are ignored, aborted/error runs clear active state, stuck streaming is force-reset by a watchdog, and older session snapshots no longer overwrite newer selected-model state.

## Evidence

- **Streaming watchdog resets stuck streaming state:** `createEventHandlers` now arms a watchdog on chat deltas and force-resets `activeChatRunId` plus activity status back to `idle` when stream updates stop arriving, instead of leaving the footer stuck in `streaming`.
  - file: [src/tui/tui-event-handlers.ts:93](https://github.com/openclaw/openclaw/blob/25a02825a5b81f8c5a2db980af9dbb9cc523b208/src/tui/tui-event-handlers.ts#L93)
  - command: `nl -ba src/tui/tui-event-handlers.ts | sed -n '1,260p'`
  - sha: [25a02825a5b8](https://github.com/openclaw/openclaw/commit/25a02825a5b81f8c5a2db980af9dbb9cc523b208)
- **Finalized and aborted runs no longer reassert stale streaming:** The TUI now ignores later `delta`/`final` events for already-finalized runs and explicitly tears down active run state on `aborted` and `error`, which matches the reported `run aborted` plus stale streaming/footer corruption behavior.
  - file: [src/tui/tui-event-handlers.ts:297](https://github.com/openclaw/openclaw/blob/25a02825a5b81f8c5a2db980af9dbb9cc523b208/src/tui/tui-event-handlers.ts#L297)

- **Session refresh rejects older model snapshots:** `applySessionInfo` compares `updatedAt` and refuses to let an older `sessions.list` snapshot overwrite newer session info, while `applySessionInfoFromPatch` applies resolved model info immediately. This directly addresses the visible-model drift described in the issue.
