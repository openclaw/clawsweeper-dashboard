## Summary

Current main already addresses the reported no-proxy gateway/plugin fetch failure by avoiding bundled undici global-dispatcher initialization unless a proxy dispatcher is actually needed; the related merged PR includes real Weixin login proof for the same `invalid content-length header` failure class.

## What This Changes

The issue asks OpenClaw to prevent gateway/plugin `fetch()` calls from failing when bundled undici 8.2.0 sees comma-separated `Content-Length` values after `setGlobalDispatcher()` changes process fetch behavior.

## Best Possible Solution

Keep the current lazy-loading/no-proxy dispatcher fix on main; if a configured proxy path still fails, track that as a separate report with proxy env details rather than weakening undici Content-Length validation broadly.

## Reproduction Assessment

No current-main failure path was established. Source inspection and regression coverage show the clean no-proxy gateway path no longer initializes or replaces the undici global dispatcher, while the related merged PR supplied live Weixin before/after proof for the original failure.

## Solution Assessment

Yes. Avoiding bundled undici dispatcher side effects on clean no-proxy startup is the narrow maintainable fix because it preserves normal Node fetch behavior without accepting ambiguous multi-value `Content-Length` headers or bypassing configured proxy behavior.
