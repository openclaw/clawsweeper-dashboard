## Summary

Keep open. Current main still lacks the Telegram direct `/status` path, Telegram suppressed-source fallback hook, and follow-up failure reply that this PR is trying to add, and the discussion includes fresh independent Telegram topic reproduction on v2026.4.27. The branch is not merge-ready because it now contains blocking privacy/API regressions and unrelated session-list performance work that should be split or reconciled before maintainer review.

## What This Changes

This PR changes Telegram native status delivery, Telegram group/topic reply fallback behavior, queued follow-up failure replies, Telegram sequencing/latency diagnostics, and session-list/model-cost performance paths with focused tests.

## Best Possible Solution

Land a narrower maintainer-approved patch that keeps the direct Telegram `/status` fast path and generic non-leaking follow-up failure reply, fixes the Telegram group/topic silent-drop regression without bypassing explicit message-tool-only privacy settings, and moves unrelated session-list performance work into a separate PR with client/test updates.

## Reproduction Assessment

Yes. Current main inspection plus the PR discussion gives a high-confidence reproduction path for Telegram group/topic silent drops on v2026.4.27, and the review findings have code-level reproduction paths: explicit `message_tool` group config for the fallback leak and `listSessionsFromStore({ opts: {} })` for the transcript usage regression.

## Solution Assessment

No. The direct status and failure-reply pieces are directionally useful, but the current branch is not the best solution because it overrides explicit group visibility policy and bundles unrelated session-list protocol/performance changes into a Telegram delivery fix.

Full review comments:

- **[P2] Honor explicit message-tool-only group visibility:** `extensions/telegram/src/bot-message-dispatch.ts:1087-1092`
  - body: This fallback is installed for every suppressed Telegram group turn, so when an operator explicitly configures `messages.groupChat.visibleReplies: "message_tool"` or global `messages.visibleReplies: "message_tool"`, normal assistant final/block/progress output is still sent visibly through `sendPayload`. That defeats the documented privacy setting; gate this compatibility fallback to the intended unset/default case or add an explicit opt-in.
  - confidence: 0.86

  - body: Changing `includeTranscriptUsage` to default false means all existing `sessions.list` callers that pass `{}` stop getting transcript-derived totals/cost/model fallback. Current tests and UI/native clients call without the new flag, so old sessions with only transcript usage lose their displayed usage unless every caller and expectation is updated or the API default stays compatible.
  - confidence: 0.84
