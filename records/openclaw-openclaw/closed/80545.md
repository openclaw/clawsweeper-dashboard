## Summary

Keep open for maintainer review: this is a plausible tracing API proposal, but the patch currently adds an `agent_start` runner without defining the hook contract it depends on, and the PR body says the broader check already fails. The external PR also lacks after-fix real behavior proof.

## What This Changes

Adds an `agent_start` hook path, threads parent run/session identifiers through subagent and gateway agent calls, and extends LLM hook ordering coverage for Opik trace-tree correlation.

## Best Possible Solution

If maintainers want this tracing surface, land a generic plugin-hook contract update that defines the event type, hook name, handler map, registration API, docs/tests, and subagent run-id threading without Opik-specific core policy.

## Reproduction Assessment

Yes for the review finding: source inspection shows `agent_start` is used by the patch but is absent from the current hook-name union and handler map. I did not run checks because this review is read-only, but the PR body independently reports the broad check failing on this same contract mismatch.

## Solution Assessment

No; adding a runner method before defining the public hook contract is not the narrow maintainable path. The safer direction is a generic tracing API change that updates hook types, SDK registration, docs, tests, and real runtime proof together.
