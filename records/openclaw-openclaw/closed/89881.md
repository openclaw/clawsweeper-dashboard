## Summary

Close: current main and the v2026.5.28 release source already route memory-triggered flush runs through append-only write wrapping; the cited suppress flag is a generic transcript-persistence mode, not the memory-flush guard boundary.

## What This Changes

The issue reports that pre-compaction daily memory flushes can silently overwrite `memory/YYYY-MM-DD.md` when user-message persistence is suppressed.

## Best Possible Solution

Keep the explicit `trigger: "memory"` plus `memoryFlushWritePath` guard and avoid broadening memory tool policy to generic suppressed transcript retries.

## Reproduction Assessment

No. Source inspection shows current main's pre-compaction memory flush path reaches tool construction with `trigger: "memory"` and `memoryFlushWritePath`, and the tests cover append-only behavior rather than a failing current-main path.

## Solution Assessment

Yes. The current implementation uses explicit memory-flush metadata as the maintainable boundary; using `suppressNextUserMessagePersistence` alone would apply memory-write restrictions to unrelated suppressed retry and room-event turns.
