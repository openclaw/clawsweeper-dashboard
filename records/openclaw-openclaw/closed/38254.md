## Summary

Keep open. Current `main` does not implement `.env` variable expansion for loaded env files, so this PR's request is not already fixed on `main` and there is not strong evidence for any closure reason.

## Evidence

- **Current dotenv reader keeps parsed values literal:** `readDotEnvFile` parses file contents with `dotenv.parse(content)` and pushes each parsed `value` into `entries` without any second-pass `${VAR}` / `$VAR` expansion step.
  - file: [src/infra/dotenv.ts:127](https://github.com/openclaw/openclaw/blob/53aac30f5104052db56da2b26bcee0f4e5140325/src/infra/dotenv.ts#L127)

  - sha: [53aac30f5104](https://github.com/openclaw/openclaw/commit/53aac30f5104052db56da2b26bcee0f4e5140325)
- **Loaded values are copied directly into process.env:** Both `loadRuntimeDotEnvFile` and `loadWorkspaceDotEnvFile` assign each parsed `value` straight to `process.env[key]` when unset. That preserves raw literals from `.env` files instead of resolving references.
  - file: [src/infra/dotenv.ts:166](https://github.com/openclaw/openclaw/blob/53aac30f5104052db56da2b26bcee0f4e5140325/src/infra/dotenv.ts#L166)
