## Label Justifications

- `P2`: The request concerns a normal-priority plugin API and safety improvement with important but bounded impact.
- `impact:security`: The requested hook is intended to support safety, policy, and truthful reporting around tool execution results.
- `impact:session-state`: The issue is about which tool-result content reaches model context and transcript state.

## Summary

Current main already provides the requested per-tool execution boundary: `before_tool_call` can rewrite/block before execution, `after_tool_call` observes results, and the shipped `registerAgentToolResultMiddleware` API lets explicitly enabled plugins transform tool results before the OpenClaw or Codex runtime feeds them back to the model.

## What This Changes

The issue requests synchronous plugin hooks around individual tool executions so plugins can inspect, veto, annotate, or substitute command results before the model sees them.

## Best Possible Solution

Keep the shipped `registerAgentToolResultMiddleware(...)` contract as the pre-model transform seam and avoid changing `after_tool_call` from observation into mutation.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a plugin API feature request. Source, docs, and release inspection show the requested pre-model tool-result mutation boundary is already present.

## Solution Assessment

Yes. The shipped manifest-gated middleware API is the maintainable solution because it preserves `after_tool_call` as observation while giving trusted plugins an awaited pre-model result transform.
