## Summary

Keep open. The performance goal is valid and the body includes useful local startup proof, but the current patch can bypass root-read safety options and also shares one cache across two different JSON reader return contracts.

## What This Changes

The PR adds a process-local stat-validated sync JSON read cache in `src/infra/json-files.ts`, routes package update root reads through it, and keeps `loadJsonFile` on the raw reader.

## Best Possible Solution

Keep the startup performance work only if cached reads preserve the full fs-safe contract; prefer domain-scoped manifest or snapshot caching, or partition and test this primitive cache by API contract and every safety-relevant option.

## Reproduction Assessment

Yes for the PR-introduced defects: source inspection shows cache hits return before fs-safe sees later root-read options, and the same map stores incompatible return shapes for two exported readers. I did not run tests because this review is read-only.

## Solution Assessment

No. The narrow maintainable path is to keep the performance win at the plugin snapshot/discovery layer, or make this cache preserve reader contracts and root-read safety options with focused regression coverage.
