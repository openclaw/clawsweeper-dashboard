## Summary

Current `main` already fixes the reported `workspaceOnly` inbound-image path gap through centralized inbound-media reference handling in the media layer, with targeted regression coverage for allowed inbound absolute paths and continued rejection of outbound/nested/symlink cases.

## Evidence

- **Old behavior matched the PR bug report:** In `v2026.4.23`, `loadImageFromRef` still enforced `assertSandboxPath` for `workspaceOnly` non-sandbox loads and then called `loadWebMedia` without a scoped local-root allowlist, which matches the PR's described `Path escapes sandbox root` failure mode for absolute inbound media paths.
  - file: [src/agents/pi-embedded-runner/run/images.ts:418](https://github.com/openclaw/openclaw/blob/a9797214338ba31c52c796adbb75afb16e0684a9/src/agents/pi-embedded-runner/run/images.ts#L418)
  - command: `git show a9797214:src/agents/pi-embedded-runner/run/images.ts | nl -ba | sed -n '418,434p'`
  - sha: [a9797214338b](https://github.com/openclaw/openclaw/commit/a9797214338ba31c52c796adbb75afb16e0684a9)
- **Current runner path uses media-layer validation:** On current `main`, `loadImageFromRef` no longer calls `assertSandboxPath` in the non-sandbox `workspaceOnly` branch. It delegates to `loadWebMedia(targetPath, { localRoots: [workspaceDir] })`, so absolute-path decisions now flow through the centralized local-media policy instead of being lexically rejected first.
