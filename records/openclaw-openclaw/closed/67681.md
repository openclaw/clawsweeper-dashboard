## Summary

Current main already implements the no-deliver behavior needed for this report: `--no-deliver` maps to `delivery.mode: "none"`, cron delivery planning marks that as not requested, and the outbound final-reply dispatch path only runs when delivery is requested. Later cron delivery fixes after the reporter's v2026.4.15 follow-up are documented in the 2026.4.20 changelog, including avoiding duplicate fallback sends after direct message-tool delivery.

## Best Possible Solution

Keep the shipped cron delivery contract: `--no-deliver` / `delivery.mode: "none"` suppresses runner-owned final-reply fallback delivery while still allowing explicit script or message-tool sends. If maintainers want a stricter mode that blocks all message tools as well, that should be a separate feature request for a hard silent/no-message-tool option.
