## Summary

Current `main` already contains the reported Telegram stream-order cleanup and follow-on preview/finalization recovery logic, with regression coverage for late assistant-message boundaries and incomplete preview finalization. The raw `message_start`/`message_stop` failure described against `v2026.3.24` is no longer the current `main` behavior.

## Evidence

- **User-facing stream-order error is now sanitized:** Current main rewrites the exact invalid event-order failure into a safe retry message instead of leaking `Unexpected event order, got message_start before receiving "message_stop"` to chat surfaces.
