## Summary

Keep open. Current main still routes Anthropic `paste-token` input through the setup-token validator, so a standard `sk-ant-api03-` API key reaches the same `sk-ant-oat01-` rejection before any credential can be stored.

## What This Changes

The issue asks for Anthropic `paste-token` to accept standard `sk-ant-api03-` API keys or provide a clear API-key auth path.

## Best Possible Solution

Teach the Anthropic paste-token path to detect API-key input and either persist it through the existing API-key credential/config semantics or fail with the exact API-key auth command before storing anything.

## Reproduction Assessment

Yes. Source inspection on current main gives a deterministic path: `openclaw models auth paste-token --provider anthropic` validates Anthropic input with `validateAnthropicSetupToken`, which rejects anything not starting `sk-ant-oat01-`; I did not run the interactive CLI in this read-only review.

## Solution Assessment

Yes for keeping this open, but no for the closed validator-only approach. The maintainable fix should preserve setup-token behavior while routing `sk-ant-api03-` through API-key auth semantics or an actionable API-key redirect.
