## Summary

Current `main` already mirrors outbound `openclaw message send` deliveries into the matching session transcript, which covers the reported need for external notifications to appear in conversation context for the next reply.

## Evidence

- **CLI send now prepares a transcript mirror for outbound notifications:** `runMessageAction()` derives an outbound session route from channel/target and passes `mirror: { sessionKey, agentId, text, mediaUrls }` into send execution, so ordinary `openclaw message send` calls are no longer channel-only.

- **Gateway send attaches mirror metadata to the resolved session:** Gateway `send` resolves the outbound session from the target, ensures the session entry exists, and passes mirror metadata so the delivered message is written back to that session transcript.
  - file: [src/gateway/server-methods/send.ts:482](https://github.com/openclaw/openclaw/blob/6b618f063516aa9ba85903aa4565c5a204c055b2/src/gateway/server-methods/send.ts#L482)

## Risks / Open Questions

- The current implementation mirrors outbound notifications as assistant transcript entries, not a caller-selectable `system` role. That is sufficient for the reported context problem, but it does not match the exact proposed API surface.
