## Summary

Keep open. Current `main` still builds the temporary download path from the remote Signal asset name before validating it, so the traversal/overwrite concern described in PR #44104 is still present. The PR also carries useful regression coverage that is not on `main`, although it likely needs rebasing because the code now lives under `extensions/signal/` instead of `src/commands/`.

## Evidence

- **Current main still joins untrusted asset names into the temp path:** `installSignalCliFromRelease` creates `archivePath` with `path.join(tmpDir, asset.name)`, downloads to that path, and only then checks whether the name looks like a supported archive. That means remote release metadata still controls the local write path on current `main`.
  - file: [extensions/signal/src/install-signal-cli.ts:250](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/extensions/signal/src/install-signal-cli.ts#L250)
  - command: `nl -ba extensions/signal/src/install-signal-cli.ts | sed -n '244,262p'`
  - sha: [5fe333ada85a](https://github.com/openclaw/openclaw/commit/5fe333ada85afd0def9e85985ce57e36aa00108a)
- **Reasonable reproduction still escapes the temp directory:** A direct Node path check on current semantics shows `path.join("/tmp/openclaw-signal-abc123", "../escape.tar.gz")` resolves to `/tmp/escape.tar.gz`, and `../../etc/passwd.tar.gz` resolves outside the temp directory entirely. That matches the PR's security concern.
  - command: `node -e 'const path=require("node:path"); const tmp="/tmp/openclaw-signal-abc123"; console.log(path.join(tmp,"../escape.tar.gz")); console.log(path.join(tmp,"nested/escape.tgz")); console.log(path.join(tmp,"../../etc/passwd.tar.gz"));'`
