## Summary

Current `main` already includes a TUI-side streaming watchdog that resets the activity indicator back to `idle` after delta silence, and the changelog says that fix shipped in `v2026.4.15` specifically to prevent the TUI from staying stuck on `streaming` after lost or late final events.

## Evidence

- **Release note documents the fix:** The `2026.4.15` changelog explicitly says TUI streaming now has a client-side watchdog that resets `streaming` to `idle` after 30s of delta silence, guarding against lost or late `state: "final"` chat events leaving the TUI stuck indefinitely.
  - file: [CHANGELOG.md:788](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/CHANGELOG.md#L788)

- **Current handler arms a watchdog on every delta:** `handleChatEvent` now sets `streaming` on each delta and calls `armStreamingWatchdog(evt.runId)` for the active run, so the UI has an explicit fallback path even if the normal final-event ordering breaks.
  - file: [src/tui/tui-event-handlers.ts:313](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/src/tui/tui-event-handlers.ts#L313)

## Risks / Open Questions

- The shipped fix is a watchdog-based recovery, not the exact immediate event-ordering rewrite proposed in the issue, so a late or lost final can still leave `streaming` visible until the watchdog window expires.
