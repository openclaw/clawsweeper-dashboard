## Summary

Current main and the latest stable release already address the WebChat duplicate-turn behavior reported against 2026.4.24. The strongest match is the Control UI duplicate-submit fix: current code coalesces identical in-flight sends before a second chat.send run is created, while adjacent WebChat history/final-reload fixes cover the related duplicate assistant rendering paths.

## What This Changes

The issue requests that WebChat render each turn once, with a single user message and a single assistant response.

## Best Possible Solution

Keep the shipped Control UI submit coalescing and final-reload reconciliation in place, and direct users on 2026.4.24 to upgrade to v2026.4.27 or newer rather than carrying another duplicate WebChat issue.

## Reproduction Assessment

Yes. The issue has a high-confidence reproduction class on 2026.4.24: duplicate rapid sends before chat.send acknowledgement could create two UI turns; current tests reproduce that timing with two handleSendChat/sendChatMessage calls and verify only one request/message remains.

## Solution Assessment

Yes. The current fix is the narrow maintainable solution: coalesce duplicate UI submits before minting a new run id, keep a controller-level chatSending guard, and leave gateway idempotency as the backend defense for reused keys.
