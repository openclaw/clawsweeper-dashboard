## Summary

Current `main` already fixes the `models list` display-path normalization bug: configured model refs are parsed with plugin/runtime normalization disabled, configured rows keep the original `provider/model` key for display, and the same code shipped in `v2026.4.23`.

## Evidence

- **Models-list parsing disables runtime provider/model normalization:** `resolveConfiguredEntries()` uses `DISPLAY_MODEL_PARSE_OPTIONS = { allowPluginNormalization: false }` for default, alias, fallback, and configured model parsing, which keeps display refs from being rewritten through provider runtime normalization.
  - file: [src/commands/models/list.configured.ts:15](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/src/commands/models/list.configured.ts#L15)
  - sha: [59523e66da03](https://github.com/openclaw/openclaw/commit/59523e66da03591bc7bcc183a11f690fa0aae641)
- **Normalization helper returns static model ids when disabled:** `normalizeProviderModelId()` returns the static model id unchanged when `allowPluginNormalization` is false, so display/status surfaces do not remap Gemini ids through another provider plugin.
  - file: [src/agents/model-selection-normalize.ts:44](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/src/agents/model-selection-normalize.ts#L44)
  - sha: [59523e66da03](https://github.com/openclaw/openclaw/commit/59523e66da03591bc7bcc183a11f690fa0aae641)
- **Configured rows print the preserved configured key:** `appendConfiguredRows()` resolves registry metadata, but still passes `entry.key` into `toModelRow()`, so the table output uses the configured `provider/model` label rather than a runtime-normalized provider prefix.
  - file: [src/commands/models/list.rows.ts:298](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/src/commands/models/list.rows.ts#L298)
  - sha: [59523e66da03](https://github.com/openclaw/openclaw/commit/59523e66da03591bc7bcc183a11f690fa0aae641)
- **Regression test locks this behavior:** The models-list regression test mocks provider-runtime normalization to throw and still expects configured entries to stay as their configured keys, proving the display path must not load runtime normalization.
  - file: [src/commands/models/list.configured.test.ts:3](https://github.com/openclaw/openclaw/blob/a9797214338ba31c52c796adbb75afb16e0684a9/src/commands/models/list.configured.test.ts#L3)
  - sha: [a9797214338b](https://github.com/openclaw/openclaw/commit/a9797214338ba31c52c796adbb75afb16e0684a9)
- **Shipped in the latest release:** The released `v2026.4.23` tree contains the same `allowPluginNormalization: false` models-list logic, so this was already shipped beyond `2026.4.1`, the version named in the report.
  - file: [src/commands/models/list.configured.ts:15](https://github.com/openclaw/openclaw/blob/a9797214338ba31c52c796adbb75afb16e0684a9/src/commands/models/list.configured.ts#L15)
  - sha: [a9797214338b](https://github.com/openclaw/openclaw/commit/a9797214338ba31c52c796adbb75afb16e0684a9)

  - sha: [9e2e1a329c33](https://github.com/openclaw/openclaw/commit/9e2e1a329c3348ba49b8135935a76caccc1960d2)
