## Summary

Close as implemented: current main already uses the memory-core manifest `configSchema` to validate `plugins.entries.memory-core.config.dreaming` and passes the validated config into the plugin runtime, so adding a duplicate schema to `definePluginEntry` is not needed for this native plugin path.

## What This Changes

Requests that memory-core declare a config schema so `plugins.entries.memory-core.config.dreaming` is accepted and reaches the dreaming runtime.

## Best Possible Solution

Keep the manifest-owned memory-core schema as the source of truth; use #67868 separately for any remaining status-rendering problem if it still reproduces.

## Reproduction Assessment

No. Source inspection of current main and shipped `v2026.5.2` shows `plugins.entries.memory-core.config.dreaming` is accepted by the memory-core manifest schema and passed through loader validation into plugin runtime config.

## Solution Assessment

Yes. The manifest-first schema is the maintainable plugin contract because OpenClaw validates native plugin config before runtime entry modules load; duplicating the same schema in `definePluginEntry` would add drift.
