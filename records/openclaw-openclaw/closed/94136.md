## Summary

Keep this PR open: the provider-owned GLM-5.2 mapping is the right layer, but the current profile can silently remap an existing/stored `minimal` thinking setting to GLM-5.2 `max`, so it needs a narrow compatibility fix before merge.

## What This Changes

The PR updates the bundled Z.AI provider and docs/tests so `zai/glm-5.2` exposes `off`, `low`, `high`, and `max` thinking levels and maps those levels into Z.AI request payload fields.

## Best Possible Solution

Preserve stored/settings compatibility by making GLM-5.2 below-low fallback resolve to `low`, `off`, or an explicit skip-thinking payload, then keep the provider-owned `reasoning_effort` mapping with focused tests.

## Reproduction Assessment

Yes, source-level. On the PR head, `resolveSupportedThinkingLevel` with a GLM-5.2 profile of `off|low|high|max` and requested `minimal` falls through to the highest non-off level, `max`.

## Solution Assessment

No, not as-is. The provider-owned mapping is the right layer, but the GLM-5.2 profile or fallback coverage needs to preserve existing low/minimal settings semantics before merge.
