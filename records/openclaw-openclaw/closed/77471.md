## Summary

This is a useful QMD MCP performance idea, but current main already keeps QMD behind the memory plugin/runtime boundary and the direct QMD HTTP service adapter fits ClawHub/plugin scope better than a new core provider mode.

## What This Changes

Request to add a `memory.provider: "qmd-mcp"`/endpoint path that sends memory search to a warm QMD MCP HTTP service instead of cold-starting the existing local QMD search path.

## Best Possible Solution

Publish a QMD MCP HTTP memory plugin through ClawHub, or use the existing `memory.qmd.mcporter` keep-alive path, while keeping core on the generic memory runtime/plugin contract.

## Reproduction Assessment

Not applicable: this is primarily a feature/plugin-boundary request. Source inspection confirms current main has QMD and mcporter-backed memory paths but no direct core `memory.provider: "qmd-mcp"` endpoint mode.

## Solution Assessment

Yes for cleanup: the narrow maintainable answer is not a new core provider id, but a ClawHub memory plugin or the existing mcporter-backed QMD path. That avoids baking one QMD HTTP service shape into core.
