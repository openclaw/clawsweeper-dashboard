## Summary

Keep open. Current main still hard-caps every Signal HTTP RPC response at 1 MiB before `fetchAttachment` can apply the configured media limit, and this open PR is the active focused implementation candidate for the paired Signal attachment bug. The earlier Greptile fallback concern was addressed in the second PR commit, but the branch is currently dirty against main and needs maintainer review/rebase before landing.

## What This Changes

The PR threads an optional Signal RPC `maxResponseBytes` cap through the HTTP client, derives the `getAttachment` cap from `mediaMaxMb` with base64/JSON headroom, adds Signal client regression tests, and updates the changelog.

## Best Possible Solution

Land a focused Signal-owned fix that keeps the default 1 MiB guard for ordinary RPCs while sizing only `getAttachment` responses from the effective media limit, with regression coverage for both the client cap and the attachment call path; then close #73564 as implemented.

## Reproduction Assessment

Yes. Current main shows the reproduction path statically: a base64 JSON `getAttachment` response over 1 MiB is rejected in `requestSignalHttpText` before `fetchAttachment` can enforce `params.maxBytes`, matching the reporter's live logs and config.

## Solution Assessment

Yes. The proposed direction is the narrowest maintainable fix: thread a bounded optional response cap through the Signal RPC client and set it only at the attachment call site, rather than removing the memory guard or adding a second user-facing config knob.
