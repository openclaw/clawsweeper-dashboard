## Summary

Current `main` fixes the actionable part of this report: Google Chat app-url auth failures are no longer silent, and startup now warns when `appPrincipal` is missing or looks like the service-account email instead of the numeric add-on principal. The `sub`-binding behavior is explicit in code and covered by tests. This appears to be fixed on `main` but not yet in the latest tagged release.

## Evidence

- **Auth still binds app-url add-on tokens to JWT `sub`:** `verifyGoogleChatRequest()` normalizes `expectedAddOnPrincipal`, compares it against `payload.sub`, and returns explicit rejection reasons for missing or mismatched bindings.
  - file: [extensions/googlechat/src/auth.ts:169](https://github.com/openclaw/openclaw/blob/c65aa1d2a6e573d49619c4dd0698a76010476e7d/extensions/googlechat/src/auth.ts#L169)
  - sha: [c65aa1d2a6e5](https://github.com/openclaw/openclaw/commit/c65aa1d2a6e573d49619c4dd0698a76010476e7d)
- **Tests cover missing, matching, and mismatched principal cases:** Current tests explicitly assert `missing add-on principal binding`, success when the binding matches `sub`, and `unexpected add-on principal: ...` when it does not.
  - file: [extensions/googlechat/src/targets.test.ts:333](https://github.com/openclaw/openclaw/blob/c65aa1d2a6e573d49619c4dd0698a76010476e7d/extensions/googlechat/src/targets.test.ts#L333)
  - sha: [c65aa1d2a6e5](https://github.com/openclaw/openclaw/commit/c65aa1d2a6e573d49619c4dd0698a76010476e7d)
- **Webhook auth failures are now logged with the concrete reason:** The Google Chat webhook handler records `Google Chat webhook auth rejected: <reason>` for rejected targets instead of returning a silent 401 path.
  - file: [extensions/googlechat/src/monitor-webhook.ts:122](https://github.com/openclaw/openclaw/blob/c65aa1d2a6e573d49619c4dd0698a76010476e7d/extensions/googlechat/src/monitor-webhook.ts#L122)
  - sha: [c65aa1d2a6e5](https://github.com/openclaw/openclaw/commit/c65aa1d2a6e573d49619c4dd0698a76010476e7d)
- **Startup warns when `appPrincipal` is missing or looks like an email:** `warnAppPrincipalMisconfiguration()` tells operators to use the numeric OAuth 2.0 client ID (`uniqueId`, 21 digits), not an email, for `audienceType: "app-url"`.
  - file: [extensions/googlechat/src/monitor-webhook.ts:163](https://github.com/openclaw/openclaw/blob/c65aa1d2a6e573d49619c4dd0698a76010476e7d/extensions/googlechat/src/monitor-webhook.ts#L163)
  - sha: [c65aa1d2a6e5](https://github.com/openclaw/openclaw/commit/c65aa1d2a6e573d49619c4dd0698a76010476e7d)
- **Google Chat monitor calls the warning on startup:** `monitorGoogleChatProvider()` invokes the misconfiguration warning before registering the webhook target, so bad `appPrincipal` values are surfaced during normal startup.
  - file: [extensions/googlechat/src/monitor.ts:347](https://github.com/openclaw/openclaw/blob/c65aa1d2a6e573d49619c4dd0698a76010476e7d/extensions/googlechat/src/monitor.ts#L347)
  - sha: [c65aa1d2a6e5](https://github.com/openclaw/openclaw/commit/c65aa1d2a6e573d49619c4dd0698a76010476e7d)
- **Latest release predates the logging/warning change:** In `v2026.4.23`, `extensions/googlechat/src/monitor-webhook.ts` still used the older boolean-only auth path and returned 401 for missing add-on tokens without logging. `git diff a9797214338ba31c52c796adbb75afb16e0684a9..HEAD -- extensions/googlechat/src/monitor-webhook.ts` shows the logging and warning additions only after that release.
  - file: [extensions/googlechat/src/monitor-webhook.ts:104](https://github.com/openclaw/openclaw/blob/a9797214338ba31c52c796adbb75afb16e0684a9/extensions/googlechat/src/monitor-webhook.ts#L104)

## Risks / Open Questions

- `docs/channels/googlechat.md` still does not appear to document `appPrincipal` explicitly; this close is based on the runtime/logging fix on `main`, not on a docs backfill.
- `channels status --probe` and doctor do not currently add an `appPrincipal`-specific warning; operators rely on startup/log output for this case.
