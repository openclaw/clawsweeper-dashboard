## Summary

Keep open. Current main does not already include these three added surfaces, and the latest PR diff still has blocking workflow correctness and supply-chain issues around Dev Container pnpm setup, privileged benchmark tooling install, SBOM target accuracy, and duplicate release upload behavior.

## What This Changes

The PR adds a Dev Container configuration plus GitHub Actions workflows for hyperfine-based benchmarking and release-time SBOM generation/attachment.

## Best Possible Solution

Land only a revised, focused implementation that prepares pnpm reliably in the Dev Container, reuses the repository setup action and existing startup benchmark harness, and generates a consumer-facing SBOM from the prepared npm tarball or production dependency set through one permissions-minimal upload path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Enable pnpm before installing in the dev container:** `.devcontainer/devcontainer.json:7`
  - body: The selected base image does not document a `pnpm` binary, and the repo's own setup paths enable/prepare pnpm through Corepack before use. A fresh Dev Container can fail at `postCreateCommand` with `pnpm` unavailable; use `corepack pnpm install --frozen-lockfile` or prepare pnpm in the image first.
  - confidence: 0.74
- **[P2] Verify hyperfine before installing it with sudo:** `.github/workflows/benchmark.yml:31-32`
  - body: The benchmark job downloads a `.deb` from GitHub releases and immediately runs `sudo dpkg -i` on it. Add a pinned checksum/signature check or install hyperfine from a trusted package source so a compromised release asset cannot execute arbitrary code in CI.
  - confidence: 0.9
- **[P2] Generate the SBOM from the release artifact:** `.github/workflows/sbom.yml:23-27`
  - body: This job installs the full workspace and scans `path: .`, so the SBOM attached to a release can include dev/test tooling and miss the exact npm tarball boundary consumers receive. Scan the prepared release package or a production dependency set instead.
  - confidence: 0.87
- **[P2] Use one SBOM release upload path:** `.github/workflows/sbom.yml:25-40`
  - body: `anchore/sbom-action` uploads artifacts and release assets by default on release events, but the job also uploads the same file as an artifact and attaches it with `softprops/action-gh-release`. Disable the Anchore upload modes or remove the duplicate steps and grant the permissions required by the chosen path.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff adds CI/release code execution and release-asset paths with concrete supply-chain and SBOM accuracy concerns.

Concerns:

- **[medium] Unverified privileged package install:** `.github/workflows/benchmark.yml:31`
  - body: The benchmark job downloads a Hyperfine `.deb` and installs it with `sudo` without verifying a checksum or signature, so a compromised release asset or delivery path could execute code in CI.
  - confidence: 0.9
- **[medium] SBOM target does not match shipped artifact:** `.github/workflows/sbom.yml:27`
  - body: The release job scans the full checkout after a full dependency install, which can publish an SBOM for development inputs rather than the actual npm release tarball or production dependency set.
  - confidence: 0.87
- **[low] Duplicate release upload behavior:** `.github/workflows/sbom.yml:25`
  - body: Anchore's action defaults to artifact and release-asset uploads, while this workflow also uses separate upload-artifact and release attachment steps; the upload owner and permissions need to be made explicit.
  - confidence: 0.84
