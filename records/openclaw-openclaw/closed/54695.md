## Summary

Current main already contains the Telegram/Groq inbound STT fix path: Telegram still uses shared audio preflight, Groq still uses the OpenAI-compatible multipart transcription helper, that helper now disables pinned DNS for multipart STT uploads, and the guarded runtime fetch layer preserves multipart FormData instead of sending a broken body. The changelog records these fixes in the 2026.4.11 release.

## Evidence

- **Telegram inbound voice notes still flow through shared preflight transcription:** Telegram replaces `<media:audio>` with the preflight transcript from `transcribeFirstAudio(...)`, so the reported Telegram voice-note path still goes through the shared STT stack.
  - file: [extensions/telegram/src/bot-message-context.body.ts:220](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/extensions/telegram/src/bot-message-context.body.ts#L220)
- **Shared preflight calls the audio transcription runner:** `transcribeFirstAudio(...)` calls `runAudioTranscription(...)` and returns the transcript when available, confirming the issue is in the shared media-understanding transcription path rather than Telegram-only parsing.
  - file: [src/media-understanding/audio-preflight.ts:49](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/src/media-understanding/audio-preflight.ts#L49)
- **Groq uses the shared OpenAI-compatible multipart STT helper:** The Groq media-understanding provider delegates `transcribeAudio` to `transcribeOpenAiCompatibleAudio(...)`, so fixes in that helper apply directly to the reported Telegram + Groq configuration.
  - file: [extensions/groq/media-understanding-provider.ts:14](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/extensions/groq/media-understanding-provider.ts#L14)
- **OpenAI-compatible STT now disables pinned DNS for multipart uploads:** `transcribeOpenAiCompatibleAudio(...)` sends the multipart body through `postTranscriptionRequest(...)` with `pinDns: false`, matching the reporter's workaround hypothesis for Groq/OpenAI-compatible transcription failures.
  - file: [src/media-understanding/openai-compatible-audio.ts:53](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/src/media-understanding/openai-compatible-audio.ts#L53)
- **Guarded runtime fetch now preserves multipart FormData correctly:** `normalizeRuntimeFormData(...)` rebuilds cross-implementation FormData bodies and strips stale `content-type`/`content-length` headers before runtime dispatch, directly addressing the reported `multipart/form-data` corruption/HTTP 400 failure mode.
  - file: [src/infra/net/runtime-fetch.ts:21](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/src/infra/net/runtime-fetch.ts#L21)
- **Regression coverage exists for both multipart fixes:** There is explicit test coverage for runtime multipart FormData normalization and for the OpenAI-compatible transcription helper forcing `pinDns: false` on multipart STT requests.
  - file: [src/infra/net/runtime-fetch.test.ts:48](https://github.com/openclaw/openclaw/blob/a6aa151653ab478e9aa23e95c6446c41adc4fce1/src/infra/net/runtime-fetch.test.ts#L48)
- **Changelog says the fix shipped in 2026.4.11:** The changelog records two relevant fixes: guarded runtime fetch no longer drops multipart FormData fields for audio transcription, and OpenAI-compatible multipart transcription disables pinned DNS so OpenAI, Groq, and Mistral transcription works again.
