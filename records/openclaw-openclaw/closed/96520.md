## Summary

Current main already covers the central opencode-go failure: the provider-owned runtime/static catalog contains the reported Anthropic-message models with the correct per-model API/base URL, and live discovery no longer caches empty model-id results or lets failed discovery replace the static fallback.

## What This Changes

The issue asks to make opencode-go Anthropic-message models reliable by syncing catalog metadata and making live model discovery resilient to transient failures.

## Best Possible Solution

Keep the provider-owned runtime/static catalog and live-discovery fallback path on main; avoid persisting full opencode-go model rows into user config unless a future migration design explicitly needs it.

## Reproduction Assessment

No high-confidence current-main failure reproduction remains: source inspection shows the affected model IDs resolve with Anthropic-message metadata and failed/empty live discovery falls back to the static catalog. I did not run an authenticated inference request against OpenCode Go.

## Solution Assessment

Yes. The current provider-owned static/runtime catalog plus shared live-discovery fallback is the maintainable fix; writing the full catalog into user config would risk stale metadata pinning.
