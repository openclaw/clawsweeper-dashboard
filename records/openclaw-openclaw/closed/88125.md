## Summary

Current main and the latest release do not match the reported auth-profile shape: OpenAI Codex OAuth flows store the auth-profile provider as `openai-codex`, and the docs already frame `openai` as the model route while keeping `openai-codex` as the auth/profile provider id.

## What This Changes

The issue asks for docs to say that `openclaw models auth login --provider openai-codex` stores auth profiles with `provider: "openai"` instead of `provider: "openai-codex"`.

## Best Possible Solution

Keep the current source and docs distinction: `openai` is the canonical model route, while `openai-codex` remains the auth/profile provider id unless a new redacted current-release reproduction proves otherwise.

## Reproduction Assessment

No. I did not run a live OAuth login in this read-only review, but current main, the latest release tag, and existing tests all show generated Codex OAuth profiles using `provider: "openai-codex"`, not `provider: "openai"`.

## Solution Assessment

No. Adding the suggested note would document the opposite of the current source and release behavior; the safer path is to request a new redacted reproduction if a current install really generates `provider: "openai"`.
