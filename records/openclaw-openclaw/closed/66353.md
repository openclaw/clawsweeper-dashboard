## Summary

The reported behavior is already covered by shipped Browser/CDP fixes and current tests: OpenClaw now reuses a healthy pre-existing loopback Chrome/CDP endpoint, marks it as running in `browser status`, and only raises `not reachable after start` after explicit CDP-ready polling. The changelog places the matching fix in the 2026.4.14 release line, after the reporter's 2026.4.12 build.

## Evidence

- **Changelog matches the reported symptom:** The 2026.4.14 changelog explicitly says managed local Chrome readiness and status probes now bypass browser SSRF for the loopback control plane so OpenClaw no longer misclassifies a healthy child browser as `not reachable after start`, and that local loopback `manual-cdp` profiles now reuse the local loopback CDP control plane so live local sessions stop showing as not running.

- **Current availability logic reuses an existing healthy loopback browser:** `ensureBrowserAvailable()` now gives an unmanaged loopback profile a longer retry window before launching anything. If the retry finds HTTP and WebSocket/CDP readiness on the existing endpoint, it returns successfully instead of relaunching Chrome; the `not reachable after start` error is only thrown after explicit CDP-ready polling for a launched browser.
