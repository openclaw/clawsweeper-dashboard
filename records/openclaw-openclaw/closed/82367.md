## Summary

Current main now addresses the reported stuck-session model behavior: auto fallback pins periodically probe and clear back to the configured primary, while exact configured custom Ollama refs are accepted by the allowlist path even when absent from the bundled catalog.

## What This Changes

The issue asks for default model resolution and `/model` allowlisting to honor `agents.defaults.model.primary` and custom `ollama/<model>` entries instead of sticking on `ollama/qwen3.6:35b`.

## Best Possible Solution

Keep the current-main session fallback recovery and exact allowlist behavior; any remaining custom Ollama rejection on a newer build should be filed as a fresh, config-backed report.

## Reproduction Assessment

Yes, source-level reproduction is high confidence for the sticky session-model part: the screenshot shows default gemma with active qwen session state, and the canonical fallback-pinning issue documents the same persisted-session pathology. I did not run a live Ollama 0.24.x repro.

## Solution Assessment

Yes. Fixing fallback/session recovery in the generic agent model-selection layer, while keeping exact custom provider/model refs accepted by the allowlist, is the narrow maintainable solution rather than adding Ollama-library-specific policy.
