## Summary

Keep this PR open: current main still lacks native WhatsApp location sends, and the earlier related WhatsApp location PRs were closed unmerged. The branch is not correct yet because PR head still has a broken fallback gateway method plus CI-reproduced WhatsApp typing and lint issues.

## What This Changes

The PR adds WhatsApp native location pins across the message tool schema, gateway send parameters, outbound action runner, WhatsApp adapter/runtime, protocol artifacts, CLI flags, and focused tests.

## Best Possible Solution

Land one coordinated native location implementation that uses a registered gateway surface, keeps WhatsApp delivery plugin-owned, preserves metadata/session mirroring, keeps generated protocol artifacts aligned, and passes focused WhatsApp, gateway, outbound, protocol, build, and changed checks.

## Reproduction Assessment

Yes for the PR defects: source inspection of PR head shows the core fallback dispatches an unregistered `location` gateway method, and CI logs reproduce the WhatsApp adapter type failure. Current-main bug reproduction is not applicable because this PR adds a new capability.

## Solution Assessment

No. The feature direction is reasonable, but the branch should route fallback location sends through the registered `send` RPC or add a complete registered RPC, type the WhatsApp outbound adapter, and satisfy lint/type gates before merge.
