## Summary

Current main routes Feishu inbound media through the current event's message_id and resource key, then saves the downloaded buffer as a fresh media file. The latest release tag v2026.4.26 contains the same message-scoped download path, so the reported image/text mismatch is already addressed in the shipped Feishu plugin path.

## Best Possible Solution

Close as implemented and keep the current message-scoped Feishu media download path. If a user can still reproduce mismatched images after v2026.4.26, the right follow-up is a new issue with the Feishu event payload metadata, message_id/file_key pairs, OpenClaw logs, and the saved local media filenames so maintainers can trace a fresh routing failure.
