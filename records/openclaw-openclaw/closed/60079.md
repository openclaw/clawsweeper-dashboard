## Summary

Current `main` already contains a post-release fix for the silent-drop path behind WhatsApp group auto-delivery of tool media, with regression coverage for block-only/media-only replies. The issue appears fixed on `main`, but not yet in the latest tagged release `v2026.4.23`.

## Evidence

- **`image_generate` still emits saved MEDIA paths for auto-delivery:** The tool returns saved local paths as `MEDIA:` lines and structured `details.media.mediaUrls`, which is the contract the auto-delivery pipeline consumes.
  - file: [src/agents/tools/image-generate-tool.ts:720](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/agents/tools/image-generate-tool.ts#L720)
  - command: `nl -ba src/agents/tools/image-generate-tool.ts | sed -n '680,740p'`
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **Tool results are queued for deferred media delivery:** The embedded PI tool handler extracts structured media from tool results and queues pending media URLs for later reply delivery when the tool itself should not emit a visible text payload.
  - file: [src/agents/pi-embedded-subscribe.handlers.tools.ts:536](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/agents/pi-embedded-subscribe.handlers.tools.ts#L536)
  - command: `nl -ba src/agents/pi-embedded-subscribe.handlers.tools.ts | sed -n '280,590p'`
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **Queued tool media is now attached to the next assistant reply or sent as media-only:** The reply handler merges queued tool media into the next block reply and also has an orphan-media path that emits a media-only reply when there is no assistant text.
  - file: [src/agents/pi-embedded-subscribe.handlers.messages.ts:180](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/agents/pi-embedded-subscribe.handlers.messages.ts#L180)
  - command: `nl -ba src/agents/pi-embedded-subscribe.handlers.messages.ts | sed -n '180,240p'`
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **WhatsApp group dispatcher no longer treats block-only replies as silent:** The WhatsApp buffered reply path now considers `counts.block > 0 || counts.final > 0` as a visible reply before returning, which matches the reported symptom where group turns with media/block output were silently skipped.
  - file: [extensions/whatsapp/src/auto-reply/monitor/inbound-dispatch.ts:276](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/extensions/whatsapp/src/auto-reply/monitor/inbound-dispatch.ts#L276)
  - command: `nl -ba extensions/whatsapp/src/auto-reply/monitor/inbound-dispatch.ts | sed -n '227,340p'`
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **Regression test covers the WhatsApp block-only silent-drop case:** A dedicated WhatsApp test now asserts that a block-only turn is treated as a visible reply instead of a silent turn, and the WhatsApp reply delivery tests use a `chatType: "group"` message when asserting media send behavior.
  - file: [extensions/whatsapp/src/auto-reply/monitor/inbound-dispatch.test.ts:417](https://github.com/openclaw/openclaw/blob/4d1ee3a73ed19063d16ac627d3071b7637fda82d/extensions/whatsapp/src/auto-reply/monitor/inbound-dispatch.test.ts#L417)

  - sha: [4d1ee3a73ed1](https://github.com/openclaw/openclaw/commit/4d1ee3a73ed19063d16ac627d3071b7637fda82d)
