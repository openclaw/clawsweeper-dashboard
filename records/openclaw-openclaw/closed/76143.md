## Summary

Keep open. The PR addresses a real current-main gap in Codex app-server dynamic-tool progress, but it should not merge as-is because the new shared tool events emit raw dynamic-tool arguments and results without the established redaction, truncation, and image-stripping path.

## What This Changes

The PR emits canonical tool start/result events for Codex app-server dynamic tool calls, normalizes responses into TUI-style tool result payloads, and adds regression coverage for that request path.

## Best Possible Solution

Emit Codex dynamic-tool progress through the same sanitized, bounded tool-event contract used by the existing tool stream, then cover success, error, redaction, truncation, and image cases and add the required changelog entry.

## Reproduction Assessment

Yes. The focused reproduction is the Codex app-server `item/tool/call` request path: current main handles the call without canonical `tool` events, and the PR test drives that handler and observes the added start/result events.

## Solution Assessment

No. Emitting from the Codex-owned request path is the right boundary, but the proposed payload shape bypasses the established sanitizer and should be changed before merge.
