## Summary

Current `main` already fixes both reported Codex OAuth image-generation regressions: the Responses transport now uses `gpt-5.5`, and legacy `openai-codex.baseUrl` values such as `https://chatgpt.com/backend-api` are canonicalized to the `/backend-api/codex` route before image requests.

## Evidence

- **Codex image requests now use gpt-5.5:** `generateOpenAICodexImage` posts to `${baseUrl}/responses` with `model: DEFAULT_OPENAI_CODEX_IMAGE_RESPONSES_MODEL`, and that constant is `"gpt-5.5"`.
  - file: [extensions/openai/image-generation-provider.ts:31](https://github.com/openclaw/openclaw/blob/16b7dee1ef2f80d62d8629caf1021f5557bc3b41/extensions/openai/image-generation-provider.ts#L31)
  - command: `nl -ba extensions/openai/image-generation-provider.ts | sed -n '500,610p'`
  - sha: [16b7dee1ef2f](https://github.com/openclaw/openclaw/commit/16b7dee1ef2f80d62d8629caf1021f5557bc3b41)
- **Configured Codex baseUrl is canonicalized before image requests:** The image-generation path passes `codexProviderConfig?.baseUrl` through `canonicalizeCodexResponsesBaseUrl(...)` before resolving request config, preventing `/backend-api` from dropping the required `/codex` segment.
