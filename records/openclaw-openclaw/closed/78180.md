## Summary

Keep open: current main still sanitizes every `text/event-stream` response before provider error parsing, so a non-OK Gemini response with a bare JSON body can lose the detail before `createProviderHttpError` reads it.

## What This Changes

The report asks streaming provider HTTP errors to preserve JSON error body details instead of collapsing Gemini 400s to a bare status.

## Best Possible Solution

Skip or bypass SSE sanitization for non-OK guarded fetch responses so provider error parsing can read the original body, while preserving the OK-response OpenAI SDK sanitizer behavior.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a 400 `text/event-stream` response with a bare JSON body passed through `buildGuardedModelFetch` and then `createProviderHttpError`; current main drops that body before error parsing.

## Solution Assessment

Yes. The narrowest maintainable fix is to preserve non-OK responses before `sanitizeOpenAISdkSseResponse`; changing `extractProviderErrorDetail` alone cannot recover a body the sanitizer already stripped.
