## Label Justifications

- `P1`: The PR targets a reported real-user Discord reply delay where completed embedded runs could hold delivery for tens of seconds.
- `merge-risk: 🚨 auth-provider`: The diff changes when auth-profile `lastGood` and usage metadata are persisted relative to successful run completion.
- `merge-risk: 🚨 session-state`: The changed ordering affects the session-lane completion path and can leave post-run state stale if the process exits before the background write finishes.

## Summary

Keep open for maintainer review: the PR addresses the reported reply-path delay with focused source and proof, but it intentionally moves auth-profile success persistence after reply delivery, so auth-provider/session-state owners need to accept that durability tradeoff before merge.

## What This Changes

The branch makes embedded-agent post-run auth-profile success bookkeeping nonblocking, adds redacted slow/failure diagnostics, and adds regression coverage for returning before that bookkeeping resolves.

## Best Possible Solution

Land the small nonblocking bookkeeping change after maintainer acceptance of the post-reply auth-state durability tradeoff; if immediate durability is required, replace the fire-and-forget call with an owned background task that can be drained during shutdown.

## Reproduction Assessment

Yes. Current main and v2026.6.1 both await `markAuthProfileSuccess` after `embedded run done`, and the PR body includes a delayed-auth-success gateway proof showing delivery should proceed before that promise completes.

## Solution Assessment

Yes, as an acceptable mitigation rather than a fully durability-preserving design. The smaller fix matches the latency failure; a drainable background task would be safer if maintainers require shutdown durability for this metadata.
