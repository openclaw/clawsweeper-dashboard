## What This Changes

Request allowing cron or agent-turn payloads to set a stable prompt cache key independent of isolated cron run session ids.

## Best Possible Solution

Keep one canonical repair in https://github.com/openclaw/openclaw/issues/62809: add a stable cron cache-affinity key that preserves isolated transcripts and provider compatibility, folding the explicit payload-key design in there if user control is desired.

## Reproduction Assessment

Yes, at source level. Current main still creates a fresh isolated cron session id per run and passes that id into OpenAI prompt_cache_key construction, while no payload field exists to override it.

## Solution Assessment

Yes for tracking consolidation. Keeping the canonical cron cache-affinity issue open is better than maintaining two competing trackers for the same cost problem and API design choice.
