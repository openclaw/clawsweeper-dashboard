## Summary

Current main already addresses the reported WS latency class by keeping session/model list RPCs on bounded read-only metadata paths and stripping runtime-only skill snapshots from persisted session stores; the proof is main-only in this checkout, not a proven shipped release.

## What This Changes

The issue reports Windows/xAI gateway-wide WebSocket latency after failed automatic compaction leaves session state that makes list/status RPCs take 8-11 seconds.

## Best Possible Solution

Keep the current main implementation that makes list/status RPCs bounded and session persistence lightweight; if latency still appears on a post-a7c5a04 build, open a new report with fresh Windows logs and the exact OpenClaw version.

## Reproduction Assessment

Yes for the reported 2026.5.2 release: the issue gives concrete Windows/xAI steps and logs showing failed compaction followed by 8-11s WS replies. I did not establish a failing current-main repro because current main contains bounded/read-only list paths and session-store bloat cleanup.

## Solution Assessment

Yes; keeping Gateway list/status RPCs off full provider discovery and removing runtime-only skill snapshots from persisted sessions is the narrow maintainable fix for this latency class without requiring users to delete sessions.
