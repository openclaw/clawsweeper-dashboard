## Summary

Keep open. The report is a concrete Azure Responses delivery bug with a detailed live reproduction, and current main still routes `azure-openai-responses` through the same shared Responses stream path without a provider-level regression test for completed Azure reasoning-plus-message output. The related Azure/OpenAI Responses fixes in the latest release address neighboring failure modes, not this direct-API-succeeds-but-OpenClaw-produces-empty-assistant-text case.

## What This Changes

The issue asks OpenClaw to make `azure-openai-responses` produce deliverable assistant text for completed Azure / Foundry Responses output instead of ending with `assistantTexts=[]` and `non_deliverable_terminal_turn`.

## Best Possible Solution

Fix this in the Azure/shared Responses normalization path with focused mock fixtures for completed reasoning-plus-message output, then verify with a redacted live Azure no-tools canary when credentials are available; keep the terminal no-deliverable guard intact.

## Reproduction Assessment

No high-confidence local reproduction was possible without Azure credentials. The reporter provided a detailed live repro, and source inspection shows the current Azure Responses route and terminal guard can plausibly produce the observed empty-assistant terminal failure when output normalization misses the final message.

## Solution Assessment

Yes, treating this as a provider/runtime extraction bug is the right direction. The best fix should normalize the provider output before terminal delivery rather than weakening `non_deliverable_terminal_turn` classification or adding a provider fallback.
