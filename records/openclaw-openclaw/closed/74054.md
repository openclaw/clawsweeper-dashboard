## Summary

Keep open. Current main bounds a single Active Memory recall attempt with a watchdog and generic embedded-runner retry caps, so the literal infinite retry wording is overstated. The remaining bug is still valid: Active Memory has no plugin-owned consecutive-timeout budget or cooldown, and timeout results are deliberately not cached, so repeated eligible prompts against the same slow model can keep spending provider calls and delaying replies.

## Best Possible Solution

Add an Active Memory-owned timeout circuit breaker: track consecutive timeout/timeout_partial results per agent/session/provider/model, skip recall for a configurable cooldown after a small threshold, persist/log a clear skipped status, reset on successful non-timeout results, and keep generic provider retry/fallback behavior out of scope.

## Risks / Open Questions

- A timeout circuit breaker should use a bounded cooldown and reset on successful or non-timeout recall so transient provider slowness does not permanently disable useful memory recall.
- Client-side aborts may not prevent provider billing once a request has been accepted, so the fix should describe reduced repeated attempts rather than guaranteed token refunds.
- Because the plugin manifest uses additionalProperties=false, any new config must update the manifest, runtime defaults, docs, and tests together.
