## Label Justifications

- `P2`: The report concerns incorrect persisted session model routing with cost and provider-choice impact, but current main and the latest release already contain the recovery path.
- `impact:session-state`: The affected state is a persisted session-level `modelOverrideSource: "auto"` override that can outlive the original fallback event.
- `impact:auth-provider`: The symptom routes turns to the wrong provider/model after transient provider failure.

## Summary

Current main and the latest release already recover persisted auto fallback pins on regular turns through the primary-probe path, so the reported 2026.5.12 behavior is obsolete for v2026.5.22+ rather than needing the proposed heartbeat-gate change.

## What This Changes

The issue requests that persisted `modelOverrideSource: "auto"` fallback overrides recover during normal inbound turns instead of only through heartbeat-specific cleanup.

## Best Possible Solution

Keep the shipped v2026.5.22 primary-probe recovery path and direct users on 2026.5.12 to upgrade before reopening with current-version proof.

## Reproduction Assessment

No high-confidence current-main failure reproduction remains. The 2026.5.12 report is credible, but current main and v2026.5.22 source show regular turns schedule a primary probe and clear matching auto pins after success.

## Solution Assessment

Yes, the shipped primary-probe path is the better solution for transient provider recovery. Removing the heartbeat gate would target stale changed-primary cleanup and would not by itself prove recovery from a provider outage whose recorded origin still matches the configured primary.
