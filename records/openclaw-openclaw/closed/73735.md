## Summary

Keep this PR open for revision. Current main still hard-fails Telegram deleteMessage errors, so the PR is not obsolete, but the latest PR head still logs benign failures before the outer catch can soft-fail them, has a forbidden-error test row that no longer matches production code, and retains boolean comparisons that the lint gate flags.

## What This Changes

The PR widens Telegram deleteMessage handling to soft-fail selected Bot API delete errors, surfaces deleted:false warnings from the Telegram action runtime, adds send.ts regression coverage, and adds a changelog entry.

## Best Possible Solution

Revise this PR in place so a single benign-delete predicate is reused both for requestWithDiag(..., { shouldLog }) and the catch, only true already-gone/stale delete states fail soft, forbidden/permission failures stay loud, the tests match that classifier, and the branch clears targeted Telegram tests plus lint.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P2] Suppress benign deletes before the API logger runs:** `extensions/telegram/src/send.ts:1094`
  - body: `requestWithDiag` is still called without `options.shouldLog`, so `withTelegramApiErrorLogging` logs `telegram deleteMessage failed: ...` before this outer catch returns a warning. That leaves the ERROR-level noise this PR is meant to stop; pass the same benign-delete predicate as `shouldLog` and reuse it in the catch.
  - confidence: 0.97
- **[P2] Move forbidden deletes out of the benign table:** `extensions/telegram/src/send.test.ts:2076`
  - body: The latest production classifier no longer matches `MESSAGE_DELETE_FORBIDDEN`, but this test row still expects it to return `{ ok: false }`. That row will reject before the assertions, so the Telegram test suite fails; move it to the non-benign rejection case.
  - confidence: 0.98
- **[P2] Use direct boolean checks so lint passes:** `extensions/telegram/src/action-runtime.ts:490`
  - body: `deleteResult.ok === false` and the matching `result.ok === false` check in `send.test.ts` are still on the latest head and trigger the boolean-literal comparison lint failure already reported in this PR. Use `!deleteResult.ok` and `!result.ok` so the lint gate can pass.
  - confidence: 0.9
