## Label Justifications

- `P1`: The report describes a recent release regression that silently corrupts isolated cron agentTurn prompts for real scheduled workflows.
- `impact:message-loss`: The configured cron prompt is effectively lost before it reaches the model, even though the run can still record success.

## Summary

Current main already fixes the reported release regression: isolated cron agentTurn prompt assembly now reads the canonical persisted `job.payload.message`, and regression tests cover both malformed dispatch input and SQLite-rehydrated manual runs. The fix is main-only after v2026.6.1, so the issue should close as implemented on main rather than queue another repair.

## What This Changes

The issue reports that isolated cron agentTurn jobs in v2026.6.1 send `[object Object]` to the LLM instead of the configured `payload.message`.

## Best Possible Solution

Keep the current-main fix from https://github.com/openclaw/openclaw/pull/91230 and let the next release carry it; separate cron store migration or delivery-default problems should remain tracked in their own issues.

## Reproduction Assessment

Yes for the release regression: v2026.6.1 source interpolates `input.message` into the isolated cron prompt, matching the reported `[object Object]` symptom when dispatch passes a wrapped object. Current main no longer has that source path.

## Solution Assessment

Yes: resolving agentTurn prompt text from canonical `job.payload.message` is the narrow durable fix, and the tests cover both malformed dispatch input and the SQLite-rehydrated manual-run path.
