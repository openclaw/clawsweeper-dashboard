## Summary

Current `main` already avoids fabricating `channels.telegram.accounts.default` when a multi-account Telegram config has no authored top-level single-account fields. The doctor migration now only promotes explicitly present channel keys, and it skips promotion entirely when none are present.

## Evidence

- **Promotion helper ignores configs with no authored top-level keys:** `resolveSingleAccountKeysToMove` filters out `accounts`, `defaultAccount`, and `enabled`, and returns `[]` when nothing else is authored. A config shaped like the report therefore has no keys eligible for promotion.
  - file: [src/channels/plugins/setup-promotion-helpers.ts:104](https://github.com/openclaw/openclaw/blob/f85806b6d78698fb6b6060d09e16cb09ae40131b/src/channels/plugins/setup-promotion-helpers.ts#L104)
  - command: `nl -ba src/channels/plugins/setup-promotion-helpers.ts | sed -n '97,145p'`
  - sha: [f85806b6d786](https://github.com/openclaw/openclaw/commit/f85806b6d78698fb6b6060d09e16cb09ae40131b)
- **Doctor only seeds accounts.default when promotable keys exist:** `seedMissingDefaultAccountsFromSingleAccountBase` calls `resolveSingleAccountKeysToMove(...)` and immediately `continue`s when `keysToMove.length === 0`, so doctor no longer synthesizes a phantom default account from absent fields.
  - file: [src/commands/doctor/shared/legacy-config-core-normalizers.ts:129](https://github.com/openclaw/openclaw/blob/f85806b6d78698fb6b6060d09e16cb09ae40131b/src/commands/doctor/shared/legacy-config-core-normalizers.ts#L129)
