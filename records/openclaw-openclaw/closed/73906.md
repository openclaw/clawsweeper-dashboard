## Summary

Current main already has the structural runtime-context leak fix. The `openclaw.runtime-context` transcript entry is still queued for history/debugging, but it now stores raw hidden context only; the human-readable header is appended as prompt-local system context, and `openclaw.runtime-context` custom messages are filtered before provider/LLM conversion. The fix is on main via 11e6928b3edc4be7a78d6397df99c61799b22807 but is not present in the latest shipped release `v2026.4.26`, whose tagged source still queues the header-bearing custom-message body.

## Best Possible Solution

Close this issue as implemented on current main, keep the structural runtime-context handling from 11e6928b3edc4be7a78d6397df99c61799b22807, and let the fix ship in the next release. The downstream Lossless-Claw defensive filter mentioned in the report remains useful as a belt-and-suspenders guard, but OpenClaw core now prevents the runtime-context custom message from entering LLM input on main.
