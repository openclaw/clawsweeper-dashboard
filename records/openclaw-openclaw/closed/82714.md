## Summary

Current main already fixes the reported unsupported-thinking Ollama failure path: non-reasoning Ollama models no longer receive truthy `think`, unsupported thinking levels are downgraded before the run, and remaining user-selected model failures are surfaced rather than silently ignored.

## What This Changes

The issue reports that switching a session model override to a non-thinking Ollama model can cause the next turn to fail with an unsupported-thinking 400, no fallback candidate, and no user-visible error.

## Best Possible Solution

Keep the current capability-gated Ollama thinking behavior and exact user-override fallback policy; users on released builds should pick up the next release containing the current-main Ollama fix.

## Reproduction Assessment

No: the reported 400 path is not reproducible from current-main source because non-reasoning Ollama models do not receive truthy `think`, and unsupported thinking defaults are downgraded before the run.

## Solution Assessment

Yes: the current fix addresses the provider request-shape bug directly while preserving the documented exact-selection semantics for user model overrides; falling through to unrelated default fallbacks would be a separate policy change.
