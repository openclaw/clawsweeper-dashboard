## Summary

The reported internal `queue-health` / `surge-plan` exec path is not present on current `main`, and current exec approvals behavior/documentation points to host-managed approvals state rather than a `tools.exec.allowlist` config key. Based on current code and repo history inspection, I could not reproduce or verify an OpenClaw-core path that still hardcodes this behavior.

## Evidence

- **Reported internal poller path is absent on current main:** A repo-wide search for `queue-health`, `surge-plan`, `cron-gates`, `~/.openclaw/state/gates`, `queue-health-gate.sh`, and `SURGE---` returned no matches on current `main`, so the specific internal path from the report is not present in the checked-out core code.
  - command: `rg -uu -n "queue-health|surge-plan|cron-gates|~/.openclaw/state/gates|queue-health-gate\.sh|SURGE---|allowlist miss" .`
  - sha: [cc0f3e0e4036](https://github.com/openclaw/openclaw/commit/cc0f3e0e40364023cc3a39227a411898ee68ce3b)
- **Current exec host merge only lets host approvals tighten requested policy:** `resolveExecHostApprovalContext` computes host exec policy from the caller-requested `security`/`ask` plus resolved approvals, with an explicit comment that the host may tighten but must not silently broaden the contract.
  - file: [src/agents/bash-tools.exec-host-shared.ts:202](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/src/agents/bash-tools.exec-host-shared.ts#L202)
  - sha: [cc0f3e0e4036](https://github.com/openclaw/openclaw/commit/cc0f3e0e40364023cc3a39227a411898ee68ce3b)
- **Current effective-policy code documents stricter-host precedence:** `resolveExecPolicyScopeSnapshot` derives `effectiveSecurity` with `minSecurity(...)` and emits the note `stricter host security wins` when host approvals are tighter than requested config.
  - file: [src/infra/exec-approvals-effective.ts:238](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/src/infra/exec-approvals-effective.ts#L238)
  - sha: [cc0f3e0e4036](https://github.com/openclaw/openclaw/commit/cc0f3e0e40364023cc3a39227a411898ee68ce3b)
- **Current user-facing docs place allowlists in exec approvals, not `tools.exec.allowlist`:** The exec approvals docs say effective policy is the stricter combination of `tools.exec.*` and approvals defaults, and they store allowlists in `~/.openclaw/exec-approvals.json` rather than under `tools.exec.*`.
  - file: [docs/tools/exec-approvals.md:16](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/docs/tools/exec-approvals.md#L16)
  - sha: [cc0f3e0e4036](https://github.com/openclaw/openclaw/commit/cc0f3e0e40364023cc3a39227a411898ee68ce3b)
- **Current CLI docs expose allowlist management via approvals commands:** The approvals CLI docs document `openclaw approvals` / `openclaw exec-policy` as the supported surface for managing host approvals and allowlists, including `openclaw approvals allowlist add|remove`.
  - file: [docs/cli/approvals.md:21](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/docs/cli/approvals.md#L21)
  - sha: [cc0f3e0e4036](https://github.com/openclaw/openclaw/commit/cc0f3e0e40364023cc3a39227a411898ee68ce3b)
- **Current config schema has no `tools.exec.allowlist` key:** The `tools.exec` schema includes `host`, `security`, `ask`, `safeBins`, `safeBinTrustedDirs`, and related fields, but no `allowlist` property, which matches the current docs and approvals model.
  - file: [src/config/zod-schema.agent-runtime.ts:456](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/src/config/zod-schema.agent-runtime.ts#L456)
