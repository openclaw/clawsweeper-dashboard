## Summary

Current main and the latest release already normalize singular modal component `value` fields before `resolveModalFieldValues()` reads RadioGroup values, so this PR would duplicate the shipped Discord modal parsing path instead of fixing a remaining gap.

## What This Changes

The PR adds a Discord RadioGroup-specific raw payload reader and routes modal radio fields through it instead of the existing `interaction.fields.getStringSelect()` path.

## Best Possible Solution

Keep the shipped `ModalFields` normalization path as the single modal-field reader and close this duplicate PR rather than adding a radio-only bypass.

## Reproduction Assessment

No against current main: source inspection shows raw modal `value` fields are already extracted into `ModalFields`, and `getStringSelect()` returns stored strings as arrays for the existing radio path. I did not run a live Discord submission in this read-only review.

## Solution Assessment

No; this PR is not the best fix because the same payload shape is already handled generically in `extensions/discord/src/internal/modal-fields.ts`. Keeping one normalized reader is narrower and less drift-prone than adding a RadioGroup-specific raw payload bypass.
