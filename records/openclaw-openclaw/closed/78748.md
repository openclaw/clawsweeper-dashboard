## Summary

Keep this PR open. The latest head fixes the earlier gateway read-scoping gap, but entering the Dreaming tab from a non-default active session still performs the automatic status/diary load before the new selected-agent state is synchronized, so the first visible data can still come from the default agent.

## What This Changes

The branch adds an agent selector to the Control UI Dreaming tab and routes Dreaming status, diary, and diary action RPCs through an optional selected agent id.

## Best Possible Solution

Synchronize the selected Dreaming agent before every automatic Dreaming data load path, keep the gateway's backward-compatible default fallback, add a tab-entry regression test, then collect redacted real multi-agent UI proof.

## Reproduction Assessment

Yes. Source inspection at PR head shows automatic Dreams tab refresh calls loadDreamingStatus/loadDreamDiary before selectedAgentId is synchronized from sessionKey, so the controller sends an empty payload and the gateway falls back to the default agent.

## Solution Assessment

No. The selector, shared session helper reuse, and gateway optional agentId fallback are the right direction, but the implementation is incomplete until automatic tab entry loads the same agent that the selector displays.
