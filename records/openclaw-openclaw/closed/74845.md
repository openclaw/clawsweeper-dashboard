## Summary

Keep open. Current main still treats node-host invocation as request/one final result, and the requested request-correlated progress/update primitive remains additive Gateway protocol, node-host, SDK, lifecycle, and security design work.

## What This Changes

The issue requests first-class support for long-running node-host commands that can report progress or intermediate updates before a final result.

## Best Possible Solution

Design an additive, version-aware node invoke progress contract that preserves one-shot behavior while updating protocol schemas, Gateway handling, node-host dispatch, plugin SDK typing, docs, and focused tests together.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives a high-confidence verification path: node invoke, protocol schema, node-host dispatch, plugin node-host commands, plugin runtime typing, and docs are still final-result oriented.

## Solution Assessment

Unclear pending maintainer design. A shared additive protocol/SDK primitive is the maintainable direction; a timeout config or generic best-effort node.event usage alone would not provide reusable request-correlated progress semantics.
