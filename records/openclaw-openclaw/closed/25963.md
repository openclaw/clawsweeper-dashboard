## Summary

Current main already implements group-to-agent routing through `bindings`, and group mention-gating is evaluated against the routed agent’s own mention patterns. That closes the reported bypass where mentioning a different agent in the same group could switch to a higher-permission agent.

## Evidence

- **Deterministic inbound agent routing:** `resolveAgentRoute` chooses one agent from binding tiers in priority order (peer, parent peer, guild+roles, guild, team, account, channel, default) and returns that routed agent/session. Mention text is not used to switch agents.

## Close Comment

Closing as implemented on `main` after Codex review. Current OpenClaw already routes each inbound group message to one configured agent via `bindings` (`src/routing/resolve-route.ts:654`), and group mention gating is then evaluated against that routed agent’s own mention patterns (`extensions/whatsapp/src/auto-reply/web-auto-reply-monitor.test.ts:440`, `extensions/slack/src/monitor/message-handler/prepare.ts:306`, `extensions/signal/src/monitor/event-handler.ts:665`). The routing docs also describe peer/group bindings as the supported way to pin a group to a specific agent (`docs/channels/channel-routing.md:62`). The latest release tree `v2026.4.22` already contains this implementation at `00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`. If anyone can still reproduce an unrestricted-agent bypass on a current build, please open a new issue with the exact channel, `bindings`/group config, and the specific mention that still escapes the routed agent.
