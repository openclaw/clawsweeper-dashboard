## Label Justifications

- `P1`: The original bug could silently block real agent runs and leave users without responses until a gateway restart.
- `impact:message-loss`: The reported failure path dropped user-visible agent responses while `chat.send` only returned `started`.
- `impact:crash-loop`: The issue is about process-level availability: a hung hook left the gateway alive but agent processing wedged until restart.

## Summary

Close: current main and release v2026.5.18 now protect legacy `before_agent_start` handlers with a default fail-open timeout and regression coverage, so the reported permanent hook hang is fixed.

## What This Changes

Ensure a hung plugin `before_agent_start` hook cannot indefinitely block agent runs without diagnostics.

## Best Possible Solution

Keep the shipped generic hook-runner timeout as the core guardrail; plugin-local OpenViking timeouts remain complementary but are no longer the only defense.

## Reproduction Assessment

Yes. The source path is clear: a never-settling `before_agent_start` handler is awaited by setup/prompt compatibility paths, and current regression tests now exercise that exact shape and verify timeout recovery.

## Solution Assessment

Yes. The implemented fix is the narrow maintainable solution because it protects the generic hook runner while preserving per-hook timeout overrides and fail-open behavior for this legacy compatibility hook.
