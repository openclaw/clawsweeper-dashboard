## Summary

Keep open: the bug is source-reproducible, but this patch suppresses every replayed assistant error turn and would hide useful provider or transport diagnostics, and the external PR still lacks real behavior proof.

## What This Changes

This PR adds a TUI history replay guard that skips assistant messages with `stopReason === "error"` and adds a regression test for synthetic fallback entries.

## Best Possible Solution

Filter only the synthetic failed-before-content fallback entries during TUI history replay, while preserving real error messages or partial visible assistant content and covering both cases in tests.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main replays assistant history through `extractTextFromMessage()`, so a sentinel text block is finalized into the chat log. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No; suppressing every `stopReason === "error"` assistant turn is broader than the reported synthetic fallback spam. The narrower maintainable fix is to detect the fallback sentinel or no-useful-content shape and keep formatted `errorMessage` or partial visible content replayable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve replayed error diagnostics:** `src/tui/tui-session-actions.ts:345`
  - body: This skip runs before `extractTextFromMessage()`, so every assistant error turn disappears, not only the synthetic `[assistant turn failed before producing content]` entries. Current formatter coverage expects empty error-turn `errorMessage` values to render as useful HTTP/provider messages, so reloading after a real failed turn would hide the failure reason. Narrow this to the sentinel/no-content fallback case and keep real error text or partial content visible.
  - confidence: 0.87
