## Summary

Close: current main and the v2026.5.28 release already route memory-flush runs through the memory trigger metadata and append-only write wrapper, so the reported replace-mode write exposure is covered.

## What This Changes

The issue reports that a pre-compaction memory flush with suppressed transcript persistence can expose the normal replace-mode `write` tool and overwrite the daily memory file.

## Best Possible Solution

Keep the shipped v2026.5.28 memory-triggered append-only write path; a new issue should focus only on a current-build unguarded continuation path if one is observed.

## Reproduction Assessment

No for current main: source inspection shows the memory flush path now forwards `trigger: "memory"` and `memoryFlushWritePath` into tool creation, which activates append-only write enforcement. The original 2026.5.27 steps are plausible for the older build but are covered by the shipped v2026.5.28 path.

## Solution Assessment

Yes: enforcing append-only behavior in tool construction is the maintainable fix because it does not rely on prompt hints or model compliance. The current implementation keeps the flush runner as the owner and narrows the tool surface for memory-triggered runs.
