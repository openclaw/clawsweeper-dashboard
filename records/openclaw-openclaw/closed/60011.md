## Summary

Current `main` does not support the issue’s claimed failure mode. `/tasks` is implemented as a gateway-handled chat slash command that returns a chat-native task board reply, not a webchat-local panel, and current gateway tests cover slash-command routing without starting an agent run.

## Evidence

- **`/tasks` is a registered chat slash command:** The shared command registry defines `tasks` with text alias `/tasks` and description `List background tasks for this session.`
  - file: [src/auto-reply/commands-registry.shared.ts:194](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/auto-reply/commands-registry.shared.ts#L194)

- **WebChat only locally intercepts explicit local commands:** Control UI chat only intercepts parsed slash commands when `executeLocal` is true. The local command set omits `tasks`, so `/tasks` is intentionally not a client-only panel action.
  - file: [ui/src/ui/app-chat.ts:354](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/ui/src/ui/app-chat.ts#L354)

- **Gateway still routes slash-command text through command handling:** `chat.send` populates `BodyForCommands` and `CommandBody` from the incoming message before dispatch, so slash-command messages are available to the gateway command handlers instead of being treated as plain agent text only.

- **Current tests cover no-agent-run slash-command routing:** The gateway chat test explicitly verifies that `chat.send` slash commands complete without increasing `agentCommand` calls, which contradicts the issue’s claim that the slash command is routed to the agent as a normal message.
  - file: [src/gateway/server.chat.gateway-server-chat.test.ts:612](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/gateway/server.chat.gateway-server-chat.test.ts#L612)
