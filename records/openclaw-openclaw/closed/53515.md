## Summary

Current main no longer reproduces the PR's claimed transcript-pollution bug: thread context stays in the live prompt path but is excluded from the persisted transcript via the separate transcript prompt flow. The proposed move into `extraSystemPrompt` would also promote untrusted thread text into system-role instructions.

## Evidence

- **Prompt/transcript split excludes thread context from persisted transcript:** `buildReplyPromptBodies` prepends `threadContextNote` only to `prefixedBody`/`queueBodyBase`, then builds `transcriptCommandBody` from `transcriptBody` without that note.
  - file: [src/auto-reply/reply/prompt-prelude.ts:31](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/auto-reply/reply/prompt-prelude.ts#L31)
  - command: `nl -ba src/auto-reply/reply/prompt-prelude.ts | sed -n '1,220p'`
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Separate transcript prompt is passed through run paths:** `runPreparedReply` carries both `followupRun.prompt` and `followupRun.transcriptPrompt`; immediate CLI execution and deferred followups both pass the transcript variant separately.
  - file: [src/auto-reply/reply/get-reply-run.ts:742](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/auto-reply/reply/get-reply-run.ts#L742)
