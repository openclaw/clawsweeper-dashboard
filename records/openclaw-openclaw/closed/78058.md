## Summary

Keep open: the PR is aimed at the right fetch-wrapper boundary, but the patch still lets non-enumerable symbol-keyed headers reach native `Headers`, and the external PR only includes unit/check output rather than after-fix real behavior proof.

## What This Changes

The PR updates the shared fetch wrapper to clone plain-object headers with enumerable symbol keys before forwarding them to fetch and adds focused regression tests.

## Best Possible Solution

Strip all own symbol keys from plain-object headers without mutating caller-owned objects, add regression coverage for non-enumerable symbols, add the required changelog entry, and include live proof that the discord.js/OpenClaw plugin path no longer crashes.

## Reproduction Assessment

Yes for the focused boundary: source inspection shows current `main` forwards plain headers through the wrapper, and a local Node check reproduces native `Headers` throwing on symbol-keyed header objects. I did not run the full OpenClaw plugin plus discord.js login path.

## Solution Assessment

No. The proposed direction is narrow, but the implementation should strip any own symbol key, not only enumerable symbols, because native `Headers` rejects both.
