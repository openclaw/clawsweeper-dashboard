## Summary

The reported v2026.3.28 Codex websocket fallback path is already fixed on main by keeping openai-codex on the HTTP responses transport instead of attempting the generic OpenAI WebSocket connection.

## Evidence

- **Issue scope:** #58807 reports v2026.3.28 with model/provider path `openai-codex` and a WebSocket attempt to `api.openai.com` failing with `ENETUNREACH` before falling back to HTTP; labels are `bug` and `regression`, with no protected labels and author association NONE.
  - command: `Provided GitHub issue context for openclaw/openclaw#58807`
- **Fix merged:** PR #55535 was merged on 2026-03-29 and states the root cause: `openai-codex` was being treated as WebSocket-eligible even though it uses the ChatGPT-backed HTTP path. The PR behavior change says only native `openai` / `openai-responses` remains WebSocket-eligible, while `openai-codex` stays on HTTP responses transport.
