## Summary

Close as cannot_reproduce. The issue body’s stated root causes do not line up with shipped OpenClaw behavior: self-hosted Telegram absolute `file_path` reads are an explicit opt-in security contract, and both current `main` and the `v2026.4.12` tag already show correct TTS MIME/extension handling in code and tests.

## Evidence

- **v2026.4.12 release notes documented the Telegram local Bot API contract:** The changelog explicitly says OpenClaw requires `channels.telegram.trustedLocalFileRoots` before reading absolute Telegram Bot API `file_path` values, and uses `channels.telegram.apiRoot` / `channels.telegram.network.dangerouslyAllowPrivateNetwork` for local or fake-IP setups. That matches the report’s 'safe read mode' complaint as a documented security requirement, not a generic regression.
  - file: [CHANGELOG.md:1487](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/CHANGELOG.md#L1487)

- **Current main still enforces trusted roots for absolute Telegram file paths:** `delivery.resolve-media.ts` resolves absolute Telegram Bot API paths only when they stay under configured `trustedLocalFileRoots`; otherwise it raises `outside trustedLocalFileRoots`. This is the current intended behavior for self-hosted/local Bot API media reads.
  - file: [extensions/telegram/src/bot/delivery.resolve-media.ts:171](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/extensions/telegram/src/bot/delivery.resolve-media.ts#L171)
