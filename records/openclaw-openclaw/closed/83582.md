## Summary

Keep open: the reported cron preflight behavior is source-reproducible, but this PR is not merge-ready because it builds the new Authorization header from raw provider config/env fallback instead of the existing resolved discovery credential contract, and it still lacks after-fix real behavior proof.

## What This Changes

The PR adds a cron preflight API-key resolver and sends an `Authorization: Bearer ...` header on local provider probe requests.

## Best Possible Solution

Resolve the probe credential through the existing provider auth/discovery-key contract, skip headers for markers and authless providers, cover plaintext/env/SecretRef/non-secret cases, and attach redacted after-fix proxy logs.

## Reproduction Assessment

Yes. Source inspection shows current main probes local OpenAI-compatible providers with `GET /models` and no headers, and the existing mocked test already covers a reachable 401 response path.

## Solution Assessment

No. Adding an auth header is the right direction, but this implementation should use the established resolved discovery credential contract instead of raw provider config strings.
