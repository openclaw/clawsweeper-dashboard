## Summary

This is useful, but it fits the opt-in plugin path rather than an OpenClaw core default. Current main does not implement a built-in CJK/Han-plus-ASCII de-spacing policy, and the existing `before_tool_call` hook can rewrite tool parameters before execution, so a ClawHub code plugin can provide this behavior without adding a global heuristic to core.

## What This Changes

The issue requests a configurable OpenClaw-layer path auto-fix that removes model-inserted spaces between Chinese/Han and ASCII characters in file paths.

## Best Possible Solution

Close the core issue and route the capability to a ClawHub code plugin that registers `before_tool_call`, rewrites only configured path-bearing parameters or carefully parsed command arguments, and documents its validation and runtime coverage. Core should only grow a new API if a plugin implementation proves a missing extension point.

## Reproduction Assessment

Not applicable: this is a feature request rather than a bug with a failing reproduction. Source inspection verifies current main lacks a built-in CJK/Han-plus-ASCII de-spacing rule while exposing a plugin hook that can implement the behavior opt-in.

## Solution Assessment

Yes: an opt-in ClawHub/plugin implementation is the narrowest maintainable direction because a global core heuristic could silently rewrite legitimate filenames or command text. The existing `before_tool_call` parameter rewrite surface is sufficient for a first implementation.

## Risks / Open Questions

- A de-spacing plugin should be existence-aware because both spaced and unspaced filenames can legitimately exist.
- Rewriting raw exec command strings can change approval matching and command semantics, so that part should be opt-in and documented by the plugin.
- Runtime-owned native tool surfaces may need explicit coverage notes from the plugin rather than assuming every harness rewrites arguments identically.
