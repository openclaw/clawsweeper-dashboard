## Summary

Current `main` includes `@slack/oauth` transitively, and its published `FileStateStore` still uses `Math.random()`, but OpenClaw does not configure or expose Slack OAuth install/callback flow on current `main`. The Slack plugin uses manual token/signing-secret setup and only registers `/slack/events`, so there is no OpenClaw-managed Slack OAuth state token path to exploit.

## Evidence

- **Slack plugin only depends on Bolt transitively for OAuth:** `extensions/slack/package.json:7` depends on `@slack/bolt`, and `pnpm-lock.yaml:10884` / `pnpm-lock.yaml:10907` show `@slack/oauth@3.0.5` only as a transitive dependency of Bolt.
  - file: [extensions/slack/package.json:7](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/extensions/slack/package.json#L7)
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **OpenClaw does not pass OAuth installer credentials:** `extensions/slack/src/monitor/provider-support.ts:121` constructs Bolt receivers with only `appToken`, `signingSecret`, `slackWebhookPath`, and `clientOptions`; `extensions/slack/src/monitor/provider.ts:186` calls it with bot/app tokens and webhook path only.
  - file: [extensions/slack/src/monitor/provider-support.ts:121](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/extensions/slack/src/monitor/provider-support.ts#L121)
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Slack HTTP routes are event webhook routes only:** `extensions/slack/src/http/plugin-routes.ts:7` registers normalized webhook paths and dispatches them to `handleSlackHttpRequest`; there is no `/slack/install` or OAuth callback route owned by OpenClaw.
  - file: [extensions/slack/src/http/plugin-routes.ts:7](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/extensions/slack/src/http/plugin-routes.ts#L7)
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Docs describe manual token setup, not OAuth install flow:** `docs/channels/slack.md:22` documents Socket Mode with `appToken`/`botToken`, and `docs/channels/slack.md:71` documents HTTP mode with `botToken`/`signingSecret` and `webhookPath`. No OpenClaw OAuth redirect/install flow is documented.
  - file: [docs/channels/slack.md:22](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/docs/channels/slack.md#L22)
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Upstream vulnerable code exists but is gated behind Bolt installer setup OpenClaw does not use:** Review of the published npm packages showed `@slack/oauth@3.0.5` `package/dist/state-stores/file-state-store.js` uses `Math.random()` and `existsSync`+`unlinkSync`, while `@slack/bolt@4.7.0` `package/dist/receivers/SocketModeReceiver.js` and `HTTPReceiver.js` only instantiate `InstallProvider` when `clientId` and `clientSecret` are provided, plus state verification requirements are met. OpenClaw provides none of those inputs.
  - command: `bash -lc 'curl -L --silent https://registry.npmjs.org/@slack/oauth/-/oauth-3.0.5.tgz | tar -xzOf - package/dist/state-stores/file-state-store.js | sed -n "1,220p"'; bash -lc 'curl -L --silent https://registry.npmjs.org/@slack/bolt/-/bolt-4.7.0.tgz | tar -xzOf - package/dist/receivers/SocketModeReceiver.js | sed -n "1,220p"'; bash -lc 'curl -L --silent https://registry.npmjs.org/@slack/bolt/-/bolt-4.7.0.tgz | tar -xzOf - package/dist/receivers/HTTPReceiver.js | sed -n "1,220p"'`

## Risks / Open Questions

- If OpenClaw later adds a Slack OAuth install/callback flow using Bolt's built-in installer on this dependency line, the `@slack/oauth` state-store implementation should be re-evaluated or overridden at that time.
