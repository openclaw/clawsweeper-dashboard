## Summary

Keep open. The report is recent, detailed, and still plausible on current `main`: the Discord preflight path still awaits configured ACP binding readiness directly, the ACP readiness chain still relies on a shared lazy registration/session-init path with no timeout at this boundary, no merged fix for this issue is present on `main`, and the issue discussion already points to an open fix PR (#68834).

## Evidence

- **Discord preflight still blocks on configured binding readiness:** Current `main` still awaits `conversationRuntime.ensureConfiguredBindingRouteReady(...)` in the Discord inbound preflight path and only logs success or an error after that await settles; there is no timeout or fallback at this callsite.

That matches the issue's suspected first-call lazy-init area.
  - file: [src/channels/plugins/binding-targets.ts:12](https://github.com/openclaw/openclaw/blob/938b53698ec17934a5d1a6cffb279c584f64d9f3/src/channels/plugins/binding-targets.ts#L12)
