## Summary

Current main now strips the unsupported top-level `metadata` field from native ChatGPT Codex Responses requests before the OpenAI SDK send path, including after payload hooks, while preserving metadata for direct OpenAI, Azure, and custom compatible endpoints. The issue can close as implemented on main by commit `388019f5b693b15456c3e2ded6ca9121c96b1af7`; it has not shipped in a tagged release yet.

## What This Changes

The issue reports that OpenClaw 2026.4.26 sends top-level OpenClaw turn metadata to the `openai-codex` ChatGPT Responses backend, causing every Codex OAuth request to fail with HTTP 400.

## Best Possible Solution

Keep the current narrow native-ChatGPT Codex sanitizer and post-hook send-path guard, because it removes the rejected body fields only for the backend that rejects them while preserving metadata and other compatibility fields for direct OpenAI, Azure, and custom Responses-compatible endpoints.

## Reproduction Assessment

Yes. The issue provides a concrete live `openclaw agent` reproduction plus captured request/response evidence, and current main has focused regression tests proving native ChatGPT Codex params no longer include `metadata`.

## Solution Assessment

Yes. The current main fix is safer than a source-only turn-state change because it also protects the final SDK send path after payload hooks, which was the failure mode called out in the follow-up comment, while keeping custom Codex-compatible endpoints untouched.
