## Summary

Current `main` already keeps device identity for authenticated loopback probes, and the released `v2026.4.22` snapshot contains the same behavior. `openclaw status` also now defaults to the lighter `presence` probe path instead of the older full detail path.

## Evidence

- **Authenticated loopback probes keep device identity on current main:** `src/gateway/probe.ts:148-167` now loads a device identity unless the target is loopback and the probe is unauthenticated. That removes the old authenticated-loopback `deviceIdentity: null` behavior that caused the false `missing scope: operator.read` result.
  - file: [src/gateway/probe.ts:148](https://github.com/openclaw/openclaw/blob/c1383680405879a3c3f43dceb20a694f54d3c13c/src/gateway/probe.ts#L148)
  - command: `nl -ba src/gateway/probe.ts | sed -n '148,167p'`
  - sha: [c13836804058](https://github.com/openclaw/openclaw/commit/c1383680405879a3c3f43dceb20a694f54d3c13c)
- **Regression coverage exists for the loopback behavior:** `src/gateway/probe.test.ts:129-193` asserts that a token-auth loopback probe sends a device identity, while an unauthenticated loopback probe still sends `null`.
  - file: [src/gateway/probe.test.ts:129](https://github.com/openclaw/openclaw/blob/c1383680405879a3c3f43dceb20a694f54d3c13c/src/gateway/probe.test.ts#L129)
