## Summary

Source review shows the PR targets a real mismatch: Bedrock `aws-sdk` auth is a valid no-api-key mode in the model registry and Bedrock runtime, but current compaction safeguard cancels when request auth has no `apiKey` or `headers`. Keep this PR open for review; the patch is plausible and small, but contributor proof is only a mocked/unit test and the external PR still needs redacted real behavior proof before merge.

## What This Changes

The PR allows the compaction safeguard to continue when the selected provider reports `aws-sdk` auth despite no explicit apiKey or headers, and adds a regression test for that path.

## Best Possible Solution

Land a narrow compaction safeguard fix that treats configured AWS SDK auth as valid while preserving cancellation for providers that truly lack request credentials, with regression coverage and redacted real AWS SDK credential-chain proof.

## Reproduction Assessment

Yes from source inspection: current `resolveModelAuth` rejects a successful no-key/no-header auth result, and the model registry intentionally returns that shape for Bedrock `auth: "aws-sdk"`. I did not run a live AWS profile or STS setup in this read-only review.

## Solution Assessment

Yes, the direction is the narrow fix: align the safeguard with the existing Bedrock `aws-sdk` auth contract instead of adding a fake key, fallback, or new config option. A typed request-auth mode would be cleaner than the local cast, but the submitted shape is small and does not introduce a definite correctness bug.
