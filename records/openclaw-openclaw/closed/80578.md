## Summary

Keep open. Current main has structured scaffolding and `isReasoning` safeguards, but the reported leak is an unmarked natural-language planning paragraph followed by the intended reply, and I did not find a shared channel-boundary guard that would strip or block that mixed-output class.

## What This Changes

The issue asks OpenClaw to prevent hidden/internal planning text from being rendered in user-visible channel output after an iMessage leak.

## Best Possible Solution

Add or confirm a shared hidden-vs-visible output boundary that prevents mixed planning-prefix plus final-answer text from reaching channel adapters, with focused iMessage and generic outbound regression coverage.

## Reproduction Assessment

No high-confidence live iMessage reproduction is available from the repo context. Source inspection does show the class is source-reproducible: an unmarked natural-language planning paragraph inside `ReplyPayload.text` would pass existing `isReasoning` and marker-based guards.

## Solution Assessment

Unclear. A final sanitizer already handles structured scaffolding, but the reported unmarked prose likely needs upstream hidden/visible separation or a carefully scoped lead-in scrubber rather than a broad all-channel heuristic.
