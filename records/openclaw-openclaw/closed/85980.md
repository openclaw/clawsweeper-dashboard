## Summary

Keep open: current main and the latest release already cover static `token` env references through the existing SecretRef path, but the requested `access`/`refresh` behavior remains explicitly outside the OAuth credential policy and needs maintainer/security direction before implementation.

## What This Changes

Request environment-variable references in `auth-profiles.json` `token`, `access`, and `refresh` credential fields.

## Best Possible Solution

Keep static credentials on the existing SecretRef path, and have maintainers decide whether OAuth `access`/`refresh` should stay excluded or gain a documented SecretRef-compatible lifecycle model.

## Reproduction Assessment

Yes, by source inspection: static `token` env refs are handled through SecretRef resolution, while OAuth `access` and `refresh` refs are rejected by policy. I did not run tests because this review is read-only.

## Solution Assessment

No, not as proposed: literal fallback on missing env vars is weaker than the current fail-fast SecretRef contract. The maintainable path is to extend or reaffirm the existing SecretRef/OAuth policy rather than add parallel loader substitution.
