## Summary

Keep this open. Current main still has a source-level orphan-recovery path that submits a synthetic resume prompt as an ordinary Gateway `agent` message without provenance, while the WebChat/SSE history projection only filters the narrower `subagent_announce` case already addressed by a related merged PR.

## What This Changes

The issue reports unexpected synthetic user messages appearing in WebChat after heartbeat wake, gateway restart, or session repair flows.

## Best Possible Solution

Propagate internal/inter-session provenance or prompt-persistence suppression through orphan-recovery resume prompts, then make the shared WebChat/SSE projection hide or system-render only provenance-marked internal rows while preserving legitimate user-authored inter-session messages.

## Reproduction Assessment

Yes, source-level. An aborted orphaned subagent follows `resumeOrphanedSession`, which submits a synthetic resume message through the Gateway `agent` method without `inputProvenance`, leaving persistence/display layers unable to distinguish it from a real user turn.

## Solution Assessment

No, current main is not complete. The maintainable fix is to extend the existing provenance/persistence contract through orphan recovery and shared projection tests, not add another text-only UI heuristic.
