## Summary

Keep open: this PR is protected by the `maintainer` label and current `main` does not yet implement the new live Mantis status-reaction scenario. The diff is directionally aligned with the existing Mantis plan, but the new REST oracle looks flaky because the scenario does not keep the terminal done reaction visible long enough to poll reliably.

## What This Changes

The branch adds a manual Mantis Discord Status Reactions workflow, docs, and a QA Lab Discord live scenario that polls message reactions and writes timeline artifacts.

## Best Possible Solution

Keep the PR open and make the reaction oracle deterministic before merge, either by holding/clearing the terminal done reaction on a known timer or by changing the oracle to observe a stable lifecycle signal.

## Reproduction Assessment

Source-reproducible for the review finding: the PR config leaves `removeAckAfterReply` at its default while current Discord processing restores the initial reaction immediately after `setDone()`. I did not run the live Discord scenario.

## Solution Assessment

No. The overall Mantis direction fits the docs, but the current oracle should not depend on catching a transient done reaction in a 250ms polling loop.
