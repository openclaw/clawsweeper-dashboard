## Summary

Current main still has the reported array-typed pairing-state bug in both device and node pairing loaders, and the provided GitHub context shows an open same-author implementation candidate, #63081, with closing syntax for this issue. The maintainer cleanup path is to review/land or explicitly close that PR, not close this issue early.

## Best Possible Solution

Keep this issue open. Review and land #63081, or an equivalent maintainer patch, with record-shape normalization in both `src/infra/device-pairing.ts` and `src/infra/node-pairing.ts`; preferably centralize the guard in `src/infra/pairing-files.ts` and add regression coverage that seeds `pending.json` and `paired.json` with `[]` and verifies request-to-approval persistence for device pairing plus mirrored or helper-level node pairing coverage.

## Evidence

- **Device pairing still passes arrays through loadState:** `loadState` reads `pending.json` and `paired.json` with `readDurableJsonFile` and assigns `pendingById: pending ?? {}` / `pairedByDeviceId: paired ?? {}`. A parsed `[]` is non-nullish, so it remains the state record.
  - file: [src/infra/device-pairing.ts:150](https://github.com/openclaw/openclaw/blob/91e835ebe0abd30ace549cec5955949903c45d5f/src/infra/device-pairing.ts#L150)

  - sha: [91e835ebe0ab](https://github.com/openclaw/openclaw/commit/91e835ebe0abd30ace549cec5955949903c45d5f)
- **Node pairing has the same loader behavior:** `node-pairing.ts` uses the same nullish fallback for `pendingById` and `pairedByNodeId`, so a device-only fix would leave the sibling state store vulnerable.
  - file: [src/infra/node-pairing.ts:134](https://github.com/openclaw/openclaw/blob/91e835ebe0abd30ace549cec5955949903c45d5f/src/infra/node-pairing.ts#L134)
