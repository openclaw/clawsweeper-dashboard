## Summary

Keep open: the linked documentation gap is valid, but this PR’s wording does not match current main. The gateway reads `x-openclaw-scopes` from the WebSocket upgrade request, and current tests show trusted-proxy Control UI scopes are preserved or bounded by that upgrade header rather than reduced by default; the PR also lacks external real behavior proof.

## What This Changes

The PR adds trusted-proxy WebSocket scope documentation and currently includes an unrelated active-memory status change.

## Best Possible Solution

Revise the docs to describe the current gateway contract: WebSocket upgrade headers can bound trusted-proxy Control UI scopes, signed but unpaired device identity differs from a paired device token, and any reduced-scope wording must match existing tests or be paired with an intentional code change.

## Reproduction Assessment

Yes, source-reproducible for the review findings: current main’s WebSocket handler reads the upgrade `x-openclaw-scopes` header, and current tests assert both preserved unpaired-device scopes and proxy-header scope bounding. I did not run a live gateway session.

## Solution Assessment

No. A documentation-only fix is appropriate, but this wording should be aligned to current source/tests instead of documenting an HTTP-only scope header or a default reduced-scope trusted-proxy WebSocket model.
