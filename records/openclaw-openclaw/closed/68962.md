## Summary

Issue #68962 is internally contradictory and not actionable as filed: it says “HTTPS for localhost,” but current main explicitly treats localhost as a secure context, while the body mostly describes remote LAN/Tailscale/NAS access, labels it as a crash, and provides no concrete reproduction, logs, or failing localhost URL.

## Evidence

- **Issue body is contradictory and lacks an actionable repro:** The title claims a localhost HTTPS requirement, but the body repeatedly discusses accessing a gateway from another device on the same LAN/Tailscale/NAS, marks the bug type as a crash, and provides no distinct steps, expected/actual delta, URL, browser, or logs. After reading the provided issue body and empty comments/timeline, there is no single reproducible localhost failure to verify.
- **Client code already treats localhost as a secure context:** The Control UI client comments and behavior explicitly say WebCrypto is available in secure contexts "(HTTPS, localhost)" and only falls back to token-only auth on plain HTTP when device identity cannot be used. The user-facing connect error also says "use HTTPS/localhost or allow insecure auth explicitly."
  - file: [ui/src/ui/gateway.ts:410](https://github.com/openclaw/openclaw/blob/938b53698ec17934a5d1a6cffb279c584f64d9f3/ui/src/ui/gateway.ts#L410)
  - sha: [938b53698ec1](https://github.com/openclaw/openclaw/commit/938b53698ec17934a5d1a6cffb279c584f64d9f3)
- **Server policy distinguishes localhost from remote HTTP:** Current gateway policy allows missing device identity for Control UI only when `gateway.controlUi.allowInsecureAuth` is enabled and the client is local; otherwise it rejects the connection. The handshake error text explicitly says Control UI requires device identity and recommends HTTPS or localhost secure context.
  - file: [src/gateway/server/ws-connection/connect-policy.ts:129](https://github.com/openclaw/openclaw/blob/938b53698ec17934a5d1a6cffb279c584f64d9f3/src/gateway/server/ws-connection/connect-policy.ts#L129)
  - sha: [938b53698ec1](https://github.com/openclaw/openclaw/commit/938b53698ec17934a5d1a6cffb279c584f64d9f3)
- **Tests cover localhost-only insecure compatibility and remote rejection:** Current tests assert that remote Control UI with `allowInsecureAuth` is still rejected, while localhost UI clients without device identity are allowed only when insecure auth is enabled.
  - file: [src/gateway/server/ws-connection/connect-policy.test.ts:67](https://github.com/openclaw/openclaw/blob/938b53698ec17934a5d1a6cffb279c584f64d9f3/src/gateway/server/ws-connection/connect-policy.test.ts#L67)
  - sha: [938b53698ec1](https://github.com/openclaw/openclaw/commit/938b53698ec17934a5d1a6cffb279c584f64d9f3)
- **Docs already describe the intended localhost vs remote behavior:** The Control UI docs state that plain HTTP over LAN/Tailscale is a non-secure context, that localhost-only insecure HTTP compatibility exists behind `gateway.controlUi.allowInsecureAuth=true`, and that remote non-localhost device-identity requirements are not relaxed.
  - file: [docs/web/control-ui.md:262](https://github.com/openclaw/openclaw/blob/938b53698ec17934a5d1a6cffb279c584f64d9f3/docs/web/control-ui.md#L262)
