## Summary

Keep open: the report command is a plausible contributor-facing DX feature with supplied real-output proof, but the latest head still has a blocking scoped-filter bug for multi-word or hyphenated surfaces in both raw-copy and locale fallback matching.

## What This Changes

This PR adds `pnpm ui:i18n:report`, a read-only Control UI i18n baseline report script, focused helper tests, UI guidance, and a changelog entry.

## Best Possible Solution

Keep the read-only baseline report, but make requested-surface matching use the same token-set model for paths and translation keys before merge.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source inspection: `--surface chat-queue` and `--surface sessions-view` stay one normalized string while path and key inputs are split into separate tokens.

## Solution Assessment

Is this the best way to solve the issue? Not yet; the report-over-baseline approach is narrow, but the matcher should tokenize the requested surface once and compare token sets consistently.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Tokenize raw-copy surface input before filtering:** `scripts/control-ui-i18n-report.ts:147-148`
  - body: `filterRawCopyEntries` tokenizes path parts but compares them to the whole normalized `--surface` value. Inputs like `chat-queue` or `chat queue` miss `chat-queue.ts`, so the report can falsely say there is no hardcoded text in that scope.
  - confidence: 0.92
- **[P2] Tokenize locale surface input before matching keys:** `scripts/control-ui-i18n-report.ts:196-198`
  - body: `filterTranslationKeysBySurface` has the same single-string comparison for fallback keys. A scope such as `sessions-view` will not match `sessionsView.*`, which can incorrectly report that no locale problems exist in scope.
  - confidence: 0.9
