## Summary

Keep this issue open. Current main still forces `encoding_format: "float"` for every memory-lancedb embedding request, which is the exact behavior the reporter’s follow-up says fails for ZhiPu-style providers that reject unknown request fields. The merged Ollama/base64 fix solves a related dimension mismatch, but it does not solve the no-`encoding_format` provider case.

## Best Possible Solution

Keep this issue open. Implement a memory-lancedb-owned embedding compatibility mode or raw request path that can omit `encoding_format` without letting the OpenAI SDK inject `base64`, explicitly normalizes float-array and base64/string responses, preserves configured dimensions, and adds regression tests for both forced-float providers and providers that reject the parameter.

## Risks / Open Questions

- A minimal fix that only removes `encoding_format: "float"` from the OpenAI SDK call is likely insufficient because the pinned SDK will inject `encoding_format: "base64"` and decode the response when the caller omits the field.
- A real fix must preserve the shipped Ollama/local-provider dimension repair while adding a provider-safe path for APIs such as ZhiPu that reject unknown request fields.
