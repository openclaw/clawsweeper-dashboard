## Summary

Current main now implements the requested gateway-node file transfer workflow via the bundled, enabled-by-default `file-transfer` plugin. It uses dedicated `file_fetch`/`file_write` tools and `file.fetch`/`file.write` node commands rather than the exact proposed `file_pull`/`file_push` action names, with policy, approval, symlink, audit, and size-limit controls that cover the observable problem well enough to close this tracking issue.

## What This Changes

Add bidirectional file transfer between the Gateway and paired node hosts, including node-side read/write commands and agent-facing pull/push actions.

## Best Possible Solution

Keep the bundled file-transfer plugin as the implementation path: it solves the file movement workflow with a safer dedicated tool and policy boundary, while any future CLI aliases or naming refinements can be tracked separately if maintainers still want them.

## Reproduction Assessment

Not applicable as a bug reproduction. There is a high-confidence current-main verification path by inspecting the bundled plugin registration, agent tools, node-host handlers, policy enforcement, redirect behavior, and colocated tests.

## Solution Assessment

Yes. A dedicated file-transfer plugin is a safer maintainable solution than adding broad raw `nodes.invoke` file actions because it centralizes path policy, approvals, audit logging, size caps, and media-store handling while still covering gateway-to-node and node-to-gateway transfer.
