## Summary

Keep open. Current main does not implement a configurable exec approval timeout, and this PR is not landable as-is because the proposed timeout is unbounded, its help text uses the obsolete two-minute default while main/docs use 30 minutes, and the configured value is not applied to the approval wait budget. Related #51287 remains open and #51292 was closed unmerged, so there is no merged superseding implementation.

## What This Changes

The PR adds a proposed `tools.exec.approvalTimeoutMs` config key to config types, schema, help, and labels, then threads it through exec tool defaults, gateway/node approval registration, and CLI node invoke defaults.

## Best Possible Solution

Expose one stable, bounded approval-timeout surface only if maintainers want this contract, preserve the current 30-minute default in user-facing help/docs, propagate the same resolved value through registration, expiry, gateway/node wait budgets, and CLI transport floors, then cover the propagation and expiry behavior with focused tests. If maintainers prefer the #51287 direction, explicitly supersede this branch under that issue.

## Reproduction Assessment

Yes for the review defects: the PR patch plus current-main source inspection show the proposed value reaches registration paths but not the later `waitDecision` budget, and the schema/help mismatches are visible in the changed files. No runtime test was run because this was a read-only review.

## Solution Assessment

No, not as-is. The direction is plausible, but a maintainable fix needs a maintainer-chosen API surface, bounded validation, documentation aligned to the 30-minute default, and end-to-end timeout plumbing before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep approval waits in sync with the configured timeout:** `src/agents/bash-tools.exec-approval-request.ts:51`
  - body: This introduces `approvalTimeoutMs` for the registration payload, but `waitForExecApprovalDecision` still waits with the fixed default request timeout. If a user configures a timeout longer than the default, the wait RPC can time out before the approval window expires, so the configured value is not honored end-to-end.
  - confidence: 0.86
- **[P2] Bound the approval timeout value:** `src/config/zod-schema.agent-runtime.ts:393`
  - body: The schema accepts any positive integer for a host-exec approval window, and the resolver also accepts any finite positive value. Very large values can leave approvals actionable far longer than intended and can outgrow transport assumptions, so this needs a documented maximum shared with runtime handling.
  - confidence: 0.82
- **[P3] Update the documented default timeout:** `src/config/schema.help.ts:447-448`
  - body: The added help text says the default is `120000ms = 2 minutes`, but current main defines the exec approval timeout as `1_800_000` ms and docs say pending exec approvals expire after 30 minutes by default. This would publish misleading config help.
  - confidence: 0.9
