## Summary

Keep open: this is a protected feature PR with a proof override and a mostly coherent implementation, but the current head introduces a stale-media fallback that can attach recent history images when the current image candidate cannot be used.

## What This Changes

The branch adds shared reply-history media metadata and agent-turn attachment resolution for recent inbound images, with Discord/Slack history-media producers and focused tests.

## Best Possible Solution

Land the shared history-media support only after current-turn media always wins by omission as well as success, so older images are attached only when the current turn has no media candidate.

## Reproduction Assessment

Yes for the review finding: source inspection shows URL-only current image media is skipped before `getBuffer`, then line 135 resolves recent history images because a current image candidate existed but did not resolve. Not applicable as a user bug report because this PR is adding new feature behavior.

## Solution Assessment

No; falling back to history after a current image candidate fails is not the safest maintainable behavior. The narrower fix is to use history images only when there is no current media candidate, or to add an explicitly safe current-URL attachment path instead of substituting older media.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Do not substitute history when the current image fails:** `src/auto-reply/reply/agent-turn-attachments.ts:135-137`
  - body: When the current turn has an image candidate that cannot be attached, such as a URL-only image or a path blocked by the local-root checks, `currentImageResolved` stays false and this branch falls through to attach recent history images. That can make the agent analyze an older image instead of the user's current media; keep history fallback for turns with no current media candidate, not for failed current images.
