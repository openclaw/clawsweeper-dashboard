## Label Justifications

- `P1`: The reported failure deterministically stops outbound replies on active messaging channels once long claude-cli sessions cross the preflight threshold.
- `impact:message-loss`: The issue is about inbound messages being accepted while outbound replies fail and never reach the user.
- `impact:session-state`: The failure depends on persisted session token state diverging from the empty OpenClaw JSONL transcript and requires session-state cleanup to recover on the affected release.

## Summary

Close as implemented: current main and v2026.5.27 treat successful non-compacted preflight compaction as a skip, so the reported no-real-conversation-messages result no longer bricks outbound replies.

## What This Changes

The issue reports that long claude-cli sessions with an empty OpenClaw JSONL transcript convert an ok:true, compacted:false preflight compaction no-op into a fatal dispatch error.

## Best Possible Solution

Keep the shipped successful-no-op preflight behavior and ask for a fresh report only if 2026.5.27 or newer still blocks replies with exact version and logs.

## Reproduction Assessment

Yes for the affected 2026.5.22 behavior: the report includes concrete gateway logs, session state, and a source trace showing ok:true/compacted:false converted into a fatal preflight error. I did not rerun a live claude-cli session, and current main/release source now routes that no-op result as a skip.

## Solution Assessment

Yes. The shipped guard that treats successful no-op compaction as non-fatal is the narrowest maintainable fix for the outage; teaching compaction to read Claude's upstream transcript can remain a separate optimization.
