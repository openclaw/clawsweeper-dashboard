## Summary

Current `main` already covers the reported OpenAI Codex OAuth restart/reuse failure: refreshed credentials are persisted before return, and `refresh_token_reused` now reloads newer on-disk credentials and retries once. Regression tests and changelog entries show this shipped in `v2026.4.8` (with an earlier persistence hardening in `v2026.3.31`).

## Evidence

- **Refresh path now persists rotated OAuth credentials before returning:** The locked refresh path writes `refreshedCredentials` back to `auth-profiles.json` with `saveAuthProfileStore(...)` before returning the API key, which directly addresses losing the new refresh token across restart.
  - file: [src/agents/auth-profiles/oauth-manager.ts:457](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/src/agents/auth-profiles/oauth-manager.ts#L457)
  - command: `nl -ba src/agents/auth-profiles/oauth-manager.ts | sed -n '340,520p'`
- **`refresh_token_reused` recovery is implemented:** On refresh failure, the manager detects `refresh_token_reused`, reloads fresh stored credentials if they changed, and otherwise retries the refresh once with the updated stored refresh token.
  - file: [src/agents/auth-profiles/oauth-manager.ts:551](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/src/agents/auth-profiles/oauth-manager.ts#L551)
