## Summary

Current `main` no longer matches the reported Telegram voice-note path-loss bug: inbound Telegram media is resolved to a required `path: string`, propagated into `allMedia`/`MediaPaths`, and covered by regression tests. Later issue discussion also indicates the remaining repro was a Deepgram plugin/config mismatch rather than this Telegram attachment-path failure.

## Evidence

- **Telegram media path is now required and sourced from saved inbound media:** `TelegramMediaRef.path` is a required string, and `resolveMedia()` returns `{ path: saved.path, ... }` after downloading/saving Telegram media. That makes the specific `allMedia[n].path === undefined` failure mode described in the issue inconsistent with current `main`.
  - file: [extensions/telegram/src/bot-message-context.types.ts:12](https://github.com/openclaw/openclaw/blob/ecac696643c604c6d854341bfe6424bfa31423e9/extensions/telegram/src/bot-message-context.types.ts#L12)
  - command: `nl -ba extensions/telegram/src/bot-message-context.types.ts | sed -n '1,80p'; nl -ba extensions/telegram/src/bot/delivery.resolve-media.ts | sed -n '353,416p'`
  - sha: [ecac696643c6](https://github.com/openclaw/openclaw/commit/ecac696643c604c6d854341bfe6424bfa31423e9)
- **Inbound handler forwards saved media paths into allMedia:** The Telegram handler builds `allMedia` from `media.path` and `media.contentType` immediately after media resolution, so downstream context/preflight code receives the saved local path rather than an optional field.
  - file: [extensions/telegram/src/bot-handlers.runtime.ts:1160](https://github.com/openclaw/openclaw/blob/ecac696643c604c6d854341bfe6424bfa31423e9/extensions/telegram/src/bot-handlers.runtime.ts#L1160)

With `TelegramMediaRef.path: string`, this pipeline no longer matches the undefined-path report.
