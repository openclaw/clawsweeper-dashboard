## Summary

Current main and the shipped v2026.5.20 tag already support the requested DeepSeek `user_id` request-body injection through the documented OpenAI-compatible `params.extra_body` pass-through, so the observable need is covered without adding a provider-specific key.

## What This Changes

Request to add an optional configured `user_id` field to OpenAI-compatible provider request bodies, especially for DeepSeek isolation and scheduling use cases.

## Best Possible Solution

Keep using the generic per-model `params.extra_body` pass-through for DeepSeek `user_id` and other OpenAI-compatible vendor fields rather than adding a dedicated provider-specific `user_id` key.

## Reproduction Assessment

Not applicable as a feature request. Source, docs, tests, and DeepSeek's dependency contract show the requested top-level `user_id` body field can be supplied through the existing `params.extra_body` path.

## Solution Assessment

Yes. The existing generic pass-through is the narrower maintainable solution because it covers DeepSeek `user_id` and other provider-specific OpenAI-compatible fields without adding one-off config keys.
