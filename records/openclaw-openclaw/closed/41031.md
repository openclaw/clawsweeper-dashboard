## Summary

Close as superseded by the linked bug’s maintainer resolution. The linked report was closed with the accepted contract that `target`/`to` control heartbeat delivery while `heartbeat.session` controls the run context; current main still matches that contract, and this branch would change it while also needing a rebase for newer async-sensitive heartbeat call sites.

## What This Changes

The PR changes heartbeat session resolution so explicit non-history `heartbeat.target` plus `heartbeat.to` derives the run session through outbound session routing, and adds a Telegram regression test for that case.

## Best Possible Solution

Keep the shipped target/session split: use `heartbeat.session` for heartbeat transcript context and `target`/`to` for outbound delivery. A future change to infer run sessions from explicit delivery targets should start as a fresh product/API proposal with docs and regression coverage across isolated, cron, and commitment heartbeat paths.
