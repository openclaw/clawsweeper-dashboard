## Summary

## What This Changes

The issue reports intermittent opencode-go/deepseek-v4-flash cron streams that remain in `model_call:stream_progress` until stuck-session recovery aborts instead of completing or failing over.

## Best Possible Solution

Land one provider-owned opencode-go stream termination guard, then separately decide whether downstream stuck-recovery classification should also land as defense-in-depth.

## Reproduction Assessment

Yes at source level, but not with a forced live provider outage in this read-only review. The issue and duplicate logs show opencode-go cron runs stuck at `model_call:stream_progress`, and current main lacks the provider-level stream termination guard proposed by the linked PRs.

## Solution Assessment

Yes, the best current fix layer is the provider-owned opencode-go stream wrapper, with downstream stuck-recovery classification considered separately. Closing this issue before a linked PR merges would be premature.
