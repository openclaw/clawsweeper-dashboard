## Summary

This PR fixes the right Codex projector-to-payload gap and the linked real behavior proof is sufficient, but it should stay open because the branch can keep a native tool error after a later successful native retry and emit a false failure reply. The protected `maintainer` label also means this needs maintainer selection and landing rather than cleanup closure.

## What This Changes

The PR records non-success Codex app-server native tool results as `lastToolError` and adds projector/payload regressions for declined command replies.

## Best Possible Solution

Land one canonical fix for https://github.com/openclaw/openclaw/issues/83093 that bridges native non-success results into `lastToolError` while preserving the existing resolved-error clearing semantics, then close the competing PRs as superseded.

## Reproduction Assessment

Yes, at source level. Current main maps a declined native `commandExecution` to a blocked tool result but does not include `lastToolError` in `buildResult()`, and the inspected before/after artifacts show zero payloads before the patch and one visible error payload after it.

## Solution Assessment

No, not quite as written. Feeding the existing `lastToolError` payload contract is the right solution shape, but the native projector path should also clear a previously recorded native error when a later successful native retry resolves the same action.
