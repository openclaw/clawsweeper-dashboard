## Summary

Keep open: the cap and follow-up live QQBot comment address the earlier passive-reply concern, but the final head still spends the limited keepalive budget before a send succeeds, so transient token/API failures can stop typing early and weaken the fix.

## What This Changes

The PR changes QQBot C2C typing from a 60-second window renewed every 50 seconds to a capped 10-second window renewed every 5 seconds, with focused tests and a changelog entry.

## Best Possible Solution

Land the 10s/5s QQBot typing cadence with a capped budget only after retry accounting is success-based and the exact-head checks pass.

## Reproduction Assessment

Yes, source-reproducible: in the PR head, sendAttempt decrements renewalsRemaining before awaiting sendInputNotify, and the retry path calls the same budgeted helper. I did not run tests because this is a read-only review, but the source path is deterministic.

## Solution Assessment

No. Shortening and capping the cadence is the right direction after the live QQBot proof, but this implementation should count only successful renewals so retry failures do not prematurely consume the cap.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86
