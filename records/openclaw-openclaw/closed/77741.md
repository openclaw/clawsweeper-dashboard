## Summary

Current main already removes the reported hardcoded lane timeout behavior: embedded agent runs now derive the queue task timeout from the resolved run timeout and the queue releases timed-out work so subsequent lane entries can proceed.

## What This Changes

The issue asks for command lane task timeouts to be configurable or aligned with agent timeout configuration instead of using a hardcoded 630-second value.

## Best Possible Solution

Keep the current main behavior that derives lane task timeout from the agent run timeout, and track any broader circuit-breaker or lower-default policy separately.
