## Label Justifications

- `P1`: The report describes a shipped session blockage where subsequent messages time out until gateway restart or stale-lock expiry.
- `impact:crash-loop`: The live gateway process can hold the session lock until restart or maxHoldMs expiry, which is a process-availability failure for that session.
- `impact:message-loss`: The leaked lock blocks subsequent user messages from receiving agent replies during the lock window.
- `impact:session-state`: The issue centers on leaked session write-lock state after embedded abort cleanup.

## Summary

Current main already fixes the lock-orphan window that matches this report: the embedded attempt session-lock controller now releases the underlying held lock in a `finally` block after ownership transfers, with focused regression coverage. The fix is current-main-only; it is not contained in `v2026.6.1`.

## What This Changes

The issue reports that `chat.abort` during an embedded abort-settle timeout can leave the gateway-held session write lock until stale expiry or gateway restart.

## Best Possible Solution

Keep the current main lock-release-in-finally behavior and let the next release carry it; reopen or file a new issue only if the same leak is observed on a build containing `bab18d567b0cd9811012a50e032f71a9c9beb440`.

## Reproduction Assessment

Yes, by source-level reproduction rather than a live run: the `v2026.6.1` code could clear `heldLock` before fence reads and only release afterward, while current main has a regression test for that orphan window. I did not run a live gateway abort scenario in this read-only review.

## Solution Assessment

Yes. Releasing the underlying file lock in `finally` at the ownership-transfer boundary is the narrowest durable fix for this class of leaked session write lock.
