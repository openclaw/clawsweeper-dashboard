## Summary

Keep open: current main still deletes the Twitch client-manager registry entry only after `disconnectAll()` completes, so a thrown cleanup path can keep a stale manager registered; I found no duplicate or current-main fix during this shell check.

## What This Changes

The issue reports that Twitch client-manager removal can leave a stale per-account registry entry if disconnecting the manager throws.

## Best Possible Solution

Move registry deletion into guaranteed cleanup for `removeClientManager`, preserve the disconnect error signal, and add a focused Twitch registry regression test.

## Reproduction Assessment

Yes, source-level: create a Twitch manager through `getOrCreateClientManager`, make that manager's `disconnectAll` throw or reject, call `removeClientManager`, and then `getClientManager` should still expose the stale entry on current main. I did not execute the test because this review is read-only.

## Solution Assessment

Yes; `try/finally` around the disconnect with `registry.delete(accountId)` in the finalizer is the narrowest maintainable fix, as long as the original disconnect error still propagates.

## Repair Work Prompt

Fix the Twitch client-manager registry cleanup bug reported in https://github.com/openclaw/openclaw/issues/83886. In `extensions/twitch/src/client-manager-registry.ts`, ensure `removeClientManager(accountId)` removes the registry entry even when `entry.manager.disconnectAll()` throws or rejects, while preserving the current disconnect error propagation unless adjacent code proves a different contract. Add a focused regression test, preferably `extensions/twitch/src/client-manager-registry.test.ts`, that registers a manager, stubs its `disconnectAll` to reject, asserts `removeClientManager` rejects, asserts `getClientManager(accountId)` is then undefined, and asserts a later `getOrCreateClientManager` returns a fresh manager. Keep the change inside the Twitch plugin lifecycle surface; do not add new config, startup migrations, core plugin-loader behavior, or dependency changes. Add a changelog entry only if maintainers consider this user-visible at landing.
