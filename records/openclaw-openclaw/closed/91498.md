## Summary

Keep open: current main still resolves OpenAI auth-profile preference at provider/session scope, and the model/runtime config surfaces do not accept a declarative preferred auth profile.

## What This Changes

Request declarative per-model-alias, model-ref, or runtime auth-profile selection so OpenAI OAuth and API-key profiles can coexist under the same `openai` provider.

## Best Possible Solution

Define one canonical model-scoped auth-profile preference contract, update schema/types/docs/status/tests, and keep `auth.order.<provider>` as the fallback.

## Reproduction Assessment

Yes from source inspection: strict model config schemas lack an auth-profile selector, and auth selection uses provider-level order plus session overrides.

## Solution Assessment

The requested direction is plausible, but maintainers should choose the permanent config/API shape before code lands.
