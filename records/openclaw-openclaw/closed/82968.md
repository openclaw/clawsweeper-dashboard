## Summary

Keep open: current main has a documented/prompted direction for time via `session_status`, plus a shared current-time helper, but the agent-facing path appears incomplete because the status builder supplies uptime without the current time line. The reporter's confirmed `date` workaround and TIME_CONTEXT notes support that this still needs a scoped upstream fix.

## What This Changes

Expose a reliable agent-facing wall-clock time authority and policy so agents do not rely on uptime for scheduling, recency, deadlines, memory chronology, or date/time reasoning.

## Best Possible Solution

Choose one official agent clock surface, preferably the existing documented `session_status` path unless maintainers want a dedicated `clock.now`, then wire wall-clock UTC and user-local time through that surface and add focused regression coverage.

## Reproduction Assessment

Yes, from source inspection: current docs and prompt say agents should use `session_status` for current date/time, but `buildStatusText` only passes uptime to the status card and never supplies the existing `timeLine`. I did not run a live prompt in this read-only review.

## Solution Assessment

Unclear until maintainers choose the public contract: wiring `resolveCronStyleNow` into `session_status` is the narrowest way to make current docs true, while a dedicated `clock.now` tool would be a larger product/API change.
