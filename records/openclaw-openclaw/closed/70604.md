## Summary

Current main already fixes the reported image-tool routing problem. The image runtime now resolves image models through OpenClaw's configured model resolver instead of calling the pi-ai registry directly, and the resolver applies configured provider `api` and `baseUrl` over discovered registry metadata. ZAI's bundled catalog also marks `glm-5v-turbo` as image-capable, with regression coverage for provider-prefixed configured vision models and provider metadata overrides.

## Best Possible Solution

Close this as implemented on current main and keep the existing resolver-based image runtime. If a ZAI-specific live failure still appears after this main build, it should be reopened as a narrower provider-runtime/request-payload issue with fresh logs from that build.
