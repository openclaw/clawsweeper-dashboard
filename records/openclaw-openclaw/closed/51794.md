## Summary

Keep open: current `main` still does not interpret `secretref-env:...` as a SecretRef for `channels.discord.token`; that value is accepted as a plain string and would be used as the Discord token literally, so the report remains a real validation/runtime UX bug rather than something already fixed.

## Evidence

- **Canonical SecretRef contract is object-form:** The current secrets docs say to use one object shape everywhere for SecretRefs: `{ source, provider, id }`.
  - file: [docs/gateway/secrets.md:81](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/docs/gateway/secrets.md#L81)
- **Discord docs do not document `secretref-env:` for channel tokens:** Discord setup docs show `channels.discord.token` as either a structured SecretRef object or the `DISCORD_BOT_TOKEN` env fallback.
  - file: [docs/channels/discord.md:122](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/docs/channels/discord.md#L122)
- **SecretRef coercion does not parse `secretref-env:` markers:** `coerceSecretRef()` accepts structured ref objects, legacy objects without `provider`, or `${ENV_VAR}` templates only. There is no parser for `secretref-env:NAME`.
  - file: [src/config/types.secrets.ts:85](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/src/config/types.secrets.ts#L85)
- **Any non-empty string is treated as an already-resolved secret:** `resolveSecretInputString()` returns any non-empty string as `status: "available"`, so `secretref-env:OPENCLAW_DISCORD_TOKEN` is accepted as plaintext instead of being resolved or rejected.
  - file: [src/config/types.secrets.ts:158](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/src/config/types.secrets.ts#L158)
- **Runtime resolution and Discord token loading rely on recognized SecretRefs only:** Secret runtime collection only queues assignments when `coerceSecretRef()` succeeds, and Discord token resolution then consumes `channels.discord.token` via `normalizeResolvedSecretInputString()`. A literal `secretref-env:...` string would therefore flow through as the token value.
  - file: [src/secrets/runtime-shared.ts:87](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/src/secrets/runtime-shared.ts#L87)
- **Discord token path uses the normalized config string directly:** `normalizeDiscordToken()` calls `normalizeResolvedSecretInputString()` and `resolveDiscordToken()` returns that config value as the token when present.
  - file: [extensions/discord/src/token.ts:13](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/extensions/discord/src/token.ts#L13)
