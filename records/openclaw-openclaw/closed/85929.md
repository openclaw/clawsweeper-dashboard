## Summary

Keep this open: the protected `maintainer` label blocks cleanup closure, and current `main` still source-reproducibly rejects both the slash-deep `oc://openclaw.json/...` shape and JSON object leaf replacement through `openclaw path set`. Existing dotted JSONC addressing and `openclaw config set` provide partial workarounds, but they do not settle the requested path-substrate UX/API change.

## What This Changes

The issue asks for `openclaw path` to support deeper `openclaw.json` paths and JSON/object-valued config replacements so nested config and SecretRef edits do not require hand-editing.

## Best Possible Solution

Maintainers should choose the public path grammar and value-mode direction, then implement either clearer dotted-path/error guidance or a scoped deep-path plus explicit JSON/SecretRef replacement mode with focused tests.

## Reproduction Assessment

Yes, from source inspection: current `parseOcPath` rejects more than four slash slots, and `setStructuredLeaf` still coerces JSONC leaf writes by the existing node type, rejecting object values. I did not execute the CLI in this read-only review, but current tests and docs codify the same behavior.

## Solution Assessment

No, raising the global depth cap alone is not clearly the best fix because current JSONC paths already support dotted subsegments and the four-slot grammar is documented. The safer path is a maintainer decision between docs/error guidance, an `openclaw.json`-scoped grammar extension, and an explicit JSON replacement mode with SecretRef-focused validation.
