## Summary

Keep open. Current `main` still kicks off ACP startup identity reconciliation immediately after plugin-service startup, while the embedded ACPX backend only becomes healthy after a detached async probe. That means `requireAcpRuntimeBackend()` can still throw `ACP_BACKEND_UNAVAILABLE` during startup warmup, matching the reported race and misleading warning.

## Evidence

- **Startup path still reconciles immediately:** `startGatewaySidecars` starts plugin services and then immediately calls `getAcpSessionManager().reconcilePendingSessionIdentities(...)`, logging a warning summary or failure from that startup reconcile path. There is no wait for ACPX backend readiness before this call.
  - file: [src/gateway/server-startup-post-attach.ts:307](https://github.com/openclaw/openclaw/blob/80b6da72f5f0ac5ade8262142f816b9287fb1cfe/src/gateway/server-startup-post-attach.ts#L307)

  - sha: [80b6da72f5f0](https://github.com/openclaw/openclaw/commit/80b6da72f5f0ac5ade8262142f816b9287fb1cfe)
- **Plugin services do not wait for backend health:** `startPluginServices` only awaits each service's `start()` method and then moves on. It has no generic readiness barrier beyond service start completion.
  - file: [src/plugins/services.ts:33](https://github.com/openclaw/openclaw/blob/80b6da72f5f0ac5ade8262142f816b9287fb1cfe/src/plugins/services.ts#L33)

- **ACPX service registers backend before async probe completes:** The embedded ACPX service registers the runtime backend with `healthy: () => runtime?.isHealthy() ?? false`, logs it as registered, and then launches `probeAvailability()` in a detached async task. The `ready` log is only emitted later if the probe succeeds, so `start()` can return while the backend is still unhealthy.
