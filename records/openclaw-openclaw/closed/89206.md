## Label Justifications

- `P2`: This is a bounded session persistence bug with real transcript replay impact but no evidence of security impact, data loss, or unusable core runtime.
- `impact:session-state`: The issue is about corrupted session JSONL state causing resumed branches to replay duplicated transcript entries.

## Summary

The issue is still necessary: current main still has the `SessionManager.persist()` path that appends the full in-memory transcript after `flushed` is reset, and the existing embedded-runner normalizer only narrows some startup cases rather than fixing the core session manager behavior.

## What This Changes

The issue requests changing first-assistant session persistence after resume or recovery so the full transcript buffer overwrites the JSONL file instead of appending duplicate existing entries.

## Best Possible Solution

Change the full-buffer flush in `SessionManager.persist()` to rewrite the JSONL file and add focused regression coverage for empty/corrupt recovery plus header-and-user-only resume.

## Reproduction Assessment

Yes, from source inspection: `setSessionFile()` can mark an existing or rewritten file as flushed, a user-only append resets `flushed`, and the first assistant then appends the entire loaded buffer. I did not run the provided tests because this review was read-only.

## Solution Assessment

Yes, rewriting on the unflushed full-buffer branch is the narrowest maintainable fix because that branch is writing the complete transcript, while the already-flushed branch should continue appending only the new entry.
