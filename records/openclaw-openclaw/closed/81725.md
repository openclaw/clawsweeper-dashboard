## Summary

Keep open for maintainer review. This focused bug-fix PR targets a valid macOS Gateway split-brain bug, but the patch still lets the Gateway LaunchAgent bootstrap repair run before the new app-ownership skip and also treats app launch-at-login as Gateway ownership even when attach-only disables app launchd management.

## What This Changes

The PR adds macOS app LaunchAgent detection, skips default CLI gateway doctor service repairs when detected, and updates related tests, docs, and changelog text.

## Best Possible Solution

Land one corrected fix that guards every CLI Gateway service lifecycle mutation, including bootstrap and service-config rewrite, on a precise app-local-mode ownership signal while preserving named profile services.

## Reproduction Assessment

Yes, source-backed rather than live macOS: current main has doctor paths that can bootstrap, install, restart, or rewrite `ai.openclaw.gateway` while the app LaunchAgent `ai.openclaw.mac` exists. The related issue comments also describe the same terminal doctor conflict and manual removal workaround.

## Solution Assessment

No. The direction is right, but this patch applies the skip too late for bootstrap repair and uses the app login LaunchAgent as a Gateway ownership signal without honoring attach-only state.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Guard the bootstrap repair before checking launchd ownership:** `src/commands/doctor-gateway-daemon-flow.ts:132-138`
  - body: The new `macAppOwnership` guard is only consulted after the Darwin bootstrap block. If a stale `ai.openclaw.gateway.plist` exists but is not loaded, `doctor --fix` can still bootstrap that LaunchAgent before reaching the skip, bringing back the duplicate Gateway process this PR is meant to prevent. Move the app-owner guard ahead of bootstrap or make the bootstrap helper honor it.
  - confidence: 0.92
- **[P2] Use a real gateway-ownership signal for the app guard:** `src/commands/doctor-gateway-daemon-flow.ts:132-138`
  - body: The guard treats any `ai.openclaw.mac.plist` that launches OpenClaw.app as proof that the app owns the Gateway, but that plist is the app launch-at-login job. `GatewayLaunchAgentManager` has an attach-only marker that explicitly disables app management of `ai.openclaw.gateway`, so terminal doctor would skip CLI repair even when the app will not manage the Gateway. Include that state or avoid using the login agent alone.
  - confidence: 0.86
