## Summary

Keep open: this is a real, source-backed performance fix with sufficient live proof, but the latest PR head still leaves blocking async-discovery defects that would keep gateway hot paths doing synchronous or duplicate metadata work.

## What This Changes

The PR adds async plugin registry/metadata auto-enable paths for gateway channel status/start/send and changes WhatsApp Web close handling so transport-class closes retry inside the monitor.

## Best Possible Solution

Keep the async read-path and WhatsApp retry direction, but make the auto-enable flow fully async and snapshot-aware, key async dedupe by full install-record contents, and add focused regression coverage before merge.

## Reproduction Assessment

Yes, source-backed. Current main still routes channel status, channel start, and send through synchronous plugin auto-enable and registry metadata checks, and the PR body plus linked issue include concrete slow-filesystem WhatsApp logs; I did not run a live reproduction.

## Solution Assessment

No, not yet. Asyncizing the persisted-read path is the right maintainable direction, but the proposed implementation still enters synchronous discovery, misses current-snapshot reuse, and has unsafe async dedupe semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Make candidate detection async before resolving candidates:** `src/config/plugin-auto-enable.apply.ts:84`
  - body: The new async entry point still calls detectPluginAutoEnableCandidates, and that detector resolves the manifest registry through the synchronous resolver before returning candidates. For WhatsApp/channel configs this can still hit loadPluginMetadataSnapshot synchronously on the gateway hot path, so the PR does not remove the event-loop block it targets.
  - confidence: 0.91
- **[P2] Reuse the current metadata snapshot before async discovery:** `src/config/plugin-auto-enable.shared.ts:998-1001`
  - body: The synchronous resolver first checks getCurrentPluginMetadataSnapshot and only falls back to disk-backed metadata loading when no compatible snapshot exists. The async resolver skips that fast path and always calls loadPluginMetadataSnapshotAsync, so status/start/send can re-read metadata after startup already produced a compatible snapshot.
  - confidence: 0.88
- **[P3] Key async dedupe by full install record inputs:** `src/plugins/plugin-registry-snapshot.ts:449-450`
  - body: buildAsyncRegistrySnapshotKey only includes the install record ids, but index construction consumes the full record contents. Concurrent callers with the same plugin ids but different specs or install paths can share the first caller's snapshot; include a stable digest of the records or avoid dedupe for custom records.
  - confidence: 0.81
