## Summary

Close as implemented. Current main and v2026.4.27 already removed the warning-emitting full-config load from the console settings path: `readLoggingConfig()` now reads the active config file directly and the old production fallback loader is a no-op, so the startup recursion route this PR guarded is gone. The PR branch is also not a good replacement because its new re-entrant stderr fallback bypasses normal console redaction.

## What This Changes

The PR wraps console settings resolution in a re-entry guard and adds a direct stderr path for console output emitted while logging configuration is resolving.

## Best Possible Solution

Close this stale PR and keep the shipped v2026.4.27 logging implementation. If maintainers still want belt-and-suspenders recursion protection, it should be a fresh, narrow patch on current main that preserves redaction and adds a regression test for re-entrant console output during logging config resolution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Redact the re-entrant stderr fallback:** `src/logging/console.ts:252`
  - body: The new `resolvingConsoleSettings` branch writes `formatted` directly to stderr before the existing `redactSensitiveText()` calls run. Any config-load warning containing an API key, token, or env assignment would now be emitted unredacted on this path, so the fallback should redact before writing.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The PR diff introduces a concrete redaction bypass in the direct stderr fallback added for re-entrant console output.

Concerns:

- **[medium] Raw stderr fallback bypasses redaction:** `src/logging/console.ts:252`
  - body: During `loggingState.resolvingConsoleSettings`, the patch writes the formatted console message straight to `process.stderr` and returns before the normal redaction path. That can leak secrets present in startup/config warnings.
  - confidence: 0.86
