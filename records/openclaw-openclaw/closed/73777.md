## Summary

Current main and the latest release already implement the manual main-session cron handoff: the run result still reports `deliveryStatus: "not-requested"`, but that field is only the fallback/announce delivery status, not proof that the targeted heartbeat wake was skipped.

## What This Changes

The issue reports that manual `cron run` for `sessionTarget="main"` and `payload.kind="systemEvent"` on v2026.4.26 completes with `deliveryStatus: "not-requested"` and does not create an observable main-session agent turn.

## Best Possible Solution

Keep the shipped main-session heartbeat handoff and treat `deliveryStatus: "not-requested"` as fallback-delivery state. If operators need clearer diagnostics, track that separately as a narrow run-history/UI wording enhancement.

## Reproduction Assessment

Yes for the reported diagnostic: current Gateway coverage intentionally reproduces `deliveryStatus: "not-requested"` for a main-session systemEvent manual run. The missing-agent-turn failure no longer follows from current source because the event is queued to the session and the heartbeat wake is retried after cron-active contention clears.

## Solution Assessment

Yes. The current implementation is the narrow maintainable path: cron records its own run outcome, enqueues the main-session system event, and relies on the existing targeted heartbeat runner for execution instead of adding a second main-session agent runner path.
