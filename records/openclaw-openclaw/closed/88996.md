## Label Justifications

- `P1`: Feishu receives messages but cannot dispatch them to agents, breaking a real channel workflow for affected users.
- `impact:message-loss`: Inbound Feishu messages are received by the gateway but fail before agent dispatch, so chat replies are effectively dropped.

## Summary

Current main fixes the Feishu dispatch failure by removing the bundled-channel lightweight startup runtime path that could leave `channel.inbound` undefined; the fix is current-main-only and not in latest release v2026.5.28.

## What This Changes

The issue reports Feishu WebSocket messages being received but failing to dispatch to agent sessions with `Cannot read properties of undefined (reading 'run')` on v2026.5.28.

## Best Possible Solution

Keep the current main full-runtime startup path for bundled Feishu channels and ship it in the next release rather than adding Feishu-local defensive checks that hide broken runtime wiring.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: v2026.5.28 could pass a startup-only runtime to bundled Feishu while Feishu dispatch calls `core.channel.inbound.run`. I did not run live Feishu credentials, so this is source-reproducible rather than locally reproduced.

## Solution Assessment

Yes. Current main fixes the owner boundary in gateway channel startup by passing the full channel runtime; adding null checks inside Feishu would only turn the same wiring bug into a quieter message drop.
