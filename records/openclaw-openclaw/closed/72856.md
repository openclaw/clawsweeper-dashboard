## Summary

Close as implemented. Current main and the shipped v2026.4.27 tree already provide a history-aware cron path for non-default agents through explicit persistent session targets: use an agent-turn job with `--agent <id> --session session:main --message ...` or another `session:<id>`. Isolated cron runs remain intentionally fresh, and the default-agent-only `--session main --system-event` guard is still by design.

## What This Changes

Request cron jobs for non-default agents to access their main-session transcript/history instead of always starting from a fresh isolated session.

## Best Possible Solution

Keep isolated cron runs fresh by default and use explicit `current` or `session:<id>` targets for scheduled jobs that intentionally need transcript continuity. If maintainers want more polish later, the narrow follow-up would be a UX/docs tweak that points the non-default `main` error toward `--session session:main --message`.

## Reproduction Assessment

Yes. The old failing spelling is clear from the report and still intentionally rejects non-default `--session main --system-event`; the supported reproduction/verification path is the source-backed `--agent ops --session session:main --message ...` route, which maps to `agent:ops:main` and does not force a fresh isolated session.

## Solution Assessment

Yes. Explicit persistent session targets are the narrowest maintainable solution because they preserve isolated cron's no-ambient-context semantics while giving history-aware jobs a deliberate transcript-sharing path.
