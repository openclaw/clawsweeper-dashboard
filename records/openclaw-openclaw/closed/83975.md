## Summary

The current source still constructs ClawHub request URLs in a way that drops non-root base pathnames, and no linked or searched PR appears to own this exact path-prefix bug yet. This should stay open as a narrow, high-confidence repair candidate.

## What This Changes

The issue reports that ClawHub API requests lose a configured path prefix from `OPENCLAW_CLAWHUB_URL` or `CLAWHUB_URL`, causing prefixed reverse-proxy deployments to receive 404s.

## Best Possible Solution

Update the shared ClawHub URL builder to join normalized base URLs and API paths without discarding an existing base pathname, then add focused regression coverage for prefixed `OPENCLAW_CLAWHUB_URL`, `CLAWHUB_URL`, and explicit `baseUrl` inputs.

## Reproduction Assessment

Yes from source inspection: current `main` passes leading-slash ClawHub API paths into a `new URL()` pattern that discards a configured base pathname. I did not run a live reverse-proxy ClawHub instance during this read-only review.

## Solution Assessment

Yes: fixing shared `src/infra/clawhub.ts` URL construction is the narrowest maintainable solution because every affected ClawHub endpoint already flows through that helper.
