## Summary

Current `main` already preserves the configured Moonshot CN `baseUrl` during implicit provider resolution, exposes the `.cn` onboarding path, and the changelog shows this shipped in `2026.3.12`.

## Evidence

- **Explicit baseUrl is preserved in implicit provider catalogs:** `buildSingleProviderApiKeyCatalog` reads `ctx.config.models.providers[providerId].baseUrl` when `allowExplicitBaseUrl` is enabled and overlays that `baseUrl` onto the generated provider config before writing the implicit catalog.
  - file: [src/plugins/provider-catalog.ts:25](https://github.com/openclaw/openclaw/blob/c6684af682bcb59bd1ee85cb49f72d4074007e2c/src/plugins/provider-catalog.ts#L25)
  - command: `nl -ba src/plugins/provider-catalog.ts | sed -n '1,90p'`
- **Moonshot plugin opts into explicit baseUrl preservation:** The bundled Moonshot provider registers `catalog.buildProvider = buildMoonshotProvider` with `allowExplicitBaseUrl: true`, so a configured `.cn` endpoint is kept instead of being replaced by the default `.ai` endpoint.
  - file: [extensions/moonshot/index.ts:53](https://github.com/openclaw/openclaw/blob/c6684af682bcb59bd1ee85cb49f72d4074007e2c/extensions/moonshot/index.ts#L53)
  - command: `nl -ba extensions/moonshot/index.ts | sed -n '1,90p'`

  - file: [extensions/moonshot/onboard.ts:36](https://github.com/openclaw/openclaw/blob/c6684af682bcb59bd1ee85cb49f72d4074007e2c/extensions/moonshot/onboard.ts#L36)
