## Summary

Close as duplicate/superseded: current main still uses Scheduled Tasks for native Windows gateway management, but the same user-facing crash-recovery gap is already tracked in the open canonical issue at https://github.com/openclaw/openclaw/issues/82125. This issue’s Windows Service/NSSM adapter details are useful design input for that thread rather than a separate work item.

## What This Changes

Request adding a native Windows Service adapter and configurable lifecycle resolver for gateway start/stop/restart/status while falling back to Scheduled Tasks.

## Best Possible Solution

Use https://github.com/openclaw/openclaw/issues/82125 as the canonical Windows crash-recovery lifecycle decision point, carrying this issue’s Windows Service/NSSM adapter and config-name details into that discussion.

## Reproduction Assessment

Yes, for the source-level gap: current main maps native Windows gateway lifecycle to Scheduled Tasks and creates an `ONLOGON` task, with no Windows Service adapter. I did not run a live Windows crash-restart reproduction in this read-only review.

## Solution Assessment

Unclear; a Windows Service adapter is plausible, but it is a product/design choice alongside the watchdog design already discussed in https://github.com/openclaw/openclaw/issues/82125. The maintainable path is to decide the Windows supervisor contract in that canonical issue rather than split the work.
