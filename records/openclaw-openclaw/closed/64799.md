## Summary

Current main and the shipped v2026.4.23 release contain the Volcengine plan provider/model wiring plus the codex models.json fix that matches the thread’s multi-provider `Unknown model` failure pattern.

## Evidence

- **Volcengine provider wiring on main:** The bundled Volcengine plugin registers both `volcengine` and `volcengine-plan` in its runtime catalog and attaches the resolved API key to both provider entries, so `volcengine-plan` is a first-class built-in provider on current main.
  - file: [extensions/volcengine/index.ts:46](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/extensions/volcengine/index.ts#L46)
  - sha: [fd65caf4b0ba](https://github.com/openclaw/openclaw/commit/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc)
- **Correct coding endpoint and default model on main:** Current source defines the coding endpoint as `https://ark.cn-beijing.volces.com/api/coding/v3` and the coding default model id as `ark-code-latest`, matching the issue’s expected provider/model pair.
  - file: [extensions/volcengine/models.ts:9](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/extensions/volcengine/models.ts#L9)
  - sha: [fd65caf4b0ba](https://github.com/openclaw/openclaw/commit/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc)
- **Current user-facing docs still describe the same built-in provider/model:** The docs page documents `volcengine` plus `volcengine-plan`, recommends `volcengine-plan/ark-code-latest`, and lists the coding endpoint as `ark.cn-beijing.volces.com/api/coding/v3`.
  - file: [docs/providers/volcengine.md:15](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/docs/providers/volcengine.md#L15)
  - sha: [fd65caf4b0ba](https://github.com/openclaw/openclaw/commit/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc)
- **Shipped codex catalog fix present in released code:** The released v2026.4.23 tree includes `apiKey: CODEX_APP_SERVER_AUTH_MARKER` in the codex provider catalog output. That is the exact hardening needed so the Pi ModelRegistry accepts the codex provider row instead of rejecting it and breaking other custom/provider rows in `models.json`.
  - file: [extensions/codex/provider-catalog.ts:65](https://github.com/openclaw/openclaw/blob/a9797214338ba31c52c796adbb75afb16e0684a9/extensions/codex/provider-catalog.ts#L65)

  - sha: [a9797214338b](https://github.com/openclaw/openclaw/commit/a9797214338ba31c52c796adbb75afb16e0684a9)
- **Release note ties the shipped fix to the reported symptom class:** The changelog in the shipped v2026.4.23 release says `#66180` makes the codex provider catalog include `apiKey` so the Pi ModelRegistry no longer rejects the entry and silently drops custom models from every provider in `models.json`, which matches the issue thread’s cross-provider `Unknown model` reports and workaround discussion.
  - file: [CHANGELOG.md:816](https://github.com/openclaw/openclaw/blob/a9797214338ba31c52c796adbb75afb16e0684a9/CHANGELOG.md#L816)

That discussion aligns with the shipped changelog fix rather than an unresolved current-main Volcengine namespace bug.
