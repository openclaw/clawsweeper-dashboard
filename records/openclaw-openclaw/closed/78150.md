## Summary

A clean published-package check of 2026.5.4 did not reproduce the reported empty-directory failure: bare `file-transfer` resolved to the bundled package path, runtime inspection loaded the plugin, and all four expected tools were registered.

## What This Changes

The issue reports that installing the bundled file-transfer plugin creates an empty state-directory install and leaves its tools unavailable.

## Best Possible Solution

Keep bundled installs path-backed to packaged `dist/extensions/file-transfer`; investigate stale local state only if a new report shows an empty configured path shadowing the bundled plugin.

## Reproduction Assessment

Yes: a clean scratch npm install of `openclaw@2026.5.4` with isolated `HOME` and `OPENCLAW_HOME`, followed by `openclaw plugins install file-transfer` and `openclaw plugins inspect file-transfer --runtime --json`, is a high-confidence reproduction path. That path did not reproduce the empty-directory or missing-tools failure.

## Solution Assessment

Yes: the existing bundled-plugin install behavior is the narrow maintainable solution for a plugin shipped inside the OpenClaw package. Copying bundled plugin files into a user state directory is not required for `file-transfer` to load.
