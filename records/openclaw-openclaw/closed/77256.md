The PR adds a typed, fail-closed `before_model_call` plugin preflight hook across Pi, CLI, and Codex model submission paths, with docs, config/API baselines, and tests.

## Best Possible Solution

Land one coherent model-call hook API only after preserving existing conversation gates, making blocked turns side-effect-free, resolving the overlap with https://github.com/openclaw/openclaw/pull/58360, and attaching real allow/block proof.

## Reproduction Assessment

Not applicable for the requested feature because this is a new public plugin hook, not a current-main bug. The blocking PR defects are source-reproducible from the PR head: hook classification drops existing gated names and runtime context queues before the preflight decision.

## Solution Assessment

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve existing conversation hook gates:** `src/plugins/hook-types.ts:171-177`
  - body: Current main gates `before_model_resolve`, `before_agent_reply`, and `before_agent_run` behind `allowConversationAccess`, but this branch's `CONVERSATION_HOOK_NAMES` drops those names while adding `before_model_call`. That lets non-bundled plugins register existing prompt/session hooks without the required trust opt-in; add the new hook without removing the existing names and restore loader/docs coverage.
  - confidence: 0.91
- **[P2] Queue runtime context only after preflight passes:** `src/agents/pi-embedded-runner/run/attempt.ts:3011-3016`
  - body: `queueRuntimeContextForNextTurn` runs before `before_model_call`. If the hook blocks or fails, prompt submission is skipped but the queued runtime context can still be delivered on a later turn; move queueing into the allowed branch after preflight succeeds and cover a blocked runtime-context turn.
  - confidence: 0.9
