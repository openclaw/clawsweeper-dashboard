## Summary

Keep open. Current main still rejects persisted `type: "aws-sdk"` auth-profile entries as `invalid_type`, so the reported Bedrock/IMDS store-load failure remains even though provider-level Bedrock AWS SDK auth is implemented.

## What This Changes

The issue asks OpenClaw to handle persisted Bedrock `auth-profiles.json` entries with `type: "aws-sdk"` so EC2 IMDS/default AWS SDK credentials keep working.

## Best Possible Solution

Add a secops-reviewed auth-profile store contract for SDK-managed AWS auth markers, with regression coverage for `amazon-bedrock:default` plus mixed real-invalid entries, while preserving provider-level AWS SDK/default-chain behavior.

## Reproduction Assessment

Yes. A stored `auth-profiles.json` profile with `type: "aws-sdk"` reaches `parseCredentialEntry`, misses the current three-value allow-list, is rejected as `invalid_type`, and is dropped during store load.

## Solution Assessment

Unclear for the exact implementation shape. PR #69753’s marker-skip approach is narrow and plausible, but the durable store contract should explicitly decide whether to skip, accept, or type SDK-managed markers.
