## Summary

Keep open. The PR is useful source work for the linked diagnostic-log growth bug, but current evidence still shows two blocking problems: it routes diagnostics into a cap that suppresses future writes instead of rotating, and it exposes new config/env knobs without complete public metadata/docs coverage.

## What This Changes

The PR adds max-byte defaults and parsing for cache-trace and Anthropic payload JSONL logs, wires `diagnostics.cacheTrace.maxFileBytes` through config schema/types, and adds focused max-byte tests.

## Best Possible Solution

Land a focused bounded-rotation implementation for queued diagnostic JSONL files that keeps a small retained archive set, continues writing to a fresh active file after the cap, preserves existing redaction and filesystem protections, and publishes the config/env/docs/changelog surface with regression tests.

## Reproduction Assessment

Yes. Fill `cache-trace.jsonl` or `anthropic-payload.jsonl` to the configured cap; the PR routes both through `safeAppendFile(..., { maxFileBytes })`, whose cap branch returns before appending future diagnostic lines.

## Solution Assessment

No. Capping by suppressing writes bounds disk usage but loses diagnostics for long-lived gateways; the narrow maintainable fix is rotation/retention matching the existing primary logger model, plus complete public config coverage.
