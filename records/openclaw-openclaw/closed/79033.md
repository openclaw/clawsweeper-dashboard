## Summary

Keep open, but the PR is not mergeable as written. It proposes a real new capability, yet broadens the prompt-bootstrap file trust boundary without inspectable real behavior proof, focused docs/tests, or a safer capped read path, and it includes unrelated lockfile metadata.

## What This Changes

The PR adds a `bootstrapTrustExternal` option to `bootstrap-extra-files` so configured bootstrap files can be read from outside the workspace, and also adds an unrelated `extensions/runs-registry` lockfile importer.

## Best Possible Solution

If maintainers want this capability, implement it as a documented opt-in trust-boundary feature with descriptor-based or otherwise byte-capped reads, focused default-reject and opt-in tests, docs/changelog coverage, and no unrelated lockfile drift.

## Reproduction Assessment

Not applicable; this PR proposes a new opt-in capability, not a broken existing behavior. Current main is source-verifiable as workspace-only through `readWorkspaceFileWithGuards` and has tests for outside-path rejection.

## Solution Assessment

No/unclear; as written, the direct `fs.readFile` path and undocumented config key are not the narrowest maintainable change for a prompt-bootstrap trust boundary. A safer version would preserve byte enforcement during the actual read and add docs plus focused opt-in/default-reject coverage.
