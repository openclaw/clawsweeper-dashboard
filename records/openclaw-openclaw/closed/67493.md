## Summary

Keep open. The branch is useful source work, but it is not merge-ready because it adds an unwired Gateway helper that would currently send chat replies as diagnostic `push.test` APNs alerts, skips canonical main-session keys, and puts reply text into push-notification content.

## What This Changes

The PR adds a new Gateway `chat-apns-notify` helper and unit test intended to send APNs alerts with chat reply text to offline main-session iOS requester devices, while deferring lifecycle wiring.

## Best Possible Solution

Complete or supersede this with an end-to-end Gateway and iOS implementation: a dedicated chat-reply APNs kind, privacy-safe or explicitly opt-in alert content, canonical main-session resolution, request-device connection tracking in both chat completion paths, focused tests, and a changelog entry.

## Reproduction Assessment

Yes. The issues are statically reproducible from the PR diff and current main: the helper is not imported by the chat lifecycle, it calls the generic `push.test` APNs sender, and the main-session guard does not use the canonical session-key helpers.

## Solution Assessment

No. The helper is a useful seed, but the maintainable fix needs a dedicated chat notification contract plus full Gateway/iOS wiring and a privacy decision before merge.

## Review Findings

  - body: `sendApnsAlert` builds the current `push.test` payload kind, so a wired chat reply would be indistinguishable from diagnostic test pushes and would not have a reliable iOS chat route. Add a dedicated chat-reply payload/send helper before using this from chat completion paths.
  - confidence: 0.95

  - body: The default guard compares `params.sessionKey` to raw `cfg.session.mainKey`, but current Gateway main sessions can be `global` or `agent:<id>:<mainKey>`. Valid main-session replies can return early and never notify.
  - confidence: 0.9
- **[P2] Keep reply text out of APNs alerts by default:** `src/gateway/chat-apns-notify.ts:57-60`
  - body: The helper turns assistant reply text into the APNs alert body. That content can traverse APNs or the relay and appear on lock screens; use a generic body or an explicit opt-in privacy setting before sending transcript text.
  - confidence: 0.84
- **[P2] Wire the helper into chat completion paths:** `src/gateway/chat-apns-notify.ts:24-34`
  - body: The new helper is never called by `server-chat` or direct `chat.send`, so merging this feature would not actually send offline chat reply alerts. Add the lifecycle integration in the same feature change or keep this preparatory module unmerged.
  - confidence: 0.88
