## Summary

Keep open: the WhatsApp 428 retry path looks like the right direction and has useful live log proof, but the latest head also adds a Pi resource-loader mitigation that does not actually bypass upstream resource discovery and lacks real Windows behavior proof.

## What This Changes

This PR routes WhatsApp opening-phase Baileys 428 closes through reconnect handling, adds retry regression coverage, and adds an embedded Pi resource-loader helper intended to avoid Windows resource-discovery stalls.

## Best Possible Solution

Keep the WhatsApp opening-phase 428 retry and regression coverage, but either split the Windows resource-loader work out or replace it with a dependency-contract-proven bypass of Pi filesystem discovery plus real Windows proof before claiming that stall is fixed.

## Reproduction Assessment

Partially: yes for https://github.com/openclaw/openclaw/issues/75736, because current main rethrows non-auth openConnection setup errors and the linked report/PR logs show Boom 428 escaping as channel exited. No high-confidence live reproduction was provided for the full Windows stall in https://github.com/openclaw/openclaw/issues/77443 on the latest head.

## Solution Assessment

No as submitted. The WhatsApp retry change is the narrow channel-owned fix, but the Pi helper uses DefaultResourceLoader flags that do not stop upstream resource discovery, so the Windows stall fix needs a true discovery bypass or should be split out.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Bypass discovery instead of only filtering loads:** `src/agents/pi-embedded-runner/resource-loader.ts:5-10`
  - body: The new helper sets `noSkills`, `noPromptTemplates`, and related flags, but in pi-coding-agent 0.74.0 `DefaultResourceLoader.reload()` still calls `DefaultPackageManager.resolve()` first. That path recursively scans project/user `.pi` resources and ancestor/user `.agents/skills` with synchronous filesystem calls before these flags are applied, so the Windows event-loop stall this changelog now says is fixed can still happen during first embedded agent startup. Please use a loader path that actually skips package/resource discovery, or split this claim out until the dependency exposes that control.
  - confidence: 0.86
