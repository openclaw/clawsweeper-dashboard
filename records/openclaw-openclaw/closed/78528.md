## Summary

Keep open: current main still has a source-backed path where resolved skill `apiKey` values are written to `process.env`, and generic host `exec` builds child environments from that same process env without stripping active skill keys. The related provider-secret issue covered a different provider config path, while the open skill-env issue and PR pull in the opposite usability direction, so this needs maintainer/security design rather than cleanup closure.

## What This Changes

The issue asks for SecretRef-backed `skills.entries.<skill>.apiKey` values to stay out of generic `exec` child process environments.

## Best Possible Solution

Define a scoped skill-secret boundary where generic `exec` does not inherit active skill SecretRef-derived keys, while intended skill-owned command paths can still receive explicit env values with tests and docs.

## Reproduction Assessment

Yes. Source inspection shows resolved skill API keys are written into `process.env`, and generic host `exec` builds child envs from that process env without receiving the active skill strip set; no live `env` command was run in this read-only review.

## Solution Assessment

Unclear as a one-line fix. Stripping active skill keys from generic host exec addresses the leak, but the final design must preserve or intentionally revise documented skill env injection and the related skill CLI usability requirement.
