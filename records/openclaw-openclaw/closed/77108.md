## Summary

Current main already handles the reported OpenAI Codex account-id refresh failure by catching the pi-ai extraction error, reusing the cached credential, and feeding the Codex app-server refresh response from the selected stored profile.

## What This Changes

The issue reports that legacy `codex/gpt-*` Codex harness runs fail after `openai-codex` OAuth refresh when pi-ai cannot extract `chatgpt_account_id`, with `openclaw models auth login --provider codex` unavailable as a recovery path.

## Best Possible Solution

Keep the current main fallback and profile-based app-server refresh behavior, then ship it so affected `v2026.5.2` users receive the fix while continuing to use `openai-codex` for auth commands.

## Reproduction Assessment

Yes. The failure condition is represented by source-level regression coverage that mocks `Failed to extract accountId from token`; I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

Yes. The current fix is narrow: provider-owned Codex OAuth refresh handles the known pi-ai extraction failure, while the app-server bridge uses the stored profile account id instead of adding a new `codex` auth login flow.
