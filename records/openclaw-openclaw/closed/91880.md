## Summary

This PR fixes the documented hyphenated naming mismatch and now keeps core and memory-host resolution aligned, but it changes a persistent workspace location that has shipped since April without migrating existing nested workspaces. That can silently disconnect non-default agents from their existing files, skills, and memory, so the branch should remain open for an explicit path-contract decision and upgrade-safe implementation.

## What This Changes

The PR changes non-default agent fallback workspaces from `<defaults.workspace>/<agentId>` to `<defaults.workspace>-<agentId>` in core and memory-host resolution, updates related tests, and adds a reproduction script.

## Best Possible Solution

Choose the canonical workspace naming contract explicitly; if hyphenated paths remain the target, add a doctor-owned migration that detects, moves, verifies, and retires shipped nested workspaces before both runtime resolvers assume the new location.

## Reproduction Assessment

Yes. Current main deterministically resolves this configuration to `<defaults.workspace>/<agentId>`, while the PR deterministically resolves it to `<defaults.workspace>-<agentId>`; the supplied Linux output also demonstrates the proposed core behavior.

## Solution Assessment

No, not as submitted. Synchronizing both resolver copies is necessary, but changing a shipped persistent path without doctor migration or an explicit compatibility decision is not upgrade-safe.
