## Summary

Keep open. Current main has generic tool-error wrapping and failed-tool warning plumbing, but the memory-flush caller still runs as a silent maintenance turn, filters normal text error payloads, ignores the result, and only logs caught failures, so the reported silent memory-flush failure path is not safely closed.

## What This Changes

The issue asks for rejected memory-flush write tools to surface a visible error or terminate cleanly instead of leaving embedded WebChat/TUI runs silently active.

## Best Possible Solution

Preserve the append-only memory safety rail, but make memory-flush restricted write failures terminate the maintenance turn promptly and emit a visible terminal/error signal or explicit diagnostic instead of being swallowed by silent-turn filtering.

## Reproduction Assessment

No high-confidence live reproduction was established because I did not run the reported macOS/Bedrock/WebChat setup. Source inspection does trace the restricted memory-flush write through silentExpected filtering and result suppression, which is enough to keep this open as source-reproducible.

## Solution Assessment

No, current main’s generic tool-error adapter is not sufficient for this specific issue because the memory-flush caller still suppresses or ignores normal text error output. The safer fix is a narrow memory-flush/embedded-runner liveness and error-propagation change, not weakening the write whitelist.
