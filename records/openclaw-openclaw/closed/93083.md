## Label Justifications

- `P1`: The patch targets a real multi-agent workspace regression but currently risks routing active session/workspace state to the wrong agent.
- `merge-risk: 🚨 session-state`: Merging can mis-associate session store, transcript, and workspace state when the session key already encodes a different target agent.
- `merge-risk: 🚨 security-boundary`: Wrong workspace selection can load the wrong AGENTS.md/SOUL.md and suppress intended per-agent safety or policy context.

## Summary

Keep open: the bugfix direction is valid, but this patch makes routed `AgentId` behave like an explicit override, so an agent-scoped target session can be resolved under the route agent instead of the session-key agent.

## What This Changes

The PR adds routed `route.agentId` to inbound channel context and passes that context `AgentId` into session-state agent resolution.

## Best Possible Solution

Add fallback-only route-agent resolution below session-key precedence, update all inbound resolver call sites consistently, and cover both unscoped-bound and scoped-target cases with focused tests.

## Reproduction Assessment

Yes, source-reproducible. Current `resolveSessionAgentIds` gives `agentId` precedence over parsed session-key agents, and this PR starts populating that field from the channel route.

## Solution Assessment

No. Carrying the routed agent is the right boundary, but it must be a fallback below the session-key agent rather than using the existing explicit `agentId` parameter.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Make route AgentId fallback-only before populating it:** `src/channels/inbound-event/context.ts:513`
  - body: This new context field is consumed by `getReplyFromConfig` and dispatch/session initialization through `resolveSessionAgentId({ agentId: ... })`, but that resolver treats `agentId` as an explicit override before parsing `sessionKey`. After this line, a native command or bound conversation targeting an `agent:other:...` session can be resolved under the route agent instead of the target session's agent, so workspace and session state can move to the wrong agent. Add fallback-only resolver semantics where the session-key agent wins, then use that for routed `AgentId`.
  - confidence: 0.93
