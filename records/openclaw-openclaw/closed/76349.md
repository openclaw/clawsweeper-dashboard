## What This Changes

The issue requests that `openclaw devices approve <requestId>` approve a pending Control UI device pairing when the local CLI operator device already has `operator.admin`.

## Best Possible Solution

Keep the current main CLI scope-resolution fix and carry it into the next release so 2026.4.29 users get the corrected approval path by updating.

## Reproduction Assessment

No, there is no current-main failing reproduction path from source inspection. The reported 2026.4.29 behavior matches the old least-privilege approval call, while current main now requests the approval-specific admin scope and has regression tests for that path.

## Solution Assessment

Yes. The current implementation is the narrowest maintainable fix: the CLI asks for the scopes needed by the pending approval while the Gateway keeps its existing approval-time authorization and cross-device guard.
