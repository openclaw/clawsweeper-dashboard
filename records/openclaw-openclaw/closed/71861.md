## Summary

Current main already supports the requested custom OpenAI-compatible provider flow: users can configure a custom provider id, base URL, API key, model id, and OpenAI-compatible adapter, and runtime sends that model through the OpenAI Chat Completions transport using the configured base URL. The broader fully custom/non-compatible transport idea is separate from this base-URL/model request.

## Best Possible Solution

Close this issue as implemented. Keep the shipped `custom-api-key` onboarding and `models.providers.<id>` configuration path for custom OpenAI-compatible endpoints; if maintainers want arbitrary non-OpenAI/Anthropic transports, track that as a separate provider plugin/API design item rather than reopening this base-URL/model request.
