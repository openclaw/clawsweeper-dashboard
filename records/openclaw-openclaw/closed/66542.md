## Summary

Keep this PR open. Current main does not implement the PR’s intended MCP reconnect behavior: bundle MCP startup is still single-pass/sequential, `callTool` still fails directly when a server session is missing, and the public config/schema only exposes `connectionTimeoutMs` plus `sessionIdleTtlMs`, not reconnect/startup retry controls. The PR is also not safely obsolete because the posted review/security comments identify unresolved correctness and security-sensitive issues in the PR’s own reconnect design.

## Best Possible Solution

Keep the PR open for normal review. The best path is to rebase it onto current main, address the posted P1/P2 and security findings, add focused tests for concurrent reconnect callers, JSON-RPC `-32000..-32099` errors, retry exhaustion/resurrection, bounded startup concurrency, and safe catalog metadata, and update the MCP config schema/docs if retry controls become supported public config.

There is no catch path that classifies transport errors, joins an in-flight reconnect, retries, or marks/resurrects dead servers.

## Risks / Open Questions

- The PR touches MCP server process/HTTP transport lifecycle code, so reconnect retries, parallel startup, and disposal behavior can cause process/socket churn or leaks if the state machine is wrong.
- The posted security review flags possible sensitive launch detail exposure through catalog metadata and error-cause propagation; those need design fixes before merge.
- The PR patch changes only `src/agents/pi-bundle-mcp-runtime.ts` in the provided context and does not include focused tests for reconnect joining, JSON-RPC error classification, dead-server resurrection, startup concurrency bounds, or metadata redaction.
