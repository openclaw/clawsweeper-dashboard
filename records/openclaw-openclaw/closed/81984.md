- `impact:data-loss`: The report and current source path concern transcript content not being durably archived or visible to ingestion after rollover.

## Summary

Keep open: current main and the latest shipped release still swallow transcript archive rename failures during `/new` rollover, and the linked fixing PR is open rather than merged.

## What This Changes

The issue requests that `/stop` followed by `/new` durably archive the prior transcript or surface archive failures instead of allowing silent transcript loss.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/82081, or an equivalent narrow fix that surfaces `/new` archive failures while preserving `/stop` abort-only semantics.

## Reproduction Assessment

Yes, at source level: if `archiveFileOnDisk` throws during `/new` rollover, `archiveSessionTranscriptsDetailed` catches and ignores it, allowing the new session to continue without surfacing the failed archive.

## Solution Assessment

Yes in direction: surfacing reset/new archive failures is the narrow maintainable fix, while changing `/stop` into a transcript finalizer would conflict with the documented command contract. Maintainers still need to decide whether warning is enough or `/new` should fail closed.
