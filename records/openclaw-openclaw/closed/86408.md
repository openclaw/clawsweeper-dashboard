## Summary

Keep open. The core compaction auth-route fix is useful and has strong live proof, but the new preflight threshold helper treats any provider id `openai` as server-compaction-enabled even when `openai` is configured to a custom base URL/proxy; that does not match the existing payload policy and can defer local compaction for setups that will not receive `context_management`.

## What This Changes

The PR routes embedded OpenAI compaction through Codex auth when configured, preserves direct OpenAI fallback, adjusts Responses server-threshold preflight gating, adds focused tests, and updates `CHANGELOG.md`.

## Best Possible Solution

Align preflight threshold detection with the existing OpenAI Responses payload-policy endpoint eligibility, keep the scoped Codex compaction routing fix, and merge after focused tests and relevant checks pass.

## Reproduction Assessment

Yes. The original bug is supported by source inspection plus redacted before/after logs, and the remaining PR defect is source-reproducible with `provider: "openai"`, a custom `models.providers.openai.baseUrl`, and `responsesCompactThreshold` without explicit server compaction.

## Solution Assessment

No. The compaction auth-route fix is the right shape, but the threshold helper should mirror the payload policy's native-endpoint/server-compaction eligibility before merge.
