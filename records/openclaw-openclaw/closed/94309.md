## Summary

Current main already restores the quote-compatible Telegram default path: ordinary Telegram text sends use standard sendMessage unless richMessages is explicitly enabled, so the reported rich-message default regression is fixed on main but not proven shipped in v2026.6.8.

## What This Changes

The issue asks for OpenClaw Telegram bot messages to use an outbound shape that lets Telegram Desktop expose selected Quote & Reply, while keeping selected quote context available inbound.

## Best Possible Solution

Keep standard Telegram text delivery as the default quote-compatible path, use richMessages only as an explicit opt-in, and let the separate inbound quote contract proceed through the existing related issue and PR.

## Reproduction Assessment

Yes. The reporter provided same-chat A/B evidence that plain sendMessage exposes Quote & Reply while OpenClaw rich-message delivery did not, and v2026.6.8 source still used direct rich sends for normal text.

## Solution Assessment

Yes for the central outbound regression. Current main restores the plain sendMessage-compatible default and keeps rich messages opt-in, which is the narrowest maintainable fix for Quote & Reply compatibility while the inbound quote contract remains separate.
