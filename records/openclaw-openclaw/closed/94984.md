## Label Justifications

- `P1`: The report describes a release regression that crash-loops Telegram channels and breaks a real channel workflow.
- `impact:crash-loop`: The affected Telegram channels repeatedly exit on startup until restart attempts are exhausted.
- `impact:message-loss`: When all Telegram channel workers are down, Telegram messages cannot reach agent sessions through that channel.

## Summary

Current main already fixes the reported Telegram startup/send path by making rich Telegram Bot API calls explicit opt-in and using standard sendMessage by default; the latest stable release v2026.6.8 is still affected, so this is a current-main-only cleanup close.

## What This Changes

The issue reports all Telegram channels crash-looping after v2026.6.8 because Telegram rich-message handling reaches the protected grammY sendRichMessage API path during startup or default delivery.

## Best Possible Solution

Keep current main's richMessages opt-in and sendMessage-default implementation, then ship or backport it; no new disable flag is needed for this report.

## Reproduction Assessment

Yes. The v2026.6.8 source/package still routes default Telegram text chunks through raw sendRichMessage while grammY's raw API is a protected Proxy; I did not run a live gateway because this was a read-only review.

## Solution Assessment

Yes. The current main fix is the narrow maintainable solution: preserve sendMessage as the compatible default and keep Bot API 10.1 rich messages behind the existing richMessages opt-in.
