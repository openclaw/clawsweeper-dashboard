## Summary

Current `main` already exposes Telegram sticker resend metadata in inbound context via `ctxPayload.Sticker.fileId` / `fileUniqueId`, documents that surface, and wires sticker sending back through the Telegram plugin. The requested resend use case is covered, even though the field lives under `Sticker.*` rather than a nested `media.telegram_file_id` key.

## Evidence

- **Inbound context includes sticker metadata:** The Telegram inbound payload builder assigns `Sticker: allMedia[0]?.stickerMetadata`, so sticker metadata is exposed on the context payload delivered to agents.

## Risks / Open Questions
