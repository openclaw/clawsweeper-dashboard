## Summary

Current main and the latest release already fix the reported Gemini cachedContent conflict: the Google provider transport now sends cachedContent while omitting per-request systemInstruction, tools, and toolConfig, with regression coverage for that exact payload shape.

## What This Changes

The issue reports Gemini direct API requests failing when cachedContent or cacheRetention are enabled because OpenClaw sent cachedContent together with system and tool settings.

## Best Possible Solution

Keep the v2026.5.28 Google provider transport behavior and ask affected users on 2026.5.18 to upgrade; reopen only with evidence from v2026.5.28 or newer showing cached requests still include forbidden fields.

## Reproduction Assessment

No current-main failure remains: source and regression tests show cachedContent requests now omit systemInstruction, tools, and toolConfig. The reporter's 2026.5.18 behavior matches the old shipped failure mode, but v2026.5.28 contains the fix.

## Solution Assessment

Yes. The fix belongs in the Google provider transport because that plugin owns direct Gemini payload shaping, and the shipped implementation covers both manual cachedContent and cacheRetention-managed cachedContents without adding a new config surface.
