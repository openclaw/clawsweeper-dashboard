## Summary

Keep open. Current main still accepts arbitrary exec `host` strings at the tool schema boundary, normalizes unknown values to `null`, and then resolves `null` as the configured/default target. Foreground and running exec results also omit requested/effective target information, so the reporter's SSH/curl workaround is consistent with a real tool-layer signal gap rather than a documentation-only problem.

## What This Changes

Reject invalid exec `host` tool arguments or surface the requested/effective target so hostname-like values cannot silently run on the sandbox/default target.

## Best Possible Solution

Fail closed before launching a command when a direct exec tool call supplies a non-empty `host` outside `auto`, `sandbox`, `gateway`, or `node`; optionally also expose requested/effective target metadata in a model-visible way after verifying provider transport behavior. Keep existing exec defaults, approval policy, and sandbox/gateway/node routing semantics unchanged.

## Repair Work Prompt

Fix the direct exec tool `host` validation bug from openclaw/openclaw#74426. Current main accepts arbitrary string `host` values, normalizes unknown strings to `null`, and falls back to the configured/default target, so hostname-like values can silently run in the sandbox/default target. Add a narrow validation boundary for direct tool-call args so any supplied non-empty `host` outside `auto|sandbox|gateway|node` fails before command execution with a clear model-visible error. Do not change `tools.exec.host` defaults, sandbox/gateway/node routing rules, exec approvals, elevated behavior, or node binding semantics. Consider adding requested/effective host fields to exec result details/text only after checking model/provider visibility; do not rely on details-only metadata as the sole fix unless verified.
