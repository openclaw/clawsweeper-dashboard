Keep open. Current main still has the Codex native-surface gates that can leave isolated Codex runs without the expected native file/edit surface under node-exec or restricted allowlist policy, and the issue is concrete enough for maintainer review rather than cleanup closure.

## What This Changes

The issue reports a v2026.5.22 regression where isolated Codex cron sessions on a Node.js-hosted gateway lose exec/read/write/edit access after native Code Mode is disabled.

## Best Possible Solution

Maintainers should choose an upgrade-safe Codex tool policy for isolated cron runs: either preserve legacy workspace-tool allowlists with explicit security tests, or provide a clear migration/diagnostic path that avoids silently removing shell and file capabilities.

## Reproduction Assessment

Yes by source inspection: restricted toolsAllow and effective exec host=node both disable Codex native Code Mode while the dynamic tool profile filters the native workspace tools. I did not run the full Linux gateway cron repro in this read-only review.

## Solution Assessment

Unclear. Restoring native Code Mode unconditionally would conflict with node-exec and allowlist safety, so the best fix needs a maintainer decision on compatibility versus explicit migration.
