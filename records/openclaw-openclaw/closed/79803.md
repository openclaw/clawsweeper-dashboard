## Summary

Current main and the shipped v2026.5.28 source now surface terminal WebChat provider errors inline instead of leaving the UI in an active polling state, with focused UI and gateway regression coverage for the error-event path.

## What This Changes

The issue asks WebChat to stop the active loading state and show the failed assistant turn immediately when provider 429 or idle-timeout errors end a chat run before content is produced.

## Best Possible Solution

Keep the current shipped WebChat terminal-error handling and treat any remaining stuck-status reports as separate current-release bugs with fresh logs.

## Reproduction Assessment

No against current main: source inspection shows the terminal error path now appends a visible assistant message and clears the run, with regression tests covering the reported behavior. The original report had strong logs and screenshots, but the current code no longer matches the failing path.

## Solution Assessment

Yes. The maintainable fix is to handle the terminal chat `error` event at the existing WebChat event boundary while the gateway sends a renderable error message; current main does that without adding new config or a parallel reload-only path.
