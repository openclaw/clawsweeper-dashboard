## Summary

Close as a duplicate of the older open media-URI image-tool report; current `main` still shows the same workspace-resolution failure mode, and the Feishu/macOS logs are useful reproduction evidence rather than a distinct remaining work item.

## What This Changes

The issue reports that Feishu inbound image references using `media://inbound/...` are passed to the image tool as workspace-relative paths, producing malformed `media:/inbound` file paths on macOS.

## Best Possible Solution

Use https://github.com/openclaw/openclaw/issues/74123 as the canonical fix target: preserve `media://` refs through image-tool workspace resolution and add regression coverage for a workspaceDir plus inbound-media URI path.

## Reproduction Assessment

Yes from source inspection: with `workspaceDir` set, `media://inbound/...` is not excluded from the image tool's workspace-relative `resolve(...)` branch before `loadWebMedia` can hydrate it. I did not run a live Feishu/macOS flow in this read-only review.

## Solution Assessment

Yes; closing this satellite and routing the fix through the older canonical issue is the best cleanup because the remaining code path and expected behavior are the same.
