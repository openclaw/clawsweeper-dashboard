## Summary

Current `main` already fixes the reported Control UI symptom through the newer grouped chat renderer: user transcript `MediaPath`/`MediaPaths` entries are rendered as images and proxied through `/__openclaw__/assistant-media`, with tests covering the exact history-reload case. The PR's server-side rewrite is therefore redundant on `main` and would also be a less conservative path than the shipped client-side handling.

## Evidence

- **Transcript still stores user uploads as MediaPath fields:** `buildChatSendTranscriptMessage` on current `main` still writes user transcript media into `MediaPath`/`MediaPaths` and `MediaType`/`MediaTypes`, so the relevant question is whether the UI now renders those fields directly.
