## Summary

Keep open. Current main still has the reported Telegram isolated polling path: it drains spooled updates through `handleUpdate()` without first initializing the grammY bot, and the issue is already paired with the focused open fix PR at https://github.com/openclaw/openclaw/pull/81975.

## What This Changes

The issue reports that isolated Telegram polling ingress creates an uninitialized grammY bot, then drains spooled updates via `handleUpdate()`, causing repeated `Bot not initialized` failures and stalled Telegram replies.

## Best Possible Solution

Review and land or otherwise resolve https://github.com/openclaw/openclaw/pull/81975, keeping the fix limited to initializing the grammY bot before isolated spool drain and covering that order with a regression test.

## Reproduction Assessment

Yes from source inspection and the live reports in the issue discussion. Current main still routes default isolated polling through direct spooled `handleUpdate()` calls without an initialization step before drain, and grammY's source throws on that exact uninitialized path.

## Solution Assessment

Yes, initializing the grammY bot before isolated spool drain is the narrow maintainable solution. The open linked PR follows that direction, so this issue should remain open until that PR lands or is rejected.
