## Summary

Close as implemented on main. Current main and shipped v2026.5.2 already provide a documented, default-off ACPX plugin-tools MCP bridge; `tools.allow` and `tools.alsoAllow` are not intended to expose plugin tools to ACP harnesses by themselves.

## What This Changes

Request to make plugin-declared tools available to ACP-backed agents instead of only exposing core OpenClaw tools.

## Best Possible Solution

Keep the shipped default-off ACPX plugin-tools MCP bridge and point affected users to enable it when they intentionally want ACP harnesses to call active OpenClaw plugin tools.

## Reproduction Assessment

Not applicable as a defect. Current source and docs define plugin-tool exposure to ACP harnesses as an explicit MCP bridge, not a normal `tools.allow` or `tools.alsoAllow` path.

## Solution Assessment

Yes. The current opt-in bridge is the narrow maintainable solution because it exposes active plugin tools to ACP through MCP while preserving the documented trust boundary and default-off behavior.
