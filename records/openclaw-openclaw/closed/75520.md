## Summary

Keep open. Current main still has the scoped startup registry and unscoped embedded tool-resolution mismatch described in the report, #75521 remains open with closing syntax for this issue, and the later web-provider cache note also matches current source.

## What This Changes

The issue requests that embedded message plugin-tool resolution reuse the compatible active gateway plugin registry instead of reloading plugins when the startup registry was scoped and the request is unscoped.

## Best Possible Solution

Review #75521 against current main and land or adjust a narrow loader compatibility fix with regression coverage. The final fix should preserve workspace, config, activation metadata, runtime mode, explicit empty scopes, disabled-plugin behavior, and should separately address the confirmed web-provider cache path if it contributes to the per-turn reloads.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: startup creates a scoped gateway-bindable active registry, ordinary embedded message tool resolution does not pass gateway binding, cache keys include the plugin-id scope, and compatibility currently has no scoped-active to unscoped-request reuse branch. The latency magnitude still needs live profiling.

## Solution Assessment

Unclear whether the currently proposed PR is complete because the issue discussion added a web-provider cache bypass that current main still contains.
