Keep open. This is a recent, high-impact, multi-host Gateway starvation report with fresh 2026.4.29 comments saying Telegram replies and per-message CPU spikes are still bad. Current main contains several relevant runtime-deps, Gateway readiness, diagnostics, and Telegram reply-path mitigations, but they are partial/unreleased in this checkout and do not yet prove the broad CPU/probe/reply regression is fixed end to end.

## What This Changes

The issue reports a Gateway high-CPU/event-loop starvation regression that stalls Telegram replies and loopback status probes, with suspected plugin runtime-deps, subagent registry, session recovery, or heartbeat hot paths.

## Best Possible Solution

Keep this issue as the canonical broad Gateway CPU/event-loop starvation tracker until maintainers can retest current main or the next packaged release on affected hosts and verify idle CPU, responsive loopback probes, and at least one Telegram/WebChat turn without pre-model Gateway stalls. Narrow root-cause fixes can continue under related issues such as #73532, #68944, #75214, and #75436/#75439, but this broad user-visible symptom should not close until the full path is proven.
