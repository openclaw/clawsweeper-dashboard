## Summary

Current main already addresses the reported cron run-session bloat: cron run session keys matching `agent:<id>:cron:<job>:run:<run>` are recognized, pruned from `sessions.json` by the cron session reaper after `cron.sessionRetention` (default `24h`), hidden from session listing, and skipped by restart recovery so they are not treated as main sessions to resume. This is fixed on current main only; no release tag contains the proof commit yet.

## Best Possible Solution

Close this issue as implemented on current main. Keep the retention-based cron reaper and restart-recovery safeguards; if maintainers later want zero-retention cleanup, that should be a separate product decision in the cron finalizer/config path rather than an `agent_end` hook workaround.
