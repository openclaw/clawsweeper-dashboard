## Summary

Current `main` already contains a targeted fix for this bug: node-targeted exec approvals now bind the exact `systemRunPlan` during approval, and the post-approval path re-invokes `node.invoke` with `command: "system.run"` on the selected node instead of falling back to gateway-host execution.

## Evidence

- **Node approval requests now require and persist node execution binding:** `exec.approval.request` rejects `host=node` requests unless `nodeId`, `systemRunPlan`, and `commandArgv` are present, then stores both `systemRunBinding` and `systemRunPlan` on the pending approval request.
  - file: [src/gateway/server-methods/exec-approval.ts:164](https://github.com/openclaw/openclaw/blob/6a61cb73c5622749d7940fd8d8b90c93adbb5081/src/gateway/server-methods/exec-approval.ts#L164)
  - command: `nl -ba src/gateway/server-methods/exec-approval.ts | sed -n '164,248p'`
  - sha: [6a61cb73c562](https://github.com/openclaw/openclaw/commit/6a61cb73c5622749d7940fd8d8b90c93adbb5081)
- **Post-approval node exec stays on `node.invoke`:** The node-host exec path builds `node.invoke` params for `system.run`, including the prepared `systemRunPlan`, and uses that path both for the async follow-up after approval and the inline approved path.
  - file: [src/agents/bash-tools.exec-host-node.ts:207](https://github.com/openclaw/openclaw/blob/6a61cb73c5622749d7940fd8d8b90c93adbb5081/src/agents/bash-tools.exec-host-node.ts#L207)
