## Summary

Keep open: current `main` still has the ordinary heartbeat reply fallback that can deliver raw heartbeat text, while the structured fix is in related PRs and a release-branch merge rather than the checked-out main branch.

## What This Changes

Heartbeat delivery should suppress or sanitize Codex tool/XML scaffolding and only send structured user-visible heartbeat text or valid media.

## Best Possible Solution

Carry the structured `heartbeat_respond` delivery guard or equivalent sanitizer fix onto `main`, keep regression coverage for protocol-only text, mixed visible text, `HEARTBEAT_OK`, and media/internal failure notices, then close with main-branch proof.

## Reproduction Assessment

Yes. Source inspection on current main shows a high-confidence reproduction path: mock a Codex heartbeat-tool-mode run where `getReplyFromConfig` returns ordinary `<tool_calls>` or `<file_contents>` text and the runner can still deliver `normalized.text` through `sendDurableMessageBatch`.

## Solution Assessment

No, current main is not sufficient. The safer fix is the release-branch direction: require structured `heartbeat_respond` delivery in heartbeat-tool mode and avoid treating ordinary assistant text/media as deliverable heartbeat output, with legacy sanitizer coverage where fallback text is still allowed.
