## Summary

Close as implemented: current main and shipped releases already use per-agent session stores by default and support `{agentId}` templating in global `session.store`, with docs and regression coverage for that supported route.

## What This Changes

Request support for per-agent session store paths, either via `agents.list[]` per-agent config or a `{agentId}` template in global `session.store`.

## Best Possible Solution

Keep the existing global `session.store` `{agentId}` contract and document usage for custom Windows paths; avoid adding a second `agents.list[].sessionStore` source of truth.

## Reproduction Assessment

Not applicable: this is a feature/config request rather than a reported failing behavior. Source and tests show the requested template route resolves to agent-specific store paths on current main.

## Solution Assessment

Yes: the existing templated global store is the narrowest maintainable solution. A new per-agent key would add config precedence and migration complexity without solving a missing capability.
