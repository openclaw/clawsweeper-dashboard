## Summary

Keep open for changes: the active-session scenario is useful and the PR includes real terminal proof, but the implementation changes the documented `maxDiskBytes` contract so referenced trajectory sidecars can make the sessions directory exceed the configured cap without being counted.

## What This Changes

The PR changes session disk-budget accounting to ignore referenced trajectory runtime/pointer sidecars and adds a regression test for active trajectory sidecars exceeding `maxDiskBytes`.

## Best Possible Solution

Keep `session.maintenance.maxDiskBytes` as a physical sessions-directory cap, and handle active trajectory pressure with a narrower, documented mechanism that still bounds trajectory storage.

## Reproduction Assessment

Yes. Source inspection shows current main counts referenced trajectory sidecars in the sessions-directory total, and the PR regression fixture demonstrates that large active sidecars can push the directory over `maxDiskBytes`; I did not execute tests because this review is read-only.

## Solution Assessment

No. The PR solves the active-session case by excluding all referenced trajectory sidecars from budget accounting, but the safer path preserves the documented physical disk cap or adds an explicit separate trajectory budget with docs and tests.
