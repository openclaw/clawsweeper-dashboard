## Summary

Keep open: the fallback direction and proof are solid, but current head still bypasses the 0700 private-store directory chmod that current main gets from fs-safe, so the exec approvals store loses an owner-only directory guarantee.

## What This Changes

The PR replaces exec approvals persistence with a temp-file writer that falls back to a guarded copy on Windows EPERM/EEXIST, adds regression tests, and updates the changelog.

## Best Possible Solution

Keep the guarded Windows fallback, but restore the private directory mode contract and add a regression asserting `.openclaw` is normalized to 0700 before merge.

## Reproduction Assessment

Yes, source-level: current main reaches `exec-approvals.json` persistence through the fs-safe private-store rename path, and #77785 gives concrete Windows EPERM failures across v2026.5.2, v2026.5.3-1, and v2026.5.4. I did not run a native Windows locked-handle gateway repro in this read-only pass.

## Solution Assessment

No, not yet. The fallback now covers the important copy-failure, symlink-swap, hard-link, temp cleanup, and final file-mode cases, but it should also preserve the private-store directory chmod/validation before merge.
