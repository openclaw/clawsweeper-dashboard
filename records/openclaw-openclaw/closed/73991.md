Full review comments:

- **[P1] Keep command lanes server-controlled:** `src/gateway/server-methods/agent.ts:1083-1084`
  - body: This prefers `request.lane` before the configured lane, but gateway agent requests accept any lane string and the queue creates an independent lane for each new name. A caller can bypass the new per-agent/default lane policy and its caps; resolve the lane from server config for ingress, or enforce a strict server-owned allowlist for any explicit override.
  - confidence: 0.93
- **[P2] Do not inherit caps across different lane ids:** `src/config/agent-command-lanes.ts:54`
  - body: When an agent sets a different `commandLane.id` but omits `maxConcurrent`, this falls back to the default lane's cap and installs that cap on the unrelated per-agent lane. Only inherit `defaults.maxConcurrent` when the agent is still using the default lane id, or require an explicit cap for the override lane.
  - confidence: 0.9
- **[P3] Apply defaults when exposing resolved command lanes:** `src/agents/agent-scope-config.ts:125`
  - body: `ResolvedAgentConfig` is expected to carry inherited defaults for adjacent fields, but this stores only the per-agent `commandLane`. Any consumer of the resolved config will miss `agents.defaults.commandLane`, so this should use the same defaults fallback as the runtime resolver.
  - confidence: 0.78
- **[P3] Remove the release-owned changelog edit:** `CHANGELOG.md:16`
  - body: `CHANGELOG.md` is release-generated in this repo; normal PRs should keep release-note context in the PR body or squash message. Please drop this direct changelog entry so release generation owns the file.
  - confidence: 0.86
