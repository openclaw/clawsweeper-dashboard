## Summary

Keep open: the linked stale Control UI busy-state bug is real, but this patch is still too narrow because it can hide Stop while a local run remains active with `chatRunId` set and `stream` null. The external PR also still has only test/mock proof, not real Control UI browser/gateway evidence.

## What This Changes

The PR gates the Control UI chat `In progress` badge and Stop button on `sending || stream !== null` and adds render tests for stale abort state.

## Best Possible Solution

Split stale session-list abortability from visible local active-run state with a guard that includes the current local run id, then add real Control UI proof for both completed idle and streamless active-run states.

## Reproduction Assessment

Yes at source level: current code uses `chatRunId` as abortability and has tested states where `chatRunId` remains set while `chatStream` is null. I did not run a live browser/gateway reproduction in this read-only review.

## Solution Assessment

No. `sending || stream !== null` fixes the stale completed-response symptom too narrowly because it drops the existing local active-run signal; the better fix distinguishes stale session-list-only rows from local `chatRunId` activity.
