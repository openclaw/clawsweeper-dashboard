## Summary

Current main implements the core request: typed plugin hooks accept per-registration timeoutMs, the hook runner races handlers against that budget and skips/logs timed-out fail-open handlers, prompt-context hooks have a default timeout, before_tool_call remains fail-closed, and Active Memory has plugin config/docs/tests for a bounded recall timeout. The fix is on main after v2026.4.26 and does not appear to be in a shipped release yet.

## Best Possible Solution

Keep the main-branch implementation: per-hook timeoutMs on api.on, default timeout/skip behavior for prompt-context hooks, fail-closed policy for before_tool_call, and Active Memory's plugin-owned timeout config. If maintainers want the exact JSON fallback/critical schema later, track that as a narrower API-design follow-up rather than keeping this broader issue open.
