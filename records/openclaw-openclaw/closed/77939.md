## Summary

Keep open: current main still only suppresses empty or same-PID dual-stack loopback Gateway listeners, so a valid LAN/wildcard Gateway listener still reaches status and doctor busy-port diagnostics.

## What This Changes

Request status and doctor to treat a healthy managed OpenClaw Gateway listener on LAN/wildcard bind as expected instead of a port conflict.

## Best Possible Solution

Add a shared expected-Gateway-listener predicate used by port hints, status-all, and doctor that accepts one managed Gateway listener on loopback/LAN/wildcard plus the existing same-PID dual-stack loopback case, while preserving warnings for real conflicts.

## Reproduction Assessment

Yes from source inspection, though not by running a live gateway here. A single `openclaw-gateway` listener on `0.0.0.0:18789` or `*:18789` is not accepted by the current dual-stack-loopback-only predicate and therefore reaches the warn/diagnostic path.

## Solution Assessment

Yes. Extending the existing listener classification seam is the narrowest maintainable fix, provided tests keep mixed owners, SSH tunnels, unknown processes, and multiple Gateway PIDs warning.

## Repair Work Prompt

Fix the false Gateway port conflict for healthy managed LAN/wildcard Gateway listeners. In current main, a single OpenClaw Gateway listener on `0.0.0.0:<port>`, `[::]:<port>`, or `*:<port>` is valid for `gateway.bind=lan`/wildcard deployments but `openclaw status --all` and doctor still report it as a busy-port conflict. Add a shared expected-Gateway-listener predicate in the port diagnostics layer, export it through `src/infra/ports.ts`, and use it in `buildPortHints`, `src/commands/status-all/diagnosis.ts`, and `src/commands/doctor-gateway-daemon-flow.ts`. Preserve warnings for unknown owners, SSH tunnels, mixed Gateway + non-Gateway listeners, multiple unrelated listeners, and multiple Gateway PIDs. Add focused regression tests for single `0.0.0.0`, `[::]`/`::`, and `*` Gateway listeners, existing dual-stack loopback, multiple Gateway PIDs, SSH, unknown, and doctor suppression. Include a single-line user-facing `CHANGELOG.md` fix entry. Do not broaden Gateway bind policy, auth defaults, or startup port availability semantics.
