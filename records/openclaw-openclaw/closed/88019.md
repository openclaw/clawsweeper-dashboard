## Summary

Keep open. Current main has replay sanitization for OpenAI Responses-family sessions, but source inspection shows the model-switch path can still drop a replayable `reasoning` item while leaving the paired assistant `msg_*` id for the next Azure Responses request; the broader related report at https://github.com/openclaw/openclaw/issues/84484 does not replace this more specific fallback/session-poisoning repro.

## What This Changes

The issue reports that an Azure Responses GPT-5.5 dashboard session can become stuck after fallback because replay keeps a `msg_*` item id without the required paired `reasoning` item.

## Best Possible Solution

Update the Responses replay sanitizer or transport conversion so `reasoning` items and their paired `msg_*` assistant message ids are preserved or stripped together after model/fallback switches, with focused coverage for Azure/OpenAI Responses replay.

## Reproduction Assessment

Yes. A high-confidence source repro is to sanitize a prior Responses assistant turn with reasoning plus text after a model snapshot change, then build the next Responses payload and observe that reasoning is dropped while the assistant `msg_*` id remains.

## Solution Assessment

Yes. The narrow maintainable fix is to repair the existing replay sanitizer/transport pairing logic, not add a config flag or require manual session cleanup.

## Repair Work Prompt

Fix the Azure/OpenAI Responses replay poisoning reported in https://github.com/openclaw/openclaw/issues/88019. The current sanitizer drops replayable OpenAI Responses reasoning after a model snapshot change, but the transport can still replay the paired assistant text `msg_*` id, causing Azure to reject the next turn. Keep the fix inside the existing Responses replay sanitizer/transport boundary: either preserve reasoning and message ids together when safe, or strip the `msg_*` text id whenever the paired `rs_*` reasoning item is removed. Cover Azure/OpenAI Responses, same-model resume after fallback, and sibling OpenAI Responses paths; do not add a user-facing config option and do not edit CHANGELOG.md. Coordinate expectations with https://github.com/openclaw/openclaw/issues/84484 but keep the PR focused on this replay pairing bug.
