## Summary

Keep open: the PR has the right core handoff direction and supplied real managed-service proof, but the current head still has a blocking status-cache regression in the final restart-sentinel retry path.

## What This Changes

The PR reroutes supervised package-managed Gateway `update.run` calls through a detached CLI managed-service handoff and updates restart sentinels, tests, and updater docs.

## Best Possible Solution

Land the detached managed-service handoff after recording the final update sentinel into the `update.status` cache before consuming it, with a focused regression test for the pending-to-final retry path.

## Reproduction Assessment

Yes. Source inspection shows current main sends supervised global `update.run` through `runGatewayUpdate` in the live Gateway process, and the linked issue plus PR proof describe the LaunchAgent-backed failure mode; I did not run the destructive self-update locally.

## Solution Assessment

Mostly yes, but not complete. Reusing the existing CLI managed-service updater is the narrow maintainable fix, while the final status sentinel still needs to refresh the cached `update.status` value before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Refresh final update status before consuming it:** `src/gateway/server-restart-sentinel.ts:517-529`
  - body: When a managed-service handoff is pending, startup can cache the pending sentinel, then this retry path later re-reads the final `ok` or `error` sentinel after the CLI finishes. The final path removes the sentinel without recording that payload, while `update.status` only returns the in-memory cache, so callers can keep seeing `restart-health-pending` after health succeeds. Record or refresh the final update payload before removing it.
  - confidence: 0.92
