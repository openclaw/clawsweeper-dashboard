## Summary

Keep open: current main still treats config env values as literal strings, and an open PR with closing syntax specifically tracks the requested `file:` env support, so this should not be cleanup-closed before maintainers resolve the config-secret contract.

## What This Changes

Support `file:` references in `openclaw.json` env values so provider API keys can be loaded from files instead of passed literally.

## Best Possible Solution

Maintainers should decide whether `file:` env values are a supported config contract; if approved, implement them through existing SecretRef/fs-safe guardrails with docs and tests, otherwise document the rejection and steer users to SecretRefs or process env.

## Reproduction Assessment

Yes, at source level for the reported literal-value behavior: current `applyConfigEnvVars` copies config env strings literally after existing-env and `${VAR}` checks. I did not run a live xAI request because the config path itself establishes the literal value reaching the env map.

## Solution Assessment

Unclear: the requested shorthand may solve the user workflow, but it is not the current documented contract. The maintainable solution is either to reuse SecretRef-grade file guardrails for the shorthand or explicitly reject it in docs.
