## Summary

Current main already resolves the reported TUI/gateway-client scope failure. The TUI identifies as `openclaw-tui` in UI mode, current gateway authorization treats `operator.admin` as sufficient for read/admin RPCs, the Control UI/gateway-client scope bundle includes `operator.read` and `operator.write`, and regression coverage verifies `operator.read` reconnects when the paired baseline is `operator.admin`. The v2026.4.5 changelog records the device-auth scope preservation fix tied to PR #46032.

## Evidence

- **Issue discussion narrowed repro:** The issue body reported a connected UI/TUI-style session whose subsequent `status`, `system-presence`, and `config.get` RPCs failed with `missing scope: operator.read`; the follow-up comment at https://github.com/openclaw/openclaw/issues/50606#issuecomment-4099667983 confirmed the strongest repro as `connect => ok` followed by `status => missing scope: operator.read`.
- **TUI uses UI client identity:** `GatewayChatClient` constructs the terminal TUI gateway client with `clientName: GATEWAY_CLIENT_NAMES.TUI`, display name `openclaw-tui`, and `mode: GATEWAY_CLIENT_MODES.UI`, matching the surface described in the report.
