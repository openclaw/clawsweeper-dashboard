## Summary

Keep open: the issue has two live confirmations on recent releases, current main still shows the Telegram turn carrying the route-derived per-peer key without a later landed fix for durable direct-session indexing, and no canonical fixing PR was found.

## What This Changes

Telegram direct messages with `session.dmScope: "per-channel-peer"` route and reply, but the per-peer `telegram:direct` session entry and history are not recoverable from the session store.

## Best Possible Solution

Add a focused regression that drives a Telegram direct DM through the production dispatch path, then make durable session recording and transcript lookup consistently reuse the route-derived per-peer key.

## Reproduction Assessment

Yes. Send consecutive Telegram DMs with `session.dmScope: "per-channel-peer"`, then inspect `sessions.json`, `openclaw sessions --json`, and the next turn's history count; I did not run the live Telegram path here, so this is source-reproducible rather than locally reproduced.

## Solution Assessment

Yes. The narrow maintainable fix is to align the existing route-derived per-peer key with durable session-store and transcript lookup behavior, not to add a new config mode or Telegram-only workaround.
