## Summary

Keep open. The feature may be useful, but the current PR applies array coercion to every string config path, can break existing string-only fields, can lose env-reference preservation on config write-back, and still lacks real behavior proof.

## What This Changes

The PR adds JSON string-array coercion for whole-value `${VAR}` config strings plus unit tests for positive and boundary cases.

## Best Possible Solution

Adopt an explicit or schema-aware list-env contract that only applies to list-valued config paths, preserves authored env references through config writes, and documents the accepted behavior.

## Reproduction Assessment

Yes for the review findings. Source inspection shows the new generic string branch runs before validation, and the current write-back restoration path cannot recover an authored string env reference after it becomes an array.

## Solution Assessment

No. Whole-value `${VAR}` is a plausible direction, but this patch needs schema/path gating or an explicit list syntax plus write-back preservation before it is maintainable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Gate array coercion to list-typed config paths:** `src/config/env-substitution.ts:200-203`
  - body: `SINGLE_VAR_REFERENCE.test(value)` runs for every config string, so a string-only field such as `agents.defaults.systemPromptOverride: '${PROMPT}'` becomes an array when `PROMPT` contains JSON array text and then fails validation. Gate this on list-typed schema/path metadata or use an explicit list syntax.
  - confidence: 0.86
- **[P2] Preserve env refs after array coercion:** `src/config/env-substitution.ts:203`
  - body: Returning `asArray` changes the resolved value from a string to an array before write-back. `restoreEnvVarRefs` only restores same-type string/string or array/array leaves, so config writes can persist the resolved array and lose the authored env reference.
  - confidence: 0.91
