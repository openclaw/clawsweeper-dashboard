## Summary

Keep this open: current main and the latest release still omit Feishu's documented full-card-content query parameter on both single-message and thread-history reads, so the reported `[Interactive Card]` placeholder remains a real Feishu plugin bug.

## What This Changes

The issue requests that Feishu message reads and thread-history reads return actual interactive-card text instead of the `[Interactive Card]` placeholder.

## Best Possible Solution

Add `card_msg_content_type: "user_card_content"` to both Feishu read API calls, keep existing parser fallbacks, and add focused tests for single-message and thread-history reads.

## Reproduction Assessment

Yes, source-reproducible. Current main routes `message(action=read)` through `getMessageFeishu()`, both read paths omit the documented card-content query parameter, and the parser falls back to `[Interactive Card]` when the API response lacks full card elements.

## Solution Assessment

Yes, the narrowest maintainable fix is in the Feishu plugin API calls, not core or config. The documented query parameter gives the existing parser full card JSON while preserving fallback behavior for unsupported payloads.
