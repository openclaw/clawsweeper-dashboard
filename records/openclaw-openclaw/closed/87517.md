## Label Justifications

- `P1`: A real silent gateway token rotation would break authenticated gateway consumers and expose an urgent auth/connectivity regression.
- `impact:security`: The report concerns unexpected mutation of a shared gateway credential used for access control.
- `impact:auth-provider`: The affected surface is gateway authentication token selection, persistence, SecretRef resolution, and consumer connectivity.

## Summary

Close as implemented: current main and shipped releases now guard `doctor` gateway-token generation behind true missing-token detection, resolve supported SecretRefs before warning, and refuse to overwrite SecretRef-managed `gateway.auth.token` with plaintext. The original audit hashes do not identify the changed config path, but the reported `gateway.auth.token` writer is now fixed.

## What This Changes

The issue reports that interactive `openclaw doctor --fix` can rotate an already valid `gateway.auth.token`, disconnecting existing gateway consumers.

## Best Possible Solution

Keep the shipped SecretRef-aware doctor guard; ask for a new report only if `gateway.auth.token` still changes on v2026.5.28 or newer with redacted changed-path or before/after config proof.

## Reproduction Assessment

Yes for the historical source path: the merged fix shows doctor previously treated resolving SecretRef-backed gateway tokens as unavailable. Current main no longer has a failing path by source inspection, and this read-only review did not run a live reproduction.

## Solution Assessment

Yes. The shipped resolver-based guard is better than a token-length early return because it covers plaintext tokens, env tokens, SecretRefs, password mode, none mode, and trusted-proxy mode in both interactive doctor and structured lint paths.
