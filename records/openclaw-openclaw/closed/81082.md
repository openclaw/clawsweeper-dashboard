## Summary

Current main fixes the reported false-positive at the shared Gateway event-loop health source, so `channels status` no longer marks utilization/CPU-only idle samples as degraded when delay is zero. The fix is current-main-only; no release tag contains the proof commit yet.

## What This Changes

The issue asks OpenClaw to stop `openclaw channels status` from reporting a persistent idle `Gateway event loop degraded` warning when `eventLoopDelayMaxMs=0` but utilization and CPU counters read saturated.

## Best Possible Solution

Keep the shared Gateway event-loop health classifier as the source of truth and let the next stable release carry the current-main fix to packaged installs.

## Reproduction Assessment

No current-main failure path remains from source inspection: the exact reported metric shape with zero delay and saturated utilization/CPU now yields no degradation reasons. I did not run a live idle container because this review was read-only, but the current regression tests cover the reported values.

## Solution Assessment

Yes. Fixing the shared Gateway health classifier is the narrow maintainable solution because `channels status`, readiness, and health consumers all receive the same filtered event-loop snapshot instead of adding formatter-side special cases.
