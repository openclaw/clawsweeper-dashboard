## Summary

The Discord v2026.4.29 CPU/event-loop report maps to the same active stuck-session recovery gap already tracked by open #71127, with #73243 open as the active implementation candidate; current main still warns and skips active embedded runs by default rather than providing a public abort/recovery threshold.

## What This Changes

The issue reports that after updating to v2026.4.29, Discord sessions can stay in `processing`, causing continuous CPU use, event-loop delays, Discord inbound timeouts, and repeated stuck-session recovery warnings.

## Best Possible Solution

Close this duplicate and keep implementation focused on #71127 and #73243, or a corrected replacement PR, adding a conservative configurable active stuck-session abort/recovery path with telemetry, docs, schema, and regression coverage.

## Reproduction Assessment
