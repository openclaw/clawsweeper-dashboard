## Summary

Current `main` already addresses the reported restart-induced Claude CLI session resets by introducing a stable `mcpResumeHash` path that canonicalizes the OpenClaw loopback URL, preferring that stable hash for session reuse, and covering the exact loopback-port churn case with regression tests.

## Evidence

- **Stable resume hash canonicalizes loopback URL:** `prepareModeSpecificBundleMcpConfig` now computes both the raw `mcpConfigHash` and a separate `mcpResumeHash`; the resume hash canonicalizes the `openclaw` MCP server URL so `http://127.0.0.1:<ephemeral>/mcp` does not change resume identity across gateway restarts.
  - file: [src/agents/cli-runner/bundle-mcp.ts:287](https://github.com/openclaw/openclaw/blob/11804a484ded89374afa55020813cfa265aa143a/src/agents/cli-runner/bundle-mcp.ts#L287)
  - command: `nl -ba src/agents/cli-runner/bundle-mcp.ts | sed -n '287,329p'`
  - sha: [11804a484ded](https://github.com/openclaw/openclaw/commit/11804a484ded89374afa55020813cfa265aa143a)
- **Session reuse prefers stable hash:** `resolveCliSessionReuse` now checks stored/current `mcpResumeHash` first and only falls back to legacy `mcpConfigHash` when the resume hash is absent, which prevents the loopback port churn from invalidating otherwise reusable Claude CLI sessions.
  - file: [src/agents/cli-session.ts:162](https://github.com/openclaw/openclaw/blob/11804a484ded89374afa55020813cfa265aa143a/src/agents/cli-session.ts#L162)

  - sha: [11804a484ded](https://github.com/openclaw/openclaw/commit/11804a484ded89374afa55020813cfa265aa143a)
- **Bindings persist resume hash on new runs:** New CLI session bindings written by the runner include `mcpResumeHash` alongside the legacy `mcpConfigHash`, so sessions created on current `main` use the stable resume identity going forward.
  - file: [src/agents/cli-runner.ts:274](https://github.com/openclaw/openclaw/blob/11804a484ded89374afa55020813cfa265aa143a/src/agents/cli-runner.ts#L274)
  - command: `nl -ba src/agents/cli-runner.ts | sed -n '262,279p'`
  - sha: [11804a484ded](https://github.com/openclaw/openclaw/commit/11804a484ded89374afa55020813cfa265aa143a)
- **Regression test covers loopback-port restart case:** The bundle-MCP tests explicitly assert that changing only the OpenClaw loopback port changes `mcpConfigHash` but keeps `mcpResumeHash` stable, and that a real semantic MCP change still invalidates the resume hash.
  - file: [src/agents/cli-runner/bundle-mcp.test.ts:193](https://github.com/openclaw/openclaw/blob/11804a484ded89374afa55020813cfa265aa143a/src/agents/cli-runner/bundle-mcp.test.ts#L193)

  - file: [src/agents/cli-session.test.ts:207](https://github.com/openclaw/openclaw/blob/11804a484ded89374afa55020813cfa265aa143a/src/agents/cli-session.test.ts#L207)

## Risks / Open Questions

- Users on released builds before a newer post-`v2026.4.23` release may still hit the bug, since the fix is confirmed on current `main` rather than a tagged release from this checkout.
- Legacy stored bindings created before `mcpResumeHash` existed can still invalidate once via the old `mcpConfigHash` fallback before being rewritten on a newer build.
