## Label Justifications

- `P1`: The report describes a currently open Telegram polling failure mode that can silently block real inbound channel workflow after container restart.
- `impact:message-loss`: Inbound Telegram updates can remain queued on disk instead of reaching the agent loop.
- `impact:session-state`: A stale `.processing` claim can leave spool/session recovery state inconsistent after gateway recreate.

## Summary

This should close as a duplicate: current main still has the reported Telegram spool PID-liveness behavior, but the same remaining bug is already tracked by the canonical issue and an open fix PR is specifically changing current-PID/different-process-id claims to recover after restart.

## What This Changes

The issue reports that Telegram polling in PID-namespaced containers can leave stale `.processing` spool claims owned by old PID 1 instances, blocking inbound message draining until the stale window expires.

## Best Possible Solution

Keep the duplicate closed and land or update https://github.com/openclaw/openclaw/pull/84690 through the canonical tracker at https://github.com/openclaw/openclaw/issues/84674, preserving this report’s PID-namespace evidence by link.

Yes. Source inspection and existing tests show that a `.json.processing` claim with a different process identity and a PID that still exists is treated as live-owned; a live Docker/Telegram replay was not run in this read-only review.

## Solution Assessment

Yes for triage. The maintainable path is to handle this through the existing canonical issue and open PR, which already targets the current-PID/different-process-id recovery condition with focused tests.
