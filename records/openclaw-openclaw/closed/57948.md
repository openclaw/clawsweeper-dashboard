## Summary

Current `main` already normalizes both Telegram sender IDs and `allowFrom` entries to strings before matching, and the reported string-vs-number mismatch is covered by existing Telegram tests.

## Evidence

- **DM sender IDs are stringified before allowlist matching:** `resolveTelegramSenderIdentity` converts `msg.from.id` with `String(from.id)` and passes that string `candidateId` into DM allowlist matching.

- **Configured allowFrom entries are normalized to strings:** `normalizeAllowFrom` maps every configured entry through `String(value)` and stores normalized string IDs in `entries` before `resolveSenderAllowMatch` compares them.
  - file: [extensions/telegram/src/bot-access.ts:43](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/extensions/telegram/src/bot-access.ts#L43)
