## Label Justifications

- `P1`: The report describes deterministic loss of persisted tool config and OAuth keyring state for containerized deployments on the latest release.
- `impact:data-loss`: Stripping inherited XDG config roots can make subprocess tools write credentials and config outside the persisted directory, losing them on restart.
- `impact:security`: The affected code is the host exec environment sanitizer, and any fix changes the trusted-inherited-env versus untrusted-override security boundary.

## Summary

Keep open: current main and the latest release still classify inherited `XDG_CONFIG_HOME` and `XDG_CONFIG_DIRS` as blocked inherited host-exec env, so agent-spawned subprocesses lose the operator-selected XDG config roots. The report is supported by current source, tests, docs, and downstream tool path contracts.

## What This Changes

The requested fix is to preserve operator-set XDG Base Directory env vars for inherited subprocess environments while still blocking agent-supplied overrides.

## Best Possible Solution

Add `XDG_CONFIG_HOME` and `XDG_CONFIG_DIRS` to the trusted inherited exception policy, keep request-scoped overrides blocked, regenerate Swift parity, and add regression coverage for both inheritance and override paths.

## Reproduction Assessment

Yes, at source level: current policy derives both XDG config vars into `blockedInheritedKeys`, and the host exec path passes `process.env` through that sanitizer before spawning commands. I did not run a live gateway subprocess in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is a policy-level inherited allowlist change plus parity/regression tests. A new config knob or downstream-tool workaround would leave the host exec inheritance contract inconsistent.
