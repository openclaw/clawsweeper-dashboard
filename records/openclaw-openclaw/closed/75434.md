## Summary

Current main and the shipped v2026.4.29 tag already carry the Feishu and shared session-path behavior needed for this report: Feishu dispatch keeps the resolved non-default agent on the route/session/store path, and the shared reply runner passes that agent/store context into transcript path resolution.

## What This Changes

The issue reports that Feishu multi-agent routing finds the correct non-default agent but builds or resolves the session under the default `main` agent instead.

## Best Possible Solution

Keep the shipped per-agent session routing implementation and direct affected users on older 2026.3.x builds to upgrade to v2026.4.29 or newer; a new report should include logs from that version if Feishu still writes a non-main account under `main`.

## Reproduction Assessment

Yes. The historical reproduction is a multi-account Feishu config with a binding to a non-main agent, then a DM to that account; current source inspection verifies that exact path now carries the routed agent into the session key, store path, and transcript resolver.

## Solution Assessment

Yes. The current fix is the narrow maintainable path because Feishu uses the shared routed channel-turn pipeline and the shared session resolver accepts agent/store context, instead of adding a Feishu-only transcript-path workaround.
