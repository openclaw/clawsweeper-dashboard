## Summary

Keep open: current main resolves `openclaw.runtimeExtensions`, but source inspection still shows configured non-bundled channel plugins are excluded from gateway startup unless they have an explicit enable marker, which matches the reported silent Discord absence after a manual npm install/update path.

## What This Changes

Ensure an installed `@openclaw/discord` package with compiled `dist/` runtime metadata is discovered and loaded by the gateway when Discord channel config is present.

## Best Possible Solution

Add a maintainer-approved official installed channel path that either persists explicit enablement during update/repair or emits an actionable diagnostic instead of silently skipping Discord.

## Reproduction Assessment

Yes from source inspection: a configured non-bundled channel plugin without explicit enablement is filtered out of gateway startup even though runtime entry discovery succeeds. I did not run a live gateway install because this review is read-only.

## Solution Assessment

Unclear: simply treating auto-enabled non-bundled channels as trusted would contradict current tests and weaken the plugin trust model. A narrower official-package recovery or diagnostic path is safer, but maintainers need to choose that policy.
