## Summary

Close as implemented on current main. The reported v2026.4.29 failure mode is the per-turn plugin runtime reload/runtime-deps hot path that the issue itself ties to #75512; current main now keeps gateway plugin runtime startup gateway-bindable, reuses compatible active registries for per-turn tool construction, and keeps bundled runtime-deps repair out of that hot path. The local node-llama-cpp setup looks like an amplifier of the same blocked event-loop symptom, not a separate unresolved source-level bug in this report.

## What This Changes

The issue asks to fix a v2026.4.29 regression where per-turn plugin re-evaluation plus local memory embeddings saturates the gateway event loop and causes Discord and Telegram transport timeouts.

## Best Possible Solution

Keep the current main registry-reuse/runtime-deps hot-path fix and ship it in the next release. Any remaining delay after this change should be narrowed to a fresh local-embeddings CPU/backpressure report with current-main traces that no longer show per-turn plugin reloads.

## Reproduction Assessment

Yes. The issue provides a high-confidence recognition path for the v2026.4.29 regression: /readyz event-loop delay/utilization, embedded-run prep stages dominated by core-plugin-tools/system-prompt, and repeated plugin loading per turn. I did not live-reproduce the WSL2 GGUF setup, but current source and tests directly cover the hot path those traces identify.

## Solution Assessment

Yes. Reusing the gateway startup plugin registry and keeping runtime-deps repair out of request-time tool construction is the narrow maintainable fix for the reported plugin re-evaluation path, and it matches the plugin cache guidance without adding a broad persistent discovery cache.
