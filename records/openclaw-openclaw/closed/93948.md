## Summary

Keep open. The PR targets a real gateway diagnostic bug, but the probe-side fix is too narrow because server-side connect rejections still set `connectError` before close and therefore remain classified as unreachable.

## What This Changes

The PR changes gateway probe and doctor health/daemon diagnostics so accepted-then-closed gateway handshakes render a specific close reason instead of unreachable/not running.

## Best Possible Solution

Track accepted/rejected gateway handshakes independently of the formatted connect error, add focused probe and doctor tests for `GatewayClientRequestError` connect rejection, and include real terminal proof for the corrected doctor/probe output.

## Reproduction Assessment

Yes. Source inspection shows the server can reject the connect request after WebSocket acceptance, the client then fires `onConnectError`, and the PR guard keeps `connectLatencyMs` null for that reachable-but-rejected path.

## Solution Assessment

No. The doctor rendering change is useful, but the probe fix should use a transport-open or request-error signal rather than `connectError == null` before it can be the best fix.
