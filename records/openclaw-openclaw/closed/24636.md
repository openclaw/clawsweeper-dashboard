## Summary

Current `main` already implements per-run cron usage tracking: `cron.runs` entries include `sessionId`/`sessionKey` plus a `usage` object with token counters, and isolated cron runs populate those fields before the gateway persists them.

## Evidence

- **`cron.runs` schema includes usage and session linkage:** The gateway protocol exposes `sessionId`, `sessionKey`, and a `usage` object on each cron run log entry, with `input_tokens`, `output_tokens`, `total_tokens`, `cache_read_tokens`, and `cache_write_tokens`.
  - file: [src/gateway/protocol/schema/cron.ts:356](https://github.com/openclaw/openclaw/blob/d3f6783b16ecc10c67b101bf019ca4294453206a/src/gateway/protocol/schema/cron.ts#L356)
  - sha: [d3f6783b16ec](https://github.com/openclaw/openclaw/commit/d3f6783b16ecc10c67b101bf019ca4294453206a)
- **Cron runs populate usage telemetry:** Isolated cron execution captures model usage into telemetry, including input/output totals and derived total tokens, then returns that telemetry with the run result.
  - file: [src/cron/isolated-agent/run.ts:752](https://github.com/openclaw/openclaw/blob/d3f6783b16ecc10c67b101bf019ca4294453206a/src/cron/isolated-agent/run.ts#L752)
  - sha: [d3f6783b16ec](https://github.com/openclaw/openclaw/commit/d3f6783b16ecc10c67b101bf019ca4294453206a)
- **Gateway persists usage into `cron.runs` history:** When a cron job finishes, the gateway appends `sessionId`, `sessionKey`, model/provider, and `usage` into the per-job run log that backs `cron.runs`.
  - file: [src/gateway/server-cron.ts:553](https://github.com/openclaw/openclaw/blob/d3f6783b16ecc10c67b101bf019ca4294453206a/src/gateway/server-cron.ts#L553)
  - sha: [d3f6783b16ec](https://github.com/openclaw/openclaw/commit/d3f6783b16ecc10c67b101bf019ca4294453206a)
- **Run-log reader returns the stored usage fields:** The cron run-log parser reads back the persisted usage counters from JSONL and includes them in returned entries.
  - file: [src/cron/run-log.ts:288](https://github.com/openclaw/openclaw/blob/d3f6783b16ecc10c67b101bf019ca4294453206a/src/cron/run-log.ts#L288)
  - sha: [d3f6783b16ec](https://github.com/openclaw/openclaw/commit/d3f6783b16ecc10c67b101bf019ca4294453206a)
- **Regression coverage exists:** `src/cron/run-log.test.ts` verifies that a cron run log entry with usage telemetry is read back with the expected token fields.
  - file: [src/cron/run-log.test.ts:288](https://github.com/openclaw/openclaw/blob/d3f6783b16ecc10c67b101bf019ca4294453206a/src/cron/run-log.test.ts#L288)
  - sha: [d3f6783b16ec](https://github.com/openclaw/openclaw/commit/d3f6783b16ecc10c67b101bf019ca4294453206a)

  - file: [src/gateway/protocol/schema/cron.ts:356](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/gateway/protocol/schema/cron.ts#L356)

## Risks / Open Questions

- There is not a dedicated `session_status` lookup by cron job ID; the implemented surface is `cron.runs`, which already returns per-run usage plus `sessionId`/`sessionKey`.

## Close Comment

Closing as implemented on main. Codex review verified that `cron.runs` now exposes per-run session linkage and token usage in the Gateway schema (`src/gateway/protocol/schema/cron.ts:356`), isolated cron runs populate those counters during execution (`src/cron/isolated-agent/run.ts:752`), and the gateway persists them into the run history returned by `cron.runs` (`src/gateway/server-cron.ts:553`, `src/cron/run-log.ts:288`). The same implementation is present in the `v2026.4.22` release commit `00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`, so this request is already covered by shipped behavior.
