## Summary

Keep open. Current main still has a plausible stale-symlink target-root failure in the bundled runtime mirror: the code hardens the enclosing dist mirror root, but a nested mirror target such as `dist/agents` can remain a symlink before file-copy temp paths are created under it.

## What This Changes

The issue reports that cross-version container upgrades can crash-loop when persisted `plugin-runtime-deps/` contains stale symlinked mirror entries that resolve into the read-only image layer.

## Best Possible Solution

Harden the bundled runtime mirror so every mirror target root is materialized as a real directory before recursive copy or file temp creation, with regression coverage for a stale symlinked directory under `plugin-runtime-deps/`. Keep the fix inside the generic mirror/runtime-deps code path rather than adding deployment-specific cleanup policy.

## Reproduction Assessment

Yes. The report includes a concrete stack trace and symlink listing, and current source tracing shows the same path can write `.openclaw-mirror-...tmp` through a symlinked target directory; no live test was run under the read-only review constraint.

## Solution Assessment

Yes, the reporter’s suggested direction is the narrow maintainable fix. The safest implementation is likely to ensure the mirror target root itself is a non-symlink directory before refreshing it, plus a regression test, rather than relying on operators to wipe `plugin-runtime-deps/`.

## Repair Work Prompt

Fix the stale symlinked bundled-runtime mirror target-root crash described in https://github.com/openclaw/openclaw/issues/75108. Current main can call `refreshBundledPluginRuntimeMirrorRoot` with a target root such as `plugin-runtime-deps/<openclaw-version>/dist/agents` that is itself a stale symlink; `copyBundledRuntimeMirrorFileAtomic` then creates `.openclaw-mirror-...tmp` through that symlink and can fail with EACCES against the read-only packaged image. Keep the change scoped to the generic bundled runtime mirror/runtime-deps path. Ensure every mirror target root is a real directory before recursive copy or temp-file creation, removing stale symlink/file entries and recreating directories as needed. Add regression coverage for a target mirror directory that is a symlink to another directory; the test should prove refresh replaces the symlink with a real directory and writes files into the staged mirror. Consider whether the fix belongs in `copyBundledPluginRuntimeRoot`, `refreshBundledPluginRuntimeMirrorRoot`, or a shared `ensureBundledRuntimeMirrorDirectory` helper. Do not change Docker volume policy, package install semantics, dependency versions, or unrelated plugin loading behavior. Add a user-facing changelog fix entry with reporter credit if required by repo policy.
