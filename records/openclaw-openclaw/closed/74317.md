## Summary

Close as `clawhub`: current main already has the plugin hook and external plugin install surfaces needed for this guardrails idea, while OpenClaw policy routes optional plugin capability to ClawHub/npm unless there is a missing core API or exceptional maintainer/security reason to bundle it. The PR discussion also frames the generic SDK seam work as separate from this concrete plugin implementation, and the proposed bundled diff still carries security-sensitive credential, network, and dynamic-code concerns.

## What This Changes

The branch adds a bundled `extensions/guardrails` plugin with `before_dispatch` allow/block checks, blacklist/HTTP/import connectors, static asset packaging, dependency lockfile updates, and tests.

## Best Possible Solution

Keep the generic `before_dispatch` and plugin install surfaces in core, and ask this implementation to live as an external native plugin published through ClawHub/npm. If #72741 uncovers a genuinely missing generic guardrails interface, that narrower SDK/API gap should be handled separately.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable for the feature request itself because this is a new plugin proposal, but yes for the review defects: the diff shows string-only API-key config and awaits connector initialization from the dispatch path.

## Solution Assessment

Is this the best way to solve the issue? No for bundling in this repository: the narrow maintainable path is an external plugin using the existing hook seam, or an explicit maintainer-owned bundled effort with SecretRef, timeout, docs, metadata, and security review completed first.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Make guardrails API keys SecretRef-aware:** `extensions/guardrails/openclaw.plugin.json:26-28`
  - body: The HTTP connector sends provider credentials, but both global and per-channel `apiKey` fields are string-only. A configured SecretRef object would fail schema/resolver handling, so this bundled credential surface would bypass the existing SecretRef resolution and redaction path.
  - confidence: 0.93
- **[P2] Bound HTTP provider initialization waits:** `extensions/guardrails/index.ts:178`
  - body: The first matching dispatch awaits `entry.initPromise` before any connector timeout or `fallbackOnError` decision is applied. A custom provider whose `init()` never settles can stall external-channel dispatch indefinitely.
  - confidence: 0.88
- **[P2] Bound import connector initialization waits:** `extensions/guardrails/index.ts:322`
  - body: The import backend waits for module load and user `init(args)` before the per-check timeout wrapper exists. A wedged local checker can hang `before_dispatch` initialization instead of honoring `timeoutMs` and fallback behavior.
  - confidence: 0.88
- **[P3] Add the required changelog entry:** `extensions/guardrails/openclaw.plugin.json:4`
  - body: This adds a user-facing feature, but the diff does not update `CHANGELOG.md`. OpenClaw policy requires user-facing `feat` changes to have a changelog entry before landing.
  - confidence: 0.9
- **[P3] Add a guardrails labeler rule:** `extensions/guardrails/openclaw.plugin.json:2`
  - body: This creates a new bundled plugin surface without a matching `.github/labeler.yml` rule, so future changes under `extensions/guardrails/**` would miss automatic surface labeling.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: Needs attention because the diff adds credential-bearing outbound checks and trusted dynamic code execution without SecretRef-ready config or bounded initialization waits.

Concerns:

- **[high] Plaintext provider API-key surface:** `extensions/guardrails/openclaw.plugin.json:26`
  - body: `http.apiKey` and channel override keys are string-only and no SecretRef contract paths are declared, so guardrails provider tokens would not use the existing SecretRef resolution/redaction pipeline if bundled.
  - confidence: 0.93
- **[medium] Unbounded initialization can stall inbound dispatch:** `extensions/guardrails/index.ts:178`
  - body: HTTP provider and import connector initialization promises are awaited from the `before_dispatch` path before configured timeout/fallback behavior applies, creating an availability risk on external-channel ingress.
  - confidence: 0.88
- **[medium] Trusted local code execution surface:** `extensions/guardrails/src/import-connector.ts:23`
  - body: The import connector executes operator-specified local modules through `jiti`; that may fit trusted deployments, but bundled inclusion needs explicit threat-model documentation around trusted paths, hot reload, and local code boundaries.
  - confidence: 0.84
