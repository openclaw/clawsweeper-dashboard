## Label Justifications

- `impact:message-loss`: Queued Telegram updates can remain undelivered behind a stale processing marker.
- `impact:session-state`: The failure is rooted in persisted per-account spool claim state surviving across gateway processes.

## Summary

Close as a duplicate: the PID-reuse failure is a valid Telegram isolated polling spool bug, but the same remaining stale `.processing` claim recovery work is already tracked by the open canonical issue https://github.com/openclaw/openclaw/issues/84674.

## What This Changes

The issue reports that Telegram isolated polling can stop draining a per-account spool when a stale `.json.processing` claim is treated as live after Docker PID reuse.

## Best Possible Solution

Keep the remaining implementation work consolidated in https://github.com/openclaw/openclaw/issues/84674, preserving this PID-reuse evidence as an additional reproduction detail for that fix.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: a fresh processing claim with a different processId but the current process.pid is treated as live and blocks later same-lane spool entries. I did not run a live Docker/Telegram reproduction in this read-only review.

## Solution Assessment

Yes for routing, not as a separate open item: the maintainable path is to fix the existing stale-claim recovery under the canonical open issue, likely by distinguishing current-process PID reuse from a genuinely live other process and adding a focused regression test.
