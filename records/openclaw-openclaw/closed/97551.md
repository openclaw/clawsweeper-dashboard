## Summary

Keep this PR open for maintainer review: current main and the latest release still have unbounded debug proxy response-body capture, while the PR appears to fix the right owner boundary with sufficient real behavior proof. The remaining decision is whether maintainers accept the fixed 16 MiB metadata-only cutoff for oversized debug captures.

## What This Changes

The PR bounds debug proxy captured response-body persistence to 16 MiB and records oversized bodies as metadata-only while adding focused proxy-capture runtime tests.

## Best Possible Solution

Land the bounded metadata-only capture behavior after maintainers accept the fixed diagnostics cutoff and the normal merge gates pass.

## Reproduction Assessment

Yes. Current main and v2026.6.10 call `response.clone().arrayBuffer()` without a byte cap during debug proxy response capture; I did not run an OOM repro in this read-only review.

## Solution Assessment

Yes. The shared proxy-capture runtime is the narrow owner boundary for both global fetch and guarded fetch capture, and the local bounded reader is justified because the shared media helper awaits cancellation while the existing text reader uses fire-and-forget cancellation for non-settling streams.
