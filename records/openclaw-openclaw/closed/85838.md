## Label Justifications

- `P1`: The reported failure breaks a real chat/tool workflow by leaving the session unable to run tools after prompt-lock session drift.
- `impact:session-state`: The failure is about session-file fingerprint drift and persistent session state becoming unusable across turns.
- `impact:message-loss`: The embedded run can fail after tool output is generated, preventing the expected chat/tool result flow from completing.

## Summary

Close as implemented: the report matches the embedded prompt-release session-lock false-takeover family from v2026.5.20, and current main plus v2026.5.22-beta.1 contain the ownership-aware session-write fix from the merged repair PR.

## What This Changes

The issue asks OpenClaw to stop corrupting or invalidating chat sessions when a local LLM/tool turn changes the session file while the embedded prompt lock is released.

## Best Possible Solution

Keep the current ownership-aware session-lock implementation and direct users on v2026.5.20 to upgrade to the fixed beta or later stable build.

## Reproduction Assessment

Yes for the historical release path: v2026.5.20 has the strict prompt-release fence and the report matches the exact EmbeddedAttemptSessionTakeoverError family fixed by https://github.com/openclaw/openclaw/pull/84250. I did not run a live current-main Ollama/qwen reproduction because current main already contains the source and test fix.

## Solution Assessment

Yes. The current implementation is the right narrow fix because it accepts explicit OpenClaw-owned session writes while preserving fail-closed detection for unowned external session-file edits.
