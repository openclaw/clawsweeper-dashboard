## Summary

The reported 2026.4.26 Docker packaging regression is fixed in the shipped v2026.4.27 artifact and current main keeps the same runtime-dependency boundary: xai/browser no longer carry the broken plugin-local TypeBox trees, and their entrypoints import successfully against the complete package-root TypeBox install.

## What This Changes

The issue reports that the 2026.4.26 Docker image shipped truncated TypeBox copies under the bundled xai and browser plugins, causing plugin load failures for missing TypeBox barrel targets.

## Best Possible Solution

Keep the v2026.4.27-and-newer packaging behavior, where xai/browser resolve TypeBox from the complete package-root dependency tree, and rely on release artifact checks to prevent reintroducing incomplete bundled dependency payloads.

## Reproduction Assessment

Yes. The original report gives a precise Docker reproduction against ghcr.io/openclaw/openclaw:2026.4.26, and the same artifact-level checks against ghcr.io/openclaw/openclaw:2026.4.27 show the missing-file path no longer exists and both affected plugin entrypoints import successfully.

## Solution Assessment

Yes. Shipping a corrected release artifact and keeping local runtime-dependency staging opt-in is the narrow maintainable fix; the immutable 2026.4.26 image remains broken, but v2026.4.27 or newer resolves the observable user problem.
