## Summary

Keep open: this is an active maintainer-labeled PR for a valid source-level iMessage bug, and the narrow owner-module fix is directionally right, but it still needs real behavior proof of the new rejection path plus one low-priority guard-order cleanup before merge.

## What This Changes

The PR adds an iMessage reply-time attachment guard and regression test covering attachment field aliases so unsupported reply-with-file calls fail explicitly instead of falling through to `sendRichMessage`.

## Best Possible Solution

Land the iMessage-owned rejection after moving the attachment guard before text validation and adding redacted live proof that `reply` with an attachment returns the explicit error; keep the combined threaded-file primitive tracked upstream at https://github.com/openclaw/imsg/issues/113.

## Reproduction Assessment

Yes, at source level. Current main routes `reply` through `sendRichMessage` after reading only text, and the runtime builds `imsg send-rich --reply-to` without any file argument, so a text-plus-media reply can succeed while dropping the media.

## Solution Assessment

Yes, with one small adjustment. Rejecting unsupported iMessage reply attachments in the iMessage action adapter is the narrowest maintainable fix until imsg supports a combined reply-and-file primitive; the attachment guard should run before the text-required check for consistent guidance.
