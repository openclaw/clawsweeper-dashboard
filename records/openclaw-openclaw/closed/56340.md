## Summary

Keep open: current main already covers the Codex WebSocket exclusion, but the public hook contract change is still absent and this PR has concrete approval-ownership and profile-loading defects in security-sensitive tool paths.

## What This Changes

The PR narrows OpenAI WebSocket transport selection, adds soft/hard `before_tool_call` block semantics with docs/tests, changes live `.profile` env loading, and broadens the `.tgz` gitignore pattern.

## Best Possible Solution

Rebase onto current main, keep the endpoint-aware WebSocket guard already present there, implement `blockMode` in the current `hook-types.ts` contract and runner/consumer paths, clear/drop approval ownership whenever soft-block ownership changes, fix profile loading to require profile-derived vars, and add focused tests plus a changelog entry.

## Reproduction Assessment

Yes. Static inspection gives high-confidence paths: clear a soft block after the same plugin requested approval, or add a different plugin's soft block after an earlier approval; for profile loading, have the first shell emit unrelated env while failing to source `.profile`.

## Solution Assessment

No, not as-is. The direction is plausible, but the branch must preserve approval ownership correctly, avoid profile fallback false positives, rebase over the current transport guard, and add the required changelog entry before it is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Clear approval state when clearing soft blocks:** `src/plugins/hooks.ts:723-728`
  - body: When a lower-priority hook returns `{ block: false }`, this branch clears the soft-block fields but can leave `acc.requireApproval` in the merged result. The consumer may still prompt for approval even though the block was explicitly cleared, so clearing a soft block is incomplete.
  - confidence: 0.8
- **[P2] Drop stale approvals when block ownership changes:** `src/plugins/hooks.ts:751-752`
  - body: If one hook requests approval and a later different plugin returns a soft block, the merger can keep the earlier approval. That lets an unrelated approval flow authorize a block that should remain blocked by default unless its owner also requested approval or the block is cleared.
  - confidence: 0.77
- **[P2] Require profile-derived env before accepting a shell:** `test/test-env.ts:92-98`
  - body: The new shell-candidate loop counts any newly applied variable from the full `env -0` output. A shell that fails to source `.profile` but emits unrelated variables can stop later fallbacks, so profile-only live credentials may still be skipped.
  - confidence: 0.74
- **[P3] Add the required changelog entry:** `src/plugins/types.ts:2152-2155`
  - body: This PR changes user-facing agent transport behavior and the public plugin hook contract, but it does not update `CHANGELOG.md`. OpenClaw release policy requires a single-line entry for this kind of fix/API-facing change.
  - confidence: 0.86
