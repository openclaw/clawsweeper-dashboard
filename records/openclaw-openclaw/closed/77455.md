Keep open: current main still has the reported response/persistence mismatch, and open closing PRs already target the same bug. The right cleanup path is to review and land one canonical persisted-response fix before closing this issue.

## What This Changes

The issue asks config.set, config.patch, and config.apply responses to return the persisted disk config instead of the pre-write in-memory config.

## Best Possible Solution

Land one canonical fix that carries persistedConfig from writeConfigFile through replaceConfigFile and commitGatewayConfigWrite into redacted write responses, with regression coverage for response/readback equality.

## Reproduction Assessment

Yes, source-reproducible. Current main config.set, config.patch, and config.apply write through a path that drops the persisted write result and then respond with parsed.config or validated.config.

## Solution Assessment

Yes. The narrow maintainable fix is to thread the persisted write result through the existing config write abstractions and use that value in all write RPC responses; the active PR cluster is already pursuing that direction.
