## Summary

Keep open: current main still has the reported trajectory flush timeout shape, and an open PR now explicitly closes this issue but has not merged yet.

## What This Changes

The issue reports that large runtime trajectory files can make `pi-trajectory-flush` time out and continue saturating the gateway event loop.

## Best Possible Solution

Review and land #77154 if its bounded trajectory runtime fix holds up, while leaving broader session and event-loop pressure to the related open reports.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection does show the current path where the cleanup timeout returns without stopping the underlying trajectory flush, and the writer continues draining its queued chain.

## Solution Assessment

Yes, the narrow bounded recorder/writer approach in the linked PR is the maintainable direction. A timeout environment knob alone would not stop the background flush from continuing after the wrapper returns.
