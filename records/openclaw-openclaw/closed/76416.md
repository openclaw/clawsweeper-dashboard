## Summary

Keep open. Current main still matches the reported regression: the late-bound `web_search` tool can classify a bundled provider as runtime-only after dropping config context, then fail when the active sub-agent plugin registry excludes provider-only plugins such as Brave.

## What This Changes

The issue reports that sub-agent `web_search` calls fail with bundled providers such as Brave when the sub-agent registry is narrowed by tool allowlists.

## Best Possible Solution

Fix the late-bound bundled-owner lookup in `createWebSearchTool` so configured bundled providers stay on bundled provider resolution even when the active sub-agent registry is narrowed, and lock it with regression coverage.

## Reproduction Assessment

Yes, at source level: create `web_search` with `lateBindRuntimeConfig: true`, active runtime metadata selecting `brave`, and an active registry scoped to plugin tools that omits Brave; the current path routes through runtime providers and can reach the reported empty-candidate error. I did not run a live provider call in this read-only sweep.

## Solution Assessment

Yes, the narrow maintainable fix is in the web_search late-bind owner lookup, not in Brave's manifest or sub-agent policy. Adding `contracts.tools` to Brave would load a provider-only plugin for the wrong reason and broaden the tool-scoping contract.
