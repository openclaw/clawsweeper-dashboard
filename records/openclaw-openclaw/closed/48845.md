## Summary

Close as superseded: the valid Feishu reaction-ID bug is now tracked by the open canonical normalizer PR, while this branch keeps overlapping, conflicting changes that regress dedupe/topic routing and has no real Feishu proof.

## What This Changes

The PR changes Feishu reaction synthetic event IDs, outbound reply anchoring, missing chatType parsing, reaction timeout handling, and related reaction tests.

## Best Possible Solution

Keep the canonical Feishu normalizer path open, land one reconciled fix that normalizes synthetic reaction IDs only at Feishu API boundaries, and preserve fail-closed chat-type handling unless maintainers explicitly change that policy.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path for the blocking PR regressions: deterministic reaction IDs collide with current 24-hour Feishu message-key dedupe, and topic routing reads `ctx.messageId` before the PR's outbound-only unwrap. I did not run a live Feishu workspace in this read-only review.

## Solution Assessment

No. The better solution is the canonical API-boundary normalizer approach: keep event-unique synthetic IDs for OpenClaw processing and strip `:reaction:` suffixes only before calling Feishu APIs, without adding unvalidated chat-id prefix inference.

## Review Findings

Full review comments:

- **[P2] Keep reaction synthetic IDs event-unique:** `extensions/feishu/src/monitor.account.ts:129-136`
  - body: The new ID is deterministic for message, action, emoji, and sender. Current Feishu processing dedupes by `event.message.message_id` with a 24-hour TTL, so a repeated same-action reaction within that window is skipped; keep an event-unique token and carry the original message ID separately for Feishu API calls.
  - confidence: 0.9

  - body: This only unwraps the ID for outbound replies after group/topic routing has already used `ctx.messageId`. In topic session modes, history and bindings can be written under the synthetic reaction key while the visible reply targets the original message; put the original anchor into reply/thread metadata before routing.
  - confidence: 0.84
- **[P2] Preserve topic_group when inferring chat types:** `extensions/feishu/src/send.ts:265-271`
  - body: The parser no longer accepts `topic_group` explicitly before prefix fallback. A fetched topic message with an `oc_` chat ID can be downgraded to `group`, losing topic-specific routing metadata; keep `topic_group` as a valid explicit chat type.
  - confidence: 0.86
