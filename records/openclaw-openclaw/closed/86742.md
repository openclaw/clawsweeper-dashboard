## Label Justifications

- `P2`: This is a confirmed restart-continuation bug with limited but real blast radius in coalesced gateway restart workflows.
- `impact:message-loss`: The accepted continuation message can be silently dropped instead of creating the promised follow-up agent turn.
- `impact:session-state`: The lost continuation leaves the post-restart agent workflow state behind what the tool response implied.

## Summary

Keep open: current main still has the source-reproducible restart continuation gap, and the linked open PR is the active fix path but has not merged yet.

## What This Changes

A gateway restart request with `continuationMessage` can coalesce with an already-emitted restart and return success without persisting or queuing the continuation.

## Best Possible Solution

Land a narrow fix like https://github.com/openclaw/openclaw/pull/83370, or otherwise guarantee exactly-once continuation queuing for the correct restart/session with regression coverage.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: emit or schedule a restart, then call `gateway.restart` with `continuationMessage` while `hasUnconsumedRestartSignal()` is true. The scheduler returns before `beforeEmit` can write the sentinel that startup later requires for continuation delivery.

## Solution Assessment

Yes, the narrowest maintainable fix is to make the scheduler/tool contract explicit: either queue the continuation exactly once for the correct pending restart or report that it was not queued. The current open PR follows the explicit unqueued-result path and avoids unsafe global sentinel rewriting.
