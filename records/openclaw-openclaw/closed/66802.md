## Summary

Current main implements the requested memory-flush model override and it shipped in v2026.4.27. The config schema accepts `agents.defaults.compaction.memoryFlush.model`, the memory-core plan carries it, the runtime uses it as an exact provider/model override without active fallbacks, and regression/docs coverage now describe the behavior.

## What This Changes

The issue requests a separate `agents.defaults.compaction.memoryFlush.model` setting so pre-compaction memory flush turns can use a cheaper model than the active session model.

## Best Possible Solution

Keep the shipped exact memory-flush model override as the end state; any future work should be a narrower follow-up for additional policy knobs such as a separate fallback chain, not a reopening of this cost-control field.

## Reproduction Assessment

Not applicable as a feature request, but the current-main verification path is high confidence: the strict schema accepts the nested key, the plan propagates it, and the runtime resolves the flush turn against that exact override.

## Solution Assessment

Yes. The shipped implementation matches the requested boundary by adding the field to config/docs/schema, carrying it through the memory plugin plan, and applying it only to the silent memory-flush turn without silently falling back to the paid active session chain.
