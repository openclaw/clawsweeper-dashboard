## Summary

Keep open: the WebChat auto-TTS bug is plausible from source, but this branch is not mergeable because it corrupts source encoding/text, weakens media normalization/trust boundaries, and still does not carry the trust flag through the final streamed TTS payload.

## What This Changes

The PR changes speech-core auto-TTS to trust generated local audio and allow block-kind TTS in final mode, skips shared reply-media normalization for trusted payloads, and changes the public ReplyPayload type.

## Best Possible Solution

Land a narrow WebChat auto-TTS fix that preserves trustedLocalMedia on the final synthetic TTS payload, keeps final-mode streaming as one post-stream audio reply, keeps shared media normalization and SDK trust boundaries intact, restores UTF-8 text, and adds focused regression coverage plus redacted real WebChat proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No live WebChat/MiniMax run was performed, but source inspection shows WebChat requires trustedLocalMedia and the current final-stream TTS payload construction does not carry that flag into the WebChat embedding path.

## Solution Assessment

Is this the best way to solve the issue? No; the patch touches relevant code but corrupts source files, changes final-mode streaming semantics, bypasses shared media normalization, expands an internal SDK trust boundary, and still misses the final synthetic payload path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Restore UTF-8 source encoding:** `src/plugin-sdk/reply-payload.ts:1`
  - body: The PR converts `src/plugin-sdk/reply-payload.ts` into UTF-16LE-looking bytes with a double BOM and NUL-separated text, so GitHub treats it as binary and TypeScript/tooling may not parse it as normal source. Please restore this file as UTF-8 text before changing the SDK contract.
  - confidence: 0.96
- **[P2] Restore the TTS max-length interpolation:** `extensions/speech-core/src/tts.ts:559`
  - body: The prompt hint changes from `≤${maxLength}` to `鈮?{maxLength}`, dropping the interpolation and leaving mojibake in an agent-facing instruction. Please restore the original text and repair the other encoding damage in the branch.
  - confidence: 0.94
- **[P2] Keep final-mode TTS out of streamed blocks:** `extensions/speech-core/src/tts.ts:1762`
  - body: In `mode: "final"`, block text is accumulated and synthesized once after streaming completes. Allowing `kind: "block"` through this gate can synthesize per block and duplicate or fragment the intended single final audio reply.
  - confidence: 0.9
- **[P2] Do not bypass shared media normalization:** `src/auto-reply/reply/dispatch-from-config.ts:595-597`
  - body: `trustedLocalMedia` is needed by WebChat embedding, but this early return runs in shared dispatch for every channel. It skips the normal staging, blocking, and media path normalization that non-WebChat deliveries still rely on.
  - confidence: 0.88
- **[P2] Carry trust into the synthetic final payload:** `extensions/speech-core/src/tts.ts:1852`
  - body: Adding `trustedLocalMedia` to the speech-core result is not enough for streamed final replies: dispatch rebuilds `ttsOnlyPayload` from selected fields and drops the flag before WebChat sees it. Copy the trust flag on that final synthetic payload instead of relying on block-mode synthesis.
  - confidence: 0.84
