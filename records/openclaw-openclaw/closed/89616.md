## Summary

Current main and the latest shipped release already preserve Slack thread routing for restart sentinel delivery; the reported 2026.3.28 behavior is covered by the newer restart-sentinel route capture and Slack transport mapping.

## What This Changes

The issue asks for gateway restart announce messages in Slack to reply in the originating thread instead of posting as top-level channel messages.

## Best Possible Solution

Keep the shipped restart-sentinel route capture and Slack reply-transport path; users on 2026.3.28 should upgrade to v2026.5.28 or newer.

## Reproduction Assessment

No current-main failing reproduction remains: the old-version steps are clear, but the shipped v2026.5.28 route now carries the restart thread into Slack delivery.

## Solution Assessment

Yes: preserving the generic delivery context in restart sentinel code and letting the Slack plugin translate it to Slack `replyToId` is the right owner-boundary fix.
