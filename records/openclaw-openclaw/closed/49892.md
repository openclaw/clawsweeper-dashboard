## Summary

The PR's original docs bug no longer reproduces on current published docs, and the proposed changes conflict with the current docs i18n pipeline and current Browserless documentation/behavior on `main`.

## Evidence

- **Published docs currently serve `zh-CN`, not `zh-Hans`:** On 2026-04-25, `curl -sSI` against the live docs returned `200` for `/zh-CN` and `/zh-CN/tools/browser`, and `404` for `/zh-Hans` and `/zh-Hans/tools/browser`. That means the PR's redirect target is not the current published route shape.
  - command: `for u in https://docs.openclaw.ai/zh-CN https://docs.openclaw.ai/zh-Hans https://docs.openclaw.ai/zh-CN/tools/browser https://docs.openclaw.ai/zh-Hans/tools/browser; do echo "URL $u"; curl -sSI "$u" | sed -n '1,12p'; echo; done`
- **Source repo no longer owns localized docs trees:** The docs i18n README says generated locale trees such as `docs/zh-CN/**` are no longer committed here; the publish repo owns them and refreshes `docs/zh-CN/**` there.
  - file: [docs/.i18n/README.md:14](https://github.com/openclaw/openclaw/blob/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5/docs/.i18n/README.md#L14)
  - sha: [84dc9f12f1b9](https://github.com/openclaw/openclaw/commit/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5)
- **Publish sync maps `zh-Hans` language metadata onto the `zh-CN` directory:** The docs sync script defines generated locale `{ language: "zh-Hans", dir: "zh-CN", navFile: "zh-Hans-navigation.json" }`, then rewrites the publish repo's `docs/docs.json` with generated locale navigation during sync.
  - file: [scripts/docs-sync-publish.mjs:27](https://github.com/openclaw/openclaw/blob/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5/scripts/docs-sync-publish.mjs#L27)
  - sha: [84dc9f12f1b9](https://github.com/openclaw/openclaw/commit/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5)
- **Current docs policy forbids editing localized docs in this repo:** `docs/AGENTS.md` explicitly says foreign-language docs are not maintained in this repo and to not add or edit localized docs under `docs/<locale>/**` here.
  - file: [docs/AGENTS.md:16](https://github.com/openclaw/openclaw/blob/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5/docs/AGENTS.md#L16)
  - sha: [84dc9f12f1b9](https://github.com/openclaw/openclaw/commit/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5)
- **Current Browserless docs on `main` support either HTTPS discovery URLs or direct WSS URLs:** The current English browser docs say OpenClaw can use Browserless HTTPS or WebSocket URLs and that users may keep an HTTPS URL and let OpenClaw discover `/json/version`, which conflicts with this PR's Browserless-only-`wss://` wording.
  - file: [docs/tools/browser.md:323](https://github.com/openclaw/openclaw/blob/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5/docs/tools/browser.md#L323)
  - sha: [84dc9f12f1b9](https://github.com/openclaw/openclaw/commit/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5)
- **Current browser code still supports HTTPS CDP discovery and normalizes to WSS:** `getChromeWebSocketUrl()` treats HTTP(S) endpoints and bare WS roots separately, using `/json/version` discovery for HTTP(S). The test suite also verifies that an HTTPS CDP URL normalizes to a WSS debugger URL.
  - file: [extensions/browser/src/browser/chrome.ts:361](https://github.com/openclaw/openclaw/blob/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5/extensions/browser/src/browser/chrome.ts#L361)
  - sha: [84dc9f12f1b9](https://github.com/openclaw/openclaw/commit/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5)
