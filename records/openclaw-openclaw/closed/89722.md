## What This Changes

Adds `isIsolatedCronSessionKey`, updates the cron session reaper to sweep base-shaped isolated cron keys, and adds regression tests for that behavior.

Yes. Source inspection shows current main only sweeps `isCronRunSessionKey` matches, while isolated jobs use stable `agent:*:cron:<jobId>` base keys that are excluded from that predicate.

## Solution Assessment

No. The PR fixes the matching gap by pruning stable base rows, but current source and docs treat those rows as durable carry-forward metadata for isolated cron jobs.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P1] Keep durable cron base sessions out of the sweep:** `src/cron/session-reaper.ts:75`
  - body: This makes `agent:*:cron:<jobId>` entries eligible for deletion when `updatedAt < cutoff`. Those base rows are not disposable run rows: `resolveCronSession` sanitizes and reuses them to carry labels plus explicit user model/auth overrides across fresh isolated runs, and the docs describe that carry-forward contract. With the default 24h retention, an active weekly or monthly isolated cron job can lose that metadata before its next run. Keep active base rows out of retention and clean up only real completed run rows or explicitly orphaned deleted-job rows.
