The branch adds a memory-core startup helper that sweeps aged `${dbPath}.tmp-*` SQLite reindex triplets before creating a persistent MemoryIndexManager, plus focused real-filesystem tests for the helper.

## Best Possible Solution

Keep one canonical memory-core startup cleanup with active-reindex ownership protection; https://github.com/openclaw/openclaw/pull/92891 is the stronger current candidate, so this branch should close rather than continue as a second cleanup path.

## Reproduction Assessment

Yes. Source inspection shows current main creates `${dbPath}.tmp-${randomUUID()}` during safe reindex and only removes it in normal success/error cleanup, so a hard kill can leave the triplet behind; I did not run a destructive SIGKILL repro in this read-only review.

## Solution Assessment

No for this branch. A startup sweep is the right general fix, but the age-only ownership guard is weaker than the open canonical PR's PID-lock and liveness guard.
