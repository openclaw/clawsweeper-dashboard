## Summary

Current `main` already resolves gateway service entrypoints through the installed CLI/package layout instead of guessing a broken `.../lib/node_modules/dist/index.js` path, and this logic was already shipped in `v2026.4.22`. The systemd unit renderer uses those resolved program arguments directly for `ExecStart`, so the npm-global failure described in #42367 is covered.

## Evidence

- **Gateway install now resolves `.bin/openclaw` to the package's real dist entrypoint:** `resolveCliEntrypointPathForService()` builds candidates from the current launcher path and, for `node_modules/.bin/openclaw`, `appendNodeModulesBinCandidates()` maps that to `node_modules/openclaw/dist/{index,index.mjs,entry,entry.mjs}`. That prevents the missing-package-segment path reported in the issue.

## Close Comment

Closing as implemented on `main` after Codex review. The current gateway install path resolves the actual installed CLI entrypoint before rendering the systemd unit, rather than guessing a broken `.../lib/node_modules/dist/index.js` path. The relevant logic is in `src/daemon/program-args.ts:18` and is covered by regression tests for npm-style shim/global-install layouts in `src/daemon/program-args.test.ts:87`. The generated systemd `ExecStart` comes from those resolved program arguments via `src/commands/daemon-install-helpers.ts:268` and `src/daemon/systemd-unit.ts:51`. This behavior was already shipped in `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`), and the changelog entry is at `CHANGELOG.md:691`.
