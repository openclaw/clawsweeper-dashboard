## Summary

Current main and the v2026.4.26 source no longer send an empty user prompt for the heartbeat/cron/exec wakeup path described here. Those wakeups enter reply generation with `isHeartbeat: true`, `runPreparedReply` replaces the visible transcript prompt with the non-empty `[OpenClaw heartbeat poll]` marker, and the runtime-context splitter therefore submits that marker as the provider prompt while carrying the real wakeup body as hidden runtime context. The generic runtime-only branch can still return `prompt: ""` for a direct empty `transcriptPrompt`, but that is not the current heartbeat/cron/exec call path.

## Best Possible Solution

Close this as implemented for the heartbeat/cron/exec wakeup path. If a separate caller can still reach `resolveRuntimeContextPromptParts({ effectivePrompt, transcriptPrompt: "" })` and hit a provider that rejects empty prompts, that should be filed as a narrower runtime-only prompt compatibility bug with that specific caller/reproduction.
