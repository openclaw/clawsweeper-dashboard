## Summary

Close #69240 as superseded by #54314. The cron concurrency bug is real and still visible on current main, but #54314 is the broader canonical PR for the same remaining work: it introduces a dedicated cron-nested lane, wires cron.maxConcurrentRuns through startup and hot reload, and addresses review feedback about not widening the shared Nested lane.

## Best Possible Solution

Close this PR as superseded and consolidate review on #54314 or an equivalent dedicated CronNested-lane implementation that covers current cron dispatch, embedded-runner lane resolution, startup/hot-reload concurrency wiring, and focused regression tests without changing unrelated shared Nested lane behavior.
