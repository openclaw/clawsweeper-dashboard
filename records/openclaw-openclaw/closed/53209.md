## Summary

Keep open: current main still lacks the requested skill-declared sensitive-env sandbox allowlist, and this PR remains a useful implementation candidate, but the latest PR head has blocking security-sensitive defects in how it selects and syncs skill credentials into sandbox exec env.

## What This Changes

The PR adds skill-derived sensitive environment allowlisting to Docker sandbox creation, mirrors injected skill env values into sandbox exec env, and expands sanitizer, sandbox, skills, and embedded-runner regression tests.

## Best Possible Solution

Rebase the branch on current main, then make sandbox-create allowlisting and sandbox exec-env syncing derive from one active, eligible, configured skill source that preserves per-agent skill filters; alternatively keep skills.entries.* host-only and limit the fix to sandbox.docker.env. Update the sandboxed-skills docs to match the chosen contract before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Restrict synced env keys to eligible skills:** `src/agents/pi-embedded-runner/run/attempt.ts:1717-1724`
  - body: On the non-snapshot path, `applySkillEnvOverridesWithResult` consumes the loaded `skillEntries` and the returned keys are copied into sandbox exec env. Because this source is not constrained to the same active eligible skill set as the sandbox-create allowlist, a disabled or otherwise ineligible configured skill can still inject a secret into sandbox execs. Filter the override/sync source to active eligible skills in both run and compaction.
  - confidence: 0.89
- **[P1] Preserve per-agent skill filters in sandbox plumbing:** `src/agents/pi-embedded-runner/run/attempt.ts:1685-1689`
  - body: Current main passes the resolved agent id into skill entry resolution and prompt building so `agents.list[].skills` is enforced. This branch's new helper and edited call sites omit that agent id, so a multi-agent setup can collect, prompt, or sync skills outside the active agent's allowlist. Thread the resolved agent id through the allowlist helper, skill entry resolution, and prompt resolution in both run and compaction.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff changes sandbox secret propagation and still has concrete paths where inactive or out-of-scope skill credentials can reach sandbox exec env.

Concerns:

- **[high] Inactive skill credentials can reach sandbox exec env:** `src/agents/pi-embedded-runner/run/attempt.ts:1717`
  - body: The PR syncs keys returned by `applySkillEnvOverridesWithResult` into sandbox.docker.env/backend.env, but that override call is not limited to the same eligible skill source used by the allowlist helper. Disabled or ineligible configured skills can therefore still contribute secret env keys to sandbox execs.
  - confidence: 0.89

  - body: The PR head omits the agentId plumbing now present on current main for skill entry resolution and prompt construction. In multi-agent setups, that can widen the skill/env surface beyond the active agent's configured allowlist.
