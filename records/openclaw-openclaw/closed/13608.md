## Summary

Current `main` already ships native observability export through the bundled `diagnostics-otel` plugin, with documented config, exported metrics/traces/logs, and focused tests. The issue’s core request for standardized metrics/observability support is therefore already implemented on `main`, even though there is not a separate built-in Prometheus `/metrics` endpoint.

## Close Comment

Closing after a Codex review because current `main` already implements the core request through the bundled `diagnostics-otel` plugin. The exporter is wired in `extensions/diagnostics-otel/src/service.ts:72`, it records native metrics for token usage, cost, run duration, webhooks, message flow, queues, sessions, and retries in `extensions/diagnostics-otel/src/service.ts:163`, the public config is documented in `docs/gateway/configuration-reference.md:3570` and schema-backed in `src/config/zod-schema.ts:284`, and the exported signals/metric names are documented in `docs/logging.md:256` with focused coverage in `extensions/diagnostics-otel/src/service.test.ts:194`. This does not add a separate Prometheus `/metrics` endpoint, but it does mean OpenClaw already has native standardized observability export on `main` via OpenTelemetry. If a first-party Prometheus pull endpoint is still wanted in addition to OTLP, that should be tracked as a narrower follow-up request.
