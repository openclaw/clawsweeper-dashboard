## Summary

Keep open. Current main still has the reported failure mode: metadata-only exec completion strings are classified as exec events, sent through the user-relay prompt, and any exec-event reply is exempted from the normal HEARTBEAT_OK skip path. The report is concrete, recent, and narrow enough for a focused fix.

## What This Changes

The issue asks OpenClaw to stop async/background exec completion relays from asking chat users for missing command logs when the completion event only contains an id or exit status and the process session is gone.

## Best Possible Solution

Normalize exec completion events before prompt construction so output-bearing completions can still be relayed, successful metadata-only completions are consumed silently, and failure metadata without output becomes a concise runtime notice rather than a prompt to retrieve or ask the user for lost logs. Let normal HEARTBEAT_OK suppression apply when there is no relayable payload, and cover the behavior with heartbeat-runner regression tests.

## Repair Work Prompt

Fix this issue: async/background exec completion relays must not ask chat users for missing logs when the completion event has only an exec id and exit status. In current main, `isExecCompletionEvent` accepts metadata-only `Exec completed (<id>, code 0)` strings, `buildExecEventPrompt` treats that as relayable event text, and `runHeartbeatOnce` prevents normal HEARTBEAT_OK suppression for all exec completions. Keep output-bearing exec completions user-visible. For success with no stdout/stderr/result payload, consume the event silently or use HEARTBEAT_OK internally. For failure with no payload, send only a concise runtime diagnostic with stable id/exit metadata; do not tell the model or user to retrieve missing logs. Avoid solving the broader #69492 event-audience design in this PR. Add focused regression coverage for a metadata-only successful exec event after session cleanup/no process logs, including that Telegram delivery is not called and the event is consumed. If behavior changes are user-facing, add a single-line changelog fix entry crediting @gkoch02; do not add broad docs unless config semantics change.
