## Label Justifications

- `P1`: The issue affects a core TUI/gateway completion path where a real final assistant reply can disappear from the user-visible transcript until a later reload.
- `impact:message-loss`: The reported failure mode removes the final assistant message from the visible TUI chat log after it was already rendered.
- `impact:session-state`: The bug is a stale history rebuild versus live stream state race that makes the local transcript view drift from the just-completed run.

## Summary

Keep open: current main still starts a non-local final-event history reload before finalizing the streamed assistant text, and `loadHistory()` still clears and rebuilds the chat log from whatever the server has persisted. The merged repaint fix at https://github.com/openclaw/openclaw/pull/87423 does not address this destructive reload path, while the closely matching fix at https://github.com/openclaw/openclaw/pull/35618 was closed unmerged.

## What This Changes

The issue asks to prevent the TUI from losing a just-finalized external assistant reply when a final-event history reload races stale server persistence.

## Best Possible Solution

Land a focused TUI fix that keeps the just-finalized displayable assistant content authoritative, either by skipping the destructive reload for displayable external finals or reconciling/restoring that final text after stale history replay, with regression coverage for the stale-history overwrite.

## Reproduction Assessment

Yes for a source-level reproduction, but not a live run in this review. Current main still reloads history for a non-local final with text, and that reload clears the chat log before replaying only persisted messages.

## Solution Assessment

Yes, the issue's direction is the right fix class: this is not a repaint problem, and the durable solution should prevent stale history replay from replacing a correct local final. The safest narrow shape is to skip or reconcile the reload only when the final event already has displayable assistant content.
