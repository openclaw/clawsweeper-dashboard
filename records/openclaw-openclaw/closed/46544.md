## Summary

Keep open. Current main still routes CLI `message send --media` through the default agent while exposing `--account` but no `--agent` selector, so a non-default account's workspace is not selected for media-root resolution. The runner and media helpers already support agent-scoped roots when an agent id is supplied, which makes this a real but security-sensitive CLI semantics gap rather than an implemented-on-main cleanup.

## What This Changes

Request `openclaw message send --media` to use agent-scoped media roots for multi-account agents instead of only default-agent/default media roots.

## Best Possible Solution

Add a documented `openclaw message ... --agent <id>` selector, validate it against configured agents, thread it through the message CLI helper and `messageCommand` into `runMessageAction`, and keep media roots scoped to the selected agent workspace. Account-to-agent inference should only be added if it is unique, deterministic, documented, and tested; current main should not include every configured workspace by default.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to configure a non-default agent workspace bound to a Telegram account and run `openclaw message send --media` with `--account`; current `messageCommand` still supplies the default agent id while the outbound runner scopes roots from the supplied id.

## Solution Assessment

No current-main solution is present. The narrow maintainable direction is an explicit validated agent selector; broadening to all workspaces or open-ended configured roots is less safe for a filesystem media allowlist.
