## Summary

Current main already replaces the vulnerable Lit decorator-based `openclaw-app` registration with an explicit guarded `customElements.define` path, and that fix shipped in `v2026.5.7`.

## What This Changes

The issue asks for the Control UI Chat view to render in Chromium/Vivaldi after pairing instead of leaving `<openclaw-app>` unregistered.

## Best Possible Solution

Keep the explicit guarded custom element registration shipped in `v2026.5.7`; if a current release still fails, track it as a new browser/runtime report with console and network diagnostics.

## Reproduction Assessment

Yes for the historical failure path: the issue gives detailed Vivaldi/Chromium DevTools evidence on `v2026.3.7`, and that release's source used only Lit decorator registration for `openclaw-app`. I did not establish a current-main failing reproduction because current main has the explicit registration fix.

## Solution Assessment

Yes. Replacing decorator-based registration with a direct guarded `customElements.define` is the narrow maintainable fix for a startup path where `customElements.get("openclaw-app")` remained undefined.
