## Summary

Current main already validates config writes before persisting/restarting and also recovers from invalid config on startup/reload via last-known-good restoration, so the requested safeguard is already implemented.

## Close Comment

Closing from a Codex review because current `main` already implements this safeguard. Config write RPCs validate before persisting or restarting (`src/gateway/server-methods/config.ts:190`, `src/gateway/server-methods/config.ts:444`, `src/gateway/server-methods/config.ts:479`, `src/gateway/server-methods/config.ts:629`), and the gateway also restores the last-known-good config if an invalid edit still reaches startup or reload (`src/gateway/server-startup-config.ts:71`, `src/gateway/config-reload.ts:218`, `src/gateway/config-reload.ts:361`). The behavior is covered by tests in `src/gateway/server.config-patch.test.ts:389`, `src/gateway/server-startup-config.recovery.test.ts:57`, and `src/gateway/config-reload.test.ts:683`, and it is documented in `docs/gateway/configuration.md:87` and `docs/gateway/configuration.md:500`. I also verified the shipped `v2026.4.22` tree (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`) contains these guards. If you can still reproduce a concrete self-modification path that bypasses them on a current build, please open a new issue with the exact config write method/tool and the rejected payload.
