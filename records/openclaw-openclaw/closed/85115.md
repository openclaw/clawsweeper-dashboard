## Summary

Keep open: this is a plausible fix for a real WebChat history-loss bug, but it should not merge yet because the proposed mirror still depends on a final-delivery path that current dispatch suppresses for WebChat room events, and one structured mirror path can persist unsanitized interactive text. The protected `maintainer` label also means this needs explicit maintainer handling.

## What This Changes

The PR captures successful internal WebChat `message` tool sends as source-reply payloads, summarizes internal-ui message tool results, and adds sanitizer and handler coverage so history can mirror message-tool replies.

## Best Possible Solution

Persist `sourceReplyTranscriptMirror` for `message_tool_only` room events without sending a duplicate final reply, sanitize all structured text fields before durable mirrors, and attach redacted real WebChat reload or terminal proof.

## Reproduction Assessment

Yes, source-level: current WebChat room events select `message_tool_only`, while dispatch suppresses marked room-event final payloads before the transcript mirror helper runs. I did not run a live browser reload in this read-only review.

## Solution Assessment

No. Capturing the message-tool payload is useful, but the best fix also needs a mirror-only room-event persistence path and sanitizer coverage for interactive structured text.
