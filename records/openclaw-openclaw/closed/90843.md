## Summary

Keep open. The report matches current source and the published v2026.6.1 package layout: `/export-session` loads templates relative to the bundled runtime at `dist/export-html`, while the build copies them to `dist/auto-reply/reply/export-html`, so the latest shipped command can fail before producing HTML.

## What This Changes

The issue requests fixing `/export-session` so the packaged HTML export templates are found at runtime instead of throwing `ENOENT`.

## Best Possible Solution

Align the runtime template lookup and packaged asset copy path, then add a packaging-focused regression check so future builds prove the bundle can read the shipped templates.

## Reproduction Assessment

Yes, there is a high-confidence source/package reproduction path: install v2026.6.1 and run `/export-session`, where the bundle reads `dist/export-html/template.css` but the package ships the assets under `dist/auto-reply/reply/export-html`. I did not execute the live channel command during this read-only review.

## Solution Assessment

Yes, the best fix is a narrow packaging/runtime alignment plus regression coverage for the packaged path. A broad export-session refactor or workaround copy by users is not needed for this specific bug.
