## Summary

Keep open. This PR is draft and protected by the `maintainer` label, and the current diff no longer matches the title/body: it lowers the subagent lane default instead of adding the described event-loop yield. More importantly, the changed default only caps work after the agent run reaches the command lane, while the current-main code still performs substantial agent preparation before that queue, so the patch does not high-confidently fix the linked event-loop saturation report.

## What This Changes

The PR lowers the default subagent lane concurrency from 8 to 3, adds config-default unit coverage for the bounded default and override path, adds a changelog entry, and leaves a whitespace-only change near `spawnSubagentDirect()`.

## Best Possible Solution

Guard the subagent startup/preparation phase that runs before `runEmbeddedPiAgent()` reaches `CommandLane.Subagent`, either by moving that preparation under a bounded subagent startup queue or by adding a narrower pre-prep throttle/offload that is validated against the #75378 workload. If the default also changes, align docs and changelog credit before landing.

## Reproduction Assessment

Unclear. The linked #75378 report has concrete v2026.4.29 reproduction steps and event-loop/liveness logs for three simultaneous heavy-model `sessions_spawn` calls, and current-main code review confirms the same pre-queue preparation path exists, but this review did not live-run that workload.

## Solution Assessment

No. Lowering `DEFAULT_SUBAGENT_MAX_CONCURRENT` is not the narrowest proven fix because that value is applied to the command lane after agent preparation has already started; the safer fix should bound or offload the preparation path itself.
