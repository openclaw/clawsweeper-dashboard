## Summary

Keep open. Current main still applies bundled runtime dependency aliases to plugin Jiti loads, and the alias resolver still selects conditional `import` targets before considering CommonJS `require`, which matches the reported `ws` constructor-shape break for custom plugins. The prior ClawSweeper comment remains accurate; #74312 is useful background but does not supersede this plugin-loader compatibility bug.

## What This Changes

The issue asks OpenClaw to preserve CommonJS `require("ws")` constructor semantics for custom plugins loaded through the plugin loader, or otherwise document and support a safe resolver pattern.

## Best Possible Solution

Preserve CommonJS semantics for plugin `require()` of bundled runtime dependencies by choosing a `require` export for sync Jiti aliases when one exists, while retaining existing ESM/subpath behavior through targeted fallback logic and regression tests.
