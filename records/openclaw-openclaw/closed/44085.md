## Summary

Current main already contains a Feishu-specific reply dispatcher that suppresses duplicate streamed/final sends. It disables block streaming for Feishu replies, tracks delivered final texts, and has targeted tests for both exact duplicate finals and the reasoning-preview-plus-final pattern described in the issue. The same implementation shipped in v2026.4.22.

## Evidence

- **Feishu dispatcher suppresses duplicate final replies:** The current Feishu reply dispatcher keeps a `deliveredFinalTexts` set, skips repeated final payloads, and records the final text after closing the streaming card. This directly prevents the second identical final send after a streamed card update.
  - file: [extensions/feishu/src/reply-dispatcher.ts:198](https://github.com/openclaw/openclaw/blob/4e4aeacae4a87f2fe09125d462d88cd71a18cb66/extensions/feishu/src/reply-dispatcher.ts#L198)

  - sha: [4e4aeacae4a8](https://github.com/openclaw/openclaw/commit/4e4aeacae4a87f2fe09125d462d88cd71a18cb66)
- **Feishu dispatcher disables block streaming for this channel:** The dispatcher returns `disableBlockStreaming: true`, and core reply directives treat that as block streaming off. That removes the extra block-level delivery path that can otherwise produce duplicate visible messages alongside the streamed/final card path.
