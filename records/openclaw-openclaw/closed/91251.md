## Label Justifications

- `P2`: The QMD root mismatch affects a real memory backend path but is limited to users who move configured workspaces or collection roots.
- `impact:session-state`: The stale QMD collection root can make memory recall resolve old files after the agent has moved to a new workspace.

## Summary

Keep open: current `main` and the latest release still take the incomplete-metadata path that can skip QMD collection rebinds when only the stored root path changed.

## What This Changes

The issue asks OpenClaw to rebind managed QMD memory collections when the configured collection root moves but the collection name and pattern stay the same.

## Best Possible Solution

Keep the issue open and fix memory-core so managed QMD collection metadata includes an authoritative path before deciding whether a same-name collection needs a safe remove/add rebind.

## Reproduction Assessment

Yes, from source inspection: when QMD list metadata has no path and the pattern matches, `shouldRebindCollection()` returns false, so `ensureCollections()` skips the rebind for a changed root.

## Solution Assessment

Yes, the proposed direction is the narrow maintainable fix: enrich listed managed collections with QMD path metadata before the rebind decision, while preserving the existing non-destructive behavior for truly unverifiable metadata.

In the memory-core QMD manager, make managed collection rebind detection use authoritative path metadata even when `qmd collection list` output omits paths. Keep the existing safety invariant: do not perform destructive remove/add from incomplete metadata unless the desired collection path is verified, and do not add new config or runtime fallback surfaces.
