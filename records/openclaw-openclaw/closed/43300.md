## Summary

Keep open. Current main has partial mitigations for trusted-proxy deployments, but headerless internal loopback self-connections still fail without a configured gateway password, and the remaining auth-model choice is security-sensitive.

## What This Changes

The issue requests a supported auth path for manager, subagent, and backend self-connections when Gateway uses trusted-proxy auth behind Traefik/authentik.

## Best Possible Solution

Resolve the remaining behavior through maintainer review of the trusted-proxy auth model, keeping proxy identity fail-closed while documenting password fallback or landing a first-class internal service identity path.

## Reproduction Assessment

Yes, from source inspection rather than a live container run. Current main still routes internal local Gateway calls to 127.0.0.1 and tests keep trusted-proxy loopback requests rejected when no password fallback is supplied.

## Solution Assessment

Unclear. The current password fallback is a narrow mitigation for configured deployments, but the exact headerless internal self-connection path remains fail-closed and the permanent service-identity or loopback-trust choice needs maintainer auth/security judgment.
