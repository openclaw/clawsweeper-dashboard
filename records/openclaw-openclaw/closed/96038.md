## Summary

Keep open for normal merge review: current main still has the unbounded Exa success JSON read, and this PR fixes that path with the shared bounded reader plus focused regression coverage and strong after-fix proof. No blocking findings; the main maintainer decision is accepting the new 16 MiB fail-closed cap for Exa success bodies.

## What This Changes

The PR bounds Exa web-search success JSON response reads with a 16 MiB cap and adds tests for valid small JSON plus oversized streamed JSON.

## Best Possible Solution

Land the Exa-specific success-body cap through the SDK-exposed response-limit helper and keep the 16 MiB provider-family limit as the default.

## Reproduction Assessment

Yes. Current main calls `response.json()` for Exa success bodies, and the PR body provides after-fix local HTTP-server proof that an oversized no-Content-Length response is capped and cancelled while a small response still parses.

## Solution Assessment

Yes. Using `readResponseWithLimit` through the plugin SDK is the narrow owner-boundary fix; using generic `readProviderJsonResponse` was considered, but that would change Exa's existing malformed-JSON error text unless the shared API grew new message-shaping options.
