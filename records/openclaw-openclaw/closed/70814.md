## Summary

Close as superseded: the linked cron/Telegram bug was already closed after current main and release `v2026.4.23` covered numeric IDs through explicit `delivery.to`, while this PR's unique fallback promotes malformed `delivery.channel` text into an explicit recipient and conflicts with target-policy semantics.

## What This Changes

The PR changes cron announce delivery resolution so a non-channel numeric `delivery.channel` value can become the explicit target when there is one configured announce channel, adds resolver regressions, and includes unrelated SwiftFormat suppression changes.

## Best Possible Solution

Keep the shipped explicit-target path and handle any legacy malformed stored jobs with validation or doctor migration that preserves target policy.

## Reproduction Assessment

Yes for the PR-introduced defect: source inspection shows the branch assigns invalid `delivery.channel` text to `explicitTo`, and current main applies `allowFrom` rerouting only in implicit mode. The original numeric `--to` path is already covered by current-main source and the closed linked issue.

## Solution Assessment

No. The narrower maintainable solution is to keep Telegram chat IDs in `delivery.to` or `--to` and reject malformed `delivery.channel`; runtime reinterpretation of the channel field is broader and less policy-safe.
