## Summary

Close as ClawHub/plugin-scope: current main has the generic SecretRef exec batching and manifest-backed plugin integration contract needed for external vault providers, so backend-specific batching and memoization belongs in installable resolver plugins rather than new OpenClaw core provider code.

## What This Changes

The issue requests a first-class efficient SecretRef pattern for external vault/entity-backed providers so one backend context can batch refs and memoize pass-scoped lookups instead of modeling each secret as its own exec command.

## Best Possible Solution

Keep OpenClaw core’s shipped generic SecretRef exec and pluginIntegration contracts stable, and publish backend-specific resolver plugins through ClawHub with pass-scoped memoization inside each resolver.

## Reproduction Assessment

Yes at source level: current core batching is by configured source/provider, so configs that model each vault value as a separate exec provider cannot be batched across those aliases. The shipped solution path is to use one backend-owned exec/plugin provider whose ids carry backend secret identity.

## Solution Assessment

No as a new core provider feature; the narrower maintainable solution is the shipped exec batching plus pluginIntegration contract, with backend-specific caching and lookup memoization owned by ClawHub/plugin resolvers.
