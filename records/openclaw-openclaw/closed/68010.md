## Summary

Keep open for maintainer/security review: the latest branch no longer implements the stated lazy compat-export fix and introduces provider auth/env contract regressions while still lacking real behavior proof.

## What This Changes

The branch rewrites `src/secrets/provider-env-vars.ts` to load plugin manifests directly, filter env var names, use null-prototype maps, copy lazy-proxy `get` values, drop auth-evidence lookup support, and adjust provider env-var lists.

## Best Possible Solution

Keep the narrow fix to making the compat export truly lazy with a post-import regression test; split provider env-secret trust hardening into an explicitly reviewed security/policy change that preserves setup `envVars`, `authEvidence`, and cache contracts.

## Reproduction Assessment

Yes for the PR defects: source inspection shows the branch removes exports still imported by current main and leaves the compat export eager. I did not run local tests because this review was required to keep the checkout read-only, but live CI is also failing relevant validation lanes.

## Solution Assessment

No. The current branch is not the narrow lazy-loading fix described by the PR body; it rewrites provider env-var resolution and security policy while breaking existing setup/auth-evidence contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Defer the compat export instead of loading manifests here:** `src/secrets/provider-env-vars.ts:114-118`
  - body: This resolver now calls `loadPluginManifestRegistry()`, while `src/agents/model-auth-env-vars.ts` still initializes `PROVIDER_ENV_API_KEY_CANDIDATES` eagerly at import time. Importing the compat module can still run manifest discovery before property access, so the fix needs to make that compat export delegate to a truly lazy proxy instead of moving the eager work here.
  - confidence: 0.9
- **[P1] Restore the auth-evidence exports before merging:** `src/secrets/provider-env-vars.ts:24-29`
  - body: The patch removes `ProviderAuthEvidence` and `resolveProviderAuthEvidence`, but current main still imports them through `src/agents/model-auth-env-vars.ts` and `src/agents/model-auth-env.ts`. This will fail typechecking and also drops the manifest `authEvidence` contract unless all callers and tests are migrated intentionally.
  - confidence: 0.96
- **[P2] Keep setup provider env vars in provider lookups:** `src/secrets/provider-env-vars.ts:125-127`
  - body: The new loop skips any plugin without `providerAuthEnvVars`, so plugins that use the preferred `setup.providers[].envVars` path are no longer included in `getProviderEnvVars()` or known secret/auth env lists. That breaks the documented manifest contract and existing dynamic coverage for setup provider env vars.
  - confidence: 0.94
- **[P2] Copy descriptor values from the lazy cache:** `src/secrets/provider-env-vars.ts:203-210`
  - body: The `get` trap returns defensive array copies, but `Object.getOwnPropertyDescriptor(PROVIDER_AUTH_ENV_VAR_CANDIDATES, id)?.value` still exposes the cached array itself. Callers can mutate shared candidate state through the descriptor path despite the intended hardening.
  - confidence: 0.88
- **[P3] Preserve the parameterless lookup cache:** `src/secrets/provider-env-vars.ts:251`
  - body: Changing `getProviderEnvVars()` to always call `resolveProviderEnvVars(params)` bypasses the existing `PROVIDER_ENV_VARS` lazy cache for ordinary parameterless lookups. That regresses the hot default lookup path that current code keeps cached unless lookup params are provided.
  - confidence: 0.82
