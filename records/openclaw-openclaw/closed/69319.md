## Summary

Keep open. Current main still lacks the requested RPC, but this branch is not mergeable against the current Gateway method registry and has source-reproducible blockers around destructive delete scope, runtime config handling, error classification, and missing real behavior proof.

## What This Changes

The PR adds a Gateway `skills.uninstall` RPC for ClawHub skill removal, including protocol schema/types, server registration, backend workspace deletion and lockfile cleanup, Swift models, docs, and tests.

## Best Possible Solution

Land a rebased, narrow Gateway uninstall RPC only after it is wired through the current core method descriptor registry, deletion is limited to ClawHub-originated or lockfile-tracked skills, client input errors are classified correctly, current protocol/docs/client artifacts are synced, and real behavior proof is added.

## Reproduction Assessment

Yes. The PR-introduced defects are source-reproducible from the diff and current main: the handler is not registered in the current descriptor registry, untracked `skills/<slug>` directories reach `fs.rm`, and malformed slugs rejected by backend normalization are returned as `UNAVAILABLE`.

## Solution Assessment

No. The requested RPC direction is reasonable, but this implementation needs a rebase plus scoped deletion, runtime-config, error-classification, descriptor-registration, and proof fixes before it is the narrow maintainable solution.

- **[P1] Register the handler in the core descriptor registry:** `src/gateway/server-methods/skills.ts:239`
  - body: On current main, core Gateway handlers are accepted only when listed in `CORE_GATEWAY_METHOD_SPECS`; otherwise registry construction throws `gateway method handler is missing a descriptor`. This PR adds a core handler but only updates the older method-list/scope files, so a rebase as-is breaks Gateway method registration.
  - confidence: 0.93
- **[P1] Refuse untracked skill folders:** `src/agents/skills-clawhub.ts:464-466`
  - body: `uninstallSkillFromClawHub` deletes `targetDir` whenever the directory exists, even when there is no `.clawhub` origin metadata or lockfile entry. A slug collision with a manually managed or uploaded skill can therefore remove content outside the documented ClawHub uninstall scope.
  - confidence: 0.92
- **[P2] Use the request runtime config:** `src/gateway/server-methods/skills.ts:254-255`
  - body: The new handler calls `loadConfig()` directly while adjacent skills handlers use `context.getRuntimeConfig()`. That can resolve the persisted/default workspace instead of the active Gateway runtime config for this request.
  - confidence: 0.86
- **[P2] Map invalid uninstall slugs to INVALID_REQUEST:** `src/gateway/server-methods/skills.ts:278`
  - body: The schema accepts any non-empty slug, so malformed values such as `../evil` pass validation and fail in backend normalization. Mapping every backend failure to `UNAVAILABLE` misclassifies client input errors as server outages.
  - confidence: 0.88
- **[P3] Report lockfile-only cleanup accurately:** `src/agents/skills-clawhub.ts:472-474`
  - body: When the directory is gone but the lockfile still contains the slug, the helper removes state while returning `removed: false`, causing the handler to say the skill was not installed even though cleanup happened.
  - confidence: 0.74
