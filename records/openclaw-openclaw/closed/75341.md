## Summary

Keep open. Current main only partially mitigates the reported Discord startup regression: it can skip the application-ID REST lookup via config or token parsing, but native command deploy still retries 429s during startup and bot identity lookup still fails startup when `fetchUser("@me")` times out.

## What This Changes

The issue reports a Discord v2026.4.29 startup regression where OAuth identity preflight requests time out and native slash-command deployment retries hit Discord REST 429s during gateway startup.

## Best Possible Solution

Fix this in the Discord plugin startup path: avoid startup REST amplification by treating native command deploy 429s as a non-fatal single-attempt skip/cooldown, and make bot identity lookup degrade through a trusted parsed/configured bot id when possible while preserving secure mention gating.

## Reproduction Assessment

Yes. Mocking `client.deployCommands` to throw a Discord 429 reproduces the current three-attempt deploy burst, and mocking `client.fetchUser("@me")` to reject reproduces the hard startup failure before lifecycle.

## Solution Assessment

No. Current main's application-ID bypass and quieter deploy logging help, but they do not remove the startup 429 retry burst or the hard `/users/@me` dependency that the report identifies.
