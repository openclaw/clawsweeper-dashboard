## Best Possible Solution

## Solution Assessment

No. The central feature should be handled by the canonical env-marker PR with trusted override semantics and proof; this branch is narrower, duplicates the cluster, and adds `OPENCLAW_SESSION_ID` without a settled public contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Scrub stale session markers before spawning exec:** `src/agents/bash-tools.exec-runtime.ts:570-571`
  - body: Because `shellRuntimeEnv` starts from `...opts.env` and only conditionally overwrites the new markers, inherited or caller-provided `OPENCLAW_SESSION_KEY` / `OPENCLAW_SESSION_ID` values survive when the current context lacks one of them. Strip these reserved names before re-injecting normalized current values so spawned commands cannot see stale or spoofed session context.
  - confidence: 0.89

## Security Review

Status: needs_attention

Summary: The diff introduces a low-severity trusted-context integrity risk by allowing stale or caller-supplied session markers to survive into exec children.

Concerns:

- **[low] Stale session markers can leak through env inheritance:** `src/agents/bash-tools.exec-runtime.ts:570`
  - body: The new markers may be consumed as routing context, but this patch only conditionally overwrites values after spreading inherited/requested env. A stale or user-provided `OPENCLAW_SESSION_KEY` or `OPENCLAW_SESSION_ID` can therefore reach a spawned command when the current value is absent.
  - confidence: 0.84
