## Summary

Close as implemented on current main. The post-v2026.5.2 mainline now has targeted fixes for both cold `models.list` and `sessions.list`: model listing uses a read-only persisted catalog fast path with a bounded configured-view fallback, and session listing uses lightweight rows plus bounded transcript reads, matching the reported first-call latency root paths.

## What This Changes

The issue reports cold first-call `models.list` and `sessions.list` WebSocket latency after a Gateway restart on v2026.5.2 and asks for prewarming, documentation, or observability around that cold-start gap.

## Best Possible Solution

Keep the current read-only catalog and lightweight session-list implementation, and ask for a fresh report only if a build containing these mainline fixes still shows >20s first-call latency.

## Reproduction Assessment

No live current-main reproduction was run. The v2026.5.2 report plus follow-up fix commits identify the old cold catalog/session row hot paths, and current main has targeted tests guarding the nonblocking list behavior.

## Solution Assessment

Yes. Fixing the request paths with bounded read-only/lightweight behavior is narrower than adding background prewarm and avoids shifting heavy work into post-ready startup while solving the reported >20s first-call waits.
