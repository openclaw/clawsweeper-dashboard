## Summary

Current main already handles the observable OpenRouter alias problem: OpenClaw resolves arbitrary OpenRouter-style refs, uses the OpenRouter provider stream path, and the pinned pi-ai stream implementation keeps assistant messages on the requested model id instead of replacing them with OpenRouter's provider-resolved id. This shipped in v2026.4.24.

## Best Possible Solution

Close this issue and keep the shipped OpenRouter provider plus pi-ai 0.70.2 stream behavior. If a future report still reproduces on v2026.4.24 or newer, it should include a fresh OpenRouter stream trace or payload evidence because the current alias/model-id path no longer matches the 2026.4.15 failure mode.
