## Summary

Keep #69735 open. Current main still forwards native Ollama response tool-call arguments unchanged in `buildAssistantMessage()`, while the existing normalization and tests cover replay/compat paths rather than the reported response assembly path. The related fixing PRs in the provided context are closed unmerged or still need maintainer handling, so the issue is not implemented on main.

## Best Possible Solution

Keep the issue open until the Ollama plugin normalizes `response.message.tool_calls[].function.arguments` inside `buildAssistantMessage()` with the existing unsafe-integer-preserving parser, updates the local `OllamaToolCall` typing if needed to reflect string-or-object runtime input, and adds a focused regression test for string-valued native Ollama response arguments. If #69910 is still viable, use it as the implementation path; otherwise land an equivalent narrow fix in `extensions/ollama/src/stream.ts` with tests in the Ollama stream test suite.
