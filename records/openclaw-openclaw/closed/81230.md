## Summary

Keep open: the PR fixes the current-main wildcard discovery bug in the right shared provider seams and has sufficient terminal proof, but it should not merge yet because the new config-backed auth path drops valid literal all-caps API keys.

## What This Changes

The PR enables vLLM/SGLang provider wildcard discovery from explicit configured endpoints, updates provider-filtered model listing, expands auth/discovery tests, and documents the wildcard pattern.

## Best Possible Solution

Land one canonical fix that keeps exact configured entries manual-only, enables wildcard discovery from configured vLLM/SGLang endpoints, and preserves existing literal API-key semantics.

## Reproduction Assessment

Yes. Current main has a high-confidence source reproduction: explicit `models.providers.vllm` or `models.providers.sglang` makes the shared helper return null before wildcard discovery can query the configured endpoint.

## Solution Assessment

Not yet. The shared-seam approach is the right shape, but the auth resolver needs to distinguish explicit env refs or known markers from arbitrary all-caps plaintext before this is safe to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve literal all-caps configured API keys:** `src/agents/models-config.providers.secrets.ts:218-227`
  - body: This branch treats any `models.providers.<id>.apiKey` matching the all-caps env-var regex as an env reference and returns no key when the env var is absent. Current secret handling intentionally treats arbitrary all-caps values such as `ALLCAPS_SAMPLE` as plaintext, so a valid self-hosted provider key in that shape would now skip wildcard discovery. Please limit env resolution to explicit `${VAR}`/known markers or otherwise fall back to the configured literal.
  - confidence: 0.89
