## Summary

Keep open, but this patch should not merge as-is: it misses the real `tools.alsoAllow` input shape and bypasses the manifest auth/config availability gate. The linked issue was already closed as external plugin manifest work, and the PR has no after-fix real behavior proof.

## What This Changes

The branch changes plugin tool runtime selection to load explicitly allowlisted plugin tool runtimes without manifest availability checks, adds optional-tool tests, and adds a changelog entry.

## Best Possible Solution

Keep manifest-backed availability as the default; if a contract-compliant plugin still fails in isolated sessions, add a narrow runtime/auth context contract with real cron or subagent proof instead of bypassing `toolMetadata` availability.

## Reproduction Assessment

Yes for the source-level filtering path: current main requires `hasManifestToolAvailability` before loading manifest-declared plugin tools. No for the linked live WeCom failure, which was not reproduced here and was already closed as an external plugin manifest issue.

## Solution Assessment

No. The patch does not match actual `tools.alsoAllow` inputs and weakens the manifest auth/config availability boundary; a plugin manifest fix or a narrower core contract would be more maintainable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Handle the real alsoAllow allowlist shape:** `src/plugins/tools.ts:447`
  - body: This branch claims to fix `tools.alsoAllow`, but real `alsoAllow`-only policies are collected with the `__openclaw_default_plugin_tools__` sentinel. Because this condition rejects any allowlist that includes default plugin tools, the new bypass will not run for the reported `agents.list[].tools.alsoAllow: ["wecom_mcp"]` shape; the added test uses `toolAllowlist: ["x_search"]`, which does not exercise the actual `alsoAllow` pipeline.
  - confidence: 0.95
- **[P2] Keep auth-gated plugin tools behind availability checks:** `src/plugins/tools.ts:493-496`
  - body: Adding the plugin before `hasManifestToolAvailability` means a manifest-declared tool with `authSignals` or `configSignals` becomes callable when the operator names it, even if there is no env/config/auth evidence. Current tests and the manifest availability helper intentionally keep those tools hidden, and the linked WeCom issue was closed because the external plugin omitted `contracts.tools`, not because OpenClaw should ignore availability metadata.
  - confidence: 0.9
