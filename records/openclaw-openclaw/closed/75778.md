## Summary

Close as implemented on current main. The report targets v2026.4.29; current main preserves the Telegram IPv4 fallback behavior, bounds Telegram control-call retries, and also fixes the logged Gateway-side `models.list`/session hot paths that could delay timers and make Telegram fallback appear minutes late.

## What This Changes

The issue asks OpenClaw to prevent multi-minute Gateway stalls on IPv6-disabled or unroutable networks by making Telegram Bot API fallback to IPv4 quickly without blocking the event loop.

## Best Possible Solution

Keep the current main behavior, ship it in the next release, and treat any reproduction after that as a fresh current-main diagnostic report rather than reopening the v2026.4.29 failure signature.

## Reproduction Assessment

Yes for the reported v2026.4.29 signature: the issue provides concrete logs for delayed `models.list`, event-loop liveness delay, and Telegram fallback promotion. I did not live-reproduce Home Assistant OS, but the release-to-main source diff matches the delayed Gateway RPC path that would also delay fallback timers.

## Solution Assessment

Yes. The maintainable fix is to bound Gateway model/session/plugin hot paths while preserving Telegram's transport-level autoSelectFamily and sticky IPv4 fallback, rather than weakening reserved `NODE_OPTIONS` filtering for service environment overrides.
