## Summary

Current `main` already has shipped guardrails for this failure mode: oversized tool results are capped before persistence, replayed image payloads are sanitized before provider submission, and embedded runs now preflight prompt size so they truncate tool-result tails and/or compact before sending an oversized request upstream.

## Evidence

- **Persisted tool results are hard-capped before they enter session history:** `installSessionToolResultGuard()` truncates `toolResult` payloads before persistence so oversized tool output cannot dominate later provider requests. The default live cap is currently 16k chars, with per-agent/config overrides via `toolResultMaxChars`.
  - file: [src/agents/session-tool-result-guard.ts:27](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/src/agents/session-tool-result-guard.ts#L27)
- **Embedded runs preflight prompt size and recover before provider submission:** `run/attempt.ts` computes a preemptive overflow route before prompt submission; when reducible tool-result tail content can absorb the overflow it truncates those results immediately, otherwise it compacts and retries. `shouldPreemptivelyCompactBeforePrompt()` explicitly accounts for aggregate tool-result reduction potential, not just a single oversized entry.
  - file: [src/agents/pi-embedded-runner/run/attempt.ts:2276](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/src/agents/pi-embedded-runner/run/attempt.ts#L2276)
- **Recovery logic is covered by regression tests for truncate-only and compact-then-truncate paths:** Current tests verify retry success after fallback tool-result truncation, mixed oversized-plus-aggregate tool-result tails, and post-compaction truncation before retry.
  - file: [src/agents/pi-embedded-runner/run.overflow-compaction.loop.test.ts:144](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/src/agents/pi-embedded-runner/run.overflow-compaction.loop.test.ts#L144)
- **Replay history sanitizes image attachments before provider calls:** Current transcript hygiene docs and implementation say image payloads are always sanitized to prevent provider-side rejection from oversized base64 images, and `sanitizeSessionMessagesImages()` applies that to user/toolResult history before replay.
  - file: [src/agents/pi-embedded-helpers/images.ts:40](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/src/agents/pi-embedded-helpers/images.ts#L40)
- **This behavior shipped after 2026.3.13:** Release notes show follow-up fixes in `2026.4.5` (`Agents/tool results: cap live tool-result persistence and overflow-recovery truncation`) and `2026.4.7` (`Agents/context overflow: combine oversized and aggregate tool-result recovery in one pass and restore a total-context overflow backstop`).
  - file: [CHANGELOG.md:1107](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/CHANGELOG.md#L1107)

## Risks / Open Questions

- I did not run an end-to-end reproduction in this read-only review; the conclusion is based on current code, docs, regression tests, and shipped changelog entries.
- The visible context indicator may still be an estimate rather than a perfect predictor of the next assembled request, but current main now has the embedded-run preflight/backstop that prevents this reported overflow mode from reaching the provider in the same way.
