## Summary

Keep open. Current main still has the reported initialization-order bug in the native `sessions_spawn` preparation path, the latest release still contains the same bare resolve path, and open PR #73904 already references this issue with closing syntax, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue requests initializing built-in context engines before `prepareContextEngineSubagentSpawn()` resolves the configured context engine so cliBackend-only cold starts can use `sessions_spawn`.

## Best Possible Solution

Review and land PR #73904, or apply the same narrow initialization guard in the native subagent spawn preparation path, then close this issue after merge. Keep the CLI compaction sibling tracked through #75129 unless maintainers choose to combine the fixes deliberately.

## Reproduction Assessment

Yes. A cold cliBackend-only process can enter `sessions_spawn` through `spawnSubagentDirect()` and reach `prepareContextEngineSubagentSpawn()` before any embedded PI path initializes the context-engine registry, which causes the exact missing-legacy-engine error.

## Solution Assessment

Yes. Initializing built-in context engines at the subagent spawn preparation boundary is the narrowest maintainable fix for this path; making registration a broad import side effect would be riskier and unnecessary for this issue.
