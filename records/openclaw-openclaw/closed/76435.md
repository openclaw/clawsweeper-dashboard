## Summary

Keep open. Current main still returns OpenAI attribution headers for the browser-side Realtime WebRTC SDP offer and the Control UI forwards them to `api.openai.com`; a live CORS preflight for that endpoint allows only `authorization,content-type`. The linked closing PR #76457 is open, so this issue should remain open until the fix is reviewed and merged.

## What This Changes

Remove browser-disallowed OpenAI attribution headers from the Control UI Talk WebRTC SDP POST while keeping Gateway-side OpenAI attribution intact.

## Best Possible Solution

Review and land a narrow fix, likely through #76457 after ensuring all browser-disallowed attribution headers are removed, with provider/UI regression coverage and a user-facing changelog entry.

## Reproduction Assessment

Yes. Current source returns `originator`/`version` in browser `offerHeaders`, the UI forwards those headers to the direct SDP `fetch`, and a live preflight for the exact OpenAI endpoint rejects them by omission from `Access-Control-Allow-Headers`.

## Solution Assessment

Yes, with one caveat: the narrow maintainable fix is to remove OpenAI attribution headers from the browser SDP offer response path while leaving Gateway-side attribution intact. The open PR should be checked for both `originator` and `version`, because OpenAI's preflight currently allows only `authorization,content-type`.
