## Label Justifications

- `P1`: Remote iMessage attachments can reach Codex as unreadable Mac filesystem paths, breaking a real channel-to-agent media workflow.
- `impact:message-loss`: The text message can arrive while the attached image content is effectively lost to the agent because the primary media path is unreachable.

## Summary

Keep open: current main still derives plugin-bound Codex media metadata from the raw remote iMessage context before dispatch-side remote-cache staging, and the linked closing PR is open rather than merged.

## What This Changes

Ensure remote iMessage attachment staging propagates the staged local cache path into final Codex-visible attachment metadata instead of exposing the raw macOS Messages path.

## Best Possible Solution

Review, repair if needed, and land https://github.com/openclaw/openclaw/pull/91803 or an equivalent dispatch-layer fix that stages remote iMessage media before plugin-bound Codex metadata is constructed.

## Reproduction Assessment

Yes, from source inspection. Remote iMessage can emit raw MediaPath(s) with MediaRemoteHost, dispatch builds plugin-bound Codex claim metadata before staging, and Codex reads localImage paths from the local filesystem.

## Solution Assessment

Yes. The best fix is to reuse OpenClaw remote media staging before plugin/Codex metadata construction; teaching Codex about channel-specific raw macOS Messages paths would put the repair in the wrong layer.
