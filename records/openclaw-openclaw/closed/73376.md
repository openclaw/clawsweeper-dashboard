## Summary

Current main has post-2026.4.26 fixes for the reported CLI hang class: default text status no longer performs live channel/status runtime work unless `--deep`/`--all` is requested, and Gateway RPC/probe clients now wait for event-loop readiness under the caller timeout before opening WebSocket connections. `cron list` is a thin Gateway RPC command, so it benefits from the same Gateway client fix.

## What This Changes

The issue reports that `openclaw status` and `openclaw cron list` hang on OpenClaw 2026.4.26 while `openclaw --version`, `openclaw health`, Gateway health, Dashboard, and chat functionality still work.

## Best Possible Solution

Keep the current main implementation: plain status should remain metadata-oriented by default, deeper live channel checks should stay behind `--deep`/`--all`, and Gateway-backed CLI commands should continue sharing the readiness-aware Gateway client path. If a user still sees a hang on a build after these commits, it should be filed as a new report with `--timeout`, Gateway logs, and whether the command stalls before or after WebSocket connect.

## Reproduction Assessment

Yes, at the symptom level: the issue gives concrete affected commands, version 2026.4.26, WSL2 environment, and passing health/dashboard checks. I did not run the WSL2 live reproduction during this read-only cleanup, but current regression tests cover the fixed status fast path and Gateway client readiness timeout path.

## Solution Assessment

Yes. The current solution is the narrow maintainable fix: it removes live channel/runtime work from plain status and fixes the shared Gateway client/probe startup timing used by status and cron RPC commands, without changing cron semantics or broad CLI dispatch behavior.
