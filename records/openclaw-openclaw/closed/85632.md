## Summary

Current main and the shipped v2026.6.6 release now cover the reported cron isolated announce TTS path: cron delivery applies the normal TTS transform with the resolved channel/account, speech-core marks Feishu/Xiaomi voice-note output with audioAsVoice, and Feishu routes voice-intent audio through Ogg/Opus native audio delivery.

## What This Changes

The issue asks for isolated cron announce delivery to preserve channel-derived TTS voice-note intent instead of sending generated audio as a raw file attachment.

## Best Possible Solution

Keep the current shipped cron TTS and Feishu voice-note implementation; if this still reproduces on v2026.6.6 or newer, reopen with a live Feishu cron log showing where audioAsVoice or transcoding is lost.

## Reproduction Assessment

No current-main failing reproduction was established; source and regression evidence now show the cron announce path applying TTS before delivery and Feishu receiving audioAsVoice-capable media. The original symptom is plausible on older or mismatched builds, but the current observable path is covered.

## Solution Assessment

Yes: the current solution is the maintainable one because cron reuses the shared TTS transform and channel/plugin capabilities decide voice-note routing, rather than adding a Feishu-specific cron special case.
