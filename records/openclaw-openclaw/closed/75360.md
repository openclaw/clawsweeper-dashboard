## Summary

Keep open. The PR is aimed at a real hot-path concern, but the current head still introduces provider contract regressions in `web_search` registration and media generation tool availability, and the protected `maintainer` label means this needs normal maintainer/contributor review rather than cleanup closure.

## What This Changes

The PR caches targeted provider hook resolution, defers web and media provider/model resolution from registration toward execution, makes typing indicator starts asynchronous, updates tests, and adds a Telegram reply latency changelog entry.

## Best Possible Solution

Revise the latency optimization so registration can stay cheap while still sourcing provider-owned web search descriptors and media generation availability from lightweight provider metadata or an equivalent contract-preserving fast path, then rerun the targeted tool/provider/channel tests and the changed gate before merge.

## Reproduction Assessment

Partially. I did not live-reproduce the Telegram RTT claim, but the review findings are reproducible by comparing the PR head to current-main registration tests and provider contracts for `web_search` descriptors and generation provider readiness.

## Solution Assessment

No. Targeted provider runtime lookup and async typing are reasonable directions, but replacing provider descriptors with a fixed schema and using a generic auth-only generation availability check is not the narrowest maintainable fix.
