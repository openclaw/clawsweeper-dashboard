## Label Justifications

- `P1`: The report describes a broken real channel workflow where Feishu users receive no reply while the response is delivered to WebChat.
- `impact:message-loss`: Replies are effectively lost from the originating Feishu conversation because they are routed to the wrong channel.

## Summary

Keep open: current main still has a source-visible route-precedence gap for sessions_send announce routing when a stored session row points at webchat while its explicit deliveryContext points at Feishu, and recent issue comments corroborate the Feishu/WebChat failure on newer releases. The related Feishu reports are closed back to this canonical thread, and no replacement PR was found in the targeted live PR search.

## What This Changes

The issue reports sessions_send replies from Feishu DM routing to webchat instead of the originating Feishu channel.

## Best Possible Solution

Make explicit persisted external deliveryContext authoritative for sessions_send announce/direct delivery when stored session route fields point at stale or internal webchat, with generic regression coverage rather than a Feishu-specific branch or new config flag.

## Reproduction Assessment

Yes. By source inspection, seed a sessions.list row for a Feishu direct session with stored webchat route fields and deliveryContext set to Feishu; deliveryContextFromSession still normalizes the stored route first, so resolveAnnounceTarget cannot reliably return the Feishu target.

## Solution Assessment

Yes. The best fix is a narrow generic routing precedence repair in session delivery-context normalization or announce-target resolution, with tests for Feishu direct/group cases and existing webchat route-preservation invariants.
