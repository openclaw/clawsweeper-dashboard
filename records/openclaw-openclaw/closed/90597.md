- `impact:session-state`: The reported provider error depends on replaying stored conversation history with incompatible tool-call/tool-result state.
- `impact:auth-provider`: The user-visible failure occurs after provider/model routing switches the session to a DeepSeek OpenAI-compatible model.

## Summary

Current main and the latest release already repair this class of model-switch/tool-history replay failure: non-raw agent turns sanitize session history before provider replay, and the shared repair drops orphan `toolResult` entries before an OpenAI-compatible provider can receive a `role: "tool"` message without a preceding assistant `tool_calls` entry.

## What This Changes

The issue asks OpenClaw to handle switching an existing tool-using conversation from Auto mode to a DeepSeek OpenAI-compatible model without sending incompatible or orphaned tool-call history.

## Best Possible Solution

Keep the shipped replay repair path and ask for a fresh report only if the same error reproduces on v2026.6.1 or newer with a redacted session/history sample.

## Reproduction Assessment

No for current main: source inspection shows non-raw model attempts now sanitize session history and drop orphan `toolResult` entries before provider replay. The reported old-version behavior is plausible, but the current source and v2026.6.1 release no longer match that failure path.

## Solution Assessment

Yes: the existing shared transcript repair is the best layer because it fixes replay pairing for provider switches without adding another model schema field or clearing whole sessions.
