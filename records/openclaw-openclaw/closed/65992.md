## Summary

Keep this PR open: current main still has a source-level same-slug overwrite path, but the branch needs contributor real behavior proof and a compatibility decision because it changes every new Memory Wiki source identity to a hash-suffixed filename.

## What This Changes

The PR changes Memory Wiki ingest to derive source page IDs and paths from a readable slug plus canonical source-path hash, reuse legacy source pages by source path, and add focused ingest regression tests.

## Best Possible Solution

Preserve the collision-resistant source identity fix while keeping legacy slug-only output for non-colliding new ingests, or explicitly document and accept the hash-suffixed identity contract before merge.

## Reproduction Assessment

Yes at source level: current main computes `pageId` and `pageRelativePath` solely from `slugifyWikiSegment(title)`, so two different files with titles like `hello world` and `hello-world` write the same source page. I did not execute the CLI because this review is read-only.

## Solution Assessment

Not yet. Collision-resistant identity is the right direction, but unconditionally suffixing all new source pages is broader than necessary unless maintainers explicitly accept that compatibility change.
