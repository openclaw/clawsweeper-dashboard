## Summary

Keep open. The PR targets a real docs gap for npm-installed users, but it is not merge-ready because the new inline Docker build fallback uses a mutable Debian base tag while the checked-in sandbox Dockerfile and tests enforce digest-pinned bases.

## What This Changes

The PR updates sandboxing-related docs and the changelog to clarify that sandbox setup scripts require a source checkout and to add npm-install guidance for building sandbox images.

## Best Possible Solution

Keep the source-checkout clarification, but make the npm-install fallback preserve the same supply-chain posture as `Dockerfile.sandbox`, preferably by using the exact digest-pinned `FROM` reference or by pointing users to an equivalent pinned build path.

## Reproduction Assessment

Yes. Current main documents source-only setup scripts, the npm package allowlist does not publish them, and the runtime missing-image error names the same script-only build path, which is enough source-level evidence for the npm global install failure described in #75485.

## Solution Assessment

No, not as currently written. A docs-only fix is the right narrow direction, but the inline Dockerfile should match the repository's digest-pinned sandbox base before this is the best solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Pin the inline sandbox base image:** `docs/gateway/sandboxing.md:386`
  - body: The new npm-install build fallback is an alternative path for creating the default sandbox image, but it starts from mutable `debian:bookworm-slim`. The checked-in `Dockerfile.sandbox` pins this base by sha256 and `src/docker-image-digests.test.ts` enforces pinned Dockerfile bases, so users following this docs path get a less reproducible supply-chain path than source-checkout users. Use the same digest-pinned `FROM` line or avoid inlining the Dockerfile.
  - confidence: 0.88
