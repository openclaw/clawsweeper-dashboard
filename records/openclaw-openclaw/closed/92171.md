## Label Justifications

- `P2`: Slack wake replies escaping an active thread are workflow-disrupting but scoped to channel delivery rather than core availability.
- `impact:message-loss`: Replies could land in the Slack channel root instead of the active thread, making updates easy to miss.
- `impact:session-state`: The report is about losing active thread context across subagent, yield, and cron wake turns.

## Summary

The reported 2026.6.1 Slack threading failure is already covered by current main and the v2026.6.5 release: subagent/internal completion routing now preserves thread context and message-tool sends infer the active Slack thread.

## What This Changes

Slack internal wake/resume turns should keep message sends in the active thread instead of posting to the channel root.

## Best Possible Solution

Keep the generic delivery-context propagation shipped in v2026.6.5; users still on 2026.6.1 should upgrade to pick up the thread-routing fix.

## Reproduction Assessment

Yes for the reported 2026.6.1 behavior from the issue body and source path, but no failing current-main run was established. Current main and v2026.6.5 source/tests show the relevant thread context is now propagated.

## Solution Assessment

Yes. The current implementation fixes this at the generic delivery-context and message-tool ambient-thread layers rather than adding a Slack-only workaround.
