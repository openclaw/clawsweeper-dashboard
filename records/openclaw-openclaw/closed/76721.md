## Summary

Current main already fixes the released CLI scope-selection failure: `devices approve` now derives the needed operator scope from the exact pending request, has regression coverage, and documents when admin is still intentionally required. The proof SHA is on main after v2026.5.2 and is not contained in a shipped release tag yet.

## What This Changes

The issue reports `openclaw devices approve <requestId>` failing in Docker on v2026.5.2 with `GatewayClientRequestError: missing scope: operator.admin`.

## Best Possible Solution

Keep the target-derived CLI approval scope logic and ship it in the next release while preserving admin enforcement for approvals that mint or inherit admin access.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source inspection gives a clear path, though I did not run the Docker compose flow: the released approval path could reach the server with only pairing scope while the approval-time guard required admin for admin-scoped requests.

## Solution Assessment

Is this the best way to solve the issue? Yes. Deriving scopes from the exact pending request is narrower than making all device approvals admin-scoped and matches the documented Gateway scope contract.
