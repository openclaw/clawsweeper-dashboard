## Summary

Keep open. The patch targets a real current-main Feishu gap, but it is not merge-ready because starter/root media still remains as parsed placeholder text, history media downloads are unbounded, and the required user-facing changelog entry is missing.

## What This Changes

The PR preserves raw content on fetched Feishu thread messages, resolves media-typed history items into saved-path attachment hints, and adds a happy-path bot test.

## Best Possible Solution

Land a bounded Feishu-owned fix that preserves rawContent for fetched starter and history messages, resolves only visible media after sender filtering, caps historical media work, keeps the existing attachment-hint contract, adds focused tests, and updates CHANGELOG.md.

## Reproduction Assessment

Yes. A source-level reproduction is to mock a Feishu thread history image whose parsed content is [image message]; current main formats that placeholder into ThreadHistoryBody. A root/starter image is also reproducible because getMessageFeishu returns parsed content without rawContent and the root is excluded from listFeishuThreadMessages.

## Solution Assessment

Partly. Reusing resolveFeishuMediaList and the existing attachment hint format is the right direction, but the patch needs starter coverage and aggregate caps before it is the narrow maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

  - body: The new resolver only runs over historyMessages, but ThreadStarterBody still comes from rootMsg?.content and getMessageFeishu does not expose rawContent.   - confidence: 0.84
