## Summary

Keep open. Current `main` still performs the reported key-only follow-up queue cleanup, and PR #68839 is already the open closing implementation candidate for the same race.

## What This Changes

The issue reports that a late follow-up drain can delete a replacement `FOLLOWUP_QUEUES` entry by key instead of checking queue identity.

## Best Possible Solution

Review and land or replace PR #68839 with an identity guard around both map deletion and callback cleanup plus a focused regression test.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: park D1 in an awaited follow-up run, clear the session queues via `/stop`, enqueue Q2 for the same key, then let D1's `finally` delete the current map entry by key.

## Solution Assessment

Yes. Guarding late cleanup with `FOLLOWUP_QUEUES.get(key) === queue` is the narrowest maintainable fix; current `main` lacks that guard, and the linked PR is the canonical review path.
