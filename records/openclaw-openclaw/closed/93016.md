## Label Justifications

- `P2`: This is a normal-priority status/task performance fix with limited blast radius, but the proposed patch has merge-blocking correctness issues.
- `merge-risk: 🚨 session-state`: The PR can return stale task registry state after task mutations because it caches reconciled task arrays without lifecycle invalidation.

## Summary

Keep open, but this PR is not merge-ready: the proposed cache is timestamped before the expensive reconciliation so it can expire before the duplicate large-ledger call it is meant to save, and it adds a global task snapshot cache without task-mutation invalidation.

## What This Changes

The PR adds a module-level 5 ms cache inside `reconcileInspectableTasks()` so immediate duplicate callers can reuse the previous reconciled task array.

## Best Possible Solution

Replace the module-level TTL cache with a per-`getStatusSummary()` shared reconciled task snapshot, or add lifecycle-owned invalidation if maintainers explicitly want cross-call caching.

## Reproduction Assessment

Yes by source inspection: current main still calls two status task helper paths that each call `reconcileInspectableTasks()` and reach the clone-and-sort `listTaskRecords()` path. I did not run a live large-ledger benchmark in this read-only review.

## Solution Assessment

No. A scoped shared snapshot inside `getStatusSummary()` is narrower and safer than a module-level 5 ms cache over mutable task registry state.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Stamp the cache after reconciliation:** `src/tasks/task-registry.maintenance.ts:901`
  - body: This stores `time: now` from before `listTaskRecords().map(...)` runs. On the reported 10K-task workload, the first reconciliation is hundreds of milliseconds, so the cached result can already be older than the 5 ms TTL before `getStatusSummary()` makes the second helper call, leaving the central performance bug unfixed. Scope the reuse to the status call or timestamp after the result is computed.
  - confidence: 0.92
- **[P2] Invalidate cached task snapshots on mutations:** `src/tasks/task-registry.maintenance.ts:886-887`
  - body: `reconcileInspectableTasks()` is shared by status, task audit, task maintenance, and active restart-blocker checks, but this module-level cache is never invalidated when task records are created, updated, restored, or deleted. A caller that reconciles, mutates the registry, then asks for a summary or restart blockers within the TTL can read the old array and miss the new task state. Keep the cache scoped to one `getStatusSummary()` call or wire it to the existing task-registry observer events with tests.
  - confidence: 0.9
