## Summary

Keep open. Current main still has the reported class of failure: the status account-note path re-enters Discord's config allowlist accessor, that accessor resolves the full Discord account, and full account resolution strictly normalizes the token, which throws on an unresolved SecretRef. The docs say read-only status paths should degrade instead of aborting, and I did not find an exact open duplicate or linked fix PR in the supplied context or targeted search.

## What This Changes

The issue asks OpenClaw to stop `openclaw gateway restart`/status scanning from crashing when Discord account tokens are configured as env SecretRefs and should be read from the resolved runtime snapshot or degraded read-only state.

## Best Possible Solution

Make the Discord read-only allowFrom/defaultTo accessor token-free, or make account-note rendering rely on already-inspected snapshot fields without re-running strict plugin account resolution. Add focused regression coverage for `status --all`/status scanning with `channels.discord.accounts.*.token` as an env SecretRef, preserving strict runtime token behavior elsewhere.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/74765. Current main can throw `channels.discord.accounts.*.token: unresolved SecretRef ...` during status/gateway restart scanning because `src/commands/status-all/channels.ts` builds account notes by calling `plugin.config.resolveAllowFrom`, and Discord's config accessor in `extensions/discord/src/shared.ts` resolves the full account/token before reading allowFrom/defaultTo. Implement a narrow fix so read-only status/config accessors do not strictly resolve Discord tokens just to format allowFrom/defaultTo. Prefer making the Discord accessor derive accountId, allowFrom, and defaultTo from merged/raw account config without calling `resolveDiscordAccount`; alternatively make the status note renderer use safe inspected/snapshot fields and catch unavailable read-only accessors. Preserve strict token resolution for gateway startup, runtime sends, REST clients, and monitor paths. Add regression coverage for a Discord multi-account config with `channels.discord.accounts.<id>.token` as `{ source: "env", provider: "default", id: "DISCORD_BOT_TOKEN" }` proving status account note/table generation does not throw and still shows configured/unavailable SecretRef state when unresolved. Also keep or add coverage that `resolveDiscordToken` still throws for unresolved SecretRefs in strict runtime paths. Changelog: add a single-line user-facing fix entry with `Thanks @PrinceOfEgypt` if behavior changes. Do not change dependency versions, release/version metadata, SecretRef provider activation semantics, or unrelated Discord runtime behavior.
