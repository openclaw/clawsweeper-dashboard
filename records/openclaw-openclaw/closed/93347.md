Keep this draft PR open, but it is not merge-ready: source review found a blocking run-scoping bug that can drop intended client restrictions for an active session, and the new public config key still needs docs/help metadata plus real gateway proof.

## What This Changes

The branch adds `gateway.tools.byClientId` schema/types, a session-key client-id registry, loopback MCP tool filtering/cache keying, and focused unit tests.

## Best Possible Solution

Bind the client id to the active agent run/lifecycle instead of a session-wide TTL map, document the new config contract, and require redacted real gateway proof before merge.

## Reproduction Assessment

Yes for the review finding from source: the patch writes a session-level client id before current active-run dedupe can return an existing run, and the dedupe key has no client-id component. I did not run a live gateway scenario in this read-only review.

## Solution Assessment

No: a session-key TTL registry is not the best security-boundary shape for per-client tool policy. The safer design is run-lifecycle binding plus docs/metadata for the new public config contract.
