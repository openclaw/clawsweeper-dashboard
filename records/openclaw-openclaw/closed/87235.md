## Label Justifications

- `P2`: The report describes a bounded provider-runtime timeout bug affecting slow self-hosted model calls, not a core outage or security emergency.
- `impact:auth-provider`: The affected behavior is model provider runtime configuration and timeout routing for configured providers.

## Summary

Close as implemented: current main and the shipped v2026.5.22 release honor explicit provider `timeoutSeconds` before the local-host heuristic, so the reported private `*.ts.net` hostname case no longer clamps to the 120s implicit watchdog when the provider timeout is set.

## What This Changes

The issue asks for `models.providers.<id>.timeoutSeconds` values above 120 seconds to be honored for private providers whose `baseUrl` hostname does not look local, such as Tailscale MagicDNS hosts.

## Best Possible Solution

Keep the existing shipped provider-timeout contract and point affected users on v2026.5.18 to upgrade to v2026.5.22 or newer, while ensuring any shorter whole-run timeout is raised when needed.

## Reproduction Assessment

No, not against current main: source and regression tests show explicit provider timeout values are honored independent of local hostname classification. The reported version, v2026.5.18, is older than the verified shipped v2026.5.22 release state.

## Solution Assessment

Yes: honoring the existing explicit provider timeout is the narrowest maintainable solution, because it preserves one documented config knob instead of adding a parallel `isLocal` classification override.
