- `impact:auth-provider`: The reported failure is an auth/provider routing mismatch between an OpenAI provider id and an `openai-codex:*` OAuth profile.

## Summary

Keep open: current main still has the inconsistent embedded compaction path described here. The normal embedded run path normalizes OpenAI plus an `openai-codex:*` auth profile through the Codex provider, but the compaction target resolver returns the caller provider unchanged when no compaction override is configured.

## What This Changes

The issue asks for embedded compaction runtime-context resolution to apply the same OpenAI Codex OAuth provider normalization used by normal embedded PI runs.

## Best Possible Solution

Reuse the existing OpenAI/Codex PI runtime-provider resolver inside the embedded compaction target path, add focused regression coverage, and avoid changing the broader OpenAI auth-selection policy.

## Reproduction Assessment

Yes, at source level: current main returns `provider: "openai"` with an `openai-codex:*` auth profile in the no-override compaction resolver, while the normal run path applies a separate Codex routing helper. I did not execute a live compaction run in this read-only review.

## Solution Assessment

Yes, the proposed direction is the narrowest maintainable fix if it reuses `resolveSelectedOpenAIPiRuntimeProvider` and adds a regression test. The repair should not broaden or redesign OpenAI auth selection while related routing policy issues remain open.
