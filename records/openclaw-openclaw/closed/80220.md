## Summary

Close as implemented: current main and the shipped v2026.5.2+ release line already honor `direct.<chatId>.threadReplies: "inbound"` for Telegram DMs by deriving thread-suffixed sessions. The canonical DM-topic key shape is `:thread:<chatId>:<topicId>` rather than `:topic:<topicId>`, but the requested per-topic isolation behavior is present.

## What This Changes

The issue requests that Telegram DM messages with different `message_thread_id` values create separate sessions when `channels.telegram.direct.<chatId>.threadReplies` is set to `"inbound"`.

## Best Possible Solution

Keep the shipped Telegram DM-topic isolation path and, if more reports appear, clarify that isolated DM-topic sessions use OpenClaw’s `:thread:<chatId>:<topicId>` suffix rather than the group-topic `:topic:<topicId>` suffix.

## Reproduction Assessment

No. The reported flat-session collapse is not reproducible from current main source because the direct chat config is read and the inbound path appends a thread session suffix when `threadReplies` is `"inbound"`; the focused regression test covers that path.

## Solution Assessment

Yes. The current implementation is the narrowest maintainable fix: keep normal DMs flat by default, preserve reply metadata, and isolate intentional DM topics only through existing `dm.threadReplies`, `direct.<chatId>.threadReplies`, `requireTopic`, or topic config.
