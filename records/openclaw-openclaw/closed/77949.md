## Summary

Keep this PR open: the main approach is useful, but the latest head still drops suppressed Codex item completion/status events whenever a channel has `onToolStart`, while normalized `tool` result events are not forwarded through that callback path.

## What This Changes

The PR adds Codex app-server tool-progress normalization, Telegram progress-draft timing changes, source-suppressed progress opt-in handling, tests, and a changelog entry.

## Best Possible Solution

Preserve the PRâ€™s normalization approach but keep item end/status callbacks, or otherwise add a result-phase channel callback, so duplicate starts are avoided without losing completion, failure, or blocked transitions.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a Codex item marked `suppressChannelProgress` with phase `end` is dropped when `onToolStart` exists, while the normalized `tool` `result` event is not forwarded to any channel progress callback.

## Solution Assessment

No. The overall direction is sound, but the current suppression guard is too broad; it should suppress only item phases that are actually represented by `onToolStart` or add a result/end callback path.
