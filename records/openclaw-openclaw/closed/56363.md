## Summary

Current main already carries the WhatsApp listener-state fixes for this report. The old split-chunk listener-loss class is covered by a process-global registry/controller design, and the changelog records the exact outbound-after-connect fix in v2026.3.22, with later WhatsApp hardening in v2026.4.11 and v2026.4.18.

## Close Comment

Closing as implemented on main after Codex review. The current WhatsApp path no longer relies on the old per-module listener map shape described here: listener ownership is now kept through a process-global connection-controller registry in `extensions/whatsapp/src/connection-controller-registry.ts:11`, the live controller is registered only after the socket and listener are ready in `extensions/whatsapp/src/connection-controller.ts:339`, and there is regression coverage for keeping the previous registered controller alive during replacement failures in `extensions/whatsapp/src/connection-controller.test.ts:96`. This exact symptom was already called out in the shipped `v2026.3.22` changelog: `CHANGELOG.md:2206` says WhatsApp outbound sends were fixed so `openclaw message send --channel whatsapp` stops failing after connect. Current main also includes later hardening for named default accounts (`CHANGELOG.md:850`) and per-account connection ownership (`CHANGELOG.md:816`). If you still see this on a current release, please open a fresh issue with the exact version and current gateway logs.
