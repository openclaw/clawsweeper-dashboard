## Label Justifications

- `P1`: The issue describes a user-facing Telegram/Codex turn that silently stops after image generation and leaves the session interrupted.
- `impact:message-loss`: The reported failure loses the final assistant response for a Telegram task after image generation work.
- `impact:session-state`: The session ends as `interrupted` after the Codex app-server timeout, which is session-state drift from the user's requested turn.

## Summary

Keep open: the report is a credible current Codex app-server liveness bug, and current main still only routes non-assistant raw completed items into the completion-idle timeout path rather than proving a clear image-generation-specific recovery or visible Telegram outcome.

## What This Changes

Codex app-server should not leave a Telegram turn silently interrupted after a completed raw `image_generation_call` item is followed by no `turn/completed` event.

## Best Possible Solution

Add a focused Codex app-server regression for raw `image_generation_call` completion without a terminal event, then implement the chosen owner policy so the turn either continues with bounded meaningful progress or fails with explicit channel-visible retry diagnostics.

## Reproduction Assessment

Yes for the core source path, but not for a full live current-main Telegram repro in this read-only review. The code path for non-assistant raw completed items leads to completion-idle timeout, while the issue provides release logs for the `image_generation_call` variant.

## Solution Assessment

Unclear. The current generic timeout guard is useful for releasing wedged turns, but it does not prove the image-generation case gets a clear user-visible recovery path; maintainers should choose whether this item type should extend liveness or fail fast with better channel diagnostics.
