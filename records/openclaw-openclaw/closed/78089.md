## Summary

Close as implemented on current main: native xAI Grok configs are routed through the xAI Responses transport, which records usage from the terminal response event, and the fallback OpenAI-completions stream loop also keeps reading trailing empty-choice usage chunks instead of stopping on `finish_reason`.

## What This Changes

The report asks OpenClaw to preserve real xAI Grok token usage and cost metadata for tool-use turns instead of writing zero-token assistant entries.

## Best Possible Solution

Keep the provider-owned xAI Responses routing on current main; if separate non-native xAI-compatible completions billing fields are needed later, track that as a narrower follow-up.

## Reproduction Assessment

No high-confidence failing current-main reproduction remains from source inspection: the reported native xAI `openai-completions` config is normalized to Responses, and both the Responses and fallback completions stream handlers consume terminal usage before finalization. I did not run a live xAI spend test because no xAI key was available in the checked profile.

## Solution Assessment

Yes. Routing native xAI traffic through the provider-owned Responses path is the narrow maintainable fix because it avoids adding xAI-specific race handling to the generic OpenAI-completions transport while still preserving streamed usage metadata.
