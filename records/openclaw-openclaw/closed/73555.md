## Summary

Current main already addresses the reported v2026.4.26 `sidecars.channels` startup stall. The release path awaited primary-model warmup inside `sidecars.channels`; current main schedules that warmup in the background, bounds it, scopes provider discovery to the configured provider, and splits trace spans into `sidecars.model-prewarm` and `sidecars.channel-start`. The prompt-processing side is also bounded by the default 15s timeout for `before_prompt_build` modifying hooks, with memory recall regression coverage.

## Best Possible Solution

Keep the current main behavior: primary-model warmup stays metadata-only, provider-scoped, bounded, and off the channel-start critical path; prompt-build plugin hooks remain bounded. Any remaining stall on 2026.4.27 or newer should be a fresh, narrower report with the split startup trace spans and per-channel timing.
