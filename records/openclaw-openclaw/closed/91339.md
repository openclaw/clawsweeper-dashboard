## Summary

Keep open: the PR fixes the central args-truncation bug, but the confirmation/reset classifier change is too broad and drops hard-reset handling for non-soft `/reset <tail>` commands; it also lacks after-fix real Control UI behavior proof.

## What This Changes

The PR modifies Control UI reset-command handling to pass `/reset` arguments through, skip confirmation for soft reset, and add app-chat regression tests.

## Best Possible Solution

Forward the full reset command while classifying only `/reset soft` or `/reset soft ...` as non-destructive; keep all other `/reset ...` tails in the hard-reset confirmation/refresh path and require real Control UI proof before merge.

## Reproduction Assessment

Yes. Source inspection reproduces the current-main bug through `handleSendChat` -> `parseSlashCommand` -> `dispatchSlashCommand`, where the reset branch sends only `/reset`; the PR-introduced classifier regression is also clear from the patch.

## Solution Assessment

No. Preserving reset args is the right layer, but the current classifier is not the best fix because it treats `/reset soft...` as non-destructive without a word boundary and stops recognizing other `/reset <tail>` commands as hard resets.
