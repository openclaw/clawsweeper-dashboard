## Summary

Current main implements the requested fix: plugin tool resolution now lazily loads missing manifest-selected tool plugins through the existing standalone runtime loader, and focused regression coverage locks the config-origin path-plugin case. The fix is present on main but not in a shipped tag yet.

## What This Changes

Path-based plugins loaded from `plugins.load.paths` should expose their manifest-declared agent tools after upgrade instead of failing with `unknown method`.

## Best Possible Solution

Keep the current main implementation and let the next release ship the restored scoped cold-load behavior for plugin tools.

## Reproduction Assessment

No current-main failing reproduction remains: the resolver now cold-loads manifest-selected tool plugin ids, and the regression test covers a config-origin path plugin without pre-warming. The original `v2026.5.2` failure path is well explained by the removed on-demand registry load.

## Solution Assessment

Yes. The current fix is the narrow maintainable path because it preserves loaded-registry fast paths and uses the existing scoped standalone runtime loader only for the remaining selected tool plugin ids.
