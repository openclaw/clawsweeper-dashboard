## Summary

Close as `clawhub`: this is a WeCom plugin-scoped inbound batching problem, not an OpenClaw core issue. Core intentionally keeps `messages.inbound` text-only for debounce, and the shipped plugin SDK already exposes channel-side debounce helpers that an external WeCom plugin can use to coalesce media bursts without changing the cross-channel core contract.

## Risks / Open Questions

- The underlying WeCom multi-image loss may still be a real bug in `@wecom/wecom-openclaw-plugin`; closing here does not resolve it upstream.
- If OpenClaw later wants a global media coalescing policy, that would need a broader design because current docs and code intentionally define media as outside `messages.inbound` debounce.
