## Label Justifications

- `P1`: The report describes a real channel session becoming stuck or failed after reset even though transport health is normal, which is an active agent/channel workflow regression.
- `impact:session-state`: The affected data is the persisted session row status and timing fields after a reset rotates sessionId for the same sessionKey.
- `impact:message-loss`: The corrupted fresh session can leave Slack appearing to stop replying despite healthy socket/auth state.

## Summary

Keep open: current main still routes lifecycle persistence by sessionKey without carrying the old run's sessionId, so a late start/end/error event can target whichever row the key now resolves to after reset.

## What This Changes

Prevent sessions.reset from letting stale lifecycle events from a pre-reset run update the newly rotated session row.

## Best Possible Solution

Carry the owning sessionId through lifecycle-producing run context and ignore lifecycle persistence/snapshot updates when the event's sessionId and current row's sessionId are both known and different.

## Reproduction Assessment

Yes at source level: current main can persist a lifecycle start/end/error by sessionKey after sessions.reset has moved that key to a new sessionId. I did not run the live Slack race, but the code path directly matches the reported sequence.

## Solution Assessment

Yes: a session-id guard in lifecycle persistence/projection is the narrowest durable fix. More aggressive abort/drain during reset may help, but it should not be the only protection against late async terminal events.
