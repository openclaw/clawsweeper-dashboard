## Summary

Keep #70470 open. The issue is recent, non-maintainer-authored, and current main still awaits WhatsApp `sock.readMessages` before enriching or dispatching accepted inbound messages to the agent. The triage comment's suggested `access.shouldMarkRead` guard has not landed, and there is no current test or changelog evidence showing that sender-side read-receipt-disabled behavior is handled independently of the message-processing path.

## Best Possible Solution

Keep the issue open and fix it in the WhatsApp plugin. The maintainer path should isolate read-receipt side effects from inbound dispatch: add focused coverage for `sock.readMessages` rejecting and, if feasible, timing out or never resolving; ensure accepted messages still reach `onMessage`; and decide whether the existing `shouldMarkRead` field should gate read receipts or whether read receipts should run asynchronously after dispatch.
