## Label Justifications

- `P1`: The report describes generated Discord replies remaining internal while users see no visible response, which is an urgent channel delivery failure.
- `impact:message-loss`: The affected path can lose visible Discord replies even though the agent run and transcript continue internally.
- `impact:auth-provider`: The failure is tied to resolving SecretRef-backed Discord bot tokens for named accounts.

## Summary

Close: current main now covers the reported SecretRef/runtime-snapshot Discord delivery path, and the relevant fixes are included in v2026.6.6. The direct message-tool send path resolves scoped Discord account SecretRefs before local send, Discord account/token resolution selects the active runtime snapshot, and the later gateway message.action fix closes the same configured_unavailable runtime-config gap for plugin actions.

## What This Changes

Ensure Discord message-tool sends for env SecretRef-backed named accounts resolve the same runtime account token material used by provider startup.

## Best Possible Solution

Keep the shipped runtime-snapshot and scoped SecretRef resolution paths; any remaining multi-account hot-reload restart behavior should stay tracked separately in the open hot-reload issue.

## Reproduction Assessment

Yes. The failing branch is source-reproducible from the Discord token resolver returning configured_unavailable for unresolved SecretRefs unless the send/action path supplies or selects the active runtime snapshot; current main has focused tests for the named-account send and runtime-snapshot paths.

## Solution Assessment

Yes. The current solution is the maintainable shape: resolve the selected channel/account SecretRefs at the message-tool boundary, let Discord account resolution select a matching active runtime snapshot, and keep unresolved configured tokens fail-closed instead of adding a process.env fallback.
