## What This Changes

The issue asks for `exec-approval-followup` CLI task runs to stop staying `running` forever after their backing agent run is gone.

## Best Possible Solution

Run-backed CLI task liveness should depend on the live agent run context, with stale exec approval follow-ups transitioning out of `running` after the existing grace period while preserving any intended legacy session-backed CLI behavior.

## Reproduction Assessment

Yes, at source level. Seed a stale `runtime: "cli"` task with inactive `sourceId`/`runId`, `childSessionKey: "agent:main:main"`, and a matching session-store row; current `hasBackingSession` keeps it backed and prevents reconciliation.

## Solution Assessment

Yes, with a narrow scope. The right repair belongs in `hasBackingSession`: run-backed CLI follow-ups should not fall back to persistent session existence after their run context disappears.

## Repair Work Prompt

Fix openclaw/openclaw#76162 on current main. The observable bug is that stale `exec-approval-followup` task runs with `runtime: "cli"`, inactive `sourceId`/`runId`, and `childSessionKey: "agent:main:main"` can stay `running` forever because `hasBackingSession` falls through from `hasActiveCliRun(task) === false` to persistent session existence. Implement a narrow task-maintenance fix in `src/tasks/task-registry.maintenance.ts` so run-backed CLI tasks are considered unbacked once their live run context is gone; preserve intended fallback behavior for CLI task records that lack run-liveness metadata unless the existing contract clearly says otherwise. Add/update regression coverage in `src/tasks/task-registry.maintenance.issue-60299.test.ts` for an inactive `exec-approval-followup`/main-session CLI task with a matching session-store row being marked `lost`, and keep existing active-run and chat-backed CLI behavior covered. Update `docs/automation/tasks.md` only if the liveness contract text changes, and add a single-line user-facing fix entry under `CHANGELOG.md` `Unreleased` fixes. Do not touch unrelated exec approval delivery issues such as #59349, #72143, or #76233.
