## Summary

Close as implemented on main: current main and v2026.4.29 already solve the reported OpenAI image timeout problem through provider-owned defaults, while this PR's runtime-wide fallback would regress Azure OpenAI by shortening its shipped 600s default to 180s.

## What This Changes

The PR adds a core image-generation runtime 180000 ms fallback and unit tests for default, config, and per-call timeout precedence.

## Best Possible Solution

Keep the shipped provider-owned timeout behavior; any future generic default should preserve omitted timeoutMs at the runtime boundary or add a typed provider-default seam.

## Reproduction Assessment

Yes. By inspection, current main leaves timeoutMs undefined when neither the tool call nor config sets it, while this PR would pass 180000 into every provider and prevent Azure from applying its tested 600000 ms default.

## Solution Assessment

No. A core runtime fallback is not the narrowest maintainable fix because OpenAI image providers already own endpoint-specific defaults and the shipped path preserves explicit per-call/config overrides.
