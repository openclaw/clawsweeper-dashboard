## Summary

Keep open: current main and latest release still materialize explicit providers wholesale and retain existing provider entries in merge mode, so the reported credential-boundary issue remains source-provable. Related open PRs address broader plaintext key stripping, but none is merged and they do not close the narrower per-agent provider scoping request.

## What This Changes

Scope each agent's generated models.json to that agent's own credential-bearing provider entries while keeping bridge cross-validation as defense in depth.

## Best Possible Solution

Define and implement an owner-reviewed per-agent credential materialization rule that removes sibling credential-bearing providers from generated models.json while preserving safe shared catalog metadata, SecretRef marker behavior, and runtime provider fallback semantics.

## Reproduction Assessment

Yes, at source level: current main still feeds all explicit models.providers into the per-agent plan and merge mode retains existing provider entries. I did not run a live saqi bridge setup, so the log-based acceptance criterion is not independently verified.

## Solution Assessment

Unclear: excluding sibling credential-bearing providers is the right security direction, but the exact scoping rule needs maintainer/security review so safe shared catalog entries, SecretRef markers, and provider fallback behavior are not accidentally removed.
