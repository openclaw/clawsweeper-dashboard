## Summary

Keep this PR open. Current main still lacks the focused model-provider proxy guidance the PR adds: the concept page does not explain that model provider requests do not automatically use HTTP_PROXY/HTTPS_PROXY, that env proxying requires an explicit request.proxy config, or that an explicit models.providers.<provider> entry must still satisfy the provider schema. The existing config reference only documents the field modes at a reference level. The PR is docs-only, and the provided review context shows the author already followed up on the placement and transport-support caveats.

## Best Possible Solution

Keep the PR open for normal maintainer docs review. The best path is to merge a polished version of the concept-page guidance, placed near the existing custom-provider proxy-route section, with explicit wording that env proxying is configured through request.proxy, explicit provider entries must remain schema-complete, and transport support is limited to the supported provider transport APIs.
