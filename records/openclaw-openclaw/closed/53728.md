## Summary

Keep this PR open for maintainer review. Current main still lacks automatic startup memory-source reporting in agent prompts, so the work is not obsolete, but the branch has two concrete prompt-startup correctness issues and mixes in an unrelated Feishu logging/privacy change that should be reviewed separately.

## What This Changes

The PR adds a memory-source report helper, injects it into embedded and command system-prompt workspace notes, and adds Feishu outbound delivery result logging with test updates.

## Best Possible Solution

Land a focused, maintainer-approved memory visibility design that uses a documented status contract, bounds all startup probes, represents unknown counts honestly, follows current file-memory semantics, and splits or separately reviews the Feishu outbound logging if that remains desired.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Bound Memoria probes before prompt construction:** `src/agents/memory-source-report.ts:39`
  - body: `buildMemorySourceReport` is awaited while constructing embedded and command system prompts, but `fetchJson` uses bare `fetch` with no timeout. A slow or blackholed configured Memoria URL can stall startup until the platform network timeout instead of degrading quickly to unavailable.
  - confidence: 0.91
- **[P2] Do not report a limited fallback as the memory count:** `src/agents/memory-source-report.ts:69-72`
  - body: The fallback request uses `/v1/memories?limit=1`, then returns `items.length` as the count. When the stats endpoint is absent, any backend with more than one memory will be reported as having one memory, making the injected startup prompt note misleading.
  - confidence: 0.94
