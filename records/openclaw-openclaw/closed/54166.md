## Summary

Keep this open: current `main` still has the `llm-task` resolver bypass reported here, while the related cron alias work fixed a different surface.

## What This Changes

The issue reports that `llm-task` model overrides are dispatched without resolving OpenClaw aliases or explicit provider/model refs through the canonical model resolver.

## Best Possible Solution

Expose or reuse a plugin-facing canonical model-ref resolver and have `llm-task` use the resolved provider/model before thinking policy, `allowedModels`, embedded dispatch, and result metadata.

## Reproduction Assessment

Yes, from source inspection. With an Anthropic/default provider and `model: "gemini-flash"` or `model: "google/gemini-3-flash-preview"`, current `llm-task` builds and dispatches the selected provider plus the raw model instead of canonical `google/gemini-3-flash-preview`.

## Solution Assessment

Yes, resolving through the shared model-ref logic before plugin policy checks and dispatch is the narrow maintainable fix. Provider-specific alias table patches would not cover explicit cross-provider refs or configured aliases.

## Repair Work Prompt

Fix OpenClaw issue #54166: `llm-task` must resolve `model` and plugin `defaultModel` through the same canonical model-ref logic used by normal agent/cron model selection before policy checks or dispatch. Current `extensions/llm-task/src/llm-task-tool.ts` splits provider and raw model locally, so `model: "gemini-flash"` or `model: "google/gemini-3-flash-preview"` under an Anthropic/default provider is validated and dispatched under the wrong provider. Add a small plugin-runtime/SDK resolver seam or equivalent runtime helper; do not make extension production code import core `src/**` directly. Use the resolved provider/model for thinking policy, `allowedModels`, `runEmbeddedPiAgent`, and returned details. Add focused regression tests for built-in alias `gemini-flash`, explicit cross-provider ref `google/gemini-3-flash-preview`, Anthropic alias coverage such as `haiku`, and canonical `allowedModels` validation. Update docs only if user-facing config semantics change, and add a single-line `CHANGELOG.md` Fixes entry.
