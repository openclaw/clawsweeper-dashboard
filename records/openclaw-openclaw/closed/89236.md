## Summary

Keep this PR open: the Slack same-account fallback is a valid bug repair, but the branch still sends a new `requesterAccountId` root field over strict v4 Gateway `message.action` RPC without a version/capability path, so an upgraded agent can fail against an older v4 Gateway before dispatch.

## What This Changes

The PR changes Slack `member-info` dispatch to infer an omitted `userId` from the same-account inbound Slack sender and propagates requester account identity through local, plugin, and Gateway action contexts.

## Best Possible Solution

Land the Slack same-account sender fallback after the Gateway identity propagation is versioned, capability-gated, or split out so v4 Gateway clients and servers remain upgrade-safe.

## Reproduction Assessment

## Solution Assessment

No, not as submitted. The Slack fallback is the right local repair, but the Gateway `requesterAccountId` propagation needs a compatibility-safe protocol path or should be split out.
