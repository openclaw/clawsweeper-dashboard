## Summary

Keep open. Current main has fixed or covered the explicit media payload, block-stream final payload, and Telegram lane-delivery parts, but local/generated Markdown image targets are still intentionally left as text and changing that needs a local-media trust policy decision.

## What This Changes

The issue requests that ordinary Telegram assistant replies preserve image attachments when text and media are delivered together.

## Best Possible Solution

Define a safe managed/generated-local-media promotion policy, or require generated images to emit explicit MEDIA:/mediaUrls metadata, then add parser, reply-pipeline, Telegram delivery tests, and redacted live Telegram proof.

## Reproduction Assessment

Yes, source-level. A local Markdown image target with extractMarkdownImages enabled still remains text because the parser only lifts remote http(s) media, and current tests assert local targets and final replies produce no mediaUrls.

## Solution Assessment

No, not as an arbitrary local Markdown lift. The maintainable path needs a maintainer-approved generated/local media allowlist tied to existing file-read and media type checks, or explicit MEDIA:/mediaUrls metadata from the image-producing path.
