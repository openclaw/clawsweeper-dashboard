## Summary

Current main and the latest shipped tag already route cron tool and Gateway updates through `CronService.update`, replace the stored job, and persist changed job definitions back to `jobs.json`; the documented split only keeps runtime execution state in `jobs-state.json`.

## What This Changes

The issue requests that cron job updates made through the gateway-backed `cron` tool persist to `cron/jobs.json` so they survive Gateway restarts.

## Best Possible Solution

Keep the current cron update persistence path and treat any remaining report as a new, narrower reproduction against `v2026.5.12` or current main with the exact tool payload and redacted before/after cron store files.

## Reproduction Assessment

No; I did not run a live Gateway restart cycle in this read-only review. Source inspection traces the current update path from the `cron` tool through Gateway `cron.update` to `persist(state)`, and the latest release contains that same path.

## Solution Assessment

Yes; the existing service-owned writer is the maintainable solution because it preserves the cron store contract and keeps runtime-only fields in `jobs-state.json` while writing definition changes to `jobs.json`.
