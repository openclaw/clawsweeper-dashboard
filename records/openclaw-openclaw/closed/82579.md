## Label Justifications

- `P1`: This PR changes a plugin security boundary and currently has a blocking compatibility regression in setup-only workspace plugin loading.
- `merge-risk: 🚨 compatibility`: The new loader and registry checks can stop existing trusted workspace plugin setup/status flows when those plugins are allowlisted but disabled for runtime.

## Summary

Keep this PR open, but it is not merge-ready: the hardening is aimed at a real plugin trust boundary, yet the current patch still uses `enabled` as a blanket trust proxy and can over-block setup metadata for trusted workspace plugins that are explicitly disabled for runtime. The setup-entry `registerChannel` error isolation looks directionally right, and the remaining blockers are compatibility/contract fixes plus removing the direct changelog edit.

## What This Changes

This PR hardens plugin channel setup loading by blocking disabled workspace channel plugins from setup-only scoped loads, adding a registry backstop for disabled workspace channel registrations, wrapping setup-entry channel registration failures as diagnostics, and adding regression coverage around catalog, loader, and registry paths.

## Best Possible Solution

Keep the untrusted workspace hardening and the setup-entry error isolation, but narrow the loader and registry checks so they reject only untrusted workspace candidates while preserving current scoped setup-only behavior for trusted disabled plugins; remove the direct changelog entry and keep release-note context in the PR body or squash message.

## Reproduction Assessment

Yes, source-reproducible: current main has a focused setup-only disabled-channel test, and the PR branch's `enabled` checks block that state for workspace-origin plugins. I did not run tests because this was a read-only review.

## Solution Assessment

No, not yet. The `registerChannel` try/catch is the right error-isolation shape, but the workspace hardening should distinguish trust cause from runtime enabled state rather than using `enabled` as a blanket proxy.
