## Summary

Current main already covers this class of failure: env-backed provider credentials are resolved from config/env before model auth is used, queued reply runs re-resolve runtime and channel/account SecretRefs before execution, and the Xiaomi provider surface is now standardized on `XIAOMI_API_KEY` rather than `MIMO_API_KEY`. The reported Feishu-vs-webchat divergence no longer matches main.

## Evidence

- **Provider auth resolves env-backed model apiKey refs:** `resolveUsableCustomProviderApiKey` coerces `models.providers.<id>.apiKey` into a SecretRef, reads the env value, and returns the resolved key; `resolveApiKeyForProvider` then uses that resolved value for runtime auth.
  - file: [src/agents/model-auth.ts:114](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/agents/model-auth.ts#L114)
  - sha: [608c08fc54da](https://github.com/openclaw/openclaw/commit/608c08fc54da2f86338366f8eeafd7e2a02713aa)
- **Reply runs resolve runtime and scoped channel secrets before execution:** Feishu/other channel follow-up runs call `resolveQueuedReplyExecutionConfig(...)` before replying, and that function resolves base runtime secrets plus scoped channel/account secrets through the gateway secret-resolution path.
  - file: [src/auto-reply/reply/agent-runner.ts:1025](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/auto-reply/reply/agent-runner.ts#L1025)
  - sha: [608c08fc54da](https://github.com/openclaw/openclaw/commit/608c08fc54da2f86338366f8eeafd7e2a02713aa)

  - file: [src/auto-reply/reply/agent-runner-utils.secret-resolution.test.ts:45](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/auto-reply/reply/agent-runner-utils.secret-resolution.test.ts#L45)
