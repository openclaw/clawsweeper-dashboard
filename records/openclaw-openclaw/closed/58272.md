Close as ClawHub scope: current main keeps read-tool path strings literal, and the remaining useful behavior is opt-in repair of model-mutated CJK/ASCII spacing that can be implemented through plugin hooks without a missing core API.

## What This Changes

The issue reports ENOENT failures when reading macOS paths containing mixed CJK and ASCII directory or filename segments.

## Best Possible Solution

Keep core path handling literal and route optional CJK/ASCII path de-spacing to a ClawHub code plugin using `before_tool_call` with validation that the original path is missing and the corrected path exists.

## Reproduction Assessment

No. The report includes concrete logs, but current source does not show OpenClaw mutating an exact mixed CJK/ASCII path; it only shows that a read will fail if the model/provider supplies a spaced path that does not exist.

## Solution Assessment

Yes for the remaining behavior. A ClawHub plugin is the narrowest maintainable solution because a core heuristic could silently retarget legitimate filenames or command text.
