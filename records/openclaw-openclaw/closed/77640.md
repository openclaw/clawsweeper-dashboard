## Summary

Current main and the latest release already implement the backend `gateway-client` loopback shared-auth behavior that this PR is trying to force in `subagent-spawn`, and the PR also adds a regression assertion for a `sessions.patch` gateway call that the current spawn flow no longer makes.

## What This Changes

The PR explicitly passes backend/gateway-client identity for subagent gateway RPCs and adds assertions for `agent` and `sessions.patch` gateway calls.

## Best Possible Solution

Keep the shipped `callGateway` backend-auth contract and close this redundant PR rather than adding duplicate defaults in the subagent wrapper.

## Reproduction Assessment

Yes for the PR defect: source inspection shows this test path does not call gateway `sessions.patch`, so the added assertion can be reproduced with `pnpm test src/agents/subagent-spawn.test.ts`. The reported runtime auth problem itself is not reproduced on current main because `callGateway` already defaults to backend/gateway-client.

## Solution Assessment

No. Current main and `v2026.5.3` already solve the runtime issue at the shared gateway call layer, and the PR's added `sessions.patch` assertion is inconsistent with the current direct session-store patch flow.
