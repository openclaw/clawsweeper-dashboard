## Summary

Keep open: current main and v2026.6.10 still run Memory Wiki wikilink extraction over markdown that includes fenced and inline code, and open candidate PRs now own the focused extractor fix.

## What This Changes

The issue asks for `openclaw wiki lint` to stop reporting `Broken wikilink target` warnings for `[[...]]` text inside fenced code blocks and inline code spans.

## Best Possible Solution

Land one focused Memory Wiki extractor fix that masks or skips fenced code blocks and inline code before link extraction, with regression coverage for both Obsidian wikilinks and Markdown links in code regions.

## Reproduction Assessment

Yes, at source level: a page containing `[[...]]` inside fenced or inline code will still be scanned by `extractWikiLinks`, and unmatched extracted tokens flow into `broken-wikilink` warnings. I did not run `wiki lint` because this review is read-only and that command can write reports/logs.

## Solution Assessment

Yes, the narrow maintainable fix is to make the Memory Wiki extractor Markdown-code-aware before `linkTargets` reach lint. Title/alias resolver normalization is adjacent and should remain with the separate resolver issue unless maintainers intentionally combine it.
