## Summary

Current main already addresses the reported Telegram mobile delivery failure by making standard Telegram HTML the default text path, gating rich messages behind an explicit setting, and falling back to plain text on Telegram HTML parse errors; the fix is current-main-only and not yet in the latest release.

## What This Changes

The issue reports Telegram outbound payloads being rejected on Android clients and requests escaping dynamic text before Telegram delivery.

## Best Possible Solution

Keep the current main behavior: standard Telegram HTML by default, rich messages opt-in only, and plain-text fallback for parse failures; reopen only with a raw Telegram API error from a remaining MarkdownV2 path.

## Reproduction Assessment

No, not against current main: the reported MarkdownV2 send path is no longer present in the inspected Telegram delivery surfaces, which now use HTML or rich messages only when explicitly enabled.

## Solution Assessment

Yes: avoiding MarkdownV2 for default delivery and gating richer client-sensitive messages behind an explicit setting is a narrower and safer fix than adding blanket MarkdownV2 escaping to dynamic text.
