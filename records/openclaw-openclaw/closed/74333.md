## Summary

Current main already implements the requested recall accounting. The memory-core `memory_search` tool queues `recordShortTermRecalls` after successful `manager.search`, HTTP and MCP tool surfaces invoke the same registered tool `execute` path, and regression tests assert `recallCount: 1` is persisted for tool hits. The proof commit is on `main` but is not contained in any local release tag, so this appears fixed on current main only.

## What This Changes

Record short-term recall counts for agent-driven `memory_search` calls so the recall store reflects tool retrievals, not only CLI searches and background scans.

## Best Possible Solution

Keep the current plugin-level recall tracking so every surface that executes the registered `memory_search` tool records recalls without duplicating HTTP-specific instrumentation; let the next release carry the main-only fix.
