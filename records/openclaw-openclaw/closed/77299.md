## Summary

Current `main` and the shipped `v2026.5.3` release already keep `/status` context usage tied to fresh prompt snapshots or post-compaction `tokensAfter`, not cumulative cache-read totals.

## What This Changes

The issue asks for `/status` context usage to report current post-compaction prompt size instead of cumulative cached-read tokens.

## Best Possible Solution

Keep the shipped prompt-snapshot accounting and have users on `2026.4.25` upgrade to `v2026.5.3` or newer.

## Reproduction Assessment

No for current `main`: the old-release steps are understandable, but source inspection shows current `/status` reads fresh prompt snapshots or post-compaction `tokensAfter`. I did not run a live Telegram session because this review is read-only.

## Solution Assessment

Yes: the current implementation is the narrow maintainable fix because it separates cumulative usage/cost telemetry from context-window display state and marks stale totals unknown.
