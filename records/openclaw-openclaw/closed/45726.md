## Summary

Current `main` already contains the fix for the leaked embedded-session lock path reported in #45726, and the same fix is present in the released `v2026.4.22` tag.

## Evidence

- **Merged fix PR linked from the issue:** The issue timeline cross-references merged PR #59194, titled "[Fix] Isolate teardown steps so session lock release is unconditional". The PR summary explicitly cites #45726 and describes the same failure mode: cleanup exceptions could skip `sessionLock.release()` and leave the session blocked by a persistent `.jsonl.lock` file.
  - command: `Opened GitHub PR #59194 conversation/commits pages via web`
- **Current main guarantees lock release during teardown:** `cleanupEmbeddedAttemptResources()` now wraps each teardown step independently and releases the session lock in an outer `finally`, so a failure in flush/dispose/WebSocket/LSP cleanup no longer leaks the `.jsonl.lock` file.

  - command: `nl -ba src/agents/pi-embedded-runner/run/attempt.subscription-cleanup.ts | sed -n '1,260p'`
