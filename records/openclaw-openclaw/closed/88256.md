## Label Justifications

- `P1`: The reported Telegram polling restart storm could break a real channel workflow and cause repeated restart/unresponsive behavior during network instability.
- `impact:message-loss`: The issue and follow-up logs describe Telegram replies not being delivered during the restart window.
- `impact:crash-loop`: The core symptom is repeated Telegram polling/provider restart cycles and forced stop timeouts.

## Summary

Current main and shipped releases now implement the requested Telegram polling hardening: the restart policy is 30s/10m, repeated stop timeouts get a separate cooldown, stop requests are deduplicated, and healthy polling resets the backoff state. The June 9 follow-up overlaps with broader in-flight reply loss, but the specific polling restart-storm policy is already fixed by the merged Telegram polling PR.

## What This Changes

The issue requests less aggressive Telegram polling restart backoff plus stop-timeout cooldown, stop-call deduplication, and healthy-cycle reset handling.

## Best Possible Solution

Keep the shipped Telegram polling restart hardening, and leave any broader in-flight reply-loss investigation to https://github.com/openclaw/openclaw/issues/78380 or a narrower delivery issue.

## Reproduction Assessment

No for current main: the old 2s/30s policy was source-reproducible in the pre-fix code and user logs, but current main and shipped release source now contain the requested 30s/10m policy and stop-timeout cooldown. I did not run a live Telegram network-flap reproduction in this read-only close pass.
