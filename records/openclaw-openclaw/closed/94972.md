## Summary

Keep open: the PR changes remote skill eligibility in a way that conflicts with the current execution contract. `system.which` is already allowed for probing bins, but eligible remote macOS skills are advertised as runnable with `exec host=node`, and that execution path still requires `system.run`.

## What This Changes

The PR changes three remote skill runtime gates and one test so macOS nodes with `system.which` but no `system.run` count as remotely eligible.

## Best Possible Solution

Keep remote skill eligibility limited to `system.run`-capable macOS nodes, and solve the linked CLI symptom at the Gateway-backed skills status boundary unless maintainers explicitly define a separate discovery-only mode.

## Reproduction Assessment

Yes from source inspection. A connected macOS node with only `system.which` and cached `memo` would become eligible under the PR, while the actual `exec host=node` runtime rejects nodes without `system.run`.

## Solution Assessment

No. The PR changes the runtime eligibility contract; the narrower maintainable fix is to keep eligibility aligned with executable nodes and address the linked CLI status-source gap separately.
