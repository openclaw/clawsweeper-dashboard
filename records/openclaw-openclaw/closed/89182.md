## Summary

Keep open: current main and v2026.5.28 still route Linux memory directory paths through chokidar, and inspected Node/chokidar source supports the reported per-entry watcher fan-out. The item is also MEMBER-authored and has the protected `maintainer` label, so cleanup review should not close it.

## What This Changes

The issue asks for Linux memory directory watching to avoid per-file inotify/watch-descriptor allocation while preserving existing macOS/Windows behavior and file-path chokidar coverage.

## Best Possible Solution

Implement a Linux-only directory watcher that uses one native watcher per directory, attaches newly-created subdirectories on demand, keeps `MEMORY.md` and file-typed `extraPaths` on chokidar, and proves descriptor count scales with directories rather than files.

## Reproduction Assessment

Yes. A high-confidence source reproduction path exists by enabling `memorySearch.sync.watch` on Linux over a synthetic large memory tree and counting inotify watch entries; I did not run it in this read-only review, but current source and dependency contracts match the issue's Docker probe.

## Solution Assessment

Yes. A Linux directory-only watcher is the narrowest maintainable fix because both Node's non-native recursive watcher and chokidar fan out per entry, while defaulting watch off would sacrifice freshness and is tracked separately at https://github.com/openclaw/openclaw/issues/71335.
