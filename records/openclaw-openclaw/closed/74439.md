## Summary

Keep open. Current main has substantial SecretRef support for external env/file/exec-backed credential references, runtime snapshots, audit/configure/apply/reload flows, and broad credential-surface docs, but it does not implement the requested first-party central vault with per-key ACLs, usage accounting, TTL/expiry policy, rotation commands, or access audit logs. This is a security-sensitive product architecture request, not a conservative cleanup close.

## What This Changes

The issue requests a centralized OpenClaw API key/secret vault with logical references, ACLs, usage tracking, TTL/expiry, rotation, migration, and access auditing across configs, channels, plugins, and gateway runtime.

## Best Possible Solution

Define the secret-management architecture first: either extend SecretRefs with a first-party vault provider and explicit ACL/audit/rotation semantics, or document that OpenClaw delegates those controls to external vaults through file/exec providers. Any accepted design should preserve existing SecretRef compatibility and update the CLI, gateway protocol, credential-surface docs, tests, and threat model together.
