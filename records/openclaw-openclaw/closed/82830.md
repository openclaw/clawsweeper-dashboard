## Summary

Current OpenClaw already has the MCP stdio lifecycle fixes that address the reported 2026.4.21 leak: stdio transports now close through OpenClaw-owned process-tree teardown, session runtimes are retired on lifecycle paths, and idle runtimes are swept. The remaining QClaw bundle upgrade is external to this repository.

## What This Changes

The issue reports that Windows stdio-mode MCP servers spawned by OpenClaw 2026.4.21 accumulate old child processes across heartbeat/session runs.

## Best Possible Solution

Keep the shipped OpenClaw MCP lifecycle implementation and route the QClaw-specific bundled-version upgrade through the already cross-referenced QClaw issue.

## Reproduction Assessment

No; I did not run a Windows Server repro, and the report targets OpenClaw 2026.4.21 rather than current main. Source and related-issue history show current main already closes stdio transports through process-tree teardown and retires/sweeps session MCP runtimes.

## Solution Assessment

Yes; the best OpenClaw-side solution is the current shared lifecycle cleanup rather than a per-MCP workaround or external process reaper. The remaining action is for QClaw to bundle a fixed OpenClaw version.
