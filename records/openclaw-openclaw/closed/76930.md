## Summary

Keep open: this protected draft PR addresses a real current-main bug, but the proposed marker still lets manifest preselection choose unrelated tool plugins before optional-tool filtering.

## What This Changes

The PR changes optional plugin tool discovery so `tools.alsoAllow` can name optional plugin tools without widening discovery to every optional plugin tool, and adds tests, docs, and changelog coverage.

## Best Possible Solution

Revise the PR so `alsoAllow` supplies explicit optional tool hints while preserving default non-optional plugin tools without using the default marker to preselect or load unrelated tool plugins.

## Reproduction Assessment

Yes. Source inspection on current main shows `tools.alsoAllow` becomes `["*", ...alsoAllow]`, `collectExplicitAllowlist` forwards that wildcard, and plugin resolution treats `*` as allowing optional plugin tools.

## Solution Assessment

No. The marker is a good narrow direction for policy provenance, but applying it in manifest-level allowlist matching is too broad because that stage lacks per-entry optional metadata.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep the marker out of manifest-wide plugin selection:** `src/plugins/tools.ts:307`
  - body: Here the default-tools marker makes `manifestToolContractMatchesAllowlist` return true for every manifest with a tool contract. `resolvePluginToolRuntimePluginIds` runs this before it has `PluginToolRegistration.optional`, then passes all matched ids into runtime loading, so `tools.alsoAllow: ["lobster"]` can still cold-load unrelated optional-tool plugins instead of only the named optional plugin. Please split the default non-optional path from optional discovery, and add coverage on selected `onlyPluginIds` or registry loading.
  - confidence: 0.87

## Security Review

Status: needs_attention
