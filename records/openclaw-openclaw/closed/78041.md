## Summary

Keep open. Current main still routes stale OpenAI Codex OAuth credentials through synchronous embedded-run auth resolution and the locked provider refresh adapter, while the latest reporter data narrows but does not yet identify the four-second substep.

## What This Changes

The issue reports recurring multi-second cold-path OpenAI Codex OAuth auth latency during embedded dispatch while warm auth resolution remains a few milliseconds.

## Best Possible Solution

Add targeted auth substage timing around store loading, OAuth manager refresh, file locks, runtime import, and the OpenAI Codex token call, then optimize only the confirmed hot subpath with credential-safe invalidation.

## Reproduction Assessment

No high-confidence live reproduction was run against current main. Source inspection confirms stale OpenAI Codex OAuth credentials still synchronously enter the locked refresh path inside embedded-run auth, but the reported four-second timing still needs current-main substep measurements.

## Solution Assessment

Unclear. The proposed cache may be the right fix, but the safer maintainer path is to measure the refresh manager and provider adapter first, then choose an owner-approved credential invalidation policy.
