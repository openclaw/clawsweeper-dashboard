## Summary

Keep open: current main and the latest release still have the mismatch the reporter described. The browser plugin intentionally keeps the standalone loopback HTTP server lazy unless `OPENCLAW_EAGER_BROWSER_CONTROL_SERVER` is truthy, while the public browser-control docs describe the HTTP API without that startup requirement.

## What This Changes

The issue requests that the browser-control HTTP API either start automatically when browser control is enabled or that the docs state `OPENCLAW_EAGER_BROWSER_CONTROL_SERVER=1` is required.

## Best Possible Solution

Update the browser-control docs to state that the standalone loopback HTTP API is opt-in via `OPENCLAW_EAGER_BROWSER_CONTROL_SERVER=1`, while the CLI and `browser.request` continue using the in-process service without that listener.

## Reproduction Assessment

Yes by source inspection: current tests and plugin startup code show the standalone HTTP listener stays lazy unless the eager env var is truthy, while current docs still describe the HTTP API without that requirement. I did not run a live gateway in this read-only review.

## Solution Assessment

Yes, documenting the existing opt-in env requirement is the narrowest maintainable fix because current tests intentionally preserve lazy startup for gateway sidecar overhead. Auto-starting the listener should be a separate maintainer product/security decision if desired.
