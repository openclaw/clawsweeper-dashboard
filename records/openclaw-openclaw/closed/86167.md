## Label Justifications

- `P1`: The reported behavior can spam visible channel messages and break active chat workflows for real users, even though it is now fixed.
- `impact:message-loss`: The issue is about duplicated visible channel messages from repeated `message(action=send)` tool calls.

## Summary

Current main and the latest release already mark successful implicit message-tool sends in `message_tool_only` mode as terminal, which addresses the reported v2026.5.19 consecutive self-send loop.

## What This Changes

The issue asks OpenClaw to stop the agent loop after a successful visible `message(action=send)` in `message_tool_only` delivery mode so the LLM does not continue from its own sent message.

## Best Possible Solution

Keep the shipped terminal message-tool-only behavior; a new narrow report should cover any v2026.5.22-or-newer mixed-tool-batch loop with concrete transcript evidence.

## Reproduction Assessment

No current-main failing reproduction remains from the provided steps. The report targets OpenClaw `2026.5.19`, while `v2026.5.22` and current main already ship terminal handling for successful message-tool-only sends.

## Solution Assessment

Yes for the reported consecutive-send loop. The current solution lives in the embedded runner/message-tool boundary and uses Pi's `terminate` contract rather than adding a user-facing config flag.
