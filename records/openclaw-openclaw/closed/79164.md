## Summary

Keep open. Current main still has validation, atomic config writes, backup rotation, restart scheduling, and invalid-config recovery, but it does not implement a configurable post-apply health gate or automatic rollback for schema-valid config that makes the gateway unhealthy; the reporter also confirmed this needs maintainer reliability/product design before implementation.

## What This Changes

The issue requests automatic backup, post-apply gateway health checking, and rollback for schema-valid config changes that make the gateway unhealthy.

## Best Possible Solution

Define a gateway-owned transactional config write contract that reuses existing validation/backups, observes an explicitly defined health signal after restart-requiring writes, and documents rollback scope, timeout, and failure semantics.

## Reproduction Assessment

Not applicable as a feature request. Source inspection confirms the requested health-gated rollback path is absent, but the issue does not provide a concrete schema-valid config sample that reproduces a current-main gateway failure.

## Solution Assessment

Unclear. The requested direction is plausible, but the maintainable solution needs maintainer design for what counts as healthy, what gets rolled back, how long to observe, and what happens when rollback itself fails.
