## Summary

Current main already rewrote the Feishu inbound routing path and route-resolution cache so bindings are resolved per channel/account/peer/parentPeer instead of sharing ambiguous state. That matches the reported multi-group misrouting symptom closely enough to close as implemented on main.

## Evidence

- **Feishu inbound routing passes group-specific peer ids into route resolution:** Current Feishu message handling resolves a per-message `peerId`/`parentPeer` and calls `resolveAgentRoute` with that scoped peer, so different groups/topics do not share the same inbound route input.
  - file: [extensions/feishu/src/bot.ts:576](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/extensions/feishu/src/bot.ts#L576)
  - command: `nl -ba extensions/feishu/src/bot.ts | sed -n '576,606p'`
  - sha: [45146913007d](https://github.com/openclaw/openclaw/commit/45146913007da723b4f06aba6a66cf4a695974c0)
- **Current route cache is keyed by peer and parent peer:** `buildResolvedRouteCacheKey` includes `channel`, `accountId`, `peer`, `parentPeer`, guild/team info, roles, and dmScope, and cached hits return a cloned route object. That removes the obvious collision path for concurrent messages from different Feishu groups.
