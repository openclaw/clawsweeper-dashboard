## Summary

Current `main` already routes Control UI/webchat attachments through the canonical inbound media flow under `media/inbound`, and the local-media allowlist trusts the managed `media` roots. That directly addresses the reported mismatch where follow-up media reuse failed because attachments lived under `~/.openclaw/attachments`.

## Evidence

- **Control UI sends attachments as inline base64 payloads, not `~/.openclaw/attachments` paths:** The Control UI reads pasted/selected images with `FileReader.readAsDataURL(...)`, then `buildApiAttachments(...)` converts those data URLs into base64 `chat.send` attachments.
  - file: [ui/src/ui/views/chat.ts:229](https://github.com/openclaw/openclaw/blob/db8f7478b43550d2949c04b2a9c9ca7fd474dad3/ui/src/ui/views/chat.ts#L229)

  - file: [src/gateway/server-methods/chat.ts:676](https://github.com/openclaw/openclaw/blob/db8f7478b43550d2949c04b2a9c9ca7fd474dad3/src/gateway/server-methods/chat.ts#L676)
