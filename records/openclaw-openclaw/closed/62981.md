## Summary

Current main still matches the reported failure mode: the CLI waits for a final Gateway `agent` response, can time out after the Gateway has accepted and continued the run, then falls back to embedded execution with the original session selectors. The session-lock hardening in main reduces adjacent stale-lock failures, but it does not prevent this accepted-run-to-embedded-fallback race.

## What This Changes

The issue asks for Gateway-timeout embedded fallback to avoid reusing a session file that may still be locked by the accepted Gateway-side run.

## Best Possible Solution

Keep this open and fix the CLI/Gateway handoff so an accepted Gateway run that later times out cannot race an embedded fallback on the same transcript. Prefer surfacing the accepted `runId` to the CLI and using `agent.wait` or abort-and-wait before any embedded fallback; if a fresh session is chosen, make that session fork explicit in output and metadata. Add focused regression coverage for the accepted-then-timeout path and coordinate the broader dispatch-timeout behavior with #71605.

## Repair Work Prompt

Fix the Gateway-timeout embedded-fallback session lock bug tracked by openclaw/openclaw#62981, coordinating with openclaw/openclaw#71605 for the broader Gateway dispatch-timeout behavior. Current main sends `openclaw agent` through the Gateway with `expectFinal: true`; the Gateway client suppresses the accepted ack, the outer call can time out, and `agentCliCommand` then calls embedded `agentCommand` with the original session options, racing the still-running Gateway attempt on the same transcript lock. Implement a narrow fix so accepted Gateway runs do not immediately fall back into the same session. Add regression coverage in `src/commands/agent-via-gateway.test.ts` for an accepted Gateway run timing out before final response and prove embedded fallback does not reuse the locked session. Do not attempt broad Gateway dispatch architecture changes, generic session-lock rewrites, provider fallback changes, or MCP cleanup changes.
