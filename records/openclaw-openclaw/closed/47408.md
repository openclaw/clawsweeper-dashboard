## Summary

Current main already addresses the reported sandbox drift: sandbox exec now maps container workdirs like `/workspace` back to the host workspace before validation, managed `/workspace` mounts are applied explicitly, custom binds are restricted to allowed workspace roots unless a dangerous override is set, and stale sandbox containers are recreated when their config hash or mount format no longer matches. The fix is documented in the v2026.4.22 changelog.

## Evidence

- **Sandbox workdir mapping fixes the reported `/workspace` fallback:** `resolveSandboxWorkdir()` now maps container paths such as `/workspace` back to the host workspace before sandbox-path validation, instead of treating `/workspace` as an unavailable host path and falling back.
  - file: [src/agents/bash-tools.shared.ts:102](https://github.com/openclaw/openclaw/blob/0e7250f37bf347c72790abf2a6692d5ec84e5ea1/src/agents/bash-tools.shared.ts#L102)
  - sha: [0e7250f37bf3](https://github.com/openclaw/openclaw/commit/0e7250f37bf347c72790abf2a6692d5ec84e5ea1)
- **Managed workspace mounts are always attached for sandbox containers:** Sandbox container creation appends the managed workspace mount at the configured workdir and, when needed, a separate `/agent` mount; this is the runtime path that makes `/workspace` exist inside the container for `workspaceAccess: "rw"`.
  - file: [src/agents/sandbox/workspace-mounts.ts:14](https://github.com/openclaw/openclaw/blob/0e7250f37bf347c72790abf2a6692d5ec84e5ea1/src/agents/sandbox/workspace-mounts.ts#L14)
  - sha: [0e7250f37bf3](https://github.com/openclaw/openclaw/commit/0e7250f37bf347c72790abf2a6692d5ec84e5ea1)
- **Custom bind mounts are constrained to workspace roots by default:** Sandbox creation validates bind mounts and passes only the sandbox workspace and agent workspace as allowed source roots, so unrelated host paths are rejected unless an explicit dangerous override is enabled.
  - file: [src/agents/sandbox/docker.ts:482](https://github.com/openclaw/openclaw/blob/0e7250f37bf347c72790abf2a6692d5ec84e5ea1/src/agents/sandbox/docker.ts#L482)
  - sha: [0e7250f37bf3](https://github.com/openclaw/openclaw/commit/0e7250f37bf347c72790abf2a6692d5ec84e5ea1)
- **Stale/broken containers are detected and recreated:** `ensureSandboxContainer()` hashes docker config, workspace access, workspace paths, and mount format version; when the existing container hash does not match, OpenClaw removes and recreates the stale runtime unless it is still hot/recently used.
  - file: [src/agents/sandbox/docker.ts:524](https://github.com/openclaw/openclaw/blob/0e7250f37bf347c72790abf2a6692d5ec84e5ea1/src/agents/sandbox/docker.ts#L524)
  - sha: [0e7250f37bf3](https://github.com/openclaw/openclaw/commit/0e7250f37bf347c72790abf2a6692d5ec84e5ea1)
- **Regression coverage and shipped release note:** Tests cover `/workspace` mapping and stale-container recreation, and the v2026.4.22 changelog explicitly calls out both the sandbox workdir mapping fix and `/workspace` mount permission hardening.
  - file: [src/agents/bash-tools.shared.test.ts:16](https://github.com/openclaw/openclaw/blob/a94763ee224e858e29dd177cd6f7f4daa438c2f1/src/agents/bash-tools.shared.test.ts#L16)

## Risks / Open Questions

- This review is code-and-release based; I did not run a live Docker reproduction in this read-only environment.
- Hot recently-used sandbox containers are not forcibly replaced in place; current code logs a recreate hint instead, so an upgraded installation may still need `openclaw sandbox recreate ...` once to swap an old runtime.
