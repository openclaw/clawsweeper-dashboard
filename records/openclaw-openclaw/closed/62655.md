## Summary

Keep open. Current `main` still registers the bundled OpenRouter text-catalog auto model as bare `auto`, while the pinned Pi catalog uses `openrouter/auto` and the pinned Pi registry still merges by raw provider/id equality, so the reported picker/runtime mismatch remains.

## What This Changes

The issue asks to remove duplicate OpenRouter auto picker entries and avoid raw `auto` session overrides by making the bundled OpenRouter catalog use the canonical `openrouter/auto` id.

## Best Possible Solution

Canonicalize the bundled OpenRouter text catalog auto id to `openrouter/auto`, add regression coverage for catalog/picker dedupe, and separately normalize already-persisted bare session overrides if current-main testing proves they keep the retry loop alive.

## Reproduction Assessment

Yes, source-backed rather than live: current `main` still has the bare OpenRouter `auto` row, the pinned Pi catalog has `openrouter/auto`, the pinned Pi registry merges by raw id equality, and the Telegram picker persists the selected raw id. That is enough to reproduce the duplicate catalog path from code inspection, though no gateway UI run was performed.

## Solution Assessment

Yes. The narrow maintainable fix is to make the bundled OpenRouter text catalog use `openrouter/auto` and add focused regression coverage; broader Pi registry normalization can remain a defensive follow-up.
