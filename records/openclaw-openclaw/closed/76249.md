## Summary

Current main and the shipped v2026.5.7 release now bound the trajectory sidecar path that matched this report: runtime trajectory payloads are capped before serialization, queued sidecar writes are budgeted, live capture stops at 10 MiB, and the docs now distinguish trajectory sidecars from the primary active transcript JSONL guarded by compaction settings.

## What This Changes

The issue reports WhatsApp turns silently dying after `stream-ready` once the trajectory sidecar grows, and asks whether transcript byte compaction knobs should cap that sidecar.

## Best Possible Solution

Keep the shipped bounded trajectory-capture implementation, and treat `forceFlushTranscriptBytes` and `maxActiveTranscriptBytes` as active primary-transcript controls rather than trajectory sidecar controls.

## Reproduction Assessment

Yes, at source level. The reporter's live WhatsApp observations and the related trajectory-flush production report map to the embedded runner recording full trajectory events after `stream-ready`; current main now bounds that recorder path, though I did not rerun the exact WhatsApp setup.

## Solution Assessment

Yes. Bounding trajectory payload shaping, queued writes, and live sidecar capture is the narrow maintainable fix; using primary transcript compaction knobs to manage trajectory sidecars would blur two different storage contracts.
