## Summary

Current `main` shows `/avatar/<agentId>` is implemented and serves local workspace-relative avatars when the file exists inside the resolved agent workspace. The report matches a workspace-path/config mismatch or an invalid local avatar (for example missing file or >2 MB), not a missing/broken route on current OpenClaw.

## Evidence

- **Avatar route is wired on current main:** The gateway registers a dedicated `control-ui-avatar` stage and resolves avatars through `resolveAgentAvatar(configSnapshot, agentId, { includeUiOverride: true })` before handing off to the Control UI avatar handler.
  - file: [src/gateway/server-http.ts:1101](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/gateway/server-http.ts#L1101)
  - command: `nl -ba src/gateway/server-http.ts | sed -n '1088,1148p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Handler returns 200 for local avatar files:** `handleControlUiAvatarRequest` returns 404 only when `resolveAvatar()` is not local; otherwise it serves the file bytes for `/avatar/<agentId>`.
  - file: [src/gateway/control-ui.ts:576](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/gateway/control-ui.ts#L576)
  - command: `nl -ba src/gateway/control-ui.ts | sed -n '550,580p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Workspace-relative avatar resolution is implemented:** `resolveAgentAvatar()` reads avatar sources from UI/config/IDENTITY, then resolves non-URL avatars relative to `resolveAgentWorkspaceDir(cfg, agentId)`, rejecting only missing/out-of-workspace/unsupported/too-large files.
  - file: [src/agents/identity-avatar.ts:23](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/agents/identity-avatar.ts#L23)
  - command: `nl -ba src/agents/identity-avatar.ts | sed -n '1,104p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Default `main` workspace is `~/.openclaw/workspace`:** For the default agent with no explicit workspace configured, `resolveAgentWorkspaceDir()` falls back to `resolveDefaultAgentWorkspaceDir()`, which resolves to `~/.openclaw/workspace`. That does not match `~/.openclaw/agents/main/workspace/...` or `~/.openclaw/media/...` from the report.
  - file: [src/agents/agent-scope-config.ts:159](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/agents/agent-scope-config.ts#L159)
  - command: `nl -ba src/agents/agent-scope-config.ts | sed -n '149,171p' && nl -ba src/agents/workspace.ts | sed -n '17,27p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Docs and tests match the implemented behavior:** Docs say avatar paths resolve relative to the workspace root, and current tests cover both local avatar resolution and a successful `GET /avatar/main` response for a local file.
  - file: [docs/cli/agents.md:156](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/docs/cli/agents.md#L156)

## Risks / Open Questions

- There is still a separate UX/docs gap: `resolveAssistantAvatarUrl()` can advertise `/avatar/<agentId>` before checking disk existence (`src/gateway/control-ui-shared.ts:35`), and the 2 MB local-avatar limit is enforced in code (`src/shared/avatar-policy.ts:4`) but not surfaced in the docs inspected here.
