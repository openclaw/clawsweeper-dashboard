## Summary

PR #62449 is now obsolete because current main already solves the stale date-rolled log-file bug it targets. The current implementation re-resolves the active rolling file on each write, preserves configured rolling-log directories, prunes/updates byte accounting on rollover, avoids the old indefinitely cached subsystem child logger, and has regression coverage for long-lived subsystem loggers crossing days.

## Best Possible Solution

Close PR #62449 as implemented on current main. Keep the current main logger implementation and regression tests; no code from this PR is needed unless maintainers want a separate, narrower performance optimization around per-emit child logger creation.

## Evidence

- **PR discussion identified a real bug:** The provided PR and linked #62381 context report long-running subsystem logs with April 7 timestamps still being written into an April 3 dated file. The Greptile review also treated the PR as a correct, narrow fix, with follow-up comments around pruning, configured rolling directories, byte accounting, and test coverage. Current main covers those same concerns with a broader implementation.
- **Write-time rolling file resolution:** `buildLogger()` initializes `activeFile` from `resolveActiveLogFile(settings.file)`, then recomputes `nextActiveFile` inside the attached transport before every write. When the date-derived file changes, it switches `activeFile`, recreates the directory, prunes stale rolling logs, refreshes the byte count, and appends to the new active file.
