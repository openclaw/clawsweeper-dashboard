## Summary

Keep open. The PR targets a real remaining directory-sweep gap, but the apply path introduces a destructive concurrency race by sweeping files after releasing the session-store writer while using a stale store snapshot.

## What This Changes

The PR adds orphaned session transcript, trajectory, and compaction-checkpoint sweeping to `sessions cleanup`, reports the result in summaries, and adds focused disk-budget tests plus a changelog entry.

## Best Possible Solution

Move orphan sweeping into a race-safe per-store cleanup path that uses a fresh store snapshot under the writer or an equivalent current-store/mtime guard, then coordinate the non-archive orphan sweep with the broader archive cleanup tracked in #75658/#75672.

## Reproduction Assessment

Yes. A code-level reproduction is gateway `sessions.cleanup` loading `afterStore`, then a concurrent session write persisting a new entry and creating its transcript before the sweep scans the directory; the stale referenced set then treats the new file as orphaned.

## Solution Assessment

No. The proposed directory scan is in the right feature area, but destructive deletes must be guarded by the same store-writer/fresh-state contract that current session maintenance uses.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Hold the store writer while sweeping orphans:** `src/config/sessions/cleanup-service.ts:289-294`
  - body: This sweep runs after `updateSessionStore` returns, so the per-store writer has been released while `afterStore` is only a snapshot. In the gateway path, another session can persist a new entry and create its `.jsonl` or trajectory file between that snapshot and the directory scan; because the referenced set is stale, the new active artifact is deleted as orphaned. Run the scan/delete under the writer with a freshly loaded store, or add an equivalent current-store/mtime guard before removal.
  - confidence: 0.88
