## Summary

Current main does not support this as an exec-core bug. Exec approval registration intentionally uses an `operator.approvals`-scoped gateway connection, current auth/docs point to `OPENCLAW_GATEWAY_TOKEN` rather than `OPENCLAW_TOKEN`, and current pairing diagnostics now surface scope-upgrade/token-drift cases explicitly with `devices list/approve/rotate` remediation.

## Evidence

- **Exec approval registration is a gateway approval RPC:** `exec` approval registration calls `exec.approval.request`, so approval-required commands depend on gateway approval auth rather than only local safe-bin execution.
  - file: [src/agents/bash-tools.exec-approval-request.ts:89](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/agents/bash-tools.exec-approval-request.ts#L89)
  - command: `sed -n '1,340p' src/agents/bash-tools.exec-approval-request.ts`
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **Approval RPCs require the approvals scope:** `exec.approval.request`/`waitDecision`/`resolve` are classified under `APPROVALS_SCOPE`, and the dedicated approvals client connects with `scopes: ["operator.approvals"]`. A paired device without that approved scope will legitimately hit pairing/scope-upgrade auth on this path.
  - file: [src/gateway/method-scopes.ts:42](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/gateway/method-scopes.ts#L42)

- **Gateway auth env is OPENCLAW_GATEWAY_TOKEN:** Current docs and tests show gateway tool auth resolving from `OPENCLAW_GATEWAY_TOKEN`, not `OPENCLAW_TOKEN`. The issue body cites only `OPENCLAW_TOKEN`, which does not match the documented gateway auth path.
  - file: [docs/cli/node.md:68](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/docs/cli/node.md#L68)

- **Current main surfaces scope-upgrade pairing state explicitly:** `PAIRING_REQUIRED` is now formatted as specific messages like `scope upgrade pending approval (requestId: ...)`, and tests cover that exact recovery path instead of a bare generic pairing error.
  - file: [src/gateway/protocol/connect-error-details.ts:272](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/gateway/protocol/connect-error-details.ts#L272)

  - file: [docs/gateway/doctor.md:407](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/docs/gateway/doctor.md#L407)
