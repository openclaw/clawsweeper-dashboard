## Label Justifications

## Summary

Keep open: current source has plugin API compatibility guards, but the downgrade path still appears to run post-core plugin sync in the pre-downgrade process without setting the target host version, so an installed newer diagnostics-prometheus package can still be evaluated against the wrong OpenClaw core before gateway restart.

## What This Changes

The issue reports that a newer diagnostics-prometheus plugin can remain installed after rolling OpenClaw core back, causing repeated log.record handler failures against the older plugin SDK runtime.

## Best Possible Solution

Make rollback post-core plugin convergence evaluate compatibility against the target downgraded core version and prevent an incompatible enabled plugin from loading before the gateway restarts.

## Reproduction Assessment

Yes, source-reproducible: the issue gives concrete rollback steps and logs, and current source shows downgrade post-core plugin work can use the pre-downgrade runtime version unless the target host version is explicitly set. I did not run a live rollback in this read-only review.

## Solution Assessment

No, suppressing repeated handler errors alone would leave incompatible plugin code loaded. The cleaner fix is compatibility enforcement during rollback/plugin convergence, with handler suppression only as a secondary hardening if maintainers want it.
