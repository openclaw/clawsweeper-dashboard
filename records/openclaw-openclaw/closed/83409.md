## Summary

Current main already supports the requested multi-user DM isolation through the shipped `session.dmScope` routing contract, with per-session queueing and isolated sub-agent sessions covering the concurrency and background-task parts of the request well enough.

## What This Changes

The issue requests per-user persistent DM sessions, overflow queueing, and isolated group-chat subtasks for team deployments.

## Best Possible Solution

Keep the shipped generic `session.dmScope` routing, queue, and sub-agent isolation contracts instead of adding a parallel channel-specific `channels.dm.routing` source of truth.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken-behavior report. Current source and docs show the supported current-main path through `session.dmScope`, session-keyed queues, and isolated sub-agent sessions.

## Solution Assessment

Yes: the existing generic session-routing contract is the narrower maintainable solution because it works across channels and avoids a second channel-specific DM routing config surface.
