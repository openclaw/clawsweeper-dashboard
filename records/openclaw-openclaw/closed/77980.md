## Summary

Keep open: current main still lacks dedicated bootstrap-token and device-signature limiter coverage before expensive pre-auth work, and the umbrella is paired with open fix PRs plus unresolved lockout/failed-verify alerting policy.

## What This Changes

Requests rate limiting, lockout, and failed-verify storm alerting for the gateway bootstrap-token and device-signature pre-auth paths.

## Best Possible Solution

Land or replace the narrow gateway auth hardening PRs, then define any remaining failed-verify lockout and alerting policy in the gateway auth contract.

## Reproduction Assessment

Yes. Source inspection on current main shows a signed device handshake with a bogus bootstrap token can reach verifyBootstrapToken before a bootstrap-token limiter, and device-signature verification runs before any device-signature-specific limiter.

## Solution Assessment

Unclear as a complete solution. The linked per-surface limiter PRs are the right narrow direction, but the umbrella still needs an explicit policy for always-on lockout and failed-verify storm alerting.
