Keep open: current main still has a source-visible LINE recipient routing bug, and two open PRs already reference this issue with closing syntax, so the issue should stay open until one fix lands.

## What This Changes

The issue reports LINE post-reply-token delivery and recovery paths sending lowercased or prefixed recipients to the LINE Messaging API, causing silent push failures.

## Best Possible Solution

Review and land one narrow fix, preferably the proof-backed open PR path, then close this issue with the merged commit and release provenance.

## Reproduction Assessment

Yes for a source-level reproduction: current main lowercases LINE peer IDs into session keys, cron fallback can copy that fragment into delivery.to, and recovery replays entry.to verbatim. No live LINE API reproduction was run in this read-only review.

## Solution Assessment

Yes, the best direction is a narrow channel-owned LINE recipient guard plus avoiding unrecoverable LINE session-key fallback, with focused cron/recovery/send regression coverage. The issue should not close until one open closing PR is merged.
