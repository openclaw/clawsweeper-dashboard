## Summary

Current `main` already blocks alias/junction escapes for write targets before any `open()` call, and that implementation shipped in `v2026.4.22`, so this PR is redundant.

## Evidence

- **Write path is prevalidated on main:** `openWritableFileWithinRoot()` calls `assertNoPathAliasEscape()` before `mkdir` and before any write-side `fs.open()`, so alias/junction escapes are rejected earlier than the PR’s proposed Windows-only parent-`realpath` branch.

  - sha: [c13836804058](https://github.com/openclaw/openclaw/commit/c1383680405879a3c3f43dceb20a694f54d3c13c)
- **Alias guard resolves missing-leaf paths through existing aliases:** `assertNoPathAliasEscape()` delegates to `resolveBoundaryPath()`, which canonicalizes the path through existing segments and applies boundary checks even when the final leaf does not exist yet.
  - file: [src/infra/path-alias-guards.ts:18](https://github.com/openclaw/openclaw/blob/c1383680405879a3c3f43dceb20a694f54d3c13c/src/infra/path-alias-guards.ts#L18)

## Risks / Open Questions

- Current direct regression tests for this alias-resolution path are POSIX-only, so the closure relies on the shared alias/junction resolution logic rather than a Windows-specific test in this file.
- The proposed PR branch also had an uncaught-ENOENT regression called out in review comments, so merging this exact patch would not improve the current released implementation.
