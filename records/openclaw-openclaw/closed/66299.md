## Repair Work Prompt

Fix the ACP resolver fallback described by this issue. Current main returns `Unable to resolve session target: <token>` when a non-empty token does not resolve, which prevents `/acp close <non-session-token>` and related ACP commands inside a bound thread from using the thread binding. In `src/auto-reply/reply/commands-acp/targets.ts`, keep successful explicit token resolution first; when token resolution returns null, fall through to `resolveBoundAcpThreadSessionKey`; if no bound target resolves and a token was provided, preserve the existing `Unable to resolve session target: ${token}` diagnostic; keep requester fallback behavior for no-token cases. Add focused regressions in `src/auto-reply/reply/commands-acp.test.ts` near the existing bound-thread close test: `/acp close <non-session-token>` in a bound thread should call closeSession and unbind, and a bad token with no binding should still return the explicit bad-token error without closing. Add a single-line user-facing fix entry under `CHANGELOG.md` Unreleased/Fixes with appropriate `Thanks @...` attribution, likely `Thanks @kindomLee` plus any implementation credit. Before starting, check whether this issue or PR #66338 now has an open replacement PR; if so, do not create a duplicate. Do not change command parsing, sessions.resolve behavior, ACP docs semantics, or add a config flag unless a maintainer asks.
