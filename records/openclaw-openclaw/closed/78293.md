## Summary

Current main and the latest release already route empty or incomplete successful embedded results through the model fallback classifier, so the reported 2026.5.3 behavior is fixed by the shipped fallback boundary.

## What This Changes

The issue asks for successful empty provider turns to advance to configured model fallbacks instead of returning the generic incomplete-response error.

## Best Possible Solution

Keep the shipped v2026.5.4 classifier-based fallback behavior; affected 2026.5.3 deployments should upgrade rather than adding another runner-specific fallback hint.

## Reproduction Assessment

No failing current-main reproduction remains: the current source classifies the empty/incomplete terminal result before runWithModelFallback accepts success. The report gives a plausible 2026.5.3 intermittent repro, but v2026.5.4 includes the fix path and regression coverage.

## Solution Assessment

Yes. The current classifier boundary is the narrow maintainable fix because it lets the fallback runner decide candidate advancement while preserving embedded-runner side-effect guards.
