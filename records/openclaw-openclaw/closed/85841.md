## Label Justifications

- `P2`: The reported behavior was a bounded exec approval routing bug that could amplify heartbeat cost without making the core runtime unusable.
- `impact:security`: The issue is about whether host exec authorization prompts have a reachable approval surface or should fail closed.

## Summary

Current main and the v2026.5.20 release already classify heartbeat as an internal non-delivery source, so heartbeat-originated exec approvals with no operator/native delivery path now hit the shared no-approval-route expiration instead of waiting for the 30-minute approval timeout.

## What This Changes

The issue requests fail-fast behavior for heartbeat-originated exec approval requests when no approver surface can receive the approval.

## Best Possible Solution

Keep the shipped non-delivery heartbeat classification and shared no-approval-route expiration; treat connected operator approval clients as the documented reachable approval path.

## Reproduction Assessment

No high-confidence current-main failing reproduction remains from source inspection: heartbeat is a non-delivery source, and the shared approval handler expires no-route requests immediately when no operator/native delivery path exists. I did not run a live heartbeat gateway scenario because this review is read-only.

## Solution Assessment

Yes. The current solution keeps the ownership boundary generic by classifying heartbeat as an internal non-delivery source and reusing the existing no-approval-route expiration instead of adding heartbeat-specific timeout policy.
