## Summary

Keep open: this targets a real Gateway/update stale-dist failure class, but the current branch is not merge-ready because supported wrapper services can bypass the guard, the auto-stop path can leave the Gateway down on failed or skipped git updates, and the supplied proof does not exercise the changed git-mode path.

## What This Changes

The PR adds a git-mode update preflight that detects a same-install running Gateway service, auto-stops it before in-place dist replacement when restart is enabled, blocks `--no-restart`, and adds update CLI tests.

## Best Possible Solution

Land a git-mode guard integrated with the existing managed-service lifecycle: resolve wrapper-backed services, track stopped services through every failure/skip path, correct `--no-restart` guidance, satisfy core typecheck, and prove a real git-mode same-install Gateway run.

## Solution Assessment

No. The PR is aimed at the right lifecycle boundary, but it needs wrapper-aware detection, failure/skip recovery, corrected guidance, and real git-mode proof before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Handle wrapper-backed gateway commands:** `src/cli/update-cli/update-command.ts:986-988`
  - body: `findGatewayServiceEntrypoint` assumes the token before `gateway` is the dist entrypoint. Supported wrapper installs use `programArguments: [wrapperPath, "gateway", ...]`, so this compares the wrapper to `dist/index.js` and allows the same running install to update.
  - confidence: 0.91
- **[P1] Recover the gateway after failed or skipped git updates:** `src/cli/update-cli/update-command.ts:3327-3358`
  - body: The PR stops the service before `runGitUpdate`, then only records `gatewayStoppedForUpdate` without using it in the error/skipped branches. A dirty checkout, failed fetch, or failed rebase can leave the previously running Gateway down with no update applied.
  - confidence: 0.9
- **[P2] Remove restart from the pre-update block guidance:** `src/cli/update-cli/update-command.ts:3341-3347`
  - body: In the `--no-restart` block, telling users to run `openclaw gateway restart` before retrying leaves the service running and triggers the same block again. The guidance should be stop, update, then restart after the update.
  - confidence: 0.86
