## Summary

Current `main` already implements the requested config rollback safety net via validated config writes, destructive-write rejection, rotating config backups, and automatic last-known-good recovery on startup/hot reload.

## Close Comment

Closing from a Codex review because current `main` already covers this request’s safety goal. Config writes now validate the full post-change config before commit and reject unresolved/bad payloads (`src/gateway/server-methods/config.ts:471`), destructive writes are refused and preserved as `openclaw.json.rejected.*` instead of clobbering the live config (`src/config/io.ts:1805`), config backups now keep a built-in 5-deep rotation ring (`src/config/backup-rotation.ts:3`), and gateway startup automatically restores the last-known-good config when a bad edit would otherwise prevent boot (`src/gateway/server-startup-config.ts:72`, `src/config/io.observe-recovery.ts:1008`). The operator docs also describe the recovery flow and the `.clobbered.*` / `.rejected.*` artifacts (`docs/help/faq.md:1640`, `docs/gateway/configuration.md:87`). The exact CLI names proposed here were not added verbatim, but the rollback-protection mechanism itself is implemented on `main` at `59a8afe6fa030f18af662e125e7dd826ce9a53ca`.
