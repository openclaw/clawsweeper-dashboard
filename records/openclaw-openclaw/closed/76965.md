## Summary

Current main already covers the reported duplicate WebChat output path with active-run dedupe in the Gateway plus regression coverage, and the issue was reported against an older 2026.4.26 build.

## What This Changes

The issue asks for Control UI/webchat to render each assistant response once instead of intermittently showing duplicate visible output.

## Best Possible Solution

Keep the main-branch Gateway dedupe and UI in-flight guard; if duplicate output reappears on a newer build, reopen with server/client stream evidence for a distinct rendering or reconciliation path.

## Reproduction Assessment

Yes, at source/test level: the Gateway regression test exercises duplicate WebChat text sends while one run is in flight and verifies they collapse to one dispatch. I did not run the test because this review is read-only and `pnpm` is unavailable in the environment.

## Solution Assessment

Yes. The current fix is narrow and maintainable: the Gateway collapses duplicate active internal WebChat sends, while the Control UI already suppresses duplicate local sends awaiting acknowledgement.
