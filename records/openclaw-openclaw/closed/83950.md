## Label Justifications

- `P1`: Default Telegram polling can silently lose its configured stall recovery path for real channel workflows.
- `impact:message-loss`: A silent isolated polling worker can leave inbound Telegram updates undelivered without the configured watchdog firing.

## Summary

Keep open: current main still defaults Telegram polling into isolated ingress while the configured stall threshold is only consumed by the embedded polling branch. The linked spool-drain fix addressed a different isolated-ingress hang mode and did not wire the parent-worker liveness watchdog to `pollingStallThresholdMs`.

## What This Changes

Honor `channels.telegram.pollingStallThresholdMs` in the default Telegram isolated polling ingress path so the configured watchdog also restarts silent worker stalls.

## Best Possible Solution

Honor the existing documented threshold inside isolated ingress by tracking worker message liveness and restarting the worker/bot cycle on the configured threshold, with focused regression coverage.

## Reproduction Assessment

Yes, at source level: current main defaults into isolated ingress, and the only `#stallThresholdMs` consumer is the embedded polling watchdog. I did not run a live Telegram stall because this review is read-only.

## Solution Assessment

Yes. Wiring the existing documented setting into the default isolated path is narrower and more maintainable than documenting it as embedded-only or adding a public opt-out just to recover old behavior.
