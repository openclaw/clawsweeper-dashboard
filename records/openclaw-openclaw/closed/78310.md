## Summary

Keep open: the narrowed Feishu topic hydration addresses a real split-session path, but the PR still performs the new Feishu message lookup before group and sender admission gates, so ignored group events can trigger authenticated reads.

## What This Changes

The branch hydrates missing Feishu native topic starter thread IDs before group-session routing, updates Feishu channel docs and changelog, and adds a regression test for stable topic session keys.

## Best Possible Solution

Keep the topic hydration approach, but move it after disabled-group, group allowlist, and sender allowlist checks while still before `resolveFeishuGroupSession` and history-key selection; add a negative test proving non-admitted group events do not call `getMessageFeishu`.

## Reproduction Assessment

Yes by source inspection: current main can route a native topic starter without `thread_id` by `messageId`, while later replies with `thread_id` route by the canonical `omt_*` key. I did not run tests because this review is read-only.

## Solution Assessment

No as-is: hydrating before session routing is the right narrow fix, but the API read should be gated behind group and sender admission checks and covered by a negative authorization regression test.
