## Summary

Current `main` already provides a supported, documented way to run operator-managed deployments without per-device identity by using identity-bearing HTTP auth (`gateway.auth.mode: "trusted-proxy"`) and trusted operator scope handling, so the requested capability is implemented even though it does not use a `fleetMode` config key.

## Evidence

- **Trusted HTTP operator scopes are implemented in code:** `resolveTrustedHttpOperatorScopes()` trusts declared scopes only for trusted identity-bearing HTTP requests and falls back to the default operator scope set when `x-openclaw-scopes` is absent.
  - file: [src/gateway/http-utils.ts:204](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/src/gateway/http-utils.ts#L204)
  - command: `nl -ba src/gateway/http-utils.ts | sed -n '204,235p'`
- **Trusted-proxy auth is documented as the supported config path:** The trusted-proxy docs explicitly document `x-openclaw-scopes` for identity-bearing HTTP requests and state that normal identity-bearing HTTP APIs fall back to standard operator default scopes when the header is absent.
  - file: [docs/gateway/trusted-proxy-auth.md:271](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/docs/gateway/trusted-proxy-auth.md#L271)

## Risks / Open Questions

- There is no dedicated `gateway.fleetMode` key; the supported path is `gateway.auth.mode: "trusted-proxy"` plus documented HTTP scope semantics.
- This evidence supports identity-bearing HTTP/operator adapter deployments. WebSocket Control UI behavior remains narrower in some cases, so a different protocol-specific gap would need a new issue.
