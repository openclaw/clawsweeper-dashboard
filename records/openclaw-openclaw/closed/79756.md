## Summary

Keep open and do not merge as-is: the branch starts channels before runtime plugin loading pins the channel registry, so a cold gateway can enumerate zero channel plugins and then skip the normal later channel start. The contributor’s follow-up acknowledges that blocker, and the linked issue remains open for a dependency-aware gateway startup fix.

## What This Changes

The PR reorders gateway post-attach startup by starting channels before `plugins.runtime-post-bind` and making the later sidecar phase await that early channel promise.

## Best Possible Solution

Introduce a dependency-aware startup path that makes channel runtime registry state available before channel start while allowing slow ACPX-dependent runtime work to continue independently.

## Reproduction Assessment

Yes for the PR defect by source inspection: the patch calls `startChannels()` before runtime plugin loading pins the channel registry, while `startChannels()` enumerates plugins from that registry. I did not run the PR branch or the Windows ACPX cold-boot path.

## Solution Assessment

No: this one-file reorder is not the best fix because it can start channels against an empty registry and then suppress the normal later start. The safer path is an explicit registry-ready or channel-runtime-loading seam before channel startup.
