## Summary

Keep open: current main still loses the explicit cron timeout signal when the payload value equals `agents.defaults.timeoutSeconds`, and this PR targets that path, but it still needs after-fix real cron-run proof before merge.

## What This Changes

The branch adds a `runTimeoutOverrideMs` signal from isolated cron setup through the embedded Pi runner, uses it for LLM idle timeout resolution, adds cron regression tests, and updates the changelog.

## Best Possible Solution

Land the narrow explicit-timeout signal plumbing after maintainer review and redacted real cron proof showing a payload timeout equal to the agent default no longer falls back to the 120 second idle watchdog.

## Reproduction Assessment

Yes, source-level. A cron payload timeout equal to `agents.defaults.timeoutSeconds` resolves to the same `timeoutMs`, the embedded attempt drops `runTimeoutMs`, and the resolver then uses the capped implicit default branch.

## Solution Assessment

Yes, the proposed direction is the narrow maintainable fix: carry the prepared explicit-run-timeout fact from cron into the embedded runner while preserving the old heuristic for unmigrated callers. Merge should still wait for real behavior proof.
