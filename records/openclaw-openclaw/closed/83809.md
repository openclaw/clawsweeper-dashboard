## Summary

Keep open: current main still has only a private Claude live-session registry and internal stream-json stdin write path, while the requested external-process injection API/UDS listener is a new security-sensitive runtime contract that needs maintainer design.

## What This Changes

The issue requests a supported live-session user-message injection surface, optionally backed by a local Unix-domain socket, so external orchestrators can wake an active TUI session through the existing Claude stream-json child.

## Best Possible Solution

Define a narrow authenticated opt-in Gateway or live-runtime contract for orchestrator wake delivery, or explicitly keep queued system events and heartbeat as the supported path.

## Reproduction Assessment

## Solution Assessment

Unclear: reusing the existing serializer is plausible, but exposing it through an API or UDS listener needs a product and security decision first. A documented, authenticated, opt-in runtime contract is safer than exporting a private Claude-only helper ad hoc.
