## Summary

Keep open. Current main has several restart hardenings, but it does not satisfy this report’s central guardrail: a systemd-supervised gateway that loses a port/lock race can still complete with exit 0 or exit 1 rather than sysexits 78, so `RestartPreventExitStatus=78` is not reliably used to stop a `Restart=always` loop.

## What This Changes

The issue requests that Linux systemd gateway restart collisions avoid an EADDRINUSE respawn loop, preferably by exiting 78 on address-in-use startup failures or by delegating restart control to systemd.

## Best Possible Solution

Add a Linux systemd-specific startup/restart guard so a supervised gateway process that loses the port or gateway-lock race exits with sysexits 78 and logs a clear message, while preserving current unmanaged and non-systemd behavior. Cover both the direct GatewayLockError path and the supervised-lock recovery helper with targeted regression tests.

## Reproduction Assessment

Yes, at code level: simulate a systemd-supervised startup where the lock path reports an existing gateway or the listener reports EADDRINUSE, and current main returns/exits 0 or 1 rather than 78. I did not run a live WSL2 systemd restart because this review had to remain read-only and avoid mutating runtime state.

## Solution Assessment

Yes, the narrowest maintainable fix is to honor the already-rendered `RestartPreventExitStatus=78` contract for systemd-supervised port/lock conflicts. Delegating every restart to `systemctl restart` from inside the gateway is broader and riskier because the service-manager restart path already exists in the CLI.
