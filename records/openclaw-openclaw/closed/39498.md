## What This Changes

The PR changes generic `normalizeSecretInputString` to call `normalizeOptionalSecretInput` and adds direct tests for trimming, blank input, non-Latin1 stripping, and Latin-1 preservation.

## Best Possible Solution

Keep the shipped header/API-key sanitization and use narrow follow-ups for any remaining header-bound credential path, while leaving generic SecretInput string handling trim-only.

## Reproduction Assessment

## Solution Assessment

No. The branch applies the right sanitizer at the wrong layer; current main's targeted API-key/header normalization is safer, with any residual header surfaces needing narrow fixes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve Unicode secrets outside HTTP headers:** `src/config/types.secrets.ts:116`
  - body: Delegating `normalizeSecretInputString` to `normalizeOptionalSecretInput` strips every code point above `0xff` for all SecretInput consumers. Non-header credentials such as Matrix passwords are sent in JSON bodies, so valid Unicode passwords can be mutated before authentication; keep generic normalization trim-only and sanitize header values at header-bound use sites.
  - confidence: 0.91
- **[P2] Keep presence checks independent from header sanitization:** `src/config/types.secrets.ts:116`
  - body: `hasConfiguredSecretInput` depends on `normalizeSecretInputString` as a presence check. With this change, an all-non-Latin1 configured value normalizes to `undefined`, so real credentials can be treated as absent and take unrelated missing-secret or SecretRef paths.
  - confidence: 0.87

## Security Review

Status: needs_attention

  - confidence: 0.86
