## Summary

Keep this issue open. Current main still models OPENCLAW_PLUGIN_STAGE_DIR as one writable bundled plugin runtime-dependency stage root, not a layered read-only baseline plus final writable install root. The issue discussion describes a concrete container/proxy-restricted deployment need that belongs in core runtime-deps staging rather than ClawHub.

## Best Possible Solution

Keep this issue open and implement it as a core bundled plugin runtime-dependency staging enhancement: parse a documented ordered stage-root list, resolve sentinels left-to-right across immutable baseline roots plus the final writable root, install and repair only into the writable root, register aliases for the actual satisfying roots, keep cache keys deterministic across OpenClaw version and runtime-deps manifests, and update Docker/systemd/offline deployment docs and focused tests. Treat #71730 and #72263 as related single-root repair history, not canonical trackers for layered stage directories.

## Risks / Open Questions

- A parser-only change would be unsafe: sentinel lookup, retained manifests, install locks, doctor repair, loader alias registration, runtime mirroring, and startup health all currently assume one install root.
- Path-list semantics need deliberate POSIX and Windows handling because path.delimiter differs and OPENCLAW_PLUGIN_STAGE_DIR currently accepts the explicit value as an opaque path.
