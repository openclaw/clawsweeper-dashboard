## Summary

Current main already implements Z.AI thinking control through the Z.AI provider stream wrapper rather than the generic OpenAI `thinkingFormat` branch. The wrapper sends `thinking: { type: "disabled" }` when OpenClaw thinking is off and sends `thinking: { type: "enabled", clear_thinking: false }` for opt-in preserved thinking; tests cover both payload shapes. Normal `/think on` does not need an explicit `enabled` payload because the current Z.AI chat-completion contract defaults `thinking.type` to `enabled`.

## Best Possible Solution

Close this as implemented on current main. Keep the provider-owned Z.AI wrapper and docs as the canonical behavior: ordinary thinking-on relies on Z.AI's default enabled mode, thinking-off sends the explicit disabled payload, and preserved thinking remains opt-in through `preserveThinking`.
