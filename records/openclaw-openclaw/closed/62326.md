## Summary

Current `main` already routes `openclaw devices ...` through the built-in subcommand path instead of plugin CLI discovery, and the plugin loader now has explicit same-snapshot reentry guards with regression tests. That matches the linked `loadOpenClawPlugins`/`recordPluginError` failure mode and is strong enough to close this older 2026.4.5 report as implemented on current code.

## Evidence

- **`devices` is a built-in subcommand:** The core subcommand registry declares `devices` as an imported built-in subcli, so it can be loaded directly without going through plugin CLI discovery.
  - file: [src/cli/program/register.subclis-core.ts:99](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/src/cli/program/register.subclis-core.ts#L99)

- **CLI startup skips plugin registration for built-in primaries:** `run-main` computes `hasBuiltinPrimary` and skips `registerPluginCliCommandsFromValidatedConfig(...)` when the requested command is already a built-in primary. That covers `openclaw devices list` and avoids the plugin-loader path implicated by the related stack-overflow report.
  - file: [src/cli/run-main.ts:255](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/src/cli/run-main.ts#L255)

  - file: [src/cli/command-registration-policy.test.ts:17](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/src/cli/command-registration-policy.test.ts#L17)
