## Label Justifications

- `P2`: This is a normal-priority security/policy improvement request with limited blast radius because the core hook seam is already shipped.
- `impact:security`: The request concerns enforcing tool/exec policy boundaries and auditing or blocking unsafe command execution.

## Summary

Current main already provides the requested pre-exec command interception through plugin `before_tool_call` hooks, including block, parameter rewrite, approval, and blocked-result behavior before `exec` or other tools execute. The exact declarative YAML helper would be optional plugin/config UX on top of the shipped core hook seam, not missing core infrastructure.

## What This Changes

Add a plugin hook system that intercepts exec commands before execution and can block, warn, redirect, and log policy violations.

## Best Possible Solution

Keep the shipped `before_tool_call` plugin hook as the generic core interception seam; handle declarative pattern packs or runtime-path bugs as separate plugin or narrow follow-up items.

## Reproduction Assessment

Not applicable: this is a feature request rather than a failing behavior report. The current docs, hook types, runtime wrapper, and regression tests give high-confidence proof that the requested core interception seam exists.

## Solution Assessment

Yes for the central request: the generic typed `before_tool_call` hook is the maintainable core solution because it covers all tools without adding a one-off exec-only config path. A declarative pattern matcher can be layered as a plugin if maintainers want that UX.
