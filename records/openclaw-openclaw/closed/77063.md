## Summary

Close as implemented on current main: the gateway startup plan now includes the selected non-legacy `plugins.slots.contextEngine` plugin before agent turns resolve the active engine, with regression coverage for the `lossless-claw` no-`activation.onStartup` case.

## What This Changes

Request that an enabled selected `lossless-claw` context-engine plugin register at gateway startup, or otherwise surface a clear load failure instead of repeatedly falling back to `legacy`.

## Best Possible Solution

Keep the current startup-plan behavior and ship it in the next post-v2026.5.2 release so selected context-engine plugins load before runtime engine resolution.

## Reproduction Assessment

Yes, from source inspection rather than a live macOS gateway run: the reported log is the registry fallback for a selected engine that was not registered, and the current regression fixture models a selected enabled `lossless-claw` plugin without `activation.onStartup`. Current main no longer has the startup-plan gap that made that path reachable.

## Solution Assessment

Yes: loading only the selected non-legacy context-engine slot plugin during gateway startup is the narrow maintainable fix, because it avoids broad-loading every enabled context-engine plugin while ensuring the configured engine can register before first use.
