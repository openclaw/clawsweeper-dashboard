## Summary

Keep open, but do not merge as-is: the PR targets a real source-backed performance bug, but the proposed hidden 5-second metadata snapshot cache conflicts with the documented plugin cache boundary and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR adds a bounded TTL cache inside `loadPluginMetadataSnapshot()`, wires cache clearing through the current metadata lifecycle, and adds cache-focused regression tests.

## Best Possible Solution

Fix the hot Gateway/subagent paths by passing a compatible current `PluginMetadataSnapshot`, `PluginLookUpTable`, or explicit manifest registry through the call chain, while keeping direct metadata reads fresh after config, workspace, install, or manifest changes.

## Reproduction Assessment

Yes, source inspection gives a credible current-main reproduction path: direct hot callers still invoke synchronous `loadPluginMetadataSnapshot()` rebuilds instead of reusing an explicit current snapshot. I did not live-run the reported Control UI freeze, so this is source-reproducible rather than fully reproduced.

## Solution Assessment

No: the PR's TTL cache is not the best fix because it violates the documented freshness contract for plugin metadata. The safer maintainable path is explicit snapshot or lookup-table reuse in the hot call chains that currently bypass it.
