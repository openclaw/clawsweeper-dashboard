## Summary

Keep open. Current main and the latest release still use the stale `lastAssistant` snapshot for the incomplete-turn tool-use guard and still replace the collected final answer with the generic incomplete-turn error when that branch fires; only the text-only deliverability part has been fixed.

## What This Changes

The issue asks OpenClaw to stop dropping a completed post-`update_plan` final answer when stale tool-use state makes the embedded runner misclassify the turn as incomplete.

## Best Possible Solution

Use the freshest terminal assistant state for incomplete-turn classification, final assistant metadata, and payload construction, then add the stale-`lastAssistant` regression while keeping broader recovery and observability ideas separate.

## Reproduction Assessment

Yes for source-level reproduction: an attempt with stale `lastAssistant.stopReason="toolUse"`, fresh `currentAttemptAssistant.stopReason="stop"`, and non-empty `assistantTexts` still follows the stale classifier path on current main. I did not run the full live WhatsApp/OpenRouter path in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is to select `currentAttemptAssistant ?? lastAssistant` at the classifier, payload, and final-metadata boundaries. The broader alarm, failed-queue, and recovery-helper ideas should be tracked separately from the message-loss fix.
