- `P0`: The reported historical behavior silently lost persisted cron jobs, which is a data-loss class issue even though current main already fixes it.

## Summary

Close as implemented: current main and the latest release use SQLite-backed cron storage, doctor-owned legacy JSON migration, and quarantine handling, so the reported split JSON hot-reload race is no longer on the active runtime path.

## What This Changes

The issue reports a shipped bug where legacy split JSON cron persistence could drop main-session jobs during gateway hot reload.

## Best Possible Solution

Keep the shipped SQLite cron store plus doctor migration path, and avoid adding another runtime legacy fallback for this already-migrated issue.

## Reproduction Assessment

Yes for the affected historical path: the report provides production backups and a source-level split JSON persistence sequence. A current-main failing repro is not expected because runtime cron state now loads from SQLite rows and legacy JSON import is doctor-owned.

## Solution Assessment

Yes. The existing SQLite runtime plus doctor-owned migration is the maintainable fix because it removes the split JSON race window and quarantines malformed rows instead of layering another steady-state legacy reader into the runtime.
