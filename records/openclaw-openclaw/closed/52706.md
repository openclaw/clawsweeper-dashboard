## Summary

Current main no longer materializes the UI's runtime-only `last` sentinel into newly saved cron delivery or failure-alert channel configs, and cron delivery now resolves an implicit `last` route through session history or configured-channel selection with a clear failure when no channel can be selected. That covers the reported `jobs.json` default-to-`last` bug; the linked PR #52769 was a useful unmerged fix attempt, but current main has shipped a narrower implementation.

## Best Possible Solution

Keep the shipped implementation: preserve `last` as a runtime implicit-route option, do not persist it as the default channel for newly created Control UI cron jobs, and handle remaining target-specific Feishu setup as explicit delivery target configuration rather than reopening this default-channel bug.

## Risks / Open Questions

- This does not make Feishu targets optional; a Feishu cron job with no previous/default recipient still needs an explicit `delivery.to` or equivalent channel target configuration. That is separate from the reported persisted `delivery.channel: "last"` default.
