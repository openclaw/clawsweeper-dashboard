## Summary

Keep open: the current source path still matches the reported regression for canonical Qwen OpenAI-completions models using `compat.thinkingFormat: "qwen-chat-template"`, and the fix is narrow enough for a focused repair PR.

## What This Changes

The issue asks for `/think off` and session `thinkingLevel: "off"` to drive `chat_template_kwargs.enable_thinking: false` for Qwen chat-template payloads instead of defaulting that nested field to enabled.

## Best Possible Solution

Make the effective session thinking level the source of truth for Qwen chat-template payloads, preferably by extending the provider-owned Qwen wrapper or a typed completions seam so the nested `chat_template_kwargs.enable_thinking` value is corrected without broad transport behavior churn.

## Reproduction Assessment

Yes from source inspection, but not from a live backend run. Current main can build `chat_template_kwargs.enable_thinking` from a default `"high"` effort while the session-off path omits `reasoning`, and the Qwen wrapper then only writes a conflicting top-level `enable_thinking: false`.

## Solution Assessment

Yes, a narrow code fix is the right path, but the proposed `options.thinkingLevel` fallback alone may not be sufficient because current tests show `thinkingLevel: "off"` can arrive as an omitted `reasoning` option. The safer fix is to use the provider-wrapper session value for `qwen-chat-template` or explicitly plumb a typed thinking-level option without changing unrelated defaults.
