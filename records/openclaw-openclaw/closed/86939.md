## What This Changes

The issue reports a webchat dashboard race where a stale current-session row with `status: "running"` makes the run-status label stay on `In progress` after the model call has ended.

## Best Possible Solution

Fix UI session/run reconciliation so a stale current-session `running` row cannot re-enable abortable chat UI after the client has already observed the terminal run, with coverage for both `sessions.list` refreshes and `sessions.changed` events.

## Reproduction Assessment

Yes from source inspection and the reporter's local proof, though not from a live run in this read-only review. A current-session row with `status: "running"` still makes `hasAbortableSessionRun` true after `chatRunId` is null, so the chat view can render `In progress` once the terminal toast clears.

## Solution Assessment

Yes, the best fix is a narrow UI session reconciliation change, not a new feature or config option. The implementation should avoid globally weakening `isSessionRunActive` without preserving sidebar/session-list semantics.
