## Summary

This PR adds a first-class core cron `payload.kind="command"` execution path, but current `main` still intentionally limits cron payloads to `systemEvent` and `agentTurn`. The requested deterministic command scheduling can be built outside core with the existing native plugin surface: plugins can register Gateway lifecycle hooks and CLI/command surfaces, observe/manage cron jobs, short-circuit cron-triggered agent turns before model execution via `before_agent_reply`, and run local commands through the public process runtime helper. The PR also expands a security-sensitive Gateway command-execution path and had substantial review feedback around delivery, timeout, and process-lifecycle correctness, so the conservative cleanup path is to move this capability to a ClawHub/plugin implementation rather than merge a new core payload kind.

## Best Possible Solution

Close this core PR and move the feature to a ClawHub/native plugin: store plugin-owned command job config, use existing cron scheduling or a plugin-owned Gateway service, short-circuit cron-triggered agent turns before model execution with `before_agent_reply`, execute via `runCommandWithTimeout`, and only propose a new plugin API if that implementation hits a concrete missing seam.
