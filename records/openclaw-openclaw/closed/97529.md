## Summary

Keep open: current main and v2026.6.10 still hard-delete the previous role-ordering-reset transcript after replaying only a bounded tail, while adjacent reset/rollover lifecycle paths archive the old transcript. The report is source-backed, narrowly scoped, and not owned by the broader archive-naming issue or the open stalled-session retry PR.

## What This Changes

The issue reports that role-ordering-conflict automatic session reset deletes the previous transcript file instead of archiving it, leaving only the bounded replayed tail in the new session.

## Best Possible Solution

Reuse the existing reset archive lifecycle so role-ordering auto-reset renames the previous transcript to a recoverable `.jsonl.reset.<timestamp>` artifact while the new session keeps the safe replay tail.

## Reproduction Assessment

Yes, at source level: the role-ordering reset caller reaches `persistSessionResetLifecycle` with cleanup enabled, and current tests assert the previous transcript disappears. I did not run a live provider or file-writing repro in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is to replace the role-ordering cleanup unlink with the same reset archive contract already used by adjacent reset lifecycle paths. The broader archive naming and never-delete policy belongs to https://github.com/openclaw/openclaw/issues/56609, not this focused defect.
