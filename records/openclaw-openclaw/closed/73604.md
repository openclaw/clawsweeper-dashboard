## Summary

Current main already addresses the reported Telegram cold-start failure mode. Startup/control Bot API calls are bounded, timed-out `deleteWebhook`/`getMe`/`setWebhook` calls force the Telegram transport to the next fallback route and retry once, and the transport keeps IPv4 and pinned-IP fallback dispatchers sticky for later Telegram API calls. The changelog records this exact `deleteWebhook` IPv6 startup storm fix in the 2026.4.27 section, after the reported 2026.4.26 environment.

## Best Possible Solution

Close this as implemented on main. Keep the current Telegram transport fallback and timeout retry behavior, and use a narrower follow-up only if operators still see slow cold starts on current main or need an explicit process-wide force-IPv4 configuration knob.
