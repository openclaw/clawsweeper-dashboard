## Summary

Keep this PR open. Current main still lacks the PR's central reconnect-grace behavior: after the startup connect grace expires, a running channel with `connected: false` is immediately evaluated as unhealthy and restarted. Main has related Discord health hardening from #72309, including restart cooldown/hourly cap accounting and stale lifecycle-state eviction, and it already has typed `lastDisconnect` status data, but those do not implement the delayed intervention proposed here.

## Best Possible Solution

Keep this PR open as a useful implementation candidate. Rebase it onto current main, add a generic `reconnectGraceMs` and `lastDisconnectAt` path in the shared health policy, consume typed `lastDisconnect.at` from runtime snapshots with a clear fallback policy, update both health-monitor and readiness call sites deliberately, and add focused tests for recent disconnects, expired disconnects, stopped channels, stale sockets, active runs, and restart caps.

## Risks / Open Questions

- The PR is old enough that merging as-is may miss the newer readiness call site, which also uses `ChannelHealthPolicy` and currently treats post-startup disconnects as failures.
- The rebased change needs to preserve current busy-run handling, stale-socket transport-activity behavior, restart cooldown/hour caps, and non-running/gave-up recovery paths.
- The `lastEventAt` fallback should stay bounded and intentional so reconnect grace does not hide a genuinely dead but noisy channel indefinitely.
