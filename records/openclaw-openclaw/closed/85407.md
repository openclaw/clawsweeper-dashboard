## Summary

Keep open: current main still has runtime paths that load OpenClaw's packaged Undici 8.3.0 through `createRequire`, and dependency-source plus a scratch repro show that loading that package mutates Undici global dispatcher state and can make native `globalThis.fetch` fail on the reported Content-Length-shaped plugin requests. The earlier Weixin fix in https://github.com/openclaw/openclaw/pull/78143 addressed no-proxy import-time side effects, but it does not fully supersede this newer report because later runtime calls can still load package Undici in the gateway process.

## What This Changes

The issue reports that loading OpenClaw's packaged Undici through `createRequire("undici")` corrupts native `globalThis.fetch`, breaking external channel plugins such as Weixin.

## Best Possible Solution

Keep OpenClaw-owned proxy/dispatcher requests on a compatible Undici runtime while preventing package Undici global-dispatcher mutation from changing native `globalThis.fetch` behavior for external plugins, with regression coverage for clean startup and post-runtime-bootstrap Content-Length fetches.

## Reproduction Assessment

Partly. The exact gateway plus Weixin setup was not run, but current source still reaches package Undici runtime loads and a scratch Undici 8.3.0 repro shows native fetch failing with the same `TypeError: fetch failed` / `UND_ERR_INVALID_ARG` class after that load.

## Solution Assessment

Unclear at implementation-detail level. The right direction is to avoid mutating native fetch's global dispatcher state while preserving OpenClaw's explicit proxy and timeout paths; the precise code shape needs a careful network-runtime patch, not a Weixin-only workaround.
