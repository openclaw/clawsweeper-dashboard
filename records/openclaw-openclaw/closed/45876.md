## Summary

Current main already implements the PR's central auth behavior through the manifest-first Google plugin path rather than the older core static map, so this PR is superseded by main. The implementation recognizes `GOOGLE_CLOUD_API_KEY` for `google-vertex`, feeds the same metadata into secret scrubbing and shell-env fallback, and has focused regression coverage; it is present on main after the latest stable release, not in `v2026.4.27`.

## What This Changes

The PR adds `GOOGLE_CLOUD_API_KEY` recognition for `google-vertex` auth detection, secret scrubbing, shell-env import, tests, and model-provider documentation.

## Best Possible Solution

Keep the current manifest-first implementation on main and close the stale PR branch; if maintainers still want the public provider docs to call out `GOOGLE_CLOUD_API_KEY`, that should be a small docs-only follow-up against the current docs structure.

## Reproduction Assessment

Yes. The reported path is deterministic: set `GOOGLE_CLOUD_API_KEY` and resolve/auth-check `google-vertex`; current main routes the manifest-provided env var through `resolveEnvApiKey()`, with unit coverage for that exact provider/env snapshot.

## Solution Assessment

Yes. The current main solution is the better maintainable direction because the env var lives in the bundled Google plugin manifest and flows into auth, scrubbing, and shell fallback generically, instead of adding a Google-specific provider id to core.
