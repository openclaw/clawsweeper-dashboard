## Summary

Keep open: current main still exposes, resolves, and tests the two reported `openai-codex` Pro refs even though the issue and duplicate report show ChatGPT/Codex OAuth rejects them at runtime, and the existing OpenAI plugin already has a suppression mechanism for similar OAuth-rejected refs.

## What This Changes

The issue asks OpenClaw to stop selecting or to clearly preflight `openai-codex/gpt-5.4-pro` and `openai-codex/gpt-5.5-pro` when the active credential is ChatGPT/Codex OAuth.

## Best Possible Solution

Keep this open and fix the OpenAI plugin so ChatGPT/Codex OAuth cannot silently select the two unsupported `openai-codex` Pro refs, while preserving direct `openai/*` Pro model handling.

## Reproduction Assessment

No live high-confidence reproduction was established here because it requires a matching ChatGPT/Codex OAuth account, but source inspection reproduces the selection path and two reports provide the same backend rejection text. Current main still advertises and resolves both reported refs.

## Solution Assessment

Yes: a provider-local suppression or clear preflight/error is the narrow maintainable fix. Core routing should remain generic, and direct `openai/gpt-5.4-pro` and `openai/gpt-5.5-pro` rows should not be removed by this fix.
