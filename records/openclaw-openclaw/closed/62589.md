## Summary

Keep open. Current `main` still has the fixed 20s browser act request timeout and the restart-the-gateway timeout guidance, so the PR is not already implemented on `main`. The PR also has an unresolved regression risk already called out in review: injecting `timeoutMs` into every act request conflicts with existing-session act kinds that currently reject timeout overrides.

## Evidence

- **Current main still hard-codes 20s for browser act requests:** `browserAct()` still sends every `/act` request through `fetchBrowserJson(..., { timeoutMs: 20000 })`, with no configurable default timeout and no wait-specific slack on current `main`. That means the PR's behavior is not already shipped.
