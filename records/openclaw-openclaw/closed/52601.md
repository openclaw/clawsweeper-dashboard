## Summary

Keep open. This is a plausible FreeBSD filesystem bugfix, but the branch is now conflicting with the current package-backed fs-safe wrapper, lacks after-fix FreeBSD proof, and the proposed fallback route still has a concrete new-target atomicity regression.

## What This Changes

The PR adds FreeBSD to the write and copy platform guards so `writeFileWithinRoot` and `copyFileWithinRoot` use the legacy fs-safe fallback instead of the Python pinned-write helper.

## Best Possible Solution

Land a current-main fix at the `@openclaw/fs-safe` boundary that handles FreeBSD helper identity without weakening write/copy atomicity, then verify helper-enabled write/copy and plugin install on FreeBSD/ZFS.

## Reproduction Assessment

No high-confidence live reproduction was established in this review. The source path is clear for helper-enabled FreeBSD/ZFS from the Python stat identity, strict Node verification, and the PR's signed `st_dev` diagnostics.

## Solution Assessment

No. The fallback route avoids the Python-vs-Node device mismatch, but it extends legacy missing-target semantics and no longer matches current main's external fs-safe package boundary; a targeted helper identity fix is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Preserve atomic writes for new FreeBSD targets:** `src/infra/fs-safe.ts:551`
  - body: Routing FreeBSD into `writeFileWithinRootLegacy` and `copyFileWithinRootLegacy` opens or creates the final destination before writing the temp file. If the process crashes before `rename`, missing FreeBSD targets can be left as zero-byte files instead of remaining absent, so the fix should normalize helper identity or use a FreeBSD path that preserves new-target atomicity.
