## Summary

Current main and the inspected shipped tag already wire `tools.web.fetch.ssrfPolicy.allowRfc2544BenchmarkRange` into `web_fetch` SSRF checks, with config/schema/docs/test coverage for the opt-in behavior.

## What This Changes

The issue reports that `web_fetch` still blocks domains resolving to RFC 2544 fake-IP addresses even when `tools.web.fetch.ssrfPolicy.allowRfc2544BenchmarkRange` is enabled.

## Best Possible Solution

Keep the existing explicit opt-in implementation and ask affected 2026.5.12 users to upgrade to a current release; proxy-side DNS resolution should use the separate `tools.web.fetch.useTrustedEnvProxy` setting when appropriate.

## Reproduction Assessment

No live WSL2/Clash current-main reproduction was run in this read-only review. Source and shipped regression tests show the current path forwards the configured RFC2544 opt-in into DNS/IP checks, so the reported failure is not expected on current code.

## Solution Assessment

Yes. The current solution is narrow: the config schema accepts the setting, `web_fetch` forwards it to the shared SSRF guard, and tests keep the default block while allowing RFC2544 only under explicit opt-in.
