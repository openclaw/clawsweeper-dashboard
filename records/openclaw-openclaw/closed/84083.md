## Summary

Keep open: this is a focused bug-fix PR with useful code, tests, and real-behavior proof; I found no blocking review findings, but the exact-head checks and intentional timeout compatibility/session-state risks still need the normal merge gate.

## What This Changes

The PR adds a shared safety-timeout wrapper for plugin-owned `ContextEngine.compact()`, threads abort signals through pi-embedded, CLI, and Codex harness compaction lanes, and updates the SDK contract, baseline, tests, and changelog.

## Best Possible Solution

Land the bounded host wrapper after exact-head checks pass, keeping the existing compaction timeout setting as the compatibility knob and adding operator-facing wording only if maintainers want extra upgrade visibility.

## Reproduction Assessment

Yes. Source inspection of current `main` shows plugin-owned queued, recovery, CLI, and Codex lanes directly await `contextEngine.compact()` while only native compaction uses the timeout helper, so a never-settling owning plugin promise reproduces the stall path.

## Solution Assessment

Yes. Reusing the existing native compaction timeout semantics through one shared helper and an optional abort signal is the narrowest maintainable fix; the remaining question is merge timing around compatibility and plugin cooperation.
