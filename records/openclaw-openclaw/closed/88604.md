Keep open. The feature is useful but the PR is not merge-ready because a failing renderer can still crash the reply process instead of falling back, and the posted proof does not show the actual OpenClaw chat/footer path after the patch.

## What This Changes

The PR adds a `messages.usageLine` config surface, a renderer helper that spawns a local command for per-response `/usage` footers, related docs, and config/reply tests.

## Best Possible Solution

Keep the opt-in renderer only if subprocess stdin/exit failures are handled as ordinary renderer failures, the new config command surface is intentionally accepted, and real Telegram or terminal proof shows the footer replacement and fallback paths in OpenClaw.

## Reproduction Assessment

Yes for the blocking patch defect: the new code writes to `child.stdin` without an error handler, and a renderer that exits before reading stdin can emit an unhandled `EPIPE`. I confirmed the Node child-process behavior with a focused subprocess probe.

## Solution Assessment

No. The renderer direction is plausible, but the subprocess contract must make stdin/write failures part of the documented fallback path before the new config surface is safe to merge.
