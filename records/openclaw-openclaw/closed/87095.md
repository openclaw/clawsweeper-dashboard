## Summary

Keep open. Current main appears to have a capped `/new` startup-memory path and an existing Dreaming disable switch, but memory-core still accepts Dreaming session-corpus entries into `short-term-recall.json` and lacks a store-size or entry-retention cap, so the remaining unbounded Dreaming-state problem is not implemented yet.

## What This Changes

The issue asks for Dreaming short-term recall and session-ingestion retention safeguards so `.dreams` state cannot grow until new sessions overflow context.

## Best Possible Solution

Add bounded internal retention for `short-term-recall.json` and session-ingestion state, exclude or prune self-generated session-corpus recall entries where appropriate, and add a doctor warning/fix path while relying on the existing `dreaming.enabled` switch for full disablement.

## Reproduction Assessment

No live runtime reproduction was run in this read-only review. Source inspection gives a clear growth path: session-corpus batches are recorded into `short-term-recall.json`, and the store writer has no entry, byte, or token retention cap.

## Solution Assessment

Yes for the bounded-retention part: a memory-core repair with regression tests is the narrowest maintainable fix. The requested extra disable flag should not be the default answer because current docs already expose `dreaming.enabled`.
