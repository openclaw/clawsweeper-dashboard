## Summary

Keep open: current main still has the CLI bootstrap ordering gap that can put plugin loader diagnostics on stdout before JSON output, and the linked open fix PR has not merged.

## What This Changes

The issue reports that non-routed plugin-provided `openclaw ... --json` commands can emit `[plugins]` loader diagnostics to stdout before the JSON payload.

## Best Possible Solution

Review and land a scoped fix through https://github.com/openclaw/openclaw/pull/81536 that routes diagnostics to stderr during early plugin registration without changing parse-only `--json` behavior, with focused regression coverage.

## Reproduction Assessment

Yes. Current source shows plugin registration happens before `parseAsync`/`preAction`, while loader debug messages still use stdout unless stderr routing is already forced; the reporter also provided a concrete `openclaw memory search ... --json | jq` failure path.

## Solution Assessment

Yes on direction, but the linked PR needs the existing scoping concern resolved before merge. The narrow maintainable fix is to move or bracket existing stderr routing around the early plugin-registration window without broadening parse-only or pass-through `--json` semantics.
