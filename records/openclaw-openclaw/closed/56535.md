## Summary

Close as implemented. Current main and the latest release have an explicit provider-request opt-in that lets trusted provider traffic bypass the default private/special-use DNS block, and Google image generation threads that option into the guarded HTTP call path. The remaining ergonomics problem of configuring the same proxy/private-network policy across many providers is now tracked separately by #74835.

## What This Changes

The issue requests proxy/Fake-IP-aware handling or a safe override for SSRF checks that block public AI provider endpoints when local DNS returns special-use fake IPs.

## Best Possible Solution

Keep the shipped explicit opt-in behavior for trusted provider/proxy environments, and continue the broader global/default provider request policy work in #74835 so operators do not have to repeat the same proxy and private-network settings for every remote provider.

## Reproduction Assessment

Yes. The report gives concrete fake-IP DNS outputs and the current SSRF tests model the same class of failure by resolving a public hostname to `198.18.0.153`; the current source path also shows the explicit per-provider opt-in that changes the guarded request policy.

## Solution Assessment

Yes for this cleanup decision. The current implementation preserves fail-closed SSRF defaults and requires an explicit operator/provider opt-in, while the less narrow global-default policy remains a separate design/ergonomics follow-up in #74835.
