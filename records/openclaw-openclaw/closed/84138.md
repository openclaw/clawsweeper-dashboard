## Label Justifications

- `P3`: The issue asks for an optional heartbeat scheduling/config refinement and current main already covers the active-turn safety behavior.
- `impact:session-state`: The report concerns heartbeat interaction with an active agent turn and session continuity during long-running work.

## Summary

Current main already prevents heartbeat wakes from interrupting an active target session: the runner skips busy reply/session lanes and the wake layer retries those skips instead of starting a competing heartbeat turn.

## What This Changes

The issue requests a heartbeat scheduling mode or config option that queues heartbeat polls until an active agent turn finishes.

## Best Possible Solution

Keep the current scheduler-level busy-skip and retry behavior as the default, because it avoids active-turn interruption without adding another config switch.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No failing current-main reproduction is available from this report, but source and existing tests give high-confidence proof that current main skips and retries instead of interrupting active target sessions.

## Solution Assessment

Is this the best way to solve the issue? Yes: the existing wake-layer deferral is narrower and safer than adding a user option that could let heartbeat turns compete with active runs.
