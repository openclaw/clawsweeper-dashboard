## Summary

Close as implemented. Current main and the v2026.4.27 shipped tree avoid the reported first-message timeout path by credential-gating lazy xAI tool factories, reusing the active runtime plugin registry for tool construction, and keeping first-pass embedded model resolution off broad provider discovery; broader startup/provider-catalog latency remains covered by related open reports.

## What This Changes

The issue reports a first-message latency regression after restart caused by bundled provider tool factories timing out sequentially during embedded-run tool initialization.

## Best Possible Solution

Keep the shipped lazy, credential-gated provider tool registration and runtime-registry reuse. Any remaining broad provider catalog or startup latency should stay with the narrower related performance reports rather than this exact first-message tool-factory timeout path.

## Reproduction Assessment

Unclear for the old v2026.4.26 stall because this was a read-only review and no live gateway timing run was performed. The current-main verification path is high confidence from source and shipped-tag inspection: unconfigured xAI tool factories no longer perform network work during tool construction.

## Solution Assessment

Yes. The current direction is the narrow maintainable fix for this symptom: avoid constructing network-backed provider tools unless credentials exist, defer network work to tool execution, and avoid broad provider discovery during the first embedded model-resolution pass.
