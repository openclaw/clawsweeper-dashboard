## Label Justifications

- `P2`: The report concerns a normal-priority provider/model configuration path for memory embeddings, with a supported path already present.
- `impact:auth-provider`: The affected surface is embedding provider/model selection for memorySearch, including OpenAI-compatible routing and model dimensions.

## Summary

Current main and the latest release already support OpenAI-compatible custom embedding models through the canonical `outputDimensionality` memorySearch key, which is accepted by schema, resolved into memory settings, and forwarded as the API `dimensions` field.

## What This Changes

Allow memorySearch configs using OpenAI-compatible custom embedding models to declare vector dimensions so BGE/M3E-style models can index without schema/runtime failures.

## Best Possible Solution

Keep the strict memorySearch schema and use `outputDimensionality` as the canonical dimension override for OpenAI-compatible custom embedding models.

## Reproduction Assessment

Partially. The strict-schema rejection for the non-canonical `dimensions` key is source-reproducible, but the reported provider blocker is not because current and release code accept `outputDimensionality` and forward it to OpenAI-compatible embeddings.

## Solution Assessment

Yes. The existing `outputDimensionality` path is the best fit because it avoids adding a duplicate config alias and matches the repository policy that runtime config reads only the current canonical schema.
