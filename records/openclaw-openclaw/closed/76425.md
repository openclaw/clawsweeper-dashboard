## Label Justifications

- `P1`: The reported upgrade regression could leave a real Gateway running but unable to dispatch channel messages after a package update.
- `impact:message-loss`: The issue directly affects Feishu, DingTalk, Slack, and Telegram message dispatch paths failing before delivery.
- `impact:crash-loop`: The failure mode is process/runtime availability: core Gateway paths crash with `ERR_MODULE_NOT_FOUND` after stale hashed chunk imports.

## Summary

Current main and the latest release now cover the stale live-Gateway and mixed-`dist` upgrade failure family well enough to close: update work is handed off outside the live Gateway, npm global updates are staged and verified before swap, and restart paths are hardened against rotated hashed chunk imports.

## What This Changes

The issue requests that OpenClaw upgrades stop leaving a running Gateway with stale hashed `dist` imports that crash channel dispatch and runtime paths with `ERR_MODULE_NOT_FOUND`.

## Best Possible Solution

Keep the current managed update handoff and staged package-swap implementation, and ask for a new narrow report only if v2026.6.6 or newer still reproduces stale chunk imports after upgrade.

## Reproduction Assessment

Yes from source and field reports, but not by running a live global package upgrade in this read-only review. The stale Gateway version mismatch and missing hashed chunks match the current source comments and tests for rotated `dist` imports.

## Solution Assessment

Yes. The current implementation fixes the right boundary by avoiding live package-tree mutation, staging and verifying package updates, and respawning the Gateway instead of adding per-channel import fallbacks.
