## Summary

The reported config-safety gap is already covered by current OpenClaw behavior, and the protections were shipped in v2026.4.23: full-schema validation, rejected destructive config writes, and automatic last-known-good recovery for invalid direct edits/startup reads.

## Evidence

- **Strict validation and recovery are documented on the user-facing config page:** The docs state that unknown keys, malformed types, or invalid values cause the Gateway to refuse startup, and that invalid configs are restored from a trusted last-known-good copy with the broken payload preserved as `.clobbered.*`.
  - file: [docs/gateway/configuration.md:70](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/docs/gateway/configuration.md#L70)

- **Startup code restores last-known-good before failing gateway boot:** Current main calls `recoverConfigFromLastKnownGood` for invalid startup snapshots, reloads the recovered snapshot, emits a recovery notice, and only then re-validates.
