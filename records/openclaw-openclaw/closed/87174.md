## Summary

Close: current main already addresses the central cron queueing problem by defaulting cron execution capacity to 8, applying it to both `cron` and `cron-nested`, and documenting the supported knob. The latest release `v2026.5.22` still defaulted the lane to 1, so this is fixed on current main only.

## What This Changes

The issue requests that cron/manual runs not wait behind one healthy long-lived isolated cron run under light load, or that OpenClaw expose an explicit supported throughput knob.

## Best Possible Solution

Keep the current main behavior: default cron execution capacity to 8 across `cron` and `cron-nested` while preserving the documented `cron.maxConcurrentRuns` knob for constrained installations.

## Reproduction Assessment

Yes. Source inspection of `v2026.5.22` shows cron lanes defaulted to one slot when `cron.maxConcurrentRuns` was unset, matching the reported waits behind a long active cron run; current main no longer has that default.

## Solution Assessment

Yes. The current implementation is the narrow maintainable fix: raise the missing-config default while preserving explicit operator limits and the existing documented tuning knob.
