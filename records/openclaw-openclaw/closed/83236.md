## Summary

Close as implemented on main: the shared channel startup handoff from the merged gateway fix now applies before any plugin `startAccount`, iMessage uses that shared lifecycle, and the latest shipped v2026.5.12 code still has the older long-lived `channels.<id>.start-account` span that this report hit.

## What This Changes

The issue asks for iMessage `startAccount` startup to stop pinning/starving gateway startup and to use the same bounded, asynchronous channel-startup hardening as the related Telegram/startup reports.

## Best Possible Solution

Keep the shared channel-manager startup handoff on main and ship it; only reopen the iMessage path if current-main logs show a distinct runtime-level iMessage RPC/blocking operation after the handoff.

## Reproduction Assessment

Yes, source-level: v2026.5.12 shows the old long-lived `channels.${channelId}.start-account` wrapper, while current main has the shared handoff and tests proving the changed startup trace behavior. I did not run a live macOS/iMessage reproduction in this read-only review.

## Solution Assessment

Yes. The shared gateway channel-manager handoff is the narrow maintainable fix because it covers iMessage through the common plugin startup lifecycle instead of duplicating channel-specific startup shims.
