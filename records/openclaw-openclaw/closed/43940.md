## Summary

Current main already carries a Telegram inbound-media IPv4 fallback and transport-threading fix that matches this report, and the workaround is documented. The changelog places the fix in release 2026.3.13.

## Evidence

  - file: [extensions/telegram/src/fetch.ts:589](https://github.com/openclaw/openclaw/blob/4e4aeacae4a87f2fe09125d462d88cd71a18cb66/extensions/telegram/src/fetch.ts#L589)

## Close Comment

Closing from a Codex review because current `main` already implements the missing Telegram media-path fallback this report was pointing at. The shipped fix is called out in `CHANGELOG.md:2533` and `CHANGELOG.md:2534` under release `2026.3.13`, and current code now threads the Telegram transport into inbound media downloads in `extensions/telegram/src/bot/delivery.resolve-media.ts:232` so file fetches reuse the same IPv4 fallback/proxy behavior built in `extensions/telegram/src/fetch.ts:589`. The operator workaround is also documented in `docs/channels/telegram.md:821` through `docs/channels/telegram.md:873`, including `channels.telegram.network.autoSelectFamily: false`. Verified against current `main` at `4e4aeacae4a87f2fe09125d462d88cd71a18cb66`. If this still reproduces on a newer build, please open a fresh issue with exact Docker Desktop/macOS versions and failing media-fetch logs.
