## Summary

Current main already avoids the reported duplicate `openclaw-unknown-*` bundled plugin runtime-deps path for packaged/global installs. Packaged bundled plugin roots now derive the external cache key from the OpenClaw package version, existing external stage roots are recognized and reused, and regression/e2e guards cover the second-generation `openclaw-unknown-*` churn described in the issue.

## Best Possible Solution

Keep the shipped current-main implementation: versioned external runtime-deps roots for packaged/global installs, reuse of existing staged mirror roots, and the regression/e2e guards against second-generation `openclaw-unknown-*` churn. Treat any future `unknown` cache key in a normal packaged install as a fresh, narrower reproduction rather than reopening this already-fixed duplicate-cache path.

## Evidence

- **Stable external cache key source:** Packaged bundled plugin roots resolve back to the OpenClaw package root, read `package.json` version, and derive the external runtime-deps key as `openclaw-${version}-${hash}`; the `unknown` fallback remains only when that package version is unavailable.

## Risks / Open Questions

- A deliberately malformed package root with no readable `package.json` version can still fall back to `unknown`; if that appears in a normal current-main packaged/global install, it should be tracked as a narrower new bug with the package root, stage root, plugin id, and log excerpt.
