## Summary

Close as superseded: the documented Canvas/A2UI runtime now goes through gateway plugin routes where the frame-compatible baseline headers are already applied, and the same canvas-header hardening was previously reviewed and closed as implemented; this stale branch also adds frame denial that conflicts with iframe-hosted canvas rendering.

## What This Changes

The PR adds X-Content-Type-Options, Referrer-Policy, and X-Frame-Options: DENY to standalone canvas-host responses and tests 200/404 header coverage.

## Best Possible Solution

Keep the shipped gateway/plugin-route baseline; any future direct startCanvasHost hardening should be a fresh current-main change that reuses the frame-compatible baseline and avoids X-Frame-Options/CSP unless maintainers explicitly change the embedding policy.

## Reproduction Assessment

Yes, but only by source inspection for the direct standalone helper: startCanvasHost on current main does not set these headers. The documented shipped Canvas/A2UI runtime is not a high-confidence failing path because gateway routing applies the baseline before dispatch.

## Solution Assessment

No, this PR is not the best solution. The maintainable path is the existing gateway-compatible baseline; this branch targets an old path and adds X-Frame-Options: DENY, which conflicts with current iframe-hosted Canvas behavior.

## Review Findings
