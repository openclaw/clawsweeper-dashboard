## Summary

Keep open. Current main still has the reported blocking order in both failover paths, and the same-author implementation PR #57283 remains open and unmerged with closing syntax for this issue.

## What This Changes

The issue asks for rate-limit auth-profile failover to rotate before any potentially blocking cooldown-store write.

## Best Possible Solution

Review and land #57283, or a narrow replacement, that advances the auth profile before any awaited cooldown write and adds regression coverage for locked cooldown marking.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction path: rate-limit failures resolve to rotate_profile, both failover branches await failure marking first, and failure marking awaits the auth-store file lock before advanceAuthProfile can run. A live multi-agent contention run was not performed because this was read-only.

## Solution Assessment

Yes, the requested direction is the narrowest maintainable fix: rotate first, then record cooldown asynchronously or with bounded waiting and error logging while preserving eventual cooldown state.
