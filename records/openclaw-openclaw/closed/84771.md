## Label Justifications

- `P1`: The report describes a recent startup regression that can break real Discord inbound delivery until reconnect or watchdog recovery.
- `impact:session-state`: The implicated startup work includes session-lock cleanup and session-store recovery paths.
- `impact:message-loss`: The reported event-loop stalls coincide with Discord READY failures, delayed typing, and missed guild messages.
- `impact:crash-loop`: The issue describes gateway restart cascades and process-level availability loss under systemd or external watchdogs.

## Summary

Keep open: current main still schedules model prewarm before channel startup and still runs the session-lock cleanup/recovery sidecar during startup; the 2026.5.20 follow-up adds credible current-release evidence, while related open work only covers part of the fanout problem.

## What This Changes

The issue requests reducing startup event-loop saturation from model prewarm and session-lock sidecars so Discord readiness, typing feedback, memory pressure, and restart recovery remain healthy.

## Best Possible Solution

Adopt a maintainer-approved startup readiness policy that moves or bounds heavyweight prewarm and recovery work outside provider handshake windows, then prove loaded Discord startup with liveness metrics and session-store scale.

## Reproduction Assessment

No live high-confidence current-main reproduction was run for the full 28-64 second symptom. Source inspection does show the named startup sidecars and synchronous session-store parse path still exist, and two reporters provided current-release production logs.

## Solution Assessment

Unclear as a final fix: deferring or bounding prewarm and session recovery is likely needed, but startup readiness semantics and live Discord proof need maintainer direction before implementation.
