## Summary

Keep open: current main still has the wrapper-first LaunchAgent writer that the report calls out, but the actual macOS launchd crash and the safest fix shape need maintainer/live confirmation because the wrapper design is also the documented secret-handling path.

## What This Changes

The issue reports that doctor/update rewrites the macOS LaunchAgent with the generated env wrapper as the first ProgramArguments entry, causing the gateway to crash on launch.

## Best Possible Solution

Confirm the failure with a macOS LaunchAgent smoke, then preserve owner-only env-file handling while making the wrapper invocation launchd-proof and covered by doctor/update regression tests.

## Reproduction Assessment

No. Source inspection on current main confirms the wrapper-first ProgramArguments path when managed service env exists, but I did not establish a live macOS launchd crash or prove that Node parses the wrapper script.

## Solution Assessment

Unclear. The inline EnvironmentVariables workaround would avoid the wrapper shape but conflicts with current docs/tests for secret handling; a safer fix is to keep owner-only env storage and harden wrapper invocation after macOS confirmation.
