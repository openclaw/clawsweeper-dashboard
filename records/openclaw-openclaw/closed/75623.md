## Summary

Keep open. The report and follow-up comments describe a concrete 2026.4.29 regression, and current main still shows the same generic bundled-channel runtime-dependency boundary: staged deps are expected, but channel entry/plugin loads can run with runtime-deps preparation disabled and without a visible `dist/node_modules` fallback for mirrored root chunks.

## What This Changes

The issue asks OpenClaw to make bundled Slack and Telegram provider startup resolve `json5` from the matching staged `plugin-runtime-deps` install root after gateway updates.

## Best Possible Solution

Fix the generic bundled plugin runtime-dependency load root so mirrored root chunks and bundled channel entry/runtime loads consistently resolve staged dependencies from the matching runtime-deps root across gateway startup, doctor/update repair, native require/import, and Jiti paths.

## Reproduction Assessment

Yes. The issue gives concrete 2026.4.29 macOS/Node 25 steps and log signatures, and two later comments confirm the same Slack/Telegram startup failure; source inspection shows the same loader boundary still present on current main.

## Solution Assessment

Unclear for any specific patch because no current PR is open, but the best direction is a generic bundled runtime-deps loader/mirror fix. Adding channel-specific package dependencies or inlining `json5` would leave the shared staged-dependency resolution gap exposed.
