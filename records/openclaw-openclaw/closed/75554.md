## Summary

Keep open: current main still lacks the plugin task lifecycle API, and the branch is not merge-ready because the create path can reuse a different taskKind record while contributor real behavior proof is still absent.

## What This Changes

The branch adds `api.runtime.tasks.runs.lifecycle` create/progress/finalize helpers, exposes `taskKind` in task DTOs, adds task-registry ID mutators, and updates docs, changelog, API baseline hashes, and runtime task tests.

## Best Possible Solution

Keep the additive owner-bound SDK seam, but make taskKind part of create identity, rebase, and merge only after real plugin/session proof and SDK-contract approval.

## Reproduction Assessment

Yes for the PR-specific defect by source inspection: with the same owner/runId/title/label and a different taskKind, `findExistingTaskForCreate` can fall through from the new taskKind check to the legacy exact match. Reproduction is not applicable to the requested feature itself because current main does not have the lifecycle API.

## Solution Assessment

Mostly yes, but not as currently implemented: the additive owner-bound SDK seam is maintainable, while create de-duplication must either stop when a provided taskKind has no match or include taskKind in every fallback match.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep taskKind from falling back to another task:** `src/tasks/task-registry.ts:720-738`
  - body: When taskKind is provided, this branch checks for a matching taskKind but then falls through to the legacy label/title match. A second lifecycle create with the same owner, runId, title, and label but a different taskKind can reuse and mutate the first task, so later progress/finalize by taskKind no longer represents the task that was created. Return no match when a provided taskKind has no match, or include taskKind in every fallback match.
  - confidence: 0.86
