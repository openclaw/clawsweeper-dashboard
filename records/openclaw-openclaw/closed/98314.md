## Label Justifications

- `P2`: The report describes a bounded but user-visible Telegram attachment delivery failure with misleading success state.
- `impact:message-loss`: The reported behavior loses the intended downloadable file attachment while still sending a visible message.

## Summary

Current main and v2026.6.11 already route message-tool `filePath`/`path` values into outbound media for Telegram, so the reported text-only fallback is no longer the current implementation.

## What This Changes

The issue asks for Telegram message-tool sends with `filePath` to upload a document or fail loudly instead of reporting success after sending only text.

## Best Possible Solution

Keep the shipped v2026.6.11 behavior that treats `filePath` as outbound media and directs unresolved future Telegram file-delivery failures to a new report with logs from that release or newer.

## Reproduction Assessment

No for current main. Source and release checks show `filePath` is collected into media and Telegram media delivery uses `InputFile` plus media send APIs instead of text-only fallback.

## Solution Assessment

Yes. The current implementation fixes this at the shared outbound and Telegram channel boundary without adding a new config surface.
