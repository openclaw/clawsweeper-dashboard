## Summary

Keep open. Current main still routes `openclaw browser status/start` through `browser.request`, and the gateway dispatch path uses a separate browser control singleton from the HTTP control server, matching the dual-state failure described in the issue. The relevant browser request/control files are unchanged from the reported broken `v2026.4.27` release, so there is not enough evidence to close this as implemented.

## What This Changes

The issue requests fixing the browser CLI/gateway WebSocket path so configured root and profile-level `browser.executablePath` settings are honored consistently with the direct browser control HTTP API.

## Best Possible Solution

Use one authoritative local browser control runtime for both the HTTP control server and gateway `browser.request` dispatch, then add a regression test proving configured executablePath/noSandbox/headless values are visible through CLI/gateway status and direct HTTP status after the same startup.

## Reproduction Assessment

Yes. The issue gives concrete CLI and HTTP comparison steps, and current main still has separate HTTP and gateway browser control state paths that plausibly reproduce the observed stale status/start behavior.

## Solution Assessment

No. Closing as implemented is not the best path; the maintainable fix is to remove or share the duplicate local browser control state rather than adding another fallback only in the status route.
