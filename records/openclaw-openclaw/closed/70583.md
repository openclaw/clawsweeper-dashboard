## Summary

Current main already fixes the stale Telegram model-picker symptom by invalidating the shared model catalog cache on model-related hot reloads. The PR's exact `useCache: false` caller change is not present, but the linked #69750 user problem is covered by the broader cache-invalidation path and regression test.

## Best Possible Solution

Close this PR as implemented on main and keep the shipped hot-reload cache invalidation path as the fix for #69750. If maintainers still want interactive model menus to bypass the catalog cache on every invocation, that should be a separate policy/performance PR rather than keeping this stale-config bug-fix branch open.
