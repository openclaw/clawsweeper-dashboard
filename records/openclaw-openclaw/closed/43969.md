## Summary

Current `main` already has persistent Feishu message-id dedupe plus startup warmup and reply-once regression coverage for duplicate replay, which matches the reported symptom of the same `message_id` being re-delivered and replied to again.

## Evidence

- **Persistent per-account Feishu dedupe:** Feishu message IDs are now recorded through a persistent dedupe store with a 24h TTL, backed by `~/.openclaw/feishu/dedup/<account>.json` (or `OPENCLAW_STATE_DIR`), instead of only transient in-memory state.
  - file: [extensions/feishu/src/dedup.ts:9](https://github.com/openclaw/openclaw/blob/4e4aeacae4a87f2fe09125d462d88cd71a18cb66/extensions/feishu/src/dedup.ts#L9)

- **Startup warmup restores dedupe state after reconnect/restart:** Each Feishu account warms the dedupe cache from disk before monitoring starts, which directly addresses duplicate redelivery after gateway restart or transport reconnect.
  - file: [extensions/feishu/src/monitor.account.ts:646](https://github.com/openclaw/openclaw/blob/4e4aeacae4a87f2fe09125d462d88cd71a18cb66/extensions/feishu/src/monitor.account.ts#L646)
