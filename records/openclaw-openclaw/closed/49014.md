## Summary

Current `main` already caches inbound Telegram stickers even when the default model supports vision. In `dispatchTelegramMessage`, the `!stickerSupportsVision` branch only controls fallback body rewriting/media pruning; `cacheSticker(...)` runs afterward regardless of vision support, and the same logic is present in shipped release `v2026.4.22`.

## Evidence

- **Vision gate no longer controls cache writes:** `stickerSupportsVision` only gates `ctxPayload.Body`/`BodyForAgent` rewriting and `pruneStickerMediaFromContext(...)`. `cacheSticker(...)` executes afterward in the same `if (description)` block, so described stickers are cached whether vision is on or off.
