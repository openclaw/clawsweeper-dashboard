## Summary

Current main already avoids the reported isolated cron/heartbeat Codex failure path: embedded Codex runs now use OpenClaw's boundary-aware OpenAI transport with injected OAuth credentials instead of pi-ai's raw atob account-id extraction. The fix is main-only and not included in v2026.4.29.

## What This Changes

The issue reports that Codex OAuth JWT account-id extraction fails in isolated cron/heartbeat sessions because the upstream pi-ai path decodes JWT payloads with raw atob instead of base64url handling.

## Best Possible Solution

Keep the current main boundary-aware Codex transport path and ship it in the next release so v2026.4.29 users get the fix.

## Reproduction Assessment

Yes. The issue's release steps are coherent, and a local static probe reproduces the raw atob/base64url failure while current-main source shows isolated embedded Codex runs no longer use that upstream decoder.

## Solution Assessment

Yes. Avoiding pi-ai's raw atob extractor in the embedded Codex runtime is safer than rewriting bearer token segments, and OpenClaw's own Codex identity helper already uses base64url decoding.
