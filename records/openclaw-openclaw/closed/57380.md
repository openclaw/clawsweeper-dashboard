## Summary

Current `main` no longer has the unconditional xAI bootstrap-noise path. The xAI plugin only synthesizes auth when a configured fallback key exists, and the same production implementation is already present in release `v2026.4.23`.

## Evidence

- **Synthetic auth now short-circuits when xAI is unconfigured:** The xAI provider hook calls `resolveFallbackXaiAuth(config)` and returns `undefined` when no fallback auth is found, so there is no unconfigured synthetic-auth path to emit the reported noise.
  - file: [extensions/xai/index.ts:186](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/extensions/xai/index.ts#L186)
  - command: `nl -ba extensions/xai/index.ts | sed -n '176,198p'`
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
- **Fallback auth is only derived from explicit config keys:** `resolveFallbackXaiAuth` only returns auth from `plugins.entries.xai.config.webSearch.apiKey` or legacy `tools.web.search.grok.apiKey`; otherwise it returns `undefined`. That matches the requested behavior of skipping unconfigured xAI.
  - file: [extensions/xai/src/tool-auth-shared.ts:118](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/extensions/xai/src/tool-auth-shared.ts#L118)
