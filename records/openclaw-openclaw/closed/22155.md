## Summary

Current main already treats TTS as a plugin-owned speech capability rather than a hardcoded core enum, so a CLI-backed TTS adapter belongs as an external/ClawHub plugin, not new core surface.

## Risks / Open Questions

- There is still no built-in config-only CLI TTS adapter on current main; this close assumes maintainers prefer the existing plugin path over adding another core provider mode.
- I did not verify an existing published ClawHub/community CLI speech plugin from this checkout, only that current main provides the extension point for one.

## Close Comment

Closing as a ClawHub/plugin-scope request based on a Codex review of current `main`. The core premise in the report is outdated: TTS providers are no longer a hardcoded enum in core (`src/config/zod-schema.core.ts:482`, `src/config/types.tts.ts:1`), and speech providers are resolved through the plugin capability registry (`src/tts/provider-registry.ts:16-53`, `src/plugins/types.ts:1613-1632`). OpenClaw’s docs already support external speech-provider plugins and show `api.registerSpeechProvider(...)` as the extension path (`docs/tools/plugin.md:11-69`, `docs/plugins/sdk-provider-plugins.md:496-506`). `VISION.md:52-60` also makes the scope call explicit: optional capability should usually ship as plugins, with a high bar for adding more core surface. A CLI-backed TTS wrapper still seems useful, but it belongs as a community/ClawHub plugin rather than new OpenClaw core code.
