## Summary

Keep open: the stale-session guard is the right fix direction, but this head still risks dropping legitimate terminal lifecycle updates after an in-run compaction rotation, and the PR still has only deterministic test proof rather than real reset-race proof.

## What This Changes

The PR stamps lifecycle events with an owning sessionId, skips stale gateway lifecycle persistence/projection after session reset, and adds focused regression tests across gateway, agent-event, followup, memory, cron, and direct agent run paths.

## Best Possible Solution

Keep the stale-event guard, but refresh or explicitly stamp the run context with the effective rotated sessionId after legitimate in-run rotations and before terminal lifecycle emission, then add regression coverage and real or maintainer-approved reset-race proof.

## Reproduction Assessment

Yes for source-level reproduction: a direct agent-command result with a rotated agentMeta sessionFile/sessionId persists the new row before emitLifecycleEnd, while the run context still owns the old sessionId. I did not run tests because this review was constrained to read-only inspection.

## Solution Assessment

No, not yet. The stale guard is the right maintainable direction, but the implementation must also carry legitimate rotated session ownership through the terminal lifecycle path before merge.

- **[P1] Refresh lifecycle owner after direct-run rotation:** `src/agents/agent-command.ts:984`
  - body: This registers the lifecycle owner as the pre-run sessionId, but later the same direct-run path can persist `effectiveSessionId` from `result.meta.agentMeta.sessionFile/sessionId` before calling `emitLifecycleEnd`.
