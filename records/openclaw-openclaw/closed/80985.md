## Summary

The PR should stay open: the direction and supplied terminal proof are good, but source review still finds blocking gaps in implicit add, inherited-mode edit, raw cron runtime handling, and one invalid docs example.

## What This Changes

The branch adds CLI/runtime validation, a WebchatNotDeliverableError, tests, docs, and a changelog entry for rejecting cron announce delivery to webchat.

## Best Possible Solution

Validate effective announce-to-webchat before webchat is normalized away, cover explicit and implicit create/edit/runtime paths with regression tests, and align the docs with commands the CLI accepts.

## Reproduction Assessment

Yes. Current main source shows isolated agentTurn cron add defaults to announce with the channel preserved, and cron runtime drops webchat before channel selection, so the broken path is source-reproducible without running the branch.

## Solution Assessment

No. Rejecting webchat as an outbound announce target is the right fix direction, but this patch checks too narrowly and documents an invalid command; the safer fix should validate the effective delivery mode and original requested channel.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Reject webchat when announce is implicit:** `src/cli/cron-cli/register.cron-add.ts:210-214`
  - body: This guard only runs when `--announce` or deprecated `--deliver` was explicit. Isolated agentTurn cron jobs default to `delivery.mode: "announce"` when neither delivery flag is passed, so `cron add --session isolated --message hello --channel webchat` can still create the forbidden announce-to-webchat config.
  - confidence: 0.94
- **[P2] Reject webchat when edit keeps announce mode:** `src/cli/cron-cli/register.cron-edit.ts:156-162`
  - body: This edit-time guard only checks `opts.announce`. Editing an existing announce job with `cron edit <id> --channel webchat` keeps the existing announce mode via delivery merge, so the same invalid config can still be persisted without hitting the new CLI error.
  - confidence: 0.88
- **[P2] Reject cron webchat before it is normalized away:** `src/infra/outbound/channel-selection.ts:221-227`
  - body: Raw cron jobs still do not pass webchat to this selector. `resolveDeliveryTarget` first calls `resolveSessionDeliveryTarget`, which drops non-deliverable webchat, then calls `resolveMessageChannelSelection({ cfg })`, so the new typed error is bypassed and the cron runtime can still surface the generic no-channel path.
  - confidence: 0.91
- **[P2] Remove the invalid main-session no-deliver example:** `docs/automation/cron-jobs.md:188`
  - body: The new default-agent example uses `--session main --system-event ... --no-deliver`, but `cron add` rejects announce/no-deliver flags outside non-main agentTurn jobs. Users following this documented shape will get the existing validation error instead of creating the reminder.
  - confidence: 0.95
