## Summary

Keep open. Current main fixes the 2026.4.10 queue-wait timeout budget issue, but a resumed CLI-backed cron run with `timeoutSeconds: 600` can still be terminated by the 180s no-output watchdog before the configured cron deadline.

## What This Changes

The issue asks cron agent-turn `timeoutSeconds` and cron update flows to honor long configured deadlines or clearly report any lower cap.

## Best Possible Solution

Make explicit cron run timeouts coordinate with lower CLI no-output and embedded idle watchdog ceilings, or emit a cron-facing diagnostic/config hint when those guards terminate first.

## Reproduction Assessment

Yes for a source-level path: a reused CLI cron session with `timeoutSeconds: 600` passes 600000ms to `runCliAgent`, but `resolveCliNoOutputTimeoutMs` with `useResume: true` returns at most 180000ms. I did not run a live multi-minute cron job in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is in runtime timeout policy: explicit cron deadlines should govern or clearly explain lower watchdog exits across CLI and embedded paths. Changing only the outer cron timer is insufficient because that path already honors the configured value.
