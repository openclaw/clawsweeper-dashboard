## Summary

Keep open: current main still lacks outbound Slack `reply_broadcast`, and this PR is a useful implementation candidate, but the branch still has blocking docs, schema, upload, and changelog issues against the current Slack message-tool surface.

## What This Changes

The PR adds a Slack `replyBroadcast` parameter across Slack send/action dispatch, message-tool forwarding, docs, and tests so threaded sends can request Slack `reply_broadcast`.

## Best Possible Solution

Land a narrow Slack-owned outbound implementation after rebase: expose a discoverable send-only `replyBroadcast` field through the current schema contribution API, forward it only to valid threaded `chat.postMessage` sends, make upload behavior explicit, and cover docs, changelog, and focused Slack tests.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: current main has no outbound `replyBroadcast`/`reply_broadcast` path, and the PR defects reproduce by tracing message-tool send params into Slack dispatch, upload, and payload construction.

## Solution Assessment

No. The feature direction is valid, but this branch is not the best mergeable form until it uses `message-tool-api.ts`, documents current `action`/`message`/`threadId` params, and rejects or explicitly implements upload broadcasts.
