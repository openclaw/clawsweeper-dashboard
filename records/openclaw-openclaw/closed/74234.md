## Summary

Keep this open. Current main and the latest release still validate only the first GnuPG fingerprint record before dearmoring the entire Docker apt key file, and the open linked PR is the direct implementation candidate rather than a reason to close the issue before merge.

## What This Changes

The issue requests hardening Dockerfile Docker apt signing-key validation so a downloaded key file with more than one primary public key is rejected instead of trusting the whole file after checking only the first fingerprint.

## Best Possible Solution

Land PR #74254 or an equivalent narrow Dockerfile change that rejects key files with more than one primary `pub` record before fingerprint comparison and `gpg --dearmor`, while keeping subkey fingerprints valid and adding focused Dockerfile ordering coverage.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: an armored key file with the expected primary key first and an extra primary public key later would pass the current first-`fpr` comparison while `gpg --dearmor` still imports the whole file into the apt keyring.

## Solution Assessment

Yes. Counting primary `pub` records before reading the pinned fingerprint and before dearmoring is the narrow maintainable fix because GnuPG's colon output distinguishes primary public keys from secondary subkeys.
