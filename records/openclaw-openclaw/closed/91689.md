## Summary

Keep open: current main has device pairing plus exec/plugin approval primitives, but not a central risk-classified, device-bound step-up gate for high-risk chat-originated actions. This is security-aligned core design work, not conservative cleanup.

## What This Changes

The issue requests a central gateway authorization layer that requires trusted-device approval before executing high-risk actions originating from chat channels.

## Best Possible Solution

Design this as a core gateway authorization gate that reuses device pairing and approval primitives, defines a risk taxonomy and upgrade-safe defaults, and then threads the policy through chat-originated tool/action execution with docs and tests.

## Reproduction Assessment

Not applicable; this is a feature/security design request, not a broken existing contract. Source inspection can confirm current main lacks the requested step-up ActionRequest and trusted-device challenge surface.

## Solution Assessment

Unclear pending design: the proposed central gateway gate is the right layer, but maintainers need to decide risk classification, enrollment rules, fail-closed defaults, and upgrade behavior before implementation.
