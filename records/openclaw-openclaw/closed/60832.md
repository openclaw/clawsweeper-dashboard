## Summary

Close as implemented on current main: the earlier ClawSweeper comment was accurate for the older checkout, but the Control UI composer plus path now creates and switches to a real dashboard session, while `/reset` remains the explicit in-place reset path.

## What This Changes

Clarify or correct the Control UI composer plus action so users can distinguish a true new session from resetting context in the current session.

## Best Possible Solution

Keep the current product direction: New Chat/New session creates a separate dashboard session, and reset remains an explicit `/reset` or reset-specific action.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: current main shows the composer plus path calling `createChatSession`, which calls `sessions.create`, while `/reset` remains the in-place reset command.

## Solution Assessment

Yes. Making the New session control actually create a session is a better fit than renaming it to Reset context, because it preserves the user expectation described in the report and keeps reset semantics explicit.
