## Summary

Keep open. The PR targets a real source-provable double-wrap bug, but the early-return guard can leave already-wrapped tools running with a previous hook context, and the supplied proof is only Vitest output rather than a real cron/isolated-agent run with a plugin hook.

## What This Changes

The PR adds an early return to `wrapToolWithBeforeToolCallHook` when the input tool already carries the `before_tool_call` wrapper marker.

## Best Possible Solution

Preserve before_tool_call source/context metadata through normalization and ensure `createOpenClawCodingTools` ends with exactly one wrapper built for the current hook context, with regression coverage and a redacted real plugin-run proof.

## Reproduction Assessment

Yes from source: an already wrapped executable tool can pass through normalization and then reach the shared hook wrapper path. I did not run the live cron or isolated-agent scenario in this read-only review.

## Solution Assessment

No. The guard prevents nested wrapper execution, but it does so by returning the old wrapper before applying the current hook context; the safer fix is to preserve metadata and rebuild or dedupe one current-context wrapper.
