## Summary

Current main already fixes the reported update transition: package-manager `update.run` now hands off to the managed CLI path and verifies the restarted Gateway version/reachability instead of swapping packages inside the live Gateway. Protocol mismatch errors also carry version details; this fix is current-main-only and is not in `v2026.5.12`.

## What This Changes

Report that updating from `v2026.5.7` to `v2026.5.12` through Gateway `update.run` can briefly surface a protocol mismatch/offline state after restart signaling before recovery.

## Best Possible Solution

Keep the current-main managed handoff and restart health/version verification path, then ship it in the next release; use `gateway status --deep` diagnostics for any residual stale-client protocol mismatch cases.

## Reproduction Assessment

No high-confidence current-main reproduction remains: the `v2026.5.12` source matches the reported transient package-swap/restart path, but current main no longer uses that path for package-manager `update.run`. I did not run a live macOS upgrade in this read-only review.

## Solution Assessment

Yes. Moving package-manager updates out of the live Gateway process and gating success on post-restart health/version checks is the narrow maintainable fix; accepting incompatible protocols or doing runtime config rollback would be less direct for this transition failure.
