## Summary

Keep open. Current `main` does not implement this cleanup change, and the proposed PR is not safe to close as obsolete: the existing Feishu dispatcher still has the reported error-path cleanup gap, but this PR's `finally`-based reset would also remove the active streaming session before the existing retry path can recover from a close failure.

## Evidence

## Risks / Open Questions

- The underlying bug on `main` still looks real: a rejected update/close path can skip Feishu streaming cleanup and leave dispatcher state stale.
- This PR as written appears to trade that bug for a retry regression: if `streaming.close()` fails once, the existing `onError` retry path would lose access to the active session.
- Even with the PR's `finally`, `partialUpdateQueue` would remain rejected after an earlier update failure, so later streaming attempts in the same dispatcher could stay broken.
