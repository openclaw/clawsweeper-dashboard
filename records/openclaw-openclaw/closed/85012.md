## Label Justifications

- `P1`: The report describes a broken multi-node agent shell workflow where explicit node execution still runs on the gateway/container.
- `impact:security`: The issue concerns a shell execution boundary where commands can run on a different host than the session selected.
- `impact:session-state`: The session accepts and carries exec host/node defaults, but the Codex native shell path does not appear to honor that state.

## Summary

This should stay open: current main still does not make the Codex app-server native shell surface honor a session `host=node` override, and an open maintainer PR is already the active fix path pending real multi-node proof.

## What This Changes

The issue reports that `/exec host=node node=<mac-node>` is accepted, but subsequent Codex agent shell commands still execute on the Linux gateway/container instead of the selected macOS node.

## Best Possible Solution

Make the effective exec host a single source of truth for Codex app-server shell surfaces: node-targeted sessions should use OpenClaw `exec`/`process` routing or fail closed with a clear unsupported-runtime message.

## Reproduction Assessment

Yes, at source level: `/exec host=node` can flow into OpenClaw dynamic tools, while the Codex app-server native shell gate does not check the node exec override. I did not live-run the required Linux gateway plus macOS node setup.

## Solution Assessment

Yes, the active direction is the maintainable fix: disable or reroute Codex native shell for node-targeted sessions so OpenClaw node exec owns the command path. It still needs real multi-node proof before this issue should close.
