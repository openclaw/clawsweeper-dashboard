## Summary

Current main and the shipped `v2026.4.23` release already route Groq STT through a shared OpenAI-compatible audio path that builds `FormData`, appends the uploaded file plus transcription fields, and posts that multipart body to `/audio/transcriptions`; the request-shape tests cover this path.

## Evidence

- **Groq provider uses shared multipart-capable audio helper:** The Groq media-understanding provider delegates `transcribeAudio` to `transcribeOpenAiCompatibleAudio`, with Groq's `/openai/v1` base URL and Whisper model defaults.
  - file: [extensions/groq/media-understanding-provider.ts:14](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/extensions/groq/media-understanding-provider.ts#L14)
  - command: `nl -ba extensions/groq/media-understanding-provider.ts | sed -n '1,120p'`
  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **Shared OpenAI-compatible audio path builds FormData and posts it directly:** `transcribeOpenAiCompatibleAudio` builds `form` via `buildAudioTranscriptionFormData(...)` and sends that form as `body` to `/audio/transcriptions` through `postTranscriptionRequest`, not through the JSON request helper.
  - file: [src/media-understanding/openai-compatible-audio.ts:21](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/media-understanding/openai-compatible-audio.ts#L21)

In the same module, `postTranscriptionRequest` forwards `body: params.body`, while `postJsonRequest` is the separate path that does `JSON.stringify(params.body)`.
