## Summary

Current main already implements the requested cross-channel/session-association behavior for direct chats: DMs share a gateway-owned session by default, Control UI/WebChat uses the same session routing as other channels, and `session.identityLinks` explicitly supports cross-channel session sharing when DM isolation is enabled.

## Evidence

- **DMs already share one session by default:** The session docs say direct messages are a shared session by default, and `main` is the default DM scope. That matches the request's core goal of one conversation history across channels/devices for direct chats.
  - file: [docs/concepts/session.md:16](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/docs/concepts/session.md#L16)

- **Gateway owns the transcript, and multiple channels can map to one session:** The messages docs state sessions are owned by the gateway, direct chats collapse into the agent main session key, and multiple devices/channels can map to the same session. They also note Control UI/TUI show the gateway-backed transcript.
  - file: [docs/concepts/messages.md:65](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/docs/concepts/messages.md#L65)

## Risks / Open Questions

- History is not fully mirrored back to every client surface; `docs/concepts/messages.md` explicitly says multiple channels can share a session while client-visible history can still diverge. That is a UX limitation, but it does not negate that the requested shared-session behavior already exists.
