## Summary

Keep open: current main still has the source-level `[object Object]` path for object-shaped skill config requirements, and an open linked PR uses closing syntax for this issue, so the issue should remain open until that implementation is reviewed, merged, or replaced.

## What This Changes

The issue asks OpenClaw to handle object-shaped skill `requires.config` entries without stringifying them as `[object Object]` or blocking valid skill status checks.

## Best Possible Solution

Review and land the linked parser/docs/tests fix or replace it with a focused change that either supports object entries end to end or rejects them clearly without producing `[object Object]`.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: an object entry in `requires.config` flows through `normalizeStringList()`, becomes `[object Object]`, and is then reported as a missing config path by the requirement evaluator.

## Solution Assessment

Unclear. Extracting `.path` is the narrowest way to remove the visible stringification, but current docs/evaluator semantics treat config requirements as truthy OpenClaw config keys, so maintainers should first choose object support versus an explicit unsupported-shape warning.
