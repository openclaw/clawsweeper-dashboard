## Summary

Current `main` already routes config-write notifications through refreshed runtime/source snapshots instead of replaying a stale redacted runtime snapshot, and gateway reload tests cover the in-process write path that would have re-persisted `__OPENCLAW_REDACTED__`. The same implementation is present in the shipped `v2026.4.23` tree.

## Evidence

- **writeConfigFile notifies with committed snapshots:** `writeConfigFile()` persists `nextCfg`, derives `committedSourceConfig`, then calls `finalizeRuntimeSnapshotWrite(...)`. Its notification closure reads `currentRuntimeConfig` from the refreshed runtime snapshot state and sends that to listeners, rather than forwarding the pre-write snapshot described in the report.
  - file: [src/config/io.ts:2053](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/config/io.ts#L2053)
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **runtime snapshot refresh happens before listener notification:** `finalizeRuntimeSnapshotWrite()` refreshes or reloads runtime/source snapshots first and only then calls `notifyCommittedWrite()`. The listener-notification ordering matches the reporter's suspected fix direction and blocks stale redacted runtime state from being replayed into reload consumers.
  - file: [src/config/runtime-snapshot.ts:88](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/config/runtime-snapshot.ts#L88)
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **gateway reloader consumes source/runtime separately:** The in-process config reload path stores `event.runtimeConfig` as the live config and `event.sourceConfig` as the compare config. That means reload planning no longer needs to infer source state from a redacted runtime snapshot.
  - file: [src/gateway/config-reload.ts:393](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/gateway/config-reload.ts#L393)
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **regression coverage for runtime-materialized secrets:** There is an explicit test asserting that in-process reloads are planned from source config and ignore runtime-materialized paths, which is the failure mode the issue described for provider secrets being written back incorrectly.
  - file: [src/gateway/config-reload.test.ts:839](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/gateway/config-reload.test.ts#L839)
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
