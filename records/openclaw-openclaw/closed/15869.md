## Summary

Current main already supports proxy-routed `web_search` for Brave and related providers via the shared web-tools transport path. The implementation landed as gateway proxy env support (`HTTP_PROXY`/`HTTPS_PROXY`/`ALL_PROXY`) rather than the exact `tools.web.search.proxy` field proposed in the issue, but it satisfies the underlying request well enough to close.

## Evidence

- **Brave web_search calls the shared trusted web-search transport:** The Brave provider runtime routes both LLM-context and normal web search requests through `withTrustedWebSearchEndpoint(...)`, so Brave no longer uses a direct bare `fetch` path for web_search.

## Close Comment

Closing from a Codex review: current `main` already supports proxy-routed `web_search` for Brave and the other managed web-search providers. The Brave runtime now calls the shared trusted web-search transport at `extensions/brave/src/brave-web-search-provider.runtime.ts:89` and `:152`; that flows through `src/agents/tools/web-search-provider-common.ts:77` into `src/agents/tools/web-guarded-fetch.ts:64`, which enables the trusted env-proxy path for web tools. The shipped changelog entry at `CHANGELOG.md:3842` also explicitly calls out `web_search` proxy support behind `HTTP_PROXY` / `HTTPS_PROXY` / `ALL_PROXY`. This landed as env-proxy support rather than the exact `tools.web.search.proxy` field from the proposed patch, but the underlying feature request is already implemented on `main` and present in release `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`).
