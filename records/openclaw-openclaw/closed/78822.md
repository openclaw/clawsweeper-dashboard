## Summary

Close as implemented: current main and the shipped v2026.5.6 tree already handle startup-pending gateway reconnects with a retry-after path that avoids the normal terminal disconnect/backoff behavior reported from 2026.5.5.

## What This Changes

The report asks for Control UI gateway restart reconnects to avoid a roughly 10-second visible disconnected window when the Gateway is listening but not yet ready.

## Best Possible Solution

Keep the shipped startup-unavailable retry path and ask for a fresh report only if 2026.5.6 or newer still shows a long restart disconnect with new timestamps and Control UI event timing.

## Reproduction Assessment

No current-main failure path was reproduced. Source and regression tests on current main show startup-pending handshakes now retry on the server-provided delay without surfacing a terminal Control UI close callback.

## Solution Assessment

Yes. The shipped fix is narrow and maintainable: the Gateway marks startup-pending connects as retryable, and the existing Control UI gateway client handles only that specific condition with short retry timing rather than changing restart policy broadly.
