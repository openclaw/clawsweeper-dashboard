## Summary

Keep open. The feature belongs in the Codex plugin, but the PR currently adds a blocking best-effort goal poll after a bound turn has already completed, so a slow or unsupported goal RPC can delay the user-visible Codex reply; it also needs real behavior proof for the new `/goal` and terminal wake flow.

## What This Changes

The PR adds Codex `/goal` and `/codex goal` controls plus a bridge from terminal Codex goal states into OpenClaw system events and heartbeat wakes.

## Best Possible Solution

Keep the feature in the Codex plugin, make terminal-goal notification non-blocking or tightly timed, rebase onto current Codex command/binding code, and prove the real `/goal` plus bound-session terminal wake path before merge.

## Reproduction Assessment

Yes, source-reproducible for the review finding: after a bound turn completes, a `thread/goal/get` call that stalls will hold the completed reply until the app-server request timeout. I did not execute this path because the checkout is read-only.

## Solution Assessment

No. The Codex plugin boundary is appropriate, but the best-effort terminal bridge should not sit on the critical path for delivering an already completed Codex reply.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Do not block the completed reply on goal polling:** `extensions/codex/src/conversation-binding.ts:449-459`

With the default timeout at 60s, a slow or wedged optional goal RPC can keep the user from seeing an already completed Codex reply; run the bridge in the background or use a much smaller timeout.
