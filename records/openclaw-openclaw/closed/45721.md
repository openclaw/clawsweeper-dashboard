## Summary

Current `main` already wires `channels.discord.proxy` into Discord REST paths, including the outbound client used for sends/replies, so the reported gap is implemented on `main` even though it does not appear to be in the latest tagged release from this checkout.

## Evidence

- **Discord outbound REST client now receives proxy-backed fetch:** `createDiscordRestClient` resolves the account proxy and passes `{ fetch: resolvedProxyFetch }` into `createDiscordRequestClient`, covering normal Discord REST sends/replies instead of only the gateway WebSocket path.

- **Local Carbon-compatible request client preserves custom fetch seam:** OpenClaw keeps a local `ProxyRequestClientCompat` specifically because Carbon removed the fetch seam, and stores `options?.fetch` as `customFetch` so proxied REST requests still work.
  - file: [extensions/discord/src/proxy-request-client.ts:132](https://github.com/openclaw/openclaw/blob/37c37eecfbe4bef413c33b8606db62cf51c2d1d4/extensions/discord/src/proxy-request-client.ts#L132)
