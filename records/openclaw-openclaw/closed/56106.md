## Summary

Current main and the latest release already route session transcript writes through OpenClaw-owned JSONL helpers that force UTF-8, and the memory side paths checked also use explicit UTF-8 writes; this issue can close as implemented.

## What This Changes

The issue reports corrupted Chinese text in Windows JSONL session transcripts and possible memory files, requesting UTF-8-safe transcript writes.

## Best Possible Solution

Keep the shipped centralized UTF-8 transcript and memory write paths; if corruption persists on v2026.6.6 or newer, open a fresh inbound/streaming-layer report with raw payload text and the persisted JSONL line.

## Reproduction Assessment

Yes, at source level: the reported v2026.3.23 message append path delegated persistence to an external SessionManager after only the header was written explicitly as UTF-8. I did not run a live Windows QQBot reproduction in this read-only cleanup review.

## Solution Assessment

Yes. Moving the OpenClaw-owned transcript persistence path to an internal SessionManager and centralized JSONL helpers that force UTF-8 is the narrow maintainable fix for the reported persisted JSONL corruption surface.
