## Summary

Close as superseded. The PR targets a real doctor SecretRef false-positive, but its current patch should not land: it can abort doctor when the SecretRef is still unresolved, and the same remaining work is now tracked by open PR #67463 with a narrower non-throwing resolver approach and regression coverage.

## What This Changes

This PR attempts to resolve SecretRef-backed `gateway.auth.token` before the doctor gateway auth health check, while also carrying unrelated sync-main drift in memory-core, generated config schema, method scopes, provider contract tests, and secrets test support.

## Best Possible Solution

Use #67463 or a narrow equivalent that resolves `gateway.auth.token` through the non-throwing token helper, preserves the documented warning when a SecretRef is unresolved, avoids broadening SecretRef execution beyond the intended doctor path, and keeps the fix focused with doctor regression tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve doctor warning when SecretRefs are unresolved:** `src/flows/doctor-health-contributions.ts:176`
  - body: `resolveGatewayTokenSecretRefValue` ultimately calls the required SecretRef resolver, which throws when the configured token ref is unavailable. In that common recovery case, `openclaw doctor` would abort instead of emitting the existing read-only warning, so this needs the non-throwing token resolution path or an explicit catch before merge.
  - confidence: 0.91

## Security Review

Status: needs_attention

Concerns:

- **[medium] Avoid broad exec SecretRef execution in doctor:** `src/flows/doctor-health-contributions.ts:176`
