## Summary

Current main now has session liveness diagnostics and recovery that cover the central request: no-progress sessions are warned on, stale lanes are released, stalled embedded runs can be abort-drained, and compaction/context-overflow failures rotate to a fresh session rather than requiring deletion of all sessions.

## What This Changes

Add a session health check that warns and automatically recovers or rotates stuck sessions after prolonged no-progress unresponsiveness.

## Best Possible Solution

Keep the current diagnostics-driven recovery design: classify no-progress sessions, release stale lanes, abort-drain only after a safer extended stall window, and rotate sessions for compaction/context-overflow failures instead of deleting all sessions.

## Solution Assessment

Yes. The current implementation is narrower and safer than a blanket sessions.json delete because it distinguishes stale bookkeeping, slow active work, stalled embedded runs, and compaction failure before recovering.
