## Summary

Close: current main already narrows plugin package install-time code safety scans to plugin-owned runtime entrypoints while preserving dependency manifest denylist checks, so large third-party dependency trees should no longer block installs with the scan file-limit error. The fix is on current main only and is not included in v2026.5.12.

## What This Changes

The issue asks for `openclaw plugins install @mem0/openclaw-mem0` to stop failing when the install-time code safety scan reaches its file limit.

## Best Possible Solution

Keep the current targeted install-time scanner behavior, and leave the broader deep-audit performance work tracked separately at https://github.com/openclaw/openclaw/issues/79274.

## Reproduction Assessment

Yes for the historical failure shape: the install scanner path and regression test cover packages whose dependency runtime files would previously trigger scanner limits or warnings. I did not run the live mem0 install because this review is read-only and plugin installation mutates local state.

## Solution Assessment

Yes. Targeting install-time code safety scans to plugin-owned runtime entrypoints while preserving dependency manifest denylist checks is the narrow owner-boundary fix for large trusted dependency trees.
