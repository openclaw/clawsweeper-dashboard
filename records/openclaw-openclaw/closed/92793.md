## Summary

Keep open: the stale-config idea is useful, but this PR is not merge-ready because it would still leave the linked Feishu named-account dynamic-agent bug unresolved and it adds a new refreshed-config return mode that only the DM bot path consumes. The external real-behavior proof is mock/unit-test only, so live Feishu routing evidence is still needed before merge.

## What This Changes

The PR makes Feishu dynamic-agent creation consult `runtime.config.current()` for an existing direct-user binding and re-resolve bot DM routes when that returns a fresher config, with helper unit coverage.

## Best Possible Solution

Land a Feishu plugin fix that makes dynamic binding detection and creation account-aware, updates every `maybeCreateDynamicAgent` caller to consume refreshed configs, preserves default-account legacy behavior, and includes redacted live Feishu routing proof.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: Feishu dynamic-agent bindings are written without `accountId`, and routing tests show account-less bindings only match the default account. I did not run a live Feishu tenant reproduction in this read-only review.

## Solution Assessment

No. The refreshed-config lookup is a plausible narrow mitigation for one DM path, but the best fix must be account-aware and must update all helper callers that can receive the refreshed config.
