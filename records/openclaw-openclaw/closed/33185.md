## Summary

Keep open. Current `main` still has a real custom-vision-model lookup gap in the media-understanding/image path, and a maintainer comment on April 25, 2026 explicitly says the issue remains the canonical open tracker pending PR #71458.

## Evidence

- **Current media image path still does bare registry lookup:** `resolveImageRuntime` still calls `modelRegistry.find(resolvedRef.provider, resolvedRef.model)` and throws `Unknown model` on miss, with no fallback for provider-prefixed custom model IDs.
  - file: [src/media-understanding/image.ts:144](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/src/media-understanding/image.ts#L144)
  - command: `nl -ba src/media-understanding/image.ts | sed -n '130,147p'`
  - sha: [5fe333ada85a](https://github.com/openclaw/openclaw/commit/5fe333ada85afd0def9e85985ce57e36aa00108a)
- **Shared helper already has the missing provider-prefixed fallback:** `resolveModelFromRegistry` retries lookup with `${provider}/${model}` when the plain model id misses, which is the exact custom-provider case this issue discusses.
  - file: [src/agents/tools/media-tool-shared.ts:430](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/src/agents/tools/media-tool-shared.ts#L430)
  - command: `nl -ba src/agents/tools/media-tool-shared.ts | sed -n '425,443p'`
  - sha: [5fe333ada85a](https://github.com/openclaw/openclaw/commit/5fe333ada85afd0def9e85985ce57e36aa00108a)

That protection is not used by `src/media-understanding/image.ts`.
  - file: [src/agents/tools/media-tool-shared.test.ts:66](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/src/agents/tools/media-tool-shared.test.ts#L66)

That documented behavior is not fully met while the bare lookup remains in place.

That is strong evidence against conservative closure.
