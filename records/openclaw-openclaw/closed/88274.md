## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/88274 as a generic OpenClaw gateway service-env bug. Current behavior can render state-dir/config service env values containing nested shell references into generated LaunchAgent env files as literal values, causing plugin credential validation to receive strings such as "$SUPERMEMORY_OPENCLAW_KEY". Keep the fix in core service-env planning/rendering: resolve only safe already-available values or omit/warn on unresolved nested references before generated macOS LaunchAgent/systemd service env files are written. Do not hard-code Supermemory plugin IDs, API-key prefixes, user examples, or generated wrapper workarounds. Add focused regression coverage for state-dir `.env` and config `env.vars` inputs on macOS LaunchAgent planning/rendering, and consider shared coverage so Linux systemd generated env files do not preserve the same bad literal. Preserve the existing owner-only env-file/wrapper security model and do not edit CHANGELOG.md; put release-note context in the PR body if user-visible behavior changes.
