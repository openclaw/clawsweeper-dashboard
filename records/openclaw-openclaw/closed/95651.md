## Summary

Current main and the latest release already bound cron run-log growth with `cron.runLog.keepLines`, insert-time SQLite pruning, paged reads, docs, and regression coverage, so this issue can close as implemented.

## What This Changes

The issue asks for automatic retention or pruning for accumulated rows in the `cron_run_logs` SQLite table and less stale run-log clutter in cron job details.

## Best Possible Solution

Keep the shipped SQLite-backed `cron.runLog.keepLines` pruning path and direct operators to configure a lower value when they want a shorter retained run history.

## Reproduction Assessment

No current-main failure reproduces: current `main` and `v2026.6.9` both have row-count retention and insert-time pruning for `cron_run_logs`. The reported old-build symptom is credible, but the present source/test evidence shows it is already fixed.

## Solution Assessment

Yes; pruning in the cron run-log storage owner is the best current fix because it bounds persisted SQLite history at write time and uses the existing `cron.runLog.keepLines` config instead of adding a competing manual SQL workflow.
