## Label Justifications

- `P2`: The issue describes a real interactive-lane hang, but current main already contains a bounded recovery path and the remaining concern is release availability or tuning.
- `impact:crash-loop`: The reported failure leaves an interactive lane wedged until timeout or recovery.
- `impact:message-loss`: The report says Discord/Telegram turns can queue behind the blocked tool and receive no timely response.
- `impact:session-state`: The affected surface is active session processing, diagnostic liveness, and blocked tool bookkeeping.

## Summary

Current main already covers the reported failure mode with generic stalled active-work recovery: blocked tool calls become recovery-eligible by active tool age and stale progress, then the diagnostic recovery runtime abort-drains the active embedded run instead of waiting for the outer run-lane backstop.

## What This Changes

The issue requests a dedicated, configurable per-active-tool timeout for bash tool calls running under the Codex app-server so wedged interactive lanes fail faster than the existing run-lane backstop.

## Best Possible Solution

Keep the generic diagnostic active-work abort-drain path as the canonical fix and tune `diagnostics.stuckSessionAbortMs` when operators need a different recovery threshold.

## Reproduction Assessment

Yes, from source inspection: the reported `blocked_tool_call` state shape is represented by active tool diagnostics, and current main has focused recovery coverage for a stale `bash` tool on an active embedded run. I did not run a live Codex app-server hang.

## Solution Assessment

Yes; the current generic diagnostic recovery is a better owner-boundary fix than a Codex-bash-only timer because it reuses the existing active embedded-run abort/drain machinery and exposes one documented timeout knob.
