## Summary

Current main covers this Telegram missing-runtime-dependency symptom: the Telegram plugin declares the grammY packages as staged bundled runtime deps, enabled bundled plugins install from their package dependency set before import, and the post-4.29 runtime-deps convergence fix prevents later lazy activation from pruning an already staged package-level dependency superset. The fix is only on main; no release tag contains it yet.

## What This Changes

The issue reports that OpenClaw 2026.4.29 fails to load the bundled Telegram channel after upgrade because the managed runtime-deps root is missing `grammy`, `@grammyjs/runner`, and `@grammyjs/transformer-throttler`.

## Best Possible Solution

Keep the generic runtime-deps staging/convergence fix on main and let the next release carry it to npm users, while leaving broader runtime-deps loop or event-loop starvation evidence in the canonical open tracker rather than reopening this narrower Telegram dependency symptom.

## Reproduction Assessment

Yes. The released-package reproduction is concrete: update from 4.23 to 4.29, restart the gateway, and observe Telegram fail to load with `Cannot find module 'grammy'`; current-main source and regression tests map that failure class to staged runtime-deps convergence rather than to a missing Telegram dependency declaration.

## Solution Assessment

Yes. Fixing bundled runtime-deps convergence generically is the maintainable solution; a Telegram-only vendoring or startup special case would duplicate the shared plugin staging contract.
