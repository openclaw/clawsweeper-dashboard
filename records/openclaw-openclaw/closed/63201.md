## Summary

Close as implemented on `main`. The current audio transcription path still submits `FormData`, but dispatcher-backed requests now pass through a runtime fetch shim that converts global `FormData` into Undici's runtime `FormData` and clears stale multipart headers before dispatch. That directly addresses the reported `HTTP 400` multipart-boundary failure, and there is a targeted regression test covering the conversion and header handling. The fix is on `main` but does not appear to be in the latest tagged release yet.

## Evidence

- **Issue discussion identified the exact failing seam:** The issue body, confirmation comment from `wangyuyan-agent`, and cross-link to PR #63202 all point to the same root cause: `FormData` built with the global Web API was being sent through an Undici dispatcher path, causing multipart requests to arrive without a usable boundary.
  - command: `Provided GitHub context for issue #63201 comments/timeline`
- **Audio transcription still sends a multipart FormData body through the guarded POST helper:** `transcribeOpenAiCompatibleAudio` builds the upload with `buildAudioTranscriptionFormData(...)` and passes that `FormData` to `postTranscriptionRequest(...)`, so this remains the exact code path the report was about.

That is the path used for guarded provider requests.

- **Runtime fetch shim fixes FormData interoperability:** `normalizeRuntimeFormData` rebuilds global `FormData` into the runtime Undici `FormData` implementation, preserves filenames, and `normalizeRuntimeRequestInit` removes `content-length` and `content-type` so Undici can generate the correct multipart headers and boundary.
