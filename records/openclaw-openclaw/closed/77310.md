## Summary

Current `main` fixes the user-visible gateway `web_search` failure by falling through from an empty active runtime registry to a scoped web-provider plugin load; Brave still does not need `activation.onStartup: true` for the assistant tool to find its provider.

## What This Changes

Issue reports that external npm web-search plugins such as Brave are enabled but not loaded by the gateway at startup, leaving gateway `web_search` with no provider.

## Best Possible Solution

Keep the current lazy on-demand web-provider loading path so gateway startup stays lean while `web_search` can still resolve installed provider plugins.

## Reproduction Assessment

No, not on current `main`: I did not run a live gateway repro in this read-only review, but source and regression tests show the empty-runtime-registry failure path now falls through to a scoped provider load instead of returning no providers.

## Solution Assessment

Yes: the current on-demand fallback is narrower than forcing every web-search plugin into gateway startup and preserves the plugin lazy-loading boundary while fixing the reported assistant-tool failure.
