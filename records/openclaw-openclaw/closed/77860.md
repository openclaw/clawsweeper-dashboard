## Summary

Current main already fixes the reported Ollama `/think` menu mismatch by exposing the Ollama thinking profile through the lightweight provider-policy surface, so reasoning-capable Ollama models no longer fall back to the generic `off, minimal, low, medium, high` choices. The fix is present on main but is not proven shipped in `v2026.5.4`.

## What This Changes

Request that Ollama `/think` choices reflect provider/model-supported thinking modes instead of the generic static list.

## Best Possible Solution

Keep the current main implementation, ship it in the next release, and treat any remaining model-specific wire-level Ollama `max` behavior as a separate focused runtime issue if still observable.

## Reproduction Assessment

No against current main: the old fallback is source-reproducible from `v2026.5.4`, where the lightweight Ollama provider-policy artifact lacks `resolveThinkingProfile`, but current main exposes the profile and has regression coverage for the corrected option list.

## Solution Assessment

Yes for the reported `/think` menu mismatch: exposing the provider-owned profile through the lightweight public artifact is the narrow fix that preserves plugin laziness without forcing Ollama startup activation. A separate live-backed issue would be the safer path for exact per-model wire-effort discovery beyond this menu regression.
