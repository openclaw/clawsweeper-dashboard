## Summary

Keep open: current main still sends managed skills through the shared realpath containment check, and open implementation PRs still target this issue, so closing before one lands would be premature.

## What This Changes

The issue reports that ClawHub or symlink-backed managed skills under `~/.openclaw/skills` are rejected as escaped paths instead of loading.

## Best Possible Solution

Land or replace the canonical managed-skills fix so managed/personal symlink-backed roots load while workspace, extra, project, bundled, and per-skill `SKILL.md` containment stay fail-closed.

## Reproduction Assessment

Yes. A managed skill directory whose realpath leaves `managedSkillsDir` still reaches `resolveContainedSkillPath()` with source `openclaw-managed` and is skipped unless the operator has configured an explicit symlink target allowlist.

## Solution Assessment

Yes, the source-scoped containment approach in https://github.com/openclaw/openclaw/pull/59219 is the narrow maintainable direction; current main has only the opt-in allowlist path, so this issue should remain open until a fix lands.
