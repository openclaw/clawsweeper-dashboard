## Summary

Keep open: the reported LM Studio tool-call argument loss still appears actionable on current main and the latest release. The reporter's hot patch matches the current shared Responses stream parser, which can overwrite buffered tool-call arguments with a missing `done` payload and then finish with `{}`.

## What This Changes

Fix LM Studio/OpenAI Responses tool calls so argument-bearing calls preserve their arguments instead of reaching tool execution as empty objects.

## Best Possible Solution

Patch the shared LLM Responses stream parser to preserve buffered arguments when a done event omits arguments, add focused regression coverage, and leave config/docs behavior unchanged.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction path from the reported LM Studio Responses stream shape: streamed argument deltas followed by a done event without usable `arguments` leave the final tool call with `{}`. I did not run a live LM Studio model in this read-only review.

## Solution Assessment

Yes, the best fix is a narrow parser repair in the shared LLM Responses stream handler plus a regression test for the LM Studio-style event sequence. A config flag or LM Studio-only workaround would be the wrong layer because the bug is argument preservation in the shared stream parser.
