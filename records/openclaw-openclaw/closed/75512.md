## Summary

Keep open. The issue is recent, has detailed cross-platform confirmations, and current main still shows embedded chat prep paths that can rebuild plugin runtime/load-context work instead of consistently reusing the gateway-owned plugin snapshot or active gateway-bindable registry.

## What This Changes

The issue reports severe per-turn embedded chat latency from repeated plugin runtime and metadata re-evaluation after upgrading past v2026.4.23.

## Best Possible Solution

Reuse explicit gateway-owned plugin metadata/runtime handles throughout embedded chat prep: pass the active gateway-bindable registry or a caller-owned lookup table/manifest registry into tool construction and middleware resolution, avoid hidden wall-clock discovery caches, and add focused tests that fail when a normal chat turn re-enters broad plugin discovery or runtime loading.

## Reproduction Assessment

Yes, the issue has a high-confidence reproduction path for affected installs: start a gateway with bundled plugins enabled, send repeated embedded chat turns through a channel, and compare `[trace:embedded-run]` prep stages plus repeated `[plugins] loading ...` logs across turns. This review did not run that live setup locally, but multiple independent comments provide platform-specific traces.

## Solution Assessment

Yes, the proposed direction is the narrow maintainable solution: thread existing explicit snapshot/registry ownership through the hot paths instead of adding a hidden process-wide metadata cache. The safer repair should preserve plugin install/config freshness and only reuse handles while they match the active gateway config/workspace.
