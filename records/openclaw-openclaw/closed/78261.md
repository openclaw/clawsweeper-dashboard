The PR was already closed by the Telegram maintainer after API/product review: current OpenClaw treats a provider-returned Telegram message id as the provider acceptance signal, so adding a new SDK-wide `delivery.providerAccepted` projection is not an accepted source change.

## What This Changes

The PR adds an optional `delivery.providerAccepted` projection to outbound and channel-message send results, annotates successful Telegram sends, and updates Telegram tests, SDK docs, and changelog text.

## Best Possible Solution

Keep the existing receipt/message-id contract unless maintainers later choose a broader delivery-state API with a separate accepted design.

## Reproduction Assessment

Not applicable as a strict bug reproduction; this PR proposes an additive result projection rather than fixing a documented broken behavior. The relevant current-main contracts lack the field, and the PR body demonstrates the proposed after-fix behavior.

## Solution Assessment

No. The maintainer decision is that the existing provider-returned Telegram message id already represents provider acceptance, while read or visibility state remains separate, so a new SDK-wide `delivery.providerAccepted` projection is not the best accepted path.
