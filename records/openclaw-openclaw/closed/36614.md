## Summary

Keep open: current main still has direct-message channel write paths that pass the shared main session into `updateLastRoute` while the documented `per-channel-peer` route contract isolates those DMs by channel and peer.

## What This Changes

Prevent inbound direct messages under `session.dmScope: per-channel-peer` from writing channel-specific last-route delivery context onto the main session.

## Best Possible Solution

Apply the existing inbound last-route session-key policy consistently across the remaining direct-message channel write sites, with regression coverage proving isolated DM sessions do not update `agent:<id>:main`.

## Reproduction Assessment

Yes, at source level: `per-channel-peer` route resolution produces an isolated `lastRoutePolicy: session` key, while multiple direct-message handlers still pass `mainSessionKey` to `recordInboundSession` for `updateLastRoute`. I did not run a live channel reproduction in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is to use `resolveInboundLastRouteSessionKey` or an equivalent central contract at the remaining write sites, rather than only adding downstream fallbacks such as the open message-tool PR.
