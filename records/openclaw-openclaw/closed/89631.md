## Summary

Keep open. Current main lacks this SDK accessor, but the branch changes a public plugin SDK subpath and likely makes existing provider-usage imports eagerly load core usage/runtime code, with no real behavior proof or SDK contract updates.

## What This Changes

The PR adds two exported provider usage-limit accessors and two result types to `src/plugin-sdk/provider-usage.ts`, wrapping the core usage summary loader with a 60-second cache and a stale-while-revalidate cached read.

## Best Possible Solution

Expose a provider-usage reader only after maintainer approval as an import-light SDK/runtime seam with docs, API baseline, focused tests, and real proof from the consuming plugin.

## Reproduction Assessment

Not applicable as a feature PR. Source inspection confirms current main lacks the accessor and the branch adds it, but this read-only review did not run the new runtime behavior and the PR has no inspectable proof.

## Solution Assessment

No. The accessor idea is plausible, but adding the core usage loader as a top-level import on the existing provider-usage SDK subpath is not the best shape for existing consumers; a lazy or dedicated runtime seam with contract updates is safer.
