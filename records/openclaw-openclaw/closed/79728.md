## Summary

Keep this PR open with a blocking finding: the new assistant-prefill format patterns also feed `isCloudCodeAssistFormatError`, and the new `allowFormatRetry` hook can make the reported rejection retryable instead of terminal.

## What This Changes

The branch adds assistant-prefill provider errors to failover format classification, changes terminal format failover policy with an allow-retry exception, adds regression tests and changelog text, and refreshes a generated Swift protocol model.

## Best Possible Solution

Separate retryable Cloud Code Assist tool-call/schema sanitization errors from terminal format rejections, then prove the exact assistant-prefill message path resolves to `surface_error` without auth-profile rotation or model fallback.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source and report evidence: current main lacks the reported format strings, and the linked issue provides concrete provider error text and lane-loop logs; I did not run the live Discord/provider scenario.

## Solution Assessment

Is this the best way to solve the issue? No: the intended fail-fast policy is narrow, but this implementation reuses the expanded format matcher for `cloudCodeAssistFormatError`, so the reported terminal error can still enter the retry path.

## Review Findings
