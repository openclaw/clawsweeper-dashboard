## Summary

Keep open: this PR targets a real current-main bug, but the patch currently pauses runs selected by `requesterSessionKey` instead of the active yielding `childSessionKey`, and the external PR still lacks required real claude-cli MCP after-fix proof.

## What This Changes

The PR forwards `sessionId` and `onYield` from the MCP loopback server/cache through `resolveGatewayScopedTools` into `createOpenClawTools`, adding an MCP-path yield callback in `src/gateway/mcp-http.ts`.

## Best Possible Solution

Land or replace this with a narrow gateway fix that forwards yield context through the MCP loopback path, pauses only the active yielding run, and proves the real claude-cli MCP flow.

## Reproduction Assessment

Yes, source-reproducible. Current main resolves MCP loopback tools without `sessionId` or `onYield`, and `createSessionsYieldTool` deterministically returns `No session context` when `sessionId` is absent; I did not run a live claude-cli MCP setup.

## Solution Assessment

No for this PR as written. Threading the existing parameters is the right direction, but the callback should identify the active yielding run by `childSessionKey` or a run-bound id before calling `markSubagentRunPausedAfterYield`.

## Review Findings
