## Summary

Keep open: current main and v2026.6.5 still use the hard-coded SDK thinking fallback that can send Ollama a non-off `think` request, and https://github.com/openclaw/openclaw/pull/91657 is the open linked repair path rather than a merged fix.

## What This Changes

The issue reports that local Ollama Gemma4/Qwen3.6 replies render only the first token in OpenClaw while direct Ollama streaming returns the full response.

## Best Possible Solution

Review and land or revise https://github.com/openclaw/openclaw/pull/91657 so SDK-created Ollama sessions honor the provider `off` default while preserving explicit/user-configured thinking levels, then close this issue after merge.

## Reproduction Assessment

## Solution Assessment

Yes, the best current fix direction is to honor the provider-declared Ollama `off` default in SDK session creation rather than changing chunk parsing or adding a new config knob. The open linked PR is the focused implementation candidate, pending maintainer review and live-provider proof.
