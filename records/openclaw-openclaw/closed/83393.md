## Summary

Keep open: current main and the latest shipped release still have a source-reproducible path where Codex/internal direct-chat delivery can suppress ordinary final replies, and a fresh maintainer assignment/comment says the 1:1 fallout needs revisiting rather than cleanup closure.

## What This Changes

Request that Signal-originated direct-session final assistant replies be delivered back to the captured Signal route even when processed through Codex, vscode, or another internal runner surface.

## Best Possible Solution

Make a maintainer-approved 1:1/direct visible-reply policy for Codex-backed channel turns, then add focused Signal/internal-runner regression coverage that preserves explicit message_tool opt-in behavior.

## Reproduction Assessment

Yes at source level: current main and v2026.5.12 show Codex direct chats can resolve to message_tool_only and suppress automatic final delivery, while internal WebChat turns without ExplicitDeliverRoute do not route to an external origin. I did not run a live Signal roundtrip.

## Solution Assessment

Unclear as an automated fix: restoring automatic 1:1 delivery for this path addresses the observed message loss, but maintainers need to choose the compatibility and privacy boundary for Codex message_tool defaults and internal-surface origin routing.
