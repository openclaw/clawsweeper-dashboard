## Summary

Keep open: current main and the latest shipped release still mark startup-interrupted cron runs as `error` and increment `consecutiveErrors`, so the reported counter/backoff inflation remains source-reproducible. The related retry PR is open and unmerged, and it targets isolated agent-turn retry behavior rather than the broader counter semantics for all persisted `runningAtMs` jobs.

## What This Changes

The issue asks cron restart interruptions to be distinguishable from agent-turn failures and not inflate `consecutiveErrors` or execution-error backoff.

## Best Possible Solution

Keep the visible restart-interruption record, but choose and document cron state semantics so normal Gateway restarts or hot reloads do not inflate execution-failure counters/backoff.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: persist a cron job with `state.runningAtMs`, start the cron service, and `markInterruptedStartupRun()` records a gateway-restart error while incrementing `consecutiveErrors`.

## Solution Assessment

Unclear. A counter-only fix could be narrow, but adding an `interrupted` state/reason or changing retry behavior affects the public cron status/protocol contract and should be settled by maintainers first.
