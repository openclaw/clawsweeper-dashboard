## Summary

Current main and the latest release already implement the requested provider-free cron command path as `payload.kind: "command"`; the shipped name differs from the proposed `localCommand`, but the central scheduler, timeout, output, delivery, and run-history behavior is present.

## What This Changes

The issue requests a deterministic local-command cron payload for bounded maintenance jobs that should run without an agent/model turn while recording execution results.

## Best Possible Solution

Keep the shipped `command` cron payload and point users to the cron command-payload docs instead of adding a second `localCommand` alias.

## Reproduction Assessment

Not applicable: this is a feature request rather than a bug report. Source, docs, tests, and release-tag inspection show the requested behavior is already available as command cron payloads.

## Solution Assessment

Yes: the shipped `command` payload is the narrow maintainable solution because it adds one explicit cron payload variant with argv and bounded execution fields instead of a parallel `localCommand` naming path.
