## Summary

Keep open. Current `main` still does not implement the Telnyx inbound handling described by PR #40188, and the PR as provided only adds interface/type scaffolding rather than the provider and manager changes needed to fix the bug.

## Evidence

- **Telnyx normalized events still omit inbound routing fields:** `TelnyxProvider.normalizeEvent` builds `baseEvent` with only `id`, `dedupeKey`, `callId`, `providerCallId`, and `timestamp`. It never sets `direction`, `from`, or `to`, so inbound Telnyx webhooks cannot supply the data the manager needs to auto-register inbound calls.
  - file: [extensions/voice-call/src/providers/telnyx.ts:140](https://github.com/openclaw/openclaw/blob/bfa6708c03f24c7caa5d0a96a298c292be0d85c1/extensions/voice-call/src/providers/telnyx.ts#L140)
  - command: `nl -ba extensions/voice-call/src/providers/telnyx.ts | sed -n '1,380p'`
  - sha: [bfa6708c03f2](https://github.com/openclaw/openclaw/commit/bfa6708c03f24c7caa5d0a96a298c292be0d85c1)
- **Manager auto-registration explicitly depends on `event.direction`:** `processEvent` only creates a webhook-backed call when `!call && providerCallId && eventDirection`. Since Telnyx events on current `main` never populate `direction`, this path is skipped for Telnyx inbound calls.
  - file: [extensions/voice-call/src/manager/events.ts:110](https://github.com/openclaw/openclaw/blob/bfa6708c03f24c7caa5d0a96a298c292be0d85c1/extensions/voice-call/src/manager/events.ts#L110)
