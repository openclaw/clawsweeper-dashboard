## Summary

Close as implemented. Current main routes `heartbeat.to` through Discord's explicit target parser, which now preserves `user:<snowflake>` instead of handing a bare snowflake to outbound resolution, and the same behavior is present in the shipped `v2026.4.29` tag.

## What This Changes

Ensure Discord heartbeat delivery with `to: user:<snowflake>` keeps the `user:` DM target instead of converting it to `channel:<snowflake>`.

## Best Possible Solution

Keep the shipped Discord plugin parser behavior and its regression coverage; no further OpenClaw core change is needed for this reported 2026.4.24 failure.

## Reproduction Assessment

Yes. The old failure path is source-reproducible by setting a Discord heartbeat `to` value to `user:<id>` and tracing `resolveHeartbeatDeliveryTarget`; current main no longer reproduces because the explicit target parser returns `user:<id>` before outbound normalization.

## Solution Assessment

Yes. Preserving the normalized Discord target at the plugin parser boundary is the narrowest maintainable fix because it keeps channel-specific target grammar inside the Discord plugin while the generic heartbeat path remains extension-agnostic.
