## Summary

Current main already addresses the operator-facing follow-up: bundled runtime-deps staging is scoped to the gateway startup plugin set and policy checks, disabled channels/plugins are honored, local memory adds only its local embedding runtime dependency, and readiness/probe paths now account for startup/runtime-deps and event-loop availability. The proof is on current main only; no release tag contains the proof SHA yet.

## What This Changes

The issue asks for local memory enablement to avoid installing unused bundled channel runtime deps, keep disabled channels from blocking readiness, make readiness/probe state accurate during runtime-deps work, and recover safely from interrupted runtime-deps installs.

## Best Possible Solution

Keep the current main behavior: scoped runtime-deps staging for the exact startup plugin set, explicit disablement precedence, event-loop-aware probes, and readiness that stays red while startup/runtime-deps work is pending. Any remaining `Unknown memory embedding provider: local` symptom should continue in the original local-provider registration thread rather than this runtime-deps follow-up.

## Reproduction Assessment

Partially. The original VPS canary was not reproduced live in this read-only review, but current-main source and regression tests cover the specific runtime-deps scoping, readiness, and probe semantics that explain the reported failure mode.

## Solution Assessment

Yes. Solving this in the gateway/plugin activation and runtime-deps planner is the narrow maintainable fix; it avoids operator-only workarounds and prevents unused bundled channels from being staged when memory-core is the trigger.
