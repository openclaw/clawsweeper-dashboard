## Summary

Close as implemented on current main: the reported release had unbounded Telegram account startup, but current main now routes channel and account startup through a shared concurrency cap of four with regression coverage; this is main-only because v2026.5.4 does not contain the proof commit.

## What This Changes

Request startup concurrency and optional delay controls so multi-account Telegram startup does not run every bot getMe probe at once.

## Best Possible Solution

Keep the generic channel startup concurrency cap as the default fix; consider a separate config-design issue only if real main builds still need operator-tunable staggering.

## Reproduction Assessment

Not applicable as a feature/config request. The reported v2026.5.4 logs are plausible, and source inspection shows current main no longer starts every configured channel account at once.

## Solution Assessment

Yes for the central startup-starvation problem: a shared startup cap is narrower than Telegram-only config knobs and protects all channel plugins. Configurable delay/concurrency can be revisited separately if evidence remains after a build containing the main fix.
