Full review comments:

- **[P1] Register curator through the CLI contract:** `extensions/skill-curator/src/cli.ts:98-106`
  - body: `api.registerCommand` only accepts a flat command definition with a `handler`; it does not support `subcommands`, `run`, `options`, `display`, or `ctx.options`. As written, the advertised `openclaw curator ...` subcommands will not register through the current plugin command contract.
  - confidence: 0.96
- **[P1] Use a valid runtime lifecycle registration:** `extensions/skill-curator/src/index.ts:19-26`
  - body: `registerRuntimeLifecycle` requires an `id` and only supports cleanup metadata. `activate` and `deactivate` are not lifecycle callbacks in the current SDK, so this registration is outside the host contract and should be replaced or removed.
  - confidence: 0.94
- **[P1] Constrain skill names before filesystem moves:** `extensions/skill-curator/src/run.ts:86-100`
  - body: `moveSkillToArchive` and restore paths join the sidecar or CLI skill key directly into source and destination paths before `fs.rename`. Validate normalized skill names and guard resolved paths inside the workspace `skills/` root before moving directories.
  - confidence: 0.9
- **[P2] Keep skill-workshop off curator private internals:** `extensions/skill-workshop/src/tool.ts:24-30`
  - body: This lazy import reaches from `skill-workshop` into another plugin's private `src/telemetry.js`, breaking the bundled plugin boundary. Expose a public API seam or keep curator-owned telemetry integration inside the curator plugin.
  - confidence: 0.87
