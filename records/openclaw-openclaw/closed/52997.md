## Summary

Keep open: current main and the latest release still use Telegram topic/thread isolation, and the requested shared-session mode would add a new opt-in config surface with privacy, sandbox, metadata, and reply-routing semantics that need maintainer product/security review.

## What This Changes

The issue requests an opt-in Telegram shared-session mode for forum topics and DM threads while preserving topic/thread metadata for correct replies.

## Best Possible Solution

Define an explicit opt-in shared-session contract that preserves default isolation, carries Telegram topic/thread metadata through the shared context, and routes replies back to the originating topic/thread with schema, docs, and focused routing tests.

## Reproduction Assessment

Yes at source level, but not as a bug reproduction. Current source, tests, docs, and v2026.5.20 show forum topics remain `:topic:` sessions, configured DM topics can become `:thread:` sessions, and no Telegram shared-session config surface exists.

## Solution Assessment

Unclear as a final implementation. The requested `sessionMode` direction is plausible, but the maintainable solution should align with the broader shared-session contract and define privacy, sandbox, metadata, and reply-routing behavior before implementation.
