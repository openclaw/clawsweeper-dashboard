## Label Justifications

- `P2`: The report concerns a real exec approval policy mismatch, but current main and the latest release already resolve the observed normal-exec security override.
- `impact:security`: The issue is about shell execution approval policy and whether model-provided tool arguments can alter the operator security boundary.

## Summary

Close as implemented: current main and the shipped v2026.5.18 release no longer let model-supplied `security=allowlist` tighten normal exec calls, which resolves the reported full/off heredoc approval storm. The remaining `ask` field is still documented as a per-call prompt mode rather than part of the ignored `security` contract.

## What This Changes

The issue asks OpenClaw to ignore model-supplied exec `security` and `ask` arguments for normal exec calls so configured exec policy remains authoritative.

## Best Possible Solution

Keep the shipped configured-only handling for normal exec `security`; if maintainers want `ask` removed as a per-call prompt control too, track that as a separate policy change rather than reopening this fixed full/off approval-storm report.

## Reproduction Assessment

No: the reported full/off plus model `security=allowlist` path no longer reproduces from current source because normal exec does not read `params.security`. The heredoc approval warning still requires effective allowlist security, which the reported model argument can no longer force.

## Solution Assessment

Yes for the reported observable bug: current main keeps normal exec security under configured policy and shipped that behavior in v2026.5.18. Ignoring `ask` as well would be a separate policy/API decision because the docs and schema still expose it as prompt behavior.
