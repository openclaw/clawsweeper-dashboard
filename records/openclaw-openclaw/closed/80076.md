## Summary

Close as implemented on current main: the normalized dreaming cron sentinel is now caught before model execution in both CLI and embedded runners, and memory-core's cron-aware handler runs the sweep/promotion path. No release tag in this checkout contains the proof SHA, so this appears fixed on main after v2026.5.7 rather than shipped in the latest tagged release.

## What This Changes

The issue asks for the managed memory-core dreaming cron sentinel to execute short-term promotion writes instead of acking a normalized isolated `agentTurn` as a no-op.

## Best Possible Solution

Keep the current main hook-gate implementation and ship it in the next release so migrated dreaming cron jobs no longer need the legacy heartbeat workaround.

## Reproduction Assessment

Yes for the reported v2026.5.7 regression path from the issue logs and source contract; no failing current-main path remains by source inspection because cron turns now invoke `before_agent_reply` and the memory-core handler processes the sentinel.

## Solution Assessment

Yes. Routing cron `agentTurn` prompts through the existing `before_agent_reply` hook before model execution is the narrow owner-boundary fix, and current main applies it to both CLI and embedded runners while keeping the promotion logic in memory-core.
