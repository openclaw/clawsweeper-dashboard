## Summary

Keep open: current `main` still schedules the shared SQLite WAL maintenance timer with a default `TRUNCATE` checkpoint through synchronous `DatabaseSync.exec()`, while the linked fixing PR is open and not merged. The issue is source-reproducible from the shared WAL helper and its SQLite-backed store callers, so cleanup should wait for review/landing or replacement of the linked implementation.

## What This Changes

The issue asks OpenClaw to stop running periodic synchronous SQLite `wal_checkpoint(TRUNCATE)` maintenance on the main event loop for default SQLite-backed gateway stores.

## Best Possible Solution

Review and land or replace the linked passive-periodic-checkpoint PR, preserving shutdown/manual `TRUNCATE` behavior and adding regression coverage for default timer behavior and explicit checkpoint overrides.

## Reproduction Assessment

Yes for source reproduction: current `main` still has a 30-minute timer that calls synchronous `DatabaseSync.exec()` with `PRAGMA wal_checkpoint(TRUNCATE)`. I did not reproduce an 8-12 second live stall in a running gateway during this read-only review.

## Solution Assessment

Unclear until the linked PR is reviewed: switching the periodic path to `PASSIVE` while keeping `TRUNCATE` for close/manual checkpoints matches the narrowest likely fix, but maintainers should confirm it preserves the WAL growth protection introduced for the earlier report.
