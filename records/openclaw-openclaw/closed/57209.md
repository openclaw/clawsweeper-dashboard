## Summary

Current `main` contains a browser-plugin startup fix that auto-enables the bundled browser plugin from browser config/tool references, registers `browser.request` on gateway startup, and covers that path with integration tests. The changelog shows this shipped in `2026.3.31`, so the `2026.3.28` regression reported in #57209 appears fixed on current main.

## Evidence

- **Gateway startup now auto-enables plugin-backed browser config before loading plugins:** `prepareGatewayPluginLoad()` applies `applyPluginAutoEnable()` to the activation config, then loads gateway plugins from that resolved config.
  - file: [src/gateway/server-plugin-bootstrap.ts:67](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/src/gateway/server-plugin-bootstrap.ts#L67)

## Close Comment

Current `main` appears to have this fixed, so this Codex review is closing the report.

- Gateway startup now runs plugin auto-enable before loading plugins, which covers browser-config and browser-tool references: `src/gateway/server-plugin-bootstrap.ts:67`.
- The bundled browser setup plugin explicitly auto-enables from `browser` config and browser tool policy: `extensions/browser/setup-api.ts:34`.
- The browser plugin still registers the `openclaw browser` CLI, the `browser.request` gateway method, and the browser control service together: `extensions/browser/plugin-registration.ts:28`.
- There is now a targeted integration test that asserts gateway startup includes both `browser.request` and the browser control service: `src/gateway/server-plugin-bootstrap.browser-plugin.integration.test.ts:42`.
- The fix is recorded in the shipped `2026.3.31` changelog: `CHANGELOG.md:1803`.

If this still reproduces on a current release after `2026.3.31`, please open a fresh issue with the exact version, `plugins.allow` / `plugins.entries.browser` config, and `openclaw gateway status --deep --json` output.
