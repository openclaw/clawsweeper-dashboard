## Summary

Keep open. Current main still lacks the requested mechanically read-only WhatsApp DM path: `dmPolicy` does not accept `observe`, blocked DMs return before any observation or persistence branch, and admitted WhatsApp DMs still enter the shared inbound turn pipeline. Existing `sendPolicy: "deny"` and generic `observeOnly` behavior are adjacent but do not meet the issue's no-session/no-turn requirement.

## What This Changes

Add a WhatsApp `dmPolicy: "observe"` mode that logs inbound direct messages for later read-only access without creating sessions, firing agent turns, or exposing reply paths.

## Best Possible Solution

Define the privacy/product contract first, then implement a WhatsApp-owned pre-routing observe policy: accept the config value, append bounded observation records before session resolution, skip hooks/route/session/agent dispatch for observed DMs, and expose a guarded read tool with account, sender, time-range, retention, deletion, and redaction controls.

## Reproduction Assessment

Yes for source-level verification, not as a live WhatsApp repro. Current main rejects `dmPolicy: "observe"`, drops denied DMs before persistence, and sends admitted DMs through `runInboundReplyTurn`.

## Solution Assessment

Unclear until maintainers approve the privacy model. The narrow maintainable direction is a WhatsApp pre-routing observe branch; existing `sendPolicy` or generic `observeOnly` behavior is not the best solution because it does not satisfy the no-session/no-turn requirement.
