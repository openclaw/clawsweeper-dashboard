## Summary

Current `main` routes gateway-host exec through approval resolution and shell allowlist checks, and unmatched commands are denied or sent through approval flow. The issue's reported behavior matches a config that only defines allowlist entries without also putting the effective policy into `security: "allowlist"` / prompting mode, which current docs and code treat as `full` + `off` by default.

## Close Comment

Closing after a Codex review of current `main`.

I could not substantiate a local-gateway allowlist bypass in the current codepath. Gateway-host exec still goes through `src/agents/bash-tools.exec.ts:1639`, which calls `processGatewayAllowlist(...)`, and that path resolves host approvals, evaluates the command against the allowlist, and denies unmatched commands (`src/agents/bash-tools.exec-host-gateway.ts:98-183`, `:296-319`). There is also a direct regression test for the deny path in `src/agents/bash-tools.exec-host-gateway.test.ts:280-295`.

The behavior described in this report matches a configuration where allowlist entries exist but the effective policy is still `security="full"` / `ask="off"`. Current approval resolution defaults to those values when `security` / `ask` are omitted (`src/infra/exec-approvals.ts:169-170`, `:701-707`), and the docs call out that omitted approvals fields inherit the requested `tools.exec.*` policy (`docs/tools/exec-approvals.md:16-22`).

If this still reproduces on a current build with an explicit effective allowlist policy, please open a new issue with the output of `openclaw approvals get --gateway`, the relevant `tools.exec.*` config, and the exact `exec-approvals.json` contents used for the repro.
