## Summary

Keep open: current main still has render-oriented Canvas embeds, A2UI/native action plumbing, live reload, a read-only diff viewer plugin, and an admin-scoped workspace file editor, but it does not provide a bidirectional Canvas Markdown editor with browser edit events, workspace writes, diff capture, refresh semantics, or next-turn agent context injection. The requested UX is plausible, but it needs maintainer product and security design before implementation because it would add a browser-origin path that can write workspace files and influence future agent turns.

## What This Changes

Request a live two-way Markdown editor embedded in Canvas so users can edit a workspace file in the browser while the agent sees diffs and refreshes the view.

## Best Possible Solution

Define a secure collaborative-editing contract first, covering authenticated edit transport, explicit workspace/file boundaries, stale-write handling, diff capture, refresh behavior, and next-turn context injection, then decide whether the editor is bundled UI or plugin-backed.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken existing behavior report. Source inspection on current main confirms the requested Canvas edit/write/diff/next-turn context path is absent while existing paths are render-oriented or admin-editor-only.

## Solution Assessment

Unclear as an implementation path: the requested UX is useful, but the maintainable route needs product and security design before code. A focused contract for transport, authorization, file scope, diffs, refresh, and context injection is safer than directly wiring generic Canvas actions to workspace writes.
