## What This Changes

The issue asks Android to split TCP connect and TLS handshake timeout handling for gateway fingerprint probes and show a distinct TLS timeout status for stalled secure endpoints.

## Best Possible Solution

Review and land or replace the linked Android fix so the probe has separate connect and handshake budgets, preserves mandatory TLS/fingerprint trust, and closes this issue only after the merged implementation is on main.

## Solution Assessment

Yes, the linked approach looks like the right shape for this issue: keep the existing remote TLS requirement, split connect and handshake timeout budgets, add a distinct TLS timeout failure, and cover the status guidance in tests. The final verdict belongs on the open PR review rather than this issue cleanup pass.
