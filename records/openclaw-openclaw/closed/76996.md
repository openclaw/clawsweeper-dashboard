## Summary

Keep open for maintainer judgment: current main intentionally makes Discord guild/channel final replies private unless the agent uses the message tool, so the reported logs match the documented tool-only reply policy rather than a simple already-fixed bug. Reviewed: https://github.com/openclaw/openclaw/issues/76996

## What This Changes

The issue asks OpenClaw to make Discord channel replies visible when the agent generates a final answer but does not call `message(action=send)`.

## Best Possible Solution

Keep the documented tool-only default unless maintainers choose a guarded fallback; users who want legacy visible channel replies can set `messages.groupChat.visibleReplies: "automatic"`.

## Reproduction Assessment

Yes for the invisible-reply path: current source and tests show Discord channel turns default to `message_tool_only`, so a final answer with `didSendViaMessagingTool: false` is intentionally not posted. I did not live-run the model-specific failure to call the message tool.

## Solution Assessment

Unclear: automatically posting final assistant text when the model skips the message tool would change the documented group-channel reply policy. A maintainer should decide between support/config guidance, stronger diagnostics/prompts, or a guarded fallback.
