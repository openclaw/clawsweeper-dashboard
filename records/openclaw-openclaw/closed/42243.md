## Label Justifications

- `P1`: The bug affects a real scheduled channel delivery workflow and can expose raw provider error JSON to external recipients.
- `impact:security`: Raw provider and operational error payloads can disclose internal details in Discord, Telegram, or other configured channels.

## Summary

Keep open: current main and the latest release still let fatal structured cron error payloads flow into normal announce delivery, and the open closing PR has not merged yet.

## What This Changes

Cron announce delivery should suppress failed-run raw provider error payloads while keeping run diagnostics and failure state available internally.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/83724 or an equivalent narrow fix that suppresses fatal structured error completion announces while preserving cron run records and configured failure notifications.

## Reproduction Assessment

Yes, at source level. Current main classifies structured cron isError payloads as fatal and still passes the resulting deliveryPayloads into dispatchCronDelivery for normal announce delivery.

## Solution Assessment

Yes. The maintainable fix is to suppress fatal structured cron completion payloads before normal announce delivery while preserving internal run diagnostics and controlled failure notifications.
