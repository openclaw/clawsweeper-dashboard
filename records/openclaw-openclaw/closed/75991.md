## Summary

Keep open. Current main still has a source-backed message-loss path where a routed non-empty final reply can be treated as successfully delivered even when outbound delivery returns zero results, which can skip the delivery-mirror append and Telegram fallback.

## What This Changes

Ensure CLI backend assistant responses with a Telegram deliveryContext are delivered, mirrored, or retried instead of silently remaining only in the transcript.

## Best Possible Solution

Make routed non-silent final replies require a provider delivery result or an explicit durable retry/claim outcome before the turn is counted as delivered, with regression coverage for mirror and Telegram fallback behavior.

Yes for a focused source-level path: make deliverOutboundPayloads return [] for a non-empty routed final reply, such as the claimed-by-other-owner path. The full intermittent macOS lock cascade remains unverified in this read-only review.

## Solution Assessment

Yes. Fixing the shared route-reply/outbound delivery contract is the narrow maintainable path; a Telegram-only watchdog or stale-lock cleanup does not close the silent success gap.
