## Summary

Close: current main and shipped v2026.6.1+ already cover the reported `XPC_SERVICE_NAME="0"` LaunchAgent environment and route gateway restart into detached launchd handoff before stale-port cleanup.

## What This Changes

The issue asks OpenClaw to avoid treating `XPC_SERVICE_NAME="0"` as a decisive non-matching launchd label during macOS gateway restart from a LaunchAgent-managed process tree.

## Best Possible Solution

Keep the shipped marker-backed launchd identity check and leave broader dual-plist or stale-service-marker launchd work to separately tracked issues such as https://github.com/openclaw/openclaw/issues/89791.

## Reproduction Assessment

Yes for the reported environment shape: the regression test drives `restartLaunchAgent` with `XPC_SERVICE_NAME="0"`, OpenClaw gateway service markers, and the gateway launchd label. It now proves the fixed path, so there is no failing current-main reproduction left for this exact issue.

## Solution Assessment

Yes. The shipped implementation fixes the decision at the launchd service-identity boundary and uses OpenClaw-owned service metadata rather than treating a nonmatching inherited XPC value as final.
