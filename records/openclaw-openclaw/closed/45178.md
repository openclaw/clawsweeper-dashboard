## Summary

Current `main` already hardens the macOS LaunchAgent restart path and folds the old `doctor --repair` bootstrap recovery into normal gateway start/restart flows, matching the reported failure mode.

## Evidence

- **Shipped release note:** The `2026.4.5` changelog explicitly says macOS gateway restarts now let launchd `KeepAlive` own in-process restarts, re-bootstrap the LaunchAgent when `kickstart -k` leaves it unloaded, and recover installed-but-unloaded LaunchAgents during `openclaw gateway start` and `restart`.

## Close Comment

Closing as implemented on current `main` after a Codex review. The macOS LaunchAgent path now has the missing recovery behavior in normal operation: `repairLaunchAgentBootstrap(...)` performs the explicit `launchctl enable -> bootstrap -> kickstart -k` repair flow (`src/daemon/launchd.ts:318`), `restartLaunchAgent(...)` re-bootstraps when `kickstart -k` finds the service unloaded (`src/daemon/launchd.ts:635`), the detached launchd handoff retries through `bootstrap` instead of silently swallowing failures (`src/daemon/launchd-restart-handoff.ts:93`), and CLI `openclaw gateway start` / `restart` now auto-recover an installed-but-not-loaded LaunchAgent (`src/cli/daemon-cli/launchd-recovery.ts:1`, `src/cli/daemon-cli/lifecycle.ts:152`). This was also shipped in the `2026.4.5` release notes (`CHANGELOG.md:1263`). If you are still seeing the separate "loaded but stale env" case mentioned later in the thread, that looks like a narrower follow-up and should be tracked separately with fresh reproduction details.
