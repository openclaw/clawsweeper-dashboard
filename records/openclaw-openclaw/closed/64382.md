## Summary

Current `main` already implements event-driven abort handling for exec tool calls, and webchat stop requests are wired into the active run's `AbortController`. The reported missing poll-loop check does not match the current implementation, and the same exec abort wiring is already present in released `v2026.4.23`.

## Evidence

- **Exec tool listens for abort and kills foreground runs:** `createExecTool()` installs an `abort` event listener on the incoming signal and calls `run.kill()` immediately for foreground exec sessions. This is event-driven abort handling, not a poll loop that waits for process completion.
  - file: [src/agents/bash-tools.exec.ts:1716](https://github.com/openclaw/openclaw/blob/11804a484ded89374afa55020813cfa265aa143a/src/agents/bash-tools.exec.ts#L1716)

  - sha: [11804a484ded](https://github.com/openclaw/openclaw/commit/11804a484ded89374afa55020813cfa265aa143a)
- **Exec runtime kill maps to supervisor manual cancel:** The exec runtime waits on the managed process exit promise, and its `kill()` implementation delegates to `managedRun.cancel("manual-cancel")`, which is the immediate subprocess termination path used by exec.
  - file: [src/agents/bash-tools.exec-runtime.ts:770](https://github.com/openclaw/openclaw/blob/11804a484ded89374afa55020813cfa265aa143a/src/agents/bash-tools.exec-runtime.ts#L770)
