## Summary

Keep open. Current `main` does not implement this PR's write-behind session store or default filesystem-lock bypass, and the proposed behavior conflicts with current documented expectations that session writes remain process-aware and file-based. The PR also has unresolved review concerns about stale-read/data-loss windows and dropped durability on failed flushes, so there is not strong evidence for a conservative close.

## Risks / Open Questions

- The underlying contention problem described by the PR may still be real for heavy concurrent session-store writers, so closing it would risk discarding a still-useful line of work.
- The exact PR implementation appears unsafe as submitted because review comments identified a plausible stale-read/data-loss window and durability regression on failed async flushes.
- Current docs and tests assume file-based cross-process locking remains part of the contract, so closing as 'implemented' or 'obsolete' would be misleading.
