## Summary

Keep open: current main still has the source-level precedence bug where stale runtime web-search metadata can outrank an explicit configured search provider in the agent tool path. The PR is useful and proof-backed, but it is merge-conflicting and must be rebased around later provider-owner/startup changes before maintainer review can land it.

## What This Changes

The PR makes explicit `tools.web.search.provider` values authoritative for agent `web_search`, adds fallback provider discovery for explicit runtime misses, and adds focused regression tests plus a changelog entry.

## Best Possible Solution

Land a rebased version of this explicit-provider precedence fix with the regression coverage, while preserving current main's global provider-owner resolution and configured-provider startup behavior.

## Reproduction Assessment

Yes, source-reproducible. Configure `tools.web.search.provider` for a custom provider while `runtimeWebSearch.selectedProvider` points at a stale provider; current main does not pass `providerId` from the agent wrapper and orders runtime metadata before explicit config.

## Solution Assessment

Yes. Passing the configured provider id at the agent boundary and giving config precedence in the runtime context is the narrowest maintainable fix; the fallback discovery piece needs a careful rebase around the later startup/provider-owner changes.
