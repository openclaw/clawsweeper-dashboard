## Summary

Current main implements the Control UI/WebChat side of this report: heartbeat and async exec wakeups no longer expose the raw internal prompt text through chat history, the stored heartbeat prompt is reduced to a benign transcript sentinel, and visible history/SSE projection drops heartbeat prompts plus HEARTBEAT_OK acknowledgements. Related broader heartbeat/session issues remain tracked separately, but the observable Control UI leak described here is fixed well enough to close this issue.

## What This Changes

The issue requests that Control UI/webchat consume async completion and heartbeat-only system events internally instead of rendering raw orchestration text or HEARTBEAT_OK rows in normal chat.

## Best Possible Solution

Keep the shipped/current split: model-facing heartbeat and exec-event prompts may still be used internally, but transcript/history surfaces should store only benign provenance or sentinels, hide no-op heartbeat rows, and continue routing broader Telegram/subagent wakeup problems through their narrower open issues.

## Reproduction Assessment

Yes for the code path: the targeted tests construct heartbeat/exec-event turns and verify transcript prompt substitution, history filtering, and no-op exec suppression. There is no minimal live Control UI repro in the report, but current source and tests cover the leak mechanism described.

## Solution Assessment

Yes. The current approach is narrower and more maintainable than only UI text matching: it keeps internal prompt text out of the transcript body, filters legacy/current heartbeat rows at the gateway projection, and leaves Control UI filtering as defense in depth.
