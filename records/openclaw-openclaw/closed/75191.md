## Summary

This PR can close because current main already covers the observable Matrix build/runtime problem: staged bundled runtime dependencies are materialized from the installed dependency tree, native files are copied rather than pruned, and Matrix has a package-local bootstrap fallback if the native crypto binding is still missing. The proposed postbuild copier duplicates shipped behavior instead of filling a missing core gap.

## What This Changes

The PR adds a runtime-postbuild phase that copies top-level `.node` files for `@matrix-org/matrix-sdk-crypto-nodejs` from root `node_modules` into `dist/extensions/matrix` after staged runtime dependency installation.

## Best Possible Solution

Keep the manifest-driven staged runtime dependency materialization and Matrix runtime bootstrap path. If a future build still misses the native binding, add a focused regression test to the generic staging code rather than a Matrix-specific postbuild copier.

## Reproduction Assessment

Yes. The high-confidence check is static: Matrix opts into staged runtime dependencies, current staging copies the installed package tree, and the prune rules preserve native `.node` files; I did not run `pnpm build` because this was a read-only review.

## Solution Assessment

No. The PR's approach is not the best maintainable fix because the generic staging path already handles native artifacts and the Matrix runtime has a fallback bootstrap; a new hard-coded Matrix copy phase would be redundant.
