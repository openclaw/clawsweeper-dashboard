## Summary

Keep open: current main and the latest release still synthesize a user-visible exec warning after an assistant reply, and two open closing PRs are already candidate fixes. This issue should stay open until one of those PRs lands or maintainers choose a different fix path.

## What This Changes

Suppress Feishu-visible exec stderr warning replies so the agent's actual response remains visible and delivered normally.

## Best Possible Solution

Land a narrow shared payload-warning fix with regression coverage that suppresses non-timeout, non-middleware exec warnings when an assistant reply already exists while preserving warnings for no-reply, timeout, heartbeat/cron, and verbose-full cases.

## Reproduction Assessment

Yes, source-reproducible: current main still builds a second visible `Exec failed` payload after assistant text for the matching tool-error path. I did not run the live Feishu/Windows scenario in this read-only review.

## Solution Assessment

Yes, the best fix location appears to be the shared embedded-run payload warning policy rather than Feishu-specific rendering. The alternate process-registry approach should be reviewed carefully because `aggregated` output is part of exec result semantics beyond channel display.
