## Label Justifications

## Summary

Keep open: current main and v2026.5.22 still store MiniMax token `expired_in` directly while shared auth treats `expires` as an absolute millisecond timestamp. The branch is a narrow provider-local normalization with focused tests and sufficient terminal proof; I found no blocking correctness finding.

## What This Changes

The branch converts MiniMax OAuth token expiry values to absolute millisecond timestamps, adds unit coverage, updates raw-fetch allowlist line numbers, and records the fix in the changelog.

## Best Possible Solution

Land the provider-local expiry normalization if maintainers accept the OAuth-only scope, and keep broader MiniMax OAuth2 refresh and API-key latency work in their separate threads.

## Reproduction Assessment

Yes. Current main stores MiniMax `expired_in` directly as `expires`, and shared auth subtracts `Date.now()` from that field, so a relative-second token lifetime becomes expired immediately.

## Solution Assessment

Yes. Normalizing at the MiniMax provider boundary is the narrowest maintainable fix for this OAuth path; broader refresh migration and API-key latency should stay separate.
