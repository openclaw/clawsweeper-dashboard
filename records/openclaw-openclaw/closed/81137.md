## Summary

Current main already implements the requested Discord text-channel auto-thread path: when a guild channel is configured with `autoThread`, OpenClaw creates a message-based thread, routes the reply to that thread, and records thread/session metadata. The proof commit is present on `main` only; no tag contains it yet.

## What This Changes

The issue requests Discord text-channel mentions to create a new thread and route the bot reply into that thread.

## Best Possible Solution

Keep the Discord-plugin-owned auto-thread implementation and its channel-level configuration rather than moving Discord-specific thread policy into core.

## Reproduction Assessment

No, not as a current-main failure. I did not run a live Discord server, and current main already has source and focused tests for creating and routing message-based Discord auto-threads.

## Solution Assessment

Yes. The current implementation is the narrow maintainable direction because it stays inside the Discord plugin, uses Discord's message-based thread route, and falls back to parent-channel delivery when auto-threading is disabled or unavailable.
