## Summary

Keep open: current main still has a source-level failure path for OpenCode-like ACP adapters that advertise `session/set_config_option` but reject the optional `timeout` key, and an open draft PR with closing syntax is already targeting this issue but has not merged.

## What This Changes

The issue reports that ACP OpenCode child sessions fail immediately when OpenClaw sends `timeoutSeconds` as a `session/set_config_option` timeout hint and the adapter rejects it.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/81603 or an equivalent narrow fix that treats only timeout config as a best-effort adapter hint while preserving strict failures for required runtime config and proving the OpenCode path.

## Reproduction Assessment

Yes at source level: current main can build a timeout config pair when advertised keys are empty and then fail before `runTurn` if `runtime.setConfigOption` rejects `timeout`. I did not run a live OpenCode adapter in this read-only review.

## Solution Assessment

Yes, the best fix direction is narrow ACP runtime-control tolerance for optional timeout keys, backed by OpenClaw's existing turn timeout and strict handling for required config such as `model`. The linked draft PR follows that direction but still needs normal review and real OpenCode proof before landing.
