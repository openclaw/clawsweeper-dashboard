## Summary

Keep open: the performance direction is useful, but the current patch is not merge-ready because the new skipPiDiscovery shortcut is too broad, still runs runtime normalization for the exact configured-row case it is meant to avoid, and lacks the required changelog entry.

## What This Changes

The PR adds a skipPiDiscovery configured-model fast path, defers PDF auto model resolution with media-tool policy coverage, and loosens two script declaration signal types.

## Best Possible Solution

Land a narrower fast path that returns only exact configured models[] rows before runtime normalization, leaves provider-level dynamic configs on the existing dynamic path, keeps the PDF deferral behavior, and records the user-facing perf change in the changelog.

## Reproduction Assessment

Yes. Static source review gives a high-confidence path: with skipPiDiscovery and a configured provider lacking an exact models[] row, the PR can return resolveConfiguredFallbackModel before provider dynamic hooks run; exact configured rows still run normalizeResolvedModel before the shortcut.

## Solution Assessment

No. The safer solution is to detect exact configured rows before resolveExplicitModelWithRegistry invokes runtime normalization, while keeping provider-only configs on prepareProviderDynamicModel/runProviderDynamicModel and adding the required changelog entry.
