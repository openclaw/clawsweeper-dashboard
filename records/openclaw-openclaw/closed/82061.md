## What This Changes

The issue requests a per-LLM-provider config opt-in that allows trusted private/internal-network provider base URLs without relaxing the global SSRF guard.

## Best Possible Solution

Keep the nested `models.providers.<id>.request.allowPrivateNetwork` contract as the supported per-provider opt-in and leave flat provider-level/private-network aliases rejected for model providers.

## Reproduction Assessment

Not applicable: this is a config feature request rather than a broken current-main contract. Source inspection shows the requested opt-in is accepted and used on current main via `models.providers.<id>.request.allowPrivateNetwork`.

## Solution Assessment

Yes. The current-main solution is narrowly scoped to model-provider request transport, preserves the SSRF guard default, and avoids broader provider-level or global private-network bypasses.
