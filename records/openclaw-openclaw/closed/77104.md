## Summary

Keep open: the PR targets a valid current-main cron/session cleanup bug, but the patch still strips resumable fields from explicit persistent cron session targets whose custom id begins with `cron:`.

## What This Changes

The PR makes missing-transcript cron metadata rows non-resumable until the transcript appears, adds cron session-state regression tests, and records a changelog fix.

## Best Possible Solution

Land the cron cleanup fix after narrowing the fallback to default isolated cron metadata rows and adding a regression for explicit `session:cron:*` targets.

## Reproduction Assessment

Yes, by source inspection. Current main can persist a cron row with `sessionId` before its transcript exists, and cleanup later prunes missing-transcript rows with `sessionId`; the PR-introduced regression is also source-reproducible with an explicit `session:cron:*` target.

## Solution Assessment

No, not as currently written. The direction is narrow, but using `isCronSessionKey` on the normalized store key conflates default isolated cron metadata with valid persistent custom session ids; the guard should use the original cron target mode or another explicit isolated-run signal.
