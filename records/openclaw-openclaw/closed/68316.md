## Summary

Close as implemented. Current main already preserves unsent media final payloads after successful block streaming, while also stripping already-streamed text and deduping media that the block pipeline already sent; the v2026.4.29 release tag contains focused regression coverage for that behavior. The PR branch is now superseded by the stronger shipped implementation.

## What This Changes

This PR changes `buildReplyPayloads` and its tests to retain media and error final payloads after successful block streaming while continuing to suppress text-only final payloads.

## Best Possible Solution

Keep the shipped current-main implementation and close this PR rather than merging the older branch; any future changes should preserve the existing text-only suppression, media preservation, and sent-media dedupe tests together.

## Reproduction Assessment

Yes. The focused reproduction is `buildReplyPayloads` with block streaming enabled, `didStream()` true, a text-plus-media final payload, and a pipeline that already sent the text-only content; current tests assert the media remains while the duplicate text is removed.

## Solution Assessment

No for the PR branch; yes for current main. The shipped implementation is the narrower maintainable solution because it preserves unsent media, keeps errors, suppresses text-only finals, strips already-streamed caption text, and dedupes media URLs already sent by the block pipeline.
