## Summary

## What This Changes

The issue asks OpenClaw to prevent Lossless/LCM summary requests for canonical `openai/gpt-5.4-mini` from being denied when older plugin config only allowlists `openai-codex/gpt-5.4-mini`.

## Best Possible Solution

Keep OpenClaw's host allowlist boundary fail-closed and rely on Lossless's plugin-owned defaults plus `openclaw doctor --fix` for Lossless config repair.

## Reproduction Assessment

Yes by source inspection: if a plugin requests `openai/gpt-5.4-mini` while its host allowlist only contains `openai-codex/gpt-5.4-mini`, OpenClaw will deny the override. I did not run a live Lossless maintenance path because the ownership decision is clear from current source and upstream plugin history.

## Solution Assessment

Yes: the best fix is plugin-owned default and doctor repair, not global host aliasing across a fail-closed model allowlist boundary. Current OpenClaw already has the generic plugin doctor contract seam needed for that repair.
