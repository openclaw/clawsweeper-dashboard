## Label Justifications

- `P2`: The issue describes a real persisted pairing-state side effect and race amplifier, but the blast radius is limited to CLI log reads and Gateway pairing metadata.
- `impact:data-loss`: The report centers on unnecessary rewrites of the persisted `devices/paired.json` state file, which is involved in related read/write race reports.
- `impact:auth-provider`: The behavior is tied to Gateway device identity, operator scopes, and paired-device authentication state.

## Summary

Keep open: current main still routes implicit local `openclaw logs --follow` through the normal CLI Gateway client identity path, which can participate in pairing and paired-device metadata writes; an open draft PR specifically references this issue but has not merged.

## What This Changes

The issue asks for `openclaw logs --follow` to read logs without registering as a paired device or rewriting `devices/paired.json`.

## Best Possible Solution

Review and land or reject https://github.com/openclaw/openclaw/pull/85380; the end state should make implicit local log reads non-mutating for `paired.json` while preserving intentional explicit remote Gateway auth behavior.

## Reproduction Assessment

Yes, at source level: current main routes `openclaw logs --follow` through CLI Gateway RPC with a device identity, and the handshake/pairing paths can persist or update paired-device state. I did not live-run the command because the reported reproduction intentionally rewrites local pairing state.

## Solution Assessment

Unclear. A passive backend identity for implicit local log reads is plausible, but maintainers still need to decide the logs/auth and follow-fallback boundary in the linked PR before this is the best final fix.
