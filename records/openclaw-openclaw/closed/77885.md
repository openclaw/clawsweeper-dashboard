## Summary

Keep open. Current main still leaves several long-running container entrypoints starting Node directly as PID 1; only the bundled Compose file and one Podman shell launcher currently opt into init handling, so the reported zombie-process risk is not resolved.

## What This Changes

Request adding an init wrapper such as tini or Docker `--init` for OpenClaw container runs so PID 1 reaps orphaned child processes.

## Best Possible Solution

Add a verified image-level init wrapper or equivalent runtime init setting for long-running OpenClaw containers, then align Docker, Fly, VM, GCP, Hetzner, and Podman Quadlet guidance with the existing Compose and Podman shell-launcher init coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No live high-confidence reproduction was established here, but the source path is clear: root-image and documented long-running container commands start Node directly as PID 1 while related launchers explicitly opt into init handling.

## Solution Assessment

Is this the best way to solve the issue? Yes in direction: fix the container launch boundary with an init wrapper or Docker/Podman init setting, rather than adding OpenClaw-specific process-reaping logic inside the Node app.
