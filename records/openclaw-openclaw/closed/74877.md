## Summary

Keep open. The branch targets a real silent-reply bug, but the current diff is still incomplete because it misses precomputed channel requested modes and can disagree with the runtime tool inventory.

## What This Changes

The PR adds `messageToolAvailable` delivery-mode plumbing and resolver tests for auto-reply dispatch and the channel reply SDK helper, plus one Persian Control UI locale correction.

## Best Possible Solution

Keep the fallback direction, but make the final dispatch decision able to override requested `message_tool_only` when `message` is truly unavailable, and derive availability from the same runtime tool inventory or shared policy helper used by the agent run. Preserve `sendPolicy: deny` and intentional suppression semantics, with targeted Discord/Slack requested-mode and runtime-policy tests.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a Discord or Slack group/channel source turn with default visible replies and a runtime policy path that removes `message`; current main suppresses automatic delivery, and this PR still misses the precomputed requested-mode path.

## Solution Assessment

No. Falling back to automatic delivery when the message tool is unavailable is the right product direction, but this implementation is not the best fix yet because it applies too late for requested modes and checks only part of the runtime policy stack.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Apply the fallback to requested channel modes:** `src/auto-reply/reply/source-reply-delivery-mode.ts:33-35`
  - body: The new fallback only runs after config-derived modes are computed, but `resolveSourceReplyDeliveryMode` still returns `params.requested` first. Discord and Slack pass precomputed `message_tool_only` through `replyOptions.sourceReplyDeliveryMode`, so those channel turns can remain silently suppressed when `message` is unavailable.
  - confidence: 0.94

  - body: This availability check only evaluates profile, provider-profile, global, and agent policies. The runtime tool pipeline also applies provider, agent-provider, group, sandbox, subagent, and runtime-disable constraints, so this can report `message` available while the actual run removes it.
  - confidence: 0.9
