## Summary

Keep open: the updated patch fixes the env-merge defect from the earlier review, but current update code also blocks package updates when the running Gateway PID is an ancestor, and this PR still spawns the updater as a Gateway child without exercising the managed-service handoff.

## What This Changes

The PR adds a sanitized base environment for startup auto-update child commands so service marker variables are omitted while preserving `OPENCLAW_AUTO_UPDATE`, and updates the startup auto-update test.

## Best Possible Solution

Make startup auto-update use the existing managed-service handoff pattern or an equivalent detached update path so package updates run outside the live Gateway process tree while preserving the safety guard for operator-run updates.

## Reproduction Assessment

Yes, source-backed: a Gateway-spawned updater remains in the Gateway process ancestry, and current `updateCommand` exits before package install when the managed Gateway runtime PID is an ancestor. I did not run a live launchd or systemd package self-update in this read-only review.

## Solution Assessment

No. Stripping service markers is a useful slice, but the best fix must also move the auto-update package command outside the live Gateway process tree or reuse the existing managed-service handoff pattern.
