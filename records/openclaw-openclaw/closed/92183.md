## Summary

Keep open: current main and v2026.6.9 still route targeted plugin-id dry-runs through the recorded exact npm spec, while docs/tests now define exact pins as sticky. The direct fix PR was closed unmerged, and the open related PR only covers bulk official-plugin sync, so this still needs maintainer product judgment.

## What This Changes

The issue asks `openclaw plugins update <id> --dry-run` to detect newer npm `latest` releases even when the tracked install record is an exact npm version.

## Best Possible Solution

Maintainers should decide the exact-pin contract first, then align CLI behavior, output wording, docs, and tests so intentional pins remain clear while stale recorded exact specs are discoverable.

## Reproduction Assessment

Yes, source-reproducible: current main sends recorded exact npm specs into dry-run probes, and live npm metadata still has newer defaults for the reported packages. I did not run a live `openclaw plugins update` command in this read-only review.

## Solution Assessment

No, not as originally stated: always comparing targeted exact pins to `latest` would conflict with the current documented pin contract. The safer solution is a maintainer-approved distinction between intentional exact pins, stale recorded exact specs, and explicit requests to move back to the registry default line.
