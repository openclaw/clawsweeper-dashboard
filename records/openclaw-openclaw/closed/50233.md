## Summary

Current `main` no longer matches this PR’s assumptions. The gateway plaintext-WS path is now enforced through `OPENCLAW_ALLOW_INSECURE_PRIVATE_WS=1` in both connection-details and client code, IPv6/private-network cases are already covered by current tests, bundled-plugin postinstall handling is implemented elsewhere and is non-fatal, and the remaining `trustedNetworks`/model-selection changes do not match current main behavior.

## Evidence

- **Gateway ws:// policy now uses the env-based break-glass path in both codepaths:** `buildGatewayConnectionDetailsWithResolvers()` rejects non-loopback plaintext `ws://` unless `OPENCLAW_ALLOW_INSECURE_PRIVATE_WS=1`, and `GatewayClient.start()` independently applies the same check. That removes the original end-to-end mismatch this PR was trying to patch via `trustedNetworks` forwarding.
