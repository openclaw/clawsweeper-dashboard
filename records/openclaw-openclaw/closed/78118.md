## Summary

Current main already prevents the reported duplicate Discord runtime: duplicate enabled accounts resolving to the same bot token are filtered before provider startup, surfaced as disabled in status, and covered by tests and docs.

## What This Changes

The issue asks OpenClaw to prevent duplicate enabled Discord accounts sharing one bot/application token from starting separate native-command runtimes that can leave slash commands unacknowledged.

## Best Possible Solution

Keep the current duplicate-token suppression in the Discord plugin and ship it in the next release; add application-id-only duplicate diagnostics only if a separate reproducible case appears.

## Reproduction Assessment

No for current main: the original live setup is clear, but current source filters duplicate-token Discord accounts before startup, so the second client path should not start. The source and tests give high confidence for the reported same-token case without needing a live Discord run.

## Solution Assessment

Yes. The current main fix is the narrow maintainable solution for the reported hazard because it preserves per-account native command settings while preventing two live clients from sharing one resolved bot token.
