## Summary

Keep open: current main still has a source-backed Telegram send path that can turn a presentation-only payload into an empty Telegram message body, and the linked fixing PR is open rather than merged.

## What This Changes

The issue reports that Telegram outbound delivery can reject presentation-only replies as empty when neither top-level text nor interactive fallback text survives into the send path.

## Best Possible Solution

Review and land the linked Telegram fallback PR, or replace it with an equivalent narrow fix that preserves presentation-derived text before Telegram’s non-empty send guard.

## Reproduction Assessment

Yes for a source-level reproduction: a Telegram payload reaching sendTelegramPayloadMessages with presentation but no text, interactive fallback, or media resolves to an empty string and reaches the Telegram non-empty guard. I did not run a live cron/Telegram end-to-end reproduction.

## Solution Assessment

Yes, the narrow solution is to preserve or re-render presentation fallback text in the Telegram outbound/reply path before the send guard. The open linked PR is the right review target, though it still needs maintainer validation and real Telegram proof before closure.
