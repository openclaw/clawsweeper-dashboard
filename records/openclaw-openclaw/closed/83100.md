## Summary

Current main already contains the MiMo replay fix for this exact `xiaomi-coding`/MiMo v2.5 failure mode: assistant replay messages now preserve or backfill `reasoning_content`, including custom Xiaomi-compatible proxy routes. The fix landed after v2026.5.12, so this report matches shipped-release behavior but is already addressed on main.

## What This Changes

The issue reports that `xiaomi-coding/mimo-v2.5` and `mimo-v2.5-pro` reject normal Telegram-backed OpenAI-compatible requests with `400 Param Incorrect` after the first few messages.

## Best Possible Solution

Keep the current main implementation and let the next release carry it to v2026.5.12 users; reopen or file a new issue only if the same provider/model still fails after a build containing the MiMo replay fix.

## Reproduction Assessment

Yes. The canonical repro is a Xiaomi-compatible custom provider using `mimo-v2.5` or `mimo-v2.5-pro` with thinking enabled, then a multi-turn/tool-call replay that returned `400 Param Incorrect`; the merged fix PR records before/after live Telegram proof for that path.

## Solution Assessment

Yes. Preserving/backfilling MiMo `reasoning_content` on assistant replay is the narrow protocol fix; disabling tools or relying on `strictMessageKeys` would only reduce capability or address a different payload-shape problem.
