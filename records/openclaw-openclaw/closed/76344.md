## Summary

Keep open. Current main still wires stateful `openclaw googlemeet` CLI session commands to a CLI-process-local `GoogleMeetRuntime`, while the #72441 fix routes only agent tool actions through gateway RPC, so CLI-created/tested sessions can disappear from the gateway-owned registry after the command exits.

## What This Changes

The issue asks for Google Meet realtime sessions started by CLI commands to remain visible and addressable through `status`, `speak`, and related follow-up commands after the starting CLI process exits.

## Best Possible Solution

Route the stateful Google Meet CLI session lifecycle commands through the existing `googlemeet.*` gateway methods while keeping OAuth/API/export helpers local unless they join or inspect live session state.

## Reproduction Assessment

Yes from source inspection, though not from a live Meet run in this read-only review. The CLI creates and queries a per-process `GoogleMeetRuntime`, and that runtime keeps sessions only in private in-memory maps.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the already-registered `googlemeet.*` gateway RPC methods for stateful CLI commands, matching the #72441 tool-action path without changing unrelated local OAuth or artifact commands.
