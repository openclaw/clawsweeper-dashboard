## Summary

Current main already has the WebChat/Control UI protections for both reported paths: stale `chat.history` refreshes preserve optimistic local messages instead of wiping visible state, active-run `session.message` reloads are deferred, and streamed `NO_REPLY` lead fragments are suppressed so `NO` is not rendered. The same NO_REPLY suppression test and implementation are present in the `v2026.4.26` release tree.

## Best Possible Solution

Close this as already implemented in the current source and shipped release tree. If the behavior still reproduces on a fresh 2026.4.26 or newer Control UI bundle, the best follow-up is a new focused bug with browser/device, exact Control UI asset version, chat event payloads around the `NO_REPLY` turn, and reconnect/history logs.
