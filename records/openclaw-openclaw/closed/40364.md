## Summary

Current main already handles the reported auth-profile case: model discovery now derives PI credentials directly from `auth-profiles.json`, and the OpenAI Codex provider catalog is emitted whenever an `openai-codex` profile exists. The same implementation is present in release `v2026.4.22`.

## Evidence

- **`models list` now loads discovery auth from the auth-profile store:** `loadModelRegistry()` builds the registry from `discoverAuthStorage(agentDir, { readOnly: true })`, not from a regenerated on-disk `models.json` auth bridge.
  - file: [src/commands/models/list.registry.ts:111](https://github.com/openclaw/openclaw/blob/171322644dece55402569c52b26370fd81769ca6/src/commands/models/list.registry.ts#L111)
  - command: `nl -ba src/commands/models/list.registry.ts | sed -n '100,140p'`
  - sha: [171322644dec](https://github.com/openclaw/openclaw/commit/171322644dece55402569c52b26370fd81769ca6)
- **PI discovery converts `openai-codex` auth profiles into runtime credentials:** `resolvePiCredentialsForDiscovery()` loads `ensureAuthProfileStore(...)`, converts store entries into PI credentials, and `discoverAuthStorage()` passes those credentials to the PI auth storage in memory.
  - file: [src/agents/pi-model-discovery.ts:287](https://github.com/openclaw/openclaw/blob/171322644dece55402569c52b26370fd81769ca6/src/agents/pi-model-discovery.ts#L287)

  - sha: [171322644dec](https://github.com/openclaw/openclaw/commit/171322644dece55402569c52b26370fd81769ca6)
- **OpenAI Codex provider catalog is injected when an auth profile exists:** The OpenAI Codex provider plugin returns its catalog when `listProfilesForProvider(authStore, "openai-codex")` is non-empty, which matches the issue's expected behavior for an `openai-codex` OAuth profile.
  - file: [extensions/openai/openai-codex-provider.ts:417](https://github.com/openclaw/openclaw/blob/171322644dece55402569c52b26370fd81769ca6/extensions/openai/openai-codex-provider.ts#L417)
