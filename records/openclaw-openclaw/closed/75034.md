## Summary

Keep this PR open for maintainer review because the underlying pain in #74876 is real, but this patch expands the agent-facing persistent config-write allowlist across a privacy-affecting room-reply setting. Current main intentionally treats the gateway tool allowlist as a model-to-operator trust boundary, while direct operator/admin `config.patch` remains available outside that agent tool guard.

## What This Changes

The PR adds `messages.visibleReplies` and `messages.groupChat.visibleReplies` to the agent gateway tool `config.apply`/`config.patch` mutation allowlist in `src/agents/tools/gateway-tool.ts`.

## Best Possible Solution

Keep visible reply modes operator-configurable, but do not add them to the agent gateway tool allowlist without an explicit maintainer-approved policy change. If maintainers accept that expansion, pair it with focused guard tests for both paths and docs that call out the privacy and room-visibility implications.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists on current main: an agent gateway `config.patch` that changes `messages.groupChat.visibleReplies` produces a changed leaf path outside `ALLOWED_GATEWAY_CONFIG_PATHS`, so the guard rejects it before forwarding to the gateway RPC.

## Solution Assessment

No. The patch is mechanically narrow, but it lets a prompted model persistently switch group/channel final replies from tool-only to automatic; the safer current path is direct operator/admin config writes or a documented maintainer-approved allowlist expansion.
