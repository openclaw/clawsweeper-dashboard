## Summary

Keep open: the runner-boundary fix is in the right area for tagged TTS, but as written it does not carry the inbound-audio signal required by the existing `messages.tts.auto = "inbound"` mode, leaving part of the linked bug unresolved.

## What This Changes

The PR applies TTS payload conversion before message-tool outbound sends across core, gateway, and plugin fallback paths, with focused regression tests and a changelog entry.

## Best Possible Solution

Keep the runner-level TTS normalization approach, but propagate inbound-audio context from the originating turn into message-tool sends and add regression coverage for inbound mode alongside tagged mode.

## Reproduction Assessment

Yes at source level: current main sends message-tool payloads to gateway/core delivery without the TTS pipeline, matching the linked report. I did not run a live Slack or Telegram TTS send in this read-only review.

## Solution Assessment

No as-is: applying TTS at the shared message-action runner boundary is the maintainable direction, but the patch must also preserve the existing inbound-mode contract by passing inbound-audio context.
