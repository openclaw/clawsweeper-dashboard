## Label Justifications

- `impact:security`: The report concerns Feishu direct-message authorization semantics and proposes broadening access without the explicit wildcard opt-in.
- `impact:message-loss`: The observed symptom is Feishu DMs being blocked by the access gate when the config lacks the required wildcard.

## Summary

This is a configuration/support mismatch rather than a Feishu runtime bug: current main and the shipped v2026.5.27 contract require an explicit `allowFrom: ["*"]` opt-in for public DMs, and Feishu setup/doctor paths add that wildcard instead of treating bare `dmPolicy: "open"` as unconditional access.

## What This Changes

The issue requests changing Feishu DM access so `channels.feishu.dmPolicy: "open"` allows all direct-message senders even when `allowFrom` is missing.

## Best Possible Solution

Keep the explicit wildcard contract for public DMs; users who want public Feishu DMs should set `channels.feishu.allowFrom: ["*"]` or run `openclaw doctor --fix` to repair the config.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `dmPolicy: "open"` without a wildcard DM allowlist reaches `dm_policy_not_allowlisted`, but that is the documented and validated contract rather than broken existing behavior.

## Solution Assessment

No. Making `senderGateForDirect` unconditional for `dmPolicy: "open"` would remove the explicit public-DM opt-in; the safer supported path is the documented wildcard config or doctor repair.
