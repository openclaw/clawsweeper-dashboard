- `impact:crash-loop`: The reported failure prevents the Gateway from starting when config references the blocked plugin.

## Summary

Keep open. Current main has improved the plugin-package diagnostic and doctor can update or reinstall some broken recorded plugin installs, but the Gateway invalid-config startup path still appends the generic `openclaw doctor --fix` recovery hint when the blocking issue is an unbuilt TypeScript plugin package, so the central misdirection is only partially fixed.

## What This Changes

The issue asks OpenClaw either to build unbuilt TypeScript plugin packages during `doctor --fix` or to stop directing users to doctor when compiled runtime output is missing.

## Best Possible Solution

Keep installed package loading strict, do not auto-build third-party plugin source, and make the startup/config-validation hint conditional so plugin-packaging failures point to update/reinstall/disable or publisher remediation instead of a generic doctor repair.

## Reproduction Assessment

Yes, at source level: an invalid startup snapshot with `includeDoctorHint` still appends the doctor hint, while plugin discovery tests prove installed source-only TypeScript plugins stay blocked. I did not run the full Gateway repro in this read-only pass.

## Solution Assessment

Yes, the best narrow fix is to correct the misleading startup guidance, not to run arbitrary third-party build toolchains from doctor. Doctor’s existing reinstall/update repair path should remain separate from plugin publisher packaging failures.
