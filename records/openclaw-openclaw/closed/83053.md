## Summary

Current main and shipped releases already have the recovery path the report says is missing: the immediate per-account crash loop still stops after ten retries, but the default gateway health monitor treats that stopped account as recoverable, resets the restart counter, and starts it again on its slower capped schedule.

## What This Changes

The issue requests changing gateway channel auto-restart so a channel account does not permanently stop after ten consecutive restart attempts.

## Best Possible Solution

Keep the existing two-tier design: the fast channel crash-loop stops after the cap, and the default health monitor performs slower capped recovery for gave-up accounts.

## Reproduction Assessment

No. The named fast-loop cap is present, but current main also contains a default health-monitor path and regression test that restart a gave-up account instead of leaving it permanently dead.

## Solution Assessment

Yes. The current implementation is the safer maintainable shape because it bounds immediate crash-loop churn while routing longer recovery through the existing health-monitor cooldown, hourly cap, and config surface.
