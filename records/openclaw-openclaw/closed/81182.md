## Label Justifications

- `P1`: The reported behavior can make an active Telegram/direct agent workflow appear hung for about 15 minutes during overflow recovery.
- `impact:crash-loop`: The issue is about a long availability stall in the active agent run loop while compaction waits for timeout.
- `impact:session-state`: The root surface is context/session transcript recovery and accumulated tool-result state.

## Summary

Keep open: current main still runs explicit overflow compaction before checking tool-result truncation, and https://github.com/openclaw/openclaw/pull/81190 is the open closing PR for this exact recovery-order bug.

## What This Changes

Request that tool-heavy overflow recovery try deterministic tool-result truncation before waiting for auto-compaction to time out.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/81190 if checks stay clean, then close this issue after the merge.

## Reproduction Assessment

Yes. Source inspection on current main shows compaction runs before tool-result truncation, and the issue includes live logs where a 912840ms compaction timeout was followed immediately by truncating 281 tool results.

## Solution Assessment

Yes for the linked fix direction. The narrow maintainable path is to reuse the existing tool-result pressure heuristic before explicit overflow compaction, while preserving compaction as the fallback when truncation is unavailable or ineffective.
