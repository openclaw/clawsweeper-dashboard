## Summary

Current `main` already contains targeted fixes for the Discord startup/readiness path and the bundled-runtime-deps doctor path described in #70057; the buggy readiness behavior is still visible in `v2026.4.23`, so this looks fixed on `main` but not yet shipped in a tagged release.

## Evidence

- **Current main waits for Discord gateway plugin registration before supervisor startup:** `createDiscordMonitorClient` now awaits `waitForDiscordGatewayPluginRegistration(gateway)` before constructing the gateway supervisor, preventing startup from proceeding before the gateway plugin is registered.
  - file: [extensions/discord/src/monitor/provider.startup.ts:189](https://github.com/openclaw/openclaw/blob/2011de69d3dc8f1a11ebc774ba4af23f54892461/extensions/discord/src/monitor/provider.startup.ts#L189)
  - command: `nl -ba extensions/discord/src/monitor/provider.startup.ts | sed -n '140,220p'`
  - sha: [2011de69d3dc](https://github.com/openclaw/openclaw/commit/2011de69d3dc8f1a11ebc774ba4af23f54892461)
- **Current main only reports READY when the Discord gateway is actually connected:** `waitForGatewayReady` now requires `params.gateway?.isConnected === true`; if READY does not arrive within 15s it marks `startup-not-ready`, restarts once, and errors if the retry still does not reach READY.
  - file: [extensions/discord/src/monitor/provider.lifecycle.ts:302](https://github.com/openclaw/openclaw/blob/2011de69d3dc8f1a11ebc774ba4af23f54892461/extensions/discord/src/monitor/provider.lifecycle.ts#L302)

- **Regression tests cover the false-ready and retry paths:** Tests assert that a missing gateway handle is not treated as ready, that startup retries once when READY never arrives, and that the retry fails loudly if READY still does not happen.
  - file: [extensions/discord/src/monitor/provider.lifecycle.test.ts:223](https://github.com/openclaw/openclaw/blob/2011de69d3dc8f1a11ebc774ba4af23f54892461/extensions/discord/src/monitor/provider.lifecycle.test.ts#L223)
