## Summary

Keep open: current main and v2026.6.6 still have the source-reproducible ACP bridge key-classification failure where bridge-created ACP-shaped sessions can be treated as stale ACP runtime sessions before normal model dispatch.

## What This Changes

The issue reports that `openclaw acp client` sessions fail on the first prompt with `ACP_SESSION_INIT_FAILED`, echoed input, and no assistant chunks.

## Best Possible Solution

Keep ACP bridge sessions out of control-plane ACP runtime stale-metadata classification unless persisted ACP metadata exists, while preserving fail-closed behavior for real spawned ACP runtime sessions.

## Reproduction Assessment

Yes by source inspection, though I did not run a live ACP client in this read-only review. The bridge creates an ACP-shaped fallback key, gateway dispatch resolves it into an ACP-shaped agent key, and agent dispatch throws on stale ACP metadata before normal model output can stream.

## Solution Assessment

Yes. The narrow maintainable fix is to separate bridge-owned ACP-shaped normal-model sessions from ACP runtime sessions that have persisted `SessionAcpMeta`, rather than adding control-plane metadata to bridge-created sessions.
