## Summary

Current main already addresses the reported Telegram dmPolicy drift. Doctor and configure now start from the authored source config instead of the materialized runtime/defaulted config, config writes merge caller changes back onto the source config without persisting schema/runtime defaults, and Telegram setup patches only explicit credential or allowlist changes. The same relevant implementation is present in v2026.4.24.

## Best Possible Solution

Close this issue as implemented. Keep the current source-config-based doctor/configure/write path, and handle any future Telegram access-policy drift with a focused regression test around an explicit channels.telegram.dmPolicy="allowlist" plus allowFrom if a new reproduction appears.
