## Summary

Keep open: this looks like a real packaged-upgrade dashboard bug rather than stale support. Current main still lazy-loads the Control UI module from the Gateway HTTP request path, and the reported missing hashed `dist/ui-*.js` chunk matches the repository’s own stale post-swap import pattern.

## What This Changes

The issue asks for the Homebrew-installed dashboard to load instead of returning a 500 `MODULE_NOT_FOUND` after reinstalling OpenClaw 2026.5.4.

## Best Possible Solution

Make the Control UI HTTP handler a stable packaged runtime boundary, or otherwise preload/restart-gate it during managed package swaps, so an already-running Gateway cannot be stranded on a deleted hashed `dist/ui-*.js` chunk.

## Reproduction Assessment

Yes, from source and the reporter’s log: a running Gateway that reaches the lazy Control UI import after a package swap can throw `MODULE_NOT_FOUND`, and current request handling turns that into `Internal Server Error`. I did not perform the macOS/Homebrew live reinstall, so this is source-reproducible rather than locally reproduced.

## Solution Assessment

No patch is proposed yet. The best fix is a narrow packaging/runtime-boundary change for the Control UI lazy import plus regression coverage; asking users to restart the Gateway is only a workaround.
