## Summary

Keep open: current main and the latest release still use the root-help fast-path error handler that logs, sets `process.exitCode = 1`, and returns as handled. An open linked PR is the implementation candidate, so this issue should stay open until that PR or a replacement lands.

## What This Changes

The issue reports that root help fast-path failures can leave the CLI process alive because the handler sets `process.exitCode` without explicitly terminating or draining active handles.

## Best Possible Solution

Land a narrow fast-path error-boundary fix through https://github.com/openclaw/openclaw/pull/97807 or a replacement that preserves injectable test seams and adds child-process coverage for active handles.

## Reproduction Assessment

No high-confidence real OpenClaw hang repro was run. Source inspection shows the fast path only sets `process.exitCode` and returns handled, and a focused Node probe confirms active handles keep such a process alive.

## Solution Assessment

Yes, the right fix is a narrow fast-path error-boundary repair rather than a new setting or broader runtime change. The linked PR should preserve the version fast-path injected `exit` seam and add focused child-process proof before merge.
