## Summary

Current main already addresses this 2026.4.29 Telegram cold-start latency report through the merged #75360 work, with source, tests, changelog, and PR validation all pointing at the same reply-startup bottleneck. The fix is on main only; no release tag in this checkout contains the proof commit yet.

## What This Changes

The issue reports a 2026.4.29 regression where Telegram replies take tens of seconds because embedded-run startup/prep work and event-loop stalls delay simple messages.

## Best Possible Solution

Keep the #75360 implementation on main and ship/consume the next release containing it; if comparable latency persists there, open a fresh issue with a diagnostics export and new embedded-run stage traces so any remaining provider-auth or stream-setup delay can be isolated separately.

## Reproduction Assessment

Yes for the reported release: the issue provides concrete v2026.4.29 traces, event-loop warnings, Telegram timeout logs, and rollback comparison steps. I did not live-run the WSL2 Telegram setup, but current main has targeted regression coverage and the merged PR includes Telegram QA RTT proof.

## Solution Assessment

Yes. The current fix is the narrow maintainable direction because it removes optional typing/media/provider work from the Telegram reply startup path without hard-coding Telegram behavior into core; residual model-auth latency should be tracked as a separate provider/runtime trace if it still appears after this fix.
