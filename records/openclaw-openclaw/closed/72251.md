## Summary

Keep open: this is a plausible fix for a real live-backup race, but the latest patch still has a retry diagnostic bug and the supplied real behavior proof predates the current head and newer delivery-queue/filter changes.

## What This Changes

The PR adds backup volatile-file filtering, tar EOF retry/error enrichment, skipped-volatile reporting, docs/changelog updates, and focused backup tests.

## Best Possible Solution

Track the actual final attempt count, add the missing retry diagnostic assertion, then provide redacted latest-head live backup proof covering current agent sessions, cron JSONL, delivery queues, and workspace lock/temp preservation before merge.

## Reproduction Assessment

Yes. Source and dependency inspection give a high-confidence path: current main feeds live state directories to node-tar `tar.c`, while node-tar `WriteEntry#onread` throws EOF when file size changes after `lstat()`.

## Solution Assessment

No, not yet. Retry plus a state-scoped volatile filter is the right maintainable direction, but the final error attempt count and latest-head proof need to be fixed before this is merge-ready.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Report the actual tar failure attempt count:** `src/infra/backup-create.ts:169-171`
  - body: `writeTarArchiveWithRetry` always formats final failures as `after 3 attempts`. Non-EOF errors break out immediately, and a non-EOF error after one retry breaks on attempt 2, so permission or cleanup failures are reported with retry attempts that never happened. Track the last attempted count and use that in the suffix.
  - confidence: 0.87
