## Summary

Current `main` no longer derives mismatched bundled-plugin `idHint` values for the bundled plugin set, so the v2026.3.22 warning storm described in #53048 is obsolete on current code.

## Evidence

- **Bundled idHint logic now prefers manifest id:** `deriveBundledPluginIdHint(...)` returns `manifestId` whenever a bundled plugin has a single declared runtime entry, which prevents the old basename/package-name hint drift that produced bundled-plugin id mismatch warnings.
  - file: [src/plugins/bundled-plugin-scan.ts:51](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/plugins/bundled-plugin-scan.ts#L51)
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Bundled metadata uses the normalized helper:** Bundled plugin metadata population calls `deriveBundledPluginIdHint(...)` when constructing each bundled candidate, so discovery uses the normalized id hint for bundled plugins.
  - file: [src/plugins/bundled-plugin-metadata.ts:149](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/plugins/bundled-plugin-metadata.ts#L149)
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Current bundled plugin set has zero computed mismatches:** Read-only audit of `extensions/*` package manifests and `openclaw.plugin.json` files found `bundled_plugins=102` and `mismatches=0` under the current bundled id-hint rules.
