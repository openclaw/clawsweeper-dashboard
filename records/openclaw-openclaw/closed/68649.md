## Summary

Keep open. Current main still omits a PDF-specific remote body-read idle timeout, so a stalled PDF download can sit in the tool path until broader session recovery intervenes instead of returning a bounded tool error.

## What This Changes

The issue asks for the PDF tool to time out stalled remote PDF work and avoid wedging the session queue behind an in-flight tool call.

## Best Possible Solution

Thread a conservative default readIdleTimeoutMs through the PDF remote-load path and WebMediaOptions, with focused regression coverage and no broad scheduler or session-policy rewrite.

## Reproduction Assessment

Yes, at source level. The PDF tool reaches a body reader that only enforces idle timeouts when readIdleTimeoutMs/chunkTimeoutMs is supplied, and current main does not supply it on the PDF remote-load path.

## Solution Assessment

Yes for the PDF-specific defect. The narrow maintainable fix is to pass an idle body-read timeout through pdf-tool.ts and web-media.ts while leaving broader slash-command and watchdog policy to separate recovery work.
