## Summary

Close as duplicate/superseded by the already-open config data-loss policy tracker #70578. Current main has source-preserving config writes and fail-closed guards for the concrete `gateway.mode` clobber path, while the remaining request to preserve unknown top-level fields is the same root config contract decision already being tracked there.

## What This Changes

The issue reports that `openclaw config set` and `openclaw plugins install` can drop unrelated top-level config fields such as `gateway`, causing later gateway startup to fail with missing `gateway.mode`.

## Best Possible Solution

Keep one canonical config-contract discussion open in #70578. Any future fix should either deliberately preserve unknown top-level config as opaque data or keep strict validation but add clear fail-closed coverage and diagnostics for config set, plugin install, and gateway startup paths.

## Reproduction Assessment

Yes. The issue body gives a concrete command sequence, but current-main source review indicates the `gateway.mode` removal path is now preserved or rejected before commit; the remaining unknown top-level passthrough behavior is reproducible as the strict-root policy tracked in #70578.
