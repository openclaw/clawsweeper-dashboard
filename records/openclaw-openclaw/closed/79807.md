## Summary

Keep open: current main still auto-inherits Slack thread context for same-channel message-tool sends, and no documented or typed parameter lets an agent force a true top-level same-channel Slack post from inside a thread.

## What This Changes

The issue requests an explicit message-tool opt-out for Slack same-channel thread inheritance so agents can send a new channel-root message from a threaded context.

## Best Possible Solution

Add a documented, backward-compatible message-tool opt-out that is handled before generic auto-thread resolution and Slack runtime auto-injection, while preserving default Slack thread inheritance.

## Reproduction Assessment

Yes, from source inspection. A same-channel Slack send from a threaded tool context resolves the active `currentThreadTs`, injects it as `threadId`/`threadTs`, and sends it as Slack `thread_ts`; I did not run a live Slack workspace test.

## Solution Assessment

Unclear until maintainers choose the public API shape. The maintainable direction is a generic explicit opt-out or sentinel consumed before both generic and Slack-specific auto-threading, not a Slack-only workaround after `thread_ts` has already been injected.
