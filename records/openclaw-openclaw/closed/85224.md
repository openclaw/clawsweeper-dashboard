## Summary

Keep open: current main and v2026.5.20 still classify macOS `XPC_SERVICE_NAME` as launchd supervision, and two open PRs already reference this issue as the fix path, so the issue should close only after one merges.

## What This Changes

The issue reports false launchd-supervised gateway respawn on macOS when a gateway child inherits generic launchd/XPC environment from a parent app.

## Best Possible Solution

Land one focused linked PR that distinguishes inherited parent launchd/XPC state from the gateway's own LaunchAgent identity, with regression tests for inherited XPC and real OpenClaw launchd markers.

## Reproduction Assessment

Yes for a source-level reproduction: current Darwin detection treats `XPC_SERVICE_NAME` as launchd, and the update respawn helper then returns `mode: "supervised"`. I did not run a live macOS LaunchAgent reproduction in this read-only review.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix: stop treating inherited generic launchd/XPC state as proof of gateway supervision while preserving OpenClaw-owned LaunchAgent markers. The issue itself should stay open because the referenced fixing PRs have not merged.
