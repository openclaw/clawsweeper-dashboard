## Summary

Keep open. The PR addresses a valid current-main startup failure and I did not find a discrete blocking bug in the diff, but it is draft and carries the protected `maintainer` label, so it needs explicit maintainer handling rather than cleanup closure.

## What This Changes

The PR adds startup-only degradation for unresolved auth-store SecretRefs, related warning/docs/changelog text, and regression coverage while keeping reload and config-level SecretRefs strict.

## Best Possible Solution

Undraft and land this after maintainer review and required checks if the broader startup auth-profile degradation policy is intended, preserving strict config/reload behavior and closing the linked bug on merge.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to load an auth store with an `api_key.keyRef` or `token.tokenRef` whose provider is absent from `secrets.providers`, then run gateway startup secret activation; current main batch-resolves that ref and turns the missing-provider error into startup failure.

## Solution Assessment

Yes. Splitting config SecretRef resolution from auth-store profile resolution, applying degradation only during startup/restart preflight, and keeping reload/config strict is the narrow maintainable direction for the reported gateway availability bug.
