## Summary

Current main already has Telegram polling stall recovery and network-error retry logic that addresses the reported symptom, and the browser proxy path is asynchronous with abort-based timeouts rather than a synchronous event-loop block.

## Evidence

- **Browser proxy call is async node invoke:** The browser tool forwards proxy actions through awaited `node.invoke` with a per-call timeout and gateway slack; it does not run a synchronous busy wait on the Telegram loop.
  - file: [extensions/browser/src/browser-tool.ts:276](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/extensions/browser/src/browser-tool.ts#L276)

  - sha: [56fe2aab9c87](https://github.com/openclaw/openclaw/commit/56fe2aab9c87596515f2280c8060dc740ea64a88)
- **Node-host timeout is abort-based:** The node-host timeout helper uses `AbortController`, `setTimeout`, and `Promise.race`, which is non-blocking timeout orchestration rather than event-loop blocking work.
  - file: [src/node-host/with-timeout.ts:1](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/src/node-host/with-timeout.ts#L1)
