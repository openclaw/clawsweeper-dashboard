## Summary

Keep open: the PR targets a real source-reproducible bundled MCP timeout problem, but it is not merge-ready because contributor real behavior proof is missing, the promised progress-reset behavior is not actually wired, the new config field is missing from the schema, and a related open PR needs timeout-policy reconciliation.

## What This Changes

The PR adds a per-server bundled MCP `requestTimeoutMs`, propagates it through transport/session state into `Client.callTool`, documents it, and adds focused tests.

## Best Possible Solution

Land one reconciled patch that schema-documents a per-server bundled MCP request-timeout contract, passes correct SDK request options, covers progress behavior if promised, and includes real long-running MCP proof.

## Reproduction Assessment

Yes, at source level: configure a bundled MCP tool that runs longer than 60 seconds and current main reaches `Client.callTool` without `RequestOptions.timeout`, so SDK 1.29.0 applies its 60-second default. I did not run a live slow MCP server in this read-only review.

## Solution Assessment

No, not as submitted. Passing `RequestOptions.timeout` is the right narrow code path, but the progress reset wiring, schema contract, real proof, and timeout-policy choice still need follow-up.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Request progress before relying on progress resets:** `src/agents/pi-bundle-mcp-runtime.ts:368`
  - body: The SDK only adds a progress token and stores a progress handler when `RequestOptions.onprogress` is provided. This PR passes `resetTimeoutOnProgress: true` without `onprogress`, so bundled MCP servers are not asked to send progress for this request and progress cannot reset the timer; either pass an appropriate progress callback or remove the promised reset behavior from code/docs/tests.
  - confidence: 0.9
- **[P2] Add requestTimeoutMs to the config schema:** `src/config/types.mcp.ts:20-21`
  - body: The PR adds `requestTimeoutMs` to the exported type, docs, and runtime, but leaves `McpServerSchema` unchanged. That omits the new public config option from schema-derived config/help/editor surfaces; add a positive finite-number field in `src/config/zod-schema.ts` with focused schema coverage.
  - confidence: 0.88
