## Summary

Keep open. Current main still has the reported source-level mismatch: sandbox explain derives sandboxed status from config/session mode, while the CLI runner records CLI-backed execution as unsandboxed and the sandbox browser error still treats a missing bridge as a browser-enabled configuration problem.

## What This Changes

The issue asks OpenClaw to make Claude CLI sandbox diagnostics and target="sandbox" browser failures match the actual CLI runner execution boundary.

## Best Possible Solution

Make the reported sandbox status and target="sandbox" browser failure reflect the prepared runner: CLI-backed turns should clearly say no OpenClaw sandbox bridge/runtime exists unless maintainers intentionally add supported sandboxed CLI execution.

## Reproduction Assessment

Yes, at source level: sandbox explain computes sandboxed from config while CLI preparation hard-codes sandbox off for CLI backend runs. A live Claude CLI OAuth/Docker reproduction was not performed in this read-only review.

## Solution Assessment

Unclear for full sandbox support. A diagnostics-first fix is the narrow maintainable repair if CLI backends remain unsandboxed; adding sandboxed Claude CLI execution needs an explicit runner/security decision.
