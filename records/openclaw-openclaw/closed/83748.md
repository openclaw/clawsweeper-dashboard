- `impact:message-loss`: A delivered Telegram sticker can reach the agent context without the usable media attachment path needed for image inspection.

## Summary

Keep open: current main and v2026.6.6 still expose Telegram cached media to the model as `telegram:file/...` refs without a hydrated media path in conversation context, and this issue has sticker-specific static WEBP behavior that is not fully owned by the broader photo issue.

## What This Changes

The issue asks for Telegram inbound stickers to be hydrated into agent-readable workspace media paths instead of only Telegram provider file refs.

## Best Possible Solution

Preserve static Telegram sticker media as a canonical local or `media://inbound/*` reference in current, reply, and conversation context while keeping Telegram file IDs as plugin-private metadata and leaving animated/video sticker support unchanged.

## Reproduction Assessment

Yes at source level: current main still serializes cached Telegram media as `telegram:file/...` and the image tool rejects that scheme. I did not run a live Telegram sticker through docker/all in this read-only review.

## Solution Assessment

Yes, the best fix is at the Telegram media/context handoff, not in the core image tool. The fix should convert or carry the existing saved sticker media path into the model-visible context instead of making core tools understand Telegram file IDs.
