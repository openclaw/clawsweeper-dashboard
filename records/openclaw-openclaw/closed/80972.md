## Summary

Current main already addresses the reported heartbeat leak: heartbeat runs are deferred when the main/session lane is busy, heartbeat turns are dropped if they still encounter an active run, and ack-only heartbeat text is kept out of pending final-delivery replay. The proof commit is on main only; no release tag in this checkout contains it yet.

## What This Changes

The issue reports that heartbeat `HEARTBEAT_OK` acknowledgments can be delivered to the Telegram/user channel during long active turns and displace the real user reply.

## Best Possible Solution

Keep the current main implementation: defer heartbeat work before reply generation when a session is active, preserve ack-only suppression, and let the next release carry the fix to affected installations.

## Reproduction Assessment

No current-main failing path was established. The release-era report is plausible, but current main has source and regression-test guards that prevent the heartbeat from running or persisting ack-only replay state during the reported active-session path.

## Solution Assessment

Yes. Deferring/dropping heartbeat turns at the existing lane and queue boundaries is narrower than adding a global turn-lock and avoids putting heartbeat-specific policy into generic outbound delivery.
