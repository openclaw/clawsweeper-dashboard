## Summary

Current main and the published v2026.3.24 package both define `autoThreadName` as a Discord guild-channel field (`channels.discord.guilds.<guild>.channels.<channel>.autoThreadName`), not a top-level `channels.discord` or guild-root key, so the reported validation error matches an unsupported config location rather than an open mainline bug.

## Evidence

- **Current type contract scopes the field to guild channels:** `autoThreadName` is declared on `DiscordGuildChannelConfig`, while `DiscordGuildEntry` only exposes `channels?: Record<string, DiscordGuildChannelConfig>`, so the supported path is under `guilds.<id>.channels.<channel>`.
  - file: [src/config/types.discord.ts:39](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/config/types.discord.ts#L39)
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Current runtime schema accepts the field only on guild channel entries:** `DiscordGuildChannelSchema` includes `autoThreadName`, which means config validation accepts it at the channel-entry level, not at `channels.discord` or `channels.discord.guilds.<id>`.
  - file: [src/config/zod-schema.providers-core.ts:449](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/config/zod-schema.providers-core.ts#L449)
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Generated bundled config metadata exposes the same channel-only field:** The generated bundled channel metadata includes `autoThreadName` inside the Discord channel override object in both emitted config metadata blocks, which is what the config surface uses for channel-level docs/UI/schema generation.
  - file: [src/config/bundled-channel-config-metadata.generated.ts:1285](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/config/bundled-channel-config-metadata.generated.ts#L1285)
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Runtime behavior consumes the setting from resolved channel config:** Discord channel resolution copies `entry.autoThreadName` from the resolved channel entry, and auto-thread creation only checks `params.channelConfig?.autoThreadName === "generated"`, confirming the feature is channel-scoped at runtime.
  - file: [extensions/discord/src/monitor/allow-list.ts:400](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/extensions/discord/src/monitor/allow-list.ts#L400)
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Published v2026.3.24 package already shipped the same channel-only contract:** The npm package for `openclaw@2026.3.24` declares `autoThreadName` on `DiscordGuildChannelConfig`, followed by `DiscordGuildEntry.channels?: Record<string, DiscordGuildChannelConfig>`, so the release-note feature was shipped as a guild-channel option rather than a top-level or guild-root option.
  - file: [package/dist/plugin-sdk/src/config/types.discord.d.ts:35](https://github.com/openclaw/openclaw/blob/6c81728723d6b64b8ee6f8f5dc5ebc7946f6957f/package/dist/plugin-sdk/src/config/types.discord.d.ts#L35)

## Risks / Open Questions

- The public docs currently describe Discord config structure at `guilds.*.channels.*`, but they do not explicitly call out `autoThreadName`, so the same misconfiguration could recur.
- I did not execute the reporter's exact installed binary; this review is based on current main plus the published v2026.3.24 package contract, which both point to an unsupported config path rather than an open code defect.
