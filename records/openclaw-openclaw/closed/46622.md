## Summary

Current main already prevents `thinking: "minimal"` from being forwarded unchanged to OpenAI-compatible NVIDIA/vLLM fallbacks, and the same behavior is present in release `v2026.4.22`.

## Evidence

- **Reasoning normalization maps `minimal` away from unsupported generic OpenAI-compatible values:** `resolveOpenAIReasoningEffortForModel` maps `minimal` to `low` when the target supports `low`, and otherwise to `medium` when needed.
  - file: [src/agents/openai-reasoning-effort.ts:119](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/src/agents/openai-reasoning-effort.ts#L119)
  - command: `nl -ba src/agents/openai-reasoning-effort.ts | sed -n '104,128p'`
  - sha: [56fe2aab9c87](https://github.com/openclaw/openclaw/commit/56fe2aab9c87596515f2280c8060dc740ea64a88)
- **OpenAI completions payload uses the normalized value before sending `reasoning_effort`:** `buildOpenAICompletionsParams` resolves the reasoning effort first and only then assigns `params.reasoning_effort`.
  - file: [src/agents/openai-transport-stream.ts:1748](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/src/agents/openai-transport-stream.ts#L1748)

- **Regression test covers the reported shape directly:** The transport test asserts that OpenAI completions requests with shared reasoning `minimal` emit `reasoning_effort === "low"`.
  - file: [src/agents/openai-transport-stream.test.ts:1370](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/src/agents/openai-transport-stream.test.ts#L1370)

  - file: [extensions/nvidia/provider-catalog.ts:16](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/extensions/nvidia/provider-catalog.ts#L16)

## Risks / Open Questions

- If someone manually configured a custom OpenAI-compatible fallback model with `reasoning: true` and an unusual backend that rejects both `minimal` and `low`, that would be a different bug than the one reported here.
