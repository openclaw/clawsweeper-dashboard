- `impact:session-state`: The issue is about task rows being associated with the wrong agent/session context for active sub-agent work.

## Summary

Keep this open: current main has the task_runs.agent_id column and task API field, but the sessions_spawn sub-agent task path still does not pass the executing child agent into the task row, so cross-agent attribution remains incomplete.

## What This Changes

The issue requests first-class agent_id attribution in task_runs for sessions_spawn child runs so dashboards and plugins do not need to parse child_session_key or labels.

## Best Possible Solution

Pass the resolved target child agent into sub-agent task creation, or infer it from the child session key before falling back to parent ownership, while preserving requester/owner fields and adding a cross-agent regression test.

## Reproduction Assessment

Yes, at source level: a parent session spawning a different child agent reaches createRunningTaskRun without explicit agentId, and task-registry inference reads the parent owner/requester before any child-session agent. I did not run a live SQLite repro because this review is read-only.

## Solution Assessment

Yes, the narrow maintainable fix is in core task attribution because task_runs.agent_id is already a persisted field consumed by plugin/dashboard readers. A reader-side child_session_key convention should remain only a workaround, not the canonical source.
