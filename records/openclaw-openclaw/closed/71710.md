## Summary

Keep open. Current main still lacks a one-shot `openclaw agent` SIGTERM/SIGINT bridge that can cancel the Gateway request, abort the accepted Gateway run, or abort the embedded/local fallback path; the newer Codex no-signal hang data point is related but does not resolve the external-supervisor SIGTERM case.

## What This Changes

The issue requests bounded SIGTERM handling, or at least cron/supervisor documentation, for `openclaw agent` processes run under wrappers such as `timeout`.

## Best Possible Solution

Add a bounded one-shot CLI shutdown path that forwards SIGTERM/SIGINT into Gateway and embedded agent execution, aborts the accepted Gateway run when known, and documents `timeout -k` as an external-supervisor backstop.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: hold a Gateway `agent` expect-final request or embedded local run pending, send SIGTERM/SIGINT to the one-shot CLI, and current main has no caller AbortSignal, accepted-run callback, or top-level signal bridge to observe.

## Solution Assessment

Is this the best way to solve the issue? Yes: bridge process signals into the existing Gateway abort controller and local runner abort surfaces, with a bounded cleanup grace and documented `timeout -k` fallback rather than adding a broader supervisor policy here.
