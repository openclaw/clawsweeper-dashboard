## Summary

Current `main` already implements the reported fix path: `DiscordMessageListener` no longer waits for the full agent turn, inbound work is handed to a separate queued worker with its own timeout handling, and the changelog records this as shipped in `2026.3.13`.

## Evidence

- **Listener no longer blocks gateway event handling:** `DiscordMessageListener.handle` now calls the message handler in fire-and-forget mode instead of awaiting it, with an inline comment that the Carbon listener should not be blocked by long agent work.
  - file: [extensions/discord/src/monitor/listeners.ts:219](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/extensions/discord/src/monitor/listeners.ts#L219)
  - command: `nl -ba extensions/discord/src/monitor/listeners.ts | sed -n '190,245p'`
- **Long-running work moved to queued inbound worker:** Discord monitor startup constructs the message handler with a separate `workerRunTimeoutMs`, then registers that handler with the listener. The message handler builds a dedicated inbound worker queue and enqueues jobs instead of running the full turn inline.
  - file: [extensions/discord/src/monitor/provider.ts:1049](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/extensions/discord/src/monitor/provider.ts#L1049)
  - command: `nl -ba extensions/discord/src/monitor/provider.ts | sed -n '1035,1098p'`
- **Worker owns long-run timeout and fallback behavior:** The inbound worker applies its own timeout budget and timeout reply path (`discord inbound worker timed out after ...`), which isolates long agent runs from the gateway listener budget.
  - file: [extensions/discord/src/monitor/inbound-worker.ts:76](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/extensions/discord/src/monitor/inbound-worker.ts#L76)
  - command: `nl -ba extensions/discord/src/monitor/inbound-worker.ts | sed -n '68,128p'`
- **Regression coverage matches the reported root cause:** Tests now assert that `DiscordMessageListener` returns immediately without awaiting handler completion, dispatches subsequent events concurrently, and no longer owns slow-listener logging.
  - file: [extensions/discord/src/monitor/listeners.test.ts:34](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/extensions/discord/src/monitor/listeners.test.ts#L34)
  - command: `nl -ba extensions/discord/src/monitor/listeners.test.ts | sed -n '33,151p'`
- **Shipped release note:** The changelog for `2026.3.13` includes: `Runtime/tool-state stability: ... serialize long-running Discord handler runs without blocking new inbound events.` That matches this issue's reported blocking-listener failure mode.
  - file: [CHANGELOG.md:3030](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/CHANGELOG.md#L3030)
  - command: `nl -ba CHANGELOG.md | sed -n '3000,3035p'`

## Close Comment

Closing as implemented on `main` after Codex review. The specific root cause described here is already addressed in current OpenClaw: `extensions/discord/src/monitor/listeners.ts:219` no longer awaits the full Discord message handler, `extensions/discord/src/monitor/message-handler.ts:79` hands work to a separate queued inbound worker, and `extensions/discord/src/monitor/inbound-worker.ts:76` applies the long-run timeout there instead of on the gateway listener path. The changelog also records this in release `2026.3.13` at `CHANGELOG.md:3030` (`serialize long-running Discord handler runs without blocking new inbound events`). Because the reported failure mode was the listener blocking the gateway event loop, this issue is now covered by code already shipped. If you still hit Discord `1006` drops on a current build, please open a new issue with fresh logs and the exact current version, since that would be a different reconnect path from the one fixed here.
