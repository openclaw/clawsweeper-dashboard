## Summary

Keep open: current main still lacks a shared foreground freshness guard for WhatsApp-style buffered auto-reply delivery, and open PR #76970 is the linked implementation candidate rather than a reason to close before merge.

## What This Changes

The issue requests that foreground WhatsApp replies be bound to their originating inbound turn so older tool-backed finalizations cannot appear after a newer same-chat message.

## Best Possible Solution

Review and land or replace #76970 with a shared buffered-dispatch freshness guard plus regression coverage, then close this issue through the merged fix.

## Reproduction Assessment

No high-confidence live reproduction was established here. Source inspection gives a medium-confidence path: a newer inbound message can steer an active run while the older run's final delivery path lacks a shared origin freshness check.

## Solution Assessment

Yes, the best direction is a shared buffered-dispatch freshness guard like #76970 proposes. A WhatsApp-only workaround would leave the generic auto-reply surface and other buffered channels exposed to the same race.
