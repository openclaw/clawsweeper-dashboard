## Summary

Current main and the latest release already address the reported v2026.4.7 resource-heavy channel diagnostics path: `channels list` no longer loads provider auth/usage snapshots, and `channels status --probe` now bounds probe/summary hooks with timeouts and concurrency.

## What This Changes

The issue reports high CPU and memory usage when running `openclaw channels list`, `openclaw channels status --probe --timeout 8000 --json`, and similar built-in commands on v2026.4.7.

## Best Possible Solution

Keep the v2026.5.28/current-main diagnostics behavior and ask for a fresh profile on the latest release if a specific command still shows excessive RSS or CPU.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? The v2026.4.7 source clearly shows `channels list` doing provider usage work by default and `channels.status` calling probe hooks directly, but I did not run a live CPU/RSS reproduction; current main and v2026.5.28 no longer use those exact paths.

## Solution Assessment

Is this the best way to solve the issue? Yes: the maintainable fix is the shipped behavior that keeps channel listing scoped to channel state and bounds live status probes, without adding a new config knob or fallback path.
