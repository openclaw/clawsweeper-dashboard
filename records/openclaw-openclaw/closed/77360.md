## Summary

Keep open: current main still omits the already-computed inbound sender/conversation metadata from the bare reset/bootstrap prompt branch, so this is a narrow source-backed prompt-composition bug.

## What This Changes

Ensure bare `/new` and `/reset` bootstrap prompts include existing user-role inbound sender and conversation metadata.

## Best Possible Solution

Prepend inbound user context to the bare reset/bootstrap user prompt and cover bare `/new` and `/reset` while leaving trusted system metadata unchanged.

## Reproduction Assessment

Yes, from source: a `runPreparedReply` case with `resetTriggered`, bare `/new` or `/reset`, and sender fields reaches `runReplyAgent` through the `isBareSessionReset` branch without `inboundUserContext` in `followupRun.prompt`. I did not run the test because this review is read-only.

## Solution Assessment

Yes. Adding `inboundUserContext` to the existing bare reset prompt array is the narrowest maintainable fix, provided the ordinary command acknowledgement path and hidden transcript marker behavior remain unchanged.

## Repair Work Prompt

Fix the sender identity omission on bare `/new` and `/reset` reset/bootstrap prompts. In `src/auto-reply/reply/get-reply-run.ts`, include the existing `inboundUserContext` in the `isBareSessionReset` `baseBodyForPrompt` array before startup/reset prompt content, without adding sender fields to `buildInboundMetaSystemPrompt`. Add focused regression coverage in `src/auto-reply/reply/get-reply-run.media-only.test.ts` for bare `/new` and bare `/reset` with sender/conversation fields, asserting `followupRun.prompt` contains `Conversation info (untrusted metadata):` and `Sender (untrusted metadata):` while preserving the hidden transcript marker and startup instruction assertions. Do not change the ordinary `commands-reset.ts` acknowledgement path for text commands. Add a single-line `CHANGELOG.md` fix entry because this is user-facing.
