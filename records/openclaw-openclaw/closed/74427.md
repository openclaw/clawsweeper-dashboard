## Summary

Keep open: the reported source-level problem is real, but this PR is still incomplete because it only applies the env-aware default in the dynamic fallback path while catalogued OpenAI rows can resolve from the static registry first, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The branch adds an OpenAI default base URL helper that reads non-empty `OPENAI_BASE_URL`, uses it in GPT dynamic fallback metadata, adds helper tests, and adds a changelog entry.

## Best Possible Solution

Use one env-aware OpenAI default base URL path across dynamic fallbacks and static/catalog model resolution, preserve explicit provider config precedence, then land after redacted real proxy proof and focused resolver/helper tests.

## Reproduction Assessment

Yes, source-level: current main hardcodes the OpenAI fallback URL and the resolver uses discovered/static `baseUrl` when provider config has no `baseUrl`. I did not run a live proxy request in this read-only review.

## Solution Assessment

No, not yet. The PR applies the env default only in dynamic fallback metadata, but catalogued refs can still return from the static registry before dynamic resolution; the fix should cover that path or add resolver proof for the intended env-only refs.

## Review Findings
