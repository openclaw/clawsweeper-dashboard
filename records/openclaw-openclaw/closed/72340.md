## Summary

Close as superseded by open PR #56544. Current main still requires a Slack reply thread target before native streaming starts, and upstream Slack docs/SDK types require `thread_ts` for `chat.startStream`, so the issue's proposed threadless native-stream gate change is not the right core fix. The remaining user-visible problem, flat Slack DMs falling back to multiple normal messages instead of a live preview, is already tracked by #56544, which changes the draft-preview eligibility gate for DMs.

## Best Possible Solution

Close this issue as superseded by open PR #56544. Keep the native Slack stream path thread-backed unless Slack changes its documented API contract; handle the flat-DM UX by merging or adapting #56544's draft-preview approach and updating Slack streaming docs to distinguish native thread streaming from draft post-and-edit preview.
