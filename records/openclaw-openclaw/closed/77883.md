## Summary

Keep open: current main still has a source-clear restart-recovery/WebChat gap. Restart recovery can resume through an internal `agent` run and the embedded runner can return the existing incomplete-turn error payload, but the WebChat/chat final path still only emits buffered assistant text or reloads persisted transcript history, so a recovered tool-only turn can still end without a visible terminal error.

## What This Changes

The issue requests that Control UI visibly surface a terminal error or failed state when restart recovery completes a resumed turn with no user-visible assistant payload.

## Best Possible Solution

Bridge the existing incomplete-turn error payload into WebChat final/history visibility for restart-recovery agent runs without changing provider empty-response fallback, external delivery defaults, or restart checkpoint semantics.

## Reproduction Assessment

Yes, at source level: restart recovery can run an internal `agent` resume that returns an incomplete-turn `isError` payload while WebChat receives only agent/tool lifecycle events and an empty final/history reload path. I did not run the full Windows/custom-provider restart scenario live.

## Solution Assessment

Yes. The narrow maintainable fix is to surface or persist the existing incomplete-turn error at the restart-recovery Gateway/WebChat boundary, rather than changing provider fallback logic or redesigning recovery checkpointing.
