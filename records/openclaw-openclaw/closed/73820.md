## Summary

Current main still has the reported empty-input path for `openai-codex-responses`: Codex instructions are moved to top-level `instructions`, the system prompt is excluded from converted `input`, and the resulting `messages` array is assigned directly to `input`. The active memory flush path can still reach the provider with an empty visible prompt and runtime/system-only context, and I did not find a merged fix or regression assertion for this case.

## Best Possible Solution

Keep this issue open and fix the OpenAI Codex Responses transport in `src/agents/openai-transport-stream.ts` so system-prompt-only Codex requests do not send `input: []`. Add a regression test for `buildOpenAIResponsesParams` with provider `openai-codex`, api `openai-codex-responses`, `systemPrompt` set, and `messages: []`; if the chosen product behavior is to reject such runs, surface a local actionable error instead of letting the ChatGPT backend timeout path hide the 400.
