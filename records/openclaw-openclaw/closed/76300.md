## Summary

Close as implemented on main: current Telegram command-menu sync trims over-budget menus, retries recoverable `BOT_COMMANDS_TOO_MUCH` responses, contains command-sync failures instead of throwing through startup, and documents the native-command disable knob.

## What This Changes

The report asks Telegram startup to remain usable when large skill/plugin command menus exceed the native Bot API payload budget, either by truncating/skipping menu registration or exposing a disable config.

## Best Possible Solution

Keep the Telegram-plugin-local behavior: trim oversized menus, retry or reduce native command registration when Telegram rejects it, and let operators disable native menus with `channels.telegram.commands.native: false` when they prefer text-only commands.

## Reproduction Assessment

Yes for a source-level reproduction path: the issue maps directly to Telegram menu sync with many commands, and current tests exercise both long-description payload trimming and mocked `BOT_COMMANDS_TOO_MUCH` retry. I did not run live Telegram because this was a read-only checkout review.

## Solution Assessment

Yes. Handling menu overflow inside the Telegram plugin, with a documented provider-level disable knob, is narrower and safer than adding core startup policy or filtering all skills globally.
