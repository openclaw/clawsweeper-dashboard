## Summary

Current main already addresses the reported Telegram callback timeout path: Telegram callback queries are acknowledged before model-picker data is rebuilt, and the catalog/auth paths named in the report have shared catalog and provider/CLI credential caching.

## What This Changes

The issue reports that Telegram /models browse picker interactions can exceed Telegram callback timeout because model catalog auth checks are repeated per render without effective cross-interaction caching.

## Best Possible Solution

Keep the current-main immediate callback ACK and cache behavior; handle any remaining post-ACK picker render latency as a narrower timing-backed follow-up.

## Reproduction Assessment

No; I did not run a live Telegram/macOS keychain reproduction, and current-main source no longer has the reported answer-after-render timeout path. The beta report remains plausible for 2026.5.16-beta.5, but the source path is changed on current main.

## Solution Assessment

Yes; acknowledging the Telegram callback before expensive picker rendering is the narrowest fix for the 30-second Bot API deadline, and current main also caches the catalog/auth probes named in the report. Further parallelization or a config flag should only be considered if fresh timing logs show remaining render latency after this implementation.
