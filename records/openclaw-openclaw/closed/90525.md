## Summary

Keep open: the report is specific, current source still allows `sessions.list` to show `status: running` while `hasActiveRun` is false, and the existing startup/recovery paths only handle marked restart-aborted or lock-backed sessions.

## What This Changes

The issue asks OpenClaw to reconcile persisted Telegram direct sessions that remain `running` after restart when no active run or task exists.

## Best Possible Solution

Add bounded startup/recovery reconciliation at the existing session-store/restart-recovery boundary so persisted `running` direct-session rows with no live owner are marked for recovery or terminalized without adding Telegram-specific policy.

## Reproduction Assessment

No live high-confidence Telegram restart repro was run. The detailed report plus current source make the inconsistent `running`/`hasActiveRun:false` state source-reproducible.

## Solution Assessment

Yes, a generic recovery reconciliation is the best direction. A Telegram-only reset would treat the symptom in one channel instead of fixing the persisted session-state invariant.
