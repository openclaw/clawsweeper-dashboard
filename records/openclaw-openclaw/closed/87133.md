## Label Justifications

- `P2`: The reported failure would break native subagent task handoff, but current source and the latest release already cover the central behavior.
- `impact:message-loss`: The issue is about a delegated task message being dropped before the child subagent sees it.
- `impact:session-state`: The symptom is an empty child subagent transcript/state at startup instead of the expected task context.

## Summary

Current main and the latest release already deliver native `sessions_spawn` task text into the child subagent’s first visible `[Subagent Task]` user message, so the reported dropped-payload behavior is no longer actionable in core.

## What This Changes

The issue reports that `sessions_spawn(mode="run", runtime="subagent", task="...")` starts a native subagent without delivering the task payload.

## Best Possible Solution

Keep the shipped first-visible-message task delivery path; if a user still sees empty child runs, collect a new report with exact OpenClaw version, sanitized child session transcript/logs, and the full tool input.

## Reproduction Assessment

No high-confidence failing current-main reproduction remains from source review: the reported task argument is required, forwarded, wrapped in `[Subagent Task]`, and sent as the child agent message. I did not run a live gateway spawn because this review was read-only.

## Solution Assessment

Yes; the current shipped solution is the narrow maintainable path because it single-sources the delegated task in the child’s first visible user message instead of duplicating it in system prompt state.
