## Summary

Keep this PR open: current main still rewrites the usage-cost cache after every stale session file and again at the end, while this branch provides a scoped batching fix with regression coverage and sufficient terminal proof. I found no blocking code-review findings; the remaining action is ordinary maintainer review, required checks, and the rebase noted by a member for Dependency Guard.

## What This Changes

The PR batches durable `.usage-cost-cache.json` rewrites during stale-session refreshes, skips no-op refresh writes, and adds a runtime proof script plus regression coverage.

## Best Possible Solution

Land the scoped checkpointing implementation after refreshing the branch against current main and passing the required checks.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: on current main, any refresh with many stale usage-counted session files calls `writeUsageCostCache` once per stale file plus a final write. I did not run the workload locally because this review is read-only, but the loop behavior is explicit in source.

## Solution Assessment

Yes, this is the best owner-boundary fix I found: throttle writes inside `refreshCostUsageCacheForPath`, where stale-file scanning and durable cache mutation already live. Caller-side throttling or a second cache path would duplicate policy and leave the core refresh loop expensive.
