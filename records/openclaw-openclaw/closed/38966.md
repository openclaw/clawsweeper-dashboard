## Label Justifications

- `impact:session-state`: The issue is specifically about TUI-visible session state diverging from the reset gateway session.

## Summary

Keep open: current main and the latest release still leave the TUI blind to external reset notifications because the gateway emits `sessions.changed` on reset while the TUI dispatches only chat and agent-family events. An open but conflicting fork PR references this issue, so maintainers should refresh or close that implementation path before queueing a competing repair.

## What This Changes

The issue asks the TUI to refresh its visible session state when the current session is reset from another client.

## Best Possible Solution

Have the TUI subscribe to and consume canonical `sessions.changed` reset/new events for its active session, reload history and session metadata, clear token/run state, and cover the handler without introducing a parallel event type.

## Reproduction Assessment

Yes, by source inspection: `sessions.reset` emits `sessions.changed` for reset/new, while the TUI only dispatches chat, side-result, and agent events, so the active view cannot refresh from an external reset.

## Solution Assessment

No, the original separate `session` event suggestion is no longer the best shape; the maintainable fix is to consume the existing `sessions.changed` contract in the TUI and preserve current run-state safeguards.
