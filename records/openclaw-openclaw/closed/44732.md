## Summary

Current main already implements the requested proxy-aware Telegram media download path. Telegram media downloads now carry the Telegram transport dispatcher policy into the guarded fetch, explicit proxies opt into trusted proxy-side DNS handling, and SSRF tests cover skipping target DNS pinning while preserving hostname policy checks. The changelog shows this shipped in v2026.4.14.

## Evidence

- **Telegram media download now forwards explicit-proxy trust into guarded media fetch:** `downloadAndSaveTelegramFile` passes both `dispatcherAttempts` from the resolved Telegram transport and `trustExplicitProxyDns: usesTrustedTelegramExplicitProxy(transport)` into `fetchRemoteMedia`, so Telegram file downloads no longer force the old direct pinned-DNS path when an explicit proxy is configured.
  - file: [extensions/telegram/src/bot/delivery.resolve-media.ts:235](https://github.com/openclaw/openclaw/blob/38caa6832d4e1d344736d635b0c5fdad37e4430b/extensions/telegram/src/bot/delivery.resolve-media.ts#L235)
  - command: `nl -ba extensions/telegram/src/bot/delivery.resolve-media.ts | sed -n '150,260p'`
  - sha: [38caa6832d4e](https://github.com/openclaw/openclaw/commit/38caa6832d4e1d344736d635b0c5fdad37e4430b)
- **Shared media fetch switches explicit proxies into trusted explicit-proxy mode:** `fetchRemoteMedia` now selects `withTrustedExplicitProxyGuardedFetchMode` whenever `trustExplicitProxyDns` is true for an `explicit-proxy` dispatcher policy, instead of always using strict pinned-DNS mode.
  - file: [src/media/fetch.ts:133](https://github.com/openclaw/openclaw/blob/38caa6832d4e1d344736d635b0c5fdad37e4430b/src/media/fetch.ts#L133)
  - command: `nl -ba src/media/fetch.ts | sed -n '120,190p'`
  - sha: [38caa6832d4e](https://github.com/openclaw/openclaw/commit/38caa6832d4e1d344736d635b0c5fdad37e4430b)
- **SSRF guard preserves hostname checks but skips target DNS pinning for trusted explicit proxies:** In `fetchWithSsrFGuard`, trusted explicit-proxy mode calls `assertHostnameAllowedWithPolicy(...)` and then builds the dispatcher with `createPolicyDispatcherWithoutPinnedDns(...)` rather than `createPinnedDispatcher(...)`, which addresses the reported proxy bypass root cause.
  - file: [src/infra/net/fetch-guard.ts:348](https://github.com/openclaw/openclaw/blob/38caa6832d4e1d344736d635b0c5fdad37e4430b/src/infra/net/fetch-guard.ts#L348)
  - command: `nl -ba src/infra/net/fetch-guard.ts | sed -n '330,430p'`
  - sha: [38caa6832d4e](https://github.com/openclaw/openclaw/commit/38caa6832d4e1d344736d635b0c5fdad37e4430b)
- **Regression tests cover the exact explicit-proxy DNS behavior:** There is a media-fetch test asserting `trusted_explicit_proxy` mode for explicit proxies, and an SSRF test asserting trusted explicit-proxy mode skips target DNS lookup and only resolves the proxy host.
  - file: [src/media/fetch.test.ts:294](https://github.com/openclaw/openclaw/blob/38caa6832d4e1d344736d635b0c5fdad37e4430b/src/media/fetch.test.ts#L294)
  - command: `nl -ba src/media/fetch.test.ts | sed -n '286,330p'`
