## Summary

Current `main` already blocks plugin command registration conflicts instead of silently letting later plugins override earlier ones, and that behavior is present in the latest release commit `00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5` (`v2026.4.22`).

## Evidence

- **Duplicate plugin commands are rejected at registration time:** `registerPluginCommand(...)` enumerates the command name plus native aliases and returns `ok: false` with `Command "..." already registered by plugin "..."` if any invocation key is already owned, rather than overwriting the existing entry.
  - file: [src/plugins/command-registration.ts:150](https://github.com/openclaw/openclaw/blob/3361593442e295e9bb1f55cbfb38861ea9413e45/src/plugins/command-registration.ts#L150)
  - sha: [3361593442e2](https://github.com/openclaw/openclaw/commit/3361593442e295e9bb1f55cbfb38861ea9413e45)
- **Built-in command hijacking is also blocked:** `validateCommandName(...)` reserves core command names like `help`, `commands`, `debug`, `exec`, and `usage`, so plugins cannot register over security-sensitive built-ins.
  - file: [src/plugins/command-registration.ts:32](https://github.com/openclaw/openclaw/blob/3361593442e295e9bb1f55cbfb38861ea9413e45/src/plugins/command-registration.ts#L32)
  - sha: [3361593442e2](https://github.com/openclaw/openclaw/commit/3361593442e295e9bb1f55cbfb38861ea9413e45)
- **Plugin load path surfaces the conflict as an error:** The plugin registry calls `registerPluginCommand(...)` during activation and emits `command registration failed: ...` diagnostics when registration is rejected, so conflicts do not silently win by load order.
  - file: [src/plugins/registry.ts:1151](https://github.com/openclaw/openclaw/blob/3361593442e295e9bb1f55cbfb38861ea9413e45/src/plugins/registry.ts#L1151)
  - sha: [3361593442e2](https://github.com/openclaw/openclaw/commit/3361593442e295e9bb1f55cbfb38861ea9413e45)
- **Regression tests cover alias collisions and reserved-name rejection:** Tests assert that a second plugin trying to claim an existing alias gets `Command "pair_device" already registered by plugin "demo-plugin"`, and that aliases like `help` are rejected as reserved.
  - file: [src/plugins/commands.test.ts:415](https://github.com/openclaw/openclaw/blob/3361593442e295e9bb1f55cbfb38861ea9413e45/src/plugins/commands.test.ts#L415)
  - sha: [3361593442e2](https://github.com/openclaw/openclaw/commit/3361593442e295e9bb1f55cbfb38861ea9413e45)
- **Same conflict-detection code is shipped in the latest release:** `git show 00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5:src/plugins/command-registration.ts` shows the same duplicate-registration guard and error path in release `v2026.4.22`.
  - file: [src/plugins/command-registration.ts:150](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/plugins/command-registration.ts#L150)
  - command: `git show 00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5:src/plugins/command-registration.ts | nl -ba | sed -n '140,195p'`
  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)
