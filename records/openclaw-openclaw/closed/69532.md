## Summary

Current `main` already handles bare `ws://host:port` remote CDP roots by doing `/json/version` discovery first, then attaching via the discovered DevTools websocket, with regression coverage for Browserless-style wildcard/token endpoints. The fix is documented in the changelog and present in the latest stable tag `v2026.4.23`.

## Evidence

- **Bare ws:// roots now use discovery before being marked down:** `isChromeReachable()` distinguishes direct `/devtools/...` websocket URLs from bare `ws://host:port` roots and normalizes bare websocket roots to HTTP(S) for `/json/version` discovery first.
  - file: [extensions/browser/src/browser/chrome.ts:286](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/extensions/browser/src/browser/chrome.ts#L286)
  - command: `nl -ba extensions/browser/src/browser/chrome.ts | sed -n '286,340p'`
  - sha: [5fe333ada85a](https://github.com/openclaw/openclaw/commit/5fe333ada85afd0def9e85985ce57e36aa00108a)
- **Runtime attach path uses discovered websocket URL:** The Playwright attach path resolves a websocket URL with `getChromeWebSocketUrl()` and uses that endpoint for `chromium.connectOverCDP()`, so runtime attach follows discovery rather than reusing the raw bare root.
  - file: [extensions/browser/src/browser/pw-session.ts:462](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/extensions/browser/src/browser/pw-session.ts#L462)

  - file: [extensions/browser/src/browser/chrome.test.ts:538](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/extensions/browser/src/browser/chrome.test.ts#L538)

## Risks / Open Questions

- `chrome.diagnostics.ts` still treats any `ws://` URL as a direct websocket health target, so diagnostics for some bare-websocket providers may be less precise than the fixed attach path.
