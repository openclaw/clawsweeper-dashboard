## Summary

Current main implements the install-alias fix: bare official external plugin ids now resolve through the official catalog before generic npm fallback, so `openclaw plugins install brave` installs and validates `@openclaw/brave-plugin`. The fix is on main but not in `v2026.5.2`; later web_search runtime/provider comments are split to #76416/#76418.

## What This Changes

After v2026.5.2 externalized Brave, `openclaw plugins install brave` should install the official Brave plugin instead of the unrelated unscoped npm package or a missing beta tag.

## Best Possible Solution

Keep the current main alias-resolution implementation, ship it in the next release, and handle the separate web_search runtime-provider issue in its own tracked thread.

## Reproduction Assessment

Yes. The release/source path is clear: packaged installs exclude bundled Brave, and current main now routes the same bare `brave` spec through the official catalog before npm fallback.

## Solution Assessment

Yes. Resolving only exact bare official plugin ids before npm fallback is the narrow maintainable fix and preserves explicit npm semantics such as `npm:brave`, scoped packages, and `brave@beta`.
