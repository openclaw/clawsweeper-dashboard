## Summary

Keep open: current main still has the Slack non-main DM route contamination bug and this PR is relevant, but the current head is not merge-ready because dispatch can still write threaded non-main DM route metadata to the base peer session and contributor real behavior proof is missing.

## What This Changes

The PR changes Slack prepare/dispatch DM last-route persistence for non-main dmScope values, adds Slack dmScope regression tests, and also adds separate ACP identity/retry runtime and test changes.

## Best Possible Solution

Land a scoped Slack routing fix that writes non-main DM last-route metadata to the prepared turn session, preserves main-DM owner-pin behavior, adds threaded non-main DM coverage, and handles ACP changes in a separate reviewed path or clearly documented scope.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main writes Slack DM last-route metadata to route.mainSessionKey for non-main dmScope values, and the PR-specific threaded case follows from prepare-routing producing a thread-suffixed prepared SessionKey while dispatch targets route.sessionKey.

## Solution Assessment

No. The prepare-side direction is right, but dispatch should target prepared.ctxPayload.SessionKey ?? route.sessionKey for non-main DMs and add threaded regression coverage; the ACP changes should be split or explicitly reviewed as part of the scope.
