## Summary

Current main already solves the reported public-surface activation problem without adding `enabledByDefault` to `memory-core`. `memory-core` is intentionally selected through the default `plugins.slots.memory` path, the activation resolver treats the selected memory slot as enabled before consulting bundled `enabledByDefault`, and the facade runtime-api guard uses that effective activation state. The exact PR patch is not present, but the observable `runtime-api.js` blocked-as-disabled problem is covered by the shipped slot-based behavior.

## Best Possible Solution

Close this PR as redundant. Keep `memory-core` activation controlled by `plugins.slots.memory` and preserve the documented `plugins.slots.memory = "none"` opt-out; any remaining doctor/update failure should be handled as a targeted reproduction against the facade activation path, not by adding `enabledByDefault` to the memory-core manifest.
