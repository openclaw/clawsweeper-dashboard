## Summary

Current main and v2026.4.24 already address the cached/wrong-image path also tracked in #37235. Feishu SDK downloads that expose writeFile are read through isolated per-download temp paths, the shared inbound store writes each provided buffer to a fresh UUID-backed file, and regression tests cover Feishu image/message-resource temp-path isolation. WeChat and WeCom are external plugin-owned surfaces, so any remaining vendor-plugin behavior belongs in those plugin repos rather than as an OpenClaw core cache bug.

## Best Possible Solution

Close this OpenClaw core issue as implemented. Keep the shipped Feishu isolated-download and UUID-backed inbound-store behavior; direct any still-reproducible WeChat or WeCom cases to the external plugin maintainers with fresh reproduction data after upgrading to v2026.4.24 or newer.
