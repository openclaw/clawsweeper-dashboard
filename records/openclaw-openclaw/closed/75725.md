## Summary

Keep this open. The report is recent, detailed, and still matches current main: Telegram direct messages enter the shared inbound reply path, then each embedded run still performs model resolution, auth/profile setup, core/plugin tool materialization, bundle tool setup, system-prompt construction, resource-loader reload, session creation, and stream setup before `stream-ready`. I did not find a merged main-branch change that removes or safely caches the reported per-turn startup/prep work.

## What This Changes

The issue asks OpenClaw to reduce Telegram direct-chat reply latency caused by repeated embedded-agent startup and preparation work before model streaming begins.

## Best Possible Solution

Add a measured warm-turn path in the shared embedded runner that reuses model/auth/tool/prompt/session preparation only behind explicit invalidation boundaries, then prove the improvement with embedded stage summaries and the Telegram RTT harness. The fix should preserve per-turn channel authorization, tool policy, provider runtime hooks, prompt contributions, and session/resource-loader correctness.

## Reproduction Assessment

Yes. The issue body and follow-up comment provide concrete stage traces, and current main still has the same stage timing hooks and Telegram-to-embedded-runner path; reproduction should be repeated trivial Telegram DM turns under trace logging or the RTT harness. I did not run live traffic in this read-only review.

## Solution Assessment

Unclear. The suggested caching areas are plausible, but the best fix should be driven by measured stage deltas and narrowly invalidated shared-runner reuse, not by a Telegram-specific shortcut that bypasses policy, auth, tool, prompt, or provider setup.
