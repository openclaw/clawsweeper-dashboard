## Summary

Keep open: current main still lets terminal session rows participate in normal freshness checks and restart-continuation recovery, so the reported wedge remains a valid core session/channel bug.

## What This Changes

The issue reports that terminal channel sessions can be reused by inbound routing and restart-sentinel continuation recovery after `/new`.

## Best Possible Solution

Make terminal lifecycle state a first-class routing guard: clear terminal fields on fresh-session rotation, treat terminal entries as stale in routing, and apply a maintainer-approved restart-continuation policy for closed sessions.

## Reproduction Assessment

Yes from source inspection, but not from a live Telegram run: a fresh terminal row can remain timestamp-fresh, be reused by resolveSession, and receive a restart continuation without a terminal guard.

## Solution Assessment

Unclear as proposed: preventing terminal-session reuse is the right fix boundary, but maintainers should choose the permanent restart-continuation behavior for a closed session.
