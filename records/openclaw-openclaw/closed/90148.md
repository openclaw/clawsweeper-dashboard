## Summary

Close as duplicate/superseded: the same Node 23 node:sqlite `StatementSync.columns()` compatibility failure is already tracked by the older canonical issue, with an open fix PR attached there.

## What This Changes

The issue reports that OpenClaw's SQLite/Kysely helpers call `statement.columns()`, which breaks plugin state database and CLI startup on Node runtimes where that experimental API is absent.

## Best Possible Solution

Keep https://github.com/openclaw/openclaw/issues/90007 as the canonical tracker and resolve the open fix PR there instead of splitting the same SQLite compatibility bug across duplicate issues.

## Reproduction Assessment

Yes from source and related-thread evidence: current main still calls `statement.columns()` in both shared SQLite execution helpers, and the older canonical issue/PR documents the same failure on Node 23.0-23.10. I did not run Node 23 locally; local Node 24.16.0 exposes `columns()` and does not exercise the missing-method path.

## Solution Assessment

No as a separate tracker: the maintainable fix belongs in the shared SQLite/Kysely execution layer already tracked by the canonical issue and open PR. The PR's typed fallback direction is likely safer than a raw `sourceSQL` SELECT-prefix-only workaround because OpenClaw also needs to preserve raw row-returning and `RETURNING` behavior.
