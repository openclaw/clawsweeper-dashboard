## Summary

Keep open: current main still has the strict cache-miss path that can fall through to a standalone plugin reload, and the issue is paired with an open closing PR, so it should remain open until that fix or a replacement lands.

## What This Changes

The issue requests eliminating redundant plugin runtime registry reloads during the first fresh dispatch after gateway startup by reusing a compatible warm startup registry.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/80691 with a compatibility-aware warm-registry fast path that preserves workspace, plugin-id, config, trust, activation, and runtime-mode reload boundaries with regression coverage.

## Reproduction Assessment

Yes, source-level. Current main shows dispatch building lean runtime load options, the active lookup enforcing strict cache compatibility, and the standalone loader calling loadOpenClawPlugins after that miss; I did not run the live heap profile.

## Solution Assessment

Yes in direction, not yet on main. Reusing the warm startup registry is the maintainable fix, but it needs compatibility guards and the linked implementation must land or be replaced before this issue can close.
