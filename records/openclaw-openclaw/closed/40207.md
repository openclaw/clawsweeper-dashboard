## Summary

Keep open. Current main and the latest release still derive the macOS LaunchAgent stdout log path from the OpenClaw state directory, so the independently confirmed symlinked/external-volume launchd `EX_CONFIG` failure remains actionable.

## What This Changes

The issue requests that macOS Gateway LaunchAgent logging avoid `~/.openclaw/logs` when that state directory is symlinked or on an external volume.

## Best Possible Solution

Generate macOS LaunchAgent stdout logs from a launchd-safe user log directory outside the OpenClaw state directory, then align status output, diagnostics, docs, and focused tests with that contract.

## Reproduction Assessment

Yes, source-backed rather than live-reproduced here. The issue has concrete macOS launchd steps plus independent confirmation, and current main still writes `StandardOutPath` from `resolveGatewayStateDir(env)/logs`.

## Solution Assessment

Yes. A focused LaunchAgent stdout path change is the narrowest maintainable fix; the separate launchd hang and external-home bootstrap reports should stay on their own tracks.
