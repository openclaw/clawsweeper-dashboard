## Summary

Keep this PR open, but not merge-ready as-is. Current main still lacks the requested runtime-pair QA axis, yet the branch’s capture logic now has a concrete correctness problem from the linked follow-up audit: it treats mock-provider planned tools as runtime tool evidence, which can misclassify Codex-native workspace-tool behavior.

## What This Changes

The PR adds a private-QA Codex-vs-Pi runtime parity harness across qa-lab CLI/reporting/suite summary code, model runtime forcing, tests, multipass forwarding, and an advisory release-check workflow lane.

## Best Possible Solution

Keep the runtime-pair harness idea, but classify runtime drift from transcript/native runtime evidence and keep mock provider-plan diagnostics separate, then let maintainers choose whether this branch or the broader corrected branch should carry it.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a feature PR; the PR body supplies terminal commands showing the new CLI/report path runs, and source inspection gives a high-confidence explanation for the reported mock-layer misclassification.

## Solution Assessment

Is this the best way to solve the issue? No, not as-is: the runtime-pair direction is useful, but the implementation should not let `/debug/requests` provider-plan snapshots stand in for actual runtime tool calls.
