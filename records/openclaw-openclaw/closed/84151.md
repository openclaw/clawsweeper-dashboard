## What This Changes

The PR adds a systemd-specific update handoff through `systemd-run --user --scope`, exit-78 supervised restart handling, a generated-unit marker delay, helper recovery restart attempts, and focused tests.

## Best Possible Solution

Land the scoped systemd handoff once maintainers accept the upgrade-safety boundary, while preserving the non-systemd detached path and exit-78 compatibility for generated systemd units.

## Reproduction Assessment

Yes, at source/proof level: current main detached-spawns the helper while generated systemd units use `KillMode=control-group`, and the PR discussion includes real systemd logs showing the helper killed without a scope and surviving with one. I did not run a live systemd update in this read-only review.

## Solution Assessment

Yes at code-shape level: a transient systemd scope is the right boundary for a helper that must outlive the gateway service cgroup while preserving the non-systemd path. The remaining question is maintainer acceptance of upgrade-safety risk, especially full version-swap and custom-unit behavior.
