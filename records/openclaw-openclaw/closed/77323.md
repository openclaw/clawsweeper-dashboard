## Summary

Keep open: current main still sends the Feishu startup identity probe through the Lark SDK Axios-backed HTTP wrapper with only timeout/User-Agent handling, while managed proxy mode still injects HTTP(S)_PROXY and clears NO_PROXY. Related Feishu proxy work remains open and needs an explicit egress-policy decision before this can be safely fixed.

## What This Changes

Fix Feishu bot identity probing so it works when `proxy.enabled=true` without breaking model-provider proxy routing.

## Best Possible Solution

Implement a Feishu-owned Lark HTTP adapter that prevents Axios double proxy handling while preserving managed Proxyline egress, timeout/User-Agent behavior, target eligibility, focused tests, docs, and redacted real startup proof.

## Reproduction Assessment

Yes, at source level: enable `proxy.enabled=true` with Feishu configured, and startup probing reaches `probeFeishu` -> `createFeishuClient` -> the Lark Axios-backed HTTP instance while managed proxy env is active. I did not run live Feishu credentials through the reported local proxy.

## Solution Assessment

No for the suggested direct-bypass workaround, because it can route Feishu outside the operator-managed proxy guardrail. The safer direction is a target-aware Feishu HTTP adapter at the Lark `httpInstance` seam that prevents Axios/proxy-runtime conflicts without silently changing egress policy.
