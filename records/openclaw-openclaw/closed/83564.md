## Summary

Keep open: current main still has a source-level path where `/status` exposes a single configured `Think:` value while runtime thinking fallback/provider mapping is handled separately. The remaining blocker is product semantics: maintainers need to decide which “resolved” layer the status line should expose.

## What This Changes

The issue requests that `/status` show both the configured `adaptive` thinking default and the effective thinking level chosen for the current turn.

## Best Possible Solution

Define an effective-thinking metadata contract for `/status` that preserves the configured value and exposes a resolved canonical level or provider effort only where that layer is well-defined and tested.

## Reproduction Assessment

Yes, at source level. The `/status` command path can receive and render configured `adaptive`, while the normal reply path can later remap unsupported adaptive through provider thinking policy; I did not run a live DeepSeek turn in this read-only review.

## Solution Assessment

Unclear. Showing `adaptive -> medium` is a plausible UI, but the maintainable fix needs a maintainer choice between canonical fallback, provider payload effort, or a paired display that names both layers.
