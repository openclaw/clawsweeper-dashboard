## Summary

Keep open. Current `main` only sends native Feishu audio bubbles for Opus/Ogg, while common TTS outputs like MP3 still fall back to generic file attachments. That means the broader request in #37868 is not implemented on `main`, and the latest maintainer discussion already says it is only partially addressed.

## Evidence

- **Current routing only promotes Opus/Ogg to Feishu audio:** `resolveFeishuOutboundMediaKind` maps only `.opus`/`.ogg` and `audio/ogg`/`audio/opus` to `msgType: "audio"`; everything else falls through to generic file-type detection.
  - file: [extensions/feishu/src/media.ts:524](https://github.com/openclaw/openclaw/blob/dfa52aaab0f4ca5a9416c1eea21b20b63631e85f/extensions/feishu/src/media.ts#L524)
  - command: `nl -ba extensions/feishu/src/media.ts | sed -n '470,700p'`
  - sha: [dfa52aaab0f4](https://github.com/openclaw/openclaw/commit/dfa52aaab0f4ca5a9416c1eea21b20b63631e85f)
- **Filename-based detection does not include MP3/WAV/M4A/AAC/FLAC/WMA:** `detectFileType` recognizes only `.opus`/`.ogg` as audio. Common TTS extensions requested in the issue are absent, so buffered sends with names like `voice.mp3` still resolve to `stream`/generic file behavior.
  - file: [extensions/feishu/src/media.ts:496](https://github.com/openclaw/openclaw/blob/dfa52aaab0f4ca5a9416c1eea21b20b63631e85f/extensions/feishu/src/media.ts#L496)
  - command: `nl -ba extensions/feishu/src/media.ts | sed -n '470,700p'`
  - sha: [dfa52aaab0f4](https://github.com/openclaw/openclaw/commit/dfa52aaab0f4ca5a9416c1eea21b20b63631e85f)
- **Tests lock in MP3 fallback as current behavior:** `media.test.ts` has an explicit test, `falls back to generic file for unsupported audio formats`, that feeds `song.mp3` with `audio/mpeg` and expects `file_type: "stream"` and `msg_type: "file"`. That is the opposite of the requested voice-bubble behavior for common TTS outputs.
  - file: [extensions/feishu/src/media.test.ts:239](https://github.com/openclaw/openclaw/blob/dfa52aaab0f4ca5a9416c1eea21b20b63631e85f/extensions/feishu/src/media.test.ts#L239)

- **Issue discussion already converged on partial, not complete, implementation:** The issue comments include proposed PRs #37899 and #37951 for broader format support, but the latest maintainer update on 2026-04-25 says current main only covers the narrow Opus/Ogg path and keeps this issue open until MP3/WAV/M4A-style TTS output is handled.
