## Summary

Current `main` already routes `openclaw browser` through plugin-registered gateway handlers, and the shipped `v2026.4.23` release contains that implementation plus integration coverage for `browser.request`.

## Evidence

- **Browser plugin now registers the missing gateway method:** `registerBrowserPlugin` registers the CLI, `browser.request`, and the browser-control service together via the plugin API.
  - file: [extensions/browser/plugin-registration.ts:28](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/extensions/browser/plugin-registration.ts#L28)
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)
- **Plugin gateway handlers are loaded into the dispatcher:** Gateway startup collects plugin-registered handlers into `pluginRegistry.gatewayHandlers`, then passes them into the websocket dispatcher as `extraHandlers`, which is exactly where `browser.request` is resolved.
  - file: [src/gateway/server.impl.ts:804](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/src/gateway/server.impl.ts#L804)
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)
- **Plugin loader exposes registered gateway methods:** `loadGatewayPlugins` builds the active method list from `pluginRegistry.gatewayHandlers`, so `browser.request` is part of the advertised/active gateway surface when the browser plugin is loaded.
  - file: [src/gateway/server-plugins.ts:493](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/src/gateway/server-plugins.ts#L493)
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)
- **Regression coverage exists for this exact path:** Integration test asserts that loading the bundled browser plugin adds `browser.request` and the `browser-control` service.
  - file: [src/gateway/server-plugin-bootstrap.browser-plugin.integration.test.ts:42](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/src/gateway/server-plugin-bootstrap.browser-plugin.integration.test.ts#L42)
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)
- **Docs reflect current expected behavior:** Current docs describe missing `browser.request` as a plugin allowlist/config problem, not an unregistered handler bug, and explicitly state the browser plugin owns the CLI, gateway method, tool, and service together.
  - file: [docs/tools/browser.md:62](https://github.com/openclaw/openclaw/blob/90877e0d4274c53fd91bf5787265c4d6889dc9a7/docs/tools/browser.md#L62)
  - sha: [90877e0d4274](https://github.com/openclaw/openclaw/commit/90877e0d4274c53fd91bf5787265c4d6889dc9a7)
- **Shipped release contains the implementation:** The tagged release `v2026.4.23` resolves to commit `a9797214338ba31c52c796adbb75afb16e0684a9`, and `git show` on that tag includes the browser plugin registration path and the browser gateway bootstrap integration test.
