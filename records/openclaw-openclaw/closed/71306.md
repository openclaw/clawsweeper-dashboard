## Summary

Current `main` already implements duplicate-delivery protection for this report: Discord inbound handling claims a per-message replay key before preflight, and the follow-up queue now refuses to enqueue the same `messageId` twice for the same routing key, with regression tests covering both duplicate inbound delivery and queued redelivery after drain restart. The fix appears newer than the latest tagged release, so it is implemented on `main` but no shipped release could be confirmed from the available evidence.

## Evidence

- **Follow-up queue now deduplicates queued messages by `messageId` plus routing:** `enqueueFollowupRun()` builds a recent-message cache key from queue key, originating channel/destination/account/thread, and `messageId`, rejects entries already seen in the recent cache, and also skips duplicates already present in the live queue.

  - file: [src/auto-reply/reply/queue.dedupe.test.ts:48](https://github.com/openclaw/openclaw/blob/28eb56dd21b61631ba916ea3dd0fc2afa002f31d/src/auto-reply/reply/queue.dedupe.test.ts#L48)
