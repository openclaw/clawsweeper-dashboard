## Summary

Keep open. Current main and the latest release still reject `#` in exec SecretRef IDs before the exec resolver can run, and the issue is already tied to an open closing PR that should be reviewed or landed before this issue closes.

## What This Changes

Allow documented AWS-style exec SecretRef IDs such as `secret-name#json-key` to validate and reach the configured exec resolver during gateway startup.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/80731 or an equivalent shared SecretRef contract fix, then close this issue from the merged commit after validation.

## Reproduction Assessment

Yes, at source level. A hash-containing exec SecretRef ID fails the shared regex in `isValidExecSecretRefId`, and `resolveSecretRefValues` throws before the exec provider receives the ID.

## Solution Assessment

Yes. The narrow maintainable solution is to update the shared exec SecretRef ID contract, schema/docs, and parity vectors together while preserving traversal-segment rejection.
