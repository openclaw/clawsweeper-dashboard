## Summary

Keep open. Current `main` still has a plausible duplicate-delivery path for Feishu card streaming failures: when `streaming.start()` fails, the dispatcher clears the streaming state and then immediately falls through to the normal card/text send path without tracking that streaming was already attempted. The current tests cover retry-after-failure, but they do not prove duplicate suppression for that path.

## Evidence

- **Dispatcher falls back after failed streaming start:** `extensions/feishu/src/reply-dispatcher.ts:268` starts streaming and, on error at `:294-297`, resets `streaming`/`streamingStartPromise` to `null`. In `deliver()` at `:391-448`, block/final card replies await that start attempt and then, if `streaming?.isActive()` is false, fall through to `sendStructuredCardFeishu(...)` / `sendMessageFeishu(...)`. There is no separate "streaming was attempted" guard.
  - file: [extensions/feishu/src/reply-dispatcher.ts:268](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/extensions/feishu/src/reply-dispatcher.ts#L268)
  - command: `nl -ba extensions/feishu/src/reply-dispatcher.ts | sed -n '260,470p'`
  - sha: [4013c658537e](https://github.com/openclaw/openclaw/commit/4013c658537ee2051c44186f197b0710e84c9049)
- **Feishu streaming start performs remote card send inside the start call:** `extensions/feishu/src/streaming-card.ts:217-294` creates the CardKit card and then sends the interactive message before marking local state active. That means the dispatcher fallback decision happens after an async start path that already performed outbound side effects, but the dispatcher only distinguishes `isActive()` vs not-active.
  - file: [extensions/feishu/src/streaming-card.ts:217](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/extensions/feishu/src/streaming-card.ts#L217)
  - command: `nl -ba extensions/feishu/src/streaming-card.ts | sed -n '150,360p'`
  - sha: [4013c658537e](https://github.com/openclaw/openclaw/commit/4013c658537ee2051c44186f197b0710e84c9049)
- **Test coverage checks recovery, not duplicate suppression:** `extensions/feishu/src/reply-dispatcher.test.ts:739-753` verifies that a failed `start()` can recover on the next delivery, but it does not assert that the failed delivery avoided sending a fallback card/message. So current tests do not demonstrate that the reported duplicate path is fixed.
  - file: [extensions/feishu/src/reply-dispatcher.test.ts:739](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/extensions/feishu/src/reply-dispatcher.test.ts#L739)
