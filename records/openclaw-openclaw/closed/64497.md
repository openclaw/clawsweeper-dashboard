## Summary

Keep open. Current `main` still has the same timeout/abort gap around isolated cron preparation, and there is no strong evidence that the inactivity-correlated spawn failure reported in #64497 was fixed on `main` or in `v2026.4.23`. The report is recent, specific, and supported by issue discussion that narrowed the likely failure area without resolving it.

## Evidence

- **Cron timeout is still an outer `Promise.race`:** `executeJobCoreWithTimeout` aborts a controller and rejects on timeout, but it only interrupts work that actually observes the abort signal.
  - file: [src/cron/service/timer.ts:91](https://github.com/openclaw/openclaw/blob/11804a484ded89374afa55020813cfa265aa143a/src/cron/service/timer.ts#L91)

- **Isolated cron preparation still does multiple awaited setup steps before execution:** `prepareCronRunContext` performs workspace setup, delivery resolution, skills snapshot work, session-store persistence, and auth-profile resolution in sequence, with no abort checks in that preparation path.
  - file: [src/cron/isolated-agent/run.ts:422](https://github.com/openclaw/openclaw/blob/11804a484ded89374afa55020813cfa265aa143a/src/cron/isolated-agent/run.ts#L422)
