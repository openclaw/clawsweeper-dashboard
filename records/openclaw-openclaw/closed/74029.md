## Summary

Keep this PR open. The underlying #73963 report is valid against current main, but the proposed builder-only metadata strip is not enough: current main still re-merges provider turn metadata immediately before `client.responses.create(...)`, so the live `openai-codex-responses` payload would still carry top-level `metadata` unless the final send boundary or OpenAI provider turn-state policy is amended.

## Best Possible Solution

Keep this PR open and request an amendment: ensure the final payload sent to `client.responses.create(...)` cannot contain top-level `metadata` for `openai-codex-responses`, while preserving metadata for regular `openai-responses` and `azure-openai-responses`. The clean fix likely belongs at the final `mergeTransportMetadata` call site or in the OpenAI provider turn-state/payload policy, with a regression test that exercises the actual final SDK payload boundary and uses the documented `openai-codex/gpt-5.5` route.
