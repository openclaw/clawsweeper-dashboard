## Summary

Current main shows `cron.maxConcurrentRuns` only limits the number of due jobs a single CronService instance runs in parallel. The reported 4-5 duplicate runs for one job are better explained by multiple gateway processes sharing the same cron store/state dir, which current docs explicitly say must be isolated per gateway instance.

## Evidence

- **Concurrency setting scope in runtime:** `cron.maxConcurrentRuns` is read in the timer and used only to size the worker pool for the current batch of due jobs in one scheduler instance (`resolveRunConcurrency`, then `concurrency = Math.min(...)`).
  - file: [src/cron/service/timer.ts:107](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/cron/service/timer.ts#L107)

- **Main-branch regression coverage:** Current tests explicitly assert that `maxConcurrentRuns` is honored for two simultaneous due jobs.
  - file: [src/cron/service/timer.regression.test.ts:887](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/cron/service/timer.regression.test.ts#L887)

  - file: [src/config/schema.help.ts:1379](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/config/schema.help.ts#L1379)

- **Default cron store is shared unless state dir/store is isolated:** If `cron.store` is unset, the cron store defaults to `resolveConfigDir()/cron/jobs.json`; multiple gateways that share the same state dir will share that cron store.
  - file: [src/cron/store.ts:26](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/cron/store.ts#L26)

- **Locking is process-local, not cross-process:** The cron lock is an in-memory `Map` keyed by `storePath`, so it serializes operations only inside one process. Separate gateway processes sharing the same store can each run the same due job.
