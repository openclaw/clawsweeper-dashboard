## Summary

Close as implemented on current main. The literal Telegram placeholder leak reported against 2026.4.26 is covered by the merged sanitizer fix from #74578: standalone `[tool calls omitted]` lines are removed before user-facing delivery, regression tests cover placeholder-only and mixed-text cases, and Telegram has a no-visible-response fallback when nothing sendable remains. The fix is newer than v2026.4.27, so I cannot identify a shipped release containing it yet.

## What This Changes

The issue reports that Telegram displays the internal `[tool calls omitted]` placeholder after tool-call turns and asks for a clearer user-facing result when the agent stalls.

## Best Possible Solution

Keep the shared sanitizer fix on main as the terminal fix for the literal placeholder leak, and route any broader “text between tool calls leaks to channels” behavior to #25592 rather than keeping this version-specific Telegram placeholder report open.

## Reproduction Assessment

Yes for the observable leak path at source level: the latest release tag lacks placeholder stripping, while current main and #74578 regression tests show `[tool calls omitted]` is removed before user-facing output. I did not live-run Telegram with the reported Ollama/MiniMax setup because this review was read-only.

## Solution Assessment

Yes for this issue: stripping standalone internal replay placeholders in the shared user-facing sanitizer is the narrowest maintainable fix because it protects Telegram and other channels without changing transcript repair. If a model still stalls without producing visible content, that is a separate runtime/provider behavior rather than this literal placeholder leak.
