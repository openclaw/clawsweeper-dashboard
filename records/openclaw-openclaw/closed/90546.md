## Summary

Current main and v2026.6.1 already make gateway startup and hot reload fail or skip invalid config instead of restoring last-known-good; the remaining last-known-good restore path is doctor repair, with audit metadata recorded.

## What This Changes

The issue requests better observability and safeguards when config recovery restores `openclaw.json.last-good` over a newly written config after restart.

## Best Possible Solution

Keep the shipped fail-closed gateway behavior and doctor-owned restore path; a new report should identify the exact command path if a restart still rewrites config automatically.

## Reproduction Assessment

No. The synthetic gateway restart path does not reproduce in current source: startup rejects invalid snapshots and hot reload skips invalid snapshots instead of restoring last-known-good.

## Solution Assessment

Yes. The current design is the narrow, maintainable fix: gateway startup/reload do not rewrite invalid config, while explicit `openclaw doctor --fix` owns last-known-good repair and audit recording.
