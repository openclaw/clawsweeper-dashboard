## Summary

Keep open: the report is fresh, concrete, and current main still advertises iMessage final media delivery while routing that media through the generic RPC `send` path instead of the bridge-backed `send-rich --file` route that the reporter proved can deliver the same file to the group.

## What This Changes

The issue reports that generated image replies in allowlisted iMessage groups create a local PNG but do not arrive through the normal OpenClaw final-reply media path, while manual `imsg send-rich --file` to the group succeeds.

## Best Possible Solution

Fix the iMessage final media path so group media replies use a verified group-capable bridge route or emit a clear retryable failure, while preserving resolver, size, and capability checks.

## Reproduction Assessment

No high-confidence live reproduction was performed in this review. The issue has repeated reporter live evidence plus a current-main source path showing final media still goes through generic RPC `send` while the known-good rich-send route is separate, so it is source-reproducible enough to keep open.

## Solution Assessment

Yes, a narrow iMessage outbound-media fix is the right direction. The safest implementation should reuse the existing media resolver and private-API capability checks rather than adding a raw filesystem send shortcut.
