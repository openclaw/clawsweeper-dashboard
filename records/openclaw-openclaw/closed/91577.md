## Label Justifications

- `P1`: The report describes a broken subagent/session workflow with no workaround on affected releases.
- `impact:session-state`: The failure prevents creation of child subagent sessions and leaves all work in the parent session.
- `impact:auth-provider`: The observable failure is a gateway auth/pairing scope-upgrade denial on loopback internal RPCs.

## Summary

Current main already fixes the reported loopback `sessions_spawn` pairing failure by keeping internal subagent gateway coordination inside the gateway request scope and by preserving backend loopback shared-auth behavior; the strongest proof commit is on main only and is not contained in a release tag yet.

## What This Changes

The issue reports that `sessions_spawn` with `runtime="subagent"` fails on older releases with WebSocket close 1008 `pairing required` during a loopback gateway scope upgrade.

## Best Possible Solution

Keep the current main owner-boundary fix and ship it in the next release so old v2026.3.x and v2026.4.x installs stop hitting the loopback pairing regression after upgrade.

## Reproduction Assessment

Yes for the affected old-version path: the report and related reports all describe `sessions_spawn runtime="subagent"` failing with loopback close 1008 during a scope upgrade, and current source comments preserve that exact failure mode. I did not run an old release live repro; current main should not reproduce because it now dispatches internally instead of depending on the old loopback pairing path.

## Solution Assessment

Yes. The current implementation fixes the owner boundary by avoiding or pre-scoping internal subagent gateway coordination, rather than asking users to repair pairing state for an internal runtime call.
