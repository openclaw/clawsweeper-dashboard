## Summary

Current main and the shipped v2026.4.23 release already decouple managed dreaming from heartbeat active hours: new and reconciled dreaming jobs run as isolated lightweight agent turns, and stale main-session jobs are migrated by runtime reconciliation or doctor repair.

## What This Changes

The issue asks for dreaming to stop being silently skipped by `heartbeat.activeHours`, or for docs/defaults to stop implying overnight dreaming works when it does not.

## Best Possible Solution

Keep the shipped isolated-cron implementation and use a narrower follow-up only if maintainers want to remove stale dreaming troubleshooting text that still mentions heartbeat-blocked status.

## Reproduction Assessment

No current-main failure path remains: source inspection shows managed dreaming jobs are isolated agent turns and stale persisted jobs are rewritten, so the heartbeat `activeHours` gate no longer applies. I did not run a live gateway scenario because this review was required to keep the checkout read-only.

## Solution Assessment

Yes. Decoupling dreaming from heartbeat is the narrowest maintainable fix for a background maintenance sweep and avoids adding a new user-facing active-hours override solely to bypass the old coupling.
