## Summary

Keep open: this recent report is concrete, matches current Codex OAuth forced-refresh source paths, and the duplicate evidence in https://github.com/openclaw/openclaw/issues/81944 reinforces the same runtime-vs-probe split. Current main still preserves hard refresh behavior rather than proving the fresh OAuthRef `token_expired` path is fixed.

## What This Changes

Reports that OpenAI Codex OAuth profiles that look healthy in `models status --probe` fail Codex harness and PI runtime turns with 401 `token_expired` during forced token refresh.

## Best Possible Solution

Make one validated OAuth access-token versus refresh-token policy shared by Codex app-server and PI routes, preserving OAuthRef storage and covering it with regression tests plus redacted live proof.

## Reproduction Assessment

No live high-confidence reproduction was established in this read-only review. Source inspection and two independent deployment reports give a credible reproduction path, but a real OAuthRef-backed `openclaw agent --agent main --message 'Reply exactly: OK' --timeout 120` run is still needed for high-confidence live proof.

## Solution Assessment

Unclear. Reusing a still-valid access token is a plausible narrow fix and aligns with https://github.com/openclaw/openclaw/pull/73807, but https://github.com/openclaw/openclaw/pull/80738 deliberately preserved hard refresh semantics, so the fix should be confirmed against the Codex app-server and PI auth contracts first.
