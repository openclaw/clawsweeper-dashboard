## Summary

Keep open for maintainer review: the PR targets a source-visible QMD scheduler gap, but the current diff treats the resolved default 60-minute embed interval as an explicit opt-in and conflicts with current BM25-only `searchMode: "search"` docs/tests. The contributor also has not provided after-fix real OpenClaw/QMD behavior proof.

## What This Changes

The PR changes QMD embed run/timer guards and QMD manager tests so a positive `embedIntervalMs` schedules `qmd update` and `qmd embed` even when `searchMode` is lexical `search`.

## Best Possible Solution

Carry explicit `update.embedInterval` intent through resolved config and align docs/tests if maintainers want lexical `search` to opt into embedding; otherwise preserve BM25-only search mode and direct users to `query` or `vsearch` for automatic embedding maintenance.

## Reproduction Assessment

Yes, source-level. Current main resolves omitted QMD search mode to lexical `search`, then both embed execution and timer scheduling return before a positive `embedIntervalMs` can matter.

## Solution Assessment

No. The proposed guard uses resolved `embedIntervalMs`, which defaults to 60 minutes and cannot distinguish omitted config from an explicit user interval; the safer path is an explicit-config signal plus maintainer-confirmed docs/test semantics.
