## Summary

Keep open. The PR targets a real systemd watchdog gap, but the heartbeat path is incorrect because it calls an unsupported `systemd-notify --watchdog`, and the supplied proof does not exercise Linux systemd receiving watchdog pings.

## What This Changes

Adds generated and documented systemd watchdog directives, a `systemd-notify` helper wired to gateway maintenance ticks, tests, docs, and a changelog entry.

## Best Possible Solution

Use the supported `WATCHDOG=1` notification payload, keep the generated unit directives aligned with the gateway-owned systemd contract, and prove the heartbeat on a Linux systemd user service before merge.

## Reproduction Assessment

Yes. Source inspection shows current main lacks the watchdog directives/notifier, and local systemd 255 rejects the PR's `systemd-notify --watchdog`; I did not live-freeze a gateway.

## Solution Assessment

No as written. Reusing the maintenance tick is a plausible narrow design, but the implementation must send `WATCHDOG=1` and be validated in a real Linux systemd service.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Send watchdog pings with WATCHDOG=1:** `src/infra/systemd-notify.ts:79`
  - body: The heartbeat path invokes `systemd-notify --watchdog`, but systemd-notify 255 and its man page do not define that option; locally it exits with `unrecognized option '--watchdog'`. With `NOTIFY_SOCKET` set, the catch disables the notifier after the first ping, so a `WatchdogSec` unit never receives keepalives. Send `WATCHDOG=1` instead and update the test/proof.
  - confidence: 0.96
