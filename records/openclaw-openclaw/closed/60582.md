## Summary

Keep open. Current main still preserves Discord channel `parentId` metadata, and the allowlist resolver can use a parent fallback, but normal message preflight only supplies thread-parent metadata, so ordinary text channels under a Discord category do not inherit category allowlist settings. Related #8972 overlaps the dynamic-channel use case but is a broader pattern/glob request, not a canonical replacement for category inheritance.

## What This Changes

The issue requests Discord guild allowlist entries that apply to every child channel in a Discord category, so new channels inherit settings such as `requireMention: false` without config edits or gateway restarts.

## Best Possible Solution

Implement the behavior inside the Discord plugin by feeding non-thread guild channel parent metadata into the existing fallback resolver, preserving direct channel overrides over category defaults and wildcard fallback, then document the chosen category config syntax and add focused preflight/schema/docs coverage.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a guild text channel whose fetched `channelInfo.parentId` matches a configured category entry while the direct channel is absent; current message preflight never passes that non-thread parent id into allowlist resolution, so the category config is not applied.

## Solution Assessment

Unclear. Reusing the existing parent-fallback resolver is likely the narrowest implementation, but maintainers should first choose the public category key syntax and confirm how this relates to #8972.
