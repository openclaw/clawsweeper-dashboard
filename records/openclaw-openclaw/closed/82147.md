## Summary

Keep open: current main still reports a successful WebChat/TUI current-run internal-source `message` send, but the successful tool result is not collected into the existing source-reply mirror path that would render it back to the visible client.

## What This Changes

The issue reports that `message.send` to the active WebChat/TUI current run reports success even though the message is not rendered in the visible client.

## Best Possible Solution

Wire successful `internal-ui` source replies from the message tool into the existing `messagingToolSourceReplyPayloads` final-payload/transcript-mirror path, preserving external channel delivery and keeping WebChat/TUI out of generic outbound routing.

## Reproduction Assessment

Yes at source level: current main documents the visible current-run WebChat path and returns a successful internal-source send, while the subscriber/attempt path does not populate the payload array that the WebChat projection uses. A live WebChat/TUI run was not executed in this read-only review.

## Solution Assessment

Yes: completing the existing internal-source reply plumbing is the narrow maintainable fix. If that path cannot be supported, the send should fail before reporting successful visible delivery rather than adding a config knob or treating WebChat as a generic outbound channel.
