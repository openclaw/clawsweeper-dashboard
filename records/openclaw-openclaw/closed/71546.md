## Summary

Close as implemented on current main. The reported lag was tied to Carbon reconnect/RESUME buffering, and current main removes Carbon from the Discord plugin in favor of an internal gateway that owns resume state and dispatches gateway MESSAGE_CREATE events directly through OpenClaw listeners.

## What This Changes

The issue requests investigation and repair for 100-400 second Discord message ingest lag during Carbon reconnect/RESUME windows, with a possible reconnect catch-up/backfill backstop.

## Best Possible Solution

Keep the internal Discord gateway implementation as the current fix, and reopen a narrower follow-up only if fresh logs on current main show delayed MESSAGE_CREATE dispatch after RESUMED or demonstrate a remaining need for bounded REST catch-up.
