## Close Comment

Closing this as implemented after Codex review.

Current `main` already avoids sending unsupported disabled OpenAI reasoning payloads, which covers the reported `openai/gpt-5.4-pro` `thinking=none` 400 and the fallback-summary variant that depended on the same request shape.

What I checked:

- **Model-specific OpenAI reasoning resolution:** `buildOpenAIResponsesParams()` resolves reasoning effort per model and returns `undefined` for unsupported `none`, so unsupported models no longer receive `reasoning.effort: "none"` at all. (`src/agents/openai-transport-stream.ts:877`)
- **Responses payload policy strips disabled reasoning:** The Responses payload policy explicitly deletes `reasoning` when the payload carries disabled reasoning on models/routes that do not support `none`. (`src/agents/openai-responses-payload-policy.ts:79`)
- **Embedded runner path uses the same policy:** Pi embedded OpenAI stream wrappers apply the Responses payload policy on outgoing payloads, so the interactive embedded-runner path from the report is covered too. (`src/agents/pi-embedded-runner/openai-stream-wrappers.ts:172`)
- **Regression coverage and release note:** Tests assert that unsupported-model disabled reasoning is omitted, and the changelog records the GPT-5 fix: “omit disabled reasoning payloads ... so GPT-5 endpoints no longer reject `reasoning.effort: "none"`.”. (`CHANGELOG.md:1899`)

So I’m closing this as already implemented rather than keeping a duplicate issue open.
