## Summary

Close as superseded by the already-linked open PR #49434, which explicitly tracks the same Discord multi-bot thread-bound subagent ownership bug and proposes the remaining fix. Current main still shows account-scoped binding resolution, so this is not implemented on main; the PR is the canonical open work item.

## Evidence

- **Canonical PR already linked:** The issue timeline cross-references PR #49434, titled "fix(discord): preserve bound thread isolation across bot accounts". Its body says it closes #49373 and describes the same fix: suppress generic routing when another Discord account already owns an active thread binding, while preserving same-account behavior.
- **Discord preflight resolves runtime bindings only for the current account:** Preflight calls `resolveRuntimeConversationBindingRoute` with `conversation.channel = "discord"`, `accountId: params.accountId`, and `conversationId` for the current inbound message. There is no cross-account ownership check before continuing normal route resolution.
  - file: [extensions/discord/src/monitor/message-handler.preflight.ts:646](https://github.com/openclaw/openclaw/blob/a1090b604312ac7a49cf2d2ce7276b334786bb79/extensions/discord/src/monitor/message-handler.preflight.ts#L646)
  - command: `nl -ba extensions/discord/src/monitor/message-handler.preflight.ts | sed -n '620,700p'`
  - sha: [a1090b604312](https://github.com/openclaw/openclaw/commit/a1090b604312ac7a49cf2d2ce7276b334786bb79)
- **Session binding adapters remain keyed by channel and account:** The session binding registry stores adapters by a `channel:accountId` key and `resolveByConversation` picks the adapter for the normalized conversation's account. That matches the issue's account-scoped root cause.
  - file: [src/infra/outbound/session-binding-service.ts:132](https://github.com/openclaw/openclaw/blob/a1090b604312ac7a49cf2d2ce7276b334786bb79/src/infra/outbound/session-binding-service.ts#L132)
  - command: `nl -ba src/infra/outbound/session-binding-service.ts | sed -n '1,260p' && nl -ba src/infra/outbound/session-binding-service.ts | sed -n '260,440p'`
  - sha: [a1090b604312](https://github.com/openclaw/openclaw/commit/a1090b604312ac7a49cf2d2ce7276b334786bb79)
- **Discord thread manager lookup is account-scoped:** `getByThreadId` builds the binding key from the manager's `accountId` plus `threadId`, then rejects records whose `accountId` differs. A different bot account therefore cannot see this manager's binding through the normal lookup path.
  - file: [extensions/discord/src/monitor/thread-bindings.manager.ts:329](https://github.com/openclaw/openclaw/blob/a1090b604312ac7a49cf2d2ce7276b334786bb79/extensions/discord/src/monitor/thread-bindings.manager.ts#L329)

- **Prior discussion supports the same diagnosis:** Ryce's issue comment describes the same ownership-boundary failure: Bot A's adapter knows the thread binding, Bot B's adapter does not, and Bot B falls back to standard routing. That comment aligns with PR #49434 as the canonical follow-up.
