## Summary

Keep open: the PR fixes the reported callback-button path and now includes redacted runtime proof, but it should not merge until disabled URL/link buttons either serialize disabled or continue degrading safely; otherwise the new Discord capability can turn a disabled link action into an enabled clickable component.

## What This Changes

The PR adds `disabled` to the message tool presentation button schema, advertises Discord disabled-button support, propagates `disabled: true` through Discord presentation/interactive specs, and adds Discord regression tests.

## Best Possible Solution

Keep the schema, mapper, and capability direction, but make final Discord component building preserve disabled state for both callback and URL/link buttons, or explicitly keep unsupported disabled link controls downgraded to non-clickable fallback text with tests.

## Reproduction Assessment

Yes. Source inspection shows current main drops disabled buttons unless `supportsDisabled` is advertised, and the PR advertises that support while the final Discord link-button builder still does not apply `params.spec.disabled`.

## Solution Assessment

No, not yet. The approach is the right narrow fix for callback buttons, but it over-advertises Discord support until URL/link buttons either serialize disabled or are deliberately kept as fallback text.
