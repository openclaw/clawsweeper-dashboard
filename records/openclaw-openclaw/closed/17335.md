## Summary

Close as `clawhub`. This is a legitimate observability/security idea, but current main treats it as third-party plugin territory rather than bundled OpenClaw scope, and the repo already exposes the plugin seams needed to build it externally.

## Close Comment

Codex review: closing this as `clawhub` rather than as core/bundled-plugin work. Current project direction is to keep optional capability out of core when it can live as a plugin: `VISION.md:52` says optional capability should usually ship as plugins, be hosted in the plugin author’s own repo, and faces a high bar for inclusion in core. The plugin docs also say external plugins should be published to ClawHub or npm instead of being added to this repository (`docs/plugins/building-plugins.md:16`), and they already expose the seams needed here: hook/service plugins are first-class (`docs/plugins/building-plugins.md:36`, `:166`) and `onDiagnosticEvent` is exported from the public SDK (`src/plugin-sdk/index.ts:97`). Meanwhile the bundled `diagnostics-otel` plugin is still intentionally narrow: it is described as "Export diagnostics events to OpenTelemetry" (`extensions/diagnostics-otel/index.ts:7`), and current main only exposes OTLP transport settings under `diagnostics.otel` with no built-in `security` config surface (`src/config/zod-schema.ts:284`). Given that scope, this is better maintained as a ClawHub/community plugin, especially since the issue already points to an external implementation.
