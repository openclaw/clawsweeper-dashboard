## Summary

Current main removes the native Telegram draft preview transport that could route generated Unicode through a ByteString-only boundary; Telegram preview/final delivery now stays on durable message send/edit paths with normal HTML/plain-text fallback. The fix is on main after the latest stable release, so users on v2026.4.27 still need a newer build or the next release.

## What This Changes

The issue reports Telegram DM replies failing when an embedded-agent response contains the Unicode black diamond character.

## Best Possible Solution

Keep the current durable message-edit preview implementation and ship it in the next release; do not strip Unicode from Telegram replies because normal Telegram message bodies should carry Unicode safely.

## Reproduction Assessment

Yes for the underlying failure shape: the reported U+25C6 character reproduces the exact ByteString error when passed to a byte-only Fetch field. The current-main Telegram path no longer sends generated preview text through the removed native draft transport that matched that boundary.

## Solution Assessment

Yes. Removing the native draft preview transport is narrower and more maintainable than sanitizing agent text, because it keeps user-visible Unicode intact while using Telegram’s standard message-body send/edit APIs.
