## Summary

Keep open; this is a concrete CLI/Gateway auth recovery bug, and the self-removal plus local fallback plus narrow re-pair path is distinct enough from https://github.com/openclaw/openclaw/issues/74484 that it should not be swept as a duplicate yet.

## What This Changes

The issue requests reliable CLI re-approval after removing the only paired CLI device, without racing rotating pending request IDs or accidentally downgrading operator scopes.

## Best Possible Solution

Define one trusted recovery contract for self-removal, likely explicit shared-token or bootstrap approval of the current pending CLI request, and preserve the intended operator scope baseline with regression tests and CLI/docs guidance.

## Reproduction Assessment

Yes, there is a source-level reproduction path: remove the paired CLI device, then let bare `devices list` / `devices approve` reconnect through pairing-required fallback with method-scoped pending requests. I did not execute the full live gateway flow, so this is source-reproducible rather than locally reproduced.

## Solution Assessment

No; an automated local patch is not the best next step because the fix must decide what authority can safely re-pair an operator device after self-removal and which scopes it may mint. Owner review should pick the recovery contract before code changes.
