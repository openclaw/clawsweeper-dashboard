## Summary

Keep open: this PR looks like a narrow, correct fix with supplied real behavior proof and green checks, but it is still an unmerged PR that needs maintainer acceptance of the session-history bearer trust boundary. Current main still rejects the reported bearer-token history path because the handler uses the strict HTTP scope resolver for both REST and SSE reauthorization.

## What This Changes

The PR switches `/sessions/:key/history` REST and SSE authorization to the OpenAI-compatible shared-secret scope resolver and adds bearer-token REST/SSE regression tests.

## Best Possible Solution

Land this narrow resolver/test change if maintainers accept session history as a shared-secret full-operator HTTP surface, then coordinate separately with https://github.com/openclaw/openclaw/pull/55928 for internal-session HTTP blocking.

## Reproduction Assessment

Yes, from source inspection. On current main, shared-secret bearer auth resolves to untrusted declared scopes, session-history uses the strict resolver, and `chat.history` authorization therefore lacks `operator.read`; I did not run local tests because this review must keep the checkout read-only.

## Solution Assessment

Yes as a code fix, subject to maintainer trust-boundary approval. Switching both the initial request and SSE reauth path to the same compat resolver is the narrowest maintainable fix; the safer landing path is to preserve the separate internal-session HTTP guard if that hardening is accepted.
