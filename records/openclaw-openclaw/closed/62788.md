## Summary

Keep open. Current `main` still does not transcribe inbound WhatsApp voice notes before agent handoff, so the user-facing gap remains real. The PR is directionally aligned with existing Telegram/Discord preflight behavior, and the discussion convincingly disposes of the media-TTL and default-format bot nits, but the head commit still has a real blocker: transcript echoes are sent with an untracked `sock.sendMessage(...)`, bypassing WhatsApp's outbound-echo suppression and risking self-reprocessing loops.

## Evidence

- **Current WhatsApp inbound path still leaves audio as media-only context:** On `main`, `enrichInboundMessage()` downloads and saves inbound media, then immediately returns the extracted body/media tuple. There is no WhatsApp call to `transcribeFirstAudio()` here, so audio-only inbound messages still stay as `<media:audio>` rather than injected transcript text.
