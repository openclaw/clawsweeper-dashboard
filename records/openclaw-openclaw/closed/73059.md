## Summary

Keep open. Current main still uses a raw `process.argv[1]` path comparison in `scripts/ui.js`, while gateway first-start Control UI building still routes through that wrapper, so the Windows junction-launched no-op remains source-backed and not superseded by the adjacent Windows `pnpm.cmd` fix.

## What This Changes

The issue asks OpenClaw to run the Control UI build wrapper correctly when `scripts/ui.js` is launched through a Windows global npm junction.

## Best Possible Solution

Canonicalize both the wrapper module path and `argv[1]` in `scripts/ui.js`, add a linked-entrypoint regression test, and preserve the existing Windows cmd.exe launcher safeguards.

## Reproduction Assessment

Yes, source-backed rather than live-run in this review. The reporter supplied concrete Windows junction steps, and current main still has the raw comparison that would skip `main()` when the junction path and real repo path differ.

## Solution Assessment

Yes. Fixing the direct-execution guard in `scripts/ui.js` is the narrowest maintainable solution because both direct `node ...\scripts\ui.js build` usage and gateway first-start auto-build route through that wrapper.
