## Summary

Keep open. Current main still has a credible managed web_search stale-config path: the runtime swaps to the resolved snapshot only for missing/runtime/matching-source configs, while managed tool wrappers can keep passing captured config and runtime metadata into Gemini provider creation, where a raw plugin-scoped SecretRef still reaches the strict SecretRef reader.

## What This Changes

Ensure managed Gemini web_search execution uses resolved active runtime snapshot credentials rather than raw plugin-scoped SecretRefs after upgrade or secrets reload.

## Best Possible Solution

Managed gateway/session-owned web_search execution should late-bind the active resolved runtime config and current runtime web-tool metadata, while explicit SDK/plugin custom config continues to use caller-supplied config. Add a regression test for an existing managed wrapper holding a raw Gemini plugin SecretRef after a resolved runtime snapshot or reload is active.

## Reproduction Assessment

Yes. The issue gives concrete config and error output, and current main gives a static reproduction path: execute managed web_search from a wrapper holding stale raw Gemini plugin config that does not match the active source snapshot, then Gemini reads that raw plugin apiKey through the strict SecretRef normalizer.

## Solution Assessment

Unclear. The requested resolved-snapshot direction is right, but the safest implementation should distinguish managed agent wrappers from explicit custom-config callers instead of globally overriding all nonmatching config inputs.
