## Summary

Current `main` already supports the underlying outcome through OpenClaw-native agent workspaces: each agent can point at its own workspace repo, OpenClaw injects per-agent personality/context files from that workspace, agent-specific skills and hooks load from workspace-scoped directories, and MCP remains intentionally delegated to `mcporter` rather than a first-class `.claude/.mcp.json` flow.

## Risks / Open Questions

- The issue specifically names `.claude/CLAUDE.md` and `.mcp.json`; current OpenClaw solves the same problem with native workspace files (`AGENTS.md`/`SOUL.md`) and mcporter, not literal `.claude` compatibility.
- There is no built-in `openclaw agents pull` repo-sync command; operators are expected to manage the workspace repo directly.

## Close Comment

Closing after a Codex review because current `main` already supports this through OpenClaw’s native agent-workspace model. Each agent can use its own workspace path (`docs/cli/agents.md:23`), the workspace is explicitly meant to live as its own private git repo outside `~/.openclaw/` (`docs/concepts/agent-workspace.md:12` and `:139`), and OpenClaw already injects per-agent workspace context such as `AGENTS.md`, `SOUL.md`, `IDENTITY.md`, and `USER.md` on every turn (`docs/concepts/system-prompt.md:101`). Agent-scoped skills and hooks are also already loaded from workspace roots like `<workspace>/skills`, `<workspace>/.agents/skills`, and `<workspace>/hooks` (`src/agents/skills/workspace.ts:533`, `docs/cli/hooks.md:157`). For MCP, the current architecture intentionally routes that through `mcporter` instead of first-class `.mcp.json` handling in core (`VISION.md:72`). So the requested outcome exists on `main`, but via OpenClaw workspaces rather than a literal `.claude` clone/sync layer.
