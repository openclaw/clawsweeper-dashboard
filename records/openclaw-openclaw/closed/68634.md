## Summary

Keep this issue open. Current main has fixed the Telegram Native Approvals startup retry loop, but the remaining CLI scope-upgrade behavior is not clearly fixed: explicit CLI gateway calls still default to the full operator scope set, and tests intentionally lock that in as legacy behavior. The maintainer comment in the issue also explicitly left it open for this second part.

## Best Possible Solution

Keep the issue open and treat the native-approval retry loop as already fixed, while resolving the remaining CLI/device-pending part. The likely product path is to make read-only CLI commands use least-privilege scopes or reuse cached approved scopes safely when shared-token CLI calls include a paired device identity, then add regression coverage proving repeated `gateway status`, `devices list`, and other read-only/inspection commands do not create persistent scope-upgrade pending entries after approval.
