## Label Justifications

## Summary

Close as a duplicate of the older open canonical tracker for configured heartbeat prompts not being respected. Current source and the latest release still support this report's core symptom, but the remaining work belongs on https://github.com/openclaw/openclaw/issues/40255 rather than a second open issue.

## What This Changes

The issue reports that configured `agents.defaults.heartbeat.prompt` is replaced by the hardcoded heartbeat transcript placeholder in model-visible heartbeat runs.

## Best Possible Solution

Keep https://github.com/openclaw/openclaw/issues/40255 as the single tracker and land a narrow fix that keeps `[OpenClaw heartbeat poll]` transcript-only while sending the resolved configured heartbeat prompt to the model.

## Reproduction Assessment

Yes. Source inspection shows the configured heartbeat prompt reaches the run setup, then the heartbeat transcript placeholder can win in the runtime prompt split and is what `context.compiled.prompt` records; I did not run a live Docker heartbeat in this read-only review.

## Solution Assessment

Yes for routing, not for a separate issue. The best fix is on the canonical tracker: preserve the transcript placeholder only for storage/export while passing the resolved heartbeat prompt as the model-visible prompt.
