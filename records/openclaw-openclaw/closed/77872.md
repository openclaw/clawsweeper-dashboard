## Summary

Keep open: current main still builds the PDF extraction fallback context without a `systemPrompt`, while the OpenAI Codex Responses request path derives `instructions` only from `context.systemPrompt`, matching the reported 400 error.

## What This Changes

Add a neutral PDF-specific system prompt to the non-native PDF extraction fallback so `openai-codex/gpt-5.5` can process extracted PDF content.

## Best Possible Solution

Add a provider-neutral PDF extraction `systemPrompt` in `buildPdfExtractionContext()`, cover both fallback branches with regression tests, and include a user-facing changelog fix entry.

## Reproduction Assessment

Yes, with source-level confidence: configure `agents.defaults.pdfModel.primary` to `openai-codex/gpt-5.5` with no fallbacks and run `pdf` on a selectable-text PDF. I did not run the live provider call, but current main still constructs the exact missing-instructions request path described in the report.

## Solution Assessment

Yes. The narrowest maintainable fix is to add a brief, neutral PDF-specific system prompt at the fallback context builder, rather than adding a provider-specific special case or changing the provider transport contract.
