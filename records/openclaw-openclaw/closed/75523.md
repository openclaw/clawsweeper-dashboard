## Summary

Keep open: the metadata/IMDS hard-deny direction is useful and the PR supplies real runtime proof, but the current diff still breaks existing exact `allowedHostnames` exceptions and leaves the public SSRF policy docs inconsistent with the behavior change.

## What This Changes

The PR changes shared SSRF host/IP classification and pinned-dispatcher validation so metadata, IMDS, and `.internal` destinations stay blocked when private-network access is enabled, with regression tests and a changelog entry.

## Best Possible Solution

Land the metadata/IMDS hard-deny after separating non-overridable metadata/IP checks from exact non-metadata hostname exceptions and updating public SSRF docs/tests to match the final policy.

## Reproduction Assessment

Yes. Current main source shows the private-network skip bypasses pre-DNS hostname classification, and the PR body supplies before/after live output for the production resolver path; the exact-exception regression is also source-reproducible from the PR call order.

## Solution Assessment

No. Hard-blocking metadata/IMDS is the right direction, but the current implementation applies the broad hostname classifier before preserving exact non-metadata exceptions; the safer fix keeps those exceptions working and documents the final policy.
