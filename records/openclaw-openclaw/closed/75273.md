## Summary

Close as implemented on main. Current main and release `v2026.4.29` already route configured custom Ollama provider IDs, including `remote-ollama`, through the generic `api: "ollama"` owner-hint path; the PR’s hard-coded alias is redundant and narrower than the shipped generic behavior.

## What This Changes

The PR adds `remote-ollama` as an Ollama provider alias in the Ollama plugin registration and plugin manifest metadata.

## Best Possible Solution

Keep the generic `models.providers.<id>.api` owner-hint path as the supported solution for alternate Ollama provider IDs, rather than adding one-off aliases for specific names like `remote-ollama`.

## Reproduction Assessment

Yes. A config with `models.providers.remote-ollama.api = "ollama"` exercises the same source path covered by the existing custom `ollama-spark` runtime test, and current main shows that it resolves the Ollama provider through the API owner hint.

## Solution Assessment

No. The proposed alias is not the best fix because current main already has a broader, provider-agnostic implementation that supports arbitrary custom Ollama provider IDs declaring `api: "ollama"`.
