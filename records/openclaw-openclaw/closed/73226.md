## What This Changes

Make WebChat direct messages respect `session.dmScope: per-channel-peer` instead of forcing the agent main session.

## Best Possible Solution

Use #67735 as the canonical thread: either add a supported WebChat direct-human/session-isolation mode that respects dmScope and identityLinks while preserving external delivery-route guards, or document WebChat as an explicit operator/main-session exception across session, WebChat, routing, and security docs.

## Reproduction Assessment

Yes. A direct WebChat runtime-policy context on `agent:main:main` reaches the resolver, `webchat` is dropped as the policy channel, and the current unit test asserts the main-session/unsandboxed result. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

Unclear. Removing only the `webchat` exclusion is the smallest local edit, but adjacent `chat.send` and delivery-route guards intentionally treat WebChat as internal, so the maintainable fix needs the canonical WebChat session-model decision in #67735.
