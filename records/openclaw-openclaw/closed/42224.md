## Summary

Current `main` already implements the Discord `maxLinesPerMessage` fix across live reply paths, and the changelog records it as shipped in 2026.3.11. The reported behavior matches an older regression that has since been addressed.

## Close Comment

Closing as implemented on `main` after Codex review. This report matches an older Discord reply-chunking regression that is already documented as fixed in `CHANGELOG.md:2708` under the `2026.3.11` release: OpenClaw now resolves the effective `channels.discord.maxLinesPerMessage` across live reply paths and preserves `chunkMode`, so replies no longer fall back to the default 17-line split unexpectedly. The current code reflects that: `extensions/discord/src/accounts.ts:80` resolves the effective value from runtime/root/account config, `extensions/discord/src/monitor/message-handler.process.ts:586` uses it in live reply handling, and `extensions/discord/src/monitor/reply-delivery.ts:166` and `:323` pass it through the outbound chunking and fast-send path. There are also regression tests covering the exact fallback/forwarding behavior in `extensions/discord/src/accounts.test.ts:107`, `extensions/discord/src/monitor/message-handler.process.test.ts:776`, and `extensions/discord/src/monitor/reply-delivery.test.ts:429`. If you still see this on a current release newer than `2026.3.11`, please open a fresh issue with the exact OpenClaw version and the active Discord account/runtime config so maintainers can check for a new regression.
