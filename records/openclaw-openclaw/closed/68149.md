No. The shipped XML-based schtasks path is the narrower maintainable fix because it sets the required Task Scheduler settings while preserving the existing fallback and lifecycle model.

- **[P1] Preserve the Windows fallback instead of failing before install:** `src/daemon/schtasks.ts:954-959`
  - body: This early admin check runs before task creation, so a non-admin Windows install errors out instead of reaching the existing Startup-folder fallback.   - body: The schtasks fallback adds `/I 0` while still creating an `ONLOGON` task. Microsoft documents `/i` as valid only with `ONIDLE`, so environments that need this fallback can fail task creation instead of installing the gateway.
  - confidence: 0.88

  - body: Returning immediately after `Unregister-ScheduledTask` skips the later cleanup that removes the Startup-folder login item and generated task script. A user with stale fallback artifacts can still have auto-start behavior after uninstall.
  - confidence: 0.9
