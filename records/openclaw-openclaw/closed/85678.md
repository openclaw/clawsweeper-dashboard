## Summary

Keep open: the PR addresses a real Telegram responsiveness gap and includes useful live proof, but the current diff sends a visible typing action before Telegram DM authorization/context checks and outside the shared sendChatAction safety path.

## What This Changes

The PR adds best-effort Telegram `sendChatAction("typing")` telemetry from the isolated polling ingress worker and logs worker typing activity in the polling session.

## Best Possible Solution

Keep the early Telegram DM acknowledgement only if the ingress path can reuse the same authorization decision and sendChatAction safety guard as the existing message-context path.

## Reproduction Assessment

Yes at source/proof level: current main only sends typing after Telegram context/access setup, while the PR body supplies live log timing showing ingress typing before inbound handling. I did not run a live Telegram probe in this read-only review.

## Solution Assessment

No. The worker-side raw cue is too early because it bypasses established access checks and sendChatAction safeguards; the maintainable fix is a prepared, authorized, guarded ingress cue or keeping the existing post-context cue.
