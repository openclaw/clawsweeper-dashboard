## Summary

Current `main` already strips inline base64 audio from `chat.history` payloads before oversized-message replacement, and the same fix shipped in `v2026.4.23` (`a9797214338ba31c52c796adbb75afb16e0684a9`).

## Evidence

- **Audio blobs are redacted from chat history blocks:** `sanitizeChatHistoryContentBlock` removes `source.data` from `type: "audio"` blocks when the source is base64, and replaces it with `omitted` plus `bytes`. That directly addresses the reported inline-TTS blob inflation in WebChat history.
  - file: [src/gateway/server-methods/chat.ts:954](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/src/gateway/server-methods/chat.ts#L954)
  - command: `nl -ba src/gateway/server-methods/chat.ts | sed -n '900,1065p'`
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **Sanitization happens before oversized placeholder replacement:** The `chat.history` handler calls `sanitizeChatHistoryMessages(...)` before `replaceOversizedChatHistoryMessages(...)`, so redacted audio blocks are processed before the 128 KB per-message hard cap can replace the whole message with `[chat.history omitted: message too large]`.
  - file: [src/gateway/server-methods/chat.ts:1908](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/src/gateway/server-methods/chat.ts#L1908)

  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
