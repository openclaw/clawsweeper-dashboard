## Summary

The reported native macOS app node behavior is the current remote-mode contract, not a repo bug: browser automation for a remote Gateway is expected to come from the CLI node-host service, and the reported browser-capable node-host is disconnected.

## What This Changes

The issue asks for the macOS app node to advertise `browser`/`browser.proxy` when Browser Control is enabled against a remote Gateway, or for the intended node-host requirement to be clarified.

## Best Possible Solution

Keep the documented remote-browser contract: start and pair the CLI node-host service on the Mac for browser control, and open a narrower report only if that service fails to connect or advertise `browser.proxy` after following the node-host setup docs.

## Reproduction Assessment

Not applicable as a bug reproduction. Source and tests show the native macOS app node intentionally omits `browser`/`browser.proxy` in remote mode; the runtime check is whether the separate CLI node-host service is installed, started, paired, and connected.

## Solution Assessment

Yes, the current documented solution is the maintainable path for the reported setup: use the headless node-host browser proxy for remote gateways. Making the native app node own remote browser relay would be a separate product decision, not a cleanup fix.
