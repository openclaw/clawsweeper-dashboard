## Summary

This issue is a duplicate of the existing stuck-session recovery track. The v2026.4.26 symptoms here, especially `state=processing`, queued messages, long session-write lock, and Telegram polling stall, map to the same remaining core gap already tracked by #71127, with #73243 open as the active implementation candidate. Current main has only conservative stale-lane recovery; it still does not add a public active-run abort threshold or default active stuck-session abort behavior.

## Best Possible Solution

Close this issue as a duplicate of #71127 and keep implementation focused on #73243 or a replacement PR that adds a conservative configurable stuck-session abort policy, typed telemetry, docs/schema updates, and regression coverage. If high RSS remains after that recovery work lands, open a narrower performance issue with a diagnostics export, heap/CPU profile, and current-version proof.
