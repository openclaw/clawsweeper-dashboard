## Summary

Current main already implements the reported WhatsApp creds hardening: queued per-account saves, atomic `creds.json` replacement, guarded backup restore, and reconnect-time queue draining. The issue was filed against `2026.4.1`, and the changelog shows the relevant fixes shipped afterward.

## Evidence

- **Atomic creds writer on main:** `writeCredsJsonAtomically()` writes `creds.json` through a sibling temp file, fsyncs the file, renames it into place, reapplies `0600`, and fsyncs the parent directory instead of truncating the live file in place.
  - file: [extensions/whatsapp/src/creds-persistence.ts:32](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/extensions/whatsapp/src/creds-persistence.ts#L32)
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **Reconnect path now waits for queued saves:** `createWaSocket()` waits for the per-auth creds save queue before auth bootstrap, then restores from backup only after that wait, and routes `creds.update` through the queued atomic save path.
  - file: [extensions/whatsapp/src/session.ts:136](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/extensions/whatsapp/src/session.ts#L136)
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **Regression tests cover the failure mode:** Tests assert temp-file-plus-rename writes for `creds.json` and also assert that a simulated rename failure leaves the previous `creds.json` valid and removes temp files.
  - file: [extensions/whatsapp/src/session.test.ts:436](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/extensions/whatsapp/src/session.test.ts#L436)
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **Atomic write shipped in release notes:** The changelog records the WhatsApp fix: queued `creds.json` updates were rewritten atomically so interrupted saves no longer leave truncated login state behind.
  - file: [CHANGELOG.md:1037](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/CHANGELOG.md#L1037)
- **Race-on-reopen hardening shipped in release notes:** The changelog also records a follow-up WhatsApp fix: auth bootstrap now drains the pending per-auth creds save queue before reopening sockets so backup restore no longer races an in-flight write.
  - file: [CHANGELOG.md:616](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/CHANGELOG.md#L616)
