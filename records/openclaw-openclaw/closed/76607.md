## Summary

Close as a duplicate of open #76500. Both reports describe the same 2026.5.2 `pdf` tool staging overhead around `pdfjs-dist`/document extraction; this issue adds useful Windows timing data but does not point to a separate source path.

## What This Changes

The issue asks for the `pdf` tool to stop adding about 20 seconds of Windows per-run overhead by avoiding `pdfjs-dist` work unless a PDF is actually processed.

## Best Possible Solution

Keep #76500 as the canonical fix thread and fix the shared PDF staging path there by making `pdf` tool registration cheap and deferring document-extract/`pdfjs-dist` work until actual PDF execution or fallback extraction.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a live Windows run here, but the report gives concrete logs and #76500 provides the same 2026.5.2 signal on Linux while current source still shows the shared PDF staging and extraction path.

## Solution Assessment

Is this the best way to solve the issue? Yes: closing this satellite as a duplicate keeps the Windows evidence linked while concentrating the code fix in #76500, the earlier open canonical report for the same root cause.
