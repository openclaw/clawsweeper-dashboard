## Summary

Close as duplicate/superseded: the report is credible and source-backed, but the remaining work is the same Claude CLI `extraSystemPromptHash` false-invalidation class already tracked by https://github.com/openclaw/openclaw/issues/69118.

## What This Changes

Prevent Claude CLI session reuse from invalidating when a direct `dmScope: "main"` conversation switches between channels whose provider labels differ.

## Best Possible Solution

Keep https://github.com/openclaw/openclaw/issues/69118 as the canonical fix path and add this cross-channel direct `dmScope: "main"` scenario as regression coverage for the shared prompt-hash/session-continuity fix.

## Reproduction Assessment

Yes. Current source shows the direct-chat provider label flows into the hashed static prompt while `dmScope: "main"` routes multiple direct channels to one session key, so alternating WebChat and Telegram can deterministically change `extraSystemPromptHash` and trigger `invalidatedReason: "system-prompt"`.

## Solution Assessment

Yes for triage: closing this duplicate keeps the fix in one canonical thread. The implementation should normalize or remove transient channel/direct context from durable CLI session identity, or otherwise scope the binding so an intentional shared DM session is not invalidated by delivery-channel labels.
