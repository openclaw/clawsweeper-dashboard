## Summary

Keep open. The reporter provided a focused live reproduction plus a direct `agent` RPC control, and current `main` still has the scope/identity shape implicated by the report rather than a proven fix.

## What This Changes

`sessions_spawn` should start native subagent runs under token auth when the local operator device token includes `operator.write` instead of failing with `missing scope: operator.write`.

## Best Possible Solution

Add a targeted regression for native `sessions_spawn -> Gateway agent` under token auth with a full-scope operator device token, then preserve only the necessary device identity/scope propagation for that internal call path without broadening remote auth or owner/admin privileges.

## Reproduction Assessment

Yes, with caveat: the reporter provided a concrete redacted live repro and direct `agent` control RPC, and current source still matches the implicated Gateway identity/scope-selection path. I did not execute that live Gateway setup against current `main`.

## Solution Assessment

Unclear. The reporter's local hotfix points at the right boundary, but the current tests intentionally expect device identity to be omitted for scoped backend loopback calls, so the maintainable fix should be regression-led and security-reviewed.
