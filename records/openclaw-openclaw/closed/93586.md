## Summary

Keep open: this targets a real QMD/mcporter 0.10 regression, but the patch is not merge-ready because it drops supported user XDG config discovery, can strip agent-scoped QMD env from mcporter-launched local qmd servers, and does not include real after-fix memory_search proof.

## What This Changes

The PR adds a separate mcporter spawn environment in memory-core QMD manager, removes XDG_CONFIG_HOME and XDG_CACHE_HOME from that environment, and updates the adjacent unit test to expect those vars to be absent.

## Best Possible Solution

Separate mcporter config discovery from QMD server execution: keep mcporter out of OpenClaw's QMD-scoped XDG tree while preserving user XDG discovery and ensuring local/generated qmd mcp execution receives agent-scoped QMD config/cache env.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.8 still pass QMD-scoped XDG env into mcporter, and mcporter v0.10.2 uses XDG_CONFIG_HOME for config discovery. I did not run a live gateway memory_search reproduction in this read-only review.

## Solution Assessment

No. The narrow env split is a plausible mitigation, but it is not the best fix until it preserves user mcporter discovery and agent-scoped QMD runtime env for local/generated qmd mcp execution.
