## Summary

Current main and the latest shipped release already manage the reported GPT-4.1 OpenAI family through the pinned Pi model registry. OpenClaw resolves configured provider/model IDs through `ModelRegistry.find()` before constructing any fallback model, and the shipped registry contains `openai/gpt-4.1`, `openai/gpt-4.1-mini`, and `openai/gpt-4.1-nano` rows with OpenAI Responses metadata and the expected context/token limits.

## What This Changes

Request that `openai/gpt-4.1`, `openai/gpt-4.1-mini`, and `openai/gpt-4.1-nano` receive managed registry metadata or a visible unmanaged-model warning instead of silently passing through.

## Best Possible Solution

Keep the registry-backed implementation shipped in `v2026.4.27` and current main; treat any future warning for truly unknown model IDs as separate generic model-configuration UX work.

## Reproduction Assessment

Yes. The current-main source and pinned dependency rows show that `openai/gpt-4.1*` resolves through `ModelRegistry.find()` and therefore does not take the unmanaged fallback path; a live OpenAI call is not needed to prove this registry-resolution behavior.

## Solution Assessment

Yes. Managing these exact model IDs through the existing Pi registry and OpenAI plugin catalog is the narrowest maintainable fix for the reported GPT-4.1 family behavior; warning UX remains relevant only for IDs that are still genuinely unknown.
