## What This Changes

The issue requests fixing a gateway crash where pinned SSRF DNS lookup can rotate dual-stack Anthropic traffic onto unreachable IPv6 and `EADDRNOTAVAIL` is not classified benign.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/80162 while preserving SSRF DNS-answer validation and adding regression coverage for IPv4-preferred automatic pinned lookups plus `EADDRNOTAVAIL` classification.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a mixed A/AAAA pinned lookup can return IPv6 on a later automatic provider request, and `EADDRNOTAVAIL` is not currently benign; I did not run the full WSL2 gateway scenario.

## Solution Assessment

Yes. The narrow maintainable solution is to fix the shared SSRF pinned lookup family selection and classify `EADDRNOTAVAIL`, with tests, rather than adding provider-specific host exceptions or relying on a resolver workaround.
