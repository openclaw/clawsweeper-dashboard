## Summary

Keep open: current main still has the source-reproducible scope mismatch and still masks explicit device-token verifier failures as `device_token_mismatch`. The issue is already paired with open closing PRs, with the maintainer-labeled https://github.com/openclaw/openclaw/pull/80189 covering the fuller end-to-end fix path, so it should close only after a fix merges.

## What This Changes

The issue reports that bootstrap-issued operator device tokens have narrower scopes than macOS/iOS/Control UI reconnect requests, and the gateway masks the verifier's `scope-mismatch` reason as `device_token_mismatch`.

## Best Possible Solution

Land a revised end-to-end fix that reuses stored approved scopes for bootstrap/cached-token reconnects and propagates scope mismatch through gateway protocol and client diagnostics without reducing shared-token operator authority.

## Reproduction Assessment

Yes, source-reproducible. Current UI/macOS/iOS defaults request `operator.admin` and `operator.pairing`, bootstrap handoff tokens do not include those scopes, `verifyDeviceToken` returns `scope-mismatch`, and `resolveConnectAuthDecision` maps explicit device-token failures to `device_token_mismatch`.

## Solution Assessment

Yes, the best fix direction is the broader end-to-end path represented by the open maintainer-labeled PR, not closing this issue early. A server-only diagnostic change is incomplete unless clients also preserve stored token scopes and handle the new scope-mismatch diagnostic correctly.
