## Summary

Keep open: current main and the latest release still throw `empty_response` for any empty claude-cli output, and `empty_response` remains a fallback/cooldown reason, so the reported wrong-model fallback path is still plausible from source.

## What This Changes

The issue reports that clean claude-cli thinking-only turns with empty visible text are treated as `empty_response` failures and rerun on a fallback model.

## Best Possible Solution

Plumb the existing silent-reply policy into the CLI runner so clean empty claude-cli results return a silent `NO_REPLY` only when the caller allows silence, while ordinary empty subprocess results still fail as `empty_response`.

## Reproduction Assessment

Yes, at source level: a claude-cli run whose `executePreparedCliRun` result has empty trimmed `text` will throw `FailoverError({ reason: "empty_response" })`, and fallback policy treats that reason as retryable. I did not run a live Claude CLI reproduction.

## Solution Assessment

No to the broad suggested fix as written; the best fix is narrower and should reuse OpenClaw's existing silent-reply policy so direct chats and genuine empty subprocess failures do not become green no-payload turns.
