## Summary

Keep open: current main still lets the Codex runtime provider from `agentMeta` win in session-store and auto-reply persistence, so `doctor --fix` can be undone by a later Codex-backed run.

## What This Changes

Prevent Codex harness runs from persisting the internal `openai-codex` runtime provider into user-facing session routes after `doctor --fix`.

## Best Possible Solution

Centralize user-facing session route normalization at the persistence boundary so `openai/*` Codex-harness runs keep `modelProvider: "openai"` while runtime metadata and auth pins can still report `openai-codex`.

## Reproduction Assessment

Yes, at source level: run an `openai/gpt-5.5` route through the Codex harness so `agentMeta.provider` reports `openai-codex`, then the current persistence code writes that provider into session route state and doctor flags it. I did not run a live packaged reproduction in this read-only sweep.

## Solution Assessment

Yes, the narrow maintainable fix is to separate internal runtime provider attribution from persisted user-facing session route state. The local patch shape is directionally right, but it should be centralized and covered across the command, auto-reply, follow-up, and doctor paths.
