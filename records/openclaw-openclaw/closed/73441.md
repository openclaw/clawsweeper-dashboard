## Summary

Current main already covers the reported Discord DM duplicate path: Discord inbound messages are claimed by account/channel/message id before debounce or preflight, duplicates return without entering the worker or auto-reply follow-up queue, and regression coverage asserts that duplicate deliveries are processed once.

## Best Possible Solution

Close as implemented. Keep the Discord replay guard and regression tests as the canonical fix; if this reappears, the next report should include current-version logs showing the same Discord message id passing preflight or the inbound worker more than once.

## Evidence

- **Discord replay key built from account, channel, and message id:** `buildDiscordInboundReplayKey` returns `${accountId}:${channelId}:${messageId}` and `claimDiscordInboundReplay` only allows `claimed` results through, so the same Discord DM message id in the same channel is treated as a replay.
