## Summary

Current `main` already replaced startup-relative heartbeat scheduling with stable per-agent phases, preserves future due slots across reloads, and has regression coverage for the restart/reload cadence behavior described here. The pre-fix `v2026.4.9` code still used `now + interval`, which matches the reporter/commenter root-cause analysis. The scheduling fix landed in PR #64560 and shipped in `v2026.4.10`.

## Evidence

- **Current runner preserves phase-based due times across reloads:** `startHeartbeatRunner()` now computes a stable `phaseMs` per agent, reuses an unchanged future `nextDueMs` on `updateConfig()`, and only advances interval wakes to the next phase slot instead of resetting from reload time.
  - file: [src/infra/heartbeat-runner.ts:1336](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/infra/heartbeat-runner.ts#L1336)
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **Current schedule helper is wall-clock phase based:** `resolveHeartbeatPhaseMs()`, `computeNextHeartbeatPhaseDueMs()`, and `resolveNextHeartbeatDueMs()` implement stable hashed phases and preserve the next future slot when interval/phase are unchanged.
  - file: [src/infra/heartbeat-schedule.ts:7](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/infra/heartbeat-schedule.ts#L7)
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **Regression test locks in reload behavior:** Scheduler tests now advance to computed phase slots, then call `runner.updateConfig(...)` and assert the next due times come from stable phase math rather than `now + interval` reload drift.
  - file: [src/infra/heartbeat-runner.scheduler.test.ts:85](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/infra/heartbeat-runner.scheduler.test.ts#L85)
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **Pre-fix v2026.4.9 matched the reported root cause:** The `v2026.4.9` runner used `lastRunMs + interval` or `now + interval` in `resolveNextDue()` / `advanceAgentSchedule()`, which would reset cadence on runner re-init or restart exactly as the issue comment described.
  - command: `curl -L --silent https://raw.githubusercontent.com/openclaw/openclaw/v2026.4.9/src/infra/heartbeat-runner.ts | sed -n '1070,1165p'`
  - sha: [60db001e96ff](https://github.com/openclaw/openclaw/commit/60db001e96ff96fe8796b3a9ff1b93125cceaa2e)
- **Fix PR and shipped release are explicit:** PR #64560 is titled `Heartbeat: spread interval runs across stable phases`, and the changelog for `2026.4.10` says heartbeat interval runs now use stable per-agent phases derived from gateway identity instead of clustering around startup-relative times.
  - file: [CHANGELOG.md:1028](https://github.com/openclaw/openclaw/blob/9a4a9a5993cc7862efa4bd876be755fe6c399702/CHANGELOG.md#L1028)

## Risks / Open Questions

- This fix stabilizes cadence across restarts/reloads, but it does not add exact top-of-hour anchors or cron-style heartbeat `at:` expressions; those are separate enhancement asks from the core bug in this issue.
