## Summary

Current `main` already handles legacy `messages.tts.edge` compat on config read, migrates it to `messages.tts.providers.microsoft` before validation, keeps the docs aligned with that runtime behavior, and no longer exposes legacy direct provider fields in `TtsConfig`. The same implementation is present in release `v2026.4.22`.

## Close Comment

Closing this as implemented on main after a Codex review. Current OpenClaw reads legacy `messages.tts.edge` config through the runtime compat path, migrates it to `messages.tts.providers.microsoft`, and only then validates the config (`src/config/io.ts:1088`, `src/config/io.ts:1428`, `src/commands/doctor/shared/runtime-compat-api.ts:6`, `src/commands/doctor/shared/legacy-config-migrations.runtime.tts.ts:10`). There is also regression coverage for legacy `messages.tts` provider-key auto-migration in `src/config/config-misc.test.ts:692`, and the docs still document the legacy alias behavior in `docs/tools/tts.md:297` while `src/config/types.tts.ts:28` now exposes only the canonical `providers` shape. The same implementation is present in release `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`), so this no longer appears to be an open bug on current main.
