## Summary

Current main resolves the reported silent failure mode: an explicit tool allowlist that resolves to no callable tools now becomes a precheck error before prompt submission, and the docs clarify that `tools.profile` is a base allowlist while `alsoAllow` is the profile-stage escape hatch. The same config should no longer complete successfully with `systemPromptReport.tools.entries: []`; users who want browser or Gmail plugin tools under `coding` should add them through `agents.list[].tools.alsoAllow` or use a broader profile.

## What This Changes

The issue reports that a top-level `tools.profile: "coding"` caused an agent with explicit browser and Gmail tool allows to receive no injected tool entries.

## Best Possible Solution

Keep the current fail-closed behavior and documented profile semantics: `tools.allow` should continue to restrict, while users who want browser or plugin tools under `coding` should grant them at the profile stage with `tools.alsoAllow` or `agents.list[].tools.alsoAllow`.

## Reproduction Assessment

Yes. The issue gives a concrete config and command, and current source shows that this shape can resolve to no effective tools, but current main now turns that into a precheck error instead of submitting a text-only prompt with empty `tools.entries`.

## Solution Assessment

Yes. Failing closed with a clear allowlist diagnostic is the safer maintainable fix for the reported silent empty-tool run; widening `tools.allow` to bypass `tools.profile` would conflict with the documented restrictive profile model.
