## Summary

Keep open: current main and the latest release still render configured fallbacks without filtering or qualifying the active user-selected session model. The runtime duplicate-call risk is not proven because the docs and fallback candidate collector indicate user `/model` overrides are strict/deduped, so the remaining bug is misleading `/status` output.

## What This Changes

The issue requests that `/status` stop showing the session-selected `/model` override as an active fallback candidate.

## Best Possible Solution

Align `/status` with effective model-routing policy by hiding or clearly qualifying configured fallbacks during user-pinned session overrides, and add regression coverage for the selected-model-in-fallbacks case.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: build a status message with a user `modelOverride` that also appears in `agent.model.fallbacks`, and the current code will print that fallback list unchanged. I did not run a live Telegram repro in this read-only review.

## Solution Assessment

No, the current display is not the best behavior because it can present configured fallbacks as active even when a user session override is selected. A narrow status-output fix plus regression test should align the UI with the documented strict override policy.
