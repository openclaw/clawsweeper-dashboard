## Summary

Current `main` contains a direct `run-node` fix for the reported upgrade/build lock hang: child-process spawn failures now settle immediately and release the artifact lock instead of waiting forever and leaving later `doctor` / `gateway start` runs stuck on `Waiting for TypeScript/runtime artifact lock.`

## Evidence

- **Current main handles child spawn errors instead of waiting forever:** `waitForSpawnedProcess` now listens for the child `error` event, logs the failure, and settles with exit code 1 instead of only waiting for `exit`.
  - file: [scripts/run-node.mjs:536](https://github.com/openclaw/openclaw/blob/a4fc6c2409f9c28e55e04e22c2e243c6900d66a3/scripts/run-node.mjs#L536)
  - command: `nl -ba scripts/run-node.mjs | sed -n '536,552p'`
  - sha: [a4fc6c2409f9](https://github.com/openclaw/openclaw/commit/a4fc6c2409f9c28e55e04e22c2e243c6900d66a3)
- **The build path holds the runtime/build lock while spawning the TypeScript compiler:** The TypeScript build runs inside `withRunNodeBuildLock(...)` and awaits `waitForSpawnedProcess(build, deps)`, so a spawn failure on this path would previously strand the lock and match the report's follow-up `Waiting for TypeScript/runtime artifact lock.` symptom.
  - file: [scripts/run-node.mjs:837](https://github.com/openclaw/openclaw/blob/a4fc6c2409f9c28e55e04e22c2e243c6900d66a3/scripts/run-node.mjs#L837)

- **Regression test added for lock release on compiler spawn error:** A dedicated test now simulates the compiler spawn error and asserts `run-node-build.lock` is removed, covering the exact stale-lock failure mode.
