## Summary

Keep open: this is a focused, source-backed fix candidate for the canonical approval-followup routing bug; current main and the latest release still drop recipient fields on the non-deliverable plugin-channel branch, while the PR preserves the captured route and adds targeted regression coverage.

## What This Changes

The PR changes exec approval followup routing so non-built-in plugin channel sessions forward the original turn-source recipient fields when resuming the agent after approval.

## Best Possible Solution

Land the focused mapper change after normal maintainer checks, keep https://github.com/openclaw/openclaw/issues/96103 open until it merges, then close the issue as implemented by this PR.

## Reproduction Assessment

Yes from source: current main can produce an agent followup with `channel` preserved but `to`/`accountId`/`threadId` omitted for an unregistered external plugin channel. I did not run a live external-channel round trip in this read-only review.

## Solution Assessment

Yes: `buildAgentFollowupArgs` is the narrow owner boundary that maps already-captured approval turn-source fields onto the gateway resume call. Changing host capture or plugin registry behavior would be broader because the fields are already captured upstream and consumed downstream.
