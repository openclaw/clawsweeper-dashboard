## Summary

Current `main` already strips `[[reply_to:*]]` tags from visible outbound text before iMessage delivery and preserves the reply target as `replyToId` metadata instead.

## Evidence

- **Shared reply-directive parser strips reply tags:** `parseReplyDirectives()` calls `parseInlineDirectives(..., { stripReplyTags: true })` and replaces the outgoing text with the stripped result while retaining `replyToId`/`replyToTag` metadata.
  - file: [src/auto-reply/reply/reply-directives.ts:23](https://github.com/openclaw/openclaw/blob/959dedb7cf59e7d77cd584520164ac911de6706d/src/auto-reply/reply/reply-directives.ts#L23)
  - command: `nl -ba src/auto-reply/reply/reply-directives.ts | sed -n '1,220p'`
  - sha: [959dedb7cf59](https://github.com/openclaw/openclaw/commit/959dedb7cf59e7d77cd584520164ac911de6706d)
- **Delivery normalization applies stripped text before transport:** `createOutboundPayloadPlan()` parses directives, stores `parsedText` as the payload text, and moves the reply target into `replyToId`, so delivery payloads no longer carry the literal `[[reply_to:*]]` prefix.
  - file: [src/infra/outbound/payloads.ts:136](https://github.com/openclaw/openclaw/blob/959dedb7cf59e7d77cd584520164ac911de6706d/src/infra/outbound/payloads.ts#L136)

  - file: [extensions/imessage/src/monitor/deliver.ts:41](https://github.com/openclaw/openclaw/blob/959dedb7cf59e7d77cd584520164ac911de6706d/extensions/imessage/src/monitor/deliver.ts#L41)

## Risks / Open Questions

- Static review only; I did not run a live macOS/iMessage reproduction in this read-only pass.
- `extensions/imessage/src/monitor/sanitize-outbound.ts` still does not strip reply tags by itself, so a future caller that bypasses the shared normalization path could reintroduce the leak.
