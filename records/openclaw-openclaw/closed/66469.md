## Summary

Current `main` already preserves memory capability state across non-activating plugin snapshot loads, and it includes a targeted regression test for the reported `activate:false`/provider-resolution path. I found strong code-level evidence that this specific bug is implemented on `main`, but no tagged release containing the fix was provable from the checkout.

## Evidence

- **Loader now snapshots and restores memory capability:** `loadOpenClawPlugins` captures `previousMemoryCapability = getMemoryCapabilityRegistration()` before plugin registration and passes that capability back into `restoreMemoryPluginState(...)` in both the `!shouldActivate` snapshot-restore path and the error rollback path. That is the missing state the issue reported.
  - file: [src/plugins/loader.ts:2873](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/src/plugins/loader.ts#L2873)
  - command: `nl -ba src/plugins/loader.ts | sed -n '2868,2918p'`
  - sha: [9a0b26cafca7](https://github.com/openclaw/openclaw/commit/9a0b26cafca76d267dd231b5eaceadb071ec749d)
- **Targeted regression test covers the exact failure mode:** A dedicated test asserts that previously registered memory public artifacts still exist after a second `loadOpenClawPlugins({ activate: false })` pass. The test comment explicitly says it simulates read-only provider-resolution style paths and verifies the memory capability survives.
  - file: [src/plugins/loader.test.ts:3364](https://github.com/openclaw/openclaw/blob/30aa7e0d4d225650b58c2fba79e71bffa5627b08/src/plugins/loader.test.ts#L3364)
  - command: `nl -ba src/plugins/loader.test.ts | sed -n '3360,3450p'`
  - sha: [30aa7e0d4d22](https://github.com/openclaw/openclaw/commit/30aa7e0d4d225650b58c2fba79e71bffa5627b08)
- **Provider resolution still uses non-activating runtime loads:** `resolvePluginProviders(...)` continues to resolve through `resolveRuntimePluginRegistry(...)`, and provider tests assert that this path uses `activate: false`. That matches the issue's reported trigger class.
  - file: [src/plugins/providers.test.ts:392](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/src/plugins/providers.test.ts#L392)
  - command: `nl -ba src/plugins/providers.test.ts | sed -n '390,410p'`
  - sha: [9a0b26cafca7](https://github.com/openclaw/openclaw/commit/9a0b26cafca76d267dd231b5eaceadb071ec749d)
- **Reported user-visible symptom still hangs off memory public artifacts:** `memory-wiki` status still computes `bridgePublicArtifactCount` by calling `listActiveMemoryPublicArtifacts(...)`, so preserving the memory capability directly addresses the reported `wiki status`/bridge artifact count symptom.
  - file: [extensions/memory-wiki/src/status.ts:220](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/extensions/memory-wiki/src/status.ts#L220)
  - command: `nl -ba extensions/memory-wiki/src/status.ts | sed -n '214,232p'`
  - sha: [9a0b26cafca7](https://github.com/openclaw/openclaw/commit/9a0b26cafca76d267dd231b5eaceadb071ec749d)
- **Issue discussion matched the now-tested root cause:** The follow-up reproduction from `schksw-ctrl` on 2026-04-15 narrowed the failure to missing in-process memory capability during wiki bridge/status flows. That lines up with the current regression test and loader fix that preserve capability across `activate:false` snapshot loads.
