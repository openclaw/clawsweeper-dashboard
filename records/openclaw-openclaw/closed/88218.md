## Label Justifications

- `P1`: The issue describes a latest-stable regression that can route a configured model to an unknown default-provider ref and break agent startup or replies for real custom-provider users.
- `impact:auth-provider`: The affected surface is model/provider routing and default model selection.

## Summary

Keep open: current main and the latest release still have resolver precedence that can remap a fully qualified configured primary when it matches an `agents.defaults.models` alias value, so the reported model-routing failure remains actionable.

## What This Changes

The issue asks OpenClaw to stop `agents.defaults.models` alias metadata from rewriting direct provider/model defaults to `openai/<alias-key>` and to make the failure easier to diagnose.

## Best Possible Solution

Adjust resolver precedence so exact configured provider/model refs are resolved before reverse alias matching, keep forward alias selection intact, and add a focused regression test for the reported custom-provider shape.

## Reproduction Assessment

Yes, by source inspection: a focused resolver test can set the primary to a custom `provider/model` ref and add an alias entry whose `alias` value equals that ref, which current code resolves through the alias key first. I did not run tests because this review is read-only.

## Solution Assessment

Yes, the best fix is narrow resolver precedence and regression coverage, not a new config surface. Exact configured provider/model refs should remain authoritative while documented aliases continue to work.
