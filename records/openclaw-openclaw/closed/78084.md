## Summary

Keep open: this is the active implementation path for dynamic plugin tool contracts, but the current patch still broadens explicit allowlists to same-namespace sibling tools, lacks real plugin/Gateway proof, and needs maintainer acceptance for the public manifest contract change.

## What This Changes

The PR adds terminal wildcard matching for plugin `contracts.tools`, applies it to registration and plugin-tool resolution, adds regression tests, and documents the new manifest syntax.

## Best Possible Solution

Land a maintainer-approved bounded wildcard ownership contract that uses patterns for owner discovery while preserving concrete explicit allowlist filtering and proving the flow with a real dynamic plugin or Gateway run.

## Reproduction Assessment

Yes, at source level. Current main exact-matches `contracts.tools`, and PR head can be inspected to show same-wildcard sibling tools passing an explicit allowlist that requested only one concrete tool.

## Solution Assessment

No, not yet. The bounded wildcard direction is plausible, but the patch must filter returned tools against the original requested concrete names unless the whole plugin/group/default set is selected, and it still needs real behavior proof plus maintainer API approval.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Filter wildcard results by requested tool names:** `src/plugins/tools.ts:1114-1116`
  - body: When an explicit allowlist requests `dynamic_alpha`, this branch selects the `dynamic_*` contract so the plugin can load, but then any factory-returned tool matching `dynamic_*`, such as `dynamic_beta`, also passes even though it was not allowlisted. Keep wildcard matching for ownership discovery, but filter returned tools by the original concrete allowlist unless the selection was for the whole plugin/group/default set.
  - confidence: 0.9
