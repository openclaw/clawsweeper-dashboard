## Summary

Current `main` already exposes plugin APIs that cover the requested dynamic context pruning surface. A plugin can replace pre-LLM context assembly through the `contextEngine` slot (`registerContextEngine(...); assemble(...)` returns the exact ordered messages sent to the model), and it can transform or suppress persisted tool results with `tool_result_persist` and `before_message_write`. Those same surfaces are present in the shipped `v2026.4.22` release.

## Risks / Open Questions

- The existing pre-request surface is an exclusive `contextEngine` slot, not a lightweight additive `context_before_request` hook, so multiple pruning plugins cannot stack by default.
- The requested API shape and names are not implemented verbatim; this close relies on the existing surfaces being functionally sufficient rather than identical.

## Close Comment

Closing from a Codex review: the requested plugin surface is already available on current `main`, and it is present in the shipped `v2026.4.22` release. For pre-request context pruning, OpenClaw already supports plugin-owned context assembly through `api.registerContextEngine(...)`; the engine’s `assemble()` method decides which messages are sent before each model run (`docs/concepts/context-engine.md:10`, `docs/concepts/context-engine.md:67`, `src/plugins/types.ts:2010`, `src/agents/pi-embedded-runner/run/attempt.context-engine-helpers.ts:223`). For post-tool pruning, OpenClaw already exposes `tool_result_persist` and `before_message_write` to rewrite or block tool-result messages before they are persisted (`docs/concepts/agent-loop.md:99`, `src/plugins/hook-types.ts:356`). That covers the feature request well enough without adding a new core API.
