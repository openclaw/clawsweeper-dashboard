## Label Justifications

- `P1`: Remote iMessage photo attachments in a documented setup reach the agent but cannot be analyzed, breaking a real channel media workflow on the latest release.
- `impact:message-loss`: The text message is delivered, but the attached image content is effectively lost to the agent because media understanding sees an unreachable raw Mac path.

## Summary

Keep open: current main and v2026.5.22 still run media understanding before sandbox media staging, while the iMessage remote-host path deliberately passes raw Mac attachment paths for later SCP staging.

## What This Changes

The issue asks OpenClaw to stage and rewrite remote iMessage attachment paths before media understanding runs.

## Best Possible Solution

Update the reply pipeline so remote inbound media is staged and path-rewritten before media understanding, while preserving existing local staging and `MediaStaged` behavior with focused regression coverage.

## Reproduction Assessment

Yes, source-reproducible: an iMessage `remoteHost` attachment produces raw Mac media paths plus `MediaRemoteHost`, and `get-reply.ts` applies media understanding before calling `stageSandboxMedia`. I did not run a live remote Mac SCP setup in this read-only review.

## Solution Assessment

Yes, the narrow fix is to stage remote media before media understanding in the reply pipeline, then keep the existing later staging path from doing duplicate work. This should be implemented with tests around the remote-host ordering and existing `MediaStaged` behavior.
