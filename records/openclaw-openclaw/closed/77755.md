## Summary

Keep open: current main still drops the WhatsApp account authDir before command authorization and group owner-bypass checks, so an owner reported only as an @lid sender can fail the existing allowlist path. The later duplicate is already closed in favor of this canonical report, and the remaining fix is narrow in the WhatsApp plugin.

## What This Changes

The issue requests accepting WhatsApp group control commands from the configured owner when Baileys reports that sender as a LID JID that maps back through the account auth directory.

## Best Possible Solution

Fix the WhatsApp plugin by threading the existing account authDir through command authorization and group owner-bypass identity resolution, with regressions for LID-only group owner commands.

## Reproduction Assessment

Yes, at source level. A group WebInboundMessage with senderJid set to an @lid value, no senderE164, and an authDir reverse mapping reaches authDir-less identity helpers, leaving groupSender empty and command authorization false.

## Solution Assessment

Yes. The narrow maintainable fix is to pass the existing WhatsApp account authDir through WhatsApp-owned authorization and group-gating call sites, reusing the current LID resolver instead of changing core policy or adding config.
