## Summary

The issue should stay open because current main still delegates Signal typing to signal-cli's `sendTyping` path, while the reported failure is in the upstream native delivery path that OpenClaw currently cannot control. The report is unusually detailed and matches the source/dependency contract review, but I did not establish a live official-client reproduction in this review.

## What This Changes

The issue reports that Signal native typing requests succeed through signal-cli but official Signal clients do not display the typing indicator during long OpenClaw replies.

## Best Possible Solution

Patch the upstream signal-service-java or signal-cli 1:1 typing delivery path to use the working delivery flag, then update OpenClaw's Signal runtime/install guidance once that fix is released while testing group typing separately.

## Reproduction Assessment

Yes for source-level reproduction: current main calls signal-cli `sendTyping`, and the inspected upstream dependency path uses the reported `online=true` flag. No for full live reproduction in this review because I did not run an official Signal client and patched signal-cli side by side.

## Solution Assessment

Unclear as an OpenClaw-only fix: the requested behavior is correct, but the narrow durable fix appears to belong upstream in signal-cli or signal-service-java rather than in OpenClaw's JSON-RPC caller. Carrying a patched runtime would need explicit maintainer approval because it changes dependency and release ownership.
