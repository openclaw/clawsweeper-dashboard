## Summary

Current main already covers the PR's central WhatsApp proxy behavior: gateway/CLI startup installs the env-proxy Undici dispatcher before plugin network work, the WhatsApp session passes ambient proxy agents/dispatchers into Baileys, tests assert the proxy-agent wiring, docs describe the supported env vars, and the later Baileys media upload hardening avoids the risky fetchAgent/dispatcher shape from this older PR. The PR's OpenShell reproduction was useful, but the current implementation is the better maintainer path.

## Best Possible Solution

Close this PR and keep the current main implementation: ambient gateway/CLI env-proxy bootstrap plus WhatsApp-owned Baileys agent/dispatcher wiring and the merged Baileys media-upload hardening. Any broader all-network proxy scope should continue under the separate global proxy tracking issue rather than this stale WhatsApp PR.
