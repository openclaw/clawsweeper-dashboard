## Label Justifications

- `P1`: The report is a release regression that can stop CLI-backed isolated cron agent turns from starting for affected users.
- `impact:auth-provider`: The failure is caused by provider/runtime routing of a cron tool policy into a CLI backend that cannot enforce runtime allow-lists.

## Summary

Current main already fixes the reported cron/claude-cli failure by dropping only auto-stamped default toolsAllow caps before CLI execution, while keeping explicit per-cron restrictions fail-closed; the fix is main-only and not in v2026.6.10.

## What This Changes

The issue reports that isolated cron agentTurn jobs using claude-cli fail because gateway-injected toolsAllow reaches a CLI backend that rejects runtime allow-lists.

## Best Possible Solution

Keep the current main implementation, ship it in the next release, and avoid adding a separate CLI runtime toolsAllow enforcement path unless maintainers later choose a new explicit CLI policy contract.

## Reproduction Assessment

Yes at source level for the affected release path: v2026.6.10 still forwards cron toolsAllow to the CLI helper shape that rejects defined allow-lists. I did not run a live cron job, and current main no longer has the reported default-cap failure.

## Solution Assessment

Yes. The current fix is the narrow runtime-boundary solution: drop only the server-stamped default cap when the backend is CLI, while explicit per-cron restrictions still fail closed in prepareCliRunContext.
