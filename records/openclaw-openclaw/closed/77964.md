## Summary

Current main already implements the missing recovery path for extended `stalled_agent_run` embedded work, while still avoiding unsafe lane resets for active reply work; the fix is on main after v2026.5.4 and is not proven shipped yet.

## What This Changes

The issue requests session recovery and visible failure handling when provider timeouts or failover leave an agent run stalled instead of draining the session lane.

## Best Possible Solution

Keep the current-main delayed abort-drain recovery path and ship it in the next release so v2026.5.4 users receive the fix.

## Reproduction Assessment

No, not as a failing current-main path. The v2026.5.4 report and attachments show the stalled/liveness symptoms, and current main has source and regression-test coverage for the delayed recovery path that addresses them.

## Solution Assessment

Yes. The current implementation is conservative: it preserves active reply work, waits for an extended no-progress stall, then abort-drains the embedded run and releases the lane instead of resetting slow-but-healthy work immediately.
