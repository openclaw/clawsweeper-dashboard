## Summary

Keep open: current main still has a source-level ordering gap where Telegram progress work is serialized after it reaches the draft queue, but embedded tool/progress callbacks can be fired without being awaited before final delivery begins.

## What This Changes

Ensure Telegram partial streaming delivers tool/progress preview updates before the final assembled reply on tool-heavy turns.

## Best Possible Solution

Add a shared pending callback drain/ack path for channel-visible progress updates, then have Telegram flush that path before final delivery with a deterministic partial-streaming regression.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows a current fire-and-forget callback path that can enqueue Telegram progress after final delivery has already started.

## Solution Assessment

No patch is under review. The best fix is a narrow generic ordering seam for delivery-affecting progress callbacks, not Telegram-specific sleeps or network timing workarounds.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/77401. Telegram partial/progress streaming can show the final assembled reply before late tool/progress preview updates because embedded subscriber callbacks call `onAgentEvent`/`onPartialReply` fire-and-forget while Telegram only drains the draft queue after callbacks have reached `enqueueDraftLaneEvent`. Add a narrow shared pending-callback drain/ack path for delivery-affecting progress callbacks and have Telegram flush it before final delivery. Add deterministic regression coverage for a tool-heavy partial-streaming turn where `onAgentEvent`/tool progress is delayed while final delivery begins. Do not add sleeps, network timing heuristics, or a Telegram-only workaround if a generic callback-drain seam is sufficient. Preserve existing preview behavior, status reaction behavior, and block reply ordering. Add a user-facing CHANGELOG.md fix entry crediting @SheraSnow unless the change ends up test-only.
