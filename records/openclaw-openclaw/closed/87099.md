## Summary

Keep open: current main still has a source-level gap between documented `/model ...@profile` pinning and session patch persistence, and the OpenAI Responses transport still contains the encrypted-reasoning retry path reported in the logs.

## What This Changes

Make canonical `openai/gpt-5.5` Pi-runtime sessions use configured `openai-codex` OAuth profiles via `/model ...@profile` and avoid repeated encrypted-reasoning fallback retries on GPT-5.5.

## Best Possible Solution

Persist and validate the `/model ...@profile` suffix through the session patch path using the existing auth-provider alias/routing helpers, then separately prove or fix the GPT-5.5 encrypted-content replay behavior in the OpenAI Responses transport.

## Reproduction Assessment

Yes for the core route mismatch: current source documents `/model ...@profile`, strips the profile suffix during model resolution, and does not persist it in the session patch path. The encrypted-retry frequency is supported by source and reporter logs, but was not live-reproduced here.

## Solution Assessment

Partly: the best first fix is a narrow session patch/profile-persistence repair that reuses existing OpenAI/Codex routing helpers. The encrypted-reasoning retry behavior should be handled with focused transport tests plus live GPT-5.5 evidence rather than guessed from logs alone.
