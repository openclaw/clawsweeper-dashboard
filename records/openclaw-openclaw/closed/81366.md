## Summary

Current main and shipped releases already address the reported Active Memory timeout-budget mismatch: the prompt-build hook, plugin watchdog, and embedded recall runner now share the configured effective budget, and docs cover the WSL2-style cold-start tuning path.

## What This Changes

The issue asks Active Memory to stop timing out independently of `config.timeoutMs` on WSL2 and to make the hook and sub-agent budgets align or be documented clearly.

## Best Possible Solution

Keep the shipped timeout-budget implementation and handle any remaining current-release WSL2 Active Memory slowness as a fresh, versioned performance report with new logs.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? The historical 2026.4.x behavior is well documented by the reporter logs and the prior linked timeout reports, but current main no longer reproduces the mismatch because the hook, watchdog, and embedded runner use the same effective budget.

## Solution Assessment

Is this the best way to solve the issue? Yes; the shipped fix keeps the behavior plugin-owned, uses the existing `timeoutMs` plus explicit `setupGraceTimeoutMs` contract, and avoids changing generic plugin hook semantics.
