## Summary

Keep open. Current main still submits the pre-compaction memory flush prompt as the embedded run's visible/transcript prompt; `silentExpected` suppresses outbound delivery, but does not make the internal memory-flush turn runtime-only or keep it out of persisted user-message history.

## Best Possible Solution

Keep this issue open as the core memory-flush transcript leak tracker. The likely fix belongs in `src/auto-reply/reply/agent-runner-memory.ts`: make pre-compaction memory flush a true runtime-only/internal embedded run, probably by passing `transcriptPrompt: ""` or adding explicit internal-run provenance. Add regression coverage proving the flush prompt is absent from session JSONL and `chat.history`, and decide whether existing polluted sessions should be sanitized. Related #51421/#12590 should stay scoped to memory-flush cadence/reliability, while #43567 is only an adjacent UI treatment.

## Risks / Open Questions

- Closing this now would leave a core transcript-pollution path open: custom `agents.defaults.compaction.memoryFlush.prompt` values would still be submitted as normal prompt text.
- A UI-only treatment like #43567 could reduce confusion but would not prevent compaction, session JSONL, or downstream consumers from seeing polluted user-message history.
- Already-polluted session JSONL may need a repair or sanitization path in addition to preventing new flush prompts from being persisted.
