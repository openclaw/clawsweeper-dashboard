## Summary

Keep open. Current `main` still intentionally restarts the Bonjour advertiser when a service stays non-`announced` for 8s, and the test suite explicitly expects the restart/warn path for a service stuck in `announcing`. OpenClaw now has disable/workaround paths for Bonjour, but that does not resolve the reported self-healing loop or log flooding.

## Evidence

- **Current main still restarts on stuck announcing:** `extensions/bonjour/src/advertiser.ts` keeps a 5s watchdog, treats any non-`announced` state older than 8s as unhealthy, and calls `recreateAdvertiser(...)`, which logs `bonjour: restarting advertiser (...)` and recreates the responder. There is no restart backoff for repeated stuck-`announcing` cycles.
  - file: [extensions/bonjour/src/advertiser.ts:69](https://github.com/openclaw/openclaw/blob/245451b6a9c529464e177afa65013adca0d63eb5/extensions/bonjour/src/advertiser.ts#L69)
  - command: `nl -ba extensions/bonjour/src/advertiser.ts | sed -n '60,120p;350,430p'`
  - sha: [245451b6a9c5](https://github.com/openclaw/openclaw/commit/245451b6a9c529464e177afa65013adca0d63eb5)
- **Current tests encode the reported behavior as expected:** `extensions/bonjour/src/advertiser.test.ts` has a regression test named `recreates the advertiser when ciao gets stuck announcing`; after 15s it expects the warning containing `restarting advertiser`, another service creation, and another advertise attempt. The next test also expects `service stuck in announcing` to trigger recreation.

## Risks / Open Questions

- Affected macOS Gateway users still need to disable Bonjour entirely to avoid the loop; there is no narrower mitigation in current main.
- Closing as implemented would be misleading because both current main and the latest release still treat this restart loop as intended behavior.
