## What This Changes

The PR changes plugin runtime logging to re-resolve child loggers on each emitted log call and adds regression tests for runtime log-level changes.

## Best Possible Solution

Land the facade-boundary runtime logging fix after ordinary maintainer approval while keeping the plugin logging API shape unchanged.

## Reproduction Assessment

Yes, source-level. Current main creates a tslog sublogger once in createRuntimeLogging(), tslog filters against that sublogger's stored minLevel, and long-lived channel monitors capture the facade logger before later shouldLogVerbose()-gated debug writes.

## Solution Assessment

Yes. Re-resolving at the plugin runtime facade is the best layer because createPluginRuntime injects this shared logger; changing each channel plugin would duplicate the same runtime invariant.
