## Summary

Keep open. Current main still has the order-sensitive isolated-cron payload classifier, and the linked draft PR that uses closing syntax is open with a blocking review finding, so this issue should remain the canonical tracker until a revised fix lands.

## What This Changes

Make isolated cron run status deterministic for recovered agent-turn tool errors so equivalent fallback runs are not classified differently based on payload order.

## Best Possible Solution

Keep this issue as the canonical tracker and land a revised cron classifier fix that uses an explicit recovery signal instead of incidental payload ordering while preserving fatal provider and claimed-success tool failures.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: feed `resolveCronPayloadOutcome` an error payload followed by success to get nonfatal, then place the same success before a trailing non-message error to get fatal, and `finalizeCronRun` maps that directly to `ok` or `error`.

## Solution Assessment

Unclear. The linked PR's final-assistant-text heuristic was reviewed as too weak; the safer fix is a narrow recovery/acknowledgement signal that does not hide genuinely fatal trailing tool or provider errors.
