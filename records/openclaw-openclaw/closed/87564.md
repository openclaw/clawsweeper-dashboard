## Summary

Keep open: current main and v2026.5.26 still collapse empty dot-path segments, and the linked fix PR is open and explicitly closes this issue. The right cleanup path is to review and land or close https://github.com/openclaw/openclaw/pull/87565, then close this issue after the fix is on main.

## What This Changes

The issue asks for `openclaw config get/set/unset` to reject malformed dot paths with empty or whitespace-only segments instead of normalizing them to a different config key.

## Best Possible Solution

Land a narrow CLI parser fix that rejects empty or whitespace-only dot segments before any config read/write, preserves escaped dots and bracket indexes, and includes regression coverage for get, set, and unset.

## Reproduction Assessment

Yes. Source inspection shows `parsePath` skips empty segments and `normalizeStringEntries` filters blanks before config get/set/unset uses the parsed path; the issue and linked PR also provide real CLI before/after commands.

## Solution Assessment

Yes. The linked PR’s approach is the narrow maintainable fix shape: reject malformed segments in the existing CLI parser while preserving the documented valid path forms and adding focused regression tests.
