## Summary

Keep open: the linked display problem is source-reproducible, but this PR changes a fresh-only helper that current runtime compaction paths still depend on and the PR body still does not show the patched `/context` or status behavior running in a real setup.

## What This Changes

The PR changes stale session token handling by returning preserved totals from `resolveFreshSessionTotalTokens`, adjusts related tests and one fork-runtime guard, and also touches changelog plus Telegram message-cache typing code.

## Best Possible Solution

Keep `resolveFreshSessionTotalTokens` fresh-only for runtime decisions, and add a separate preserved-total display/search path for `/context`, status, Control UI, and session listing with regression coverage proving compaction and fork paths still re-estimate stale totals.

## Reproduction Assessment

Yes. Source inspection shows stale-but-preserved finite `totalTokens` can exist after the merged preservation change, while display callers still pass through the fresh-only helper and current tests assert stale totals render as unknown.

## Solution Assessment

No. The display behavior is worth fixing, but globally broadening `resolveFreshSessionTotalTokens` is not the narrowest maintainable solution because runtime compaction/fork callers still depend on fresh-only semantics.
