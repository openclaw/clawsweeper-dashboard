## Summary

Keep open for human review. The Slack hook coverage gap is still source-reproducible on current main, but this draft PR is not mergeable as-is because it misses the default native-streamed reply path, lacks real behavior proof, and adds a bundled guard with account-scoping and rewrite-boundary bugs.

## What This Changes

The branch wraps Slack `deliverReplies()` sends in `message_sending` hooks and adds a bundled `slack-addressee-guard` plugin with manifest config and tests.

## Best Possible Solution

Land a focused Slack hook seam that covers normal, block/media, preview, and native-streamed reply delivery without double-firing, then split or rework the addressee guard as an account-aware plugin with explicit product approval or as a ClawHub follow-up.

## Reproduction Assessment

Yes, by source inspection. Current main still sends normal Slack replies directly through `sendMessageSlack()` and default native-streamed replies through `startSlackStream()`/`appendSlackStream()` without a hook-bearing seam; I did not run a live Slack workspace reproduction.

## Solution Assessment

No. The hook direction is useful, but the PR only wraps `deliverReplies()`, misses native streaming, and bundles a guard that needs account-scoped credentials plus a product-boundary decision.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Cover native streamed replies with hooks:** `extensions/slack/src/monitor/replies.ts:48-49`
  - body: Slack defaults to partial native streaming, and that path sends through `startSlackStream()` and `appendSlackStream()` instead of this `deliverReplies()` wrapper. Wrapping only `deliverReplies()` leaves default visible agent replies invisible to `message_sending` policy plugins.
  - confidence: 0.9
- **[P2] Resolve Slack credentials for the sending account:** `extensions/slack-addressee-guard/index.ts:294`
  - body: The guard reads only `slackTokenOverride` or `SLACK_BOT_TOKEN`, but the actual Slack send is account-scoped and can use configured account credentials or SecretRefs. Multi-account installs can query the wrong workspace or fail open while sending with a different token.
  - confidence: 0.89
- **[P2] Require a leading target mention before rewriting:** `extensions/slack-addressee-guard/index.ts:351`
  - body: The guard activates when the configured target is mentioned anywhere, but `repairContent()` prepends the last human when that mention is not leading. A normal reply like `I checked with <@target>` would be rewritten even though the plugin is scoped to accidental leading addresses.
  - confidence: 0.9
- **[P2] Accept Slack DM and private-channel IDs:** `extensions/slack-addressee-guard/index.ts:130-131`
  - body: `parseChannelAndThread()` rejects conversation IDs that do not start with `C`, so Slack DMs, MPIMs, and private channels using `D...` or `G...` silently skip the guard even though Slack routing supports those surfaces.
  - confidence: 0.85
- **[P3] Remove or implement the logPath setting:** `extensions/slack-addressee-guard/openclaw.plugin.json:20`
  - body: The manifest exposes `logPath` and says it controls JSONL decision logging, but the implementation never reads that setting or writes decision records. Either implement the promised logging or remove the field before shipping the plugin.
  - confidence: 0.88
- **[P3] Add label routing for the new Slack plugin:** `extensions/slack-addressee-guard/openclaw.plugin.json:2`
  - body: This adds a new bundled Slack-related plugin under `extensions/slack-addressee-guard/**`, but current labeler rules only route Slack changes under `extensions/slack/**`. Add a Slack or plugin-specific mapping so future changes are routed consistently.
  - confidence: 0.82
