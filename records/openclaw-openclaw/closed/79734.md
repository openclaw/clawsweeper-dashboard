## Summary

Keep open: the feature is not implemented on current main, and the PR still has blocking dry-run safety issues around secret exposure, repair-mode side effects, and missing proof of the patched no-write behavior.

## What This Changes

The PR adds a `doctor --dry-run` CLI flag, threads it through doctor options/config flow/write guards, emits proposed config diffs, updates doctor docs, and adds finalize-flow tests.

## Best Possible Solution

Implement dry-run as an end-to-end read-only doctor planning mode with redacted original-vs-candidate output, effective repair mode disabled for all prompters/contributions, guards on every write or side-effect path, focused regression tests, and real `openclaw doctor --fix --dry-run` no-write proof.

## Reproduction Assessment

Yes. Source inspection of the PR head and current helper contracts shows the raw diff serialization, repair-mode prompter behavior, and `shouldRepair` mutation baseline issue without needing to run a mutating doctor flow.

## Solution Assessment

No. A doctor-level dry-run flag is a plausible solution, but this implementation is not safe until dry-run is read-only across the whole doctor flow and renders redacted diffs from an immutable baseline.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Redact dry-run diff values before printing:** `src/commands/doctor/finalize-config-flow.ts:25-29`
  - body: The diff renders arbitrary config values with `JSON.stringify` and sends them to `note()`. If doctor proposes changes under token/API-key/SecretRef-like paths, `--dry-run` leaks secrets into terminal logs; build the diff from redacted objects or mask sensitive paths before rendering.
  - confidence: 0.93
- **[P1] Force dry-run out of repair mode before prompting:** `src/flows/doctor-health.ts:11-13`
  - body: This only skips the Nix write assertion; `createDoctorPrompter({ options })` still sees `--fix`/`--repair`, so health contributions that call `confirmAutoFix()`, `confirmRuntimeRepair()`, or check `prompter.shouldRepair` can still write auth, cron, plugin, state, UI, or service files during dry-run. Derive read-only effective options/repair mode when `dryRun` is true.
  - confidence: 0.91
- **[P2] Diff dry-runs against an immutable original config:** `src/commands/doctor-config-flow.ts:70-72`
  - body: `--fix --dry-run` leaves `shouldRepair` true, and existing mutation plumbing assigns `cfg` to each proposed mutation before finalization. The dry-run finalizer can then compare the already-mutated `cfg` to `candidate` and print no changes for mutations that would be applied; keep the original baseline or collect mutations in effective non-repair mode.
  - confidence: 0.89
- **[P3] Show the clean dry-run result when nothing is pending:** `src/commands/doctor/finalize-config-flow.ts:13`
  - body: The dry-run branch is gated on `pendingChanges`, so a clean `openclaw doctor --dry-run` falls through without the documented `No config changes detected.` note. Enter the dry-run branch whenever `dryRun` is true; the current no-changes test still passes `pendingChanges: true`.
  - confidence: 0.82
