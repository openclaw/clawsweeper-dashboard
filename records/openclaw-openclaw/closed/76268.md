## Summary

Current main already uses a structural plugin-loader cache key rather than hashing the full root config, and request-time runtime registry resolution reuses a compatible active registry before falling back to `loadOpenClawPlugins`.

## What This Changes

The issue asks for plugin registry cache keys to ignore bookkeeping fields like `meta.lastTouchedAt` so `core-plugin-tools` does not reload plugins on every request.

## Best Possible Solution

Keep the current structural cache-key and active-registry reuse behavior, then let the next release carry this main-only fix to users on the latest stable build.

## Reproduction Assessment

No. The reported full-config cache-key mechanism does not reproduce on current main; the high-confidence verification is source and regression-test inspection rather than a live timing run.

## Solution Assessment

Yes. The current structural cache key is the narrow maintainable solution because it ignores unrelated root bookkeeping while preserving invalidation for actual plugin config and activation inputs.
