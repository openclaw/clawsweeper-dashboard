## Summary

Keep open. Current main does not have `openclaw proxy validate`, and the PR is a useful follow-up to the merged operator-managed proxy work, but the latest head still has one blocking functional defect: schema-invalid configured `proxy.proxyUrl` values fail through global config loading before the new command can emit the promised text/JSON validation result. The earlier proxy URL suffix redaction concern appears addressed in the latest head, with tests covering userinfo, query, and fragment redaction.

## What This Changes

This PR adds `openclaw proxy validate`, a proxy validation runtime for proxy URL resolution and allowed/denied destination checks, CLI/security docs, tests, and a changelog entry.

## Best Possible Solution

Keep the validation command direction, but make the CLI command handle invalid configured proxy URLs inside the validation formatter path so `--json` and text output remain structured for proxy-specific config failures while preserving global fail-closed config loading elsewhere.

## Reproduction Assessment

Yes. On the PR head, configure `proxy.enabled=true` with a schema-invalid `proxy.proxyUrl` such as an `https://` proxy URL and run `openclaw proxy validate --json`; the command reaches `getRuntimeConfig()` before `runProxyValidation`, so the global config loader throws before the new JSON formatter can report the invalid proxy URL.

## Solution Assessment

No. The validation engine, deny canary semantics, docs, and proxy URL redaction are in the right direction, but the CLI entrypoint should not use the normal throwing runtime config path for the specific invalid proxy URL states it promises to report.
