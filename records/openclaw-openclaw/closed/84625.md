## Label Justifications

- `P2`: The report describes a real but limited WhatsApp secondary-account group delivery failure, not a core outage.
- `impact:message-loss`: The reported symptom is assistant text being generated but not delivered back to the originating WhatsApp group.
- `impact:auth-provider`: The failure mode depends on resolving the non-default WhatsApp account id for outbound delivery.

## Summary

Current main and the shipped v2026.5.18 source already carry the WhatsApp account-scoped group route through auto-reply delivery, so the 2026.5.12 report is addressed by upgrading.

## What This Changes

The issue reports failed WhatsApp auto-reply delivery for secondary-account group sessions whose session key includes a `:thread:whatsapp-account-<name>` suffix.

## Best Possible Solution

Keep the shipped v2026.5.18/current-main implementation and ask for a fresh report only if the same flow still fails after upgrading with gateway logs.

## Reproduction Assessment

No live two-account WhatsApp group was run, so there is not a high-confidence live reproduction path here. Source and existing tests do cover the reported session-key/account route and show current main takes the fixed path.

## Solution Assessment

Yes. The maintainable fix is already in the WhatsApp plugin boundary: route construction owns the account-scoped group session key and final delivery uses the resolved route account id.
