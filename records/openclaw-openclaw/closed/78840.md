## Summary

Close as implemented on current main: the reported v2026.5.6 startup phases map to code that now makes model prewarm fire-and-forget/bounded, moves session-lock cleanup out of the startup gate, and keeps Feishu WebSocket recovery on a retrying backoff loop. The fix is main-only right now and is not included in v2026.5.6.

## What This Changes

The issue asks for gateway startup model prewarm and session-lock cleanup to stop blocking Feishu WebSocket heartbeats during startup.

## Best Possible Solution

Keep the current main startup-sidecar behavior and let it ship in the next release; no new Feishu-specific startup workaround is needed beyond the existing reconnect loop.

## Reproduction Assessment

No current-main failing repro was established. The report includes precise v2026.5.6 logs for `sidecars.model-prewarm` and `sidecars.session-locks`, and current-main source plus focused tests show those phases no longer block channel startup or readiness in the same way.

## Solution Assessment

Yes. The current fix is the narrow maintainable direction: keep startup prewarm bounded and non-gating, move session-lock cleanup out of the ready path, and rely on Feishu's existing retrying WebSocket monitor instead of adding channel-specific heartbeat policy.
