## Label Justifications

- `P1`: The PR changes final delivery behavior for real agent/channel workflows and currently has a plausible wrong-recipient failure mode.
- `merge-risk: 🚨 message-delivery`: Merging as written could deliver final payloads to a fresh route that belongs to another logical session.
- `merge-risk: 🚨 session-state`: The patch trusts refreshed session-store state without proving it is the same session state as the run being finalized.
- `merge-risk: 🚨 security-boundary`: Wrong-route final delivery can expose private agent output to the wrong channel, thread, or account.

## Summary

Keep open: the refresh idea is focused and now has credible live Discord proof, but the patch accepts any fresh same-key session entry without checking that it belongs to the same logical session, which can reroute final payloads across session rotations.

## What This Changes

The PR adds a final-delivery routing reread from fresh session-store state before no-send failures, updates agent delivery coverage/runtime exports, and adds a changelog entry.

## Best Possible Solution

Keep the refresh-before-no-send behavior, but accept refreshed routing only when the fresh entry matches the original session identity or an equivalent same-run invariant.

## Reproduction Assessment

No high-confidence current-main live repro was established here. Source inspection shows current main fails closed after one stale routing resolution, and the PR body supplies after-fix live Discord proof for the refreshed route path.

## Solution Assessment

No, not as written. Refreshing once before a no-send failure is the right boundary, but the implementation must prove the fresh route belongs to the same logical session before sending.
