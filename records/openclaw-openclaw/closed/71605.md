## Summary

Current main implements the two observable failure mitigations: Gateway `agent` requests now acknowledge with a run id before deferred heavy dispatch, and CLI Gateway-timeout fallback now forks into an explicit fresh `gateway-fallback-*` session instead of racing the Gateway-owned transcript lock. The fix is on main but I could not prove it is included in the latest stable tag.

## What This Changes

The issue asks for Gateway `agent` RPC dispatch to return promptly and for embedded fallback or local agent execution to avoid confusing session-file lock contention with a running daemon, or to document the limitation.

## Best Possible Solution

Keep the current split implementation: Gateway should acknowledge accepted agent runs before heavy dispatch, and CLI timeout fallback should be explicit about forking to a fresh session rather than contending with the daemon-owned transcript. Any future report of a different starvation source should be tracked with fresh logs rather than reopening the already-fixed fallback lock race.

## Reproduction Assessment

Yes for the historical failure path: the issue gives concrete `openclaw agent --session-id ... --timeout 30 --json` steps and the old code path reused the original session after a Gateway timeout. Current main cannot be live-tested here without writing session state, but source and tests now cover the relevant timeout-fallback race.

## Solution Assessment

Yes. The current implementation is the narrowest maintainable fix: it flushes/defers the Gateway accepted handoff and isolates only timeout fallback sessions, avoiding broad changes to the session-lock contract or daemon/local state layout.
