## Summary

Close as implemented on main: current OpenClaw already handles Telegram long replies through the outbound chunking path and already repairs enabled bundled-plugin runtime dependencies before import, so this PR is superseded by shipped behavior on `main`.

## Risks / Open Questions

- `main` solves Telegram long-output handling via chunking, not via the PR's summary-and-button transcript callback; if that exact UX is still desired, it would need a new scoped proposal.
- The supported startup self-repair path is intentionally limited to bundled/OpenClaw-owned plugin runtime dependencies; third-party plugin repair remains explicit by design.
