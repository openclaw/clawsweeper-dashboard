## Summary

Current `main` and the latest release already have `cron list`, `cron status`, and `cron get` routed through the same loaded cron store, and the CLI does not add an agent filter unless `--agent` is explicitly passed, so the reported 2026.6.1 mismatch is addressed by the current implementation.

## What This Changes

The issue reports that `openclaw cron list` returns fewer cron jobs than `openclaw cron status` and `cron get`, causing enabled scheduled jobs to be omitted from listing output.

## Best Possible Solution

Keep the current shared-store cron list/status/get implementation and ask for a fresh latest-release report only if the mismatch still reproduces with exact `cron list --json` and `cron status` output.
