## Summary

Current main already covers the central request: channel/account lifecycle failures are isolated, restarted with bounded exponential backoff, and eventually stopped instead of retrying forever. Telegram invalid-token/auth failures are not classified as recoverable polling network errors, so they surface to that channel-level guard; Telegram also has a narrower 401 backoff/suspension guard for sendChatAction.

## Best Possible Solution

Close this issue as implemented by the shipped generic channel/account lifecycle guard plus Telegram-specific auth retry handling. Keep the current implementation; if maintainers want the exact proposed threshold of 5 or an auth-specific configurable counter, that should be a narrower follow-up product/config issue rather than keeping this retry-storm report open.
