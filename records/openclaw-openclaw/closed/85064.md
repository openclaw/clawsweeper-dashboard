## Best Possible Solution

Land a segment-aware filename-only guard that rejects separators and `.`/`..` path segments, preserves legal bare filenames and the full-path override, and includes Windows upgrade proof.

## Reproduction Assessment

## Solution Assessment

No, not as written: the filename-only direction is right, but substring matching `..` is broader than the security boundary requires. The safer fix is segment-aware validation plus explicit proof that full-path users can keep using `OPENCLAW_TASK_SCRIPT`.
