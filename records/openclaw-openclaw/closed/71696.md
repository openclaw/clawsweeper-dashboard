## Summary

Keep open: current main does not implement Telegram chat-type reply modes, and the PR contains useful Telegram reply-threading work, but the latest head still makes omitted per-chat settings override runtime/account `replyToMode` fallbacks and needs release-note coverage before merge.

## What This Changes

The PR adds Telegram `replyToModeByChatType`, applies it to normal and native command delivery, restores explicit reply threading, and passes Telegram message IDs into plugin command contexts.

## Best Possible Solution

Land a repaired version that keeps the per-chat config optional, preserves fallback precedence, regenerates metadata/tests, and documents the Telegram behavior.

## Reproduction Assessment

Yes. On the PR head, a Telegram config with only `replyToMode` or only a runtime fallback gets a parsed `replyToModeByChatType` default, and the resolver returns that default before the fallback.

## Solution Assessment

No. The feature belongs in Telegram config/delivery, but direct/group defaults should be the resolver's final fallback rather than schema-level required config.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve replyToMode fallback before per-chat defaults:** `src/config/zod-schema.providers-core.ts:237-240`
  - body: The schema default makes every parsed Telegram config look like it explicitly set `{ direct: "off", group: "all" }`. Because the resolver checks `replyToModeByChatType` before `fallbackReplyToMode` and account `replyToMode`, existing configs or runtime overrides such as `replyToMode: "first"` are ignored after validation. Keep this field optional in the schema and apply direct/group defaults only as the resolver's final fallback.
  - confidence: 0.9
