## Summary

Keep open: the report identifies a source-backed remaining Codex OAuth runtime bug, not the older refresh-handshake bug that v2026.5.7 fixed. Current main still resolves and injects only the access token into embedded Codex request paths, while the stored `accountId` is not carried into request-time Codex auth.

## What This Changes

Ensure Codex/OpenAI Codex embedded agent runs keep using stored ChatGPT account IDs after refreshed OAuth access tokens no longer carry `chatgpt_account_id`.

## Best Possible Solution

Carry the stored OpenAI Codex `accountId` as prepared runtime auth metadata and use it when building native Codex request auth/headers instead of relying on refreshed JWT claims.

## Reproduction Assessment

Yes, from source inspection: current main preserves `accountId` in stored OAuth credentials but drops it before embedded Codex request transport, and the pinned pi-ai Codex transport throws when the access-token JWT lacks the account claim. I did not live-run the ChatGPT OAuth refresh flow.

## Solution Assessment

Yes, the best fix is to carry the existing stored `accountId` through the runtime auth plan and request transport, with focused regression coverage; catching this later in failover would keep the Codex path slow or unusable.
