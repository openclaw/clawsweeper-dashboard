## Summary

Keep open: the PR targets a real source-provable config persistence bug, but the normal CLI write path still loses the new explicit-path option in the exported writer, and the supplied proof is helper-level rather than a real CLI/file write.

## What This Changes

The PR adds explicit config write paths from config set/patch operations into config write preparation so default-equal user writes can be persisted.

## Best Possible Solution

Forward explicitSetPaths through the exported writeConfigFile path, including the runtime-snapshot projection path, then add regression coverage and real CLI/file persistence proof for default-valued config writes.

## Reproduction Assessment

Yes, source-reproducible. The linked issue gives concrete CLI steps, and current main shows default-valued writes can disappear when persistence diffs nextConfig against runtimeConfig defaults; I did not run live writes in this read-only review.

## Solution Assessment

No. The helper approach is a good direction, but this patch is incomplete until explicitSetPaths survives the exported writer path that normal CLI writes use and is proven through an end-to-end write path.
