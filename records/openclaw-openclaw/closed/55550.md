## Summary

Close as superseded. Current main already moved Telegram health reporting onto the shared runtime-backed health snapshot path that preserves lifecycle state and credential metadata, while this PR's Telegram-only fallback would incorrectly treat a successful bot API probe as proof that the polling/webhook runtime is running. The remaining cached `health --json` mismatch is tracked separately in #59287.

## What This Changes

The PR changes Telegram status summaries to backfill `tokenSource` from the resolved account, infer `running` from successful probes when snapshots omit it, and add regression tests for that fallback behavior.

## Best Possible Solution

Keep the shared runtime-backed health contract as the source of truth for `running`, keep credential metadata projected through account inspection, and handle any remaining cached-health behavior in #59287 rather than adding Telegram-only probe-derived lifecycle state.

## Reproduction Assessment

Yes. The old failure can be modeled with a configured Telegram account whose health snapshot lacks `running` and `tokenSource`; current main now fills those through inspected account metadata plus gateway runtime snapshots, while the PR's branch can be reviewed directly from its diff.

## Solution Assessment

No. This PR is not the best solution because a successful Telegram probe proves token/API reachability, not that the polling or webhook runtime is active; the merged shared runtime-backed health path is the safer maintainable fix.
