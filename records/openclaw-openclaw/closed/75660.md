## Summary

Current main already covers the reported old-version Discord gateway failure mode: the bundled Discord plugin no longer depends on discord.js for inbound gateway liveness, records close/reconnect/READY failures, heartbeat-ACK timeouts trigger reconnect, transport activity feeds the channel health monitor, and unhealthy disconnected or stale sockets are restarted instead of requiring a full gateway restart.

## What This Changes

The issue requests logging and recovery for silent Discord WebSocket drops that stop inbound Discord messages while outbound REST sends still work.

## Best Possible Solution

Keep the shipped Discord gateway lifecycle and health-monitor implementation; future work should be opened only with current-version logs showing the READY/watchdog/health-monitor paths failing to detect a live silent transport loss.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a live Discord/NAT reproduction in this read-only pass, but the issue’s 2026.4.15 log timeline maps directly to current covered code paths and regression tests for heartbeat, close/reconnect, READY timeout, and health-monitor restart behavior.

## Solution Assessment

Is this the best way to solve the issue? Yes; handling this in the Discord plugin gateway lifecycle plus the generic channel health monitor is the narrow maintainable fix, because it avoids core Discord-specific policy while still restarting stale or disconnected channel runtimes.
