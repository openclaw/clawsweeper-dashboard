The PR changes daemon gateway diagnostics so non-Darwin restart error selection prefers stderr matches over stale stdout matches and adds a Linux regression test.

## Best Possible Solution

Land the focused helper-order change after ordinary maintainer review, keeping `readLastGatewayErrorLine` as the single source for status, doctor, onboard, and status-all report diagnostics.

Yes. Source inspection gives a high-confidence reproduction path: seed matching stdout and stderr gateway logs on Linux and call `readLastGatewayErrorLine`; current main checks stdout first because it is appended after stderr and scanned from the end.

## Solution Assessment

Yes. The helper owns the selection that all visible callers print, so reversing the combined log order is the narrow maintainable fix; timestamp parsing would be a larger contract change than the current stderr-priority behavior needs.
