## Summary

Keep open: this draft maintainer PR is useful but not merge-ready because the new isolation path appears to turn npm integrity-drift aborts into skipped plugin outcomes, letting package update succeed after a supply-chain mismatch.

## What This Changes

This PR adds a post-core `openclaw update` plugin failure-isolation mode, skips disabled missing plugin payloads, adds focused tests, and records a changelog entry.

## Best Possible Solution

Keep isolation for ordinary broken plugin package updates, but preserve fatal outcomes for integrity drift and security/policy blocks, then align focused tests and update docs with that boundary.

## Reproduction Assessment

Yes. Source inspection is enough: rejected npm integrity drift returns `ok: false`, and the PR's `recordFailure` would convert that failure to a skipped disabled plugin when post-core update passes `disableOnFailure: true`.

## Solution Assessment

No. The current approach is too broad because it treats supply-chain integrity aborts the same as ordinary install failures; the safer fix is to classify those failures before converting outcomes to skipped.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep integrity drift failures fatal:** `src/plugins/update.ts:774-789`
  - body: This helper is used for every installer failure when `disableOnFailure` is enabled by post-core `openclaw update`, including the `aborted: npm package integrity drift detected...` result produced when exact-pinned npm artifact integrity changes. Rewriting that to `status: "skipped"` makes `runPostCorePluginUpdate` return ok because it only fails on error outcomes, so a package update can succeed and restart after a supply-chain mismatch. Keep integrity-drift/security failures as errors while isolating ordinary broken plugin packages.
  - confidence: 0.9
