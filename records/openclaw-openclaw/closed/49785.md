## Summary

Current `main` already preserves Slack `replyToMode: "first"` threading for multi-part block deliveries, with a narrower block-only fallback plus regression coverage, so PR #49785 is superseded.

## Evidence

- **Block-only thread reuse on current main:** `dispatchPreparedSlackMessage` now resolves delivery thread targets by preferring an explicit/planned thread and then falling back to `usedBlockReplyThreadTs` only for `kind === "block"`. That preserves multi-block threading in `first` mode without forcing later non-block replies into the thread.
