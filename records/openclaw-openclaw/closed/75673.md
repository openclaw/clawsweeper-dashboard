## Summary

Current main already prevents the reported disabled-qqbot path from touching the bundled runtime-deps mirror: startup staging is scoped to selected startup plugins, qqbot opts out of startup activation, and explicit plugin/channel disablement wins before default-enabled dependency repair.

## What This Changes

The issue asks for macOS packaged Gateway restarts to stop failing on a qqbot `plugin-runtime-deps` cache permission error when qqbot is disabled.

## Best Possible Solution

Keep the current main runtime-deps scoping behavior and ship it in the next release; users on 2026.4.26 should update to a build containing the current main fix rather than chmodding the cache after each restart.

## Reproduction Assessment

Yes. The high-confidence source-level reproduction is a disabled bundled channel/plugin config with packaged runtime-deps staging; current main filters that plugin out before repair or mirror preparation, and tests cover the disabled-channel plan behavior.

## Solution Assessment

Yes. Scoping repair and mirror preparation to selected startup/configured plugins is the narrow maintainable fix because it prevents disabled qqbot from touching the cache without broadening filesystem permissions or adding a chmod workaround.
