## Summary

Close as duplicate/superseded: the same plugin SDK redispatch gap is already tracked by the open canonical issue at https://github.com/openclaw/openclaw/issues/77147, and current main confirms this report is a narrower takeover/dedupe variant of that remaining work.

## What This Changes

Request to add a `before_dispatch` takeover/deferred-dispatch contract so plugins can queue an inbound message and later redispatch it without committing inbound dedupe.

## Best Possible Solution

Keep https://github.com/openclaw/openclaw/issues/77147 as the canonical design thread and preserve this WhatsApp FIFO/dedupe context by linking it there.

## Reproduction Assessment

Yes for current behavior by source inspection: `{ handled: true }` in `before_dispatch` records completion and commits inbound dedupe. I did not run a live WhatsApp plugin because the remaining ask is a new SDK capability, not a broken documented behavior.

## Solution Assessment

No for the standalone takeover flag as the complete solution; the safer path is the broader trust-gated redispatch contract already tracked in the canonical issue.
