## Summary

Keep open: current main and v2026.6.9 still have the reported `message_tool_only` accounting gap, and the open linked pull request is the live fix candidate rather than a reason to close the issue before merge.

## What This Changes

The issue requests that Discord `message_tool_only` source replies delivered by `message(action=send)` count as user-facing replies so stale tool-failed warnings are not appended after a successful send.

## Best Possible Solution

Land or close the canonical fix at https://github.com/openclaw/openclaw/pull/94072; after it merges, close this issue with the fixing commit and release provenance.

## Reproduction Assessment

Yes at source level: current main can set `deliveredSourceReplyViaMessageTool` while `hasUserFacingAssistantReply` remains false, causing the warning policy to treat the turn as having no user-facing reply. I did not run a live Discord reproduction in this read-only review.

## Solution Assessment

Yes. Counting the existing runner-owned delivery fact in the payload warning gate is the narrowest maintainable fix; Discord-specific warning logic would duplicate ownership and weakening tool-error warnings globally would be broader than needed.
