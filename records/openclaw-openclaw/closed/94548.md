## Label Justifications

- `P1`: The reported release regression makes Telegram agent replies unreadable on real user clients, which breaks a major channel workflow.
- `impact:message-loss`: Readable Telegram reply content is replaced by client-side unsupported-message placeholders, effectively losing the delivered message for affected users.

## Summary

Current main already restores readable Telegram text delivery by making Bot API 10.1 rich messages explicit opt-in and accepting the documented `richMessages` config key; the reported v2026.6.8 release remains affected, but the central problem is fixed on main with source, test, docs, and history proof.

## What This Changes

The issue reports that Telegram plain text replies are sent through `sendRichMessage` in v2026.6.8, making Telegram Web and Apple Watch unreadable and leaving the documented `richMessages` off-switch absent from the shipped schema.

## Best Possible Solution

Keep the current-main default-off rich-message implementation from https://github.com/openclaw/openclaw/pull/93279 and ship it in the next patch release rather than adding a second Telegram delivery switch.

## Reproduction Assessment

Yes. Source inspection reproduces the shipped v2026.6.8 problem: the release tag lacks the documented `richMessages` key and routes normal text through rich sends, while related reports and Mantis proof show the client-side unsupported rendering.

## Solution Assessment

Yes. Current main is the best fix shape: standard `sendMessage`/HTML remains the default for compatibility, Bot API 10.1 rich messages are explicit opt-in, and docs/tests now match that contract.
