## Summary

Current main and the shipped v2026.4.27 release already cover the reported DeepSeek V4 replay failure: the DeepSeek provider wraps V4 OpenAI-compatible payloads, enables DeepSeek thinking, and backfills missing `reasoning_content` on replayed assistant messages, with unit, live-regression, docs, and changelog evidence. The issue discussion and the follow-up comment from zilvercat match the same /think multi-step failure mode now covered by that implementation.

## What This Changes

Request to preserve/replay DeepSeek V4 `reasoning_content` for multi-turn thinking/tool conversations so follow-up OpenAI-compatible requests do not fail with provider 400s.

## Best Possible Solution

Keep the shipped DeepSeek provider wrapper, tests, and docs as the maintained solution; it preserves core/provider boundaries and avoids making OpenAI Responses replay conversion DeepSeek-specific.

## Reproduction Assessment

Yes. The report gives a concrete DeepSeek V4 thinking/multistep path, the upstream DeepSeek docs explain the same 400 condition, and current main has a live regression scenario for tool-call replay plus a follow-up turn.

## Solution Assessment

Yes. The implemented fix is the narrowest maintainable path because DeepSeek-specific replay shaping stays in the DeepSeek provider wrapper while the shared helper remains provider-family scoped and tested.
