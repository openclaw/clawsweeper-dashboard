## Summary

Current main resolves the reported built-in `codex-cli` false-success path by removing the bundled direct Codex CLI backend and routing legacy Codex CLI refs to the Codex app-server runtime. The earlier structured-JSONL fix attempt is useful context, but the supported core path no longer executes `codex exec --json` through a bundled `codex-cli` backend.

## What This Changes

The issue asks OpenClaw to detect sandbox-denied Codex CLI JSONL runs that exit 0 and treat them as failed instead of successful.

## Best Possible Solution

Keep the Codex app-server migration as the supported OpenAI Codex path, and treat any future false-success report for an explicit custom CLI backend as a new plugin-owned or generic CLI contract issue with current-main config evidence.

## Reproduction Assessment

No current-main reproduction remains for the reported built-in `codex-cli` path. Current source and docs show that OpenClaw no longer bundles that backend and migrates legacy Codex CLI refs to the Codex app-server route.

## Solution Assessment

Yes, for the reported supported core path. Removing the bundled direct Codex CLI route is the narrower maintainable core solution; Codex-specific JSONL failure parsing should only return if a supported plugin/runtime contract needs it.
