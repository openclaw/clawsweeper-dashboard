## What This Changes

The issue reports a MemoryIndexManager.close()/sync race where embedding provider and SQLite resources could be closed while sync work was still initializing or running.

## Best Possible Solution

Keep the merged PR 86765 implementation and close this as shipped; any new memory lifecycle race should be filed with a fresh reproduction against current main or v2026.5.26.

## Reproduction Assessment

Yes for the original bug at source level: the report identified the close/sync interleaving, and current tests encode both pending-sync and provider-init race cases. Current main should not reproduce the failure because the close path now waits for the active sync before provider and DB teardown.

## Solution Assessment

Yes. The merged fix is the narrow maintainable solution: make sync visible before provider initialization and defer provider/SQLite teardown until the current sync has settled, without adding config or product surface.
