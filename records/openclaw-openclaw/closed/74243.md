## Summary

Keep open: the PR targets a real shared media optimizer failure cluster, but the latest fallback is still too broad for unvalidated `image/*` inputs when the optimizer backend is unavailable, and the user-facing media fix is missing a changelog entry.

## What This Changes

The PR adds an under-cap non-HEIC original-image fallback when shared image optimization fails, preserves optimizer causes, adds media regression tests, and marks Discord system-event text untrusted.

## Best Possible Solution

Land the shared media fallback only after original bytes have passed an explicit image input guard, then document the user-facing fix in `CHANGELOG.md`; the Discord untrusted-system-event change is directionally sound and can stay if the PR remains bundled.

## Reproduction Assessment

Yes. Current main has a source-level reproduction where optimizer failures abort `loadWebMedia`, while the remaining guard concern is source-reproducible from the `image/*` classification plus non-sips unknown-metadata return path when the optimizer is unavailable.

## Solution Assessment

No. The shared loader is the right place for MIME-preserving fallback, but the fallback needs a positive dimension/pixel preflight or a stricter supported-format gate before it is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Require a positive image guard before fallback:** `src/media/web-media.ts:407-418`
  - body: The fallback returns original bytes for any under-cap non-HEIC `image/*` after optimizer failure. On non-sips backends, `assertImagePixelLimit` returns when header dimensions are unknown, so a missing optimizer can bypass the intended decoded-pixel or unknown-dimension guard for unparsed image types. Require an explicit successful preflight, or limit fallback to formats whose dimensions were positively validated.
  - confidence: 0.88

  - body: This is a user-facing media fix intended to close active image-tool reports, but the PR does not update `CHANGELOG.md`. Add a single-line active-version Fixes entry before merge.
  - confidence: 0.96
