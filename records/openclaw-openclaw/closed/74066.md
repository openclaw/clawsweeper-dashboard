## Summary

Keep this issue open. Current main still calls `context.cron.add(...)` and `context.cron.update(...)` without a cron-pattern validation/catch boundary, while Croner parsing still happens downstream during next-run computation. The provided GitHub context and `git ls-remote` also show PR #74068 is the open implementation candidate with closing syntax, so this issue should stay open until that PR or an equivalent fix lands.

## Best Possible Solution

Keep this issue open and review/land PR #74068, or an equivalent narrow gateway-side validation fix. The fix should validate `schedule.kind === "cron"` expressions for `cron.add` and `cron.update` before service mutation, return `ErrorCodes.INVALID_REQUEST` with sanitized user-facing text, preserve valid cron/every/at behavior, and add regression tests for bad field count, out-of-range fields, invalid timezone, and defensive Croner parse failures.
