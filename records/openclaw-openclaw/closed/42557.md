## Summary

Current `main` already fixes the reported `exec host=node` cross-platform cwd/workdir bug by bypassing gateway-side cwd fallback for node exec, forwarding only an explicitly requested workdir, and covering that behavior with targeted tests.

## Evidence

- **Node exec skips gateway workdir resolution:** `src/agents/bash-tools.exec.ts` now has a dedicated `host === "node"` branch that sets `workdir = explicitWorkdir` and explicitly documents avoiding the cross-platform `SYSTEM_RUN_DENIED: approval requires an existing canonical cwd` failure instead of calling `resolveWorkdir(...)`.
  - file: [src/agents/bash-tools.exec.ts:1519](https://github.com/openclaw/openclaw/blob/d29eaeafc1fc1ca86f0ccb5c6b2b3d3668502132/src/agents/bash-tools.exec.ts#L1519)

## Close Comment

Closing as implemented on current `main`. Codex review verified that `src/agents/bash-tools.exec.ts:1519` now special-cases `host=node` and forwards only an explicitly requested `workdir`, instead of resolving/falling back through the gateway filesystem. `src/agents/bash-tools.exec-host-node.ts:109` only includes `cwd` in `system.run.prepare` when a workdir was actually provided, and `src/agents/bash-tools.exec.approval-id.test.ts:557` covers the exact regression by asserting the gateway default cwd is not forwarded to node exec. `git blame` ties this behavior to commit `6f04eee2a12e665fb4b0e3b0021dbbf90efdf2a2`, which is already contained in current `main`. I could not verify a released tag containing that commit yet, so this looks fixed on `main` but not necessarily shipped in `v2026.4.22`.
