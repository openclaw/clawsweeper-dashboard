## Summary

Keep open: current main still has a source-proven path where a subagent completion direct message can reach the external channel, fail later on transcript mirroring, be reported as undelivered, and then retry into duplicate user-visible sends.

## What This Changes

The issue reports duplicate Telegram delivery when subagent completion auto-announce direct delivery is retried after a session transcript lock conflict.

## Best Possible Solution

Preserve at-most-once user-visible completion delivery by crediting sent-before-mirror-failure as delivered and keeping transcript mirror recovery separate from external resend retries.

## Reproduction Assessment

Yes, from source inspection: the direct text fallback can send externally, then throw on transcript mirror failure, and the registry retry budget can call the announce flow again. I did not run the live Telegram scenario, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

Yes for keeping the issue open. The narrow maintainable fix is to make the retry decision distinguish external delivery success from transcript mirror failure, not add a config knob or suppress all retries.
