## Summary

Close as implemented on current main. The reported infinite Anthropic 400 loop is now handled before provider replay by repairing the corrupted assistant-only transcript under the session write lock; the separate reset-readiness greeting concern remains tracked by #75314. The fix is present on main but not in a tag I could prove shipped.

## What This Changes

The issue requests recovery for leading-assistant/no-user corrupted session transcripts that loop on Anthropic empty-message 400s, plus clearer ready feedback after `/new` or auto-reset.

## Best Possible Solution

Keep the current read-time session-file repair as the primary fix for this corruption class, because it prevents the bad transcript from reaching any provider. Handle the separate fresh-session greeting or ready-signal UX in #75314 rather than reopening the provider-loop bug.

## Reproduction Assessment

Yes. The issue gives a concrete assistant-only/no-user JSONL shape, and current tests cover the same class by trimming an orphan assistant-only transcript to the session header before replay.

## Solution Assessment

Yes. Repairing the transcript before provider submission is narrower and more durable than adding a transport-only empty-message fallback; the exact post-400 reset copy was not added, but the persistent loop is removed earlier in the pipeline.
