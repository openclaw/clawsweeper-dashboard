## Summary

Close as duplicate/superseded: current code already has provider-level request proxy support outside auth profiles, and the remaining per-account/profile routing design is covered by the open provider-profile routing issue.

## What This Changes

Requested adding an optional `proxyUrl` field to auth-profile credentials so each provider account can route model traffic through a dedicated proxy.

## Best Possible Solution

Keep auth profiles credential-only, use `models.providers.*.request.proxy` for direct provider proxy cases, and track per-account proxy/routing semantics in the canonical provider-profile routing design.

## Reproduction Assessment

Not applicable as a feature request. Source review shows current main lacks an auth-profile `proxyUrl` field while provider-level proxy config exists and the remaining per-profile routing is tracked in the canonical issue.

## Solution Assessment

No: adding `proxyUrl` directly to auth profiles is not the best standalone path because current docs and types keep auth profiles credential-only. The maintainable path is the broader provider-profile routing contract plus existing provider request proxy config.
