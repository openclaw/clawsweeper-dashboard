## Summary

Keep open: current main and the latest release still leave bundled MCP catalog `tools/list` on the MCP SDK default request timeout, while this PR targets that exact call site and removed the public config surface after maintainer feedback. I found no blocking correctness or security findings, but merge still carries the explicit compatibility choice of accepting an unconfigurable 1500 ms catalog-list cap.

## What This Changes

The PR adds an internal 1500 ms timeout to bundled MCP catalog `tools/list` discovery, adds slow-valid and hung catalog regression tests, and refreshes generated config baseline hashes.

## Best Possible Solution

Land the no-config internal timeout if maintainers accept the 1500 ms cutoff, or tune that single constant before merge while preserving the slow-valid and hung catalog tests.

## Reproduction Assessment

Yes at source level: current main and `v2026.5.22` call `client.listTools` without request options, and the pinned MCP SDK defaults requests to 60000 ms when no timeout is supplied. I did not run a local repro because this review is read-only and the focused runtime tests create temporary process artifacts.

## Solution Assessment

Yes, with maintainer acceptance of the cutoff: the PR bounds the exact post-connect catalog request without adding a public config key, matching the latest review direction. The safer alternative is only to tune the internal constant if maintainers prefer a different compatibility/UX tradeoff.
