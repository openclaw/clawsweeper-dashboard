## Summary

Current main already addresses the reported v2026.4.29 plugin-tool fanout by scoping tool-runtime plugin loads from manifest metadata before runtime import, so the release behavior that could materialize broad plugin runtime during `core-plugin-tools` is no longer present.

## What This Changes

The issue reports severe Windows Gateway event-loop blocking on v2026.4.29 when each message builds tools and appears to scan or initialize about 160 bundled plugins.

## Best Possible Solution

Keep the scoped plugin-runtime loading now on main and ship it in the next release so v2026.4.29 users can upgrade out of the Gateway stall.

## Reproduction Assessment

Yes. The report gives concrete v2026.4.29 Windows steps and timing, and the v2026.4.29 source has the unscoped plugin-tool runtime path that current main removes.

## Solution Assessment

Yes. Scoping plugin tool runtime from manifest metadata is the narrow maintainable fix because it prevents unrelated bundled plugins from loading during each agent turn without hardcoding MiniMax or Windows-specific behavior.
