## Summary

Keep this PR open, but do not merge it as-is. The observability gap still exists on current main, and the PR is directionally useful, but the diff instruments the lazy generated-entry catch while the runtime-deps lock timeout that disables bundled channels is caught later through the plugin-loader runtime path.

## What This Changes

The PR adds structured bundled-channel runtime-deps lock-timeout warning metadata, regression tests for timeout versus generic import-error logging, and a changelog entry.

## Best Possible Solution

Add a shared internal classifier for bundled runtime-deps lock timeouts and apply it at the plugin-loader runtime entry failure surface, preserving the human-readable timeout detail and current channel-skip behavior while adding structured metadata where log sinks can consume it.

## Reproduction Assessment

Unclear for a full production reproduction in this review. The PR test forces a timeout-shaped error into the lazy-entry catch, but source inspection shows the real runtime-deps acquisition path reaches the plugin loader after loadChannelPlugin() runs with default options.

## Solution Assessment

No. Catch-side classification is a reasonable direction, but this PR applies it to a catch branch that normally disables runtime-deps installation; the safer fix is to classify the timeout where the bundled runtime load error is recorded.
