## Summary

Keep open: current main still builds model-picker runtime choices with a hard-coded Pi default while the execution policy resolves official OpenAI `openai/gpt-*` agent runs to Codex. The related wildcard-runtime work tracks a separate resolver issue and does not supersede this readback mismatch.

## What This Changes

The issue reports that model picker runtime-choice metadata labels the default official OpenAI model route as Pi even though harness selection resolves it to Codex.

## Best Possible Solution

Derive model-picker runtime readback from `resolveAgentHarnessPolicy`, reuse centralized runtime labels, and keep the runtime metadata read-only unless a supported mutation flow exists.

## Reproduction Assessment

Yes, source-level: `buildModelsProviderData` with `openai/gpt-5.5` initializes OpenAI runtime choices with Pi, while `resolveAgentHarnessPolicy` for the same provider/model returns Codex through the official OpenAI default. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

Yes. The narrow maintainable fix is to make picker/runtime readback call the same effective harness policy used for execution, without changing execution policy or adding a session runtime override flow.
