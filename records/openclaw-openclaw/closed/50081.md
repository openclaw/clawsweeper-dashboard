## Summary

Keep open: current main still has a source-visible inline Control UI attachment path where a small HEIC/HEIF image can be forwarded to the current run as `image/heic`, and #50320 remains the open implementation candidate with closing syntax for this issue.

## What This Changes

Normalize current-run HEIC/HEIF webchat prompt images to JPEG before they are handed to the provider.

## Best Possible Solution

Land #50320 or a focused replacement that converts HEIC/HEIF chat attachments to JPEG before the active prompt call, rechecks converted image limits, preserves non-HEIC behavior, and adds gateway attachment regression coverage.

## Reproduction Assessment

Yes. The reporter gave concrete Control UI HEIC upload steps and hot-patch proof, and current source still preserves inline HEIC MIME through the current-run prompt-image path when no prompt refs force later loading.

## Solution Assessment

Yes. Normalizing before the current-run provider send is the narrow fix boundary because persistence/history normalization and offloaded media loaders occur too late for inline in-flight prompt images.
