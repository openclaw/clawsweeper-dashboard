## Summary

Keep this open: current main has partial support for fully node-bound Codex sessions, but the reported WebChat/Codex `host=auto` plus configured node mixed-session case still does not have a clearly supported node-exec surface or diagnostic, and related open PRs do not safely close this issue yet.

## What This Changes

The issue requests a usable WebChat/OpenAI Codex path for approved remote node shell execution when a connected node is configured, especially from `tools.exec.host=auto` sessions.

## Best Possible Solution

Define and land a maintainer-approved Codex mixed-session node-exec surface, either by safely restoring the existing OpenClaw exec tool where `host=node` can be requested or by adding a dedicated pinned node-run wrapper, with approval-policy preservation and WebChat/Codex regression proof.

## Reproduction Assessment

Yes for a source-level reproduction, but not a live Windows/WebChat run here. Current main filters OpenClaw exec/process from Codex dynamic tools, only restores them when effective exec host is node, and resolves `host=auto` without sandbox to gateway even when `tools.exec.node` is set.

## Solution Assessment

Unclear until maintainers choose the intended mixed-session shape. A narrow fix should not bypass Codex native local tools or OpenClaw node approvals; it should compose with the existing exec tool contract and the adjacent direct-shell/node-hosted-tool PRs.
