## Summary

Current main and v2026.5.3 already route operational voicecall CLI commands through the Gateway before falling back to a standalone runtime, with regression coverage asserting that no new VoiceCall runtime is created when Gateway RPC succeeds.

## What This Changes

The issue requests gateway-aware voicecall CLI subcommands so they do not bind a second webhook server when the Gateway is already running.

## Best Possible Solution

Keep the gateway-first operational CLI path and direct affected 2026.4.x users to upgrade to v2026.5.3 or newer.

## Reproduction Assessment

No current-main failing reproduction remains. Source inspection and regression tests show the reported commands delegate to Gateway RPC and skip createVoiceCallRuntime when a Gateway responds.

## Solution Assessment

Yes. The current implementation matches the requested maintainable shape: use the Gateway-owned voice-call runtime when available, with standalone runtime reserved for gateway-unavailable fallback.
