## Summary

Close as implemented: current main and v2026.5.7 now support configurable memory embedding dimensions through the existing `memorySearch.outputDimensionality` contract, pass it to OpenAI-compatible inline and batch embedding requests as `dimensions`, and include it in provider cache/reindex identity. The literal `memorySearch.dimensions` alias is still not accepted, but the shipped public config field solves the reported OpenAI-compatible memory-search problem without adding a duplicate key.

## What This Changes

Request support for configuring memory-search embedding dimensions and passing the value to OpenAI-compatible embedding requests.

## Best Possible Solution

Keep the shipped `memorySearch.outputDimensionality` contract as the single public dimension setting for compatible memory embedding providers, and leave local GGUF truncation to the separate open local-provider follow-up.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source and test inspection on current main gives high confidence that the OpenAI-compatible memory embedding dimension path is now implemented through `outputDimensionality`.

## Solution Assessment

Yes. Reusing the existing public `outputDimensionality` setting and mapping it to OpenAI-compatible `dimensions` is narrower than adding a duplicate config key, and the current adapter includes dimension changes in cache/reindex identity.
