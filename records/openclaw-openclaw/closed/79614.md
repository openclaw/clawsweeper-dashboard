## Summary

Keep open. Current main has partial stale-reply protections, including a foreground delivery fence and bounded `before_agent_finalize` retries, but it still lacks the requested host-owned latest-user snapshot, complaint signal, configurable re-anchor contract, and observability semantics.

## What This Changes

Request a first-class runtime contract that re-checks the latest inbound user message at finalization time and can force one bounded re-anchor pass before replying.

## Best Possible Solution

Define a host-owned freshness and re-anchor contract that composes with the existing foreground freshness fence, `before_agent_finalize`, retry prompts, and queue/orphan repair paths, then document the config and add focused Pi/Codex regression coverage.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection does show current main lacks the requested finalize-time latest-user snapshot, complaint flag, and built-in re-anchor contract.

## Solution Assessment

Unclear until maintainer product/API review. A bounded finalization recheck is plausible, but it should be designed around the existing hook security model, retry semantics, side-effectful tool turns, and foreground freshness fence rather than copied directly from a local heuristic plugin.
