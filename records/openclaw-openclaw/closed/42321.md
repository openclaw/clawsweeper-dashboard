## Summary

Current main already replaced the static messaging-tool check with plugin-aware detection, records successful plugin tool sends as messaging delivery, and suppresses the isolated-cron follow-up retry when `didSendViaMessagingTool` is true.

## Evidence

- **Plugin tools now count as messaging tools:** `isMessagingTool()` no longer relies only on the static `CORE_MESSAGING_TOOLS` set; it also treats channel/plugin tool names as messaging tools when the channel plugin exposes `actions`, and `isMessagingToolSendAction()` uses `actions.extractToolSend` to recognize a real send target.
  - file: [src/agents/pi-embedded-messaging.ts:9](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/agents/pi-embedded-messaging.ts#L9)

- **Successful plugin sends feed delivery tracking:** Tool execution handling records pending messaging sends for recognized plugin tools, then commits the sent text on successful tool completion; `didSendViaMessagingTool()` is derived from those committed sends.
  - file: [src/agents/pi-embedded-subscribe.handlers.tools.ts:673](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/agents/pi-embedded-subscribe.handlers.tools.ts#L673)

## Risks / Open Questions

- This fix path applies to plugin/channel tools that expose the supported `actions.extractToolSend` seam; arbitrary non-channel MCP tools without that contract are still outside the messaging-delivery detector.

## Close Comment

Closing from a Codex review because current `main` already covers this behavior. The old static `CORE_MESSAGING_TOOLS` limitation is gone: `src/agents/pi-embedded-messaging.ts:9` recognizes plugin/channel tools as messaging tools via `actions.extractToolSend`, `src/agents/pi-embedded-subscribe.handlers.tools.ts:673` records successful sends from those tools, and `src/agents/pi-embedded-runner/run/incomplete-turn.ts:190` suppresses the isolated-cron continuation/retry once `didSendViaMessagingTool` is true. This was also shipped in `v2026.4.22`; see `CHANGELOG.md:404` for the cron duplicate-delivery note. If this still reproduces on `v2026.4.22` or newer, please open a fresh issue with the exact plugin tool name, args shape, and a minimal repro showing that it does not implement the supported send-extraction seam.
