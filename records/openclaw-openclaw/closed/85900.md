## Label Justifications

- `P1`: A scheduled agent workflow can miss an entire run when a primary provider stalls even though healthy fallback models are configured.
- `impact:message-loss`: The reported timeout drops the cron job's expected response or delivery before any fallback model can produce output.
- `impact:auth-provider`: The affected surface is provider/model fallback routing and timeout classification for model calls.

## Summary

Keep open: current main and v2026.6.9 still use the explicit cron payload timeout as both the cron wall-clock watchdog and the cron-triggered model idle ceiling, so a no-first-chunk stream stall can still consume the budget that should leave room for fallback.

## What This Changes

The issue asks OpenClaw to treat cron provider streams that open but never yield a first chunk as failover-worthy failures so configured model fallbacks get a useful attempt before the cron watchdog kills the run.

## Best Possible Solution

Define a provider-attempt stream-stall budget that fails the current candidate and advances model fallback before the cron wall-clock deadline, while preserving explicit slow-provider timeout behavior.

## Reproduction Assessment

Yes at source level, but not with a live stalled-provider replay. A cron job with payload.model, configured fallbacks, and an opened-but-silent stream uses the same explicit timeout for the stream idle ceiling and the outer cron watchdog.

## Solution Assessment

Unclear until maintainers choose the timeout hierarchy. Treating no-first-chunk stalls as failover-worthy matches the documented fallback contract, but the implementation should avoid shortening all cron or provider timeouts unconditionally.
