## Summary

## What This Changes

The PR adds a Slack-only tool status label map and writes tool-specific Slack assistant thread status from the dispatch `onToolStart` handler.

## Best Possible Solution

Derive Slack assistant status from the shared progress display model, update after typing status is active, dedupe unchanged writes, cover start/update/clear behavior in tests, and require redacted live Slack proof before merge.

## Solution Assessment

No. The `onToolStart` seam is appropriate, but this patch can miss the first tool status, duplicates the shared tool display model, and lacks live Slack proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Update status after typing is active:** `extensions/slack/src/monitor/message-handler/dispatch.ts:1915`
  - body: This guard only writes the tool label when `didSetStatus` is already true, but the agent runner invokes `onToolStart` before awaiting the typing-start signal and has a test that requires progress callbacks to proceed before slow typing resolves. For a first or only tool event that starts Slack typing, this can skip the tool-specific status and leave the user seeing the generic typing text.
  - confidence: 0.9
- **[P1] Derive labels from shared progress display:** `extensions/slack/src/monitor/message-handler/dispatch.ts:129-150`
  - body: This adds a Slack-only tool label table even though the same handler already builds shared progress lines through `buildChannelProgressDraftLineForEntry`, which resolves canonical labels and fallbacks via `resolveToolDisplay`. Duplicating the map means supported or plugin-provided tools can show progress drafts but never update the assistant status consistently.
  - confidence: 0.9
- **[P2] Deduplicate unchanged Slack status writes:** `extensions/slack/src/monitor/message-handler/dispatch.ts:1915-1924`
  - body: `onToolStart` is called for both `start` and `update` phases, so this block can repeatedly call `assistant.threads.setStatus` with the same label during long-running tools. Track the last status or update only when the rendered status changes to avoid unnecessary Slack API churn.
  - confidence: 0.84
