## Summary

Keep open. Current `main` has improved explicit model persistence, but the issue discussion’s later failure mode is still a source-level gap: native `sessions_spawn` can accept a resolved model and report `modelApplied` after session-store persistence without validating that the selected provider has a runnable CLI backend before the child run starts.

## What This Changes

The issue asks for `sessions_spawn(model=...)` to make spawned subagents run on the requested model instead of silently falling back to the orchestrator/default model.

## Best Possible Solution

Add a narrow spawn-time validation or equivalent atomic run-start contract so an explicit resolved model either has a runnable backend before `accepted`/`modelApplied` is returned or fails with a clear configuration error.

## Reproduction Assessment

Yes, at source level. Configure an explicit `provider/model` whose provider has no configured or registered CLI backend; current `main` can persist the requested model and mark it applied before any backend-availability preflight in the spawn path.

## Solution Assessment

Yes. The narrow maintainable fix is to validate the resolved provider with the same CLI backend resolver before accepting the spawn, or pass the resolved model through an equivalent run-start contract that cannot silently fall back.
