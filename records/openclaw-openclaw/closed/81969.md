## Summary

Close: current main already prevents native OpenAI strict tool-schema mode from being forced onto non-native OpenAI-compatible completions providers, so the reported Xiaomi/custom-provider request shape is no longer the current-main behavior.

## What This Changes

The issue asks OpenClaw to stop sending `strict: true` and strict OpenAI tool schemas to Xiaomi MiMo and other custom OpenAI-compatible providers that reject them.

## Best Possible Solution

Keep the route-based strict-tool gating on main and ship it in the next release, with custom providers using the existing non-native compat path rather than a Xiaomi-specific core special case.

## Reproduction Assessment

No high-confidence current-main reproduction exists. Source inspection shows current main classifies non-native custom base URLs as proxy/custom routes and omits `function.strict` for those tool payloads.

## Solution Assessment

Yes. The current implementation is the narrowest maintainable fix because native OpenAI routes keep strict tools while generic OpenAI-compatible providers do not receive `strict: true`.
