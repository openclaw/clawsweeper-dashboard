## Summary

Current `main` and the latest shipped release already validate `send`/`chat.send` param types into structured `INVALID_REQUEST` responses instead of letting bad payloads crash the gateway, and the issue thread already contains a same-version reproduction that got the expected validation error with no restart.

## Evidence

- **`send` rejects invalid params before dispatch:** The `send` handler checks `validateSendParams(p)` and immediately responds with `ErrorCodes.INVALID_REQUEST` plus formatted validation errors when params fail schema validation.
  - file: [src/gateway/server-methods/send.ts:376](https://github.com/openclaw/openclaw/blob/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2/src/gateway/server-methods/send.ts#L376)
  - command: `nl -ba src/gateway/server-methods/send.ts | sed -n '340,440p'`
  - sha: [bc21f500d4c1](https://github.com/openclaw/openclaw/commit/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2)
- **`send` schema requires strings and forbids extra fields:** `SendParamsSchema` requires `to` and `idempotencyKey` as strings and sets `additionalProperties: false`, so the reported payload shape (`to:{}`, numeric `idempotencyKey`, extra `text`) is expected to fail validation with structured errors.
  - file: [src/gateway/protocol/schema/agent.ts:87](https://github.com/openclaw/openclaw/blob/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2/src/gateway/protocol/schema/agent.ts#L87)

  - file: [src/gateway/protocol/index.ts:526](https://github.com/openclaw/openclaw/blob/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2/src/gateway/protocol/index.ts#L526)
