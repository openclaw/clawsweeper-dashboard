## Summary

Keep open: this draft, `maintainer`-labeled PR still removes public named Plugin SDK exports from package-exported subpaths, so external plugins with existing named imports would break. The earlier `provider-auth-login` subpath-removal problem appears addressed in the latest head, but SDK compatibility judgment and contributor real behavior proof are still required.

## What This Changes

The PR narrows and deprecates Plugin SDK public surfaces, moves bundled plugin imports toward focused or provider-owned seams, and updates SDK docs, metadata, scripts, tests, and changelog entries.

## Best Possible Solution

Keep bundled plugins off broad or provider-owned SDK seams, but preserve deprecated public named re-export shims until an explicit major-version SDK removal decision is made.

## Reproduction Assessment

Yes for the review findings: source comparison shows current main exports and documents the affected public SDK names while the PR head removes them from package-exported subpaths. An external plugin using those named imports would fail at module binding after upgrade.

## Solution Assessment

No. Deprecating and moving bundled plugin imports is reasonable, but deleting public SDK named exports is not the narrowest compatible path; deprecated shims or an explicit major-version process are safer.
