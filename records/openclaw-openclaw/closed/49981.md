## Summary

Keep this PR open. Current main still has the same underlying remote embedding batch HTTP timeout gap: the configured batch timeout reaches the provider batch runtime and the guarded fetch can enforce `timeoutMs`, but the active memory-host SDK HTTP helpers and OpenAI/Voyage/Gemini batch upload/create/status/error-read/download paths do not accept or forward `timeoutMs`. The PR is stale against the newer provider-plugin layout and has review concerns, but the intended fix is still actionable.

## Best Possible Solution

Keep this PR open as useful but stale implementation work. The best maintainer path is a fresh port that adds optional `timeoutMs` or a shared remaining-deadline budget to `packages/memory-host-sdk/src/host/{remote-http,post-json,batch-http,batch-upload}.ts`, threads it through `extensions/{openai,voyage,google}/embedding-batch.ts` upload/create/status/error-read/download calls, avoids resubmitting already accepted provider batches on abort, drops unrelated formatting/test-support churn, and adds focused regression tests for SDK forwarding plus provider batch wait/download paths.

## Risks / Open Questions

- Merging the branch as-is is risky because it targets older `src/memory/*` files while the active implementation now lives under `packages/memory-host-sdk/src/host/*` and `extensions/{openai,voyage,google}/embedding-batch.ts`.
- A naive `AbortError` retry that reruns the whole batch operation can resubmit provider jobs after upload/create already succeeded, causing duplicate remote work, cost, or orphaned files/jobs.
- The correct fix should carry one remaining deadline through upload, create, polling, terminal error reads, result downloads, and retry attempts; passing the original timeout to each phase can still exceed the documented batch timeout.
