## Summary

Close as ClawHub/plugin-surface work. Current main already bridges internal hooks for the message receive/send portions, while `before_dispatch` and `agent_end` are deliberately exposed as typed Plugin SDK hooks; VISION.md and the plugin hook docs route optional runtime integrations to plugins instead of mirroring every plugin lifecycle event into the managed HOOK.md system.

## What This Changes

Request to make managed internal HOOK.md hooks receive plugin lifecycle/message events such as `before_dispatch`, `agent_end`, `message_sent`, and `message_received`.

## Best Possible Solution

Keep the internal HOOK.md event surface narrow and direct lifecycle/message-flow integrations that need `before_dispatch` or `agent_end` to a code plugin published through ClawHub or loaded locally with `api.on(...)`. Reopen core work only for a concrete missing Plugin SDK capability that blocks an external plugin implementation.

## Reproduction Assessment

Not applicable as a feature/scope request. Static inspection of current main gives a high-confidence behavior check: internal `message:received` and `message:sent` fire in the relevant paths, while `before_dispatch` and `agent_end` remain typed plugin hooks only.

## Solution Assessment

Yes for conservative cleanup: routing this to the Plugin SDK/ClawHub path is the narrowest maintainable solution because the requested lifecycle surfaces already exist for plugins. Mirroring those decision and conversation hooks into managed internal hooks would expand core policy without a demonstrated missing plugin API.
