## Summary

Keep open. Current main still routes proactive WhatsApp phone-number sends through plain PN JID normalization, and the exact PN-to-LID fix is an open, unmerged closing PR rather than shipped behavior.

## What This Changes

The issue requests that proactive WhatsApp sends resolve phone-number recipients through Baileys LID mapping before falling back to PN JIDs.

## Best Possible Solution

Review and land #74925, or an equivalent WhatsApp-plugin-only fix, so `createWebSendApi` resolves phone-number targets through the active account authDir/Baileys forward LID mapping while preserving qualified JIDs and the PN fallback.

## Reproduction Assessment

Yes. The issue gives concrete live WhatsApp steps, and current main has a source-level reproduction: a phone-number target still becomes `{digits}@s.whatsapp.net` even when Baileys would have a forward `lid-mapping-{digits}.json` entry.

## Solution Assessment

Yes. Resolving at the active web-send boundary with the listener account authDir is the narrowest maintainable fix because text/media sends, polls, and composing presence all flow through `createWebSendApi`.
