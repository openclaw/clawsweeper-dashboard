## Summary

Keep open: current main still lacks a per-session minimum-interval or rolling-hour compaction throttle, and the two production reports make the runaway-compaction failure mode credible. The proposed public config keys and default cancellation/monitoring behavior need runtime owner approval before a fix PR is safe.

## What This Changes

Add `agents.defaults.compaction.minIntervalSeconds` and `maxPerHour` guardrails to bound repeated compactions for a single session.

## Best Possible Solution

Design and implement an owner-approved compaction-rate guard in the runtime/config contract, with schema/help/docs updates, structured warning or lifecycle signaling, and focused coverage for both minimum interval and hourly cap decisions.

## Reproduction Assessment

Yes, at source level: current main has no per-session minimum interval or rolling-hour cap in the safeguard, runtime state, config schema, or docs. I did not live-reproduce the full production storm.

## Solution Assessment

Unclear as a final design: `minIntervalSeconds` plus `maxPerHour` is a plausible narrow guardrail, but the defaults, cancellation behavior, and monitoring event contract need runtime owner approval before implementation.
