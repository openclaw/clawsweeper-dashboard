## Summary

Keep open. Current main and the latest release still gate post-run deletion on `schedule.kind === "at"`, and the only direct fix path in the provided context is an open, unmerged closing-syntax PR.

## What This Changes

The issue reports that `deleteAfterRun: true` is ignored for successful `every` and `cron` schedules, causing those jobs to keep running instead of deleting after the first run.

## Best Possible Solution

Review and land the existing narrow fix PR, or close that PR only if maintainers choose a different fix for the same successful-recurring-job deletion contract.

## Reproduction Assessment

Yes. Source inspection of current main shows `applyJobResult` still gates deletion on `job.schedule.kind === "at"`, while successful `every` and `cron` jobs continue into next-run scheduling; no tests were run because this review is read-only.

## Solution Assessment

Yes. The narrow maintainable fix is to remove the `at`-only guard for successful runs while keeping failure deletion semantics separate from the newer related product question.
