## Summary

Keep open. Current `main` preserves `enabled: false` for existing named accounts in the shared account-patching helper, and this PR would change that shared behavior in ways that can silently reactivate disabled accounts outside the setup-only path.

## Evidence

- **Shared helper preserves existing disabled named accounts on main:** `src/channels/plugins/setup-helpers.ts:363` keeps an existing boolean `enabled` value for named accounts instead of forcing `true`, even when `ensureAccountEnabled` resolves truthy. That is the current shipped behavior on `main`. 
  - file: [src/channels/plugins/setup-helpers.ts:363](https://github.com/openclaw/openclaw/blob/459d277076488d5833a181cca7d8bf5c0f9696dc/src/channels/plugins/setup-helpers.ts#L363)
  - command: `nl -ba src/channels/plugins/setup-helpers.ts | sed -n '340,390p'`
  - sha: [459d27707648](https://github.com/openclaw/openclaw/commit/459d277076488d5833a181cca7d8bf5c0f9696dc)
- **Current helper test locks in the preserve-disabled invariant:** `src/channels/plugins/setup-helpers.test.ts:114` explicitly asserts that `applySetupAccountConfigPatch` leaves `accounts.work.enabled` as `false` while still enabling the channel root and applying the patch.
  - file: [src/channels/plugins/setup-helpers.test.ts:114](https://github.com/openclaw/openclaw/blob/459d277076488d5833a181cca7d8bf5c0f9696dc/src/channels/plugins/setup-helpers.test.ts#L114)
  - command: `nl -ba src/channels/plugins/setup-helpers.test.ts | sed -n '114,140p'`
  - sha: [459d27707648](https://github.com/openclaw/openclaw/commit/459d277076488d5833a181cca7d8bf5c0f9696dc)
- **Current wizard-path test also expects disabled accounts to stay disabled:** `src/channels/plugins/setup-wizard-helpers.test.ts:992` asserts `patchChannelConfigForAccount(...)` preserves `accounts.work.enabled = false`. This is the exact expectation the PR changes, so the PR conflicts with current in-tree behavior rather than matching it.
  - file: [src/channels/plugins/setup-wizard-helpers.test.ts:992](https://github.com/openclaw/openclaw/blob/459d277076488d5833a181cca7d8bf5c0f9696dc/src/channels/plugins/setup-wizard-helpers.test.ts#L992)

- **Shared helper is used by non-setup account edits:** `extensions/nextcloud-talk/src/setup-core.ts:48` routes generic account config updates through `patchScopedAccountConfig`, and `extensions/nextcloud-talk/src/setup-core.ts:185` uses that path for DM policy updates. `extensions/zalouser/src/setup-surface.ts:45` does the same, and `extensions/zalouser/src/setup-surface.ts:60` / `:81` use it for DM/group policy changes. A helper-wide force-enable would therefore reactivate disabled accounts during ordinary settings edits, not just initial setup.
  - file: [extensions/nextcloud-talk/src/setup-core.ts:48](https://github.com/openclaw/openclaw/blob/459d277076488d5833a181cca7d8bf5c0f9696dc/extensions/nextcloud-talk/src/setup-core.ts#L48)

  - sha: [6f79793c2a49](https://github.com/openclaw/openclaw/commit/6f79793c2a49350bc904f154bfa3bfc75461a111)

## Risks / Open Questions

- Merging this PR as-is would change a shared helper contract and can silently re-enable disabled named accounts during DM policy, allowlist, or group-policy edits in existing plugins.
- There may still be a real setup-only bug to solve, but the evidence supports a narrower fix in the wizard/setup call path rather than changing `patchScopedAccountConfig` semantics globally.
