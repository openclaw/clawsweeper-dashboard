## Summary

Keep this PR open. Current main still has no stale or idle-gap cutoff for tool-call history, and production loop detection still reads retained history before recording the next call. The linked bug remains open, this PR is a plausible focused fix candidate, and the overlapping open PR #40798 is not clearly canonical or merged.

## Best Possible Solution

Keep this PR open for maintainer review and consolidate it with #40144 and #40798. The best fix should prune or scope stale tool history before detection in the real detect-then-record path, use cutoff semantics that preserve slow-tool loop protection, reset or scope warning dedupe consistently, and include regression coverage plus aligned config/schema/docs updates.

## Risks / Open Questions

- The PR's stale cutoff is based on ToolCallRecord.timestamp; if that timestamp represents call start time and outcome recording does not refresh it, slow back-to-back tool calls could be treated as stale and weaken loop protection.
- Clearing toolCallHistory without resetting or scoping toolLoopWarningBuckets can suppress later warning-level loops that fall into an already-emitted warning bucket.
- If staleThresholdMs becomes public config, schema labels/help, generated config schema, and user docs should be aligned before landing.
- There are multiple open fix attempts for #40144, so maintainers should consolidate the implementation path rather than close this PR as obsolete without choosing a canonical replacement.
