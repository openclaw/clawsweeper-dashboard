## Summary

Keep open: current main and release `v2026.5.20` still build the `clickCoords` request body from unguarded `Number()` results, and the linked fix PR at https://github.com/openclaw/openclaw/pull/84208 is now closed unmerged with no open replacement that closes this issue.

## What This Changes

The issue asks OpenClaw to reject non-finite `browser click-coords` coordinates at the CLI boundary instead of sending `NaN` values that serialize to JSON `null`.

## Best Possible Solution

Add a finite-number guard for `click-coords` before `runElementAction`, leave resize validation centralized in `runBrowserResizeWithOutput`, and add focused CLI coverage for the invalid-coordinate no-request path.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: `browser click-coords abc 200` flows through `Number("abc")` without a guard before the `/act` body is built, and JavaScript serialization turns that `NaN` into JSON `null`. I did not run the full CLI because this review is read-only.

## Solution Assessment

Yes. The narrow maintainable fix is a local finite-number guard for `click-coords` plus regression coverage, while keeping the existing resize validation in its shared helper.
