## Summary

Close: the ACPX #73071 investigation path is superseded by the already-applied #73548 fix on current main, while this PR branch still carries a gateway scope-upgrade approval regression and an unlogged health-check fallback that should not merge.

## What This Changes

The branch wraps the ACPX startup health callback to return healthy when `isHealthy()` throws and removes the gateway WS `scope-upgrade` non-silent pairing override.

## Best Possible Solution

Keep the focused #73548 ACPX runtime-mode fix on main, close this unsafe superseded branch, and handle any remaining ACPX startup-probe observability improvement as a separate narrow PR that preserves explicit scope-upgrade approval.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Restore explicit approval for scope upgrades:** `src/gateway/server/ws-connection/message-handler.ts:1002-1007`
  - body: Removing the `scope-upgrade` override makes `silent` follow `allowSilentLocalPairing`, and that helper returns true for local scope-upgrade requests. A locally connected client asking for broader scopes could therefore be auto-approved, contradicting current main and the pairing docs.
  - confidence: 0.93
- **[P2] Log health-check exceptions before treating backend as healthy:** `extensions/acpx/src/service.ts:241-243`
  - body: The new catch converts any `isHealthy()` exception into `true` without logging. The registry currently treats thrown health checks as unavailable, so this masks backend probe failures and can send ACP work to a broken runtime with no diagnostic trail.
  - confidence: 0.82

## Security Review

Status: needs_attention
