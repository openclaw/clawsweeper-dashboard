## Summary

Current main already implements the group-session exec/ACP wake path described here: background exec completions enqueue a session-scoped `exec-event` wake, targeted heartbeat wakes preserve `sessionKey`, and regression coverage verifies delivery back to the original Telegram group/topic route. The remaining `heartbeat.target: "none"` and `activeHours` behavior is documented configuration, not the old group-routing bug.

## Evidence

- **`notifyOnExit` uses canonical session-scoped `exec-event` wakes:** Background exec completion enqueues the system event on the originating `sessionKey` and immediately requests a heartbeat with `scopedHeartbeatWakeOptions(sessionKey, { reason: "exec-event", coalesceMs: 0 })`.

- **Targeted heartbeat wakes keep `sessionKey` and `target:last` intact:** Regression coverage asserts that targeted wakes preserve `agentId`, `sessionKey`, and `heartbeat.target: "last"` across dispatch and retry, including group/channel-shaped session keys.
  - file: [src/infra/heartbeat-wake.test.ts:262](https://github.com/openclaw/openclaw/blob/0e7250f37bf347c72790abf2a6692d5ec84e5ea1/src/infra/heartbeat-wake.test.ts#L262)

  - file: [src/infra/outbound/targets.test.ts:606](https://github.com/openclaw/openclaw/blob/0e7250f37bf347c72790abf2a6692d5ec84e5ea1/src/infra/outbound/targets.test.ts#L606)
  - sha: [0e7250f37bf3](https://github.com/openclaw/openclaw/commit/0e7250f37bf347c72790abf2a6692d5ec84e5ea1)

## Risks / Open Questions

- `docs/channels/groups.md:75` still says heartbeats are skipped for group sessions, so docs may lag the implemented behavior.
- Visible heartbeat replies still require heartbeat delivery to be configured (`docs/gateway/heartbeat.md:25` documents default `target: "none"`); quiet-hours via `activeHours` also still suppress wakes by configuration.
