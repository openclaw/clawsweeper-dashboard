## Summary

Current `main` does not gate `/hooks/agent` execution on `deliver`. The hook path always dispatches an isolated cron agent turn; `deliver: false` is translated to `delivery.mode: "none"`, and current tests cover that `mode: "none"` jobs still execute while only suppressing outbound delivery/summary posting. The report's "empty session" observation is consistent with inspecting the hook response UUID instead of the actual hook session key.

## Evidence

- **Hook session key is separate from the response UUID:** Hook requests resolve a real session key from the explicit `sessionKey`, `hooks.defaultSessionKey`, or a generated `hook:<uuid>` value before dispatch. That means the persisted session identity is not derived from the HTTP response `runId`.
  - file: [src/gateway/hooks.ts:325](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/gateway/hooks.ts#L325)
  - command: `nl -ba src/gateway/hooks.ts | sed -n '320,350p'`
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **HTTP handler dispatches with sessionKey, then returns a hook-layer runId:** `/hooks/agent` resolves `sessionKey`, normalizes it, calls `dispatchAgentHook(...)`, remembers that returned UUID for idempotency replay, and returns it in `{ ok: true, runId }`. The returned UUID is not the session key used for the isolated run.
  - file: [src/gateway/server-http.ts:673](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/gateway/server-http.ts#L673)
  - command: `nl -ba src/gateway/server-http.ts | sed -n '663,728p'`
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **`deliver: false` only changes delivery mode:** The hook dispatcher maps `deliver: false` to `delivery: { mode: "none" }` and still unconditionally calls `runCronIsolatedAgentTurn(...)` for the isolated agent turn.
  - file: [src/gateway/server/hooks.ts:49](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/gateway/server/hooks.ts#L49)
  - command: `nl -ba src/gateway/server/hooks.ts | sed -n '43,90p'`
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **Isolated-agent tests prove `mode: none` still runs the agent:** Current tests explicitly assert that `delivery.mode = "none"` keeps the message tool enabled and still invokes the embedded agent runtime, so execution is not suppressed by delivery being off.
  - file: [src/cron/isolated-agent/run.message-tool-policy.test.ts:266](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/cron/isolated-agent/run.message-tool-policy.test.ts#L266)
  - command: `nl -ba src/cron/isolated-agent/run.message-tool-policy.test.ts | sed -n '266,340p'`
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **Cron service tests treat `mode: none` as run-without-posting:** The cron service test expects an isolated job with `delivery.mode = "none"` to return `{ ok: true, ran: true }` while only suppressing the summary post, matching the intended hook behavior.
  - file: [src/cron/service.delivery-plan.test.ts:73](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/cron/service.delivery-plan.test.ts#L73)

## Risks / Open Questions

- This was a read-only code review; I did not run a live hook request in this environment.
- The `/hooks/agent` response field named `runId` is easy to misread as a waitable agent/session identifier even though the hook path dispatches using a separate session key. That API/docs ambiguity may still deserve a separate follow-up.
