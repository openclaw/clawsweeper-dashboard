## Summary

Keep open: this PR targets a real Telegram reply-loss path, but its current implementation scopes the polling session's shared abort signal and would stop shutdown from aborting overlapping `sendMessage` requests; it also lacks the required changelog entry and currently reports as dirty against `main`.

## What This Changes

The PR threads an optional fetch-abort method allow-list into Telegram bot creation, scopes polling-session aborts to `getUpdates`, and adds fetch-abort regression coverage.

## Best Possible Solution

Keep the scoped-abort fix, but split restart-only and shutdown abort lifetimes so watchdog recovery aborts only `getUpdates` while gateway/session shutdown still aborts all in-flight Telegram API calls, then update tests and changelog before merge.

## Reproduction Assessment

Yes, source-level reproduction is high confidence. Current `main` uses one polling-session abort controller for restart and shutdown, while the PR gates that same signal to `getUpdates`, so an overlapping `sendMessage` no longer observes shutdown abort.

## Solution Assessment

No. The allow-list is a good narrow mechanism for restart recovery, but it is attached at the wrong lifetime boundary; separate restart-only and shutdown abort signals would preserve both behaviors.
