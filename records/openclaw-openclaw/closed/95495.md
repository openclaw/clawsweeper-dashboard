## Summary

Keep open: current main and the latest v2026.6.10 release still use the per-agent memory SQLite database while only migrating legacy tables already inside the opened database. The strongest fix path is the open, proof-positive Memory Core doctor migration PR, so this issue should remain open until that PR lands or maintainers choose a replacement migration/warning policy.

## What This Changes

The issue reports that upgrading to 2026.6.9 moved built-in memory indexing from the legacy global SQLite store to the per-agent SQLite database without migration or warning, leaving vector search empty until a full reindex.

## Best Possible Solution

Land or replace the open Memory Core doctor-migration PR so legacy sidecar memory indexes are imported or explicitly preserved/warned during upgrade while runtime stays on the canonical per-agent DB.

## Reproduction Assessment

Yes, source-reproducible: current main opens built-in memory through the per-agent SQLite path, while the current schema migration only handles legacy rows already inside that DB and the issue/comments provide field logs of an intact legacy sidecar plus empty active DB. I did not run a packaged 2026.6.8 to 2026.6.9 upgrade reproduction in this read-only review.

## Solution Assessment

No, current main is not the best solution yet because it documents and uses the new per-agent store without a sidecar import/warning for existing shipped indexes. The best available path is the open doctor-owned Memory Core migration PR, pending maintainer landing or replacement.
