## Label Justifications

- `P0`: The report describes a shared-secret auth boundary bypass from hook credential possession to full Gateway operator access.
- `impact:security`: The affected behavior crosses the documented Gateway authentication and trusted-operator boundary.
- `impact:auth-provider`: The issue is specifically about Gateway auth secret reuse and password-mode shared-secret authentication.

## Summary

Keep open: current main and the latest release still have the password-mode hook-token reuse path, and the issue is already paired with an open closing PR that should be reviewed before closure.

## What This Changes

The issue reports that reusing the same secret for `hooks.token` and `gateway.auth.password` lets a hook ingress bearer authenticate to full Gateway HTTP operator surfaces.

## Best Possible Solution

Land the linked PR or an equivalent focused fix that rejects hook-token reuse with both Gateway token and password auth, updates audit coverage, and keeps regression tests for both modes.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure hooks with a token equal to `gateway.auth.password`, then reuse that bearer against a password-mode Gateway HTTP surface; I did not run a live gateway because this review is read-only.

## Solution Assessment

Yes. The narrow maintainable fix is to keep hook ingress credentials distinct from all Gateway shared-secret operator credentials in startup/reload validation and security audit, rather than changing `/tools/invoke` trusted-operator semantics.
