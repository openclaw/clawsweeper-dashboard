## Summary

Keep open: current main still treats user-pinned session model selections as strict exact selections, so the reported one-candidate fallback chain remains a product-policy question rather than an implemented fix or cleanup close. The directly related expansion PR was closed unmerged, and the merged silent-fallback PR is adjacent but does not change strict candidate construction.

## What This Changes

The issue requests fallback-chain expansion or a warning when a user-pinned session model is also the only configured fallback, instead of collapsing to a single strict candidate.

## Best Possible Solution

Maintainers should decide whether strict user pins remain the contract; if that policy changes, update the session-pin boundary, candidate builder, docs, changelog, and regression tests together.

## Reproduction Assessment

Yes, source-reproducible: a user session model override resolves to an empty fallback override, and the fallback builder treats that as exact, leaving one candidate and fallbackConfigured:false. I did not run a live timeout transcript.

## Solution Assessment

No, not as an automatic bug fix: appending the configured primary for empty overrides would broaden the documented strict-selection contract. A session-pin-specific policy decision should come first, followed by a narrow implementation if maintainers choose that behavior.
