Keep open: current main still has the source-level approval/execution executable identity mismatch, and a linked PR with closing syntax remains open and conflicting rather than merged.

## What This Changes

The issue reports that exec approval trust checks use the PATH/symlink executable path while approval-backed execution pins the executable realpath.

## Best Possible Solution

Land or replace the linked PR with a maintainer-approved executable identity fix that aligns matching, trust checks, persistence/audit metadata, diagnostics, and execution with focused symlink regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction: an ExecutableResolution with a symlink resolvedPath and distinct resolvedRealPath is matched against the symlink path while approved execution pins the realpath.

## Solution Assessment

Unclear as a complete fix. Realpath canonicalization is the right direction, but maintainers need to decide whether symlink-form approvals remain valid and ensure the adjacent trust and persistence surfaces follow the chosen identity.
