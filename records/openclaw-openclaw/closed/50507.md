Overall confidence: 0.94

Full review comments:

- **[P1] Expose location through WhatsApp discovery:** `src/agents/tools/message-tool.ts:123-130`
  - body: Configured message tools derive valid actions from channel discovery. This PR adds location fields to the schema, but WhatsApp discovery still advertises only `react` and `poll`, so normal configured tools reject `action: "location"` before the new fields can be used.
  - confidence: 0.91
- **[P1] Wire location into the shipped WhatsApp outbound surface:** `extensions/whatsapp/src/outbound-adapter.ts:91-109`
  - body: The PR adds `sendLocation` to `whatsappOutbound`, but the active WhatsApp plugin surface is built through the channel outbound/base path, which still lacks `sendLocation`. Real sends can still fail as an unsupported location channel before reaching Baileys.
  - confidence: 0.9
- **[P1] Register a gateway location route before calling it:** `src/infra/outbound/message.ts:424-426`
  - body: Every non-dry-run location send calls gateway method `location`, but the gateway send handlers register `message.action`, `send`, and `poll` only. WhatsApp gateway delivery will fail with an unknown method before delivery.
  - confidence: 0.93
- **[P2] Stop logging exact location coordinates:** `extensions/whatsapp/src/send.ts:184-186`
  - body: Latitude and longitude are precise location data. Logging them in structured outbound logs can expose sensitive places; keep the redacted recipient/JID and omit or coarsen coordinates.
  - confidence: 0.92
- **[P3] Forward accuracy through the core location path:** `src/infra/outbound/outbound-send-service.ts:204-244`
  - body: The new public location types include `accuracyInMeters`, but `executeLocationAction()` neither accepts nor forwards it to `sendLocation`, so that metadata is silently dropped on the core path.
  - confidence: 0.82
