## Summary

Keep open: current main still lacks heartbeat object-form fallback support and this PR is the canonical implementation thread, but the branch should not merge yet because it drops explicit empty fallback overrides, loses object-form fallbacks on a direct heartbeat reply path, is stale against current main, and lacks after-fix real behavior proof.

## What This Changes

The PR widens `agents.*.heartbeat.model` to accept `{ primary, fallbacks }`, threads heartbeat fallback lists into model fallback execution, and updates pricing metadata, docs, generated config help, tests, and changelog.

## Best Possible Solution

Land a rebased implementation that treats heartbeat fallback lists as an explicit per-run override, preserves string compatibility, updates the current generated config surfaces and docs, and includes redacted real heartbeat fallback proof.

## Reproduction Assessment

Yes, at source level: current main's schema and public type are string-only, and the heartbeat runner forwards only a single model override. I did not reproduce a live provider-429 heartbeat failover path.

## Solution Assessment

No, not as currently written. The object-form direction matches the existing agent model config contract, but the patch needs empty-array preservation, direct-call fallback forwarding, a current-main rebase, and real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve explicit empty heartbeat fallbacks:** `src/infra/heartbeat-runner.ts:1038`
  - body: The PR documents `fallbacks: []` as disabling the agent-level fallback chain, and the downstream fallback contract treats `[]` differently from `undefined`. Because this line only passes `heartbeatModelFallbacks` when the list is non-empty, an explicit empty array is dropped and the run can still inherit configured fallbacks.
  - confidence: 0.95

  - body: This fallback path now resolves the object-form heartbeat primary when no runner override is supplied, but it never derives the matching fallback list. Direct heartbeat callers using object-form defaults would run only the primary and lose the configured failover chain.
  - confidence: 0.86
