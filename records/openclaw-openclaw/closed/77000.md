## Summary

Current main implements the reported Telegram restart-loss fix: Telegram update watermarks are persisted after completed dispatch with a safe completed watermark, and final automatic Telegram replies now use the durable message lifecycle/write-ahead outbound queue. The fix is included in the stable v2026.5.12 release, so this issue can close as implemented.

## What This Changes

Prevent Telegram forum-topic replies from being permanently lost when the gateway restarts between accepting an inbound update and confirming outbound final delivery.

## Best Possible Solution

Keep the shipped durable final-delivery and after-dispatch safe-watermark implementation; use the separate session-lock issue for unrelated lost-response cases.

## Reproduction Assessment

No, not against current main: the old source-level race is addressed by after-dispatch watermark persistence and durable final-send queueing. I did not run a live Telegram restart reproduction in this read-only pass.

## Solution Assessment

Yes. The current implementation is stronger than raw offset deferral because it combines a safe completed update watermark with durable outbound final delivery, reducing permanent loss without blindly advancing past pending or failed updates.
