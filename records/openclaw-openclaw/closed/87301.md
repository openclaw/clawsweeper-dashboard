This PR adds a logging hostname bootstrap imported before tslog and replaces the structured file-log hostname constant with a cached lazy getter.

## Best Possible Solution

Land a repaired version that makes both tslog `_meta.hostname` and OpenClaw's top-level `hostname` recover after an initial empty hostname lookup, with focused regression coverage and macOS log proof.

## Reproduction Assessment

Full review comments:

- **[P1] Handle the first empty hostname before tslog snapshots meta:** `src/logging/logger-hostname-init.ts:8-10`
  - body: The reported failure is that hostname lookup is empty during module load, but this initializer also does a single module-load `os.hostname()` call and leaves `HOSTNAME` unset when that call is empty. tslog 4.10.2 builds `_meta.hostname` from env once when the logger environment is created, so that first-empty sequence still leaves `_meta.hostname` as `unknown` for the process lifetime.
  - confidence: 0.92
- **[P2] Do not cache the fallback hostname:** `src/logging/logger.ts:108`
  - body: The lazy getter stores `unknown` when its first `os.hostname()` call is empty. If hostname lookup stabilizes after gateway startup, later log writes cannot recover; cache only a non-empty hostname or retry until a real value is available.
  - confidence: 0.82
