## Label Justifications

- `P2`: The reported behavior affects Telegram startup in proxy-heavy environments but is limited to one channel/plugin and already has a shipped configuration path.
- `impact:message-loss`: A false Telegram startup failure can leave the Telegram channel offline and prevent inbound/outbound message handling until configuration or release state is corrected.

## Summary

Close: current main and shipped releases already route Telegram startup getMe through the Telegram timeout resolver, with a 15s default and higher `channels.telegram.timeoutSeconds` support; the reported 10s startup behavior matches older code.

## What This Changes

The issue requests a configurable or longer Telegram startup getMe health-check timeout for proxy-heavy environments.

## Best Possible Solution

Keep the existing Telegram-owned `timeoutSeconds` contract and direct proxy-heavy operators to raise it instead of adding a second health-check-only timeout knob.

## Reproduction Assessment

Not applicable as a feature/config request. Source and tests show the reported 10s startup timeout no longer matches current main: startup getMe uses a 15s default and honors higher `timeoutSeconds`.

## Solution Assessment

Yes. Reusing the existing Telegram API `timeoutSeconds` knob is narrower and less confusing than adding a second health-check-only timeout setting, and current tests cover the startup getMe path.
