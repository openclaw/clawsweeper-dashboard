## Summary

This should close as superseded: the remaining Feishu message delete/recall and generic `unsend` work is already concentrated in the active Feishu message-management PR, and the issue's proposed `recall()` API does not match the Feishu/Lark SDK contract.

## What This Changes

The issue asks for Feishu/Lark message `unsend` to recall a message for all chat participants instead of using the wrong message deletion semantics.

## Best Possible Solution

Review and land or explicitly close https://github.com/openclaw/openclaw/pull/83517, keeping the Feishu/Lark dependency contract that recall is exposed as the SDK `delete` method.

## Reproduction Assessment

No for the exact old-package API-mismatch report: current main does not contain a Feishu `unsend` handler that calls delete. Source inspection does show the remaining capability gap, because Feishu does not advertise or dispatch `unsend` today.

## Solution Assessment

No, replacing `client.im.message.delete()` with `client.im.message.recall()` is not the best fix because the Feishu/Lark SDK and docs expose recall through the DELETE/delete method. The maintainable path is the already-open Feishu message-management PR or an equivalent narrow implementation with tests and an opt-in gate.
