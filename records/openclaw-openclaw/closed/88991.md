## Summary

This should close as a duplicate/superseded tracker: current main still has the reported daily-freshness WebChat rotation path, but the same user-visible WebChat reset/lost-history problem is already tracked in the older open WebChat session-reset reports, while the separate adaptive reset-mode policy work is on an open PR.

## What This Changes

The issue reports that WebChat can silently rotate a stale daily session on the first post-reconnect send, archiving the prior transcript and clearing visible history without warning.

## Best Possible Solution

## Solution Assessment

No as a standalone tracker: the remaining fix needs the canonical WebChat session-recovery discussion plus maintainer choice on whether to change daily reset semantics, add a proactive UI event, or both.
