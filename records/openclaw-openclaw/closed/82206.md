## Summary

Keep open: the bug-fix direction is valid, but the patch currently exposes a runtime-only `sourceReplyDeliveryMode` knob through the public gateway `agent` RPC, and the PR has the protected `maintainer` label.

## What This Changes

The PR propagates `sourceReplyDeliveryMode` through generated-media completion handoffs, embedded-run queue state, gateway agent RPC, generated Swift protocol models, tests, and the changelog.

## Best Possible Solution

Preserve the generated-media completion delivery mode across requester wake and fallback paths, but derive or accept `message_tool_only` only for trusted internal completion handoffs and add a regression test that untrusted gateway callers cannot force it.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main computes that some group/channel media completions need message-tool delivery, but the queued wake and gateway fallback handoff do not carry that mode.

## Solution Assessment

No, not as written. Carrying the mode is the right fix, but accepting it directly on the public gateway `agent` schema is broader than the internal completion handoff that needs it.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Restrict sourceReplyDeliveryMode to trusted handoffs:** `src/gateway/protocol/schema/agent.ts:182-184`
  - body: Adding this field to `AgentParamsSchema` makes it accepted on the public `agent` RPC, which is only `operator.write`. The handler then forwards it into `agentCommand`, and `message_tool_only` forces the embedded `message` tool/prompt path, so untrusted gateway callers can opt into a runtime-only delivery mode. Please derive this value for the media-completion handoff or reject/ignore it for callers that are not trusted internal handoffs.
  - confidence: 0.87
