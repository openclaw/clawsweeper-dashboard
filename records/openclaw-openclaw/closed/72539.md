## Summary

Current main and the v2026.4.27 release now implement the requested outbound pre-send plugin SDK surface as `message_sending`. It runs before channel delivery, exposes outbound routing/content/media metadata, and lets plugins rewrite content or cancel delivery, so the original missing-hook report is implemented well enough without adding a separate `MessagePreSent` name.

## What This Changes

The issue requests a pre-send Plugin SDK hook that lets plugins inspect, validate, modify, or reject outbound messages before gateway/channel dispatch.

## Best Possible Solution

Keep the shipped `message_sending` hook as the public pre-send Plugin SDK contract, and use the broader open runtime-policy issue for classifier/stateful fake-progress enforcement that goes beyond a generic SDK interception point.

## Reproduction Assessment

Yes. On current main or v2026.4.27, register a plugin `api.on("message_sending", ...)` handler that returns rewritten `content` or `{ cancel: true }`; the existing outbound, Telegram, auto-reply, and hook-runner tests cover the expected pre-send behavior.

## Solution Assessment

Yes. Reusing the documented `message_sending` hook is narrower and more maintainable than adding a parallel `MessagePreSent` event name, while still satisfying the report's structural pre-send filtering need.
