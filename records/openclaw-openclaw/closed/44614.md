## What This Changes

The PR adds a Windows-only detached `.cmd` updater helper, delegates global npm updates to it, exits the Gateway after responding to `update.run`, and adds helper tests.

## Best Possible Solution

Close this source PR and continue review on #75649 as the canonical replacement for #40540, preserving the original contributor credit while landing the safer staged-update design.

## Reproduction Assessment

Yes. The reproduction path is source-backed by #40540's Windows EBUSY logs plus current main's live Gateway/global npm package update path; I did not run a live Windows update in this read-only review.

## Solution Assessment

No. The detached-helper direction is plausible, but this PR is not the best implementation because it bypasses current staged-update safeguards, drops env handling, builds unsafe cmd commands, and reports success before the detached update is reconciled; #75649 is the safer replacement path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Default the detached exit delay before clamping:** `src/gateway/server-methods/update.ts:121`
  - body: `restartDelayMs` is optional, so `Math.max(restartDelayMs, 2000)` becomes `NaN` for the default request. Node treats that timer like an immediate timeout, which can exit the Gateway before the response flushes.
  - confidence: 0.95
- **[P1] Gate forced exit on restart supervision:** `src/gateway/server-methods/update.ts:119`
  - body: This branch reports a successful detached restart and then calls `process.exit(0)`, but the helper only relaunches through `schtasks`. On unsupervised Windows runs or missing task installs, `update.run` can return success and leave the Gateway down.
  - confidence: 0.88
- **[P1] Pass the prepared install env to the helper:** `src/infra/update-detached-win32.ts:147-151`
  - body: `spawnDetachedUpdate` receives the prepared `globalInstallEnv`, but the `spawn()` options omit `env`. The detached cmd then runs with the ambient environment and can lose Windows npm/Corepack/portable-Git settings needed for package updates.
  - confidence: 0.92
- **[P1] Escape install argv before embedding it in cmd:** `src/infra/update-detached-win32.ts:134-136`
  - body: The helper converts argv to a batch command by only quoting arguments that contain spaces. Cmd metacharacters such as `&`, `|`, `%`, `!`, and `^` can still be interpreted by the shell, unlike the prior argv-based execution.
  - confidence: 0.9
- **[P2] Preserve npm's omit-optional retry:** `src/infra/update-runner.ts:896-901`
  - body: The Windows branch delegates only the primary `globalInstallArgs(...)` command and returns before the existing `globalInstallFallbackArgs` retry can run. Machines where the first npm install fails but the `--omit=optional` retry succeeds will regress to failed updates.
  - confidence: 0.86
- **[P2] Do not report delegated installs as complete:** `src/infra/update-runner.ts:933-940`
  - body: This returns `status: "ok"` immediately after spawning the helper, before npm has installed anything. If the helper later records `install-failed` or `pid-wait-timeout`, the API response and restart sentinel still look like a successful update.
  - confidence: 0.89

## Security Review

Status: needs_attention

Summary: The PR introduces a Windows command-execution risk in the detached update helper.

Concerns:

- **[medium] Generated cmd script does not preserve argv semantics:** `src/infra/update-detached-win32.ts:134`
  - body: `installArgv` is converted to one cmd command with only space-based quoting, so shell metacharacters in package specs or paths can alter the command that runs in a supply-chain-sensitive update path.
  - confidence: 0.9
