## Label Justifications

- `P1`: A real reported KiloCode image workflow silently drops image blocks before the model request, breaking a user-visible agent path for affected users.
- `impact:message-loss`: The reported failure causes a submitted image to be stored as a media reference instead of being delivered to the active model.
- `impact:auth-provider`: The root path is provider catalog discovery and model capability metadata for a bundled provider.

## Summary

Keep open: current main and the latest release still treat only `runtime` manifest discovery as requiring full provider runtime loading, while the documented `refreshable` contract says manifest rows are seeds rather than authoritative catalog coverage.

## What This Changes

The issue reports that refreshable manifest provider catalogs can satisfy discovery with seed rows and skip provider runtime loading, causing KiloCode-discovered image-capable models to be treated as text-only.

## Best Possible Solution

Update shared provider discovery so refreshable manifest rows remain seed/supplement rows and do not satisfy full runtime discovery coverage; add regression coverage using a refreshable Kilo-like manifest catalog.

## Reproduction Assessment

Yes at source level: current main can classify a refreshable manifest seed as complete because only `runtime` discovery forces full plugin loading. I did not run a live KiloCode account path, so this is not marked as an end-to-end reproduced failure.

## Solution Assessment

Yes, the best fix is in shared provider discovery, not in KiloCode alone. Refreshable manifest catalogs should keep their fast seed rows but still trigger the provider runtime path during full discovery.
