## Summary

Keep open. #70831 is shipped on main and in v2026.4.24 for the guarded-dispatcher 60s timeout, but this issue was explicitly left open as a separate LLM-idle-timeout/local-Ollama path, and a fresh v2026.4.24 report says the timeout still occurs even with `agents.defaults.llm.idleTimeoutSeconds` set to 300. Current main still contains the same idle-timeout/failover surfaces and I found no newer Ollama timeout fix after v2026.4.24.

## Best Possible Solution

Keep #64710 open as the canonical tracker for the remaining local Ollama main-chat/agent idle-timeout failure. Maintainers should retest current main with native Ollama `/api/chat`, tools enabled, and the configured idle timeout, capture the OpenClaw request/stream behavior versus direct Ollama, then fix the implicated Ollama transport or embedded-run timeout budget path with a focused regression test or live smoke.
