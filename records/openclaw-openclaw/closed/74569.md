## Summary

Current main already addresses the reported Matrix restart replay path with two independent guards: stale Matrix sync cursors are only trusted after a clean shutdown persist, and inbound Matrix room events are durably deduped by room/event id across restarts. The latest release tag available in this checkout, v2026.4.27, contains the same Matrix sync/dedupe implementation and release-note coverage, so this can close as implemented.

## What This Changes

Prevent old Matrix room events from being processed as fresh inbound messages after a gateway restart or host reboot by preserving safe sync state and suppressing already-seen event ids.

## Best Possible Solution

Keep the current Matrix restart safety model: only trust restart backlog after a clean shutdown persist, keep durable event-id dedupe for already-handled Matrix room events, and ask for a new focused report if v2026.4.27 or newer still replays stale Matrix history with the sync and dedupe stores present.
