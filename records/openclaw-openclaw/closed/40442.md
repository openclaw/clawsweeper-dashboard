## Summary

Current main already guards bare `/new` and `/reset` turns from inheriting stale implicit reply targets, and Telegram native-command delivery falls back to the visible command message when no explicit reply target exists. The same implementation is present in release `v2026.4.22`.

## Evidence

- **Bare reset turns disable implicit reply-to-current threading:** `runPreparedReply` sets `replyThreadingOverride.implicitCurrentMessage = "deny"` for bare session resets, so `/new` and `/reset` turns no longer inherit the generic implicit current-message reply target.
  - file: [src/auto-reply/reply/get-reply-run.ts:771](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/auto-reply/reply/get-reply-run.ts#L771)
  - command: `git show 00bd2cf7:src/auto-reply/reply/get-reply-run.ts | sed -n '760,780p'`
  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)
- **Telegram native commands anchor fallback replies to the visible command message:** When a native Telegram command reply has no explicit `replyToId`, delivery rewrites it to `replyToId: String(msg.message_id)`, which ties the first reply to the visible `/new` or `/reset` message instead of an older hidden target.
  - file: [extensions/telegram/src/bot-native-commands.ts:954](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/extensions/telegram/src/bot-native-commands.ts#L954)
