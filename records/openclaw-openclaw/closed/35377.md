## Summary

Current `main` already implements inbound audio/voice-message transcription through the shared `tools.media.audio` pipeline, including provider/CLI STT options, transcript injection into reply context, and voice-note handling in channel flows such as Feishu and Telegram.

## Evidence

- **Core audio transcription docs:** `docs/nodes/audio.md:12` documents automatic inbound audio transcription, transcript injection, ordered provider/CLI fallback, and Whisper CLI/OpenAI examples; `docs/nodes/audio.md:163` documents preflight transcription for mention-gated voice notes.
  - file: [docs/nodes/audio.md:12](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/docs/nodes/audio.md#L12)
  - sha: [9153598e659d](https://github.com/openclaw/openclaw/commit/9153598e659daf5a52c479731f5b4ab8a5bf16b7)
- **Core implementation:** `src/media-understanding/audio-preflight.ts:10` implements transcription of the first audio attachment before mention checks, using the shared media-understanding runtime and returning the transcript for reply handling.
  - file: [src/media-understanding/audio-preflight.ts:10](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/src/media-understanding/audio-preflight.ts#L10)
  - sha: [9153598e659d](https://github.com/openclaw/openclaw/commit/9153598e659daf5a52c479731f5b4ab8a5bf16b7)
- **Feishu/Lark inbound audio path:** `extensions/feishu/src/bot-content.ts:307` maps Feishu `audio` messages to `<media:audio>`, and `extensions/feishu/src/bot-content.ts:329` downloads/saves Feishu media attachments into the generic inbound media pipeline, which is what the shared audio transcription code consumes.
  - file: [extensions/feishu/src/bot-content.ts:307](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/extensions/feishu/src/bot-content.ts#L307)
  - sha: [9153598e659d](https://github.com/openclaw/openclaw/commit/9153598e659daf5a52c479731f5b4ab8a5bf16b7)
- **Regression coverage for voice notes:** `extensions/telegram/src/bot-message-context.body.test.ts:149` asserts that DM voice notes are transcribed via preflight and become message body text instead of staying as raw audio placeholders.
  - file: [extensions/telegram/src/bot-message-context.body.test.ts:149](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/extensions/telegram/src/bot-message-context.body.test.ts#L149)
  - sha: [9153598e659d](https://github.com/openclaw/openclaw/commit/9153598e659daf5a52c479731f5b4ab8a5bf16b7)

  - file: [docs/providers/openai.md:26](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/docs/providers/openai.md#L26)

- This review was read-only and `pnpm` was unavailable in the checkout, so the conclusion is based on current code, docs, tests, and release-tag inspection rather than a live channel reproduction.
