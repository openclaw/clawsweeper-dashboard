## Summary

Keep open: this is useful media filename security hardening, but the branch is stale against current main's cross-platform basename helpers, the supplied proof only exercises the helper directly, and the remaining bidi-mark scope needs maintainer security/i18n judgment.

## What This Changes

The PR adds a shared outbound filename control-character stripper, wires it into remote Content-Disposition and local web-media filename extraction, and adds focused media tests.

## Best Possible Solution

Land a rebased media-boundary sanitizer that composes with basenameFromAnyPath/extnameFromAnyPath, locks the agreed bidi policy in tests, and includes real fetch/local-media or Telegram-label proof.

## Reproduction Assessment

Yes, from source inspection: current main returns remote and local media basenames without filtering these invisible/control characters. I did not run a live harness in this read-only review, so this is source-reproducible rather than executed proof.

## Solution Assessment

No, not as currently mergeable. Centralizing the strip at the media filename boundary is the right shape, but the branch must preserve current basename helpers, supply boundary-visible proof, and settle the bidi-mark policy.
