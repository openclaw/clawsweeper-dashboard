## Summary

Close this PR as superseded by the sandbox backend architecture already on current main. Main now has a registry-backed SandboxBackendHandle API, plugin SDK exports for registering backends, built-in Docker/SSH backends, and an OpenShell plugin backend. The remaining gVisor idea is not shipped, but this branch adds an obsolete parallel ISandboxProvider abstraction and has multiple unresolved functional/security review findings, so the best path is a fresh narrow gVisor backend against the current API.

## Best Possible Solution

Close this PR as superseded. Keep the shipped SandboxBackendHandle/registerSandboxBackend architecture, and invite any remaining gVisor work as a fresh focused backend implementation that reuses the current Docker container creation/hardening path where possible, includes tests for workspace mounts, scope naming, registry/list/recreate/prune, config hash/recreate behavior, browser support if claimed, and a dedicated sandbox security review.
