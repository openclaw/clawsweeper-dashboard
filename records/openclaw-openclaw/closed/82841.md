## Summary

Keep open: the earlier 2026.5.12 appcast mismatch was fixed, but this report is a different rollback direction involving a stale `cli v2026.5.12` client. Current source still treats protocol mismatch as a generic connect failure for clients, so a long-lived stale helper can keep reconnecting instead of stopping on a terminal version incompatibility.

## What This Changes

The issue reports rollback log spam from a stale local client that still identifies as cli v2026.5.12 while reconnecting to a 2026.5.7 gateway.

## Best Possible Solution

Keep strict gateway protocol compatibility, but make protocol incompatibility terminal and diagnosable for clients so stale helpers stop retrying and operators can identify the mismatched process/version.

## Reproduction Assessment

No for the full macOS rollback setup; source inspection gives a medium-confidence reproduction of the core retry path: a newer long-lived GatewayClient receiving a protocol-mismatch connect failure has no structured terminal detail code and falls through to reconnect scheduling.

## Solution Assessment

Yes, the narrow maintainable fix is client/protocol error classification and reconnect gating, not allowing incompatible non-probe clients through the gateway.
