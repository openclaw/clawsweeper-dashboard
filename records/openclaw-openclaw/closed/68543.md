The PR changes `retryAsync` to keep jittered delays at or above honor-able `retryAfterMs` values, adds regression tests for boundary/non-integer/over-cap cases, and records a changelog entry plus small test cleanups.

## Best Possible Solution

Keep the merged main implementation and let the next release carry the Retry-After jitter fix to users.

## Reproduction Assessment

## Solution Assessment

Yes. The merged fix is narrow: it changes only the private jitter mode and the one `retryAsync` call-site decision, while preserving symmetric jitter for pure backoff and for over-cap Retry-After values.
