## Summary

Close as duplicate of #73525. This report describes the same remaining bug: default `openclaw status` can render an empty Channels table while live gateway-backed status paths still show channel information. Current main still builds the visible default Channels table from the read-only/config discovery path, while `--deep` fetches gateway health separately.

## Best Possible Solution

Keep #73525 as the canonical open fix. The likely product/code path is to make default `openclaw status` populate the visible Channels table from the reachable gateway account snapshot when available, while preserving the existing read-only/config table as the fallback for unreachable gateways and adding a regression test for a live gateway payload producing non-empty default status rows.

## Evidence

- **Canonical open duplicate:** The provided related context shows #73525 is open and tracks the same remaining user problem: `openclaw status` shows an empty Channels table while a live channel status path shows the configured channels correctly.
- **Default status fetches live channel status but does not use it for table rows:** `collectStatusScanOverview` calls gateway method `channels.status`, derives `channelIssues` from that payload, then separately calls `buildChannelsTable(cfg, { sourceConfig })` for the visible Channels table. The live `channelsStatus.channelAccounts` payload is not the table source.
