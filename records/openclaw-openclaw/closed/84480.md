## Summary

Keep this PR open for normal maintainer review: the refreshed patch looks correct and has sufficient source-entry CLI proof, but it intentionally changes malformed `--limit` values from lenient coercion to fail-fast parsing, so maintainers should accept that compatibility change before merge.

## What This Changes

This PR replaces `parseInt`-based `--limit` parsing in `plugins search` and `skills search` with a Commander-scoped strict positive-integer parser and adds command-path tests for invalid and signed valid values.

## Best Possible Solution

Land the strict parser and focused tests after maintainers accept the CLI compatibility change, then let the linked bug close when the fix merges.

## Reproduction Assessment

Yes. Source inspection on current main shows both affected search commands still use `Number.parseInt(value, 10)`, and the PR body supplies before/after source-entry CLI output for the invalid and valid paths.

## Solution Assessment

Yes. The PR uses the existing strict positive-integer parser behind a small Commander-only adapter, which is narrower than changing shared helpers or the search action runtime paths.
