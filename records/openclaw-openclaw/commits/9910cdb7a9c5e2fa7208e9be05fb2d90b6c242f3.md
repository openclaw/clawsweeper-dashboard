# Commit 9910cdb

## Summary

Found one low-severity live-test reliability regression: the new retry wrapper allows more per-attempt time than the enclosing Vitest test timeout can actually provide.

## Findings

### Low: second live retry cannot receive its declared timeout budget

- Kind: reliability
- File: `src/agents/openai-ws-stream.e2e.test.ts`
- Line: 387
- Evidence: the test loops over two attempts and wraps each attempt with a `75_000` ms timeout, but the Vitest timeout for the whole test remains `120_000` ms at line 474. If attempt 1 stalls until the new helper timeout, attempt 2 starts around 75s into the test and has only about 45s before Vitest aborts the whole test.
- Impact: the commit intends to retry a stalled WebSocket reasoning turn, but a valid second attempt that needs between 45s and 75s will still fail with the outer Vitest timeout. That can keep the keyed live/release lane flaky, especially for the high-reasoning WebSocket turn this test exercises.
- Suggested fix: align the budgets: either raise the test timeout to at least two attempt timeouts plus cleanup/headroom, e.g. `180_000`, or reduce the per-attempt timeout so two attempts fit inside `120_000`. Defining shared constants would keep the relationship explicit.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes, by the static timeout budget: first attempt reaches `75_000`, then second attempt is interrupted by the enclosing `120_000` timeout before its own `75_000` budget can expire. Local live reproduction was blocked because no OpenAI key is configured.
- Is this the best way to solve the issue? yes, the fix should be to make the outer test timeout match the retry budget rather than changing production WebSocket behavior.
