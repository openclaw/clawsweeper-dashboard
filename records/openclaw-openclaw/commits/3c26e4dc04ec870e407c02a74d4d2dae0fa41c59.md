# Commit 3c26e4dc04

## Summary

Found one concrete security hardening regression in the new mode-preservation path: it preserves POSIX special mode bits, not just normal read/write/execute permissions.

## Findings

### Medium: Sandbox writes preserve setuid/setgid bits on atomic replacement

- Kind: security
- File: `src/agents/sandbox/fs-bridge-mutation-helper.ts`
- Line: 98
- Evidence: `existing_regular_file_mode` returns `stat.S_IMODE(target_stat.st_mode)`, then `write_atomic` applies that value with `os.fchmod(temp_fd, target_mode)`. Python `S_IMODE` includes special bits such as setuid/setgid/sticky, not only `0o777`. A focused live check against the exported helper rewrote a `0o4755` file and left it `0o4755` afterward.
- Impact: In a sandboxed writable mount, editing an existing setuid/setgid file now keeps that privilege bit on attacker-controlled replacement content. This is a new behavior from the commit: before this patch, the temp replacement was created as `0600`, so the special executable bit did not survive. The practical exploit requires a preexisting special-bit file in a writable sandbox mount and a runtime able to set those bits, but that is exactly the sort of host-boundary footgun the sandbox bridge should avoid.
- Suggested fix: preserve only ordinary permission bits for write replacement, e.g. return/apply `stat.S_IMODE(target_stat.st_mode) & 0o777`, and add a regression test proving a `04755` or `02755` target becomes `0755`/`0755` after write while `0644` and `0600` still preserve correctly.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The exported helper preserves `0o4755` in a temp directory with `write`, producing `{"before":"0o4755","after":"0o4755","specialPreserved":true}`.
- Is this the best way to solve the issue? yes. Masking to `0o777` keeps the intended fix for #44077 while avoiding privilege-bit preservation during temp-file replacement.
