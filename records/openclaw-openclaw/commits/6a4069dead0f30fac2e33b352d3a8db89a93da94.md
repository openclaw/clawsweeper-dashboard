# Commit 6a4069d

## Summary

Found one actionable regression in the QQBot STT path. The commit replaced raw `fetch` with the shared SSRF-guarded fetch, but did not provide a policy or config opt-in for the operator-configured STT endpoint. That makes the existing QQBot STT unit test fail before its mocked `fetch` runs and blocks localhost/self-hosted OpenAI-compatible transcription URLs in production.

## Findings

### Medium: QQBot STT now blocks configured localhost/self-hosted transcription endpoints

- Kind: regression
- File: `extensions/qqbot/src/engine/utils/stt.ts`
- Line: 83
- Evidence: `transcribeAudio` now calls `fetchWithSsrFGuard` with only `url`, `auditContext`, and `init`. The QQBot STT schema exposes `baseUrl`, `apiKey`, and `model`, but no private-network policy or opt-in at `extensions/qqbot/src/config-schema.ts:16`. The focused existing test `pnpm test extensions/qqbot/src/engine/utils/stt.test.ts` now fails with `getaddrinfo ENOTFOUND api.example.test` because the guard resolves DNS before the test’s mocked `fetch` is called. A direct local smoke with `channels.qqbot.stt.baseUrl: "http://127.0.0.1:65535/v1"` fails immediately with `SsrFBlockedError: Blocked hostname or private/internal/special-use IP address`.
- Impact: QQBot voice transcription can stop working for users who point `channels.qqbot.stt.baseUrl` or the `tools.media.audio.models[0]` fallback at a local/self-hosted OpenAI-compatible STT endpoint. This is a behavioral regression from the parent commit, where the same configured URL was passed directly to `fetch`.
- Suggested fix: Treat the STT base URL as operator-configured endpoint metadata and pass an explicit SSRF policy for that configured hostname, or add/propagate a QQBot STT network opt-in equivalent to other private-network-aware providers. Also update `stt.test.ts` so the guarded fetch path is covered without relying on DNS for mocked fetches.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Run `pnpm test extensions/qqbot/src/engine/utils/stt.test.ts`; the `posts audio to OpenAI-compatible transcription endpoint` test fails with `getaddrinfo ENOTFOUND api.example.test`. A local endpoint smoke also fails before network I/O with `SsrFBlockedError`.
- Is this the best way to solve the issue? yes, directionally. The URL is from operator config, not arbitrary user input, so the safe fix is to carry a prepared policy or explicit opt-in into the guarded fetch rather than reverting to an unguarded request.
