# Commit a4fd45c

## Summary

Found one actionable regression in the new StepFun setup metadata. The added generic `authMethods: ["api-key"]` descriptors create extra onboarding choices that do not map to any StepFun runtime auth method.

## Findings

### Medium: StepFun setup now advertises broken generic auth choices

- Kind: regression
- File: `extensions/stepfun/openclaw.plugin.json`
- Line: 13
- Evidence: The commit adds `setup.providers[].authMethods: ["api-key"]` for `stepfun` and `stepfun-plan`, but the runtime provider only registers `standard-api-key-cn`, `standard-api-key-intl`, `plan-api-key-cn`, and `plan-api-key-intl` in `extensions/stepfun/index.ts`. A focused smoke showed `resolveManifestProviderAuthChoices()` now exposes extra `stepfun-api-key` and `stepfun-plan-api-key` choices, while `resolveProviderPluginChoice()` returns `null` for both.
- Impact: Provider setup/onboarding surfaces can present two additional StepFun choices that cannot be applied. Selecting one will fail to resolve an auth method and bounce back to selection instead of configuring StepFun.
- Suggested fix: Either omit `authMethods` from the StepFun setup descriptors and keep only `envVars`, or list the exact runtime method ids already covered by `providerAuthChoices` so descriptor-derived choices are deduped.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Run `resolveManifestProviderAuthChoices().filter(c => c.pluginId === "stepfun")`; it includes `stepfun-api-key` and `stepfun-plan-api-key`. Then load StepFun setup providers and call `resolveProviderPluginChoice()` for those values; both return `null`.
- Is this the best way to solve the issue? yes. The bug is in the manifest descriptor contract, not runtime auth execution. Aligning or removing `authMethods` keeps the new env metadata without advertising nonexistent auth methods.
