# Commit 42773cb8

## Summary

Found one source-tree regression from removing the jiti fallback in the bundled channel module loader. Focused channel tests now fail because source `.ts` bundled channel artifacts are rejected in Vitest/source-checkout contexts that do not install a native CommonJS `.ts` require hook.

## Findings

### Medium: Source-tree bundled channel modules are no longer loadable without a native `.ts` require hook

- Kind: regression
- File: `src/channels/plugins/module-loader.ts`
- Line: 19
- Evidence: `loadChannelPluginModule()` now throws for non-JavaScript module paths unless `nodeRequire.extensions[extension]` exists. In a normal Vitest run, that hook is absent, while the source checkout resolves bundled channel entries and package-state probes to files like `extensions/slack/index.ts` and `extensions/discord/configured-state.ts`. The previous loader used `getCachedPluginJitiLoader(...)`, which handled these source `.ts` modules.
- Impact: Source-tree channel metadata paths silently degrade. `bundled.ts` catches the loader failure and treats the Slack entry as missing, so `hasBundledChannelEntryFeature("slack", "accountInspect")` returns `false`. `package-state-probes.ts` catches the same failure and returns `null`, so configured-state checks such as Discord env-token detection return `false`. This breaks focused tests and can misreport channel setup/configured state in source-tree/test contexts.
- Suggested fix: Restore a jiti fallback for source `.ts`/`.mts`/`.cts` modules when no native require hook is present, or keep the native path only for built JavaScript and route source artifacts through the existing cached plugin jiti loader.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. After `pnpm install`, `pnpm test src/channels/plugins/bundled.shape-guard.test.ts` fails at `src/channels/plugins/bundled.shape-guard.test.ts:164`, and `pnpm test src/channels/config-presence.test.ts src/channels/plugins/configured-state.ts src/channels/plugins/persisted-auth-state.ts` fails via `src/channels/plugins/configured-state.test.ts:21`.
- Is this the best way to solve the issue? yes, likely by restoring source-module jiti fallback while preserving native require for built JS. That keeps the intended “without jiti” path for bundled built artifacts without breaking source-checkout contracts.
