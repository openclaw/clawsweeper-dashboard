# Commit 42d8255

## Summary

Found one low-severity lookup regression in the new bounded TUI session refresh path. The picker recency limit itself looks intentional and covered.

## Findings

### Low: Exact session refresh can miss the active session when newer prefix matches fill the lookup limit

- Kind: bug
- File: `src/tui/tui-session-actions.ts`
- Line: 238
- Evidence: `refreshSessionInfo()` now calls `sessions.list` with `limit: 5` and `search: state.currentSessionKey`, then looks for the exact key only within those returned rows. Gateway search is substring-based across display fields and `key` (`src/gateway/session-utils.ts:1966`), and the limit is applied after sorting by `updatedAt` (`src/gateway/session-utils.ts:1997`). A store with active key `agent:main:main` plus five newer keys like `agent:main:main-1` through `agent:main:main-5` will satisfy the search with the newer prefix matches and exclude the exact older active row before the TUI exact-match check runs. The remembered-session restore path has the same pattern in `src/tui/tui.ts:639`.
- Impact: The TUI can fail to refresh metadata for the active or remembered session, leaving footer/model/session details stale or preventing restore of the last selected session. Before this commit, the unbounded agent-filtered list would eventually include the exact row.
- Suggested fix: Use a real exact-key lookup for these refresh paths, or have `sessions.list` prioritize/include exact `key` matches before applying `limit` when used as a lookup. Avoid relying on substring `search` plus a small recency-sorted limit for exact identity.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes: create a session store where the target key is older than five newer keys sharing the target as a prefix, then call `sessions.list` with `{ limit: 5, search: "<target>", agentId: "main" }` and observe that the target is absent.
- Is this the best way to solve the issue? The recency-bound picker is reasonable, but the active-session and remembered-session refreshes need exact lookup semantics rather than picker-style fuzzy search.
