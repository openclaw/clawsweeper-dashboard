# Commit 2d8fa23447

## Summary

Found one actionable regression in the new async media completion mediation path. The code applies message-tool-only group/channel policy, but it does not make the `message` tool available to the completion agent or fall back to automatic delivery when the tool is unavailable.

## Findings

### Medium: Async media completions can be silently dropped for default coding-profile group/channel sessions

- Kind: regression
- File: `src/agents/subagent-announce-delivery.ts`
- Line: 876
- Evidence: For `music_generate`/`video_generate`, the new path treats default group/channel policy as requiring message-tool delivery (`completionRequiresMessageToolDelivery` returns true unless group visible replies are `"automatic"`), sets `shouldDeliverAgentFinal` false, and calls the requester agent with `deliver: false`. It then returns `delivered: false` if the agent result lacks messaging-tool evidence. However, the normal source-turn path explicitly adds `message` to the runtime profile when message-tool delivery is preferred and falls back to automatic mode when the message tool is unavailable. This async media path does neither. Under `tools.profile: "coding"`, `message` is not in the coding profile, so a generated media completion in a default group/channel route can complete successfully, run the requester agent without the required tool, then only log `Media generation completion wake failed` instead of delivering the finished media.
- Impact: Users in Slack/Discord/etc. group/channel sessions with the common coding tool profile can lose the completion notification and media attachment for successful background `music_generate`/`video_generate` jobs. The task ledger says generation succeeded, but the requester chat never receives the result unless the agent happens to have `message` through a full/messaging profile or explicit allowlist.
- Suggested fix: Reuse the same effective source-reply policy logic as source turns: when a media completion requires message-tool delivery, force/also-allow `message` for the requester completion run, or detect that `message` is unavailable and fall back to `deliver: true` automatic delivery. A focused regression test should cover `tools.profile: "coding"` plus default group/channel visible replies.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. Configure an agent with `tools.profile: "coding"` and default group/channel visible replies, start `music_generate` or `video_generate` from a group/channel session, and let the background job complete. The completion path will call the requester agent with `deliver: false`; if the agent cannot call `message`, `hasGatewayAgentMessagingToolDelivery` remains false and delivery is reported failed.
- Is this the best way to solve the issue? No. The commit duplicated only the config check, not the existing effective source-turn policy that accounts for tool availability. Reusing that policy or adding an equivalent message-tool availability check is safer than relying on the model to use a tool that may not exist.
