# Commit 60e6ccd

## Summary

Found one concrete regression risk in the new settling-state auto retry path for first-time TLS gateway pairing.

## Findings

### Medium: Settling auto-retry can overwrite first TLS pairing with a false cached-endpoint error

- Kind: regression
- File: `apps/android/app/src/main/java/ai/openclaw/app/ui/OnboardingFlow.kt`
- Line: 526
- Evidence: The new `gatewayRecoveryUiState()` returns `Finishing` whenever `connectSettling` is true, and `Finishing.canAutoRetry` is true. `GatewayRecoveryScreen` therefore starts `PairingAutoRetryEffect` immediately after a pair attempt. That effect now fires after 1.5s. For TLS gateways, `NodeRuntime.beginConnect()` does not set `connectedEndpoint` until after the async TLS probe/trust flow reaches `connectAfterTlsCheck`; meanwhile `refreshGatewayConnection()` reports `Failed: no cached gateway endpoint` when called without `connectedEndpoint`.
- Impact: First-time `wss://` or trusted remote gateway onboarding can show or copy a false “no cached gateway endpoint” failure while the TLS fingerprint prompt/probe is still in progress. This is a regression from the previous behavior, which only auto-retried when the status looked like pairing/approval, not during generic settling.
- Suggested fix: Do not enable auto retry solely for the initial `connectSettling` state. Keep auto retry for actual pairing/partial-connect statuses, or make the runtime refresh path no-op while an initial connect attempt is still awaiting TLS trust/caching.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Unclear; the code path is concrete, but I did not run an Android UI/TLS-prompt scenario. Repro path should be: start onboarding with a first-time TLS gateway requiring fingerprint trust, wait 1.5s before the endpoint is cached/trusted, observe `refreshGatewayConnection()` setting `Failed: no cached gateway endpoint`.
- Is this the best way to solve the issue? Yes for the UI side: separate “settling” display from “safe to retry” behavior so the screen can stay calm without issuing reconnects before the runtime has a cached endpoint.
