# Commit ff45bc1

## Summary

Found one user-facing bug in the native WebChat transcript bridge: Talk Mode live transcript events can render the full Talk prompt wrapper instead of just the spoken transcript.

## Findings

### Medium: Talk transcript rows can expose the internal Talk prompt wrapper

- Kind: bug
- File: `apps/shared/OpenClawKit/Sources/OpenClawChatUI/ChatViewModel.swift`
- Line: 982
- Evidence: Talk Mode builds a prompt with UI-internal instructions before the spoken text (`TalkPromptBuilder.build`, `apps/shared/OpenClawKit/Sources/OpenClawKit/TalkPromptBuilder.swift:7`) and sends that full prompt through `chatSend` (`apps/macos/Sources/OpenClaw/TalkModeRuntime.swift:386`). The Gateway then emits the same `parsedMessage` as the user transcript event content (`src/gateway/server-methods/chat.ts:785`, `src/gateway/server-methods/chat.ts:2187`). The new Swift handler appends the projected `session.message` after only generic inbound-metadata stripping (`apps/shared/OpenClawKit/Sources/OpenClawChatUI/ChatViewModel.swift:982`), and that stripper does not know about the Talk prompt header (`apps/shared/OpenClawKit/Sources/OpenClawChatUI/ChatMarkdownPreprocessor.swift:7`). The added test only covers a clean `"spoken transcript"` payload, not the real Talk prompt shape (`apps/shared/OpenClawKit/Tests/OpenClawKitTests/ChatViewModelTests.swift:692`).
- Impact: macOS Talk Mode user rows may show implementation instructions such as “Talk Mode active...” and the ElevenLabs JSON hint above the actual spoken text. That leaves the reported feature only partially fixed and exposes internal prompt text in the chat thread.
- Suggested fix: keep display text separate from agent prompt text for Talk Mode. Prefer emitting/persisting the raw spoken transcript as the visible user message while delivering Talk instructions through an explicit non-display field or system/provenance path; alternatively add a targeted Talk prompt display sanitizer with tests using `TalkPromptBuilder.build(...)`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes, source-level: open native WebChat, use macOS Talk Mode, and inspect the `session.message` user row. The code path sends `TalkPromptBuilder.build(transcript:)` as `chat.send.message`, the Gateway broadcasts that as transcript content, and the Swift handler appends it without stripping the Talk-specific wrapper.
- Is this the best way to solve the issue? No. Subscribing native WebChat to `session.message` is the right bridge, but the display contract should avoid using the agent prompt as the user-visible transcript.
