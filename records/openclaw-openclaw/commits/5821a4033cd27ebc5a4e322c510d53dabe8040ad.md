# Commit 5821a4033c

## Summary

Found one delivery edge case in the new native image output projection. Current `main` still has the reviewed behavior unchanged.

## Findings

### Medium: Native image media is dropped when any messaging-tool media was sent

- Kind: bug
- File: `extensions/codex/src/app-server/event-projector.ts`
- Line: 692
- Evidence: `buildToolMediaUrls()` only appends `nativeGeneratedMediaUrls` when `messagingToolSentMediaUrls` is empty. But `messagingToolSentMediaUrls` is global telemetry collected from any messaging-tool send in the turn (`extensions/codex/src/app-server/dynamic-tools.ts:287`), while final reply delivery has route-aware media dedupe using `messagingToolSentTargets` (`src/auto-reply/reply/agent-runner-payloads.ts:223`, `src/auto-reply/reply/reply-payloads-dedupe.ts:281`). Because the projector drops native saved paths before that layer, a turn that both emits a Codex `imageGeneration.savedPath` and sends unrelated or cross-target media through a messaging tool never gives the native image to the final delivery path.
- Impact: Users can still get a blank/no-media final reply for native Codex image generation in mixed media/message-tool turns, which is the class of regression this commit is intended to fix.
- Suggested fix: Always pass native `savedPath` values into `toolMediaUrls` and let existing route-aware final payload dedupe suppress only true same-route duplicates, or make the suppression compare the native path against same-route sent media instead of checking only global array length.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes at the projector/unit level: send a `turn/completed` notification containing an `imageGeneration` item with `savedPath`, then call `buildResult()` with `messagingToolSentMediaUrls` containing a different media URL and a non-matching `messagingToolSentTargets` route. The current code returns no native media because line 692 gates on array length only.
- Is this the best way to solve the issue? Unclear. The safer direction is to keep dedupe in the existing final delivery layer, because it already knows origin route and messaging-tool targets.
