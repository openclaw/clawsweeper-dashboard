# Commit f976454

## Summary

Found one regression in the new Xiaomi provider hook registration: the provider-local MiMo matcher is stricter than the generic fallback it now suppresses, so suffixed/prefixed MiMo model ids can lose the Xiaomi thinking payload wrapper.

## Findings

### Medium: Xiaomi hook disables the more permissive MiMo fallback for suffixed model ids

- Kind: regression
- File: `extensions/xiaomi/index.ts`
- Line: 39
- Evidence: The new `wrapStreamFn` always installs a provider wrapper for Xiaomi when a base stream exists, which makes `applyExtraParamsToAgent` treat the provider wrapper as handled and skip the generic MiMo fallback in `src/agents/pi-embedded-runner/extra-params.ts:748`. That fallback normalizes model ids by stripping `:` suffixes and `/` prefixes before matching MiMo ids at `src/agents/pi-embedded-runner/extra-params.ts:829`. The new Xiaomi wrapper calls `isMiMoReasoningModelRef`, which only accepts exact ids from `extensions/xiaomi/thinking.ts:3` and rejects ids like `mimo-v2.5-pro:cloud` or `xiaomi/mimo-v2.5-pro:cloud`.
- Impact: Custom or routed Xiaomi MiMo model ids that previously got `thinking: { type: "enabled" }`, `reasoning_effort`, and replayed assistant `reasoning_content` backfills can now be sent without those fields. That can break multi-turn tool continuations for MiMo reasoning models. Current transport tests already encode the suffixed-id contract for reasoning-content replay in `src/agents/openai-transport-stream.test.ts:5557`.
- Suggested fix: Move the MiMo id normalization into `extensions/xiaomi/thinking.ts` and use it for `isMiMoReasoningModelId`, `isMiMoReasoningModelRef`, `resolveMiMoThinkingProfile`, and `isModernModelRef`, matching the existing fallback behavior: lowercase, strip a `:` suffix, split on `/`, and match the final segment.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. Source path: configure/use provider `xiaomi` with model id `mimo-v2.5-pro:cloud` or `xiaomi/mimo-v2.5-pro:cloud`; the new provider wrapper is considered handled, but its exact matcher returns false, so the generic normalizing fallback no longer patches the payload.
- Is this the best way to solve the issue? Yes. Normalizing in the Xiaomi-owned matcher preserves the new provider-local ownership while retaining the prior supported model-id forms.
