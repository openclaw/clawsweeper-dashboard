# Commit 2bb00f6

## Summary

Found one concurrency regression in the session write fence. The new fence refresh hook marks Pi `message_end` transcript writes as owned after they have already been appended, but the real Pi event persistence path is not wrapped by the new session write lock.

## Findings

### Medium: Pi `message_end` appends can bypass takeover checks and then refresh the fence

- Kind: concurrency
- File: `src/agents/pi-embedded-runner/run/attempt.ts`
- Line: 2122
- Evidence: The commit wires `onMessagePersisted` to `sessionLockController.refreshAfterOwnedSessionWrite()` for every guarded `SessionManager.appendMessage`. That callback is invoked immediately after `originalAppend()` in `src/agents/session-tool-result-guard.ts:601`. However, the lock installer in `src/agents/pi-embedded-runner/run/attempt.session-lock.ts:249` only wraps a private `_processAgentEvent` method. The pinned `@earendil-works/pi-coding-agent@0.75.4` source has no `_processAgentEvent` or `_agentEventQueue`; it subscribes `AgentSession._handleAgentEvent` and persists `message_end` via `this.sessionManager.appendMessage(event.message)` in `dist/core/agent-session.js`.
- Impact: After `releaseForPrompt()`, a provider response can persist assistant/tool messages through Pi’s real `_handleAgentEvent` path without first calling `sessionLockController.withSessionWriteLock()`. If another owner advanced the session file while the model call was in flight, this path appends anyway and then refreshes the fence to the new fingerprint, so later cleanup no longer reports takeover. That defeats the commit’s intended “reject post-prompt writes after takeover” contract and can merge stale assistant output into a session another owner has already advanced.
- Suggested fix: Fence the actual Pi `AgentSession._handleAgentEvent` `message_end` persistence path, or wrap `SessionManager.appendMessage` itself with `sessionLockController.withSessionWriteLock()` before `originalAppend()`. Only refresh the fence from inside a lock-owned write after `assertSessionFileFence()` has passed.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. A focused repro would use the pinned Pi `AgentSession` shape: call `releaseForPrompt()`, externally append to the session file, then trigger a Pi `message_end` append through `SessionManager.appendMessage`; the current hook refreshes after append without the pre-write fence check.
- Is this the best way to solve the issue? Yes. The fence must be checked before the write that claims ownership; refreshing after an unfenced append masks the takeover.
