# Commit 84dd9c7

## Summary

The commit closes the trusted-proxy password fallback at the server auth gate, but the rest of the current `main` contract still treats `gateway.auth.password` as the local/internal credential for trusted-proxy deployments. That leaves documented internal Gateway callers resolving and sending a password that the server now always rejects.

## Findings

### High: Trusted-proxy local password callers still send an auth path the server now rejects

- Kind: regression
- File: `src/gateway/auth.ts`
- Line: 469
- Evidence: `authorizeGatewayConnectCore` now returns the trusted-proxy failure immediately for `auth.mode === "trusted-proxy"` once proxy identity checks fail, so the password branch is unreachable. But local credential resolution still marks trusted-proxy password as winning in `src/gateway/credentials.ts:122`, and `src/gateway/call.test.ts:1485` still asserts that local Gateway calls resolve `gateway.auth.password` when auth mode is trusted-proxy. The docs also still tell internal Gateway clients to use `gateway.auth.password` / `OPENCLAW_GATEWAY_PASSWORD` in this mode (`docs/gateway/trusted-proxy-auth.md:99`, `docs/gateway/trusted-proxy-auth.md:308`, `docs/gateway/configuration-reference.md:436`). A direct helper smoke confirmed the combined behavior: credentials resolve to `{password:"local-password"}`, then auth returns `{ok:false, reason:"trusted_proxy_loopback_source"}` for the same loopback request.
- Impact: Deployments behind identity-aware proxies that configured `gateway.auth.password` for local/internal OpenClaw callers regress back to the failure mode tracked in #17761: CLI RPC, node host, ACP/TUI, cron, sessions, and other direct loopback clients can no longer authenticate unless they go through the reverse proxy. This is especially likely because current client code and docs continue to steer those callers toward the now-dead password path.
- Suggested fix: Make the policy consistent across server, client credential selection, docs, and tests. If local internal callers must remain supported, restore a narrowly scoped password fallback or add an explicit opt-in internal-auth mode with rate limiting. If fail-closed proxy-only auth is final, reject or ignore `gateway.auth.password` in trusted-proxy mode, stop resolving/sending it from Gateway call paths, update the docs/changelog, and provide a replacement migration path for #17761-style internal callers.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Configure local Gateway auth as `mode: "trusted-proxy"` with `gateway.auth.password`, then make a direct loopback Gateway client call using that password. The client credential resolver returns the password, while `authorizeGatewayConnect` rejects with `trusted_proxy_loopback_source`.
- Is this the best way to solve the issue? no. The server-side fail-closed change may be a valid security policy, but it is incomplete while client credential resolution, tests, and docs still advertise and exercise the removed password path.
