# Commit 26f2241

## Summary

Found one low-severity regression: the fix imports ClawHub error-code constants from the heavy ClawHub installer module, which defeats an existing lazy import boundary in onboarding.

## Findings

### Low: Static ClawHub import makes the onboarding lazy installer import ineffective

- Kind: reliability
- File: `src/commands/onboarding-plugin-install.ts`
- Line: 12
- Evidence: The commit adds `import { CLAWHUB_INSTALL_ERROR_CODE } from "../plugins/clawhub.js";` while the same file still dynamically imports `../plugins/clawhub.js` at `src/commands/onboarding-plugin-install.ts:665`. The base file had only the dynamic import. The focused dynamic-import checker reports this exact new advisory: `runtime static + dynamic import of "../plugins/clawhub.js" (static line 12)`.
- Impact: Any caller that imports onboarding plugin install logic now eagerly loads `src/plugins/clawhub.ts` and its archive/registry dependencies even when the user selects npm, local, or skip. That regresses the intended startup/import boundary and can make non-ClawHub onboarding paths pay ClawHub installer cost or fail earlier if that heavy module has an import-time problem.
- Suggested fix: Keep the fallback predicate local without importing the installer module at runtime, for example compare against local string constants, or move `CLAWHUB_INSTALL_ERROR_CODE` into a small sidecar module that does not import the ClawHub installer.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Run `pnpm exec node scripts/check-dynamic-import-warts.mjs --json` on current `main`; it reports `src/commands/onboarding-plugin-install.ts:665`. Running the same checker function against the base version of this file returns `[]`.
- Is this the best way to solve the issue? yes. The functional fallback change matches existing doctor/update behavior; the safer fix is only to preserve the lazy boundary while keeping the new package/version-only fallback predicate.
