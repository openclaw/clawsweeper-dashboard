# Commit 8a1a862

## Summary

The intended `thread.id -> thread.sessionId` compatibility fill is narrow and covered. The landed commit also accepts the reverse shape, synthesizing `thread.id` from `thread.sessionId`; that can persist a session-tree id as OpenClaw’s thread id and send it back as `threadId` on future app-server calls.

## Findings

### Medium: Do not synthesize `thread.id` from `sessionId`

- Kind: bug
- File: `extensions/codex/src/app-server/protocol-validators.ts`
- Line: 161
- Evidence: `normalizeThreadResponse()` fills missing `thread.id` from `thread.sessionId`, and the new test asserts that behavior in `extensions/codex/src/app-server/protocol-validators.test.ts:52`. The generated Codex app-server schema keeps `id` and `sessionId` as distinct required fields, with `sessionId` documented as shared by threads in the same session tree (`extensions/codex/src/app-server/protocol-generated/json/v2/ThreadStartResponse.json:1418`, `extensions/codex/src/app-server/protocol-generated/json/v2/ThreadStartResponse.json:1506`). OpenClaw stores `response.thread.id` as the binding `threadId` and later sends that value to `thread/resume` and `turn/start` (`extensions/codex/src/app-server/thread-lifecycle.ts:155`, `extensions/codex/src/app-server/thread-lifecycle.ts:224`, `extensions/codex/src/app-server/thread-lifecycle.ts:333`, `extensions/codex/src/app-server/thread-lifecycle.ts:356`).
- Impact: A malformed response with only `sessionId` would now pass validation and poison the binding with a session-tree id, causing later resume/turn/compaction/notification matching to use the wrong identifier instead of failing at the protocol boundary.
- Suggested fix: Keep only the observed compatibility fill from `id` to missing `sessionId`; let missing `id` remain invalid and update the regression test to assert rejection for `sessionId`-only responses.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the code path: feed a `thread/start` or `thread/resume` response with `thread.sessionId` but no `thread.id`; the validator now returns `thread.id = sessionId`, and `startOrResumeThread()` persists that as the binding `threadId`. I did not reproduce against a live Codex app-server that emits this malformed reverse shape.
- Is this the best way to solve the issue? No. The reported/live gap is `id` present and `sessionId` absent; reverse filling broadens the accepted protocol beyond the upstream contract and conflicts with OpenClaw’s use of `thread.id` as the runtime thread handle.
