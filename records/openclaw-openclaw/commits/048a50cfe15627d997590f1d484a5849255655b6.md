# Commit 048a50c

## Summary

Found one test regression. The tightened timestamp assertion now fails on current `main` because the formatter returns `UTC` in this environment while the test expects `GMT`.

## Findings

### Medium: tightened timestamp assertion expects the wrong timezone abbreviation

- Kind: regression
- File: `src/auto-reply/reply/inbound-meta.test.ts`
- Line: 392
- Evidence: The commit changed the assertion from `expect.any(String)` to the exact string `"Sun 2026-02-15 13:35 GMT"`. The production path calls `formatEnvelopeTimestamp()` with default local timezone, which delegates to `formatZonedTimestamp()` and `Intl.DateTimeFormat(..., { timeZoneName: "short" })`. On this current UTC host, that returns `"Sun 2026-02-15 13:35 UTC"`, not `GMT`. Focused test run failed with:
  `expected 'Sun 2026-02-15 13:35 UTC' to be 'Sun 2026-02-15 13:35 GMT'`.
- Impact: The auto-reply test lane fails on current `main`, so changed-test/CI validation for this area can be blocked even though runtime behavior is unchanged.
- Suggested fix: Make the test deterministic against the actual contract. Either set an explicit timezone and expect the formatter’s real output (`UTC` for `TZ=UTC`), pass an envelope option such as `{ timezone: "utc" }` and assert the UTC format, or assert the stable date/time parts without hardcoding the environment-dependent abbreviation.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: after installing missing deps, run `pnpm test src/auto-reply/reply/commands-tts.test.ts src/auto-reply/reply/inbound-meta.test.ts`; `inbound-meta.test.ts` fails at line 392.
- Is this the best way to solve the issue? yes: the production formatter intentionally uses `Intl`, whose short timezone labels vary by runtime/timezone. The test should pin the input timezone or avoid asserting an implementation/platform-dependent label.
