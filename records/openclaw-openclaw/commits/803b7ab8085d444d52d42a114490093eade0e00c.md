# Commit 803b7ab

## Summary

Found one concrete missed diagnostic in the new doctor warning: tilde-prefixed `hooks.transformsDir` values are checked before the same path normalization that Gateway startup uses, so doctor can stay silent for a config that the Gateway still rejects.

## Findings

### Medium: Doctor misses invalid tilde-prefixed hook transform directories

- Kind: reliability
- File: `src/commands/doctor-config-flow.ts`
- Line: 40
- Evidence: `collectInvalidHookTransformsDirWarnings` reads `cfg.hooks?.transformsDir` from the doctor source config and passes non-absolute values to `path.resolve(transformsRoot, transformsDir)`. A value authored as a tilde-prefixed home path is therefore treated as a literal relative child of `hooks/transforms` and does not warn. Runtime materialization does normalize path-like config fields before Gateway hook mapping resolution, and `src/gateway/hooks-mapping.ts:142` then rejects the same normalized workspace skill directory as outside the canonical transforms root.
- Impact: A user can still have a Gateway crash loop from `hooks.transformsDir` pointing at a workspace skill directory, then run `openclaw doctor` and get no `hooks.transformsDir` recovery hint if their config used the documented home-relative tilde form. That leaves the new recovery path incomplete for an equivalent invalid configuration.
- Suggested fix: Apply the same path normalization before this doctor check, preferably by sharing the Gateway contained-path resolver or by checking a cloned/materialized config so `hooks.transformsDir` is interpreted exactly as startup interprets it.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. In a temp-home smoke, `openclaw doctor --non-interactive` emitted the new warning for an absolute invalid workspace skill path, but emitted no `hooks.transformsDir` warning for the equivalent tilde-prefixed value. A focused `tsx` probe showed `normalizeConfigPaths` converts that tilde value to the workspace skill path and `resolveHookMappings` rejects it as outside the hooks transforms root.
- Is this the best way to solve the issue? no. The current fix only matches already-absolute source config values; the safer direction is to reuse the same normalized path semantics as Gateway startup.
