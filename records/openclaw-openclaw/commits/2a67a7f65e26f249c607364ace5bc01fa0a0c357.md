# Commit 2a67a7f

## Summary

Found one cleanup regression in the new managed peer pruning path: uninstall only performs one peer-sync/install pass, so chained managed peer dependencies can remain installed after their owning plugin is removed.

## Findings

### Medium: Transitive managed peers can survive plugin uninstall

- Kind: bug
- File: `src/plugins/uninstall.ts`
- Line: 641
- Evidence: `applyPluginUninstallDirectoryRemoval` calls `syncManagedNpmRootPeerDependencies` once, then runs one `npm install --omit=peer` cleanup if the manifest changed. The sync helper computes desired peer pins from the currently installed `node_modules` tree. If plugin `owner-pkg` peers on `direct-peer`, and `direct-peer` peers on `transitive-peer`, the first sync after uninstall sees `direct-peer` still installed, removes only `direct-peer` from root dependencies, and keeps `transitive-peer`. The following install prunes `direct-peer`, but no second sync runs to discover that `transitive-peer` is now orphaned. Install-side code already loops until managed peers settle in `src/plugins/install.ts:695`, which is the contract this uninstall path needs as well.
- Impact: Uninstall can leave third-party peer packages in the managed npm root even after the plugin that required them is gone. This defeats the commit’s cleanup goal and keeps stale third-party code in `package.json`, `package-lock.json`, and `node_modules` until a later operation happens to reconcile it.
- Suggested fix: Mirror the install-side settle loop for uninstall cleanup: repeatedly run `syncManagedNpmRootPeerDependencies` followed by the safe npm install command until sync returns false, with the same bounded-pass behavior and warning policy appropriate for uninstall.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused temporary npm-root check with packed local packages reproduced it: after uninstalling `owner-pkg`, the result had no warnings but still left `transitive-peer` in root dependencies, `openclaw.managedPeerDependencies`, and `node_modules`.
- Is this the best way to solve the issue? yes. The underlying helper is state-based over the installed package tree, so removing one managed peer can expose another now-orphaned peer only after npm prunes the first one. A bounded settle loop is already the install-side solution.
