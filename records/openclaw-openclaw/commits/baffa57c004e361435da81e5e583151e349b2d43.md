# Commit baffa57

## Summary

Found one low-severity behavior mismatch: the commit restores rolling-label semantics for text progress drafts, but Slack rich progress drafts still render the label as a separate sticky block.

## Findings

### Low: Slack rich progress drafts keep the label sticky while text drafts roll it away

- Kind: bug
- File: `extensions/slack/src/monitor/message-handler/dispatch.ts`
- Line: 972
- Evidence: `src/plugin-sdk/channel-streaming.ts:795` now prepends the draft label to the same line list and `src/plugin-sdk/channel-streaming.ts:810` slices the combined list by `maxLines`, so text output with label `Shelling...`, `maxLines: 3`, and three tool lines renders only the three tool lines. The Slack rich path still passes `label: resolveChannelProgressDraftLabel(...)` separately at `extensions/slack/src/monitor/message-handler/dispatch.ts:972`, and `extensions/slack/src/progress-blocks.ts:50` always renders that label block before the progress rows, slicing only by Slack’s 50-block cap.
- Impact: users with `channels.slack.streaming.progress.render: "rich"` see different behavior from the shared text fallback and the updated progress-drafts docs: the starter label does not scroll away after enough work appears. This can leave rich Slack drafts with one extra visible row and stale starter text while text mode, Discord, Teams, Telegram, and Matrix follow the restored rolling behavior.
- Suggested fix: make the Slack rich block builder use the same rolling budget as `formatChannelProgressDraftText`, for example by treating the label as a first pseudo-row before applying `streaming.progress.maxLines`, then rendering the surviving rows as blocks.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused `node --import tsx` check showed `formatChannelProgressDraftText({ label: "Shelling...", maxLines: 3, lines: [one, two, three] })` outputs only the three tool rows, while `buildSlackProgressDraftBlocks({ label: "Shelling...", lines })` still outputs a `Shelling...` section plus all three tool rows.
- Is this the best way to solve the issue? yes. The shared formatter is now the canonical text behavior, so Slack rich rendering should derive its block list from the same label-plus-lines rolling model rather than owning separate sticky-label semantics.
