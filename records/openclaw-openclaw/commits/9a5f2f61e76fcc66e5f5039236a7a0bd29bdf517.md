# Commit 9a5f2f6

## Summary

Found one concrete regression: `doctor --lint` is documented and implemented as a read-only mode, but the final config validation health check still performs an observing config snapshot read and writes `config-health.json`.

## Findings

### Medium: `doctor --lint` still writes config health state

- Kind: regression
- File: `src/flows/doctor-core-checks.ts`
- Line: 177
- Evidence: `runDoctorLintCli()` starts with `readConfigFileSnapshot({ observe: false })`, but it then runs registered checks. The final config validation check calls `readConfigFileSnapshot()` without `{ observe: false }`, and config IO defaults `observe` to true before calling `observeConfigSnapshot()`. A focused smoke with a temp HOME reproduced this: `node scripts/run-node.mjs doctor --lint --only core/doctor/final-config-validation --json` exited `0` with `{"ok":true,"checksRun":1,"checksSkipped":4,"findings":[]}` and still created `.openclaw/logs/config-health.json`.
- Impact: This violates the new lint contract promised in docs as “read-only” and “does not ... rewrite config/state.” CI/preflight runs can now mutate OpenClaw state even when operators chose lint specifically to avoid writes.
- Suggested fix: Make `finalConfigValidationCheck.detect()` use the already-loaded lint context/config snapshot, or call `readConfigFileSnapshot({ observe: false })`. Add a regression test that a valid `doctor --lint` run, including `core/doctor/final-config-validation`, does not create `config-health.json`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: run `doctor --lint --only core/doctor/final-config-validation --json` against a temp HOME with a valid `openclaw.json`; the command succeeds and creates `.openclaw/logs/config-health.json`.
- Is this the best way to solve the issue? yes: the intended non-observing read already exists, and this check is the remaining path that bypasses it.
