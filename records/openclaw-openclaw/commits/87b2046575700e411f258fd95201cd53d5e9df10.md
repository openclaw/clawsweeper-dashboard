# Commit 87b2046

## Summary

The commit fixes legacy reload history for targetless `message.send` replies, but the projection fallback now creates visible assistant mirrors without coordinating with the existing durable `delivery-mirror` path and without using the delivered/sanitized payload as source of truth.

## Findings

### Medium: Synthetic message-tool mirrors duplicate durable delivery mirrors

- Kind: regression
- File: `src/gateway/chat-display-projection.ts`
- Line: 793
- Evidence: `flushSucceededMirrors` unconditionally pushes a synthetic assistant mirror once a successful targetless `message.send` is followed by a silent control reply. Current main also persists delivered internal source replies as assistant messages with `provider: "openclaw"` and `model: "delivery-mirror"` via `sourceReplyTranscriptMirror` metadata and `appendAssistantMessageToSessionTranscript` (`src/agents/pi-embedded-runner/run/payloads.ts:567`, `src/auto-reply/reply/dispatch-from-config.ts:1510`, `src/config/sessions/transcript.ts:250`). A targeted projection smoke with `toolCall -> toolResult -> NO_REPLY -> delivery-mirror` returned two visible assistant entries with the same text: one `openclawMessageToolMirror` and one durable `delivery-mirror`.
- Impact: Current message-tool-only runs that already append a durable transcript mirror can show the same assistant reply twice after `chat.history` reload/refresh.
- Suggested fix: Scope the synthetic projection mirror to legacy transcripts without a durable `delivery-mirror` for the same delivered reply, or dedupe/merge synthetic mirrors against delivery mirrors before returning display history.
- Confidence: high

### Medium: Projection mirror uses raw tool-call text instead of delivered sanitized text

- Kind: privacy
- File: `src/gateway/chat-display-projection.ts`
- Line: 561
- Evidence: `readMessageToolVisibleText` reads `message`, `text`, `content`, `body`, or `caption` directly from the tool-call arguments and only applies `stripInlineDirectiveTagsForDisplay`. The actual send path parses reply directives and strips unsupported citation control markers before delivery (`src/infra/outbound/message-action-runner.ts:775`, `src/infra/outbound/message-action-runner.ts:802`), and the existing send-validation test asserts an internal UI source reply removes `turn2view0` from the delivered payload (`src/infra/outbound/message-action-runner.send-validation.test.ts:121`). A targeted projection smoke with raw text `v2026.5.20 release note citeturn2view0` still emitted that marker in the synthetic assistant mirror.
- Impact: Reload history can display text that was not actually delivered, including citation/control markers or directive-bearing raw argument text that the delivery path strips or normalizes.
- Suggested fix: Build the history mirror from sanitized delivered payload/result data such as `sourceReply` / `sourceReplyTranscriptMirror`, or apply the same delivery normalization before mirroring; skip the fallback when sanitized delivery evidence is unavailable.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Both findings were reproduced with focused `node --import tsx --input-type=module` projection probes against `projectChatDisplayMessages`.
- Is this the best way to solve the issue? yes. The durable delivered payload is the safer source of truth; the projection fallback should only backfill legacy history and should not reconstruct visible chat text from raw model arguments.
