# Commit f62618f

## Summary

Found one compatibility/reliability bug in the new hostname-scoped remote sandbox handling.

## Findings

### Medium: Remote sandbox defaults can miss FQDN-only host matches and still send YOLO

- Kind: bug
- File: `extensions/codex/src/app-server/config.ts`
- Line: 551
- Evidence: OpenClaw matches `remote_sandbox_config` using `params.hostName ?? os.hostname()`, but Codex 0.129.0 first normalizes the kernel hostname and then prefers the resolver-provided canonical FQDN before applying `remote_sandbox_config`. A focused resolver check showed a remote-only policy with `hostname_patterns = ["runner-*.ci.example.com"]` resolves to `never` + `danger-full-access` for `hostName: "runner-01"`, but to guardian + `workspace-write` for `hostName: "runner-01.ci.example.com"`.
- Impact: On managed hosts whose local hostname is short but Codex resolves an FQDN that matches remote sandbox requirements, OpenClaw still sends `danger-full-access`. Codex then applies the remote requirement internally and rejects the thread/turn, so the hostname-scoped case this commit documents can still fail.
- Suggested fix: Mirror Codex’s FQDN hostname resolution before matching, or better, start/initialize app-server and use Codex’s loaded `configRequirements/read` data before building thread defaults so system, MDM, cloud, and remote-applied requirements cannot drift from Codex.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes: run the resolver with a remote-only FQDN pattern and a short hostname, then compare with the FQDN hostname. The short-host case returns YOLO while Codex’s own host-name code can classify by FQDN.
- Is this the best way to solve the issue? Unclear. Matching Codex’s hostname canonicalization is the narrow fix; querying Codex’s loaded requirements is safer because it avoids reimplementing TOML, source precedence, and cloud/MDM layers.
