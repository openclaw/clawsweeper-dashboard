# Commit d0a74db

## Summary

Found one concrete path-boundary regression in the new memory-flush target pre-create helper. The focused regression tests pass, but they do not cover symlinked workspace paths.

## Findings

### Medium: Memory flush target pre-create follows workspace symlinks outside the workspace

- Kind: security
- File: `src/auto-reply/reply/agent-runner-memory.ts`
- Line: 109
- Evidence: `ensureMemoryFlushTargetFile` validates `relativePath` lexically under `workspaceRoot`, then calls `fs.promises.mkdir(path.dirname(targetPath), { recursive: true })` and `fs.promises.open(targetPath, "a")`. Both operate on the raw resolved path, so a pre-existing `memory` directory symlink or `memory/YYYY-MM-DD.md` symlink is followed after the lexical check. A focused Node probe matching this logic created `outside/created.md` through a workspace symlink at `memory/2026-05-22.md`. The existing append-only memory write path uses the guarded `fsRoot(...).append(...)` path in `src/agents/pi-tools.read.ts`, which is covered by fs-safe symlink/hardlink rejection tests.
- Impact: A workspace-controlled symlink can make the automatic memory flush pre-create step create or open a file outside the workspace with OpenClaw’s process privileges. The helper does not write content, but arbitrary out-of-workspace file creation/opening is still a workspace boundary bypass and weakens the safety contract this commit is trying to enforce.
- Suggested fix: Route the pre-create through the same safe filesystem abstraction used by the memory write tool, or add a no-follow “touch if missing” helper that rejects symlink and hardlink aliases for every path component and the final target. If sandboxed writes are active, use the sandbox bridge rather than a raw host `fs.open`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The reproduction is: create a temp workspace, symlink `workspace/memory/2026-05-22.md` to a missing file in a separate temp directory, run the same lexical `path.resolve/path.relative` check, then `fs.open(targetPath, "a")`; the outside file is created.
- Is this the best way to solve the issue? yes. The fix should reuse the existing guarded write boundary instead of adding another raw filesystem path that must independently handle symlinks, hardlinks, sandbox roots, and TOCTOU cases.
