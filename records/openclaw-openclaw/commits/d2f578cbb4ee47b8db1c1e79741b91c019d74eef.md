# Commit d2f578c

## Summary

Found one concurrency risk in the new timeout-retirement path: a single timed-out Codex attempt now closes the process-level shared app-server client, which can reject unrelated in-flight requests from other active Codex sessions using the same shared client.

## Findings

### Medium: A timed-out run can close the shared Codex app-server under other active runs

- Kind: concurrency
- File: `extensions/codex/src/app-server/run-attempt.ts`
- Line: 1509
- Evidence: the new abort listener calls `retireCodexAppServerClientAfterTimedOutTurn(client, ...)` whenever the local attempt sets `timedOut`. The default run-attempt client factory uses `getSharedCodexAppServerClient`, not an isolated per-run client (`extensions/codex/src/app-server/client-factory.ts:16`). Retiring a current shared client clears the global shared slot and calls `client.close()` (`extensions/codex/src/app-server/shared-client.ts:157`), and `close()` rejects every pending request on that client with `codex app-server client is closed` (`extensions/codex/src/app-server/client.ts:256`, `extensions/codex/src/app-server/client.ts:461`). Active embedded runs are tracked per session and can coexist (`src/agents/pi-embedded-runner/runs.ts:417`).
- Impact: if two Discord/Codex sessions share the same app-server process, a timeout in session A can abruptly fail session B’s healthy in-flight `turn/start`, `turn/steer`, tool response, or completion wait. That is a user-visible regression from retiring only the poisoned turn to retiring the whole shared client without coordinating other active turns.
- Suggested fix: make run attempts use an isolated app-server client, or add shared-client active-turn coordination before closing. If the intended policy is “try bounded interrupt, then retire,” wait for the bounded interrupt to fail/timeout before closing the shared process, and explicitly abort/report any sibling runs that must be sacrificed.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes: start two concurrent `runCodexAppServerAttempt` calls with the default same-auth shared client, keep run B in flight, and let run A hit `params.timeoutMs`. Run A’s abort listener will clear/close the shared client, causing run B’s pending or next client request to reject.
- Is this the best way to solve the issue? Unclear. If Codex app-server is considered unsafe after any turn timeout, coordinated global abort/reporting may be the right behavior. If only the timed-out turn is suspect, bounded interrupt before shared-client retirement is safer.
