# Commit 02fe0d8

## Summary

Found two medium-confidence-to-high-confidence routing regressions in the runtime policy migration. Both are current on `main`.

## Findings

### Medium: Raw OpenAI model runs no longer force the PI path

- Kind: regression
- File: `src/agents/harness/selection.ts`
- Line: 77
- Evidence: `runAgentAttempt` still marks raw one-shot model runs with `requestedAgentHarnessId = "pi"` and passes it to `runEmbeddedPiAgent` (`src/agents/command/attempt-execution.ts:411`, `src/agents/command/attempt-execution.ts:606`). This commit removed the selector’s pinned policy path, so `selectAgentHarness` ignores `agentHarnessId` and resolves only provider/model policy. For official `openai`, that policy returns `codex` by default (`src/agents/harness/policy.ts:38`). A focused policy check returned `codex` for `openai/gpt-5.4`, and the existing selector test now explicitly proves `agentHarnessId: "pi"` is ignored (`src/agents/harness/selection.test.ts:343`).
- Impact: one-shot raw OpenAI model runs can route through the Codex harness instead of the raw embedded model path, requiring Codex/plugin auth and changing the behavior that `modelRun`/`promptMode: "none"` is trying to preserve.
- Suggested fix: keep stale session pins ignored, but introduce a separate explicit per-attempt force such as `requestedAgentHarnessId`/`forceAgentHarnessId` that `runEmbeddedPiAgent` and `selectAgentHarness` honor for raw runs.
- Confidence: high

### Medium: Doctor can rewrite legacy Codex refs onto a custom OpenAI endpoint without preserving Codex runtime

- Kind: regression
- File: `src/commands/doctor/shared/codex-route-warnings.ts`
- Line: 317
- Evidence: `maybeRepairCodexRoutes` rewrites `openai-codex/*` to `openai/*` without adding any provider/model-scoped `agentRuntime` (`src/commands/doctor/shared/codex-route-warnings.ts:317`, `src/commands/doctor/shared/codex-route-warnings.ts:570`). That is only behavior-preserving when `models.providers.openai.baseUrl` is the official OpenAI endpoint; the runtime policy returns `auto` for a custom OpenAI-compatible base URL. Focused checks showed repair produces `openai/gpt-5.5` with no runtime policy, and `resolveAgentHarnessPolicy` returns `auto` for that repaired model under a custom `openai` base URL.
- Impact: users with a custom `models.providers.openai.baseUrl` plus legacy `openai-codex/*` routes can have `doctor --fix` silently move those Codex runs to the custom OpenAI-compatible provider/PI path.
- Suggested fix: when canonicalizing legacy `openai-codex/*`, add model-scoped `agentRuntime: { id: "codex" }` if the current `openai` provider would not default to Codex, or skip the automatic rewrite with an explicit warning.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. For the first issue, run an OpenAI one-shot `modelRun`/`promptMode: "none"` path and observe that the selector ignores the `agentHarnessId: "pi"` force. For the second, run `maybeRepairCodexRoutes` on a config with `models.providers.openai.baseUrl = "https://example.test/v1"` and `agents.defaults.model = "openai-codex/gpt-5.5"`.
- Is this the best way to solve the issue? Yes for the first: separating explicit per-attempt force from stale session pins preserves the commit’s intent. For the second: preserving runtime only when OpenAI no longer defaults to Codex avoids reintroducing broad whole-agent pins.
