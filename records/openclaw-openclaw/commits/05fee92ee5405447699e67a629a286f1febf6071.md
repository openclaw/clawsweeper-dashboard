## Findings

### Low: Timer assertion now accepts the unarmed `null` state

- Kind: test_gap
- File: `src/cron/service/ops.test.ts`
- Line: 185
- Evidence: The commit changed the timer assertion from `expect(state.timer).toEqual(expect.anything())` to `expect(state.timer).toBeDefined()`. `CronServiceState.timer` is `NodeJS.Timeout | null`, and `armTimer` explicitly sets `state.timer = null` before arming. Local matcher check confirmed `expect(null).toBeDefined()` passes while `expect(null).toEqual(expect.anything())` fails.
- Impact: This test is named to prove startup “arms the timer”, but it would now pass if `start()` left `state.timer` as `null`. The same test still verifies `setTimeout` was called, but it no longer proves the service retained the timer handle, which matters for later `clearTimeout`, rearming, and duplicate timer prevention.
- Suggested fix: Restore the old assertion or use an explicit null check, for example `expect(state.timer).not.toBeNull()`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A local Vitest matcher probe showed `toBeDefined()` accepts `null`, and current source shows `null` is the service’s unarmed timer sentinel.
- Is this the best way to solve the issue? yes. The intended contract is simply “timer handle exists”, so restoring `expect.anything()` or asserting non-null is the narrow fix.
