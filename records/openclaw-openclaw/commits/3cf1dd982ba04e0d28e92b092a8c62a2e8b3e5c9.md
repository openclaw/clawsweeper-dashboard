# Commit 3cf1dd9

## Summary

Found one workspace-scoping regression in the new media availability gate.

## Findings

### Medium: `image` tool availability ignores workspace-scoped plugin metadata

- Kind: regression
- File: `src/agents/openclaw-tools.ts`
- Line: 106
- Evidence: `resolveImageToolFactoryAvailable()` calls `loadCapabilityMetadataSnapshot({ config })` without `workspaceDir`, while `createOpenClawTools()` has already resolved `workspaceDir` and passes it to the generation-tool planner at `src/agents/openclaw-tools.ts:346`. Workspace-scoped metadata snapshots are rejected when the caller omits `workspaceDir` (`src/plugins/current-plugin-metadata-snapshot.ts:79`), and workspace plugins are only discovered from `<workspace>/.openclaw/extensions` when a workspace is supplied (`src/plugins/roots.ts:21`).
- Impact: a workspace-installed plugin that declares `contracts.mediaUnderstandingProviders` can be ignored by the new `image` tool gate, so `createOpenClawTools()` may omit the `image` tool even when the session workspace has an available media-understanding provider and auth evidence. This is a regression from the prior behavior, which registered `image` whenever `agentDir` existed unless model/config checks disabled it.
- Suggested fix: add `workspaceDir?: string` to `resolveImageToolFactoryAvailable()`, pass the resolved `workspaceDir` from `createOpenClawTools()`, and thread it into `loadCapabilityMetadataSnapshot()`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Configure a workspace plugin under `.openclaw/extensions` with `contracts.mediaUnderstandingProviders`, call `createOpenClawTools()` with `workspaceDir`, `agentDir`, `modelHasVision: false`, and matching auth evidence; the image availability helper does not query that workspace metadata.
- Is this the best way to solve the issue? yes. The adjacent generation tool planner already passes `workspaceDir`; the image helper should use the same metadata scope.
