# Commit 9eb79bcf

## Summary

Found one regression: Mattermost’s deleted `src/config-schema.ts` shim was replaced only in the generated-metadata script, but the runtime bundled-channel config collector still does not look for `src/config-surface.*`.

## Findings

### Medium: Runtime bundled metadata no longer discovers Mattermost channel config

- Kind: regression
- File: `src/plugins/bundled-channel-config-metadata.ts`
- Line: 26
- Evidence: This commit deletes `extensions/mattermost/src/config-schema.ts`, leaving the schema at `extensions/mattermost/src/config-surface.ts:4`. The generator was updated to scan `config-surface.*` at `scripts/generate-bundled-channel-config-metadata.ts:80`, but the runtime collector’s `SOURCE_CONFIG_SCHEMA_CANDIDATES` still only includes `src/config-schema.*`, then falls back only to top-level `channel-config-api.*`. A focused probe confirmed current main returns no Mattermost channel config: `collectBundledChannelConfigs({ pluginDir: "extensions/mattermost", ... })` printed `{ "hasMattermost": false }`.
- Impact: Source/dev runtime manifest hydration and config-doc baseline flows that use `collectBundledChannelConfigs` lose Mattermost `channelConfigs` metadata. Mattermost’s `openclaw.plugin.json` declares `channels: ["mattermost"]` but has no inline `channelConfigs`, so setup/config surfaces that rely on manifest metadata before full runtime load can treat Mattermost as missing its config schema.
- Suggested fix: Add `src/config-surface.{ts,js,mts,mjs,cts,cjs}` to `SOURCE_CONFIG_SCHEMA_CANDIDATES` in `src/plugins/bundled-channel-config-metadata.ts`, or expose Mattermost through the existing top-level `channel-config-api.*` public surface. Add a regression test using `src/config-surface.js`, not only `src/config-schema.js`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Run the focused `node --import tsx -e ...collectBundledChannelConfigs...` probe against `extensions/mattermost`; it returns no `mattermost` entry.
- Is this the best way to solve the issue? yes. The changed generator and runtime collector are duplicate discovery paths, so aligning their candidate lists is the narrow fix.
