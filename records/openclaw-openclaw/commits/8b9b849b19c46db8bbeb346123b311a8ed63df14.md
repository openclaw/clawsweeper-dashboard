## Findings

### Low: write symlink-race test passes when the rebind never fires

- Kind: test_gap
- File: `src/infra/fs-safe.test.ts`
- Line: 37
- Evidence: `runWriteOpenRace` now accepts a resolved `params.runWrite()` and only asserts if an error is thrown. Its race trigger only flips on `fs.realpath` inputs ending in `slot/target.txt`, while `withRealpathSymlinkRebindRace` does not fail when its internal `flipped` flag remains false. I verified the current locked `@openclaw/fs-safe@0.1.1` write path with a focused temp-dir script: `write flipped=false outcome=resolved ... inside=new-content outside=SAFE`, while append and copy did flip and reject. The affected targeted tests still pass with `pnpm test src/infra/fs-safe.test.ts -t "symlink retarget races"`, so this no-op race is hidden.
- Impact: The test named `does not truncate out-of-root file when symlink retarget races write open` no longer proves a retarget race for `write`; it can pass as an ordinary in-root write. That weakens coverage around a security-sensitive fs boundary and could hide a future write traversal/clobber regression.
- Suggested fix: Make the race helper fail when `shouldFlip` never matches, or return/expose the flip status and assert it in this test. Then adjust the `write` trigger to an actual path the current write implementation realpaths, or add a test hook around the pinned write path.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused temp-dir probe using the same trigger showed `write flipped=false outcome=resolved`, and the changed Vitest target still passed.
- Is this the best way to solve the issue? yes. The core problem is that the harness no longer proves the race occurred; asserting the flip keeps the intended expectation explicit without requiring writes to always reject when a race is safely handled.
