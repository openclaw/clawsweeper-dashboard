# Commit c528f36

## Summary

Found one narrow regression in the Feishu wiki `move` path. The commit preserves pre-client validation for source `space_id`, but accidentally moved `target_space_id` validation after Lark client creation.

## Findings

### Low: `target_space_id` is validated after client creation in wiki move

- Kind: bug
- File: `extensions/feishu/src/wiki.ts`
- Line: 245
- Evidence: In the reviewed parent, `validateWikiSpaceIds()` validated both `space_id` and `target_space_id` before `createFeishuToolClient()`. In current `main`, the `move` case calls `moveNode(createClient(), spaceId, p.node_token, optionalWikiSpaceId(...))`; JavaScript evaluates arguments left-to-right, so `createClient()` runs before `optionalWikiSpaceId(p.target_space_id, "target_space_id")`.
- Impact: A numeric `target_space_id` is no longer rejected before Feishu/Lark client creation. With unresolved SecretRef credentials, the user can get a credential-resolution error instead of the intended precision-loss error. With valid credentials, the invalid request still does not reach the Lark API, but the fix’s “reject numeric wiki space IDs before Lark client creation” guarantee is incomplete for cross-space moves.
- Suggested fix: In the `move` case, compute `const targetSpaceId = optionalWikiSpaceId(p.target_space_id, "target_space_id");` before `createClient()`, then pass `targetSpaceId` to `moveNode()`. Add a focused regression test for numeric `target_space_id` asserting the Feishu client mock is not called and the error names `target_space_id`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes, source-level: call `feishu_wiki` with `action: "move"`, a quoted string `space_id`, and numeric `target_space_id`; the current argument order creates/resolves the client before validating `target_space_id`.
- Is this the best way to solve the issue? yes. It restores the parent commit’s validation order without changing Lark request behavior or account routing.
