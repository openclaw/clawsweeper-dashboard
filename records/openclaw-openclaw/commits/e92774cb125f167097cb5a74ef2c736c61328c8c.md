# Commit e92774c

## Summary

Found one upgrade-path gap: the fix suppresses duplicate announces for newly spawned inline ACP sessions, but existing persistent ACP thread sessions created before this commit can keep the old announce-enabled registry flag across follow-up turns.

## Findings

### Medium: Existing ACP thread sessions can keep duplicate announces after upgrade

- Kind: bug
- File: `src/agents/subagent-registry-run-manager.ts`
- Line: 334
- Evidence: The commit only changes new ACP spawn registration: `src/agents/tools/sessions-spawn-tool.ts:337` maps `result.inlineDelivery` to `expectsCompletionMessage: false`. Follow-up messages in an already-bound session do not call `sessions_spawn`; they use `reactivateCompletedSubagentSession`, which fetches the latest completed run and calls `replaceSubagentRunAfterSteer` with that existing record (`src/gateway/session-subagent-reactivation.ts:15`). The replacement record is built with `...source` and does not override `expectsCompletionMessage` (`src/agents/subagent-registry-run-manager.ts:334`). Pre-fix ACP session records were registered with the default `expectsCompletionMessage: true`, so those records keep the announce path enabled on follow-up turns.
- Impact: Users who update while a Discord ACP thread-bound session is active can still see the same duplicate response behavior for that existing thread until the old registry record is swept or the session is recreated. Active use can keep reactivating the same copied flag.
- Suggested fix: On reactivation, normalize legacy inline ACP session records to `expectsCompletionMessage: false` when the child session is an ACP session-mode thread-bound run, or migrate/repair persisted matching records on restore.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the code path: create an ACP thread-bound session on the parent commit so its persisted run has `expectsCompletionMessage: true`, update to this commit, then send a follow-up in that bound thread before the registry record is swept. I did not run live Discord/ACP.
- Is this the best way to solve the issue? Mostly yes for fresh sessions. The missing piece is a compatibility guard for legacy persisted session-mode ACP run records during reactivation.
