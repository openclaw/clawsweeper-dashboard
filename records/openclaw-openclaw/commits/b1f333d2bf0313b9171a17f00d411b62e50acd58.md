# Commit b1f333d

## Summary

Found one release-lane hardening bug: the new OpenAI installer E2E env vars are set in the lane command but are not forwarded through the Docker wrapper, so the container still uses the old defaults.

## Findings

### Medium: OpenAI installer hardening env vars are dropped before Docker

- Kind: reliability
- File: `scripts/lib/docker-e2e-scenarios.mjs`
- Line: 528
- Evidence: The lane command sets `OPENCLAW_INSTALL_E2E_OPENAI_MODEL=openai/gpt-5.4-mini` and `OPENCLAW_INSTALL_E2E_OPENAI_PROVIDER_TIMEOUT_SECONDS=300` before `pnpm test:install:e2e`. That package script runs `bash scripts/test-install-sh-e2e-docker.sh` (`package.json:1609`). The wrapper’s `docker run` allowlist forwards `OPENCLAW_INSTALL_E2E_AGENT_TURN_TIMEOUT_SECONDS` and `OPENCLAW_INSTALL_E2E_AGENT_TURNS_PARALLEL`, but not the two new OpenAI vars (`scripts/test-install-sh-e2e-docker.sh:21`). Inside the container, `run.sh` therefore falls back to `openai/gpt-5.5` and provider timeout equal to the outer agent-turn timeout (`scripts/docker/install-sh-e2e/run.sh:28`, `scripts/docker/install-sh-e2e/run.sh:29`).
- Impact: The release OpenAI installer proof lane silently does not use the intended cheaper/faster model or 300s provider timeout, so the commit’s hardening is ineffective and the lane can continue to hit the slow/hanging behavior it was meant to avoid.
- Suggested fix: Forward `OPENCLAW_INSTALL_E2E_OPENAI_MODEL` and `OPENCLAW_INSTALL_E2E_OPENAI_PROVIDER_TIMEOUT_SECONDS` in `scripts/test-install-sh-e2e-docker.sh`, and update the helper test to assert the wrapper forwards them, not only that the runner reads them.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. Static call path: scenario env prefix -> `pnpm test:install:e2e` -> Docker wrapper -> in-container `run.sh`; the wrapper does not pass the new variables.
- Is this the best way to solve the issue? Yes. The intended contract is already env-driven; the missing step is forwarding those env values across the Docker boundary.
