## Findings

### Low: Hot reload can make active group handling read identity data from a different WhatsApp auth directory

- Kind: regression
- File: `extensions/whatsapp/src/auto-reply/monitor/on-message.ts`
- Line: 229
- Evidence: The handler now resolves `account` from the current hot-reloaded config for each message, then passes `account.authDir` into `applyGroupGating`. Before this commit, group gating used the listener-startup `params.account.authDir`. The active Baileys socket and inbound normalization still belong to the startup auth directory, but `channels.whatsapp` changes are declared reload no-ops in `extensions/whatsapp/src/shared.ts:222`, so changing `channels.whatsapp.accounts.<id>.authDir` while the listener is running does not restart the socket. After such a reload, group mention resolution and reply-context identity resolution can read LID mappings from the new configured auth directory instead of the directory backing the active socket.
- Impact: Existing group messages that rely on auth-dir LID mapping, such as WhatsApp `@lid` mentions of the bot, can stop being recognized until restart after an `authDir` config edit. The same drift can affect reply-context identity normalization through the current config passed to `processMessage`.
- Suggested fix: Keep transport-bound connection facts from the listener snapshot. Use the current config for policy/routing fields that are safe to hot apply, but continue passing the startup listener `authDir` to identity resolution/group gating, or thread a separate immutable listener account/auth context into `processMessage`.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes, by source path: start a listener with auth dir A containing the LID mapping, hot-reload the account `authDir` to B without restarting, then send a group message whose mention is represented only as a LID. The current handler resolves mention identities against B. I did not run a live WhatsApp account.
- Is this the best way to solve the issue? yes. The safe boundary is to keep socket/auth-directory facts immutable for the lifetime of the active listener while still hot-applying `dmPolicy` and `allowFrom`.
