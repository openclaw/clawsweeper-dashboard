# Commit af8d2f2

## Summary

Found two regressions in persisted-boundary hardening. Both are still present on current `main`: no later commit changed the affected auth or cron files.

## Findings

### Medium: Legacy `auth.json` entries without embedded `provider` are now dropped

- Kind: regression
- File: `src/agents/auth-profiles/persisted.ts`
- Line: 130
- Evidence: `normalizeCommonCredentialFields()` now always emits `provider: ""` when `entry.provider` is missing. `parseCredentialEntry()` then uses `typed.provider ?? fallbackProvider`, so the empty string blocks the legacy fallback provider key and fails as `missing_provider` at `src/agents/auth-profiles/persisted.ts:656`. The legacy loader still calls `parseCredentialEntry(value, key)` at `src/agents/auth-profiles/persisted.ts:700`, so the fallback path is clearly intended.
- Impact: legacy provider-keyed `auth.json` entries such as `{ "openai": { "type": "api_key", "key": "..." } }` are ignored instead of migrated, leaving users with no auth profile from that legacy file.
- Suggested fix: preserve fallback semantics by omitting `provider` when absent/blank, or by resolving `const provider = normalizeProviderId(typed.provider ?? "") || fallbackProvider` before rejecting. Add a regression test for `loadLegacyAuthProfileStore()` with a provider-keyed entry that has no embedded provider.
- Confidence: high

### Medium: Legacy numeric-string `everyMs` cron jobs are skipped before hydration

- Kind: regression
- File: `src/cron/persisted-shape.ts`
- Line: 34
- Evidence: the new shape guard requires `schedule.everyMs` to already be a number, and `ensureLoaded()` skips the job on any invalid reason at `src/cron/service/store.ts:102`. But `normalizeCronJobInput()` preserves `everyMs: "60000"` for `kind: "every"` schedules, while the scheduler explicitly supports numeric strings through `coerceFiniteScheduleNumber()` and has a test named “handles string-typed everyMs and anchorMs from legacy persisted data” at `src/cron/schedule.test.ts:89`.
- Impact: persisted legacy `every` jobs with numeric-string intervals no longer load or run until `openclaw doctor --fix` is run, even though the runtime scheduling code can still execute them correctly.
- Suggested fix: coerce numeric-string `everyMs` during runtime normalization before the persisted-shape check, or allow the shape checker to accept numeric strings after validation. Add an `ensureLoaded()` regression test with `schedule: { kind: "every", everyMs: "60000" }`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused `node --import tsx` probe showed `loadLegacyAuthProfileStore()` returns `null` for provider-keyed legacy `auth.json` without embedded provider. A second probe showed `{ kind: "every", everyMs: "60000" }` normalizes unchanged, is classified as `invalid-schedule`, yet `computeNextRunAtMs()` returns a valid next timestamp.
- Is this the best way to solve the issue? yes for direction: both fixes should keep hardening at the persisted boundary while preserving documented/intentional legacy repair paths.
