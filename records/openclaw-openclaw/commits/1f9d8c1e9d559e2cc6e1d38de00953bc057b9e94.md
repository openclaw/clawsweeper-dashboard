# Commit 1f9d8c1

## Summary

Found one actionable regression in the source-checkout `qa coverage` launcher path. The new `--tools` command works through the Commander registration/runtime tests, but fails through `pnpm openclaw qa coverage --tools` when private QA dist is absent.

## Findings

### Medium: `qa coverage --tools` fails in source checkouts without private QA dist

- Kind: regression
- File: `scripts/qa-coverage-report.ts`
- Line: 45
- Evidence: `scripts/run-node.mjs:1189` routes `openclaw qa coverage` to `scripts/qa-coverage-report.ts` when private QA dist is missing. This commit adds `--tools` and `--summary` to `extensions/qa-lab/src/cli.ts:376`, but the source wrapper still only parses `--json`, `--output`, and `--repo-root`, then throws on any other flag at `scripts/qa-coverage-report.ts:45`. In the current no-dist checkout, `pnpm openclaw qa coverage --tools --json` fails with `Unknown qa coverage option: --tools`, while `pnpm openclaw qa coverage --json` still succeeds through the same source fallback.
- Impact: The newly wired tool coverage report is unavailable through the normal source-checkout CLI path that this repo intentionally supports for QA coverage without rebuilding private QA dist. Developers or CI jobs in that state cannot generate the tool coverage report.
- Suggested fix: Update `scripts/qa-coverage-report.ts` to parse and forward `--tools` and `--summary <path>` to `runQaCoverageReportCommand`, update its help text, and add a `src/infra/run-node.test.ts` case proving the source fallback forwards these flags.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: in a source checkout without `dist/plugin-sdk/qa-lab.js`, run `pnpm openclaw qa coverage --tools --json`; it exits 1 with `Unknown qa coverage option: --tools`.
- Is this the best way to solve the issue? yes: the Commander path already forwards the options correctly, so the source fallback shim needs the same option surface.
