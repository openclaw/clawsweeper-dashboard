# Commit bec83c5116

## Summary

Found one privacy regression in the new startup error message field: stability bundles now force built-in redaction but skip user-configured `logging.redactPatterns`, even though docs and adjacent safety-boundary code say custom patterns still apply to diagnostic/support surfaces.

## Findings

### Medium: Startup stability bundles can leak values covered only by custom redaction patterns

- Kind: privacy
- File: `src/logging/diagnostic-stability-bundle.ts`
- Line: 127
- Evidence: `readErrorMessage` now persists `error.message` after `redactSensitiveText(message, { mode: "tools" })`. Passing `{ mode: "tools" }` without `patterns` uses only built-in defaults, so configured `logging.redactPatterns` are ignored. The repo’s dedicated safety-boundary helper `redactToolPayloadText` explicitly merges user patterns with defaults, and docs say custom patterns still apply to support bundles/diagnostics surfaces even when `logging.redactSensitive` is off.
- Impact: Before this commit, raw startup error text was omitted from stability bundles. After this commit, a startup error such as `Startup failed for internal-ticket-AbC12345` is written verbatim if the operator relies on `logging.redactPatterns` to hide that token shape. These bundles are later included in diagnostics support exports, so users can unintentionally attach project-specific secrets or identifiers they configured OpenClaw to redact.
- Suggested fix: Redact bundle error messages with the same policy used for safety-boundary tool payloads, e.g. `redactToolPayloadText(message)`, or explicitly merge `getDefaultRedactPatterns()` with configured `logging.redactPatterns` while still forcing `mode: "tools"`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. With `OPENCLAW_CONFIG_PATH` pointing at a config containing `logging.redactSensitive: "off"` and `logging.redactPatterns: ["\\binternal-ticket-[A-Za-z0-9]+\\b"]`, a focused `node --import tsx` smoke wrote a failure stability bundle whose `error.message` still contained `internal-ticket-AbC12345`.
- Is this the best way to solve the issue? yes. The existing helper and docs already define the intended behavior for safety-boundary payload redaction, so reusing that policy is lower-risk than inventing bundle-specific redaction semantics.
