# Commit a77bd213ce

## Summary

Found one low-severity retry regression in the `Retry-After === maxDelayMs` boundary logic when callers pass a value that has already been clamped to the local max.

## Findings

### Low: Pre-clamped over-cap Retry-After values lose the symmetric fallback

- Kind: reliability
- File: `src/infra/retry.ts`
- Line: 162
- Evidence: This commit changes `canHonorRetryAfter` to `retryAfterMs <= maxDelayMs`, so equality always uses positive jitter. For a real equality boundary that preserves the server lower-bound contract, but `extensions/discord/src/api.ts:121` returns `Math.min(Math.max(0, err.retryAfter * 1000), retryConfig.maxDelayMs)`. A Discord server hint above the local cap therefore enters `retryAsync` as exactly `maxDelayMs`, not as the original over-cap value. The code then treats it as honor-able equality, applies positive jitter, and clamps the result back to exactly `maxDelayMs` at `src/infra/retry.ts:164`, synchronizing all clients. The existing raw over-cap test in `src/infra/retry.retry-after-lower-bound.test.ts:79` still documents that `retryAfterMs > maxDelayMs` should use symmetric jitter to avoid this herd shape.
- Impact: Discord API paths through `requestDiscord`/`fetchDiscord` can reintroduce the lockstep retry behavior this code is explicitly trying to avoid whenever Discord returns a large `retry_after` or the user configures a max below the server hint.
- Suggested fix: keep the reviewed `<=` behavior for true raw equality, but stop pre-clamping Discord’s `retryAfterMs` before it reaches `retryAsync`, or extend the retry contract so `retryAsync` can distinguish raw server hints from caller-capped hints. Add a regression test for a caller returning `Math.min(serverRetryAfter, maxDelayMs)`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused TSX probe using `retryAfterMs: () => Math.min(2, 1)`, `maxDelayMs: 1`, and `jitter: 1` produced `{"onRetryDelays":[1],"timeoutDelays":[1]}`, confirming the pre-clamped value is treated as exact-boundary positive jitter and then clamped to the cap.
- Is this the best way to solve the issue? yes. Preserving the raw server hint into `retryAsync` is safer than reverting the commit’s `<=` change, because true `retryAfterMs === maxDelayMs` still should not schedule below the server lower bound.
