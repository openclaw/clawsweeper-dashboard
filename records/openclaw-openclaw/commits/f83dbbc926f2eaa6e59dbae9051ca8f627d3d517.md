# Commit f83dbbc926

## Summary

Found two actionable regressions: the extension test type lane is broken by the changed test file, and the new exact-speech queue can stall Discord realtime voice when the selected realtime provider does not emit OpenAI-style `response.done` events.

## Findings

### Medium: changed Discord voice test file fails extension test typecheck

- Kind: regression
- File: `extensions/discord/src/voice/manager.e2e.test.ts`
- Line: 1199
- Evidence: `pnpm tsgo:extensions:test` fails with `TS2339: Property 'onEvent' does not exist on type ...` at lines 1199 and 1613. The added calls use `bridgeParams?.onEvent?.(...)`, but those local narrowed types only declare `onTranscript` or `onToolCall`/`onTranscript`.
- Impact: the extension test type gate fails on current `main`, so the changed surface cannot pass the expected changed-gate typecheck.
- Suggested fix: add `onEvent?: (event: { direction: "server"; type: string }) => void` to the affected local `bridgeParams` type annotations, then rerun `pnpm tsgo:extensions:test`.
- Confidence: high

### Medium: exact-speech queue can stall for realtime providers without completion events

- Kind: bug
- File: `extensions/discord/src/voice/realtime.ts`
- Line: 613
- Evidence: `enqueueExactSpeechMessage` queues later exact-speech answers whenever `exactSpeechResponseActive` or `hasInterruptibleOutputAudio()` is true. The queue only drains from `completeExactSpeechResponse`, which is reached via OpenAI-style `response.done`/`response.cancelled`, stream close, or Discord player idle. The bundled Google realtime voice bridge registers as a realtime voice provider but does not call `config.onEvent` at all, and its `handleServerContent` does not emit a turn-complete event when `content.turnComplete` arrives.
- Impact: with `channels.discord.voice.realtime.provider: "google"` in default `agent-proxy` mode, the first forced OpenClaw answer can set `exactSpeechResponseActive` and open an output stream, but subsequent forced answers are queued indefinitely because Discord never receives a provider completion signal to end the stream and drain the queue.
- Suggested fix: add a provider-neutral output completion signal and have Google emit it on Live `turnComplete`, or have Discord realtime use an explicit provider capability/fallback timeout for providers that cannot emit response completion. The fix should include a Discord test using a mock realtime provider that emits audio without `response.done`.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the typecheck regression: run `pnpm tsgo:extensions:test`. For the provider queue issue, the reproduction path is static but concrete: configure Discord realtime voice with provider `google`, let one forced exact-speech answer produce audio, then finish a second forced answer while the first output stream remains active.
- Is this the best way to solve the issue? Yes for the type annotation fix. For the provider queue issue, the safer fix is a contract-level provider completion signal rather than assuming all realtime voice providers emit OpenAI event names.
