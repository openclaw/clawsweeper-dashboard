# Commit f83dbbc926

## Summary

Found one provider-compatibility regression in the Discord realtime exact-speech queue. The OpenAI path is covered and passes, but configured non-OpenAI realtime voice providers can leave queued Discord voice answers stuck indefinitely.

## Findings

### Medium: Discord exact-speech queue waits forever for Google realtime voice

- Kind: compatibility
- File: `extensions/discord/src/voice/realtime.ts`
- Line: 442
- Evidence: Discord now only treats server events named `response.done` or `response.cancelled` as realtime response completion, and only then calls `finishOutputAudioStream()` to end the Discord `PassThrough` playback stream. Queued exact-speech answers drain only after `completeExactSpeechResponse()` and `hasInterruptibleOutputAudio()` becomes false. The Google realtime voice bridge is a registered backend realtime voice provider, emits audio and transcripts from `handleServerContent()`, but has no `config.onEvent` emission at all, so it never emits those OpenAI-style completion events.
- Impact: A Discord voice config using `voice.realtime.provider: "google"` in default `agent-proxy` mode can speak the first forced OpenClaw answer, then keep the output stream and `exactSpeechResponseActive` state alive. Later exact-speech answers enter `queuedExactSpeechMessages` and never dequeue, so the voice session appears to stop answering after the first response.
- Suggested fix: Normalize realtime response-completion at the provider/SDK boundary. For Google, emit a bridge event on `serverContent.turnComplete` or add a provider-agnostic output-complete callback, then have Discord end playback and complete exact speech from that normalized signal rather than only OpenAI event names.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes at the code-path level: configure Discord realtime voice with provider `google`, trigger one agent-proxy forced answer that emits audio, then trigger another forced answer. The second answer queues because `exactSpeechResponseActive` or output audio is still active, while Google never emits the completion event Discord requires to end the stream and drain the queue.
- Is this the best way to solve the issue? Yes. The bug is a provider contract mismatch, so fixing it by normalizing provider completion is safer than adding more provider-specific names in Discord.
