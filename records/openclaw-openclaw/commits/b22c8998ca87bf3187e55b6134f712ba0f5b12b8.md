# Commit b22c899

## Summary

Found one bug: the new load-path doctor contract discovery still misses load-path plugins when the persisted plugin registry exists and is considered fresh by the current policy hash.

## Findings

### Medium: Load-path doctor contracts are skipped behind a stale persisted registry

- Kind: bug
- File: `src/plugins/doctor-contract-registry.ts`
- Line: 268
- Evidence: `resolvePluginDoctorContracts` now passes `config` into `loadPluginManifestRegistryForPluginRegistry`, but that path can still reuse the persisted plugin registry. The persisted freshness check compares only `policyHash` in `src/plugins/plugin-registry-snapshot.ts:215`, and the policy hash in `src/plugins/installed-plugin-index-policy.ts:33` does not include `plugins.load.paths`. The new load-path tests use a fresh hermetic state with no persisted registry, so they do not cover this stale-persisted case.
- Impact: `openclaw doctor` / `openclaw doctor --fix` can still fail to discover a configured load-path plugin’s `doctor-contract-api` until the user manually refreshes or disables the persisted registry. That preserves the reported failure mode for setups where a registry already exists and a filesystem plugin is added through active config.
- Suggested fix: include normalized `plugins.load.paths` in persisted-registry freshness, or have doctor contract discovery force a derived manifest registry when active config contains load paths. Add a regression test that writes a persisted registry without the load-path plugin, then verifies active `plugins.load.paths` still discovers the sidecar.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. I created a temp doctor plugin with a `doctor-contract-api.cjs`, wrote a persisted registry from an otherwise identical config without `plugins.load.paths`, then called `listPluginDoctorLegacyConfigRules` with active config containing the load path. The normal persisted path returned `[]`; the same call with persisted registry disabled returned the expected legacy rule.
- Is this the best way to solve the issue? likely. The robust fix is to make registry freshness account for discovery roots, because other plugin-registry consumers can have the same blind spot.
