## Findings

### Low: Failed Telegram spool tombstones retain raw update payloads indefinitely

- Kind: privacy
- File: `extensions/telegram/src/telegram-ingress-spool.ts`
- Line: 333
- Evidence: `failTelegramSpooledUpdateClaim()` writes `.json.failed` tombstones that include `update: parsed.update`, and current code only uses tombstone filenames to suppress reprocessing. `listTelegramSpooledUpdates()` skips when `${entry}.failed` exists, and `writeTelegramSpooledUpdate()` checks the tombstone path, so the full raw update body is not needed for the runtime skip contract.
- Impact: timed-out Telegram updates can contain message text, sender/chat metadata, captions, or other user data. This commit changes those updates from temporary spool/processing files into terminal `.json.failed` files with no cleanup path found in current source, increasing local retention after a failure.
- Suggested fix: make failed tombstones minimal, for example `{ version, updateId, receivedAt, failure }`, or store only redacted diagnostic metadata. Keep the filename-based tombstone behavior so duplicate redelivery is still skipped.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Claim any spooled update containing message text, call `failTelegramSpooledUpdateClaim()`, and inspect the resulting `*.json.failed`; the code writes `parsed.update` into the tombstone.
- Is this the best way to solve the issue? yes. The runtime contract only needs a durable update-id tombstone, so removing the raw update from terminal failures preserves behavior while reducing retained user data.
