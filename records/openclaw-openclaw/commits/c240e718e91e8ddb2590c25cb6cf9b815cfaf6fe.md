# Commit c240e718e9

## Summary

Found two durable-delivery regressions that still exist on current `main`. The main risk is that a restart after the final assistant message is already written can discard the newly persisted pending final reply instead of delivering it.

## Findings

### Medium: Pending final reply is cleared as failed when the transcript tail is already an assistant reply

- Kind: reliability
- File: `src/agents/main-session-restart-recovery.ts`
- Line: 301
- Evidence: `recoverStore()` rejects any interrupted main session whose last meaningful transcript message is not `user`, `tool`, or `toolResult`; `markSessionFailed()` then clears `pendingFinalDelivery*` fields at lines 146-152. A no-repo-mutation probe with a `running`, `abortedLastRun`, `pendingFinalDelivery` session and a final assistant tail produced `recovered=0 failed=1` and removed the pending text from the store.
- Impact: This defeats the commit’s stated durable-delivery case when the model completed and persisted the assistant reply, but the process restarted before channel delivery. Recovery treats the already-completed assistant tail as non-resumable and deletes the only durable final-delivery payload, so the user can still lose the completed result.
- Suggested fix: If `pendingFinalDeliveryText` exists, prioritize direct replay/heartbeat delivery or mark the delivery attempt without applying the non-resumable-tail failure path. Do not clear pending final delivery merely because the transcript tail is an assistant message.
- Confidence: high

### Medium: Heartbeat replay delivers pending final text without clearing the durable marker

- Kind: reliability
- File: `src/infra/heartbeat-runner.ts`
- Line: 1849
- Evidence: `getReply()` returns `pendingFinalDeliveryText` for heartbeat runs and only updates attempt metadata (`src/auto-reply/reply/get-reply.ts:314`). The heartbeat runner then calls `deliverOutboundPayloads()` directly and only records `lastHeartbeatText` / `lastHeartbeatSentAt`; it does not clear `pendingFinalDelivery*`. The clear helper exists only in `dispatch-from-config.ts:330`, which this heartbeat path bypasses.
- Impact: After a successful heartbeat replay, the session remains permanently marked as pending final delivery. Later heartbeats keep re-entering the replay path, causing repeated work and duplicate suppression; once the 24-hour duplicate window expires or dedupe state changes, the old final reply can be sent again.
- Suggested fix: After successful heartbeat delivery of a pending-final replay, clear the same `pendingFinalDelivery*` fields that `dispatch-from-config` clears after final dispatch success.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. For the first finding, seed a temp session store with `status: "running"`, `abortedLastRun: true`, `pendingFinalDelivery: true`, `pendingFinalDeliveryText`, and a transcript ending in an assistant message, then call `recoverRestartAbortedMainSessions()`. The pending fields are cleared and the session is marked failed.
- Is this the best way to solve the issue? likely. Both fixes should preserve the durable marker until a concrete delivery success or explicit retry exhaustion, rather than routing pending-final state through generic interrupted-run resume/failure behavior.
