# Commit 39bcd1e088

## Summary

Found one concrete bypass in the new installed dependency runtime scan: dependency entrypoints that are valid Node package entrypoints but have no scanned extension are not scanned, so dangerous runtime code can still pass install validation.

## Findings

### Medium: Installed dependency scan misses extensionless Node package entrypoints

- Kind: security
- File: `src/security/skill-scanner.ts`
- Line: 42
- Evidence: `scanDirectoryWithSummary()` only collects files whose extension is in `SCANNABLE_EXTENSIONS` (`.js`, `.ts`, `.mjs`, `.cjs`, etc.), and the new post-install path at `src/plugins/install-security-scan.runtime.ts:1199` scans dependency package directories without adding package-manifest entrypoints as forced files. Node’s package contract allows `"main"` to define the package entrypoint as a path, and Node documents extensionless package subpaths as a traditional supported style. ([nodejs.org](https://nodejs.org/api/packages.html))
- Impact: A dependency installed under `node_modules` can put executable CommonJS in an extensionless `main` target, for example `package.json` with `"main": "payload"` and `payload` containing `child_process.execSync(...)`. A focused smoke check showed `installPluginFromInstalledPackageDir()` returned `ok: true` with no warnings for that fixture, even though the plugin loaded it with `require("extensionless-helper")`.
- Suggested fix: When scanning installed package dependency roots, parse each dependency package manifest and force-scan runtime entrypoints from `main`, `exports`, `module`/common legacy fields if supported locally, and `bin`, even when the target has no scannable extension. Resolve symlinks and keep the existing install-root boundary checks.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Create an installed package plugin with `dependencies: { "extensionless-helper": "1.0.0" }`, `index.js` requiring that dependency, and `node_modules/extensionless-helper/package.json` containing `"main": "payload"` where `payload` has `child_process.execSync("node -v")`; `installPluginFromInstalledPackageDir()` succeeds.
- Is this the best way to solve the issue? yes. The scanner should keep the broad extension filter for normal traversal, but installed package validation needs manifest-derived runtime entrypoints as explicit includes because Node can execute files that extension-based walking intentionally ignores.
