# Commit 1235f7f

## Summary

Found one regression in the new metadata reuse fast path. It can reuse a current plugin manifest registry after plugin discovery inputs change, because the fallback compares only the installed-plugin policy hash and bypasses the existing control-plane fingerprint check.

## Findings

### Medium: Auto-enable can reuse a stale manifest registry after plugin load paths or roots change

- Kind: regression
- File: `src/config/plugin-auto-enable.shared.ts`
- Line: 958
- Evidence: The first lookup calls `getCurrentPluginMetadataSnapshot({ config, env, allowWorkspaceScopedSnapshot: true })`, which checks the full control-plane fingerprint. That fingerprint includes discovery inputs such as `plugins.load.paths`, workspace roots, and env-resolved plugin roots via `src/plugins/plugin-control-plane-context.ts:41`. The new fallback immediately calls `getCurrentPluginMetadataSnapshot({ env, allowWorkspaceScopedSnapshot: true })` without `config`, then accepts the snapshot when only `snapshot.policyHash === resolveInstalledPluginIndexPolicyHash(params.config)` at `src/config/plugin-auto-enable.shared.ts:962`. The policy hash in `src/plugins/installed-plugin-index-policy.ts:19` includes allow/deny/slots/enabled state, but not `plugins.load.paths` or env-resolved roots.
- Impact: In a long-lived process with a current metadata snapshot, `applyPluginAutoEnable` can make auto-enable decisions from an old manifest registry after the requested config points at different plugin load paths. That can fail to auto-enable a newly configured path plugin under a restrictive allowlist, or preserve/allowlist a plugin id from a removed path. Gateway channel/status/send surfaces call `applyPluginAutoEnable` against runtime config without passing an explicit manifest registry in paths such as `src/gateway/server.impl.ts:797`, so the stale registry can affect runtime channel/plugin resolution helpers.
- Suggested fix: Do not use the no-config snapshot fallback here unless the requested config’s full control-plane fingerprint is known compatible. Prefer passing `compatibleConfigs` when storing the snapshot, or call `getCurrentPluginMetadataSnapshot` with `config` and rely on its existing compatible fingerprint/policy handling. At minimum, guard this fallback the way other callers do by refusing it when `normalizePluginsConfig(config.plugins).loadPaths.length > 0`, but that still would not cover env/root changes as completely as the fingerprint check.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused smoke in the current checkout set a current snapshot for config A with `plugins.load.paths: ["/plugins/one"]`, then resolved auto-enable metadata for config B with the same policy hash but `plugins.load.paths: ["/plugins/two"]`. `resolvePluginAutoEnableManifestRegistry` returned the stale registry entry `stale:/plugins/one/stale`.
- Is this the best way to solve the issue? yes, the safest fix direction is to keep the existing full compatibility check authoritative. Policy-only equality is too weak for manifest registry reuse because plugin discovery metadata is intentionally part of the control-plane fingerprint.
