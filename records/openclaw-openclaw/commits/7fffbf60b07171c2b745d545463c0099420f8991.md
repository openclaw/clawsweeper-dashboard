# Commit 7fffbf60

## Summary

The commit substantially hardens public package URL downloads and the focused resolver tests pass. I found one remaining security/supply-chain regression around the new trusted package token being exposed to non-`trusted-url` package resolution paths.

## Findings

### Medium: Trusted package token is inherited by `source=ref` dependency lifecycle scripts

- Kind: security
- File: `.github/workflows/package-acceptance.yml`
- Line: 372
- Evidence: The workflow injects `OPENCLAW_TRUSTED_PACKAGE_TOKEN` into the entire “Resolve package candidate” step unconditionally, regardless of `inputs.source`. For `source=ref`, `scripts/resolve-openclaw-package-candidate.mjs` calls `installPackageSourceDeps()` and runs `pnpm install --ignore-scripts=false --config.enable-pre-post-scripts=true` in the selected package worktree. The script’s `run()` helper does not set `env`, so child processes inherit the resolver process environment. `update-migration.yml:36-46` calls this reusable workflow with `source: ref` and `secrets: inherit`, which makes the new secret available to that path when configured.
- Impact: A package ref install lifecycle script, or a compromised dependency lifecycle script during that install, can read and exfiltrate the trusted private mirror bearer token even though `source=ref` does not need that token. This weakens the intended `source=trusted-url` credential boundary introduced by the commit.
- Suggested fix: Scope `OPENCLAW_TRUSTED_PACKAGE_TOKEN` to `source=trusted-url` only in the workflow, and defensively scrub that variable from generic spawned commands in `scripts/resolve-openclaw-package-candidate.mjs` so only the HTTPS request header construction path can use it.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Source-level reproduction is direct: configure `OPENCLAW_TRUSTED_PACKAGE_TOKEN`, run Package Acceptance through `source=ref` such as `update-migration.yml`, and any package/dependency lifecycle script spawned by `pnpm install` can read the variable. I also verified locally with a sentinel value that Node child processes inherit environment variables when `spawn()` is called without an explicit `env`.
- Is this the best way to solve the issue? yes. The token is only needed for `source=trusted-url` request headers, so scoping plus child-process env scrubbing preserves the new trusted mirror feature without widening credential exposure to unrelated package resolution modes.
