# Commit 49db190

## Summary

Found one actionable regression risk in the QMD collection conflict repair path. The reviewed commit avoids unsafe rebinding, but the verification it now requires is not available from the currently published QMD CLI list output, so a real stale path-pattern conflict can remain unrepaired and block OpenClaw’s managed QMD collection creation.

## Findings

### Medium: Current QMD stderr-only path conflicts can no longer auto-repair

- Kind: regression
- File: `extensions/memory-core/src/memory/qmd-manager.ts`
- Line: 675
- Evidence: Parent `e422bcfc2aec374ce971f4c49b9d18217a4c2879` used the parsed QMD add-conflict name when `collection list` could not identify a path-pattern match. This commit now only accepts that parsed name when `existing.get(parsedConflictName)` also has `path` and `pattern` metadata. With current `@tobilu/qmd@2.1.0`, `qmd collection list --json` still prints human table output and includes `Pattern`, `Files`, and `Updated`, but not the collection path. A focused live check also showed a second `collection add` for the same path and pattern exits with `A collection already exists for this path and pattern` and the stale `Name`, while `collection list --json` does not provide the path OpenClaw now requires for verification. The current `main` follow-up `c39f85822a` adds a manual repair warning, but still returns `false` for this case.
- Impact: If a user has a stale differently named QMD collection occupying the same path and pattern, OpenClaw will skip adding the canonical managed collection on startup instead of rebinding it. Memory search for that managed collection can stay unavailable until manual `qmd collection remove <old-name>` repair.
- Suggested fix: Verify against QMD’s YAML/config source that contains both `path` and `pattern`, or add/use a QMD list/API mode that exposes path metadata, before refusing the parsed add-conflict name. If the intended behavior is manual repair only, update docs/tests to make that explicit and remove the unreachable-looking parsed-name verification branch.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. With `@tobilu/qmd@2.1.0`, create a collection for a temporary workspace path, then try adding a second collection with another name and the same `--mask`. QMD reports the conflicting collection name from the add error, but `collection list --json` does not expose path metadata, matching the reviewed test’s refused-rebind scenario.
- Is this the best way to solve the issue? Unclear. The commit’s safety goal is valid, but the current verification source is insufficient for the published QMD CLI. Reading QMD config/YAML or getting structured list output would preserve safety without making the repair path unavailable.
