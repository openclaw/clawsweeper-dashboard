# Commit ea5f2abb

## Summary

Found two actionable issues in the new read-target authorization guards. Slack read checks ignore the channel-name matching mode they opt into, which can bypass disabled name-based channel entries. Discord read checks treat non-guild DM channels as guild allowlist misses once any guild config exists, breaking DM read/reaction workflows.

## Findings

### Medium: Slack read guard ignores name-matched channel policy entries

- Kind: security
- File: `extensions/slack/src/action-runtime.ts`
- Line: 173
- Evidence: `assertSlackReadTargetAllowed` calls `resolveSlackChannelConfig` with `channelId` and `allowNameMatching`, but never supplies `channelName`. Existing Slack ingress paths pass both `channelId` and `channelName` for the same matcher, and the matcher only considers name keys when a channel name is present. A focused source probe showed `resolveSlackChannelConfig({ channelId: "C1", channelName: "ops-room", channels: { "ops-room": { enabled: false } }, allowNameMatching: true })` resolves the disabled entry, while the same call without `channelName` does not. A runtime probe with `groupPolicy: "open"`, `dangerouslyAllowNameMatching: true`, and `"ops-room": { enabled: false }` allowed `readMessages` for `C1` and called `readSlackMessages`.
- Impact: Operators using the documented break-glass name matching mode can block inbound messages for a channel by name, but an agent can still read that channel by ID through read-like actions because the new guard treats it as an unlisted open-policy channel. The same omission also falsely rejects name-allowed channels under allowlist policy.
- Suggested fix: Resolve/carry the Slack channel name before action-time read authorization, then pass it into `resolveSlackChannelConfig`; alternatively fail closed for configured name-only entries if action-time name resolution is unavailable.
- Confidence: high

### Medium: Discord DM read targets are rejected when guild entries exist

- Kind: compatibility
- File: `extensions/discord/src/actions/runtime.messaging.shared.ts`
- Line: 335
- Evidence: `assertReadTargetAllowed` only returns early for open policy when there are no guild entries. If a target channel has no `guildId`, it falls through to checking whether that channel ID is explicitly listed under some guild. Discord DM channels have no guild ID, so with any configured guild entry they are rejected even when Discord DMs are enabled separately by `dmPolicy`. A focused runtime probe with `dmPolicy: "open"`, `allowFrom: ["*"]`, `groupPolicy: "open"`, one configured guild, and `fetchChannelInfoDiscord` returning `{ type: 1 }` rejected a `reactions` read with `Discord read target channel is not allowed.` Docs also describe DM policy and message-tool reactions resolving `user:<id>` DM targets independently from guild policy.
- Impact: Discord users with both guild configuration and enabled DMs can no longer use read-like actions against DM targets, including reaction reads resolved from `user:<id>`, even though DM access is governed by `dmPolicy`/`allowFrom`.
- Suggested fix: Treat Discord non-guild channel types separately from guild channel allowlists. Authorize explicit/current DM reads through the DM policy path or preserve enough target context to verify the DM participant before allowing the read.
- Confidence: high
