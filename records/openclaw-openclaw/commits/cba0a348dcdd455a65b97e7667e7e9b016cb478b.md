# Commit cba0a348

## Summary

Found one regression outside the stated Windows `.env` hardening path: the new prepared Pi settings wrapper flattens trusted project settings into global in-memory settings, which can drop project-local Pi resources that must resolve relative to `<workspace>/.pi`.

## Findings

### Medium: Trusted prepared Pi settings flatten project scope and drop project-local resources

- Kind: regression
- File: `src/agents/pi-project-settings.ts`
- Line: 41
- Evidence: `createRuntimeEmbeddedPiSettingsManager()` builds a merged snapshot from `getGlobalSettings()` and `getProjectSettings()`, then passes it to `SettingsManager.inMemory()`. The upstream Pi `SettingsManager.inMemory()` treats that object as global settings; `getProjectSettings()` becomes `{}`. Pi resource resolution uses project settings for project-scope resources relative to `<cwd>/.pi`, while global settings resolve relative to `agentDir`. A focused temp-dir smoke with trusted project policy and `.pi/settings.json` containing `{"skills":["custom/project-skill"]}` produced `global { skills: ["custom/project-skill"] }`, `project {}`, and loaded no skill. The pre-change path using the file-backed manager preserved `project { skills: ["custom/project-skill"] }` and loaded the project skill.
- Impact: trusted-policy workspaces can silently lose project-local Pi skills, prompts, extensions, themes, or package entries, especially explicit project paths outside auto-discovery directories. That changes prior trusted workspace behavior and can make agent runs miss project-provided capabilities.
- Suggested fix: preserve global and project scopes when creating the runtime in-memory manager. Use Pi storage APIs such as a scoped in-memory storage adapter, or otherwise initialize separate global/project settings before applying runtime-only compaction overrides. Add a regression test that loads an explicit project-local resource path under `projectSettingsPolicy: "trusted"`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Create a temp workspace with `.pi/settings.json` containing an explicit project skill path such as `custom/project-skill`, add `.pi/custom/project-skill/SKILL.md`, construct `createPreparedEmbeddedPiSettingsManager()` with trusted project settings, then call Pi `DefaultResourceLoader.reload()`. Current code puts the skill path in global settings, leaves project settings empty, and loads no skill.
- Is this the best way to solve the issue? yes, preserving Pi’s global/project distinction is the right fix direction. Sanitizing and runtime overrides are still useful, but resource-bearing project settings must remain project-scoped.
