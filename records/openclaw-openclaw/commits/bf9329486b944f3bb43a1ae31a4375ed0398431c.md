# Commit bf93294

## Summary

One display regression remains in the Telegram picker callback path. The commit made `formatModelsAvailableHeader` depend on `agentId` to compute the effective OpenAI/Codex auth-provider order, but the Telegram caller still omits the session agent id.

## Findings

### Low: Telegram model picker headers ignore agent-scoped runtime policy

- Kind: regression
- File: `extensions/telegram/src/bot-handlers.runtime.ts`
- Line: 2351
- Evidence: `formatModelsAvailableHeader` now resolves `resolveAgentHarnessPolicy({ provider, agentId })` before auth labeling in `src/auto-reply/reply/commands-models.ts:396`, but this Telegram callback passes only `provider`, `cfg`, `agentDir`, and `sessionEntry`. A focused check with a default `main` agent using implicit Codex and a `worker` agent pinning `openai/*` to Pi produced different labels: without `agentId` it printed `oauth (openai-codex:user@example.com)`, while with `agentId: "worker"` it printed `api-key (openai:default)`.
- Impact: In Telegram sessions routed to a non-default agent with agent-scoped OpenAI runtime policy, the picker can show the default agent’s Codex auth label instead of the session agent’s effective auth path. This reintroduces the same kind of misleading auth/cost-control header the commit is meant to fix, but only for that Telegram subagent path.
- Suggested fix: Pass `agentId: sessionState.agentId` into the `formatModelsAvailableHeader` call at `extensions/telegram/src/bot-handlers.runtime.ts:2351`, and add a focused Telegram callback/header test for a non-default agent with `openai/*` runtime policy.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The focused `node --import tsx --eval` check exercised the real `formatModelsAvailableHeader` path with and without `agentId` and showed the header label changing for the same agent auth store/config.
- Is this the best way to solve the issue? yes. The text command path already threads `agentId`; the Telegram callback has `sessionState.agentId` available next to the existing `resolveAgentDir(cfg, sessionState.agentId)` call.
