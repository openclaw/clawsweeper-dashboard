# Commit be21d64d08

## Summary

Found one regression in proxied HTTPS canvas host URL generation.

## Findings

### Medium: HTTPS proxy canvas URLs are forced back to HTTP

- Kind: regression
- File: `src/gateway/server.impl.ts`
- Line: 1332
- Evidence: `server.impl.ts:1332` now sets `canvasHostScheme = gatewayTls.enabled ? "https" : "http"` and passes it into the WS handler. `src/gateway/server/ws-connection.ts:252` then forwards that as `scheme` to `resolveCanvasHostUrl`. But `src/infra/canvas-host-url.ts:63` only uses `x-forwarded-proto` when `params.scheme` is undefined, and its comment at `src/infra/canvas-host-url.ts:77` explicitly covers HTTPS proxy/Tailscale Serve behavior. The focused check showed the regression: forwarded HTTPS alone resolves to `https://gateway.example.com:443`, but adding `scheme: "http"` resolves to `http://gateway.example.com:80`.
- Impact: Gateways behind Tailscale Serve or another HTTPS reverse proxy, with Gateway TLS disabled internally, can advertise insecure canvas links on port 80. HTTPS Control UI clients may hit the wrong URL or block canvas content as mixed content.
- Suggested fix: only override the scheme for direct Gateway TLS, e.g. pass `canvasHostScheme: gatewayTls.enabled ? "https" : undefined`, or otherwise let `x-forwarded-proto` win when present. Add a WS/server-path test for `x-forwarded-proto: https` with `gatewayTls.enabled === false`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The helper-level reproduction is `resolveCanvasHostUrl({ canvasPort: 18789, requestHost: "gateway.example.com", forwardedProto: "https" })` returning `https://gateway.example.com:443`, while the new forced `scheme: "http"` path returns `http://gateway.example.com:80`.
- Is this the best way to solve the issue? yes. Direct TLS only needs an explicit HTTPS override when the Gateway itself owns TLS; otherwise the existing forwarded-proto contract should remain authoritative.
