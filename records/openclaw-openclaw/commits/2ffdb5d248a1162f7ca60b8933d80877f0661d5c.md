# Commit 2ffdb5d

## Summary

Found regressions in the archive transcript visibility/indexing path. The main issue is that the QMD session backend still cannot resolve reset/deleted archive hits after this change, so those hits are filtered out before reaching `memory_search`.

## Findings

### Medium: QMD reset/deleted archive hits are still dropped by visibility filtering

- Kind: regression
- File: `src/plugin-sdk/session-transcript-hit.ts`
- Line: 43
- Evidence: QMD exports session files as Markdown with `const targetName = \`${path.basename(sessionFile, ".jsonl")}.md\`` in `extensions/memory-core/src/memory/qmd-manager.ts:2219`. For an archive like `deleted-stem.jsonl.deleted.2026-02-16T22-27-33.000Z`, that becomes `deleted-stem.jsonl.deleted.2026-02-16T22-27-33.000Z.md`. `extractTranscriptIdentityFromSessionsMemoryHit` calls `parseUsageCountedSessionIdFromFileName(base)` before stripping `.md`, so the archive parser fails; the `.md` branch then returns stem `deleted-stem.jsonl.deleted.2026-02-16T22-27-33.000Z` with `archived: false`. With the live store entry gone, `resolveTranscriptStemToSessionKeys` returns no keys and `filterMemorySearchHitsBySessionVisibility` drops the hit at `extensions/memory-core/src/session-search-visibility.ts:56`.
- Impact: the commit’s main fix does not work for QMD-backed memory session search. Reset/deleted archive content can be indexed and returned by QMD, then discarded by the new visibility filter, so `memory_search corpus=sessions` still cannot recover the archived transcript.
- Suggested fix: make archive parsing handle QMD `.md` archive exports before the `.md` stem branch, and carry enough owner metadata for deleted archives, either from the QMD session collection/agent id or by exporting archives under an agent-scoped path.
- Confidence: high

### Medium: Dirty-only sync can prune all legacy session rows after the path-format migration

- Kind: regression
- File: `packages/memory-host-sdk/src/host/session-files.ts`
- Line: 319
- Evidence: `sessionPathForFile` now changes existing indexed paths from `sessions/<file>` to `sessions/<agent>/<file>`. In the built-in memory sync, `activePaths` is built with the new path format from all files, but dirty-only mode skips every non-dirty file at `extensions/memory-core/src/memory/manager-sync-ops.ts:846`; after that, it deletes every existing row whose old path is not in `activePaths` at `extensions/memory-core/src/memory/manager-sync-ops.ts:902`. A focused check of `resolveMemorySessionSyncPlan` with existing rows `sessions/a.jsonl`, `sessions/b.jsonl`, files now resolving to `sessions/main/a.jsonl`, `sessions/main/b.jsonl`, and only `a.jsonl` dirty marked both old rows stale while `indexAll` was false.
- Impact: after upgrading, the first session-delta sync can delete previously indexed, unchanged session transcripts from the memory DB without reindexing them under the new path. Those sessions stay missing from `memory_search` until a later forced/full reindex.
- Suggested fix: force a full session reindex when the session path scheme changes, include a path-schema version in memory index metadata, or make the stale-prune step understand legacy `sessions/<basename>` aliases until a full reindex has migrated them.
- Confidence: high

### Low: Literal user `[cron:...]` messages can make archived user sessions opaque

- Kind: bug
- File: `packages/memory-host-sdk/src/host/session-files.ts`
- Line: 604
- Evidence: for reset/deleted archives, any user message whose normalized text matches `DIRECT_CRON_PROMPT_RE` sets `generatedByCronRun = true`, clears already collected content, and suppresses the rest of the transcript. The nearby comment at `packages/memory-host-sdk/src/host/session-files.ts:617` already notes that user-typed text can match `[cron:...]`.
- Impact: a normal user session archived after reset/delete can lose all searchable transcript content if the user’s text starts with a literal cron-looking prefix, for example while asking about cron syntax or quoting an internal message.
- Suggested fix: do not classify an entire archived transcript as cron-generated from the prompt text alone. Prefer durable `sessionKey`/metadata evidence, or only drop the generated prompt line while preserving later user-visible assistant content when provenance is ambiguous.
- Confidence: medium
