## Summary

Found one still-actionable native tool policy gap. The follow-up on current `main` fixed restricted `toolsAllow` turns, but `disableTools: true` still leaves Codex app-server native code mode enabled.

## Findings

### Medium: `disableTools` runs still enable Codex native code mode

- Kind: security
- File: `extensions/codex/src/app-server/run-attempt.ts`
- Line: 3136
- Evidence: `shouldEnableCodexAppServerNativeToolSurface()` only evaluates the forced `toolsAllow` value; when `params.disableTools === true` and `toolsAllow` is unset, it returns `true`. A direct probe on current `main` printed `true` for that input. `buildDynamicTools()` does honor `disableTools` by returning no OpenClaw dynamic tools, but `startOrResumeThread()` then receives `nativeCodeModeEnabled: true`, and `buildThreadStartParams()` emits `{"features.code_mode":true,"features.code_mode_only":true}`.
- Impact: Callers use `disableTools` for raw/model-only runs (`src/agents/command/attempt-execution.ts:684`). With the Codex app-server harness, those runs lose OpenClaw dynamic tools but still expose Codex’s native shell/file code-mode surface, bypassing the intended no-tools contract.
- Suggested fix: Treat `params.disableTools === true` as `nativeToolSurfaceEnabled === false` in the new gate, and keep the current follow-up behavior that starts a transient restricted thread, disables user MCP servers, clears environments, and sends the deny-all app patch.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The focused probes showed `shouldEnableCodexAppServerNativeToolSurface({ disableTools: true, toolsAllow: undefined })` returns `true`, and that native-enabled thread start config turns on `features.code_mode` and `features.code_mode_only`.
- Is this the best way to solve the issue? yes. The existing restricted-native-tools path already has the right downstream behavior; the missing input is `disableTools`.
