# Commit 00a44b08

## Summary

Found one regression in the new active-run fallback: it treats any active abort-controller entry with the same run id as an active `agent` retry, including `chat.send` entries.

## Findings

### Medium: Active chat.send run can suppress a distinct agent request with the same idempotency key

- Kind: regression
- File: `src/gateway/server-methods/agent.ts`
- Line: 1287
- Evidence: The new fallback returns `{ status: "in_flight" }` whenever `registerChatAbortController` did not register and `context.chatAbortControllers.has(runId)` is true. It does not verify that the existing entry is `kind === "agent"`. The shared entry contract says absent `kind` is treated as `"chat-send"` (`src/gateway/chat-abort.ts:15`), and the maintenance change itself uses the stricter namespace check for dedupe preservation (`src/gateway/server-maintenance.ts:96`). The updated test now demonstrates the problematic case by inserting a pre-existing `"chat-send-session"` entry with no `kind`, then asserting the agent command is not called (`src/gateway/server-methods/agent.test.ts:3333`).
- Impact: `agent` and `chat.send` dedupe records are namespaced as `agent:<id>` and `chat:<id>`, but this new run-id guard is global. If a client legitimately reuses an idempotency key across the two RPC methods while a `chat.send` run is active, the gateway reports the agent request as cached/in-flight and never dispatches it. The requested agent turn is effectively dropped behind an unrelated chat run.
- Suggested fix: Only use this fallback for an existing active agent entry, e.g. read `const existing = context.chatAbortControllers.get(runId)` and return cached `in_flight` only when `existing?.kind === "agent"`. Keep the old behavior for `chat-send`/undefined entries: avoid overwriting the entry, but still dispatch the agent run with its own cleanup guard.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The current committed test constructs the failing shape directly: a `chatAbortControllers` entry with no `kind` and the same idempotency key as an `agent` RPC now causes `mocks.agentCommand` not to be called.
- Is this the best way to solve the issue? yes. The commit’s goal is to preserve active agent dedupe retries; checking `kind === "agent"` matches the existing ownership contract and the new maintenance cleanup’s own namespace-specific logic.
