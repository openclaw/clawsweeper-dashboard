## Findings

### Low: Fallback test now asserts a non-production OpenRouter candidate

- Kind: test_gap
- File: `src/agents/model-fallback.test.ts`
- Line: 958
- Evidence: The new top-level mock makes `normalizeProviderModelIdWithManifest` always return `undefined` at `src/agents/model-fallback.test.ts:30`, and the OpenRouter fallback case now expects `{ provider: "openrouter", model: "deepseek-chat" }` at `src/agents/model-fallback.test.ts:958`. In current production code, `normalizeStaticProviderModelId` calls the manifest normalizer by default (`src/agents/model-ref-shared.ts:27`), and OpenRouter’s manifest has `prefixWhenBare: "openrouter"` (`extensions/openrouter/openclaw.plugin.json:8`). A focused live check of `__testing.resolveFallbackCandidates` without the test mock returned `{"provider":"openrouter","model":"openrouter/deepseek-chat"}` for this same config.
- Impact: A future regression that disables manifest model-id normalization in the fallback resolver could keep this test green while OpenRouter fallbacks are sent as bare model ids. That is a concrete risk because OpenRouter provider/model refs intentionally keep the upstream provider prefix inside the model id.
- Suggested fix: Keep the hot fallback unit test isolated, but either use a provider/model fixture not affected by manifest normalization in this ordering test, or add a focused unmocked composition test proving `resolveFallbackCandidates` still applies manifest normalization for OpenRouter fallbacks.
- Confidence: high

- Is this the best way to solve the issue? unclear. The intent to keep the hot fallback test from loading plugin metadata is reasonable; the safer fix is to avoid asserting OpenRouter’s production-normalized model id inside the mocked fixture, while preserving one small production-composition proof elsewhere.
