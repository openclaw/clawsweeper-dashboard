# Commit 997edf6

## Summary

Found one incomplete-fix bug: disabled realtime input interruption is honored for the OpenAI GA session shape, but not for Azure deployment realtime sessions.

## Findings

### Medium: Azure realtime sessions still omit `interrupt_response: false`

- Kind: bug
- File: `extensions/openai/realtime-voice-provider.ts`
- Line: 672
- Evidence: The GA payload maps `interruptResponseOnInputAudio` into `audio.input.turn_detection.interrupt_response` at `extensions/openai/realtime-voice-provider.ts:622`, but the Azure deployment/beta payload at `extensions/openai/realtime-voice-provider.ts:672` only sends `create_response` and omits `interrupt_response`. The touched tests assert the disabled-interruption setting only for the GA shape; the Azure deployment test at `extensions/openai/realtime-voice-provider.test.ts:187` checks only format compatibility. OpenAI’s Realtime API reference documents `interrupt_response` as the setting for automatic interruption, and Azure’s realtime audio reference documents the same field with default `true`. ([platform.openai.com](https://platform.openai.com/docs/api-reference/realtime-client-events/session?utm_source=openai))
- Impact: Users configured with `azureEndpoint`, `azureDeployment`, and `interruptResponseOnInputAudio: false` still send an Azure session update that leaves server-side response interruption at its default. The local `speech_started` handler no longer clears playback, but the Azure realtime service can still interrupt the active response on VAD start, so Discord audio may continue to cut off despite the new setting.
- Suggested fix: Include `interrupt_response: cfg.interruptResponseOnInputAudio ?? cfg.autoRespondToAudio ?? true` in `resolveBetaSessionUpdatePayload().turn_detection`, if the targeted Azure API version accepts it. If older Azure deployment APIs reject the field, gate by API version/capability and document that disabled server interruption is unsupported on that path. Add a focused Azure deployment test with `interruptResponseOnInputAudio: false`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Create an OpenAI realtime bridge with `azureEndpoint`, `azureDeployment`, `autoRespondToAudio: true`, and `interruptResponseOnInputAudio: false`; the emitted `session.update.session.turn_detection` omits `interrupt_response`.
- Is this the best way to solve the issue? yes, unless the legacy Azure deployment endpoint rejects the field. The safest fix is to send the same boolean contract in the Azure payload, with an API-version guard only if live compatibility requires it.
