# Commit f3c9203

## Summary

Found one replay regression in the fix. The streamed reply no longer persists `[object Object]`, but the new synthetic `thinkingSignature: "content"` collides with the OpenAI-compatible replay converter’s use of `thinkingSignature` as an outbound message field name.

## Findings

### Medium: Structured thinking replay overwrites assistant answer content

- Kind: regression
- File: `src/agents/openai-transport-stream.ts`
- Line: 1646
- Evidence: `getCompletionsContentDeltas()` emits structured thinking as `{ kind: "thinking", signature: "content", text }`, which persists as `thinkingSignature: "content"`. On the next same-model OpenAI-completions request, the dependency converter preserves signed thinking blocks and assigns `assistantMsg[signature] = thinkingText`; with `signature === "content"`, this overwrites the visible assistant answer. A focused payload check using the committed block shape produced replay messages of `[{"role":"assistant","content":"Need to think."},{"role":"user","content":"follow up"}]`, dropping `Visible answer.`.
- Impact: Follow-up turns after a Mistral structured-thinking response can send hidden thinking as the prior assistant message content and omit the actual visible answer. That corrupts conversation context and can expose reasoning text back to the provider as normal assistant content.
- Suggested fix: Do not store `"content"` as a replay signature. For structured `delta.content` thinking blocks, either omit `thinkingSignature` or use a parser/replay path that reconstructs provider-supported structured chunks without colliding with `content`. Add a regression test for `buildOpenAICompletionsParams()`/`convertMessages()` with a persisted thinking block plus visible text.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Persist an assistant message like the new test expects, with `{ type: "thinking", thinking: "Need to think.", thinkingSignature: "content" }` followed by `{ type: "text", text: "Visible answer." }`, then build OpenAI-completions params for a follow-up user turn. The serialized assistant message content becomes `Need to think.` instead of `Visible answer.`.
- Is this the best way to solve the issue? yes, directionally. The bug is the synthetic replay signature, not the structured-content extraction itself; avoiding a reserved field-name signature is the narrowest fix.
