# Commit 4b8c600f99

## Summary

Found one low-severity test regression in the rewritten assertion. The production behavior currently still passes, but the new test would miss a concrete invalid shape for the external CLI provider scope.

## Findings

### Low: Provider scope assertion now accepts a plain string

- Kind: test_gap
- File: `src/gateway/server-methods/models-auth-status.test.ts`
- Line: 239
- Evidence: The commit replaced `expect.arrayContaining(["opencode-go"])` with `expect(externalCli.providerIds).toContain("opencode-go")`. Vitest `toContain` passes for substring checks on strings; a focused matcher probe showed `"opencode-go"` passes the new assertion but fails the previous `arrayContaining` assertion. `ExternalCliAuthDiscovery.providerIds` is typed as `Iterable<string>` in `src/agents/auth-profiles/external-cli-discovery.ts`, so a plain string is type-compatible but would be consumed downstream as characters rather than provider IDs.
- Impact: A future regression that returns `providerIds: "opencode-go"` instead of `["opencode-go"]` would still pass this test while breaking external CLI auth overlay scoping for the provider.
- Suggested fix: Restore an array-shape assertion, for example `expect(externalCli.providerIds).toEqual(expect.arrayContaining(["opencode-go"]))`, or assert `Array.isArray(externalCli.providerIds)` before `toContain`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A local Vitest matcher probe showed `expect("opencode-go").toContain("opencode-go")` passes while `expect("opencode-go").toEqual(expect.arrayContaining(["opencode-go"]))` fails.
- Is this the best way to solve the issue? yes. The test is specifically protecting the scoped provider list passed to `ensureAuthProfileStore`, so preserving the array/collection shape check is the narrow fix.
