# Commit 99a6b1c

## Summary

Found two actionable issues. The main one is introduced by this commit: the isolated Codex ACP config now inherits `sandbox_mode`, which is execution policy rather than safe model/provider routing. The commit also marks #80079 fixed while a source-level failure path from that issue still remains.

## Findings

### Medium: Isolated Codex ACP config inherits host sandbox policy

- Kind: security
- File: `extensions/acpx/src/codex-trust-config.ts`
- Line: 166
- Evidence: `sandbox_mode` is included in `INHERITED_TOP_LEVEL_CODEX_CONFIG_KEYS`, and `renderIsolatedCodexConfig()` copies inherited top-level keys into the generated isolated `CODEX_HOME`. The generated wrapper then launches Codex ACP with `CODEX_HOME` pointing at that generated config. The added test also asserts `sandbox_mode = "workspace-write"` is copied.
- Impact: A host Codex config such as `sandbox_mode = "danger-full-access"` now silently carries into OpenClaw-managed ACP child sessions. That contradicts the docs/PR wording that only trusted project entries plus safe model/provider routing are copied, and it can broaden filesystem/process permissions without an explicit OpenClaw ACP policy decision.
- Suggested fix: Remove `sandbox_mode` from the inherited whitelist unless there is a separate, explicit ACP permission contract that validates and documents this inheritance.
- Confidence: high

### Medium: #80079 is closed while the output-without-done failure path remains

- Kind: bug
- File: `src/acp/control-plane/manager.core.ts`
- Line: 949
- Evidence: The linked issue reports that ACP turn completion should tolerate adapters that close the stream after output without a terminal `done` event. Current `consumeAcpTurnStream()` tracks `sawOutput`, but `runTurn` still unconditionally throws `ACP turn ended without a terminal done event` when `sawTerminalEvent` is false. This commit’s changelog says `Fixes #80079`, but this reported path is still present on current `main`.
- Impact: ACP sessions can still be marked failed after producing useful output, which can keep bound thread delivery/task completion from succeeding for the same issue cluster.
- Suggested fix: Decide the intended contract for output-bearing stream closure, then either treat `sawOutput && !sawTerminalEvent` as a successful completion for affected ACP adapters or keep #80079 open/narrow the changelog claim.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Partially. The sandbox finding is static and directly covered by the new code/test. The #80079 finding is source-level; I did not run a live Discord/Codex ACP flow.
- Is this the best way to solve the issue? For sandbox inheritance, yes: remove it from the safe routing whitelist unless explicitly approved. For output-without-done, unclear: the runtime contract needs a maintainer decision because accepting output-only closure may mask real adapter crashes.
