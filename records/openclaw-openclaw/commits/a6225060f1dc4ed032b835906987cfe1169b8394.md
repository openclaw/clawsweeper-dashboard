# Commit a622506

## Summary

The embedding timeout/abort plumbing looks sound in the reviewed paths, but the commit also added a CLI config-guard bypass that can apply to parsed `config set` invocations when `--section` appears before the subcommand.

## Findings

### Medium: `config --section ... set` can skip config bootstrap

- Kind: regression
- File: `src/cli/program/preaction.ts`
- Line: 127
- Evidence: `resolveCliArgvInvocation(["node","openclaw","config","--section","auth","set","gateway.auth.mode","none"]).commandPath` returns `["config","auth"]`. The new `isGuidedConfigCommandPath` treats that as guided config, while Commander parses the same argv as the `config set` action. Plain `config set` is expected to run `ensureConfigReady` in `src/cli/program/preaction.test.ts`.
- Impact: mutating config commands can skip `ensureCliExecutionBootstrap`, bypassing config readiness/invalid-config handling that `config set` normally receives.
- Suggested fix: remove the argv-derived guided-config bypass, or gate it on the parsed `actionCommand` being the parent `config` command rather than a subcommand.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: the focused Node probes above show raw argv path classification diverges from Commander’s parsed action for `openclaw config --section auth set gateway.auth.mode none`.
- Is this the best way to solve the issue? yes: the existing parsed-action check already covers bare guided `config --section ...`; subcommands should not use a raw positional heuristic.
