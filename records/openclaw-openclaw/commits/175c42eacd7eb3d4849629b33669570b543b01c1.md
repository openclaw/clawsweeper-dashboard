# Commit 175c42e

## Summary

Found two regressions in the sub-agent completion side of the commit. The Telegram select callback change looks covered and reasonable, but the commit also removes two guards added by the parent commit.

## Findings

### Medium: Sub-agent completions bypass message-tool-only group/channel delivery

- Kind: privacy
- File: `src/agents/subagent-announce-delivery.ts`
- Line: 59
- Evidence: `AGENT_MEDIATED_COMPLETION_TOOLS` now contains only `music_generate` and `video_generate`, so `requiresAgentMediatedCompletionDelivery()` is false for `sourceTool: "subagent_announce"`. That makes `shouldDeliverAgentFinal` stay true and calls `agent` with `deliver: true` even when `messages.groupChat.visibleReplies: "message_tool"`.
- Impact: Group/channel rooms default to message-tool-only visible replies. Sub-agent completion summaries can now be automatically posted to Slack/Discord/Telegram-style rooms even though the configured policy says normal final replies should stay private unless the agent calls `message(action=send)`.
- Suggested fix: Restore `"subagent_announce"` to the mediated completion tool set, or otherwise enforce `completionRequiresMessageToolDelivery()` for `expectsCompletionMessage` sub-agent announcements in group/channel routes.
- Confidence: high

### Medium: Child completion output is injected into prompts without a size bound

- Kind: reliability
- File: `src/agents/internal-events.ts`
- Line: 44
- Evidence: `formatChildResultDataBlock()` now passes `event.result` directly to `wrapPromptDataBlock()` with no `maxChars`. The parent commit capped child results at 4,000 chars and had a regression test; this commit removes both. A probe with a 6,000-character child result produced a 6,496-character internal prompt block with no truncation marker.
- Impact: Large sub-agent outputs can inflate or overflow the requester-agent completion prompt, increasing cost and making completion delivery fail or degrade on long logs/tool output.
- Suggested fix: Reinstate a bounded child-result prompt cap, ideally via `wrapPromptDataBlock({ maxChars: ... })` plus a truncation marker.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. For the delivery issue, a direct probe of `deliverSubagentAnnouncement()` with `messages.groupChat.visibleReplies: "message_tool"` and `sourceTool: "subagent_announce"` returned `delivered: true` and called gateway `agent` with `deliver: true`, `channel: "slack"`, `to: "channel:C123"`.
- Is this the best way to solve the issue? yes. The safest fix is to restore the removed guard for `subagent_announce`; media completions still use the same mechanism.
