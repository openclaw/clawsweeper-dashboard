# Commit 175c42e

## Summary

Found two regressions outside the Telegram callback parser fix: subagent completions can bypass group/channel `message_tool` visibility policy, and child completion prompt context is no longer bounded.

## Findings

### Medium: Subagent completions bypass tool-only group/channel delivery

- Kind: privacy
- File: `src/agents/subagent-announce-delivery.ts`
- Line: 62
- Evidence: The commit removes `"subagent_announce"` from `AGENT_MEDIATED_COMPLETION_TOOLS`, so `requiresMessageToolDelivery` is false for normal subagent completions. That makes `shouldDeliverAgentFinal` true at `src/agents/subagent-announce-delivery.ts:738`, causing the gateway `agent` request to use `deliver: true` even when `messages.groupChat.visibleReplies` is `"message_tool"`. A focused probe with a Slack channel requester, `sourceTool: "subagent_announce"`, and `messages.groupChat.visibleReplies: "message_tool"` returned `delivered: true` and called the gateway with `deliver: true`, `channel: "slack"`, and `to: "channel:C123"`.
- Impact: Group/channel rooms are documented to keep normal final replies private unless the agent uses `message(action=send)`. This regression can auto-post requester-agent final text for background subagent completions into a room whose config requires message-tool delivery.
- Suggested fix: Restore `"subagent_announce"` to the mediated-completion set, and restore/add the removed regression test that expects `deliver: false` plus message-tool evidence for group/channel subagent completions.
- Confidence: high

### Medium: Large child results are injected into prompts without a cap

- Kind: reliability
- File: `src/agents/internal-events.ts`
- Line: 44
- Evidence: The commit removes `MAX_CHILD_RESULT_PROMPT_CHARS` and `truncateChildResultForPrompt`, then passes the raw `event.result` to `wrapPromptDataBlock` without `maxChars`. The removed test had asserted a 6,000-character child result was capped below 5,000 characters. Current focused probe of `formatAgentInternalEventsForPrompt` with `result: "x".repeat(6000)` returned `{"length":6526,"hasTruncation":false}`.
- Impact: A large child/subagent result can balloon the parent completion prompt, increasing token cost and risking context overflow or model request failure. The gateway schema accepts `internalEvents[].result` as an unbounded string, and subagent output collection does not impose a cap before this formatter.
- Suggested fix: Restore a child-result prompt cap, either with the previous truncation helper or by passing an explicit `maxChars` to `wrapPromptDataBlock`, and keep the truncation marker test.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The delivery regression reproduces with `deliverSubagentAnnouncement` using a Slack channel origin, `sourceTool: "subagent_announce"`, and `messages.groupChat.visibleReplies: "message_tool"`; the gateway params show `deliver: true`. The prompt-size regression reproduces by calling `formatAgentInternalEventsForPrompt` with a 6,000-character child result.
- Is this the best way to solve the issue? yes for the delivery regression: the commit removed the exact classification needed for the existing gate. For the prompt-size regression, the safest fix is to restore the previous bounded formatter or use the existing `wrapPromptDataBlock` `maxChars` option.
