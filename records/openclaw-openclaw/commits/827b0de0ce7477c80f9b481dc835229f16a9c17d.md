## Findings

### Low: Chutes OAuth rejects IPv6 loopback redirect URIs

- Kind: compatibility
- File: `extensions/chutes/oauth.ts`
- Line: 42
- Evidence: The commit replaced the old Chutes OAuth local callback path, which used `isLoopbackHost(hostname)`, with an exact hostname allowlist: `localhost`, `127.0.0.1`, and `::1`. In Node, `new URL("http://[::1]:1456/oauth-callback").hostname` returns `[::1]`, not `::1`, so the new check rejects a valid loopback redirect. A focused smoke on current `main` fails with: `Chutes OAuth redirect hostname must be loopback (got [::1])...`; `isLoopbackHost("[::1]")` still returns `true`.
- Impact: Users configuring `CHUTES_OAUTH_REDIRECT_URI=http://[::1]:...` cannot use the local Chutes OAuth callback and are forced into failure/manual fallback even though the address is loopback-safe. The old implementation accepted this host.
- Suggested fix: Use the SDK loopback helper, for example `isLoopbackHost(url.hostname)` from an allowed plugin SDK subpath, or normalize bracketed IPv6 hostnames before the allowlist check.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: call `loginChutes({ app: { redirectUri: "http://[::1]:1456/oauth-callback", ... }, manual: false, ... })` from `extensions/chutes/oauth.ts`; it rejects before opening auth.
- Is this the best way to solve the issue? yes: reuse the existing loopback host contract instead of maintaining a narrower ad hoc hostname check.
