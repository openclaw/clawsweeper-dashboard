# Commit 6c07de0

## Summary

The commit intended to clear the pairing allowlist cache before installing the read spy, but it also stopped passing the computed `filePath` into the spy factory. That breaks both test typechecking and the targeted runtime test.

## Findings

### Medium: cache invalidation test no longer passes its file path to the read spy

- Kind: regression
- File: `src/pairing/pairing-store.test.ts`
- Line: 189
- Evidence: `withAllowFromCacheReadSpy` still declares `createReadSpy: (filePath: string) => FileReadSpy` and computes `filePath`, but now calls `params.createReadSpy()` with no argument. The variant callbacks close over that argument and count reads with `countFileReads(spy, filePath)`, so they count reads to `undefined`, not the actual allowFrom file.
- Impact: `pnpm tsgo:core:test` fails with `TS2554: Expected 1 arguments, but got 0`, and `pnpm test:serial src/pairing/pairing-store.test.ts` fails because `readCount()` is `0` instead of `1`. This leaves current `main` with a broken touched test/type lane.
- Suggested fix: keep the new cache clear, but restore the argument: `const readSpy = params.createReadSpy(filePath);`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: `pnpm tsgo:core:test --pretty false --incremental false --tsBuildInfoFile /tmp/openclaw-tsgo-core-test-review.tsbuildinfo` reports `src/pairing/pairing-store.test.ts(189,26): error TS2554`, and `OPENCLAW_VITEST_FS_MODULE_CACHE_PATH=/tmp/openclaw-vitest-cache-pairing-review pnpm test:serial src/pairing/pairing-store.test.ts` fails the cache invalidation test.
- Is this the best way to solve the issue? yes: the commit’s intended cache clear is fine, but the spy factory still needs the same `filePath` used by `countFileReads`.
