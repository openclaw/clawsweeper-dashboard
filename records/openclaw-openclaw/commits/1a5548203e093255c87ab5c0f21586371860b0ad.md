# Commit 1a55482

## Summary

Found one low-severity progress-draft regression in the new keyed merge helper: keyed item lines that only change display-affecting fields such as `status` are treated as unchanged because the merge identity compares only `text`.

## Findings

### Low: Keyed progress lines ignore status-only updates

- Kind: bug
- File: `src/plugin-sdk/channel-streaming.ts`
- Line: 827
- Evidence: `buildChannelProgressDraftLine` stores `status` separately on non-tool item lines, and `formatChannelProgressDraftText` renders that status as visible text. But `mergeChannelProgressDraftLine` returns the existing line whenever the same keyed line has the same normalized `.text`, so a later event with the same `itemId` and text but a changed status is discarded. I confirmed with a focused TS probe: merging a `Context compaction` item from `running` to `completed` still renders `• Context compaction: running`.
- Impact: progress drafts can leave keyed item statuses stale until the draft is replaced, for example Codex `contextCompaction` start/end item events emitted from `extensions/codex/src/app-server/event-projector.ts` carry the same title and item id but different statuses.
- Suggested fix: when an `id` match is found, compare the full display-affecting line shape or always replace the existing object unless it is actually equivalent, rather than using `.text` alone.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: construct two keyed `item` progress lines with the same `itemId` and title but statuses `running` then `completed`, merge them, and format the result; the output remains `running`.
- Is this the best way to solve the issue? yes: keyed replacement should preserve the in-place update behavior added by this commit, but equality should account for fields used by rendering.
