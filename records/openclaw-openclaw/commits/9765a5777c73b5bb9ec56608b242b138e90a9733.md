# Commit 9765a57

## Summary

Found one regression in the update-doctor path: deferred channel plugin installs can be treated as a completed one-time release migration, causing the config to be stamped so the automatic install pass will not run later.

## Findings

### Medium: Update-deferred channel plugin installs can permanently mark the release repair complete

- Kind: regression
- File: `src/commands/doctor/shared/missing-configured-plugin-install.ts`
- Line: 547
- Evidence: During `OPENCLAW_UPDATE_IN_PROGRESS`, the new code adds every `collectUpdateDeferredPluginIds` result to `deferredPluginIds` before checking whether there is an install record. For a channel-selected plugin with no persisted install record, line 549 continues without a change or warning, then lines 597-620 exclude that plugin from `missingPluginIds` and block it from `collectDownloadableInstallCandidates`. The caller `maybeRunConfiguredPluginInstallReleaseStep` treats `warnings.length === 0` as completed and returns `touchedConfig: true` in `src/commands/doctor/shared/release-configured-plugin-installs.ts:357`, and `runReleaseConfiguredPluginInstallsHealth` stamps `meta.lastTouchedVersion` in `src/flows/doctor-health-contributions.ts:296`. Git and package update flows set both `OPENCLAW_UPDATE_IN_PROGRESS=1` and `OPENCLAW_UPDATE_PARENT_SUPPORTS_DOCTOR_CONFIG_WRITE=1`, so that stamp is persisted during update doctor.
- Impact: A user updating from a pre-2026.5.2 config with a configured externalized channel plugin can skip the automatic install, get `meta.lastTouchedVersion` advanced, and never get the one-time release install pass again. `plugins update --all` does not cover missing plugins without install records because it selects `Object.keys(installs)`.
- Suggested fix: Do not report a deferred no-record install as a successful release repair. Either make the release step return `touchedConfig: false` when update mode defers configured install candidates, or make the helper surface a warning/deferred status that prevents the one-time release stamp until a non-update doctor can actually install the plugin.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: use an old config with `channels.discord` configured, no `plugins.installs.discord`, and `meta.lastTouchedVersion` before `2026.5.2-beta.1`; run update doctor with `OPENCLAW_UPDATE_IN_PROGRESS=1` and config-write support enabled. The helper skips installation without warning, and the release health step can stamp `meta.lastTouchedVersion`.
- Is this the best way to solve the issue? yes, the safer direction is to keep deferral during package updates but prevent the one-time release migration from being marked complete until the install actually runs or explicitly fails.
