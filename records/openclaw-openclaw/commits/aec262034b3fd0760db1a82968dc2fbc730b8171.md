# Commit aec2620

## Summary

Found one upgrade-path regression in the Telegram reply-chain cache persistence format. Fresh JSONL cache files work and the targeted test passes, but an existing pre-commit JSON-array cache file can poison post-upgrade persistence across restarts.

## Findings

### Low: Existing JSON-array cache files can block loading new appended records after upgrade

- Kind: regression
- File: `extensions/telegram/src/message-cache.ts`
- Line: 169
- Evidence: Before this commit, `persistMessages` wrote the cache as a single JSON array with no trailing newline. The new reader treats the file as newline-delimited JSON objects, and `appendPersistedMessage` appends a JSONL record directly to the existing file. With an old file, the first post-upgrade append produces `<old-json-array><new-json-object>\n`; on the next restart, `JSON.parse(line)` throws on that first line and the outer `try/catch` exits the whole replay loop, so later valid JSONL lines are ignored too.
- Impact: Telegram reply-chain context persisted before the upgrade is lost, and more importantly newly appended reply-chain records may also fail to reload across gateway restarts until a single long-running process reaches compaction. With the default `maxMessages` of 5000, that can mean up to 10001 records before the poisoned file is rewritten.
- Suggested fix: Either read and migrate the old JSON-array snapshot shape before switching to JSONL, or make append insert a newline when appending to a non-empty file that does not end in `\n`. Also consider handling per-line parse failures inside the loop so one malformed line cannot suppress later valid records.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Start from a pre-commit cache file containing `JSON.stringify([{ key, node }])`, append one new JSONL record without inserting a newline, then restart and replay with the new loop in `readPersistedMessages`; the first line is invalid JSON and replay stops before later records.
- Is this the best way to solve the issue? yes. Supporting the old snapshot shape or normalizing the file boundary before append preserves current behavior without giving up the append performance win.
