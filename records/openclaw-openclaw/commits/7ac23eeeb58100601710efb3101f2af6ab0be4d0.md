# Commit 7ac23eee

## Summary

Found one compatibility regression risk: third-party startup-only plugins that have not added `activation.onStartup: true` now stop loading at Gateway startup and no longer get a compatibility/status warning.

## Findings

### Medium: Startup-only plugins without `activation.onStartup` are now silently skipped

- Kind: regression
- File: `src/plugins/gateway-startup-plugin-ids.ts`
- Line: 94
- Evidence: Current startup planning only admits explicit `manifest.activation.onStartup === true` or memory plugins in `shouldConsiderForGatewayStartup`; the installed-index builder now sets `startup.sidecar` only from explicit `onStartup: true` and no longer emits the previous `legacy-implicit-startup-sidecar` compat code. Status warnings were also narrowed to `legacy-before-agent-start` and `hook-only`, so affected plugins no longer get a migration notice. Docs now state that omitted `onStartup` no longer startup-loads plugins, but base behavior explicitly loaded no-channel/no-runtime-surface plugins and warned authors.
- Impact: Existing external or workspace plugins whose `register()` starts hooks/services at Gateway boot, and whose manifest predates `activation.onStartup`, will be enabled but absent from `startupPluginIds`; their startup side effects will not run. This is especially easy to miss because `plugins status` no longer reports the removed compatibility notice.
- Suggested fix: If the behavior change is intended now, keep a diagnostic/status/doctor notice for manifests that look like legacy startup sidecars but omit `activation.onStartup`, so operators know why startup behavior changed. If this is not intended to break external plugins yet, keep the fallback until the documented removal window and only remove the env gate later.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. A focused `pnpm exec tsx` smoke calling `resolveGatewayStartupPluginIdsFromRegistry` with an explicitly enabled global plugin, no channels/providers/CLI backends, and no `activation.onStartup` returned `[]`; the same fixture with `activation.onStartup: true` returned `["startup-sidecar"]`.
- Is this the best way to solve the issue? Unclear. For a deliberate breaking change, a retained diagnostic is the narrowest fix. For compatibility preservation, restoring the fallback until the prior removal date is safer.
