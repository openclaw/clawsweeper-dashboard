# Commit 314903417f

## Summary

Found one remaining boundary-case leak in the new dreaming-fence guard: managed fence marker lines themselves are not treated as overlapping the managed block, so they can still be promoted into `MEMORY.md`.

## Findings

### Low: Dreaming fence marker-only ranges still bypass the promotion fence

- Kind: bug
- File: `extensions/memory-core/src/short-term-promotion.ts`
- Line: 1520
- Evidence: `lineRangeOverlapsDreamingFence` sets `insideFence = true` for a start marker and `insideFence = false` for an end marker, then `continue`s before checking whether that marker line is inside the candidate range. A focused probe against the exported test helper returned `false` for `2-2` on `<!-- openclaw:dreaming:light:start -->` and `false` for `4-4` on `<!-- openclaw:dreaming:light:end -->`, while returning `true` for the body line and full range.
- Impact: A recalled/ranked candidate whose live source range is exactly a managed dreaming marker is not rejected by the new rehydration guard and is also not caught by `isContaminatedDreamingSnippet`. I reproduced this through `recordShortTermRecalls -> rankShortTermPromotionCandidates -> applyShortTermPromotions` with default promotion thresholds after 10 recall signals; `MEMORY.md` appended `- <!-- openclaw:dreaming:light:start --> [...]`.
- Suggested fix: Treat start and end marker lines as overlapping when `oneIndexed` is within `safeStart..safeEnd`, or build explicit managed-block intervals that include both boundary markers. Add marker-only tests for start and end markers plus an apply-path regression.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The temp-workspace probe wrote a daily note containing a light-sleep fence, recorded 10 recalls for the start marker line, ranked candidates with default thresholds, and applied promotions. Result: `applied 1 appended 1`, with the start marker written into `MEMORY.md`.
- Is this the best way to solve the issue? yes. The intended guard is already line-range based; including boundary marker lines is the narrowest correction and matches the managed-block contract used by `dreaming-markdown.ts`.
