# Commit b258c3f

## Summary

Found two actionable issues: the service-plan fix does not cover exec SecretRefs stored in `auth-profiles.json`, and the Docker compose change silently raises the minimum supported Docker Compose version without an updated preflight or docs.

## Findings

### Medium: auth-profile exec SecretRefs still lose their resolver environment in managed service plans

- Kind: bug
- File: `src/commands/daemon-install-helpers.ts`
- Line: 183
- Evidence: The new passEnv collector only iterates `discoverConfigSecretTargets(params.config)` at `src/commands/daemon-install-helpers.ts:183`. Auth-profile refs are loaded separately, but `collectAuthProfileServiceEnvVars` only preserves refs where `ref.source === "env"` and skips everything else at `src/commands/daemon-install-helpers.ts:90`. Runtime does resolve `auth-profiles.json` `keyRef`/`tokenRef` assignments (`src/secrets/runtime-auth-collectors.ts:53`, `src/secrets/runtime-auth-collectors.ts:92`), and exec providers build their child env exclusively from `providerConfig.passEnv` keys present in the runtime env (`src/secrets/resolve.ts:697`).
- Impact: A user with an auth profile such as `keyRef: { source: "exec", provider: "onepassword", id: "..." }` and `secrets.providers.onepassword.passEnv: ["OP_CONNECT_TOKEN"]` can install or repair the Gateway service from a shell where `OP_CONNECT_TOKEN` exists, but the installed service environment will omit `OP_CONNECT_TOKEN`. On restart, the Gateway’s runtime SecretRef snapshot cannot resolve that auth profile, breaking provider/model auth even though the commit claims managed service plans preserve exec SecretRef `passEnv` keys.
- Suggested fix: Extend the new passEnv collection to inspect loaded auth-profile `keyRef` and `tokenRef` exec refs, using the same provider lookup and host-env filtering as config refs. Add a focused `buildGatewayInstallPlan` test where `authStore.profiles.*.keyRef` or `tokenRef` uses an exec provider with `passEnv`.
- Confidence: high

### Low: docker-compose.yml now requires Docker Compose 2.24+ but docs still allow any v2

- Kind: compatibility
- File: `docker-compose.yml`
- Line: 5
- Evidence: The commit adds object-form `env_file` entries with `required: false` for both services (`docker-compose.yml:5` and `docker-compose.yml:77`). Official Docker docs state that `env_file.required` requires Docker Compose 2.24.0 or later, while this repo’s Docker prerequisites still say only “Docker Compose v2” at `docs/install/docker.md:19`. 
- Impact: Users on older Compose v2 releases can fail before container startup just parsing the root compose file, including manual `docker compose up` workflows documented by the repo. The setup script only checks that `docker compose version` works; it does not enforce 2.24+ before using this file.
- Suggested fix: Either avoid the 2.24-only object form in the root compose file, or update Docker docs and `scripts/docker/setup.sh` preflight to require Docker Compose >= 2.24.0 with a clear error.
- Confidence: medium
