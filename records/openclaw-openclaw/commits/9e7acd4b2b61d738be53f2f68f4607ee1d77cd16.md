# Commit 9e7acd4b2b

## Summary

The commit fixes plugin-scoped stale diagnostics, but the new `pluginId` guard also preserves stale non-plugin discovery diagnostics even after the user removes the bad `plugins.load.paths` entry.

## Findings

### Low: Removed `plugins.load.paths` errors can stay cached

- Kind: regression
- File: `src/plugins/plugin-registry-snapshot.ts`
- Line: 138
- Evidence: `hasStalePersistedPluginDiagnostics` now only invalidates a missing diagnostic `source` when `diag.pluginId` is present. Missing configured plugin paths are emitted by discovery without `pluginId`, and `resolveInstalledPluginIndexPolicyHash` does not include `plugins.load.paths`, so removing the bad path does not make the persisted registry stale. I reproduced this by seeding a persisted index from `{ plugins: { load: { paths: [missingPath] } } }`, then loading with `{}`: the policy hash was unchanged, `result.source` was `"persisted"`, and `snapshot.diagnostics` still contained `plugin path not found: <missingPath>`.
- Impact: after a user fixes config by deleting a bad plugin load path, status/plugin registry/doctor-style metadata surfaces can keep reporting the old missing-path error until a manual registry refresh. This is the same class of stale diagnostic noise the surrounding change is trying to remove, just for configured load-path diagnostics instead of plugin-owned diagnostics.
- Suggested fix: include plugin discovery roots such as `plugins.load.paths` in the persisted registry policy/control-plane fingerprint, or re-derive when a non-plugin missing-path diagnostic no longer corresponds to the current configured load paths. The broader policy-hash fix is safer because it also covers path changes that add or replace plugin roots.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: the focused `pnpm exec tsx` smoke described above reproduced the stale persisted diagnostic on current `main`.
- Is this the best way to solve the issue? unclear: the narrow fix is to special-case stale non-plugin path diagnostics, but the plugin architecture docs say metadata compatibility checks should include plugin discovery roots, so extending the policy/fingerprint to cover `plugins.load.paths` is likely the cleaner fix.
