# Commit 2b4b60b551

## Summary

Found one low-severity display bug in the new Claude CLI status-label fallback. The intended OAuth case is covered and the focused changed tests pass, but the new branch labels any readable Claude CLI credential as OAuth even though the credential reader can return a token credential.

## Findings

### Low: Token-only Claude CLI credentials are mislabeled as OAuth

- Kind: bug
- File: `src/agents/model-auth-label.ts`
- Line: 88
- Evidence: The new fallback returns `oauth (claude-cli)` whenever `readClaudeCliCredentialsCached(...)` returns a truthy value. The reader can return `{ type: "token", provider: "anthropic", token, expires }` when the Claude CLI credential has an access token and expiry but no refresh token (`src/agents/cli-credentials.ts:115`). The nearby profile-label path already distinguishes `oauth` from `token` (`src/agents/model-auth-label.ts:60`), and external CLI sync deliberately ignores non-OAuth Claude CLI credentials (`src/agents/auth-profiles/external-cli-sync.ts:119`).
- Impact: `/status` can report `oauth (claude-cli)` for an access-token-only Claude CLI credential. This is display-only, but it gives users the wrong auth mode and can hide that the runtime does not have a refreshable OAuth credential.
- Suggested fix: Store the returned Claude credential in a local variable and branch on `credential.type`; return `oauth (claude-cli)` only for OAuth, and either return `token (claude-cli)` for token credentials or keep the existing external-sync policy by falling through to `unknown`.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes at source level: a Claude CLI credential object with `type: "token"` is accepted by the new truthiness check and produces the OAuth label. I did not add a temporary regression test because this is a report-only review.
- Is this the best way to solve the issue? Yes, the narrow fix is to preserve the credential type already exposed by the reader and used by adjacent auth code.
