# Commit 66b02c91b1

## Summary

Found one regression in the Docker build fallback path: the commit re-runs A2UI asset bundling inside `build:docker`, after `Dockerfile` has already caught a bundling failure and written a stub for QEMU cross-arch builds.

## Findings

### Medium: `build:docker` bypasses the Dockerfile A2UI fallback on cross-arch builds

- Kind: compatibility
- File: `package.json`
- Line: 1305
- Evidence: `Dockerfile:95` says A2UI bundling may fail under QEMU cross-compilation and `Dockerfile:98` intentionally catches `pnpm canvas:a2ui:bundle` failure, writes a stub `a2ui.bundle.js`, and writes `.bundle.hash` as `stub`. The new `package.json:1305` command then runs `pnpm plugins:assets:build` during `RUN pnpm build:docker`; hook discovery confirms that phase runs the canvas command `node scripts/bundle-a2ui.mjs`. In `extensions/canvas/scripts/bundle-a2ui.mjs:170`, the script sees the app dir and installed `@a2ui/lit`; in `extensions/canvas/scripts/bundle-a2ui.mjs:193`, the stub hash cannot match the computed current hash, so it proceeds to run rolldown again at `extensions/canvas/scripts/bundle-a2ui.mjs:211`. That repeats the same operation the Dockerfile just treated as non-fatal.
- Impact: local cross-arch Docker builds that previously succeeded with the stub fallback can fail again at `RUN pnpm build:docker`, defeating the documented Dockerfile compatibility path.
- Suggested fix: keep the Dockerfile fallback authoritative. Either avoid running `plugins:assets:build` inside `build:docker` for the Dockerfile path, or add an explicit “keep existing/stubbed A2UI bundle” mode that `build:docker` respects after the Dockerfile fallback.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes: run a cross-arch Docker build in the environment described by `Dockerfile:95` where the first `pnpm canvas:a2ui:bundle` fails under QEMU. The fallback writes the stub, then `pnpm build:docker` now invokes `plugins:assets:build` and retries the same bundle.
- Is this the best way to solve the issue? No. The intent to build assets before copying is valid for package builds, but the Dockerfile already has a failure-tolerant asset-build boundary. The safer fix is to preserve that boundary rather than retrying the failing bundler inside the later aggregate script.
