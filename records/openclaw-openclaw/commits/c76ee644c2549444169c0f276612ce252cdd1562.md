# Commit c76ee644c

## Summary

Found one concurrency gap in the persistent restore path. The normal in-memory registry path consumes sibling component entries as intended, but after restart, concurrent sibling clicks can still both consume different persistent entries before the asynchronous sibling deletes land.

## Findings

### Medium: Persistent component groups are not consumed atomically

- Kind: concurrency
- File: `extensions/discord/src/components-registry.ts`
- Line: 321
- Evidence: `resolvePersistentRegistryEntry` atomically consumes only `params.id`, then `resolveDiscordComponentEntryWithPersistence` calls `deletePersistentComponentConsumptionGroup(persisted)` afterward. That helper starts `store.delete(id)` calls with `void` and does not await them. In a persistent-only state, two concurrent sibling interactions can each consume their own key before either sibling delete removes the other key.
- Impact: After gateway restart, a non-reusable Discord confirmation panel can still deliver both sibling callbacks if two authorized interactions arrive concurrently. That preserves the original dangerous case for the persistent fallback path, e.g. Cancel and Confirm can both dispatch.
- Suggested fix: Make group consumption atomic in the persistent path. A durable group guard key consumed before dispatch, or a per-`consumptionGroupId` lock that wraps persistent consume plus awaited sibling deletion, would prevent sibling callbacks from racing.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. I reproduced it with an inline `tsx` check using a persistent-only mock store containing `btn_confirm` and `btn_cancel` in the same `consumptionGroupId`; concurrent `resolveDiscordComponentEntryWithPersistence` calls returned both ids.
- Is this the best way to solve the issue? yes on direction, though the exact implementation should fit the registry/store API. The important property is a single group-level consume decision before any callback is returned.
