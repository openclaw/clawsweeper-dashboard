# Commit 68f9710

## Summary

Found one concrete permission-routing bug in the new raw ACP approval relay path. The focused tests pass, but the raw `exec.approval.requested` broadcast can be matched to the wrong active ACP prompt when another run uses the same Gateway session key.

## Findings

### Medium: Raw approval broadcasts can be relayed to the wrong same-session ACP turn

- Kind: security
- File: `src/acp/translator.ts`
- Line: 1236
- Evidence: `handleExecApprovalRequestEvent` accepts global `exec.approval.requested` broadcasts, extracts only `request.sessionKey`, and calls `startApprovalRelay` without a run id. `startApprovalRelay` then falls back to `findUniquePendingBySessionKey` at `src/acp/translator.ts:1249`, which only proves this ACP bridge has one active prompt for that session key. It does not prove the approval belongs to that ACP prompt’s `idempotencyKey`.
- Impact: If an ACP bridge is attached to a shared Gateway session such as `agent:main:main`, and another channel/client/run in that same session requests an exec approval while an ACP prompt is active, the ACP client can approve or deny that unrelated command. I confirmed this with an in-process smoke check: an active ACP prompt in `agent:main:main` received a synthetic raw `exec.approval.requested` for `external-approval` with the same `sessionKey` and no run id, then the bridge called `requestPermission` once and resolved `exec.approval.resolve` with `{ id: "external-approval", decision: "allow-once" }`.
- Suggested fix: Do not resolve raw approval broadcasts by `sessionKey` alone. Add a run/idempotency correlation field to the Gateway exec approval request/event and require it here, or keep raw events as early display hints only and resolve them only after a matching run-scoped `agent` approval event arrives.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: start one pending ACP prompt for `agent:main:main`, then deliver an `exec.approval.requested` event whose payload has `{ id, request: { sessionKey: "agent:main:main" } }` but no run id. Current code relays and resolves it against the ACP prompt.
- Is this the best way to solve the issue? yes: the relay needs run-level correlation. Session-level matching is not enough because Gateway supports multiple active runs per session key.
