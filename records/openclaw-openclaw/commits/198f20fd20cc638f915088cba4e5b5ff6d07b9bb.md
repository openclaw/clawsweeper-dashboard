# Commit 198f20f

## Summary

Found one security-boundary regression in the approval runtime token forwarding. The new agent gateway helper suppresses the local approval runtime token only for explicit tool `gatewayUrl` overrides, but the lower gateway client can still resolve remote targets from `OPENCLAW_GATEWAY_URL` or `gateway.mode=remote`.

## Findings

### Medium: Approval runtime token is forwarded to remote gateway targets

- Kind: security
- File: `src/agents/tools/gateway.ts`
- Line: 163
- Evidence: `resolveApprovalRuntimeTokenForGatewayTool` returns `getOperatorApprovalRuntimeToken()` whenever the method is an approval request/wait and `opts.gatewayUrl` is absent. But `callGateway` later resolves `OPENCLAW_GATEWAY_URL` and `gateway.remote.url` when `opts.url` is undefined (`src/gateway/call.ts:450`, `src/gateway/connection-details.ts:44`). Read-only injected probes confirmed both remote paths create a `GatewayClient` with `hasApprovalRuntimeToken: true`.
- Impact: A process-local approval runtime capability is disclosed to remote gateway endpoints. The token also will not validate against a different gateway process, so remote-mode approval waits remain outside the intended runtime trust path.
- Suggested fix: Gate the token on the resolved final target/source, not raw tool input. Only attach it for the implicit local loopback gateway; add negative tests for `OPENCLAW_GATEWAY_URL` and `gateway.mode=remote` / `gateway.remote.url`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. With no tool `gatewayUrl`, an injected `OPENCLAW_GATEWAY_URL=wss://remote.example/ws` produced `{"url":"wss://remote.example/ws","hasApprovalRuntimeToken":true,"scopes":["operator.approvals"]}`. A configured remote gateway produced the same result for `wss://configured.example/ws`.
- Is this the best way to solve the issue? yes. The lower gateway path already computes the effective URL/source; the approval-runtime token decision needs that resolved trust boundary.
