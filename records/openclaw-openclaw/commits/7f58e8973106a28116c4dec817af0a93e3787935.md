# Commit 7f58e89

## Summary

Found one regression in the new sibling runtime-deps pruning path. A reused previous root is correctly kept during the first repair, but a later no-op repair can delete that same root while the current root still symlinks `node_modules` into it.

## Findings

### Medium: No-op repair can delete the active reused runtime-deps root

- Kind: regression
- File: `src/plugins/bundled-runtime-deps.ts`
- Line: 407
- Evidence: `repairBundledRuntimeDepsPackagePlanAsync` now calls `pruneSiblingExternalBundledRuntimeDepsRoots` whenever `plan.missingSpecs.length === 0`. That includes the state created by the existing reuse path: `tryReuseBundledRuntimeDepsRoot` links `installRoot/node_modules` to `previousRoot/node_modules` and intentionally does not prune on the first repair when `reuseResult.status === "reused"`. On the next repair, the symlink makes the plan look materialized, so the new no-op branch deletes `previousRoot`; the current `node_modules` symlink is then broken.
- Impact: packaged upgrades that reuse a prior same-package runtime-deps root can become non-materialized after a later `doctor --fix`, `plugins deps --repair`, gateway startup prestage, or config hot reload pass. The command reports no repaired specs, but the active dependency root has been removed. Subsequent plugin runtime preparation can fail with missing bundled runtime dependencies until another repair recreates a concrete install.
- Suggested fix: before pruning sibling roots, detect whether `installRoot/node_modules` is a symlink into a sibling root and skip that target root. A stricter alternative is to prune siblings only when the current install root has concrete `node_modules`, not a reuse symlink.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. I ran a focused `pnpm exec tsx --eval` repro using the repo’s bundled runtime-deps fixture helpers. First repair reused `openclaw-2026.4.28-<hash>` and created a symlink; second no-op repair returned `repairedSpecs: []`, deleted the previous root, left `nodeModulesIsSymlink: true`, `symlinkTargetExists: false`, and `materialized: false`.
- Is this the best way to solve the issue? yes, the safest fix is to preserve any sibling root currently referenced by the current install root’s `node_modules` symlink. That keeps the intended pruning behavior for stale concrete roots without invalidating the reuse optimization.
