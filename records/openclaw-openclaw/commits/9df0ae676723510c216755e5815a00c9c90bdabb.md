# Commit 9df0ae6

## Summary

I found one low-severity attribution gap. The patch adds `sessionId`/`lane` context to coercion, but pre-existing `FailoverError` objects are returned unchanged, so several direct inner failover paths can still lose the new request attribution.

## Findings

### Low: Existing FailoverError instances bypass session/lane enrichment

- Kind: bug
- File: `src/agents/failover-error.ts`
- Line: 476
- Evidence: `coerceToFailoverError(err, context)` immediately returns `err` when `isFailoverError(err)` is true, ignoring the new `context.sessionId` and `context.lane`. `runWithModelFallback` now passes attribution into this function, but helper paths still create `FailoverError` without those fields, for example `src/agents/pi-embedded-runner/run/assistant-failover.ts:193`, `src/agents/pi-embedded-runner/run/auth-controller.ts:335`, and `src/agents/cli-runner/claude-live-session.ts:390`. A focused smoke confirmed the behavior: coercing an existing `FailoverError` with `{ sessionId: "session:demo", lane: "main" }` returns the same object with both fields still `null`.
- Impact: single-candidate/no-fallback failures, and any structured consumer that inspects the inner `FailoverError` or `describeFailoverError()` result, can still miss session/lane attribution even though the caller supplied it to the fallback boundary.
- Suggested fix: when `coerceToFailoverError` receives a `FailoverError` plus missing context fields, return a cloned/wrapped `FailoverError` with existing metadata preserved and context fields filled, or thread `sessionId`/`lane` into every direct `new FailoverError(...)` helper path.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the helper behavior: construct a `FailoverError` without `sessionId`/`lane`, pass it to `coerceToFailoverError` with attribution context, and observe the fields are not filled.
- Is this the best way to solve the issue? Unclear. Cloning in `coerceToFailoverError` is the smallest central fix, but explicitly passing attribution through all helper constructors may be clearer where session/lane are already in scope.
