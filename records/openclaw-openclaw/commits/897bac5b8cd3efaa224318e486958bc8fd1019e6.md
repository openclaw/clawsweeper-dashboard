# Commit 897bac5b8c

## Summary

Found one crash-durability regression risk in the session-store fsync opt-out. The latency fix is intentional and well targeted for #73655, but current recovery behavior makes a crash-corrupted `sessions.json` get treated as an empty store and then overwritten.

## Findings

### Medium: Non-durable session-store writes can turn crash corruption into silent session metadata loss

- Kind: data_loss
- File: `src/config/sessions/store.ts`
- Line: 502
- Evidence: The commit writes the session store with `writeTextAtomic(..., { durable: false, mode: 0o600 })`. The pinned `@openclaw/fs-safe` implementation maps `durable: false` to `syncTempFile: false` and `syncParentDir: false`, so the temp file and parent directory are no longer fsynced before/after rename. `loadSessionStore` initializes `store = {}` and swallows final read/parse failures, so an invalid or truncated `sessions.json` is returned as an empty store. I verified this with a focused temp-file smoke: an invalid `sessions.json` loaded with `0` keys, and `updateSessionStore` then rewrote the file from that empty store.
- Impact: A kernel panic, power loss, VM reset, or filesystem crash during the no-fsync replacement window can leave the renamed target missing, stale, truncated, or unparsable on filesystems that do not guarantee crash consistency without file and directory fsync. On the next gateway write, OpenClaw can silently replace the prior session registry with a new partial store. This can lose non-reconstructible session state such as `pendingFinalDeliveryText`, delivery context, model/auth-profile overrides, group activation, ACP metadata, subagent recovery markers, plugin state, and transcript path references.
- Suggested fix: Keep the session-store latency fix, but fail closed or recover on invalid stores before overwriting. Practical options: maintain a last-known-good `sessions.json.bak` and load it on parse failure; make `loadSessionStore` surface invalid JSON for write paths instead of returning `{}`; or use a background/group-commit durability path that keeps a crash-safe checkpoint outside the writer lock.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? No for the actual crash window; it requires OS/filesystem crash timing. The downstream failure path is reproducible: invalid `sessions.json` is loaded as empty and a later write rewrites from that empty store.
- Is this the best way to solve the issue? Unclear. The #73655 latency fix is addressing a real starvation path, so reverting to in-lock fsync is probably not the best fix. Adding a last-known-good recovery path or fail-closed invalid-store handling is safer.
