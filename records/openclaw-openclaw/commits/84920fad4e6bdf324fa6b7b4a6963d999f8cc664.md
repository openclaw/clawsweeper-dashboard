# Commit 84920fa

## Summary

Found one concrete redaction gap in the new payment credential handling: payment fields are redacted only when their values are strings. Numeric `card_number` / `cvc` values with the same sensitive keys still pass through text redaction and structured tool-payload sanitization unchanged.

## Findings

### High: Numeric payment credential fields bypass the new redaction

- Kind: security
- File: `src/logging/redact.ts`
- Line: 29
- Evidence: The added JSON-field pattern only matches quoted string values: `"...payment key...": "([^"]+)"`. The structured sanitizer in `src/agents/pi-embedded-subscribe.tools.ts:136` only calls `redactSensitiveFieldValue(key, child)` when `typeof child === "string"`; numeric primitives recurse unchanged. A focused live check confirmed both paths leak unchanged:
  - `redactSensitiveText(JSON.stringify({card_number:4242424242424242,cvc:123,amount:4200}), {mode:"tools"})` returned `{"card_number":4242424242424242,"cvc":123,"amount":4200}`.
  - `sanitizeToolResult({details:{structuredContent:{cardNumber:4242424242424242,cvc:123,amount:4200}}})` returned the same numeric `cardNumber` and `cvc` values.
- Impact: A wallet-style MCP/tool result that emits card number or CVC as JSON numbers rather than strings can still expose raw payment credentials in Control UI tool events. The same text pattern gap also affects log/transcript strings that contain numeric JSON payment fields.
- Suggested fix: Make payment-key redaction key-aware for non-string primitives too. For text, extend the JSON payment-field pattern to match string and numeric scalar values. For structured payloads, when a key matches `STRUCTURED_SECRET_FIELD_RE`, mask any primitive value, not only strings. Add focused tests for numeric `card_number`, `cardNumber`, and `cvc` in both JSON text and `details.structuredContent`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The focused `pnpm exec tsx -e ...` smoke above exercised the committed redaction helpers directly and showed numeric payment fields remain raw.
- Is this the best way to solve the issue? yes. The fix should stay key-scoped, matching the PR’s intent to avoid broad card-number scanning while still redacting values under explicitly sensitive payment field names.
