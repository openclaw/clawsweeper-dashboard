## Findings

### Low: OpenAI provider docs still advertise WebSocket-first as the default

- Kind: regression
- File: `docs/providers/openai.md`
- Line: 741
- Evidence: The commit changes direct OpenAI provider defaults to `buildOpenAIResponsesProviderHooks({ transport: "sse" })` in `extensions/openai/openai-provider.ts:230`, and the changed provider test now expects `transport: "sse"` plus no default `openaiWsWarmup` in `extensions/openai/openai-provider.test.ts:510`. The docs still say `"auto"` is the default WebSocket-first transport in `docs/providers/openai.md:741`, and the concept docs still say `openai` default transport is `auto` in `docs/concepts/model-providers.md:95`.
- Impact: Users following the published docs will expect direct `openai/*` turns to try WebSocket first, while the actual default is now SSE. That can mislead troubleshooting and makes the documented opt-in/opt-out behavior backwards from the shipped default.
- Suggested fix: Update the OpenAI provider docs to say direct `openai/*` defaults to SSE, `transport: "auto"` opts back into WebSocket-first behavior, and clarify whether `openai-codex/*` still defaults differently.
- Confidence: high
