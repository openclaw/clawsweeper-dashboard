# Commit f0adbd4

## Summary

Found one concrete reliability regression in the new internal Discord gateway heartbeat scheduler.

## Findings

### Medium: Gateway can reconnect spuriously when the first heartbeat jitter is high

- Kind: reliability
- File: `extensions/discord/src/internal/gateway-lifecycle.ts`
- Line: 16
- Evidence: `GatewayHeartbeatTimers.start()` schedules the first heartbeat at `intervalMs * random()` and also starts the recurring `setInterval(..., intervalMs)` immediately from the Hello timestamp. `sendHeartbeat()` flips `lastHeartbeatAck` to `false` in `extensions/discord/src/internal/gateway.ts:313`, and the interval path treats a false ACK as a timeout at `extensions/discord/src/internal/gateway.ts:298`. With a valid high jitter value, the first heartbeat can be sent just before the interval’s ACK check. A direct probe with `intervalMs=20` and `random=() => 1` produced `{"heartbeats":1,"timeouts":1}`.
- Impact: Discord gateway sessions can be torn down and reconnected even though the first heartbeat was just sent and the server has not had a full heartbeat interval to ACK it. In production with a 45s interval and jitter near `1`, this can leave only hundreds of milliseconds for the ACK, causing avoidable reconnect churn or startup instability.
- Suggested fix: start the recurring ACK-check interval after the first heartbeat fires, or schedule the first recurring check for `firstHeartbeatDelay + intervalMs`. Add a timer test with high jitter proving the client does not timeout until a full interval after the sent heartbeat.
- Confidence: high
