# Commit 9eee202

## Summary

Found two actionable regressions in the new main-session cron lane handling. The execution lane can target the wrong agent when the configured default agent is not `main`, and manual `cron.run` task records still point at the human main session while the actual cron work runs in the new cron lane.

## Findings

### Medium: Main cron jobs without an explicit agent can run on `agent:main` instead of the configured default agent

- Kind: regression
- File: `src/cron/service/timer.ts`
- Line: 439
- Evidence: `resolveMainSessionCronRunSessionKey()` derives the lane agent from `job.agentId` or `resolveAgentIdFromSessionKey(job.sessionKey)`. For a normal `sessionTarget: "main"` job with no `agentId` and no `sessionKey`, that falls back to `main`, ignoring `state.deps.defaultAgentId`. The gateway then derives `agentId` from the generated `agent:main:cron:...` key in `src/gateway/server-cron.ts:228`, so a config whose default agent is `primary` but also has a non-default `main` agent wakes `agentId: "main"` and queues the system event under `agent:main:cron:...`.
- Impact: Main-session cron jobs are constrained to the default agent by `assertMainSessionAgentId`, but this commit can route them to a different configured agent. That can use the wrong model/workspace/auth defaults and misses the intended default-agent lane isolation.
- Suggested fix: Resolve the main cron run lane from the effective cron target agent. Pass `state.deps.defaultAgentId` into the helper, and only derive from `job.sessionKey` when a real target session key is present; otherwise use the default agent.
- Confidence: high

### Low: Manual main cron task records still attach to the human main lane

- Kind: regression
- File: `src/cron/service/ops.ts`
- Line: 576
- Evidence: The timer/startup task path now uses `resolveCronTaskChildSessionKey()` and records the new cron run key, but `tryCreateManualTaskRun()` still writes `childSessionKey: params.job.sessionKey`. A focused harness for `cron.run(job.id, "force")` observed `taskChildSessionKey: "agent:main:main"` while both the queued event and heartbeat used `agent:main:cron:<job>:run:<timestamp>`.
- Impact: Manual/forced main cron runs leave task history and session-filtered task status attached to the human lane instead of the actual execution lane. That undermines the task ledger and operator inspection surfaces introduced by this fix.
- Suggested fix: Share the same main cron child-session resolver between `timer.ts` and `ops.ts`, and add a manual `cron.run` regression test asserting the task record matches the execution session key.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. I ran two focused inline harnesses: one through `buildGatewayCronService` showing a default `primary` config waking `agentId: "main"`, and one through `CronService.run` showing the task/event session-key mismatch.
- Is this the best way to solve the issue? yes. The fix direction should centralize effective main cron lane resolution and reuse it for timer, startup, and manual paths.
