# Commit fe9a89a2d2

## Summary

Found one Discord realtime voice regression. The commit makes every first non-empty audio chunk from an active speaker report provider playback as active, even when Discord playback is idle. With the OpenAI realtime bridge, that can truncate the previous completed assistant audio item on the next normal user turn.

## Findings

### Medium: Normal follow-up speech can truncate the previous realtime assistant item

- Kind: regression
- File: `extensions/discord/src/voice/realtime.ts`
- Line: 278
- Evidence: `sendInputAudioForTurn()` now calls `this.handleBargeIn()` on the first non-empty audio chunk for every speaker turn when barge-in is enabled, independent of whether Discord playback is currently active. `handleBargeIn()` then always calls the bridge with `{ audioPlaybackActive: true }` at `extensions/discord/src/voice/realtime.ts:293`.
- Evidence: OpenAI's bridge treats `audioPlaybackActive: true` as sufficient to send `conversation.item.truncate` whenever it still has `lastAssistantItemId` and `responseStartTimestamp`, even after `response.done` only flips `responseActive` false. See `extensions/openai/realtime-voice-provider.ts:728` and `extensions/openai/realtime-voice-provider.ts:781`.
- Evidence: Discord never calls `setMediaTimestamp()`; repo search only found the method definition and non-Discord callers. OpenAI defaults `latestMediaTimestamp` to `0`, so a later ordinary user turn can truncate the prior assistant item to `audio_end_ms: 0`.
- Impact: In Discord realtime voice, after an assistant response finishes naturally, the next user utterance can mutate the provider conversation by truncating the prior assistant audio item even though no barge-in happened. That can drop or corrupt the prior assistant context and may produce provider errors or degraded follow-up behavior.
- Suggested fix: Do not pass `audioPlaybackActive: true` from ordinary `sendInputAudioForTurn()` unless Discord output is actually playing or the realtime session has an explicit still-active provider response state. Restore playback-aware gating for normal input, and add a regression test where a completed assistant audio response is followed by a new user turn with no active Discord playback and no `conversation.item.truncate` is sent.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Simulate an OpenAI realtime session with an assistant audio delta, immediate mark acknowledgement, `response.done`, no Discord `setMediaTimestamp()`, then send the first audio chunk of a new Discord speaker turn. The current Discord code calls `handleBargeIn({ audioPlaybackActive: true })`, and the OpenAI bridge path sends `conversation.item.truncate`.
- Is this the best way to solve the issue? yes. The fix should keep explicit Discord barge-in for active playback, but avoid claiming active playback for every normal first input chunk. A precise active-response/playback flag is safer than using `audioPlaybackActive: true` as a broad provider-state interrupt.
