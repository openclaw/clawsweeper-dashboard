# Commit 014b527

## Summary

The commit fixes the reported denial loop for ordinary denied approvals and `exec.denied` node events, and the focused regression tests pass. I found one concrete follow-on regression risk: the new terminal-denial branch treats internal execution failures encoded as `Exec denied` the same as user denials, which can silently drop approved async command failures for internal/webchat-style sessions.

## Findings

### Medium: Approved async exec failures can be silently dropped as “terminal denials”

- Kind: reliability
- File: `src/agents/bash-tools.exec-approval-followup.ts`
- Line: 295
- Evidence: `sendExecApprovalFollowup` now routes every parsed `Exec denied (...)` result through the terminal denied branch, returning `false` when there is no external direct delivery target. But async gateway/node callers also encode non-denial failures as `Exec denied`, including approval plumbing failure at `src/agents/bash-tools.exec-host-gateway.ts:704`, gateway spawn failure at `src/agents/bash-tools.exec-host-gateway.ts:744`, and node invoke failure at `src/agents/bash-tools.exec-host-node.ts:259`. A focused probe on current `main` showed `sendExecApprovalFollowup` returns `false` for `spawn-failed` and `invoke-failed` with `sessionKey: "agent:main:main"` and `turnSourceChannel: "webchat"`, so no agent follow-up and no direct message is sent.
- Impact: if an async approval is granted but the gateway process cannot start the command, or the node invoke fails, internal/webchat-style sessions can remain stuck at the original `approval-pending` result with no visible failure. That is different from suppressing user-denied approvals; these are execution/reporting failures after the approval path.
- Suggested fix: split approval-denial reasons from execution/reporting failure reasons. Keep `user-denied`, `approval-timeout`, and similar policy denials terminal, but route `approval-request-failed`, `spawn-failed`, and `invoke-failed` to an operator-visible failure path that does not create a command retry loop.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes, with a focused source-level probe of `sendExecApprovalFollowup` on current `main`; a full live WebChat/node reproduction was not run.
- Is this the best way to solve the issue? unclear; the safest direction is to preserve the no-agent-wakeup behavior for true denials while adding a non-model-visible or tightly bounded failure notification path for internal execution failures.
