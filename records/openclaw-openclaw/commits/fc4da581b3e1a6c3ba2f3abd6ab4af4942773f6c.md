# Commit fc4da581b3

## Summary

Found one low-severity behavioral gap in the newly advertised Discord `upload-file` action: it can silently drop the shared `caption` parameter.

## Findings

### Low: `upload-file` ignores the shared `caption` field

- Kind: bug
- File: `extensions/discord/src/actions/handle-action.ts`
- Line: 136
- Evidence: the shared message tool exposes `caption` for attachment sends, and the core `send` path promotes `caption` to the message text when `message` is empty in `src/infra/outbound/message-action-runner.ts:610`. The new Discord `upload-file` branch only reads `message` and `content` before forwarding `content: content ?? ""` to `sendMessage`, so `message({ action: "upload-file", filePath: "...", caption: "..." })` sends the file with an empty Discord caption.
- Impact: agents using the attachment-specific `caption` field can deliver the file but lose the user-visible caption text. This is silent output loss, not a send failure.
- Suggested fix: read `caption` as a fallback in the Discord `upload-file` handler, or normalize `caption` to `message` for plugin-owned `upload-file` actions before dispatch. Add a focused test for `caption` on Discord `upload-file`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: invoke the message tool or `handleDiscordMessageAction` with `action: "upload-file"`, a valid `filePath`, and `caption` but no `message`/`content`; the current handler forwards an empty `content`.
- Is this the best way to solve the issue? yes for this commit scope: adding a `caption` fallback preserves the existing `message` behavior and matches the shared send-path contract.
