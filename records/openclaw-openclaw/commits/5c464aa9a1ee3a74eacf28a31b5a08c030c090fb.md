# Commit 5c464aa

## Summary

The commit correctly models the documented Telegram long-final behavior where a long final can reuse the preview as chunk one and send a second remaining chunk. I found one low-severity QA guard weakness: the new range can also accept the stale-preview-plus-full-final duplicate pattern this scenario is meant to catch.

## Findings

### Low: Two-message range can mask stale preview plus duplicate final

- Kind: test_gap
- File: `extensions/qa-lab/src/live-transports/telegram/telegram-live.runtime.ts`
- Line: 445
- Evidence: The scenario rationale says it guards against "long streamed finals leaving stale preview messages behind" at `extensions/qa-lab/src/live-transports/telegram/telegram-live.runtime.ts:435`, but the assertion now only requires `expectedSutMessageCountRange: [1, 2]` plus joined text containing begin/end. The helper counts SUT messages and then checks `messages.map(...).join("")` for marker inclusion at `extensions/qa-lab/src/live-transports/telegram/telegram-live.runtime.ts:1024` and `extensions/qa-lab/src/live-transports/telegram/telegram-live.runtime.ts:1034`. A stale preview message containing `TELEGRAM-LONG-FINAL-BEGIN` followed by a separate full final message containing both begin/end would therefore pass as two messages, even though that is the duplicate/stale-preview shape the scenario title and docs say should be guarded.
- Impact: The mock Telegram live lane can pass while a regression leaves a visible stale preview plus a full final reply, as long as exactly two SUT messages are observed. That weakens the regression signal for Telegram preview finalization.
- Suggested fix: Keep accepting legitimate two-chunk finals, but make the assertion distinguish chunks from duplicate finals. For example, require the start/end markers to occur exactly once across the joined SUT text, or add a scenario-specific validator that rejects two-message observations where the second message contains the start marker again or the first message is not the first final chunk.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes, at helper level: add a unit case beside `asserts long Telegram final replies reuse the streamed preview message` with two observed SUT messages, first `TELEGRAM-LONG-FINAL-BEGIN stale preview`, second `TELEGRAM-LONG-FINAL-BEGIN complete final TELEGRAM-LONG-FINAL-END`. Current code accepts it because count is within `[1, 2]` and joined text includes both expected markers.
- Is this the best way to solve the issue? Yes: preserve the range for legitimate chunking, but assert marker uniqueness or chunk shape so the live QA still catches duplicate final delivery.
