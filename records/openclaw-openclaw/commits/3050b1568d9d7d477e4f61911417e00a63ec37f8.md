# Commit 3050b15

## Summary

Found one low-severity compatibility edge case in the new Playwright cache probing order.

## Findings

### Low: Linux arm64 can select an x64 Playwright Chromium from a shared cache

- Kind: compatibility
- File: `extensions/browser/src/browser/chrome.executables.ts`
- Line: 496
- Evidence: The new loop probes `chrome-linux64/chrome` before `chrome-linux/chrome` for every Linux host. Playwright’s own `playwright-core@1.59.1` registry maps Chromium `linux-x64` to `chrome-linux64/chrome` and `linux-arm64` to `chrome-linux/chrome`, while OpenClaw’s docs recommend persistent `PLAYWRIGHT_BROWSERS_PATH` caches and the Docker release workflow builds both `linux/amd64` and `linux/arm64`. A focused local probe with system candidates hidden and both cache layouts present returns the `chrome-linux64` path.
- Impact: On an arm64 Linux host with a reused or copied Playwright cache containing both layouts under the same `chromium-*` revision, OpenClaw will try to launch the x64 Chromium binary instead of the arm64 one, likely failing browser startup with an exec-format/runtime error.
- Suggested fix: Make the Playwright cache candidate order architecture-aware: prefer `chrome-linux64` for `process.arch === "x64"` and `chrome-linux` for `process.arch === "arm64"`, with the other layout only as an explicit fallback if desired. Add a regression test for the both-directories-present arm64 ordering.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Partially. I reproduced the deterministic resolver choice with a temp cache containing both layouts and system browser paths hidden. I did not have an arm64 host here to live-prove the final exec-format failure.
- Is this the best way to solve the issue? Yes. Matching Playwright’s own host-platform executable mapping avoids guessing from filesystem presence alone.
