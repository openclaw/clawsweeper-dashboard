# Commit 3050b15

## Summary

Found one low-severity compatibility edge in the new Playwright Linux cache detection: mixed x64/arm64 caches can make arm64 hosts select the x64 Chromium binary first.

## Findings

### Low: Playwright mixed-architecture cache can select the x64 Chromium binary on arm64

- Kind: compatibility
- File: `extensions/browser/src/browser/chrome.executables.ts`
- Line: 496
- Evidence: Current `main` still probes Playwright cache candidates in fixed order: `chrome-linux64` before `chrome-linux`. Playwright 1.59.1’s own registry maps Chromium `linux-x64` to `chrome-linux64/chrome` and `linux-arm64` to `chrome-linux/chrome`, while the cache directory is still the shared `chromium-<revision>` shape. A focused local probe with both subdirectories present returned the `chrome-linux64/chrome` candidate first; because this OpenClaw code never checks `process.arch`, the same selection order applies on arm64.
- Impact: On Linux arm64 systems that reuse, mount, or copy a Playwright cache containing both architecture layouts, OpenClaw can launch the x64 Chromium binary instead of the arm64 one, leading to an exec-format launch failure even though a valid arm64 binary exists next to it.
- Suggested fix: Order the Playwright Linux subdirectory candidates by `process.arch` or by a small arch-to-Playwright-layout helper: `x64` should prefer `chrome-linux64`, `arm64` should prefer `chrome-linux`, and only then fall back to the other layout for legacy or unusual caches.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Partially. The selection bug is directly reproducible with a mixed Playwright cache probe, but I did not run the final launch failure on a real Linux arm64 host.
- Is this the best way to solve the issue? Yes. Matching Playwright’s own `linux-x64`/`linux-arm64` executable mapping preserves the new x64 cache support while avoiding wrong-architecture selection.
