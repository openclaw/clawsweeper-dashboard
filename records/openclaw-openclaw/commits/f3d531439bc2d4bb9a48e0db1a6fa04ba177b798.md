# Commit f3d5314

## Summary

Found two reliability issues in the new Mantis evidence publishing path. The more important one prevents Slack failure evidence from being published, which undermines the workflow’s main review value.

## Findings

### Medium: Slack failure runs skip manifest creation, so PR evidence publishing fails

- Kind: reliability
- File: `.github/workflows/mantis-slack-desktop-smoke.yml`
- Line: 219
- Evidence: The workflow uses `set -euo pipefail`, then runs `pnpm openclaw qa mantis slack-desktop-smoke` before writing `mantis-evidence.json` at lines 252-278. The CLI sets `process.exitCode = 1` when the Slack desktop smoke result is `fail` in `extensions/qa-lab/src/mantis/cli.runtime.ts:61`, while the runtime still writes failure summaries/reports in `extensions/qa-lab/src/mantis/slack-desktop-smoke.runtime.ts:773`. Because of `set -e`, any legitimate failed smoke aborts before the manifest is created. The later `always()` comment step still runs with `output_dir` set and calls `scripts/mantis/publish-pr-evidence.mjs`, which requires the missing manifest at `scripts/mantis/publish-pr-evidence.mjs:440`.
- Impact: A failed Slack desktop smoke uploads raw artifacts but does not publish the inline PR evidence comment, exactly when maintainers need screenshot/video/error context.
- Suggested fix: Capture the CLI exit code instead of letting `set -e` abort, generate the preview and manifest from the summary/report/error artifacts, publish evidence, then exit non-zero based on the recorded status.
- Confidence: high

### Low: Concurrent artifact publishes can lose to non-fast-forward pushes

- Kind: reliability
- File: `scripts/mantis/publish-pr-evidence.mjs`
- Line: 340
- Evidence: Each publisher fetches `qa-artifacts` once, commits under its own artifact root, then pushes `HEAD:qa-artifacts` at line 368 with no retry/rebase. Workflow concurrency is scoped by PR/ref/scenario, so separate PRs or scenarios can publish simultaneously to the same global branch.
- Impact: One of two otherwise successful Mantis runs can fail only at evidence publication with a non-fast-forward push, leaving the PR without inline evidence.
- Suggested fix: Add a bounded fetch/rebase-or-retry loop around the `qa-artifacts` push, or serialize all `qa-artifacts` writes with a shared workflow concurrency key.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. For the first finding, any Slack smoke result that returns `status: "fail"` makes the CLI exit 1, and the workflow shell exits before `mantis-evidence.json` is written.
- Is this the best way to solve the issue? Yes: preserving the failure status while still generating/publishing evidence keeps CI semantics and fixes the missing review artifact.
