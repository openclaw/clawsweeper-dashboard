# Commit 30b201e

## Summary

Found one compatibility regression in the Google Meet plugin config schema. The runtime still accepts legacy `defaultMode: "realtime"` as promised, but the manifest JSON Schema now rejects that same existing config value before the runtime resolver can normalize it.

## Findings

### Medium: Legacy `defaultMode: "realtime"` configs now fail plugin config validation

- Kind: compatibility
- File: `extensions/google-meet/openclaw.plugin.json`
- Line: 229
- Evidence: `extensions/google-meet/openclaw.plugin.json:229` defines `defaultMode.enum` as only `["agent", "bidi", "transcribe"]`. But `extensions/google-meet/src/config.ts:327` still treats `"realtime"` as a supported legacy alias and normalizes it to `"agent"`. Core config validation validates plugin entry config against the manifest schema when the plugin is enabled or has config, before relying on plugin runtime normalization (`src/config/validation.ts:1607`). A direct AJV check of the manifest schema against `{ "defaultMode": "realtime" }` fails with allowed values `agent`, `bidi`, `transcribe`.
- Impact: Users who already have `plugins.entries.google-meet.config.defaultMode: "realtime"` can fail OpenClaw config validation after upgrading, despite the changelog/docs saying the alias remains accepted for compatibility.
- Suggested fix: Keep the manifest validation path compatible while hiding the alias from new choices. For example, add support for hidden/deprecated enum aliases in the config schema/UI pipeline, or otherwise allow `"realtime"` in validation while filtering it out of agent/UI advertised options.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Validate `extensions/google-meet/openclaw.plugin.json` with AJV against `{ "defaultMode": "realtime" }`; it fails on `/defaultMode` enum. This matches the runtime/config-validation path in `src/config/validation.ts`.
- Is this the best way to solve the issue? unclear. The intent to hide `"realtime"` from new agent/UI choices is sound, but compatibility needs a schema mechanism that accepts old persisted values without advertising them.
