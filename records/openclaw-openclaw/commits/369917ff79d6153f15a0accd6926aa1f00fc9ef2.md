# Commit 369917f

## Summary

The commit adds visibility-only `openclaw doctor` diagnostics for cron `payload.model` pins. I found one low-severity gap where the new diagnostic can miss legacy cron stores on the same doctor run that migrates a top-level model pin into `payload.model`.

## Findings

### Low: Doctor reports model pins before legacy top-level model fields are migrated

- Kind: bug
- File: `src/commands/doctor-cron.ts`
- Line: 324
- Evidence: `noteCronModelOverrides` only counts `payload.model` (`src/commands/doctor-cron.ts:142`-`149`), but `maybeRepairLegacyCronStore` calls it before `normalizeStoredCronJobs` (`src/commands/doctor-cron.ts:324` then `:326`). The normalizer later copies legacy top-level `model` into `payload.model` for agent-turn jobs (`src/commands/doctor-cron-store-migration.ts:119`-`130`, `:353`).
- Impact: A legacy cron job with top-level `model: "openai/gpt-5.4"` can be repaired into a persistent `payload.model` pin without the new “Cron model overrides detected” note appearing until a second doctor run. That weakens the intended billing/auth investigation diagnostic for users with older cron store shapes.
- Suggested fix: Run the model override diagnostic after `normalizeStoredCronJobs(rawJobs)`, or teach it to also inspect legacy top-level model fields that the normalizer will migrate into `payload.model`.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes, by source trace: create a legacy cron job with top-level `message` and `model`, run doctor once, and the new note sees no `payload.model`; `doctor --fix` can then persist the migrated `payload.model`, making the note appear only on a later run.
- Is this the best way to solve the issue? Yes; moving the diagnostic after in-memory normalization reuses the existing migration contract and avoids adding a second legacy-shape detector.
