# Commit 7134a95c90

## Summary

Found one public SDK compatibility regression in the new QMD availability result type. Runtime QMD cwd handling and the focused regression tests otherwise look consistent with the intended fix.

## Findings

### Medium: QMD availability type no longer accepts the full old unavailable shape

- Kind: compatibility
- File: `packages/memory-host-sdk/src/host/qmd-process.ts`
- Line: 21
- Evidence: Before this commit, `QmdBinaryAvailability` was `{ available: boolean; error?: string }`, so `{ available: false }` was source-valid. The new public SDK type is `{ available: true } | QmdBinaryUnavailable`, and `QmdBinaryUnavailable.error` is required. This subpath is exported as `./plugin-sdk/memory-core-host-engine-qmd` in `package.json` and listed as a public plugin-owned SDK entrypoint in `src/plugin-sdk/entrypoints.ts`.
- Impact: External plugin code, typed mocks, or wrappers that still construct the old valid `{ available: false }` result will fail TypeScript after upgrading, despite the commit preserving optional `reason` for compatibility. The new callers also assume `qmdCheck.error.trim()`, so the runtime handling no longer tolerates the previous optional-error contract either.
- Suggested fix: Keep `error?: string` on `QmdBinaryUnavailable`, use a fallback in callers (`qmdCheck.error?.trim() || "unknown error"`), and add compatibility coverage for `{ available: false }` as well as `{ available: false, error }`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused TypeScript compiler probe against the current type rejected `const legacyNoError: QmdBinaryAvailability = { available: false };` with `TS2322: Property 'error' is missing`.
- Is this the best way to solve the issue? yes. Keeping `error` optional preserves the old SDK shape while still allowing the new `reason` discriminant and workspace-specific diagnostics.
