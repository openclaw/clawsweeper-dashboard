# Commit 5fce2f6

## Summary

Found one low-severity regression in the new config surface: the runtime config accepts and applies `gateway.controlUi.chatMessageMaxWidth`, but the generated JSON schema leaves the field untyped, so schema-driven Control UI config forms treat it as unsupported.

## Findings

### Low: `chatMessageMaxWidth` is unsupported in schema-driven config forms

- Kind: bug
- File: `src/config/schema.base.generated.ts`
- Line: 22296
- Evidence: The generated schema node for `gateway.controlUi.chatMessageMaxWidth` has only `title` and `description`, with no `type: "string"`. The Control UI schema analyzer resolves type from `schema.type` and, for nodes without a primitive type/object/array/enum, adds the path to `unsupportedPaths`; `renderNode` then displays “Unsupported schema node. Use Raw mode.” for that path. The runtime path itself is wired: the gateway includes `chatMessageMaxWidth` in the bootstrap payload and `app-render.ts` applies it as `--chat-message-max-width`.
- Impact: Users can configure the new setting through raw JSON/CLI, but the Control UI’s schema-driven config editor cannot render an input for the newly documented field. That makes the new public setting harder to discover and edit from the main UI surface.
- Suggested fix: Keep the runtime validation/normalization, but make the generated JSON schema for this field explicitly represent it as a string. For example, restructure the Zod schema or add a schema-generation override so `chatMessageMaxWidth` emits at least `{ type: "string" }`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Source path: open the generated schema at `src/config/schema.base.generated.ts:22296`, then trace `ui/src/ui/views/config-form.analyze.ts:50` and `ui/src/ui/views/config-form.node.ts:453`; the field is classified unsupported because it has no schema type.
- Is this the best way to solve the issue? yes. The feature already uses the right runtime/data path; the narrower fix is preserving schema representability for this single config field rather than changing the bootstrap or CSS behavior.
