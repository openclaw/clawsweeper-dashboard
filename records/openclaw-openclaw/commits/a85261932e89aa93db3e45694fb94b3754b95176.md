# Commit a85261932e

## Summary

Found one actionable regression risk in the new macOS `sips` fallback: it can bypass the existing HEIC image pixel budget when Sharp fails before conversion.

## Findings

### Medium: macOS HEIC fallback can bypass the input pixel limit

- Kind: reliability
- File: `src/media/image-ops.ts`
- Line: 581
- Evidence: `convertHeicToJpeg` calls `assertImagePixelLimit`, but that guard only reads PNG/GIF/WebP/JPEG headers and returns without validation when metadata is unknown unless `prefersSips()` is already true (`src/media/image-ops.ts:211`, `src/media/image-ops.ts:251`). The bundled Sharp path is configured with `limitInputPixels: MAX_IMAGE_INPUT_PIXELS` (`extensions/media-understanding-core/image-ops.ts:48`) and Sharp throws `Input image exceeds pixel limit` when decoded pixels exceed that limit (`node_modules/sharp/src/common.cc:581`). This commit catches every `ops.convertHeicToJpeg` error on Darwin and immediately runs `sipsConvertToJpeg` (`src/media/image-ops.ts:575`), so a Sharp pixel-limit failure, unsupported-decoder failure, or other safety failure all fall through to unbounded `sips` conversion. Callers such as input image normalization and CLI model-run files do not add a pixel-dimension check after conversion (`src/media/input-files.ts:262`, `src/cli/capability-cli.ts:619`).
- Impact: A small-but-huge-dimension HEIC/HEIF input on macOS can force `sips` to decode/convert an image that OpenClaw’s 25,000,000-pixel guard is meant to reject before expensive processing. URL/base64 input paths cap bytes, but they still run the expensive conversion first; the CLI model-run path has no comparable byte or pixel recheck around the converted buffer.
- Suggested fix: Before falling back to `sips`, query dimensions with the existing `sipsMetadataFromBuffer` helper and apply `validateImagePixelLimit`; fail closed if dimensions cannot be read. Also avoid catching Sharp pixel-limit errors as “try another backend” errors, or only fall back for known missing/unsupported HEIC decoder cases.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? No live macOS reproduction from this Linux checkout. The code path is concrete: on Darwin, any `ops.convertHeicToJpeg` rejection is swallowed and `sipsConvertToJpeg` runs without the HEIC dimension guard.
- Is this the best way to solve the issue? Yes. The fallback should preserve the same pre-decode pixel budget that the Sharp path enforces, and this file already has both `sipsMetadataFromBuffer` and `validateImagePixelLimit` available.
