# Commit cdd8e81

## Summary

Found one security hardening issue in the new release-publish workflow: it executes code from the release tag before proving the tag is reachable from `main` or `release/*`.

## Findings

### Medium: Untrusted release tag code runs before the reachability guard

- Kind: security
- File: `.github/workflows/openclaw-release-publish.yml`
- Line: 104
- Evidence: The workflow grants `actions: write` at workflow scope, checks out `refs/tags/${{ inputs.tag }}` at lines 97-102, then runs the repo-local setup action at lines 104-109. The tag reachability check does not run until lines 115-130. That setup action is code from the checked-out tag and runs another local action plus `pnpm install` with lifecycle scripts enabled (`--ignore-scripts=false`, `--config.enable-pre-post-scripts=true`) in `.github/actions/setup-node-env/action.yml`.
- Impact: A malformed or intentionally unreachable release tag can execute tag-controlled workflow/helper/package lifecycle code before the workflow rejects it. In this workflow that code runs with the parent job’s `GITHUB_TOKEN` carrying `actions: write`, so the pre-guard code can mutate Actions state, such as dispatching or cancelling workflow runs, even though the release target is later rejected.
- Suggested fix: Move the reachability validation immediately after checkout and before `setup-node-env` or any package install/script execution. Also scope permissions per job: keep `resolve_release_target` at `contents: read`, and grant `actions: write` only to the later dispatch job.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. Dispatch `OpenClaw Release Publish` with a syntactically valid tag that points to a commit not reachable from `main` or `release/*` and whose checked-out `.github/actions/setup-node-env/action.yml` or package lifecycle scripts have observable side effects. The side effect runs before the reachability step rejects the tag.
- Is this the best way to solve the issue? Yes. The release target must be proven trusted before running repo-local actions or package lifecycle scripts from that target. Job-level permission narrowing removes the unnecessary `actions: write` token from the resolving/preflight job.
