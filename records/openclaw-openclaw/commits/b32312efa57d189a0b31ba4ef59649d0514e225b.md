# Commit b32312e

## Summary

Found two regressions in the deferred auth-profile failure marking. The fix unblocks rotation, but some fallback exits now either return before cooldown state is durable or throw before scheduling the marker at all.

## Findings

### Medium: Fallback exits can finish before profile cooldown is persisted

- Kind: regression
- File: `src/agents/pi-embedded-runner/run/assistant-failover.ts`
- Line: 154
- Evidence: `markFailedProfile()` fires `maybeMarkAuthProfileFailure(...).catch(...)` without awaiting it, then the code can immediately fall through to `fallback_model`/`surface_error` after `advanceAuthProfile()` returns `false`. The existing focused E2E test `src/agents/model-fallback.run-embedded.e2e.test.ts` now fails at line 570: after all fallback candidates are overloaded, `usageStats["openai:p1"]?.cooldownUntil` is still `undefined`.
- Impact: callers can observe a completed failed/fallback run before the failed profile cooldown is written. Immediate follow-up turns can reselect the same overloaded/rate-limited credential, and tests/diagnostics no longer have a reliable persisted failure record.
- Suggested fix: keep the rotation-success path non-blocking if needed, but await the failed-profile mark before throwing/returning from no-rotation and model-fallback exhaustion paths. A helper that returns the marker promise would make the required happens-before explicit.
- Confidence: high

### Medium: Rate-limit rotation cap throws before marking the cap-triggering profile

- Kind: regression
- File: `src/agents/pi-embedded-runner/run/assistant-failover.ts`
- Line: 146
- Evidence: `maybeEscalateRateLimitProfileFallback()` increments the rotation count and throws once `rateLimitProfileRotations > rateLimitProfileRotationLimit` (`src/agents/pi-embedded-runner/run.ts:891`). The new assistant path calls it before `markFailedProfile()` at line 154, and the prompt path calls it at `src/agents/pi-embedded-runner/run.ts:2025` before the deferred marker at line 2045. In the parent commit, both paths marked the failed profile before this escalation point.
- Impact: with the default `rateLimitedProfileRotations: 1`, the second rate-limited profile that triggers cross-model fallback is never marked for cooldown, so it remains eligible on later turns despite having just failed.
- Suggested fix: capture and schedule/await the failed-profile mark before any cap helper that can throw, or make the cap helper return a decision instead of throwing before cleanup.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. `pnpm test src/agents/pi-embedded-runner/run/assistant-failover.test.ts src/agents/pi-embedded-runner.run-embedded-pi-agent.auth-profile-rotation.e2e.test.ts src/agents/model-fallback.run-embedded.e2e.test.ts -- --run` fails the existing model-fallback cooldown assertion after dependency install.
- Is this the best way to solve the issue? yes: preserve the commit’s useful rotate-before-lock behavior only for successful same-provider rotation, but restore durable marking before leaving the provider/run or before synchronous rate-limit cap throws.
