# Commit 94ac563

## Summary

The Highlight.js major upgrade breaks the Control UI production build. The existing UI imports `highlight.js/lib/core.js`, but Highlight.js 11 exports `./lib/core` without the `.js` suffix, and Vite 8 rejects the `.js` subpath.

## Findings

### High: Highlight.js 11 upgrade breaks UI build resolution

- Kind: regression
- File: `ui/src/ui/markdown.ts`
- Line: 2
- Evidence: `ui/package.json:15` upgrades `highlight.js` from `10.7.3` to `11.11.1`, while `ui/src/ui/markdown.ts:2` still imports `highlight.js/lib/core.js`. `highlight.js@11.11.1` package metadata exports `./lib/core`, not `./lib/core.js`. A focused Vite 8.0.13 probe with `highlight.js@11.11.1` failed with: `"./lib/core.js" is not exported ... from package ... highlight.js`. The same probe with `highlight.js@10.7.3` built successfully.
- Impact: `pnpm ui:build` / Control UI production builds fail before bundling, so the dependency update can block releases or any full build lane including the UI.
- Suggested fix: Change the Highlight.js imports to exported subpaths, e.g. `highlight.js/lib/core` and `highlight.js/lib/languages/bash` without `.js`, then update or remove `ui/src/types/highlight-js-subpaths.d.ts` accordingly. Downgrading Highlight.js is the fallback if the no-extension imports reveal further migration work.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A temp project using `vite@8.0.13` and `highlight.js@11.11.1` fails to build when importing `highlight.js/lib/core.js`; the same setup using `highlight.js@10.7.3` succeeds.
- Is this the best way to solve the issue? yes. The package export map explicitly exposes the no-extension subpath, and a Node ESM probe confirmed `highlight.js/lib/core` works while `highlight.js/lib/core.js` is blocked.
