# Commit 1daba52

## Summary

Found one low-severity regression in the new memory-wiki session visibility filter: allowed deleted/reset session archive hits are denied because the wiki copy of the visibility lookup drops archive owner metadata that the existing memory-core filter preserves.

## Findings

### Low: Wiki visibility filter drops same-agent archived session hits

- Kind: regression
- File: `extensions/memory-wiki/src/query.ts`
- Line: 1253
- Evidence: `createSessionMemoryPathVisibilityChecker()` extracts only the transcript stem and calls `resolveTranscriptStemToSessionKeys({ store, stem })` without `archivedOwnerAgentId`. For archive paths such as `sessions/main/deleted-stem.jsonl.deleted.2026-02-16T22-27-33.000Z`, `src/plugin-sdk/session-transcript-hit.ts:39` can return `{ stem, ownerAgentId, archived: true }`, and `resolveTranscriptStemToSessionKeys()` only synthesizes a fallback session key when `archivedOwnerAgentId` is provided at `src/plugin-sdk/session-transcript-hit.ts:91`. The established memory-core filter does pass that metadata at `extensions/memory-core/src/session-search-visibility.ts:45`, with regression coverage at `extensions/memory-core/src/session-search-visibility.test.ts:154`.
- Impact: After this commit, `wiki_search` / `wiki_get` with shared memory can hide same-agent reset/deleted transcript archive results when the live session store entry is gone, even though memory-core’s own session memory search intentionally allows those archive hits under the same visibility policy. That breaks post-reset/deleted-session recall through the wiki shared-memory path.
- Suggested fix: Use `extractTranscriptIdentityFromSessionsMemoryHit()` in memory-wiki and pass `archivedOwnerAgentId` when `identity.archived && identity.ownerAgentId`, matching `extensions/memory-core/src/session-search-visibility.ts`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. Configure memory-wiki shared search, mock or create a session archive hit path like `sessions/main/deleted-stem.jsonl.deleted.2026-02-16T22-27-33.000Z`, leave the combined session store without a live matching entry, and call `searchMemoryWiki()` or `getMemoryWikiPage()` from `agent:main:main` with `tools.sessions.visibility=agent`; memory-core keeps this hit, memory-wiki filters it out.
- Is this the best way to solve the issue? Yes. Reusing the identity helper keeps wiki behavior aligned with the existing dependency contract and avoids weakening the visibility guard.
