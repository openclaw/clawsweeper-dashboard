# Commit 72f6016

## Summary

Found one actionable regression in follow-up delivery: the new route-aware dedupe treats “some target metadata exists” as “all sent messages are route-classified,” but message sends can still lack target metadata when the message tool uses its default current-channel target.

## Findings

### Medium: Follow-up replies can duplicate implicit-target message sends when another routed send occurred

- Kind: reliability
- File: `src/auto-reply/reply/followup-delivery.ts`
- Line: 76
- Evidence: `resolveFollowupDeliveryPayloads()` now only runs text/media dedupe when `messagingToolPayloadDedupe.shouldDedupePayloads` is true. That helper returns `false` whenever `sentTargets.length > 0` and no sent target matches the origin route (`src/auto-reply/reply/reply-payloads-dedupe.ts:227`). However, `handleToolExecutionStart()` records sent text from raw message-tool args (`src/agents/pi-embedded-subscribe.handlers.tools.ts:704`) while `extractMessagingToolSend()` returns no target if the model omitted `to`/`target` (`src/agents/pi-embedded-subscribe.tools.ts:527`). The message tool can still infer the omitted target from the current source channel later (`src/infra/outbound/message-action-normalization.ts:36`), so a same-route send can have text/media metadata but no route entry. A focused read-only repro showed `extractMessagingToolSend("message", { action: "send", channel: "slack", message: "already sent to source" })` returns `null`, while normalization infers `target: "channel:C1"`, and `resolveFollowupDeliveryPayloads()` returns the duplicate final payload when any unrelated cross-target send is also present.
- Impact: A follow-up run that sends one message via the implicit current-chat target and also sends another message to a different route can now auto-deliver the already-sent implicit message again. For `sessions_send` or internal follow-up text, the same shape can also expose text that was meant for another session back to the requester chat.
- Suggested fix: Do not disable content/media fallback dedupe merely because at least one routed send exists. Either record normalized route metadata for every successful message send, including inferred/default current-channel sends, or keep content/media dedupe enabled when any sent text/media lacks an associated route.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The exported helpers reproduce the state shape without mutating files: implicit message-tool send target extraction returns no target, normalization infers the source target, and follow-up delivery keeps the duplicate final payload when an unrelated routed send exists.
- Is this the best way to solve the issue? no. The new route-aware behavior is useful for true cross-target sends, but it needs per-send target coverage or a fallback for unclassified sent text/media.
