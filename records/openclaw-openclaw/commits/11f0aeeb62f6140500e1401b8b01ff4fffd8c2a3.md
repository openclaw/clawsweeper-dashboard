# Commit 11f0aeeb

## Summary

Found one low-severity Docker live-test scheduler regression: the `live-gateway` lane now self-builds a custom image, but the aggregate scheduler still pre-builds the default shared live image first.

## Findings

### Low: `live-gateway` selected runs build two live Docker images

- Kind: reliability
- File: `scripts/lib/docker-e2e-scenarios.mjs`
- Line: 136
- Evidence: `live-gateway` now sets `OPENCLAW_IMAGE=openclaw:local-live-gateway` and passes `{ skipBuild: false }`, so its command intentionally omits `OPENCLAW_SKIP_DOCKER_BUILD=1`. The script then calls `scripts/test-live-gateway-models-docker.sh:168`, which invokes `scripts/test-live-build-docker.sh`; that build script tags `${OPENCLAW_IMAGE}-live` from `scripts/test-live-build-docker.sh:8` and builds it at `scripts/test-live-build-docker.sh:42`. Separately, the aggregate scheduler still adds `shared live-test image once` for any live lane at `scripts/test-docker-all.mjs:1239`, using the base env/default image rather than the lane’s `OPENCLAW_IMAGE`.
- Impact: a targeted `OPENCLAW_DOCKER_ALL_LANES=live-gateway pnpm test:docker:all` does an expensive default `openclaw:local-live` build that the lane no longer uses, then builds `openclaw:local-live-gateway-live`. That can waste CI/Testbox time and increase timeout risk for a lane that is already a live Docker validation path.
- Suggested fix: model live image ownership in the lane metadata. Either have the scheduler build the lane-specific live-gateway image once and run the lane with skip-build reuse, or mark this lane as self-building and make the scheduler skip the shared live image when selected lanes do not use it.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The code path is deterministic: select only `live-gateway`, leave `OPENCLAW_DOCKER_ALL_BUILD` at its default true value, and `scripts/test-docker-all.mjs:1239` schedules the shared live build before the lane command performs its own custom image build.
- Is this the best way to solve the issue? yes, at direction level. The safer fix is scheduler metadata rather than removing the lane-local build blindly, because the custom `OPENCLAW_IMAGE` and `OPENCLAW_DOCKER_BUILD_EXTENSIONS=matrix` need to stay aligned.
