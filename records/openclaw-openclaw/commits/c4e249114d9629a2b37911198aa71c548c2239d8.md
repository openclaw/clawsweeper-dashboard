## Summary

Found one concrete propagation gap: the commit makes workspace plugin auth evidence resolvable when `workspaceDir` reaches `resolveApiKeyForProvider`, but the main embedded PI auth bootstrap and compaction auth lookup still drop that workspace before credential resolution.

## Findings

### Medium: Embedded PI auth bootstrap still drops workspace auth evidence

- Kind: bug
- File: `src/agents/pi-embedded-runner/run/auth-controller.ts`
- Line: 350
- Evidence: `run.ts` constructs `createEmbeddedRunAuthController` with `workspaceDir: resolvedWorkspace` and then calls `initializeAuthProfile()` for the normal PI transport path, but `auth-controller.ts` calls `getApiKeyForModel({ model, cfg, profileId, store, agentDir, lockedProfile })` without forwarding `workspaceDir`. The new workspace plugin test shows this field is required: `resolveEnvApiKey("workspace-cloud", ..., { config: cfg })` returns `null`, while adding `{ workspaceDir }` returns the workspace credential marker. `src/agents/pi-embedded-runner/compact.ts:388` has the same omitted `workspaceDir` on its direct `getApiKeyForModel` call.
- Impact: A trusted workspace plugin provider can now appear authenticated in picker/status helpers, but a normal embedded run or compaction using that provider can still fail auth with no API key because the actual credential lookup cannot see the workspace plugin manifest evidence.
- Suggested fix: Pass `workspaceDir: params.workspaceDir` from `createEmbeddedRunAuthController` into `getApiKeyForModel`, and pass `workspaceDir: resolvedWorkspace` in `compactEmbeddedPiSessionDirect`. Add focused tests asserting the auth controller and compaction lookup forward workspace context.
- Confidence: high
