# Commit 7504fc3

## Summary

Found one actionable regression: the fix correctly targets `grok-4.3`, but it also enables OpenClaw thinking controls for other xAI reasoning models that xAI documents as automatic-reasoning models which reject explicit reasoning effort.

## Findings

### Medium: Reasoning effort is re-enabled for Grok models that reject it

- Kind: regression
- File: `extensions/xai/runtime-model-compat.ts`
- Line: 35
- Evidence: `applyXaiRuntimeModelCompat` now maps every model with `reasoning: true` to `low`/`medium`/`high` thinking efforts, `resolveThinkingProfile` exposes those levels from the catalog `reasoning` flag, and `stream.ts` preserves `reasoning` whenever `model.reasoning === true`. The xAI catalog marks `grok-4-fast`, `grok-4-1-fast`, and `grok-4.20-beta-latest-reasoning` as `reasoning: true`. Current xAI docs say `grok-4.3` accepts `reasoning.effort`; the summary table lists only `grok-4.3` and `grok-4.20-multi-agent`, while the guide explicitly says `reasoning_effort` is not supported by `grok-4.20` or `grok-4-1-fast` and specifying it returns an error. ([docs.x.ai](https://docs.x.ai/developers/model-capabilities/text/reasoning))
- Impact: users configured on `xai/grok-4-1-fast`, `xai/grok-4.20-*` reasoning aliases, or other automatic-reasoning Grok routes can now get a default/explicit `/think low|medium|high` payload and fail with an upstream 400 instead of letting the model reason automatically.
- Suggested fix: separate “model reasons” from “model accepts controllable reasoning effort.” Keep `grok-4.3` effort-capable, but leave automatic-reasoning Grok routes off-only in `resolveThinkingProfile`, map their `thinkingLevelMap` entries to `null`, and keep stripping `reasoning` for them. Add regression tests for `grok-4-1-fast` and Grok 4.20 with `reasoning: true`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes by source and contract: select an affected xAI model with default/low thinking and the current code path preserves `reasoning: { effort: "low" }`; xAI docs say those models reject explicit effort. Live HTTP was not run because no xAI key was present.
- Is this the best way to solve the issue? No. The commit fixes `grok-4.3`, but the safer model is an explicit effort-capability allowlist rather than using the broad `reasoning` catalog boolean.
