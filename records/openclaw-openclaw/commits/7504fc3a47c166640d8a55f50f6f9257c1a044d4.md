# Commit 7504fc3

## Summary

Found two medium regressions in the xAI reasoning-control mapping. The commit uses the broad catalog `reasoning: true` flag as if it meant “accepts configurable reasoning effort,” and it also makes `/think off` clamp back to low effort for Grok 4.3.

## Findings

### Medium: xAI thinking controls are exposed for models that do not support configurable effort

- Kind: regression
- File: `extensions/xai/provider-policy-api.ts`
- Line: 10
- Evidence: `resolveThinkingProfile` enables `minimal|low|medium|high` whenever `ctx.reasoning` or the catalog entry’s `reasoning` is true. The catalog marks several non-4.3 models as `reasoning: true`, including `grok-3-mini`, `grok-4`, `grok-4-fast`, `grok-4-1-fast`, `grok-4.20-beta-latest-reasoning`, and `grok-code-fast-1` in `extensions/xai/model-definitions.ts:84`, `extensions/xai/model-definitions.ts:111`, `extensions/xai/model-definitions.ts:129`, `extensions/xai/model-definitions.ts:147`, `extensions/xai/model-definitions.ts:165`, and `extensions/xai/model-definitions.ts:183`. The stream wrapper now preserves reasoning payloads for any `model.reasoning === true` at `extensions/xai/stream.ts:235`.
- Impact: Users selecting `/think high` on `xai/grok-4-fast`, `xai/grok-4-1-fast`, or forward-compatible `grok-4.20...` models now send `reasoning: { effort: "high" }`. The official xAI reasoning docs document configurable `reasoning_effort` for Grok 4.3 and Grok 4.20 Multi Agent, not for those ordinary Grok Fast / mini / code models. This can reintroduce request failures like the earlier “Invalid reasoning effort” path that `ea26a9dba0` had fixed by omitting xAI reasoning efforts. citeturn0view0
- Suggested fix: Split “model can reason” from “model accepts configurable reasoning effort.” Gate `resolveThinkingProfile`, `XAI_REASONING_EFFORTS`, and `supportsReasoningControls` on a narrow allowlist or catalog field for docs/live-verified configurable models, currently at least `grok-4.3`.
- Confidence: high

### Medium: `/think off` sends low reasoning for Grok 4.3 instead of disabling reasoning

- Kind: regression
- File: `extensions/xai/runtime-model-compat.ts`
- Line: 22
- Evidence: `XAI_REASONING_EFFORTS.off` is `null`. In `@mariozechner/pi-ai`, `getSupportedThinkingLevels` removes levels mapped to `null`, so `clampThinkingLevel(model, "off")` advances to the next available level. A focused payload probe with the committed map produced `{"reasoningOption":"off","payloadReasoning":{"effort":"low","summary":"auto"}}`.
- Impact: The provider profile advertises an `off` level at `extensions/xai/provider-policy-api.ts:15`, but using it does not disable Grok 4.3 reasoning. xAI docs describe `reasoning_effort` control and state that the default is low, so omitting a true “none” effort leaves users unable to turn reasoning off through OpenClaw’s `/think off` control. citeturn0view0
- Suggested fix: For xAI models that support configurable effort, map `off` to xAI’s disable value (`none`) instead of `null`, and add a test that captures the outgoing Grok 4.3 Responses payload for `/think off`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for payload construction: local payload probes show unsupported model ids receive `reasoning.effort`, and `/think off` on Grok 4.3 becomes low effort. Live provider rejection was not re-run because no `XAI_API_KEY` was present in the environment or mentioned in `~/.profile`.
- Is this the best way to solve the issue? Yes: keep the new Grok 4.3 support, but introduce a provider-owned configurable-effort capability separate from the existing reasoning-capable catalog hint.
