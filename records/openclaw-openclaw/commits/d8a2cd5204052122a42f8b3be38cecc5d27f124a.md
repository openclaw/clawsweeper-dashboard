# Commit d8a2cd5204

## Summary

Found one remaining verifier false positive in the hardlink path added by this commit. Current `main` already includes follow-up `77d1157618` to reject missing hardlink targets, but `backup verify` can still pass a tar where a hardlink target appears later in the archive, while normal extraction fails because the target does not exist yet.

## Findings

### Medium: `backup verify` accepts hardlinks whose targets appear later and cannot be extracted

- Kind: bug
- File: `src/commands/backup-verify.ts`
- Line: 274
- Evidence: current `main` checks only `entries.has(target.normalized)` after confirming the target stays under `archiveRoot`. It does not track tar stream order. A crafted archive with `manifest.json`, then `Link hardlink.txt -> target.txt`, then `target.txt` passes `backupVerifyCommand`, but `tar -xzf` exits with status 2: `Cannot hard link ... target.txt: No such file or directory`.
- Impact: `openclaw backup verify` can still certify an archive that broad extraction rejects. This preserves part of the failure mode from #54242 for malformed or externally produced archives, even though current `backup create` now dereferences hardlinks and should not emit this shape.
- Suggested fix: while collecting entries, keep normalized path, type, and stream index. For `Link` entries, require the normalized target to refer to an earlier regular file entry, or reject `Link` entries entirely now that OpenClaw-created backups dereference them.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes; a focused `node --import tsx` smoke generated a small gzip tar with the hardlink before its target. Current `backupVerifyCommand` passed it, then normal tar extraction failed with status 2.
- Is this the best way to solve the issue? yes; hardlink extraction uses an immediate filesystem link operation, so the verifier needs to validate extraction-suitable ordering and target type, not just target membership.
