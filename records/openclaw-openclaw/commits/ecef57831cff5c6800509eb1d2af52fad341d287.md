# Commit ecef578

## Summary

Found two concrete routing gaps in the macOS Voice Wake selected-session path. The fix works for the hard-coded channel enum cases, but it can still drop delivery for dynamically supported channels and lose the selected account for multi-account channel sessions.

## Findings

### Medium: Selected sessions on channels missing from the macOS enum are converted to WebChat

- Kind: bug
- File: `apps/macos/Sources/OpenClaw/VoiceWakeForwarder.swift`
- Line: 90
- Evidence: `forwardOptions` maps the selected session row’s `deliveryContext.channel` through `GatewayAgentChannel(rawValue:)` and falls back to `.webchat` on any unknown value. `GatewayAgentChannel` in `apps/macos/Sources/OpenClaw/GatewayConnection.swift:9` only includes a subset of built-in channels, while the gateway accepts bundled/registered channel IDs dynamically via `src/utils/message-channel-normalize.ts:23`. For a selected Matrix/LINE/Feishu/plugin channel session, this sets `channel = .webchat`; `shouldDeliver` then forces `deliver = false`.
- Impact: Voice Wake/Push-to-Talk still won’t deliver replies to selected sessions for valid channels outside the Swift enum, despite those sessions appearing in `sessions.list`.
- Suggested fix: Carry the raw channel string through `GatewayAgentInvocation` instead of a closed Swift enum, or add a dynamic/raw case whose delivery behavior treats non-`webchat` channels as deliverable and lets the gateway validate them.
- Confidence: high

### Medium: Multi-account selected sessions lose `accountId` during delivery

- Kind: bug
- File: `apps/macos/Sources/OpenClaw/VoiceWakeForwarder.swift`
- Line: 56
- Evidence: `SessionRouteEntry.DeliveryContext` decodes only `channel` and `to`, and `GatewayAgentInvocation` sends only `channel`/`to`. But `sessions.list` exposes `deliveryContext.accountId` (`src/gateway/server.sessions.store-rpc.test.ts:158`), and the agent RPC schema accepts `accountId` (`src/gateway/protocol/schema/agent.ts:148`). Because macOS passes `to`, gateway delivery becomes explicit; `resolveAgentDeliveryPlan` intentionally does not infer the session account in explicit mode (`src/infra/outbound/agent-delivery.ts:110`).
- Impact: Selecting a session from a non-default Telegram/Slack/WhatsApp/etc. account can send the voice-wake reply with `accountId` unset, so it may use the default account or fail to resolve the target.
- Suggested fix: Decode `deliveryContext.accountId`/`lastAccountId`, add `accountId` to `GatewayAgentInvocation`, and pass it through `sendAgent`. Alternatively, avoid turning session-derived `to` into an explicit delivery target when the selected session entry already has a complete delivery context.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. Both failures are visible through pure routing helpers: construct `VoiceWakeForwarder.forwardOptions` with a route entry whose channel is valid in the gateway but absent from `GatewayAgentChannel`, or with `deliveryContext.accountId = "work"` and observe that no Swift type/agent param can preserve that account.
- Is this the best way to solve the issue? The current approach is incomplete because it narrows gateway routing metadata at the macOS boundary. The safer fix is to pass the selected session’s raw delivery context through to the gateway.
