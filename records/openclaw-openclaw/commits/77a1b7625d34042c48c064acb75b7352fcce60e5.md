## Findings

### Medium: Gemini 3.1 Pro `medium` thinking is still rejected and downgraded

- Kind: bug
- File: `extensions/google/provider-policy.ts`
- Line: 193
- Evidence: `resolveGoogleThinkingProfile()` returns Pro levels as only `off`, `low`, `adaptive`, and `high`; the new test at `extensions/google/provider-policy-api.test.ts:179` locks in the same list for `gemini-3.1-pro-preview`. Google’s Gemini 3 developer guide lists `medium` as supported for Gemini 3.1 Pro, with the page last updated 2026-05-07. ([ai.google.dev](https://ai.google.dev/gemini-api/docs/gemini-3?hl=en))
- Impact: A cron or agent default of `thinkingDefault: "medium"` for `google/gemini-3.1-pro-preview` is treated as unsupported. The cron path at `src/cron/isolated-agent/run.ts:650` downgrades unsupported levels, and a current-source probe returned `mediumSupported: false` and `fallbackFromMedium: "adaptive"`.
- Suggested fix: Add `medium` to the Gemini 3.1 Pro profile, update `resolveGoogleGemini3ThinkingLevel()` in `src/plugin-sdk/provider-stream-shared.ts` so Pro `medium` maps to the provider `MEDIUM` level instead of `HIGH`, and add regression coverage for provider-policy, shared thinking validation, and cron preservation.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Running the current source resolver with catalog `{ provider:"google", id:"gemini-3.1-pro-preview", reasoning:false }` reports levels `["off","low","adaptive","high"]`, `mediumSupported: false`, and `fallbackFromMedium: "adaptive"`.
- Is this the best way to solve the issue? yes. The bug is in the provider-owned Gemini 3 thinking contract and the payload mapper; accepting `medium` in only one layer would either keep validation wrong or send the wrong upstream level.
