# Commit 5d2ee19

## Summary

Found one low-severity reliability regression in the new fast acknowledgement path. The status comment is created before the actual `clawsweeper_comment` dispatch, and unlike the existing reaction acknowledgement, that API call is not best-effort.

## Findings

### Low: Status-comment failure prevents the command router dispatch

- Kind: reliability
- File: `.github/workflows/clawsweeper-dispatch.yml`
- Line: 228
- Evidence: The workflow runs with `set -euo pipefail`. For OWNER/MEMBER/COLLABORATOR comments, it executes `status_response="$(GH_TOKEN="$TARGET_TOKEN" gh api "repos/$TARGET_REPO/issues/$ITEM_NUMBER/comments" --method POST ...)"`
  before building and posting the `repos/openclaw/clawsweeper/dispatches` payload at line 244. If the status comment POST exits non-zero, bash exits the step before the router dispatch. The older reaction path immediately above is guarded and continues on failure; the new status-comment path is not.
- Impact: A transient GitHub API failure, permission mismatch, secondary rate-limit/abuse response, or other comment-create failure can drop the exact command dispatch for trusted maintainers. At best, processing waits for the scheduled fallback; at worst, operators see an acknowledgement path fail without the command being routed.
- Suggested fix: Wrap the status comment creation in an `if status_response="$(...)"; then ... else echo "::warning::..."; fi` block and leave `status_comment_id` empty on failure so the workflow still dispatches `clawsweeper_comment`.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A local shell probe with a stubbed failing `gh api repos/$TARGET_REPO/issues/$ITEM_NUMBER/comments` exits before reaching the dispatch call under `set -euo pipefail`.
- Is this the best way to solve the issue? yes. The status comment is an acknowledgement optimization; dispatch should remain the durable action, matching the existing best-effort reaction behavior.
