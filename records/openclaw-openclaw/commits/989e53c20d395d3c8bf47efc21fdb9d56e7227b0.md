# Commit 989e53c

## Summary

Found one current-main security regression in the Android Gateway settings form: it can reuse and send a saved bootstrap credential when the user changes the manual endpoint.

## Findings

### Medium: Gateway settings can send a saved bootstrap token to a different endpoint

- Kind: security
- File: `apps/android/app/src/main/java/ai/openclaw/app/ui/SettingsScreens.kt`
- Line: 721
- Evidence: `GatewaySettingsScreen` builds the endpoint from the current form fields, but computes `bootstrapToken` with `.ifEmpty { bootstrapTokenInput.trim().ifEmpty { savedBootstrapToken } }` regardless of whether the host/port/TLS changed, then saves it and passes it into `viewModel.connect` at lines 744-751. The shared resolver used by the main connect flow explicitly preserves a saved bootstrap token only when the saved and requested manual endpoints match, and `GatewayConfigResolverTest` asserts the token is dropped when the endpoint changes.
- Impact: after pairing with one gateway, editing Gateway settings to another host with the Bootstrap field blank can disclose the old bootstrap credential to the new endpoint, or at minimum attempt unintended pairing/auth against the wrong gateway.
- Suggested fix: route `GatewaySettingsScreen` through `resolveGatewayConnectConfig`, or apply the same endpoint-match preservation rule before falling back to `savedBootstrapToken`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. Save a bootstrap token for one manual endpoint, open Settings > Gateway, change host/port/TLS, leave Bootstrap blank, and tap Save & Connect; the code passes the saved bootstrap token to the new `GatewayEndpoint.manual(...)`.
- Is this the best way to solve the issue? Yes. Reusing the existing resolver keeps Settings aligned with Connect/Onboarding and the existing endpoint-change test contract.
