# Commit 814b125

## Summary

Found one low-severity progress-draft lifecycle regression. Final replies still send normally, but a multi-assistant-message Telegram turn in `progress` mode can lose later progress draft updates after the first final reply clears the draft stream.

## Findings

### Low: Progress mode stops the draft stream for later assistant messages in the same turn

- Kind: bug
- File: `extensions/telegram/src/bot-message-dispatch.ts`
- Line: 908
- Evidence: `deliverProgressModeFinalAnswer` calls `answerLane.stream?.clear()` and then `resetDraftLaneState(answerLane)`. Telegram draft `clear()` uses `stopForClear`, which marks the stream stopped; stopped streams ignore future `update()` calls. The only reset that reopens the stream is `forceNewMessage()`, but after `resetDraftLaneState()` and message-id clearing, `rotateLaneForNewMessage()` returns early at `extensions/telegram/src/bot-message-dispatch.ts:580` without calling `forceNewMessage()`. Existing dispatch tests already model multiple assistant messages in one Telegram turn at `extensions/telegram/src/bot-message-dispatch.test.ts:881`, but the new progress-mode test covers only one final reply.
- Impact: If a provider or runtime emits another assistant message after a progress-mode final in the same inbound turn, later tool/progress updates cannot create a new editable progress draft. The final answer still uses normal delivery, so this is a visibility/progress regression rather than message loss.
- Suggested fix: After clearing a progress-mode draft for a final answer, either call `answerLane.stream?.forceNewMessage()` before marking the lane reusable, or make `rotateLaneForNewMessage()` reset stopped streams when `answerLane.finalized` came from progress-mode clear.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes from the existing unit harness shape: in `streamMode: "progress"`, emit tool progress, deliver final A, call `onAssistantMessageStart`, then emit another tool progress event before final B. Source shows the second progress update is ignored because the stream remains stopped. I did not add a new failing test because this was report-only.
- Is this the best way to solve the issue? Yes, narrowly reopening the draft stream after progress-final cleanup preserves the commit’s intended “clear draft, send final normally” behavior while restoring the existing multi-assistant-message lane lifecycle.
