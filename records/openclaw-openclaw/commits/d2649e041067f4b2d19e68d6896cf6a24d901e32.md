# Commit d2649e0410

## Summary

Found one current low-severity formatter regression in the added Telegram list-spacing preprocessor.

## Findings

### Low: Fence tracking can insert blank lines inside valid code blocks

- Kind: regression
- File: `extensions/telegram/src/format.ts`
- Line: 113
- Evidence: `preserveTelegramListBoundarySpacing` treats any line starting with either `` ``` `` or `~~~` as a fence toggle. Markdown-it, which `src/markdown/ir.ts` uses, closes fenced code only with the same marker character and at least the opening marker length. A live probe on current `main` with a backtick-opened code block containing a tilde line reproduced the mutation:
  `<pre><code>~~~ literal\n  • literal bullet\n\n3. literal number\n</code></pre>`
  The input code block only had one blank-free pair of code lines after the tilde line.
- Impact: Telegram rendering can corrupt copied code/log output by adding a blank line when code contains a fence-looking line of the other marker type, or a shorter same-marker fence inside a longer fence.
- Suggested fix: Track the active fence marker and opening length, and only clear fence state for a valid closing fence matching markdown-it/CommonMark rules. Reuse or mirror the existing Markdown fence parser if possible.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Run the formatter on:
  ```ts
  ["```", "~~~ literal", "  • literal bullet", "3. literal number", "```"].join("\n")
  ```
  Current output inserts an extra blank line before `3. literal number` inside the `<pre><code>` block.
- Is this the best way to solve the issue? yes. The bug is in the pre-parser fence state, so matching the actual Markdown fence close contract avoids changing list-spacing behavior outside code blocks.
