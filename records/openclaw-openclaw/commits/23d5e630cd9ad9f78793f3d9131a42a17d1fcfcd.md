# Commit 23d5e630cd

## Summary

Found one low-severity regression in the new parser error helpers: invalid non-string values can now throw a raw `TypeError` before the intended helpful parser error is built.

## Findings

### Low: Parser error helpers can throw `raw.trim is not a function`

- Kind: compatibility
- File: `src/cli/parse-duration.ts`
- Line: 18
- Evidence: `parseDurationMs` first normalizes `raw` through `normalizeOptionalString`, which treats non-strings as empty, but the new `invalidDuration(raw, "empty")` path calls `raw.trim()` directly. On current `main`, `pnpm tsx -e "import { parseDurationMs } from './src/cli/parse-duration.ts'; try { parseDurationMs(5 as never); } catch (err) { console.log(err instanceof Error ? err.message : String(err)); }"` prints `raw.trim is not a function`. The same pattern exists in `src/cli/parse-bytes.ts:22`.
- Impact: `parseDurationMs` is re-exported from the public plugin CLI runtime surface in `src/plugin-sdk/cli-runtime.ts:12`. A JavaScript plugin, or any unchecked config/CLI adapter path that passes a non-string, now gets an implementation `TypeError` instead of the parser’s actionable `Invalid duration... Use values like...` message. Before this commit, the same empty-normalization branch threw the parser-owned invalid-duration error.
- Suggested fix: Make the error helper tolerate `unknown` or coerce before trimming, for example `const display = typeof raw === "string" ? raw : String(raw ?? "")`, and apply the same defensive handling to `invalidByteSize`.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes: call `parseDurationMs(5 as never)` or `parseByteSize(5 as never)` from `pnpm tsx`; current `main` throws `raw.trim is not a function`.
- Is this the best way to solve the issue? yes: keep the improved examples, but make the display formatting helper robust so invalid inputs still produce parser-owned errors.
