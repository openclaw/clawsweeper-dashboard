# Commit 51d3ec7

## Summary

Found one actionable issue in the recovery predicate added by this commit. The new test models the compiled-runtime config issue as path `plugins`, but the real config validator scopes plugin diagnostics with a plugin id to `plugins.entries.<id>`, so the official plugin recovery path can still reject the source-only shadow it is meant to repair.

## Findings

### Medium: compiled-runtime recovery misses plugin-scoped validation issues

- Kind: bug
- File: `src/cli/plugins-install-command.ts`
- Line: 438
- Evidence: `isAllowedPluginRecoveryIssue` only allows the new compiled-runtime case when `issue.path === "plugins"`. But config validation maps registry diagnostics with `diag.pluginId` to `plugins.entries.${diag.pluginId}` in `src/config/validation.ts:817`, and source-only package discovery supplies a plugin id hint from the manifest/package name in `src/plugins/discovery.ts:723`. The official Discord package has package metadata/channel id `discord` in `extensions/discord/package.json:31`, so a compiled-runtime diagnostic for Discord is expected to become `plugins.entries.discord`, not bare `plugins`. A focused `tsx` probe against `validateConfigObjectWithPlugins` confirmed that an error diagnostic with `pluginId: "discord"` produces path `plugins.entries.discord`, while only diagnostics without a plugin id produce `plugins`.
- Impact: `openclaw plugins install npm:@openclaw/discord --force` can still fail before reinstall with “Config invalid outside the plugin recovery path for discord” when the compiled-runtime issue is plugin-scoped, leaving the source-only install shadow unrepaired. The added unit test does not catch this because it hand-builds the issue path as `plugins`.
- Suggested fix: allow the same-plugin scoped path, for example `plugins.entries.${pluginId}`, for the compiled-runtime diagnostic. If retaining bare `plugins`, add an identity check through diagnostic/plugin source data where available so unrelated plugin failures do not become recoverable.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Partially. The validator path mismatch is deterministic and was reproduced with a focused `tsx` probe. I did not produce a full CLI state fixture for a persisted source-only package shadow.
- Is this the best way to solve the issue? Yes: match the validator’s plugin-scoped path for the requested plugin, rather than relying only on the unscoped `plugins` path.
