# Commit a38c2c2

## Summary

Found one diagnostic regression in the new builtin semantic-vector readiness split.

## Findings

### Medium: Semantic vectors can be reported ready when the embedding provider ping failed

- Kind: bug
- File: `extensions/memory-core/src/memory/manager.ts`
- Line: 841
- Evidence: `runMemoryStatus` probes embeddings at `extensions/memory-core/src/cli.runtime.ts:693` and then probes semantic vectors at `extensions/memory-core/src/cli.runtime.ts:697`. `probeEmbeddingAvailability()` correctly catches live embedding failures from `embedBatchWithRetry(["ping"])` and returns `{ ok: false }` at `extensions/memory-core/src/memory/manager.ts:894`, but `probeVectorAvailability()` only checks that a provider object exists and that the vector store loads before setting `this.vector.semanticAvailable = ready` at `extensions/memory-core/src/memory/manager.ts:841`. Remote embedding providers are created before the live request and only fail when `embedBatch` calls `/embeddings` (`packages/memory-host-sdk/src/host/embeddings-remote-provider.ts:26`), so a bad/expired API key can produce `Embeddings: unavailable` and `Semantic vectors: ready` in the same deep status result.
- Impact: `openclaw memory status --deep` and `--json` can falsely mark semantic vector search ready during provider/auth outages. That defeats the commit’s new diagnostic contract and can mislead users or scripts into treating semantic recall as healthy when query embedding would fail.
- Suggested fix: make builtin `probeVectorAvailability()` require embedding readiness as well as store readiness, preferably by consulting the cached embedding probe result or calling `probeEmbeddingAvailability()` before setting `semanticAvailable`. Keep `probeVectorStoreAvailability()` as the store-only check.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. Configure builtin memory with sqlite-vec loadable and a remote embedding provider that constructs from a present but invalid/expired API key, then run `openclaw memory status --deep --json`. The embedding probe will return false after the live `/embeddings` call, while the later vector probe can still set `vector.semanticAvailable` true if sqlite-vec loads.
- Is this the best way to solve the issue? Yes. The new `storeAvailable` field should remain store-only, but `semanticAvailable` should be derived from both embedding-provider readiness and store readiness.
