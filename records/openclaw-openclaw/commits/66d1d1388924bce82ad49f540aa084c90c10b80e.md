# Commit 66d1d13889

## Summary

Found one low-severity regression: the new expected service PATH is threaded into doctor, but not into `openclaw status`, so status can falsely report a newly installed healthy service as non-standard when the active `openclaw` bin lives in pnpm/asdf/fnm-style paths.

## Findings

### Low: `openclaw status` can warn on service PATHs this commit now intentionally installs

- Kind: bug
- File: `src/cli/daemon-cli/status.gather.ts`
- Line: 515
- Evidence: The commit adds the active CLI bin directory to generated service PATH via `resolveDaemonServicePathDirs` in `src/commands/daemon-install-helpers.ts:550`, and doctor avoids false positives by passing `expectedServicePath: expectedPlan.environment.PATH` at `src/commands/doctor-gateway-services.ts:416`. But `openclaw status` still calls `auditGatewayServiceConfig({ env: process.env, command })` without that expected PATH. The audit only suppresses package-manager-looking PATH entries when they are present in `normalizedExpected`; otherwise `isNonMinimalServicePathEntry` flags entries such as `/Users/testuser/Library/pnpm`.
- Impact: A service installed after this fix can work correctly and pass doctor, but `openclaw status` can print ‚ÄúService config looks out of date or non-standard‚Äù and recommend doctor for the same expected PATH. This is especially likely for pnpm/asdf/fnm installs, which the issue/PR explicitly intends to support.
- Suggested fix: Make status build or reuse the same expected gateway service plan before auditing, then pass `expectedServicePath`, or move the ‚Äúactive OpenClaw bin is allowed‚Äù decision into the audit inputs shared by doctor and status.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused `node --import tsx` audit check with service PATH `/opt/homebrew/opt/node/bin:/Users/testuser/Library/pnpm:...` produced `gateway-path-nonminimal` without `expectedServicePath`, and no issues with the same expected path.
- Is this the best way to solve the issue? yes, the status path should use the same expected service PATH contract as doctor instead of independently falling back to the older minimal PATH baseline.
