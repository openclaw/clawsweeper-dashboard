# Commit 2cb03ee

## Summary

Found one actionable regression: the new `openai-codex` audio provider advertises batch transcription, but the runtime resolves `openai-codex` OAuth credentials and sends them to the public OpenAI `/v1/audio/transcriptions` API. That API is documented as API-key authenticated, so the advertised Codex-only path is likely to fail for the users this change targets.

## Findings

### Medium: Codex audio uses Codex OAuth against the OpenAI API-key transcription endpoint

- Kind: bug
- File: `extensions/openai/media-understanding-provider.ts`
- Line: 21
- Evidence: `transcribeOpenAiCodexAudio()` sets `provider: "openai-codex"` but keeps `defaultBaseUrl: "https://api.openai.com/v1"`; the shared helper then posts to `${baseUrl}/audio/transcriptions` with `Authorization: Bearer ${params.apiKey}`. The audio runner resolves credentials for the entry provider id, so an auto-selected Codex entry calls `resolveApiKeyForProvider({ provider: "openai-codex", ... })`. The manifest only exposes `OPENAI_API_KEY` for `openai`; `openai-codex` auth choices are browser/device OAuth. OpenAI’s API docs state the platform API uses API keys for Bearer auth, and the speech-to-text guide shows `/v1/audio/transcriptions` called with `Authorization: Bearer $OPENAI_API_KEY`. ([platform.openai.com](https://platform.openai.com/docs/api-reference/authentication?api-mode=responses))
- Impact: A user whose active model is `openai-codex/gpt-5.5` and who only has Codex OAuth configured will now auto-select `openai-codex` for inbound audio, but the transcription request is sent to the OpenAI Platform endpoint with a Codex OAuth token. That likely returns an auth failure and also prevents fallback to later audio providers because auto resolution returns a single provider entry.
- Suggested fix: Either do not advertise `openai-codex` batch audio until there is a valid Codex transcription backend, or route this path through `openai`/`OPENAI_API_KEY` auth and only select it when direct OpenAI API auth is available. If Codex OAuth is intended to support transcription, add a provider-specific backend analogous to the existing Codex image route instead of using `api.openai.com`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes: configure only an `openai-codex` OAuth profile, use active model `openai-codex/gpt-5.5`, and process an inbound audio attachment. The code path selects `openai-codex`, resolves Codex credentials, and posts to `https://api.openai.com/v1/audio/transcriptions`.
- Is this the best way to solve the issue? The safest fix direction is to keep provider id, credential type, and endpoint family aligned. The current OpenAI image docs/code already distinguish Codex OAuth image routing through the Codex Responses backend from direct OpenAI API-key routing.
