# Commit efc8641

## Summary

Found three actionable issues: one medium docs/runtime mismatch in the BlueBubbles migration path, one low CLI fallback bug for invalid filters, and one low compatibility risk in the generated Swift protocol init.

## Findings

### Medium: Stopped-Gateway migration probe cannot produce live iMessage probe results

- Kind: bug
- File: `docs/channels/imessage-from-bluebubbles.md`
- Line: 192
- Evidence: The guide says to run `openclaw gateway stop`, enable `channels.imessage`, then run `openclaw channels status --probe --channel imessage` expecting `imessage.privateApi.available: true`. But `channels status --probe` first calls Gateway RPC, and on failure falls back to config-only output; the CLI docs also state live probe/audit runs only on a reachable Gateway.
- Impact: Migrating users cannot actually verify the `imsg` private API bridge in the documented stopped-Gateway state, so they may cut over without catching permission, install, or private API failures.
- Suggested fix: Update the workflow to run a Gateway with only iMessage enabled before probing, or implement a real local/offline iMessage probe path before documenting this command as stopped-Gateway proof.
- Confidence: high

### Low: Unknown channel filters are silently dropped on config-only fallback

- Kind: bug
- File: `src/commands/channels/status.ts`
- Line: 210
- Evidence: `normalizeChannelId(opts.channel)` returns `null` for an unknown channel. The Gateway handler returns `INVALID_REQUEST` for that same unknown channel, but `channelsStatusCommand` catches all errors as “Gateway not reachable” and the fallback filter uses `!requestedChannel || ...`, so a misspelled channel becomes unfiltered config-only output.
- Impact: `openclaw channels status --channel imesage --json` can report all configured channels instead of failing the bad filter, masking operator typos and producing misleading status output.
- Suggested fix: Validate `opts.channel` before the Gateway call/fallback, or only enter config-only fallback for transport errors and preserve Gateway `INVALID_REQUEST` errors.
- Confidence: high

### Low: Generated Swift params init is source-breaking for existing clients

- Kind: compatibility
- File: `apps/shared/OpenClawKit/Sources/OpenClawProtocol/GatewayModels.swift`
- Line: 3632
- Evidence: `ChannelsStatusParams` changed its public initializer from `init(probe:timeoutms:)` to `init(probe:timeoutms:channel:)`, and the new optional `channel` parameter has no default.
- Impact: Existing Swift clients using `ChannelsStatusParams(probe:timeoutms:)` no longer compile after an additive wire-protocol change.
- Suggested fix: Add a backward-compatible overload or have the Swift generator emit `= nil` defaults for optional initializer parameters.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the docs/runtime and invalid-filter issues from source inspection; the Swift issue is reproducible by compiling an existing two-argument initializer call against the new generated model.
- Is this the best way to solve the issue? Yes for the docs and CLI validation fixes; for Swift, an overload is the narrowest fix, while generator-level optional defaults may be better if source compatibility is expected broadly.
