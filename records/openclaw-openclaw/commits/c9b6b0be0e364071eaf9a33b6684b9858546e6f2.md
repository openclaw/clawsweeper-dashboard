# Commit c9b6b0b

## Summary

Found two actionable regressions in the Mantis evidence publisher change: artifact comments now list filenames that are not actually present in uploaded Actions artifacts for existing workflows, and an adjacent Mantis test still asserts the removed inline rendering behavior.

## Findings

### Medium: Artifact comments list `targetPath` names that workflows no longer materialize

- Kind: bug
- File: `scripts/mantis/publish-pr-evidence.mjs`
- Line: 123
- Evidence: `renderArtifactFileList` now prints `artifact.targetPath`, but the commit removed the only code that copied artifacts into those `targetPath` names. Existing workflows still upload the original output directory directly, for example `mantis-discord-status-reactions` declares `path: "baseline/discord-status-reactions-tool-only-timeline.png"` with `targetPath: "baseline.png"` at `.github/workflows/mantis-discord-status-reactions.yml:494`, then uploads `steps.run_mantis.outputs.output_dir` at `.github/workflows/mantis-discord-status-reactions.yml:528`. `actions/upload-artifact@v4` uses the search path as the artifact root for a single directory upload, so it does not rename that file to `baseline.png`.
- Impact: Mantis PR comments become misleading: the bundle link is correct, but the file list points maintainers at filenames such as `baseline.png`, `candidate.png`, and `baseline-desktop.mp4` that are not in the uploaded artifact for several workflows. Since the commit intentionally removed inline images, this file list is now the primary map for inspecting screenshots/videos.
- Suggested fix: Either stage/copy the declared manifest artifacts into an upload bundle rooted at `targetPath` before `actions/upload-artifact`, or render the actual uploaded relative `artifact.path` values and update the schema/docs so `targetPath` is not presented as the artifact bundle path.
- Confidence: high

### Low: Telegram desktop proof test still expects removed inline tables

- Kind: regression
- File: `test/scripts/mantis-build-telegram-desktop-proof-evidence.test.ts`
- Line: 97
- Evidence: The test still calls `renderEvidenceComment` with removed `artifactRoot`, `rawBase`, and `treeUrl` fields and expects `<th width="50%">Main</th>` plus inline image markup. Current `renderEvidenceComment` ignores those fields and renders only an Actions artifact link plus file list. A focused Node probe reproduced the mismatch and failed with: `Current renderEvidenceComment output no longer contains the table expected by test/scripts/mantis-build-telegram-desktop-proof-evidence.test.ts:97`.
- Impact: The script test suite has a stale expectation after this commit and will fail once this adjacent test is run in CI or by changed-test selection.
- Suggested fix: Update this test to assert the new artifact-only comment contract, or restore inline rendering through a non-git artifact-safe mechanism if that behavior is still required.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The filename mismatch is visible by comparing the workflow manifest `path`/`targetPath` pairs with direct directory upload, and a focused Node probe showed the comment rendering `baseline.png` while the manifest source remained `baseline/actual-name.png`.
- Is this the best way to solve the issue? unclear. The safest direction depends on the intended contract: keep `targetPath` by staging files before upload, or remove `targetPath` from the user-facing artifact list and document actual uploaded paths.
