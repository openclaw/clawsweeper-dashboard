# Commit 5d01be1

## Summary

Found one low-severity UI regression in the new chat picker search flow. Server-side search and pagination work, but live `sessions.changed` events can repopulate a searched picker with sessions that do not match the active query.

## Findings

### Low: Live session events bypass the active chat picker search

- Kind: bug
- File: `ui/src/ui/controllers/sessions.ts`
- Line: 376
- Evidence: `createChatSessionsLoadOverrides` includes the applied search query in `sessions.list` requests (`ui/src/ui/app-chat.ts:114`), and the search controls set `chatSessionSearchAppliedQuery` before refreshing (`ui/src/ui/chat/session-controls.ts:133`). After that, `applySessionsChangedEvent` accepts any event with a reliable source, including a `sessionId`, and inserts missing rows into `state.sessionsResult` at `ui/src/ui/controllers/sessions.ts:419` without checking the active search. A one-off repro with `chatSessionSearchAppliedQuery: "telegram"` and an incoming `sessions.changed` payload for `"Slack chat"` produced `{"result":{"applied":true,"change":"inserted"},"keys":["agent:main:slack","agent:main:telegram"]}`.
- Impact: A user can search the chat session picker for one conversation, then unrelated session updates from the gateway can appear in that filtered picker until the next full refresh. This makes the search result set unstable and can make the new bounded picker show non-matching sessions.
- Suggested fix: Keep the active session-list query/search in the controller state used by `applySessionsChangedEvent`, and reject inserted event rows that do not match it. Existing rows can still be updated, or removed if the updated row no longer matches the active filter.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Load a searched `sessionsResult`, then call `applySessionsChangedEvent` with a non-matching event payload that includes `sessionId`; the controller inserts it because the event is considered reliable.
- Is this the best way to solve the issue? yes. The bug is in the client-side incremental merge path, not the server pagination path; the merge needs the same active filter contract as the list request or it should trigger a filtered reload.
