# Commit a24d5fe

## Summary

Found one regression in the cached channel-probe path. Configs that have both a configured channel and a material plugin entry can now skip channel auto-enable candidate generation.

## Findings

### Medium: Plugin config entries suppress channel auto-enable candidates

- Kind: regression
- File: `src/config/plugin-auto-enable.shared.ts`
- Line: 495
- Evidence: `resolvePluginAutoEnableReadiness()` returns `{ mayNeedAutoEnable: true, configuredChannelIds: [] }` before probing channels when `hasPluginAllowlistWithMaterialEntries()` or `hasConfiguredPluginConfigEntry()` is true. `detectPluginAutoEnableCandidates()` then passes that empty array through, and `resolveConfiguredPluginAutoEnableCandidates()` uses `params.configuredChannelIds ?? collectConfiguredChannelIds(...)`, so `[]` is treated as authoritative and channel collection never runs. In the parent commit, `resolveConfiguredPluginAutoEnableCandidates()` always collected channel ids itself after readiness passed.
- Impact: mixed configs can stop materializing configured channels into the effective auto-enabled config. Focused smoke: with `channels.slack.botToken` plus `plugins.entries.firecrawl.config.webFetch` under `plugins.allow: ["telegram"]`, detection returned only the Firecrawl candidate, left `channels.slack.enabled` unset, and left `plugins.allow` as `["telegram", "firecrawl"]`. The same Slack config without the plugin entry returned the Slack candidate, set `channels.slack.enabled: true`, and appended `"slack"` to `plugins.allow`. Gateway startup persists auto-enable changes from this shared API, so the dropped channel candidate is a real config-materialization regression.
- Suggested fix: distinguish “channel ids were probed and none were found” from “channel ids were not probed yet”. For example, make `configuredChannelIds` optional in the readiness result and only pass it when `collectConfiguredChannelIds()` actually ran, or compute channel ids before every early `true` return.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The inline `pnpm exec tsx -e ...` smoke against `applyPluginAutoEnable()` reproduced the dropped Slack candidate with a combined Slack + Firecrawl config, and the control case without the plugin entry still auto-enabled Slack.
- Is this the best way to solve the issue? yes. The optimization should cache channel probes only when performed; an empty array cannot safely represent both “none found” and “not checked”.
