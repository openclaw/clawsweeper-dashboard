# Commit 530b892

## Summary

Found one actionable issue. The new `channel:<channelId>:<senderId>` sender-key support works in the generic resolver, but several plugin-owned group policy adapters still call `resolveToolsBySender` without a channel/provider value, so channel-scoped group policies silently miss.

## Findings

### Medium: Channel-scoped group sender policies are ignored in plugin-owned group adapters

- Kind: security
- File: `extensions/discord/src/group-policy.ts`
- Line: 69
- Evidence: `src/config/group-policy.ts:272` only checks the channel bucket when `params.messageProvider` normalizes to a channel. The Discord, Slack, and Microsoft Teams group adapters call `resolveToolsBySender` with sender fields only, omitting any channel value at `extensions/discord/src/group-policy.ts:69`, `extensions/slack/src/group-policy.ts:55`, and `extensions/msteams/src/policy.ts:116`. The docs now advertise `channel:<channelId>:<senderId>` for group `toolsBySender` at `docs/channels/groups.md:365`.
- Impact: A group config such as `tools: { allow: ["read", "exec"] }` plus `toolsBySender: { "channel:discord:guest": { deny: ["exec"] } }` will not apply the deny for that Discord sender. The adapter falls back to the broader group `tools`, leaving tools visible that the admin tried to remove for that sender. Wildcard policies can also be applied instead of the intended channel-specific override.
- Suggested fix: Thread a verified channel id through `ChannelGroupContext`, or have each built-in adapter pass its canonical channel id to `resolveToolsBySender` (`messageProvider: "discord"`, `"slack"`, `"msteams"`, etc.). Add regression tests for at least one plugin adapter with `channel:<channelId>:<senderId>` plus a conflicting wildcard/group fallback.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused `tsx` probe showed `resolveToolsBySender({ messageProvider: "discord", senderId: "alice", toolsBySender: { "channel:discord:alice": { allow: ["exec"] }, "*": { deny: ["write"] } } })` returns `{"allow":["exec"]}`, while `resolveDiscordGroupToolPolicy` with the same `toolsBySender` returns the wildcard `{"deny":["write"]}` because the adapter never passes the provider.
- Is this the best way to solve the issue? yes. The generic matcher already has the correct channel-bucket behavior; the missing contract is getting the canonical channel id into plugin group policy calls.
