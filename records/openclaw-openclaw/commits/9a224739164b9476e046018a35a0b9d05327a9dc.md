# Commit 9a22473916

## Summary

Found one regression risk in the new stalled embedded-run recovery window: the abort threshold is measured from diagnostic session-state age, not the last observed run progress age.

## Findings

### Medium: Stalled-run abort can fire minutes after recent diagnostic run progress

- Kind: reliability
- File: `src/logging/diagnostic.ts`
- Line: 367
- Evidence: `isStalledEmbeddedRunRecoveryEligible` compares `params.ageMs` against the extended abort threshold, and the heartbeat passes `ageMs = now - state.lastActivity` from `src/logging/diagnostic.ts:834`. But `run.progress` updates only the diagnostic activity snapshot in `src/logging/diagnostic-run-activity.ts:214`, not `state.lastActivity`; Codex app-server emits these progress events in `extensions/codex/src/app-server/run-attempt.ts:789`. A focused smoke check with a mocked heartbeat emitted `run.progress` at 9 minutes and then recovered at 10.5 minutes, only 90 seconds after that progress: `allowActiveAbort: true`.
- Impact: The docs and changelog describe abort-draining after an extended no-progress window, but a run that recently emitted `run.progress` can still be abort-drained as soon as the older session-state age crosses 10 minutes/5x threshold. That can prematurely kill a slow but still progressing embedded diagnostic/Codex app-server turn and start queued work over a stale active task.
- Suggested fix: Base stalled embedded-run recovery on the activity no-progress age, e.g. require `lastProgressAgeMs >= resolveStalledEmbeddedRunAbortMs(...)`, or make `run.progress` refresh the same session-state activity used for the abort window. Add a regression test where `run.progress` occurs shortly before the 10-minute recovery boundary.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. I ran a no-file-change TSX smoke that patched the heartbeat callback, started an embedded run with `stuckSessionWarnMs: 30_000`, emitted `run.progress` at 9 minutes, advanced to 10.5 minutes, and observed recovery called with `allowActiveAbort: true`.
- Is this the best way to solve the issue? yes. The recovery threshold should use the same “no recent progress” clock that classifies the run as stalled; otherwise the new active abort path can contradict the liveness contract.
