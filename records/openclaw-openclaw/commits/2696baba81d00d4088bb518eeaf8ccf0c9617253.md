# Commit 2696bab

## Summary

Found one plausible regression in the Telegram supersession fence: read-only/side-channel text commands that intentionally bypass the busy Telegram lane can now suppress the in-flight main reply.

## Findings

### Medium: Control-lane text commands can suppress the main Telegram reply

- Kind: regression
- File: `extensions/telegram/src/bot-message-dispatch.ts`
- Line: 190
- Evidence: The current supersession predicate returns true for every non-abort dispatch, and `dispatchTelegramMessage` begins the per-session reply fence with that value. Telegram’s sequential key intentionally lets read-only status commands and `/btw` bypass the busy per-topic lane (`extensions/telegram/src/sequential-key.ts:30`, `extensions/telegram/src/sequential-key.ts:74`, `extensions/telegram/src/sequential-key.ts:80`). Text-message handling still routes accepted messages through `dispatchTelegramMessage` (`extensions/telegram/src/bot-message.ts:114`), and `/btw` is explicitly a side question command that should leave the main run alone (`docs/tools/btw.md:9`, `docs/tools/btw.md:29`, `src/auto-reply/commands-registry.shared.ts:269`).
- Impact: If native Telegram commands are disabled or a command reaches the text-message path, a user can send `/status`, `/context`, or `/btw ...` while a normal answer is in flight. Because those commands use a control/side lane but share the same session fence key, the command increments the reply generation and the older main answer is treated as superseded, so the user gets the command reply but loses the answer they were waiting for.
- Suggested fix: Do not supersede the Telegram reply fence for text commands that are explicitly control/read-only or side-result commands. Reuse/share the same command classification used by `getTelegramSequentialKey`, or make supersession depend on whether the inbound turn can start/replace a normal assistant turn rather than simply “non-abort”.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes at source level: start an in-flight text-path Telegram dispatch for a session, then dispatch `/btw ...` or a read-only status command for the same session before the first finalizes; the second dispatch increments the fence generation and the first dispatch’s final delivery path returns early as superseded.
- Is this the best way to solve the issue? Yes directionally: the new fence is useful for ordinary newer user turns, but control/side commands need a non-superseding classification so they can interleave without canceling the main answer.
