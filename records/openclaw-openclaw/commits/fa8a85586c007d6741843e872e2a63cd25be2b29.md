# Commit fa8a855

## Summary

Found two concrete release-publish regressions in the new GitHub release creation step. The first currently breaks release note extraction for the repo’s `CHANGELOG.md` size; the second breaks prerelease note matching for historical beta changelog headings.

## Findings

### Medium: GitHub Contents API returns empty content for the current changelog size

- Kind: regression
- File: `.github/workflows/openclaw-release-publish.yml`
- Line: 277
- Evidence: The workflow fetches `CHANGELOG.md` with `gh api .../contents/CHANGELOG.md?ref=${TARGET_SHA} --jq '.content' | base64 --decode`. The file is currently about 1.96 MB, and a live API check for this reviewed SHA returned `{"size":1961487,"encoding":"none","contentLen":0}`. Running the same decode path produced `api-decoded changelog bytes=0 notes bytes=0`, so line 284 exits with `CHANGELOG.md does not contain release notes...`. GitHub’s REST Contents API docs say 1-100 MB files require the raw or object custom media types, and with object media type the `content` field is empty and `encoding` is `none`. citeturn4search0
- Impact: After the OpenClaw npm publish child succeeds, the parent release workflow will fail while creating the GitHub release page. That leaves a published package with a failed release orchestration run and no generated GitHub release notes.
- Suggested fix: Fetch the raw file instead of `.content | base64 --decode`, for example `gh api -H 'Accept: application/vnd.github.raw+json' "repos/${GITHUB_REPOSITORY}/contents/CHANGELOG.md?ref=${TARGET_SHA}" > "${changelog_file}"`, or use another API path that returns the full blob for large files.
- Confidence: high

### Medium: Beta tags look for the stable heading instead of the beta changelog heading

- Kind: regression
- File: `.github/workflows/openclaw-release-publish.yml`
- Line: 270
- Evidence: The new regex rewrites `vYYYY.M.D-beta.N` to `YYYY.M.D` before running the `awk` section match. Historical release tags in this repo keep beta-specific headings: at their tagged commits, `v2026.4.19-beta.2`, `v2026.4.1-beta.1`, `v2026.3.31-beta.1`, and `v2026.3.24-beta.2` each have `## <full-beta-version>` headings. Simulating the reviewed extraction logic against those tagged changelogs produced zero-byte notes for all four because it searched for the unsuffixed stable heading.
- Impact: Even after fixing the large-file fetch, beta release publishes whose changelog section is named for the actual beta version will fail after npm publish and skip GitHub release creation.
- Suggested fix: Match `release_version` first for prereleases and stable correction releases, then optionally fall back to the base stable version only if that is an intentional supported changelog shape.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. For the large-file issue, `gh api repos/openclaw/openclaw/contents/CHANGELOG.md?ref=fa8a85586c007d6741843e872e2a63cd25be2b29 --jq '{size,encoding,contentLen:(.content|length)}'` returns empty content, and the workflow pipeline writes empty notes. For the beta issue, running the added `notes_version` rewrite plus `awk` extraction against historical beta tags yields empty release notes.
- Is this the best way to solve the issue? yes for the direction: use a raw/full-file fetch for `CHANGELOG.md`, and preserve exact tag-version matching before any fallback.
