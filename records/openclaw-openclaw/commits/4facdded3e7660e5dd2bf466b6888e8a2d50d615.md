# Commit 4facdde

## Summary

Found one low-severity bug in the new repo-owned Telegram user driver: `probe --thread-id` waits in the requested thread but sends the probe message outside that thread.

## Findings

### Low: `probe --thread-id` does not send the message to the requested thread

- Kind: bug
- File: `scripts/e2e/telegram-user-driver.py`
- Line: 669
- Evidence: `send` forwards `args.thread_id` to `send_text` at `scripts/e2e/telegram-user-driver.py:638`, but `probe` calls `driver.send_text(chat_id, text, args.reply_to)` at `scripts/e2e/telegram-user-driver.py:669` while still passing `thread_id=args.thread_id` into `wait_for_message` at `scripts/e2e/telegram-user-driver.py:674`. A focused monkeypatched invocation with `thread_id=777` showed `send_text_args=(123, 'hi', None, 0)` and `wait_thread_id=777`.
- Impact: In topic-enabled Telegram groups, `python3 scripts/e2e/telegram-user-driver.py probe --thread-id <topic>` can send the prompt to the main chat, then wait for a bot reply in the topic and time out even if the bot responds correctly to the actual message.
- Suggested fix: Change the probe call to `driver.send_text(chat_id, text, args.reply_to, args.thread_id)` and add a small test or monkeypatched unit check for the probe thread-id path.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes; the static call path is direct, and a monkeypatched `command_probe` invocation confirmed the send path receives `thread_id=0` while the wait path receives the requested thread id.
- Is this the best way to solve the issue? yes; `send_text` already supports `thread_id`, and `command_send` already uses that parameter correctly.
