# Commit b55d9fa

## Summary

The commit correctly adds context-engine metadata to Codex app-server bindings and rotates incompatible native Codex threads, but the new fresh-thread retry path has one cleanup regression when the retry cannot start a new thread.

## Findings

### Medium: Fresh-thread retry startup failures bypass turn-start cleanup

- Kind: reliability
- File: `extensions/codex/src/app-server/run-attempt.ts`
- Line: 1499
- Evidence: After a resumed context-engine thread hits a context-window error, the new retry branch clears the binding and awaits `restartContextEngineCodexThread()` before entering the existing `turn === undefined` failure block. If that `thread/start` path throws, execution skips the cleanup/reporting block at lines 1510-1567. The normal post-turn `finally` at lines 1806-1844 is only reached after a turn is accepted, so it does not cover this pre-turn retry failure.
- Impact: A transient `thread/start` failure during the retry can leave notification/request handlers and the native hook relay registered, skip `turn_start_failed`, skip `agent_end`/LLM output hooks, and skip trajectory flush for that failed attempt.
- Suggested fix: Include `restartContextEngineCodexThread()` in the retry `try/catch` and assign any restart failure to `turnStartError`, leaving `turn` undefined so the existing turn-start failure cleanup path runs.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Use the existing app-server test harness with a compatible context-engine binding, make `thread/resume` return `thread-old`, make the first `turn/start` throw a context-window error, then make retry `thread/start` throw; source control flow shows the throw escapes before the failure cleanup block.
- Is this the best way to solve the issue? yes. The surrounding code already has one canonical pre-turn failure cleanup path, so catching restart failure and routing it through that path is smaller and safer than duplicating cleanup.
