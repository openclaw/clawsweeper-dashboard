# Commit f9c0dc2

## Summary

Found one medium-confidence regression risk in the new Feishu fallback guard. The fallback is meant to be limited to normal group quoted replies, but the dispatcher derives that from only `threadReply/rootId/replyToMessageId`, which can also match native-topic-looking inbound metadata when the group is not configured with a topic session scope.

## Findings

### Medium: Top-level fallback can apply to default-scoped Feishu topic replies

- Kind: regression
- File: `extensions/feishu/src/reply-dispatcher.ts`
- Line: 150
- Evidence: `allowTopLevelReplyFallback` is true whenever `threadReply` is true, `rootId` exists, and `sendReplyToMessageId !== rootId`. The dispatcher does not know whether the inbound event came from a normal group quoted reply or a Feishu native topic. In the real bot path, default group scope chooses `replyTargetMessageId = ctx.replyTargetMessageId ?? ctx.messageId` unless the group is configured as `group_topic`/`group_topic_sender` or has explicit `replyInThread`, while still passing `threadReply` and `rootId` into the dispatcher. That means a `topic_group`/thread event handled under default `group` scope can satisfy the new fallback condition and, on withdrawn/not-found reply failure, `send.ts` falls back to `message.create` into the chat instead of preserving the hard failure.
- Impact: A failed native-topic reply can be degraded into a top-level group message, leaking thread-local context into the main group view and contradicting the commit’s stated native topic safety goal.
- Suggested fix: Move the opt-in decision to the bot routing layer or pass enough context into the dispatcher to require a known normal group quote, for example `chatType === "group"` plus default non-topic session scope, rather than inferring only from distinct root/reply IDs.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Unclear. The source path is concrete, but I did not run a live Feishu delivery. A focused regression test could construct a `chat_type: "topic_group"` event with default group scope, `root_id`, and a child `message_id`, then assert the dispatcher does not pass `allowTopLevelReplyFallback: true`.
- Is this the best way to solve the issue? Yes for the fix direction: the fallback policy needs source-context proof from inbound routing, not just derived message-id shape.
