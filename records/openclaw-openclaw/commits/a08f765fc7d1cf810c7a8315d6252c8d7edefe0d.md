## Findings

### Low: Parent assertions now pass when the expected span is missing

- Kind: test_gap
- File: `extensions/diagnostics-otel/src/service.test.ts`
- Line: 1962
- Evidence: The prior `toMatchObject({ "openclaw.run": undefined })` assertion failed if the key was absent. A Vitest probe confirmed `expect({}).toMatchObject({ a: undefined })` fails, while `expect({}.a).toBeUndefined()` passes. The new assertions at `extensions/diagnostics-otel/src/service.test.ts:1962`, `extensions/diagnostics-otel/src/service.test.ts:2004`, and `extensions/diagnostics-otel/src/service.test.ts:2060` only read `parentBySpanName[...]`, so a regression that stops creating `openclaw.run`, `openclaw.model.call`, or `openclaw.tool.execution` spans would still satisfy these checks. The production paths currently create those spans in `extensions/diagnostics-otel/src/service.ts:1692`, `extensions/diagnostics-otel/src/service.ts:1980`, and `extensions/diagnostics-otel/src/service.ts:2104`.
- Impact: These tests are intended to guard lifecycle span parenting for completed-only trusted diagnostics and untrusted injected trace ids. With the new matcher shape, they can miss a concrete OTEL tracing regression where the span is absent instead of merely unparented.
- Suggested fix: Assert both presence and undefined parent, for example `expect(parentBySpanName).toHaveProperty("openclaw.run")` plus `expect(parentBySpanName["openclaw.run"]).toBeUndefined()`, or keep the original `toMatchObject` assertion.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes; the focused Vitest matcher probe demonstrates the exact assertion semantic change, and the surrounding tests do not otherwise assert span presence for the three changed cases.
- Is this the best way to solve the issue? yes; preserving explicit key presence keeps the intended “span exists but has no parent context” contract without changing production behavior.
