# Commit 55a0c9b1f4

## Summary

Found one concrete routing regression risk in the new WebChat text-only image path: when only some images are described by media understanding, the remaining images are rehydrated as native model images even though the gateway already determined the session model is text-only.

## Findings

### Medium: Undescribed WebChat images can still be sent to text-only reply models

- Kind: regression
- File: `src/auto-reply/reply/current-turn-images.ts`
- Line: 90
- Evidence: `chat.send` routes image uploads into `ctx.MediaPaths` when `supportsSessionModelImages` is false (`src/gateway/server-methods/chat.ts:2399`, `src/gateway/server-methods/chat.ts:2410`, `src/gateway/server-methods/chat.ts:2438`). Media understanding then selects only one image by default (`src/media-understanding/attachments.select.ts:9`, `src/media-understanding/attachments.select.ts:83`). After that, `resolveCurrentTurnImages` filters out described attachment indexes but rehydrates every remaining undescribed image into `ImageContent` (`src/auto-reply/reply/current-turn-images.ts:81`, `src/auto-reply/reply/current-turn-images.ts:90`, `src/auto-reply/reply/current-turn-images.ts:108`) without knowing whether the final reply model supports image input.
- Impact: A text-only WebChat session with multiple uploaded images and default media-understanding attachment policy will describe the first image, then pass the remaining images as native images to the text-only reply model. Before this commit, configured image-model routing switched the run to a vision model; after the override removal, this can produce provider errors or silent loss of the extra images on the restored text-only routing path.
- Suggested fix: Thread the final model image capability or an explicit ŌĆ£text-only media-path routeŌĆØ flag into `resolveCurrentTurnImages`, and suppress native image rehydration for that case. If multiple images should be preserved for text-only sessions, run image understanding over all staged image offloads or add an explicit omitted-image note instead of passing native images.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes, source-level: configure a WebChat session model that does not support images, configure `agents.defaults.imageModel`, and send two image attachments. The gateway stages both as `MediaPaths`; default image understanding describes only one; the second is then placed in `followupRun.images`.
- Is this the best way to solve the issue? The safest direction is to keep the gatewayŌĆÖs text-only capability decision attached to the turn and prevent later native image hydration from overriding it. Expanding media-understanding coverage to all text-only image offloads is a product choice, but sending native images to a known text-only model should be avoided.
