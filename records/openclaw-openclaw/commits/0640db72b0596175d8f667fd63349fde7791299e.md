# Commit 0640db72

## Summary

The refreshed generated channel metadata accidentally drops the Mattermost channel schema. Current `main` still has the Mattermost plugin and its schema, but the generated baseline no longer exposes it to config schema lookup or config validation.

## Findings

### Medium: Mattermost channel config schema is no longer included in generated metadata

- Kind: regression
- File: `src/config/bundled-channel-config-metadata.generated.ts`
- Line: 7818
- Evidence: The commit deletes the full `pluginId: "mattermost"` generated entry, so current metadata jumps from Matrix to `pluginId: "msteams"` at `src/config/bundled-channel-config-metadata.generated.ts:7818`. The generator only probes `src/config-schema.{ts,js,mts,mjs}` and skips a channel when no such module exists (`scripts/generate-bundled-channel-config-metadata.ts:74`, `scripts/generate-bundled-channel-config-metadata.ts:182`), but Mattermost’s generated surface is currently `extensions/mattermost/src/config-surface.ts:4`. The base config schema and validation both consume `GENERATED_BUNDLED_CHANNEL_CONFIG_METADATA` (`src/config/schema.ts:509`, `src/config/schema.ts:542`, `src/config/validation.ts:70`). Focused checks confirmed `buildConfigSchema()` no longer has `channels.mattermost.botToken`, `baseUrl`, or `dmPolicy`, and `validateConfigObjectWithPlugins({ channels: { mattermost: { dmPolicy: "open" }}})` returns `ok: true` even though `MattermostConfigSchema.safeParse({ dmPolicy: "open" })` correctly rejects it with the missing wildcard `allowFrom` error.
- Impact: Mattermost config loses schema-backed UI/docs metadata and validation. Invalid or unsafe Mattermost config can now be accepted silently instead of failing with the plugin-owned validation contract.
- Suggested fix: restore Mattermost discovery by adding a `channel-config-api.ts`/`src/config-schema.ts` export for `MattermostChannelConfigSchema`, or update the generator/runtime discovery candidates to include the Mattermost `src/config-surface.ts` pattern, then regenerate the channel metadata and config baselines.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: on current `main`, `buildConfigSchema()` lacks Mattermost field lookups, and `validateConfigObjectWithPlugins` accepts a Mattermost config that the plugin’s own schema rejects.
- Is this the best way to solve the issue? yes: the generated metadata should include the plugin-owned schema surface; fixing discovery/export keeps core generic and restores the intended generated contract.
