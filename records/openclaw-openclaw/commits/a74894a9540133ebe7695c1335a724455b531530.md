# Commit a74894a

## Summary

Found one edge-case regression in the new fail-fast guard. The direct session-lock case is fixed and covered, but the guard can also reject errors that the existing failover classifier intentionally treats as provider failover when explicit provider metadata is present.

## Findings

### Medium: Explicit provider failover errors can be blocked by a nested session-lock cause

- Kind: regression
- File: `src/agents/model-fallback.ts`
- Line: 1052
- Evidence: The new guard checks `hasSessionWriteLockTimeout(normalized)` after `coerceToFailoverError(...)`. For `Error` instances, `coerceToFailoverError` preserves the original error as the `FailoverError` cause, so a provider error with explicit metadata like `status: 429` / `code: RESOURCE_EXHAUSTED` and a nested `SessionWriteLockTimeoutError` cause still makes `hasSessionWriteLockTimeout(normalized)` true. That bypasses the existing classifier contract in `src/agents/failover-error.ts`, where explicit provider failover metadata is authoritative over nested session-lock text, with regression tests at `src/agents/failover-error.test.ts:387` and `src/agents/failover-error.test.ts:402`.
- Impact: A real provider rate-limit/overload/auth-style failure wrapped as an `Error` with a nested session-lock cause can now fail fast instead of trying configured fallback models, contradicting the existing failover-error behavior.
- Suggested fix: Only fail fast when the original error contains a session write-lock timeout and `coerceToFailoverError(...)` did not classify it as a `FailoverError`; or expose a helper that distinguishes “session-lock-only” errors from explicit provider failover metadata.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes, by adding a focused unit case where `runWithModelFallback` receives an `Error` instance with `status: 429`, `code: RESOURCE_EXHAUSTED`, and `cause: new SessionWriteLockTimeoutError(...)`; current source tracing shows it throws before fallback.
- Is this the best way to solve the issue? The direct fail-fast idea is right for plain session-lock contention, but the guard should preserve the existing explicit-provider-metadata exception.
