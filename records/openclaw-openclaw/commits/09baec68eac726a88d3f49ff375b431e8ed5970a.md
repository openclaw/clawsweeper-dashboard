# Commit 09baec68ea

## Summary

Found one compatibility regression in the new global Codex dynamic-tool timeout.

## Findings

### Medium: Codex harness now times out long-running OpenClaw tools before their advertised tool timeout

- Kind: regression
- File: `extensions/codex/src/app-server/run-attempt.ts`
- Line: 482
- Evidence: every matching Codex `item/tool/call` is wrapped with `handleDynamicToolCallWithTimeout(... timeoutMs: CODEX_DYNAMIC_TOOL_TIMEOUT_MS)`, and that helper clamps to the 30s constant before returning a failed tool response. The lower-level client also has an independent 30s fallback for every `item/tool/call` in `extensions/codex/src/app-server/client.ts:347`. This applies to all OpenClaw dynamic tools, not just tools without their own timeout contract.
- Impact: tools such as `image_generate` expose and honor longer provider timeouts. `src/agents/tools/image-generate-tool.ts:169` advertises `timeoutMs`, `src/agents/tools/image-generate-tool.ts:667` forwards the configured/per-call timeout to `generateImage`, and `src/agents/tools/image-generate-tool.test.ts:560` verifies a configured `180_000` ms timeout. `docs/tools/image-generation.md:219` explicitly tells users to set this for slow backends. Under the Codex harness, the same call now fails after 30s even when the tool/provider timeout is 180s, so slow image backends and similar long-running tool calls regress from working to deterministic failure.
- Suggested fix: make the Codex bridge timeout tool-aware or configurable, and keep the client fallback in sync. For long-running media/subagent-style tools, either return an async/background handle before 30s or allow the tool’s declared timeout budget. If the hard 30s cap is intentional for Codex, clamp or remove longer `timeoutMs` affordances from Codex-advertised tool specs and update the affected tool docs.
- Confidence: high
