# Commit e873c1e1f8

## Summary

Found one regression in the DNS precedence change. The code now treats Node's implicit default `verbatim` result order as a process override, so the documented and previous Node 22+ Telegram fallback to `ipv4first` is bypassed on default installs.

## Findings

### Medium: Default Node 22+ Telegram transport no longer uses IPv4-first DNS

- Kind: regression
- File: `extensions/telegram/src/network-config.ts`
- Line: 99
- Evidence: The new process-default branch accepts `dns.getDefaultResultOrder()` before the Node 22+ fallback. On the local supported runtime, `dns.getDefaultResultOrder()` returns `verbatim` without any explicit DNS flag, so `resolveTelegramDnsResultOrderDecision({ env: {}, nodeMajor: 22 })` now returns `{ value: "verbatim", source: "process-default" }`. Before this commit, the same no-override path returned `{ value: "ipv4first", source: "default-node22" }`. The docs still say Node 22+ falls back to `ipv4first` when no override applies at `docs/channels/telegram.md:881`.
- Impact: Default Telegram Bot API transports on Node 22+ lose the IPv4-first mitigation that was explicitly there to avoid common IPv6 fetch failures. On flaky dual-stack hosts this can reintroduce first-attempt failures and sticky fallback churn, now mostly hidden at debug level. The affected transport is used by polling startup, sends, probes, and username lookup through `extensions/telegram/src/bot-core.ts:192`, `extensions/telegram/src/send.ts:270`, `extensions/telegram/src/probe.ts:87`, and `extensions/telegram/src/api-fetch.ts:27`.
- Suggested fix: Preserve the old Node 22+ `ipv4first` fallback when the process is only reporting Node's implicit default `verbatim`. If process-level inheritance should only honor explicit operator choices, detect `NODE_OPTIONS` / exec args for `--dns-result-order` or require the Telegram-specific env/config override for `verbatim`. Add a test where `defaultResultOrder: "verbatim"` and `nodeMajor: 22` asserts the intended behavior.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused `tsx` probe on current `main` returned `actual: { value: "verbatim", source: "process-default" }`, while forcing `defaultResultOrder: null` returned the old `{ value: "ipv4first", source: "default-node22" }`.
- Is this the best way to solve the issue? unclear. If maintainers intentionally wanted Node's implicit `verbatim` default to replace the Telegram IPv4-first default, then docs/tests/changelog should be updated. If the old mitigation should remain, the process-default branch needs to distinguish explicit overrides from Node's default.
