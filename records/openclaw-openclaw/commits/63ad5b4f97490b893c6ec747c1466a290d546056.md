# Commit 63ad5b4

## Summary

Found one actionable bug in the structured attachment send path. The commit makes `attachments` deliverable, but local file attachments are discovered too late to participate in outbound media access root expansion, so live sends can reject files that the equivalent top-level `media` param can send.

## Findings

### Medium: Structured local attachments miss media access root expansion

- Kind: bug
- File: `src/infra/outbound/message-action-runner.ts`
- Line: 1247
- Evidence: `runMessageAction` builds `mediaAccess` from `collectActionMediaSourceHints(params, ...)` before `buildSendPayloadParts` extracts `actionParams.attachments` into `mergedMediaUrls` and synthesizes `actionParams.media`. Because `collectActionMediaSourceHints` only sees top-level media fields at that point, attachment paths are absent from the source list used by `getAgentScopedMediaLocalRootsForSources`.
- Impact: A live send with `attachments: [{ path: "/outside/default/roots/photo.png" }]` can fail with `Local media path is not under an allowed directory`, while the same path passed as top-level `media` succeeds when `tools.fs.workspaceOnly=false` allows parent-root expansion. This breaks the new structured attachment path for generated/local files outside the default media roots.
- Suggested fix: Normalize and collect structured attachment media hints before resolving `mediaAccess`, or extend `collectActionMediaSourceHints` / sandbox normalization to include `attachments[].media`, `mediaUrl`, `path`, `filePath`, `fileUrl`, and `url` before the media access policy is built. Keep the normalized attachment sources and `actionParams.media`/`mediaUrls` in sync.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. I ran an ad hoc `node --import tsx` probe with a direct test channel and a local image in a temporary directory outside default media roots. Top-level `media` added that directory to `mediaLocalRoots`; structured `attachments: [{ path }]` did not. A second probe that actually called `loadWebMedia` sent the top-level `media` case and rejected the structured attachment case with `Local media path is not under an allowed directory`.
- Is this the best way to solve the issue? yes. The failure is caused by ordering and source collection, so fixing the pre-access media-source collector is safer than widening local file policy later in delivery.
