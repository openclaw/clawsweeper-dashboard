# Commit 5d7262c

## Summary

Found one concrete test gap: this commit makes the threaded-reply test opt out of Telegram’s default preview streaming path, but that default path drops reply metadata from follow-up chunks of long final replies.

## Findings

### Medium: Default Telegram streaming is no longer testing `replyToMode="all"` for long replies

- Kind: test_gap
- File: `extensions/telegram/src/bot.create-telegram-bot.test.ts`
- Line: 2812
- Evidence: The test named `honors threaded replies for replyToMode=first/all` now injects `streamMode: "off"`, even though Telegram preview streaming defaults to `partial` in `extensions/telegram/src/preview-streaming.ts:14`. In the default streaming path, long final text is split; `extensions/telegram/src/lane-delivery-text-deliverer.ts:181` sends remaining chunks through `followUpPayload(...)`, and Telegram’s production `applyTextToFollowUpPayload` strips `replyToId`, `replyToCurrent`, and `replyToTag` at `extensions/telegram/src/bot-message-dispatch.ts:757`. A focused `tsx` harness of `createLaneTextDeliverer` with a split final payload `{ replyToId: "101" }` produced `sent:[{"text":"ijklmnop"}]`, confirming the follow-up chunk has no reply target.
- Impact: Users with `channels.telegram.replyToMode: "all"` can get only the first preview/final chunk threaded under the triggering message when the final answer exceeds Telegram’s preview chunk size. Follow-up chunks become unthreaded, despite the documented `all` mode in `docs/channels/telegram.md:534`.
- Suggested fix: Keep a test that exercises the default preview-streaming path for `replyToMode="all"` and make follow-up chunk reply metadata conditional on `replyToMode`. For `all`, preserve the reply target on every follow-up chunk; for `first`, stripping it is still appropriate.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The focused lane-delivery harness reproduced the metadata drop without network access; the touched test would have covered this path before it forced `streamMode: "off"`.
- Is this the best way to solve the issue? yes. The issue is at the boundary between Telegram preview finalization and follow-up chunk delivery; fixing the follow-up payload policy by mode is narrower than disabling preview streaming.
