# Commit 2a22eb6

## Summary

Found one regression: official npm plugin install trust is now explicit, but the npm update paths do not propagate or reconstruct that provenance, so updates for official packages with reviewed launch code can be blocked by the scanner.

## Findings

### Medium: Official npm plugin updates lose scanner trust provenance

- Kind: regression
- File: `src/plugins/update.ts`
- Line: 1030
- Evidence: `src/plugins/install.ts:757` now passes only `params.trustedSourceLinkedOfficialInstall` into `scanPackageInstallSource`, and `src/plugins/install-security-scan.runtime.ts:571` blocks critical scanner findings unless that flag or `dangerouslyForceUnsafeInstall` is set. New install flows pass the flag, but `updateNpmInstalledPlugins` calls `installPluginFromNpmSpec` for `record.source === "npm"` without it at `src/plugins/update.ts:1030`; the externalized bundled-plugin npm path also omits it at `src/plugins/update.ts:1390` and `src/plugins/update.ts:1398`. Official npm packages such as `@openclaw/voice-call`, `@openclaw/acpx`, `@openclaw/google-meet`, and `@openclaw/codex` are catalogued for npm publishing and contain scanner-critical `child_process` launch code in repo sources. A focused dry-run scan of `extensions/voice-call` returned `security_scan_blocked` without the flag for `extensions/voice-call/src/tunnel.ts:62` and `extensions/voice-call/src/webhook/tailscale.ts:14`, and returned `ok: true` with `trustedSourceLinkedOfficialInstall: true`.
- Impact: A plugin installed through the now-trusted catalog/onboarding/doctor npm path is persisted as a normal npm install record, then later `openclaw plugins update <id>` can fail on the same reviewed official code unless the user uses the unsafe override. Update-channel migration/fallback for externalized bundled plugins can hit the same block.
- Suggested fix: Propagate official provenance into update calls. The safer shape is to persist a source-linked official marker on install records, or reconstruct it during update only from trusted official catalog/bridge metadata with plugin-id and npm-spec/integrity checks, then pass `trustedSourceLinkedOfficialInstall: true` to the relevant `installPluginFromNpmSpec` calls.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The scanner behavior is directly reproducible with a dry-run install of `extensions/voice-call`: without the trust flag it returns `security_scan_blocked`; with the flag it succeeds. The update path deterministically omits that flag for npm records.
- Is this the best way to solve the issue? no. The commit correctly removes package-name-only trust for direct installs, but update needs its own provenance source instead of requiring users to bypass the scanner.
