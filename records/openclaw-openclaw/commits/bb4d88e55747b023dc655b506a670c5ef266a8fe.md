### Medium: committed shrinkwraps are stale under the repo generator

- Kind: supply_chain

- Line: 1570
- Evidence: This commit adds `peer: true` metadata to the root and bundled plugin shrinkwraps, for example `npm-shrinkwrap.json:1570` and `extensions/acpx/npm-shrinkwrap.json:906`. The repo generator explicitly normalizes this metadata away in `scripts/generate-npm-shrinkwrap.mjs:301`, with the comment that npm 11 patch releases disagree on package-lock v3 `peer`/`libc` fields. Running `node scripts/generate-npm-shrinkwrap.mjs --all --check` now fails with `.: npm-shrinkwrap.json is stale. Run \`pnpm deps:shrinkwrap:generate\`.` Individual checks also report stale shrinkwraps for every touched package dir.
- Impact: The dependency-generation gate is now red on `main`, and the committed npm publish/install metadata includes fields the repo intentionally strips for stable Node 24/npm 11 output. This is not a runtime exploit, but it is a supply-chain/release metadata regression that maintainers will likely need to clean up.
- Suggested fix: Regenerate with the current repo generator and commit the normalized output, which should remove the added `peer: true` fields. If the intent is to keep `peer` metadata, update `normalizeNpmVersionDrift` and its rationale/tests in the same change.
- Confidence: high
