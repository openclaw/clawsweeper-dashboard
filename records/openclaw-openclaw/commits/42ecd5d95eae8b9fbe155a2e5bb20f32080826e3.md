# Commit 42ecd5d

## Summary

Found one reliability issue in the new ACPX process cleanup fallback. The leased path is careful, but when the sidecar lease is missing the fallback can still kill a live generated wrapper that advertises a different lease id.

## Findings

### Medium: Lease-less fallback can kill a different live ACPX wrapper after PID reuse

- Kind: reliability
- File: `extensions/acpx/src/runtime.ts`
- Line: 789
- Evidence: `cleanupProcessTreeForRecord` passes `expectedLeaseId` and `expectedGatewayInstanceId` when it finds a sidecar lease, but the fallback call at `extensions/acpx/src/runtime.ts:789` calls `cleanupOpenClawOwnedAcpxProcessTree` with only `rootPid`, `rootCommand`, and `wrapperRoot`. The reaper then treats missing expected lease values as a match: `commandOptionEquals` returns `true` when `expected` is absent, and `liveCommandMatchesLeaseIdentity` returns `true` when both expected ids are absent in `extensions/acpx/src/process-reaper.ts:114`. New generated wrappers do carry lease flags from `extensions/acpx/src/process-lease.ts:159`, so the live process may prove it belongs to a different lease but still be terminated by this fallback.
- Impact: If `process-leases.json` is missing/corrupt/stale, or an old pre-lease session record is being closed, and the stored PID has been reused by another same-root ACPX wrapper, closing or cleaning the stale session can terminate the unrelated active ACP session. That interrupts active work and defeats the lease identity protection this commit adds.
- Suggested fix: In the fallback path, fail closed when the live generated-wrapper command contains OpenClaw lease flags but the caller cannot provide matching expected lease ids. Keep the legacy no-lease cleanup only for live commands that do not advertise a lease, or recover the expected lease from the live record before signaling. Add a regression test where `rootCommand` is a generated wrapper, the live command has `--openclaw-acpx-lease-id other-lease`, and no expected lease id is passed.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused `tsx` probe called `cleanupOpenClawOwnedAcpxProcessTree` with a stored generated-wrapper command, no expected lease ids, and a live same-root wrapper command containing `--openclaw-acpx-lease-id other-lease`; it returned `terminatedPids: [12345]` and invoked SIGTERM/SIGKILL.
- Is this the best way to solve the issue? yes: the fix should be in ownership verification, not broad cleanup policy. A leased live wrapper is stronger evidence than a stale session PID.
