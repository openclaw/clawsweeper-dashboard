# Commit 77c3bdb3

## Summary

Found one concrete attribution gap: the new timeout contract declares preflight attribution, but embedded-run timeouts before provider submission still default to provider/providerStarted. That can preserve the auth-profile cooldown poisoning this commit is meant to avoid for slow preflight paths.

## Findings

### Medium: Pre-provider run timeouts are still attributed as provider-started

- Kind: bug
- File: `src/agents/pi-embedded-runner/run.ts`
- Line: 2491
- Evidence: `src/agents/run-timeout-attribution.ts:1` declares `preflight`, but no production code writes that phase. The attempt timeout is armed before pre-provider work such as prompt hooks, image loading, and preflight compaction checks in `src/agents/pi-embedded-runner/run/attempt.ts:3427`, while the provider prompt is not submitted until `src/agents/pi-embedded-runner/run/attempt.ts:4164`. If the run timeout fires in that window, `src/agents/pi-embedded-runner/run.ts:2491` computes `assistantProviderStarted` as true solely from `timedOut && !timedOutDuringCompaction && !timedOutDuringToolExecution`. The timeout payload path also defaults missing attribution to `timeoutPhase: "provider"` and `providerStarted: true` at `src/agents/pi-embedded-runner/run.ts:2794`.
- Impact: A slow preflight hook, prompt assembly, image load, or pre-prompt check can still be surfaced through `agent.wait` as a provider-started timeout and can feed `resolveAuthProfileFailureReason(... providerStarted: true)`, cooling the auth profile even though the model request may not have started.
- Suggested fix: carry an explicit provider-submission flag from the exact `promptActiveSession` boundary, default pre-submission timeouts to `timeoutPhase: "preflight", providerStarted: false`, and only emit provider attribution after the prompt has actually been submitted.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes, source-level: configure a slow `before_agent_run`/prompt-build path with a short run timeout, so the attempt timer fires before `promptActiveSession`; then inspect `agent.wait` attribution and auth-profile usage stats.
- Is this the best way to solve the issue? Yes. The commit already added the public fields; the missing piece is using a real provider-submission fact instead of deriving it from broad timeout booleans.
