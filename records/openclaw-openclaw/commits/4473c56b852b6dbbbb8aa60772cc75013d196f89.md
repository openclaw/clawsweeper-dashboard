# Commit 4473c56

## Summary

Found one low-severity regression in the new fallback message: for supported ClawHub slugs that are not npm package names, the message tells users to run an invalid `npm:` install spec.

## Findings

### Low: Missing-artifact fallback can suggest an invalid npm package

- Kind: bug
- File: `src/plugins/clawhub.ts`
- Line: 382
- Evidence: `formatClawHubMissingArtifactMetadataError` builds the fallback command from the ClawHub package name: `npm:${params.packageName}@${params.version}`. Existing catalog contract tests cover ClawHub slugs like `clawhub:openclaw/moonshot@2026.5.2` and `clawhub:openclaw/vllm@2026.5.2` whose npm fallbacks are different scoped packages, `@openclaw/plugin-moonshot@2026.5.2` and `@openclaw/plugin-vllm@2026.5.2` (`src/plugins/provider-install-catalog.test.ts:609`, `src/plugins/provider-install-catalog.test.ts:792`). The npm spec validator rejects unscoped names containing `/` (`src/infra/npm-registry-spec.ts:42`). Focused check: `validateRegistryNpmSpec("openclaw/vllm@2026.5.2")` returned `unsupported npm spec: expected <name> or <name>@<version> from the npm registry`, while `@openclaw/plugin-vllm@2026.5.2` returned `ok`.
- Impact: When a rollout-state ClawHub artifact is missing for one of these slash-style ClawHub packages, the user-facing error recommends a command that fails before npm resolution, so the intended npm fallback path is harder to discover.
- Suggested fix: Only print an exact `npm:` command when the installer knows a valid npm fallback spec. Otherwise pass optional fallback metadata from catalog/onboarding callers, or use generic wording like “install the npm fallback package from the plugin catalog” instead of synthesizing one from the ClawHub slug.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the invalid fallback spec: the repo npm parser rejects `openclaw/vllm@2026.5.2`, and catalog tests show the valid fallback is `@openclaw/plugin-vllm@2026.5.2`. I did not call live ClawHub to force a missing-artifact response.
- Is this the best way to solve the issue? Unclear. The safest direction is to avoid synthesizing npm package names from ClawHub slugs unless a valid `npmSpec` is available.
