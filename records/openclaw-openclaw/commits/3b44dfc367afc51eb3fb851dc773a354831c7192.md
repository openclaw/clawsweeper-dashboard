# Commit 3b44dfc

## Summary

Found one concrete regression: the commit breaks the image/video provider-registry unit tests by replacing hoisted module mocks with `vi.spyOn` mocks that are invalidated by `vi.resetModules()`.

## Findings

### Medium: Provider-registry tests no longer mock the capability boundary

- Kind: regression
- File: `src/image-generation/provider-registry.test.ts`
- Line: 6
- Evidence: The test creates `resolvePluginCapabilityProvidersMock` with `vi.spyOn(...)`, then each `beforeEach` calls `vi.resetModules()` before `loadRegistry()` dynamically imports `provider-registry.js`. The imported registry gets a fresh `../plugins/capability-provider-runtime.js` module instance, so the spy is not the function it calls. Running `pnpm test src/image-generation/provider-registry.test.ts src/video-generation/provider-registry.test.ts` fails all 6 tests; the mocked empty/custom provider lists are ignored and real bundled providers are loaded.
- Impact: This leaves current `main` with a focused unit-fast failure in both provider registry suites and can block changed/full test gates.
- Suggested fix: Restore the hoisted `vi.mock("../plugins/capability-provider-runtime.js", ...)` pattern, or recreate the mock after `vi.resetModules()` using `vi.doMock` before importing the registry so the registry and test share the same mocked module instance. Apply the same fix to `src/video-generation/provider-registry.test.ts`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: `pnpm test src/image-generation/provider-registry.test.ts src/video-generation/provider-registry.test.ts` fails consistently with 6 failed tests.
- Is this the best way to solve the issue? yes: the failure is test isolation caused by module-cache reset plus non-hoisted spies; using a hoisted/mock-factory module mock matches the pre-commit working pattern.
