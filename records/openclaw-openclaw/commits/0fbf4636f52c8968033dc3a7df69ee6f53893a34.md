# Commit 0fbf463

## Summary

Found one low-severity regression: the Brave web-search plugin now declares `setup.providers[].authMethods`, which creates a generic provider onboarding choice even though Brave does not register a model/setup provider.

## Findings

### Low: Brave Search appears as a text-model provider auth choice

- Kind: regression
- File: `extensions/brave/openclaw.plugin.json`
- Line: 13
- Evidence: The commit adds `authMethods: ["api-key"]` under Brave `setup.providers`. `src/plugins/provider-auth-choices.ts:146` derives manifest auth choices from every `setup.providers[].authMethods`, and `src/flows/provider-flow.ts:95` includes unscoped choices in the default `text-inference` setup flow. A focused probe showed `brave-api-key` returned by both `resolveManifestProviderAuthChoices()` and `resolveProviderSetupFlowContributions({ scope: "text-inference" })`. Brave only registers a web-search provider in `extensions/brave/index.ts:8`, not a setup/model provider, so choosing that option cannot complete the normal provider auth flow.
- Impact: Users can see and select a “Brave API key” option in text-model onboarding, then hit a dead-end/retry path instead of configuring a usable model provider.
- Suggested fix: Keep `setup.providers[].envVars` for the warning suppression, but remove `authMethods` from Brave unless there is a scoped web-search setup flow. Add a regression test that Brave/web-search-only setup metadata does not produce a text-inference provider setup contribution.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Run a focused provider-flow probe on current `main`; it returns a `brave-api-key` manifest choice and text-inference setup contribution for the Brave plugin.
- Is this the best way to solve the issue? yes. The commit only needs env-var mirroring to suppress the diagnostic; `authMethods` is the extra field that turns metadata into an onboarding auth choice.
