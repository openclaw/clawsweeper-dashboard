# Commit 5c02b72

## Summary

Found one regression in the doctor repair path: `doctor --fix` can now preserve and write a root `defaultModel` key that only doctor-specific validation accepts. The final doctor validation uses the same bypass, so doctor can complete successfully while the normal config read path still rejects the file as invalid.

## Findings

### Medium: Doctor can leave a config that normal OpenClaw validation rejects

- Kind: bug
- File: `src/flows/doctor-health-contributions.ts`
- Line: 623
- Evidence: `runWriteConfigHealth()` passes `preservedLegacyRootKeys` into `replaceConfigFile()`, and `runFinalConfigValidationHealth()` also passes it into `readConfigFileSnapshot()` at `src/flows/doctor-health-contributions.ts:663`. That bypass strips `defaultModel` only for validation (`src/config/validation.ts:670`), but ordinary `readConfigFileSnapshot()` defaults to no preserved keys (`src/config/io.ts:2385`). A focused smoke using the real doctor config flow and config write API wrote a config with `{ defaultModel, agents.list[].description, unexpected }`; doctor removed `unexpected`, preserved `defaultModel`, and doctor-mode validation passed, but a normal `readConfigFileSnapshot()` immediately returned `valid: false` with `Unrecognized key: "defaultModel"`.
- Impact: Users running `openclaw doctor --fix` to repair an unrelated unknown key can be left with an on-disk config that doctor says is valid, while normal gateway/config guard reads still fail on the preserved `defaultModel`. This keeps the user in a broken repair loop instead of producing a config the rest of OpenClaw can load.
- Suggested fix: Make the accepted contract consistent. Either make the normal config read/validation path tolerate this legacy root metadata key while keeping it out of runtime/public typed config, or migrate/drop `defaultModel` during doctor repair with an explicit warning. Doctor’s final validation should not accept a narrower special-case config than normal OpenClaw startup/config validation accepts.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Create a temp config containing root `defaultModel`, `agents.list[].description`, and another unknown root key; run the doctor config flow with repair and write using the returned `preservedLegacyRootKeys`; then call ordinary `readConfigFileSnapshot()`. The persisted file keeps `defaultModel`, drops the other unknown key, doctor-mode validation passes, and normal validation fails on `defaultModel`.
- Is this the best way to solve the issue? no. The current fix preserves data but only by creating a doctor-only validation contract. The safer fix is to align normal read validation with the legacy preservation decision, or to perform an explicit migration/removal that produces a normal-valid config.
