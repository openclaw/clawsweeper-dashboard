# Commit 189ab9f5d1

## Summary

Found one remaining SSRF bypass in the new Firecrawl scrape target guard. The landed fix correctly blocks literal private, loopback, metadata, and non-HTTP(S) URLs before cache/network work, but it does not check DNS answers for ordinary hostnames that resolve to private or special-use IPs.

## Findings

### Medium: DNS-resolving private hosts bypass the Firecrawl scrape target guard

- Kind: security
- File: `extensions/firecrawl/src/firecrawl-client.ts`
- Line: 98
- Evidence: `assertFirecrawlScrapeTargetAllowed()` parses the target URL and calls only `isBlockedHostnameOrIp(parsed.hostname)`. That helper checks reserved hostnames and literal/private IP strings, but it does not resolve DNS. The shared SSRF path’s stronger `resolvePinnedHostnameWithPolicy()` does resolve and reject private/special-use DNS answers. A focused live check showed `169.254.169.254.nip.io` resolves to `169.254.169.254`; `isBlockedHostnameOrIp("169.254.169.254.nip.io")` returns `false`, while `resolvePinnedHostnameWithPolicy("169.254.169.254.nip.io")` blocks with `SsrFBlockedError`.
- Impact: an injected or model-chosen `firecrawl_scrape` target such as `http://169.254.169.254.nip.io/latest/meta-data/` can still be forwarded to Firecrawl. This is especially relevant for the now-supported self-hosted/private Firecrawl `baseUrl` mode, where Firecrawl may resolve and fetch the target from inside the user’s network.
- Suggested fix: make scrape target validation async and run the target hostname through `resolvePinnedHostnameWithPolicy()` before forwarding to Firecrawl, at least for self-hosted Firecrawl endpoints. Keep the existing literal-host fast block, and add regression coverage for a hostname that resolves to a private/link-local IP.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Resolve `169.254.169.254.nip.io`, then compare `isBlockedHostnameOrIp()` with `resolvePinnedHostnameWithPolicy()` as above; the current guard follows the permissive path.
- Is this the best way to solve the issue? unclear. DNS resolution for hosted Firecrawl is not a perfect pin because Firecrawl resolves remotely, but rejecting locally observable private DNS answers would close a concrete self-hosted/private-network bypass and better match the existing web-fetch target-safety contract.
