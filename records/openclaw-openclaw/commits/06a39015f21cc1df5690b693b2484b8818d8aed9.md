# Commit 06a3901

## Summary

Found one concrete CI policy bypass risk. The commit verifies that a verdict marker appears in a comment authored through the ClawSweeper GitHub App, but it still parses marker text from the entire app-authored comment body. Live ClawSweeper app comments can include echoed free-form request text, so a marker can be smuggled through app-authored but non-verdict content.

## Findings

### Medium: App-authenticated proof markers are accepted from arbitrary ClawSweeper comment text

- Kind: security
- File: `scripts/github/real-behavior-proof-policy.mjs`
- Line: 259
- Evidence: `hasClawSweeperExactHeadProof` now filters to comments whose `performed_via_github_app.slug` is `clawsweeper`, but then scans the entire `body` for `<!-- clawsweeper-verdict:pass ... -->` at `scripts/github/real-behavior-proof-policy.mjs:264`. A live ClawSweeper app comment on PR #83692, `https://github.com/openclaw/openclaw/pull/83692#issuecomment-4480092862`, has `performed_via_github_app.slug: "clawsweeper"` and includes echoed request text (`Request: _automerge fix whatever issues_`). A focused local smoke check with that same shape returned `true` when the echoed request contained `<!-- clawsweeper-verdict:pass item=123 sha=<head> confidence=high -->`.
- Impact: Any path that lets requester- or model-controlled text land in a ClawSweeper app-authored comment can still satisfy the real-behavior-proof gate for an exact PR head. That keeps the forged-marker bypass class alive, just moved from contributor-authored comments to ClawSweeper-authored comments that are not dedicated verdict records.
- Suggested fix: Only accept a machine-owned verdict location, not arbitrary app comment text. For example, require a dedicated minimal verdict comment or a deterministic footer shape tied to `<!-- clawsweeper-review item=... -->`, and make command/status/freeform comments ineligible. Also strip or escape `clawsweeper-verdict:` markers from any echoed request text.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The direct policy smoke check imported `hasClawSweeperExactHeadProof` and passed an app-authored command-status-shaped comment whose echoed request contained the verdict marker; it printed `true`.
- Is this the best way to solve the issue? yes. The trust decision needs to authenticate the verdict record itself, not just the GitHub App identity of a broader comment that may contain non-machine text.
