# Commit c5ca8a17

## Summary

Found two concrete reliability bugs in the new release CI helper scripts: the run summary can report unrelated workflow runs as children, and the OpenRouter secret preflight can pass an invalid key.

## Findings

### Medium: Release summary reports unrelated runs as child workflows

- Kind: reliability
- File: `.agents/skills/openclaw-release-ci/scripts/release-ci-summary.mjs`
- Line: 61
- Evidence: The helper lists child runs by querying `actions/runs?per_page=100` and selecting any run created after the parent with a matching workflow name. It does not use the parent’s recorded child run IDs, dispatch inputs, `headSha`, or parent job outputs. A live check on parent run `25901122541` showed the helper reporting unrelated CI runs at SHAs `30a26db...`, `6546c3e...`, and `931b6d1...`; the parent verifier log recorded the actual children as `25901133095`, `25901133453`, `25901133402`, and `25901229022`, all at `b672be5...`.
- Impact: Release triage can focus on the wrong runs and miss the actual failed child, which is especially risky because this helper is meant to be the compact evidence source for release validation.
- Suggested fix: Derive exact child run IDs from the parent job outputs or verifier/dispatch logs, then validate each child’s `headSha` against the parent target. Avoid broad repo run-list matching as the source of truth.
- Confidence: high

### Medium: OpenRouter preflight accepts a fake API key

- Kind: reliability
- File: `.agents/skills/openclaw-release-ci/scripts/verify-provider-secrets.mjs`
- Line: 83
- Evidence: OpenRouter is checked with `https://openrouter.ai/api/v1/models`, and any `response.ok` is treated as a valid secret. A live fake-token check returned `openrouter: ok env=OPENROUTER_API_KEY required` and exit `0` for `OPENROUTER_API_KEY=or_fake`. OpenRouter documents `/api/v1/models` as a model listing endpoint, while `/api/v1/key` is the current-key endpoint and documents `401 Unauthorized` errors, making `/api/v1/key` the better validity probe. ([openrouter.ai](https://openrouter.ai/docs/api/api-reference/models/get-models))
- Impact: An invalid OpenRouter key from env or 1Password can pass preflight, undermining the guardrail this skill adds before expensive full release provider matrices.
- Suggested fix: Change OpenRouter validation to `GET /api/v1/key` with the bearer token, or another authenticated endpoint that fails invalid credentials, and add a fake-token regression check.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The two reproductions were `node .agents/skills/openclaw-release-ci/scripts/release-ci-summary.mjs 25901122541` and `OPENROUTER_API_KEY=or_fake node .agents/skills/openclaw-release-ci/scripts/verify-provider-secrets.mjs --required openrouter --timeout-ms 8000`.
- Is this the best way to solve the issue? yes. Both fixes should make the helpers validate exact evidence rather than broad, potentially misleading success signals.
