# Commit 6290050

## Summary

Found one low-severity test regression: the tightened `execStart` assertion assumes the temp entrypoint path will not be shell-quoted, but the production formatter quotes paths containing spaces.

## Findings

### Low: `execStart` test fails when the temp directory path needs shell quoting

- Kind: compatibility
- File: `src/commands/status.service-summary.test.ts`
- Line: 129
- Evidence: `summarizeGatewayServiceLayout` formats `execStart` by shell-quoting each program argument in `src/daemon/service-layout.ts:20`. The changed assertion now expects ``/usr/bin/node ${entrypoint} gateway run`` exactly. A focused run with `TMPDIR` set to a path containing a space fails because the received value is `/usr/bin/node '<entrypoint>' gateway run`, while the expected value omits the quotes.
- Impact: Contributors or CI jobs with temp directories containing spaces get a false unit-test failure. The previous `toContain("gateway run")` assertion did not have this path-shape dependency.
- Suggested fix: Build the expected string through the same quoting behavior, or assert the stable parts separately while also checking that `layout.execStart` includes the formatted command semantics.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Create/use a temp directory whose path contains a space, set `TMPDIR` to it, then run `pnpm test src/commands/status.service-summary.test.ts`; the test fails at `src/commands/status.service-summary.test.ts:129`.
- Is this the best way to solve the issue? yes. The implementation is behaving as designed by quoting shell-unsafe arguments, so the test should account for that formatter instead of expecting raw interpolation.
