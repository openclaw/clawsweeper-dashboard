## Findings

### Low: MiniMax Docker gateway shard no longer exercises `minimax-portal`

- Kind: test_gap
- File: `.github/workflows/openclaw-live-and-e2e-checks-reusable.yml`
- Line: 2215
- Evidence: The workflow now runs `OPENCLAW_LIVE_GATEWAY_PROVIDERS=minimax,minimax-portal OPENCLAW_LIVE_GATEWAY_MAX_MODELS=1`. The gateway live test applies that cap globally before running probes (`src/gateway/gateway-models.profiles.live.test.ts:2446`). The selector prioritizes `minimax/minimax-m2.7` before `minimax-portal/minimax-m2.7` (`src/agents/live-model-filter.ts:20`, `src/agents/live-model-filter.ts:31`) and returns once `maxItems` is reached (`src/agents/live-model-filter.ts:305`). With the previous cap of `2`, both provider IDs could be selected; with `1`, `minimax-portal` is skipped.
- Impact: Stable/full release Docker live validation can pass while the package/container path for `minimax-portal` auth, catalog, or gateway runtime is broken. The native source MiniMax lane still covers both providers, so this is not a product runtime regression by itself, but it weakens the release Docker/package proof.
- Suggested fix: Keep this lane at `OPENCLAW_LIVE_GATEWAY_MAX_MODELS=2`, or split MiniMax and MiniMax Portal into separate one-model Docker shards. If skipping portal is intentional, remove `minimax-portal` from this command so the lane does not imply coverage it no longer provides.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The static path is enough: run the changed workflow command with both provider candidates available and `OPENCLAW_LIVE_GATEWAY_MAX_MODELS=1`; selection caps before probes and picks `minimax/minimax-m2.7` first.
- Is this the best way to solve the issue? unclear. If the goal is to preserve both auth/provider surfaces, the safest fix is two one-model shards. If the goal is only to reduce the MiniMax lane to one total model, the workflow should stop listing `minimax-portal`.
