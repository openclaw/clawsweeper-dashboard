## Findings

### Medium: model-list normalization passes partial row models to provider runtime hooks

- Kind: compatibility
- File: `src/commands/models/list.rows.ts`
- Line: 126
- Evidence: `normalizeListRowWithProviderPlugin` passes `params.model as ProviderRuntimeModel` into `normalizeProviderResolvedModelWithPlugin` at `src/commands/models/list.rows.ts:126`. Some callers supply display-only row models: fallback configured rows contain only `provider`, `id`, `name`, `input`, and `contextWindow` at `src/commands/models/list.rows.ts:265`; manifest rows similarly omit required runtime fields at `src/commands/models/list.rows.ts:226`. The hook contract says `context.model` is `ProviderRuntimeModel` at `src/plugins/types.ts:535`, and `ProviderRuntimeModel` is based on `@earendil-works/pi-ai` `Model`, whose dependency type requires fields such as `api`, `baseUrl`, `reasoning`, `cost`, and `maxTokens`.
- Impact: provider plugins can reasonably dereference required fields in `normalizeResolvedModel`. After this commit, `openclaw models list` can call that hook on configured or catalog display rows that are not fully resolved runtime models, so an installed provider plugin that does something like `ctx.model.api.startsWith(...)` or reads `ctx.model.cost.input` can crash model listing for stale/configured refs. The bundled hooks I read are mostly tolerant of missing fields, which is why this is compatibility risk rather than a confirmed bundled-provider break.
- Suggested fix: avoid reusing `normalizeResolvedModel` with partial display rows. Either build a real `ProviderRuntimeModel` before calling the hook, or add/use a row-specific provider hook with a narrower `ListRowModel`-style contract. If Anthropic is the only needed fallback, keep the provider-owned policy but expose it through a hook whose input matches the list-row surface.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? No. I did not create a throwaway external provider plugin; the evidence is from the source path and the public hook/type contract.
- Is this the best way to solve the issue? Unclear. The direction is right, but the current fix crosses a contract boundary by casting partial display data into a runtime model shape.
