# Commit bc4f27c

## Summary

I found one low-severity regression: changelog-only PRs now skip the whole `CI` workflow, including the lightweight private-key guard that previously still ran for those PRs.

## Findings

### Low: changelog-only PRs bypass the committed-private-key guard

- Kind: security
- File: `.github/workflows/ci.yml`
- Line: 23
- Evidence: Current `CI` adds `pull_request.paths-ignore: CHANGELOG.md` at `.github/workflows/ci.yml:21`, while the same workflow owns `Detect committed private keys` via `pre-commit ... detect-private-key` at `.github/workflows/ci.yml:365`. In the parent commit, the `pull_request` trigger had no path filter, so a `CHANGELOG.md`-only PR still started `CI` and reached the always-on security jobs. GitHub documents that `paths-ignore` skips a workflow when all changed paths match the ignored patterns, so the private-key job is no longer scheduled for that PR shape. ([docs.github.com](https://docs.github.com/en/actions/reference/workflow-syntax-for-github-actions?utm_source=openai))
- Impact: A PR that only edits root `CHANGELOG.md` can now add a private-key block without the repository’s existing CI private-key detector running before merge. The PR diff is public, so this is not a secret-prevention guarantee by itself, but it is still a regression in the repo’s automated “do not merge key material” guard.
- Suggested fix: Keep changelog-only PRs out of the heavy fanout, but move `detect-private-key` into an always-on lightweight PR hygiene workflow, or keep `CI` scheduled and gate only the expensive jobs after preflight. Consider doing the same for `workflow-sanity`’s tracked conflict-marker check if changelog-only PRs should retain that guard.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes at source/contract level: open a PR whose only changed path is `CHANGELOG.md`; GitHub’s path filter contract skips `CI`, so `.github/workflows/ci.yml:365` is not scheduled. I did not live-fire a disposable PR.
- Is this the best way to solve the issue? Unclear. Splitting out an always-on lightweight hygiene workflow preserves the commit’s goal while keeping the security guard; leaving the entire workflow unfiltered would restore safety but also restore some workflow fanout.
