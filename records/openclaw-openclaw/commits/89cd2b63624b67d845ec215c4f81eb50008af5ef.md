# Commit 89cd2b6362

## Summary

Found two actionable reliability issues in the new Active Memory circuit breaker change. The focused active-memory test target is currently red, and the implementation misses a runner timeout path that can bypass the new breaker.

## Findings

### Medium: Focused active-memory tests fail on the new reset scenario

- Kind: reliability
- File: `extensions/active-memory/index.test.ts`
- Line: 3215
- Evidence: `pnpm test extensions/active-memory/index.test.ts extensions/active-memory/config.test.ts` fails with `expected "vi.fn()" to be called 3 times, but got 2 times` in `resets circuit breaker after a successful recall`. The failing assertion is part of this commit’s new coverage and expects the fourth recall to run after a cooldown-expired success.
- Impact: The changed plugin test lane is red, and the failing scenario suggests the breaker can still skip recall after the test’s intended successful reset path.
- Suggested fix: Debug the reset/cooldown path and make the test assert breaker state after the third call. If cache or test mock state is intentionally suppressing the fourth call, update the implementation or test to prove the breaker is actually closed after success.
- Confidence: high

### Medium: Runner timeout error payloads are classified as successful recall

- Kind: bug
- File: `extensions/active-memory/index.ts`
- Line: 2162
- Evidence: `runRecallSubagent` joins all returned `result.payloads` text without checking `payload.isError`. The embedded runner has a timeout path that returns an `isError: true` payload with text like `Request timed out before a response was generated` in `src/agents/pi-embedded-runner/run.ts:1933`. Because `maybeResolveActiveRecall` treats any non-empty `rawReply` as `status: "ok"` at `extensions/active-memory/index.ts:2345`, that timeout payload resets the new circuit breaker at `extensions/active-memory/index.ts:2376` instead of recording a timeout.
- Impact: Some active-memory sub-agent timeouts can still run on every eligible turn, defeating the fix for #74054. The timeout text can also be cached/injected as active-memory context.
- Suggested fix: Treat `isError` timeout/abort payloads from `runEmbeddedPiAgent` as timeout/unavailable results before summary normalization, and record breaker timeouts for the timeout classification. Add a regression test where `runEmbeddedPiAgent` returns `{ payloads: [{ text: "...timed out...", isError: true }], meta: { aborted: true } }`.
- Confidence: high
