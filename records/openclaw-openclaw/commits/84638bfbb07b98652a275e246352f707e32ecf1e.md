# Commit 84638bf

## Summary

Found one low-severity compatibility regression in the new non-mac iMessage probe guard.

## Findings

### Low: Explicit `imsg` PATH wrappers are treated as the unsupported default

- Kind: compatibility
- File: `extensions/imessage/src/probe.ts`
- Line: 96
- Evidence: `probeIMessage` collapses both an omitted `cliPath` and an explicitly configured `cliPath: "imsg"` to the same string at line 90, then `resolveIMessageNonMacHostError` rejects that string on non-darwin before `detectBinary` or the RPC wrapper can run. Before this commit, bare executable names were supported by `detectBinary`, which resolves PATH commands with `which`/`where`, and `createIMessageRpcClient` would spawn that executable with the `rpc` args.
- Impact: A Linux/Windows remote-Mac setup that follows the `cliPath` wrapper contract by installing an SSH wrapper named `imsg` on PATH now fatally reports ÔÇťdefault imsg CLI must run on macOSÔÇŁ and cannot start, even though the configured executable may be a valid stdio wrapper. The same predicate can also misclassify Windows path-like wrapper values ending in `\imsg` because it only checks for `/`.
- Suggested fix: Preserve whether `cliPath` was omitted versus explicitly configured and only fail fast for the omitted default, or make the default-local predicate only match the exact implicit default after both `/` and `\` path separators are ruled out. Let explicit configured values continue through `detectBinary`/RPC probing.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes by code path: on a non-darwin host configure `channels.imessage.cliPath` to a PATH wrapper named `imsg`; `probeIMessage` returns the new fatal error before binary detection. I also checked Node path semantics for a Windows-style `C:\OpenClaw\imsg` value.
- Is this the best way to solve the issue? The intended fail-fast behavior is useful for the implicit default, but the implementation should not infer ÔÇťdefaultÔÇŁ from the executable basename alone because `cliPath` is the userÔÇÖs wrapper contract.
