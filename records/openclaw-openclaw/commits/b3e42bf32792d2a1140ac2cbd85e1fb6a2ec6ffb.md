# Commit b3e42bf

## Summary

Found one low-severity bug in the new catalog-backed warning path: it can match a channel id whose real plugin id differs, then suggest only an install command that does not make the existing `plugins.allow` or `plugins.entries` config valid.

## Findings

### Low: Install hint can leave channel-id plugin config broken

- Kind: bug
- File: `src/config/validation.ts`
- Line: 100
- Evidence: `formatMissingOfficialExternalPluginWarning()` calls `getOfficialExternalPluginCatalogEntry(pluginId)`, and that catalog lookup accepts both plugin ids and channel ids. For catalog entries such as `wecom`, the channel id is `wecom` but the plugin id is `wecom-openclaw-plugin` (`scripts/lib/official-external-channel-catalog.json:4`, `scripts/lib/official-external-channel-catalog.json:17`, `scripts/lib/official-external-channel-catalog.json:38`). A focused check against current code shows `plugins.allow: ["wecom"]` emits: `plugin not installed: wecom — install the official external plugin with: openclaw plugins install @wecom/wecom-openclaw-plugin@2026.4.23`. Rechecking with a registry that already contains plugin id `wecom-openclaw-plugin` still emits the same warning for `plugins.allow: ["wecom"]`, because the allowlist is validated against plugin ids.
- Impact: Operators can follow the new actionable hint, install the suggested package, and still have the plugin blocked or the config warning unresolved because `plugins.allow` / `plugins.entries` must use `wecom-openclaw-plugin`, not the channel id `wecom`. The same issue applies to `yuanbao` / `openclaw-plugin-yuanbao`.
- Suggested fix: When the catalog match is via channel/provider alias rather than canonical plugin id, include the canonical plugin id config rewrite in the warning, or avoid the install-only hint for `plugins.allow` / `plugins.entries` alias matches. Also consider avoiding "official" wording for catalog entries whose `source` is `external`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Run config validation with `plugins.allow: ["wecom"]` and an empty registry; then run it again with a registry containing `wecom-openclaw-plugin`. Both produce the install-only warning for `wecom`.
- Is this the best way to solve the issue? yes. The broken path is diagnostic/config guidance, so the narrow fix is to distinguish canonical plugin-id matches from channel-id aliases and tell users what config id must change.
