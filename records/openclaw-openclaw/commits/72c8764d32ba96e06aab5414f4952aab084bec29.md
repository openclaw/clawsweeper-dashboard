# Commit 72c8764

## Summary

Found one remaining bug in the media forwarding fix: Slack now forwards `mediaLocalRoots` and `mediaReadFile`, but it still drops `mediaAccess.workspaceDir`, so relative agent-workspace upload paths can still fail the local-media guard.

## Findings

### Medium: Relative workspace upload paths still lose `workspaceDir`

- Kind: bug
- File: `extensions/slack/src/channel-actions.ts`
- Line: 51
- Evidence: `runMessageAction` builds a full `mediaAccess` object and dispatches it to plugins, including separately derived `mediaLocalRoots` and `mediaReadFile` (`src/infra/outbound/message-action-runner.ts:885`, `src/infra/outbound/message-action-runner.ts:977`). This commit’s `resolveSlackActionContext` only copies `ctx.mediaLocalRoots` and `ctx.mediaReadFile` into the Slack context (`extensions/slack/src/channel-actions.ts:51`), while `SlackActionContext` and `uploadFile` forwarding only carry those legacy fields (`extensions/slack/src/action-runtime.ts:88`, `extensions/slack/src/action-runtime.ts:311`). The shared media loader explicitly preserves `mediaAccess.workspaceDir` for resolving relative media paths (`src/media/load-options.ts:58`).
- Impact: `message` tool calls such as Slack `upload-file` with `filePath: "renders/file.wav"` from an agent workspace can still fail with `LocalMediaAccessError: Local media path is not under an allowed directory`, even though the changelog says agent-scoped workspace files can be attached. Absolute workspace paths are covered by the new test, but relative workspace paths remain broken.
- Suggested fix: Carry `ctx.mediaAccess` or at least `ctx.mediaAccess.workspaceDir` through `SlackActionContext`, then pass it into `sendSlackMessage` for both `sendMessage` media and `uploadFile`. Add a regression test with a relative `filePath` and `mediaAccess.workspaceDir`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused loader check showed roots/readFile alone failed for `renders/file.wav` with `path-not-allowed`, while the same call with `mediaAccess: { localRoots, readFile, workspaceDir }` succeeded.
- Is this the best way to solve the issue? yes. Slack already has the shared `sendSlackMessage` media access plumbing; the narrow fix is to forward the full media access contract instead of just two compatibility fields.
