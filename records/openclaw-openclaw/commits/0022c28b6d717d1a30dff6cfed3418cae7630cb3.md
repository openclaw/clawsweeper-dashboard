# Commit 0022c28

## Summary

Found one regression in the workflow path fix: the Mantis Discord thread attachment lane now passes an absolute `--output-dir`, but the QA CLI rejects absolute output directories before running the scenario.

## Findings

### Medium: Absolute QA output path makes the Mantis workflow fail before running

- Kind: regression
- File: `.github/workflows/mantis-discord-thread-attachment.yml`
- Line: 333
- Evidence: `run_lane()` now sets `output_dir="${GITHUB_WORKSPACE}/${root}/${lane}"` and passes it to `pnpm --dir "$repo_root" openclaw qa discord --repo-root "$repo_root" --output-dir "$output_dir"`. The live transport CLI resolves output directories through `resolveRepoRelativeOutputDir`, which throws on absolute paths (`extensions/qa-lab/src/cli-paths.ts:8`) and is called by `extensions/qa-lab/src/live-transports/shared/live-transport-cli.runtime.ts:21`. A CLI smoke with the same absolute-output-dir shape failed immediately with `--output-dir must be a relative path inside the repo root.`
- Impact: Every baseline/candidate lane in this workflow will fail before producing `discord-qa-summary.json`, so the Mantis evidence workflow cannot validate or comment on the Discord thread attachment scenario.
- Suggested fix: Keep `repo_root` absolute, but pass a repo-root-relative output directory to the QA CLI, then copy artifacts back to the harness workspace for upload/comment publishing. For example, use `output_dir=".artifacts/qa-e2e/mantis/discord-thread-attachment/$lane"` for the CLI and copy from `$repo_root/$output_dir` into `$GITHUB_WORKSPACE/$root/$lane`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: `OPENCLAW_BUILD_PRIVATE_QA=1 OPENCLAW_ENABLE_PRIVATE_QA_CLI=1 pnpm openclaw qa discord --repo-root "$PWD" --output-dir "$PWD/.artifacts/qa-e2e/mantis/discord-thread-attachment/baseline" --provider-mode mock-openai --scenario discord-thread-reply-filepath-attachment --allow-failures` fails at CLI startup with the same output-dir validation error.
- Is this the best way to solve the issue? no: the commit correctly makes `repo_root` absolute, but moving `--output-dir` outside the selected worktree conflicts with the QA CLI contract. Restoring a relative output dir plus copying artifacts preserves both the fixed repo-root resolution and the upload layout.
