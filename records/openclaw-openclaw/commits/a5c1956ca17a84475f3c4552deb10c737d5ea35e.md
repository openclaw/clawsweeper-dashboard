# Commit a5c1956

## Summary

Found two actionable issues in the new Codex CLI node binding path. The more important one is that conversations bound to a CLI node session still route app-server control and detach cleanup through the ambient OpenClaw `ctx.sessionFile`, which can mutate or delete an unrelated app-server binding.

## Findings

### Medium: CLI-node bindings can mutate or clear the ambient app-server binding

- Kind: regression
- File: `extensions/codex/src/command-handlers.ts`
- Line: 741
- Evidence: `resolveControlSessionFile()` returns `ctx.sessionFile` for any current binding that is not `codex-app-server-session`, including `codex-cli-node-session`. A focused smoke command with a CLI-node binding and `ctx.sessionFile: "/tmp/ambient-session.jsonl"` showed `/codex fast on` calling `setCodexConversationFastMode({ sessionFile: "/tmp/ambient-session.jsonl", enabled: true })`. `detachConversation()` has the same issue at lines 479-482: a CLI-node binding falls into the `else if (ctx.sessionFile)` branch and calls `clearCodexAppServerBinding(ctx.sessionFile)`.
- Impact: Once a conversation is bound to a Codex CLI node session, app-server-only commands such as `/codex fast`, `/codex model`, `/codex permissions`, `/codex steer`, and `/codex stop` can operate on a stale or unrelated app-server binding for the host session file. `/codex detach` can delete that persisted app-server binding, losing the attached Codex thread association.
- Suggested fix: Treat recognized non-app-server bindings as unsupported for app-server controls. Return `undefined` from `resolveControlSessionFile()` when `data.kind === "codex-cli-node-session"`, and in `detachConversation()` only fall back to `ctx.sessionFile` when there is no readable current Codex binding data.
- Confidence: high

### Low: Session listing reads every Codex session file before applying limit

- Kind: reliability
- File: `extensions/codex/src/node-cli-sessions.ts`
- Line: 202
- Evidence: `listLocalCodexCliSessions()` reads `history.jsonl`, calls `hydrateSessionFiles()`, then calls `hydrateSessionsFromSessionFiles()` before filtering and slicing to the requested limit. `hydrateSessionsFromSessionFiles()` scans all `~/.codex/sessions` files and `readSessionFileSummary()` reads each full JSONL transcript into memory.
- Impact: On a long-lived Codex install with many or large session transcripts, `/codex sessions --host ... --limit 10` can still read the entire Codex session corpus, potentially blocking the node host, timing out the 15s node invoke, or causing high memory use.
- Suggested fix: Apply bounded discovery before hydration: sort/cap by mtime, read only first metadata plus a bounded tail for last-message data, avoid the second full tree walk when history already provides enough candidates, and enforce a max file count/byte budget.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the first finding: a direct `node --import tsx` smoke against `handleCodexCommand()` with a `codex-cli-node-session` binding showed `/codex fast on` using the ambient `ctx.sessionFile`, and `/codex detach` clearing that same ambient app-server binding. For the second finding, the failure mode is from code inspection; I did not build a huge synthetic Codex history.
- Is this the best way to solve the issue? Yes for the first finding: the app-server control path should fail closed when the active binding is a CLI-node binding. For the second, the exact cap policy is product judgment, but listing should not require reading every transcript file fully.
