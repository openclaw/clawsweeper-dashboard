# Commit d932d89778

## Summary

Found one low-severity test regression: the rewritten assertions still check method names, but no longer prove `hello.features.methods` is actually an array as required by the gateway protocol.

## Findings

### Low: Hello method assertions now pass for a string value

- Kind: test_gap
- File: `src/gateway/server.sessions.store-rpc.test.ts`
- Line: 78
- Evidence: The commit changed the single `expect.arrayContaining([...])` assertion to `const methods = ... ?? []` followed by multiple `expect(methods).toContain(...)` checks. Vitest `toContain` passes for strings, while the old `arrayContaining` assertion fails for a string. The protocol schema requires `features.methods` to be `Type.Array(NonEmptyString)` in `src/gateway/protocol/schema/frames.ts:85`, and the server currently sends `features: { methods: gatewayMethods, events }` in `src/gateway/server/ws-connection/message-handler.ts:1533`.
- Impact: A future regression that serializes advertised methods as a comma-delimited/string value could pass this RPC test as long as the expected method names are substrings, even though clients and the protocol contract require an array.
- Suggested fix: Keep the clearer per-method assertions, but first assert the runtime type, for example `expect(Array.isArray(methods)).toBe(true)`, or revert this block to `expect(hello.features?.methods).toEqual(expect.arrayContaining([...]))`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused matcher probe showed `expect("sessions.list,sessions.preview,...").toContain("sessions.list")` passes, while `expect("sessions.list,sessions.preview").toEqual(expect.arrayContaining(["sessions.list"]))` fails.
- Is this the best way to solve the issue? yes. Adding an explicit array assertion preserves the new individual failure messages while restoring coverage for the protocol shape.
