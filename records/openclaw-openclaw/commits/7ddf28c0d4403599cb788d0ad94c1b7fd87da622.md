# Commit 7ddf28c

## Summary

Found two actionable issues in the new git plugin install path. Focused tests for the changed surface pass, but they do not cover credential-bearing git URLs or relative git specs across later updates.

## Findings

### Medium: Git install URLs with credentials are persisted and re-emitted

- Kind: privacy
- File: `src/cli/plugins-install-command.ts`
- Line: 351
- Evidence: git installs persist the raw `spec: params.spec` and `gitUrl: result.git.url` into the managed install record at `src/cli/plugins-install-command.ts:346`. The chat install path does the same at `src/auto-reply/reply/commands-plugins.ts:214`. A URL such as `git:https://token@example.com/org/private.git@main` is accepted by `parseGitPluginSpec` as an HTTP URL, cloned, then stored unredacted. That same install record is emitted by `openclaw plugins inspect --json` at `src/cli/plugins-inspect-command.ts:229`, printed as `Spec:` in non-JSON inspect output at `src/cli/plugins-inspect-command.ts:57`, and returned in chat `/plugin inspect` JSON at `src/auto-reply/reply/commands-plugins.ts:483`. The git installer already uses `redactSensitiveUrlLikeString` for clone log messages, but the persisted/displayed install metadata bypasses that redaction.
- Impact: users installing private repos with HTTPS credentials can end up with access tokens/passwords in `plugins/installs.json`, terminal output, JSON diagnostics, and chat replies. That puts repo credentials outside the documented secret stores and makes accidental support/debug paste leaks likely.
- Suggested fix: reject git URLs with URL userinfo or sensitive query params and tell users to use SSH, `file://`, or Git credential helpers; also redact install metadata before any inspect/chat/registry JSON output. If token-bearing URLs must be supported, store credentials as SecretRefs and keep `spec`/`gitUrl` credential-free.
- Confidence: high

### Low: Relative git specs are accepted but updates resolve them from the wrong cwd

- Kind: reliability
- File: `src/plugins/update.ts`
- Line: 550
- Evidence: `parseGitPluginSpec` explicitly accepts `./`, `../`, and `~/` git repo forms and normalizes relative paths to an absolute URL/spec at `src/plugins/git-install.ts:137`. The install result exposes that normalized `git.url`, but the CLI and chat installers persist the original raw spec instead (`src/cli/plugins-install-command.ts:351`, `src/auto-reply/reply/commands-plugins.ts:219`). Later updates compute `effectiveSpec` from `record.spec` at `src/plugins/update.ts:550` and pass it back into `installPluginFromGitSpec` at `src/plugins/update.ts:816`. So an install run as `openclaw plugins install git:../my-plugin@main` only updates correctly when the future `plugins update` command happens to run from the same working directory.
- Impact: accepted installs from relative local git repositories can become non-updatable or, worse, update from a different repository with the same relative path when run from another directory/service context.
- Suggested fix: persist `gitSpec`/`spec` as the canonical `ParsedGitPluginSpec.normalizedSpec`, or rebuild the update spec from persisted `gitUrl` plus `gitRef`. Add a regression test that installs `git:../repo@main`, changes cwd, then verifies update still uses the original absolute repo path.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. For the credential issue, install with a credential-bearing `git:https://user:token@host/org/repo.git@main` spec and inspect the persisted install record or `plugins inspect --json`; the raw `spec` and `gitUrl` are carried through directly. For the relative-path issue, install from `git:../repo@main`, run update from a different cwd, and observe `updateNpmInstalledPlugins` re-parsing the stale relative `record.spec`.
- Is this the best way to solve the issue? yes for rejecting/redacting URL credentials; storing repo credentials as ordinary install metadata conflicts with the existing secret-handling model. For relative specs, canonicalizing to `normalizedSpec` is the narrowest fix because the parser already computes the stable value.
