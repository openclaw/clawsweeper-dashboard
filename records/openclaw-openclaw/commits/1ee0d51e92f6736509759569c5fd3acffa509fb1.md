# Commit 1ee0d51

## Summary

Found two actionable issues: the merged commit leaves the Plugin SDK API baseline failing, and the new Codex app-server MCP merge path can produce hybrid server configs when user MCP and bundled MCP share a server name.

## Findings

### Medium: Plugin SDK API baseline was not updated for new public exports

- Kind: compatibility
- File: `src/plugin-sdk/agent-harness-runtime.ts`
- Line: 141
- Evidence: The commit adds public `CodexBundleMcpThreadConfig`, `LoadCodexBundleMcpThreadConfigParams`, and `loadCodexBundleMcpThreadConfig` exports from `openclaw/plugin-sdk/agent-harness-runtime`, but the range does not update `docs/.generated/plugin-sdk-api-baseline.sha256`. `pnpm plugin-sdk:api:check` fails on current `main` with: `Plugin SDK API baseline drift detected. Hash mismatch: docs/.generated/plugin-sdk-api-baseline.sha256`.
- Impact: The SDK API drift guard is broken on `main`, and the public plugin SDK surface changed without the tracked baseline artifact. This can block CI/release validation and hides intentional-vs-accidental public API changes.
- Suggested fix: If the new export is intentional, run `pnpm plugin-sdk:api:gen` and commit the updated `docs/.generated/plugin-sdk-api-baseline.sha256`. If not, keep the helper internal instead of exporting it through the public SDK surface.
- Confidence: high

### Medium: User MCP server overrides are deep-merged with bundled MCP servers on name collision

- Kind: bug
- File: `extensions/codex/src/app-server/thread-lifecycle.ts`
- Line: 285
- Evidence: `run-attempt.ts` passes bundled MCP as `params.config`, then `startOrResumeThread` merges `params.config` with `userMcpServersConfigPatch` via `mergeCodexThreadConfigs`. That helper recursively merges nested objects, so a bundled `mcp_servers.search` with `{ url, env }` plus a user `mcp_servers.search` with `{ command, args }` becomes one hybrid server containing all four fields. The existing CLI merge contract in `src/agents/bundle-mcp-config.ts` says user config overrides bundle defaults and implements that with user entries spread after bundled entries.
- Impact: A user trying to replace a bundled server with the same name in Codex app-server can get stale bundled URL/env/header fields mixed into their user stdio/http definition. That can make the MCP server fail to start or connect to the wrong endpoint with the wrong inherited config.
- Suggested fix: Merge `mcp_servers` by server name as replacement, not recursive field merge, or build a single bundled-plus-user MCP map using the same “user wins” contract before normalizing for Codex. Add start/resume regression coverage for a same-name bundled/user MCP server.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. The baseline issue reproduces with `pnpm plugin-sdk:api:check`. The MCP merge issue reproduces by calling `mergeCodexThreadConfigs` with same-name bundled and user `mcp_servers` patches and observing the hybrid output.
- Is this the best way to solve the issue? Yes for the baseline artifact. For the MCP merge, the safest direction is to preserve the existing OpenClaw merge contract: user MCP config should replace bundled defaults at the server-entry level.
