# Commit 9444b2ad

## Summary

Found one remaining replay edge case: the new broad OpenAI-compatible strip policy still preserves reasoning on active tool-call continuations, which can leave the same non-standard `reasoning` / `reasoning_content` fields in a vLLM/oMLX-style follow-up request when the model used a tool.

## Findings

### Medium: Tool continuation replay can still send Qwen reasoning fields

- Kind: reliability
- File: `src/plugins/provider-replay-helpers.ts`
- Line: 48
- Evidence: This commit enables `dropReasoningFromHistory` by default for `openai-completions`, but the sanitizer it activates intentionally skips an assistant message when it has a tool call followed by a tool result after the latest user turn (`src/agents/pi-embedded-runner/thinking.ts:279`). The new test locks that in for OpenAI-compatible replay (`src/agents/pi-embedded-runner.sanitize-session-history.test.ts:1229`). Pi’s Chat Completions converter then maps a preserved thinking block’s `thinkingSignature` back onto the outbound assistant message as a field such as `reasoning` or `reasoning_content`.
- Impact: A Qwen/vLLM or oMLX session that produces reasoning plus a tool call can still send the tool-continuation request with the same per-message reasoning field that #46637 reports as causing vLLM stalls and oMLX request rejection. Non-tool historical turns are fixed; tool-using turns can remain slow or fail.
- Suggested fix: Make active tool-turn reasoning preservation provider-policy controlled. For strict OpenAI-compatible default replay, strip reasoning from the assistant tool-call message too; keep preservation only for provider-owned exceptions that require it, such as DeepSeek/Z.AI preserved-thinking paths.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Unclear. I did not run a live vLLM/oMLX server, but the source path is concrete: `openai-completions` Qwen/vLLM, assistant thinking block with `thinkingSignature: "reasoning"` plus a tool call, then a tool result continuation.
- Is this the best way to solve the issue? Likely yes. The safer fix direction is policy-controlled preservation rather than a blanket change, because the commit already identifies DeepSeek and Z.AI as protocols that may need replayed reasoning metadata.
