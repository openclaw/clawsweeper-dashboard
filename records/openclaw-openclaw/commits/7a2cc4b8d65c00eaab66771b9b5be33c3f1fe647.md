# Commit 7a2cc4b

## Summary

Found one remaining wrong-surface delivery path in the message/action send helper: DM topic sends addressed by username or `t.me` target can still fall back to plain DMs after the target resolves to a positive Telegram user ID.

## Findings

### Medium: Username DM topic sends still allow threadless fallback

- Kind: bug
- File: `extensions/telegram/src/send.ts`
- Line: 667
- Evidence: `sendMessageTelegram` parses the raw target before `getChat` resolution, then keeps using `target.chatType` for both `resolveTelegramSendThreadSpec` and `allowThreadlessRetry` (`extensions/telegram/src/send.ts:604`, `extensions/telegram/src/send.ts:620`, `extensions/telegram/src/send.ts:667`). `parseTelegramTarget("@name")` and `parseTelegramTarget("https://t.me/name")` classify the chat as `unknown` (`extensions/telegram/src/targets.ts:80`), while docs explicitly allow username send targets (`docs/channels/telegram.md:783`). If `getChat` resolves that username to a positive DM chat ID, `target.chatType !== "direct"` remains true, so `withTelegramThreadFallback` strips `message_thread_id` and retries as a plain DM on `message thread not found`.
- Impact: This preserves the exact false-success/wrong-surface behavior fixed for numeric DM IDs, but for username/t.me direct targets used by CLI, message tools, and automation. The same raw-target guard is also used in media sends, stickers, and polls (`extensions/telegram/src/send.ts:864`, `extensions/telegram/src/send.ts:1518`, `extensions/telegram/src/send.ts:1626`).
- Suggested fix: After `resolveAndPersistChatId`, derive the effective chat type from the resolved numeric `chatId` and use that for `resolveTelegramSendThreadSpec` and `allowThreadlessRetry`. That keeps group usernames resolving to negative IDs eligible for forum fallback while making positive resolved DM IDs fail closed.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. A focused mock of `getChat("@dmuser") -> { id: 123456789 }`, first `sendMessage(...message_thread_id...)` throwing `400: Bad Request: message thread not found`, and second `sendMessage` succeeding will return success after a threadless retry because the fallback guard still sees the original `unknown` target type.
- Is this the best way to solve the issue? Yes. The fix should not disable fallback for all unknown targets before lookup, because a username can resolve to a group/channel negative ID. The safer boundary is the resolved numeric chat ID.
