# Commit e57b137

## Summary

Found one concrete platform-specific policy-enforcement gap in the Codex native hook trust config. The Linux-focused behavior covered by the PR tests passes, but Windows Codex app-server runs can still miss the trusted session-flags hook state key that makes the relay executable.

## Findings

### Medium: Windows Codex native relay hooks are not trusted

- Kind: security
- File: `extensions/codex/src/app-server/native-hook-relay.ts`
- Line: 31
- Evidence: `CODEX_SESSION_FLAGS_HOOK_SOURCE_PATHS` only emits `/<session-flags>/config.toml` and `<session-flags>/config.toml` keys, then uses those to populate `hooks.state`. OpenAI Codex 0.130.0 builds the session-flags hook source path with `synthetic_layer_path("<session-flags>/config.toml")`; on Windows that resolves against `C:\`, and hook state lookup uses `source_path.display().to_string()` in the persisted key. The same Codex discovery code only adds non-managed command hooks to active handlers when the trusted hash matches.
- Impact: on Windows, the generated key will not match the Codex key shape such as `C:\<session-flags>\config.toml:pre_tool_use:0:0`, so the relay command hooks remain untrusted and are not activated. That leaves Codex-native tool execution outside OpenClaw `before_tool_call` policy for paths that rely on the native relay, especially explicit yolo configurations and native operations that do not produce an approval request.
- Suggested fix: include Codex-compatible Windows session-flags hook-state keys, or derive the key variants from a helper that models Codex `synthetic_layer_path` for each supported app-server platform. Add focused coverage for the Windows key shape.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the contract path: on a Windows Codex app-server run, inspect `hooks.state` keys from `thread/start` and compare them to Codex’s discovered session-flags hook key, or run an explicit-yolo Codex harness with a blocking `before_tool_call` policy and verify the native relay does not fire. I did not run a live Windows app-server here.
- Is this the best way to solve the issue? Yes: the existing fix depends on pre-trusting Codex’s exact hook-state identity, so the safe repair is to generate all platform key identities Codex may use rather than weakening trust checks.
