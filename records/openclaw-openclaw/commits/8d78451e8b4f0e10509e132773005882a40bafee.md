# Commit 8d78451

## Summary

Found one low-severity regression in local `openclaw sessions --json` provider reporting for wrapper providers with slash-containing model IDs.

## Findings

### Low: Local sessions JSON rewrites wrapper providers to the nested model vendor

- Kind: regression
- File: `src/commands/sessions-display-model.ts`
- Line: 143
- Evidence: `resolveSessionDisplayModelRef` now parses `row.model` with `parseModelRef(row.model, row.modelProvider ?? defaultRef.provider)` before preserving non-CLI `row.modelProvider`. Then `src/commands/sessions.ts:200` overwrites the JSON rowâ€™s `modelProvider` with `modelRef.provider`. A focused probe returned `{"provider":"anthropic","model":"claude-haiku-4.5"}` for `{ modelProvider: "openrouter", model: "anthropic/claude-haiku-4.5" }`, and `{"provider":"moonshotai","model":"kimi-k2.5"}` for `{ modelProvider: "nvidia", model: "moonshotai/kimi-k2.5" }`.
- Impact: local `openclaw sessions --json` can misreport stored wrapper-provider sessions as the nested vendor provider. OpenRouter, NVIDIA, Vercel AI Gateway, and similar providers commonly use slash-containing model IDs, so scripts or humans inspecting session JSON may see the wrong provider and diagnose/auth/switch the wrong backend. This is new for `modelProvider`: before this commit the JSON spread preserved the stored `row.modelProvider`.
- Suggested fix: when `row.modelProvider` is present and is not a CLI runtime provider, preserve `{ provider: row.modelProvider, model: row.model }` instead of parsing the slash-containing model as a provider/model ref. Keep the CLI-runtime canonicalization path only for CLI providers, or reuse the gateway display resolver semantics.
- Confidence: high
