# Commit 2f8cb86381

## Summary

Found one actionable scan-coverage regression in the changed OpenGrep wrapper.

## Findings

### Medium: Changed OpenGrep scan silently drops every first-party symlink

- Kind: security
- File: `scripts/run-opengrep.sh`
- Line: 119
- Evidence: The PR-diff workflow runs `scripts/run-opengrep.sh --changed --sarif --error`, and the changed-path selector includes all changed paths under `src/`, `extensions/`, `apps/`, `packages/`, and `scripts/`. This commit adds `if [[ -L "$path" ]]; then continue; fi`, so any changed symlink in those first-party trees is removed before OpenGrep runs. If the skipped symlink is the only first-party changed path and no rulepack path changed, the script exits successfully with `No changed first-party paths for opengrep.` The nearby `.semgrepignore` rationale is specifically about `CLAUDE.md` guide symlinks, and current tree inventory confirms existing tracked symlinks in these trees are `CLAUDE.md -> AGENTS.md`; the implementation is broader than that intent.
- Impact: A future PR can add or change a symlink under a source/script/package path and bypass the blocking OpenGrep PR-diff scan entirely for that path. Before this change, that class of PR at least failed the security scan instead of passing with the path omitted.
- Suggested fix: Restrict the skip to the known guide alias shape, for example basename `CLAUDE.md` with target `AGENTS.md`, or fail explicitly on other first-party symlinks. If non-guide symlinks are intentionally allowed, resolve and scan the in-repo target path when it remains inside the repository and within an intended scan scope.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the wrapper behavior: the code path is deterministic, and a temporary `opengrep` stub showed the reviewed range still scans `scripts/run-opengrep.sh`; the preceding symlink-plus-ignore range skips `src/infra/outbound/CLAUDE.md` and falls back only because `.semgrepignore` also changed. A symlink-only changed range would hit the same `-L` branch and, without rulepack changes, exit 0.
- Is this the best way to solve the issue? No. The bug being fixed is specific to redundant `CLAUDE.md` symlink aliases, but the implemented filter covers all first-party symlinks. Narrowing the exception keeps the original CI unblock while preserving the previous block-on-unexpected-symlink posture.
