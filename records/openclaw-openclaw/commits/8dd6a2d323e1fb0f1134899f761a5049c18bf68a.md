## Findings

### Low: WhatsApp outbound mention boundary still accepts Unicode word suffixes

- Kind: bug
- File: `extensions/whatsapp/src/inbound/outbound-mentions.ts`
- Line: 186
- Evidence: `shouldSkipMentionAt` treats a mention as embedded only when the adjacent character matches `/[\w@]/`. In JavaScript, `\w` is ASCII-only, so a token followed by a non-ASCII letter is still resolved as a native mention. Focused check on current `main`:
  ```bash
  pnpm exec tsx -e "import { resolveWhatsAppOutboundMentions } from './extensions/whatsapp/src/inbound/outbound-mentions.ts'; console.log(JSON.stringify(resolveWhatsAppOutboundMentions({ chatJid: '120363000000000000@g.us', text: 'literal @15551234567号', participants: [{ id: '15551234567@s.whatsapp.net' }] })))"
  ```
  returned:
  ```json
  {"text":"literal @15551234567号","mentionedJids":["15551234567@s.whatsapp.net"]}
  ```
- Impact: The commit intends to prevent phone-number prefixes inside longer strings from creating hidden native mentions, but non-ASCII suffixes still trigger that path. In multilingual WhatsApp groups, text like `@15551234567号` can unexpectedly mention the matching participant.
- Suggested fix: Replace the ASCII boundary check with a Unicode-aware adjacent-character predicate, for example using `/[\p{L}\p{N}_@]/u` or an equivalent helper, and add regression cases for CJK/non-Latin suffixes and prefixes.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes; the one-line `tsx` invocation above reproduces it on current `main`.
- Is this the best way to solve the issue? yes; keeping the fix inside the existing boundary helper preserves the commit’s design while making “longer string” detection Unicode-aware.
