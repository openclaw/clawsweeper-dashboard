# Commit f784349

## Summary

Found one regression in update-channel persistence for configs that keep `update` in a top-level `$include`.

## Findings

### Medium: update channel writes now fail for `$include`-owned `update` config

- Kind: regression
- File: `src/cli/update-cli/update-command.ts`
- Line: 1930
- Evidence: `persistRequestedUpdateChannel` now always calls `mutateConfigFileWithRetry` with `writeOptions: { skipPluginValidation: true }`. The config mutation helper already disables its single-top-level-include fast path when `skipPluginValidation` is true (`src/config/mutate.ts:277`), then the root writer rejects touching an include-owned path (`src/config/io.write-prepare.ts:473`). A focused temp-config smoke using the same mutation shape showed:
  - `skip=false`: wrote `update.json` and changed `"channel"` to `"beta"`.
  - `skip=true`: failed with `Config write would flatten $include-owned config at update; edit that include file directly or remove the $include first.`
- Impact: `openclaw update --channel beta` or `--channel dev` can now fail for users whose root config has `update: { "$include": "./update.json" }`, even when the config is otherwise valid. Before this commit, the update-channel mutation did not force `skipPluginValidation`, so the include fast path could update the included file.
- Suggested fix: keep the include fast path available when `skipPluginValidation` is requested by making that path skip-aware: validate with plugin validation skipped and refresh/read back with `readConfigFileSnapshotForWrite({ skipPluginValidation: true })`. Avoid simply removing the skip from update-channel writes, since the commit is trying to preserve package-update repair behavior while plugin schemas are converging.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A temp config with root `update: { "$include": "./update.json" }` and included `{ "channel": "stable" }` reproduces the failure by calling `mutateConfigFileWithRetry` with the same `writeOptions` and `draft.update.channel = "beta"` mutation used by `persistRequestedUpdateChannel`.
- Is this the best way to solve the issue? yes. The include fast path is the intended support path for single top-level include writes; making it honor the skip option preserves both include behavior and the new package-update plugin-validation bypass.
