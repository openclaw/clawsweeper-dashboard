# Commit 8847d390

## Summary

Found one low-severity POSIX edge case in the Windows-style path detection. The main Windows regression is fixed and the focused tests pass, but the guard still treats any home path containing a literal backslash as Windows-style.

## Findings

### Low: POSIX home paths containing a literal backslash are misclassified as Windows paths

- Kind: bug
- File: `src/agents/skills/workspace.ts`
- Line: 91
- Evidence: `compactHomePrefixesForHome()` uses `home.includes("\\")` to add a backslash home-boundary prefix, and `normalizeCompactedSkillPath()` uses `matchedHomePrefix.includes("\\")` to rewrite all suffix backslashes. On POSIX, backslash is a legal filename character. A focused probe on this Linux checkout showed `compactHomePath("/tmp/alice\\corp/.openclaw-test-skills\\literal-skill/SKILL.md", ["/tmp/alice\\corp"])` returns `~/.openclaw-test-skills/literal-skill/SKILL.md`, changing the literal backslash in the skill directory.
- Impact: Rare POSIX users whose home path contains a literal backslash can get prompt `<location>` values that point at a different path, causing failed skill reads. The extra `home + "\\"` prefix can also compact a sibling POSIX path that merely starts with the home string plus a literal backslash as if it were under `~/`.
- Suggested fix: Classify Windows-style homes more narrowly, for example by native Windows path semantics or drive/UNC syntax, and normalize only when the matched home boundary is actually a Windows separator. A new regression test should use a POSIX home containing `\\`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The focused `node --import tsx --input-type=module` probe above exercises the real `__testing.compactHomePath` helper and shows the literal POSIX backslash being rewritten.
- Is this the best way to solve the issue? yes. The Windows fix should remain, but the Windows-style detection should not be `includes("\\")` over the whole home prefix.
