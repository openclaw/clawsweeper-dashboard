# Commit 530e4f9

## Summary

Found one compatibility regression in the global package update swap path. The new `movePathWithCopyFallback` wrapper rejects a symlinked npm global `node_modules` root before the package swap, even though the previous raw `fs.rename` path works through that layout.

## Findings

### Medium: Global self-update fails when the npm global root is a symlink

- Kind: regression
- File: `src/infra/package-update-steps.ts`
- Line: 287
- Evidence: `swapStagedNpmInstall` now passes `targetPackageRoot` and `params.stage.packageRoot` directly to `movePathWithCopyFallback`. The dependency’s `guardedRename` creates guards with `lstat(path.dirname(...))` and rejects a symlink directory component with `directory component must be a directory`. A focused smoke check with `globalRoot` as a symlinked `node_modules` path produced a failed `global install swap` step with that stderr and left the active package at `1.0.0`. A control check showed raw `fs.rename` can move the package through the same symlinked root.
- Impact: OpenClaw self-update can fail for installations where `npm root -g` preserves a symlinked global root, such as custom prefixes or managed layouts that symlink `lib/node_modules` onto another volume. The install command succeeds in the staged prefix, verification succeeds, then the final swap fails and the user stays on the old package.
- Suggested fix: resolve the filesystem move parents to real paths before calling `movePathWithCopyFallback` for the package-root swap, while preserving the original npm prefix/bin layout for install commands and shim placement. Alternatively, extend the fs-safe move helper to guard the resolved parent/nearest stable root so symlinked trusted roots remain supported.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Create a temp npm prefix where `lib/node_modules` is a symlink to a real global root, seed `openclaw@1.0.0`, then call `runGlobalPackageUpdateSteps` with that symlink path as `installTarget.globalRoot`. The update step writes `2.0.0` into the staged prefix, but `global install swap` fails with `directory component must be a directory`.
- Is this the best way to solve the issue? yes directionally. The failing paths are internal installer-controlled paths, but fs-safe’s atomic helper intentionally rejects symlink parent components. Canonicalizing only the move paths keeps the new guarded move behavior without breaking npm layouts that report symlinked roots.
