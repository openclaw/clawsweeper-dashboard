# Commit 7c2802b

## Summary

Found one regression in the fallback merge path: the new explicit auth-mutation opt-in is applied after the recursive “preserve existing keys” merge, so it cannot actually remove or replace old `gateway.auth` fields when the macOS app falls back to direct file writes.

## Findings

### Medium: Local fallback preserves old gateway auth even for explicit auth mutation saves

- Kind: bug
- File: `apps/macos/Sources/OpenClaw/OpenClawConfigFile.swift`
- Line: 71
- Evidence: `ConfigStore.save` falls back to `OpenClawConfigFile.saveDict(root, preserveExistingKeys: true, allowGatewayAuthMutation: allowGatewayAuthMutation)` after a failed Gateway save at `apps/macos/Sources/OpenClaw/ConfigStore.swift:66`. `saveDict` then recursively merges `previousRoot` into the new dictionary before checking `allowGatewayAuthMutation`. Because `mergeExistingConfig` keeps missing nested keys, an explicit auth mutation that omits or partially replaces `gateway.auth` has already had the previous auth merged back by `apps/macos/Sources/OpenClaw/OpenClawConfigFile.swift:71` and `apps/macos/Sources/OpenClaw/OpenClawConfigFile.swift:361`.
- Impact: The Tailscale settings path was updated to pass `allowGatewayAuthMutation: true`, but in the fallback path it still cannot clear old auth. For example, switching Tailscale Serve to “no credentials” builds `gateway.auth = ["allowTailscale": true]` at `apps/macos/Sources/OpenClaw/TailscaleIntegrationSection.swift:327`, but the merge preserves previous `mode` and `password`, leaving password auth enabled after the UI says settings were saved. Turning Tailscale off removes `gateway.auth` at `apps/macos/Sources/OpenClaw/TailscaleIntegrationSection.swift:323`, but the fallback merge restores the old block.
- Suggested fix: When `allowGatewayAuthMutation` is true, apply `dict.gateway.auth` as an exact replacement/removal after the preserve-existing merge, or make the merge path path-aware so auth mutation callers can delete `gateway.auth` and its nested keys. Add a regression test covering `preserveExistingKeys: true` plus `allowGatewayAuthMutation: true`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Source-level repro: start with existing `gateway.auth.mode/password`, trigger `ConfigStore.save(..., allowGatewayAuthMutation: true)` where `saveToGateway` fails and fallback runs, with a new root whose `gateway.auth` is absent or only `["allowTailscale": true]`. The recursive merge restores the old nested auth fields before the auth opt-in is checked.
- Is this the best way to solve the issue? yes for a narrow repair: auth mutation callers need exact replacement/removal semantics after the broad preservation merge. Longer term, fallback writes probably need patch/delete semantics instead of an unconditional deep merge.
