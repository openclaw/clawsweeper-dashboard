## Summary

Found one concrete regression: the landed commit reintroduced a direct `resolveIsNixMode` import from `config/paths` in `plugin-metadata-snapshot.ts`, which breaks existing CLI plugin tests that partially mock `../config/paths.js`.

## Findings

### Low: Metadata memo key breaks existing partial `config/paths` mocks

- Kind: compatibility
- File: `src/plugins/plugin-metadata-snapshot.ts`
- Line: 175
- Evidence: `plugin-metadata-snapshot.ts` imports `resolveIsNixMode` from `../config/paths.js` and calls it while computing the memo key. `src/cli/plugins-cli-test-helpers.ts:206` intentionally mocks `../config/paths.js` with only `resolveStateDir`. Running `node scripts/run-vitest.mjs run src/cli/plugins-cli.list.test.ts --maxWorkers=1 --reporter=dot` fails 4 tests with: `No "resolveIsNixMode" export is defined on the "../config/paths.js" mock`, through `computePluginMetadataSnapshotMemoKey`.
- Impact: Existing plugin CLI doctor/list coverage is red on current `main` when it exercises stale-plugin-config scanning through `loadPluginMetadataSnapshot`. This is a repo test regression and can hide real plugin CLI regressions until the mock/import mismatch is fixed.
- Suggested fix: Avoid importing `resolveIsNixMode` in `plugin-metadata-snapshot.ts`; compute the memo-key bit directly from `env.OPENCLAW_NIX_MODE === "1"` or drop the redundant `pathPolicy.nixMode` entry since `OPENCLAW_NIX_MODE` is already in `MEMO_RELEVANT_ENV_KEYS`. Updating every partial mock is more brittle.
- Confidence: high

- Do we have a high-confidence way to reproduce the issue? yes: run `node scripts/run-vitest.mjs run src/cli/plugins-cli.list.test.ts --maxWorkers=1 --reporter=dot`.
