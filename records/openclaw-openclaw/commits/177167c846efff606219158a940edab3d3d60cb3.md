# Commit 177167c

## Summary

Found one low-severity documentation regression: Google video generation now explicitly does not support generated audio, but the Google provider docs still say it does.

## Findings

### Low: Google provider docs still advertise video audio support

- Kind: bug
- File: `docs/providers/google.md`
- Line: 229
- Evidence: `docs/providers/google.md:229` says Google video generation supports `audio`. This commit changes all Google video mode capabilities to `supportsAudio: false` in `extensions/google/generation-provider-metadata.ts:91`, `:104`, and `:117`; core normalization drops `audio` when unsupported in `src/video-generation/normalization.ts:164`; and the Google provider request config no longer sends `generateAudio` in `extensions/google/video-generation-provider.ts:426`.
- Impact: users following the provider page may request Google video output with audio, but current runtime will ignore the `audio` override and return a warning instead of producing audio. That can waste paid/slow video jobs and makes the provider page contradict `video_generate action=list`.
- Suggested fix: update `docs/providers/google.md` to remove `audio` from the supported Google video controls, or explicitly state that generated audio is not currently forwarded on the Gemini API path.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes; static source proof is enough here. The docs advertise `audio`, while the current provider metadata and normalization path mark it unsupported and strip it before provider submission.
- Is this the best way to solve the issue? yes; the runtime behavior matches the installed `@google/genai` 1.51.0 dependency, whose Gemini API serializer rejects `generateAudio` when present, so the actionable fix is docs alignment.
