# Commit 67fe2097f3

## Summary

Found two actionable issues in the new Mantis workflow wiring. The main one makes the new Discord thread attachment workflow fail when it reaches the QA run step.

## Findings

### Medium: QA lanes pass a relative repo root after `pnpm --dir`, resolving to the wrong checkout

- Kind: bug
- File: `.github/workflows/mantis-discord-thread-attachment.yml`
- Line: 334
- Evidence: `run_lane` sets `repo_root="$worktree_root/$lane"`, then runs `pnpm --dir "$repo_root" openclaw qa discord --repo-root "$repo_root"`. `pnpm --dir` changes the command cwd to that worktree. The QA CLI resolves `--repo-root` with `path.resolve(opts.repoRoot ?? process.cwd())`, so the relative `.artifacts/.../baseline` argument is resolved relative to the baseline worktree itself, producing a nested non-repo path. The gateway startup path then uses that wrong repo root for `dist/index.js` and plugin staging/package reads.
- Impact: both baseline and candidate lanes should fail before producing the Discord evidence, so the new workflow is effectively unusable.
- Suggested fix: make `repo_root` absolute before invoking `pnpm --dir`, for example `repo_root="$(realpath "$worktree_root/$lane")"`, and copy from the absolute output path. Alternatively, do not combine `pnpm --dir` with a relative `--repo-root`.
- Confidence: high

### Low: Generic Mantis dispatcher mislabels the synthetic baseline

- Kind: bug
- File: `.github/workflows/mantis-scenario.yml`
- Line: 73
- Evidence: the generic dispatcher keeps `baseline_ref` defaulted to `0bf06e953fdda290799fc9fb9244a8f67fdae593`, then passes it to `mantis-discord-thread-attachment.yml`. The new target workflow treats `baseline_ref` only as a display label for a synthetic baseline created by patching the candidate worktree, and its direct default is `synthetic-reverted-thread-filepath-fix`.
- Impact: if a maintainer runs the new scenario through `Mantis Scenario` without overriding `baseline_ref`, the report and PR comment will claim the baseline is `0bf06e...` even though that ref was never checked out for the baseline lane.
- Suggested fix: make `mantis-scenario.yml` use a scenario-specific baseline default for this case, or leave the generic input blank and let each target workflow apply its own default.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. A focused shell check confirmed `pnpm --dir <dir>` changes the child cwd, and the QA CLI source resolves relative `--repo-root` from that cwd.
- Is this the best way to solve the issue? Yes. Passing an absolute repo root preserves the intended candidate worktree while keeping the lane on that worktree’s dependency graph.
