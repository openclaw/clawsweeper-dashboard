# Commit 562d460d

## Summary

I found one medium-confidence compatibility/reliability risk in the new timeout selection. The fix is intentionally trying to fail fast after post-tool raw assistant silence, but it now uses a hard-coded, non-user-configurable 10s assistant idle budget for a path that the existing config/docs describe as controlled by `appServer.turnCompletionIdleTimeoutMs`.

## Findings

### Medium: Post-tool raw assistant silence bypasses the documented configurable timeout

- Kind: compatibility
- File: `extensions/codex/src/app-server/run-attempt.ts`
- Line: 1556
- Evidence: `postToolRawAssistantCompletionNeedsTerminalGuard` arms the completion watchdog with `timeoutMs: turnAssistantCompletionIdleTimeoutMs`. That value is resolved only from test/internal options or the hard-coded `CODEX_TURN_ASSISTANT_COMPLETION_IDLE_TIMEOUT_MS = 10_000`; it is not part of `CodexPluginConfig.appServer`, schema keys, or runtime config. Current docs still say `appServer.turnCompletionIdleTimeoutMs` is the quiet window to raise for “slow post-tool or status-only synthesis phases,” and say post-tool raw assistant progress waits for `turn/completed` or the terminal watchdog.
- Impact: A deployment that raised `appServer.turnCompletionIdleTimeoutMs` above 60s to tolerate slow post-tool/status-only synthesis can still be interrupted after about 10s if Codex emits a post-tool raw assistant completion and then goes quiet before `turn/completed`. The regression test itself uses this status-like payload: `"I'm writing the report now."`
- Suggested fix: Either use `turnCompletionIdleTimeoutMs` for this post-tool raw-assistant guard, or expose/document a production `appServer.turnAssistantCompletionIdleTimeoutMs` knob and update the Codex harness docs/reference so operators can tune the new 10s behavior.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes at source level: configure a long `appServer.turnCompletionIdleTimeoutMs`, run a turn that crosses a tool handoff, emit `rawResponseItem/completed` with an assistant message, then delay `turn/completed` beyond 10s. The new branch arms the watchdog with the assistant timeout override, not the configured completion timeout.
- Is this the best way to solve the issue? Unclear. The fail-fast behavior fixes #82775, but the timeout budget should either follow the existing documented config contract or add an explicit new config contract.
