# Commit afd7339

## Summary

Found one concrete regression in the session-boundary fix. The new timestamp boundary filters Telegram “Conversation context”, but the older replied-to message is still sent through the separate Telegram reply-chain path.

## Findings

### Medium: Session-start boundary still leaks stale Telegram reply chains

- Kind: bug
- File: `extensions/telegram/src/bot-handlers.runtime.ts`
- Line: 821
- Evidence: `handleInboundMessageLike` now resolves `promptContextMinTimestampMs` from the session entry and passes it into inbound processing, but `processMessageWithReplyChain` still builds `replyChainNodes` first and passes the resulting `replyChain` to `processMessage` unchanged. The timestamp boundary is only applied when building `promptContext`. Downstream, `extensions/telegram/src/bot-message-context.session.ts:305` turns any nonempty `replyChain` into `visibleReplyChain`, appends it to the model body at `extensions/telegram/src/bot-message-context.session.ts:331`, and exposes it as `ReplyChain` at `extensions/telegram/src/bot-message-context.session.ts:502`. The new regression test replies to stale message `84670`, but only asserts `UntrustedStructuredContext`, so it would miss the same stale text in `payload.ReplyChain` and the envelope body.
- Impact: A user can reply to a pre-reset or pre-session Telegram message and still carry that stale text into the new session, despite the new boundary. That can resurrect old instructions or sensitive prior context after a reset.
- Suggested fix: Apply the same session/reset boundary to reply-chain nodes before `resolveReplyMediaForChain` and before `replyChain` reaches `processMessage`. Add assertions to the new bot test that `payload.ReplyChain` and `payload.Body` do not contain the stale instruction.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. Extend the new `omits stale Telegram topic context before the persisted session start` test to assert `payload.ReplyChain` and `payload.Body` do not contain `staleInstruction`; the current code path still builds the stale reply target before applying the prompt-context timestamp filter.
- Is this the best way to solve the issue? Yes. Filtering only `UntrustedStructuredContext` leaves another first-class model input unbounded; the boundary needs to be enforced at reply-chain construction or immediately before reply-chain formatting.
