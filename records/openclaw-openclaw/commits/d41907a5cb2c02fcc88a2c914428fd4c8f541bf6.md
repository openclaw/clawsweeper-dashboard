## Findings

### Medium: Malformed Slack media redirects are only handled in an unused helper

- Kind: bug
- File: `extensions/slack/src/monitor/media.ts`
- Line: 106
- Evidence: The new try/catch is in `fetchWithSlackAuth`, but `rg` shows that helper is only called by `extensions/slack/src/monitor/media.test.ts`. Production Slack media downloads call `resolveSlackMedia`, which reaches `downloadSlackMediaFile`, then `createSlackMediaFetch`, then `saveRemoteMedia` at `extensions/slack/src/monitor/media.ts:310`. That path uses the shared guarded fetch implementation, whose redirect handling still does `new URL(location, parsedUrl)` without a catch at `src/infra/net/fetch-guard.ts:508`. A focused Node check confirmed the added test value, `http://[::1`, throws `TypeError: Invalid URL`.
- Impact: A Slack private-file response with a malformed `Location` header still fails the real media download path instead of being treated as an unfollowable redirect. The new regression test passes while exercising only the unused helper, so this can ship with the user-visible Slack media behavior unchanged.
- Suggested fix: Move the malformed-redirect handling to the production path, either in the guarded fetch redirect parser or in SlackŌĆÖs media fetch wrapper, and add a `resolveSlackMedia` test where the first fetch returns `302 Location: http://[::1`.
- Confidence: high
