# Commit 863fe90

## Summary

Found one workflow regression in the website installer sync path. The first deletion sync can work, but later installer syncs can fail after `public/install.cmd` is already gone from `openclaw.ai`.

## Findings

### Medium: Future installer syncs can fail after `install.cmd` is removed from the website repo

- Kind: reliability
- File: `.github/workflows/website-installer-sync.yml`
- Line: 199
- Evidence: The workflow removes `openclaw.ai/public/install.cmd` at line 155, but the commit step still runs `git add public/install.sh public/install-cli.sh public/install.ps1 public/install.cmd`. In a temp git repo matching the future state where `public/install.cmd` is no longer tracked and does not exist, `git add public/install.sh public/install.cmd` fails with `fatal: pathspec 'public/install.cmd' did not match any files` and exits 128.
- Impact: After the website repo has accepted the deletion once, any later `scripts/install.sh`, `scripts/install-cli.sh`, or `scripts/install.ps1` change that triggers `Website Installer Sync` can fail during the push step, leaving `openclaw.ai` serving stale installer scripts.
- Suggested fix: Stage existing installer files separately and use an ignore-safe removal for the deleted CMD installer, for example `git add public/install.sh public/install-cli.sh public/install.ps1` plus `git rm -f --ignore-unmatch public/install.cmd`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. In a temp repo with `public/install.sh` tracked and `public/install.cmd` absent/untracked, changing `public/install.sh` then running the workflow’s `git add public/install.sh public/install.cmd` exits 128.
- Is this the best way to solve the issue? yes. `git rm -f --ignore-unmatch public/install.cmd` stages the deletion when it exists and stays successful after the path is already removed.
