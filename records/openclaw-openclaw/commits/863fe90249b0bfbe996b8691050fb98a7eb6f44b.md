# Commit 863fe902

## Summary

Found one repeat-run regression in the website installer sync workflow. The commit correctly removes `install.cmd`, but the later staging step still hard-requires that path to exist or be tracked.

## Findings

### Medium: Website installer sync now fails after `install.cmd` has already been removed

- Kind: reliability
- File: `.github/workflows/website-installer-sync.yml`
- Line: 199
- Evidence: The sync step removes `openclaw.ai/public/install.cmd`, but the commit step still runs `git add public/install.sh public/install-cli.sh public/install.ps1 public/install.cmd`. `openclaw.ai` currently no longer lists `public/install.cmd` in `public/` via the GitHub Contents API (`https://api.github.com/repos/openclaw/openclaw.ai/contents/public`). A local git repro with the same absent, untracked path exits `128` with `fatal: pathspec 'public/install.cmd' did not match any files`.
- Impact: The first sync could succeed while `public/install.cmd` was still tracked, but any later installer sync with `changed=true` will fail in the commit step once the file is absent from `openclaw.ai`, leaving public website installer files stale.
- Suggested fix: Stage the remaining installer files normally, and remove the legacy file with an idempotent deletion command, for example `git add public/install.sh public/install-cli.sh public/install.ps1` followed by `git rm --ignore-unmatch public/install.cmd`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. In a temp git repo with only `public/install.sh`, `public/install-cli.sh`, and `public/install.ps1` tracked, modifying `public/install.sh` and running the workflow’s `git add public/install.sh public/install-cli.sh public/install.ps1 public/install.cmd` fails with exit `128`.
- Is this the best way to solve the issue? yes. `git rm --ignore-unmatch public/install.cmd` handles both states: staging the deletion when the file is still tracked, and no-oping once it is already gone.
