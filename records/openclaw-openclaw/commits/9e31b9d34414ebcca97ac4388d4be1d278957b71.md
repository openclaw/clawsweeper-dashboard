# Commit 9e31b9d

## Summary

This commit adds `file_path` aliases to the edit recovery helper, but the exposed OpenClaw edit tools still reject alias-only edit calls before that recovery helper can run. The new regression test exercises `wrapEditToolWithRecovery()` directly with a fake base tool, so it does not prove the production `createOpenClawCodingTools()` path that PR #81909 and issue #80066 depend on.

## Findings

### Medium: `file_path` edit recovery is unreachable through production edit tools

- Kind: bug
- File: `src/agents/pi-tools.host-edit.ts`
- Line: 63
- Evidence: The commit makes `readEditToolParams()` accept `file_path`, but production edit tools are returned as `wrapToolParamValidation(withRecovery, REQUIRED_PARAM_GROUPS.edit)` in `src/agents/pi-tools.read.ts:721` and `src/agents/pi-tools.read.ts:739`. `REQUIRED_PARAM_GROUPS.edit` still requires canonical `path` in `src/agents/pi-tools.params.ts:87`, so alias-only calls fail before `wrapEditToolWithRecovery()` runs. A live probe against `createOpenClawCodingTools({ workspaceDir })` with `{ file_path: "demo.txt", edits: [...] }` returned `Missing required parameter: path (received: file_path, edits=<array>)` and left the file unchanged; the same probe with `{ path: "demo.txt", edits: [...] }` succeeded and wrote the file.
- Impact: The merged commit can close #80066 while not fixing the exposed edit-tool boundary. If the reported Feishu warning path depends on `file_path` edit calls, users still see a tool failure rather than recovery; if production never accepts `file_path`, the new alias recovery code is only covered by a helper-level synthetic test and not the actual runtime path.
- Suggested fix: Add a production-path regression test using `createOpenClawCodingTools()` or `createHostWorkspaceEditTool()` for the exact failing call shape. Then either normalize the accepted alias to canonical `path` before validation/base edit execution, or keep canonical-only validation and fix the real warning/recovery path that produces the post-write failure.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Construct the production edit tool via `createOpenClawCodingTools({ workspaceDir })`, call `edit.execute()` with `file_path` and valid `edits`, and observe the missing-`path` failure before any write. The canonical `path` version succeeds in the same probe.
- Is this the best way to solve the issue? unclear. If `file_path` is intended to be supported, normalization must happen before validation and before the underlying edit tool executes. If canonical `path` remains the contract, this commit should not claim to fix `file_path` production recovery and should instead target the actual Feishu false-warning source.
