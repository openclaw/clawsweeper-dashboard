# Commit d92501d

## Summary

Found one regression in the new metadata-driven PDF text-extraction fallback path: text-only MiniMax CN alias configurations can be bypassed and replaced with the canonical global MiniMax provider.

## Findings

### Medium: PDF fallback drops text-only MiniMax CN execution aliases

- Kind: regression
- File: `src/agents/tools/pdf-tool.model-config.ts`
- Line: 123
- Evidence: `resolveTextExtractionCandidateRefs` now seeds PDF text-extraction candidates from `params.primary.provider` plus `resolveAutoMediaKeyProviders({ capability: "image" })`. `resolveAutoMediaKeyProviders` only inserts configured execution aliases such as `minimax-cn` when the configured model has `input: ["image"]`; text-only configured providers are not inserted. The fallback then asks for the canonical `minimax` metadata and does not find the configured `minimax-cn` text model because provider normalization for model config does not collapse `minimax-cn` to `minimax`.
- Impact: A user with default model `openai/gpt-5.4` and a configured text-only `models.providers["minimax-cn"]` PDF extraction model no longer gets `minimax-cn/...`; the resolver returns `minimax/MiniMax-M2.7` and `minimax-portal/MiniMax-M2.7`. That loses the CN execution provider/base URL and can send PDF fallback through the wrong MiniMax endpoint or fail auth/region routing. The removed implementation iterated configured MiniMax providers directly, so it preserved `minimax-cn` when present.
- Suggested fix: Include configured providers with document text-extraction metadata, not only image-capable configured providers, or explicitly scan `cfg.models.providers` for providers whose `normalizeMediaProviderId(providerKey)` has a PDF `textExtraction` document model while preserving `normalizeMediaExecutionProviderId(providerKey)` in the emitted ref.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A one-off probe with `MINIMAX_API_KEY=minimax-test`, default `openai/gpt-5.4`, and `models.providers["minimax-cn"]` containing only `MiniMax-M2.7-highspeed` with `input: ["text"]` returned:
  `{"primary":"minimax/MiniMax-M2.7","fallbacks":["minimax-portal/MiniMax-M2.7"]}`.
- Is this the best way to solve the issue? yes. The PDF fallback policy is now metadata-driven, but the candidate enumeration still needs to preserve execution aliases from config; fixing enumeration keeps policy in metadata without reintroducing MiniMax-specific hardcoding.
