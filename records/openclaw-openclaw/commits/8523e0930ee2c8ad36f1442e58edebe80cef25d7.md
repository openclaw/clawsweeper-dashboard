# Commit 8523e09

## Summary

Found one medium-confidence regression risk: the patch correctly keeps tool-only interrupted turns eligible for no-visible-answer handling, but it also makes interrupted turns with partial assistant text look like successful `stop` replies.

## Findings

### Medium: Interrupted partial replies can now be promoted as successful completions

- Kind: regression
- File: `extensions/codex/src/app-server/event-projector.ts`
- Line: 333
- Evidence: Current `buildResult()` returns `aborted: this.aborted` and no longer considers `completedTurn.status === "interrupted"`. `createAssistantMessage()` then assigns `stopReason: "stop"` whenever `this.aborted` and `promptError` are false. A focused projector probe with `turn/completed status="interrupted"` plus an `agentMessage` item produced `aborted=false`, `promptError=null`, `assistantTexts=["partial answer"]`, and `lastAssistant.stopReason="stop"`. The base commit set `this.aborted = true` for interrupted turns and returned `aborted: this.aborted || turnInterrupted`.
- Impact: If Codex app-server emits `interrupted` after streaming partial assistant text without an OpenClaw-side abort signal, OpenClaw can now record/emit that partial text as a normal successful answer. That bypasses both aborted handling and the no-visible-answer guard, so users may see an incomplete response treated as final.
- Suggested fix: Keep the no-visible-answer path unblocked, but preserve raw interrupted terminal state separately from success. For example, carry an `interrupted`/non-success terminal classification into the attempt result and avoid assigning `stopReason: "stop"` or lifecycle `success` for interrupted turns that contain assistant text unless there is explicit evidence the text is complete.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes, at source level: instantiate `CodexAppServerEventProjector`, feed an interrupted `turn/completed` containing one `agentMessage`, then inspect `buildResult()`. The observed result was `aborted=false`, `promptError=null`, and `stopReason="stop"`.
- Is this the best way to solve the issue? unclear. The commit’s intended explicit-abort split is sound for tool-only/no-answer turns, but interrupted status probably needs a distinct terminal signal so partial visible text does not become indistinguishable from a completed answer.
