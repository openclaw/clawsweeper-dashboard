# Commit 1d772f3b1a

## Summary

Found one release-test regression in the Open WebUI Docker timeout calculation. The default local path is extended, but the release-path lane that sets a shorter provider timeout is now more likely to be killed by the Docker wrapper before the scriptŌĆÖs own probe budgets are exhausted.

## Findings

### Low: Release Open WebUI lane can time out before its own probe budget

- Kind: regression
- File: `scripts/e2e/openwebui-docker.sh`
- Line: 26
- Evidence: `DOCKER_COMMAND_TIMEOUT` now defaults to `PROVIDER_TIMEOUT_SECONDS + 90`. The release Open WebUI lane sets `OPENCLAW_OPENWEBUI_PROVIDER_TIMEOUT_SECONDS=300` in `scripts/lib/docker-e2e-scenarios.mjs`, so this commit reduces that laneŌĆÖs Docker command timeout from the old fixed `600s` to `390s`. The final `docker exec` runs `openwebui-probe.mjs`, which can wait `72 * 5000ms = 360s` for Open WebUI to expose the model before starting the chat request, and the chat request gets `PROVIDER_TIMEOUT_SECONDS * 1000 + 60000 = 360000ms`.
- Impact: a valid but slow release-path run can be killed by GNU `timeout` around 390 seconds, even though the Open WebUI probe still has allowed wait time left. For example, 60-120 seconds of Open WebUI model propagation plus a slow but still in-budget 300-second provider response passed more often under the previous 600-second wrapper but can fail after this commit.
- Suggested fix: make the Docker command timeout cover the final probeŌĆÖs full worst-case budget, or preserve a floor higher than the previous 600 seconds for release-path runs. A direct fix would account for model retry budget plus fetch timeout plus buffer, rather than deriving the Docker timeout only from provider timeout.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Focused arithmetic check with `OPENCLAW_OPENWEBUI_PROVIDER_TIMEOUT_SECONDS=300` gives `fetch_timeout=360000ms`, `docker_command_timeout=390s`, and `model_wait_budget=360s`; the wrapper timeout is shorter than the probeŌĆÖs possible model-wait-plus-chat path.
- Is this the best way to solve the issue? no. The intent is to extend the Docker wrapper around slow provider calls, but tying it only to provider timeout ignores the probeŌĆÖs preceding Open WebUI model retry loop and regresses the release-path override.
