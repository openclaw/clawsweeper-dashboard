# Commit e179525

## Summary

Found one compatibility regression risk: removing the external Dockerfile frontend pin makes the root Dockerfile depend on the builder’s bundled frontend, but the file still uses `RUN --mount` variable expansion that older Dockerfile frontends do not support.

## Findings

### Medium: Root Dockerfile can fail on older bundled Dockerfile frontends

- Kind: compatibility
- File: `Dockerfile`
- Line: 31
- Evidence: The commit removes `# syntax=docker/dockerfile:1.7`, while `Dockerfile:31` still uses `${OPENCLAW_BUNDLED_PLUGIN_DIR}` inside `RUN --mount` options. Docker documents `RUN --mount` as Dockerfile v1.2, but variable expansion in `RUN --mount` option values was added in Dockerfile v1.3. Docker also documents that the external syntax image is what keeps builders on the same frontend and allows latest features without updating Docker Engine. ([docs.docker.com](https://docs.docker.com/reference/builder))
- Impact: Users whose Docker/BuildKit bundled frontend is effectively v1.2 can no longer rely on the removed `docker/dockerfile:1.7` directive. A build reaches the first `RUN --mount` and treats `${OPENCLAW_BUNDLED_PLUGIN_DIR}` literally, failing with `"/${OPENCLAW_BUNDLED_PLUGIN_DIR}": not found`. The public Docker install docs only require Docker Desktop or Docker Engine plus Compose v2, with no frontend/Docker Engine minimum at `docs/install/docker.md:19`.
- Suggested fix: Either keep a syntax directive for the root Dockerfile, or rewrite the first `RUN --mount` so mount options do not require variable expansion. If the intended contract is “no external frontend pull”, document and enforce a minimum Docker/BuildKit frontend of at least v1.3.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. `docker buildx build --target workspace-deps --build-arg BUILDKIT_SYNTAX=docker/dockerfile:1.2 -f Dockerfile .` fails at the dynamic bind mount with `"/${OPENCLAW_BUNDLED_PLUGIN_DIR}": not found`; the same target with `docker/dockerfile:1.3` succeeds.
- Is this the best way to solve the issue? Unclear. Restoring the directive is the smallest compatibility fix, but it conflicts with this commit’s goal of avoiding an external frontend pull. Removing the v1.3-only mount interpolation preserves that goal.
