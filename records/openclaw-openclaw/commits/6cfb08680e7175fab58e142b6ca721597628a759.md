# Commit 6cfb086

## Summary

Found one low-severity transport regression. The stdio shutdown path works and the focused tests pass, but the shared close helper also serves WebSocket transports, where closing `stdin` does not close the socket.

## Findings

### Low: WebSocket app-server clients now remain connected until the force-kill timer

- Kind: reliability
- File: `extensions/codex/src/app-server/transport.ts`
- Line: 31
- Evidence: `closeCodexAppServerTransport()` now only calls `child.stdin.end?.()` / `child.stdin.destroy?.()` and defers `signalCodexAppServerTransport(child, "SIGKILL")` until `forceKillDelayMs`. For WebSocket transports, `stdin` is only a local `Writable`; the real socket is closed only by `kill`, via `socket.close()` in `extensions/codex/src/app-server/transport-websocket.ts:71`. A focused probe with `forceKillDelayMs: 25` showed the server-side WebSocket was still open after 5 ms and closed only after the fallback delay.
- Impact: configured WebSocket app-server users get delayed client teardown, defaulting to about 1 second. During shared-client rotation or short-lived CLI disposal, old sockets can overlap with new clients and hold server/client resources longer than before.
- Suggested fix: keep the stdin-first stdio behavior, but make WebSocket close immediate. Either give transports an explicit graceful close method, have the WebSocket transport’s `stdin.destroy()` close the socket, or branch the fallback so non-process transports call `kill`/`close` immediately.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Create a local `WebSocketServer`, connect through `createWebSocketTransport()`, call `closeCodexAppServerTransport(transport, { forceKillDelayMs: 25 })`, and observe that the server-side `close` event does not fire until the fallback timer.
- Is this the best way to solve the issue? unclear. The stdio change is appropriate for managed Codex app-server shutdown; the safer fix is to separate stdio EOF semantics from WebSocket socket-close semantics.
