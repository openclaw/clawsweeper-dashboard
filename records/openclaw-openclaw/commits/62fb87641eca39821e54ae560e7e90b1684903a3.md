# Commit 62fb876

## Summary

Found two regressions in the QQBot slash-command changes. The serious one is in the new pre-dispatch admin authorization: mixed wildcard allowlists authorize everyone for sensitive commands.

## Findings

### High: Mixed wildcard allowlists authorize any sender for QQBot admin commands

- Kind: security
- File: `extensions/qqbot/src/engine/commands/slash-command-auth.ts`
- Line: 41
- Evidence: `resolveSlashCommandAuth()` only checks that the normalized list has at least one non-wildcard entry, then passes the original list to `createQQBotSenderMatcher()`. That matcher treats any `"*"` entry as allow-all, so `allowFrom: ["*", "OWNER"]` authorizes `ATTACKER`. I verified with a focused smoke: `wildcardOnly=false`, `mixedWildcard=true`, `owner=true`, and a fake `requireAuth` registry command executed as `ATTACKER` with `allowFrom: ["*", "OWNER"]`.
- Impact: Operators can reasonably use `allowFrom: ["*", "OWNER"]` or `dmPolicy: "open"` plus an explicit owner to keep chat open while limiting admin commands, which is exactly the intent described in this new file. With this commit, any sender can run `requireAuth` QQBot commands such as `/bot-logs`, `/bot-clear-storage`, and `/bot-approve` in that configuration, enabling log disclosure, file deletion, or approval-policy changes.
- Suggested fix: For slash-command admin auth, remove `"*"` before matching and require the sender to match a concrete entry. Also consider passing `dmPolicy`/`groupPolicy` through this pre-dispatch auth path or delegating to `resolveQQBotAccess()` so explicit disabled policies still fail closed before command execution.
- Confidence: high

### Low: `/bot-version` is now blocked in group chats

- Kind: regression
- File: `extensions/qqbot/src/engine/commands/builtin/register-basic.ts`
- Line: 73
- Evidence: The commit added `c2cOnly: true` to `/bot-version`, but the commit message’s c2cOnly list names `/bot-upgrade`, `/bot-streaming`, `/bot-logs`, `/bot-clear-storage`, and `/bot-approve`, not `/bot-version`. Before the commit, group help only hid `bot-upgrade`, `bot-clear-storage`, and `bot-streaming`; `/bot-version` was available. A focused smoke of `matchSlashCommand()` for a group `/bot-version` now returns `💡 请在私聊中使用此指令`.
- Impact: Group users lose a harmless diagnostic command and `/bot-help` hides it in groups because help filters all `c2cOnly` commands.
- Suggested fix: Remove `c2cOnly: true` from `/bot-version` unless version disclosure in groups is intentionally prohibited, and add a small command-registry test for group-visible basic commands.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. For the auth issue, call `resolveSlashCommandAuth({ senderId: "ATTACKER", isGroup: false, allowFrom: ["*", "OWNER"] })`; it returns `true`, and a `requireAuth` pre-dispatch registry command executes. For `/bot-version`, call `matchSlashCommand()` with `type: "group"` and `rawContent: "/bot-version"`; it returns the private-chat-only rejection.
- Is this the best way to solve the issue? yes. The auth bug is local to the new pre-dispatch authorization helper and should be fixed by matching only concrete entries for admin commands. The version regression is a one-line command metadata fix unless maintainers explicitly want group version blocking.
