# Commit 466f718

## Summary

One actionable regression risk found: the commit makes native Talk PTT node commands reachable, but the iOS `pttStart` failure path can leave Voice Wake suspended.

## Findings

### Medium: Failed iOS PTT start can leave Voice Wake suspended

- Kind: reliability
- File: `apps/ios/Sources/Model/NodeAppModel.swift`
- Line: 1460
- Evidence: This commit allowlists `talk.ptt.*` for Talk-capable nodes in `src/gateway/node-command-policy.ts:206` and iOS now advertises the `talk` capability in `apps/ios/Sources/Gateway/GatewayConnectionController.swift:779`. The iOS PTT start handler suspends Voice Wake at `apps/ios/Sources/Model/NodeAppModel.swift:1460`, then calls `try await self.talkMode.beginPushToTalk()` at line 1461. `beginPushToTalk()` can throw on microphone denial, speech permission denial, or recognition/audio-session startup failure at `apps/ios/Sources/Voice/TalkModeManager.swift:321`, `apps/ios/Sources/Voice/TalkModeManager.swift:328`, and `apps/ios/Sources/Voice/TalkModeManager.swift:339`. Unlike `pttOnce`, `pttStart` has no `defer`/`catch` cleanup; the outer invoke catch only returns an unavailable response at `apps/ios/Sources/Model/NodeAppModel.swift:854`. Voice Wake suspension tears down recognition and only resumes through `resumeAfterExternalAudioCapture` when called later, as shown in `apps/ios/Sources/Voice/VoiceWakeManager.swift:224`.
- Impact: If Voice Wake is active and a remote/native PTT start fails, Voice Wake can remain paused until another explicit stop/cancel path or user action resumes it. First-use permission denial and speech/audio startup errors are plausible failure modes.
- Suggested fix: Mirror the `pttOnce` cleanup pattern for `pttStart`: use a local suspended flag or `do/catch` around `beginPushToTalk()`, resume Voice Wake and clear `pttVoiceWakeSuspended` on any thrown start failure. Also consider clearing `activePTTCaptureId` inside `beginPushToTalk()` failure cleanup.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for a focused iOS unit path: enable/listen with Voice Wake, invoke `talk.ptt.start`, force `beginPushToTalk()` to throw before it marks PTT active, then assert Voice Wake is resumed. I did not live-verify on an iOS device.
- Is this the best way to solve the issue? Yes; cleanup belongs around the owner-owned iOS PTT start handoff because Android already has analogous failure cleanup in `runPreparedTalkPttCommand`.
