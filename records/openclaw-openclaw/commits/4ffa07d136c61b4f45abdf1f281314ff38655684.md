# Commit 4ffa07d

## Summary

Found two actionable issues: a realtime tool-policy bypass for the new Discord control tool, and a committed e2e test that times out under the repo’s e2e Vitest config.

## Findings

### Medium: `toolPolicy: "none"` still executes `openclaw_agent_control` tool calls

- Kind: security
- File: `extensions/discord/src/voice/realtime.ts`
- Line: 962
- Evidence: `resolveDiscordRealtimeVoiceAgentConsultTools()` only advertises `openclaw_agent_control` when the policy is not `"none"` (`extensions/discord/src/voice/realtime.ts:46`), but `handleToolCall()` executes `REALTIME_VOICE_AGENT_CONTROL_TOOL_NAME` before the existing `"none"` rejection for consult tools (`extensions/discord/src/voice/realtime.ts:962`). That path then calls `controlRealtimeVoiceAgentRun()` and can steer/cancel the active embedded run (`extensions/discord/src/voice/realtime.ts:1062`).
- Impact: a buggy, custom, or hallucinating realtime provider can emit an unadvertised `openclaw_agent_control` call and still control the active run even when the operator disabled realtime tools with `voice.realtime.toolPolicy: "none"`.
- Suggested fix: reject control calls when `consultToolPolicy === "none"`, or more generally track the advertised tool names for the bridge session and reject any unadvertised tool event before dispatch.
- Confidence: high

### Medium: Added STT/TTS active-run control e2e test times out

- Kind: reliability
- File: `extensions/discord/src/voice/manager.e2e.test.ts`
- Line: 3771
- Evidence: `node scripts/run-vitest.mjs --config test/vitest/vitest.e2e.config.ts extensions/discord/src/voice/manager.e2e.test.ts -t "handles semantic realtime agent-control tool calls in Discord VC|routes active-run"` ran the new manager-level control cases; two passed, but `routes active-run STT/TTS transcripts to voice control before agent turns` timed out after 120000ms at `extensions/discord/src/voice/manager.e2e.test.ts:3771`.
- Impact: the changed e2e coverage is not reliably runnable under the repo’s e2e config and can block the e2e lane or mask regressions in the STT/TTS control path.
- Suggested fix: make the STT/TTS test settle deterministically. If the hang is in production flow, fix `processDiscordVoiceSegment`/playback scheduling; if it is test-only, adjust the mocked playback/voice state path so `processVoiceSegment()` returns and no pending playback promise holds the test open.
- Confidence: high

- Is this the best way to solve the issue? yes for the policy bypass: local dispatch must enforce the same policy as tool advertisement. For the timeout, the safest fix depends on whether the hang is production or test harness, but the failing test gives a focused starting point.
