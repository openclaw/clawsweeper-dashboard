# Commit eb73486a94

## Summary

The added Codex home precreation breaks the Mantis Telegram Desktop Proof workflow before the agent starts.

## Findings

### Medium: Precreated Codex home makes `codex-action` fail during home resolution

- Kind: reliability
- File: `.github/workflows/mantis-telegram-desktop-proof.yml`
- Line: 349
- Evidence: The new step precreates the `codex` user's `.codex` directory with restrictive ownership/mode. The pinned `openai/codex-action` first tries `fs.stat(codexHome)`, catches stat errors, then runs plain `sudo mkdir` instead of `mkdir -p`; if the directory exists but is not visible to the runner process, that fails with “File exists”. A live run on this exact SHA reached the changed job and failed at `resolve-codex-home` immediately after the new prepare step ran: https://github.com/openclaw/openclaw/actions/runs/25652068950/job/75292342605. The relevant action source is `openai/codex-action` at `5c3f4ccdb2b8790f73d6b21751ac00e602aa0c02`, `src/main.ts` lines 418-432.
- Impact: Any non-skipped Mantis Telegram Desktop Proof run using this workflow fails before Codex starts, so the workflow cannot produce native Telegram proof artifacts.
- Suggested fix: Do not precreate the default unprivileged Codex home in a way the action cannot stat. Either let `codex-action` create its expected home, or pass `codex-home` to a shared directory and grant both the runner-side proxy/config steps and the `codex` user the exact access they need.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Re-run the workflow path that reaches `Run Codex Mantis Telegram agent`; run `25652068950` already reproduces the failure on the reviewed SHA.
- Is this the best way to solve the issue? unclear. The safest direction is to avoid precreating the action’s default home; if a shared home is required, use an explicit `codex-home` contract and permissions tested against the pinned action.
