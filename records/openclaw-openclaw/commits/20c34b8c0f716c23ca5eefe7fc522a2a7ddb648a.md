# Commit 20c34b8

## Summary

Found one concrete gap in the new implicit announce preflight: route modifiers such as `accountId` or `threadId` are treated as an explicit delivery target, so an account-only or thread-only announce job can still execute the model before failing delivery resolution.

## Findings

### Medium: Account/thread-only announce jobs bypass the new unresolved-route preflight

- Kind: bug
- File: `src/cron/isolated-agent/run.ts`
- Line: 324
- Evidence: `hasExplicitCronDeliveryTarget()` returns true when only `plan.accountId` or `plan.threadId` is set. The new early-fail guard only runs when `!hasExplicitCronDeliveryTarget(...)`, so jobs with `delivery.mode="announce"`, implicit `channel="last"`, no `to`, and only `accountId` or only `threadId` skip the preflight. The CLI can create this shape: `src/cli/cron-cli/register.cron-add.ts:261` builds delivery from `--account`/`--thread-id`, and `src/cli/cron-cli.test.ts:382` confirms `--account` alone is accepted. `resolveDeliveryTarget()` still cannot route without a channel when there is no previous/single fallback channel, returning `Channel is required when delivery.channel=last has no previous channel.` at `src/cron/isolated-agent/delivery-target.ts:226`.
- Impact: A recurring isolated announce job created with `--account` or `--thread-id` but without `--channel`/`--to` can still spend model tokens before hitting the same permanent delivery-target error this commit intended to avoid.
- Suggested fix: Split ÔÇťexplicit targetÔÇŁ from ÔÇťtarget modifiers.ÔÇŁ For the preflight, treat `accountId` and `threadId` alone as still implicit unless accompanied by a route-defining field such as a non-`last` channel or a recipient target that can select/resolve a channel. Add regression tests for announce-mode account-only and thread-only jobs.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes, from source inspection of the exact call path: CLI accepts account-only delivery, delivery planning produces `channel="last"` plus `accountId`, the guard treats that as explicit, and target resolution can still fail before delivery because account id does not provide a channel or recipient.
- Is this the best way to solve the issue? Yes. The narrow fix is to change the guardÔÇÖs route-definition predicate, not to remove account/thread support; those fields remain useful once a real channel/recipient route exists.
