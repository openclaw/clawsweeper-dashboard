# Commit 626e078

## Summary

Found one medium-confidence privacy/regression risk in the new Codex auth repair path.

## Findings

### Medium: Invalidated Codex repair can switch a per-agent profile to a different OpenAI account

- Kind: privacy
- File: `src/agents/auth-profiles/persisted.ts`
- Line: 279
- Evidence: `findMainStoreOAuthReplacementForInvalidatedProfile` accepts any usable main-store `openai-codex` OAuth profile, then picks base order/lastGood/single candidate, without checking `accountId` or `email` compatibility. The adjacent legacy repair path does have identity safeguards through `isSafeToAdoptMainStoreOAuthIdentity` / conflict checks. The new regression test also encodes a cross-identity switch: stale per-agent `val@viewdue.ai` is replaced by main `bunsthedev@gmail.com`, and order/lastGood are rewritten to the main profile.
- Impact: A per-agent `auth.order.openai-codex` can intentionally pin a worker to a separate Codex account. After any recorded auth invalidation signal on that profile, runtime merge can silently route that agent to the main agent’s healthy Codex profile instead. That can send prompts/code to the wrong OpenAI account and affect billing, org entitlements, or account-specific data boundaries. The trigger is also broad: `hasAuthInvalidationSignal` treats historical `failureCounts.auth` as enough even if the cooldown window is no longer active.
- Suggested fix: Require same `accountId` or normalized `email` before adopting a main-store replacement, reusing the existing OAuth identity helpers. If cross-account relogin is a desired repair, keep it behind an explicit local login/promotion in that agent or an explicit doctor/fix action rather than automatic merge-time repair.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. The added test in `src/agents/auth-profiles.ensureauthprofilestore.test.ts` constructs separate main/per-agent Codex profiles with different emails and confirms the per-agent order is switched to the main profile.
- Is this the best way to solve the issue? No. The repair should preserve the existing identity-safety invariant for OAuth adoption, or make cross-identity repair explicit.
