# Commit e1fec3c

## Summary

Found two concrete issues: the new BlueBubbles Zod refinement is not reached by the real plugin-aware config validator, and the committed Plugin SDK API baseline hash is stale.

## Findings

### Medium: BlueBubbles policy refinements are bypassed by real config validation

- Kind: bug
- File: `src/config/validation.ts`
- Line: 1267
- Evidence: `extensions/bluebubbles/src/config-schema.ts:139` adds `superRefine` checks for `dmPolicy="allowlist"` and `dmPolicy="open"`, and `BlueBubblesConfigSchema.safeParse({ dmPolicy: "allowlist" })` rejects as intended. But `validateConfigObjectWithPlugins({ channels: { bluebubbles: { dmPolicy: "allowlist" }}})` returns `ok: true`, and so do `dmPolicy: "open"` with no `"*"` wildcard and account-level `accounts.work.dmPolicy="allowlist"` without inherited `allowFrom`. The validator path at `src/config/validation.ts:1267` only runs `validateJsonSchemaValue(...)` over channel JSON schema metadata, so Zod `superRefine` runtime checks are not applied.
- Impact: users can save BlueBubbles configs that the plugin-owned schema says are invalid. At runtime these policies can silently block all DMs instead of failing validation with the intended actionable error.
- Suggested fix: make plugin-aware channel validation use `channelConfig.runtime.safeParse` when available, or add an equivalent generated/runtime validation seam for channel refinements that JSON Schema cannot express. Add a `validateConfigObjectWithPlugins` regression test for BlueBubbles top-level and account-level `dmPolicy` constraints.
- Confidence: high

### Medium: Plugin SDK API baseline hash is stale

- Kind: regression
- File: `docs/.generated/plugin-sdk-api-baseline.sha256`
- Line: 1
- Evidence: `pnpm plugin-sdk:api:check` fails on current `main` with `Plugin SDK API baseline drift detected`. Re-rendering the baseline without mutating the repo produced expected hashes `259c4d22d5fe84ca6b27f9d061dddca4459e4eddd516729564f250d823dbd819` and `b2bf16d3c5859b06240b5f41ed5803af8e82338da3e56ace1569520ac0732a5a`, but the committed file has `25d5e7d7532d99d06187813e1cf75d3192fb2bf330862bfd1d4d4a50c91c77c9` and `9f9758f36694c4004ab8c2fb772815c91cc2035de872a0e53f053537b2156910`.
- Impact: Workflow Sanity and release checks that run `pnpm plugin-sdk:api:check` will fail even though the SDK export map itself is synced.
- Suggested fix: rerun `pnpm plugin-sdk:api:gen` on current `main` and commit the refreshed `docs/.generated/plugin-sdk-api-baseline.sha256`.
- Confidence: high
