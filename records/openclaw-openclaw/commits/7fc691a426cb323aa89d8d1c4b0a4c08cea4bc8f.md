## Findings

### Medium: Configured Telegram table mode is still ignored on generic chunked sends

- Kind: bug
- File: `extensions/telegram/src/outbound-adapter.ts`
- Line: 56
- Evidence: The new Telegram chunker only reads `ctx?.formatting?.tableMode`. The shared chunk context type only carries `formatting` (`src/channels/plugins/outbound.types.ts:134`), and the shared planner only passes a third argument when `params.formatting` is already present (`src/infra/outbound/message-plan.ts:59`). `deliverOutboundPayloadsCore` passes `formatting: params.formatting` into planning (`src/infra/outbound/deliver.ts:1414`) but does not resolve `channels.telegram.markdown.tables` from `cfg`/`accountId` when formatting is absent. A direct runtime probe of `telegramOutbound.chunker` with a cfg-like context containing `channels.telegram.markdown.tables = "bullets"` still returned the raw table, while the same call with `formatting.tableMode = "bullets"` rendered `<b>A</b>\nâ€˘ Value: 1`.
- Impact: Telegram bot dispatch paths that already resolve and pass `formatting.tableMode` are fixed, but generic durable/direct outbound sends, compatibility paths, or queued/recovered sends that rely on channel config can still deliver raw `| --- |` table syntax once chunking happens. That leaves the documented/configured table-mode behavior only partially honored.
- Suggested fix: Make the chunker context carry `cfg` and `accountId` additively, pass those through `planOutboundTextMessageUnits` and `sendTextMediaPayload`, and let the Telegram chunker resolve `resolveMarkdownTableMode({ cfg, channel: "telegram", accountId })` when `formatting.tableMode` is absent. Alternatively, resolve `tableMode` into `params.formatting` before planning for Telegram.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes for the chunker seam: call `telegramOutbound.chunker` on a markdown table with no explicit `formatting.tableMode`; even a cfg-like context containing Telegram `markdown.tables: "bullets"` is ignored because the context type and implementation do not read it.
- Is this the best way to solve the issue? yes, likely. The safer fix is additive context propagation, matching the already-open broader PR shape and avoiding channel-specific policy in core.
