# Commit 6de9d71

## Summary

Found two actionable CI/security-pipeline issues: the new workflows execute a downloaded OpenGrep binary without enforcing signature or digest verification, and the new rule metadata guard is wired into local `pnpm check` but not into the sharded CI check job.

## Findings

### Medium: OpenGrep binary install is not actually pinned or verified

- Kind: supply_chain
- File: `.github/workflows/opengrep-precise.yml`
- Line: 43
- Evidence: Both OpenGrep workflows pipe the pinned installer script into `bash -s -- -v "$OPENGREP_VERSION"` but do not pass `--verify-signatures`, install `cosign`, or verify the downloaded release asset digest. Running the exact installer path locally for `v1.19.0` completed with: `Warning: cosign needed for signature validation; the package will still be installed.` The GitHub release metadata for `opengrep/opengrep` `v1.19.0` exposes asset SHA-256 digests, and the release is not immutable. Source checked: `https://github.com/opengrep/opengrep/releases/tag/v1.19.0`.
- Impact: A mutable or compromised OpenGrep release asset can execute arbitrary code on the CI runner before the scan. Pinning only the installer commit does not pin the binary bytes that are executed.
- Suggested fix: Install `cosign` and invoke the installer with `--verify-signatures`, or download the exact platform asset and verify its expected SHA-256 before adding it to `PATH`. Apply the same fix to `.github/workflows/opengrep-precise-full.yml`.
- Confidence: high

### Low: CI does not run the new OpenGrep metadata guard

- Kind: regression
- File: `.github/workflows/ci.yml`
- Line: 1395
- Evidence: The commit adds `check:opengrep-rule-metadata` in `package.json:1284` and includes it in local `scripts/check.mjs:45`, but the CI `check-shard` job manually enumerates the equivalent preflight checks and omits it. A dry-run changed-check plan for this exact range also omits `check:opengrep-rule-metadata`, selecting only conflict markers, changelog attribution, wildcard re-export guards, runtime sidecar guard, `tsgo:all`, lint, and import cycles.
- Impact: Future PRs can change `security/opengrep/precise.yml` with missing or inconsistent provenance metadata and still pass the normal CI check shards. The OpenGrep scan validates rule loading/findings, not this repository’s advisory provenance contract.
- Suggested fix: Add `pnpm check:opengrep-rule-metadata` to the CI `preflight-guards` shard or a dedicated check shard, and include it in changed-check planning for `security/opengrep/**` / `.semgrepignore` rulepack surfaces.
- Confidence: high
