# Commit 4a188e7

## Summary

Found one concrete regression: the `@grammyjs/types` bump creates a duplicate/incompatible Telegram type graph and breaks the production TypeScript lane.

## Findings

### Medium: Telegram typecheck now fails from mismatched grammy type packages

- Kind: regression
- File: `package.json`
- Line: 1805
- Evidence: This commit bumps root `@grammyjs/types` to `3.27.2`, while `grammy@1.42.0` still depends on `@grammyjs/types@3.26.0` in `pnpm-lock.yaml`. `pnpm tsgo:prod` now fails in `extensions/telegram/src/bot-core.ts:170` and related Telegram files with “Two different types with this name exist” errors between `node_modules/grammy/node_modules/@grammyjs/types/...` and root `node_modules/@grammyjs/types/...`; the representative mismatch is `Poll.members_only` existing only in the root `3.27.2` type.
- Impact: The repo’s production typecheck is broken on current `main`, blocking release/build validation and any gate that runs `pnpm tsgo:prod` or `pnpm tsgo:extensions`.
- Suggested fix: Revert the root `@grammyjs/types` devDependency to `3.26.0` until `grammy` publishes a version that depends on `3.27.x`, or update `grammy` and the root type package together once those versions align.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: install from the frozen lockfile, then run `pnpm tsgo:prod`.
- Is this the best way to solve the issue? yes: the failure is caused by two structurally divergent copies of `@grammyjs/types`; aligning the root devDependency with `grammy@1.42.0`’s declared dependency removes the duplicate type identity problem.
