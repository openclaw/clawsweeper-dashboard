# Commit 28550a7

## Summary

Found one doctor false-negative introduced by the runtime-grant mirror: provider-specific message-tool vetoes are checked against the agent/default model only, but runtime can switch the actual channel run to a different provider through `channels.modelByChannel`.

## Findings

### Medium: channel model overrides can hide provider-policy message-tool failures

- Kind: regression
- File: `src/commands/doctor/shared/preview-warnings.ts`
- Line: 153
- Evidence: `resolveMessageToolAvailability` resolves provider policy from `resolvePrimaryModelRef(params.cfg, agentConfig?.model)`, which only considers the agent model, `agents.defaults.model`, then the hardcoded default. Runtime model selection can override that per channel: `src/auto-reply/reply/get-reply.ts` resolves `cfg.channels.modelByChannel` and replaces `provider/model` before the run, and `src/channels/model-overrides.ts` supports wildcard channel matches. A focused probe on current `main` with `channels.modelByChannel.telegram["*"] = "anthropic/claude-sonnet-4.6"`, `tools.profile = "coding"`, and `tools.byProvider.anthropic.allow = ["read"]` returned no visible-reply or channel-bound warnings, even though a Telegram group run would select Anthropic and the provider allowlist vetoes `message`.
- Impact: `doctor`/preview output can now say nothing for a config where source replies are routed through message-tool-only delivery but the runtime-selected provider strips the message tool. Before this commit, the generic `coding` profile path still warned; the new runtime-grant suppression removes that warning unless the denying provider is also the default/agent model provider.
- Suggested fix: when checking channel-bound or visible-reply source paths, resolve provider policy per affected channel using `channels.modelByChannel` where possible, or conservatively keep the warning when a routed channel can select a provider/model whose `tools.byProvider` policy denies `message`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A one-off current-main probe of `collectVisibleReplyToolPolicyWarnings` and `collectChannelBoundMessageToolPolicyWarnings` with a Telegram wildcard channel model override to Anthropic plus `tools.byProvider.anthropic.allow = ["read"]` returned `[]` for both warning sets.
- Is this the best way to solve the issue? yes, directionally. The warning logic needs to use the same channel model override inputs as runtime model selection, or fail conservative when static doctor context cannot prove the route’s provider.
