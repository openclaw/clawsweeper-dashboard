# Commit f3f6a866

## Summary

Found two concrete validator regressions in the new MS Teams inline image base64 canonicalizer. The main user-facing issue is that valid unpadded `data:` image payloads are now silently dropped.

## Findings

### Medium: Valid unpadded inline data images are rejected

- Kind: regression
- File: `extensions/msteams/src/attachments/shared.ts`
- Line: 339
- Evidence: `canonicalizeInlineBase64Payload()` now returns `undefined` unless the cleaned payload length is divisible by 4. That rejects `data:image/png;base64,aGVsbG8`, while Node decodes the same payload to `hello` and the previous implementation would have accepted it via `Buffer.from(payload, "base64")`. The current `downloadMSTeamsAttachments()` path consumes these candidates at `extensions/msteams/src/attachments/download.ts:212`, so rejected candidates are not saved as inbound media. The Fetch Standard processes `data:` base64 bodies with forgiving-base64 decode, and Infra’s forgiving-base64 algorithm only fails length remainder 1, allowing unpadded remainder 2 or 3 payloads. ([fetch.spec.whatwg.org](https://fetch.spec.whatwg.org/?utm_source=openai))
- Impact: MS Teams inline HTML images emitted with omitted base64 padding are silently ignored: no media buffer is saved, and `buildMSTeamsAttachmentPlaceholder()` also undercounts them through the same helper at `extensions/msteams/src/attachments/html.ts:115`.
- Suggested fix: canonicalize by stripping allowed ASCII whitespace, reject misplaced padding, but accept unpadded lengths with remainder 2 or 3 by adding the required padding before estimating/decoding.
- Confidence: high

### Low: Non-whitespace C0 controls are treated as ignorable base64 whitespace

- Kind: bug
- File: `extensions/msteams/src/attachments/shared.ts`
- Line: 316
- Evidence: The new loop ignores every code point `<= 0x20`, which includes NUL and BEL, not just ASCII whitespace. A focused probe against current `main` showed `aGV\u0000sbG8=` and `aGV\u0007sbG8=` both produce an inline data candidate decoding to `hello`. Infra defines ASCII whitespace as tab, LF, FF, CR, and space; other C0 controls are a separate class. ([infra.spec.whatwg.org](https://infra.spec.whatwg.org/?utm_source=openai))
- Impact: The validator still accepts malformed base64 payloads containing arbitrary low control characters, so the new validation is stricter than needed for padding but weaker than intended for invalid characters.
- Suggested fix: replace `code <= 0x20` with an explicit ASCII whitespace check for `0x09`, `0x0a`, `0x0c`, `0x0d`, and `0x20`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. `node --import tsx/esm` importing `extractInlineImageCandidates()` returns `0` candidates for `data:image/png;base64,aGVsbG8`, but `Buffer.from("aGVsbG8", "base64")` decodes to 5 bytes.
- Is this the best way to solve the issue? yes. Matching forgiving-base64 semantics preserves the intended malformed-padding rejection without dropping web-compatible unpadded payloads.
