# Commit 7308f40

## Summary

The freshness fence fixes the tested same-session stale-final race, but it is installed before inbound dedupe and only guards dispatcher deliveries. That leaves one concrete dropped-reply regression for duplicate inbound events and one still-stale delivery path for routed foreground replies.

## Findings

### Medium: Duplicate in-flight inbound events can suppress the only valid reply

- Kind: regression
- File: `src/auto-reply/dispatch.ts`
- Line: 285
- Evidence: `dispatchInboundMessageWithBufferedDispatcher` begins the foreground fence before `dispatchReplyFromConfig` reaches inbound dedupe. A duplicate provider event with the same `MessageSid` increments the same fence generation, then `claimInboundDedupe(ctx)` returns `inflight` and exits with no reply at `src/auto-reply/reply/dispatch-from-config.ts:800`. The original run’s pending final then sees the newer generation in `beforeDeliver` and is cancelled. I confirmed the dedupe state with an ad hoc check: first claim for a WhatsApp context is `claimed`, second same-message claim is `inflight`.
- Impact: provider retries or duplicate webhooks for the same in-flight WhatsApp/Telegram/etc. message can cause OpenClaw to drop the user’s response entirely. This is especially plausible because the code already has explicit in-flight dedupe handling.
- Suggested fix: do not advance the freshness generation for duplicate/in-flight copies of the same inbound message. Move/coordinate the freshness claim after inbound dedupe, or include inbound message identity in a preliminary check so same-message replays do not supersede the original run.
- Confidence: high

### Medium: Routed foreground replies bypass the freshness guard

- Kind: bug
- File: `src/auto-reply/dispatch.ts`
- Line: 290
- Evidence: the new guard is wired as `beforeDeliver` on the `ReplyDispatcher`. However cross-provider/origin-routed replies in `dispatch-from-config` call `routeReplyToOriginating` directly from `sendFinalPayload` at `src/auto-reply/reply/dispatch-from-config.ts:1004` and from `sendPayloadAsync` at `src/auto-reply/reply/dispatch-from-config.ts:624`, without passing through `dispatcher.sendFinalReply` or the guarded `beforeDeliver`.
- Impact: a stale foreground final from an older run can still be delivered when the reply is routed back to an originating channel different from the current surface, despite a newer same-session inbound starting. The commit’s guard gives protection to direct dispatcher delivery but not to this channel-delivery boundary.
- Suggested fix: apply the same freshness predicate inside the routed-send helpers, or move the freshness check to a common “about to send foreground payload” boundary used by both dispatcher and route-reply paths.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the duplicate claim path: two identical inbound contexts produce `claimed` then `inflight`, and the commit’s generation fence is advanced before that second claim is rejected. The routed-reply issue is source-proven but not live-reproduced.
- Is this the best way to solve the issue? Yes for the direction: freshness should be coordinated with inbound dedupe and enforced at all foreground delivery exits, not only the buffered dispatcher’s `beforeDeliver`.
