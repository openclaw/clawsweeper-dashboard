# Commit 2de0113608

## Summary

Found two actionable issues: one runtime env/state-dir mismatch in the new `GatewayClient.env` contract, and one Docker-lane test gap that can miss a restart regression.

## Findings

### Medium: `GatewayClient.env` does not control the device identity path

- Kind: bug
- File: `src/gateway/client.ts`
- Line: 248
- Evidence: The commit adds `env?: NodeJS.ProcessEnv` and threads it into device-token load/store/clear calls, but the constructor still defaults `deviceIdentity` via `loadOrCreateDeviceIdentity()` with no env. That helper resolves `OPENCLAW_STATE_DIR` from `process.env`, while `loadStoredDeviceAuth()` now reads the token from `this.opts.env`.
- Impact: A caller using `new GatewayClient({ env })` for a service or alternate state dir can load/create an identity in the caller process state dir, then look for a token for that wrong device id in the service env state dir. Stored service auth is not used, and the constructor can also mutate the wrong local state dir.
- Suggested fix: When `opts.env` is provided and `opts.deviceIdentity` is omitted, resolve the identity path from `resolveStateDir(opts.env)` before calling `loadOrCreateDeviceIdentity(...)`. Add a real temp-dir test where process env and `opts.env.OPENCLAW_STATE_DIR` differ.
- Confidence: medium

### Low: `update-restart-auth` can pass without proving a new gateway process

- Kind: test_gap
- File: `scripts/e2e/upgrade-survivor-docker.sh`
- Line: 269
- Evidence: The lane installs the candidate tarball, starts a gateway from that same candidate, then runs `openclaw update --tag <same tarball>` and only checks update JSON plus HTTP/RPC health afterward. Because the pre-update and post-update gateway versions are identical, the later probes do not prove that the shim actually received a restart or that the PID changed.
- Impact: A regression where the update command skips or fails to initiate the managed restart can be masked if the already-running gateway remains healthy.
- Suggested fix: Record the pre-update PID and require the systemctl shim log to include `stop`/`restart` or require `SYSTEMCTL_SHIM_PID_FILE` to change before the health probes pass.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? For the env bug, yes: use distinct `process.env.OPENCLAW_STATE_DIR` and `GatewayClient({ env })` state dirs with a cached token only in `env`; the constructor resolves identity from the wrong state dir. For the Docker lane gap, the failure mode is a concrete missing assertion rather than a live reproduced failure.
- Is this the best way to solve the issue? Yes for the env bug: identity and token storage must resolve from the same env. For the lane gap, PID/shim-log proof is the narrowest check.
