# Commit 314ab1e

## Summary

Found one low-severity regression in the new `tokenRef` placeholder path: malformed token expirations can now make read-only PI model discovery treat an invalid token profile as configured.

## Findings

### Low: malformed `tokenRef` expirations are treated as configured

- Kind: bug
- File: `src/agents/pi-auth-credentials.ts`
- Line: 58
- Evidence: The new placeholder path rejects token credentials only when `expires` is a finite number in the past. If a `tokenRef` profile has `expires: null`, a string, `NaN`, or `Infinity`, lines 49-58 skip invalid-expiry rejection and return the secret-ref placeholder. Canonical auth semantics reject those same values as `invalid_expires` in `src/agents/auth-profiles/credential-state.ts:25` and `docs/auth-credential-semantics.md:36`.
- Impact: Read-only model discovery can mark models for that token-backed provider as available even though runtime auth resolution rejects the profile. That reintroduces selection-time/runtime drift for malformed token profiles and can lead to model picker/list availability that later fails or falls back.
- Suggested fix: Use `resolveTokenExpiryState(cred.expires)` in `convertAuthProfileCredentialToPi()` for token credentials, and reject `expired` and `invalid_expires` before returning either inline-token credentials or secret-ref placeholders. Add a regression case for wrong-type/non-finite `expires` with `tokenRef`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused `tsx` probe showed `resolvePiCredentialMapFromStore(..., { includeSecretRefPlaceholders: true })` returns placeholder credentials for `tokenRef` profiles with `expires: "not-a-number"` and `expires: null`; `evaluateStoredCredentialEligibility()` rejects both as `invalid_expires`.
- Is this the best way to solve the issue? yes. Reusing the existing token expiry helper keeps PI discovery aligned with the documented auth-profile contract and the runtime resolver.
