# Commit 1a7669b

## Summary

The `@openclaw/fs-safe` update changes `tryReadSecretFileSync` so policy failures such as rejected symlinks now throw. OpenClaw still re-exports that helper directly and several channel credential resolvers still expect the old lenient behavior, so symlinked secret-file configs now throw out of account/token resolution instead of returning unavailable credentials.

## Findings

### Medium: `tryReadSecretFileSync` now throws through channel credential resolution

- Kind: regression
- File: `src/infra/secret-file.ts`
- Line: 10
- Evidence: `src/infra/secret-file.ts:10` directly re-exports `tryReadSecretFileSync` from `@openclaw/fs-safe/secret`. With `package.json:1783` updated to `@openclaw/fs-safe@0.2.7`, a focused package comparison showed `0.2.6` returned `undefined` for a rejected symlink secret file while `0.2.7` throws `FsSafeError`. OpenClaw’s own expectations still require the lenient behavior: `src/infra/secret-file.test.ts:145`, `extensions/telegram/src/token.ts:178`, `extensions/irc/src/accounts.ts:82`, `extensions/line/src/accounts.ts:19`, `extensions/nextcloud-talk/src/accounts.ts:89`, `extensions/zalo/src/token.ts:13`, and `extensions/telegram/src/account-inspect.ts:41`.
- Impact: Configured secret/token/password file paths that hit a validation failure now crash credential resolution paths instead of producing `source: "none"` or `configured_unavailable`. The failing tests cover Telegram, IRC, LINE, Nextcloud Talk, Zalo, and Telegram account inspection.
- Suggested fix: Decide the OpenClaw contract at the SDK boundary. If channel resolvers should keep treating rejected credential files as unavailable, wrap the upstream helper in `src/infra/secret-file.ts` or catch `FsSafeError` at the channel call sites and map it to the existing unavailable states. If fail-closed throwing is intended, update the channel tests and any user-facing status/error paths deliberately.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Run the focused channel tests below; each rejected symlink credential path throws `FsSafeError` instead of returning the expected unavailable result.
- Is this the best way to solve the issue? unclear. Upstream `fs-safe` now documents throwing for validation failures, but OpenClaw’s channel tests and resolver behavior still encode the old lenient contract. The fix should be at the OpenClaw boundary where maintainers choose the desired user-facing behavior.
