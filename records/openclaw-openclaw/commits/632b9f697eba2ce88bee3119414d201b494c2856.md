# Commit 632b9f6

## Summary

Found one regression: the new repository validation is enforced, but the existing publishable plugin manifests were not updated, so current plugin release checks now reject most packages before publishing can proceed.

## Findings

### Medium: Repository validation blocks existing publishable plugin releases

- Kind: regression
- File: `scripts/lib/plugin-npm-release.ts`
- Line: 235
- Evidence: the new check requires every publishable plugin candidate to have `repository.url === "https://github.com/openclaw/openclaw"`, and `collectPublishablePluginPackages` throws on validation errors before returning a release plan. Current main still has publishable plugin manifests without `repository`, for example `extensions/zalo/package.json:50` has `publishToClawHub` and `publishToNpm` set but no repository field. After installing dependencies, `pnpm release:plugins:npm:check -- --plugins @openclaw/zalo` fails with `zalo: package.json repository.url must be "https://github.com/openclaw/openclaw" ... found "<missing>"`. A manifest scan found only `twitch` compliant and 17 publishable plugins missing or mismatching the new requirement.
- Impact: selected NPM releases for existing packages such as `@openclaw/zalo` cannot reach the release plan/publish steps. The shared validation also makes `pnpm release:plugins:clawhub:check` fail before ClawHub planning, so the manual ClawHub release workflow is blocked by the same missing metadata. The GitHub release workflows call these checks before planning at `.github/workflows/plugin-npm-release.yml:78` and `.github/workflows/plugin-clawhub-release.yml:68`.
- Suggested fix: add the required `repository` metadata to every current `publishToNpm` plugin manifest in the same change, or stage the validator so it only gates packages after their manifests have been migrated. Add a focused real-manifest regression check for at least a selected current package.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: run `pnpm release:plugins:npm:check -- --plugins @openclaw/zalo` on current main after `pnpm install`.
- Is this the best way to solve the issue? yes: the validator may be reasonable, but the repository state must satisfy it before the release gates enforce it.
