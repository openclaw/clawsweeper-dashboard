# Commit fb73f21

## Summary

Found one actionable regression in the array-index unset fix. It stops double-deleting shifted array entries, but the shifted entries can now be persisted with resolved `${ENV}` values instead of preserving the authored env reference.

## Findings

### Medium: shifted array entries can write resolved env refs back to config

- Kind: privacy
- File: `src/cli/config-cli.ts`
- Line: 1706
- Evidence: `runConfigUnset` clones `snapshot.resolved`, removes the array element, then omits `writeOptions.unsetPaths` for array leaves. The write layer later computes changed paths by index in `src/config/io.ts:2010`; `restoreEnvRefsFromMap` only restores env refs for paths not considered changed in `src/config/io.write-prepare.ts:548`. After an array deletion, every following element shifts index, so env refs in those retained elements are treated as changed and are written as resolved values.
- Impact: `openclaw config unset array[index]` can silently materialize secrets or private env-backed values into `openclaw.json` when a later array element contains `${ENV}`. Focused smoke reproduced this with `agents.defaults.cliBackends.codex.args`: after unsetting index 1, the retained third arg was persisted as `resolved-value-not-secret` instead of `${OPENCLAW_ARG_SECRET}`.
- Suggested fix: preserve authored values for shifted array entries while still applying the deletion once. One safe direction is to make array-leaf unset a first-class write intent that can suppress source-value restoration for the removed index without replaying a second splice, or rebuild the persisted array from the authored source with only the target index removed before env/tilde restoration.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Use a temp `OPENCLAW_CONFIG_PATH` whose `agents.defaults.cliBackends.codex.args` is `["--keep", "--remove", "${OPENCLAW_ARG_SECRET}"]`, set `OPENCLAW_ARG_SECRET`, then run `pnpm openclaw config unset 'agents.defaults.cliBackends.codex.args[1]'`. Current main exits 0 and persists `["--keep", "resolved-value-not-secret"]`.
- Is this the best way to solve the issue? yes for the fix direction: the original double-splice must stay fixed, but the write layer still needs deletion intent or authored-array restoration so shifted elements do not lose env refs.
