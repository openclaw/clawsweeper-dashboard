# Commit 4be4c475ea

## Summary

Found one scanner bypass introduced by the benign `.exec()` suppression. The current `main` source still has the issue.

## Findings

### Medium: `child_process` aliases other than `cp` bypass dangerous exec scanning

- Kind: security
- File: `src/security/skill-scanner.ts`
- Line: 233
- Evidence: `isBenignMemberExecMatch` suppresses any member `.exec(` unless the same line matches only `cp.exec`, `childProcess.exec`, or `child_process.exec`. A skill file containing `const proc = require("child_process"); proc.exec("node server.js");` has the required `child_process` context, but the `proc.exec(` line is treated as benign and produces no `dangerous-exec` finding. A focused live check on current `main` printed `cp dangerous-exec:2`, `proc none`, `bare dangerous-exec:2`, `regexp none`; `scanDirectoryWithSummary` for the `proc.exec` sample returned `critical:0` and no findings.
- Impact: Critical `dangerous-exec` findings are the default block condition for skill install source scans. This bypass lets untrusted skill/plugin source hide shell execution behind any normal alias name, weakening the install-time dangerous-code gate.
- Suggested fix: Do not hard-code a tiny set of child-process object names. Prefer parsing/import tracking for aliases from `require("child_process")` and `import("node:child_process")`; at minimum, keep scanning subsequent matches and only suppress `.exec()` when the receiver is clearly a benign `RegExp`/known-safe expression rather than an arbitrary object.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: run `node --import tsx -e` against `scanSource` with `const proc = require("child_process"); proc.exec("node server.js");`; it returns no `dangerous-exec` finding on current `main`.
- Is this the best way to solve the issue? no: the current fix handles the false positive but adds a narrow alias allowlist. Alias discovery or AST-based callee classification is safer.
