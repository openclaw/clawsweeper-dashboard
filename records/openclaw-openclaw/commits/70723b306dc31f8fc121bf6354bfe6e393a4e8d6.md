# Commit 70723b3

## Summary

Found one incomplete-normalization bug. The new Kilo supplemental catalog path emits the canonical nested Gemini ID, but the later generic configured-catalog merge still appends the original retired ID, so users can still see and select `kilocode/google/gemini-3-pro-preview`.

## Findings

### Low: Configured Kilo catalogs can still publish the retired nested Gemini ID

- Kind: bug
- File: `src/plugin-sdk/provider-catalog-shared.ts`
- Line: 189
- Evidence: `readConfiguredProviderCatalogEntries` now rewrites `google/gemini-3-pro-preview` to `google/gemini-3.1-pro-preview`, but `loadModelCatalog` merges plugin supplemental rows before appending `buildConfiguredModelCatalog` rows. That later generic path uses `normalizeStaticProviderModelId(providerId, rawId)` in `src/agents/model-selection-shared.ts:842`; for provider `kilocode`, it does not normalize the nested `google/...` model ID. A focused import check showed the split directly: supplemental produced `kilocode/google/gemini-3.1-pro-preview`, while configured produced `kilocode/google/gemini-3-pro-preview`.
- Impact: a config row under `models.providers.kilocode.models[]` with the old Kilo nested Gemini ref can still add the retired ref to the final model catalog after the canonical supplemental row. Model pickers or allowlists may still expose/select the retired Kilo model, sending `google/gemini-3-pro-preview` to Kilo instead of the intended `google/gemini-3.1-pro-preview`.
- Suggested fix: apply the same nested proxy-provider ID canonicalization in the generic configured catalog path, or dedupe configured rows against supplemental rows using a canonicalized provider/model key. Add an integration test for `loadModelCatalog` or the merge pair that asserts the old Kilo ref is absent when the configured provider model is `google/gemini-3-pro-preview`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The focused TSX check imported `readConfiguredProviderCatalogEntries` and `buildConfiguredModelCatalog` with the same Kilo config row and showed the divergent refs: supplemental canonical, configured retired.
- Is this the best way to solve the issue? no. The commit fixes only the plugin augmentation seam; the durable fix needs the configured-catalog merge path to share the same canonicalization or dedupe semantics.
