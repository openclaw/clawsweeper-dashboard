### Medium: Plugin registry metadata mock leaves install-persist tests failing

- Kind: reliability
- File: `src/cli/plugins-cli-test-helpers.ts`

- Evidence: The new `loadPluginRegistrySnapshotWithMetadata` mock always returns `snapshot: { plugins: [] }`. Running `pnpm test src/agents/subagent-spawn.workspace.test.ts src/cli/plugins-cli.install.test.ts src/cli/plugins-install-persist.test.ts src/cli/plugins-cli.uninstall.test.ts src/cli/plugins-cli.update.test.ts src/cli/plugins-cli.list.test.ts` after `pnpm install` fails in `src/cli/plugins-install-persist.test.ts` with 3 assertion failures. The failing assertions expect `loadPluginManifestRegistry` to be called with targeted options like `{ includeDisabled: true, pluginIds: ["memory-b"] }`, but the new mock drives `loadPluginMetadataSnapshot` through the empty-index fallback, which calls the manifest registry with `{ diagnostics: [], env: process.env, installRecords: {}, workspaceDir: undefined }`.
- Impact: The touched plugin CLI test lane is red on current `main`, so this fixture change can block CI/changed-test validation for plugin install work.
- Suggested fix: Make the helper’s metadata snapshot mock configurable/coherent with the existing manifest-registry mock, or update the install-persist tests to assert the current metadata snapshot call contract while preserving the important behavior checks: selected plugin only for runtime diagnostics and no runtime sibling loading.
- Confidence: high
