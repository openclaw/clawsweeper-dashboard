# Commit 0496063264

## Summary

Found one supply-chain policy regression: the lockfile pins two npm versions that were still inside the repo’s configured 48-hour `minimumReleaseAge` window at review time.

## Findings

### Medium: Dependency refresh bypasses the repo’s pnpm release-age gate

- Kind: supply_chain
- File: `package.json`
- Line: 1699
- Evidence: `pnpm-workspace.yaml:7` sets `minimumReleaseAge: 2880`, and neither `@google/genai` nor `@zed-industries/codex-acp` is listed in `minimumReleaseAgeExclude` at `pnpm-workspace.yaml:9`. This commit pins `@google/genai` to `^2.0.1` in `package.json:1699` and `@zed-industries/codex-acp` to `0.14.0` in `extensions/acpx/package.json:12`, with resolved lockfile entries at `pnpm-lock.yaml:2439` and `pnpm-lock.yaml:4819`. Fresh pnpm probes with the same `minimumReleaseAge: 2880` failed for both exact versions with `ERR_PNPM_NO_MATURE_MATCHING_VERSION`: `@google/genai@2.0.1` was reported as released 23 hours ago, and `@zed-industries/codex-acp@0.14.0` as released 37 hours ago. pnpm documents `minimumReleaseAge` as the number of minutes that must pass after publish before installation, and `minimumReleaseAgeExclude` as the explicit escape hatch. ([pnpm.io](https://pnpm.io/settings))
- Impact: The committed lockfile allows immediate consumption of package versions that the repository’s own supply-chain delay would reject during a normal fresh resolution. That weakens the dependency-compromise buffer the workspace config is meant to enforce.
- Suggested fix: Wait until the versions satisfy the 48-hour age gate and regenerate the lockfile, or add narrowly scoped `minimumReleaseAgeExclude` entries for the exact packages/versions with maintainer-approved rationale.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. In a temp project with `minimumReleaseAge: 2880`, `pnpm add @google/genai@2.0.1 --lockfile-only` and `pnpm add @zed-industries/codex-acp@0.14.0 --lockfile-only` both fail with `ERR_PNPM_NO_MATURE_MATCHING_VERSION`.
- Is this the best way to solve the issue? yes: either respect the existing maturity window, or make the exception explicit in the workspace config.
