# Commit 665b0ef

## Summary

Found one concrete security bypass in the new trust boundary. The patch validates `groupId`, but still forwards caller-controlled `groupChannel` and `groupSpace` whenever `groupId` is absent or matches the session-derived ID. Several channel policy resolvers use those selector fields to choose group-scoped tool policies, so a scoped caller can still select a more permissive channel/guild policy without forging the `groupId`.

## Findings

### High: forged groupChannel/groupSpace can still select a permissive group tool policy

- Kind: security
- File: `src/agents/pi-tools.policy.ts`
- Line: 499
- Evidence: `resolveGroupToolPolicy()` validates only `params.groupId` via `resolveTrustedGroupIdFromContexts()`, then passes `params.groupChannel` and `params.groupSpace` through to the loaded channel plugin whenever `trustedGroup.dropped` is false. If the caller omits `groupId`, `resolveTrustedGroupIdFromContexts()` returns `dropped: false`, so selector fields are trusted even though no session-derived value vouches for them. The gateway path also persists request selectors for first-turn trusted sessions when the request group ID is empty or matches (`src/gateway/server-methods/agent.ts:963`).
- Evidence: Slack’s resolver explicitly uses `groupChannel` as a policy lookup candidate after the real `groupId` and before wildcard fallback (`extensions/slack/src/group-policy.ts:30`). Discord similarly uses `groupSpace` for guild selection and `groupChannel` for channel selection (`extensions/discord/src/group-policy.ts:21`, `extensions/discord/src/group-policy.ts:44`).
- Evidence: focused repro with the Slack plugin pinned showed the bypass: a config with wildcard `allow: ["read"]` and channel-name `admin` allowing `["exec","read","write","edit"]` returned the admin policy for `sessionKey: "agent:main:slack:group:C999"` plus only `groupChannel: "#admin"`. The same call without `groupChannel` returned the wildcard read-only policy.
- Impact: this leaves the same class of tool-policy escalation open for channels where operators configure policies by names/slugs rather than immutable IDs. A caller who can invoke the gateway agent method for a legitimate group session can claim another channel or guild label and unlock tools such as `exec`, `write`, or `edit`.
- Suggested fix: treat `groupChannel` and `groupSpace` as trusted only when they come from stored/session metadata or a server-verified inbound transport event. In `resolveGroupToolPolicy()`, do not pass caller selector fields unless a trusted metadata object explicitly marks them as verified; in the gateway first-turn path, only persist request selectors when the request includes a matching verified `groupId` and the caller is an internal trusted channel runtime.
- Confidence: high
