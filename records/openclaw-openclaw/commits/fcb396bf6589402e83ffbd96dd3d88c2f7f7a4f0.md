# Commit fcb396b

## Summary

Found one concrete edge-case bug in the new `registerIfAbsent` claim semantics: a successful claim can be immediately evicted and therefore not persisted when the namespace is at capacity and timestamps tie.

## Findings

### Medium: `registerIfAbsent` can return `true` for a claim that eviction immediately deletes

- Kind: bug
- File: `src/plugin-state/plugin-state-store.sqlite.ts`
- Line: 485
- Evidence: `pluginStateRegisterIfAbsent` inserts the new key, checks `result.changes`, then calls `enforcePostRegisterLimits` and unconditionally returns `true`. The shared eviction query deletes rows by `ORDER BY created_at ASC, entry_key ASC`, so if an existing row and the new row have the same `Date.now()` millisecond and the new key sorts earlier, the just-inserted row is the one evicted. I reproduced the SQL path with Node SQLite: seed key `z` at `created_at=1000`, insert key `a` at `created_at=1000`, run the same `deleteOldestNamespace` query with `maxEntries=1`; the remaining row is `z`, not `a`, even though the insert succeeded.
- Impact: a plugin using `registerIfAbsent` for dedupe can receive `true` and process work believing the key was claimed, but the claim is absent immediately afterward. A following duplicate can then also claim the same work. This is plausible during bursts because `Date.now()` has millisecond granularity, especially with small `maxEntries` or many same-tick claims.
- Suggested fix: make namespace eviction preserve the just-inserted key for `registerIfAbsent`, for example by deleting oldest rows excluding `params.key`, or by otherwise verifying the inserted key survived before returning `true`. Add a regression test with fake timers, `maxEntries: 1`, existing key `z`, new key `a`, and same system time.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused `node --experimental-sqlite` smoke using the same table shape and eviction query leaves only `z` after inserting `a` at the same `created_at`, proving the new inserted claim can be selected as the oldest row.
- Is this the best way to solve the issue? yes. The contract documented in `docs/plugins/sdk-runtime.md` says `true` means the key was registered; preserving the inserted key during post-insert eviction is the narrowest fix for that contract.
