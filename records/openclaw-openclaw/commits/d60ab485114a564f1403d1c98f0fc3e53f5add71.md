## Summary

Found one low-severity bug in the new Telegram preview-flow proof script. The production Telegram/Discord/Feishu reasoning and progress-draft paths covered by focused tests did not show an actionable regression.

## Findings

### Low: `working-final` can report preview proof that did not actually happen

- Kind: reliability
- File: `scripts/dev/channel-message-flows.ts`
- Line: 353
- Evidence: `runTelegramWorkingFinalFlow` loops while `elapsedMs < durationMs`, but only sleeps when `elapsedMs + updateIntervalMs < durationMs`, so the default `durationMs: 12000` with the default 2s cadence sends the final after 10s of sleeps, not 12s. The same loop increments `previewUpdates` after `await draft.update(previewText)` without checking the returned boolean, so a native draft update that returns `false` is still counted as a successful preview update.
- Impact: This script is meant to provide real Telegram preview-flow proof. It can under-run the requested working duration and print a successful preview update count even when `sendMessageDraft` failed or was disabled after the first API error, which weakens the evidence maintainers get from `working-final`.
- Suggested fix: Track successful updates from the boolean return value and fail or report a proof gap when an update returns `false`. Also make `durationMs` match wall-clock simulated work, for example by sleeping after the last emitted update until the requested duration or by changing the loop condition/cadence accounting.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused local probe of `runTelegramWorkingFinalFlow({ durationMs: 12000 })` with mocked sleep recorded `totalSleep: 10000`, and a probe with `draft.update` returning `false` still returned `previewUpdates: 2`.
- Is this the best way to solve the issue? yes. The fix should stay inside the dev runner: count only successful native draft updates and make the duration option’s behavior match its help text.
