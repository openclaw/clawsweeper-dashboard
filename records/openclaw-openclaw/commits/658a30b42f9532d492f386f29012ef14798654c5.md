# Commit 658a30b42f

## Summary

Found one low-severity regression: default exec/bash failures now bypass the documented `messages.suppressToolErrors` config.

## Findings

### Low: `messages.suppressToolErrors` no longer suppresses exec/bash failure warnings

- Kind: regression
- File: `src/agents/pi-embedded-runner/run/payloads.ts`
- Line: 151
- Evidence: The commit moved exec/bash handling behind the mutating-tool branch. Since `exec` and `bash` are mutating tool names, `resolveToolErrorWarningPolicy` returns from the mutating branch at lines 151-155 before reaching the `params.suppressToolErrors` check at line 160. The documented config contract in `src/config/schema.help.ts:1776` and `src/config/types.messages.ts:134` says `messages.suppressToolErrors` suppresses tool-error warnings shown to the user. A focused probe on current `main` with `config.messages.suppressToolErrors: true`, assistant text `"Done."`, and `lastToolError: { toolName: "exec", error: "command failed" }` still emitted `âš ď¸Ź đź› ď¸Ź Exec failed`.
- Impact: Users who explicitly disabled tool-error warning payloads can still receive exec/bash warning messages after this commit. This is especially likely in the exact path the commit changed: default verbosity plus failed `exec`/`bash`.
- Suggested fix: Honor `messages.suppressToolErrors` before the mutating-tool return, or add a tested explicit override for exec/bash warnings when that config is true. If mutating failures are intentionally exempt, update the public config help/types to say so and add coverage for the exemption.
