# Commit 7db44b9

## Summary

Found two low-severity live-test coverage regressions in the new account-drift tolerance paths. Both can skip Anthropic proof while a later configured key is available.

## Findings

### Low: Anthropic disabled cache lane ignores fallback keys

- Kind: test_gap
- File: `src/agents/live-cache-regression-runner.ts`
- Line: 658
- Evidence: Normal Anthropic cache lanes iterate `fixture.apiKeys` and retry account drift with the next key, but `runAnthropicDisabledCacheLane` calls `runAnthropicDisabledLane` once with the original fixture. `resolveLiveDirectModel` sets `apiKey` to the first key and `apiKeys` to all keys in `src/agents/live-cache-test-support.ts:190`, so `OPENCLAW_LIVE_ANTHROPIC_KEYS="bad good"` lets stable/tool/image/mcp reach the good key while the disabled lane still skips on the bad first key.
- Impact: The cache-disabled baseline can be silently skipped even in a run that already proved a later Anthropic key works. A regression where `cacheRetention: "none"` starts producing cache reads/writes would not fail live cache validation.
- Suggested fix: Use the same key-rotation helper for the disabled lane, or carry forward a known-good Anthropic key from the successful lanes. Only mark disabled skipped after all configured keys fail with account drift or empty response.
- Confidence: high

### Low: Anthropic live model sweeps skip auth drift before trying remaining keys

- Kind: test_gap
- File: `src/agents/models.profiles.live.test.ts`
- Line: 1326
- Evidence: The agents live model suite computes `attemptMax` from all Anthropic keys and already retries next keys for rate-limit and billing errors, but the new auth-drift branch immediately records a skip and breaks. The gateway suite has the same pattern in `src/gateway/gateway-models.profiles.live.test.ts:2471`.
- Impact: In modern live CI, if the first Anthropic key is revoked/invalid but a later configured key is valid, the suites report Anthropic as skipped instead of exercising the model, agent, gateway, and tool-probe paths. That reduces exactly the live coverage these multi-key loops are meant to preserve.
- Suggested fix: Add an Anthropic-specific `isAuthErrorMessage` retry branch before the generic skip, mirroring the existing billing/rate-limit retry behavior.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Configure multiple Anthropic keys with the first key invalid and a later key valid, then run the affected live suites; the code paths break/skip before exhausting keys in the disabled cache lane and auth-drift model sweeps.
- Is this the best way to solve the issue? yes. The narrow fix is to reuse the existing Anthropic key rotation semantics for auth drift instead of widening skip behavior.
