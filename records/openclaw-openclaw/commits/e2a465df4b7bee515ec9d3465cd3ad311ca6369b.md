# Commit e2a465d

## Summary

Found one regression in the new Windows SSE smoke setup: the generated `config set` path does not escape model IDs containing dots, including the default Windows OpenAI model.

## Findings

### Medium: Windows OpenAI smoke fails before the agent turn for dotted model IDs

- Kind: regression
- File: `scripts/e2e/parallels/powershell.ts`
- Line: 29
- Evidence: `windowsModelProviderTimeoutScript()` writes `path: \`agents.defaults.models.${modelId}\``. The Windows default model is `openai/gpt-4.1-mini` from `scripts/e2e/parallels/provider-auth.ts:72`, and `scripts/e2e/parallels/windows-smoke.ts:896` runs this helper before the agent turn. A focused CLI dry-run of the generated batch failed with `agents.defaults.models.openai/gpt-4: Unrecognized key: "1-mini"`, because `config set` dot paths split `gpt-4.1-mini` into nested path segments. The same value validated successfully when addressed as `agents.defaults.models["openai/gpt-4.1-mini"]` or when merged into `agents.defaults.models` as an object.
- Impact: The default Windows OpenAI Parallels smoke now aborts during agent-turn setup with `model provider timeout config set failed`, so the commit’s intended forced-SSE smoke does not run for the default model or other dotted OpenAI model IDs such as `openai/gpt-5.5`.
- Suggested fix: Build this batch entry using bracket notation for the model key, or set/merge the whole `agents.defaults.models` object with `{ [modelId]: { alias: "GPT", params: { transport: "sse" } } }`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The generated `config set --batch-json ... --strict-json --dry-run` fails on the default `openai/gpt-4.1-mini` path with the schema error above.
- Is this the best way to solve the issue? yes. Escaping the model key or merging the map matches the existing config path contract and preserves dotted model IDs as single keys.
