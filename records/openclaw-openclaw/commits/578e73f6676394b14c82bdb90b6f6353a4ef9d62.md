# Commit 578e73f

## Summary

Found one low-severity regression in debug proxy capture metadata handling. The targeted tests pass, but they do not cover caller-supplied guarded-fetch capture metadata under the new duplicate-capture suppression path.

## Findings

### Low: Guarded fetch metadata is dropped when global fetch capture suppresses the guarded capture

- Kind: regression
- File: `src/infra/net/fetch-guard.ts`
- Line: 272
- Evidence: `captureGuardedFetchExchange` now returns early when `capturedByGlobalFetchPatch` is true and the global fetch patch is installed. That skips the guarded capture path that supplies `flowId: params.capture?.flowId` and merges `auditContext` plus `params.capture?.meta` at `src/infra/net/fetch-guard.ts:282`. The global fetch patch capture at `src/proxy-capture/runtime.ts:205` only records `{ captureOrigin: "global-fetch", source }`.
- Impact: For guarded fetch calls that intentionally use ambient patched `globalThis.fetch` instead of `fetchWithRuntimeDispatcher` such as `pinDns: false`, debug proxy capture no longer preserves caller-provided `capture.flowId`, provider metadata, or `auditContext`. That breaks correlation and filtering in the capture UI even though the HTTP exchange still records once.
- Suggested fix: Preserve guarded metadata when suppressing the duplicate. One safe direction is to mark guarded fetch requests so the global fetch patch can record the guarded `flowId`/meta/audit context, or only suppress the guarded capture when there is no caller-supplied `capture` object and no `auditContext`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. I ran an inline Node/tsx smoke with `initializeDebugProxyCapture("test")`, patched `globalThis.fetch`, then called `fetchWithSsrFGuard({ url: "https://example.com/test", pinDns: false, auditContext: "qa-audit", capture: { flowId: "flow-1", meta: { foo: "bar" } } })`. The stored events had a random flow ID and `{"captureOrigin":"global-fetch","source":"openclaw"}`, with no `flow-1`, `qa-audit`, or `foo`.
- Is this the best way to solve the issue? unclear. Avoiding duplicate capture is reasonable, but the duplicate suppression needs to preserve the richer guarded-fetch metadata.
