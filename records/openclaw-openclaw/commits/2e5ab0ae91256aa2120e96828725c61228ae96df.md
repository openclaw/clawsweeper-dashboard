# Commit 2e5ab0a

## Summary

Found two concrete issues: realtime Discord voice now logs user/assistant content at info level, and the documented `dm:` voice agent-session target is parsed as a channel target.

## Findings

### Medium: Realtime Discord voice leaks transcript and answer text into info logs

- Kind: privacy
- File: `extensions/discord/src/voice/realtime.ts`
- Line: 203
- Evidence: `onTranscript` logs final realtime transcript text with `logger.info(...)` at `extensions/discord/src/voice/realtime.ts:203`. Realtime consult questions and answers are also logged at info at `extensions/discord/src/voice/realtime.ts:364` and `extensions/discord/src/voice/realtime.ts:385`; agent turn prompts/answers are logged at info at `extensions/discord/src/voice/manager.ts:794` and `extensions/discord/src/voice/manager.ts:821`. Existing STT voice transcript previews use verbose logging instead at `extensions/discord/src/voice/segment.ts:75`, and the shared Talk logging tests explicitly assert that private transcript text is not written to info/file logs in `src/talk/logging.test.ts:38`.
- Impact: Any Discord realtime voice room can put spoken user text, consult questions, and assistant answers into normal gateway logs. Those logs can persist to file/diagnostics when info logging is enabled, widening exposure for voice conversations compared with the existing verbose-only STT path.
- Suggested fix: Move content-bearing previews back behind `logVoiceVerbose` or add an explicit opt-in/redacted diagnostic flag. Keep info logs to event metadata, lengths, route/session ids, and statuses.
- Confidence: high

### Low: Documented `dm:` voice target routes as a channel session

- Kind: bug
- File: `extensions/discord/src/target-parsing.ts`
- Line: 31
- Evidence: The new docs say `target: "dm:123456789012345678"` routes through a direct-message session at `docs/channels/discord.md:1331`, but `parseDiscordTarget` only recognizes `user:`, `channel:`, `discord:`, and `discord:user/channel:` prefixes at `extensions/discord/src/target-parsing.ts:31` and `extensions/discord/src/target-parsing.ts:55`. A `dm:123` string falls through to the default channel branch at `extensions/discord/src/target-parsing.ts:51`; `resolveDiscordVoiceAgentRoute` then maps every non-user parsed target to `peer.kind: "channel"` at `extensions/discord/src/voice/manager.ts:138`.
- Impact: A config copied from the docs will not attach voice to the intended DM conversation. It creates/uses a channel-style session keyed around `dm:<id>`, which can select the wrong agent binding and lose the intended DM transcript context.
- Suggested fix: Either support `dm:` as a user/direct alias in `parseDiscordTarget`, or remove `dm:` from the docs and config hints. Add a regression test for `voice.agentSession.target: "dm:<id>"`.
- Confidence: high

- Is this the best way to solve the issue? yes for direction: keep sensitive voice content out of normal logs, and make target parsing match the documented contract.
