# Commit d656cda

## Summary

The Windows env-key normalization regresses the existing `env: { KEY: undefined }` contract. Child process env overrides that intentionally unset inherited variables now leave the inherited value in place, including security-sensitive cases such as `CLOUDSDK_PYTHON` and git environment controls.

## Findings

### Medium: Undefined env overrides no longer clear inherited child variables

- Kind: regression
- File: `src/process/exec.ts`
- Line: 62
- Evidence: `assignChildEnvValue` returns immediately when `value === undefined`, before deleting any matching key already copied from `baseEnv`. Before this commit, `resolveCommandEnv` did `{ ...baseEnv, ...params.env }` and then filtered `undefined`, so `env: { KEY: undefined }` removed `KEY` from the child env. Current `main` reproduces the regression: `resolveCommandEnv({ baseEnv: { CLOUDSDK_PYTHON: "/tmp/attacker-python" }, env: { CLOUDSDK_PYTHON: undefined } })` returns `CLOUDSDK_PYTHON=/tmp/attacker-python`; the old merge/filter behavior returns it unset.
- Impact: call sites already use `undefined` as a deliberate sanitization signal. `src/hooks/gmail-setup-utils.ts:152` says it must override inherited `CLOUDSDK_PYTHON` so gcloud cannot select a workspace-controlled interpreter, and its test expects `env: { CLOUDSDK_PYTHON: undefined }`. `src/plugins/git-install.ts:170`-`178` similarly clears `GIT_DIR`, `GIT_WORK_TREE`, `GIT_EXEC_PATH`, `GIT_SSL_NO_VERIFY`, and related git variables before plugin git operations. After this commit, those inherited variables survive into the child process, so gcloud/git can run with hostile or unsafe inherited configuration.
- Suggested fix: when an override value is `undefined`, delete the existing key from the resolved env instead of returning. On Windows, delete all case-insensitive matches; on other platforms, delete the exact key. Keep filtering `undefined` before passing the final env to `spawn`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Run `node --import tsx` and import `resolveCommandEnv` from `src/process/exec.ts`; pass `baseEnv: { CLOUDSDK_PYTHON: "/tmp/attacker-python", GIT_SSL_NO_VERIFY: "1" }` with matching `env` keys set to `undefined`. Both inherited values remain on current `main`.
- Is this the best way to solve the issue? yes. The old contract was merge-then-drop-undefined; preserving that means `undefined` must delete already-merged keys, while the new Windows-specific case-folding should only change which matching keys are deleted.
