# Commit 64ab50e

## Summary

Found one concurrency/data-loss regression in the new pre-update plugin install record snapshot path.

## Findings

### Medium: Post-update plugin sync can overwrite plugin install records changed during the core update

- Kind: data_loss
- File: `src/cli/update-cli/update-command.ts`
- Line: 2217
- Evidence: The command now snapshots install records before the core update at `src/cli/update-cli/update-command.ts:2217`, then always passes that stale map into post-core plugin sync at `src/cli/update-cli/update-command.ts:2378` and `src/cli/update-cli/update-command.ts:2397`. `updatePluginsAfterCoreUpdate` uses that supplied map instead of re-reading the current installed-plugin index at `src/cli/update-cli/update-command.ts:1144`, and if plugin sync changes anything it commits `nextInstallRecords` derived from that stale map while also using it as the rollback base at `src/cli/update-cli/update-command.ts:1288`. The commit helper writes the new index first and rolls back to `previousInstallRecords` on config-write failure at `src/cli/plugins-install-record-commit.ts:31`.
- Impact: If another `openclaw plugins install/update/uninstall` writes the installed-plugin index while `openclaw update` is running, the post-update sync can drop those newer records. On a config hash conflict, the rollback path can also restore the pre-update snapshot, discarding the concurrent plugin index update. This is especially plausible for git updates or `--no-restart` package updates where the Gateway/CLI can remain usable during the update.
- Suggested fix: Use the pre-update snapshot only as warning context for records missing from the current index. Before committing, re-read the current installed-plugin index, merge in only absent legacy/corrupt records needed for post-update checks, and use the current index as the commit rollback base. If both maps changed for the same plugin id, detect the conflict and retry or warn instead of blindly writing the stale snapshot.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? No live reproduction was run. A focused unit harness can reproduce it by snapshotting `{a}` before update, making the persisted index `{a,b}` before `updatePluginsAfterCoreUpdate` commits a changed plugin result, then asserting the write/rollback path drops `b`.
- Is this the best way to solve the issue? The current approach is the right direction for preserving corrupt-plugin warning context, but using the snapshot as the authoritative record set is not safe. Merging it with a fresh post-core read keeps the warning context without clobbering concurrent install records.
