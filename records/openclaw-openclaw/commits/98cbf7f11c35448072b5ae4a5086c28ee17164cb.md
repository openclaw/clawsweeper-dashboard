# Commit 98cbf7f

## Summary

Found one low-severity display bug in the new Telegram `/think` picker title. The command still executes safely; the issue is that the “current thinking level” can be shown as a raw unsupported value instead of the effective level runtime will actually use.

## Findings

### Low: Telegram `/think` picker can display an unsupported “current” level

- Kind: bug
- File: `extensions/telegram/src/bot-native-commands.ts`
- Line: 266
- Evidence: `resolveTelegramCommandMenuModelContext` copies `entry?.thinkingLevel` as a plain string, and `resolveTelegramThinkMenuCurrentLevel` returns that explicit value immediately. It also returns `resolveAgentConfig(...).thinkingDefault` immediately, without checking whether either level is supported by the target provider/model. The title then renders that raw value as `Current thinking level: ...`.
- Impact: For providers with restricted profiles, the picker can claim an active level that the runtime will not use. For example, Moonshot exposes only `off` and `low`/`on` in its thinking profile (`extensions/moonshot/index.ts:68`). If a session or agent default contains `high` while the target model is Moonshot, Telegram will show `Current thinking level: high` even though the normal reply path remaps unsupported non-explicit levels through `resolveSupportedThinkingLevel` before running (`src/auto-reply/reply/get-reply-run.ts:774`). The menu buttons can therefore say `off`/`on` while the header says `high`.
- Suggested fix: Resolve the displayed value through the same support policy used by the run path: normalize the raw configured/session value, check it against the target provider/model thinking profile, remap with `resolveSupportedThinkingLevel` when unsupported, and ideally display the provider label for the resolved option.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the code path: configure a Telegram-targeted session or agent with `thinkingLevel`/`thinkingDefault: "high"` and a binary-thinking model such as Moonshot, then invoke `/think` without args. The new title path returns `high`; the runtime path remaps unsupported non-explicit levels before execution.
- Is this the best way to solve the issue? Yes, the display should reuse the same provider/model support resolution as execution rather than showing raw stored/configured state.
