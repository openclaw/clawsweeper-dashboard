# Commit d24a847

## Summary

Found one config-contract mismatch introduced by the commit: `AgentConfig` now exposes per-agent `compaction`, but the strict runtime config schema still rejects the same field under `agents.list[]`.

## Findings

### Medium: Typed per-agent compaction is still rejected by config validation

- Kind: compatibility
- File: `src/config/types.agents.ts`
- Line: 94
- Evidence: The commit adds `AgentConfig.compaction?: AgentDefaultsConfig["compaction"]` at `src/config/types.agents.ts:94`, and `OpenClawConfig`/`AgentConfig` are exported through config and plugin SDK type surfaces. The runtime schema for `AgentEntrySchema` in `src/config/zod-schema.agent-runtime.ts:933` does not include `compaction` and ends with `.strict()` at `src/config/zod-schema.agent-runtime.ts:1011`.
- Impact: Type-correct code can now create or mutate `OpenClawConfig` with `agents.list[].compaction`, but persisted config validation rejects it. A focused probe returned `agents.list.0: Unrecognized key: "compaction"` while `agents.defaults.compaction` validated successfully. This is especially risky because the new comment says the field is kept for raw doctor migration/repair, but raw validation still treats it as invalid.
- Suggested fix: Either add a deprecated `compaction` field to `AgentEntrySchema` using the same schema as defaults, or remove the public `AgentConfig` field and keep doctor tests on an explicit raw/legacy fixture type.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: run a raw config validation with `{ agents: { list: [{ id: "main", compaction: { model: "openai/gpt-5.4" } }] } }`; it rejects `compaction`.
- Is this the best way to solve the issue? yes: config type, schema, and doctor migration contract should agree. If per-agent compaction is intentionally legacy-only, the type should not advertise it on the public config contract.
