# Commit 1b7bf4a

## Summary

One low-severity current-main issue remains around the new read-only workspace skill overlays. The code and tests assume a protected bind mounted after custom binds can override conflicting custom bind targets, but Docker rejects exact duplicate targets and preserves nested child mounts as writable.

## Findings

### Low: Protected skill overlays do not reliably override custom binds

- Kind: reliability
- File: `src/agents/sandbox/docker.ts`
- Line: 578
- Evidence: Current `main` appends custom binds, then appends read-only skill overlays at `src/agents/sandbox/docker.ts:577` and `src/agents/sandbox/docker.ts:578`; browser does the same at `src/agents/sandbox/browser.ts:355` and `src/agents/sandbox/browser.ts:360`. A live Docker 28.0.4 probe with `-v custom:/workspace/skills:rw -v workspace/skills:/workspace/skills:ro,z` fails container creation with `Duplicate mount point: /workspace/skills`. A nested custom bind probe with `-v custom:/workspace/skills/demo:rw -v workspace/skills:/workspace/skills:ro,z` starts, reads the custom content, and allows writes under `/workspace/skills/demo`.
- Impact: Configurations that explicitly allow reserved container targets can either fail to create the sandbox when they bind exactly `/workspace/skills`, or silently keep nested skill paths writable despite the protected overlay. This makes the current ordering tests overstate the real Docker behavior.
- Suggested fix: Detect custom bind targets equal to or under resolved protected skill container paths before appending Docker args. Either reject them with a clear config error when skill protection is active, or explicitly document and implement dangerous override semantics instead of relying on mount order.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The exact-target Docker probe returns `Duplicate mount point: /workspace/skills`; the nested-target probe shows the custom mount remains writable under the protected skill root.
- Is this the best way to solve the issue? yes. Docker mount ordering cannot enforce this policy, so OpenClaw needs explicit target conflict handling before creating the container.
