# Commit 70f5800

## Summary

Found one concrete QA regression in the new share-safe diagnostics scenario: the redaction leak checks are case-sensitive, so the scenario can pass while the artifact or final reply repeats transformed private content or the fake secret.

## Findings

### Medium: Redaction scenario misses case-variant leaks

- Kind: test_gap
- File: `qa/scenarios/personal/share-safe-diagnostics-artifact.md`
- Line: 134
- Evidence: The scenario normalizes expected artifact fields at lines 123 and 128, but checks forbidden material with raw `artifact.includes(needle)` and `outbound.text.includes(needle)` at lines 134 and 145. Existing adjacent personal scenario checks normalize forbidden text before comparison. A model response containing `qa_fake_personal_diagnostic_secret_77123`, `QA-ACCOUNT-998877`, or a title-cased copy of the private sentence would pass these assertions despite leaking the source material.
- Impact: This new high-risk privacy QA case can produce false passes for the exact behavior it is meant to guard: share-safe diagnostics artifacts and replies that omit raw personal content and fake secret material.
- Suggested fix: Normalize both haystack and needles for forbidden checks, matching the nearby `personal-task-followthrough-status` pattern, for example compare `normalizeLowercaseStringOrEmpty(artifact)` and `normalizeLowercaseStringOrEmpty(outbound.text)` against normalized `config.forbiddenNeedles`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. The failure mode follows directly from the current qa-flow expressions: JavaScript `String.prototype.includes` is case-sensitive, while the forbidden list contains mixed-case fake identifiers and a lowercase raw sentence.
- Is this the best way to solve the issue? Yes. The scenario already depends on `normalizeLowercaseStringOrEmpty`, and using it for forbidden checks keeps the contract local to the scenario without changing QA-Lab runtime behavior.
