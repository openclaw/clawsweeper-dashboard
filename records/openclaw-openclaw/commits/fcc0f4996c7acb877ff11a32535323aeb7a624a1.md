# Commit fcc0f4996c

## Summary

Found one low-severity test-harness bug in the new RTT marker plumbing. The default Docker RTT path is not affected, but the newly added `OPENCLAW_NPM_TELEGRAM_SUCCESS_MARKER` override cannot work with the mock provider.

## Findings

### Low: Custom RTT success markers are not echoed with sample suffixes

- Kind: bug
- File: `scripts/e2e/mock-openai-server.mjs`
- Line: 103
- Evidence: `scripts/e2e/npm-telegram-rtt-driver.mjs:18` allows `OPENCLAW_NPM_TELEGRAM_SUCCESS_MARKER`, and warm samples wait for `${successMarker}_${index + 1}` at `scripts/e2e/npm-telegram-rtt-driver.mjs:225`. The mock server only extracts hard-coded `OPENCLAW_E2E_OK(?:_\d+)?` at `scripts/e2e/mock-openai-server.mjs:103`, falling back to `SUCCESS_MARKER` without the sample suffix. A local probe with `SUCCESS_MARKER=CUSTOM_OK` and a prompt containing `CUSTOM_OK_1` returned `CUSTOM_OK`.
- Impact: Maintainers trying to run the RTT harness with a custom marker will get systematic sample timeouts under the mock OpenAI provider, because the SUT never sends the suffixed marker the driver is waiting for.
- Suggested fix: Use the configured marker when building the extraction regexp, escaping it first, or remove the unsupported driver override. The mock and driver should share one env name/contract.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: start `scripts/e2e/mock-openai-server.mjs` with `SUCCESS_MARKER=CUSTOM_OK`, POST a non-streaming Responses request whose input asks for `CUSTOM_OK_1`, and observe the returned output text is `CUSTOM_OK`, not `CUSTOM_OK_1`.
- Is this the best way to solve the issue? yes: the driver’s marker-matching contract and the mock’s echo logic need the same configurable marker source; broad Telegram changes are unnecessary.
