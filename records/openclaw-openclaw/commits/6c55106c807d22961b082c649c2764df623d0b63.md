# Commit 6c55106

## Summary

Found one concrete bug: the new Discord component persistence path rejects normal component entries before writing them to the real plugin state store, so active buttons/selects/forms still do not survive Gateway restarts.

## Findings

### Medium: Component entries are not JSON-safe for the real persistent store

- Kind: bug
- File: `extensions/discord/src/components-registry.ts`
- Line: 192
- Evidence: `registerPersistentRegistryEntries` writes `{ version: 1, entry }` directly to `openKeyedStore.register`. The real plugin state store calls `assertJsonSerializable` before `JSON.stringify`, and rejects `undefined` properties. Normal Discord entries are built with explicit optional fields such as `callbackData`, `modalId`, and `allowedUsers` at `extensions/discord/src/components.builders.ts:95`, and `addEntry` adds more explicit optional fields such as `sessionKey`, `agentId`, `accountId`, and `reusable` at `extensions/discord/src/components.builders.ts:252`. A focused probe with the real `createPluginStateKeyedStore` built a normal button entry, then logged `PluginStateStoreError: plugin state value at value.entry.modalId must be JSON-serializable`; after clearing memory, `resolveDiscordComponentEntryWithPersistence` returned `null`.
- Impact: the commit’s advertised restart recovery does not work for ordinary component messages. The first failed register disables persistent registry access for the process, leaving only the old in-memory behavior.
- Suggested fix: normalize entries before persistence by recursively omitting `undefined` optional fields, or construct component/modal entries without explicit `undefined` properties. Add a test that uses the real plugin keyed store with an isolated `OPENCLAW_STATE_DIR`, not only a permissive mocked store.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Build a normal Discord component message, set Discord runtime state to `createPluginStateKeyedStore("discord", opts)`, call `registerDiscordComponentEntries`, clear the in-memory registry, then resolve with persistence. The persistent lookup returns `null` and the warning reports the JSON-serializability failure.
- Is this the best way to solve the issue? yes. The dependency contract is explicit in `src/plugin-state/plugin-state-store.ts:223`: values must pass the JSON-serializable validator before persistence. Sanitizing at the Discord registry boundary preserves the existing component types while satisfying the store contract.
