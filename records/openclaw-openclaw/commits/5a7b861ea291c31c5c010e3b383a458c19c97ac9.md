# Commit 5a7b861

## Summary

The commit fixes unrelated discovered plugin diagnostics, but it also downgrades errors for plugins explicitly sourced through `plugins.load.paths`. That makes `openclaw config validate` report success for a broken configured plugin load path.

## Findings

### Medium: Configured `plugins.load.paths` plugin errors now pass validation

- Kind: regression
- File: `src/config/validation.ts`
- Line: 957
- Evidence: `collectExplicitPluginReferences` records `plugins.entries`, `plugins.allow`, `plugins.deny`, and `plugins.slots`, but not `plugins.load.paths`. `pushRegistryDiagnostics` now only promotes error diagnostics to issues when `explicitPath` exists or the diagnostic has no `pluginId`, so an error diagnostic with a `pluginId` from a configured load path is pushed as a warning.
- Impact: A config that explicitly asks OpenClaw to load a local plugin can now validate as `ok: true` even when that plugin’s runtime entry is missing. I reproduced this with a temporary plugin in `plugins.load.paths` whose package declared `runtimeExtensions: ["./dist/index.js"]`; validation returned no issues and only warned: `plugin broken-load-path: runtime extension entry not found: ./dist/index.js`. This weakens `config validate` for intentionally configured local plugins and can let broken plugin setups reach startup/runtime.
- Suggested fix: Treat registry diagnostics whose `source` is equal to or inside a configured `plugins.load.paths` entry as explicit config diagnostics. Keep those error-level diagnostics fatal, and add a regression test for a broken load-path plugin with a missing runtime entry.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Create a plugin directory referenced only by `plugins.load.paths`, give it `openclaw.plugin.json` id `broken-load-path`, `package.json` with `openclaw.extensions: ["./src/index.ts"]` and `runtimeExtensions: ["./dist/index.js"]`, omit `dist/index.js`, then call `validateConfigObjectWithPlugins`; current `main` returns `ok: true` with the diagnostic in `warnings`.
- Is this the best way to solve the issue? yes. The intended fix is config-scoped diagnostics, and `plugins.load.paths` is an explicit plugin config surface, so source/path matching preserves the new unrelated-plugin behavior without weakening configured local plugin validation.
