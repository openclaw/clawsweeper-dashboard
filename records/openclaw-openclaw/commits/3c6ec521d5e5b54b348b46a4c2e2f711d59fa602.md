# Commit 3c6ec521d5

## Summary

Found one low-severity completion-path bug: the new Windows profile resolver only distinguishes Windows PowerShell from pwsh when `SHELL` is populated, so explicit PowerShell installs can still write to the wrong profile directory.

## Findings

### Low: Windows PowerShell profile resolution falls back to the pwsh profile when `SHELL` is unset

- Kind: bug
- File: `src/cli/completion-runtime.ts`
- Line: 178
- Evidence: `resolveCompletionProfilePath("powershell")` now chooses `WindowsPowerShell` only when `env.SHELL` resolves exactly to `powershell`; every other Windows case, including an empty `SHELL`, falls back to `Documents\PowerShell`. `installCompletion("powershell", ...)` calls this resolver without passing any explicit host/version context at `src/cli/completion-runtime.ts:271`. A focused helper probe returned `C:\Users\Ada\Documents\PowerShell\Microsoft.PowerShell_profile.ps1` for `{ platform: "win32", env: { USERPROFILE: "C:\\Users\\Ada" } }`, while Microsoft documents Windows PowerShell’s current-user/current-host profile under `Documents\WindowsPowerShell\Microsoft.PowerShell_profile.ps1` and PowerShell 7’s profile under `Documents\PowerShell\Microsoft.PowerShell_profile.ps1`. citeturn3search0turn0search0
- Impact: A Windows PowerShell 5.1 user running `openclaw completion --install --shell powershell` in an environment without `SHELL` gets OpenClaw completion written to the PowerShell 7 profile path, so restarting Windows PowerShell will not load the installed completion.
- Suggested fix: Do not treat missing `SHELL` as pwsh. Pass a concrete host/profile signal into the resolver, or on Windows prefer the active/existing PowerShell profile directory and only choose `Documents\PowerShell` when pwsh is actually detected.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the resolver path: call `resolveCompletionProfilePath("powershell", { platform: "win32", env: { USERPROFILE: "C:\\Users\\Ada" } })` and it returns the pwsh profile directory. Live Windows shell behavior was not exercised.
- Is this the best way to solve the issue? Yes directionally: the fix belongs in the shared resolver so setup, doctor, status, and install use the same concrete profile decision.
