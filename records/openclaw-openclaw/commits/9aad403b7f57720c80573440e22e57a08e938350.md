# Commit 9aad403b7f

## Summary

Found one actionable bug in the new black-box Telegram RTT Docker script: it accepts and forwards provider mode, but always runs the mock OpenAI path.

## Findings

### Medium: `live-frontier` RTT runs are silently measured against the mock provider

- Kind: bug
- File: `scripts/e2e/npm-telegram-rtt-docker.sh`
- Line: 91
- Evidence: Current `scripts/rtt.ts` exposes `--provider mock-openai|live-frontier` and records that selected mode in the result, while `scripts/lib/rtt-harness.ts` now invokes this new `npm-telegram-rtt-docker.sh` script. The script forwards `OPENCLAW_NPM_TELEGRAM_PROVIDER_MODE`, but never branches on it; instead it always exports fake `OPENAI_API_KEY`, starts `scripts/e2e/mock-openai-server.mjs`, and writes config through `npm-telegram-rtt-config.mjs`, which unconditionally sets `models.providers.openai.baseUrl` to `http://127.0.0.1:<mock_port>/v1`.
- Impact: `pnpm rtt openclaw@latest --provider live-frontier` can produce result JSON labeled `live-frontier` while the SUT actually talks to the local mock server. That gives false RTT data and can mask live provider regressions in release/performance validation.
- Suggested fix: Either reject `OPENCLAW_NPM_TELEGRAM_PROVIDER_MODE != mock-openai` in this script, or implement the live-provider branch by forwarding the real provider key/model settings and generating config without the mock `baseUrl` or mock server.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Static path: `scripts/rtt.ts --provider live-frontier` passes `OPENCLAW_NPM_TELEGRAM_PROVIDER_MODE=live-frontier`; the Docker script still starts `mock-openai-server.mjs` and the config generator produces `baseUrl: http://127.0.0.1:44080/v1`.
- Is this the best way to solve the issue? yes. Failing fast for unsupported live mode is the safest narrow fix; implementing a real live branch is appropriate if RTT is meant to compare live providers.
