# Commit d1638f1

## Summary

One low-severity trajectory completeness issue found. The normal focused tests pass, but the new native trajectory recording can emit a `tool.result` without the matching `tool.call` on an existing completed-item-only path.

## Findings

### Low: Completed-only native tool notifications can record result without call

- Kind: bug
- File: `extensions/codex/src/app-server/event-projector.ts`
- Line: 542
- Evidence: `handleItemCompleted` marks an item completed, then calls `emitNormalizedToolItemEvent({ phase: "result", item })`. The new `recordToolTrajectoryEvent` only records `tool.call` for phase `"start"` and records `tool.result` for phase `"result"`. If Codex sends `item/completed` without a prior `item/started`, the adjacent transcript mirror still backfills the tool-call message via `recordNativeToolTranscriptCall`, but the trajectory stream does not backfill `tool.call`. A later turn snapshot also cannot repair it because `emitSnapshotOnlyNativeToolProgress` returns early for ids already in `completedItemIds`.
- Impact: Runtime trajectory sidecars can contain native `tool.result` events with no matching `tool.call`, so exported/debug trajectories lose the native tool arguments and have an incomplete call/result pair for that item.
- Suggested fix: Add a deduped native trajectory call-id set and record a missing `tool.call` before the result on completed-only native items, matching the existing transcript-call backfill behavior. Add a projector test for `item/completed` only with a `trajectoryRecorder`.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes: instantiate `CodexAppServerEventProjector` with a mock `trajectoryRecorder`, send only `item/completed` for a `commandExecution`, and inspect `recordEvent` calls. The code path records `tool.result` but never calls the `"start"` phase.
- Is this the best way to solve the issue? yes: trajectory recording should dedupe independently and mirror the transcript backfill instead of relying on app-server notification ordering.
