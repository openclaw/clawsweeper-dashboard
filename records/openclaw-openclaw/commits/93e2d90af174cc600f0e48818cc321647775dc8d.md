# Commit 93e2d90

## Summary

Found one actionable concurrency/reliability issue in the Discord IDENTIFY reconnect path. The committed regression test covers a closed current socket, but not a stale pending IDENTIFY waiter after a reconnect has already replaced `this.ws`.

## Findings

### Medium: Stale IDENTIFY waiters can authenticate the replacement socket twice

- Kind: concurrency
- File: `extensions/discord/src/internal/gateway.ts`
- Line: 333
- Evidence: `identifyWithConcurrency()` awaits the shared limiter, then reads `this.ws` after the await and calls `this.identify()`. If socket A receives HELLO and starts this wait, then closes and schedules reconnect, socket B can become `this.ws` before A's waiter resumes. A's stale waiter then sends IDENTIFY on B; B's own HELLO waiter later sends another IDENTIFY on the same socket.
- Impact: Discord can close the replacement connection as already authenticated (`GatewayCloseCodes.AlreadyAuthenticated = 4005`, treated as non-resumable here), causing avoidable reconnect churn and potentially reintroducing startup unresponsiveness under the same close-during-identify timing this commit is trying to repair.
- Suggested fix: capture the socket or a connection generation when handling HELLO, pass it into `identifyWithConcurrency`, and after the limiter await only IDENTIFY if that captured socket is still the active `this.ws`. Stale waiters should return without sending; the current socket's own waiter should own IDENTIFY/reconnect decisions.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A one-off in-memory script seeded the limiter, emitted HELLO on socket A, emitted close 1006, let reconnect create socket B, emitted HELLO on B, then observed B send `op:2` once at about 5s from A's stale waiter and again at about 10s from B's own waiter.
- Is this the best way to solve the issue? yes. The failure is caused by looking up mutable `this.ws` after an async wait; binding the wait to the socket/generation that received HELLO is the narrow fix and matches the existing stale socket guards in `setupWebSocket`.
