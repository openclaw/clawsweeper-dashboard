# Commit c6ddb1a

## Summary

Found one concrete propagation bug: the new `sourceReplyDeliveryMode` gateway/agent option reaches embedded runs, but is dropped when the same agent command selects a CLI backend. That leaves `message_tool_only` completion handoffs ineffective for CLI-backed agents.

## Findings

### Medium: CLI-backed agent runs drop `sourceReplyDeliveryMode`

- Kind: bug
- File: `src/agents/command/attempt-execution.ts`
- Line: 493
- Evidence: `agent` RPC now forwards `request.sourceReplyDeliveryMode` into `AgentCommandOpts` in `src/gateway/server-methods/agent.ts:1494`, and the embedded branch passes it through at `src/agents/command/attempt-execution.ts:633`. The CLI branch’s `runCliAgent({ ... })` object at `src/agents/command/attempt-execution.ts:493-528` omits `sourceReplyDeliveryMode`, even though `RunCliAgentParams` defines it at `src/agents/cli-runner/types.ts:42` and `prepareCliRunContext` consumes it for the CLI system prompt at `src/agents/cli-runner/prepare.ts:354`.
- Impact: If a generated media completion fallback runs through a CLI provider, the gateway request can carry `sourceReplyDeliveryMode: "message_tool_only"` but the CLI runtime sees `undefined`. The CLI prompt therefore does not get the message-tool-only delivery guidance, so group/channel media completions can still complete without posting the generated attachment through the required message-tool path.
- Suggested fix: Add `sourceReplyDeliveryMode: params.opts.sourceReplyDeliveryMode` to the CLI `runCliAgent` call in `runAgentAttempt`, and add a focused test for CLI-provider propagation alongside the existing embedded/gateway forwarding tests.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Code-level reproduction is to route `agent` through a CLI backend with `sourceReplyDeliveryMode: "message_tool_only"`; `server-methods/agent.ts` forwards the option, but the CLI branch in `attempt-execution.ts` drops it before `prepareCliRunContext`.
- Is this the best way to solve the issue? yes. The CLI runner already has a typed parameter and prompt plumbing for this option, so the narrow fix is to pass the existing value through.
