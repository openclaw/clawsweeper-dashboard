# Commit d9c6c5f

## Summary

Found one actionable bug: the commit preserves OpenAI-compatible usage aliases in gateway chat history, but the Control UI chat context renderer still ignores those aliases, so the advertised llama.cpp context display path can remain blank.

## Findings

### Medium: Preserved OpenAI usage aliases are not consumed by the chat context renderer

- Kind: bug
- File: `ui/src/ui/chat/grouped-render.ts`
- Line: 511
- Evidence: The commit now keeps `prompt_tokens`, `completion_tokens`, and `total_tokens` in sanitized assistant usage (`src/gateway/chat-display-projection.ts:243`), and the new regression test only verifies preservation (`src/gateway/server-methods/server-methods.test.ts:533`). But the Control UI renderer reads only `usage.input ?? usage.inputTokens`, `usage.output ?? usage.outputTokens`, and cache aliases when building the Context badge (`ui/src/ui/chat/grouped-render.ts:511`). A live JSDOM render probe with `usage: { prompt_tokens: 11000, completion_tokens: 250, total_tokens: 11250 }` and `contextWindow: 131000` rendered the model label but reported `ctx=missing` and `tokens=missing`.
- Impact: `chat.history` can now deliver the llama.cpp/OpenAI-compatible usage fields, but WebChat/Control UI still does not display token counts or context percentage for those messages. That leaves the “keep context counts” behavior incomplete for at least the chat display surface.
- Suggested fix: Normalize chat message usage before rendering, or extend `extractGroupMeta` to read the same aliases as `normalizeUsage`: `input_tokens`/`promptTokens`/`prompt_tokens` for input and `output_tokens`/`completionTokens`/`completion_tokens` for output. Add a grouped-render regression test using the preserved OpenAI alias shape.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The source path is `chat.history` -> `projectRecentChatDisplayMessages` -> UI `renderMessageGroup`; the ad-hoc JSDOM render probe confirmed preserved OpenAI alias usage renders no token count and no context percentage.
- Is this the best way to solve the issue? yes. The gateway sanitizer should keep safe numeric aliases, but display consumers also need to normalize them before computing context metadata.
