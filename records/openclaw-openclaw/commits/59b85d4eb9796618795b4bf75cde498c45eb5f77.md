## Findings

### Low: Matrix intro wait can reject without a handler if the trigger send fails

- Kind: reliability
- File: `extensions/qa-matrix/src/runners/contract/scenario-runtime-room.ts`
- Line: 320
- Evidence: `introPromise` is created before sending the trigger, but it is not awaited or otherwise given a rejection handler until after `client.sendTextMessage(...)` completes at lines 337-342. The Matrix wait path throws on timeout/missing match (`extensions/qa-matrix/src/substrate/sync.ts:184`), and `sendTextMessage` can reject through the Matrix send request or missing `event_id` (`extensions/qa-matrix/src/substrate/client.ts:440`, `extensions/qa-matrix/src/substrate/client.ts:564`).
- Impact: if the Matrix trigger send fails, or if the intro wait rejects quickly while the send is still pending, the wait promise can surface as an unhandled rejection and obscure the original release-validation failure. In strict unhandled-rejection modes, this can terminate the process.
- Suggested fix: attach the wait handler immediately, for example by awaiting send and intro through `Promise.all`, or by catching/draining `introPromise` on send failure before rethrowing.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes, with a focused unit around `runSubagentThreadSpawnScenario` that mocks `waitForRoomEvent` to reject and `sendTextMessage` to reject or delay; the current code leaves `introPromise` unhandled until after the send await.
- Is this the best way to solve the issue? Yes; keeping the pre-armed wait is reasonable for the race it fixes, but its rejection needs to be handled immediately.
