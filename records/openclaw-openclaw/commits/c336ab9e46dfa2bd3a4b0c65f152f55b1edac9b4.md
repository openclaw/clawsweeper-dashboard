## Findings

### Medium: Release check calls an unregistered `plugins deps --repair` command

- Kind: regression
- File: `scripts/release-check.ts`
- Line: 370
- Evidence: `runPackedBundledPluginActivationSmoke` now executes `[openclaw.mjs, "plugins", "deps", "--repair"]`, but `src/cli/plugins-cli.ts` registers only `list`, `search`, `inspect`, `enable`, `disable`, `uninstall`, `install`, `update`, `registry`, `doctor`, and `marketplace` under `plugins`. A focused isolated CLI check on current `main` failed with `error: unknown option '--repair'`; `openclaw plugins --help` also shows no `deps` subcommand, and `openclaw plugins deps` fails with `too many arguments for 'plugins'`.
- Impact: `pnpm release:check` reaches this new smoke during `runPackedBundledChannelEntrySmoke` and fails on every run, blocking the release preflight the commit was meant to repair.
- Suggested fix: restore/register the intended `plugins deps --repair` command and add a command-registration or packed CLI smoke assertion that executes it, or change the release smoke to an existing supported repair path.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: after installing missing deps, run `HOME=$(mktemp -d) OPENCLAW_STATE_DIR=$(mktemp -d) OPENCLAW_NO_ONBOARD=1 OPENCLAW_SUPPRESS_NOTES=1 pnpm --silent openclaw plugins deps --repair`; it exits 1 with `error: unknown option '--repair'`.
- Is this the best way to solve the issue? unclear: the changelog/docs mention `openclaw plugins deps --repair`, so registering that command appears intended, but using an existing doctor repair path may be safer if the command was deliberately removed.
