# Commit cc4a596

## Summary

Found two actionable regressions in the realtime Discord barge-in path. Existing targeted tests pass, but they do not cover the affected sequences.

## Findings

### Medium: A skipped early barge-in consumes the only same-speaker retry

- Kind: regression
- File: `extensions/discord/src/voice/realtime.ts`
- Line: 314
- Evidence: `sendInputAudioForTurn` sets `turn.interruptedPlayback = true` before calling `this.handleBargeIn()`. The new OpenAI provider guard returns early when `audioEndMs < minBargeInAudioEndMs` without canceling, truncating, or clearing audio (`extensions/openai/realtime-voice-provider.ts:832`). After that first skipped chunk, later chunks from the same speaker turn cannot retry because `interruptedPlayback` is already true.
- Impact: a real user who starts speaking during the first 250 ms of assistant playback can be ignored for the rest of that captured turn, so the assistant continues talking over them even after the configured minimum window has elapsed.
- Suggested fix: do not mark the Discord speaker turn as interrupted until the provider actually accepts/clears the barge-in, or make skipped barge-ins retryable for the same turn. A small return value/event contract from `handleBargeIn` would make this explicit.
- Confidence: high

### Medium: Non-OpenAI realtime providers can no longer stop Discord playback on barge-in

- Kind: regression
- File: `extensions/discord/src/voice/realtime.ts`
- Line: 329
- Evidence: this commit removes Discordâ€™s local `clearOutputAudio()` fallback after `this.bridge?.handleBargeIn({ audioPlaybackActive: true })`. But the session wrapper calls provider `handleBargeIn` with optional chaining, so providers without an implementation no-op (`src/talk/session-runtime.ts:76`). The Google realtime provider advertises `supportsBargeIn: true` but does not implement `handleBargeIn` (`extensions/google/realtime-voice-provider.ts:915`), meaning Discord playback is not stopped immediately anymore for Google Live voice sessions.
- Impact: Discord realtime voice with a non-OpenAI provider can keep playing assistant audio over the userâ€™s new speech, recreating the feedback/overlap behavior the barge-in path is meant to avoid.
- Suggested fix: keep a Discord-side clear fallback when the provider cannot confirm it handled barge-in, or require provider `handleBargeIn` to return whether it cleared/interrupted playback and update Google accordingly.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. For the first finding, simulate an OpenAI response audio delta at media timestamp `1000`, call Discord same-speaker `sendInputAudio` while `latestMediaTimestamp` is still below `1250`, then advance timestamp past `1250` and send another chunk from the same turn. The current code skips the first provider interruption and never retries. For the second finding, configure Discord realtime with provider `google`, start playback, then trigger `handleSpeakingStart`; the wrapperâ€™s optional `handleBargeIn` call no-ops and Discord no longer clears output itself.
- Is this the best way to solve the issue? No. Letting the provider own clearing is reasonable, but the current contract gives Discord no success/failure signal and does not require every barge-in-capable provider to implement the new behavior.
