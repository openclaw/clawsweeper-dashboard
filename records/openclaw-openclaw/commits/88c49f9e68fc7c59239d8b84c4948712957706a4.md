# Commit 88c49f9

## Summary

Found one low-severity regression risk in the xAI stream wrapper. The rest of the helper extractions I checked were behavior-preserving under focused tests.

## Findings

### Low: xAI tool-call entity decoding no longer fully decodes amp-prefixed entities

- Kind: regression
- File: `extensions/xai/stream.ts`
- Line: 63
- Evidence: The commit replaced the previous ordered chained replacements with a single regex replacement. On current `main`, an xAI tool-call argument value like `&amp;lt;tag&amp;gt;` is decoded only to `&lt;tag&gt;`, not `<tag>`. The prior implementation decoded `&amp;` first, then decoded the resulting `&lt;`, `&quot;`, and numeric entities in later replacements. The xAI compat contract declares tool-call argument encoding as `html-entities` in `extensions/xai/model-compat.ts:18`.
- Impact: If xAI or a routed xAI model returns double-escaped tool arguments, tools receive still-escaped strings. This can break concrete values such as selectors, HTML/XML snippets, quoted strings, or numeric character references.
- Suggested fix: Preserve the old multi-stage behavior, either by keeping the ordered replacements or by looping the known-entity decoder until stable with a small bound. Add a regression test for `&amp;lt;`, `&amp;quot;`, and `&amp;#123;`.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes. I ran the current wrapper with a toolCall message containing `&amp;lt;tag&amp;gt;`, `&amp;quot;x&amp;quot;`, and `&amp;#123;`; the decoded arguments remained `&lt;tag&gt;`, `&quot;x&quot;`, and `&#123;`.
- Is this the best way to solve the issue? yes, if the old double-decode behavior was intentional. If only single-encoded xAI output is supported, add an explicit test documenting that narrower contract instead.
