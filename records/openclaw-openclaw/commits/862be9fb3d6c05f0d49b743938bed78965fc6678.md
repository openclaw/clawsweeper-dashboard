# Commit 862be9f

## Summary

Found one concrete compatibility gap: the new schema rewrite is only reachable through the bundled `xiaomi` provider hook, while the reported failing Token Plan setup used a custom OpenAI-compatible provider id. Current config/catalog contracts also reject or drop the new compat flag, so affected custom Xiaomi routes cannot opt into the fix.

## Findings

### Medium: Custom Xiaomi Token Plan providers cannot enable the new array-items rewrite

- Kind: regression
- File: `src/config/zod-schema.core.ts`
- Line: 224
- Evidence: The commit sets `omitEmptyArrayItems` only in the bundled Xiaomi plugin via `normalizeResolvedModel` at `extensions/xiaomi/index.ts:42`, and the schema stripping only runs when `modelCompat.omitEmptyArrayItems` is present at `src/agents/pi-tools-parameter-schema.ts:339`. The public model compat config remains strict and does not include that key at `src/config/zod-schema.core.ts:190-224`; a focused probe for `models.providers["xiaomi-mimo-tp"].models[0].compat.omitEmptyArrayItems: true` fails with `Unrecognized key: "omitEmptyArrayItems"`. Even if validation is bypassed, `normalizeModelCatalogCompat` only copies its known boolean fields at `src/model-catalog/normalize.ts:163-179`, and a probe showed it drops `omitEmptyArrayItems`.
- Impact: Issue #82447’s original failing shape was a custom Token Plan provider such as `xiaomi-mimo-tp/mimo-v2.5` on `token-plan-cn.xiaomimimo.com/v1`, not necessarily the bundled `xiaomi` provider id. Those users continue sending the synthetic `items: {}` payload and have no supported config escape hatch for the new compat behavior.
- Suggested fix: Either promote `omitEmptyArrayItems` into `ModelCompatConfig`, `ModelCompatSchema`, model-catalog normalization, generated docs/baselines, and add a custom-provider regression; or add a provider/baseUrl-aware compat hook that marks Xiaomi MiMo Token Plan routes even when the provider id is custom.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the config/contract failure: `OpenClawSchema.safeParse` rejects the custom-provider compat key, and `normalizeModelCatalogRows` drops it when validation is bypassed. Live Token Plan request reproduction remains unverified because no Token Plan credential is available here.
- Is this the best way to solve the issue? Yes, if the intended fix is to cover OpenAI-compatible Xiaomi routes beyond the bundled `xiaomi` id. Exposing the compat flag through the same model compat contract used by other custom-provider quirks is the smallest general fix; baseUrl detection is more automatic but needs careful provider ownership boundaries.
