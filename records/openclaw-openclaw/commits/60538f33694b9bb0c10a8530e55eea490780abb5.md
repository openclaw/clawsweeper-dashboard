# Commit 60538f3

## Summary

Found one compatibility regression: the commit removes existing exports from a public plugin SDK subpath.

## Findings

### Medium: Public media-runtime exports were removed without a compatible alias

- Kind: compatibility
- File: `src/media/qr-image.ts`
- Line: 13
- Evidence: `openclaw/plugin-sdk/media-runtime` is a package export in `package.json:349` and re-exports `src/media/png-encode.ts` and `src/media/qr-image.ts` from `src/plugin-sdk/media-runtime.ts:17`. The commit removes exported `crc32`, `pngChunk`, `QrPngRenderOptions`, `QrPngTempFileOptions`, and `QrPngTempFile`. The scoped SDK guidance says `src/plugin-sdk` is a public contract for bundled and third-party plugins and that breaking removals are major-version work, not drive-by cleanup (`src/plugin-sdk/AGENTS.md:3`, `src/plugin-sdk/AGENTS.md:93`).
- Impact: third-party plugins importing `type QrPngRenderOptions` / `type QrPngTempFileOptions` / `type QrPngTempFile` from `openclaw/plugin-sdk/media-runtime` will now fail to typecheck, and JS/TS runtime imports of `crc32` or `pngChunk` from that same public subpath will fail module instantiation.
- Suggested fix: restore these exports as compatibility aliases, or move the removal through the repo’s documented breaking-change/versioning path. If the intent is only to discourage use, keep the exports and mark them deprecated.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused TypeScript compiler API probe over `src/plugin-sdk/media-runtime.ts` on current `main` reports the five removed names as missing while the remaining media helpers are still exported; the parent commit source shows those five names were exported.
- Is this the best way to solve the issue? no. The safer fix is to preserve the public names and deprecate them, because existing plugin consumers may import them from the published `openclaw/plugin-sdk/media-runtime` subpath.
