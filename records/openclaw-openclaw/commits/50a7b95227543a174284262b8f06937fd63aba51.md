# Commit 50a7b95227

## Summary

Found one reconnect readiness regression in the OpenAI realtime voice bridge.

## Findings

### Medium: Pre-ready reconnect can leave `connect()` permanently pending

- Kind: reliability
- File: `extensions/openai/realtime-voice-provider.ts`
- Line: 413
- Evidence: The commit moved `connect()` resolution from WebSocket `open` to `session.updated`, but the unintentional `close` handler still only starts `void this.attemptReconnect()` and never settles the original `doConnect()` promise. If the first socket opens, closes before `session.updated`, and the background reconnect reaches `session.updated` before the first attempt's 10s timeout, `this.sessionConfigured` becomes true on the later socket, the first timeout does not reject, and the original `connect()` promise has no remaining path to resolve. Google Meet awaits that promise before returning the realtime audio bridge handle in `extensions/google-meet/src/realtime.ts:475` and `extensions/google-meet/src/realtime-node.ts:212`, so a transient close during session configuration can hang Meet join/startup even after reconnect succeeds.
- Impact: A short network/provider close in the new readiness window can strand callers waiting for `connect()`, especially Google Meet paths that now depend on readiness before completing setup.
- Suggested fix: Make pre-ready close/reconnect part of the same `connect()` settlement path. Either reject immediately on close before `session.updated`, or chain the reconnect attempt so the original promise resolves when a replacement socket reaches readiness and rejects when reconnect is exhausted.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. Event ordering: call `bridge.connect()`, emit WebSocket `open`, emit `close` before any `session.updated`, let `attemptReconnect()` create a second socket, then emit `open` and `session.updated` on the second socket before the first timeout. The second `doConnect()` resolves internally, but the first `doConnect()` promise awaited by the original caller remains unsettled.
- Is this the best way to solve the issue? Unclear. The safer design is to define the bridge contract explicitly: during initial readiness, either fail fast on pre-ready close or make reconnect transparent to the initial `connect()` promise. Transparent reconnect best preserves the existing reconnect intent, but it needs `attemptReconnect()` to surface success/failure rather than only firing callbacks.
