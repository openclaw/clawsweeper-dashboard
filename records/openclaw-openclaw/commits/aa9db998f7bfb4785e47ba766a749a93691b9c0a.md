# Commit aa9db998

## Summary

Found one concrete regression risk in the CLI proxy bootstrap ordering. The commit lets foreground gateway recovery proceed past invalid config, but the managed network proxy is still bootstrapped before that recovery happens, using the invalid pre-recovery config.

## Findings

### Medium: Gateway recovery can start without the proxy restored by last-known-good config

- Kind: security
- File: `src/cli/run-main.ts`
- Line: 349
- Evidence: `runCli()` now loads `readBestEffortConfig()` and immediately calls `startProxy(config?.proxy ?? undefined)` before the gateway fast path runs. `readBestEffortConfig()` returns `result.snapshot.config` when the current file is invalid, so this can be the malformed rejected config. `startProxy()` then treats anything other than `config.enabled === true` as disabled. The actual last-known-good recovery occurs later in `src/cli/gateway-cli/run.ts`, after proxy bootstrap, and `src/gateway/server-startup-config.ts` can then restore and start from a valid recovered config without retrying proxy startup.
- Impact: If a user has a valid last-known-good config with `proxy.enabled=true`, then directly edits `openclaw.json` into an invalid shape that also changes `proxy.enabled` to a non-boolean value, foreground `openclaw gateway run` can skip managed proxy activation from the rejected config, recover the valid last-known-good file, and continue gateway startup in a process where `OPENCLAW_PROXY_ACTIVE` was never set. That bypasses the operator-managed egress proxy for the recovered gateway process until restart.
- Suggested fix: Bootstrap the managed proxy from the post-recovery gateway config, not the pre-recovery best-effort config. A narrow fix is to move foreground gateway config recovery before proxy activation, or retry/reconcile proxy startup after `readGatewayStartupConfig()` rereads a recovered snapshot. At minimum, malformed `proxy` config from an invalid best-effort snapshot should fail closed rather than silently treating `enabled !== true` as disabled.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the code path: seed a valid last-known-good config with `proxy.enabled=true`, then write an invalid current config whose recovery is allowed and whose `proxy.enabled` is not boolean. Start `openclaw gateway run`; the proxy decision is made before recovery and is not revisited.
- Is this the best way to solve the issue? The best direction is to make recovery happen before proxy activation for the foreground gateway path, because validating or coercing only `proxy` still leaves bootstrap using rejected state.
