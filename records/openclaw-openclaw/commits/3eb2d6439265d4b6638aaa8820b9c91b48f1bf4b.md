# Commit 3eb2d64

## Summary

Found two actionable issues in the new live Codex plugin release lane: one security-risky default for custom plugin specs, and one documented `git:` path that fails after install.

## Findings

### Medium: Custom Codex plugin specs bypass the dangerous-code scanner by default

- Kind: security
- File: `scripts/e2e/codex-npm-plugin-live-docker.sh`
- Line: 110
- Evidence: The lane defaults `OPENCLAW_CODEX_NPM_PLUGIN_FORCE_UNSAFE_INSTALL` to `1`, then appends `--dangerously-force-unsafe-install` to every `openclaw plugins install "$CODEX_PLUGIN_SPEC"` call. The same Docker run passes live `OPENAI_API_KEY`/`OPENAI_BASE_URL`, and the new docs explicitly allow operator-supplied `npm:`, `npm-pack:`, and `git:` plugin specs. Repo docs describe this flag as break-glass for scanner false positives.
- Impact: A typo, compromised package, or intentionally custom plugin spec in this secret-bearing release lane can skip the built-in critical dangerous-code block and then load plugin code with live CI credentials available in the container.
- Suggested fix: Default the unsafe override to off for explicit `codex_plugin_spec` values. If the same-ref packed Codex tarball truly needs the override, apply it only to that generated trusted tarball or require a separate explicit workflow input/env opt-in for custom specs.
- Confidence: high

### Medium: Documented `git:` Codex plugin specs fail the live assertion

- Kind: compatibility
- File: `scripts/e2e/lib/codex-npm-plugin-live/assertions.mjs`
- Line: 232
- Evidence: `normalizePluginSpec()` accepts `git:` specs, and `docs/ci.md:157` documents `git:` as a supported `codex_plugin_spec`. But `assertNpmDeps()` unconditionally requires the Codex install path and package file to resolve inside the managed npm root. Git plugin installs persist `installPath: result.targetDir`, where `result.targetDir` is the managed git repo root, not the npm root. A temp-state probe of `assert-npm-deps` with a git install record failed at this exact path-inside assertion.
- Impact: `codex_plugin_spec=git:...` can install successfully but the live lane then fails before validating the agent turn, so the newly documented custom git source is unusable.
- Suggested fix: Either remove `git:` from the documented/accepted live lane contract, or branch the dependency assertion by install source so git installs verify `@openai/codex` under the git install root while keeping npm/npm-pack installs constrained to the managed npm root.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The unsafe-install issue follows directly from the default env and install flag path. The `git:` issue was reproduced with a temp `OPENCLAW_STATE_DIR` install record that matches the git install contract and failed in `assert-npm-deps`.
- Is this the best way to solve the issue? yes for the fix direction: keep scanner bypass explicit and source-scoped, and align assertions with the documented install source contract.
