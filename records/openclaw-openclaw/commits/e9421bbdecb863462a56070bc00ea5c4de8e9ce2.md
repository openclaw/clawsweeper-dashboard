# Commit e9421bb

## Summary

The new already-installed Codex path can report success even when config policy blocks enabling the plugin.

## Findings

### Medium: Existing Codex install path ignores enablement failure

- Kind: regression
- File: `src/commands/codex-runtime-plugin-install.ts`
- Line: 68
- Evidence: The new branch calls `enablePluginInConfig(...)`, but returns `installed: true` and `status: "installed"` even when `enableResult.enabled` is false. The normal onboarding install path handles the same condition by noting the policy block and returning `installed: false`, `status: "failed"` from `src/commands/onboarding-plugin-install.ts:1179`.
- Impact: Re-running onboarding on a host with an existing Codex install record and `plugins.enabled: false`, a denylist entry, or an allowlist that excludes `codex` now silently reports success while leaving Codex disabled. Callers that only consume `.cfg` can persist the selected OpenAI/Codex model and continue, but the Codex plugin still will not load.
- Suggested fix: In the existing-record branch, honor `enableResult.enabled`. Return `installed: false`, `status: "failed"` and emit the same note/error as `applyPluginEnablement`, or factor that helper so both paths share the policy behavior.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused `node --import tsx --eval` probe wrote a temp `codex` install record with a real `package.json`, then called `ensureCodexRuntimePluginForModelSelection` with `cfg: { plugins: { enabled: false } }`; it returned `required: true`, `installed: true`, `status: "installed"`, left `plugins.enabled: false`, and emitted no note/error.
- Is this the best way to solve the issue? yes. The existing install path already treats blocked plugin enablement as install failure, so the new fast path should preserve that contract.
