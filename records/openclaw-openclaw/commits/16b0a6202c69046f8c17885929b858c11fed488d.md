## Summary

Found one media-dedupe regression in the copy-on-write rewrite. The other queue-index/splice changes appear behavior-preserving.

## Findings

### Medium: Stripping duplicate `mediaUrl` can drop unrelated `mediaUrls`

- Kind: data_loss
- File: `src/auto-reply/reply/reply-payloads-dedupe.ts`
- Line: 87
- Evidence: `filteredUrls` is only initialized when an entry inside `mediaUrls` is stripped. If `mediaUrl` matches `sentMediaUrls` but every `mediaUrls` entry is non-duplicate, `stripSingle` triggers the copy at lines 85-88 while `filteredUrls` remains `undefined`, so the returned payload clears `mediaUrls`. Focused probe on current `main`: `payloads: [{ text: "gallery", mediaUrl: "file:///tmp/a.jpg", mediaUrls: ["file:///tmp/b.jpg"] }], sentMediaUrls: ["file:///tmp/a.jpg"]` returns `[{"text":"gallery"}]`. Before this commit, the old `mediaUrls?.filter(...)` path preserved `["file:///tmp/b.jpg"]`.
- Impact: final reply/media dedupe can suppress the already-sent single media item but accidentally drop additional unsent attachments from the same payload, so users may never receive those files.
- Suggested fix: when `stripSingle` is true and `mediaUrls` exists but no array entries were stripped, preserve the original `mediaUrls` array, or initialize `filteredUrls` to `mediaUrls` before constructing `nextPayload`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The helper-level input above reproduces the dropped `mediaUrls` on current `main`.
- Is this the best way to solve the issue? yes. The fix should stay local to `filterMessagingToolMediaDuplicates` and preserve the old filtering semantics while keeping the new copy-on-write optimization.
