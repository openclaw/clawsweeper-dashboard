# Commit c8fa0fd1c9

## Summary

Found one onboarding regression: the new Codex app-server auth choice is visible only while the external Codex plugin is not installed. Once Codex is installed, bundled, or present in a source checkout, the cold onboarding prompt loses that choice.

## Findings

### Medium: Codex setup choice disappears for installed or bundled Codex plugin

- Kind: bug
- File: `extensions/codex/openclaw.plugin.json`
- Line: 5
- Evidence: The commit adds the Codex app-server auth method only to runtime provider code at `extensions/codex/provider.ts:60`, but onboarding’s cold provider setup surface reads manifest/setup metadata via `plugin.providerAuthChoices` and setup-provider metadata at `src/plugins/provider-auth-choices.ts:203`. The official external install-catalog fallback is skipped when the plugin id is already installed at `src/plugins/provider-install-catalog.ts:279`. The Codex manifest declares `"providers": ["codex"]` at `extensions/codex/openclaw.plugin.json:5`, but has no `providerAuthChoices` or `setup.providers` metadata for the `"codex"` choice.
- Impact: Fresh packaged installs can see `Codex app-server` before installing `@openclaw/codex`, but after installation, re-running setup no longer shows that provider choice. Source checkouts hit the same state immediately because `extensions/codex` is already present. A focused query of `resolveProviderSetupFlowContributions` on current `main` returned only `openai-codex` and `openai-codex-device-code`, even with `plugins.entries.codex.enabled=true`; no `codex` / `Codex app-server` setup contribution was produced.
- Suggested fix: Mirror the Codex app-server setup choice into cold manifest metadata, preferably `providerAuthChoices` or a `setup.providers` entry with the same choice id, label, group, hint, priority, method id, and onboarding scopes as `extensions/codex/provider.ts`. Add a regression test for the installed/bundled Codex case, not only the external-not-installed catalog case.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Run a cold setup contribution query in the repo checkout: `pnpm exec tsx -e "import { resolveProviderSetupFlowContributions } from './src/flows/provider-flow.ts'; for (const cfg of [{}, {plugins:{entries:{codex:{enabled:true}}}}]) console.log(JSON.stringify(resolveProviderSetupFlowContributions({config:cfg as any,workspaceDir:process.cwd()}).filter(c=>c.providerId==='codex'||String(c.option.value).includes('codex')).map(c=>({value:c.option.value,label:c.option.label,source:c.source,pluginId:c.pluginId}))))"`. It prints only the OpenAI Codex choices from the OpenAI plugin, not the new Codex app-server choice.
- Is this the best way to solve the issue? yes. The repo’s current setup flow intentionally uses manifest/install-catalog metadata before provider runtime loads, so the durable fix is to publish the Codex auth choice in the Codex manifest metadata rather than relying on `provider.ts` runtime auth metadata.
