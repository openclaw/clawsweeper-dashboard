# Commit 3e6f749

## Summary

Found one low-severity regression in the new default-port preservation logic. The main fixed scenario passes, but explicit `:80`/`:443` URLs can be corrupted when the path, query, or fragment contains `@`.

## Findings

### Low: Default-port normalization treats `@` in query/path as userinfo

- Kind: bug
- File: `extensions/browser/src/browser/cdp.helpers.ts`
- Line: 85
- Evidence: In the `hasExplicitPort && !parsed.port` branch, `rest` contains the normalized authority plus path/query/hash, but `rest.indexOf("@")` searches all of it. `resolveProfile` then uses this `normalizedWithPort` whenever the raw URL had an explicit default port at `extensions/browser/src/browser/config.ts:506`.
- Impact: A profile like `https://remote.example.com:443/json/version?token=user@example.com` resolves to `https://remote.example.com/json/version?token=user@example.com:443`, corrupting the token/path instead of preserving `:443` in the authority. Remote CDP endpoints that use default HTTPS ports plus email-like or `@`-bearing tokens can fail authentication or discovery.
- Suggested fix: Limit the `@` search to the authority portion before the first `/`, `?`, or `#`, preferably using `lastIndexOf("@")` within that authority slice, then insert the port before the authority terminator. Add regression tests for `@` in query and path with explicit default ports.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Running current `main` through `resolveProfile` with `cdpUrl: "https://remote.example.com:443/json/version?token=user@example.com"` produced `cdpUrl: "https://remote.example.com/json/version?token=user@example.com:443"`.
- Is this the best way to solve the issue? yes. The bug is local to the port-preserving reconstruction logic; bounding userinfo detection to the URL authority preserves the intended fix without changing the profile precedence rule.
