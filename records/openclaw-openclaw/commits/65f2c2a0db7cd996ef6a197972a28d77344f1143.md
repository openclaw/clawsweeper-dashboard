# Commit 65f2c2a

## Summary

Found one privacy/security regression in the new diagnostic cron-session context parser.

## Findings

### Medium: Non-canonical cron-like session keys can path-traverse transcript reads

- Kind: privacy
- File: `src/logging/diagnostic-session-context.ts`
- Line: 25
- Evidence: `parseCronRunSessionKey` accepts any agent session key with a later `cron` segment and then any later `run` segment (`parts.indexOf("cron")`, `parts.indexOf("run")`) instead of requiring the canonical `agent:<agent>:cron:<job>:run:<id>` shape. Existing cron-key contract code only accepts `^cron:[^:]+:run:[^:]+$` after the agent prefix, and its tests explicitly reject `agent:main:slack:cron:job:run:uuid` as non-canonical. The new resolver then builds `path.join(resolveStateDir(), "agents", agentId, "sessions", `${runId}.jsonl`)` with the raw parsed run id, so `../` segments are honored before reading the tail and logging `lastAssistant`.
- Impact: A crafted or channel-derived non-cron session key containing `:cron:<job>:run:<path>` can make stalled-session or stuck-session recovery diagnostics read a transcript outside the intended `agents/<agent>/sessions` directory and include that assistant snippet in warning logs. I reproduced this with `agent:main:slack:cron:job:run:../../other/sessions/secret`: `isCronRunSessionKey` returned `false`, but the new diagnostic parser emitted `lastAssistant="SECRET_FROM_OTHER_AGENT"`.
- Suggested fix: Reuse the canonical session-key parser/contract (`parseAgentSessionKey` plus exact `cron:<job>:run:<id>` matching, or `isCronRunSessionKey` plus capture), reject non-canonical cron-like keys, validate `runId` with the session transcript ID rules, and resolve the transcript through the existing safe session path helper rather than raw `path.join`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused `pnpm exec tsx --eval` probe created a temp state dir with `agents/other/sessions/secret.jsonl`, passed `agent:main:slack:cron:job:run:../../other/sessions/secret`, and observed the new diagnostic formatter include the other agent’s assistant text while the canonical cron classifier returned `false`.
- Is this the best way to solve the issue? yes. The bug is in accepting and path-resolving an over-broad shape; constraining to the existing canonical cron-run session contract and safe transcript path helpers fixes both the false-positive cron classification and traversal.
