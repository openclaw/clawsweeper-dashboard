# Commit 0db0979

## Summary

Found one low-severity false-positive regression in the new module-access screening. The runtime hardening mostly holds up under the focused code-mode tests, but benign template strings containing `import ...` are rejected before execution.

## Findings

### Low: Template strings containing ordinary `import` text are rejected as module access

- Kind: bug
- File: `src/agents/code-mode.ts`
- Line: 340
- Evidence: `maskCodeLiteralsAndComments()` masks comments plus single and double quoted strings, but it does not mask template literal raw text. `rejectsModuleAccess()` then scans the unmasked template content with `/\bimport\b.../`, so `const message = \`import docs later\`; return message;` fails with `code mode module access is disabled.`
- Impact: Code mode rejects valid JavaScript that only uses a string value. Template literals are common for building messages and structured output, so this can break benign generated code despite the commit adding coverage for “strings that contain import”.
- Suggested fix: Make module-access detection template-aware. Mask raw template chunks, but still scan `${...}` expressions so `` `${import('node:fs')}` `` remains blocked. A parser or TypeScript scanner would be safer than expanding the regex state machine.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Running the actual current `createCodeModeTools()` path with `code: 'const message = \`import docs later\`; return message;'` returned `{"status":"failed","error":"code mode module access is disabled.","code":"invalid_input",...}`.
- Is this the best way to solve the issue? yes, at the contract level. The current scanner already tries to distinguish code from string content; the missing template-literal state is the narrow bug.
