# Commit c33968e

## Summary

Found one low-severity chunking regression in the new CJK punctuation fallback.

## Findings

### Low: Early CJK punctuation can force tiny Discord chunks

- Kind: bug
- File: `extensions/discord/src/chunk.ts`
- Line: 103
- Evidence: `findCjkPunctuationBreak()` accepts any punctuation break with `start > 0`, so the first punctuation inside the window wins even if it is only a couple characters into a 2000-character window. A focused smoke with `chunkDiscordText("注：" + "a".repeat(25), { maxChars: 10 })` now returns `["注：","aaaaaaaaaa","aaaaaaaaaa","aaaaa"]`.
- Impact: Long no-space mixed replies such as CJK labels followed by URLs, identifiers, or minified text can produce a nearly empty first Discord message before the useful content. The prior hard fallback would have filled the first chunk up to the limit, and the issue/PR context explicitly called for a guarded CJK punctuation fallback to avoid degenerate tiny chunks.
- Suggested fix: Add a minimum-progress guard for CJK punctuation breaks, for example requiring the punctuation break to be past a small fraction of `window.length`, and fall back to the code-point-safe hard boundary when the only punctuation is too early.
- Confidence: high
