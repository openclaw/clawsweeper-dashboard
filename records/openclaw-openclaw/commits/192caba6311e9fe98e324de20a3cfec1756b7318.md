# Commit 192caba

## Summary

The fix reports invalid JSON and one narrow bad-message shape, but the new validator still accepts malformed transcript rows as valid session entries. Those rows can be counted/exported without warnings, and malformed assistant entries can break the generated HTML viewer.

## Findings

### Medium: Malformed session entries still pass the new warning parser

- Kind: bug
- File: `src/auto-reply/reply/commands-export-session.ts`
- Line: 167
- Evidence: `isSessionFileEntry` returns `true` for every non-`message` object with a string `type`, and for `message` rows it only checks `message.role`. The upstream `@earendil-works/pi-coding-agent@0.74.0` `FileEntry` contract requires session headers to include `id`, `timestamp`, and `cwd`, and session entries to include `id`, `parentId`, and `timestamp`; assistant messages also require array `content`. The export HTML renderer then assumes those fields exist, including `entry.id`/`entry.parentId` in `src/auto-reply/reply/export-html/template.js:40` and `src/auto-reply/reply/export-html/template.js:91`, and `for (const block of msg.content)` for assistant rows in `src/auto-reply/reply/export-html/template.js:1361`.
- Impact: A transcript row such as `{"type":"message","id":"a","parentId":null,"timestamp":"2026-05-16T00:00:00.000Z","message":{"role":"assistant"}}` is malformed, but `/export-session` accepts it with no skipped-row warning and writes it into the archive. If it is the active leaf, opening the export attempts to iterate `msg.content` and throws instead of rendering the session. Non-message rows like `{"type":"custom"}` are likewise silently counted/exported despite not satisfying the session-entry shape.
- Suggested fix: Replace the permissive guard with shape validation for `SessionHeader` and each supported `SessionEntry` type, at minimum enforcing shared `id`, `parentId`, and `timestamp` fields for non-header entries and required message payload fields by role before pushing rows into `fileEntries`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Put the malformed assistant row above into a targeted session transcript and run `/export-session`; the reply will not include an `invalid-session-row` warning, and the exported HTML has a deterministic renderer failure when that row is visible/current.
- Is this the best way to solve the issue? yes. The new parser is the right boundary for diagnostics, but it needs to validate the actual `FileEntry` contract before rows reach migration and HTML serialization.
