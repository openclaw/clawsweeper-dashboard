# Commit 5d64454

## Summary

Found one concrete regression in the doctor repair collector: enabled-only channel configs are no longer treated as configured, even though the runtime/plugin activation path still treats them as explicit channel config.

## Findings

### Medium: Doctor no longer repairs plugins for `channels.<id>.enabled: true`

- Kind: regression
- File: `src/commands/doctor/shared/missing-configured-plugin-install.ts`
- Line: 139
- Evidence: The commit changed `collectConfiguredChannelIds` to rely on `listPotentialConfiguredChannelIds`. That helper only adds config signals when the channel entry has a key other than `enabled` (`src/channels/config-presence.ts:36` and `src/channels/config-presence.ts:132`). The actual plugin activation path still treats `enabled === true` as explicit channel config (`src/plugins/channel-presence-policy.ts:92`) and includes those ids via `listExplicitConfiguredChannelIdsForConfig` (`src/plugins/effective-plugin-ids.ts:27`). A focused probe on current `main` returned `{"doctorChannelIds":[],"startupExplicitChannelIds":["matrix"]}` for `{ channels: { matrix: { enabled: true } } }`. The diff also changed the existing doctor test case from `matrix: { enabled: true }` to `matrix: { enabled: true, homeserver: ... }`, masking the regression.
- Impact: A user with an enabled-only channel config, which is a supported config shape used in setup/tests, can still have that channel considered by runtime/plugin activation but `doctor --fix` will not install or repair the missing downloadable channel plugin. After plugin externalization or a missing install record, doctor can report no repair while the channel remains unusable.
- Suggested fix: Make `collectConfiguredChannelIds` include explicit enabled channel configs using `listExplicitConfiguredChannelIdsForConfig` or equivalent `enabled === true || meaningful config` logic, while keeping the new disabled-channel filtering.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: run the focused helper probe above, or add a unit case for `channels: { matrix: { enabled: true } }` and expect `repairMissingConfiguredPluginInstalls` to install the catalog candidate.
- Is this the best way to solve the issue? yes: doctor should use the same explicit-channel semantics as startup/effective plugin id resolution, then layer the new disabled filtering on top.
