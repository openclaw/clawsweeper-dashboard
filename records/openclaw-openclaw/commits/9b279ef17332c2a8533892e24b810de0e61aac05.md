# Commit 9b279ef

## Summary

Found one concurrency regression in the new stale session lock reclamation path. The issue is still present on current `main`.

## Findings

### Medium: Stale-lock reclamation can delete a fresh holder's lock

- Kind: concurrency
- File: `src/agents/session-write-lock.ts`
- Line: 398
- Evidence: After `@openclaw/fs-safe` throws `file_lock_stale`, `acquireSessionWriteLock` re-reads the lock payload and then removes `params.lockPath` with `fs.rm(..., { force: true })`. The upstream lock manager explicitly fails closed here because a stale reclaimer cannot bind a path delete to the file it inspected; a concurrent release plus fresh acquire can otherwise lose its lock. I also ran a focused injected-race smoke check that replaced the stale file with a marker fresh lock immediately before this `fs.rm`; the fresh marker did not survive: `{"injected":true,"markerSurvived":false,"finalPid":7934}`.
- Impact: If process A decides a lock is stale, then process B removes the stale file and acquires a fresh lock before A reaches line 398, A can delete B's active `.lock` and loop into its own acquisition. That permits two writers into the same session transcript path, with plausible lost writes or transcript corruption.
- Suggested fix: Do not path-unlink from the `file_lock_stale` catch path unless the stale-reclaim protocol can prevent competing reclaimers and fresh acquirers from racing. Prefer moving safe reclamation into `@openclaw/fs-safe` with a serialized stale-reclaimer protocol, or fail closed and direct users through the existing doctor/fix cleanup path.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes; the temporary `tsx` smoke check monkey-patched `fs.rm` at this boundary to simulate a concurrent fresh acquire between stale reinspection and removal, and showed the fresh lock was deleted.
- Is this the best way to solve the issue? no; it bypasses the dependency's deliberate fail-closed behavior for stale locks and reintroduces the exact path-delete race that dependency warns about.
