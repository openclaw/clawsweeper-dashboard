# Commit ac2e3a23b9

## Summary

Found one actionable security issue in the new RTT Convex credential path: broker secrets are forwarded into the runtime container that executes the installed package under test.

## Findings

### Medium: Convex broker secrets are exposed to the package under RTT test

- Kind: security
- File: `scripts/e2e/npm-telegram-rtt-docker.sh`
- Line: 228
- Evidence: `OPENCLAW_QA_CONVEX_SECRET_CI` and `OPENCLAW_QA_CONVEX_SECRET_MAINTAINER` are appended to `docker_env` at `scripts/e2e/npm-telegram-rtt-docker.sh:228`, then the runtime container is launched with `"${docker_env[@]}"` at `scripts/e2e/npm-telegram-rtt-docker.sh:272`. Inside that same container, the installed `openclaw` package is executed via `openclaw --version` and `openclaw gateway run --verbose` at `scripts/e2e/npm-telegram-rtt-docker.sh:359` and `scripts/e2e/npm-telegram-rtt-docker.sh:381`.
- Impact: A compromised or buggy package candidate supplied through `OPENCLAW_NPM_TELEGRAM_PACKAGE_TGZ` can read the Convex broker secret from its process environment. If the maintainer secret is present, this grants admin-level credential-broker access, not just the leased Telegram credential needed for the RTT run. The commit already avoids passing these secrets to the install container via `install_env`, but the runtime package still receives them.
- Suggested fix: Keep broker secrets out of the SUT environment. Pass only the selected role secret to `npm-telegram-rtt-credentials.mjs` acquire/heartbeat/release, then unset Convex secrets before invoking `openclaw --version` or `openclaw gateway run`. If heartbeat/release need the token later, keep it in the helper process environment or pass a minimal env only for those helper calls.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. Build any `.tgz` with an `openclaw` bin that prints `process.env.OPENCLAW_QA_CONVEX_SECRET_MAINTAINER`, run the RTT wrapper in Convex mode with that package tarball, and the runtime `openclaw --version` path will receive the secret.
- Is this the best way to solve the issue? Yes. The credential helper needs broker auth, but the package under test only needs the leased Telegram values and generated config.
