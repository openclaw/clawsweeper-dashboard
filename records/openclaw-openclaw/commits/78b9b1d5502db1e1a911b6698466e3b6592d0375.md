The internal auth-profile success write consolidation looks covered by focused tests, but the commit still removes public SDK exports from `openclaw/plugin-sdk/agent-runtime`.

## Findings

### Medium: Public agent-runtime SDK exports were removed

- Kind: compatibility
- File: `src/plugin-sdk/agent-runtime.ts`
- Line: 39
- Evidence: `./plugin-sdk/agent-runtime` is a package-exported SDK subpath in `package.json:425`. Before this commit it re-exported `markAuthProfileGood` and `markAuthProfileUsed`; now it only exports `markAuthProfileSuccess` from the auth profile surface. The scoped SDK policy identifies `src/plugin-sdk` as the plugin/core public contract and says breaking removals or renames are major-version work. `pnpm plugin-sdk:api:check` also fails with `Plugin SDK API baseline drift detected`.
- Impact: Third-party plugins importing `markAuthProfileGood` or `markAuthProfileUsed` from `openclaw/plugin-sdk/agent-runtime` will fail to typecheck or load after upgrading, even though the runtime behavior change was only meant to reduce duplicate auth-store writes.
- Suggested fix: Keep `markAuthProfileSuccess` for new internal use, but preserve deprecated compatibility exports. `markAuthProfileGood` can wrap the new helper; `markAuthProfileUsed` should either keep the prior usage-stat implementation or get a compatibility wrapper that preserves its old `{ store, profileId, agentDir }` contract.
- Confidence: high

- Do we have a high-confidence way to reproduce the issue? yes. The package subpath is public, the removed symbols are visible in the base commit, and `pnpm plugin-sdk:api:check` fails on the current commit with SDK API baseline drift.
- Is this the best way to solve the issue? yes. The safe fix preserves source compatibility while still allowing the new single locked success write for OpenClaw’s own runtime path.
