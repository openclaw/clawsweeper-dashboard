# Commit 817ca4bf65

## Summary

Found one low-severity correctness issue in the new Android skills status UI path: agent-filtered skills can be shown as ready even though the gateway skill-status contract treats them as unavailable to the active/default agent.

## Findings

### Low: Android skills status ignores agent-filtered availability

- Kind: bug
- File: `apps/android/app/src/main/java/ai/openclaw/app/ui/SkillsSettingsScreen.kt`
- Line: 107
- Evidence: The Android skills UI considers a skill ready when `!skill.disabled && skill.eligible && skill.missingCount == 0`, and its parsed `GatewaySkillSummary` has no `blockedByAgentFilter`, `modelVisible`, or `commandVisible` field. The gateway contract separates raw eligibility from agent availability: `src/agents/skills-status.ts` sets `availableToAgent = eligible && !blockedByAgentFilter`, and existing tests assert an otherwise eligible filtered skill has `blockedByAgentFilter == true` with `modelVisible/commandVisible == false`. Current `src/gateway/server-methods/skills.ts` also resolves an `agentId` for `skills.status` but does not pass it into `buildWorkspaceSkillStatus`, so the mobile request path has no reliable way to reflect the default agent filter.
- Impact: On Android, a skill excluded by the active/default agent’s skill filter can appear as “Ready” in the Skills settings screen, even though the agent will not see or invoke it. That can send users toward debugging missing runtime dependencies when the real reason is agent policy/filtering.
- Suggested fix: Carry agent-scoped availability through the mobile contract. At minimum, pass the resolved `agentId` into `buildWorkspaceSkillStatus` in `skills.status`, parse `blockedByAgentFilter` on Android, and treat filtered skills separately from ready/setup states. Prefer using `modelVisible`/`commandVisible` if the UI is meant to answer “available to this agent”.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes, at the unit-contract level: configure an agent with a skill filter that includes `alpha` but not `beta`; `buildWorkspaceSkillStatus(..., { agentId })` marks `beta` as `blockedByAgentFilter` while leaving it otherwise eligible, and the Android UI currently has no field/check for that state.
- Is this the best way to solve the issue? yes: the fix should preserve the gateway’s existing distinction between requirement eligibility and agent-visible availability, then render that distinction on Android instead of folding everything into “Ready” or “Setup”.
