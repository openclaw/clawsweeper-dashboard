## Findings

### Medium: Resumed threads can keep stale direct/searchable tool presentation

- Kind: regression
- File: `extensions/codex/src/app-server/thread-lifecycle.ts`
- Line: 845
- Evidence: The commit now ignores `namespace` and `deferLoading` in the durable dynamic-tool fingerprint, so a direct tool and searchable/deferred tool compare equal. But `thread/start` sends `dynamicTools`, while `thread/resume` does not send any updated `dynamicTools`; it only sends developer instructions. `run-attempt.ts` still makes `message` direct only when `sourceReplyDeliveryMode === "message_tool_only"`, and passes `toolBridge.specs` as the thread schema. A normal turn can therefore start a thread with `message` registered as searchable, then a later `message_tool_only` turn can resume that same thread without updating Codex’s registered tool presentation.
- Impact: `message_tool_only` turns depend on the visible reply going through the `message` tool while automatic source delivery is suppressed. If the existing Codex thread still has `message` only as deferred/searchable, the current developer instructions can say to “use the `message` tool” without the app-server actually exposing it as a direct tool for that turn. This can make visible replies unreliable for sessions whose first turn used searchable presentation and a later turn requires direct `message`.
- Suggested fix: Do not treat `namespace`/`deferLoading` as fingerprint-compatible unless `thread/resume` can update the dynamic tool registry. Either keep presentation in the durable fingerprint for tools whose direct availability matters, or make the durable registered presentation stable in a way that satisfies all resumed turn modes.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Unclear. The source path is concrete: start a Codex app-server thread on a normal turn where `message` is deferred/searchable, then resume it with `sourceReplyDeliveryMode = "message_tool_only"` and assert the resume request carries no `dynamicTools` update even though the current turn expects direct `message`.
- Is this the best way to solve the issue? Unclear. The safest fix depends on the Codex app-server contract: if dynamic tool presentation is only set on `thread/start`, the fingerprint should keep presentation-sensitive fields or the initial durable schema must choose a presentation that remains valid for later turns.
