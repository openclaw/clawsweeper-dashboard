# Commit d253392

## Summary

Found one likely regression in the new explicit bundled web-provider fast path: it now filters scoped provider ids against the raw config before plugin auto-enable has a chance to widen restrictive allowlists for an explicitly selected web provider.

## Findings

### Medium: Explicit web providers can be filtered out before auto-enable widens `plugins.allow`

- Kind: regression
- File: `src/plugins/web-provider-public-artifacts.ts`
- Line: 57
- Evidence: the new `onlyPluginIds !== undefined` branch returns before `resolveBundledWebSearchResolutionConfig` / `resolveBundledWebFetchResolutionConfig` runs, and filters with `params.config` directly at line 59. The existing auto-enable contract says `tools.web.search.provider: "brave"` should add `brave` to a restrictive `plugins.allow: ["telegram"]` config (`src/config/plugin-auto-enable.providers.test.ts:76`). A focused helper smoke check on current `main` returned `[]` for `resolveBundledWebSearchProvidersFromPublicArtifacts({ config: { tools.web.search.provider: "brave", plugins.allow: ["telegram"] }, bundledAllowlistCompat: true, onlyPluginIds: ["brave"] })`.
- Impact: setup/control-plane paths that use the public artifact helper with `onlyPluginIds` can drop the explicitly selected bundled provider before falling back. In `mode: "setup"`, `resolvePluginWebProviders` returns any non-null artifact result directly (`src/plugins/web-provider-runtime-shared.ts:170`), and search setup uses that mode (`src/flows/search-setup.ts:101`), so an explicitly configured web provider can disappear from setup/provider surfaces under restrictive allowlists.
- Suggested fix: keep the explicit fast path, but run the same activation/auto-enable allowlist normalization before `filterAllowlistedBundledPluginIds`, or otherwise explicitly preserve the provider ids that are auto-enable eligible for the configured web provider before returning `[]`.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Partly: yes for the changed helper returning `[]`; the full setup-mode smoke was blocked by an ad hoc `tsx` package-export resolution issue, but the source path from setup mode to the helper is direct.
- Is this the best way to solve the issue? Unclear; the safest direction is to preserve the new no-manifest-scan goal while restoring the prior activation-aware allowlist behavior for scoped provider ids.
