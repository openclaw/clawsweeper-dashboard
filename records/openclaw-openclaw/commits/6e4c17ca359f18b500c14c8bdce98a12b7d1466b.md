# Commit 6e4c17c

## Summary

Found one compatibility regression in the screenshot path handling for documented custom or pinned Chrome DevTools MCP launches.

## Findings

### Medium: Existing-session screenshots now fail with pinned older Chrome MCP builds

- Kind: compatibility
- File: `extensions/browser/src/browser/chrome-mcp.ts:923`
- Evidence: `takeChromeMcpScreenshot` passes the bare temporary `filePath` to `take_screenshot`, then unconditionally reads `${filePath}.${format}`. That matches current `chrome-devtools-mcp@0.25.0`, whose `saveFile` applies `ensureExtension(..., ".<format>")`, but OpenClaw documents `mcpCommand`/`mcpArgs` for offline hosts, pinned versions, and vendored binaries in `docs/tools/browser.md:623`. Inspecting `chrome-devtools-mcp@0.20.0` package source shows `saveFile(data, filename)` wrote exactly `path.resolve(filename)` without adding the format extension, so this commit makes those supported pinned/custom launches write the bare temp path while OpenClaw reads the suffixed path.
- Impact: Existing-session `browser screenshot` and labeled snapshot screenshot flows can regress to `ENOENT` for users who pinned or vendored an older Chrome DevTools MCP, even though the MCP call succeeded and wrote the image.
- Suggested fix: Build the suffixed output path before the tool call, pass that exact path as `filePath`, and read the same path. Current Chrome MCP’s extension-normalizing implementation keeps an already-matching `.png`/`.jpeg` path unchanged, while older builds write the exact provided path.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused in-memory Chrome MCP session that writes screenshot bytes to the exact requested `filePath`, matching `chrome-devtools-mcp@0.20.0`, makes current `takeChromeMcpScreenshot({ format: "png" })` fail with `ENOENT`.
- Is this the best way to solve the issue? yes. Passing the final path into the dependency avoids probing multiple filenames and remains compatible with the current dependency behavior.
