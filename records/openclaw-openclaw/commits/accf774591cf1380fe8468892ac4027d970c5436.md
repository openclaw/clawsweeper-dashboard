# Commit accf774

## Summary

Found one low-severity Slack rich progress rendering regression. Text-mode progress drafts roll the label away correctly, but rich Block Kit progress drafts usually keep the label forever.

## Findings

### Low: Slack rich progress labels do not roll away under the configured progress-line cap

- Kind: regression
- File: `extensions/slack/src/progress-blocks.ts`
- Line: 50
- Evidence: `buildSlackProgressDraftBlocks` now combines the optional label block with all passed progress-line blocks and slices only to Slack's `SLACK_MAX_BLOCKS` limit. The Slack dispatcher already caps `previewToolProgressLines` to `resolveChannelProgressDraftMaxLines(account.config)` before calling this helper (`extensions/slack/src/monitor/message-handler/dispatch.ts:1035` and `:972`). With the default `maxLines: 8`, the helper receives at most 8 rows plus the label, so the label remains visible as a ninth Block Kit block even after the text formatter has rolled it off. A focused local probe with 8 progress rows produced text without `Shelling`, but `buildSlackProgressDraftBlocks({ label: "Shelling", lines })` returned 9 blocks and block 0 was the `Shelling` label.
- Impact: Users who enable Slack `streaming.progress.render: "rich"` still see the starter/status label pinned at the top for ordinary/default progress drafts, contrary to this commit's changelog/docs claim that labels scroll away with progress lines. This is visual behavior only; final delivery is not broken.
- Suggested fix: Apply the configured progress-line budget to the rich block payload, not only the Slack 50-block transport limit. For example, combine the optional label and progress rows at the dispatcher or helper level, slice by `resolveChannelProgressDraftMaxLines(account.config)`, then map to blocks and finally cap to `SLACK_MAX_BLOCKS`. Add a Slack rich-mode test with `maxLines: 8` and 8 progress rows.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: configure Slack progress drafts with `streaming.mode: "progress"` and `streaming.progress.render: "rich"` using the default or explicit `maxLines: 8`, then emit 8 distinct progress rows. The text fallback rolls off the label, while the visual Block Kit payload still starts with the label.
- Is this the best way to solve the issue? yes: the bug is an ordering/budget mismatch between the shared formatter and Slack rich block construction, so the rich block path should use the same rolling window semantics before rendering blocks.
