## Findings

### Medium: Reported manual update jobs are not disarmed by the new label filter

- Kind: bug
- File: `src/daemon/launchd.ts`
- Line: 48
- Evidence: `CHANGELOG.md:16` says this fixes #82167, whose issue body identifies the relaunching submitted job as `ai.openclaw.manual-update-2026-05-14-2026-5-12`. The new helper only accepts labels starting with `OPENCLAW_UPDATE_LAUNCHD_LABEL_PREFIX = "ai.openclaw.update."`, and `disableCurrentOpenClawUpdateLaunchdJob()` returns `false` unless one of the env labels passes that prefix filter.
- Impact: If the same reported submitted manual update job runs `openclaw update`, this patch will not disable it before the gateway stop/update flow, so the KeepAlive relaunch loop can continue for that exact incident class.
- Suggested fix: Include the known `ai.openclaw.manual-update-` submitted-job label shape in the updater-job classifier, with the same gateway-label guard and regression coverage. If current launchd env does not reliably expose submitted labels, add a macOS-specific fallback that identifies the current launchd job rather than relying only on env.
- Confidence: medium
