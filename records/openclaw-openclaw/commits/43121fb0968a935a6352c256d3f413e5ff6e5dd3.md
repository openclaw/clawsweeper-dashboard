# Commit 43121fb

## Summary

Found one remaining provider-prefix coercion edge in the WhatsApp target normalizer. The main core guard covers loaded plugin prefixes, but the WhatsApp local fallback guard added in this commit misses underscored prefixes.

## Findings

### Medium: WhatsApp still accepts underscored provider-prefixed numeric targets

- Kind: privacy
- File: `extensions/whatsapp/src/normalize-target.ts`
- Line: 7
- Evidence: `NON_WHATSAPP_PROVIDER_PREFIX_RE` is `/^[a-z][a-z0-9-]*:/i`, so it rejects `telegram:1234567890` and `synology-chat:1234567890`, but not `synology_chat:1234567890`. The focused live check showed `normalizeWhatsAppTarget("synology_chat:1234567890")` returns `+1234567890` and `looksLikeWhatsAppTargetId(...)` returns `true`.
- Impact: this leaves the same class of bug described by #56839 for provider aliases containing `_`: a Synology Chat numeric target can still be coerced into a WhatsApp phone number when the shared loaded-plugin prefix guard is unavailable or bypassed and the WhatsApp adapter normalizes the target directly.
- Suggested fix: use the same provider-prefix character class as `src/infra/outbound/channel-target-prefix.ts`, for example allow `_` in the WhatsApp non-provider guard, and add a regression assertion for `synology_chat:<digits>` in `extensions/whatsapp/src/resolve-target.test.ts` or `extensions/whatsapp/src/channel-outbound.test.ts`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: `pnpm exec tsx -e "import { normalizeWhatsAppTarget, looksLikeWhatsAppTargetId } from './extensions/whatsapp/src/normalize-target.ts'; console.log(normalizeWhatsAppTarget('synology_chat:1234567890'), looksLikeWhatsAppTargetId('synology_chat:1234567890'))"` prints `+1234567890 true`.
- Is this the best way to solve the issue? yes: aligning the WhatsApp regex with the core provider-prefix grammar is narrow and preserves the existing `whatsapp:` stripping behavior.
