# Commit bb07cbc2

## Summary

Found one concrete sanitizer regression: malformed persisted plugin extension keys can become inherited properties during load and then be projected as real plugin session state.

## Findings

### Medium: Sanitizing `__proto__` plugin extension keys can inject inherited plugin state

- Kind: security
- File: `src/config/sessions/store-load.ts`
- Line: 159
- Evidence: `normalizePluginExtensions` rebuilds `normalizedExtensions` and `normalizedPluginState` as plain `{}` objects, then writes attacker-controlled persisted keys at lines 179 and 185. A session row containing a namespace key `__proto__` plus any sibling that causes `changed = true` makes `normalizedPluginState.__proto__` point at attacker JSON. `projectPluginSessionExtensionsSync` later reads with `params.entry.pluginExtensions?.[registration.pluginId]?.[registration.extension.namespace]` in `src/plugins/host-hook-state.ts:676`, so inherited properties are treated as valid extension state. This reaches gateway session listing because `sessions.list` calls `loadCombinedSessionStoreForGateway`, which loads stores with `{ clone: false }` in `src/config/sessions/combined-store-gateway.ts:72` and `src/config/sessions/combined-store-gateway.ts:104`.
- Impact: A corrupted `sessions.json` can cause arbitrary JSON under a `__proto__` key to be exposed as another registered plugin’s projected session extension state during the first clone-false hydration path. That undercuts the commit’s hardening goal of stripping poisoned plugin metadata before replay/projection paths.
- Suggested fix: Build normalized plugin records with `Object.create(null)` or define properties with `Object.defineProperty`, and reject/skip prototype-sensitive keys such as `__proto__`, `constructor`, and `prototype` for plugin ids and namespaces. Also consider own-property checks in projection reads.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A no-repo-write `node --import tsx` probe wrote a temporary `sessions.json` with `pluginExtensions["victim-plugin"]["__proto__"] = { workflow: { state: "polluted" } }` plus `" trim-me "` to force normalization. Loading with `{ skipCache: true, clone: false }` produced own namespaces `["trim-me"]`, but `entry.pluginExtensions["victim-plugin"].workflow` resolved from the prototype and `projectPluginSessionExtensionsSync` emitted `{ pluginId: "victim-plugin", namespace: "workflow", value: { state: "polluted" } }`.
- Is this the best way to solve the issue? yes. Null-prototype accumulators and/or explicit reserved-key rejection keeps JSON map semantics without letting magic object keys change prototypes.
