# Commit 364ec53

## Summary

Found one regression in the installer Docker E2E OpenAI profile: it now configures `agents.defaults.imageModel` to an OpenRouter image-generation shortcut under the direct `openai/` provider, so the later `image` tool smoke can fail before it reaches the live API.

## Findings

### Medium: OpenAI installer E2E configures a text-only image model

- Kind: regression
- File: `scripts/docker/install-sh-e2e/run.sh`
- Line: 610
- Evidence: The OpenAI branch now runs `set_image_model "$profile" "openai/gpt-5.4-image-2"`, then the same profile requires the `image` tool to inspect `proof.png` and write `LEFT=RED RIGHT=GREEN` at `scripts/docker/install-sh-e2e/run.sh:756`. In current source, direct OpenAI media understanding defaults to `gpt-5.4-mini` (`extensions/openai/media-understanding-provider.ts:24`), the OpenAI plugin marks `gpt-5.5` and `gpt-5.4-mini` as `["text", "image"]`, and `openai/gpt-5.4-image-2` appears as an OpenRouter image-generation shortcut, not a direct OpenAI media-understanding model (`extensions/openrouter/image-generation-provider.ts:27`). A local model-resolution probe for `openai/gpt-5.4-image-2` with an OpenAI provider config resolved it with `input:["text"]`; `src/media-understanding/image.ts:191` rejects such models with `Model does not support images`.
- Impact: `OPENCLAW_E2E_MODELS=openai` or `both` can fail the official installer Docker E2E at the image-tool turn even though the text agent model was configured correctly, blocking release/package validation lanes.
- Suggested fix: Set the image-understanding model to a model OpenClaw knows supports image input, such as `openai/gpt-5.5` or `openai/gpt-5.4-mini`. If the intent was image generation, use `agents.defaults.imageGenerationModel` with the OpenAI image provider model instead; this script is exercising the `image` analysis tool, not `image_generate`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. The local probe returned `{"id":"gpt-5.4-image-2","input":["text"]}` for the configured direct OpenAI provider path, and the `image` tool rejects text-only models before inference.
- Is this the best way to solve the issue? Yes. The failing setting is specifically for `imageModel`, so it should use a vision-capable mainline OpenAI model, not an OpenRouter image-generation shortcut. Official OpenAI image-generation docs also distinguish GPT Image models from the mainline `model` field and list supported mainline models separately.
