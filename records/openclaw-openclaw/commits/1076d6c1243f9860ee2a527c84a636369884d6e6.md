# Commit 1076d6c

## Summary

Found one actionable issue: the Discord text-DM owner pin is wired to guild/channel user allowlists, not the configured Discord DM owner allowlist, so the intended route-hijack protection is ineffective for real text DMs.

## Findings

### Medium: Discord text DM owner pin reads the wrong allowlist

- Kind: security
- File: `extensions/discord/src/monitor/message-handler.context.ts`
- Line: 117
- Evidence: `pinnedMainDmOwner` is computed from `channelConfig?.users ?? guildInfo?.users`. Production DM preflight sets `channelConfig` to `null` for non-guild messages in `extensions/discord/src/monitor/message-handler.preflight-channel-context.ts:31`, and `guildInfo` is also `null` for DMs. The actual configured DM allowlist is passed into the message handler as `allowFrom` from provider startup (`extensions/discord/src/monitor/provider.ts:563`) and exists only on `DiscordMessagePreflightParams`, not on `DiscordMessagePreflightContext` (`extensions/discord/src/monitor/message-handler.preflight.types.ts:96`). The new regression test passes by manually injecting `channelConfig: { users: ["user:111"] }` into a direct-message context (`extensions/discord/src/monitor/message-handler.process.test.ts:823`), which does not match the real preflight path.
- Impact: For `session.dmScope=main`, an allowed/bound non-owner Discord DM can still update the main session `lastRoute` to itself, so future main-session deliveries can be redirected away from the configured owner. That is the exact privacy/security behavior the changelog says this commit fixes.
- Suggested fix: Carry the resolved account DM `allowFrom` onto `DiscordMessagePreflightContext` or compute the pin before dropping preflight-only fields, then pass that list to `resolvePinnedMainDmOwnerFromAllowlist`. Reuse the Discord DM security normalizer so documented forms like `discord:<@!123>` normalize consistently.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. Follow the production path for a DM: `resolveDiscordPreflightChannelContext` returns `channelConfig: null` whenever `isGuildMessage` is false, while `buildDiscordMessageProcessContext` only looks at `channelConfig?.users ?? guildInfo?.users`; therefore `pinnedMainDmOwner` is null and `mainDmOwnerPin` is omitted.
- Is this the best way to solve the issue? Yes: the pin must come from the account-level Discord DM owner allowlist, matching Slack/Signal/Line/Mattermost patterns that pass account `allowFrom` into `resolvePinnedMainDmOwnerFromAllowlist`.
