# Commit 47de32a

## Summary

The commit removes the broad `detect-secrets` hook and baseline, leaving repository secret scanning limited to private-key marker detection. That is a concrete security guardrail regression for common committed tokens such as GitHub tokens.

## Findings

### Medium: Secret scanning no longer catches API tokens

- Kind: security
- File: `.pre-commit-config.yaml`
- Line: 21
- Evidence: Current config keeps only `detect-private-key` at `.pre-commit-config.yaml:21`, and CI runs exactly `pre-commit ... detect-private-key` at `.github/workflows/ci.yml:364`. The removed parent config had `Yelp/detect-secrets` with `GitHubTokenDetector`, `OpenAIDetector`, `SlackDetector`, `TelegramBotTokenDetector`, high-entropy detectors, and other token detectors. A focused dependency check against `pre-commit-hooks==6.0.0` showed `detect-private-key` only scans for private-key header byte strings like `BEGIN RSA PRIVATE KEY`; it exited 0 for a sample file containing a `ghp_...` token. The old `detect-secrets-hook==1.5.0` with the parent baseline exited 1 on the same sample and reported `Secret Type: GitHub Token`.
- Impact: A developer or CI path using the repo's documented security hook can now accept newly committed service tokens that are not private keys. That is especially risky in this repo because workflows and docs contain many provider token names and release/live-test credential surfaces.
- Suggested fix: Restore a broad token scanner, either by regenerating/auditing a current `detect-secrets` baseline or replacing it with an equivalent changed-file/all-file scanner such as detect-secrets, gitleaks, or trufflehog. Keep `detect-private-key` as a narrow extra check, not the only secret scanner.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Run the remaining `detect-private-key` hook against a sample file containing a GitHub-style token; it passes. Run the removed `detect-secrets-hook` with the parent baseline against the same file; it fails with `GitHub Token`.
- Is this the best way to solve the issue? yes, directionally. The exact scanner can change, but the fix needs broad token detection, not just private-key header detection.
