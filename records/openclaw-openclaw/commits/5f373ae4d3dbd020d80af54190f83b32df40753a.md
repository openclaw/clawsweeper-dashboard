# Commit 5f373ae

## Summary

Found one low-severity TUI abort edge case in the new pending-run tracking.

## Findings

### Low: Pending abort can target the wrong run after duplicate in-flight sends

- Kind: bug
- File: `src/tui/tui-command-handlers.ts`
- Line: 645
- Evidence: `sendMessage` passes a locally generated `runId` into `client.sendChat`, ignores the returned `{ runId }`, then stores the local ID in `state.pendingChatRunId`. The gateway can legally return a different active run ID for duplicate in-flight internal text sends (`src/gateway/server-methods/chat.ts:2071`), and the existing gateway test proves that duplicate request `idem-active-b` receives active run `idem-active-a` (`src/gateway/server.chat.gateway-server-chat-b.test.ts:417`). `GatewayChatClient.sendChat` also discards the RPC payload and always returns the local idempotency key (`src/tui/gateway-chat.ts:181`).
- Impact: During the same pre-event waiting gap this commit is trying to fix, a user can submit the same text again before the first chat event arrives. The second `chat.send` may be deduped to the first run, but TUI overwrites `pendingChatRunId` with the second, nonexistent ID. Pressing Esc then sends `chat.abort` for the wrong run, so the real run continues while the UI reports `aborted`.
- Suggested fix: Return the actual RPC `runId` from `GatewayChatClient.sendChat`, store that returned ID in `pendingChatRunId`, and add a regression test where `sendChat` resolves a different `runId` than the one passed in. Also consider making `abortActive` check `aborted` before showing aborted status.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: start a TUI send, submit the same text again before the first chat event arrives, then press Esc before any event binds `activeChatRunId`.
- Is this the best way to solve the issue? yes: the gateway already defines the response `runId` as authoritative for duplicate in-flight sends, so the TUI should track that value.
