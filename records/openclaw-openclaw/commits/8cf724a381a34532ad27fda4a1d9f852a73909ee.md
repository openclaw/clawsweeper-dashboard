# Commit 8cf724a

## Summary

Found two actionable issues in the bundled runtime-deps staging change. One is directly proven by a failing focused test added in this commit.

## Findings

### Medium: Bundled channel discovery still loads through staged runtime deps

- Kind: regression
- File: `src/channels/plugins/bundled.ts`
- Line: 554
- Evidence: `getBundledChannelPluginForRoot()` still calls `entry.loadChannelPlugin()` without `{ installRuntimeDeps: false }`. The entry helper defaults to staging runtime deps for built bundled roots at `src/plugin-sdk/channel-entry-contract.ts:355`. The new guard test at `src/channels/plugins/bundled.shape-guard.test.ts:736` fails: `getBundledChannelPlugin("alpha")` returns a plugin loaded from staged `alpha-runtime-dep` instead of `undefined`.
- Impact: control-plane/discovery callers such as `listBundledChannelPlugins()`, setup/security/doctor helpers, and channel registry fallbacks can still materialize staged runtime deps and execute heavy runtime entry code while trying to inspect bundled channel metadata. That preserves the startup/setup laziness regression this commit is trying to remove.
- Suggested fix: pass `{ installRuntimeDeps: false }` through `entry.loadChannelPlugin(...)` for discovery/control-plane bundled channel access, or split full runtime loading from metadata/setup plugin resolution so activation paths can still opt into installs explicitly.
- Confidence: high

### Medium: Release check does not enforce the new explicit mirrored-root metadata contract

- Kind: test_gap
- File: `scripts/lib/bundled-plugin-root-runtime-mirrors.mjs`
- Line: 202
- Evidence: runtime collection now only installs root mirrored deps named by `openclaw.bundle.mirroredRootRuntimeDependencies` (`src/plugins/bundled-runtime-deps.ts:546`, used at `src/plugins/bundled-runtime-deps.ts:1500`). But the release-check helper still accepts any required root mirror if it exists in root `dependencies`/`optionalDependencies` (`scripts/lib/bundled-plugin-root-runtime-mirrors.mjs:202`), and the new metadata validator only checks that declared metadata entries are backed by root deps (`scripts/lib/bundled-plugin-root-runtime-mirrors.mjs:217`). A focused Node probe with a required mirror present in root deps but absent from `mirroredRootRuntimeDependencies` returned no errors from both validators.
- Impact: a future root dist chunk can import a bundled plugin runtime dependency, pass release validation because the package is in root deps, but fail in packaged runtime because the explicit metadata omitted it and startup no longer infers root mirrors.
- Suggested fix: make `collectBundledPluginRootRuntimeMirrorErrors()` also require every `requiredRootMirrors` dependency to be present in `openclaw.bundle.mirroredRootRuntimeDependencies`, and add a release-check test for the omitted-metadata case.
- Confidence: high
