# Commit ea75cd8

## Summary

Found one concrete regression in the ZaloUser startup gate: it also disables stable group-id alias normalization, not just mutable display-name lookup.

## Findings

### Medium: Stable `zlu:<numeric>` group allowlist aliases no longer normalize unless dangerous name matching is enabled

- Kind: regression
- File: `extensions/zalouser/src/monitor.ts`
- Line: 873
- Evidence: Before this commit, startup always entered the group normalization block, stripped `zalouser:`/`zlu:` with `normalizeZalouserEntry()`, and copied numeric cleaned keys into `groups` without requiring a name match. After the new `if (allowNameMatching && groupKeys.length > 0)`, that stable-id canonicalization is skipped by default. Current runtime candidates for a group id only include the raw id and `group:<id>` when name matching is disabled, so a config key like `zlu:123` is not matched. The security audit treats `zlu:123` as non-mutable/stable by stripping `zlu:` before the numeric check in `extensions/zalouser/src/security-audit.ts`.
- Impact: Users with a stable provider-prefixed numeric Zalo group allowlist can have group messages dropped after upgrade unless they enable `dangerouslyAllowNameMatching`, which is broader and less safe than needed for stable ID aliases.
- Suggested fix: Keep network friend/group display-name resolution behind `dangerouslyAllowNameMatching`, but normalize stable group-id aliases independently. Either canonicalize `zlu:<numeric>`/`zalouser:<numeric>` into raw numeric group keys before the gated lookup, or make group candidate matching understand those stable aliases without requiring mutable name matching.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes, at helper/contract level: `pnpm exec tsx -e ...` showed `isZalouserMutableGroupEntry("zlu:123") === false`, runtime id-only candidates are `["123","group:123"]`, and `findZalouserGroupEntry({"zlu:123": ...}, candidates)` does not match.
- Is this the best way to solve the issue? yes: separating stable alias canonicalization from mutable display-name lookup preserves the security intent of the commit without forcing operators onto `dangerouslyAllowNameMatching`.
