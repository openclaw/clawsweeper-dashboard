# Commit 01bd2f2

## Summary

Found one high-confidence regression: the new active-registry shortcut can return an empty/incompatible active registry and skip the fallback plugin load, disabling web-search/web-fetch provider discovery in existing covered paths.

## Findings

### High: Incompatible active registries now suppress web provider loading

- Kind: regression
- File: `src/plugins/web-provider-runtime-shared.ts`
- Line: 189
- Evidence: After `resolveCompatibleRuntimePluginRegistry(loadOptions)` rejects the active registry, the new code still calls `getActivePluginRegistry()` and returns `deps.mapRegistryProviders({ registry: activeRegistry, onlyPluginIds: context.onlyPluginIds })` for any truthy active registry. That bypasses `loadOpenClawPlugins(loadOptions)` even when the active registry is empty, stale, or scoped differently from the derived web-provider candidates.
- Impact: `resolvePluginWebSearchProviders({})` and `resolvePluginWebFetchProviders({})` can return `[]` instead of loading bundled providers. The focused provider tests reproduce this directly: `src/plugins/web-search-providers.runtime.test.ts:373` expected bundled web-search providers but received `[]`, and `src/plugins/web-fetch-providers.runtime.test.ts:128` expected `firecrawl:firecrawl` but received `[]`. This can make web_search/web_fetch unavailable whenever an active registry exists but does not already contain the requested web provider registrations.
- Suggested fix: Do not return an arbitrary active registry after the compatibility check fails. Either keep the old loader fallback unless the active registry is proven to cover the derived candidate scope, or map the active registry only as an optimization and fall back to `loadOpenClawPlugins(loadOptions)` when the requested non-empty candidate scope produces no active providers. Preserve the explicit empty-scope case.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. After installing missing dependencies, `pnpm test src/plugins/web-provider-runtime-shared.test.ts src/plugins/web-search-providers.runtime.test.ts src/plugins/web-fetch-providers.runtime.test.ts` failed with 11 provider-resolution regressions.
- Is this the best way to solve the issue? unclear. The performance goal from #75513 is valid, but the shortcut needs a coverage check so it only reuses the active registry when that registry actually contains the requested provider candidates.
