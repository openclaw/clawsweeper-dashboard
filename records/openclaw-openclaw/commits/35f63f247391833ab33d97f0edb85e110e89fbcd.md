# Commit 35f63f2

## Summary

Found one regression in the new typed error-frame parsing. The helper flattens top-level gateway error fields into the same dictionary used for connect-auth detail fields, overwriting the nested auth/pairing detail code.

## Findings

### Medium: Connect auth detail codes are replaced by generic gateway error codes

- Kind: regression
- File: `apps/shared/OpenClawKit/Sources/OpenClawKit/GatewayChannel.swift`
- Line: 133
- Evidence: `gatewayErrorDetails` first merges `error.details`, then overwrites `details["code"]` with the top-level `ErrorShape.code`. `handleConnectResponse` later reads `details["code"]` as `detailCodeRaw` for `GatewayConnectAuthError`. Current server connect-auth failures send top-level protocol codes like `INVALID_REQUEST` / `NOT_PAIRED` while the actionable auth detail remains nested at `error.details.code`, for example `AUTH_TOKEN_MISSING`, `AUTH_TOKEN_MISMATCH`, `DEVICE_IDENTITY_REQUIRED`, or `PAIRING_REQUIRED` in `src/gateway/server/ws-connection/message-handler.ts:647` and `src/gateway/server/ws-connection/message-handler.ts:1150`.
- Impact: macOS/iOS clients no longer classify connect failures through `GatewayConnectAuthDetailCode`. That degrades recovery UI in `GatewayConnectionProblem`, and non-recoverable auth failures can stop matching `authError.isNonRecoverable`, so reconnect behavior and pairing/auth remediation can become generic or noisy instead of pausing with the correct action.
- Suggested fix: keep the nested detail `code` for connect-auth details. Either do not overwrite an existing nested `details["code"]`, or split top-level gateway metadata into separate keys such as `gatewayCode` / `gatewayMessage` and have `handleConnectResponse` read `error.details.code` before adding top-level fields.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. A typed connect response with `error: { code: "INVALID_REQUEST", message: "...", details: { code: "AUTH_TOKEN_MISSING" } }` will produce `GatewayConnectAuthError.detailCodeRaw == "INVALID_REQUEST"` because of the overwrite at `GatewayChannel.swift:133`, then the switch in `GatewayConnectionProblem.swift:185` falls through to the unknown case.
- Is this the best way to solve the issue? Yes. The protocol already separates top-level gateway error codes from nested connect-auth detail codes; preserving that separation in Swift is the narrow fix.
