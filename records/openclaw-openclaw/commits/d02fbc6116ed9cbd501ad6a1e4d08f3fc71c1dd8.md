# Commit d02fbc6

## Summary

Found one sandbox host-path policy regression in the new Windows drive-letter support: Windows drive roots are not treated like POSIX `/`, and `..` traversal above a drive root can normalize to a relative-looking path that bypasses the blocked-root check when external bind sources are explicitly allowed.

## Findings

### Medium: Windows drive roots can be bind-mounted despite root-mount blocking

- Kind: security
- File: `src/agents/sandbox/validate-sandbox-security.ts`
- Line: 143
- Evidence: `getBlockedReasonForSourcePath` only treats `sourceNormalized === "/"` as a root-covering bind. The new Windows normalization in `src/agents/sandbox/host-paths.ts:44` uses `path.posix.normalize`, so `C:/` stays `C:/` and is not blocked, while `C:/../` normalizes to `.` and also avoids the blocked-root check. Focused probe on current `main`: `validateBindMounts(["C:/:/host:ro"], { allowSourcesOutsideAllowedRoots: true })`, `validateBindMounts(["C:/../:/host:ro"], { allowSourcesOutsideAllowedRoots: true })`, and `validateBindMounts(["C:/Windows/../:/host:ro"], { allowSourcesOutsideAllowedRoots: true })` all returned allowed.
- Impact: `dangerouslyAllowExternalBindSources` is meant to bypass the allowed-root fence, but blocked host roots still remain blocked for POSIX `/`. After this commit, the Windows equivalent can expose an entire drive to the sandbox container.
- Suggested fix: normalize drive-letter paths with `path.win32.normalize` semantics, convert back to `/`, preserve the drive root when `..` climbs above it, and make `getBlockedReasonForSourcePath` return `covers` for drive roots like `C:/`. Add tests for `C:/:/host:ro`, `C:/../:/host:ro`, and audit/runtime behavior with `dangerouslyAllowExternalBindSources`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The direct validation probe above reproduces the policy bypass without Docker. A live Windows Docker run was not performed.
- Is this the best way to solve the issue? yes, directionally. The fix should be in the shared host-path policy helper and blocked-root predicate so config/runtime/audit callers stay aligned.
