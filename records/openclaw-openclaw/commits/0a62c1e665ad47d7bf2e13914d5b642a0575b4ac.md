# Commit 0a62c1e

## Summary

Found one concrete Telegram sticker regression introduced by the new saved-MIME placeholder override.

## Findings

### Medium: Cached sticker descriptions are replaced with `<media:image>` and the media is then removed

- Kind: regression
- File: `extensions/telegram/src/bot-message-context.body.ts`
- Line: 290
- Evidence: For a cached static sticker, lines 207-216 replace the primary `<media:sticker>` placeholder with the cached `[Sticker ...] description` and set `stickerCacheHit`. The new block at lines 289-291 then sees saved `image/webp` media and overwrites that description with `<media:image>`. Later, `extensions/telegram/src/bot-message-context.session.ts:332` removes sticker media from context when `stickerCacheHit` is true, so the agent receives an image placeholder with no media instead of the cached sticker description. The docs still state static WEBP stickers use placeholder `<media:sticker>` in `docs/channels/telegram.md:636`.
- Impact: Repeated/cached Telegram sticker messages can regress to the same class of contradictory prompt as the fixed photo bug: the model sees `<media:image>` without an attached image, losing the sticker description and sticker semantics.
- Suggested fix: Do not apply `formatSavedMediaPlaceholder()` to sticker messages or to a placeholder that has already been replaced with a cached sticker description. A narrow guard on `allMedia.some((m) => m.stickerMetadata)` or on the original primary media placeholder would preserve existing sticker behavior while keeping the photo/document fix.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Add or unskip a focused sticker case with `msg.sticker`, `allMedia: [{ contentType: "image/webp", stickerMetadata: { cachedDescription: "..." } }]`, and a non-vision default model; expected `BodyForAgent` should contain the cached `[Sticker ...]` description, but this commit routes it to `<media:image>` while `stickerCacheHit` removes media.
- Is this the best way to solve the issue? yes. The MIME-derived placeholder is appropriate for regular saved media, but sticker metadata is an existing Telegram-owned semantic layer and should bypass this generic classifier.
