## Findings

### Low: CDP relay ignores the entrypoint’s default CDP port env

- Kind: compatibility
- File: `scripts/sandbox-browser-entrypoint.sh`
- Line: 192
- Evidence: The shell wrapper defaults `CDP_PORT="${OPENCLAW_BROWSER_CDP_PORT:-9222}"`, but the embedded Python reads `os.environ["OPENCLAW_BROWSER_CDP_PORT"]` directly. A disposable container with `OPENCLAW_BROWSER_CDP_AUTH_TOKEN` but no explicit `OPENCLAW_BROWSER_CDP_PORT` started Chromium, then the Python relay crashed with `KeyError: 'OPENCLAW_BROWSER_CDP_PORT'`.
- Impact: Direct users of the sandbox browser image can no longer rely on the documented/default CDP port when enabling relay auth; the relay process exits and the container stops instead of listening on `9222`. The normal OpenClaw create path is not affected because `src/agents/sandbox/browser.ts` injects `OPENCLAW_BROWSER_CDP_PORT`.
- Suggested fix: Pass the shell-defaulted value into Python, for example `OPENCLAW_BROWSER_CDP_PORT="${CDP_PORT}" ... python3`, or have Python read `os.environ.get("OPENCLAW_BROWSER_CDP_PORT", "9222")`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: build `scripts/sandbox-browser-setup.sh`, then run the image with `OPENCLAW_BROWSER_CDP_AUTH_TOKEN` and without `OPENCLAW_BROWSER_CDP_PORT`; the embedded relay raises `KeyError`.
- Is this the best way to solve the issue? yes: keeping Python aligned with the shell default preserves the entrypoint contract without weakening relay auth.
