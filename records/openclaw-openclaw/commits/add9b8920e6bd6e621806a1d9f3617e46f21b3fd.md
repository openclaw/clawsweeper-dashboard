# Commit add9b89

## Summary

Found one test-regression issue: the Android node live-test record helper now accepts `null`.

## Findings

### Low: `expectRecord` no longer rejects null payloads

- Kind: test_gap
- File: `src/gateway/android-node.capabilities.live.test.ts`
- Line: 50
- Evidence: `expectRecord` was changed from `typeof value === "object"` plus `not.toBeNull()` to `expect(value).toEqual(expect.any(Object))`, followed only by `Array.isArray(value) === false`. In Vitest 4.1.5, `expect.any(Object)` is implemented as `typeof other == "object"`, so it matches `null`; `Array.isArray(null)` is also false. A focused check confirmed `new Any(Object).asymmetricMatch(null) === true`.
- Impact: `expectRecord(null, ...)` now passes and returns `null as Record<string, unknown>`. The live suite can miss Android/Gateway regressions where `device.permissions`, `device.health.memory`, or the fetched remote `gateway` config is `null` instead of a JSON object.
- Suggested fix: restore an explicit non-null object predicate, for example `expect(value, label).toSatisfy((candidate) => typeof candidate === "object" && candidate !== null && !Array.isArray(candidate))`, or throw from the helper before returning the cast.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: `node --input-type=module -e "import { Any } from '@vitest/expect'; console.log(new Any(Object).asymmetricMatch(null))"` prints `true`, and the helper’s remaining array check does not reject null.
- Is this the best way to solve the issue? yes: keep the matcher-scan cleanup goal, but use a predicate that preserves the old non-null object contract.
