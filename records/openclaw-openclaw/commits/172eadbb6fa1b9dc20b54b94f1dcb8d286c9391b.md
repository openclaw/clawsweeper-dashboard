# Commit 172eadbb6f

## Summary

Found one concrete relay contract validation hole. The default bundled paths look aligned, and the focused gateway tests pass, but the new guard still lets some valid bundled provider PCM encodings through while the relay tells clients to send G.711 u-law/8 kHz.

## Findings

### Medium: Relay accepts some non-u-law provider encodings and then advertises u-law audio

- Kind: bug
- File: `src/gateway/talk-transcription-relay.ts`
- Line: 67
- Evidence: `normalizeRelayInputEncoding` recognizes `pcm_s16le` as PCM, but returns `undefined` for other Mistral-supported PCM encodings such as `pcm_s32le`, `pcm_f16le`, and `pcm_f32le`. Because `assertRelayInputAudioConfig` only rejects when a recognized encoding differs from `g711_ulaw`, `{ encoding: "pcm_s32le", sampleRate: 8000 }` passes validation. The relay then returns `audio.inputEncoding: "g711_ulaw"` and forwards client bytes unchanged, while the Mistral provider can configure its upstream session with `encoding: "pcm_s32le"` from `extensions/mistral/realtime-transcription-provider.ts:12` and sends that encoding in the provider `audio_format` at `extensions/mistral/realtime-transcription-provider.ts:181`.
- Impact: A configured Mistral transcription relay can be created with an upstream PCM contract while downstream clients are told to send G.711 u-law. Audio is then interpreted under the wrong format, so transcription can fail or produce bad output instead of rejecting the incompatible session at create time.
- Suggested fix: Make relay validation cover all provider-supported non-u-law encodings, including Mistral’s `pcm_s32le`, `pcm_f16le`, and `pcm_f32le`, or move this to an explicit provider audio-contract capability. Also change the sample-rate check to reject defined non-8000 values with `sampleRate !== undefined && sampleRate !== RELAY_INPUT_SAMPLE_RATE_HZ`, so `0` cannot bypass the guard.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused `node --import tsx --input-type=module` probe against `createTalkTranscriptionRelaySession` created sessions for both `{ encoding: "pcm_s32le", sampleRate: 8000 }` and `{ encoding: "mulaw", sampleRate: 0 }`, returning `g711_ulaw/8000` instead of rejecting.
- Is this the best way to solve the issue? yes, for the narrow fix. The relay now owns a fixed `g711_ulaw/8000` downstream contract, so it should fail closed for any known incompatible upstream provider config.
