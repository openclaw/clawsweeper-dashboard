# Commit b217cd0

## Summary

Found active edge-case bugs in the new Discord channel-audience DM access path. A later `main` commit already fixed the role-overwrite ordering issue from this commit, but the current permission calculator still misses thread parent permissions and guild-owner bypass semantics.

## Findings

### Medium: Thread channel audiences can authorize users who cannot view the parent channel

- Kind: security
- File: `extensions/discord/src/send.permissions.ts`
- Line: 157
- Evidence: `canViewDiscordGuildChannel` fetches only the configured `channelId` and computes `ViewChannel` from that object’s own `permission_overwrites`; it never detects thread channel types or fetches/applies the thread `parent_id`. Discord documents that threads inherit permissions from their parent channel and users need `VIEW_CHANNEL` on the parent to view threads. ([docs.discord.com](https://docs.discord.com/developers/topics/permissions))
- Impact: If an operator configures `accessGroups.<name>.channelId` to a thread under a private channel, a guild member with base `ViewChannel` but denied access to the private parent can be treated as in the channel audience and allowed to DM/control the bot.
- Suggested fix: For `isThreadChannelType(channel.type)`, require `parent_id`, fetch the parent channel, and compute `ViewChannel` against the parent channel’s overwrites. Add a regression test for a public/private thread whose parent denies `ViewChannel` to `@everyone`.
- Confidence: high

### Low: Guild owners are falsely excluded from channel-audience groups

- Kind: bug
- File: `extensions/discord/src/send.permissions.ts`
- Line: 63
- Evidence: `resolveMemberGuildPermissionBits` only ORs `@everyone` and member role permissions; it does not accept `userId` or inspect `guild.owner_id`. Discord’s permission pseudocode returns all permissions when the member is the guild owner. ([docs.discord.com](https://docs.discord.com/developers/topics/permissions))
- Impact: A guild owner can be blocked from `discord.channelAudience` DM access for private channels where they have owner-level visibility but no role grants the computed `ViewChannel` bit.
- Suggested fix: Include `owner_id` in the guild shape, pass `userId` into the guild-permission helper, and return `ALL_PERMISSIONS` when `guild.owner_id === userId`. Add tests for `canViewDiscordGuildChannel` and `fetchMemberGuildPermissionsDiscord`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. Mock `getChannel` as a thread with `guild_id`, `parent_id`, and no effective overwrites, mock the parent as denying `ViewChannel` to `@everyone`, then call `canViewDiscordGuildChannel`; current code never reads the parent and can return true from guild base permissions. For the owner case, mock `getGuild` with `owner_id: "user1"` and no role `ViewChannel`; current code returns false.
- Is this the best way to solve the issue? Yes. The fix belongs in the shared Discord permission calculator introduced for access groups, so both diagnostics and channel-audience authorization follow Discord’s documented permission model.
