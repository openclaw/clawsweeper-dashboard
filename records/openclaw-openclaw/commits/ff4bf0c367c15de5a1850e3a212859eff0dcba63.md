## Findings

### Low: Document or separate private pip index credentials from the package build arg

- Kind: privacy
- File: `Dockerfile`
- Line: 223
- Evidence: The new `OPENCLAW_IMAGE_PIP_PACKAGES` Docker build arg is expanded into `python3 -m pip install ... $OPENCLAW_IMAGE_PIP_PACKAGES`, and `scripts/docker/setup.sh` forwards the same value via `--build-arg`. Pip package installs can include package index options, while Docker documents build args as unsuitable for secrets because they can persist in image metadata/history. ([pip.pypa.io](https://pip.pypa.io/en/stable/cli/pip_install/?highlight=hash&utm_source=openai))
- Impact: Operators needing private Python packages may naturally put `--index-url`, `--extra-index-url`, or a direct URL containing a token into `OPENCLAW_IMAGE_PIP_PACKAGES`. That can leak package registry credentials through image metadata/provenance/history when the image is shared or inspected.
- Suggested fix: Keep `OPENCLAW_IMAGE_PIP_PACKAGES` for non-secret package specs, but explicitly warn in Docker/Podman docs not to put credentials or tokenized URLs in it. If private indexes are intended, add a BuildKit secret/pip.conf path and pass it through setup separately.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? No. I did not run a Docker build with a real secret value; the failure mode is based on the new build-arg path plus Docker’s documented build-arg secret behavior.
- Is this the best way to solve the issue? Unclear. A docs-only warning is probably enough for the current opt-in feature; BuildKit secret support is safer if authenticated private indexes are a supported use case.
