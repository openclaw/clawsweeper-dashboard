## Findings

### Medium: Swift config-only edits skip Swift validation

- Kind: regression
- File: `scripts/ci-changed-scope.mjs`
- Line: 33
- Evidence: `macos-swift` runs only when preflight emits `run_macos_swift == 'true'` in `.github/workflows/ci.yml:1939`, and that job is where `swiftlint lint --config config/swiftlint.yml` and `swiftformat --lint ... --config config/swiftformat` run in `.github/workflows/ci.yml:1999`. Current `MACOS_NATIVE_RE` in `scripts/ci-changed-scope.mjs:33` only matches `apps/...` native paths, not `config/swiftformat` or `config/swiftlint.yml`. A focused probe on current `main` returns `runMacos: false` for both new config paths. Local changed-gate routing has the same blind spot: `scripts/changed-lanes.mjs:8` only treats `apps/` as the app lane, while `config/` is tooling at `scripts/changed-lanes.mjs:11`; the resulting plan runs `lint:scripts` but not `lint:apps`, and `lint:apps` is the package script that delegates to `lint:swift` in `package.json:1401`.
- Impact: a future PR that changes only `config/swiftformat` or `config/swiftlint.yml` can merge without the macOS Swift CI lane or local `lint:swift` checking that the moved config is valid. A bad SwiftLint/SwiftFormat config would then surface later on unrelated native changes instead of at the PR that introduced it.
- Suggested fix: add the two Swift config paths to the native/macOS CI scope and to local changed-lane app routing, or add a dedicated Swift-config scope that schedules `macos-swift` and `lint:apps`; cover both `config/swiftformat` and `config/swiftlint.yml` in `src/scripts/ci-changed-scope.test.ts` and `test/scripts/changed-lanes.test.ts`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Running `detectChangedScope(["config/swiftformat", "config/swiftlint.yml"])` returns `runMacos: false`; running `detectChangedLanes(...)` plus `createChangedCheckPlan(...)` returns only tooling commands and omits `lint:apps`.
- Is this the best way to solve the issue? yes. The configs now directly control the Swift lint/format job, so the routing source of truth should classify them with the Swift app/native validation surface.
