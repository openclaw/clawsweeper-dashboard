# Commit c91e20a

## Summary

Found one release-automation regression: the new candidate checklist runs the post-publish beta smoke by default before the candidate has been published to npm.

## Findings

### Medium: Candidate checklist blocks on unpublished npm beta during default Parallels step

- Kind: regression
- File: `scripts/release-candidate-checklist.mjs`
- Line: 400
- Evidence: `release:candidate` dispatches/validates the npm preflight only at `scripts/release-candidate-checklist.mjs:434`, downloads and validates the prepared tarball, then calls `pnpm release:beta-smoke -- --beta <version> --skip-telegram` at `scripts/release-candidate-checklist.mjs:400` before it prints the publish command at `scripts/release-candidate-checklist.mjs:507`. That beta-smoke path runs Parallels by default (`scripts/release-beta-smoke.ts:350`), and the Parallels beta-validation helper resolves the target via `npm view openclaw@<version>` (`scripts/e2e/parallels/npm-update-smoke.ts:947`, `scripts/e2e/parallels/package-artifact.ts:33`). A focused registry probe for an unpublished exact beta version returned npm `E404`.
- Impact: the default `pnpm release:candidate -- --tag vYYYY.M.D-beta.N` flow can fail after all expensive preflight validation succeeds, because the exact beta version is not on npm until the publish command that this script has not run yet. Operators must discover and use `--skip-parallels`, or the release evidence script cannot reach its final evidence/publish-command output.
- Suggested fix: do not run `release:beta-smoke` by default in the pre-publish candidate checklist. Either mark Parallels as a post-publish step, or add a tarball/artifact mode so the Parallels update/fresh-target smoke uses the already downloaded preflight `.tgz` instead of resolving `openclaw@<version>` from npm.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Use a new unpublished beta tag with default `pnpm release:candidate -- --tag vYYYY.M.D-beta.N`; after preflight artifacts validate, the default Parallels call resolves `openclaw@YYYY.M.D-beta.N` from npm and fails before evidence is written. I did not run the full Parallels flow because it requires VMs and provider credentials.
- Is this the best way to solve the issue? yes. The script is pre-publish evidence automation, while `release:beta-smoke` is documented and implemented as a published-package beta smoke; the safer fix is to separate those phases or explicitly pass a prepared tarball.
