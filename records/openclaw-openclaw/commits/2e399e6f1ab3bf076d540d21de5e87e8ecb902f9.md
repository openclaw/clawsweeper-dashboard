## Summary

Found one actionable release-check reliability bug. Current `main` has already fixed the versioned tarball-name part in a later commit, but the timeout matcher still does not match the real command shape because the real packaged-upgrade command includes `--no-restart`.

## Findings

### Medium: Windows packaged-upgrade timeout fallback misses the real command

- Kind: reliability
- File: `scripts/openclaw-cross-os-release-checks.ts`
- Line: 1400
- Evidence: `buildPackagedUpgradeUpdateArgs()` builds `update --tag ... --yes --json --no-restart --timeout ...` at `scripts/openclaw-cross-os-release-checks.ts:1360`, and `runCommand()` formats timeout errors from exactly `command + args.join(" ")`. The new recognizer at `scripts/openclaw-cross-os-release-checks.ts:1400` expects `--yes --json --timeout`, with no `--no-restart`, and the test fixture at `test/scripts/openclaw-cross-os-release-checks.test.ts:750` has the same omission. A focused probe using the real arg builder printed the real command and returned `false` from `isRecoverableWindowsPackagedUpgradeTimeoutError(..., "win32")`.
- Impact: the intended recovery for the known Windows packaged-upgrade timeout will not trigger on the real release-check command. The catch block at `scripts/openclaw-cross-os-release-checks.ts:781` rethrows, so the Windows packaged-upgrade lane can still fail instead of falling back to direct candidate install.
- Suggested fix: make the matcher include `--no-restart` in the expected sequence, or better, derive/compare against `buildPackagedUpgradeUpdateArgs()` so the recognizer and test fixture cannot drift when update arguments change.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Import the script, build the real packaged-upgrade args, construct the timeout message the local runner emits, and call `isRecoverableWindowsPackagedUpgradeTimeoutError`; it returns `false`.
- Is this the best way to solve the issue? yes. The fallback is intentionally narrow, so the safe fix is to keep it narrow but align it with the actual command generated by the same script.
