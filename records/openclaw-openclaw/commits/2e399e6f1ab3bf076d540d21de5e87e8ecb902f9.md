# Commit 2e399e6f1a

## Summary

Found one concrete release-check regression. The Windows timeout fallback added by this commit does not match the actual packaged-upgrade command line on current `main`, so the intended recovery path is skipped.

## Findings

### Medium: Windows packaged-upgrade timeout fallback misses the real `--no-restart` command

- Kind: reliability
- File: `scripts/openclaw-cross-os-release-checks.ts`
- Line: 1400
- Evidence: `runUpgradeLane` calls `buildPackagedUpgradeUpdateArgs()`, which builds `update --tag <url> --yes --json --no-restart --timeout <n>` at `scripts/openclaw-cross-os-release-checks.ts:1360`. `runCommand` reports timeouts as `Command timed out: <node> <openclaw.mjs> ${args.join(" ")}` at `scripts/openclaw-cross-os-release-checks.ts:3354`. The new matcher expects `--yes --json --timeout`, with no `--no-restart`, so it returns false for the actual generated args. Focused probe output: `update --tag http://127.0.0.1:49951/openclaw-current.tgz --yes --json --no-restart --timeout 1200` followed by `false`.
- Impact: the known Windows packaged upgrade timeout will still throw out of `runUpgradeLane` instead of setting `usedWindowsPackagedUpgradeTimeoutFallback` and installing the candidate directly. That can keep the cross-OS release validation red even though this commit was meant to recover it. The current test passes because it hand-writes a command string that omits `--no-restart`.
- Suggested fix: make `isRecoverableWindowsPackagedUpgradeTimeoutError` accept the real `--no-restart` argument, preferably by constructing the expected command from `buildPackagedUpgradeUpdateArgs()` or updating the regex to allow `--no-restart` between `--json` and `--timeout`. Update the test to build its timeout message from `buildPackagedUpgradeUpdateArgs()`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Import the helper and pass a timeout message built from `buildPackagedUpgradeUpdateArgs("http://127.0.0.1:49951/openclaw-current.tgz")`; current `main` returns `false`.
- Is this the best way to solve the issue? yes. The recognizer should match the actual command builder used by the same flow, instead of a copied command shape.
