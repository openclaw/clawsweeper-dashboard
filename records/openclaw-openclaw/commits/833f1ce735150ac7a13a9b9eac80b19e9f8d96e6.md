## Findings

### Low: Fanout phase-2 fallback hijacks unrelated mock requests

- Kind: reliability
- File: `extensions/qa-lab/src/providers/mock-openai/server.ts`
- Line: 2056
- Evidence: `scenarioState` is shared per mock server instance at `extensions/qa-lab/src/providers/mock-openai/server.ts:2539`. After the second fanout spawn sets `subagentFanoutPhase = 2`, the new branch at `extensions/qa-lab/src/providers/mock-openai/server.ts:2056` returns `"subagent-1: ok\nsubagent-2: ok"` for any later request with a non-empty `prompt`, before other deterministic branches like `visible skill marker` at `extensions/qa-lab/src/providers/mock-openai/server.ts:2140`.
- Impact: a different scenario or concurrent request sharing the same mock provider can receive the fanout completion instead of its own scripted response. I reproduced this by issuing two fanout spawn turns, then posting `Visible skill marker QA.`; it returned the fanout summary instead of `VISIBLE-SKILL-OK`.
- Suggested fix: do not let global fanout phase alone preempt unrelated prompts. Narrow the branch to beta/fanout continuation evidence, move it after other prompt-specific branches, or key fanout state by conversation/session rather than server-wide state.
- Confidence: high

- Is this the best way to solve the issue? unclear. The safest fix is per-conversation state; the smaller fix is to narrow/reorder the phase-2 fallback so it cannot override unrelated scenario branches.
