# Commit e201fbf

## Summary

Found one narrow fallback hole that still matters on current `main`: a stale first xAI API-key auth profile can block fallback to a later valid xAI API-key profile when no env/config key is available.

## Findings

### Low: Stale xAI API-key profile can still block later profile fallback

- Kind: reliability
- File: `extensions/xai/src/web-search-provider.runtime.ts`
- Line: 328
- Evidence: after this commit, any profile-backed 401 reaches `resolveXaiWebSearchApiKeyFallback`, but that fallback first calls `resolveXaiProviderAuthCredential(... credentialPrecedence: "env-first")` and immediately returns any non-OAuth provider-auth result. In `src/agents/model-auth.ts:654`, `env-first` only checks env first; if no env key exists, it still resolves the first profile in auth order. For two xAI API-key profiles where the first key is stale and the second key is valid, fallback resolves the same first key again, returns before scanning `resolveXaiProviderApiKeyProfileFallback`, and `extensions/xai/src/web-search-provider.runtime.ts:407` rethrows because `fallback.apiKey === auth.apiKey`.
- Impact: Grok `web_search` still fails for users with multiple xAI API-key auth profiles if the first profile is stale and no `XAI_API_KEY` or plugin web-search key is configured. The new regression test covers stale API-key profile to env fallback, but not stale API-key profile to later API-key profile fallback.
- Suggested fix: pass the failed `profileId` or failed `apiKey` into `resolveXaiWebSearchApiKeyFallback`, skip that same profile/key when resolving env-first/provider profile fallback, then continue scanning usable non-OAuth xAI profiles until a different key is found. Add a regression test with two API-key profiles and no env/config key.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes: seed xAI auth order with two `api_key` profiles, mock the first request as 401, leave `XAI_API_KEY` and plugin config unset, and expect the second profile to be tried. I did not add or run that new repro because this was report-only.
- Is this the best way to solve the issue? Yes: the fix should stay in the xAI fallback path and skip only the already-failed profile/key, preserving the env/config fallback behavior added by this commit.
