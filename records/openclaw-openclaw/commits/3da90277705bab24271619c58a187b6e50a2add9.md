# Commit 3da9027

## Summary

Found one concrete data-loss recovery risk: update-time doctor now allows config size-drop writes while relying on a `.pre-update` file that is first-write-wins across all future updates, so a later update can overwrite a large config and point users at a stale backup.

## Findings

### Medium: update size-drop writes can rely on a stale `.pre-update` backup

- Kind: data_loss
- File: `src/flows/doctor-health-contributions.ts`
- Line: 609
- Evidence: `runWriteConfigHealth` now sets `allowConfigSizeDrop` whenever `updateDoctorRun` is true, bypassing the size-drop rejection in `src/config/io.ts:420`. The recovery message points to `${ctx.configPath}.pre-update` at `src/flows/doctor-health-contributions.ts:614`, but `createPreUpdateConfigSnapshot` writes that path with `flag: "wx"` and silently returns on `EEXIST` in `src/config/backup-rotation.ts:131`, so an old snapshot from a prior update is never refreshed.
- Impact: after one update has created `openclaw.json.pre-update`, a later update that triggers a large doctor config shrink can now commit the shrink instead of being rejected. The user is told to use the pre-update backup, but that file may contain config from an earlier update rather than the state immediately before the destructive write.
- Suggested fix: make the pre-update snapshot scoped to the current update run, or refresh/rotate it once at the beginning of each top-level update before any update-time config writes. Only allow update-time size drops after proving the backup path is fresh for that update.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes: create `openclaw.json.pre-update`, change `openclaw.json`, then run an update doctor path that shrinks config by more than 50%. The current code will pass `allowConfigSizeDrop: true` and preserve the old `.pre-update` because `createPreUpdateConfigSnapshot` returns on `EEXIST`.
- Is this the best way to solve the issue? Yes, the fix should be in the snapshot freshness contract, not by only changing the log. The new behavior explicitly relies on the backup as the safety net.
