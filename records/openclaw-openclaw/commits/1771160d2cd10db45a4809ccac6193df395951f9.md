# Commit 1771160

## Summary

Found one compatibility regression in the public plugin SDK surface. The security goal is sound for bundled public search providers, but the commit changes the runtime behavior of the existing `withTrusted*` helpers that third-party plugins can already import.

## Findings

### Medium: Public `withTrusted*` SDK helpers now block private self-hosted endpoints

- Kind: compatibility
- File: `src/agents/tools/web-guarded-fetch.ts`
- Line: 10
- Evidence: The commit changes `WEB_TOOLS_TRUSTED_NETWORK_SSRF_POLICY` from `{ dangerouslyAllowPrivateNetwork: true, allowRfc2544BenchmarkRange: true }` to `{}`, while keeping `withTrustedWebToolsEndpoint()` as the same exported helper. That helper is still re-exported from public SDK entrypoints in `src/plugin-sdk/provider-web-fetch.ts` and `src/plugin-sdk/provider-web-search.ts`. A local smoke against `http://127.0.0.1:<port>` confirmed the new behavior: `withTrustedWebToolsEndpoint` throws `SsrFBlockedError: Blocked hostname or private/internal/special-use IP address`, while the new `withSelfHostedWebToolsEndpoint` succeeds.
- Impact: Existing third-party provider plugins that used the public `withTrustedWebToolsEndpoint` or `withTrustedWebSearchEndpoint` for operator-configured private/self-hosted endpoints will fail at runtime after this commit, with no type error and no migration guidance. The bundled SearXNG and Firecrawl plugins were migrated, but external plugins cannot be migrated by this commit.
- Suggested fix: Preserve compatibility by keeping `withTrusted*` as the private/self-hosted-compatible helper, add a new strict/public helper for bundled public web providers, and migrate bundled public providers to that helper. If the semantic break is intentional, document it explicitly in the SDK migration docs and changelog.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Import `withTrustedWebToolsEndpoint` from the SDK/internal helper and call it against a local loopback HTTP server; it now blocks with `SsrFBlockedError`. Calling `withSelfHostedWebToolsEndpoint` against the same URL succeeds.
- Is this the best way to solve the issue? unclear. The security fix needs public providers off the permissive policy, but renaming the strict path or adding a new strict/public helper would avoid silently changing an existing public SDK helper.
