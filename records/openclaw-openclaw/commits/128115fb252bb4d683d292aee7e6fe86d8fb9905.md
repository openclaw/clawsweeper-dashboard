# Commit 128115f

## Summary

Found one bug in the new compile-cache prune path: it removes a directory layout Node does not use, so the install-time prune is ineffective for both configured and default compile caches.

## Findings

### Medium: postinstall prunes the wrong Node compile-cache directory

- Kind: bug
- File: `scripts/postinstall-bundled-plugins.mjs`
- Line: 876
- Evidence: `openclaw.mjs:83` enables Node's compile cache with `module.enableCompileCache()` and no OpenClaw-specific subdirectory. A live Node check on v24.14.1 showed `NODE_COMPILE_CACHE=/tmp/x/cache-root` resolves to `/tmp/x/cache-root/v24.14.1-x64-...`, and the default base similarly uses `<tmp>/node-compile-cache/v...`. The new helper instead checks and removes `<base>/openclaw`, and a focused smoke left the real `v24.14.1-x64-...` cache in place after calling `pruneOpenClawCompileCache`.
- Impact: installs and updates still keep stale OpenClaw bytecode cache entries in the actual Node compile-cache directory, so this commit does not fix the stale-cache update failure it is meant to address.
- Suggested fix: prune the actual directory returned/used by Node for each base, such as versioned children under `NODE_COMPILE_CACHE` and `tmpdir()/node-compile-cache`, or move OpenClaw to an explicit `enableCompileCache({ directory: join(base, "openclaw") })` contract and align docs/tests with that.
- Confidence: high
