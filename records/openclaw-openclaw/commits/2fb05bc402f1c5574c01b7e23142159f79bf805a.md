# Commit 2fb05bc

## Summary

Found two concrete regressions in the auth-profile recovery changes. Both convert previously nonblocking cooldown marking into awaited work on retry paths, and both are still present on current `main`.

## Findings

### Medium: Prompt profile rotation now waits for cooldown persistence before retrying

- Kind: reliability
- File: `src/agents/pi-embedded-runner/run.ts`
- Line: 2046
- Evidence: The prompt `rotate_profile` branch now awaits `maybeMarkAuthProfileFailure()` before it can record the retry and `continue`. In the base commit this call was fire-and-forget with a `.catch(...)`.
- Impact: A slow or contended auth-profile store lock blocks prompt-side recovery, so the agent does not start the next profile attempt even though `advanceAuthProfile()` already succeeded.
- Suggested fix: Restore nonblocking marking in the retry path, or start the retry before awaiting durable cooldown persistence.
- Confidence: high

### Medium: Assistant profile rotation now waits for cooldown persistence before returning retry

- Kind: reliability
- File: `src/agents/pi-embedded-runner/run/assistant-failover.ts`
- Line: 156
- Evidence: `handleAssistantFailover()` now awaits `markFailedProfile()` immediately after `advanceAuthProfile()`. The existing test `src/agents/pi-embedded-runner/run/assistant-failover.test.ts:61` intentionally holds the mark promise and expects the retry outcome first; it now times out after 120s.
- Impact: Assistant-side rate-limit/auth recovery can hang behind the auth-store write instead of rotating promptly to the next usable account.
- Suggested fix: Restore `void markFailedProfile()` for successful profile rotation, keeping logging inside the helper.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The prompt-side focused test fails because the second attempt never starts while the mark promise is held; the assistant-side focused test times out waiting for `handleAssistantFailover()` to return.
- Is this the best way to solve the issue? no. The transcript and sandbox fixes use reasonable seams, but cooldown marking should remain off the hot retry path.
