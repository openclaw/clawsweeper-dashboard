# Commit 1470b43

## Summary

Found one low-severity Telegram streaming cleanup regression: the fix avoids reusing the stale preview, but can leave that old preview message visible because its message id is dropped before cleanup can delete it.

## Findings

### Low: Discarded preview rotation loses the old preview id before cleanup

- Kind: bug
- File: `extensions/telegram/src/bot-message-dispatch.ts`
- Line: 604
- Evidence: When a visible non-final tool/media payload lands, `markVisibleNonPreviewBoundary()` sets `discardAnswerPreviewOnNextRotation = true` if an answer preview has streamed. On the next assistant-message start, `rotateAnswerLaneForNewAssistantMessage()` materializes the old preview but skips adding it to `archivedAnswerPreviews` when that flag is true, then still calls `answerLane.stream?.forceNewMessage()`. The draft stream’s `forceNewMessage()` clears `streamMessageId` in `extensions/telegram/src/draft-stream.ts:279`, while cleanup only deletes the active stream message or ids retained in `archivedAnswerPreviews`. The discarded old preview is therefore neither active nor archived.
- Impact: In Telegram preview streaming, a pre-tool draft such as tool progress or an incomplete assistant partial can remain permanently visible after a tool/media bubble and the next assistant final. This is most concrete for tool-progress previews, where the previous behavior archived the preview with `deleteIfUnused: true` and cleanup deleted it.
- Suggested fix: When `discardAnswerPreviewOnNextRotation` is true and `previewMessageId` is numeric, still track the preview for cleanup, for example by adding it to `archivedAnswerPreviews` with `deleteIfUnused: true`, or explicitly delete it before/after forcing a new message.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: stream an answer/tool-progress preview, trigger compaction, deliver a visible non-final tool/media payload, then start the next assistant message. The rotation path skips archive tracking and `forceNewMessage()` drops the only active id.
- Is this the best way to solve the issue? yes: keep the new “do not reuse this preview for finals” behavior, but preserve the message id long enough for cleanup.
