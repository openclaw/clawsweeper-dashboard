# Commit d2db67e

## Summary

Found one concrete regression in the `cron.update` error handling path: malformed cron edits now return `INVALID_REQUEST`, but the existing job has already been mutated in memory before Croner throws, and a later cron write can persist that rejected schedule.

## Findings

### Medium: Failed cron.update can leave and later persist the rejected cron schedule

- Kind: data_loss
- File: `src/gateway/server-methods/cron.ts`
- Line: 341
- Evidence: The new handler catches `TypeError`/`RangeError` from `context.cron.update(jobId, patch)` and returns `INVALID_REQUEST`. But `CronService.update` applies the patch to the live job first at `src/cron/service/ops.ts:356`; `applyJobPatch` assigns `job.schedule` at `src/cron/service/jobs.ts:633`; only afterward does `computeJobNextRunAtMs` run at `src/cron/service/ops.ts:379`, eventually constructing `new Cron(...)` at `src/cron/schedule.ts:32` where invalid expressions throw. A focused live check through `cronHandlers["cron.update"]` returned `false INVALID_REQUEST` for `99 * * * *`, while `cron.getJob(id).schedule` was already `{"kind":"cron","expr":"99 * * * *"}`; after an unrelated `cron.add`, the rejected schedule was written to `jobs.json`.
- Impact: A user-visible failed edit can still corrupt the in-memory cron job, and the next successful cron mutation can persist the invalid schedule. That can leave the job with a stale old `nextRunAtMs`, break future scheduling/reload behavior, and make the `INVALID_REQUEST` response misleading because the requested change was partially applied.
- Suggested fix: Validate Croner-backed schedule changes before mutating the stored job, or make `CronService.update` transactional by applying patches to a clone, computing the next run on the clone, then committing the clone only after all validation succeeds. Add a regression test that a rejected cron update leaves `context.cron.getJob(id).schedule` and persisted job definitions unchanged.
- Confidence: high
