# Commit c659590

## Summary

Found one contract regression in the restored external-output helper. Current `main` still has the issue.

## Findings

### Medium: Local external-output wrapper drops the upstream helper contract

- Kind: regression
- File: `src/infra/fs-safe.ts`
- Line: 51
- Evidence: Before this commit, `writeExternalFileWithinRoot` was re-exported from `@openclaw/fs-safe/output`. The locked dependency’s contract is generic, returns `{ path, result }`, supports `maxBytes` and `mode`, and finalizes with `mkdir: true` (`node_modules/@openclaw/fs-safe/dist/output.d.ts:1`, `node_modules/@openclaw/fs-safe/dist/output.js:57`). The replacement wrapper narrows `write` to `Promise<void>`, returns only `{ path }`, omits `maxBytes`/`mode`, and delegates to `writeViaSiblingTempPath` (`src/infra/fs-safe.ts:51`, `src/infra/fs-safe.ts:67`). A live probe confirmed current OpenClaw ignores `maxBytes: 1` and writes a 9-byte file, while the upstream helper rejects with `too-large`. A second probe confirmed `path: "nested/out.txt"` fails with `not-found` in OpenClaw but succeeds through the upstream helper because it creates missing parents.
- Impact: This is exported through the plugin SDK (`src/plugin-sdk/security-runtime.ts:35`), so existing or third-party plugins compiled against the prior helper can lose callback results, silently lose size/mode enforcement at runtime, and fail for nested output paths that previously worked. The lost `maxBytes` enforcement is the highest-risk part because external writers can produce unexpectedly large files.
- Suggested fix: Restore the `@openclaw/fs-safe/output` re-export if possible. If a local wrapper is required, mirror the upstream signature and behavior: generic callback result, `maxBytes`, `mode`, `mkdir: true`, `sourceHardlinks: "reject"`, canonical final path, and return `{ path, result }`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. I ran focused `tsx` probes comparing `./src/infra/fs-safe.ts` against `@openclaw/fs-safe/output`: OpenClaw returned only `{ path }`, did not enforce `maxBytes`, and rejected a nested output path; the upstream helper returned `result`, rejected `too-large`, and created the nested parent.
- Is this the best way to solve the issue? yes. The cleanest fix is to keep the public helper aligned with the dependency it previously re-exported, either by restoring that re-export or by copying the full contract.
