# Commit 38456f5

## Summary

Found one Control UI regression in supported `session.scope = "global"` setups. The new agent-scoped refreshes send `agentId: "main"` even when the active chat session is the canonical `global` session, and the Gateway’s `agentId` filter removes `global` rows even when `includeGlobal: true`.

## Findings

### Medium: Global-scope chat refreshes drop the active `global` session from the picker

- Kind: regression
- File: `ui/src/ui/app-chat.ts`
- Line: 766
- Evidence: `refreshChat()` now calls `loadSessions(..., { includeGlobal: true, includeUnknown: true, agentId: resolveAgentIdForSession(host) ?? undefined })`. For non-agent session keys, `resolveAgentIdForSession()` returns the default agent or `"main"` at `ui/src/ui/app-chat.ts:824`, so a `global` chat session becomes a scoped `agentId=main` list. The Gateway list filter then excludes `global` and `unknown` whenever `agentId` is present at `src/gateway/session-utils.ts:1994`. A focused live check confirmed the contract: broad list returned `["global","agent:main:main"]`, scoped list returned only `["agent:main:main"]`.
- Impact: In `session.scope = "global"`, `applySessionDefaults()` maps main aliases to the canonical `global` session (`ui/src/ui/app-gateway.ts:333`). After a manual refresh, slash-command refresh, model/thinking refresh, or post-chat final refresh, the session list no longer contains the active `global` row. `renderChatSessionSelect()` disables/falls back when the current key is absent from option groups, so the chat picker can show the wrong option or lose the active global session after the first reply.
- Suggested fix: Only pass `agentId` for parsed `agent:<id>:...` session keys. For `global`, `unknown`, and other non-agent keys, leave `agentId` undefined so `includeGlobal`/`includeUnknown` keep working. Apply the same guard to `resolveChatEventSessionListAgentId()` and `resolveSessionOptionsAgentId()`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Configure/use a `global` session, trigger a chat refresh path, and inspect the `sessions.list` payload: it includes `agentId: "main"` even though the active key is `global`; the Gateway then filters the `global` row out.
- Is this the best way to solve the issue? yes. The intended optimization is agent scoping for agent sessions; non-agent sentinel sessions should keep the existing broad list behavior.
