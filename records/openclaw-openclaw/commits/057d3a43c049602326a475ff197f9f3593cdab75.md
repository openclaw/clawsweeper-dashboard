# Commit 057d3a43

## Summary

Found three actionable issues in the new logged-in Discord Web capture path: secret-bearing env is exposed to candidate worktree code, documented `$HOME`/`~/` profile paths do not expand on the remote machine, and raw Discord UI metadata is published despite metadata redaction.

## Findings

### High: Candidate worktree code inherits the new Discord viewer and Crabbox secrets

- Kind: security
- File: `.github/workflows/mantis-discord-thread-attachment.yml`
- Line: 323
- Evidence: The new `MANTIS_DISCORD_VIEWER_CHROME_PROFILE_TGZ_B64`, Crabbox coordinator token, and Crabbox access client secrets are added to the whole `Run baseline and candidate` step env, before `run_lane candidate` executes `pnpm --dir "$repo_root" openclaw qa discord` from the candidate worktree. Candidate validation allows an `open-pr-head` from the same repository, so unreviewed candidate code can read these env vars before the later `capture_candidate_discord_web` function needs them.
- Impact: A malicious or compromised same-repo candidate branch can exfiltrate the logged-in Discord viewer profile archive and Crabbox credentials. This is broader than the intended visual-capture helper scope.
- Suggested fix: Move the new Discord viewer and Crabbox secrets into a separate capture-only step that runs trusted harness code after baseline/candidate lanes finish, or explicitly run candidate/baseline lanes under a sanitized env that only contains the QA credentials they require.
- Confidence: high

### Medium: `$HOME` and `~/` profile dirs are accepted but used literally

- Kind: bug
- File: `extensions/qa-lab/src/mantis/desktop-browser-smoke.runtime.ts`
- Line: 218
- Evidence: `assertSafeRemoteProfileDir` accepts `$HOME/...` and `~/...`, and docs/tests use `$HOME/.config/openclaw-mantis/...`, but `renderRemoteScript` wraps the path with `shellQuote` and emits `profile='$HOME/...'`. In shell, that assigns a literal `$HOME/...` string; it does not expand to the remote home directory. A focused shell check created `./$HOME/.config/...` and did not create the real `$HOME/.config/...`.
- Impact: The configured persistent Chrome profile is not reused, so the logged-in Discord Web capture can silently launch with a fresh unauthenticated profile and miss the intended evidence.
- Suggested fix: Normalize `$HOME/` and `~/` to a real remote path in the generated script before quoting, or reject non-absolute paths and document `/home/<user>/...` only. Add a test that executes the generated assignment behavior rather than only checking the rendered string.
- Confidence: high

### Medium: Public evidence includes raw Discord URL and IDs despite redaction

- Kind: privacy
- File: `.github/workflows/mantis-discord-thread-attachment.yml`
- Line: 522
- Evidence: The workflow sets `OPENCLAW_QA_REDACT_PUBLIC_METADATA=1`, but `writeDiscordThreadReplyAttachmentEvidence` writes `guildId`, `channelId`, `threadId`, `messageId`, `parentMessageId`, and `discordWebUrl` into `discord-thread-reply-filepath-attachment-ui.json`. The workflow then includes that file in `mantis-evidence.json`, and `scripts/mantis/publish-pr-evidence.mjs` copies every manifest artifact to the `qa-artifacts` branch.
- Impact: Real Discord guild/channel/thread/message identifiers and the private Discord Web URL are published alongside PR evidence even though the run requested public metadata redaction.
- Suggested fix: Keep the UI JSON as a private intra-workflow artifact only, omit it from the public evidence manifest, or publish a redacted metadata copy that excludes Discord IDs and URLs.
- Confidence: high

### Medium: Crabbox is built from a mutable default branch inside a secret-bearing workflow

- Kind: supply_chain
- File: `.github/workflows/mantis-discord-thread-attachment.yml`
- Line: 260
- Evidence: The workflow clones `https://github.com/openclaw/crabbox.git` with `--depth 1` and builds whatever the default branch contains. The resulting binary is then executed in the step that has QA, Discord viewer, and Crabbox secrets.
- Impact: The workflow is not reproducible and trusts mutable external code at run time. A bad Crabbox default-branch change can run with the workflow’s secret env.
- Suggested fix: Pin Crabbox to an audited commit/ref or signed release artifact, and make the chosen ref explicit in the workflow inputs/env.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The secret exposure follows directly from the step-level env and `pnpm --dir "$repo_root"` candidate execution. The profile path bug was reproduced with a focused shell assignment check matching the generated `profile='$HOME/...'` shape.
- Is this the best way to solve the issue? yes for the first two: scope secrets to trusted capture-only code, and expand or reject home-relative remote paths. For the metadata leak, the safer direction is to separate private capture metadata from public evidence.
