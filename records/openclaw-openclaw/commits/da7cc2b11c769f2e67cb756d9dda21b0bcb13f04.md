# Commit da7cc2b

## Summary

Found one low-severity privacy regression in the new default Feishu manual setup path.

## Findings

### Low: Default manual setup asks for App Secret through an unmasked text prompt

- Kind: privacy
- File: `extensions/feishu/src/setup-surface.ts`
- Line: 376
- Evidence: The commit makes `promptFeishuSetupMethod` default to `"manual"` and the manual branch prompts for `"App Secret"` through `promptSingleChannelSecretInput`. That helper reaches `promptSingleChannelToken`, which calls `prompter.text({ message, validate })` without `sensitive: true`; the Clack prompter only switches to a masked password prompt when `params.sensitive` is set.
- Impact: A user following the new default Feishu setup path can type the App Secret visibly into the terminal, leaving it in scrollback, terminal recordings, or screenshots. This existed in the old fallback path, but the commit makes that path the default for Feishu.
- Suggested fix: Add a sensitive/masked mode to `promptSingleChannelSecretInput` or make `promptSingleChannelToken` use `sensitive: true` for plaintext secret entry, then cover Feishu’s App Secret prompt in setup tests.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. Run Feishu setup on a new config, accept the default manual path, choose plaintext App Secret entry, and observe that the flow uses the normal text prompt rather than the masked password prompt.
- Is this the best way to solve the issue? Yes. The existing `WizardTextParams.sensitive` contract already exists for exactly this terminal-leak class, so the fix should reuse it rather than adding Feishu-specific masking.
