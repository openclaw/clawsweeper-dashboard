# Commit 297f4c6e

## Summary

Found three actionable i18n regressions: the touched i18n test file no longer parses, the committed generated metadata is not in the generator’s canonical format so `ui:i18n:check` fails, and five newly exposed UI locales ship as 100% English fallback bundles.

## Findings

### Medium: New locale imports collide with Vitest bindings and break the UI test shard

- Kind: bug
- File: `ui/src/i18n/test/translate.test.ts`
- Line: 11
- Evidence: line 1 imports Vitest’s `it` and `vi`; lines 11 and 20 import locale exports named `it` and `vi`. `pnpm test ui/src/i18n/test/translate.test.ts` fails during transform with `Identifier it has already been declared` and `Identifier vi has already been declared`.
- Impact: the UI test shard cannot parse this file, so relevant CI/test gates fail before executing the i18n coverage added by the commit.
- Suggested fix: alias the locale imports, for example `it as itLocale` and `vi as viLocale`, and update the assertions/object entries to use those aliases.
- Confidence: high

### Medium: Committed generated metadata is out of sync with `scripts/control-ui-i18n.ts`

- Kind: reliability
- File: `ui/src/i18n/.i18n/de.meta.json`
- Line: 2
- Evidence: the commit writes fallback arrays inline, e.g. `fallbackKeys` on one line, but `renderMeta()` in `scripts/control-ui-i18n.ts` emits `JSON.stringify(meta, null, 2)`. `pnpm ui:i18n:check` fails with `control-ui-i18n drift detected`; a temp sync probe rewrote `ui/src/i18n/.i18n/de.meta.json` only by expanding that array to canonical multi-line JSON.
- Impact: the repo’s i18n generated-output check fails even without translation provider secrets. This can block changed gates or produce bot churn on main.
- Suggested fix: regenerate with `node --import tsx scripts/control-ui-i18n.ts sync --write` and commit the canonical metadata formatting for all touched locale meta files.
- Confidence: high

### Low: Newly supported UI locales are exposed while fully untranslated

- Kind: bug
- File: `ui/src/i18n/lib/registry.ts`
- Line: 22
- Evidence: `ar`, `it`, `vi`, `nl`, and `fa` are added to `LAZY_LOCALES` and browser-language resolution, but their metadata reports `translatedKeys: 0`, `totalKeys: 768`, and `fallbackKeys: 768`; `ui/src/i18n/locales/ar.ts:6` still has `health: "Health"` and the surrounding UI strings are English.
- Impact: users with Arabic, Persian, Italian, Dutch, or Vietnamese browser locales, or users selecting those locales, get an apparently supported language that renders the Control UI in English.
- Suggested fix: either keep these locales out of `SUPPORTED_LOCALES` until translated bundles are committed, or run the UI i18n sync with provider credentials and commit real translated bundles/TM before exposing them.
- Confidence: high
