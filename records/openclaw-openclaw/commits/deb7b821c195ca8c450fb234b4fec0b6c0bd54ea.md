# Commit deb7b82

## Summary

Found one concrete follow-up bug: the new startup planner normalizes the selected context-engine slot for plugin loading, but the actual context-engine resolver still uses the raw slot string. Mixed-case or alias slot values can now preload the plugin yet still fall back to `legacy`.

## Findings

### Medium: Normalized startup slot can still resolve to `legacy`

- Kind: bug
- File: `src/context-engine/registry.ts`
- Line: 504
- Evidence: The commit normalizes `plugins.slots.contextEngine` before startup filtering in `src/plugins/gateway-startup-plugin-ids.ts:112`, and adds a regression test where `Lossless-Claw` includes `lossless-claw` in startup IDs at `src/plugins/channel-plugin-ids.test.ts:1142`. But `resolveContextEngine()` later reads `config.plugins.slots.contextEngine`, only trims it, and does an exact `engines.get(engineId)` lookup at `src/context-engine/registry.ts:504` and `src/context-engine/registry.ts:519`. A focused smoke check registered `lossless-claw`, then resolved configs for `lossless-claw` and `Lossless-Claw`; the lower-case config returned `lossless-claw`, while `Lossless-Claw` logged the fallback warning and returned `legacy`.
- Impact: A user can have the plugin loaded by the new startup plan but still not use the selected context engine at runtime. This preserves the core symptom for slot values that the new test implies are normalized.
- Suggested fix: Normalize the context-engine slot value on the runtime resolution path as well, or materialize the normalized slot into the runtime config before `resolveContextEngine()` sees it. Add an end-to-end test that loads/registers a selected context-engine plugin and then resolves the active engine for a mixed-case slot value.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: register a `lossless-claw` context engine, then call `resolveContextEngine({ plugins: { slots: { contextEngine: "Lossless-Claw" } } })`; it falls back to `legacy` despite the startup planner including `lossless-claw`.
- Is this the best way to solve the issue? unclear: the safest fix depends on whether context-engine ids are intended to be plugin ids, aliases, or independently case-sensitive engine ids. The important contract is that startup planning and runtime resolution must use the same normalized id.
