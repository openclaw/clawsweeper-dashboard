# Commit c307a612

## Summary

Found one current regression: the new reply-chain prompt block can leak into user-visible transcript/UI text because the existing inbound metadata stripper does not recognize the new sentinel.

## Findings

### Medium: Reply-chain metadata survives UI transcript stripping

- Kind: privacy
- File: `src/agents/pi-embedded-runner/run/runtime-context-prompt.ts`
- Line: 87
- Evidence: This commit adds a new persisted prompt suffix labelled `Reply chain of current user message (untrusted, nearest first):`. The prompt is appended into `promptForModel` at `src/agents/pi-embedded-runner/run/attempt.ts:2828`, and pi-agent emits prompt messages into transcript state. The UI stripper only recognizes the older sentinels in `src/auto-reply/reply/strip-inbound-meta.ts:21`; it includes `Reply target...` but not `Reply chain...`.
- Impact: User-visible chat history surfaces that call `stripInboundMetadata` can show AI-facing reply-chain JSON, including quoted bodies, sender IDs/usernames, forwarded metadata, and media paths/refs. This violates the stripper contract that injected metadata blocks “must never surface in user-visible chat history.”
- Suggested fix: Add the reply-chain sentinel to `INBOUND_META_SENTINELS` and add a regression test using a JSON array block from `buildCurrentTurnPromptContextSuffix`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused `pnpm exec tsx -e` probe built a reply-chain suffix, passed `why?${suffix}` through `stripInboundMetadata`, and confirmed `strippedContainsReplyChain: true`.
- Is this the best way to solve the issue? yes. The existing stripper is sentinel-based and already strips the single-reply block; extending that list preserves the established contract with a narrow fix.
