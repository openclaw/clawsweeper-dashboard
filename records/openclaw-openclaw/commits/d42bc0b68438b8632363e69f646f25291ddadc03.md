# Commit d42bc0b

## Summary

Found one security regression in the manual checkout helper: the workflow now persists the job token in `remote.origin.url` before running checked-out repo code.

## Findings

### Medium: Manual checkout leaves `github.token` readable from `.git/config`

- Kind: security
- File: `.github/workflows/ci.yml`
- Line: 430
- Evidence: This commit changed the manual checkout from an ephemeral `git -c http.https://github.com/.extraheader=... fetch` to `git remote add origin "https://x-access-token:${CHECKOUT_TOKEN}@github.com/${CHECKOUT_REPO}.git"`. `CHECKOUT_TOKEN` is `${{ github.token }}`, and there is no later `remote set-url`, `remote remove`, or credential cleanup before the jobs run PR-controlled commands such as `pnpm build`, `pnpm test:*`, `pnpm lint`, and `./gradlew`. The same pattern is repeated at lines 638, 725, 806, 884, 960, 1083, 1214, 1364, and 1712.
- Impact: Any code running after these checkout steps can read `git remote get-url origin` or `.git/config` and recover the workflow token. The workflow permissions are `contents: read`, so this is not a write-token takeover, but it still exposes a repository-scoped credential to untrusted PR code and makes the change the opposite of auth hardening.
- Suggested fix: Keep the token out of persistent git config. Revert to the previous `http.extraheader` fetch, or fetch with a temporary credential mechanism and immediately reset `origin` to `https://github.com/${CHECKOUT_REPO}.git` before any repo code runs.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused local proof with a dummy token showed `git remote add origin "https://x-access-token:dummy-token-123@github.com/openclaw/openclaw.git"` stores the token verbatim in `remote.origin.url`.
- Is this the best way to solve the issue? yes. The previous `git -c http.extraheader=... fetch` shape authenticated only the fetch command and did not persist the token in the checkout that later executes repository code.
