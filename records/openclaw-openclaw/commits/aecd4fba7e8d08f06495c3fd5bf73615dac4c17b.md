# Commit aecd4fba

## Summary

Found one remaining Feishu topic-threading bug in a touched outbound path: rich/card payloads that also include media still send the media without `replyInThread`, so the attachment can leave the topic while the final card stays in it.

## Findings

### Medium: Card-plus-media payloads drop `replyInThread` for the media send

- Kind: bug
- File: `extensions/feishu/src/outbound.ts`
- Line: 572
- Evidence: In `sendPayload`, Feishu computes `replyToMessageId` from `ctx.replyToId`/`ctx.threadId`, then sends each media item through `sendMediaFeishu({ ..., replyToMessageId })` without `replyInThread`. The final card immediately below passes `replyInThread: ctx.threadId != null && !ctx.replyToId`, so only the card is topic-threaded. `sendMediaFeishu` only adds Feishu’s `reply_in_thread` API field when its own `replyInThread` parameter is true, and the local SDK types confirm `reply_in_thread?: boolean` is the supported reply flag.
- Impact: In a `group_topic` or `group_topic_sender` session, a reply payload with interactive/card content plus an image/file can split delivery: the attachment is sent as a non-thread reply/root-visible message, while the card lands in the topic. That is the same user-visible failure mode this commit is trying to eliminate for topic-scoped Feishu replies.
- Suggested fix: Reuse the normalized reply-mode helper in `sendPayload` and pass `replyInThread` into both the media `sendMediaFeishu` call and the final `sendCardFeishu` call. Add a regression test for `feishuOutbound.sendPayload` with `threadId`, interactive/card content, and `mediaUrl`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. Call `feishuOutbound.sendPayload` with `threadId: "om_topic_root"` and a payload containing both `interactive` or `presentation` content and `mediaUrl`. The existing test fixture at `extensions/feishu/src/outbound.test.ts:660` exercises this branch but does not assert `replyInThread` on the media call.
- Is this the best way to solve the issue? Yes. The rest of this commit already normalizes `threadId` into `{ replyToMessageId, replyInThread }` for text/media adapters; applying the same normalized value inside `sendPayload` keeps the Feishu-owned outbound behavior consistent.
