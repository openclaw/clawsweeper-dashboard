# Commit b1cfba2

## Summary

Found one regression in the new blocked-result path: after a critical tool-loop block is returned as a normal blocked tool result, that blocked result is recorded into loop history and resets the no-progress streak. A model that retries the same call after seeing the guardrail can get another run of the same tool instead of being blocked again.

## Findings

### Medium: Critical loop blocks reset the no-progress streak

- Kind: reliability
- File: `src/agents/pi-tools.before-tool-call.ts`
- Line: 643
- Evidence: The commit changes critical loop stops to `kind: "veto"` at `src/agents/pi-tools.before-tool-call.ts:438`, so `wrapToolWithBeforeToolCallHook` now builds a blocked result and records it via `recordLoopOutcome` at `src/agents/pi-tools.before-tool-call.ts:639` and `src/agents/pi-tools.before-tool-call.ts:643`. Because the critical path returns before `recordToolCall`, `recordToolCallOutcome` cannot match an existing in-flight history entry and pushes the blocked result as a new history item at `src/agents/tool-loop-detection.ts:719`. The no-progress detector only counts the latest identical `resultHash` and stops on the first different hash at `src/agents/tool-loop-detection.ts:374` and `src/agents/tool-loop-detection.ts:379`, so the blocked result becomes the latest hash and breaks the previous 20/30 identical no-progress outcomes.
- Impact: The critical breaker is no longer sticky. After one `details.status: "blocked"` result, the next identical polling/read call can execute again, and the guard only re-trips after another full threshold of identical tool outcomes. That weakens the runaway-loop protection this commit is trying to improve.
- Suggested fix: Do not feed tool-loop veto results into the no-progress outcome history, or record them with a marker/hash policy that preserves the existing critical streak instead of replacing the latest result hash. Add a regression test that executes the same call immediately after the first blocked result and asserts it is blocked again without invoking the underlying tool.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused `pnpm exec tsx -e ...` check using `detectToolCallLoop`, `recordToolCall`, and `recordToolCallOutcome` showed `beforeBlock.stuck: true` for the 20-call critical polling loop, then `afterBlock.stuck: false` immediately after recording the blocked result exactly as the new veto wrapper does.
- Is this the best way to solve the issue? yes, at direction level. The model should receive a blocked result, but that presentation result should not reset the internal critical-loop evidence.
