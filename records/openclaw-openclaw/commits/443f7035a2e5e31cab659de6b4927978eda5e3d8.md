# Commit 443f703

## Summary

Found one cache-path regression in the optional-tool availability fix. Fresh runtime factory results are filtered by per-tool manifest availability, but cached plugin tool descriptors can still re-expose an unavailable optional tool after the descriptor cache has been warmed.

## Findings

### Medium: Cached optional descriptors bypass manifest availability filtering

- Kind: regression
- File: `src/plugins/tools.ts`
- Line: 631
- Evidence: `resolveCachedPluginTools` computes `availableToolNames` at `src/plugins/tools.ts:589`, but the cached descriptor loop only applies that list to non-optional descriptors at `src/plugins/tools.ts:623`. Optional cached descriptors are only checked with `isOptionalToolAllowed` at `src/plugins/tools.ts:631`, then pushed at `src/plugins/tools.ts:650`. The fresh factory path added by this commit filters every returned tool through `isManifestToolNameAvailable` before optional allowlist handling at `src/plugins/tools.ts:987`, so the cached path no longer matches fresh behavior. The descriptor cache key in `src/plugins/tool-descriptor-cache.ts:117` includes config/context/source data but not the current env/auth evidence, so a descriptor cached while auth is present can be reused when the same config later lacks that auth.
- Impact: A plugin with an available default tool plus an auth-gated optional sibling can still show the optional sibling in the agent tool list after a previous authenticated resolution cached its descriptor. That reintroduces the issue this commit is trying to fix for warm descriptor-cache flows, and the new test only covers a warm registry with factories not called, not a warmed descriptor cache.
- Suggested fix: In `resolveCachedPluginTools`, require every cached descriptor name, optional or not, to be present in `availableToolNames` before creating the cached tool. Also consider using the filtered availability list when deciding whether to write descriptor cache entries, so cache coverage and cache reads use the same availability contract.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Warm the descriptor cache with a manifest plugin whose `contracts.tools` include `other_tool` plus `optional_tool`, where `optional_tool` has an auth signal and `tools.alsoAllow` names it while auth env is present. Resolve again with the same config/context and no auth env. `availableToolNames` becomes `["other_tool"]`, but the cached `optional_tool` descriptor still passes the optional allowlist branch and is returned.
- Is this the best way to solve the issue? yes. The fresh runtime path already has the correct per-tool availability predicate; the cached path should apply the same predicate before optional allowlist handling.
