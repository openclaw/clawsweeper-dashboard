# Commit 83a1080

## Summary

Found one remaining canonicalization hole in the changed payload selection logic: media-only final answers still replay accumulated pre-final text as an extra outbound payload.

## Findings

### Medium: Media-only canonical final answers still leak accumulated pre-final text

- Kind: bug
- File: `src/agents/pi-embedded-runner/run/payloads.ts`
- Line: 369
- Evidence: `shouldUseCanonicalFinalAnswer` requires `normalizedFallbackAnswerSourceText.length > 0`. For a valid final answer that is only a reply directive, such as `MEDIA:...` or `NO_REPLY` plus media/audio, `parseReplyDirectives(...).text` is empty even though the payload has outbound media. A focused smoke of `buildEmbeddedRunPayloads` with `assistantTexts: ["Preparing audio...", "NO_REPLY\nMEDIA:...\n[[audio_as_voice]]"]` returned two payloads: the stale progress text and the media payload.
- Impact: Telegram or another final-delivery channel can still receive an unintended extra text message before the intended media-only final answer. This preserves part of the duplicate/leak class this commit is trying to fix for supported media-only reply payloads.
- Suggested fix: make the canonical-final predicate content-aware instead of text-only. Parse `fallbackAnswerSourceText` once and allow canonical collapse when it has cleaned text, media URLs, or `audioAsVoice`; then reuse that parsed result or equivalent `hasOutboundReplyContent` check.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Call `buildEmbeddedRunPayloads` with two accumulated assistant texts where the final `lastAssistant` `final_answer` block is media-only or `NO_REPLY` plus media. The helper returns two payloads instead of collapsing to the canonical final media payload.
- Is this the best way to solve the issue? yes. The bug is in the new text-only gate, so the narrow fix is to treat directive-carried media/audio as canonical outbound content.
