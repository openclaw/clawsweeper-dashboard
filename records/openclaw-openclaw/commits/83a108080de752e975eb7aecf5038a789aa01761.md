# Commit 83a1080

## Summary

The text-answer canonicalization works for captioned/text final answers, but it still leaves the same multi-payload Telegram leak class open for media-only final answers.

## Findings

### Medium: Media-only final answers still replay accumulated pre-tool text

- Kind: bug
- File: `src/agents/pi-embedded-runner/run/payloads.ts`
- Line: 369
- Evidence: The new `shouldUseCanonicalFinalAnswer` requires `normalizedFallbackAnswerSourceText.length > 0`. For a valid final assistant answer that is only directives, such as `MEDIA:/tmp/reply.opus\n[[audio_as_voice]]`, `parseReplyDirectives()` produces deliverable media/audio but empty cleaned text, so the canonical final-answer path is skipped. The function then falls back to `nonEmptyAssistantTexts`, and each parsed answer text becomes a separate reply item. A focused probe on current `main` returned two payloads for `assistantTexts: ["Preparing media...", "MEDIA:/tmp/reply.opus\n[[audio_as_voice]]"]`: one text payload `"Preparing media..."` plus one media payload. The final dispatcher iterates reply arrays and sends each final payload separately in `src/auto-reply/reply/dispatch-from-config.ts:1548`.
- Impact: A Telegram/media turn can still send preliminary progress, raw-looking accumulated text, or internal/status text before the intended media-only final answer. This is the same user-visible duplication/leak class the commit is meant to close, and media-only replies are an explicitly supported delivery shape in the surrounding reply code.
- Suggested fix: Treat the canonical final answer as deliverable when parsed directives contain text, media URLs, or `audioAsVoice`; do not require non-empty cleaned text. Keep the no-content `NO_REPLY` case suppressed by checking parsed outbound content rather than only normalized text.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The focused `pnpm exec tsx -e ...` probe against current `main` produced two payloads from a media-only final answer with accumulated pre-tool text.
- Is this the best way to solve the issue? yes. The bug is in the canonical-answer eligibility guard; widening that guard to directive-backed outbound content preserves captioned media, media-only replies, and silent no-reply semantics without changing final dispatch.
