# Commit b744010

## Summary

Found one reliability issue: the Gateway `models.list` path still has a fresh plugin metadata/setup discovery path during auth visibility checks, even when the commit passes `runtimeAuthDiscovery: false`.

## Findings

### Medium: `models.list` static auth checks can still rescan plugin metadata

- Kind: reliability
- File: `src/agents/model-auth.ts`
- Line: 337
- Evidence: `models.list` now passes `runtimeAuthDiscovery: false` into `resolveVisibleModelCatalog`, which disables external CLI overlay and plugin synthetic auth. But `hasRuntimeAvailableProviderAuth` still unconditionally calls `resolveEnvApiKey(...)`. That helper builds provider env/auth-evidence maps through `resolveProviderEnvApiKeyCandidates` and `resolveProviderEnvAuthEvidence`, both of which call `loadPluginMetadataSnapshot({ preferPersisted: false })` via `src/secrets/provider-env-vars.ts:121` and `src/secrets/provider-env-vars.ts:158`. For providers without a candidate map, it can also fall through to `resolvePluginSetupProvider` at `src/agents/model-auth-env.ts:147`, which loads setup manifest/runtime registration.
- Impact: a default/configured Gateway `models.list` with any non-empty catalog row can still synchronously scan plugin metadata while resolving visibility. That leaves a concrete path for the same event-loop/CPU stalls this commit is trying to avoid, especially after startup prewarm creates a persisted `models.json` and the read-only catalog returns rows.
- Suggested fix: add a true static auth mode for `runtimeAuthDiscovery: false`: use core env candidates plus the current Gateway plugin metadata snapshot, or precompute env/evidence maps from that snapshot, and skip `loadPluginMetadataSnapshot` plus setup-runtime fallback in this path. Add a regression test that calls the real `resolveVisibleModelCatalog` or `models.list` with a non-empty catalog while `loadPluginMetadataSnapshot`/setup lookup is mocked to throw.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Source tracing gives the path: `models.list` -> `resolveVisibleModelCatalog` -> `createProviderAuthChecker` -> `hasRuntimeAvailableProviderAuth` -> `resolveEnvApiKey` -> fresh plugin metadata/setup lookup.
- Is this the best way to solve the issue? unclear. The fix direction is right, but the static-auth boundary needs to include env/evidence lookup, not only external CLI and synthetic provider auth.
