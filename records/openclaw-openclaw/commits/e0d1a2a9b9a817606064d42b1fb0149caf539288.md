## Summary

Found one concrete prompt-authority regression risk in the Codex app-server path. Focused Codex tests pass, but they assert the new behavior rather than proving the authority boundary is safe.

## Findings

### Medium: Workspace-editable profile files now become Codex developer instructions

- Kind: security
- File: `extensions/codex/src/app-server/run-attempt.ts`
- Line: 4581
- Evidence: `loadWorkspaceBootstrapFiles` loads `SOUL.md`, `TOOLS.md`, `IDENTITY.md`, and `USER.md` from the active workspace, `buildBootstrapContextFiles` forwards their raw/truncated contents, and this commit selects those basenames into `developerInstructionFiles`. `renderCodexWorkspaceDeveloperInstructions` then tells Codex to “Internalize and follow them accordingly,” and `baseDeveloperInstructions` appends that rendered content into the `developerInstructions` sent on `thread/start`/`thread/resume`.
- Impact: Before this commit, these workspace files were turn input reference context and explicitly lower than Codex/OpenClaw developer policy. They now share the developer-instruction channel with OpenClaw runtime guidance, so user/workspace-editable text can conflict with or override message-delivery, tool-use, or safety guidance that used to sit above it.
- Suggested fix: Keep raw workspace files in turn input, or introduce a narrow, sanitized developer-level bridge that only promotes explicitly approved fields from these files and states conflict precedence against OpenClaw runtime guidance.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the request-shape change: create a workspace with `SOUL.md`/`IDENTITY.md`/`TOOLS.md`/`USER.md` and run a Codex app-server attempt; the test and source show the contents appear in `thread/start.developerInstructions`. The downstream model-obedience/security impact is necessarily behavioral.
- Is this the best way to solve the issue? Unclear. If maintainers intentionally want these files at developer priority, the safer shape is still to make the trust-boundary change explicit and constrain what can be promoted.
