## Findings

### Low: Standalone oxlint config now enables `no-underscore-dangle`

- Kind: compatibility
- File: `.oxlintrc.json`
- Line: 27
- Evidence: This commit removes `"eslint/no-underscore-dangle": "off"` while `.oxlintrc.json:4` still enables the full `suspicious` category as errors. With oxlint `1.62.0`, `eslint/no-underscore-dangle` is in `suspicious`; `pnpm exec oxlint --config .oxlintrc.json --print-config test/scripts/oxlint-config.test.ts` resolves it to `deny`. A direct config run on `ui/src/ui/chat/deleted-messages.ts` reports six `eslint(no-underscore-dangle)` errors for `this._keys`; `src/infra/net/undici-global-dispatcher.ts` reports one for `_globalUndiciStreamTimeoutMs`.
- Impact: CI/package lint scripts call repo wrappers that inject `--allow eslint/no-underscore-dangle`, but direct oxlint usage, editor integrations, or any external consumer that relies on `.oxlintrc.json` alone will now flag valid existing repo code.
- Suggested fix: Make the config and wrapper agree. Either keep the explicit `off` rule and require an oxlint version that supports it, or avoid enabling this rule through broad `suspicious` category activation for standalone config consumers.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: run `pnpm exec oxlint --config .oxlintrc.json --type-aware --tsconfig tsconfig.oxlint.json --report-unused-disable-directives-severity error --threads=1 ui/src/ui/chat/deleted-messages.ts`.
- Is this the best way to solve the issue? unclear: the right fix depends on whether cross-version oxlint support or standalone config correctness is the higher-priority contract, but the current state makes those two paths inconsistent.
