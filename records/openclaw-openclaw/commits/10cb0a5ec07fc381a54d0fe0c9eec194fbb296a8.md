# Commit 10cb0a5ec0

## Summary

Found one security regression: the commit restores silent browser-origin Control UI/WebChat pairing before browser-minted device tokens are bound to the shared gateway auth generation.

## Findings

### Medium: Browser auto-pairing can mint durable operator tokens that outlive shared-token rotation

- Kind: security
- File: `src/gateway/server/ws-connection/handshake-auth-helpers.ts`
- Line: 87
- Evidence: The restored guard allows browser-origin Control UI/WebChat clients past the browser-origin block. That feeds `allowSilentLocalPairing` into `requestDevicePairing({ silent })` at `src/gateway/server/ws-connection/message-handler.ts:1077`, and silent requests are immediately approved at `src/gateway/server/ws-connection/message-handler.ts:1164`. Once approved, the handshake issues a device token through `ensureDeviceToken` at `src/gateway/server/ws-connection/message-handler.ts:1420`. The updated regression test now asserts a loopback Control UI browser with a valid gateway token is paired with no pending approval at `src/gateway/server.auth.browser-hardening.test.ts:394`. Existing rotation coverage confirms shared-token sessions are disconnected after rotation, while device-token sessions remain connected at `src/gateway/server.shared-auth-rotation.test.ts:139` and `src/gateway/server.shared-auth-rotation.test.ts:155`.
- Impact: A browser-origin Control UI session that has the shared gateway token can convert it into a durable operator device token without owner approval. Rotating the shared gateway token then does not invalidate that minted device token, so shared-token rotation no longer fully contains browser-side token exposure for this path.
- Suggested fix: Keep browser-origin fresh pairing and role upgrades explicit until browser-origin auto-minted device tokens record the shared-auth generation that created them and are rejected or revoked after gateway token rotation. Add a regression that auto-pairs a browser Control UI device with token A, rotates to token B, then verifies the device token from token A no longer authenticates.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The focused tests prove the two required legs: browser Control UI loopback auto-pairing succeeds, and device-token sessions survive shared-token rotation. Source inspection ties those together through the same silent approval and `ensureDeviceToken` path.
- Is this the best way to solve the issue? yes. The PR body itself calls generation-bound browser device tokens the longer-term fix; keeping explicit browser pairing is the safer interim behavior until that guard exists.
