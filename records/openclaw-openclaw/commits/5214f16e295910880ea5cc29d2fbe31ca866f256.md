# Commit 5214f16

## Summary

Found one high-confidence regression: a failed plugin update can remove the last `plugins.allow` entry, turning a restrictive plugin allowlist into an open plugin policy.

## Findings

### High: Failed plugin update can open a restrictive plugin allowlist

- Kind: security
- File: `src/plugins/update.ts`
- Line: 805
- Evidence: `disablePluginConfigEntry()` removes the failed plugin from `plugins.allow`; `removeDisabledPluginIdFromList()` returns `undefined` when that was the only allowlisted id (`src/plugins/update.ts:771`). Activation policy only enforces allowlist blocking when `allow.length > 0` (`src/plugins/config-activation-shared.ts:226`), while docs define `plugins.allow` as an exclusive allowlist (`docs/tools/plugin.md:153`).
- Impact: If a user has `plugins.allow: ["demo"]` and `demo` fails during an update with `disableOnFailure`, the persisted config becomes equivalent to no allowlist. Previously blocked global installed plugins and bundled default-enabled plugins can then activate on registry refresh or restart. I confirmed with a focused activation probe that a global plugin changes from `disabled: not in allowlist` before this config shape to `enabled: default` after it.
- Suggested fix: Do not collapse the last allowlist entry to `undefined` when disabling after an update failure. Preserve the restrictive allow boundary, or add an explicit fail-closed representation before removing the last allowed id. Add a regression test for `allow: ["failed-plugin"]` plus another installed/default plugin that must remain blocked.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The activation probe using `normalizePluginsConfig()` and `resolvePluginActivationState()` shows `origin: "global"` changing from `reason: "not in allowlist"` to enabled/default when the singleton allowlist is removed.
- Is this the best way to solve the issue? yes: the repair should preserve the allowlist’s restrictive semantics. Keeping the disabled plugin entry alone is not enough because `entries.<id>.enabled: false` only blocks that one id.
