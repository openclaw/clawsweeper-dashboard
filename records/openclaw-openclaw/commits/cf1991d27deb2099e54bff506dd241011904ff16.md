# Commit cf1991d27d

## Summary

Found one concrete sudo carrier parsing gap that remains on current `main`. The new sudo env-assignment stripping handles `sudo VAR=value command`, but not the valid sudo form `sudo VAR=value -- command`.

## Findings

### Medium: sudo env assignments before `--` hide the real carried command

- Kind: security
- File: `src/infra/command-carriers.ts`
- Line: 340
- Evidence: `resolveSudoLikeCarriedArgv()` returns immediately when it sees the first non-option token and calls `stripSudoEnvAssignmentsFromCommandArgv()`. For `["sudo","OPENCLAW_ENV=1","--","/approve","abc","allow-once"]`, the helper strips only the env assignment and returns `["--","/approve","abc","allow-once"]`. `buildCommandPayloadCandidates()` then emits `"-- /approve abc allow-once"`, so `parseExecApprovalShellCommand()` in `src/agents/bash-tools.exec.ts:1156` does not reject it. Focused check output: `["sudo","OPENCLAW_ENV=1","--","/approve","abc","allow-once"] => ["-- /approve abc allow-once"] inline= null`.
- Impact: sudo itself accepts this syntax and runs the command after `--`; local sudo 1.9.15p5 executed `sudo -n FOO=bar -- python3 -c 'print(123)'` as Python and printed `123`. That means the exec unsafe-command guard can miss restricted commands such as `sudo OPENCLAW_ENV=1 -- /approve abc allow-once` and `sudo OPENCLAW_ENV=1 -- openclaw channels login`, and inline-eval risk detection also misses `sudo OPENCLAW_ENV=1 -- python3 -c ...`.
- Suggested fix: parse sudo env assignments in the sudo scanner instead of stripping them as a post-process. If env assignments appear before the option terminator, skip them, then consume a following `--` separator before returning the carried command. Preserve the opposite case: `sudo -- VAR=value command` should not strip `VAR=value`, because sudo treats it as the command after an explicit option terminator.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Run `pnpm exec tsx -e "import { buildCommandPayloadCandidates, detectInlineEvalArgv } from './src/infra/command-analysis/risks.ts'; console.log(buildCommandPayloadCandidates(['sudo','OPENCLAW_ENV=1','--','/approve','abc','allow-once'])); console.log(detectInlineEvalArgv(['sudo','OPENCLAW_ENV=1','--','python3','-c','print(1)']));"` and compare with `sudo -n FOO=bar -- python3 -c 'print(123)'`.
- Is this the best way to solve the issue? yes. The parser needs to model sudo’s boundary: env assignments before `--` are sudo env options, while env-looking tokens after an already-seen `--` are the command.
