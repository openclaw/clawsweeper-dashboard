# Commit ca093d8

## Summary

Found one regression in the runtime dependency staging cache. The new fast fingerprint can miss changed installed dependency contents and reuse stale `dist/extensions/*/node_modules`.

## Findings

### Medium: Runtime dependency staging reuses stale copied dependencies

- Kind: regression
- File: `scripts/stage-bundled-plugin-runtime-deps.mjs`
- Line: 492
- Evidence: This commit removed the previous recursive content fingerprint and now hashes only `depName`, installed version, `package.json` mtime/size, and real root at `scripts/stage-bundled-plugin-runtime-deps.mjs:492`. The skip path at `scripts/stage-bundled-plugin-runtime-deps.mjs:1281` reuses existing staged `node_modules` when that fingerprint matches. `scripts/tsdown-build.mjs:146` intentionally preserves staged plugin `node_modules` across rebuild cleanup, so stale reuse is reachable in normal incremental/package builds. The existing regression test confirms it: `pnpm test test/scripts/stage-bundled-plugin-runtime-deps.test.ts` fails because after changing the installed dependency’s `index.js`, the second staging pass still leaves the copied file as `module.exports = 'first';` instead of `second` at `test/scripts/stage-bundled-plugin-runtime-deps.test.ts:574`.
- Impact: Package/build outputs can ship stale bundled plugin runtime dependencies when installed package contents change without changing the dependency package.json metadata. That includes local rebuilds, postinstall-produced files, patched or regenerated dependency files, and any cache/workspace reuse where `dist/extensions/<plugin>/node_modules` survives.
- Suggested fix: Restore a content-sensitive fingerprint for the selected dependency roots, or replace it with an equivalent optimized fingerprint that still changes when any copied runtime file changes. Keep the existing `restages when installed root runtime dependency contents change` test green.
- Confidence: high
