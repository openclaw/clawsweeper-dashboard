# Commit fe121632

## Summary

Found one chat picker search race that can still apply stale results after the user changes the input back to the already-applied query.

## Findings

### Medium: In-flight search can overwrite results after reverting to the applied query

- Kind: bug
- File: `ui/src/ui/chat/session-controls.ts`
- Line: 385
- Evidence: `updateChatSessionPickerSearchQuery` only calls `invalidateChatSessionPickerSearchRequests` when the normalized input differs from `chatSessionPickerAppliedQuery` or there is no current result. If a user has applied `telegram`, starts an async search for `telegramp`, then edits the input back to `telegram` before the `telegramp` response returns, the `else` branch only clears the timer. The `telegramp` request remains the active request, so `loadChatSessionPickerPage` accepts it at lines 321-327 and sets `chatSessionPickerAppliedQuery = "telegramp"` even though the visible input is back to `telegram`.
- Impact: The picker can show results for a query the user no longer has in the search field, reintroducing stale search state in a common correction path.
- Suggested fix: Invalidate active search requests whenever the input value changes away from the active request query, even when it equals the already-applied query. A second guard before applying results should also compare the request query with the current normalized `chatSessionPickerQuery` for replace searches.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: apply a search for `telegram`, start a slower search for another value such as `telegramp`, edit the input back to `telegram` before the slower response resolves, then resolve the slower response.
- Is this the best way to solve the issue? yes: the controller already owns active request invalidation, so keeping the active request tied to the current input is the narrowest fix.
