# Commit 575936473d

## Summary

Found two low-severity issues. The runtime suppression behavior itself is covered and passed focused tests, but the landed docs still contain contradictory fallback wording, and the new “metadata” debug log includes a snippet of suppressed final text.

## Findings

### Low: Docs still claim message-tool group replies fall back to final text

- Kind: regression
- File: `docs/channels/discord.md:253`, `docs/gateway/configuration-examples.md:113`
- Evidence: These docs say normal Discord/group requests “fall back” to assistant final text when the model misses `message(action=send)`. Current runtime does the opposite: `dispatch-from-config.ts:1636` continues without delivery when `suppressDelivery` is true and the payload is not explicitly marked deliverable, and `dispatch-from-config.test.ts:4933` asserts group/channel finals stay private.
- Impact: Operators following the Discord setup or expanded config example can keep `messages.groupChat.visibleReplies: "message_tool"` expecting fallback posting, then see typing/token usage with no room message.
- Suggested fix: Replace the fallback wording with the strict contract used in `docs/channels/groups.md`: final text stays private, and `"automatic"` is required for legacy final-reply posting.
- Confidence: high

### Low: Suppressed “metadata” log includes private final text preview

- Kind: privacy
- File: `src/auto-reply/reply/dispatch-from-config.ts:153`
- Evidence: `formatSuppressedReplyPayloadForLog` adds `textPreview=${JSON.stringify(textPreview)}` with the first 160 chars of suppressed final text. The changelog and docs describe the new signal as suppressed payload metadata, and the mode’s user-facing contract says normal final text stays private.
- Impact: When verbose console logging or debug file logging is enabled, content that was intentionally not posted to the room is copied into gateway logs and possibly log tail/export surfaces. This is debug-only and local-operator scoped, so severity is low, but it is broader than “metadata.”
- Suggested fix: Log only non-content facts such as text length, media count, rich-part flags, and maybe a hash/run id; or explicitly document that debug logs include a capped content preview.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes. For the docs mismatch, read the cited docs and compare with `dispatch-from-config.ts:1636` plus the focused test at `dispatch-from-config.test.ts:4933`. For the log issue, run a message-tool-only group/channel turn with debug logging and a final text reply that does not call the message tool.
- Is this the best way to solve the issue? yes for the docs mismatch: align the two stale snippets with the strict runtime contract. For the log issue, removing `textPreview` is the safer default unless maintainers intentionally want content previews in debug logs.
