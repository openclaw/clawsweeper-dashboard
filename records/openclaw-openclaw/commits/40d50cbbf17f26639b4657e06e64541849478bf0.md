# Commit 40d50cb

## Summary

Found one low-severity regression in the deadcode guard parser: the new pnpm-progress filter can also hide valid Knip unused-file reports for root-level project files.

## Findings

### Low: Deadcode guard now ignores root-level Knip file findings

- Kind: regression
- File: `scripts/check-deadcode-unused-files.mjs`
- Line: 31
- Evidence: `isLikelyRepoFilePath()` only accepts paths beginning with `apps/`, `docs/`, `extensions/`, `packages/`, `scripts/`, `src/`, `test/`, or `ui/`. But `knip.config.ts` includes root workspace project files via `*.config.{js,mjs,cjs,ts,mts,cts}!` and `*.mjs!`, and `docs/ci.md` documents that `pnpm deadcode:unused-files` should fail when a PR adds a new unreviewed unused file. A direct parser probe confirms `parseKnipCompactUnusedFiles("orphan.config.ts: orphan.config.ts\nopenclaw-extra.mjs: openclaw-extra.mjs\nsrc/a.ts: src/a.ts\n")` returns only `["src/a.ts"]`.
- Impact: a future unused root-level config/module file covered by Knip’s project globs can be reported by Knip but silently dropped by this guard, making `pnpm deadcode:unused-files` pass when it should flag a new unreviewed unused file.
- Suggested fix: filter the known pnpm progress shape directly, or validate against Knip-covered path patterns including root-level project files, instead of using a hard-coded directory-prefix allowlist.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes; import `parseKnipCompactUnusedFiles` and feed it compact files-only output for root-level project files such as `orphan.config.ts: orphan.config.ts`.
- Is this the best way to solve the issue? no; ignoring pnpm progress is reasonable, but narrowing accepted file reports below Knip’s configured project scope creates false negatives.
