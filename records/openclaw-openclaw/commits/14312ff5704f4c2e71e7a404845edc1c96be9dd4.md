# Commit 14312ff

## Summary

Found one low-severity regression in the ordinary media upload failure path.

## Findings

### Low: Ordinary media upload failures duplicate already-sent caption text

- Kind: regression
- File: `extensions/feishu/src/outbound.ts`
- Line: 709
- Evidence: For non-voice media, `sendMedia` sends `text` before uploading media at `extensions/feishu/src/outbound.ts:682`. If `sendMediaFeishu` then throws, the new fallback builds `fallbackText` from both the same `text` and the attachment link at `extensions/feishu/src/outbound.ts:709`, so the user gets the caption once as a normal message and again in the fallback message.
- Impact: A failed ordinary image/file/audio upload now produces duplicate visible reply text. Before this commit, the failure fallback sent only the attachment link after the already-sent caption.
- Suggested fix: Include the original text in the fallback only when it was suppressed for voice media; otherwise keep the fallback to just `📎 ${mediaUrl}`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: call `feishuOutbound.sendMedia` with `text: "caption text"`, a non-voice `mediaUrl`, and a mocked `sendMediaFeishu` rejection. Current control flow sends `"caption text"` first, then sends `"caption text\n\n📎 <url>"` from the catch path.
- Is this the best way to solve the issue? yes: the commit correctly needs text-preserving fallback for suppressed voice replies, but that should be conditional on `suppressTextForVoiceMedia`.
