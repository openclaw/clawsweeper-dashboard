# Commit 5dbc969

## Summary

Found one Telegram presentation regression. The commit preserves valid legacy `interactive` button precedence, but it treats any truthy `payload.interactive` object as authoritative before normalization.

## Findings

### Medium: Empty legacy interactive data suppresses Telegram presentation buttons

- Kind: regression
- File: `extensions/telegram/src/outbound-adapter.ts`
- Line: 213
- Evidence: `renderPresentation` now sets `hasExplicitButtons` when `payload.interactive` is truthy. Core later removes `presentation` from rendered payloads in `src/infra/outbound/deliver.ts:964`, so `sendTelegramPayloadMessages` no longer has the presentation fallback available. `normalizeInteractiveReply` returns `undefined` for `{ blocks: [] }` or malformed empty blocks in `src/interactive/payload.ts:263`, and `resolveTelegramInlineButtons` would otherwise fall through to presentation buttons in `extensions/telegram/src/button-types.ts:135`.
- Impact: A payload with valid `presentation` buttons plus stale/empty legacy `interactive` data sends Telegram fallback text but no inline keyboard. That can break button-driven flows such as approvals or action choices for mixed old/new payload producers.
- Suggested fix: Base the suppression on normalized legacy interactive buttons, not raw truthiness. For example, only skip presentation button materialization when normalized `interactive` actually yields Telegram buttons, while preserving explicit `channelData.telegram.buttons` precedence.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A read-only script showed `no-interactive` sends `[[{"text":"Generic","callback_data":"generic"}]]`, while both `interactive: { blocks: [] }` and malformed empty button blocks send `undefined` buttons after mimicking core’s presentation stripping.
- Is this the best way to solve the issue? yes. The regression is caused by checking raw legacy data instead of the same normalized button contract used by final send resolution.
