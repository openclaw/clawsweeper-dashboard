# Commit bf2711b

## Summary

Found one medium reliability issue: the new limiter caps only the manager’s pre-handoff wrapper, not the actual channel `startAccount` work. For Telegram, the expensive startup probe is inside `startAccount`, so multi-account startup can still fan out all accounts concurrently.

## Findings

### Medium: Account startup limiter releases before `startAccount` does the expensive work

- Kind: reliability
- File: `src/gateway/server-channels.ts`
- Line: 510
- Evidence: `runTasksWithConcurrency` wraps each account at `src/gateway/server-channels.ts:381`, but each wrapper creates `task = Promise.resolve().then(... startAccount(...))` at `src/gateway/server-channels.ts:510`, stores `trackedPromise` at `src/gateway/server-channels.ts:599`, then returns without awaiting that task. That frees the concurrency slot while the channel startup work is still running. The new tests block `isConfigured` and assert that preflight is limited (`src/gateway/server-channels.test.ts:650`), but they do not block `startAccount`. A focused harness with six accounts whose `startAccount` stays pending printed `{"startCalls":["one","two","three","four","five","six"],"maxActive":6,"active":6,"releases":6}` after `await manager.startChannel("discord")`, showing all six active despite the limit of four.
- Impact: The referenced issue reports eight Telegram accounts saturating the Windows event loop during startup. Telegram’s `startAccount` performs the startup probe before starting the monitor (`extensions/telegram/src/channel.ts:879`, `extensions/telegram/src/channel.ts:897`), so this commit can still launch all Telegram probes concurrently and may not fix the Windows multi-bot startup stall it claims to address.
- Suggested fix: Keep the limiter active through the real startup phase, or add a plugin/channel contract that separates “startup completed / monitor handed off” from the long-running monitor promise. For a narrow repair, put a Telegram-owned semaphore around the probe/monitor initialization path so the expensive Telegram startup work is actually staggered.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A read-only `node --import tsx --input-type=module` harness registered six accounts with blocking `startAccount` promises and observed `maxActive: 6`.
- Is this the best way to solve the issue? no. The current implementation limits `isConfigured`/runtime/bootstrap preflight, but the reported Telegram stall is in provider startup work, which remains unbounded.
