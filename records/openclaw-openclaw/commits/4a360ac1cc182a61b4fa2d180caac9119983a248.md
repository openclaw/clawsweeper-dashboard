# Commit 4a360ac

## Summary

Found one doctor migration gap: `openclaw doctor --fix` can still preserve a stale legacy `plugins.installs` path record during first-run registry migration, letting old compiled bundled output continue to shadow the current bundled plugin.

## Findings

### Medium: First-run doctor migration re-persists stale legacy config install records

- Kind: bug
- File: `src/commands/doctor/shared/plugin-registry-migration.ts`
- Line: 296
- Evidence: `maybeRepairPluginRegistryState` only asks `listStaleLocalBundledPluginInstallRecords` to inspect `loadInstalledPluginIndexInstallRecords(params)` from the persisted index at `src/commands/doctor-plugin-registry.ts:168`, then passes filtered durable records into migration. But `migratePluginRegistryForInstall` immediately rebuilds `installRecords` by merging `extractShippedPluginInstallConfigRecords(rawConfig)` back in at `src/commands/doctor/shared/plugin-registry-migration.ts:294-300`. A focused repro with no persisted index and a legacy config `plugins.installs.discord = { source: "path", installPath: old/dist/extensions/discord, version: "2026.5.4-beta.3" }` showed `doctor --fix` printed “Plugin registry rebuilt”, but persisted `discord` with `origin: "global"` and `rootDir` equal to the stale old compiled path. Rebuilding the registry with those persisted records kept `discord` resolving from the stale global path.
- Impact: Users still on the legacy config-backed install-record layout can run the new doctor repair and remain broken after the first migration. The stale old compiled bundled plugin can keep shadowing the current bundled plugin, so the current bundled schemas/runtime do not win until a second doctor run or manual record removal.
- Suggested fix: When doctor repair has stale bundled plugin IDs to remove, filter both durable index records and legacy config-extracted records during migration, or detect stale local bundled shadows from the merged config+index install-record set before `migratePluginRegistryForInstall` writes the first persisted index.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Create current and stale `dist/extensions/discord` plugin dirs with different versions, put only the stale path record in `config.plugins.installs`, run `maybeRepairPluginRegistryState({ shouldRepair: true })` with no existing persisted index, then inspect the persisted index.
- Is this the best way to solve the issue? yes. The pruning contract needs to apply before migration persists any install-record source, including legacy config records that migration intentionally imports.
