# Commit 86a563e

## Summary

Found one tooling regression: the RTT CLI still accepts and records `--provider live-frontier`, but after this commit it runs the new RTT Docker harness, which always starts/configures the mock OpenAI server.

## Findings

### Medium: `--provider live-frontier` now produces mock-backed RTT runs

- Kind: regression
- File: `scripts/lib/rtt-harness.ts`
- Line: 166
- Evidence: `pnpm rtt` still documents and parses `--provider live-frontier` in `scripts/rtt.ts:26` and `test/scripts/rtt-harness.test.ts:147`, and `createHarnessEnv` forwards it as `OPENCLAW_NPM_TELEGRAM_PROVIDER_MODE` in `scripts/lib/rtt-harness.ts:105`. This commit changes `runHarness` to execute `scripts/e2e/npm-telegram-rtt-docker.sh`, but that script unconditionally starts `mock-openai-server.mjs` at `scripts/e2e/npm-telegram-rtt-docker.sh:152`, and the generated config always sets `baseUrl` to `http://127.0.0.1:${mockPort}/v1` in `scripts/e2e/npm-telegram-rtt-config.mjs:41`. The old live harness path passed the provider mode into `runTelegramQaLive` in `scripts/e2e/npm-telegram-live-runner.ts:70`.
- Impact: `pnpm rtt openclaw@latest --provider live-frontier` can report `mode.providerMode: "live-frontier"` while actually measuring a mock OpenAI backend. That can give false release or regression confidence for real-provider Telegram RTT checks.
- Suggested fix: either make `npm-telegram-rtt-docker.sh` honor `OPENCLAW_NPM_TELEGRAM_PROVIDER_MODE=live-frontier` by configuring real provider credentials and skipping the mock server, or reject `live-frontier` in the RTT CLI while this black-box harness is mock-only.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes, by static trace of the `pnpm rtt openclaw@latest --provider live-frontier` path; live execution was not run because Telegram/OpenAI credentials were absent from env and `~/.profile`.
- Is this the best way to solve the issue? Unclear; the safe requirement is to stop silently labeling mock-backed runs as live-frontier. The implementation choice depends on whether RTT is intended to support live providers.
