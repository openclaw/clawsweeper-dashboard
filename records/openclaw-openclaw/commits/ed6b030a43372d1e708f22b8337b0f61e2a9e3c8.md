# Commit ed6b030a43

## Summary

Found one PTY input regression in the new stdin writability tracking. The commit is present on current `main` (`153add18aaaba90457c7dfbee614eb3ec87dc5f0`), and the issue still applies.

## Findings

### Medium: PTY EOF permanently disables further process input

- Kind: regression
- File: `src/process/supervisor/adapters/pty.ts`
- Line: 143
- Evidence: `stdin.end()` for PTY sessions now sets `stdinEnded = true`, making `stdin.writable` false. `process` then rejects all later `write`, `send-keys`, `submit`, and `paste` actions because `isWritableStdin()` treats `writableEnded` as non-writable in `src/agents/bash-tools.process.ts:76`. A live smoke with a background PTY read loop showed: send `first\n` succeeds, send `eof: true` emits `EOF_SEEN` while the process keeps running, a later `process write` returns `Session <id> stdin is not writable`, but directly calling the underlying PTY write still accepts `direct-after-eof\n` and the loop prints `LINE:direct-after-eof`.
- Impact: after an operator sends EOF/Ctrl-D to an interactive PTY process, OpenClaw can no longer recover or continue interacting with that still-running session through the documented process input actions. This regresses the attach/recover workflow this commit is meant to improve.
- Suggested fix: do not mark PTY stdin as ended from `end()`; treat it as “sent EOF key” and keep `writable` true until PTY exit, dispose, or destroy. Keep the ended tracking for real child pipe stdin, where `end()` actually closes the stream.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Start a PTY background command that loops on `read`, call `process write` with `eof: true`, then call `process write` again. The tool rejects the second write even though the PTY and child are still alive and direct PTY input still works.
- Is this the best way to solve the issue? yes. The problematic state transition is PTY-specific; child pipe semantics should continue to mark stdin ended, while PTY EOF should remain a key/control sequence rather than a permanent close.
