# Commit b436927

## Summary

Found one low-severity bug in the new inline image sniffing path: it still accepts common non-image byte payloads when the data URL declares `image/*`, because the MIME detector is called with that declared MIME as a fallback.

## Findings

### Low: Inline data-image sniffing still trusts the spoofed `image/*` header when bytes are unrecognized

- Kind: bug
- File: `extensions/msteams/src/attachments/download.ts`
- Line: 115
- Evidence: `resolveInlineDataImageMime` calls `detectMime({ buffer: inline.data, headerMime: inline.contentType })`, then accepts the result if it starts with `image/`. The local detector falls back to `headerMime` when byte sniffing returns no match (`src/media/mime.ts:212`). A live probe with a Teams HTML attachment containing `<img src="data:image/png;base64,<base64 html>">` returned `{"saved":1,"media":[{"path":"/tmp/x.png","contentType":"image/png","placeholder":"<media:image>"}]}`.
- Impact: The changelog and test intent say non-image inline `data:image/*` payloads should be skipped before staging, but text/HTML or other unrecognized non-image bytes are still saved and propagated as `image/png` through the MS Teams media payload. That can leave spoofed inline attachments agent-visible as images and can still trigger downstream provider/media failures the sniff was meant to prevent.
- Suggested fix: Do the pre-save verification with a bytes-only sniff, or add a detector mode that does not fall back to the declared header. Only fall back to `inline.contentType` for explicitly supported non-magic image formats after separate validation, such as SVG/XML handling; skip unknown bytes instead of accepting the declared data URL MIME.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Run a small `node --import tsx --input-type=module --eval ...` probe that sets the MS Teams runtime, calls `downloadMSTeamsAttachments` with a `text/html` attachment containing a `data:image/png;base64,` payload whose decoded bytes are HTML, and observe `saveMediaBuffer` is called once with `image/png`.
- Is this the best way to solve the issue? yes, directionally. The bug is specifically caused by using the declared `image/*` value as MIME-detection fallback during validation, so the validation path needs a no-header or no-fallback sniff before storage.
