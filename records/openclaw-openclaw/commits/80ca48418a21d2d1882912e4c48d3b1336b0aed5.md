# Commit 80ca484

## Summary

Found one concrete persistence regression in the forced context-engine compaction retry path. The focused tests pass, but they do not assert where the successful retry turn is mirrored after the compaction returns a successor transcript file.

## Findings

### Medium: Retry after successor compaction mirrors the completed turn to the stale transcript

- Kind: bug
- File: `extensions/codex/src/app-server/run-attempt.ts`
- Line: 1950
- Evidence: The overflow retry path adopts successor handles from `contextEngine.compact()` by updating `activeSessionId` and `activeSessionFile`, then restarts Codex with `buildActiveRunAttemptParams()`. After the retry succeeds, `mirrorTranscriptBestEffort()` is still called with the original `params`, and that helper writes to `params.params.sessionFile`, not `activeSessionFile`. Final context-engine turn finalization then reads `activeSessionFile`; if the successor file exists, it uses that stale compacted transcript instead of `historyMessages.concat(result.messagesSnapshot)`, so the just-completed user/assistant turn is not finalized into the active successor transcript.
- Impact: When Codex overflows on a resumed context-engine thread and the owning engine compacts to a successor transcript, the successful retry can split state: the new Codex binding is written beside the successor file, but the completed turn is mirrored into the old file and may be skipped by context-engine `afterTurn`/ingest. The next turn can lose the answer/tool context from the retry that just succeeded.
- Suggested fix: Carry the active post-compaction session handles through completion. Mirror the retry result to `activeSessionFile`, return/update `sessionIdUsed` and `sessionFileUsed` from the active handles, and make finalization include `result.messagesSnapshot` when the successor transcript was not yet mirrored.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. Extend `extensions/codex/src/app-server/run-attempt.context-engine.test.ts`’s “retries a resumed context-engine thread on a fresh Codex thread after early context overflow” case to assert that the successor transcript contains the retry prompt and `"fresh answer"` after `await run`; from the current code path, those mirrored messages are written through `params.sessionFile` instead.
- Is this the best way to solve the issue? Yes. The runtime already tracks `activeSessionId`/`activeSessionFile`; the missing part is using those handles consistently for mirroring, result metadata, and finalization after forced compaction.
