# Commit 97283f0

## Summary

Found one concrete issue in the new CLI response-time contract: the benchmark counts its own RSS marker as first output, so a silent CLI command can satisfy the “first output” contract.

## Findings

### Low: First-output contract accepts benchmark RSS marker as CLI output

- Kind: test_gap
- File: `scripts/bench-cli-startup.ts`
- Line: 596
- Evidence: `buildRssHook` injects an `exit` handler that writes `__OPENCLAW_MAX_RSS_KB__=...` to stderr (`scripts/bench-cli-startup.ts:469`). The new first-output timer calls `markFirstOutput()` for all stderr data (`scripts/bench-cli-startup.ts:596`), and the budget checker treats any finite `summary.firstOutputMs.max` as satisfying the response contract (`scripts/test-cli-startup-bench-budget.mjs:227`). A focused smoke with `--entry /dev/null --case version` produced no real CLI output, but still recorded `firstOutputMs: 28.837994`, `exitCode: 0`, and `node scripts/test-cli-startup-bench-budget.mjs --report /tmp/openclaw-silent-cli-bench.json --skip-baseline` exited 0.
- Impact: The new `test:cli-response:contract` can miss a regression where a covered CLI surface exits quickly but prints nothing, because the benchmark’s synthetic RSS stderr line is counted as user-visible first output.
- Suggested fix: Do not call `markFirstOutput()` for stderr chunks that are only the RSS marker, or move RSS collection out of stdout/stderr used for response timing. A stricter option is to track first non-marker stdout/stderr and keep RSS parsing separate.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: run `node --import tsx scripts/bench-cli-startup.ts --entry /dev/null --case version --runs 1 --warmup 0 --timeout-ms 10000 --output /tmp/openclaw-silent-cli-bench.json --json`, then `node scripts/test-cli-startup-bench-budget.mjs --report /tmp/openclaw-silent-cli-bench.json --skip-baseline`; the budget check passes despite the silent entry.
- Is this the best way to solve the issue? yes: separating benchmark-internal RSS output from measured CLI output directly fixes the false positive without weakening the response budget.
