## Summary

Found one low-severity test compatibility regression. The stricter exact console warning assertion fails when ANSI color is forced in the test environment.

## Findings

### Low: Exact warning assertion breaks under forced color output

- Kind: compatibility
- File: `src/agents/model-fallback.test.ts`
- Line: 1259
- Evidence: The new assertion expects the raw string `[model-fallback] Model "openai/gpt-6" not found...`. `src/logging/subsystem.ts:95` honors `FORCE_COLOR`, and `src/logging/subsystem.ts:259` through `src/logging/subsystem.ts:283` colorizes both the subsystem prefix and warning message before calling `console.warn`. Reproduced with `FORCE_COLOR=1 OPENCLAW_VITEST_MAX_WORKERS=1 pnpm test src/agents/model-fallback.test.ts`; the third warn call contained ANSI-colored `[model-fallback]` and `Model ...`, so `toHaveBeenCalledWith` did not match.
- Impact: Any CI or developer environment that sets `FORCE_COLOR=1` fails this otherwise unrelated agent test file, blocking checks despite correct runtime behavior.
- Suggested fix: Keep the tightened full-message intent but strip ANSI before comparing, or assert against `expect.stringContaining(...)` for the semantic warning. Another option is to force color off in this specific test before the logger formats console output.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: `FORCE_COLOR=1 OPENCLAW_VITEST_MAX_WORKERS=1 pnpm test src/agents/model-fallback.test.ts` fails at `src/agents/model-fallback.test.ts:1259`.
- Is this the best way to solve the issue? yes: the runtime logger legitimately colorizes console output under `FORCE_COLOR`, so the test should normalize captured console strings or avoid exact raw-console matching.
