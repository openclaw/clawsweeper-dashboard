# Commit 84229d9

## Summary

Found one concrete regression: the fix correctly avoids using Claude’s cumulative `result.usage` as the context snapshot, but it also drops that cumulative result usage from the aggregate run-usage path. Downstream code still treats `agentMeta.usage` as cumulative run usage for cost/status snapshots, so tool-heavy Claude CLI turns can now underreport usage and estimated cost.

## Findings

### Medium: Claude CLI aggregate usage is lost when assistant usage exists

- Kind: regression
- File: `src/agents/cli-output.ts`
- Line: 414
- Evidence: `createCliJsonlStreamingParser` and `parseCliJsonl` now skip `result` usage whenever a prior assistant usage exists (`!isClaudeStreamJsonResult(...) || !usage`), so a stream with assistant `cache_read_input_tokens: 125` and final result `cache_read_input_tokens: 300` returns only `125`. The CLI runner then assigns the same `resultParams.output.usage` to both `agentMeta.usage` and `agentMeta.lastCallUsage` in `src/agents/cli-runner.ts:518`, and `session-store` estimates cost from `usage` while documenting that this value is cumulative run usage in `src/agents/command/session-store.ts:221`. Anthropic’s Claude Code cost-tracking docs describe assistant usage as per-step data and the final `result` usage as cumulative/authoritative for total usage and cost. ([docs.anthropic.com](https://docs.anthropic.com/id/docs/claude-code/sdk/sdk-cost-tracking?utm_source=openai))
- Impact: After this commit, multi-step/tool-heavy `claude-cli` runs can record the last assistant API call as the whole run’s `inputTokens`, `outputTokens`, `cacheRead`, `cacheWrite`, and `estimatedCostUsd`. This fixes the compaction snapshot inflation but regresses cost and usage reporting for the same affected workload.
- Suggested fix: Split the parser output into aggregate `usage` and `lastCallUsage`/context usage. For Claude stream-json, keep final `result.usage` as aggregate `usage`, keep the latest deduped assistant `message.usage` as `lastCallUsage`, and update `cli-runner` to pass those separately into `agentMeta`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The new regression fixture itself demonstrates that the parser returns assistant usage instead of final result usage; tracing `cli-runner.ts:518` and `session-store.ts:196-225` shows there is no separate aggregate usage path after parsing.
- Is this the best way to solve the issue? yes. The original fix direction is right for context snapshots, but aggregate run usage and last-call context usage need separate fields.
