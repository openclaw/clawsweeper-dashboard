# Commit 3110c621

## Summary

Found one regression in the new mixed tool/text projection: when a stored assistant message contains commentary text, a tool call, and final-answer text in the same turn, `chat.history` now returns both text blocks without phase metadata. The Control UI then renders the commentary alongside the final answer, violating the existing final-answer preference contract.

## Findings

### Medium: Mixed tool projection revives commentary when a final answer exists

- Kind: regression
- File: `src/gateway/chat-display-projection.ts`
- Line: 180
- Evidence: `projectAssistantTextFromMixedToolContent` collects every nonblank text block from a message with any tool-history block and replaces the content with unphased `{ type: "text", text }` blocks. This bypasses the existing `sanitizeAssistantPhasedContentBlocks` path, which filters phased text to `final_answer`. Existing contracts prefer `final_answer` over `commentary` in `src/shared/chat-message-content.ts` and the UI tests, and `src/agents/openai-ws-stream.test.ts` has a stored-message shape where commentary text, a `toolCall`, and final-answer text share one assistant message. A focused inline check on current `main` projected that shape to `["Working... ", "Done."]`, and `ui/src/ui/chat/message-extract.ts` rendered `Working... \nDone.`.
- Impact: Reloaded Control UI/WebChat history can show progress/commentary text that live chat suppresses, before the final answer, for tool-using OpenAI Responses turns that persist commentary and final answer in one assistant message.
- Suggested fix: Make the mixed-tool projection phase-aware. If any `final_answer` text blocks exist, project only those; otherwise preserve visible text from the mixed tool/commentary case this commit intended to fix. Alternatively keep enough `textSignature` metadata for downstream visible-text extraction to retain the final-answer preference.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Run an inline projection with an assistant message containing `commentary` text, a `toolCall`, and `final_answer` text; current `main` returns both text blocks, and UI extraction renders both.
- Is this the best way to solve the issue? yes. The narrow fix is in the new projection helper because it is the point that strips phase metadata and bypasses the existing final-answer filtering.
