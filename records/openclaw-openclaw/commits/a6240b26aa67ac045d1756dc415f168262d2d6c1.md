# Commit a6240b26

## Summary

Found one concrete MiniMax usage-routing bug: the new base-URL-aware usage fetch still sends env-only/default global MiniMax setups to the CN usage host.

## Findings

### Medium: Default global MiniMax setups still poll the CN usage host

- Kind: bug
- File: `src/infra/provider-usage.fetch.minimax.ts`
- Line: 379
- Evidence: `fetchMinimaxUsage()` falls back to `https://api.minimaxi.com/v1/token_plan/remains` when no explicit `baseUrl` option is passed, while the MiniMax provider catalog defaults chat traffic to `https://api.minimax.io/anthropic` when `MINIMAX_API_HOST` is unset in `extensions/minimax/provider-catalog.ts:13` and `extensions/minimax/provider-catalog.ts:70`. The plugin only passes `models.providers.minimax-portal.baseUrl` or `models.providers.minimax.baseUrl` from saved config in `extensions/minimax/provider-registration.ts:76`, so an env-key/default-provider setup has no config base URL and still uses CN usage. A focused smoke confirmed this mismatch: `buildMinimaxProvider({})` returned `https://api.minimax.io/anthropic`, while `fetchMinimaxUsage(..., undefined)` requested `https://api.minimaxi.com/v1/token_plan/remains`. MiniMax’s current Token Plan FAQ documents separate global/CN usage endpoints for checking Token Plan usage. 
- Impact: users who rely on the built-in MiniMax provider default or env-only `MINIMAX_API_KEY`/Token Plan setup can still see wrong/stale MiniMax quota or CN auth errors in `openclaw status --usage`, despite this commit’s changelog saying global setups no longer query the CN usage host.
- Suggested fix: when no explicit MiniMax provider base URL is saved, derive the usage base URL from the same MiniMax catalog/env helper used for chat provider construction, or pass the resolved provider catalog base URL into the usage snapshot hook. CN fallback should apply only when config/env indicates CN or when preserving truly legacy unknown behavior.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Run a usage snapshot path with no `models.providers.minimax*.baseUrl` and a MiniMax env credential; the provider catalog uses the global chat host, but `fetchMinimaxUsage` requests the CN usage host.
- Is this the best way to solve the issue? yes. Reusing the provider catalog/env host resolution keeps usage routing aligned with the actual MiniMax runtime host and avoids hardcoding a different default.
