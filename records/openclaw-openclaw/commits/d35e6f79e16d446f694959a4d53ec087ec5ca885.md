# Commit d35e6f7

## Summary

Found one narrow compatibility regression in the legacy fallback behavior.

## Findings

### Low: Canonical placeholder entries shadow legacy double-prefixed params

- Kind: regression
- File: `src/agents/pi-embedded-runner/extra-params.ts`
- Line: 79
- Evidence: The new lookup chooses `configuredModels?.[canonicalKey]` before checking `legacyKey`. If a config has both `"openrouter/auto": {}` as the canonical allowlist entry and `"openrouter/openrouter/auto": { params: ... }` as the pre-fix workaround entry, the canonical empty object wins and `globalParams` becomes undefined at line 80. The parent code looked up only `${provider}/${modelId}`, so that same config applied the legacy params.
- Impact: Users who kept the normal canonical allowlist entry and added a double-prefixed params entry to work around the old bug silently lose runtime params after this commit, despite the changelog saying legacy double-prefixed fallback entries are preserved.
- Suggested fix: Resolve fallback at the `params` level, for example prefer `configuredModels?.[canonicalKey]?.params` when present, otherwise fall back to `configuredModels?.[legacyKey]?.params`. Add a regression test where both keys exist and only the legacy entry has params.
- Confidence: high
