# Commit b34cf2f

## Summary

Found one actionable regression in the Matrix externalization path: the new Matrix docs say to install from ClawHub, but the command shown routes through npm today and npm currently resolves to an older Matrix package.

## Findings

### Medium: Matrix install docs route users to stale npm package instead of ClawHub

- Kind: regression
- File: `docs/channels/matrix.md`
- Line: 17
- Evidence: The new Matrix doc says "Install Matrix from ClawHub" but shows `openclaw plugins install @openclaw/matrix`. The installer only handles ClawHub when the spec is explicitly prefixed; `parseClawHubPluginSpec("@openclaw/matrix")` returns `null`, and the scoped package falls through to the npm install path at `src/cli/plugins-install-command.ts:967`. A focused probe confirmed `@openclaw/matrix` does not take the official external pre-npm plan, while `clawhub:@openclaw/matrix` does. `npm view @openclaw/matrix version` currently returns `2026.3.13`; ClawHub package metadata currently reports latest `2026.3.22` and readiness `ready:false`, while this commit marks the source package as built for `2026.5.6` at `extensions/matrix/package.json:88`.
- Impact: Published OpenClaw 2026.5.6 no longer includes `dist/extensions/matrix/**`, so users following the Matrix setup page can install an older external Matrix plugin instead of the intended current externalized plugin. That can drop newer Matrix setup/runtime behavior and make the cutover look broken.
- Suggested fix: Change the Matrix install command to `openclaw plugins install clawhub:@openclaw/matrix` until ClawHub has the current package ready, or update the installer so official external scoped package specs honor `defaultChoice: "clawhub"` before npm, with a regression test for `@openclaw/matrix`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The spec-resolution probe shows `@openclaw/matrix` is not a ClawHub spec and falls through to npm trust/install; npm metadata shows latest `@openclaw/matrix` is `2026.3.13`.
- Is this the best way to solve the issue? yes. Either make the docs use the explicit ClawHub spec they already mention, or make installer behavior match the catalog’s `defaultChoice`.
