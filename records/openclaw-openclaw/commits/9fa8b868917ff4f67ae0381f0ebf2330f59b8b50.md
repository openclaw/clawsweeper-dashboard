# Commit 9fa8b86

## Summary

Found one compatibility regression in the hardened backend fallback path. The change correctly avoids chaining after real decode failures, but it also treats ImageMagick missing-format support as a real processing error, so HEIC conversion can stop before reaching a working `ffmpeg` fallback.

## Findings

### Medium: ImageMagick missing HEIC delegate can block the later ffmpeg fallback

- Kind: compatibility
- File: `src/media/image-ops.ts`
- Line: 225
- Evidence: `convertHeicToJpeg` uses `runWithImageBackends("convertHeicToJpeg")`, whose auto order includes ImageMagick before `ffmpeg`. After this commit, `runWithImageBackends` throws immediately on the first error that `isImageBackendUnavailableCause` does not recognize. The classifier includes Sharp/libvips’ `"support for this compression format has not been built in"` message, but not ImageMagick’s missing delegate failure such as `"no decode delegate for this image format \`HEIC'"`. ImageMagick documents HEIC as requiring the `libheif` delegate, and the ImageMagick issue tracker shows that missing HEIC support is reported with the `no decode delegate ... HEIC` error. ([imagemagick.org](https://imagemagick.org/formats/?utm_source=openai))
- Impact: On a host where Sharp lacks HEIC support, ImageMagick is installed but built without the HEIC delegate, and `ffmpeg` is installed, HEIC inputs can now fail at ImageMagick instead of reaching `ffmpeg`. That regresses the fallback contract for HEIC conversion in `src/media/input-files.ts` and `src/media/web-media.ts`.
- Suggested fix: Treat ImageMagick/GraphicsMagick missing decoder/delegate errors for known, pre-guarded image inputs as backend-unavailable errors, so the chain can continue to later backends. Keep corrupt-image decode errors fail-fast.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? No. The local container does not have `magick`, `convert`, `gm`, or `ffmpeg` available, so I could not live-reproduce the exact mixed-backend host state.
- Is this the best way to solve the issue? Unclear. The safest direction is a narrow availability classification for missing decoder/delegate support, not a broad return to trying every backend after arbitrary processing errors.
