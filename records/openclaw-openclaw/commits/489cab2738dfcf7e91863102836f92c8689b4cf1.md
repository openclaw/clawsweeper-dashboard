# Commit 489cab2

## Summary

Found one dry-run validation gap in the new `config unset --dry-run` support. Nested removals under `secrets.defaults` can change how existing shorthand SecretRefs resolve, but the dry-run path reports success without checking any affected refs.

## Findings

### Medium: Nested secret-default unsets skip affected SecretRef validation

- Kind: bug
- File: `src/cli/config-cli.ts`
- Line: 1045
- Evidence: `touchesSecretDefaults()` only marks `secrets` and `secrets.defaults` as affecting all discovered refs. A path like `secrets.defaults.env` therefore reaches `buildUnsetOperation()` without `touchesAllSecretRefs`, a target path, or a provider alias, so `collectDryRunRefs()` returns an empty ref set. I reproduced this with a temp config where `${WEB_SEARCH_API_KEY}` resolved through `secrets.defaults.env = "vaultenv"`: `node scripts/run-node.mjs config unset secrets.defaults.env --dry-run --json` exited 0 with `"ok": true` and `"refsChecked": 0`. On the same config, unsetting `secrets.defaults` correctly exited 1 with `"refsChecked": 1`.
- Impact: A user can dry-run removal of `secrets.defaults.env`, `secrets.defaults.file`, or `secrets.defaults.exec`, get a successful validation report, then perform the actual unset and break existing shorthand SecretRefs that now fall back to a different provider/default.
- Suggested fix: Treat `secrets.defaults` descendants as all-ref-affecting, not just the collection itself. Add focused tests for `secrets.defaults.env`, `secrets.defaults.file`, and `secrets.defaults.exec` unsets.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: create a temp config with `secrets.defaults.env = "vaultenv"` and a `${WEB_SEARCH_API_KEY}` shorthand SecretRef, then run `OPENCLAW_CONFIG_PATH=<cfg> node scripts/run-node.mjs config unset secrets.defaults.env --dry-run --json`; it reports success with `refsChecked: 0`.
- Is this the best way to solve the issue? yes: the dry-run selection already has the all-ref path for broad defaults; the missing piece is recognizing nested defaults paths as changing global SecretRef resolution.
