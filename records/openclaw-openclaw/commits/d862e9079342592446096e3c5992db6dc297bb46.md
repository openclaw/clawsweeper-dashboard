# Commit d862e90

## Summary

Found one live-test coverage regression: the default Fireworks live sweep now has no catalog-backed candidate, so Fireworks-specific release lanes can silently skip.

## Findings

### Medium: Fireworks live profile lanes now filter out every bundled Fireworks model

- Kind: regression
- File: `src/agents/live-model-filter.ts`
- Line: 29
- Evidence: The commit removes both bundled Fireworks Kimi ids from `HIGH_SIGNAL_LIVE_MODEL_PRIORITY`, leaving only `accounts/fireworks/models/glm-5` and `accounts/fireworks/models/glm-5p1` as accepted Fireworks high-signal ids. But the Fireworks plugin manifest only catalogs the two Kimi rows at `extensions/fireworks/openclaw.plugin.json:31`, and the provider registers `resolveDynamicModel` but no `augmentModelCatalog` at `extensions/fireworks/index.ts:76`. The live suites filter catalog entries through `isHighSignalLiveModelRef` before auth lookup at `src/agents/models.profiles.live.test.ts:777` and `src/gateway/gateway-models.profiles.live.test.ts:2188`; with the current bundled catalog, both Fireworks rows evaluate false. Focused smoke confirmed:
  ```json
  [
    { "id": "accounts/fireworks/models/kimi-k2p6", "highSignal": false },
    { "id": "accounts/fireworks/routers/kimi-k2p5-turbo", "highSignal": false }
  ]
  ```
  The release workflow still has Fireworks-only/default-modern lanes, for example `OPENCLAW_LIVE_GATEWAY_PROVIDERS=fireworks` at `.github/workflows/openclaw-live-and-e2e-checks-reusable.yml:1923` and DeepSeek/Fireworks Docker gateway coverage at `.github/workflows/openclaw-live-and-e2e-checks-reusable.yml:2219`.
- Impact: Fireworks provider regressions can pass release validation because the Fireworks shard reaches “no API keys found; skipping” or only exercises DeepSeek in the combined shard instead of testing Fireworks.
- Suggested fix: Either add the intended GLM Fireworks rows to the Fireworks catalog or `augmentModelCatalog`, or make the Fireworks live workflow pass explicit Fireworks model ids when relying on dynamic models.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Run the Fireworks catalog/filter smoke above or run a default-modern Fireworks live profile with `OPENCLAW_LIVE_GATEWAY_PROVIDERS=fireworks`; the catalog-backed Kimi rows are filtered before auth lookup.
- Is this the best way to solve the issue? yes, if the intent is to keep Fireworks in advisory coverage while dropping off-only Kimi from the high-thinking sweep. The safer fix is to make the intended GLM rows catalog-visible, because both live suites select from discovered catalog entries.
