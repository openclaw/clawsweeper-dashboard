# Commit 524185a

## Summary

Found two security regressions in the realpath trust change. Current `main` still has the relevant source unchanged since this commit, except `CHANGELOG.md`.

## Findings

### High: Allow Always can persist versioned interpreter realpaths as broad executable approvals

- Kind: security
- File: `src/infra/exec-approvals-allowlist.ts`
- Line: 1146
- Evidence: `collectAllowAlwaysPatterns` now derives `candidatePath` with `resolveExecutionTargetTrustPath`, then checks `isInterpreterLikeAllowlistPattern(candidatePath)`. For a common symlink like `python3 -> python3.12`, the candidate becomes the realpath ending in `python3.12`, but `isInterpreterLikeAllowlistPattern` only recognizes fixed names such as `python3`, not versioned interpreter basenames. A production-module probe persisted `["/tmp/.../bin/python3.12"]` for approving `python3 -c "print(1)"`; a later different command `python3 -c "import os; print(os.environ)"` then had `laterAllowlistSatisfied: true` and `laterRequiresApproval: false`.
- Impact: A single Allow Always approval for one interpreter inline-eval command can become a durable approval for arbitrary future commands through that interpreter realpath. The pre-existing guard was meant to fall back to command-specific durability for interpreter-like commands.
- Suggested fix: Preserve interpreter classification against the user-facing executable identity and/or enhance the helper to recognize versioned realpath basenames such as `python3.12`, `node20`, etc. Do not persist unbound path-only Allow Always entries for interpreter-like tools.
- Confidence: high

### High: One symlinked safe bin can trust sibling executables in its realpath target directory

- Kind: security
- File: `src/infra/exec-safe-bin-trust.ts`
- Line: 174
- Evidence: `buildTrustedSafeBinDirs` adds whole `targetDirs` from safe-bin symlink realpaths to the trusted directory set. A probe with a trusted non-writable dir containing only `jq -> /tmp/world-target/jq` produced trusted dirs including `/tmp/world-target`, emitted no warning, and allowed `/tmp/world-target/sort` as a safe bin even though `sort` was not linked from the explicit trusted dir.
- Impact: Trust expands from â€śthis configured symlinked safe binâ€ť to â€śany configured safe-bin name in the symlink target directory.â€ť If the target dir is writable or attacker-controlled, another executable can inherit safe-bin auto-allow without being explicitly trusted.
- Suggested fix: Bind derived trust to exact `(bin name, realpath)` targets instead of whole directories, or at minimum reject/warn on writable derived target dirs and require each trusted executable to be reachable from the explicit trusted dir.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Both findings were reproduced with `tsx` probes using production modules, real temp executables, symlinks, and `evaluateShellAllowlist` / `resolveExecSafeBinRuntimePolicy`.
- Is this the best way to solve the issue? yes for the direction: keep realpath binding, but do not lose interpreter classification and do not turn exact symlink target trust into broad target-directory trust.
