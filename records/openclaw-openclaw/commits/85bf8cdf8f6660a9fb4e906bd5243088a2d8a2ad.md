# Commit 85bf8cdf

## Summary

Found one privacy/user-visible regression. The new delivery hint is inserted before the inbound metadata blocks, but the sanitizers expect metadata to be the leading content. That lets internal prompt text and, in the TUI path, the full metadata block survive stripping.

## Findings

### Medium: Delivery hint before metadata bypasses inbound metadata stripping

- Kind: privacy
- File: `src/auto-reply/reply/inbound-meta.ts`
- Line: 423
- Evidence: The commit prepends `Delivery: to send a message, use the \`message\` tool.` before `Conversation info`. `stripLeadingInboundMetadata` only strips when the first non-empty line is one of `INBOUND_META_SENTINELS`; otherwise it returns the text unchanged (`src/auto-reply/reply/strip-inbound-meta.ts:250`, `src/auto-reply/reply/strip-inbound-meta.ts:265`). TUI user-message rendering calls this helper (`src/tui/tui-formatters.ts:409`). A focused probe with `buildInboundUserContextPrefix(..., { sourceReplyDeliveryMode: "message_tool_only" })` showed `stripLeadingInboundMetadata` returning the delivery line plus the full `Conversation info` JSON, including `chat_id`, `message_id`, and sender.
- Impact: Message-tool-only source turns can expose AI-facing delivery guidance and untrusted metadata in user-visible history surfaces. For WhatsApp/Telegram/etc. this metadata can include chat IDs, message IDs, sender names, and phone-like identifiers.
- Suggested fix: Keep the delivery hint inside a known strip-safe metadata block, add a dedicated sentinel for it to both stripping helpers, or place it after stripped metadata in a way that remains runtime-only and never enters user-visible text. Add regression coverage for both `stripInboundMetadata` and `stripLeadingInboundMetadata` with the new message-tool-only prefix.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Run a probe that builds the new message-tool-only prefix, appends `Actual user message`, then passes it to `stripLeadingInboundMetadata`; the result still contains `Delivery: ...` and the full `Conversation info` / `Sender` blocks.
- Is this the best way to solve the issue? unclear. The smallest fix is to update the strip contract in lockstep with the new prefix, but a safer prompt design would keep the delivery instruction in an already runtime-only/system/tool surface rather than adding free text before strip-sensitive metadata.
