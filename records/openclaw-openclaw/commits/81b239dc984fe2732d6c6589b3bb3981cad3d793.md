# Commit 81b239d

## Summary

Found one packaging regression: the commit externalizes Slack, OpenShell, and Anthropic Vertex in package metadata, but does not opt them out of the bundled root dist build. The published-package `files` excludes are not the same as excluding these plugins from the core build graph.

## Findings

### Medium: Externalized plugins still participate in the core bundled dist graph

- Kind: regression
- File: `extensions/slack/package.json`
- Line: 69
- Evidence: `scripts/lib/bundled-plugin-build-entries.mjs:34` only skips a plugin build entry when `openclaw.build.bundledDist === false`. The earlier externalized Amazon providers have that flag, but the new externalized packages only set `openclaw.build.openclawVersion` at `extensions/anthropic-vertex/package.json:30`, `extensions/openshell/package.json:29`, and `extensions/slack/package.json:69`. A focused build-config probe confirmed the unified root dist graph still includes `extensions/anthropic-vertex/*`, `extensions/openshell/index`, and 8+ `extensions/slack/*` entries, while `amazon-bedrock` and `amazon-bedrock-mantle` are absent.
- Impact: this undercuts the commit’s externalization contract. Root builds still compile these plugin runtimes into the core dist graph, so Slack/OpenShell/Anthropic Vertex code and dependency imports can continue to affect core build output, package size, chunking, and dependency verification even though root `package.json` removed their runtime dependencies. It also leaves source/Docker dist behavior different from the intended “installed plugin roots only” runtime path.
- Suggested fix: add `"bundledDist": false` under `openclaw.build` for `anthropic-vertex`, `openshell`, and `slack`, and add a regression assertion beside the existing Amazon coverage in `test/scripts/bundled-plugin-build-entries.test.ts` or `src/infra/tsdown-config.test.ts`.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes: import `tsdown.config.ts` with `node --import tsx` and inspect unified graph keys. The probe showed Amazon externalized providers as `<none>`, but showed root graph entries for `anthropic-vertex`, `openshell`, and `slack`.
- Is this the best way to solve the issue? yes: the local build-entry contract already has the `openclaw.build.bundledDist: false` mechanism, and the prior Amazon externalization uses it.
