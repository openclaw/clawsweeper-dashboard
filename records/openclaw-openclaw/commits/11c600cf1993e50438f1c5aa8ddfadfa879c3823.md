# Commit 11c600c

## Summary

Found two compatibility regressions in the Google Meet realtime provider split.

## Findings

### Medium: Legacy agent-mode STT providers are ignored

- Kind: regression
- File: `extensions/google-meet/src/config.ts`
- Line: 542
- Evidence: `resolveGoogleMeetConfigWithEnv()` now always materializes `realtime.transcriptionProvider` to the default `"openai"` when the scoped field is unset. The new STT resolver then chooses `params.config.realtime.transcriptionProvider ?? params.config.realtime.provider` in `extensions/google-meet/src/realtime.ts:380`, so a legacy config such as `realtime.provider: "deepgram"` no longer reaches the Deepgram realtime transcription provider. In the parent commit, the STT resolver used `params.config.realtime.provider` directly.
- Impact: Existing Google Meet agent-mode users who selected another bundled or third-party realtime transcription provider through the old `realtime.provider` field are silently moved back to OpenAI STT. This contradicts the new docs/manifest text that says `realtime.provider` remains the compatibility fallback when scoped fields are unset.
- Suggested fix: Preserve whether `transcriptionProvider` was explicitly configured. For example, only default `transcriptionProvider` to `"openai"` when neither `realtime.transcriptionProvider` nor legacy `realtime.provider` is present, so legacy provider IDs still fall through.
- Confidence: high

### Low: The Google Meet doctor migration is not wired into `doctor --fix`

- Kind: regression
- File: `extensions/google-meet/doctor-contract-api.ts`
- Line: 1
- Evidence: The commit exports `normalizeCompatibilityConfig`, and `extensions/google-meet/src/config-compat.ts:40` implements the legacy Gemini migration. But the doctor fix path in `src/commands/doctor/shared/legacy-config-core-migrate.ts:16` only invokes `runPluginSetupConfigMigrations()` plus channel migrations. `runPluginSetupConfigMigrations()` only runs migrations registered by a plugin setup API (`src/plugins/setup-registry.ts:677`), and `extensions/google-meet` has no `setup-api.ts`.
- Impact: `doctor --fix` can report the legacy Google Meet provider shape via the new rule, but it will not apply the advertised migration. Users keep getting the legacy warning and the changelog promise is not fulfilled.
- Suggested fix: Either add a Google Meet `setup-api.ts` that registers `migrateGoogleMeetLegacyRealtimeProvider`, or wire plugin doctor-contract normalizers into the generic doctor compatibility normalization path for non-channel plugins.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. For the first issue, resolve a Google Meet config with `realtime.provider: "deepgram"` and no scoped `transcriptionProvider`; the resolved config selects OpenAI for STT. For the second, run the doctor config normalization path with `plugins.entries.google-meet.config.realtime.provider: "google"`; the direct normalizer works, but the doctor fix pipeline does not call it.
