## Findings

### Low: Codex harness docs still promise per-agent HOME isolation

- Kind: regression
- File: `docs/plugins/codex-harness-reference.md`
- Line: 170
- Evidence: `extensions/codex/src/app-server/auth-bridge.ts:263` now leaves `nativeHome` undefined unless `startOptions.env.HOME` is explicitly provided, and `auth-bridge.ts:274` only injects `CODEX_HOME` by default. The updated test at `extensions/codex/src/app-server/auth-bridge.test.ts:175` asserts default launches do not override `HOME`. However, `docs/plugins/codex-harness-reference.md:170` still says OpenClaw sets both `CODEX_HOME` and `HOME` to per-agent directories, and `docs/plugins/codex-harness.md:483` still says `HOME` is reserved for per-agent Codex isolation.
- Impact: Users reading the Codex harness docs can make the wrong security/privacy assumption: subprocesses and anything resolving `$HOME` in a local Codex app-server launch now see the inherited process home by default, not the per-agent synthetic home described by those docs.
- Suggested fix: Update the Codex harness docs and reference docs to match `docs/cli/migrate.md`: per-agent isolation is through `CODEX_HOME`; `HOME` is inherited unless an internal launch explicitly overrides it; `appServer.clearEnv` does not clear `HOME` or `CODEX_HOME` after the bridge reserves them.
- Confidence: high
