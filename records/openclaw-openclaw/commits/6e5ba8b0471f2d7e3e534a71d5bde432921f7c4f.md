# Commit 6e5ba8b

## Summary

Found one config-surface regression: the new `channels.discord.voice.captureSilenceGraceMs` option was added to source schema/docs, but the generated bundled channel metadata was not regenerated. That makes repo guards fail and causes bundled-channel validation/schema consumers to reject or omit the new key.

## Findings

### Medium: New Discord voice config key is rejected by generated channel metadata

- Kind: regression
- File: `src/config/bundled-channel-config-metadata.generated.ts`
- Line: 10
- Evidence: `src/config/zod-schema.providers-core.ts` now accepts `captureSilenceGraceMs`, and `extensions/discord/src/config-ui-hints.ts` documents a UI hint, but generated Discord channel metadata still lacks both the voice schema property and `voice.captureSilenceGraceMs` UI hint. `pnpm -s config:channels:check` fails with stale generated output. A focused validation probe returned `channels.discord.voice: invalid config: must NOT have additional properties: "captureSilenceGraceMs"`.
- Impact: users following the new docs cannot reliably set the new knob through config tooling; config dry-run/schema validation paths that use bundled channel metadata reject the documented key, and Control UI/config schema metadata omits it.
- Suggested fix: run and commit the generated channel metadata update, and update generated config docs baseline as required by `pnpm config:docs:gen`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: `pnpm -s config:channels:check` fails, and `validateConfigObjectRaw({ channels: { discord: { groupPolicy: "allowlist", voice: { captureSilenceGraceMs: 3500 } } } }, { validateBundledChannels: true })` rejects the new key.
- Is this the best way to solve the issue? yes: this is generated drift from a public config-surface change; regenerating the bundled channel metadata and docs baseline aligns source schema, runtime metadata, and docs.
