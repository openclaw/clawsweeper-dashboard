Found one low-severity compatibility issue in the new no-pnpm UI build path. The test-only changes otherwise matched the current production behavior and adjacent contracts I inspected.

## Findings

### Low: `OPENCLAW_BUILD_ALL_NO_PNPM` UI build still requires `pnpm`

- Kind: compatibility
- File: `scripts/ui.js`
- Line: 171
- Evidence: `scripts/package-openclaw-for-docker.mjs:139` now invokes `node scripts/ui.js build` with `OPENCLAW_BUILD_ALL_NO_PNPM=1`, and `scripts/ui.js:189`/`scripts/ui.js:202` adds a direct `node node_modules/vite/bin/vite.js build` fallback. But `scripts/ui.js:171` resolves the pnpm runner and exits on a missing runner before either no-pnpm branch can run. I reproduced this after deps were installed by running the UI build help path with `PATH` stripped of the pnpm shim and `OPENCLAW_BUILD_ALL_NO_PNPM=1`; it failed with `Missing UI runner: install pnpm, then retry.` Directly invoking Vite under the same no-pnpm PATH returned Vite help successfully.
- Impact: prerelease/Docker package validation from a checkout with dependencies present but without `pnpm` on `PATH` still fails, even though this commit explicitly routes the package UI build through the no-pnpm mode.
- Suggested fix: handle `OPENCLAW_BUILD_ALL_NO_PNPM=1 && action === "build"` before resolving `pnpm`, or lazily resolve the runner only for actions/branches that actually need it. Add a focused `scripts/ui.js` test for no-pnpm build dispatch with no runner.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The focused smoke strips `pnpm` from `PATH`, sets `OPENCLAW_BUILD_ALL_NO_PNPM=1`, and runs `node scripts/ui.js build --help`; it exits before the direct Vite path with `Missing UI runner: install pnpm, then retry.`
- Is this the best way to solve the issue? yes. The failure is ordering, not Vite itself; direct Vite invocation works once dependencies are installed.
