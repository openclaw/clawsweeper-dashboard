# Commit d0034e2f9

## Summary

Found one low-severity security hardening regression: the new Graph guard wraps `globalThis.fetch` in a custom `fetchImpl`, which makes the SSRF guard treat the fetch as an explicit dispatcher-capable implementation instead of using its runtime dispatcher fallback.

## Findings

### Low: Graph guard wrapper can bypass the guard's pinned dispatcher path

- Kind: security
- File: `extensions/msteams/src/graph.ts`
- Line: 53
- Evidence: `requestGraph` passes `fetchImpl: async (...) => currentFetch(...)`. In `src/infra/net/fetch-guard.ts:460`, any explicit `fetchImpl` that is not exactly `globalThis.fetch` is treated as dispatcher-capable, so `shouldUseRuntimeFetch` becomes false. The guard’s own comments at `src/infra/net/fetch-guard.ts:467` say the runtime fallback is needed because ambient global fetch may not honor per-request dispatchers, and the guard tests at `src/infra/net/fetch-guard.ssrf.test.ts:392` and `src/infra/net/fetch-guard.ssrf.test.ts:535` assert that ambient global fetch must not be used when a pinned dispatcher is attached.
- Impact: The request still gets URL parsing, hostname policy checks, and manual redirect handling, but the DNS-pinned dispatcher that closes the check/use gap can be skipped for normal MS Teams Graph helper calls. The immediate target is the fixed Graph origin, so this is low severity, but it weakens the security property this commit is trying to add.
- Suggested fix: Omit `fetchImpl` here and let `fetchWithSsrFGuard` choose the default/runtime fetch path. Test-installed `globalThis.fetch` mocks are already handled by `isMockedFetch`, and passing no wrapper preserves the guard’s pinned dispatcher behavior in production.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the control-flow issue: the changed wrapper is an explicit function, and the guard source selects explicit non-ambient fetch implementations instead of the runtime dispatcher fallback. I did not run a live DNS-rebinding proof.
- Is this the best way to solve the issue? Yes: removing the wrapper is the narrowest fix and aligns with the guard’s existing ambient-fetch and mocked-fetch detection.
