# Commit 5e3265b09b

## Summary

Found one low-severity edge-case regression in the new `extraBody` collision behavior: overriding `response_format` through `extraBody` can make OpenClaw label and deliver the returned audio as the wrong format.

## Findings

### Low: `extraBody.response_format` can desynchronize the request format from returned TTS metadata

- Kind: bug
- File: `extensions/openai/speech-provider.ts`
- Line: 318
- Evidence: `openaiTTS` serializes generated fields and then spreads sanitized `extraBody` after them, so `extraBody.response_format` can overwrite the actual JSON request format sent to `/audio/speech` (`extensions/openai/tts.ts:138`). The provider still returns `outputFormat`, `fileExtension`, and `voiceCompatible` from the pre-merge `responseFormat` variable (`extensions/openai/speech-provider.ts:316`). The TTS runtime then writes the buffer using that stale extension (`extensions/speech-core/src/tts.ts:1108`). The added test also confirms collision precedence by allowing `extraBody.speed` to overwrite generated `speed` (`extensions/openai/tts.test.ts:188`).
- Impact: A config such as `extraBody: { response_format: "wav" }` with default file TTS would send a WAV request but return/write it as `.mp3`. Voice-note or telephony paths can similarly report compatibility/sample metadata for the generated format while the server returned another format, causing playback or delivery failures.
- Suggested fix: Treat local metadata-affecting keys like `response_format` as reserved, or derive `outputFormat`, `fileExtension`, and voice compatibility from the final sanitized request body after collision handling. Telephony should probably reject or ignore `response_format` overrides because it assumes PCM/24 kHz.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes: configure OpenAI-compatible TTS with `responseFormat: "mp3"` or default file output and `extraBody: { response_format: "wav" }`; code inspection shows the request body uses `wav` while the provider result remains `mp3`/`.mp3`.
- Is this the best way to solve the issue? unclear: blocking reserved collisions is safer for delivery metadata, while deriving metadata from the final body preserves the documented “extraBody after generated fields” precedence.
