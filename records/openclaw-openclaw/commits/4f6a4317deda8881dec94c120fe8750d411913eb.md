# Commit 4f6a431

## Summary

Found one low-severity setup diagnostic bug in the new Twilio dial-plan check.

## Findings

### Low: Twilio setup can report an invalid default dial-in number as ready

- Kind: bug
- File: `extensions/google-meet/src/setup.ts`
- Line: 200
- Evidence: `getGoogleMeetSetupStatus` treats `config.twilio.defaultDialInNumber` as valid with `Boolean(config.twilio.defaultDialInNumber)` and then reports `"Twilio default Meet dial-in number is configured"` with `ok: true`. The actual join path validates the same default through `normalizeDialInNumber` in `extensions/google-meet/src/runtime.ts:448`, and that helper throws unless the compacted value matches `^\+?[0-9]{5,20}$` in `extensions/google-meet/src/transports/twilio.ts:10`.
- Impact: a user who accidentally configures `twilio.defaultDialInNumber` as a Meet URL, meeting code, or other non-phone string gets a green `twilio-dial-plan` setup check, but the subsequent Twilio join fails before dialing. That undercuts the commit’s goal of making setup_status a blocker agents can trust before joining.
- Suggested fix: validate the configured default with the same `normalizeDialInNumber` helper used by `join`, but convert validation failure into a failed `twilio-dial-plan` setup check instead of throwing. Add a regression test beside the new missing/request-provided dial-plan tests for an invalid configured default.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: configure `twilio.defaultDialInNumber` to a non-empty non-phone value, call `setup_status` with `transport: "twilio"`, then call `join` without request `dialInNumber`. The setup check goes green from `Boolean(...)`; join rejects through `normalizeDialInNumber`.
- Is this the best way to solve the issue? yes: the setup check should share the join path’s phone-number validation so preflight and runtime agree.
