# Commit 8612af7

## Summary

Found one regression in the new default context handling for thread-bound native subagent sessions: cross-agent thread spawns now fail unless the caller explicitly overrides `context: "isolated"`.

## Findings

### Medium: Cross-agent thread-bound subagent spawns default to an impossible fork mode

- Kind: regression
- File: `src/agents/subagent-spawn.ts`
- Line: 565
- Evidence: `resolveSubagentContextMode()` now returns the channel thread-binding policy default for any `thread=true` spawn, and `resolveThreadBindingSpawnPolicy()` defaults `defaultSpawnContext` to `"fork"`. That context is computed before target-agent compatibility is considered. Later, `prepareSubagentSessionContext()` rejects forked context when `targetAgentId !== requesterAgentId` with `context="fork" currently requires the same target agent as the requester; use context="isolated" for cross-agent spawns.` (`src/agents/subagent-spawn.ts:356`).
- Impact: a previously valid cross-agent native subagent spawn such as `sessions_spawn({ agentId: "ops", thread: true })` from a thread-capable channel now fails by default even when `agents.*.subagents.allowAgents` permits the target. Cross-agent non-thread spawns remain covered by tests, but the new thread-bound default path is not covered.
- Suggested fix: when `params.context` is omitted and `targetAgentId !== requesterAgentId`, force `contextMode` to `"isolated"` or defer default context resolution until after the target agent is known. Keep explicit `context: "fork"` rejected for cross-agent spawns.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: configure a requester agent with `subagents.allowAgents` allowing another agent, invoke native `sessions_spawn` with `agentId` set to that other agent and `thread: true` from a Discord/Matrix thread-capable channel, and omit `context`. The code resolves omitted context to `"fork"` and then rejects cross-agent fork preparation before thread binding or agent start.
- Is this the best way to solve the issue? yes: preserving explicit `fork` rejection is correct, but the inferred default should not select a mode the implementation already knows cannot work for cross-agent targets.
