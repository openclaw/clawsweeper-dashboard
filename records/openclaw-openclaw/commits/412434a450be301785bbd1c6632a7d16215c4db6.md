# Commit 412434a

## Summary

Found one test-harness regression: the new ClawHub install-boundary assertion does not match the default `plugins` Docker fixture package, so the default `test:docker:plugins` ClawHub block should fail.

## Findings

### Medium: ClawHub fixture cannot satisfy the new external install contract assertion

- Kind: regression
- File: `scripts/e2e/lib/plugins/assertions.mjs`
- Line: 357
- Evidence: `assertClawHubInstalled()` now always calls `assertClawHubExternalInstallContract()`, which requires `node_modules/openclaw` to be a symlink and `node_modules/is-number/package.json` to exist. The generic plugins lane starts the local fixture with the `plugins` profile (`scripts/e2e/lib/plugins/clawhub.sh:16`), but that fixture package only declares `name`, `version`, and `openclaw.extensions` (`scripts/e2e/lib/clawhub-fixture-server.cjs:146`); it has no `dependencies.is-number` and no `peerDependencies.openclaw`. The installer only links the OpenClaw peer when `peerDependencies` contains `openclaw` (`src/plugins/install.ts:546`) and only installs declared dependencies/optional dependencies (`src/plugins/install.ts:734`).
- Impact: default `pnpm test:docker:plugins` and the plugin prerelease `plugins` Docker lane can fail after a successful ClawHub install, blocking release/pre-release validation instead of proving the intended boundary.
- Suggested fix: update the `plugins` fixture package to declare the dependency and OpenClaw peer that the assertion requires, or keep this assertion gated to a fixture/package that actually declares that contract.
- Confidence: high
