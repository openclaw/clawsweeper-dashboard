# Commit c5071a8061

## Summary

Found one actionable privacy regression in the new RequestError details path.

## Findings

### Medium: RequestError details bypass always-on UI redaction when log redaction is disabled

- Kind: privacy
- File: `src/acp/runtime/errors.ts`
- Line: 59
- Evidence: `readAcpRequestErrorDetails()` redacts appended `data.details` with `redactSensitiveText(...)`, which honors `logging.redactSensitive: "off"`. The resulting `AcpRuntimeError.message` is passed to user-facing ACP replies via `formatAcpRuntimeErrorText()` in `src/auto-reply/reply/dispatch-acp.ts:579` and ACP command text through `src/auto-reply/reply/commands-acp/shared.ts:515`. Repo docs say `redactSensitive: "off"` only disables general log/transcript policy and that UI/tool/diagnostic safety surfaces still redact secrets before emission (`docs/gateway/configuration-reference.md:1005`); `redactToolPayloadText()` is the helper that forces tools-mode regardless of logging config (`src/logging/redact.ts:230`).
- Impact: A user with log redaction disabled can now have secret-shaped backend `RequestError.data.details` surfaced into ACP chat/channel error messages. Before this commit, those details were not appended to the runtime message at all.
- Suggested fix: Use `redactToolPayloadText()` or `redactSensitiveText(..., { mode: "tools" })` for RequestError details that feed ACP user-visible text, and add a regression test with `logging.redactSensitive: "off"`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused `tsx` probe with a temp `OPENCLAW_CONFIG_PATH` containing `{"logging":{"redactSensitive":"off"}}` produced `leak_detected` for a numeric `RequestError` with secret-shaped `data.details`.
- Is this the best way to solve the issue? yes. The repo already has a purpose-built helper for UI/tool safety surfaces that ignores the log redaction off switch.
