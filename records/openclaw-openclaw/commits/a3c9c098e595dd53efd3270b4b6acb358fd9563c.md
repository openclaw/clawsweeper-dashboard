# Commit a3c9c09

## Summary

Found one low-severity reliability regression in the new live-active queued typing path.

## Findings

### Low: Queued typing controllers can be started without any later cleanup owner

- Kind: reliability
- File: `src/auto-reply/reply/agent-runner.ts`
- Line: 1038
- Evidence: The enqueue-followup branch ignores the boolean returned by `enqueueFollowupRun()` and, whenever `isRunActive?.() === true`, starts the current message's typing controller via `typingSignals.signalToolStart()` without calling `typing.cleanup()`. But `enqueueFollowupRun()` can return `false` before registering the followup callback for duplicate message ids, existing queued duplicates, or queue drop policy. It also stores only one drain callback per queue key, and each later successful enqueue overwrites the previous callback. The followup runner's final cleanup only marks the typing controller captured by the callback that actually drains, so earlier started controllers are left to the 2-minute TTL instead of being stopped when the queued work resolves.
- Impact: A redelivered/dropped followup, or the first of multiple followups queued behind the same active run, can keep sending typing keepalives after it is no longer associated with any queued turn. On Discord and similar channels this can leave a stale typing indicator for up to the TTL after the active run/drain has completed, partially trading the original "looks idle" bug for "looks busy after work is done" in duplicate and multi-message queue cases.
- Suggested fix: Capture the return value from `enqueueFollowupRun()` and clean up immediately when nothing was queued. For successful enqueues, either attach cleanup ownership to each queued item/typing controller, or clean up/transfer the previous queued typing controller when the per-queue drain callback is overwritten so every started controller has a deterministic stop path.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Source-level repro: while `isRunActive()` returns `true`, call `runReplyAgent` twice for the same queue before the active run drains. Both calls start their own typing controllers, but `queue/drain.ts` keeps only the latest callback for the queue key, so only the latest controller is cleaned in `followup-runner.ts`. A duplicate message-id path is even simpler: make `enqueueFollowupRun()` return `false`; the branch still starts typing and no queued callback exists to clean it.
- Is this the best way to solve the issue? unclear. The current fix is correct for the single successfully queued followup case from #65685, but the cleanup ownership needs to follow actual queue acceptance and callback replacement semantics.
