# Commit 8f17356392

## Summary

Found one current reliability gap in the Kimi Anthropic thinking path. Current `main` already contains a follow-up budget hardening commit, but the sanitizer bypass still misses Kimi thinking that is enabled through provider `extraParams`.

## Findings

### Medium: Kimi thinking enabled by extraParams still uses the SSE sanitizer that this commit bypasses for thinking streams

- Kind: reliability
- File: `src/agents/anthropic-transport-stream.ts`
- Line: 658
- Evidence: `createAnthropicTransportClient` chooses `buildGuardedModelFetch(model, undefined, { sanitizeSse: false })` only when `options?.thinkingEnabled === true`, and the adjacent comment says the sanitizer can stall before the text block for Kimi Anthropic thinking SSE. But `resolveAnthropicTransportOptions` sets `thinkingEnabled = false` whenever raw stream options have no `reasoning`. The Kimi plugin independently receives `extraParams` and can inject `payload.thinking = { type: "enabled", ... }` later via `onPayload`; that `onPayload` is invoked only after the client/fetch has already been created.
- Impact: A valid Kimi configuration such as model params `thinking: { type: "enabled" }` with the session thinking level left off can still send an Anthropic thinking request, but it will use `buildGuardedModelFetch(model)` instead of disabling SSE sanitization. That leaves this supported thinking path exposed to the exact stream stall this commit was intended to avoid.
- Suggested fix: Base the Kimi sanitizer bypass on the final request payload after provider `onPayload` patches, or let the Kimi wrapper set an internal transport flag when it injects enabled thinking. The decision should not depend only on raw `options.reasoning`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Add a focused variant of the existing Kimi sanitizer test where `reasoning` is omitted and `onPayload` injects `thinking: { type: "enabled", budget_tokens: 1024 }`; current code will call `buildGuardedModelFetch(model)` because `thinkingEnabled` was computed before `onPayload`.
- Is this the best way to solve the issue? yes. The transport decision needs to reflect the final payload, since provider wrappers are explicitly allowed to mutate the payload after generic transport options are resolved.
