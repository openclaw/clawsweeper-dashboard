# Commit 5b1c2ee

## Summary

Found one concrete reliability issue in the new Slack user-group mention resolver: one inbound message can trigger an unbounded sequence of Slack Web API membership lookups before the message is gated.

## Findings

### Medium: Unbounded subteam token expansion can rate-limit Slack handling

- Kind: reliability
- File: `extensions/slack/src/monitor/message-handler/subteam-mentions.ts`
- Line: 90
- Evidence: `extractSlackSubteamMentionIds()` returns every unique `<!subteam^...>` token from untrusted `message.text`, and `isSlackSubteamMentionForBot()` then awaits `readSlackSubteamUsers()` once per ID in a sequential loop. `readSlackSubteamUsers()` calls `params.client.usergroups.users.list()` for every uncached ID. The caller in `prepare.ts` invokes this path for any message containing a subteam token before mention gating finishes. Slack documents `usergroups.users.list` as a Web API method with a required `usergroup` argument and a Tier 2 rate limit, so many distinct fake or irrelevant subteam IDs in one message can burn Slack API calls and delay the Slack event handler. ([docs.slack.dev](https://docs.slack.dev/reference/methods/usergroups.users.list))
- Impact: A channel participant can post a single message with many distinct `<!subteam^S...>` tokens and force OpenClaw to make many Slack API calls, potentially pausing on rate limits and delaying or dropping subsequent Slack processing for the account. OpenClaw’s Slack client retries read calls, so transient failures/rate limits can amplify the stall.
- Suggested fix: Cap the number of subteam IDs resolved per message to a small bound, cache negative `ok: false`/missing-scope/no-such-subteam outcomes briefly, and consider bailing early once the cap is exceeded with a verbose/default diagnostic. A per-message cap preserves normal user-group mention behavior while bounding external API work.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for source-level reproduction: post a monitored Slack channel message containing many distinct `<!subteam^S...>` tokens and observe one `usergroups.users.list` call per token until a match or exhaustion. I did not run a live Slack workspace proof.
- Is this the best way to solve the issue? Yes, bounding and negative-caching the API expansion is the narrow fix; it avoids changing the intended semantics that a Slack user-group mention should wake only bots that Slack reports as group members. Slack’s documented user-group mention syntax is `<!subteam^ID>`, and this commit’s API-backed direction matches that contract.
