# Commit 09e7eb6

## Summary

Found one concrete cached-path regression in the new optional metadata preservation. The runtime path correctly filters unavailable manifest-optional sibling tools, but once the descriptor is cached as optional, the cached path can return it later without rechecking manifest availability.

## Findings

### Medium: Cached manifest-optional sibling tools bypass auth/config availability after becoming optional

- Kind: regression
- File: `src/plugins/tools.ts`
- Line: 686
- Evidence: The commit now captures manifest-optional sibling descriptors with `optional: true` at `src/plugins/tools.ts:1176`. On cache reuse, `resolveCachedPluginTools` computes `availableToolNames` with auth/config availability at `src/plugins/tools.ts:644`, but the loop only checks `availableToolNames` for non-optional descriptors. Optional descriptors only need `isOptionalToolAllowed`, so a cached optional descriptor can be returned even when `filterManifestToolNamesForAvailability` just excluded it.
- Impact: A plugin with a shared non-optional factory and a manifest-optional sibling gated by `authSignals` or `configSignals` can expose that optional tool after a previous authorized/cache-warming resolution. This matters for per-run auth profile availability and env/auth drift; the model/catalog can see a tool that should be unavailable.
- Suggested fix: In the cached descriptor loop, require every cached descriptor name to be present in `availableToolNames`; then additionally apply `isOptionalToolAllowed` for optional descriptors. A normalized `availableToolNameSet` would make this straightforward.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. I ran a TS smoke that seeded a plugin manifest with `other_tool` plus `optional_tool`, where `optional_tool` had `optional: true` and an `authSignals` requirement. First resolution used `env: { XAI_API_KEY: "present" }` and cached both tools; second resolution used `env: {}` with the same allowlist. The second result still returned `optional_tool` from cache and the factory was called only once: `{"first":[["other_tool",false],["optional_tool",true]],"second":[["other_tool",false],["optional_tool",true]],"factoryCalls":1}`.
- Is this the best way to solve the issue? yes. The cache key does not include every availability input, and the resolver already computes `availableToolNames` per call, so the safest local fix is to honor that set for optional descriptors too.
