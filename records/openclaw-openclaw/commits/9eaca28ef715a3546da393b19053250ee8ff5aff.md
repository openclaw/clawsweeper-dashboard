# Commit 9eaca28

## Summary

Found one remaining reliability bug in the new volatile-file handling. The added tar filter skips stable queue files, but it cannot skip queue files that disappear before `node-tar` reaches the filter, so live backups can still abort on the same delivery-queue race this commit documents as skipped.

## Findings

### Medium: Deleted volatile queue entries still abort backup before filtering

- Kind: reliability
- File: `src/infra/backup-create.ts`
- Line: 506
- Evidence: `createBackupArchive` relies on `tar.c({ filter: tarFilter })` to skip volatile queue files, and the filter checks `delivery-queue` / `session-delivery-queue` JSON paths in `src/infra/backup-volatile-filter.ts:114`. But `node-tar` calls `lstat` before invoking the create filter: `node_modules/tar/dist/commonjs/pack.js:262` emits the stat error immediately, and only after a successful stat does `node_modules/tar/dist/commonjs/pack.js:277` call `this.filter(...)`. I reproduced this deterministically by deleting `state/delivery-queue/race.json` from a patched `fs.lstat` hook during `createBackupArchive`; the command failed with `Backup archive write failed: ENOENT ... lstat '<temp-state>/delivery-queue/race.json' ... after 3 attempts`.
- Impact: A live gateway can still make `openclaw backup create` fail when a delivery queue entry is acknowledged/removed between tar’s directory read and stat. The docs now say delivery queues are skipped during archive creation, but the current implementation only skips queue files that survive long enough to be statted.
- Suggested fix: Treat `ENOENT` under known volatile queue/session/log paths as a retryable live-write race, or pre-walk/filter volatile paths before passing the source list to `node-tar`. If using retry, track and report the actual attempt count; the current non-EOF failure wrapper says `after 3 attempts` even when it failed after one attempt.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused live check created a real OpenClaw test state with `delivery-queue/race.json`, patched `fs.lstat` to delete that file exactly when `node-tar` tried to stat it, then called `createBackupArchive({ includeWorkspace: false })`. It failed before the volatile filter could run.
- Is this the best way to solve the issue? no. The filter is correct for already-statted files, but disappeared-file races need handling before or around `node-tar`’s stat phase.
