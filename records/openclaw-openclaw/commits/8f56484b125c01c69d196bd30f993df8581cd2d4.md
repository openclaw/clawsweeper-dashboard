# Commit 8f56484b12

## Summary

Found one public SDK compatibility regression. The runtime behavior still passes the focused contract tests, but the commit changes typed plugin-facing APIs and breaks existing TypeScript callers that supplied the previously supported type argument.

## Findings

### Medium: Public plugin SDK context readers no longer accept their documented generic call form

- Kind: compatibility
- File: `src/plugins/types.ts`
- Line: 2606
- Evidence: `OpenClawPluginApi.getRunContext` changed from `<T extends PluginJsonValue = PluginJsonValue>(...) => T | undefined` to `(params) => PluginJsonValue | undefined`. `PluginHookToolContext.getSessionExtension` similarly changed at `src/plugins/hook-types.ts:411`. The old comments explicitly said these JSON reads are caller-typed by namespace. A focused temporary TypeScript probe using `api.getRunContext<State>({ runId, namespace })` and `ctx.getSessionExtension<State>("policy")` now fails with `TS2558: Expected 0 type arguments, but got 1` on both calls. `pnpm plugin-sdk:api:check` also fails with Plugin SDK API baseline drift.
- Impact: External TypeScript plugins using the previous generic helper form will stop compiling after this commit, even though this was a chore/import cleanup. This touches `OpenClawPluginApi`, which is exported through `openclaw/plugin-sdk/plugin-entry` and `openclaw/plugin-sdk/core`, so it is a public plugin author surface rather than only an internal refactor.
- Suggested fix: Restore the generic signatures and keep the lint suppression comments, or replace them with an explicitly compatible overload that still accepts type arguments. If the breaking change is intentional, it needs SDK versioning/docs/baseline follow-through rather than landing as stale import cleanup.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A temporary TypeScript file importing `OpenClawPluginApi` from `src/plugin-sdk/plugin-entry.ts` and calling `api.getRunContext<State>(...)` fails with `TS2558`; the same happens for `getSessionExtension<State>(...)`.
- Is this the best way to solve the issue? yes. Restoring the generic public signatures preserves existing plugin source compatibility while keeping the current runtime implementation unchanged.
