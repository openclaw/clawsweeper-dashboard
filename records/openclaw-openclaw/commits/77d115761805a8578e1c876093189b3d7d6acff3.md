# Commit 77d1157

## Summary

The new hardlink-target check only proves that the target path appears somewhere in the archive. It still lets `backup verify` pass archives whose hardlink target has not materialized as a regular file yet, including self-links and target-later archives that fail ordinary tar extraction.

## Findings

### Medium: `backup verify` still accepts non-extractable hardlink targets

- Kind: data_loss
- File: `src/commands/backup-verify.ts`
- Line: 274
- Evidence: `verifyHardlinkTargetsAgainstArchiveRoot` checks `entries.has(target.normalized)`, but `entries` contains every normalized archive path, including the hardlink entry itself and entries that appear later in the tar stream. A focused smoke script built three in-root archives: `self-link` and `target-later` both returned `verify=passed`, but `tar -xzf` exited 2 with `Cannot hard link ... No such file or directory`; the `target-earlier` control both verified and extracted successfully. The added test in `src/commands/backup-verify.test.ts:346` only covers a target absent from the entry set, so it misses this case.
- Impact: `openclaw backup verify` can still certify an archive that cannot be broadly restored. This is the same user-facing risk class as the hardlink restore bug this area is repairing: backup confidence is overstated, especially for older or third-party archives that contain `Link` entries.
- Suggested fix: either reject `Link` entries outright now that `backup create` dereferences hardlinks, or verify hardlinks against extraction semantics: target must not equal the link entry, must have appeared earlier in archive order, and must resolve to a materialized non-link payload entry rather than another unresolved link.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Build a backup-shaped tar with `manifest.json`, then a `Link` entry whose `linkpath` is the same path or a file entry that appears later. Current `backupVerifyCommand` passes it; system tar extraction fails before materializing the link.
- Is this the best way to solve the issue? yes. Since current backup creation intentionally avoids emitting hardlink entries, fail-closed rejection is the simplest safe path. If compatibility with old safe hardlink archives matters, order-aware validation is the narrower alternative.
