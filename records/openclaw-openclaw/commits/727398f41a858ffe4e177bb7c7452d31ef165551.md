# Commit 727398f

## Summary

Found one concrete residual credential-echo path. The Node Clack CLI path honors the new `sensitive` flag, and focused Node tests pass, but the bundled macOS `openclaw-mac wizard` CLI decodes `WizardStep.sensitive` and still reads sensitive text steps with echoing `readLine()`.

## Findings

### Medium: macOS wizard CLI still echoes sensitive wizard text answers

- Kind: security
- File: `apps/macos/Sources/OpenClawMacCLI/WizardCommand.swift`
- Line: 445
- Evidence: This commit now forwards `sensitive: true` through `WizardSessionPrompter.text` for credential prompts (`src/wizard/session.ts:122`) and the Swift protocol model already decodes `WizardStep.sensitive` (`apps/shared/OpenClawKit/Sources/OpenClawProtocol/GatewayModels.swift:2402`). The SwiftUI app handles it with `SecureField` (`apps/macos/Sources/OpenClaw/OnboardingWizard.swift:303`), but `openclaw-mac wizard` handles every text step with `readLineWithPrompt`, prints any initial value in brackets, and `readLineWithPrompt` calls echoing `readLine()` (`apps/macos/Sources/OpenClawMacCLI/WizardCommand.swift:445`, `apps/macos/Sources/OpenClawMacCLI/WizardCommand.swift:520`).
- Impact: Users running the macOS CLI wizard against the Gateway can still expose provider API keys, gateway tokens/passwords, web-search keys, or skill env values in terminal scrollback, shell transcripts, screenshots, and screen shares. This is the same leak class the commit fixes for the Node Clack CLI.
- Suggested fix: In the Swift CLI `case "text"` branch, check `step.sensitive == true`; for sensitive steps use a termios no-echo reader, suppress bracketed initial-value display, and preserve long inputs without truncation.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. On macOS, run `openclaw-mac wizard` against a Gateway flow that emits a sensitive text step, such as model auth key entry after this commit; the Swift CLI uses `readLine()` for that step, so terminal echo remains enabled.
- Is this the best way to solve the issue? yes. The protocol already carries `sensitive`; the missing behavior is client-side rendering in the Swift CLI. Node Clack and SwiftUI already have the expected split.
