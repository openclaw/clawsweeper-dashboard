# Commit e7f1b10

## Summary

Found one low-severity gateway watch regression: auto-doctor now runs after clean `gateway` child exits, not just failed startup exits.

## Findings

### Low: clean gateway watch exits trigger doctor repair and retry

- Kind: bug
- File: `scripts/watch-node.mjs`
- Line: 349
- Evidence: `shouldRunAutoDoctor` only checks that the watched command starts with `gateway` and that auto-doctor has not already run. The exit handler calls it after every non-restartable child exit, without checking `exitCode !== 0` or `exitSignal`. The previous test covered `args: ["gateway", "--force", "--help"]` returning child exit code `0`; this commit changed that test to `["config", "validate"]`, avoiding the new gateway-specific path. A focused harness confirmed that a clean `gateway --help` child exit spawns `scripts/run-node.mjs doctor --fix --non-interactive`, then restarts `gateway --help`.
- Impact: `pnpm gateway:watch --help`, or any other legitimate clean one-shot `gateway` exit under the watcher, now runs a mutating `doctor --fix --non-interactive` pass and repeats the command instead of simply returning `0`. That can unexpectedly modify local config/state and produces confusing duplicate command output.
- Suggested fix: pass `exitCode`/`exitSignal` into the auto-doctor predicate and require a failed exit, for example `exitSignal || exitCode !== 0`, before running doctor. Restore/add a regression test for `args: ["gateway", "--force", "--help"]` exiting `0` with no doctor spawn.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The mock `runWatchMain` harness emitted `exit(0, null)` from a `gateway --help` child and observed doctor spawn before restart.
- Is this the best way to solve the issue? yes. The feature is meant for failed startup repair; preserving clean exit semantics only needs an exit-status gate plus the prior help-path regression test.
