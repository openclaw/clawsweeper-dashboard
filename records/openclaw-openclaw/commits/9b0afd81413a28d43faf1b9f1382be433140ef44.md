# Commit 9b0afd8

## Summary

Found one diagnostic regression in the new human `gateway status --deep` output. JSON receives the handoff details, but human output still falls through to the old stopped-service error for the same clean handoff state.

## Findings

### Low: Restart handoffs still print the opaque stopped-service error

- Kind: bug
- File: `src/cli/daemon-cli/status.print.ts`
- Line: 267
- Evidence: The commit adds handoff rendering at `src/cli/daemon-cli/status.print.ts:184`, but the later `else if (service.loaded && service.runtime?.status === "stopped")` branch at `src/cli/daemon-cli/status.print.ts:267` is unchanged and does not exclude `service.restartHandoff`. A focused render probe with `loaded: true`, `runtime.status: "stopped"`, and a fresh `restartHandoff` printed both `Recent restart handoff: full-process via launchd...` and `Service is loaded but not running (likely exited immediately).`
- Impact: This undercuts the feature’s stated behavior: clean supervisor restart handoffs are still accompanied by the misleading immediate-exit diagnostic and restart-log guidance. Users can still be sent toward failure troubleshooting during a normal supervised restart window.
- Suggested fix: Suppress or replace the stopped-service branch when `service.restartHandoff` is present, while keeping the existing stopped-service diagnostics for non-handoff cases.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: call `printDaemonStatus` with a loaded stopped service and `service.restartHandoff`; the focused probe reproduced the mixed output on current `main`.
- Is this the best way to solve the issue? yes: the handoff is already validated and TTL-bounded by `readGatewayRestartHandoffSync`, so it is the right condition for treating this stopped state as an expected transient handoff instead of an immediate-exit failure.
