## Findings

### Low: Default status paths do not actually receive model-pricing health

- Kind: bug
- File: `src/commands/status-overview-rows.ts`
- Line: 37
- Evidence: `buildStatusCommandOverviewRows` can render model-pricing only from `params.health` or `surface.gatewayProbe.health` (`src/commands/status-overview-rows.ts:30`). But normal status only fetches `health` when `opts.deep` is true (`src/commands/status-runtime-shared.ts:154`), and the gateway probe defaults to `detailLevel: "presence"` (`src/commands/status.scan.shared.ts:226`), whose probe result explicitly sets `health: null` (`src/gateway/probe.ts:332`). The JSON path has the same problem: it only includes `health` when `opts.deep` is passed (`src/commands/status-json-runtime.ts:64`) and otherwise relies on `gatewayProbe.health` for `gateway.modelPricing` (`src/commands/status-all/format.ts:492`). `status --all` fetches health separately (`src/commands/status-all/report-data.ts:78`) but does not pass it into `buildStatusAllOverviewRows` (`src/commands/status-all/report-data.ts:177`).
- Impact: After a pricing bootstrap/source failure, users can see degradation in `openclaw health` and `openclaw gateway status`, but the advertised `openclaw status` and status JSON surfaces can still report no model-pricing warning unless `--deep` is used. This leaves the original â€śany user-visible output/statusâ€ť gap partly open.
- Suggested fix: Thread the already-fetched health summary into the status overview/JSON surfaces, and for normal `status`/`status --json` either request lightweight health when the gateway is reachable or avoid claiming those surfaces unless `--deep` is required. Add an end-to-end-ish test through `scanStatus`/`runStatusJsonCommand`, not only a payload test with fabricated `gatewayProbe.health`.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes by source path: force model-pricing degradation, run `openclaw status` or `openclaw status --json` without `--deep`, and observe that neither `params.health` nor `gatewayProbe.health` carries the model-pricing state into the renderer.
- Is this the best way to solve the issue? Yes: the health state already exists; the safer fix is to pass that structured state through the status surfaces rather than relying on the default presence-only probe.
