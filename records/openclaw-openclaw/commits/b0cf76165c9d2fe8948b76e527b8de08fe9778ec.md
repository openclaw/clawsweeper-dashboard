# Commit b0cf761

## Summary

Found one concurrency/regression risk in the gateway cleanup flag plumbing. The kitchen-sink env-var change looked safe.

## Findings

### Medium: Public agent RPC can close the shared CLI bundle MCP loopback server

- Kind: concurrency
- File: `src/gateway/server-methods/agent.ts`
- Line: 1376
- Evidence: This commit forwards caller-supplied `cleanupBundleMcpOnRunEnd` from the gateway `agent` request into `agentCommandFromIngress` at `src/gateway/server-methods/agent.ts:1376`. The field remains part of the public agent schema and is still documented there as an older-client compatibility no-op at `src/gateway/protocol/schema/agent.ts:157`. For CLI backends, that flag unconditionally calls `closeMcpLoopbackServer()` at `src/agents/cli-runner.ts:114`, while CLI bundle MCP setup reuses one process-wide loopback runtime via `getActiveMcpLoopbackRuntime()` / `ensureMcpLoopbackServer()` at `src/agents/cli-runner/prepare.ts:177`. `closeMcpLoopbackServer()` then clears and closes that single active server without a per-run lease/refcount at `src/gateway/mcp-http.ts:235`.
- Impact: A normal gateway `agent` request, or an older client that still sends this compatibility field, can now close the shared CLI bundle MCP loopback server when its run ends. If another CLI-backend run is active and using bundle MCP tools, later MCP tool calls can fail because their configured loopback port has been closed. This also changes the documented compatibility behavior from “accepted but ignored” to “active process-wide cleanup.”
- Suggested fix: Do not trust/forward the public request field directly. Derive cleanup from server-side one-shot context only, or make the CLI MCP loopback runtime lease/refcounted and close it only when no active prepared CLI runs still reference it. Update the schema comment once the intended contract is explicit.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes: configure a gateway CLI backend with `bundleMcp: true`, start one long-running gateway agent turn that can call OpenClaw MCP tools, then complete a second `agent` RPC with `cleanupBundleMcpOnRunEnd: true`. The second run reaches `closeMcpLoopbackServer()`, closing the shared loopback server used by the first run.
- Is this the best way to solve the issue? No. The release leak fix is valid, but the cleanup authority needs to be internal/per-run or refcounted because the CLI MCP loopback server is process-wide.
