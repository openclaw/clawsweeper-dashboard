# Commit 6330fe6

## Summary

Found one release-path regression: alpha npm releases are still accepted by the workflow, but the new prepublish/packed-install verification rejects alpha versions because the reused version normalizer only recognizes stable, correction, and beta versions.

## Findings

### Medium: Alpha npm releases fail the new tarball verification

- Kind: regression
- File: `scripts/openclaw-npm-prepublish-verify.ts`
- Line: 65
- Evidence: The workflow still accepts `vYYYY.M.D-alpha.N` and requires the `alpha` dist-tag in `.github/workflows/openclaw-npm-release.yml:62` and `.github/workflows/openclaw-npm-release.yml:70`, but the new verifier calls `collectInstalledPackageErrors` and compares `normalizeInstalledBinaryVersion(installedBinaryVersion)` against the expected package version in `scripts/openclaw-npm-prepublish-verify.ts:65` and `scripts/openclaw-npm-prepublish-verify.ts:80`. That normalizer is `/\b\d{4}\.\d{1,2}\.\d{1,2}(?:-\d+|-beta\.\d+)?\b/u` in `scripts/openclaw-npm-postpublish-verify.ts:164`, so `2026.5.14-alpha.3` normalizes to `2026.5.14`. The same failure is now wired into `release:check` through `scripts/release-check.ts:345` and `scripts/release-check.ts:352`.
- Impact: Any alpha preflight or `pnpm release:check` for an alpha package will fail before publish with misleading version mismatch errors, blocking a supported release channel.
- Suggested fix: Update `normalizeInstalledBinaryVersion` to parse the same release forms as the release checker, including `-alpha.N`, and add alpha cases to `test/openclaw-npm-postpublish-verify.test.ts` plus the new packed-install release-check test path.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused probe returned `normalizeInstalledBinaryVersion("OpenClaw 2026.5.14-alpha.3 (abc1234)") -> "2026.5.14"`, and `collectPackedInstalledPackageVerificationErrors` on a minimal alpha package returned both package and binary version mismatch errors.
- Is this the best way to solve the issue? yes. The verifier should share the release version grammar rather than special-casing only beta.
