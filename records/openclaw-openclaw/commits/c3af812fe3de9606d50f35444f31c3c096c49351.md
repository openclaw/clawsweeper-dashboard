# Commit c3af812f

## Summary

The commit refreshes Codex account rate limits after subscription-limit failures and improves `/codex account` formatting. I found one actionable edge case where the new summary/error path can report the wrong reset time when more than one Codex quota window is exhausted.

## Findings

### Medium: Multiple exhausted Codex windows can report the earliest reset even though a longer window still blocks usage

- Kind: bug
- File: `extensions/codex/src/app-server/rate-limits.ts`
- Line: 333
- Evidence: `selectSnapshotBlockingReset()` filters exhausted windows, sorts by `resetsAtMs` ascending, and returns `candidates[0]`. `selectNextRateLimitReset()` uses the same earliest-reset selection for channel errors. The generated Codex app-server protocol for `@openai/codex@0.130.0` has both `primary` and `secondary` `RateLimitWindow`s on one snapshot, each with `usedPercent`, `windowDurationMins`, and `resetsAt`. A local probe with `primary` at 100% resetting in 1 hour and `secondary` at 100% resetting in 7 days produced: `Codex is paused until May 11 at 1:00 AM UTC (in 1 hour)` and `Next reset in 1 hour`.
- Impact: If both quota windows are exhausted, the user will still be blocked after the short-term window resets, but Discord and `/codex account` can tell them Codex is paused only until the earlier reset. That undermines the commit’s main user-facing promise of accurate subscription reset times.
- Suggested fix: For “paused until” / “try again” messaging, choose the latest reset among exhausted blocking windows, or list each exhausted window separately. If upstream later exposes which specific window caused `rateLimitReachedType`, prefer that explicit field.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the formatter behavior: call `summarizeCodexAccountRateLimits()` or `formatCodexUsageLimitErrorMessage()` with one Codex snapshot where both `primary` and `secondary` have `usedPercent: 100`, with different future `resetsAt` values. Live Codex production behavior with both windows exhausted was not available in this review.
- Is this the best way to solve the issue? Likely yes: user-facing availability should not be based on the earliest exhausted-window reset unless that is explicitly the only blocking window.
