# Commit 6fd35f6

## Summary

Found two parser recovery regressions in the new multiline Codex app-server buffering path.

## Findings

### Medium: One malformed partial frame can consume later valid JSON-RPC frames

- Kind: reliability
- File: `extensions/codex/src/app-server/client.ts`
- Line: 302
- Evidence: `handlePendingParseLine` always appends the next stdout line to `pending.text` and, if the combined candidate still fails, keeps buffering until 1 MB or 1,000 fragments. It does not detect that the new line may itself be a fresh valid JSON-RPC frame. A focused harness probe writing an unterminated JSON-looking line followed by `{"method":"next/valid","params":{"ok":true}}` produced no notifications.
- Impact: after one malformed/incomplete app-server stdout line matching `shouldBufferCodexAppServerParseFailure`, later valid notifications and responses can be swallowed until the buffer limit is reached. In app-server turn paths, that can hide `turn/completed` and output deltas, leading to request or idle timeouts instead of continued recovery.
- Suggested fix: when the buffered candidate fails, check whether the new raw line is a plausible standalone JSON-RPC frame; if so, log/drop the pending fragment and process the new line normally. Add a regression test for malformed partial frame followed by a valid notification.
- Confidence: high

### Low: Recovered multiline output loses trailing spaces before the first raw newline

- Kind: bug
- File: `extensions/codex/src/app-server/client.ts`
- Line: 279
- Evidence: the first buffered fragment is stored from `trimmed`, not `rawLine`, so significant trailing whitespace inside the JSON string is removed before reconstruction. A focused probe with `delta: "first   \nsecond"` delivered `delta: "first\nsecond"`.
- Impact: command/file output delivered through `item/commandExecution/outputDelta` or similar notifications can be subtly corrupted, which matters for whitespace-sensitive command output and patches.
- Suggested fix: keep the raw fragment for pending reconstruction while using a separately trimmed value only for blank-line detection and initial parse/log preview decisions. Add a test that preserves spaces before a recovered newline.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Both issues reproduce through `CodexAppServerClient.fromTransportForTests` with a fake stdout transport; no live app-server is required.
- Is this the best way to solve the issue? unclear. The intended multiline recovery is reasonable, but the buffer needs a resynchronization path and should preserve raw payload bytes where possible.
