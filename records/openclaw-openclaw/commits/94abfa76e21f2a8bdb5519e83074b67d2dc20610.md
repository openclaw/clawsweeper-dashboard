# Commit 94abfa7

## Summary

Found two concrete regressions in the converted `doctor --lint` health-check path. Both are false-clean or ignored-option problems introduced by moving legacy note-based checks into structured lint checks.

## Findings

### Medium: Browser lint drops several real browser readiness warnings

- Kind: regression
- File: `src/flows/doctor-core-checks.ts`
- Line: 334
- Evidence: `core/doctor/browser` captures legacy browser notes through `createNoteCollector()`, but `createNoteCollector()` discards any note inferred as `info`. The inference only treats text containing `CRITICAL`, `- Fix:`, `unavailable`, `not found`, `missing`, `not readable`, `not writable`, or `readonly` as actionable. Existing browser readiness warnings include managed-browser no-executable text (`"No Chromium-based browser executable was found"`), no display/no-sandbox text, and Chrome-too-old text in `extensions/browser/src/doctor-browser.ts:245`, `extensions/browser/src/doctor-browser.ts:256`, and `extensions/browser/src/doctor-browser.ts:324`; those do not match the collector’s warning triggers, so `doctor --lint --only core/doctor/browser` can return no findings for configurations that legacy `doctor` still warns about.
- Impact: Browser readiness lint can falsely pass while managed browser launch is known to fail or Chrome MCP is too old, undermining the conversion’s goal of preserving read-only doctor diagnostics.
- Suggested fix: Do not infer/drop severity for browser notes. Prefer a structured browser detector, or at least have `core/doctor/browser` preserve every emitted browser note as a warning unless the browser surface explicitly marks it informational.
- Confidence: high

### Low: `--no-workspace-suggestions` is ignored for the new lint workspace check

- Kind: compatibility
- File: `src/cli/program/register.maintenance.ts`
- Line: 48
- Evidence: The command defines `--no-workspace-suggestions` at `src/cli/program/register.maintenance.ts:16`, but the `--lint` branch calls `runDoctorLintCli()` with only `json`, `severityMin`, `skipIds`, and `onlyIds`. The new `core/doctor/workspace-suggestions` lint check at `src/flows/doctor-core-checks.ts:683` has no access to the doctor option and always emits backup/memory-system info findings when applicable.
- Impact: `openclaw doctor --lint --no-workspace-suggestions` can still fail or emit workspace suggestion findings, even though the user explicitly disabled that doctor surface. This matters because default lint visibility/exit behavior includes `info`.
- Suggested fix: Thread `workspaceSuggestions` through `DoctorLintCliOptions` and either skip `core/doctor/workspace-suggestions` when false or include it in `skipIds` before running lint checks.
- Confidence: high
