# Commit 41810a4

## Summary

Found one low-severity bug: the new Discord `suppressEmbeds` default is applied to the main bot send path, but not to webhook persona delivery or thread initial-message creation paths.

## Findings

### Low: Some Discord outbound paths still omit `SUPPRESS_EMBEDS`

- Kind: bug
- File: `extensions/discord/src/send.webhook.ts`
- Line: 105
- Evidence: `discordOutbound.sendText` tries `maybeSendDiscordWebhookText` before falling back to `sendMessageDiscord` for bound thread replies (`extensions/discord/src/outbound-adapter.ts:160`). That webhook path posts JSON with `content`, `username`, `avatar_url`, and `message_reference`, but no `flags`, so URL previews still expand in webhook persona replies (`extensions/discord/src/send.webhook.ts:105`). The same default is also skipped for message-action thread initial messages: forum/media thread starters use `body.message = { content: starterContent }`, and non-forum/message-attached threads send `{ content: payload.content }` (`extensions/discord/src/send.messages.ts:164`, `extensions/discord/src/send.messages.ts:182`). Discord’s Execute Webhook and message-create contracts both support `flags` with `SUPPRESS_EMBEDS`, so these paths can carry the same suppression bit. ([docs.discord.com](https://docs.discord.com/developers/resources/webhook?utm_source=openai))
- Impact: Agent-sent URLs can still render Discord link previews in bound thread webhook/persona replies, despite the new config/docs claiming outbound Discord URLs are suppressed by default.
- Suggested fix: Share the new suppress-embed resolver with `sendWebhookMessageDiscord` and `createThreadDiscord`, derive the effective account config from `opts.cfg`/`accountId`, and include `flags: MessageFlags.SuppressEmbeds` unless explicitly disabled.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: configure a bound Discord thread with webhook persona delivery, then send text containing `https://example.com`; the adapter calls `sendWebhookMessageDiscord` and the request body has no `flags`.
- Is this the best way to solve the issue? yes: the main send path already centralizes `SUPPRESS_EMBEDS`; the bypassing paths should use the same account-aware default.
