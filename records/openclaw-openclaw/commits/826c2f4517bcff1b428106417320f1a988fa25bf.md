# Commit 826c2f4

## Summary

Found one QA-lab regression: the new “live-only” Codex read canary is still selected by implicit mock QA suite runs, but its flow has no mock-mode skip and the mock provider cannot produce the seeded file marker for this prompt shape.

## Findings

### Medium: Live-only canary runs in mock suites and fails deterministically

- Kind: regression
- File: `qa/scenarios/runtime/codex-pi-shaped-read-vocabulary.md`
- Line: 28
- Evidence: The scenario is marked `runtimeParityTier: live-only`, but its execution config does not set `requiredProviderMode: live-frontier` and the flow always runs `runAgentPrompt`. Current selection code includes any scenario without `requiredProviderMode` for non-live providers because `selectQaSuiteScenarios` returns true when `getQaProvider(providerMode).kind !== "live"` (`extensions/qa-lab/src/suite-planning.ts:41`). A focused import check confirmed the new scenario is selected for an implicit `mock-openai` suite: `mock-openai true 76`. In that lane, the mock provider’s read-path parser only extracts backticked, quoted, `repo/...`, or `QA_*.md` paths and otherwise falls back to `repo/package.json` (`extensions/qa-lab/src/providers/mock-openai/server.ts:561`, `extensions/qa-lab/src/providers/mock-openai/server.ts:587`), while this prompt supplies an unquoted absolute `.txt` path and asks for a marker contained only in the seeded file.
- Impact: Unfiltered mock QA suite or mock runtime-pair runs now pick up this live-only canary and fail on a harness/provider mismatch rather than a real product regression. That can block maintainer confidence gates that use mock-openai for cheap runtime parity proof.
- Suggested fix: Either add `requiredProviderMode: live-frontier` to the scenario config, or wrap the live probe in the same `if env.providerMode !== 'live-frontier'` skip pattern used by other live-only scenarios. If runtime tier selection is intended to own this, wire `runtimeParityTier: live-only` into suite selection before relying on it.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: select implicit mock scenarios with `selectQaSuiteScenarios({ providerMode: "mock-openai", primaryModel: "mock-openai/gpt-5.5" })`; the canary is included. The mock response path then cannot read the seeded absolute `.txt` fixture from the prompt shape.
- Is this the best way to solve the issue? yes: adding an explicit provider-mode guard is the narrowest fix and matches existing live scenario patterns.
