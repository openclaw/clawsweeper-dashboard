# Commit c885a1c

## Summary

Found one low-severity reliability issue in the Android v2 attachment flow: the picker caps each selection batch at 8 images, but the retained composer list is unbounded across repeated picker launches.

## Findings

### Low: Repeated image picks can build an oversized chat frame

- Kind: reliability
- File: `apps/android/app/src/main/java/ai/openclaw/app/ui/chat/ChatScreen.kt`
- Line: 111
- Evidence: Current main still has this commit’s blamed flow after the later `V2ChatScreen.kt` rename. The picker loads `uris.take(8)`, then appends with `attachments.addAll(next)` without clamping the existing list. Send maps every retained attachment into the RPC payload. Each Android image attachment is compressed up to `CHAT_IMAGE_MAX_BASE64_CHARS = 300 * 1024`, while the gateway WebSocket post-auth max payload is `25 * 1024 * 1024`; gateway attachment parsing enforces per-attachment size but does not cap attachment count.
- Impact: A user can tap Attach repeatedly, selecting up to 8 images each time, and accumulate enough images for `chat.send` to exceed the gateway frame limit. That can fail the send by closing/rejecting the WebSocket frame, and the composer clears the attachment list before the request result is known.
- Suggested fix: Enforce a total composer attachment count or total encoded-size budget before `attachments.addAll(next)`, ideally shared with `ChatSheetContent`. For example, compute remaining slots from a single `MAX_CHAT_ATTACHMENTS` value, append only that many, disable Attach at the limit, and keep the budget comfortably below `MAX_PAYLOAD_BYTES`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: on Android chat, launch Attach repeatedly and select 8 large/high-detail photos per batch until the retained attachment count is high enough to exceed the 25 MB gateway frame budget, then send. I did not run this on a device, but the code path and limits are deterministic.
- Is this the best way to solve the issue? yes: the failure is caused by missing total-budget enforcement in the UI state; fixing it at append time prevents both local memory growth and oversized gateway frames. A server-side max-items guard would be a useful backstop, but it would still leave the Android composer able to accumulate a failing payload.
