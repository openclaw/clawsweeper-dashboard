# Commit 1df3850a17

## Summary

Found one low-severity regression in the live Docker test script: local runs that rely on `$HOME/.profile` for `OPENAI_API_KEY` no longer pick it up by default.

## Findings

### Low: Codex npm plugin live Docker lane no longer falls back to `$HOME/.profile`

- Kind: regression
- File: `scripts/e2e/codex-npm-plugin-live-docker.sh`
- Line: 14
- Evidence: Before this commit, `PROFILE_FILE` defaulted to `$HOME/.profile`. After the commit it defaults to `${OPENCLAW_TESTBOX_PROFILE_FILE:-$HOME/.openclaw-testbox-live.profile}` unless `OPENCLAW_CODEX_NPM_PLUGIN_PROFILE_FILE` is set. The script only mounts/sources the selected file when it exists, then passes `OPENAI_API_KEY` into Docker from the host environment and the container fails if the key is still missing. A focused default-resolution check with a temp home containing only `.profile` resolved `PROFILE_FILE` to `.openclaw-testbox-live.profile` and reported it missing.
- Impact: `pnpm test:docker:live-codex-npm-plugin` regresses for local developers/operators who have OpenAI credentials in `$HOME/.profile`, which was the previous default and is still the fallback used by related live Docker auth helper logic. These runs now fail before exercising the intended npm plugin path unless callers discover and set `OPENCLAW_CODEX_NPM_PLUGIN_PROFILE_FILE=$HOME/.profile`.
- Suggested fix: Preserve the Testbox preference without dropping the local fallback, for example by using the shared `openclaw_live_default_profile_file` behavior or by resolving to `$HOME/.openclaw-testbox-live.profile` only when it exists, then falling back to `$HOME/.profile`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. With a home directory containing `.profile` but no `.openclaw-testbox-live.profile`, the new default resolves to the missing Testbox profile, so no profile is mounted or sourced for the Docker run.
- Is this the best way to solve the issue? yes. Restoring fallback selection keeps the intended Testbox profile support while preserving the pre-existing local credential workflow.
