# Commit 37426a6e

## Summary

Found regressions in the generic directory forwarding path added for Slack. The most concrete issue breaks Microsoft Teams `directory self`; a second Slack edge loses config-backed directory results when live Slack readers exist but no readable token is available.

## Findings

### Medium: Live directory forwarder overwrites existing/default `self` resolvers

- Kind: regression
- File: `src/channels/plugins/runtime-forwarders.ts`
- Line: 42
- Evidence: `createRuntimeDirectoryLiveAdapter` now always returns a `self` property, set to `undefined` when no `self` resolver is passed. Channels spread that result into directory adapter params. In `extensions/msteams/src/channel.ts:460`, Microsoft Teams defines a real `self` resolver, but the later spread at `extensions/msteams/src/channel.ts:498` overwrites it with `undefined`. The existing focused test now fails: `pnpm test extensions/msteams/src/channel.directory.test.ts` errors at `extensions/msteams/src/channel.directory.test.ts:15` with `expected msteams directory.self`.
- Impact: `openclaw directory self --channel msteams` regresses from returning the configured bot identity/null to being unsupported. Channels that relied on `createChannelDirectoryAdapter`’s default null self resolver, such as Discord/Matrix/Feishu, can also lose the default and turn `directory self` into an unsupported-channel error.
- Suggested fix: Only include forwarded keys when the corresponding resolver is provided, or make `createChannelDirectoryAdapter` ignore `undefined` overrides after spreads. Add a regression test for both omitted `self` and an explicit `self` followed by a live-adapter spread.
- Confidence: high

### Medium: Slack CLI live preference drops config-backed fallback when no token is available

- Kind: regression
- File: `src/cli/directory-cli.ts`
- Line: 171
- Evidence: `runDirectoryList` now chooses `listPeersLive ?? listPeers` / `listGroupsLive ?? listGroups`. Slack always registers live readers, so the config reader is never called. But Slack live readers return `[]` when `resolveReadToken` cannot produce a token (`extensions/slack/src/directory-live.ts:127` and `extensions/slack/src/directory-live.ts:174`), while Slack’s config-backed directory contract explicitly remains readable with unresolved SecretRefs (`extensions/slack/src/directory-contract.test.ts:56`). This contradicts the linked issue’s expected repair direction to prefer live readers while preserving config fallback.
- Impact: Slack installs with allowlists/configured groups but missing or unresolved Slack tokens now get empty `openclaw directory peers/groups list --channel slack` output where the previous CLI returned config-backed IDs.
- Suggested fix: Preserve fallback only when the live path is unavailable, for example by having Slack live readers delegate to config readers when no read token is resolved, or by adding a typed unavailable signal so the CLI can fall back without treating a legitimate empty live result as failure.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the first finding: `pnpm test extensions/msteams/src/channel.directory.test.ts` fails on current `main`. The second is source-level reproducible with a Slack config containing directory allowlist/group entries but no resolvable read token.
- Is this the best way to solve the issue? Yes for the first finding: omit undefined forwarded keys. For the second, the safest direction is an explicit unavailable-token fallback, not a blind fallback on every empty live result.
