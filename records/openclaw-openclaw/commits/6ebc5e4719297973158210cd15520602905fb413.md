## Findings

### Medium: Timeout fallback accepts missing parent synthesis reply

- Kind: test_gap
- File: `qa/scenarios/agents/subagent-fanout-synthesis.md`
- Line: 150
- Evidence: The scenario objective requires the final answer to synthesize both worker outputs in one reply (`qa/scenarios/agents/subagent-fanout-synthesis.md:12-16`). The normal path waits for an outbound message containing both expected reply groups, then stores `outbound.text` as details (`qa/scenarios/agents/subagent-fanout-synthesis.md:103-147`). The new catch path handles any mock timeout, checks only that child rows and at least two `sessions_spawn` debug requests exist, then fabricates `details: "subagent-1: ok\nsubagent-2: ok"` and sets `lastError` to `__done__` (`qa/scenarios/agents/subagent-fanout-synthesis.md:150-175`). The flow runner does not rethrow caught errors after catch actions run (`extensions/qa-lab/src/scenario-flow-runner.ts:247-268`), and the final assertion only checks `lastError === '__done__'` (`qa/scenarios/agents/subagent-fanout-synthesis.md:194-198`).
- Impact: A regression where the parent successfully spawns both subagents but never wakes, waits, or sends the final synthesized reply will be reported as a passing release QA scenario in `mock-openai` mode. That is exactly the third success criterion this scenario is supposed to guard.
- Suggested fix: Do not mark the timeout path done unless it can verify a real parent outbound/transcript reply containing both expected groups. If the fallback is only meant to tolerate transport observation races, read the actual parent session output or transport state and use that text as `details`; otherwise keep the timeout as a failure.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. In `mock-openai`, any run that records two `sessions_spawn` calls and creates `qa-fanout-alpha` / `qa-fanout-beta` child rows but then times out before the parent outbound synthesis will take the new catch branch and pass the final assertion.
- Is this the best way to solve the issue? yes: preserve the spawn evidence checks, but require real parent synthesis evidence before setting `lastError` to `__done__`.
