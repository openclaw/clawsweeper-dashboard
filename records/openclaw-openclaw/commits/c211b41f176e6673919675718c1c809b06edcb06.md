## Summary

Found one low-severity freshness regression in the new module-level bundled alias cache.

## Findings

### Low: Bundled alias normalization can stay stale for the process lifetime

- Kind: regression
- File: `src/plugins/config-state.ts`
- Line: 47
- Evidence: `getBundledPluginAliasLookup()` now stores the first `listBundledPluginMetadata({ includeChannelConfigs: false })` result in `bundledPluginAliasLookup` and never invalidates it. The plugin metadata layer itself re-reads manifests on each call (`src/plugins/bundled-plugin-metadata.ts:178`) and has coverage asserting manifest edits are reflected on the next read (`src/plugins/bundled-plugin-metadata.test.ts:511`, `src/plugins/bundled-plugin-metadata.test.ts:519`). The architecture doc also says manifest/discovery metadata should stay fresh unless a caller holds an explicit snapshot/lookup table/manifest registry, and hidden metadata caches are not part of plugin loading (`docs/plugins/architecture.md:168`).
- Impact: if bundled plugin manifests are repaired, regenerated, or replaced while a CLI/gateway/test process remains loaded, `normalizePluginId()` can keep resolving provider and legacy aliases through the old manifest set. That can make `plugins.allow`, `plugins.deny`, and `plugins.entries` normalize to stale owners, or fail to recognize newly added bundled aliases until restart.
- Suggested fix: keep the perf improvement behind an explicit metadata snapshot/lookup table passed by callers, or add cache invalidation keyed to the resolved bundled plugin root/manifest state. At minimum, add a focused regression test for `normalizePluginId()` across manifest alias changes if this cache is intentionally process-static.
- Confidence: medium
