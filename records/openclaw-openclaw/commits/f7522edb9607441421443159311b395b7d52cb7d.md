# Commit f7522ed

## Summary

Found one migration/data-loss gap: the new manifest-driven `npm install` preserves siblings only after the managed npm-root `package.json` already lists them. Users upgrading from the shipped broken state can still have installed plugin directories and install records but no managed manifest entries, and the first post-upgrade install/update can prune those existing siblings.

## Findings

### Medium: First post-upgrade npm install can still prune existing unmanifested plugin siblings

- Kind: data_loss
- File: `src/plugins/install.ts`
- Line: 1217
- Evidence: The commit now runs `npm install --prefix <npmRoot>` without an explicit spec, so npm reconciles disk from the managed root manifest. But `upsertManagedNpmRootDependency` only preserves dependencies already present in `package.json` and then adds the current package; it does not seed that manifest from existing `node_modules` siblings or `plugins.installs`. For a pre-fix npm root with `node_modules/@openclaw/voice-call` but no `package.json`, installing `@openclaw/whatsapp` creates a manifest containing only whatsapp, then bare `npm install` prunes voice-call as extraneous. I reproduced the npm behavior in `/tmp`: before install the root had `@openclaw/voice-call`; after a manifest-only install for `@openclaw/whatsapp`, npm left only `@openclaw/whatsapp`.
- Impact: Users who installed one official npm plugin on 2026.5.2 and then upgrade to this fix can still lose that existing plugin when they install or update another plugin. The registry can continue to claim both plugins are installed while one install path is gone, matching the user-visible breakage from #76571 for the first post-upgrade operation.
- Suggested fix: Before the first manifest-driven install in an existing npm root, hydrate `package.json` from trusted existing npm install records whose `installPath` is under the managed npm root and still exists, or from existing managed `node_modules` package manifests with corresponding install records. Add a regression test where `node_modules/@openclaw/voice-call` exists but `package.json` is missing, then install whatsapp and assert both remain.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Create a managed npm root with an existing `node_modules/@openclaw/voice-call` package and no dependency entry for it, write a root manifest with only `@openclaw/whatsapp`, then run `npm install --prefix <root>`; npm prunes the unmanifested voice-call directory.
- Is this the best way to solve the issue? no. The manifest-driven install is the right direction for fresh/current installs, but it needs a one-time hydration path for already-shipped npm roots before using the manifest as the source of truth.
