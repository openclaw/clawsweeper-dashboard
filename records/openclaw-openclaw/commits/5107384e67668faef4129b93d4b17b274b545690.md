# Commit 5107384

## Summary

Found one concrete reliability regression risk in the new Matrix approval retry path: it retries non-idempotent Matrix sends/direct-room creation without a stable Matrix transaction id.

## Findings

### Medium: Matrix approval retries can duplicate stale approval prompts

- Kind: reliability
- File: `extensions/matrix/src/approval-handler.runtime.ts`
- Line: 453
- Evidence: `retryMatrixApprovalDelivery` re-invokes `sendSingleTextMessage` and chunked `sendMessage` on any thrown send error except the local ÔÇťtext exceeds limitÔÇŁ error. The Matrix send wrapper eventually calls `client.sendMessage(roomId, content)` without a transaction id in `extensions/matrix/src/matrix/send.ts:507` and `extensions/matrix/src/matrix/sdk.ts:911`. Installed `matrix-js-sdk@41.4.0` source shows `sendMessage` delegates to `sendEvent`, and `sendEvent` creates a fresh `makeTxnId()` when none is supplied; its docs describe `txnId` as the optional id used ÔÇťto deduplicate requests in case of repeated attempts.ÔÇŁ ([matrix-org.github.io](https://matrix-org.github.io/matrix-js-sdk/classes/matrix.MatrixClient.html?utm_source=openai))
- Impact: if the first Matrix PUT reaches the homeserver but the client promise rejects due to a timeout/connection loss before the response is observed, this retry sends the same approval again with a new transaction id. The native approval runtime only tracks the successful returned entry, so the earlier sent-but-thrown prompt is not updated/deleted on approval resolution in `src/infra/approval-handler-runtime.ts:589`. Users can be left with an actionable-looking stale approval message containing command details; the same retry wrapper also wraps direct-room repair at `extensions/matrix/src/approval-handler.runtime.ts:220`, so an ambiguous successful `createDirectRoom` can create duplicate DMs on retry.
- Suggested fix: make Matrix approval delivery retries idempotent. Generate a stable transaction id per approval/target/message part and pass it through the Matrix send wrappers to `matrix-js-sdk`, or restrict retrying to failures known to occur before a Matrix send/create side effect. For direct-room repair, avoid retrying `createDirectRoom` unless the code can prove no room was created.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? No live repro was run; the failure requires an ambiguous Matrix send/create success where the server accepted the request but the client saw an error. The code path and dependency contract are concrete.
- Is this the best way to solve the issue? Unclear; stable Matrix transaction ids for approval retries are the safest direction for message sends, while direct-room creation likely needs a narrower retry boundary or a post-error room discovery step.
