## Findings

### Low: Protocol sync output no longer matches the checked-in generated JSON format

- Kind: regression
- File: `scripts/sync-codex-app-server-protocol.ts`
- Line: 23
- Evidence: The sync script now writes selected JSON schemas with `JSON.stringify(JSON.parse(schemaSource))`, which produces a single-line JSON file. The checked-in generated schemas are still multi-line compact JSON; for example, `extensions/codex/src/app-server/protocol-generated/json/v2/ThreadStartResponse.json` has 1377 lines. A direct local check against the current file showed `same as sync would write false` and `current bytes 49311`, `compact bytes 32637`.
- Impact: Running `pnpm codex-app-server:protocol:sync` will create formatting churn even when schema semantics are unchanged. Because `scripts/check-codex-app-server-protocol.ts` now normalizes JSON before comparing, CI can pass while the committed mirror is not reproducible from the sync command, making future protocol refreshes harder to review.
- Suggested fix: Either regenerate and commit the selected JSON files with the new one-line sync output, or update the sync script to emit the same compact multi-line format that is checked in and add a check that locks that canonical format.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The sync write expression is deterministic, and applying the same expression to the current generated JSON produces different bytes.
- Is this the best way to solve the issue? unclear. Fully regenerating with the new sync output is simplest; preserving the current multi-line compact shape may be better for reviewability if that shape was intentional.
