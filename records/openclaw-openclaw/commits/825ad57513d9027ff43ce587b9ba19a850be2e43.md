# Commit 825ad575

## Summary

Found one current-main gap in the compaction hook-message delivery path: the new forwarding only covers explicit `contextEngine.compact()` recovery/manual paths, while in-attempt auto-compaction events still emit no internal hook messages.

## Findings

### Medium: Compaction notifier messages are still dropped for in-attempt auto-compaction

- Kind: bug
- File: `src/agents/pi-embedded-subscribe.handlers.compaction.ts`
- Line: 17
- Evidence: This commit teaches `runAgentTurnWithFallback` to deliver hook text only when compaction events contain `data.messages`, and wires `onCompactionHookMessages` into the explicit timeout/overflow `contextEngine.compact()` calls. Current `handleCompactionStart()` and `handleCompactionEnd()` still emit `{ phase: "start" }` and `{ phase: "end", ... }` without messages, and they run only the legacy `before_compaction` / `after_compaction` hook runner, not the internal `session:compact:*` hook event whose `event.messages` powers the bundled `compaction-notifier`.
- Impact: Enabling the bundled `compaction-notifier` still cannot produce visible chat notices for the auto-compaction path emitted by the embedded agent subscription handlers, which is the mid-turn compaction case most likely to look stalled to chat users.
- Suggested fix: Route `handleCompactionStart()` / `handleCompactionEnd()` through the same internal `session:compact:before/after` hook event path, or otherwise collect internal hook `event.messages` there and include them in `ctx.params.onAgentEvent({ stream: "compaction", data: { ..., messages } })`. Preserve the existing plugin hook calls or migrate them through the shared helper without dropping their metrics.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. At unit level, trigger `handleCompactionStart()` / `handleCompactionEnd()` with an enabled internal compaction hook that pushes `event.messages`; the current handler never triggers that internal hook and calls `onAgentEvent` with no `messages`, so `runAgentTurnWithFallback` has nothing to deliver.
- Is this the best way to solve the issue? Yes, directionally. The explicit compaction path already has shared `runBeforeCompactionHooks` / `runAfterCompactionHooks` plumbing for internal hook messages; the auto-compaction subscription path should use equivalent plumbing instead of a parallel message-less event.
