# Commit b3ab3cde

## Summary

Found one incomplete-fix path: the new sanitizer covers the normal harness assemble/finalize lifecycle, but context-engine-owned mid-turn compaction still sends hidden runtime-context custom messages directly to context-engine hooks.

## Findings

### Medium: Context-engine loop hook bypasses runtime-context filtering

- Kind: privacy
- File: `src/agents/pi-embedded-runner/tool-result-context-guard.ts`
- Line: 278
- Evidence: This commit filters runtime-context messages in `src/agents/harness/context-engine-lifecycle.ts:75` and rebuilds filtered post-turn snapshots at `src/agents/harness/context-engine-lifecycle.ts:187`. However, when `activeContextEngine.info.ownsCompaction === true`, `src/agents/pi-embedded-runner/run/attempt.ts:1698` installs `installContextEngineLoopHook`, which calls `contextEngine.afterTurn` with raw `sourceMessages` at `src/agents/pi-embedded-runner/tool-result-context-guard.ts:278`, falls back to raw `sourceMessages.slice(...)` for `ingestBatch`/`ingest` at `src/agents/pi-embedded-runner/tool-result-context-guard.ts:291`, and calls `contextEngine.assemble` with raw `sourceMessages` at `src/agents/pi-embedded-runner/tool-result-context-guard.ts:311`. Runtime context is queued as a hidden `openclaw.runtime-context` custom message at `src/agents/pi-embedded-runner/run/runtime-context-prompt.ts:142`, before prompt submission at `src/agents/pi-embedded-runner/run/attempt.ts:3010`, so a tool-loop transform can still expose that hidden message to owning context engines.
- Impact: The changelog says transcript reconstruction plugins should only see conversation messages, but engines that own compaction can still ingest/index/render hidden OpenClaw runtime context during mid-turn loops. That can leak internal runtime context into plugin stores or assembled prompts and reintroduces the behavior this commit is trying to prevent for a supported context-engine mode.
- Suggested fix: Apply the same `stripRuntimeContextCustomMessages` boundary inside `installContextEngineLoopHook`. Recompute the loop `prePromptMessageCount` across filtered pre-prompt and turn slices, then pass the filtered snapshot to `afterTurn`, `ingestBatch`, `ingest`, `assemble`, and the `getRuntimeContext` callback.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Configure an active context engine with `info.ownsCompaction === true`, queue a `customType: "openclaw.runtime-context"` message, then enter a tool-loop transform; the hook path in `tool-result-context-guard.ts` forwards the raw message array to the context engine.
- Is this the best way to solve the issue? yes. The normal lifecycle fix already establishes the intended sanitizer and pre-prompt boundary adjustment; the mid-turn hook needs the same boundary because it duplicates the lifecycle calls.
