# Commit e437763

## Summary

Found one conditional regression in the new non-streaming direct media-block path: captioned media blocks can now trigger an extra final-mode auto-TTS send after the media block was already delivered.

## Findings

### Medium: Captioned media blocks can synthesize duplicate final TTS audio

- Kind: regression
- File: `src/auto-reply/reply/dispatch-from-config.ts`
- Line: 1415
- Evidence: This commit makes non-streaming media-bearing blocks send directly in `src/auto-reply/reply/reply-delivery.ts:160`, and the updated regression test confirms `runReplyAgent` now returns `undefined` after directly sending a captioned media payload in `src/auto-reply/reply/agent-runner.media-paths.test.ts:271`. The dispatch layer accumulates every block payload with text into `accumulatedBlockTtsText` before checking whether the block already has media in `src/auto-reply/reply/dispatch-from-config.ts:1415`. The TTS layer correctly skips applying TTS to the block itself when `reply.hasMedia` is true in `extensions/speech-core/src/tts.ts:1574`, but later, because the direct block send suppresses the final reply and leaves `replies.length === 0`, final-mode auto-TTS synthesizes a new audio-only reply from the accumulated block text in `src/auto-reply/reply/dispatch-from-config.ts:1562`.
- Impact: With `messages.tts` enabled in final mode, an agent TTS/tool media block such as `{ text: "spoken confirmation", mediaUrls: ["/tmp/voice.opus"], audioAsVoice: true }` can deliver the intended voice/media block, then generate and send a second TTS audio reply from the same caption text. That duplicates user-facing audio and can incur an unnecessary TTS provider call.
- Suggested fix: Do not add media-bearing block replies to `accumulatedBlockTtsText` for final-mode synthetic TTS, or track whether accumulated text came from already-media-bearing blocks and skip the final synthetic TTS fallback when there is no unsent text-only block content.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes: enable final-mode auto-TTS, disable block streaming, have the agent emit a captioned media-bearing block reply that is also suppressed from final payloads by `directlySentBlockKeys`, then observe one `sendBlockReply`/route send for the original media and one later final TTS-only send from `accumulatedBlockTtsText`.
- Is this the best way to solve the issue? Yes: final-mode synthetic TTS is meant to recover audio for text that was streamed as blocks and therefore has no final reply.
