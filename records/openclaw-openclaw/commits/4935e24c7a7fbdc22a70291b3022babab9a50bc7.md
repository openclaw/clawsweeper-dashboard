# Commit 4935e24c

## Summary

Found one remaining Control UI run-lifecycle regression in the path this commit is meant to harden: terminal `sessions.changed` updates can still leave the local `chatRunId` active, so the composer keeps rendering Stop / In progress even after the session row is terminal.

## Findings

### Medium: terminal session updates do not reconcile the local active run

- Kind: bug
- File: `ui/src/ui/controllers/sessions.ts`
- Line: 332
- Evidence: `applySessionsChangedEvent` turns a terminal row update into `hasActiveRun = false`, but `ui/src/ui/app-gateway.ts:871` returns immediately when a `sessions.changed` update is applied. That skips the new `reconcileChatRunFromCurrentSessionRow` helper, which is only wired after full `sessions.list` reloads. Because `hasAbortableSessionRun` returns `true` as soon as `chatRunId` is set (`ui/src/ui/app-chat.ts:101`), the row can say `done` while the UI still treats the run as abortable.
- Impact: if the `chat` terminal event is missed but the terminal `sessions.changed` event arrives, the original stale Stop-button / In progress state remains. Follow-up sends can still be treated as active-run queue/steer input instead of a clean next turn.
- Suggested fix: after a successful current-session terminal `sessions.changed` update, call `reconcileChatRunFromCurrentSessionRow` or route the applied result through `reconcileChatRunLifecycle` before returning from the gateway event handler. Add a regression test where `chatRunId` is set, `sessions.changed` applies `{status: "done"}`, and `hasAbortableSessionRun` becomes false.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A read-only `pnpm exec tsx -e` probe applied a terminal session update to a state with `chatRunId: "run-1"`; the row became `{hasActiveRun: false, status: "done"}`, but `chatRunId` stayed `"run-1"` and `hasAbortableSessionRun(state)` returned `true`.
- Is this the best way to solve the issue? yes. The commit already adds the right reconciliation helper; the missing piece is invoking it for incremental terminal session updates, not only full session-list reloads.
