# Commit 2b82c05

## Summary

Found one incomplete inherited-persona startup path: Discord voice `voice.tts` overrides are still scanned raw, even though runtime merges them over `messages.tts`.

## Findings

### Medium: Discord voice persona providers can still be omitted from gateway startup

- Kind: bug
- File: `src/plugins/gateway-startup-speech-providers.ts`
- Line: 171
- Evidence: The new code resolves inherited config for `agents.list[].tts`, `channels.<id>.tts`, and `channels.<id>.accounts.<id>.tts`, but still passes `channelConfig.voice.tts` and `accountConfig.voice.tts` directly at lines 171 and 183. Discord voice runtime merges `channels.discord.voice.tts` over `messages.tts` before resolving TTS, so a voice override like `voice.tts.persona: "narrator"` should inherit `messages.tts.personas.narrator.provider`. A focused startup-plan check confirms `channels.discord.tts.persona` includes `microsoft`, while `channels.discord.voice.tts.persona` and account `voice.tts.persona` do not.
- Impact: Users who configure a Discord voice persona by inheriting the persona definition/provider from `messages.tts` can still start the gateway without the required speech provider plugin in the startup set, preserving the same class of TTS startup activation failure this commit is fixing for channel/account TTS.
- Suggested fix: Apply the same effective merge to voice TTS startup scanning, for example by constructing a config where `messages.tts` is merged with `channelConfig.voice.tts` / `accountConfig.voice.tts`, or by adding a shared helper matching `extensions/discord/src/voice/tts.ts` merge semantics. Add regression cases for `channels.discord.voice.tts.persona` and `channels.discord.accounts.<id>.voice.tts.persona`.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the startup-plan omission: `resolveGatewayStartupPluginIds` includes `microsoft` for `channels.discord.tts.persona`, but not for equivalent `channels.discord.voice.tts.persona` or account voice persona configs.
- Is this the best way to solve the issue? Unclear. The safest direction is to reuse or centralize the same TTS merge semantics used by Discord voice runtime, rather than adding another ad hoc scan rule.
