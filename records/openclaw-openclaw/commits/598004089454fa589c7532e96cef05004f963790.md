# Commit 5980040894

## Summary

Found one regression: Gateway startup now pins `getRuntimeConfig()` to a config snapshot profile that intentionally omits load-time defaults.

## Findings

### Medium: Gateway runtime loses default compaction safeguards after startup

- Kind: regression
- File: `src/gateway/server.impl.ts`
- Line: 598
- Evidence: The new `setRuntimeConfigSnapshot(cfgAtStart, startupLastGoodSnapshot.sourceConfig)` pins the process runtime config. `cfgAtStart` comes from `configSnapshot.config`, and valid config snapshots are materialized with `"snapshot"` mode in `src/config/io.ts:1815`; that profile excludes compaction and context-pruning defaults in `src/config/materialize.ts:37`. The previous lazy `getRuntimeConfig()` path uses `"load"` mode via `readBestEffortConfig`/`loadConfig`, which includes those defaults in `src/config/io.ts:1940`. A focused probe confirmed the mismatch: snapshot/pinned runtime compaction was `null`, while load-mode compaction was `{ "mode": "safeguard" }`.
- Impact: Configs with `agents.defaults` but no explicit `agents.defaults.compaction.mode` no longer get the default `"safeguard"` compaction mode in live Gateway runtime lookups. `src/agents/pi-embedded-runner/extensions.ts:126` falls back to `"default"` unless it sees `compaction.mode === "safeguard"`, so long-running agent sessions can lose the default safeguard extension and hit context overflow or weaker compaction behavior. Anthropic load-time context-pruning defaults are also omitted for the same reason.
- Suggested fix: Pin a load-materialized runtime config, e.g. apply `materializeRuntimeConfig(cfgAtStart, "load")` before `setRuntimeConfigSnapshot`, while keeping `startupLastGoodSnapshot.sourceConfig` for source/write comparisons. Add a startup/runtime test asserting `getRuntimeConfig().agents.defaults.compaction.mode === "safeguard"` when `agents.defaults` exists and compaction is unset.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A temp-config probe comparing snapshot, load, and pinned runtime output showed:
  ```json
  {
    "snapshotCompaction": null,
    "loadCompaction": { "mode": "safeguard" },
    "pinnedRuntimeCompaction": null
  }
  ```
- Is this the best way to solve the issue? yes. Load-materializing before pinning preserves the commit’s goal of avoiding another config file read while keeping the prior `getRuntimeConfig()` contract.
