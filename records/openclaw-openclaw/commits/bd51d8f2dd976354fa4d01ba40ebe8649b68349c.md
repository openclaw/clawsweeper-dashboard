# Commit bd51d8f2

## Summary

Found one regression: the new generic partial-preview dedupe drops final payloads before channel dispatch, but Slack draft-preview delivery needs that final payload to edit and mark the draft preview as the committed answer.

## Findings

### Medium: Preview dedupe bypasses Slack draft finalization

- Kind: regression
- File: `src/auto-reply/reply/agent-runner-payloads.ts`
- Line: 435
- Evidence: `runReplyAgent()` records any `opts.onPartialReply` text and passes it as `previewStreamedText` into `buildReplyPayloads()` (`src/auto-reply/reply/agent-runner.ts:1280`, `src/auto-reply/reply/agent-runner.ts:1754`). When block streaming is off, `buildReplyPayloads()` now drops matching final text payloads (`src/auto-reply/reply/agent-runner-payloads.ts:435`). Slack enables `onPartialReply` for draft previews (`extensions/slack/src/monitor/message-handler/dispatch.ts:1201`) and forces block streaming off whenever a Slack draft stream is active (`extensions/slack/src/monitor/message-handler/dispatch.ts:187`). The final payload is the path that calls `deliverWithFinalizableLivePreviewAdapter()` and runs `onPreviewFinalized`, which marks the draft as committed and records delivery (`extensions/slack/src/monitor/message-handler/dispatch.ts:898`, `extensions/slack/src/monitor/message-handler/dispatch.ts:948`). If the final text exactly matches the preview, the new core filter returns no final reply, so `dispatch-from-config` has nothing to send through the Slack finalizer (`src/auto-reply/reply/dispatch-from-config.ts:1657`).
- Impact: Slack users with draft preview streaming can be left with an unfinalized draft preview and missing final delivery bookkeeping for the common case where the final answer equals the streamed preview.
- Suggested fix: Do not run this dedupe for every `onPartialReply` user. Scope it to the Telegram path or pass an explicit channel capability/option that says a preview has already been finalized and final text-only payloads should be suppressed. Channels with finalizable draft previews still need the final payload delivered to their finalizer.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Source path: enable Slack preview streaming so `shouldUseDraftStream` is true, have the agent emit `onPartialReply({ text: "done" })` and return final payload `{ text: "done" }`; `buildReplyPayloads()` suppresses the final reply before Slack’s `deliverWithFinalizableLivePreviewAdapter()` can run.
- Is this the best way to solve the issue? yes. The dedupe needs a narrower contract because `onPartialReply` means “draft preview update” for Slack, not “final delivery already completed.”
