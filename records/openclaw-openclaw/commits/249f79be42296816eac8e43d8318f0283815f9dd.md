# Commit 249f79be

## Summary

Found one supply-chain regression in the new portable optional bundled dependency install path. Current `main` still has the issue.

## Findings

### Medium: Portable optional dependency install can ignore the shrinkwrap and bundle newer code

- Kind: supply_chain
- File: `scripts/lib/plugin-npm-package-manifest.mjs`
- Line: 254
- Evidence: `collectMissingOptionalBundledDependencySpecs` builds install specs from the installed dependency package’s `optionalDependencies` manifest range, then `installMissingOptionalBundledDependencies` runs `npm install ... --package-lock=false --save=false ... <name>@<spec>` at `scripts/lib/plugin-npm-package-manifest.mjs:278`. A temp repro with `npm-shrinkwrap.json` locking `picocolors` to `1.0.0` but an optional range of `^1.0.0` installed `picocolors` `1.1.1` during this second pass.
- Impact: plugin publish can pass the shrinkwrap check and then bundle optional dependency code that was not reviewed or pinned by the plugin shrinkwrap. This weakens the intended package reproducibility/security boundary for bundled plugin packages.
- Suggested fix: resolve missing optional install specs from `npm-shrinkwrap.json` package entries, using the locked version/resolved/integrity or locked `file:` target, instead of reinstalling from dependency manifest ranges. Add a regression test where the optional range admits a newer version than the shrinkwrap pin.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. In a temp npm package, run the same `npm ci` command followed by the commit’s `npm install --force --package-lock=false --save=false picocolors@^1.0.0`; npm 11.12.1 installs `picocolors` `1.1.1` despite the shrinkwrap entry for `1.0.0`.
- Is this the best way to solve the issue? yes. The publish path already requires and checks `npm-shrinkwrap.json`, so the missing optional dependency repair should consume that lock data rather than asking npm to resolve ranges again.
