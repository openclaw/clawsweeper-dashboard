# Commit a224810

## Summary

Found one regression risk from making omitted `sessions.list.limit` mean “100 rows” globally: some internal authorization and resolution paths still call `sessions.list` as a complete visibility set, so older spawned child sessions can become invisible once a parent has more than 100 visible children.

## Findings

### Medium: default list cap can deny access to older spawned child sessions

- Kind: regression
- File: `src/gateway/session-utils.ts`
- Line: 2005
- Evidence: `listSessionsFromStore` now passes `defaultLimit: SESSIONS_LIST_DEFAULT_LIMIT`, so any caller that omits `limit` receives only the newest 100 filtered rows. Existing internal callers still omit a limit when they need complete spawned-session visibility: `src/plugin-sdk/session-visibility.ts:43` calls `sessions.list` with `spawnedBy` and no `limit`, then `src/plugin-sdk/session-visibility.ts:233` denies tree access if the target key is absent from that truncated set. `src/gateway/sessions-resolve.ts:67` also checks key visibility by calling `listSessionsFromStore` with `spawnedBy` and no explicit limit, so key-based `sessions.resolve` can return “No session found” for a valid older child.
- Impact: a sandboxed/default `tree`-visibility agent session with more than 100 spawned children can lose access to older children by key for `sessions_history`, `sessions_send`, and status/visibility checks. This is a behavioral regression from the previous unbounded internal visibility lookup and can make retained sessions look unauthorized or missing even though they still exist.
- Suggested fix: keep the bounded default for public/display `sessions.list`, but do not reuse it for completeness-sensitive internal lookups. For example, make `isResolvedSessionKeyVisible` use `filterAndSortSessionEntries` or a direct exact metadata check, and make `listSpawnedSessionKeys` request an explicit unbounded/completeness mode or page until `hasMore` is false.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Create 101 session entries with the same `spawnedBy`, descending `updatedAt`, then resolve/check the oldest child by key under that same `spawnedBy`; the default cap omits it from the visibility set.
- Is this the best way to solve the issue? no. The public cap is reasonable for the reported Gateway starvation issue, but internal authorization/resolution paths need complete membership checks instead of truncated display lists.
