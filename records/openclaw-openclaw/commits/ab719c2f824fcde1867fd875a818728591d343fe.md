# Commit ab719c2

## Summary

The core fix works for the tested allowlist path, but two Telegram consumers still read raw `accounts.<id>.groups ?? channels.telegram.groups`. In the new single-account `{}` case, those paths still treat the empty map as authoritative and miss the effective root groups.

## Findings

### Medium: Topic-level `requireMention` root config is still masked by `accounts.default.groups: {}`

- Kind: bug
- File: `extensions/telegram/src/group-policy.ts`
- Line: 38
- Evidence: `resolveTelegramRequireMention` reads `cfg.channels?.telegram?.accounts?.[accountId]?.groups ?? cfg.channels?.telegram?.groups`, so an explicit `{}` prevents its topic-aware lookup from seeing root `groups`. It then falls back to `resolveChannelGroupRequireMention` with only the chat id, which cannot apply Telegram `topics.<id>.requireMention`. A focused probe with root `groups["-1001"].requireMention=true`, root `topics["77"].requireMention=false`, and `accounts.default.groups={}` returned `true` for `resolveTelegramGroupRequireMention({ accountId: "default", groupId: "-1001:topic:77" })`; the effective merged root topic config should return `false`.
- Impact: Telegram topic workflows that rely on root topic `requireMention: false` can still require explicit mentions after this commit when the single account has `groups: {}`. That contradicts the commit’s new “root groups still apply” behavior for the affected config shape.
- Suggested fix: make the Telegram topic-aware resolver use the same effective group map as `mergeTelegramAccountConfig`, or extract a shared Telegram effective-groups helper and add a regression test for root topic overrides with `accounts.default.groups: {}`.
- Confidence: high

### Low: Telegram status/probe diagnostics still ignore effective root groups

- Kind: bug
- File: `extensions/telegram/src/channel.ts`
- Line: 813
- Evidence: `auditAccount` and `resolveAccountSnapshot` both compute groups from `cfg.channels?.telegram?.accounts?.[account.accountId]?.groups ?? cfg.channels?.telegram?.groups`. A focused helper probe showed `mergeTelegramAccountConfig(cfg, "default").groups` resolves `{"-100123":{"requireMention":false}}`, while the raw status expression resolves `{}`; consequently `collectTelegramUnmentionedGroupIds` sees `["-100123"]` with the effective groups but `[]` on the status path.
- Impact: `openclaw channels status --probe` can skip membership probing and fail to surface the BotFather privacy warning for root `requireMention: false` groups in the exact single-account empty-map setup this commit fixes.
- Suggested fix: use the same effective Telegram groups helper in status audit/snapshot code and add a status regression test for `accounts.default.groups: {}` plus root `requireMention: false`.
- Confidence: high

- Is this the best way to solve the issue? yes; centralizing effective Telegram group resolution avoids more raw `??` fallbacks drifting from the fixed contract.
