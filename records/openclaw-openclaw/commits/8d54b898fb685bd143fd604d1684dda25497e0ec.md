# Commit 8d54b898

## Summary

Found one concrete runtime-resolution regression risk in the QA Lab runtime surface. The focused source/Vitest checks pass, but the changed import bypasses the existing private QA plugin-loader alias path and is only covered by TS/Vitest path aliases.

## Findings

### Medium: QA Lab runtime imports QA Channel through a specifier the plugin loader does not alias

- Kind: regression
- File: `extensions/qa-lab/src/runtime-api.ts`
- Line: 23
- Evidence: The commit changed QA Lab’s runtime re-export from `openclaw/plugin-sdk/qa-channel` to `@openclaw/qa-channel/api.js`. `extensions/qa-lab/package.json` does not declare `@openclaw/qa-channel` as a dependency, and after `pnpm install`, `node -e "import('@openclaw/qa-channel/api.js')"` fails with `ERR_MODULE_NOT_FOUND`. The source/Vitest path is masked by aliases: `tsconfig.json` maps `@openclaw/*` to `extensions/*`, and `test/vitest/vitest.shared.config.ts` has a dedicated `@openclaw/qa-channel/api.js` alias. The runtime plugin-loader alias map under `OPENCLAW_ENABLE_PRIVATE_QA_CLI=1` includes the old private SDK specifiers (`openclaw/plugin-sdk/qa-channel`, `@openclaw/plugin-sdk/qa-channel`) but not `@openclaw/qa-channel/api.js`.
- Impact: QA Lab’s runtime facade can fail when loaded outside TS/Vitest path resolution or from built/runtime plugin-loader flows, even though source tests pass. That can break private QA runtime consumers that load `qa-lab/runtime-api.js`.
- Suggested fix: Keep the runtime surface on `openclaw/plugin-sdk/qa-channel`, or add an explicit runtime/plugin-loader alias plus package metadata coverage for `@openclaw/qa-channel/api.js`. Add a non-Vitest facade/load check so this cannot be hidden by test aliases.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Partially. The missing package resolution and alias-map mismatch are directly reproducible. I did not get a complete `loadQaRuntimeModule()` end-to-end failure because that exploratory local load hung and was killed.
- Is this the best way to solve the issue? Unclear. Reverting to the previous private SDK facade is the safest narrow fix unless the intent is to make `@openclaw/qa-channel` a real runtime dependency and loader-supported specifier.
