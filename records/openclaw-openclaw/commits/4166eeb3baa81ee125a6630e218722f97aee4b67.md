# Commit 4166eeb

## Summary

Found one compatibility regression in the new prerelease `minHostVersion` support. The code now accepts prerelease floors, but the comparison still ignores prerelease identifiers, so older same-base prereleases can pass the gate.

## Findings

### Medium: Prerelease host floors accept older same-base prereleases

- Kind: compatibility
- File: `src/plugins/min-host-version.ts`
- Line: 84
- Evidence: `checkMinHostVersion` parses both the host and required floor with `parseSemver` from `src/infra/runtime-guard.ts:24`, which only keeps `major/minor/patch`. It then calls `isAtLeast` at `src/plugins/min-host-version.ts:85`, so `2026.5.1-alpha.1`, `2026.5.1-beta.0`, and `2026.5.1-beta.1` all compare as the same version. A direct probe on current `main` returned `ok: true` for both `currentVersion: "2026.5.1-beta.0"` and `"2026.5.1-alpha.1"` with `minHostVersion: ">=2026.5.1-beta.1"`.
- Impact: External plugin install and non-bundled manifest loading can accept a plugin that requires a later beta API on an older same-date prerelease host. That can let incompatible plugin code install/load and fail later at runtime instead of being blocked by the version gate.
- Suggested fix: Use the prerelease-aware comparator already present in `src/infra/semver-compare.ts` for `minHostVersion` checks, or add equivalent SemVer prerelease comparison here. Add tests that reject `2026.5.1-beta.0` and `2026.5.1-alpha.1` for `>=2026.5.1-beta.1`, while accepting `2026.5.1` stable.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: run `pnpm exec tsx -e "import { checkMinHostVersion } from './src/plugins/min-host-version.ts'; console.log(checkMinHostVersion({ currentVersion: '2026.5.1-beta.0', minHostVersion: '>=2026.5.1-beta.1' }));"` and observe `ok: true`.
- Is this the best way to solve the issue? no; accepting prerelease floor syntax is useful, but the check must use prerelease-aware ordering when the floor includes prerelease metadata.
