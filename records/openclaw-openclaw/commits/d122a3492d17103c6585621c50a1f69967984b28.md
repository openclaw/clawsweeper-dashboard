# Commit d122a3492

## Summary

Found one concrete regression in the oversized-line metadata recovery path. The new prefix scanner can mistake nested message payload fields for top-level transcript tree metadata, causing legacy linear transcripts to be treated as tree transcripts and dropping surrounding chat history.

## Findings

### Medium: Nested `id`/`parentId` fields in oversized legacy rows can hide recent history

- Kind: regression
- File: `src/gateway/session-utils.fs.ts`
- Line: 302
- Evidence: `buildOversizedTranscriptRecord()` extracts `"id"`, `"parentId"`, `"type"`, and `"role"` with unscoped regexes over the first 64 KiB of the whole JSON line. `tailRecordHasTreeLink()` then treats any record with an `id` and a present `parentId` as a tree node. A valid legacy transcript row like `{message:{role:"assistant",content:[{output:{id:"nested-leaf",parentId:"missing-parent",payload:"..."}}]}}` is therefore converted into a fake tree leaf. A focused smoke using `readRecentSessionMessagesAsync()` returned only the oversized placeholder with `__openclaw.id: "nested-leaf"` and dropped the surrounding `"before"` and `"after"` messages.
- Impact: `chat.history` can lose recent messages for legacy or non-tree transcript files when an oversized message contains arbitrary nested JSON fields named `id` and `parentId`, such as tool output or imported data. This regresses the linear-transcript fallback that should preserve nearby small messages while replacing only the oversized row.
- Suggested fix: only recover tree metadata from top-level transcript entry fields. A small JSON prefix scanner should track object depth and stop before the `message` value, or otherwise only mark an oversized placeholder as a tree node when top-level `type`, `id`, and `parentId` were confidently found. If top-level metadata is uncertain, keep the placeholder non-tree as before.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Create a v1-style JSONL transcript with a small user row, an oversized assistant row whose nested payload contains `id` and `parentId`, and a later small user row; then call `readRecentSessionMessagesAsync(sessionId, storePath, undefined, { maxMessages: 10 })`. Current code returns only the placeholder instead of preserving the small rows.
- Is this the best way to solve the issue? yes for the suggested direction. The pinned Pi session format documents top-level `id`/`parentId` as entry metadata and the session header as not part of the tree, so the fallback parser should not infer tree structure from arbitrary nested message content.
