# Commit b8f6e16

## Summary

Found one concrete update-ordering gap: the new OpenClaw release comparator is only reached for bare package versions, not `v`-prefixed git tags, so source-checkout beta channel selection can still order same-day stable correction tags below beta tags.

## Findings

### Medium: Git beta channel can skip same-day stable correction tags

- Kind: bug
- File: `src/infra/update-check.ts`
- Line: 389
- Evidence: `compareSemverStrings` now calls `compareOpenClawReleaseVersions(a, b)`, but that parser only accepts `YYYY.M.D...` forms, not the `vYYYY.M.D...` tag forms used by git release tags. A focused check showed `compareSemverStrings("v2026.5.3-beta.9", "v2026.5.3-1")` returns `1`, while the same bare versions return `-1`. `update-runner` passes git tags directly into this helper when deciding whether beta is older than stable (`src/infra/update-runner.ts:307`).
- Impact: For source checkouts on the beta channel, a tag set like `v2026.5.3-beta.9`, `v2026.5.3-1`, and `v2026.5.3` can choose the beta tag instead of the stable correction tag, leaving users on an older prerelease path and missing the correction.
- Suggested fix: Normalize an optional leading `v` before `compareOpenClawReleaseVersions`, or teach the OpenClaw release parser to accept `v` prefixes. Add coverage for `compareSemverStrings("v2026.5.3-1", "v2026.5.3")` and the beta-channel git tag selection case.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: `pnpm exec tsx --eval "import { compareSemverStrings } from './src/infra/update-check.ts'; console.log(compareSemverStrings('v2026.5.3-beta.9','v2026.5.3-1')); console.log(compareSemverStrings('2026.5.3-beta.9','2026.5.3-1'));"` prints `1` then `-1`.
- Is this the best way to solve the issue? yes: accepting the same optional `v` prefix already supported by the generic semver parser keeps npm package-version behavior unchanged while fixing git-tag callers.
