# Commit c89da2a

## Summary

Found one low-severity logging regression in the new liveness throttling split. The current `main` code still has the same diagnostic logic; later commits only touched the changelog for these paths.

## Findings

### Low: Telemetry-only liveness events can bypass the visible warning cooldown

- Kind: reliability
- File: `src/logging/diagnostic.ts`
- Line: 312
- Evidence: `startDiagnosticHeartbeat` computes `shouldEmitLivenessReport` as `shouldEmitLivenessEvent || shouldEmitLivenessWarning`, then calls `emitDiagnosticLivenessWarning` for either case. Inside `emitDiagnosticLivenessWarning`, the log level is chosen only from `hasOpenDiagnosticWork(work)`, so a telemetry event that passed the event cooldown but failed the warning cooldown still logs via `diag.warn` whenever work is open.
- Impact: after an idle liveness event, if work becomes queued/active soon after, visible Gateway warning logs can recur sooner than `DEFAULT_LIVENESS_WARN_COOLDOWN_MS`. Example: idle event at 30s, active warning at 60s, then another visible warning at 150s because the event cooldown has elapsed from 30s even though only 90s elapsed since the last warning at 60s. This partially undermines the commitâ€™s goal of reducing liveness warning noise.
- Suggested fix: pass an explicit `logAsWarning` boolean from `shouldEmitLivenessWarning` into the emitter, and use `diag.warn` only when that boolean is true; event-only reports should stay telemetry/debug even when work is open.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. With fake timers and a constant non-null liveness sample: start diagnostics, advance 30s idle, queue a message, advance 30s for the first active warning, then advance 90s. The current code reaches the event-only path and still calls `diag.warn` because work remains open.
- Is this the best way to solve the issue? Yes. The separate event cooldown is useful for keeping idle samples in telemetry, but visible log severity should be driven by the warning throttle result rather than recomputed inside the emitter from work presence alone.
