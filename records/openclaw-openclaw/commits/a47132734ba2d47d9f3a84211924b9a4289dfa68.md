# Commit a471327

## Summary

Found one bootstrap-routing regression in an edge path for completed `continuation-skip` turns when the workspace is still bootstrap-pending but the run initially lacks `read` tool access.

## Findings

### Medium: Pending BOOTSTRAP can no longer upgrade limited routing to full injection on completed continuation turns without read access

- Kind: regression
- File: `src/agents/pi-embedded-runner/run/attempt.ts`
- Line: 1133
- Evidence: On a completed `continuation-skip` turn, the new code first calls `resolveBootstrapRouting()` without preloaded files. If the workspace is bootstrap-pending but `bootstrapHasFileAccess` is false, routing is `limited`; line 1133 then skips `resolveBootstrapFilesForRun`. `resolveAttemptBootstrapContext` treats any non-`full` bootstrap mode as a continuation and returns empty bootstrap context when the completed marker exists (`src/agents/pi-embedded-runner/run/attempt.context-engine-helpers.ts:37`). But routing can only upgrade no-file-access runs to `full` when `bootstrapFiles` contain BOOTSTRAP content (`src/agents/pi-embedded-runner/run/attempt-bootstrap-routing.ts:81`), and the parent commit always preloaded those files before routing.
- Impact: A user-facing constrained run, such as `disableTools`, a no-tool model, or a tool construction plan with no `read` tool, can have a still-pending canonical `BOOTSTRAP.md` after a prior completed full bootstrap marker. Before this commit, the runner could preload BOOTSTRAP and inject the full bootstrap content; after this commit it stays in limited mode and only gives generic pending-bootstrap wording, so the agent misses the actual BOOTSTRAP instructions.
- Suggested fix: For the `shouldProbeContinuationSkip` path, only skip preloading when the cheap route is `none`. If the cheap route is `limited`, resolve bootstrap files once and reroute so real BOOTSTRAP content can still upgrade file access to `full`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes, by creating a completed-bootstrap session marker, leaving canonical `BOOTSTRAP.md` pending, configuring `contextInjection: "continuation-skip"`, and running a user-triggered primary attempt with no initial `read` access.
- Is this the best way to solve the issue? yes, the targeted fix preserves the intended latency win for “no pending bootstrap” continuation turns while keeping the existing pending-bootstrap upgrade contract.
