# Commit 010f7a58

## Summary

Found one release metadata regression: three already-published plugin packages gained install metadata without version bumps, so that metadata cannot be published for those package versions.

## Findings

### Medium: Install metadata changes target already-published plugin versions

- Kind: supply_chain
- File: `extensions/diagnostics-prometheus/package.json`
- Line: 17
- Evidence: This commit adds `openclaw.install.npmSpec/defaultChoice` to `diagnostics-prometheus`, `lobster`, and `voice-call`, but leaves their versions at `2026.5.1-beta.2`, `2026.5.1-beta.1`, and `2026.5.1-beta.1`. `pnpm release:plugins:clawhub:check -- --base-ref 10c8b9085a447a8e08a977eb09be8ae578baf815 --head-ref 010f7a58a1aaa0efff541a6844c7b3d684a2bacc` fails with version-bump errors for all three. `scripts/plugin-npm-release-plan.ts` also reports all three as `alreadyPublished: true`, so the npm publish lane will skip them.
- Impact: The newly added install metadata will remain repo-local for those three package versions; users or registries consuming the already-published package manifests still see the old metadata. `npm view` confirmed the published `@openclaw/diagnostics-prometheus@2026.5.1-beta.2` and `@openclaw/lobster@2026.5.1-beta.1` manifests lack `openclaw.install`, while `@openclaw/voice-call@2026.5.1-beta.1` has only `minHostVersion`, not the new `npmSpec/defaultChoice`.
- Suggested fix: Bump the three plugin package versions before changing publishable package metadata, or defer these metadata edits until the next publishable version bump.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: run the ClawHub release check for the reviewed range; it deterministically rejects the unchanged versions.
- Is this the best way to solve the issue? yes: the existing release contract requires version bumps for changed ClawHub-publishable plugin manifests, and npm cannot republish the same package version.
