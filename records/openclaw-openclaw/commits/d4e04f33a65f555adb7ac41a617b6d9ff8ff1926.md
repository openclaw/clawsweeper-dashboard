# Commit d4e04f33a6

## Summary

Found two concrete issues in the new `--fix-dm-scope` applied-cleanup path: retired sessions leave trajectory sidecars behind, and applied JSON summaries misreport entry counts.

## Findings

### Medium: `--fix-dm-scope` leaves retired-session trajectory artifacts behind

- Kind: privacy
- File: `src/config/sessions/cleanup-service.ts`
- Line: 451
- Evidence: The dmScope retirement apply path archives removed transcripts with `archiveRemovedSessionTranscripts(...)` but never calls `removeRemovedSessionTrajectoryArtifacts(...)`. The normal stale/cap deletion path in `src/config/sessions/store.ts:322` does call that trajectory cleanup helper after archiving. A focused temp-store smoke with a retired direct-DM row plus `direct-session.trajectory.jsonl` and `direct-session.trajectory-path.json` showed the transcript was archived, but both trajectory files remained in the sessions directory after cleanup.
- Impact: Operators using the new cleanup to retire stale direct-DM rows can remove the row from `sessions.json` while leaving unreferenced runtime traces and pointer files on disk. Those sidecars can contain detailed session/runtime data and, in warn/default mode or while fresh, may survive outside the explicit cleanup action.
- Suggested fix: After `archiveRemovedSessionTranscripts(...)`, call `removeRemovedSessionTrajectoryArtifacts(...)` with the same `dmScopeRemovedSessionFiles`, referenced IDs, `storePath`, and `restrictToStoreDir: true`. Extend the new integration test to create pointer/runtime sidecars and assert they are removed.
- Confidence: high

### Low: Applied cleanup counts exclude the rows removed by `--fix-dm-scope`

- Kind: bug
- File: `src/config/sessions/cleanup-service.ts`
- Line: 512
- Evidence: `retireMainScopeDirectSessionEntries(...)` deletes dmScope rows inside the `updateSessionStore` mutator before `saveSessionStoreUnlocked` computes `appliedReport.beforeCount`. The applied summary then copies `appliedReport.beforeCount` and `afterCount`, so a temp-store smoke with two initial rows returned `beforeCount: 1`, `afterCount: 1`, and `dmScopeRetired: 1`.
- Impact: CLI/Gateway JSON consumers get internally inconsistent cleanup summaries for the new repair: the result says a direct-DM row was retired but shows no entry-count reduction. That weakens auditability for an operator-facing cleanup command.
- Suggested fix: Preserve the pre-mutation count for explicit repair removals, or build applied summaries from the preview plus the actual post-apply store count. Add an assertion to the applied `--fix-dm-scope` integration test for `beforeCount: 2` and `afterCount: 1`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Seed a temp `sessions.json` with `agent:main:main` plus `agent:main:telegram:direct:<peer>`, create transcript and trajectory sidecar files for the direct row, then run `runSessionsCleanup({ opts: { enforce: true, fixDmScope: true } })`.
- Is this the best way to solve the issue? yes. Reusing the existing removed-session trajectory cleanup keeps the new explicit repair aligned with the established stale/cap deletion behavior.
