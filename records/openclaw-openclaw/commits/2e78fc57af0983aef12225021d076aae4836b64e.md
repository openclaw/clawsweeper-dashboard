## Summary

Found one regression gap in the fix: the exact `auth-profiles.json` shape from #69708 can still be dropped, now as `missing_provider` instead of `invalid_type`.

## Findings

### Medium: aws-sdk profiles without an inline provider are still rejected

- Kind: regression
- File: `src/agents/auth-profiles/persisted.ts`
- Line: 147
- Evidence: #69708â€™s reproducer stores `"amazon-bedrock:default": { "type": "aws-sdk", "createdAt": ... }` without a `provider` field. Current persisted-store loading calls `parseCredentialEntry(value)` with no fallback provider, while `parseCredentialEntry` rejects any entry whose normalized `provider` is absent. A focused smoke on current `main` printed `profiles []` and then `No credentials found for profile "amazon-bedrock:default"` for that exact shape. The new regression tests cover only the modified shape that adds `provider: "amazon-bedrock"`.
- Impact: Existing EC2/IMDS Bedrock deployments that have the reported profile shape remain broken when an explicit `amazon-bedrock:default` profile is selected or ordered; the commit closes the `invalid_type` path but still silently drops the profile during store load.
- Suggested fix: When loading persisted `auth-profiles.json`, infer the provider from the profile id prefix for secretless `aws-sdk` profiles that omit `provider` (for example `amazon-bedrock:default` -> `amazon-bedrock`), and add a regression test using the exact #69708 JSON. Keep `missing_provider` for entries where no safe provider can be inferred.
- Confidence: high
