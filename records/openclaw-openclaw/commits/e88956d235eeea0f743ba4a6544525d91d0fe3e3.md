## Findings

### High: Google Chat auth test now dereferences a nullable credential result

- Kind: regression
- File: `extensions/googlechat/src/google-auth.runtime.test.ts`
- Line: 457
- Evidence: The commit changed the tests to store `const credentials = await resolveValidatedGoogleChatCredentials(...)` and then read `credentials.client_email`, `credentials.token_uri`, and `credentials.type` at lines 457-459 and 499-501. The resolver is typed as `Promise<GoogleChatServiceAccountCredentials | null>` in `extensions/googlechat/src/google-auth.runtime.ts:546`, so the extension test typecheck lane reports `TS18047: 'credentials' is possibly 'null'` for all six property reads.
- Impact: `pnpm tsgo:extensions:test` fails on the touched file, so the extension test typecheck / test-types gate is broken even though the runtime Vitest file passes.
- Suggested fix: keep the awaited value but narrow it before property access, for example `if (!credentials) throw new Error("Expected credentials");`, or assert the object directly with `expect(credentials).toMatchObject(...)` without dereferencing nullable properties.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: run `pnpm tsgo:extensions:test`; it reports `TS18047` on `extensions/googlechat/src/google-auth.runtime.test.ts:457`, `458`, `459`, `499`, `500`, and `501`.
- Is this the best way to solve the issue? yes: the runtime behavior is correct for file credentials, but the test must satisfy the resolver’s public nullable type before property access.
