Found one low-severity user-visible regression: one first-run FAQ still says Bedrock is bundled even though this commit externalizes it to npm.

## Findings

### Low: Bedrock first-run FAQ still says the provider is bundled

- Kind: regression
- File: `docs/help/faq-first-run.md`
- Line: 592
- Evidence: The FAQ says OpenClaw has a bundled Amazon Bedrock provider. This commit excludes `dist/extensions/amazon-bedrock/**` and `dist/extensions/amazon-bedrock-mantle/**` from the root package, marks `extensions/amazon-bedrock/package.json` with `openclaw.build.bundledDist: false`, and the generated plugin reference now says `Install route: npm`.
- Impact: Users following first-run docs can reasonably expect Bedrock to work from the core install with AWS env markers alone. On the externalized package path, they must install `@openclaw/amazon-bedrock-provider` first, so the FAQ sends them down a stale setup path.
- Suggested fix: Update the FAQ to say Bedrock is supported through the official external npm plugin, include the install command, then describe the same AWS env-marker/discovery behavior after the plugin is installed.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: compare `docs/help/faq-first-run.md:592` against `package.json:35`, `extensions/amazon-bedrock/package.json:32`, and `docs/plugins/reference/amazon-bedrock.md:14`.
- Is this the best way to solve the issue? yes: the implementation and generated plugin inventory are internally consistent; the stale FAQ text should be updated rather than changing packaging behavior.
