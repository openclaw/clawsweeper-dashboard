# Commit 8f2dd02

## Summary

Found one residual bug in the new DeepSeek catalog hydrator: it backfills token and cost fields but leaves other catalog-owned runtime metadata missing, including DeepSeek V4 `compat.maxTokensField: "max_tokens"`.

## Findings

### Medium: Minimal DeepSeek V4 config still misses catalog compat and can send the wrong max-token field

- Kind: bug
- File: `extensions/deepseek/provider-policy-api.ts`
- Line: 66
- Evidence: The new normalizer only patches `contextWindow`, `maxTokens`, and `cost` for catalog matches (`extensions/deepseek/provider-policy-api.ts:66`, `extensions/deepseek/provider-policy-api.ts:72`, `extensions/deepseek/provider-policy-api.ts:78`). The DeepSeek catalog also marks V4 models as `reasoning: true` and sets `compat.maxTokensField` to `"max_tokens"` (`extensions/deepseek/openclaw.plugin.json:31`, `extensions/deepseek/openclaw.plugin.json:41`). For a configured row like `{ id, name }`, `applyModelDefaults` then fills missing `reasoning` with `false` and adds the provider-level API (`src/config/defaults.ts:167`, `src/config/defaults.ts:206`). Runtime resolution treats configured models with an API as inline models before falling back to the registry/catalog (`src/agents/pi-embedded-runner/model.ts:592`), so the missing catalog `compat` is not recovered later. The OpenAI-compatible payload builder sends `max_completion_tokens` unless `compat.maxTokensField === "max_tokens"` (`src/agents/openai-transport-stream.ts:1775`), and the generic detector does not classify DeepSeek as a `max_tokens` provider (`src/agents/openai-completions-compat.ts:70`).
- Impact: Users with minimal `models.providers.deepseek.models[]` rows get the intended 1M context/cost from this commit, but DeepSeek V4 requests that pass `maxTokens` can still be shaped with `max_completion_tokens` instead of the catalog-declared `max_tokens`. The same row is also no longer represented as a reasoning model. This leaves the catalog hydration incomplete for the exact configured-model path the commit is meant to repair.
- Suggested fix: Hydrate missing catalog metadata as a structured merge, not only three fields. Preserve explicit user overrides, but fill absent `reasoning`, `input`, and missing `compat` keys from the matching catalog entry, especially `compat.maxTokensField`.
- Confidence: medium
