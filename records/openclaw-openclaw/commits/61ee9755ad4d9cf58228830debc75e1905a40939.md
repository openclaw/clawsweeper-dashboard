# Commit 61ee975

## Summary

Found one actionable privacy/security issue in the fresh post-core update handoff: the commit serializes the full resolved pre-update config to a temporary file. Normal completion removes the directory, but an interrupted update can leave env-substituted secrets on disk.

## Findings

### Medium: Post-core handoff can leave resolved config secrets in temp storage

- Kind: privacy
- File: `src/cli/update-cli/update-command.ts`
- Line: 2233
- Evidence: `writePostCoreSourceConfigFile` writes `JSON.stringify(preUpdateConfig)` to `source-config.json`. That payload includes `sourceConfig`, and the fresh-process caller populates it from `configSnapshot.sourceConfig` at `src/cli/update-cli/update-command.ts:3147`. Current config IO documents and implements `sourceConfig` as the config after `$include` and `${ENV}` substitution, so provider/channel tokens that were authored as env refs can be materialized into this temp file.
- Impact: If the parent updater is killed, crashes, or the host loses power before the `finally` cleanup removes `openclaw-update-post-core-*`, secrets intended to live only in environment variables can remain on disk. The temp directory narrows exposure, but this is still avoidable secret persistence and covers the full config, not only the channels needed for repair.
- Suggested fix: hand off only the minimum authored channel data needed for restore, and let the child resolve includes/env from the inherited environment and config path. If a resolved handoff is unavoidable, keep it channel-only, write with owner-only mode, and avoid serializing unrelated provider/gateway secrets.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Configure a pre-update value such as `channels.whatsapp.token: "${WHATSAPP_TOKEN}"`, run a package/git update path that respawns post-core work, and interrupt after `source-config.json` is written but before cleanup; the handoff payload comes from resolved `configSnapshot.sourceConfig`.
- Is this the best way to solve the issue? yes. The restore behavior is correct, but the handoff should preserve authored/unresolved channel shape and avoid writing the resolved full config.
