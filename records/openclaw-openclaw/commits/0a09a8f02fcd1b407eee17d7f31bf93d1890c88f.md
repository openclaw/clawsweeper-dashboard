## Findings

### Medium: Vydra image generation still drops SSRF policy after job creation

- Kind: security
- File: `extensions/vydra/image-generation-provider.ts`
- Line: 77
- Evidence: The patched Vydra POST forwards `ssrfPolicy` at `extensions/vydra/image-generation-provider.ts:70`, but the same request flow then calls `resolveCompletedVydraPayload` without policy at `extensions/vydra/image-generation-provider.ts:77` and `downloadVydraAsset` without policy at `extensions/vydra/image-generation-provider.ts:90`. Those helpers use raw timeout fetches at `extensions/vydra/shared.ts:209` and `extensions/vydra/shared.ts:242`, not `fetchWithTimeoutGuarded` or `postJsonRequest`.
- Impact: A Vydra response can still drive OpenClaw to fetch a returned `imageUrl` outside the guarded SSRF path. That means the new image-generation SSRF policy does not actually cover the complete Vydra provider request chain, and a compromised/custom Vydra-compatible endpoint could return a private/internal or RFC2544 URL for the final image download.
- Suggested fix: Thread `req.ssrfPolicy` through `resolveCompletedVydraPayload` and `downloadVydraAsset`; use `fetchWithTimeoutGuarded` for the job-status GET and asset download, preserving the resolved dispatcher policy/allow-private-network behavior where appropriate.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes from source: make the Vydra create response return a non-completed job or an `imageUrl` like `http://169.254.169.254/...`; the next GET/download path goes through `fetchWithTimeout`, bypassing the SSRF guard.
- Is this the best way to solve the issue? Yes directionally: centralizing Vydra GET/downloads on the same guarded helper model avoids provider-specific bypasses while preserving the new request policy.
