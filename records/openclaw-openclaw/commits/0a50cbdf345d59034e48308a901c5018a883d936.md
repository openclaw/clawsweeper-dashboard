## Findings

### Low: Opt-in local PTY smoke can time out on a cold source checkout

- Kind: reliability
- File: `src/tui/tui-pty-local.e2e.test.ts`
- Line: 30
- Evidence: After a fresh `pnpm install`, the documented opt-in command `OPENCLAW_TUI_PTY_INCLUDE_LOCAL=1 node scripts/run-vitest.mjs run --config test/vitest/vitest.tui-pty.config.ts` failed once. The TUI reached `local ready`, accepted `send the local PTY smoke response`, and showed the local busy spinner, but `fixture.mockModel.requests()` stayed empty until `LOCAL_OUTPUT_TIMEOUT_MS` expired at 35s. The same local test passed when rerun by itself, and the combined opt-in command passed on a warmed retry.
- Impact: This makes the newly documented `OPENCLAW_TUI_PTY_INCLUDE_LOCAL=1` proof path flaky for a first run in a cold checkout, so a maintainer or contributor can see a false failure even though the mocked model path works after warmup.
- Suggested fix: Make the local smoke tolerant of cold source checkout startup, either by warming/building before the timed model-request wait, increasing the local request timeout, or waiting on a more specific “local backend ready for first turn” signal before typing the prompt.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? No. I observed the failure once on the first post-install combined opt-in run, then could not reproduce it after rerunning the local test and the combined command.
- Is this the best way to solve the issue? Unclear. The safest direction is to separate cold build/setup latency from the asserted model-request timeout, but the exact fix depends on whether the first-run delay is build/cache warmup or embedded backend readiness.
