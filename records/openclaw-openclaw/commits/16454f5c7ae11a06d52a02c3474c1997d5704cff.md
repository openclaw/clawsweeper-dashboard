# Commit 16454f5c

## Summary

Found one UI correctness issue: the new Sessions table Runtime column can show the configured agent/default runtime instead of the effective runtime persisted on that session.

## Findings

### Low: Runtime column ignores per-session runtime overrides

- Kind: bug
- File: `ui/src/ui/views/sessions.ts`
- Line: 760
- Evidence: The new column renders `resolveAgentRuntimeLabel(row.agentRuntime)` and filtering uses the same field at `ui/src/ui/views/sessions.ts:182`. Current session rows populate `agentRuntime` from `resolveAgentRuntimeMetadata(cfg, sessionAgentId)` at `src/gateway/session-utils.ts:1646` and return that value at `src/gateway/session-utils.ts:1765`, which only reflects env/agent/default/implicit config. But session entries can persist an effective runtime override from `/model ... --runtime ...` at `src/auto-reply/reply/directive-handling.persist.ts:249`, and execution reads `sessionEntry.agentRuntimeOverride` at `src/auto-reply/reply/agent-runner-execution.ts:1404`. Status rendering also gives that override priority at `src/status/status-message.ts:223`.
- Impact: A session running through an override such as `--runtime codex` can appear as `pi` or the agent default in the Sessions table, and searching/filtering by the actual runtime will miss it.
- Suggested fix: Expose the effective session runtime in the sessions-list payload, resolving `entry.agentRuntimeOverride` / `entry.agentHarnessId` before falling back to `resolveAgentRuntimeMetadata`, then render and filter that effective value in the UI.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: create or inspect a session entry with `agentRuntimeOverride: "codex"` while the agent/default runtime resolves to `pi`, then call `sessions.list` and open the Sessions table. The code path returns `agentRuntime: { id: "pi", source: "implicit" }`, so the new UI column cannot display `codex`.
- Is this the best way to solve the issue? yes: the UI does not receive `agentRuntimeOverride`, so the backend payload needs to expose the effective runtime rather than making the table infer from unavailable state.
