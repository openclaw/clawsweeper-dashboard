# Commit 1b77145

## Summary

Found one concrete reliability regression in the new trusted fingerprint bookkeeping. The patch still falsely reports a session takeover after a normal local Pi-owned write advances a controller’s fence, that controller releases again, and another same-process controller publishes a transcript append.

## Findings

### Medium: Trusted session baseline goes stale after local owned writes

- Kind: reliability
- File: `src/agents/pi-embedded-runner/run/attempt.session-lock.ts`
- Line: 191
- Evidence: `trustSessionFileState()` only initializes `trustedSessionFileStates` once; after that, a different fingerprint returns `undefined` instead of advancing trust. `refreshAfterOwnedSessionWrite()` updates only the controller-local `fenceFingerprint`, so after a prompt reacquire plus Pi-owned persistence, a later `releaseForPrompt()` records the new local fence but leaves the global trusted baseline stale. A subsequent `runWithOwnedSessionTranscriptWritePublication()` then refuses to record the append because `publishOwnedSessionFileWriteIfChanged()` requires `beforeWrite` to match `trustedSessionFileStates`.
- Impact: the commit’s target race can still fail in a realistic multi-release attempt: provider stream returns, Pi persists an owned assistant/error message, the attempt releases again for post-prompt/compaction wait, and another in-process delivery-mirror transcript append occurs. The released controller then throws `EmbeddedAttemptSessionTakeoverError`, marks takeover, and can skip normal cleanup/session flush even though all writes were OpenClaw-owned.
- Suggested fix: when a controller proves a local owned write by holding/reacquiring the session lock and calling `refreshAfterOwnedSessionWrite()`, advance the trusted baseline or record an owned generation for that new fingerprint before a later release uses it as the fence baseline. Keep publication narrow so external drift before/inside broad callbacks is still rejected.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. I ran a focused source-level probe with real temp session files: first controller `releaseForPrompt()` -> `reacquireAfterPrompt()` -> append a local Pi-owned message -> `refreshAfterOwnedSessionWrite()` -> second `releaseForPrompt()`; then a second controller used `runWithOwnedSessionTranscriptWritePublication()` to append. The first controller’s next `withSessionWriteLock()` printed `takeover` and `firstTakeover: true`.
- Is this the best way to solve the issue? yes, directionally. The bug is in trust-generation bookkeeping, not in the broader transcript append API.
