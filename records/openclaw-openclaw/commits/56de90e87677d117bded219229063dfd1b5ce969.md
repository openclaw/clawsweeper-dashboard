Found one concrete regression in the newly added Windows-hosted Testbox workflows: the pinned Blacksmith Testbox action fails on the Windows runner before checkout, so the new maintainer harness cannot start.

## Findings

### Low: Windows-hosted Testbox workflows fail before checkout

- Kind: compatibility
- File: `.github/workflows/windows-testbox.yml`
- Line: 28
- Evidence: The reviewed SHA already has a manual `Windows Testbox` run at `https://github.com/openclaw/openclaw/actions/runs/25292097042/job/74145297833` that failed in `Begin Testbox` before checkout. The log shows `jq: parse error: Invalid numeric literal at line 1, column 10` inside `useblacksmith/begin-testbox@d0e04585c26905fdd92c94a09c159544c7ee1b67`, followed by `run-testbox` failing from partial/missing state. The pinned `begin-testbox` action source builds JSON by interpolating `${GITHUB_WORKSPACE}` directly into the payload and later runs Linux-specific setup (`sudo apt-get install -y rsync`), so using it unchanged on `blacksmith-16vcpu-windows-2025` is not currently compatible. `.github/workflows/wsl2-testbox.yml:25` has the same first step on the same Windows runner, so it is exposed to the same failure before its WSL2 verification can run.
- Impact: The new Windows and likely WSL2 maintainer harnesses are advertised in the changelog but cannot reliably open a Testbox session, blocking the plugin-surface debugging workflow this commit adds.
- Suggested fix: Use a Windows-compatible Blacksmith Testbox action/pin, or wrap/replace the begin/run steps for Windows so the phone-home payload is JSON-safe for Windows paths and the setup avoids Linux-only package-manager commands. Then validate with a real warmed Windows and WSL2 Testbox dispatch.
- Confidence: high
