The lock acquisition change is source-correct and the focused tests pass. I found one low-severity contract/documentation mismatch left by the new behavior: `maxHoldMs` is no longer watchdog-only, but the user-facing config help and docs still say it is.

## Findings

### Low: Update maxHoldMs help for acquisition-time reclaim

- Kind: compatibility
- File: `src/config/schema.help.ts`
- Line: 1644
- Evidence: Current code records `maxHoldMs` in the lock payload and passes `respectMaxHold: !heldByThisProcess` through both `shouldReclaim` and `shouldRemoveStaleLock`, so a contender can delete an unchanged live-holder lock after the holder-recorded max hold expires (`src/agents/session-write-lock.ts:746`, `src/agents/session-write-lock.ts:752`, `src/agents/session-write-lock.ts:764`). The config help still describes `session.writeLock.maxHoldMs` only as “a held in-process session transcript lock may remain held before the watchdog releases it,” and the docs similarly say it controls the “in-process watchdog release threshold” (`docs/reference/session-management-compaction.md:101`).
- Impact: Operators tuning this setting are now tuning cross-process acquisition reclaim too. Setting it low can make other processes reclaim live-holder locks sooner; setting it high can delay the new fix. The current docs hide that changed runtime contract.
- Suggested fix: Update `src/config/schema.help.ts` and `docs/reference/session-management-compaction.md` to say `maxHoldMs` is the holder-recorded max lock age used by the in-process watchdog and by acquisition-time reclaim for non-self-held locks.
- Confidence: high
