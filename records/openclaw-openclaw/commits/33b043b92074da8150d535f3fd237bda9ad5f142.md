# Commit 33b043b

## Summary

Found one multi-account migration edge case in the Discord doctor fix.

## Findings

### Medium: Root Discord `agentId` migration becomes default-account-only

- Kind: bug
- File: `extensions/discord/src/doctor-contract.ts`
- Line: 281
- Evidence: The root `channels.discord.guilds...channels...agentId` migration builds a binding without `match.accountId` unless `params.accountId` is set. Discord account config inherits top-level channel config through `mergeDiscordAccountConfig`, but route binding resolution treats a missing `accountId` as the default account only, not any account. A focused smoke showed doctor migration produced `bindings: [{ match: { channel: "discord", guildId: "g1", peer: { kind: "channel", id: "c1" } } }]`; resolving the same guild/channel for `accountId: "work"` returned `agentId: "main", matchedBy: "default"`, while `accountId: "default"` returned `agentId: "video", matchedBy: "binding.peer"`.
- Impact: In a multi-account Discord config where a named account inherits top-level `guilds` config, `openclaw doctor --fix` removes the invalid nested `agentId` but does not create a route that applies to that named account. The config becomes valid, but messages for that account continue on the default agent instead of the intended migrated agent.
- Suggested fix: When migrating top-level Discord channel `agentId`, preserve account inheritance. Generate explicit account-scoped bindings for configured accounts that inherit the root guild config, or use `accountId: "*"` only when it is safe for every configured account. Add a regression test with `channels.discord.accounts.work` plus root `channels.discord.guilds...agentId`.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. The `tsx` smoke above uses current `main` code: normalize the legacy config, then call `resolveAgentRoute` for `accountId: "work"` and the same Discord guild/channel.
- Is this the best way to solve the issue? Unclear. The safest direction is per-account generated bindings because Discord account config inheritance is shallow and account-local `guilds` overrides can opt accounts out of root guild config.
