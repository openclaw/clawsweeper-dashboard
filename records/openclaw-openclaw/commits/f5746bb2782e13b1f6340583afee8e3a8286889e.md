# Commit f5746bb

## Summary

Found one low-severity regression in the live model test harness: the new priority-only discovery path also applies when `OPENCLAW_LIVE_MAX_MODELS=0`, which is documented as the escape hatch for an exhaustive modern sweep.

## Findings

### Low: `OPENCLAW_LIVE_MAX_MODELS=0` no longer makes the modern live sweep exhaustive

- Kind: regression
- File: `src/agents/models.profiles.live.test.ts`
- Line: 805
- Evidence: `useDefaultPriorityOnly` is true for every `OPENCLAW_LIVE_MODELS=modern|all` run without model/provider filters, and that branch returns `loadPrioritizedHighSignalModels()` at `src/agents/models.profiles.live.test.ts:808` before `OPENCLAW_LIVE_MAX_MODELS` is read at `src/agents/models.profiles.live.test.ts:838`. The documented contract still says `OPENCLAW_LIVE_MAX_MODELS=0` is an exhaustive modern sweep in `docs/help/testing-live.md:83`.
- Impact: Maintainers asking for an exhaustive default modern live run now only test the static priority list. That can silently miss configured provider/plugin catalog models. One concrete example is `minimax-portal/minimax-m2.7`: it remains in the priority list at `src/agents/live-model-filter.ts:31`, the MiniMax plugin registers `minimax-portal` and supplies `MiniMax-M2.7` from its catalog at `extensions/minimax/provider-registration.ts:101`, but `@mariozechner/pi-ai` static providers do not include `minimax-portal`, so the new loader skips that route entirely.
- Suggested fix: Make the priority-only shortcut conditional on the default capped sweep only. If `OPENCLAW_LIVE_MAX_MODELS=0`, keep the old `discoverAuthStorage` / `discoverModels` path and then apply the existing high-signal filter with no cap. Add a focused test for this branch selection.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the selection logic: run the direct live model test with `OPENCLAW_LIVE_MODELS=modern OPENCLAW_LIVE_MAX_MODELS=0` and no provider/model filters; the code still takes the priority-only branch. I did not run external live provider calls.
- Is this the best way to solve the issue? Yes: preserving the new default cap while routing the explicit `0` escape hatch through discovery keeps the commit’s performance goal without breaking the documented exhaustive mode.
