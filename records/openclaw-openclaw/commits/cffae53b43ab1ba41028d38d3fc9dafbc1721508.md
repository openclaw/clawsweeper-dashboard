## Summary

Found one concrete issue: the security fix is implemented only as a pnpm workspace patch to `@openclaw/fs-safe@0.2.2`. Recommended/package installs still resolve the unpatched published dependency, so the Windows ACL audit bypass remains for npm-installed OpenClaw packages.

## Findings

### High: Windows ACL security fix does not reach npm/package installs

- Kind: security
- File: `pnpm-workspace.yaml`
- Line: 98
- Evidence: OpenClaw imports ACL audit behavior from `@openclaw/fs-safe` (`src/infra/permissions.ts:2-21`) and still declares the dependency as exact `@openclaw/fs-safe: "0.2.2"` (`package.json:1755`). The commit adds the SID fix only through `patchedDependencies` (`pnpm-workspace.yaml:97-98`) and a patch file (`patches/@openclaw__fs-safe@0.2.2.patch:1-47`). The recommended install path is `npm install -g openclaw@latest` (`README.md:101-103`), which installs the published `@openclaw/fs-safe@0.2.2` package, not this repo’s pnpm workspace patch. A focused npm install probe of `@openclaw/fs-safe@0.2.2` confirmed `*S-1-5-7` still classifies as `group` (`{"trusted":0,"world":0,"group":1}`), and the published package contents do not contain `anonymous logon` or `s-1-5-7`.
- Impact: A future npm/package release from this commit can claim the Windows ACL audit bypass is fixed while package-installed users still downgrade broad principals such as Anonymous Logon from critical `worldWritable` findings to warn-level `groupWritable` findings.
- Suggested fix: publish a fixed `@openclaw/fs-safe` version and bump OpenClaw to it, or vendor/override this classifier in OpenClaw’s shipped `dist` path so npm package installs execute the fixed code. Add package-installed proof that imports the installed package and verifies `*S-1-5-7` is world-classified.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Install `@openclaw/fs-safe@0.2.2` from npm and call `summarizeWindowsAcl([{ principal: "*S-1-5-7", rawRights: "(F)", rights: ["F"], canRead: true, canWrite: true }])`; it returns the entry under `untrustedGroup`, not `untrustedWorld`.
- Is this the best way to solve the issue? yes. Since OpenClaw’s npm package depends on `@openclaw/fs-safe`, the durable fix needs to ship through that package version or through OpenClaw’s packaged runtime code, not only the source workspace patch.
