# Commit 4ad29d2

## Summary

Found one regression in the POSIX standalone Parallels smoke scripts: a failed batch config write can now be masked by the cleanup command.

## Findings

### Medium: POSIX smoke scripts ignore failed provider config batch writes

- Kind: reliability
- File: `scripts/e2e/parallels/linux-smoke.ts`
- Line: 693
- Evidence: `verifyLocalTurn()` now writes a temp batch file, runs `openclaw config set --batch-file "$provider_config_batch" --strict-json`, then runs `rm -f "$provider_config_batch"` with no `set -e` or captured exit status. The macOS path has the same pattern in `scripts/e2e/parallels/macos-smoke.ts:977`. These scripts are executed by `LinuxGuest.bash()` / `MacosGuest.sh()` as plain `bash <scriptPath>`, so the final `rm -f` status becomes the script status. A focused shell check reproduced the masking: `false; rm -f "$tmp"` exits `0`, while the captured-status pattern used in `scripts/e2e/parallels/npm-update-scripts.ts:30` exits `1`.
- Impact: if the batch config set fails, the Linux/macOS standalone Parallels smoke can continue as if the OpenAI provider timeout/SSE config was applied. That can produce misleading later failures or false green smoke results that no longer prove the intended provider config path works.
- Suggested fix: use the same captured-exit cleanup pattern already added in `npm-update-scripts.ts`, or add `set -e` plus a cleanup trap before the temp-file block in both standalone POSIX smoke scripts.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The generated block is a plain bash script without `set -e`; local shell reproduction confirmed the nonzero command is masked when followed by successful `rm -f`.
- Is this the best way to solve the issue? yes. Capturing the `config set` exit and cleaning up before exiting preserves temp-file cleanup without hiding failures; it also matches the already-correct npm-update POSIX helper.
