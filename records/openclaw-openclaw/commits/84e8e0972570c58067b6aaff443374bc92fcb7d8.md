# Commit 84e8e09725

## Summary

Found three actionable issues in the new QA/live-check plumbing. The highest-risk one is unsafe extraction of WhatsApp auth archives before handing the directory to Baileys.

## Findings

### Medium: WhatsApp auth archives can plant symlinks that Baileys later writes through

- Kind: security
- File: `extensions/qa-lab/src/live-transports/whatsapp/whatsapp-live.runtime.ts`
- Line: 440
- Evidence: `assertSafeArchiveEntries` only rejects absolute paths and `..` path segments, then `unpackWhatsAppAuthArchive` runs `tar -xzf` into the auth dir. A tarball containing a symlink entry named `creds.json` passes the current entry check because `tar -tzf` only reports `creds.json`. GNU tar extracts that symlink. Baileys’ local `useMultiFileAuthState` implementation writes auth files with `fs.promises.writeFile(filePath, ...)`, and Node follows symlinks on `writeFile`, so a crafted archive can redirect later auth writes outside the extracted auth directory.
- Impact: A malformed or malicious WhatsApp QA credential archive can write files as the CI/local runner user when the QA lane starts the driver or SUT session. This is contained to QA credential inputs, but it is still an archive traversal class bug in a secret-bearing live lane.
- Suggested fix: Inspect tar entry type/link metadata before extraction and reject symlinks, hardlinks, device nodes, and other non-regular/non-directory entries. Prefer extracting with a library that exposes entry metadata, or run tar with hardening flags and verify the extracted tree with `lstat` before Baileys sees it.
- Confidence: high

### Low: Mention-gating can pass even when the SUT replies to the quiet group message

- Kind: bug
- File: `extensions/qa-lab/src/live-transports/whatsapp/whatsapp-live.runtime.ts`
- Line: 537
- Evidence: The group scenario sends an unmentioned `quietInput`, then only treats a reply as unexpected when it matches `quietMatchText`. Since `quietMatchText` is set to the generated quiet token, any SUT reply that does not include that exact token is ignored, even though mention gating’s invariant is “no visible reply before the mention gate is triggered.”
- Impact: The new WhatsApp mention-gating lane can false-pass if group mention gating regresses but the agent replies with text that does not echo the quiet token.
- Suggested fix: For the quiet window, fail on any incoming group message from the SUT after `quietStartedAt`, or make the scenario omit `quietMatchText` so the existing fallback checks any reply.
- Confidence: high

### Low: Release QA gates normalize `1`/`yes` as enabled but job `if` still skips them

- Kind: bug
- File: `.github/workflows/openclaw-release-checks.yml`
- Line: 1063
- Evidence: The input step normalizes `OPENCLAW_RELEASE_QA_WHATSAPP_LIVE_CI_ENABLED` values of `true`, `1`, and `yes` to `qa_live_whatsapp_enabled=true`, but the job condition also requires `vars.OPENCLAW_RELEASE_QA_WHATSAPP_LIVE_CI_ENABLED == 'true'`. The same pattern exists for Discord and Slack. A focused shell check confirms `1` and `yes` normalize to true while the strict job condition remains false.
- Impact: Release summaries can say a QA live lane is enabled while GitHub skips the job, so an operator using accepted truthy values (`1`/`yes`) silently loses the requested Discord/WhatsApp/Slack release QA coverage.
- Suggested fix: Trust the normalized `needs.resolve_target.outputs.qa_live_*_enabled == 'true'` value in the job `if`, or apply the same truthy normalization in the job condition.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. The archive issue is reproducible with a tarball containing a safe-looking symlink entry and confirmed by Baileys’ `writeFile` auth-store behavior. The mention-gating false pass follows directly from the quiet-window predicate. The release gate mismatch is reproducible by setting the release QA variable to `1` or `yes`.
- Is this the best way to solve the issue? Yes for the archive hardening and gate normalization. For mention gating, the safest fix is checking any SUT group reply during the quiet window rather than relying on model compliance with the quiet token instruction.
