# Commit a2f1d1d

## Summary

Found one regression that still matters on current `main`: text-source turns can bypass the default group/channel `message_tool_only` suppression even when the inbound text is not actually a slash command.

## Findings

### Medium: Authorized non-command group messages can become automatic visible replies

- Kind: regression
- File: `src/auto-reply/reply/source-reply-delivery-mode.ts`
- Line: 17
- Evidence: The reviewed commit made all `CommandSource: "text"` turns explicit source replies. Current `main` adds `CommandAuthorized === true`, but still does not verify that `CommandBody` is a command. Some producers mark normal text turns as text-source turns: Discord always sets `CommandSource: "text"` in `extensions/discord/src/monitor/message-handler.context.ts:353`, and Matrix does the same in `extensions/matrix/src/matrix/monitor/handler.ts:1351`. Their `CommandAuthorized` comes from `resolveControlCommandGate`, whose `commandAuthorized` is based on sender authorizers and is true even when `hasControlCommand` is false (`src/channels/command-gating.ts:38`). A focused live check with `hasControlCommand: false`, authorized sender, and `CommandBody: "hello group"` produced `sourceReplyDeliveryMode: "automatic"` and `suppressDelivery: false`.
- Impact: Allowed/authorized Discord or Matrix group senders can cause ordinary assistant final replies to post visibly in the room, bypassing the intended default that normal group/channel final output stays private/suppressed unless the model uses the `message` tool. This is a privacy and behavior regression for group/channel defaults.
- Suggested fix: Make explicit text-source reply detection require an actual text command, not just an authorized sender. For example, pass a command-present flag from command detection, or have `isExplicitSourceReplyCommand` verify normalized `CommandBody` starts with a supported command before returning true for `CommandSource: "text"`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The focused check combined `resolveControlCommandGate({ hasControlCommand: false, authorizers: [{ configured: true, allowed: true }] })` with `resolveSourceReplyVisibilityPolicy` for a group text-source context and got automatic visible delivery.
- Is this the best way to solve the issue? yes, directionally. The text-command visibility exception should be keyed to command detection, while sender authorization should remain a separate permission signal.
