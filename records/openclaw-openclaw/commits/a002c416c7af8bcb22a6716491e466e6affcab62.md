# Commit a002c416

## Summary

Found one regression in the missing-package-version path: omitting `OPENCLAW_COMPATIBILITY_HOST_VERSION` does not reliably clear it. The doctor child can inherit a stale parent value and then validate plugin compatibility against the wrong host version.

## Findings

### Medium: Missing package version path can reuse stale compatibility host version

- Kind: regression
- File: `src/cli/update-cli/update-command.ts`
- Line: 1425
- Evidence: The CLI package-update doctor env spreads `resolvePostInstallDoctorEnv(...)`, which starts from `process.env` at `src/cli/update-cli/update-command.ts:1043`. After this commit, when `readPackageVersion(verifiedPackageRoot)` returns `null`, the object spread at `src/cli/update-cli/update-command.ts:1425` adds no replacement value, so any parent `OPENCLAW_COMPATIBILITY_HOST_VERSION` remains in the child env. The service update runner has the same shape at `src/infra/update-runner.ts:1490`; even though it does not spread `process.env` directly there, the default command runner merges the child env over inherited env at `src/process/exec.ts:81` and `src/process/exec.ts:304`. `resolveCompatibilityHostVersion` treats this env var as authoritative at `src/version.ts:147`, and plugin manifest loading uses that value for `minHostVersion` checks at `src/plugins/manifest-registry.ts:956`.
- Impact: In the exact case this commit targets, a stale parent compatibility version can make post-update doctor/plugin validation reject plugins that should load, or allow plugins whose `minHostVersion` exceeds the actual updated runtime. That can break package updates or leave the refreshed plugin registry inconsistent with the runtime being installed.
- Suggested fix: When the candidate package version is missing, explicitly neutralize this internal override instead of omitting it. Given the current env merge helper cannot delete inherited keys with `undefined`, use a value already ignored by `firstNonEmpty` such as `"null"`/`"undefined"`, or add a real delete/sanitize path in the command env construction and cover both update call sites with tests where `process.env.OPENCLAW_COMPATIBILITY_HOST_VERSION` is pre-set and `readPackageVersion()` returns `null`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Set `process.env.OPENCLAW_COMPATIBILITY_HOST_VERSION` to a stale version, make the verified package root have no readable `package.json` version, and run the package-update post-verify doctor path. The focused env smoke showed that an omitted override preserves the parent value, while the old `"null"` override reaches the child as `null`, which `resolveCompatibilityHostVersion` ignores.
- Is this the best way to solve the issue? yes. The intended contract is “do not use a missing candidate package version as the compatibility host version”; the fix needs to clear the internal override across the existing child-env merge behavior, not just omit a property from one object literal.
