# Commit 1e8de76

## Summary

Found one concrete regression in diagnostic session alias merging. The commit can merge unrelated conversation diagnostics when callers pass a shared agent id as `sessionId`.

## Findings

### Medium: Shared agent ids can collapse unrelated diagnostic sessions

- Kind: regression
- File: `src/logging/diagnostic-session-state.ts`
- Line: 136
- Evidence: `getDiagnosticSessionState()` now merges `direct` session-key state with the first state found by `sessionId`. Tool loop diagnostics call it with `sessionId: args.ctx?.agentId` at `src/agents/pi-tools.before-tool-call.ts:410`, and dispatch creates per-conversation states with only `sessionKey` at `src/auto-reply/reply/dispatch-from-config.ts:373`. For two active keys under the same agent, a focused probe on current `main` reduced two states to one:
  `size: 1`, key `agent:main:telegram:dm:b`, `sessionId: "main"`, `queueDepth: 2`.
- Impact: Diagnostic state for separate conversations using the same agent can be merged, deleting one session’s entry and adding its queue depth/history to another. That can produce wrong liveness counts and, when stuck recovery runs, target the wrong session lane or miss the actually stale one.
- Suggested fix: Do not merge two non-empty, different `sessionKey` states solely because `sessionId` matches. Also change tool-loop diagnostic lookups to use `args.ctx.sessionId` for diagnostic `sessionId`, not `args.ctx.agentId`, or keep `agentId` in a separate field.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Creating two keyed states, then resolving both with `sessionId: "main"` collapses them into one state on current `main`.
- Is this the best way to solve the issue? yes. The canonicalization should only join true aliases for the same runtime session, not all sessions sharing an agent id.
