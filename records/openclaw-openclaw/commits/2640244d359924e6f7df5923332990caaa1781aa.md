# Commit 2640244d35

## Summary

The ACP runtime-cache signature change is sound for calls that receive the current config. One linked issue path still appears actionable: Discord queued jobs can retain a pre-change `cfg`, so the new ACP signature can still compare and reuse against the stale config snapshot.

## Findings

### Medium: Discord queued jobs can still run with stale exec policy after config changes

- Kind: security
- File: `extensions/discord/src/monitor/inbound-job.ts`
- Line: 54
- Evidence: `buildDiscordInboundJob()` removes runtime-only fields but leaves the rest of the preflight context in `payload`, including `cfg`. Later, `processDiscordQueuedMessage()` calls `materializeDiscordInboundJob()` and dispatches that saved payload. The new ACP guard in `src/acp/control-plane/manager.core.ts:1474` hashes `params.cfg`, so a job queued before a config change still supplies the old config signature and can keep using a cached handle created under that old policy.
- Impact: This leaves the secondary stale-config path from issue #59273 unresolved for queued Discord work crossing a config-change/restart window. If an operator tightens `tools.exec.security` or `safeBins`, an already queued Discord ACP job can still process under the older policy.
- Suggested fix: Re-materialize queued Discord jobs with the current runtime config, or store a config generation/fingerprint on enqueue and re-preflight/drop jobs when the generation changes before processing. Add a regression that enqueues with old exec config, refreshes runtime config, then proves queued processing uses or forces the new policy.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes at source level: enqueue a Discord inbound job from a preflight context with old `cfg`, update the runtime config before queue processing, then observe `processDiscordMessage` receives the saved old `cfg` and ACP computes the old runtime config cache key.
- Is this the best way to solve the issue? The ACP cache-signature part is appropriate, but the queued Discord path needs its own freshness boundary because ACP can only hash the config object it is given.
