# Commit 5aefe6a

## Summary

Found two actionable issues: streamed ElevenLabs audio can be aborted by the request timeout while Discord is still playing it, and the Plugin SDK API baseline hash in the commit is stale.

## Findings

### Medium: Streamed TTS playback can be cut off by the fetch timeout

- Kind: reliability
- File: `extensions/elevenlabs/tts.ts`
- Line: 173
- Evidence: `elevenLabsTTSStream` passes the normal TTS `timeoutMs` into `fetchWithSsrFGuard` and then returns `response.body` to Discord for later consumption at `extensions/elevenlabs/tts.ts:162-186`. `fetchWithSsrFGuard` keeps the timeout active until `release()` is called at `src/infra/net/fetch-guard.ts:356-370`, and `release()` is only called after Discord playback exits at `extensions/discord/src/voice/segment.ts:168-169`. The default TTS timeout is 30s at `extensions/speech-core/src/tts.ts:62`.
- Impact: any ElevenLabs stream that remains open longer than `messages.tts.timeoutMs` can abort mid-playback. The local reproduction using `elevenLabsTTSStream` with a slow streaming HTTP response and `timeoutMs: 1000` read the first chunk, then the stream failed with `TimeoutError request timed out` after the timeout fired.
- Suggested fix: split the timeout into a request/header timeout and a body/playback lifetime, or clear/refresh the guarded-fetch timeout once the streaming body is handed off. If keeping a body timeout, wrap the stream so each chunk calls `refreshTimeout`, and make Discord’s playback timeout the user-facing playback limit.
- Confidence: high

### Low: Plugin SDK API baseline hash is stale

- Kind: compatibility
- File: `docs/.generated/plugin-sdk-api-baseline.sha256`
- Line: 1
- Evidence: the commit adds public SDK/runtime surface including `SpeechProviderPlugin.streamSynthesize` at `src/plugins/types.ts:1812` and `PluginRuntimeCore.tts.textToSpeechStream` at `src/plugins/runtime/types-core.ts:194`, but `pnpm plugin-sdk:api:check` fails on current `main` with `Plugin SDK API baseline drift detected`. Recomputing under Node 22 produced `c6b4de2d... plugin-sdk-api-baseline.json` and `d4f6a5bd... plugin-sdk-api-baseline.jsonl`, while the committed file contains `e26753b5...` and `7b998abd...`.
- Impact: the Plugin SDK API check gate fails for this commit/current `main`, so release or changed-surface validation that includes the API baseline check will block until the hash is regenerated.
- Suggested fix: rerun `pnpm plugin-sdk:api:gen` from a clean tree and commit the updated `docs/.generated/plugin-sdk-api-baseline.sha256`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The stream timeout issue reproduces with a local streaming HTTP server through `elevenLabsTTSStream`; the API baseline issue reproduces with `pnpm plugin-sdk:api:check`.
- Is this the best way to solve the issue? yes for the API hash. For the stream timeout, the safest direction is to avoid applying the request timeout to the full playback body; exact implementation should fit the guarded-fetch timeout model.
