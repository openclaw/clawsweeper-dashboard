# Commit 50ee68c

## Summary

Found one concrete regression in the new plugin host-hook Gateway surface: Control UI descriptors from plugins that fail during registration can still be returned to Gateway clients.

## Findings

### Medium: Failed plugins can still publish Control UI descriptors

- Kind: regression
- File: `src/gateway/server-methods/plugin-host-hooks.ts`
- Line: 56
- Evidence: `plugins.uiDescriptors` maps `getActivePluginRegistry()?.controlUiDescriptors` directly and does not filter by loaded plugin status. If a plugin registers `api.session.controls.registerControlUiDescriptor(...)` and then throws during `register`, the loader marks the plugin `error`, but the rollback snapshot in `src/plugins/loader.ts:321` only includes `sessionActions` from the workflow arrays and still omits `controlUiDescriptors`. The descriptor is pushed in `src/plugins/registry.ts:2018`, survives the failed registration, and is then projected by the new Gateway method.
- Impact: Operator clients can render controls for a plugin that did not load. Invoking related workflow controls will fail or point at stale plugin state, and the same rollback gap can also leave failed-plugin workflow cleanup/scheduler records behind.
- Suggested fix: Filter `plugins.uiDescriptors` to descriptors whose `pluginId` belongs to a `loaded` plugin, and complete the loader rollback snapshot for all mutable workflow arrays such as `controlUiDescriptors`, `sessionExtensions`, `trustedToolPolicies`, `toolMetadata`, `runtimeLifecycles`, `agentEventSubscriptions`, and `sessionSchedulerJobs`. Add a regression test where a plugin registers a descriptor and then throws.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused current-main smoke loaded a throwaway plugin that registers a Control UI descriptor and then throws; the returned registry showed plugin status `error`, while `plugins.uiDescriptors` still returned the descriptor.
- Is this the best way to solve the issue? yes. Filtering the Gateway projection prevents user-visible stale controls, and fixing rollback removes the underlying leaked registration state.
