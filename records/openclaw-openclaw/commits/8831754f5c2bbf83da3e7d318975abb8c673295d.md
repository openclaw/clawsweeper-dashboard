# Commit 8831754

## Summary

Found one regression in provider auth resolution: arbitrary all-caps literal `models.providers.<id>.apiKey` values are now treated as env-var references.

## Findings

### Medium: all-caps literal provider API keys are dropped during catalog auth resolution

- Kind: regression
- File: `src/agents/models-config.providers.secrets.ts`
- Line: 218
- Evidence: `resolveConfigBackedProviderAuth()` now tests the normalized configured key with `ENV_VAR_NAME_RE`. A literal key like `ALLCAPS_SAMPLE` matches that regex, so with no `ALLCAPS_SAMPLE` env var the resolver returns `undefined` instead of the configured key. Existing adjacent contracts say arbitrary all-caps `models.json` `apiKey` values are plaintext, not markers: `src/commands/models/list.probe.targets.test.ts`, `src/secrets/audit.test.ts`, `src/infra/provider-usage.auth.normalizes-keys.test.ts`, and `src/agents/models-config.providers.secrets.bedrock-apikey.test.ts`.
- Impact: plugin catalog discovery paths call this resolver from `src/agents/models-config.providers.implicit.ts:487` and `src/commands/models/list.provider-catalog.ts:267`; self-hosted setup returns `null` when `ctx.resolveProviderApiKey()` has no `apiKey` in `src/plugins/provider-self-hosted-setup.ts:415`. A configured custom/self-hosted provider whose literal token is all-caps/underscore shaped can disappear from catalog discovery or be reported unauthenticated.
- Suggested fix: keep custom env-marker support only for unambiguous marker syntax/source, such as original `"${VAR}"` syntax or SecretRef/env-ref metadata. Do not infer every bare all-caps string as an env marker; preserve bare arbitrary values as plaintext unless they are known markers or have source metadata proving they came from an env ref.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused smoke check on current `main` returned `{"apiKey":null,"discoveryApiKey":null}` for `apiKey: "ALLCAPS_SAMPLE"` with an empty env, while `apiKey: "proof-key"` resolved normally.
- Is this the best way to solve the issue? yes, at the contract level. The unsafe part is the ambiguous broad regex on a persisted/plain string; the fix should preserve source information or require explicit marker syntax.
