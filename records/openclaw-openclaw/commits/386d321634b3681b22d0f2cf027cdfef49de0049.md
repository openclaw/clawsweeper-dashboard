# Commit 386d321

## Summary

Found two approval-routing regressions. The more serious one breaks normal two-phase exec approvals from backend gateway clients because ownership is bound to a transient WebSocket connection, but the requester waits on a later connection.

## Findings

### High: Two-phase exec approval requesters lose access after the registration connection closes

- Kind: regression
- File: `src/gateway/server-methods/approval-shared.ts`
- Line: 234
- Evidence: `exec.approval.request` stamps `requestedByConnId`, `requestedByDeviceId`, and `requestedByClientId` on the record (`src/gateway/server-methods/exec-approval.ts:335`). The new `handleApprovalWaitDecision` rejects unless `isApprovalRecordVisibleToClient` accepts the caller (`src/gateway/server-methods/approval-shared.ts:234`). For no-device backend gateway clients, visibility falls back to exact `connId` matching (`src/gateway/server-methods/approval-shared.ts:105`), but the standard agent path registers with `expectFinal: false` (`src/agents/bash-tools.exec-approval-request.ts:121`) and waits later via a fresh gateway call (`src/agents/bash-tools.exec-approval-request.ts:137`). `callGateway` creates a new `GatewayClient` per call (`src/gateway/call.ts:639`), so the wait connection cannot match the original registration `connId`.
- Impact: Approval-gated exec/tool runs can immediately treat their own pending approval as “approval expired or not found,” returning `null` before the operator can approve. The same transient-connection binding also risks breaking `/approve` flows that use `callGateway` instead of the new approval-runtime client.
- Suggested fix: Do not use transient `connId` as the sole owner for two-phase request/wait flows. Keep the requester on the same connection through the final response, or issue a stable request-side secret/handle for `waitDecision`, or add a tightly-scoped trusted backend path that does not reopen cross-client approval replay.
- Confidence: high

### Medium: Hidden exact IDs still block visible prefix lookup

- Kind: bug
- File: `src/gateway/exec-approval-manager.ts`
- Line: 212
- Evidence: `lookupApprovalId` returns `none` immediately when an exact ID exists but is filtered out (`src/gateway/exec-approval-manager.ts:212`). That prevents scanning visible prefix matches. A focused probe with hidden id `approval` and visible id `approval-visible`, filtered to the visible owner, returned `{"kind":"none"}` instead of `{"kind":"prefix","id":"approval-visible"}`.
- Impact: A caller can fail to resolve an approval by a normally unique visible prefix if another caller has a hidden exact ID with that prefix. This can make short approval IDs unreliable after the requester filter was added.
- Suggested fix: If an exact record exists but is excluded only by the visibility filter, continue into the prefix scan over visible records. Preserve the existing resolved-exact behavior separately if that is still required.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused `handleApprovalWaitDecision` probe with a record owned by `conn-register` and a later backend gateway-client caller on `conn-wait` returned `approval expired or not found`.
- Is this the best way to solve the issue? unclear. The intent to narrow approval visibility is sound, but the fix needs a stable requester identity or request token rather than a WebSocket `connId` that disappears between two-phase calls.
