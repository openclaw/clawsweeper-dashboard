# Commit 73be4ea

## Summary

Found one concrete release-validation coverage regression: two of the new two-provider Docker live gateway shards can still select two models from only one provider, so the advertised paired provider is not actually exercised when both providers have credentials.

## Findings

### Medium: Split Docker gateway shards can skip one provider in each pair

- Kind: regression
- File: `.github/workflows/openclaw-live-and-e2e-checks-reusable.yml`
- Line: 2209
- Evidence: The new advisory shards set `OPENCLAW_LIVE_GATEWAY_PROVIDERS=deepseek,fireworks` and `opencode-go,openrouter` while also setting `OPENCLAW_LIVE_GATEWAY_MAX_MODELS=2`. The gateway live test caps candidates through `selectHighSignalLiveItems`, which consumes the global priority list before falling back to provider spread (`src/agents/live-model-filter.ts:307`). In that priority list, both DeepSeek entries come before Fireworks, and two OpenRouter entries come before `opencode-go/glm-5` (`src/agents/live-model-filter.ts:18`, `src/agents/live-model-filter.ts:23`). A focused local probe of that selector returned only `deepseek/deepseek-v4-flash, deepseek/deepseek-v4-pro` for the DeepSeek/Fireworks shard and only `openrouter/openai/gpt-5.2-chat, openrouter/minimax/minimax-m2.7` for the OpenCode/OpenRouter shard.
- Impact: Full release validation can report the new advisory Docker gateway shards as green without running Fireworks in the DeepSeek/Fireworks shard or OpenCode Go in the OpenCode/OpenRouter shard. That weakens the release gate for provider-specific Docker gateway regressions while docs say those provider shards are covered.
- Suggested fix: Use explicit `OPENCLAW_LIVE_GATEWAY_MODELS` per shard, or raise/split caps so each configured provider is guaranteed at least one selected model. Add a workflow/selector test that proves each advisory shard selects at least one model per provider.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: import `selectHighSignalLiveItems` and pass the high-signal candidates for each paired shard with `maxItems=2`; the selected list contains only DeepSeek for `deepseek,fireworks` and only OpenRouter for `opencode-go,openrouter`.
- Is this the best way to solve the issue? yes: explicit per-shard model lists are safest because the release gate should not depend on incidental global priority ordering.
