# Commit 096b91c

## Summary

Found two actionable issues: one product/privacy edge case in the new direct Slack DM path, and one stale adjacent Slack test that now fails the Slack plugin test lane.

## Findings

### Medium: `user:` targets with channel IDs now post to channels instead of failing as invalid DMs

- Kind: privacy
- File: `extensions/slack/src/send.ts`
- Line: 315
- Evidence: `parseSlackTarget` treats any `user:` prefix as `kind: "user"` without validating the ID shape (`extensions/slack/src/target-parsing.ts:25`). The new `resolveDirectUserPostChannelId` branch returns `params.recipient.id` for every `recipient.kind === "user"` when there is no media/thread (`extensions/slack/src/send.ts:303`, `extensions/slack/src/send.ts:312`, `extensions/slack/src/send.ts:315`), so `sendMessageSlack("user:C123CHAN", "secret", ...)` calls `chat.postMessage({ channel: "C123CHAN", text: "secret" })`. I verified this with an inline `pnpm exec tsx` mock, which printed `{"calls":[["post",{"channel":"C123CHAN","text":"secret"}]],"result":{"messageId":"1.000","channelId":"C123CHAN"}}`.
- Impact: A misconfigured or tool-provided target that is explicitly marked as a DM (`user:<id>`) can now publish the message to a Slack channel if the ID is actually channel-like. Before this commit, the same target went through `conversations.open({ users: ... })`, which would fail for a channel ID before any message was posted.
- Suggested fix: Only direct-post user targets whose IDs are valid Slack user IDs, and otherwise reject the target or keep the previous `conversations.open` failure path. Do not treat `recipient.kind === "user"` alone as enough to pass the raw ID to `chat.postMessage`.
- Confidence: high

### Low: Slack plugin test lane is red after the behavior change

- Kind: regression
- File: `extensions/slack/src/send.identity-fallback.test.ts`
- Line: 193
- Evidence: `pnpm test extensions/slack` fails one test: `sendMessageSlack customize-scope fallback > preserves Slack missing-scope details while opening DMs`. The test still expects text-only `sendMessageSlack("user:U123", "hello", ...)` to reject from `conversations.open`, but the commit intentionally bypasses `conversations.open` and resolves with `{ messageId: "171234.567", channelId: "U123" }`.
- Impact: The Slack plugin test lane fails on current `main` with 1 failed test out of 885, which can block future Slack validation and changed-gate runs.
- Suggested fix: Update this test to cover a path that still opens DMs, such as media uploads or threaded sends, if the missing-scope enrichment behavior still needs coverage.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. For the privacy edge case, call `sendMessageSlack("user:<real C-channel-id>", "secret", ...)` on this commit; the code path sends the raw channel-like ID to `chat.postMessage`. For the test regression, run `pnpm test extensions/slack`.
- Is this the best way to solve the issue? yes. The direct-DM fix should stay in `extensions/slack/src/send.ts`, but the direct path needs an ID-shape guard so malformed explicit DM targets cannot become channel sends.
