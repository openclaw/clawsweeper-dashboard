# Commit bca4e440bb

## Summary

Found one plausible regression: the broadened webhook suppression is named and described as bound-thread-only, but one call path passes generic configured channel bindings into the same helper. That can drop legitimate webhook-authored messages in configured Discord channel bindings, not just duplicate webhook copies in Discord session threads.

## Findings

### Medium: Webhook messages are suppressed for configured channel bindings, not only bound threads

- Kind: regression
- File: `extensions/discord/src/monitor/message-handler.preflight-helpers.ts`
- Line: 156
- Evidence: `shouldIgnoreBoundThreadWebhookMessage` now returns `true` for any non-empty `threadId` plus any `threadBinding`, regardless of whether the conversation is actually a Discord thread or whether the binding is the Discord thread-binding manager record. The second preflight call passes `threadBinding` from `resolveDiscordPreflightRoute`, and that value is set from `configuredBinding.record` when a normal configured channel binding exists (`extensions/discord/src/monitor/message-handler.routing-preflight.ts:73`, `extensions/discord/src/monitor/message-handler.routing-preflight.ts:88`). Existing configured ACP binding tests model this as a regular Discord channel binding for `channel-1`, with metadata `{ source: "config", mode: "persistent", agentId: "codex" }`, not a thread webhook binding (`extensions/discord/src/monitor/message-handler.preflight.acp-bindings.test.ts:35`).
- Impact: A Discord channel pinned to a configured ACP/session binding can no longer accept webhook-authored messages from integrations when `allowBots: true`; preflight returns `null` before mention/access handling and before `ensureConfiguredBindingRouteReady`. That is wider than the commit subject and changelog claim, which are specifically about duplicate webhook copies in already-bound Discord session threads.
- Suggested fix: Gate the broadened “any webhook id in a binding” suppression on an actual Discord thread-bound context. For example, pass `isBoundThreadSession`/`earlyThreadChannel` into the helper or keep the broad branch only for records resolved by `params.threadBindings.getByThreadId`; configured channel bindings should retain the previous exact-webhook-id/recently-unbound behavior.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the code path: configure a Discord channel ACP/session binding, set `allowBots: true`, then send a webhook-authored Discord message with `webhookId` into that channel. Static tracing shows the route resolver converts the configured binding into `threadBinding`, and the helper returns `true` before normal processing. I did not run a live Discord webhook replay.
- Is this the best way to solve the issue? Yes: the fix should narrow the new broad suppression to real thread-bound sessions rather than weakening the PluralKit duplicate fix globally.
