## Findings

### Low: `response.done` failures discard the OpenAI failure detail

- Kind: bug
- File: `ui/src/ui/chat/realtime-talk-webrtc.ts`
- Line: 151
- Evidence: The new `response.done` branch always calls `onStatus("listening", this.extractResponseStatus(event))`; `extractResponseStatus` only returns `Response ${status}` for non-completed statuses and ignores `response.status_details`. OpenAI documents `response.done` as always emitted, says clients should check `response.status`, and documents `status_details.error` as populated when status is `failed`. 
- Impact: A failed realtime response such as model access, server error, or content-filter failure can still show only `Response failed` while keeping Talk in the listening state, losing the actionable provider message that this commit was intended to surface.
- Suggested fix: For `response.done` with `status === "failed"`, extract `response.status_details.error.message/code/type` and report `onStatus("error", detail)`. For `incomplete`/`cancelled`, include `status_details.reason` when present.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Dispatch a data-channel message shaped like `{"type":"response.done","response":{"status":"failed","status_details":{"type":"failed","error":{"type":"invalid_request_error","code":"model_not_found","message":"model unavailable"}}}}`; current code reports only `listening` / `Response failed`.
- Is this the best way to solve the issue? yes. The fix belongs in the existing browser WebRTC event parser and can be covered by extending `ui/src/ui/realtime-talk-webrtc.test.ts`.
