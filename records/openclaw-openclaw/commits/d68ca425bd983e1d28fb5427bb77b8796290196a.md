# Commit d68ca42

## Summary

Found two low-severity status-reporting regressions in the new beta-channel update status plumbing.

## Findings

### Low: Channel-aware registry checks drop fetch errors

- Kind: regression
- File: `src/infra/update-check.ts`
- Line: 306
- Evidence: `fetchNpmLatestVersion()` preserves `fetchNpmTagVersion()` errors, but the new `fetchNpmRegistryVersionForChannel()` returns only `{ latestVersion, tag }`. `checkUpdateStatus()` now uses that helper whenever `registryChannel` is passed, and both `status` paths pass it via `src/commands/status.update.ts:28` and `src/cli/update-cli/status.ts:52`. A mocked 404 probe returned `latest: { latestVersion: null, error: "HTTP 404" }` but `stableChannel: { latestVersion: null, tag: "latest" }`.
- Impact: `openclaw status` / `openclaw update status` no longer surface npm registry errors for the channel-aware path. Since `formatUpdateOneLiner()` only prints `npm latest unknown` when `registry.error` exists, an offline/404 registry check can disappear from human output and JSON.
- Suggested fix: carry the selected tag and the underlying error through `resolveNpmChannelTag()` / `fetchNpmRegistryVersionForChannel()`, at least when the returned version is null.
- Confidence: high

### Low: Installed beta version masks git-derived channel source

- Kind: regression
- File: `src/infra/update-channels.ts`
- Line: 65
- Evidence: the new installed-version branch runs before config/git metadata. With `currentVersion: "2026.5.2-beta.1"`, `installKind: "git"`, and `gitTag: "v2026.5.2-beta.1"`, `resolveUpdateChannelDisplay()` returns `{ source: "installed-version", label: "beta (installed version)" }`; without `currentVersion`, the same git metadata returns `{ source: "git-tag", label: "beta (v2026.5.2-beta.1)" }`.
- Impact: beta git installs lose the more precise tag/branch source in status output and JSON `updateChannelSource`, even though the existing channel display contract had git-specific labels.
- Suggested fix: restrict the installed-version override to package installs, or place it after git tag/branch resolution so git metadata remains authoritative for git installs.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Both issues reproduce with focused inline probes against the touched helpers.
- Is this the best way to solve the issue? yes for error propagation; for channel precedence, the safer fix is to scope installed-version inference to package installs unless maintainers explicitly want git metadata hidden.
