# Commit d0cae0d

## Summary

Found one low-severity diagnostic/behavior regression in the new active-memory error classifier: it preserves allowlist-filtered memory errors, but still suppresses equivalent deny-policy-filtered memory errors.

## Findings

### Low: Memory tool deny policies are misclassified as missing memory plugins

- Kind: regression
- File: `extensions/active-memory/index.ts`
- Line: 518
- Evidence: `isMissingRegisteredMemoryToolsError` returns true when the allowlist-error sources contain only the runtime memory allowlist, or contain wildcard allow sources. Deny policies are applied before the embedded runner builds `effectiveTools` (`src/agents/pi-tools.ts:797`), and `buildEmptyExplicitToolAllowlistError` only receives the post-policy callable tool names plus allowlist sources (`src/agents/pi-embedded-runner/run/attempt.ts:1167`). A config such as `tools.deny: ["memory_search", "memory_get", "memory_recall"]` can therefore leave zero memory tools while producing an error source like `runtime toolsAllow: memory_recall, memory_search, memory_get`, which this new matcher treats as тАЬno memory tools registered.тАЭ
- Impact: active-memory reports `status=empty` and logs only debug тАЬno memory tools registeredтАЭ even when memory tools exist but policy explicitly denies them. That hides a real policy/configuration block, contrary to the new regression test that keeps allowlist-filtered policy errors visible.
- Suggested fix: make the core empty-tool error distinguish тАЬregistered but removed by policyтАЭ from тАЬno registered tools matched,тАЭ or have the active-memory matcher inspect relevant deny policies before suppressing. Add a regression test for memory tools removed by `tools.deny` or an agent/group/sandbox deny policy.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes: enable active-memory with a memory backend available, configure a tool policy deny for all active-memory memory tools, then send an eligible conversational message. I did not run that full live config, but the source path is deterministic: policy filtering precedes the empty explicit-allowlist error, and deny sources are not represented in the message that the new matcher parses.
- Is this the best way to solve the issue? Unclear. The safest direction is structured error metadata from the tool-policy/allowlist guard; parsing string text in the plugin is brittle and cannot see deny-policy causes.
