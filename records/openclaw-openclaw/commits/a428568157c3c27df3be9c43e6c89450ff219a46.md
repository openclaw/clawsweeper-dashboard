# Commit a428568

## Summary

Found one remaining trust-gating hole in the direct Gemini transport. The OpenAI-compatible path now treats API surface changes as cross-route, but the direct Gemini path still trusts a raw tool-call `thoughtSignature` when only provider and model match.

## Findings

### Medium: Direct Gemini replay still trusts cross-API tool-call thought signatures

- Kind: bug
- File: `extensions/google/transport-stream.ts`
- Line: 422
- Evidence: `convertGoogleMessages` computes `isSameProviderAndModel` from only `msg.provider === model.provider && msg.model === model.id`. For Gemini 3, the commit still calls `transformTransportMessages(..., { preserveCrossModelToolCallThoughtSignature: true })`, so a prior assistant turn from `api: "openai-completions"`, `provider: "google"`, and the same Gemini 3 model id keeps its captured `thoughtSignature`. The direct Gemini converter then treats it as same-source and emits that raw signature instead of `skip_thought_signature_validator`. The OpenAI-compatible fix in `src/agents/openai-transport-stream.ts:1835` correctly includes `source.api === model.api`, so the two paths now disagree on the trust boundary.
- Impact: Switching a session from Google’s OpenAI-compatible endpoint to the direct Gemini endpoint with the same Gemini 3 model can still replay an opaque signature from a different API surface. That can either reintroduce provider rejection for invalid cross-route signatures or bypass the intended “only trust same route” guard this commit is adding.
- Suggested fix: Make the direct Gemini same-source check include API equality, for example `msg.provider === model.provider && msg.api === model.api && msg.model === model.id`, and add a direct-transport test for same provider/model but `api: "openai-completions"` replaying to `api: "google-generative-ai"` and receiving the skip-validator signature.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the payload construction path: create a stored assistant tool-call turn with `api: "openai-completions"`, `provider: "google"`, `model: "gemini-3-flash-preview"`, and `thoughtSignature`, then build a direct `google-generative-ai` Gemini 3 payload for the same provider/model. Code inspection shows the raw signature survives and is emitted because the direct same-source check omits `api`.
- Is this the best way to solve the issue? Yes. The existing OpenAI-compatible side already defines the safer route identity as API + provider + model, so direct Gemini should use the same route identity and reserve `skip_thought_signature_validator` for cross-route Gemini 3 replay.
