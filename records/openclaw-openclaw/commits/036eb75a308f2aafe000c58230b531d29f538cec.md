# Commit 036eb75

## Summary

Found two actionable regressions: one breaks existing model-config tests because the new provider-barrel export was not added to mocks, and one can crash `models.json` planning on a parseable existing file containing a `null` provider row.

## Findings

### Medium: New provider-barrel import breaks existing mocked model-config tests

- Kind: regression
- File: `src/agents/models-config.plan.ts`
- Line: 172
- Evidence: The commit adds a required call to `normalizeProviderCatalogModelsForConfig(...)`, imported from `./models-config.providers.js`. Existing tests mock that barrel without the new export in `src/agents/models-config.uses-first-github-copilot-profile-env-tokens.test.ts:30` and `src/agents/models-config.skips-writing-models-json-no-env-token.test.ts:39`. Running `pnpm test src/agents/models-config.skips-writing-models-json-no-env-token.test.ts src/agents/models-config.uses-first-github-copilot-profile-env-tokens.test.ts -- --reporter=verbose` fails 9 tests with `No "normalizeProviderCatalogModelsForConfig" export is defined on the "./models-config.providers.js" mock`.
- Impact: The affected focused agent test shard is red on current `main`, blocking normal validation even though the production change itself is narrow.
- Suggested fix: Add `normalizeProviderCatalogModelsForConfig: (providers) => providers` to these mocks, or partially mock with `importOriginal()` and override only the functions each test needs.
- Confidence: high

### Low: Normalizing preserved providers can crash on null existing rows

- Kind: reliability
- File: `src/agents/models-config.providers.normalize.ts`
- Line: 49
- Evidence: `normalizeProviderCatalogModelsForConfig()` iterates merged providers and calls `normalizeProviderModelsForConfig(providerKey, provider)`, which immediately reads `provider.models`. Existing `models.json` data is parsed as `unknown` and cast in merge mode; a parseable file like `{ "providers": { "broken": null } }` now throws `TypeError: Cannot read properties of null (reading 'models')` through `src/agents/models-config.plan.ts:172`. I reproduced this with a focused `pnpm exec tsx -e ...planOpenClawModelsJsonWithDeps(...)` probe.
- Impact: A malformed but parseable existing `models.json` can make model-config preparation fail instead of regenerating or preserving/skipping the bad row. Adjacent code already treats non-object provider entries defensively in source-managed enforcement.
- Suggested fix: Guard `normalizeProviderModelsForConfig()` or `normalizeProviderCatalogModelsForConfig()` with the same record check used elsewhere, preserving non-object rows or skipping normalization for them.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The test regression reproduces with the two-file `pnpm test` command above. The null-row crash reproduces by calling `planOpenClawModelsJsonWithDeps()` in merge mode with `existingParsed.providers.broken = null`.
- Is this the best way to solve the issue? yes for the test regression: updating the mocks matches the new module contract. For the null-row crash, the safest fix is a narrow runtime guard around catalog-model normalization.
