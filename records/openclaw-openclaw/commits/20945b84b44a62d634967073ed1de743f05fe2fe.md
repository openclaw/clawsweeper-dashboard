# Commit 20945b8

## Summary

Found one concrete access-control regression in the Zalo-family wiring. Access groups are expanded only in the later command-authorization helper, but some inbound access gates still run first on the raw `accessGroup:<name>` strings and block otherwise-authorized senders.

## Findings

### Medium: Zalo/Zalo Personal access-group allowlists are checked too late

- Kind: bug
- File: `extensions/zalouser/src/monitor.ts`
- Line: 372
- Evidence: Zalo Personal builds `configAllowFrom` and `configGroupAllowFrom` directly from config, then calls `resolveDmGroupAccessWithLists` before the new `resolveSenderCommandAuthorization` access-group expansion can run. A DM config like `allowFrom: ["accessGroup:ops"]` with `accessGroups.ops.members.zalouser: ["123"]` is therefore denied by `isSenderAllowed("123", ["accessGroup:ops"])` and returns before expansion. Zalo groups have the same ordering problem: `extensions/zalo/src/monitor.ts:429` calls `evaluateZaloGroupAccess` with raw `groupAllowFrom`, and returns at `extensions/zalo/src/monitor.ts:457` before the expanded command-auth path at `extensions/zalo/src/monitor.ts:462`.
- Impact: Documented/static `message.senders` groups do not actually authorize Zalo group senders, and do not authorize Zalo Personal DM or group senders, even though the commit wires `channel: "zalo"` / `channel: "zalouser"` into the shared helper and current docs list both as supported.
- Suggested fix: Expand `allowFrom` / `groupAllowFrom` with `expandAllowFromWithAccessGroups` before the first inbound access decision in both monitors, then pass the expanded lists into `evaluateZaloGroupAccess` / `resolveDmGroupAccessWithLists` and command authorization. Add focused tests for Zalo group `groupAllowFrom: ["accessGroup:..."]` and Zalo Personal DM/group access groups.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Configure `accessGroups.ops = { type: "message.senders", members: { zalouser: ["123"] } }` and `channels.zalouser.dmPolicy = "allowlist"; channels.zalouser.allowFrom = ["accessGroup:ops"]`; an inbound DM from sender `123` is blocked at the raw allowlist gate before access-group expansion.
- Is this the best way to solve the issue? yes. The expansion needs to happen before the access-control decision, not only before command authorization.
