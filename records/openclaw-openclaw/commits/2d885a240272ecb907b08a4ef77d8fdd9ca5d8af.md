# Commit 2d885a2402

## Summary

Found one concrete regression in the new lock-owner disambiguation. The code compares two different timestamp formats with exact equality, so a live active installer can be mistaken for a recycled PID.

## Findings

### Medium: Active runtime-deps locks can be reclaimed because start times rarely compare exactly

- Kind: concurrency
- File: `src/plugins/bundled-runtime-deps.ts`
- Line: 606
- Evidence: Lock writers persist `pidStartTimeMs` from `Date.now() - process.uptime() * 1000` at `src/plugins/bundled-runtime-deps.ts:462`, `src/plugins/bundled-runtime-deps.ts:698`, and `src/plugins/bundled-runtime-deps.ts:2479`. The reader recomputes a live PID start time from `/proc/<pid>/stat` jiffies plus `Date.now() - os.uptime() * 1000` at `src/plugins/bundled-runtime-deps.ts:511`, then treats any non-null exact mismatch as stale at `src/plugins/bundled-runtime-deps.ts:606`. On this Linux runner, the same live process produced mismatches in 43 of 50 samples, with diffs from `-7ms` to `2ms`. A parent/child smoke check also showed a live child lock owner with `diffMs: -5` and `wouldRemoveActiveLock: true`.
- Impact: Any contended runtime-deps install on Linux can remove another still-running process's `.openclaw-runtime-deps.lock`, allowing concurrent `npm install` and `node_modules` replacement in the same install root. That breaks the active-install protection this lock is supposed to provide and can cause startup/install races or corrupted staged dependencies.
- Suggested fix: Store and compare the raw Linux `/proc/<pid>/stat` starttime jiffies instead of converted epoch milliseconds, ideally by reusing `src/shared/pid-alive.ts:getProcessStartTime`. Preserve `null` fallback for non-Linux or legacy owner files. Add a live/self-process or injected-rounding regression test; the current tests only mock exact `1_000 === 1_000` equality.
- Confidence: high
