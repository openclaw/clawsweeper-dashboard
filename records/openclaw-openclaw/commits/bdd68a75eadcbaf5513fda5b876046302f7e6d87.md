# Commit bdd68a75ea

## Summary

Found one low-severity incomplete-fix bug in the channel auto-enable path. The doctor repair path can now install missing official plugins, but gateway/config auto-enable can still re-add a missing known channel id to `plugins.allow` when no manifest declares that channel, preserving the warning loop that #76872 described whenever install repair cannot complete.

## Findings

### Low: Known channels without manifests can still be re-added to `plugins.allow`

- Kind: bug
- File: `src/config/plugin-auto-enable.shared.ts`
- Line: 282
- Evidence: When no manifest claims a configured channel, `collectPluginIdsForConfiguredChannel` still returns `builtInId` for known channels. Materialization then calls `ensurePluginAllowlisted` at `src/config/plugin-auto-enable.shared.ts:1009`, so a packaged install with no Discord manifest can still write `discord` into an existing restrictive allowlist. A focused smoke check against current `main` reproduced this: `channels.discord.enabled: true`, `DISCORD_BOT_TOKEN`, `plugins.allow: ["openai"]`, and an empty manifest registry produced `plugins.allow: ["openai", "discord"]` plus `Discord configured, enabled automatically.`
- Impact: If the external plugin install succeeds, doctor masks this by installing the plugin. If install repair fails or cannot run, the config still gets a `plugins.allow` entry that validation treats as missing/stale, so removing the warning remains non-sticky for the same class of Discord setup reported in #76872.
- Suggested fix: Only allowlist channel auto-enable candidates when a manifest actually declares the channel/plugin, or register the built-in channel enabled state without calling `ensurePluginAllowlisted` for manifestless known channels.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: run `applyPluginAutoEnable` with `channels.discord.enabled`, `DISCORD_BOT_TOKEN`, `plugins.allow: ["openai"]`, and `manifestRegistry: { plugins: [], diagnostics: [] }`.
- Is this the best way to solve the issue? yes: keep doctor’s missing-plugin install repair, but prevent manifestless channel fallback candidates from being persisted as plugin allowlist ids.
