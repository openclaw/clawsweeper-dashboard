# Commit 8d70f7e

## Summary

Found one runtime regression: the new Mistral Medium 3.5 reasoning catalog flag makes the default “off” thinking profile send `reasoning_effort=high` unless a caller explicitly passes `reasoning: "off"`.

## Findings

### Medium: Default thinking off still sends Mistral high reasoning

- Kind: regression
- File: `extensions/mistral/openclaw.plugin.json`
- Line: 97
- Evidence: The commit marks `mistral-medium-3-5` as `"reasoning": true` and `extensions/mistral/index.ts:50` advertises a thinking profile with `defaultLevel: "off"`. In the actual Chat Completions path, the upstream agent core maps `thinkingLevel === "off"` to `reasoning: undefined`, while `src/agents/openai-transport-stream.ts:1830` then defaults missing completions reasoning to `"high"` and `src/agents/openai-transport-stream.ts:2042` emits `reasoning_effort` whenever `model.reasoning` and `compat.supportsReasoningEffort` are true. The Mistral compat map only converts an explicit `off` to `"none"` at `extensions/mistral/api.ts:23`, so the advertised default-off state does not reach the request payload.
- Impact: A default Medium 3.5 session can run in high reasoning mode even though OpenClaw reports/selects off by default. Mistral documents `reasoning_effort="high"` as producing a full thinking chunk with increased token usage, while `"none"` omits the thinking chunk, so this can unexpectedly increase cost/latency and surface reasoning output. ([docs.mistral.ai](https://docs.mistral.ai/studio-api/conversations/reasoning/adjustable))
- Suggested fix: For Mistral adjustable-reasoning models, make the default-off path explicit before `buildOpenAICompletionsParams` runs. For example, add a Mistral provider transport patch/wrapper that sets `reasoning: "off"` or `reasoningEffort: "off"` when `thinkingLevel` is `off`, while still mapping enabled levels to `"high"`. Add a focused test that builds/sends a Medium 3.5 completions payload with default/off thinking and asserts `reasoning_effort: "none"` rather than `"high"`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. Code path reproduction: select `mistral/mistral-medium-3-5` with default thinking/off; the model is now `reasoning: true`, the agent core passes no `reasoning` option for off, and the OpenAI-compatible completions transport defaults missing reasoning to `"high"`.
- Is this the best way to solve the issue? Yes, the safest fix is provider-local request shaping for Mistral so the advertised thinking profile and the transport payload stay aligned without changing generic OpenAI completions defaults.
