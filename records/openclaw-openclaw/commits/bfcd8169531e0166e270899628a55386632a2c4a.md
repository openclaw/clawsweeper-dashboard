## Summary

One low-severity test regression found: a DeepSeek live assertion was rewritten in a way that no longer rejects extra replayed tool calls.

## Findings

### Low: DeepSeek live replay assertion no longer checks tool call count

- Kind: test_gap
- File: `extensions/deepseek/deepseek.live.test.ts`
- Line: 169
- Evidence: The base test used `toMatchObject({ tool_calls: [ ... ] })`; Vitest requires arrays passed to `toMatchObject` to have the same length, so a payload with two `tool_calls` failed. The new code only checks `Array.isArray(toolCalls)` and validates `toolCalls[0]`, so `[expectedCall, unexpectedExtraCall]` now passes.
- Impact: This live test is meant to prove DeepSeek V4 replay payload compatibility after a prior provider tool call. A regression that duplicates or appends an unintended replayed tool call would no longer be caught, even though that can create invalid or stale tool-call history for the provider.
- Suggested fix: Add `expect(toolCalls).toHaveLength(1)` before reading `[0]`, or restore an exact array assertion for `tool_calls`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused Vitest matcher probe confirmed `expect([{a:1},{a:2}]).toMatchObject([{a:1}])` fails, while the new first-element pattern would pass.
- Is this the best way to solve the issue? yes. A length assertion preserves the intended stricter coverage without changing the clearer per-field checks.
