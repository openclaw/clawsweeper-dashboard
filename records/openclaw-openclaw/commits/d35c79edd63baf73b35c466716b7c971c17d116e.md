# Commit d35c79e

## Summary

Found one transcript-loss regression in the new retry suppression path. The fix tracks `appendMessage(user)` as “persisted”, but pi’s session manager does not actually write an initial user message to disk until an assistant message exists.

## Findings

### Medium: Prompt-side fallback can drop the only persisted user turn for a new session

- Kind: data_loss
- File: `src/agents/session-tool-result-guard.ts`
- Line: 486
- Evidence: `installSessionToolResultGuard()` calls `onUserMessagePersisted` immediately after `originalAppend(finalMessage)` for user messages (`src/agents/session-tool-result-guard.ts:468`, `src/agents/session-tool-result-guard.ts:486`). `agent-command` then suppresses the next retry user when that flag is set (`src/agents/agent-command.ts:1026`). But `@mariozechner/pi-coding-agent@0.71.1` only flushes a new session’s initial user message once an assistant message exists. A focused smoke check showed: after appending only the first user, the session file did not exist; after simulating the retry user suppression and appending a successful assistant, the JSONL roles were `["session","assistant"]`.
- Impact: On a new session where the primary model emits the user `message_end` and then fails before any assistant response, common prompt-side failover paths such as rate limit, overload, timeout, or network errors can set `currentTurnUserMessagePersisted = true` without durable transcript content. The fallback retry then suppresses the user append, so the final successful assistant reply is persisted without the user request. Future turns, Control UI history, and transcript-based recovery lose the original prompt.
- Suggested fix: gate retry suppression on durable session content, not just the append callback. A narrow fix would include `await sessionFileHasContent(sessionFile)` in the suppression condition, or delay the callback until the current user entry is verifiably flushed to disk.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The dependency source emits user `message_end` before stream execution, and the smoke check confirmed pi’s session manager keeps an initial user append in memory only until an assistant flushes it.
- Is this the best way to solve the issue? unclear. The simplest safe direction is to require durable session content before suppressing a retry user; the more precise fix is tracking the specific current-turn user entry until it is actually flushed.
