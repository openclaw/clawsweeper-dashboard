# Commit f6a1d70

## Summary

Found one Mattermost regression in the new DM main-route pinning path. ID-based `allowFrom` works, but username/name-based Mattermost allowlists can now skip the legitimate owner’s main-session `lastRoute` update.

## Findings

### Medium: Mattermost username allowlists pin against sender IDs and skip the owner route

- Kind: regression
- File: `extensions/mattermost/src/mattermost/monitor.ts`
- Line: 1764
- Evidence: Mattermost direct-message access can allow a sender by username when `dangerouslyAllowNameMatching` is enabled: the normal access path passes both `senderId` and `senderName` through `isMattermostSenderAllowed` at `extensions/mattermost/src/mattermost/monitor.ts:1254`, and `normalizeMattermostAllowEntry` strips `@`, `user:`, and `mattermost:` prefixes at `extensions/mattermost/src/mattermost/monitor-auth.ts:13`. The new pin path instead derives `ownerRecipient` from raw `account.config.allowFrom` at `extensions/mattermost/src/mattermost/monitor.ts:1575`, but compares it only with `senderId` at `extensions/mattermost/src/mattermost/monitor.ts:1764`. For `allowFrom: ["@alice"]`, `dangerouslyAllowNameMatching: true`, and an inbound post from `user_id: "user-1"`, `sender_name: "alice"`, auth allows the DM, but the pin becomes owner `"alice"` vs sender `"user-1"`. `src/channels/session.ts:22` then treats that mismatch as non-owner and returns before updating `lastRoute` at `src/channels/session.ts:60`.
- Impact: A supported Mattermost DM configuration using username matching can no longer refresh the main session route when the owner sends a DM. Future shared-main-session replies may keep using a stale route instead of the owner’s current DM.
- Suggested fix: Only create `mainDmOwnerPin` for Mattermost when the single configured owner is a concrete user ID, or resolve username allowlist entries to concrete Mattermost user IDs before pinning. If name matching remains supported for DMs, the pin comparison needs to use the same match semantics or skip pinning for non-concrete name entries.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Configure Mattermost with `dmPolicy: "allowlist"`, `allowFrom: ["@alice"]`, and `dangerouslyAllowNameMatching: true`; send a direct post where Mattermost reports `user_id: "user-1"` and `sender_name: "alice"`. The normal auth path allows it, but the new pin asks session recording to skip the main `lastRoute` update.
- Is this the best way to solve the issue? yes, directionally. The route owner pin should be based on a stable recipient ID, not on mutable username strings.
