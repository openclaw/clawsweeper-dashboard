# Commit b830beb

## Summary

I found one low-severity status accuracy regression. The new status formatter can report an update restart as `verified` while the restart is only scheduled and has not run yet.

## Findings

### Low: `status` can show update restart as verified before the scheduled restart runs

- Kind: bug
- File: `src/commands/status-update-restart.ts`
- Line: 59
- Evidence: `update.run` writes the update restart sentinel before scheduling the gateway restart in `src/gateway/server-methods/update.ts:199` and `src/gateway/server-methods/update.ts:218`. `scheduleGatewaySigusr1Restart` defaults to a delayed restart and can also honor a requested delay, returning the pending delay at `src/infra/restart.ts:692` and `src/infra/restart.ts:813`. The new status path reads that sentinel at `src/commands/status.command.ts:288`, and `formatUpdateRestartStatusValue` formats any `status === "ok"` update sentinel as `verified` at `src/commands/status-update-restart.ts:58`.
- Impact: after a successful in-process `update.run`, `openclaw status` can display `Update restart: verified · gateway <version>` during the pending restart window, even though the old gateway process is still running and the restart has not been verified. This is especially visible when `restartDelayMs` is non-zero or a restart is coalesced/delayed.
- Suggested fix: distinguish “update succeeded and restart is scheduled/pending” from “restarted gateway was verified.” For example, only render `verified` after startup/finalization or health verification writes an explicit marker, and render existing `ok` sentinels as pending/scheduled until then.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. Trigger `update.run` on a git/dev install with a non-zero `restartDelayMs`, then run `openclaw status` before the scheduled SIGUSR1 fires; the sentinel is already `ok`, so the new formatter reports `verified`.
- Is this the best way to solve the issue? Yes: the status surface needs a separate verified state or should avoid claiming verification from a pre-restart `ok` sentinel.
