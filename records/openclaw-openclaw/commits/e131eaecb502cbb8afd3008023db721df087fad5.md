# Commit e131eae

## Summary

Found two concrete reliability gaps in the new “forced” package-update restart path. Both can still leave the old Gateway process running after the package tree has been swapped, which is the stale hashed chunk failure class described in #75206.

## Findings

### Medium: Package-update restarts still honor the 30s restart cooldown

- Kind: reliability
- File: `src/infra/restart.ts`
- Line: 676
- Evidence: `scheduleGatewaySigusr1Restart` always computes `cooldownMsApplied` from `lastRestartEmittedAt` and adds it into `requestedDueAt`. The new `update.run` call passes `delayMs: 0` and `skipDeferral: true` for package swaps, but `skipDeferral` only bypasses the pre-restart idle poll; it does not bypass the cooldown. Existing coverage proves a second restart after a handled restart is delayed 30s, and the new tests do not cover `skipDeferral` with a recent restart.
- Impact: If a package-manager `update.run` happens within the cooldown window, the old process can keep running for up to 30s after the global package tree was replaced. That preserves the exact stale in-memory chunk risk this commit is intended to eliminate.
- Suggested fix: Treat package-swap update restarts as cooldown-exempt, either by making `skipDeferral` also skip cooldown or by adding an explicit `forceImmediate`/`skipCooldown` option used only after a successful package swap.
- Confidence: high

### Medium: Update respawn is still delayed by normal active-work drain

- Kind: reliability
- File: `src/gateway/server-methods/update.ts`
- Line: 146
- Evidence: The handler schedules `reason: "update.run"` with `skipDeferral`, but the SIGUSR1 run loop still performs the normal restart drain before it reaches `respawnGatewayProcessForUpdate`. In `src/cli/gateway-cli/run-loop.ts`, all restarts wait on active tasks, embedded runs, and runtime-deps installs before `server.close`; only after close does `handleRestartAfterServerClose` branch on `restartReason === "update.run"` and hard-respawn. The new QA scenario explicitly exercises an agent calling `gateway.update.run`, which naturally leaves an active embedded run during the restart request.
- Impact: Agent-initiated package self-updates can continue executing old in-memory Gateway code after the package swap while waiting for the current run or active work to drain. Any lazy import during that window can still hit removed `dist` chunks and break inbound/reply recovery.
- Suggested fix: Carry the package-swap/update restart mode into the run-loop drain decision and skip or sharply bound active-work draining before update respawn. The restart sentinel/note should be the recovery path, not a long post-swap drain in the old process.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the cooldown path via the existing scheduler semantics; a focused test can emit and handle one restart, then schedule `skipDeferral: true` and observe `delayMs: 30000`. The drain path is source-evident and can be reproduced with a run-loop unit that sets active counts for an `update.run` restart.
- Is this the best way to solve the issue? Yes: the update-specific restart path should make the old process stop loading code from a replaced package tree as soon as the successful swap is known.
