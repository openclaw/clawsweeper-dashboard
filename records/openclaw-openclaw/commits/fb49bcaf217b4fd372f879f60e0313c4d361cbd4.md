# Commit fb49bca

## Summary

Found one security regression in the secret target registry fast path. The new current-snapshot reuse can load channel secret contract code from workspace plugins that are only discovered in metadata and were not enabled or trusted by config.

## Findings

### High: Secret target registry can execute untrusted workspace channel plugin contracts

- Kind: security
- File: `src/secrets/target-registry-data.ts`
- Line: 452
- Evidence: `loadSecretTargetRegistryFromPluginMetadata()` now accepts `getCurrentPluginMetadataSnapshot({ env, allowWorkspaceScopedSnapshot: true })` without passing any config or workspace policy, then treats every `record.channels.length > 0` plugin as a channel plugin and calls `listChannelSecretTargetRegistryEntries()` on it at `src/secrets/target-registry-data.ts:463`. That path calls `loadChannelSecretContractApiForRecord()`, and non-bundled records load the plugin contract module at `src/secrets/channel-contract-api.ts:138`. Workspace plugins are auto-discovered into workspace-scoped metadata snapshots from `.openclaw/extensions` at `src/plugins/discovery.ts:1058`, even before they are allowed/enabled for runtime.
- Impact: after gateway startup has stored a workspace-scoped plugin metadata snapshot, any workspace channel plugin present under the workspace plugin directory can run `contract-api.ts/js` side effects during secret target registry enumeration, including `getSecretTargetRegistry()`, `listSecretTargetRegistryEntries()`, and secrets audit/configure/apply paths. This bypasses the existing workspace plugin trust/activation gates and changes the previous behavior where this registry was loaded from `config: {}` without a workspace snapshot.
- Suggested fix: do not reuse the unscoped current snapshot for `getSecretTargetRegistry()`, or filter external channel records to enabled/trusted/loadable plugins before loading their contract API. The safest narrow fix is to keep this registry on the old `loadPluginMetadataSnapshot({ config: {}, env })` path, or only reuse a current snapshot when it is proven compatible with that same config and not merely workspace-scoped.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A temp workspace plugin with `channels: ["workspace-channel"]` and a `contract-api.ts` side effect was discovered in a metadata snapshot under `config: { plugins: {} }`. After `setCurrentPluginMetadataSnapshot(snapshot, { config, workspaceDir })`, `getSecretTargetRegistry()` returned the plugin’s target and the side-effect marker was written. Clearing the current snapshot before the same call returned `false` and did not write the marker.
- Is this the best way to solve the issue? yes for the fix direction. The risky part is specifically reusing an unscoped current snapshot for a path that executes external plugin contract code; keeping snapshot reuse config-compatible or filtering to enabled/trusted channel plugins preserves the performance goal without widening plugin execution.
