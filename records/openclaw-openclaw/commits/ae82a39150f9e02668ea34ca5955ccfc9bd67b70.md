# Commit ae82a39

## Summary

Found one regression risk in the new Active Memory fast-fail path: it treats the first empty `memory_search`/`memory_recall` tool result in the transcript as terminal, even though the recall prompt explicitly allows fallback and broader retry behavior.

## Findings

### Medium: First empty memory result aborts valid fallback recall

- Kind: regression
- File: `extensions/active-memory/index.ts`
- Line: 1635
- Evidence: `readTerminalMemorySearchResult` stops at the first record where `extractTerminalMemorySearchResultFromSessionRecord` returns empty, and `maybeResolveActiveRecall` races that watcher against the subagent, aborting the subagent and returning cached `status=empty` at `extensions/active-memory/index.ts:2501`. The same prompt tells the subagent to fall back from `memory_recall` to `memory_search` and to use permissive recall before deciding no useful memory exists at `extensions/active-memory/index.ts:933`.
- Impact: If the subagent performs an initial narrow search that writes `details.results: []`, then broadens the query or falls back to another memory tool that would return useful results, Active Memory now aborts before that useful result can be used. This can drop valid memory context and cache the empty result for the session/query.
- Suggested fix: Do not treat the first empty tool result as terminal. At minimum, scan the currently written transcript for any later non-empty memory result before resolving empty, and add a regression test where an empty `memory_search` is followed by a non-empty `memory_search` or fallback result. A safer design is to fast-fail only on explicit unavailable payloads, or after a small settle window that proves no follow-up successful memory result was written.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. In `extensions/active-memory/index.test.ts`, mock `runEmbeddedPiAgent` to write a transcript containing an empty `memory_search` result followed by a non-empty `memory_search` result, then delay before returning a useful payload. The current watcher reads the first empty result, wins the race, aborts, and returns no `prependContext`.
- Is this the best way to solve the issue? unclear. The fast-fail goal is valid for stalled terminal zero-hit paths, but the current transcript signal is not terminal. The fix should preserve fast failure for true unavailable/stalled cases without aborting fallback searches.
