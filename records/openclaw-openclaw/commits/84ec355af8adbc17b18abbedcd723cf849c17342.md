# Commit 84ec355

## Summary

Found one low-severity validation bug in the new QA-channel inline attachment base64 check. The focused QA-channel inbound test passes, but the validator is not a strict canonical base64 parser.

## Findings

### Low: Base64 round-trip check still accepts malformed padding

- Kind: bug
- File: `extensions/qa-channel/src/inbound.ts`
- Line: 22
- Evidence: `normalizeBase64ForCompare()` strips all trailing `=` before comparing decoded bytes back to the input. A local probe of the exact helper logic showed `"AAA@@@" -> rejected`, but `"====" -> 0 bytes`, `"AA====" -> 1 bytes`, and `"aGVsbG8====" -> 5 bytes`. Those malformed padded strings therefore reach `saveMediaBuffer()` at `extensions/qa-channel/src/inbound.ts:49`; `src/media/store.ts:523` only rejects oversized buffers, so a zero-byte/corrupt attachment can still be staged. The same helper also rejects line-wrapped base64 such as `"aG Vs bG8="`, while the existing canonical parser in `src/media/base64.ts:53` strips ASCII whitespace and enforces padding order/count.
- Impact: The commit’s hardening claim is incomplete: some malformed inline QA attachments are still accepted and surfaced as media paths, while some otherwise decodable wrapped payloads are dropped.
- Suggested fix: Decode only after canonicalizing with the SDK/media canonical base64 helper, or use equivalent local logic that skips ASCII whitespace, rejects padding before data, rejects more than two padding characters, requires non-empty canonical data, and decodes the canonical string. Add tests for `"===="`, extra trailing padding, mid-string padding, and wrapped valid base64.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Send a QA inbound attachment with `contentBase64: "===="` and a MIME type; the new helper returns an empty `Buffer` instead of `null`, so the attachment is staged.
- Is this the best way to solve the issue? yes. The repo already has strict canonical base64 logic in `src/media/base64.ts`, and current Teams inline-image code uses the same shape of padding/whitespace validation.
