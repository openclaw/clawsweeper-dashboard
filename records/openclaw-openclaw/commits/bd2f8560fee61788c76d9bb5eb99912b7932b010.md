# Commit bd2f856

## Summary

Found one regression in the new active WebChat dedupe path: the Gateway can now return an active run id that differs from the submitted idempotency key, but the browser Control UI still tracks the locally generated id and ignores the returned id.

## Findings

### Medium: Deduped Control UI sends can stop tracking the active run

- Kind: regression
- File: `src/gateway/server-methods/chat.ts`
- Line: 2075
- Evidence: This commit adds a path that responds to a duplicate active internal text send with `{ runId: activeRunId, status: "in_flight" }`, where `activeRunId` is the first run, not the duplicate request's `idempotencyKey` (`src/gateway/server-methods/chat.ts:2071`-`2078`). The browser Control UI call path ignores the `chat.send` response (`ui/src/ui/controllers/chat.ts:497`) and keeps `state.chatRunId` set to the newly generated local id (`ui/src/ui/controllers/chat.ts:629`-`635`). When subsequent chat events arrive for the actual active run id, `handleChatEvent` treats them as another run: deltas are dropped, and final events append without clearing `chatRunId` (`ui/src/ui/controllers/chat.ts:738`-`747`, `ui/src/ui/controllers/chat.ts:769`-`771`).
- Impact: A rapid duplicate submit after the first ack but before the first run completes can put browser WebChat into a stale active-run state. Streaming updates for the real active run are ignored, the final response may be handled as an unrelated run, and UI controls that depend on `chatRunId` can remain busy or abort the wrong run. This is directly on the bug surface this commit is trying to fix.
- Suggested fix: Make the WebChat client honor the accepted `runId` from `chat.send` and update/merge pending state when the server returns a different active run id, or keep server-side active-send dedupe from returning a different run id to clients that are known to track their generated id. Add a UI regression test where `chat.send` resolves `{ runId: "first", status: "in_flight" }` after the client generated `"second"`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Send `chat.send` from browser Control UI, let the first request ack while its run remains active, then submit the same trimmed text again with a fresh idempotency key. The Gateway returns the first run id by design; the browser client continues tracking the second id and filters the first run's delta events.
- Is this the best way to solve the issue? unclear. The Gateway-side dedupe is a reasonable guard for duplicate dispatch, but it needs a matching client contract update for surfaces that filter events by run id. Android and shared Swift chat code already appear to remap pending state when the returned run id differs; the browser Control UI should do the same.
