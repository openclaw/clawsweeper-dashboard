# Commit 1e8d9666b0

## Summary

Found one cache-state regression risk in the Dockerfile prune warmup.

## Findings

### Medium: pnpm store warmup can be skipped when the install layer is cached

- Kind: reliability
- File: `Dockerfile`
- Line: 71
- Evidence: `pnpm store add source-map@0.6.1` now runs in the install-layer `RUN` at `Dockerfile:71`, while the offline prune still runs later in `runtime-assets` at `Dockerfile:125` with `--config.offline=true`. Both steps rely on the same BuildKit cache mount id, but cache-mount writes are not part of the cached image layer. I reproduced the mechanism with an isolated two-stage BuildKit Dockerfile: after pruning only `exec.cachemount`, the second build reused the warmup step as `CACHED`, then the later same-id cache-mount check failed because the mount was empty.
- Impact: A Docker build can reuse the install layer while the pnpm exec cache is cold or evicted. In that state, `runtime-assets` can still execute `pnpm prune --prod --config.offline=true`, but the `source-map@0.6.1` warmup no longer necessarily happened in that build execution. This makes root Dockerfile release/smoke builds cache-state dependent again.
- Suggested fix: Put `pnpm store add source-map@0.6.1` back in the `runtime-assets` `RUN` immediately before `CI=true pnpm prune --prod`, or otherwise make the warmup execute in the same non-cached step as the offline prune. Update `src/dockerfile.test.ts:92` to assert the warmup is in the prune `RUN`, not only textually before prune.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Unclear for the full OpenClaw image because I did not run the heavyweight root Docker build, but yes for the underlying BuildKit cache failure mode: a cached warmup `RUN` does not repopulate an evicted cache mount before a later `RUN`.
- Is this the best way to solve the issue? Yes; the dependency is on a cache-mount side effect, so the robust fix is to keep the warmup in the same executing Docker step as the offline prune.
