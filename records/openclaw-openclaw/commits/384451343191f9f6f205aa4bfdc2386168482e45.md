## Findings

### Low: Timeout test now accepts an unreachable `minimum` branch

- Kind: test_gap
- File: `test/scripts/package-acceptance-workflow.test.ts`
- Line: 981
- Evidence: The changed assertion expects `release_checks.timeout-minutes` to be `${{ inputs.release_profile != 'minimum' && 240 || 60 }}`. On current `main`, `.github/workflows/full-release-validation.yml:29` defines `release_profile` as `type: choice`, but its options are only `beta`, `stable`, and `full` at `.github/workflows/full-release-validation.yml:34`. The actual monitored jobs now use the same `!= 'minimum'` expression at `.github/workflows/full-release-validation.yml:218` and `.github/workflows/full-release-validation.yml:418`, while the docs still describe `minimum` as a release profile at `docs/reference/full-release-validation.md:108`. GitHub’s workflow syntax docs describe `workflow_dispatch` `choice` inputs as single selectable options with explicit `options`, so a human manual dispatch will not have `minimum` available from this workflow. ([docs.github.com](https://docs.github.com/en/actions/reference/workflows-and-actions/workflow-syntax?utm_source=openai))
- Impact: The test no longer protects the previous budget distinction where non-full release monitors had a 60-minute parent timeout. Because `minimum` is absent from the top-level workflow choices, `normal_ci` and `release_checks` effectively get 240 minutes for every exposed profile. That can make beta or other focused release validation hangs tie up the parent monitor for four hours instead of one, and it leaves the documented `minimum` profile inconsistent with the runnable workflow.
- Suggested fix: Either add `minimum` to the Full Release Validation and Release Checks `release_profile` choice options and keep the new timeout expression, or change the timeout expression and tests to match the actually supported profiles, for example making only `stable`/`full` use the longer budget if `beta` is still intended to remain short.
- Confidence: high
