# Commit 3dcff3b

## Summary

Found one narrow gap in the HEIC fail-closed predicate: HEIC/HEIF sequence MIME types can still take the optional-Sharp-unavailable original-media fallback when the filename does not end in `.heic` or `.heif`.

## Findings

### Low: HEIC sequence MIME types still bypass the fail-closed Sharp fallback

- Kind: bug
- File: `src/media/web-media.ts`
- Line: 96
- Evidence: The new fallback guard rejects originals only when `isHeicSource(meta)` is true, but `HEIC_MIME_RE` is `^image\/hei[cf]$`. The installed `file-type` source classifies ISO BMFF brands `hevc` and `msf1` as `image/heic-sequence` and `image/heif-sequence`; a focused `detectMime` probe confirmed `hevc -> image/heic-sequence` and `msf1 -> image/heif-sequence`. Those MIME values are `image/*`, so `loadWebMedia` treats them as images, but `isHeicSource` returns false unless the filename also has a `.heic`/`.heif` suffix. If Sharp is unavailable and the buffer is under the cap, the changed guard at `src/media/web-media.ts:417` still returns the original sequence media.
- Impact: Generic/download-style HEIC sequence attachments without a HEIC filename can still be sent unconverted in the exact missing-Sharp fallback case this commit is trying to fail closed. That can produce unsupported outbound attachments or downstream provider/channel failures.
- Suggested fix: Include HEIC/HEIF sequence MIME values in `isHeicSource`, for example `^image\/hei[cf](?:-sequence)?$`, and add a regression test with `contentType: image/heic-sequence` or sniffed `hevc`/`msf1` data and no HEIC filename.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Mock `./image-ops.js` to throw the optional Sharp error as the existing test helper does, then call `loadWebMedia` on an under-cap image whose MIME is `image/heic-sequence` or `image/heif-sequence` and whose filename lacks `.heic`/`.heif`; the current predicate will allow the original fallback.
- Is this the best way to solve the issue? yes. The bug is in HEIC source classification, not in the fallback structure. Expanding the MIME predicate keeps the existing PNG/JPEG fallback behavior intact.
