# Commit c18cc76

## Summary

The commit fixes the prior detached-probe bug by awaiting ACPX startup probing by default. One reliability gap remains: the configured startup timeout only wraps `probeAvailability()`, not the follow-up `doctor()` call used when the probe completes unhealthy.

## Findings

### Medium: Unbounded doctor rerun can still block gateway ready after an unhealthy probe

- Kind: reliability
- File: `extensions/acpx/src/service.ts`
- Line: 397
- Evidence: `service.start()` wraps only `runtime.probeAvailability()` in `withStartupProbeTimeout()` at `extensions/acpx/src/service.ts:385`, then, when `runtime.isHealthy()` is false, awaits `runtime?.doctor?.()` with no timeout at `extensions/acpx/src/service.ts:397`. The pinned `acpx@0.7.0` package contract shows `doctor()` reruns `runProbe()`, and that probe awaits `client.start()` without the package’s `withTimeout()` helper. A focused smoke with `timeoutSeconds: 0.001`, a successful-but-unhealthy probe, and a never-resolving doctor left `service.start()` `still-pending-after-100ms`.
- Impact: With the new default startup probe enabled, a misbehaving ACP agent can still keep Gateway startup from reaching `gateway ready` after the initial probe reports unhealthy. Since plugin services are awaited in `src/plugins/services.ts:72` and readiness is marked after sidecars finish in `src/gateway/server-startup-post-attach.ts:803`, this can block channels/sidecars and contradict the changelog claim that ready waits until ACPX is usable or has reported a probe failure.
- Suggested fix: Bound the diagnostic `doctor()` call with the same startup timeout or remaining budget, or skip the second probe during startup and log a bounded generic unhealthy result. Add a regression where `probeAvailability()` resolves, `isHealthy()` returns false, and `doctor()` hangs; `service.start()` should still resolve after the configured timeout with a warning.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The focused `pnpm exec tsx` smoke described above proves the service-level timeout gap without needing a live ACP process.
- Is this the best way to solve the issue? yes. The startup readiness contract needs all probe-related work on the startup path to be bounded; wrapping or removing the diagnostic rerun is the narrow fix.
