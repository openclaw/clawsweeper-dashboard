# Commit c982358

## Summary

Found one narrow diagnostic-dedupe gap. The payload behavior still stays on `strict=false`, and the focused OpenAI transport tests pass, but the new dedupe key can miss identical tool inventories when their incoming order changes.

## Findings

### Low: Strict downgrade diagnostics are keyed before the transport's stable tool sort

- Kind: bug
- File: `src/agents/openai-transport-stream.ts`
- Line: 980
- Evidence: `resolveOpenAIStrictToolFlagWithDiagnostics` builds diagnostics from the raw `tools` array before `convertResponsesTools` and `convertTools` sort that same inventory for stable provider payloads. The key also includes `toolIndex` and preserves diagnostics array order at `src/agents/openai-transport-stream.ts:1014`, while existing tests already treat `[zeta, alpha]` and `[alpha, zeta]` as the same stable Responses payload in `src/agents/openai-transport-stream.test.ts:2935`.
- Impact: If a gateway exposes the same incompatible tool set in a different order across requests, the new cache logs the same provider/model/transport/tool-violation set again under a different hash. That weakens the intended "once per unique signature" behavior and can still produce repeated debug noise for nondeterministic tool ordering.
- Suggested fix: Build the diagnostic key from a canonical order, such as diagnostics sorted by `toolName`/violation text, or pass the same `sortTransportToolsByName(tools)` inventory into diagnostic generation before hashing.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes at source level: call the request builder twice with the same invalid named tools in opposite orders and debug logging enabled. The provider payloads sort to the same order, but the diagnostic hash differs because raw `toolIndex` and diagnostics order differ.
- Is this the best way to solve the issue? Yes: canonicalizing only the diagnostic key keeps strict-mode selection and payload shaping unchanged while matching the existing stable-tool-order contract.
