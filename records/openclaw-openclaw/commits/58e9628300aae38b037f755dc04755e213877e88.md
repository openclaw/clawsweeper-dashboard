# Commit 58e9628

## Summary

Found two actionable wrapper regressions around the new temporary full-checkout path for clean sparse Blacksmith/Testbox runs.

## Findings

### Medium: Blacksmith proof artifacts are written under the temporary checkout and then deleted

- Kind: data_loss
- File: `scripts/crabbox-wrapper.mjs`
- Line: 611
- Evidence: When a clean sparse checkout uses `provider=blacksmith-testbox`, the wrapper switches the child process `cwd` to a temporary full checkout at `scripts/crabbox-wrapper.mjs:557-613`. It rewrites explicit local path flags like `--emit-proof`, but Crabbox itself derives Blacksmith proof transcript artifact paths from its discovered repo root. Upstream Crabbox source stores successful Blacksmith `--emit-proof` artifacts under `req.Repo.Root/.crabbox/runs/...`, and the proof renderer includes those artifact paths in `Additional evidence`. With this wrapper, `req.Repo.Root` is the temporary checkout, and `cleanupOnce()` removes that checkout immediately after the child exits.
- Impact: A clean sparse Testbox run with `--emit-proof` can successfully write the proof file back to the original checkout, but the linked stdout/stderr/timing/metadata evidence files are deleted with the temporary checkout. The proof then points at dead local evidence paths.
- Suggested fix: Keep all local proof artifacts rooted at the original checkout when the child runs from a temporary sync checkout. A clean fix is an explicit Crabbox local-artifact-root override; a wrapper-only fix would need to copy/rewrite `.crabbox/runs` artifacts and proof paths before cleanup.
- Confidence: high

### Low: The supported `blacksmith` provider alias bypasses the sparse-checkout fix

- Kind: regression
- File: `scripts/crabbox-wrapper.mjs`
- Line: 372
- Evidence: The wrapper accepts provider aliases through `providerAliases`, including `blacksmith -> blacksmith-testbox`, but `selectedProvider()` returns the raw value. `shouldUseFullCheckoutForCleanSparseBlacksmithSync()` only matches `providerName !== "blacksmith-testbox"`, so `--provider blacksmith` runs from the sparse checkout even though Crabbox documents and supports that alias.
- Impact: Users invoking the supported alias still get the old sparse sync behavior: Blacksmith sync sees the sparse worktree rather than the temporary full checkout, so files outside the sparse cone can be absent from the remote Testbox.
- Suggested fix: Canonicalize the selected provider once and use the canonical name for Blacksmith-specific warnings and temporary-checkout routing.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. For the artifact issue, run a successful clean sparse `node scripts/crabbox-wrapper.mjs run --provider blacksmith-testbox ... --emit-proof <repo-relative-proof> -- <command>` with Blacksmith auth; the proof is written to the original checkout while additional evidence paths are rooted in the temporary checkout that the wrapper deletes. For the alias issue, use the same clean sparse setup with `--provider blacksmith`.
- Is this the best way to solve the issue? Unclear for the artifact issue until Crabbox decides whether local artifact root should be an upstream contract or wrapper-owned post-processing. The alias fix is straightforward provider canonicalization.
