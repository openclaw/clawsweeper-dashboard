# Commit 156e86a

## Summary

Found one low-severity compatibility regression in the new synchronous loader path.

## Findings

### Low: `plugins build` now rejects ESM tool entries that use top-level await

- Kind: compatibility
- File: `src/cli/plugins-authoring-command.ts`
- Line: 101
- Evidence: Before this commit, `importToolPluginEntry` used `await import(pathToFileURL(entryPath).href)`, preserving native ESM semantics. The new code calls the synchronous jiti loader via `loader(toSafeImportPath(entryPath))`. A focused probe on current `main` with a built `dist/index.js` tool plugin containing `await Promise.resolve()` before `export default defineToolPlugin(...)` fails with `SyntaxError: await is only valid in async functions and the top level bodies of modules`. The same entry loads through `await jiti.import(entry)` with the same SDK alias map.
- Impact: Valid ESM tool plugin entries that previously loaded during `openclaw plugins build` or `openclaw plugins validate` can now fail if they use top-level await. This is an edge case, but it is a behavioral regression from the async native import path.
- Suggested fix: Keep the SDK alias behavior but load authoring entries through jiti’s async import API, for example `await loader.import(toSafeImportPath(entryPath))`, then apply the same default/createEntry/entry unwrapping. Add a regression test for a built ESM `.js` entry with top-level await.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes; create a temporary ESM tool plugin entry with `await Promise.resolve()` before the `defineToolPlugin` default export, then call `loadToolPlugin` on `dist/index.js`.
- Is this the best way to solve the issue? likely; the command is already async, and jiti’s documented async import path preserves the alias support added by this commit while restoring async ESM loading semantics.
