# Commit f4b5e58

## Summary

Found one migration gap: providerless OpenAI retired model refs are still accepted by runtime compatibility paths, but `doctor --fix` does not upgrade them after this commit removes the retired catalog rows.

## Findings

### Medium: Doctor misses bare retired OpenAI model refs

- Kind: regression
- File: `src/commands/doctor/shared/legacy-config-migrations.runtime.models.ts`
- Line: 308
- Evidence: `upgradeRetiredModelRef` only calls `upgradeRetiredOpenAiModelId` when the parsed ref has provider `openai`, `openai-codex`, or `github-copilot`. For a providerless value like `gpt-5.2`, `provider` is `undefined`, so it skips the new retired OpenAI mapping and falls through to Claude-only handling. Runtime still accepts providerless model strings for compatibility and defaults unresolved bare refs to the default provider, which is `openai` (`src/agents/model-selection-shared.ts:612`, `src/agents/defaults.ts:3`).
- Impact: Existing configs using accepted bare refs such as `agents.defaults.model.primary: "gpt-5.2"` or fallback `"gpt-4o-mini"` are not repaired by `openclaw doctor --fix`, even though this commit prunes those OpenAI catalog entries and changelog/docs tell users the doctor migration upgrades existing configs. I reproduced this with `node --import tsx -e 'import { migrateLegacyConfig } from "./src/commands/doctor/shared/legacy-config-migrate.ts"; ... "gpt-5.2" ...'`, which returned `config: null` and `changes: []`; the same shape with `openai/gpt-5.2` rewrites to `openai/gpt-5.5`.
- Suggested fix: Extend `upgradeRetiredModelRef` to treat providerless retired OpenAI-family IDs as OpenAI refs, preferably rewriting them to explicit `openai/<current-model>` refs, and add coverage for bare `gpt-5.2`, `gpt-4o-mini`, and similar pruned IDs.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The focused `migrateLegacyConfig` probe shows no migration for bare `gpt-5.2`, while the provider-qualified equivalent is upgraded.
- Is this the best way to solve the issue? yes. The bug is in the migration decision boundary; fixing the owner inference there keeps the runtime clean and matches the commit’s doctor-based upgrade contract.
