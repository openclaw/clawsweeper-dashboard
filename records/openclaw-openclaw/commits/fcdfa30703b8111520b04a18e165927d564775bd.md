# Commit fcdfa30

## Summary

The normal gateway WhatsApp listener path now threads `authDir` into `createWebSendApi`, and the new PN-to-LID resolver matches Baileys' multi-file auth storage shape. I found one missed shipped call path: the exported WhatsApp QA driver has the same real send API but still omits its `authDir`, so its `sendText` helper remains PN-only for LID-backed contacts.

## Findings

### Low: WhatsApp QA driver sends still skip the LID forward mapping

- Kind: bug
- File: `extensions/whatsapp/src/qa-driver.runtime.ts`
- Line: 152
- Evidence: `startWhatsAppQaDriverSession` accepts `params.authDir`, passes it to `createWaSocket`, and uses it for inbound `jidToE164` normalization, but creates `createWebSendApi({ sock, defaultAccountId: "qa-driver" })` without `authDir`. The new resolver in `extensions/whatsapp/src/inbound/send-api.ts:50` only uses `toWhatsappJidWithLid` when `params.authDir` is set; otherwise it falls back to `toWhatsappJid`, preserving the old `{digits}@s.whatsapp.net` routing.
- Impact: Live QA driver sessions exported through `extensions/whatsapp/api.ts` can still reproduce the ghost-chat failure this commit fixes for the gateway path when `session.sendText("+...")` targets a LID-backed contact. That also weakens live validation because the QA helper does not exercise the fixed behavior.
- Suggested fix: Pass `authDir: params.authDir` into `createWebSendApi` in `qa-driver.runtime.ts` and extend `qa-driver.runtime.test.ts` so its mock asserts the auth directory is threaded into the send API.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused mocked test can start `startWhatsAppQaDriverSession({ authDir })`, call `sendText`, and assert either `createWebSendApi` receives `authDir` or the underlying socket send targets `{lid}@lid` when `lid-mapping-{digits}.json` exists.
- Is this the best way to solve the issue? yes. The QA driver already owns the correct auth directory and the shared send API already implements the fixed resolver; this is a one-argument wiring fix.
