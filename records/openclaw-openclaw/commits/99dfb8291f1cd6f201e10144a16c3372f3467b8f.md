# Commit 99dfb829

## Summary

Found one remaining replay bug in the new model-id allowlist normalization. The committed tests cover plain `moonshotai/kimi-k2.6`, bare Kimi/MiMo IDs, and colon suffixes, but not colon-prefixed model IDs such as `hf:moonshotai/Kimi-K2-Thinking`.

## Findings

### Medium: Colon-prefixed Kimi model IDs still lose required reasoning replay

- Kind: bug
- File: `src/agents/transcript-policy.ts`
- Line: 173
- Evidence: `requiresReasoningContentReplay()` normalizes with `.split(":", 1)[0]` before taking the final slash segment. For `hf:moonshotai/Kimi-K2-Thinking`, that produces `hf`, so `dropReasoningFromHistory` stays `true`. The transport has the same normalization at `src/agents/openai-transport-stream.ts:2499`, so a manual probe of `buildOpenAICompletionsParams()` for `hf:moonshotai/Kimi-K2-Thinking` emitted an assistant tool-call message with no `reasoning_content`.
- Impact: Kimi thinking routes using this valid OpenClaw model-id shape still hit the exact class of follow-up tool-call replay failure this commit is meant to fix. Kimi’s docs describe `kimi-k2-thinking`, `kimi-k2.6`, and `kimi-k2.5` as reasoning-capable and say multi-step/tool-call contexts should include prior `reasoning_content`. ([platform.kimi.com](https://platform.kimi.com/docs/guide/use-kimi-k2-thinking-model))
- Suggested fix: normalize by taking the final slash segment first, then stripping only a suffix after the model segment, or share a helper that handles both `model:variant` and `hf:vendor/model` forms. Add regression coverage in both transcript policy and transport.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Run `node --import tsx` against `resolveTranscriptPolicy({ provider: "custom-openai-proxy", modelId: "hf:moonshotai/Kimi-K2-Thinking", modelApi: "openai-completions" })`; it returns `dropReasoningFromHistory: true`. A second probe through `buildOpenAICompletionsParams()` strips `reasoning_content` from the replayed assistant tool-call message.
- Is this the best way to solve the issue? yes. The fix should be narrow normalization repair plus tests, not a broader replay-policy change.
