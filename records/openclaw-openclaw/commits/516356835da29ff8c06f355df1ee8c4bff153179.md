## Findings

### Medium: scoped dynamic HTTP route registration is not shared across duplicate SDK module copies

- Kind: regression
- File: `src/plugins/http-registry.ts`
- Line: 13
- Evidence: `withPluginHttpRouteRegistry()` stores the server-local route registry in a module-local `new AsyncLocalStorage<PluginRegistry>()`, and `registerPluginHttpRoute()` reads only that local store before falling back to `requireActivePluginHttpRouteRegistry()` at `src/plugins/http-registry.ts:33`. But `registerPluginHttpRoute` is exported through public SDK subpaths (`src/plugin-sdk/webhook-targets.ts:2`, `package.json:1319`), while comparable request scope code uses `resolveGlobalSingleton(Symbol.for(...))` specifically to survive duplicate runtime module copies (`src/plugins/runtime/gateway-request-scope.ts:22`). The fallback registry state itself is process-global via `Symbol.for("openclaw.pluginRegistryState")`, so a plugin using a second SDK copy can miss the scoped store and register into whichever process-global HTTP route registry is currently pinned.
- Impact: external or separately staged plugins that import `openclaw/plugin-sdk/webhook-targets` / `webhook-ingress` can still register dynamic HTTP routes against another live gateway instance in the same process. That undermines the commit’s hardening for channel startup, plugin services, and `gateway_start` hooks, and can cause webhooks/control-plane plugin routes to execute on or disappear into the wrong gateway.
- Suggested fix: make `pluginHttpRouteRegistryScope` a process-global singleton using `resolveGlobalSingleton(Symbol.for("openclaw.pluginHttpRouteRegistryScope"), () => new AsyncLocalStorage<PluginRegistry>())`, matching the gateway request scope pattern, and add a regression test that imports/registers through an isolated SDK module copy.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Unclear. The source path is concrete, but I did not build a packaged external-plugin fixture with a second `openclaw` SDK copy during this read-only review.
- Is this the best way to solve the issue? Yes. The existing gateway request-scope singleton pattern is the narrow fix and preserves the intended server-local route scoping.
