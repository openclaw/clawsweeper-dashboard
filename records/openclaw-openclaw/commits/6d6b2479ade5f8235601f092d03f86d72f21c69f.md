# Commit 6d6b247

## Summary

Found one actionable ordering regression in the imported history merge comparator.

## Findings

### Medium: Mixed timestamped and untimestamped history can now sort inconsistently

- Kind: bug
- File: `src/gateway/cli-session-history.merge.ts`
- Line: 131
- Evidence: `compareHistoryMessages` now compares by timestamp only when both messages have timestamps, otherwise by insertion order. With local messages `[timestamp=1000, no timestamp]` and an imported message `timestamp=0`, the comparator forms a cycle: local timestamped `<` local untimestamped by order, local untimestamped `<` imported by order, but imported `<` local timestamped by timestamp. A focused smoke check of `mergeImportedChatHistoryMessages` returned `local timestamped later | local untimestamped | imported older`, leaving an older imported message after a newer timestamped local message.
- Impact: `chat.history` can display merged Claude CLI history out of chronological order for sessions that contain a mix of timestamped and untimestamped local transcript entries. Because the comparator is not a total order, future Node/V8 sort behavior or input shape can also produce hard-to-reason-about ordering.
- Suggested fix: make the merge use one consistent total ordering policy. For example, detect missing timestamps and either preserve insertion order for the whole merge, or compute explicit stable sort keys/segments before sorting so pairwise comparisons cannot switch between timestamp and insertion-order semantics.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes: run a small `node --import tsx` call importing `mergeImportedChatHistoryMessages` with the mixed timestamp scenario above.
- Is this the best way to solve the issue? unclear: the intended product rule for untimestamped local messages needs to be explicit, but the comparator should be made transitive whichever rule is chosen.
