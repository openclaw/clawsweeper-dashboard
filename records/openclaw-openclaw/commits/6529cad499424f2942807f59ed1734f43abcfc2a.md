# Commit 6529cad

## Summary

Found one compatibility regression in the new preload skip path.

## Findings

### Medium: Gateway-owned preload skip can bypass configured bundled-plugin overrides

- Kind: regression
- File: `src/cli/program/message/helpers.ts`
- Line: 98
- Evidence: The new preload planner skips `ensurePluginRegistryLoaded(...)` when `getChannelPlugin(scopedChannel)?.actions?.resolveExecutionMode(...)` returns `"gateway"`. In a fresh CLI process, `getChannelPlugin` falls back to the bundled plugin when no runtime registry is loaded (`src/channels/plugins/registry.ts:32`). The later outbound resolver also falls back the same way after a no-op bootstrap (`src/infra/outbound/channel-resolution.ts:72`). That bypasses the previous scoped registry load path, which maps `onlyChannelIds` to the effective owner plugin ids before loading (`src/plugins/runtime/runtime-registry-loader.ts:136`). Existing loader coverage establishes that config-loaded and installed duplicate plugin ids can override bundled plugins after the registry is loaded (`src/plugins/loader.test.ts:5842`, `src/plugins/loader.test.ts:5911`).
- Impact: A user with an installed or explicitly loaded override for a bundled gateway-owned channel plugin, such as a patched `telegram` or `discord` plugin, can run `openclaw message ... --channel telegram` or a prefixed target and have the CLI choose the bundled channel contract instead of the configured override. That can route to the wrong execution mode, action handler, or channel metadata.
- Suggested fix: Do not skip preload based only on the bundled fallback. Either preload when manifest/source precedence says the scoped channel may be owned by a configured or installed plugin, or only allow the fast path when the active loaded plugin is already the effective owner and no higher-precedence override is pending.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes, source-level: configure or install a duplicate plugin id for a bundled gateway-owned channel, then run an explicit gateway-owned message action in a fresh CLI process. The new helper can skip the registry load that would apply source precedence.
- Is this the best way to solve the issue? Yes in direction: keep the gateway fast path, but gate it on effective owner resolution rather than `getChannelPlugin`’s bundled fallback.
