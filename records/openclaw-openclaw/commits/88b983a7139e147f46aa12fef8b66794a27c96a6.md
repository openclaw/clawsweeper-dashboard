# Commit 88b983a

## Summary

Found three actionable regressions: the touched Google Meet test file fails on Linux, realtime Meet transcripts are now logged at info level, and deferred bridge startup can leave an in-call Chrome participant untracked if bridge startup fails after joining.

## Findings

### High: New local Chrome realtime tests fail on Linux

- Kind: compatibility
- File: `extensions/google-meet/index.test.ts`
- Line: 1868
- Evidence: `assertBlackHole2chAvailable` rejects non-macOS hosts at `extensions/google-meet/src/transports/chrome.ts:57`, but the three new local Chrome realtime tests at `extensions/google-meet/index.test.ts:1868`, `extensions/google-meet/index.test.ts:1913`, and `extensions/google-meet/index.test.ts:1990` do not mock `process.platform` to `darwin`. Running `pnpm test extensions/google-meet/index.test.ts extensions/openai/realtime-voice-provider.test.ts` fails those three tests with `expected false to be true`.
- Impact: the changed Google Meet test shard is red on Linux, so the commit can break normal CI/changed-lane validation even though the production code path is macOS-only.
- Suggested fix: wrap these tests with the same `process.platform = "darwin"` override used by adjacent local Chrome tests, or make them exercise `chrome-node`/non-realtime paths when running on Linux.
- Confidence: high

### Medium: Private meeting transcript text is now written to info logs

- Kind: privacy
- File: `extensions/google-meet/src/realtime.ts`
- Line: 412
- Evidence: final realtime transcripts are now logged with `params.logger.info("[google-meet] realtime ${role}: ${text}")`. The paired-node path does the same at `extensions/google-meet/src/realtime-node.ts:159`. Before this commit, the same transcript text was logged only through optional debug logging.
- Impact: Google Meet speech content can be persisted or surfaced in normal OpenClaw logs by default. That is a privacy regression for private meetings, especially because both user speech and assistant replies are logged verbatim.
- Suggested fix: keep verbatim transcript logging behind debug/trace logging, or log only counts/timestamps at info level while exposing transcript text through explicit status/debug commands.
- Confidence: high

### Medium: Bridge-start failure after join leaves Chrome Meet participant untracked

- Kind: reliability
- File: `extensions/google-meet/src/transports/chrome.ts`
- Line: 177
- Evidence: `launchChromeMeet` now opens/joins the Meet first, then starts the realtime bridge only after `browser.inCall === true`. If `startRealtimeAudioBridge()` throws at that point, the exception propagates back through `GoogleMeetRuntime.join`, whose catch at `extensions/google-meet/src/runtime.ts:494` rethrows before the session is stored at `extensions/google-meet/src/runtime.ts:499`.
- Impact: a bridge command failure or provider bridge connection failure after Meet admission can leave the browser participant in the meeting with no tracked OpenClaw session, so `googlemeet leave/status/speak` cannot manage or clean it up.
- Suggested fix: once Chrome is in-call, either record a session with `speechBlockedReason: "audio-bridge-unavailable"` and the bridge error, or explicitly clean up/leave/close the tab before throwing.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The test regression reproduces with `pnpm test extensions/google-meet/index.test.ts extensions/openai/realtime-voice-provider.test.ts` on Linux. The untracked participant path follows directly from `openMeetWithBrowserRequest` returning in-call, `startRealtimeAudioBridge()` throwing, and `join` not storing the session until after `launchChromeMeet` returns.
- Is this the best way to solve the issue? yes for the test fix and transcript logging fix. For bridge startup, the safer product behavior is to keep the session trackable after Chrome joins; cleanup-before-throw is acceptable if the browser transport has a reliable leave/close primitive.
