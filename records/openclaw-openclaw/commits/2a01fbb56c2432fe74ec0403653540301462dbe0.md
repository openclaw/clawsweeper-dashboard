# Commit 2a01fbb

## Summary

Found one release automation regression. The commit makes ClawHub non-blocking when `wait_for_clawhub=false`, but it also stops the wrapper from auto-approving the protected ClawHub publish environment, so the default sidecar publish can remain pending instead of running in the background.

## Findings

### Medium: Default ClawHub sidecar can remain stuck pending environment approval

- Kind: regression
- File: `.github/workflows/openclaw-release-publish.yml`
- Line: 736
- Evidence: `wait_for_run()` is the only code path that calls `approve_pending_deployments` for child runs at `.github/workflows/openclaw-release-publish.yml:512`. After this commit, the ClawHub child is only waited when `WAIT_FOR_CLAWHUB == "true"` at `.github/workflows/openclaw-release-publish.yml:736`; otherwise the workflow just writes “not awaited”. The ClawHub child publish job uses `environment: clawhub-plugin-release` at `.github/workflows/plugin-clawhub-release.yml:295`. Live repository environment metadata from `gh api repos/openclaw/openclaw/environments/clawhub-plugin-release` shows that environment has required reviewers.
- Impact: The default release-candidate publish command sets `wait_for_clawhub=false`, and the workflow input also defaults to false. With that default, `OpenClaw Release Publish` still dispatches `plugin-clawhub-release.yml`, but no longer performs the approval step that previously let the protected publish job proceed while OpenClaw npm was being published. Unless a release manager manually notices and approves the separate child run, the ClawHub sidecar can sit pending instead of publishing asynchronously.
- Suggested fix: Keep ClawHub non-blocking, but decouple environment approval from final waiting. After dispatching the ClawHub child, poll/approve its pending deployments briefly even when `WAIT_FOR_CLAWHUB=false`, then stop waiting and keep `--skip-clawhub` for final postpublish verification.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes: dispatch `OpenClaw Release Publish` with the default `wait_for_clawhub=false` and a ClawHub-publishable plugin set; the wrapper will dispatch the ClawHub child but will not call the only approval helper for its protected environment. I did not execute this because it would mutate release/publish state.
- Is this the best way to solve the issue? Yes: approving the child environment independently preserves the intended advisory/non-blocking ClawHub behavior without restoring ClawHub as a top-level completion gate.
