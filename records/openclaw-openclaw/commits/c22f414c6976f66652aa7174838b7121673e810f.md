# Commit c22f414

## Summary

The watchdog fix is covered by focused tests and matches the #77984 failure mode, but the new timeout diagnostic stores raw assistant text before filtering notifications to the active turn. With the shared app-server client, that can put assistant text from the wrong concurrent Codex turn into another run's warn logs and trajectory timeout event.

## Findings

### Medium: timeout diagnostics can log raw assistant text before current-turn filtering

- Kind: privacy
- File: `extensions/codex/src/app-server/run-attempt.ts`
- Line: 918
- Evidence: `handleNotification` calls `touchTurnCompletionActivity(..., { details: describeNotificationActivity(notification) })` for every app-server notification before it computes `isCurrentTurnNotification` at `extensions/codex/src/app-server/run-attempt.ts:927`. `describeNotificationActivity` extracts up to 240 chars of assistant content into `lastAssistantTextPreview` at `extensions/codex/src/app-server/run-attempt.ts:1854`, and both timeout paths spread those details into trajectory events and `embeddedAgentLog.warn` at `extensions/codex/src/app-server/run-attempt.ts:772` and `extensions/codex/src/app-server/run-attempt.ts:818`. The default factory uses a shared app-server client (`extensions/codex/src/app-server/client-factory.ts:19`), and that client broadcasts every notification to every registered handler (`extensions/codex/src/app-server/client.ts:246`, `extensions/codex/src/app-server/client.ts:437`). The new test also asserts warn metadata includes `lastAssistantTextPreview` (`extensions/codex/src/app-server/run-attempt.test.ts:965`).
- Impact: In concurrent Codex runs sharing the same app-server process, a `rawResponseItem/completed` from one thread can become the last activity details for another thread and later be written to that other run's timeout log/trajectory. Even without concurrency, this moves arbitrary assistant text into warn-level gateway diagnostics; standard log redaction masks token-like secrets, not general private transcript content.
- Suggested fix: Only compute/store notification activity details after confirming the notification belongs to the current `threadId`/`turnId`, and avoid logging assistant text previews by default. Keep item metadata such as method/id/type/role, or gate text previews behind an explicit debug/trajectory option with stronger redaction.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the raw-text log path: the added unit test proves `lastAssistantTextPreview` reaches `embeddedAgentLog.warn`. The cross-turn bleed follows from the shared client broadcasting every notification to each run handler, but I did not run a live concurrent Codex app-server scenario.
- Is this the best way to solve the issue? Yes: filtering before capturing details preserves the watchdog behavior while avoiding cross-session diagnostic contamination. Dropping the text preview is safer because metadata already explains the timeout shape.
