# Commit 6f93365

## Summary

Found one low-severity bug: the patch changes `deriveIdHint`, but the linked warning path on current `main` is not produced by that branch for manifest-backed bundled plugins, and stale persisted mismatch diagnostics can still be replayed into gateway config warnings.

## Findings

### Low: Linked plugin-id warnings can survive because the fix is applied outside the active warning path

- Kind: bug
- File: `src/plugins/discovery.ts`
- Line: 550
- Evidence: `deriveIdHint` returns `rawManifestId` before package-name normalization, so manifest-backed packages like `@openclaw/xai-plugin` with manifest id `xai` never reach the new `-plugin` stripping branch. Current registry tests also assert manifest id is authoritative and id-hint mismatches do not warn. The still-live replay path copies persisted installed-index diagnostics in `src/plugins/manifest-registry-installed.ts:455` and config validation surfaces them as warnings in `src/config/validation.ts:1009`. A focused probe confirmed a persisted `plugin id mismatch (manifest uses "xai", entry hints "xai-plugin")` diagnostic is replayed unchanged.
- Impact: Users with existing persisted plugin-registry diagnostics can still see the startup warning that #85048 was closed as fixed, even after this commit.
- Suggested fix: Filter or invalidate obsolete persisted `manifest uses ... entry hints ...` diagnostics, or refresh the installed plugin index for that diagnostic shape. Keep real current loader/export id mismatches intact.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes, source-level: construct an installed index with an existing-source `plugin id mismatch (manifest uses "xai", entry hints "xai-plugin")` diagnostic and pass it through `loadPluginManifestRegistryForInstalledIndex`; it remains in registry diagnostics.
- Is this the best way to solve the issue? unclear. The safest direction is stale-diagnostic invalidation/filtering, not changing manifest-id semantics in discovery.
