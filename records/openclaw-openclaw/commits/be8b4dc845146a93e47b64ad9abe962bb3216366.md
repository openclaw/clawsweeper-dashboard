# Commit be8b4dc

## Summary

Found one regression in the new bootstrap routing check: it treats ordinary leftover `BOOTSTRAP.md` files as hook-provided bootstrap content, bypassing the completed-workspace state.

## Findings

### Medium: Completed workspaces with a leftover `BOOTSTRAP.md` are forced back into bootstrap

- Kind: regression
- File: `src/agents/pi-embedded-runner/run/attempt-bootstrap-routing.ts`
- Line: 81
- Evidence: `resolveWorkspaceBootstrapStatus()` treats `setupCompletedAt` as authoritative and returns `"complete"` before checking whether `BOOTSTRAP.md` still exists (`src/agents/workspace.ts:377`). This commit passes all preloaded bootstrap files into routing (`src/agents/pi-embedded-runner/run/attempt.ts:954`), and `hasBootstrapFileContent()` then treats any non-empty `BOOTSTRAP.md` in that list as pending hook content (`src/agents/pi-embedded-runner/run/attempt-bootstrap-routing.ts:63`). That list comes from normal workspace loading as well as hooks, so an ordinary completed workspace with a residual `BOOTSTRAP.md` now gets `bootstrapMode: "full"` even though the workspace status is complete.
- Impact: Users with a completed workspace state but a stale or intentionally retained `BOOTSTRAP.md` can be pushed back into the first-run bootstrap prompt. The system prompt says bootstrap is pending and tells the agent to follow `BOOTSTRAP.md` before replying normally, which can block normal conversations or repeat onboarding.
- Suggested fix: Preserve the workspace completion state for filesystem `BOOTSTRAP.md`. Only OR hook-added or hook-overridden `BOOTSTRAP.md` content into `workspaceBootstrapPending`, or carry source metadata / compare against the unhooked baseline so ordinary loaded files do not override `isWorkspaceBootstrapPending()`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused smoke created a temporary workspace with `.openclaw/workspace-state.json` containing `setupCompletedAt` plus a leftover `BOOTSTRAP.md`; `resolveWorkspaceBootstrapStatus()` returned `"complete"`, but `resolveAttemptWorkspaceBootstrapRouting()` returned `{"bootstrapMode":"full","includeBootstrapInSystemContext":true,"includeBootstrapInRuntimeContext":false}`.
- Is this the best way to solve the issue? unclear. The fix direction should keep the intended hook-injected `BOOTSTRAP.md` behavior, but the current helper needs a way to distinguish hook output from ordinary workspace files.
