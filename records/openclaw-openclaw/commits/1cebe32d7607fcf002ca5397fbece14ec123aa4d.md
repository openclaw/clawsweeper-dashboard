# Commit 1cebe32d76

## Summary

Found one current compatibility regression in the registry hardening: the commit changed shard filenames from raw `containerName.json` to SHA-256 filenames, but current `main` still has no fallback or migration for raw-named shard files written by the immediately preceding sharded-registry implementation.

## Findings

### Medium: Existing raw-named sandbox shards become stranded after the hash filename switch

- Kind: regression
- File: `src/agents/sandbox/registry.ts`
- Line: 132
- Evidence: The parent implementation stored each shard at `path.join(dir, `${containerName}.json`)`; this commit switched all single-entry lookup/write/remove paths to `hashTextSha256(containerName).json`. Current `readRegistry()` still scans raw `.json` shards, but `readRegistryEntry()`, `updateRegistry()`, and `removeRegistryEntry()` only address the hashed path. A focused current-main check with an old-style `containers/container-a.json` showed: `readRegistryEntry("container-a")` returns `null`; `updateRegistry()` creates both `container-a.json` and the hashed shard; `removeRegistryEntry("container-a")` removes only the hashed shard, leaving `container-a.json`; `readRegistry()` still reports `container-a`.
- Impact: Users or dev environments that ran the raw-sharded registry before this commit can get duplicate/stale sandbox entries. `openclaw sandbox recreate` and prune paths can remove the runtime but fail to remove the old raw shard, so later `sandbox list`/recreate cycles keep seeing a deleted container. Docker hot paths can also miss existing registry metadata such as `createdAtMs`/`configHash` when they use `readRegistryEntry()`.
- Suggested fix: Add a backward-compatible raw-shard migration/fallback for safe legacy filenames. On lookup/update/remove, check a contained legacy raw path for simple pre-hash container names, verify the parsed `containerName` matches, write the hashed shard preserving existing fields, then remove the raw shard. Also de-duplicate `readShardedEntries()` by `containerName`, preferring the hashed/current shard.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Seed `SANDBOX_CONTAINERS_DIR/container-a.json` with a valid old-style entry, then call `readRegistryEntry("container-a")`, `updateRegistry(...)`, and `removeRegistryEntry("container-a")` on current `main`; the old raw shard remains visible to `readRegistry()`.
- Is this the best way to solve the issue? yes. A compatibility migration/fallback keeps the security hardening from hashed filenames while preserving metadata and cleanup semantics for already-written raw shards. A broad return to raw filenames would re-open the path-traversal hardening this commit was trying to close.
