# Commit 428cc541

## Summary

Found one provider-boundary regression in the new Anthropic API-key Haiku allowlist backfill. The intended direct-Anthropic default can also be added for Anthropic-on-Bedrock defaults because the commit reuses the broader cache-retention predicate.

## Findings

### Medium: Bedrock Claude defaults now allow direct Anthropic Haiku overrides

- Kind: regression
- File: `extensions/anthropic/config-defaults.ts`
- Line: 271
- Evidence: `hasAnthropicApiKeyModel` is computed with `isAnthropicCacheRetentionTarget(...)`, but that helper intentionally returns true for both direct `anthropic/*` refs and `amazon-bedrock/*anthropic.claude*` refs. The new loop then adds `anthropic/claude-haiku-4-5` to `agents.defaults.models` whenever that broader predicate matches. A focused probe confirms an API-key config whose primary model is `amazon-bedrock/us.anthropic.claude-opus-4-6-v1` is materialized with both the Bedrock model and `anthropic/claude-haiku-4-5`, and `resolveAllowedModelRef` then accepts a cron/model override to direct Anthropic Haiku.
- Impact: Users who intentionally route Claude through Bedrock can have their model allowlist widened to the direct Anthropic provider just because an Anthropic API key/profile is present. That weakens the provider boundary enforced by `agents.defaults.models` and can redirect explicit cron/session overrides away from Bedrock’s billing, IAM, logging, or data-residency controls.
- Suggested fix: Keep the cache-retention predicate broad, but use a narrower predicate for this allowlist backfill, for example only seed `ANTHROPIC_API_KEY_DEFAULT_ALLOWLIST_REFS` when an existing/default model ref parses to `provider === "anthropic"`. Add a regression test showing a Bedrock Claude primary still gets `cacheRetention: "short"` without adding `anthropic/claude-haiku-4-5`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: `pnpm exec tsx -e ... applyConfigDefaults(...)` with an Anthropic API-key profile and Bedrock Claude primary produced a direct `anthropic/claude-haiku-4-5` allowlist entry; a second probe showed `resolveAllowedModelRef` accepts that direct Anthropic model under the materialized config.
- Is this the best way to solve the issue? yes: the fix should split “Claude cache-retention target” from “direct Anthropic API-key allowlist seed target” so Bedrock behavior remains unchanged while direct Anthropic configs still get the intended Haiku backfill.
