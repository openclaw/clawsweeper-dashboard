# Commit 3af81481

## Summary

Found one concrete retry-path bug. The new Gemini first-response retry handles the case where `fetch()` itself stalls, but can miss the case where Google returns SSE headers and then stalls before the first event.

## Findings

### Medium: Header-only SSE stalls can be treated as an empty successful response instead of triggering the retry

- Kind: reliability
- File: `extensions/google/transport-stream.ts`
- Line: 826
- Evidence: `openGoogleSseAttempt()` waits for `iterator.next()` to get the first SSE chunk. The retry deadline aborts the child signal, and `parseGoogleSseChunks()` responds by cancelling the reader at `extensions/google/transport-stream.ts:956`, but after `reader.read()` resolves it breaks on `done` at `extensions/google/transport-stream.ts:966` without rechecking `signal.aborted`. `openGoogleSseAttempt()` then clears the deadline and returns `type: "ready"` when `first.done` is true, so `openGoogleSseChunks()` never runs the lean retry.
- Impact: if the Google endpoint sends HTTP/SSE headers but no first `data:` event before the deadline, OpenClaw can finalize an empty assistant message as a successful `stop` instead of retrying or failing over. That is directly in the surface this commit is intended to fix.
- Suggested fix: after `await iterator.next()`, check `attemptSignal?.timedOut()` before treating `first.done` as ready, or make `parseGoogleSseChunks()` recheck `signal.aborted` immediately after `reader.read()` returns and before the `done` break. Add a regression test where `guardedFetchMock` resolves to a `Response` whose body never enqueues a first chunk.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the stream primitive: a focused Node Web Streams check showed that cancelling a pending `reader.read()` resolves `{ done: true }`, which matches the code path above. I did not run a live Google call that proves Google flushes headers before a stalled first SSE event.
- Is this the best way to solve the issue? Yes: keep the provider-owned retry, but classify child-deadline cancellation during first-chunk wait as `timeout` instead of an empty ready stream.
