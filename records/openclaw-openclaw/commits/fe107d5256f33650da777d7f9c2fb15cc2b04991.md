# Commit fe107d5

## Summary

Found one setup-status false positive in the touched Google Meet Twilio readiness path.

## Findings

### Medium: Config-only `voice-call` entries are still reported as enabled

- Kind: bug
- File: `extensions/google-meet/src/setup.ts`
- Line: 229
- Evidence: The new check only requires `Object.hasOwn(pluginEntries, "voice-call")` and `voiceCallEntry.enabled !== false`, so a config-only entry like `plugins.entries.voice-call.config.provider = "twilio"` is reported as enabled. Plugin activation for bundled plugins requires `entry.enabled === true`; otherwise a bundled plugin without `enabledByDefault` falls through to disabled. `extensions/voice-call/openclaw.plugin.json` does not declare `enabledByDefault`, and the Google Meet docs say `twilio-voice-call-plugin` should fail when `voice-call` is not enabled.
- Impact: `googlemeet setup --transport twilio` can report a green `twilio-voice-call-plugin` check even though the Voice Call plugin will not load. Agents/scripts may proceed past setup and then fail at join time when `voicecall.start` is unavailable.
- Suggested fix: Change the readiness check to require `voiceCallEntry.enabled === true`, matching bundled plugin activation and the docs. Add a regression test for a present `voice-call` entry with config but no `enabled: true`.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes: configure `plugins.allow: ["google-meet", "voice-call"]`, add `plugins.entries.voice-call.config` with Twilio provider/public URL but omit `enabled: true`, then run `setup_status` for Twilio with dial-in and credentials. The source path reports the plugin check as ready even though activation rules do not enable the bundled plugin.
- Is this the best way to solve the issue? Yes; setup status should mirror plugin activation for this explicit dependency. Requiring `enabled === true` is narrower than trying to infer activation from config presence.
