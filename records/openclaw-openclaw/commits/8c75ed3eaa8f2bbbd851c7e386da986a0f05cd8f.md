# Commit 8c75ed3

## Summary

Found one low-severity concurrency/data-quality issue in the new Codex native subagent task mirror.

## Findings

### Low: Unrelated Codex projectors can overwrite mirrored subagent terminal summaries

- Kind: concurrency
- File: `extensions/codex/src/app-server/native-subagent-task-mirror.ts`
- Line: 122
- Evidence: `CodexAppServerClient` broadcasts every server notification to every registered handler (`extensions/codex/src/app-server/client.ts:436`), and app-server attempts use the shared client by default (`extensions/codex/src/app-server/client-factory.ts:16`). The new projector intentionally calls `nativeSubagentTaskMirror.handleNotification()` before current-turn filtering (`extensions/codex/src/app-server/event-projector.ts:159`). For `thread/status/changed`, the mirror calls `applyStatus()` for any `threadId` without checking that this projector previously mirrored that child thread (`extensions/codex/src/app-server/native-subagent-task-mirror.ts:122`). Since `finalizeTaskRunByRunId()` scopes only by `runId` and `runtime`, another active projector on the same shared client can process a child thread’s later `idle` event and overwrite a completed task’s useful `terminalSummary` with the generic `"Codex native subagent finished."`
- Impact: Concurrent or overlapping Codex app-server runs can make Task Registry rows for native subagents lose their real completion message. Status is unlikely to reopen, but user-visible task/status details can be degraded by an unrelated session handler.
- Suggested fix: Track child thread ownership before applying `thread/status/changed`; only call `applyStatus()` when `mirroredThreadIds.has(threadId)` or after a verified parent/thread source match. Keep the collab spawn and `thread/started` paths as the places that register a child thread for this projector.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes in a focused unit-style scenario: attach two `CodexAppServerEventProjector` instances to the same shared notification stream, let one mirror and complete a child task from a `collabAgentToolCall` with a specific message, then deliver `thread/status/changed` idle for that child while the second projector is active. The second mirror has no `terminalRunIds` entry and can finalize the same run id with the generic summary.
- Is this the best way to solve the issue? Yes. The mirror already tracks `mirroredThreadIds`; using it to gate status updates keeps cross-projector notifications from mutating rows they do not own while preserving the intended pre-filter handling for child-thread events.
