# Commit 62375ae

## Summary

Found one low-severity regression in the lint rewrite: the model-entry copy changed from object spread to `Object.assign({}, ...)`, which lets a config-derived `__proto__` property mutate the returned model object's prototype during the Ollama doctor migration.

## Findings

### Low: Ollama doctor migration can return a prototype-poisoned model object

- Kind: security
- File: `src/commands/doctor/shared/legacy-config-core-normalizers.ts`
- Line: 1283
- Evidence: The commit replaced the model copy with `Object.assign({}, model, { params: ... })`. For a JSON-parsed model entry containing an own `__proto__` key, `Object.assign` invokes the inherited `__proto__` setter on the fresh `{}` target. A focused probe against current `main` showed `Object.hasOwn(migrated, "__proto__") === false`, `Object.getPrototypeOf(migrated).polluted === true`, and `migrated.polluted === true` after `normalizeCompatibilityConfigValues` migrated a native Ollama model with `contextWindow`.
- Impact: This does not pollute `Object.prototype`, but it does create a normalized in-memory config model whose prototype is controlled by config input. Doctor continues to pass that normalized config through later repair/warning/write logic, so inherited properties can be observed by code paths that do not require own-property checks. The previous object-spread form preserved `__proto__` as an own data property instead of changing the model object's prototype.
- Suggested fix: Avoid `Object.assign({}, model, ...)` for config-derived objects here. Revert this copy to object spread with an intentional lint suppression if needed, or copy through a helper that filters blocked object keys such as `__proto__`, `constructor`, and `prototype`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Run a focused `tsx` probe importing `normalizeCompatibilityConfigValues`, pass an Ollama model parsed from JSON with `{"__proto__":{"polluted":true},"contextWindow":32768}`, and inspect the migrated model prototype after normalization.
- Is this the best way to solve the issue? yes. The regression is caused by the copy primitive, not the `num_ctx` migration logic. Keeping a data-property copy or filtering blocked prototype keys preserves the lint goal without introducing prototype-chain state.
