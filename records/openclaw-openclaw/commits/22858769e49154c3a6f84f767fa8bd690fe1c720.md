# Commit 22858769e4

## Summary

The commit adds a TUI consumer for lifecycle `fallback_step` events, but the normal TUI chat-send path still does not emit those events during model fallback. The footer update therefore remains dead for the reported local/gateway fallback flow.

## Findings

### Medium: Fallback model display still has no TUI-facing producer

- Kind: bug
- File: `src/tui/tui-event-handlers.ts`
- Line: 506
- Evidence: `handleAgentEvent` now updates `state.sessionInfo.modelProvider` / `model` only after a run-scoped lifecycle event with `phase: "fallback_step"` reaches the TUI. Current `src/agents/agent-command.ts:1061` still handles `onFallbackStep` by recording only `model.fallback_step` to trajectory data, and `src/tui/embedded-backend.ts:525` only forwards already-emitted agent events. The embedded TUI `agentCommandFromIngress` call at `src/tui/embedded-backend.ts:591` does not pass any fallback/model-switch callback.
- Impact: During a real TUI fallback chain, the footer can continue showing the stale persisted/default model even though this commit closes the linked issue as fixed. The new tests prove the consumer works for synthetic events, not that the live chat path produces them.
- Suggested fix: Emit a run/session-scoped lifecycle `fallback_step` agent event from the `agentCommandFromIngress` fallback callback, alongside the trajectory recording, and add focused producer/embedded-backend coverage proving the event reaches `handleAgentEvent`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes, source-level: configure a fallback chain, force the primary to fail, and send via TUI. The fallback callback has the target model data but records it only to trajectory, so the TUI handler never receives the event it now requires.
- Is this the best way to solve the issue? yes: the consumer-side parser is reasonable, but the fix needs the matching live producer in the chat-send/embedded TUI path rather than only synthetic handler tests.
