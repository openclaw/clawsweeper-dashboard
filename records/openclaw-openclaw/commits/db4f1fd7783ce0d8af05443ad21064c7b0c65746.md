# Commit db4f1fd

## Summary

Found one current regression in the untargeted cron wake path: the queued heartbeat fanout behavior is preserved, but the synchronous `runHeartbeatOnce` path can lose default-agent heartbeat overrides before the runner gets a chance to merge them.

## Findings

### Medium: Untargeted cron wake-now can override default-agent heartbeat settings with global defaults

- Kind: regression
- File: `src/gateway/server-cron.ts`
- Line: 313
- Evidence: Current `resolveCronTarget(..., preserveUntargeted: true)` returns `agentId: undefined` when no target is supplied at lines 211-212. `runHeartbeatOnce` then calls `resolveCronHeartbeatOverride` with that undefined `agentId` at lines 313-325. Inside `resolveCronHeartbeatOverride`, an undefined agent skips `findAgentEntry`, so the override is built from `agents.defaults.heartbeat` only at lines 243-254. The downstream runner does resolve the default agent, but it merges the already-expanded requested heartbeat over that agent config, so default fields can overwrite the default agent’s heartbeat overrides.
- Impact: A `wakeMode: "now"` main cron job with no explicit `agentId` or `sessionKey` can run the default agent heartbeat with the global default prompt/model/deliveryFormat/directPolicy instead of the default agent’s override. Example: if `agents.defaults.heartbeat.prompt = "generic"` and the default agent has `heartbeat.prompt = "ops-specific"`, the cron wake-now path passes `"generic"` as the requested heartbeat and masks `"ops-specific"`.
- Suggested fix: Keep untargeted fanout for queued `requestHeartbeat`, but resolve the default agent for `runHeartbeatOnce` heartbeat override calculation, or make `resolveCronHeartbeatOverride` return only the caller override when `agentId` is undefined so `runHeartbeatOnce` can merge it against the runner-resolved agent config.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Add a gateway cron unit case with a default agent that has an agent-level heartbeat prompt/model override plus conflicting `agents.defaults.heartbeat`, then run a main `wakeMode: "now"` cron job with no explicit target and assert the mocked `runHeartbeatOnce` options preserve the agent override.
- Is this the best way to solve the issue? yes. The fix should be local to the synchronous wake target/override path; broadening queued heartbeat targeting would undo the fanout behavior this commit intentionally preserves.
