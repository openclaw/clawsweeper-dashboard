# Commit 7544bee

## Summary

Found one compatibility regression in the new explicit embedded dispatcher timeout path: it can replace existing custom Undici dispatchers on normal no-proxy embedded runs.

## Findings

### Medium: Embedded timeout setup overwrites custom Undici dispatchers

- Kind: compatibility
- File: `src/infra/net/undici-global-dispatcher.ts`
- Line: 36
- Evidence: `resolveDispatcherKind()` treats any constructor name containing `Agent` as a tunable bare agent, and `applyGlobalDispatcherStreamTimeouts()` then replaces it with `new runtime.Agent(...)`. The changed embedded runtime now calls `ensureGlobalUndiciDispatcherStreamTimeouts()` unconditionally after proxy bootstrap, so a process that already installed Undici `MockAgent`, `RetryAgent`, `SnapshotAgent`, or a custom `*Agent` dispatcher loses that dispatcher on the first embedded attempt. Focused smoke on current `main`: `before=MockAgent`, then after `configureEmbeddedAttemptHttpRuntime({ timeoutMs: 1_900_000 })`, `after=Agent`.
- Impact: Existing process-level dispatcher behavior can be bypassed for embedded runs. That can break tests/offline harnesses using `MockAgent.disableNetConnect()`, operator-installed retry/instrumentation dispatchers, or custom dispatcher policy. This is the same process-global fetch surface this PR is trying to avoid disturbing.
- Suggested fix: Classify only the default Undici `Agent` and OpenClaw-managed `EnvHttpProxyAgent` as replaceable, for example by exact constructor name or constructor identity where safe. Treat `MockAgent`, `RetryAgent`, `SnapshotAgent`, subclasses, and unknown custom dispatchers as unsupported while still updating `_globalUndiciStreamTimeoutMs`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Run a no-proxy smoke that sets `setGlobalDispatcher(new MockAgent())`, calls `configureEmbeddedAttemptHttpRuntime({ timeoutMs: 1_900_000 })`, and checks `getGlobalDispatcher().constructor.name`; current `main` changes it from `MockAgent` to `Agent`.
- Is this the best way to solve the issue? yes, directionally. The existing code already avoids replacing unsupported proxy dispatchers; the same boundary should apply to non-default dispatcher implementations whose options cannot be safely preserved.
