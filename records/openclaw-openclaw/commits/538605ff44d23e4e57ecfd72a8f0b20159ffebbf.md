# Commit 538605f

## Summary

Found one high-confidence filesystem safety regression. OpenClaw now disables the `@openclaw/fs-safe` Python helper by default, which makes `root.write()` use the package's Node fallback. That fallback can follow a parent symlink after validation and overwrite a file outside the root before post-write verification rejects.

## Findings

### High: Default fs-safe write fallback can clobber files outside the root during a symlink-retarget race

- Kind: security
- File: `src/infra/fs-safe-defaults.ts`
- Line: 7
- Evidence: `src/infra/fs-safe-defaults.ts:7` configures `configureFsSafePython({ mode: "off" })` unless an env override is set. `src/infra/fs-safe.ts:75` routes `writeFileWithinRoot()` through `fsSafeRoot(...).write(...)`. In `@openclaw/fs-safe@0.1.0`, that default reaches the Node fallback path, which builds the parent path lexically from `rootPath + relativeParentPath`, writes `.<basename>.fallback.tmp`, renames it, and only then performs post-write verification. A focused live probe with a root-internal symlink retargeted to an outside directory during the fallback temp write produced: `outcome: rejected:outside-workspace`, `flipped: true`, `inside: inside`, `outside: new-content`. With `OPENCLAW_FS_SAFE_PYTHON_MODE=require`, the same probe rejected before the fallback write and left the outside file unchanged.
- Impact: Any caller relying on `root.write()` or `writeFileWithinRoot()` as a root boundary can report failure while still overwriting an attacker-controlled outside path. This breaks the intended safety contract and can cause data loss or boundary escape when a mutable symlink exists under an otherwise allowed root.
- Suggested fix: Do not silently use this Node fallback for POSIX root mutations. Either keep the fd-relative helper required for write/copy-style mutation primitives, or change the fallback to hold and operate on parent directories fd-relative with no-follow semantics before writing or renaming. The post-write verification is useful, but it is too late to prevent the clobber.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The temp-directory probe above deterministically retargets `root/slot` from `root/inside` to an outside directory just before the fallback temp write; the outside `target.txt` becomes `new-content` even though the API rejects.
- Is this the best way to solve the issue? yes for the fix direction: the mutation primitive must not degrade to a path-string fallback after a symlink validation step. Exact implementation can live upstream in `@openclaw/fs-safe`, but OpenClaw should not default to the unsafe fallback.
