# Commit 0ecda68

## Summary

Found one low-severity sanitizer edge case that still exists on current `main`. The normal legacy `[TOOL_CALL]...[/TOOL_CALL]` path is covered and passes, but a close delimiter inside a quoted legacy payload terminates stripping early and leaves raw payload tail text visible.

## Findings

### Low: Legacy TOOL_CALL sanitizer closes on delimiters inside quoted args

- Kind: bug
- File: `src/shared/text/assistant-visible-text.ts`
- Line: 397
- Evidence: `stripLegacyBracketToolCallBlocks` searches for the first `[/TOOL_CALL]` with a regex and does not account for quoted payload content. A focused live check on current `main`:
  ```text
  normalizeReplyPayload({
    text: 'Before\n[TOOL_CALL]{tool => "web_search", args => {"query":"what is [/TOOL_CALL]", "note":"after marker"}}[/TOOL_CALL]\nAfter'
  })?.text
  ```
  returned:
  ```text
  Before
  ", "note":"after marker"}}[/TOOL_CALL]
  After
  ```
- Impact: A WhatsApp/user-facing reply can still leak malformed legacy tool-call syntax if the model includes the closing marker inside an argument string, which is plausible when the user asks about or includes `[/TOOL_CALL]`. This undercuts the commit’s intended delivery sanitizer for that edge.
- Suggested fix: Make the legacy bracket stripper quote/balanced-payload aware, similar to the existing XML tool-call stripper’s quote-balance handling, or parse the `{tool => ..., args => ...}` payload enough to ignore bracket delimiters inside quoted strings before selecting the closing tag.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The `pnpm exec tsx` check above reproduces on current `main`; the normal case strips correctly, while a quoted `[/TOOL_CALL]` leaves the payload tail.
- Is this the best way to solve the issue? unclear. The safest fix direction is to make close-delimiter detection quote-aware. A simpler “use the last close tag” approach may break multiple legacy blocks in one reply.
