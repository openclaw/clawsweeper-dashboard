# Commit d448415

## Summary

Found one regression in the new config-backed SecretRef handling: provider catalog auth now bypasses env SecretRef provider allowlists and can expose an allowlist-excluded env value as `discoveryApiKey`.

## Findings

### Medium: Catalog auth resolves env SecretRefs without enforcing the secret provider allowlist

- Kind: security
- File: `src/agents/models-config.providers.secrets.ts`
- Line: 217
- Evidence: The new SecretRef branch reads `params.env?.[envVar]` directly for `configuredApiKeyRef.source === "env"` and returns it as `discoveryApiKey`. Existing runtime auth explicitly gates this path through `canResolveEnvSecretRefInReadOnlyPath`, which rejects refs whose `secrets.providers.<provider>.allowlist` does not include the env id (`src/agents/model-auth.ts:110`, covered by `src/agents/model-auth.test.ts:566`). The general secret resolver enforces the same allowlist (`src/secrets/resolve.ts:332`).
- Impact: A config such as `models.providers.custom.apiKey = { source: "env", provider: "custom-env", id: "MY_CUSTOM_KEY" }` with `secrets.providers.custom-env.allowlist = ["OPENAI_API_KEY"]` should not resolve `MY_CUSTOM_KEY`. After this commit, `createProviderApiKeyResolver` and `createProviderAuthResolver` return `discoveryApiKey: "sk-should-not-resolve"` for that excluded ref, so provider discovery/catalog code can receive a secret that the configured SecretRef provider policy denies.
- Suggested fix: Reuse the same allowlist/source check used by `resolveUsableCustomProviderApiKey`, or route env SecretRef resolution through the existing SecretRef resolver/policy before returning a discovery key. If the ref provider is not the default env provider or the id is excluded, return `undefined` rather than reading `params.env`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused `tsx --eval` check against `createProviderApiKeyResolver` with `MY_CUSTOM_KEY` set and an allowlist excluding it returned `{"apiKey":"MY_CUSTOM_KEY","discoveryApiKey":"sk-should-not-resolve"}`.
- Is this the best way to solve the issue? yes. The rest of the code already treats env SecretRef allowlists as a read boundary, so the catalog resolver should use the same predicate/contract.
