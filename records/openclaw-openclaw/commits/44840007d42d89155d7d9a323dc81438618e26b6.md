# Commit 4484000

## Summary

Found one concrete SSRF-policy regression in the new `allowedOrigins` path. The normal pinned-DNS path blocks metadata targets, but trusted proxy modes skip target DNS and now let a matching `allowedOrigins` entry suppress the remaining literal-host block before dispatch.

## Findings

### Medium: `allowedOrigins` can bypass metadata blocking in trusted proxy modes

- Kind: security
- File: `src/infra/net/fetch-guard.ts`
- Line: 421
- Evidence: `resolveSsrFPolicyForUrl()` promotes a matching `allowedOrigins` entry into `allowedHostnames`, and `resolveHostnamePolicyChecks()` then skips `assertAllowedHostOrIpOrThrow()` when the hostname is allowed. In the trusted env-proxy and trusted explicit-proxy branches, `fetchWithSsrFGuard()` only calls `assertHostnameAllowedWithPolicy()` before dispatching, so the resolver-side metadata/link-local check never runs for those modes.
- Impact: a caller using the SDK-exposed SSRF helpers can make `http://169.254.169.254/...` reach the fetch implementation when `HTTP_PROXY` is configured and `mode: "trusted_env_proxy"` is active. The same control request without `allowedOrigins` is blocked before dispatch. This contradicts the commit’s intended contract that plugin-supplied `allowedOrigins` metadata entries are rejected at the guard boundary.
- Suggested fix: keep exact-origin private-host trust, but make metadata/link-local/cloud-metadata host rejection independent of `allowedHostnames` in no-target-DNS modes. Add regression tests for `TRUSTED_ENV_PROXY` and `TRUSTED_EXPLICIT_PROXY` using `allowedOrigins` for metadata literals and metadata hostnames.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused `node --import tsx` probe on current `main` set `HTTP_PROXY`, called `fetchWithSsrFGuard({ url: "http://169.254.169.254/latest/meta-data/", mode: TRUSTED_ENV_PROXY, policy: { allowedOrigins: ["http://169.254.169.254"] } })`, and observed `FETCH_CALLED ... RESULT allowed`. The same URL without `allowedOrigins` logged the SSRF block and threw `SsrFBlockedError`.
- Is this the best way to solve the issue? yes, directionally. The bug is in the no-DNS proxy branches trusting the promoted hostname too early; the provider-specific filter is not enough because the SDK exports the policy and fetch helpers.
