## Findings

### Low: `--rerun-failed-clawhub` can be skipped by earlier registry verification

- Kind: reliability
- File: `scripts/lib/release-beta-verifier.ts`
- Line: 413
- Evidence: `verifyBetaRelease` verifies every expected ClawHub package/version first at `scripts/lib/release-beta-verifier.ts:413`. The only path that actually performs `gh run rerun --failed` is inside `verifyWorkflowRun` at `scripts/lib/release-beta-verifier.ts:332`, but the Plugin ClawHub workflow run is not checked until `scripts/lib/release-beta-verifier.ts:435`, after all ClawHub registry checks have already succeeded. The docs added for this commit tell operators to use `--rerun-failed-clawhub` when the ClawHub sidecar failed in retryable jobs at `docs/reference/RELEASING.md:101`.
- Impact: if a retryable ClawHub job fails before publishing one plugin, or before that version becomes visible, `pnpm release:verify-beta -- ... --rerun-failed-clawhub` fails on the missing ClawHub version/artifact and never reaches the rerun logic. That leaves the documented repair flag ineffective for a common failed-publish recovery path.
- Suggested fix: when `--rerun-failed-clawhub` and a ClawHub run id are provided, check/rerun that workflow before ClawHub registry verification, or catch ClawHub registry failures and run the failed-job rerun path before exiting with the “rerun verifier after it finishes” message.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. The static call order is enough: a missing ClawHub package throws before the verifier reaches the rerun-capable workflow check.
- Is this the best way to solve the issue? Yes. The rerun decision should happen before assertions that depend on the failed workflow having completed successfully.
