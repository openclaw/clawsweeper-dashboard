# Commit c614b59

## Summary

Found one UI regression in the delayed run-status publishing path: failed or timed-out session rows can be rewritten to `killed` after the new deferred status publish runs.

## Findings

### Low: Deferred run-status publish collapses failed/timeout session status to killed

- Kind: regression
- File: `ui/src/ui/app-gateway.ts`
- Line: 795
- Evidence: `applySessionsChangedEvent` now suppresses immediate status publishing and returns only `phase: "done" | "interrupted"` in `clearedChatRunStatus` at `ui/src/ui/controllers/sessions.ts:469`. When `flushChatQueueAfterSessionRunReconcile` later publishes that status, it calls `reconcileChatRunLifecycle` with only `outcome` and no `sessionStatus`. `reconcileSessionRows` defaults every interrupted outcome to `killed` at `ui/src/ui/chat/run-lifecycle.ts:131`, so a row that was just reconciled as `failed` or `timeout` is rewritten to `killed`.
- Impact: WebChat/session UI can misreport failed or timed-out runs as killed/aborted after session-driven run completion, hiding the actual terminal reason from the user.
- Suggested fix: Carry the original `SessionRunStatus` through `clearedChatRunStatus` and pass it as `sessionStatus`, or publish the toast without re-running session-row reconciliation.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A read-only `node --import tsx --input-type=module` probe showed `afterApply: "failed"` and `afterPublish: "killed"` for the same row.
- Is this the best way to solve the issue? yes. The new delayed publish path needs to preserve the more specific session terminal status instead of reconstructing it from the coarse UI toast phase.
