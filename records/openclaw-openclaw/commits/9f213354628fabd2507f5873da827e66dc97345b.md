## Summary

Found one low-severity diagnostic bug in the new Full Release Validation queue summary. The runner-size changes and workflow syntax checks look otherwise sound.

## Findings

### Low: Queue summary counts dependency wait as runner queue time

- Kind: bug
- File: `.github/workflows/full-release-validation.yml`
- Line: 713
- Evidence: `summarize_child_timing` reads the workflow run `createdAt` and computes each job’s `queueMin` as `job.startedAt - run.createdAt`. That is not runner queue time for jobs blocked behind `needs`; it includes all upstream DAG wait. A live GitHub API spot check showed the failure mode: the new formula reported `checks-node-core` as `2.7` queue minutes on run `25121753316`, while `gh api repos/openclaw/openclaw/actions/runs/25121753316/jobs` showed that job’s own `created_at` and `started_at` were both `2026-04-29T16:46:34Z`.
- Impact: Full Release Validation evidence can misidentify downstream jobs as the longest runner queues, which undermines the purpose of diagnosing release runner pressure and can send maintainers toward the wrong runner-class changes.
- Suggested fix: compute actual queue time from per-job `created_at` to `started_at` via the Actions jobs API, or rename the table to something like “minutes before job start” and avoid using it as queue-pressure evidence.
- Confidence: high
