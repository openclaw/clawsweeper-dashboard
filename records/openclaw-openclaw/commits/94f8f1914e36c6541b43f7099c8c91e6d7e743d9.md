# Commit 94f8f191

## Summary

Found one actionable release-check reliability bug: the widened Windows timeout matcher now accepts versioned tarball names, but still does not match the actual timeout command produced by this script because the real args include `--no-restart`.

## Findings

### Medium: Windows packaged-upgrade timeout fallback still misses the actual command

- Kind: reliability
- File: `scripts/openclaw-cross-os-release-checks.ts`
- Line: 1400
- Evidence: `buildPackagedUpgradeUpdateArgs()` includes `--no-restart` before `--timeout` at `scripts/openclaw-cross-os-release-checks.ts:1367`, and `runOpenClaw()` passes those args into the command that `runCommand()` reports on timeout. The updated matcher at `scripts/openclaw-cross-os-release-checks.ts:1400` expects `--yes --json --timeout`, with no `--no-restart`. The new test fixture in `test/scripts/openclaw-cross-os-release-checks.test.ts:750` also omits `--no-restart`, so it passes without exercising the real command shape.
- Impact: On Windows, a packaged-upgrade timeout with a versioned tarball can still be rethrown at `scripts/openclaw-cross-os-release-checks.ts:782` instead of taking the direct candidate install fallback at `scripts/openclaw-cross-os-release-checks.ts:799`. That keeps the intended recoverable timeout path dead for the actual script-generated command and can fail the cross-OS release check.
- Suggested fix: Update the matcher and test fixture to include the real `--no-restart` argument, preferably by constructing the test timeout string from `buildPackagedUpgradeUpdateArgs()` so future arg changes cannot drift again.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused import check built the timeout message from `buildPackagedUpgradeUpdateArgs("http://127.0.0.1:49951/openclaw-2026.5.4-beta.1.tgz")`; `isRecoverableWindowsPackagedUpgradeTimeoutError(..., "win32")` returned `false`.
- Is this the best way to solve the issue? yes. The fallback is intentionally narrow to local candidate tarball update timeouts, so aligning the regex and test with the actual generated args is safer than broadly loosening command matching.
