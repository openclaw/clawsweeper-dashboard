# Commit 6283c8247c

## Summary

Found one actionable regression in the YAML-kind removal: unsupported `.yaml` files now fall through to the Markdown parser and can be rewritten as Markdown by `openclaw path set`.

## Findings

### Medium: Unsupported YAML files can be silently rewritten as Markdown

- Kind: data_loss
- File: `extensions/oc-path/src/cli.ts`
- Line: 162
- Evidence: `inferKind()` no longer maps `.yaml`, `.yml`, or `.lobster`, but `loadAst()` only handles `jsonc`/`jsonl` and then defaults every other extension to `parseMd(raw).ast`. In the base commit, those extensions mapped to `yaml`.
- Impact: A user or script that still targets a YAML workflow/config gets a successful write instead of an unsupported-kind error. Focused smoke on current `main`: `pathSetCommand("oc://workflow.yaml/+", "Appended Section", ...)` exited `0` and rewrote `workflow.yaml` from YAML into `name: old\nsteps:\n  - id: build\n\n\n## Appended Section`.
- Suggested fix: Make the CLI reject `inferKind(fileName) === null` with a structured unsupported-kind error before parsing/writing. If Markdown fallback is intentional for some legacy extensionless files, gate it explicitly rather than applying it to `.yaml/.yml/.lobster`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: create a `.yaml` file and call `pathSetCommand("oc://workflow.yaml/+", "Appended Section", { cwd, json: true }, rt)`. It succeeds and writes Markdown into the YAML file.
- Is this the best way to solve the issue? yes: after removing YAML support, refusing unsupported file kinds at the CLI load boundary is safer than treating unknown extensions as Markdown.
