# Commit 9ff4d36

## Summary

Found one test-runner false-success path in the new explicit-target validation.

## Findings

### Medium: Source targets can still pass after resolving to a missing mapped test file

- Kind: reliability
- File: `scripts/test-projects.test-support.mjs`
- Line: 839
- Evidence: `findUnmatchedExplicitTestTargets()` validates non-test targets using the broad inferred `toScopedIncludePattern(targetArg, cwd)` instead of the same resolved include patterns that the runner will execute. Current `main` has `test/vitest/vitest.commands-light-paths.mjs:9` mapping `src/commands/doctor-browser.ts` to missing `src/commands/doctor-browser.test.ts`, while a nearby real test is `src/commands/doctor-browser.facade.test.ts`. The validator returns `[]` for `src/commands/doctor-browser.ts`, then `node scripts/test-projects.mjs src/commands/doctor-browser.ts --reporter=dot` exits 0 with Vitest reporting “No test files found” and `include: doctor-browser.test.ts`.
- Impact: An explicit source-file target can still report a passing test run while executing zero tests, which is the false-success class this commit is meant to close. This can hide stale light-lane mappings or renamed tests from local and CI targeted runs.
- Suggested fix: Validate the final planned include patterns, or have `findUnmatchedExplicitTestTargets()` use `resolveLightLaneIncludePatterns(kind, targetArg, cwd)` before falling back to `toScopedIncludePattern()`. Also correct or remove the stale `doctor-browser` light-lane mapping.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: `node scripts/test-projects.mjs src/commands/doctor-browser.ts --reporter=dot` exits successfully with zero tests and `include: doctor-browser.test.ts`.
- Is this the best way to solve the issue? yes for the validator direction: checking the exact include patterns that will be run avoids drift between validation and routing. The stale mapping should also be fixed.
