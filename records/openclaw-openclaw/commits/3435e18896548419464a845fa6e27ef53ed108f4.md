# Commit 3435e18896

## Summary

Found one low-severity test regression. Runtime cleanup still appears correct on current `main`, but the updated assertion no longer proves that the stale symlink itself was removed.

## Findings

### Low: Symlink cleanup assertion now follows the symlink and can miss a leftover dangling link

- Kind: test_gap
- File: `src/commands/doctor/shared/plugin-dependency-cleanup.test.ts`
- Line: 132
- Evidence: `expectPathMissing` uses `fs.stat()` at line 8, and the symlink cleanup test now calls it for `slackLink`. `fs.stat()` follows symlinks, so after `cleanupLegacyPluginDependencyState` removes `legacyRuntimeRoot`, a leftover `slackLink` would also report `ENOENT`. A focused Node check confirmed this behavior: for a dangling symlink, `stat` returns `ENOENT` while `lstat` still succeeds and reports `isSymbolicLink: true`.
- Impact: The test named “removes dangling global plugin-runtime symlinks” can pass while a stale global `node_modules` symlink remains on disk. That is the exact failure this test is meant to guard against, and such stale plugin-runtime symlinks are repaired because they can break bundled plugin imports.
- Suggested fix: Keep the shared helper for ordinary file/directory paths, but use `fs.lstat(slackLink)` for symlink entry absence, e.g. `await expect(fs.lstat(slackLink)).rejects.toMatchObject({ code: "ENOENT" });`, or add a separate no-follow helper.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Create a directory, symlink to it, delete the target, then compare `fs.stat(link)` and `fs.lstat(link)`: `stat` reports `ENOENT`, while `lstat` still sees the symlink.
- Is this the best way to solve the issue? yes. The production contract here is removal of the symlink directory entry, so the assertion must use `lstat` rather than a target-following stat.
