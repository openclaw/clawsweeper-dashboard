# Commit 4df34cb

## Summary

Found one release-version metadata regression: the `llm-task` plugin package was bumped to `2026.5.22`, but its explicit plugin manifest version stayed on `2026.5.21`. Current `main` still has the mismatch.

## Findings

### Low: `llm-task` manifest still reports the previous release

- Kind: regression
- File: `extensions/llm-task/openclaw.plugin.json`
- Line: 36
- Evidence: `extensions/llm-task/package.json:3` is now `2026.5.22`, but `extensions/llm-task/openclaw.plugin.json:36` remains `2026.5.21`. This is the only extension manifest with an explicit `version` field. `src/plugins/manifest-registry.ts:503` prefers `manifest.version` over the package version, and `src/plugins/loader-records.ts:36` carries that into the plugin record, so plugin discovery/status metadata can report `llm-task` as `2026.5.21` in a `2026.5.22` release.
- Impact: user-visible and registry-facing plugin metadata is stale for a bundled startup plugin, which undermines release/version alignment and can confuse diagnostics or support checks that rely on plugin record versions.
- Suggested fix: update `extensions/llm-task/openclaw.plugin.json` to `2026.5.22`, and extend `scripts/sync-plugin-versions.ts` plus its test to sync explicit `openclaw.plugin.json#version` values when present.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Read `extensions/llm-task/package.json:3` and `extensions/llm-task/openclaw.plugin.json:36`; then trace `src/plugins/manifest-registry.ts:503`, which selects the stale manifest version before package version fallback.
- Is this the best way to solve the issue? yes. Since manifest `version` is optional and only one plugin declares it, either keep it aligned when present or remove it. If keeping it, the release sync script should own it so the next bump does not repeat the drift.
