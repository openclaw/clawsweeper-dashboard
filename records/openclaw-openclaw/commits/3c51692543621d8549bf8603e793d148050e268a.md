# Commit 3c51692543

## Summary

One low-severity user-facing regression found: after the new snapshot cap is reached, recovery can still tell users that the rejected/original config was saved as `.clobbered.*` even when the new helper intentionally skipped writing a new snapshot.

## Findings

### Low: Recovery messages overpromise `.clobbered.*` preservation after the cap is reached

- Kind: bug
- File: `src/commands/doctor-config-preflight.ts`
- Line: 114
- Evidence: `persistBoundedClobberedConfigSnapshot` now returns `null` when the per-path cap is reached (`src/config/io.clobber-snapshot.ts:176`). Prefix recovery still overwrites `openclaw.json` with the recovered JSON after that, and its own warning only includes the saved path when `clobberedPath` is truthy (`src/config/io.ts:995`). But the doctor preflight note unconditionally says `original saved as .clobbered.*` whenever `recoverConfigFromJsonRootSuffix` returns true. The same overclaim exists in the gateway recovery notice text for last-known-good recovery (`src/gateway/config-recovery-notice.ts:25`) even though `recoverConfigFromLastKnownGood` records `clobberedPath` as nullable (`src/config/io.observe-recovery.ts:893`).
- Impact: In the exact repeated recovery loops this commit is fixing, post-cap recoveries can overwrite the current rejected/prefixed payload without creating a per-incident forensic snapshot while telling the operator that one exists. That can send users to inspect a backup that was not written for the current recovery.
- Suggested fix: Plumb `clobberedPath` or a `snapshotSaved` flag through the recovery result, then make doctor and gateway notices conditional. If the cap was reached, say that no additional `.clobbered.*` snapshot was written and point users to existing snapshots/audit logs.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. Seed 32 `openclaw.json.clobbered.*` files, put a prefixed invalid `openclaw.json`, and run the doctor prefix repair path; the bounded helper returns `null`, recovery succeeds, and the preflight note at `src/commands/doctor-config-preflight.ts:114` still claims the original was saved.
- Is this the best way to solve the issue? Yes. The cap itself is reasonable; the safer fix is to make the notice reflect whether a snapshot was actually written.
