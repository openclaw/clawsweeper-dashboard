# Commit 8b09aa9

## Summary

Found one low-severity test portability regression in the tightened WhatsApp outbound ffmpeg assertions.

## Findings

### Low: ffmpeg path assertions are POSIX-only

- Kind: compatibility
- File: `extensions/whatsapp/src/send.test.ts`
- Line: 303
- Evidence: The new assertions require `ffmpegArgs[5]` to contain `/input.${media.name}` and the output path to contain `/fs-safe-output-`. The implementation builds these paths through `workspace.write(...)` and `writeExternalFileWithinRoot(...)`, which pass native `node:path` paths into `runFfmpeg` in `extensions/whatsapp/src/outbound-media-contract.ts:192` and `extensions/whatsapp/src/outbound-media-contract.ts:217`. On Windows those paths use `\`, so valid paths such as `C:\...\input.mp3` do not contain `/input.mp3`.
- Impact: `pnpm test extensions/whatsapp/src/send.test.ts -- --reporter=verbose` can fail on Windows for test-only reasons even though the runtime ffmpeg arguments are correct.
- Suggested fix: Assert on path components instead of POSIX substrings, for example `path.basename(ffmpegArgs[5]) === "input.mp3"` and either normalize separators or check `path.basename(outputPath)` / `path.dirname(outputPath)` for the generated temp-output naming.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Run the focused test on Windows; the new `/input...` and `/fs-safe-output-` substring checks will not match native backslash-separated paths. I also verified with `path.win32` that equivalent Windows paths fail those substring checks.
- Is this the best way to solve the issue? yes. The test is checking path intent, not slash style, so using `path.basename` or separator normalization keeps the tightened assertion without making it platform-specific.
