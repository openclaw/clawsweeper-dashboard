## Findings

### Low: Nested non-POSIX wrappers still receive generated approval highlights

- Kind: bug
- File: `src/infra/command-explainer/format.ts`
- Line: 38
- Evidence: `hasUnsupportedShellWrapper()` checks only `explanation.topLevelCommands`, while `formatCommandSpans()` later emits spans for both `topLevelCommands` and `nestedCommands`. A focused probe on current `main` showed `sh -c "pwsh -Command Get-ChildItem"` returns spans for `sh` and `pwsh`; `bash -lc "cmd.exe /d /s /c dir"` returns spans for `bash` and `cmd.exe`.
- Impact: Approval prompts can still show generated highlights for unsupported PowerShell/CMD execution when those wrappers are invoked from a parseable POSIX shell payload, even though the same non-POSIX wrappers are intentionally suppressed at top level. The full command text is still displayed, so this is a UI clarity/safety issue rather than an approval bypass.
- Suggested fix: Apply the unsupported-wrapper check to the same command set used for span emission, or explicitly scan nested wrapper-payload commands before returning generated spans. Add regression coverage for POSIX-wrapper payloads containing `pwsh -Command ...` and `cmd.exe /c ...`.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes: run a formatter probe for `sh -c "pwsh -Command Get-ChildItem"` or `bash -lc "cmd.exe /d /s /c dir"` and inspect `formatCommandSpans()` output.
- Is this the best way to solve the issue? yes, if the intended contract is “no generated command spans whenever a recognized non-POSIX shell wrapper is present.” If nested POSIX payload highlights are intentionally allowed, the contract should be made explicit in tests.
