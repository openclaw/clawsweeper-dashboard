# Commit 44afab6

## Summary

Found one actionable regression in the media factory prefilter. The current `main` follow-up commits fixed several manifest/provider availability holes from this commit, but the prefilter still ignores `tools.alsoAllow` / provider `alsoAllow` before deciding whether to construct media tools.

## Findings

### Medium: `alsoAllow` cannot re-enable media generation/PDF tools before the factory prefilter drops them

- Kind: regression
- File: `src/agents/pi-tools.ts`
- Line: 606
- Evidence: `profilePolicyWithAlsoAllow` is computed at `src/agents/pi-tools.ts:433` and used by the final policy pipeline at `src/agents/pi-tools.ts:780`, but `pluginToolAllowlist` passed into `createOpenClawTools` is built from the raw `profilePolicy` / `providerProfilePolicy` at `src/agents/pi-tools.ts:606` and passed at `src/agents/pi-tools.ts:705`. The reviewed commit made `createOpenClawTools` use that prefilter for `image_generate`, `video_generate`, `music_generate`, and `pdf` at `src/agents/openclaw-tools.ts:157` and `src/agents/openclaw-tools.ts:346`. A focused smoke showed `tools.profile: "messaging"` plus `tools.alsoAllow: ["image_generate"]` resolves to a merged policy containing `image_generate`, while the prefilter allowlist does not; the factory plan then returns `"imageGenerate": false` even with explicit `agents.defaults.imageGenerationModel`.
- Impact: Restrictive profiles documented to be widened with `tools.alsoAllow` can no longer enable configured media generation/PDF tools. The final policy would allow the tool, but the tool is absent because the factory was skipped first. The same raw-prefilter pattern exists in gateway tool resolution at `src/gateway/tool-resolution.ts:95`, so gateway-scoped tool listing can also omit tools that `alsoAllow` should expose.
- Suggested fix: Build the factory prefilter from the same effective policies used by the final policy pipeline: use `profilePolicyWithAlsoAllow` and `providerProfilePolicyWithAlsoAllow` in `collectExplicitAllowlist`, and mirror that in `src/gateway/tool-resolution.ts`. Add a regression test for `tools.profile: "messaging"` plus `tools.alsoAllow: ["image_generate"]` with explicit image generation config.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Direct policy smoke: `profileAlsoAllow` contains `image_generate`, the merged profile policy contains it, but the prefilter source does not. Direct factory-plan smoke with explicit `imageGenerationModel` and a prefilter lacking `image_generate` returns `imageGenerate: false`.
- Is this the best way to solve the issue? yes. The factory prefilter is only an optimization; it should not be stricter than the final effective policy.
