# Commit df9d26e

## Summary

Found one low-severity regression in the narrowed hook timeout.

## Findings

### Low: Live timeout increases can still be capped by the startup hook budget

- Kind: regression
- File: `extensions/active-memory/index.ts`
- Line: 2434
- Evidence: The hook registration now freezes `beforePromptBuildTimeoutMs` from the startup `config.timeoutMs + setupGraceTimeoutMs`. The handler then calls `refreshLiveConfigFromRuntime()` on every invocation before running recall, and `maybeResolveActiveRecall()` uses that refreshed `params.config.timeoutMs` for its own watchdog. The hook runner stores the registration `timeoutMs` and races the handler against that fixed value.
- Impact: If Active Memory starts with the default timeout, the registered hook budget is 45s. If live runtime config later raises `plugins.entries.active-memory.config.timeoutMs` to 90s or 120s while the process remains up, the subagent is still skipped by the hook runner at 45s. Before this commit, the hook was registered for the maximum 150s envelope, so live increases could complete. This matters because the plugin already has tests and code paths for live runtime config changes, and the existing tests only cover the initial configured timeout case.
- Suggested fix: Register the hook with the maximum allowed envelope, `120_000 + setupGraceTimeoutMs`, or add a dynamic hook timeout/update mechanism. Keep the embedded subagent timeout itself at the live `config.timeoutMs`.
- Confidence: high
