# Commit 6baa2b38

## Summary

Found one concrete publisher/prompt contract bug: some failed Telegram capture-infrastructure manifests can still publish a public PR comment that says the run was merely skipped.

## Findings

### Medium: Capture failures can be published as skipped no-visual proof

- Kind: bug
- File: `scripts/mantis/publish-pr-evidence.mjs`
- Line: 347
- Evidence: The prompt now tells Mantis that visual PRs blocked by Telegram session state, authorization, credentials, Crabbox, or other capture infrastructure should emit `comparison.pass: false`, skipped lanes, no artifacts, and a summary starting `Mantis could not capture Telegram Desktop proof because` (`.github/codex/prompts/mantis-telegram-desktop-proof.md:80`). The publisher, however, treats every no-visible-artifact manifest with both lanes `skipped` as `isSkippedNoVisualProof` regardless of `comparison.pass` (`scripts/mantis/publish-pr-evidence.mjs:319`), and suppresses PR comments only when the summary matches a narrow regex (`scripts/mantis/publish-pr-evidence.mjs:347`). A focused probe with summary `Mantis could not capture Telegram Desktop proof because Crabbox timed out before the native Telegram session started.` returned `shouldPublishPrComment= true` and rendered `Summary: Mantis did not generate before/after GIFs...` plus `- Overall: \`skipped\``.
- Impact: The workflow posts the evidence comment before the later failure step (`.github/workflows/mantis-telegram-desktop-proof.yml:501`, `.github/workflows/mantis-telegram-desktop-proof.yml:535`), so an infrastructure failure can leave a public PR comment that looks like an intentional non-visual skip instead of a failed proof attempt. Maintainers reading the PR thread can be misled about whether Telegram Desktop proof was actually captured or intentionally skipped.
- Suggested fix: Make capture-failure detection structural, not wording-dependent. For example, if both lanes are skipped, no visible artifacts exist, and `comparison.pass === false`, do not publish the PR comment; additionally suppress summaries with the documented `Mantis could not capture Telegram Desktop proof because` prefix. Keep `overallStatus` from rendering `skipped` for failed capture manifests if they ever are rendered.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Create a schema v1 manifest with no artifacts, both `baseline.status` and `candidate.status` set to `skipped`, `comparison.pass: false`, and a documented capture-failure summary mentioning `Crabbox timed out`; `shouldPublishPrComment(manifest)` returns `true`, and `renderEvidenceComment` renders the sanitized no-visual summary with `Overall: skipped`.
- Is this the best way to solve the issue? yes. The prompt already defines capture failures structurally with `comparison.pass: false`; relying on a small set of English substrings is weaker than the manifest contract.
