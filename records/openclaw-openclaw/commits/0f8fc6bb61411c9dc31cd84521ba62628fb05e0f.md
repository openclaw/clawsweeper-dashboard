# Commit 0f8fc6b

## Summary

Found one low-severity test reliability regression that still exists on current `main`: the Bonjour advertiser tests now fail when the test process itself runs with Fly Machine environment variables.

## Findings

### Low: Bonjour advertiser tests inherit ambient Fly env and skip advertising paths

- Kind: reliability
- File: `extensions/bonjour/src/advertiser.test.ts`
- Line: 130
- Evidence: `advertiser.test.ts` snapshots the original `process.env` and restores it after each test, but it never clears `FLY_MACHINE_ID` / `FLY_APP_NAME` before tests that expect Bonjour to start. The commit made `extensions/bonjour/src/advertiser.ts:138` treat both variables as a container signal, so `isDisabledByEnv()` returns true and skips service creation.
- Impact: Running the Bonjour test file in a Fly Machine, or any CI/dev environment that has both Fly variables set, produces widespread false failures. I reproduced this on current `main` with `FLY_MACHINE_ID=test-machine FLY_APP_NAME=test-app pnpm test extensions/bonjour/src/advertiser.test.ts -- --reporter=verbose`: 20 of 27 tests failed, starting with `expected "vi.fn()" to be called 1 times, but got 0 times` at `extensions/bonjour/src/advertiser.test.ts:175`.
- Suggested fix: Mirror the current `src/gateway/net.test.ts` Fly-env isolation helper in the Bonjour advertiser suite: clear `FLY_MACHINE_ID` and `FLY_APP_NAME` before each test, restore them after each test, and set them only in the Fly-specific regression test.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: run `FLY_MACHINE_ID=test-machine FLY_APP_NAME=test-app pnpm test extensions/bonjour/src/advertiser.test.ts -- --reporter=verbose`.
- Is this the best way to solve the issue? yes: clearing ambient Fly variables preserves the intended production behavior while keeping unit tests independent of the host runtime environment. This is the same pattern already added on current `main` in `src/gateway/net.test.ts`.
