# Commit 0519107

## Summary

Found two concrete regressions in the shared helper extraction. The serious one makes packaged installs and Docker install layers fail because the new helper is not included wherever `postinstall-bundled-plugins.mjs` is shipped or copied.

## Findings

### High: New postinstall helper is not shipped or copied before postinstall runs

- Kind: regression
- File: `scripts/postinstall-bundled-plugins.mjs`
- Line: 27
- Evidence: This commit adds `import ... from "./lib/bundled-runtime-deps-install.mjs"` in `scripts/postinstall-bundled-plugins.mjs`, and `package.json` still ships `scripts/postinstall-bundled-plugins.mjs` but not `scripts/lib/bundled-runtime-deps-install.mjs` in `files` (`package.json:58-62`). The Docker install layers have the same fixed copy list: `Dockerfile:65-66` and `scripts/docker/cleanup-smoke/Dockerfile:22-23` copy `postinstall-bundled-plugins.mjs` and `scripts/lib/package-dist-imports.mjs`, but not the new helper, before `pnpm install`/postinstall runs.
- Impact: Installing the published `openclaw` package will fail during `postinstall` with `ERR_MODULE_NOT_FOUND` before the script can reach its non-fatal runtime-dependency handling. Docker builds using the current Dockerfiles will also fail in the install/prune layer for the same missing import.
- Suggested fix: Add `scripts/lib/bundled-runtime-deps-install.mjs` to `package.json` `files`, `scripts/release-check.ts` package file expectations, both Docker copy lists, and the Dockerfile/package-file guard tests that already cover `package-dist-imports.mjs`.
- Confidence: high

### Medium: Fallback runtime npm installs lost their timeout

- Kind: reliability
- File: `scripts/stage-bundled-plugin-runtime-deps.mjs`
- Line: 921
- Evidence: `runNpmInstall` now passes `timeout: params.timeoutMs ?? 5 * 60 * 1000` into `runBundledRuntimeDependencyNpmInstall`, but the shared helper only forwards `params.timeoutMs` to `spawnSync` (`scripts/lib/bundled-runtime-deps-install.mjs:57`). The previous inline implementation passed `timeout: params.timeoutMs ?? 5 * 60 * 1000` directly to `spawnSync`, so the default 5-minute cap was effective.
- Impact: If root graph staging falls back to a real `npm install`, a stalled registry/network/process can now block the build indefinitely instead of timing out and retrying/failing. That affects bundled plugin runtime dependency staging during builds/package preparation.
- Suggested fix: Pass `timeoutMs` from `runNpmInstall`, or make the helper accept `timeout` consistently. Add a focused test asserting the fallback installer forwards the default timeout to `spawnSync`.
- Confidence: high
