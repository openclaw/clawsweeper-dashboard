# Commit 5e8e77e

## Summary

Found one low-severity behavior bug in the new action allowlist enforcement: allowing `broadcast` without also allowing `send` exposes an action that cannot successfully deliver.

## Findings

### Low: `broadcast`-only allowlists make every broadcast target fail

- Kind: bug
- File: `src/infra/outbound/message-action-runner.ts`
- Line: 558
- Evidence: `runMessageAction` authorizes the outer `broadcast` action at `src/infra/outbound/message-action-runner.ts:1046`, then `handleBroadcastAction` implements each delivery by recursively calling `runMessageAction({ ...input, action: "send", ... })` at `src/infra/outbound/message-action-runner.ts:558`. With `tools.message.actions.allow: ["broadcast"]`, the outer action is allowed, but each internal send is rejected by the same allowlist with `Message action "send" is disabled for this agent.` A focused live check returned `{"results":[{"channel":"workspace","to":"C12345678","ok":false,"error":"Message action \"send\" is disabled for this agent."}]}`.
- Impact: A valid-looking config can expose and accept `broadcast` while making the action unusable. The tool call returns a broadcast result with failed targets rather than rejecting the configuration or succeeding, so agents/operators can get confusing delivery failures.
- Suggested fix: Once `broadcast` itself has passed the allowlist, execute its internal sends through a helper that does not re-check the external action allowlist, or define/validate the contract so `broadcast` requires `send` in the allowlist and reject/hide `broadcast` when `send` is absent. Add a regression test for `actions.allow: ["broadcast"]`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Configure `tools.message.actions.allow` to `["broadcast"]`, run a dry broadcast through the test workspace plugin, and observe each target failing with `Message action "send" is disabled for this agent.`
- Is this the best way to solve the issue? yes, if `broadcast` is intended to be an independently allowlisted action. If maintainers want `broadcast` to require `send`, the safer alternative is config/schema filtering plus a clear validation error.
