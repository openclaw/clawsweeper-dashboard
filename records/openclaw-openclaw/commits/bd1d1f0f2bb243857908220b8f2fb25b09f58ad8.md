# Commit bd1d1f0

## Summary

Found one Slack access-control gap: normal Slack DM messages now use the stricter open-DM allowlist semantics, but Slack DM system events still treat an empty allowlist as allow-all.

## Findings

### Medium: Slack DM system events still bypass the new open-DM wildcard requirement

- Kind: security
- File: `extensions/slack/src/monitor/auth.ts`
- Line: 144
- Evidence: `isSlackSenderAllowListed` returns true when `allowListLower.length === 0`. `authorizeSlackSystemEventSender` uses that helper for DM channels at `extensions/slack/src/monitor/auth.ts:260`, so `dmPolicy: "open"` with no `allowFrom: ["*"]` still authorizes any DM sender for this path. The system-event wrapper then treats that as authorized at `extensions/slack/src/monitor/events/system-event-context.ts:19`, and reaction/pin/message subtype handlers enqueue system events from that context at `extensions/slack/src/monitor/events/reactions.ts:45`, `extensions/slack/src/monitor/events/pins.ts:41`, and `extensions/slack/src/monitor/events/messages.ts:176`. Existing tests also encode this: `extensions/slack/src/monitor/events/reactions.test.ts:87` expects a DM reaction with `dmPolicy: "open"` and default empty `allowFrom` to enqueue.
- Impact: A legacy or invalid Slack config with `dmPolicy: "open"` but no wildcard now blocks regular DM messages through `authorizeSlackDirectMessage`, but still accepts DM reactions, pins, and message subtype events from arbitrary users as agent system events. That leaves a non-message injection surface open after this commit’s stated policy change.
- Suggested fix: For Slack DM system events, use strict allowlist matching like `authorizeSlackDirectMessage` does, or make `isSlackSenderAllowListed` accept an `allowWhenEmpty` option and pass `false` for `channelType === "im"`. Update Slack system-event tests so open DMs without `allowFrom: ["*"]` do not enqueue.
