# Commit f2a0b3d2e2

## Summary

Found one low-severity regression in the new Crabbox wrapper parser. It does not alter the command forwarded to Crabbox, but it can make the wrapper reason about the wrong provider and skip the intended stale-provider guard/warnings.

## Findings

### Low: Valid `run` value flags can hide a later `--provider` from wrapper diagnostics

- Kind: regression
- File: `scripts/crabbox-wrapper.mjs`
- Line: 46
- Evidence: `runCommandBounds()` only skips values for names in `runValueOptions`; upstream Crabbox `run` also defines value flags such as `--preset`, `--scenario`, `--emit-proof`, `--proof-template`, `--stop-after`, plus repeatable `--preset-var` and `--artifact-glob`. When one of those appears before `--provider`, the wrapper treats the value as the remote command start and `commandProvider()` ignores the later provider. A focused fake-Crabbox smoke showed `node scripts/crabbox-wrapper.mjs run --preset qa-live --provider blacksmith-testbox --shell -- "pnpm test"` logging `provider=aws` while forwarding `--provider blacksmith-testbox` unchanged.
- Impact: valid Crabbox invocations can get incorrect wrapper provider logging, miss the `blacksmith-testbox` fallback warning, and bypass the wrapper’s “selected binary does not advertise provider” preflight for the actual provider. The remote command still reaches Crabbox, so this is diagnostic/guard drift rather than command execution corruption.
- Suggested fix: include all current Crabbox `run` value flags in `runValueOptions` and add focused tests for `--provider` after `--preset`, `--scenario`, and repeatable value flags. A more robust fix would avoid a partial hand-maintained parser for provider detection.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A fake `crabbox` on `PATH` that reports normal `--version` and `run --help` output reproduced the mismatch: wrapper logged `provider=aws` for a command whose argv still contained `--provider blacksmith-testbox`.
- Is this the best way to solve the issue? yes for a narrow repair: update the value-flag list and add regression tests. Longer term, deriving option metadata from Crabbox help or parsing only until an explicit remote `--` delimiter would reduce future drift.
