# Commit 04cd861

## Summary

Found one low-severity redaction regression in the new fallback regex.

## Findings

### Low: URL userinfo with whitespace can now leak when embedded in error text

- Kind: privacy
- File: `src/shared/net/redact-sensitive-url.ts`
- Line: 73
- Evidence: The new fallback regex excludes whitespace from the userinfo match: `.replace(/\/\/([^@/?#\s]+)@/g, "//***:***@")`. A standalone URL like `https://user:pa ss@example.com/path` is accepted by `new URL(...)` and redacted by `redactSensitiveUrl`, but the same URL embedded in arbitrary text now falls through and is not redacted. Focused check output: `fatal https://user:pa ss@example.com/path and https://***:***@example.net/path`.
- Impact: Any git/plugin/MCP error text that includes an unescaped space in URL userinfo can expose that credential. This is a regression from the previous fallback regex, which redacted that first occurrence.
- Suggested fix: Keep the global repeated-occurrence behavior, but handle URL-like userinfo with whitespace using a bounded URL-token parser or a more precise second pass, and add a regression test for embedded `https://user:pa ss@example.com/path`.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the helper behavior: `pnpm tsx -e` calling `redactSensitiveUrlLikeString("fatal https://user:pa ss@example.com/path and https://x:y@example.net/path")` leaves the first credential visible.
- Is this the best way to solve the issue? Unclear. A parser/tokenizer is safer than relaxing the regex back to cross whitespace broadly, because the commit also fixes repeated non-whitespace URL occurrences.
