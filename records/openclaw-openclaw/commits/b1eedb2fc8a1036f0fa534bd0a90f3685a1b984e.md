# Commit b1eedb2

## Summary

Found one concrete ACP ledger correctness regression. The new ledger can keep a session marked complete while the stored event history is partial or stale, so `loadSession` trusts the ledger and skips the safer Gateway transcript fallback.

## Findings

### Medium: `loadSession` can replay partial or stale ledger history as complete

- Kind: reliability
- File: `src/acp/event-ledger.ts`
- Line: 241
- Evidence: `getOrCreateSession` preserves completeness with `existing.complete = existing.complete || params.complete`, so a later `startSession(... complete: false)` cannot downgrade an existing complete ledger. Separately, `newSession` starts the ledger as complete at `src/acp/translator.ts:651`, and accepted prompts are appended at `src/acp/translator.ts:979` without first marking the session incomplete while the run is active. `loadSession` then treats `ledgerReplay.complete` as authoritative and skips `sessions.get` at `src/acp/translator.ts:707`.
- Impact: If the ACP bridge exits, crashes, or is restarted after `chat.send` is accepted but before the terminal Gateway event is recorded, the persisted ledger still reads as complete. A later `loadSession` replays only the initial updates and user prompt, and does not call `sessions.get`, so the IDE misses the final Gateway transcript. The same root issue also rekeys stale complete history: starting an existing complete ledger session with a new `sessionKey` and `complete: false` makes the old events replay as complete under the new key.
- Suggested fix: Treat an accepted/in-flight run and session-key reroutes as incomplete until a terminal final/error/cancel path has recorded all corresponding ACP updates. In `getOrCreateSession`, do not preserve `complete: true` when `params.complete` is false or when the session key changes; add an explicit “mark complete” path after terminal event recording succeeds.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused `pnpm exec tsx` harness showed `replayCompleteBeforeTerminalEvent: true` with an active run and only `session_info_update`, `available_commands_update`, and `user_message_chunk` in the ledger; a subsequent `loadSession` reported `loadCalledSessionsGet: false`.
- Is this the best way to solve the issue? yes, directionally. The bug is in the ledger completeness contract, not the transcript fallback; the fallback already works when `complete` is false.
