## Summary

Found one low-severity live-test runner gap. The new image-probe override works in the test helper, but the primary Docker ACP bind runner does not forward the override env var into the container, so the forced OpenCode image probe cannot be exercised there.

## Findings

### Low: Docker ACP bind runner drops the new image-probe override

- Kind: test_gap
- File: `scripts/test-live-acp-bind-docker.sh`
- Line: 344
- Evidence: `src/gateway/gateway-acp-bind.live.test.ts:914` reads `process.env.OPENCLAW_LIVE_ACP_BIND_IMAGE_PROBE`, and `src/gateway/live-agent-probes.test.ts:35` asserts `override: "1"` forces the OpenCode image probe. The Docker runner’s `docker run` env list forwards ACP bind agent/model/command values at `scripts/test-live-acp-bind-docker.sh:344`, but it never includes `OPENCLAW_LIVE_ACP_BIND_IMAGE_PROBE`.
- Impact: `OPENCLAW_LIVE_ACP_BIND_IMAGE_PROBE=1 pnpm test:docker:live-acp-bind:opencode` will still skip the image probe inside the container. That blocks the intended opt-in coverage for debugging or future image-capable OpenCode ACP behavior in the main live Docker lane.
- Suggested fix: forward the env var in `scripts/test-live-acp-bind-docker.sh`, for example `-e OPENCLAW_LIVE_ACP_BIND_IMAGE_PROBE`, and document it with the other ACP bind overrides if it is meant to be user-facing.
- Confidence: high

- Is this the best way to solve the issue? Yes. The helper/test behavior is fine; the missing integration piece is the Docker env forwarding.
