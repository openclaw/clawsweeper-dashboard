## Findings

### Low: rewritten account payload checks no longer reject extra rows

- Kind: test_gap
- File: `src/gateway/server-methods/channels.status.test.ts`
- Line: 212
- Evidence: Before this commit, both affected checks used exact array/object equality, e.g. `channelAccounts.whatsapp` had to equal a single-element array. The new code only verifies `channelAccounts.whatsapp` is an array, destructures the first element, and asserts fields on that first account. The same pattern appears again at line 298.
- Impact: A regression that returns duplicate or stale account rows for a single configured channel would now pass these tests. The fallback-summary case also no longer rejects extra `channelAccounts` keys that the previous `toEqual({ whatsapp: [...] })` rejected.
- Suggested fix: Keep the explicit field assertions, but add exact-shape checks such as `expect(channelAccounts.whatsapp).toHaveLength(1)` and, where the old object equality mattered, `expect(Object.keys(channelAccounts)).toEqual(["whatsapp"])`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes; make `channels.status` append a second `whatsapp` account row in either touched scenario. The current assertions inspect only the first row, while the pre-commit `toEqual([expect.objectContaining(...)])` assertions would fail.
- Is this the best way to solve the issue? yes; adding length/key checks preserves the commit’s explicit payload assertions while restoring the old exact-shape coverage.
