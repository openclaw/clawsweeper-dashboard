# Commit eb6dd2c

## Summary

Found one remaining help/dispatch mismatch in the new config-sensitive root help path. The intended configured `memory-lancedb` path works once the plugin has an enabled, valid config, but a slot-only or incomplete config can now advertise `ltm` in top-level help even though `openclaw ltm --help` does not load the LanceDB command.

## Findings

### Medium: Root help can advertise `ltm` before the LanceDB CLI can dispatch

- Kind: regression
- File: `extensions/memory-lancedb/cli-metadata.ts`
- Line: 8
- Evidence: The new metadata descriptor unconditionally advertises `ltm`, and root help switches to live plugin descriptors whenever `plugins.slots` is present (`src/cli/root-help-live-config.ts:23`, `src/cli/run-main.ts:536`). But the real LanceDB plugin returns before registering its CLI when `memoryConfigSchema.parse(api.pluginConfig)` fails (`extensions/memory-lancedb/index.ts:640` and `extensions/memory-lancedb/index.ts:652`). Focused smoke: with only `{"plugins":{"slots":{"memory":"memory-lancedb"}}}`, `node --import tsx src/entry.ts --help` prints `ltm                  Inspect and query LanceDB-backed memory`, while `node --import tsx src/entry.ts ltm --help` fails with “bundled plugin is disabled by default.” With `entries.memory-lancedb.enabled=true` but no plugin config, root help still advertises `ltm`, and `ltm --help` exits 0 with generic root help instead of the LanceDB subcommand help.
- Impact: This reintroduces the same class of bug the commit is fixing: top-level help can point users to a plugin command that is unavailable at dispatch time. It affects partial/manual memory plugin configs and is especially plausible because the docs say selecting `plugins.slots.memory = "memory-lancedb"` enables the active memory plugin.
- Suggested fix: Make root-help descriptor loading use the same effective enablement and runtime registration eligibility as plugin CLI dispatch, or make the LanceDB CLI registrar available for `--help` even when runtime memory config is incomplete. At minimum, do not include the `ltm` descriptor for slot-only/default-disabled or invalid LanceDB configs.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: create a temp config with only `plugins.slots.memory = "memory-lancedb"`, run root help, then run `ltm --help`; root help lists `ltm`, dispatch rejects it.
- Is this the best way to solve the issue? yes: root help should be generated from the same availability contract as command dispatch, or the plugin should register help-only command surfaces even when runtime services are not configured.
