# Commit ba3c0fc

## Summary

Found one rollback-path issue: failed npm plugin updates can remove the existing managed install instead of preserving or restoring it.

## Findings

### Medium: Failed npm update rollback can remove the installed plugin

- Kind: data_loss
- File: `src/plugins/install.ts`
- Line: 1288
- Evidence: `installPluginFromNpmSpec` is used for both fresh installs and updates. `src/plugins/update.ts:1030` calls it with `mode: "update"`, but when post-install validation returns `!result.ok`, `src/plugins/install.ts:1288` unconditionally calls `rollbackManagedNpmPluginInstall`, which runs `npm uninstall`, removes `installRoot`, and removes the package from the managed root manifest. There is no branch that preserves the pre-existing install or restores the prior dependency spec for update mode.
- Impact: If an npm plugin update installs a package that later fails validation, such as security scan, manifest validation, min-host-version, or expected plugin id checks, the update reports failure but the previously installed plugin can be deleted from `~/.openclaw/npm/node_modules` and removed from the managed npm root manifest. That turns a failed update into plugin removal.
- Suggested fix: Treat update rollback separately from fresh-install rollback. Stage npm updates in a temporary managed root and validate before replacing the active package, or record the previous dependency spec and reinstall/restore it on failure. At minimum, do not call the destructive uninstall/rm rollback for `effectiveMode === "update"` unless a restore path succeeds.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes: set up an existing managed npm plugin directory, call `installPluginFromNpmSpec` with `mode: "update"`, mock npm install to succeed with a package that fails the post-install scan, then assert the old install path and managed dependency remain. The current code removes them.
- Is this the best way to solve the issue? Unclear. The safest direction is staged validation before replacing the active npm root entry; a fallback reinstall of the previous spec is less robust but would reduce data loss.
