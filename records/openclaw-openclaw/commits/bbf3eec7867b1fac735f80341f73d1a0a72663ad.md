## Summary

One actionable QA regression found: the new install-race markdown scenario deadlocks when executed by the real QA flow runner.

## Findings

### Medium: Codex install-race scenario waits forever before marking install complete

- Kind: reliability
- File: `qa/scenarios/runtime/codex-plugin-install-race.md`
- Line: 41
- Evidence: The scenario stores `turn` with `expr: "gate.runFirstTurnAfterInstall(...)"` at lines 41-43, then calls `gate.markInstalled` at line 48. But the flow runner resolves every `set` value through `resolveValue`, and `resolveValue` awaits `evalExpr` for expression values in `extensions/qa-lab/src/scenario-flow-runner.ts:111`. `evalExpr` also awaits the returned expression result at `extensions/qa-lab/src/scenario-flow-runner.ts:81`. A focused smoke of just this scenario returned `timed-out` after 1s; the other five new flows completed.
- Impact: `codex-plugin-install-race` is `runtimeParityTier: standard`, and unfiltered QA suite selection includes matching catalog scenarios. Running this scenario stalls before `gate.markInstalled()` can execute, so the new standard QA coverage can hang or time out instead of proving the intended race behavior.
- Suggested fix: Do not start the pending install-gated promise through a `set` expression that the flow runner auto-awaits. Use a synchronous wrapper/value that stores a non-thenable handle, add explicit flow support for “do not await” promise capture, or collapse the race assertion into a helper function that internally starts the turn, asserts the waiting event, marks installed, awaits completion, and returns the result.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A minimal `runScenarioFlow` smoke for `codex-plugin-install-race` raced the scenario against a 1s timer and printed `timed-out`.
- Is this the best way to solve the issue? yes directionally. The failure is caused by the flow engine’s intentional auto-await contract, so the scenario must avoid representing a pending promise as a normal `set` expression unless the engine gains an explicit non-awaiting primitive.
