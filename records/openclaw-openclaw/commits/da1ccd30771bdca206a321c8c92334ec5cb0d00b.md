# Commit da1ccd3

## Summary

Found one regression in the rich-only block reply path: delivering a presentation-only block now marks the block pipeline as streamed, which can suppress unrelated final text that was never streamed.

## Findings

### Medium: Rich-only block replies can suppress unsent final text

- Kind: regression
- File: `src/auto-reply/reply/block-reply-pipeline.ts`
- Line: 244
- Evidence: The changed branch sends any non-text content immediately. `sendPayload` then marks `didStream = true` for every successful payload, including presentation-only payloads. Later, `buildReplyPayloads` treats `didStream() && !isAborted()` as proof that final payloads are covered and drops non-media final payloads without checking whether their text was sent.
- Impact: A turn that emits a rich-only block reply, such as buttons/cards, and later returns normal final text can deliver the rich block but drop the final answer. A focused TSX probe confirmed this: after enqueuing a presentation-only block, `pipeline.didStream()` was `true`, `pipeline.hasSentPayload({ text: "final answer" })` was `false`, and `buildReplyPayloads(... payloads: [{ text: "final answer" }])` returned `[]`.
- Suggested fix: Separate “sent any block payload” from “streamed final text coverage.” Use a text-streamed flag, or have final suppression preserve non-media payloads when `hasSentPayload(payload)` is false. Rich-only final duplicates can still be suppressed through the content key.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Run a one-off probe that creates a coalescing `createBlockReplyPipeline`, enqueues `{ presentation: { blocks: [...] } }`, flushes it, then calls `buildReplyPayloads` with `blockStreamingEnabled: true`, the same pipeline, and final payload `{ text: "final answer" }`; current `main` drops the final payload.
- Is this the best way to solve the issue? yes, directionally. The bug is caused by `didStream` becoming too broad for its downstream suppression contract, so the safest fix is to narrow the suppression signal or fall back to `hasSentPayload` for final text.
