# Commit 8483d03

## Summary

Found one regression in the new `configuredAgentsOnly` flow: unconfigured discovered stores are now merged before filtering, so their entries can win key collisions and surface as configured sessions.

## Findings

### Medium: Hidden discovered stores can override configured session rows before filtering

- Kind: regression
- File: `src/gateway/server-methods/sessions.ts`
- Line: 900
- Evidence: `sessions.list` no longer passes `configuredAgentsOnly` into `loadCombinedSessionStoreForGateway`, so the loader uses `resolveAllAgentSessionStoreTargetsSync` and merges configured plus disk-discovered stores before `filterSessionStoreToConfiguredAgents` runs. The merge keeps the freshest entry for a canonical key, while the later filter only sees the merged key and accepts `agent:main:main` as configured.
- Impact: A retired/unregistered store under the agents root that contains a fully qualified configured key, such as `agent:main:main`, can replace the real configured row in `sessions.list configuredAgentsOnly` if its `updatedAt` is newer. That can show stale or wrong metadata/session ids in configured-only UI/API lists and reintroduces hidden stores through key collisions.
- Suggested fix: Filter per target before merging, or preserve store origin through the merge and prevent unconfigured store entries from overwriting configured-agent canonical keys. Child inclusion should be added without allowing unrelated discovered stores to win configured-key collisions.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused inline `node --import tsx` repro created `main` and unregistered `local` stores; `local` contained `agent:main:main` with newer `updatedAt`, and `loadCombinedSessionStoreForGateway(cfg)` returned `sessionId: hidden-local-overrides-main` for `agent:main:main`.
- Is this the best way to solve the issue? yes. The problem is merge order and missing origin-aware filtering, not ACP target expansion itself.
