# Commit a1a6cd6

## Summary

Found one low-severity regression: the new shared inbound-history helper preserves `messageId`, but several converted channel paths previously stripped it from structured `InboundHistory`. Current main still has this behavior, and representative channel tests fail on the changed shape.

## Findings

### Low: `InboundHistory` now exposes historical `messageId` for converted channels

- Kind: regression
- File: `src/auto-reply/reply/history.ts`
- Line: 151
- Evidence: Before this commit, Signal and Zalo built `InboundHistory` with only `sender`, `body`, and `timestamp`. The new `buildInboundHistoryFromEntries` copies `entry.messageId`, and `buildInboundUserContextPrefix` renders it as `message_id`. Focused tests fail in `extensions/signal/src/monitor/event-handler.inbound-context.test.ts:209` and `extensions/zalouser/src/monitor.group-gating.test.ts:942` because received history entries now include `messageId`.
- Impact: This does not preserve the previous structured history contract for converted channels. It also changes prompt-visible structured history by adding historical provider IDs, which can affect model behavior and currently breaks existing channel tests.
- Suggested fix: If preserving existing channel behavior is required, make message ID inclusion opt-in or have converted call sites strip `messageId` unless that channel intentionally wants it. If the new shape is intentional, update the affected channel tests and document the changed `InboundHistory` contract.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: `node scripts/run-vitest.mjs extensions/signal/src/monitor/event-handler.inbound-context.test.ts extensions/zalouser/src/monitor.group-gating.test.ts` reproduces the `messageId` expectation failures.
- Is this the best way to solve the issue? unclear: code should omit `messageId` to preserve prior behavior, but if the product decision is to expose history message IDs, tests and docs should be updated instead.
