# Commit 20d891a

## Summary

One low-severity test regression found. Current production code still dynamically imports `sqlite-vec`, and the focused test passes, but this commit weakens the test that was guarding the optional-dependency contract.

## Findings

### Low: Static loader import no longer proves explicit extension paths avoid bundled sqlite-vec

- Kind: test_gap
- File: `packages/memory-host-sdk/src/host/sqlite-vec.test.ts`
- Line: 2
- Evidence: The test now imports `loadSqliteVecExtension` at module load time, before `mockMissingSqliteVecPackage()` runs in each test. Since `sqlite-vec` is an optional dependency in `package.json`, normal dev installs include it, so a future top-level `sqlite-vec` import in `./sqlite-vec.js` would resolve before the test’s `vi.doMock("sqlite-vec", ...)` is installed. The prior version mocked before dynamically importing the loader, so it actually enforced the “explicit extensionPath without importing bundled sqlite-vec” contract.
- Impact: A future regression could make configured `memorySearch.store.vector.extensionPath` unusable on installs where optional `sqlite-vec` is absent, while this test would no longer catch it.
- Suggested fix: Keep the speed-up for tests that do not need module-import isolation, but preserve one isolated test that calls `vi.doMock("sqlite-vec")` before dynamically importing `./sqlite-vec.js`, or otherwise assert that the explicit `extensionPath` path does not touch the `sqlite-vec` module.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the test gap: static import order in the touched test makes the mock too late to guard module-load-time imports. The current runtime code still uses `import(SQLITE_VEC_MODULE_ID)` inside `loadSqliteVecModule()`, so no current production failure reproduced.
- Is this the best way to solve the issue? No. The speed-up is reasonable for most of the file, but this specific contract needs import-time isolation.
