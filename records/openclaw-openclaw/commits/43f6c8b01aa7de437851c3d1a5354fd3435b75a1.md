## Summary

Found two actionable issues: one SDK result-status bug and one lockfile drift issue for the new workspace package.

## Findings

### Medium: Cancelled SDK runs can be reported as timed out

- Kind: bug
- File: `packages/sdk/src/client.ts`
- Line: 37
- Evidence: `runStatusFromWaitPayload()` maps wait payloads only by `payload.status`; `status: "timeout"` always becomes `timed_out`. Current `agent.wait` snapshots use `status: "timeout"` whenever lifecycle data has `aborted`, while carrying the actual cancellation reason separately as `stopReason` (`src/gateway/server-methods/agent-job.ts:145`). The SDK event normalizer already treats `stopReason: "rpc" | "user" | "stop"` as cancellation, but `Run.wait()` does not.
- Impact: SDK consumers that cancel a run and then call `run.wait()` can see `timed_out` instead of `cancelled`, which changes UI state, retry logic, and automation decisions.
- Suggested fix: teach `runStatusFromWaitPayload()` to inspect `stopReason` and `aborted` before mapping timeout statuses, matching `normalizeLifecycleEndEventType()`, and add a unit test for `status: "timeout", stopReason: "rpc"`.
- Confidence: high

### Low: New SDK workspace package is missing from the lockfile

- Kind: supply_chain
- File: `pnpm-lock.yaml`
- Line: 1526
- Evidence: The commit adds `packages/sdk/package.json`, and `pnpm-workspace.yaml` includes `packages/*`, but `pnpm-lock.yaml` has no `packages/sdk` importer. Running `pnpm install` added:
  ```yaml
  packages/sdk: {}
  ```
  `pnpm install --frozen-lockfile --ignore-scripts` also left the same tracked lockfile change.
- Impact: Fresh installs dirty the worktree and the committed dependency graph no longer exactly represents the workspace package set. This can confuse CI/release/package acceptance checks that expect a stable lockfile after install.
- Suggested fix: run `pnpm install` and commit the `packages/sdk: {}` importer in `pnpm-lock.yaml`.
- Confidence: high
