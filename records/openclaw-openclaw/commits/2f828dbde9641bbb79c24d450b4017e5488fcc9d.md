# Commit 2f828dbde9

## Summary

Found one low-severity formatting regression in the new markdown chunk planner path. The primary long-message failure from #75868 is fixed for normal long markdown where rendered HTML needs chunking, but a markdown payload that becomes shorter after rendering can now be sent as raw plain text chunks even when the rendered Telegram HTML would fit and was previously deliverable.

## Findings

### Low: Markdown that shrinks during rendering can lose Telegram HTML formatting

- Kind: regression
- File: `extensions/telegram/src/send.ts`
- Line: 737
- Evidence: The commit routes default markdown sends through `buildChunkedTextPlan`, but that planner compares raw/plain fallback chunks against rendered HTML chunks and returns plain chunks when `fixedPlainTextChunks.length > htmlChunks.length`. For default markdown, that means inputs like 1300 `# a` headings have raw length 5199, rendered Telegram HTML length 3898, raw chunks 2, HTML chunks 1. Current code therefore sends two plain raw markdown messages instead of the one `parse_mode: "HTML"` rendered message that the previous markdown path would send successfully.
- Impact: Some long markdown messages that previously delivered as formatted Telegram HTML now arrive as raw markdown/plain text and may be split unnecessarily. This contradicts the Telegram docs contract that outbound markdown-ish text is rendered to Telegram-safe HTML.
- Suggested fix: For default markdown planning, derive chunks from the markdown IR/`markdownToTelegramChunks`-style path so each chunk carries rendered HTML plus display-text fallback, instead of using raw markdown as the fallback chunk-count guard. Keep the raw plain fallback behavior for caller-supplied `textMode: "html"`/`plainText` cases where preserving the supplied fallback is the safer contract.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. Source-level reproduction: render `Array.from({length:1300}, () => "# a").join("\n")`; it produces raw length 5199 and rendered HTML length 3898, so `buildChunkedTextPlan` takes the plain fallback branch at `extensions/telegram/src/send.ts:737`.
- Is this the best way to solve the issue? Mostly yes for the reported oversized-message failure, but the markdown fallback chunk selection needs refinement so rendered-safe markdown is not downgraded to plain text when rendering shrinks the payload.
