## Summary

Found one concrete doctor false-positive in the new context-engine host compatibility collector. Runtime enforcement paths for CLI, Pi, and Codex otherwise matched the declared host capabilities in the focused code paths reviewed.

## Findings

### Low: Doctor treats explicit `auto` as a concrete Pi host

- Kind: bug
- File: `src/commands/doctor/shared/context-engine-host-compat.ts`
- Line: 220
- Evidence: `collectExplicitRuntimeRefs()` pushes explicit `agentRuntime.id: "auto"` into the host map, and `runtimeHostCandidate()` maps `auto` to `PI_EMBEDDED_CONTEXT_ENGINE_HOST` at lines 164-166. The actual runtime policy for an OpenAI model with explicit `agentRuntime.id: "auto"` resolves to Codex in `src/agents/harness/policy.ts:41`, not Pi.
- Impact: `openclaw doctor` can warn that a compatible OpenAI/Codex config has an incompatible Pi runtime. For a context engine requiring Codex-only capability `thread-bootstrap-projection`, doctor reports a mixed compatible/incompatible setup and tells the operator to reconfigure or switch to `legacy`, even though the selected model actually runs on Codex and satisfies the requirement.
- Suggested fix: Do not add explicit `auto` as a standalone host candidate. Resolve `auto` through `resolveAgentHarnessPolicy()` for selected model refs, and add a regression test for `openai/gpt-5.5` with `agentRuntime.id: "auto"` plus a context engine requiring `thread-bootstrap-projection`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A no-write `node --import tsx` probe showed `collectConfiguredContextEngineAgentRunHosts()` returning both `auto`/`pi-embedded` and `codex` for an OpenAI model explicitly set to `auto`, while `resolveAgentHarnessPolicy()` returned `{ runtime: "codex", runtimeSource: "model" }`.
- Is this the best way to solve the issue? yes. The doctor collector should mirror runtime selection instead of interpreting `auto` as Pi before model/provider policy is applied.
