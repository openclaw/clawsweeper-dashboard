# Commit c5224a3

## Summary

Found one low-severity determinism issue in the new descriptor planner sort order.

## Findings

### Low: Descriptor ordering depends on host locale

- Kind: reliability
- File: `src/tools/planner.ts`
- Line: 13
- Evidence: `compareDescriptors` sorts `sortKey` / `name` with `localeCompare`. Live check against current `main` showed `buildToolPlan` returns `i,I,z` with `LC_ALL=en_US.UTF-8`, but `I,i,z` with `LC_ALL=tr_TR.UTF-8` for the same descriptors.
- Impact: Tool descriptor order can vary across host locale. This weakens the planner’s cache-stable/deterministic ordering goal, especially for plugin/MCP descriptors or `sortKey` values containing uppercase/non-ASCII characters.
- Suggested fix: Use a locale-independent comparator, for example direct code-unit comparison with `<` / `>`, for both `sortKey` and `name`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: run `LC_ALL=en_US.UTF-8 pnpm exec tsx -e '...'` and `LC_ALL=tr_TR.UTF-8 pnpm exec tsx -e '...'` importing `buildToolPlan` with descriptors named `i`, `I`, and `z`.
- Is this the best way to solve the issue? yes: descriptor planning wants stable ordering, and locale-independent comparison is narrower than constraining all future descriptor names/sort keys.
