# Commit 476ac66d

## Summary

One low-severity regression found in the direct Docker RTT wrapper: it now accepts `openclaw@main` without a local package tarball, but that spec is not installable from npm and the CLI path already treats it as tarball-only.

## Findings

### Low: Direct RTT Docker script accepts `openclaw@main` without the required tarball

- Kind: bug
- File: `scripts/e2e/npm-telegram-rtt-docker.sh`
- Line: 16
- Evidence: The commit adds `main` to `validate_openclaw_package_spec`, so `OPENCLAW_NPM_TELEGRAM_PACKAGE_SPEC=openclaw@main` passes validation when `OPENCLAW_NPM_TELEGRAM_PACKAGE_TGZ` / `OPENCLAW_CURRENT_PACKAGE_TGZ` is unset. In that branch, the script leaves `package_install_source="$PACKAGE_SPEC"` and later runs `npm install -g "$install_source"`, so it attempts `npm install -g openclaw@main`. The new CLI wrapper disagrees with that contract and explicitly throws `openclaw@main requires --package-tgz.`. A focused package metadata check also showed `npm view openclaw dist-tags --json` returns only `beta` and `latest`, not `main`.
- Impact: Direct users of `scripts/e2e/npm-telegram-rtt-docker.sh` can start an expensive Docker/Telegram RTT run with an accepted-looking `openclaw@main` spec only to fail during npm install. This is especially confusing because the `pnpm rtt` path correctly models `main` as requiring a local tarball.
- Suggested fix: Keep `main` out of the Docker script’s no-tarball validator, or add an early check that rejects `PACKAGE_SPEC=openclaw@main` unless `resolved_package_tgz` is non-empty. Optionally mirror the CLI error text.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: run the Docker wrapper with required Telegram env/Docker available, set `OPENCLAW_NPM_TELEGRAM_PACKAGE_SPEC=openclaw@main`, and leave both package tarball env vars unset. Static tracing shows it reaches `npm install -g openclaw@main`; npm metadata checked during review showed no `main` dist-tag.
- Is this the best way to solve the issue? yes: the intended `main` measurement path is tarball-backed, so the script should reject the no-tarball `main` case before Docker build/install work.
