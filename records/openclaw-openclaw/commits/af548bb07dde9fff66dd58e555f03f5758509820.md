## Findings

### Low: Set Default silently no-ops for visible disk-only agents

- Kind: bug
- File: `ui/src/ui/controllers/config.ts`
- Line: 369
- Evidence: `stageDefaultAgentConfigEntry` only looks for the selected agent inside the current config form/snapshot and returns `false` when `findAgentConfigEntryIndex` cannot find it (`ui/src/ui/controllers/config.ts:367`). The Agents UI still shows the Set Default button for any selected row from `agents.list` RPC (`ui/src/ui/views/agents.ts:183`), and the click handler ignores the helper return value (`ui/src/ui/app-render.ts:2151`). Separately, the gateway can include agent directories from disk in the UI list when there is no explicit `agents.list` (`src/gateway/agent-list.ts:40`, `src/gateway/agent-list.ts:74`), so a visible non-default disk-only agent can be selected but cannot be staged as default. A direct helper smoke confirmed this state returns `false`, leaves `configFormDirty=false`, and leaves `configForm=null`.
- Impact: Users with agent directories discovered from disk, but not yet represented in config, can click **Set Default** and see no dirty state, no saveable config change, and no error. This leaves the default agent unchanged.
- Suggested fix: Make the Set Default path either create a valid `agents.list` entry for selected visible agents, preserving the existing/default visible agents as needed, or disable/explain the action when the selected agent is not config-backed.
- Confidence: high
