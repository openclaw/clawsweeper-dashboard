# Commit d44aeb6

## Summary

Found two transcript-mirroring bugs in the Telegram fix. Both are source-backed and affect documented or existing Telegram delivery modes.

## Findings

### Medium: Final-only and progress-mode Telegram replies still bypass transcript mirroring

- Kind: bug
- File: `extensions/telegram/src/bot-message-dispatch.ts`
- Line: 947
- Evidence: `deliveryBaseOptions.transcriptMirror` is only consumed by `deliverReplies`, but final sends first try `deliverInboundReplyWithMessageSendContext(...)`. When Telegram durable delivery returns `handled_visible`, the function marks delivery and returns before `deliverReplies` or the preview-finalized mirror runs. Telegram exposes durable final capabilities through its message adapter, and existing tests cover this path with `deliverReplies` not called.
- Impact: Telegram sessions using `streaming.mode: "off"`, `streaming.mode: "progress"`, media finals, stale preview fallbacks, or any other non-preview final delivery can still send a visible reply without appending the `delivery-mirror` assistant message. That leaves the `.jsonl` session transcript missing the assistant output this commit is meant to restore.
- Suggested fix: mirror after successful durable final delivery, or pass a real mirror context through `deliverInboundReplyWithMessageSendContext`/`sendDurableMessageBatch` into `deliverOutboundPayloads`. Add a regression test where `SessionKey` is present, durable delivery returns `handled_visible`, and `appendAssistantMessageToSessionTranscript` or equivalent is called once.
- Confidence: high

### Medium: Long streamed finals can be mirrored twice

- Kind: bug
- File: `extensions/telegram/src/bot-message-dispatch.ts`
- Line: 1013
- Evidence: `createLaneTextDeliverer` streams the first chunk of an over-limit final answer, then sends overflow chunks with `sendPayload(...)` (`extensions/telegram/src/lane-delivery-text-deliverer.ts:181`). Those overflow sends call `deliverReplies`, whose new mirror aggregates delivered contents into the transcript (`extensions/telegram/src/bot/delivery.replies.ts:912`). After that, `emitPreviewFinalizedHook` mirrors the full final text again at `extensions/telegram/src/bot-message-dispatch.ts:1013`.
- Impact: For long Telegram streaming responses, the session transcript can contain the full assistant answer plus duplicate overflow chunks. Future turns may replay bloated or misleading assistant history.
- Suggested fix: make preview-finalized overflow mirroring single-source. Prefer mirroring the full final answer once and suppressing transcript mirroring for the generated overflow follow-up sends, or return exact sent parts from lane delivery and mirror only those once.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. For the first finding, run a dispatch with `SessionKey`, `streamMode: "off"` or `"progress"`, and `deliverInboundReplyWithMessageSendContext` returning `handled_visible`; current tests already show this bypasses `deliverReplies`. For the second, use the existing long-final path where `textLimit` forces follow-up chunks and a session key is present.
- Is this the best way to solve the issue? yes. Both fixes should centralize transcript mirroring around the actual final delivery outcome and avoid per-path ad hoc duplication.
