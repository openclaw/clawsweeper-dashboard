# Commit 21ac476

## Summary

Found one concrete packaging regression: the commit adds legacy provider-dispatcher alias entries, but the actual build does not emit the root stable file those aliases depend on, so the aliases are silently skipped.

## Findings

### Medium: Provider-dispatcher legacy shims are never written

- Kind: regression
- File: `scripts/runtime-postbuild.mjs`
- Line: 191
- Evidence: `LEGACY_ROOT_RUNTIME_COMPAT_ALIASES` maps old root chunks like `provider-dispatcher-6EQEtc-t.js` to `provider-dispatcher.js`, but `writeLegacyRootRuntimeCompatAliases` skips each alias unless `dist/provider-dispatcher.js` exists. A live build plus postbuild emitted `dist/auto-reply/reply/provider-dispatcher.js` and `dist/provider-dispatcher-k2AktooJ.js`, but left `dist/provider-dispatcher.js`, `dist/provider-dispatcher-6EQEtc-t.js`, `dist/provider-dispatcher-BpL2E92x.js`, and `dist/provider-dispatcher-JG96SkLX.js` missing.
- Impact: already-running gateways that still resolve one of those stale lazy chunks after a package/dist replacement can still hit module-not-found. That is the exact update/runtime break this commit claims to fix.
- Suggested fix: emit a root stable dispatcher entry, for example `provider-dispatcher: "src/auto-reply/reply/provider-dispatcher.ts"`, or make the legacy aliases point to the nested stable file with a valid relative export. Add an artifact-level test that runs the real tsdown config plus `runtime-postbuild` and asserts the expected stale root chunk files exist.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: run `node scripts/tsdown-build.mjs && node scripts/runtime-postbuild.mjs`, then check for `dist/provider-dispatcher-6EQEtc-t.js`. It is missing while `dist/auto-reply/reply/provider-dispatcher.js` exists.
- Is this the best way to solve the issue? no. The intent is sound, but the stable entry path and alias target are misaligned with real build output.
