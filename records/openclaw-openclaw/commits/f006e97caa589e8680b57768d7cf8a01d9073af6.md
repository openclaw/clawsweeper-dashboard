Found one low-severity test regression: the new exact `~/` assertion hard-codes `HOME`, but the production path resolver intentionally lets `OPENCLAW_HOME` override `HOME`.

## Findings

### Low: `~/` image parser test fails when `OPENCLAW_HOME` is set

- Kind: reliability
- File: `src/agents/pi-embedded-runner/run/images.test.ts`
- Line: 76
- Evidence: The test now expects `resolved: path.join(process.env.HOME ?? os.homedir(), ...)`, but `detectImageReferences` resolves `~/...` through `resolveUserPath` in `src/agents/pi-embedded-runner/run/images.ts:258`. That resolver uses `resolveRequiredHomeDir`, whose contract prefers `OPENCLAW_HOME` over `HOME` in `src/infra/home-dir.ts:24`.
- Impact: A normal test run with an ambient `OPENCLAW_HOME` now fails even though production behavior is correct. Reproduced with `OPENCLAW_HOME=/tmp/openclaw-review-home pnpm test src/agents/pi-embedded-runner/run/images.test.ts -- --reporter=dot`; the test expected the runner temp `HOME` path but received `/tmp/openclaw-review-home/Pictures/vacation.png`.
- Suggested fix: Make the expectation use the same contract as production, for example `resolveUserPath("~/Pictures/vacation.png")`, or explicitly clear/stub `OPENCLAW_HOME` in this test if it is meant to verify OS-home behavior only.
- Confidence: high

## Details
