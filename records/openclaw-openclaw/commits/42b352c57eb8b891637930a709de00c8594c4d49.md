Found one low-severity catalog completeness issue: the new GitHub Copilot manifest is marked static-authoritative, but omits GPT-5.5, which GitHub’s current supported-models documentation lists as available.

## Findings

### Low: Static Copilot catalog omits GPT-5.5

- Kind: bug
- File: `extensions/github-copilot/openclaw.plugin.json`
- Line: 177
- Evidence: The manifest’s static catalog lists GPT-5.4 nano and then moves to Grok/Raptor/Goldeneye, with no GPT-5.5 row. The same omission exists in `extensions/github-copilot/models-defaults.ts:10`. Because `modelCatalog.discovery.github-copilot` is `"static"`, `models list --all --provider github-copilot` uses the manifest source plan and does not fall back to runtime/provider-index rows once static rows exist (`src/commands/models/list.source-plan.ts:71`). A focused TS probe returned `{"count":21,"hasGpt55":false}` for `loadStaticManifestCatalogRowsForList({ providerFilter: "github-copilot" })`. GitHub’s supported Copilot models page lists GPT-5.5 alongside GPT-5.4/GPT-5.4 mini/GPT-5.4 nano, while also noting model availability is plan/client dependent and subject to change. ([docs.github.com](https://docs.github.com/en/copilot/reference/ai-models/supported-models))
- Impact: Users asking OpenClaw for the authoritative GitHub Copilot model list will not see GPT-5.5, even though the new static catalog is intended to publish Copilot’s available model catalog. Manual dynamic use may still work, but listing/picker surfaces will be stale.
- Suggested fix: Add a `gpt-5.5` manifest row and keep `models-defaults.ts` in sync, or mark the catalog non-authoritative until it can include all supported current rows.
- Confidence: medium
