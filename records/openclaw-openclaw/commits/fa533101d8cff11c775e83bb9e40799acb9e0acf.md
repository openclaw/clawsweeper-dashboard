# Commit fa53310

## Summary

Found one reliability regression in the new post-global-update doctor path.

## Findings

### Medium: Doctor failure after package swap suppresses the required update restart

- Kind: reliability
- File: `src/infra/update-runner.ts`
- Line: 1447
- Evidence: `runGlobalPackageUpdateSteps` runs `postVerifyStep` after verification and after the staged npm `global install swap`; the new hook returns an `"openclaw doctor"` step. If that step exits nonzero, `runGatewayUpdate` returns `status: "error"` / `reason: "doctor-failed"`. `src/gateway/server-methods/update.ts:151` only schedules the SIGUSR1 restart when `result.status === "ok"`.
- Impact: a Control UI global package update can successfully replace the on-disk package tree, then fail doctor, then skip restart. That leaves the old Gateway process running against a replaced `dist` tree. The repo docs for package-manager `update.run` explicitly rely on immediate no-deferral restart after package swap because the old process may lazy-load removed chunks.
- Suggested fix: avoid treating a post-swap doctor failure as a plain update failure that blocks restart. Safer options are running doctor before the npm staged swap when possible, or returning a distinct partial-success/package-swapped result that still schedules the update restart while surfacing the doctor failure.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The added test covers the first half: doctor failure makes `runGatewayUpdate` return `status: "error"` / `reason: "doctor-failed"`. The update handler then deterministically skips restart for any non-`ok` result.
- Is this the best way to solve the issue? unclear. The intended config migration before restart is valid, but running a potentially failing repair after the package tree is already swapped changes the restart safety contract.
