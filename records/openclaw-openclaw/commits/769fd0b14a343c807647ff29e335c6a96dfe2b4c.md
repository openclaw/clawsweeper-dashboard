# Commit 769fd0b

## Summary

Found one rollback consistency issue. The new failure path restores the tracked git checkout, but it does not restore ignored dependency state after `pnpm install`/equivalent has already run.

## Findings

### Medium: Failed post-install updates can roll code back while leaving candidate dependencies installed

- Kind: reliability
- File: `src/infra/update-runner.ts`
- Line: 808
- Evidence: `rollbackGitCheckout()` only runs git commands: `reset --hard`, `checkout`, and another `reset --hard` to `beforeSha`. The rollback is used after `deps install`, `build`, `ui:build`, doctor, and UI repair failures. In the build/UI/doctor failure cases, the package-manager install has already succeeded in the main checkout, but rollback never re-runs the package manager after returning to `beforeSha`; `node_modules/` is ignored by git, so `git reset --hard` cannot restore it.
- Impact: A failed update can leave the repository files at the previous commit while the dependency tree remains partially or fully converged to the failed candidate. If the candidate changed, removed, or upgraded runtime dependencies, the rolled-back OpenClaw checkout can restart or rerun with a mismatched `node_modules`, causing crashes or follow-up failures despite the rollback reporting the old checkout.
- Suggested fix: Make rollback converge dependencies for the restored checkout, or avoid mutating the live checkout until validation succeeds. A bounded fix would run the resolved package manager install after successfully checking out `beforeSha` and record that as a rollback step; a stronger fix is to validate/install in a disposable worktree and only swap the live checkout after all post-checkout checks pass.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. Use a git-backed install where the candidate commit changes package dependencies, let `deps install` succeed, force `build` or `ui:build` to fail, then observe that `HEAD` is reset to `beforeSha` while ignored `node_modules/` remains outside git rollback.
- Is this the best way to solve the issue? No. Git-only rollback is incomplete for failures after dependency installation; dependency convergence or an isolated update workspace is needed to make the rollback operationally consistent.
