# Commit 72f50dd

## Summary

The commit fixes Slack message reads by converting `before` / `after` ISO timestamps into Slack `latest` / `oldest` Unix timestamp strings and adds focused Slack regression tests. I found one low-severity precision edge case in the new ISO parser.

## Findings

### Low: ISO fractional seconds beyond milliseconds are accepted but silently truncated

- Kind: bug
- File: `extensions/slack/src/actions.ts`
- Line: 84
- Evidence: `z.iso.datetime({ offset: true })` accepts timestamps such as `2024-04-05T12:34:56.123456Z`, but `normalizeSlackReadTimestamp` then uses `Date.parse` and `formatEpochSeconds`, which formats only millisecond precision via `seconds.toFixed(3)`. A focused probe showed Zod accepts that value while `Date.parse` returns `1712320496123`, so the Slack bound becomes `1712320496.123` instead of preserving `.123456`.
- Impact: Slack message timestamps are microsecond-style strings. For `before`, this can exclude messages between `.123000` and `.123456` that should match; for `after`, it can include messages in that same sub-millisecond range that should not match. Most human-entered ISO timestamps use milliseconds or whole seconds, so this is narrow.
- Suggested fix: Either reject ISO inputs with more than 3 fractional-second digits, or parse and preserve up to Slack-compatible fractional precision instead of relying on `Date.parse`.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the parser behavior: local Zod/Date probes show accepted six-digit fractional ISO input is reduced to milliseconds before Slack-bound formatting. I did not run a live Slack workspace call for a sub-millisecond boundary.
- Is this the best way to solve the issue? Unclear. Rejection is simpler and safer if OpenClaw only wants JS `Date` precision; preserving microseconds is better if â€śISO-8601 date stringâ€ť is intended literally.
