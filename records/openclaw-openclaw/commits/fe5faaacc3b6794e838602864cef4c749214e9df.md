# Commit fe5faaa

## Summary

Found one concrete regression in channel account promotion. The helper now ignores the channel-owned promotion contract and applies Matrix-like targeting to every channel, which can overwrite an existing named account when adding another account.

## Findings

### Medium: Single-account promotion can overwrite the sole named account and ignore plugin-owned promotion keys

- Kind: regression
- File: `src/channels/plugins/setup-helpers.ts`
- Line: 524
- Evidence: this commit replaced the plugin-aware `setup-promotion-helpers` import with local static `resolveSingleAccountKeysToMove` / `resolveSingleAccountPromotionTarget` functions. The local target resolver no longer receives `channelKey` or checks the loaded/bundled channel setup contract, and it returns the sole named account for any channel. A focused smoke on current `main` with `discord: { token: "legacy-default-token", accounts: { work: { token: "work-token" } } }` produced `accounts.work.token = "legacy-default-token"` and no `accounts.default`, overwriting the named account. A second smoke with a loaded `custom-chat` plugin declaring `setup.singleAccountKeysToMove = ["customAuth"]` left `customAuth` at the channel root, proving the plugin contract is ignored. The user path is `src/commands/channels/add.ts:378`, which calls this helper before adding a non-default account.
- Impact: adding a non-default account to a mixed legacy/multi-account config can corrupt an existing named account’s credentials/settings. Third-party or future bundled channel plugins that rely on `singleAccountKeysToMove`, `namedAccountPromotionKeys`, or `resolveSingleAccountPromotionTarget` also lose their documented promotion behavior.
- Suggested fix: route `moveSingleAccountChannelSectionToDefaultAccount` back through `setup-promotion-helpers` or otherwise pass `channelKey` and honor the loaded/bundled setup promotion surface. The default fallback should remain `accounts.default`; only channel-owned promotion hooks should target a sole named account. Add regression tests for a non-Matrix sole named account and a custom plugin promotion key.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Run `moveSingleAccountChannelSectionToDefaultAccount` with a non-Matrix channel containing top-level `token` and one existing named account; the named account is overwritten. Also reproducible with a synthetic loaded channel plugin whose `singleAccountKeysToMove` contains a custom key.
- Is this the best way to solve the issue? yes: restore the existing plugin-aware promotion helper path and keep static fallback behavior narrow.
