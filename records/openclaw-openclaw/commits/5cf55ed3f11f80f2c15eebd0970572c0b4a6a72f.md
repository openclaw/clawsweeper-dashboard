# Commit 5cf55ed

## Summary

Found one low-severity compatibility risk in the new suppression metadata: the added `openai-codex` suppressions are unconditional even though the stated failure is specific to ChatGPT/Codex OAuth accounts.

## Findings

### Low: Stale-model suppressions also block explicit custom `openai-codex` endpoints

- Kind: compatibility
- File: `extensions/openai/openclaw.plugin.json`
- Line: 729
- Evidence: The new `openai-codex/gpt-5.2` and adjacent 5.1/5.3 suppression entries have no `when` clause, so core treats them as absolute suppressions. `src/agents/pi-embedded-runner/model.ts:648` blocks even explicit inline provider models with `api` when `shouldUnconditionallySuppress()` matches, and `src/config/validation.ts:1151` turns every no-`when` suppression into a config error for matching model refs. Existing runner tests show `models.providers.openai-codex.baseUrl` custom endpoints are supported for this provider, so a previously explicit custom `openai-codex` endpoint serving one of these ids can no longer be configured or resolved.
- Impact: Operators using an OpenAI-Codex-compatible custom endpoint under the `openai-codex` provider lose the documented explicit-override path, even if their endpoint is not a ChatGPT/Codex OAuth account and still supports the model id.
- Suggested fix: Scope these suppressions to the default ChatGPT/Codex OAuth route, or add a condition that preserves explicitly configured custom provider models/base URLs. If manifest `when.baseUrlHosts` is used, make sure the default `openai-codex` backend still suppresses when no user `baseUrl` is configured.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? No live endpoint was available, but the source path is concrete: configure `models.providers.openai-codex` with a custom `baseUrl`, explicit `models: [{ id: "gpt-5.2", api: "openai-codex-responses" }]`, and `agents.defaults.model.primary: "openai-codex/gpt-5.2"`. Current validation will reject the ref via the unconditional manifest suppression, and runtime resolution will return `suppressed` before using the explicit inline model.
- Is this the best way to solve the issue? Unclear. Global suppression is appropriate for the reported ChatGPT/Codex OAuth entitlement failure, but the safer fix is to keep that block account/backend-scoped so explicit custom endpoint overrides retain their existing behavior.
