# Commit 1ed50b0

## Summary

Found one low-severity regression in the new queue outcome reporting: reply-run failures are still collapsed to a boolean, so active reply runs can be reported as `no_active_run`.

## Findings

### Low: Active reply-run queue failures are mislabeled as no active run

- Kind: bug
- File: `src/agents/pi-embedded-runner/runs.ts`
- Line: 121
- Evidence: `queueEmbeddedPiMessageWithOutcome()` falls back to `queueReplyRunMessage(sessionId, text)`, but `queueReplyRunMessage()` only returns `false` for several distinct active states: operation not `running`, backend missing `queueMessage`, or backend not streaming (`src/auto-reply/reply/reply-run-registry.ts:469`). That single `false` is then always converted to `reason: "no_active_run"` at `src/agents/pi-embedded-runner/runs.ts:131`. A one-off probe with an active running reply operation whose backend `isStreaming()` returns false produced `{"queued":false,"sessionId":"reply-session-1","reason":"no_active_run","gatewayHealth":"live"}`.
- Impact: `/steer` and subagent completion wake diagnostics can tell users/operators that the session has no active run when it actually has an active reply run that is queued, compacting, missing a queue hook, or not currently streaming. This defeats the commitâ€™s stated goal of exposing useful queue failure reasons and can hide compacting/not-streaming states.
- Suggested fix: expose a structured reply-run queue outcome from `reply-run-registry`, then map reply operation phases/backend state into the same reason contract before falling back to `no_active_run`. For example, preflight/memory-flushing compacting phases should not become `no_active_run`; running-but-not-streaming should map to `not_streaming` or a more precise additive reason.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Create an active reply operation, set it to `running`, attach a backend with `queueMessage` and `isStreaming() => false`, then call `queueEmbeddedPiMessageWithOutcome(sessionId, "hello")`; it returns `reason:"no_active_run"` even though the reply operation is active.
- Is this the best way to solve the issue? yes. The loss happens at the boolean reply-run boundary, so preserving reason at that boundary is safer than guessing after `queueReplyRunMessage()` returns `false`.
