# Commit ec69c07b27

## Summary

Found one regression in the Twilio notify path: the commit now sends the notify message as direct create-call TwiML, but the manager still replays the same `initialMessage` when Twilio reports the call as answered.

## Findings

### Medium: Twilio notify calls can replay the direct TwiML message on answer

- Kind: regression
- File: `extensions/voice-call/src/providers/twilio.ts`
- Line: 567
- Evidence: `initiateCall` now sends `params.Twiml = input.inlineTwiml` for notify calls while still registering `StatusCallbackEvent: ["initiated", "ringing", "answered", "completed"]` at `extensions/voice-call/src/providers/twilio.ts:562`. Twilio documents that create-call `twiml` is executed without fetching `url`, and that `StatusCallbackEvent` causes requests to the supplied callback URL. ([twilio.com](https://www.twilio.com/docs/voice/api/call-resource)) The generated notify TwiML already contains `<Say>` and `<Hangup/>` at `extensions/voice-call/src/manager/twiml.ts:3`, but `maybeSpeakInitialMessageOnAnswered` still calls `speakInitialMessage` for `mode === "notify"` at `extensions/voice-call/src/manager.ts:333`, which then calls `speak` and schedules notify hangup at `extensions/voice-call/src/manager/outbound.ts:333`.
- Impact: a one-shot Twilio notify call can speak the message from the create-call `Twiml`, then issue a second live call update with another `<Say>` when the answered status callback is processed. Depending on Twilio timing, this can duplicate the spoken notification or replace the direct `<Hangup/>`, keeping the call open until OpenClaw's delayed auto-hangup.
- Suggested fix: mark Twilio direct-notify calls as provider-delivered and skip `speakInitialMessage` for that mode, or clear `metadata.initialMessage` after successful Twilio direct-`Twiml` initiation while still letting status callbacks update call state.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused local smoke using a mocked `TwilioProvider.apiRequest` initiated a notify call, then injected a `call.answered` event. It recorded `POST /Calls.json` with direct `Twiml`, followed by `POST /Calls/CA123.json` with another `<Say>` TwiML update.
- Is this the best way to solve the issue? yes. The direct TwiML path is the right owner for one-shot notify audio, so the safer fix is to prevent the generic answered hook from replaying the same initial message for Twilio direct-notify calls.
