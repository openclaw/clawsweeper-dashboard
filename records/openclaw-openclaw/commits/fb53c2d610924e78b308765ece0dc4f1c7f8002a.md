# Commit fb53c2d

## Summary

Found one low-severity false positive in the new doctor diagnostic: current bundled skill paths that were compacted to `~` can be reported as stale when `OPENCLAW_HOME` differs from the OS home.

## Findings

### Low: Current compacted bundled skill paths can be flagged as stale with `OPENCLAW_HOME`

- Kind: bug
- File: `src/commands/doctor-session-snapshots.ts`
- Line: 174
- Evidence: `resolveExpectedBundledSkillPath()` expands `~` with `expandHomePrefix(... env ...)`, which uses the effective OpenClaw home. But skill prompt compaction can replace either the effective OpenClaw home, OS home, or native home with `~` in `src/agents/skills/workspace.ts:56`. In a setup where `OPENCLAW_HOME=/state` and the OpenClaw package is installed under the OS home, a current prompt path like `~/.bun/install/global/node_modules/openclaw/skills/doctor/SKILL.md` expands to `/state/.bun/...`, fails the `isInsidePath(bundledSkillsDir, expandedCachedPath)` current-root suppression, then gets reported as stale even though `expectedPath` is the same current bundled skill file.
- Impact: `openclaw doctor` can warn users to start fresh sessions or reset long-lived sessions even when cached metadata already points at the live bundled skills root. This specifically violates the commit’s intended “current runtime paths quiet” behavior for users who set `OPENCLAW_HOME` separately from their OS home.
- Suggested fix: For `~` cached paths, try the same home candidates used by skill compaction, or at least effective home plus OS home, before classifying. Suppress the finding if any expansion is inside `bundledSkillsDir`; only report stale when the compacted path cannot represent the current bundled root.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. I ran a focused `node --import tsx -e` probe against `scanSessionStoreForStaleRuntimeSnapshotPaths()` with `OPENCLAW_HOME` and `HOME` set to different roots, `bundledSkillsDir` under `HOME`, and a cached current `~/.bun/.../node_modules/openclaw/skills/doctor/SKILL.md`; the scanner returned a stale finding.
- Is this the best way to solve the issue? yes. Matching the scanner’s `~` expansion candidates to the prompt compaction contract preserves the intended stale-path detection while preventing this false positive.
