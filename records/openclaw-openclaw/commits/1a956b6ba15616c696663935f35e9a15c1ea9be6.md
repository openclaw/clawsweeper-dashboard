# Commit 1a956b6

## Summary

Found one medium reliability regression in the fallback path for generated media completions. The new direct handoff correctly requires `message_tool_only`, but a later fallback steer can still requeue the same completion without that delivery mode and mark it delivered.

## Findings

### Medium: Failed message-tool handoff can fall back to an unguarded active-run steer

- Kind: regression
- File: `src/agents/subagent-announce-delivery.ts`
- Line: 468
- Evidence: The direct completion path now sets `sourceReplyDeliveryMode: "message_tool_only"` for generated music/video completions, including the active-run wake at lines 648-660. If that wake fails with `source_reply_delivery_mode_mismatch`, it falls through to the forced announce-agent handoff. When that handoff returns no message-tool evidence, `runSubagentAnnounceDispatch` still runs the generic `steer-fallback`; `maybeSteerSubagentAnnounce` queues with only `{ steeringMode: "all", debounceMs }` and no `sourceReplyDeliveryMode`, so an active requester run that rejected the guarded wake can accept the unguarded fallback.
- Impact: A generated media completion can be reported as delivered even though the message-tool-only announce agent failed. The completion is then handed to the active requester under its old automatic reply contract, which undermines this commit’s “do not auto-post / require message tool” behavior and can either lose the generated media or let the active run deliver it through the path the commit was trying to prevent.
- Suggested fix: Carry the required `sourceReplyDeliveryMode` into the fallback steer path, or skip fallback steering when the direct generated-media handoff failed specifically because message-tool delivery was missing. The guarded mode should be applied consistently to both primary and fallback queue attempts.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. I ran an in-process probe where the primary active wake rejects `message_tool_only` with `source_reply_delivery_mode_mismatch`, the forced announce-agent call returns `{ result: { payloads: [] } }`, and the fallback steer then queues successfully without `sourceReplyDeliveryMode`. The final result was `delivered: true`, `path: "steered"`, with phases showing the direct failure `"completion agent did not deliver through the message tool"` followed by successful fallback steer.
- Is this the best way to solve the issue? yes. The contract is already represented by `sourceReplyDeliveryMode`; the bug is that one queue path drops it. Propagating or enforcing that same mode is narrower than adding another delivery heuristic.
