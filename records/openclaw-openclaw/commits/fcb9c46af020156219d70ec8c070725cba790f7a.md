# Commit fcb9c46af0

## Summary

Found one CI regression in the live-test image preparation path. The new retry helper is called from the selected ref in one job, so validating older refs that do not contain the newly added script now fails before GHCR login.

## Findings

### Medium: Live-suite validation of older refs now fails before GHCR login

- Kind: regression
- File: `.github/workflows/openclaw-live-and-e2e-checks-reusable.yml`
- Line: 1584
- Evidence: The reusable workflow accepts a `ref` input described as `Ref, tag, or SHA to validate` and `validate_selected_ref` allows commits reachable from `origin/main` or release tags. `prepare_live_test_image` checks out that selected ref at lines 1565-1569, then runs `bash scripts/ci-docker-login-ghcr.sh` at line 1584. The new script does not exist in the parent/base commit: `git cat-file -e d42bc0b68438b8632363e69f646f25291ddadc03:scripts/ci-docker-login-ghcr.sh` fails with “path ... not in” that commit. In contrast, the other changed login call sites use `.release-harness/scripts/ci-docker-login-ghcr.sh`, which comes from the trusted workflow checkout.
- Impact: Dispatching this workflow to validate an older main ancestor or release tag with live suites enabled can fail immediately with `bash: scripts/ci-docker-login-ghcr.sh: No such file or directory`, even though that ref is explicitly allowed by the workflow’s trust check. This blocks historical release/live validation rather than retrying GHCR login.
- Suggested fix: In `prepare_live_test_image`, either add the same trusted `.release-harness` checkout used by the neighboring Docker jobs and run `.release-harness/scripts/ci-docker-login-ghcr.sh`, or keep this one call on `docker/login-action` so it does not depend on files in the selected ref.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Run the reusable workflow from current `main` with `ref` set to `d42bc0b68438b8632363e69f646f25291ddadc03` and live suites enabled; `prepare_live_test_image` checks out that ref, where `scripts/ci-docker-login-ghcr.sh` is absent, then calls it.
- Is this the best way to solve the issue? yes. The workflow already uses `.release-harness` for trusted harness scripts in adjacent jobs, so using the same path here preserves old-ref validation without reintroducing the un-retried login action.
