# Commit 2ab3a4e

## Summary

Found one concrete gap in the new heartbeat span filter: response-tool heartbeat prompts generated for due `HEARTBEAT.md` tasks are not recognized as heartbeat prompts, so their silent `heartbeat_respond notify=false` artifacts still replay into later embedded-runner context.

## Findings

### Medium: Task-based response-tool heartbeat spans are not filtered

- Kind: bug
- File: `src/auto-reply/heartbeat-filter.ts`
- Line: 327
- Evidence: The new recognizer only treats task prompts as heartbeat prompts when they include `After completing all due tasks, reply HEARTBEAT_OK.`. But the production heartbeat runner emits a different task prompt when response-tool mode is active: `After completing all due tasks, use heartbeat_respond to report the outcome...` at `src/infra/heartbeat-runner.ts:1157`. A focused probe against current `main` returned `isHeartbeatUserMessage: false` and retained the whole span, including `heartbeat_respond`, for that exact prompt shape.
- Impact: Users with due heartbeat tasks and Codex/message-tool response mode can still carry silent task heartbeat prompts, `heartbeat_respond notify=false` calls, and tool results into later normal prompts/context snapshots. That is the same pollution class this commit is intended to remove, just through the task prompt path.
- Suggested fix: Teach `isHeartbeatUserMessage` to recognize the response-tool task completion instruction as a heartbeat task prompt, ideally by matching the task prefix plus either the old `HEARTBEAT_OK` instruction or the new `heartbeat_respond` instruction. Add a regression case with a due-task response-tool prompt and a silent `heartbeat_respond notify=false` span.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Use `filterHeartbeatTranscriptArtifacts` with a user message starting `Run the following periodic tasks...` and ending with the response-tool task instruction from `src/infra/heartbeat-runner.ts:1157`, followed by a silent `heartbeat_respond` tool call/result and then a real user turn. The current filter returns all four messages instead of dropping the heartbeat span.
- Is this the best way to solve the issue? yes. The production runner already has two task completion strings; the shared recognizer should accept both so the existing span filter can run unchanged.
