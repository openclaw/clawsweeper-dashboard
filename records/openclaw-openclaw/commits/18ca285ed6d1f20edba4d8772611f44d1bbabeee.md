# Commit 18ca285e

## Summary

Found one remaining peer-dependency contract bug in the managed npm root reconciliation path.

## Findings

### Medium: Later plugins can install successfully with an incompatible preserved peer

- Kind: bug
- File: `src/infra/npm-managed-root.ts`
- Line: 533
- Evidence: `collectManagedNpmRootPeerDependencyPins` records only the first seen peer name and skips later declarations for the same peer. `syncManagedNpmRootPeerDependencies` then preserves any existing dependency spec with `dependencies[packageName] ?? dependencySpec`, so a managed peer already pinned for an older plugin is never checked against a newly installed plugin’s peer range. The install path runs npm with `--legacy-peer-deps`, whose npm 11 local config definition says peer dependencies are completely ignored while building the tree, so npm does not reject the conflict.
- Impact: A plugin install/update can return success while the new plugin resolves an incompatible peer at runtime. I reproduced the npm side with local tarballs: plugin A required `peer-lib-conflict-demo@^1.0.0`, plugin B required `^2.0.0`, the managed root kept `peer-lib-conflict-demo@1.0.0`, `npm install --omit=dev --omit=peer --legacy-peer-deps` succeeded, `require()` from plugin B loaded `v1`, and `npm ls` reported `ELSPROBLEMS invalid`.
- Suggested fix: collect all non-optional peer ranges for each peer name and validate the installed/pinned version satisfies every requiring package before accepting the install. If the ranges conflict, rollback and return an actionable error. A final peer validation step should exclude the special `openclaw` host peer but should not rely on `legacy-peer-deps` to enforce third-party peers.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes; install one managed plugin whose required third-party peer resolves to v1, then install another managed plugin requiring the same peer at an incompatible v2 range. The current sync logic preserves the old root dependency and the legacy npm install accepts it.
- Is this the best way to solve the issue? yes; the managed root is intentionally bypassing npm peer enforcement, so OpenClaw needs its own validation before reporting success.
