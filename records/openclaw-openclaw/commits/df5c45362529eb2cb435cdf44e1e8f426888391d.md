## Summary

Found one actionable issue: the change allowlists `@whiskeysockets/libsignal-node` for build scripts, but the reported pnpm failure is an exotic-subdependency resolver rejection, not a blocked-build-script rejection. The stated fix does not unblock installs when `block-exotic-subdeps=true`.

## Findings

### Medium: `onlyBuiltDependencies` does not fix `ERR_PNPM_EXOTIC_SUBDEP`

- Kind: bug
- File: `package.json:1793` and `pnpm-workspace.yaml:43`
- Evidence: The changelog says adding `@whiskeysockets/libsignal-node` to `onlyBuiltDependencies` makes pnpm stop rejecting BaileysРђЎ git-tarball subdependency. pnpm 10.33.2РђЎs resolver throws `ERR_PNPM_EXOTIC_SUBDEP` from the exotic-subdependency check when `blockExoticSubdeps` is enabled; `onlyBuiltDependencies` is only consulted by the build-script allow function. A throwaway minimal install with `@whiskeysockets/baileys@7.0.0-rc.9` and the same `onlyBuiltDependencies` entries still failed under `pnpm install --lockfile-only --config.block-exotic-subdeps=true` with `ERR_PNPM_EXOTIC_SUBDEP Exotic dependency "libsignal" ... is not allowed`. The same install succeeded when `block-exotic-subdeps=false`.
- Impact: Any environment that actually has `block-exotic-subdeps=true` still cannot resolve the Baileys `libsignal` git subdependency, so the release note claims a fix that does not work for the stated failure mode. Impact is narrower than the changelog text implies because current pnpm 10.33.2 defaults this setting to false and current main excludes WhatsApp from the core package payload.
- Suggested fix: Do not rely on `onlyBuiltDependencies` for this resolver error. Either set/force project-level `block-exotic-subdeps=false` after accepting that trust tradeoff, replace or patch the Baileys `libsignal` dependency so it is not an exotic subdependency, or update the changelog/issue resolution to reflect that current main avoids the packaged-core path rather than fixing `blockExoticSubdeps`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A minimal package depending on `@whiskeysockets/baileys@7.0.0-rc.9` with `@whiskeysockets/baileys` and `@whiskeysockets/libsignal-node` in `pnpm.onlyBuiltDependencies` still fails with pnpm 10.33.2 when run with `--config.block-exotic-subdeps=true`.
- Is this the best way to solve the issue? no. The commit changes the build-script allowlist, but the concrete failure happens during dependency resolution before build-script execution matters.
