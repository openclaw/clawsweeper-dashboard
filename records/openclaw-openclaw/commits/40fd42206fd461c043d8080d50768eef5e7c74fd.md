# Commit 40fd422

## Summary

Found one current, actionable regression: compact chat-window prompt context is no longer stripped from user-visible chat history.

## Findings

### Medium: Compact chat-window context leaks into displayed user messages

- Kind: privacy
- File: `src/auto-reply/reply/inbound-meta.ts`
- Line: 441
- Evidence: The commit adds `formatChatWindowStructuredContext()` and pushes its plain-text output into the inbound user prompt. The UI stripping code in `src/auto-reply/reply/strip-inbound-meta.ts` only recognizes fixed fenced-JSON sentinels from `INBOUND_META_SENTINELS`, and the Swift `ChatMarkdownPreprocessor` has the same fixed-header model. The new dynamic headers like `Current local chat window (untrusted, chronological, before current message):` are not recognized, so they survive stripping.
- Impact: AI-facing nearby chat transcript context can appear in TUI/Web UI/macOS/session-history surfaces as if it were part of the user's message, including previous local chat text that was only meant for model context.
- Suggested fix: Add a stable non-user-controlled sentinel for compact chat-window blocks and update both TS and Swift stripping logic/tests to remove it, or keep chat-window context inside an existing fenced JSON metadata block that current strippers already understand.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused local check building a `chat_window` prefix and passing it through `stripInboundMetadata` returned: `"Current local chat window (untrusted, chronological, before current message):\n#34273 Sam: private previous text\n\nActual user message"`.
- Is this the best way to solve the issue? yes: the compact block needs a stripping contract. Field sanitization already landed later on `main`, but that does not make the AI-facing block user-visible safe.
