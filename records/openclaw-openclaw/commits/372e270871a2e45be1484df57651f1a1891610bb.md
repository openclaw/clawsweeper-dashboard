# Commit 372e270

## Summary

The commit fixes the reported false-positive send telemetry for empty outbound results, but it also makes every terminal no-send path look retryable to the main-session pending-final-delivery cleanup. Documented `message_sending` cancellations return `[]` without an adapter result, so the reviewed change can leave an intentionally blocked final reply stuck as pending.

## Findings

### Medium: Terminal hook cancellations now keep final delivery pending

- Kind: reliability
- File: `src/agents/command/delivery.ts`
- Line: 398
- Evidence: The changed line now sets `deliverySucceeded = deliveryResults.length > 0 && !deliveryHadError`. But `deliverOutboundPayloads` also returns `[]` for documented terminal no-send paths: `message_sending` hooks with `{ cancel: true }` short-circuit at `src/infra/outbound/deliver.ts:1404`, and repo docs state `message_sending` cancellation is terminal. The caller in `src/agents/agent-command.ts:1299` only clears `pendingFinalDelivery` when `deliverySucceeded === true`, so an intentional hook cancellation leaves the durable pending marker and pending text in the session.
- Impact: A main-session `openclaw agent --deliver` final reply that a policy hook intentionally cancels can remain marked for recovery/retry instead of being treated as a completed terminal no-send. That can cause stale pending-final-delivery state, repeated recovery attempts after restart, short heartbeat deferrals, and longer retention of final reply text that the hook explicitly blocked from delivery.
- Suggested fix: Split “sent” from “handled/terminal”. Keep `deliverySucceeded` false for telemetry when no platform send happened, but return another terminal outcome from `deliverAgentCommandResult` or `deliverOutboundPayloads` so `agent-command` can clear pending final delivery for hook-cancelled or otherwise handled no-send outcomes. Alternatively, have outbound delivery return a typed no-send result for hook cancellation instead of a bare empty array.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes at source level: install or mock a `message_sending` hook that returns `{ cancel: true }`, run a delivered main-session agent command with final text, and observe `deliverOutboundPayloads` return `[]`; after this commit line 398 reports false, so `src/agents/agent-command.ts:1299` does not clear the pending final delivery fields. I did not run a live channel delivery with a real plugin hook.
- Is this the best way to solve the issue? The reviewed change is the right direction for #78532’s claim-conflict/no-adapter false positive, but it needs a separate terminal no-send signal. Reverting to `!deliveryHadError` would reintroduce the original bug; overloading an empty result as retryable failure creates the new edge case.
