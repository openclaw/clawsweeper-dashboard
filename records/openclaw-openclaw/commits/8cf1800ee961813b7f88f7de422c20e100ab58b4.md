# Commit 8cf1800ee9

## Summary

Found one continuity regression in the new transient no-tool thread path. Existing focused tests pass, but they only assert that the original binding is preserved, not that the next resumed native thread sees the transient turn that happened on the side thread.

## Findings

### Medium: Tool-enabled resume can skip the transient no-tool turn it just preserved around

- Kind: regression
- File: `extensions/codex/src/app-server/run-attempt.ts`
- Line: 1576
- Evidence: `startOrResumeThread` preserves the old binding when a previously tool-enabled binding is followed by an empty dynamic-tool fingerprint, starts a transient `thread/start`, and deliberately skips writing that new thread binding in `extensions/codex/src/app-server/thread-lifecycle.ts:61` and `extensions/codex/src/app-server/thread-lifecycle.ts:162`. That transient run is mirrored only into the OpenClaw session file by `mirrorCodexAppServerTranscript`, not into the preserved native Codex thread. On the next tool-enabled run, `shouldProjectMirroredHistoryForCodexStart` returns false when the preserved binding fingerprint again matches the restored tool fingerprint, so `turn/start` sends only the current prompt while resuming the old native thread.
- Impact: a sequence of tool-enabled turn -> no-tool/maintenance turn -> tool-enabled turn can make Codex resume the original native thread without the intervening no-tool turn. If the no-tool turn produced user-visible decisions, summaries, or instructions, the next user follow-up can lose exactly that recent context.
- Suggested fix: record that a transient no-tool turn occurred, for example on the preserved binding as a `needsMirroredHistoryProjection`/watermark marker, and force the next compatible resume to project the mirrored history delta or full mirrored context once before clearing the marker. Alternatively start a new native thread with projected history after any transient turn that appended transcript messages.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes from the code path: create a session binding with a non-empty dynamic-tool fingerprint, run the same session with no dynamic tools so the transient branch preserves the old binding, complete that turn so `mirrorCodexAppServerTranscript` appends it locally, then run again with the original tools. The second tool-enabled run resumes the preserved old thread and does not project the mirrored transient turn.
- Is this the best way to solve the issue? Unclear. Preserving the native binding is a reasonable fix for the original bug, but it needs an explicit ŌĆ£native thread missed local transcriptŌĆØ marker or equivalent projection gate so preservation does not create a hidden fork.
