# Commit f3fd0ee

## Summary

Found two reliability gaps in the memory-core atomic swap path. The new rename retry helps the reported main-file `EBUSY` case, but the swap can still strand the live SQLite file when a sidecar rename fails, and cleanup still does not retry the transient Windows delete/unlink failures reported in #64187.

## Findings

### Medium: Sidecar rename failure during backup can leave the live DB path missing

- Kind: reliability
- File: `extensions/memory-core/src/memory/manager-atomic-reindex.ts`
- Line: 81
- Evidence: `swapMemoryIndexFiles()` calls `await moveMemoryIndexFiles(targetPath, backupPath)` before entering the restore `try` block. `moveMemoryIndexFiles()` renames suffixes sequentially: main DB, then `-wal`, then `-shm`. A focused injected check confirmed that if `index.sqlite` rename succeeds and `index.sqlite-wal` keeps throwing `EBUSY`, the helper exits after moving the main DB to backup with no compensating restore.
- Impact: On Windows WAL-mode stores, a persistent lock on `main.sqlite-wal` or `main.sqlite-shm` can leave `main.sqlite` moved to `main.sqlite.backup-...`, so subsequent memory reads/searches see the live DB path missing or inconsistent until manual repair.
- Suggested fix: Put the initial target-to-backup move under rollback handling, or make the grouped move helper track completed suffixes and restore them if a later suffix fails.
- Confidence: medium

### Medium: Cleanup still fails on transient Windows rm/unlink locks

- Kind: reliability
- File: `extensions/memory-core/src/memory/manager-atomic-reindex.ts`
- Line: 76
- Evidence: `removeMemoryIndexFiles()` still calls `fs.rm(path, { force: true })` directly for temp and backup files. Node’s local type docs note `fs.rm` retries only in recursive mode; this call is non-recursive, so `EBUSY`, `EPERM`, or `EACCES` from Windows file locks still propagate. Issue #64187 includes a reproduction comment where the failure was `unlink '...sqlite.tmp-<uuid>' EBUSY`.
- Impact: A memory reindex can still fail or leave accumulating `.tmp-*`/backup files after the rename phase, even though the commit now retries rename locks.
- Suggested fix: Add the same bounded transient retry behavior to cleanup deletes, preserving the original error when cleanup fails during an error path.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? No for live Windows behavior in this Linux checkout. The sidecar failure shape was reproduced with injected `moveMemoryIndexFiles()` file ops; the cleanup risk is based on current code, Node type docs, and the existing Windows issue report.
- Is this the best way to solve the issue? Unclear. Rename retry is the right narrow location, but the atomic move needs rollback coverage and cleanup needs retry coverage to address the full Windows lock surface.
