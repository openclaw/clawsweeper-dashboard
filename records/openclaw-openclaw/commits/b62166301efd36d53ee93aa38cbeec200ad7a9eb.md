# Commit b621663

## Summary

Found one concrete issue: the new Codex `message_tool_only` hint is only in the dynamic tool description, but existing bound Codex threads are explicitly reused when only dynamic tool descriptions change. That means the new hint is not delivered for resumed threads, which leaves the reported failure mode in place for existing sessions.

## Findings

### Medium: Resumed Codex threads can keep the stale generic message tool spec

- Kind: bug
- File: `extensions/codex/src/app-server/thread-lifecycle.ts`
- Line: 418
- Evidence: This commit appends the visible-reply instruction only to `message` tool `description` (`src/agents/tools/message-tool.ts:703`) and sends dynamic tools only on `thread/start` (`extensions/codex/src/app-server/thread-lifecycle.ts:309`). On resume, `buildThreadResumeParams` does not include `dynamicTools` (`extensions/codex/src/app-server/thread-lifecycle.ts:332`), and the compatibility fingerprint strips `description` before comparing dynamic tool catalogs (`extensions/codex/src/app-server/thread-lifecycle.ts:414`). Existing coverage also asserts that a bound Codex thread resumes when only dynamic tool descriptions change (`extensions/codex/src/app-server/run-attempt.test.ts:3317`).
- Impact: A user with an already-bound Codex thread can enter a `message_tool_only` direct-chat turn, but Codex may continue using the old generic `message` tool metadata from the existing thread. Since the root cause was that models missed the visible-reply requirement when it was absent from tool metadata, these resumed turns can still produce private final text instead of calling `message(action="send")`.
- Suggested fix: Treat this delivery-mode hint as semantic, not cosmetic. Add a stable non-description fingerprint input for the source reply delivery mode, or otherwise force a new Codex thread when the `message` tool visible-reply contract changes. Avoid including all descriptions blindly unless that is acceptable for prompt-cache/thread churn.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Start or bind a Codex app-server thread with the generic `message` tool, then run a later turn where `sourceReplyDeliveryMode` is `message_tool_only`; current source will take the `thread/resume` path because the only relevant dynamic tool change is description text.
- Is this the best way to solve the issue? unclear. The best fix depends on whether Codex app-server can refresh dynamic tool specs on resume; if not, invalidating only on the source-reply delivery contract is safer than making every description edit restart threads.
