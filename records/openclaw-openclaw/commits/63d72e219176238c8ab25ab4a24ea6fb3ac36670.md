# Commit 63d72e2

## Summary

Found one low-severity hardening gap: malformed Twilio media-stream JSON is wrapped, but the runtime still logs the wrapped error object, which includes the raw `SyntaxError` as `cause`.

## Findings

### Low: Runtime logging still exposes the raw JSON parser failure

- Kind: bug
- File: `extensions/voice-call/src/media-stream.ts`
- Line: 275
- Evidence: `parseTwilioMediaMessage` wraps `JSON.parse` failures with `new Error("Twilio media stream message was malformed JSON", { cause })` at `extensions/voice-call/src/media-stream.ts:124`, but the WebSocket handler catches and logs the entire error object at `extensions/voice-call/src/media-stream.ts:275`. A focused Node check of that exact logging shape prints `[cause]: SyntaxError: Expected property name or '}' in JSON...`, so the raw parser failure still reaches logs. The new test only checks the helper’s thrown type/cause and does not exercise the runtime logging path.
- Impact: The changelog says malformed media-stream JSON is reported “with an owned parser error instead of logging raw parser failures,” but unauthenticated malformed WebSocket frames still produce nested raw parser errors in logs. This is mostly a hardening/diagnostic contract miss, not a payload leak.
- Suggested fix: For this known parser error, log only the stable owned message in the WebSocket catch path, or introduce a small owned parse-error class/code and special-case that branch while preserving full error logging for non-parser processing failures.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Running the same `console.error("[MediaStream] Error processing message:", new Error("Twilio media stream message was malformed JSON", { cause }))` shape under Node prints the nested `[cause]: SyntaxError...`.
- Is this the best way to solve the issue? yes. The wrapper can keep `cause` for programmatic inspection, but the boundary log needs to avoid passing the full Error object for this expected malformed-input case.
