# Commit e650f893

## Summary

Found one low-severity test portability regression in the new archive fixture helper.

## Findings

### Low: New test helper fails when POSIX `find` is unavailable

- Kind: compatibility
- File: `src/plugins/install.test.ts`
- Line: 129
- Evidence: `listFindFlatRootArchiveEntries()` shells out to `find` and returns `null` on nonzero status, then `listFlatRootArchiveEntries()` falls back to `fs.readdirSync(pkgDir)` at line 125. The new test at line 694 asserts `fs.readdirSync` was not called. A focused local probe with `PATH` emptied reproduced the fallback path with `readDirCalls: 1`; the selected Vitest passes on Linux only when POSIX `find` is available.
- Impact: contributors or CI environments without GNU/POSIX `find`, including likely Windows-native shells where `find` may be missing or resolve to the Windows text-search command, will fail this test despite the install behavior being unchanged.
- Suggested fix: avoid depending on external POSIX `find` for a unit-test helper. Use a cross-platform child Node process to list direct entries, pass expected archive entries into `packToArchive`, or make the assertion explicitly skip when the external lister is unavailable.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Hide `find` from `PATH` and the helper falls back to `fs.readdirSync`, contradicting the new assertion.
- Is this the best way to solve the issue? yes. The test is trying to avoid current-process `fs.readdirSync` calls, not require a POSIX userland tool.
