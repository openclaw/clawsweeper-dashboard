# Commit e7c784f

## Summary

Found one low-severity privacy regression in the new active-process runtime snapshot.

## Findings

### Low: Active process snapshots expose live exec output to context engines

- Kind: privacy
- File: `src/agents/bash-process-references.ts`
- Line: 62
- Evidence: `listActiveProcessSessionReferences()` copies `session.tail` into `ActiveProcessSessionReference`, and compaction/after-turn runtime contexts pass those references into context-engine `compact()` / `maintain()` calls. The system prompt renderer only needs `sessionId`, `status`, `pid`, `cwd`, and `name`, so the live output tail is not necessary for the advertised recovery path.
- Impact: a background exec can emit secrets or sensitive file contents before the agent polls/logs it. This change can hand that output to context-engine plugins as runtime metadata even when it was never intentionally included in transcript context.
- Suggested fix: remove `tail` from `ActiveProcessSessionReference` for compaction/runtime-context handoff, or gate it behind an explicit consumer that proves it needs live output. Keep the prompt-facing recovery data to session id/status/pid/cwd/name/truncated.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes by source path: start a scoped background exec that writes sensitive output, then trigger compaction or after-turn context-engine maintenance; `activeProcessSessions[].tail` is populated from the process registry and delivered in runtime context.
- Is this the best way to solve the issue? Yes: session reattachment only needs identifiers and lightweight metadata, not process output.
