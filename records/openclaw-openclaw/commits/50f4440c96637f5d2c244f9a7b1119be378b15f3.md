# Commit 50f4440

## Summary

The commit hardens inline shell-wrapper approval, but the new guard still lets a bare `busybox` or `toybox` allowlist entry approve arbitrary shell applet payloads without evaluating the extracted `sh -c` command.

## Findings

### Medium: Shell multiplexer allowlist entries still bypass inline payload matching

- Kind: security
- File: `src/infra/exec-approvals-allowlist.ts`
- Line: 416
- Evidence: `matchExecutableAllowlistForSegment` only requires an argument-bound match when `allowlistTargetIsExecutionTarget` is true. For `busybox sh -c ...` and `toybox sh -c ...`, command resolution keeps the policy target as the multiplexer but execution target as `sh`, so the guard is skipped even though `inlineCommand` is present. A focused probe on current `main` showed `evaluateShellAllowlist` returns `allowlistSatisfied: true` for `busybox sh -c unlisted` with only `{ pattern: <busybox path> }`, and the segment is marked `allowlist`.
- Impact: A user or existing config that allowlists the multiplexer binary can still run arbitrary inline shell payloads through `busybox sh -c` or `toybox sh -c` without payload approval. This leaves a gap in the commit’s intended “inline shell wrapper payload matching” policy.
- Suggested fix: Treat shell-multiplexer policy targets that unwrap to shell wrapper execution targets like shell wrapper entries for this guard: require an `argPattern`, wildcard, script/payload allowlist match, or evaluate the extracted inline payload before accepting the bare multiplexer match. Add regression coverage for `busybox sh -c unlisted` and `toybox sh -c unlisted`.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes: create fake executable `busybox`, `toybox`, `sh`, and `unlisted` files in a temp PATH, then call `evaluateShellAllowlist({ command: "busybox sh -c unlisted", allowlist: [{ pattern: busyboxPath }], safeBins: new Set(), ... })`; current `main` returns `allowlistSatisfied: true`.
- Is this the best way to solve the issue? unclear: the code already intentionally keeps shell multiplexers as separate policy targets, so maintainers should decide whether manual bare multiplexer allowlists are meant to trust every applet. If not, the smallest fix is to extend this commit’s bound-argument/payload-evaluation guard to multiplexer-unwrapped shell invocations.
