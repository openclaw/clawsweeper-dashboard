# Commit 63ff924

## Summary

The commit widens `RunEmbeddedPiAgentParams.onAgentEvent` to allow async callbacks, but one native compaction path still fires that callback with `void`. Async compaction notice delivery can therefore race the rest of the run instead of being awaited.

## Findings

### Medium: Native compaction events still drop the async `onAgentEvent` promise

- Kind: regression
- File: `src/agents/pi-embedded-subscribe.handlers.compaction.ts`
- Line: 17
- Evidence: The reviewed commit changes `src/agents/pi-embedded-runner/run/params.ts:154` so `onAgentEvent` may return `Promise<void>`. `run/attempt.ts:2355` forwards that callback into `subscribeEmbeddedPiSession`, but native PI compaction handlers still call it as `void ctx.params.onAgentEvent?.(...)` at `src/agents/pi-embedded-subscribe.handlers.compaction.ts:17` and `src/agents/pi-embedded-subscribe.handlers.compaction.ts:87`. The auto-reply implementation is async and awaits notice delivery in `src/auto-reply/reply/agent-runner-execution.ts:1505` and `src/auto-reply/reply/agent-runner-execution.ts:1617`.
- Impact: For automatic PI compaction events, async user-visible compaction notices are not part of the event chain. A final reply can continue or complete before the compaction notice callback settles, and callback rejections from custom compaction handlers can become detached from the run. This undercuts the new async callback contract for the path most directly named by the commit.
- Suggested fix: Make the compaction start/end handlers return and await the `onAgentEvent` promise, letting `scheduleEvent` serialize those events, or explicitly collect these callback promises in the subscription’s pending delivery work before the run can finish.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused `tsx` smoke against `handleCompactionStart` with an async `onAgentEvent` printed `after-return,callback`, confirming the handler returns before the callback settles.
- Is this the best way to solve the issue? yes. The repo already uses `scheduleEvent` to preserve async event ordering when handlers return promises, so returning/awaiting compaction callback promises fits the existing design.
