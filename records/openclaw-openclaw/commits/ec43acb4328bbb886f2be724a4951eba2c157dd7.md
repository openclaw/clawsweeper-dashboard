# Commit ec43acb432

## Summary

The commit makes Microsoft Foundry DeepSeek V4 deployments default to `openai-responses`. Current Microsoft Foundry catalog/docs and DeepSeek’s own API docs still point these V4 models at chat completions, so fresh onboarding can persist an API adapter that calls `/responses` for models advertised as `chat-completion`.

## Findings

### Medium: DeepSeek V4 Foundry deployments are routed to the wrong API surface

- Kind: regression
- File: `extensions/microsoft-foundry/shared.ts`
- Line: 124
- Evidence: The added `normalized.startsWith("deepseek-v4")` makes `resolveFoundryApi()` return `openai-responses` for `DeepSeek-V4-Pro` and `DeepSeek-V4-Flash`. That value is used while discovering deployments in `extensions/microsoft-foundry/auth.ts:133` and is persisted into provider/model config through `buildFoundryProviderConfig()` at `extensions/microsoft-foundry/shared.ts:279` and `extensions/microsoft-foundry/shared.ts:301`. Connection tests then call `${baseUrl}/responses` for `openai-responses` in `extensions/microsoft-foundry/onboard.ts:323`. Microsoft Foundry’s DeepSeek catalog currently labels both `DeepSeek-V4-Pro` and `DeepSeek-V4-Flash` as `chat-completion`; Microsoft’s Responses API supported-model list includes DeepSeek V3/R1 entries but not V4 and says unsupported models return `400 Model not supported`; DeepSeek’s own API docs expose V4 examples under `POST /chat/completions`.
- Impact: New Microsoft Foundry users selecting DeepSeek V4 via discovery/onboarding can get a persisted config that immediately calls `/responses` against a chat-completion deployment, causing setup/test or runtime model calls to fail.
- Suggested fix: Remove the DeepSeek V4 Responses default unless provider-owner live proof shows these exact Foundry deployments support `/responses`; otherwise keep V4 on `openai-completions` and fix any reported schema/tool failure in the chat-completions compatibility path.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the local routing: `DeepSeek-V4-Pro` now resolves to `openai-responses`, and the onboarding connection test builds a `/responses` request. I could not run a live Foundry request because no Foundry credentials/endpoints are available.
- Is this the best way to solve the issue? Yes: API adapter defaults should follow the provider contract. A live provider proof could justify a different fix, but the current public contract points to chat completions.
