# Commit 79e3142

## Summary

Found one low-severity guidance regression in the new login failure classifier.

## Findings

### Low: Device-auth and bootstrap-auth failures fall through to network guidance

- Kind: bug
- File: `ui/src/ui/views/login-gate.ts`
- Line: 231
- Evidence: `resolveLoginFailureFeedback` classifies known pairing, rate-limit, insecure-context, origin, protocol, and shared-auth cases, then returns the generic `network` feedback. Current gateway code can emit `DEVICE_AUTH_*` detail codes from `src/gateway/server/ws-connection/message-handler.ts:731` and `AUTH_BOOTSTRAP_TOKEN_INVALID` from auth resolution. A focused live check showed both `DEVICE_AUTH_SIGNATURE_INVALID` and `AUTH_BOOTSTRAP_TOKEN_INVALID` render as `network` / “Could not connect” with steps to check gateway status, WebSocket URL, and `wss://`.
- Impact: users with a corrupted/stale browser device identity, bad clock/nonce/signature, or expired bootstrap/setup token lose the previously visible auth-specific error as primary guidance and get transport troubleshooting instead. The raw error remains collapsed under details, but the main remediation points at the wrong subsystem.
- Suggested fix: handle remaining auth/device-auth detail codes before the network fallback. For device-auth codes, show device repair/re-pairing guidance; for bootstrap-token invalid, show fresh setup-code/dashboard-token guidance or at least classify it as auth failure rather than transport.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: call `resolveLoginFailureFeedback` with `lastErrorCode: DEVICE_AUTH_SIGNATURE_INVALID` or `AUTH_BOOTSTRAP_TOKEN_INVALID`; it returns `kind: "network"` and the generic transport steps.
- Is this the best way to solve the issue? yes: the fix belongs in the login failure classifier because the gateway already supplies precise detail codes and the rest of the new UI is built around those codes.
