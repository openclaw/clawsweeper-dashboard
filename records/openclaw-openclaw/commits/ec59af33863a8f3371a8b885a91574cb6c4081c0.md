# Commit ec59af3

## Summary

Found two actionable regressions in the new bounded transcript paths. The higher-risk issue is that the large-transcript raw append path can create a parentless current leaf, causing later `SessionManager` reads to hide prior context.

## Findings

### Medium: Raw append creates a parentless leaf and hides prior transcript context

- Kind: regression
- File: `src/gateway/server-methods/chat-transcript-inject.ts`
- Line: 110
- Evidence: `appendRawAssistantMessageToTranscript` writes `{ type, id, timestamp, message }` without `parentId`, and `shouldUseRawAppend` routes any file over 8 MiB with no parent-linked entries into that path. The `@mariozechner/pi-coding-agent` session format requires v2/v3 entries to have `id`/`parentId`, and `SessionManager.getBranch()` walks `parentId`. A focused smoke check with a 10,000,648-byte v3 transcript containing a prior parentless user message then calling the actual `appendInjectedAssistantMessageToTranscript` returned `rawPathUsed: true` and `SessionManager.open(file).getBranch()` returned only the injected assistant message (`branchLength: 1`). The pre-commit `SessionManager.appendMessage` path sets `parentId` to the prior entry and keeps both messages in the branch.
- Impact: after `chat.inject` or `/stop` writes into an oversized v3 append-only transcript, the next run can resume from the injected assistant message as the root and silently lose prior user/assistant context from the active branch.
- Suggested fix: do not raw-append parentless entries into v2/v3 transcripts. Either keep using `SessionManager` for those files, or stream enough metadata to set `parentId` to the last reachable entry id. Add a regression test that reopens the transcript with `SessionManager.open(...).getBranch()` and asserts prior context is still reachable.
- Confidence: high

### Low: Bounded history can falsely report no older history when the tail is fully hidden

- Kind: bug
- File: `src/gateway/session-history-state.ts`
- Line: 132
- Evidence: `buildSessionHistorySnapshot` only sets `hasMore = true` for truncated bounded snapshots when `history.messages.length > 0`. `projectChatDisplayMessages` can hide all tail messages, for example heartbeat prompt / `HEARTBEAT_OK` pairs. A focused check with 40 heartbeat tail messages, `totalRawMessages: 99`, and `limit: 1` returned `{"items":[],"messages":[],"hasMore":false}` even though older raw transcript messages exist.
- Impact: `/sessions/<key>/history?limit=1` and SSE first-page history can tell clients there is no history after enough hidden heartbeat/control messages accumulate at the tail, preventing pagination back to older meaningful messages.
- Suggested fix: when a bounded snapshot is known to be truncated, avoid returning a terminal empty page. Either set `hasMore` with a cursor such as `rawTranscriptSeq + 1`, or widen/fallback the read until a visible page or true beginning is found.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Both findings were reproduced with focused `node --import tsx` temp-file checks; the raw append check used the actual `appendInjectedAssistantMessageToTranscript`.
- Is this the best way to solve the issue? yes for direction: preserve the `SessionManager` parent chain on raw append, and avoid false terminal pages for truncated hidden tails.
