# Commit 5de8f8e

## Summary

Found two user-visible voice UI regressions that still exist on current `main` after the V2 files were renamed to canonical Android UI files.

## Findings

### Medium: Voice settings no longer exposes dictation or realtime talk controls

- Kind: regression
- File: `apps/android/app/src/main/java/ai/openclaw/app/ui/SettingsScreens.kt`
- Line: 344
- Evidence: Before this commit, the Voice settings screen rendered `V2SettingsToggleRow` controls for `Speaker`, `Dictation`, and `Realtime Talk`, wired to `viewModel::setSpeakerEnabled`, `viewModel::setMicEnabled`, and `viewModel::setTalkModeEnabled`. This commit replaced that with the “Talk Provider Setup” surface where only the speaker row has an `onClick`; the `VoiceSetupPanel` rows for `Realtime Provider`, `Voice`, and `Transport` are status-only and always show `ready = true` / `Configured` without calling `setMicEnabled` or `setTalkModeEnabled`.
- Impact: Users who enter Settings -> Voice can no longer start/stop dictation or realtime talk from the settings page, even though that page previously advertised and implemented “Control talk, dictation, and playback.” Current `VoiceScreen` can route active dictation/talk users into this settings page, but the page cannot stop realtime talk; it only lets them toggle speaker or leave.
- Suggested fix: Restore functional rows for dictation and realtime talk, or move the provider setup UI behind a separate section while keeping explicit `setMicEnabled` and `setTalkModeEnabled` controls in Voice settings.
- Confidence: high

### Low: Provider card reports “Ready” while the gateway is still connecting

- Kind: bug
- File: `apps/android/app/src/main/java/ai/openclaw/app/ui/VoiceScreen.kt`
- Line: 693
- Evidence: This commit added `VoiceProviderCard`, which derives readiness from `gatewayStatus.isVoiceGatewayReady()`. That helper returns true for any status string that does not contain `offline`, `not connected`, `failed`, or `error`. Runtime connection paths set user-visible statuses such as `Connecting…`, `Reconnecting…`, and `Verify gateway TLS fingerprint…`; those are not ready states, but the helper classifies them as ready and the card displays `Connected and ready` / `Ready`.
- Impact: The Voice screen can tell users the provider is ready during connection, reconnect, or TLS verification states, which makes voice availability look healthy before the gateway is usable.
- Suggested fix: Pass an explicit readiness boolean from `viewModel.isConnected` or a typed gateway state into the provider card, or make the helper require a positive connected state instead of treating unknown transitional strings as ready.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. For the settings regression, compare the reviewed diff: the `Dictation` and `Realtime Talk` toggle rows were removed and current `SettingsScreens.kt` still has only the speaker `onClick`. For the provider-card bug, set or reach `statusText = "Connecting…"` via `refreshGatewayConnection`; `isVoiceGatewayReady()` returns true and the provider card renders `Ready`.
- Is this the best way to solve the issue? yes. Both fixes should use existing view-model state and restore explicit controls rather than adding more status-string interpretation.
