# Commit 7439d782

## Summary

The commit removes the final branch provenance check from the npm publish workflow. The remaining checks compare the downloaded artifact’s self-reported manifest fields to the release tag/SHA, but they no longer verify that the artifact was produced by the trusted workflow ref.

## Findings

### High: npm publish can promote a forged preflight artifact from another workflow ref

- Kind: supply_chain
- File: `.github/workflows/openclaw-npm-release.yml`
- Line: 523
- Evidence: Before this commit, the publish job required the referenced preflight run and full release validation run to have `headBranch == github.ref_name`. Now it only checks `workflowName`, `event`, and success at lines 522-533, then downloads the preflight artifact and trusts `preflight-manifest.json` for `releaseTag`, `releaseSha`, `npmDistTag`, and tarball digest at lines 614-652. That manifest is generated inside the workflow artifact itself, so a workflow run from another branch can claim the release tag/SHA and include a matching digest for a different tarball. The parent release workflow also only validates the downloaded preflight manifest at `.github/workflows/openclaw-release-publish.yml:213` and then dispatches this npm workflow with that run id at `.github/workflows/openclaw-release-publish.yml:873`. GitHub supports manually running a workflow on a selected branch/ref, and `gh workflow run --ref` selects the branch or tag containing the workflow file to run, so branch provenance matters here. ([docs.github.com](https://docs.github.com/en/actions/how-tos/manage-workflow-runs/manually-run-a-workflow)) ([cli.github.com](https://cli.github.com/manual/gh_workflow_run))
- Impact: A prepared npm tarball from an untrusted or modified workflow ref can pass the final publish checks if it forges the manifest values for a real release tag. The `npm-release` approval gate still exists, but this removes the workflow-side guarantee that the artifact was built by the trusted release workflow, creating a package supply-chain bypass.
- Suggested fix: Keep accepting SHA-pinned evidence, but verify GitHub-provided run provenance independently of the artifact. For example, request `headSha`/`headBranch` from `gh run view` and require either a trusted release branch/ref or a run `headSha`/workflow SHA that is tied to the trusted release commit/ref. Do the same for the full release validation run; do not rely on manifest SHA alone.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. In a read-only review I did not create a branch/run, but the code path is direct: run `OpenClaw NPM Release` from another branch/ref with a crafted `openclaw-npm-preflight-<tag>` artifact whose manifest claims the release tag SHA, then dispatch the publish flow with that `preflight_run_id`. The current publish job checks the artifact’s self-reported manifest, not the run’s branch/workflow SHA.
- Is this the best way to solve the issue? Yes. The intended compatibility is to allow SHA-verified evidence from alternate refs, so the narrow fix is to compare against GitHub-provided run provenance (`headSha`, trusted branch/ref, or workflow SHA) rather than restoring a strict same-branch-only rule.
