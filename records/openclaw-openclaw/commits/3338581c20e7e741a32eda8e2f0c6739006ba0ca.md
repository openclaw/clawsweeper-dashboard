# Commit 3338581

## Summary

Found one concrete regression: the commit makes diagnostics onboarding prefer ClawHub, but both exact ClawHub specs currently fail the installer before npm fallback.

## Findings

### Medium: Diagnostics onboarding now defaults to non-installable ClawHub specs

- Kind: regression
- File: `scripts/lib/official-external-plugin-catalog.json`
- Line: 65
- Evidence: The catalog now sets both `diagnostics-otel` and `diagnostics-prometheus` to `defaultChoice: "clawhub"` at `scripts/lib/official-external-plugin-catalog.json:65` and `scripts/lib/official-external-plugin-catalog.json:83`. In onboarding, `promptInstall: false` makes `ensureOnboardingPluginInstalled` use that default directly, then the ClawHub failure path only asks for npm afterward (`src/commands/onboarding-plugin-install.ts:732`, `src/commands/onboarding-plugin-install.ts:788`, `src/commands/onboarding-plugin-install.ts:865`). A live dry-run of the exact specs returned `missing_archive_integrity` for `clawhub:@openclaw/diagnostics-otel` and `no_installable_version` for `clawhub:@openclaw/diagnostics-prometheus`.
- Impact: Users selecting these optional diagnostics plugins now hit a failed default install path and an extra fallback prompt instead of the previously direct npm install. In nonstandard prompt/default handling, or if the user declines the fallback after seeing the failure, the selected plugin is not installed.
- Suggested fix: Keep these diagnostics entries npm-default until the ClawHub package metadata is installable, or pin/update the ClawHub specs and registry artifact metadata so `installPluginFromClawHub` succeeds. Add a focused test or live catalog check that the default source for official onboarding entries is actually installable.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: run a dry-run installer probe for `clawhub:@openclaw/diagnostics-otel` and `clawhub:@openclaw/diagnostics-prometheus`; both fail before npm fallback.
- Is this the best way to solve the issue? no: preferring ClawHub is fine once those ClawHub packages are installable, but the catalog default should not point first at a currently failing source.
