# Commit c919702

## Summary

The commit adds Moonshot provider aliases, but the runtime resolver still passes the raw alias provider into registry-backed lookup. Direct aliases work only when a matching inline `models.providers.moonshot` config is present; they still fail when the model exists in the generated `models.json`/PI registry under canonical `moonshot`.

## Findings

### Medium: Moonshot aliases still miss registry-backed model resolution

- Kind: bug
- File: `src/agents/pi-embedded-runner/model.ts`
- Line: 1085
- Evidence: `normalizeProviderId("moonshotai")` now returns `moonshot` in `src/agents/provider-id.ts:23`, but `resolveModel()`, `resolveModelAsync()`, and `resolveModelWithRegistry()` build `normalizedRef.provider` from the raw `provider` argument at `src/agents/pi-embedded-runner/model.ts:963`, `src/agents/pi-embedded-runner/model.ts:1028`, and `src/agents/pi-embedded-runner/model.ts:1085`. That raw value is then passed to `resolveExplicitModelWithRegistry()`, which reaches `modelRegistry.find(provider, modelId)`. The PI registry lookup is exact on both provider and model id, so canonical `moonshot/kimi-k2.6` rows are not found when the user requested `moonshotai/kimi-k2.6`.
- Impact: Users relying on generated provider discovery / `models.json` can still get `Unknown model: moonshotai/kimi-k2.6`, even though `moonshot/kimi-k2.6` resolves and the commit claims direct `moonshotai/...` refs are accepted. The new tests cover only inline config with `models.providers.moonshot`, which masks this path.
- Suggested fix: Normalize the provider part before calling registry and runtime resolution paths, for example set `normalizedRef.provider` to `normalizeProviderId(provider)`/`normalizeProviderId(params.provider)` in `resolveModel()`, `resolveModelAsync()`, and `resolveModelWithRegistry()`, then add a regression with a registry row under `moonshot` and no inline provider config.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused runtime probe with a fake registry containing only `moonshot/kimi-k2.6` showed `moonshot` resolves, while `moonshotai` queries the registry as `moonshotai/kimi-k2.6` twice and returns `Unknown model: moonshotai/kimi-k2.6`.
- Is this the best way to solve the issue? yes. The alias should be canonicalized at the resolver boundary before all registry, dynamic, fallback, and error paths use the provider id.
