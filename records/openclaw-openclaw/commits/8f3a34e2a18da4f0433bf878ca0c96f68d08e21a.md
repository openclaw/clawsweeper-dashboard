## Findings

### Low: GitHub fs-safe dependency now runs an unpublished source build during installs

- Kind: supply_chain
- File: `package.json`
- Line: 1698
- Evidence: `package.json` changed `@openclaw/fs-safe` from a registry range to `github:openclaw/fs-safe#3c508734af0b`, and `pnpm-lock.yaml` resolves it as a GitHub codeload tarball without an npm integrity field. The commit also adds `@openclaw/fs-safe` to `onlyBuiltDependencies` at `package.json:1804`. A fresh `pnpm install --frozen-lockfile` executed the dependency’s nested install and `prepack` build before linking it. `npm view @openclaw/fs-safe@0.1.2` shows a registry tarball with integrity, but `npm pack @openclaw/fs-safe@0.1.2` does not contain the new durable-queue exports this commit imports, so current `main` is depending on an unreleased source snapshot even though its package version is still `0.1.2`.
- Impact: OpenClaw installs now depend on a GitHub source tarball and install-time build scripts for this security-sensitive filesystem helper, instead of a registry artifact with lockfile integrity. That widens the install failure and supply-chain surface and creates confusing version skew between the installed Git snapshot and the published `0.1.2` package. OpenClaw docs describe `@openclaw/fs-safe` as the guardrail for security-sensitive local file operations, so preserving reproducible dependency provenance matters here. ([docs.openclaw.ai](https://docs.openclaw.ai/gateway/security/secure-file-operations?utm_source=openai))
- Suggested fix: publish the fs-safe commit that includes `readRootJsonObjectSync`, `readRootStructuredFileSync`, and the JSON durable queue helpers as a new registry version, then depend on that semver version with an integrity-backed lockfile entry. Remove `@openclaw/fs-safe` from `onlyBuiltDependencies` if the registry artifact does not need install-time scripts.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. From a clean dependency state, `pnpm install --frozen-lockfile` ran the Git dependency’s nested install and `prepack`; `pnpm-lock.yaml` shows the GitHub tarball resolution without npm integrity.
- Is this the best way to solve the issue? yes. The code needs helpers that are not in the currently published `0.1.2` registry tarball, so the durable fix is to publish a new fs-safe registry version and switch OpenClaw back to the registry artifact.
