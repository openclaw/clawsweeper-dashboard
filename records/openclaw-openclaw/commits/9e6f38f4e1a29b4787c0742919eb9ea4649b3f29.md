## Summary

Found one actionable reliability issue in the new gateway-relay browser barge-in logic. The follow-up `f1636d5e28` renamed the relay RPC/event surface on current `main`, but the same counter reset remains in the current source, so the issue still matters.

## Findings

### Medium: Streaming audio chunks can prevent gateway-relay barge-in from ever reaching the speech-frame threshold

- Kind: reliability
- File: `ui/src/ui/chat/realtime-talk-gateway-relay.ts`
- Line: 147
- Evidence: The microphone path only cancels playback after `BARGE_IN_CONSECUTIVE_SPEECH_FRAMES` speech frames, but every incoming relay `audio` frame resets `speechFramesDuringPlayback` to `0` before queuing playback. The gateway forwards each provider audio delta as its own browser `audio` event (`src/gateway/talk-realtime-relay.ts:185`), and OpenAI’s bridge calls `onAudio` for every `response.audio.delta` (`extensions/openai/realtime-voice-provider.ts:581`). If provider audio chunks arrive between microphone processor callbacks, the counter is repeatedly reset between user-speech frames and never reaches `2`.
- Impact: Users speaking over gateway-relay assistant playback can fail to interrupt the assistant during normal streamed output, especially with small/frequent audio deltas. The focused test added by this commit only covers one initial audio frame followed by speech, so it misses the realistic interleaving of `audio -> speech frame -> audio -> speech frame`.
- Suggested fix: Do not reset `speechFramesDuringPlayback` on every `audio` chunk. Reset it only when playback starts from idle/new turn, on `clear`/`stopOutput`, or after a cancel is sent. Also keep `cancelRequestedForPlayback` true until the gateway confirms a clear/turn boundary so late chunks do not immediately re-arm playback.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes, with a unit-level event sequence: start `GatewayRelayRealtimeTalkTransport`, emit one relay `audio` frame, pump one speechy microphone frame, emit another relay `audio` frame, then pump another speechy microphone frame. Expected behavior is one `cancelOutput`/relay-cancel request after two speech frames; the current code resets the counter on the second audio frame, so no cancel is sent. I did not add that test because this was report-only.
- Is this the best way to solve the issue? Yes: the counter reset belongs to playback lifecycle boundaries, not every streamed audio delta. That keeps the consecutive-speech heuristic meaningful while preserving the existing silence/stop reset behavior.
