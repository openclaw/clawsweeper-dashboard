# Commit 6918134

## Summary

Found one Google Vertex auth regression.

## Findings

### Medium: Google Vertex ADC is bypassed after adding manifest env metadata

- Kind: regression
- File: `extensions/google/openclaw.plugin.json`
- Line: 79
- Evidence: The new `setup.providers[].envVars` entry adds `GOOGLE_CLOUD_API_KEY` to the generic candidate map for `google-vertex`. `resolveEnvApiKey` returns `null` immediately when a provider has candidate env vars but none are set (`src/agents/model-auth-env.ts:65`), so it never reaches the existing Google Vertex ADC fallback at `src/agents/model-auth-env.ts:76`. A focused smoke with `GOOGLE_APPLICATION_CREDENTIALS`, `GOOGLE_CLOUD_PROJECT`, and `GOOGLE_CLOUD_LOCATION` set returned `resolveEnvApiKey= null`; the same call with an empty candidate map returned `{"apiKey":"gcp-vertex-credentials","source":"gcloud adc"}`.
- Impact: Users authenticated to Google Vertex via Application Default Credentials, but without `GOOGLE_CLOUD_API_KEY`, lose env auth detection for `google-vertex`. That affects model list/status/probe and runtime paths that call `resolveEnvApiKey("google-vertex")`.
- Suggested fix: Let `google-vertex` fall through to `resolveGoogleVertexEnvApiKey` when declared env candidates are absent, or handle the ADC fallback before returning `null` for an exhausted candidate list. Add an unmocked test covering manifest-backed `setup.providers[].envVars` plus ADC credentials.
- Confidence: high
