# Commit e2501b2d6d

## Summary

Found two observability regressions in the new diagnostics export paths. The focused tests added by the commit pass, but they do not exercise the real trust metadata paths used at runtime.

## Findings

### Medium: Prometheus session recovery metrics never record real recovery events

- Kind: regression
- File: `extensions/diagnostics-prometheus/src/service.ts`
- Line: 449
- Evidence: `recordDiagnosticEvent` returns immediately when `metadata.trusted` is false at `extensions/diagnostics-prometheus/src/service.ts:449`, but the real session recovery coordinator emits both `session.recovery.requested` and `session.recovery.completed` with `emitDiagnosticEvent`, not `emitTrustedDiagnosticEvent`, at `src/logging/diagnostic-session-recovery-coordinator.ts:25` and `src/logging/diagnostic-session-recovery-coordinator.ts:44`. The new Prometheus recovery cases at `extensions/diagnostics-prometheus/src/service.ts:585` are therefore unreachable for the real producer. A direct probe rendered `""` for an untrusted `session.recovery.completed` event and only emitted `openclaw_session_recovery_total` when the same event was marked trusted.
- Impact: The documented `openclaw_session_recovery_total` and `openclaw_session_recovery_age_seconds` metrics do not export actual Gateway session recovery activity, so Prometheus dashboards and alerts silently miss the recovery path this commit claims to expose.
- Suggested fix: Either emit session recovery diagnostics through the trusted path if they are core-owned internal telemetry, or let the Prometheus exporter admit these specific core recovery event types while continuing to reject untrusted plugin-controlled events. Update the Prometheus test to use the real emission path or `trusted: false` for recovery.
- Confidence: high

### Medium: Runtime Talk events are not included in stability snapshots

- Kind: regression
- File: `src/logging/diagnostic-stability.ts`
- Line: 642
- Evidence: Talk controllers now call `options.onEvent` from `src/talk/talk-session-controller.ts:75`, and all new call sites pass `recordTalkDiagnosticEvent`, which emits via `emitTrustedDiagnosticEvent` at `src/talk/diagnostics.ts:28`. The stability recorder subscribes through `onDiagnosticEvent` at `src/logging/diagnostic-stability.ts:642`, and `onDiagnosticEvent` drops trusted events at `src/infra/diagnostic-events.ts:857`. A read-only `tsx` probe using a real `createTalkSessionController(..., { onEvent: recordTalkDiagnosticEvent })` emitted `session.started`, waited for the async drain, and produced a stability snapshot with `"count": 0`.
- Impact: Support/stability snapshots and bundles omit real Talk lifecycle/audio events even though this commit added a `talk.event` sanitizer and test coverage for snapshots. That makes Talk failures harder to diagnose from the stability bundle surface.
- Suggested fix: Subscribe the stability recorder to internal diagnostic events and sanitize the trusted Talk subset, or provide a bounded public emission path for Talk stability records. Add a test that wires a real Talk controller to `recordTalkDiagnosticEvent` and asserts the snapshot records the sanitized event.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. For Prometheus, call `__test__.recordDiagnosticEvent` with a `session.recovery.completed` event and `{ trusted: false }`; the rendered metrics are empty, matching the real coordinator’s `emitDiagnosticEvent` path. For stability, start the recorder, create a Talk controller with `{ onEvent: recordTalkDiagnosticEvent }`, emit `session.started`, wait one tick, and inspect an empty snapshot.
- Is this the best way to solve the issue? unclear. The safer direction is to align each consumer with the existing trust model rather than broadly weakening the Prometheus trusted-event guard.
