# Commit cce00498cd

## Summary

Found one regression in the legacy runtime migration path. The commit preserves default legacy Claude CLI intent, but it can ignore a per-agent legacy PI/auto opt-out and then route that agent through the newly materialized default model runtime policy.

## Findings

### Medium: Per-agent legacy PI runtime opt-outs can be lost under a default Claude CLI migration

- Kind: regression
- File: `src/commands/doctor/shared/legacy-config-core-normalizers.ts:220`; also `src/commands/doctor/shared/legacy-config-migrations.runtime.agents.ts:297`
- Evidence: Both preservation helpers treat `agentRuntime.id` values of `pi` and `auto` as non-recoverable and return without writing a per-agent model runtime policy. The new default-preservation path writes `agents.defaults.models["anthropic/..."].agentRuntime.id = "claude-cli"`, and `resolveModelRuntimePolicy` checks `agentEntry?.models` first, then `agents.defaults.models` at `src/agents/model-runtime-policy.ts:145`. A direct probe with `agents.defaults.agentRuntime.id = "claude-cli"` plus `agents.list[0].agentRuntime.id = "pi"` for the same Anthropic model resolved the per-agent run to `{ id: "claude-cli" }`, not PI. `doctor --fix` also removed the per-agent `agentRuntime` while leaving the inherited default model policy in place.
- Impact: Configs that used a default legacy Claude CLI runtime but explicitly opted a particular agent back into PI/API runtime can be silently changed. After normalization or `doctor --fix`, that agent inherits the default Claude CLI model policy, changing auth/runtime/tool behavior and permanently deleting the legacy per-agent opt-out during migration.
- Suggested fix: When preserving a default legacy CLI runtime into `agents.defaults.models`, also preserve per-agent legacy `pi`/`auto` overrides as per-agent `agents.list[].models[ref].agentRuntime` entries for matching selected refs, or otherwise skip/default-scope the migrated policy so explicit per-agent legacy runtime choices still win. Apply the same rule to compatibility normalization and `doctor --fix`, with regression tests for default `claude-cli` plus per-agent `pi`/`auto`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused `node --import tsx` probe against current main showed `normalizeCompatibilityConfigValues` writes the default Anthropic model policy and `resolveModelRuntimePolicy({ agentId: "local" })` returns `claude-cli` despite `agents.list[0].agentRuntime.id = "pi"`. The same probe through `migrateLegacyConfig` showed `doctor --fix` removes the per-agent PI pin and still resolves the agent to `claude-cli`.
- Is this the best way to solve the issue? yes, at the migration boundary. Runtime selection intentionally ignores whole-agent legacy keys, so the fix should preserve override intent while translating legacy config into model-scoped policy.
