# Commit dafbdb6f20

## Summary

Found one concrete regression in the new invalid-identity preservation path: `loadOrCreate` now returns an unsaved random identity for recognized invalid identity files, so repeated launches keep changing `deviceId` and cannot reuse stored device tokens.

## Findings

### Medium: Invalid identity files now create throwaway device IDs on every load

- Kind: regression
- File: `src/infra/device-identity.ts`
- Line: 238
- Evidence: `normalizeStoredIdentity` marks recognized but invalid PEM/raw identity shapes as `recognized-invalid`, and `loadOrCreateDeviceIdentity` returns `generateIdentity()` without writing it. A focused smoke with an invalid `{ version: 1, publicKeyPem, privateKeyPem }` file produced two different device IDs from two consecutive `loadOrCreateDeviceIdentity(path)` calls while leaving `device.json` unchanged. Swift has the same branch at `apps/shared/OpenClawKit/Sources/OpenClawKit/DeviceIdentity.swift:59`.
- Impact: device tokens are keyed by the current `deviceId` (`src/shared/device-auth-store.ts` only reuses tokens when the stored `deviceId` matches). With an invalid recognized identity file, every CLI/app process can present as a fresh device, store auth under a throwaway ID, and then lose it on the next load. This recreates the pairing churn class the commit is meant to prevent, just for malformed or partially corrupted recognized OpenClaw identity files.
- Suggested fix: do not return an unsaved generated identity for `recognized-invalid`. Either surface a clear identity-file error / omit device identity so callers do not pair a throwaway key, or preserve diagnostic evidence by quarantining/copying the bad file and then persisting one stable replacement identity. Apply the same behavior in Swift and TS.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Create `identity/device.json` with `version: 1`, `deviceId`, invalid `publicKeyPem`, invalid `privateKeyPem`, and `createdAtMs`; call `loadOrCreateDeviceIdentity(path)` twice. The returned IDs differ and the invalid file remains in place.
- Is this the best way to solve the issue? no. The bidirectional PEM/raw migration is the right direction, but the invalid-file branch should not silently pair with an identity that cannot be reloaded.
