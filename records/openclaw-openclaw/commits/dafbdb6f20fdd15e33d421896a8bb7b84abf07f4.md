# Commit dafbdb6

## Summary

Found one regression in the new invalid-identity preservation path. When the identity file is invalid but recognized, both loaders keep the bad file and return a newly generated identity without saving it, so every process launch gets a different device id.

## Findings

### Medium: Invalid identity files now create a fresh unsaved device id on every load

- Kind: regression
- File: `src/infra/device-identity.ts`
- Line: 238
- Evidence: `loadOrCreateDeviceIdentity()` returns `generateIdentity()` for `recognized-invalid` without writing the generated identity. Its catch path also returns `generateIdentity()` without writing whenever `identityFileExists(filePath)` is true. The Swift path mirrors this at `apps/shared/OpenClawKit/Sources/OpenClawKit/DeviceIdentity.swift:59`, where `.recognizedInvalid` returns `self.generate()` without `save()`.
- Impact: a bad `identity/device.json` leaves the client with a disposable in-memory identity. On the next CLI/gateway/app launch, the same invalid file is read again and a different device id is generated, so stored device tokens keyed by device id cannot be reused and pairing prompts can repeat. This reintroduces the class of pairing churn the commit is meant to prevent, especially for the macOS app because `DeviceAuthStore` only loads tokens when the stored auth file’s `deviceId` matches the current identity.
- Suggested fix: do not silently continue with an unsaved random identity while preserving the invalid file. Either fail with a clear diagnostic, or move/quarantine the invalid file and save a generated replacement so subsequent loads are stable while still preserving evidence.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused TS probe wrote an invalid PEM-shaped `identity/device.json`, called `loadOrCreateDeviceIdentity()` twice, and observed `sameDeviceId: false` while the invalid file remained preserved.
- Is this the best way to solve the issue? yes. Pairing stability requires either a durable identity or an explicit failure; a transient generated identity is the unsafe middle ground.
