# Commit bbc47cb

## Summary

Found one regression in the fix path. The new skip guard prevents downloading the older ClawHub/marketplace package, but it leaves the stale external install record in place. Installed globals intentionally outrank bundled duplicates, so an already-installed older external plugin can keep shadowing the newer bundled plugin after `openclaw update`.

## Findings

### Medium: Skip leaves stale external install active over the newer bundled plugin

- Kind: regression
- File: `src/plugins/update.ts`
- Line: 593
- Evidence: The new branch at `src/plugins/update.ts:593` emits a skipped outcome and `continue`s when bundled version is newer than the recorded ClawHub/marketplace version. It does not remove or rewrite the install record. Release-channel sync also ignores non-path/non-npm records at `src/plugins/update.ts:1089`. Meanwhile registry precedence ranks a global candidate matching an install record ahead of bundled at `src/plugins/manifest-registry.ts:479`, and the existing loader test asserts “installed global beats bundled duplicate” in `src/plugins/loader.test.ts:7278`.
- Impact: A user who already has `plugins.installs.whatsapp` pointing at an older ClawHub package, with that package present under the managed plugin root, will still load the older global plugin after update. The command now warns/skips, but the newer bundled WhatsApp remains shadowed, so the reported broken-channel state can persist.
- Suggested fix: When skipping because bundled is newer, switch the install metadata back to the bundled path or remove the stale ClawHub/marketplace install record so registry resolution uses the bundled plugin. Add a test that combines `updateNpmInstalledPlugins` with duplicate registry precedence, proving the bundled plugin wins after this skip.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: seed a ClawHub install record for `whatsapp` with version `2026.2.9` and `installPath` pointing to an existing managed global plugin, expose bundled `whatsapp` at `2026.4.25`, run `updateNpmInstalledPlugins`, then load the plugin registry with both candidates. The updater skips with `changed: false`, and registry precedence keeps the global install ahead of bundled.
- Is this the best way to solve the issue? no: skipping the network update is only half the repair. The safer fix also has to make the active plugin source become bundled, or the stale external package keeps winning.
