# Commit fd244fd

## Summary

Found regressions in the new Telegram isolated ingress path. The most severe is a packaged-build worker sidecar path mismatch that prevents the default Telegram polling path from loading its worker at runtime.

## Findings

### High: Packaged Telegram polling resolves the worker sidecar from the wrong dist directory

- Kind: regression
- File: `extensions/telegram/src/telegram-ingress-worker.ts`
- Line: 50
- Evidence: `createTelegramIngressWorker` uses `new URL("./telegram-ingress-worker.runtime.js", import.meta.url)`. After `pnpm build`, this code is bundled into `dist/monitor-polling.runtime-*.js`, so the URL resolves to `dist/telegram-ingress-worker.runtime.js`. The actual emitted sidecar is `dist/extensions/telegram/telegram-ingress-worker.runtime.js`; `dist/telegram-ingress-worker.runtime.js` is absent. A focused worker-load probe against the built chunk failed with `Cannot find module .../dist/telegram-ingress-worker.runtime.js` and `exit:1`.
- Impact: `monitor.ts` now enables isolated ingress by default, so packaged/NPM Telegram polling can fail before any Bot API polling happens.
- Suggested fix: emit the worker sidecar next to the bundled polling runtime chunk, or resolve the worker URL to the actual stable emitted plugin sidecar. Add a built-dist smoke test that starts or at least loads the Telegram ingress worker.
- Confidence: high

### Medium: Isolated draining serializes all Telegram updates and ignores configured runner concurrency

- Kind: regression
- File: `extensions/telegram/src/polling-session.ts`
- Line: 269
- Evidence: `#drainSpooledUpdates` loops through spooled files and awaits each `bot.handleUpdate` before reading the next update. The previous `@grammyjs/runner` path uses a concurrent sink, and `createTelegramRunnerOptions` still configures `sink.concurrency` from `resolveAgentMaxConcurrent`, but the isolated path never applies it.
- Impact: one long-running Telegram update can block every later spooled update, including unrelated chats, while the worker keeps advancing offsets. That regresses the prior runner behavior where different update keys could proceed concurrently.
- Suggested fix: drain with a bounded concurrent sink that respects `runnerOptions.sink.concurrency`, while preserving per-update delete-after-success and retry-on-failure behavior.
- Confidence: high

### Medium: Transient Telegram API errors are now fatal to the worker

- Kind: reliability
- File: `extensions/telegram/src/telegram-ingress-worker.runtime.ts`
- Line: 73
- Evidence: `fetchJson` parses JSON and throws a plain `Error(description)` for any non-OK or `ok:false` response, losing Telegram `error_code` and `parameters`. The catch block only retries `isRecoverableTelegramNetworkError`; it rethrows everything else. The previous `@grammyjs/runner@2.0.3` fetcher retries polling errors until `maxRetryTime`, only treating 401/409 as unrecoverable and honoring 429 `retry_after`.
- Impact: a transient Bot API 429/5xx, or a proxy returning a non-JSON 502 page, can stop isolated ingress instead of retrying polling.
- Suggested fix: preserve Telegram API error metadata and retry polling errors using runner-equivalent semantics: fatal for unauthorized/conflict, honor 429 retry-after, retry transient 5xx/parse/proxy failures with bounded backoff.
- Confidence: high

### Medium: Custom `timeoutSeconds` can make quiet long polls self-timeout forever

- Kind: compatibility
- File: `extensions/telegram/src/telegram-ingress-worker.runtime.ts`
- Line: 95
- Evidence: the worker sends `timeout: options.timeoutSeconds` to `getUpdates`, but the local fetch guard always aborts at `TELEGRAM_GET_UPDATES_REQUEST_TIMEOUT_MS` (`45_000`). Existing request-timeout tests explicitly keep `getupdates` at 45s even when configured `timeoutSeconds` is higher.
- Impact: an account configured with `timeoutSeconds > 45` asks Telegram to hold the long poll longer than OpenClaw allows, so quiet polls abort locally, log errors, and back off repeatedly.
- Suggested fix: clamp the Bot API long-poll `timeout` below the local getUpdates guard, or derive both from the same getUpdates timeout contract.
- Confidence: high
