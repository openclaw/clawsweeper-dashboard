# Commit 841cb12

## Summary

Found one regression in the auth-retry cancellation logic. The change uses Twurple’s `manual` disconnect flag to distinguish user cancellation from auth retry reconnects, but Twurple’s own auth retry path also closes the IRC connection on demand and can emit `manual=true`.

## Findings

### Medium: Auth retry reconnects can be mistaken for user cancellation

- Kind: regression
- File: `extensions/twitch/src/twitch-client.ts`
- Line: 235
- Evidence: The new handler only ignores auth-retry disconnects when `authRetryPending && !manual`; otherwise a manual disconnect with no reason rejects with `Twitch connection cancelled`. In `@twurple/chat@8.1.4`, authentication failure emits `onAuthenticationFailure`, then calls `this._ircClient.reconnect()` after the retry delay. `ircv3@0.33.1` implements `reconnect()` as `quit(); connect()`, and `quitAbruptly()` calls `_connection.disconnect()`. The connection layer emits `onDisconnect(true)` for on-demand WebSocket/direct closes, and `ChatClient` forwards that boolean to `onDisconnect`.
- Impact: A real Twitch auth failure that Twurple intends to retry can cause OpenClaw’s pending `getClient()` promise to reject and call `client.quit()`, tearing down the retry path instead of waiting for Twurple’s next authentication attempt. This regresses the prior behavior that ignored all disconnects during `authRetryPending`.
- Suggested fix: Do not use Twurple’s `manual` flag alone as the cancellation signal during auth retry. Track explicit manager-initiated pending cancellations, for example with a cancellation flag/WeakSet set by `disconnect()` and `disconnectAll()` before calling `quit()`, and only reject auth-retry disconnects when that explicit cancellation is present. Add a focused test where auth failure is followed by `onDisconnect(true)` without manager cancellation and the connection remains pending/retryable.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Source path: invalid/stale Twitch auth emits `onAuthenticationFailure`; Twurple’s first `fibWithLimit(120)` retry delay is `0`; Twurple then calls `_ircClient.reconnect()`, which closes the current connection on demand and propagates `onDisconnect(true)`. A focused unit repro can simulate `authFailureHandlers[0]()` followed by `disconnectHandlers[0]?.(true)` without calling `manager.disconnect()` or `disconnectAll()`.
- Is this the best way to solve the issue? yes. The plugin needs its own explicit cancellation state because Twurple’s `manual` means “connection closed on demand,” not necessarily “OpenClaw user/operator cancelled this pending connection.”
