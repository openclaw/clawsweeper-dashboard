# Commit 31e2276

## Summary

Found one regression: relaxing the `memory-lancedb` manifest schema also removes the signal that bundled install uses to keep config-required plugins disabled until configured.

## Findings

### Medium: Unconfigured Memory LanceDB bundled installs are now treated as valid

- Kind: regression
- File: `extensions/memory-lancedb/openclaw.plugin.json`
- Line: 129
- Evidence: This commit removes top-level `configSchema.required: ["embedding"]`. `resolveBundledPluginSources` derives `requiresConfig` only from top-level required fields at `src/plugins/bundled-sources.ts:79`, and `hasValidBundledPluginConfig` immediately accepts any bundled plugin whose `requiresConfig` is false at `src/cli/plugins-install-command.ts:77`. A direct metadata probe now returns `{ "pluginId": "memory-lancedb", "requiresConfig": false, "hasRequired": null }`.
- Impact: `openclaw plugins install memory-lancedb` with an incomplete existing entry such as `{ config: {} }` no longer strips that incomplete entry or prints the “requires configuration first” warning. The focused install test now persists `{ config: {} }` instead of leaving the plugin disabled, so users can end up with an apparently enabled Memory LanceDB install whose runtime parser still rejects missing `embedding` and registers only the disabled service.
- Suggested fix: Preserve metadata discovery without losing install gating. For example, add an explicit manifest/install metadata flag for “requires configuration before activation”, or update bundled install gating to use plugin-specific setup/config-contract metadata instead of relying solely on JSON Schema `required`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Run `OPENCLAW_VITEST_MAX_WORKERS=1 pnpm test src/cli/plugins-cli.install.test.ts`; the existing test `does not persist incomplete config entries for config-gated bundled installs` fails because `writtenConfig.plugins.entries["memory-lancedb"]` is `{ config: {} }` instead of `undefined`.
- Is this the best way to solve the issue? unclear. The commit’s schema relaxation is valid for setup discovery, but install gating needs a separate required-config signal so metadata loading and activation safety do not share one overloaded JSON Schema field.
