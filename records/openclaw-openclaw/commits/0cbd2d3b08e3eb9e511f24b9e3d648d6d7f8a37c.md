## Findings

### Medium: Inline code wrapping reverts the copy-fidelity guard

- Kind: regression
- File: `ui/src/styles/chat/text.css`
- Line: 96
- Evidence: This commit changes `.chat-text :where(:not(pre) > code)` back to `overflow-wrap: anywhere` and `word-break: break-word`. Prior commit `7c90ef7c52cdc8a10cf5c3e636bcc200af947c50` added `overflow-wrap: normal` and `word-break: keep-all` to this same selector to fix `openclaw/openclaw#32230`, where copied UUIDs/long tokens gained inserted spaces at wrap points. The new regression test at `ui/src/ui/chat/chat-responsive.browser.test.ts:414` checks visual containment only; it does not select/copy/paste the wrapped token.
- Impact: Users copying long inline code tokens such as job IDs, hashes, or command arguments from chat can get corrupted pasted values on affected browser/platform combinations, causing commands to fail or identifiers to be misused.
- Suggested fix: Preserve the visual containment fix, but add a browser copy/paste assertion for a wrapped long inline token and adjust the CSS only once that exact copied text is proven across the supported browser path. If the intended support matrix proves copy fidelity with this CSS, keep the CSS and add that proof as a guardrail.
- Confidence: medium

- Is this the best way to solve the issue? Unclear. The visual fix direction is right, but the safe solution should prove both no clipping and byte-for-byte copy fidelity.
