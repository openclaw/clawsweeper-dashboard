# Commit aca8440

## Summary

Found two diagnostics regressions: Prometheus does not export the newly documented session recovery metrics from the real recovery path, and Talk audio byte histograms miss existing STT/TTS producers that use `payload.bytes`.

## Findings

### Low: Prometheus recovery metrics are dropped for real recovery events

- Kind: bug
- File: `extensions/diagnostics-prometheus/src/service.ts`
- Line: 449
- Evidence: `recordDiagnosticEvent` returns immediately for `!metadata.trusted`, but `src/logging/diagnostic-session-recovery-coordinator.ts:25` and `src/logging/diagnostic-session-recovery-coordinator.ts:44` emit `session.recovery.requested` / `session.recovery.completed` with `emitDiagnosticEvent`, not `emitTrustedDiagnosticEvent`. The added test passes `trusted` directly at `extensions/diagnostics-prometheus/src/service.test.ts:168`, so it does not cover the production emitter path.
- Impact: The documented `openclaw_session_recovery_total` and `openclaw_session_recovery_age_seconds` Prometheus metrics stay absent during actual stuck-session recovery, even though the commit claims to export them.
- Suggested fix: Either emit recovery exporter events through a trusted internal path, or intentionally allow these bounded recovery event types through the Prometheus trust gate. Add a test that starts the Prometheus service and emits recovery through the real recovery diagnostic path.
- Confidence: high

### Low: Talk audio byte metrics miss producers that use `payload.bytes`

- Kind: bug
- File: `src/talk/diagnostics.ts`
- Line: 23
- Evidence: `createTalkDiagnosticEvent` only reads `byteLength` and `audioBytes`, but existing audio Talk events use `bytes` in `extensions/google-meet/src/realtime.ts:619`, `extensions/google-meet/src/realtime.ts:769`, `extensions/voice-call/src/media-stream.ts:245`, and `extensions/voice-call/src/media-stream.ts:617`. A focused mapper check with `{ payload: { bytes: 320 } }` returned `byteLength: undefined`.
- Impact: `openclaw.talk.audio.bytes` / `openclaw_talk_audio_bytes` underreport audio traffic for the STT/TTS Google Meet and Voice Call bridges.
- Suggested fix: Normalize all audio producers to `byteLength`, or have the diagnostic mapper also accept bounded numeric `bytes`. Add coverage for both existing payload shapes.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused Prometheus service check emitted `session.recovery.completed` through `emitDiagnosticEvent` and `exporter.render().includes("openclaw_session_recovery_total")` printed `false`. A focused Talk mapper check with `payload.bytes` printed `undefined`.
- Is this the best way to solve the issue? yes for the byte mapper normalization. For recovery metrics, the safest fix direction depends on whether maintainers want Prometheus to keep trusting only official internal events or mirror OTel’s broader handling for bounded diagnostics.
