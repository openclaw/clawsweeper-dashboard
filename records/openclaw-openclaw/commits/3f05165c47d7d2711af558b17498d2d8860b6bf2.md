# Commit 3f05165

## Summary

Found one contract regression in Codex auth ordering. The new alias merge can silently add native `openai-codex:*` profiles even when the user explicitly ordered only an `openai:*` API-key profile under `auth.order.openai`.

## Findings

### Medium: Explicit OpenAI auth order no longer excludes omitted Codex profiles

- Kind: regression
- File: `src/agents/auth-profiles/order.ts`
- Line: 280
- Evidence: `aliasExplicitOrder` from `auth.order.openai` is passed into `mergeAliasOrderWithNativeProfiles`, which prepends/appends every native Codex profile from the store before eligibility filtering. A focused check with `auth.order.openai: ["openai:api-key"]` plus stored `openai-codex:sub` returned `["openai-codex:sub","openai:api-key"]`.
- Impact: Users can no longer rely on the documented explicit order to restrict Codex auth to the profiles they listed. This can route a Codex run through a ChatGPT/Codex subscription profile even when the user set the user-facing OpenAI order to an API-key-only profile, affecting billing/route choice and `models status --probe` exclusion semantics.
- Suggested fix: Treat explicit alias order as authoritative: do not add native Codex profiles unless they are explicitly listed, or repair/suggest the desired subscription-first order into `auth.order.openai` outside the hot runtime resolver.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The focused `pnpm exec tsx -e ...` resolver check on current `main` shows an omitted native Codex profile being inserted ahead of the only explicitly ordered OpenAI API-key profile.
- Is this the best way to solve the issue? yes, if `auth.order.openai` remains the documented user-facing Codex order. If maintainers intentionally want implicit native Codex prepending, the docs and probe exclusion contract need to be updated instead.
