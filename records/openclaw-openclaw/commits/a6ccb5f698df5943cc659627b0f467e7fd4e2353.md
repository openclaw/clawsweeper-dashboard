# Commit a6ccb5f

## Summary

Found one medium-confidence reliability regression: ambiguous Discord request timeouts and transport failures are now retried for message/thread creation without an idempotency guard, which can duplicate outbound Discord messages if the first POST reached Discord but the response was lost.

## Findings

### Medium: Ambiguous Discord message POST failures can now be replayed without dedupe

- Kind: reliability
- File: `extensions/discord/src/retry.ts`
- Line: 73
- Evidence: The new classifier treats every `AbortError` as retryable and also retries broad transport messages/codes. `createDiscordRetryRunner` is used by `createDiscordClient`, and `sendDiscordText` wraps `createChannelMessage` POSTs in that request runner at `extensions/discord/src/send.shared.ts:324`. The message payload builder has no nonce/idempotency field, while `RequestClient` aborts in-flight fetches on its timeout at `extensions/discord/src/internal/rest.ts:150`.
- Impact: For `POST /channels/{id}/messages`, forum thread creation, media sends, voice sends, and other non-idempotent Discord writes, a local timeout/reset after Discord accepts the request can cause OpenClaw to submit the same send again. That can produce duplicate cron/agent replies or duplicate forum/thread starter messages during the exact outage class this commit targets.
- Suggested fix: Make retry eligibility method/route aware. Keep retries for clear HTTP 429/408/5xx responses and pre-connect failures, but do not retry ambiguous local `AbortError`/body-timeout/socket-close failures for non-idempotent create-message/create-thread routes unless the request includes a reliable idempotency/dedupe key and recovery path.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? No live Discord proof was run. A focused unit reproduction is straightforward with a fetch/rest stub that records a successful server-side create, then rejects the first client attempt with `AbortError`; the current runner will call the POST twice when attempts are `2`.
- Is this the best way to solve the issue? Unclear. The commit correctly fixes the original no-retry gap from #52396, but the safer shape is to split retry policy by request idempotency instead of applying the same ambiguous-failure retry predicate to every Discord REST call.
