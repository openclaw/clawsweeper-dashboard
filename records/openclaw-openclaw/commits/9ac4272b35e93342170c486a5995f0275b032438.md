# Commit 9ac4272

## Summary

Found one concrete regression in the new nested inline-chain rewrite path. The safe-bin hardening is broadly scoped correctly, and the targeted tests pass, but nested `sh -c` chains that mix explicit allowlist commands with safe-bin commands can now silently execute different argv for the allowlisted segment.

## Findings

### Medium: Nested safe-bin chain rewrite literalizes allowlisted segments too

- Kind: regression
- File: `src/infra/exec-approvals-analysis.ts`
- Line: 1079
- Evidence: `src/infra/exec-approvals-allowlist.ts:706` collapses an entire wrapper segment to `"inlineChain"` when any inner segment is satisfied by `safeBins`/`inlineChain`. Then `renderInlineChainSegmentArgv` re-analyzes the wrapper payload and calls `buildEnforcedShellCommand`, which quotes and canonicalizes every inner segment, not just the safe-bin segment. A focused probe showed `sh -c 'echo $HOME && head -c 16'` is allowlist-satisfied with `echo` explicitly allowlisted and `head` safe-binned, but rebuilds to a payload containing `'/usr/bin/echo' '$HOME' && '/usr/bin/head' '-c' '16'`. The original command expands `$HOME`; the dispatched command passes literal `$HOME`.
- Impact: Existing allowlist-mode workflows that wrap mixed chains in `sh -c` can silently change behavior after this commit whenever one segment needs safe-bin literalization. Top-level chains preserve non-safe segments, so nested chains now behave inconsistently.
- Suggested fix: Preserve the inner segment plan and `segmentSatisfiedBy` metadata from inline-chain evaluation, then rebuild the inner payload with `buildSafeBinsShellCommand` so only inner safe-bin/inline-chain segments are literalized. Alternatively, fail closed for nested mixed chains until the exact plan can be preserved.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The focused `tsx` probe against `evaluateShellAllowlist` and `buildSafeBinsShellCommand` reproduces the altered command plan without needing external services.
- Is this the best way to solve the issue? yes for the fix direction. The root problem is loss of per-inner-segment satisfaction metadata; carrying that metadata forward avoids both silent behavior changes and over-broad literalization.
