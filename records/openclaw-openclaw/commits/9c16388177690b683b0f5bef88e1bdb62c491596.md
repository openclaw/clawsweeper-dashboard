# Commit 9c16388177

## Summary

Found one regression in the new Mantis Telegram serialization step. Both workflows run `gh run list` before checkout, but the command does not pass `--repo`, so GitHub CLI cannot determine the repository in the empty job workspace.

## Findings

### Medium: Wait step fails before checkout because `gh run list` has no repo context

- Kind: regression
- File: `.github/workflows/mantis-telegram-desktop-proof.yml:242`, `.github/workflows/mantis-telegram-live.yml:263`
- Evidence: the new first step runs before `actions/checkout@v6` and calls `gh run list --workflow "$workflow" ...` without `--repo`. A focused probe from a temp directory with `GITHUB_REPOSITORY=openclaw/openclaw` failed with `failed to determine base repo: failed to run git: fatal: not a git repository`, while the same command with `--repo openclaw/openclaw` succeeded.
- Impact: both Mantis Telegram workflows exit in the serialization step before checkout and before any proof work runs, making the Telegram live and desktop proof workflows unusable.
- Suggested fix: add `--repo "$GITHUB_REPOSITORY"` to both `gh run list` invocations, matching the explicit repo used by the preceding `gh api` call.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: run `GITHUB_REPOSITORY=openclaw/openclaw gh run list --workflow mantis-telegram-live.yml --limit 1 --json databaseId,status,createdAt,url` outside a git checkout.
- Is this the best way to solve the issue? yes: passing `--repo "$GITHUB_REPOSITORY"` preserves the current pre-checkout placement and makes the repo context explicit.
