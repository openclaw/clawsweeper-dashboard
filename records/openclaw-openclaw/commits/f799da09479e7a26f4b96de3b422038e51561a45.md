# Commit f799da0947

## Summary

Found one concrete parser bug in the new lockfile package seeding path. The focused test added by this commit fails, and the malformed package spec it emits is rejected by `pnpm store add`.

## Findings

### Medium: Peer-suffixed lockfile keys are split at the peer dependency version

- Kind: bug
- File: `scripts/list-prod-store-packages.mjs`
- Line: 29
- Evidence: `packageSpecFromLockfileKey()` uses `normalizedKey.lastIndexOf("@")`. For `recma-jsx@1.0.1(acorn@8.16.0)`, that splits at `acorn@8.16.0`, producing `recma-jsx@1.0.1(acorn@8.16.0)` instead of `recma-jsx@1.0.1`. The committed test at `test/scripts/list-prod-store-packages.test.ts:121` fails with exactly that received value.
- Impact: The focused test suite is red. If a peer-suffixed lockfile key reaches the Docker store seeding pipeline, `pnpm store add` rejects it with `ERR_PNPM_SPEC_NOT_SUPPORTED_BY_ANY_RESOLVER`, aborting the Docker runtime-assets build.
- Suggested fix: Strip the peer suffix before finding the package/version separator, for example split on the package part before the first `(` and then apply `lastIndexOf("@")`, or use a pnpm-compatible lockfile key parser.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Run `node scripts/run-vitest.mjs test/scripts/list-prod-store-packages.test.ts`; the new test fails. A direct smoke also prints `recma-jsx@1.0.1(acorn@8.16.0)` for the fixture lockfile key.
- Is this the best way to solve the issue? yes. The bug is in lockfile-key normalization before `pnpm store add`; normalizing the key before package/version splitting fixes both the failing test and the rejected pnpm argument.
