# Commit 89db1e5

## Summary

Found two concrete issues in the new cron diagnostics path: persisted diagnostics bypass configured custom redaction patterns, and the claimed unavailable-tool cron fix still misses the representative #75763 final-answer wording.

## Findings

### Medium: Cron diagnostics ignore custom redaction patterns before persisting tool output

- Kind: privacy
- File: `src/cron/run-diagnostics.ts`
- Line: 80
- Evidence: Diagnostic messages and summaries call `redactSensitiveText(..., { mode: "tools" })` at `src/cron/run-diagnostics.ts:80` and `src/cron/run-diagnostics.ts:148`. That path uses built-in/default patterns only when explicit options are passed, while the established tool-surface helper `redactToolPayloadText` merges user `logging.redactPatterns` with defaults at `src/logging/redact.ts:181`. Docs also say custom patterns can protect safety/UI/diagnostic surfaces at `docs/logging.md:227`. A focused probe with `logging.redactSensitive: "off"` and a custom `custom-secret-*` pattern showed `normalizeCronRunDiagnostics` preserved the synthetic secret in both summary and entry, while `redactToolPayloadText` masked it.
- Impact: Cron diagnostics are now persisted into cron job state/run logs and exposed through Gateway/CLI status surfaces. Project-specific secrets covered only by `logging.redactPatterns` can leak where older tool/UI diagnostic surfaces would redact them.
- Suggested fix: Use `redactToolPayloadText` for cron diagnostic text, or explicitly merge `getDefaultRedactPatterns()` with configured `logging.redactPatterns` in this path. Add a regression test with `redactSensitive: "off"` plus a custom pattern.
- Confidence: high

### Medium: Representative unavailable-tool cron output still stays `ok` with no diagnostic

- Kind: bug
- File: `src/cron/isolated-agent/helpers.ts`
- Line: 45
- Evidence: The changelog says this fixes #75763 for unavailable requested tools at `CHANGELOG.md:49`, but the denial classifier only matches tokens like `could not run`, `did not run`, and `was denied` at `src/cron/isolated-agent/helpers.ts:45`. The new diagnostics extractor only records tool error payloads/details, `meta.error`, or `meta.failureSignal` at `src/cron/run-diagnostics.ts:293`. A focused probe using representative final assistant text, `this cron session does not have exec available`, produced `hasFatalErrorPayload: false` and no diagnostics.
- Impact: The common #75763 path can still look like a successful delivered cron run: status remains `ok`, `cron show` prints no diagnostic, and users must manually read the delivered text to notice the tool-policy failure.
- Suggested fix: Add narrow unavailable-tool patterns for the issue examples, such as missing requested tool availability and `not found or not runnable` when classified as a tool-policy failure, then assert those examples in cron payload outcome/diagnostics tests.
- Confidence: high
