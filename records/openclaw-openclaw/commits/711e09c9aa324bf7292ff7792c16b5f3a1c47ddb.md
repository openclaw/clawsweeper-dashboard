# Commit 711e09c

## Summary

Found regressions in the new Discord realtime forced-consult skip path. The native opus opt-in path itself looked sound in the focused checks.

## Findings

### Medium: Forced consult skip can drop valid short commands ending in common words

- Kind: regression
- File: `extensions/discord/src/voice/realtime.ts`
- Line: 159
- Evidence: `classifySkippableForcedAgentProxyTranscript()` treats any transcript whose final word is in `DISCORD_REALTIME_FORCED_CONSULT_TRAILING_FRAGMENT_WORDS` as a `trailing-fragment`. That set includes words that are often valid command endings, notably `that` and `then`. In default `agent-proxy` / `consultPolicy: "always"` mode, `scheduleForcedAgentProxyConsult()` returns immediately on that classification, so a turn like “do that”, “fix that”, or “try that” can be ignored instead of being sent to the routed OpenClaw agent.
- Impact: Discord voice users can lose actionable follow-up commands in the exact fallback path meant to protect them when the realtime model produces a final transcript without calling `openclaw_agent_consult`.
- Suggested fix: Narrow the trailing-fragment heuristic to true dangling connectors, or require stronger evidence of incompleteness before skipping. At minimum, do not classify pronoun/adverb endings such as `that` and `then` as incomplete by themselves.
- Confidence: high

### Medium: Skipped transcripts consume speaker context before a later tool call can use it

- Kind: regression
- File: `extensions/discord/src/voice/realtime.ts`
- Line: 879
- Evidence: `scheduleForcedAgentProxyConsult()` calls `consumePendingSpeakerContext()` before `classifySkippableForcedAgentProxyTranscript()`. If the transcript is classified as skippable, the method logs and returns without storing a pending or recent consult context. If the realtime provider then emits a real `openclaw_agent_consult` tool call for that same transcript, `handleToolCall()` can no longer match a pending context and eventually reports `No Discord speaker context available`.
- Impact: The skip was intended to suppress only stale forced fallback answers, but it can also break provider-initiated consults for the same speaker turn. That can drop owner identity and tool authorization for an otherwise valid Discord voice request.
- Suggested fix: Classify before consuming the speaker context, or keep a pending/recent context for skipped transcripts so a later explicit tool call can still bind to the correct speaker.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Use the existing `DiscordVoiceManager` realtime agent-proxy test setup, begin a speaker turn with audio, emit a final transcript like `do that` or `Get this working and...`, and wait past the forced consult delay. For the second case, emit a matching consult tool call afterward; current code has already consumed the only pending speaker context.
- Is this the best way to solve the issue? yes for the direction: the skip decision should not consume state needed by the normal consult path, and the fragment heuristic should avoid common valid command endings.
