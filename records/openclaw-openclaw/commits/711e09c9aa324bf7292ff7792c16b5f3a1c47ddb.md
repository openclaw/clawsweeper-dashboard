## Findings

### Medium: Complete voice commands can be skipped as “trailing fragments”

- Kind: regression
- File: `extensions/discord/src/voice/realtime.ts`
- Line: 168
- Evidence: `classifySkippableForcedAgentProxyTranscript` treats any final transcript whose last word is in `DISCORD_REALTIME_FORCED_CONSULT_TRAILING_FRAGMENT_WORDS` as `trailing-fragment`. That list includes complete-utterance endings like `on`, `in`, `to`, `with`, and `for`. In default `agent-proxy` mode with `consultPolicy: "always"`, `scheduleForcedAgentProxyConsult` consumes the pending speaker context before this check and returns without calling `runAgentTurn` when the classifier skips the transcript (`extensions/discord/src/voice/realtime.ts:879`, `extensions/discord/src/voice/realtime.ts:881`). A focused shell reproduction of the same predicate classified complete utterances as skipped: `what are you working on => trailing-fragment`, `turn it on => trailing-fragment`, and `can you help me log in => trailing-fragment`.
- Impact: Normal Discord voice requests can be silently ignored when the realtime transcript lacks punctuation and happens to end in one of these words. This regresses the previous “always consult” behavior for complete commands/questions such as “turn it on” or “what are you working on”.
- Suggested fix: Do not classify trailing connector words as skippable by word alone. Limit fragment skipping to explicit ellipsis/truncated transcript signals, require stronger syntax evidence such as a minimum trailing phrase pattern, or only suppress closings. Add regression coverage for complete utterances ending in `on`, `in`, and `to`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The failure path is direct from the added classifier into `scheduleForcedAgentProxyConsult`; the reproduction command evaluated the exact normalization/last-word predicate and showed complete utterances being skipped.
- Is this the best way to solve the issue? yes, at the filter boundary. The forced-consult queueing logic is otherwise doing the right thing; the unsafe part is the broad lexical skip list.
