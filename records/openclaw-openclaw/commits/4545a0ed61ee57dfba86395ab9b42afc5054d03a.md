# Commit 4545a0e

## Summary

Found one regression: the commit changes ACPX, Google Chat, and LINE official install specs from explicit `@beta` specs to bare npm package names, but the live npm `latest` tag for all three packages still resolves to a prerelease. OpenClaw’s npm installer rejects bare specs that resolve to prerelease versions, so catalog-driven installs and doctor repair fail.

## Findings

### Medium: Bare official launch specs fail while npm latest is still beta

- Kind: regression
- File: `scripts/lib/official-external-plugin-catalog.json`
- Line: 14
- Evidence: `scripts/lib/official-external-plugin-catalog.json:14`, `scripts/lib/official-external-channel-catalog.json:179`, and `scripts/lib/official-external-channel-catalog.json:204` now advertise bare specs. `src/plugins/install.ts:1156` rejects prerelease resolutions unless `src/infra/npm-registry-spec.ts:125` sees an explicit selector. Live npm metadata currently resolves `@openclaw/acpx`, `@openclaw/googlechat`, and `@openclaw/line` to `2026.5.2-beta.2` for both `latest` and `beta`. Focused dry-runs confirmed `@openclaw/acpx`, `@openclaw/googlechat`, and `@openclaw/line` fail with the prerelease guard, while the explicit `@beta` specs succeed.
- Impact: missing configured plugin repair calls `installPluginFromNpmSpec` with the catalog/fallback spec at `src/commands/doctor/shared/missing-configured-plugin-install.ts:427`, and onboarding does the same through `src/commands/onboarding-plugin-install.ts:594` and `src/commands/onboarding-plugin-install.ts:904`. Users trying to repair/install ACPX, Google Chat, or LINE from the official catalog will get a hard install failure until npm `latest` is promoted to a stable version.
- Suggested fix: keep these specs as explicit `@beta` until stable package versions are published/promoted to `latest`, or make catalog-driven install/doctor/onboarding resolve default specs through the active beta channel before calling `installPluginFromNpmSpec`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: run `node --import tsx --eval 'import { installPluginFromNpmSpec } from "./src/plugins/install.ts"; console.log(await installPluginFromNpmSpec({ spec: "@openclaw/acpx", dryRun: true, mode: "install", expectedPluginId: "acpx" }));'` with current npm metadata.
- Is this the best way to solve the issue? yes for the immediate release window: either restore explicit prerelease specs or publish/promote stable `latest` artifacts before relying on bare specs. A longer-term channel-aware resolver would be safer than embedding launch-time beta assumptions in static catalog specs.
