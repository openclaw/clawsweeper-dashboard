# Commit 07a11c4

## Summary

Two low-severity robustness issues in the new PR comment/publish path. The exact-SHA live Mantis run succeeded, and I did not find a security, privacy, or high-impact CI regression.

## Findings

### Low: Upsert can overwrite a non-bot comment containing the marker

- Kind: bug
- File: `.github/workflows/mantis-discord-status-reactions.yml`
- Line: 351
- Evidence: The comment lookup selects any issue comment whose body contains `<!-- mantis-discord-status-reactions -->`, then patches the last matching id. It does not filter to comments authored by `github-actions[bot]`.
- Impact: If a user posts a later comment containing that marker, the workflow can update that user-authored comment instead of the bot-owned QA comment, leaving stale bot evidence and misattributing the generated QA evidence.
- Suggested fix: Filter the comment search by bot author, for example `.user.login == "github-actions[bot]"`, before selecting the marker comment.
- Confidence: high

### Low: Concurrent Mantis runs can race when pushing the shared `qa-artifacts` branch

- Kind: concurrency
- File: `.github/workflows/mantis-discord-status-reactions.yml`
- Line: 307
- Evidence: Each run fetches `qa-artifacts`, commits a disjoint artifact path, then pushes `HEAD:qa-artifacts` once. The workflow concurrency key is scoped to baseline/candidate/run attempt, while all runs publish to the same branch.
- Impact: Two different Mantis runs that finish around the same time can both base their commit on the same branch tip; the later push will be rejected as non-fast-forward, failing an otherwise successful QA run/comment publish.
- Suggested fix: Add a short fetch/rebase/push retry loop around the artifact push, or serialize only the branch-publish section with a shared concurrency key.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the comment-selection bug via a PR comment containing the marker after the bot comment; yes for the branch race by running two dispatches that publish to `qa-artifacts` from different concurrency groups.
- Is this the best way to solve the issue? Yes: bot-author filtering keeps the upsert target stable, and retry/serialization is the usual fix for shared branch appenders.
