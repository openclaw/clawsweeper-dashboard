Found one current Telnyx realtime failure-handling gap in the surface this test commit tried to cover.

## Findings

### Medium: Telnyx `streaming.failed` webhooks are still acknowledged and dropped

- Kind: bug
- File: `extensions/voice-call/src/providers/telnyx.ts`
- Line: 211
- Evidence: The reviewed commit added coverage for `call.streaming.failed`, but Telnyx’s Call Control API docs for both dial and answer say the stream lifecycle webhook names are `streaming.started`, `streaming.stopped`, or `streaming.failed` when `stream_url` is set (`https://developers.telnyx.com/api-reference/call-control/dial-call`, `https://developers.telnyx.com/api-reference/call-commands/answer-call`). Current `main` handles only `streaming.started` and `streaming.stopped` at `extensions/voice-call/src/providers/telnyx.ts:211`, then falls through to `default: return null`. A focused parser probe with `event_type: "streaming.failed"` returned `{"events":[],"statusCode":200}`.
- Impact: A Telnyx media-stream setup failure is silently acknowledged instead of becoming a `call.error`, so the manager never marks or surfaces the realtime failure. Operators can be left with a call that appears healthy while the realtime stream failed to attach.
- Suggested fix: Add a `case "streaming.failed"` in `TelnyxProvider.normalizeEvent` that maps to non-retryable `call.error`, using Telnyx’s failure reason fields when present. Add the corresponding provider test with the unprefixed event name.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: instantiate `TelnyxProvider`, call `parseWebhookEvent` with a signed/verified-equivalent body containing `data.event_type: "streaming.failed"`, and observe `events: []`.
- Is this the best way to solve the issue? yes: Telnyx sends this as a webhook lifecycle event, while WS `event:"error"` frames cover the already-established socket path; both paths should be handled.
