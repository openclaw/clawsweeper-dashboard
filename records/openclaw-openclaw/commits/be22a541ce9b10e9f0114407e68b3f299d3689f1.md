# Commit be22a541

## Summary

Found one actionable edge case in the blank tool-name fix. The commit stops a final assistant message when every tool call is an unrecoverable blank, but a mixed batch containing one valid tool call and one blank tool call is classified as `allowed`, so the blank call still reaches pi-agent-core dispatch and produces the same `Tool  not found` class of error.

## Findings

### Medium: Mixed valid and blank tool-call batches still dispatch the blank call

- Kind: bug
- File: `src/agents/pi-embedded-runner/run/attempt.tool-call-normalization.ts`
- Line: 707
- Evidence: `classifyToolCallMessage` records `sawBlankStringToolCall`, but returns `{ kind: "allowed" }` as soon as any block resolves to an allowed tool, before the blank-malformed branch at line 710 can run. A focused one-off probe of `wrapStreamFnTrimToolCallNames` with `[{name:"read"}, {name:""}]` and `allowedToolNames = Set(["read"])` returned both tool calls unchanged, including the blank one. The installed `@mariozechner/pi-agent-core@0.73.1` dispatch loop then filters all `type === "toolCall"` blocks and resolves each by exact `toolCall.name`; missing matches become `Tool ${toolCall.name} not found` in `node_modules/@mariozechner/pi-agent-core/dist/agent-loop.js:330`.
- Impact: Models can emit parallel tool-call batches. If one call is valid and another malformed blank, the commit’s final-result guard is bypassed and OpenClaw can still persist an error result for the blank name, continue the retry loop, and leak the malformed call id path the fix was meant to stop.
- Suggested fix: Do not let `sawAllowedToolCall` short-circuit unrecoverable blank string calls. Fail closed on any unrecoverable blank block before dispatch, or prune only the malformed blank block while preserving valid calls. Add a regression test with one allowed tool call plus one blank tool call.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A direct wrapper probe showed the mixed final message remains `{"content":[{"name":"read"},{"name":""}]}` after normalization; pi-agent-core source confirms both tool calls are dispatched.
- Is this the best way to solve the issue? yes for direction. The safest fix is still generic runner-side malformed tool-call handling, but it needs per-block malformed detection instead of whole-message “allowed” classification.
