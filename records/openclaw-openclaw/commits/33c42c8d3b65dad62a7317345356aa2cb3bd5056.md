# Commit 33c42c8d

## Summary

Found one low-severity label automation regression: Barnacle can clear `proof: sufficient` after a title-only PR edit, even though the reviewed proof and head commit are unchanged.

## Findings

### Low: Title-only PR edits clear `proof: sufficient`

- Kind: regression
- File: `scripts/github/barnacle-auto-response.mjs`
- Line: 780
- Evidence: `shouldRemoveProofSufficientLabel` returns true for every `edited` action when proof still evaluates as passed. The workflow subscribes to `pull_request_target` `edited`, and GitHub documents `edited` as a supported pull request activity; GitHub also describes edit webhook payloads for issue or pull request title/body edits and says the payload points out what changed. ([docs.github.com](https://docs.github.com/en/actions/reference/events-that-trigger-workflows?ref=fabians-dev-blog)) A focused local smoke using `payload.changes: { title: { from: "Old title" } }`, valid proof, and existing `proof: sufficient` removed `proof: sufficient`.
- Impact: A contributor changing only the PR title can erase ClawSweeper’s stronger proof judgment, causing avoidable re-review or label churn without any proof/body/head change.
- Suggested fix: For `edited`, inspect `context.payload.changes` and only clear `proof: sufficient` when `changes.body` is present. Keep clearing on `synchronize`, and keep clearing whenever the proof evaluation no longer passes.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes; call `runBarnacleAutoResponse` with action `edited`, `changes.title`, valid real behavior proof, and labels `proof: supplied` plus `proof: sufficient`; it removes `proof: sufficient`.
- Is this the best way to solve the issue? yes; using the webhook `changes` object preserves the intended invalidation behavior while avoiding title-only churn.
