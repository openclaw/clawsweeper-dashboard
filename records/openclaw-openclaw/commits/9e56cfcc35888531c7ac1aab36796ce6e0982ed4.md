# Commit 9e56cfc

## Summary

Found one medium-confidence regression in Codex CLI session invalidation for macOS Keychain-only credentials.

## Findings

### Medium: Keychain-only Codex CLI auth no longer contributes to session invalidation

- Kind: regression
- File: `src/agents/cli-auth-epoch.ts`
- Line: 127
- Evidence: `resolveCliAuthEpoch()` is used by `prepareCliRunContext()` before `resolveCliSessionReuse()` decides whether an existing CLI session can be resumed (`src/agents/cli-runner/prepare.ts:225`, `src/agents/cli-runner/prepare.ts:259`). This commit changed the Codex CLI credential read to `readCodexCliCredentialsCached({ ttlMs: 5000, allowKeychainPrompt: false })` (`src/agents/cli-auth-epoch.ts:127`). On Darwin, `readCodexKeychainAuthRecord()` returns `null` whenever `allowKeychainPrompt === false` (`src/agents/cli-credentials.ts:255`), and `readCodexCliCredentials()` only falls back to `auth.json` after that (`src/agents/cli-credentials.ts:606`). For users whose Codex auth exists only in Keychain, `authEpoch` becomes `undefined`; if the stored binding also has no epoch, `resolveCliSessionReuse()` treats it as reusable because both normalized epochs match (`src/agents/cli-session.ts:156`).
- Impact: Switching Codex accounts in Keychain can fail to invalidate an existing `codex-cli` resume session, so a later run may resume a conversation created under a previous account instead of starting fresh. That is a context/privacy boundary regression, especially because the OpenAI Codex CLI backend uses `codex exec resume {sessionId}` (`extensions/openai/cli-backend.ts:37`).
- Suggested fix: Keep non-prompting reads for startup/status metadata, but preserve an execution-path auth epoch for CLI session reuse. Options: let `prepareCliRunContext()` request a prompting/identity-capable Codex credential read when evaluating an existing binding, or disable reuse when the local credential source is known to be Codex Keychain but cannot be fingerprinted without prompting.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Unclear. The code path is deterministic, but I could not live-verify macOS Keychain behavior from this Linux checkout. A focused reproduction would seed a `codex-cli` session binding without `authEpoch`, provide only a mocked Darwin Keychain credential, then show that `allowKeychainPrompt: false` makes the current epoch undefined and `resolveCliSessionReuse()` reuses the stored session after the mocked Keychain account changes.
- Is this the best way to solve the issue? Unclear. The goal of avoiding Keychain prompts on startup/status paths is sound; the safer fix is to split metadata probing from execution-time session invalidation rather than reverting all non-prompting reads.
