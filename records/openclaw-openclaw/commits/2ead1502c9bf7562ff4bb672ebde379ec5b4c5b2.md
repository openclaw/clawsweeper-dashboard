# Commit 2ead150

## Summary

Found one recovery regression in the new durable send lifecycle. A transient adapter failure after the queue entry is marked `send_attempt_started` can leave the queued message in a state that recovery refuses to replay for channels without reconciliation support.

## Findings

### Medium: Transient send failures can poison queued delivery recovery

- Kind: reliability
- File: `src/infra/outbound/deliver.ts`
- Line: 1221
- Evidence: `deliverOutboundPayloadsWithQueueCleanup` marks the queued entry as `send_attempt_started` before invoking the adapter (`onPlatformSendStart`, lines 1170-1177). If the adapter then throws before returning results, the catch path calls `failDelivery(queueId, ...)` (line 1221). `failDelivery` increments retry state but does not clear `recoveryState` in `src/infra/outbound/delivery-queue-storage.ts:212`, so the entry remains `send_attempt_started`. Recovery treats both `send_attempt_started` and `unknown_after_send` as unsafe to replay unless the adapter implements `durableFinal.reconcileUnknownSend`; without that, it fails the entry again without calling `deliver` in `src/infra/outbound/delivery-queue-recovery.ts:366`.
- Impact: A normal transient outbound error, such as a network failure or provider SDK exception, can stop later startup/reconnect recovery from retrying that message for most channels. The queued delivery stays pending, burns retry attempts with “refusing blind replay without adapter reconciliation,” and can eventually move to failed instead of delivering.
- Suggested fix: On handled adapter failure, either clear/downgrade the recovery state before `failDelivery`, or classify adapter errors so only truly ambiguous post-platform outcomes remain `send_attempt_started`/`unknown_after_send`. Recovery should be able to retry ordinary failed sends after backoff.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. Unit path: enqueue a delivery, call `markDeliveryPlatformSendAttemptStarted`, then `failDelivery` with a transient error, then drain/recover it with no `reconcileUnknownSend` adapter. Existing tests already assert recovery refuses `send_attempt_started` entries without reconciliation.
- Is this the best way to solve the issue? Yes directionally: preserve the no-blind-replay protection for genuinely ambiguous sends, but do not keep that state for known failed sends that should use ordinary retry/backoff.
