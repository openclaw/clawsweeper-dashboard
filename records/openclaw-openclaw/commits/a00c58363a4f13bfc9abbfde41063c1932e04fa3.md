# Commit a00c583

## Summary

The commit suppresses Linux systemd-unavailable hints in `openclaw gateway status` when RPC succeeds, but the linked issue and changelog claim `openclaw status --deep` is fixed. Current `main` still routes the top-level service rows through a separate summary/formatter path, so a reachable system-scoped Gateway can still be reported as `systemd user not installed`.

## Findings

### Medium: `openclaw status --deep` still reports the user-systemd service as missing for reachable system-scoped gateways

- Kind: bug
- File: `src/cli/daemon-cli/status.print.ts`
- Line: 323
- Evidence: The new guard only affects `printDaemonStatus`, the `gateway status` printer. Top-level `openclaw status --deep` computes `gatewayReachable` in `src/commands/status.command.ts:146`, but service rows still come from `readServiceStatusSummary`, which marks a service installed only when the user-managed service command exists or `state.running` is true (`src/commands/status.service-summary.ts:24-28`). On Linux, the service implementation still reads `~/.config/systemd/user/<unit>.service` and probes via `systemctl --user show` (`src/daemon/systemd.ts:55-57`, `src/daemon/systemd.ts:1088-1096`). If the user unit is absent while the Gateway is reachable under a system-level unit, `formatStatusServiceValue` still returns `<label> not installed` (`src/commands/status-all/format.ts:151-152`). The changelog entry says this fixes `openclaw status --deep` and closes #85094.
- Impact: The reported user-facing diagnostic remains false on the issue’s primary surface, and the merged commit closed the issue/changelog as fixed. Linux operators running a documented system-scoped service can still be told the Gateway/Node user service is not installed even while the Gateway RPC probe is healthy.
- Suggested fix: Extend the top-level status overview path to carry `gatewayReachable` into service-row classification, or add a Linux status-only system-scope fallback/external-management summary for active system-level OpenClaw units. Keep lifecycle commands user-systemd-owned unless a separate owner decision changes that contract.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Source inspection matches the linked issue’s repro: reachable Gateway is computed separately, but the top-level service row remains based on user-systemd service state. I did not run a live Ubuntu systemd-system fixture.
- Is this the best way to solve the issue? yes. The fix should cover the `openclaw status --deep` overview/service-summary path because that is the reported and changelog-stated surface; the current `gateway status` suppression is useful but incomplete.
