# Commit 579acc3

## Summary

Found one startup-planning regression in the new policy-only registry refresh path. The optimization correctly updates enabled state, but it can also preserve stale startup metadata after a plugin manifest changes, so a later `plugins enable` can say the plugin is enabled while the gateway startup plan still omits it.

## Findings

### Medium: Policy refresh can preserve stale startup metadata and skip an enabled plugin at gateway startup

- Kind: regression
- File: `src/plugins/installed-plugin-index-store.ts`
- Line: 286
- Evidence: `refreshPersistedInstalledPluginIndex` now takes the fast path for `reason: "policy-changed"` when the persisted index exists and the target plugin id is present. That path only rewrites `policyHash`, `generatedAtMs`, `refreshReason`, and each record’s `enabled` flag; it does not rebuild or validate source-derived fields such as `startup.memory`, `startup.sidecar`, or `startup.agentHarnesses`. Gateway startup planning still consumes those persisted `plugin.startup.*` fields in `src/plugins/gateway-startup-plugin-ids.ts:138` and `src/plugins/gateway-startup-plugin-ids.ts:424`.
- Impact: if a local/path plugin’s manifest changes in place before a policy toggle, startup-affecting metadata can remain stale. I reproduced this with a temp plugin initially indexed as a provider, then changed its manifest to `kind: "memory"`, then ran the new `policy-changed` refresh with `plugins.entries.demo.enabled=true` and `plugins.slots.memory="demo"`. The resulting manifest registry saw `kind: "memory"`, but the persisted record still had `startup.memory:false`, and `loadPluginLookUpTable(...).startup.pluginIds` was `[]`. After `openclaw plugins enable demo`, a gateway restart would therefore not load the selected memory plugin until a manual source rebuild refresh occurs.
- Suggested fix: keep the fast path only when target records are source-fresh for startup-affecting metadata. A narrow fix would rebuild or hash the targeted plugin manifest/package record when `policyPluginIds` is provided and fall back to the normal `refreshInstalledPluginIndex` path if any target’s manifest hash, package hash, root/source path, or startup classification has changed. Alternatively, explicitly derive startup planning from the current manifest registry instead of persisted `plugin.startup` fields.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The focused temp-dir smoke described above produced `{"startup":{"sidecar":false,"memory":false,...},"manifestKind":"memory","startupPluginIds":[]}` after the new policy fast path.
- Is this the best way to solve the issue? unclear. The safest behavior is to preserve the no-global-rebuild optimization while validating the targeted plugin’s source-derived record before overwriting the persisted policy hash. A broader fallback to full rebuild is simpler but gives back some of the performance win this commit intended.
