# Commit 5c7980f

## Summary

Found one low-severity behavioral bug in the new approval-reaction shortcut: removed approval tapbacks are intended to be swallowed, but currently fall through to the normal iMessage reaction event pipeline.

## Findings

### Low: Removed approval tapbacks can leak into normal reaction notifications

- Kind: bug
- File: `extensions/imessage/src/approval-reactions.ts`
- Line: 585
- Evidence: `maybeResolveIMessageApprovalReaction()` says removed tapbacks should be owned by the approval shortcut and kept quiet, but the removed branch returns `false`. `monitor-provider.ts` only stops normal inbound processing when this function returns `true`, so the same tapback can continue into `resolveIMessageInboundDecision()`, where reaction contexts become model-visible reaction system events when `reactionNotifications` is `"own"` or `"all"`.
- Impact: Untapping or switching a 👍/👎 on an approval prompt can create a noisy normal iMessage reaction event instead of being silently consumed as the commit message and inline comment describe.
- Suggested fix: For removed tapbacks that target a known approval binding, consume the event without resolving the approval. Keep or tombstone the binding deliberately depending on the intended toggle behavior, but return `true` for approval-owned removals so they do not fall through to the general reaction pipeline.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. The code path is direct: removed tapback returns `false` in `approval-reactions.ts`, and `monitor-provider.ts` only short-circuits on `true`; existing inbound reaction logic then emits reaction decisions for bot-authored tapback targets.
- Is this the best way to solve the issue? Yes, if the intended contract is the one documented in the commit and comment: approval-owned removed tapbacks should be consumed by the approval shortcut.
