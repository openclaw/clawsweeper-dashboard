# Commit 8a1e220

## Summary

Found one low-severity QA regression risk: the relaxed Matrix tool-progress matcher now accepts non-tool bullet lines, so the live QA scenario can pass without proving tool progress was folded into the Matrix preview.

## Findings

### Low: Generic bullet matcher can mask missing Matrix tool progress

- Kind: test_gap
- File: `extensions/qa-matrix/src/runners/contract/scenario-runtime-room.ts`
- Line: 693
- Evidence: `hasMatrixQaToolProgressPreviewLine` accepts any bullet-like line matching `[-*•] ...`, and `runToolProgressPreviewScenario` enables that fallback. A local regex probe showed `- \`planning\``, `* ordinary bullet`, and `• \`approval requested\`` all match. Production Matrix preview code separately emits real tool starts as `tool: ${toolName}`, while plan, approval, item, command, and patch callbacks can also append preview lines.
- Impact: If the Matrix preview stops receiving actual tool-start progress but still receives another progress bullet plus the final marker, `matrix-room-tool-progress-preview` can pass while the behavior named by the scenario is broken.
- Suggested fix: Keep the name relaxation but require the generic fallback to identify tool progress, for example a bullet/code line beginning with `tool:`. Add a negative unit case such as `Working...\n- \`planning\`` that must not satisfy the tool-progress preview scenario.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes at unit level: the introduced regex accepts non-tool progress bullets. The existing mocked scenario shape would resolve if the update body were `Working...\n- \`planning\``.
- Is this the best way to solve the issue? Yes: requiring a `tool:` marker preserves the intended relaxation from `tool: read` to other tool names without turning the assertion into “any progress bullet appeared.”
