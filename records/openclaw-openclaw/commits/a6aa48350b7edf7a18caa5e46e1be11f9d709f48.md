## Findings

### Low: Testing docs still route broad proof to Blacksmith

- Kind: regression
- File: `docs/ci.md`
- Line: 520
- Evidence: The changed skills now say normal broad proof should omit `--provider` and let `.crabbox.yaml` choose brokered AWS (`.agents/skills/crabbox/SKILL.md:12`, `.agents/skills/openclaw-testing/SKILL.md:49`). The wrapper now derives the default provider from `.crabbox.yaml` and validates that selected provider (`scripts/crabbox-wrapper.mjs:32`, `scripts/crabbox-wrapper.mjs:80`, `scripts/crabbox-wrapper.mjs:91`). But `docs/ci.md:520` still says the wrapper refuses binaries without `blacksmith-testbox` and tells users to pass `--provider blacksmith-testbox`; the changed-gate, focused rerun, and full-suite examples at `docs/ci.md:529`, `docs/ci.md:544`, and `docs/ci.md:559` all keep the Blacksmith flags. `docs/reference/test.md:18` also still tells Codex worktrees to use `node scripts/crabbox-wrapper.mjs run --provider blacksmith-testbox ...`, and `.agents/skills/openclaw-testing/SKILL.md:13` tells agents to read that doc first.
- Impact: Operators or agents following the documented test path will still send broad proof to Blacksmith by default, which defeats this commit’s intended brokered-AWS fallback behavior when Blacksmith queues, fails capacity, or is unavailable.
- Suggested fix: Update `docs/ci.md` and `docs/reference/test.md` to match the new default examples: omit `--provider` for normal broad proof, remove the normal-path Blacksmith org/workflow/job/ref flags, and keep Blacksmith only in an explicit opt-in/fallback section.
- Confidence: high
