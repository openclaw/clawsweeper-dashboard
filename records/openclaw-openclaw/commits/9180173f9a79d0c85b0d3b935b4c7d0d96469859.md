# Commit 9180173

## Summary

Found two actionable gaps in the changed behavior: one sanitizer case still leaks raw tool XML, and one mixed system-event queue case can route an exec completion using another event's delivery context.

## Findings

### Medium: Bare MiniMax `<invoke>` tool calls still reach user-facing replies

- Kind: bug
- File: `src/shared/text/assistant-visible-text.ts`
- Line: 343
- Evidence: `sanitizeUserFacingText` now chains `stripMinimaxToolCallXml` and `stripToolCallXmlTags`, but `stripMinimaxToolCallXml` returns early unless the text contains `minimax:tool_call`. The generic XML stripper also does not recognize a top-level `<invoke>...</invoke>` wrapper. Focused check: `sanitizeUserFacingText('Before\n<invoke name="exec"><parameter name="cmd">ls</parameter></invoke>\nAfter')` returned the raw `<invoke>` block unchanged.
- Impact: the shared delivery sanitizer still leaks one of the tool-call text forms described in #62820, including tool names and arguments, to messaging channels.
- Suggested fix: strip standalone `<invoke ...>...</invoke>` blocks outside inline/fenced code when they look like tool-call payloads, while preserving examples in code regions.
- Confidence: high

### Medium: Mixed event queues can route exec completions to the wrong chat/thread

- Kind: privacy
- File: `src/infra/heartbeat-runner.ts`
- Line: 715
- Evidence: preflight computes `turnSourceDeliveryContext` from all pending events, but the prompt and consumption logic now select only exec events when `hasExecCompletion` is true. `resolveSystemEventDeliveryContext` merges later events as primary, so a later generic event can override the exec event's delivery context. Focused check showed pending `[exec topic 47, generic topic 99]` resolves `all` to topic 99 while `execOnly` resolves to topic 47.
- Impact: an async exec completion can be delivered to a later generic event's channel/thread instead of the requester route attached to the exec event, potentially exposing command output to the wrong conversation.
- Suggested fix: derive heartbeat `turnSource` from the same selected event set used to build the heartbeat prompt, or recompute delivery context after `selectSystemEventsConsumedByHeartbeat`.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the sanitizer via the focused `tsx` call above. The routing issue is source-reproduced through the queue/context resolver; I did not run a live channel send.
- Is this the best way to solve the issue? Yes for the sanitizer. For routing, the safest direction is to keep prompt selection, event consumption, and delivery-context selection on the same event subset.
