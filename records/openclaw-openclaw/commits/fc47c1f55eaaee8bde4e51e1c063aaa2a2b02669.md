# Commit fc47c1f

## Summary

The commit adds `--agent` to `infer/capability model auth logout`, but the scoped logout path is still not reliable. It ignores the documented env-scoped agent directory when `--agent` is omitted, and the new explicit `--agent` path can report inherited default-agent profiles as removed even when the selected agent store was not changed.

## Findings

### Medium: Scoped model-auth logout can still report success without removing the selected agent’s profile

- Kind: bug
- File: `src/cli/capability-cli.ts`
- Line: 925
- Evidence: `runModelAuthLogout` resolves `agentDir` as `agent?.trim() || resolveDefaultAgentId(cfg)`, so omitted `--agent` never checks `OPENCLAW_AGENT_DIR` or `PI_CODING_AGENT_DIR`. That conflicts with the documented models status scoping behavior in `docs/cli/models.md:39` and the existing status implementation in `src/commands/models/list.status-command.ts:79` and `src/commands/models/list.status-command.ts:238`. For explicit non-default agents, `src/cli/capability-cli.ts:927` loads `loadAuthProfileStoreForRuntime(agentDir)`, while `src/agents/auth-profiles/store.ts:582` merges the main auth store into non-main agent runtime stores. The command then returns `removedProfiles: profileIds` from that merged store at `src/cli/capability-cli.ts:959`, but the locked writer reloads only the selected local store at `src/agents/auth-profiles/store.ts:449` and returns a store even when `shouldSave` is false at `src/agents/auth-profiles/store.ts:450`.
- Impact: `OPENCLAW_AGENT_DIR=<agent-dir> openclaw infer model auth logout --provider <p>` still targets the configured default agent instead of the env-selected store, so the linked issue workflow can remain broken. Also, `openclaw infer model auth logout --provider <p> --agent <id>` can claim an inherited default-agent profile was removed from the selected agent even though no selected-agent file was saved and the profile remains visible through inheritance.
- Suggested fix: centralize target directory resolution for logout as explicit `--agent`, then `OPENCLAW_AGENT_DIR`/`PI_CODING_AGENT_DIR`, then configured default. For removal, compute and report removed IDs from the locked persisted store actually being mutated, or explicitly resolve/remove the owning persisted store if inherited main credentials are intended to be removable from a scoped agent view.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Source proof is direct: omitted `--agent` has no env override branch, and explicit non-default `--agent` lists provider profiles from a merged runtime store but mutates only the selected persisted store. A focused repro would create a default-agent provider profile, leave the selected agent without a local provider profile, then run logout with `--agent <id>` and observe JSON listing the inherited profile while the selected auth store remains unchanged.
- Is this the best way to solve the issue? yes. The fix should move target-store selection and removal accounting to the same persisted auth-store boundary; otherwise read/write path drift will keep producing false-success logout results.
