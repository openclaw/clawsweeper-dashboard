# Commit 955b025

## Summary

Found one low-severity correctness bug in the new native SQLite Kysely dialect: insert queries prefixed by a Kysely CTE lose `insertId`.

## Findings

### Low: CTE-prefixed insert queries do not return the generated insert id

- Kind: bug
- File: `src/infra/kysely-node-sqlite.ts`
- Line: 159
- Evidence: `executeQuery` only sets `insertId` when `isInsertStatement(sql)` returns true, and that helper checks `sql.trimStart().toLowerCase().startsWith("insert")`. Kysely compiles valid insert builders with CTEs as `with "src" as (...) insert into ...`, so the query changes one row but returns `InsertResult { insertId: undefined, numInsertedOrUpdatedRows: 1n }`.
- Impact: Callers using Kysely `.with(...).insertInto(...).executeTakeFirst()` against an autoincrement table can lose the generated id even though the insert succeeded. Plain inserts still work; this is an edge case around CTE inserts.
- Suggested fix: Decide insert-ness from `compiledQuery.query` using Kysely’s operation node kind, e.g. `InsertQueryNode.is(compiledQuery.query)`, instead of SQL prefix text. Keep the `changes > 0` guard for ignored/conflicted inserts.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A local `node --import tsx` smoke created an in-memory SQLite DB and ran a Kysely CTE insert; the compiled SQL began with `with`, the row was inserted, and `insertId` was undefined.
- Is this the best way to solve the issue? yes. Kysely already carries the typed root operation node on `CompiledQuery`, which avoids SQL-string parsing edge cases while preserving the stale `lastInsertRowid` fix.
