# Commit 56024b7

## Summary

Found one cancellation-semantics regression risk. The stale-recovery path is narrow and covered, but the same predicate also makes fresh childless Codex-native task mirrors report as successfully cancelled even though OpenClaw has no child session or Codex app-server interrupt path for that mirror.

## Findings

### Medium: `tasks cancel` can mark a live Codex-native subagent cancelled without stopping it

- Kind: regression
- File: `src/tasks/task-registry.ts`
- Line: 1911
- Evidence: `cancelTaskById` now lets childless Codex-native mirrors bypass the old “no cancellable child session” return and falls through to `updateTask(... status: "cancelled")`. These mirrors are intentionally created without `childSessionKey` in `extensions/codex/src/app-server/native-subagent-task-mirror.ts`, so this path does not call ACP cancel, `killSubagentRunAdmin`, or any Codex app-server interrupt. A focused smoke confirmed the durable effect: after cancelling such a task, a later `finalizeTaskRunByRunId(... status: "succeeded")` leaves the task `cancelled` with no terminal summary.
- Impact: An operator or RPC client can cancel a fresh, still-running Codex-native subagent mirror and receive success, while the native Codex work may continue. Later native completion cannot correct the task status, so task state can hide active/completed work and mislead audits or status UIs.
- Suggested fix: Either wire cancellation to an authoritative Codex app-server interrupt for the native thread, or keep registry-only cancellation limited to stale mirrors that have exceeded the same no-progress grace threshold. If this is only cleanup, consider marking stale records `lost`/cleared rather than `cancelled`, or allow later native terminal events to override registry-only cancellation when no real interrupt occurred.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Create a running `runtime: "subagent"`, `taskKind: "codex-native"`, `runId/sourceId: "codex-thread:..."` task with no `childSessionKey`; call `cancelTaskById`; then call `finalizeTaskRunByRunId` with `status: "succeeded"`. The smoke returned `{"cancelled":true,"finalStatus":"cancelled","terminalSummary":null}`.
- Is this the best way to solve the issue? unclear. Registry-only cleanup is reasonable for stale abandoned mirrors, but reporting fresh work as cancelled without an actual cancellation backend changes the user-visible contract documented for `openclaw tasks cancel`.
