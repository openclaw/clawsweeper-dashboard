# Commit 1466878

## Summary

Found one regression in the cached descriptor path: a cached plugin tool can be advertised after the owning runtime registry is no longer loaded, then fail at execution time.

## Findings

### Medium: Cached descriptors can expose plugin tools without an executable runtime

- Kind: regression
- File: `src/plugins/tools.ts`
- Line: 533
- Evidence: `resolveCachedPluginTools` accepts cached descriptors using only manifest availability/cache coverage, then marks the plugin handled at `src/plugins/tools.ts:596`. It never verifies that the owning runtime registry is still loaded. The wrapper later calls `resolvePluginToolRegistry` during `execute`, and if the active/channel registry was replaced, it throws `plugin tool runtime unavailable` at `src/plugins/tools.ts:451`.
- Impact: prompt-time planning can show a plugin tool that cannot execute. The old path first resolved a loaded registry before returning any tool; after this commit, a stale descriptor cache can bypass that guard.
- Suggested fix: make cached execution actually load or rehydrate the owning plugin runtime when missing, for example by using the standalone/runtime loader on the execution path before looking up the factory. Alternatively, only return cached descriptor wrappers when a compatible runtime registry for the plugin is present.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. I ran an in-memory smoke script that installed a manifest snapshot, set an active registry with `demo_tool`, called `resolvePluginTools` once to populate the descriptor cache, replaced the active registry with an empty registry, then called `resolvePluginTools` again. The second call still returned `["demo_tool"]`, and executing it rejected with `plugin tool runtime unavailable (demo): demo_tool`.
- Is this the best way to solve the issue? yes, directionally. The commit’s stated contract says planning can skip plugin runtime loading while execution still loads the live plugin tool; the current wrapper only looks for an already-loaded registry, so the fix should make execution load/rehydrate the owner or avoid advertising the cached wrapper when execution cannot be resolved.
