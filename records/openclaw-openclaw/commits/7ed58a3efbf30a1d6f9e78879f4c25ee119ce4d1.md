# Commit 7ed58a3

## Summary

The resolved Discord setup allowlist path is fixed, but the new mapper drops every unresolved setup row. That leaves a concrete setup failure when lookup cannot run, including the existing no-token fallback path.

## Findings

### Medium: Unresolved Discord setup allowlist entries are silently discarded

- Kind: regression
- File: `extensions/discord/src/setup-core.ts`
- Line: 52
- Evidence: `mapDiscordSetupAllowlistEntries` returns `[]` for every row with `resolved === false`. The Discord setup resolver emits exactly that shape when no usable token is available (`extensions/discord/src/setup-surface.ts:96` and `extensions/discord/src/resolve-channels.ts:158`). The setup helper then still sets `groupPolicy: "allowlist"` and applies the empty mapped allowlist. A focused runtime check of `groupAccess.applyAllowlist` with unresolved rows for `guild:111` and `111/222` produced `{"enabled":true,"guilds":{}}`.
- Impact: Users who configure Discord setup with a SecretRef/env token unavailable in the current process, or who hit a transient Discord lookup failure, can enter deterministic guild/channel IDs and finish setup with an empty guild allowlist. Runtime allowlist mode then fails closed for guild messages, so the configured server/channel never works.
- Suggested fix: Preserve parseable raw ID fallback rows instead of dropping all unresolved rows. For example, map unresolved `guild:<id>`, `<guildId>/<channelId>`, and maybe `channel:<id>` only when they are syntactically deterministic; continue ignoring ambiguous unresolved names.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Invoke the setup group allowlist apply path with unresolved fallback rows, as produced by no-token resolution: `[{ input: "guild:111", resolved: false }, { input: "111/222", resolved: false }]`. Current output is `channels.discord.guilds: {}`.
- Is this the best way to solve the issue? yes, with a narrow fallback parser. It preserves the new fix for resolved `guildId`/`channelId` rows without reintroducing the previous wildcard behavior for unknown shapes.
