# Commit 28c852f

## Summary

Found one low-severity reliability regression in the new error-detail formatter.

## Findings

### Low: Non-Error thrown values can bypass the secrets handlerâ€™s generic failure response

- Kind: reliability
- File: `src/gateway/server-methods/secrets.ts`
- Line: 12
- Evidence: `errorMessage(error)` calls `String(error)` for non-`Error` values. Some valid JavaScript thrown values, such as `Object.create(null)`, cannot be stringified this way and throw `TypeError: Cannot convert object to primitive value`. A focused smoke against `createSecretsHandlers` confirmed that `secrets.reload` rejected before calling `respond` when `reloadSecrets` threw that value.
- Impact: before this commit, any reload/resolve exception reached the generic `secrets.reload failed` or `secrets.resolve failed` response. Now an unusual non-`Error` rejection can turn the error handler itself into the failure path, producing an outer handler error instead of the intended generic secrets RPC error.
- Suggested fix: replace the local helper with the existing safe `formatErrorMessage` path, or wrap non-`Error` formatting so it cannot throw.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes: inject `reloadSecrets: async () => { throw Object.create(null); }` into `createSecretsHandlers` and call `secrets.reload`; the handler rejects with `Cannot convert object to primitive value` and does not call its own `respond`.
- Is this the best way to solve the issue? yes: using the repoâ€™s existing safe error formatter preserves the intended diagnostic logging while keeping the failure response path non-throwing.
