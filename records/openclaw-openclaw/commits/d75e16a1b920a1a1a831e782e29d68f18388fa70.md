# Commit d75e16a

## Summary

Found two actionable reliability/concurrency issues in the new stale-run filter for the Mantis Telegram workflow queue.

## Findings

### Medium: Stale queued runs can still start later and overlap the newer run that skipped them

- Kind: concurrency
- File: `.github/workflows/mantis-telegram-desktop-proof.yml`
- Line: 321
- Evidence: The new filter excludes every older run whose `.createdAt < $stale_before`, across `queued`, `in_progress`, `waiting`, `pending`, and `requested`. The same logic exists in `.github/workflows/mantis-telegram-live.yml:285`. But each run only waits for runs older than its own `current_created`; if an older queued/requested run is skipped as stale and later receives a runner, that older run will not wait for the newer run because the newer run has a later `createdAt`.
- Impact: Two secret-bearing Mantis Telegram jobs can use the shared Telegram account at the same time, corrupting evidence or active sessions. This undercuts the test’s stated contract that these workflows “serialize all Mantis Telegram account runs.”
- Suggested fix: Do not treat “old” as equivalent to “safe to ignore” unless the stale run is also cancelled or externally locked out. Safer options: cancel/supersede stale queued/requested/waiting runs before proceeding, use a real shared account lock/lease, and always block on active `in_progress` runs regardless of age.
- Confidence: high

### Medium: The stale cutoff is frozen, so blockers that age out while waiting remain blockers until timeout

- Kind: reliability
- File: `.github/workflows/mantis-telegram-desktop-proof.yml`
- Line: 311
- Evidence: `stale_before` is computed once before `while true`, then reused on every poll. A blocker created just after that cutoff, for example 7h59m old when this step starts, continues matching `.createdAt >= $stale_before` even after it becomes older than eight hours. The same frozen cutoff exists in `.github/workflows/mantis-telegram-live.yml:275`.
- Impact: A run can still wait until the job timeout on a stuck older run, so the commit does not reliably fix stale Telegram queue blockers except for blockers already stale before the wait step begins.
- Suggested fix: Recompute `stale_before` inside the polling loop, or compute age from the current poll time before applying the blocker filter.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Static tracing shows the newer run skips stale older runs, while the older run’s ordering rule ignores newer runs. A focused `jq` check also confirmed a blocker created one minute after the fixed cutoff remains selected even after it should have aged out under a moving cutoff.
- Is this the best way to solve the issue? yes. The fix direction should preserve serialization with cancellation or a real lock, and recompute the age cutoff per poll if age-based ignoring remains.
