# Commit f2e342b

## Summary

Found one regression in the update-time deferral path: it deletes the only install metadata for configured third-party plugins whose payload is missing, so a later doctor or plugin update can no longer repair them.

## Findings

### Medium: Update-time deferral drops third-party plugin install records

- Kind: data_loss
- File: `src/commands/doctor/shared/missing-configured-plugin-install.ts`
- Line: 526
- Evidence: During `OPENCLAW_UPDATE_IN_PROGRESS=1`, the new loop deletes any configured plugin record whose `installPath` no longer has `package.json`, then persists `nextRecords` at lines 609-610. The committed test for Discord asserts the index is written as `{}` in this path (`src/commands/doctor/shared/missing-configured-plugin-install.test.ts:617`). That is safe only for catalog-backed plugins like Discord. The existing non-update test at `src/commands/doctor/shared/missing-configured-plugin-install.test.ts:905` shows arbitrary plugin `demo` is repaired solely from its persisted record (`spec: "@openclaw/plugin-demo@1.0.0"`), via `updateNpmInstalledPlugins`. Once the update-time path deletes that record, a later non-update doctor has no catalog candidate and no recorded npm/git/ClawHub spec to reinstall.
- Impact: Users with configured third-party plugins can lose the install record during a package/git update doctor pass if the plugin payload is temporarily absent or moved. Subsequent `openclaw doctor --fix` or `openclaw plugins update --all` cannot reconstruct the package source, leaving the configured plugin broken until the user manually reinstalls it from memory.
- Suggested fix: Defer by skipping update/install work for missing records while `OPENCLAW_UPDATE_IN_PROGRESS=1`, but do not delete or persist-delete the record. Keep the record available for the later doctor/plugin-update pass. If a cleanup write is needed for bundled plugins, keep that separate from external missing-payload deferral.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. Use the existing `demo` persisted-record repair scenario, add `env: { OPENCLAW_UPDATE_IN_PROGRESS: "1" }`, no catalog entries, and a missing `installPath`. Current code will write `{}` instead of preserving `demo`; the next non-update repair has no source spec left.
- Is this the best way to solve the issue? No. The current implementation conflates “skip package-manager work during update” with “remove install metadata.” Preserving the record while blocking immediate repair keeps the update safe without losing repairability.
