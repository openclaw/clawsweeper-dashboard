# Commit 4047f4d0b4

## Summary

Found one recovery-path gap: the new install-record loader can recover managed npm records, but registry snapshot/refresh paths still bypass that recovery by passing an explicit empty map from the missing/empty persisted index.

## Findings

### Medium: Plugin registry refresh bypasses recovered npm install records

- Kind: bug
- File: `src/plugins/plugin-registry-snapshot.ts`
- Line: 246
- Evidence: `loadInstalledPluginIndexInstallRecordsSync({ stateDir })` recovers managed npm records from `stateDir/npm`, but `loadPluginRegistrySnapshotWithMetadata` and `refreshPersistedInstalledPluginIndex` pass `installRecords: extractPluginInstallRecordsFromInstalledPluginIndex(persistedIndex)` even when the persisted index is missing/empty. That returns `{}`, so `loadInstalledPluginIndex` never calls the recovered loader. A focused smoke with a managed npm plugin and no `plugins/installs.json` printed `direct records [ 'review-ledger-plugin' ]`, then `refreshPluginRegistry(...)` produced `refreshed records []` and `refreshed has fake plugin false`.
- Impact: `openclaw plugins registry --refresh` and derived registry consumers can continue to omit recovered managed npm plugins after ledger loss, so plugin list/registry repair can still show a broken or empty view even though the direct install-record loader recovered the package. This undercuts the commit’s stated empty/partial ledger recovery path.
- Suggested fix: route registry snapshot/refresh fallback through `loadInstalledPluginIndexInstallRecordsSync` when no caller-supplied `installRecords` exists, or merge recovered managed npm records with records extracted from the persisted index before passing them into `loadInstalledPluginIndex`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Create a temp state dir with `npm/package.json` containing a direct plugin dependency and `npm/node_modules/<package>/package.json` plus `openclaw.plugin.json`, leave `plugins/installs.json` missing, then compare `loadInstalledPluginIndexInstallRecordsSync({ stateDir })` with `refreshPluginRegistry({ stateDir, config, reason: "manual" })`.
- Is this the best way to solve the issue? yes, directionally. The recovery needs to be available at the registry layer, not only at direct install-record readers, because the registry layer intentionally passes an explicit record map today.
