# Commit dd09e6fe40

## Summary

Found one concrete gap: the new `supportsTools: false` metadata only reaches newly built Arcee catalog entries. Existing users who already onboarded Arcee can keep stale `trinity-large-thinking` model entries without the compat flag, so the reported main-session tool/request-shape failure remains reachable for exactly the users this fix is meant to protect.

## Findings

### High: Existing Arcee configs are not repaired, so stale Trinity entries still enable tools

- Kind: bug
- File: `extensions/arcee/index.ts`
- Line: 123
- Evidence: `extensions/arcee/models.ts:48` adds `compat.supportsTools: false` to the bundled catalog, but `extensions/arcee/index.ts:123` only normalizes stale OpenRouter base URLs and never adds the new compat flag to existing configured `models.providers.arcee.models` entries. Existing onboarding persists catalog models into config via `extensions/arcee/onboard.ts:21` and `extensions/arcee/onboard.ts:32`; rerunning onboarding preserves existing model objects over refreshed catalog rows in `src/plugin-sdk/provider-onboard.ts:395`. - Impact: A user onboarded before this commit can still run `arcee/trinity-large-thinking` with tools enabled and hit the same main-session request shape that #62851 was closed as fixed for. The commit’s new tests only cover fresh direct/OpenRouter catalog construction, not stale config normalization or resolved runtime model compat.
- Suggested fix: Add an Arcee-owned stale-config/resolved-model compat repair for `trinity-large-thinking` that merges `supportsTools: false` into existing direct and OpenRouter-backed config/runtime models, preserving other compat keys. Add regression coverage for stale direct config, stale OpenRouter config, and resolved model normalization.
- Confidence: high
