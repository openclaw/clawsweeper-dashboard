# Commit c35634c

## Summary

The catalog pin and npm registry metadata agree for `@tencent-weixin/openclaw-weixin@2.4.3`, but the official-plugin update path does not actually carry the new catalog `expectedIntegrity` into the npm installer when this commit changes the official spec from `2.4.1` to `2.4.3`.

## Findings

### Medium: Official catalog updates drop the new expected integrity when the pinned spec changes

- Kind: supply_chain
- File: `src/plugins/update.ts`
- Line: 1018
- Evidence: This commit adds the 2.4.3 exact npm pin and integrity in `scripts/lib/official-external-channel-catalog.json:117`. `resolveOfficialExternalPluginInstall()` preserves catalog `expectedIntegrity` at `src/plugins/official-external-plugin-catalog.ts:166`, but `updateNpmInstalledPlugins()` computes `expectedIntegrity` only when `effectiveSpec === record.spec` at `src/plugins/update.ts:1018`. For a user installed from the previous catalog, `record.spec` is `@tencent-weixin/openclaw-weixin@2.4.1` and `syncOfficialPluginInstalls` makes `effectiveSpec` `@tencent-weixin/openclaw-weixin@2.4.3`, so the installer calls at `src/plugins/update.ts:1168` and `src/plugins/update.ts:1369` receive `expectedIntegrity: undefined`.
- Impact: The catalog now says the trusted update target is exact-with-integrity, and docs state official exact specs should fail closed on npm artifact drift, but `openclaw update` / `openclaw plugins update` can install the 2.4.3 tarball without comparing it to the catalog pin during this official-spec migration.
- Suggested fix: Return or look up the full official install metadata alongside `officialNpmSpec`, and when the effective update spec equals the official catalog `npmSpec`, pass `officialInstall.expectedIntegrity` to `installPluginFromNpmSpec`. Add a regression test for an `openclaw-weixin` install record pinned to 2.4.1 with `syncOfficialPluginInstalls: true`, asserting the installer receives the 2.4.3 catalog integrity.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. Source-level reproduction is enough: use an install record with `pluginId: "openclaw-weixin"`, `source: "npm"`, `spec: "@tencent-weixin/openclaw-weixin@2.4.1"`, and run `updateNpmInstalledPlugins({ syncOfficialPluginInstalls: true })`; the current branch selects the 2.4.3 official spec but computes `expectedIntegrity` as `undefined`.
- Is this the best way to solve the issue? Yes. The catalog already owns the trusted official npm pin and integrity; the safer fix is to carry that existing contract into the update call, not to weaken catalog policy or rely on recorded old-version integrity.
