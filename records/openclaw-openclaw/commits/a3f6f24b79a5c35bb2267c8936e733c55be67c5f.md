# Commit a3f6f24b79

## Summary

Found one release-validation regression: the new Slack credential gate cannot re-enable the Slack QA lane for the default “all QA live lanes” release path, even when the repository variable is set to `true`.

## Findings

### Medium: Slack QA stays skipped for default full release checks after the opt-in variable is enabled

- Kind: regression
- File: `.github/workflows/openclaw-release-checks.yml`
- Line: 217
- Evidence: `resolve_target` now initializes `qa_live_slack_enabled=false`. The normalized `qa_live_slack_ci_enabled` value is only assigned when the filter token is explicitly `qa-live-slack|qa-slack|slack` at lines 257-259. For the blank filter, which the workflow input describes as “runs all selected live suites”, no filter block runs, so Slack remains false. For `qa-live|qa-live-all|qa-all`, lines 239-243 select Matrix and Telegram but no longer set Slack, so Slack also remains false. The job condition at line 892 then skips `qa_live_slack_release_checks` despite `vars.OPENCLAW_QA_SLACK_LIVE_CI_ENABLED == 'true'`.
- Impact: Once Slack CI credentials are provisioned and `OPENCLAW_QA_SLACK_LIVE_CI_ENABLED=true` is set, broad release checks (`rerun_group=all`, `qa`, or `qa-live` with blank filter) still silently omit Slack coverage. Only an explicit `live_suite_filter=qa-live-slack` can run it. That regresses the release workflow’s “blank runs all selected live suites” contract and can let Slack-specific release breakage miss normal full validation.
- Suggested fix: Use the normalized gate as the Slack default and all-QA selection, for example initialize `qa_live_slack_enabled="$qa_live_slack_ci_enabled"` and set `slack_selected="$qa_live_slack_ci_enabled"` for `qa-live|qa-live-all|qa-all`. Prefer one normalized source of truth for the job condition instead of also requiring the raw repo variable to equal exactly `'true'`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A local shell simulation of the changed `Capture selected inputs` logic produced `filter='' gate=true => ... slack=false` and `filter=qa-live gate=true => ... slack=false`, while `filter=qa-live-slack gate=true => ... slack=true`.
- Is this the best way to solve the issue? yes. Applying the normalized gate to the default/all filter paths preserves the new “disabled until credentials are provisioned” behavior without breaking broad release validation once the credentials gate is enabled.
