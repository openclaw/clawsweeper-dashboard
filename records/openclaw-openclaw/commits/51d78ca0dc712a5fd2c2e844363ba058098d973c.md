## Summary

Found one reliability regression in the Parallels npm-update smoke path: the shared model timeout now allows 30 minutes on macOS/Windows, but the aggregate update harness still kills those update lanes after 20 minutes.

## Findings

### Low: npm-update Parallels lanes still time out before the new GPT-5.5 timeout

- Kind: reliability
- File: `scripts/e2e/parallels/provider-auth.ts`
- Line: 85
- Evidence: This commit raises `resolveParallelsModelTimeoutSeconds("macos" | "windows")` to `1800`, but `scripts/e2e/parallels/npm-update-smoke.ts:68` still defaults `OPENCLAW_PARALLELS_NPM_UPDATE_TIMEOUT_S` to `1200`. The npm-update macOS lane applies the OpenAI provider timeout before its agent turn in `scripts/e2e/parallels/npm-update-scripts.ts:134`, and the Windows lane passes `--timeout ${resolveParallelsModelTimeoutSeconds("windows")}` at `scripts/e2e/parallels/npm-update-scripts.ts:218`. Both are run under the 1200-second outer update timeout at `scripts/e2e/parallels/npm-update-smoke.ts:372` and `scripts/e2e/parallels/npm-update-smoke.ts:376`.
- Impact: `pnpm test:parallels:npm-update` can still fail the macOS/Windows GPT-5.5 update lanes after 20 minutes, before the newly intended 30-minute model timeout can expire. That keeps the aggregate update smoke flaky on the same slow desktop guests this commit is trying to accommodate.
- Suggested fix: raise the npm-update default timeout alongside the model timeout, or compute it from the selected platform/model timeout plus update/gateway overhead. A fixed default in the same range as the standalone OS smoke agent phase, at least `2700`, would avoid this immediate timeout inversion.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the timeout inversion: evaluating the current helpers with no env overrides returns `updateTimeoutSeconds: 1200`, `macosProviderTimeoutSeconds: 1800`, and `windowsCliTimeoutSeconds: 1800`. A full live failure requires a macOS or Windows npm-update GPT-5.5 agent turn that runs past 1200 seconds.
- Is this the best way to solve the issue? The safest fix is to align the aggregate npm-update outer timeout with the largest inner model timeout plus expected update/gateway overhead, because the lower outer timeout currently wins regardless of the provider timeout.
