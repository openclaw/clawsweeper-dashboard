# Commit ec67290

## Summary

Found one regression in the new schema traversal: some valid schema-bearing keywords are now treated as opaque values, so local refs under those keywords are left dangling after `$defs` is removed.

## Findings

### Medium: Schema refs under unevaluated/additional/dependencies keywords are no longer inlined

- Kind: regression
- File: `src/agents/pi-tools-parameter-schema.ts`
- Line: 418
- Evidence: The new `SCHEMA_OBJECT_KEYS` / `SCHEMA_MAP_KEYS` traversal lists omit schema-bearing keywords such as `unevaluatedProperties`, `unevaluatedItems`, `additionalItems`, and draft-07 `dependencies`. The old implementation recursively processed every non-definition key, but the new code copies unrecognized keys unchanged while still dropping top-level `$defs`. A focused probe on current `main`:
  `normalizeToolParameterSchema({ type: "object", unevaluatedProperties: { $ref: "#/$defs/Foo" }, $defs: { Foo: { type: "string" } } })`
  returns `{"type":"object","unevaluatedProperties":{"$ref":"#/$defs/Foo"},"properties":{}}`. Compiling that normalized schema with Ajv 2020 fails with `can't resolve reference #/$defs/Foo from id #`.
- Impact: Plugin/MCP/OpenAPI 3.1 tool schemas that use local refs under these valid JSON Schema keywords can now be sent to providers with unresolved `$ref`s and no corresponding `$defs`, causing provider-side schema rejection or broken validation. TypeBox 1.1.38 in this checkout has runtime/type support for `unevaluatedProperties`, `unevaluatedItems`, and `additionalItems`, and OpenClaw’s own OpenAI strict compat helper already treats `unevaluatedItems` / `unevaluatedProperties` as nested schema keywords.
- Suggested fix: Add the missing schema-bearing keywords to the traversal model. At minimum, recurse object-valued `unevaluatedProperties`, `unevaluatedItems`, and `additionalItems`, and recurse schema-valued entries under `dependencies` while preserving dependency string arrays.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The direct `tsx` probe above shows the dangling `$ref`, and the Ajv 2020 compile check rejects the normalized schema.
- Is this the best way to solve the issue? yes. The new selective traversal is the right direction for preserving literals, but the keyword allowlists need to cover the schema-bearing keywords the runtime already supports.
