# Commit 36e76ef

## Summary

Found two concrete regressions in the new completion classifier plumbing. Both can leave successful child work recorded as blocked.

## Findings

### Medium: Refreshed subagent final output does not clear an earlier blocked terminal outcome

- Kind: regression
- File: `src/agents/subagent-registry-lifecycle.ts`
- Line: 284
- Evidence: `safeFinalizeSubagentTaskRun` passes `terminalOutcome: terminalResult.terminalOutcome`. For a later real final report, that value is `undefined`; `src/tasks/task-registry.ts:1685` only updates terminal outcome when the param is not `undefined`, so an earlier `"blocked"` value remains. This matters because `refreshFrozenResultFromSession` can replace progress/missing frozen output with later real output for pending completion runs at `src/agents/subagent-registry-lifecycle.ts:387`.
- Impact: a required subagent can first be marked blocked due progress-only/missing output, later capture and deliver a real final answer, but the durable task/flow remains blocked. Parent review/UI can keep prompting follow-up even though the final deliverable arrived.
- Suggested fix: when a required completion is now accepted as real final output, pass an explicit clearing outcome such as `terminalOutcome: null` or `"succeeded"` through `completeTaskRunByRunId`, and add a regression where initial progress-only output is refreshed to final output and the task no longer has `terminalOutcome: "blocked"`.
- Confidence: high

### Medium: ACP completion classification uses the 240-character display summary instead of full output

- Kind: regression
- File: `src/acp/control-plane/manager.core.ts`
- Line: 129
- Evidence: ACP output is accumulated into `taskProgressSummary`, but `appendBackgroundTaskProgressSummary` caps that value at 240 chars (`src/acp/control-plane/manager.core.ts:95`, `src/acp/control-plane/manager.core.ts:118`). The new required-completion classifier is then run on that capped summary at `src/acp/control-plane/manager.core.ts:129` and persisted at `src/acp/control-plane/manager.core.ts:970`. A focused probe with a long “I’ll inspect...” preamble followed by “The fix is...” showed the capped summary ended before the final text and `resolveRequiredCompletionTerminalResult` returned `terminalOutcome: "blocked"`.
- Impact: valid ACP completions can be marked blocked whenever a verbose progress preamble pushes the actual final deliverable past the 240-character display cap. That contradicts the PR’s “preserve real final reports” goal for longer outputs.
- Suggested fix: keep a separate uncapped terminal-output buffer, or consume structured final output from the ACP runtime, and reserve the 240-character value only for UI progress summary.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. I reproduced the stale blocked outcome with a focused task-registry probe: complete a run once with `terminalOutcome: "blocked"`, then complete it again with final text and `terminalOutcome: undefined`; the record kept `terminalOutcome: "blocked"`. I also reproduced the ACP classifier failure with the same 240-character cap logic and the new helper.
- Is this the best way to solve the issue? Unclear for ACP because structured final output would be better than text heuristics. For the subagent stale outcome, explicitly clearing or setting the terminal outcome on accepted final output is the narrow fix.
