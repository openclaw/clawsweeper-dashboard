# Commit aec83af

## Summary

Found one reliability regression in the new bounded chat history path: current tree-format transcripts still take the old unbounded full-file read path.

## Findings

### Medium: Current tree transcripts bypass the bounded reader

- Kind: reliability
- File: `src/gateway/session-utils.fs.ts`
- Line: 227
- Evidence: `chat.history` now calls `readRecentSessionMessages` from `src/gateway/server-methods/chat.ts:1672`, but `readRecentSessionMessages` falls back to `readSessionMessages(...).slice(-maxMessages)` whenever the sampled tail contains a tree entry. Current Pi/Gateway transcript entries are tree entries: dependency types define every `SessionEntryBase` with `id` and `parentId`, and Gateway transcript writes use `SessionManager.appendMessage` in `src/config/sessions/transcript.ts:279` and `src/gateway/server-methods/chat-transcript-inject.ts:105`. `SessionManager.open()` loads the whole JSONL via `loadEntriesFromFile`, so the fallback defeats the intended byte bound for normal current transcripts.
- Impact: Large current-format session logs can still force `chat.history` to read and parse the entire transcript, preserving the Gateway OOM/freeze mode this commit is meant to fix. The new tests cover legacy append-only lines without `parentId`, so they miss the common current format.
- Suggested fix: Add a bounded path for tree transcripts instead of calling `readSessionMessages` unconditionally. At minimum, detect append-only linear tree tails and project those bounded entries directly; only use the full branch load when the tail proves active-branch ambiguity that cannot be resolved from the bounded window.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused local repro created a transcript with `SessionManager.create()` and 1500 user/assistant pairs, patched `SessionManager.open` to count calls, then called `readRecentSessionMessages(..., { maxMessages: 1, maxBytes: 64 * 1024 })`. Output showed `openCalls: 1` on a 3.25 MB tree transcript, proving the bounded reader entered the full `SessionManager.open` fallback.
- Is this the best way to solve the issue? no. The commit bounds legacy append-only transcripts, but current tree transcripts need their own bounded implementation or the fix does not cover the normal Gateway transcript format.
