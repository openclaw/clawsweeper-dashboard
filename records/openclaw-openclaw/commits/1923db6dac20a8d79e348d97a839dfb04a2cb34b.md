# Commit 1923db6

## Summary

Found two low-severity regressions in `openclaw models list` configured-entry resolution. The runtime model-selection fix itself is covered by tests, but the CLI listing path now mixes alias resolution and proxy-provider normalization in ways that can hide or split configured rows.

## Findings

### Low: configured model keys can be swallowed by aliases

- Kind: regression
- File: `src/commands/models/list.configured.ts`
- Line: 76
- Evidence: The configured-model key loop now calls `resolveModelRefFromString({ cfg, raw: key, aliasIndex, ... })`. That resolver checks aliases before parsing provider/model refs. With valid config such as `"openai/gpt-5.5": { alias: "anthropic/claude-sonnet-4-6" }` plus an explicit `"anthropic/claude-sonnet-4-6"` entry, `resolveConfiguredEntries()` returns only `openai/gpt-5.5`; the Anthropic configured row disappears.
- Impact: `openclaw models list` can omit an explicitly allowlisted model when another configured alias has the same text, including slash-form aliases that the model resolver already supports.
- Suggested fix: Resolve `agents.defaults.models` declaration keys with a literal key parser that still handles exact configured proxy-provider normalization, but does not consult `aliasIndex.byAlias`.
- Confidence: high

### Low: fallback rows still bypass proxy-provider normalization

- Kind: bug
- File: `src/commands/models/list.configured.ts`
- Line: 45
- Evidence: `addResolvedModelRef()` resolves defaults/fallback/image refs without passing `cfg`, while the configured-key loop now does pass `cfg`. A smoke check with `fallbacks: ["kilocode/google/gemini-3-pro-preview"]` and the same configured model returns separate rows for `kilocode/google/gemini-3-pro-preview` and `kilocode/google/gemini-3.1-pro-preview`.
- Impact: `openclaw models list` can show the fallback tag on the retired Gemini ref and the configured/alias tags on the canonical ref, making one configured model look like two different models.
- Suggested fix: Pass `cfg` through `addResolvedModelRef()` so fallback/image refs use the same exact configured proxy-provider normalization as configured rows.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Both issues reproduce with `resolveConfiguredEntries()` using small in-memory configs.
- Is this the best way to solve the issue? yes for the direction: declaration keys should not resolve through aliases, while fallback/image references should receive the same `cfg`-aware normalization as other config model refs.
