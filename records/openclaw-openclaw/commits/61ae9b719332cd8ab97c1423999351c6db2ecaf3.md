# Commit 61ae9b7

## Summary

Found two actionable bugs in the new pre-update snapshot safety net. The strip guard and update-time size-drop guard look sound, and focused tests pass, but the snapshot can be written for the wrong config path and then becomes stale forever after its first creation.

## Findings

### Medium: Post-install doctor can mutate a different config than the one snapshotted

- Kind: data_loss
- File: `src/cli/update-cli/update-command.ts`
- Line: 1038
- Evidence: `createUpdateConfigSnapshot()` always snapshots module `CONFIG_PATH` (`src/cli/update-cli/update-command.ts:144`), then the post-install doctor child is launched with `resolvePostInstallDoctorEnv(...)` (`src/cli/update-cli/update-command.ts:1043`). That helper intentionally lets managed service env override `OPENCLAW_STATE_DIR`, `OPENCLAW_CONFIG_PATH`, and `OPENCLAW_PROFILE` (`src/cli/update-cli/update-command.ts:761`), and the existing test proves a service-relative `OPENCLAW_CONFIG_PATH` replaces the caller path (`src/cli/update-cli/update-command.test.ts:119`).
- Impact: package updates for a managed service with a custom config path can snapshot the caller/default config while the update doctor repairs and writes the service config. If that service config is damaged, the new `.pre-update` recovery file may not exist beside the affected file.
- Suggested fix: build the post-install doctor env first, resolve the config path from that same env, and pass that path into the snapshot writer before launching the child. Apply the same pattern to restart paths that operate with `params.serviceEnv`.
- Confidence: high

### Medium: `.pre-update` is first-write-wins across all future updates

- Kind: reliability
- File: `src/config/backup-rotation.ts`
- Line: 128
- Evidence: the snapshot path is fixed as `${configPath}.pre-update`, written with `flag: "wx"`, and `EEXIST` is ignored (`src/config/backup-rotation.ts:128`). The new test explicitly locks in that a second call never updates the snapshot (`src/config/config.backup-rotation.test.ts:197`).
- Impact: after one successful update, later updates cannot capture their own pre-update state. A future repair failure would leave users with an old snapshot that may omit agents, channels, credentials references, or plugin state added since the first update.
- Suggested fix: keep first-write-wins only within a single update run, but refresh or rotate snapshots between runs. A timestamped/bounded retention scheme, or replacing `.pre-update` at update start while preserving a small history, would preserve recovery value without racing the multiple writes inside one update.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes, source-level. The service-path mismatch follows from `resolvePostInstallDoctorEnv` and its existing unit test; the stale snapshot behavior is directly covered by the new first-write-wins test.
- Is this the best way to solve the issue? Yes. Both fixes keep the commit’s intended safety layers but make the snapshot correspond to the actual update target and the current update run.
