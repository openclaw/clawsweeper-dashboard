# Commit d8b82df5

## Summary

Found one concrete regression in the Mantis PR-comment publication path: the workflow now calls a REST endpoint that GitHub App installation tokens cannot use, so successful QA runs with a PR target can fail before upserting the PR evidence comment.

## Findings

### Medium: Mantis PR comment step cannot resolve bot login with an installation token

- Kind: regression
- File: `.github/workflows/mantis-discord-status-reactions.yml`
- Line: 454
- Evidence: The comment step sets `GH_TOKEN` to `steps.mantis_app_token.outputs.token`, then runs `mantis_bot_login="$(gh api user --jq '.login')"`. The `actions/create-github-app-token` README documents that this output is a GitHub App installation access token and exposes `app-slug` for deriving the bot identity; its own examples query `/users/${app-slug}[bot]`, not `/user`. GitHub’s REST docs for `GET /user` list GitHub App user access tokens and fine-grained PATs for that endpoint, not GitHub App installation access tokens. A focused local check with this session’s integration token reproduced the same failure: `gh api user --jq '.login'` returned HTTP 403 `Resource not accessible by integration`. ([github.com](https://github.com/actions/create-github-app-token))
- Impact: Any Mantis run with `pr_number` set, including the new PR-comment trigger, can complete the expensive live QA and publish `qa-artifacts`, then fail under `set -euo pipefail` before finding/updating the Mantis-owned PR comment. That makes the workflow report failure and withholds the inline QA evidence from the PR.
- Suggested fix: Do not call `gh api user`. Use `steps.mantis_app_token.outputs.app-slug` to derive `${app_slug}[bot]`, or restore an explicit bot-login variable fallback. If an API lookup is needed, query `/users/${app_slug}[bot]` as shown by the action README.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Run `GH_TOKEN=<GitHub App installation token> gh api user --jq '.login'`; in this checkout, the same endpoint returned HTTP 403 with `Resource not accessible by integration`.
- Is this the best way to solve the issue? yes. The workflow only needs the bot login for comment ownership filtering, and `actions/create-github-app-token@v3` already outputs the app slug, so deriving `<app-slug>[bot]` avoids an unsupported authenticated-user endpoint.
