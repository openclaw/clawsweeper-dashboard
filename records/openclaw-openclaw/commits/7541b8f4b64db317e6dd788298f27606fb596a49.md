# Commit 7541b8f

## Summary

Found one reliability regression in the new OpenAI realtime response-create guard: recoverable non-active `error` events can leave the bridge permanently refusing future `response.create` sends.

## Findings

### Medium: Recoverable realtime errors can wedge future manual responses

- Kind: reliability
- File: `extensions/openai/realtime-voice-provider.ts`
- Line: 794
- Evidence: `requestResponseCreate()` sets `responseCreateInFlight = true` before sending `response.create`, and later suppresses all future creates while `responseCreateInFlight` or `responseCancelInFlight` is true. The new handler clears those flags on `response.done`/`response.cancelled`, and clears only `responseCreateInFlight` for the specific active-response error prefix. For any other `error`, it only calls `onError` and returns, leaving in-flight guards set.
- Impact: OpenAI documents realtime `error` events as often recoverable with the session staying open, and errors may be tied to a specific client event. If a `response.create` or `response.cancel` receives a recoverable non-active error and no `response.done` follows, later `sendUserMessage()` or `submitToolResult()` calls only set `responseCreatePending`; they never send another `response.create`, so Discord/Voice Call/Talk relay can remain connected but stop speaking. OpenAI separately documents `response.done` as always emitted for a Response that exists, but an `error` for a rejected client event is not itself such a completed Response. ([platform.openai.com](https://platform.openai.com/docs/api-reference/realtime-server-events/error?lang=node&utm_source=openai))
- Suggested fix: track client `event_id`s for `response.create` and `response.cancel`, clear the matching in-flight guard when an `error.error.event_id` reports that event failed, and flush pending creates when no response is active. Add a unit test for: send `response.create`, receive a generic recoverable `error`, then call `sendUserMessage()` and assert a fresh `response.create` is sent.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes at the mocked websocket level: connect, call `triggerGreeting()` or `submitToolResult()` so `responseCreateInFlight` becomes true, inject a non-active `error` event, then call `sendUserMessage()`; current code will not send `response.create`.
- Is this the best way to solve the issue? Yes directionally: correlate provider errors to the guarded client events instead of clearing all guards on every unrelated error.
