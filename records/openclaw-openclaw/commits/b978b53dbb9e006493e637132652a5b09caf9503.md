# Commit b978b53

## Summary

Found one concrete policy regression in the new per-agent message cross-context override.

## Findings

### Medium: Legacy global cross-context allow bypasses per-agent deny

- Kind: security
- File: `src/infra/outbound/outbound-policy.ts`
- Line: 160
- Evidence: `resolveAgentMessageToolsConfig` merges `...globalConfig` into the per-agent effective config at `src/infra/outbound/outbound-policy.ts:106`, then merges `crossContext` separately. If the global config still has deprecated `tools.message.allowCrossContextSend: true`, that flag survives into `messageConfig`. `enforceCrossContextPolicy` then returns early on `messageConfig?.allowCrossContextSend` at `src/infra/outbound/outbound-policy.ts:160`, before reading the agent’s `crossContext.allowWithinProvider: false` or `allowAcrossProviders: false`. The new docs example at `docs/tools/multi-agent-sandbox-tools.md:55` uses only the canonical per-agent `crossContext` fields for a current-conversation-only public/family agent. A focused live check with global `allowCrossContextSend: true`, agent `crossContext` both false, and a same-provider send from `C123` to `C999` returned `allowed`.
- Impact: Existing installs that have not run the doctor migration from legacy `tools.message.allowCrossContextSend` can add the documented per-agent restriction and still let that agent send cross-context messages. For sandboxed or public agents, that widens message delivery beyond the current conversation and can post into unrelated chats.
- Suggested fix: Treat legacy `allowCrossContextSend` as a global default only when the agent does not provide its own message cross-context policy, or normalize the legacy flag into `crossContext` before merging so agent canonical settings override it. Add a regression test with global `allowCrossContextSend: true` and per-agent `crossContext` both false.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: the focused `pnpm exec tsx -e ... enforceCrossContextPolicy(...)` probe described above returned `allowed` for a cross-context send that the documented per-agent config should block.
- Is this the best way to solve the issue? yes: the effective policy resolver should make canonical per-agent settings authoritative over legacy global compatibility flags, rather than requiring users to set the deprecated field per agent.
