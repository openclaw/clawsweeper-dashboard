# Commit 772034d

## Summary

Found one concrete edge case in the new stripping wrapper: `parallel_tool_calls` can be re-added after the stripper runs when it is configured through extra params.

## Findings

### Medium: `parallel_tool_calls` can survive `supportsTools=false`

- Kind: bug
- File: `src/agents/pi-embedded-runner/extra-params.ts`
- Line: 691
- Evidence: The new `createOpenAICompletionsToolsCompatWrapper` deletes `tools`, `tool_choice`, and `parallel_tool_calls`, but it is installed before the later extra-param wrappers. `streamWithPayloadPatch` applies inner patches first, then calls the outer `onPayload`. Since `createParallelToolCallsWrapper` is installed later at `src/agents/pi-embedded-runner/extra-params.ts:745`, its patch runs after the new stripper and writes `payloadObj.parallel_tool_calls = enabled` at `src/agents/pi-embedded-runner/extra-params.ts:526`.
- Impact: A model configured with `api: "openai-completions"` and `compat.supportsTools: false` can still send the tool-related `parallel_tool_calls` field whenever inherited/model/runtime params set `parallel_tool_calls` or `parallelToolCalls`. The referenced issue is about chat-only OpenAI-compatible endpoints rejecting tool payload fields, so this leaves a plausible failure path for users who have that global or model-specific param set.
- Suggested fix: Move the tool-compat stripper after `createParallelToolCallsWrapper`, or make the parallel wrapper skip injection when `model.api === "openai-completions"` and `model.compat.supportsTools === false`. Add an integration test through `applyExtraParamsToAgent` with `supportsTools:false` plus `parallel_tool_calls:true`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Configure an `openai-completions` model with `compat.supportsTools: false`, set `agents.defaults.params.parallel_tool_calls` or a model override to `true`, then capture the outbound payload through the existing `onPayload` hook. Source ordering shows `parallel_tool_calls` is added after the strip patch.
- Is this the best way to solve the issue? yes. The stripper needs final precedence over generated tool-related fields; user pass-through via `extra_body` can remain explicit, but the generated `parallel_tool_calls` wrapper should not override `supportsTools:false`.
