## Summary

Found one concurrency/reliability issue in the new managed update handoff sentinel ownership path. Current `main` has no later changes to the relevant update files, so the issue still applies.

## Findings

### Medium: Stale CLI handoffs can overwrite a newer update sentinel

- Kind: concurrency
- File: `src/cli/update-cli/update-command.ts`
- Line: 2237
- Evidence: The gateway handoff path assigns a `handoffId` and writes it into the pending sentinel, and the helper script explicitly avoids overwriting a newer pending sentinel whose `stats.handoffId` differs. But once the detached CLI starts, `writeControlPlaneUpdateRestartSentinelBestEffort()` calls `writeControlPlaneUpdateRestartSentinel()` unconditionally, and that delegates to `writeRestartSentinel()` without checking the current sentinel owner. The failure marker path has the same issue: `markControlPlaneUpdateRestartSentinelFailure()` delegates to `markUpdateRestartSentinelFailure()`, which rewrites any current update sentinel without validating `handoffId`.
- Impact: If an old handoff CLI is still running and a newer `update.run` handoff has already written a new pending sentinel, the old CLI can replace the newer pending status with its own final `ok`, `error`, or `restart-unhealthy` result. That can make `update.status` report the wrong run, unblock/deliver the wrong continuation, or stop the restarted gateway from polling the newer pending handoff.
- Suggested fix: Make the CLI-side final and failure sentinel writes owner-aware. When `meta.handoffId` is present, read the current sentinel and only rewrite it if it is absent or has the same `stats.handoffId`; otherwise leave the newer handoff sentinel intact. Reuse the same guard semantics already implemented in `src/gateway/server-methods/update-managed-service-handoff.ts`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Unclear from live service proof, but the code path is direct: create pending sentinel A, create newer pending sentinel B with a different `handoffId`, then let CLI A hit either the final write at `src/cli/update-cli/update-command.ts:2846` or the restart failure marker at `src/cli/update-cli/update-command.ts:2837`.
- Is this the best way to solve the issue? Yes. The handoff helper already treats `handoffId` as the ownership boundary, so the CLI continuation should enforce the same contract instead of relying on whole-file replacement.
