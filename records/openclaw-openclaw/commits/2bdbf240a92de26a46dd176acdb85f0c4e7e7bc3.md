# Commit 2bdbf240a9

## Summary

Found one concrete bypass in the new staged-extension guard. The MIME detector normalizes headers case-insensitively, but the new image-header check does not, so ZIP bytes with `Content-Type: Image/PNG` still get staged as `.png`.

## Findings

### Medium: Uppercase image MIME headers still preserve image extensions for ZIP bytes

- Kind: bug
- File: `src/media/store.ts`
- Line: 337
- Evidence: `detectMime()` normalizes the header with `toLowerCase()` in `src/media/mime.ts:102` and reports the ZIP bytes as `application/zip`, but `isImageHeaderMime()` checks `startsWith("image/")` on the raw trimmed header at `src/media/store.ts:337`. A direct smoke check using ZIP bytes, `contentType: "Image/PNG"`, and `originalFilename: "fake.png"` returned `{"basename":"fake---<uuid>.png","contentType":"application/zip"}`.
- Impact: The commit’s intended protection can be bypassed by casing the image MIME header differently. Callers then receive a saved media path with an image extension even though the stored content is a generic container.
- Suggested fix: Normalize the header MIME before image/generic comparisons, preferably by reusing `normalizeMimeType()` or lowercasing the trimmed media type in `isImageHeaderMime()` and `extensionForAuthoritativeHeaderMime()`. Add uppercase/mixed-case image header coverage for buffer and stream paths.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. `saveMediaBuffer(zipBuffer, "Image/PNG", "inbound", 5 * 1024 * 1024, "fake.png")` saves a `.png` path while returning `contentType: "application/zip"`.
- Is this the best way to solve the issue? yes. The detector already treats MIME case as normalized; the extension trust gate should use the same normalization contract.
