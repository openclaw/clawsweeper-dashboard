# Commit 3a901b5e

## Summary

Found one high-confidence regression. The revert claims to restore `openai-codex/*` as a valid PI OAuth route, but it also restores doctor code/tests that still rewrite that route to `openai/*` plus PI. That reproduces the production lockout and possible unexpected OpenAI API billing described in #78407.

## Findings

### High: `doctor --fix` still rewrites valid Codex OAuth PI routes to direct OpenAI PI routes

- Kind: regression
- File: `src/commands/doctor/shared/codex-route-warnings.ts`
- Line: 56
- Evidence: Current docs and changelog introduced by this revert say `openai-codex/gpt-5.5` is the explicit Codex OAuth route through PI and that doctor should not rewrite it (`docs/providers/openai.md:35`, `docs/providers/openai.md:52`, `docs/providers/openai.md:230`, `CHANGELOG.md:177`). The code still maps `openai-codex/<model>` to `openai/<model>` in `toCanonicalOpenAIModelRef` (`src/commands/doctor/shared/codex-route-warnings.ts:56`), applies that rewrite in `maybeRepairCodexRoutes` (`src/commands/doctor/shared/codex-route-warnings.ts:651`), and chooses `agentRuntime.id: "pi"` unless the Codex plugin is installed, enabled, and has usable OAuth (`src/commands/doctor/shared/codex-route-warnings.ts:593`). The restored tests assert the bad behavior: `openai-codex/gpt-5.5` becomes `openai/gpt-5.5` plus `agentRuntime.id: "pi"` (`src/commands/doctor/repair-sequencing.test.ts:435`).
- Impact: OAuth-only ChatGPT/Codex users are moved from the working `openai-codex/*` provider to direct `openai/*` on PI, which then requires an OpenAI API key. Mixed setups with an OpenAI API key can silently switch expected subscription traffic onto API billing. This is the exact failure mode reported in https://github.com/openclaw/openclaw/issues/78407.
- Suggested fix: Preserve `openai-codex/*` refs when the selected/repair runtime would be PI, and only migrate to `openai/*` when also forcing a proven Codex runtime route. Update `codex-route-warnings` tests so doctor repair rejects `openai/* + pi` for Codex OAuth-only configs.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. `runDoctorRepairSequence` with `agents.defaults.model: "openai-codex/gpt-5.5"` currently asserts the repaired config is `openai/gpt-5.5` and `agentRuntime.id: "pi"` (`src/commands/doctor/repair-sequencing.test.ts:412`).
- Is this the best way to solve the issue? yes. The safe invariant is that doctor must not change a subscription/OAuth provider route into a direct API-key route unless it can prove the runtime/auth route remains subscription-backed.
