## Findings

### Low: Filtered native plugin specs can be paired with the wrong text command in `commands.list`

- Kind: bug
- File: `src/gateway/server-methods/commands.ts`
- Line: 184
- Evidence: `buildPluginCommandEntries()` builds `pluginTextSpecs = listPluginCommands()` and `pluginNativeSpecs = getPluginCommandSpecs(provider, ...)`, then pairs them by array index. This commit changed `listProviderPluginCommandSpecs()` to filter native specs by channel in `src/plugins/command-specs.ts:64`, while `listPluginCommands()` still returns every registered plugin command. If a QQBot-scoped command is registered before an unscoped command, `commands.list` for `provider: "discord"` can assign the unscoped command’s native name to the QQBot text command, then shift or drop the real command entry.
- Impact: Native command catalogs returned over the Gateway protocol can show a command with the wrong name, text alias, description, and plugin metadata. Clients that use `commands.list` for command menus or palettes can display misleading commands after channel-scoped plugin commands exist alongside other plugin commands.
- Suggested fix: Do not correlate plugin text and native specs by index. Build both text and native metadata from the same registered command object, or have native specs include a stable canonical command id/plugin id and join by that key after filtering.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Register a channel-scoped plugin command before an unscoped plugin command with a provider-native alias, then call `buildCommandsListResult({ provider: "discord", ... })`; the index join at `src/gateway/server-methods/commands.ts:183` pairs the filtered native spec with the wrong text spec.
- Is this the best way to solve the issue? yes. The safer direction is a keyed or single-source mapping; preserving index coupling is fragile now that one side is provider/channel-filtered.
