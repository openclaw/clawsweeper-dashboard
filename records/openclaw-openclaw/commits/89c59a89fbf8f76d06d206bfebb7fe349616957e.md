# Commit 89c59a8

## Summary

Found one actionable privacy regression in the new error-body metadata path. The intended structured metadata is useful, but its redaction honors `logging.redactSensitive: "off"` even though these error bodies can be emitted through transport error payloads/UI-facing surfaces.

## Findings

### Medium: Provider error bodies can bypass safety redaction when log redaction is off

- Kind: privacy
- File: `src/agents/transport-stream-shared.ts`
- Line: 178
- Evidence: The new transport metadata path serializes provider/OpenAI error objects into `errorBody` and redacts with `redactSensitiveText(text)` at `src/agents/transport-stream-shared.ts:178`. The provider HTTP helper does the same for raw response bodies at `src/agents/provider-http-errors.ts:22`. `redactSensitiveText` reads `logging.redactSensitive` and returns the original text when that config is `"off"` (`src/logging/redact.ts:200`), while the documented contract says `"off"` only disables general log/transcript redaction and safety-boundary payloads such as provider error observations should still redact (`docs/logging.md:268`, `docs/gateway/configuration-reference.md:1017`). A focused probe with `OPENCLAW_CONFIG_PATH` pointing at `{ "logging": { "redactSensitive": "off" } }` showed `extractProviderErrorInfo(...).body` and `extractTransportErrorDetails(...).errorBody` both contained `sk-secret1234567890abcd`; with `"tools"` they did not.
- Impact: A user who disables general log redaction can now receive raw provider error body metadata containing API keys, request auth material, or other echoed secrets in transport error payloads. This is broader than the previous formatted message path because `errorBody` can preserve full raw JSON fields that are not part of the user-facing detail string.
- Suggested fix: Treat provider/transport `errorBody` and structured provider error detail as safety-boundary payloads. Use `redactToolPayloadText` or `redactSensitiveText(..., { mode: "tools" })` with the existing custom-pattern merge semantics instead of config-default `redactSensitiveText`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Set `OPENCLAW_CONFIG_PATH` to a config with `logging.redactSensitive: "off"`, then call `extractProviderErrorInfo` on a JSON error response containing an extra secret field, or call `extractTransportErrorDetails` with an OpenAI-shaped `{ error: { ... } }`; the returned `body` / `errorBody` contains the secret.
- Is this the best way to solve the issue? yes. The repo already has a dedicated always-tools-mode redaction helper for UI/tool/safety surfaces, and the docs explicitly classify provider error observations as still redacted even when general log redaction is off.
