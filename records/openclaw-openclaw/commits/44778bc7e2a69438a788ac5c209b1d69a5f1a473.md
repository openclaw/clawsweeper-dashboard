# Commit 44778bc

## Summary

Found one low-severity regression in the web-provider runtime fallback path. The commit makes an explicit empty provider scope fall through to a fresh plugin load, breaking the existing resolver contract and its focused unit test.

## Findings

### Low: Empty web-provider scopes now trigger a loader call

- Kind: regression
- File: `src/plugins/web-provider-runtime-shared.ts`
- Line: 195
- Evidence: The new active-registry branch only returns when `activeProviders.length > 0`; otherwise it falls through to `loadOpenClawPlugins(loadOptions)`. For an explicit empty candidate scope, `context.onlyPluginIds` is `[]`, so the correct result is already known to be empty. The existing focused test `src/plugins/web-provider-runtime-shared.test.ts` reproduces this: `pnpm test src/plugins/web-provider-runtime-shared.test.ts src/plugins/web-search-providers.runtime.test.ts src/plugins/web-fetch-providers.runtime.test.ts src/plugins/web-provider-resolution-candidates.test.ts test/scripts/plugin-prerelease-test-plan.test.ts` fails with `loadOpenClawPlugins` called once as `{ activate: false, cache: false, onlyPluginIds: [] }`.
- Impact: This breaks the established empty-scope contract and adds unnecessary plugin-loader work. Current production callers I found do not pass `activate: true`, so the direct blast radius is low; however, the empty-scope loader branch can activate an empty registry and clear plugin runtime state when `activate: true`.
- Suggested fix: Return `[]` before consulting/loading registries when `context.onlyPluginIds` is explicitly empty, or otherwise keep the active-registry fallthrough from running for `[]`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: run the focused test command above after `pnpm install`; the relevant failure is `src/plugins/web-provider-runtime-shared.test.ts > preserves explicit empty candidate scopes when reusing the active registry`.
- Is this the best way to solve the issue? yes: an explicit empty scope is already a terminal “no candidates” signal, so short-circuiting avoids both the regression and extra loader side effects.
