# Commit 432b949

## Summary

The scrubber correctly redacts historical `argv`/`execArgv` records and current `main` includes a follow-up for one append race, but the rewrite is still not serialized with audit appends. A config audit record can be lost if it is appended after the final size check and before the `rename`.

## Findings

### Medium: Config audit scrub can still drop concurrent appends

- Kind: concurrency
- File: `src/config/io.audit.ts`
- Line: 597
- Evidence: `scrubConfigAuditLog` checks `stat(auditPath).size`, then calls `rename(tmpPath, auditPath)`. `appendConfigAuditRecord` independently uses `appendFile(auditPath, ...)` at `src/config/io.audit.ts:615` with no lock or coordination. Current `main` added a second stat in `14a9bb98b7`, but an append that opens/writes after that final stat and before rename still lands on the old inode and is replaced.
- Impact: `openclaw doctor --fix` can silently drop a concurrent config audit entry while the gateway or another CLI process writes to `config-audit.jsonl`. This does not corrupt user config, but it weakens the security/forensic audit log during a security-sensitive repair path.
- Suggested fix: serialize scrub and append through a lock that both paths honor, or require/prove no active writer before rewriting. Size checks alone cannot make `stat` plus `rename` atomic with respect to `appendFile`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. I ran a temp-home `tsx` probe against `scrubConfigAuditLog` with a mocked `fs.stat` that appends immediately after the second size check returns and before `rename`; the scrub returned `aborted:false`, and the final file did not contain the appended record.
- Is this the best way to solve the issue? yes for the direction: the writer and scrubber need a shared exclusion mechanism or the scrub needs to refuse to run while writers are active. A third size check cannot close the final check-to-rename race.
