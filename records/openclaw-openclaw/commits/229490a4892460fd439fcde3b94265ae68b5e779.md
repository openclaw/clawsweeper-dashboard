# Commit 229490a

## Summary

One low-severity compatibility regression remains: the new filename guard rejects legal bare Windows filenames that contain `..`, even when they do not traverse out of the state directory.

## Findings

### Low: Legal dotted task script filenames are rejected

- Kind: compatibility
- File: `src/daemon/schtasks.ts`
- Line: 53
- Evidence: `resolveTaskScriptPath()` now throws when `/[/\\]|\.\./.test(scriptName)` matches. The new test suite also locks in `OPENCLAW_TASK_SCRIPT_NAME=gateway..cmd` as invalid at `src/daemon/schtasks.test.ts:178`, but a focused `path.win32.join` probe shows `gateway..cmd` resolves to `C:\Users\test\.openclaw\gateway..cmd` and stays under the state directory.
- Impact: Existing custom Windows configurations using a legal bare filename with a double-dot substring now fail during install, restart handoff, uninstall cleanup, and runtime command reads because they all call `resolveTaskScriptPath()`. This is broader than the stated “file name only, not a path” contract.
- Suggested fix: Reject path separators and exact `.` / `..` path segments, or use a basename-style validation that preserves legal bare filenames such as `gateway..cmd`. Add a regression test that accepts a dotted bare filename.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: set `OPENCLAW_TASK_SCRIPT_NAME=gateway..cmd` and call `resolveTaskScriptPath()`; current main throws before path construction even though Windows path resolution keeps that name inside the state directory.
- Is this the best way to solve the issue? yes: narrowing validation to path traversal/path separators preserves the security hardening without breaking valid bare filenames.
