# Commit dee0aa4

## Summary

Found one release-test regression. The commit correctly allows a config-only plugin row, but it also allows no plugin index evidence at all for the configured external Brave plugin.

## Findings

### Medium: Configured plugin install survivor check now passes when Brave was never installed or indexed

- Kind: test_gap
- File: `scripts/e2e/lib/upgrade-survivor/assertions.mjs`
- Line: 453
- Evidence: `assertOptionalConfiguredPluginIndex()` only validates `record` when present and `plugin` when present. If both are absent, it returns successfully. A temp fixture with `OPENCLAW_UPGRADE_SURVIVOR_SCENARIO=configured-plugin-installs`, accepted coverage, and `plugins/installs.json` containing `{ "installRecords": {}, "plugins": [] }` exited `0` for `node scripts/e2e/lib/upgrade-survivor/assertions.mjs assert-state`.
- Impact: The published upgrade survivor scenario is documented as covering automatic external plugin install repair. With this change, the lane can pass even if `openclaw doctor --fix` fails to install or index the configured external Brave plugin, so release validation can miss the user-visible regression it was built to catch.
- Suggested fix: Keep accepting either an install record or a config/plugin row, but assert that at least one exists for required configured external plugins. For Brave, require `record || plugin`, validate the record when present, and validate `plugin.enabled !== false` when a row is present.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Build a minimal temp survivor state with configured-plugin coverage and an empty installed plugin index, then run `node scripts/e2e/lib/upgrade-survivor/assertions.mjs assert-state`; current `main` succeeds.
- Is this the best way to solve the issue? yes. Requiring “record or plugin row” preserves the commit’s config-only-row tolerance while restoring the missing-install failure signal.
