# Commit 48bf037

## Summary

Found one actionable regression: builtin extra memory path expansion now bypasses the existing `OPENCLAW_HOME` effective-home contract, while the QMD path resolver still honors it.

## Findings

### Medium: `~` extra paths bypass `OPENCLAW_HOME` for builtin memory

- Kind: compatibility
- File: `packages/memory-host-sdk/src/host/internal.ts`
- Line: 80
- Evidence: `expandHomePath()` expands `~` and `~/...` with `os.homedir()` directly. Existing path resolution in `packages/memory-host-sdk/src/host/config-utils.ts:188` and `packages/memory-host-sdk/src/host/config-utils.ts:199` resolves `~` through `OPENCLAW_HOME` first, and QMD extra memory paths use that resolver via `packages/memory-host-sdk/src/host/backend-config.ts:173` and `packages/memory-host-sdk/src/host/backend-config.ts:381`.
- Impact: With `OPENCLAW_HOME` set for service-account isolation, the same `memorySearch.extraPaths: ["~/notes"]` resolves to the OS home for builtin memory but to `OPENCLAW_HOME` for QMD. That can index or allow reads from the wrong directory and breaks the documented effective-home behavior.
- Suggested fix: Use the existing `resolveUserPath()` path contract for tilde expansion in `normalizeExtraMemoryPaths()` while preserving workspace-relative handling for non-absolute, non-tilde paths. Add regression coverage with `OPENCLAW_HOME`, `HOME`, and `~/...`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. I ran a focused import smoke check with `OPENCLAW_HOME=/tmp/openclaw-effective-home HOME=/tmp/os-home`; builtin normalization returned `/tmp/os-home/notes`, while QMD resolution returned `/tmp/openclaw-effective-home/notes`.
- Is this the best way to solve the issue? yes. Reusing `resolveUserPath()` keeps one home-expansion contract instead of creating memory-backend-specific semantics.
