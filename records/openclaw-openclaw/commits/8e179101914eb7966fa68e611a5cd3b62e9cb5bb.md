# Commit 8e17910

## Summary

Found one regression in the status/listing surfaces for the new config-only AWS SDK auth profiles. Runtime auth resolution accepts these profiles, but `models status --check` still reports the provider as missing because the overview/probe/list availability code only counts persisted store credentials or explicit AWS env vars.

## Findings

### Medium: Config-only AWS SDK auth profiles are reported as missing by model status/list surfaces

- Kind: regression
- File: `src/commands/models/list.auth-overview.ts`
- Line: 74
- Evidence: The new auth contract makes `auth.profiles.<id>.mode: "aws-sdk"` a valid profile with no `auth-profiles.json` entry. Runtime honors that path: `resolveApiKeyForProvider` checks `resolveAuthProfileOrder` and returns `resolveConfiguredAwsSdkProfileAuth` for config-only candidates at `src/agents/model-auth.ts:603`. `resolveAuthProfileEligibility` also treats a missing stored credential as eligible when `isConfiguredAwsSdkAuthProfileForProvider` matches at `src/agents/auth-profiles/order.ts:75`. However, `resolveProviderAuthOverview` still starts from `listProfilesForProvider(store, provider)` at `src/commands/models/list.auth-overview.ts:74`, so a config-only AWS SDK profile produces `profiles.count = 0`, no env, no models.json auth, and `effective: missing`. `models status` then filters that overview row out at `src/commands/models/list.status-command.ts:338` and adds the active provider to `missingProvidersInUse` at `src/commands/models/list.status-command.ts:347`. The same store-only assumption exists in `src/commands/models/list.auth-index.ts:86` and `src/commands/models/list.probe.ts:294`, so model availability/probe targeting can miss the same valid route.
- Impact: A migrated Bedrock setup that relies on the AWS SDK default credential chain, shared AWS config files, SSO/process credentials, web identity, ECS, or instance metadata can run through the new config-only profile path, but `openclaw models status --check` exits `1` and reports `amazon-bedrock` as missing. This is especially likely after `doctor --fix` moves legacy `type: "aws-sdk"` markers out of `auth-profiles.json`.
- Suggested fix: Teach the status/list/probe auth surfaces to include eligible config-only AWS SDK profile IDs from `resolveAuthProfileOrder` or `cfg.auth.profiles`, not only `store.profiles`. For status, count/render them as `aws-sdk` and include them in `providerAuth` so `missingProvidersInUse` does not fire. For list/probe, use the same eligibility helper used by runtime resolution.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. With a temp config whose default model is `amazon-bedrock/amazon.nova-micro-v1:0`, `models.providers.amazon-bedrock.auth = "aws-sdk"`, and `auth.profiles.amazon-bedrock:default.mode = "aws-sdk"` with an empty auth store, `openclaw models status --json --check` exits `1`, returns `missingProvidersInUse=["amazon-bedrock"]`, and returns `providers=[]`.
- Is this the best way to solve the issue? yes. The runtime selection path is already centralized in `resolveAuthProfileOrder` and `isConfiguredAwsSdkAuthProfileForProvider`; the status/list/probe surfaces should reuse that contract instead of separately inferring auth only from stored credentials/env.
