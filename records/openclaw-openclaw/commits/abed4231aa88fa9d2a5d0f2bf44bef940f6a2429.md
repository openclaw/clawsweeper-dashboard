# Commit abed4231aa

## Summary

Found one actionable regression. The embedded runner now gives the model reply context by appending it to the user prompt string passed into `activeSession.prompt()`. That makes the quoted/replied message durable as part of the stored `role: "user"` turn, even though the new `currentTurnContext` contract says it is current-turn-only and not persisted as user text.

## Findings

### Medium: Reply context is persisted into the durable user turn

- Kind: regression
- File: `src/agents/pi-embedded-runner/run/attempt.ts`
- Line: 3068
- Evidence: `currentTurnContext` is documented as model-visible but not persisted as user text in `src/agents/pi-embedded-runner/run/params.ts:107`. The runner builds `promptForModel = promptSubmission.prompt + currentTurnPromptContextSuffix` at `src/agents/pi-embedded-runner/run/attempt.ts:2798`, then submits that same string to `activeSession.prompt()` at `src/agents/pi-embedded-runner/run/attempt.ts:3068` and `src/agents/pi-embedded-runner/run/attempt.ts:3071`. The installed `@mariozechner/pi-coding-agent@0.71.1` implementation builds a `role: "user"` message from the exact prompt text and persists `message_end` user messages with `sessionManager.appendMessage()`. OpenClaw also strips `openclaw.runtime-context` custom messages before the LLM boundary at `src/agents/pi-embedded-runner/run/attempt.ts:553`, but it cannot strip the reply suffix once it is embedded in a normal user message.
- Impact: A quoted/replied message that should only inform the current turn becomes part of durable user history and future model replay. This can make later turns treat stale quote context as user-authored history, duplicate context already handled through runtime context, and violate the transcript hygiene rule that stored visible user turns use the transcript body rather than the runtime-enriched prompt.
- Suggested fix: Keep the persisted user prompt as `promptSubmission.prompt` / `transcriptPrompt`, and carry `currentTurnContext` through a non-user, current-turn-only channel. If pi-coding-agent lacks that seam, either add a submit-with-transcript override or rewrite the just-persisted user entry back to the transcript prompt after submission. Add an integration assertion against the session file or session manager branch, not only the mocked prompt argument.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Send an embedded-run channel reply where `ReplyToBody` is set; `runPreparedReply()` creates `currentTurnContext`, the embedded path appends the `Replied message (untrusted, for context):` suffix to `promptForModel`, and `activeSession.prompt(promptForModel)` persists that whole string as a user message.
- Is this the best way to solve the issue? yes. The fix should preserve model visibility while separating provider input from durable transcript text, matching the existing `transcriptPrompt` and runtime-context split.
