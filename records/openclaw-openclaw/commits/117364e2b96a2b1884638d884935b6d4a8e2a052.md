# Commit 117364e

## Summary

Found one low-severity incomplete coverage/behavior issue in the adjacent exec script-preflight parser: the commit updates the shared env carrier parser for `env -P`, but the separate exec script-preflight env stripper still does not know that `-P` consumes the following path argument.

## Findings

### Low: `env -P` still bypasses exec script preflight unwrapping

- Kind: reliability
- File: `src/agents/bash-tools.exec.ts`
- Line: 99
- Evidence: `PREFLIGHT_ENV_OPTIONS_WITH_VALUES` includes `-C`, `-S`, and `-u`, but not `-P`; `stripPreflightEnvPrefix` only skips the next token when this local set contains the option. The same commit adds `-P` to the shared `ENV_OPTIONS_WITH_VALUE` in `src/infra/command-carriers.ts:12`, and existing preflight tests already assert plain `env python bad.py` is validated in `src/agents/bash-tools.exec.script-preflight.test.ts:227`. FreeBSD `env(1)` documents `-P altpath`, so the parser must consume the following path token before the utility. ([man.freebsd.org](https://man.freebsd.org/env%281%29?utm_source=openai))
- Impact: commands like `env -P /usr/bin python bad.py` are now handled correctly by approval-command and strict inline-eval checks, but script preflight can still treat `/usr/bin` as the command and skip validation of `bad.py`. That leaves the “shell variable leaked into Python/JS source” guard inconsistent for the same BSD/macOS env carrier form this commit is fixing.
- Suggested fix: reuse `parseEnvInvocationPrelude`/`unwrapEnvInvocation` for preflight stripping, or at minimum add `"-P"` to `PREFLIGHT_ENV_OPTIONS_WITH_VALUES`; add a script-preflight test for `env -P /usr/bin python bad.py`.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes: extend the existing env-prefixed preflight test with `command: "env -P /usr/bin python bad.py"` and a `bad.py` containing `payload = $DM_JSON`; the source path at `src/agents/bash-tools.exec.ts:99` and `src/agents/bash-tools.exec.ts:176` will not consume `/usr/bin` as the `-P` value.
- Is this the best way to solve the issue? yes: the safer fix is to remove the duplicated env option table and call the shared carrier parser, so approval, inline-eval, display, and preflight stay aligned.
