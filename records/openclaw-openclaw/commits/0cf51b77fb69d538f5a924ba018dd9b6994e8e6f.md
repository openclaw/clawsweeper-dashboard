# Commit 0cf51b7

## Summary

Found one compatibility regression in the collapsed memory registration state. A memory plugin that mixes deprecated split registration calls with the new unified capability call can silently lose its prompt section, flush plan, and runtime.

## Findings

### Medium: Mixed old/new memory registration can erase previously registered memory slots

- Kind: compatibility
- File: `src/plugins/memory-state.ts`
- Line: 168
- Evidence: `registerMemoryCapability()` now replaces the whole active capability with only the fields passed in. Deprecated plugin API calls are adapted through `registerMemoryPromptSectionForPlugin()`, `registerMemoryFlushPlanResolverForPlugin()`, and `registerMemoryRuntimeForPlugin()`, which patch the active capability for the same plugin. But if a plugin first calls deprecated split APIs and later calls `registerMemoryCapability({ publicArtifacts })`, the later unified call discards the earlier `promptBuilder`, `flushPlanResolver`, and `runtime`. I reproduced this with the current source: after registering prompt/flush/runtime for `memory-ext` and then `registerMemoryCapability("memory-ext", { publicArtifacts })`, `buildMemoryPromptSection()` returned `[]`, `resolveMemoryFlushPlan()` returned `null`, and `getMemoryRuntime()` returned `undefined`.
- Impact: Third-party memory plugins still using deprecated APIs can break during incremental migration, especially when adding `publicArtifacts` through the new API while leaving prompt/flush/runtime on old calls. That disables memory prompt guidance, pre-compaction memory flushes, and memory search/status runtime for that plugin without a load error.
- Suggested fix: Preserve same-plugin existing fields when registering a partial unified capability, or route `api.registerMemoryCapability()` through a merge/patch helper for the active plugin. If full replacement is required, add an explicit replacement path rather than making partial migration erase existing slots.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused `node --import tsx -e` check against `src/plugins/memory-state.ts` reproduced the lost prompt, flush plan, and runtime after old-style registrations followed by a partial unified capability registration.
- Is this the best way to solve the issue? yes. The deprecated APIs remain public in `src/plugins/types.ts`, and `src/plugins/registry.ts` still routes them into the same memory capability state, so same-plugin merging preserves the intended compatibility path while keeping the new collapsed state model.
