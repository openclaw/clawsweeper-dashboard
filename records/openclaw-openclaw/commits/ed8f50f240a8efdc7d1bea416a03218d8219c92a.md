# Commit ed8f50f

## Summary

Found regressions in the new plugin dependency cleanup/repair paths. The highest-risk issue is that `doctor --fix` can delete legitimate dependency installs for copied archive/path plugins.

## Findings

### Medium: `doctor --fix` deletes valid dependencies from installed path/archive plugins

- Kind: data_loss
- File: `src/commands/doctor/shared/plugin-dependency-cleanup.ts`
- Line: 37
- Evidence: `isLegacyDependencyDebrisName()` classifies any child named `node_modules` as legacy debris, and `collectLegacyPluginDependencyTargets()` scans both `resolveStateDir(env)/extensions` and `resolveConfigDir(env)/extensions` before `cleanupLegacyPluginDependencyState()` recursively removes each target. The default plugin install root is also `resolveConfigDir(env)/extensions`. Archive installs explicitly set `installDependencies: true`, and `installPackageDir()` runs `npm install` in the staged plugin before publishing it, so a copied archive/path plugin with dependencies legitimately ends up with `~/.openclaw/extensions/<id>/node_modules`.
- Impact: Running `openclaw doctor --fix` after installing such a plugin removes its runtime dependencies and the plugin will later fail with missing-module errors until reinstalled.
- Suggested fix: Do not delete plain `node_modules` under the configured plugin install root unless it is tied to a legacy staging marker/manifest. Use install records to skip managed path/archive installs, or limit generic `node_modules` cleanup to old bundled/dist roots with explicit legacy evidence.
- Confidence: high

### Medium: Nix mode still runs npm plugin repair during doctor repair

- Kind: compatibility
- File: `src/commands/doctor/shared/missing-configured-plugin-install.ts`
- Line: 117
- Evidence: Current docs say `OPENCLAW_NIX_MODE=1` disables auto-install and self-mutation flows and should surface Nix-specific remediation. `runDoctorRepairSequence()` unconditionally calls `repairMissingConfiguredPluginInstalls()`, which calls `updateNpmInstalledPlugins()` for missing recorded installs and `installPluginFromNpmSpec()` for missing downloadable catalog candidates without checking `resolveIsNixMode(env)`.
- Impact: `openclaw doctor --fix` in Nix mode can perform network/npm installs and write plugin install records instead of remaining deterministic/read-only. That contradicts the Nix contract and can fail or dirty Nix-managed deployments.
- Suggested fix: Add an early Nix-mode guard in the missing-plugin repair path that reports the missing plugin with Nix-specific remediation and skips npm/git/install-record mutation.
- Confidence: high

### Low: Blocked npm plugin installs leave downloaded package contents in the managed npm root

- Kind: security
- File: `src/plugins/install.ts`
- Line: 1173
- Evidence: `installPluginFromNpmSpec()` runs `npm install --prefix <npmRoot> <spec>` before calling `installPluginFromInstalledPackageDir()`. That later validation path can reject the package for invalid metadata, id mismatch, or install security scan failure, but the error is returned directly and no cleanup removes `npmRoot/node_modules/<package>` or dependencies that npm already wrote.
- Impact: A blocked or mismatched npm plugin is not recorded or loaded, but its code remains in OpenClaw’s managed plugin npm root after a failed install. This weakens the “installation blocked” behavior and leaves stale untrusted package contents for manual cleanup.
- Suggested fix: Install npm candidates into a temporary prefix and only publish/adopt the package after validation succeeds, or clean the installed package/dependency tree on every post-install validation failure.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. For the cleanup issue, install a copied archive/path plugin with a runtime dependency so `~/.openclaw/extensions/<id>/node_modules` exists, then run `openclaw doctor --fix`; the cleanup target collector will remove that directory. For Nix mode, set `OPENCLAW_NIX_MODE=1` and configure a missing downloadable plugin/channel before `doctor --fix`; the repair path still reaches npm install/update calls. For npm failure cleanup, use an npm plugin package that fails manifest/id/security validation after download and inspect the npm root.
- Is this the best way to solve the issue? No. The dependency model change is sound, but cleanup/repair needs tighter ownership checks: cleanup should only remove proven legacy generated artifacts, Nix mode should skip auto-installs, and failed npm installs need transactional cleanup.
