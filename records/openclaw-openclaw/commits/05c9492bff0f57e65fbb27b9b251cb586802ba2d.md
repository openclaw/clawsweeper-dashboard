# Commit 05c9492bff

## Summary

Found two actionable regressions still present on current `main` (`c8ab22997b`): official non-Codex npm plugin installs can now be blocked by the scanner, and the public `chat.transcribeAudio` Gateway RPC was removed without a compatibility path.

## Findings

### Medium: Official npm plugin trust is now limited to Codex only

- Kind: regression
- File: `src/plugins/install.ts`
- Line: 113
- Evidence: The commit replaced the catalog-derived trusted official npm package map with `new Map([["@openclaw/codex", "codex"]])`. The official catalog still marks packages like `@openclaw/feishu` as `source: "official"` with `defaultChoice: "npm"` (`scripts/lib/official-external-channel-catalog.json:104`, `scripts/lib/official-external-channel-catalog.json:120`), and docs still list `@openclaw/feishu` as an npm-installable official plugin (`docs/tools/plugin.md:193`). A focused temp-package probe using package name `@openclaw/feishu`, manifest id `feishu`, and normal channel-style `process.env` plus `fetch(...)` code returned `{"ok":false,"code":"security_scan_blocked"}`. The same probe for `@openclaw/codex` was allowed as official.
- Impact: npm install/update/doctor/onboarding fallback paths for official external plugins other than Codex can fail on expected plugin code patterns, despite the catalog advertising npm as the default install source. Users would need force flags or a different install path for official packages that previously passed the trusted official scanner path.
- Suggested fix: Restore a trusted official npm package list from the official catalog, or generate a small static allowlist from catalog data so the install hot path stays cheap. Keep the package-name/plugin-id match checks and restore coverage for both an allowed official non-Codex package and a blocked spoof.
- Confidence: high

### Medium: `chat.transcribeAudio` was removed as a breaking Gateway protocol change

- Kind: compatibility
- File: `src/gateway/server-methods.ts`
- Line: 80
- Evidence: The commit deletes `src/gateway/server-methods/chat-transcribe-audio.ts`, removes its handler spread from `coreGatewayHandlers`, removes `chat.transcribeAudio` from `server-methods-list.ts` and `method-scopes.ts`, and removes the Control UI dictation button/caller. Before the commit, docs described WebChat using `chat.transcribeAudio`; current docs still contain stale UI copy saying Talk is “next to the microphone dictation button” (`docs/web/control-ui.md:168`), but that button no longer exists.
- Impact: Existing WebSocket clients or older Control UI bundles that call `chat.transcribeAudio` now get an unknown/unauthorized method instead of server-side draft dictation. This is an incompatible Gateway API removal without a versioned/deprecated compatibility shim.
- Suggested fix: Keep a deprecated `chat.transcribeAudio` handler until a protocol version/removal window exists, or explicitly version the removal and update all docs/client capability discovery so older clients can degrade cleanly.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. For the install regression, the temp-package probe directly exercises `installPluginFromInstalledPackageDir` with `installPolicyRequest.kind = "plugin-npm"`. For the Gateway regression, the method has been removed from the handler registry and advertised method list, so any `chat.transcribeAudio` request no longer has a server handler.
- Is this the best way to solve the issue? No. The install change should preserve the lazy/performance goal without dropping catalog-backed official trust. The Gateway change needs either backward compatibility or explicit protocol versioning.
