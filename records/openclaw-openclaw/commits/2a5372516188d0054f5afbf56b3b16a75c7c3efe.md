# Commit 2a53725161

## Summary

One low-severity test regression found. The production Bonjour advertiser path was not changed, but the new warning check accidentally became a no-op.

## Findings

### Low: warning absence assertion does not assert the boolean result

- Kind: test_gap
- File: `extensions/bonjour/src/advertiser.test.ts`
- Line: 751
- Evidence: The commit replaced the previous `expect(logger.warn).not.toHaveBeenCalledWith(...)` assertion with `expect(warnMessages().every(...))` but did not add a matcher such as `.toBe(true)`. Vitest accepts a bare `expect(false)` without throwing, verified with `node --input-type=module -e "const { expect } = await import('vitest'); expect(false); console.log('bare expect returned without throwing');"`.
- Impact: The test named `does not re-advertise while ciao is still probing` no longer fails if the unwanted `"watchdog detected non-announced service; attempting re-advertise"` warning is logged during the first 15 seconds. That weakens the regression coverage this commit is meant to add.
- Suggested fix: Assert the boolean, e.g. append `.toBe(true)`, or use an array assertion such as `expect(warnMessages()).not.toEqual(expect.arrayContaining([expect.stringContaining(...)]))`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes; the bare Vitest `expect(false)` check returns without throwing, so a false result from the new `.every(...)` expression would not fail the test.
- Is this the best way to solve the issue? yes; the intended behavior is already encoded in the predicate, it just needs a matcher or equivalent negative array assertion.
