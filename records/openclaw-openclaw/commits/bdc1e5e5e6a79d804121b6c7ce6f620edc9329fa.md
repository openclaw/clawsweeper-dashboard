# Commit bdc1e5e

## Summary

Found one regression in the new Slack message-tool schema contributions: the descriptions are additive for current-Slack sessions, but their default `current-channel` visibility also removes several Slack actions from cross-channel message-tool schemas.

## Findings

### Medium: Slack management/file actions disappear from cross-channel message tool schemas

- Kind: regression
- File: `extensions/slack/src/message-tool-api.ts`
- Line: 49
- Evidence: The new schema contributions set `actions: ["download-file"]` and `actions: ["react", "reactions", "edit", "delete", "pin", "unpin"]` without `visibility`. Core treats schema contributions as `current-channel` by default and removes those action names from cross-channel discovery in `src/channels/plugins/message-action-discovery.ts:255`.
- Impact: When Slack is configured but the current channel is another plugin, the generated `message` tool action enum no longer includes Slack `download-file`, `react`, `reactions`, `edit`, `delete`, `pin`, or `unpin`. This turns previously valid cross-channel Slack workflows into schema validation failures before dispatch. A focused smoke check with Slack plus a dummy current channel produced only `read,upload-file,member-info,emoji-list` from the Slack action subset; current Slack and no-current-channel contexts still included `download-file`.
- Suggested fix: Mark these Slack schema fragments so they do not block cross-channel actions. Since `messageId`, `message_id`, and `fileId` already exist in the shared message-tool schema, `actions: []` would preserve current-channel-only Slack descriptions without removing cross-channel actions. Alternatively use `visibility: "all-configured"` if Slack-specific hints should also be visible in cross-channel contexts. Add a regression test around `createMessageTool` or `listCrossChannelSchemaSupportedMessageActions` with Slack configured and a different current channel.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Register Slack plus a non-Slack current-channel plugin, create the message tool with `currentChannelProvider` set to the non-Slack plugin, and inspect `parameters.properties.action.enum`; the affected Slack actions are missing.
- Is this the best way to solve the issue? yes, directionally. The intended change is a schema hint/error clarification, not a cross-channel capability reduction, and the core schema-blocking contract already supports empty `actions` for fragments that should not block cross-channel actions.
