# Commit c40f894

## Summary

Found two actionable issues in the Mantis PR comment changes. The main regression is that the new human summary can claim success even when the workflow failed and is posting failure evidence.

## Findings

### Medium: Failure comments now include a success summary

- Kind: bug
- File: `.github/workflows/mantis-discord-status-reactions.yml`
- Line: 337
- Evidence: `run_mantis` writes `comparison.json` before failing on unexpected baseline/candidate status at `.github/workflows/mantis-discord-status-reactions.yml:226` and `.github/workflows/mantis-discord-status-reactions.yml:250`. The comment step is guarded with `always()` at `.github/workflows/mantis-discord-status-reactions.yml:270`, so it can still run after that failure. The newly added summary at `.github/workflows/mantis-discord-status-reactions.yml:337` unconditionally says the baseline reproduced the bug and the candidate showed the expected sequence, regardless of `.pass`, `.baseline.status`, or `.candidate.status`.
- Impact: A failed Mantis run with screenshots and `comparison.json` can leave a PR comment whose first substantive line says the fix passed. That is exactly the evidence surface maintainers are meant to scan before merging.
- Suggested fix: Generate the summary from `pass`, `baseline_status`, and `candidate_status`; for non-passing or harness-failure paths, say the run did not verify the fix and point readers to the status lines/artifact.
- Confidence: high

### Low: The docs now promise bug-report comments, but the workflow still only accepts PRs

- Kind: bug
- File: `.github/workflows/mantis-discord-status-reactions.yml`
- Line: 298
- Evidence: The new input description says the number can be a "bug or fix PR" at `.github/workflows/mantis-discord-status-reactions.yml:17`, and the docs say the workflow should comment on a "bug report or fix PR" at `docs/concepts/mantis.md:349`. The implementation still validates `repos/${GITHUB_REPOSITORY}/pulls/${TARGET_PR}` and later uses `gh pr comment "$TARGET_PR"` at `.github/workflows/mantis-discord-status-reactions.yml:298` and `.github/workflows/mantis-discord-status-reactions.yml:365`, which fail for a plain issue number.
- Impact: A maintainer following the updated docs can dispatch the workflow against a bug issue, get a successful Mantis evidence run, and then have the final comment step fail instead of attaching proof to the issue.
- Suggested fix: Either keep the contract PR-only in the input/docs, or rename/generalize the input and use issue comment APIs for issue targets while preserving PR support.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. For the summary bug, run the workflow with a candidate that produces evidence files but has `candidate_status != pass`; the `always()` comment step will build the new success summary after the run step fails. For the issue-target bug, dispatch with a GitHub issue number that is not a PR; the `pulls/<number>` validation rejects it.
- Is this the best way to solve the issue? yes. Both fixes should stay in the workflow/comment contract: make the summary status-aware, and make the target type either honestly PR-only or actually issue-compatible.
