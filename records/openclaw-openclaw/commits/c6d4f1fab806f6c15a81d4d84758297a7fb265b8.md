# Commit c6d4f1f

## Summary

Found one concrete regression in the Discord target resolver bridge. Bare numeric Discord channel IDs, which the plugin normalizer explicitly treats as channel targets, now throw as ambiguous when used through the generic message target resolver.

## Findings

### Medium: Bare numeric Discord channel IDs now fail target resolution

- Kind: regression
- File: `extensions/discord/src/channel.ts`
- Line: 330
- Evidence: `normalizeDiscordMessagingTarget` defaults bare numeric IDs to `channel:<id>` for stable tool routing (`extensions/discord/src/normalize.ts:3`). The new `messaging.targetResolver.resolveTarget` bridge calls `resolveDiscordTarget(input, ..., {})` when no `preferredKind` is provided (`extensions/discord/src/channel.ts:330`). On `input: "123456789"`, `resolveDiscordTarget` reaches `parseDiscordSendTarget(trimmed, {})` and throws the ambiguous-recipient error (`extensions/discord/src/target-resolver.ts:42`). This path is reachable from message sends because `resolveActionTarget` calls `resolveResolvedTargetOrThrow` for `args.to` without a preferred kind (`src/infra/outbound/message-action-runner.ts:380`), and id-like targets call the plugin resolver before falling back to the normalized channel target (`src/infra/outbound/target-resolver.ts:357`, `src/infra/outbound/target-id-resolution.ts:19`).
- Impact: Discord message tool sends that use a raw channel ID in `to`, a supported and documented hint shape (`<channelId|user:ID|channel:ID>`), can fail with `Ambiguous Discord recipient` instead of resolving to `channel:<id>`. Before this commit, Discord had no plugin `resolveTarget`, so the generic id-like branch fell through to the normalized channel target.
- Suggested fix: In the Discord bridge, pass `{ defaultKind: "channel" }` when `normalized` is a channel target and no explicit user preference exists, or catch the ambiguous parse and return `null` so the generic normalized fallback can preserve `channel:<id>`. Add a regression test through `resolveChannelTarget` or the message-action runner for `channel: "discord", input/to: "123456789"`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused runtime probe of `discordPlugin.messaging.targetResolver.resolveTarget({ input: "123456789", normalized: "channel:123456789" })` throws `Ambiguous Discord recipient "123456789"...`.
- Is this the best way to solve the issue? yes for the suggested direction: the fix should preserve the existing Discord normalizer contract and avoid letting plugin fallback exceptions bypass the generic normalized id-like fallback.
