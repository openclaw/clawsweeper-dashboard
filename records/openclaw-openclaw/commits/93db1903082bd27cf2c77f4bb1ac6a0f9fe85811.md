# Commit 93db190

## Summary

Found one medium reliability regression in the new aggregate channel-restart behavior. When one channel fails but a later channel restarts successfully, the managed reloader still treats the hot reload as failed and restores the old runtime snapshot, leaving the successfully restarted channel running with a config that the gateway rolled back.

## Findings

### Medium: Later channels can keep running the rejected config after an aggregate restart failure

- Kind: reliability
- File: `src/gateway/server-reload-handlers.ts`
- Line: 486
- Evidence: `applyHotReload` now catches each channel restart failure, continues to later channels, then throws an aggregate error after the loop at `src/gateway/server-reload-handlers.ts:495`. The managed reloader activates the next runtime config before `applyHotReload` at `src/gateway/server-reload-handlers.ts:661`, but restores the previous runtime snapshot when `applyHotReload` throws at `src/gateway/server-reload-handlers.ts:678`. Channel startup captures `getRuntimeConfig()` and passes that captured `cfg` into the channel task in `src/gateway/server-channels.ts:378` and `src/gateway/server-channels.ts:531`, so a later successfully restarted channel keeps running the next config even after the gateway rolls back the active snapshot.
- Impact: A failed multi-channel hot reload can leave the process split-brained: gateway/runtime config is old, while channels that restarted after the first failure continue running with the rejected new config until another restart/reload. Before this commit, the first failure aborted the loop, so later channels were not started under a config that would immediately be rolled back.
- Suggested fix: Make the aggregate failure path transactional. Either accept the new runtime snapshot even when some channels fail and surface per-channel failures without throwing through the managed rollback path, or track channels successfully restarted under the next config and, after restoring the previous snapshot, restart those channels again against the previous config.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused in-memory harness made `telegram` stop fail, let `discord` stop/start continue, then mimicked the managed rollback; it produced `startedWith:[["discord","new"]]` while `activeAfterRollback:"old"`.
- Is this the best way to solve the issue? unclear. The code needs one consistent contract: partial channel failures either still accept the config with degraded channel status, or a failed reload must roll successfully restarted channels back to the previous runtime config.
