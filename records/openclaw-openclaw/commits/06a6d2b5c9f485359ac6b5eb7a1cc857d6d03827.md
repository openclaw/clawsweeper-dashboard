## Findings

### Medium: Auth-profile watcher misses hot-reloaded agent auth stores

- Kind: bug
- File: `src/gateway/server-startup-post-attach.ts`
- Line: 1047
- Evidence: The watcher is created once with `cfg: params.cfgAtStart`, and `watchAuthProfilesForChanges` computes a fixed path list from that config at `src/agents/auth-profiles-watcher.ts:25`. But `agents.list` is explicitly hot-reloadable at `src/gateway/config-reload-plan.ts:100`, and reload handling rewarms provider auth with `nextConfig` at `src/gateway/server-reload-handlers.ts:503` without replacing or updating the watcher. Once warmed, `hasAuthForModelProvider` returns the prepared boolean directly at `src/agents/model-provider-auth.ts:120`, so a stale `false` can persist.
- Impact: After adding an agent or changing an agent `agentDir` via hot reload, externally added credentials for that agent’s `auth-profiles.json` will not clear/rewarm the prepared auth map. Model listing/status can keep reporting no auth for that agent until another config reload or gateway restart.
- Suggested fix: Make the auth-profile watcher part of reloadable runtime state. On hot reloads that affect `agents.list`, `agents.defaults`, or any path influencing `resolveAgentDir`, stop the old watcher and start a new one for `nextConfig`, or support updating the watched path set in place. Rewarm with the current runtime config, not `cfgAtStart`.
- Confidence: high

### Low: Detached watcher registration can race gateway close

- Kind: reliability
- File: `src/gateway/server-startup-post-attach.ts`
- Line: 1021
- Evidence: `onPostReadySidecars` publishes the sidecar array at `src/gateway/server-startup-post-attach.ts:969`, but the auth watcher is created later in a detached `sidecarsPromise.then` and pushed at `src/gateway/server-startup-post-attach.ts:1046`. Gateway close marks closing and drains the currently registered sidecars at `src/gateway/server.impl.ts:1673`; `stopRegisteredPostReadySidecars` then clears `runtimeState.postReadySidecars` at `src/gateway/server.impl.ts:977`. If close starts while the detached auth callback is still awaiting imports, the callback can create a new chokidar watcher after the drain and push it into the old array with no second close-start check.
- Impact: A fast shutdown/restart during post-ready auth prewarm can leave the chokidar watcher open, which can keep the process alive or leak handles in tests and managed restarts.
- Suggested fix: Register the auth watcher before `onPostReadySidecars` fires, or expose a `registerPostReadySidecar` helper from `server.impl` that stops immediately when `closePreludeStarted` is already true. Add a lifecycle test with delayed auth imports and immediate close.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the hot-reload bug: start with one configured agent, hot reload `agents.list` to add another agent, let auth rewarm with no credentials, then write the new agent’s `auth-profiles.json`; the watcher remains bound to startup paths and does not invalidate the prepared `false`. The shutdown race is a concrete async ordering issue but was not live-reproduced.
- Is this the best way to solve the issue? Yes: the watcher should be owned by the same runtime/reload lifecycle as the config-derived agent directory set, rather than permanently bound to startup config.
