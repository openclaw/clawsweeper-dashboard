# Commit d4af125

## Summary

Found one low-severity CLI default bug in the new Mantis before/after runner: local runs now force the CI Convex credential role instead of reusing the existing live-transport default of `maintainer` outside CI.

## Findings

### Low: Local Mantis runs default to CI credentials outside CI

- Kind: bug
- File: `extensions/qa-lab/src/mantis/cli.ts`
- Line: 65
- Evidence: `qa mantis run` sets `--credential-role` default to `"ci"`, and `runMantisBeforeAfter` also fills missing roles with `DEFAULT_CREDENTIAL_ROLE = "ci"` before passing `--credential-role ci` into each underlying `qa discord` run (`extensions/qa-lab/src/mantis/run.runtime.ts:82`, `extensions/qa-lab/src/mantis/run.runtime.ts:342`, `extensions/qa-lab/src/mantis/run.runtime.ts:286`). That bypasses the shared live credential default, which intentionally uses `maintainer` when `CI` is not truthy (`extensions/qa-lab/src/live-transports/shared/credential-lease.runtime.ts:120`).
- Impact: The documented local command in `docs/concepts/mantis.md` omits `--credential-role`. On a maintainer machine with the normal maintainer Convex secret but no CI secret, the new command will request CI credentials and fail with `Missing OPENCLAW_QA_CONVEX_SECRET_CI`, even though the existing `qa discord --credential-source convex` path would work.
- Suggested fix: Do not default `credentialRole` in the Mantis CLI/runtime. Pass it only when the user provides it, and let the shared credential lease code choose `ci` in CI and `maintainer` otherwise.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. `pnpm openclaw qa mantis run --help` shows `--credential-role` defaulting to `"ci"`, and the code path always forwards that value to the underlying Discord live lane.
- Is this the best way to solve the issue? yes. Reusing the existing shared credential-role default keeps Mantis aligned with Telegram/Discord live QA behavior and avoids duplicating environment-sensitive credential policy.
