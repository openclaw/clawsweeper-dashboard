# Commit c96795d

## Summary

Found one release-gate regression: the live/E2E reusable workflow tries to detect `workflow_call` through `github.event_name`, but reusable workflows inherit the caller’s `github` context. Manual release-check calls therefore still select Blacksmith runners for the live/E2E jobs instead of hosted runners.

## Findings

### Medium: live/E2E release jobs still select Blacksmith when called by manual release checks

- Kind: reliability
- File: `.github/workflows/openclaw-live-and-e2e-checks-reusable.yml`
- Line: 458
- Evidence: `openclaw-release-checks.yml:530` and `openclaw-release-checks.yml:595` call `openclaw-live-and-e2e-checks-reusable.yml` from the manual `OpenClaw Release Checks` workflow. The called workflow then uses expressions such as `github.event_name == 'workflow_call' && 'ubuntu-24.04' || 'blacksmith-8vcpu-ubuntu-2404'` at `openclaw-live-and-e2e-checks-reusable.yml:458`, `:611`, `:1242`, `:1486`, `:1559`, `:2207`, and `:2426`. GitHub documents that a reusable workflow’s `github` context is associated with the caller workflow, and its OIDC example for a job in a called workflow shows `event_name: "workflow_dispatch"` while `job_workflow_ref` identifies the called workflow. ([docs.github.com](https://docs.github.com/en/actions/reference/workflows-and-actions/reusing-workflow-configurations))
- Impact: When `OpenClaw Release Checks` is manually dispatched, these live/E2E jobs evaluate the condition as `workflow_dispatch`, not `workflow_call`, and keep using Blacksmith labels. That defeats the commit’s hosted-runner fallback for the release live/E2E gate, leaving release validation exposed to the same Blacksmith queue/unavailability path this change is trying to bypass.
- Suggested fix: Do not infer the runner profile from `github.event_name` inside the reusable workflow. Add an explicit `workflow_call` input such as `use_hosted_runners` or `runner_profile`, pass it from `openclaw-release-checks.yml`, and branch `runs-on` from that input. Alternatively add explicit runner-label inputs like the cross-OS reusable workflow already uses.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes; dispatch `OpenClaw Release Checks` with `rerun_group=live-e2e` or `run_release_soak=true`, then inspect the called live/E2E jobs’ selected runner labels. Static evidence is enough to predict the evaluation path because the caller is `workflow_dispatch` and the called workflow uses the caller-associated `github.event_name`.
- Is this the best way to solve the issue? yes; an explicit input keeps runner selection under the release workflow’s control and avoids relying on reusable-workflow context subtleties.
