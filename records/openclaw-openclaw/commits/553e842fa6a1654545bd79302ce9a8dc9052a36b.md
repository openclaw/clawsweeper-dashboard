# Commit 553e842

## Summary

Found one regression: the full-schema payload cap is also applied to `config.schema.lookup`, so the documented path-scoped lookup API loses plugin field schemas exactly when clients need a smaller drill-down response.

## Findings

### Medium: Oversized full-schema cap breaks path-scoped plugin schema lookup

- Kind: regression
- File: `src/config/schema.ts`
- Line: 555
- Evidence: `buildConfigSchema()` now calls `limitExtensionSchemas()` before all schema consumers. `config.schema.lookup` loads that same capped schema via `loadSchemaWithPlugins()` and then calls `lookupConfigSchema()` in `src/gateway/server-methods/config.ts:288`. A focused check with one 300 KB plugin schema showed `config.schema` omitted the large marker, but `lookupConfigSchema(res, "plugins.entries.huge.config.huge")` returned `null`.
- Impact: `config.schema.lookup` is documented as the path-scoped drill-down API for Control UI, CLI, and custom clients. After this commit, any plugin schema omitted from the full `config.schema` response can no longer be queried field-by-field, so clients cannot inspect field docs, constraints, child summaries, or sensitive hints for those plugin config fields.
- Suggested fix: Keep the cap on the full `config.schema` response, but make `config.schema.lookup` resolve the requested plugin/channel schema from uncapped metadata and return only the shallow lookup payload. If that is too large for one path, apply a separate lookup response cap at the requested subtree, not at the whole merged schema.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. One oversized plugin schema with `properties.huge` over the per-schema budget causes the full response to omit the marker and `lookupConfigSchema(..., "plugins.entries.huge.config.huge")` to return `null`.
- Is this the best way to solve the issue? unclear. The right fix shape depends on whether maintainers want lookup to remain full-fidelity for omitted schemas, but the current implementation contradicts the documented lookup contract in `docs/gateway/protocol.md:380` and `docs/cli/config.md:56`.
