# Commit 8df3500

## Summary

Found one concrete reliability regression risk: the UI now omits `agentId` from chat session refreshes, but the gateway does not pass `configuredAgentsOnly` into the store-loading layer. That means configured-only UI refreshes can still discover and load disk-only/unconfigured agent stores before filtering them out.

## Findings

### Medium: Configured-only chat refreshes can still do broad disk-store discovery

- Kind: reliability
- File: `ui/src/ui/app-chat.ts`
- Line: 783
- Evidence: This commit makes chat refreshes call `loadSessions` without `agentId` and with `CHAT_SESSIONS_ACTIVE_MINUTES = 0`, so `activeMinutes` is omitted and no agent scope is sent (`ui/src/ui/app-chat.ts:88`, `ui/src/ui/app-chat.ts:783`). The same no-`agentId` refresh shape is used after session creation/switch (`ui/src/ui/app-render.helpers.ts:703`, `ui/src/ui/app-render.helpers.ts:734`) and active chat/session events (`ui/src/ui/app-gateway.ts:691`, `ui/src/ui/app-gateway.ts:840`). `loadSessions` does add `configuredAgentsOnly: true` by default (`ui/src/ui/controllers/sessions.ts:485`), but the gateway handler ignores that flag when loading stores: it calls `loadCombinedSessionStoreForGateway(cfg, { agentId: p.agentId })` and only filters configured agents after store loading (`src/gateway/server-methods/sessions.ts:897`). With no `agentId` and no `configuredAgentsOnly` passed to the loader, `loadCombinedSessionStoreForGateway` takes the broad `resolveAllAgentSessionStoreTargetsSync(cfg)` path, which discovers disk-only stores under agent roots (`src/config/sessions/combined-store-gateway.ts:95`, `src/config/sessions/targets.ts:170`).
- Impact: The PR intentionally removes current-agent scoping from frequent Control UI chat refreshes, but the advertised configured-agent performance guardrail is not preserved at the expensive store discovery/load stage. Users with old or disk-only agent stores can pay broad discovery and JSON load cost on chat refresh, session switch, new chat, post-run refresh, and mid-run session-message refreshes, which can reintroduce the responsiveness regression this bounded refresh path was meant to avoid.
- Suggested fix: Pass `configuredAgentsOnly` through to `loadCombinedSessionStoreForGateway` in `sessions.list`, e.g. `{ agentId: p.agentId, configuredAgentsOnly }`, so no-agent configured-only callers use `resolveSessionStoreTargets(cfg, { allAgents: true })` instead of broad disk discovery. Add a gateway/unit regression where `sessions.list` with `{ configuredAgentsOnly: true }` and no `agentId` does not load a disk-only store.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. Configure one or more normal agents plus a disk-only `agents/<id>/sessions/sessions.json` store, then call the same Control UI-shaped RPC payload: `sessions.list` with `configuredAgentsOnly: true`, no `agentId`, `limit: 100`, and no `activeMinutes`. Code inspection shows the disk-only store is loaded before rows are filtered out.
- Is this the best way to solve the issue? Yes. The desired UI behavior is broad configured-agent visibility, not broad disk-only recovery discovery; the loader already has a `configuredAgentsOnly` option, but the gateway handler does not forward it.
