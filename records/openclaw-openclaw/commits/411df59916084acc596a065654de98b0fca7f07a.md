# Commit 411df59

## Summary

Found one likely incomplete-fix bug in the official bare-id install path: the new alias resolver runs only after config loading succeeds, but the linked Brave upgrade failure can leave config invalid until that same plugin is installed.

## Findings

### Medium: Bare official plugin aliases do not participate in install-time invalid-config recovery

- Kind: bug
- File: `src/cli/plugins-install-command.ts`
- Line: 763
- Evidence: `runPluginInstallCommand` resolves request context and calls `loadConfigForInstall` at lines 529-538 before the new `resolveOfficialExternalInstallPlanBeforeNpm` branch at line 763. `resolveOfficialExternalInstallRecoveryMetadata` only matches official entries by npm spec/name, not by bare plugin id, so `resolvePluginInstallRequestContext({ rawSpec: "brave" })` returns no recovered plugin id while `@openclaw/brave-plugin` does. Config validation rejects `tools.web.search.provider: "brave"` when the Brave provider is not active. Packaged OpenClaw excludes `dist/extensions/brave/**`, so upgraded packaged installs can hit this before the alias branch ever runs.
- Impact: Users in the linked #76373 state can still be unable to run the advertised repair command `openclaw plugins install brave` if their config already selects Brave web search and the packaged build no longer has the bundled Brave source. The command can fail at config validation instead of installing `@openclaw/brave-plugin`.
- Suggested fix: Make install request recovery recognize exact official bare plugin ids before config loading, and allow the narrowly matching “requested installable provider/channel is unavailable” config issue through for that same plugin id. Add a packaged/no-bundled test for `rawSpec: "brave"` with `tools.web.search.provider = "brave"`.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? unclear. The code path is concrete, and `gh issue view 76373` shows the affected Brave upgrade state, but this checkout is a source tree where `pnpm openclaw plugins install brave` uses the bundled `dist/extensions/brave` source after build, so it does not reproduce the packaged no-bundled branch directly.
- Is this the best way to solve the issue? unclear. The safe direction is to keep the recovery narrow to the requested official plugin id and the specific unavailable-provider/channel config issue, not to broadly relax install-time invalid-config handling.
