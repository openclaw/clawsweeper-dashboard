# Commit 34b17c82da

## Summary

Found one concrete incomplete fix in the OAuth adoption path: two changed call sites still can read runtime snapshot auth, so runtime-only external OAuth can still be treated as the main persisted credential.

## Findings

### Medium: main-store adoption can still use runtime-only external OAuth from active snapshots

- Kind: bug
- File: `src/agents/auth-profiles/oauth-manager.ts`
- Line: 225
- Evidence: The commit replaced `ensureAuthProfileStore(undefined)` with `ensureAuthProfileStoreWithoutExternalProfiles(undefined)` for main-store adoption, but that helper first returns `resolveRuntimeAuthProfileStore(agentDir)` when an active runtime snapshot exists (`src/agents/auth-profiles/store.ts:493`). Secrets runtime snapshots can contain external overlays because the non-fast path loads auth stores through `loadAuthProfileStoreForSecretsRuntime` before `replaceRuntimeAuthProfileStoreSnapshots` (`src/secrets/runtime.ts:390`, `src/secrets/runtime.ts:451`). I reproduced this with an active main runtime snapshot containing `runtime-external-access`: `resolveOAuthAccess` logged “adopted newer OAuth credentials from main agent”, returned `runtime-external-access`, and made `0` refresh calls even though persisted main/sub stores were expired.
- Impact: The commit’s intended persisted-store boundary is bypassed whenever a runtime auth snapshot is active. A subagent with an expired local OAuth profile can adopt a runtime-only external main profile instead of refreshing the persisted main store, which can use the wrong transient account/token and skip the refresh path this commit is trying to protect.
- Suggested fix: In the main-store adoption and catch fallback paths, use `loadAuthProfileStoreWithoutExternalProfiles(undefined)` or add a truly persisted-only/runtime-snapshot-bypassing helper. Avoid `ensureAuthProfileStoreWithoutExternalProfiles` where the correctness requirement is “no runtime overlay.”
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Activate a runtime auth snapshot for the main store containing a fresh `openai-codex:default` OAuth credential, persist expired main and subagent credentials for the same profile, then call `createOAuthManager(...).resolveOAuthAccess(...)` for the subagent. The manager returns the snapshot credential and does not call `refreshCredential`.
- Is this the best way to solve the issue? yes. The commit already uses disk-backed `loadAuthProfileStoreWithoutExternalProfiles` for the lock/reload paths; applying the same persisted-only read to the pre-lock and error fallback main-store checks aligns those paths with the stated fix.
