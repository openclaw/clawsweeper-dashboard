## Summary

Found one current outbound voice regression after the ffmpeg fallback removal. A snapshot-only missing `debugLog` import from this commit was already fixed on current `main` by `3c8de6e`, so I did not report it as actionable.

## Findings

### Medium: Non-native voice formats now fall through to raw QQ voice upload

- Kind: regression
- File: `extensions/qqbot/src/engine/messaging/outbound-media-send.ts`
- Line: 438
- Evidence: `shouldTranscodeVoice()` still marks formats like `.ogg`, `.opus`, `.flac`, and `.aac` as needing transcode, but `audioFileToSilkBase64()` now only handles direct native formats, SILK detection, PCM, WAV, and MP3 before returning `null` for unsupported formats at `extensions/qqbot/src/engine/utils/audio.ts:260`. When that `null` comes back, `sendVoiceFromLocal()` reads the original file and sends it as raw base64 voice data instead of returning an error. A focused probe showed `{"shouldTranscode":true,"converted":null}` for `.ogg`.
- Impact: before this commit, hosts with ffmpeg could convert these non-native audio formats to SILK. Now the default `transcodeEnabled: true` path attempts a raw voice upload for formats the plugin itself says cannot be uploaded directly, likely causing failed QQ voice sends and preventing higher-level `sendMedia()` fallback-to-document behavior from running cleanly.
- Suggested fix: when `needsTranscode` is true and `audioFileToSilkBase64()` returns `null`, return an explicit voice-transcoding error instead of raw-uploading. Alternatively, narrow `shouldTranscodeVoice()`/audio policy to only the formats still convertible without native subprocesses.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the local conversion branch: create a `.ogg` file and call `shouldTranscodeVoice()` plus `audioFileToSilkBase64()`. Full QQ delivery was not run because no QQ credentials were present in env or `~/.profile`.
- Is this the best way to solve the issue? Yes: the sender already has an error/fallback model for formats that cannot be uploaded directly, so the safest fix is to stop raw-uploading when a required conversion failed.
