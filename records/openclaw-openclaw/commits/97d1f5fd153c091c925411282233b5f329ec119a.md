# Commit 97d1f5f

## Summary

Found two concrete bypass regressions. The larger one means the original `min-release-age=7` npmrc case can still fail OpenClaw-managed installs, because npm reports that setting as a derived `before` value and the new helper responds with `--before=...`, which npm rejects as mutually exclusive with `min-release-age`.

## Findings

### High: `min-release-age` npmrc is converted into a conflicting `--before` override

- Kind: regression
- File: `src/infra/npm-install-env.ts`
- Line: 67
- Evidence: `createNpmFreshnessBypassArgs()` probes `npm config get min-release-age`; with npm 11.12.1 and a temp npmrc containing `min-release-age=7`, that probe returns `null`, while `npm config get before` returns a derived cutoff. The helper therefore returns `["--before=<now>"]`, and `createNpmProjectInstallEnv()` sets `npm_config_before=<now>`. Running npm with that env fails before config resolving: `--min-release-age cannot be provided when using --before`. The same derived-before logic is duplicated in `scripts/install.sh:718`, `scripts/install-cli.sh:629`, and `scripts/install.ps1:514`. npm’s own config docs state `before` and `min-release-age` cannot be used together, and npm/cli issue #9199 documents that `min-release-age` may display as `before` while `min-release-age` appears null. ([docs.npmjs.com](https://docs.npmjs.com/cli/v11/using-npm/config?utm_source=openai))
- Impact: The exact reported hardening config, `min-release-age=7` in npmrc, still breaks `openclaw update`, plugin npm installs, and hosted installer npm installs. The issue was closed and docs now claim this policy is cleared, but the common file-backed case remains broken.
- Suggested fix: Do not treat a derived `before` as proof of explicit `before`. Prefer `--min-release-age=0` for real `min-release-age` configs, or validate the candidate override against npm config before using it. Add file-backed npmrc tests for `min-release-age=7` across the TS helper and all three installer scripts.
- Confidence: high

### Medium: Global update probes plain `npm` instead of the selected npm command

- Kind: compatibility
- File: `src/infra/update-global.ts`
- Line: 732
- Evidence: `globalInstallArgs()` installs with `resolved.command`, but appends `...createNpmFreshnessBypassArgs()` with no selected command/env. That helper hardcodes `execFileSync("npm", ["config", "get", ...])` in `src/infra/npm-install-env.ts:51`. A focused probe with a selected npm wrapper that sees an explicit `before` npmrc produced argv ending in `--min-release-age=0`; running the selected npm with that arg failed with `--before cannot be provided when using --min-release-age`, while `--before=<now>` succeeded.
- Impact: Prefix-local or Homebrew npm installs can still fail when the npm selected for the update sees config that the PATH npm probe does not. The original report included `/opt/homebrew/etc/npmrc`, so this is on the affected path.
- Suggested fix: Thread the resolved npm command and effective child env into the freshness probe, and use that same command/env for `globalInstallArgs()`, `globalInstallFallbackArgs()`, and `createGlobalInstallEnv()`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Temp npmrc probes with npm 11.12.1 reproduce both failures without touching repo files.
- Is this the best way to solve the issue? yes directionally: compute the bypass from the same npm runtime/config that will perform the install, and add regression tests for file-backed `min-release-age` plus selected-npm config drift.
