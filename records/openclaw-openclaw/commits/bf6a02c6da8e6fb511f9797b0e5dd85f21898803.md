# Commit bf6a02c

## Summary

Found one surviving regression in the Feishu manifest metadata added by this commit. It can hide Feishu plugin tools for the documented multi-account configuration shape.

## Findings

### Medium: Feishu tool metadata ignores account-scoped credentials

- Kind: regression
- File: `extensions/feishu/openclaw.plugin.json`
- Line: 27
- Evidence: The new `toolMetadata` entries for all Feishu tools only pass availability when `channels.feishu.appId` and `channels.feishu.appSecret` exist. The documented Feishu multi-account config stores credentials at `channels.feishu.accounts.<id>.appId/appSecret`, and Feishu runtime account resolution merges those account entries before registering tools. The manifest tool gate in `src/plugins/tools.ts:289` calls `hasManifestToolAvailability`; if every declared tool has metadata and the signal fails, the Feishu runtime is not loaded for tool discovery.
- Impact: A user using the documented account-only Feishu config can have valid enabled Feishu accounts, but `feishu_wiki`, `feishu_doc`, `feishu_drive`, `feishu_chat`, and related tools are omitted from plugin tool discovery/catalog because the manifest-level availability check never sees the account credentials.
- Suggested fix: Make Feishu `toolMetadata` account-aware. If manifest `configSignals` cannot express wildcard account maps, either add support for account-map credential signals or avoid applying `toolMetadata` to Feishu tools until the signal can match both top-level and `accounts.<id>` credentials.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused check against current `main` returned `account-only false` and `root true` for `hasManifestToolAvailability({ toolNames: ["feishu_wiki"] })` using the actual Feishu manifest. The account-only config used `channels.feishu.accounts.main.appId/appSecret`, matching the docs.
- Is this the best way to solve the issue? yes, directionally. The bug is in the static availability evidence, not in Feishu runtime account resolution; fixing the manifest signal expressiveness or removing the incomplete metadata keeps the lazy runtime behavior without rejecting valid configs.
