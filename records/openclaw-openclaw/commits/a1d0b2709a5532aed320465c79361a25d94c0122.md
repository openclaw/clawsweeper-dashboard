# Commit a1d0b270

## Summary

Found one approval-policy regression in the new security audit suppression edit guard. The audit suppression filtering and active visibility marker looked consistent with the merged tests, but the exec approval path still lets suppression edits run via `askFallback=full` when an approval times out or has no route.

## Findings

### Medium: Suppression edit approval can be bypassed by askFallback

- Kind: security
- File: `src/agents/bash-tools.exec-host-gateway.ts`
- Line: 528
- Evidence: `requiresSecurityAuditSuppressionApproval` is added to `requiresAsk`, but once the approval request is unavailable or times out, the decision path only applies `enforceStrictInlineEvalApprovalBoundary`. `createExecApprovalDecisionState` approves a null/no-decision result when `askFallback` is `"full"` in `src/agents/bash-tools.exec-host-shared.ts:183`, and there is no equivalent boundary for `requiresSecurityAuditSuppressionApproval` in either the inline no-route path (`src/agents/bash-tools.exec-host-gateway.ts:528`) or async timeout path (`src/agents/bash-tools.exec-host-gateway.ts:597`).
- Impact: A command such as `openclaw config set security.audit.suppressions '[]'` can still run without explicit approval in non-yolo configurations like `security="full"`, `ask="always"`, `askFallback="full"` if no approver route exists for cron/headless execution or the approval times out. That contradicts the new warning/docs that suppression changes require explicit approval unless exec is already `security="full"` and `ask="off"`.
- Suggested fix: Treat `requiresSecurityAuditSuppressionApproval` like strict inline eval: deny fallback/no-route/timeouts unless the original policy exemption `hostSecurity === "full" && hostAsk === "off"` applies. Add tests for cron no-route and timeout fallback with `askFallback="full"`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Unit-level path: mock `processGatewayAllowlist` with a suppression mutation command, `trigger: "cron"`, effective `hostSecurity: "full"`, `hostAsk: "always"`, `askFallback: "full"`, and registration returning `finalDecision: null`; the current inline branch approves because only strict inline-eval fallback is blocked.
- Is this the best way to solve the issue? yes. The command classifier already identifies suppression mutations; the missing piece is carrying that requirement through fallback decision enforcement.
