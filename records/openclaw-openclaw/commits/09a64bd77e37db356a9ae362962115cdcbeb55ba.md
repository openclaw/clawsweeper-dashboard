# Commit 09a64bd

## Summary

Found one low-severity cancellation regression in the Ollama timeout propagation path.

## Findings

### Low: Ollama `timeoutMs` masks caller abort signals

- Kind: regression
- File: `extensions/ollama/src/stream.ts`
- Line: 1004
- Evidence: The Ollama stream wrapper passes `options.signal` inside `init.signal`, while the commit now also passes `options.timeoutMs` as top-level `timeoutMs` to `fetchWithSsrFGuard`. That guard composes only its top-level `signal` with the timeout signal and then overwrites `init.signal`, so a caller-supplied abort signal is no longer observed whenever `timeoutMs` is present. A focused probe confirmed `init.signal` was replaced and did not abort after the original caller signal was aborted.
- Impact: Direct Ollama stream callers that pass both standard `timeoutMs` and an `AbortSignal` can no longer cancel promptly, so `/stop`-style cancellation or early aborts may keep the HTTP request alive until the timeout or upstream response completes. This path becomes more likely now that standard provider `timeoutMs` is propagated.
- Suggested fix: Pass `options?.signal` to `fetchWithSsrFGuard` as the top-level `signal` parameter, and omit or leave `init.signal` only when no guard-level signal is needed.
- Confidence: high
