# Commit 9dc3271

## Summary

Found one actionable workflow security issue that is still present on latest remote `main`.

## Findings

### Medium: `target_ref` is interpolated directly into a PowerShell run block

- Kind: security
- File: `.github/workflows/windows-testbox-probe.yml`
- Line: 50
- Evidence: The workflow writes `Write-Host "target_ref=${{ inputs.target_ref || github.ref }}"` inside a `pwsh` `run:` block. `pipx run zizmor==1.22.0 --config .github/zizmor.yml --persona=regular --min-severity=medium --min-confidence=medium .github/workflows/windows-testbox-probe.yml` reports high-confidence `template-injection` at this line. A ref name like `probe";whoami;#` passes `git check-ref-format --branch`, and the equivalent PowerShell line executes `whoami` after closing the string.
- Impact: A user who can dispatch the manual workflow and target a crafted valid branch/tag ref can execute arbitrary PowerShell on the Windows runner after checkout, bypassing the probe-only intent. Current permissions are read-only and no explicit secrets are passed, so I’m rating this medium rather than high, but it is still command injection on CI infrastructure.
- Suggested fix: Pass the expression through `env` and only reference the shell variable in PowerShell, e.g. set `TARGET_REF: ${{ inputs.target_ref || github.ref }}` on the step and use `Write-Host "target_ref=$env:TARGET_REF"`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The static audit flags the exact line, `git check-ref-format --branch 'probe";whoami;#'` accepts the crafted ref locally, and `pwsh` executes the injected `whoami` when the interpolated line is evaluated.
- Is this the best way to solve the issue? yes. Moving GitHub expressions out of `run:` blocks and into `env` is the narrow fix and preserves the workflow behavior.
