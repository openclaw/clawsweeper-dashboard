# Commit d593f5b

## Summary

Found one low-severity Android/provider status bug that still exists on current `main`.

## Findings

### Low: Android provider refresh can show stale auth readiness for up to 60s

- Kind: bug
- File: `apps/android/app/src/main/java/ai/openclaw/app/NodeRuntime.kt`
- Line: 1716
- Evidence: The Android refresh path calls `models.authStatus` with `"{}"` at `NodeRuntime.kt:1716`, while the gateway returns a cached result for 60 seconds unless `params.refresh` is true in `src/gateway/server-methods/models-auth-status.ts:400`. The current Android UI wires the visible Refresh button to `viewModel::refreshModelCatalog` in `apps/android/app/src/main/java/ai/openclaw/app/ui/ProvidersModelsScreen.kt:109` and `:323`, so user-initiated refreshes do not bypass that cache. The web UI explicitly documents and sends `{ refresh: true }` for this exact case in `ui/src/ui/controllers/model-auth-status.ts:19`.
- Impact: After adding, removing, or changing provider credentials on the Gateway, the Android Providers & Models screen can continue to display stale “Needs setup” / “Ready” state until the gateway cache expires, even after tapping Refresh.
- Suggested fix: Let Android distinguish initial/background catalog loads from user refreshes, and send `{"refresh":true}` to `models.authStatus` for explicit refresh actions. Keeping cached reads for passive overview refreshes is fine.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: load provider status once, change provider auth state on the Gateway, then tap Android Refresh within 60 seconds; the Android call omits `refresh:true`, so the gateway serves the cached status.
- Is this the best way to solve the issue? yes: the gateway already owns the cache contract and exposes the bypass flag; Android should use that contract rather than adding client-side delays or duplicate invalidation.
