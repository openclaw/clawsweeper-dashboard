# Commit f50c65f

## Summary

Found one concrete regression in the new raw assistant completion release guard. It blocks post-tool/native-tool raw assistant progress, but request-style handoffs can still release early.

## Findings

### Medium: Request handoffs can still release interim raw assistant text

- Kind: regression
- File: `extensions/codex/src/app-server/run-attempt.ts`
- Line: 1520
- Evidence: `mcpServer/elicitation/request`, `item/tool/requestUserInput`, and app-server approval requests set `armCompletionWatchOnResponse = true`, but unlike `item/tool/call` at line 1557 they never set `turnCrossedToolHandoff = true`. The new release predicate returns true for any non-commentary raw assistant message while `!turnCrossedToolHandoff` at line 3049.
- Impact: after a user-input, MCP elicitation, or approval response, a subsequent raw assistant `rawResponseItem/completed` can fire the assistant idle release, send `turn/interrupt`, and return that raw text before Codex sends the real `turn/completed`. That can deliver interim text and cut off the actual final answer.
- Suggested fix: mark all current-turn request handoffs that arm the post-request completion watch as crossing the handoff boundary, or replace the boolean with a clearer “raw assistant release still allowed” state that is disabled for every request/approval/elicitation response path. Add regression coverage for `item/tool/requestUserInput` plus one approval or MCP elicitation path.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A one-off stdin script using the real `runCodexAppServerAttempt` with a fake app-server client answered `item/tool/requestUserInput`, then sent raw assistant `rawResponseItem/completed`; current `main` sent one `turn/interrupt` before `turn/completed` and returned `assistantTexts: ["Interim text before final answer."]`.
- Is this the best way to solve the issue? yes. The bug is the missing handoff state on the same request paths that already rearm the post-request watchdog, so setting the same blocker there is the narrow fix.
