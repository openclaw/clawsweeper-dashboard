# Commit a31b75f

## Summary

Found one concrete regression risk in the macOS deep-link delivery path after removing the `bluebubbles` enum case.

## Findings

### Medium: Legacy BlueBubbles deep links now silently fall back to `last`

- Kind: regression
- File: `apps/macos/Sources/OpenClaw/GatewayConnection.swift`
- Line: 23
- Evidence: `GatewayAgentChannel(raw:)` maps any unrecognized channel string to `.last`, and this commit removes the previously recognized `.bluebubbles` case. `DeepLinkAgentPolicy.effectiveDelivery` passes the link channel through that initializer when an unattended key is valid (`apps/macos/Sources/OpenClaw/DeepLinks.swift:39`), and `sendAgent` serializes the enum raw value into the gateway request (`apps/macos/Sources/OpenClaw/GatewayConnection.swift:497`). On the gateway side, `channel="last"` deliberately reuses the previous delivery route; this is covered by `src/gateway/server.agent.gateway-server-agent-b.test.ts:197`. Unknown channels sent as-is are rejected by gateway validation, as covered by `src/gateway/server.agent.gateway-server-agent-b.test.ts:356`.
- Impact: Existing keyed macOS automations or shortcuts using `channel=bluebubbles&deliver=1` no longer fail or target the retired channel. They become `channel=last`, so a message can be delivered through whichever channel was last used for that session, possibly with the legacy `to` value. That is a wrong-channel delivery/privacy risk for users migrating from BlueBubbles.
- Suggested fix: Preserve a legacy mapping from raw `"bluebubbles"` to `.imessage`, or make retired/unknown explicit channels non-deliverable so they fail closed instead of becoming `.last`.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes: on macOS, use a valid unattended deep link such as `openclaw://agent?message=hi&deliver=1&channel=bluebubbles&to=%2B15555550123&key=<valid-key>` after the target session has a non-iMessage last route. The current Swift code will send `channel: "last"` to the gateway.
- Is this the best way to solve the issue? Unclear; mapping to `imessage` best matches the migration docs if BlueBubbles target shapes are still compatible, while fail-closed unknown-channel handling is safer if maintainers want retired channel ids to stop working entirely.
