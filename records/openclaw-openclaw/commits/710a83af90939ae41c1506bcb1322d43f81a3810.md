# Commit 710a83af90

## Summary

Found one actionable gap in the rich-text recovery path: the recovered full Slack rich-text body is used for `RawBody`/`BodyForAgent`, but Slack ingress decisions and command bodies still read the truncated top-level `message.text`.

## Findings

### Medium: rich-text recovery does not feed Slack mention and command handling

- Kind: bug
- File: `extensions/slack/src/monitor/message-handler/prepare.ts`
- Line: 607
- Evidence: The commit makes `chooseSlackPrimaryText` prefer longer `rich_text` block content in `extensions/slack/src/monitor/message-handler/prepare-content.ts:257`, but `prepareSlackMessage` still computes mention metadata from `message.text` before content recovery, strips commands from `message.text` at `extensions/slack/src/monitor/message-handler/prepare.ts:607`, and stores `CommandBody`/`BodyForCommands` from that same preview at `extensions/slack/src/monitor/message-handler/prepare.ts:928`. The full body is not resolved until later, after room mention/command ingress gates can skip the message.
- Impact: In mention-required Slack rooms, a long message whose bot mention appears after Slack’s socket-mode preview cutoff can still be dropped even though the full `rich_text` block contains the mention. Long Slack text commands such as `/new <long instructions>` or `/reset <long instructions>` can also enter the agent with full `BodyForAgent` but truncated `CommandBody`, so reset/session/directive handling can silently use the preview tail instead of the recovered full text.
- Suggested fix: Split text-only rich-text extraction from media hydration and run it before mention/command ingress. Use the recovered primary text, with Slack mention stripping, for mention metadata, command detection, pending history, `CommandBody`, and `BodyForCommands`; keep the existing media/content resolver for attachments. Add tests for a long rich-text channel message with `<@B1>` only after the preview cutoff and a long `/new` DM whose command tail differs from the preview.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: construct a Slack channel message with truncated `text` that lacks `<@B1>`, a longer `blocks: [{ type: "rich_text", ... }]` body that includes `<@B1>` after the cutoff, and `requireMention: true`; `prepareSlackMessage` will evaluate ingress from `message.text` before rich-text recovery. For the command variant, use a DM with `text` as Slack’s truncated `/new ...` preview and full rich text in blocks; `RawBody` is recovered but `CommandBody` remains preview-derived.
- Is this the best way to solve the issue? yes: the recovered Slack text needs to become the canonical text fact before ingress, while media hydration can remain lazy/later.
