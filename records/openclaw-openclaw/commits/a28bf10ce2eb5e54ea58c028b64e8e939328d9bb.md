# Commit a28bf10c

## Summary

Found one low-severity compatibility gap in the new native-tool `after_tool_call` observation path.

## Findings

### Low: Declined native shell/patch items are suppressed when PostToolUse relay is enabled

- Kind: compatibility
- File: `extensions/codex/src/app-server/event-projector.ts`
- Line: 895
- Evidence: `shouldEmitAfterToolCallObservation()` suppresses every `commandExecution`, `fileChange`, and `mcpToolCall` item whenever `nativePostToolUseRelayEnabled` is true. `run-attempt.ts:1451` sets that flag from `nativeHookRelay.allowedEvents.includes("post_tool_use")`, which is the default relay set. But Codex command/file schemas include terminal `declined` status, and the projector already maps that to blocked/error payloads. A declined item is not actually covered by the PostToolUse relay path, because OpenClaw’s native relay only calls `runAgentHarnessAfterToolCallHook()` from `runNativeHookRelayPostToolUse()`.
- Impact: With default native relay settings, plugin `after_tool_call` hooks do not observe denied native shell or patch attempts, while the same projector emits those observations when the relay is disabled and dynamic tools do emit `after_tool_call` for blocked attempts. Plugins that audit, count, or react to blocked tool attempts can miss native Codex denials.
- Suggested fix: Do not suppress projector observations for statuses that cannot be covered by PostToolUse, at least `itemStatus(item) === "blocked"` / raw `declined`. Keep suppressing executed completed/failed items that the native PostToolUse relay can cover, or track actual relay completions by item id if Codex exposes a stable id.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes: instantiate `CodexAppServerEventProjector` with `{ nativePostToolUseRelayEnabled: true }`, install an `after_tool_call` hook, and send an `item/completed` notification for a `commandExecution` item with `status: "declined"`. The suppression branch prevents the hook; with the option false, the new `itemToolError()` path would emit it.
- Is this the best way to solve the issue? Yes, narrowly: status-aware suppression preserves duplicate prevention for real PostToolUse completions while restoring blocked-attempt observations.
