# Commit 9d68c67

## Summary

Found one fail-open regression in the OpenGrep PR workflow. The new shallow checkout relies on `ensure-base-commit`, but that helper can finish successfully even when the base SHA is still unavailable; the scanner script then suppresses the failed diff and exits 0 without scanning changed PR paths.

## Findings

### Medium: OpenGrep PR scan can silently skip when the base fetch does not resolve

- Kind: security
- File: `.github/workflows/opengrep-precise.yml`
- Line: 52
- Evidence: This commit replaces `fetch-depth: 0` with `fetch-depth: 1` and adds `.github/actions/ensure-base-commit` before running `scripts/run-opengrep.sh --changed`. The helper warns on failed fetches and, if the base is still unavailable, reaches its final `echo` without exiting non-zero. `scripts/run-opengrep.sh` then runs `git diff ... 2>/dev/null || true`; with a missing base SHA it produces no paths and exits 0 with “No changed first-party paths for opengrep.” I reproduced that exact skip path with a fake `opengrep` on `PATH` and a missing valid-looking base ref; the script exited `0` without invoking the scanner.
- Impact: A transient fetch failure, stale/unreachable PR base SHA, or helper fetch regression can turn a PR OpenGrep scan into a successful no-op. Before this change, the full checkout made the diff base available as part of checkout, so this fail-open mode was not in this workflow’s critical path.
- Suggested fix: Make `ensure-base-commit` fail closed when a non-empty concrete `base-sha` remains unavailable after all fetch attempts, or add an explicit `git rev-parse --verify "${{ github.event.pull_request.base.sha }}^{commit}"` assertion in this workflow before running OpenGrep. Consider also making `scripts/run-opengrep.sh --changed` fail when an explicit `OPENCLAW_OPENGREP_BASE_REF` cannot be diffed in CI.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. With `OPENCLAW_OPENGREP_BASE_REF` set to a missing 40-character SHA and a stub `opengrep` command, `scripts/run-opengrep.sh --changed --sarif --error` prints “No changed first-party paths for opengrep.” and exits `0`.
- Is this the best way to solve the issue? yes. The workflow is a security scanner, so failure to establish the PR diff base should fail the job instead of converting the scan to an empty path set.
