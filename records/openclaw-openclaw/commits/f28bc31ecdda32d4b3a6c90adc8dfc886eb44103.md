## Findings

### High: Active ClawSweeper PRs using `clownfish/` heads can now be auto-closed

- Kind: regression
- File: `.github/workflows/labeler.yml`
- Line: 378
- Evidence: The labeler exemption now only checks `headRefName.startsWith("clawsweeper/")`, then falls through to the open-PR count path that adds `r: too-many-prs` when an author has more than 10 open PRs at `.github/workflows/labeler.yml:399`. Barnacle likewise now only removes that label for `clawsweeper/` heads at `scripts/github/barnacle-auto-response.mjs:241` and `scripts/github/barnacle-auto-response.mjs:1034`, while `r: too-many-prs` is a closing rule at `scripts/github/barnacle-auto-response.mjs:30`.
- Evidence: `gh pr list --repo openclaw/openclaw --state open --author app/openclaw-clownfish --limit 200` returned 42 open PRs, all sampled heads still under `clownfish/...` such as `clownfish/ghcrawl-199269-agentic-merge`; the same check for `app/openclaw-clawsweeper` returned 0 open PRs. A direct import check also returned `false` for `isClawsweeperPullRequest({ head: { ref: "clownfish/ghcrawl-199269-agentic-merge" } })`.
- Impact: On the next opened/synchronized/edited automation PR while this producer still uses the old branch namespace, labeler can add `r: too-many-prs` because the author has 42 open PRs. The auto-response workflow then sees the label and closes the PR instead of removing the stale limit label. This breaks the automation queue this exemption was added to protect.
- Suggested fix: Accept both `clownfish/` and `clawsweeper/` prefixes during the transition in both labeler and Barnacle, and keep a regression test using a current `app/openclaw-clownfish` + `clownfish/...` fixture. Remove the legacy prefix only after the producer and existing queue have fully cut over.
