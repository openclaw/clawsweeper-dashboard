# Commit 4ea0556

## Summary

Found one medium-severity validation false positive in the new proxy preflight command. The touched code is still present on current `main`; only `CHANGELOG.md` changed after this commit.

## Findings

### Medium: Default denied canary can falsely pass for non-local proxies

- Kind: security
- File: `src/infra/net/proxy/proxy-validation.ts`
- Line: 247
- Evidence: `createLoopbackDeniedCanary()` binds the canary to the CLI host’s `127.0.0.1`, then marks transport errors as blocked. `runDeniedCheck()` returns `ok: true` for those transport errors. With a proxy URL such as `http://proxy.internal:3128`, the denied URL `http://127.0.0.1:<port>/` is resolved from the proxy endpoint, not from the CLI host. A permissive remote/container proxy can therefore fail to connect simply because no service exists on its own loopback port, and the command reports the denied check as passed.
- Impact: Operators can get a passing `openclaw proxy validate` result even though the proxy has not proven it blocks loopback/private destinations. That undercuts the command’s security purpose before deploying proxy-routed OpenClaw commands.
- Suggested fix: Only treat the built-in loopback canary transport error as proof when the configured proxy endpoint is local to the validation host/network namespace. For non-loopback proxy hosts, fail as unverifiable and require an explicit `--denied-url` that is reachable if the proxy permits it, or provide a remote canary strategy.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. A focused `tsx` check with `proxyUrl: "http://proxy.internal:3128"` and a denied transport error returned `ok: true` with a generated `http://127.0.0.1:<port>/` denied URL.
- Is this the best way to solve the issue? Yes directionally. The default local canary is only authoritative when the proxy’s loopback is the same loopback being served by the validator; otherwise the check should be marked unverifiable instead of passing.
