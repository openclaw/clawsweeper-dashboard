# Commit f738263

## Summary

Found one packaging guard regression. The test now allows Discord to remain inside the core npm package, but the core package does not declare all Discord runtime dependencies needed by the packaged runtime sidecars.

## Findings

### Medium: Discord is whitelisted as core-packaged without dependency proof

- Kind: regression
- File: `test/plugin-npm-release.test.ts`
- Line: 193
- Evidence: The new `corePackageRuntimePluginIds = new Set(["discord"])` filters Discord out of the assertion that publishable plugin dist trees must be excluded from the core package. `extensions/discord/package.json:10` declares `@discordjs/voice`, `discord-api-types`, and `opusscript`; `package.json:1644` root dependencies include shared deps like `https-proxy-agent`, `typebox`, `undici`, and `ws`, but not those three Discord-specific packages. Packaged Discord runtime surfaces re-export code that imports `discord-api-types` at runtime, for example `extensions/discord/src/monitor/native-command.ts:1`.
- Impact: A published `openclaw` tarball can include `dist/extensions/discord/**` while npm does not install all dependencies required by those files. Importing or activating the packaged Discord runtime can then fail with module resolution errors instead of prompting/installing the external `@openclaw/discord` package path.
- Suggested fix: Either keep Discord excluded from the core package, or make the core package truly own the packaged Discord runtime by declaring the missing runtime deps and adding a guard that every whitelisted core-packaged plugin has all runtime deps available from the core install. A package acceptance import smoke for Discord’s packaged runtime would also close the gap.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes, at source/package-contract level: comparing non-excluded packaged plugin manifests against root dependencies reports `discord: @discordjs/voice`, `discord: discord-api-types`, and `discord: opusscript` as missing. I did not build and install a fresh tarball to hit the runtime `ERR_MODULE_NOT_FOUND`.
- Is this the best way to solve the issue? Unclear. If Discord is intentionally packaged in core, root dependency ownership plus an explicit guard is the safer direction. If Discord should remain independently installed like other publishable plugins, restoring the package exclusion is cleaner.
