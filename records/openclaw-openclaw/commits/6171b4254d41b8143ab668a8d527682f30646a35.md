# Commit 6171b42

## Summary

Found one regression: the new Discord picker runtime override is honored for normal agent turns, but not for status or compaction paths.

## Findings

### Medium: Session runtime overrides only affect normal runs, leaving status and compaction on the wrong harness

- Kind: regression
- File: `src/status/status-text.ts`
- Line: 141
- Evidence: The picker persists explicit runtime choices as `agentRuntimeOverride` and deletes `agentHarnessId` in `extensions/discord/src/monitor/native-command-model-picker-apply.ts:60`. Normal turns then read that field and pass `agentHarnessRuntimeOverride` into `runEmbeddedPiAgent` in `src/auto-reply/reply/agent-runner-execution.ts:1529` and `src/auto-reply/reply/agent-runner-execution.ts:1811`. Status still calls `selectAgentHarness` with only `agentHarnessId` in `src/status/status-text.ts:141`, and compaction callers likewise pass only `agentHarnessId` in `src/auto-reply/reply/agent-runner-memory.ts:654`, `src/auto-reply/reply/commands-compact.ts:257`, and `src/gateway/server-methods/sessions.ts:2154`.
- Impact: A user can select Pi for an official OpenAI route in the Discord picker, and normal replies use Pi, but `/status` reports `Runtime: OpenAI Codex`; preflight/manual/gateway compaction can also resolve back to Codex. That can misreport auth/billing-sensitive runtime state or use the wrong compaction backend.
- Suggested fix: Thread the validated session runtime override through status and compaction too, preferably via a shared helper, and add `agentHarnessRuntimeOverride` to compaction params/callers.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused runtime probe showed `selectAgentHarness({ provider: "openai", agentHarnessId: "pi" })` returns `codex`, while `agentHarnessRuntimeOverride: "pi"` returns `pi`. A status probe with `sessionEntry.agentRuntimeOverride: "pi"` printed `Runtime: OpenAI Codex`.
- Is this the best way to solve the issue? yes. The new runtime override is the effective contract for session runtime selection, so every runtime-sensitive caller should consume the same field after provider validation.
