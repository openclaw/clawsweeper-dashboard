# Commit c8a953a

## Summary

Found one regression: the new cron recovery logic treats any `⚠️ 🛠️ ...` error payload as recoverable when final assistant text is preferred, which also matches real exec failures that the payload builder intentionally emits as fatal warnings.

## Findings

### Medium: Real exec failures can be marked successful on final-text cron channels

- Kind: regression
- File: `src/cron/isolated-agent/helpers.ts`
- Line: 275
- Evidence: `hasRecoveredToolWarning` now accepts every error payload whose text starts with `⚠️ 🛠️ `, then clears `hasFatalStructuredErrorPayload`. `run.ts` uses that result for the run status. But `src/agents/pi-embedded-runner/run/payloads.ts:502` emits real tool failures as `⚠️ ${toolSummary} failed`, and existing tests at `src/agents/pi-embedded-runner/run/payloads.errors.test.ts:471` assert exec failures after a success-claiming assistant reply are `isError: true` and not marked `nonTerminalToolErrorWarning`.
- Impact: For Telegram/Discord cron delivery, where `preferFinalAssistantVisibleText` is true, a run with final text like “saved in your workspace” plus a real failed exec payload `⚠️ 🛠️ Exec failed` is now recorded as `ok` and delivers only the final text. The actual failed tool warning is dropped from delivery and downgraded in diagnostics because `run.ts` passes `finalStatus: "ok"`.
- Suggested fix: Do not use the display prefix as the recoverability contract. Require structured metadata such as `nonTerminalToolErrorWarning`, or add a specific diagnostic-only/recovered-tool marker for the intended cron case, while keeping ordinary exec/mutating tool failures fatal.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused probe using `buildEmbeddedRunPayloads` produced `[final text, { text: "⚠️ 🛠️ Exec failed", isError: true }]`; passing that through `resolveCronPayloadOutcome` with `preferFinalAssistantVisibleText: true` returned `hasFatalErrorPayload: false` and delivered only the final text.
- Is this the best way to solve the issue? yes. The existing code already has a structured metadata path for non-terminal tool warnings; preserving that boundary avoids confusing real tool failures with display-only warning text.
