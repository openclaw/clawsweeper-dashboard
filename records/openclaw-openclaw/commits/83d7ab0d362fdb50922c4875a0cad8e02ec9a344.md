# Commit 83d7ab0d36

## Summary

Found one low-severity changelog gate regression: the scanner can reject a valid changelog line when a forbidden handle is mentioned before the actual `Thanks @human` attribution.

## Findings

### Low: Forbidden-handle scanner treats pre-Thanks mentions as Thanks attributions

- Kind: regression
- File: `scripts/check-changelog-attributions.mjs`
- Line: 75
- Evidence: The new scanner first checks whether a line contains `Thanks`, then runs `text.matchAll(THANKED_HANDLE_PATTERN)` over the whole line. A valid line like `- Changelog: document @dependabot[bot] attribution behavior. (#1) Thanks @alice.` returns a violation for `dependabot[bot]`, even though the bot handle is part of the changelog description, not the thanked attribution. The base implementation only matched forbidden handles after `Thanks`.
- Impact: Legitimate changelog entries that describe bot/app handles or `@openclaw` before thanking a human contributor are blocked by `validate_changelog_attribution_policy`.
- Suggested fix: After finding `Thanks`, scan only the substring from the `Thanks` match onward, or use a regex that anchors forbidden handle detection to the attribution text after `Thanks`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: a direct Node probe against current `findForbiddenChangelogThanks` returns a violation for `@dependabot[bot]` before `Thanks @alice`, while the base commit’s scanner returns no violation for the analogous pre-`Thanks @openclaw` case.
- Is this the best way to solve the issue? yes: preserving the new multi-handle/bot rules while limiting scanning to post-`Thanks` attribution text matches the original contract and avoids false positives.
