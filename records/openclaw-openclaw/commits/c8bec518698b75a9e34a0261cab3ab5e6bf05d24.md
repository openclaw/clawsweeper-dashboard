# Commit c8bec51

## Summary

Found one low-severity bug in the new dry-run diff formatter: EOF newline-only changes are now visible, but the emitted text is not a valid unified diff/patch for that case.

## Findings

### Low: Final-newline-only changes emit an invalid unified diff

- Kind: bug
- File: `extensions/oc-path/src/cli.ts`
- Line: 206
- Evidence: `splitDiffLines()` preserves the trailing empty `split("\n")` element, and the hunk builder emits it as a normal removed or added empty line. For `formatUnifiedDiff("a\n", "a", "sample.txt")`, current output is:

```diff
--- sample.txt
+++ sample.txt
@@ -1,2 +1,1 @@
 a
-
```

A focused `/tmp` check with that exact formatter output failed under `patch -p0` with `1 out of 1 hunk ignored`; standard `diff -u` represents this byte change with the `\ No newline at end of file` marker instead. The committed test at `extensions/oc-path/src/cli.test.ts:176` codifies the pseudo empty-line deletion.

- Impact: `openclaw path set --dry-run --diff` advertises unified diff output, but automation or reviewers that treat it as a real patch cannot apply or faithfully interpret EOF-newline-only previews.
- Suggested fix: Use a proven unified-diff formatter, or model line terminators explicitly and emit standard no-newline markers. Add a regression that verifies final-newline add/remove output is accepted by `patch` or matches `diff -u` semantics.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: call `formatUnifiedDiff("a\n", "a", "sample.txt")`, save the output, then try applying it to a file containing `a\n` with `patch -p0`.
- Is this the best way to solve the issue? yes: the formatter needs standard EOF newline handling; the rest of the command shape and dry-run-only guard are narrow and appropriate.
