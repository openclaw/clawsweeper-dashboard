# Commit 5a6cedc

## Summary

Found one actionable workflow reliability issue: the new `curl -L` follows Blacksmith’s 302 phone-home redirect, but it does not preserve the JSON request body, so the redirected phone-home lands as an empty POST and returns 404 in the live Windows workflow.

## Findings

### Medium: Redirected phone-home loses the POST body and still fails

- Kind: reliability
- File: `.github/workflows/windows-blacksmith-testbox.yml`
- Line: 158
- Evidence: The reviewed commit adds `-L` to the ready phone-home POST. A live run at this exact SHA (`25302568168`, https://github.com/openclaw/openclaw/actions/runs/25302568168) logged `phone_home_ready_http=404`. The same workflow content before this commit saw the Blacksmith endpoint return `302`, so the new `-L` is exercising that redirect. A focused local probe confirmed `curl -L -X POST --data-binary ...` follows 301/302/303 as `POST` with an empty body unless `--post302` or equivalent is used; 307/308 preserve the body.
- Impact: The Windows Testbox workflow can print “Testbox ready!” while the Blacksmith backend never receives a valid ready payload. That can leave the warmed Windows box unregistered or degraded for SSH/Testbox use. The hydrating phone-home at line 68 uses the same redirect pattern and can miss the SSH key/idle timeout response as well.
- Suggested fix: Preserve the POST body when following Blacksmith’s 302, for example add `--post302` to both phone-home `curl -L` calls, or resolve/use the canonical backend URL before posting. Also consider failing or warning loudly when the ready phone-home is not 2xx.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: inspect run `25302568168` at the reviewed SHA and see `phone_home_ready_http=404`; locally, a 302 redirect probe reproduces curl’s empty-body redirected POST behavior.
- Is this the best way to solve the issue? yes: preserving the POST body is the narrow fix for this workflow’s redirect-following intent. Resolving the final backend URL first would also be safe.
