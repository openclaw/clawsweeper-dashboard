# Commit 0ef1f36

## Summary

Found two concrete service-tier configuration edge cases. The Bedrock payload shape itself is correct, but some documented or config-typed parameter sources can be ignored before the new wrapper sees them.

## Findings

### Medium: Bedrock service tier ignores resolved model `params`

- Kind: bug
- File: `extensions/amazon-bedrock/register.sync.runtime.ts`
- Line: 531
- Evidence: `ModelDefinitionConfig.params` is documented as “Provider-specific request/runtime parameters passed through to provider plugins” in `src/config/types.models.ts:111`, and model resolution merges configured model params into `model.params` in `src/agents/pi-embedded-runner/model.ts:578` and attaches them at `src/agents/pi-embedded-runner/model.ts:640`. The new Bedrock hook only calls `resolveBedrockServiceTier(extraParams, ...)` and never checks `model?.params`. A focused probe with a runtime model containing `params: { serviceTier: "flex" }` returned `{}` from `resolvePreparedExtraParams(...)`, so the wrapper would not inject any `serviceTier`.
- Impact: Users who put `serviceTier` in explicit provider/model config params, including the config surface the issue requested and the type contract says is provider runtime data, get normal/default Bedrock service instead of the requested tier. For `flex`, that means the intended cost-saving behavior silently does not apply.
- Suggested fix: Merge resolved `model.params` into prepared extra params with the same precedence rules as `agents.defaults.params` / per-model defaults / call overrides, or have the Bedrock wrapper resolve from `model?.params` plus `extraParams` with explicit `extraParams` taking final precedence.
- Confidence: high

### Low: Mixed `serviceTier` / `service_tier` aliases can defeat per-model overrides

- Kind: bug
- File: `extensions/amazon-bedrock/register.sync.runtime.ts`
- Line: 48
- Evidence: `resolveExtraParams` merges default, per-model, and per-agent params with `Object.assign` in `src/agents/pi-embedded-runner/extra-params.ts:91`, so a global `serviceTier: "default"` plus a per-model `service_tier: "priority"` leaves both keys in the merged object. The Bedrock resolver then uses `extraParams?.serviceTier ?? extraParams?.service_tier`, so it chooses the lower-precedence global camelCase value. A focused probe produced `{"serviceTier":"default","service_tier":"priority"}` from `resolveExtraParams(...)`, and the Bedrock wrapper injected `{"serviceTier":{"type":"default"}}`.
- Impact: The docs say either spelling is accepted and that per-model params override global params, but the override silently fails when the two layers use different aliases.
- Suggested fix: Canonicalize Bedrock service tier aliases during extra-param preparation, using the existing source-aware alias precedence pattern (`resolveAliasedParamValue` / `applyCanonicalAliasedParamValue`), then make the Bedrock wrapper read the canonical key.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The two `node --import tsx --eval ...` probes reproduced the ignored `model.params` path and the alias-precedence failure without needing live AWS.
- Is this the best way to solve the issue? unclear. The safest direction is probably generic extra-param canonicalization/merging because the same precedence rules apply beyond Bedrock, but a Bedrock-local fix can be narrow if maintainers want minimal blast radius.
