# Commit 705bde4

## Summary

Found CI/test regressions from switching `jiti` to lazy `createRequire("jiti")` loads. The production fast path is plausible, and local `jiti` does export `createJiti`, but several existing tests rely on `vi.doMock("jiti")`; `createRequire` bypasses those mocks, so focused suites now call real `jiti` and fail.

## Findings

### Medium: Lazy `createRequire("jiti")` bypasses plugin loader test mocks

- Kind: regression
- File: `src/plugins/plugin-module-loader-cache.ts`
- Line: 115
- Evidence: `loadCreateJitiLoaderFactory()` now resolves `jiti` with `requireForJiti("jiti")`. Focused checks fail because existing tests mock `jiti` through Vitest ESM mocking but the new CJS require path loads real `jiti` instead:
  - `pnpm test src/channels/plugins/module-loader.test.ts` fails the TypeScript fallback test; real `jiti` loads the fixture and returns `{ ok: true }` instead of the mocked `{ loadedBy: "jiti" }`.
  - `pnpm test src/plugin-sdk/channel-entry-contract.test.ts` fails 3 Windows/fallback tests because real `jiti` receives simulated Windows paths and throws before the mocked loader assertions can run.
- Impact: The changed plugin loader breaks existing channel/plugin-sdk focused tests and likely the changed gate for this surface, even though the runtime fast path itself may be correct.
- Suggested fix: Keep the lazy production require, but make the fallback factory test-controllable everywhere it is used. Either route affected consumers through an injectable `createLoader` seam/test setter, or update the tests to mock `node:module`/`createRequire` consistently. Reset any global override after each test to avoid cross-file leakage.
- Confidence: high

### Medium: Channel config surface script has the same unmockable lazy Jiti path

- Kind: regression
- File: `scripts/load-channel-config-surface.ts`
- Line: 22
- Evidence: The script also replaced the static `jiti` import with `requireForJiti("jiti")`. `pnpm test src/config/load-channel-config-surface.test.ts` now fails `falls back to bun when the source-aware loader fails` at `src/config/load-channel-config-surface.test.ts:163` because the mocked `createJiti` is never called.
- Impact: The Bun fallback branch is no longer tested as intended, and the runtime-config test shard fails.
- Suggested fix: Add the same injectable/mockable Jiti factory seam for this script, or update the test to mock `createRequire` instead of `jiti`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: run `pnpm test src/plugin-sdk/channel-entry-contract.test.ts src/channels/plugins/module-loader.test.ts src/config/load-channel-config-surface.test.ts`, or run the individual failing files listed above.
- Is this the best way to solve the issue? unclear: the performance goal is sound, but the safer fix direction is to preserve lazy loading while restoring deterministic test injection/mocking for fallback paths.
