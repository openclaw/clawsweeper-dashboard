# Commit 1143f73

## Summary

Found three actionable issues: one generated API baseline gate failure, plus two cancellation paths where aborted web search work can still continue.

## Findings

### Medium: Plugin SDK API baseline was not updated

- Kind: compatibility
- File: `src/plugins/web-provider-types.ts`
- Line: 17
- Evidence: The commit changes the public `WebSearchProviderToolDefinition.execute` signature and exports `WebSearchProviderToolExecutionContext`, but `pnpm plugin-sdk:api:check` fails with `Plugin SDK API baseline drift detected` for `docs/.generated/plugin-sdk-api-baseline.sha256`.
- Impact: The plugin SDK API drift gate is red on current `main` for this commit.
- Suggested fix: Run `pnpm plugin-sdk:api:gen` and commit the updated generated hash, or avoid changing the public SDK surface.
- Confidence: high

### Medium: Auto-detect fallback can continue after caller cancellation

- Kind: reliability
- File: `src/web-search/runtime.ts`
- Line: 373
- Evidence: `runWebSearch` passes `params.signal` into each provider, but when `allowFallback` is true it catches provider errors and continues without checking `params.signal?.aborted`. Auto-detected Gemini aborts can therefore fall through to later providers; DuckDuckGo currently ignores the execution context and starts its own guarded fetch without the caller signal.
- Impact: A stopped or timed-out run can keep doing web-search network work and may return a fallback provider result instead of honoring cancellation.
- Suggested fix: Treat an already-aborted caller signal as terminal before each candidate and in the catch path, and avoid fallback after cancellation-origin aborts.
- Confidence: high

### Low: Gemini citation redirect resolution ignores the abort signal

- Kind: reliability
- File: `extensions/google/src/gemini-web-search-provider.runtime.ts`
- Line: 236
- Evidence: The main Gemini fetch receives `params.signal`, but citation redirect HEAD requests call `resolveCitationRedirectUrl(citation.url)` with no signal. That helper only accepts a URL and uses its own 5s timeout.
- Impact: If cancellation happens after the Gemini response but while resolving citations, the tool can continue redirect HEAD batches despite the run abort.
- Suggested fix: Add optional signal support to `resolveCitationRedirectUrl` and pass the Gemini execution context signal through, or skip citation resolution once aborted.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the API baseline failure: run `pnpm plugin-sdk:api:check`. The cancellation issues are source-level traces; I did not run live Gemini.
- Is this the best way to solve the issue? Yes for the baseline update. For cancellation, the safer direction is to make caller abort terminal across web-search fallback and all secondary provider fetches.
