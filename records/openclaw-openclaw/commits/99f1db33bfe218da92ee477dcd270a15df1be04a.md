# Commit 99f1db3

## Summary

Found one concrete regression in the integrated managed dreaming sweep path: the primary workspace is added to the workspace fan-out, but the primary workspace is not forwarded into phase-level session transcript ownership resolution, so main-agent session transcripts are still skipped for the primary workspace during the normal short-term dreaming sweep.

## Findings

### Medium: Managed dreaming sweep skips primary session transcripts

- Kind: bug
- File: `extensions/memory-core/src/dreaming.ts`
- Line: 557
- Evidence: `runShortTermDreamingPromotionIfTriggered` builds `workspaces` with `resolveMemoryDreamingWorkspaces(..., { primaryWorkspaceDir: fallbackWorkspaceDir, primaryAgentId: "main" })`, so the primary workspace is included. But when it calls `runDreamingSweepPhases`, it only passes the current `workspaceDir`. In `extensions/memory-core/src/dreaming-phases.ts:1755` and `extensions/memory-core/src/dreaming-phases.ts:1771`, `runDreamingSweepPhases` calls `runLightDreaming` and `runRemDreaming` without `primaryWorkspaceDir`. Those phase functions pass `params.primaryWorkspaceDir` into `ingestSessionTranscriptSignals`, and `resolveSessionAgentsForWorkspace` returns no agent IDs for the primary workspace when that option is undefined and the primary workspace is not already in `agents.list`.
- Impact: The commit fixes daily/recall promotion for the primary workspace, but session transcript ingestion remains silently disabled for the primary workspace under the issue’s configuration shape: `agents.list` contains subagent workspaces while the runtime primary workspace comes from `ctx.workspaceDir`. The newly added phase test covers the legacy light-phase trigger path, which does pass `primaryWorkspaceDir`, but not the managed short-term sweep path that runs from the managed cron.
- Suggested fix: Add `primaryWorkspaceDir?: string` to `runDreamingSweepPhases`, pass it through to `runLightDreaming` and `runRemDreaming`, and pass `fallbackWorkspaceDir` from `runShortTermDreamingPromotionIfTriggered` when invoking `runDreamingSweepPhases`. Add a `runShortTermDreamingPromotionIfTriggered` test with main and subagent session transcripts to prove the main corpus is written under the primary workspace and not copied to subagent workspaces.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Create a config with `agents.list` containing only subagent workspaces, write a `main` transcript under `OPENCLAW_STATE_DIR/agents/main/sessions`, run `runShortTermDreamingPromotionIfTriggered` with `workspaceDir` set to the primary workspace, and check that `memory/.dreams/session-corpus/<day>.txt` is not written for the primary workspace despite the workspace being swept.
- Is this the best way to solve the issue? yes. The new resolver already has the right option; the managed sweep path just drops it before session ingestion. Passing that value through preserves the intended dedupe and subagent separation behavior.
