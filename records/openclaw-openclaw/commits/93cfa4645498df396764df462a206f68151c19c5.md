# Commit 93cfa46454

## Summary

One actionable test regression found. The deduped helper weakens assertions for symlink cleanup paths.

## Findings

### Low: Missing-path helper treats dangling symlinks as deleted

- Kind: test_gap
- File: `test/scripts/postinstall-bundled-plugins.test.ts`
- Line: 34
- Evidence: `expectPathMissing` now uses `fs.access(filePath)`. `fs.access` follows symlinks, so it rejects with `ENOENT` for a dangling symlink even though the symlink directory entry still exists. This helper replaced prior `fs.lstat(...).rejects` checks for `legacySymlink` and `slackLink` at the legacy plugin runtime cleanup assertions.
- Impact: The tests can now pass if postinstall deletes the legacy runtime target but leaves the global `node_modules` symlink behind. That is exactly the cleanup behavior those `lstat` assertions were protecting, and it could hide a future regression leaving broken package-manager/runtime debris.
- Suggested fix: Use `fs.lstat` for missing-path assertions, or add a separate `expectPathEntryMissing` helper for symlink-sensitive cleanup checks and use it for `legacySymlink` and `slackLink`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused Node check with a dangling symlink shows `fs.access(link)` rejects `ENOENT` while `fs.lstat(link)` succeeds.
- Is this the best way to solve the issue? yes. The intended assertion is about the path entry itself being removed, so `lstat` is the correct filesystem primitive for symlink paths.
