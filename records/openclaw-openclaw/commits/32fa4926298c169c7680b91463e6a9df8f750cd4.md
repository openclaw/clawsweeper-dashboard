## Findings

### Medium: New mock-internals assertion breaks extension test typecheck

- Kind: regression
- File: `extensions/codex/index.test.ts`
- Line: 103
- Evidence: The commit changed the assertion to `api.registerProvider.mock.calls[0]?.[0].id`. `createTestPluginApi` returns `OpenClawPluginApi` (`src/plugin-sdk/plugin-test-api.ts`), and `OpenClawPluginApi.registerProvider` is typed as `(provider: ProviderPlugin) => void` (`src/plugins/types.ts:2525`), so `.mock` is not part of the type. `pnpm tsgo:extensions:test` reports `extensions/codex/index.test.ts(103,33): error TS2339: Property 'mock' does not exist on type '(provider: ProviderPlugin) => void'.`
- Impact: The extension test typecheck lane fails on current `main`. This lane is selected for extension test changes by `scripts/check-changed.mjs` and is also run in CI via `pnpm check:test-types`.
- Suggested fix: Keep a typed mock reference before wrapping it in `createTestPluginApi`, for example `const registerProvider = vi.fn(); ... registerProvider`, then assert `expect(registerProvider).toHaveBeenCalledTimes(1)` and inspect `registerProvider.mock.calls[0]?.[0].id`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: run `pnpm tsgo:extensions:test`; it fails with TS2339 on the changed line.
- Is this the best way to solve the issue? yes: asserting through the original `vi.fn()` preserves the stricter runtime assertion without lying about the SDK API type.
