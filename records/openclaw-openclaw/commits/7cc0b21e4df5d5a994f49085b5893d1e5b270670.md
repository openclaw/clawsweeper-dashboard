# Commit 7cc0b21e4d

## Summary

Found one actionable security regression in the custom `ensureAbsoluteDirectory` replacement.

## Findings

### Medium: Existing output directories behind symlinked parents are accepted

- Kind: security
- File: `extensions/browser/src/sdk-security-runtime.ts`
- Line: 48
- Evidence: The new wrapper returns success when `existingAncestor === absolutePath` and `fs.lstat(absolutePath)` is a real directory. `lstat` only checks the final segment, so an already-existing directory reached through a symlinked parent is accepted. The same pattern exists in the core export at `src/infra/fs-safe.ts:80`. I confirmed with a focused smoke that `ensureOutputDirectory(link/existing)` prints `ACCEPTED_EXISTING_SYMLINK_PARENT`, while the pinned `@openclaw/fs-safe` helper rejects the same path with `symlink directory path traverses a symlink within output directory`.
- Impact: Browser output/download paths use `ensureOutputDirectory` through `extensions/browser/src/browser/output-directories.ts:25`. A symlinked output parent can redirect writes to a different directory if the target directory already exists, bypassing the symlink rejection that the old dependency helper provided. This can become an arbitrary-output-location/data overwrite risk anywhere an output root or output path is influenced by config, plugin code, or browser tool input.
- Suggested fix: Do not short-circuit on the final directory. Reuse the pinned `@openclaw/fs-safe` `ensureAbsoluteDirectory` behavior after the `/tmp` and `/var` alias canonicalization, or port its forward `lstat` walk so every existing path segment is checked for symlinks before accepting an existing directory.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Create a symlinked parent whose target already contains the requested directory, then call `ensureOutputDirectory` on the symlink path. Current code accepts it; the pinned dependency rejects it.
- Is this the best way to solve the issue? yes. The safer direction is to preserve the dependency’s path-segment validation and only special-case canonical system aliases before calling it.
