# Commit 6785633d

## Summary

The main race fix is covered by focused tests and the normal delayed-send path works. I found two narrower edge-case regressions in the new pending-switch barrier: one stale pending-state path after synchronous request failures, and one attachment-loss path when a delayed send becomes queued.

## Findings

### Medium: synchronous model-patch throws can leave a permanent pending switch

- Kind: reliability
- File: `ui/src/ui/chat/session-controls.ts`
- Line: 358
- Evidence: `switchChatModel` starts the async IIFE before storing `switchPromise` in `state.chatModelSwitchPromises`. If `client.request("sessions.patch", ...)` throws synchronously, the IIFE catches and runs `finally` before lines 376-379 store the promise, so `clearPendingSwitch()` cannot delete it. The real Gateway client has a synchronous `ws.send(JSON.stringify(frame))` call after the ready-state check at `ui/src/ui/gateway.ts:743`, so a send-time exception can take this path. I also ran a no-edit jsdom smoke that dispatched the model select with a synchronous-throwing request client and observed `hasPending: true` after the failure.
- Impact: the active session can get stuck with `chatModelSwitchPromises[sessionKey]` set to a resolved-false promise. The model picker remains disabled, and future normal sends hit the pending switch in `ui/src/ui/app-chat.ts:594` and return without calling `chat.send` until the UI state is reset or the page reloads.
- Suggested fix: create the pending promise so its body starts after the state map is assigned, for example `const switchPromise = Promise.resolve().then(async () => { ... }).catch(...).finally(clearPendingSwitch); state.chatModelSwitchPromises = { ..., [targetSessionKey]: switchPromise };`. This makes cleanup run after the promise is visible.
- Confidence: medium

### Medium: queued delayed sends can drop attachment bytes after composer removal

- Kind: data_loss
- File: `ui/src/ui/app-chat.ts`
- Line: 610
- Evidence: the commit snapshots attachment `dataUrl`s before waiting, but if the model switch finishes after the session becomes busy, `handleSendChat` queues `attachmentsToSend` at line 610. `enqueueChatMessage` then strips inline payloads with `cloneChatAttachmentsMetadata` at line 188. If the user removed the original composer attachment during the wait, the payload store entry has already been released, so queue flush later reaches `buildApiAttachments` and filters the attachment out because `getChatAttachmentDataUrl` returns null. I verified this with a no-edit node smoke: delayed send with a PDF, remove/release the attachment during the wait, force the busy queue path, then flush; the resulting `chat.send` payload had `attachments: []`.
- Impact: the queued message text is sent, but the user’s attached file is silently omitted from the agent turn. This can happen when another run becomes active for the same session while the model patch is pending, then the queued item flushes on a terminal event or reconnect.
- Suggested fix: preserve complete snapshotted attachment payloads in queued items for this path, or make `enqueueChatMessage` avoid stripping `dataUrl` when the input attachment already carries an inline snapshot. Release/discard after the queued item is sent or removed.
- Confidence: medium
