# Commit 1c924c3

## Summary

Found one reliability regression in the Mantis workflow: the workflow now treats MP4 output as mandatory, but the command it calls still defines video as optional and can pass with only a screenshot.

## Findings

### Medium: Workflow requires an optional desktop MP4 artifact

- Kind: reliability
- File: `.github/workflows/mantis-discord-status-reactions.yml`
- Line: 404
- Evidence: The workflow runs `pnpm openclaw qa mantis desktop-browser-smoke`, then unconditionally copies `desktop-browser-smoke.mp4` and later requires both baseline/candidate MP4s before posting evidence. The called runtime explicitly skips video when `ffmpeg` is unavailable (`ffmpeg missing; video artifact skipped`), only asserts the PNG exists, returns `videoPath?: string`, and has a passing unit test where rsync copies no MP4.
- Impact: A valid desktop smoke run can now fail the status-reactions workflow after the expensive live baseline/candidate work, before comparison/report generation and PR evidence publishing. This is especially plausible on Crabbox images or transient apt/ffmpeg failures where screenshot capture still succeeds.
- Suggested fix: Align the contract. Either make this workflow copy/link MP4s conditionally, or add a `--require-video` mode to `desktop-browser-smoke` and fail inside that command with a clear Mantis report when the MP4 is missing.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The focused unit test `extensions/qa-lab/src/mantis/desktop-browser-smoke.runtime.test.ts` already models a passing desktop smoke with no MP4 copied back; under this workflow, the next `cp "$desktop_dir/desktop-browser-smoke.mp4"` would fail.
- Is this the best way to solve the issue? unclear. If MP4 is required evidence for this workflow, enforce it in the Mantis runtime with a clear option/report. If video is best-effort, keep the PR links conditional.
