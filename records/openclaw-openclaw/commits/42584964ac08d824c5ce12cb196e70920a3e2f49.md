# Commit 42584964ac

## Summary

Found one low-severity edge in the new opt-in `preassembly_may_overflow` path. The commit snapshots pre-assembly messages with `slice()`, which preserves the array shape but not the message objects or nested content. If a context engine mutates message objects in place while assembling, the precheck no longer has the true pre-assembly view it is explicitly trying to preserve.

## Findings

### Low: Preassembly snapshot can be mutated through shared message objects

- Kind: bug
- File: `src/agents/pi-embedded-runner/run/attempt.ts`
- Line: 2081
- Evidence: The new snapshot is `activeSession.messages.slice()`, then `assembleAttemptContextEngine` receives the original `activeSession.messages`. The nearby comment says the snapshot must preserve the true pre-assembly state because the assemble contract does not require immutable input, but `slice()` only copies the array. The added regression at `src/agents/pi-embedded-runner/run/attempt.spawn-workspace.context-engine.test.ts:394` covers array windowing (`messages.length = 0`) but not object/content mutation such as `messages[0].content = "windowed"`.
- Impact: An engine that opts into `promptAuthority: "preassembly_may_overflow"` and truncates/summarizes message objects in place can accidentally shrink `unwindowedMessages` too. The hard precheck then estimates the mutated/windowed content instead of the raw transcript, bypassing the exact overflow guard the opt-in mode is meant to request.
- Suggested fix: Snapshot the pre-assembly messages deeply enough for token estimation before calling `assemble`, for example with a message clone helper or `structuredClone` if `AgentMessage` is guaranteed JSON-like here. Add a regression where `assemble` mutates a message object's `content` in place and returns `promptAuthority: "preassembly_may_overflow"`.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes: extend the new attempt-level test so the context engine mutates `messages[0].content` from huge text to a small string before returning an assembled prompt with `promptAuthority: "preassembly_may_overflow"`. With the current shallow snapshot, the captured `unwindowedMessages` shares that mutated object.
- Is this the best way to solve the issue? yes: the contract says the runner owns the pre-assembly snapshot, so the snapshot needs to be independent of allowed context-engine input mutation. A deep-enough clone is narrower than changing the public `assemble` contract to require immutability.
