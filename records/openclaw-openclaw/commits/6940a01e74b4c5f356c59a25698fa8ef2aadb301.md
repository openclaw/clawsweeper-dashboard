# Commit 6940a01

## Summary

Found one user-visible transcript streaming bug. Final transcript text still lands, but realtime partials from at least the OpenAI realtime provider are displayed as only the latest delta chunk.

## Findings

### Low: Realtime transcript bubbles overwrite with delta chunks instead of accumulating partial text

- Kind: bug
- File: `apps/android/app/src/main/java/ai/openclaw/app/voice/TalkModeManager.kt`
- Line: 1115
- Evidence: Android handles relay `type: "transcript"` events by passing `obj["text"]` straight into `upsertRealtimeConversation`, and an existing entry is updated to exactly that text. The gateway relay forwards provider callback text unchanged in the flattened event at `src/gateway/talk-realtime-relay.ts:238` and labels non-final user/assistant text as `transcript.delta` / `output.text.delta`. The OpenAI realtime provider calls `onTranscript(..., event.delta, false)` for both assistant and user partials at `extensions/openai/realtime-voice-provider.ts:898` and `extensions/openai/realtime-voice-provider.ts:918`.
- Impact: During realtime talk, a phrase streamed as partial deltas such as `hel` then `lo` will make the Android bubble show `lo`, not `hello`, until a final transcript arrives. That undercuts the new realtime transcript feature for the live part of the UI and can leave misleading text if a turn does not produce a final event before interruption/close.
- Suggested fix: Normalize partials before updating the Android conversation. Prefer making the gateway relayâ€™s flattened Android-facing `text` cumulative per role/turn while preserving canonical nested `talkEvent` delta payloads, or have Android inspect the nested `talkEvent.type` and accumulate known delta events per active entry without duplicating providers that already emit cumulative partials.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Set `realtimeSessionId`, then deliver two `talk.event` payloads for the same role with `type: "transcript"`, `final: false`, and text chunks matching OpenAI delta behavior; the current code updates the same entry to the second chunk instead of appending it.
- Is this the best way to solve the issue? unclear. The safest fix boundary depends on whether the flattened relay payload is meant to be cumulative for mobile clients; the canonical `talkEvent` already exposes delta vs done semantics.
