# Commit a0f6ce0

## Summary

Found one concrete issue: the new mount ordering only applies when a sandbox container is newly created, but the container hash/version did not change, so existing containers with the old writable overlay order can be reused.

## Findings

### Medium: Existing sandbox containers keep the old writable skill overlay order

- Kind: security
- File: `src/agents/sandbox/workspace-mounts.ts`
- Line: 8
- Evidence: This commit moves read-only skill overlays after custom Docker binds in both `src/agents/sandbox/docker.ts:568` and `src/agents/sandbox/browser.ts:346`, but `SANDBOX_MOUNT_FORMAT_VERSION` remains `3`, the same value as the parent commit. The recreate checks in `src/agents/sandbox/docker.ts:623` and `src/agents/sandbox/browser.ts:242` hash the Docker config, workspace paths, mount format version, and skill mount list, but not the relative ordering change. A container created by the parent with order `workspace -> read-only skill overlay -> custom bind` can therefore have the same `openclaw.configHash` that current `main` expects, causing `ensureSandboxContainer` / `ensureSandboxBrowser` to skip recreation and leave the custom bind taking precedence over the read-only skill overlay.
- Impact: For users who already created a sandbox/browser container with a custom bind overlapping `/workspace/skills` or `/workspace/.agents/skills`, the security fix is not actually applied to that reused container. A sandboxed agent may still be able to modify skill instructions until the operator manually recreates the container or some unrelated config change forces recreation.
- Suggested fix: Bump the mount format/security epoch or add an explicit hash input for the new “read-only skill overlays after custom binds” contract, and add stale-container tests that seed the old hash then expect recreation for both normal and browser sandboxes. For this security boundary, also consider bypassing the hot-container “log only” path.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Create a parent-version sandbox with `workspaceAccess: "rw"`, an existing `skills/` root, and a custom bind to `/workspace/skills:rw`; then run current `main` with the same config. Because the expected hash is unchanged, the existing container is reused instead of recreated with the new bind order.
- Is this the best way to solve the issue? yes. The behavior contract changed at the Docker mount layer, so the recreate key needs to change with it.
