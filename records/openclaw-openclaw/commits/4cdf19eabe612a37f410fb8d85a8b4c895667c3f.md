# Commit 4cdf19e

## Summary

Found one actionable privacy/regression issue. The new Telegram local chat windows are injected as structured prompt context without applying the existing `contextVisibility` allowlist filter.

## Findings

### Medium: Telegram chat-window context bypasses `contextVisibility: "allowlist"`

- Kind: privacy
- File: `extensions/telegram/src/bot-handlers.runtime.ts`
- Line: 581
- Evidence: `buildPromptContextForMessage` builds `currentWindow` and `replyTargetWindow` directly from the persisted `messageCache` and maps cached `sender_id`, `sender_username`, and `body` into `UntrustedStructuredContext`. `extensions/telegram/src/bot-message-context.session.ts:459` then injects that prompt context as-is. The existing reply/forward path in the same file computes `contextVisibilityMode` and calls `evaluateSupplementalContextVisibility`, but the new chat-window path never evaluates cached message senders against the active allowlist.
- Impact: If a Telegram group previously cached messages while a sender was allowed, then the operator tightens `groupAllowFrom` and uses `contextVisibility: "allowlist"`, later allowed users can still cause old non-allowlisted cached messages to be sent to the model via the new local chat window. This bypasses the documented supplemental-context filtering contract and can expose group text the operator expected to withhold from model context.
- Suggested fix: Filter each cached chat-window message at context assembly/finalization time using the current `contextVisibilityMode` and `effectiveGroupAllow`. Treat the current local window as history/thread context, and decide explicitly whether the reply target itself gets the `allowlist_quote` exception while nearby non-target messages remain filtered.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. In a Telegram group, run with `contextVisibility: "allowlist"` and a permissive or broader group allowlist, send/cache a message from sender `2`, then tighten the allowlist to sender `1` and have sender `1` trigger a reply in the same group. Current `main` still builds `currentWindow` from the old cached message and passes it into `UntrustedStructuredContext` without checking sender `2` against the current allowlist.
- Is this the best way to solve the issue? yes. The filtering belongs where current config and `effectiveGroupAllow` are available, not in the cache itself, because cached messages can outlive config changes.
