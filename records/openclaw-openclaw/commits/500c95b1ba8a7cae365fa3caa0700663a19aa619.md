# Commit 500c95b

## Summary

Found one low-severity regression in the new `git grep` prefilter: it is not behavior-equivalent to the existing JS scanner for CRLF files containing a standalone `=======` marker line.

## Findings

### Low: CRLF separator-only conflict markers are skipped by the prefilter

- Kind: regression
- File: `scripts/check-no-conflict-markers.mjs`
- Line: 8
- Evidence: `findConflictMarkerLines()` splits on `/\r?\n/u`, so the old full scan reports a CRLF line containing exactly `=======` as a violation. The new prefilter uses `git grep -E "=======$"`, and `git grep` does not match `=======\r\n` because the `\r` remains before `$`. A focused temp-repo probe showed `findConflictMarkersInFiles(listTrackedFiles(root))` reported `only-crlf.txt`, while `findConflictMarkersInTrackedFiles(root)` omitted it.
- Impact: On CRLF checkouts or CRLF-tracked files, `check:no-conflict-markers` can now pass a tracked file that the previous scanner would reject. Full conflict blocks with `<<<<<<<` or `>>>>>>>` still get caught; this affects separator-only/partially edited marker cases.
- Suggested fix: Make the grep candidate pattern at least as broad as the JS predicate for separator lines, for example match `^=======` and let `findConflictMarkerLines()` do the exact filtering, or use a CR-aware separator expression. Add a test with a tracked file containing only `=======\r\n`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. In a temp git repo, add one tracked file containing `=======\r\n`; the old exported full-scan path reports it, while the new tracked prefilter path returns no violation for that file.
- Is this the best way to solve the issue? yes. The bug is in the candidate filter being narrower than the authoritative JS marker predicate; widening the prefilter and keeping JS as the final filter preserves the optimization without changing scanner semantics.
