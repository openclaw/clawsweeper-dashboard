# Commit 46a67d3

## Summary

Found one concrete persisted-state guard regression in the cron shape validator. The Discord and pairing changes looked safe in the reviewed paths.

## Findings

### Medium: String-schedule rows bypass payload validation

- Kind: reliability
- File: `src/cron/persisted-shape.ts`
- Line: 21
- Evidence: `getInvalidPersistedCronJobReason()` now returns `null` as soon as `schedule` is a string, before checking `payload`. A direct probe returned `missing-payload:null` and `array-payload:null` for rows with `schedule: "0 17 * * *"` and missing/array payloads, while the same array payload with an object schedule returned `missing-payload`.
- Impact: `ensureLoaded()` accepts that hydrated job into `state.store.jobs` at `src/cron/service/store.ts:99`. Timer/startup paths then dereference `job.payload.kind`, for example `src/cron/service/timer.ts:165` and `src/cron/service/timer.ts:1447`, so a persisted legacy string-schedule row with missing payload can crash scheduler startup/tick handling instead of being skipped for `openclaw doctor --fix`.
- Suggested fix: treat string schedules as schedule-compatible but continue into the existing payload validation block. Add regression coverage for string-schedule rows with missing payload and array payload, while preserving the intended acceptance of legacy string schedules and blank string `systemEvent.text`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: run `node --import tsx --input-type=module -e "import { getInvalidPersistedCronJobReason } from './src/cron/persisted-shape.ts'; const cases = [ { id: 'missing-payload', schedule: '0 17 * * *' }, { id: 'array-payload', schedule: '0 17 * * *', payload: ['systemEvent', 'tick'] }, { id: 'object-schedule-array-payload', schedule: { kind: 'every', everyMs: 60000 }, payload: ['systemEvent', 'tick'] } ]; for (const c of cases) console.log(c.id + ':' + String(getInvalidPersistedCronJobReason(c)));"`.
- Is this the best way to solve the issue? yes: the intended schedule compatibility should be separated from payload validation, rather than returning before the rest of the persisted-row guard runs.
