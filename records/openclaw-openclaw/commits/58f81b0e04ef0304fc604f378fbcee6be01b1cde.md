# Commit 58f81b0

## Summary

Found one concrete status-path regression risk: the runtime Codex context cap can still be shadowed by the older selected-provider context value in the normal `/status` command path.

## Findings

### Medium: `/status` can still report the stale OpenAI cap for native Codex runs

- Kind: bug
- File: `src/status/status-message.ts`
- Line: 750
- Evidence: `buildStatusText` now computes `runtimeContextTokens` using the Codex-aware provider mapping, but it also forwards the caller’s existing `contextTokens` into `agent.contextTokens` at `src/status/status-text.ts:377`. The normal reply/directive paths compute that value earlier with `resolveContextTokens({ provider, model })`, where native Codex still has `provider: "openai"` (`src/auto-reply/reply/get-reply-directives.ts:556`). In `buildStatusMessage`, that forwarded value becomes `cappedAgentContextTokens` and is used before `explicitRuntimeContextTokens` in `contextTokensOverride` (`src/status/status-message.ts:750`). The new `commands-status` test proves the happy path only when `contextTokens` is omitted (`src/auto-reply/reply/commands-status.test.ts:667`), so it misses the production command path that passes it.
- Impact: A user on the documented native Codex route (`openai/gpt-*` plus Codex harness) can still see `/status` report the old OpenAI/runtime cap, such as `272k`, even though `models.providers.openai-codex.models[].contextTokens` resolves to `1.0m`. That preserves the confusing status mismatch this commit says it fixes.
- Suggested fix: For status rendering, prefer the Codex-aware `runtimeContextTokens` over the previously resolved selected-provider `contextTokens` when the active runtime provider is remapped to `openai-codex`, or compute the forwarded `contextTokens` with the same context-provider remap before it reaches `buildStatusMessage`. Add a test that calls `buildStatusText` with `contextTokens: 272_000`, Codex harness config, and `openai-codex` model context set to `1_000_000`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes, source-level: follow the normal `/status` command path where `get-reply-directives` resolves `contextTokens` for `openai/gpt-5.5`, then `buildStatusText` also resolves `runtimeContextTokens` for `openai-codex/gpt-5.5`; `buildStatusMessage` prefers the former via `cappedAgentContextTokens`.
- Is this the best way to solve the issue? Yes, the safest direction is to make status use one Codex-aware context source consistently, rather than allowing an earlier selected-provider cap to override the runtime cap.
