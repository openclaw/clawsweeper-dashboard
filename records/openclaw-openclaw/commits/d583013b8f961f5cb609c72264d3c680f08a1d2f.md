# Commit d583013b8f

## Summary

Found one low-severity argument-matching gap in the new cross-platform `argPattern` enforcement.

## Findings

### Low: Direct argv allowlist checks can ignore a trailing redirection-like literal argument

- Kind: bug
- File: `src/infra/exec-command-resolution.ts`
- Line: 306
- Evidence: `matchArgPattern` strips trailing `2>&1` / `2>/dev/null`-style suffixes before applying a hand-authored space-joined `argPattern`. After this commit removes the non-Windows platform gate, direct argv evaluation on Linux/macOS now uses that stripping too. A focused check showed `{ pattern: "python3", argPattern: "^safe\\.py$" }` returns `allowlistSatisfied: true` for argv `["python3", "safe.py", "2>/dev/null"]`, while correctly returning `false` for `["python3", "safe.py", "--verbose"]`.
- Impact: An operator who intends an exact argv shape can still allow a direct `system.run` argv request with an extra literal argument if it looks like a shell redirection. Shell command text with `>` is rejected before matching, so this is limited to argv/direct execution paths, but it violates the documented “parsed command arguments” restriction.
- Suggested fix: Only apply `stripTrailingRedirections` when matching shell-command text where the token is known to be syntactic redirection, or remove that fallback from argv-based matching. Add a direct `analyzeArgvCommand`/`evaluateExecAllowlist` regression test for trailing redirection-like argv tokens.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes; a focused `pnpm exec tsx` probe against `analyzeArgvCommand` and `evaluateExecAllowlist` reproduced the mismatch.
- Is this the best way to solve the issue? unclear; preserving legacy shell-redirection tolerance may matter, but argv/direct execution should not drop literal arguments before exact matching.
