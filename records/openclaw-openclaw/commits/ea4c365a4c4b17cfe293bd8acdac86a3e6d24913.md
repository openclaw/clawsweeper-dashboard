## Findings

### Low: Exact exec description assertions fail on Windows

- Kind: compatibility
- File: `src/agents/pi-tools.deferred-followup-guidance.test.ts`
- Line: 24 and 37
- Evidence: The commit changed the test from substring checks to exact `toBe(...)` assertions for `exec.description`. Current production `describeExecTool()` returns the pinned base string only when `process.platform !== "win32"`; on Windows it appends the `"IMPORTANT (Windows): ..."` guidance and may append allowlist guidance from exec approvals (`src/agents/bash-tools.descriptions.ts:28`). A focused probe with `process.platform` set to `win32` showed `execMatchesPinnedExpectation: false` and `execHasWindowsGuidance: true`.
- Impact: Running this focused test, or a broader unit suite that includes it, on Windows fails even though the extra Windows guidance is intentional production behavior. The repository’s default Windows CI is focused and does not currently include this file, so this is not an immediate main CI break, but it regresses Windows test portability.
- Suggested fix: Keep exact assertions for the stable non-Windows base only behind a platform guard or mocked non-Windows platform, and add a separate Windows-aware assertion that accepts/includes the Windows guidance.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. On Windows, `pnpm test src/agents/pi-tools.deferred-followup-guidance.test.ts` would compare the `exec` description against a Unix-only expected string while `describeExecTool()` appends Windows guidance.
- Is this the best way to solve the issue? Yes. The production platform branch is intentional; the test should either normalize/mock platform for exact copy pinning or assert the Windows suffix separately.
