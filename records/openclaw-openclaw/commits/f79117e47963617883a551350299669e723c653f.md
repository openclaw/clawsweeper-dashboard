# Commit f79117e

## Summary

Found one actionable regression in the new Telegram edit-cache path: edited messages are cached without applying the same Telegram ingress authorization checks used for normal messages. The touched files have not changed in later `main` commits, so the issue still applies on current `main`.

## Findings

### Medium: Edited Telegram messages bypass ingress authorization before entering prompt context

- Kind: security
- File: `extensions/telegram/src/bot-handlers.runtime.ts`
- Line: 2250
- Evidence: `recordEditedMessageForReplyChain` only checks dedupe, resolves forum/thread metadata, then calls `recordMessageForReplyChain` at `extensions/telegram/src/bot-handlers.runtime.ts:2230`. Normal inbound messages go through group/channel/DM authorization first and only record after those gates at `extensions/telegram/src/bot-handlers.runtime.ts:2253` and `extensions/telegram/src/bot-handlers.runtime.ts:2329`. Cached nodes are then selected into `UntrustedStructuredContext` and reply chains at `extensions/telegram/src/bot-handlers.runtime.ts:698` and `extensions/telegram/src/bot-handlers.runtime.ts:786`.
- Impact: In a group with `groupPolicy: allowlist` or per-group `allowFrom`, an unauthorized sender can edit a visible message and have the edited text persisted in the Telegram message cache. The next authorized user message in the same chat/thread can include that unauthorized edited content in the model’s conversation context, letting a blocked sender influence the bot despite being rejected by normal message ingress.
- Suggested fix: Make edit-cache recording use the same authorization decision as normal inbound messages before writing to `messageCache`. For `edited_channel_post`, also preserve the `requireConfiguredGroup` behavior used by `channel_post`. If edit updates should never trigger processing, split the shared path so it authorizes and records only.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Configure a Telegram group allowlist that excludes user A and includes user B. Have user A edit a message in the group, then user B send a normal authorized message in the same chat/thread. The edited text can be selected through `recentBefore`/conversation context even though user A’s normal inbound message would have been rejected before caching.
- Is this the best way to solve the issue? yes. The cache is downstream prompt input, so the safest fix is to preserve the existing ingress authorization boundary before cache writes instead of trying to filter later prompt-context reads.
