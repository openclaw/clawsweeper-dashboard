## Findings

### Low: Bitwarden docs incorrectly say underscore exec ids are rejected

- Kind: bug
- File: `docs/gateway/secrets.md`
- Line: 357
- Evidence: The new docs say env-var style keys with underscores are rejected before the resolver runs. Current config validation for exec refs uses `isValidExecSecretRefId` (`src/config/zod-schema.core.ts:66`, `src/secrets/ref-contract.ts:9`), whose pattern allows underscores. A focused validation probe accepted `apiKey: { source: "exec", provider: "bws", id: "OPENAI_API_KEY" }`.
- Impact: Users with Bitwarden secret keys like `OPENAI_API_KEY` may rename or duplicate secrets unnecessarily, even though OpenClaw accepts and resolves that id shape.
- Suggested fix: Remove the sentence or replace it with the actual exec id constraints: starts with an alphanumeric, may contain letters, digits, `.`, `_`, `:`, `/`, `-`, and cannot include `.` or `..` path segments.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Run config validation with an exec SecretRef id of `OPENAI_API_KEY`; current validation accepts it, contradicting the docs.
- Is this the best way to solve the issue? yes. The code contract is already covered by parity tests; this is a documentation correction, not a runtime change.
