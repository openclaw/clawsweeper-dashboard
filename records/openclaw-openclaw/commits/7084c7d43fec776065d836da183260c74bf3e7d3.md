## Summary

Found one low-severity test regression: the new major-only assertion can accept a `PNPM_VERSION` value that the workflow setup action rejects at runtime.

## Findings

### Low: Major-only pnpm assertion no longer proves the workflow input is runnable

- Kind: test_gap
- File: `test/scripts/mantis-telegram-desktop-proof-workflow.test.ts`
- Line: 56
- Evidence: The test now checks only `workflow.env?.PNPM_VERSION?.split(".", 1)[0]`, so `PNPM_VERSION: "11"` would pass. The workflows pass that value into `.github/actions/setup-node-env`, which forwards it to `.github/actions/setup-pnpm-store-cache/action.yml`; that action rejects values that do not match `^[0-9]+(\.[0-9]+){1,2}...$`. A focused shell probe confirmed `11` is rejected while `11.0.8` is accepted.
- Impact: A future Mantis workflow edit can pass this test while making the secret-bearing Telegram proof/live workflow fail before dependency setup. The previous exact comparison against `package.json`’s package manager pin would have caught that class of drift.
- Suggested fix: Keep the major comparison only if that is the intended policy, but also assert `PNPM_VERSION` satisfies the setup action’s full semver input contract. If exact reproducibility is still desired, restore the exact comparison against the `packageManager` version with the integrity suffix stripped.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes: set either Mantis workflow’s `env.PNPM_VERSION` to `"11"` and the updated test’s major assertion accepts it, while the setup action’s regex rejects it before `corepack prepare`.
- Is this the best way to solve the issue? yes: the narrowest fix is to align the test with the local composite action contract it is meant to protect.
