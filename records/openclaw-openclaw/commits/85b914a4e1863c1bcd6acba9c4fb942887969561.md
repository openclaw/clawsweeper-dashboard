# Commit 85b914a

## Summary

Found one replay-repair bug in the snake_case tool-call handling added by this commit. The sanitizer now accepts `tool_call_id` / `call_id` blocks, but the pairing repair still extracts only camel-case `toolCall` / `toolUse` / `functionCall` blocks with `.id`, so matching tool results can be dropped as orphans immediately after the new sanitizer preserves the assistant tool call.

## Findings

### Medium: snake_case tool-call ids are accepted by input repair but missed by result pairing

- Kind: bug
- File: `src/agents/session-transcript-repair.ts`
- Line: 513
- Evidence: `hasToolCallId` now treats `call_id`, `tool_call_id`, `toolUseId`, etc. as valid ids, and `RAW_TOOL_CALL_BLOCK_TYPES` now includes `tool_call`, `tool_use`, and `function_call`. But `repairToolUseResultPairing()` still calls `extractToolCallsFromAssistant()`, whose extractor only recognizes `toolCall`, `toolUse`, and `functionCall` blocks with an `id` field. A focused check confirms the mismatch: `sanitizeToolCallInputs()` preserves `{ type: "tool_call", tool_call_id: "tool_2", ... }`, then `sanitizeToolUseResultPairing()` drops the matching `toolResult` as an orphan.
- Impact: replay history can keep a snake_case assistant tool-call block while removing its real result, losing tool output and potentially sending strict providers malformed or incomplete tool history. This is especially risky because the changed replay path in `src/agents/pi-embedded-runner/replay-history.ts` runs `sanitizeToolCallInputs()` and then `sanitizeToolUseResultPairing()` sequentially.
- Suggested fix: make the shared extractor understand the same block types and id fields as `session-transcript-repair.ts`, or canonicalize snake_case blocks to the internal `toolCall` shape before pairing. Add a regression test that runs `sanitizeToolCallInputs()` followed by `sanitizeToolUseResultPairing()` for `tool_call_id` and `call_id`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. I ran a focused `pnpm exec tsx -e` check against `sanitizeToolCallInputs()` plus `sanitizeToolUseResultPairing()` with a `tool_call_id` assistant block and matching `toolResult`; the paired output dropped the result.
- Is this the best way to solve the issue? yes. The safest fix is to align extraction/pairing with the widened sanitizer contract instead of adding provider-specific exceptions later.
