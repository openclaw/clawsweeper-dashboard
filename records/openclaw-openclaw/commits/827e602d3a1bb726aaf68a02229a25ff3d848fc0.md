# Commit 827e602

## Summary

The sanitizer added for `talk.event` is correct for untrusted synthetic events, but current runtime talk diagnostics are emitted as trusted events and the stability recorder drops trusted diagnostics before sanitization. The committed test uses `emitDiagnosticEvent()` directly, so it does not cover the production path.

## Findings

### Low: Runtime talk events still do not reach stability snapshots

- Kind: bug
- File: `src/logging/diagnostic-stability.ts`
- Line: 642
- Evidence: `startDiagnosticStabilityRecorder()` subscribes with `onDiagnosticEvent()`, but `onDiagnosticEvent()` returns early for `metadata.trusted` in `src/infra/diagnostic-events.ts:855`. Runtime talk sessions wire `{ onEvent: recordTalkDiagnosticEvent }` in paths like `src/gateway/talk-realtime-relay.ts:172` and `src/gateway/talk-transcription-relay.ts:150`; `recordTalkDiagnosticEvent()` calls `emitTrustedDiagnosticEvent()` in `src/talk/diagnostics.ts:27`. A focused smoke using `recordTalkDiagnosticEvent()` produced `{"count":0,"events":[]}` from `getDiagnosticStabilitySnapshot({ limit: 10 })`.
- Impact: the advertised fix does not work for real gateway talk/realtime/transcription flows, so support bundles and `diagnostics.stability` still omit talk latency/audio-byte events when debugging voice/talk stability.
- Suggested fix: either make the stability recorder subscribe via `onInternalDiagnosticEvent()` and explicitly accept sanitized trusted `talk.event` records, or provide a narrow diagnostic subscription that includes trusted event types known to be safe after `sanitizeDiagnosticEvent()`. Add a test that calls `recordTalkDiagnosticEvent()` instead of only direct `emitDiagnosticEvent()`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Start the stability recorder, call `recordTalkDiagnosticEvent()` with a minimal talk event, wait one `setImmediate`, then read `getDiagnosticStabilitySnapshot({ limit: 10 })`; current `main` returns `count: 0`.
- Is this the best way to solve the issue? unclear. The safest direction is a narrow allowlist for trusted events at the stability-recorder boundary so existing protection against trusted raw/log events is not widened accidentally.
