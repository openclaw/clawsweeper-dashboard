# Commit 3abb2ee

## Summary

Found one test reliability regression. The new exact equality assertion assumes the main auth store is empty, but runtime intentionally merges main auth profiles into non-main agent auth context.

## Findings

### Low: read-only auth overlay test fails when main auth profiles are staged

- Kind: reliability
- File: `src/agents/auth-profiles.readonly-sync.test.ts`
- Line: 89 on current `main`; line 76 in the reviewed commit snapshot
- Evidence: The commit added `expect(externalAuthCall?.context?.store?.profiles).toStrictEqual(baseline.profiles)`. For non-main `agentDir`, `loadAuthProfileStoreForRuntime` loads the main store and passes `mergeAuthProfileStores(mainStore, store)` into `overlayExternalAuthProfiles` (`src/agents/auth-profiles/store.ts:606`), and the external auth resolver receives that merged store (`src/agents/auth-profiles/external-auth.ts:56`). A focused check with a fake staged main profile failed exactly at this assertion because the received profiles contained both `anthropic:main` and `openai:default`.
- Impact: Normal isolated test runs pass when main auth is empty, but live-mode test runs that stage auth profiles, or any same-worker test contamination of the shared test home, can fail this test despite correct runtime behavior. That can block auth/live validation for an unrelated persisted main profile.
- Suggested fix: Make the test isolate the main auth store explicitly by setting `OPENCLAW_STATE_DIR`/`OPENCLAW_AGENT_DIR` to a temp root with an empty main store, or loosen the assertion to verify the baseline profile is included in the resolver context instead of requiring exact equality.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. I created a temp `HOME` with `.openclaw/agents/main/agent/auth-profiles.json` containing a fake `anthropic:main` profile, then ran `HOME=$tmp OPENCLAW_LIVE_TEST=1 pnpm test src/agents/auth-profiles.readonly-sync.test.ts -- --reporter=verbose`; it failed with the received profile map containing both `anthropic:main` and `openai:default`.
- Is this the best way to solve the issue? yes. The runtime merge is intentional for non-main agents, so the test should either isolate the main store or assert only the relevant agent profile/context fields.
