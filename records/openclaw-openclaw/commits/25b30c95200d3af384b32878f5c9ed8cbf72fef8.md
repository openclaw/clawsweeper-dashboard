# Commit 25b30c9

## Summary

Found one allowlist enforcement gap in the bundled MCP/LSP path.

## Findings

### Medium: Bundled runtime tools are started for narrow allowlists but not filtered by that allowlist

- Kind: security
- File: `src/agents/pi-embedded-runner/run/attempt.ts`
- Line: 1059
- Evidence: `shouldCreateBundleMcpRuntimeForAttempt()` and `shouldCreateBundleLspRuntimeForAttempt()` start bundled runtimes for exact allowlist entries like `strict__strict_probe` and `lsp_hover_typescript`, but after materialization `applyFinalEffectiveToolPolicy()` receives only config/sandbox/subagent policy inputs, not `params.toolsAllow`. The final `effectiveTools` append can therefore include every tool produced by the selected bundled runtime, not just the runtime allowlist entry.
- Impact: A cron/subagent/embedded run intended to expose one bundled MCP or LSP tool can expose sibling tools from the same runtime. For LSP, a request for `lsp_hover_typescript` can also expose `lsp_definition_typescript` / `lsp_references_typescript` if the server advertises them. For bundled MCP, one exact `server__tool` allowlist can expose the server’s other materialized tools.
- Suggested fix: After `applyFinalEffectiveToolPolicy()`, run the bundled tools through the same runtime allowlist filter used for core/plugin tools, with `getPluginToolMeta()` so `bundle-mcp`, plugin ids, and `group:plugins` still expand intentionally.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. Configure a bundled LSP server that advertises hover and definition, then run an embedded attempt with `toolsAllow: ["lsp_hover_<server>"]`; current code starts the LSP runtime and appends all policy-allowed LSP tools because `params.toolsAllow` is not applied to `filteredBundledTools`.
- Is this the best way to solve the issue? Yes. Filtering after materialization preserves the existing startup optimization while making runtime allowlists authoritative for bundled tools too.
