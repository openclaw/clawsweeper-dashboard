# Commit 16ef041

## Summary

Found one regression in the new implicit default-account preservation path: channels that merge top-level credentials into named accounts can now list and start both `default` and a named account with the same credential set.

## Findings

### Medium: Implicit default preservation can duplicate top-level credentials across enabled named accounts

- Kind: regression
- File: `src/channels/plugins/account-helpers.ts`
- Line: 77
- Evidence: `createAccountListHelpers` now adds `DEFAULT_ACCOUNT_ID` whenever `hasImplicitDefaultAccount` is true, while `mergeAccountConfig` still merges top-level channel fields into every named account at `src/channels/plugins/account-helpers.ts:167`. For Slack, a config with top-level `botToken`/`appToken` and `accounts.work: {}` now produces `["default","work"]`, and both accounts resolve enabled with the same tokens because `extensions/slack/src/accounts.ts:76` merges the root config into `work`, `extensions/slack/src/accounts.ts:194` only filters by `enabled`, and the gateway starts every listed/enabled/configured account in `src/gateway/server-channels.ts:381` and `src/gateway/server-channels.ts:438`.
- Impact: Existing configs that had top-level credentials plus a named account relying on inherited top-level fields previously started one logical Slack account (`work`). After this commit they start `default` and `work` with the same Slack Socket Mode credentials, which can duplicate inbound handling or create provider connection conflicts. The same pattern is reproducible for other opted-in channels without duplicate-owner suppression, including Zalo and Mattermost.
- Suggested fix: Suppress duplicate inherited credential owners when synthesizing the implicit default, or make opted-in channels require named accounts to provide their own account credential before treating them as independently configured. Slack can also mirror Discord’s duplicate-token owner handling so only one account starts for a given credential pair.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. I ran a focused TSX probe on current `main` with top-level Slack tokens and `accounts.work: {}`; it returned `ids: ["default","work"]` and `enabled` entries for both accounts with the same bot/app tokens.
- Is this the best way to solve the issue? unclear. The safest generic direction is to prevent synthesized implicit defaults from causing duplicate runtime credentials, but whether that belongs in the shared helper or channel-specific duplicate-owner logic depends on the intended account inheritance contract.
