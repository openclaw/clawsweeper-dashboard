# Commit 4b864109

## Summary

Found two concrete regressions in the localization plumbing. Built-in and skill command localizations now reach Discord serialization, but localized plugin command descriptions are dropped in the Discord provider assembly path. Also, reconcile compares localized desired payloads against Discord list results fetched without full localization dictionaries, so localized commands can be patched again every process restart.

## Findings

### Medium: Discord drops plugin command description localizations before command creation

- Kind: bug
- File: `extensions/discord/src/monitor/provider.commands.ts`
- Line: 54
- Evidence: `src/plugins/command-specs.ts:74` now copies `cmd.descriptionLocalizations` into plugin command specs, and `extensions/discord/src/monitor/native-command.ts:150` serializes `command.descriptionLocalizations` into the Discord command. But the actual Discord merge path at `extensions/discord/src/monitor/provider.commands.ts:54` reconstructs each plugin command as only `{ name, description, acceptsArgs }`, omitting `descriptionLocalizations`.
- Impact: `api.registerCommand({ descriptionLocalizations: ... })` appears supported by the new public/plugin-facing types and docs, but localized descriptions for plugin-provided Discord slash commands are still published as English-only defaults.
- Suggested fix: include `descriptionLocalizations: pluginCommand.descriptionLocalizations` when pushing plugin commands in `appendPluginCommandSpecs`, and add a focused provider command-spec test that feeds a localized plugin command through `resolveDiscordProviderCommandSpecs`.
- Confidence: high

### Low: Reconcile fetches Discord commands without full localization dictionaries

- Kind: reliability
- File: `extensions/discord/src/internal/api.commands.ts`
- Line: 8
- Evidence: `reconcileGlobalCommands` compares the fetched command object to the serialized desired command and patches when they differ (`extensions/discord/src/internal/command-deploy.ts:97`). The fetch path calls `rest.get(Routes.applicationCommands(clientId))` with no query. Discord’s command-list API defaults `with_localizations` to false, returning localized single-value fields instead of full `name_localizations` / `description_localizations` dictionaries; the local `discord-api-types` definition says the same. citeturn0search0
- Impact: once desired commands include `description_localizations`, a fresh deployer instance can see live commands as missing the full dictionary and issue unnecessary PATCH calls on every Discord startup/reload, increasing API churn and rate-limit exposure for localized command sets.
- Suggested fix: fetch application commands for reconcile with `with_localizations: true`, using the existing `RequestClient.get(path, query)` support, and update `extensions/discord/src/internal/api.test.ts`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. For the first finding, pass a localized plugin command through `resolveDiscordProviderCommandSpecs`; the merge object currently omits the localization field. For the second, Discord’s documented/default list-command response omits full localization dictionaries unless `with_localizations=true`, while reconcile compares against desired payloads containing `description_localizations`.
- Is this the best way to solve the issue? yes. Both fixes preserve the additive metadata contract introduced by this commit and avoid broader command-system changes.
