## Findings

### Low: Public gateway pairing docs still promise stable pending request IDs

- Kind: regression
- File: `docs/gateway/pairing.md`
- Line: 66
- Evidence: `src/infra/node-pairing.ts:291` now refreshes a single pending request only when caps, commands, and permissions match; changed approval surfaces create a replacement request and delete the old pending ID. The new test in `src/infra/node-pairing.test.ts` asserts the old request ID can no longer be approved after supersession. But `docs/gateway/pairing.md:66` still says repeated `node.pair.request` calls return the same pending request, and `docs/gateway/pairing.md:68` says repeated requests refresh the latest command snapshot.
- Impact: Operators or client authors following the public protocol docs can cache and approve a stale request ID after a node retries with changed caps, commands, or permissions, which now fails as an unknown request. The new behavior is valid, but the documented contract is stale.
- Suggested fix: Update `docs/gateway/pairing.md` to say metadata-only repeats reuse the pending request, while caps/commands/permissions changes supersede the previous request and require re-listing before approval.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes; compare the current code path in `requestNodePairing` with the public docs, or run the added infra test that proves stale IDs cannot be approved after approval-surface changes.
- Is this the best way to solve the issue? yes; the code behavior appears intentional security hardening, so the fix is to update the public protocol docs rather than revert the replacement semantics.
