# Commit 4d06491

## Summary

Found one concrete boundary regression in the new source-module Jiti fallbacks. The existing focused tests pass, but the fallback now retries TypeScript modules after any `loadChannelPluginModule` failure, including plugin-root boundary failures.

## Findings

### Medium: Source fallback can bypass bundled plugin root boundary checks

- Kind: security
- File: `src/channels/plugins/bundled.ts`
- Line: 233
- Evidence: `loadGeneratedBundledChannelModule` first calls `loadChannelPluginModule`, which performs the plugin-root boundary check, but the new catch block retries any `.ts`/`.mts`/`.cts` path directly through Jiti. That means a boundary rejection is no longer final. The same pattern exists for package-state checkers in `src/channels/plugins/package-state-probes.ts:52`.
- Impact: A malformed bundled channel `openclaw.extensions` entry or `openclaw.channel.persistedAuthState/configuredState.specifier` that resolves outside the plugin root can still be imported if it points at a TypeScript source file. Before this commit, `loadChannelPluginModule` rejected that path and stopped. A focused smoke using a temporary bundled plugin tree confirmed both surfaces load an escaping `../outside.ts` module and return success.
- Suggested fix: Preserve the boundary-validated path across the fallback. Either move the TypeScript/Jiti fallback into `loadChannelPluginModule` after `openBoundaryFileSync` succeeds, or classify boundary failures separately and never retry them. Add regression tests for escaping `.ts` bundled channel entries and package-state checker specifiers.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused smoke created a temporary bundled plugin whose package metadata pointed `openclaw.extensions` and `persistedAuthState.specifier` at `../outside.ts`; both `getBundledChannelPlugin("evil")` and `hasBundledChannelPersistedAuthState(...)` loaded the escaping module on current main.
- Is this the best way to solve the issue? yes. The fix should keep Jiti support for valid source modules while making the existing boundary check authoritative.
