# Commit 6fb9e9e

## Summary

Found one actionable bug: the gateway now rejects strict missing delivery targets before dispatch, but the CLI treats that validation error as a generic gateway failure and starts an embedded fallback run. That can still make `openclaw agent --deliver` run an agent turn even though delivery is known to be impossible.

## Findings

### Medium: Strict delivery preflight can still fall through to embedded CLI execution

- Kind: bug
- File: `src/commands/agent-via-gateway.ts`
- Line: 257
- Evidence: The new gateway preflight returns `INVALID_REQUEST` before dispatch when `deliver=true`, the channel is deliverable, and no target resolves (`src/gateway/server-methods/agent.ts:1174`). Gateway error responses are rejected as `GatewayClientRequestError` (`src/gateway/client.ts:862`). `agentCliCommand` catches every non-timeout gateway error and runs `agentCommand` embedded with the same options (`src/commands/agent-via-gateway.ts:231`, `src/commands/agent-via-gateway.ts:257`).
- Impact: A strict `openclaw agent --deliver --reply-channel <channel>` request with no `--to`/`--reply-to` can still start a local embedded agent run after the gateway correctly rejects it. This defeats the commit’s user-facing goal: avoiding a completed agent turn that cannot deliver anywhere. It can also duplicate side effects/costs for failures that should be surfaced immediately.
- Suggested fix: Do not embedded-fallback on gateway request validation errors. For example, detect `GatewayClientRequestError` with `gatewayCode === "INVALID_REQUEST"` in `agentCliCommand` and rethrow/surface it instead of calling local `agentCommand`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. The code path is deterministic: gateway strict preflight returns `INVALID_REQUEST`; `GatewayClient.request` rejects non-ok responses; `agentCliCommand` catches that rejection and invokes embedded fallback for all non-timeout errors.
- Is this the best way to solve the issue? Yes. The fallback should remain for transport failures, but user/input validation errors should not be retried through embedded execution.
