# Commit b2c3202

## Summary

Found two actionable issues in the new failure-notice/rate-limit plumbing. The main risk is that one final-payload transform can still strip the new delivery-bypass metadata, so some marked runtime failures can remain silent in message-tool-only mode.

## Findings

### Medium: Response usage appending strips the failure-notice delivery marker

- Kind: regression
- File: `src/auto-reply/reply/agent-runner-usage-line.ts`
- Line: 68
- Evidence: runtime failure notices are tagged through `markReplyPayloadForSourceSuppressionDelivery`, and `dispatchReplyFromConfig` only bypasses message-tool-only suppression when `getReplyPayloadMetadata(reply)?.deliverDespiteSourceReplySuppression === true`. The success-path injected error payload at `src/auto-reply/reply/agent-runner-execution.ts:2207` can later flow through `appendUsageLine` from `src/auto-reply/reply/agent-runner.ts:1839` when `responseUsage` is enabled and usage is nonzero. `appendUsageLine` creates `next = { ...existing, text: ... }` but does not call `copyReplyPayloadMetadata`, so the WeakMap metadata is lost before dispatch.
- Impact: users with `responseUsage: "tokens"` or `"full"` can still get no visible runtime failure notice in `message_tool_only` source-reply mode for mid-turn rate-limit/overload failures that produced usage metadata, which partially reintroduces the silent-failure behavior this commit is meant to fix.
- Suggested fix: update `appendUsageLine` to preserve payload metadata with `copyReplyPayloadMetadata(existing, next)`, and add a regression test covering a marked `isError` payload plus `responseUsage` under message-tool-only delivery.
- Confidence: medium

### Low: Codex rate-limit reset cache is process-global, not scoped to auth profile/account

- Kind: bug
- File: `extensions/codex/src/app-server/rate-limit-cache.ts`
- Line: 3
- Evidence: `rememberCodexRateLimits` stores one global value under `Symbol.for("openclaw.codexRateLimitCacheState")`, and fallback reads use `readRecentCodexRateLimits()` without a key from both `extensions/codex/src/app-server/event-projector.ts:538` and `extensions/codex/src/app-server/run-attempt.ts:1668`. The same run path resolves a concrete `startupAuthProfileId` at `extensions/codex/src/app-server/run-attempt.ts:390`, so multiple Codex auth profiles/accounts can exist while sharing one recent rate-limit snapshot.
- Impact: if one profile/account receives `/codex account` or a rate-limit notification, a different profile/account that hits a usage-limit error without embedded reset details in the next 10 minutes can display the wrong reset window. That is misleading and can expose another account’s coarse usage/reset state.
- Suggested fix: key the cache by the resolved Codex auth profile/account context and only use a cached snapshot for the same key; if no same-key snapshot exists, omit the reset time.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the metadata loss path: create a marked failure payload, enable response usage so `appendUsageLine` runs, then verify `getReplyPayloadMetadata` is absent on the cloned payload before `dispatchReplyFromConfig` checks it. The cross-profile cache issue is code-evident but I did not run a live multi-profile Codex app-server check.
- Is this the best way to solve the issue? Yes for preserving metadata in `appendUsageLine`; it matches the new metadata-copy pattern already added to reply normalization/threading. For the cache issue, scoping by resolved auth profile is the safest narrow fix.
