# Commit 59b5058c

## Summary

Found one production-path timeout interaction that likely keeps the intended partial-recovery fix from working through the real hook runner.

## Findings

### Medium: Partial timeout recovery can be cut off by the outer hook timeout

- Kind: regression
- File: `extensions/active-memory/index.ts`
- Line: 2949
- Evidence: The commit increases `TIMEOUT_PARTIAL_DATA_GRACE_MS` to 500 ms at `extensions/active-memory/index.ts:48`, and timeout recovery now waits for the subagent rejection/attached partial data at `extensions/active-memory/index.ts:1876` and `extensions/active-memory/index.ts:1904`. But the plugin registers `before_prompt_build` with `{ timeoutMs: config.timeoutMs + config.setupGraceTimeoutMs }` at `extensions/active-memory/index.ts:2837` and `extensions/active-memory/index.ts:2949`, the same budget used by the internal watchdog before the extra 500 ms recovery work. The real hook runner enforces modifying hook timeouts with `Promise.race` and rejects at the configured timeout in `src/plugins/hooks.ts:493` and `src/plugins/hooks.ts:502`.
- Impact: In production, when Active Memory hits its configured timeout and needs the new 500 ms settle window to recover a temporary transcript summary, the outer `before_prompt_build` hook timeout can fire first and discard the hook result. That means the feature this commit is trying to stabilize can still return no partial memory context and log a hook timeout instead of `timeout_partial`.
- Suggested fix: Give the registered hook timeout enough recovery headroom, for example `config.timeoutMs + config.setupGraceTimeoutMs + TIMEOUT_PARTIAL_DATA_GRACE_MS + smallSchedulerMargin`, or fire the internal watchdog early enough that `buildTimeoutRecallResult` can finish before the hook runner deadline. Add a focused test that runs Active Memory through `createHookRunner`, not only by directly invoking the captured hook function.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. Use the real plugin hook runner with Active Memory configured to a small `timeoutMs`, make the embedded subagent reject on abort after writing a temporary assistant transcript, and assert the hook result. The source path shows the hook runner timeout and Active Memory timeout share the same deadline, while the new partial-recovery path needs up to 500 ms after that deadline.
- Is this the best way to solve the issue? The fix direction is clear: the outer hook timeout must include the plugin’s post-timeout recovery window, or the internal timeout must fire before the outer timeout.
