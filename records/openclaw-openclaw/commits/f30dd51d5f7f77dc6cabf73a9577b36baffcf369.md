# Commit f30dd51

## Summary

Found one remaining race in the lazy cron shutdown path. The commit fixes the case where `stop()` wins before the lazy import resolves, but it still lets the old cron service finish startup after hot reload if `stop()` lands while the real `cron.start()` is already in flight.

## Findings

### Medium: In-flight lazy cron start can still arm the old scheduler after hot reload

- Kind: reliability
- File: `src/gateway/server-cron-lazy.ts`
- Line: 51
- Evidence: `start()` sets `resolved.started = true` before awaiting `resolved.state.cron.start()`. If hot reload calls `stop()` while that awaited start is still running, `stop()` sets `loaded.started = false` and calls `cron.stop()`. When the original `start()` resumes, the cleanup guard is `if (stopped && resolved.started)`, so it skips the second stop because `resolved.started` was already cleared. The real cron startup path can still arm timers at the end of `ops.start()` after async load/missed-job work (`src/cron/service/ops.ts:145`, `src/cron/service/ops.ts:168`, `src/cron/service/ops.ts:198`), while `stop()` only clears the current timer (`src/cron/service/ops.ts:210`, `src/cron/service/timer.ts:1605`).
- Impact: A hot reload that replaces cron state during lazy startup can still leave the old cron service armed. That can produce duplicate scheduling against old config/store state, which is the failure mode this commit is trying to eliminate.
- Suggested fix: After `await resolved.state.cron.start()`, if `stopped` is true, always mark `resolved.started = false` and call `resolved.state.cron.stop()`. Add a unit test where the mocked `cron.start()` blocks, `lazy.cron.stop()` runs after `cron.start()` has been entered, then the mock start resolves; the old service should be stopped after startup completes.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused unit test can block the mocked underlying `cron.start()`, call `lazy.cron.stop()` after the mock was entered, then resolve the start promise. Current code only records the stop made during the pending start and skips the post-start cleanup because `resolved.started` is false.
- Is this the best way to solve the issue? yes. The cleanup condition should be based on the external stopped state after startup returns, not on the bookkeeping flag that `stop()` mutates during the race.
