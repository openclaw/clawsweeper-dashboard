# Commit aaf85166

## Summary

Found one regression in the mock OpenAI fanout state machine. The commit fixes the generic beta follow-up case, but it also lets any unrelated later prompt consume the server-global fanout completion state.

## Findings

### Medium: fanout completion state can hijack unrelated mock requests

- Kind: regression
- File: `extensions/qa-lab/src/providers/mock-openai/server.ts`
- Line: 1113
- Evidence: `scenarioState.subagentFanoutPhase` is server-global and is set to `2` after the alpha result plans the beta spawn at `extensions/qa-lab/src/providers/mock-openai/server.ts:2047`. The changed completion branch now only checks `scenarioState.subagentFanoutPhase === 2 && prompt`, so the next non-empty prompt that reaches `buildAssistantText()` returns `subagent-1: ok\nsubagent-2: ok` even when that request has no fanout prompt or beta result. I verified this with a live probe: after first fanout spawn plus alpha output, an unrelated request with `Summarize the current QA status in one short sentence.` returned the fanout completion text.
- Impact: a long-lived mock provider server can corrupt the next generic QA/model request after a fanout reaches phase 2. That can create false positives or false failures in QA Lab/release lanes that reuse the same mock server while multiple scripted checks or agent turns are in flight.
- Suggested fix: keep the new generic follow-up support, but gate phase-2 completion on fanout-specific evidence, such as beta `function_call_output`/`childSessionKey`, prior fanout text in `allInputText`, or per-conversation fanout state instead of any prompt on the shared server.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Start the mock server, POST the fanout prompt to get `qa-fanout-alpha`, POST the same fanout prompt with alpha output to get `qa-fanout-beta`, then POST an unrelated generic prompt. Current `main` returns the fanout completion text.
- Is this the best way to solve the issue? yes for the immediate fix direction: the completion branch needs a fanout/beta-result discriminator, or the phase state needs to be scoped per conversation/session rather than only per server.
