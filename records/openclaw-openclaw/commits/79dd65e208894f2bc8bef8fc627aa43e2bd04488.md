# Commit 79dd65e

## Summary

Found one actionable regression in the delegated Google Meet Twilio flow: intro speech is no longer reliably attached to the call startup path.

## Findings

### Medium: Google Meet Twilio intro can be silently dropped before the realtime bridge is ready

- Kind: regression
- File: `extensions/google-meet/src/voice-call-gateway.ts`
- Line: 96
- Evidence: `joinMeetViaVoiceCallGateway()` receives `params.message`, but the `voicecall.start` request only sends `to`, `mode`, `dtmfSequence`, `requesterSessionKey`, and `sessionKey`. It then waits a fixed delay and calls `voicecall.speak`; if the realtime bridge is not active yet, the new code catches the error or `success: false`, logs a warning, returns `introSent: false`, and never retries. The Voice Call plugin already supports `message` on `voicecall.start`, stores it as `metadata.initialMessage`, and `RealtimeCallHandler` speaks that initial greeting on bridge readiness. The changed Google Meet UI help also says Twilio Meet joins now carry the intro as the initial Voice Call message, but this gateway path does not do that.
- Impact: In Google Meet `twilio` transport with talk-back mode, the dial-in leg can join without the configured intro being spoken, especially because `voicecall.start` now returns before answer/stream readiness while DTMF is queued as pre-connect TwiML. This makes the Meet voice agent appear silent until a participant speaks.
- Suggested fix: Include `message: params.message` in the `voicecall.start` request when present, and avoid a duplicate later `voicecall.speak` path or make it conditional on not using the initial-message startup path.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The existing `extensions/google-meet/src/voice-call-gateway.test.ts` path calls `joinMeetViaVoiceCallGateway()` with a message and asserts the first gateway request omits `message`; its second test also proves `voicecall.speak` failure is treated as a skipped intro.
- Is this the best way to solve the issue? yes. The Voice Call startup contract already supports initial messages and triggers them when the realtime bridge is ready, which is more reliable than a fixed post-start sleep plus best-effort `voicecall.speak`.
