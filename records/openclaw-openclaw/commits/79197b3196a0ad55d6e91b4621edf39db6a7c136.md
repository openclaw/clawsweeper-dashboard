## Findings

### Low: `NODE_VERSION=24.x` never matches hosted-toolcache Node paths

- Kind: reliability
- File: `.github/actions/setup-pnpm-store-cache/action.yml`
- Line: 58
- Evidence: The wildcard branch scans `*/bin/node`, then checks `$(NF-2)` for the major version. Hosted toolcache node paths are shaped like `/opt/hostedtoolcache/node/24.15.0/x64/bin/node`, where `$(NF-2)` is `x64`; the version is `$(NF-3)`. Several workflows set `NODE_VERSION: "24.x"` and call this action through `.github/actions/setup-node-env/action.yml`, for example `.github/workflows/mantis-telegram-live.yml:43` and `.github/workflows/mantis-telegram-live.yml:320`.
- Impact: The new repair path silently does nothing for wildcard Node versions. Most current callers still survive because `actions/setup-node` already put the right Node on `PATH`, but if this repair is needed to recover from a shadowed or stale Node/Corepack path, the `24.x` jobs keep using the wrong executable and can still fail the pnpm setup gate.
- Suggested fix: Match the version component instead of the architecture component, ideally by deriving the path relative to `$RUNNER_TOOL_CACHE/node` and using semver-aware sorting such as `sort -V`.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the lookup bug: splitting `/opt/hostedtoolcache/node/24.15.0/x64/bin/node` shows `NF-2=x64` and `NF-3=24.15.0`. The full CI failure remains conditional on the repair branch actually being needed.
- Is this the best way to solve the issue? Yes; the fix should stay in the local action and make the wildcard lookup select the requested major’s installed Node.
