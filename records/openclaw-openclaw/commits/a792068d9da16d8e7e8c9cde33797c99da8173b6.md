# Commit a792068

## Summary

Found one low-severity diagnostic bug in the new `sessions` runtime-policy alias display. It is still present on current `main`.

## Findings

### Low: Discord DM runtime-policy aliases use the conversation target as the sender id

- Kind: bug
- File: `src/commands/sessions.ts`
- Line: 269
- Evidence: `resolveDisplayRuntimePolicySessionKey` derives `peerId` from `to` before `from`, then passes that value as `SenderId`. Discord direct contexts persist `From: "discord:U1"` and `To`/`OriginatingTo: "user:U1"` (`extensions/discord/src/monitor/message-handler.process.test.ts:1187`). The actual runtime resolver prefers `SenderId` before `OriginatingTo` (`src/auto-reply/reply/runtime-policy-session-key.ts:44`), so live runtime policy resolves `agent:main:discord:default:direct:u1`, while `sessionsCommand` displayed `agent:main:discord:default:direct:user:u1` in a focused smoke check.
- Impact: `openclaw sessions --json` and the table `policy:` flag can point operators to a policy key that actual Discord DM runs do not use. Copied diagnostics/config edits can target the wrong runtime-policy session key.
- Suggested fix: do not synthesize `SenderId` from an `OriginatingTo`/conversation-target value. Persist and prefer a native direct user id for Discord direct contexts, or add a channel-aware direct-peer resolver that strips Discord `user:` targets consistently with live `SenderId`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A temporary `sessionsCommand` smoke with a persisted Discord DM main session emitted `agent:main:discord:default:direct:user:u1`; directly calling `resolveRuntimePolicySessionKey` with the equivalent live inbound context and `SenderId: "U1"` emitted `agent:main:discord:default:direct:u1`.
- Is this the best way to solve the issue? unclear. The safest direction is to mirror the live context contract rather than adding more provider-specific string guesses in `sessions.ts`.
