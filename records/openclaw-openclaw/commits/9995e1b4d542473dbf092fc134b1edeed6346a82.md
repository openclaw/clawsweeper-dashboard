## Findings

### Low: Nextcloud Talk reaction docs name the wrong bot feature

- Kind: bug
- File: `docs/tools/reactions.md`
- Line: 47
- Evidence: The new docs say Nextcloud Talk reactions require the bot to be registered with the `reaction` feature. Upstream Nextcloud Talk docs define `response` as the feature that lets a bot post messages and reactions, while `reaction` only notifies the bot about added/removed reactions. The upstream reaction endpoint itself requires `bots-v1` and posts to `/bot/{token}/reaction/{messageId}`; it does not name the `reaction` notification feature as the send prerequisite. ([nextcloud-talk.readthedocs.io](https://nextcloud-talk.readthedocs.io/en/stable/occ/))
- Impact: Users repairing an existing bot can enable the wrong feature or miss the actual send prerequisite. A bot with `reaction` but without `response` still cannot send reactions.
- Suggested fix: Change the line to require the `response` feature for sending reactions. If desired, mention that OpenClaw’s setup command also includes `reaction` for reaction webhook events.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. Compare `docs/tools/reactions.md:47` with the upstream Nextcloud Talk OCC feature descriptions and bot reaction API docs.
- Is this the best way to solve the issue? Yes. This is a wording fix; the setup docs already include a full command with `webhook`, `response`, and `reaction`.
