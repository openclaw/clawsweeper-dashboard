# Commit c14b01e

## Summary

Found one reliability regression in the workflow timeout change. The commit caps advisory native live shards by lowering the whole job timeout, but the advisory wrapper can only downgrade failures after the command returns.

## Findings

### Medium: Job-level timeout can turn advisory live shards into blocking failures

- Kind: reliability
- File: `.github/workflows/openclaw-live-and-e2e-checks-reusable.yml`
- Line: 1886
- Evidence: `validate_live_provider_suites` and `validate_live_media_provider_suites` set `timeout-minutes: ${{ matrix.timeout_minutes }}` at the job level, and this commit lowers many full-profile advisory rows from 60 to 30 minutes. The run step only converts advisory command failures to warnings after `ci-live-command-retry.sh` or `${{ matrix.command }}` returns (`.github/workflows/openclaw-live-and-e2e-checks-reusable.yml:2199`, `.github/workflows/openclaw-live-and-e2e-checks-reusable.yml:2603`). GitHub’s workflow syntax defines `jobs.<job_id>.timeout-minutes` as the maximum time before GitHub automatically cancels the job, while step `timeout-minutes` kills only the step process. ([docs.github.com](https://docs.github.com/en/actions/reference/workflows-and-actions/workflow-syntax?s=09&utm_source=openai))
- Impact: A slow advisory shard that reaches 30 minutes is canceled by GitHub before the shell can emit `::warning::Advisory live suite failed...` and exit 0. That makes the full release advisory sweep fail as a blocking job timeout instead of staying advisory. This is especially plausible for gateway profile shards: the live test file intentionally allows an uncapped explicit model sweep up to 60 minutes (`src/gateway/gateway-models.profiles.live.test.ts:616`, `src/gateway/gateway-models.profiles.live.test.ts:655`, `src/gateway/gateway-models.profiles.live.test.ts:2846`).
- Suggested fix: Keep a job-level timeout with setup slack, but enforce the advisory cap inside the run command or retry helper, for example `timeout --foreground --kill-after=30s 25m ...`, then let the existing advisory branch convert exit 124 or other failures into a warning. The Docker live gateway advisory rows already use this shape with a 25-minute command timeout under a 30-minute job cap.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? No. Reproduction needs full-profile live release secrets and a slow provider shard, or a temporary workflow probe that sleeps past 30 minutes. The static failure path is concrete: GitHub cancels the job before the advisory shell branch can run.
- Is this the best way to solve the issue? Yes. The intended cap should apply to the live command, not the whole job, so the workflow can still report the suite as advisory and successful.
