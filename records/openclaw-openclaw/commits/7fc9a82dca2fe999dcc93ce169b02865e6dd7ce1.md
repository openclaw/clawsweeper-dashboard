# Commit 7fc9a82

## Summary

Found one pacing bug that still exists on current `main`: audio is paced only when multiple frames are already queued in one `sendAudio()` call. Realtime providers that deliver one 20 ms frame per callback can still send frames to Twilio back-to-back.

## Findings

### Medium: Pacer bypasses delay for separate realtime audio callbacks

- Kind: reliability
- File: `extensions/voice-call/src/webhook/realtime-audio-pacer.ts`
- Line: 122
- Evidence: `ensurePump()` calls `pump()` whenever `timer` is null, and `pump()` clears `timer` before sending the current item. It only schedules the next timeout when `queue.length > 0`, so a single-frame `sendAudio()` leaves no cooldown timer. A focused snippet on current `main` reproduced the issue: two separate `sendAudio(Buffer.alloc(160))` calls produced `{"sentImmediately":2,"events":["media","media"]}`. This matters because Google Live and OpenAI realtime both call `config.onAudio(...)` per received audio part/delta, not necessarily as large multi-frame buffers (`extensions/google/realtime-voice-provider.ts:707`, `extensions/openai/realtime-voice-provider.ts:584`).
- Impact: The commit’s intended Twilio pacing can be defeated by provider callback shape, reintroducing bursty outbound media, Twilio-side buffering, delayed barge-in clears, and possible downstream media discard if enough audio buffers. Twilio documents that outbound media messages are buffered and played in receive order, and that `clear` is needed to interrupt buffered audio; Twilio also documents error 31931 for downstream buffer overflow after more than 10 minutes of buffered audio. ([twilio.com](https://www.twilio.com/docs/voice/media-streams/websocket-messages?utm_source=openai))
- Suggested fix: Keep a cooldown timer after every audio send, even if the queue is empty at that moment. New `sendAudio()` calls during that cooldown should append to the queue and wait for the timer-driven pump. Add a test with two separate 160-byte `sendAudio()` calls asserting the second is not sent until 20 ms advances.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: `pnpm exec tsx -e "import { RealtimeTwilioAudioPacer } from './extensions/voice-call/src/webhook/realtime-audio-pacer.ts'; const sent:any[]=[]; const pacer=new RealtimeTwilioAudioPacer({streamSid:'MZ-test', sendJson:(m)=>{sent.push(m); return true;}}); pacer.sendAudio(Buffer.alloc(160,0x7f)); pacer.sendAudio(Buffer.alloc(160,0x7f)); console.log(JSON.stringify({sentImmediately:sent.length, events:sent.map((x)=>x.event)}));"` prints two immediate media sends.
- Is this the best way to solve the issue? yes: preserving a post-send cooldown in the pacer directly fixes the timing invariant without changing provider APIs or Twilio handler ownership.
