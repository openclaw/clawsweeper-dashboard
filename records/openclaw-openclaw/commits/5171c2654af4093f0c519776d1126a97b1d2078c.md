# Commit 5171c265

## Summary

Found one compatibility regression in the new restore step. It preserves the old `agents.defaults.model` even when the provider patch is explicitly replacing/renaming the model allowlist, which can leave a legacy primary outside the rewritten allowlist.

## Findings

### Medium: Raw model restore can undo `replaceDefaultModels` migrations

- Kind: compatibility
- File: `src/plugins/provider-auth-choice-helpers.ts`
- Line: 261
- Evidence: `src/commands/models/auth.ts:266` first applies provider `configPatch` with `replaceDefaultModels`, then `src/commands/models/auth.ts:278` restores the pre-login `agents.defaults.model` whenever `--set-default` is omitted. The restore helper at `src/plugins/provider-auth-choice-helpers.ts:261` writes the old model object back wholesale. The only current `replaceDefaultModels` producer, `extensions/anthropic/cli-migration.ts:198`, rewrites legacy `claude-cli/*` model refs to canonical `anthropic/*` refs and replaces `agents.defaults.models` at `extensions/anthropic/cli-migration.ts:204`.
- Impact: a user with legacy `agents.defaults.model.primary: "claude-cli/claude-opus-4-7"` who runs `openclaw models auth login --provider anthropic --method cli` without `--set-default` can end up with `model.primary` restored to `claude-cli/...` while `agents.defaults.models` has been replaced with `anthropic/...` keys. Runtime model visibility code treats a default outside `agents.defaults.models` as not allowed and may silently fall back to the first allowed model or fail when no allowed model is available (`src/auto-reply/reply/model-selection.ts:291`).
- Suggested fix: when `result.replaceDefaultModels` is true, do not blindly restore the raw prior `model`. Either preserve the migrated `configPatch.agents.defaults.model` when it is a normalization of the prior primary, or restore only after ensuring the restored primary/fallbacks are present in the replacement allowlist. For the Anthropic CLI migration, preserving the canonical `anthropic/*` rewrite is the safer behavior.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes, source-level: start with a config whose default primary and allowlist use legacy `claude-cli/*`, run the Anthropic CLI auth path without `--set-default`, and inspect the persisted `agents.defaults.model` versus `agents.defaults.models`.
- Is this the best way to solve the issue? Unclear. The general restore guard is correct for normal provider onboarding, but migrations that set `replaceDefaultModels` need a narrower preservation rule so they can keep renamed model refs consistent.
