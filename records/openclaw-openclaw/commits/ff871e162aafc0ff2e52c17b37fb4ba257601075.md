# Commit ff871e162a

## Summary

Found two concrete follow-up issues in the provider-overlay handling. The focused tests added by the commit pass, but the new overlay shape can still produce an invalid Foundry base URL and a validated config object that fails the same validator on a second pass.

## Findings

### Medium: Foundry timeout-only overlays write a relative `/openai/v1` base URL

- Kind: bug
- File: `extensions/microsoft-foundry/provider.ts`
- Line: 43
- Evidence: `onModelSelected` now accepts `providerConfig.models ?? []`, but it also normalizes `providerConfig.baseUrl ?? ""` and then always writes `baseUrl: buildFoundryProviderBaseUrl(...)`. With no configured Foundry endpoint, `normalizeFoundryEndpoint("")` returns `""`, and `buildFoundryV1BaseUrl("")` returns `"/openai/v1"` in `extensions/microsoft-foundry/shared.ts:176`. A direct probe of the new test shape produced:
  ```json
  {
    "timeoutSeconds": 120,
    "baseUrl": "/openai/v1",
    "api": "openai-responses",
    "models": [{ "id": "gpt-5.4" }]
  }
  ```
- Impact: A user with `models.providers.microsoft-foundry.timeoutSeconds` but no Foundry endpoint can select a Foundry model and get a persisted provider config with a relative base URL. Later model calls will not have a valid Azure/Foundry endpoint and can fail URL construction or route to an unusable relative path.
- Suggested fix: In the Foundry selection hook, only rebuild/write provider config when an endpoint is present, or derive the endpoint from an existing auth profile/selection. For overlay-only Foundry config, preserve the overlay and avoid adding a model/baseUrl until onboarding has captured a real endpoint.
- Confidence: high

### Low: Materialized bundled overlays are not accepted by the same schema

- Kind: compatibility
- File: `src/config/validation.ts`
- Line: 95
- Evidence: `materializeBundledModelProviderOverlays` fills missing bundled overlay fields with `baseUrl: ""` and `models: []`, but `ModelProviderSchema` still defines `baseUrl` as `z.string().min(1).optional()` in `src/config/zod-schema.core.ts:448`. A focused probe shows `validateConfigObjectRaw({ models: { providers: { openai: { timeoutSeconds: 600 }}}})` succeeds, but passing its returned `config` back to `validateConfigObjectRaw` fails with `models.providers.openai.baseUrl: Too small: expected string to have >=1 characters`.
- Impact: `validateConfigObjectRaw` returns an `OpenClawConfig` that is not schema-valid as input. Any caller that revalidates, serializes, or uses the returned object as a write candidate can reject a config that just passed validation.
- Suggested fix: Keep the materialized representation schema-valid. Prefer leaving `baseUrl` absent for overlay entries and updating the few consumers that assumed it exists, or explicitly separate raw overlay input from strict persisted/runtime provider declarations.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The Foundry issue reproduces by calling `buildMicrosoftFoundryProvider().onModelSelected` with the exact timeout-only overlay from the added test and observing `baseUrl: "/openai/v1"`. The schema issue reproduces by validating the returned `validateConfigObjectRaw` config a second time.
- Is this the best way to solve the issue? yes for direction. Foundry should not synthesize endpoint-backed config without an endpoint. Config validation should not return a shape that its own schema rejects.
