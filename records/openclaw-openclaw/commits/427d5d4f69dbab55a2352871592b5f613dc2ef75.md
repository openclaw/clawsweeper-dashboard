# Commit 427d5d4

## Summary

Found one CI reliability regression in the new unused-file guard. The check can fail on a fresh `pnpm dlx` cache by parsing pnpm/Corepack install progress as unused file paths.

## Findings

### Medium: Fresh `pnpm dlx` progress is parsed as unused files

- Kind: reliability
- File: `scripts/check-deadcode-unused-files.mjs`
- Line: 108
- Evidence: `runKnipUnusedFiles()` concatenates `stdout` and `stderr`, and `parseKnipCompactUnusedFiles()` accepts every line containing `": "` when Knip emits files-only compact output without an `Unused files (...)` header. With fresh `XDG_CACHE_HOME` and `npm_config_store_dir`, the script parsed pnpm/Corepack progress lines as findings: `+21`, `resolved 1, reused 0, downloaded 0, added 0`, and similar. The first local `pnpm deadcode:unused-files` run failed with those exact “Unexpected unused files”; rerunning after the dlx cache warmed passed.
- Impact: `.github/workflows/ci.yml:1289` adds this command to the `check-dependencies` shard, so a fresh runner/cache can fail CI even when Knip reports exactly the allowlisted 72 unused files.
- Suggested fix: parse only Knip stdout, keep stderr only for diagnostics, or suppress pnpm/Corepack progress with a parent pnpm reporter/silent flag and add a test fixture containing stderr progress after files-only compact output.
- Confidence: high
