## Summary

Found one concrete data-loss race in the new config caching path. Current `main` has not changed the touched files since this commit, so the issue still applies.

## Findings

### Medium: Saving before the first config snapshot can overwrite the config with a partial draft

- Kind: data_loss
- File: `apps/macos/Sources/OpenClaw/ChannelsStore+Config.swift`
- Line: 79
- Evidence: Settings now loads schema and config concurrently in `SettingsRootView` (`loadConfigSchema()` and `loadConfig(force: false)` at `apps/macos/Sources/OpenClaw/SettingsRootView.swift:80`). Once schema arrives, `ChannelConfigForm` renders editable controls from schema defaults even if `configLoaded` is still false (`apps/macos/Sources/OpenClaw/ChannelConfigForm.swift:13`). Editing any field writes into the initially empty `configDraft` and sets `configDirty = true` (`apps/macos/Sources/OpenClaw/ChannelsStore+Config.swift:126`). When the delayed non-forced `config.get` returns, `applyConfigSnapshot` now ignores it because the draft is dirty (`apps/macos/Sources/OpenClaw/ChannelsStore+Config.swift:79`), leaving the draft partial. The Channels save button is enabled solely by `configDirty` (`apps/macos/Sources/OpenClaw/ChannelsSettings+ChannelSections.swift:115`), and `saveConfigDraft()` writes `self.configDraft` as the full config (`apps/macos/Sources/OpenClaw/ChannelsStore+Config.swift:139`).
- Impact: A user who edits and saves while the initial config snapshot is still in flight can persist only the edited subtree, dropping unrelated existing config keys.
- Suggested fix: Do not allow edits/saves until the first config snapshot has hydrated the draft, or merge early edits on top of the first snapshot before marking the draft saveable. Also guard `saveConfigDraft()` against `!configLoaded` so a partial pre-load draft cannot be written.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: delay `config.get`, let `config.schema` resolve first, edit a channel/config field, then click Save before `config.get` resolves. The same state can be reproduced at unit level by starting with `configLoaded == false`, calling `updateConfigValue`, then applying a non-forced snapshot and observing it is ignored.
- Is this the best way to solve the issue? yes: the safe boundary is the first successful config snapshot. Caching dirty drafts after a loaded baseline is fine; dirty drafts before any baseline should not be saveable as the full config.
