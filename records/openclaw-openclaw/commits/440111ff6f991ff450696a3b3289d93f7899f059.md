# Commit 440111f

## Summary

The commit fixes the main reported suppression path: recent or in-flight non-`getUpdates` Telegram API calls no longer prevent the polling watchdog from restarting a stale inbound poller.

I found one remaining concrete liveness gap in the same touched tracker: repeated `getUpdates` errors still refresh the stall clock, so the watchdog can stay quiet while grammY retries internally for up to the configured 1 hour.

## Findings

### Medium: repeated getUpdates errors still reset the watchdog clock

- Kind: reliability
- File: `extensions/telegram/src/polling-liveness.ts`
- Line: 85
- Evidence: `noteGetUpdatesError` records failures into `#lastGetUpdatesFinishedAt` and `#lastApiActivityAt` (`extensions/telegram/src/polling-liveness.ts:85`). `detectStall` then computes idle elapsed from `#lastGetUpdatesFinishedAt ?? #lastGetUpdatesAt` (`extensions/telegram/src/polling-liveness.ts:135`), so any failed `getUpdates` attempt that returns before `pollingStallThresholdMs` resets the watchdog. The session middleware calls `noteGetUpdatesError` for failed `getUpdates` calls (`extensions/telegram/src/polling-session.ts:252`). OpenClaw‚Äôs runner config suppresses grammY fetch logs and retries for `60 * 60 * 1000` ms (`extensions/telegram/src/monitor.ts:40`), while `@grammyjs/runner` catches `getUpdates` errors and loops until `maxRetryTime` for recoverable errors (`node_modules/@grammyjs/runner/out/runner.js:95`). A focused `tsx` simulation of failed `getUpdates` completions at 1ms, 45s, 90s, 135s, and 180s returned `no-stall` each time; the stall only appeared 120s after the final error stopped.
- Impact: a network path that repeatedly times out or fails `getUpdates` every ~45s can avoid the 120s watchdog restart and transport rebuild, delaying recovery until grammY‚Äôs 1-hour retry window expires. That conflicts with the documented/status contract that polling liveness is based on successful/completed long-poll activity, not continuous failed attempts.
- Suggested fix: separate ‚Äúlast successful getUpdates liveness‚Äù from ‚Äúlast failed getUpdates diagnostic.‚Äù Keep `finishedAt`, duration, and error fields for logs, but base `detectStall` on successful `getUpdates` completion or active request age. Add a regression test for repeated `noteGetUpdatesError` calls below the threshold over a total window greater than the threshold.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the tracker-level failure mode: drive `noteGetUpdatesStarted`/`noteGetUpdatesError`/`noteGetUpdatesFinished` every 45s, then call `detectStall({ thresholdMs: 120000 })`; it returns `null` despite no successful poll liveness for more than 120s. I did not reproduce a live Telegram network wedge.
- Is this the best way to solve the issue? Unclear. The narrowest safe direction is to keep error diagnostics but stop treating failed `getUpdates` completions as healthy liveness for the watchdog. Maintainers may want to tune restart policy if they prefer grammY‚Äôs internal retry loop for transient outage handling.
