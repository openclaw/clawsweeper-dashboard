# Commit d73f3ac

## Summary

I found two low-severity actionable regressions, both exposed by focused tests on current `main`. One is a real state-normalization leak in reactivated subagent delivery records; the other is a new gateway test assertion that is not isolated from earlier mock calls and makes the file fail.

## Findings

### Low: Reactivated subagent runs retain `delivery.lastError: null`

- Kind: regression
- File: `src/agents/subagent-delivery-state.ts`
- Line: 118
- Evidence: `replaceSubagentRunAfterSteer` creates the replacement run with a fresh `delivery: { status: "pending" }` at `src/agents/subagent-registry-run-manager.ts:450`, but `normalizeSubagentRunState` merges that with a restored active-run delivery state whose default `lastError` is `null`. The focused test `src/agents/subagent-registry.test.ts:1247` fails with `expected null to be undefined`.
- Impact: A reactivated run is supposed to clear suspended delivery fields before retrying. Persisting a raw `lastError: null` leaves the replacement record in a partially dirty state and currently breaks the regression test for that contract.
- Suggested fix: Avoid backfilling `lastError` from restored defaults when a current delivery object exists, or make active/no-error restored delivery omit `lastError` instead of setting it to `null`.
- Confidence: high

### Low: New gateway internal-effects test fails from stale mock call history

- Kind: reliability
- File: `src/gateway/server-methods/agent.test.ts`
- Line: 1695
- Evidence: The added test asserts `mocks.updateSessionStore` was never called, but the mock is not cleared before this test. Running the focused gateway test file reports 25 earlier `updateSessionStore` calls and fails at this assertion.
- Impact: The gateway test file is red even though the runtime path appears to skip visible session-store writes for `sessionEffects: "internal"`. This can block CI and obscure actual regressions.
- Suggested fix: Clear `mocks.updateSessionStore` before invoking the backend internal-effects request, or capture the call count before the request and assert it is unchanged.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: the focused Vitest wrapper fails on both affected tests.
- Is this the best way to solve the issue? yes for the test isolation failure; for the delivery state leak, the safest fix is to keep normalization from manufacturing no-op error fields on fresh replacement runs.
