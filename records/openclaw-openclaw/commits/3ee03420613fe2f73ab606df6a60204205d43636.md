# Commit 3ee0342

## Summary

Found one security-sensitive regression in the new Docker bind mirroring path. The patch mirrors `sandbox.docker.binds` into Codex host-side `writableRoots`, but it can consume changed bind config even when Docker deliberately keeps using a hot existing container whose changed config was not validated or applied.

## Findings

### High: Hot Docker config changes can grant Codex host writes to unvalidated bind roots

- Kind: security
- File: `extensions/codex/src/app-server/run-attempt.ts`
- Line: 459
- Evidence: `resolveCodexAppServerSandboxPolicyForOpenClawSandbox` now adds every writable Docker bind host root from `sandbox.docker.binds` to Codex `writableRoots` (`extensions/codex/src/app-server/run-attempt.ts:458-468`). But `resolveSandboxContext` carries `resolvedCfg.docker` into `SandboxContext` after `ensureSandboxContainer` returns, and `ensureSandboxContainer` skips `buildSandboxCreateArgs`/`validateSandboxSecurity` when an existing running container has a hash mismatch but was used within `HOT_CONTAINER_WINDOW_MS`; it only logs “recreate to apply” and records the old hash (`src/agents/sandbox/docker.ts:629-640`, `src/agents/sandbox/docker.ts:660-669`). Before this commit, the Codex policy used only `[cwd]`, so stale or rejected bind config was not reflected into host-side Codex write roots.
- Impact: A recently used Docker sandbox can keep running with its old safe mounts while a changed config such as a same-path writable bind to a blocked host root is still mirrored into Codex native Code Mode. Because the Codex app-server runs on the Gateway host and same-path binds do not disable native Code Mode, this can bypass OpenClaw’s Docker bind validation and allow host writes outside the actual sandbox/container contract.
- Suggested fix: Validate `sandbox.docker.binds` before mirroring them into Codex writable roots, even on hot hash-mismatch containers. Safer still, only mirror roots from the applied/current container config, or disable native Code Mode and use sandbox-routed tools whenever the Docker container hash does not match the requested config.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: start a Docker sandbox, keep it hot, change `sandbox.docker.binds` to a same host/container writable bind that normal validation would reject, then start a Codex app-server turn. The container path stays old because recreation is deferred, but `writableRoots` is built from the new config.
- Is this the best way to solve the issue? yes: the fix should keep Codex’s host writable roots tied to validated/applied sandbox state, not raw desired config during the hot-container grace path.
