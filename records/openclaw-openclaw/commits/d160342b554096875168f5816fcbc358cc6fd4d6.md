# Commit d160342

## Summary

Found one low-severity UI regression risk in the new fixed-position delete confirmation placement. The viewport clamp fixes edge clipping, but the popover is no longer anchored during chat scroll, so it can stay visually attached to the wrong message before confirmation.

## Findings

### Low: Delete confirm can drift away from the message it will delete

- Kind: bug
- File: `ui/src/ui/chat/grouped-render.ts`
- Line: 766
- Evidence: The popover is now `position: fixed` in `ui/src/styles/chat/grouped.css:539`, and `placeDeleteConfirmPopover()` runs only when the popover opens and once more in `requestAnimationFrame()` at `ui/src/ui/chat/grouped-render.ts:730` and `ui/src/ui/chat/grouped-render.ts:766`. The chat list itself is scrollable (`ui/src/styles/chat/layout.css:53`), but the popover registers only a document click listener, not scroll, resize, or visualViewport listeners.
- Impact: If a user opens a delete confirmation, scrolls the chat thread, then clicks the still-visible fixed popover, the popover can appear near a different message while `onDelete()` still hides the original message key from `ui/src/ui/views/chat.ts:1180`.
- Suggested fix: Reposition or dismiss the popover on `.chat-thread` scroll, `window` resize, and `visualViewport` scroll/resize, with cleanup in `dismissPopover()`. Add a focused test that changes the trigger rect after a scroll-like event.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes from the code path: open a chat message delete confirm, scroll `.chat-thread` before pressing Delete, then observe that the fixed popover remains at its previous viewport coordinates while the closure still deletes the original group key.
- Is this the best way to solve the issue? Yes: fixed positioning needs lifecycle tracking, so either dismissing on scroll or subscribing to scroll/resize and recomputing placement is the direct fix.
