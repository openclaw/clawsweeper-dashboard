# Commit 3683559

## Summary

Found one privacy regression: the new Discord voice transcript preview is sent through the verbose console path without applying OpenClaw’s documented sensitive-token redaction.

## Findings

### Medium: Voice transcript preview can print unredacted secrets in verbose console logs

- Kind: privacy
- File: `extensions/discord/src/voice/segment.ts`
- Line: 95
- Evidence: The commit logs `formatVoiceTranscriptLogPreview(transcript)` through `logVoiceVerbose(...)`. `logVoiceVerbose` delegates to `logVerbose`, and `src/globals.ts:22` prints `theme.muted(message)` directly when verbose mode is enabled. The logging docs say sensitive tokens can be redacted before they hit console output. A focused smoke check confirmed this path prints a fake credential-shaped value raw: `pnpm exec tsx -e "... logVerbose('discord voice: transcript preview OPENAI_API_KEY=sk-1234567890abcdef')"` emitted the unredacted token.
- Impact: If a Discord voice speaker says or spells an API key, payment token, password-like assignment, or other credential-shaped text while the gateway is running with verbose console output, the STT preview can expose that sensitive text in terminal logs despite the documented redaction policy. File debug logs still go through logger redaction; the console verbose path is the leaking surface.
- Suggested fix: Redact the transcript preview before passing it to `logVoiceVerbose`, or fix `logVerbose`/`logVerboseConsole` to apply `redactSensitiveText` on console output consistently. Add a regression test with a credential-shaped transcript and assert the verbose transcript preview contains the redaction marker, not the raw token.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The pure logging path reproduces with `setVerbose(true)` and `logVerbose(...)`; the Discord voice path reaches the same sink through `segment.ts:95` and `session.ts:44`.
- Is this the best way to solve the issue? no. Logging a bounded transcript preview is useful, but it should preserve the existing redaction contract for console output.
