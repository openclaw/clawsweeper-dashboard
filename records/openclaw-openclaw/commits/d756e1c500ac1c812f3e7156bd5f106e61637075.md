# Commit d756e1c

## Summary

Found one concrete regression in the Docker E2E catalog guard. The new lane is intentionally planned without an `e2eImageKind`, but the existing boundary checker still rejects every non-live Docker E2E lane that lacks one, so `pnpm lint:docker-e2e` and `pnpm lint:scripts` now fail on current `main`.

## Findings

### Medium: New source Docker lane breaks the Docker E2E boundary lint

- Kind: regression
- File: `scripts/lib/docker-e2e-scenarios.mjs`
- Line: 465
- Evidence: The new `plugin-binding-command-escape` lane is added to both `mainLanes` and release-path `core` with `e2eImageKind: false` (`scripts/lib/docker-e2e-scenarios.mjs:465` and `scripts/lib/docker-e2e-scenarios.mjs:721`). Current `scripts/check-docker-e2e-boundaries.mjs:85` rejects any non-live lane without an image kind. Running `node scripts/check-docker-e2e-boundaries.mjs` fails with:
  `release-path: package Docker E2E lane 'plugin-binding-command-escape' must declare an e2e image kind`
  and
  `main: package Docker E2E lane 'plugin-binding-command-escape' must declare an e2e image kind`.
- Impact: `package.json:1498` wires this guard as `lint:docker-e2e`, and `package.json:1518` includes it in `lint:scripts`, so the commit breaks a normal repository lint/check lane even though the planner test for the new lane passes.
- Suggested fix: Either make the lane package-backed with a valid `e2eImageKind`, or update the boundary checker and scheduler model to explicitly allow this new source-built Docker proof class, including its build/reuse behavior.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: run `node scripts/check-docker-e2e-boundaries.mjs` on current `main`.
- Is this the best way to solve the issue? yes: the catalog, checker, and planner need the same contract for source-built Docker lanes; otherwise the lint failure will persist.
