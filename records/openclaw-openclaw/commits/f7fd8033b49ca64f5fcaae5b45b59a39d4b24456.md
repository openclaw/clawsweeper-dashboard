# Commit f7fd803

## Summary

Found one remaining secret-exposure path adjacent to the fix. The changed installer now redacts authenticated git URLs in its own git command error, but plugin update error formatting can append the original authenticated install spec back onto that redacted error.

## Findings

### Medium: Plugin update errors can re-add the redacted authenticated git spec

- Kind: privacy
- File: `src/plugins/update.ts`
- Line: 116
- Evidence: `src/plugins/git-install.ts:238` now redacts `stderr`/`stdout` before returning `result.error`, and `src/plugins/git-install.test.ts:203` covers that direct installer error. But `formatGitInstallFailure` appends raw `params.spec` as `(git ${params.spec})` at `src/plugins/update.ts:116`. For git install records, `effectiveSpec` is `record.spec` at `src/plugins/update.ts:559`, then passed into `formatGitInstallFailure` on dry-run check failures at `src/plugins/update.ts:761` and update failures at `src/plugins/update.ts:892`. `src/cli/plugins-install-command.ts:350` persists the user-provided git install spec, so an install record like `git:https://token:secret@github.com/acme/demo.git` can be printed in update error output even after the installer redacts its own error detail.
- Impact: `openclaw update` / plugin update JSON can expose git URL credentials in terminal logs or automation output when an authenticated git plugin update fails.
- Suggested fix: redact and sanitize the git spec before interpolating it in `formatGitInstallFailure`, using the same `redactSensitiveUrlLikeString` + `sanitizeForLog` path as `git-install.ts`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: create or mock a git install record with `spec: "git:https://token:secret@github.com/acme/demo.git"`, force `installPluginFromGitSpec` to return an error, and call `updateNpmInstalledPlugins`; the resulting outcome message formats `(git git:https://token:secret@github.com/acme/demo.git)`.
- Is this the best way to solve the issue? yes: the update layer is the point that reintroduces the raw spec, so redacting there is the narrow fix. A broader follow-up would review whether authenticated git specs should be persisted raw at all.
