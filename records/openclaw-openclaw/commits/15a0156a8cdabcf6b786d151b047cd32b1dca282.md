# Commit 15a0156

## Summary

Found one low-severity Windows installer regression in the new replacement guidance for `main` source installs.

## Findings

### Low: Windows `-InstallMethod git -Tag main` guidance does not actually force `main`

- Kind: bug
- File: `scripts/install.ps1`
- Line: 618
- Evidence: The new npm rejection tells users to run `-InstallMethod git -Tag main` (`scripts/install.ps1:550`), and the docs now show that same command (`docs/install/installer.md:320`). But `Install-OpenClawFromGit` only accepts `RepoDir` and `SkipUpdate`, then clones or runs `git pull --rebase`; it never reads `$Tag`, checks out `main`, or resolves a requested ref (`scripts/install.ps1:618`, `scripts/install.ps1:631`, `scripts/install.ps1:642`).
- Impact: A Windows user with an existing `%USERPROFILE%\openclaw` checkout on a tag or non-main branch who follows the new replacement command after `-Tag main` is rejected will rebuild whatever checkout is already there, not the moving `main` checkout promised by the error/docs.
- Suggested fix: Pass the requested tag/ref into `Install-OpenClawFromGit` and implement the same ref resolution/checkout behavior as the POSIX installers, at least ensuring `-Tag main` checks out and updates `main` before build.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: create or use a Windows git checkout at `$GitDir` on a non-main branch/tag, then run the documented `-InstallMethod git -Tag main`; static code shows the git installer ignores `$Tag` and only pulls the current checkout.
- Is this the best way to solve the issue? yes: the POSIX installers already honor `--version main` for git installs, so aligning PowerShell behavior removes the platform mismatch.
