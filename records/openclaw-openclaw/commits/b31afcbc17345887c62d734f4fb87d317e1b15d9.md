# Commit b31afcbc

## Summary

Found one still-plausible context-boundary leak. The commit filters selected Telegram conversation context after `/new` or non-soft `/reset`, and current `main` adds the persisted `sessionStartedAt` filter, but the separate Telegram reply-chain path is still built before those filters and is passed into the model-facing message body.

## Findings

### Medium: Pre-reset direct replies can still replay old session text through `ReplyChain`

- Kind: bug
- File: `extensions/telegram/src/message-cache.ts`
- Line: 581
- Evidence: `buildTelegramReplyChain` still follows `reply_to_message` from the current Telegram message without taking a reset/session boundary. `extensions/telegram/src/bot-handlers.runtime.ts:821` builds that chain before `buildPromptContextForMessage`, and `extensions/telegram/src/bot-message-context.session.ts:305` uses the raw chain to append `[Reply chain - nearest first]` to the inbound body. The current regression test itself sends message `87227` as a reply to stale pre-session message `84670` at `extensions/telegram/src/bot.test.ts:1777`, but it only asserts `UntrustedStructuredContext`, not `Body`/`ReplyChain`.
- Impact: A user message after `/new` or after the persisted session start can still carry an old instruction into the model if it is a Telegram reply to the old message. That weakens the intended session isolation and can re-activate stale config/restart instructions outside the selected conversation-context block.
- Suggested fix: Apply the same reset/session-start boundary to `replyChainNodes` before `resolveReplyMediaForChain` and before passing `replyChain` to `processMessage`, or make `buildTelegramReplyChain` accept the boundary. Add a bot-level assertion that `payload.Body` and `payload.ReplyChain` omit pre-boundary message `84670`.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes as a source-level path: run the existing stale Telegram topic setup and inspect `payload.Body` or `payload.ReplyChain` for the pre-session reply target. I did not edit tests in this report-only review.
- Is this the best way to solve the issue? Likely yes. Filtering before media resolution and prompt construction keeps stale text and stale reply media from entering the model payload at all.
