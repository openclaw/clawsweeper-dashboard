# Commit 48af99e

## Summary

Found two concrete issues in the new incoming `details` sanitizer. The main one is that JSON-safe payloads invalid only by key count or depth still get replaced by the middleware failure sentinel when any no-op middleware, such as tokenjuice, is registered.

## Findings

### Medium: Sanitized JSON-safe details can still fail validator caps

- Kind: bug
- File: `src/agents/harness/tool-result-middleware.ts`
- Line: 138
- Evidence: `sanitizeMiddlewareDetailsValue()` only checks serialized length before `JSON.parse(serialized)`, but `isValidMiddlewareDetails()` also rejects depth over 20 and more than 1000 aggregate keys. A direct current-main smoke with a no-op handler and `details = Object.fromEntries(1001 keys)` logs `[pi] discarded invalid tool result middleware output for exec` and returns `{ status: "error", middlewareError: true }`.
- Impact: tool results with otherwise valid structured details can still be silently replaced by the failure sentinel whenever tokenjuice or another registered middleware leaves the result unchanged. The fix specifically claims to give middleware a validator-safe incoming projection, but key/depth-only invalid payloads still break that contract.
- Suggested fix: after parsing the sanitized projection, run `isValidMiddlewareDetails(parsed)` and collapse invalid key/depth payloads to a safe truncation marker, or replace the stringify pass with a cap-aware traversal that enforces bytes, depth, and key count together.
- Confidence: high

### Low: Oversized sanitization serializes the full payload before truncating

- Kind: reliability
- File: `src/agents/harness/tool-result-middleware.ts`
- Line: 120
- Evidence: after the validator rejects an oversized `details` payload, the sanitizer calls `JSON.stringify(value, ...)` over the entire original object and only then checks `serialized.length > MAX_MIDDLEWARE_DETAILS_BYTES`.
- Impact: a large tool result that previously failed closed after the validator crossed its 100 KB budget can now allocate and traverse the complete payload before being collapsed. User-triggered tools can place external API/browser/channel payloads in `details`, so this can add avoidable latency or memory pressure in the tool-result path.
- Suggested fix: use a bounded sanitizer that stops once the byte/key/depth budget is exceeded and returns the truncation marker without serializing the whole payload.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Create `createAgentToolResultMiddlewareRunner({ runtime: "pi" }, [() => undefined])`, pass a result with text content and `details` containing 1001 small keys, and the runner returns the middleware failure sentinel on current `main`.
- Is this the best way to solve the issue? yes for the direction: keep the harness-level fix, but make the sanitizer enforce the same validator caps it promises to satisfy.
