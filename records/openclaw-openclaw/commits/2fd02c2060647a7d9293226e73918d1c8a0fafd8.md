# Commit 2fd02c2060

## Summary

The commit adds the resolved-target guard to several Full Release Validation child-dispatch jobs, but misses `npm_telegram`. That job can still run after `resolve_target` fails and dispatch `npm-telegram-beta-e2e.yml` with an empty `harness_ref`.

## Findings

### Medium: `npm_telegram` can dispatch after target resolution fails

- Kind: bug
- File: `.github/workflows/full-release-validation.yml`
- Line: 666
- Evidence: `npm_telegram` has `needs: [resolve_target, prepare_release_package]`, but its job `if` starts with `always()` and does not include `needs.resolve_target.result == 'success'`. The dispatch step then sets `TARGET_SHA` from `needs.resolve_target.outputs.sha` and passes `-f harness_ref="$TARGET_SHA"` to `npm-telegram-beta-e2e.yml`. That child workflow defaults an empty `harness_ref` to `github.sha` at checkout.
- Impact: A manual Full Release Validation run with `rerun_group=npm-telegram` plus `release_package_spec` or `npm_telegram_package_spec` can run and potentially pass even when `inputs.ref` failed to resolve. The resulting Telegram proof is then tied to the workflow ref, not the requested target SHA.
- Suggested fix: Add `needs.resolve_target.result == 'success'` to the `npm_telegram.if` condition, and add a test assertion covering every Full Release Validation job that consumes `needs.resolve_target.outputs.sha` for dispatch/package resolution.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Dispatch Full Release Validation with an invalid `ref`, `rerun_group=npm-telegram`, and `release_package_spec=openclaw@beta`; `resolve_target` fails, but `npm_telegram.if` remains true because it uses `always()` and does not check `needs.resolve_target.result`.
- Is this the best way to solve the issue? yes. The same guard added by this commit to `normal_ci`, `plugin_prerelease`, `release_checks`, and `prepare_release_package` is the direct fix for this remaining child dispatch path.
