## Findings

### Low: `brave.http` logs cache writes even when caching is disabled

- Kind: bug
- File: `extensions/brave/src/brave-web-search-provider.runtime.ts`
- Line: 425
- Evidence: `executeBraveSearch` always emits `brave http cache write` after calling `writeCachedSearchPayload(...)` for both `llm-context` and `web` modes (`extensions/brave/src/brave-web-search-provider.runtime.ts:425`, `extensions/brave/src/brave-web-search-provider.runtime.ts:462`). But `cacheTtlMinutes: 0` is valid config (`src/config/zod-schema.agent-runtime.ts:326`), resolves to `ttlMs === 0` (`src/agents/tools/web-shared.ts:18`), and `writeCache` returns without storing anything when `ttlMs <= 0` (`src/agents/tools/web-shared.ts:43`). With `brave.http` enabled and caching disabled, the logs report a cache write that did not happen; the next identical query will still be a cache miss and refetch.
- Impact: Users enabling the new diagnostics specifically to verify cache hit/miss/write behavior can be misled into thinking Brave results are cached when they configured caching off. This does not change request behavior or leak secrets, but it weakens the new troubleshooting signal.
- Suggested fix: Only log `cache write` when `cacheTtlMs > 0`, or log a separate `cache skip` / `cache disabled` event when the write helper will no-op.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. Configure Brave web search with `tools.web.search.cacheTtlMinutes: 0` and `diagnostics.flags: ["brave.http"]`, then run the same Brave query twice. Source inspection shows the first call logs `cache write` with `ttlMs: 0`, but no cache entry is stored, so the second call is another miss/fetch.
- Is this the best way to solve the issue? Yes. The fix should stay local to the new diagnostics calls and preserve existing cache behavior.
