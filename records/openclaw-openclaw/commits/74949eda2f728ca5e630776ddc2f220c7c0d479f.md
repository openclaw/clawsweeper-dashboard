# Commit 74949eda

## Summary

The commit improves Telegram raw-update logging, but the new denylist redactor still lets sensitive fields from default Telegram update families through. Current `main` still has this formatter unchanged in the runtime path.

## Findings

### Medium: Raw-update redaction still leaks payment, passport, address, and media metadata

- Kind: privacy
- File: `extensions/telegram/src/raw-update-log.ts`
- Line: 4
- Evidence: `bot-core.ts` logs `stringifyTelegramRawUpdateForLog(ctx.update)` when verbose/debug logging is enabled. The redaction key sets omit sensitive Telegram fields that are reachable through the plugin’s default allowed updates, including `invoice_payload`, `paid_media_payload`, shipping address fields such as `city`, `street_line1`, and `post_code`, passport `hash`/`secret`, venue `address`, and upload `file_name`. A direct formatter probe on current `main` printed dummy values like `internal invoice payload secret`, `1 Infinite Loop`, `paid media payload secret`, `credential hash secret`, `rsa secret value`, `private-tax-return.pdf`, and `1 Secret Address`.
- Impact: Users who enable verbose/debug logs can still write private Telegram payment, address, passport credential, venue, or file metadata into gateway logs while believing this security hardening redacts raw updates.
- Suggested fix: Broaden the formatter to cover sensitive fields across the default Telegram update set, preferably with schema-aware denylist groups for payment, passport, location/venue, and media metadata, and add regression cases for these shapes.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Import `stringifyTelegramRawUpdateForLog` with `node --experimental-strip-types --input-type=module`, pass a `shipping_query`, `pre_checkout_query`, `purchased_paid_media`, `message.passport_data`, `message.successful_payment`, `message.document`, or `message.venue`, and inspect the returned string for the sensitive values listed above.
- Is this the best way to solve the issue? yes. The logging seam is already centralized; the repair should stay in the formatter and its tests rather than changing Telegram dispatch or logger plumbing.
