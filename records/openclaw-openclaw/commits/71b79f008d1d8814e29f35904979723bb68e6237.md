# Commit 71b79f0

## Summary

The patch covers malformed `data:image/...` replay well, but it misses the raw-base64 `image_url` shape that the linked issue explicitly calls out. A Responses payload containing `input_image.image_url: "/9j..."` still passes through the new sanitizer unchanged, even though the OpenAI Responses contract requires either a fully qualified URL or a base64 image inside a data URL. ([platform.openai.com](https://platform.openai.com/docs/api-reference/responses/compact?api-mode=responses&utm_source=openai))

## Findings

### Medium: Raw base64 `image_url` replay still bypasses the sanitizer

- Kind: bug
- File: `src/agents/responses-image-payload-sanitizer.ts`
- Line: 50
- Evidence: `sanitizeImageUrl()` returns any non-`data:` string unchanged at `src/agents/responses-image-payload-sanitizer.ts:50`, and `sanitizeValue()` then preserves that value for `type: "input_image"` records at `src/agents/responses-image-payload-sanitizer.ts:93`. A local runtime probe showed `{ type: "input_image", image_url: "/9j/4AAQSkZJRgABAQAAAQABAAD=" }` remains an `input_image` with the raw base64 string. The same pass-through exists for Codex mirrored-history `inputImage.imageUrl` at `extensions/codex/src/app-server/image-payload-sanitizer.ts:77`.
- Impact: The exact poisoned replay class from #82878 can still wedge a session when a Responses-shaped replay payload contains raw JPEG/base64 in `image_url` rather than a malformed `data:` URL. OpenAI’s Responses SDK/API contract says `image_url` must be a fully qualified URL or base64 in a data URL, so raw base64 will still be sent to `responses.create` and rejected. ([platform.openai.com](https://platform.openai.com/docs/api-reference/responses/compact?api-mode=responses&utm_source=openai))
- Suggested fix: Treat non-`data:` image URLs as valid only if they parse as fully qualified `http:` or `https:` URLs. Replace raw base64, relative paths, and other non-URL strings with the same `input_text` placeholder.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Run the sanitizer with a Responses-shaped `function_call_output.output` containing `{ type: "input_image", image_url: "/9j/4AAQSkZJRgABAQAAAQABAAD=" }`; it returns the raw `input_image` unchanged.
- Is this the best way to solve the issue? yes. The fix belongs at this final provider-request sanitizer boundary because payload hooks and replay repair can still hand Responses-shaped objects to the transport.
