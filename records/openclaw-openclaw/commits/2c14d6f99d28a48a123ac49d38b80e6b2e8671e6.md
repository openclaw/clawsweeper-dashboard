# Commit 2c14d6f

## Summary

Found one concrete regression risk in the new bounded shutdown hook path: the timeout is unref’d, so it may not fire before Node exits naturally, bypassing the intended explicit CLI exit code.

## Findings

### Medium: Unref’d stop-hook timeout can turn failed message commands into exit 0

- Kind: bug
- File: `src/cli/program/message/helpers.ts`
- Line: 39
- Evidence: `runMessageAction` records delivery failure in `failed`, then awaits `runPluginStopHooks()` before calling `defaultRuntime.exit(failed ? 1 : 0)`. The new `runPluginStopHooks()` races the hook promise against a timeout, but immediately calls `timeout.unref?.()`. A pending Promise does not keep Node alive, and an unref’d timer does not either. I verified the same control flow with `node -e`: with only an unresolved hook promise plus an unref’d timeout, the process exits before the timeout callback and returns `exit:0`; with a separate live handle keeping the loop alive, the timeout fires and the process exits `1`.
- Impact: if `messageCommand` fails and a registered `gateway_stop` hook never settles without any other ref’d handles, the CLI can exit naturally with status 0 before reaching `defaultRuntime.exit(1)`. That makes failed `openclaw message send`/`poll`/mutation actions look successful to scripts and CI. The existing fake-timer unit test covers the timeout branch but not this process-liveness behavior.
- Suggested fix: keep this shutdown timeout ref’d until the race resolves, or set `process.exitCode = failed ? 1 : 0` before awaiting stop hooks. For this CLI path, removing `timeout.unref?.()` is the simplest fix because the timeout is the mechanism that guarantees the later explicit exit.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: run a process with the same `Promise.race([neverSettlingHook.then(...), unrefTimeout])` shape after a simulated failure; it exits `0` before the timeout fires. Adding any ref’d handle lets the timeout fire and preserves exit `1`, confirming the liveness dependency.
- Is this the best way to solve the issue? yes: the timeout must remain capable of driving control flow to `defaultRuntime.exit(...)`; unref is appropriate for background cleanup timers, but not for a timer that gates an explicit CLI exit code.
