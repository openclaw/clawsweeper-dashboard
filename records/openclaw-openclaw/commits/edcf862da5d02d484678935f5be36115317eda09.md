# Commit edcf862

## Summary

Found a high-severity security regression in the new cleanup pass: it recursively trusts `session.json` files under `.artifacts/qa-e2e`, which includes untrusted fork PR worktrees, then runs the trusted proof finisher with CI secrets in its environment.

## Findings

### High: forged PR worktree sessions can execute local SSH ProxyCommand during secret-bearing cleanup

- Kind: security
- File: `.github/workflows/mantis-telegram-desktop-proof.yml`
- Line: 497
- Evidence: the workflow prompt places the untrusted fork candidate worktree under `.artifacts/qa-e2e/mantis/telegram-desktop-proof-worktrees/candidate`, while the new cleanup scans all `.artifacts/qa-e2e` paths matching `*/telegram-user-crabbox/*/session.json` and runs trusted `finish --session "$session_file"` with `OPENCLAW_QA_CONVEX_SECRET_CI` in the environment. `finish` trusts `session.crabbox.inspect.sshUser` and `host`, builds `target: \`${inspect.sshUser}@${inspect.host}\`` in `scripts/e2e/telegram-user-crabbox-proof.ts:1062`, and passes it to `ssh` before the remote command at `scripts/e2e/telegram-user-crabbox-proof.ts:1116`. A fork PR can force-add ignored `.artifacts/.../telegram-user-crabbox/.../session.json` and `.session/lease.json` into its tree, causing that forged file to be discovered after checkout.
- Impact: a forged `sshUser` beginning with `-oProxyCommand=...` is parsed by OpenSSH as a local option, not as a hostname. I verified this with `ssh -G`, which reported the injected `proxycommand`. That gives a malicious fork PR a plausible local code-execution path inside the cleanup process, with the Convex CI secret and site URL available for exfiltration.
- Suggested fix: do not recursively process session files from PR-controlled worktrees. Track cleanup targets in a trusted, workflow-owned manifest or output root, validate every session path and JSON field against that manifest, constrain `localRoot`/`outputDir`, reject SSH fields that start with `-` or contain unsafe characters, and pass `--` before SSH/SCP targets where supported.
- Confidence: high

### Medium: cleanup skips sessions once credential release has removed `lease.json`

- Kind: reliability
- File: `.github/workflows/mantis-telegram-desktop-proof.yml`
- Line: 499
- Evidence: the cleanup derives `.session/lease.json` from each `session.json` and skips `finish` when that lease file is missing. But `finishSession` already tolerates a missing credential lease via `releaseCredential` returning early at `scripts/e2e/telegram-user-crabbox-proof.ts:1375`, and still has independent cleanup duties such as killing local SUT PIDs and stopping a created Crabbox lease at `scripts/e2e/telegram-user-crabbox-proof.ts:1974`.
- Impact: if an agent or earlier `finish` is interrupted after releasing the credential lease but before stopping the Crabbox lease or removing `.session`, the workflow now ignores the remaining `session.json`, leaving the desktop lease/session cleanup incomplete.
- Suggested fix: for trusted sessions, run `finish` based on the session file itself and let credential release no-op when the lease file is already gone; keep the separate `lease.json` sweep only as a fallback.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The security path follows from the workflow’s recursive `find`, the prompt’s untrusted worktree location, and the proof script’s trusted use of session JSON for SSH. A local `ssh -G` probe confirmed that a target argument beginning with `-oProxyCommand=...` is parsed as `proxycommand`.
- Is this the best way to solve the issue? yes: the cleanup boundary must stop trusting files discovered inside untrusted PR checkouts. SSH argument hardening is useful defense-in-depth but does not address forged `localRoot`, `outputDir`, and session lifecycle fields.
