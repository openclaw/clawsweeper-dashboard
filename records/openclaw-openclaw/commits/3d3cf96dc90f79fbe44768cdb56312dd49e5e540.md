# Commit 3d3cf96

## Summary

Found one low-severity diagnostics bug: apply-mode JSON can report that task maintenance reconciled stale running tasks while omitting the new per-task diagnostic rows for those same tasks.

## Findings

### Low: `--apply --json` omits diagnostics for tasks reconciled by the apply pass

- Kind: bug
- File: `src/commands/tasks.ts`
- Line: 670
- Evidence: `tasksMaintenanceCommand` runs `runTaskRegistryMaintenance()` first when `opts.apply` is true, then calls `getTaskRegistryMaintenanceDiagnostics()`. The diagnostics helper only includes records still marked `running` (`src/tasks/task-registry.maintenance.ts:1021`), so any stale task just marked `lost` by the apply pass is filtered out. A focused smoke check with a stale CLI task produced `maintenance.tasks.reconciled: 1`, `diagnostics.staleRunningTasks: []`, and final task status `lost`.
- Impact: Operators running `openclaw tasks maintenance --apply --json` do not get the new explanation for reconciled stale-running tasks, even though the changelog and PR body say retained and reconcile candidates should be explained. Preview mode still shows the diagnostic; apply mode loses it exactly when the action is taken.
- Suggested fix: Capture the JSON diagnostic snapshot before mutating task records in apply mode, while still before session-registry pruning, or have `runTaskRegistryMaintenance()` return the decision trace it applied so diagnostics describe the same pass.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes; create a stale running CLI task with no active run context, run `tasksMaintenanceCommand({ json: true, apply: true }, runtime)`, and inspect the JSON output.
- Is this the best way to solve the issue? yes; the diagnostic needs to be taken from the pre-apply task state or carried out of the apply pass before status mutation hides the candidate.
