# Commit d94e7f5

## Summary

Found one medium-confidence privacy regression in the Discord progress draft path. The changed Discord code displays reasoning stream payload text whenever a draft stream exists, but the Codex app-server harness can emit those callbacks without checking that the session requested `/reasoning stream`.

## Findings

### Medium: Discord progress drafts can expose Codex reasoning when reasoning visibility is off

- Kind: privacy
- File: `extensions/discord/src/monitor/message-handler.process.ts`
- Line: 663
- Evidence: Discord now always passes reasoning payload text into `draftPreview.pushReasoningProgress` when a draft stream exists (`extensions/discord/src/monitor/message-handler.process.ts:663`). `pushReasoningProgress` then adds the normalized text to progress draft lines and renders it in progress mode (`extensions/discord/src/monitor/message-handler.draft-preview.ts:210`). The default resolved reasoning visibility is `off` (`src/auto-reply/reply/get-reply-directives.ts:455`), and the normal PI subscriber gates streamed reasoning on `reasoningMode === "stream"` (`src/agents/pi-embedded-subscribe.ts:136`). The Codex app-server projector does not have that gate; it calls `this.params.onReasoningStream?.({ text: delta })` for every reasoning delta (`extensions/codex/src/app-server/event-projector.ts:331`).
- Impact: A Discord user with `channels.discord.streaming.mode="progress"` can see Codex reasoning text in the editable progress draft even when the session has not opted into `/reasoning stream`. That weakens the reasoning visibility contract and can expose internal planning/thinking in Discord.
- Suggested fix: Gate user-visible Discord reasoning progress on the resolved session/directive reasoning level being `stream`, or move the contract lower so Codex app-server only invokes `onReasoningStream` when `params.reasoningLevel === "stream"`. Keep status reactions on a separate non-text activity signal if Discord still needs a thinking indicator.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes at a harness level: run Discord progress mode with a Codex app-server reasoning notification (`item/reasoning/textDelta`) while the session has no `/reasoning stream` override; the projector calls `onReasoningStream`, and Discord renders the payload into the progress draft.
- Is this the best way to solve the issue? Unclear. The safest direction is to preserve the existing visibility contract by gating user-visible reasoning at the producer or Discord draft layer; exact placement depends on whether maintainers want `onReasoningStream` to mean “visible reasoning” or “reasoning activity.”
