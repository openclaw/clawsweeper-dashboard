## Findings

### Medium: ClickClack inbound ignores `allowFrom` and authorizes every text command

- Kind: security
- File: `extensions/clickclack/src/inbound.ts`
- Line: 93
- Evidence: `resolveClickClackAccount` preserves account `allowFrom` and defaults it at `extensions/clickclack/src/accounts.ts:118`, and the channel exposes it via `resolveAllowFrom` at `extensions/clickclack/src/channel.ts:88`. But `handleClickClackInbound` routes every human ClickClack message directly into model or agent dispatch without checking `params.account.allowFrom`; model mode replies at `extensions/clickclack/src/inbound.ts:99`, and agent mode sets `CommandAuthorized: true` unconditionally at `extensions/clickclack/src/inbound.ts:153`.
- Impact: An operator who configures `channels.clickclack.allowFrom` or per-account `allowFrom` to restrict senders still allows any non-bot ClickClack user seen by the bot token to trigger model replies or agent runs. In agent mode, text control commands are also treated as authorized before shared command gating sees them.
- Suggested fix: Gate inbound before `resolveAccountAgentRoute` for both model and agent modes. Match `author_id` and any stable ClickClack user aliases against `account.allowFrom`, drop or log denied senders, and compute `CommandAuthorized` through the shared command authorization runtime instead of hardcoding `true`.
- Confidence: high

### Medium: Failed websocket message handling leaves the old socket alive during restart

- Kind: reliability
- File: `extensions/clickclack/src/gateway.ts`
- Line: 123
- Evidence: The websocket wait promise rejects from `JSON.parse`, `processEvent`, or `socket.on("error")` via `reject` at `extensions/clickclack/src/gateway.ts:129` and `extensions/clickclack/src/gateway.ts:146`, but cleanup only happens in the `"close"` handler at `extensions/clickclack/src/gateway.ts:142`. If a message handler rejects while the socket remains open, `startClickClackGatewayAccount` exits, the gateway supervisor restarts the account, and the original socket is not closed or detached.
- Impact: A transient handler failure can leave a live old websocket processing future ClickClack events while the restarted account opens a second connection, causing duplicate inbound handling, duplicate replies, and leaked listeners.
- Suggested fix: Wrap the websocket promise in `try/finally` or centralize `finish(error?)` so every reject path removes listeners and closes or terminates the socket. Prefer catching/logging per-message failures if the connection should remain healthy after one bad event.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. For the access-control issue, configure an account with `allowFrom: ["usr_allowed"]`, send an inbound message from `usr_intruder`, and observe `handleClickClackInbound` still calls `runtime.llm.complete` or `runtime.channel.turn.runPrepared`. For the socket issue, make `processEvent` reject while the websocket stays open and observe the old listener remains after the account supervisor restarts.
- Is this the best way to solve the issue? Yes. Both fixes belong inside the ClickClack plugin because the channel owns sender identity mapping, inbound access policy, and its websocket lifecycle.
