# Commit d4b4660

## Summary

Found one concrete regression in the new runtime-only gateway token path: the gateway can start with a generated token that no separate client process can discover.

## Findings

### Medium: Runtime-generated gateway tokens can lock clients out

- Kind: regression
- File: `src/gateway/startup-auth.ts`
- Line: 189
- Evidence: `ensureGatewayStartupAuth` still generates a random token when token auth has no configured token, but now always returns `persistedGeneratedToken: false` at `src/gateway/startup-auth.ts:213`. The CLI start path explicitly allows this unconfigured-token bootstrap state via `canBootstrapToken` at `src/cli/gateway-cli/run.ts:721`. Normal client bootstrap then resolves credentials only from explicit auth/config/env at `src/gateway/client-bootstrap.ts:35`, and token auth rejects missing client credentials at `src/gateway/auth.ts:352`.
- Impact: after `openclaw gateway` starts from a config with token auth but no token, the server accepts only the in-memory generated token, while a later `openclaw ...` client process reading the same unchanged config has no token to send. Before this commit, startup persisted the generated token, so subsequent clients could authenticate.
- Suggested fix: for non-Nix installs, keep persisting generated startup tokens or write them to a durable local credential channel that client bootstrap reads. For Nix mode, fail startup with the existing Nix guidance when token auth lacks a configured/env token instead of generating an undiscoverable runtime token.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Focused proof with `pnpm exec tsx` showed `ensureGatewayStartupAuth({ cfg: { gateway: { auth: { mode: "token" } } }, env: {} })` generated a server token, while `resolveGatewayClientBootstrap` on the unchanged config returned `clientToken: null`.
- Is this the best way to solve the issue? yes for the fix direction. The generated secret must either be discoverable by future clients or startup should refuse the configuration; an in-memory-only shared secret breaks the auth contract for separate processes.
