# Commit 1725eebe

## Summary

One low-severity compatibility bug found in the JSON-to-SSE fallback path. The main minified JSON fallback case from #77870 is covered and passed focused tests, but valid pretty-printed JSON fallback bodies still fail in the OpenAI SDK SSE parser.

## Findings

### Low: Pretty-printed JSON fallback bodies are emitted as invalid single-line SSE data

- Kind: compatibility
- File: `src/agents/provider-transport-fetch.ts`
- Line: 77
- Evidence: `sanitizeOpenAISdkSseResponse` buffers a JSON response and emits it as ``data: ${data}\n\n``. If the JSON body contains literal newlines, only the first line is treated as SSE `data`; subsequent JSON lines are ignored by `Stream.fromSSEResponse` because they do not start with `data:`. A focused SDK check using the same synthesized frame shape failed with `Could not parse message into JSON: {`.
- Impact: OpenAI-compatible providers or local proxies that return pretty-printed `application/json` chat-completion bodies despite `stream: true` still error instead of using the new fallback. Most hosted providers minify responses, so this is low severity, but pretty-printed JSON is valid JSON and common in custom/self-hosted compatibility layers.
- Suggested fix: When synthesizing SSE, either parse and re-serialize the JSON body to compact JSON before enqueueing, or prefix every physical JSON line with `data:` so SSE decoding reconstructs the full payload.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Construct an SSE frame using the committed shape, ``data: ${JSON.stringify(chatCompletion, null, 2)}\n\n``, then parse it with `openai@6.37.0` `Stream.fromSSEResponse`; it fails before yielding the completion.
- Is this the best way to solve the issue? yes. The fix belongs in the generic JSON-to-SSE synthesis, not in provider-specific code, because the bug is in SSE framing.
