# Commit 1725eebe

## Summary

Found one concrete fallback bug. The new JSON-to-SSE synthesis works for compact JSON, but it emits multiline JSON as one `data:` field. The OpenAI SDK only joins lines that are individually prefixed with `data:`, so a valid pretty-printed `application/json` chat-completion body can still fail the streaming fallback path with a JSON parse error.

## Findings

### Medium: Pretty-printed JSON fallback bodies are framed as invalid SSE data

- Kind: bug
- File: `src/agents/provider-transport-fetch.ts`
- Line: 77
- Evidence: The JSON fallback path trims the full response body and enqueues it as ``data: ${data}\n\n``. If `data` contains physical newlines, only the first line is an SSE `data:` field; the remaining JSON lines are parsed as non-data SSE fields and ignored by OpenAI SDK 6.37.0. The SDK then calls `JSON.parse(sse.data)` on only the first line (`"{"`) and throws. The focused probe reproduced this exactly: `single-data-line: SyntaxError: Expected property name or '}' in JSON at position 1`, while the same JSON succeeded when each physical line was prefixed with `data:`.
- Impact: Any OpenAI-compatible provider that returns valid formatted JSON to a `stream: true` request can still fail the agent turn instead of producing the intended fallback answer. This is the same network-boundary path introduced to recover from JSON chat-completion fallbacks, so the fix is incomplete for a valid `application/json` response shape.
- Suggested fix: When synthesizing SSE, either prefix every physical JSON line as its own `data:` line before the blank event delimiter, or parse and re-stringify the JSON into one compact line before writing the synthetic SSE frame. Add a regression test with pretty-printed JSON.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A direct OpenAI SDK parser probe using the same synthetic framing pattern from `src/agents/provider-transport-fetch.ts:77` fails on pretty-printed JSON and succeeds when each JSON line is `data:`-prefixed.
- Is this the best way to solve the issue? yes. The failure is in synthetic SSE framing, not in the downstream chat-completion parser; fixing the frame construction preserves the generic fallback and does not require provider-specific behavior.
