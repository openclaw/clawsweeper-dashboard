# Commit e6825fc

## Summary

Found one remaining regression in current `main`: explicit wildcard tool allowlists stop exposing plugin tools.

## Findings

### Medium: `tools.allow: ["*"]` loads plugin owners but filters out their tools

- Kind: regression
- File: `src/plugins/tools.ts`
- Line: 215
- Evidence: `manifestToolContractMatchesAllowlist` treats `"*"` as a match at `src/plugins/tools.ts:229`, so wildcard policy scopes the owning plugin runtimes in. But the final registry-entry filter at `src/plugins/tools.ts:421` calls `pluginToolNamesMatchAllowlist`, which delegates non-empty allowlists to `isOptionalToolEntryPotentiallyAllowed`; that helper does not recognize `"*"`. Required named plugin tools are therefore skipped whenever the explicit allowlist is `["*"]`. Optional tools also remain filtered by `isOptionalToolAllowed`, which likewise does not recognize `"*"`.
- Impact: documented wildcard tool policy says `tools.allow` supports `*` wildcards, and `plugins.allow` docs explicitly discuss `tools.allow` containing `"*"`. A user who enables a plugin and sets `tools.allow: ["*"]` expects all callable tools, but plugin tools like `browser`, `tavily_search`, or `file_fetch` are omitted from agent runs.
- Suggested fix: treat `allowlist.has("*")` as allowed in both `pluginToolNamesMatchAllowlist` / `isOptionalToolEntryPotentiallyAllowed` and `isOptionalToolAllowed`, and add a regression test for `toolAllowlist: ["*"]` covering at least one required and one optional plugin tool.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: in `src/plugins/tools.optional.test.ts`, a registry entry with `optional: false`, `names: ["declared_tool"]`, and manifest `contracts.tools: ["declared_tool"]` will currently resolve with no tools when `toolAllowlist: ["*"]`, even though the manifest scope admits the plugin.
- Is this the best way to solve the issue? yes: wildcard handling belongs in the same allowlist predicate used for per-entry and optional returned-tool filtering, so manifest scoping and registry filtering share the same policy semantics.
