## Findings

### Low: Embedded release notes do not escape angle-bracket code spans

- Kind: bug
- File: `appcast.xml`
- Line: 23
- Evidence: The new 2026.5.2 `<description><![CDATA[...]]>` HTML includes code spans such as `<code>--wait <duration></code>` and later `<code>accessGroup:<name></code>`, `<code><@U...></code>`, and `<code><media:image></code>`. Because Sparkle embedded descriptions are HTML, an HTML parser treats those placeholder fragments as tags, not literal text. A focused `parse5` check serialized the first example as `--wait <duration></duration></code>...`, and its text content becomes `--wait ` instead of `--wait <duration>`. Sparkle documents embedded `<description>` release notes as unescaped HTML in CDATA, so code text still needs HTML escaping inside the HTML fragment. 
- Impact: Users viewing the macOS update dialog can see truncated or malformed release-note snippets for the new 2026.5.2 item. The updater still installs correctly; this is a user-facing notes rendering issue, not an update delivery failure.
- Suggested fix: Escape Markdown code-span contents when generating appcast HTML, especially `<`, `>`, and `&`, then regenerate `appcast.xml`. The source generator is `scripts/changelog-to-html.sh`, whose `sed 's/`\([^`]*\)`/<code>\1<\/code>/g'` path currently inserts raw code contents.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Parse the new 2026.5.2 description as HTML with `parse5`; the literal `<duration>` placeholder is consumed as an HTML element instead of displayed as code text.
- Is this the best way to solve the issue? yes. Escaping at `scripts/changelog-to-html.sh` fixes this release and prevents future generated appcast descriptions from malformed inline-code HTML.
