# Commit 8142e67

## Summary

Found one low-severity startup-planner policy gap. Normal Gateway startup has an outer `plugins.enabled: false` guard, but the exported planner and effective-plugin paths can still select configured speech providers when plugins are globally disabled.

## Findings

### Low: Configured speech providers bypass the global plugin-disable check in the startup planner

- Kind: bug
- File: `src/plugins/gateway-startup-plugin-ids.ts`
- Line: 181
- Evidence: `canStartConfiguredSpeechProviderPlugin` only checks speech-provider ownership, deny lists, and per-plugin `enabled: false`; unlike `canStartConfiguredRootPlugin` and `canStartConfiguredChannelPlugin`, it does not check `pluginsConfig.enabled` or the activation-source enabled state. A focused planner probe returned `["microsoft"]` for config with `plugins.enabled: false` and `messages.tts.provider: "microsoft"`. `resolveEffectivePluginIds` returned the same.
- Impact: Direct users of `resolveGatewayStartupPluginIds` / `loadGatewayStartupPluginPlan`, and effective-only diagnostics such as `plugins doctor`, can treat a speech plugin as effective even when the operator globally disabled plugins. The main Gateway startup wrapper currently masks this, so severity is low, but the exported planner now has a misleading and potentially load-triggering result.
- Suggested fix: Add a global enabled check to `canStartConfiguredSpeechProviderPlugin`, mirroring the root/channel startup checks, and add regression coverage for `plugins.enabled: false` plus configured `messages.tts.provider`.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the planner/effective-plugin path: call the startup/effective resolver with `plugins.enabled: false` and `messages.tts.provider: "microsoft"`; it returns `microsoft`.
- Is this the best way to solve the issue? Yes: the speech-provider startup path should honor the same global plugin disable contract as adjacent startup paths. The outer Gateway guard should remain, but the reusable planner should be internally consistent.
