# Commit 089a306

## Summary

Found one ownership cleanup regression in the startup-timeout path. The commit protects newer shared clients from older lanes, but if startup times out before the default shared factory returns a client, cleanup now becomes a no-op and leaves the pending shared app-server client/promise installed.

## Findings

### Medium: Startup timeout can leave the shared Codex app-server permanently stuck

- Kind: reliability
- File: `extensions/codex/src/app-server/run-attempt.ts`
- Line: 573
- Evidence: `startupClientForCleanup` is assigned only after `clientFactory(...)` resolves at `extensions/codex/src/app-server/run-attempt.ts:522`. The outer startup catch now calls `clearSharedCodexAppServerClientIfCurrent(startupClientForCleanup)` at `extensions/codex/src/app-server/run-attempt.ts:573`, which returns immediately for `undefined` at `extensions/codex/src/app-server/shared-client.ts:140`. The default factory calls `getSharedCodexAppServerClient` without a timeout at `extensions/codex/src/app-server/client-factory.ts:15`, while `getSharedCodexAppServerClient` stores `state.client` and `state.promise` before awaiting `client.initialize()` at `extensions/codex/src/app-server/shared-client.ts:52`. If `withCodexStartupTimeout` fires while initialize is still pending, the run-attempt catch cannot identify the client, so it does not clear or close the shared singleton.
- Impact: A hung Codex app-server initialize can leave the shared client promise in place after the user-visible startup timeout. Later Codex lanes with the same start key will reuse the same stuck promise instead of spawning a fresh app-server, so Codex can remain unusable until the process exits, a different key forces cleanup, or the gateway restarts.
- Suggested fix: Preserve ownership-aware cleanup for known clients, but also make the pending startup ownership trackable before the factory resolves. For example, pass the startup timeout into the shared-client path, expose a token/key/promise-scoped clear helper, or have the factory return/record the started shared client before `initialize()` completes so timeout cleanup can clear only the still-current pending startup.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. Use the default shared Codex app-server factory with an app-server process that starts but never replies to `initialize`, then run a Codex app-server attempt with a short startup timeout. The first attempt rejects with `codex app-server startup timed out`; a second same-key attempt reuses the uncleared shared `state.promise` instead of starting fresh.
- Is this the best way to solve the issue? Unclear. The intended ownership-aware direction is right, but the fix needs a pending-startup ownership token, not only a resolved client object.
