# Commit 2c8f78e

## Summary

Found one concrete routing-safety regression in the new fresh-session final-delivery retry path. The focused delivery tests pass, but the new test only covers same-session refresh and does not guard against reusing a newer logical session’s route.

## Findings

### Medium: Fresh delivery retry can send a run’s final reply through a different logical session

- Kind: privacy
- File: `src/agents/command/delivery.ts`
- Line: 470
- Evidence: The new retry branch accepts any `freshSessionEntry` returned for the same key when it is a different object, then replaces `deliveryRouting` if the fresh route is valid. There is no check that `freshSessionEntry.sessionId` matches the run’s original `sessionEntry.sessionId`. The caller reloads the current disk entry by `sessionKey` at `src/agents/agent-command.ts:1531` and returns `freshStore[sessionKey]` at `src/agents/agent-command.ts:1535`. Same-key session rotation is a supported state: `src/agents/command/session.ts:328` creates a new `sessionId` for a stale key, and `src/config/sessions/session-file.ts:42` explicitly persists a changed `sessionId` under the same `sessionKey`.
- Impact: If a run reaches final delivery with stale/no-target routing while the same `sessionKey` has been rotated or rebound to a newer entry, this change can turn the old run’s previous no-send failure into a platform send using the newer entry’s `deliveryContext` channel/to/account/thread. That can expose the old run’s final content in the wrong channel/thread and can also clear pending final-delivery state on the newer entry after a successful send.
- Suggested fix: Only accept a refreshed entry when it is proven to be the same logical session, for example `freshSessionEntry.sessionId === sessionEntry?.sessionId` or another explicit run/session invariant. Do not update the in-memory `sessionStore[sessionKey]` until after that guard. Add a regression where the resolver returns a valid route with a different `sessionId` and delivery is not attempted.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes, at unit level: use the new `deliverAgentCommandResult` refresh test shape, but make the initial `sessionEntry.sessionId` `"old-session"` and the resolver return `"new-session"` with a valid `deliveryContext`; current code still sends to the fresh target because it checks object identity only.
- Is this the best way to solve the issue? Yes: the intended recovery is same-session stale routing, so constraining refresh acceptance to the same logical session preserves the durable send path without allowing cross-session route reuse.
