# Commit b9fbc57

## Summary

Found one actionable approval-binding regression in the new shared POSIX option advancement path. The fix covers Bash combined `-o` forms, but it now misses valid zsh invocations where the `-o` option name is attached in the same token.

## Findings

### Medium: zsh attached `-o` option names skip the script operand binding

- Kind: security
- File: `src/node-host/invoke-system-run-plan.ts`
- Line: 628
- Evidence: `resolvePosixShellScriptOperandIndex` now applies `advancePosixInlineOptionScan(token)` to all POSIX shell option tokens. That helper counts any `o`/`O` inside a short-option cluster as consuming a separate following value. For `zsh -xoshwordsplit ./run.zsh`, zsh treats `shwordsplit` as an attached option name and executes `./run.zsh`; the zsh invocation manual explicitly says `-xoshwordsplit` is equivalent to `-x -o shwordsplit`, and that the following non-option argument is the script file. ([zsh.sourceforge.io](https://zsh.sourceforge.io/Doc/Release/Invocation.html)) The current code skips `./run.zsh` as if it were the separate `-o` value, so `resolveMutableFileOperandSnapshotSync({ argv: ["zsh", "-xoshwordsplit", "./run.zsh"], ... })` returns `{ ok: true, snapshot: null }`. The same probe binds correctly for `["zsh", "-xo", "shwordsplit", "./run.zsh"]`.
- Impact: Approved `system.run` zsh script executions using attached `-o` names are no longer hashed and revalidated. A mutable script can change between approval and execution without triggering the script-drift denial that this approval hardening is meant to enforce.
- Suggested fix: Make the combined option advancement shell-aware. For zsh, treat `-...oNAME` / `+...oNAME` as consuming the option name from the same token when `o` is not the final character, and only skip the next argv token for `-...o` / `+...o`. Add regression coverage for `zsh -xoshwordsplit ./run.zsh` and the matching plus form if supported.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Run a direct probe of `resolveMutableFileOperandSnapshotSync` with `["zsh", "-xoshwordsplit", "./run.zsh"]`; it returns `snapshot: null` on this commit, while the separated forms return a snapshot.
- Is this the best way to solve the issue? yes for the fix direction. The Bash-centered helper is still useful, but zsh needs its documented attached `-o` syntax modeled instead of blindly skipping the following operand.
