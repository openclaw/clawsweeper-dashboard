# Commit 6dbd5bd

## Summary

Found two policy false negatives. Both let `policy check` emit a clean attestation even though current OpenClaw config contains settings that the authored policy appears to deny.

## Findings

### Medium: Model provider policy misses channel and other non-agent model refs

- Kind: bug
- File: `extensions/policy/src/policy-state.ts`
- Line: 217
- Evidence: `scanPolicyModelRefs` only scans `cfg.agents`, then `agents.defaults.models` / `agents.list[].models`. The existing model-ref contract in `src/config/model-refs.ts` also enumerates `channels.modelByChannel`, hooks model refs, `messages.tts.summaryModel`, and `channels.discord.voice.model`. A temp-workspace CLI smoke with `channels.modelByChannel.slack.C123 = "anthropic/claude-opus-4-6"` and policy `{ "models": { "providers": { "allow": ["openai"] } } }` returned `ok: true`, with evidence containing only `agents.defaults.model: "openai/gpt-5.5"` and no finding for the Anthropic channel override.
- Impact: Operators can record a clean `expectedAttestationHash` while channel routing still pins traffic to a denied or unapproved provider.
- Suggested fix: Use one canonical configured-model-ref collector for policy evidence. Because plugin production code should not deep-import `src/config/model-refs.ts`, promote a narrow SDK/public helper or mirror the full enumerated contract with tests for `channels.modelByChannel`, hooks, message TTS, Discord voice, heartbeat/subagent/compaction refs, and agent model maps.
- Confidence: high

### Medium: Private-network policy ignores model provider request opt-ins

- Kind: security
- File: `extensions/policy/src/policy-state.ts`
- Line: 230
- Evidence: `scanPolicyNetwork` checks browser SSRF and `tools.web.fetch.ssrfPolicy` only. `models.providers.*.request.allowPrivateNetwork` is a supported config key (`src/config/types.provider-request.ts`) and is documented as allowing model-provider HTTP requests to private/CGNAT-like ranges. A temp-workspace CLI smoke with `models.providers.local.request.allowPrivateNetwork: true` and policy `{ "network": { "privateNetwork": { "allow": false } } }` returned `ok: true`, with `network: []` and no `policy/network-private-access-enabled` finding.
- Impact: A workspace can explicitly allow private-network model-provider requests while the policy attestation still claims private-network access is denied.
- Suggested fix: Extend network evidence to enumerate `models.providers.<id>.request.allowPrivateNetwork === true` with an `oc://openclaw.config/models/providers/<id>/request/allowPrivateNetwork` source, and add allow/deny tests plus attestation evidence coverage. Consider auditing other documented private-network opt-ins before calling the posture complete.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Both findings were reproduced with temp-workspace `pnpm openclaw policy check --json` runs that returned `ok: true` despite denied/unapproved config.
- Is this the best way to solve the issue? yes directionally. The policy evidence layer should reuse or exactly match the config contract it claims to attest; otherwise clean attestations are incomplete.
