# Commit a04e40e6fa

## Summary

Found one regression in the Docker prune command. The commit follows pnpm v11 migration guidance in general, but `pnpm prune` on `pnpm@11.0.8` does not honor `pnpm_config_frozen_lockfile=false` for disabling its CI frozen-lockfile path.

## Findings

### Medium: Docker prune no longer disables frozen lockfile for pnpm prune

- Kind: regression
- File: `Dockerfile`
- Line: 127
- Evidence: The changed command is `CI=true pnpm_config_frozen_lockfile=false pnpm prune --prod`. In an isolated pnpm 11.0.8 fixture with an intentionally outdated lockfile, this exact form failed with `ERR_PNPM_OUTDATED_LOCKFILE`. The previous `CI=true NPM_CONFIG_FROZEN_LOCKFILE=false pnpm prune --prod` form succeeded, and `CI=true pnpm prune --prod --config.frozen-lockfile=false` also succeeded. `PNPM_CONFIG_FROZEN_LOCKFILE=false` failed as well. Local pnpm 11.0.8 source also shows `prune` exposes only `dev`, `optional`, `production`, and `ignore-scripts` as command rc options before delegating to the install handler, which explains why the normal `pnpm_config_*` setting is filtered out for this command.
- Impact: The Dockerfile comment says prune must be allowed to run against the generated runtime workspace subset. With the new env var, CI frozen-lockfile behavior is still active for `pnpm prune`, so Docker builds can fail at the runtime-assets prune step whenever pnpm considers the generated workspace or lockfile state to need a non-frozen install/prune. The adjacent test now asserts the string but does not prove the command behavior.
- Suggested fix: Use an explicit command config that `prune` accepts in practice, for example `CI=true pnpm prune --prod --config.frozen-lockfile=false`, and update `src/dockerfile.test.ts` to check that exact command. If keeping an env var, add a focused fixture/smoke check proving `pnpm prune` honors it under the pinned pnpm version.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. In `/tmp`, create a small pnpm project, install once, add a dependency to `package.json` without updating `pnpm-lock.yaml`, then run `CI=true pnpm_config_frozen_lockfile=false pnpm prune --prod` with pnpm 11.0.8. It fails with `ERR_PNPM_OUTDATED_LOCKFILE`; the old uppercase env and `--config.frozen-lockfile=false` variants succeed.
- Is this the best way to solve the issue? yes. `--no-frozen-lockfile` is not accepted by `pnpm prune`, but `--config.frozen-lockfile=false` was verified against the same pnpm 11.0.8 fixture and avoids relying on prune’s command-specific env filtering.
