# Commit 973e240

## Summary

Found one compatibility regression in the new progress-tool-status scoping logic. Slack legacy progress mode still maps `streamMode: "status_final"` to progress mode, but the changed SDK resolver does not recognize that Slack-specific legacy value, so `streaming.progress.toolProgress: false` is ignored for that still-supported config shape.

## Findings

### Low: Slack legacy progress mode ignores `streaming.progress.toolProgress=false`

- Kind: compatibility
- File: `src/plugin-sdk/channel-streaming.ts`
- Line: 427
- Evidence: The new resolver only reads `config.progress.toolProgress` when `resolveChannelPreviewStreamMode(entry, "partial") === "progress"` (`src/plugin-sdk/channel-streaming.ts:427`). Slack still maps legacy `streamMode: "status_final"` to unified progress mode (`extensions/slack/src/streaming-compat.ts:75`, `extensions/slack/src/streaming-compat.ts:77`, and `extensions/slack/src/streaming-compat.ts:47`), and dispatch passes the raw account config into this SDK helper (`extensions/slack/src/monitor/message-handler/dispatch.ts:876`). A focused probe with `{ streamMode: "status_final", streaming: { progress: { toolProgress: false } } }` returned `slackMode:"progress"` but `previewToolProgress:true`.
- Impact: Users with an unmigrated Slack `streamMode: "status_final"` config who set the new nested `streaming.progress.toolProgress=false` still get compact tool/status lines in Slack progress drafts. The parent implementation honored that key because it read `progress.toolProgress` before `preview.toolProgress`.
- Suggested fix: Either pass the effective resolved preview mode into `resolveChannelStreamingPreviewToolProgress`, or add a Slack wrapper/call-site normalization so Slack legacy `status_final` is treated as progress before applying the progress-only fallback.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Run:
  `pnpm exec tsx -e "import { resolveChannelPreviewStreamMode, resolveChannelStreamingPreviewToolProgress } from './src/plugin-sdk/channel-streaming.ts'; import { resolveSlackStreamingConfig } from './extensions/slack/src/stream-mode.ts'; const cfg = { streamMode: 'status_final', streaming: { progress: { toolProgress: false } } }; console.log(JSON.stringify({ sdkMode: resolveChannelPreviewStreamMode(cfg, 'partial'), slackMode: resolveSlackStreamingConfig(cfg).mode, slackDraftMode: resolveSlackStreamingConfig(cfg).draftMode, previewToolProgress: resolveChannelStreamingPreviewToolProgress(cfg) }));"`
  It prints `{"sdkMode":"partial","slackMode":"progress","slackDraftMode":"status_final","previewToolProgress":true}`.
- Is this the best way to solve the issue? unclear. The safest direction is to avoid making the generic helper infer channel-specific legacy modes by itself; the Slack caller already has the effective mode, so passing/normalizing that mode would keep the SDK helper generic.
