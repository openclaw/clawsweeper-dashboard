## Summary

The commit adds `openclaw cron run --wait`, exact `cron.runs --run-id` filtering, protocol/Swift schema updates, tests, and docs. I found one concrete timeout-bound bug in the new wait loop.

## Findings

### Medium: `--wait-timeout` can be exceeded by a full poll interval

- Kind: reliability
- File: `src/cli/cron-cli/register.cron-simple.ts`
- Line: 77
- Evidence: `waitForCronRunCompletion` checks `Date.now() - startedAt >= timeoutMs`, then unconditionally sleeps `params.pollIntervalMs`. With `--wait-timeout 1s --poll-interval 10s` and no terminal `cron.runs` entry, the command performs the first poll, sees the timeout has not expired, sleeps for 10 seconds, and only then throws. This contradicts the CLI text that `--wait-timeout` is the maximum wait.
- Impact: shutdown/pre-stop automation can block far longer than its configured budget, especially when users choose a larger poll interval to reduce Gateway polling.
- Suggested fix: compute a deadline and sleep only `min(pollIntervalMs, remainingMs)`, then recheck/throw immediately when the remaining time is exhausted.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Mock `cron.run` to return `{ ok: true, enqueued: true, runId }`, mock `cron.runs` to return no entries, then invoke `cron run job-1 --wait --wait-timeout 1s --poll-interval 10s`; the source path sleeps for 10 seconds before timing out.
- Is this the best way to solve the issue? yes. The design of polling by exact `runId` is sound; only the timer needs to clamp each sleep to the remaining timeout.
