# Commit e03fe1e

## Summary

Found two actionable Telegram long-final regressions. The new path reuses the preview even when existing rules require a fresh final after stale or overwritten previews, and it drops Telegram markdown formatting by converting formatted chunks to plain text before delivery.

## Findings

### Medium: Long finals bypass the fresh-final guard for stale or overwritten previews

- Kind: regression
- File: `extensions/telegram/src/lane-delivery-text-deliverer.ts`
- Line: 677
- Evidence: The short-final path checks `shouldUseFreshFinalForLane(lane)` and sends a fresh final when the preview is older than the threshold or was pushed up by a visible non-preview message (`extensions/telegram/src/lane-delivery-text-deliverer.ts:641`). The new long-final branch calls `tryDeliverLongFinalThroughPreview` directly (`extensions/telegram/src/lane-delivery-text-deliverer.ts:677`), and that helper only checks stream existence, buttons, and transient lifecycle (`extensions/telegram/src/lane-delivery-text-deliverer.ts:369`). It never consults `shouldUseFreshFinalForLane`, even though docs still promise fresh finals after intermediate output and after about one minute (`docs/channels/telegram.md:348`, `docs/channels/telegram.md:349`). A focused inline harness with `visibleSinceMs=10000`, `now=70000`, `draftMaxChars=10`, and two split chunks returned `preview-finalized` and edited message `999` instead of taking the fresh-send/clear path.
- Impact: Long Telegram replies can put the first final chunk back in the old preview bubble above tool/error/intermediate output, or keep a stale timestamp for long-running replies. That reintroduces the ordering/timestamp problem the existing fresh-final logic was designed to avoid.
- Suggested fix: Gate `tryDeliverLongFinalThroughPreview` behind the same freshness check. If `shouldUseFreshFinalForLane(lane)` is true, keep the existing fresh-send-and-clear behavior for long finals, with fallback to preview reuse only if the fresh send returns false.
- Confidence: high

### Medium: Long-final preview reuse strips Telegram markdown formatting

- Kind: regression
- File: `extensions/telegram/src/bot-message-dispatch.ts`
- Line: 803
- Evidence: `markdownToTelegramChunks` returns both rendered Telegram HTML and plain source text (`extensions/telegram/src/format.ts:450`). Normal `deliverReplies` sends `chunk.html` with `textMode: "html"` (`extensions/telegram/src/bot/delivery.replies.ts:190`). The new splitter discards `html` and returns only `telegramChunk.text` (`extensions/telegram/src/bot-message-dispatch.ts:803`), then edits the preview with default markdown rendering (`extensions/telegram/src/bot-message-dispatch.ts:872`, `extensions/telegram/src/send.ts:1358`) and sends the remaining plain text via `sendPayload` (`extensions/telegram/src/lane-delivery-text-deliverer.ts:403`). A focused probe showed `markdownToTelegramChunks('**bold** and [link](https://example.com) ...', 120)` returns HTML like `<b>bold</b> and <a href="https://example.com">link</a>` but `.text` as `bold and link ...`.
- Impact: Any long final that takes this new preview-reuse path loses bold/italic/code/link formatting across the preview chunk and remaining chunks. Before this commit, the standard long-send path preserved formatting through `chunk.html`.
- Suggested fix: Preserve structured chunk data through the long-final path. Edit the preview using the rendered HTML with `textMode: "html"` or a dedicated edit helper, and send follow-up chunks through the same HTML-aware delivery path instead of joining plain `.text`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The stale-preview regression was reproduced with a focused inline `createLaneTextDeliverer` harness; the markdown regression was reproduced with a focused `markdownToTelegramChunks` probe showing formatted `html` is discarded.
- Is this the best way to solve the issue? yes for the fix direction: reuse the existing fresh-final predicate before preview reuse, and keep Telegram chunk formatting structured rather than flattening to plain text.
