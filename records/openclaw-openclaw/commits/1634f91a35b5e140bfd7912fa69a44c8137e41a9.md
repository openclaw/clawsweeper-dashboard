# Commit 1634f91

## Summary

Found one medium-severity reliability regression in the new staged Google Meet Twilio join path: if DTMF or intro speech fails after the voice call has already started, the live phone call is left without a Google Meet session stop handler.

## Findings

### Medium: Failed post-start join steps can leave an untracked live Twilio call

- Kind: reliability
- File: `extensions/google-meet/src/voice-call-gateway.ts`
- Line: 119
- Evidence: `joinMeetViaVoiceCallGateway` now calls `voicecall.start` first, then separately calls `voicecall.dtmf` and `voicecall.speak`. If either later request rejects or returns `success: false`, the function throws after a `callId` already exists. The only `finally` cleanup stops the gateway client, not the voice call. The Google Meet runtime only stores the session stop handler after `joinMeetViaVoiceCallGateway` returns successfully, so this failure path never registers `googlemeet leave` cleanup for that call (`extensions/google-meet/src/runtime.ts:460`, `extensions/google-meet/src/runtime.ts:483`).
- Impact: a Twilio phone leg can remain active after a failed Meet join attempt, potentially incurring cost and leaving an operator with no Google Meet session ID to hang it up. Voice Call may eventually time out after answered webhooks, but that is delayed fallback and may not run if webhook state is the reason the post-start command failed.
- Suggested fix: once `voicecall.start` returns a `callId`, wrap the DTMF/speak sequence in a `try`/`catch` that best-effort calls `voicecall.end` before rethrowing, or return a partial tracked session with a stop handler even when the post-start sequence fails.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: mock `voicecall.start` to return `{ callId: "call-1" }`, then make `voicecall.dtmf` or `voicecall.speak` fail; `joinMeetViaVoiceCallGateway` rejects without issuing `voicecall.end`.
- Is this the best way to solve the issue? yes: cleanup belongs next to the staged sequence because Google Meet owns the orchestration and knows a started phone leg must be unwound if later join stages fail.
