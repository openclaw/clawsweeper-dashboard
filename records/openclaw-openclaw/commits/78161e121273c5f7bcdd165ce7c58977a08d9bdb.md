# Commit 78161e1

## Summary

Found one concrete issue: the new `includeInCore: false` filter is applied to bundled text-provider and manifest-contract paths, but not to capability-provider fallback loading. That leaves the externalized Codex plugin visible as a bundled media-understanding provider from a source checkout.

## Findings

### Medium: Externalized Codex still auto-loads through bundled capability fallback

- Kind: regression
- File: `src/plugins/capability-provider-runtime.ts`
- Line: 82
- Evidence: The commit adds `isPackageIncludedInCoreBundle(...)` filtering in `src/plugins/providers.ts:81` and `src/plugins/plugin-registry-contributions.ts:123`, but `resolveBundledCapabilityCompatPluginIds` still accepts every `plugin.origin === "bundled"` manifest with a matching capability contract. `extensions/codex/package.json:29` marks Codex as `"includeInCore": false`, while `extensions/codex/openclaw.plugin.json:6` declares `mediaUnderstandingProviders: ["codex"]`. A focused probe confirms the mismatch: `resolveManifestContractPluginIds({ contract: "mediaUnderstandingProviders", origin: "bundled" })` excludes `codex`, but `resolveBundledCapabilityProviderIds({ key: "mediaUnderstandingProviders" })` still returns `codex`. `timeout 20s node --import tsx -e '...'` for `resolvePluginCapabilityProvider({ key: "mediaUnderstandingProviders", providerId: "codex" })` printed `{"found":true,"id":"codex"}` and then timed out, showing this path loads the externalized Codex plugin and leaves the process alive.
- Impact: Core media/capability discovery can still treat an externalized plugin as a bundled core capability in source-checkout/runtime fallback paths. That can unexpectedly load the Codex app-server plugin and its heavier runtime side effects even though this commit intended bundled provider contracts to align with externalized plugins.
- Suggested fix: Apply the same `includeInCore !== false` guard to bundled capability provider resolution, especially `resolveBundledCapabilityCompatPluginIds` and `resolveBundledCapabilityProviderIds`, and add a contract test that `codex` is absent from bundled media-understanding fallback ids unless installed as a non-bundled plugin.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: run a focused `node --import tsx` probe against `src/plugins/capability-provider-runtime.ts` and compare it with `resolveManifestContractPluginIds` for `mediaUnderstandingProviders`.
- Is this the best way to solve the issue? yes: the commit already establishes the correct predicate; the safer fix is to reuse that policy consistently across all bundled capability-contract fallback paths.
