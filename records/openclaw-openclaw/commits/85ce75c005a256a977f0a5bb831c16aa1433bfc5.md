# Commit 85ce75c005

## Summary

Found one concrete edge case in the macOS PATH fix: installs launched under a version-managed Node can still write that version-manager Node directory into the LaunchAgent PATH via the selected runtime `nodePath`.

## Findings

### Medium: macOS version-manager runtime paths still get prepended to the service PATH

- Kind: bug
- File: `src/commands/daemon-install-helpers.ts`
- Line: 421
- Evidence: `resolvePreferredNodePath` still prefers the current supported `process.execPath` for `runtime: "node"` (`src/daemon/runtime-paths.ts:168`), including fnm/Volta/asdf paths. `buildGatewayInstallPlan` then passes `resolveDaemonNodeBinDir(nodePath)` into `buildServiceEnvironment` as `extraPathDirs` (`src/commands/daemon-install-helpers.ts:413-422`), and `getMinimalServicePathParts` always prepends `extraDirs` before the new canonical macOS system dirs (`src/daemon/service-env.ts:331-360`). A focused smoke check produced `/Users/Nick/.volta/tools/image/node/22.14.0/bin:/usr/local/bin:/usr/bin:/bin:/usr/sbin:/sbin`, and the service audit reported `gateway-path-nonminimal,gateway-runtime-node-version-manager`.
- Impact: the default macOS install path can still create a LaunchAgent whose PATH is not canonical when the installer is run with a supported version-managed Node. That undercuts the commit’s stated fix for #75233 because `node` child-process resolution can still be influenced by Volta/fnm/asdf, and `openclaw gateway status --deep` can immediately flag the generated service as non-minimal.
- Suggested fix: distinguish explicit user-provided `nodePath` from auto-selected `process.execPath`, and on Darwin avoid adding auto-selected version-manager directories to PATH. Alternatively, prefer a supported system/Homebrew Node for daemon installs when available, or only prepend runtime dirs that are classified as system Node paths.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the source-level behavior: run the install planner or `buildServiceEnvironment` on Darwin with a version-manager `nodePath` selected from the current runtime. I did not run a live macOS launchd install in this Linux checkout.
- Is this the best way to solve the issue? Unclear. The safest direction is to keep absolute `ProgramArguments` usable while preventing auto-discovered version-manager bins from becoming child-process PATH precedence. Explicit operator-provided runtime overrides may deserve different treatment.
