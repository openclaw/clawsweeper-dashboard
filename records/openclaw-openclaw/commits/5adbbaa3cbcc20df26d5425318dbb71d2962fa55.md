# Commit 5adbbaa3cb

## Summary

Found one low-severity Windows compatibility regression in the new `.cmd`/`.bat` UI runner path. The DEP0190 fix direction is sound, and focused tests pass, but the shared cmd.exe builder now validates the resolved executable path as if it were a forwarded argument.

## Findings

### Low: UI runner now fails when the resolved pnpm shim path contains valid Windows shell metacharacters

- Kind: compatibility
- File: `scripts/ui.js`
- Line: 77
- Evidence: `resolveSpawnCall()` now passes the resolved runner path `cmd` into `buildCmdExeCommandLine(cmd, args)`. That helper rejects any value containing `[&|<>%\r\n]`, so a valid Windows install path such as `C:\tools\R&D\pnpm.cmd` throws `unsafe Windows cmd.exe argument detected` before spawning. The previous code only validated forwarded args, while quoting the command path separately. Microsoft’s Windows filename rules allow characters except the documented reserved set, which does not include `&` or `%`. ([learn.microsoft.com](https://learn.microsoft.com/en-us/windows/win32/fileio/naming-a-file?utm_source=openai))
- Impact: `pnpm ui:build`, `pnpm ui:dev`, and `pnpm ui:install` can fail on Windows when `PATH` resolves `pnpm.cmd` from a directory containing `&` or `%`, even though the metacharacter is in the trusted executable path rather than user-supplied args.
- Suggested fix: handle the executable path separately from forwarded args. Keep rejecting unsafe forwarded args, but quote the `.cmd`/`.bat` path with a command-path-specific helper, or add a test documenting that such paths intentionally fail closed.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes for the pre-spawn failure: `resolveSpawnCall(String.raw\`C:\tools\R&D\pnpm.cmd\`, ["run", "build"], undefined, { platform: "win32", comSpec: "cmd.exe" })` throws locally before any Windows syscall is involved.
- Is this the best way to solve the issue? unclear. The commit correctly avoids Node’s DEP0190 pattern, which Node documents as a runtime deprecation for `spawn`/`execFile` with `shell: true` plus args. ([nodejs.org](https://nodejs.org/download/release/latest/docs/api/deprecations.html?utm_source=openai)) The safer refinement is to preserve that `shell: false` cmd.exe wrapper while not treating the trusted executable path exactly like untrusted forwarded arguments.
