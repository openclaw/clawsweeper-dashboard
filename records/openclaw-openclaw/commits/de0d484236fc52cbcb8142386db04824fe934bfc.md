# Commit de0d484

## Summary

Found one concrete data-retention edge case in the new preservation predicate: real external channel/group sessions whose key happens to end with `:heartbeat` are still treated as synthetic and can be evicted by stale-age, max-entry, and disk-budget maintenance.

## Findings

### Medium: Real channel sessions ending in `:heartbeat` bypass durable-entry protection

- Kind: data_loss
- File: `src/config/sessions/store-maintenance.ts`
- Line: 239
- Evidence: `isSyntheticSessionMaintenanceKey()` returns true for any parsed rest that `endsWith(":heartbeat")` or contains `:heartbeat:` at `src/config/sessions/store-maintenance.ts:217`, and `isProtectedSessionMaintenanceEntry()` immediately returns false before checking group/channel keys at `src/config/sessions/store-maintenance.ts:239`. The session entry type explicitly distinguishes synthetic isolated heartbeat sessions by `heartbeatIsolatedBaseSessionKey` and notes that real user/session-scoped keys may also end in `:heartbeat` at `src/config/sessions/types.ts:159`. A focused live check showed `isProtectedSessionMaintenanceEntry("agent:main:workspace:channel:heartbeat", { chatType: "channel" })` returns `false`, while `agent:main:workspace:channel:alerts` returns `true`; a prune/cap smoke then removed the `channel:heartbeat` entry.
- Impact: A durable external conversation named or keyed `heartbeat` can still lose its `sessions.json` pointer under exactly the maintenance paths this commit intended to protect. That can drop long-lived channel context for custom/workspace-style channels using human-readable channel IDs.
- Suggested fix: Make heartbeat synthetic classification entry-aware. Prefer treating `:heartbeat` as synthetic only when `entry.heartbeatIsolatedBaseSessionKey` is present, or when the key is not otherwise recognized as an external group/channel/thread. Keep a legacy fallback for old isolated heartbeat entries, but let explicit group/channel keys and `chatType: "group" | "channel" | "thread"` win.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Run a focused predicate/prune smoke against current `main`: `agent:main:workspace:channel:heartbeat` with `chatType: "channel"` is not protected and is removed by both `pruneStaleEntries()` and `capEntryCount()`.
- Is this the best way to solve the issue? unclear. The direction is clear, but the safest exact fix should account for legacy heartbeat-isolated entries without `heartbeatIsolatedBaseSessionKey`.
