# Commit 95343af

## Summary

Found one residual stale-cache path from removing the prepared auth TTL. Current `main` added auth-profile file watching after this commit, but that watcher is still tied to startup config and does not cover agents added by hot reload.

## Findings

### Medium: Auth added for a hot-reloaded agent can stay invisible indefinitely

- Kind: regression
- File: `src/agents/model-provider-auth.ts`
- Line: 120
- Evidence: `hasAuthForModelProvider` now returns the prepared per-agent answer whenever config fingerprint, agent ID, and workspace match, with no TTL or freshness fallback. Current `main` tries to compensate with `watchAuthProfilesForChanges`, but that watcher is constructed from `params.cfgAtStart` in `src/gateway/server-startup-post-attach.ts:1046`, and `agents.list` is a hot-reload path in `src/gateway/config-reload-plan.ts:100`. Hot reload rewarms the auth map for `nextConfig` in `src/gateway/server-reload-handlers.ts:503`, but it does not recreate the auth-profile watcher except in the unrelated Gmail watcher path. A `/models` request for the newly added agent uses that warmed per-agent map through `src/auto-reply/reply/commands-models.ts:248`, so if the new agent had no auth at hot reload time and its `auth-profiles.json` is later created externally, no watcher invalidates the prepared false answer.
- Impact: A user can add an agent through config hot reload, then add credentials for that agent with an external login/tool or direct auth-profile write, and `/models` can keep hiding auth-backed models for that agent until another config reload, restart, auth failure hook, or manual cache clear. Before this commit, the removed 10-second TTL eventually forced a recompute.
- Suggested fix: Make the auth-profile watcher lifecycle follow the current runtime config on `agents.list` hot reload, or have the watcher read/update paths from the current config. A bounded stale fallback for prepared false answers would also cover missed filesystem events.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes: start the gateway with one agent, hot-reload `agents.list` to add a second agent, let `/models` or auth prewarm cache a no-auth answer for the second agent, then create that second agent’s `auth-profiles.json`; `/models` should continue using the prepared false answer because the watcher was never pointed at the new agent path.
- Is this the best way to solve the issue? Unclear. Updating/restarting the watcher on agent-list reload is the cleanest ownership fix; reintroducing a TTL is a broader fallback but weakens the intended prewarm behavior.
