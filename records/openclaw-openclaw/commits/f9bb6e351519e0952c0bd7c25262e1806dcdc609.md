# Commit f9bb6e3

## Summary

The embedded PI path now uses `discord-voice` as the tool-policy key and hides `tts`, but the CLI runtime path still exposes the `tts` tool through bundle MCP. That leaves a concrete reproduction path for the original Discord voice failure when an agent is configured to run through `claude-cli`/`codex-cli` with bundle MCP enabled.

## Findings

### Medium: CLI-backed Discord voice turns still expose the `tts` tool

- Kind: bug
- File: `src/agents/command/attempt-execution.ts`
- Line: 484
- Evidence: The commit forwards `messageProvider: params.opts.messageProvider ?? params.messageChannel` into `runCliAgent`, and the new test only asserts that the CLI runner receives `discord-voice`. Downstream, however, CLI bundle MCP builds `OPENCLAW_MCP_MESSAGE_CHANNEL` from `params.messageChannel ?? params.messageProvider`, so Discord voice turns send `discord`, not `discord-voice`, into the loopback MCP request context (`src/agents/cli-runner/prepare.ts:205`). The MCP tool resolver then creates OpenClaw tools without applying `filterToolsByMessageProvider`, and `createOpenClawTools` always includes `createTtsTool` (`src/gateway/tool-resolution.ts:83`, `src/agents/openclaw-tools.ts:258`). Issue #61536’s failure mode is specifically “agent calls `tts`, then final `NO_REPLY`, Discord voice has no text to play.”
- Impact: Users whose Discord voice sessions route through a CLI backend can still hit the original no-spoken-reply behavior even after this fix, because the tool remains available in the CLI harness and its description explicitly tells the model to reply with `NO_REPLY` after a successful TTS call.
- Suggested fix: Carry a separate tool-policy provider into CLI bundle MCP, or apply the same `discord-voice` deny policy inside `resolveGatewayScopedTools`/loopback tool resolution. Keep `messageChannel: "discord"` for routing, but use `messageProvider: "discord-voice"` for MCP tool filtering.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. Configure a Discord voice session to run through a bundle-MCP CLI runtime such as `claude-cli`, speak after `/vc join`, and inspect the CLI MCP tool list or session transcript; the source path still makes `tts` available through loopback MCP.
- Is this the best way to solve the issue? Yes. Hiding `tts` at every tool exposure surface is safer than relying on the spoken-output prompt alone, because the failing issue is caused by the tool being available.
