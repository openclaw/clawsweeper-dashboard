# Commit 148e546

## Summary

Found one actionable Telnyx lifecycle parsing bug that still exists on current `main`.

## Findings

### Medium: `streaming.failed` webhooks are still acknowledged and dropped

- Kind: bug
- File: `extensions/voice-call/src/providers/telnyx.ts`
- Line: 211
- Evidence: The commit removed the bogus `call.streaming.failed` case but only acknowledges `streaming.started` and `streaming.stopped`; `streaming.failed` falls through to `default` and returns no event. A focused one-off parse check for a Telnyx-shaped `streaming.failed` body returned `{"events":[],"statusCode":200}`. Telnyx’s current Dial and Answer references list `streaming.started`, `streaming.stopped`, or `streaming.failed` as expected webhooks when `stream_url` is set, and the callback reference shows `event_type: "streaming.failed"` with `failure_reason`, `stream_id`, and `call_control_id`. ([developers.telnyx.com](https://developers.telnyx.com/api-reference/call-commands))
- Impact: When OpenClaw embeds `stream_url` for Telnyx dial/answer realtime calls, stream setup failure can be silently treated as a successful no-op webhook. The manager will not receive a non-retryable `call.error`, so failed realtime setup can leave a Telnyx call active but without the realtime media bridge the user expects.
- Suggested fix: Add a `case "streaming.failed"` branch that emits `call.error` with `retryable: false`, using `payload.failure_reason` first and falling back to `payload.reason` for compatibility. Add a regression test beside the existing `streaming.started` / `streaming.stopped` test.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: call `TelnyxProvider.parseWebhookEvent` with `data.event_type: "streaming.failed"` and a payload containing `call_control_id`, Base64 `client_state`, and `failure_reason`; current code returns an empty event list with 200.
- Is this the best way to solve the issue? yes: Telnyx documents this as a carrier webhook, so it belongs in the provider webhook parser rather than only in the WebSocket frame adapter.
