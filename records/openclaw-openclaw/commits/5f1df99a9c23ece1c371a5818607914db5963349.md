# Commit 5f1df99

## Summary

Found one low-severity user-facing regression: the Grok `web_search` missing-credential message now tells users to onboard with xAI OAuth, but that web-search path still only resolves API-key sources.

## Findings

### Low: Grok web_search suggests xAI OAuth even though it cannot use OAuth credentials

- Kind: bug
- File: `extensions/xai/src/web-search-provider.runtime.ts`
- Line: 200
- Evidence: `executeXaiWebSearchProviderTool` resolves credentials only through `resolveXaiWebSearchCredential`, which calls `resolveWebSearchProviderCredential` with scoped config and `XAI_API_KEY` at `extensions/xai/src/web-search-provider.runtime.ts:174`. The underlying resolver only reads configured secrets or env vars, and the web-search execution context exposes only `signal`, not provider auth/profile access. The new error message nevertheless says to run `openclaw onboard --auth-choice xai-oauth`.
- Impact: An OAuth-only xAI user who follows this message will still hit `missing_xai_api_key` for Grok `web_search`, because onboarding with OAuth does not populate `plugins.entries.xai.config.webSearch.apiKey` or `XAI_API_KEY`. This is especially confusing because nearby xAI tools do now support OAuth.
- Suggested fix: Either remove the OAuth recommendation from the Grok `web_search` missing-credential message, or add real OAuth/profile resolution to the web-search provider runtime before advertising it.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Configure only an xAI OAuth auth profile, leave `XAI_API_KEY` and `plugins.entries.xai.config.webSearch.apiKey` unset, then execute the Grok `web_search` provider tool. Source shows the credential resolver returns undefined on that path while the message recommends OAuth.
- Is this the best way to solve the issue? unclear. If OAuth is intended for `web_search`, the best fix is to thread auth-profile support into the web-search provider contract/runtime. If not, the narrow fix is to revert that one message to API-key-only guidance.
