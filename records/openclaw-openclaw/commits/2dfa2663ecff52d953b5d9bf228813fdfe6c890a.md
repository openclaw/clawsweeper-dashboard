# Commit 2dfa266

## Summary

Found one low-severity reliability/concurrency risk in the new split Slack media-plus-block delivery path. No security or supply-chain issue found.

## Findings

### Low: Split Slack media/control sends can interleave with other sends or fail after partial delivery

- Kind: concurrency
- File: `extensions/slack/src/action-runtime.ts`
- Line: 258
- Evidence: `handleSlackAction` now implements `mediaUrl && blocks` as two independent awaited `sendSlackMessage` calls: upload media first, then send the block/control message. The actual Slack sender serializes only each individual call with a per-target queue in `extensions/slack/src/send.ts:377`, so a second concurrent compound send to the same channel can enqueue between the first action’s media call and its block call. A local reproduction of that queue algorithm produced `A:media,B:media,A:blocks,B:blocks`.
- Impact: two concurrent Slack message-tool sends with files plus interactive/presentation blocks can display files and controls out of adjacency, making a control message appear to belong to the wrong attachment. If the media send succeeds but the follow-up block send fails, the tool reports failure after already delivering the file, so retry paths can duplicate the attachment.
- Suggested fix: make the compound media-plus-block operation queue as one unit for a given account/recipient/thread, or expose a lower-level compound send helper that holds the same Slack send queue across the media sequence and final block send. Consider returning/reporting partial success explicitly if the final send fails after media delivery.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the queue ordering: run two concurrent compound sends to the same Slack target with the first media upload delayed; the current per-call queue allows the second media send to run before the first final block send. I did not run this against a live Slack workspace.
- Is this the best way to solve the issue? Unclear. Holding a compound queue fixes OpenClaw-originated interleaving; partial Slack/network failure after media delivery still needs an explicit retry/idempotency decision.
