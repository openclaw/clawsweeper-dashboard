# Commit f5f8562384

## Summary

Found one release-test regression in the kitchen-sink sweep. The new `plugins inspect --all --runtime` command broad-loads unrelated bundled plugins, and the existing kitchen-sink assertion treats unrelated runtime-dependency diagnostics as kitchen-sink failures.

## Findings

### Medium: `inspect --all --runtime` can fail the kitchen-sink lane on unrelated bundled plugin diagnostics

- Kind: regression
- File: `scripts/e2e/lib/kitchen-sink-plugin/sweep.sh`
- Line: 85
- Evidence: The commit changes the all-plugin inspection from static `plugins inspect --all --json` to runtime `plugins inspect --all --runtime --json`. In a clean isolated state, `node openclaw.mjs plugins inspect --all --runtime --json` exits 0 but returns error diagnostics for unrelated default bundled plugins including `browser`, `document-extract`, `memory-core`, `ollama`, `web-readability`, and `xai`, all reporting missing bundled runtime deps. The prior static command returns 0 error diagnostics. `scripts/e2e/lib/kitchen-sink-plugin/assertions.mjs:244` folds `allInspect.diagnostics` into the kitchen-sink diagnostic set, and `assertExpectedDiagnostics` then rejects unexpected error messages for every success scenario.
- Impact: The kitchen-sink Docker E2E lane can fail even when the kitchen-sink plugin is fine, because the new all-runtime inspect now depends on unrelated bundled plugin runtime deps being staged in the test environment.
- Suggested fix: Keep the all-plugin inspection static, or filter/assert `allInspect` diagnostics to the kitchen-sink plugin only. The targeted `plugins inspect "$KITCHEN_SINK_ID" --runtime --json` already exercises the kitchen-sink runtime surface.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. In a clean isolated state, `node openclaw.mjs plugins inspect --all --runtime --json` produced 12 unrelated `level: "error"` diagnostics, while `node openclaw.mjs plugins inspect --all --json` produced 0. The assertion code includes all of those diagnostics in the kitchen-sink expected-diagnostics check.
- Is this the best way to solve the issue? yes: the narrowest fix is to avoid broad runtime loading for `--all`, or filter non-kitchen-sink diagnostics before asserting.
