# Commit 8e9d5c43

## Summary

Found one concrete gap: the new message-tool TTS path still cannot honor `messages.tts.auto = "inbound"` because the runner never receives or forwards whether the current turn came from audio.

## Findings

### Medium: Message-tool TTS still cannot work in inbound-only mode

- Kind: bug
- File: `src/infra/outbound/message-action-runner.ts`
- Line: 933
- Evidence: `RunMessageActionParams` carries `inboundEventKind` but no `inboundAudio` field (`src/infra/outbound/message-action-runner.ts:107`), and the new call to `maybeApplyTtsToMessageActionSendPayload` omits `inboundAudio` (`src/infra/outbound/message-action-runner.ts:933`). The helper has an `inboundAudio?: boolean` parameter and forwards it to the TTS runtime (`src/infra/outbound/message-action-tts.ts:43`, `src/infra/outbound/message-action-tts.ts:71`), while the runtime explicitly skips synthesis for `autoMode === "inbound"` unless `params.inboundAudio === true` (`extensions/speech-core/src/tts.ts:1759`). The message tool caller likewise only passes `inboundEventKind`, not inbound audio state (`src/agents/tools/message-tool.ts:557`, `src/agents/tools/message-tool.ts:1028`).
- Impact: the issue fixed by this commit explicitly reproduced with `messages.tts.auto = "inbound"` and an audio user input. That mode remains broken for `message(action=send)`: even in an audio-originated turn, the TTS runtime sees `inboundAudio` as undefined and returns the cleaned payload instead of voice media. For hidden `[[tts:text]]...[[/tts:text]]` sends, that can strip the visible tags into an empty/undeliverable message rather than attaching audio.
- Suggested fix: plumb the existing inbound-audio fact from reply dispatch/tool construction into `createMessageTool`, `RunMessageActionParams`, and `maybeApplyTtsToMessageActionSendPayload`, then add focused tests for `messages.tts.auto = "inbound"` with `inboundAudio: true` and `false`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: configure `messages.tts.auto = "inbound"`, enter an audio-originated turn, and call `message(action="send", message="[[tts:text]]Hello world.[[/tts:text]]")`. Source tracing shows the message action TTS helper is called without the audio-origin flag, so synthesis is skipped before any provider call.
- Is this the best way to solve the issue? yes: the TTS runtime already has the correct `inboundAudio` contract; the missing piece is preserving that prepared fact through the message-tool path rather than deriving it from `inboundEventKind`.
