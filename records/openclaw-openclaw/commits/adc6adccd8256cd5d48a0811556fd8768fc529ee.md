# Commit adc6adc

## Summary

Found one concrete regression: the shrinkwrap refresh adds npm `peer: true` metadata that the repository’s own shrinkwrap generator intentionally strips, so the dependency shrinkwrap guard now fails.

## Findings

### Medium: Shrinkwrap files are stale against the repo generator

- Kind: supply_chain
- File: `npm-shrinkwrap.json`
- Line: 1570
- Evidence: This commit adds `peer: true` entries to the root and bundled-plugin `npm-shrinkwrap.json` files. The generator at `scripts/generate-npm-shrinkwrap.mjs:301` explicitly deletes `metadata.peer === true` to keep npm 11 patch-version drift stable. Running `node scripts/generate-npm-shrinkwrap.mjs --all --check` fails with `.: npm-shrinkwrap.json is stale`, and checking the touched package dirs individually reports stale shrinkwraps for root plus all 10 changed plugin shrinkwraps.
- Impact: Any dependency/shrinkwrap validation lane that runs `deps:shrinkwrap:check` will fail on this commit. This also leaves published install metadata out of sync with the repo’s canonical generator.
- Suggested fix: Regenerate with the current repo script so the added `peer: true` metadata is removed, or intentionally change `normalizeNpmVersionDrift` if the project now wants to preserve that metadata.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: after `pnpm install`, run `node scripts/generate-npm-shrinkwrap.mjs --all --check`.
- Is this the best way to solve the issue? yes: the current generator is the repo contract, and its comment says these fields are deliberately normalized away across npm 11 patch versions.
