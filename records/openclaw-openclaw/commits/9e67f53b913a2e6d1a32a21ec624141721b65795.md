# Commit 9e67f53

## Summary

Found one concrete regression in the new provider-override SecretRef path: `infer web fetch --provider firecrawl` no longer resolves the supported same-plugin `webSearch.apiKey` fallback when no `webFetch.apiKey` is configured.

## Findings

### Medium: Firecrawl fetch provider override marks its webSearch credential fallback inactive

- Kind: regression
- File: `src/secrets/runtime-command-secrets.ts`
- Line: 132
- Evidence: `getWebFetchCommandSecretTargets` intentionally selects `plugins.entries.firecrawl.config.webSearch.apiKey` as an allowed fallback for Firecrawl fetch when the same plugin has no `webFetch.apiKey` (`src/cli/command-secret-targets.ts:476`). But the new provider-override classifier only treats `webFetch` override paths as active when the path is a direct fetch key or ends in `.config.webFetch.apiKey` (`src/secrets/runtime-command-secrets.ts:132`). The duplicated CLI local fallback classifier has the same shape (`src/cli/command-secret-gateway.ts:203`). As a result, the allowed Firecrawl `webSearch.apiKey` fallback is restored as inactive and overwritten back to the SecretRef instead of being returned as an assignment.
- Impact: A user with `plugins.entries.firecrawl.config.webSearch.apiKey` as a SecretRef, `tools.web.fetch.provider = "firecrawl"`, and no separate `webFetch.apiKey` can run configured fetch without an override, but `openclaw infer web fetch --provider firecrawl ...` fails to materialize the gateway-resolved key. That is the exact class of provider-scoped CLI SecretRef path this commit is trying to repair.
- Suggested fix: Treat the selected web-fetch provider’s same-plugin `webSearch.apiKey` fallback as active when no same-plugin `webFetch.apiKey` is configured, matching `getWebFetchCommandSecretTargets`; apply this in both `runtime-command-secrets.ts` and `command-secret-gateway.ts`, with regression coverage for `--provider firecrawl` plus only `plugins.entries.firecrawl.config.webSearch.apiKey`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Use a config with `tools.web.fetch.provider = "firecrawl"` and a SecretRef at `plugins.entries.firecrawl.config.webSearch.apiKey`, with the secret available only to the active gateway/runtime env. Then run `openclaw infer web fetch --provider firecrawl --url https://example.com --json`; the target selector includes the fallback path, but provider-override classification marks it inactive before assignment.
- Is this the best way to solve the issue? yes. The target-selection contract already says this fallback is part of the Firecrawl fetch command surface, so the command-secret provider override classifier should use the same active-path rule rather than inventing a narrower fetch-only rule.
