# Commit e782f47

## Summary

Found one live QA harness regression. The new Matrix reaction-echo wait can miss the just-sent reaction because it starts a fresh `/sync` observer after `sendReaction` returns, with no pre-send cursor or shared driver sync stream.

## Findings

### Medium: Reaction echo wait can miss the sent reaction and time out live Matrix approval scenarios

- Kind: reliability
- File: `extensions/qa-matrix/src/runners/contract/scenario-runtime-approval.ts`
- Line: 182
- Evidence: `reactToApproval` creates a Matrix client with only `accessToken` and `baseUrl`, so `client.waitForRoomEvent` falls through to a new one-shot observer rather than the shared driver observer. The commit adds that wait after `client.sendReaction(...)` returns. In the sync helper, a fresh observer with no `since` first calls `primeMatrixQaRoom`, which reads only `next_batch` and records no timeline events, then later polls from that cursor. If the homeserver has already persisted the reaction by the time the post-send wait primes, the reaction is before the new cursor and the wait can time out even though the reaction echo exists.
- Impact: live Matrix approval scenarios that call `reactToApproval` can fail with `Matrix approval reaction <id> was not observed...` and burn the full scenario timeout. This affects approval coverage in the transport profile and at least the fast-profile approval scenarios.
- Suggested fix: start observing before sending the driver reaction. Either reuse/pass the already primed driver scenario client into `reactToApproval`, create the driver scenario client with shared `syncState`/`syncStreams`, or capture a `since` cursor before `sendReaction` and pass it to the post-send wait.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. A focused reproduction is a fake Matrix sync where `sendReaction` returns an event id, the following no-`since` prime returns a `next_batch` after that event while discarding its timeline, and later polls omit the event. The existing source already shows this behavior: `primeMatrixQaRoom` returns only `next_batch`, and `createMatrixQaRoomObserver` starts from an empty local event list.
- Is this the best way to solve the issue? Yes. The fix should move the observation cursor before the send, or use the existing shared driver observer, so the reaction cannot fall into the gap between `sendReaction` and the wait’s prime.
