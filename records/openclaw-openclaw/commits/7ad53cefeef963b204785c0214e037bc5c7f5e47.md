## Findings

### High: Expanded JS/MJS Knip project glob breaks `check-dependencies`

- Kind: regression
- File: `config/knip.config.ts`
- Line: 175
- Evidence: The commit changes bundled plugin `project` from `["index.ts!", "src/**/*.ts!"]` to `["index.ts!", "src/**/*.{js,mjs,ts}!"]`. With current `main`, `pnpm deadcode:dependencies` exits 0, but `pnpm deadcode:unused-files` exits 1 and reports unexpected files:
  - `extensions/acpx/src/runtime-internals/error-format.mjs`
  - `extensions/acpx/src/runtime-internals/mcp-command-line.mjs`
  - `extensions/acpx/src/runtime-internals/mcp-proxy.mjs`
  - `extensions/canvas/src/host/a2ui-app/bootstrap.js`
  - `extensions/canvas/src/host/a2ui-app/rolldown.config.mjs`
  - `extensions/matrix/src/plugin-entry.runtime.js`
- Impact: CI’s `dependencies` shard runs `pnpm deadcode:dependencies` followed by `pnpm deadcode:unused-files`, so Node-relevant PRs and `main` validation will fail even though the commit is meant to repair CI.
- Suggested fix: Either mark the intentional JS/MJS dynamic/plugin build inputs as Knip entries or add the newly surfaced files to `scripts/deadcode-unused-files.allowlist.mjs`. If only canvas A2UI imports are needed for dependency accounting, prefer a narrower canvas/A2UI entry pattern over broadening every bundled plugin’s JS/MJS project files.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: after `pnpm install`, run `pnpm deadcode:unused-files` on current `main`; it fails with the six unexpected files above.
- Is this the best way to solve the issue? yes, directionally: the dependency check and unused-file guard share the same Knip config, so the fix needs to keep the A2UI imports visible without leaving newly visible dynamic JS/MJS files unclassified.
