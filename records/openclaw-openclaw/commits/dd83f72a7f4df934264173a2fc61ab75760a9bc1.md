# Commit dd83f72

## Summary

Found one concrete bypass of the new cron-safe persistence path. The commit removes `sessionId`/`sessionFile` for pre-transcript cron rows when `persistSessionEntry()` is used, but cron auth-profile auto-selection can write the original `cronSession.sessionEntry` back to `sessions.json` directly before the transcript exists.

## Findings

### Medium: Auth-profile persistence can re-register a pre-transcript cron row as resumable

- Kind: regression
- File: `src/agents/auth-profiles/session-override.ts`
- Line: 164
- Evidence: `createPersistCronSessionEntry()` now stores a sanitized clone when the cron key has a `sessionId` but the transcript path does not exist (`src/cron/isolated-agent/run-session-state.ts:51`). Immediately afterward, isolated cron calls `resolveSessionAuthProfileOverride()` with the same `cronSession.sessionEntry`, `cronSession.store`, `agentSessionKey`, and `storePath` (`src/cron/isolated-agent/run.ts:713`). When an auth profile is selected or refreshed, the generic resolver writes `sessionStore[sessionKey] = sessionEntry` and then persists that same entry to disk (`src/agents/auth-profiles/session-override.ts:164` and `src/agents/auth-profiles/session-override.ts:169`). That entry is the unsanitized cron session entry, so it includes `sessionId` and `sessionFile` even though the transcript can still be missing.
- Impact: For users with configured or stored auth profiles, the commit’s “non-resumable until transcript exists” invariant is not reliable. A crash, restart, doctor run, or `sessions cleanup --fix-missing` during this window can again see a resumable cron row pointing at a missing transcript, which is the failure mode this commit is trying to eliminate. This is especially relevant for short isolated cron runs and shell/CLI-backed cron paths where the transcript may not be present yet, or may never be created.
- Suggested fix: Route auth-profile persistence through the cron-safe persistence helper, or add a resolver persistence hook so cron can persist auth-profile mutations via `persistSessionEntry()` instead of the generic `updateSessionStore()` write. Regression coverage should combine a missing transcript cron entry with auth-profile auto-selection and assert the stored row remains without `sessionId`/`sessionFile`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Configure an auth profile order/source that causes `resolveSessionAuthProfileOverride()` to select or refresh an auto profile, run an isolated cron agent turn whose `sessionFile` does not yet exist, and inspect the session store after auth-profile resolution or simulate a crash before the later final cron persist. Source inspection shows the direct write path precisely.
- Is this the best way to solve the issue? yes. The new behavior is a persistence contract for cron rows, so all cron-time session-store writes for that row need to share the same sanitization decision.
