# Commit 48529f1

## Summary

Found two actionable issues in the new post-install Codex migration offer. The main correctness risk is the accepted interactive migration path mutating config through `openclaw migrate` while setup continues with a stale in-memory config and later writes it back.

## Findings

### Medium: Accepted onboarding migration can be overwritten by the later setup config write

- Kind: regression
- File: `src/plugins/provider-auth-choice.ts`
- Line: 175
- Evidence: The auth-choice path calls `offerPostInstallMigrations` with the in-memory `nextConfig` after Codex install/repair (`src/plugins/provider-auth-choice.ts:165-180`). If the user accepts, the helper invokes `migrateDefaultCommand` (`src/wizard/setup.post-install-migration.ts:149-151`). The Codex migration apply path can write native Codex plugin config via `ctx.runtime.config.mutateConfigFile` (`extensions/codex/src/migration/apply.ts:223-240`). After returning, setup still keeps the pre-migration `nextConfig` and later writes it to disk (`src/wizard/setup.ts:741`), with no re-read or merge of config changes made by the migration command.
- Impact: A user can accept the new onboarding migration, see native Codex plugin migration apply, then lose the migrated `plugins.entries.codex.config.codexPlugins` config when setup persists its stale config object. Skills copied to disk may survive, but plugin activation/config can be silently undone.
- Suggested fix: Persist the pending onboarding config before invoking migration, then re-read and return the post-migration config, or make `offerPostInstallMigrations` return an updated config snapshot that callers must continue with. Add an accepted-migration setup test that proves migrated Codex plugin config survives the final wizard write.
- Confidence: high

### Low: Non-interactive setup can block on Codex app-server just to emit an optional migration hint

- Kind: reliability
- File: `src/commands/onboard-non-interactive/local/auth-choice.plugin-providers.ts`
- Line: 222
- Evidence: The non-interactive path calls `offerPostInstallMigrations(..., nonInteractive: true)` when Codex is installed (`src/commands/onboard-non-interactive/local/auth-choice.plugin-providers.ts:215-227`). The helper still resolves candidates before deciding to only log a hint, so it runs `provider.detect` (`src/wizard/setup.post-install-migration.ts:113-123`). Codex detect calls `discoverCodexSource` (`extensions/codex/src/migration/provider.ts:20-23`), which always calls `discoverInstalledCuratedPlugins` (`extensions/codex/src/migration/source.ts:586`), which starts the Codex app-server with a 60s request timeout for `plugin/list` (`extensions/codex/src/migration/source.ts:203-209`, `extensions/codex/src/migration/source.ts:265-271`).
- Impact: Scripted onboarding can hang for up to 60 seconds on a slow or wedged Codex app-server only to decide whether to print a dry-run hint. This is especially surprising on the repair path, where the comment intentionally runs on already-present harness installs.
- Suggested fix: In non-interactive mode, either emit a cheap static `openclaw migrate codex --dry-run` hint without detection, or add a lightweight filesystem-only detect mode with a short timeout and defer app-server inventory to the explicit migration command.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the stale-config issue by source path: run interactive setup through an auth choice that installs/enables Codex, accept the migration prompt, select a native Codex plugin, then continue setup until the final config write. The non-interactive delay is source-proven but depends on a slow/hung `codex app-server`.
- Is this the best way to solve the issue? Yes for the config issue: setup must not keep writing a stale snapshot after a nested command mutates config. For the delay issue, a lighter detection mode is safer than spawning app-server in a hint-only path.
