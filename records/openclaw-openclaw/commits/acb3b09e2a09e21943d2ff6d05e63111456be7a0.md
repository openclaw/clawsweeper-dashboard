# Commit acb3b09

## Summary

Found one low-severity consistency bug: the shared text formatter now keeps the progress label outside the rolling line cap, but Slack rich progress drafts can still drop the visible label when configured above Slack's block cap.

## Findings

### Low: Slack rich progress drafts can still drop the label

- Kind: bug
- File: `extensions/slack/src/progress-blocks.ts`
- Line: 63
- Evidence: `buildSlackProgressDraftBlocks` builds `[labelBlock, ...progressLineBlocks].slice(-SLACK_MAX_BLOCKS)`. With `streaming.progress.render: "rich"` and `streaming.progress.maxLines` above 49, the newest 50 progress blocks are retained and the label block is discarded. A focused script on current `main` produced text fallback first line `Shelling...`, `blockCount: 50`, first block `Exec 10`, and `hasLabelBlock: false` for 60 progress rows.
- Impact: The commit’s intended behavior and docs say the label stays visible above the rolling progress rows, but Slack users using rich progress drafts and a high `maxLines` value will lose the visible label in Block Kit. The fallback `text` still contains the label, but the rendered Slack message blocks do not.
- Suggested fix: In `buildSlackProgressDraftBlocks`, reserve one Slack block for the label when present and slice only the progress blocks to `SLACK_MAX_BLOCKS - 1`; add a rich Slack test for `label + newest 49 rows`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Configure Slack progress drafts with `render: "rich"` and `maxLines: 60`, then render 60 progress rows through `buildSlackProgressDraftBlocks`; the label block is absent.
- Is this the best way to solve the issue? yes. The platform cap still needs to be honored, but the label should be kept separately and the rolling rows should absorb the cap.
