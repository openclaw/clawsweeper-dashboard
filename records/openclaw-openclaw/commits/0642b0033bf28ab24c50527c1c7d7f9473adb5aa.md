# Commit 0642b00

## Summary

Found one concrete gap in the new npm `files` negation guard. The committed test covers direct file glob negations, but npm also treats directory negations and root-anchored negations as excluding runtime files, while the new matcher lets those through.

## Findings

### Medium: Some npm `files` negations still publish tarballs without advertised runtime files

- Kind: bug
- File: `scripts/lib/plugin-npm-package-manifest.mjs`
- Line: 42
- Evidence: `packFilePatternMatchesPath()` normalizes `!dist` to `dist` and only compares it to exact file paths or glob regexes, so it does not treat a directory pattern as matching descendants. It also leaves a leading slash in `!/dist/setup-entry.js`, so that root-anchored npm pattern does not match `./dist/setup-entry.js`.
- Impact: A publishable plugin package can still advertise `openclaw.runtimeExtensions` / `runtimeSetupEntry` while the npm tarball omits those files. In focused probes, npm 11.12.1 omitted `dist` files for `files: ["dist/**", "!dist"]` and omitted `dist/setup-entry.js` for `files: ["dist/**", "!/dist/setup-entry.js"]`, but `node scripts/lib/plugin-npm-package-manifest.mjs --run ... -- npm pack --dry-run --json --ignore-scripts` exited 0 instead of rejecting the package.
- Suggested fix: avoid partially reimplementing npm packlist semantics, or extend the guard to cover npm-compatible directory descendants and root-anchored patterns. The most robust direction is to validate the actual staged pack file list, using npm’s packlist behavior, against `listPluginNpmRuntimeBuildOutputs(plan)` before publish.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Create a throwaway publishable plugin package with built `dist/index.js` and `dist/setup-entry.js`, set `files` to `["dist/**", "!dist"]` or `["dist/**", "!/dist/setup-entry.js"]`, then run the manifest wrapper around `npm pack --dry-run --json --ignore-scripts`. npm omits the runtime file(s), but the wrapper exits successfully.
- Is this the best way to solve the issue? unclear. Matching npm semantics locally is possible, but comparing the staged package’s actual pack file list to required runtime outputs is less likely to drift.
