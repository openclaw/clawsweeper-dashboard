# Commit 5f1d8a2e

## Summary

Found one policy-bypass regression in the new plugin-harness deny-all handling.

## Findings

### High: Agent-scoped sender deny-all does not restrict Codex plugin harnesses

- Kind: security
- File: `src/agents/harness/selection.ts`
- Line: 291
- Evidence: `resolvePluginHarnessDenyAllToolPolicyPrompt` calls `resolveEffectiveToolPolicy`, which can resolve the active agent id from `sessionKey`, but it discards that returned `agentId` and calls `resolveSenderToolPolicy` with only `params.agentId`. `resolveSenderToolPolicy` only reads agent-scoped `toolsBySender` when `params.agentId` is present, while the PI policy path passes the resolved `agentId`. A focused script showed this mismatch: with `sessionKey: "agent:main:..."` and `agents.list[].tools.toolsBySender["id:test-denied-sender"].deny = ["*"]`, the Codex harness received no `toolsAllow` or denial prompt unless `agentId: "main"` was explicitly set.
- Impact: Common session-derived agent runs with agent-scoped sender deny-all policy can still start the Codex plugin harness without `toolsAllow: []`. Downstream Codex code treats `toolsAllow === undefined` as native tool surface enabled, so code mode/apps/MCP are not failed closed for that denied sender.
- Suggested fix: Capture the `agentId` returned by `resolveEffectiveToolPolicy` and pass it to `resolveSenderToolPolicy`, matching `src/agents/pi-embedded-runner/effective-tool-policy.ts`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Register a Codex harness, run `runAgentHarnessAttempt` with an agent-scoped `toolsBySender` deny-all policy and a session key that resolves to that agent, but omit `params.agentId`; the harness receives unrestricted params.
- Is this the best way to solve the issue? yes. The PI path already uses the resolved agent id for sender policy, so using the same resolved id in selection restores contract parity without broadening policy behavior.
