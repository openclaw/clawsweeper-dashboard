# Commit ea098bca

## Summary

Found two actionable test/release-gate regressions. Both make live agent-turn failures look like successful validation, which can hide a broken agent path in release or live backend checks.

## Findings

### Medium: Cross-OS release checks now pass when the agent turn returns the wrong output

- Kind: regression
- File: `scripts/openclaw-cross-os-release-checks.ts`
- Line: 2784
- Evidence: `OPENCLAW_CROSS_OS_AGENT_TURN_OPTIONAL` defaults to `true` at line 41. `runAgentTurn` and `runInstalledAgentTurn` throw `Agent output did not contain the expected OK marker.` when the CLI completes but does not produce the expected `OK` marker, then `shouldRetryCrossOsAgentTurnError` classifies that assertion as retryable at line 2822. After the second attempt, `maybeBuildOptionalAgentTurnSkipResult` returns a zero-status skipped result at lines 2784-2800. The fresh, upgrade, installer-fresh, and dev-update lanes still return `status: "pass"` after this path.
- Impact: A real release regression where `openclaw agent --json` runs but produces malformed JSON, no assistant text, or the wrong assistant output can now make every cross-OS lane green by being summarized as `{"status":"skipped"}`. That weakens the documented cross-OS runtime validation from “agent turn works” to “gateway/dashboard works, agent may be skipped.”
- Suggested fix: Keep wrong-output / missing-OK-marker failures hard-failing. If live-provider unavailability must be optional, gate only externally attributable timeout/capacity errors, make the default fail-closed in release workflows, or surface a lane-level skipped/non-pass status that release approval cannot silently treat as success.
- Confidence: high

### Low: Codex CLI backend live smoke treats gateway timeouts as success

- Kind: test_gap
- File: `src/gateway/gateway-cli-backend.live.test.ts`
- Line: 431
- Evidence: The live test sends a Codex CLI backend agent request and previously failed any non-`ok` payload. The new branch returns successfully when `providerId === "codex-cli"` and `payload.status === "timeout"`, before checking status or asserting the nonce appears in the assistant text.
- Impact: A regression where the Codex CLI backend starts but never produces a model response now passes this live smoke. That is a concrete gap because the test’s only product assertion for Codex is that a real agent turn returns `CLI-BACKEND-${nonce}`.
- Suggested fix: Only skip known external capacity/auth outages, or require an explicit opt-in env for timeout tolerance. Backend timeouts from the gateway agent request should remain failures in the live smoke by default.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. For the first finding, any cross-OS agent command that exits successfully but omits the expected `OK` marker follows the `Agent output did not contain the expected OK marker` path and is converted to a skipped zero-status result after retry. For the second, a Codex live payload with `status: "timeout"` returns before the nonce assertion.
- Is this the best way to solve the issue? Yes for the fix direction: keep externally flaky provider availability separate from product-output assertions, and keep release/live validation fail-closed unless explicitly downgraded.
