# Commit 103cdd9

## Summary

Found two actionable issues in the new safe restart path: one policy bypass for `commands.restart=false`, and one low-severity blocker reporting bug.

## Findings

### Medium: `gateway.restart.request` bypasses `commands.restart=false`

- Kind: regression
- File: `src/gateway/server-methods/restart.ts`
- Line: 12
- Evidence: the new handler calls `requestSafeGatewayRestart(...)` directly and always schedules an authorized SIGUSR1 restart. Existing manual restart entry points check the same config flag first: `/restart` returns disabled when `commands.restart=false`, the gateway tool throws, and external SIGUSR1 is ignored when restart is not enabled.
- Impact: an operator who disabled manual gateway restarts can still be restarted through the new RPC path, including `openclaw gateway restart --safe` / `openclaw daemon restart --safe`.
- Suggested fix: gate `gateway.restart.request` with `isRestartEnabled(getRuntimeConfig())` before scheduling, return a structured RPC error when disabled, and add a test that `commands.restart=false` rejects this method.
- Confidence: high

### Low: safe restart preflight drops the “+N more” task blocker summary

- Kind: bug
- File: `src/infra/restart-coordinator.ts`
- Line: 129
- Evidence: the coordinator renders only `taskBlockers.slice(0, 8)`, but computes `omitted` as `counts.activeTasks - taskBlockers.length`. When there are 10 task blockers and the count matches, `omitted` is `0`, so the summary shows only 8 tasks with no indication that 2 more blockers exist. The adjacent reload handler uses the correct `blockers.length - shown.length` pattern.
- Impact: CLI/RPC preflight output can under-report active blockers during busy restarts, making operator diagnostics misleading.
- Suggested fix: compute omitted from the shown count, e.g. `const shown = taskBlockers.slice(0, 8)` and `const omitted = Math.max(0, counts.activeTasks - shown.length)`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. For the first issue, set `commands.restart=false` and call `gateway.restart.request`; the handler has no config check before scheduling. For the second, run preflight with more than eight task blockers.
- Is this the best way to solve the issue? yes. The restart-disable check belongs at the RPC boundary, and the omission bug is a local arithmetic fix.
