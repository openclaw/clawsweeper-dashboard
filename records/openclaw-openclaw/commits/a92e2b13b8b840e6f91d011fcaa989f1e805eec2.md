# Commit a92e2b13

## Summary

Found one concrete regression risk in the new blanket `toolUse` terminal detection. The fix correctly catches the reported interrupted tool-chain shape, but it also classifies a supported Pi agent early-termination path as incomplete.

## Findings

### Medium: `terminate: true` tool turns can now surface false incomplete-turn errors

- Kind: regression
- File: `src/agents/pi-embedded-runner/run/incomplete-turn.ts`
- Line: 98
- Evidence: `isIncompleteTerminalAssistantTurn` now returns true for every terminal assistant with `stopReason === "toolUse"`, and `resolveIncompleteTurnPayloadText` now allows the incomplete-turn error path even when `payloadCount > 0` if the last assistant is `toolUse` (`src/agents/pi-embedded-runner/run/incomplete-turn.ts:233-244`). The upstream `@mariozechner/pi-agent-core` contract explicitly supports tools returning `terminate: true` to skip the automatic post-tool LLM call; in that valid case the last assistant remains the tool-call assistant with `stopReason: "toolUse"`. OpenClaw’s public tool types preserve that result shape through `AnyAgentTool` / `AgentToolResult`, and plugin tools are invoked directly through `src/plugins/tools.ts`.
- Impact: A plugin or custom tool that intentionally returns `terminate: true` after successful execution can now be reported to the user as `Agent couldn't generate a response` and marked replay-invalid/abandoned, even though the agent loop completed as designed. This is a user-visible false failure and can break third-party plugin workflows that rely on the documented early-termination contract.
- Suggested fix: Carry an explicit “terminated by tool result” flag from `tool_execution_end` / attempt state into `EmbeddedRunAttemptResult`, and exclude that case from the incomplete-terminal guard. The guard should distinguish “missing post-tool assistant after a normal tool result” from “post-tool assistant intentionally skipped because all tool results terminated the batch.”
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Unclear for an in-repo bundled tool, because I did not find a bundled OpenClaw tool currently returning `terminate: true`. The failure mode is concrete for the public Pi/OpenClaw tool contract: a successful tool result with `terminate: true` leaves a terminal `toolUse` assistant by design, and the changed guard has no state to distinguish that from an interrupted tool chain.
- Is this the best way to solve the issue? The reported bug belongs in the terminal run guard, but the guard needs one more bit of attempt state from the agent loop/tool execution path. Inferring from `stopReason: "toolUse"` alone is too broad because it conflates interrupted turns with intentional tool-result termination.
