# Commit 1c42c774

## Summary

Found two actionable regressions: one in blocked-turn transcript lifecycle, and one compatibility break for existing non-bundled plugin hooks.

## Findings

### Medium: Blocked user turns are treated as orphan prompts on the next run

- Kind: data_loss
- File: `src/agents/pi-embedded-runner/run/attempt.ts`
- Line: 2807
- Evidence: `before_agent_run` blocks append only a redacted `role: "user"` message at `src/agents/pi-embedded-runner/run/attempt.ts:2807`, with no assistant companion turn. On the next run, the existing orphan-user repair sees that trailing user leaf at `src/agents/pi-embedded-runner/run/attempt.ts:2707` and calls `mergeOrphanedTrailingUserPrompt`, which always removes a non-empty user leaf and prepends its text to the next prompt at `src/agents/pi-embedded-runner/run/attempt.prompt-helpers.ts:434`. A focused SessionManager smoke confirmed the active branch after the next run contains one merged current user prompt plus the assistant reply, while the prior blocked entry remains only as an inactive JSONL line.
- Impact: The feature promises a persisted redacted blocked turn, but a normal next message removes that blocked turn from the active transcript/history and injects the redacted block notice into the next model prompt as queued user text. That loses the durable audit/display entry and pollutes the next request.
- Suggested fix: Do not leave blocked runs as a trailing user-only leaf. Persist an assistant block response after the redacted user stub, or teach orphan repair to handle `__openclaw.beforeAgentRunBlocked` entries without merging them into the next prompt while still preserving provider role ordering.
- Confidence: high

### Medium: Existing non-bundled `before_model_resolve` and `before_agent_reply` hooks now require new config

- Kind: compatibility
- File: `src/plugins/hook-types.ts`
- Line: 173
- Evidence: Before this commit, `CONVERSATION_HOOK_NAMES` only included `llm_input`, `llm_output`, `before_agent_finalize`, and `agent_end`. The commit adds pre-existing public hooks `before_model_resolve` and `before_agent_reply` to that gated list at `src/plugins/hook-types.ts:173`. Registration then blocks any non-bundled conversation hook unless `plugins.entries.<id>.hooks.allowConversationAccess=true` is set, at `src/plugins/registry.ts:2115`.
- Impact: Third-party plugins that already use `before_model_resolve` for model routing or `before_agent_reply` for synthetic replies will silently stop registering after upgrade unless every user adds the new config flag. That is a backward-incompatible plugin contract change outside the new `before_agent_run` surface.
- Suggested fix: Limit the new hard requirement to `before_agent_run`, or add a compatibility migration/grace path for existing hooks with explicit release-note treatment. If the stricter gate is intentional security hardening, make the break deliberate and discoverable rather than describing it as only a clarification.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes. For the blocked-turn issue, create a transcript with a `beforeAgentRunBlocked` user leaf, then run the same orphan repair path before a next prompt; the redacted leaf is removed from the active branch and merged into the next prompt. For the plugin compatibility issue, register a non-bundled plugin with `before_model_resolve` or `before_agent_reply` and no `allowConversationAccess`; the loader blocks it via the new conversation-hook gate.
- Is this the best way to solve the issue? unclear. The blocked-turn fix should preserve role ordering and the audit entry, likely by writing an assistant block companion. The hook gating fix depends on whether maintainers intend a security-breaking change or a backward-compatible feature addition.
