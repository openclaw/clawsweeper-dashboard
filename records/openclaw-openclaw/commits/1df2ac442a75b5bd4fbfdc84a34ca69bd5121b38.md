# Commit 1df2ac4

## Summary

Found one concrete regression: plugin skills using the documented one-level grouped layout still are not published into the generated plugin skill discovery directory.

## Findings

### Medium: Grouped plugin skill roots are skipped by generated publication

- Kind: regression
- File: `src/agents/skills/plugin-skills.ts`
- Line: 139
- Evidence: `collectSkillTargets()` publishes a direct `SKILL.md`, otherwise scans only immediate children and requires `child/SKILL.md`. It does not mirror the existing workspace loader’s supported grouped traversal in `src/agents/skills/workspace.ts:657`, which descends one level deeper for `skills/<group>/<skill>/SKILL.md`. The docs also state configured skill roots support one grouping level in `docs/tools/skills.md:141`. A focused smoke of `publishPluginSkills([skillsRoot])` with `skills/group/grouped-skill/SKILL.md` produced `{"published":false,"entries":[]}`.
- Impact: a plugin manifest that declares `skills: ["./skills"]` and uses the documented grouped layout still loads only through the original plugin path, not through `~/.openclaw/plugin-skills/`. That leaves the file-discovery publication bug unresolved for this supported layout.
- Suggested fix: make plugin skill publication share the same candidate discovery logic as `loadSkills()`, or extend `collectSkillTargets()` to include one grouped level with the same realpath and size constraints before creating generated links.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: create a plugin skill root with `skills/group/grouped-skill/SKILL.md`, call `publishPluginSkills([skillsRoot])`, and observe no generated link is created.
- Is this the best way to solve the issue? yes: publication should match the loader’s documented skill-root traversal, preferably by sharing a helper so direct prompt loading and generated discovery cannot drift again.
