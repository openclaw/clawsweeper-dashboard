# Commit d1e40ca3e8

## Summary

Found one regression in the runtime facade resolver. The new disabled-bundled check also bypasses the registry fallback used for installed/config-loaded plugin public surfaces.

## Findings

### Medium: Disabled bundled plugins now block installed facade fallback

- Kind: regression
- File: `src/plugin-sdk/facade-runtime.ts`
- Line: 78
- Evidence: `resolveFacadeModuleLocationUncached()` now returns `null` immediately when `OPENCLAW_DISABLE_BUNDLED_PLUGINS` is set, before reaching `return resolveRegistryPluginModuleLocation(params)`. The base code attempted bundled resolution first, then fell through to registry resolution. Current tests also document this fallback: `src/plugin-sdk/facade-runtime.test.ts` has registry resolution coverage for globally installed plugin roots, and `src/plugins/contracts/runtime-seams.contract.test.ts` verifies activated runtime facades from installed-style manifest records.
- Impact: Operators/tests that disable bundled plugins can still load external/config plugins elsewhere, but facade-runtime can no longer resolve their public artifacts through the manifest registry. A focused smoke created a config-loaded `line` plugin with `runtime-api.js`: the registry helper resolved `/tmp/.../runtime-api.js`, while `__testing.resolveFacadeModuleLocation({ dirName: "line", artifactBasename: "runtime-api.js", env: { OPENCLAW_DISABLE_BUNDLED_PLUGINS: "1" } })` returned `null`.
- Suggested fix: keep the disabled check in the bundled-only resolver, but let `facade-runtime` continue to call `resolveRegistryPluginModuleLocation(params)` when bundled plugins are disabled. Equivalently, change the new early return to `return resolveRegistryPluginModuleLocation(params)`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Create a temp plugin with `openclaw.plugin.json`, `runtime-api.js`, configure `plugins.load.paths`, set `OPENCLAW_DISABLE_BUNDLED_PLUGINS=1`, then compare `resolveRegistryPluginModuleLocationFromRegistry(...)` with `__testing.resolveFacadeModuleLocation(...)`.
- Is this the best way to solve the issue? yes. The commit already moved the bundled skip into `resolveBundledFacadeModuleLocation()`, so the outer runtime resolver does not need to suppress non-bundled registry fallback.
