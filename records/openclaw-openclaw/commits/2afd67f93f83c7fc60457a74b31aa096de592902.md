# Commit 2afd67f

## Summary

Found one concrete regression in the OpenTelemetry diagnostics consumer. The commit fixes FIFO ordering for blocked tool diagnostics, but that newly-correct ordering exposes an existing terminal-handler mismatch: blocked tool events do not consume the live span created by the preceding started event.

## Findings

### Medium: Blocked tool spans are duplicated and the started span is left active

- Kind: reliability
- File: `extensions/diagnostics-otel/src/service.ts`
- Line: 2165
- Evidence: The commit adds `tool.execution.blocked` to `ASYNC_DIAGNOSTIC_EVENT_TYPES` in `src/infra/diagnostic-events.ts:655`, so a Codex native tool start and blocked terminal event are dispatched FIFO from the async queue. Codex emits the start at `extensions/codex/src/app-server/event-projector.ts:806` and the blocked terminal event at `extensions/codex/src/app-server/event-projector.ts:822`. In the OTEL service, `tool.execution.started` tracks a live span with `trackTrustedSpan(...)` at `extensions/diagnostics-otel/src/service.ts:2071`, while completed/error terminal handlers consume that span with `takeTrackedTrustedSpan(...)` at `extensions/diagnostics-otel/src/service.ts:2103` and `extensions/diagnostics-otel/src/service.ts:2139`. The blocked handler instead always creates a fresh zero-duration span at `extensions/diagnostics-otel/src/service.ts:2165` and never removes the tracked started span.
- Impact: With OTEL tracing enabled and a trusted diagnostic trace present, each started-then-blocked native tool call now leaves the started span in `activeTrustedSpans` until service shutdown, while also exporting a separate blocked span. Long-running processes can accumulate stale active spans and traces show duplicate/incomplete tool execution spans for denied native tools.
- Suggested fix: Make `recordToolExecutionBlocked` mirror completed/error: call `takeTrackedTrustedSpan(evt, metadata) ?? spanWithDuration(...)`, then set blocked attributes and end the consumed span. Add a diagnostics-otel test that emits trusted `tool.execution.started` followed by `tool.execution.blocked` with the same trace span id and asserts only one `openclaw.tool.execution` span is started/ended.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Enable diagnostics-otel traces, emit trusted `tool.execution.started` and `tool.execution.blocked` with the same diagnostic trace span id, then inspect `openclaw.tool.execution` spans: the current blocked handler starts a second span and the started span remains tracked until shutdown.
- Is this the best way to solve the issue? yes. Completed and error already use `takeTrackedTrustedSpan`; blocked is the missing terminal case and should follow the same lifecycle contract.
