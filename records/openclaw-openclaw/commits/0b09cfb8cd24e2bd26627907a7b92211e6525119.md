# Commit 0b09cfb8

## Summary

Found one Linux/systemd reliability regression in the new “stop then allow” path for package updates launched from inside the gateway service environment.

## Findings

### Medium: In-service package update can kill its own updater on systemd

- Kind: reliability
- File: `src/cli/update-cli/update-command.ts`
- Line: 291
- Evidence: When `OPENCLAW_SERVICE_MARKER=openclaw` is present, the new flow calls `maybeStopManagedServiceBeforePackageUpdate`, which stops a running managed service at `src/cli/update-cli/update-command.ts:236`, then `shouldBlockPackageUpdateFromGatewayServiceEnv` returns `false` solely because `stopState.stopped` is true at `src/cli/update-cli/update-command.ts:291`. On Linux, OpenClaw’s generated systemd unit explicitly uses `KillMode=control-group` at `src/daemon/systemd-unit.ts:85`, and stopping the service is implemented as `systemctl ... stop <unit>` at `src/daemon/systemd.ts:716`. The systemd manual states that `KillMode=control-group` kills all remaining processes in the unit control group when the unit stops. ([freedesktop.org](https://www.freedesktop.org/software/systemd/man/latest/systemd.kill.html?utm_source=openai))
- Impact: If `openclaw update` is launched by the running gateway service on a Linux systemd install, the updater is a descendant in the same service control group. The new default path can stop the unit, causing systemd to terminate the updater before it performs or completes the package replacement and restart. That leaves the gateway stopped and the update incomplete, rather than cleanly blocking with the intended error.
- Suggested fix: Keep blocking package replacement when the current process has gateway service markers and the managed service was running, or first hand off to an updater process that is provably outside the service control group before calling `service.stop`. A systemd-specific implementation would need to escape the unit cgroup before the stop, then verify the service is stopped before replacing the package.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes in principle: install/run the gateway as the Linux systemd user service, launch `openclaw update --yes` from a child process of that service with the inherited gateway service env, and observe the updater being terminated when it executes `systemctl --user stop <openclaw unit>`. I did not run a live systemd repro in this container.
- Is this the best way to solve the issue? The current fix is incomplete for systemd. The safer fix is to block in-service package replacement unless the updater has been re-execed outside the service lifecycle boundary.
