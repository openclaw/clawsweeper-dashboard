# Commit c2004fe

## Summary

Found one concrete bug in the new blocked-liveness dedupe normalization: real blocked agent results can still lose the actionable error text and surface `error: "completed"` to `agent.wait` callers.

## Findings

### Medium: Blocked dedupe snapshots use `"completed"` instead of the real blocked error

- Kind: bug
- File: `src/gateway/server-methods/agent-wait-dedupe.ts`
- Line: 106
- Evidence: `readTerminalSnapshotFromDedupeEntry` reads `payload.error`, then `payload.summary`, then `entry.error?.message`, but real `agent` dedupe entries are written with `summary: "completed"` and the agent result under `result`. The blocked details produced by the runner live in `result.meta.livenessState` and `result.meta.error.message`. The new code reads `result.meta.livenessState` but not `result.meta.error.message`, so a blocked success-shaped agent payload normalizes to an error with message `"completed"`.
- Impact: Parent-facing subagent and `sessions_send` callers can receive `status: "error"` with `error: "completed"` instead of the actionable context-overflow / hook-block / image-size message. That still avoids the prior false-success status, but it fails the commit’s stated behavior of surfacing the blocked/error message and makes the repair path opaque to the user.
- Suggested fix: When `livenessState` normalizes a success-shaped dedupe entry to error, prefer `result.meta.error.message` or another explicit blocked result message before falling back to `payload.summary`. Add a regression using the real `dispatchAgentRunFromGateway` payload shape: `{ status: "ok", summary: "completed", result: { meta: { livenessState: "blocked", error: { message }}}}`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A direct probe of `readTerminalSnapshotFromGatewayDedupe` with the real dedupe payload shape returned:
  ```json
  {
    "status": "error",
    "endedAt": 123,
    "error": "completed",
    "livenessState": "blocked"
  }
  ```
- Is this the best way to solve the issue? yes. The narrow fix is to extend the existing dedupe snapshot parser to read the nested runner error message it already partially consumes via `result.meta`.
