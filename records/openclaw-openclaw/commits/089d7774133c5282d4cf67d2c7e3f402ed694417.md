# Commit 089d777

## Summary

Found one regression in the new blockquote span trimming. It fixes the reported paragraph case, but blockquotes ending in fenced or indented code can now produce crossing style spans, which the shared marker renderer cannot close correctly.

## Findings

### Medium: Blockquoted code blocks can render malformed Telegram HTML

- Kind: regression
- File: `src/markdown/ir.ts`
- Line: 690
- Evidence: `blockquote_close` now calls `closeStyle(..., { trimTrailingParagraphSeparator: true })`, and `closeStyle` blindly shortens the blockquote end by two bytes when text ends in `\n\n`. Code blocks also end with `\n\n`, but `renderCodeBlock` includes the first newline inside the `code_block` span. Repro on current `main`:

```text
markdownToTelegramHtml("> text\n> ```\n> code\n> ```")
=> <blockquote>text

<pre><code>code
</code></pre>
```

The output is missing `</blockquote>`. The underlying IR has crossing spans: blockquote `[0,10]`, code_block `[6,11]`.

- Impact: Telegram HTML output for a quoted paragraph followed by quoted code is malformed and may be rejected or rendered incorrectly. A pure blockquoted code block also nests `<blockquote>` inside `<pre><code>`, reversing the intended container relationship.
- Suggested fix: When trimming the terminal paragraph separator, do not trim the blockquote end before the end of any child style span opened inside that blockquote. For example, use `max(candidateTrimmedEnd, maxChildStyleEndWithinBlockquote)` so code/table block content remains inside the blockquote while only the actual separator falls outside. Add focused tests for blockquoted fenced code and paragraph-plus-code rendering.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. `pnpm exec tsx -e ... markdownToTelegramHtml("> text\n> ```\n> code\n> ```")` prints malformed HTML missing the blockquote close tag.
- Is this the best way to solve the issue? yes. The problem is in shared IR span boundaries, not Telegram-specific formatting; preserving well-nested spans keeps all marker renderers safe.
