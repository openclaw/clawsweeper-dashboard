# Commit f126f72

## Summary

Found two concrete Windows regressions in the new executable resolution paths: long-lived Gmail `.cmd` watcher processes are now tracked as `cmd.exe` without process-tree shutdown, and the resolver can promote a lower-priority `.cmd` from later in `PATH` over an earlier `.exe`.

## Findings

### Medium: Windows Gmail watcher can leave `gog serve` running after shutdown

- Kind: reliability
- File: `src/hooks/gmail-watcher.ts`
- Line: 104
- Evidence: `resolveGogServeInvocation` wraps `.cmd`/`.bat` `gog` as `cmd.exe /d /s /c ...`, and `spawnGogServe` stores that wrapper as `watcherProcess`. `stopGmailWatcher` later calls only `watcherProcess.kill("SIGTERM")` and then `watcherProcess.kill("SIGKILL")`, so on Windows it is killing the wrapper process, not explicitly the spawned `gog` process tree. The manual CLI path has the same pattern in `src/hooks/gmail-ops.ts:373` after spawning through the same wrapper at `src/hooks/gmail-ops.ts:398`. The repo’s timeout helper uses `taskkill /PID <pid> /T /F` for Windows process-tree termination in `src/process/exec.ts:346`, but the new long-lived Gmail watcher path does not.
- Impact: on native Windows with `gog` resolved to a `.cmd` or `.bat` shim, stopping the gateway or `openclaw webhooks gmail run` can leave the real `gog gmail watch serve` process behind, still bound to the port and outside OpenClaw lifecycle control. A restart can then hit the existing “address already in use” path or keep forwarding Gmail Pub/Sub callbacks with stale config.
- Suggested fix: avoid a shell wrapper for long-lived `gog` by unwrapping the shim to its real node/exe entrypoint, or add Windows process-tree termination for the wrapper PID in both gateway watcher and CLI shutdown paths. Add a Windows `.cmd` watcher lifecycle test that verifies shutdown uses tree kill or a direct child process.
- Confidence: medium

### Medium: Resolver ignores PATH precedence by preferring any `.cmd` over earlier `.exe`

- Kind: supply_chain
- File: `src/infra/executable-path.ts`
- Line: 156
- Evidence: `resolveExecutable` scans all PATH/PATHEXT candidates into `matches`, then returns the first `.cmd` anywhere in that list before considering any `.exe`. A focused synthetic check with `PATH=first;second`, `PATHEXT=.EXE;.CMD`, `first/tool.exe`, and `second/tool.cmd` returned the later `tool.cmd`. The adjacent existing Windows resolver in `src/plugin-sdk/windows-spawn.ts:61` preserves PATH entry order and PATHEXT order instead.
- Impact: `gcloud`, `gog`, or `tailscale` can resolve to a lower-priority user or tool shim later in `PATH` instead of the earlier executable Windows users would expect. That is a Windows compatibility regression and can become a supply-chain/security footgun on machines with mixed trusted system directories and user-writable tool directories.
- Suggested fix: preserve PATH entry order when selecting candidates. If `.cmd` needs special handling, prefer it only within the same PATH entry after deciding that entry wins, or reuse/adapt the existing ordered Windows resolver and wrapper-unwrapping logic.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Partial. The PATH-precedence issue was reproduced with a focused synthetic resolver check. The orphaned `gog` process issue needs native Windows process-tree proof, but the code path is concrete and matches the repo’s existing use of `taskkill /T` where tree termination is required.
- Is this the best way to solve the issue? Unclear. The narrow fix direction is to keep explicit executable resolution, but preserve command precedence and avoid shell-wrapper lifecycle ownership for long-lived processes.
