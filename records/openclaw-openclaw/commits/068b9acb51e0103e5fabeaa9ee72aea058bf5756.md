# Commit 068b9acb

## Summary

Found one low-severity regression in the new read-side duplicate filter. It fixes the reported ACP/WebChat duplicate, but the predicate is broad enough to hide a legitimate `chat.inject` assistant row on later `chat.history` reads when the injected text exactly matches the preceding ACP assistant reply.

## Findings

### Low: Same-text `chat.inject` rows can disappear from chat history after ACP replies

- Kind: regression
- File: `src/gateway/chat-display-projection.ts`
- Line: 1204
- Evidence: `isDuplicateAcpGatewayInjectedMessage` drops any visible `openclaw/acp-runtime` assistant row followed by equal-text `openclaw/gateway-injected`, without checking that the second row came from the WebChat final-delivery duplicate path. But `chat.inject` also appends through `appendAssistantTranscriptMessage` without an `idempotencyKey` at `src/gateway/server-methods/chat.ts:3643`, and the transcript helper stamps all such injected rows as `provider: "openclaw", model: "gateway-injected"` at `src/gateway/server-methods/chat-transcript-inject.ts:95`.
- Impact: an operator/admin-injected assistant message can appear in the live broadcast, then vanish after reload/history fetch if it repeats the immediately preceding ACP reply text. I confirmed the projection reduces `[acp-runtime "Done.", gateway-injected "Done."]` to only the ACP row.
- Suggested fix: narrow the dedupe to WebChat final-delivery rows, for example by requiring the current `gateway-injected` row to carry the final-delivery idempotency marker currently passed at `src/gateway/server-methods/chat.ts:3144`, or add an explicit metadata marker for this duplicate-prone path and filter only that marker.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: call `projectRecentChatDisplayMessages` with an `openclaw/acp-runtime` assistant text row followed by an equal-text `openclaw/gateway-injected` assistant row lacking WebChat final-delivery metadata; current projection drops the second row.
- Is this the best way to solve the issue? yes: narrowing the predicate preserves the ACP/WebChat duplicate fix while avoiding a broader semantic meaning for all `gateway-injected` transcript rows.
