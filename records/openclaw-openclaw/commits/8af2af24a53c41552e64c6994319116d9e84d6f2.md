# Commit 8af2af2

## Summary

Found one reliability issue in the new startup catch-up path. The catch-up can mark stale session files dirty, then have its intended `session-startup-catchup` sync coalesced into an already-running sync that is allowed to skip sessions, leaving the files dirty but unindexed.

## Findings

### Medium: Startup catch-up can be swallowed by an in-flight sync

- Kind: reliability
- File: `extensions/memory-core/src/memory/manager-sync-ops.ts`
- Line: 559
- Evidence: `runSessionStartupCatchup()` marks dirty files, then calls `this.sync({ reason: "session-startup-catchup" })`. But `MemoryIndexManager.sync()` returns the existing `this.syncing` promise for non-targeted sync requests when any sync is already running (`extensions/memory-core/src/memory/manager.ts:629`). If the in-flight sync is `session-start` or `watch`, `shouldSyncSessionsForReindex()` explicitly returns false for those reasons (`extensions/memory-core/src/memory/manager-session-reindex.ts:24`), and `runSync()` only preserves `sessionsDirty` without scheduling another pass. Current `main` has not changed this logic after the reviewed commit.
- Impact: a clean restart can still miss stale session transcript indexing if the first `session-start` or watch/search sync overlaps the async startup catch-up scan. That preserves the same user-visible failure this commit is meant to fix: `memory_search` silently misses session content until a later force/manual recovery path runs.
- Suggested fix: queue a follow-up non-excluded catch-up sync behind any in-flight sync when `markSessionStartupCatchupDirtyFiles()` finds dirty files, or add a general non-targeted dirty-session queue that reruns after `this.syncing` settles if dirty session files remain. Add a regression that starts a `session-start` sync, marks startup catch-up files dirty while it is in flight, then proves a later non-excluded sync indexes/clears them.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes, by source-level scheduling: start a non-targeted `session-start` sync, let startup catch-up mark stale files and call `sync({ reason: "session-startup-catchup" })` before the first sync settles, then observe that `sync()` returns the in-flight promise and no second sync is queued.
- Is this the best way to solve the issue? Yes. The fix should preserve the bounded catch-up scan, but must make the actual indexing pass durable across existing sync coalescing.
