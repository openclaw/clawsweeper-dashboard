# Commit 0031ef3

## Summary

Found one regression: legacy `secretref-env:<ENV_VAR>` markers now pass normal config validation, but runtime SecretRef resolution does not resolve that marker format. Doctor still migrates them, but startup/config validation can now accept a config that later uses the literal marker string as a credential.

## Findings

### Medium: Legacy SecretRef markers now validate but runtime treats them as plaintext credentials

- Kind: regression
- File: `src/config/validation.ts`
- Line: 493
- Evidence: `collectUnsupportedSecretRefPolicyIssues` now returns only unsupported mutable SecretRef issues, and the updated policy test explicitly expects `validateConfigObjectRaw({ channels.discord.token: "secretref-env:DISCORD_BOT_TOKEN" })` to succeed in `src/config/validation.policy.test.ts:101`. Doctor migration still converts the marker in `src/secrets/legacy-secretref-env-marker.ts:48`, but runtime SecretRef assignment collection only calls `coerceSecretRef` in `src/secrets/runtime-shared.ts:97`; `coerceSecretRef` does not parse `secretref-env:` markers in `src/config/types.secrets.ts:108`. For Discord, `extensions/discord/src/token.ts:13` accepts any non-empty normalized string and `extensions/discord/src/token.ts:47` returns it as a config token.
- Impact: A user with a valid legacy config such as `channels.discord.token: "secretref-env:DISCORD_BOT_TOKEN"` no longer gets a validation failure telling them to run doctor, and the gateway/channel path can proceed with `secretref-env:DISCORD_BOT_TOKEN` as the literal bot token. That silently breaks the channel until doctor is run manually.
- Suggested fix: Keep doctor as the only mutating migration path, but restore a non-mutating validation guard for legacy markers on supported SecretRef credential paths, or make runtime resolution intentionally parse legacy markers until the doctor migration window ends. If doctor needs to operate on invalid configs, exempt only the doctor preflight/repair path rather than changing normal raw validation globally.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. I ran `pnpm exec tsx -e ...` with a config containing `channels.discord.token: "secretref-env:DISCORD_BOT_TOKEN"`; it printed `{"validationOk":true,"resolved":"secretref-env:DISCORD_BOT_TOKEN"}`.
- Is this the best way to solve the issue? yes, directionally. The safest behavior is to keep startup/config validation fail-closed for this legacy credential shape while leaving `openclaw doctor --fix` responsible for rewriting it.
