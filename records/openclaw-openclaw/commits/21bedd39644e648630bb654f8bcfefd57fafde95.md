# Commit 21bedd3964

## Summary

Found one install-behavior regression in the shrinkwrap normalizer. Current `main` still contains the reviewed change.

## Findings

### Medium: Removing `libc` from shrinkwrap entries makes npm install wrong-libc native optional packages

- Kind: supply_chain
- File: `scripts/generate-npm-shrinkwrap.mjs`
- Line: 303
- Evidence: The new `normalizeNpmVersionDrift()` deletes `metadata.libc` for every package entry. npm 11.12.1 Arborist loads package metadata from the shrinkwrap into `node.package`, then runs optional-package platform checks against that metadata; `npm-install-checks` treats missing `target.libc` as compatible with the current libc.
- Impact: Published shrinkwraps no longer let npm prune glibc-vs-musl optional native packages. A focused comparison with `sharp@0.34.5` showed that with `libc` present, npm installed only `sharp-libvips-linux-x64` and `sharp-linux-x64` on Linux x64 glibc; after deleting only `libc`, the same `npm ci` also installed `sharp-libvips-linuxmusl-x64` and `sharp-linuxmusl-x64`. This widens the install/download surface for root and plugin packages that ship libc-specific native optional dependencies.
- Suggested fix: Preserve `libc` in generated shrinkwraps. If npm patch-version drift needs normalization, constrain it to fields npm does not use for platform pruning, or run the generator under a pinned npm version.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Generate a minimal `sharp@0.34.5` shrinkwrap, copy it, delete only `packages[*].libc` from the copy, then run `npm ci --ignore-scripts --no-audit --no-fund` in both directories and compare installed `node_modules/@img/*` packages.
- Is this the best way to solve the issue? yes. The install behavior comes from npm’s documented lock metadata path in Arborist at runtime, so preserving `libc` is safer than deleting it for file stability.
