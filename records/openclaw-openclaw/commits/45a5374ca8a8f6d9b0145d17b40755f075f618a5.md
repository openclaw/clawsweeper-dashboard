# Commit 45a5374

## Summary

Found one validation regression. The generated bundled-channel schema check exists, but current `main` only calls it when unrelated SecretRef policy issues are already present, so raw and `pluginValidation: "skip"` validation now accepts invalid bundled channel config that full plugin validation rejects.

## Findings

### Medium: Raw validation no longer rejects invalid bundled channel config

- Kind: regression
- File: `src/config/validation.ts`
- Line: 669
- Evidence: `validateConfigObjectRaw` computes `channelIssues` as `policyIssues.length > 0 ? collectRawBundledChannelConfigIssues(validatedConfig) : []`, so ordinary configs never run the new static bundled-channel AJV validation. A live focused check confirmed `validateConfigObjectRaw({ channels: { discord: { notARealKey: true } } })` returns `ok: true`, while `validateConfigObjectRawWithPlugins(..., { pluginValidation: "full" })` rejects the same input with `channels.discord: invalid config: must NOT have additional properties`.
- Impact: Callers that intentionally use raw or skipped plugin validation can treat invalid bundled channel config as valid. This affects raw validation surfaces such as config dry-run/schema checks and skip-validation snapshot reads in setup/status paths, and it regresses the previous `ChannelsSchema` behavior that validated plugin-owned channel config during raw parsing.
- Suggested fix: Run `collectRawBundledChannelConfigIssues(validatedConfig)` unconditionally after `OpenClawSchema.safeParse` succeeds, then merge with policy issues as intended. The helper uses generated metadata and AJV, so it preserves the startup/runtime laziness goal without loading plugin runtime metadata.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Run a focused validation probe against current `main`: `validateConfigObjectRaw({ channels: { discord: { notARealKey: true } } })` returns `ok: true`; full plugin validation rejects the same object.
- Is this the best way to solve the issue? yes. The commit already added `collectRawBundledChannelConfigIssues`; the bug is the guard that only invokes it when `policyIssues.length > 0`.
