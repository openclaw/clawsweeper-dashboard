# Commit 8cc1aee

## Summary

Found one regression in the new reasoning-only stream wrapper. The fix can truncate streamed MiMo V2 answers after the first `reasoning_content` chunk because it rewrites mutable partial output before the transport has finished appending later deltas.

## Findings

### Medium: Streaming MiMo reasoning-only answers can be truncated after the first delta

- Kind: bug
- File: `src/plugin-sdk/provider-stream-shared.ts`
- Line: 314
- Evidence: `wrapThinkingOnlyFinalTextStream()` calls `promoteThinkingOnlyFinalOutputToText(event.partial)` for every streamed event. OpenAI completions streams push the same mutable `output` object as `partial` on every thinking delta and as the final `done.message` (`src/agents/openai-transport-stream.ts:1829` and `src/agents/openai-transport-stream.ts:1741`). Since `output.stopReason` starts as `"stop"`, the first non-empty `reasoning_content` delta qualifies as “final” and replaces `output.content` with a text block. Later reasoning deltas continue appending to the old `currentBlock` reference, no longer present in `output.content`, so the final visible answer keeps only the first chunk.
- Impact: The target bug report specifically shows streaming `delta.reasoning_content` arriving in multiple chunks. For `mimo-v2-pro` / `mimo-v2-omni`, users can see a non-blank but truncated reply, often just the first token or first small chunk.
- Suggested fix: Do not promote `event.partial`. Promote only terminal event payloads (`event.message`) and `stream.result()`, or gate promotion to event types that are known final (`done` / final message-end) without touching shared mutable partial state. Add a regression test with two streamed thinking deltas sharing one output object.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Source path is direct: OpenAI completions uses one mutable `output` object for partials and final done; the new wrapper mutates that object on the first non-empty partial. A small local simulation reproduced final content `"hello "` while the transport’s old thinking block contained `"hello world"`.
- Is this the best way to solve the issue? yes. The intended behavior is terminal-only promotion, so skipping mutable partials preserves streaming internals and still fixes the blank final result through `done.message` and `result()`.
