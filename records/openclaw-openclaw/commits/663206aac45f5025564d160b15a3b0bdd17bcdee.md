## Summary

Found one compatibility regression in the Mantis workflow dispatch path. The Telegram desktop proof workflow now requires `pr_number` and no longer declares `baseline_ref` / `candidate_ref`, but the generic `Mantis Scenario` workflow still dispatches it with the removed inputs and still treats `pr_number` as optional.

## Findings

### Medium: `Mantis Scenario` can no longer dispatch Telegram desktop proof

- Kind: regression
- File: `.github/workflows/mantis-telegram-desktop-proof.yml`
- Line: 8
- Evidence: The changed callee declares only required `pr_number` plus optional instructions/provider/lease inputs in `.github/workflows/mantis-telegram-desktop-proof.yml:7`. Current `main` still has `.github/workflows/mantis-scenario.yml:107` dispatching `mantis-telegram-desktop-proof.yml` with `-f "baseline_ref=..."` and `-f "candidate_ref=..."`, and it only appends `-f "pr_number=..."` when `PR_NUMBER` is non-empty at `.github/workflows/mantis-scenario.yml:119`. The docs also still describe `Mantis Scenario` as a supported entry point for this proof lane.
- Impact: Running `Mantis Scenario` with `scenario_id=telegram-desktop-proof` can fail before the proof workflow starts because the child workflow now requires an input the parent does not require/pass consistently, and the parent sends inputs the child no longer defines. Operators lose the generic scenario entry point for Telegram desktop proof, especially the previously supported candidate-ref-only/manual path.
- Suggested fix: Update `.github/workflows/mantis-scenario.yml` for `telegram-desktop-proof` to match the new PR-derived contract: require `pr_number` for that scenario, stop passing `baseline_ref` / `candidate_ref`, and pass any supported `instructions` if needed. If candidate-ref-only scenario dispatch is still intended, restore compatible optional inputs in the child workflow and gate the PR-derived behavior separately.
- Confidence: high
