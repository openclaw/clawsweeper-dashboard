# Commit a1304c9

## Summary

Found one low-severity UI regression in the Sessions filter tooltip when the active-minutes filter is cleared.

## Findings

### Low: Clearing the Sessions active filter renders a raw `{count}` placeholder

- Kind: bug
- File: `ui/src/ui/views/sessions.ts`
- Line: 307
- Evidence: The commit changes the tooltip to `t("sessionsView.activeTooltip", { count: props.activeMinutes.trim() })`. The active-minutes input can be cleared and forwards `activeMinutes: ""` through `onFiltersChange`; `app-render.ts` then treats `Number(next.activeMinutes) || 0` as no active filter. The i18n runtime preserves placeholders when the replacement is falsy (`params[k] || "{k}"`), and a focused runtime check prints `Updated in the last {count} minutes.` for `{ count: "" }`.
- Impact: Users who clear the active filter to show all sessions see raw implementation copy in the tooltip instead of useful text.
- Suggested fix: Branch the tooltip copy for an empty trimmed value, e.g. a separate localized “No active-window filter.” string, or otherwise avoid calling the `{count}` string without a non-empty count.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: clear the Sessions “Active” filter input; the rendered `data-tooltip` is produced from `t("sessionsView.activeTooltip", { count: "" })`, which live-checked to `Updated in the last {count} minutes.`.
- Is this the best way to solve the issue? yes: the bug is at the call-site contract between optional filter state and a required placeholder, so the fix should choose explicit no-filter copy before invoking the placeholder string.
