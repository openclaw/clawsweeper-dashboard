# Commit b0f841e

## Summary

Found one regression risk in the new beta-channel install behavior: onboarding and doctor first-install flows now synthesize `@beta` specs but do not use the fallback specs the new helper provides, unlike the update path.

## Findings

### Medium: Beta first-installs can fail instead of falling back to the catalog default

- Kind: regression
- File: `src/commands/onboarding-plugin-install.ts`
- Line: 996
- Evidence: `resolveNpmInstallSpecsForUpdateChannel` and `resolveClawHubInstallSpecsForUpdateChannel` return `fallbackSpec: params.spec` for default specs rewritten to beta (`src/plugins/install-channel-specs.ts:54`, `src/plugins/install-channel-specs.ts:80`). `updateNpmInstalledPlugins` consumes those fallback specs after beta npm/ClawHub failures (`src/plugins/update.ts:1218`, `src/plugins/update.ts:1246`). The new onboarding path installs `npmInstallSpec` and, on failure, only reports the beta failure or offers a local path; it never retries `npmSpecs.fallbackSpec` (`src/commands/onboarding-plugin-install.ts:996`). Doctor repair has the same gap for `npmInstallSpec` and returns a warning without retrying `npmSpecs.fallbackSpec` (`src/commands/doctor/shared/missing-configured-plugin-install.ts:548`).
- Impact: On beta-channel installs, a default catalog spec such as `@vendor/plugin` or `clawhub:vendor-plugin` is rewritten to `@vendor/plugin@beta` / `clawhub:vendor-plugin@beta`. If that beta tag/version is absent but the default catalog install is valid, first-time onboarding or `doctor --fix` repair fails where it previously installed the default package, and where the update path already falls back.
- Suggested fix: Mirror the update path fallback in onboarding and doctor: retry `npmSpecs.fallbackSpec` on beta npm package/tag-not-found style failures, retry `clawhubSpecs.fallbackSpec` on beta ClawHub package/version-not-found failures, and only apply catalog `expectedIntegrity` when installing the fallback/default spec.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Use a beta-channel config with an install catalog entry whose `npmSpec` is a bare/default package and whose package lacks a `beta` dist-tag; onboarding/doctor will call the beta spec and return failure without trying the helper’s fallback default.
- Is this the best way to solve the issue? yes. The new helper already models the fallback contract, and `updateNpmInstalledPlugins` already implements the same fallback behavior for plugin updates.
