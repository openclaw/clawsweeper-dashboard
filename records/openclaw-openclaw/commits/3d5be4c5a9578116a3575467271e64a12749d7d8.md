# Commit 3d5be4c5a

## Summary

Found one low-severity privacy/reliability issue that still exists on current `main`: the Android app downloads the full Dream Diary during broad refresh paths, not only when the user opens the Dreaming detail.

## Findings

### Low: Android broad refresh eagerly fetches the full Dream Diary

- Kind: privacy
- File: `apps/android/app/src/main/java/ai/openclaw/app/NodeRuntime.kt`
- Line: 1879
- Evidence: `refreshHomeCanvasOverviewIfConnected()` includes `refreshDreamingFromGateway()` in the broad connected overview refresh at line 701. `refreshDreamingFromGateway()` always calls both `doctor.memory.status` and `doctor.memory.dreamDiary`; the gateway implementation of `doctor.memory.dreamDiary` reads and returns the entire `DREAMS.md` content (`src/gateway/server-methods/doctor.ts:859`, `src/gateway/server-methods/doctor.ts:991`). The Android settings list only needs `dreamingSummaryText()` / `dreamingStatus()` for the row, so this pulls diary contents before the Dreaming detail is opened.
- Impact: A normal Android connect or settings-home refresh transfers and retains potentially personal Dream Diary text on the device unnecessarily. Large `DREAMS.md` files can also make broad refresh slower or more fragile, and a diary read failure prevents the status summary from updating because both RPCs are in one try block.
- Suggested fix: Split Android refresh into status-only and diary-loading paths. Use `doctor.memory.status` for overview/settings rows and call `doctor.memory.dreamDiary` only from `DreamingSettingsScreen` or an explicit diary refresh.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes from the code path: connect the Android app to a gateway with a populated `DREAMS.md`, then trigger the broad overview/settings refresh; it will call `doctor.memory.dreamDiary` before the detail screen needs diary content. I did not run a live Android device capture.
- Is this the best way to solve the issue? Yes. Status and diary are already separate gateway RPCs, and the desktop UI keeps `loadDreamingStatus()` and `loadDreamDiary()` separate.
