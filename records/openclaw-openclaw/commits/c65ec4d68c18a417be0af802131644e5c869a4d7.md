# Commit c65ec4d

## Summary

Found one policy-regression bug: the Clownfish exemption is keyed only on a PR branch prefix, so any contributor can bypass the documented hard active-PR limit by naming their branch `clownfish/...`.

## Findings

### Medium: Clownfish active-limit exemption is spoofable by branch name

- Kind: regression
- File: `scripts/github/barnacle-auto-response.mjs`
- Line: 243
- Evidence: `isClownfishPullRequest()` returns true whenever `headRefName` or `head.ref` starts with `clownfish/`, without checking `pullRequest.user.login`, app identity, repository owner, or another trusted signal. The exemption then removes `r: too-many-prs` before the generic close rule runs at `scripts/github/barnacle-auto-response.mjs:1034`. `CONTRIBUTING.md:96` documents the 10-open-PR cap as a hard per-author limit, and `.github/workflows/labeler.yml:397`-`417` applies `r: too-many-prs` based on author open-PR count.
- Impact: A contributor over the active PR limit can open or rename a branch to `clownfish/manual-bypass`; Barnacle removes `r: too-many-prs`, does not post the active-limit comment, and does not close the PR. I confirmed this with a read-only Node smoke that produced `removeLabel: r: too-many-prs`, `createComment: []`, and `update: []` for a `CONTRIBUTOR` user.
- Suggested fix: Require the known Clownfish automation identity in addition to the branch prefix, and add a negative test for a normal contributor using `headRefName: "clownfish/..."`. Consider applying the same predicate in `labeler.yml` to avoid label churn.
- Confidence: high
