# Commit 01a22d4

## Summary

Found one low-severity regression: shared Telegram interactive buttons are now rendered by delivery, but delivery does not apply the same `inlineButtons` scope policy that callback handling applies. In disabled or wrong-scope configs, users can see buttons that Telegram will later ignore.

## Findings

### Low: Telegram can render interactive buttons that callback handling ignores

- Kind: regression
- File: `extensions/telegram/src/bot/delivery.replies.ts`
- Line: 806
- Evidence: `deliverReplies` now derives `reply_markup` from `reply.interactive` unconditionally via `resolveTelegramInlineButtons` at `extensions/telegram/src/bot/delivery.replies.ts:806`. Plugin binding approval payloads use `interactive` buttons with `pluginbind:*` callback data at `src/plugins/conversation-binding.ts:305` and `src/plugins/conversation-binding.ts:681`. But Telegram callback handling checks `resolveTelegramInlineButtonsScope` first, and for non-exec-approval callbacks returns immediately when `inlineButtonsScope === "off"` or the DM/group scope does not match at `extensions/telegram/src/bot-handlers.runtime.ts:1318` and `extensions/telegram/src/bot-handlers.runtime.ts:1329`. The plugin binding callback is only parsed afterward at `extensions/telegram/src/bot-handlers.runtime.ts:1385`.
- Impact: With `channels.telegram.capabilities.inlineButtons: "off"` or a DM/group-scoped mismatch, OpenClaw can send visible Allow/Deny-style buttons that do nothing when tapped. This is especially misleading for plugin bind approvals because the prompt text asks the user to choose, but the callback is dropped before `pluginbind:*` handling.
- Suggested fix: Gate derived `reply.interactive` Telegram buttons in `deliverReplies` with the same inline-button scope policy used by callback handling. If the current chat is outside scope or inline buttons are off, omit the markup or render an explicit text fallback. Keep explicit `channelData.telegram.buttons` behavior only if that is intentionally scope-managed upstream.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. Configure Telegram with `capabilities.inlineButtons: "off"` or `inlineButtons: "dm"` in a group, deliver a plugin binding approval payload through `deliverReplies`, and observe `sendMessage` receives `reply_markup`; tapping a `pluginbind:*` button then returns early in callback handling before `parsePluginBindingApprovalCustomId`.
- Is this the best way to solve the issue? Yes. Rendering and callback acceptance need to share the same scope predicate; otherwise the UI advertises actions that runtime policy rejects.
