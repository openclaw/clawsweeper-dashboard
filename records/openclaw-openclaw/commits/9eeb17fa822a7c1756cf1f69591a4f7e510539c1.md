# Commit 9eeb17f

## Summary

Found two response-schema hardening gaps. One is a likely Gemini compatibility regression for valid responses with grounding metadata but no `groundingChunks`; the other is an xAI malformed-response path that still leaks a raw TypeError instead of the provider-owned malformed JSON error this commit is trying to guarantee.

## Findings

### Medium: Gemini treats optional `groundingChunks` as required once `groundingMetadata` exists

- Kind: regression
- File: `extensions/google/src/gemini-web-search-provider.runtime.ts`
- Line: 246
- Evidence: The new parser allows `groundingMetadata === undefined`, but if `groundingMetadata` is present it requires `groundingMetadata.groundingChunks` to be an array. The previous code used `candidate?.groundingMetadata?.groundingChunks ?? []`, so a response with answer text and `groundingMetadata` fields such as `webSearchQueries`/`searchEntryPoint` but no citation chunks still succeeded with empty citations. The installed upstream `@google/genai` types define `GroundingMetadata.groundingChunks?` as optional alongside other optional grounding fields.
- Impact: A valid Gemini Google Search grounding response variant can now fail the whole `web_search` tool as `Gemini API error: malformed JSON response`, even though the answer text is present. That is a user-facing search failure and a behavior regression from the pre-commit fallback to empty citations.
- Suggested fix: Treat missing `groundingChunks` the same as absent `groundingMetadata`: `[]`. Still reject `groundingChunks` when the field is present and not an array.
- Confidence: medium

### Low: xAI `output` with the wrong container type still throws a raw TypeError

- Kind: bug
- File: `extensions/xai/src/responses-tool-shared.ts`
- Line: 50
- Evidence: `extractXaiWebSearchContent` iterates `for (const output of data.output ?? [])` without first checking that `data.output` is an array. A successful JSON object like `{ "output": {} }` passes the new root-object guard, then throws `object is not iterable (cannot read property Symbol(Symbol.iterator))` instead of `xAI web search failed: malformed JSON response`. I verified this directly with `requireXaiResponseTextAndCitations({ output: {} }, "xAI web search failed")`.
- Impact: Malformed xAI web search, x_search, or code_execution success payloads can still leak unstable raw parser/runtime errors despite the changelog promise of provider-owned malformed-response errors.
- Suggested fix: In `extractXaiWebSearchContent`, reject a present non-array `output` via the callerâ€™s provider label, or make the `require*` helpers validate `output` before iterating.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes for the xAI raw TypeError via the direct helper call above; yes for the Gemini control flow by feeding a candidate with `content.parts` text and `groundingMetadata` lacking `groundingChunks`.
- Is this the best way to solve the issue? yes for Gemini: preserve optional citation chunks while still validating wrong-shaped chunks. For xAI, the cleanest fix is to move labeled validation into the `require*` path so all three tools share the same provider-owned error.
