# Commit bc924889be

## Summary

Found one actionable regression in the Open WebUI Docker E2E lane: the plan is now image-free, but the executor still passes the generic functional image through the environment, so the script can still try to reuse or pull an image that the plan no longer prepares.

## Findings

### Medium: Open WebUI still falls back to the unprepared functional E2E image

- Kind: regression
- File: `scripts/lib/docker-e2e-scenarios.mjs`
- Line: 160
- Evidence: This commit changes `openwebui` to `liveLane(...)`, making its `imageKind` undefined and `needs.functionalImage/package` false. `OPENCLAW_DOCKER_ALL_LANES=openwebui node scripts/test-docker-all.mjs --plan-json` confirms `imageKinds: []`, `functionalImage: false`, and `package: false`. But `scripts/test-docker-all.mjs:1148` still sets `baseEnv.OPENCLAW_DOCKER_E2E_IMAGE` to the functional image for every run, and `laneEnv` copies that env for lanes with no `e2eImageKind`. Then `scripts/e2e/openwebui-docker.sh:9` calls `docker_e2e_resolve_image`, whose fallback in `scripts/lib/docker-e2e-image.sh:28` prefers `OPENCLAW_DOCKER_E2E_IMAGE` over the Open WebUI default. Focused check: `OPENCLAW_DOCKER_E2E_IMAGE=openclaw-docker-e2e-functional:local bash -lc 'ROOT_DIR="$PWD"; source scripts/lib/docker-e2e-image.sh; docker_e2e_resolve_image openclaw-openwebui-e2e OPENCLAW_OPENWEBUI_E2E_IMAGE'` returns `openclaw-docker-e2e-functional:local`.
- Impact: Open WebUI-only jobs now skip package download and functional image pull/build based on the plan, while the runtime script can still try to reuse/pull that functional image. On a clean CI runner, this can fail before the Open WebUI smoke starts, or accidentally use a stale/preexisting functional image when one happens to be present.
- Suggested fix: For lanes without `e2eImageKind`, remove `OPENCLAW_DOCKER_E2E_IMAGE` from `laneEnv`, or make `openwebui-docker.sh` ignore the generic fallback and use only `OPENCLAW_OPENWEBUI_E2E_IMAGE` plus its own default. Add a test that covers executor env/image resolution, not only plan JSON.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: the plan JSON says no image/package is needed, while the same executor/script image resolver still selects the functional image from `OPENCLAW_DOCKER_E2E_IMAGE`.
- Is this the best way to solve the issue? yes: the safest fix is to align executor env with the lane plan so image-free lanes cannot see the generic package-image fallback.
