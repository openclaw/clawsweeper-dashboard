# Commit 375afbad2d

## Summary

Found two release-workflow concurrency issues. The more important one is that a later alpha preflight can cancel an in-progress real alpha npm publish because both modes share the same concurrency group.

## Findings

### Medium: Alpha preflight cancellation can cancel a real npm publish

- Kind: supply_chain
- File: `.github/workflows/openclaw-npm-release.yml`
- Line: 35
- Evidence: The workflow-level group is keyed only by `inputs.tag` and `inputs.npm_dist_tag`, so preflight and publish runs for the same alpha tag share `openclaw-npm-release-<tag>-alpha` (`.github/workflows/openclaw-npm-release.yml:34`). The new `cancel-in-progress` expression is true for any workflow-dispatched alpha preflight (`.github/workflows/openclaw-npm-release.yml:35`), while the real publish path is the same workflow with `preflight_only=false` and reaches `publish_openclaw_npm`/`bash scripts/openclaw-npm-publish.sh --publish ...` (`.github/workflows/openclaw-npm-release.yml:366`, `.github/workflows/openclaw-npm-release.yml:403`, `.github/workflows/openclaw-npm-release.yml:645`). GitHub Actions documents that `cancel-in-progress: true` cancels currently running jobs or workflows in the same concurrency group. ([docs.github.com](https://docs.github.com/en/actions/writing-workflows/choosing-what-your-workflow-does/control-the-concurrency-of-workflows-and-jobs?apiVersion=2022-11-28&utm_source=openai))
- Impact: If an operator or automation dispatches a duplicate alpha preflight for the same tag while the real alpha publish is running, the preflight run can cancel the publish run. A cancellation during or near `npm publish` can leave release state ambiguous and require manual recovery.
- Suggested fix: Separate preflight and publish concurrency groups, for example by adding a mode discriminator such as `${{ inputs.preflight_only && 'preflight' || 'publish' }}` to the group, then keep `cancel-in-progress` enabled only for the alpha preflight group.
- Confidence: high

### Low: Tideclaw alpha umbrella cancellation misses tag/SHA-based dispatches

- Kind: reliability
- File: `.github/workflows/full-release-validation.yml`
- Line: 111
- Evidence: The new Full Release Validation cancellation condition only checks `inputs.ref` for `tideclaw/alpha/` (`.github/workflows/full-release-validation.yml:111`). Current release tooling dispatches this workflow with `ref: options.tag` while the trusted workflow branch is passed separately as `options.workflowRef` (`scripts/release-candidate-checklist.mjs:581`, `scripts/release-candidate-checklist.mjs:583`, `scripts/release-candidate-checklist.mjs:584`). The child release-check workflow already keys Tideclaw alpha behavior off `github.ref`/the workflow branch (`.github/workflows/openclaw-release-checks.yml:84`, `.github/workflows/openclaw-release-checks.yml:123`).
- Impact: Duplicate Tideclaw alpha Full Release Validation runs dispatched from a `tideclaw/alpha/...` workflow branch but targeting an alpha tag or full SHA will not cancel the older umbrella run, so the expensive duplicate-run problem remains for that release path.
- Suggested fix: Include the workflow ref in the parent condition, e.g. also cancel when `startsWith(github.ref, 'refs/heads/tideclaw/alpha/')`, while retaining the existing `inputs.ref` check if branch-target dispatches are supported.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. For the npm issue, start a real alpha publish and then dispatch `OpenClaw NPM Release` again with the same `tag`, `npm_dist_tag=alpha`, and `preflight_only=true`; the second run has the same group and cancellation enabled. I did not run this because it would mutate release state.
- Is this the best way to solve the issue? Yes for the main issue: separating preflight and publish concurrency groups preserves duplicate-preflight cancellation without allowing a validation rerun to kill publishing.
