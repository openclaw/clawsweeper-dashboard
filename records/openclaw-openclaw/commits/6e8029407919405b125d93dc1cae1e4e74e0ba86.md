# Commit 6e80294

## Summary

Found one concrete playback/barge-in regression in the new Discord realtime preroll path. The issue is not a security problem, but it can make the realtime provider truncate assistant audio that Discord never actually played.

## Findings

### Medium: Preroll barge-in reports buffered, unheard audio as played

- Kind: regression
- File: `extensions/discord/src/voice/realtime.ts`
- Line: 633
- Evidence: The commit buffers realtime output until `DISCORD_REALTIME_OUTPUT_PREROLL_FRAMES` is reached (`extensions/discord/src/voice/realtime.ts:50`, `extensions/discord/src/voice/realtime.ts:689`), but `outputAudioTimestampMs` is still advanced for every provider audio delta before Discord playback starts (`extensions/discord/src/voice/realtime.ts:660`). `hasInterruptibleOutputAudio()` then publishes that generated timestamp to the provider and treats the prebuffer as interruptible output (`extensions/discord/src/voice/realtime.ts:633`), and `handleBargeIn()` always calls the provider with `audioPlaybackActive: true` (`extensions/discord/src/voice/realtime.ts:622`). The OpenAI bridge uses that timestamp to send `conversation.item.truncate` with `audio_end_ms` (`extensions/openai/realtime-voice-provider.ts:1012`, `extensions/openai/realtime-voice-provider.ts:1038`).
- Impact: If a user starts speaking while, for example, 300ms of assistant audio is buffered but still below the 500ms preroll threshold, Discord has played none of it, but OpenAI receives a truncate event saying 300ms was heard. That can leave the provider-side conversation state believing the user heard part of an answer that was actually discarded locally.
- Suggested fix: Track a playback timestamp separately from generated/buffered audio. Before `outputPlaybackStarted`, barge-in should cancel/clear buffered output without reporting a positive played offset; after playback starts, report elapsed played audio capped to generated duration rather than the full generated buffer.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. Feed Discord realtime output with more than `minBargeInAudioEndMs` of audio but less than the preroll threshold, then send speaker input before `startOutputPlayback()` runs. The code path will set the provider media timestamp from buffered duration and call `handleBargeIn({ audioPlaybackActive: true })`.
- Is this the best way to solve the issue? Yes. The provider needs the amount of audio actually heard, not the amount generated or buffered. Using a played-audio clock keeps cancellation responsive without corrupting provider conversation state.
