# Commit 3bc433c

## Summary

Found one low-severity contract gap: the runtime provider-auth write path now normalizes per-agent Gemini refs, but the public SDK OAuth auth-result helper still emits stale per-agent refs in its returned `configPatch`.

## Findings

### Low: SDK OAuth auth-result helper does not normalize per-agent config patches

- Kind: bug
- File: `src/plugin-sdk/provider-auth-result.ts`
- Line: 63
- Evidence: `normalizeProviderAuthConfigPatchModelRefs` normalizes only `patch.agents.defaults` and `patch.models.providers`; it never visits `patch.agents.list`. A focused probe showed `buildOauthProviderAuthResult({ configPatch: { agents: { list: [...] }}}).configPatch` still contains `google/gemini-3-pro-preview`, while applying that patch through `applyProviderAuthConfigPatch` produces `google/gemini-3.1-pro-preview`.
- Impact: The main CLI/provider-auth write path is covered because `runProviderPluginAuthMethod` applies `applyProviderAuthConfigPatch` before writing config. The uncovered surface is the public SDK helper result itself: bundled/third-party provider tests and direct consumers that inspect or forward `ProviderAuthResult.configPatch` can still observe stale retired Gemini per-agent refs, despite the new changelog entry saying per-agent auth patches are normalized.
- Suggested fix: Extend `normalizeProviderAuthConfigPatchModelRefs` to normalize `agents.list[].model` and `agents.list[].models`, mirroring the new logic in `src/plugins/provider-auth-choice-helpers.ts`.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes: call `buildOauthProviderAuthResult` with a `configPatch.agents.list` entry containing `google/gemini-3-pro-preview`; the returned `configPatch` keeps the old id.
- Is this the best way to solve the issue? yes: the public SDK helper already owns normalization for explicit auth-result config patches, so adding the same per-agent traversal there keeps emitted helper output aligned with the runtime write path.
