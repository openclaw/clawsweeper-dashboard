# Commit f453904

## Summary

The commit adds fal and OpenRouter as `music_generate` providers and wires the related plugin metadata, docs, SDK exports, and focused tests. I found two concrete provider behavior issues: OpenRouter can corrupt streamed audio when chunks split base64 mid-quantum, and fal’s default MiniMax model incorrectly strips a valid `wav` request.

## Findings

### Medium: OpenRouter stream decoding can corrupt generated audio

- Kind: bug
- File: `extensions/openrouter/music-generation-provider.ts`
- Line: 134
- Evidence: `processOpenRouterSseLine` decodes each `delta.audio.data` fragment immediately with `Buffer.from(audio.data, "base64")`, then concatenates decoded buffers. OpenRouter’s audio streaming docs show collecting all `audio.data` chunks, joining them into one base64 string, and decoding once after streaming completes. The added unit test only splits at `audioBase64.slice(0, 4)`, which is aligned on a valid base64 quantum and misses the failure. A focused local check splitting `"wav-bytes"` at offset 5 produced `matches:false` and different decoded bytes. ([openrouter.ai](https://openrouter.ai/docs/guides/overview/multimodal/audio))
- Impact: Valid OpenRouter audio streams can produce corrupted audio files whenever SSE chunking splits base64 at a non-4-character boundary.
- Suggested fix: Accumulate `audio.data` strings in order and decode `Buffer.from(joinedAudioBase64, "base64")` once after `[DONE]`; add a regression test with a non-aligned split.
- Confidence: high

### Low: fal MiniMax `wav` output is declared unsupported even though the provider accepts it

- Kind: bug
- File: `extensions/fal/music-generation-provider.ts`
- Line: 134
- Evidence: The fal provider advertises `supportsFormat: true`, but `supportedFormatsByModel` restricts the default `fal-ai/minimax-music/v2.6` model to `["mp3"]`. The shared normalization therefore strips a user’s `format: "wav"` request before `buildFalMinimaxBody`, even though that body would pass `audio_setting.format` through and fal’s MiniMax 2.6 schema lists `mp3`, `wav`, and `pcm` as valid format enum values. OpenClaw’s public type supports only `mp3 | wav`, so `wav` should remain available. ([fal.ai](https://fal.ai/models/fal-ai/minimax-music/v2.6/api))
- Impact: Users asking the default fal music model for WAV output silently get MP3 instead, and the tool reports the format as ignored despite upstream support.
- Suggested fix: Change the default MiniMax entry to include `"wav"` in `supportedFormatsByModel`; keep `pcm` out unless OpenClaw’s public output format type expands.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. For OpenRouter, split a base64 audio string at a non-4-character boundary and decode each fragment independently, matching the current code path; the bytes differ from decoding the joined string. For fal, the runtime normalization path strips `format: "wav"` because the default model capability lists only `mp3`.
- Is this the best way to solve the issue? yes. Both fixes are narrow provider-contract corrections: OpenRouter should follow the documented stream assembly contract, and fal should align the capability declaration with the upstream model schema.
