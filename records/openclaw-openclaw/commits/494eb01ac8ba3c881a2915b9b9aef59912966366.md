# Commit 494eb01

## Summary

Found one concrete issue in the new black-box Telegram RTT driver: the “mentioned message reply” scenario is implemented as another `/status` command and accepts any SUT bot message, so it can pass without proving the normal mention/agent reply path.

## Findings

### Medium: RTT mention scenario only verifies a native status command

- Kind: test_gap
- File: `scripts/e2e/npm-telegram-rtt-driver.mjs`
- Line: 201
- Evidence: The scenario named `telegram-mentioned-message-reply` sends `/status@${sutMe.username} RTT marker ...` instead of a normal `@bot ...` group mention, and passes `allowAnySutReply: true`. In `waitForSutReply`, `anySutReplyMatches` makes the predicate succeed for any fresh message from the SUT bot before the `matchText` marker has to match (`scripts/e2e/npm-telegram-rtt-driver.mjs:115`-`120`). The established QA live scenario for the same ID sends a normal mention text (`extensions/qa-lab/src/live-transports/telegram/telegram-live.runtime.ts:274`-`283`).
- Impact: A package with broken Telegram mention routing, agent turn handling, or model-backed group replies can still pass this new RTT lane as long as the native `/status` command replies. The reported `mentionReplyMs` then measures native command RTT rather than the user-visible mention/agent path the scenario name and RTT harness imply.
- Suggested fix: Keep the canary as `/status`, but make `telegram-mentioned-message-reply` send a normal mention message and require either `reply_to_message.message_id === request.message_id` or a unique marker echoed by the model. Do not set `allowAnySutReply` for marker-validated scenarios.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes by code path: for this scenario, any fresh SUT message satisfies the predicate because `allowAnySutReply` is true. I could not run the live Telegram lane because the required Telegram env vars were absent.
- Is this the best way to solve the issue? Yes: preserving the `/status` canary while requiring a real mention reply for the RTT scenario aligns this black-box driver with the existing QA scenario contract and prevents false positives.
