# Commit 024528c

## Summary

Found one low-severity doctor/status-preflight regression risk: the new Nextcloud Talk bot response probe treats failures from an admin-only bot-list endpoint as doctor warnings even though the existing `apiUser` / `apiPassword` config is documented and prompted as room-lookup credentials, not admin credentials.

## Findings

### Low: Doctor Can Warn On Valid Non-Admin Room-Lookup Credentials

- Kind: bug
- File: `extensions/nextcloud-talk/src/doctor.ts`
- Line: 26
- Evidence: `bot-preflight.ts` uses configured `apiUser` / `apiPassword` to call `/ocs/v2.php/apps/spreed/api/v1/bot/admin`, and any non-OK response becomes `code: "api_error"`. `doctor.ts` then reports `api_error` as a preview warning. The same credentials are still documented as â€śAPI user for room lookups (DM detection)â€ť and the setup prompt says â€śConfigure optional Nextcloud Talk API credentials for room lookups?â€ť, so a normal room participant/moderator credential can be valid for the pre-existing room lookup feature but still fail this admin endpoint. Official Nextcloud Talk docs say `bot/admin` returns `403 Forbidden` when the current user is not an administrator. - Impact: Users with working Nextcloud Talk room lookup credentials but without admin credentials can get a new doctor warning like â€śbot response feature probe failed (403)â€ť even when the channel is otherwise configured correctly. - Suggested fix: Treat `401` / `403` from the admin bot-list endpoint as a skipped probe with an explicit â€śadmin credentials required for response-feature preflightâ€ť message, or add separate admin-only config/docs and only warn when the probe actually had admin-capable credentials.
- Confidence: medium
