# Commit 89a3b9a

## Summary

Found one actionable regression in the Telegram isolated ingress spool drain. The new live-claim protection is sound for the covered cases, but the same commit also changed pending spool enumeration from unbounded to the first 1,000 files, which can reintroduce lane starvation for large same-lane backlogs.

## Findings

### Medium: Large blocked Telegram lane can starve unrelated lanes beyond the first 1,000 pending files

- Kind: regression
- File: `extensions/telegram/src/polling-session.ts`
- Line: 437
- Evidence: `#drainSpooledUpdates` now calls `listTelegramSpooledUpdates({ limit: TELEGRAM_SPOOLED_DRAIN_SCAN_LIMIT })`, where the scan limit is `100 * 10`. The drain loop skips entries whose lane is already active or claimed, but because every cycle re-reads the same sorted first 1,000 pending files, an unrelated lane at file 1,001 or later is never considered while those earlier blocked same-lane files remain pending. The parent behavior used `limit: "all"`, and the adjacent regression test only covers 100 blocked same-lane updates before an unrelated lane.
- Impact: A busy Telegram topic/thread with more than 1,000 queued updates behind an active or live-claimed lane can prevent later independent topics from being delivered, even though the isolated ingress drain is explicitly supposed to scan past active-lane backlogs and start unrelated lanes.
- Suggested fix: Preserve a bounded start limit, but do not use a fixed first-N scan window for pending files. Continue scanning sorted pending entries until either `TELEGRAM_SPOOLED_DRAIN_START_LIMIT` handlers have started or the pending list is exhausted, or add a paged/cursor-based scan that can advance past blocked entries.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Adapt `extensions/telegram/src/polling-session.test.ts:815` so it writes 1,001 blocked topic-10 updates before the topic-11 update; with the current cap, `handled:1002` will not be reached until topic 10 releases.
- Is this the best way to solve the issue? yes. The intended concurrency contract is lane-based, so blocked entries should not consume the entire discovery window for unrelated lanes.
