# Commit 22657861c8

## Summary

Found one medium-confidence regression risk in the new Codex auto-activation helper. The code change is still present on current `main`; only `CHANGELOG.md` changed after this commit.

## Findings

### Medium: Scoped Codex activation can replace the active plugin registry mid-run

- Kind: regression
- File: `src/agents/harness/runtime-plugin.ts`
- Line: 31
- Evidence: `ensureSelectedAgentHarnessPlugin` activates `onlyPluginIds: ["codex"]`. When that scoped plugin is not already active, `ensurePluginRegistryLoaded` loads that scoped registry, and the loader clears activated plugin runtime state before activating the new registry (`src/plugins/loader.ts:1528`, `src/plugins/loader.ts:2437`). `setActivePluginRegistry` then replaces the active/channel registry unless it was explicitly pinned (`src/plugins/runtime.ts:133`). This helper is called after normal runtime plugin loading in both `src/agents/agent-command.ts:922` and `src/agents/pi-embedded-runner/run.ts:500`.
- Impact: In local embedded/fallback runs where channel/tool/hook plugins were already loaded but `codex` was not, a one-shot `openai/...` model override can swap the active registry to Codex-only before the attempt runs. That can drop loaded channel plugins, channel agent tools, plugin hooks, or plugin-owned runtime surfaces for the same run. Gateway startup pins channel registries, but local `openclaw agent --local --model openai/... --deliver` and embedded fallback paths do not appear to have the same protection.
- Suggested fix: Do not activate a Codex-only registry over an existing active runtime. Load the union of currently active plugin ids plus `codex`, or add a targeted harness-registration path that loads Codex without clearing/replacing the current active registry. Add a regression test with an existing active channel/plugin registry and assert it survives `ensureSelectedAgentHarnessPlugin`.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Unclear. The code path is concrete, and the loader replacement behavior is direct, but my ad hoc real plugin-load smoke hung during Codex loading and was killed, so I did not complete a live end-to-end reproduction.
- Is this the best way to solve the issue? No. The intent is right, but activating a scoped registry is broader than needed. Preserving the existing active registry while adding Codex harness availability is safer.
