# Commit e259938

## Summary

Found one startup reliability regression that still applies on current `main`: `sessions.list` is now startup-gated, but key session-list clients do not retry the new retryable startup error. Discord reply-safety edge cases introduced by this commit were already fixed later on `main` by `5572c8137c` and `343f859b90`.

## Findings

### Medium: Startup-gated `sessions.list` can leave session UIs stuck on a transient startup error

- Kind: regression
- File: `src/gateway/server-startup-unavailable-methods.ts`
- Line: 5
- Evidence: This commit adds `sessions.list` to `STARTUP_UNAVAILABLE_GATEWAY_METHODS`. While sidecars are pending, `handleGatewayRequest` now returns retryable `UNAVAILABLE` before dispatch for any listed method (`src/gateway/server-methods.ts:136`). Control UI reconnect calls `refreshActiveTab`, which calls `loadSessions` when the active tab is `sessions` (`ui/src/ui/app-gateway.ts:493`, `ui/src/ui/app-settings.ts:375`), but `loadSessionsOnce` sends one `sessions.list` request and catches failures by setting `state.sessionsError = String(err)` with no retry (`ui/src/ui/controllers/sessions.ts:439`, `ui/src/ui/controllers/sessions.ts:467`). TUI has the same mismatch: `chat.history` has a bounded retry loop for startup `UNAVAILABLE`, but `listSessions` sends one `sessions.list` request and propagates the error (`src/tui/gateway-chat.ts:204`, `src/tui/gateway-chat.ts:222`).
- Impact: During the exact startup window this commit creates, opening/reconnecting the Control UI on the Sessions tab, refreshing session options, or using TUI session-list flows can show/fail with `sessions.list unavailable during gateway startup` instead of waiting until sidecars clear the gate. If no later manual refresh or unrelated `sessions.changed` event occurs, the initial sessions view can remain stale/erroring after startup has actually recovered.
- Suggested fix: Reuse the existing bounded startup retry helper for `sessions.list` callers, especially `ui/src/ui/controllers/sessions.ts` and `src/tui/gateway-chat.ts`, or avoid gating `sessions.list` until the shipped clients consistently retry retryable startup `UNAVAILABLE`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Start or simulate a gateway with `unavailableGatewayMethods` containing `sessions.list`; existing server behavior returns retryable `UNAVAILABLE`. Then call Control UI `loadSessions` or TUI `GatewayChatClient.listSessions`; both make a single request and surface/store the error instead of retrying.
- Is this the best way to solve the issue? yes. The server-side gate is intentionally retryable, and `chat.history` already demonstrates the expected bounded retry shape. Applying the same retry contract to `sessions.list` preserves startup load shedding without breaking startup clients.
