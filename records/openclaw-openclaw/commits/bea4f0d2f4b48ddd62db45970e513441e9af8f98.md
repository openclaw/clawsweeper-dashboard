# Commit bea4f0d2

## Summary

Found one concrete remaining race in the fix: the heartbeat guard checks `replyRunRegistry`, but Gateway has a short accepted-before-dispatch window where the user-visible agent run is already registered in `chatAbortControllers` and the reply registry has not been populated yet.

## Findings

### Medium: Heartbeats can still start during the accepted-before-dispatch reply window

- Kind: bug
- File: `src/infra/heartbeat-runner.ts`
- Line: 1357
- Evidence: The new guard only calls `replyRunRegistry.isActive(sessionKey)` before falling through to the session-lane check. Gateway registers the run as active in `chatAbortControllers` at `src/gateway/server-methods/agent.ts:1431`, sends the accepted response at `src/gateway/server-methods/agent.ts:1471`, then waits through `yieldAfterAgentAcceptedAck()` at `src/gateway/server-methods/agent.ts:1477`; that helper is a 10ms `setTimeout` at `src/gateway/server-methods/agent.ts:600`. The reply registry entry is not created until `createReplyOperation()` in `src/auto-reply/reply/agent-runner.ts:1284`, after dispatch reaches the auto-reply runner.
- Impact: A heartbeat timer or wake that fires in that 10ms accepted-before-dispatch gap will see no active reply operation and no busy embedded session lane, so it can still start a heartbeat turn ahead of the user’s accepted reply. That preserves a small version of the WebChat interleaving failure this commit claims to fix.
- Suggested fix: Include Gateway’s accepted active-run state in the heartbeat busy predicate, or create/bind the `ReplyOperation` before the accepted ack/yield and pass it through to the runner so `replyRunRegistry.isActive(sessionKey)` is true for the whole accepted run lifetime.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? No live WebChat reproduction was run, but the source-level race is concrete: accepted agent run registration and a deliberate 10ms dispatch delay happen before `replyRunRegistry` can become active.
- Is this the best way to solve the issue? Unclear; the safest direction is to make heartbeat deferral consume the earliest canonical active-run fact rather than adding another later-stage lane check.
