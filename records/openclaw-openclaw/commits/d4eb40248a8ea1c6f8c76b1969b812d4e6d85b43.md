# Commit d4eb402

## Summary

Found one low-severity picker edge case introduced by normalizing option values without normalizing the matching preferred-provider initial value.

## Findings

### Low: Preferred-provider picker can seed autocomplete with a value that no longer exists in the options

- Kind: bug
- File: `src/flows/model-picker.ts`
- Line: 820
- Evidence: `addModelSelectOption` now normalizes each catalog row before pushing the option value, so a row like `google/gemini-3-pro-preview` becomes option value `google/gemini-3.1-pro-preview` at `src/flows/model-picker.ts:235`. Later, the preferred-provider default path still computes `initialValue = modelKey(firstModel.provider, firstModel.id)` from the raw row at `src/flows/model-picker.ts:820`. In the Google case, the searchable prompt receives `initialValue: "google/gemini-3-pro-preview"` while the option list contains only `"google/gemini-3.1-pro-preview"`.
- Impact: In the flow where `allowKeep` is true, a preferred provider is selected, the current model is on another provider, and the first preferred catalog row is a normalized Google alias, pressing Enter on the intended default can fail to select the new provider model. `src/wizard/clack-prompter.ts:78` passes this initial value straight to `@clack/prompts` autocomplete; `@clack/core@1.3.0` only initializes the selected value if the initial value exactly matches an option, so an unmatched initial value can submit `undefined`, which `promptDefaultModel` converts to `""` before running the selected-model hook. The existing BytePlus test covers this preferred-provider initial-selection shape but uses a model that is not rewritten.
- Suggested fix: Normalize the `firstModel` through the same path used for option values before assigning `initialValue`, or derive the initial value from the first normalized option. Also consider returning `{}` before `runProviderModelSelectedHook` when `selectedValue` is empty.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the code path: use a `promptDefaultModel` unit fixture matching the existing preferred-provider BytePlus test, but with `preferredProvider: "google"`, current model `openai/gpt-5.5`, and first catalog row `{ provider: "google", id: "gemini-3-pro-preview" }`; then assert the prompt `initialValue` is not present in `options`. I did not run an interactive TTY reproduction.
- Is this the best way to solve the issue? Yes. The picker should compute `initialValue` from the same normalized key used in `addModelSelectOption`, keeping the behavior local and preserving the new catalog-write normalization.
