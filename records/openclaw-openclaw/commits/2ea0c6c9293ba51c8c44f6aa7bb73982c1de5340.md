# Commit 2ea0c6c

## Summary

Found one low-severity docs regression: the new Slack unfurl docs overstate what the new default suppresses. Runtime defaults `unfurl_links` to `false`, but still omits `unfurl_media` unless configured, so media previews remain governed by Slack/default behavior.

## Findings

### Low: Slack unfurl docs overstate the default for media previews

- Kind: bug
- File: `docs/gateway/config-channels.md`
- Line: 490
- Evidence: The docs say `unfurlLinks` defaults to `false` so outbound bot links do not expand inline unless enabled, and the changelog says agent-sent URLs no longer expand unless `channels.slack.unfurlLinks` is enabled. Current runtime sends `unfurl_links: options?.unfurlLinks ?? false` but only includes `unfurl_media` when configured (`extensions/slack/src/send.ts:129`), and the focused test confirms the default payload omits `unfurl_media` (`extensions/slack/src/send.unfurl.test.ts:50`). Slack documents `unfurl_links` and `unfurl_media` as separate controls, with `unfurl_links` for primarily text-based content and `unfurl_media` for media content; Slack also says suppressing link unfurls in Block Kit messages requires both fields set to false. ([docs.slack.dev](https://docs.slack.dev/reference/methods/chat.postMessage?utm_source=openai))
- Impact: Operators can still see media URL previews after upgrade while the docs/changelog imply `unfurlLinks` alone disables URL previews. They may not discover that `unfurlMedia: false` is still required.
- Suggested fix: Qualify the docs/changelog to say `unfurlLinks` disables text-based link unfurls by default, and set `unfurlMedia: false` when media previews should also be suppressed. If the intended product change is “no URL previews at all,” change runtime to default `unfurl_media` to `false` too.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The focused unit test proves the default OpenClaw payload sends `unfurl_links: false` and omits `unfurl_media`; Slack’s API contract separates media unfurls from text-based link unfurls.
- Is this the best way to solve the issue? unclear. The safest docs-only fix is to narrow the wording; a runtime change to default `unfurlMedia` would be a product behavior change.
