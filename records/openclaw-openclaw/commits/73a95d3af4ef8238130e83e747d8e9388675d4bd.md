# Commit 73a95d3

## Summary

Found one regression: the new read-only default for `loadGatewayModelCatalog()` leaks into image-capability checks, so image-capable built-in models can be treated as text-only when `models.json` has any persisted custom rows but does not include that built-in model.

## Findings

### Medium: Image-capability checks now use the partial read-only catalog

- Kind: regression
- File: `src/gateway/server-model-catalog.ts`
- Line: 69
- Evidence: `loadGatewayModelCatalog()` now defaults `readOnly` to true, and the fast path in `src/agents/model-catalog.ts:212` returns only `models.json` provider model rows once it finds any usable row. Existing callers in `src/gateway/server-methods/chat.ts:2023`, `src/gateway/server-methods/agent.ts:639`, and node-event handling still pass `context.loadGatewayModelCatalog` without `{ readOnly: false }` into `resolveGatewayModelSupportsImages()`. That helper returns `false` when the active model is absent from the catalog (`src/gateway/session-utils.ts:1269` and `src/gateway/session-utils.ts:1326`). The upstream PI `ModelRegistry` full path still includes built-ins plus custom models, per `node_modules/@mariozechner/pi-coding-agent/dist/core/model-registry.js:263` and `:269`.
- Impact: A user with a persisted custom model row in `models.json` can have the read-only catalog contain only that row. If their session model is an image-capable built-in such as `openai/gpt-4.1`, attachment handling sees no catalog entry and treats it as text-only. `agent.run` then rejects image attachments via `supportsInlineImages: false`, while `chat.send` can unnecessarily offload images instead of sending them inline.
- Suggested fix: Keep the read-only fast path scoped to model-listing/picker calls. Capability checks should call `loadGatewayModelCatalog({ readOnly: false })`, or `resolveGatewayModelSupportsImages` should fall back to a full catalog lookup when a requested provider/model is missing from the partial read-only catalog.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused `tsx` check with a temp `models.json` containing only `custom-openai/text-only` showed `loadModelCatalog({ readOnly: true })` returns just that row and `resolveGatewayModelSupportsImages({ provider: "openai", model: "gpt-4.1" })` returns `false`. A dependency check showed PI’s built-in registry has `openai/gpt-4.1` with `input: ["text", "image"]`.
- Is this the best way to solve the issue? yes, the safest fix is to keep exact capability checks on the full catalog while leaving `models.list` read-only performance behavior intact.
