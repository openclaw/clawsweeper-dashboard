# Commit 39a931f

## Summary

Found two concrete routing regressions in the new per-number voice-call routing. The main issue is that Twilio Media Streams, one of the documented TTS paths, does not actually use the per-number TTS config when synthesizing audio.

## Findings

### Medium: Per-number TTS overrides are ignored for Twilio Media Streams playback

- Kind: bug
- File: `extensions/voice-call/src/providers/twilio.ts`
- Line: 621
- Evidence: `speak` resolves the effective per-number config and passes only a `voice` string into `provider.playTts` at `extensions/voice-call/src/manager/outbound.ts:249`. The Twilio active-stream path then calls `playTtsViaStream(input.text, streamSid)` and never reads `input.voice`. That stream synthesis uses `this.ttsProvider`, which is created once at startup from global `config.tts` in `extensions/voice-call/src/runtime.ts:458`.
- Impact: With `streaming.enabled` and a route like `numbers["+15550001111"].tts.providers.openai.voice = "alloy"`, inbound calls to that number still synthesize with the global TTS config. This breaks the advertised per-number TTS voice/persona use case for Twilio Media Streams.
- Suggested fix: Carry the effective per-call TTS config into stream synthesis, not just a voice string. For example, extend the playback input/provider boundary to accept an effective TTS override and create/resolve the telephony TTS config per playback or per call.
- Confidence: high

### Low: Outbound calls can accidentally use inbound per-number routes

- Kind: bug
- File: `extensions/voice-call/src/manager/outbound.ts`
- Line: 249
- Evidence: When `call.metadata.numberRouteKey` is absent, `speak` falls back to `call.to` and resolves `numbers` against it. The same fallback appears in `extensions/voice-call/src/webhook.ts:880` and `extensions/voice-call/src/runtime.ts:343`. The docs and issue close comment say outbound calls do not use `numbers`.
- Impact: An outbound conversation or playback to a number that exists in `config.numbers` can pick up the inbound route’s TTS/agent/model settings even though outbound calls are supposed to remain unchanged.
- Suggested fix: Resolve routes from `metadata.numberRouteKey` only, or fall back to `call.to` only when `call.direction === "inbound"`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. For the first finding, configure Twilio streaming with a global TTS voice and a different per-number TTS voice, receive an inbound call to the routed number, and observe the stream path synthesizes through the globally constructed telephony TTS provider. For the second, create an outbound active call whose `to` matches a `numbers` key and call `speak`; current code resolves that inbound route.
- Is this the best way to solve the issue? Yes. The route should be a per-inbound-call effective config carried through every response/playback path; outbound calls should not route unless they have an explicit stored inbound route key.
