# Commit 03012ac

## Summary

Found one runtime/documentation mismatch introduced by updating xAI’s advertised image model to `grok-imagine-image-quality`: OpenClaw still advertises and permits up to five edit reference images, but xAI’s current Imagine/Image docs cap multi-image editing at three source images.

## Findings

### Medium: xAI quality image edits still advertise and allow too many reference images

- Kind: bug
- File: `extensions/xai/image-generation-provider.ts`
- Line: 96
- Evidence: The xAI provider declares `edit.maxInputImages: 5`, and the shared OpenAI-compatible image provider only rejects edit requests when `inputImages.length > maxInputImages`, so four- and five-image edits are accepted locally and sent as `images` to `/images/edits`. Current xAI docs for the replacement `grok-imagine-image-quality` flow state that multi-image editing supports up to three source/reference images. ([docs.x.ai](https://docs.x.ai/docs/guides/image-generations?_bhlid=4c8d7a32edb59abbdf2780c6b4890f1dc05936bc&utm_source=openai))
- Impact: Users following OpenClaw’s current docs or `image_generate action=list` can submit 4 or 5 reference images for xAI. OpenClaw will not reject or normalize the request before the network call, so the provider request can fail at xAI instead of giving a local, actionable validation error.
- Suggested fix: Change xAI image edit capability/docs/tests from 5 to 3 for the quality model path, including `extensions/xai/image-generation-provider.ts`, `extensions/xai/image-generation-provider.test.ts`, `docs/providers/xai.md`, and `docs/tools/image-generation.md`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. Configure xAI image generation, call `image_generate` with model `xai/grok-imagine-image-quality` and 4 reference images; current code permits the request because `maxInputImages` is 5, while xAI documents a 3-image edit limit.
- Is this the best way to solve the issue? Yes. The safest fix is to align the provider capability with xAI’s current contract so core validation rejects invalid requests before making the external API call.
