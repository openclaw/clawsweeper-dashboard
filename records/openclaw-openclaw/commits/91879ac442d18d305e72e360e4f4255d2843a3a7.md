## Summary

Found one actionable issue: the permission hardening was added to a helper that current production config reads do not call, so the changelogged hardening does not appear to affect real runtime behavior.

## Findings

### Medium: Backup restore chmod is only covered on an unused helper path

- Kind: test_gap
- File: `src/config/io.observe-recovery.ts`
- Line: 642
- Evidence: The commit adds `chmod(..., 0o600)` after `copyFile` in `maybeRecoverSuspiciousConfigRead`, and the new test calls that helper directly. But `rg` shows `maybeRecoverSuspiciousConfigRead` and `maybeRecoverSuspiciousConfigReadSync` are referenced only by `src/config/io.observe-recovery.test.ts`. The production config IO module imports only `promoteConfigSnapshotToLastKnownGood` and `recoverConfigFromLastKnownGood` from this file (`src/config/io.ts:55`), while the active read observer in `src/config/io.ts:610` logs/audits suspicious reads with `restoredFromBackup: false` and never copies or chmods the `.bak`. The adjacent production-path test also asserts `readConfigFileSnapshot()` does not restore suspicious direct edits from `.bak` (`src/config/io.best-effort.test.ts:11`).
- Impact: The commit’s user-facing claim in `CHANGELOG.md:328` says restored `openclaw.json` is chmodded after suspicious-read backup recovery, but current runtime reads do not use the hardened restore helper. If the intended security fix was for real config reads, users remain unprotected by this patch; if auto-restore is no longer intended, the new tests and changelog are proving/reporting behavior that is not product behavior.
- Suggested fix: Either wire the recovery helper into the active config read path and add a production-path permission test around `readConfigFileSnapshot()`/`loadConfig()`, or remove/adjust the changelog and direct-helper tests to match the intentionally observe-only runtime behavior.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. `rg` shows no production callers for the changed helper, and the focused test run confirms both the direct helper test and the production-path “does not restore from .bak” test pass.
- Is this the best way to solve the issue? unclear. If suspicious-read auto-restore is still desired, the safest direction is to move the hardening to the active read/recovery path and test that path. If observe-only behavior is intentional, the safer direction is to remove the misleading restore claim.
