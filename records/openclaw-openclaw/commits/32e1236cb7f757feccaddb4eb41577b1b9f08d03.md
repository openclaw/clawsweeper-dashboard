# Commit 32e1236c

## Summary

Found three actionable lifecycle issues in the new held Telegram Crabbox session flow. The docs promise a long-lived exclusive session, but the implementation can leak scarce leases on early startup failures, lets the Convex credential lease expire without heartbeat, and can fail to stop non-default Crabbox providers when finishing from the session file alone.

## Findings

### Medium: early `start` failures leak the exclusive credential lease

- Kind: reliability
- File: `scripts/e2e/telegram-user-crabbox-proof.ts`
- Line: 1310
- Evidence: `startSession` acquires the Convex `telegram-user` credential at line 1310, then runs `sutIdentity`, `prepareRemoteState`, `warmupCrabbox`, and `inspectCrabbox` before entering the `try` block at line 1323. The cleanup catch that releases the credential and stops the created Crabbox lease only covers failures after line 1323.
- Impact: if SUT identity lookup, local state preparation, Crabbox warmup, or Crabbox inspect fails, the exclusive burner-account lease remains held until TTL expiry. If `warmupCrabbox` succeeded and `inspectCrabbox` fails, the Crabbox lease can also be left running.
- Suggested fix: move the whole post-acquire startup sequence into a `try/finally` or declare `credential`, `leaseId`, and `createdLease` before acquisition and use one cleanup block that covers every failure after `leaseCredential`.
- Confidence: high

### Medium: held sessions do not heartbeat the Convex lease

- Kind: concurrency
- File: `scripts/e2e/telegram-user-crabbox-proof.ts`
- Line: 1310
- Evidence: the held-session path calls `leaseCredential`, writes `session.json`, and returns to the operator; `rg` shows no heartbeat call in `telegram-user-crabbox-proof.ts`. The credential helper defaults the lease TTL to 20 minutes in `scripts/e2e/telegram-user-credential.ts:262`, and the broker expires leases at `qa/convex-credential-broker/convex/credentials.ts:395`. The docs say `start` “holds one exclusive Convex telegram-user lease” while agents continue testing.
- Impact: after the TTL, the broker can hand the same Telegram burner account to another job while the first Crabbox desktop, TDLib state, local SUT, and recorder are still active. That can create cross-run message interference and make `finish` fail to release because the lease owner/token changed.
- Suggested fix: persist enough lease metadata from acquire and start a heartbeat process/timer for held sessions, or add an explicit heartbeat subcommand that `start` supervises until `finish`. A simpler fallback is to request the maximum TTL and document/enforce a hard session expiry, but that still would not satisfy arbitrary long review sessions.
- Confidence: high

### Low: `finish` and `status` ignore the session provider when stopping or inspecting

- Kind: reliability
- File: `scripts/e2e/telegram-user-crabbox-proof.ts`
- Line: 1558
- Evidence: the session stores `session.crabbox.provider` and `session.crabbox.target`, and screenshot/WebVNC use those saved values, but `finishSession` calls `stopCrabbox(root, opts, session.crabbox.id)`. `stopCrabbox` uses `opts.provider` at `scripts/e2e/telegram-user-crabbox-proof.ts:1104`. `statusSession` similarly calls `inspectCrabbox(opts, root, session.crabbox.id)` at line 1461.
- Impact: a session started with `--provider` or `--target` can be finished with the documented `finish --session ...` command but inspect/stop the default provider instead of the provider saved in the session. The credential and local SUT may be cleaned up while the actual Crabbox lease remains running.
- Suggested fix: make status/finish stop paths use `session.crabbox.provider` and `session.crabbox.target`, or pass a session-derived options object into the Crabbox helpers.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. The first and third issues are directly reproducible with stubbed helper commands or a real broker/Crabbox setup; the second can be reproduced by setting a short allowed credential TTL, starting a held session, waiting past expiry, then acquiring another `telegram-user` lease.
- Is this the best way to solve the issue? Yes. The fixes should keep cleanup and lease ownership tied to the session lifecycle instead of relying on operator timing or repeated CLI flags.
