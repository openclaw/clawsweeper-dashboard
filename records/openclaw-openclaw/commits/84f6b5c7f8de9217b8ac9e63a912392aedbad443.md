# Commit 84f6b5c

## Summary

Found one regression in the new npm Git-source update path. The code assumes `npm pack <git spec>` runs OpenClaw’s `prepack` build, but npm runs `prepare` for git specs instead; OpenClaw’s build is in `prepack`, while `prepare` only configures git hooks. That makes `openclaw update --tag main` likely produce a source tarball without `dist`, then fail staged verification.

## Findings

### Medium: Git-source package updates pack an unbuilt OpenClaw tarball

- Kind: regression
- File: `src/infra/package-update-steps.ts`
- Line: 223
- Evidence: The new path detects npm git specs and runs `npm pack <installSpec> --pack-destination <tmp>` before installing the produced tarball. OpenClaw’s package build is wired to `prepack` in `package.json:1558`, while `prepare` in `package.json:1559` only sets `core.hooksPath`. `dist/` is ignored and not tracked, so a git-sourced tarball must run the build lifecycle to contain the runtime. A focused npm 11.12.1 fixture showed the contract mismatch: `npm pack <local folder>` with `prepack` included `dist/generated.txt`, but `npm pack git+file://...#main` with the same `prepack` produced only `package.json`; changing the fixture script to `prepare` made the git-spec tarball include the generated file.
- Impact: The newly advertised `openclaw update --tag main` / GitHub source package flow can pack and install a tarball missing OpenClaw’s built `dist` payload. In the normal staged npm path this should fail verification rather than corrupt the live package, but it turns the feature enabled by this commit into a late update failure instead of a usable main-branch package update.
- Suggested fix: Do not rely on `npm pack <git spec>` to run OpenClaw’s `prepack`. Either clone/fetch the target into a temporary checkout, run the existing build/prepack preparation there, then `npm pack <local checkout>`, or restore the explicit rejection until the git-source build path is implemented. Add a real npm fixture/integration test that verifies a git-source pack includes a generated `dist` file rather than mocking the tarball.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The focused npm fixture reproduced the lifecycle behavior without touching the repo, and the OpenClaw package scripts plus ignored/untracked `dist` make the failure path concrete for GitHub source specs.
- Is this the best way to solve the issue? yes on direction: the update path needs an explicit build-from-source step before local packing, or it should keep rejecting source package specs. Moving all build work into `prepare` may work technically but is riskier because npm runs `prepare` for git dependencies broadly.
