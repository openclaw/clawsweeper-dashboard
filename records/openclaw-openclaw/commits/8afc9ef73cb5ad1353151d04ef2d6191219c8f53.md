# Commit 8afc9ef

## Summary

Found one plugin SDK regression in the new `before_agent_finalize.retry` budget logic: retry budgets are keyed by run and idempotency key only, so separate plugins can accidentally consume each other’s retry attempts.

## Findings

### Medium: Finalize retry budgets collide across plugins

- Kind: bug
- File: `src/agents/harness/lifecycle-hook-helpers.ts`
- Line: 161
- Evidence: `normalizeBeforeAgentFinalizeResult` uses `retryRunId` plus `retry.idempotencyKey` or an instruction hash as the budget identity, but it never includes the plugin id. The hook runner receives each hook registration at `src/plugins/hooks.ts:581`, including `hook.pluginId`, but `mergeBeforeAgentFinalize` drops that ownership when building `retryCandidates`. A focused smoke script with two plugins returning `{ idempotencyKey: "shared", maxAttempts: 1 }` produced a first revise for plugin A and then `continue`, so plugin B’s still-unspent retry was skipped.
- Impact: Two independent plugins using a common key such as `tests`, `finalize`, or the same fallback instruction can suppress each other’s bounded finalization retry. That can let a natural final answer through even though a lower-priority plugin requested its own corrective pass.
- Suggested fix: Carry plugin ownership in the internal retry candidate metadata when merging `before_agent_finalize` results, and include that plugin id in the budget key. Add a contract test with two plugins using the same idempotency key and `maxAttempts: 1`, expecting the second plugin’s retry to be honored after the first is spent.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. I ran a focused `tsx` smoke script that registered two `before_agent_finalize` hooks from `plugin-a` and `plugin-b`, both using `idempotencyKey: "shared"` and `maxAttempts: 1`; the second finalize call returned `{"action":"continue"}` instead of honoring plugin B’s retry.
- Is this the best way to solve the issue? yes. The public API can stay unchanged; the runner can attach non-public plugin ownership to retry candidates before the harness budget code evaluates them.
