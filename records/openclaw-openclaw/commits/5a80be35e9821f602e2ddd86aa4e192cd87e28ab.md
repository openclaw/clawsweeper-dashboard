# Commit 5a80be3

## Summary

Found one likely remaining regression in the Codex app-server cron milestone path: the new `turn_accepted` milestone is emitted only after `turn/start` returns, so it does not protect long-running `turn/start` requests from the pre-execution watchdog.

## Findings

### Medium: Codex `turn/start` progress is reported too late to clear the cron pre-execution watchdog

- Kind: bug
- File: `extensions/codex/src/app-server/run-attempt.ts`
- Line: 1421
- Evidence: The cron watchdog starts the pre-execution timer after runner entry and clears it only when `onExecutionPhase` reports an execution milestone (`src/cron/service/timer.ts:168`, `src/cron/service/timer.ts:185`). In the Codex app-server path, `emitExecutionPhaseOnce("turn_accepted", ...)` runs after `await client.request("turn/start", ...)` has resolved (`extensions/codex/src/app-server/run-attempt.ts:1342`, `extensions/codex/src/app-server/run-attempt.ts:1421`). Notifications received before `turnId`/`projector` exist are only queued and return without calling `reportCodexExecutionNotification` (`extensions/codex/src/app-server/run-attempt.ts:1111`). PR #80283’s live proof/comment specifically describes needing progress at `turn/start` dispatch time to cover long first turns.
- Impact: If Codex accepts a turn and starts native/tool work while the `turn/start` RPC remains pending longer than the 60s pre-execution watchdog, isolated cron jobs can still be aborted as “stalled before execution start” even though work has begun. That is the false-timeout class this commit is meant to fix.
- Suggested fix: Emit a recognized execution milestone when dispatching `turn/start` before awaiting its response, or process `turn/started` pending notifications before `turnId`/`projector` setup so they clear the watchdog. Keep the post-response dedupe as a fallback.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes at source level: use a Codex app-server client where `turn/start` remains pending for more than the pre-execution watchdog while the server emits current-turn start/tool notifications. Current code queues those notifications and does not report progress until the RPC resolves.
- Is this the best way to solve the issue? Yes: moving or adding the milestone at dispatch/pending-notification time matches the intended watchdog contract without weakening cron timeout policy.
