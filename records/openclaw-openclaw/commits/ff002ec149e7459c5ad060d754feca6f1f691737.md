# Commit ff002ec

## Summary

Found one regression: default active-run steering now drops the inbound user context that this commit moved out of `followupRun.prompt`.

## Findings

### Medium: Steered messages lose inbound sender/reply/chat context

- Kind: regression
- File: `src/auto-reply/reply/get-reply-run.ts`
- Line: 630
- Evidence: This commit changes normal turns from `[inboundUserContext, baseBodyFinal]` in the prompt to only `baseBodyFinal`, and media-only turns from `[inboundUserContext, "[User sent media without caption]"]` to only the media marker. The inbound block is now stored as `followupRun.currentTurnContext` at `src/auto-reply/reply/get-reply-run.ts:752`. That works for a normal followup run because `runEmbeddedAttemptWithBackend` prepends `currentTurnContext` to the model prompt at `src/agents/pi-embedded-runner/run/attempt.ts:3081`. But the active streaming steering path does not call that attempt path. It calls `queueEmbeddedPiMessageWithOutcome(steerSessionId, followupRun.prompt, ...)` at `src/auto-reply/reply/agent-runner.ts:1050`, and the embedded handle forwards only that string to `activeSession.steer(text)` at `src/agents/pi-embedded-runner/run/attempt.ts:2715`. Since `messages.queue.mode` defaults to `"steer"` at `src/auto-reply/reply/queue/settings.ts:7`, this affects the default path for messages that arrive while a session is streaming.
- Impact: A second inbound message during an active run can be steered without the sender, chat id, reply target, structured chat window, or media-only conversation context that the model previously saw in `followupRun.prompt`. In group channels this can make the model misattribute who spoke or miss the quoted message being discussed. For pure media or bare-mention history turns, the steered text can be especially under-specified because the meaningful context now lives only in `currentTurnContext`.
- Suggested fix: Teach the steering queue to carry `currentTurnContext`, or compose that context into the steered runtime prompt when `effectiveShouldSteer` is true. If preserving prompt-local transcript behavior cannot be guaranteed for steering, fall back to a normal followup when `currentTurnContext` is present.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes, code-level: start a session run that is active and streaming with default `steer` queue mode, then send a second group-channel message with reply context or sender metadata. `runPreparedReply` will produce `followupRun.prompt` without inbound context and `followupRun.currentTurnContext` with it, while `runReplyAgent` queues only `followupRun.prompt` into steering.
- Is this the best way to solve the issue? yes, the fix needs to be at the steering boundary. Normal runs already consume `currentTurnContext`; steering is the remaining string-only path.
