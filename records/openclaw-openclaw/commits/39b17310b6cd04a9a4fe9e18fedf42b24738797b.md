# Commit 39b17310

## Summary

Found release-orchestration regressions in the new parallel publish flow. Both are in the parent workflow that dispatches mutating child release workflows.

## Findings

### Medium: OpenClaw npm can publish even after ClawHub publish has already failed

- Kind: regression
- File: `.github/workflows/openclaw-release-publish.yml`
- Line: 279
- Evidence: The workflow dispatches plugin npm and ClawHub at lines 269-270, waits only for plugin npm at lines 272-276, then dispatches `openclaw-npm-release.yml` at lines 279-284. It does not start or check the ClawHub wait until lines 289-318, after the OpenClaw npm publish has already been dispatched.
- Impact: If ClawHub fails while plugin npm is still running, the parent will still publish `openclaw` to npm once plugin npm succeeds. The parent workflow may later fail, but the npm package is already live, leaving a partial release that the previous serialized flow prevented.
- Suggested fix: Keep ClawHub publish as a hard gate before dispatching OpenClaw npm, or add a fail-fast watcher that cancels/blocks OpenClaw npm if ClawHub completes unsuccessfully before OpenClaw publish starts.
- Confidence: high

### Medium: Cancelling the parent release workflow no longer cancels active child publishes

- Kind: reliability
- File: `.github/workflows/openclaw-release-publish.yml`
- Line: 269
- Evidence: The base workflow installed `trap cancel_child EXIT INT TERM` around each dispatched child run. The new flow records child run IDs but has no equivalent trap after dispatching plugin npm, ClawHub, or OpenClaw npm. Since these child workflows are independent `workflow_dispatch` runs, parent cancellation/timeout can leave mutating publish jobs running.
- Impact: An operator can cancel or lose the parent `OpenClaw Release Publish` run and still have plugin npm, ClawHub, or OpenClaw npm publishing continue outside the parent’s control and summary.
- Suggested fix: Track all active child run IDs, install an `EXIT INT TERM` trap that cancels unfinished children, and clear each run from the active set only after it has completed successfully or been intentionally allowed to continue.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. The ordering is visible directly in `.github/workflows/openclaw-release-publish.yml`: OpenClaw npm dispatch happens before any ClawHub conclusion is read. The cancellation issue is also directly visible from the removed trap and the absence of a replacement.
- Is this the best way to solve the issue? Yes for the cancellation fix. For the ordering fix, either serialize ClawHub before OpenClaw again or add explicit fail-fast coordination before OpenClaw dispatch; simply failing the parent after both children finish does not undo npm publication.
