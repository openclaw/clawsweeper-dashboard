# Commit eb1a0aa

## Summary

Found one remaining auth-order propagation bug in the `/codex` command surface. The new request layer accepts `ctx.config`, but several command handlers still call it without config, so they can still start/use the app-server under the default Codex profile instead of `auth.order.openai-codex`.

## Findings

### Medium: Several `/codex` app-server commands still ignore configured auth order

- Kind: bug
- File: `extensions/codex/src/command-handlers.ts`
- Line: 267
- Evidence: This commit adds a config-aware fourth argument to `codexControlRequest`/`safeCodexControlRequest` in `extensions/codex/src/command-rpc.ts:39`, and `requestCodexAppServerJson` passes that config to `getSharedCodexAppServerClient` in `extensions/codex/src/app-server/request.ts:39`. But `CodexControlRequestFn` and `SafeCodexControlRequestFn` in `extensions/codex/src/command-handlers.ts:72` still expose only three arguments, and command paths such as `/codex mcp`, `/codex skills`, `/codex account`, `/codex threads`, `/codex resume`, `/codex diagnostics`, `/codex compact`, and `/codex review` call the app-server request helpers without `{ config: ctx.config }` at `extensions/codex/src/command-handlers.ts:270`, `extensions/codex/src/command-handlers.ts:280`, `extensions/codex/src/command-handlers.ts:287`, `extensions/codex/src/command-handlers.ts:426`, `extensions/codex/src/command-handlers.ts:446`, `extensions/codex/src/command-handlers.ts:816`, and `extensions/codex/src/command-handlers.ts:1409`. The changed status/models paths do pass `ctx.config` at `extensions/codex/src/command-handlers.ts:189` and `extensions/codex/src/command-handlers.ts:196`, so the behavior now differs by subcommand.
- Impact: With both `openai-codex:default` and `openai-codex:work` present and `auth.order.openai-codex` selecting `work`, `/codex status` and `/codex models` use `work`, but `/codex account`, `/codex threads`, `/codex resume`, feedback upload, and thread actions can still reuse/start the shared client under the default profile. That can show the wrong account/rate limits, list or resume the wrong account’s threads, or fail to find the configured account’s thread.
- Suggested fix: Extend `CodexControlRequestFn` and `SafeCodexControlRequestFn` to accept the existing `{ config }` option, then pass `{ config: ctx.config }` through every command-handler app-server control request. For helpers such as `buildThreads`, `resumeThread`, and `startThreadAction`, thread `ctx.config` through their parameters.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Configure two Codex profiles with `auth.order.openai-codex` selecting the non-default profile, then run `/codex account` or `/codex threads`; these paths do not pass `ctx.config` into the config-aware request layer, unlike `/codex status` and `/codex models`.
- Is this the best way to solve the issue? yes. The commit already introduced the lower-level config option; the remaining fix is to propagate it consistently from the command surface.
