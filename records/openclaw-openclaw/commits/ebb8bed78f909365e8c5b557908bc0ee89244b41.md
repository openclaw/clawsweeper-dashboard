# Commit ebb8bed

## Summary

Found one current-main bug: the new cap only budgets one fs-safe temp suffix and only applies to `createWikiPageFilename`. Other memory-wiki page writes can still generate filenames that fit the final `.md` component but fail when fs-safe creates its sibling temp file.

## Findings

### Medium: Long synthesis titles still exceed the fs-safe temp filename budget

- Kind: bug
- File: `extensions/memory-wiki/src/markdown.ts`
- Line: 112
- Evidence: `MAX_WIKI_SAFE_WRITE_FILENAME_COMPONENT_BYTES` budgets for `.${fileName}.fallback.tmp`, but `slugifyWikiSegment` still allows a 240-byte slug. `applyCreateSynthesisMutation` then writes `syntheses/${slug}.md` through fs-safe at `extensions/memory-wiki/src/apply.ts:185`, producing a 243-byte final filename and a 257-byte fallback temp filename. A focused live check using the same `slugifyWikiSegment` plus `openclaw/plugin-sdk/security-runtime` `root.write` produced `slugBytes: 240`, `fileNameBytes: 243`, `fallbackTempBytes: 257`, then failed instead of writing.
- Impact: `wiki_apply` / `create_synthesis` can still crash for long ASCII titles, which is the exact class of failure this commit is trying to prevent for safe writes.
- Suggested fix: route synthesis page creation through `createWikiPageFilename`, or lower/rework the shared slug cap so every generated page filename plus every fs-safe sibling temp suffix fits the path-component limit. Also consider budgeting the documented Python helper path: with `OPENCLAW_FS_SAFE_PYTHON_MODE=require`, `createWikiPageFilename` generated a 241-byte filename whose Python-helper temp component was 259 bytes and failed with `File name too long`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The focused write check reproduced the default fs-safe failure from the generated slug filename. A second check with `OPENCLAW_FS_SAFE_PYTHON_MODE=require` reproduced the same class for `createWikiPageFilename`.
- Is this the best way to solve the issue? yes directionally: the cap needs to be applied at the actual filename-producing call sites and should be derived from the longest fs-safe temp-name strategy that OpenClaw supports.
