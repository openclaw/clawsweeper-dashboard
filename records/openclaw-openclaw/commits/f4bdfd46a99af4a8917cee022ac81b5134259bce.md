# Commit f4bdfd46a9

## Summary

Found one actionable security regression in the OpenClaw Crabbox wrapper: local-container runs now receive host Docker socket pass-through by default.

## Findings

### Medium: local-container Crabbox runs now mount the host Docker daemon by default

- Kind: security
- File: `scripts/crabbox-wrapper.mjs`
- Line: 433
- Evidence: The wrapper builds `childEnv` from `process.env` and, for `local-container` / `docker` / `container` / `local-docker`, sets `CRABBOX_LOCAL_CONTAINER_DOCKER_SOCKET=1` unless the env var or `--local-container-docker-socket` option is already present. A focused fake-Crabbox probe confirmed `node scripts/crabbox-wrapper.mjs run --provider local-container -- echo ok` passes `dockerSocket:"1"` and `workRoot:"/tmp/openclaw-crabbox-docker-work"` to the child process. Upstream Crabbox documents `localContainer.dockerSocket: false`, says socket pass-through is opt-in, and says Crabbox normally does not mount the Docker socket by default. ([github.com](https://github.com/openclaw/crabbox/blob/main/docs/providers/local-container.md))
- Impact: Any OpenClaw `pnpm crabbox:run -- --provider local-container ...` command now gives code inside the lease access to the host Docker daemon unless the caller knows to override the wrapper. That weakens the isolation boundary for local smoke runs, especially when running untrusted PR code or generated test commands. Docker’s own hardening docs treat Docker socket mounts as sensitive because they allow malicious access to Docker Engine, and Docker daemon access can allow non-root users to gain root access on the host. ([docs.docker.com](https://docs.docker.com/security/for-admins/hardened-desktop/enhanced-container-isolation/config/))
- Suggested fix: Keep the wrapper default off and require an explicit opt-in for Docker E2E lanes, for example `OPENCLAW_CRABBOX_ENABLE_DOCKER_SOCKET=1` or command-specific wiring in the Docker test scripts. If the project wants this default for all OpenClaw local-container runs, document the privilege change prominently and add wrapper tests for default enable, explicit disable, and custom work-root behavior.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: run the wrapper with a fake `crabbox` on `PATH` and inspect the child environment for `CRABBOX_LOCAL_CONTAINER_DOCKER_SOCKET=1`.
- Is this the best way to solve the issue? yes: the least risky fix is to move socket pass-through back to explicit opt-in and scope it to Docker-in-Docker proof lanes.
