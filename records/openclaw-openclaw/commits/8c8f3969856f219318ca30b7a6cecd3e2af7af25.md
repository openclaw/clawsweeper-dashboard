# Commit 8c8f396

## Summary

Found one lifecycle race in the new late-final suppression state. The focused test passes, but the guard is armed only after an async card close finishes, while the shared dispatcher does not await `onIdle`.

## Findings

### Medium: Slow streaming close can still let late finals create duplicate cards

- Kind: concurrency
- File: `extensions/feishu/src/reply-dispatcher.ts`
- Line: 377
- Evidence: `closeStreaming()` sets `streamingClosedForReply = true` only after `await streaming.close(...)` returns (`extensions/feishu/src/reply-dispatcher.ts:377-385`). But the shared dispatcher calls `options.onIdle?.()` without awaiting it (`src/auto-reply/reply/reply-dispatcher.ts:279-283`), and Feishu’s `onIdle` is async (`extensions/feishu/src/reply-dispatcher.ts:582-584`). During `FeishuStreamingSession.close()`, the session flips `closed = true` before awaiting queued/network close work (`extensions/feishu/src/streaming-card.ts:431-437`), making `streaming?.isActive()` false (`extensions/feishu/src/streaming-card.ts:487-489`) while `streamingClosedForReply` is still false. A final delivered in that window reaches the normal card fallback path instead of the new suppression guard.
- Impact: On a slow Feishu close/update request, the original symptom can still happen: a distinct late `final` arriving just after idle close starts can open/send another card before `streamingClosedForReply` is armed.
- Suggested fix: Mark a “closing/closed for reply” state before awaiting `streaming.close()`, then clear it if close fails and recovery should be allowed. Alternatively make the shared dispatcher await async `onIdle`/`onError` handlers, but that is a broader contract change.
- Confidence: high
