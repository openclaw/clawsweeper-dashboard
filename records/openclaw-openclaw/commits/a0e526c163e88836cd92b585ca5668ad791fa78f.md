# Commit a0e526c

## Summary

Found one UI session-state regression: skipping `sessions.list` reloads for lifecycle `sessions.changed` events leaves the dynamic `hasActiveRun` flag stale because those lifecycle event payloads do not include it.

## Findings

### Medium: Chat lifecycle events no longer refresh `hasActiveRun`

- Kind: regression
- File: `ui/src/ui/app-gateway.ts`
- Line: 759
- Evidence: The commit now returns early for chat-turn `sessions.changed` payloads after only `applySessionsChangedEvent(...)`, so `loadSessions(...)` is skipped. `applySessionsChangedEvent` preserves existing row fields unless the event includes that field (`ui/src/ui/controllers/sessions.ts:264` through `ui/src/ui/controllers/sessions.ts:280`). `hasActiveRun` is computed dynamically by `sessions.list` (`src/gateway/server-methods/sessions.ts:287` through `src/gateway/server-methods/sessions.ts:292`), but lifecycle `sessions.changed` payloads from chat only include status/timing/session metadata and omit `hasActiveRun` (`src/gateway/server-chat.ts:233` through `src/gateway/server-chat.ts:281`). The chat controls depend on `session.hasActiveRun === true` when there is no local `chatRunId` (`ui/src/ui/app-chat.ts:93` through `ui/src/ui/app-chat.ts:106`).
- Impact: A Control UI that is attached to an already-loaded session but did not start the run locally can miss the transition to abortable when a lifecycle `phase: "start"` event arrives, because the row remains `hasActiveRun: false`. Conversely, if the row was loaded while a remote run was active, terminal `phase: "end"`/`"error"` events can leave `hasActiveRun: true`, keeping the Stop control visible after completion. Before this commit, those lifecycle events fell through to `loadSessions`, which recomputed the flag.
- Suggested fix: Either include `hasActiveRun` on lifecycle `sessions.changed` payloads using the same active-run source as `sessions.list`, or have the UI patcher explicitly derive it for lifecycle start/terminal events before taking the no-reload fast path. The server-side payload is safer if multiple active runs per session are possible.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. I ran the actual `applySessionsChangedEvent` with a row containing `hasActiveRun: true` and a terminal lifecycle-style payload without `hasActiveRun`; it returned `{ applied: true, change: "updated" }` and left the row as `{"hasActiveRun":true,"status":"done"}`.
- Is this the best way to solve the issue? no. Avoiding reloads is reasonable, but the fast path needs parity for dynamic fields that only `sessions.list` used to compute.
