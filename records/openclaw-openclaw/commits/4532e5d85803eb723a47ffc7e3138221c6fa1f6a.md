# Commit 4532e5d858

## Summary

Found one low-severity UI state bug in the recovered Stop flow. The main abort path works, but a recovered `hasActiveRun: true` can stay true after the run finishes, leaving Stop visible when no abortable run remains.

## Findings

### Low: Recovered Stop state can remain stuck on after the run finishes

- Kind: bug
- File: `ui/src/ui/app-chat.ts`
- Line: 93
- Evidence: `renderApp` now drives Stop from `hasAbortableSessionRun(state)` in `ui/src/ui/app-render.ts:2376`, and `hasAbortableSessionRun` returns true from `sessionsResult.sessions[].hasActiveRun` when `chatRunId` is already null. A terminal chat event clears only local run state in `ui/src/ui/controllers/chat.ts:755` through `ui/src/ui/controllers/chat.ts:771`; it does not clear or refresh the recovered `sessionsResult` flag. The transcript/session broadcast helper in `src/gateway/server-session-events.ts:21` through `src/gateway/server-session-events.ts:83` also omits `hasActiveRun`, so normal `session.message` / `sessions.changed` updates do not overwrite the stale true value.
- Impact: In the fixed scenario, reload/reconnect during a long run, `sessions.list` can correctly recover `hasActiveRun: true`. If that run then finishes normally before the user clicks Stop, the UI can keep showing Stop. Clicking it sends a no-op session-scoped `chat.abort`, and it can still clear the draft via `handleAbortChat`.
- Suggested fix: After terminal `chat` events for the active session, refresh sessions or update the matching session row from authoritative state. A server-side post-cleanup `sessions.changed` with fresh `hasActiveRun` would be strongest; if fixed client-side, avoid clearing blindly when multiple active runs for the same session are possible.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused TS check against current main kept `hasAbortableSessionRun` true before and after a synthetic final event for the same session: `before true`, `after true`.
- Is this the best way to solve the issue? unclear. The safest fix is to make the active-run flag authoritative after cleanup, either by reloading `sessions.list` on terminal chat events or by broadcasting a fresh post-cleanup session update.
