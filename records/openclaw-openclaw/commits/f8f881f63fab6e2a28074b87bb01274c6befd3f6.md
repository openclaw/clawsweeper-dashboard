# Commit f8f881f

## Summary

Found one concrete gap in the fix. The new systemd source metadata is threaded through doctor/configure/onboard paths, but the primary `openclaw gateway install --force` path still drops it. That can move file-owned provider secrets out of `gateway.systemd.env` and back into inline systemd `Environment=` entries during forced install or update service refresh.

## Findings

### Medium: Forced gateway install drops EnvironmentFile source metadata

- Kind: privacy
- File: `src/cli/daemon-cli/install.ts`
- Line: 209
- Evidence: `runDaemonInstall` reads `existingServiceCommand?.environment`, but does not pass `existingServiceCommand?.environmentValueSources` into `buildGatewayInstallPlan`, and then destructures/passes only `environment` to `service.install` at `src/cli/daemon-cli/install.ts:232`. The new preservation model depends on those sources: `buildGatewayInstallPlan` defaults preserved existing vars to `"inline"` when `existingEnvironmentValueSources` is absent (`src/commands/daemon-install-helpers.ts:445`), and `writeSystemdUnit` uses `environmentValueSources` to decide which keys must stay file-owned versus inline-managed (`src/daemon/systemd.ts:585`). With sources missing, file-backed provider secrets preserved from the current unit are treated as inline, removed from the env file merge, and rendered as inline `Environment=` entries.
- Impact: `openclaw gateway install --force` and the update refresh path that shells out to `gateway install --force` can undo the commit’s intended secret-placement fix for systemd installs. The secret value is not necessarily lost, but it is moved from the owner-only `gateway.systemd.env` file into the systemd unit metadata, contrary to the issue fix and docs.
- Suggested fix: Thread `existingServiceCommand?.environmentValueSources` into `buildGatewayInstallPlan`, destructure `environmentValueSources`, and pass it to `service.install`. Apply the same pattern to `src/cli/daemon-cli/start-repair.ts`, which repairs loaded services from `params.state.command` and has the same omission.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. Source path: install a systemd service whose `readCommand()` returns an `OPENROUTER_API_KEY` from `EnvironmentFile` with `environmentValueSources.OPENROUTER_API_KEY = "file"`, then run `openclaw gateway install --force`. The current caller drops the source map before planning/installing, so systemd cannot know the key should remain file-owned.
- Is this the best way to solve the issue? Yes. The commit already introduced the right generic contract; this is a caller propagation bug, not a systemd merge-algorithm problem.
