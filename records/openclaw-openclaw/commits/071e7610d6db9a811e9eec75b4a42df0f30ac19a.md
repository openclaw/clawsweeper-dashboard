# Commit 071e761

## Summary

Found one regression: the new hardcoded Telegram Bot API timers can override the existing user-configured Telegram API timeout.

## Findings

### Medium: outbound Telegram request caps ignore `channels.telegram.timeoutSeconds`

- Kind: regression
- File: `extensions/telegram/src/request-timeouts.ts`
- Line: 13
- Evidence: `extensions/telegram/src/request-timeouts.ts:13` through `extensions/telegram/src/request-timeouts.ts:21` now hardcode 10-30s caps for outbound methods like `sendmessage`, `sendphoto`, `sendvideo`, and `sendvoice`. `extensions/telegram/src/bot-core.ts:195` through `extensions/telegram/src/bot-core.ts:238` applies that timer in the fetch wrapper before the request reaches grammY’s own timeout handling. The existing configured `timeoutSeconds` is still only passed to grammY later at `extensions/telegram/src/bot-core.ts:295` through `extensions/telegram/src/bot-core.ts:305`. Docs and config describe `channels.telegram.timeoutSeconds` as the Telegram API client timeout override in `docs/channels/telegram.md:729`, `extensions/telegram/src/config-ui-hints.ts:92`, and `src/config/types.telegram.ts:153`.
- Impact: operators who set `channels.telegram.timeoutSeconds` above these new caps for slow proxies, WSL2 networking, self-hosted Bot API servers, or larger media uploads will still get aborts at 20s for text sends and 30s for media sends. That breaks a documented/configured escape hatch that previously flowed to grammY.
- Suggested fix: make the per-method caps defaults only when `timeoutSeconds` is unset, or otherwise compute an effective timeout that explicitly honors the configured Telegram API timeout contract.
- Confidence: high
