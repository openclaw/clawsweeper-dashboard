# Commit 46c99cff

## Summary

One low-severity display bug found in the new CLI status runtime column. Aggregate recent session rows can show the wrong runtime for legacy, unprefixed session keys in non-default agent stores.

## Findings

### Low: Aggregate status rows lose the store agent id before resolving runtime

- Kind: bug
- File: `src/commands/status.summary.ts`
- Line: 272
- Evidence: `byAgent` rows call `buildSessionRows(store, { agentIdOverride: agent.id })`, but aggregate `sessions.recent` rebuilds rows from only the unique store path with `buildSessionRows(loadStore(storePath))`. For unprefixed legacy keys, line 196 leaves `agentId` undefined, and `resolveSessionRuntimeLabel()` then resolves metadata as the default/main agent. A temp-state smoke check with a `research` agent configured for `codex` and an unprefixed `legacy-chat` row produced `byAgent.research.runtime = "OpenAI Codex"` but aggregate `sessions.recent[0].runtime = "OpenClaw Pi Default"`.
- Impact: `openclaw status` renders `summary.sessions.recent`, so the new `Runtime` column can mislead users about which harness/runtime a recent session uses when older/non-agent-prefixed session keys live in a per-agent store.
- Suggested fix: preserve a `storePath -> agentId` association for unique per-agent stores and pass that `agentIdOverride` when building aggregate rows. If multiple agents share one store path, keep the current parse-only fallback for ambiguous unprefixed rows.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: create a temp `OPENCLAW_STATE_DIR` with `agents/research/sessions/sessions.json` containing an unprefixed session, configure `research.agentRuntime.id = "codex"`, then call `getStatusSummary({ includeChannelSummary: false })`; aggregate and by-agent runtime labels diverge.
- Is this the best way to solve the issue? yes: the runtime resolver is fine when it receives the correct agent id; the bug is the aggregate rebuild dropping the store target context.
