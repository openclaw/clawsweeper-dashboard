# Commit 4332037

## Summary

Found one supported-configuration gap in the new Ollama cloud watchdog detection.

## Findings

### Medium: Ollama `:cloud` watchdog still disables for custom non-`ollama*` provider IDs

- Kind: reliability
- File: `src/agents/pi-embedded-runner/run/llm-idle-timeout.ts`
- Line: 101
- Evidence: `isOllamaCloudModel()` rejects any model whose `provider` is present and does not start with `"ollama"`. The resolved runtime model also carries `api`, and the Ollama docs explicitly support custom provider IDs that set `api: "ollama"` and use their own model refs. A config like provider `"gpu-box"` with `api: "ollama"`, `baseUrl: "http://127.0.0.1:11434"`, and model `"kimi-k2.5:cloud"` therefore still takes the local-provider branch and returns `0`, disabling the idle watchdog.
- Impact: users with custom Ollama provider names that do not begin with `ollama` can still hit the original silent stall/failover delay this commit is meant to fix for cloud models routed through a local signed-in daemon.
- Suggested fix: pass/read `model.api` in `resolveLlmIdleTimeoutMs()` and treat `api === "ollama"` plus `id.endsWith(":cloud")` as the authoritative Ollama cloud signal; keep provider-name matching only as a fallback if needed.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the resolver path: call `resolveLlmIdleTimeoutMs({ model: { provider: "gpu-box", api: "ollama", id: "kimi-k2.5:cloud", baseUrl: "http://127.0.0.1:11434" } })`; current code has no `api` field in the resolver input and returns `0`.
- Is this the best way to solve the issue? Likely yes. The runtime model already carries `api`, so using it avoids coupling behavior to provider naming while preserving local non-Ollama behavior.
