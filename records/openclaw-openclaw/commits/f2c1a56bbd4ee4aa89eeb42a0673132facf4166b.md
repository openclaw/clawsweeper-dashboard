# Commit f2c1a56

## Summary

Found one low-severity compatibility issue in the new `end-active-conference` path: canonical `spaces/{id}` inputs are forced through an unnecessary `spaces.get` call, which can require a broader OAuth scope than Google requires for ending the active conference.

## Findings

### Low: `spaces/{id}` end-conference path unnecessarily requires the read scope

- Kind: compatibility
- File: `extensions/google-meet/src/meet.ts`
- Line: 422
- Evidence: `endGoogleMeetActiveConference` always calls `fetchGoogleMeetSpace` before posting `:endActiveConference`. That helper performs `GET /v2/spaces/...` and reports `meetings.space.readonly` as the required scope on failure at `extensions/google-meet/src/meet.ts:352`. But Google documents `spaces.endActiveConference` as `POST https://meet.googleapis.com/v2/{name=spaces/*}:endActiveConference`, with `name` already being the required `spaces/{space}` resource name, an empty request body, and only the `meetings.space.created` OAuth scope. 
- Impact: A caller with a valid `meetings.space.created` token, or one following the docs added at `docs/plugins/google-meet.md:750`, can pass a canonical `spaces/{id}` value and still fail before the end call because OpenClaw first performs a read-scope `spaces.get`. Normal `googlemeet auth login` asks for both scopes, so this mainly affects scoped access-token overrides and least-privilege integrations.
- Suggested fix: When the input is already an explicit `spaces/{id}` resource name, call `endActiveConference` directly. Keep the existing `spaces.get` resolution for Meet URLs and meeting codes, and document that URL/code resolution needs `meetings.space.readonly`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: invoke `googlemeet end-active-conference spaces/<space-id> --access-token <token-with-meetings.space.created-only>`. The code attempts `spaces.get` first, so it can fail before reaching the API call that Google says only needs `meetings.space.created`.
- Is this the best way to solve the issue? yes: branching only for explicit resource names preserves URL/code convenience while avoiding unnecessary scope widening for callers who already have the canonical space resource.
