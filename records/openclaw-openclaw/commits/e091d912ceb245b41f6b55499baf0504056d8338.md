# Commit e091d91

## Summary

Found one low-severity UX regression in the new allowlist repair guidance: retry commands drop supported auth-profile suffixes.

## Findings

### Low: blocked model retry guidance drops auth profile overrides

- Kind: bug
- File: `src/auto-reply/reply/model-selection-directive.ts`
- Line: 32
- Evidence: `extractModelDirective` parses `/model openai/gpt-5.5@work --runtime codex` into `rawModel: "openai/gpt-5.5"`, `rawProfile: "work"`, and `rawRuntime: "codex"` in `src/auto-reply/model.ts:39`. The not-allowed error path only receives `rawRuntime` and rebuilds the retry as `/model ${modelRef} --runtime ${rawRuntime}` in `src/auto-reply/reply/model-selection-directive.ts:32`, so the generated retry omits `@work`.
- Impact: A user who intentionally selected an auth profile on a blocked model is told to retry a different command after adding the model to the allowlist. Following the guidance persists the model/runtime switch without the profile override, which can use the wrong auth account or fail when only that selected profile has usable credentials.
- Suggested fix: Pass the parsed profile, or the original normalized selection token, into `formatNotAllowedError`. Keep the additive allowlist command using the bare `provider/model`, but build the retry as `/model provider/model@profile --runtime runtime` when `rawModelProfile` is present. Add a blocked-model test covering profile plus runtime.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused parser check showed the profile is parsed separately, and the formatted blocked-model error prints `Then retry: /model openai/gpt-5.5 --runtime codex` with no profile suffix.
- Is this the best way to solve the issue? yes. The allowlist key should remain the bare model ref, while only the retry command needs to preserve the user’s profile-bearing selection.
