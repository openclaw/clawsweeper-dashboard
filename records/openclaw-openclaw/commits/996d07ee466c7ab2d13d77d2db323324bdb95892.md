# Commit 996d07e

## Summary

Found two actionable regressions in the Telegram topic cache change. The production-impacting one is that the cache now requests 2,048 plugin-state rows even though plugin state enforces a 1,000 live-row cap per plugin, so forum topic handling can start throwing once enough topic names are cached.

## Findings

### Medium: Topic cache can crash forum handling after 1,000 plugin-state rows

- Kind: reliability
- File: `extensions/telegram/src/topic-name-cache.ts`
- Line: 80
- Evidence: `openTopicNamePersistentStore()` opens a keyed store with `maxEntries: MAX_ENTRIES`, where `MAX_ENTRIES` is 2,048. `updateTopicName()` writes each topic as a plugin-state row at `extensions/telegram/src/topic-name-cache.ts:193`. The plugin-state backend hard-caps all live rows for the `telegram` plugin at 1,000 in `src/plugin-state/plugin-state-store.sqlite.ts:19` and throws in `src/plugin-state/plugin-state-store.sqlite.ts:424` when exceeded. A focused script using real `createPluginStateKeyedStore("telegram", ...)` and calling `updateTopicName()` for 1,001 distinct topics reproduced: `PluginStateStoreError: Plugin state for telegram exceeds the 1000 live row limit.`
- Impact: once Telegram accumulates 1,001 topic-cache rows, or fewer if other Telegram plugin-state namespaces already have rows, `persistentStore.register()` throws. `buildTelegramMessageContext()` awaits `updateTopicName()` / `getTopicName()` in the forum path at `extensions/telegram/src/bot-message-context.ts:207` and `extensions/telegram/src/bot-message-context.ts:210`, so an optional topic-name cache failure can drop forum message processing. The previous file-backed cache allowed 2,048 entries without this plugin-wide row ceiling.
- Suggested fix: keep the topic cache within the plugin-state contract before writing, or change the storage shape/contract so 2,048 topics is actually supported. At minimum, do not let plugin-state write failures abort inbound message handling; topic names are metadata and should degrade to missing `TopicName`.
- Confidence: high

### Medium: Forum message-context tests now fail without an installed Telegram runtime

- Kind: regression
- File: `extensions/telegram/src/topic-name-cache.ts`
- Line: 80
- Evidence: the new cache opens persistence through `getTelegramRuntime().state.openKeyedStore(...)`. Existing forum message-context tests build contexts through the test harness without setting `setTelegramRuntime()`, and the focused test run failed 5 forum-topic tests with `Error: Telegram runtime not initialized` from `src/plugin-sdk/runtime-store.ts:92`.
- Impact: the affected Telegram test file is currently red on `main` for forum-topic paths, which blocks reliable CI signal and hides whether the behavior change works outside the narrow cache unit test.
- Suggested fix: install a minimal runtime/keyed-store in the message-context test harness or pass a topic-cache persistence seam through the message-context path so these tests do not depend on global plugin runtime setup.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The row-cap failure reproduces with a one-off `tsx` script that sets a Telegram runtime backed by `createPluginStateKeyedStore` and writes 1,001 distinct topic names. The test regression reproduces with the focused Vitest wrapper command below.
- Is this the best way to solve the issue? unclear. Lowering the topic-cache limit avoids the immediate impossible contract, but a robust fix should also make topic-cache persistence nonfatal because topic names are optional inbound context metadata.
