# Commit e2f4aa4

## Summary

Found one regression in the new `env -S` detector: it now depends on a stricter shared `env` parser, so valid GNU `env` modifier forms can hide a split-string carrier from approval explanations.

## Findings

### Medium: `env -S` risk detection misses valid split-string forms after optional `env` modifiers

- Kind: security
- File: `src/infra/command-analysis/risks.ts`
- Line: 173
- Evidence: `detectEnvSplitStringFlag` now returns early unless `parseEnvInvocationPrelude(argv)` returns `splitArgv`. The parser treats `--block-signal`, `--default-signal`, and `--ignore-signal` as value-consuming options when they have no `=SIG`, and does not model `--debug`; GNU `env` accepts those as modifiers before `-S`. Focused checks showed `env --block-signal -S 'printf %s ok'`, `env --default-signal -S 'printf %s ok'`, `env --ignore-signal -S 'printf %s ok'`, and `env --debug -S 'printf %s ok'` all execute the split payload, while `detectCommandCarrierArgv(["env","--block-signal","-S","sh -c id"])` returns `[]`. `explainShellCommand("env --block-signal -S 'sh -c id'")` also returns no risks.
- Impact: an approval request can omit the command-carrier warning for a real `env -S` split payload, so reviewers may not see that a shell wrapper or hidden command is being carried through `env`. The parent implementation’s broad scan would have still reported `-S` for this form.
- Suggested fix: either fix `parseEnvInvocationPrelude` to match GNU `env` option semantics for optional signal arguments and `--debug`, or make `detectEnvSplitStringFlag` scan option tokens independently enough to find `-S` before the command boundary without requiring full `splitArgv` resolution.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: run `node --import tsx -e` against `detectCommandCarrierArgv(["env","--block-signal","-S","sh -c id"])` on current `main`; it returns no hit, while local GNU `env --block-signal -S 'printf %s ok'` executes successfully.
- Is this the best way to solve the issue? yes: the detector should use an `env` option parser whose command-boundary behavior matches real `env`, otherwise this fix trades one missed carrier form for another.
