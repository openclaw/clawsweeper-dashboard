# Commit 39cbe94

## Summary

The commit fixes the normal Teams message handler path, but the same plugin still has an adjacent feedback-invoke path that appends Teams thread suffixes with the generic helper and can still produce `:thread:OLD:thread:NEW` session keys.

## Findings

### Medium: Teams feedback invokes still compound pre-suffixed route session keys

- Kind: bug
- File: `extensions/msteams/src/monitor-handler.ts`
- Line: 227
- Evidence: The feedback invoke handler computes `feedbackThreadId`, then calls `resolveThreadSessionKeys({ baseSessionKey: route.sessionKey, threadId: feedbackThreadId, parentSessionKey: route.sessionKey })` directly at `extensions/msteams/src/monitor-handler.ts:223`. That bypasses the new `resolveMSTeamsRouteSessionKey` normalization added in this commit. The generic helper still blindly appends at `src/routing/session-key.ts:251`, and a focused check returned `agent:main:msteams:channel:19:channel@thread.tacv2:thread:old-root:thread:new-root` for a pre-suffixed base. The pre-suffixed base is still reachable because `message-handler.ts` continues mutating `route.sessionKey` after `resolveAgentRoute`, while `src/routing/resolve-route.ts:689` stores and returns the same cache-miss route object.
- Impact: Teams thumbs-up/down feedback and negative-feedback reflection can land in a malformed transcript/reflection session instead of the intended per-thread transcript, so the bug fixed for normal messages remains for a user-visible Teams feedback workflow.
- Suggested fix: Route feedback invokes through `resolveMSTeamsRouteSessionKey` too, passing `isChannel`, `conversationMessageId`, and `replyToId`, or extract/reuse the clean-base normalization before any Teams thread suffix is appended. Add a channel-thread feedback regression test with a pre-suffixed `route.sessionKey`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Source-level reproduction is direct: import `resolveThreadSessionKeys`, pass a base ending in `:thread:old-root` plus `threadId: "new-root"`, and it emits `:thread:old-root:thread:new-root`. The feedback handler uses that exact helper shape.
- Is this the best way to solve the issue? yes. The new Teams-specific helper is already the local fix boundary for this bug; the feedback handler comment says it should match the message handler’s thread-aware session key, so reusing that helper is the narrowest consistent fix.
