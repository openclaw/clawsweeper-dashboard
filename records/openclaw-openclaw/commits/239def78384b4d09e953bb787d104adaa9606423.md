# Commit 239def7

## Summary

The commit adds a process-local cache for hydrated `resolvedSkills`. The cache key covers workspace, snapshot version, filter, agent, remote eligibility, and redacted config, but it still misses local runtime eligibility inputs used by skill filtering.

## Findings

### Medium: cached skill hydration can ignore env and PATH eligibility changes

- Kind: regression
- File: `src/auto-reply/reply/session-updates.ts`
- Line: 278
- Evidence: `snapshotCacheKey` is built from workspace/version/filter/agent/remote eligibility/config only (`src/auto-reply/reply/session-updates.ts:277`). But `shouldIncludeSkill` also checks `process.env[envName]` for `requires.env` (`src/agents/skills/config.ts:97`) and local binary availability through `hasBinary` for `requires.bins` / `requires.anyBins` (`src/shared/config-eval.ts:66`, `src/shared/config-eval.ts:79`). Before this commit, hydrating a stripped snapshot called `buildWorkspaceSkillSnapshot`; after this commit, `cachedRebuild` can return the prior `resolvedSkills` without re-evaluating those runtime inputs.
- Impact: In a long-lived gateway, adding/removing an env var or changing PATH/PATHEXT can leave skills incorrectly absent or present across warm turns until another snapshot-version/config/filter key change evicts the cache. That can advertise unusable skills or hide newly eligible ones.
- Suggested fix: include a redacted runtime eligibility fingerprint in the cache boundary, at least env-var presence and PATH/PATHEXT state, or skip this cache for snapshots whose loaded skills include `requires.env`, `requires.bins`, or `requires.anyBins`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: create a persisted stripped snapshot for a workspace skill gated by `requires.env`, call `ensureSkillSnapshot` with the env var absent to populate the cache, then set `process.env` and call again with the same workspace/version/filter/config. The current key is unchanged, so `cachedRebuild` returns stale `resolvedSkills`.
- Is this the best way to solve the issue? yes: the cache is reasonable, but its reuse boundary needs to match all inputs used by `buildWorkspaceSkillSnapshot` eligibility.
