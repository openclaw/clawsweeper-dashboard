# Commit a40499b

## Summary

Found one low-severity regression in the test/E2E helper surface: the new generated auth-profile key is not preserved by the shared E2E state-env snapshot helper, so fresh shell steps can lose access to OAuth profile secrets written by the original test-state process.

## Findings

### Low: E2E state snapshots drop the generated auth-profile secret key

- Kind: regression
- File: `scripts/lib/openclaw-e2e-instance.sh`
- Line: 80
- Evidence: This commit adds `OPENCLAW_AUTH_PROFILE_SECRET_KEY` to generated test state in `scripts/lib/openclaw-test-state.mjs:260`, `scripts/lib/openclaw-test-state.mjs:307`, `scripts/lib/openclaw-test-state.mjs:356`, and `scripts/lib/openclaw-test-state.mjs:400`. Auth-profile persistence reads that env key first and uses it for OAuth secret encryption/decryption in `src/agents/auth-profiles/persisted.ts:344` and `src/agents/auth-profiles/persisted.ts:369`. But `openclaw_e2e_write_state_env` writes only HOME/config/agent paths and omits `OPENCLAW_AUTH_PROFILE_SECRET_KEY`; current Docker E2E fresh-shell consumers source that snapshot in `scripts/e2e/browser-cdp-snapshot-docker.sh:87` and `scripts/e2e/config-reload-source-docker.sh:24`.
- Impact: A process started from the generated state can persist OAuth profile secrets with the new env key, while a later fresh shell that sources `/tmp/openclaw-test-state-env` falls back to another key source and cannot decrypt those secrets. I reproduced this: after `openclaw_e2e_write_state_env`, a fresh `env -i bash` had `OPENCLAW_AUTH_PROFILE_SECRET_KEY` unset; loading an OAuth profile saved by the original process returned only `oauthRef` and no `access`/`refresh`. Re-exporting the key from `.openclaw-test-auth-profile-secret-key` made the same load return both tokens.
- Suggested fix: Include `OPENCLAW_AUTH_PROFILE_SECRET_KEY` in `openclaw_e2e_write_state_env` when set, or have it export the key from `$HOME/.openclaw-test-auth-profile-secret-key`. Add a focused regression check for source-snapshot/fresh-shell auth-profile decryption.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes; the focused shell/`tsx` probe above demonstrates the missing env key and the resulting OAuth secret decrypt miss.
- Is this the best way to solve the issue? yes; the state-env helper is explicitly a fresh-shell snapshot, so it should carry the same secret seed as the generated test state.
