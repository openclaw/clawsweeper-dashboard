# Commit 47c020b

## Summary

Found two low-severity follow-up issues in the new cron TTS delivery path. The main TTS transform is wired into cron announce delivery and the focused tests pass, but audio-only TTS and replay behavior still have edge-case regressions.

## Findings

### Low: Audio-only TTS cron deliveries skip main-session awareness

- Kind: regression
- File: `src/cron/isolated-agent/delivery-dispatch.ts`
- Line: 388
- Evidence: `queueCronAwarenessSystemEvent` uses `pickLastNonEmptyTextFromPayloads(params.deliveryPayloads)` whenever transformed delivery payloads are passed, and only falls back to `outputText` / `synthesizedText` when `deliveryPayloads` is absent. The new caller passes `payloadsForDelivery` after TTS transformation at line 699. For a valid audio-only directive such as `[[tts:text]]Morning briefing complete[[/tts:text]]`, the TTS directive parser removes the visible text and `maybeApplyTtsToPayload` returns a media payload with `spokenText`, not `text`; the awareness picker ignores `spokenText`, so it returns before enqueueing anything.
- Impact: isolated cron runs that successfully deliver voice-only TTS can disappear from the main session’s awareness queue, even though delivered isolated cron text is an existing tested contract.
- Suggested fix: when queueing cron awareness from transformed payloads, fall back to `spokenText` for audio-only TTS payloads, or fall back to the cleaned/original `outputText` / `synthesizedText` when transformed payloads contain media but no visible text.
- Confidence: medium

### Low: Replay cache is checked only after TTS synthesis side effects

- Kind: reliability
- File: `src/cron/isolated-agent/delivery-dispatch.ts`
- Line: 621
- Evidence: `deliverViaDirect` now calls `maybeApplyTtsToCronPayloads(...)` before `getCompletedDirectCronDelivery(deliveryIdempotencyKey)`. The cache is meant to make same-run replay idempotent, but for a cached TTS cron execution the replay still runs the TTS transform first, which can call external TTS providers and create temp audio files before discovering that delivery already succeeded.
- Impact: duplicate dispatch of the same cron execution does not duplicate the chat send, but it can repeat paid/slow TTS work and update TTS runtime state unnecessarily. If the transform throws before the cache check, a replay of an already delivered cron run can be reported as a delivery error instead of using the cached success.
- Suggested fix: check the completed direct-delivery cache after raw payload normalization/stale checks and before TTS transformation. The idempotency key does not depend on transformed payload content.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the replay ordering from the static call path; no live TTS provider was used. The awareness issue is a static path through `[[tts:text]]` hidden speech, transformed payloads, and the existing awareness picker.
- Is this the best way to solve the issue? Unclear. Both suggested fixes preserve the commit’s main design while tightening the new edge cases.
