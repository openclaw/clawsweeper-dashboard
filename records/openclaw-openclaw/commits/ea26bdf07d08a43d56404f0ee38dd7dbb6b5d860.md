# Commit ea26bdf

## Summary

Found one low-severity contract-test gap introduced by removing package runtime dependency assertions without replacing them with a root-owned-dependency-safe assertion.

## Findings

### Low: Root-owned plugin runtime deps no longer have manifest coverage

- Kind: test_gap
- File: `src/plugins/contracts/package-manifest.contract.test.ts`
- Line: 17
- Evidence: The commit removes `pluginLocalRuntimeDeps` entries for root-owned runtime deps such as `@google/genai`, Bedrock SDK packages, `minimatch`, Slack SDK packages, Telegram grammY packages, and `express` for `msteams`. The existing helper only supports “plugin has it and root does not” assertions, so removing these entries drops package-manifest coverage entirely for those deps. `pnpm deps:root-ownership:check` passed, but its audit classifies `@google/genai` as `mixed_noncore` because it is also used by `scripts/dev/realtime-talk-live-smoke.ts`, and classifies `express` as root-owned because `browser` internalizes it, even though `msteams` is excluded from the core package and imports `express` in `extensions/msteams/src/monitor.ts`.
- Impact: A future package metadata edit could remove required runtime deps from an owning plugin manifest while the focused checks still pass. For published/externalized plugins, that can ship an installable plugin missing dependencies even though docs state runtime deps belong in the owning plugin package.
- Suggested fix: Add a separate contract helper parameter for runtime deps that must be declared by the plugin, without requiring absence from root `package.json`, and use it for intentionally root-owned/internalized deps.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes by static path: remove one of the affected manifest deps, then `pnpm deps:root-ownership:check` can still pass for mixed or multi-owner cases such as `@google/genai` or `msteams` `express`. I did not mutate the repo to prove this live because this is a report-only review.
- Is this the best way to solve the issue? Unclear; a new helper for “declared by plugin, root ownership allowed” seems safer than reusing `pluginLocalRuntimeDeps`, whose current root-absence assertion is intentionally incompatible with internalized deps.
