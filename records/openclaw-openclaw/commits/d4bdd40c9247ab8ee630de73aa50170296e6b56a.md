# Commit d4bdd40c

## Summary

Found one remaining regression in the issue path this commit is meant to supersede. The new lookup is safer than the raw-binding PR because it uses `resolveFirstBoundAccountId`, but it is still scoped to the caller’s resolved agent id. The CLI path from `#66807` still runs message sends as the default agent, so a target bound to another agent’s Slack account is not selected.

## Findings

### Medium: CLI sends still cannot select the target peer’s account across agents

- Kind: bug
- File: `src/infra/outbound/message-action-runner.ts`
- Line: 354
- Evidence: `resolveTargetBoundAccountId` calls `resolveFirstBoundAccountId` with `agentId: params.agentId`, so it only considers bindings owned by the resolved agent. The CLI message command still invokes `runMessageAction` with `agentId: resolveDefaultAgentId(cfg)` in `src/commands/message.ts:74`. In the `#66807` repro, `openclaw message send --channel slack --target C_CONTENT` runs under the default agent (`eng-hygiene`) while `C_CONTENT` is bound to agent `anisha`, so this commit’s resolver cannot see the `anisha` account binding. A focused resolver smoke with that config returned `{ "defaultAgentLookup": null, "targetAgentLookup": "anisha" }`.
- Impact: The reported multi-workspace Slack failure can remain: the CLI send falls through without `params.accountId`, so Slack delivery uses the channel default account instead of the account uniquely bound to the target peer. That can produce the original `channel_not_found` failure or send through the wrong bot identity when both bots can access the target.
- Suggested fix: Preserve the route-only and ambiguity-safe behavior, but before the agent-scoped fallback either select a unique route binding for `(channel, peer)` across agents, or make `openclaw message` inherit/accept the intended agent id so the existing agent-scoped resolver runs with `anisha` instead of the default agent.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. Use the two-agent/two-Slack-account config from `#66807`, with the default agent bound to `C_CODE` and agent `anisha` bound to `C_CONTENT`, then run `openclaw message send --channel slack --target C_CONTENT --message hi` without an explicit account. The code path still supplies the default agent id, and the new resolver cannot select the `anisha` binding.
- Is this the best way to solve the issue? Unclear. The commit correctly avoids the prior unsafe raw-binding scan, but the fix needs one more account-resolution step that is either unique-by-target across route bindings or backed by an inherited/explicit CLI agent id.
