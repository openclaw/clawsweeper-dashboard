# Commit c499ef1

## Summary

Found one low-severity reliability regression in stale lock reclamation. The intended EPERM behavior is sound, and the later zombie-owner gap was already fixed on current `main`, but oversized numeric PID payloads still fail closed indefinitely.

## Findings

### Low: Oversized PID payloads can permanently pin stale locks

- Kind: reliability
- File: `src/infra/stale-lock-file.ts`
- Line: 68
- Evidence: This commit replaced `!isPidAlive(pid)` with `isPidDefinitelyDead(pid)`. The new helper only pre-rejects non-integers and `pid <= 0`, then treats only `process.kill(pid, 0)` `ESRCH` as dead. Node throws `ERR_INVALID_ARG_TYPE` for values such as `2 ** 31` and `Number.MAX_SAFE_INTEGER`, so those invalid owners are treated as possibly live. The commit also changed the existing test dead PID from `Number.MAX_SAFE_INTEGER` to `-1`, so the regression is not covered. Current `main` still has the same behavior after the helper moved to `src/shared/pid-alive.ts:39`.
- Impact: A malformed or corrupted `.lock` payload like `{"pid":2147483648,"createdAt":"old"}` is never reclaimed, because `shouldRemoveDeadOwnerOrExpiredLock` takes the PID branch and ignores the expired timestamp. SDK file-lock users then time out and leave the lock in place until manual deletion; this can block auth store writes, OAuth refresh locks, config mutation locks, and plugin lock users.
- Suggested fix: Treat `process.kill` argument validation failures for integer PIDs outside Nodeâ€™s accepted PID range as definitely dead, or make `isValidPid` enforce the accepted upper bound before probing. Add a regression test using `2 ** 31` or `Number.MAX_SAFE_INTEGER`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A direct probe showed `isPidDefinitelyDead(2147483648) === false`, then `acquireFileLock` against an expired lock file with `pid: 2147483648` returned `file_lock_timeout` and left the stale lock contents intact.
- Is this the best way to solve the issue? yes. The EPERM preservation should stay narrow; invalid PID argument errors are not evidence of a live, non-signalable owner.
