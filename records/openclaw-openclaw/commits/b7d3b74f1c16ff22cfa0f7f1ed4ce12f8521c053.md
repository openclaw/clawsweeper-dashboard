# Commit b7d3b74

## Summary

Found one low-severity UI ordering regression in the new global chat item timestamp sort. The main reported history-vs-tool ordering fix works, but an existing live-stream invariant can now be reversed.

## Findings

### Low: Pre-tool streamed text can render below the tool card

- Kind: regression
- File: `ui/src/ui/chat/build-chat-items.ts`
- Line: 432
- Evidence: `ui/src/ui/app-tool-stream.ts:549` says in-progress assistant text is committed as a stream segment so it renders above the tool card. That segment is currently stamped with client `now` at `ui/src/ui/app-tool-stream.ts:557`, while the tool message uses the server event timestamp at `ui/src/ui/app-tool-stream.ts:568`. The new sort at `ui/src/ui/chat/build-chat-items.ts:283` and `ui/src/ui/chat/build-chat-items.ts:432` orders by timestamp, so a normal server tool event timestamp that is slightly earlier than the client-side segment timestamp puts the tool group first.
- Impact: During live runs, assistant text emitted before a tool call can appear after the tool card, making the turn read out of order and contradicting the existing stream/tool rendering contract.
- Suggested fix: When committing `host.chatStream` before the first tool, stamp the segment with `host.chatStreamStartedAt ?? now` instead of `now`, and add a regression test with one stream segment and one tool message where the tool event timestamp is earlier.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused `buildChatItems()` check with `streamSegments: [{ text: "assistant text before tool", ts: 2000 }]` and a tool message at `timestamp: 1000` renders the tool group before the stream.
- Is this the best way to solve the issue? yes, likely. The stream segment represents text that began before the tool call, and `chatStreamStartedAt` is already the available source timestamp for that visible item.
