# Commit 146ca95

## Summary

Found one test regression. Current `extensions/openshell/src/mirror.ts` still skips copied symlinks, so I did not find a current runtime bug. The commit does weaken the security-relevant symlink absence assertions enough that a future symlink-copy regression could pass.

## Findings

### Medium: Symlink absence assertions now follow dangling symlinks

- Kind: test_gap
- File: `extensions/openshell/src/mirror.test.ts`
- Line: 19
- Evidence: `expectPathMissing()` now uses `fs.access(targetPath)`. The converted symlink assertions at `extensions/openshell/src/mirror.test.ts:149`, `extensions/openshell/src/mirror.test.ts:150`, `extensions/openshell/src/mirror.test.ts:184`, and `extensions/openshell/src/mirror.test.ts:185` previously used `fs.lstat(...)`, which checks the directory entry itself. `fs.access(...)` follows symlinks, so a copied dangling symlink to `/tmp/host-secret` rejects with `ENOENT` even though the symlink entry exists.
- Impact: The tests named “skips symbolic links when copying into the host workspace” and “stages upload content without symbolic links” no longer prove the symlink entries are absent. A regression that copies dangling sandbox symlinks into the host mirror/staging directory could pass these tests, weakening coverage around a host/sandbox boundary.
- Suggested fix: Make the missing-path helper use `fs.lstat(targetPath)` with `code: "ENOENT"`, or add a separate `expectPathEntryMissing()` helper for symlink-sensitive assertions and use it for these four checks.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused Node check showed `fs.access()` rejects with `ENOENT` for a dangling symlink while `fs.lstat()` resolves and reports `isSymbolicLink() === true`.
- Is this the best way to solve the issue? yes. The tests intend to assert the filesystem entry is absent, and `lstat` is the right API because it does not follow symlinks.
