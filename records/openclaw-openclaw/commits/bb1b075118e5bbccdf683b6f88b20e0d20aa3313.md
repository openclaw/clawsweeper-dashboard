# Commit bb1b075

## Summary

Found two Telnyx realtime reliability regressions that still matter on current `main`. Later commits fixed the original Telnyx start-frame binding/parsing issues, but stream failure reporting and outbound token timing remain vulnerable.

## Findings

### Medium: Telnyx `streaming.failed` webhooks are silently dropped

- Kind: reliability
- File: `extensions/voice-call/src/providers/telnyx.ts`
- Line: 211
- Evidence: `normalizeEvent` only acknowledges `streaming.started` and `streaming.stopped`; any `streaming.failed` falls through to `default` and returns no event. Telnyx documents `streaming.started`, `streaming.stopped`, or `streaming.failed` as expected webhooks when `stream_url` is set on answer/dial. A focused parser check for `event_type: "streaming.failed"` returned `{"events":[],"statusCode":200}`. ([developers.telnyx.com](https://developers.telnyx.com/api-reference/call-commands/answer-call?utm_source=openai))
- Impact: if Telnyx cannot establish the realtime media stream, OpenClaw acknowledges the webhook but never emits `call.error` or tears down/marks the realtime call. The caller can remain in a connected call with no realtime bridge while operators have no normalized failure.
- Suggested fix: add `case "streaming.failed"` and map it to a non-retryable `call.error`, preserving `client_state`, `call_control_id`, and any provider reason/detail.
- Confidence: high

### Medium: Outbound Telnyx stream tokens can expire before the callee answers

- Kind: reliability
- File: `extensions/voice-call/src/webhook/realtime-handler.ts`
- Line: 41
- Evidence: outbound Telnyx calls issue the stream session before the Telnyx dial request (`extensions/voice-call/src/manager/outbound.ts:210`), and `issueStreamToken` expires it after 30 seconds (`extensions/voice-call/src/webhook/realtime-handler.ts:41`, `:516`). The Telnyx dial payload sets `timeout_secs: 30` (`extensions/voice-call/src/providers/telnyx.ts:265`), and Telnyx documents dial timeout default 30 seconds plus `stream_establish_before_call_originate` defaulting to false. A focused check showed `consumeStreamToken` returns `null` after advancing `Date.now()` by 31 seconds. ([developers.telnyx.com](https://developers.telnyx.com/api-reference/call-commands/dial?utm_source=openai))
- Impact: a callee who answers near the end of the 30-second ring window can trigger Telnyx’s WebSocket upgrade after OpenClaw has expired the only accepted token, causing the stream upgrade to get `401 Unauthorized` and the realtime bridge to never start.
- Suggested fix: make Telnyx outbound stream-token TTL exceed the configured dial timeout plus connection grace, or set `stream_establish_before_call_originate: true` and verify token-bound call matching still works for that pre-originate stream.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes for `streaming.failed`: parse a Telnyx webhook with `event_type: "streaming.failed"` and observe no normalized event. partial for token expiry: the token expiry path is deterministic, but a full carrier reproduction needs a Telnyx call answered near timeout.
- Is this the best way to solve the issue? yes for `streaming.failed`; unclear for token timing because either a longer Telnyx-specific TTL or pre-originate streaming could be correct, but the current 30s/30s boundary has no grace.
