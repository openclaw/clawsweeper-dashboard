# Commit cd15ce3

## Summary

Found one regression in the Mantis Telegram Desktop proof credential helper: the new import path points at `scripts/qa/...`, but the validator lives at repo-root `qa/...`. This breaks the credential script before it can lease or release the Telegram user credential.

## Findings

### Medium: Telegram user credential script imports the broker validator from the wrong relative path

- Kind: bug
- File: `scripts/e2e/telegram-user-credential.ts`
- Line: 6
- Evidence: `scripts/e2e/telegram-user-credential.ts:6` imports `../qa/convex-credential-broker/convex/payload-validation.js`. From `scripts/e2e`, that resolves to `scripts/qa/convex-credential-broker/convex/payload-validation.js`, but the commit tree only contains `qa/convex-credential-broker/convex/payload-validation.ts`; there is no `scripts/qa/...`. The proof runner calls this script through `node --import tsx` in `scripts/e2e/telegram-user-crabbox-proof.ts:1340`, so the module-resolution failure happens before `lease-restore` can acquire credentials.
- Impact: Mantis Telegram Desktop proof sessions cannot start because the credential lease/restore helper fails to load. The workflow still installs and invokes the trusted proof command, but the first credential operation will fail before Crabbox proof can run.
- Suggested fix: change the import to `../../qa/convex-credential-broker/convex/payload-validation.js` and update `test/scripts/mantis-telegram-desktop-proof-workflow.test.ts:203` to assert the executable path actually resolves or loads, not just that the intended string is present.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. With dependencies installed, run `node --import tsx scripts/e2e/telegram-user-credential.ts --help`; the ESM resolver will try `scripts/qa/convex-credential-broker/convex/payload-validation.js` before reaching usage output.
- Is this the best way to solve the issue? yes. The commit deliberately moved the parser out of `extensions/qa-lab/runtime-api.ts`, so the direct repo-root broker validator import is the right dependency direction; only the relative path is wrong.
