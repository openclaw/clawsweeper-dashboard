# Commit 83b8289

## Summary

I found two WhatsApp status-reaction lifecycle gaps. The main landed paths and prior ClawSweeper repair cases are covered by focused tests, but broadcast voice-note preflight and thrown dispatch errors can leave reactions without a final owner.

## Findings

### Medium: DM broadcast voice-note status reactions are orphaned after the queued preflight

- Kind: reliability
- File: `extensions/whatsapp/src/auto-reply/monitor/on-message.ts`
- Line: 319
- Evidence: For a direct audio message, `runAudioPreflightOnce()` creates a `statusReactionController` and sends `setQueued()` before STT when `messages.statusReactions.enabled` is true. If the DM peer is configured in `broadcast`, `on-message.ts` calls `maybeBroadcastMessage()` and passes only `ackAlreadySent: true` for that controller, then returns when broadcast handles the message. `broadcast.ts` cannot forward a `statusReactionController`, and each broadcast agent receives `ackAlreadySent: true`; `process-message.ts` then skips creating a controller because its creation condition is `statusReactions.enabled === true && !params.ackAlreadySent`.
- Impact: WhatsApp broadcast DMs are documented as supported. A broadcast voice note with status reactions enabled can leave the trigger message stuck at the queued emoji, with no thinking/tool/done/error transition and no `removeAckAfterReply` cleanup after all agents finish.
- Suggested fix: Let the broadcast path own a single shared status controller and finalize it once after all agents settle, or avoid sending the preflight status reaction before a broadcast path unless it can be finalized. Do not use `ackAlreadySent` to suppress per-agent lifecycle without another finalizer.
- Confidence: high

### Medium: Thrown WhatsApp dispatches skip status-reaction error cleanup

- Kind: reliability
- File: `extensions/whatsapp/src/auto-reply/monitor/inbound-dispatch.ts`
- Line: 499
- Evidence: WhatsApp sets the controller to thinking before `await dispatchReplyWithBufferedBlockDispatcher(...)`, but all calls to `finalizeWhatsAppStatusReaction()` happen after that await. If the dispatcher or delivery path throws, control leaves before lines 618-644 run. `process-message.ts` calls `runInboundReplyTurn()` without an adapter `onFinalize`, even though the shared kernel only runs failed-turn finalizers when an adapter provides one.
- Impact: Delivery/model/dispatcher failures can leave a WhatsApp status reaction in thinking/tool/stall state instead of transitioning to error or restoring/clearing the ack reaction. This is exactly the cleanup responsibility added by the new lifecycle feature.
- Suggested fix: Move WhatsApp status cleanup into the channel-turn `onFinalize` path, or wrap the dispatch await in `try/catch/finally` so failures set error and apply the configured restore/clear behavior before rethrowing.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. The broadcast issue follows directly from the direct-audio preflight branch plus the broadcast option type and `processMessage` controller gate. The dispatch-error issue follows from the await/finalize ordering and the missing `onFinalize` adapter hook.
- Is this the best way to solve the issue? Yes for direction: both fixes should give one owner responsibility for final status cleanup instead of letting lifecycle state escape through alternate control flow.
