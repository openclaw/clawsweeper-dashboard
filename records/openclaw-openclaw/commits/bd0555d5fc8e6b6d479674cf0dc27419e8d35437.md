# Commit bd0555d

## Summary

Found one migration gap: `openclaw doctor --fix` still skips a valid native Ollama config shape that omits `api`, even though the Ollama provider runtime later defaults that same explicit `models.providers.ollama.models` config to native `api: "ollama"`.

## Findings

### Medium: Doctor skips canonical native Ollama configs that omit `api`

- Kind: regression
- File: `src/commands/doctor/shared/legacy-config-core-normalizers.ts`
- Line: 1145
- Evidence: The new migration only treats provider-level budgets as native when `provider.api === "ollama"` and only treats model entries as native when `model.api` or `provider.api` is set to `"ollama"` (`src/commands/doctor/shared/legacy-config-core-normalizers.ts:1145`, `src/commands/doctor/shared/legacy-config-core-normalizers.ts:1153`). But the Ollama provider runtime defaults explicit canonical `models.providers.ollama.models` configs to native by returning `api: explicit.api ?? "ollama"` (`extensions/ollama/src/discovery-shared.ts:238`). Native requests then only send `num_ctx` from `params.num_ctx` (`extensions/ollama/src/stream.ts:281`, `extensions/ollama/src/stream.ts:307`). A focused probe against current main with `models.providers.ollama.contextWindow`, an explicit model `contextWindow`/`maxTokens`, and no `api` returned `changes: []` and left `params.num_ctx` unset.
- Impact: Users with older canonical Ollama configs that relied on `contextWindow`/`maxTokens` but did not spell out `api: "ollama"` remain broken after `openclaw doctor --fix`; their native `/api/chat` requests still omit `options.num_ctx` and can fall back to the small Ollama default the issue was meant to repair.
- Suggested fix: Match the migration’s native detection to the Ollama provider’s runtime defaulting. At minimum, treat provider id `ollama` with omitted `api` as native when the base URL is absent or not an OpenAI-compatible `/v1` URL, while preserving the existing skip for `/v1` configs. Add tests for omitted-`api` canonical native configs and omitted-`api` `/v1` configs.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Run `normalizeCompatibilityConfigValues` on a config with `models.providers.ollama.contextWindow`, an explicit `models` entry with `contextWindow`/`maxTokens`, and no `api`; it produces no changes. The Ollama discovery path then proves that same explicit canonical provider is native at runtime.
- Is this the best way to solve the issue? yes. The fix should broaden native detection only for config shapes the Ollama runtime already treats as native, and keep the `/v1` OpenAI-compatible exclusion covered by tests.
