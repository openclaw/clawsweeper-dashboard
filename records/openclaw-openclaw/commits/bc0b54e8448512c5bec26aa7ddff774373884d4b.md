# Commit bc0b54e

## Summary

Found one concrete compatibility miss: the new legacy shutdown aliases do not cover the actual `server-close` chunk shipped in `openclaw@2026.5.3-beta.3`, so users upgrading manually from that beta can still hit the same unclean-shutdown failure this commit is meant to fix.

## Findings

### High: Missing beta.3 `server-close` compatibility alias

- Kind: regression
- File: `scripts/runtime-postbuild.mjs`
- Line: 41
- Evidence: The commit writes aliases for `server-close-DsVPJDIx.js` and `server-close-DvAvfgr8.js`, but the published `openclaw@2026.5.3-beta.3` tarball contains `dist/server-close-D_I3Pd3d.js`, and its `dist/server.impl-CkBWAj3-.js` imports `./server-close-D_I3Pd3d.js`. A local invocation of `writeLegacyRootRuntimeCompatAliases` against a temp `dist/server-close.runtime.js` created the two committed aliases but left `server-close-D_I3Pd3d.js` missing.
- Impact: A running Gateway from `2026.5.3-beta.3` that is manually upgraded via package replacement can still fail graceful shutdown with `ERR_MODULE_NOT_FOUND` when it resolves the old shutdown chunk after the new package tree lands. That preserves the “cannot shut down cleanly until force-kill/launchd kickstart” failure mode for beta.3 users.
- Suggested fix: Add `["server-close-D_I3Pd3d.js", "server-close.runtime.js"]` to `LEGACY_ROOT_RUNTIME_COMPAT_ALIASES`, and extend `test/scripts/runtime-postbuild.test.ts` to assert that alias.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Inspect the published beta.3 package: `npm pack openclaw@2026.5.3-beta.3`, then search `package/dist` for `server-close`; it shows `server.impl-CkBWAj3-.js` importing `./server-close-D_I3Pd3d.js`. Then run the current alias writer with only `server-close.runtime.js` present; `server-close-D_I3Pd3d.js` is not generated.
- Is this the best way to solve the issue? yes. The stable runtime boundary fixes future versions, but already-published broken versions need exact legacy filename shims.
