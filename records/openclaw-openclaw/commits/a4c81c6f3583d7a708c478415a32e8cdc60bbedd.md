# Commit a4c81c6

## Summary

Found two actionable regressions in the timeout-recovery path. The main issue is that the recovery trusts the session’s last assistant message, which can be from a previous turn, so a plain prompt timeout in an existing session can be reported as a successful replay of stale assistant text.

## Findings

### High: Prompt timeouts can return the previous assistant answer as the current result

- Kind: regression
- File: `src/agents/pi-embedded-runner/run.ts`
- Line: 2675
- Evidence: The new recovery derives `finalAssistantVisibleText` and `finalAssistantStopReason` from `sessionLastAssistant` and treats `stop`/`end_turn`/`completed` text as success during `timedOutDuringPrompt`. But `run/attempt.ts:4311` sets `lastAssistant` to the last assistant anywhere in `messagesSnapshot`, while `currentAttemptAssistant` is separately sliced from `prePromptMessageCount` in `run/attempt.context-engine-helpers.ts:106`. If the current prompt times out before producing a new assistant message, an existing session can still have a prior `lastAssistant` with `stop` and final text.
- Impact: A new user turn that times out can return the previous turn’s final answer as a successful current response instead of the timeout error. That is a user-visible stale-response regression and can make downstream lifecycle metadata mark the timed-out attempt as `working`/success.
- Suggested fix: Gate recovery on the current attempt assistant, not the whole-session `lastAssistant`. For example, require `currentAttemptAssistant` to be present and to be the source of the recovered final text/stop reason, or carry an explicit current-turn completion flag from the harness.
- Confidence: high

### Medium: Synthesized recovery payloads discard already-built source-reply/media payloads

- Kind: bug
- File: `src/agents/pi-embedded-runner/run.ts`
- Line: 2791
- Evidence: When `recoveredFinalAssistantPayloadsAfterPromptTimeout` is set, `payloadsForTerminalPath` becomes only `[{ text: recoveredFinalAssistantTextAfterPromptTimeout }]`, replacing `payloadsWithToolMedia`. That can discard payload metadata and media produced by `buildEmbeddedRunPayloads`. In particular, `run/payloads.ts:271` suppresses normal assistant artifacts when source-reply payloads exist, and `run/payloads.test.ts:224` verifies those source replies must carry `deliverDespiteSourceReplySuppression` metadata. Dispatch only delivers under source suppression when that metadata is present (`dispatch-from-config.ts:2136`).
- Impact: In `message_tool_only` runs, a timeout race with final assistant text can drop the suppression-safe source-reply payload and return an unmarked raw assistant payload instead. Dispatch then suppresses that unmarked payload, so the intended source reply/transcript mirror can disappear. Similar replacement can lose parsed media/reply directive fields when the built payload text differs from the raw final text.
- Suggested fix: Do not replace non-empty `payloadsWithToolMedia` with a raw synthesized payload. Prefer the already-built payloads when they contain visible non-error content, and only synthesize a text payload when the normal projection is empty.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes. For the stale-answer case, use the existing runner harness with `timedOut: true`, no current-attempt assistant text, and a prior `lastAssistant` carrying `stop` plus text; the new branch will skip the timeout return and emit that stale text.
- Is this the best way to solve the issue? yes. The fix should keep the intended recovery but bind it to current-turn evidence and preserve the existing payload projection rather than reconstructing raw text.
