# Commit ad450a7

## Summary

The fast image-model path is covered by focused tests and the main event-loop fix looks sound, but the commit also threads agent `workspaceDir` into media attachment resolution. That still exists on current `main` and broadens local attachment reads for relative `MediaPath` values.

## Findings

### Medium: Agent workspace is accepted as an inbound media attachment root

- Kind: security
- File: `src/media-understanding/apply.ts`
- Line: 531
- Evidence: `applyMediaUnderstanding` now falls back from `ctx.MediaWorkspaceDir` to `params.workspaceDir`, then passes that as the attachment cache `workspaceDir`. `resolveMediaAttachmentLocalRoots` also falls back to `params.workspaceDir` and adds it to allowed roots in `src/media-understanding/runner.ts:282`. `MediaAttachmentCache` resolves relative attachment paths under its `workspaceDir` and permits reads when the resolved path is under `localPathRoots` in `src/media-understanding/attachments.cache.ts:311` and `src/media-understanding/attachments.cache.ts:318`. A no-edit cache probe using the same constructed options read `secret.jpg` from an agent workspace via a relative attachment path.
- Impact: A relative inbound `MediaPath` without `MediaWorkspaceDir` can be interpreted as a file inside the agent workspace and sent to an image/audio/video provider. That mixes provider/auth workspace scoping with media authorization and can expose workspace files through media-understanding if a channel/plugin/ACP path supplies or preserves a relative media path.
- Suggested fix: Keep `workspaceDir` for provider/auth/model resolution only. For attachment resolution, use only `ctx.MediaWorkspaceDir`, default media roots, and channel-configured inbound roots. Add a regression test that `MediaPath: "secret.jpg"` with no `MediaWorkspaceDir` does not read from `params.workspaceDir`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Construct a media context with a relative `MediaPath`, no `MediaWorkspaceDir`, and `workspaceDir` pointing at a directory containing that file; the current cache composition resolves and reads that workspace file.
- Is this the best way to solve the issue? yes. The fast image runtime should still receive `workspaceDir` for provider auth, but media path authorization should remain tied to staged media roots and channel media roots.
