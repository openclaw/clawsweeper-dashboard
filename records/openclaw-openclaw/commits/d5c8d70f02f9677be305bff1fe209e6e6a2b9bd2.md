# Commit d5c8d70

## Summary

Found one actionable supply-chain hardening regression in the publish job. The preview-pack half of the change is justified by the new publish script dependency path, but the same default install now also runs dependency lifecycle code inside the npm release job that has OIDC publishing authority.

## Findings

### Medium: Publish job now runs dependency lifecycle scripts with OIDC publishing permission

- Kind: supply_chain
- File: `.github/workflows/plugin-npm-release.yml`
- Line: 193
- Evidence: This commit removes `install-deps: "false"` from `publish_plugins_npm`, so `.github/actions/setup-node-env` uses its default `install-deps: "true"`. That action runs `pnpm install` with `--ignore-scripts=false` and `--config.enable-pre-post-scripts=true` at `.github/actions/setup-node-env/action.yml:91`, while the same job grants `id-token: write` at `.github/workflows/plugin-npm-release.yml:180`. Current `package.json` also has root lifecycle hooks and approved dependency build scripts (`package.json:1413`, `package.json:1709`). GitHub documents that jobs with this permission can request OIDC tokens via runner environment variables, and npm trusted publishing uses that OIDC capability to authorize publishes. 
- Impact: A compromised dependency lifecycle/build script, or an unintended install-time script in the dependency graph, now executes before the publish step in a job whose identity is trusted for npm publishing/provenance. That widens the release workflow’s package-publishing trust boundary beyond the checked publish script.
- Suggested fix: Keep dependency installation in `preview_plugin_pack`, but avoid running install scripts in `publish_plugins_npm`. Safer options are to publish a tarball/artifact produced by the unprivileged preview job from a tiny OIDC-only publish job, or run a minimal `pnpm install --ignore-scripts`/script-specific dependency setup before `npm publish`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. The diff makes the setup action’s default install run in `publish_plugins_npm`; GitHub run `25241140603` for this exact SHA shows the publish jobs executed `Setup Node environment` successfully before the `Publish` step.
- Is this the best way to solve the issue? No. Installing dependencies is needed for the pack preview path, but the privileged publish job should not need install-time lifecycle execution. Splitting pack/build from OIDC publish keeps the required functionality with a smaller release credential surface.
