# Commit 2d9ef76

## Summary

Found one concrete parser regression risk in the new batch response normalization: malformed-but-JSON batch rows can now be dropped, shifting later vectors onto earlier inputs.

## Findings

### Medium: Batch parser can shift embeddings after an invalid row

- Kind: data_loss
- File: `extensions/amazon-bedrock/embedding-provider.ts`
- Line: 249
- Evidence: `asNumberArrayBatch()` uses `value.filter(Array.isArray)`, and `parseCohereBatch()` returns that filtered list. For a response like `{"embeddings":[null,[0.25,0.75]]}`, the parser returns `[[0.25,0.75]]`. `manager-embedding-ops.ts` consumes `batchEmbeddings[i]` by positional index and writes it to the corresponding chunk, so the second vector can be cached/written for the first chunk.
- Impact: A malformed successful Bedrock/Cohere batch response can silently corrupt memory search embeddings instead of throwing or preserving indexes with an empty vector. The new malformed-JSON tests do not cover this shape-preservation case.
- Suggested fix: Preserve batch positions with `value.map(asNumberArray)` or reject invalid entries with a provider-owned shape error. Do not filter entries out of a provider batch response.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes at unit level: call `__testing.parseCohereBatch("cohere-v3", JSON.stringify({ embeddings: [null, [0.25, 0.75]] }))`; current code returns only the second vector. I also confirmed the JavaScript `filter(Array.isArray)` behavior with `node -e`.
- Is this the best way to solve the issue? Yes: preserving the input/output cardinality matches the provider batch contract and the downstream positional mapping. Throwing a stable shape error is also safe; silently shortening the array is the unsafe part.
