# Commit 20333bd

## Summary

Found one low-severity caching regression in the new Gemini time-filter path. The API payload shape is consistent with Google’s `GoogleSearch.timeRangeFilter` contract, but freshness and open-ended `date_after` calls now generate a moving timestamp before the cache lookup, making repeated identical searches miss the configured cache.

## Findings

### Low: Gemini freshness searches use a moving timestamp in the cache key

- Kind: reliability
- File: `extensions/google/src/gemini-web-search-provider.runtime.ts`
- Line: 281
- Evidence: `resolveGeminiTimeRangeFilter()` defaults `now = new Date()` and uses `now.toISOString()` for freshness `endTime` and `date_after`-only `endTime`. `executeGeminiSearch()` then includes `timeRange.timeRangeFilter?.startTime` and `?.endTime` in `buildSearchCacheKey()`. For `freshness: "week"`, two identical calls a few milliseconds apart produce different `startTime` and `endTime` values, so `readCachedSearchPayload()` cannot hit the existing 15-minute search cache.
- Impact: Repeated Gemini searches with `freshness` or open-ended `date_after` pay the provider/API cost and latency every time instead of using `tools.web.search.cacheTtlMinutes`. Other provider paths cache symbolic filters or date buckets rather than per-millisecond timestamps.
- Suggested fix: Keep the request `timeRangeFilter` as-is, but build the cache key from normalized user filters (`freshness`, `dateAfter`, `dateBefore`) or a coarse TTL/date bucket instead of the generated `now` timestamps.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Call Gemini `web_search` twice with the same `query` and `freshness` after a small delay; the generated timestamp components differ before the cache lookup, so the second call is not marked `cached: true`.
- Is this the best way to solve the issue? yes. The API still needs paired timestamps, but cache identity should represent the requested filter under the configured TTL rather than the exact instant the request was prepared.
