# Commit f82deaff

## Summary

Found one low-severity regression in explicit config unsets for old Gemini model-map keys. The new write-time normalizer can re-create the retired key under the canonical Gemini 3.1 key after the caller deleted it.

## Findings

### Low: Retired Gemini model-map deletes are persisted as canonical entries instead of being removed

- Kind: regression
- File: `src/config/io.write-prepare.ts`
- Line: 305
- Evidence: `resolvePersistCandidateForWrite()` now calls `normalizeAgentDefaultModelRefsForWrite()` before `writeConfigFile()` later applies `applyUnsetPathsForWrite()`. For an authored config containing `agents.defaults.models["google/gemini-3-pro-preview"]`, an unset path such as `["agents","defaults","models","google/gemini-3-pro-preview"]` removes the source key from `nextConfig`, but the merge patch is computed against the runtime-normalized canonical key. The projected source still carries the retired key, then line 305 normalizes the map to `google/gemini-3.1-pro-preview`. The later unset still targets the retired key, so it no longer matches anything.
- Impact: `openclaw config unset agents.defaults.models.google/gemini-3-pro-preview` on an older config can report a successful write while leaving the model entry present under `google/gemini-3.1-pro-preview`. That blocks users from removing stale aliases/params through the path that previously worked.
- Suggested fix: normalize relevant `unsetPaths` for `agents.defaults.models.<modelRef>` before the final `applyUnsetPathsForWrite()`, or pass unset paths into `normalizeAgentDefaultModelRefsForWrite()` and skip/re-delete normalized model refs whose retired source key was explicitly unset.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused `pnpm exec tsx -e` call to `resolvePersistCandidateForWrite()` plus `applyUnsetPathsForWrite()` with source key `google/gemini-3-pro-preview` and unset path for that same key produced `{"google/gemini-3.1-pro-preview": ...}` after the unset pass.
- Is this the best way to solve the issue? unclear. Normalizing writes is reasonable, but explicit delete paths need to be normalized or honored before/after this new map normalization.
