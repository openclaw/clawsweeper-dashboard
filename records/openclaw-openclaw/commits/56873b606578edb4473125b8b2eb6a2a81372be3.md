# Commit 56873b6065

## Summary

Found one medium-confidence reliability regression: healthy Telegram polling cycles can now be restarted while the runner is only backpressured by active long-running update handlers.

## Findings

### Medium: Watchdog can restart healthy backpressured Telegram handlers

- Kind: regression
- File: `extensions/telegram/src/polling-liveness.ts`
- Line: 72
- Evidence: This commit removes non-`getUpdates` API liveness tracking, and `detectStall` now returns a stall solely when elapsed `getUpdates` time exceeds the threshold. The session middleware now lets non-`getUpdates` calls pass through without updating liveness at `extensions/telegram/src/polling-session.ts:240`.
- Impact: `@grammyjs/runner` can legitimately stop fetching more updates while its sink is at capacity: its runner awaits `sink.handle(updates)` before the next fetch (`node_modules/@grammyjs/runner/out/runner.js:142`), and the concurrent sink waits for queue capacity (`node_modules/@grammyjs/runner/out/sink.js:82`, `node_modules/@grammyjs/runner/out/queue.js:113`). With `agents.defaults.maxConcurrent: 1`, or the default concurrency saturated by long turns, a >120s Telegram agent turn can have no completed `getUpdates` even though the handler is alive and sending Telegram API calls. The new test at `extensions/telegram/src/polling-session.test.ts:811` codifies restarting even after a `sendMessage` succeeds. Restarting during an unfinished handler is risky because Telegram update offsets are acknowledged after agent dispatch (`extensions/telegram/src/bot-core.ts:371`, `extensions/telegram/src/bot-update-tracker.ts:145`), so the replacement bot can start from a stale persisted offset and reprocess the same update while the old handler continues.
- Suggested fix: Distinguish a real polling/network stall from runner backpressure. Keep active stuck-`getUpdates` detection, but suppress idle “no completed getUpdates” restarts while update handling is at capacity or while handler-owned Telegram liveness is recent. A narrow fix would carry handler/backpressure liveness into `TelegramPollingLivenessTracker` instead of treating all unrelated API activity as polling health.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the code path: configure Telegram runner concurrency to 1, process an inbound update for longer than `pollingStallThresholdMs`, and emit a Telegram API call from that handler after the last `getUpdates`. The current tests simulate the latter condition and now expect a restart.
- Is this the best way to solve the issue? Unclear. The prior “any API call keeps polling alive” rule could mask a real wedged `getUpdates`; the safer direction is a more specific backpressure/active-handler signal.
