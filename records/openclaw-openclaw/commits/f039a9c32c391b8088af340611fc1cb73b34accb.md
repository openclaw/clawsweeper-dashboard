# Commit f039a9c

## Summary

Found one concrete bug: the new QQBot WebSocket helper creates a `ProxyAgent`, but `proxy-agent` does not route `wss://` through `HTTPS_PROXY` or `HTTP_PROXY` by default. That leaves the advertised ambient proxy path broken for the env vars named by the commit.

## Findings

### Medium: QQBot `wss://` gateway still bypasses `HTTPS_PROXY`/`HTTP_PROXY`

- Kind: bug
- File: `extensions/qqbot/src/engine/gateway/ws-client.ts`
- Line: 11
- Evidence: `createQQWSClient()` calls `resolveAmbientNodeProxyAgent()` and passes the returned `ProxyAgent` to `ws`. `proxy-agent@8.0.1` detects WebSocket upgrades as `wss:` URLs and delegates proxy selection to `proxy-from-env`; that package looks up `${proto}_proxy` or `all_proxy`, so `wss://...` checks `WSS_PROXY`/`wss_proxy` and `ALL_PROXY`, not `HTTPS_PROXY` or `HTTP_PROXY`. A focused live check with a local HTTP proxy and only `HTTPS_PROXY` set produced `WS_ERROR ENOTFOUND proxySeen=false`; the proxy never saw a `CONNECT`.
- Impact: Deployments using the documented/common `https_proxy`, `HTTPS_PROXY`, or `HTTP_PROXY` environment variables still connect to `wss://api.sgroup.qq.com/websocket` directly, so the PR does not fix private-network environments that need the QQ gateway WebSocket to go through an HTTP proxy.
- Suggested fix: Resolve the proxy URL for `wss:` using HTTPS semantics before constructing the agent, preserving `NO_PROXY` behavior by mapping the target to `https://...` for proxy matching. Then pass an explicit `HttpsProxyAgent`/compatible agent, or update `resolveAmbientNodeProxyAgent` to provide a custom `getProxyForUrl` that maps `wss -> https` and `ws -> http`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: start a local HTTP proxy, set only `HTTPS_PROXY=http://127.0.0.1:<port>`, then create `new WebSocket("wss://qq.example.test/websocket", { agent: new ProxyAgent() })`. The request fails with direct DNS lookup and the proxy receives no connection.
- Is this the best way to solve the issue? yes, directionally: the problem is dependency proxy-selection semantics, so the fix should explicitly map WebSocket schemes to HTTP(S) proxy env semantics rather than only asserting that an `agent` option exists.
