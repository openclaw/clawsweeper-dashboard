# Commit 06dfa6f

## Summary

Found one still-actionable lifecycle regression from the sticky `stopped` flag added by this commit. Current `main` has follow-up fixes for the no-binding cleanup contract, but the stopped state still never resets for a reusable approval runtime.

## Findings

### Medium: Native approval handler stays stopped after a stop/start lifecycle

- Kind: reliability
- File: `src/infra/approval-handler-runtime.ts`
- Line: 451
- Evidence: `createChannelApprovalHandlerFromCapability` stores `let stopped = false`, `onStopped` sets it to `true`, and `deliverTarget` returns `null` whenever it sees `stopped`. There is no corresponding reset on `start()`, while `ExecApprovalChannelRuntime` exposes reusable `start()`/`stop()` methods in `src/infra/exec-approval-channel-runtime.types.ts`.
- Impact: If the same native approval handler instance is stopped and later reused, future approval requests still call `deliverPending` but are treated as canceled before `bindPending`/active tracking. That silently drops native approval delivery after restart until a new handler object is created.
- Suggested fix: Replace the sticky boolean with a stop generation token captured per delivery. Increment the generation in `onStopped`, and only cancel a delivery when its captured generation is stale. That preserves the original in-flight cleanup without poisoning later lifecycle runs.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused `tsx` probe created a handler, delivered one request, called `stop()`, then delivered a second request on the same runtime. Output was `{"deliverCount":2,"bindCount":1,"unbindCount":1,"cancelCount":1}`, proving the second request was canceled before binding because `stopped` remained true.
- Is this the best way to solve the issue? yes. A generation token is safer than simply resetting the boolean, because it still lets old in-flight deliveries notice that their own lifecycle was stopped even if a later lifecycle has started.
