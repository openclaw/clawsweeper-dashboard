# Commit 5e05052

## Summary

Found one regression in the new LINE account-level validation: it rejects a named-account `dmPolicy: "open"` config even when the effective account config inherits `allowFrom: ["*"]` from `channels.line`, which current runtime resolution supports.

## Findings

### Medium: Named LINE accounts cannot inherit the wildcard allowlist for open DMs

- Kind: regression
- File: `extensions/line/src/config-schema.ts`
- Line: 54
- Evidence: `LineAccountConfigSchema` calls `requireChannelOpenAllowFrom` with only `value.allowFrom`, so account validation fails before considering the parent channel config. Current LINE runtime resolution merges parent channel fields into named accounts in `extensions/line/src/accounts.ts:115`, and the shared DM helper resolves account allowFrom before falling back to parent allowFrom in `src/channels/plugins/dm-access.ts:139`. A focused live check confirmed the mismatch: `{ channels: { line: { allowFrom: ["*"], accounts: { work: { dmPolicy: "open", ... } } } } }` parses with `success: false` at `accounts.work.allowFrom`, while `resolveLineAccount({ accountId: "work" }).config` has `dmPolicy: "open"` and `allowFrom: ["*"]`.
- Impact: Existing multi-account LINE configs that intentionally keep `allowFrom: ["*"]` at `channels.line` and set `channels.line.accounts.<id>.dmPolicy = "open"` were previously runtime-valid and are now rejected during config validation after upgrade. That can disable or block startup/config reload for a working LINE account.
- Suggested fix: validate named accounts against the effective allowlist, e.g. in `LineConfigSchema.superRefine` iterate `accounts` and use `account.allowFrom ?? value.allowFrom` with a path like `["accounts", accountId, "allowFrom"]`; remove or narrow the account-only refinement so it does not reject inherited parent wildcard configs.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The focused `pnpm exec tsx -e ...` check parsed a config with root `allowFrom: ["*"]` plus a named account `dmPolicy: "open"` and showed schema failure while runtime account resolution produced the inherited wildcard.
- Is this the best way to solve the issue? yes. The validation should match the effective LINE account config contract already used by `resolveLineAccount`; copying `"*"` into every named account is a setup-wizard workaround, not a safe assumption for manual or existing configs.
