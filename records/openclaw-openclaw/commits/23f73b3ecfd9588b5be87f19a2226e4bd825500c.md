# Commit 23f73b3

## Summary

I found two medium-severity incomplete-fix paths. The new tests cover the clean reset/orphan cases, but already-bad reset entries and recorded-but-dead ACPX wrapper leases can still preserve the production failure modes from #82364.

## Findings

### Medium: Reset keeps an already-stale generated transcript file

- Kind: bug
- File: `src/gateway/session-reset-service.ts`
- Line: 83
- Evidence: `resolveResetSessionFile` rewrites only when `currentEntry.sessionFile` is based on `currentEntry.sessionId`; otherwise it falls back to `currentEntry.sessionFile`. For a store already in the reported bad shape, `sessionId=B` with `sessionFile=A-topic-456.jsonl`, the next reset to `C` still resolves to `A-topic-456.jsonl` because `resolveSessionFilePath` accepts the explicit in-store file path before deriving `C-topic-456.jsonl`.
- Impact: Users who were already affected by the shipped reset bug can remain stuck on the unrelated/stale transcript even after upgrading and resetting again.
- Suggested fix: When rotation misses, distinguish true custom `sessionFile` overrides from generated transcript filenames. If the basename looks like a generated session transcript or topic transcript but does not match the current session id, derive a fresh path for `nextSessionId` instead of preserving it.
- Confidence: high

### Medium: Startup reaping misses orphan adapters when the recorded wrapper PID is gone

- Kind: reliability
- File: `extensions/acpx/src/service.ts`
- Line: 289
- Evidence: `reapOpenAcpxProcessLeases` calls `reapStaleOpenClawOwnedAcpxOrphans` only for open leases with `rootPid <= 0`. For leases with a recorded positive wrapper PID, it calls `cleanupOpenClawOwnedAcpxProcessTree`; when that PID is no longer visible, cleanup returns `skippedReason: "unverified-root"` and the adapter-orphan sweep is never run. A local source probe showed that exact shape leaves the adapter untouched by cleanup while the new orphan reaper would terminate it if invoked.
- Impact: If the wrapper PID was saved before the wrapper crashed and left a plugin-local `codex-acp` child reparented to PID 1, gateway startup marks the lease lost but does not reap the orphan process.
- Suggested fix: On positive-root cleanup failure with `unverified-root` or no terminated PIDs, run the wrapper-root orphan sweep before marking the lease lost.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes for the reset path: a focused source probe with `sessionId=B`, `sessionFile=A-topic-456.jsonl`, and `nextSessionId=C` resolved back to `A-topic-456.jsonl`. yes for the ACPX call-path gap by source and a focused `process-reaper` probe; I did not run a full service-level live ACP failed-initialize reproduction.
- Is this the best way to solve the issue? yes for the reset path: repair generated transcript naming at the reset boundary while preserving true explicit overrides. likely for ACPX: the narrowest fix is to invoke the already-added orphan sweep when recorded-root cleanup cannot verify the saved PID.
