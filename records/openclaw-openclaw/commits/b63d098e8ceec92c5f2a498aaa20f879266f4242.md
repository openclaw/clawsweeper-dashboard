# Commit b63d098

## Summary

Found one concrete retry-state bug in the new one-time configured-plugin install repair. The happy-path tests pass, but a failed repair can still be marked as completed when doctor writes the config for an unrelated repair.

## Findings

### Medium: Failed configured-plugin repair can be skipped forever after an unrelated config write

- Kind: reliability
- File: `src/flows/doctor-health-contributions.ts`
- Line: 303
- Evidence: The new health contribution only mutates `ctx.cfg.meta.lastTouchedVersion` when `maybeRunConfiguredPluginInstallReleaseStep()` reports `touchedConfig: true` (`src/flows/doctor-health-contributions.ts:297-306`), and the release step deliberately returns `touchedConfig: false` when install repair emits warnings (`src/commands/doctor/shared/release-configured-plugin-installs.ts:308-314`). However, the final doctor writer still writes whenever `ctx.configResult.shouldWriteConfig` is already true or any other contribution changed `ctx.cfg` (`src/flows/doctor-health-contributions.ts:543-556`), and config writes always stamp `meta.lastTouchedVersion` to the current `VERSION` (`src/config/io.ts:2057-2061`). On the next run, `shouldRunConfiguredPluginInstallReleaseStep()` skips once `touchedVersion >= 2026.5.2` (`src/commands/doctor/shared/release-configured-plugin-installs.ts:223-232`).
- Impact: If the configured-plugin install fails transiently, for example ClawHub/npm/network failure for a provider or channel plugin, and the same `doctor --fix` run also repairs any unrelated config item, the config is stamped as touched by `2026.5.2`. Future doctor runs then skip the one-time release repair, leaving the downloadable plugin missing unless the user installs it manually or another non-release repair path happens to cover that exact plugin.
- Suggested fix: Do not use the global `meta.lastTouchedVersion` as the sole completion marker for this repair. Persist a release-specific completion flag only after successful configured-plugin repair, or carry an explicit “do not mark this release repair complete” state through the final config write when warnings occurred.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. Create a config last touched before `2026.5.2` that triggers `maybeRunConfiguredPluginInstallReleaseStep()` warnings for a release-collected plugin, and also triggers any unrelated doctor config write. After `openclaw doctor --fix`, the config write path stamps `meta.lastTouchedVersion` to current, so a second doctor run skips the release step.
- Is this the best way to solve the issue? The safest direction is a release-specific completion marker. Trying to suppress the global config stamp for one doctor write would conflict with the existing config IO contract.
