# Commit efb1a7c

## Summary

Found two low-severity accessibility/input regressions in the new pointer/focus behavior.

## Findings

### Low: Secondary pointers can hijack or cancel an active divider drag

- Kind: bug
- File: `ui/src/ui/components/resizable-divider.ts`
- Line: 98
- Evidence: `handlePointerDown` stores `activePointerId`, but `handlePointerMove` and `handlePointerUp` process every document-level pointer event without checking `e.pointerId`. A second touch/pen/mouse pointer can move the divider using its own `clientX`, or stop the first pointer’s drag by firing `pointerup`/`pointercancel`.
- Impact: Touch resizing can jump or end unexpectedly during multi-touch/stray pointer input, which undercuts the new input-method agnostic behavior.
- Suggested fix: Store the initiating pointer id and ignore `pointermove`, `pointerup`, and `pointercancel` events whose `pointerId` does not match. Add a multi-pointer test.
- Confidence: high

### Low: Mobile sidebar exposes a focusable resize separator when there is no split view to resize

- Kind: regression
- File: `ui/src/ui/views/chat.ts`
- Line: 1069
- Evidence: The divider is rendered whenever `sidebarOpen` is true and now always gets `tabindex="0"` plus separator ARIA. Mobile CSS hides `.chat-main` in `ui/src/styles/chat/sidebar.css:454`, but does not hide or inert the `resizable-divider`, so mobile keyboard/screen-reader users can focus a resize control whose target pane is not visible.
- Impact: Adds a misleading focus stop on the mobile full-screen sidebar; keyboard resize can also persist split ratio changes with no visible feedback.
- Suggested fix: Hide/inert the divider in the mobile sidebar layout, or render it only when the split view is actually active.
- Confidence: high
