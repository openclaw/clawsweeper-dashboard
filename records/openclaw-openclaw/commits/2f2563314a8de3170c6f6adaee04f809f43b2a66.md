# Commit 2f25633

## Summary

Found two actionable regressions in the Codex migration apply path: the normal CLI apply path can leave the shared Codex app-server alive, and real missing-plugin failures are now downgraded to successful warning results.

## Findings

### High: `openclaw migrate codex --yes` can leave the target app-server running

- Kind: reliability
- File: `extensions/codex/src/migration/apply.ts`
- Line: 169
- Evidence: Codex plugin install requests now pass `isolated: false`, so `requestCodexAppServerJson` uses `getSharedCodexAppServerClient` instead of `createIsolatedCodexAppServerClient`. That helper only closes the child in its `finally` block when `params.isolated` is true (`extensions/codex/src/app-server/request.ts:48`, `extensions/codex/src/app-server/request.ts:60`). The new `prepareApply` disposer is only called from onboarding (`src/wizard/setup.post-install-migration.ts:200`, `src/wizard/setup.post-install-migration.ts:220`); `runMigrationApply` calls `params.provider.apply(ctx, selectedPlan)` directly and never invokes or disposes `provider.prepareApply` (`src/commands/migrate/apply.ts:97`, `src/commands/migrate/apply.ts:109`). The stdio transport is spawned with live pipes (`extensions/codex/src/app-server/transport-stdio.ts:100`), and those pipes are only unref’d during explicit close (`extensions/codex/src/app-server/transport.ts:27`).
- Impact: a normal `openclaw migrate codex --yes` run that installs selected native Codex plugins can finish the migration logic but keep the CLI process alive on the shared app-server child. The PR only fixed the post-install onboarding wrapper, not the standalone migrate command it tells users to rerun.
- Suggested fix: have the generic migrate apply runner call `provider.prepareApply(ctx)` and dispose it in a `finally`, or keep standalone target plugin activation isolated and reserve the shared prewarm path for the wizard-owned lifecycle.
- Confidence: high

### Medium: Missing curated plugins are reported as successful warnings

- Kind: regression
- File: `extensions/codex/src/migration/apply.ts`
- Line: 209
- Evidence: after polling, `requestTargetCodexAppServerJson` returns as soon as the `openai-curated` marketplace is present (`extensions/codex/src/migration/apply.ts:294`). At that point, `ensureCodexPluginActivation` returns `plugin_missing` when the selected plugin is absent from the loaded marketplace (`extensions/codex/src/app-server/plugin-activation.ts:63`). This commit converts both `plugin_missing` and `marketplace_missing` into `status: "warning"` (`extensions/codex/src/migration/apply.ts:209`), and `assertApplySucceeded` ignores warnings because it only fails on errors or conflicts (`src/commands/migrate/output.ts:261`).
- Impact: if a source-installed Codex plugin was renamed, removed, or no longer available in the target curated marketplace, migration exits successfully with generic rerun guidance instead of failing with the prior actionable missing-plugin error. That can silently omit a requested plugin config entry.
- Suggested fix: only downgrade the delayed-marketplace case to a warning when the final `plugin/list` response still lacks `openai-curated` or inventory timed out. Preserve `plugin_missing` as an error once the marketplace has loaded.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the missing-plugin downgrade by returning a target `plugin/list` response containing `openai-curated` without the selected plugin. The shared-client hang is established from the command lifecycle and transport cleanup path; local live reproduction was blocked by missing dev dependencies.
- Is this the best way to solve the issue? Yes: the fixes should preserve the new marketplace polling but put shared app-server lifetime under the actual migrate apply lifecycle, and keep real missing-plugin failures distinct from delayed marketplace availability.
