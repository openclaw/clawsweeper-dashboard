# Commit 68b5610

## Summary

Found one low-severity diagnostic regression: one CLI warning still directs users to `plugins doctor`, but this commit changed `plugins doctor` to hide that same healthy-shadowing state.

## Findings

### Low: shadowed install warning can point to a doctor report that now suppresses the condition

- Kind: bug
- File: `src/cli/plugins-install-persist.ts`
- Line: 93
- Evidence: `logShadowedNpmInstallWarning` warns for any active `origin === "config"` source outside the npm install path and prints `Run openclaw plugins doctor for repair options` without checking `active.status`. This commit changed `plugins doctor` to only include config-selected shadow diagnostics when the active config plugin is `status === "error"` (`src/cli/plugins-cli.ts:363`), and the new test asserts healthy config-selected shadowing reports `No plugin issues detected` (`src/cli/plugins-cli.list.test.ts:115`).
- Impact: if a user installs an npm plugin while a healthy config-selected source with the same id is still active, the install command warns that the npm install is shadowed and sends them to `openclaw plugins doctor`, but doctor now intentionally hides that case. The user loses the promised repair path for switching from the config-selected source to the installed npm source.
- Suggested fix: either only emit the install-time doctor guidance when `active.status === "error"`, or keep the healthy-shadow warning but include inline repair steps that do not depend on `plugins doctor`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: make `buildPluginSnapshotReport` return the installed plugin id with `origin: "config"`, `status: "loaded"`, and a source outside the npm `installPath`; `persistPluginInstall` will warn and point to doctor, while `plugins doctor` suppresses the healthy shadow diagnostic by design.
- Is this the best way to solve the issue? yes: the smallest fix is to align the install-time warning predicate or message with the new doctor predicate.
