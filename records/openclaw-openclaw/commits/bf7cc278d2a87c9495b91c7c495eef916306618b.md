# Commit bf7cc278

## Summary

Found one low-severity diagnostic regression. The new missing-registration hint fixes the reported custom-provider case, but it is returned before provider-owned unknown-model hints, so configured allowlist entries can hide more specific setup/auth guidance for providers such as Ollama and vLLM.

## Findings

### Low: Missing-registration hint can mask provider-specific setup hints

- Kind: regression
- File: `src/agents/pi-embedded-runner/model.ts`
- Line: 1231
- Evidence: `buildUnknownModelError` now returns `registrationHint` before calling `runtimeHooks.buildProviderUnknownModelHintWithPlugin`. The registration hint triggers for any exact `agents.defaults.models["provider/model"]` entry without a matching `models.providers[provider].models[]`, not just custom providers. Existing provider plugins already expose more specific unknown-model hints: Ollama tells users to set `OLLAMA_API_KEY` or run configure, and vLLM tells users to set `VLLM_API_KEY` or run configure. Existing tests also assert those auth hints for unknown Ollama/vLLM models, but only without an `agents.defaults.models` entry.
- Impact: A user who allowlists a local/self-hosted model, for example `agents.defaults.models["ollama/gemma3:4b"]`, can now get told to add `models.providers["ollama"].models[]` instead of seeing the provider-owned auth/setup hint. That sends them down the wrong recovery path for providers whose runtime registration depends on auth/setup/discovery rather than only a static provider model row.
- Suggested fix: Prefer provider-owned unknown-model hints when they exist, or append the registration hint after the provider hint instead of returning early. Add a regression test with `agents.defaults.models["ollama/..."]` or `agents.defaults.models["vllm/..."]` proving the auth/setup hint is preserved.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes by code path: with an exact `agents.defaults.models["ollama/<id>"]` entry and no matching provider model row, `buildMissingProviderModelRegistrationHint` returns a string, so the provider hook at `src/agents/pi-embedded-runner/model.ts:1235` is never called.
- Is this the best way to solve the issue? Unclear; the narrowest safe fix is to preserve provider-owned hints and use this new registration hint only when no provider-specific hint exists, or include both in a deterministic order.
