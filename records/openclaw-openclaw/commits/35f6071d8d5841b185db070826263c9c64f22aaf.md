# Commit 35f6071

## Summary

One low-severity config/runtime mismatch found. The commit adds Mattermost streaming config fields, including tool-progress disable knobs, but the Mattermost monitor still always writes tool-start text into the draft whenever preview streaming is not off.

## Findings

### Low: Mattermost accepts tool-progress disable config but still emits tool activity drafts

- Kind: bug
- File: `extensions/mattermost/src/mattermost/monitor.ts`
- Line: 1850
- Evidence: `extensions/mattermost/src/config-schema-core.ts:82` accepts `streaming.progress.toolProgress`, and `extensions/mattermost/src/config-schema-core.ts:90` accepts `streaming.preview.toolProgress`. The shared resolver in `src/plugin-sdk/channel-streaming.ts:153` gives those fields semantics, and docs say `streaming.preview.toolProgress: false` hides tool-progress lines. But the Mattermost dispatch path only checks `account.streamingMode !== "off"` when creating the draft stream and unconditionally runs `draftStream.update(buildMattermostToolStatusText(payload))` in `onToolStart` at `extensions/mattermost/src/mattermost/monitor.ts:1850`.
- Impact: A user can now provide valid config such as `channels.mattermost.streaming.preview.toolProgress: false` or `channels.mattermost.streaming.progress.toolProgress: false`, but Mattermost will still show tool activity like `Running \`lookup\`...` in the channel. This is user-visible config drift and can expose tool names/activity a user explicitly tried to suppress.
- Suggested fix: Resolve `resolveChannelStreamingPreviewToolProgress(account.config)` for Mattermost and guard `onToolStart`/progress draft updates with it. For `mode: "progress"`, use the shared progress helpers so `progress.toolProgress`, labels, and bounded progress lines are applied consistently.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes by code path: configure Mattermost with `streaming: { mode: "partial", preview: { toolProgress: false } }`, trigger a turn that starts a tool, and `onToolStart` still calls `draftStream.update(...)` with no config guard.
- Is this the best way to solve the issue? Yes, the existing shared `channel-streaming` helpers already encode the contract and are used by other channel implementations.
