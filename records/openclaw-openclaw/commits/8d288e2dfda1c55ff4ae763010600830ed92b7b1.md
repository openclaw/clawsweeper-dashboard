## Findings

### Medium: OpenAI web-search Docker E2E imports a removed export

- Kind: regression
- File: `scripts/e2e/lib/openai-web-search-minimal/assertions.mjs`
- Line: 7
- Evidence: `assertPatchBehavior()` dynamically imports `dist/extensions/openai/native-web-search.js` and destructures `patchOpenAINativeWebSearchPayload`, then calls it at lines 11 and 28. This commit changed `extensions/openai/native-web-search.ts:68` from `export function patchOpenAINativeWebSearchPayload` to a private `function`, and a focused esbuild transpile of current `main` showed the compiled module exports only `createOpenAINativeWebSearchWrapper`. `scripts/e2e/lib/openai-web-search-minimal/scenario.sh:57` runs this assertion before the fixture, mock server, or gateway startup.
- Impact: `pnpm test:docker:openai-web-search-minimal` will fail with `TypeError: patchOpenAINativeWebSearchPayload is not a function` in built/package acceptance contexts. That lane is wired through `package.json` and Docker E2E scenario lists, including package acceptance selection.
- Suggested fix: either keep `patchOpenAINativeWebSearchPayload` exported as an intentionally test-facing helper, or stop the E2E assertion from deep-importing the private module and validate the same behavior through the public wrapper/request path.
- Confidence: high

## Details

- Is this the best way to solve the issue? yes, if the assertion is still valuable as a preflight, re-export the helper. If the goal is truly to trim internal exports, move the assertion to a public-path check and remove the deep import.
