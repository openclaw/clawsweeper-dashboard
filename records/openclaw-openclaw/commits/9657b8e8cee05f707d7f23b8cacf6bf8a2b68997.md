# Commit 9657b8e

## Summary

One low-severity follow-up issue found: the commit makes multiple active image-generation tasks possible, but the `image_generate` status action still reports only one unfiltered active task.

## Findings

### Low: `image_generate` status is ambiguous once multiple image tasks can be active

- Kind: bug
- File: `src/agents/tools/image-generate-tool.actions.ts`
- Line: 80
- Evidence: `createImageGenerateStatusActionResult()` still uses `findActiveImageGenerationTaskForSession(sessionKey)` without prompt or task id filtering, while `findActiveSessionTask()` returns one matching active task, preferring a running task and otherwise the first match. A focused smoke with two active image tasks showed prompt-scoped duplicate guards resolved both tasks correctly, but unfiltered status returned only the newer `Prompt B` task.
- Impact: after this commit, an agent/user can have two distinct image tasks active, then ask for progress on the earlier one. `action:"status"` has no way to select it and may report the wrong image task, causing false progress/status messaging.
- Suggested fix: make image status return all active image tasks, or add an optional `prompt`/`taskId` status selector to the tool schema and status action. Cover two active image tasks where status can distinguish or enumerate both.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Create two running `image_generation` task records in one session, then call `createImageGenerateStatusActionResult(sessionKey)`; it returns one unfiltered task even though prompt-scoped duplicate guard can resolve each task.
- Is this the best way to solve the issue? unclear. Returning all active image tasks is the safest generic status behavior; a selector is useful too, but requires schema/docs updates.
