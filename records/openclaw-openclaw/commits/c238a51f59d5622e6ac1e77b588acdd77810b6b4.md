# Commit c238a51

## Summary

Found one concrete regression in the canonical Gemini model-key preservation logic. The new path normalization correctly moves legacy Gemini params in the common case, but it does that before honoring ownership/removal semantics tied to the original authored key.

## Findings

### Medium: Legacy Gemini params can bypass unset/include ownership during config writes

- Kind: data_loss
- File: `src/config/io.write-prepare.ts`
- Line: 232
- Evidence: `preserveAuthoredAgentParams` normalizes the source model id into `modelPath`, deletes the original `modelId` key when it differs, then calls `preserveSourceValueAtPath` only for the normalized path. That means `unsetPaths` and `$include` ownership attached to `agents.defaults.models["google/gemini-3-pro-preview"]` are not considered before the code resurrects the old params under `google/gemini-3.1-pro-preview`.
- Impact: A user trying to remove legacy Gemini per-model params can have them reappear under the canonical key. A more damaging case is a nested `$include` at the legacy model entry: an unrelated config write drops the `$include` reference and writes the resolved params into the root config, flattening modular config that the write-preparation layer is otherwise designed to preserve.
- Suggested fix: Before deleting or preserving under the normalized model path, check ownership and unset overlap for both the source path and normalized path. If the source legacy model entry or params path is include-owned, leave the preserved `$include` path alone instead of moving it. If the source legacy params path is explicitly unset, do not preserve those params under the normalized key.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused `pnpm exec tsx --eval` probe against `resolvePersistCandidateForWrite` showed `unsetPaths: [["agents","defaults","models","google/gemini-3-pro-preview","params"]]` still persisted `agents.defaults.models["google/gemini-3.1-pro-preview"].params`, and a root-authored nested `$include` at the legacy model key was replaced by an inline canonical model entry during an unrelated `gateway.port` write.
- Is this the best way to solve the issue? yes, directionally. The regression is local to the new normalized preservation path; preserving the existing include/unset contract requires checking the original authored path before moving the value.
