# Commit 58c4f9e

## Summary

Found one concrete bug: the Slack thread fallback added by this commit can still fail for resumed/gateway paths where the current Slack target is stored as `channel:C123`.

## Findings

### Medium: Prefixed Slack currentChannelId prevents ambient thread injection

- Kind: bug
- File: `src/agents/tools/message-tool.ts`
- Line: 848
- Evidence: The commit now derives `currentThreadTs` from `agentThreadId` and defaults `replyToMode` to `"all"` when a thread exists, then forwards `options.currentChannelId` unchanged into `toolContext`. Gateway run context can populate `currentChannelId` from `opts.to`, producing `channel:C123` (`src/agents/command/run-context.ts:49`). Slack auto-threading compares the parsed target’s bare id (`C123`) directly to `toolContext.currentChannelId`, so `channel:C123` does not match (`extensions/slack/src/action-threading.ts:25`). A focused probe confirmed `resolveSlackAutoThreadId({ to: "channel:C123", currentChannelId: "channel:C123", currentThreadTs: "111.222", replyToMode: "all" })` returns no thread id, while the same context with `currentChannelId: "C123"` returns `"111.222"`.
- Impact: The fix can be ineffective for the resumed Slack parent-send path it targets when the ambient target is stored in normalized/prefixed form. The send falls back to the channel root instead of the originating Slack thread.
- Suggested fix: Normalize Slack `currentChannelId` before passing it into Slack threading, or make `resolveSlackAutoThreadId` parse/normalize both the target and `toolContext.currentChannelId`. Add an integration-style test that runs through message action threading and asserts `params.threadId` is injected for `currentChannelId: "channel:C123"`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The focused `tsx` probes exercised `resolveAgentRunContext`, `resolveSlackAutoThreadId`, and `resolveAndApplyOutboundThreadId` and showed the prefixed target path leaves `threadId` unset.
- Is this the best way to solve the issue? yes. Normalizing at the Slack helper boundary is safest because Slack accepts both bare and prefixed target forms elsewhere.
