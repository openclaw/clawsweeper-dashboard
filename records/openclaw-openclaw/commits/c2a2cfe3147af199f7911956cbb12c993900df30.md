# Commit c2a2cfe

## Summary

Found one actionable protocol/codegen regression: the commit adds `WizardStep.format` to the gateway protocol schema but does not update the generated Swift protocol models that are checked by `pnpm protocol:check`.

## Findings

### Medium: Wizard protocol schema change leaves generated Swift models stale

- Kind: compatibility
- File: `src/gateway/protocol/schema/wizard.ts`
- Line: 69
- Evidence: `WizardStepSchema` now permits `format: "plain"` and `src/wizard/session.ts:73` emits that field for plain wizard output. The generated Swift `WizardStep` structs in `apps/macos/Sources/OpenClawProtocol/GatewayModels.swift:2341` and `apps/shared/OpenClawKit/Sources/OpenClawProtocol/GatewayModels.swift:2341` still list only `id`, `type`, `title`, `message`, `options`, `initialValue`, `placeholder`, `sensitive`, and `executor`. `package.json:1419` defines `pnpm protocol:check` as regenerating Swift protocol models and diffing those tracked files.
- Impact: The protocol check will report generated-model drift once run, and Swift clients cannot observe the new `format` marker, so macOS/OpenClawKit wizard surfaces cannot render the Slack manifest with the new plain-output semantics.
- Suggested fix: Regenerate and commit the Swift protocol models with `pnpm protocol:gen:swift` or `pnpm protocol:check`, then update Swift wizard rendering/CLI handling to honor `format == "plain"` if the plain-copy behavior is intended for RPC clients too.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. Static inspection shows the TypeBox schema and session payload include `format`, while both tracked generated Swift `WizardStep` structs omit it. The repo’s protocol docs and `package.json` script require generated models to stay in sync.
- Is this the best way to solve the issue? Yes. This is generated protocol drift; the narrow fix is to regenerate the Swift models and add client handling for the new field where needed.
