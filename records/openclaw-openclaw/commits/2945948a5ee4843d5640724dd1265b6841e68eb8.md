# Commit 2945948

## Summary

Found one concrete cancellation regression in the new `tasks.cancel` RPC path. The list/get/read surfaces otherwise matched the protocol and SDK tests I reviewed.

## Findings

### Medium: `tasks.cancel` bypasses the existing registered-runtime fallback

- Kind: regression
- File: `src/gateway/server-methods/tasks.ts`
- Line: 208
- Evidence: The new handler imports `cancelDetachedTaskRunById` from `src/tasks/detached-task-runtime.ts`, then returns that result directly. The detached runtime contract says registered runtimes should return `found: false` when they do not own a task so core can fall back, and the actual fallback implementation lives in `src/tasks/task-executor.ts:683` through `src/tasks/task-executor.ts:695`. That behavior is already locked by `src/tasks/task-executor.test.ts:715`.
- Impact: If any plugin registers a detached task runtime and declines ownership for a normal core task, `tasks.cancel` reports `{ found: false, cancelled: false }` and leaves the task running. Existing CLI/plugin cancellation through `task-executor` would fall back and cancel the same task.
- Suggested fix: Route the RPC through the `src/tasks/task-executor.ts` cancellation wrapper and extend its params to preserve the optional `reason`, or move the fallback behavior into the detached-runtime wrapper so every public cancellation path honors the contract.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. I ran a one-off `node --import tsx --eval` harness that created a running core CLI task, registered a detached runtime whose `cancelDetachedTaskRunById` returned `{ found: false, cancelled: false, reason: "not owned by runtime" }`, then invoked `tasksHandlers["tasks.cancel"]`. The RPC response was `{ found: false, cancelled: false, reason: "not owned by runtime" }` and `statusAfter` remained `"running"`.
- Is this the best way to solve the issue? yes. The fallback contract already exists and is tested in `task-executor`; the RPC should reuse that contract instead of calling the lower-level registered runtime dispatcher directly.
