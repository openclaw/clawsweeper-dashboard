# Commit 92d33e4

## Summary

The object-shaped presentation fields added in this commit are sanitized, but supported presentation inputs can still reach channel renderers with `<think>...</think>` content intact.

## Findings

### Medium: presentation reasoning sanitizer is bypassed for supported JSON-string and text-alias payloads

- Kind: privacy
- File: `src/agents/tools/message-tool.ts`
- Line: 767
- Evidence: `sanitizePresentationTextFields` returns any non-object unchanged at `src/agents/tools/message-tool.ts:68`, then `createMessageTool` forwards that value at `src/agents/tools/message-tool.ts:767`. `runMessageAction` later parses string `presentation` values at `src/infra/outbound/message-action-runner.ts:1002` through `parseJsonMessageParam` (`src/infra/outbound/message-action-params.ts:402`), and the existing test `src/infra/outbound/message-action-runner.plugin-dispatch.test.ts:1222` confirms JSON-string presentation is supported. A direct smoke showed `createMessageTool` forwards `presentation: "{\"title\":\"<think>internal</think>Visible\",...}"` unchanged. Separately, `normalizeButton` and `normalizeOption` accept `text` as a label alias at `src/interactive/payload.ts:131` and `src/interactive/payload.ts:153`, but the new sanitizer only strips `label`; a smoke with `{ buttons: [{ text: "<think>why</think>Approve" }] }` normalized into a visible label containing the reasoning tag.
- Impact: an agent can still leak hidden planning/reasoning into rich channel payloads when it supplies `presentation` as JSON text or uses the already-supported `text` aliases for button/select labels. That defeats the stated fix for visible presentation titles, blocks, buttons, and select labels.
- Suggested fix: parse/sanitize presentation after the same normalization/parsing boundary used by outbound sending, or move a recursive presentation-text sanitizer into the shared presentation normalization path. Include `text` label aliases for buttons/options, and add regression tests for JSON-string `presentation` plus button/option `text` aliases.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Use `createMessageTool` with a stub `runMessageAction` and pass `presentation` as a JSON string containing `<think>internal</think>` in `title` or block `text`; the captured params still contain the raw string. A second smoke with object-shaped button/option `text` aliases keeps `<think>why</think>` through the tool and normalizes it into visible labels.
- Is this the best way to solve the issue? yes. Sanitizing after presentation JSON parsing and before channel dispatch is safer than maintaining a partial pre-parse object walker in the agent tool, because it covers CLI/gateway-compatible payloads and all channel renderers consistently.
