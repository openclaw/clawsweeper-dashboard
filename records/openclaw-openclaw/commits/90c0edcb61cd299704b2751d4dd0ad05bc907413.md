# Commit 90c0edc

## Summary

Found one compatibility regression in the downloadable Mattermost plugin metadata.

## Findings

### Medium: Mattermost plugin can load on hosts that do not export the new SDK formatter

- Kind: compatibility
- File: `extensions/mattermost/src/mattermost/draft-stream.ts`
- Line: 2
- Evidence: This commit adds a static named import of `formatChannelProgressDraftLine` from `openclaw/plugin-sdk/channel-streaming`. The Mattermost package still advertises `openclaw.install.minHostVersion: ">=2026.4.10"` in `extensions/mattermost/package.json:40`, and the plugin install/load gates use that field, not the `peerDependencies.openclaw` range, for host compatibility checks (`src/plugins/install.ts:823`, `src/plugins/manifest-registry.ts:786`). `git show v2026.4.10:src/plugin-sdk/channel-streaming.ts` shows the subpath exists but does not export `formatChannelProgressDraftLine`; the base commit does.
- Impact: A user on an allowed `2026.4.10` host can install or keep loading the latest `@openclaw/mattermost` plugin, but module evaluation will fail when Node resolves the static named export against that older host SDK. That disables the Mattermost plugin instead of just missing the new progress-label behavior.
- Suggested fix: Raise `extensions/mattermost/package.json` `openclaw.install.minHostVersion` to the first OpenClaw release that contains `formatChannelProgressDraftLine` (likely matching the existing peer floor, `>=2026.5.3`), or avoid the new named export on older hosts via a compatible fallback.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Install/load this Mattermost plugin on an OpenClaw `2026.4.10` host, which the package metadata allows; the static named import resolves to a `channel-streaming` module without that export.
- Is this the best way to solve the issue? yes. The plugin intentionally relies on a newer SDK surface, so the install/load compatibility floor should move with that dependency unless the code adds a real fallback.
