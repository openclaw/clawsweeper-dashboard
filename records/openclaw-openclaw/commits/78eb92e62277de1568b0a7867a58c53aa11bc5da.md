# Commit 78eb92e

## Summary

The Codex app-server path looks covered, but the new internal UI source-reply contract is not wired through the non-Codex embedded tool telemetry path. There is also a schema gap for rich internal UI replies.

## Findings

### Medium: Internal UI source replies are dropped outside the Codex dynamic-tool bridge

- Kind: bug
- File: `src/agents/pi-embedded-subscribe.handlers.tools.ts`
- Line: 951
- Evidence: `runMessageAction` can now return an internal-source send payload with `sourceReplySink: "internal-ui"` and `sourceReply` for any message-tool run (`src/infra/outbound/message-action-runner.ts:636`). The Codex bridge extracts that payload into `messagingToolSourceReplyPayloads` (`extensions/codex/src/app-server/dynamic-tools.ts:285`), but the normal embedded subscription path only commits `messagingToolSentTexts`, `messagingToolSentTargets`, and `messagingToolSentMediaUrls` (`src/agents/pi-embedded-subscribe.handlers.tools.ts:951`). `run/attempt.ts` returns those old fields but never returns `messagingToolSourceReplyPayloads` (`src/agents/pi-embedded-runner/run/attempt.ts:4099`), while `buildEmbeddedRunPayloads` only creates suppression-safe visible UI payloads from `messagingToolSourceReplyPayloads` (`src/agents/pi-embedded-runner/run/payloads.ts:227`).
- Impact: A non-Codex harness or configured direct WebChat/TUI run with `sourceReplyDeliveryMode: "message_tool_only"` can get a successful targetless `message.send` result, but the reply text is not converted into the deliver-despite-suppression final payload and can still disappear from the active UI/history path.
- Suggested fix: Teach `pi-embedded-subscribe` to extract `sourceReplySink: "internal-ui"` results into a new `messagingToolSourceReplyPayloads` state field, return it from the subscription and `run/attempt.ts`, and add a focused non-Codex embedded-run regression test.
- Confidence: medium

### Low: WebChat/TUI message tool schema does not advertise rich presentation replies

- Kind: bug
- File: `src/agents/tools/message-tool.ts`
- Line: 638
- Evidence: For a current internal WebChat run, `resolveIncludeCapability` asks `channelSupportsMessageCapabilityForChannel` for `webchat`, but WebChat is not a plugin-backed message-action channel, so the generated `message` tool schema omits `presentation`. A focused schema probe with `createMessageTool({ currentChannelProvider: "webchat", sourceReplyDeliveryMode: "message_tool_only" })` showed only `action` and `message` fields relevant to sending, even though the new payload path supports presentation-only replies (`src/agents/pi-embedded-runner/run/payloads.ts:176`) and the changelog/docs describe rich/button-only internal replies.
- Impact: Codex is less likely to produce button-only or rich WebChat/TUI replies because the tool schema does not expose the supported `presentation` argument on the exact internal UI path this commit adds.
- Suggested fix: Treat the internal WebChat/TUI source sink as presentation-capable when building the message tool schema, and add a schema test for `currentChannelProvider: "webchat"` plus `message_tool_only`.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the schema gap via the focused `createMessageTool` schema probe; partly by source inspection for the non-Codex telemetry gap.
- Is this the best way to solve the issue? Yes: the fix should preserve the current Codex bridge behavior while adding the same source-reply payload extraction to the shared embedded subscription path.
