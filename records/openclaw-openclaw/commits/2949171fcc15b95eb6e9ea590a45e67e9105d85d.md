# Commit 2949171

## Summary

Found one shutdown race introduced by deferring gateway maintenance and cron startup until 250 ms after readiness.

## Findings

### Medium: Delayed maintenance can start after the gateway has already closed

- Kind: reliability
- File: `src/gateway/server.impl.ts`
- Line: 1493
- Evidence: The new `setTimeout(..., POST_READY_MAINTENANCE_DELAY_MS)` starts maintenance timers and cron later, but the timeout handle is local and never cleared. `close()` captures and stops the current `runtimeState.tickInterval`, `healthInterval`, `dedupeCleanup`, `mediaCleanup`, and cron at close time, then returns. If callers close the server within the 250 ms window, the delayed callback can still run afterward, assign fresh maintenance intervals, and call `startGatewayCronWithLogging` on an already closed gateway.
- Impact: Fast restart/shutdown paths and tests can leave cron or gateway maintenance loops running after shutdown, which can run scheduled work against closed runtime state or keep background activity alive unexpectedly.
- Suggested fix: Track the post-ready timeout and cancel it in `runClosePrelude`/close, and guard the callback with a closed flag before and after `await earlyRuntime.startMaintenance()`. If the callback races with close after creating handles, clear those handles immediately and skip cron startup.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes: start a non-minimal gateway, call `close()` before 250 ms elapses, then advance timers or wait past the delay. The expected failing assertion is that maintenance/cron start is invoked after `close()` has already stopped the pre-existing handles.
- Is this the best way to solve the issue? Yes: the fix should keep the latency win while making the deferred startup cancellable and close-aware, instead of moving all maintenance back onto the readiness path.
