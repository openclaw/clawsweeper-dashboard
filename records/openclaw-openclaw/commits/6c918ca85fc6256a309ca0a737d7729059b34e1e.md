# Commit 6c918ca

## Summary

The commit correctly threads inherited tool policy through the main spawn handoff points, but the inherited allowlist is captured before the embedded runner appends bundled MCP/LSP tools. That can make delegated sessions stricter than their parent and break restricted-tool workflows that rely on bundled runtime tools.

## Findings

### Medium: Delegated sessions lose bundled MCP/LSP tools from inherited allowlists

- Kind: regression
- File: `src/agents/pi-tools.ts`
- Line: 999
- Evidence: `createOpenClawCodingTools()` populates the mutable `inheritedToolAllowlist` from `subagentFiltered` before returning the OpenClaw tools, and `createSessionsSpawnTool()` receives that same list through `src/agents/openclaw-tools.ts:403` and `src/agents/openclaw-tools.ts:417`. Bundled MCP/LSP tools are materialized later in the embedded runner and appended through `filteredBundledTools` at `src/agents/pi-embedded-runner/run/attempt.ts:1262` and `src/agents/pi-embedded-runner/run/attempt.ts:1284`, but nothing refreshes the inherited allowlist after those tools are added.
- Impact: A parent with a restrictive allowlist such as `["sessions_spawn", "mcp__bundle__fs_read"]` can still receive the bundled MCP tool after `applyFinalEffectiveToolPolicy()`, but a child spawned from that parent is persisted with an inherited allowlist that only reflects the pre-bundle tool surface. The child then filters out the bundled tool even though the parent had it, breaking delegated sessions under restrictive bundled MCP/LSP configurations.
- Suggested fix: Update the inherited allowlist from the final effective tool surface after bundled MCP/LSP tools and tool-search compaction are resolved, or move inherited allowlist computation to a shared runtime point where it can see every callable tool before `sessions_spawn` executes. Add a regression test with a restrictive allowlist that includes `sessions_spawn` plus a bundled MCP/LSP tool.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes, source-reproducible with a focused unit test: configure a parent allowlist containing `sessions_spawn` plus a bundled MCP/LSP tool, assert the parent final tool surface includes the bundled tool, execute `sessions_spawn`, and assert the stored child `inheritedToolAllow` also includes that bundled tool.
- Is this the best way to solve the issue? Yes in direction: the inherited policy should reflect the final parent callable surface, not the earlier core/OpenClaw-only surface. The safer alternative is recomputing from visible tools at `sessions_spawn` execution time.
