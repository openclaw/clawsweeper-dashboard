# Commit 45928ef

## Summary

Found one low-severity authorization ordering issue that still exists on current `main`: cached ancestor media in a Telegram reply chain can be downloaded before the normal DM authorization gate drops an unauthorized DM.

## Findings

### Low: Cached reply-chain media is fetched before DM authorization

- Kind: privacy
- File: `extensions/telegram/src/bot-handlers.runtime.ts`
- Line: 599
- Evidence: `processMessageWithReplyChain` builds and resolves the cached reply chain, including media downloads in `resolveReplyMediaForChain`, before calling `processMessage` at `extensions/telegram/src/bot-handlers.runtime.ts:599-601`. The DM authorization gate that drops unauthorized direct messages runs later inside `buildTelegramMessageContext` at `extensions/telegram/src/bot-message-context.ts:349-362`. The early guard in `handleInboundMessageLike` only pre-authorizes DMs when the current message or its immediate reply target has media (`extensions/telegram/src/bot-handlers.runtime.ts:1986`), so a text-only reply to a cached text parent can still cause a cached media grandparent to be fetched first.
- Impact: If a previously authorized DM has cached media in its reply chain and is later revoked or the bot restarts under a stricter policy, that now-unauthorized sender can send a text-only reply to the cached text parent and make OpenClaw fetch the cached ancestor media before the message is dropped. The media is not sent to the agent after the later auth drop, but it crosses the intended “do not fetch reply media for unauthorized DM replies” boundary and can create unnecessary local media downloads.
- Suggested fix: Move DM access enforcement ahead of reply-chain media resolution, or split reply-chain construction from media hydration so media is fetched only after `buildTelegramMessageContext` has accepted the sender and after context-visibility filtering has selected visible chain entries.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes in code path form: authorize a Telegram DM, receive/cache `9000` with media and `9001` text replying to `9000`, revoke the sender or restart with a restrictive DM policy, then have the same sender send text-only `9002` replying to `9001`. The immediate reply target is text, so the early media auth gate is skipped, but cached `9000` media is fetched during reply-chain hydration before the later DM gate drops the turn.
- Is this the best way to solve the issue? Unclear. The safest direction is to keep reply-chain ID/body hydration early but delay media fetches until after sender authorization and visibility filtering.
