# Commit dddf871

## Summary

Found one low-severity compatibility regression in fast-mode config lookup for legacy double-prefixed provider model keys.

## Findings

### Low: Fast mode no longer honors legacy double-prefixed model config keys

- Kind: regression
- File: `src/agents/fast-mode.ts`
- Line: 17
- Evidence: `resolveConfiguredFastModeRaw` now only reads `agents.defaults.models[modelKey(provider, model)]`. For provider-prefixed model IDs such as provider `openrouter` + model `openrouter/hunter-alpha`, `modelKey()` resolves to `openrouter/hunter-alpha`, so a legacy key like `openrouter/openrouter/hunter-alpha` is ignored. A focused probe returned `{"enabled":false,"source":"default"}` for a config with `"openrouter/openrouter/hunter-alpha": { params: { fastMode: true } }`. Current adjacent contracts still preserve that legacy key shape for other per-model params: `resolveExtraParams` checks both canonical and `legacyModelKey`, `findConfiguredAgentModelParams` checks both `modelKey(provider, modelId)` and raw `${provider}/${modelId}`, and tests explicitly cover legacy duplicated OpenRouter keys for thinking and extra params.
- Impact: Existing users with legacy duplicated keys for provider-prefixed model IDs can silently lose `params.fastMode` on embedded runs. Because this commit now passes the resolved boolean into `runEmbeddedPiAgent`, the ignored config becomes `fastMode: false`, overriding the model params path that otherwise still recognizes the legacy key.
- Suggested fix: Mirror the existing model-param lookup behavior: check the canonical key first and then the raw legacy key `${provider}/${model}` when it differs. This keeps the new provider-qualified-model fix while preserving legacy provider-prefixed model configs.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Run `resolveFastModeState({ cfg: { agents: { defaults: { models: { "openrouter/openrouter/hunter-alpha": { params: { fastMode: true } } } } } }, provider: "openrouter", model: "openrouter/hunter-alpha" })`; current main returns disabled/default.
- Is this the best way to solve the issue? yes. The safer fix is a narrow fallback lookup in `fast-mode.ts`, matching the existing compatibility behavior for per-model thinking and extra params without changing model selection semantics.
