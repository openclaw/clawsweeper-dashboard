## Summary

Found one actionable test-regression issue. The new assertion makes the Bedrock npm override pack-compatible, but it no longer proves the npm-installed package is pinned below the Windows ARM Node 24 resolver failure that the test and changelog describe.

## Findings

### Medium: npm install path is no longer guarded to the pinned Bedrock runtime

- Kind: test_gap
- File: `test/scripts/root-package-overrides.test.ts`
- Line: 26
- Evidence: The test still says it “pins the Bedrock runtime below the Windows ARM Node 24 npm resolver failure”, but the new assertion only checks `dependencies["@aws-sdk/client-bedrock-runtime"]` exists and `overrides["@aws-sdk/client-bedrock-runtime"]` is `$@aws-sdk/client-bedrock-runtime`. On current `main`, `package.json` has the direct dependency at `3.1041.0`, npm override as the self-reference, and only `pnpm.overrides` pinned to `3.1024.0` (`package.json:1650`, `package.json:1729`, `package.json:1744`). A focused temp npm install from a packed artifact resolved `node_modules/@aws-sdk/client-bedrock-runtime` to `3.1041.0`, not `3.1024.0`.
- Impact: The test now passes while the npm/global install path can still receive the version that the changelog says must be pinned below the Windows ARM Node 24 npm resolver failure (`CHANGELOG.md:266`). That can let the original install-before-onboarding failure regress for npm users while CI stays green.
- Suggested fix: Pin the direct dependency itself to `3.1024.0` for npm compatibility, keeping the npm override as `$@aws-sdk/client-bedrock-runtime` only if needed to avoid npm `EOVERRIDE`; add a focused package-lock/package-acceptance assertion that npm resolves `@aws-sdk/client-bedrock-runtime` to `3.1024.0`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Pack current `main` with npm ignoring scripts, install that artifact into a temp npm project with `--package-lock-only --ignore-scripts`, and inspect the package lock: the resolved Bedrock runtime is `3.1041.0`.
- Is this the best way to solve the issue? no. The commit solves the local npm override syntax conflict, but the safer fix is to make the npm-visible direct dependency carry the intended pin and test actual npm resolution.
