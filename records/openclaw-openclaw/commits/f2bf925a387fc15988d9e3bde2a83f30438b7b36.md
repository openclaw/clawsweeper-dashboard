# Commit f2bf925

## Summary

Found one concrete data-loss gap in the same cleanup path this commit hardens: directory EXDEV fallback now identity-checks copied entries before removing them, but top-level regular-file EXDEV fallback still unlinks the source name without checking that it is the same file copied from the pinned fd.

## Findings

### Medium: Regular-file EXDEV fallback can delete a replacement source file

- Kind: data_loss
- File: `src/agents/sandbox/fs-bridge-mutation-helper.ts`
- Line: 325
- Evidence: The commit adds `entry_identity` / `same_identity` and uses it through `remove_copied_entry(...)` for directory fallback cleanup at `src/agents/sandbox/fs-bridge-mutation-helper.ts:286`. The regular-file fallback still copies from `src_fd`, publishes the destination with `os.replace(...)`, and then runs `os.unlink(src_basename, dir_fd=src_parent_fd)` at `src/agents/sandbox/fs-bridge-mutation-helper.ts:325` without comparing `src_file_stat` to the current source path.
- Impact: If the source path is replaced after the copy from the open fd but before cleanup, the helper deletes the replacement and reports success. The destination contains the old bytes, while the new source bytes are lost. This affects cross-device sandbox rename fallback for regular files, including remote/open-shell style mounts where EXDEV fallback is the intended path.
- Suggested fix: Route top-level regular-file fallback cleanup through the same identity guard: capture `entry_identity(src_file_stat)`, `os.lstat(src_basename, dir_fd=src_parent_fd)` immediately before unlink, raise `source_changed_error(...)` on mismatch, and only unlink when the name still identifies the copied file. Add the analogous forced-EXDEV test for a top-level file replacement.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. I ran a focused synthetic check using the current helper source with test-only forced `EXDEV` and a replacement inserted immediately before the regular-file cleanup unlink. The helper exited `0`, destination contained the original payload, and the source replacement was gone (`ENOENT`).
- Is this the best way to solve the issue? yes. The commit already introduces the identity primitive and error path; applying it before the top-level file unlink keeps behavior consistent with the new directory cleanup guard.
