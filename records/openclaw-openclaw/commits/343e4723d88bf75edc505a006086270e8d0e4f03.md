# Commit 343e4723d8

## Summary

Found one actionable regression in the new persisted Discord slash-command deploy cache: the cache is shared across Discord accounts but keyed only by command target shape, not by Discord application/client id.

## Findings

### Medium: Shared deploy hash cache can skip slash-command registration for additional Discord bot accounts

- Kind: regression
- File: `extensions/discord/src/internal/command-deploy.ts`
- Line: 80
- Evidence: `putCommandSetIfChanged()` stores and checks hashes under keys such as `global:reconcile`, `global:overwrite`, and `guild:<guildId>` without including `this.params.clientId`. Startup wires every Discord account to the same state file at `discord/command-deploy-cache.json` in `extensions/discord/src/monitor/provider.startup.ts:141`, while docs support multiple bots with distinct `channels.discord.accounts.<id>.applicationId`. A focused smoke with two `Client` instances sharing the same hash store but using different `clientId` values produced `{"appAGet":1,"appAPost":1,"appBGet":0,"appBPost":0}`, showing the second application skipped all REST deploy/reconcile calls.
- Impact: In a multi-bot Discord setup, after the first account persists the desired command hash, later accounts with the same command set can skip their own application-command reconciliation. New accounts may never receive native slash commands, and existing secondary accounts can remain stale after command changes.
- Suggested fix: Scope persisted deploy hashes by Discord application id, for example by prefixing every cache key with `clientId` or using an account/application-specific cache file. Keep the existing command-set hash inside that per-application scope.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: create two Discord clients/accounts with different `applicationId`/`clientId` values, the same commands, and the same `commandDeployHashStorePath`; deploy the first, then deploy the second. The second skips `GET`/`POST` entirely because the persisted `global:reconcile` hash matches.
- Is this the best way to solve the issue? no: the optimization is useful, but the persisted state must be scoped to the remote Discord application whose command state it represents.
