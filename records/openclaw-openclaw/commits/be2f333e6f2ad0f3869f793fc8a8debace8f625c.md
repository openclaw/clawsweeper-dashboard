# Commit be2f333

## Summary

Found two low-severity CLI error-message regressions that still exist on current `main`.

## Findings

### Low: Strict JSON parse errors echo secret-bearing config values

- Kind: privacy
- File: `src/cli/error-format.ts`
- Line: 57
- Evidence: `formatStrictJsonParseFailure` includes `JSON.stringify(preview)` in the error text, and `parseValue` passes the raw CLI value from `config set` before any path-aware redaction at `src/cli/config-cli.ts:1223`. A focused formatter probe printed `Could not parse "SECRET_TOKEN_1234567890" as JSON for --strict-json...`.
- Impact: mistyped `openclaw config set channels.<id>.token <secret> --strict-json` or similar secret-bearing paths now copy the first 48 characters of the supplied secret into stderr/log captures. The previous message did not include the raw value.
- Suggested fix: do not echo the raw value in strict JSON parse failures, or pass the target path into the formatter and redact for secret-sensitive paths before previewing.
- Confidence: high

### Low: `secrets reload` reports Gateway non-response for real RPC failures

- Kind: bug
- File: `src/cli/secrets-cli.ts`
- Line: 86
- Evidence: all `callGatewayFromCli("secrets.reload", ...)` errors are formatted with `formatGatewayCommandFailure`, which hard-codes `because the Gateway did not respond`. But the Gateway can respond with a method error: `src/gateway/server-methods/secrets.ts:46` catches reload failures and sends `errorShape(..., "secrets.reload failed")`; `GatewayClient` converts that response into `GatewayClientRequestError` at `src/gateway/client.ts:863`. A focused formatter probe printed `Could not reload secrets because the Gateway did not respond: secrets.reload failed...`.
- Impact: when reload fails after reaching the Gateway, for example secret activation or channel restart failure, the CLI tells the operator the Gateway did not respond and points only at status inspection. That obscures the actual class of failure.
- Suggested fix: use the new Gateway recovery text only for transport/connect/timeout errors, and preserve a `Secrets reload failed: <message>` style for `GatewayClientRequestError` or other method-level failures.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. The strict JSON leak is reproducible through the exported formatter and the `config set` parse path. The reload misclassification is reproducible through the exported formatter and the documented RPC error path from `secrets.reload`.
- Is this the best way to solve the issue? Yes for direction: avoid value echoing unless redacted, and distinguish transport failures from RPC/application failures.
