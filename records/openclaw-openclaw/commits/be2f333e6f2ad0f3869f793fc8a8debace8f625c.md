# Commit be2f333

## Summary

Found one low-severity diagnostic regression in the new `secrets reload` error formatter. The tests added in the commit cover offline transport failures, but the same formatter is also used for server-side `secrets.reload` failures where the Gateway did respond.

## Findings

### Low: `secrets reload` says the Gateway did not respond for Gateway-returned reload failures

- Kind: bug
- File: `src/cli/error-format.ts`
- Line: 79
- Evidence: `src/cli/secrets-cli.ts:83` sends every caught `secrets.reload` error through `formatGatewayCommandFailure`, which always returns `Could not reload secrets because the Gateway did not respond...`. But `src/gateway/server-methods/secrets.ts:50` catches application reload failures from a live Gateway and responds with `errorShape(..., "secrets.reload failed")`; `src/gateway/server-aux-handlers.ts:157` through `src/gateway/server-aux-handlers.ts:190` shows concrete live failure causes such as required channel restarts being skipped or restart failures. A focused formatter check produces: `Could not reload secrets because the Gateway did not respond: secrets.reload failed.`
- Impact: Users with a reachable Gateway but a failed reload get a false transport diagnosis and are pointed at Gateway status instead of the reload/config/channel-secret failure. That undercuts the commit’s recovery-text goal and can send operators down the wrong troubleshooting path.
- Suggested fix: Only use the “Gateway did not respond” wording for transport errors, for example by detecting `GatewayTransportError`/closed/timeout cases. For `GatewayClientRequestError` or generic server RPC failures, say `Could not reload secrets: <message>` and keep the status command as a secondary inspection hint if desired.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Force the Gateway `secrets.reload` handler’s `reloadSecrets` callback to throw, or hit an existing reload path that throws during channel restart. The handler responds over the established RPC with `secrets.reload failed`, and the CLI formats that as a non-response.
- Is this the best way to solve the issue? no. The formatter conflates transport and application failures; classifying the caught error before choosing wording is safer and keeps offline recovery text without mislabeling healthy-Gateway failures.
