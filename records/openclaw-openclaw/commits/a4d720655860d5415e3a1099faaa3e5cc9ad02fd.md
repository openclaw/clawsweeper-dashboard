# Commit a4d7206

## Summary

Found one low-severity regression in the new Discord voice permission audit: disabled voice auto-join config is still audited.

## Findings

### Low: Disabled Discord voice auto-join targets are still audited

- Kind: regression
- File: `extensions/discord/src/audit-core.ts`
- Line: 101
- Evidence: `collectDiscordAuditChannelIdsForAccount` always iterates `config.voice?.autoJoin`, but runtime voice explicitly treats `voice.enabled: false` as disabled in `extensions/discord/src/voice/config.ts:3`, and `DiscordVoiceManager.autoJoin()` returns immediately when `voiceEnabled` is false in `extensions/discord/src/voice/manager.ts:128`. A focused probe confirms the collector still returns `{"channelIds":["222"],"unresolvedChannels":0}` for `{ voice: { enabled: false, autoJoin: [...] } }`.
- Impact: `openclaw channels status --probe` can report Discord permission failures for stale or intentionally disabled voice auto-join channels. That can make an otherwise valid text-only Discord account look degraded after the user disables voice but leaves `voice.autoJoin` in config.
- Suggested fix: include `enabled?: boolean` in the collector’s voice shape and skip `voice.autoJoin` when effective voice is disabled, matching `resolveDiscordVoiceEnabled` semantics. Add a regression test for `voice: { enabled: false, autoJoin: [...] }`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Configure `channels.discord.voice.enabled: false` with a numeric `voice.autoJoin[].channelId`; the collector still returns that channel for audit, and `auditAccount` will probe it during `channels status --probe`.
- Is this the best way to solve the issue? yes. The audit target selection should match the runtime’s effective voice enablement before checking auto-join permissions.
