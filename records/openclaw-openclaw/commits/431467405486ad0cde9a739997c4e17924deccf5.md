# Commit 4314674

## Summary

The commit introduces a shared plugin metadata snapshot resolver, but several call sites now opt unscoped requests into workspace-scoped current metadata. Existing focused tests reproduce wrong model/tool planning behavior without a workspace context.

## Findings

### Medium: Unscoped model ID normalization reuses workspace-scoped metadata

- Kind: regression
- File: `src/plugins/manifest-model-id-normalization.ts`
- Line: 50
- Evidence: `resolveMetadataSnapshotForPolicies` always passes `allowWorkspaceScopedCurrent: true`. The existing test `manifest-model-id-normalization.test.ts` fails: `normalizeDemoModel()` returns `"alpha/demo-model"` from a `/workspace/a` snapshot when no workspace context is active, but the test expects `undefined`.
- Impact: Model references normalized outside a workspace can inherit another workspace plugin’s `modelIdNormalization` policy, producing wrong provider model IDs.
- Suggested fix: Only pass `allowWorkspaceScopedCurrent` when `workspaceDir` was explicit or came from active plugin runtime state; keep bare unscoped callers on unscoped current metadata or a fresh metadata load.
- Confidence: high

### Medium: Optional media tools become enabled from unrelated workspace metadata

- Kind: regression
- File: `src/agents/tools/manifest-capability-availability.ts`
- Line: 85
- Evidence: `loadCapabilityMetadataSnapshot` passes `allowWorkspaceScopedCurrent: workspaceDir === undefined`; the existing `openclaw-tools.media-factory-plan.test.ts` now reports `imageGenerate === true` for an unscoped request that should not see the `/workspace/a` image provider.
- Impact: Sessions or control-plane paths without a workspace can expose image/video/music factories based on a workspace-only plugin and auth signal.
- Suggested fix: Do not allow workspace-scoped current snapshots when no workspace context exists; require an explicit `workspaceDir` or active runtime workspace before using workspace plugin capabilities.
- Confidence: high

### Medium: Explicit agent-dir model planning reuses default workspace plugin metadata

- Kind: regression
- File: `src/agents/models-config.ts`
- Line: 181
- Evidence: `ensureOpenClawModelsJson({}, agentDir)` sets no workspace context, but now passes `allowWorkspaceScopedCurrent: workspaceDir === undefined`. Existing `models-config.write-serialization.test.ts` fails because the plan receives the default workspace snapshot object for a non-default explicit agent dir.
- Impact: `models.json` planning for explicit agent directories can use the wrong workspace plugin registry, affecting model normalization and provider catalog planning.
- Suggested fix: Preserve the previous behavior for `agentDirOverride` without `options.workspaceDir`: do not opt into workspace-scoped current metadata.
- Confidence: high

### Low: Manifest contract eligibility test mock was not updated for the new resolver import

- Kind: reliability
- File: `src/plugins/manifest-contract-eligibility.test.ts`
- Line: 8
- Evidence: The test mocks `./plugin-metadata-snapshot.js` with only `loadPluginMetadataSnapshot`; after the source switched to `resolvePluginMetadataSnapshot`, all four tests in this file fail with “No `resolvePluginMetadataSnapshot` export is defined”.
- Impact: Focused/broader Vitest lanes covering this file fail on current `main`.
- Suggested fix: Update the mock to export `resolvePluginMetadataSnapshot` and adjust expectations to the new resolver contract.
- Confidence: high
