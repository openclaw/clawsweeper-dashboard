# Commit 6aafdf1

## Summary

Found one regression risk in the doctor repair: it treats every bare `payload.model: "default"` as an invalid sentinel even though cron runtime can resolve a bare model id named `default` as a valid explicit override.

## Findings

### Medium: Doctor can delete a valid explicit `default` model override

- Kind: regression
- File: `src/commands/doctor-cron-store-migration.ts`
- Line: 238
- Evidence: `isInvalidCronPayloadModelSentinel` classifies the normalized string `"default"` as invalid, and `normalizeStoredCronJobs` then deletes `payloadRecord.model` for agent-turn jobs. But cron runtime treats any nonempty `payload.model` as an explicit override and resolves it through `resolveAllowedModelRef`; that resolver accepts bare model ids by inferring the provider from configured models or the default provider. I verified source-level behavior with a config that allows `minimax-portal/default`: `resolveAllowedModelRef({ raw: "default", defaultProvider: "minimax-portal" })` returns `{ provider: "minimax-portal", model: "default" }`, while the new doctor normalization removes the same stored `payload.model`.
- Impact: Running `openclaw doctor --fix` can silently change a cron job that intentionally targets a provider model whose id is literally `default`. After repair, the job inherits the agent/default model instead, which can change provider/model, cost, capability, and output behavior.
- Suggested fix: Do not blanket-delete `"default"` before consulting model configuration. Either only remove `"default"` when it fails the same `resolveAllowedModelRef` path used by cron preflight, or limit this repair to `null`, `"null"`, and blank values while requiring users to remove ambiguous `"default"` manually.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Source-level reproduction: configure an allowed model key such as `minimax-portal/default`, resolve bare `payload.model: "default"` with cron’s model resolver, then run `normalizeStoredCronJobs`; the resolver accepts it but doctor deletes it.
- Is this the best way to solve the issue? no. The repair should share cron’s model-resolution contract or avoid mutating ambiguous valid strings.
