# Commit 6115087

## Summary

Found one concrete bug in the new `--host-ip` support: the aggregate npm update smoke consumes the override for its own update tarball server, but does not forward it to the per-platform fresh install child smoke scripts.

## Findings

### Low: `--host-ip` is not forwarded to fresh baseline child smokes

- Kind: bug
- File: `scripts/e2e/parallels/npm-update-smoke.ts`
- Line: 370
- Evidence: `parseArgs` stores `options.hostIp` and `run()` resolves it at line 260, but `spawnFresh()` builds child args for `scripts/e2e/parallels/${platform}-smoke.ts` without adding `--host-ip`. Linux and Windows fresh child scripts call `resolveHostIp(this.options.hostIp)` and serve/fetch tgz artifacts through that IP.
- Impact: On the hosts this flag is meant to support, `parallels-npm-update-smoke.sh --host-ip <ip>` can still fail during fresh baseline or fresh target phases because Linux/Windows child smokes receive no override and fall back to autodetecting a `10.211.*` address.
- Suggested fix: Add the parent host override to `spawnFresh()` args when `this.options.hostIp` is set, covering both `runFreshBaselines()` and `runFreshTargetInstalls()`. Update the test to assert the child command includes `--host-ip`, not only that the parent resolves it.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. On a Parallels host where autodetection fails or the reachable host address is not the first `10.211.*` address, run `bash scripts/e2e/parallels-npm-update-smoke.sh --platform linux --host-ip <reachable-ip>`. The parent accepts the flag, then the Linux fresh child is spawned without it and calls `resolveHostIp(undefined)`.
- Is this the best way to solve the issue? Yes. The child smokes already support `--host-ip`; the aggregate wrapper just needs to propagate the option through its existing child-arg construction.
