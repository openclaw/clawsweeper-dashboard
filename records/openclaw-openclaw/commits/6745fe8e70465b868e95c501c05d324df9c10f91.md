# Commit 6745fe8e7046

## Summary

Found one diagnostic false negative in the new doctor warning. A single exact MCP tool deny entry is treated as if all MCP tools were intentionally denied, so `openclaw doctor` suppresses the new warning even though sandbox policy still hides the rest of the configured MCP server tools.

## Findings

### Low: Exact MCP tool deny suppresses the sandbox MCP warning for the whole server

- Kind: bug
- File: `src/commands/doctor/shared/plugin-tool-allowlist-warnings.ts`
- Line: 451
- Evidence: `toolPolicyDeniesAllMcpServers()` reuses `entriesMatchEveryMcpTool()`. That matcher counts any normalized entry longer than the server prefix and starting with that prefix as a server match, so `deny: ["outlook__send_mail"]` is classified as “denies all MCP servers.”
- Impact: With `agents.defaults.sandbox.mode: "all"`, `mcp.servers.outlook`, and only `tools.sandbox.tools.deny: ["outlook__send_mail"]`, doctor returns no warning. A runtime probe showed the resolved sandbox policy still filters both `outlook__send_mail` and `outlook__read_mail`, so the user gets no diagnostic for the still-hidden MCP tools.
- Suggested fix: Use stricter deny-all detection than allow matching. Treat `*`, `bundle-mcp`, `group:plugins`, and full server-wide patterns such as `<server>__*` as intentional all-MCP denies; do not treat arbitrary exact server tool names or narrower globs as all-server denies.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused `node --import tsx` probe against `collectPluginToolAllowlistWarnings()` returned the warning for no deny, but returned `[]` for `tools.sandbox.tools.deny: ["outlook__send_mail"]` and for `tools.deny: ["outlook__send_mail"]`.
- Is this the best way to solve the issue? yes. The bug is in the diagnostic’s deny classification, not runtime sandbox filtering.
