# Commit 7b7e651

## Summary

Found two actionable compatibility/behavior issues in the markdown parser refactor. The main runtime bug is that ordered-list items are now resolvable and writable, but the markdown edit path still only rewrites dash bullets, so `set` can report success while writing unchanged bytes.

## Findings

### Medium: Ordered-list markdown item writes silently no-op

- Kind: bug
- File: `extensions/oc-path/src/oc-path/edit.ts`
- Line: 84
- Evidence: `extensions/oc-path/src/oc-path/parse.ts:150` now treats every `list_item_open` as an `AstItem`, including ordered lists. But `rebuildBlockBody` still uses `^(\\s*-\\s*KEY\\s*:\\s*).*$`, which only matches dash bullets. A focused smoke with `## H\n1. timeout: 5\n` parsed `timeout` as a kv item, `setMdOcPath(..., "60")` returned `ok: true`, and `emitMd(r.ast)` remained `## H\n1. timeout: 5\n`.
- Impact: `openclaw path set oc://X.md/h/timeout/timeout 60` can tell scripts it wrote successfully while leaving the target file unchanged for newly-supported ordered-list items.
- Suggested fix: Either preserve and use the original list marker/line span when rewriting item kv values, or reject non-dash markdown kv items as `not-writable` until the edit path supports them. Add an edit/CLI regression test for ordered-list kv items.
- Confidence: high

### Low: Public AST shape changed without the documented SDK version bump

- Kind: compatibility
- File: `extensions/oc-path/src/oc-path/index.ts`
- Line: 34
- Evidence: The commit removes exported `AstTable` and `AstCodeBlock` types and removes `tables`/`codeBlocks` from `AstBlock`, but leaves `SDK_VERSION = "0.1.0"`. The adjacent comment says this value is bumped on every breaking change to AST shape and public verbs.
- Impact: Any plugin pack or internal consumer using the exported oc-path AST/version contract can see the same SDK version for an incompatible AST surface, so version-based compatibility checks cannot distinguish pre/post-removal behavior.
- Suggested fix: Bump `SDK_VERSION` for the AST break, or keep deprecated empty `tables`/`codeBlocks` fields/types until the next intentional version boundary.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The ordered-list no-op reproduces with `parseMd("## H\n1. timeout: 5\n")`, `setMdOcPath(... "oc://X.md/h/timeout/timeout", "60")`, and `emitMd`, which returns unchanged bytes despite `ok: true`.
- Is this the best way to solve the issue? yes for the suggested direction: parser and writer need the same supported item contract. Either write all item markers the parser resolves, or make unsupported markers non-writable.
