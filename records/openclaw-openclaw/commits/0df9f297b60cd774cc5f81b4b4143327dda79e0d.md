# Commit 0df9f297b6

## Summary

Found one same-feature regression in the new transcript mirror: accepted Telegram topic source sends with an explicit unprefixed topic target can still be delivered but skipped by the mirror predicate.

## Findings

### Medium: Explicit Telegram topic source sends can miss the transcript mirror

- Kind: bug
- File: `src/infra/outbound/source-reply-mirror.ts`
- Line: 120
- Evidence: `isCurrentSourceConversation` compares the requested target string directly to `toolContext.currentChannelId`, or to `resolveCurrentChannelId(requestedTarget, threadId)`. Real Telegram tool context preserves `context.To` as values like `telegram:-1001234567890:topic:42` (`extensions/telegram/src/threading-tool-context.ts:30`), while the message tool accepts explicit topic targets like `-1001234567890:topic:42` with `threadId: "42"` (`src/agents/tools/message-tool.test.ts:953`). Telegram’s `resolveCurrentChannelId` returns the unprefixed `to` unchanged when it already contains `:topic:` (`extensions/telegram/src/channel.ts:1126`), so both comparisons are false and `mirrorDeliveredSourceReplyToTranscript` returns before appending.
- Impact: A successful same-topic Telegram `message(action="send")` can remain only as an elided tool result, so later transcript/history/context still misses the visible reply. This is the same class of user-visible gap the commit is meant to fix, but for an explicit target shape the tool already supports.
- Suggested fix: Normalize same-source comparison through channel-owned target parsing/canonicalization before comparing, or compare alias candidates for both requested and current targets. Add a regression with `currentChannelId: "telegram:-1001234567890:topic:42"` and `target: "-1001234567890:topic:42"` plus `threadId: "42"`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. The static path is concrete: `runMessageAction` normalizes and forwards gateway plugin sends with `params` intact (`src/infra/outbound/message-action-runner.ts:948` and `src/infra/outbound/message-action-runner.ts:580`), and the gateway passes those params into the mirror (`src/gateway/server-methods/send.ts:394`). The string comparison in the mirror then rejects the same Telegram topic when one side has the `telegram:` prefix and the other does not.
- Is this the best way to solve the issue? Yes: the fix should be in the mirror’s source-conversation matcher, not in Telegram delivery, because delivery already accepts both target spellings.
