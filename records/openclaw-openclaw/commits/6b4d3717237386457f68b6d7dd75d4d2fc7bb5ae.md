# Commit 6b4d371

## Summary

Found one low-severity docs/behavior mismatch introduced by the commit: the docs now advertise `"${OPENAI_API_KEY}"` as a SecretInput shorthand, but for `openclaw.json` config fields that form is still consumed by config env substitution before secrets audit can classify it as a SecretRef.

## Findings

### Low: `${VAR}` shorthand is documented for SecretInput fields but still fails config-field secrets audit

- Kind: bug
- File: `docs/gateway/secrets.md`
- Line: 91
- Evidence: The new docs say supported SecretInput fields accept both `"${OPENAI_API_KEY}"` and `"$OPENAI_API_KEY"` (`docs/gateway/secrets.md:91`). Current config loading resolves `${ENV}` before the snapshot used by audit (`src/config/io.ts:1159`, `src/config/types.openclaw.ts:184`), and `collectConfigSecrets` only skips plaintext when `resolveSecretInputRef` sees the original ref-shaped value (`src/secrets/audit.ts:220`). A focused smoke confirmed the mismatch: `gateway.auth.token: "${OPENCLAW_GATEWAY_TOKEN}"` with the env var set reports `PLAINTEXT_FOUND` at `gateway.auth.token`, while `gateway.auth.token: "$OPENCLAW_GATEWAY_TOKEN"` is clean.
- Impact: Operators following the updated SecretRef docs for `openclaw.json` can still get the exact false-positive audit state this commit is meant to reduce. The object form and `$VAR` form still work, so this is not a blocker, but the new docs make the unresolved `${VAR}` audit caveat user-facing.
- Suggested fix: Either make config-field audit source-aware for exact `${VAR}` values, or narrow the docs to recommend `$VAR`/object SecretRefs for `openclaw.json` and mention that `${VAR}` is only audit-safe on non-config stores such as auth profiles until the config substitution path is fixed.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The smoke used `runSecretsAudit` against a temporary config with `gateway.auth.mode: "token"` and compared `"${OPENCLAW_GATEWAY_TOKEN}"` versus `"$OPENCLAW_GATEWAY_TOKEN"`.
- Is this the best way to solve the issue? unclear. The safest code fix likely needs authored-value/source provenance, which overlaps the existing `${VAR}` audit work; the narrow docs fix is lower risk if maintainers do not want to change audit semantics here.
