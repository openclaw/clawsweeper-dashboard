# Commit 186b8e44

## Summary

Found one compatibility regression in auth-backed media tool availability.

## Findings

### Medium: Runtime-only external CLI auth profiles stop enabling media tools

- Kind: regression
- File: `src/agents/tools/model-config.helpers.ts`
- Line: 47
- Evidence: `hasAuthForProvider` now short-circuits whenever an `authStore` is supplied, returning only `listProfilesForProvider(params.authStore, ...)`. The embedded run supplies that store from `ensureAuthProfileStoreWithoutExternalProfiles` in `src/agents/pi-embedded-runner/run.ts:514`, so runtime-only external CLI overlays are absent. The fallback path that still calls `ensureAuthProfileStore(... externalCliDiscoveryForProviderAuth(...))` is now skipped for media tools. That matters for declared external CLI-backed providers such as `openai-codex` in `src/agents/auth-profiles/external-cli-sync.ts:99` and `minimax-portal` in `src/agents/auth-profiles/external-cli-sync.ts:126`; MiniMax declares media/image/music/video generation contracts for `minimax-portal` in `extensions/minimax/openclaw.plugin.json:69`, and OpenAI declares `openai-codex` as a media/image auth signal in `extensions/openai/openclaw.plugin.json:757` and `extensions/openai/openclaw.plugin.json:769`.
- Impact: Users whose media capability is authenticated only through a readable external CLI credential, rather than a persisted OpenClaw auth profile or env var, can lose `image`, `pdf`, `image_generate`, `video_generate`, or `music_generate` availability during reply startup. The old helper path could discover those scoped runtime profiles; the new supplied-store path cannot.
- Suggested fix: Build the run media-availability auth store with the same scoped external profile overlay, or make `hasAuthForProvider` fall back to scoped external CLI discovery when the supplied store has no matching provider and `agentDir` is available. To preserve the perf goal, prefer computing one scoped overlay for all media provider ids instead of probing per helper call.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Unclear live; I did not have external CLI credentials in this checkout. Source-level reproduction is concrete: use only runtime external CLI auth for `minimax-portal` or `openai-codex`, no persisted profile/env, then build media tools through an embedded run.
- Is this the best way to solve the issue? Yes directionally: keep the reusable store optimization, but ensure the reused store has the same external auth facts as the previous availability check.
