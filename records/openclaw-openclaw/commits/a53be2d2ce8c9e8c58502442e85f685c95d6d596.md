# Commit a53be2d2ce

## Summary

Found one regression in the setup/TUI timeout change. The intended 5-minute timeout for the bootstrap hatch message is passed as a process-wide TUI option, so it also applies to every later message typed in that setup-launched TUI session.

## Findings

### Medium: setup hatch timeout leaks to later TUI messages

- Kind: regression
- File: `src/wizard/setup.finalize.ts`
- Line: 462
- Evidence: setup now launches the TUI with `timeoutMs: HATCH_TUI_TIMEOUT_MS` at `src/wizard/setup.finalize.ts:458`. The TUI sends the initial hatch message through the normal `sendMessage(autoMessage)` path at `src/tui/tui.ts:1159`, and that same command handler forwards `opts.timeoutMs` on every chat send at `src/tui/tui-command-handlers.ts:635`. The default agent timeout is 48 hours when no override is supplied (`src/agents/timeout.ts:3`), so this changes later user messages in the setup-launched TUI from the normal default to 300 seconds.
- Impact: a first-run user who keeps using the terminal hatch session for a long task can have normal, post-hatch agent turns aborted after 5 minutes. The PR intended to bound only the bootstrap wake run, not normal user-started TUI messages.
- Suggested fix: scope the hatch timeout to the initial auto-message only. For example, add a separate `initialMessageTimeoutMs` option and have `runTui`/`sendMessage` use it only when `autoMessageSent` is false and the auto-message is being dispatched; leave `opts.timeoutMs` undefined for later manual sends unless the user explicitly passed `--timeout-ms`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Launch setup hatch with the new `timeoutMs`, wait for the auto-message path to run, then send any later message in the same TUI; source shows the later send still receives `timeoutMs: opts.timeoutMs`, which the embedded backend converts to `timeout: "300"`.
- Is this the best way to solve the issue? yes. The timeout needs a narrower lifetime; keeping it on `TuiOptions.timeoutMs` is too broad because that option is designed as the default for all TUI sends.
