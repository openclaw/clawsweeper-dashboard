# Commit a35067f

## Summary

Found two media model resolution regressions in the shared candidate resolver.

## Findings

### Medium: Exact provider refs can be rerouted to another provider's model catalog entry

- Kind: regression
- File: `src/media-generation/runtime-shared.ts`
- Line: 206
- Evidence: The new resolver returns `resolveProviderModelOnlyRef(...) ?? parsed`, so provider metadata now wins before honoring an already parsed provider/model ref. In the parent, `providerMatchesId` gave exact provider IDs and aliases precedence before trying model-only resolution. With Google and OpenRouter providers registered, `imageGenerationModel.primary: "google/gemini-3.1-flash-image-preview"` resolves first to `{ provider: "openrouter", model: "google/gemini-3.1-flash-image-preview" }` because OpenRouter declares that full string as one of its model IDs.
- Impact: A documented native Google image config can execute through OpenRouter instead when OpenRouter is registered/configured, changing provider, auth path, billing, request semantics, and provider-specific options. If OpenRouter is not authenticated, it can also add a spurious failed attempt before the requested provider.
- Suggested fix: Restore exact parsed provider precedence when the parsed provider matches a registered provider or alias, and only use provider model-only metadata when the raw value is not an exact provider ref. Add coverage for `google/gemini-3.1-flash-image-preview` with an OpenRouter provider present.
- Confidence: high

### Low: Disabling auto fallback now drops model-only explicit defaults

- Kind: compatibility
- File: `src/media-generation/runtime-shared.ts`
- Line: 232
- Evidence: Explicit `primary` and `fallbacks` now pass `useProviderMetadata: autoProviderFallbackEnabled`. When `agents.defaults.mediaGenerationAutoProviderFallback` is `false`, metadata lookup is skipped entirely. Current behavior for `primary: "gpt-image-2"` returns no candidates, and `primary: "fal-ai/flux/dev"` resolves to provider `fal-ai` instead of registered provider `fal`. The parent resolver still used provider metadata for explicit entries even when implicit provider fallback was disabled.
- Impact: Users who disabled implicit cross-provider fallback can lose explicit model-only primary/fallback entries, despite docs saying the flag should keep explicit `model`, `primary`, and `fallbacks` entries.
- Suggested fix: Decouple model-only qualification from implicit provider fallback. `mediaGenerationAutoProviderFallback=false` should suppress appended auth-backed provider defaults, not metadata needed to resolve explicit entries.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Stub-provider runtime checks against current `main` showed `generateImage` returning `resultProvider: "openrouter"` for explicit `google/gemini-3.1-flash-image-preview`, and showed `auto=false` resolving `fal-ai/flux/dev` to provider `fal-ai`.
- Is this the best way to solve the issue? yes: restore the old precedence split between exact provider refs and model-only metadata, while preserving the commit's goal of avoiding provider listing for unambiguous exact refs.
