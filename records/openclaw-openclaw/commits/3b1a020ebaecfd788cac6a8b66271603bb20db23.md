# Commit 3b1a020

## Summary

Found regressions in the new `gateway start` auto-repair path. The repair reuses install logic from a start command, but it does not preserve some existing service-owned settings and bypasses install-only guardrails.

## Findings

### Medium: Start repair can silently move a custom-port service back to the default port

- Kind: regression
- File: `src/cli/daemon-cli/start-repair.ts`
- Line: 33
- Evidence: `repairLoadedGatewayServiceForStart` derives `port` from `resolveGatewayPort(cfg)` instead of the stale service command/environment. The existing service env is read, but `mergeInstallInvocationEnv` intentionally drops `OPENCLAW_*` keys, including `OPENCLAW_GATEWAY_PORT`, before building the replacement install plan. A service installed with `openclaw gateway install --port 19001` can therefore be repaired by `openclaw gateway start` into a default/config port such as `18789`.
- Impact: After an OpenClaw update, running `gateway start` against an otherwise valid stale service can change the listening port, breaking existing clients, docs-generated URLs, reverse proxies, or colliding with another local gateway.
- Suggested fix: For start-time repair, preserve the effective installed service port first, from `state.command.programArguments` and `state.command.environment.OPENCLAW_GATEWAY_PORT`, then fall back to current config/env. Add a test where a stale service installed on a non-default port is repaired without changing that port.
- Confidence: high

### Medium: Nix mode can still rewrite the managed service from `gateway start`

- Kind: regression
- File: `src/cli/daemon-cli/lifecycle.ts`
- Line: 163
- Evidence: `runDaemonStart` unconditionally passes `repairLoadedService`, and the repair path calls `params.service.install` in `start-repair.ts:73`. The normal install command blocks this with `failIfNixDaemonInstallMode` in `src/cli/daemon-cli/install.ts:80`, and the Nix docs say `OPENCLAW_NIX_MODE=1` disables auto-install/self-mutation flows.
- Impact: Declarative Nix/Home Manager installs can be overwritten by an imperative `openclaw gateway start` if the service version is stale, violating the Nix-mode contract and potentially fighting rollback/rebuild state.
- Suggested fix: Apply the same Nix-mode install guard before invoking start repair, or make repair report the stale service and point users to their Nix-managed update path.
- Confidence: high

### Medium: Windows repair updates and runs the task without stopping the old gateway first

- Kind: reliability
- File: `src/cli/daemon-cli/start-repair.ts`
- Line: 73
- Evidence: The repair path calls `service.install`. On Windows that reaches `installScheduledTask`, whose existing-task path changes `/TR` and calls `/Run` (`src/daemon/schtasks.ts:645` and `src/daemon/schtasks.ts:655`). The normal restart path first ends the task and terminates listeners before rerunning it (`src/daemon/schtasks.ts:993` through `src/daemon/schtasks.ts:1008`).
- Impact: If a stale Windows gateway is already running, `gateway start` can try to run the updated task while the old listener still owns the port, causing a failed launch or leaving the old process serving after the command reports repair/start intent.
- Suggested fix: On Windows repair, stop/terminate the existing task/listeners before or immediately after rewriting the task script, then launch through the same restart semantics.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the first two by constructing stale service state with `OPENCLAW_SERVICE_VERSION` mismatch and either a custom `OPENCLAW_GATEWAY_PORT`/`--port` or `OPENCLAW_NIX_MODE=1`. The Windows finding is code-path based; I did not run a Windows live check.
- Is this the best way to solve the issue? No. Start-time repair needs a narrower repair contract than install: preserve installed lifecycle parameters unless explicitly overridden, honor Nix mode, and use platform restart semantics after rewriting.
