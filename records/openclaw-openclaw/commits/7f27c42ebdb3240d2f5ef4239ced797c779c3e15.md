# Commit 7f27c42

## Summary

Found one low-severity reliability issue in the new optional preview generation path. The FFmpeg command shape is consistent with the documented palettegen/paletteuse flow, but the Bash error handling around it can silently treat failed conversions as success. ([ffmpeg.org](https://ffmpeg.org/ffmpeg-filters.html))

## Findings

### Low: Preview conversion failures are masked as success

- Kind: reliability
- File: `.github/workflows/mantis-discord-status-reactions.yml`
- Line: 424
- Evidence: `make_desktop_preview` relies on `set -e` to stop after either FFmpeg command fails, but the function is invoked inside `if ! make_desktop_preview baseline || ! make_desktop_preview candidate; then`. Bash ignores `errexit` in commands whose status is inverted with `!` or used in non-final `||`/`&&` positions, so a failing `ffmpeg` can be followed by `rm -f "$palette"`, returning success and skipping the warning/cleanup branch. I reproduced the control-flow behavior locally with a reduced Bash function: `false; echo after; rm -f /tmp/nope` still printed `after` and the outer `if ! f` path reported success. ([gnu.org](https://www.gnu.org/s/bash/manual/bash.html?utm_source=openai))
- Impact: If GIF conversion fails because the MP4 is unreadable, unsupported, or FFmpeg errors mid-write, the workflow may omit previews without warning, or publish broken preview files if partial outputs exist. The main Mantis result and MP4/screenshot evidence still work, so this is low severity.
- Suggested fix: Make `make_desktop_preview` return explicitly on each conversion failure, for example `ffmpeg ... || return 1`, and clean up the palette/output on the failing path before returning.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The Bash control-flow behavior reproduces with a minimal local shell function called under `if ! f || ! g`; this directly matches the workflow’s function invocation shape.
- Is this the best way to solve the issue? yes. Explicit `|| return 1` checks are the narrowest fix and avoid depending on `errexit` in a context where Bash documents that it is ignored.
