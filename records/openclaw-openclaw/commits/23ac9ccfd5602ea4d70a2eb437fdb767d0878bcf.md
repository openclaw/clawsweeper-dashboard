# Commit 23ac9cc

## Summary

Found two trust-boundary issues in the new Codex npm-plugin exception. The intended happy path is covered and passes, but the trust checks are broader than “the official npm Codex package.”

## Findings

### Medium: Official Codex unsafe-scan bypass trusts package name without registry/provenance proof

- Kind: supply_chain
- File: `src/plugins/install.ts`
- Line: 189
- Evidence: `isTrustedOfficialNpmPluginInstall` returns trusted when the request is `plugin-npm`, the requested package name is `@openclaw/codex`, the installed package name is `@openclaw/codex`, and the plugin id is `codex`. That trusted bit is passed into `scanPackageInstallSource` at `src/plugins/install.ts:721`, allowing critical dangerous-code findings without `--dangerously-force-unsafe-install`. The npm commands inherit process npm configuration via `resolveCommandEnv` merging `process.env` at `src/process/exec.ts:242` and `src/process/exec.ts:256`, and `createNpmProjectInstallEnv` does not reset registry configuration.
- Impact: an environment with a scoped/private registry, registry compromise, or registry misconfiguration can serve a package named `@openclaw/codex` with manifest id `codex` and receive the official-package scanner bypass. That defeats the install-time unsafe-code block this commit is relaxing only for the official public package.
- Suggested fix: make this exception provenance-backed, not name-backed. For example, require a verified public npm registry source plus expected package integrity/provenance, or keep requiring an explicit trusted-source flag supplied only by a higher-level resolver that has verified the official artifact.
- Confidence: high

### Low: Reserved `/codex` command ownership can be claimed by path substring spoofing

- Kind: security
- File: `src/plugins/registry.ts`
- Line: 253
- Evidence: `isOfficialNpmCodexPluginRecord` only checks `record.id === "codex"` and whether normalized `rootDir ?? source` contains the substring `/node_modules/@openclaw/codex`. `canClaimReservedCommandOwnership` then allows reserved command ownership for that record regardless of origin. A non-official plugin with id `codex` under a path like `.../node_modules/@openclaw/codex-fake` or another configured path containing that substring passes this check and can register `ownership: "reserved"` for `/codex`.
- Impact: this bypasses the reserved-command guard that otherwise blocks external plugins from overriding built-in command names and receiving reserved-command context such as owner/diagnostic flags.
- Suggested fix: require exact package path segment matching, require `origin === "global"`, and ideally tie the permission to the installed plugin record for source `npm`, spec/resolved name `@openclaw/codex`, and exact install path rather than path text alone.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. For the first issue, serve `@openclaw/codex` from a custom npm registry with manifest id `codex` and dangerous install-source code, then install `npm:@openclaw/codex@beta`; the current trust predicate is satisfied before the scanner decision. For the second, register a plugin record with id `codex`, non-bundled origin, and a root/source path containing `/node_modules/@openclaw/codex-fake`, then register a reserved `/codex` command.
- Is this the best way to solve the issue? The current approach solves the happy path but is too broad for a trust exception. The safer fix is to bind trust to verified package provenance and exact installed-record ownership.
