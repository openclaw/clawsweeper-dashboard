# Commit 7c5222a

## Summary

Found one regression in the new preserve-key plumbing. The commit normalizes preserve keys, but the maintenance predicate still compares them to raw store keys, so legacy mixed-case active session keys can lose the protection they had before this change.

## Findings

### Medium: Normalized preserve keys do not match legacy mixed-case store keys

- Kind: regression
- File: `src/config/sessions/store-maintenance.ts`
- Line: 318
- Evidence: `collectSessionMaintenancePreserveKeys` lowercases every base/provider key via `normalizeStoreSessionKey` at `src/config/sessions/store-maintenance-preserve.ts:20`. `saveSessionStoreUnlocked` now sends `opts.activeSessionKey` through that collector at `src/config/sessions/store.ts:286`; before this commit it used `new Set([opts.activeSessionKey])`. But `shouldPreserveMaintenanceEntry` checks `params.preserveKeys?.has(params.key)` against the raw object key at `src/config/sessions/store-maintenance.ts:318`.
- Impact: a legacy or externally-created mixed-case active session key can be pruned/capped under maintenance pressure even when the caller passes the exact active key. Focused probe on current `main`: seed `Agent:Main:Slack:direct:U1` as the active key plus a newer `agent:main:old`, run `saveSessionStore` with `activeSessionKey: "Agent:Main:Slack:direct:U1"` and `maxEntries: 1`; the persisted store contains only `agent:main:old`, so the active session is evicted.
- Suggested fix: compare preserve keys using the normalized form of the store key, or preserve both raw and normalized values. For example, have `shouldPreserveMaintenanceEntry` check `preserveKeys.has(params.key) || preserveKeys.has(normalizeStoreSessionKey(params.key))`, with a regression test for a mixed-case active key.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The focused `node --import tsx --input-type=module` probe described above reproduces the active-entry eviction on current `main`.
- Is this the best way to solve the issue? yes. Normalizing the lookup side keeps the new case-insensitive provider behavior while restoring exact active-key protection for legacy keys.
