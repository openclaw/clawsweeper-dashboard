# Commit 4c210e2

## Summary

Found one concrete regression in the adaptive image compression path: model override compression metadata can diverge from the actual image fallback execution candidates.

## Findings

### Medium: Image compression drops configured fallbacks when a model override is used

- Kind: regression
- File: `src/agents/tools/image-tool.ts`
- Line: 302
- Evidence: `resolveCompressionModelCandidates()` chooses `overrideConfig ?? configuredImageModelConfig`, then writes that override-only config into `agents.defaults.imageModel` before calling `resolveImageFallbackCandidates()`. But execution later calls `runWithImageModelFallback()` with the original configured image model plus `modelOverride`, and `src/agents/model-fallback.ts:588` keeps the override first and configured `imageModel.fallbacks` after it. A read-only probe showed compression models only included `openai/gpt-5.5` limits while execution candidates were `openai/gpt-5.5` then `anthropic/claude-opus-4-6`.
- Impact: If the override model fails and the image tool falls back to a stricter configured model, the image may have been compressed for only the override model. For example, OpenAI `gpt-5.5` metadata allows `maxSidePx: 6000`, while Anthropic metadata in this commit allows `maxSidePx: 1568`, so fallback can receive an oversized image and fail instead of recovering.
- Suggested fix: Resolve compression candidates from the same effective candidate list used by `runWithImageModelFallback()`: keep `modelOverride` as the first candidate but preserve configured `imageModel.fallbacks` when building compression policy.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Configure `agents.defaults.imageModel.primary` as `openai/gpt-5.5`, `fallbacks` as `anthropic/claude-opus-4-6`, then call the image tool with `model: "openai/gpt-5.5"` and force the first attempt to fail. Compression policy only includes OpenAI metadata, while execution falls through to Anthropic.
- Is this the best way to solve the issue? yes. The safest direction is to make compression and execution share candidate resolution semantics rather than adding provider-specific fallback special cases.
