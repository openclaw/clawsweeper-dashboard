## Findings

### Low: `--order-by` ignores the requested limit until after materializing all matching rows

- Kind: reliability
- File: `extensions/memory-lancedb/index.ts`
- Line: 868
- Evidence: When `opts.orderBy` is set, the code intentionally skips `query.limit(limit)` at `extensions/memory-lancedb/index.ts:864` and then calls `await query.toArray()` before sorting and slicing at `extensions/memory-lancedb/index.ts:868-881`. LanceDB 0.27.2’s types document that a plain query has no default limit and that `toArray()` collects results as an array.
- Impact: `openclaw ltm query --order-by createdAt:desc --limit 10` can load every matching memory row into Node before returning 10 rows. A large local or object-storage-backed LanceDB table can make this slow or exhaust memory despite the user passing a small limit.
- Suggested fix: Avoid full materialization for ordered queries. Stream batches and keep a bounded top-K heap for the requested sort, add a hard scanned-row cap with a clear error, or use a native LanceDB order/sort API if one becomes available.
- Confidence: high
