# Commit 439d8ed

## Summary

Found two actionable issues in current `main`: one product-path mismatch for the new Codex direct-chat delivery default, and one isolated-test failure introduced around the same dispatch coverage.

## Findings

### Medium: Codex source delivery default can be missed on first auto-selected Codex turns

- Kind: regression
- File: `src/auto-reply/reply/dispatch-from-config.ts`
- Line: 309
- Evidence: `resolveHarnessSourceVisibleRepliesDefault` resolves the harness from `entry.modelProvider` first, but when the session entry is absent it falls back to `ctx.Provider` / `ctx.Surface`, which are channel ids such as `telegram`, not the configured model provider. Codex’s harness only supports provider id `codex`, and the new docs promise Codex source chat turns default to `message_tool` when `messages.visibleReplies` is unset.
- Impact: a new direct chat running Codex through auto provider selection can stay on automatic final-reply delivery for the first turn, instead of suppressing visible final output unless the agent calls `message(action="send")`. The added test only covers a pre-populated session entry with `agentHarnessId: "codex"`, so this first-turn path is not covered.
- Suggested fix: resolve the effective model provider/model from the same model-selection path used for the upcoming run when no session entry has provider/harness metadata, or persist/derive the harness id before source delivery policy is computed. Add a test with no existing `sessionStoreEntry.entry`, a Codex default model/provider, and a Codex harness that only supports `codex`.
- Confidence: medium

### Low: New dispatch harness-default test fails when run in isolation

- Kind: reliability
- File: `src/auto-reply/reply/dispatch-from-config.test.ts`
- Line: 3975
- Evidence: `pnpm test src/auto-reply/reply/dispatch-from-config.test.ts -t "uses harness defaults"` fails with `TypeError: Cannot read properties of undefined (reading 'threadId')` at `dispatch-from-config.ts:436`. The enclosing `sendPolicy deny` `beforeEach` does not initialize `threadInfoMocks.parseSessionThreadInfo`, while `dispatchReplyFromConfig` unconditionally reads `.threadId`.
- Impact: targeted test runs for the new coverage fail depending on suite ordering, making local repair/review loops unreliable.
- Suggested fix: initialize `threadInfoMocks.parseSessionThreadInfo.mockImplementation(parseGenericThreadSessionInfo)` in the `sendPolicy deny` `beforeEach`, or move the new test under the describe block that already owns that setup.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the test isolation issue with the exact command above. For the Codex first-turn default, the source path is concrete, but I did not add a temporary test in this report-only review.
- Is this the best way to solve the issue? For the test issue, yes: the mock setup belongs in that describe. For the Codex default, the safer fix is to use the actual pending run’s resolved provider/harness rather than channel provider fallback.
