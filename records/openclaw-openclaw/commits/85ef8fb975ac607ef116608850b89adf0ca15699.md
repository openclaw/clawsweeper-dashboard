# Commit 85ef8fb

## Summary

Found one actionable regression in the v2 Android notification settings: forwarding notification events is now gated on `POST_NOTIFICATIONS`, which is the wrong Android permission for reading/forwarding listener events.

## Findings

### Medium: Notification forwarding is blocked by the app-posting notification permission

- Kind: regression
- File: `apps/android/app/src/main/java/ai/openclaw/app/ui/V2SettingsScreens.kt`
- Line: 578
- Evidence: `setForwarding(true)` now requests `Manifest.permission.POST_NOTIFICATIONS` on API 33+ and only calls `viewModel.setNotificationForwardingEnabled(true)` when that permission is granted. The callback also writes the denial directly into `notificationForwardingEnabled`. But OpenClaw’s forwarding path is `DeviceNotificationListenerService`, and the legacy/current settings sheet treats “System Notifications” (`POST_NOTIFICATIONS`) separately from “Notification Listener Access”, with forwarding controlled by listener access. Android documents `POST_NOTIFICATIONS` as the runtime permission for sending app notifications, while `NotificationListenerService` is the separate listener-service API used for notification access. ([developer.android.google.cn](https://developer.android.google.cn/develop/ui/views/notifications/notification-permission?hl=en&utm_source=openai))
- Impact: On Android 13+, a user who grants Notification Listener Access but denies app notification posting cannot enable forwarded notification events from the v2 settings screen, even though the listener pipeline can read and forward notifications. This regresses the previous v2 toggle behavior and the older settings sheet’s permission split.
- Suggested fix: Do not gate `notificationForwardingEnabled` on `POST_NOTIFICATIONS`. For forwarding, require or open Notification Listener Access; keep `POST_NOTIFICATIONS` as a separate system-notifications/foreground-service permission control.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: on Android 13+, deny OpenClaw `POST_NOTIFICATIONS`, grant Notification Listener Access, then toggle v2 `Forward Notifications`; the callback keeps forwarding disabled despite listener access being available.
- Is this the best way to solve the issue? yes: forwarding should follow `DeviceNotificationListenerService.isAccessEnabled(context)`, not the app’s notification-posting permission.
