# Commit 03f7e26

## Summary

Found one low-severity logging correctness issue in the new startup model-mode details.

## Findings

### Low: Startup log can report `reasoning=off` for reasoning-capable model defaults

- Kind: bug
- File: `src/gateway/server-startup-log.ts`
- Line: 77
- Evidence: `formatAgentModelStartupDetails()` calls `resolveReasoningDefault({ provider, model })` without a catalog. `resolveReasoningDefault()` returns `"on"` only when the supplied catalog entry has `reasoning === true`; without a catalog, it always falls through to `"off"`. The runtime reply path explicitly loads/passes a model catalog before resolving model-capability reasoning defaults, and documents that behavior in code.
- Impact: For a default model whose catalog marks `reasoning: true`, with no explicit `agents.defaults.reasoningDefault` or per-agent `reasoningDefault`, the new startup line can say `reasoning=off` even though new replies can inherit `reasoning=on` when thinking is inactive. That undermines the operator-facing log this commit adds.
- Suggested fix: Either pass the same model catalog used by the reply model-selection path into the startup formatter, or avoid claiming a resolved model-capability reasoning default in startup logs when the catalog is unavailable.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: use a config with no explicit reasoning default and a default model whose loaded catalog entry has `reasoning: true`; the startup formatter has no catalog parameter, so its fallback resolves to `off`, while the reply path loads the catalog before resolving the model default.
- Is this the best way to solve the issue? unclear: passing the catalog is the most accurate fix, but it may have startup-cost implications; a safer low-cost alternative is to label/log only configured reasoning defaults unless catalog-backed resolution is available.
