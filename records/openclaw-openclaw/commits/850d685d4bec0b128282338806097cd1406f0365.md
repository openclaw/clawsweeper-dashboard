# Commit 850d685

## Summary

Found one concrete refresh-path risk in the new runtime-only native Codex CLI OAuth handling. The first app-server token refresh can succeed, but the rotated refresh token is not written to any durable or shared source, so later refreshes can reuse the old native CLI refresh token.

## Findings

### Medium: Native Codex CLI OAuth refresh drops the rotated refresh token

- Kind: reliability
- File: `extensions/codex/src/app-server/auth-bridge.ts`
- Line: 474
- Evidence: For runtime-only overlaid native Codex CLI OAuth, `resolveOAuthCredentialForCodexAppServer()` force-refreshes via `refreshOAuthCredentialForRuntime()` and returns the refreshed credential, but it does not save it or update the native Codex auth source. Codex CLI external auth is marked `bootstrapOnly`/`runtime-only` in `src/agents/auth-profiles/external-cli-sync.ts`, and `readCodexCliCredentials()` rereads the unchanged `auth.json`/keychain source in `src/agents/cli-credentials.ts`. The app-server refresh response only returns `accessToken`, `chatgptAccountId`, and `chatgptPlanType`, so the child app-server also does not receive the new refresh token. The pinned upstream `@earendil-works/pi-ai@0.74.0` `refreshOpenAICodexToken()` contract returns both a new `access` and a new `refresh`.
- Impact: A long-running Codex app-server session authenticated from native CLI OAuth can pass startup and the first `account/chatgptAuthTokens/refresh`, then fail on the next refresh by retrying the stale refresh token from the native CLI file/keychain. Concurrent refresh requests can also bypass the repo’s normal OAuth refresh lock/queue, the path explicitly documented in `src/agents/auth-profiles/constants.ts` as preventing `refresh_token_reused`.
- Suggested fix: Route this runtime-only Codex CLI refresh through the same locked OAuth manager path and persist/adopt the rotated credential into a canonical OpenClaw auth store, or update the native Codex credential source under an equivalent lock. Add a regression that calls `refreshCodexAppServerAuthTokens({ agentDir })` twice for native CLI OAuth and asserts the second call uses the first call’s rotated refresh token.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Unclear without live OAuth tokens. The code path is concrete, and the upstream refresh contract rotates refresh tokens, but I did not run a live multi-refresh Codex OAuth session.
- Is this the best way to solve the issue? Unclear. Persisting into OpenClaw’s auth store is the safest fit for existing refresh locking, but if OpenClaw must never persist native CLI auth, then it needs a locked runtime credential cache that survives repeated app-server refresh callbacks for the process lifetime.
