# Commit de186a8

## Summary

The commit fixes the exact `tools.byProvider["provider/model"].deny` false-positive from #80118, but the new audit path still does not model documented provider-scoped profiles. A small model restricted at runtime by `tools.byProvider["provider/model"].profile = "minimal"` is still reported as `critical` with web exposure.

## Findings

### Medium: small-model audit still ignores provider-scoped tool profiles

- Kind: bug
- File: `src/security/audit-extra.summary.ts`
- Line: 85
- Evidence: `resolveToolPolicies()` now resolves `tools.byProvider` entries, but pushes the returned config directly as a sandbox policy. That honors `allow`/`deny`, while ignoring `profile` and `alsoAllow`. Runtime policy resolution keeps `providerProfile` separate at `src/agents/pi-tools.policy.ts:488`, and docs define `tools.byProvider` order as `base profile -> provider profile -> allow/deny` in `docs/gateway/config-tools.md:67`.
- Impact: A documented mitigation such as `tools.profile = "coding"` plus `tools.byProvider["openrouter/google/gemma-3-4b-it:free"].profile = "minimal"` disables `web_search`, `web_fetch`, and `browser` at runtime, but the audit still reports `models.small_params` as `critical` with `web=[web_search, web_fetch]`. That keeps the security audit noisy for a valid provider/model restriction and can train operators to ignore the critical signal.
- Suggested fix: Reuse the runtime effective-policy stack for audit exposure checks, or explicitly include `providerProfile`/`providerProfileAlsoAllow` in `resolveToolPolicies()` before computing `isToolAllowedByPolicies(...)`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused `tsx` probe showed `collectSmallModelRiskFindings()` returning `critical` and `web=[web_search, web_fetch]` for the config above, while `resolveEffectiveToolPolicy()` resolved `providerProfile: "minimal"` and `isToolAllowedByPolicies()` returned false for `web_search`, `web_fetch`, and `browser`.
- Is this the best way to solve the issue? yes. The audit should not partially duplicate runtime provider policy semantics; sharing or mirroring the effective-policy stack avoids further drift.
