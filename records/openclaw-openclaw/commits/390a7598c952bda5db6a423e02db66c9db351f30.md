# Commit 390a7598

## Summary

Found one low-severity compatibility risk in the new `tools` reservation path.

## Findings

### Low: `tools` is now silently blocked as a plugin CLI root

- Kind: compatibility
- File: `src/cli/command-registration-policy.ts`
- Line: 4
- Evidence: The commit adds `RESERVED_NON_PLUGIN_COMMAND_ROOTS = new Set(["tools"])`, and `shouldSkipPluginCommandRegistration` now returns `true` for any primary command named `tools` before plugin CLI registration runs. In `run-main.ts:553`, that skip prevents `registerPluginCliCommandsFromValidatedConfig(...)` from loading plugin CLI registrars. The public plugin docs describe `api.registerCli(..., { commands/descriptors })` as owning root CLI commands, and the capture path accepts descriptor/command names without a corresponding `tools` reserved-name rejection.
- Impact: Any existing third-party plugin that owns a root CLI command named `tools` will now be unreachable as `openclaw tools ...`; the CLI will treat it as a reserved non-plugin root and fail as an unknown built-in path. This is likely rare, but it is a backwards-compatibility break that is not surfaced at plugin registration/install time.
- Suggested fix: Either narrow the fast path to the intended stray Gateway RPC invocations (`tools effective`, `tools catalog`, and help), or formally reserve `tools` in the plugin CLI contract and reject plugin CLI descriptors/commands named `tools` with a clear diagnostic.
- Confidence: medium
