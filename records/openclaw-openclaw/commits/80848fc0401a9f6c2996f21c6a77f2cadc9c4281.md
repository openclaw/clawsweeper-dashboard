# Commit 80848fc

## Summary

Found one low-severity config contract bug introduced while adding the typed `gateway.remote` schema shape: `gateway.remote.enabled` is now accepted by validation, but current remote consumers ignore it.

## Findings

### Low: `gateway.remote.enabled: false` validates but does not disable remote targeting

- Kind: bug
- File: `src/config/zod-schema.ts`
- Line: 59
- Evidence: `GatewayRemoteConfig` documents `enabled?: boolean` as controlling whether remote gateway surfaces are enabled, and the new schema now accepts it. Current call paths still choose `gateway.remote.url` based only on `gateway.mode` and URL presence, for example `src/commands/gateway-status/helpers.ts:98` and `src/gateway/connection-details.ts:36`. A live probe with `{ gateway: { mode: "remote", remote: { enabled: false, url: "wss://gateway.example.test" } } }` returned validation `ok: true` and an active `configRemote` target.
- Impact: A config author can now set `gateway.remote.enabled: false` and still have CLI/status/call paths use the configured remote URL and remote credentials. That is a misleading accepted config state, and could cause unexpected remote connections.
- Suggested fix: Either remove `enabled` from the accepted core schema until it is implemented, or honor `remote.enabled === false` consistently in remote URL/target/auth resolution paths.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: validate a config with `gateway.mode="remote"`, `gateway.remote.enabled=false`, and `gateway.remote.url`, then call `resolveTargets`; the remote target remains active.
- Is this the best way to solve the issue? unclear: if `enabled` is intended public config, consumers should honor it; if it was only pulled in by the type-backed schema shape, rejecting it is the smaller fix.
