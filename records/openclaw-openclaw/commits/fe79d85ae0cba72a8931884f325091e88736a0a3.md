# Commit fe79d85

## Summary

Found one actionable security boundary issue: iMessage group-management actions are advertised and executable by default without an owner/trusted-sender check.

## Findings

### Medium: iMessage group-management actions are enabled without sender authorization

- Kind: security
- File: `extensions/imessage/src/actions.ts`
- Line: 502
- Evidence: `IMESSAGE_ACTIONS` includes `renameGroup`, `setGroupIcon`, `addParticipant`, `removeParticipant`, and `leaveGroup` as normal action gates, and `createActionGate` defaults unset gates to enabled. The new discovery test explicitly expects those actions with `cfg()` and no action config at `extensions/imessage/src/actions.test.ts:96`. Dispatch only enforces trusted sender identity when a plugin declares `requiresTrustedRequesterSender`, but the iMessage action adapter does not declare that hook and `handleAction` ignores `requesterSenderId`/`senderIsOwner`; it proceeds to call the group mutation runtime paths at `extensions/imessage/src/actions.ts:502`, `extensions/imessage/src/actions.ts:531`, and `extensions/imessage/src/actions.ts:554`.
- Impact: Any accepted iMessage group participant whose prompt causes the agent to call the `message` tool can ask the owner’s Messages account to rename a group, change its icon, add/remove participants, or leave the group when the private API bridge is enabled. That is a larger permission surface than normal reply/reaction actions.
- Suggested fix: Treat group-management actions as privileged. Either default these gates to false, hide them unless `senderIsOwner === true`, or add `requiresTrustedRequesterSender` plus execution-time owner/trusted-sender enforcement before invoking the bridge.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the static dispatch path: configure iMessage with private API available, enter an iMessage group context, and the message tool advertises group-management actions by default. A live macOS/iMessage bridge was not available here to execute an end-to-end group mutation.
- Is this the best way to solve the issue? Unclear. The safest fix is an owner/trusted-sender gate plus discovery hiding for non-owner contexts; default-off group-management gates would also reduce risk but is a product decision.
