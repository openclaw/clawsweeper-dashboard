# Commit 205c595

## Summary

Found one concrete regression in the new missing-refresh-token short-circuit: it returns `null` before the existing Codex CLI fallback path can recover a same-account fresh CLI credential when the stored OAuth profile still has stale inline access material but no refresh token.

## Findings

### Medium: Missing-refresh guard bypasses Codex CLI fallback for access-only OAuth records

- Kind: regression
- File: `src/agents/auth-profiles/oauth-manager.ts`
- Line: 537
- Evidence: The new guard returns `null` whenever `credentialToRefresh.refresh` normalizes to missing. The existing fallback recovery only runs in the `catch` path at `src/agents/auth-profiles/oauth-manager.ts:752`, so this new non-error return skips `readFallbackCredential`. For OpenAI Codex, bootstrap also does not cover stale access-only records because `readExternalCliBootstrapCredential` refuses bootstrap-only providers when the local credential has any inline token material at `src/agents/auth-profiles/external-cli-sync.ts:164`. A focused smoke check with expired `access`, empty `refresh`, matching `accountId`, and a same-account fresh fallback produced `{"result":null,"calls":{"refresh":0,"fallback":0}}`.
- Impact: A stored OpenAI Codex OAuth profile with stale access material but missing refresh material can now fail as missing auth even when a fresh same-account Codex CLI credential is available. Before this guard, the refresh attempt would error and the existing fallback path could use that CLI credential, as covered for refresh-failure cases in `src/agents/auth-profiles/oauth.openai-codex-refresh-fallback.test.ts:468`.
- Suggested fix: Preserve the fail-fast behavior, but run the same fallback recovery before returning `null` for missing refresh material, or route the missing-refresh condition through a local recoverable error path that reaches the existing fallback logic. Keep the guard after main-store adoption and bootstrap as it is now.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The focused `node --import tsx --input-type=module` smoke check exercised `createOAuthManager` with a stale access-only OAuth credential and a matching fresh fallback, and current main returned `null` without calling refresh or fallback.
- Is this the best way to solve the issue? yes. The bug is not that refresh is skipped; it is that the already-supported fallback is skipped too. Factoring fallback recovery before the `return null` keeps the intended no-refresh fail-fast path without losing the same-account external CLI recovery contract.
