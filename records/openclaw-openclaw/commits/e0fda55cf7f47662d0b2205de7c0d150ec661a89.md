## Summary

Found two actionable issues in the new Upstash Box install guide. The rest of the docs/navigation changes and the system-prompt MEDIA wording look consistent with current code and docs checks.

## Findings

### Medium: SSH-tunnel guide widens the Gateway bind to all interfaces

- Kind: security
- File: `docs/install/upstash.md`
- Line: 61
- Evidence: The page says to use an SSH tunnel and not expose the Gateway port directly (`docs/install/upstash.md:12`, `docs/install/upstash.md:35`), but then tells users to run `openclaw config set gateway.bind lan`. In current code, `gateway.bind: "lan"` resolves to `0.0.0.0` (`src/gateway/net.ts:287`), and the configuration reference documents `lan` as `0.0.0.0` (`docs/gateway/configuration-reference.md:529`). The new exposure runbook in the same commit says the SSH-tunnel pattern should keep `gateway.bind: "loopback"` (`docs/gateway/security/exposure-runbook.md:26`).
- Impact: Operators following the Upstash guide can unnecessarily expose the Gateway listener on every Box network interface instead of only the loopback endpoint used by the SSH tunnel. Gateway auth mitigates this, but it still widens the network attack surface and contradicts the new security runbook.
- Suggested fix: Remove the `gateway.bind lan` step for the SSH-tunnel path, or set/keep `gateway.bind loopback`. If Upstash requires a non-loopback bind for a separate provider-specific reason, document that reason and add the expected firewall/auth verification steps.
- Confidence: high

### Low: The Upstash flow mixes daemon install with a manual nohup gateway

- Kind: reliability
- File: `docs/install/upstash.md`
- Line: 51
- Evidence: The guide runs `openclaw onboard --install-daemon` (`docs/install/upstash.md:51`) and then starts `nohup openclaw gateway` manually (`docs/install/upstash.md:62`) and asks users to add the same `nohup` command as an init script (`docs/install/upstash.md:77`). On Linux where daemon install succeeds, OpenClaw’s systemd install path enables and restarts the service (`src/daemon/systemd.ts:946`, `src/daemon/systemd.ts:951`), so the later unmanaged `nohup` gateway can contend with the already-started managed service on the same port/runtime lock.
- Impact: On systems with working user systemd, users can end up with a failed manual start, confusing logs, or mixed managed/unmanaged lifecycle instructions.
- Suggested fix: Pick one lifecycle path. For the Upstash init-script/nohup path, use onboarding without daemon install, or condition the manual `nohup` steps on daemon install being unavailable/skipped. If daemon install is the intended path, replace the manual commands with daemon status/restart guidance.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the bind issue: the docs command sets `gateway.bind` to `lan`, and current code resolves that mode to `0.0.0.0`. For the daemon/nohup issue, reproduction depends on a Linux environment where user systemd is available.
- Is this the best way to solve the issue? Yes: keep the SSH-tunnel guide loopback-only, and make the lifecycle instructions choose either daemon management or manual init-script management.
