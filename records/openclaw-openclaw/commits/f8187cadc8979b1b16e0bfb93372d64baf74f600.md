# Commit f8187ca

## Summary

Found one regression in configured-provider inference for bare legacy Gemini IDs.

## Findings

### Medium: Bare retired Gemini IDs no longer infer the configured Google provider

- Kind: regression
- File: `src/agents/model-selection-shared.ts`
- Line: 126
- Evidence: The commit changed configured provider matching to compare only `normalizedModelId` against the raw requested bare model. With config containing `models.providers.google.models: [{ id: "gemini-3-pro-preview" }]` and `agents.defaults.model.primary: "gemini-3-pro-preview"`, current `resolveConfiguredModelRef({ defaultProvider: "openai" })` returns `{"provider":"openai","model":"gemini-3-pro-preview"}` and logs a fallback warning. Before this change, the raw configured catalog ID matched and inferred `google`.
- Impact: A previously supported bare configured-provider model can be rerouted to `openai/gemini-3-pro-preview`, which is not the configured Google model and is likely to fail model selection or inference.
- Suggested fix: In the configured-provider loop, compare both the raw and canonical configured ID against both the raw and provider-normalized requested model. When a bare model is resolved via provider-specific normalization, return/use the canonical model ID, e.g. `google/gemini-3.1-pro-preview`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: `pnpm exec tsx -e ...` with the config above produced `infer old bare = undefined` and `default = {"provider":"openai","model":"gemini-3-pro-preview"}`.
- Is this the best way to solve the issue? no: canonicalizing configured catalog rows is reasonable, but the lookup needs to preserve legacy bare alias matching while canonicalizing the selected model.
