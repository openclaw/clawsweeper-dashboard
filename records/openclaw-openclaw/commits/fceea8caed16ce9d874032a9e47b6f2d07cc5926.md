# Commit fceea8c

## Summary

One low-severity compatibility regression was found in the public SDK test-helper typing. Runtime behavior and the touched proxy tests look clean.

## Findings

### Low: Provider HTTP mock helper now rejects typed mock implementations

- Kind: compatibility
- File: `src/plugin-sdk/test-helpers/provider-http-mocks.ts`
- Line: 23
- Evidence: The documented `getProviderHttpMocks()` helper returns mocks typed through `AnyMock = Mock<(...args: unknown[]) => unknown>`. With Vitest’s `Mock` typing, `mockImplementation` must accept `unknown` arguments, so normal typed implementations like `postJsonRequestMock.mockImplementation((request: { url: string }) => ...)` now fail with TS2345. The previous base type, `Mock<(...args: any[]) => any>`, accepted the same implementation.
- Impact: Third-party plugin tests using the documented `openclaw/plugin-sdk/provider-http-test-mocks` helper can fail typecheck after upgrading, even though their runtime mock usage is valid.
- Suggested fix: Prefer exact mock signatures for `postJsonRequest`, `fetchWithTimeout`, and related helpers. If exact generic signatures are too awkward, keep flexible argument typing on the public mock surface while tightening return types separately.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused `tsgo` probe importing `getProviderHttpMocks()` and calling `postJsonRequestMock.mockImplementation((request: { url: string }) => ...)` fails with TS2345 on current `main`; the equivalent old `Mock<(...args: any[]) => any>` probe passes.
- Is this the best way to solve the issue? yes. Real function signatures preserve type safety without narrowing the mock helper into an unusable `unknown[]` implementation contract.
