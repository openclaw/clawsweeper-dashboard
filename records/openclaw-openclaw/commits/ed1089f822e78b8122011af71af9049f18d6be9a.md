# Commit ed1089f

## Summary

Found one low-severity test-script regression: the new `bash -lc` wrapper sources Testbox auth correctly, but it drops extra arguments passed through `pnpm ... -- ...`.

## Findings

### Low: `test:plugins:kitchen-sink-live` no longer forwards extra QA flags

- Kind: regression
- File: `package.json`
- Line: 1617
- Evidence: The parent script was a direct `pnpm openclaw qa suite ...` invocation, so appended script args reached the QA CLI. The new `bash -lc '...'` wrapper never references `"$@"`; a focused dry run with a fake inner `pnpm` showed `pnpm test:plugins:kitchen-sink-live -- --output-dir custom-artifacts --allow-failures` now invokes only `openclaw qa suite --provider-mode live-frontier --scenario kitchen-sink-live-openai`, while the parent script included the appended `--output-dir custom-artifacts --allow-failures`.
- Impact: Maintainers can no longer use the convenience script to pass common QA controls such as custom output directories, `--allow-failures`, or other scenario-runner flags. That makes the live Kitchen Sink lane less debuggable and can turn intended artifact-collection runs into hard failures.
- Suggested fix: Preserve positional args through the wrapper, ideally in a small script, or add a bash sentinel and forward `"$@"`, for example building the QA argv in bash and `exec`ing either `openclaw-testbox-env "${cmd[@]}"` or `"${cmd[@]}"`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: run the package script with extra args and a fake inner `pnpm`; the current script drops appended flags, while the parent script forwards them.
- Is this the best way to solve the issue? yes: keep the Testbox helper behavior but preserve npm/pnpm script argument forwarding.
