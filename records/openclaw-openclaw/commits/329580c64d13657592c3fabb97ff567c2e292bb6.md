# Commit 329580c

## Summary

Found one regression in the new stale-config reinstall fallback: the disabled-config guard only checks the channel id, but catalog installs enable the catalog/plugin manifest id. That can re-enable a plugin explicitly disabled under `plugins.entries.<pluginId>.enabled=false` when `pluginId !== channelId`.

## Findings

### Medium: Catalog fallback can bypass plugin-id disables for external channel plugins

- Kind: regression
- File: `src/flows/channel-setup.ts`
- Line: 603
- Evidence: Both new fallback paths call `resolveConfigDisabledHint(channel)` before installing, but that helper only checks `plugins.entries[channel]?.enabled === false`. Catalog entries carry a separate `pluginId`, and repo tests already document third-party channel plugins where `pluginId !== channelId`. The install flow then enables `result.pluginId`, and `enablePluginInConfig` does not preserve an existing `plugins.entries[pluginId].enabled=false`; it writes the plugin enabled unless global plugins, denylist, or allowlist block it.
- Impact: A config with stale `channels.<channel>` data and `plugins.entries.<actual-plugin-id>.enabled=false` can still run the new catalog reinstall path and flip the disabled plugin back on. That violates the guard this commit intended to preserve for operator-disabled channels/plugins.
- Suggested fix: Make the guard check the catalog/plugin id as well as the channel id before both fallback installs, for example `entry.pluginId ?? channel`, and add regression tests using `pluginId: "@vendor/external-chat-plugin"` with `plugins.entries["@vendor/external-chat-plugin"].enabled = false`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. Adapt the new disabled fallback tests to disable `plugins.entries["@vendor/external-chat-plugin"]` instead of `plugins.entries["external-chat"]`; the new guard misses it, then `ensureChannelSetupPluginInstalled` enables the plugin id.
- Is this the best way to solve the issue? Yes. The intended policy is already expressed in adjacent code that checks `entry.pluginId ?? entry.id` for plugin-disabled status; the setup fallback should use the same id.
