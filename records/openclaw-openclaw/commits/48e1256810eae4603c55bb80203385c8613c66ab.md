# Commit 48e1256

## Summary

Found one CI reliability bug: the new live-image build does not feed the image that the OpenWebUI lane actually runs.

## Findings

### Medium: OpenWebUI still runs the unbuilt E2E image instead of the newly built live image

- Kind: reliability
- File: `.github/workflows/openclaw-live-and-e2e-checks-reusable.yml`
- Line: 1195
- Evidence: The OpenWebUI-only job exports `OPENCLAW_DOCKER_E2E_IMAGE` from `prepare_docker_e2e_image.outputs.image` at `.github/workflows/openclaw-live-and-e2e-checks-reusable.yml:1101`, but the OpenWebUI plan needs only `needs_live_image=1` and `needs_functional_image=0`. The new step builds the default live image through `test-live-build-docker.sh`, whose default tag is `openclaw:local-live` (`scripts/test-live-build-docker.sh:8`). The lane then runs `pnpm test:docker:openwebui`; that script resolves its gateway image from `OPENCLAW_DOCKER_E2E_IMAGE` before its default (`scripts/lib/docker-e2e-image.sh:28`, `scripts/e2e/openwebui-docker.sh:9`). So the built `openclaw:local-live` image is not the image OpenWebUI consumes.
- Impact: OpenWebUI-only release checks can still fail for a new SHA because no functional E2E image is built or pulled for that plan, while the locally built live image is unused. In mixed release-path chunks, OpenWebUI can also silently run against the functional E2E image instead of the live image this commit tried to prepare.
- Suggested fix: Make the built live image and OpenWebUI’s consumed image the same image. For example, set/export an `OPENCLAW_OPENWEBUI_E2E_IMAGE` or `OPENCLAW_LIVE_IMAGE` value before both the prebuild and scheduler run, or update the OpenWebUI lane/script so live lanes do not inherit `OPENCLAW_DOCKER_E2E_IMAGE` unless they explicitly need a functional/bare E2E image.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. A focused planner check for the OpenWebUI chunk returns `needs_live_image=1`, `needs_functional_image=0`, and `needs_e2e_image=0`; static tracing then shows the workflow builds `openclaw:local-live` but the OpenWebUI script uses `OPENCLAW_DOCKER_E2E_IMAGE`.
- Is this the best way to solve the issue? No. The fix should wire the image name through to the lane that consumes it, not only build a default-tagged live image beforehand.
