# Commit 5d12774

## Summary

Found two actionable regressions in the new unconditional Anthropic 1M context override. The resolver now upgrades too many Claude 4 model IDs, and it also bypasses explicit `contextTokens` runtime caps for the supported models.

## Findings

### Medium: Broad Claude 4 prefix reports 1M for Sonnet 4.5

- Kind: bug
- File: `src/agents/context.ts`
- Line: 459
- Evidence: `ANTHROPIC_1M_MODEL_PREFIXES` is `["claude-opus-4", "claude-sonnet-4"]`, and `isAnthropic1MModel()` uses `startsWith()`. After this commit, any explicit Anthropic/Claude CLI model matching that broad prefix returns `1_048_576` before config/cache lookup. A focused check on current `main` prints `claude-sonnet-4-5=1048576`. Anthropic’s current context-window docs say Sonnet 4.5/Sonnet 4 moved out of the 1M beta and the old 1M beta header has no effect for those models. citeturn6search0
- Impact: Users on `anthropic/claude-sonnet-4-5` can get `/status`, session persistence, and compaction thresholds based on a 1M window even though the model is effectively 200k. That can delay compaction and produce over-context request failures.
- Suggested fix: Replace the broad prefixes with an exact allowlist/prefix set for the currently 1M models, such as Opus 4.7, Opus 4.6, and Sonnet 4.6 variants, and add a regression test for `claude-sonnet-4-5`.
- Confidence: high

### Medium: Forced 1M bypasses explicit `contextTokens` caps

- Kind: regression
- File: `src/agents/context.ts`
- Line: 459
- Evidence: The new early return runs before `resolveConfiguredProviderContextTokens()`, so a config entry like `models.providers.anthropic.models[{ id: "claude-sonnet-4-6", contextWindow: 1048576, contextTokens: 64000 }]` resolves to `1048576`. Current docs define model-level `contextTokens` as the optional runtime cap used when a smaller effective context budget is wanted (`docs/gateway/config-tools.md:474`), and existing tests preserve that contract for non-Anthropic providers (`src/agents/context.test.ts:463`).
- Impact: Users or generated configs that cap Anthropic runtime budget below the native window, for proxy/server limits or cost/latency control, silently lose that cap for Opus/Sonnet 4 models. Memory flush, model selection, status, and session persistence all consume this resolver.
- Suggested fix: Split configured context lookup so explicit `contextTokens` caps are honored before the Anthropic default upgrade, while stale `contextWindow: 200000` metadata can still be upgraded for the fixed issue.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. I reproduced both with focused `pnpm exec tsx -e ...` resolver checks, and verified the existing focused context tests pass.
- Is this the best way to solve the issue? yes for direction: use a narrower Anthropic 1M model predicate and preserve explicit runtime caps before applying model-family defaults.
