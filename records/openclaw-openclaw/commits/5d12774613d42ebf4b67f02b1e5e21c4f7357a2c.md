# Commit 5d127746

## Summary

Found one low-severity regression in the new Anthropic 1M context reporting path: it bypasses explicit configured `contextTokens`/`contextWindow` caps for Anthropic Opus/Sonnet 4 refs.

## Findings

### Low: Anthropic 1M shortcut ignores explicit configured context caps

- Kind: regression
- File: `src/agents/context.ts`
- Line: 459
- Evidence: The new early return runs before `resolveConfiguredProviderContextTokens(...)`, so any explicit `models.providers.anthropic.models[].contextTokens` or `contextWindow` is ignored when the model id starts with `claude-opus-4` or `claude-sonnet-4`. A focused probe on current `main` with `claude-opus-4-6` configured as `contextWindow: 1048576, contextTokens: 160000` returned `1048576` instead of the configured runtime cap. Existing guard code treats `models.providers.*.models[].contextTokens` as the preferred configured limit (`src/agents/context-window-guard.ts:34`) and tests that it wins over `contextWindow` (`src/agents/context-window-guard.test.ts:98`).
- Impact: Operators who intentionally cap an Anthropic model below its native window, or who configure a lower effective window for account/backend constraints, now get session/status/UI denominators that overstate the usable context. The Control UI context notice uses `session.contextTokens` as the warning/compact denominator (`ui/src/ui/chat/context-notice.ts:69`), so a 160k-capped session at 145k tokens would no longer show the near-limit warning.
- Suggested fix: Preserve explicit `contextTokens` as a hard cap before the Anthropic auto-upgrade. If the intent is to ignore only stale generated `contextWindow: 200000` metadata for known 1M Anthropic models, narrow the shortcut to that case and do not bypass explicit `contextTokens`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Run a resolver probe with an explicit Anthropic model config containing `contextTokens: 160000`; current `main` returns `1048576`.
- Is this the best way to solve the issue? yes. The existing config contract already distinguishes native `contextWindow` from effective `contextTokens`, so respecting `contextTokens` before the 1M reporting shortcut preserves the intended fix while keeping operator caps meaningful.
