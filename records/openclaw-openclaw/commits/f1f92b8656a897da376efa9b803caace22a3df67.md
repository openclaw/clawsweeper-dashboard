# Commit f1f92b8

## Summary

Found two lifecycle races introduced by the new reconnect behavior. The focused Android gateway tests pass, but they do not cover disconnect/connect overlap or the existing `connect()` plus `reconnect()` caller path.

## Findings

### Medium: Disconnect can race with a new connect and leave stale lifecycle callbacks behind

- Kind: concurrency
- File: `apps/android/app/src/main/java/ai/openclaw/app/gateway/GatewaySession.kt`
- Line: 198
- Evidence: `disconnect()` now sets `job = null` inside the lifecycle lock before the old `jobToCancel` is cancelled, then launches cancellation asynchronously at lines 201-207. A quick following `connect()` sees `job?.isActive != true` and starts a new `runLoop()` at lines 181-182 while the old loop may not have been cancelled yet. The stale disconnect task also calls `onDisconnected("Offline")` unconditionally at line 207 even when `desired` was repopulated by a new connect; `NodeRuntime` uses that callback to mark operator/node state disconnected in `apps/android/app/src/main/java/ai/openclaw/app/NodeRuntime.kt:333` and `apps/android/app/src/main/java/ai/openclaw/app/NodeRuntime.kt:370`.
- Impact: A rapid disconnect followed by reconnect or switching gateways can briefly or persistently clobber the new session’s UI/runtime state back to `Offline`, and can also allow the old run loop to race the new run loop against the same `desired` connection.
- Suggested fix: Cancel the old job before exposing `job = null`, or add a connection generation token so old disconnect jobs and old run loops cannot act on a newer `desired` connection. Also gate the final `onDisconnected("Offline")` on the same generation or `desired == null`.
- Confidence: medium

### Low: Existing reconnect caller can close the replacement connection that `connect()` now starts

- Kind: reliability
- File: `apps/android/app/src/main/java/ai/openclaw/app/gateway/GatewaySession.kt`
- Line: 185
- Evidence: `connect()` now closes the active connection itself at line 185. The existing refresh path still calls `operatorSession.connect(...)` / `nodeSession.connect(...)` and then immediately calls `operatorSession.reconnect()` / `nodeSession.reconnect()` when `reconnect = true` in `apps/android/app/src/main/java/ai/openclaw/app/NodeRuntime.kt:1138` and `apps/android/app/src/main/java/ai/openclaw/app/NodeRuntime.kt:1155`. If the run loop has already assigned the replacement at `GatewaySession.kt:977`, the following `reconnect()` closes `currentConnection` again at lines 211-213.
- Impact: Manual or foreground reconnect can close the fresh replacement websocket during startup, producing an avoidable reconnect cycle, transient gateway error, or backoff delay.
- Suggested fix: Remove the post-`connect()` `reconnect()` calls for this path now that `connect()` restarts the session, or make `connect()`/`reconnect()` share a generation-aware restart primitive.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Unclear; both issues depend on coroutine scheduling around websocket close/resume. The code paths are concrete, but the existing tests do not force the stale job/callback interleavings.
- Is this the best way to solve the issue? Unclear; the safest direction is to serialize lifecycle generations so stale jobs cannot affect newer connects.
