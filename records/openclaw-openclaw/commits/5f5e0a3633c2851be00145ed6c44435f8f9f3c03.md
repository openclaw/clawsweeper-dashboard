## Findings

### Low: Reported `terminated` embedding transport errors still abort without retry

- Kind: bug
- File: `extensions/memory-core/src/memory/manager-embedding-policy.ts`
- Line: 84
- Evidence: The new regex matches `fetch failed`, `UND_ERR_*`, `ECONNRESET`, `socket hang up`, and timeout strings, but not `terminated`. A focused probe on current `main` returned `false` for `isRetryableMemoryEmbeddingError("TypeError: terminated")`. The linked issue #44166 lists `terminated` alongside `TypeError: fetch failed`, `ECONNRESET`, `socket hang up`, and `other side closed` as transient embedding transport failures. The batch embedding paths pass this classifier directly into `runMemoryEmbeddingRetryLoop`, so a `terminated` batch error still throws without retry from `extensions/memory-core/src/memory/manager-embedding-ops.ts:299` and `extensions/memory-core/src/memory/manager-embedding-ops.ts:331`.
- Impact: Large remote memory reindex runs can still abort mid-run on one of the reported transient socket failure strings, losing the benefit of this retry fix for that case.
- Suggested fix: Extend the classifier and tests to include narrowly scoped connection-termination forms, at least `terminated`; consider `premature close` / `connection closed` if maintainers want the changelog’s broader “closed” wording to match behavior.
- Confidence: medium
