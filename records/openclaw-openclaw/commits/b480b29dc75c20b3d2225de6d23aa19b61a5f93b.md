# Commit b480b29

## Summary

One low-severity formatting bug found: the new Telegram HTML preservation allowlist does not match Telegram’s supported HTML contract, so some supported raw HTML is still escaped while one unsupported `tg-time` variant is preserved.

## Findings

### Low: Telegram HTML allowlist omits supported tags and allows an unsupported `tg-time` attribute

- Kind: bug
- File: `extensions/telegram/src/format.ts`
- Line: 183
- Evidence: `TELEGRAM_ATTR_HTML_TAG_PATTERNS` only allows `a href`, `span class="tg-spoiler"`, `tg-emoji emoji-id`, and `tg-time datetime`; `TELEGRAM_SIMPLE_HTML_TAGS` also requires zero attributes for `code` and `blockquote`. Telegram’s Bot API formatting docs list supported HTML that includes `<pre><code class="language-python">...</code></pre>`, `<blockquote expandable>...</blockquote>`, and `<tg-time unix="1647531900" format="t">...</tg-time>`, not `datetime`. ([core.telegram.org](https://core.telegram.org/bots/api%0A?utm_source=openai)) A focused probe confirms the formatter escapes those supported forms and preserves the unsupported `datetime` form:
  - `<pre><code class="language-python">x</code></pre>` renders as `<pre>&lt;code class="language-python"&gt;x&lt;/code&gt;</pre>`
  - `<blockquote expandable>x</blockquote>` renders as `&lt;blockquote expandable&gt;x&lt;/blockquote&gt;`
  - `<tg-time unix="1647531900" format="t">22:45</tg-time>` renders escaped
  - `<tg-time datetime="2026-05-12T00:00:00Z">today</tg-time>` is preserved
- Impact: Replies using Telegram-supported raw HTML still show literal tags instead of formatted output, while the unsupported `tg-time datetime` case can still trigger Telegram parse fallback. This undercuts the commit’s documented “Supported Telegram HTML tags are preserved” behavior.
- Suggested fix: Update the allowlist to match the Bot API contract: allow `code class="language-..."` only in the nested `pre > code` case, allow `blockquote expandable`, allow `tg-time unix="..."` with optional `format="..."`, and remove `datetime`. Add regression tests for those exact forms.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Run the formatter probe against `markdownToTelegramHtml` or `renderTelegramHtmlText` with the three supported Telegram examples above.
- Is this the best way to solve the issue? yes. The bug is in the local supported-tag contract, so fixing the allowlist and adding contract tests is narrower than relying on Telegram parse fallback.
