# Commit 7175b1b

## Summary

Found one medium regression in the startup catch-up path. The new future-slot repair is disabled while `runMissedJobs()` writes deferred overflow slots, but `start()` immediately runs another maintenance recompute with repair enabled, so non-agent cron overflow deferrals can be rewritten before the timer gets a chance to execute them.

## Findings

### Medium: Startup overflow cron deferrals are repaired before they can run

- Kind: regression
- File: `src/cron/service/jobs.ts`
- Line: 616
- Evidence: `applyStartupCatchupOutcomes()` writes overflow deferrals at `src/cron/service/timer.ts:1308` and explicitly disables future repair at `src/cron/service/timer.ts:1317`, but `start()` immediately calls `recomputeNextRunsForMaintenance(state, { recomputeExpired: true })` at `src/cron/service/ops.ts:181`. Because this commit defaults `repairFutureCronNextRunAtMs` to `true` and repairs non-agent cron slots before the natural next slot at `src/cron/service/jobs.ts:220`, that second pass undoes the startup deferral. A source probe with three overdue hourly `systemEvent` cron jobs, `maxMissedJobsPerRestart: 1`, and `missedJobStaggerMs: 5000` showed deferred slots at `17:00:05` and `17:00:10` immediately after `runMissedJobs()`, then `17:01:12.401` and `17:03:36.458` after the same recompute that `start()` runs.
- Impact: When more non-agent cron jobs are missed than `maxMissedJobsPerRestart`, overflow jobs no longer reliably catch up shortly after startup. For less frequent schedules this can skip the intended missed-run catch-up until the next natural schedule, such as the next day.
- Suggested fix: Preserve startup deferrals through the post-catchup `start()` maintenance pass, for example by passing `repairFutureCronNextRunAtMs: false` there when startup catch-up just ran or by making `runMissedJobs()` report that it wrote deferrals. Add a `start()`-level regression test; the current direct `runMissedJobs()` test does not cover the second recompute.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The read-only `tsx` probe above follows the same sequence as `start()`: `runMissedJobs()` then `recomputeNextRunsForMaintenance({ recomputeExpired: true })`, and shows the deferred slots being rewritten.
- Is this the best way to solve the issue? yes. The repair should still run in ordinary maintenance paths, but startup catch-up needs a scoped exemption until its intentionally deferred slots fire.
