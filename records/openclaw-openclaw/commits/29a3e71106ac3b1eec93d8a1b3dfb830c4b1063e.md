# Commit 29a3e711

## Summary

Found one concrete gap in the failure isolation: normal tracked plugin update failures are now disabled/skipped, but earlier externalized bundled-plugin channel-sync install failures still make the whole update fail.

## Findings

### Medium: Externalized plugin sync failures still fail the OpenClaw update

- Kind: reliability
- File: `src/cli/update-cli/update-command.ts`
- Line: 1201
- Evidence: `updatePluginsAfterCoreUpdate` still returns plugin post-update `status: "error"` whenever `syncResult.summary.errors.length > 0`. Those errors are produced in `syncPluginsForUpdateChannel` when an externalized bundled plugin install fails, for example `installPluginFromNpmSpec`/`installPluginFromClawHub` returning `ok: false` at `src/plugins/update.ts:1596`. The new `disableOnFailure: true` is only passed to the later `updateNpmInstalledPlugins` calls at `src/cli/update-cli/update-command.ts:1130` and `src/cli/update-cli/update-command.ts:1151`, so it cannot isolate failures that happen in the preceding sync phase.
- Impact: A user updating after an enabled bundled plugin has moved to an external package can still get `Update failed during plugin post-update sync` and a non-zero update result if that plugin package is unavailable or times out. This is the same class of “broken plugin package fails an otherwise successful OpenClaw update” behavior the commit is trying to isolate, and it can also prevent the normal post-update restart path.
- Suggested fix: Extend the same disable/skip behavior to externalized bundled-plugin sync install failures, or have `updatePluginsAfterCoreUpdate` translate non-security sync install failures into disabled plugin entries instead of `sync.errors`. Keep explicit integrity-drift/security failures fail-closed if that is the intended contract.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The existing `src/plugins/update.test.ts` case “leaves config unchanged when externalized plugin installation fails” proves `syncPluginsForUpdateChannel` returns `summary.errors` for a failed externalized npm install; `src/cli/update-cli/update-command.ts:1201` then maps any such sync error to post-update `status: "error"`.
- Is this the best way to solve the issue? unclear. Disabling the plugin matches this commit’s tracked-plugin update path, but integrity drift may deserve a separate fatal path.
