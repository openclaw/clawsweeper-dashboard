# Commit bb294bc

## Summary

Found one release-automation bug: adding `alpha` as a selectable npm dist-tag also permits stable and stable-correction OpenClaw releases to publish to `openclaw@alpha`, despite the documented lane rules saying only alpha prerelease tags should publish there.

## Findings

### Medium: Stable OpenClaw releases can be published to the alpha npm dist-tag

- Kind: bug
- File: `scripts/openclaw-npm-release-check.ts`
- Line: 204
- Evidence: `resolveNpmPublishPlan("2026.5.2", undefined, "alpha")` currently returns `{ channel: "stable", publishTag: "alpha", mirrorDistTags: [] }`. The workflow inputs now expose `npm_dist_tag=alpha` in `.github/workflows/openclaw-npm-release.yml:24` and `.github/workflows/openclaw-release-publish.yml:19`, but their guards only reject alpha prerelease tags with non-alpha tags and beta prerelease tags with non-beta tags. They do not reject stable or correction tags with `alpha`. The docs added in `docs/reference/RELEASING.md:626` say stable/correction tags may publish only to `beta` or `latest`, and `docs/reference/RELEASING.md:627` says alpha prerelease tags may publish only to `alpha`.
- Impact: A manual official release dispatch for a stable tag such as `v2026.5.2` with `npm_dist_tag=alpha` can publish the stable package under the `alpha` dist-tag. That pollutes the alpha install lane and makes `openclaw@alpha` resolve to a stable release until corrected.
- Suggested fix: In `resolveNpmPublishPlan`, reject `publishTag === "alpha"` unless `parsedVersion.channel === "alpha"`. Mirror that guard in both workflow validation blocks and add a focused test that stable and stable-correction versions reject requested `alpha`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The focused helper probe showed stable and correction versions resolve to `publishTag: "alpha"` when requested, while alpha prereleases correctly reject `beta`.
- Is this the best way to solve the issue? yes. Keeping the invariant in the shared publish-plan helper prevents direct script use from bypassing workflow validation, and workflow guards fail earlier for operator mistakes.
