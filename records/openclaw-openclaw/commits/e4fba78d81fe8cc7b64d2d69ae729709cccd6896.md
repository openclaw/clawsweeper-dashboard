# Commit e4fba78

## Summary

The commit fixes the intended ClawSweeper proof handoff in shape, but it leaves two concrete regressions on `main`: the new proof marker is trusted from any issue comment body, and the required proof workflow usually cannot read comments with the token it prefers.

## Findings

### High: External commenters can forge ClawSweeper proof verdicts

- Kind: security
- File: `scripts/github/real-behavior-proof-policy.mjs`
- Line: 245
- Evidence: `hasClawSweeperExactHeadProof` scans every supplied issue comment body for `<!-- clawsweeper-verdict:pass ... -->` and returns `true` when `item` and `sha` match the PR number/head SHA, but it never checks the comment author, bot/app identity, or `performed_via_github_app`. A read-only probe with a comment authored by `external-contributor` returned `hasProof: true` and `passed: true`. The Barnacle path also kept `proof: sufficient` when the only marker came from a forged contributor comment.
- Impact: An external PR author can post the hidden marker for the current head SHA, then trigger a `pull_request_target` `edited` run by editing the PR body. Once the comment-read path works, the required real-behavior proof check can pass without ClawSweeper or a maintainer accepting proof. On Barnacle runs, a forged exact-head marker can also preserve an existing `proof: sufficient` label after a contributor changes the branch.
- Suggested fix: Accept exact-head verdict markers only from a trusted ClawSweeper app/bot comment, preferably by validating the REST comment author plus GitHub App metadata/app slug or app id. Add regression tests proving contributor-authored markers are rejected in both the proof policy and Barnacle label-sync path.
- Confidence: high

### Medium: Proof check prefers a members-only app token for issue comments

- Kind: reliability
- File: `scripts/github/real-behavior-proof-check.mjs`
- Line: 54
- Evidence: The script sets `const token = appToken || process.env.GITHUB_TOKEN`, while `.github/workflows/real-behavior-proof.yml` creates `GH_APP_TOKEN` with only `permission-members: read`. The workflow adds job-level `issues: read`, but it does not pass `GITHUB_TOKEN` into the run step and the script prefers the app token anyway. `actions/create-github-app-token` source builds a permissions object from the explicit `permission-*` inputs, and its action metadata defines `permission-issues` as the permission for issues and related comments. A mocked run with `GH_APP_TOKEN=app-token` and `GITHUB_TOKEN=github-token` failed at the comments fetch with `Failed to fetch PR comments for proof verdicts: 403`; the same mocked run without `GH_APP_TOKEN` accepted the marker.
- Impact: The intended exact-head ClawSweeper verdict fallback can still fail closed in CI because the comments API call uses a token that was scoped for team membership, not issue comments. External PRs without body proof may also fail with an unhandled fetch error instead of the normal proof-required annotation.
- Suggested fix: Either request `permission-issues: read` on both app-token creation steps, or keep the app token only for maintainer membership and explicitly pass/use `GITHUB_TOKEN` for issue comment reads. Handle comment-fetch permission failures with a clear warning/fallback instead of throwing before the normal proof-required message.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The forged-comment policy probe returns `hasProof: true`; the Barnacle probe preserves `proof: sufficient` with a contributor-authored marker; the mocked workflow probe fails with 403 when an app token is present and succeeds when only the issue-readable token path is used.
- Is this the best way to solve the issue? yes. The exact-head marker contract is reasonable, but it must authenticate the marker source and use a token that can actually read issue comments.
