# Commit 59aef2ff0

## Summary

Found one compatibility regression in the shared binding parser: the new multi-account wildcard default also applies to `agents unbind`, so an existing default-scoped/channel-only binding can no longer be removed with the same bare `--bind <channel>` shorthand that created or removed it before.

## Findings

### Medium: Bare multi-account unbind no longer removes existing default-scoped bindings

- Kind: regression
- File: `src/commands/agents.bindings.ts`
- Line: 269
- Evidence: `resolveBindingAccountId()` now returns `*` whenever the channel plugin reports more than one configured account. That helper is called by `parseBindingSpecs()`, and `agentsUnbindCommand()` uses `parseBindingSpecs()` for removal in `src/commands/agents.commands.bind.ts:363`. In a focused check with two configured WhatsApp accounts and an existing binding `{ match: { channel: "whatsapp" } }`, parsing `--bind whatsapp` produced `{ accountId: "*" }`, and `removeAgentBindings()` returned `removed: []`, `missing: [whatsapp:*]`. Parsing `--bind whatsapp:default` did remove the same stored channel-only binding.
- Impact: Users with pre-existing channel-only/default-account bindings on a channel that later has multiple accounts cannot remove those bindings with the previous bare shorthand. The stale binding remains active and can keep routing default-account traffic to the old agent after the operator attempted to unbind it.
- Suggested fix: Keep wildcard defaulting on creation paths only, or add an unbind-specific parse mode that preserves omitted account IDs as default-scoped when matching existing bindings. At minimum, add a regression test for removing a channel-only binding after multiple accounts exist.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Run a TS snippet importing `parseBindingSpecs` and `removeAgentBindings` with `channels.whatsapp.accounts.default` and `channels.whatsapp.accounts.biz`, plus an existing `{ agentId: "work", match: { channel: "whatsapp" } }` binding; `specs: ["whatsapp"]` misses it, while `specs: ["whatsapp:default"]` removes it.
- Is this the best way to solve the issue? yes, for the suggested direction. The maintainer explicitly accepted wildcard behavior for new omitted-account bindings in PR #49769, so the narrow fix is to avoid applying that creation convenience to removal matching.
