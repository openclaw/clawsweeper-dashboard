# Commit f436b431

## Summary

Found two actionable Feishu CardKit streaming risks in the landed commit. The strongest issue is that CardKit API body errors can still be treated as successful sends, which breaks the new delivered-delta baseline and can drop streamed text.

## Findings

### Medium: CardKit body errors advance `sentText` as if content was delivered

- Kind: bug
- File: `extensions/feishu/src/streaming-card.ts`
- Line: 354
- Evidence: `updateCardContent()` and `replaceCardContent()` only check `response.ok`; on any HTTP 200 response they return `true` without reading the Feishu `{ code, msg }` body. The caller then advances `state.sentText` at `extensions/feishu/src/streaming-card.ts:465` and `extensions/feishu/src/streaming-card.ts:525`. The pinned Feishu SDK dependency is `@larksuiteoapi/node-sdk` 1.64.0 (`extensions/feishu/package.json:11`), and its CardKit element methods return `{ code?: number; msg?: string }` (`node_modules/@larksuiteoapi/node-sdk/types/index.d.ts:32619`); the same dependency comments call out CardKit failures such as `code: 230099 / ErrCode: 11310`.
- Impact: If Feishu returns HTTP 200 with a non-zero CardKit code for sequence, size, rate, or element errors, OpenClaw marks the delta or replacement delivered. Later retries send only text after the false baseline, so the rejected content can be permanently skipped.
- Suggested fix: Parse the JSON body for content and replacement updates, require `code === 0`, log/include `msg` on failure, and return `false` without advancing `sentText`. Add regression coverage for HTTP 200 `{ code: 230099, msg: ... }`.
- Confidence: high

### Medium: Non-prefix streaming rewrites still go through the append/delta path

- Kind: bug
- File: `extensions/feishu/src/streaming-card.ts`
- Line: 456
- Evidence: `resolveStreamingCardAppendContent()` returns the whole `nextText` when it does not start with `sentText`, but `update()` still sends that value through `updateCardContent()` unconditionally. The dispatcher can build non-prefix rewrites by prepending reasoning before already-streamed answer text (`extensions/feishu/src/reply-dispatcher.ts:259`, `extensions/feishu/src/reply-dispatcher.ts:693`), and `mergeStreamingText()` accepts such a rewrite when the new body merely contains the previous text (`extensions/feishu/src/streaming-card.ts:161`). The final close path already has the safer replacement branch at `extensions/feishu/src/streaming-card.ts:516`.
- Impact: On the append-mode CardKit behavior this commit is working around, a later reasoning/status rewrite like `Thinking...\n---\nanswer` after `answer` was delivered can append the whole rewritten body after the old body, then mark it sent. If close sees the same final text, it skips the replacement and the duplicated visible card can persist.
- Suggested fix: In `update()`, use `replaceCardContent()` whenever `mergedText` does not start with `state.sentText`, mirroring `close()`. Add a streaming-card test for `sentText = "answer"` then `update("> Thinking...\n\n---\n\nanswer")`.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the first finding with a mocked CardKit HTTP 200 body carrying a non-zero `code`; the current tests only cover thrown errors and non-OK HTTP. The second finding is source-level with a plausible dispatcher path, but I did not have live Feishu credentials to verify client rendering.
- Is this the best way to solve the issue? Yes for the first finding: Feishu API success must include `code === 0` before advancing the delivered baseline. For the second, reusing the existing replacement branch is the narrowest fix because final close already uses that behavior.
