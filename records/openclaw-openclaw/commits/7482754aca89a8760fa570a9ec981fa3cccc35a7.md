# Commit 7482754

## Summary

Found one compatibility regression in the native plugin loader fallback path.

## Findings

### Medium: Nested SDK alias imports can no longer fall back to the source transformer

- Kind: compatibility
- File: `src/plugins/plugin-module-loader-cache.ts`
- Line: 285
- Evidence: The commit stops passing `fallbackOnMissingDependency: true` into `tryNativeRequireJavaScriptModule`. The remaining SDK-alias guard only scans the top-level target file before native `require()`. A focused temp-file smoke showed a CJS plugin entry that only does `require("./nested.cjs")` now throws `MODULE_NOT_FOUND` when the nested file requires an alias-covered `openclaw/plugin-sdk/*` specifier; the same fixture loads successfully when `tryNative: false`, proving the jiti alias path can resolve it.
- Impact: External/custom CommonJS plugins can use a small entry shim and put their `openclaw/plugin-sdk/*` import in a nested runtime file. Those plugins now fail during native load before jiti alias rewriting gets a chance to run.
- Suggested fix: Keep the broad missing-dependency retry removed, but preserve SDK-alias compatibility by either recursively scanning static relative JS dependencies before native load, or narrowly falling back only when the missing specifier matches an SDK alias key. Add a regression test for an entry shim with the SDK import in a nested file.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes: create a temp CJS entry requiring a nested CJS file, make the nested file require an alias-backed `openclaw/plugin-sdk/*` specifier, then call `getCachedPluginModuleLoader` with `tryNative: true`. Current main throws; `tryNative: false` loads.
- Is this the best way to solve the issue? no: the duplicate-side-effect fix is directionally right for arbitrary missing dependencies, but it is too broad for alias-backed SDK misses that the loader contract already supports.
