# Commit 98cc6df

## Summary

Found one concrete gap in the new Anthropic image-capability repair. It fixes unprefixed stale Claude rows, but still misses provider-prefixed configured rows that the model resolver supports and can return.

## Findings

### Medium: Provider-prefixed Anthropic configured rows still resolve text-only

- Kind: bug
- File: `extensions/anthropic/register.runtime.ts`
- Line: 396
- Evidence: `applyAnthropicImageInputCapability` checks only `ctx.modelId` and `model.name` for a `claude-*` prefix. For configured provider rows, core preserves the configured `model.id` and passes it into the provider hook as `modelId` (`src/agents/pi-embedded-runner/model.ts:265`), while inline rows can legally keep ids like `anthropic/claude-sonnet-4-5` (`src/agents/pi-embedded-runner/model.inline-provider.ts:163`). A focused production-hook check with a stale configured row `{ id: "anthropic/claude-sonnet-4-5", input: ["text"] }` resolved `modelInput: ["text"]` and `preservedImages: 0`.
- Impact: Users with provider-prefixed `models.providers.anthropic.models[]` rows can still hit the same image-drop path fixed by this commit: native `/v1/responses` images are stripped before the Anthropic request even though the Claude model supports vision.
- Suggested fix: Include `params.model.id` in the image-capability candidates and normalize/strip a matching `anthropic/` or `claude-cli/` provider prefix before matching known Claude model prefixes. Add a regression where the configured row id is `anthropic/claude-sonnet-4-5`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Use the production Anthropic provider normalizer through `resolveModelWithRegistry` with `models.providers.anthropic.models[0].id = "anthropic/claude-sonnet-4-5"` and `input: ["text"]`; `detectAndLoadPromptImages` preserves 0 existing images.
- Is this the best way to solve the issue? yes. The provider hook owns this repair, and normalizing the candidate id keeps the fix local without moving Anthropic policy into core.
