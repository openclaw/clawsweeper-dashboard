# Commit a9176e9

## Summary

Found one release-check validation gap: embedded fallback detection runs only after the OK-marker check, so a fallback turn that times out before producing `OK` can still be converted into an optional skipped success.

## Findings

### Medium: Fallback timeout can still be skipped as optional instead of rejected

- Kind: bug
- File: `scripts/openclaw-cross-os-release-checks.ts`
- Line: 2229 and 3045
- Evidence: both agent-turn paths first throw `"Agent output did not contain the expected OK marker."` when `agentOutputHasExpectedOkMarker(...)` is false, and only then check `agentTurnUsedEmbeddedFallback(...)`. The catch path calls `maybeBuildOptionalAgentTurnSkipResult`, whose default-enabled optional logic returns a success-like skipped result when the error is an OK-marker miss and the log contains `"status":"timeout"` or `Request timed out before a response was generated` (`scripts/openclaw-cross-os-release-checks.ts:43`, `scripts/openclaw-cross-os-release-checks.ts:3072`, `scripts/openclaw-cross-os-release-checks.ts:3091`).
- Impact: a gateway failure followed by embedded fallback that times out without `OK` can bypass the new fallback rejection and pass the release lane as an optional skipped agent turn, leaving the cross-OS release proof without gateway-agent coverage.
- Suggested fix: check `agentTurnUsedEmbeddedFallback(result, { logText })` before the OK-marker validation in both `runInstalledAgentTurn` and `runAgentTurn`, or make the optional skip helper refuse to skip when stdout/stderr/current log text contains the embedded fallback marker. Add a regression test for fallback marker plus timeout payload plus no `OK`.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes at the helper/branch level: a synthetic current-turn log containing `EMBEDDED FALLBACK:` plus a timeout payload returns `fallback=true`, `ok=false`, and `optionalSkip=true`, matching the branch ordering that would skip before fallback rejection. I did not run a live provider/gateway scenario.
- Is this the best way to solve the issue? Yes: fallback is the stronger release-proof disqualifier and should be detected before optional model-timeout handling.
