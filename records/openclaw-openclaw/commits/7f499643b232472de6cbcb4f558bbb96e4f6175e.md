# Commit 7f49964

## Summary

Found one concrete multi-account Slack approval-routing regression. Existing focused Slack tests pass, but they do not cover unscoped `approvals.plugin.targets` with multiple Slack accounts.

## Findings

### Medium: Unscoped Slack plugin approval targets can be handled by every Slack account

- Kind: security
- File: `extensions/slack/src/approval-native-gates.ts`
- Line: 72
- Evidence: `hasSlackPluginForwardingTarget` treats a Slack forwarding target with no `accountId` as matching any runtime account because `matchesSlackAccount` returns true when either side is missing (`extensions/slack/src/approval-native-gates.ts:55`). That feeds `canPluginForwardingRouteToSlack` and `shouldHandleSlackPluginNativeApprovalRequest`, so each Slack account runtime with plugin approvers can handle the same plugin approval (`extensions/slack/src/approval-native-gates.ts:128`, `extensions/slack/src/approval-native-gates.ts:202`). The native adapter then sends DMs to that account’s plugin approvers (`extensions/slack/src/approval-native.ts:190`). Gateway bootstraps native approval handlers per account (`src/gateway/server-channels.ts:505`). This conflicts with documented multi-account semantics: omitted `accountId` means the default account, while `accountId: "*"` is the all-account fallback (`docs/gateway/config-channels.md`, `docs/concepts/multi-agent.md`).
- Impact: In a multi-workspace Slack setup, `approvals.plugin.targets: [{ channel: "slack", to: "U123" }]` can be treated as deliverable by non-default Slack account runtimes too. Those other workspaces’ plugin approvers can receive and resolve a plugin approval that was configured for the default/unscoped target, widening approval authority and leaking plugin approval details across Slack accounts.
- Suggested fix: Make Slack plugin-forwarding target matching account-aware: unscoped targets should resolve to the channel default account, named accounts should require an exact `target.accountId`, and all-account behavior should require an explicit wildcard contract if supported. Add a multi-account Slack test for plugin native delivery/suppression with an unscoped target and a named account.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes at source level: configure two Slack accounts, give both account-scoped `allowFrom` plugin approvers, enable `approvals.plugin.mode: "targets"` with a Slack target that omits `accountId`, then observe that the new gate returns true for both account ids.
- Is this the best way to solve the issue? Yes: the fix should happen in the new Slack native plugin approval gate so native delivery, fallback suppression, and runtime registration all share the same account-scope semantics.
