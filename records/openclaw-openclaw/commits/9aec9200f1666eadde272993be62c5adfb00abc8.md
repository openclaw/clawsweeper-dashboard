# Commit 9aec920

## Summary

The agent workspace resolver fix works for the core agent path, but the memory host SDK still has a duplicate default workspace resolver that ignores `OPENCLAW_WORKSPACE_DIR`. That leaves QMD memory startup/default collections and `memory_get` reads pointed at the old home-derived workspace in the same env-mounted deployment this commit is meant to fix.

## Findings

### Medium: QMD memory paths still ignore `OPENCLAW_WORKSPACE_DIR`

- Kind: bug
- File: `packages/memory-host-sdk/src/host/config-utils.ts`
- Line: 240
- Evidence: The commit added `OPENCLAW_WORKSPACE_DIR` handling to `src/agents/workspace-default.ts:10`, but the memory host SDK copy at `packages/memory-host-sdk/src/host/config-utils.ts:240` still derives defaults only from home/profile. That copy is runtime-reachable: `src/memory-host-sdk/host/backend-config.ts:1` re-exports the package backend config, `src/gateway/server-startup-memory.ts:49` calls it for QMD startup, `packages/memory-host-sdk/src/host/backend-config.ts:374` uses its resolver for default memory collections, and `packages/memory-host-sdk/src/host/read-file.ts:148` uses it for agent memory reads. A focused probe on current `main` with `OPENCLAW_WORKSPACE_DIR=/srv/openclaw-workspace` and `OPENCLAW_HOME=/srv/openclaw-home` returned `{"core":"/srv/openclaw-workspace","memory":"/srv/openclaw-home/.openclaw/workspace"}`.
- Impact: Docker/ECS users can now see the correct workspace in agent prompt/runtime paths while QMD memory indexing and memory file reads still miss `MEMORY.md` and `memory/*.md` on the mounted workspace. This preserves a concrete part of #66786’s reported failure mode: memory operations defaulting to the wrong path.
- Suggested fix: Align `packages/memory-host-sdk/src/host/config-utils.ts` with the core resolver precedence, or route package runtime file helpers through the existing core-backed runtime facade where appropriate. Add a regression test in `packages/memory-host-sdk/src/host/backend-config.test.ts` or `read-file.test.ts` that asserts default memory collections/read paths honor `OPENCLAW_WORKSPACE_DIR`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Set `OPENCLAW_WORKSPACE_DIR` and `OPENCLAW_HOME`, leave `agents.defaults.workspace` and per-agent workspace unset, then resolve core agent workspace and QMD memory backend config; the focused probe above shows they diverge.
- Is this the best way to solve the issue? yes. The problem is a duplicated default workspace resolver in the memory host SDK path, so the fix should make that resolver obey the same env precedence or remove the duplication for runtime-backed use.
