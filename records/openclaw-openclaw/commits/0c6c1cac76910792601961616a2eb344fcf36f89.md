# Commit 0c6c1cac

## Summary

Found one actionable bug: the new ClawHub-preferred cutover path is not reachable from the real update flow. The unit tests inject ClawHub bridges directly, but the production bridge builder still only emits npm bridges.

## Findings

### Medium: ClawHub-preferred bundled cutovers are never produced by the CLI update path

- Kind: bug
- File: `src/cli/plugins-location-bridges.ts`
- Line: 17
- Evidence: `updatePluginsAfterCoreUpdate` passes `externalizedBundledPluginBridges` only from `listPersistedBundledPluginLocationBridges` (`src/cli/update-cli/update-command.ts:775`). That builder reads only `record.packageInstall?.npm?.spec`, returns `null` without it, and hardcodes `preferredSource: "npm"` (`src/cli/plugins-location-bridges.ts:17-25`). The new ClawHub branch in `syncPluginsForUpdateChannel` only runs when a bridge has `preferredSource === "clawhub"` plus `clawhubSpec` (`src/plugins/update.ts:1136-1158`), but no current production caller supplies such a bridge.
- Impact: official bundled-plugin cutovers still install from npm instead of preferring ClawHub/ClawPack, so the changelogged behavior does not happen for users updating through `openclaw update`.
- Suggested fix: extend the persisted bridge source/manifest install metadata to carry ClawHub package/spec data, or add an explicit official cutover registry that `listPersistedBundledPluginLocationBridges` merges into its bridge output. Also make the post-core update caller consume `switchedToClawHub` when skipping already-synced plugins.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. Inspect the production call path: update CLI obtains bridges only from `listPersistedBundledPluginLocationBridges`, and that function cannot return a ClawHub-preferred bridge.
- Is this the best way to solve the issue? Unclear. The safest fix depends on whether ClawHub cutover data should live in plugin manifests, persisted indexes, or a maintainer-owned cutover table.
