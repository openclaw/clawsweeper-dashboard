# Commit 7c2425a

## Summary

Found one concrete compatibility gap in the HTTPS proxy support: the new IP-SNI hardening is not applied to `proxy-fetch.ts`, so some custom fetch paths can still fail with HTTPS proxy endpoints addressed by IP.

## Findings

### Medium: Proxy fetch helpers still use raw Undici proxy agents without IP-SNI stripping

- Kind: compatibility
- File: `src/infra/net/proxy-fetch.ts`
- Line: 81
- Evidence: The commit adds `createHttp1ProxyAgent` / `createHttp1EnvHttpProxyAgent` wrappers in `src/infra/net/undici-runtime.ts` that install an IP-safe `clientFactory` and strip IP-literal `servername` before Undici connects to an HTTPS proxy. However, `makeProxyFetch()` still constructs `new ProxyAgent(addActiveManagedProxyTlsOptions({ uri: proxyUrl }))` directly, and `resolveProxyFetchFromEnv()` still constructs `new EnvHttpProxyAgent(proxyOptions)` directly at `src/infra/net/proxy-fetch.ts:129`. In Undici 8.3.0, `ProxyAgent` sets CONNECT `servername` to `proxyHostname`, and `EnvHttpProxyAgent` builds internal `ProxyAgent` instances from its options, so these two paths miss the wrapper behavior.
- Impact: Managed HTTPS proxy endpoints like `https://127.0.0.1:8443` or `https://[::1]:8443` can pass `openclaw proxy validate` and the global dispatcher path, then fail in callers that use `resolveProxyFetchFromEnv()` or `makeProxyFetch()` because they still send IP-literal SNI to the proxy TLS endpoint. Affected call paths include `src/media-understanding/runner.entries.ts:623`, `src/agents/pi-embedded-runner/openrouter-model-capabilities.ts:200`, and several Telegram/Zalo/Discord explicit proxy fetch paths.
- Suggested fix: Change `proxy-fetch.ts` to create dispatchers through `createHttp1ProxyAgent()` and `createHttp1EnvHttpProxyAgent()` while still loading Undici `fetch`/`FormData` as it does today. Add a focused test that `makeProxyFetch("https://127.0.0.1:8443")` and `resolveProxyFetchFromEnv({ HTTPS_PROXY: "https://127.0.0.1:8443" })` install the same IP-safe client factory behavior covered by `src/infra/net/undici-runtime.test.ts`.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes at the code-contract level: configure an HTTPS proxy endpoint by IP with `proxy.tls.caFile`, then exercise a `resolveProxyFetchFromEnv()` or `makeProxyFetch()` caller. I did not run a live generated-cert proxy repro for this report.
- Is this the best way to solve the issue? Yes. The repo already introduced the correct shared wrappers in this commit; using them in `proxy-fetch.ts` avoids duplicating Undici internals and keeps proxy fetch behavior aligned with validation/global dispatcher paths.
