# Commit 7b5a18a

## Summary

Found two concrete regressions in the new Google Meet CLI gateway delegation. The main risk is that several delegated stateful commands still use a 5s RPC timeout even though their gateway-side work can validly take 10-30s or more.

## Findings

### Medium: Delegated session commands can time out before valid gateway work completes

- Kind: regression
- File: `extensions/google-meet/src/cli.ts`
- Line: 2277
- Evidence: `callGoogleMeetGateway` defaults to `GOOGLE_MEET_GATEWAY_DEFAULT_TIMEOUT_MS = 5000` when no timeout is supplied, and `status`, `doctor`, `leave`, and `speak` all call it without `timeoutMs`. Gateway-side `leave`/`speak` can call Voice Call RPCs with `config.voiceCall.requestTimeoutMs` defaulting to 30s, and `status`/`doctor` can refresh browser health through 5s tab lookup plus 5s focus plus 10s evaluation paths.
- Impact: `openclaw googlemeet leave <session>` or `speak <session>` can report `gateway request timeout` against a healthy gateway while the runtime is still doing a valid hangup/speech operation. `status`/`doctor` can also fail for slow Chrome-node/browser inspection. This weakens the commitâ€™s stated goal because follow-up CLI control of gateway-owned sessions becomes flaky under normal configured timeouts.
- Suggested fix: Pass a method-appropriate timeout for all delegated lifecycle/status calls. Use at least `operationTimeoutMs` for `status`, `doctor`, `leave`, and `speak`; for `test-listen`, include the parsed `--timeout-ms` plus a small buffer because the runtime accepts up to 120s.
- Confidence: high

### Medium: Delegated create ignores CLI `--expires-at`

- Kind: regression
- File: `extensions/google-meet/src/cli.ts`
- Line: 1479
- Evidence: delegated `create` sends raw Commander `options` to `googlemeet.create`, so `expiresAt` is a string. The gateway create path in `extensions/google-meet/src/create.ts` only accepts `raw.expiresAt` when it is already a number; otherwise it falls back to configured OAuth expiry. The pre-commit local CLI path parsed `--expires-at` with `parseOptionalNumber`.
- Impact: `openclaw googlemeet create --access-token <token> --expires-at <future>` works through the local path, but with a running gateway and default join behavior it is delegated and can reject with missing refresh credentials because the valid access-token expiry was ignored.
- Suggested fix: Normalize `expiresAt` before sending the gateway payload, or update the gateway create parameter normalization to accept numeric strings consistently with the CLI local path.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. Both issues follow directly from the current source paths. A minimal unit repro can inject a gateway call that delays over 5s for `leave`/`speak`, or route `create` with `--access-token` plus string `--expires-at` through the gateway handler.
- Is this the best way to solve the issue? Yes. Keep the session ownership change, but make the delegated CLI payload and timeout contracts match the runtime methods they now call.
