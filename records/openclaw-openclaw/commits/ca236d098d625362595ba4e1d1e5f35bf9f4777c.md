# Commit ca236d0

## Summary

Found one actionable reliability gap in the launchd hardening path. New installs and explicit restarts rewrite old LaunchAgent plists to the hardened stdout/stdin/stderr paths, but the macOS installed-but-not-loaded recovery path still bootstraps the stale plist unchanged.

## Findings

### Medium: `gateway start` can re-bootstrap stale macOS LaunchAgent plists without the hardened paths

- Kind: reliability
- File: `src/daemon/launchd.ts`
- Line: 586
- Evidence: This commit adds `rewriteLaunchAgentPlistForRestart()` and wires it into `restartLaunchAgent()` before kickstart/bootstrap, so old plists get rewritten to `~/Library/Logs/openclaw`, `/dev/null` stderr, and `/dev/null` stdin before restart. But `repairLaunchAgentBootstrap()` still only runs `launchctl enable` and `launchctl bootstrap` against the existing plist at `src/daemon/launchd.ts:585-586`. `openclaw gateway start` takes this repair path first when launchd is not loaded but the plist exists: `recoverInstalledLaunchAgent()` calls `repairLaunchAgentBootstrap()` and then returns handled at `src/cli/daemon-cli/launchd-recovery.ts:26-37`, so `startGatewayService()` never reaches the new restart rewrite path.
- Impact: Existing macOS users with a pre-hardening plist and an unloaded LaunchAgent can run `openclaw gateway start` or the doctor bootstrap repair and still boot the old plist. That leaves the old state-dir stdout path and missing stdin hardening in place, which is exactly the class of launchd startup/log-path issue this commit is trying to repair for installed services.
- Suggested fix: Reuse the same plist rewrite logic before `repairLaunchAgentBootstrap()` calls `bootstrap`, or route the installed-but-not-loaded recovery through a start/restart helper that rewrites the plist before bootstrapping. Add a focused test with an old plist, `serviceLoaded=false`, and `recoverInstalledLaunchAgent()`/`repairLaunchAgentBootstrap()` asserting the plist is rewritten before bootstrap.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: create an old-style LaunchAgent plist with `StandardOutPath` under the state dir and no `StandardInPath`, boot it out so the plist exists but launchd is not loaded, then run `openclaw gateway start`. The current code path bootstraps via `repairLaunchAgentBootstrap()` and returns before the new restart rewrite can run.
- Is this the best way to solve the issue? yes: the commit already centralizes the intended rewrite in `rewriteLaunchAgentPlistForRestart()`, so the repair/start recovery path should use that same preparation before bootstrap.
