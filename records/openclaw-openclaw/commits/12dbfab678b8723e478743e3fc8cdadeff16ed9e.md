# Commit 12dbfab

## Summary

Found one low-severity Teams progress-stream bug: command-output progress lines are wired to a field the core event payload does not send, so command completions collapse to a generic message.

## Findings

### Low: Command-output progress loses command name and exit status

- Kind: bug
- File: `extensions/msteams/src/reply-dispatcher.ts`
- Line: 388
- Evidence: The new Teams `onCommandOutput` handler reads `payload.summary`, but `ReplyOptions.onCommandOutput` only defines `title`, `name`, `output`, `status`, `exitCode`, `durationMs`, and `cwd` (`src/auto-reply/get-reply-options.types.ts:127`). The agent runner sends those fields and no `summary` for `command_output` events (`src/auto-reply/reply/agent-runner-execution.ts:1583`).
- Impact: In Microsoft Teams native progress mode, completed commands will stream `command output ready` instead of useful labels like the command/tool name and exit status. Other progress-stream channels format this from `name`/`title`/`exitCode`, so Teams now has a concrete behavior gap.
- Suggested fix: Format Teams command-output progress from the actual payload fields, e.g. mirror Slack/Telegram/Discord: use `payload.name` with success/exit-code suffix, otherwise `payload.title`, and fall back only when both are absent.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. Configure a personal Microsoft Teams account with `streaming.mode="progress"` and tool progress enabled, then emit a `command_output` event with `phase: "end"`, `name`, and `exitCode`; the Teams callback cannot see `summary`, so it pushes the fallback text.
- Is this the best way to solve the issue? Yes. The stable callback contract already exposes the command fields; the Teams handler should consume that contract instead of inventing `summary`.
