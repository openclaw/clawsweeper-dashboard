# Commit b19b7539a8

## Summary

Found one actionable security/privacy issue in the API-key Docker live-test path: the cleanup added by the commit recursively makes the container home world-readable after `codex login --with-api-key` has used the live API key.

## Findings

### Medium: API-key Docker cleanup can expose Codex credentials in host temp storage

- Kind: security
- File: `scripts/test-live-codex-harness-docker.sh`
- Line: 188
- Evidence: In API-key mode the script creates and bind-mounts a host temp home at `/home/node` (`scripts/test-live-codex-harness-docker.sh:109`), passes `OPENAI_API_KEY` into the container (`scripts/test-live-codex-harness-docker.sh:165`), then runs `codex login --with-api-key` inside that home (`scripts/test-live-codex-harness-docker.sh:230`). The new EXIT trap runs `chmod -R a+rwX "$HOME" "$NPM_CONFIG_PREFIX" "$XDG_CACHE_HOME"` (`scripts/test-live-codex-harness-docker.sh:188`), so any credential/config material written under the bind-mounted home becomes readable and writable by all local users before the host-side cleanup removes the temp directory.
- Impact: On shared or self-hosted runners, or local hosts where `${RUNNER_TEMP:-/tmp}` is traversable, another local user can race the cleanup window and read live Codex/OpenAI credential material. This is a test-only path, but it handles real live API keys.
- Suggested fix: Keep API-key temp homes private by running the container as the host uid/gid in API-key mode and avoiding recursive `a+rwX`; if non-host uids must be supported, do cleanup inside the container or use a private Docker volume/tmpfs rather than broadening permissions on a host bind mount.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes: run `OPENCLAW_LIVE_CODEX_HARNESS_AUTH=api-key` with a live API key on a host temp path that another local user can traverse, then observe that the bind-mounted `/home/node` tree is recursively changed to world-readable/world-writable during container exit. I did not run the live lane because it requires live OpenAI credentials and a Docker image build.
- Is this the best way to solve the issue? Unclear on the exact implementation, because the intended non-CI uid contract is not documented here. The safe direction is clear: do not recursively apply `a+rwX` to a bind-mounted home that can contain live auth state.
