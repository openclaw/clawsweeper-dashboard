# Commit c589680

## Summary

Found one low-severity doctor diagnostic regression in the new alternate memory-slot suppression.

## Findings

### Low: `enabled: true` or empty config can hide an inactive `memory-lancedb` setup

- Kind: regression
- File: `src/commands/doctor-memory-search.ts`
- Line: 330
- Evidence: `hasActiveAlternateMemoryPluginSlot` now returns true when the selected alternate slot entry has `enabled === true` or any `config` value. But `memory-lancedb` disables itself during registration if `api.pluginConfig` has no `embedding` object: `extensions/memory-lancedb/index.ts:643` catches `memoryConfigSchema.parse`, and `extensions/memory-lancedb/config.ts:120` throws `embedding config required`. The manifest schema permits this bad state because `extensions/memory-lancedb/openclaw.plugin.json:84` defines `embedding` as optional.
- Impact: `openclaw doctor` can now silently pass a config such as `plugins.slots.memory = "memory-lancedb"` with `plugins.entries.memory-lancedb.enabled = true` but no embedding config, even though the plugin registers only a disabled warning service and memory tools/hooks are not active. Before this commit, the null backend path still emitted `No active memory plugin is registered`.
- Suggested fix: do not treat `enabled: true` alone, or an empty/invalid `config`, as proof that an alternate memory plugin is active. Either require a non-placeholder valid config before suppressing this note, align the `memory-lancedb` manifest schema with its runtime parser by requiring `embedding`, or consult plugin diagnostics/runtime registration before returning.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. I ran a temp-home `openclaw doctor --non-interactive --no-workspace-suggestions` with `memory-lancedb` selected and `entries.memory-lancedb.enabled = true` but no config. The filtered output matched only `Doctor complete`; no missing-memory-plugin, `memory-lancedb`, or `embedding config required` warning appeared.
- Is this the best way to solve the issue? unclear. The safest direction is to stop using raw slot-entry presence as active-plugin proof unless the entry has valid plugin config or the plugin runtime/diagnostics confirm it loaded.
