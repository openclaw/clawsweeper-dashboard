# Commit 14142a9

## Summary

Found one concrete bug: the new inline-image guard still accepts arbitrary bytes when the `data:` URL declares `image/*`, unless the bytes happen to sniff as a known non-image format.

## Findings

### Medium: Inline image sniffing falls back to the attacker-declared MIME

- Kind: bug
- File: `extensions/msteams/src/attachments/download.ts`
- Line: 115
- Evidence: `resolveInlineDataImageMime` calls `detectMime({ buffer: inline.data, headerMime: inline.contentType })`, then accepts the result when it starts with `image/`. The shared detector falls back to any non-generic header MIME when byte sniffing returns nothing (`src/media/mime.ts:201` and `src/media/mime.ts:212`). A focused probe on current `main` returned `image/png` for both `Buffer.from("not an image")` and `Buffer.from("<html>...")` when `headerMime` was `image/png`; only recognizable signatures like ZIP/PDF overrode the header.
- Impact: `data:image/png;base64,...` payloads containing unknown text/HTML/random bytes are still saved and surfaced as image media. This leaves the spoofing path mostly open for payloads without a file-type signature, despite the commit’s stated goal of skipping payloads that are not actually images.
- Suggested fix: use a strict byte-only sniff for inline `data:image/*` candidates before saving. Do not pass the declared `data:` MIME into the acceptance check unless the format has a separate validator, such as a deliberate SVG parser/sanitizer. Add a test with real detector behavior or a mock returning `undefined` for unknown bytes plus `inline.contentType: "image/png"`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Run a focused MIME probe: `node --import tsx --input-type=module --eval 'import { detectMime } from "./src/media/mime.ts"; console.log(await detectMime({ buffer: Buffer.from("not an image"), headerMime: "image/png" }));'`. It prints `image/png`, so the new `startsWith("image/")` check accepts the spoof.
- Is this the best way to solve the issue? yes. The risk is caused by using a header-trusting detector for a trust decision; the safer boundary is a byte-only or source-aware detector for this path.
