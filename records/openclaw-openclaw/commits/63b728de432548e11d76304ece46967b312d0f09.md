# Commit 63b728d

## Summary

Found one reliability bug in the new low-level Telegram 421 response retry path. It retries by reusing the same `RequestInit`, which is fine for JSON/string bodies but breaks Telegram media uploads because grammY builds multipart uploads as one-shot streams.

## Findings

### Medium: 421 response retry reuses consumed multipart upload bodies

- Kind: bug
- File: `extensions/telegram/src/client-fetch.ts`
- Line: 204
- Evidence: `createTelegramClientFetch` calls `runFetch()` again after a `response.status === 421`, and `runFetch()` resubmits the same `input` and spread `init` object. grammY 1.42.0 builds the fetch config once per API call; for uploads, `createFormDataPayload` sets `body` to a `Readable` multipart stream. Current `main` still has this exact logic, with no later changes to the touched files.
- Impact: `sendPhoto`, `sendDocument`, `sendVideo`, and other upload methods can fail instead of retrying when Telegram returns 421. The first request consumes the multipart stream; the immediate second fetch throws before reaching the server, losing the original 421 status and bypassing the higher-level retry path that would have rebuilt the grammY request body.
- Suggested fix: Do not low-level retry 421 responses when `init.body` is a non-replayable stream. Either throw a status-tagged 421 error so the existing higher-level Telegram send retry can call `api.send*` again and rebuild the body, or only perform the inline retry for replayable bodies such as strings/URLSearchParams/Blob/Buffer. Add a regression test with a stream body or consuming fetch mock.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A local Node fetch probe that retries a 421 response with the same `Readable` request body fails with `TypeError Response body object should not be disturbed or locked` and only one server request is observed.
- Is this the best way to solve the issue? yes. The retry needs to happen at a layer that can recreate the request body, or the inline retry must be limited to replayable bodies. The current custom fetch hook receives an already-created stream and cannot safely replay it.
