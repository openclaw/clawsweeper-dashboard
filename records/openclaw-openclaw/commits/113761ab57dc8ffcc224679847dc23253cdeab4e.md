# Commit 113761a

## Summary

The `@openclaw/fs-safe` bump changes stale sidecar lock behavior from reclaiming stale locks to throwing `file_lock_stale`. Current OpenClaw lock wrappers still expect stale lock reclamation, and the existing session-lock regression tests now fail on current `main`.

## Findings

### Medium: `fs-safe` 0.2.0 breaks OpenClaw stale lock recovery

- Kind: regression
- File: `package.json`
- Line: 1700
- Evidence: The dependency pin moves `@openclaw/fs-safe` to `c7ccb99d...`. That upstream range includes `fix: fail closed on stale sidecar locks`, and current OpenClaw still calls `SESSION_LOCKS.acquire()` with a `shouldReclaim` callback that returns true for dead PID, recycled PID, malformed, payload-less, and orphan self-lock cases at `src/agents/session-write-lock.ts:506`. The focused test run `pnpm test src/plugin-sdk/file-lock.test.ts src/agents/session-write-lock.test.ts` now fails 6 session-lock tests, all with `Error: file lock stale for ...`, including `reclaims stale lock files`, `reclaims payload-less orphan lock files`, and `reclaims malformed lock files once they are old enough` from `src/agents/session-write-lock.test.ts:242` and `src/agents/session-write-lock.test.ts:270`. The upstream compare shows the 16-commit dependency range and the stale-lock fail-closed commit. ([github.com](https://github.com/openclaw/fs-safe/compare/ce4137f028ca4b09f26a93ffa3e0d32fbfbd516c...c7ccb99d3058f2acf2ad2758ad2470c7e113a53c))
- Impact: A stale session `.jsonl.lock` left by a crashed or killed process can now block transcript/session writes instead of being reclaimed by `acquireSessionWriteLock()`. The same behavior reaches `src/plugin-sdk/file-lock.ts`: a direct smoke with a dead-PID lock returned `error file_lock_stale file lock stale for .../state.json`, so plugin file-lock users can also become stuck until manual cleanup.
- Suggested fix: Either adapt OpenClaw wrappers to the new fail-closed fs-safe contract by explicitly performing their existing stale-lock cleanup before retrying acquisition, or update `fs-safe` to provide an opt-in, identity-checked reclaim mode that preserves OpenClaw’s existing `shouldReclaim` contract. Add/keep targeted coverage for stale plugin SDK locks as well as the existing session lock tests.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. `pnpm test src/plugin-sdk/file-lock.test.ts src/agents/session-write-lock.test.ts` fails 6 existing session stale-lock recovery tests on current `main`.
- Is this the best way to solve the issue? unclear. The upstream fail-closed change may be the safer primitive default, but OpenClaw’s higher-level lock owners need an explicit recovery path or a documented breaking-contract change.
