# Commit a3c7fea

## Summary

Found one issue: the new helper disables Pi resource *loading*, but pi-coding-agent still performs synchronous default filesystem discovery before those `no*` flags take effect. That leaves the Windows event-loop-stall failure mode only partially addressed.

## Findings

### Medium: `no*` loader flags still allow synchronous Pi resource discovery

- Kind: reliability
- File: `src/agents/pi-embedded-runner/resource-loader.ts`
- Line: 19
- Evidence: `createEmbeddedPiResourceLoader()` still returns `new DefaultResourceLoader(...)`, and both embedded attempt and compaction immediately call `resourceLoader.reload()` at `src/agents/pi-embedded-runner/run/attempt.ts:1721` and `src/agents/pi-embedded-runner/compact.ts:1005`. In `@earendil-works/pi-coding-agent@0.74.0`, `DefaultResourceLoader.reload()` calls `DefaultPackageManager.resolve()` before applying `noExtensions`, `noSkills`, `noPromptTemplates`, or `noThemes`; `resolve()` unconditionally auto-discovers `.pi/{extensions,skills,prompts,themes}`, `.agents/skills`, and agent-dir resource folders with sync `existsSync`/`readdirSync`/`statSync`.
- Impact: the commit is meant to avoid Pi resource discovery stalls during first embedded startup, but workspaces with `.pi` or `.agents` skill/resource trees can still block the Node event loop inside `reload()`. On the reported Windows path, that can still starve channel keepalive timers and trigger the same first-inbound stall/428 recovery loop.
- Suggested fix: avoid `DefaultPackageManager.resolve()` for embedded runs. Use a minimal custom `ResourceLoader` that loads only the OpenClaw inline `extensionFactories`, or add/upstream a loader option that skips package-manager auto-discovery before `resolve()` runs.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the contract mismatch: a temp-workspace probe with `.pi/skills` and `.agents/skills`, all five `no*` options enabled, and `DefaultResourceLoader.reload()` still recorded sync `existsSync`, `readdirSync`, and `statSync` calls under those folders while returning zero loaded skills/prompts/themes/context files. No for the full Windows WhatsApp live stall; I did not have a Windows linked-WhatsApp environment here.
- Is this the best way to solve the issue? No. The change suppresses loaded resources after discovery, but the stall source is the synchronous discovery path itself.
