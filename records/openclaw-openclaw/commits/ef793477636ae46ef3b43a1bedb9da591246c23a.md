# Commit ef79347763

## Summary

Found one actionable UI behavior regression in the Talk retry change: the error-state Talk control still presents itself as “Stop Talk” while the new handler restarts Talk.

## Findings

### Medium: Error-state Talk button says “Stop Talk” but starts a new Talk session

- Kind: bug
- File: `ui/src/ui/app.ts`
- Line: 911
- Evidence: `toggleRealtimeTalk()` now treats `realtimeTalkStatus === "error"` as a retry path: it stops the stale session, then falls through to create and start a new `RealtimeTalkSession`. Runtime Talk errors keep `realtimeTalkActive = true` via the status callback at `ui/src/ui/app.ts:936`. The rendered button label and title are still derived only from `realtimeTalkActive`, so that same state is announced and shown as “Stop Talk” in `ui/src/ui/views/chat.ts:1308`.
- Impact: after a runtime Talk error, a user or screen-reader user activating the visible “Stop Talk” control will unexpectedly start a fresh microphone/realtime session instead of stopping or clearing the failed one. That is especially sensitive for a voice feature because the action can reopen microphone capture under a misleading command label.
- Suggested fix: make the rendered Talk control account for `realtimeTalkStatus === "error"` and label/title it as “Retry Talk” or “Start Talk”, with matching icon/active styling. Alternatively, if “Stop Talk” remains the label, preserve stop-only behavior and add a separate retry affordance.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Put the app in `realtimeTalkActive = true`, `realtimeTalkStatus = "error"`, and `realtimeTalkSession != null`; the existing renderer labels the button as “Stop Talk”, while the new test path proves clicking it calls `start()` for a new session.
- Is this the best way to solve the issue? yes. The intended behavior appears to be retry-on-click, so the lowest-risk fix is to make the UI affordance match that command rather than changing the retry state machine again.
