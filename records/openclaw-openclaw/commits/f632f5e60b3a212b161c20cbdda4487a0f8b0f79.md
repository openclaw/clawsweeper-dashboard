# Commit f632f5e

## Summary

Found one actionable QA regression: the documented `--gateway-setup --scenario ...` Slack desktop smoke path can report pass without running the Slack QA scenario or validating the gateway it starts.

## Findings

### Medium: `--gateway-setup` skips the requested Slack QA scenario and can false-pass

- Kind: bug
- File: `extensions/qa-lab/src/mantis/slack-desktop-smoke.runtime.ts`
- Line: 334
- Evidence: In the generated remote script, the `setup_gateway=1` branch only patches config, starts `pnpm openclaw gateway run`, writes the PID, and sleeps. The actual `pnpm openclaw qa slack ... ${scenarioArgs}` command is in the `else` branch at lines 369-374, so the documented command with `--gateway-setup --scenario slack-canary` never runs `slack-canary`. `qa_status` remains `0` unless setup commands fail, the screenshot check can pass, and the optional `slack-qa/` artifact copy is explicitly ignored on failure at lines 557-569.
- Impact: Maintainers can get a green Mantis Slack desktop smoke report for a VM that merely opened Slack Web and launched a background gateway. This defeats the purpose of the new QA primitive and can hide Slack channel regressions.
- Suggested fix: Make gateway setup either run a real Slack scenario against the persistent gateway and verify gateway health before returning pass, or split it into an explicit setup-only mode that does not accept `--scenario` and does not report QA success/artifacts.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Run `pnpm openclaw qa mantis slack-desktop-smoke --gateway-setup --scenario slack-canary --keep-lease` with normal Mantis inputs; source control flow shows the `openclaw qa slack` command is bypassed whenever `setup_gateway=1`.
- Is this the best way to solve the issue? yes, if the command is meant to be a smoke. It should fail or run the requested scenario before declaring pass. If the intended behavior is only “leave me a VM”, the safer alternative is to rename/document it as setup-only and remove scenario/artifact success claims from that path.
