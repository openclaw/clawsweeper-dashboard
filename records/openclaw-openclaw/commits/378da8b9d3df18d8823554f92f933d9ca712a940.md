# Commit 378da8b

## Summary

Found three actionable issues in the new Telegram Mantis evidence surfaces. The most important one is that the desktop builder can keep a Telegram gateway VM alive after returning the brokered Telegram credential to the shared pool.

## Findings

### Medium: kept Telegram gateway releases its exclusive credential lease

- Kind: reliability
- File: `extensions/qa-lab/src/mantis/telegram-desktop-builder.runtime.ts`
- Line: 881
- Evidence: `gatewaySetup` defaults to true, `keepLease` defaults to true when gateway setup is enabled, and the VM is only stopped when `!keepLease`. The remote script starts and disowns `openclaw gateway run`, but the `finally` block always stops the heartbeat and calls `credentialLease.release()`.
- Impact: the Convex broker can hand the same Telegram group/bot tokens to another QA run while the kept VM is still polling and replying with them. That can cause Telegram Bot API conflicts, lost updates, cross-run replies, and leaked QA traffic between supposedly exclusive runs.
- Suggested fix: when using brokered credentials, either stop the gateway/VM before releasing the lease, force `--no-keep-lease`, or keep the broker lease heartbeat tied to the retained VM lifecycle with an explicit cleanup command.
- Confidence: high

### Medium: secret-bearing workflow builds and executes Crabbox from mutable `main`

- Kind: supply_chain
- File: `.github/workflows/mantis-telegram-live.yml`
- Line: 50
- Evidence: the workflow sets `CRABBOX_REF: main`, fetches `openclaw/crabbox` at that moving ref, builds the CLI, then later runs `crabbox` in a step carrying OpenAI, Convex, and Crabbox coordinator/access secrets.
- Impact: a later change or compromise in the external Crabbox `main` branch changes what code this reviewed workflow executes with live QA secrets, without any target-repo review or reproducible pin.
- Suggested fix: pin Crabbox to a reviewed commit SHA or signed release artifact, and update the pin intentionally when the workflow needs new Crabbox behavior.
- Confidence: high

### Low: downloaded Telegram Desktop inherits QA secrets

- Kind: security
- File: `extensions/qa-lab/src/mantis/telegram-desktop-builder.runtime.ts`
- Line: 416
- Evidence: the builder downloads/extracts Telegram Desktop, then launches it from the same remote shell environment that was populated with `OPENCLAW_MANTIS_TELEGRAM_*` tokens and OpenAI/QA env needed later by gateway setup.
- Impact: the third-party desktop process does not need bot/API credentials, but can read inherited environment variables. A compromised or unexpected downloaded binary could exfiltrate tokens.
- Suggested fix: launch Telegram Desktop with a scrubbed environment such as `env -i HOME="$HOME" DISPLAY="$DISPLAY" PATH="$PATH" ...`, or start it before injecting QA secrets. Prefer a pinned/verifiable Telegram Desktop artifact if this stays automated.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the credential lease issue: run the builder with Convex credentials and default gateway setup, or a mocked runner like the existing tests; it will keep the Crabbox lease while still calling the broker release path. The other two are workflow/code-inspection reproductions.
- Is this the best way to solve the issue? Yes for the first two: credential exclusivity must match the lifetime of the running gateway, and secret-bearing CI should not execute a mutable external branch. For Telegram Desktop env inheritance, env scrubbing is the least disruptive fix.
