# Commit 378da8b

## Summary

Found two actionable issues in the new Telegram desktop builder. The highest-risk one can copy a real Telegram Desktop profile archive into the local Mantis artifact directory.

## Findings

### High: Telegram Desktop profile archive is copied back into artifacts

- Kind: security
- File: `extensions/qa-lab/src/mantis/telegram-desktop-builder.runtime.ts`
- Line: 400
- Evidence: The remote script decodes `OPENCLAW_MANTIS_TELEGRAM_DESKTOP_PROFILE_TGZ_B64` to `$out/telegram-profile.tgz` and extracts it, but never deletes it. Later `copyRemoteArtifacts` rsyncs the entire remote output directory back to the local output directory with no exclude list at `extensions/qa-lab/src/mantis/telegram-desktop-builder.runtime.ts:605`.
- Impact: A Telegram Desktop profile archive is login/session material. Any run using `--telegram-profile-archive-env` leaves that archive in `.artifacts/...`; if that artifact directory is uploaded, shared, or attached to evidence later, it can expose the Telegram user session.
- Suggested fix: Decode the archive to a temp path outside `$out`, or delete `$out/telegram-profile.tgz` immediately after extraction. Also add an rsync exclude for `telegram-profile.tgz` and a regression test that a supplied profile archive is not present in copied artifacts.
- Confidence: high

### Medium: Default kept gateway releases its pooled Telegram credential while still running

- Kind: reliability
- File: `extensions/qa-lab/src/mantis/telegram-desktop-builder.runtime.ts`
- Line: 671
- Evidence: `gatewaySetup` defaults on, and `keepLease` defaults true when gateway setup is enabled. The remote script starts and disowns `openclaw gateway run` at `extensions/qa-lab/src/mantis/telegram-desktop-builder.runtime.ts:505`, but the `finally` block skips stopping the VM when `keepLease` is true while still stopping the heartbeat and releasing the Convex credential lease at `extensions/qa-lab/src/mantis/telegram-desktop-builder.runtime.ts:876`.
- Impact: The kept VM can continue polling Telegram with the leased SUT bot token after the broker marks that credential available. A later QA run can receive the same token, creating duplicate Telegram pollers and likely `getUpdates` conflicts or stolen updates.
- Suggested fix: Do not release pooled credentials while a kept gateway remains alive, or refuse `--keep-lease` with Convex credentials unless the remote gateway is stopped first. For manual long-lived VNC sessions, require explicit env-owned credentials or add an explicit long-lived lease mode.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. For the archive leak, run the builder with `--telegram-profile-archive-env` and inspect the copied output directory; the code writes the archive under `$out` and rsyncs `$out/` wholesale. For the credential issue, run the default Convex-backed builder successfully; the code path keeps the VM/gateway while releasing the credential lease.
- Is this the best way to solve the issue? yes for the first issue: secret session archives should never live under copied artifact roots. For the second, the safer contract is to keep credential lifetime aligned with the live gateway lifetime.
