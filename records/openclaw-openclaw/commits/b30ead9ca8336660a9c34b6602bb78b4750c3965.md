# Commit b30ead9ca8

## Summary

Found two actionable issues in the new hiding/suppression path. The intended trusted subagent announce handoff is fixed, but the new trust decision is based on request fields that the public `agent` RPC schema accepts, and the display fallback can hide ordinary user messages by substring.

## Findings

### Medium: Public agent callers can spoof subagent announce metadata to suppress transcript persistence

- Kind: security
- File: `src/gateway/server-methods/agent.ts`
- Line: 515
- Evidence: `AgentParamsSchema` accepts caller-provided `internalEvents` and `inputProvenance` at `src/gateway/protocol/schema/agent.ts:172`, while `agent` only requires `operator.write` at `src/gateway/method-scopes.ts:133`. The new `shouldSuppressAgentPromptPersistence` trusts those request fields when `sourceTool === "subagent_announce"` plus a `task_completion` event from `subagent`, then passes `suppressPromptPersistence: true` into `agentCommand` at `src/gateway/server-methods/agent.ts:1436`. That becomes `suppressNextUserMessagePersistence` at `src/agents/command/attempt-execution.ts:639`, and the session guard drops the next persisted user message at `src/agents/session-tool-result-guard.ts:464`.
- Impact: Any authenticated write-scope gateway client can send an arbitrary prompt to the model while making that user turn absent from the session transcript by supplying forged internal handoff metadata. That weakens auditability and can hide prompts from later history/debugging.
- Suggested fix: Treat announce suppression as an internal-only capability. For example, ignore or sanitize public `inputProvenance`/`internalEvents` on `agent` requests unless the request comes from a trusted internal dispatch path, or require a non-user-spoofable internal flag/client marker and validate the source against an actual registered subagent completion.
- Confidence: high

### Low: Display projection hides any user message containing subagent announce marker substrings

- Kind: bug
- File: `src/gateway/chat-display-projection.ts`
- Line: 537
- Evidence: The legacy fallback drops a user message whenever projected text includes both `[Inter-session message]` and `sourceTool=subagent_announce`, regardless of whether the message has trusted provenance. A focused check confirmed `projectChatDisplayMessages([{ role: "user", content: "please explain [Inter-session message] sourceTool=subagent_announce" }])` returns `[]`.
- Impact: A legitimate user asking about the marker text, or a caller deliberately including those substrings, disappears from Control UI/WebChat history even though it was a normal user message.
- Suggested fix: Only apply the fallback to a tightly matched legacy wrapper shape, preferably with the internal-context envelope or structured provenance. Avoid broad `includes` checks on arbitrary user text.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. The suppression issue is source-reproducible through the public schema to session guard path. The display issue was directly checked with `projectChatDisplayMessages`.
- Is this the best way to solve the issue? Yes. The durable fix is to make internal announce metadata non-spoofable and tighten legacy display matching, while keeping the current trusted handoff behavior.
