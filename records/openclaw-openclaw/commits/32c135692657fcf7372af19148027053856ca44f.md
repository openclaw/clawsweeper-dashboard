# Commit 32c1356

## Summary

Found two actionable gaps in the HEIC normalization path. The main one breaks the new behavior on the supported Bun/macOS `sips` backend; the second leaves valid HEIC/HEIF sequence MIME variants unnormalized.

## Findings

### Medium: HEIC model-run conversion fails before `sips` can run on Bun/macOS

- Kind: compatibility
- File: `src/cli/capability-cli.ts`
- Line: 619
- Evidence: `readModelRunImageFiles` now calls `convertHeicToJpeg(buffer)` for `image/heic` and `image/heif`. That helper first calls `assertImagePixelLimit`; on the `sips` backend, `assertImagePixelLimit` throws when metadata cannot be read. The local header parser only handles PNG/GIF/WebP/JPEG, so HEIC metadata is never available before conversion. `prefersSips()` is the default on Bun + macOS unless `OPENCLAW_IMAGE_BACKEND=sharp` is set. A focused smoke with `OPENCLAW_IMAGE_BACKEND=sips` returned `Unable to determine image dimensions; refusing to process`.
- Impact: `openclaw infer model run --file photo.heic` fails on the Bun/macOS path that repo policy says must keep working, instead of normalizing the HEIC file to JPEG.
- Suggested fix: make HEIC conversion validate size through a HEIC-aware path before failing closed, or route HEIC through `sips`/sharp conversion with the converter’s own pixel limits before applying the header-only guard. Add a test that exercises the `sips` backend path rather than mocking `convertHeicToJpeg`.
- Confidence: high

### Low: HEIC/HEIF sequence files bypass normalization

- Kind: bug
- File: `src/cli/capability-cli.ts`
- Line: 94
- Evidence: `HEIC_MODEL_RUN_MIMES` includes only `image/heic` and `image/heif`. `detectMime` prefers non-generic sniffed MIME types over extension mappings, and installed `file-type` 22.0.1 returns `image/heif-sequence` for `msf1` and `image/heic-sequence` for `hevc`/`hevx`. A focused `detectMime` smoke returned `msf1=image/heif-sequence` and `hevc=image/heic-sequence`.
- Impact: valid HEIC-family files with sequence brands are accepted as `image/*` but sent to providers unconverted, defeating the normalization fix for those inputs.
- Suggested fix: include `image/heif-sequence` and `image/heic-sequence` in the HEIC normalization set, with a unit test covering those detected MIME values.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Finding 1 is reproducible by forcing `OPENCLAW_IMAGE_BACKEND=sips`; real Bun/macOS was not available, but the code path uses the same backend predicate. Finding 2 is reproducible through `detectMime` with HEIF/HEIC sequence `ftyp` brands.
- Is this the best way to solve the issue? yes for the MIME allowlist; for the `sips` path, the safest fix is to keep pixel-limit protection but make that protection HEIC-aware instead of using the PNG/JPEG-only header probe before conversion.
