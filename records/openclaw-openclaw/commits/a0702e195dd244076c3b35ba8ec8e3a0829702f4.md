# Commit a0702e195d

## Summary

Found one actionable CI reliability regression in the new pnpm cache setup. Some package/release jobs call `setup-node-env` with `install-deps: "false"` and can take npm/url/artifact paths that never run `pnpm install`; on a cache miss, `pnpm/action-setup` still tries to save the pnpm store in its post action and fails if the store directory was never created.

## Findings

### Medium: pnpm cache post step can fail package jobs that do not install dependencies

- Kind: reliability
- File: `.github/actions/setup-pnpm-store-cache/action.yml`
- Line: 45
- Evidence: `setup-pnpm-store-cache` defaults `use-actions-cache` to `true` and passes it directly to `pnpm/action-setup` as `cache`. `setup-node-env` also defaults `use-actions-cache` to `true`, even when `install-deps` is `false`. The affected current workflows include `.github/workflows/package-acceptance.yml:318` with `install-deps: "false"`, and its `source=npm`, `source=url`, and `source=artifact` paths only run `node scripts/resolve-openclaw-package-candidate.mjs`; that script runs `pnpm install` only for `source=ref`. Locally, `pnpm store path --silent` did not create the returned store directory in an empty HOME. Upstream `@actions/cache@4.1.0` throws `Path Validation Error: Path(s) specified in the action for caching do(es) not exist` when saving a missing cache path, and `pnpm/action-setup` catches that in its post cache-save wrapper by calling `setFailed`.
- Impact: Package Acceptance and release-package resolution jobs can fail at post-job cleanup on cold cache or lockfile-key misses despite the package resolution itself succeeding. This is most direct for `package-acceptance.yml` with non-ref package sources, and also plausible for release checks when a published `release_package_spec` avoids the ref worktree install.
- Suggested fix: Disable the built-in `pnpm/action-setup` cache for setup paths that do not run `pnpm install`, or restore the old split restore/save behavior so saving is skipped or non-fatal unless a store was actually populated. For `package-acceptance.yml`, pass `use-actions-cache: ${{ inputs.source == 'ref' && 'true' || 'false' }}` or equivalent.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes: run a workflow path that uses `setup-node-env` with `install-deps: "false"` and `package-acceptance` `source=npm`, `source=url`, or `source=artifact` after a cache miss. The post step will attempt to save the missing pnpm store path.
- Is this the best way to solve the issue? Yes: the cache should be disabled or save-skipped when no pnpm install will populate the store. Creating an empty store directory would avoid the crash but would save a useless cache.
