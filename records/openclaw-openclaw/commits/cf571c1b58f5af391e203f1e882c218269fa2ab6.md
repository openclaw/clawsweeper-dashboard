# Commit cf571c1

## Summary

Found one install-scan regression: the new runtime graph resolver misses a normal Node local directory import resolved through that directory’s `package.json`.

## Findings

### Medium: Runtime graph scan misses local directory modules with package.json main

- Kind: security
- File: `src/plugins/install-security-scan.runtime.ts`
- Line: 868
- Evidence: `resolveRuntimeGraphFileCandidate()` resolves extensionless imports by trying the raw path, `path + extension`, and `path/index.*`. It never checks `path/package.json` `main`/exports. Since `scanPackageInstallSourceRuntime()` now scans only `runtimeGraphScanEntries` with `onlyIncludeFiles: true`, a reachable local runtime file can be omitted entirely.
- Impact: A package plugin can put dangerous code in `payload/runtime.js`, set `payload/package.json` to `{ "main": "runtime.js" }`, and have `index.js` call `require("./payload")`. Node resolves that import to `payload/runtime.js`, but the install scanner skips it and allows the install with no warning.
- Suggested fix: Resolve relative runtime imports with Node-compatible package-directory handling before limiting the scanner to included files. At minimum, when an extensionless candidate is a directory, read its `package.json` and enqueue its `main` target plus `index.*` fallbacks while preserving package-boundary checks. Add a regression test for `require("./payload")` with `payload/package.json` pointing at a dangerous runtime file.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A temporary `installPluginFromDir` fixture with `index.js` requiring `./payload`, `payload/package.json` main set to `runtime.js`, and dangerous `child_process.execSync` in `payload/runtime.js` returned `{ ok: true, warnings: [] }`. The same fixture’s `createRequire(...).resolve("./payload")` returned `payload/runtime.js`.
- Is this the best way to solve the issue? yes. The intended scope is plugin-owned runtime graph, so the safer fix is to make graph resolution match normal local runtime resolution instead of broadening back to all dependency code.
