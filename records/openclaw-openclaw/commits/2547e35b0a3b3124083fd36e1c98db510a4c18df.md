# Commit 2547e35

## Summary

Found one low-severity QA harness false-negative risk in the new WebChat direct-reply sentinel.

## Findings

### Low: WebChat direct-reply scenario can pass message-tool self-sends that do not end in `Sent.`

- Kind: test_gap
- File: `qa/scenarios/channels/webchat-direct-reply-routing.md`
- Line: 59
- Evidence: The scenario accepts any outbound message containing the marker, then only asserts `!transcriptSummary.hasDirectReplySelfMessage`. That summary is backed by `scanDirectReplyTranscriptSentinels`, which only reports a self-message when a current-chat `message(action=send)` call exists and the final assistant text normalizes exactly to `Sent.` (`extensions/qa-lab/src/gateway-log-sentinel.ts:282`).
- Impact: A regression where the agent calls `message(action=send)` with the marker and then returns empty text, `NO_REPLY`, or any non-`Sent.` final text would still pass. A duplicate-delivery case can also pass because the scenario does not assert exactly one matching outbound message despite its success criteria.
- Suggested fix: For this scenario, expose/assert a broader transcript flag for any current-chat `message(action=send)` call, and also collect matching outbound messages since `startIndex` to assert exactly one visible marker reply.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. A transcript containing an assistant `message` tool call to the current QA operator plus a final assistant text other than `Sent.` makes `hasDirectReplySelfMessage` false, while the current outbound predicate passes as soon as one marker message is visible.
- Is this the best way to solve the issue? Yes. The sentinel should assert the exact contract it documents: no current-chat message tool call for this direct reply, and exactly one visible marker-bearing outbound reply.
