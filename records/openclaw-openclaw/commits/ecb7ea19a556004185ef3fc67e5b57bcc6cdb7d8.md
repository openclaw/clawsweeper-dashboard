# Commit ecb7ea1

## Summary

Found two actionable issues in the new Telegram real-user Crabbox proof path: the Convex credential lease is not heartbeated while the long-running proof owns the burner account, and the documented/prebaked TDLib path can install a native library fetched over an unauthenticated transport with a checksum fetched from the same transport.

## Findings

### Medium: Telegram user credential leases can expire while the proof is still using the account

- Kind: concurrency
- File: `scripts/e2e/telegram-user-credential.ts`
- Line: 513
- Evidence: `lease-restore` sends `leaseTtlMs` and `heartbeatIntervalMs` to `acquire`, writes a lease file, and only implements `release`; there is no `heartbeat` broker call in `scripts/e2e/telegram-user-credential.ts` or `scripts/e2e/telegram-user-crabbox-proof.ts`. The shared QA lease runtime has a separate `heartbeat()` and `startQaCredentialLeaseHeartbeat()` contract, so the broker expects long holders to refresh the lease.
- Impact: the default credential TTL is 20 minutes, while the Crabbox proof can run longer during apt install, TDLib build/download, desktop setup, recording, or current held-session use. Once the lease expires, the pool can hand the same Telegram burner account to another job while the first job still has TDLib/Desktop restored and active, causing cross-job message races and invalid visual proof.
- Suggested fix: make the Crabbox proof process heartbeat the acquired lease until `release`/cleanup completes, reusing the shared credential lease helper if possible. Persist enough lease metadata for held sessions to heartbeat, and fail the session if heartbeat fails.
- Confidence: high

### Medium: Prebaked TDLib downloads can be tampered with before root install

- Kind: supply_chain
- File: `scripts/e2e/telegram-user-crabbox-proof.ts`
- Line: 916
- Evidence: when `--tdlib-url` is passed, the remote setup script downloads `$tdlib_url`, and if `--tdlib-sha256` is absent it downloads `$tdlib_url.sha256` from the same URL base before installing the extracted `libtdjson.so` with `sudo install`. Current docs show `--tdlib-url http://artifacts.openclaw.ai/tdlib-v1.8.0-linux-x64.tgz`, and say the default verification source is the sibling `.sha256` file.
- Impact: a network attacker on the Crabbox path can replace both the HTTP tarball and the sibling checksum, gaining native code execution as root on the remote box and access to the restored Telegram burner state/SUT bot token used for the proof.
- Suggested fix: change docs to HTTPS, reject non-HTTPS `--tdlib-url` unless an explicit `--tdlib-sha256` is supplied out of band, and preferably pin the official TDLib artifact hash in docs or script examples.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. For the lease issue, run a proof with a short `--lease-ttl-ms` or a setup phase longer than the default 20 minutes and observe no `POST /heartbeat` call before release. For the TDLib issue, run with an HTTP `--tdlib-url` and no explicit `--tdlib-sha256`; both the tarball and checksum are fetched from the same unauthenticated URL before root install.
- Is this the best way to solve the issue? Yes. Heartbeating matches the existing broker contract and prevents duplicate account leases. Requiring HTTPS or an out-of-band hash closes the checksum-from-same-untrusted-channel gap without removing the fast prebaked TDLib path.
