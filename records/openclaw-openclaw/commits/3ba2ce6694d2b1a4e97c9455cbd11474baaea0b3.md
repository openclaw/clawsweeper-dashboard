# Commit 3ba2ce66

## Summary

Found one regression risk in the `--prefix .` removal: npm can now climb to a parent workspace when the managed npm root is nested under one, so plugin install/uninstall/prune may operate on the wrong npm project.

## Findings

### Medium: Managed npm commands can target a parent npm workspace

- Kind: regression
- File: `src/plugins/install.ts`
- Line: 489
- Evidence: The commit removed `--prefix .` from managed npm commands and now relies only on `cwd: npmRoot` at `src/plugins/install.ts:489`, rollback uninstall at `src/plugins/install.ts:322`, peer repair at `src/infra/npm-managed-root.ts:239`, and uninstall cleanup at `src/plugins/uninstall.ts:606`. `resolveDefaultPluginNpmDir()` derives the root from `$OPENCLAW_STATE_DIR` at `src/plugins/install-paths.ts:83`, and that override is public/configurable. In a focused npm check, when `npmRoot` was a child package inside a parent npm workspace, `npm install ...` from `cwd: npmRoot` without `--prefix .` tried to resolve the parent workspace dependency and failed with `E404`; the old `--prefix .` command wrote the child lockfile successfully. `npm install ... --workspaces=false` also stayed in the child.
- Impact: Users or test/dev setups that put `OPENCLAW_STATE_DIR` under an npm workspace can get failed plugin installs or, worse, managed plugin uninstall/prune/install commands mutating the parent project’s `package-lock.json`/dependencies instead of `$OPENCLAW_STATE_DIR/npm`.
- Suggested fix: Keep avoiding `--prefix .` for the Windows Arborist crash, but make managed-root npm commands explicitly non-workspace, e.g. add `--workspaces=false` to install/uninstall/prune invocations and cover a managed npm root nested under a parent workspace.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Create a parent `package.json` with `workspaces: ["managed-npm"]`, put the managed npm `package.json` in `managed-npm`, and run the new npm install argv from that child. Local npm 11.11.0 selected the parent workspace; old `--prefix .` and `--workspaces=false` selected the child.
- Is this the best way to solve the issue? no. The Windows fix direction is sound, but it needs an explicit workspace-disabling flag so cwd-only npm does not rediscover a parent project.
