# Commit 6c6b6b7

## Summary

Found one concrete issue in the new `/context map` path: it writes a sensitive diagnostic image into the generic OS temp directory with default file permissions, and that same path choice can make delivery drop the image under hardened media policies.

## Findings

### Medium: Sensitive context maps are written to world-readable `/tmp`

- Kind: privacy
- File: `src/auto-reply/reply/context-treemap.ts`
- Line: 490
- Evidence: `renderContextTreemapPng` writes the PNG to `path.join(os.tmpdir(), ...)` with plain `writeFile(...)`. The caller returns that path as local media and explicitly marks it `sensitiveMedia: true` in `src/auto-reply/reply/commands-context-report.ts:127`. A focused probe generated `/tmp/openclaw-context-map-...png` with mode `0644`.
- Impact: On shared hosts, another local user can enumerate/read these context-map files from `/tmp`. The image includes context composition metadata and workspace file labels. Also, under `tools.profile="messaging"` or `tools.fs.workspaceOnly=true`, the reply-media normalizer does not widen roots to `/tmp`, so the advertised `/context map` command degrades to text plus `⚠️ Media failed.` instead of sending the image.
- Suggested fix: Write the PNG under `resolvePreferredOpenClawTmpDir()` with restrictive mode, or better stage it through the managed outbound media store. That puts it under the media roots already trusted by delivery and avoids world-readable generic temp files.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Generate a map and stat the returned path; it is under `/tmp` and mode `0644`. Then normalize the payload with `cfg: { tools: { profile: "messaging" } }`; the media is dropped and the text gains `⚠️ Media failed.`.
- Is this the best way to solve the issue? yes. The repo already centralizes safe temp/media roots in `src/media/local-roots.ts:37` via `resolvePreferredOpenClawTmpDir()`, and delivery trusts those roots.
