# Commit 003bed0c

## Summary

Found one low-severity alias gap in the new Fireworks thinking policy surface. Canonical `fireworks/...` Kimi refs are pinned to `off`, but the user-facing provider alias `fireworks-ai` still resolves through the generic thinking ladder on the cold public-policy path.

## Findings

### Low: Fireworks alias bypasses the new public thinking policy

- Kind: bug
- File: `extensions/fireworks/provider-policy-api.ts`
- Line: 3
- Evidence: `extensions/fireworks/index.ts:60` declares `fireworks-ai` as a provider alias, and the stream wrapper already treats it as Fireworks. The new public policy artifact works for `fireworks`, but `resolveBundledProviderPolicySurface` only finds bundled policy owners through manifest `providers`; `extensions/fireworks/openclaw.plugin.json:7` declares only `fireworks`. Focused smoke confirmed `listThinkingLevels("fireworks-ai", "accounts/fireworks/models/kimi-k2p6")` returns `["off","minimal","low","medium","high"]` and resolves `high` as supported, while canonical `fireworks` returns only `["off"]`.
- Impact: cold control-plane surfaces or CLI validation using the supported `fireworks-ai/...` provider alias can still expose and accept non-off thinking levels for Kimi. The request wrapper later forces `thinking: disabled`, so this is likely UX/state inconsistency rather than a provider rejection, but it violates this commit’s “thinking-off-only” contract for the alias.
- Suggested fix: make bundled provider policy lookup alias-aware for Fireworks, either by declaring the alias in the manifest ownership path if that is the intended contract, or by teaching `resolveBundledProviderPolicyPluginId` to consider provider/model-catalog aliases. Add a regression check for `fireworks-ai` resolving Kimi to `["off"]`.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the policy miss: the direct resolver smoke above reproduces it without credentials. I did not live-run Fireworks because `FIREWORKS_API_KEY` was absent from the environment and `~/.profile`.
- Is this the best way to solve the issue? Likely yes directionally: the fix should make the public policy artifact reachable for the same user-facing alias already accepted by Fireworks runtime hooks, instead of special-casing command validation.
