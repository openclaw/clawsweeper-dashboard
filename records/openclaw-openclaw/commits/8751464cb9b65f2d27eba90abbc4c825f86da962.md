# Commit 8751464

## Summary

Found one regression in the RTT driver scenario parsing. The lint rewrite changes configured scenario IDs into single-character set entries, so the normal warm Telegram RTT scenario is skipped.

## Findings

### Medium: Configured RTT scenarios are parsed as characters, skipping the warm reply scenario

- Kind: regression
- File: `scripts/e2e/npm-telegram-rtt-driver.mjs`
- Line: 19
- Evidence: The new code builds `new Set(process.env.OPENCLAW_NPM_TELEGRAM_SCENARIOS ?? "telegram-mentioned-message-reply".split(",")...)`. When the env var is set, `new Set("telegram-mentioned-message-reply")` creates a character set, so `scenarioIds.has("telegram-mentioned-message-reply")` at `scripts/e2e/npm-telegram-rtt-driver.mjs:313` is false. Both real paths set that env var: `scripts/lib/rtt-harness.ts:148` joins the CLI scenarios, and `scripts/e2e/npm-telegram-rtt-docker.sh:90` passes a default if unset.
- Impact: `pnpm rtt` / the RTT Docker harness can pass after only the canary scenario and silently skip the intended warm sample loop, producing no mention-reply RTT metrics and masking regressions in the measured Telegram reply path.
- Suggested fix: Parenthesize before splitting: `new Set((process.env.OPENCLAW_NPM_TELEGRAM_SCENARIOS ?? "telegram-mentioned-message-reply").split(",").map(...).filter(Boolean))`. Add a focused test for the driver scenario parser with `OPENCLAW_NPM_TELEGRAM_SCENARIOS=telegram-mentioned-message-reply`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused Node probe with `OPENCLAW_NPM_TELEGRAM_SCENARIOS=telegram-mentioned-message-reply` produced `hasMention:false`; the same expression without the env var produced `hasMention:true`.
- Is this the best way to solve the issue? yes. The pre-commit behavior split the coalesced string before filtering; restoring those parentheses keeps the lint-friendly `Set` change without changing the scenario contract.
