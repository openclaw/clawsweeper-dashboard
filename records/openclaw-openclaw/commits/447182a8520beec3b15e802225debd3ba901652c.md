# Commit 447182a

## Summary

The commit adds Telegram inbound delivery correlation, but the main `sendMessage` path still compares the active inbound chat id against the pre-send target string. Real message-tool targets are commonly normalized or user-facing aliases, so the successful send can still leave the inbound turn marked undelivered and trigger the silent fallback the commit is meant to suppress. I also found the analogous Telegram poll action path has no delivery notification at all.

## Findings

### Medium: Telegram message-tool sends still miss same-chat delivery after target normalization

- Kind: bug
- File: `extensions/telegram/src/action-runtime.ts`
- Line: 399
- Evidence: `dispatchTelegramMessage` registers the active inbound turn with `outboundTo: String(chatId)` in `extensions/telegram/src/bot-message-dispatch.ts:688`. `notifyTelegramInboundTurnOutboundSuccess` then requires exact string equality in `extensions/telegram/src/inbound-turn-delivery.ts:34`. But after `sendMessageTelegram` succeeds, `handleTelegramAction` passes the original `to` value to the notifier in `extensions/telegram/src/action-runtime.ts:399`, even though `sendMessageTelegram` resolves the actual Telegram chat id and returns it as `result.chatId` in `extensions/telegram/src/send.ts:591`. The Telegram target normalizer returns provider-prefixed targets like `telegram:123` in `extensions/telegram/src/normalize.ts:38`, while the inbound registry stores `123`. A focused probe confirmed this exact mismatch: `normalizeTelegramMessagingTarget("123")` produced `telegram:123`; notifying with `telegram:123` left the delivery count at `0`, while notifying with raw `123` marked it delivered.
- Impact: A same-chat `message(action=send)` from an inbound Telegram turn can still emit the unwanted rewritten silent fallback after the visible tool send succeeds. This reproduces the user-facing duplicate/filler behavior the commit claims to fix, especially on gateway-normalized `target=current`/numeric targets and aliases such as usernames.
- Suggested fix: Notify with the successful send result’s resolved `chatId` instead of `to`, or normalize both sides through the same Telegram chat-id canonicalization. Add a regression test where the active turn stores `outboundTo: "123"` and the action uses `to: "telegram:123"` or a username while `sendMessageTelegram` returns `chatId: "123"`.
- Confidence: high

### Low: Telegram poll message actions never mark inbound delivery

- Kind: bug
- File: `extensions/telegram/src/action-runtime.ts`
- Line: 462
- Evidence: `message(action=poll)` can execute through the same gateway plugin action path in `src/infra/outbound/message-action-runner.ts:859`. The Telegram poll branch sends successfully via `sendPollTelegram` and returns `messageId`, `chatId`, and `pollId` in `extensions/telegram/src/action-runtime.ts:462`, but unlike the send-message branch, it never calls `notifyTelegramInboundTurnOutboundSuccess`. The commit forwards `sessionKey` through Telegram channel actions, but the poll branch does not use it.
- Impact: A Telegram inbound turn that creates a poll as its visible reply can still be considered undelivered and receive the silent fallback afterward.
- Suggested fix: After a successful `sendPollTelegram`, call the same delivery notifier using `result.chatId`, guarded by the session/account just like the fixed send-message path should be.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. The source path is deterministic: inbound dispatch stores raw `String(chatId)`, message action target resolution can use provider-prefixed targets, and the notifier compares exact strings. The focused `tsx` probe reproduced the mismatch without Telegram credentials.
- Is this the best way to solve the issue? Yes for the main finding: use the resolved Telegram `result.chatId` returned by the successful API send, because it is the same canonical value inbound dispatch stores.
