# Commit 1c33990108

## Summary

Found one regression in the OpenAI API-key first-run path. The commit routes default `openai/*` agent runs through the Codex harness, but standard OpenAI API-key onboarding still creates only an `openai:*` profile. That profile is reported as making the default OpenAI model available, but it is not selected or forwarded to the Codex harness.

## Findings

### Medium: OpenAI API-key onboarding no longer authenticates the default Codex-routed agent path

- Kind: regression
- File: `extensions/openai/openai-provider.ts`
- Line: 221
- Evidence: OpenAI API-key onboarding is still registered as provider `openai` and writes the default OpenAI model/profile path (`providerId: PROVIDER_ID`, `defaultModel: OPENAI_DEFAULT_MODEL`) at `extensions/openai/openai-provider.ts:221`. The new default harness policy routes ordinary OpenAI agent runs to `codex` at `src/agents/harness/selection.ts:318`. For a Codex OpenAI run, accepted auth providers are only `openai-codex` at `src/agents/openai-codex-routing.ts:127`, and the runtime auth plan only forwards a profile when the selected profile provider matches the harness auth provider at `src/agents/runtime-plan/auth.ts:41`. The committed test at `src/agents/runtime-plan/build.test.ts:118` confirms `openai:work` is intentionally not forwarded into the Codex harness auth slot.
- Impact: A user who runs the documented/standard API-key setup gets `openai:default` and `openai/gpt-5.5`, but a default agent turn through Codex has no forwarded auth profile. With no inherited `OPENAI_API_KEY`, the agent run can fail despite `models status` reporting no missing provider. The status/list UI also reinforces the bad state: the smoke check showed `missingProvidersInUse: []` and OpenAI models marked available with only `openai:default`.
- Suggested fix: Either make OpenAI API-key onboarding also create/order an `openai-codex:*` API-key profile for agent runs, or keep API-key-onboarded `openai/*` agent defaults on the PI/direct runtime. Align `models status`, `models list`, and docs with whichever contract is intended.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. In temp state with `OPENAI_API_KEY` unset, `pnpm openclaw onboard --openai-api-key sk-test-review-only --non-interactive --accept-risk` wrote `openai:default` only and defaulted to `openai/gpt-5.5`; `pnpm openclaw models status --json` then reported no missing provider. A direct resolver probe with that same profile shape returned `{"acceptedProviderIds":["openai-codex"],"selected":null}` for the Codex OpenAI runtime.
- Is this the best way to solve the issue? unclear. The safest direction is to preserve the existing first-run promise by creating/ordering a matching `openai-codex` API-key profile when API-key onboarding selects a Codex-routed OpenAI agent model. If the product intent is to require explicit Codex auth for all default OpenAI agent runs, then onboarding/status should stop presenting `openai:default` as sufficient for that default.
