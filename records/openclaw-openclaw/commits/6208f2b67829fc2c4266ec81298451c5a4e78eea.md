# Commit 6208f2b

## Summary

Found one low-severity user-visible regression in the OAuth failure path: the new `cause` preserves useful diagnostics, but the outer error message already embeds the formatted inner error, and OpenClaw’s formatter also walks `.cause`.

## Findings

### Low: xAI OAuth login failures duplicate the underlying error text

- Kind: bug
- File: `extensions/xai/xai-oauth.ts`
- Line: 304
- Evidence: The commit changed the catch block to `new Error(\`xAI OAuth failed: ${formatErrorMessage(err)}\`, { cause: err })`. `formatErrorMessage` in `src/infra/errors.ts:57` starts with `err.message` and then appends each `.cause` message with ` | `. CLI failure output uses `formatErrorMessage(options.error)` for the displayed `Reason` in `src/cli/failure-output.ts:29`, reached by CLI catches such as `src/index.ts:104`.
- Impact: A failed xAI OAuth login now reports duplicated detail, for example `xAI OAuth failed: discovery failed | discovery failed`. This does not break auth, but it regresses the user-facing error string and can make logs/diagnostics noisier.
- Suggested fix: Preserve the cause without pre-flattening it into the outer message, for example `new Error("xAI OAuth failed", { cause: err })`, or use a helper/formatter pattern that avoids appending an identical cause twice.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused formatter probe produced `xAI OAuth failed: discovery failed | discovery failed` for the exact wrapping pattern introduced here.
- Is this the best way to solve the issue? yes. Keeping `cause` is useful, but the outer message should not also include the already-formatted cause when the standard formatter traverses causes.
