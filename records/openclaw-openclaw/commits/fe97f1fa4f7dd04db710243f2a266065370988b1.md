# Commit fe97f1f

## Summary

Found one low-severity validation gap in the new base64 check. The main malformed-character case is covered, but malformed padding-only or over-padded payloads still pass and reach the realtime bridge.

## Findings

### Low: Padding-only and over-padded payloads still pass validation

- Kind: bug
- File: `extensions/voice-call/src/webhook/stream-frame-adapter.ts`
- Line: 51
- Evidence: `normalizeBase64ForCompare()` strips all trailing `=` before comparing decoded/re-encoded data. A focused probe against the actual adapter showed `payload: "===="` returns `{kind:"media","payloadBase64":"===="}` for both Twilio and Telnyx, and `payload: "AAA==="` is also accepted.
- Impact: malformed provider media-frame base64 is still forwarded into the realtime bridge path. `====` decodes to a zero-byte audio buffer at `extensions/voice-call/src/webhook/realtime-handler.ts:436`, which then emits an input audio event and calls the configured realtime provider bridge instead of being ignored.
- Suggested fix: validate padding before round-tripping, or use an existing strict pattern/helper shape: reject empty decoded payloads, reject `=` unless it appears only as the canonical final padding, and reject lengths/padding combinations like `====` and `AAA===`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Run `node --import tsx` importing `TwilioStreamFrameAdapter` / `TelnyxStreamFrameAdapter`, then parse a media frame with `payload: "===="` or `"AAA==="`; the adapter returns `kind: "media"`.
- Is this the best way to solve the issue? yes. The bug is in the new validator, before the downstream bridge decode, so strict padding validation at `isValidBase64Payload()` is the narrowest fix.
