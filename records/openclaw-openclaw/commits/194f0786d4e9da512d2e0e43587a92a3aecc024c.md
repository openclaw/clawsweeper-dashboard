# Commit 194f078

## Summary

Found one remaining security gap in the WhatsApp auth hardening. The commit rejects symlinked `creds.json` paths before auth checks and socket startup, and later commits on current `main` also guard atomic `creds.json`/backup writes, but the live socket still hands the auth directory to Baileys’ multi-file key store, which reads and writes other auth key files through symlinks.

## Findings

### Medium: Baileys auth key files can still be read and written through symlinks

- Kind: security
- File: `extensions/whatsapp/src/session.ts`
- Line: 162
- Evidence: Current `createWaSocket` only rejects `resolveWebCredsPath(authDir)` before calling `useMultiFileAuthState(authDir)`. Baileys `7.0.0-rc12` then reads and writes key files such as `${type}-${id}.json` with plain `fs.promises.readFile`, `writeFile`, and `unlink`. I verified this dependency behavior with the installed package: a symlinked `pre-key-1.json` was read through to an outside target, and `keys.set({ "pre-key": { "2": ... } })` wrote through a symlinked `pre-key-2.json` while leaving the symlink in place. OpenClaw already treats `app-state-sync-*`, `session-*`, `sender-key-*`, and `pre-key-*` as WhatsApp/Baileys auth files in `extensions/whatsapp/src/auth-store.ts`.
- Impact: A symlinked non-`creds.json` auth file under the WhatsApp auth directory can still cause OpenClaw/Baileys to read from or write to an arbitrary same-user filesystem target. That leaves part of the symlinked credential-file class unfixed despite the new `creds.json` checks.
- Suggested fix: Stop exposing Baileys’ raw multi-file key store to runtime paths. Wrap `state.keys` with no-follow, parent-safe read/write/remove helpers for all Baileys auth filenames, or replace `useMultiFileAuthState` with an OpenClaw-owned safe multi-file auth store. At minimum, reject or quarantine symlinked Baileys auth files before socket startup and before key writes.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. With `baileys@7.0.0-rc12`, create an auth dir containing regular `creds.json` plus symlinked `pre-key-1.json`/`pre-key-2.json`, call `useMultiFileAuthState(authDir)`, then `state.keys.get("pre-key", ["1"])` and `state.keys.set({ "pre-key": { "2": value } })`; Baileys follows both symlinks.
- Is this the best way to solve the issue? yes. The boundary that currently needs hardening is the key store passed to Baileys, not only the pre-start `creds.json` probe.
