# Commit 38b3e736

## Summary

Found one actionable diagnostics bug. The new deep Gateway client inspection can miss the stale client process it is meant to surface when Unix `lsof -F` reports multiple socket names under the same PID.

## Findings

### Medium: Lsof connection parsing drops later sockets for the same PID

- Kind: bug
- File: `src/infra/ports-inspect.ts`
- Line: 147
- Evidence: `parseLsofConnectionFieldOutput` feeds connection output through `parseLsofFieldOutput`, but `parseLsofFieldOutput` keeps only the first `n` address per process (`if (!current.address) current.address = ...`). A live focused check with one Node process owning a loopback server and two clients produced one `p...` record with four `n...` socket lines; `inspectPortConnections` returned only the first socket, classified as `server`.
- Impact: `openclaw gateway status --deep` and `openclaw doctor --deep` both filter out `direction === "server"` connections, so this shape displays no stale client even though the same PID has client-side Gateway connections. That weakens the rollback protocol-mismatch diagnostic path added by this commit.
- Suggested fix: parse `lsof -F` file/socket records as separate connection entries while carrying the current PID/command forward, instead of collapsing all `n` fields into one process-level address. Add a regression where one PID has a server-side `n` first and a client-side Gateway `n` later.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. I reproduced the parser behavior against real `lsof` output from a local TCP server/client process; `inspectPortConnections` returned only the first server-side socket, which the new status/doctor paths would filter out.
- Is this the best way to solve the issue? yes. The bug is in the lsof field parser, and lsof success prevents the Unix `ss` fallback from running. Fixing record parsing preserves macOS support, where `ss` is not a portable fallback.
