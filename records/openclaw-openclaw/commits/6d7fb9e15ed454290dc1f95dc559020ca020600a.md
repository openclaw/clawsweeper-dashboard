# Commit 6d7fb9e

## Summary

Found one concrete regression risk in the new bundled OpenRouter Kimi K2.5 catalog row: it advertises the full context window as the generation cap.

## Findings

### Medium: Kimi K2.5 fallback advertises an oversized completion cap

- Kind: reliability
- File: `extensions/openrouter/provider-catalog.ts`
- Line: 84
- Evidence: The new static `moonshotai/kimi-k2.5` row sets `maxTokens: 262144`, but OpenRouter’s live `/api/v1/models` metadata for `moonshotai/kimi-k2.5` returned `top_provider.max_completion_tokens: 65535`. OpenClaw’s OpenRouter live capability parser already maps that upstream field into `maxTokens` at `src/agents/pi-embedded-runner/openrouter-model-capabilities.ts:178`, and the OpenAI-compatible transport sends `model.maxTokens` as `max_completion_tokens`/`max_tokens` when no runtime override is provided at `src/agents/openai-transport-stream.ts:2046`. The OpenRouter model page also confirms the 262,144-token value is the context size, not the output cap. ([openrouter.ai](https://openrouter.ai/moonshotai/kimi-k2.5))
- Impact: Users who select the bundled fallback K2.5 row can get runtime requests carrying an impossible default completion cap, likely producing provider-side request rejection instead of a working first turn. This is exactly the class of failure the static fallback is meant to avoid when live discovery is unavailable or when the catalog is merged into configured provider metadata.
- Suggested fix: Set the K2.5 static row’s `maxTokens` to the live OpenRouter completion cap, currently `65535`, and add a focused assertion for the metadata values, not only the model id.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the bad metadata path: `pnpm tsx -e` against `buildOpenrouterProvider()` returns `maxTokens: 262144`, while `curl -fsSL https://openrouter.ai/api/v1/models | jq ...` returns `top_provider.max_completion_tokens: 65535` for the same model. A live completion rejection was not sent because no `OPENROUTER_API_KEY` was present in the environment/profile and using a real provider call would spend user credentials.
- Is this the best way to solve the issue? Yes. The static fallback should mirror the same upstream field already used by OpenClaw’s dynamic OpenRouter capability parser, so the fix is a narrow metadata correction plus regression assertion.
