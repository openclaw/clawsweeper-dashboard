# Commit d1b2d81752

## Summary

Found one user-facing regression in the browser OpenAI Realtime WebRTC path: the new attribution headers are sent from browser JavaScript to OpenAI’s SDP endpoint, but OpenAI’s current CORS preflight response does not allow those headers.

## Findings

### Medium: Browser Realtime Talk SDP setup now fails CORS preflight

- Kind: regression
- File: `ui/src/ui/chat/realtime-talk-webrtc.ts`
- Line: 75
- Evidence: `extensions/openai/realtime-voice-provider.ts:632` builds `offerHeaders` with `originator` and `version`, returns them at `extensions/openai/realtime-voice-provider.ts:722`, and the browser sends them in `fetch()` at `ui/src/ui/chat/realtime-talk-webrtc.ts:75`. OpenAI’s WebRTC docs show the direct browser SDP POST using `Authorization` and `Content-Type` for `/v1/realtime/calls`, not these extra custom headers. citeturn0search0 A live preflight check on 2026-04-29 confirmed `OPTIONS https://api.openai.com/v1/realtime/calls` with `Access-Control-Request-Headers: authorization,content-type,originator,version` returns `access-control-allow-headers: authorization,content-type`, omitting `originator` and `version`.
- Impact: Native OpenAI Realtime Talk browser sessions can fail before receiving the SDP answer because the browser blocks the POST during CORS preflight. The new unit test only mocks `fetch`, so it locks in the failing headers without exercising browser CORS.
- Suggested fix: Do not pass OpenClaw attribution headers to the browser-side SDP offer request. Keep browser offer headers to CORS-allowed headers, or proxy the SDP offer through a Gateway/server endpoint and add attribution headers server-side where CORS does not apply.
- Confidence: high
