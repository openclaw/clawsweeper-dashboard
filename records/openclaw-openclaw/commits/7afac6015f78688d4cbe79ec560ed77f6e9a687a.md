# Commit 7afac6015f

## Summary

Found one high-confidence security regression: action-triggered modal dialog state can be returned from `/act` without enforcing the current-tab URL policy, leaking dialog text/default values from a denied page.

## Findings

### High: Blocked `/act` dialog responses bypass current-URL policy

- Kind: security
- File: `extensions/browser/src/browser/routes/agent.act.ts`
- Line: 657
- Evidence: `/act` enters `withRouteTabContext` at `extensions/browser/src/browser/routes/agent.act.ts:385` without `enforceCurrentUrlAllowed`, then returns `browserState` directly when `executeActViaPlaywright` reports `blockedByDialog` at `extensions/browser/src/browser/routes/agent.act.ts:657`. That state includes dialog `message` and optional `defaultValue` from `extensions/browser/src/browser/pw-session.ts:71`. The guarded paths show the intended policy boundary: snapshot checks `assertBrowserNavigationResultAllowed` before collecting observed dialog state at `extensions/browser/src/browser/routes/agent.snapshot.ts:551`, and debug `/dialogs` uses `enforceCurrentUrlAllowed: true` at `extensions/browser/src/browser/routes/agent.debug.ts:111`.
- Impact: If the selected tab is on a URL denied by browser SSRF policy, an action that hits or observes a modal can still return page-controlled dialog text through `/act`. That bypasses the current-tab read guard and can expose private/local-page content that snapshot/debug routes would block.
- Suggested fix: Gate the blocked-dialog `/act` response with the same current-tab URL policy before including `browserState`, either by setting `enforceCurrentUrlAllowed: true` where safe for `/act` or by explicitly asserting the current tab URL only before returning dialog state. Add a denied-current-URL regression test for the blocked `/act` path.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Configure browser SSRF policy to deny private network URLs, put the current tab on `http://127.0.0.1:...`, trigger or mock an action returning `blockedByDialog` with pending dialog state, then `POST /act`; current code returns `ok: true` plus `browserState.dialogs.pending[].message` instead of failing closed.
- Is this the best way to solve the issue? yes. The feature is reasonable, but dialog text is tab-scoped page content, so it must use the same current URL guard as snapshot/debug dialog state.
