# Commit 5618a8f

## Summary

Found one concrete sandbox integration bug: newly allowed symlinked skills are copied into sandbox workspaces as symlinks, then the sandbox read boundary rejects them.

## Findings

### Medium: Allowed symlink skills become unreadable in sandboxed workspaces

- Kind: bug
- File: `src/agents/skills/workspace.ts`
- Line: 1173
- Evidence: The commit makes `loadSkillEntries` accept candidates whose realpath is inside `skills.load.allowSymlinkTargets` (`src/agents/skills/workspace.ts:308`). Non-`rw` sandbox setup then syncs loaded skills into the sandbox workspace (`src/agents/sandbox/context.ts:54`), but `syncSkillsToWorkspace` copies `entry.skill.baseDir` with `fsp.cp(..., { recursive: true, force: true })` and no dereference option (`src/agents/skills/workspace.ts:1173`). A focused probe showed the synced skill directory remains a symlink, the sandbox workspace loader still advertises the skill, and `createSandboxFsBridge().readFile("/workspace/skills/manager/SKILL.md")` fails with `Symlink escapes sandbox mount root`.
- Impact: Agents running with sandbox `workspaceAccess: "none"` or `"ro"` can see the allowed symlinked skill in the prompt but cannot read its `SKILL.md`, so the new feature silently breaks for the sandboxed path that intentionally reloads skills from the copied workspace.
- Suggested fix: When syncing a loaded skill whose source directory is a symlink admitted by `allowSymlinkTargets`, materialize the resolved skill directory into `targetWorkspace/skills` instead of preserving the symlink. Keep the existing `.git`/`node_modules` filters and continue rejecting nested symlink escapes.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Create a source workspace skill symlink to an allowed external root, run `syncSkillsToWorkspace` with `skills.load.allowSymlinkTargets`, then read the synced skill through `createSandboxFsBridge` with `workspaceAccess: "none"`. The synced entry is a symlink and the read fails at the sandbox mount boundary.
- Is this the best way to solve the issue? yes. The prompt path is inside the sandbox workspace, so the synced filesystem content needs to be inside that workspace too; mounting or prompting the original external target would broaden sandbox access and does not match the existing sync model.
