# Commit 579334f

## Summary

Found one regression in the new preservation logic: it also preserves `params` when the user explicitly replaces the containing config object/map, so `openclaw config set --replace` can no longer fully remove stale model entries or nested provider params.

## Findings

### Medium: Explicit model config replacements keep stale `params` and removed model entries

- Kind: regression
- File: `src/config/io.write-prepare.ts`
- Line: 171
- Evidence: `preserveSourceValueAtPath` only treats the exact preserved path as caller-owned when `getPathValue(nextConfig, params.path) !== undefined`. For a replacement of `agents.defaults.models["openai/gpt-5.4"]` with `{"alias":"New"}`, the caller touched the model entry, but not the nested `params` path, so lines 225-230 re-add the old `params`. For a `--replace` of the whole `agents.defaults.models` map that omits an old model entry with params, lines 216-222 re-add the entire old model entry. This conflicts with the documented config contract that object assignment replaces the target path by default and `--replace` makes the provided value the complete protected map value (`docs/cli/config.md:97`, `docs/cli/config.md:108`), and with the CLI write path that applies replacement via `setAtPath` (`src/cli/config-cli.ts:1223`).
- Impact: Users cannot reliably remove stale transport/runtime overrides or stale model catalog entries with documented replacement commands. I confirmed `openclaw config set agents.defaults.models '{"anthropic/sonnet-4.6":{"alias":"Keep"}}' --strict-json --replace` leaves the old `openai/gpt-5.4` entry on disk when that old entry had `params`.
- Suggested fix: make the preservation hook aware of the containing object/map being explicitly present in `nextConfig`. For example, preserve `agents.defaults.params` only when `agents.defaults` is absent from a narrowed write, preserve model-entry `params` only when that exact model entry is absent from `nextConfig`, and never resurrect an omitted model when `agents.defaults.models` is explicitly present. Alternatively pass explicit write-intent paths into `resolvePersistCandidateForWrite`.
- Confidence: high
