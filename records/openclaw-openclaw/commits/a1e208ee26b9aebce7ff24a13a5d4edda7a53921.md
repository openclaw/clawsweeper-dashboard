## Summary

Found one compatibility regression in the restart drain path. The diagnostic blocked-tool-call recovery looks consistent and the focused tests pass.

## Findings

### Medium: `deferralTimeoutMs: 0` no longer gives an indefinite restart drain for embedded runs

- Kind: regression
- File: `src/cli/gateway-cli/run-loop.ts`
- Line: 353
- Evidence: `resolveGatewayRestartDeferralTimeoutMs()` uses `undefined` for the documented `gateway.reload.deferralTimeoutMs: 0` indefinite wait. The new `resolveActiveRunDrainWaitMs()` remaps that `undefined` to `RESTART_ACTIVE_EMBEDDED_RUN_ABORT_GRACE_MS` when any embedded run is active, then `waitForActiveEmbeddedRuns(30_000)` times out and `abortEmbeddedPiRun(undefined, { mode: "all" })` runs. Current docs still say `deferralTimeoutMs: 0` waits indefinitely for in-flight operations, and the hot-reload path still has a test proving that contract.
- Impact: operators who set `deferralTimeoutMs: 0` to avoid interrupting long active turns can now have active embedded runs aborted after 30 seconds during gateway restart, inconsistent with docs and channel hot reload behavior.
- Suggested fix: preserve `undefined` for active embedded run drain when the configured restart drain is indefinite, or introduce/document a separate explicit stale-run grace override and cover the `deferralTimeoutMs: 0` restart case in `run-loop.test.ts`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. Configure `gateway.reload.deferralTimeoutMs: 0`, have `getActiveEmbeddedRunCount()` return `1`, trigger a non-forced restart, and observe that current `runGatewayLoop` calls `waitForActiveEmbeddedRuns(30_000)` instead of `waitForActiveEmbeddedRuns(undefined)`, then aborts active runs on timeout.
- Is this the best way to solve the issue? Yes, if the documented indefinite-drain contract is still intended. If the new 30s cap is intentional even for `0`, update docs and tests so the contract is explicit.
