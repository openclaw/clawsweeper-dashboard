# Commit cd019cfa

## Summary

Found one low-severity build reliability regression: the new suppression is intended for bundled `zod/v4/locales/*.d.cts` warnings, but it suppresses the same `rolldown-plugin-dts:fake-js` CommonJS-dts warning from any path.

## Findings

### Low: CommonJS-dts warning suppression is broader than the zod locale case

- Kind: reliability
- File: `tsdown.config.ts`
- Line: 92
- Evidence: `isSuppressedLog` checks only `PLUGIN_WARNING`, `rolldown-plugin-dts:fake-js`, and message text containing `uses CommonJS dts syntax`; it does not inspect the warning path. Upstream `rolldown-plugin-dts@0.25.1` emits this warning for any declaration input containing `TSExportAssignment` or external `TSImportEqualsDeclaration`, not just zod locales. A focused one-off check with `/abs/path/node_modules/example-lib/index.d.cts uses CommonJS dts syntax...` returned `handled=0`, proving current main hides a non-zod same-class warning.
- Impact: normal build and CI output can now hide future actionable declaration-bundling warnings from non-zod dependencies or sources, even though the commit message and changelog scope the suppression to noisy zod locale declarations.
- Suggested fix: normalize the log message/path and require it to match `zod/v4/locales/*.d.cts` before suppressing. Add a regression test proving the same CommonJS-dts warning from a non-zod path is forwarded.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: import `tsdown.config.ts`, get the unified graph `onLog`, pass a `rolldown-plugin-dts:fake-js` CommonJS-dts warning for `example-lib/index.d.cts`, and observe the default handler is not called.
- Is this the best way to solve the issue? yes: keeping the zod warning filter is reasonable, but path-scoping preserves the intended build-log signal for unrelated declaration bundling problems.
