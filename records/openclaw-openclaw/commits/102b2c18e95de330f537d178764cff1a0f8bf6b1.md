# Commit 102b2c18

## Summary

Found one low-severity progress regression in the new verify-stage counter. The commit fixes the `[4/3]` path when verification is actually reached, but it also counts verification on first-install onboarding paths that return or `exec` before `verify_installation`.

## Findings

### Low: `--verify` can now leave first-install progress stuck at `[3/4]`

- Kind: bug
- File: `scripts/install.sh`
- Line: 342
- Evidence: `configure_install_stage_total` unconditionally sets `INSTALL_STAGE_TOTAL=4` whenever `VERIFY_INSTALL=1`, but fresh install onboarding paths return or `exec "$claw" onboard` before the later `verify_installation` call at `scripts/install.sh:3124`. In a stubbed no-TTY first-install run, this commit prints `[1/4]`, `[2/4]`, `[3/4] Finalizing setup`, then returns with “No TTY; run /tmp/openclaw onboard to finish setup”; the base commit printed `[3/3]`.
- Impact: Users running `--verify` during a fresh install that still needs onboarding can see an incomplete progress counter and never see the counted verify stage. This is especially visible in noninteractive installs that exit successfully before verification.
- Suggested fix: either run `verify_installation` before onboarding return/exec paths when `--verify` is requested, or only include the verify stage in `INSTALL_STAGE_TOTAL` after the flow knows verification will be reached.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes; I sourced `scripts/install.sh` with `OPENCLAW_INSTALL_SH_NO_RUN=1`, parsed `--verify`, stubbed install side effects, forced `is_promptable` false, and ran `main`. Current commit ended at `[3/4]`; base ended at `[3/3]`.
- Is this the best way to solve the issue? unclear; moving verification before onboarding honors the `--verify` contract more directly, while dynamic counting is safer if onboarding must remain terminal before verification.
