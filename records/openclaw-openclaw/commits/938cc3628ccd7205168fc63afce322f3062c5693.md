# Commit 938cc36

## Summary

Found one low-severity CLI guidance regression: a new remediation suggests a `channels status --channel` selector that the command does not implement.

## Findings

### Low: Auth error points users to a non-existent channel status selector

- Kind: bug
- File: `src/cli/channel-auth.ts`
- Line: 121
- Evidence: The new unsupported-auth-mode error says to run `openclaw channels status --channel <id>` “for its supported actions.” `src/cli/channels-cli.ts:129` registers `channels status` with only `--probe`, `--timeout`, and `--json`, and `src/commands/channels/status.ts:26` has no `channel` option or filtering path. `pnpm openclaw channels status --help` also lists no `--channel`.
- Impact: Users who hit `openclaw channels login --channel <channel-without-login>` or the matching logout path get directed to a command form that is not a real selector and does not report supported actions for that channel.
- Suggested fix: Replace this hint with an existing valid command, such as `openclaw channels list` or `openclaw channels capabilities --channel <id>`, or add a real `--channel` filter to `channels status` with tests.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Trigger the `!plugin || !supportsChannelAuthMode(plugin, mode)` path, for example with a configured channel plugin that does not support `login`; then compare the suggested `channels status --channel <id>` against `openclaw channels status --help`.
- Is this the best way to solve the issue? no. The current hint names a command shape not present in the CLI; the safer fix is to point at an existing status/list/capabilities surface or implement the advertised filter.
