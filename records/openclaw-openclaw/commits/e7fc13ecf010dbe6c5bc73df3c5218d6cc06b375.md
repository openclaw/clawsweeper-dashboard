# Commit e7fc13e

## Summary

Found one regression in the merged-PR baseline calculation. The workflow now handles rebase-merged PRs, but it produces the wrong “before” ref for squash-merged or normal merge-commit PRs with more than one source commit. The affected code is still present on current `main`.

## Findings

### Medium: Merged squash PRs compare against an unrelated older baseline

- Kind: regression
- File: `.github/workflows/mantis-telegram-desktop-proof.yml`
- Line: 180
- Evidence: For merged PR comments, the workflow sets `mergedBaseline` to `${mergedCandidate}~${commits.length}`. That is only correct when the landed commits are a first-parent chain matching the PR commit count. A real recent merged PR shows the failure: `gh pr view 80456` reports 7 PR commits and merge commit `c3af812fe3de9606d50f35444f31c3c096c49351`; the landed commit is a single squash-style commit whose real pre-PR parent is `8ffb7566141281c34d9583724183f931a2c39dec`, while the workflow would use `c3af812fe3de9606d50f35444f31c3c096c49351~7`, which resolves to `b241458f15468f3e2d3b3ac9e6be223dbcaba530`.
- Impact: A maintainer triggering `@Mantis telegram desktop proof` after a multi-commit squash merge gets baseline evidence from several unrelated main commits earlier, not the state immediately before the PR. That can invalidate before/after Telegram proof, hide the PR’s actual effect, or attribute unrelated behavior changes to the PR. The same ancestry math is also wrong for normal two-parent merge commits with more than one PR commit, and `per_page: 100` makes the count unreliable for very large rebased PRs.
- Suggested fix: Do not derive the landed baseline from source PR commit count alone. Detect the landed merge shape from the candidate commit: use the first parent for squash commits and merge commits, and only use `~<count>` when the landed first-parent chain is proven to correspond to a rebase merge. If the strategy cannot be proven, require an explicit `baseline=` rather than emitting misleading proof. Paginate `pulls.listCommits` if commit count remains part of the rebase path.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. For PR `80456`, the reviewed logic would emit `baseline_ref=c3af812fe3de9606d50f35444f31c3c096c49351~7`, and local Git resolves that to `b241458f15468f3e2d3b3ac9e6be223dbcaba530` instead of the landed commit’s parent `8ffb7566141281c34d9583724183f931a2c39dec`.
- Is this the best way to solve the issue? yes, directionally. The fix should use the landed commit graph, not just the PR source commit count, because GitHub merge strategies land multi-commit PRs differently.
