# Commit 99176e19

## Summary

Found one regression in the shared command payload extraction. Some valid carrier forms now fail to expose the final executable payload, so the exec guard can miss forbidden control commands such as `openclaw channels login` or `/approve` when wrapped.

## Findings

### Medium: Valid carrier forms can hide forbidden exec control commands

- Kind: regression
- File: `src/infra/command-analysis/risks.ts`
- Line: 282
- Evidence: `rejectUnsafeControlShellCommand` now relies on `buildCommandPayloadCandidates` before checking `/approve` and `openclaw channels login` in `src/agents/bash-tools.exec.ts:1145`. The new helper delegates to `resolveCarrierCommandArgv`; when that resolver returns `null`, it keeps only the wrapper command. Focused checks showed:
  - `["env","-uFOO","openclaw","channels","login"] => ["env -uFOO openclaw channels login"]`
  - `["sudo","-urunner","openclaw","channels","login"] => ["sudo -urunner openclaw channels login"]`
  - `["exec","-aopenclaw","openclaw","channels","login"] => ["exec -aopenclaw openclaw channels login"]`
  - six nested `env` wrappers never produced `openclaw channels login`
- Impact: these are valid command forms that execute the carried command, but the guard no longer sees the carried payload, so interactive channel login and approval-command shell guards can be bypassed by wrapping. The parent implementation kept stripping prefixes until the real command for these cases.
- Suggested fix: make payload extraction preserve the old guard behavior for control-command scanning: recognize attached short option values such as `env -uFOO`, `sudo -urunner`, and `exec -aopenclaw`, and avoid using the inline-eval carrier depth cap in a way that hides the final payload. Add regression tests for those examples and deeply nested carriers.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: call `buildCommandPayloadCandidates` with the examples above; none returns the forbidden `openclaw channels login` payload. Live shell checks confirmed `env -uFOO /usr/bin/printf ok`, `sudo -u$(id -un) /usr/bin/printf ok`, and `bash -lc 'exec -aopenclaw /usr/bin/printf ok'` execute successfully.
- Is this the best way to solve the issue? yes: the shared helper should expose executable payload candidates for valid carrier syntax before the exec guard applies control-command checks.
