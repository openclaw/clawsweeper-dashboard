# Commit d419bcf

## Summary

Found one low-severity display-label edge case in the new Unicode slug normalizer.

## Findings

### Low: Emoji variation selectors can survive after their base emoji is stripped

- Kind: bug
- File: `extensions/discord/src/monitor/allow-list.ts`
- Line: 102
- Evidence: `normalizeDiscordDisplaySlug()` keeps `\p{M}` marks but replaces the non-letter/non-number base symbol with `-`. For emoji sequences that include U+FE0F, the base emoji is removed while the variation selector remains. Focused check: `normalizeDiscordDisplaySlug("❤️-ops")` returns `"️-ops"` with code points `fe0f,2d,6f,70,73`.
- Impact: inbound Discord context builds `GroupChannel` / `GroupSubject` from this value in `extensions/discord/src/monitor/message-handler.context.ts:97`, and component events do the same in `extensions/discord/src/monitor/agent-components.dispatch.ts:131`. Emoji-prefixed channel labels can therefore appear to agents as malformed labels like `#️-ops` instead of a clean fallback such as `#ops` or a preserved emoji label.
- Suggested fix: either preserve full display-safe emoji/symbol grapheme clusters, or strip variation selectors/leading combining marks that are left behind after removing unsupported base characters. Add a regression test for an emoji-prefixed channel name such as `❤️-ops`.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes: `pnpm exec tsx -e "import { normalizeDiscordDisplaySlug } from './extensions/discord/src/monitor/allow-list.ts'; const value = normalizeDiscordDisplaySlug('❤️-ops'); console.log(JSON.stringify(value)); console.log([...value].map((ch) => ch.codePointAt(0)?.toString(16)).join(','));"` prints `"️-ops"` and `fe0f,2d,6f,70,73`.
- Is this the best way to solve the issue? unclear: the safer fix depends on whether display labels should preserve emoji or keep slug-like text only. Either way, dangling marks should not be emitted.
