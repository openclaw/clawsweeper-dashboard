## Findings

### Low: Valid unmatched Telegram access-group entries are logged as invalid allowFrom entries

- Kind: bug
- File: `extensions/telegram/src/access-groups.ts`
- Line: 33
- Evidence: The new unmatched path returns the original `accessGroup:operators` token so group allowlist policy can fail closed. That token is later normalized by `normalizeAllowFrom`, which treats every non-wildcard, non-numeric entry as invalid and logs `Invalid allowFrom entry: "accessGroup:operators"` even though `accessGroup:<name>` is documented as a valid allowlist reference. A focused runtime probe reproduced the warning with `expanded:["accessGroup:operators"]` and `invalidEntries:["accessGroup:operators"]`.
- Impact: Operators using valid Telegram access groups can get misleading runtime warnings telling them their config is invalid and suggesting unrelated Telegram group-ID fixes. The access decision still fails closed, so this is not an authorization bypass.
- Suggested fix: Preserve unmatched access-group references as configured allowlist entries for policy purposes, but exclude parsed `accessGroup:<name>` tokens from Telegram numeric-ID invalid-entry warnings, or carry separate “configured access group” metadata instead of feeding the raw token to invalid-ID validation.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes; call `expandTelegramAllowFromWithAccessGroups` with `allowFrom:["accessGroup:operators"]`, a config where `operators.members.telegram` does not include the sender, then pass the returned list to `normalizeAllowFrom`.
- Is this the best way to solve the issue? unclear; the behavioral fix is right, but the warning path needs a small follow-up so valid access-group syntax is not reported as invalid Telegram sender syntax.
