# Commit e71d7d48

## Summary

Found one concrete reliability regression in the new stdin-backed `ffprobe` path. The Telegram behavior change is otherwise well-scoped, covered by focused tests, and the SDK baseline/format checks passed.

## Findings

### Medium: ffprobe stdin EPIPE can escape the helper and crash the process

- Kind: reliability
- File: `src/media/ffmpeg-exec.ts`
- Line: 59
- Evidence: The new `runFfprobe(..., { input })` path starts `execFile(...)`, then calls `proc.stdin?.end(options.input)` without attaching an `error` handler to the child stdin stream. If the child process exits before consuming the full buffer, Node emits an `EPIPE` error on `proc.stdin`; with no listener, that is an uncaught exception rather than a rejected `runFfprobe` promise. I reproduced the same write pattern locally against `/bin/true` with a large buffer and Node exited through `uncaughtException: EPIPE`.
- Impact: Telegram video replies now run this path on user-supplied video buffers from remote or local media. A malformed or mislabeled `.mp4`/`video/*` payload can make `ffprobe` exit early while OpenClaw is still writing the buffer, turning what should be a graceful `probeVideoDimensions()` fallback into a gateway process crash on systems with `ffprobe` installed.
- Suggested fix: Attach a stdin `error` listener before writing, and settle the promise through that path without throwing globally. Prefer preserving the child callback error when available, but at minimum handle/ignore `EPIPE` so `probeVideoDimensions()` can return `undefined` as intended.
- Confidence: high
