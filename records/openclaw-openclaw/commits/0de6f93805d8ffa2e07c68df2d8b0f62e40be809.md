# Commit 0de6f938

## Summary

Found one concrete reliability risk in the new probe fallback path. The patch fixes the first repeated timeout, but the cached transport’s built-in primary recovery probes can still recreate the same `getMe` timeout on persistent broken-IPv6 hosts.

## Findings

### Medium: Sticky fallback recovery can still reintroduce periodic getMe timeouts

- Kind: reliability
- File: `extensions/telegram/src/probe.ts`
- Line: 195
- Evidence: `probeTelegram` now calls `transport.forceFallback?.("probe timeout/network error")` after any `getMe` fetch error. That handles the first `fetchWithTimeout` `TimeoutError`, but the cached transport still schedules a primary dispatcher recovery probe after five sticky fallback successes in `extensions/telegram/src/fetch.ts:730`. When that recovery probe runs, `fetch.ts:746` starts again at attempt 0. The same `TimeoutError` shape is not classified as retryable by `fetch.ts:461`, so it escapes back to the probe catch and logs another timeout. If the sticky attempt is already the IPv4 fallback, the unconditional `forceFallback` also advances to the pinned-IP fallback via `fetch.ts:842` instead of just keeping the existing IPv4 fallback.
- Impact: On the persistent no-IPv6-egress network this commit targets, health checks can still produce recurring `getMe` timeout logs after enough successful fallback probes. The cadence is lower than the pre-fix 60-second storm, but it does not fully satisfy the “stop logging repeated probe timeouts” behavior and can still add a 10-15s delayed health refresh periodically.
- Suggested fix: Make forced probe fallback idempotent when the transport is already sticky, or expose a transport method that marks/keeps fallback without advancing to the next attempt after a failed primary recovery probe. A focused regression test should simulate first timeout, five successful fallback `getMe` calls, then a primary recovery `TimeoutError`.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes at source/test level: mock `probeTelegram` with a transport that succeeds after the first forced fallback, then reaches the transport’s five-success primary recovery threshold and throws the same `TimeoutError` again. I did not run a live broken-IPv6 Telegram host.
- Is this the best way to solve the issue? Unclear. The intended primary recovery probing is useful, so the safer fix is not to remove recovery entirely, but to avoid treating a failed recovery probe as a reason to advance deeper or to produce frequent long timeout probes.
