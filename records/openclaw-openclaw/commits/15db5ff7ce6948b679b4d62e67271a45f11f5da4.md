# Commit 15db5ff

## Summary

Found one regression in the external-CLI auth path. The commit prevents reply startup from discovering external profiles, but the stripped store is also reused for actual auth resolution, so scoped external-CLI profiles selected by probe/status paths can no longer resolve during the embedded run.

## Findings

### Medium: External-CLI probe profiles are dropped before runtime auth resolution

- Kind: regression
- File: `src/agents/pi-embedded-runner/run.ts`
- Line: 516
- Evidence: `runEmbeddedPiAgent` now builds the non-plugin `authStore` with `ensureAuthProfileStoreWithoutExternalProfiles(...)`. `models status --probe` still builds targets from `ensureAuthProfileStore(... externalCliDiscoveryScoped(...))` and passes `target.profileId` into `runEmbeddedPiAgent` as a user-selected profile at `src/commands/models/list.probe.ts:260` and `src/commands/models/list.probe.ts:486`. In the run path, a user-selected profile that only exists as an external overlay is absent from the stripped store, so `lockedProfileId` is cleared at `src/agents/pi-embedded-runner/run.ts:575`, profile ordering is computed from the stripped store at `src/agents/pi-embedded-runner/run.ts:609`, and `getApiKeyForModel` is then called with that same stripped store at `src/agents/pi-embedded-runner/run/auth-controller.ts:349`. Because a store is supplied, `resolveApiKeyForProvider` cannot fall back to its own scoped external-CLI discovery.
- Impact: file-backed external CLI credentials, such as Claude CLI or MiniMax portal profiles that are intentionally runtime-only, can be listed as probe targets but then fail the actual probe or embedded run as missing auth. This conflicts with the documented contract that scoped external CLI credentials are discovered when the provider/runtime/auth profile is in scope.
- Suggested fix: keep the lightweight store for early reply-prep decisions, but do not pass it as the final authority for actual credential resolution when no persisted candidate was found. For example, let `getApiKeyForModel` resolve without `store` for the `[undefined]` fallback candidate, or overlay scoped external profiles only inside the auth-controller resolution step with `allowKeychainPrompt: false`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes, with a file-backed external CLI profile that is available through `externalCliDiscoveryScoped` but not persisted in `auth-profiles.json`; `buildProbeTargets` will emit it, then `probeTarget`/`runEmbeddedPiAgent` will resolve auth from the stripped store and fail.
- Is this the best way to solve the issue? Unclear. The intent to avoid external discovery during reply prep is reasonable; the safer direction is to defer scoped discovery until actual auth resolution rather than removing it from the runtime credential path entirely.
