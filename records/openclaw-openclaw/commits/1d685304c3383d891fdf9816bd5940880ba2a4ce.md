# Commit 1d68530

## Summary

Found one CI regression in the draft-PR skip logic. The new job-level guard skips draft PR runs, but the workflows do not opt into `ready_for_review`, so a PR opened as draft can keep skipped-success Testbox checks and never rerun them when marked ready.

## Findings

### Medium: Draft workflow PRs do not rerun Testbox checks when marked ready

- Kind: regression
- File: `.github/workflows/ci-check-testbox.yml` and `.github/workflows/ci-build-artifacts-testbox.yml`
- Line: 9 and 10
- Evidence: Both workflows define `pull_request` with only `paths`, then the commit adds `if: ${{ github.event_name != 'pull_request' || !github.event.pull_request.draft }}` at job level. GitHub documents that `pull_request` defaults to only `opened`, `synchronize`, and `reopened`; `ready_for_review` must be added via `types`. GitHub also documents skipped jobs report success and do not block merging, even as required checks. ([docs.github.com](https://docs.github.com/en/actions/writing-workflows/choosing-when-your-workflow-runs/events-that-trigger-workflows))
- Impact: A workflow-changing PR opened as draft runs these workflows once, the jobs skip as success, and marking the PR ready does not trigger these Testbox jobs again unless another commit is pushed. That weakens the intended workflow-change validation path for exactly the draft PR case this commit changes.
- Suggested fix: Add explicit PR activity types to both workflows, matching the main CI pattern, for example `types: [opened, reopened, synchronize, ready_for_review, converted_to_draft]` under `pull_request` while keeping the existing `paths` filter.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: open a draft PR that changes `.github/workflows/**`; the jobs are skipped by the new condition. Then mark the PR ready without pushing another commit; these workflows will not rerun because `ready_for_review` is not in their trigger activity set.
- Is this the best way to solve the issue? no: the draft skip is reasonable, but it needs trigger types that include `ready_for_review` so skipped draft checks are replaced by real checks when the PR becomes mergeable.
