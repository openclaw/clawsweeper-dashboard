# Commit 02b1075a

## Summary

Found one low-severity regression: the deferred PDF factory can now expose an unusable `pdf` tool for audio-only media-understanding provider setups.

## Findings

### Low: Audio-only media providers can register a PDF tool that cannot resolve a PDF model

- Kind: regression
- File: `src/agents/openclaw-tools.ts`
- Line: 417
- Evidence: `resolveOptionalMediaToolFactoryPlan` enables `pdf` for any available `mediaUnderstandingProviders` contract at `src/agents/openclaw-tools.ts:223`, including audio-only plugins such as Deepgram, whose manifest declares `mediaUnderstandingProviders: ["deepgram"]` but only `capabilities: ["audio"]` in `extensions/deepgram/openclaw.plugin.json:10`. The commit then passes `deferAutoModelResolution: true` into `createPdfTool` at `src/agents/openclaw-tools.ts:417`; with deferral enabled, `createPdfTool` returns a tool without checking `resolvePdfModelConfigForTool` (`src/agents/tools/pdf-tool.ts:276`). Execution later rejects with `No PDF model configured.` (`src/agents/tools/pdf-tool.ts:346`). A focused probe with only a Deepgram auth profile showed `{"eager":false,"deferred":true}` and then `{"execute":"rejected","message":"No PDF model configured."}`.
- Impact: users with only audio media providers configured can now see a `pdf` tool in agent tool prep even though no PDF-capable image/native-PDF model exists. That violates the documented availability contract and can lead the model to call a tool that deterministically fails.
- Suggested fix: keep registration lazy, but make the cheap factory planner PDF-specific. For example, gate `optionalMediaTools.pdf` on explicit `pdfModel`/`imageModel`, configured image-model auth, or media-understanding metadata that declares image capability or native PDF input, not any media-understanding provider.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. With model-provider env vars unset and only a Deepgram auth profile, `createPdfTool({ deferAutoModelResolution: false })` returns null while `createPdfTool({ deferAutoModelResolution: true })` returns `pdf`; executing it rejects before file loading with `No PDF model configured.`
- Is this the best way to solve the issue? yes. The regression is in availability gating, not PDF execution. Restoring a PDF-capable availability predicate preserves the performance fix without reintroducing eager auth/model probing on every turn.
