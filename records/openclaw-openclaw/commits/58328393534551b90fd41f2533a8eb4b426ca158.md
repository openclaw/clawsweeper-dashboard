# Commit 5832839353

## Summary

Found one release-CI reliability regression: the new Open WebUI `models` smoke mode is planned, but it is not forwarded into the container process that runs the probe, so the release lane still runs the full chat completion path.

## Findings

### Medium: Release Open WebUI lane drops `OPENWEBUI_SMOKE_MODE`

- Kind: reliability
- File: `scripts/e2e/openwebui-docker.sh`
- Line: 146
- Evidence: `scripts/lib/docker-e2e-scenarios.mjs:10` sets the release command to `OPENWEBUI_SMOKE_MODE=models ... pnpm test:docker:openwebui`, and `scripts/e2e/openwebui-probe.mjs:11` only enters model-only mode when `OPENWEBUI_SMOKE_MODE` or `OPENCLAW_OPENWEBUI_SMOKE_MODE` is present. But `scripts/e2e/openwebui-docker.sh:146` runs the probe via `docker_e2e_docker_cmd exec` with an explicit env list and does not include either smoke-mode variable. That means the probe defaults to `"chat"` and still calls `/api/chat/completions`.
- Impact: The release-path Open WebUI smoke remains dependent on a live model completion even though this commit intends to stop after sign-in and model discovery. That keeps the lane nondeterministic and timeout-prone.
- Suggested fix: Resolve the smoke mode in `openwebui-docker.sh` and pass it to the probe exec, for example `-e "OPENWEBUI_SMOKE_MODE=${OPENWEBUI_SMOKE_MODE:-${OPENCLAW_OPENWEBUI_SMOKE_MODE:-chat}}"`, with a focused test/assertion covering wrapper propagation.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Static path: release planner emits `OPENWEBUI_SMOKE_MODE=models`, wrapper `docker exec` omits it, probe defaults to `chat`; a full Docker run with a live OpenAI key would still reach the chat-completions block.
- Is this the best way to solve the issue? yes. The contract boundary is the shell wrapper launching the probe, so forwarding the env there is the narrow fix.
