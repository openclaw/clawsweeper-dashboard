# Commit bca6709

## Summary

Found one medium-confidence regression in the new Codex session repair path: it can repin valid native Codex sessions to PI when Codex is temporarily not considered ready, even when the session contains no legacy `openai-codex/*` route state.

## Findings

### Medium: Doctor can permanently repin valid Codex sessions to PI

- Kind: regression
- File: `src/commands/doctor/shared/codex-route-warnings.ts`
- Line: 749
- Evidence: `scanCodexSessionStoreRoutes` treats any `entry.agentHarnessId === "codex"` as stale whenever `resolveCodexRepairRuntime` returns `pi`, regardless of whether the session has legacy `openai-codex/*` provider/model state. `repairCodexSessionStoreRoutes` then writes both `entry.agentHarnessId = params.runtime` and `entry.agentRuntimeOverride = params.runtime`. The added test at `src/commands/doctor/shared/codex-route-warnings.test.ts:263` explicitly covers a session with only `agentHarnessId: "codex"` and asserts it is changed to PI.
- Impact: `openclaw doctor --fix` can run while Codex is temporarily not ready, for example due to a missing/disabled plugin record or unusable OAuth profile, and permanently force an existing native Codex conversation to PI via `agentRuntimeOverride: "pi"`. Later turns read `agentRuntimeOverride` from the session and pass it into harness selection, so this can override a user's configured Codex runtime and replay a Codex-pinned transcript through PI.
- Suggested fix: only repin `agentHarnessId`/`agentRuntimeOverride` when the session also has legacy `openai-codex` route evidence, or delegate pinned-runtime cleanup to the existing session-state owner repair path that checks whether the current configured route still allows the owner. At minimum, do not synthesize `agentRuntimeOverride: "pi"` for a session whose only matched field is `agentHarnessId: "codex"`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Call `repairCodexSessionStoreRoutes({ store: { main: { sessionId: "s1", updatedAt: 1, agentHarnessId: "codex" } }, runtime: "pi" })`; current code changes that session to `agentHarnessId: "pi"` and `agentRuntimeOverride: "pi"`.
- Is this the best way to solve the issue? no. The new repair path conflates “legacy `openai-codex/*` route state” with any Codex harness pin. The safer repair is to require actual legacy provider/model evidence before converting to PI, and leave generic stale harness cleanup to the existing configured-route-aware doctor repair.
