# Commit e9e7c432

## Summary

Found one regression in the bare plugin install path: current main still depends on a ClawHub `/readiness` endpoint that is returning 404 for public plugin packages, so bare ClawHub-only package installs fall through to npm and can fail even though the package exists and has installable ClawHub artifact metadata.

## Findings

### Medium: Bare ClawHub-only plugin installs fall through to npm while readiness is not live

- Kind: regression
- File: `src/cli/plugins-command-helpers.ts`
- Line: 248
- Evidence: `resolvePreferredClawHubSpec()` calls `fetchClawHubPackageReadiness({ name })` and catches every failure by returning `null`. Both CLI and chat install paths only try ClawHub when this helper returns a spec; otherwise `plugins install <bare>` proceeds to npm fallback at `src/cli/plugins-install-command.ts:807`, and `/plugins install <bare>` does the same in `src/auto-reply/reply/commands-plugins.ts:259`.
- Live evidence: ClawHub search currently returns `stalwart-jmap` as a `code-plugin`, and its package detail plus `/versions/0.3.1/artifact` endpoints return package/version metadata, but `https://clawhub.ai/api/v1/packages/stalwart-jmap/readiness` returns 404. `npm view stalwart-jmap version --json` also returns npm 404. That means `openclaw plugins install stalwart-jmap` has no chance to use the existing ClawHub package through the bare shorthand.
- Impact: Users can discover an installable ClawHub plugin, then bare install fails or installs a stale/different npm package instead of the ClawHub artifact. This breaks the documented bare install shorthand for ClawHub-first plugin installs.
- Suggested fix: Either deploy/support the readiness endpoint before gating bare installs on it, or treat readiness-route 404 as “readiness API unavailable” and fall back to the previous ClawHub install attempt for package names that exist on ClawHub. Only fall through to npm after an explicit not-ready response or package/version not-found decision.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Public ClawHub search/detail/artifact endpoints prove `stalwart-jmap` exists as a ClawHub code-plugin, the readiness endpoint returns 404, and npm has no `stalwart-jmap` package.
- Is this the best way to solve the issue? unclear. The safest fix depends on rollout intent: either ship the readiness service endpoint first, or make the client preserve the old ClawHub install attempt when only the readiness route is missing.
