# Commit 46d4238

## Summary

Found one low-severity onboarding edge case: external search provider selection can still be saved without installing the selected plugin when the user leaves the credential prompt blank.

## Findings

### Low: Blank external search key skips plugin install while saving provider config

- Kind: bug
- File: `src/flows/search-setup.ts`
- Line: 582
- Evidence: The new install path only runs through `finalizeSearchProviderSetup`, which calls `ensureOnboardingPluginInstalled`, in the key/ref/env/keyless branches. If the selected provider needs a credential and the user leaves the prompt blank with no existing key or env var, the branch at `src/flows/search-setup.ts:573` notes that no key is stored, then returns a config containing `tools.web.search.provider = choice` at `src/flows/search-setup.ts:582` without calling `finalizeSearchProviderSetup`.
- Impact: In packaged builds where Brave is external, a user can select Brave during onboarding, skip entering the key, and end with `tools.web.search.provider: "brave"` but no installed `brave` plugin. Adding `BRAVE_API_KEY` later still leaves `web_search` unavailable until a separate doctor repair/install step runs.
- Suggested fix: Route the no-key branch through `finalizeSearchProviderSetup`, likely with `applySearchProviderSelection(config, choice)`, so selected install-catalog providers are installed before any provider config is saved even when credentials are deferred.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: run the search setup flow, choose the catalog-only Brave provider, leave the API key prompt blank, and inspect the returned config. The code path returns before the install helper is invoked.
- Is this the best way to solve the issue? yes: the commit’s stated behavior is to install external web-search plugins before saving provider config, and the existing `finalizeSearchProviderSetup` helper already centralizes that installation behavior.
