# Commit d763b83

## Summary

One low-severity diagnostic regression remains on current `main`: the new missing-command message assumes every non-`enabledByDefault` command alias is a bundled disabled-by-default plugin, but manifest alias metadata can also come from external/workspace/global plugins.

## Findings

### Low: Disabled-alias diagnostic is not origin-aware

- Kind: bug
- File: `src/cli/run-main-policy.ts`
- Line: 145
- Evidence: `resolveMissingPluginCommandMessage` emits the â€śbundled plugin is disabled by defaultâ€ť repair whenever an alias is not `runtime-slash`, lacks `enabledByDefault: true`, and `plugins.entries.<id>.enabled` is not true. But `PluginManifestCommandAliasRecord` only carries `pluginId` and `enabledByDefault`, not plugin `origin`, while activation policy treats non-bundled plugins as enabled by default and even marks non-bundled allowlist entries as explicit selections (`src/plugins/config-activation-shared.ts:121`, `src/plugins/config-activation-shared.ts:292`).
- Impact: An external/workspace/global plugin that declares a CLI-like `commandAliases` entry can receive incorrect guidance saying it is a bundled plugin disabled by default. The suggested `openclaw plugins enable <plugin>` can distract from the real cause, such as a missing or failed CLI registration in an already allowed external plugin.
- Suggested fix: Carry plugin `origin` or effective activation state through `resolveManifestCommandAliasOwnerInRegistry`, and only emit the disabled-by-default bundled-plugin hint for bundled aliases whose effective state is bundled-disabled-by-default.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused call with an allowed alias owner still prints the bundled-disabled message: `resolveMissingPluginCommandMessage("demo", { plugins: { allow: ["demo-plugin"] } }, { registry: { plugins: [{ id: "demo-plugin", commandAliases: [{ name: "demo" }] }] } })`.
- Is this the best way to solve the issue? yes. The message needs metadata that distinguishes bundled default-disablement from non-bundled plugin activation; checking only `enabledByDefault` collapses different plugin contracts.
