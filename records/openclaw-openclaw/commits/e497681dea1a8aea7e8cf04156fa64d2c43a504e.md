# Commit e497681

## Summary

Found one timeout-ordering bug in the Telegram fix. The commit raises OpenClaw's outbound guard to 60s, but it also floors grammY's client timeout to the same 60s, and grammY starts its own timer before it calls OpenClaw's wrapped fetch. At the timeout boundary, grammY can still win the race and bypass the wrapper's authoritative timeout/fallback path.

## Findings

### Low: grammY can still preempt the OpenClaw outbound timeout guard

- Kind: reliability
- File: `extensions/telegram/src/bot-core.ts`
- Line: 333
- Evidence: `resolveTelegramOutboundClientTimeoutFloorSeconds()` returns the guarded `sendmessage` timeout as exactly `60`, and `resolveTelegramClientTimeoutSeconds()` passes that same value to grammY as `client.timeoutSeconds`. The wrapped fetch also arms OpenClaw's method timeout at 60,000ms. In grammY 1.42.0, `createTimeout()` is called before `this.fetch(...)`, so the grammY timer is inserted before OpenClaw's fetch-wrapper timer for the same delay. A focused reproduction with a hanging `sendChatAction` showed two 60,000ms timers, grammY's firing first, and `forceFallbackCalls: 0` with `HttpError: Network request for 'sendChatAction' failed!`.
- Impact: low `channels.telegram.timeoutSeconds` values are not guaranteed to keep OpenClaw's outbound guard authoritative. On a hung outbound call, grammY can still produce the failure first, so the new timeout-triggered `sendChatAction` transport fallback does not run; send failures also lose the wrapper's timeout/origin semantics.
- Suggested fix: make grammY's client timeout strictly greater than the OpenClaw method guard, for example floor to `Math.ceil(requestTimeoutMs / 1000) + 1` when the wrapper owns a method guard. Add an integration-style test that goes through actual grammY timer ordering, not only direct `clientFetch(...)` calls or a mocked Bot constructor.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The reproduction uses grammY's real `Bot` with an OpenClaw-shaped wrapped fetch and controlled timers; firing the first same-delay timer returns grammY's `HttpError` and never calls the fallback path.
- Is this the best way to solve the issue? yes. The intended contract is that OpenClaw's guard fires before grammY's client timeout, so the client timeout must be strictly above the guard, not equal to it.
