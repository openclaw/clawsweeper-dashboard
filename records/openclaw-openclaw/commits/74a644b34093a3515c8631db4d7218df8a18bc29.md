# Commit 74a644b

## Summary

Found one low-severity test regression: the new shared absence helper uses `fs.access`, which follows symlinks and can report `ENOENT` for a dangling symlink entry that still exists.

## Findings

### Low: Absence helper can miss dangling symlink entries

- Kind: test_gap
- File: `src/agents/sandbox/fs-bridge-mutation-helper.test.ts`
- Line: 59
- Evidence: `expectPathMissing()` asserts `fs.access(targetPath)` rejects with `ENOENT`. A focused Node probe confirmed that for a dangling symlink, `fs.access` rejects with `ENOENT` while `fs.lstat` and `fs.readlink` both succeed. The helper is used at `src/agents/sandbox/fs-bridge-mutation-helper.test.ts:266` in the test named `remove unlinks the symlink itself`, replacing the prior `fs.readlink(...).rejects.toThrow()` assertion.
- Impact: A future regression that leaves `workspace/link.txt` behind as a dangling symlink would now pass this test, even though the behavior under test is that removal unlinks the symlink entry itself. This is in sandbox filesystem mutation coverage, where symlink handling is security-sensitive.
- Suggested fix: Make the shared helper assert entry absence with `fs.lstat(targetPath)` rejecting with `ENOENT`, or use a separate `expectDirectoryEntryMissing` helper for tests that need path-entry absence rather than target accessibility.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Create a dangling symlink, then compare `fs.access`, `fs.lstat`, and `fs.readlink`: `fs.access` reports `ENOENT`, while `lstat` and `readlink` still see the symlink entry.
- Is this the best way to solve the issue? no. Deduping the assertions is fine, but the helper should use `lstat` so it checks the directory entry itself.
