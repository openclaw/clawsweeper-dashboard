# Commit 9bedcff

## Summary

Found one concrete packaging regression: QQ Bot is now excluded from bundled build entries, but the metadata copy step still stages its package metadata into `dist/extensions/qqbot`, leaving `prepack`’s bundled channel smoke looking for JS files that were intentionally not built.

## Findings

### Medium: `includeInCore: false` skips QQ Bot JS but still stages QQ Bot metadata for prepack smoke

- Kind: regression
- File: `scripts/copy-bundled-plugin-metadata.mjs`
- Line: 334
- Evidence: The commit adds `openclaw.bundle.includeInCore: false` for QQ Bot in `extensions/qqbot/package.json:58` and makes `collectBundledPluginBuildEntries` skip those entries at `scripts/lib/bundled-plugin-build-entries.mjs:118`, so `dist/extensions/qqbot/index.js` and `setup-entry.js` are no longer build outputs. But `copyBundledPluginMetadata` only checks `NON_PACKAGED_BUNDLED_PLUGIN_DIRS` and `shouldBuildBundledCluster` at `scripts/copy-bundled-plugin-metadata.mjs:334` and `scripts/copy-bundled-plugin-metadata.mjs:338`, so it still writes `dist/extensions/qqbot/package.json` with rewritten `openclaw.extensions`/`setupEntry`. `scripts/openclaw-prepack.ts:104` then runs `scripts/test-built-bundled-channel-entry-smoke.mjs`, which scans every `dist/extensions/*/package.json` channel at `scripts/test-built-bundled-channel-entry-smoke.mjs:70` and asserts the declared JS entry exists at `scripts/test-built-bundled-channel-entry-smoke.mjs:138`.
- Impact: `pnpm prepack`/`npm pack` can fail after the build phase because local `dist` contains QQ Bot metadata without its built entrypoints. This blocks root package packing/publishing even though the npm tarball later excludes `dist/extensions/qqbot/**`.
- Suggested fix: Share the new `includeInCore` predicate with `copyBundledPluginMetadata`, or otherwise make metadata staging skip/remove plugins where `openclaw.bundle.includeInCore === false`. Add a focused test that runs metadata copy after the build-entry filter and asserts QQ Bot metadata is not left in `dist/extensions`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A temp fixture using the current QQ Bot package and manifest showed `listBundledPluginBuildEntries` and `listBundledPluginPackArtifacts` exclude QQ Bot, while `copyBundledPluginMetadata` still creates `dist/extensions/qqbot/package.json` and `openclaw.plugin.json`. Running `scripts/test-built-bundled-channel-entry-smoke.mjs --package-root <temp-fixture>` then fails with `qqbot channel entry missing from packed dist`.
- Is this the best way to solve the issue? yes. The root problem is that build-entry inclusion and metadata-staging inclusion now use different policies; sharing the opt-out keeps `dist` internally consistent.
