# Commit 7917632e

## Summary

Found one regression in the new session picker reorder pass: nested subagent descendants can disappear from the picker.

## Findings

### Medium: Nested subagent grandchildren are dropped from session picker options

- Kind: regression
- File: `ui/src/ui/chat/session-controls.ts`
- Line: 748
- Evidence: The new reorder pass skips every option with a visible `parentKey`, then only appends `childrenByParent.get(option.key)` while iterating original root options. If a visible child is itself a parent, it is appended as a child and never visited as an `option`, so its own children are omitted. A focused runtime probe of `resolveSessionOptionGroups()` with `parent -> child -> grandchild` returned only `agent:main:main` and `agent:main:subagent:parent`; `agent:main:subagent:parent:subagent:child` was absent.
- Impact: Nested subagents are supported and documented (`maxSpawnDepth` can allow sub-agents to spawn children), and adjacent UI slash-command tests already use `agent:main:subagent:parent:subagent:child` shapes. Those descendant sessions become unavailable in the Control UI session picker whenever their parent chain is present.
- Suggested fix: Build the reordered list recursively or with a DFS/queue that appends each child and then that child’s descendants, while preserving sibling input order and guarding cycles.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Run a focused probe importing `resolveSessionOptionGroups()` with rows ordered `[grandchild spawnedBy child, child spawnedBy parent, parent]`; current main returns only parent and child.
- Is this the best way to solve the issue? yes. The bug is local to the reorder algorithm; recursively appending descendants preserves the intended sibling-order fix without hiding deeper children.
