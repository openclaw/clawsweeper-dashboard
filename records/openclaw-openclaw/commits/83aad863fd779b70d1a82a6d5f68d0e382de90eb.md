# Commit 83aad86

## Summary

Found security-audit/doctor false negatives in the new exec filesystem drift helper. The added warning does not fully match the runtime tool-policy chain, so some real read-only-looking configs still produce no warning.

## Findings

### Medium: Exec filesystem drift warning misses `profile` + `alsoAllow` configs

- Kind: security
- File: `src/security/exec-filesystem-policy.ts`
- Line: 29
- Evidence: `resolveToolPolicies` applies `resolveToolProfilePolicy(profile)` directly, then adds `pickSandboxToolPolicy(cfg.tools)`. Runtime instead merges `tools.alsoAllow` into the profile stage before filtering tools (`src/agents/pi-tools.ts:432`). A focused check with `tools.profile="messaging"`, `tools.alsoAllow=["read","exec","process"]`, and `tools.deny=["write","edit","apply_patch"]` showed runtime policy has `exec=true`, `write=false`, `edit=false`, `apply_patch=false`, but `collectExecFilesystemPolicyDriftHits` returns `[]`.
- Impact: `openclaw security audit` and doctor can fail to warn on a plausible config that starts from a restricted profile and adds shell access while disabling filesystem tools.
- Suggested fix: mirror runtime policy composition by merging explicit `alsoAllow` into the profile policy, as `createOpenClawTools` does, before checking runtime/filesystem tool availability. Add a regression test for `profile + alsoAllow + deny mutating fs`.
- Confidence: high

### Medium: Non-main sandbox tool policy drift is not checked

- Kind: security
- File: `src/security/exec-filesystem-policy.ts`
- Line: 45
- Evidence: the helper only adds sandbox tool policy when `sandboxMode === "all"`, and `isExecFilesystemConstrained` treats every non-`all` mode as unconstrained. But `sandbox.mode="non-main"` really does sandbox non-main sessions (`src/agents/sandbox/runtime-status.ts:16`), and `host=auto` resolves to sandbox for those sessions. A focused check with `sandbox.mode="non-main"`, `workspaceAccess="rw"`, and `tools.sandbox.tools.allow=["read","exec","process"]` showed sandbox policy blocks `write` while allowing `exec`, yet `collectExecFilesystemPolicyDriftHits` returns `[]`.
- Impact: common non-main sandbox deployments can have the same “filesystem tools disabled but exec can still write” drift in non-main sessions without an audit/doctor warning.
- Suggested fix: model sandboxed and unsandboxed scopes separately for `non-main`, or add a non-main sandbox context that includes `resolveSandboxToolPolicyForAgent` when checking sandboxed sessions. Cover `workspaceAccess="rw"` and `"ro"` cases in tests.
- Confidence: high

## Details

- Is this the best way to solve the issue? yes: the safest fix is to make the diagnostic helper reuse or closely mirror the runtime policy composition instead of maintaining a partial copy.
