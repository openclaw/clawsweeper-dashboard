## Findings

### Low: Voice health marks error states as OK

- Kind: bug
- File: `apps/android/app/src/main/java/ai/openclaw/app/ui/HealthLogsSettingsScreen.kt`
- Line: 78
- Evidence: Health computes `voiceReady = talkStatus.lowercase() != "off"`, then renders the Voice row as success when `voiceReady` is true. `TalkModeManager` uses non-`Off` text for failure states, including `"Gateway not connected"`, `"Microphone permission required"`, `"Speech recognizer unavailable"`, and `"Talk failed: ..."`, so those error states are displayed as healthy.
- Impact: The Health screen can show a green/success Voice row while the voice subsystem is unavailable or failed, which defeats the diagnostic purpose of this screen and can send users toward the wrong troubleshooting path.
- Suggested fix: Replace the string inequality with an explicit health classifier or expose a structured voice status from the runtime. At minimum, treat statuses containing `required`, `unavailable`, `failed`, or `not connected` as warning/danger instead of success.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Start Talk Mode with no microphone permission or while the gateway is disconnected; `TalkModeManager` sets `talkModeStatusText` to an error string, and the Health screen treats any value other than exactly `Off` as healthy.
- Is this the best way to solve the issue? yes. The UI should not infer health from arbitrary display text; a structured status or central classifier keeps future Talk Mode text changes from silently changing diagnostics.
