# Commit 532a6a7a

## Summary

Found one medium-confidence privacy risk in the new heap-checkpoint artifact path.

## Findings

### Medium: Heap snapshots can copy live secrets into QA artifacts

- Kind: privacy
- File: `extensions/qa-lab/src/suite.ts`
- Line: 920
- Evidence: `captureGatewayHeapSnapshotCheckpoint` sends `SIGUSR2`, waits for the raw Node `.heapsnapshot`, then copies it into `${outputDir}/artifacts/gateway-heap-snapshots`. Heap snapshots are not redacted. A focused local check with `NODE_OPTIONS=--heapsnapshot-signal=SIGUSR2` and a fake `OPENAI_API_KEY` confirmed the fake secret string was present in the generated `.heapsnapshot`. Existing QA artifact upload patterns publish `.artifacts/qa-e2e/` wholesale, e.g. `.github/workflows/openclaw-release-checks.yml:795`.
- Impact: If `OPENCLAW_QA_GATEWAY_HEAP_CHECKPOINTS=1` is enabled for a live/provider-backed QA suite, the uploaded QA artifact can contain API keys, auth profiles, gateway tokens, prompts, user content, or channel credentials from process memory. The existing gateway debug artifact path deliberately preserves only sanitized logs and avoids copying the runtime temp root because it may contain credentials, but this new path copies a raw memory artifact.
- Suggested fix: Keep heap snapshots out of normal/uploaded QA output by default, or gate copying them behind an explicit unsafe artifact opt-in. At minimum, block heap checkpoint artifacts for live/provider modes unless an additional acknowledged secret-bearing flag is set, and document that these artifacts must not be uploaded publicly.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes, with a fake secret: start Node with `NODE_OPTIONS=--heapsnapshot-signal=SIGUSR2`, retain `process.env.OPENAI_API_KEY` in memory, signal `SIGUSR2`, and search the generated `.heapsnapshot`; the fake key appears in the raw snapshot.
- Is this the best way to solve the issue? unclear. Heap snapshots are inherently raw diagnostics, so redaction is not realistic. The safer direction is to avoid placing them in standard QA artifact trees unless the operator explicitly opts into secret-bearing artifacts.
