# Commit 76da347

## Summary

Found one low-severity macOS Config settings regression: the new lazy schema lookup path renders form controls from a stripped schema that drops `default`, so unset fields no longer display their schema defaults.

## Findings

### Low: Lazy config forms lose schema defaults

- Kind: regression
- File: `apps/macos/Sources/OpenClaw/ConfigSettings.swift`
- Line: 289
- Evidence: `ConfigSettings` now loads `config.schema.lookup` (`apps/macos/Sources/OpenClaw/ConfigSettings.swift:27`) and passes the returned `node.schema` directly into `ConfigSchemaForm` (`apps/macos/Sources/OpenClaw/ConfigSettings.swift:289`). `ConfigSchemaForm` uses `schema.explicitDefault` when the config value is absent (`apps/macos/Sources/OpenClaw/ChannelConfigForm.swift:31`), and `explicitDefault` reads the schema `default` key (`apps/macos/Sources/OpenClaw/ConfigSchemaSupport.swift:58`). But `stripSchemaForLookup` only preserves selected string/number/boolean schema metadata, `type`, `enum`, and `const`; it never preserves `default` (`src/config/schema.ts:663`). A live probe showed full schema defaults still exist while lookup schemas omit them: `channels.discord.groupPolicy` full default is `"allowlist"` but lookup default is absent; `channels.whatsapp.debounceMs` full default is `0` but lookup default is absent.
- Impact: In the macOS Config tab, unset defaulted settings can render as blank, false, zero, or “Select…” instead of the actual default. That is misleading for policy/config fields such as channel `groupPolicy`, and it is a behavior regression from the previous full-schema rendering path.
- Suggested fix: Preserve JSON-schema `default` values in `stripSchemaForLookup` when they are JSON-safe primitives/arrays/objects, or have the macOS Config tab use a lookup mode that includes all form metadata required by `ConfigSchemaForm`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Run a schema lookup for a defaulted path and compare it with the full schema; the lookup drops `default` while the full schema keeps it.
- Is this the best way to solve the issue? yes. The macOS form contract already depends on `default`, so the lookup response should preserve it for any schema node intended for form rendering.
