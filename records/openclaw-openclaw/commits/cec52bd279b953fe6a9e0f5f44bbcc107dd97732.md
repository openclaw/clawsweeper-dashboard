# Commit cec52bd

## Summary

Found one actionable regression in the new voice readiness gate. The Android voice UI now infers gateway readiness from display text, which can route users away from a working operator connection when only the node session is offline.

## Findings

### Medium: Voice is disabled when the operator gateway is connected but the node session is offline

- Kind: regression
- File: `apps/android/app/src/main/java/ai/openclaw/app/ui/VoiceScreen.kt`
- Line: 118
- Evidence: `VoiceScreen` sets `gatewayReady = gatewayStatus.isVoiceGatewayReady()`, and the helper returns false for any status containing `offline` at `apps/android/app/src/main/java/ai/openclaw/app/ui/VoiceScreen.kt:985`. `NodeRuntime.updateStatus()` emits `Connected (node offline)` when `operatorConnected` is true but `_nodeConnected` is false at `apps/android/app/src/main/java/ai/openclaw/app/NodeRuntime.kt:651`. Voice capture and talk mode use the operator session and operator connection state, not the node session, via `MicCaptureManager` / `TalkModeManager` construction around `apps/android/app/src/main/java/ai/openclaw/app/NodeRuntime.kt:515` and `apps/android/app/src/main/java/ai/openclaw/app/NodeRuntime.kt:606`.
- Impact: In the real split-session state where the operator UI connection is online but the node channel is offline, the voice screen now shows gateway setup/offline UI, disables the Realtime Talk and Dictation rows, and changes the primary action to `Connect Gateway` at `apps/android/app/src/main/java/ai/openclaw/app/ui/VoiceScreen.kt:642`. Before this commit, those controls remained usable.
- Suggested fix: Base voice readiness on the actual operator connection, e.g. collect `viewModel.isConnected` or expose a dedicated `voiceGatewayReady` state from `NodeRuntime`, and keep `statusText` only for labels.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: reach or simulate `operatorConnected == true` and `_nodeConnected == false`; `statusText` becomes `Connected (node offline)`, then the Voice tab computes `gatewayReady == false` and routes the primary action to Gateway settings.
- Is this the best way to solve the issue? yes: the UI should consume the existing connection boolean or a dedicated voice-ready boolean instead of parsing localized/status display text.
