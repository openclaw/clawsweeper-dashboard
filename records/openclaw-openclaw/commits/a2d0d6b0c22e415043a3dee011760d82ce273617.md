# Commit a2d0d6b0c2

## Summary

Found one cleanup regression in the new candidate de-duplication path. The hardening works for the covered install-stage and symlink cases, but an explicit `OPENCLAW_PLUGIN_STAGE_DIR` entry can now shadow the canonical legacy `plugin-runtime-deps` target and prevent `doctor --fix` from removing it.

## Findings

### Medium: Explicit stage entries can suppress canonical `plugin-runtime-deps` cleanup

- Kind: regression
- File: `src/commands/doctor/shared/plugin-dependency-cleanup.ts`
- Line: 228
- Evidence: `filterLegacyStaleRootCandidates` de-dupes by resolved path before the target kind is validated. Explicit stage targets are collected before legacy roots, so when `OPENCLAW_PLUGIN_STAGE_DIR` equals the canonical `OPENCLAW_STATE_DIR/plugin-runtime-deps`, the explicit-stage candidate wins. Later, `resolveSafeRemovalTarget` treats that same path as an explicit stage and rejects it as `unexpected path name` unless it is named `.openclaw-install-stage*` or has explicit-stage proof. A temp-dir probe confirmed the behavior: without `OPENCLAW_PLUGIN_STAGE_DIR`, `state/plugin-runtime-deps` was removed; with `OPENCLAW_PLUGIN_STAGE_DIR=state/plugin-runtime-deps`, cleanup emitted an unexpected-name warning and left the legacy root in place.
- Impact: `openclaw doctor --fix` can fail to remove a valid legacy runtime dependency root precisely when the environment also lists it as a plugin stage root. The repo changelog documents that `OPENCLAW_PLUGIN_STAGE_DIR` can contain layered runtime-dependency roots, so this is a plausible upgrade/repair configuration rather than a purely hostile input case.
- Suggested fix: Preserve or prefer the `legacy` classification when duplicate paths are collected for known legacy names under cleanup roots. Concretely, either de-dupe after safety resolution, or collapse duplicates by path with `legacy` taking precedence over `explicit-stage` for `plugin-runtime-deps`, `bundled-plugin-runtime-deps`, `.local/bundled-plugin-runtime-deps`, and recognized legacy debris names.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Create a temp `state/plugin-runtime-deps/demo/.openclaw-runtime-deps-stamp.json`, run `cleanupLegacyPluginDependencyState` once with only `OPENCLAW_STATE_DIR` and once with `OPENCLAW_PLUGIN_STAGE_DIR` set to that same `state/plugin-runtime-deps` path. The first removes the root; the second leaves it and warns.
- Is this the best way to solve the issue? no. The containment checks are the right direction, but duplicate candidate handling should not let a less-specific explicit-stage interpretation override a canonical legacy cleanup target.
