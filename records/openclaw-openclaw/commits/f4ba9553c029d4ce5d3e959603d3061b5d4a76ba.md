## Findings

### Low: Model failover docs still describe the removed cross-provider fallback gate

- Kind: regression
- File: `docs/concepts/model-failover.md`
- Line: 232
- Evidence: Current code now always uses `agents.defaults.model.fallbacks` when `fallbacksOverride` is undefined, then appends the configured primary at the end (`src/agents/model-fallback.ts:608`). The new tests assert exactly that behavior for a different-provider override and for duplicate provider model IDs (`src/agents/model-fallback.test.ts:1032`, `src/agents/model-fallback.test.ts:1058`). The docs still say a different-provider current run “does not append unrelated configured fallbacks from another provider” (`docs/concepts/model-failover.md:232`).
- Impact: Operators and maintainers reading the public failover contract can make the wrong provider-routing assumption. This matters because configured fallback chains can cross providers; explicit user selections are still strict via `fallbacksOverride: []` (`src/agents/agent-scope.ts:215`), so the docs need to distinguish caller-supplied strict overrides from the fallback runner’s default candidate construction.
- Suggested fix: Update the candidate-chain rules to say: requested model first, configured fallback list next when no explicit override list is supplied, configured primary last; `fallbacksOverride`, including `[]`, is the mechanism that keeps user/strict selections from walking global fallbacks.
- Confidence: high
