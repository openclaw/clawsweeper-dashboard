# Commit 86b53aa

## Summary

Found one actionable regression in the landed quiet-DM path: when Telegram status reactions are enabled, the successful “no visible DM reply” path is still finalized as an error.

## Findings

### Medium: Quiet DM no-response turns still set Telegram error status reactions

- Kind: regression
- File: `extensions/telegram/src/bot-message-dispatch.ts`
- Line: 1652
- Evidence: The new guard only synthesizes the fallback when `isGroup` is true at `extensions/telegram/src/bot-message-dispatch.ts:1616`, so a direct no-response turn with `queuedFinal === false`, no delivery, and no dispatch error leaves `sentFallback === false`. `hasFinalResponse` then remains false at `extensions/telegram/src/bot-message-dispatch.ts:1652`, and the dispatcher calls `finalizeTelegramStatusReaction({ outcome: "error", hasFinalResponse: false })` at `extensions/telegram/src/bot-message-dispatch.ts:1655`. Telegram creates that controller for opt-in status reactions when an ack reaction is active at `extensions/telegram/src/bot-message-context.ts:498`.
- Impact: For users with `messages.statusReactions.enabled` and direct-message ack reactions enabled, the fix suppresses the unwanted filler message but replaces a successful quiet DM turn with an error reaction. That is visible, misleading, and can make normal `NO_REPLY`/empty turns look failed.
- Suggested fix: Treat the quiet direct no-response path as handled for finalization purposes, for example by adding an explicit `quietNoResponseHandled = !isGroup && !dispatchError && !deliverySummary.delivered && !suppressSilentReplyFallback && !queuedFinal` and including it in the success/done status finalization without re-enabling DM fallback delivery.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A dispatcher unit scenario matching the new DM test plus a `statusReactionController` would hit the branch: `deliverReplies` is not called, `queuedFinal` is false, `hasFinalResponse` is false, and `setError` is called.
- Is this the best way to solve the issue? yes. The `isGroup` delivery guard is the right boundary for suppressing DM chatter; the missing piece is to count the intentionally quiet DM completion as handled for status cleanup.
