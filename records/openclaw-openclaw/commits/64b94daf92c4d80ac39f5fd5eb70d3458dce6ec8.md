# Commit 64b94daf

## Summary

The new `models.authLogout` path correctly removes auth-profile credentials, refreshes runtime secrets, and aborts active runs that have matching provider metadata. I found one remaining user-facing removal path that does not go through that new abort logic.

## Findings

### Medium: Config API key removal still bypasses active-run abort

- Kind: bug
- File: `src/gateway/server-methods/models-auth-status.ts`
- Line: 353
- Evidence: The new abort behavior is only implemented inside `models.authLogout`, which deletes saved auth profiles and then calls `abortChatRunsForProvider` at `src/gateway/server-methods/models-auth-status.ts:353`. The existing Settings/config-form API-key path edits `models.providers.<provider>.apiKey` through `config.set` / `config.apply` (`ui/src/ui/controllers/config.ts:219`, `ui/src/ui/controllers/config.ts:244`, `ui/src/ui/controllers/config.ts:248`), and the config-form test shows provider API keys are edited at `models.providers.openai.apiKey` (`ui/src/ui/config-form.browser.test.ts:352`). The config handlers hot-reload secrets after config changes (`src/gateway/server-methods/config.ts:421`, `src/gateway/server-reload-handlers.ts:506`) but do not call the new provider-run abort helper.
- Impact: If a user removes an API key stored in config rather than in `auth-profiles.json`, active runs can continue using the credential they already resolved. That is the spend/safety failure mode this commit is meant to close for “Remove key” style flows.
- Suggested fix: Add a shared provider-auth removal/rotation detector for config writes, or route the Settings “Remove key” affordance through a method that both removes the config/profile credential and aborts matching active runs. Add a regression test that changes/removes `models.providers.<id>.apiKey` while an active run for that provider is registered and verifies `stopReason: "auth-revoked"`.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the source path: remove or change `models.providers.<provider>.apiKey` via `config.set` / `config.apply` while an active run is registered for that provider; the current config path refreshes secrets but never calls `abortChatRunsForProvider`.
- Is this the best way to solve the issue? Unclear. The new `models.authLogout` implementation is a good primitive, but config-backed credentials need the same lifecycle policy or the UI must avoid using config writes for “Remove key”.
