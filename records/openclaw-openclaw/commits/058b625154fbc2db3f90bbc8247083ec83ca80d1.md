# Commit 058b625

## Summary

Found one low-severity retry-classification issue in the new `openclaw logs --follow` reconnect path.

## Findings

### Low: follow-mode retry treats protocol close codes as transient

- Kind: reliability
- File: `src/cli/logs-cli.ts`
- Line: 164
- Evidence: `isTransientFollowError` returns true for every `GatewayTransportError` close code except `1008` and `4000-4999`. The gateway has non-transient close codes outside that set, including protocol mismatch: `src/gateway/server/ws-connection/message-handler.ts:486` closes with `1002`, and `src/gateway/call.ts:540` wraps close codes into `GatewayTransportError`.
- Impact: `openclaw logs --follow --url ...` can spend all 8 retry attempts and backoff warnings on a non-recoverable protocol mismatch instead of failing immediately with actionable feedback. This is a regression from the old immediate-fail behavior for that class of error.
- Suggested fix: retry only known transient close cases, or explicitly exclude protocol/unsupported-payload close codes such as `1002`. Add a focused follow-mode test for `GatewayTransportError` code `1002` that asserts no retry warning and one gateway call.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes at unit level: mock `callGatewayFromCli` to reject with `GatewayTransportError({ kind: "closed", code: 1002, reason: "protocol mismatch", ... })` and run `logs --follow --url ...`; current code will enter the retry branch.
- Is this the best way to solve the issue? Yes. The narrow fix is in retry classification; the rest of the reconnect loop and cursor handling can stay as-is.
