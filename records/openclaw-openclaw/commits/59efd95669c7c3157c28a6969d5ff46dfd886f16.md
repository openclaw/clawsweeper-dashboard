# Commit 59efd95

## Summary

Found one authorization regression in the new label trigger. The workflow now treats any matching `pull_request_target` label event as authorized before checking whether the actor has write/maintain/admin access.

## Findings

### High: Label trigger bypasses the maintainer permission check for a privileged proof workflow

- Kind: security
- File: `.github/workflows/mantis-telegram-desktop-proof.yml`
- Line: 84
- Evidence: The new `pull_request_target` path enters on the `mantis: telegram-visible-proof` label, then returns `authorized=true` at lines 84-87 before the existing `allowed = ["admin", "maintain", "write"]` check runs. The parent workflow applied that permission check to all non-dispatch requests. After this authorization, the workflow can run with `issues: write`/`pull-requests: write`, Mantis/OpenAI/Crabbox/Convex/R2 secrets, and a generated Mantis GitHub App token at lines 39-43, 403-423, and 460-481.
- Impact: GitHub’s role matrix allows Triage users to apply/dismiss labels while not allowing them to create/edit/run Actions workflows; this commit turns label application into an indirect way to start this privileged `pull_request_target` workflow. GitHub documents `pull_request_target` `labeled` as a supported trigger and warns to avoid it when building/running PR code because it runs in the base repo context with privileged capabilities; GitHub Security Lab specifically notes label-based gating should require someone with write privileges. ([docs.github.com](https://docs.github.com/en/actions/reference/events-that-trigger-workflows))
- Suggested fix: Do not special-case `pull_request_target` as automatically authorized. Reuse the existing write/maintain/admin permission check for label events, or add an exact allowlist for the intended ClawSweeper/Mantis bot/app identity and keep the permission check for all human actors.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. Grant a non-write collaborator Triage access, have them apply `mantis: telegram-visible-proof` to a PR, and the workflow’s current line 84 branch authorizes the run without querying collaborator permission.
- Is this the best way to solve the issue? Yes. The original contract was explicit maintainer-level authorization; preserving that check on the new trigger closes the regression without changing the proof workflow itself.
