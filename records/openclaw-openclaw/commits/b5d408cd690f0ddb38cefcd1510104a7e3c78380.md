## Findings

### Medium: Slack dispatch preview fallback tests now fail under the new SDK imports

- Kind: regression
- File: `extensions/slack/src/monitor/message-handler/dispatch.preview-fallback.test.ts`
- Line: 233
- Evidence: `dispatch.ts` now imports `resolveChannelProgressDraftRender` at `extensions/slack/src/monitor/message-handler/dispatch.ts:895` and `buildChannelProgressDraftLine` at `extensions/slack/src/monitor/message-handler/dispatch.ts:1125`, but the full `vi.mock("openclaw/plugin-sdk/channel-streaming", ...)` in `dispatch.preview-fallback.test.ts` still only provides the older `formatChannelProgressDraftLine` surface. Running `pnpm test extensions/slack/src/monitor/message-handler/dispatch.preview-fallback.test.ts` fails 5 tests with missing mock exports for those two functions.
- Impact: the Slack extension test lane can fail even though the runtime code compiles, blocking changed-gate/CI for this touched Slack dispatch surface.
- Suggested fix: update the test mock to include `buildChannelProgressDraftLine` and `resolveChannelProgressDraftRender`, or switch it to a partial mock that spreads the real module and overrides only the specific helpers the test needs.
- Confidence: high

### Low: Rich progress blocks can show stale lines when `maxLines` exceeds Slack's block cap

- Kind: bug
- File: `extensions/slack/src/progress-blocks.ts`
- Line: 57
- Evidence: the dispatch path keeps the newest configured progress lines with `.slice(-resolveChannelProgressDraftMaxLines(...))`, but `buildSlackProgressDraftBlocks` then caps rich blocks with `params.lines.slice(0, availableLineBlocks)`. If a user sets `streaming.progress.maxLines` above the available Slack block count, the visible Block Kit fields keep the oldest entries in the retained window while newer progress only appears in the text fallback.
- Impact: rich Slack progress drafts can lag behind current work during long tool-heavy runs with high `maxLines`, which defeats the purpose of progress drafts.
- Suggested fix: cap the rich line blocks with the newest entries, e.g. `params.lines.slice(-availableLineBlocks)`, while preserving the label block.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes: `pnpm test extensions/slack/src/monitor/message-handler/dispatch.preview-fallback.test.ts` reproduces the test regression directly.
- Is this the best way to solve the issue? yes for the test regression: the production imports changed, so the owned test mock should be kept in sync or made partial. For the rich-block ordering edge case, keeping the newest entries matches the existing progress-line retention behavior.
