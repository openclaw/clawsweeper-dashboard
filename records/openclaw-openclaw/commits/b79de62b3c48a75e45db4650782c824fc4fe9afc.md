## Summary

Found one low-severity compatibility regression in the tsdown build wrapper added by this commit. The Codex dynamic-tool profile removal path otherwise looks internally consistent and has doctor migration coverage.

## Findings

### Low: tsdown heap default overrides explicit underscore-form Node heap settings

- Kind: compatibility
- File: `scripts/tsdown-build.mjs`
- Line: 205
- Evidence: `resolveTsdownEnv` only detects `--max-old-space-size`, but Node also accepts `--max_old_space_size`. With `NODE_OPTIONS="--trace-warnings --max_old_space_size=8192"`, the wrapper now returns `--trace-warnings --max_old_space_size=8192 --max-old-space-size=6144`; Node applies the later 6144 MB setting.
- Impact: users or CI lanes that already set the underscore-form V8 heap option can have their explicit larger heap reduced to 6144 MB, reintroducing tsdown OOM/build failures despite an explicit override.
- Suggested fix: treat both spellings as explicit heap settings, for example by matching `--max[-_]old[-_]space[-_]size(?:=|\s+)`, and add a test for `--max_old_space_size=8192`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. `resolveTsdownBuildInvocation({ env: { NODE_OPTIONS: "--trace-warnings --max_old_space_size=8192" } })` appends `--max-old-space-size=6144`; `NODE_OPTIONS="--max_old_space_size=8192 --max-old-space-size=6144" node -e "console.log(require('v8').getHeapStatistics().heap_size_limit)"` reports a ~6 GiB heap.
- Is this the best way to solve the issue? yes. The intended contract is to add a default only when the operator has not already supplied a heap limit, so recognizing Node’s accepted underscore spelling preserves that contract with a narrow change.
