## Findings

### Low: tsdown heap default overrides Node's accepted underscore heap flag

- Kind: compatibility
- File: `scripts/tsdown-build.mjs`
- Line: 205
- Evidence: `resolveTsdownEnv` treats only `--max-old-space-size` as an explicit heap setting. Node also accepts `--max_old_space_size`; with `NODE_OPTIONS="--max_old_space_size=8192"`, `resolveTsdownBuildInvocation` now returns `--max_old_space_size=8192 --max-old-space-size=6144`. A live Node check showed the later `6144` setting wins.
- Impact: CI or local builders using Node's accepted underscore spelling can have an explicit larger tsdown heap silently lowered to 6144 MB, reintroducing OOM risk for the build this change is trying to protect.
- Suggested fix: recognize both spellings, for example `--max[-_]old[-_]space[-_]size(?:=|\s+)`, and add a unit test for `--max_old_space_size=8192`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Run `resolveTsdownBuildInvocation` with `env: { NODE_OPTIONS: "--max_old_space_size=8192" }`; it appends `--max-old-space-size=6144`. Running Node with both flags confirms the appended lower value wins.
- Is this the best way to solve the issue? yes. This is a parser/recognition bug in the new preservation guard; widening the regex preserves the intended behavior without changing the default for users with no heap setting.
