# Commit 6a482584ee

## Summary

Found one cron edit regression that can overwrite an existing delivery mode when a user only meant to disable best-effort delivery while editing an agent payload.

## Findings

### Medium: `--no-best-effort-deliver` plus payload edits forces delivery back to announce

- Kind: regression
- File: `src/cli/cron-cli/register.cron-edit.ts`
- Line: 306
- Evidence: The new condition sets `delivery.mode = "announce"` whenever there is any agent-turn payload field and `opts.bestEffortDeliver` is boolean. That includes `--no-best-effort-deliver`, not just `--best-effort-deliver`. The service merge code in `src/cron/service/jobs.ts:891` then treats the mode as explicit and overwrites the existing job delivery mode.
- Impact: Editing an existing webhook or disabled-delivery cron job with `openclaw cron edit <id> --message "new" --no-best-effort-deliver` changes the job from its existing `webhook` or `none` delivery mode to `announce`. That can break webhook delivery or unexpectedly re-enable chat fallback delivery.
- Suggested fix: Only infer `mode: "announce"` for the true case, e.g. `opts.bestEffortDeliver === true`, and do not set `delivery.mode` for `--no-best-effort-deliver` unless the user also passed an explicit delivery mode flag such as `--announce`, `--deliver`, or `--no-deliver`. Add a regression test covering an existing webhook job or at least the CLI patch shape for `--message ... --no-best-effort-deliver`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Existing job: `delivery: { mode: "webhook", to: "https://example.invalid/hook", bestEffort: true }`. The CLI code now emits `{ delivery: { mode: "announce", bestEffort: false } }` for `--message new --no-best-effort-deliver`; a focused `applyJobPatch` probe confirmed the resulting delivery becomes `{"mode":"announce","to":"https://example.invalid/hook","bestEffort":false}`.
- Is this the best way to solve the issue? yes. The compatibility behavior should be scoped to the true best-effort flag; the false flag is a delivery error-policy patch, not an implicit delivery-mode selection.
