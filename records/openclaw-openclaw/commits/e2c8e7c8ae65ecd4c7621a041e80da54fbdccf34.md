# Commit e2c8e7c8ae65

## Summary

Found one concrete regression in an existing caller: `openclaw node run --port 99999` now silently falls back to the saved/default port instead of rejecting the explicit invalid port. The shared parser fix is correct, but this caller collapses `null` into fallback without checking whether the user supplied `--port`.

## Findings

### Medium: `node run --port` silently ignores out-of-range values

- Kind: regression
- File: `src/cli/node-cli/register.ts`
- Line: 57
- Evidence: `parsePortWithFallback` returns `parsed ?? fallback` at `src/cli/node-cli/register.ts:18`, and `node run` calls it for `opts.port` at `src/cli/node-cli/register.ts:57`. After this commit, `parsePort("99999")` returns `null`, so the explicit CLI value becomes `18789` or the existing saved config port. A focused smoke check showed `99999: parsePort=null nodeRunPort=18789`.
- Impact: The commit’s intended CLI-level rejection does not hold for `openclaw node run`; an invalid explicit user value is silently ignored and the node host may connect to, and persist, a different gateway port than requested.
- Suggested fix: In the `node run` action, parse `opts.port` directly and fail when `opts.port !== undefined && parsed === null`, matching `runNodeDaemonInstall` and gateway/daemon install behavior. Keep fallback only for omitted `--port`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Source path: `openclaw node run --port 99999` -> `parsePortWithFallback(opts.port, existing?.gateway?.port ?? 18789)` -> `parsePort("99999") === null` -> fallback port. I did not run the full CLI because `runNodeHost` writes node-host config and starts a long-running gateway client.
- Is this the best way to solve the issue? yes. The parser should keep returning `null` for invalid ports; the caller needs to distinguish omitted input from explicit invalid input.
