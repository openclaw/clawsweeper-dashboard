# Commit 9210dfc

## Summary

The commit fixes the skills status/missing-requirements path for `openclaw.os: macos`, but it leaves runtime inclusion on the old literal `macos` vs `darwin` comparison. After this merge, `skills check` can report the skill eligible while the prompt/runtime loader still filters the same skill out.

## Findings

### Medium: `macos` skills can be reported eligible but still excluded at runtime

- Kind: regression
- File: `src/shared/config-eval.ts`
- Line: 119
- Evidence: `resolveMissingOs` now normalizes `macos` to `darwin` in `src/shared/requirements.ts:81`, and `buildWorkspaceSkillStatus` consumes that path through `evaluateEntryRequirementsForCurrentPlatform` in `src/agents/skills-status.ts:228`. Actual skill loading still calls `shouldIncludeSkill` in `src/agents/skills/config.ts:89`, which delegates to `evaluateRuntimeEligibility`; that helper still checks `!osList.includes(resolveRuntimePlatform())` and remote platforms literally. A direct probe on current `main` with `process.platform = "darwin"` returned `{"statusMissingOs":[],"runtimeEligible":false,"remoteStatusMissingOs":[],"remoteRuntimeEligible":false}` for `os: ["macos"]`.
- Impact: a macOS-only skill using the now-accepted `openclaw.os: macos` metadata can move from “missing requirements” to “eligible” in status output, but still not appear in the actual available skills prompt or runtime-loaded skill set. This makes the fix incomplete and creates a misleading state for the issue this commit claims to close.
- Suggested fix: share the same OS requirement canonicalization with `evaluateRuntimeEligibility` and any skills install-option OS filtering, then add focused coverage for local and remote Darwin matches in `src/shared/config-eval.test.ts` and the skill status/runtime caller tests.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. On current `main`, mocking `process.platform` to `darwin` and comparing `resolveMissingOs({ required: ["macos"], localPlatform: "darwin" })` with `evaluateRuntimeEligibility({ os: ["macos"], ...satisfiedRuntimeChecks })` shows status returns no missing OS while runtime returns `false`.
- Is this the best way to solve the issue? yes. The alias should be one shared requirement/platform contract rather than a status-only normalization, otherwise status, runtime loading, hooks, and installer filtering can drift.
