# Commit 284dcc5

## Summary

Found one heartbeat regression: `notify:false` heartbeat tool responses now look like rich outbound content because their internal metadata lives in `channelData`, so cron no-change heartbeats are no longer suppressed.

## Findings

### Medium: Heartbeat metadata prevents silent heartbeat suppression

- Kind: regression
- File: `src/cron/heartbeat-policy.ts`
- Line: 20
- Evidence: `shouldSkipHeartbeatOnlyDelivery` now checks `hasOutboundReplyContent({ ...payload, text: undefined })`, and that treats any non-empty `channelData` as content. But `createHeartbeatToolResponsePayload` always adds `channelData.openclawHeartbeatResponse`, even when `notify:false` sets visible text to `HEARTBEAT_OK` in `src/auto-reply/heartbeat-tool-response.ts:92`. Cron uses this predicate to set `skipHeartbeatDelivery` in `src/cron/isolated-agent/run.ts:966`.
- Impact: A no-change heartbeat response from the heartbeat tool, which is explicitly supposed to avoid notifying the user when `notify:false`, is now delivered as a non-empty payload instead of being treated as silent. Recurring cron heartbeats can leak `HEARTBEAT_OK`/metadata-backed empty messages into the configured delivery channel.
- Suggested fix: Do not count heartbeat response metadata as user-visible non-text content for heartbeat skip decisions. Either ignore `channelData` in this specific `hasAnyNonTextContent` check, or special-case `openclawHeartbeatResponse` with `notify:false` before applying the generic rich-content predicate.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. I ran:
  `node --import tsx -e "const [{shouldSkipHeartbeatOnlyDelivery},{createHeartbeatToolResponsePayload}] = await Promise.all([import('./src/cron/heartbeat-policy.ts'), import('./src/auto-reply/heartbeat-tool-response.ts')]); const payload = createHeartbeatToolResponsePayload({outcome:'no_change', notify:false, summary:'All clear'}); console.log(JSON.stringify(payload)); console.log('skip=' + shouldSkipHeartbeatOnlyDelivery([payload], 300));"`
  It printed a `HEARTBEAT_OK` payload with `channelData.openclawHeartbeatResponse` and `skip=false`.
- Is this the best way to solve the issue? yes, scoped to heartbeat delivery policy. The generic rich-content helper should continue to treat channel data as content elsewhere; heartbeat’s internal metadata is the exceptional case.
