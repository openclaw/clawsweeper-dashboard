# Commit e65b490f11

## Summary

Found one actionable regression in the widened Telegram draft-streaming path. The core fix for plain current-message replies is valid, but the new guard can now stream replies whose final payload explicitly targets a different Telegram message, and final preview editing cannot apply that different reply target.

## Findings

### Medium: Streaming finalization can drop explicit Telegram reply targets

- Kind: regression
- File: `extensions/telegram/src/bot-message-dispatch.ts`
- Line: 399
- Evidence: The commit enables answer drafts whenever `replyQuoteText == null`, even though `replyQuoteByMessageId` may still contain reply candidates and final payloads may carry an explicit `replyToId`. Draft preview creation always binds the first preview message to `draftReplyToMessageId`, derived from the triggering message at `extensions/telegram/src/bot-message-dispatch.ts:406`. Later, text-only finals are finalized by editing the existing preview when `canEditViaPreview` is true, without checking `payload.replyToId`, at `extensions/telegram/src/lane-delivery-text-deliverer.ts:507` and `extensions/telegram/src/lane-delivery-text-deliverer.ts:529`. I confirmed this helper path with a focused `tsx` harness: a final payload `{ replyToId: "9001" }` returned `preview-finalized`, called `editPreview`, and never called `sendPayload`.
- Impact: With `channels.telegram.replyToMode: "first"` or `"all"` and partial streaming enabled, a model reply containing `[[reply_to:<id>]]` can visibly finish as an edit to the draft preview that was originally sent as a reply to the triggering message, not the explicitly requested message. Before this commit, non-off `replyToMode` disabled the answer draft because the quote-candidate map was always non-empty, so this case went through `deliverReplies` and preserved the explicit reply target. The docs promise explicit `[[reply_to_*]]` tags are honored at `docs/channels/telegram.md:488`.
- Suggested fix: Track the reply target used for the preview and bypass preview finalization when the final payload has an explicit `replyToId`/reply tag that resolves to a different target, falling back to `sendPayload`. Add a regression test with an active draft stream plus final payload `replyToId: "9001"`.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes at source level: enable a draft stream, deliver a final answer payload with `replyToId` different from the preview‚Äôs `replyToMessageId`, and observe preview finalization edits the existing message instead of calling `sendPayload`.
- Is this the best way to solve the issue? Unclear. The safest local fix is a final-delivery fallback when reply parameters differ from the preview message. A broader fix would make preview delivery carry enough reply metadata to compare all Telegram reply constraints before editing.
