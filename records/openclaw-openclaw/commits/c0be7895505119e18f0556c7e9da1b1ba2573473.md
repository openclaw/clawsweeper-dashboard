## Summary

Found one low-severity helper bug in the changed parent-directory calculation. The shared Telegram user credential kind/parser wiring otherwise matches the broker payload contract and passed the focused checks.

## Findings

### Low: Bare output paths create an unintended truncated directory

- Kind: bug
- File: `scripts/e2e/telegram-user-credential.ts`
- Line: 171
- Evidence: `writePrivateJson()` now computes the parent with `expanded.slice(0, expanded.lastIndexOf("/"))`. For a documented-style bare file name such as `payload.json`, `lastIndexOf("/")` is `-1`, so this evaluates to `payload.jso`. The helper then creates that directory before writing `payload.json` in the current directory.
- Impact: Running `export --output payload.json`, `lease-restore --lease-file lease.json`, or `--payload-output payload.json` leaves an unexpected directory named after the output file minus its last character. If that path already exists as a file, the command can also fail before writing the JSON.
- Suggested fix: Use `dirname` from `node:path` after `expandHome()`, and only call `mkdir()` when the directory is not `"."`; alternatively normalize no-slash paths to the current directory.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The path expression itself reproduces it: `"payload.json".slice(0, "payload.json".lastIndexOf("/"))` returns `payload.jso`.
- Is this the best way to solve the issue? yes. `path.dirname()` is the standard contract for parent directory extraction and avoids the no-slash `-1` edge case.
