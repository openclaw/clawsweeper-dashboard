Found one low-severity concurrency/UX regression risk in the new asynchronous image processing path. The image resize logic itself looks narrowly scoped, dependency-free, and aligned with the linked issue.

## Findings

### Low: Selected photos can still be sent before processing finishes

- Kind: concurrency
- File: `apps/shared/OpenClawKit/Sources/OpenClawChatUI/ChatComposer.swift`
- Line: 418
- Evidence: `loadPhotosPickerItems` calls `self.viewModel.addImageAttachment(...)` without awaiting completion. That public method starts a `Task`, and the commit now makes the private add path wait on detached JPEG processing before appending to `attachments` at `apps/shared/OpenClawKit/Sources/OpenClawChatUI/ChatViewModel.swift:1315` and `apps/shared/OpenClawKit/Sources/OpenClawChatUI/ChatViewModel.swift:1334`. During that processing window, `canSend` and `performSend` only see the current `attachments` array, so a user who picked a large photo and quickly sends a caption can send text without the selected image; the processed image can then appear afterward as a stale pending attachment.
- Impact: Large images are exactly the new path’s target, and they create the longest processing window. This can drop an intended attachment from the current send and leave it queued for a later message.
- Suggested fix: Track pending attachment processing in the view model and disable send until it drains, or change the picker path to await/slot attachment processing so selected-item order and send readiness are explicit.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes by source path: pick a large image, type a caption, and send before the attachment strip appears. I did not run this on a physical iPhone in this environment.
- Is this the best way to solve the issue? Yes for the direction: the model needs an explicit “processing attachments” state or ordered pending slots because processing now has meaningful latency.
