# Commit 4b9aa302

## Summary

Found one actionable regression in the guardrail test change. Exporting `LEGACY_DAEMON_CLI_EXPORTS` looks like a correct compile fix for `scripts/write-cli-compat.ts`, but removing Mattermost from the generated channel metadata guardrail lets current `main` accept invalid Mattermost channel config without validation.

## Findings

### Medium: Mattermost channel config is no longer guarded by generated metadata validation

- Kind: test_gap
- File: `src/plugins/contracts/config-footprint-guardrails.test.ts`
- Line: 114
- Evidence: The commit removes `mattermost` from the private-network generated metadata assertion. Current `scripts/generate-bundled-channel-config-metadata.ts:74` only discovers `src/config-schema.*`, while Mattermost exposes its channel schema through `extensions/mattermost/src/config-surface.ts:4` and has no generated metadata entry. `src/config/validation.ts:928` seeds channel schemas from `GENERATED_BUNDLED_CHANNEL_CONFIG_METADATA`, and `src/config/validation.ts:1071` skips channel validation entirely when no schema exists. A focused probe confirmed `validateConfigObjectWithPlugins({ channels: { mattermost: { network: { dangerouslyAllowPrivateNetwork: "yes" } } } })` returns `ok: true`, while the same bad type for `matrix` returns an invalid-config issue. Mattermost’s own schema requires a boolean at `extensions/mattermost/src/config-schema-core.ts:73`.
- Impact: Release guardrails now pass while Mattermost channel config is absent from generated channel metadata. Users can save misspelled or wrongly typed Mattermost settings, including the private-network opt-in and nested group config, and OpenClaw will silently keep running with ineffective config instead of failing validation.
- Suggested fix: Make Mattermost discoverable by the generated metadata path, for example by adding the expected `extensions/mattermost/src/config-schema.ts` surface or teaching the generator to load the existing `config-surface.ts`, then restore `mattermost` to this guardrail list.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The direct `node --import tsx` validation probe accepts invalid Mattermost config and rejects the comparable Matrix config.
- Is this the best way to solve the issue? yes. The runtime schema already exists; the safer fix is to wire it into the manifest/generated metadata path and keep the release guardrail covering it.
