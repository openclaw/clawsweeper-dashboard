# Commit dd0a9bf

## Summary

Found one still-present guard regression in the CodeQL replacement. The old raw-socket guard allowed exact callsites; the new query allows entire owner functions, so future raw socket egress added inside an already-allowed function can land unclassified.

## Findings

### Medium: Raw socket allowlist now permits new callsites inside allowed functions

- Kind: regression
- File: `.github/codeql/openclaw-boundary/queries/raw-socket-callsite-classification.ql`
- Line: 51
- Evidence: `allowedOwnerScope` allows any raw socket call whose ancestor function matches an allowed path/name. The removed script’s allowlist was intentionally exact `path:line` entries, with the comment “Each entry is intentionally a concrete callsite so new raw socket egress fails until reviewed.” I verified this with a scratch CodeQL database: adding a second `net.connect({ host: "example.com", port: 6667 })` inside `src/infra/jsonl-socket.ts`’s allowed `requestJsonlSocket` function produced zero CodeQL findings, while the same query reported all 10 raw-socket idioms in an unallowed scratch file.
- Impact: a future direct `net`/`tls`/`http2` client call added inside allowed functions such as `startDebugProxyServer`, `connectIrcClient`, or `requestJsonlSocket` bypasses the “classify before landing” boundary. That weakens the managed-proxy/raw-egress review gate compared with the previous exact-callsite guard.
- Suggested fix: make the CodeQL allowlist callsite-granular again, for example by matching file plus start line/start column or by moving approved raw socket calls into small named wrapper functions where the wrapper is the only allowed scope. Add CodeQL fixture coverage for “extra raw socket call inside an allowed owner function is reported.”
- Confidence: high
