One supply-chain hardening issue found. The functional CI/UI changes otherwise checked out cleanly.

## Findings

### Medium: Publish tokens are now job-wide and reach unrelated action/install/build steps

- Kind: supply_chain
- File: `.github/workflows/website-installer-sync.yml`
- Line: 137
- Evidence: The commit moved `OPENCLAW_GH_TOKEN` into `jobs.sync-website.env`, so it is available throughout the job. That includes `oven-sh/setup-bun@v2` at line 180 and the later `bun install` / `bun run build` step in the checked-out `openclaw.ai` repo. The same pattern was added for `OPENCLAW_DOCS_SYNC_TOKEN` in `.github/workflows/docs-sync-publish.yml:19`, making the docs publish token available to setup and npm/MDX-check steps that do not need it. GitHub documents job-level env as applying at the job scope, and separately warns that a compromised third-party action can access secrets available to the workflow; GitHub also notes secrets cannot be referenced directly in `if:` conditionals, but that does not require keeping the secret value in every subsequent step. ([docs.github.com](https://docs.github.com/en/actions/reference/workflows-and-actions/workflow-syntax?apiversion=2022-11-28&utm_source=openai))
- Impact: A compromise or malicious change in an unrelated action, npm/bun dependency lifecycle, or checked-out website build code can read and exfiltrate a token whose only intended use is checkout/push. That is a regression from the previous narrower exposure.
- Suggested fix: Add an initial token-presence probe step with the secret scoped only to that step, emit a non-secret output such as `has_token=true`, and gate later steps on that output. Pass the secret only to the checkout/clone/push steps that require it, preferably through `with.token` or step-local `env`.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes, by inspecting the workflow: the secret is assigned at job scope and unrelated later steps run in that job when sync changes exist. Exploitability depends on a compromised action/dependency/repo script.
- Is this the best way to solve the issue? yes; a non-secret probe output preserves the “skip when secret is missing” behavior without expanding token scope.
