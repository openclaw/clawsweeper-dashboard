# Commit 4b0f16d

## Summary

Found one actionable gap: the new visible fallback/recovery notices are implemented for the main `runReplyAgent` path, but queued followup turns still run their own model fallback path and deliver only the agent result payloads.

## Findings

### Medium: Queued followup fallback transitions remain silent

- Kind: bug
- File: `src/auto-reply/reply/followup-runner.ts`
- Line: 356
- Evidence: `createFollowupRunner` calls `runWithModelFallback` and captures `fallbackProvider` / `fallbackModel` at `src/auto-reply/reply/followup-runner.ts:356`, then builds delivery only from `runResult.payloads` at `src/auto-reply/reply/followup-runner.ts:513`. It never calls `resolveFallbackTransition`, never updates `fallbackNoticeSelectedModel` / `fallbackNoticeActiveModel`, and never prepends an `isFallbackNotice` payload like the new main path does in `src/auto-reply/reply/agent-runner.ts:1659`.
- Impact: followup turns that hit auto fallback or recover back to the selected model still complete without the promised chat notice, so users can see fallback/recovery notices on direct turns but not on queued followup work.
- Suggested fix: share the fallback notice/state helper with `createFollowupRunner` after `fallbackResult` is known, then thread/dedupe/deliver the notice with the followup payloads using the same `isFallbackNotice` contract.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Queue a followup run where `runWithModelFallback` returns a different provider/model than the selected run. The queued path will route only `runResult.payloads`; there is no code path that constructs or delivers the fallback transition notice.
- Is this the best way to solve the issue? yes. The normal turn path already has the intended notice/state contract; the followup runner should reuse that behavior rather than inventing a separate notice format.
