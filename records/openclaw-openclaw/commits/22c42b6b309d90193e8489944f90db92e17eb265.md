### High: `modelAllowlist.loadCatalog` is outside the provider wizard contract

- Kind: compatibility
- File: `extensions/github-copilot/index.ts`
- Line: 325
- Evidence: `pnpm tsgo:extensions` reports `extensions/github-copilot/index.ts(325,13): error TS2353: Object literal may only specify known properties, and 'loadCatalog' does not exist in type '{ allowedKeys?: string[] | undefined; initialSelections?: string[] | undefined; message?: string | undefined; }'.` The declared contract at `src/plugins/types.ts:1101` only permits `allowedKeys`, `initialSelections`, and `message`.
- Impact: This is a second touched-code typecheck failure in the same production lane.
- Suggested fix: Either move the behavior to an existing supported field, such as `wizard.setup.modelSelection.promptWhenAuthChoiceProvided`, or intentionally extend `ProviderPluginWizardSetup.modelAllowlist` and the relevant consumers/tests to support `loadCatalog`.
- Confidence: high

### Medium: Existing Copilot profile reuse bypasses token expiry eligibility

- Kind: regression
- File: `extensions/github-copilot/index.ts`
- Line: 79
- Evidence: `resolveExistingCopilotTokenProfileId` accepts the first Copilot token profile with a token or token ref, but it does not apply `resolveAuthProfileEligibility` / `resolveAuthProfileOrder`. The new interactive path then defaults to returning that profile when the user declines “Re-run login?” at `extensions/github-copilot/index.ts:260`. `docs/auth-credential-semantics.md` says token profiles with invalid or past `expires` are ineligible.
- Impact: An expired or invalid Copilot token profile can now be reused by default during interactive onboarding. Before this commit, the interactive flow always ran device login and refreshed the stored credential.
- Suggested fix: Select existing Copilot profiles through the canonical auth-order/eligibility resolver with `ctx.config`, and skip the reuse prompt when no eligible profile remains.
- Confidence: medium
