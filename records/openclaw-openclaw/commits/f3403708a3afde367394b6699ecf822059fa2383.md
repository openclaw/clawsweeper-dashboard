# Commit f3403708a3

## Summary

Found one regression in the new fallback session classification. It fixes the synthetic test path by adding a parser to `fallbackchat`, but real fallback-only plugins such as Google Chat and IRC still do not expose `parseExplicitTarget`, `inferTargetChatType`, or `resolveOutboundSessionRoute`, so native group targets now fall through to `direct`.

## Findings

### Medium: Fallback-only group targets are mirrored as direct sessions

- Kind: regression
- File: `src/infra/outbound/outbound-session.ts`
- Line: 121
- Evidence: The new fallback table only recognizes `user:`, `dm:`, `channel:`, `conversation:`, `thread:`, `group:`, and `room:` before defaulting to `"direct"` at `src/infra/outbound/outbound-session.ts:121`. Google Chat has `normalizeTarget`/`targetResolver` for `spaces/{space}` but no parser/infer/session-route hook at `extensions/googlechat/src/channel.ts:166`, while cron calls `resolveOutboundSessionRoute` with only `target`, not `resolvedTarget`, at `src/cron/isolated-agent/delivery-dispatch.ts:601`.
- Impact: Direct cron delivery still sends to the correct provider target, but the transcript mirror/session metadata is written to the wrong OpenClaw session. A Google Chat space target like `spaces/AAA` now resolves as `peer.kind: "direct"`, `to: "user:spaces/AAA"`, `sessionKey: "agent:main:main"` instead of the group session. IRC `#openclaw` has the same direct fallback. That can hide the delivery from the actual group history and contaminate the main/direct session context used by later turns.
- Suggested fix: Keep classification plugin-owned, but add `parseExplicitTarget` or `inferTargetChatType` for fallback-only plugins whose native IDs encode kind, at least Google Chat `spaces/` vs `users/` and IRC `#`/`&` channels. Alternatively, carry the resolved target kind from delivery target resolution into the cron mirror route before calling `resolveOutboundSessionRoute`. Add a test using the real Google Chat or IRC plugin fallback path, not only `fallbackchat`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused `node --import tsx --input-type=module` probe on current `main` with the real Google Chat plugin returns `chatType: "direct"` and `to: "user:spaces/AAA"` for `resolveOutboundSessionRoute({ channel: "googlechat", target: "spaces/AAA" })`; the same probe with `resolvedTarget: { kind: "group" }` returns the expected group route.
- Is this the best way to solve the issue? yes, if the goal is tightened generic fallback behavior. The safe direction is to move kind inference into the affected plugins or pass already-resolved kind information through cron, rather than restoring broad global guesses for every plugin.
