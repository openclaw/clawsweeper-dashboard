# Commit dffc295

## Summary

Found one reliability risk in the new scenario expansion path: the documented `release-history` plus `reported-issues` package-acceptance shape expands many Docker lanes inside a single targeted job, but the workflow still groups only the raw lane name and keeps the old 90-minute job timeout.

## Findings

### Medium: Scenario-expanded published survivor lanes are not sharded before the 90-minute targeted Docker job

- Kind: reliability
- File: `.github/workflows/openclaw-live-and-e2e-checks-reusable.yml`
- Line: 860
- Evidence: `plan_docker_lane_groups` splits only `inputs.docker_lanes`, so `published-upgrade-survivor` remains one matrix group. The new planner then expands that one lane across every baseline and scenario in `scripts/lib/docker-e2e-plan.mjs:150`. With eight release-history-style baselines and `reported-issues`, the plan expands to 40 lanes in one job. Each `published-upgrade-survivor` lane has weight 3 and a 25-minute timeout in `scripts/lib/docker-e2e-scenarios.mjs:287`, while default Docker/npm resource caps allow only three such lanes at once. The job timeout is still 90 minutes at `.github/workflows/openclaw-live-and-e2e-checks-reusable.yml:895`.
- Impact: The newly documented `published_upgrade_survivor_baselines=release-history` plus `published_upgrade_survivor_scenarios=reported-issues` path can be killed by the outer workflow timeout before the scheduler’s own per-lane timeouts and artifact summaries complete. That makes Package Acceptance fail as a harness timeout rather than as a product regression.
- Suggested fix: Expand `published-upgrade-survivor` baseline/scenario matrices before `plan_docker_lane_groups` builds the GitHub matrix, or teach that grouping step to call the Docker planner and group expanded lane names. A smaller fallback is to raise the targeted job timeout only when the expanded plan size requires it, but sharding gives better artifacts and reruns.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the scheduling shape: a local planner probe with eight baseline specs and `reported-issues` produced one raw workflow group but 40 expanded lanes. I did not run the full Docker matrix, so the wall-clock timeout depends on actual lane duration.
- Is this the best way to solve the issue? Likely yes: sharding expanded lanes preserves existing per-lane timeout semantics and keeps rerun artifacts focused. Raising the single job timeout is safer than the current state but keeps one oversized job.
