# Commit 57020dae

## Summary

The commit enables `dropThinkingBlocks` for custom Anthropic-compatible models with `compat.supportsReasoningEffort: false`, but the sanitizer behind that flag intentionally preserves the latest assistant thinking block. That leaves the normal follow-up path able to resend exactly the `thinking` block this change is meant to suppress.

## Findings

### Medium: Latest prior assistant thinking still replays to unsupported Anthropic-compatible providers

- Kind: bug
- File: `src/agents/transcript-policy.ts`
- Line: 143
- Evidence: The new policy path sets `dropThinkingBlocks: true` for unowned Anthropic-compatible models with `supportsReasoningEffort: false`. Both session-history sanitization and the outbound stream wrapper then call `dropThinkingBlocks(...)`, but that helper skips the latest assistant message unconditionally at `src/agents/pi-embedded-runner/thinking.ts:176`. Existing tests confirm this behavior even when a user follow-up comes after the assistant turn (`src/agents/pi-embedded-runner/thinking.test.ts:74`, `src/agents/pi-embedded-runner.sanitize-session-history.test.ts:1287`).
- Impact: For a transcript shaped like `user -> assistant(thinking,text) -> user follow-up`, the previous assistant response is still the latest assistant message, so its `type: "thinking"` block remains in the next Anthropic Messages payload. A Kimi/Qiniu-compatible endpoint that rejects `thinking` content can still return the reported HTTP 400 on the first follow-up after a thinking-bearing assistant turn.
- Suggested fix: Separate “drop legacy Claude thinking except latest signed turn” from “provider rejects thinking blocks.” For the latter, strip thinking from completed prior assistant turns, for example by using a `dropReasoningFromHistory`-style path or a new policy mode that preserves only an active same-turn tool continuation when that is intentionally supported.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Source-level repro: resolve policy for provider `qiniu`, API `anthropic-messages`, model `moonshotai/kimi-k2.5`, and `compat.supportsReasoningEffort: false`; then sanitize a transcript containing `user -> assistant({type:"thinking"}, text) -> user`. The policy enables `dropThinkingBlocks`, but `dropThinkingBlocks` preserves that assistant thinking block because it is the latest assistant turn.
- Is this the best way to solve the issue? yes, directionally. The fix needs a stricter unsupported-thinking replay policy, not just the existing Claude-compatible “drop older thinking” helper.
