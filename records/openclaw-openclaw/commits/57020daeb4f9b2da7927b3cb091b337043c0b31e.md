# Commit 57020da

## Summary

Found one incomplete-fix bug. The new compat flag enables the existing `dropThinkingBlocks` policy for custom Anthropic-compatible models, but that sanitizer intentionally preserves the latest assistant turn. A normal follow-up or model switch can still replay a `thinking` block to the reported Qiniu/Kimi route.

## Findings

### Medium: Latest assistant thinking block is still replayed to providers that reject `thinking`

- Kind: bug
- File: `src/agents/transcript-policy.ts`
- Line: 141
- Evidence: The commit maps `compat.supportsReasoningEffort === false` to `dropThinkingBlocks: true`, but both replay history and the outbound stream wrapper implement that by calling `dropThinkingBlocks(...)`. That helper explicitly skips the latest assistant message (`if (i === latestAssistantIndex) out.push(msg)`), and its existing tests assert that the latest assistant thinking block is preserved.
- Impact: A transcript ending with an assistant `thinking` block, such as after a Claude turn, a model switch, or a previous Kimi turn that persisted parsed thinking, will still resend that latest `thinking` content block on the next Qiniu/Kimi Anthropic Messages request. That matches the closed issue’s HTTP 400 failure mode: `Invalid value: thinking`.
- Suggested fix: Use a separate policy/helper for models that do not support Anthropic thinking content at all, stripping all completed assistant thinking blocks while preserving only active same-turn tool continuations if needed. Add a regression test with `[user, assistant(thinking+text), user]` for `qiniu/moonshotai/kimi-k2.5` and `supportsReasoningEffort: false`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Source-level path: resolve policy for custom provider `qiniu`, `modelApi: "anthropic-messages"`, model compat `supportsReasoningEffort: false`; then replay history or outbound wrapper calls `dropThinkingBlocks`, which preserves the latest assistant `thinking` block.
- Is this the best way to solve the issue? unclear. The fix direction should distinguish Claude-style “drop old thinking but preserve latest signed replay” from Kimi-style “this endpoint rejects `thinking` content entirely.”
