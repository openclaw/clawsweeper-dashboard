# Commit c031274

## Summary

Found one concrete cross-surface regression risk: the shared git spec parser now accepts slash-containing refs, but the existing plugin git installer still cannot check out remote branches by that parsed ref. The new skill installer handles this correctly; the plugin path needs the same ref-resolution fallback.

## Findings

### Medium: Slash refs now parse for plugin git installs but still fail checkout

- Kind: bug
- File: `src/plugins/git-install.ts`
- Line: 62
- Evidence: This commit changed `splitGitSpecRef` so `git:owner/repo@feature/foo` parses as `{ ref: "feature/foo" }`. That parser is shared with plugin installs, but `installPluginFromGitSpec` still checks out the parsed ref with `git switch --detach -- <ref>` at `src/plugins/git-install.ts:323`. In a fresh clone, Git has only `remotes/origin/feature/foo`, so `git switch --detach -- feature/foo` fails with `fatal: '--detach' cannot be used with '-b/-B/--orphan'`. I reproduced that exact Git command against a local repo with a `feature/foo` branch, and `parseGitPluginSpec("git:owner/repo@feature/foo")` now returns `ref: "feature/foo"`.
- Impact: `openclaw plugins install git:owner/repo@feature/foo` and `/plugins install git:owner/repo@feature/foo` now pass git-spec validation and route into plugin git install, but fail during checkout. Plugin docs already describe `@<ref>` as supporting branches, tags, and commits (`docs/cli/plugins.md:182`), and slash-containing branch names are common. The new skill path fixed this by resolving `ref` or `origin/ref` before checkout, but the plugin path was left behind.
- Suggested fix: Share the new ref-resolution logic with plugin git installs, or add the same `ref` then `origin/<ref>` `rev-parse --verify <candidate>^{commit}` step before `git switch --detach <commit>`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A local repo with a remote-only `feature/foo` branch reproduces the exact failing checkout command used by `installPluginFromGitSpec`; the parser probe confirms this commit now routes that spec into the plugin git path.
- Is this the best way to solve the issue? yes. The skill installer already implements the right shape by resolving to a commit before checkout; using that same behavior for plugins avoids branch/tag ambiguity and keeps recorded commit metadata accurate.
