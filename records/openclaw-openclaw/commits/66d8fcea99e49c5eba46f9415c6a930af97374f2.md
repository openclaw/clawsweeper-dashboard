# Commit 66d8fcea

## Summary

Found one security regression in the new `release-ci/*` workflow-ref allowlist. The release checks workflow passes many live/provider secrets to child jobs, and this commit allows it to run from a temporary branch namespace that is not covered by the repo's active branch rulesets.

## Findings

### High: Secret-bearing release checks can run from unprotected `release-ci/*` refs

- Kind: security
- File: `.github/workflows/openclaw-release-checks.yml`
- Line: 92
- Evidence: The new guard allows `refs/heads/release-ci/[0-9a-f]{12}-[0-9]+`. The same workflow later forwards release/live secrets such as `OPENAI_API_KEY`, `ANTHROPIC_API_KEY`, browser/live setup tokens, Codex/Claude credential JSON, and QA Convex secrets to child workflows at `.github/workflows/openclaw-release-checks.yml:335` and `.github/workflows/openclaw-release-checks.yml:361`. `Full Release Validation` dispatches this child by branch name via `gh workflow run ... --ref "$CHILD_WORKFLOW_REF"` while passing `expected_sha`, and the helper creates `release-ci/<short-sha>-<timestamp>` branches with `git push origin <sha>:refs/heads/<branch>`. GitHub documents that manual workflows require write access and that `gh workflow run --ref` runs the workflow file version from the specified branch or tag. ([docs.github.com](https://docs.github.com/en/actions/using-workflows/manually-running-a-workflow)) Read-only GitHub ruleset inspection showed active protections for `main`, `stable/**/*`, `release/**`, and `releases/**`, but no active rule covering `refs/heads/release-ci/**`; a remote `origin/release-ci/66d8fcea99e4-1777696193155` branch currently exists.
- Impact: Any actor or automation able to push to unprotected repository branches can create or move a matching `release-ci/<12hex>-<digits>` ref and manually dispatch `OpenClaw Release Checks` from that ref. Because the workflow definition itself is selected by `--ref`, a malicious or moved temporary branch can execute altered workflow code with the release workflow's secret surface before any parent verifier notices a `headSha` mismatch.
- Suggested fix: Do not allow secret-bearing release workflows from a mutable unprotected namespace. Either dispatch release checks from `main` or a protected release ref and pass the target SHA as data, or add an active ruleset for `refs/heads/release-ci/**` that restricts creation and updates to trusted release automation/admin bypass only and prevents non-fast-forward moves during runs. A runtime check inside the workflow is not enough if the workflow file can be changed on the selected ref.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes in principle: create or move a same-repo branch matching `release-ci/<12hex>-<digits>` with a modified `openclaw-release-checks.yml`, then run `gh workflow run openclaw-release-checks.yml --ref <that-branch> -f ref=<sha>`. I did not mutate the repository to prove this live, but the local regex probe accepted the branch shape and GitHub's documented `--ref` behavior plus current ruleset inspection establish the path.
- Is this the best way to solve the issue? The safest direction is to avoid unprotected branch workflow refs for secrets. If exact child `headSha` proof requires temporary refs, the branch namespace needs protection equivalent to release branches before this allowlist is safe.
