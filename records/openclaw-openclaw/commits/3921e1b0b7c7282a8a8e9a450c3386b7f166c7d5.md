# Commit 3921e1b

## Summary

Found one Windows timeout error-handling regression in the new `taskkill` path.

## Findings

### Medium: Windows timeout kill can crash when `taskkill` cannot be spawned

- Kind: reliability
- File: `src/process/exec.ts`
- Line: 338
- Evidence: the timeout handler now calls `spawn("taskkill", ...)` and immediately returns. The returned child is discarded and has no `error` listener, so asynchronous spawn failures are not caught by the surrounding `try/catch`. A focused Node smoke check confirmed that spawning a missing command without an `error` listener exits with an unhandled `error` event. The added test at `src/process/exec.windows.test.ts:373` only covers the successful secondary spawn path.
- Impact: on Windows, if `taskkill` is unavailable from the gateway process environment, blocked by policy, or otherwise fails during spawn, any timed-out `runCommandWithTimeout` call can crash the process instead of falling back to `child.kill("SIGKILL")`. The original timed-out command may also remain running because the fallback is skipped after the attempted `taskkill` spawn.
- Suggested fix: keep the spawned `taskkill` child, attach an `error` listener that swallows/logs the failure and falls back to direct `child.kill("SIGKILL")`; consider also handling non-zero `taskkill` close if direct fallback is useful there. Add a Windows-path unit test where the second spawn emits `error`.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the Node failure mode: spawning a missing command without an `error` listener crashes with an unhandled `error` event. The exact OpenClaw path needs a Windows or mocked test where the second spawn fails.
- Is this the best way to solve the issue? Yes: the new tree-kill behavior should stay, but the secondary kill helper needs its own error listener and fallback.
