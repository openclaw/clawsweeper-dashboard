# Commit 3921e1b

## Summary

Found one actionable Windows timeout-path issue: the new `taskkill` fallback is launched as an unqualified executable and has no `error`/nonzero-exit handling before skipping the previous direct `child.kill("SIGKILL")` fallback.

## Findings

### Medium: Windows timeout path can run the wrong `taskkill` or crash on spawn failure

- Kind: security
- File: `src/process/exec.ts`
- Line: 338
- Evidence: `killChild()` now does `spawn("taskkill", ["/PID", String(child.pid), "/T", "/F"], ...)` and immediately `return`s, so direct `child.kill("SIGKILL")` is skipped after a successful `spawn()` call. The returned `ChildProcess` has no `error` or `exit` listener. Node documents that failed child spawns emit an `error` event, and Microsoft documents that unqualified Windows executable lookup can search the application/current directory before System32. The repo already avoids that pattern in nearby Windows process code by resolving `SystemRoot\\System32\\taskkill.exe` in `src/infra/restart-stale-pids.ts:475` and `src/daemon/schtasks.ts:493`. ([nodejs.org](https://nodejs.org/download/release/v22.12.0/docs/api/child_process.html))
- Impact: If a Windows OpenClaw process times out while a malicious or accidental `taskkill.exe` is earlier in the process search path, OpenClaw can execute that binary under the user’s account. If `taskkill` cannot be spawned, the unhandled `error` event can crash the OpenClaw process instead of falling back to direct child termination.
- Suggested fix: Resolve the absolute taskkill path via the existing Windows install-root helper, and attach `error` plus nonzero `exit` fallback handlers that call `child.kill("SIGKILL")` when the tree kill cannot be started or fails. Add tests for taskkill `error` and nonzero exit.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the unhandled spawn-failure behavior: `node -e "const { spawn } = require('node:child_process'); spawn('definitely-missing-openclaw-taskkill-probe-xyz'); setTimeout(() => console.log('survived'), 50);"` crashes with an unhandled `error` event on Node 24. The exact Windows search-path hijack was not live-run here because this runner is Linux.
- Is this the best way to solve the issue? No. The intent is right, but this should use an absolute Windows system binary path and preserve a direct-kill fallback for failure cases.
