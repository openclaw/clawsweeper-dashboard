# Commit d609859

## Summary

Found one Windows harness regression in the extracted background PowerShell runner.

## Findings

### Low: Windows background runner breaks when the temp script path contains spaces

- Kind: compatibility
- File: `scripts/e2e/parallels/guest-transports.ts`
- Line: 117
- Evidence: The new shared launcher passes `$scriptPath` as an unquoted element in `Start-Process ... -ArgumentList @(..., '-File', $scriptPath)`. `Start-Process` joins `ArgumentList` array entries with spaces and does not preserve grouping/quotes; Microsoft documents that callers should include needed quote characters themselves. ([learn.microsoft.com](https://learn.microsoft.com/en-us/powershell/module/microsoft.powershell.management/start-process?view=powershell-5.1&utm_source=openai)) I also reproduced this locally with PowerShell 7.4.14: launching a script under a temp directory containing spaces caused the child `pwsh -File` to see only the first path segment and fail. This is a regression for `scripts/e2e/parallels/npm-update-smoke.ts:593`, because the prior Windows npm-update launcher used `cmd.exe start ... -File "%TEMP%\\${fileBase}.ps1"`, which quoted the temp path.
- Impact: Windows npm-update smoke can fail to launch the detached update script on VMs whose `%TEMP%` path contains spaces, such as user profiles with spaced display/user names. The harness then retries until materialization fails or times out, without testing the update path.
- Suggested fix: Quote `$scriptPath` inside a single `-ArgumentList` string, for example `"-NoProfile -ExecutionPolicy Bypass -File `"$scriptPath`""`, or restore the previous quoted `cmd.exe start ... -File "%TEMP%\\...ps1"` launch shape for this helper.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Create a PowerShell script under a directory with a space and launch it with `Start-Process -FilePath pwsh -ArgumentList @('-NoProfile', '-File', $scriptPath)`: the child receives the split path and fails before running the script.
- Is this the best way to solve the issue? yes. The failure is in argument quoting at the launch boundary; quoting the script path in the command line is narrower and safer than changing polling or retry behavior.
