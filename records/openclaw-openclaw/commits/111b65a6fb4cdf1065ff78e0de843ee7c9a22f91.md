# Commit 111b65a

## Summary

Found two status-schema regressions in the new strict video response validation. Both reject documented non-terminal provider statuses, so normal async video jobs can fail immediately instead of polling to completion.

## Findings

### Medium: OpenRouter polling rejects documented `in_progress` jobs

- Kind: regression
- File: `extensions/openrouter/video-generation-provider.ts`
- Line: 312
- Evidence: `pollOpenRouterVideo` now treats any status outside `queued`, `pending`, `processing`, `running`, `completed`, and terminal failures as malformed. OpenRouter’s video docs list `pending`, `in_progress`, `completed`, and `failed`; `in_progress` is the documented “video is being generated” poll state. ([openrouter.ai](https://openrouter.ai/docs/guides/overview/multimodal/video-generation?utm_source=openai))
- Impact: An OpenRouter video job that enters `in_progress` will throw `OpenRouter video generation response malformed` instead of continuing to poll, breaking valid long-running video generations.
- Suggested fix: Accept `in_progress` as a non-terminal status, preferably through a single normalized status set shared by submit and poll validation.
- Confidence: high

### Medium: xAI polling rejects documented `pending` jobs

- Kind: regression
- File: `extensions/xai/video-generation-provider.ts`
- Line: 95
- Evidence: `readXaiStatusResponse` only accepts `queued`, `processing`, `done`, `failed`, and `expired`. xAI’s REST video docs say poll responses use `pending`, `done`, `expired`, and `failed`, with `pending` meaning the video is still being generated. ([docs.x.ai](https://docs.x.ai/docs/guides/video-generations?utm_source=openai))
- Impact: xAI video jobs that return the normal `pending` state will be reported as malformed before polling can continue.
- Suggested fix: Accept `pending` as the non-terminal xAI status; consider normalizing status strings to lowercase before validation.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Mock an OpenRouter poll response with `status: "in_progress"` after a valid submit response, or mock an xAI poll response with `status: "pending"` after a valid create response; current `main` rejects both before the next poll.
- Is this the best way to solve the issue? yes. The hardening intent is sound, but the allowlists need to match each provider’s documented async lifecycle states.
