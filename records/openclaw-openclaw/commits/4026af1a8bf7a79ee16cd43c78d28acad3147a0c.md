# Commit 4026af1

## Summary

Found one low-severity regression path that still appears present on current `main`. A later commit fixed the bundled-plugin variant of this behavior, but non-bundled plugins that are already discovered from explicit/local sources can still be reinstalled unnecessarily.

## Findings

### Low: Doctor may install a plugin that is already discovered but has no install record

- Kind: regression
- File: `src/commands/doctor/shared/missing-configured-plugin-install.ts`
- Line: 597
- Evidence: `collectDownloadableInstallCandidates` still receives all configured plugin/channel ids, not only `missingPluginIds`, so a configured catalog candidate can be returned even when `knownIds` already contains the plugin. The reviewed commit changed the skip guard to continue only when `knownIds.has(candidate.pluginId) && hasUsableRecord`; with no persisted install record, `hasUsableRecord` is false and the code falls through to `installCandidate`. This affects plugins already discovered from `plugins.load.paths` or other non-recorded local/config sources.
- Impact: `openclaw doctor --fix` can call npm/ClawHub installation for a plugin that is not missing. At best this adds noisy network failures during repair; at worst it writes a managed install record for a plugin the operator intended to load from an explicit local path, creating source-plane drift for later diagnostics/update flows.
- Suggested fix: Preserve the old “known means not missing” guard unless the plugin is in `missingPluginIds` because an existing record is unusable. For example, skip when `knownIds.has(candidate.pluginId) && !missingPluginIds.has(candidate.pluginId)`, then separately allow the missing-record reinstall path.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes at unit level: use the existing mocked test shape with `snapshot.plugins` containing a non-bundled `matrix` plugin, empty install records, a downloadable `matrix` catalog entry, and config enabling the `matrix` channel. The current guard reaches `installPluginFromNpmSpec` even though the plugin is already known. I did not add this test because this review is report-only.
- Is this the best way to solve the issue? Yes, the safest fix is to make the fallback install path conditional on actual missingness, not just lack of a persisted record.
