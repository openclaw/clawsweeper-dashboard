# Commit f072835

## Summary

Found one low-severity test regression in the matcher cleanup. The runtime setup wizard code on current `main` still passes the expected arguments; the issue is that two rewritten assertions no longer fail if required mock-call arguments become `undefined`.

## Findings

### Low: Replacing `expect.anything()` with `not.toBeNull()` allows missing arguments

- Kind: test_gap
- File: `src/wizard/setup.test.ts`
- Line: 698
- Evidence: `getMockCallArg` returns `call[argIndex]` without checking whether that argument exists (`src/wizard/setup.test.ts:180`), and the new assertions at `src/wizard/setup.test.ts:698`, `src/wizard/setup.test.ts:699`, `src/wizard/setup.test.ts:700`, `src/wizard/setup.test.ts:767`, and `src/wizard/setup.test.ts:768` only use `.not.toBeNull()`. `undefined` passes that matcher. The previous `expect.anything()` assertions rejected both `null` and `undefined`.
- Impact: These tests would now pass if `runSetupWizard` accidentally called `setupChannels` without `cfg`, `runtime`, or `prompter`, or called `warnIfModelConfigLooksOff` without `nextConfig` or `prompter`. Those are concrete required call arguments: `setupChannels(cfg, runtime, prompter, options?)` and `warnIfModelConfigLooksOff(config, prompter, options?)`.
- Suggested fix: Make `getMockCallArg` throw when `argIndex` is not present in the call, or use assertions that reject both `null` and `undefined` for these required arguments.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. `expect(undefined).not.toBeNull()` passes, while the previous `expect.anything()` would fail for `undefined`.
- Is this the best way to solve the issue? yes. Tightening `getMockCallArg` is the lowest-friction fix because all call-argument assertions would then fail on missing arguments.
