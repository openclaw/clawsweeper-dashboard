# Commit e7dafaf2af

## Summary

Found two actionable issues in the new `Crabbox Hydrate` workflow: one security issue from direct `workflow_dispatch` input interpolation into Bash after provider secrets are hydrated, and one reliability issue shown by the live hydrate run for this SHA exiting as a failed job.

## Findings

### Medium: Dispatch inputs can execute shell before the keep-alive step

- Kind: security
- File: `.github/workflows/crabbox-hydrate.yml`
- Line: 85
- Evidence: The workflow hydrates provider secrets into the runner at lines 54-78 via `scripts/ci-hydrate-testbox-env.sh`, which writes a profile under `$HOME` by default. Later steps interpolate `${{ inputs.crabbox_id }}` and `${{ inputs.crabbox_keep_alive_minutes }}` directly into Bash at lines 85, 98, and 102. GitHub documents that expressions inside `${{ }}` are substituted before the temporary shell script runs, which can turn attacker-controlled context strings into shell command injection. ([docs.github.com](https://docs.github.com/en/actions/concepts/security/script-injections?utm_source=openai)) A focused local proof showed that a `crabbox_id` shaped like `$(printf injected > "$proof")` executes during `state="...${{ inputs.crabbox_id }}.env"` even though the assignment is double-quoted.
- Impact: Anyone or any integration able to dispatch this workflow with chosen inputs can run arbitrary shell in the self-hosted Crabbox job after model provider credentials have been materialized in the runner home. That is enough to read or exfiltrate the hydrated profile and helper files.
- Suggested fix: Validate inputs in an early step before hydrating secrets, pass them through step `env` variables, and only use shell variables after strict allowlist checks. For example, allow `crabbox_id`/runner labels to match a narrow identifier pattern and parse `crabbox_keep_alive_minutes` from `$CRABBOX_KEEP_ALIVE_MINUTES`, not from direct `${{ inputs.* }}` interpolation inside `run`.
- Confidence: high

### Low: The hydrate job reports failure when the runner is terminated

- Kind: reliability
- File: `.github/workflows/crabbox-hydrate.yml`
- Line: 98
- Evidence: The keep-alive step loops until the deadline or a stop file appears, but has no `TERM` handling. The actual `Crabbox Hydrate` workflow run for this commit, run `25198902816`, reached the keep-alive step with `minutes="45"` and then failed after about two minutes with `Process completed with exit code 143`.
- Impact: Crabbox hydration can appear failed in GitHub Actions even after the runner was prepared and marked ready. If Crabbox or downstream automation keys off the workflow conclusion, it may treat successful hydration as a failed lease; even if not, this creates persistent false failures for manual hydrate runs.
- Suggested fix: Align the shutdown contract with Crabbox: either ensure Crabbox writes the expected `.stop` file before stopping the runner and waits for the loop to exit, or make the keep-alive step handle the intended termination signal explicitly without masking real setup failures.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes for the input injection: dispatch the workflow with a `crabbox_id` containing shell command substitution, or reproduce locally by placing that text in the same quoted Bash assignment. yes for the failed hydrate conclusion: the live `Crabbox Hydrate` run for this SHA already failed with exit 143 in the keep-alive step.
- Is this the best way to solve the issue? yes for the injection fix direction: environment variables plus strict validation are the standard safe pattern here. unclear for the exit-143 behavior until Crabbox’s runner shutdown contract is confirmed.
