# Commit 7330778

## Summary

Found one low-severity test reliability regression. Runtime behavior is unchanged, but the tightened assertion now rejects a valid built-tree resolver output.

## Findings

### Low: ACPX config test assumes source MCP args even when built MCP output exists

- Kind: reliability
- File: `extensions/acpx/src/config.test.ts`
- Line: 165
- Evidence: The test now requires `openclaw-plugin-tools` args to equal the source-mode `["--import", tsx, "src/mcp/plugin-tools-serve.ts"]`. The implementation in `extensions/acpx/src/config.ts:152` intentionally returns `[dist/mcp/plugin-tools-serve.js]` whenever that built file exists, and `tsdown.config.ts:242` declares `mcp/plugin-tools-serve` as a build entry reached by `pnpm build` through `scripts/build-all.mjs`. A focused temp-layout probe confirmed `resolveAcpxPluginConfig({ pluginToolsMcpBridge: true })` returns the built `dist/mcp/plugin-tools-serve.js` arg when that file is present.
- Impact: `extensions/acpx/src/config.test.ts` passes in a clean source tree but can fail after a build in the same worktree, which is a common local and CI/debug sequence for this repo.
- Suggested fix: Keep the tighter command assertion, but make the args assertion cover both valid resolver modes, or inject a `moduleUrl`/fixture layout that deterministically exercises source mode without depending on ambient repo `dist/` state.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Create or build the `dist/mcp/plugin-tools-serve.js` entry, then run `node scripts/run-vitest.mjs run --config test/vitest/vitest.extension-acpx.config.ts extensions/acpx/src/config.test.ts`; the resolver will return built args while the test expects source args.
- Is this the best way to solve the issue? yes. The resolver’s source-vs-dist branch is intentional, so the test should either assert the branch explicitly with controlled fixtures or accept both valid outputs.
