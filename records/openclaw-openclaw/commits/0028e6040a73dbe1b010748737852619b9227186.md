# Commit 0028e6040a

## Summary

Found one concrete regression in a second caller of the shared Feishu queue helper. The message starvation fix is plausible for normal Feishu messages, but the new default timeout also changes Drive comment notice acknowledgement semantics.

## Findings

### Medium: Queue timeout can mark Drive comment notices processed before handling completes

- Kind: reliability
- File: `extensions/feishu/src/sequential-queue.ts`
- Line: 41
- Evidence: `createSequentialQueue()` now defaults `taskTimeoutMs` to 5 minutes and `boundedRun()` resolves the returned queue promise on timeout. `extensions/feishu/src/monitor.comment-notice-handler.ts:34` uses `createSequentialQueue()` with no options, then records the synthetic Drive comment event as processed after `await enqueue(...)` at `extensions/feishu/src/monitor.comment-notice-handler.ts:80`. Existing tests encode the intended retry behavior: retryable comment failures must release without recording at `extensions/feishu/src/monitor.comment.test.ts:914`.
- Impact: a Drive comment handler that runs longer than 5 minutes is treated as successfully finished by the caller. The event is recorded processed and replay is closed while `handleFeishuCommentEvent()` continues detached. If the gateway exits before it finishes, or if the detached task later fails with `FeishuRetryableSyntheticEventError`, Feishu retry is suppressed and the comment notice can be lost.
- Suggested fix: opt the comment notice queue into legacy behavior with `createSequentialQueue({ taskTimeoutMs: 0 })`, or change the queue API to report timeout separately so callers that perform durable acknowledgement only record completion after the task actually settles.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. In the comment notice path, hold `handleFeishuCommentEvent()` past the queue timeout and observe `await enqueue(...)` resolve, followed by `recordProcessedFeishuMessage(...)`, before the held task settles.
- Is this the best way to solve the issue? Yes for the narrow regression: Drive comment notices have explicit retryable-vs-non-retryable acknowledgement logic, so they should not inherit a timeout that resolves as success unless that caller is changed to handle timeout as a separate state.
