# Commit e730e9b

## Summary

Found one actionable regression in the new iOS realtime relay path: push-to-talk does not suspend the realtime microphone relay before starting native PTT capture.

## Findings

### Medium: Push-to-talk leaves realtime relay capture running

- Kind: bug
- File: `apps/ios/Sources/Voice/TalkModeManager.swift`
- Line: 385
- Evidence: `startRealtimeRelay()` stores an active `realtimeRelaySession` whose `RealtimeTalkRelaySession.startMicrophonePump()` installs its own microphone tap and starts its own `AVAudioEngine` (`apps/ios/Sources/Voice/TalkModeManager.swift:243`, `apps/ios/Sources/Voice/RealtimeTalkRelaySession.swift:441`). When PTT starts, `beginPushToTalk()` only calls `stopSpeaking()` and `stopRecognition()` before starting native speech recognition (`apps/ios/Sources/Voice/TalkModeManager.swift:385`, `apps/ios/Sources/Voice/TalkModeManager.swift:393`, `apps/ios/Sources/Voice/TalkModeManager.swift:424`). `stopRecognition()` does not stop the realtime relay session.
- Impact: If Talk is running in realtime relay mode and a gateway/watch PTT command starts, iOS keeps streaming microphone audio to the realtime session while also starting native PTT recognition. That can duplicate the user request, race two voice flows, or fail native capture because another audio engine tap is still active.
- Suggested fix: In `beginPushToTalk()`, after recording `resumeContinuousAfterPTT`, stop the realtime relay before starting native recognition. If PTT setup fails, either restore the prior realtime session when `resumeContinuousAfterPTT` is true or fully reset the state so the existing relay is not left running unexpectedly.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes from the code path: enable realtime relay Talk, start continuous Talk, then invoke `pttStart` or `pttOnce` through `NodeAppModel.handleTalkInvoke()` (`apps/ios/Sources/Model/NodeAppModel.swift:1470`). The realtime relay microphone pump remains active while PTT starts.
- Is this the best way to solve the issue? Yes. PTT already suspends native continuous recognition before taking over the microphone; realtime relay needs the same ownership transition.
