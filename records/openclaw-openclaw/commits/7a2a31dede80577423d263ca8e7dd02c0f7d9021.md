# Commit 7a2a31d

## Summary

Found one concrete regression in the new OAuth-backed OpenAI realtime transcription path. The focused repo tests pass, but they assert the same mismatched request shape, so they do not catch the external API contract issue.

## Findings

### Medium: OAuth realtime transcription mints client secrets with the wrong request body shape

- Kind: bug
- File: `extensions/openai/realtime-transcription-provider.ts`
- Line: 113
- Evidence: `resolveOpenAIRealtimeTranscriptionAuthorization()` calls `createOpenAIRealtimeTranscriptionClientSecret()` with `buildOpenAIRealtimeTranscriptionSessionCreateConfig(config)`, which returns a GA-style nested body: `{ type: "transcription", audio: { input: { format, transcription, turn_detection } } }`. The shared helper then posts that object directly to `https://api.openai.com/v1/realtime/transcription_sessions` at `extensions/openai/realtime-provider-shared.ts:159-163`. OpenAI’s current `Create transcription session` API for `/v1/realtime/transcription_sessions` documents the request body as the transcription-session fields directly, such as `input_audio_format`, `input_audio_transcription`, `input_audio_noise_reduction`, `turn_detection`, and `include`, and says it is configured with the same parameters as `transcription_session.update`. ([platform.openai.com](https://platform.openai.com/docs/api-reference/realtime-beta-sessions/transcription_session_object?lang=python&utm_source=openai))
- Impact: OAuth-only setups that rely on OpenAI realtime transcription, including Voice Call streaming and Google Meet transcription paths that this commit now wires through `openai-codex`, can fail before the WebSocket opens with an OpenAI request validation error, or mint a default transcription session that does not match the intended G.711/VAD/model settings. Direct API-key transcription still sends a WebSocket update later, but the new OAuth client-secret minting path is the headline behavior added here and uses the incompatible REST payload.
- Suggested fix: Build the `/v1/realtime/transcription_sessions` body with the documented flat transcription-session fields: `input_audio_format: "g711_ulaw"`, `input_audio_transcription: { model, language?, prompt? }`, and `turn_detection: { type: "server_vad", threshold, prefix_padding_ms, silence_duration_ms }`. Keep the nested `{ type: "transcription", audio: ... }` shape only if using the newer `/v1/realtime/client_secrets` endpoint with a `session` object that explicitly supports transcription session configurations.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. Configure only an `openai-codex` OAuth profile, no direct `OPENAI_API_KEY`, then start a realtime transcription session through Voice Call streaming or Google Meet with the OpenAI transcription provider. The failing outbound request is the POST body generated before WebSocket creation. I could not live-hit OpenAI because this checkout has no usable OpenAI key/profile in the environment.
- Is this the best way to solve the issue? Yes. The provider already has a legacy-compatible flat builder for `transcription_session.update`; the OAuth REST minting path should use the same contract shape for `/realtime/transcription_sessions`, or switch endpoints deliberately.
