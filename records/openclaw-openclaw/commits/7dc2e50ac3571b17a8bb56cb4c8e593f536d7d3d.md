# Commit 7dc2e50

## Summary

Found one concrete regression risk: Mattermost and Microsoft Teams now use the abort-aware detector for command authorization/mention gates, not only for debounce bypass. That can drop normal group/channel messages like `wait` from senders who are allowed to chat but are not command-authorized.

## Findings

### Medium: bare abort words can be dropped as unauthorized commands in Mattermost and Teams

- Kind: regression
- File: `extensions/mattermost/src/mattermost/monitor.ts`
- Line: 1322
- Evidence: `isControlCommandMessage()` returns true for abort trigger words including `wait`, `stop`, and `abort` (`src/auto-reply/command-detection.ts:54`, `src/auto-reply/reply/abort-primitives.ts:5`). This commit passes that broader value into Mattermost command access (`extensions/mattermost/src/mattermost/monitor.ts:1322`) and drops group messages when `commandAccess.shouldBlockControlCommand` is true (`extensions/mattermost/src/mattermost/monitor.ts:1394`). Teams has the same pattern at `extensions/msteams/src/monitor-handler/message-handler.ts:289` and drops at `extensions/msteams/src/monitor-handler/message-handler.ts:424`.
- Impact: In an allowed group/channel conversation where normal chat is permitted but command authorization is not granted, standalone words such as `wait` used to be normal text because `hasControlCommand("wait")` is false. After this commit, they become control commands; `resolveControlCommandGate()` blocks them when unauthorized (`src/channels/command-gating.ts:43`). That is message loss for common conversational text.
- Suggested fix: keep `isControlCommandMessage()` in debounce predicates (`extensions/mattermost/src/mattermost/monitor.ts:2093`, `extensions/msteams/src/monitor-handler/message-handler.ts:947`), but keep auth/mention-gating inputs slash-command scoped with `hasControlCommand()`, or add a separate abort-aware path that does not turn unauthorized bare abort words into dropped commands.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A source probe showed `hasControlCommand("wait") === false`, `isControlCommandMessage("wait") === true`, and flipping `hasControlCommand` from false to true changes `resolveControlCommandGate()` from `{ shouldBlock: false }` to `{ shouldBlock: true }` with no configured command authorizer.
- Is this the best way to solve the issue? yes, directionally. The original fix should stay scoped to debounce bypass; command auth and mention gates need either slash-only detection or a separate bare-abort authorization decision.
