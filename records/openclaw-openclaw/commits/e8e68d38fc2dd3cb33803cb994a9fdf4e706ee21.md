# Commit e8e68d38fc

## Summary

Found one current release-validation coverage regression. The reviewed commit added `OPENCLAW_INSTALL_E2E_AGENT_TOOL_SMOKE=0` to the OpenAI installer lane; that variable is now honored on current `main`, so the package install lane exits before running any OpenAI agent/tool turn.

## Findings

### Medium: OpenAI install lane skips its package-installed agent proof

- Kind: regression
- File: `scripts/lib/docker-e2e-scenarios.mjs`
- Line: 535
- Evidence: The lane command sets `OPENCLAW_INSTALL_E2E_AGENT_TOOL_SMOKE=0`. Current `scripts/test-install-sh-e2e-docker.sh:31` passes that env into Docker, and `scripts/docker/install-sh-e2e/run.sh:677` returns immediately after gateway health when it is `0`, before the `openclaw agent` turns and transcript tool assertions at `scripts/docker/install-sh-e2e/run.sh:701`. The beta `package-update-openai` plan contains this lane plus `codex-on-demand`, and the release workflow includes that chunk for beta/minimum/stable/full at `.github/workflows/openclaw-live-and-e2e-checks-reusable.yml:620`.
- Impact: The release-path OpenAI installer lane can pass after install, onboard, and gateway health without proving the published package can run an OpenAI agent turn or use the expected tools. A regression in packaged OpenAI runtime wiring, provider auth behavior after onboard, model invocation, or agent tool plumbing could pass this release gate.
- Suggested fix: Keep the parallelization, but do not disable the agent smoke for the release OpenAI install lane. If runtime is the concern, split out a shorter minimal OpenAI agent proof or skip only the slow image turn instead of skipping all agent/tool turns.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the control-flow regression: the focused plan command shows the lane command with `OPENCLAW_INSTALL_E2E_AGENT_TOOL_SMOKE=0`, and the current runner exits before agent turns when that env is set. I did not run the live Docker/OpenAI lane.
- Is this the best way to solve the issue? Unclear; the safest direction is to preserve at least one package-installed OpenAI agent turn in this release chunk, but maintainers may prefer a separate shorter lane if the full tool smoke is too slow.
