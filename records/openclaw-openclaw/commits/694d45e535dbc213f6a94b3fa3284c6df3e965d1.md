# Commit 694d45e

## Summary

One actionable regression found in the new static image compression metadata shortcut.

## Findings

### Medium: Partial static pixel metadata can skip Anthropic runtime side limits

- Kind: regression
- File: `src/agents/tools/image-tool.ts`
- Line: 409
- Evidence: `imageCompressionPolicyHasDimensionLimit()` treats `maxPixels` alone as a complete dimension policy, so `resolveCompressionModelPolicy()` returns the static/configured policy before calling runtime augment hooks. For an Anthropic dated Sonnet config with only `mediaInput.image.maxPixels`, the current policy resolves to `{"maxPixels":10000000}`. Calling `resolveModelAsync()` with provider runtime hooks for the same model resolves `{"maxSidePx":1568,"preferredSidePx":1568,"tokenMode":"provider","maxPixels":10000000}`.
- Impact: `src/media/web-media.ts:395` converts `maxPixels` to a square long-side budget. With only `maxPixels: 10000000`, the balanced compression ladder starts at `2048`; with the Anthropic runtime side limit merged, it starts at `1568`. Configured dated Anthropic variants can therefore send images larger than the provider-owned side limit, causing avoidable provider rejection or token/cost regressions.
- Suggested fix: Do not treat `maxPixels` alone as sufficient for runtime-augment providers. Merge runtime metadata whenever static/configured metadata is partial, or at least require the provider-owned hard side limit (`maxSidePx`) before skipping Anthropic runtime augment.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused `node --import tsx -e` probe against `testing.resolveImageCompressionPolicy()` reproduced the missing side limit, and a matching `resolveModelAsync()` probe showed runtime hooks still provide it.
- Is this the best way to solve the issue? yes. The bug is in the completeness predicate for skipping runtime augment, not in the manifest rows themselves.
