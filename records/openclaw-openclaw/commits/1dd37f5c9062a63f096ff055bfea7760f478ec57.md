# Commit 1dd37f5

## Summary

Found three concrete regressions from bridging `src/memory-host-sdk/host/*` to the package-owned implementations. Two are already caught by unchanged focused tests.

## Findings

### Medium: Default QMD scope now denies channel sessions

- Kind: regression
- File: `src/memory-host-sdk/host/backend-config.ts`
- Line: 1
- Evidence: The bridge now exports `packages/memory-host-sdk/src/host/backend-config.js`. The parent `src` implementation allowed both direct and channel sessions in `DEFAULT_QMD_SCOPE`; the package implementation only allows `{ chatType: "direct" }` at `packages/memory-host-sdk/src/host/backend-config.ts:106`. The unchanged test at `src/memory-host-sdk/host/backend-config.test.ts:87` expects channel sessions to be allowed, and `pnpm test src/memory-host-sdk/host/backend-config.test.ts` fails at line 98 with `expected false to be true`.
- Impact: Users with `memory.backend = "qmd"` lose QMD memory in channel sessions by default, even though the prior `src` host contract allowed those sessions.
- Suggested fix: Preserve the previous `src` default in the package bridge by adding the channel allow rule, or intentionally update the contract/tests/docs if channel denial is desired.
- Confidence: high

### Low: `memory_get` can no longer read documented top-level `DREAMS.md`

- Kind: regression
- File: `src/memory-host-sdk/host/internal.ts`
- Line: 1
- Evidence: The bridge now uses package `isMemoryPath`, which accepts `dreams.md` but not `DREAMS.md` at `packages/memory-host-sdk/src/host/internal.ts:91`. The parent implementation accepted `DREAMS.md`, docs list `DREAMS.md` as a memory file, and the unchanged test `src/memory-host-sdk/host/internal.test.ts:268` fails with `expected false to be true`.
- Impact: `readMemoryFile` checks `isMemoryPath` before reading workspace files, so the `memory_get` tool rejects the documented Dream Diary path unless it is lowercase or explicitly configured as an extra path.
- Suggested fix: Accept both `DREAMS.md` and `dreams.md` in the package implementation, matching the documented surface and previous `src` behavior.
- Confidence: high

### Medium: Agent memory-read context limits stop inheriting defaults

- Kind: regression
- File: `src/memory-host-sdk/host/read-file.ts`
- Line: 1
- Evidence: The bridged `readAgentMemoryFile` uses package `resolveAgentContextLimits`, then passes `memoryGetDefaultLines` and `memoryGetMaxChars` into `readMemoryFile` at `packages/memory-host-sdk/src/host/read-file.ts:100`. The core helper merged defaults with agent overrides (`src/agents/agent-scope-config.ts:128`), but the package helper returns the agent object directly (`packages/memory-host-sdk/src/host/config-utils.ts:303`). A focused `tsx` comparison showed core returns `{"memoryGetMaxChars":1000,"memoryGetDefaultLines":5}` while the package helper returns only `{"memoryGetDefaultLines":5}` for defaults plus a partial agent override.
- Impact: An agent that overrides only one context limit silently drops the configured default `memoryGetMaxChars`, falling back to the built-in 12,000-character read cap and potentially returning more memory content than configured.
- Suggested fix: Make package `resolveAgentContextLimits` merge `{ ...defaults, ...agent.contextLimits }`, matching core behavior, and add a bridge/contract test for partial overrides.
- Confidence: high
