# Commit b6cbd92

## Summary

Found one allowlist regression in the new cold memory-runtime fallback. `plugins.allow` is documented as exclusive, but the new standalone doctor/status path can load the implicit default `memory-core` plugin even when the operator set a restrictive allowlist that does not include it.

## Findings

### Medium: Cold doctor/status memory fallback bypasses `plugins.allow` for the implicit default memory plugin

- Kind: security
- File: `src/plugins/memory-runtime.ts`
- Line: 36
- Evidence: `resolveMemoryRuntimePluginIds()` uses `normalizePluginsConfig(config.plugins).slots.memory`, which fills an absent memory slot with the implicit default `"memory-core"`, then only checks global disable, deny, and `entries[pluginId].enabled === false`. It does not check whether the slot was explicitly configured or whether a non-empty `plugins.allow` includes the plugin before `ensureStandaloneRuntimePluginRegistryLoaded()` loads it. A focused smoke check reproduced this with `cfg = { plugins: { allow: ["browser"] } }`: before the call `getMemoryRuntime()` was false, `resolveActiveMemoryBackendConfig({ cfg, agentId: "main" })` returned the builtin memory backend, and after the call `getMemoryRuntime()` was true.
- Impact: Controlled deployments that use `plugins.allow` to restrict the approved plugin inventory can still execute the bundled memory plugin in standalone `doctor` / `status` memory resolution when the memory slot is only implicit. This conflicts with the documented contract that a non-empty `plugins.allow` means only listed plugins are eligible to load, with only explicitly configured bundled chat channels called out as an exception.
- Suggested fix: Treat the implicit default memory slot differently from an explicit `plugins.slots.memory`. If `plugins.allow` is non-empty and the raw config did not explicitly select the memory slot, return no memory plugin unless the allowlist contains the default plugin. Keep explicit `plugins.slots.memory = "memory-core"` as the existing selected-slot behavior if that bypass is intentional, and consider normalizing explicit slot IDs through the same registry-aware path used by startup filtering.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Run a cold-process check that clears plugin loader state, sets `cfg.plugins.allow = ["browser"]` with no explicit `plugins.slots.memory`, then calls `resolveActiveMemoryBackendConfig({ cfg, agentId: "main" })`; the memory runtime is loaded despite not being allowlisted.
- Is this the best way to solve the issue? yes. The bug is in the new memory-runtime bootstrap deciding that the implicit default slot is enough to perform a scoped runtime load. The safer fix is to make that decision honor explicit allowlist policy before calling the standalone loader.
