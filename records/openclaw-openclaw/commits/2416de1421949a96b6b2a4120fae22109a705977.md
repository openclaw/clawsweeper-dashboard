# Commit 2416de1

## Summary

The commit fixes the reported web-search SecretRef path, but it also routes several local `infer` commands through broad command-secret resolution before the selected provider/model is known. That can make an unused providerâ€™s missing SecretRef fail an otherwise explicit local command.

## Findings

### Medium: Explicit local infer commands can fail on unrelated provider SecretRefs

- Kind: regression
- File: `src/cli/capability-cli.ts`
- Line: 735
- Evidence: `runModelRun` resolves command secrets before `canonicalizeModelRunRef` and before `resolveModelRefOverride`, passing `getModelsCommandSecretTargetIds()` with no `allowedPaths` at `src/cli/capability-cli.ts:735`. That helper returns the full static model target family, including `models.providers.*.apiKey`, at `src/cli/command-secret-targets.ts:16` and `src/cli/command-secret-targets.ts:286`. The collector treats every enabled model provider as active, not just the requested one, at `src/secrets/runtime-config-collectors-core.ts:42`. The same broad model family is also included in the new memory embedding and TTS target sets at `src/cli/command-secret-targets.ts:40` and `src/cli/command-secret-targets.ts:45`.
- Impact: A user running an explicit local command such as `infer model run --model openai/...` can be blocked by an unrelated enabled provider, for example `models.providers.anthropic.apiKey` pointing at an env SecretRef unavailable in the CLI environment. That contradicts the selected-provider scoping expectation in the linked bug and expands the original failure mode beyond web search.
- Suggested fix: Determine the effective provider/model for explicit local infer commands before enforcing SecretRef resolution, then pass `allowedPaths` for only the selected providerâ€™s credential paths. For auto-selection paths, resolve only the actual default/fallback candidates needed by that runtime rather than the entire `models.providers.*` family.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused resolver probe with `models.providers.openai.apiKey` resolvable and `models.providers.anthropic.apiKey` missing failed through the same `infer model run` target set, while the same probe with only the OpenAI ref resolved successfully.
- Is this the best way to solve the issue? yes for explicit provider/model commands: scope command-secret `allowedPaths` to the selected provider. Auto-selection may need a narrower candidate-resolution pass rather than resolving every configured provider up front.
