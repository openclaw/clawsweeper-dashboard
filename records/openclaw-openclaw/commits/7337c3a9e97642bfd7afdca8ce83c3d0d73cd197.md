## Summary

Found one concrete contract gap: the new `currentTurnContext.promptJoiner` is honored by the PI runner but not by the Codex app-server harness, which is the default runtime for OpenAI/Codex routes.

## Findings

### Medium: Codex harness still splits Telegram inline reply body from the sender header

- Kind: bug
- File: `extensions/codex/src/app-server/run-attempt.ts`
- Line: 2607
- Evidence: The commit sets `currentTurnContext.promptJoiner` in `src/auto-reply/reply/get-reply-run.ts:760`, and the PI path uses it in `src/agents/pi-embedded-runner/run/runtime-context-prompt.ts:48`. The Codex app-server path consumes the same `currentTurnContext` type but still hardcodes `[text, prompt].join("\n\n")` in `extensions/codex/src/app-server/run-attempt.ts:2607`.
- Impact: Telegram inline replies using the Codex/OpenAI default runtime still send the model `#34975 obviyus:\n\nWhat do you mean hidden?` instead of `#34975 obviyus: What do you mean hidden?`, so the fix does not apply on a major runtime path. `src/agents/harness/policy.ts:38` routes default OpenAI and openai-codex runs to the Codex harness.
- Suggested fix: Update `prependCurrentTurnContext` to join with `context.promptJoiner ?? "\n\n"` and add Codex harness coverage for `promptJoiner: " "`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes; run a Codex app-server attempt with `currentTurnContext: { text: "Current message:\n#34975 obviyus:", promptJoiner: " " }` and inspect the `turn/start` input text. The current helper will insert `\n\n`.
- Is this the best way to solve the issue? yes; honoring the existing field at the Codex consumption point is the narrow fix and preserves the default separator for all existing context prompts.
