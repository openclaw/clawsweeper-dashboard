# Commit 1877c600ea

## Summary

Found one low-severity `/codex account` status bug in the API-key fallback inference path. Later commits fixed several adjacent stale-state issues, but the fallback still can mark an unusable API-key profile as active.

## Findings

### Low: API-key fallback inference can mark an unusable profile active

- Kind: bug
- File: `extensions/codex/src/command-account.ts`
- Line: 200
- Evidence: When the rate-limit probe fails with `"chatgpt authentication required"`, `resolveActiveProfileId` returns the first ordered profile whose stored credential has `type === "api_key"` without checking `resolveAuthProfileEligibility` or `isActiveProfileCandidate`. Active rows then bypass inactive status checks and are formatted as `"active now"` with `"billed per token"`.
- Impact: `/codex account` can claim OpenClaw is now using an API-key backup even when that profile is missing its key/keyRef, disabled, blocked, or otherwise unusable. A concrete repro is an explicit auth order containing a ChatGPT profile plus an API-key profile with no key, with `account/read` returning `requiresOpenaiAuth: true` and `account/rateLimits/read` failing with the ChatGPT-auth-required error.
- Suggested fix: In the `shouldInferApiKeyActiveFromRateLimitProbe` branch, only select API-key profiles that pass `resolveAuthProfileEligibility({ provider: "openai-codex" })` and `isActiveProfileCandidate`, or select from `resolveAuthProfileOrder` instead of raw display order.
- Confidence: medium
