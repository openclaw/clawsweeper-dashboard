# Commit 005eeca

## Summary

Found one regression in the PR OpenGrep workflow trigger. The scan itself is still diff-scoped, but the new top-level `paths:` filter can prevent the workflow from starting on large mixed-file PRs.

## Findings

### Medium: Path-filtered security workflow can be skipped on large PRs

- Kind: security
- File: `.github/workflows/opengrep-precise.yml`
- Line: 13
- Evidence: This commit changes `on.pull_request` from an unconditional PR trigger to `types` plus `paths` filters at `.github/workflows/opengrep-precise.yml:11`. GitHub documents that path-filter diffs are limited to 300 files, and if matching files are not in the first 300 files returned for the filter, the workflow will not run. The previous workflow always started on PR events and let `scripts/run-opengrep.sh --changed` decide whether there were first-party paths to scan. ([docs.github.com](https://docs.github.com/en/actions/how-tos/writing-workflows/choosing-when-your-workflow-runs/triggering-a-workflow))
- Impact: A PR with more than 300 changed files can include source or OpenGrep-rule changes while keeping the matching path outside GitHub’s first 300-file trigger window, causing `OpenGrep — PR Diff` not to run and not to upload SARIF for that PR.
- Suggested fix: Remove the top-level `paths:` filter and keep the existing script-level no-op for irrelevant PRs, or replace it with an always-triggered lightweight preflight job that computes the complete changed-file set via Git/GitHub API and gates the scan job internally.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes; create a PR whose three-dot diff has 300 non-matching files before a matching `src/**`, `scripts/**`, `security/opengrep/**`, etc. path, then observe that GitHub does not start this workflow because the trigger filter never matches.
- Is this the best way to solve the issue? yes; security workflows should avoid top-level `paths:` filters when the workflow’s own diff logic can cheaply no-op after checkout.
