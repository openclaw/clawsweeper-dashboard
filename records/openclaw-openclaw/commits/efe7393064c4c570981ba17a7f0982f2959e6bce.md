# Commit efe7393

## Summary

Found one user-visible Android regression: the new Approvals settings surface is wired to generic running tool events, not actual approval requests.

## Findings

### Medium: Approvals settings reports running tools as approval requests and misses real approvals

- Kind: bug
- File: `apps/android/app/src/main/java/ai/openclaw/app/ui/SettingsScreens.kt`
- Line: 290
- Evidence: The current renamed screen collects `viewModel.chatPendingToolCalls` and labels those rows as "Approvals", "Review actions that need your attention", and "Waiting for review". The settings entry also uses `pendingToolCalls.size` for "No pending approvals"/"`N` pending" in `apps/android/app/src/main/java/ai/openclaw/app/ui/ShellScreen.kt:351` and `apps/android/app/src/main/java/ai/openclaw/app/ui/ShellScreen.kt:715`. But `chatPendingToolCalls` is populated only from generic `stream == "tool"` events on `phase == "start"` in `apps/android/app/src/main/java/ai/openclaw/app/chat/ChatController.kt:422`, while backend tool starts are emitted for ordinary tool execution in `src/agents/pi-embedded-subscribe.handlers.tools.ts:736`. Actual command approvals are emitted separately as `stream: "approval"` in `src/agents/pi-embedded-subscribe.handlers.tools.ts:1117`, and the Android `handleAgentEvent` switch has no `approval` case.
- Impact: Any normal long-running tool can show up as a pending approval needing review even when no approval is requested. Conversely, when a real exec approval is requested, Android clears the tool row on `phase == "result"` and ignores the subsequent approval event, so this new screen does not surface the actionable approval it claims to show.
- Suggested fix: Split Android state for tool progress from approval requests. Either rename this screen/row to tool activity, or parse and store `stream == "approval"` events with approval ids/details and wire actions through the gateway approval list/resolve contract.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Start any chat run that emits a `tool` `phase=start` event; the settings row will count it as a pending approval. For the inverse path, an exec approval request emits `stream=approval`, which current Android code does not handle.
- Is this the best way to solve the issue? yes: the fix needs to use the backend approval contract instead of overloading running tool state.
