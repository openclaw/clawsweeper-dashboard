# Commit 3064d61

## Summary

Found one compatibility regression in the new `git ls-files` fast path. Current `main` still contains the issue.

## Findings

### Medium: Sparse checkouts can crash static asset discovery

- Kind: compatibility
- File: `scripts/lib/static-extension-assets.mjs`
- Line: 107
- Evidence: `listTrackedExtensionPackageDirs()` marks every `git ls-files` result as `hasPackageJson: true`, so `discoverStaticExtensionAssets()` skips the filesystem existence check and calls `readFileSync()` directly. In a temp sparse-checkout repro, `git ls-files -- ':(glob)extensions/*/package.json'` still returned `extensions/demo/package.json` while that file was absent from the worktree, and `discoverStaticExtensionAssets({ rootDir })` threw `ENOENT`.
- Impact: Scripts that import this helper, including runtime-postbuild/run-node/release checks, can fail at startup in a sparse checkout that omits some or all `extensions/*/package.json` files. The previous directory-scan path skipped absent files.
- Suggested fix: Keep the fast path, but tolerate missing tracked files before parsing metadata, for example by catching `ENOENT` around `readJsonFile(packageJsonPath, fsImpl)` and continuing.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: create a git repo with tracked `extensions/demo/package.json`, enable sparse checkout excluding `extensions/`, then call `discoverStaticExtensionAssets({ rootDir })`; current code throws `ENOENT`.
- Is this the best way to solve the issue? yes: skipping absent tracked metadata preserves the no-directory-scan behavior while restoring sparse-checkout tolerance.
