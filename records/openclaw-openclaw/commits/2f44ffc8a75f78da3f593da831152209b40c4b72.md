# Commit 2f44ffc

## Summary

Found one regression in the model catalog listing path: no-provider-filter manifest catalog rows now include disabled plugins because the new metadata snapshot exposes an include-disabled manifest registry.

## Findings

### Medium: `models list --all` can show model rows for disabled plugins

- Kind: regression
- File: `src/commands/models/list.manifest-catalog.ts`
- Line: 107
- Evidence: `loadPluginMetadataSnapshot()` builds `snapshot.manifestRegistry` with `includeDisabled: true` in `src/plugins/plugin-metadata-snapshot.ts:220`, and the no-provider-filter branch passes that registry straight into `planManifestModelCatalogRows()` without filtering enabled plugin ids. The previous code path loaded `loadPluginManifestRegistryForInstalledIndex()` without `includeDisabled`, so disabled plugin records were filtered out before planning rows.
- Impact: an exclusive `plugins.allow` no longer constrains the all-provider manifest model catalog. Focused smoke check with `cfg = { plugins: { allow: ["openai"] } }` returned 155 static rows including disabled provider `moonshot`, while `isPluginEnabled(..., "moonshot")` was `false`. The old equivalent enabled-registry path returned only OpenAI/OpenAI-Codex rows. This can make disabled providers appear available in `openclaw models list --all` and any no-filter supplemental catalog surface.
- Suggested fix: before the no-provider-filter `loadManifestCatalogRowsForPluginIds()` call, filter `snapshot.manifestRegistry.plugins` to plugin ids enabled in `snapshot.index` for the current config, or use the old enabled-registry loader for model-list catalog planning. Add a disabled-plugin/allowlist regression test to `src/commands/models/list.manifest-catalog.test.ts`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: `pnpm exec tsx -e` calling `loadStaticManifestCatalogRowsForList({ cfg: { plugins: { allow: ["openai"] } } })` returned `moonshot: true`, while a second focused check showed `isPluginEnabled({ pluginId: "moonshot" })` was `false`.
- Is this the best way to solve the issue? yes: model-list catalog planning should use enabled plugin metadata, while the broader metadata snapshot can remain include-disabled for doctor/config/diagnostic consumers.
