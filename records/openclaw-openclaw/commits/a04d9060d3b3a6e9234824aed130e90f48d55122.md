# Commit a04d906

## Summary

Found two actionable issues in the Mantis workflow change. The new Crabbox capability guard currently fails every run, and the workflow now executes a moving `openclaw/crabbox` `main` binary inside a secret-bearing job.

## Findings

### Medium: Crabbox desktop capability check always fails

- Kind: regression
- File: `.github/workflows/mantis-discord-status-reactions.yml`
- Line: 274
- Evidence: The workflow runs `"$HOME/.local/bin/crabbox" warmup --help 2>&1 | grep -q -- "--desktop"`. Crabbox is a Go `flag` CLI, and its help output lists `-desktop`, not `--desktop`. A focused local build of current `openclaw/crabbox` showed `grep --desktop status=1` and `grep -desktop status=0`. The live workflow-dispatch run on this exact commit also failed in `Install Crabbox CLI` immediately after printing Crabbox `0.5.0`, before baseline/candidate worktrees were prepared: https://github.com/openclaw/openclaw/actions/runs/25296495946
- Impact: `Mantis Discord Status Reactions` cannot reach the actual QA scenario; all valid manual/comment-triggered runs fail during CLI installation.
- Suggested fix: Change the guard to match the actual help output, for example `grep -q -- "-desktop"`, or use a functional smoke such as `crabbox warmup -h` plus a more robust parser. Better still, check a machine-readable version/capability output if Crabbox exposes one.
- Confidence: high

### Medium: Secret-bearing workflow executes unpinned Crabbox main

- Kind: supply_chain
- File: `.github/workflows/mantis-discord-status-reactions.yml`
- Line: 270
- Evidence: The workflow now does `git clone --depth 1 https://github.com/openclaw/crabbox.git` and `go build ... ./cmd/crabbox` without pinning a branch SHA or tag. The resulting binary is later executed in the `Run baseline and candidate` step with live QA secrets in the environment, including OpenAI, Convex, Crabbox coordinator, and access client credentials. Current `openclaw/crabbox` `main` resolved to `343ca7baa92f71d764370c2795fd0a970a57c054` during review, but the workflow records no expected SHA.
- Impact: The same OpenClaw commit can execute different Crabbox code over time. A bad or compromised Crabbox `main` update can break Mantis runs nondeterministically or run arbitrary code with the workflow’s live QA secrets.
- Suggested fix: Pin Crabbox to an audited commit SHA or signed release artifact, ideally through a workflow input/default such as `CRABBOX_REF` plus `git checkout "$CRABBOX_REF"` and an expected full SHA check. If the workflow must temporarily consume unreleased Crabbox behavior, pin the exact unreleased commit and update it intentionally.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. For the first finding, build current Crabbox and run `crabbox warmup --help 2>&1 | grep -q -- "--desktop"`; it exits 1 because the help output uses `-desktop`. The exact GitHub workflow run on this commit failed at the same step.
- Is this the best way to solve the issue? yes for the immediate guard fix. For the supply-chain issue, pinning a Crabbox SHA preserves access to unreleased flags while keeping workflow behavior reproducible and reviewable.
