# Commit 0b708a2

## Summary

Found two YAML edge-case regressions in the restored OC Path adapter. The important one can make `setOcPath` write a different YAML key than the key that `resolveOcPath` just reported.

## Findings

### Medium: YAML scalar map keys can resolve but set the wrong key

- Kind: bug
- File: `extensions/oc-path/src/oc-path/yaml/edit.ts`
- Line: 49
- Evidence: `resolveYamlOcPath` matches map keys by `String(k)` in `extensions/oc-path/src/oc-path/yaml/resolve.ts:95`, so a YAML numeric key `0` resolves as `oc://x.yaml/root/0`. The edit path also finds by `String(k)` in `extensions/oc-path/src/oc-path/yaml/edit.ts:133`, but then calls `ast.doc.hasIn(segments)` / `cloned.setIn(segments, newValue)` with string segments at `extensions/oc-path/src/oc-path/yaml/edit.ts:49` and `extensions/oc-path/src/oc-path/yaml/edit.ts:54`. The upstream `yaml` map lookup compares actual scalar values, so number `0` and string `"0"` are distinct keys. Focused probe:
  ```text
  root:
    0: numeric
    "0": string

  resolve oc://x.yaml/root/0 -> "numeric"
  set oc://x.yaml/root/0 updated -> updates "0": updated, leaving 0: numeric
  ```
  With only `0: numeric`, resolve succeeds but set returns `{ ok: false, reason: "unresolved" }`.
- Impact: A caller can resolve a YAML leaf and then either fail to edit it or overwrite a different sibling key with the same string rendering. That breaks the core resolve/set contract and can corrupt YAML maps containing numeric or other non-string scalar keys.
- Suggested fix: Preserve the matched YAML key identity when building edit paths, e.g. return `unknown[]` path segments for YAML map keys and push the matched scalar key value rather than `String(k)`. If non-string keys are intentionally unsupported, reject them consistently in both resolve/find and set instead of resolving them as writable paths.
- Confidence: high

### Low: Backslash map keys resolve but YAML set/find throw `OC_PATH_UNQUOTABLE`

- Kind: bug
- File: `extensions/oc-path/src/oc-path/yaml/edit.ts`
- Line: 37
- Evidence: The path parser accepts an unquoted segment containing a backslash, and YAML resolve can read it:
  ```text
  root:
    a\b: x

  resolve oc://x.yaml/root/a\b -> "x"
  ```
  But `setYamlOcPath` calls `formatOcPath(path)` for the sentinel guard at `extensions/oc-path/src/oc-path/yaml/edit.ts:37`; `formatOcPath` uses `quoteSeg`, which throws `OC_PATH_UNQUOTABLE` for any `\`. `findOcPaths` has the same failure mode for wildcard enumeration because YAML map enumeration calls `quoteSeg(String(k))` at `extensions/oc-path/src/oc-path/find.ts:496`.
- Impact: Valid YAML keys can be read but not written, and a wildcard find over a map containing such a key aborts instead of returning matches or a structured unsupported-key result. In the CLI set path this exception is not converted into a `SetResult`.
- Suggested fix: Avoid canonical formatting for internal guard labels, or make the segment formatter only reject values that actually require impossible quoting. For `find`, use a non-throwing segment encoder and add tests for YAML keys containing backslashes.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. I reproduced both issues with `node --import tsx --input-type=module` against the current source modules.
- Is this the best way to solve the issue? yes for the first finding: YAML edit resolution needs to carry the same key identity used by the `yaml` `Document` API. For the second, the safer direction is to stop using canonical path formatting as an internal guard label when formatting can throw for an already parsed path.
