# Commit 02182d5a30

## Summary

Found two actionable public-SDK compatibility regressions. The larger one can also widen access for already-installed plugin/channel tools that relied on the previous `ownerOnly` metadata contract.

## Findings

### Medium: Existing `ownerOnly` plugin tools are no longer honored

- Kind: security
- File: `src/agents/pi-tools.ts`
- Line: 1014
- Evidence: Current tool assembly explicitly says sender identity no longer filters model tools, then passes `toolsForModelProvider` directly into the tool-policy pipeline at `src/agents/pi-tools.ts:1014`. Direct HTTP invocation now selects `tools.find(...)` at `src/gateway/tools-invoke-shared.ts:227`, and the plugin MCP bridge now wraps/maps every tool at `src/mcp/plugin-tools-handlers.ts:22`. The commit also removes `ownerOnly` from public tool types at `src/agents/tools/common.ts:30` and `src/channels/plugins/types.core.ts:28`. In the reviewed base, `applyOwnerOnlyToolPolicy` filtered/wrapped tools whose `ownerOnly` metadata was true, direct HTTP used that filter, and the MCP bridge filtered `!tool.ownerOnly`.
- Impact: An installed external plugin or channel tool that previously relied on `ownerOnly: true` as the host-enforced guard can now be listed and executed by non-owner sessions when normal tool policy otherwise allows it. That is a privilege-widening regression for custom admin tools.
- Suggested fix: Keep `ownerOnly` as deprecated public metadata and continue enforcing it centrally, or add an explicit compatibility/migration layer that rejects/diagnoses old `ownerOnly` tools instead of silently ignoring the flag. If the contract is intentionally removed, make it a documented SDK breaking change with a replacement owner-gated pattern.
- Confidence: high

### Low: Public system-event trust options were removed from exported SDK types

- Kind: compatibility
- File: `src/infra/system-events.ts`
- Line: 35
- Evidence: `enqueueSystemEvent` is exported through public SDK subpaths at `src/plugin-sdk/system-event-runtime.ts:3` and `src/plugin-sdk/channel-runtime.ts:21`, but its options type now only accepts `sessionKey`, `contextKey`, and `deliveryContext` at `src/infra/system-events.ts:35`. In the reviewed base, both `forceSenderIsOwnerFalse?: boolean` and deprecated `trusted?: boolean` were present, with the comment saying `trusted` was kept for installed plugin compatibility.
- Impact: External TypeScript plugins that still pass `{ trusted: false }` or `{ forceSenderIsOwnerFalse: true }` to the public SDK export now fail excess-property type checks even though these were explicitly kept as compatibility fields before this commit.
- Suggested fix: Restore the optional fields in `SystemEvent` / `SystemEventOptions` as deprecated no-op compatibility fields if the owner-downgrade semantics are intentionally retired, and document the replacement behavior.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. For the `ownerOnly` issue, a plugin returning `{ name: "admin_tool", ownerOnly: true, execute: ... }` now flows through the current tool list/invoke paths without a central owner filter. For the system-event issue, a public SDK call using `{ trusted: false }` is now rejected by the exported TypeScript signature.
- Is this the best way to solve the issue? yes for compatibility: keep deprecated fields/metadata accepted while migrating callers to configured tool policy and owner-gated channel actions.
