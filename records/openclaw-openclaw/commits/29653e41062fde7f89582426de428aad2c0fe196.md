# Commit 29653e4

## Summary

Found one actionable regression in Mac gateway discovery transport selection.

## Findings

### Medium: TLS-enabled loopback gateways are selected as direct even when not reachable

- Kind: regression
- File: `apps/macos/Sources/OpenClaw/GatewayDiscoverySelectionSupport.swift`
- Line: 68
- Evidence: The server now advertises reachability separately with `gatewayDirectReachable: !isLoopbackHost(bindHost)` in `src/gateway/server.impl.ts:1045`, and Bonjour only emits `gatewayDirectReachable=1` when that flag is true in `extensions/bonjour/src/advertiser.ts:455`. But the Mac selection logic treats `gateway.gatewayTls` alone as enough to prefer direct transport. A default loopback-bound gateway with `gateway.tls.enabled: true` will advertise `gatewayTls=1` and no `gatewayDirectReachable`; selecting it still switches the Mac app from SSH to `wss://<serviceHost>:<port>`, which cannot connect to a loopback-only remote gateway.
- Impact: Users running TLS on a loopback-only gateway can have discovery overwrite a working SSH transport with an unreachable direct URL, breaking Dashboard/control connection until they manually switch back.
- Suggested fix: Use `gatewayTls` only for URL scheme selection. Prefer direct based on `gatewayDirectReachable`, Tailscale Serve, or the existing legacy `.ts.net` fallback, not TLS alone. Add a regression test for `gatewayTls: true`, `gatewayDirectReachable: false`, `.local` host preserving SSH.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: construct a `DiscoveredGateway` with `serviceHost: "nearby-gateway.local"`, `servicePort: 18789`, `gatewayTls: true`, `gatewayDirectReachable: false`, and current transport `.ssh`; `preferredTransport` returns `.direct`.
- Is this the best way to solve the issue? yes: reachability and TLS are intentionally separate advertised facts, so transport choice should follow reachability while TLS only controls `ws` vs `wss`.
