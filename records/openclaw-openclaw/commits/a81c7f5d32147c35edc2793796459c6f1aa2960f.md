# Commit a81c7f5

## Summary

Found two actionable regressions in the new default GraphQL PR push path. The first blocks the maintained prepare/merge workflow for normal prepared pushes. The second can publish a PR branch tree that differs from the prepared local tree for executable-bit or symlink changes.

## Findings

### High: GraphQL prep pushes record a remote commit SHA that the local prep branch never moves to

- Kind: regression
- File: `scripts/pr-lib/push.sh`
- Line: 210
- Evidence: `push_prep_head_once` now defaults to `graphql_push_to_fork`, which creates a new GitHub commit and returns its OID. `prepare_push` then writes that returned OID as `PREP_HEAD_SHA` in `.local/prep.env`, but the checked-out prep branch remains at the original local commit. `merge_verify` immediately calls `verify_prep_branch_matches_prepared_head`, which requires the local `pr-<n>-prep` branch head to equal `PREP_HEAD_SHA`; sibling commits with the same tree still fail this check.
- Impact: a successful default `prepare-push` leaves artifacts that `merge-run` rejects before merge. This breaks the maintainer PR preparation flow for any real push where GitHub creates the verified commit.
- Suggested fix: after a successful GraphQL push, fetch the PR head commit and move/reset the local prep branch to the returned OID before writing `PREP_HEAD_SHA`, or otherwise change the merge verifier to track separate local-prep and remote-match-head SHAs safely.
- Confidence: high

### Medium: Default GraphQL pushes cannot preserve file modes or symlinks

- Kind: data_loss
- File: `scripts/pr-lib/push.sh`
- Line: 210
- Evidence: GitHub GraphQL schema introspection shows `FileAddition` only accepts `path` and base64 `contents`, with no mode field. The helper now sends all default pushes through that API when PR head metadata is present. Local checks show executable-bit-only changes are included by `git diff --diff-filter=AM`, and symlinks appear as `120000 blob`; the current guard only rejects non-blob entries and submodules (`160000`), so a symlink can be sent as a regular file containing the link target text.
- Impact: preparing a PR that adds/changes executable scripts or symlinks can mutate the contributor branch to a different tree than the reviewed local prep commit. This is especially risky for scripts because losing `100755` can break installed or CI-invoked tooling.
- Suggested fix: before using GraphQL, reject any diff that includes mode changes, symlinks (`120000`), submodules, or other unsupported tree entries and require the explicit git-push override; alternatively use an API/path that can create the exact Git tree.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A local git-object check with two commits sharing the same parent and tree but different commit metadata showed the GitHub-created commit shape is not equal to, nor an ancestor of, the local prep commit, which is exactly what `merge_verify` rejects.
- Is this the best way to solve the issue? unclear. Keeping verified commits is reasonable, but the local prep branch and artifact contract must be updated to the remote verified commit, and unsupported tree changes must fail closed or use a lower-level exact-tree push path.
