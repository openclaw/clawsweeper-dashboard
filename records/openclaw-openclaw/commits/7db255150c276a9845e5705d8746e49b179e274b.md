# Commit 7db2551

## Summary

Found one remaining consistency issue on current `main`: Telegram plugin-native commands still use the flat DM session even when `dm.threadReplies` or per-DM `threadReplies` opts a DM topic into thread-scoped sessions.

## Findings

### Medium: Plugin-native commands bypass opt-in Telegram DM topic session isolation

- Kind: bug
- File: `extensions/telegram/src/bot-native-commands.ts`
- Line: 1212
- Evidence: normal inbound messages apply `shouldUseTelegramDmThreadSession(...)` and wrap the base key with `resolveThreadSessionKeys(...)` when DM topic threading is enabled (`extensions/telegram/src/bot-message-context.ts:386`). Built-in native commands do the same through `resolveTargetSessionKey()` (`extensions/telegram/src/bot-native-commands.ts:879`). The plugin-native command branch, however, passes `sessionKey: route.sessionKey` to `executePluginCommand` and uses `route.sessionKey` for delivery policy/internal hooks (`extensions/telegram/src/bot-native-commands.ts:1163`, `extensions/telegram/src/bot-native-commands.ts:1212`). `route.sessionKey` is the base direct-chat route; it does not include the `:thread:<chatId>:<dmThreadId>` suffix added by the new policy.
- Impact: a Telegram DM topic configured with `channels.telegram.dm.threadReplies: "inbound"` or `direct.<chatId>.threadReplies: "inbound"` gets isolated agent sessions for normal messages and built-in native commands, but plugin commands invoked in that same topic receive/use the flat DM session. Plugin commands that inspect, mutate, bind, or emit replies against `ctx.sessionKey` can cross topic boundaries despite the user explicitly opting into DM topic isolation.
- Suggested fix: share the built-in command session-key resolution with the plugin-command branch, then use the resolved target session key for `executePluginCommand({ sessionKey })`, `sessionKeyForInternalHooks`, and `policySessionKey`. Add a Telegram plugin-command test for a private `message_thread_id` with `dm.threadReplies: "inbound"` asserting the plugin command receives `agent:main:main:thread:<chatId>:<threadId>`.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. Configure Telegram with `dm.threadReplies: "inbound"`, register a provider plugin command, invoke it from a private Telegram message carrying `message_thread_id`, and inspect the `executePluginCommand` call: the current plugin branch passes `route.sessionKey`, while the normal/built-in branches compute the thread key.
- Is this the best way to solve the issue? Yes. The existing helper path already encodes the intended policy and should be reused so all Telegram command surfaces agree.
