# Commit 1fdc73a

## Summary

Found one concrete regression in the new chat session picker: it renders raw `sessions.list` rows instead of applying the existing chat-session eligibility filters.

## Findings

### Medium: Picker exposes sessions the chat controls previously hid

- Kind: regression
- File: `ui/src/ui/chat/session-controls.ts`
- Line: 357
- Evidence: `renderChatSessionPickerPopover` uses `const rows = result?.sessions ?? []` and then renders every row as a selectable picker option. The existing `resolveSessionOptionGroups` path in the same file still filters session choices to the active agent and excludes non-current `global`/`unknown`, hidden cron, and inactive subagent rows at `ui/src/ui/chat/session-controls.ts:1009`. Existing helper tests also document those contracts for inactive subagents and active-agent filtering in `ui/src/ui/app-render.helpers.node.test.ts:562` and `ui/src/ui/app-render.helpers.node.test.ts:580`. A read-only render probe against current `main` with active `agent:alpha:main` showed the picker options included `agent:beta:main`, `global`, `unknown`, and `agent:alpha:subagent:worker`.
- Impact: The chat header picker now bypasses the agent filter and can switch users into other-agent rows, aggregate/internal rows, or spawned subagent rows that the old select deliberately hid. `switchChatSession` will accept those keys and load chat history for them.
- Suggested fix: Reuse or refactor the existing eligibility logic for picker rows before rendering and before switching. Keep picker-local search/pagination state, but project `SessionsListResult.sessions` through the same “active agent, current session exception, no global/unknown, no hidden cron, no inactive subagents” contract.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Render `renderChatSessionSelect` with `chatSessionPickerOpen=true`, active `agent:alpha:main`, and mixed rows for `agent:beta:main`, `global`, `unknown`, and a spawned alpha subagent; the popover renders all of them as `[data-chat-session-picker-option]`.
- Is this the best way to solve the issue? yes. The new UI should preserve the old eligibility contract while changing where search and pagination state live.
