# Commit aaeb64b

## Summary

Found one release-smoke reliability issue in the call path covered by the updated assertion. The test now asserts that the release Open WebUI command includes `OPENWEBUI_SMOKE_MODE=models`, but the shell runner does not propagate that variable into the containerized probe, so the probe still defaults to chat mode.

## Findings

### Medium: Open WebUI release smoke mode is not passed to the probe

- Kind: reliability
- File: `scripts/e2e/openwebui-docker.sh`
- Line: 146
- Evidence: `scripts/lib/docker-e2e-scenarios.mjs` defines the release command with `OPENWEBUI_SMOKE_MODE=models`, and the updated test asserts that string. But `scripts/e2e/openwebui-docker.sh` starts the gateway container with selected `-e` variables at lines 68-80 and later runs `node /app/scripts/e2e/openwebui-probe.mjs` with selected `docker exec -e` variables at lines 146-155; neither forwards `OPENWEBUI_SMOKE_MODE` nor `OPENCLAW_OPENWEBUI_SMOKE_MODE`. The probe reads those variables and otherwise defaults to `"chat"` at `scripts/e2e/openwebui-probe.mjs:11`, only skipping chat completion when `smokeMode === "models"` at `scripts/e2e/openwebui-probe.mjs:113`.
- Impact: Release-path CI commands that intend to stop after Open WebUI sign-in and model discovery still run the live `/api/chat/completions` path. That can add unnecessary live-provider dependency, latency, timeout risk, and false release-check failures despite the scenario command and test claiming `models` mode.
- Suggested fix: Normalize a shell `SMOKE_MODE` from `OPENWEBUI_SMOKE_MODE` / `OPENCLAW_OPENWEBUI_SMOKE_MODE`, pass it into the `docker exec` that runs `openwebui-probe.mjs`, and add a focused assertion that `openwebui-docker.sh` forwards the smoke-mode env to the probe.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. Run the Open WebUI Docker smoke with `OPENWEBUI_SMOKE_MODE=models`; from static command flow, the probe process in the gateway container will not receive that env var and will default to chat mode. I did not run the full Docker/live smoke because it requires Docker plus live OpenAI credentials and pulls an external Open WebUI image.
- Is this the best way to solve the issue? Yes. The release scenario is already passing the intended mode at the top-level command; the runner needs to carry it across the `docker exec` boundary where the probe actually reads it.
