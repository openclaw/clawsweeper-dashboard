# Commit d580ec4

## Summary

Found one low-severity Telegram picker display bug: the new provider-prefixed fallback is still truncated from the front, so many real nested OpenRouter model refs do not actually show the `openrouter/` prefix.

## Findings

### Low: Long nested OpenRouter labels still drop the provider prefix

- Kind: bug
- File: `extensions/telegram/src/model-buttons.ts`
- Line: 222
- Evidence: The commit builds `fallbackLabel` as `openrouter/<nested-id>` when `model.includes("/")`, then immediately calls `truncateModelId(displayLabel, 38)`. `truncateModelId` preserves only the suffix with a leading ellipsis. A focused smoke check of the pure helper shows `provider="openrouter", model="anthropic/claude-3-5-sonnet-20241022"` renders `…/anthropic/claude-3-5-sonnet-20241022`, and `model="google/gemini-2.0-flash-thinking-exp:free"` renders `…le/gemini-2.0-flash-thinking-exp:free`; neither exposes the `openrouter/` prefix the commit is meant to preserve.
- Impact: Common OpenRouter refs longer than 38 characters can still appear without the outer provider in Telegram’s model picker, keeping the ambiguity this fix addresses for long but realistic model ids. Selection callback data remains correct, so this is display-only.
- Suggested fix: For nested fallback labels, truncate in the middle while preserving the outer provider prefix and enough model suffix, or raise/skip the display truncation if Telegram button text length allows it. Add a regression test with a long OpenRouter id.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Call `buildModelsKeyboard({ provider: "openrouter", models: ["anthropic/claude-3-5-sonnet-20241022"], currentPage: 1, totalPages: 1 })` and inspect the first button text.
- Is this the best way to solve the issue? yes. The selection path is sound; the issue is localized to display truncation after the new fallback label is computed.
