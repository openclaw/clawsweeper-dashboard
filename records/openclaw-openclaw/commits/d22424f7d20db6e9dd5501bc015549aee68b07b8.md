## Findings

### Low: Docker patch smoke runs shell command substitution from the JavaScript template literal

- Kind: reliability
- File: `.github/workflows/install-smoke.yml`
- Line: 336
- Evidence: The workflow passes a Node program through `sh -lc '... node -e "..." ...'`, but the JavaScript error uses a template literal: ``throw new Error(`missing patch for ${dep}: ${rel}`);``. Inside the shell’s double-quoted `node -e` argument, backticks are command substitution, so the shell evaluates `missing patch for ...` before Node starts. A focused local reproduction with the same quoting prints `sh: 1: missing: not found` and exits 0 even when the JS loop is empty; adding a temporary `missing` executable in `PATH` shows that command is actually run.
- Impact: Every full install-smoke Docker patch check can emit unrelated shell errors, and any `missing` executable found in the container `PATH` would be executed during CI command expansion. If a patch is actually missing, the intended diagnostic is also stripped down to a generic `Error`.
- Suggested fix: Avoid shell-interpreted JavaScript template literals in inline `node -e` snippets. Escape the backticks, use string concatenation, or move the check into a small checked-in script/heredoc quoted so the shell cannot perform command substitution.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Running the same `sh -lc 'node -e "..."'` quoting pattern locally triggers command substitution before Node executes; a temporary `missing` executable confirms the shell runs it.
- Is this the best way to solve the issue? yes. The bug is shell quoting, so the safest fix is to remove shell-special backticks from the inline JavaScript or move the JavaScript out of the shell string.
