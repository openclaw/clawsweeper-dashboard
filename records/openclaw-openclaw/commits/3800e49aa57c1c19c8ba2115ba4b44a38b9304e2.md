## Summary

Found one release-blocking regression in the NPM plugin publish path. The ClawHub workflow also failed on the exact SHA, but the failing no-token/OIDC-fallback path was not introduced by this diff.

## Findings

### High: Plugin NPM release now ignores the available token and fails when trusted publishing is not configured

- Kind: regression
- File: `.github/workflows/plugin-npm-release.yml`
- Line: 215
- Evidence: The workflow still provides `NODE_AUTH_TOKEN` and `NPM_TOKEN`, but line 215 sets `OPENCLAW_NPM_PUBLISH_AUTH_MODE=trusted-publisher`. `scripts/plugin-npm-publish.sh:86` then clears `publish_auth_token`, so `npm publish` runs without the token-backed `NPM_CONFIG_USERCONFIG`. The exact SHA workflow-dispatch run `25241329315`, job `74017757019`, shows `NODE_AUTH_TOKEN: ***`, `NPM_TOKEN: ***`, `OPENCLAW_NPM_PUBLISH_AUTH_MODE: trusted-publisher`, then `Publish auth: GitHub OIDC trusted publishing`, followed by `npm error code ENEEDAUTH`.
- Impact: Real plugin NPM publishes fail for packages that do not already have npm trusted publisher settings matching `openclaw/openclaw`, `plugin-npm-release.yml`, and the `npm-release` environment. In the observed run, all five `publish_plugins_npm` matrix jobs failed.
- Suggested fix: Do not force tokenless trusted publishing until every plugin package has the npm trusted publisher configured. Safer path: keep the token-backed publish userconfig while granting `id-token: write`; npm documents that trusted publishing is preferred when available and traditional tokens remain the fallback. Alternatively, gate `trusted-publisher` mode behind a verified per-package trust setup.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Dispatch `Plugin NPM Release` for this SHA with selected unpublished plugin beta versions; `gh run view 25241329315 --job 74017757019 --log` shows the token env was present but ignored, and npm returned `ENEEDAUTH`.
- Is this the best way to solve the issue? no. The current change removes the token fallback before the external npm trusted-publisher configuration is proven. npm’s docs say trusted publishing requires package settings and exact workflow configuration, and ENEEDAUTH should be expected when that setup does not match. They also say OIDC is limited to publish and the CLI falls back to traditional tokens.
