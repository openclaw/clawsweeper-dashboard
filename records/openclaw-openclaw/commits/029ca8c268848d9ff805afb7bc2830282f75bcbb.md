# Commit 029ca8c

## Summary

Found four actionable issues in the new suspension and handoff flow. The focused changed-file tests pass, but they do not cover restart persistence, broad non-quota suspension, or the missing recovery-summary/diagnostic paths.

## Findings

### Medium: persisted suspension is not enforced after gateway restart

- Kind: reliability
- File: `src/agents/pi-embedded-runner/run/attempt.ts`
- Line: 2233
- Evidence: `suspendSession` persists `quotaSuspension.state = "suspended"` and drops only the in-process lane concurrency to 0 in `src/agents/session-suspension.ts:101-134`. On a restart, command-lane state and the auto-resume timer are lost. The next attempt calls `runQuotaSuspensionMaintenance`, but if `expectedResumeBy` is still in the future it leaves the state as `"suspended"`, and the runner only branches on `"resuming"` before proceeding normally.
- Impact: a restart during the 30-minute suspension window lets the same lane run immediately and retry the quota/billing-limited provider, defeating the durable circuit breaker.
- Suggested fix: restore/schedule active persisted suspensions at startup or block/requeue attempts while the stored state is `"suspended"` and `expectedResumeBy` is still in the future.
- Confidence: high

### Medium: all multi-candidate fallback failures suspend the lane as `circuit_open`

- Kind: bug
- File: `src/agents/model-fallback.ts`
- Line: 409
- Evidence: `throwFallbackFailureSummary` calls `suspendSession` for any attributed multi-attempt fallback summary and hard-codes `reason: "circuit_open"` without inspecting attempt reasons. This catches non-quota conditions such as `model_not_found`, `format`, or auth/config failures after two candidates fail.
- Impact: ordinary misconfiguration or incompatible model errors can drop the shared lane to concurrency 0 for the TTL, making later user turns wait instead of surfacing the same actionable error.
- Suggested fix: gate summary suspension to suspend-worthy reasons only, or derive the persisted reason from the actual attempts and exclude configuration/schema/model-not-found failures.
- Confidence: high

### Low: resume handoff usually injects no recovery briefing

- Kind: bug
- File: `src/agents/pi-embedded-runner/run/attempt.ts`
- Line: 2246
- Evidence: the resume path injects `suspension.summary ?? "No recovery briefing was captured."`, but production `suspendSession` callers in `src/agents/model-fallback.ts:941`, `src/agents/model-fallback.ts:410`, and `src/agents/pi-embedded-runner/run.ts:1883`/`2264` do not pass `summary`; the new `summarizeForHandoff` helper is not called by production code.
- Impact: the advertised state-aware handoff does not provide the new model a concise recovery briefing, especially in long sessions where the full prior context may be trimmed.
- Suggested fix: generate and persist the handoff summary before suspension, or do not inject a handoff message that claims a recovery briefing exists.
- Confidence: high

### Low: direct runner suspensions do not emit `model.failover`

- Kind: reliability
- File: `src/agents/pi-embedded-runner/run.ts`
- Line: 1882
- Evidence: the prompt and assistant failover paths call `suspendSession` directly when `FailoverError.suspend` is set, but only the auth-cooldown skip path in `src/agents/model-fallback.ts:933` emits `emitFailoverEvent`.
- Impact: real rate-limit/billing suspensions from these runner paths are invisible to the new diagnostic event and OTLP span, leaving operators without the event the commit adds.
- Suggested fix: centralize suspension behind a helper that persists state and emits `model.failover`, or emit the event at each direct suspension site.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the first two via source-level paths: restart clears in-process lane state while persisted `"suspended"` is not checked, and any attributed multi-attempt fallback summary reaches the unconditional suspension block. The changed tests do not exercise either case.
- Is this the best way to solve the issue? The feature direction is plausible, but the durable state machine needs an enforcement path, and suspension needs narrower reason gating plus consistent diagnostics and summary capture.
