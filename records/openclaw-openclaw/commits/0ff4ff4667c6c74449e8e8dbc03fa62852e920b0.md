Found one concrete QA-lab mock-routing regression. The new prompt-local marker fast path can still let stale conversation history override the Telegram `session_status` scenario, including the real current-session prompt shape.

## Findings

### Low: Stale marker history can bypass the Telegram `session_status` tool call

- Kind: regression
- File: `extensions/qa-lab/src/providers/mock-openai/server.ts`
- Line: 1424
- Evidence: The new checks return `exactReplyDirective` / `exactMarkerDirective` before the `QA_TELEGRAM_CURRENT_SESSION_STATUS_PROMPT_RE` branch. Those directive variables still fall back to `allInputText`, so a previous exact-marker instruction can be reused when the latest prompt merely contains the word `marker`. The real Telegram current-session prompt contains “exact QA marker” but does not define an exact marker value (`extensions/qa-lab/src/live-transports/telegram/telegram-live.runtime.ts:324`).
- Impact: In a session with prior exact-marker history, the mock answers the old marker instead of calling `session_status`, so the opt-in Telegram current-session QA scenario can fail or silently stop testing the intended tool path. A related direct probe also showed stale `session_status` history can still hijack a later unrelated tool-result turn and return `QA-TELEGRAM-CURRENT-SESSION-BAD missing-session-key`.
- Suggested fix: Let the latest `session_status` prompt take precedence over generic marker handling, and only enter the tool-output continuation when the latest/last function call output belongs to `session_status`. For the prompt-local marker fast path, use directives extracted from `prompt` only, not stale `allInputText` fallback.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A direct mock-server request with stale input `Reply exactly: QA-OLD-MARKER`, followed by the current-session prompt and a declared `session_status` tool, returns `QA-OLD-MARKER` instead of a `session_status` function call.
- Is this the best way to solve the issue? unclear. The intended ordering is right for stale `session_status` versus latest exact-marker prompts, but the implementation needs tighter “latest prompt” and “last tool was session_status” scoping.
