# Commit 10caa764

## Summary

Found one actionable bug in the migration branch of the doctor registry repair. The new install-record filtering only applies to the refresh path, so `openclaw doctor --fix` can still rebuild a registry with stale npm install metadata after it has removed the shadowing managed package.

## Findings

### Medium: Registry migration path preserves stale managed install records

- Kind: bug
- File: `src/commands/doctor-plugin-registry.ts`
- Line: 278
- Evidence: `maybeRepairPluginRegistryState` records stale plugin ids before removal, then removes the managed npm package. But when `preflight.action === "migrate"`, it calls `migratePluginRegistryForInstall(migrationParams)` without passing filtered install records. The filtering added in this commit is only inside the later `refreshPluginRegistry` branch at `src/commands/doctor-plugin-registry.ts:291`. `migratePluginRegistryForInstall` then reloads install records from the persisted index at `src/commands/doctor/shared/plugin-registry-migration.ts:290`, so an invalid/forced-migration registry that still contains `installRecords.google-meet` writes that stale npm record back even though the package directory was removed.
- Impact: Doctor can report `Plugin registry rebuilt` after deleting the shadowing npm package, while the regenerated registry still says the bundled plugin has an npm install record pointing at a deleted `node_modules/@openclaw/...` path. Later plugin management flows load those top-level install records, so update/uninstall/repair decisions can continue treating a bundled plugin as a managed npm install.
- Suggested fix: make the migrate branch use the same filtered install-record map as the refresh branch. Since `migratePluginRegistryForInstall` currently ignores `params.installRecords`, either teach it to honor an explicit install-record override or refresh/write the migrated registry with `loadInstallRecordsWithoutPluginIds(...)` before returning. Add a test where the persisted registry is invalid or force-migrated and contains a stale top-level install record.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused temp-state script wrote an invalid persisted plugin index containing `installRecords.google-meet`, created a managed npm `@openclaw/google-meet` package shadowing a bundled candidate, then ran `maybeRepairPluginRegistryState(... shouldRepair: true)`. Output showed `packageExists: false` but the rebuilt persisted `installRecords` still contained `google-meet` with the deleted npm path.
- Is this the best way to solve the issue? yes, directionally. The canonical record filtering should happen before both rebuild mechanisms, not only the skip-existing refresh mechanism.
