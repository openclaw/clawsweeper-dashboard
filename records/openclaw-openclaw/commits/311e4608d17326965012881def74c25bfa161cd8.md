# Commit 311e460

## Summary

Found two actionable regressions: one OpenRouter video capability overlay bug that can incorrectly skip valid reference-image requests, and one Ollama discovery cache regression that breaks the existing provider-discovery test file.

## Findings

### Medium: OpenRouter live video capabilities turn reference images into a two-frame limit

- Kind: regression
- File: `extensions/openrouter/video-model-catalog.ts`
- Line: 134
- Evidence: the new live catalog maps `supported_frame_images` into `imageToVideo.enabled` and `maxInputImages = frameImages.length`. The runtime then applies that `maxInputImages` to every image reference at `src/video-generation/capability-overlays.ts:49`, but the provider code sends `role: "reference_image"` inputs to `input_references`, not `frame_images`, at `extensions/openrouter/video-generation-provider.ts:94`. OpenRouter documents `frame_images` and `input_references` as separate request fields, with `input_references` used for style/content guidance rather than exact first/last-frame control. ([openrouter.ai](https://openrouter.ai/docs/guides/overview/multimodal/video-generation/?utm_source=openai))
- Impact: when live model discovery succeeds, a model reporting `supported_frame_images: ["first_frame", "last_frame"]` makes OpenClaw skip a request with three `reference_image` inputs before it reaches OpenRouter, even though the static provider capability previously allowed up to four images and the request builder would have put those extra references in `input_references`.
- Suggested fix: keep frame-role support separate from the generic reference-image count, or preserve the static OpenRouter `maxInputImages: 4` for `reference_image` inputs unless the live API exposes an explicit `input_references` limit.
- Confidence: high

### Low: Ollama discovery caching makes the existing provider-discovery tests fail

- Kind: reliability
- File: `extensions/ollama/src/discovery-shared.ts`
- Line: 272
- Evidence: the commit wraps `params.buildProvider()` in `getCachedLiveCatalogValue()` with a 30s default TTL and a key of provider/baseUrl/API-key/quiet. `extensions/ollama/provider-discovery.test.ts` swaps fetch fixtures per test, but does not clear this shared cache, so later tests reuse an earlier model list and fail without making the expected `/api/tags` and `/api/show` calls.
- Impact: `pnpm test extensions/ollama/provider-discovery.test.ts` now fails 3 tests on current `main`. In production, the same cache can also briefly hide local Ollama model changes after a pull/delete because the dynamic local catalog is reused by key rather than refreshed.
- Suggested fix: either avoid the shared live-catalog cache for local Ollama discovery, add an explicit short/cache-bypass path for dynamic local catalogs, or clear the cache in the affected tests if the 30s stale window is intentional.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. The Ollama failure reproduces with `pnpm test extensions/ollama/provider-discovery.test.ts`. The OpenRouter skip reproduces with the new runtime guard using a live overlay that sets `imageToVideo.maxInputImages: 2`, which returns `openrouter/google/veo-3.1-fast supports at most 2 reference image(s), 3 requested; skipping`.
- Is this the best way to solve the issue? For OpenRouter, no: frame images and reference images need separate capability accounting. For Ollama, unclear: caching may be intentional for remote/live catalogs, but the current shared cache breaks the existing local-discovery contract.
