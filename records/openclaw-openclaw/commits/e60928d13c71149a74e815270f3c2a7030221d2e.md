# Commit e60928d13c

## Summary

Found two actionable regressions in the installer changes: Windows `-Tag main` now resolves to a nonexistent npm dist-tag, and the new Unix `install-cli.sh` removes a legacy subdirectory during the default npm flow before confirming it is operating on a git checkout.

## Findings

### Medium: Windows `-Tag main` now installs `openclaw@main` instead of GitHub main

- Kind: regression
- File: `scripts/install.ps1`
- Line: 483
- Evidence: The old resolver special-cased `main` to `github:openclaw/openclaw#main`; the new `Resolve-NpmOpenClawInstallSpec` only recognizes explicit URL/git/file specs and otherwise returns `"$PackageName@$trimmedTag"`, so `-Tag main` becomes `openclaw@main`. A direct helper check returned `openclaw@main`, and `npm view openclaw@main version` fails with `E404 No match found for version main`. `docs/install/installer.md` still documents `-Tag main` as “GitHub main via npm.”
- Impact: Windows users following the documented `-Tag main` path can no longer install from the repository’s main branch via the PowerShell installer.
- Suggested fix: Restore the `main` special case in `Resolve-NpmOpenClawInstallSpec`, matching `scripts/install.sh:2297`.
- Confidence: high

### Medium: Default `install-cli.sh` npm installs can delete `~/openclaw/Peekaboo`

- Kind: data_loss
- File: `scripts/install-cli.sh`
- Line: 711
- Evidence: `main()` calls `cleanup_legacy_submodules` before checking `INSTALL_METHOD`; that helper defaults to `${HOME}/openclaw` and recursively removes `${repo_dir}/Peekaboo`. I reproduced this in a temp home: sourcing the helper removed `$HOME/openclaw/Peekaboo` during the default path. The interactive Unix installer only calls this cleanup inside `install_openclaw_from_git`, and the PowerShell tests explicitly assert the Windows cleanup is only in the selected git checkout.
- Impact: Running the new local-prefix installer in its default npm mode can remove a matching directory under `~/openclaw` even when the user did not request a git checkout install.
- Suggested fix: Remove the early `cleanup_legacy_submodules` call from `main()` and keep cleanup scoped to `install_openclaw_from_git "$repo_dir"`. Ideally verify `$repo_dir/.git` before deleting.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The resolver can be extracted and called with `RequestedTag main`; the cleanup helper removes a temp `$HOME/openclaw/Peekaboo` before any git-mode decision.
- Is this the best way to solve the issue? yes. Both fixes restore behavior already present in adjacent installer paths and keep cleanup tied to the owner checkout.
