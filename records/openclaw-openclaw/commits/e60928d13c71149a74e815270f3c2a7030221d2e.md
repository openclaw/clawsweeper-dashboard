# Commit e60928d

## Summary

Found three actionable issues that still matter on current `main`: one Windows installer regression, one noninteractive CLI installer data-loss risk, and one Windows installer supply-chain risk.

## Findings

### Medium: `install.ps1 -Tag main` no longer installs GitHub main

- Kind: regression
- File: `scripts/install.ps1`
- Line: 495
- Evidence: the base resolver mapped `main` to `github:openclaw/openclaw#main`, but the new `Resolve-NpmOpenClawInstallSpec` falls through to `openclaw@main`. Current docs still advertise `-Tag main` as “GitHub main via npm”, and `npm view openclaw dist-tags --json` shows only `latest` and `beta`.
- Impact: Windows users following the documented main install path can fail or install the wrong target because there is no `main` npm dist-tag.
- Suggested fix: restore the special case for trimmed tag `main` to return `github:openclaw/openclaw#main`, matching `scripts/install.sh`.
- Confidence: high

### Medium: `install-cli.sh --npm` deletes legacy submodule paths outside the selected install

- Kind: data_loss
- File: `scripts/install-cli.sh`
- Line: 711
- Evidence: `main()` calls `cleanup_legacy_submodules` before method dispatch and without passing the parsed git dir. The helper defaults to `${OPENCLAW_GIT_DIR:-${HOME}/openclaw}` and removes `${repo_dir}/Peekaboo`. An isolated mocked run of default npm install deleted a temp `$HOME/openclaw/Peekaboo`.
- Impact: a noninteractive npm install, including one with a custom `--prefix`, can delete files in the default source checkout path even though the selected install method is not using that checkout.
- Suggested fix: remove the unconditional call and keep cleanup only in `install_openclaw_from_git "$GIT_DIR"`, where the selected checkout is explicit.
- Confidence: high

### Medium: Windows installer extracts mutable portable Git without verification

- Kind: supply_chain
- File: `scripts/install.ps1`
- Line: 287
- Evidence: `Resolve-PortableGitDownload` resolves `git-for-windows/git/releases/latest`, picks a MinGit ZIP, then `Install-PortableGit` downloads and expands it. There is no pinned version, checksum, or signature verification before the extracted binary is added to the process path.
- Impact: fresh Windows installs without Git now trust a mutable third-party binary path outside OpenClaw’s release artifacts before installing OpenClaw.
- Suggested fix: either remove the portable-Git bootstrap and require a package-manager/manual Git install, or pin the release and verify a digest/signature from an OpenClaw-owned manifest before extraction.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. I reproduced the CLI cleanup with a temp `$HOME` and mocked install functions, reproduced the PowerShell resolver output as `openclaw@main`, and confirmed npm has no `main` dist-tag.
- Is this the best way to solve the issue? Yes for the two behavioral regressions: restore the previous `main` mapping and scope cleanup to the selected git checkout. For portable Git, the safest fix direction is to remove the mutable fallback or add pinned verification.
