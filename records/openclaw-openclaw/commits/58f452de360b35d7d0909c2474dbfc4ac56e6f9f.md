## Findings

### Low: Recommended cropped finish path depends on an unchecked host `ffmpeg`

- Kind: compatibility
- File: `scripts/e2e/telegram-user-crabbox-proof.ts`
- Line: 925
- Evidence: `finish --preview-crop telegram-window` now calls local `ffmpeg` directly in `createCroppedMotionPreview`, while the updated skill tells agents to use that option by default at `.agents/skills/telegram-crabbox-e2e-proof/SKILL.md:125`. The script installs `ffmpeg` on the remote Crabbox for recording, but it does not preflight or provide a repo dependency for host-side `ffmpeg`; after `pnpm install`, this review environment still had no `ffmpeg` on `PATH` and no `node_modules/.bin/ffmpeg`.
- Impact: Agents following the new recommended `finish --preview-crop telegram-window` workflow can get a failed finish after the session recording has already been stopped and copied. Cleanup still runs, but the cropped artifact is not produced and the proof flow exits non-zero, blocking the intended publish path on hosts that otherwise have the repo dependencies installed.
- Suggested fix: Either generate the crop through an existing Crabbox/media helper that owns its binary prerequisites, run the crop on the Crabbox where `ffmpeg` is installed, or preflight host `ffmpeg` before stopping the session and fall back to the uncropped GIF with an explicit warning.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Unclear for a full Telegram run because `crabbox` was not available in this environment. The host prerequisite failure is concrete: the new code shells out to `ffmpeg`, and `command -v ffmpeg` failed here after installing repo deps.
- Is this the best way to solve the issue? Unclear. The safest direction is to avoid adding an implicit host dependency to the recommended finish path; a best-effort fallback to the existing uncropped GIF would also keep proof cleanup/publishing usable.
