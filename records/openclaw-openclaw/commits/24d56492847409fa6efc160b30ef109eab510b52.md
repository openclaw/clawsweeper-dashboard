# Commit 24d5649

## Summary

Found one regression in the disabled-reasoning path. The new OpenRouter Anthropic prefill stripper treats `reasoning: { effort: "none" }` as enabled reasoning, but pi-ai emits exactly that object for OpenRouter reasoning-capable models when no reasoning effort is requested. That makes the real request path strip assistant prefills even though the added test says disabled reasoning should keep them.

## Findings

### Medium: Disabled OpenRouter reasoning still strips assistant prefills

- Kind: regression
- File: `extensions/openrouter/stream.ts`
- Line: 47
- Evidence: `isEnabledReasoningValue` returns `true` for any non-null object, and `isOpenRouterReasoningPayloadEnabled` then strips trailing assistant messages when `payload.reasoning` is an object. In pi-aiŌĆÖs installed OpenAI-compatible request builder, OpenRouter-compatible reasoning models set `reasoning: { effort: "none" }` when `options.reasoningEffort` is absent. OpenClawŌĆÖs OpenRouter reasoning wrapper returns early for `thinkingLevel === "off"` after deleting only `reasoning_effort`, so it leaves that disabled object in place. The added disabled-reasoning test at `extensions/openrouter/index.test.ts:267` misses this because its mocked base payload never goes through pi-aiŌĆÖs real OpenRouter payload builder.
- Impact: Users who explicitly disable thinking for `openrouter/anthropic/...` models can still lose a trailing assistant prefill such as a JSON prefix. Anthropic documents that prefilling is not compatible when thinking is enabled, while non-extended-thinking prefill remains a supported prompting mode, so stripping in the disabled path changes output-control behavior unnecessarily. ([platform.claude.com](https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/extended-thinking-tips?_bhlid=067ff38f4d95d0ae9990035ad3160cc03dcfde89))
- Suggested fix: Treat OpenRouter `reasoning` objects with `effort: "none"` or `effort: "off"` as disabled, or key the prefill wrapper off the resolved OpenClaw thinking level instead of truthiness of the final `reasoning` object. Add a regression test that uses the actual pi-ai `streamSimple`/payload builder or seeds `reasoning: { effort: "none" }`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Build an OpenRouter Anthropic `openai-completions` payload with `model.reasoning: true`, `thinkingLevel: "off"`, and messages ending in an assistant prefill. The installed pi-ai builder emits `reasoning: { effort: "none" }`; applying the reviewed wrapper strips the assistant message.
- Is this the best way to solve the issue? yes. The narrow fix is to make the new OpenRouter wrapper distinguish disabled OpenRouter reasoning objects from enabled ones, preserving the commitŌĆÖs verified-route and Anthropic-model scoping.
