# Commit 6a3f5d0

## Summary

Found one low-severity compatibility regression in the new missing-plugin guard.

## Findings

### Low: Channel aliases are rejected before the existing channel-id resolver can canonicalize them

- Kind: regression
- File: `src/cli/plugins-cli.ts`
- Line: 125
- Evidence: The new guard canonicalizes the CLI argument with `normalizePluginId(id)` and checks `report.plugins.some((plugin) => plugin.id === id)` before calling `enablePluginInConfig` or `setPluginEnabledInConfig`. Those mutation helpers still resolve built-in channel aliases through `normalizeChatChannelId`, so inputs like `imsg`, `gchat`, `google-chat`, and `internet-relay-chat` previously reached `imessage`, `googlechat`, or `irc`, but now fail the new registry check because `normalizePluginId` does not apply channel aliases.
- Impact: Existing scripts or operators using built-in channel aliases with `openclaw plugins enable/disable` now get `Plugin not found` for valid bundled channel plugins.
- Suggested fix: Normalize the requested id through the same channel-aware path used by the mutation helpers before the registry membership check, or share a single resolver for policy CLI ids.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the code path: `src/channels/ids.test.ts` confirms aliases such as `imsg`, `gchat`, `google-chat`, and `internet-relay-chat`; `src/plugins/enable.ts` and `src/plugins/toggle-config.ts` canonicalize via `normalizeChatChannelId`; the new CLI preflight does not.
- Is this the best way to solve the issue? Unclear. The safest direction is to reuse the existing mutation-id resolver before validation so the guard does not define a narrower accepted-id contract than the write path.
