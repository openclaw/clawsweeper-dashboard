# Commit dd53545

## Summary

One low-severity test regression found. The runtime code is unchanged, but two updated assertions no longer prove that `channels remove --delete` actually persists the removed external channel config.

## Findings

### Low: Channel remove tests can pass without any config write

- Kind: test_gap
- File: `src/commands/channels.remove.test.ts`
- Line: 177
- Evidence: The commit replaced `toHaveBeenCalledWith(...)` with `const writtenConfig = configMocks.writeConfigFile.mock.calls[0]?.[0] ...; expect(writtenConfig?.channels?.["external-chat"]).toBeUndefined();` at lines 177 and 243. If `writeConfigFile` is never called, `writtenConfig` is `undefined`, and the optional-chain assertion still passes. The production path persists the deletion through `replaceConfigFile({ nextConfig: next, ... })` in `src/commands/channels/remove.ts:251`, so losing that write would leave the external channel in config while these two tests could still pass.
- Impact: A future regression where the external channel delete path stops writing the updated config could be missed, even though the command may still avoid `runtime.error`/`runtime.exit` and appear successful.
- Suggested fix: Add `expect(configMocks.writeConfigFile).toHaveBeenCalledTimes(1)` before reading `mock.calls[0]` in both updated tests, or use a helper that fails when no config was written.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes; by inspection, optional chaining makes the zero-call case evaluate to `undefined`, satisfying `toBeUndefined()`.
- Is this the best way to solve the issue? yes; restoring an explicit write-call assertion keeps the new direct property check while preserving the previous persistence guarantee.
