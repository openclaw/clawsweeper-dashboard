# Commit e4332f7

## Summary

Found one regression in the new `--` handling: a leading package-manager separator is now treated as the boundary before all harness args, so existing `pnpm ... -- <script args>` call sites can lose their actual script arguments.

## Findings

### Medium: Leading `--` breaks extension batch package-script invocations

- Kind: regression
- File: `scripts/test-extension-batch.mjs`
- Line: 18
- Evidence: `parseExtensionIds()` now splits on the first `--` before collecting extension IDs. When invoked as `pnpm test:extensions:batch -- cerebras`, the script receives a leading separator, parses zero extension IDs, prints usage, and exits 1. The current workflow still invokes `pnpm test:extensions:batch -- "$OPENCLAW_EXTENSION_BATCH"` at `.github/workflows/plugin-prerelease.yml:347`, so those prerelease extension shards will fail before running tests.
- Impact: Plugin prerelease extension validation is broken for the existing workflow command shape. The same leading-separator pattern also affects `scripts/test-live-media.ts:125`: `pnpm test:live:media -- image --image-providers __none__` ignored the requested suite/provider options and defaulted to all media suites.
- Suggested fix: Strip a single leading package-manager `--` before script-level parsing, or only treat `--` as a Vitest passthrough boundary after at least one harness target/suite has been parsed. Keep the new behavior for `telegram -- <vitest args>` and add tests for leading separators.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: `pnpm --silent test:extensions:batch -- cerebras` exits 1 with usage, while `node scripts/test-extension-batch.mjs cerebras` exits 0 with the expected no-tests skip.
- Is this the best way to solve the issue? yes: the regression is in argument normalization, not extension planning or Vitest execution.
