### Low: Positive Matrix tool-progress test no longer synthesizes the expected marker

- Kind: test_gap
- File: `extensions/qa-matrix/src/runners/contract/scenarios.test.ts`
- Line: 109
- Evidence: `readMatrixQaReplyDirective` only extracts markers from `reply exactly \`...\``. This commit changed `buildMatrixToolProgressPrompt` to say `reply with only this exact marker and no other text: \`...\`` in `extensions/qa-matrix/src/runners/contract/scenario-runtime-shared.ts:93`. The positive test at `extensions/qa-matrix/src/runners/contract/scenarios.test.ts:2526` therefore falls back to `MATRIX_QA_TOOL_PROGRESS_FIXED`, while the scenario waits for a generated `MATRIX_QA_TOOL_PROGRESS_<uuid>` token via `doesMatrixQaReplyBodyMatchToken` at `extensions/qa-matrix/src/runners/contract/scenario-runtime-room.ts:874`. The mock `waitForRoomEvent` returns queued events without applying the predicate, so the test still passes with a body the real client would reject.
- Impact: The main success-path unit test for `matrix-room-tool-progress-preview` can pass while the final marker is wrong. Future regressions in final-token matching or prompt/token extraction could be hidden until the live Matrix QA lane fails.
- Suggested fix: Update `readMatrixQaReplyDirective` to parse both the old `reply exactly` wording and the new `reply with only this exact marker...` wording, and assert `reply.tokenMatched: true` in the positive Matrix tool-progress test. A stronger fix is to make the room-client mock apply the supplied predicate before returning events.
- Confidence: high
