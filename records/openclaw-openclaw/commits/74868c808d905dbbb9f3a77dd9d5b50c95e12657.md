# Commit 74868c808d

## Summary

One low-severity test-gap finding. The commit makes the concurrency test deterministic, but moves the bounded-concurrency assertion before all mapped work has finished, so later oversubscription can escape the test.

## Findings

### Low: Concurrency bound is only asserted before the queued workers are released

- Kind: test_gap
- File: `extensions/qa-lab/src/suite-planning.test.ts`
- Line: 114
- Evidence: The revised test checks `expect(maxActive).toBe(2)` immediately after the first two tasks start, then releases all queued task resolvers and awaits `resultPromise` without checking `maxActive` again. Since `maxActive` is still updated by later mapper calls, a regression that starts items 3/4 with more than two active workers after the release would not fail this test.
- Impact: The test name still claims “bounded concurrency while preserving order,” but the bound is no longer verified over the full mapped run. That can hide a future QA suite worker oversubscription regression.
- Suggested fix: Keep the early assertion if it is useful for deterministic startup proof, but add or move `expect(maxActive).toBe(2)` after `const result = await resultPromise;`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes; by inspection of the changed test control flow, and the focused test passes even though the final post-release `maxActive` value is never asserted.
- Is this the best way to solve the issue? yes; a final `maxActive` assertion preserves the deterministic release behavior while restoring the original full-run concurrency check.
