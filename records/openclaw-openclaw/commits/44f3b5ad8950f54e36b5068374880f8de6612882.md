# Commit 44f3b5a

## Summary

Found two actionable launch/externalization regressions. The main risk is that this commit removes bundled Discord from the core package and changes diagnostics metadata toward npm-first installs, but the live npm packages do not currently match the repo metadata needed for a 2026.5.2 stable migration.

## Findings

### High: npm-first externalization resolves to obsolete or rejected plugin packages

- Kind: regression
- File: `package.json`
- Line: 43
- Evidence: The commit excludes `dist/extensions/discord/**` from the core package, so packaged installs must rely on external installation for Discord. Current repo metadata says Discord should be `@openclaw/discord` with `pluginApi >=2026.5.2` (`extensions/discord/package.json:61`), and diagnostics now default to npm (`extensions/diagnostics-otel/package.json:30`, `extensions/diagnostics-prometheus/package.json:17`). Live npm metadata does not match: `npm view @openclaw/discord@latest` returns `2026.3.13` and deprecated; `npm view @openclaw/diagnostics-otel@latest` returns deprecated `2026.3.13`; `npm view @openclaw/diagnostics-prometheus@latest` returns prerelease `2026.5.1-beta.2`. A focused install probe confirmed `installPluginFromNpmSpec({ spec: "@openclaw/discord", expectedPluginId: "discord" })` succeeds with `version: "2026.3.13"`, while bare `@openclaw/diagnostics-prometheus` is rejected by the prerelease guard in `src/plugins/install.ts:1127`.
- Impact: Stable users can update into a package that no longer contains bundled Discord, then migrate to an obsolete deprecated Discord plugin. Diagnostics npm-first installs either install an obsolete OTEL plugin or fail for Prometheus because the bare npm spec resolves to a prerelease.
- Suggested fix: Keep Discord bundled, or publish and verify `@openclaw/discord@2026.5.2` on the stable `latest` tag before excluding it. For diagnostics, keep ClawHub as default or pin exact npm versions/integrity that are actually published and stable.
- Confidence: high

### Medium: persisted bridge skips configured channels unless `enabled: true` was present

- Kind: regression
- File: `src/cli/plugins-location-bridges.ts`
- Line: 14
- Evidence: `buildBridgeFromPersistedBundledRecord` returns `null` when the persisted record is not `enabled`. But channel startup treats a meaningful channel config section or env var as configured: `channels.discord.token` and `DISCORD_BOT_TOKEN` both make Discord a startup plugin via `src/channels/config-presence.ts:36`, `src/channels/config-presence.ts:132`, and `src/plugins/gateway-startup-plugin-ids.ts:50`. A focused probe with `channels.discord.token` but no `enabled: true` produced `configuredChannels: ["discord"]`, `startupHasDiscord: true`, `indexEnabled: false`, and `hasNpmSpec: true`. The same happened for env-only `DISCORD_BOT_TOKEN`.
- Impact: Users whose bundled Discord worked because the channel was configured, but did not have `channels.discord.enabled: true`, get no externalized install bridge after this commit removes Discord from the packaged core. Their channel can disappear on update.
- Suggested fix: Bridge persisted bundled channel plugins based on the same configured-channel semantics used by startup, not only `record.enabled`; at minimum include records with npm metadata whose manifest channels are configured or env-detected and not explicitly disabled.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The npm issue reproduces with `npm view` and the focused `installPluginFromNpmSpec` probes. The bridge issue reproduces with `loadInstalledPluginIndex`, `listPotentialConfiguredChannelIds`, and `loadGatewayStartupPluginPlan` on a config containing `channels.discord.token` without `enabled: true`.
- Is this the best way to solve the issue? no. The safer path is to keep bundled Discord until the npm package and migration bridge are both proven against stable package metadata, or pin exact install metadata that cannot resolve to deprecated/prerelease artifacts.
