# Commit 86c1622

## Summary

Found one concrete regression: the new ACP error-chain formatter uses config-sensitive log redaction for lifecycle strings that are broadcast or stored on UI/tool/task surfaces. With `logging.redactSensitive: "off"`, nested ACP cause messages can now expose secret-shaped tokens.

## Findings

### Medium: ACP lifecycle error chains can leak secrets when log redaction is disabled

- Kind: privacy
- File: `src/acp/runtime/errors.ts`
- Line: 85
- Evidence: `formatAcpErrorChain` calls `redactSensitiveText(...)` without forcing a safety-boundary mode at `src/acp/runtime/errors.ts:85` and `src/acp/runtime/errors.ts:100`. That helper honors `logging.redactSensitive: "off"`, while docs state that setting only disables general log/transcript policy and that UI/tool/diagnostic safety surfaces still redact secrets (`docs/logging.md:235`, `docs/gateway/configuration-reference.md:1001`). The formatted string is emitted as lifecycle event data at `src/agents/command/attempt-execution.ts:700`, then broadcast as raw `agentPayload` to Control UI and node/channel subscribers at `src/gateway/server-chat.ts:684` and `src/gateway/server-chat.ts:693`. It is also stored as ACP background-task/session error state at `src/acp/control-plane/manager.core.ts:918` and `src/acp/control-plane/manager.core.ts:927`.
- Impact: A user who intentionally disables general log redaction can still reasonably expect UI/tool safety surfaces to redact. This commit newly flattens nested cause-chain text into those surfaces, so upstream ACP errors containing API keys, bearer tokens, or command diagnostics can be exposed to WebSocket clients, node/channel subscribers, task status, and ACP session state.
- Suggested fix: Use a safety-boundary redactor for `formatAcpErrorChain`, for example `redactToolPayloadText(...)`, or pass forced `mode: "tools"` plus default patterns while preserving custom `logging.redactPatterns`. Add a regression test with `logging.redactSensitive: "off"` against `emitAcpLifecycleError` or `formatAcpErrorChain`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. I ran a focused `pnpm exec tsx` probe with `OPENCLAW_CONFIG_PATH` pointing at a config containing `{ logging: { redactSensitive: "off" } }`, then called `emitAcpLifecycleError` with an `AcpRuntimeError` whose cause message included `sk-abcdefghijklmnopqrstuvwxyz123456`. The captured lifecycle event JSON still contained the full token and printed `contains-token=true`.
- Is this the best way to solve the issue? yes. The failure is at the formatting boundary: the formatter is intentionally used for gateway replies, lifecycle events, and task/tool-visible text, so the safest fix is to make that formatter always redact like other safety-boundary payloads.
