# Commit d4c7519989

## Summary

Found one high-confidence security regression: the workflow now accepts fork PR head code in a privileged, secret-bearing Mantis job, but the isolation is enforced only by agent prompt text and partial child-process env scrubbing.

## Findings

### High: Fork PR code can execute in the secret-bearing Mantis proof job

- Kind: security
- File: `.github/workflows/mantis-telegram-desktop-proof.yml`
- Line: 282
- Evidence: The new path accepts a fork PR head when `ALLOW_FORK_CANDIDATE` is true and the revision matches the PR head (`.github/workflows/mantis-telegram-desktop-proof.yml:282`). The same job then runs Codex with `GH_TOKEN`, Crabbox credentials, Convex secrets, and an OpenAI key/action input in the agent environment (`.github/workflows/mantis-telegram-desktop-proof.yml:428`, `.github/workflows/mantis-telegram-desktop-proof.yml:435`). The prompt tells the agent not to pass secrets into candidate install/build/runtime (`.github/codex/prompts/mantis-telegram-desktop-proof.md:47`), but the workflow does not enforce a no-secrets environment for those candidate commands. The proof helper only scrubs env for the mock server and gateway children (`scripts/e2e/telegram-user-crabbox-proof.ts:420`), while its generic command runner still defaults to `process.env` (`scripts/e2e/telegram-user-crabbox-proof.ts:485`).
- Impact: A malicious fork can change package scripts, build tooling, or module resolution and exfiltrate the job token or QA secrets after a maintainer adds `fork-ok` or dispatches with `allow_fork_candidate=true`. This matches the known privileged-workflow pattern where `issue_comment` or similar privileged triggers run untrusted PR code with repository secrets; CodeQL documents this as a critical Actions risk when modified build scripts can execute in the privileged job. ([codeql.github.com](https://codeql.github.com/codeql-query-help/actions/actions-untrusted-checkout-critical/))
- Suggested fix: Do not build or run fork code in this secret-bearing job. Split fork candidate execution into an unprivileged no-secrets workflow/job that uploads artifacts, then have a separate privileged job consume verified artifacts and comment. If a temporary same-job path is unavoidable, enforce isolation in deterministic shell wrappers, not the prompt: launch candidate install/build/proof with `env -i`, avoid resolving `tsx` or any loader from the candidate cwd, remove GitHub/Crabbox/Convex/OpenAI secrets from the agent process before candidate work, and run candidate code under a separate UID/container with no access to the secret-bearing process environment.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A fork PR can add a malicious install/build script or hijack tooling loaded from the candidate cwd, then a maintainer-triggered `fork-ok` run executes that fork in the privileged Mantis job. Local probes confirmed that `node --import tsx` resolves `tsx` from the current working directory and that a same-UID child with a scrubbed env can read a parent process’s initial env through `/proc/$PPID/environ`, so partial child env filtering is not a sufficient boundary if the proof command itself is launched from a secret-bearing process.
- Is this the best way to solve the issue? yes. The robust fix is a privilege split: untrusted fork code runs without secrets, privileged comment/upload/token work runs later on verified artifacts. GitHub’s secure-use guidance also treats secrets and write-scoped tokens as high-value job-wide assets and recommends minimizing exposed token permissions and secret access. ([docs.github.com](https://docs.github.com/en/actions/security-for-github-actions/security-guides/security-hardening-for-github-actions))
