## Findings

### Medium: Public SDK export leaves the API baseline failing

- Kind: regression
- File: `src/plugin-sdk/session-store-runtime.ts`
- Line: 7
- Evidence: The commit adds `readLatestAssistantTextFromSessionTranscript` to public `openclaw/plugin-sdk/session-store-runtime`, but does not update `docs/.generated/plugin-sdk-api-baseline.sha256`. On current `main`, `pnpm plugin-sdk:api:check` fails with `Plugin SDK API baseline drift detected. Hash mismatch: docs/.generated/plugin-sdk-api-baseline.sha256`.
- Impact: SDK/change validation and release preflight can fail until the tracked baseline is regenerated, and the public Plugin SDK surface change is not recorded.
- Suggested fix: Run `pnpm plugin-sdk:api:gen` and commit the updated `docs/.generated/plugin-sdk-api-baseline.sha256`, or keep the helper out of the public SDK facade if the export was not intended.
- Confidence: high

### Low: Long intentional ellipsis finals can retain stale streamed text

- Kind: bug
- File: `extensions/telegram/src/lane-delivery-text-deliverer.ts`
- Line: 115
- Evidence: Any final with at least 48 characters before trailing `...`/`…` is treated as a potential truncation, and a longer prior candidate is retained when it shares that prefix and has a 24-character alphanumeric continuation. The retained branch then finalizes `previewText` instead of the canonical final. Existing intentional-ellipsis tests only cover short prefixes.
- Impact: If a model streams a longer draft, then intentionally finalizes a long open-ended answer ending in ellipsis, Telegram can keep the earlier continuation and mirror that stale text to hooks/transcript.
- Suggested fix: Add a regression test for a long intentional ellipsis final, and require stronger corroboration before retaining a longer candidate.
- Confidence: medium
