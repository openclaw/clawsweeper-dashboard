# Commit 788cff1

## Summary

Found one low-severity behavioral gap that still appears present on current `main`: `trackToolCalls: true` is ignored in Discord `message_tool_only` source delivery, even when status reactions are explicitly enabled.

Later `main` commit `5039a35a33` already fixed two other risks from the reviewed snapshot: raw Discord target reuse for tracked reactions and status emoji override precedence.

## Findings

### Low: Tracked reaction opt-in is ignored for explicit tool-only Discord status reactions

- Kind: bug
- File: `extensions/discord/src/monitor/message-handler.process.ts`
- Line: 264
- Evidence: `statusReactionsEnabled` explicitly allows status reactions in `message_tool_only` mode when `messages.statusReactions.enabled === true` at `extensions/discord/src/monitor/message-handler.process.ts:197`, and the existing test covers that expectation at `extensions/discord/src/monitor/message-handler.process.test.ts:1020`. But `maybeBindStatusReactionsToToolReaction` returns immediately when `sourceRepliesAreToolOnly` is true, before it checks for `message` `react` calls with `trackToolCalls: true`. The positive tracking test at `extensions/discord/src/monitor/message-handler.process.test.ts:619` only covers the non-tool-only path.
- Impact: In always-on Discord guild replies, where source delivery is `message_tool_only`, an agent can opt into status reactions globally and still fail to move progress reactions to the explicitly reacted message. Subsequent tool/done reactions stay on the original inbound status target instead of the reacted target promised by `docs/tools/reactions.md:25`.
- Suggested fix: Change the guard to allow tracked reaction binding when status reactions are explicitly enabled, for example by checking `sourceRepliesAreToolOnly && !statusReactionsExplicitlyEnabled`, and add a regression test combining `sourceReplyDeliveryMode: "message_tool_only"`, `statusReactions.enabled: true`, and `trackToolCalls: true`.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes, source-level: use the existing always-on Discord setup from `extensions/discord/src/monitor/message-handler.process.test.ts:1020`, have `dispatchInboundMessage` emit an `onToolStart` payload for `message`/`react` with `trackToolCalls: true`, and observe the early return at `extensions/discord/src/monitor/message-handler.process.ts:264`.
- Is this the best way to solve the issue? Likely yes; the smallest fix is to make the tracking guard match the explicit tool-only status reaction policy already established by `statusReactionsExplicitlyEnabled`.
