# Commit ddc9581

## Summary

Found one concrete policy/tool-surface bug in the new Codex side-thread path: `/btw` forks no longer get the OpenClaw-managed Codex plugin-app config that normal Codex app-server threads receive.

## Findings

### Medium: Side Codex forks omit OpenClaw plugin-app config

- Kind: security
- File: `extensions/codex/src/app-server/side-question.ts`
- Line: 209
- Evidence: normal Codex app-server starts merge `pluginThreadConfig.configPatch` into the thread config and persist the plugin app policy context in `extensions/codex/src/app-server/thread-lifecycle.ts:201`. That patch is built as an `apps` config in `extensions/codex/src/app-server/plugin-thread-config.ts:212`. The new side fork only sends `config: buildCodexRuntimeThreadConfig(undefined)` at `extensions/codex/src/app-server/side-question.ts:209`, so it drops the same `apps` patch. Codex 0.130.0 `thread/fork` builds a fresh config from request overrides/current cwd rather than inheriting the source thread’s full config, while only dynamic tools are recovered from persisted history.
- Impact: in sessions using OpenClaw-managed Codex plugin apps, `/btw` can run with a plugin-app surface that differs from the parent thread. Selected apps may disappear, or globally configured Codex apps may become available outside the OpenClaw plugin policy that was applied to the parent. This contradicts the new “keeps the current Codex permissions and native tool surface” behavior and can widen app/tool access relative to the configured OpenClaw plugin policy.
- Suggested fix: rebuild or reuse the same plugin-thread config for side forks and merge its `configPatch` into the fork config, alongside `buildCodexRuntimeThreadConfig`. Add a focused test that a binding with plugin app policy causes `thread/fork` to receive the expected `apps` config.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Unclear as a live repro; the static path is concrete. Configure `codexPlugins`, start a normal Codex app-server thread, then run `/btw`. The parent start receives the plugin `apps` config; the side `thread/fork` request does not.
- Is this the best way to solve the issue? Yes directionally: side forks should use the same OpenClaw-managed Codex app policy construction as parent threads, not only the stored policy context.
