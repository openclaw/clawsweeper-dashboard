## Summary

Found one remaining supply-chain risk in the new performance workflow. Later commits on current `main` already fixed the original token-presence and empty-report-repo issues, but the workflow still executes a mutable third-party installer before later secret-bearing steps.

## Findings

### Medium: Performance workflow executes mutable third-party installer before secret-bearing steps

- Kind: supply_chain
- File: `.github/workflows/openclaw-performance.yml`
- Line: 157
- Evidence: The workflow pipes `https://raw.githubusercontent.com/shakkernerd/ocm/main/install.sh` directly into `bash`, then appends `$HOME/.local/bin` to `GITHUB_PATH` at line 167. Later steps run `kova` with `OPENAI_API_KEY`/`OPENAI_BASE_URL` at lines 223-224 and configure authenticated git access with `CLAWGRIT_REPORTS_TOKEN` at lines 315 and 323-324. GitHub documents that writing a directory to `GITHUB_PATH` prepends it for all subsequent steps in the job, so code from the mutable installer can place binaries that later intercept secret-bearing commands. The current installer fetched from `main` does download an OCM release selected by `--version`, but the installer script itself is still mutable because the workflow fetches it from the branch head. ([docs.github.com](https://docs.github.com/en/actions/reference/workflow-commands-for-github-actions))
- Impact: If the external `shakkernerd/ocm` branch, maintainer account, or raw GitHub delivery path is compromised, the scheduled/manual performance job can execute attacker-controlled setup code on trusted runners and persist code into later steps that receive OpenAI credentials and the clawgrit reports write token.
- Suggested fix: Fetch the installer from an immutable tag/SHA and verify a pinned checksum, or vendor a minimal OCM install helper in this repo. Keep third-party-installed directories out of `PATH` for steps that receive secrets unless the exact installed binary has been verified.
- Confidence: high
