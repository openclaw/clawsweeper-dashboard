# Commit 87eb450

## Summary

Found one concrete regression in the `openclaw update --yes` path: the new ClawHub risk acknowledgement helper can still prompt interactively during post-update plugin sync.

## Findings

### Medium: `openclaw update --yes` can still prompt for ClawHub risk warnings

- Kind: regression
- File: `src/cli/update-cli/update-command.ts`
- Line: 184
- Evidence: `resolveUpdateClawHubRiskAcknowledgementOptions` returns an `onClawHubRisk` confirmation callback whenever `opts.json` is false and `process.stdin.isTTY` is true, but it does not check `opts.yes`. The callback is then spread into `syncPluginsForUpdateChannel` and `updateNpmInstalledPlugins` during post-core plugin work. The public docs still define `--yes` as “skip confirmation prompts”.
- Impact: `openclaw update --yes` can hang automation running under a pseudo-TTY if a post-update ClawHub plugin release has risky trust signals. Without explicit `--acknowledge-clawhub-risk`, this path should fail closed instead of prompting.
- Suggested fix: Treat `opts.yes` like other non-interactive modes in this helper: do not install an `onClawHubRisk` callback unless `--acknowledge-clawhub-risk` is present. Add focused coverage for `updateCommand({ yes: true })` verifying no `onClawHubRisk` is forwarded.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Run `openclaw update --yes` in a TTY with post-update ClawHub plugin sync selecting a release whose `/security` trust response has a risk reason; current code will invoke the confirmation callback instead of failing closed.
- Is this the best way to solve the issue? yes. `--yes` should preserve non-interactive behavior without silently acknowledging registry risk; `--acknowledge-clawhub-risk` is already the explicit override.
