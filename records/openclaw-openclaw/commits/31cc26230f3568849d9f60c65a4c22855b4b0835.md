# Commit 31cc262

## Summary

Found one low-severity regression in the new Git-index discovery path: tracked plugin metadata paths are treated as readable even when the worktree does not contain the file.

## Findings

### Low: Git-index discovery crashes when tracked plugin metadata is missing from the worktree

- Kind: compatibility
- File: `scripts/lib/bundled-plugin-source-utils.mjs`
- Line: 100
- Evidence: `collectTrackedBundledPluginSourceCandidates()` now records manifest/package paths directly from `git ls-files`, and `collectBundledPluginSources()` then calls `fs.readFileSync(manifestPath)` and `fs.readFileSync(packageJsonPath)` without checking that those indexed files exist in the worktree. A focused `/tmp` repo reproduction with `extensions/foo/package.json` tracked in the index but deleted from the worktree returned `ENOENT: no such file or directory, open '/tmp/.../extensions/foo/package.json'`. The same applies to a missing tracked `openclaw.plugin.json`.
- Impact: `config:channels:check`, `config:channels:gen`, and release preflight can now crash in dirty delete/rename states or sparse/partial checkouts where `git ls-files` still reports tracked plugin metadata absent from disk. The previous implementation checked `fs.existsSync()` before reading and skipped absent metadata instead of throwing.
- Suggested fix: Keep the Git-index candidate source, but validate `manifestPath` and `packageJsonPath` with `fs.existsSync()` or `fs.statSync()` before returning/reading candidates. For `requirePackageJson`, skip candidates whose package file is not present, matching the old worktree-based contract.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes; create a temporary git repo, add `extensions/foo/openclaw.plugin.json` and `extensions/foo/package.json` to the index, delete either file from the worktree, then call `collectBundledPluginSources({ repoRoot, requirePackageJson: true })`.
- Is this the best way to solve the issue? yes; existence-checking the tracked candidates preserves the commit’s goal of avoiding directory scans while restoring the prior “missing metadata is skipped” behavior.
