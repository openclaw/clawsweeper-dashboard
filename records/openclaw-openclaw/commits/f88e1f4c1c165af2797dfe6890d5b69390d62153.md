# Commit f88e1f4

## Summary

Found one reliability regression in the new pre-ready close failure path: Google Meet command-pair realtime startup can now fail fast without cleaning up the audio bridge resources it already spawned.

## Findings

### Medium: Pre-ready close rejection skips Google Meet audio bridge cleanup

- Kind: reliability
- File: `extensions/openai/realtime-voice-provider.ts`
- Line: 428
- Evidence: The new `close` branch rejects `connect()` when the socket closes before `session.updated`, then returns without invoking `onError` or `onClose("error")`. Google Meet’s local command bridge spawns the output and input processes before `await bridge.connect()` at `extensions/google-meet/src/realtime.ts:196` and `extensions/google-meet/src/realtime.ts:197`; its cleanup is in `stop()` at `extensions/google-meet/src/realtime.ts:252`, but the `await bridge.connect()` at `extensions/google-meet/src/realtime.ts:475` is not wrapped in a catch/finally. The paired-node bridge has the same shape: it creates the bridge at `extensions/google-meet/src/realtime-node.ts:103`, then awaits `bridge.connect()` at `extensions/google-meet/src/realtime-node.ts:212`; remote stop only happens through its `stop()` path.
- Impact: A realtime socket close before readiness now makes the Meet join fail instead of hanging, but the already-started local capture/playback commands, or paired-node Chrome audio bridge, can be left running because no cleanup callback fires and no returned handle exists for the caller to stop.
- Suggested fix: Wrap the Google Meet `await bridge.connect()` calls in `try/catch` and call `await stop()` before rethrowing. Optionally also have the OpenAI close-before-ready branch report the error through `onError` before rejecting, but caller-side cleanup is the safer contract because any provider can reject `connect()`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: start Google Meet realtime in command-pair or paired-node mode, force the OpenAI realtime WebSocket to close after `open` but before `session.updated`, and observe `startCommandRealtimeAudioBridge` / `startNodeRealtimeAudioBridge` throw before calling their `stop()` cleanup.
- Is this the best way to solve the issue? yes: cleanup belongs around the resource-owning `await bridge.connect()` call, so all provider connect rejections are handled consistently.
