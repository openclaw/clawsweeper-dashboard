# Commit e7c1d10

## Summary

The commit is test-only, but it seals a privacy-sensitive redaction behavior that leaves the secret portion of a colon-delimited token visible in logs.

## Findings

### Medium: Matrix logger test now asserts leakage of most of a bearer token

- Kind: privacy
- File: `extensions/matrix/src/matrix/sdk/logger.test.ts`
- Line: 20
- Evidence: The changed assertion expects `[MatrixHttpClient] Authorization: Bearer ***:abcdefghijklmnopqrstuvwxyzABCDEFG`, so only the numeric prefix is masked while the long secret suffix remains visible. The current redaction patterns in `src/logging/redact.ts:52` match bearer tokens only as `[A-Za-z0-9._\-+=]+`, so they stop at `:` and replace just `123456`; after that, the bot-token pattern at `src/logging/redact.ts:79` can no longer match the original full token.
- Impact: If Matrix SDK or adjacent HTTP diagnostics log an `Authorization: Bearer` value containing a colon-delimited secret, OpenClaw logs retain the sensitive suffix. The new exact test locks in that output and would fail a future fix that redacts the full token.
- Suggested fix: Change the contract to require the full sensitive value not be present, and fix the bearer-header redaction to consume colon-delimited token values before partial masking. A focused regression should assert the suffix is not present.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. `pnpm test extensions/matrix/src/matrix/sdk/logger.test.ts -- --reporter=verbose` passes and the test’s expected value is the leaked log line.
- Is this the best way to solve the issue? yes. The root cause is the redaction pattern ordering/character class, while this commit’s test expectation should stop asserting the leaked suffix.
