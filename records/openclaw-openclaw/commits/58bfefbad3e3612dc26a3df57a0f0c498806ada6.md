# Commit 58bfefb

## Summary

Found one low-severity Docker scheduler regression in the new lane metadata. The release journey lane itself is registered and the focused script/plan checks pass, but it double-counts the `npm` resource while running.

## Findings

### Low: release-user-journey double-reserves the npm scheduler resource

- Kind: reliability
- File: `scripts/lib/docker-e2e-scenarios.mjs`
- Line: 243
- Evidence: `release-user-journey` is defined with `npmLane(...)` but also passes `resources: ["npm", "service"]`. `npmLane` already prepends `"npm"` at `scripts/lib/docker-e2e-scenarios.mjs:100`, and the Docker scheduler reserves resources by iterating every `laneResources(candidate)` entry at `scripts/test-docker-all.mjs:923`. A focused plan check shows the lane as `["docker","npm","npm","service"]`.
- Impact: while this lane is active with weight 4, the scheduler records `npm=8` instead of `npm=4`. With the default npm cap of 10, that blocks another weight-3 npm/service lane that should otherwise fit, reducing aggregate Docker E2E parallelism and making CI/local `test:docker:all` runs slower or more serialized.
- Suggested fix: remove `"npm"` from the lane options and leave `resources: ["service"]`; `npmLane` will keep the npm resource.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: `OPENCLAW_DOCKER_ALL_LANES=release-user-journey node scripts/test-docker-all.mjs --plan-json` prints duplicate `npm` resources for the lane, and the scheduler reserve path accounts each duplicate entry.
- Is this the best way to solve the issue? yes: the local helper contract is that `npmLane` owns the npm resource tag; other npm lanes with service needs pass only `resources: ["service"]`.
