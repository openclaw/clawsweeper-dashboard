# Commit ffdc7aa

## Summary

Found two actionable regressions: one secondary delivery path still drops the new Slack DM transport thread id, and the public Slack docs still describe the old thread-session contract.

## Findings

### Medium: Inferred follow-up commitments from Slack DM threads lose the Slack thread target

- Kind: regression
- File: `src/auto-reply/reply/agent-runner.ts`
- Line: 983
- Evidence: Flattened Slack DM thread turns intentionally clear `MessageThreadId` and put the Slack UI thread in `TransportThreadId` (`extensions/slack/src/monitor/message-handler/prepare.ts:1103`, `extensions/slack/src/monitor/message-handler/prepare.ts:1148`). But commitment extraction still stores `threadId` only from `sessionCtx.MessageThreadId` (`src/auto-reply/reply/agent-runner.ts:983`). The commitment runtime only persists a thread when the enqueue input has `threadId` (`src/commitments/runtime.ts:138`), and heartbeat delivery later builds the commitment delivery context from that stored `commitment.threadId` (`src/infra/heartbeat-runner.ts:1380`). `resolveHeartbeatDeliveryTarget` explicitly passes only `params.turnSource?.threadId` as the heartbeat thread source, so the session’s persisted `lastThreadId` cannot recover the omitted Slack thread (`src/infra/outbound/targets.ts:130`).
- Impact: A check-in inferred from an agent promise made inside a Slack DM thread can later post to the main DM instead of the Slack thread where the promise was made. This undercuts the commit’s “preserve Slack UI thread delivery” goal for delayed inferred follow-ups.
- Suggested fix: Use the new routed-thread helper, or equivalent `MessageThreadId ?? TransportThreadId`, when enqueuing commitment extraction. Add a focused test where a Slack DM thread context has `TransportThreadId` and no `MessageThreadId`, then assert the stored commitment and heartbeat route keep the thread id.
- Confidence: medium

### Low: Slack docs still state the old all-thread-replies-become-thread-sessions contract

- Kind: compatibility
- File: `docs/channels/slack.md`
- Line: 952
- Evidence: Current docs say “Slack thread replies use the parent Slack `thread_ts` for session suffixes” and the note at line 974 says `replyToMode="off"` does not flatten inbound Slack thread sessions. This commit changes the DM exception: Slack DM thread replies now keep the main/direct DM session and carry the UI thread separately via `TransportThreadId`.
- Impact: Operators reading the published Slack session contract will expect DM thread replies to create/use `:thread:<threadTs>` sessions, which is now false for the main behavior this commit changes.
- Suggested fix: Update the Slack threading docs to distinguish room/channel threads from DM UI threads, and mention that explicit runtime DM thread bindings still win.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Source-level yes. For the first finding, prepare a Slack DM thread context with `MessageThreadId: undefined` and `TransportThreadId: "650.000"`, then run commitment extraction; the enqueue path omits `threadId`, and heartbeat commitment routing cannot recover it from the session. I did not run a live Slack workspace roundtrip.
- Is this the best way to solve the issue? Yes for the suggested direction: centralizing on `resolveRoutedDeliveryThreadId` matches the helper added by this commit and avoids reintroducing thread-scoped session keys.
