# Commit ef0eb12615

## Summary

Found one concrete session-scoping bug in the new `tools.invoke` RPC path. The existing focused tests pass, but they do not cover custom `session.mainKey`, global session scope, or unknown `agentId` values.

## Findings

### Medium: `agentId`-only invokes use a synthetic noncanonical main session

- Kind: bug
- File: `src/gateway/tools-invoke-shared.ts`
- Line: 63
- Evidence: `resolveSessionKey()` maps an `agentId` request with no explicit non-`main` `sessionKey` to `canonicalizeSessionKeyForAgent(agentId, "main")`, producing `agent:<id>:main`. That bypasses the existing main-session helpers in `src/config/sessions/main-session.ts`, where `resolveAgentMainSessionKey()` honors `session.mainKey` and `canonicalizeMainSessionAlias()` maps aliases like `agent:ops:main` to the configured main key. The handler also never validates that the requested `agentId` exists, unlike `tools.catalog` and `tools.effective`.
- Impact: SDK callers using the documented `agentId` parameter can invoke tools against the wrong session when `session.mainKey` is customized, and can get synthetic sessions for unknown agents. Session-aware tools, loop detection, approval context, and any tool behavior tied to `ctx.sessionKey` can read or write under `agent:<id>:main` instead of the operator’s configured main session.
- Suggested fix: validate `agentId` against `listAgentIds(cfg)` in the RPC/shared path, and when resolving `agentId` with an omitted or `"main"` session key, use `resolveAgentMainSessionKey({ cfg, agentId })` or `canonicalizeMainSessionAlias()` rather than hard-coding `"main"`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Configure `session.mainKey: "primary"` with an agent such as `ops`, call `tools.invoke` with `{ name, agentId: "ops" }`, and inspect the hook context or tool context: the current code resolves `sessionKey` to `agent:ops:main` instead of `agent:ops:primary`.
- Is this the best way to solve the issue? yes. The repo already has canonical helpers for agent main-session resolution; reusing them keeps the new RPC aligned with existing session storage and other Gateway method behavior.
