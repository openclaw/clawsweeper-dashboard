# Commit aeb7d073

## Summary

Found one concrete regression in the new raw transcript reseed gate. The commit blocks explicit `auth-profile` and `auth-epoch` reseed reasons, but `claude-cli` missing-transcript detection runs before auth/session reuse resolution, so a combined missing transcript plus auth change is still classified as `missing-transcript` and can raw-reseed prior transcript content under new credentials.

## Findings

### Medium: Missing Claude transcript can mask auth-boundary invalidation

- Kind: privacy
- File: `src/agents/cli-runner/prepare.ts`
- Line: 284
- Evidence: `prepareCliRunContext` sets `reusableCliSession` to `{ invalidatedReason: "missing-transcript" }` whenever the saved Claude CLI transcript is absent, before calling `resolveCliSessionReuse`. The auth-profile and auth-epoch checks live only in the skipped `resolveCliSessionReuse` branch. Later, `rawTranscriptReseedReason` becomes `"missing-transcript"` and `loadCliSessionReseedMessages` allows raw-tail reseed for that reason when `reseedFromRawTranscriptWhenUncompacted` is enabled.
- Impact: An opted-in `claude-cli` backend can send raw prior OpenClaw transcript history into a fresh CLI run after a credential/profile change if the old Claude transcript is also missing. That contradicts the new docs and commit intent that auth profile or credential-epoch changes never raw-reseed.
- Suggested fix: Resolve the binding auth/session reuse state before, or alongside, the missing-transcript probe. If auth-profile/auth-epoch would invalidate the binding, preserve that reason and suppress raw reseed even when the transcript is missing. Add prepare-level coverage for missing transcript plus auth-profile/auth-epoch mismatch.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes from source: set an opted-in `claude-cli` backend, pass a saved `cliSessionBinding` whose auth profile or auth epoch no longer matches, and make `claudeCliSessionTranscriptHasContent` return false. Current code reports `missing-transcript`, not the auth-boundary reason, and the new allow-list treats that as raw-reseed eligible.
- Is this the best way to solve the issue? Yes: the bug is precedence, not the raw-tail helper itself. Auth-boundary classification needs to win before any safe-reseed reason is chosen.
