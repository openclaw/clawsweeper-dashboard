# Commit 01c5ab8d13

## Summary

Found one low-severity release-script reliability regression: the verifier now expects plain `npm pack` output but does not force npm’s `json` config off, so an inherited `NPM_CONFIG_JSON=true` or npmrc `json=true` makes the post-publish verifier fail.

## Findings

### Low: Verifier can still receive JSON npm output from inherited npm config

- Kind: compatibility
- File: `scripts/verify-plugin-npm-published-runtime.mjs`
- Line: 206
- Evidence: The commit removed the explicit `--json` flag and now parses the last non-empty stdout line as a `.tgz` filename, but the command does not pass `--json=false`. npm still honors inherited config: `NPM_CONFIG_JSON=true OPENCLAW_PLUGIN_NPM_VERIFY_ATTEMPTS=1 node scripts/verify-plugin-npm-published-runtime.mjs @openclaw/zalo@2026.5.22` fails with `plugin-npm-published-runtime-check: npm pack did not report a tarball filename`. For the large package named by the new test, the same config reintroduces the buffered JSON failure: `@openclaw/msteams@2026.5.24-beta.1` fails with `spawnSync npm ENOBUFS`.
- Impact: A release or local verification environment with npm JSON mode enabled can fail after publishing a plugin, even though the package is valid. The default GitHub workflow environment does not appear to set this, so this is environment-specific rather than a default CI failure.
- Suggested fix: Add `--json=false` to the `npm pack` args, or sanitize `NPM_CONFIG_JSON`/`npm_config_json` for this subprocess, so the script always receives the plain tarball filename format it now parses.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes: run the verifier with `NPM_CONFIG_JSON=true` against `@openclaw/zalo@2026.5.22` or `@openclaw/msteams@2026.5.24-beta.1`.
- Is this the best way to solve the issue? yes: forcing `--json=false` keeps the intended plain-output fix while avoiding inherited npm config drift.
