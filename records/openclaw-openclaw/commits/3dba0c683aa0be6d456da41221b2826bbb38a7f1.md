# Commit 3dba0c6

## Summary

Found one workflow regression: the new custom Codex home is written under `/tmp`, but the pinned `openai/codex-action` runs `codex exec` through `sudo -u codex` without preserving `CODEX_HOME`. A live run of this exact workflow/commit failed before the agent could start or produce evidence.

## Findings

### Medium: Custom `codex-home` is not preserved into the unprivileged Codex process

- Kind: regression
- File: `.github/workflows/mantis-telegram-desktop-proof.yml`
- Line: 382
- Evidence: The workflow now creates `/tmp/mantis-codex-home-${GITHUB_RUN_ID}` and passes it as `codex-home`. The pinned `openai/codex-action@5c3f4ccdb2b8790f73d6b21751ac00e602aa0c02` sets `env.CODEX_HOME`, but for `safety-strategy: unprivileged-user` it invokes `sudo -u codex -- <codex>` without `-E` or `--preserve-env`; default `sudo` drops `CODEX_HOME`. The exact post-commit run `https://github.com/openclaw/openclaw/actions/runs/25652345291` resolved and wrote proxy state under `/tmp/mantis-codex-home-25652345291`, then `Run Codex Mantis Telegram agent` failed immediately with `Error: Permission denied (os error 13)` and no `mantis-evidence.json`.
- Impact: Mantis Telegram Desktop Proof now fails before the agent starts, so maintainers cannot get native Telegram proof from this workflow.
- Suggested fix: Avoid a custom home that depends on `CODEX_HOME` crossing `sudo`, or make it explicit. The safer workflow-side fix is to use the action’s default `/home/codex/.codex` path, or pass `/home/codex/.codex` and prepare ACLs there so Codex’s default home still matches even when `CODEX_HOME` is stripped. Alternatively, update/pin an action version that preserves `CODEX_HOME` for `unprivileged-user`, or add a sudoers `env_keep` entry before invoking the action.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Trigger `Mantis Telegram Desktop Proof` on this commit; run `25652345291` failed in `Run Codex Mantis Telegram agent` before evidence generation.
- Is this the best way to solve the issue? yes. Keeping the proxy config and the actual Codex process on the same home path removes the sudo environment dependency.
