# Commit f98ba66

## Summary

Found one reliability issue in the Parallels Linux smoke bootstrap: the new timeout does not make `apt-get update` wait for the apt lists lock, so the smoke can still fail immediately during fresh Ubuntu boot background updates.

## Findings

### Low: `apt-get update` still fails immediately on the apt lists lock

- Kind: reliability
- File: `scripts/e2e/parallels/linux-smoke.ts`
- Line: 445
- Evidence: The commit adds `-o DPkg::Lock::Timeout=300` to `apt-get update`, but Debian apt tracks `/var/lib/apt/lists/lock` separately from the dpkg/admin locks. Debian bug #1069167 specifically asks for an option for `apt-get update` to wait on `/var/lib/apt/lists/lock`, and bug #1012173 records that `DPkg::Lock::Timeout` applied to the admin/status-file lock rather than other apt locks. I also reproduced this locally with apt 2.8.3 by holding a temp `lists/lock` with `fcntl` and running `apt-get -o DPkg::Lock::Timeout=3 update` against temp state dirs; it exited `100` in `0s` with `Could not get lock .../lists/lock`. ([bugs-devel.debian.org](https://bugs-devel.debian.org/1069167))
- Impact: On a freshly restored Ubuntu snapshot, `apt-daily`/background update work can hold the lists lock. The Parallels Linux smoke’s `fresh.bootstrap-guest` or `upgrade.bootstrap-guest` phase can still fail immediately instead of waiting, so the commit only partially fixes the apt-lock flake.
- Suggested fix: Wrap the `apt-get update` call in a guest-side retry/wait helper for apt lock failures, or wait until holders of `/var/lib/apt/lists/lock`, `/var/lib/dpkg/lock-frontend`, `/var/lib/dpkg/lock`, and `/var/cache/apt/archives/lock` exit before running apt. Keep `DPkg::Lock::Timeout` for install, but do not rely on it for the lists lock.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Hold the apt lists lock with a POSIX `fcntl` lock in a temp apt state directory, then run `apt-get -o DPkg::Lock::Timeout=3 update` pointed at that temp state. It fails immediately with the lists-lock error.
- Is this the best way to solve the issue? no. The timeout is useful for dpkg/frontend lock waits, but the `apt-get update` lists-lock case needs an explicit wait/retry path.
