# Commit f29aaa2

## Summary

Found one low-severity race in the new fallback path for resolving the dispatched `NPM Telegram Beta E2E` workflow run. The direct URL parse path is fine; the risk is limited to the case where `gh workflow run` returns no run URL.

## Findings

### Low: Fallback run lookup can attach to another concurrent Telegram beta run

- Kind: concurrency
- File: `scripts/release-beta-smoke.ts`
- Line: 189
- Evidence: `selectNewestDispatchedRunId` filters out run IDs seen before dispatch, then selects the newest remaining `workflow_dispatch` run by `createdAt`. `listWorkflowDispatchRuns` only fetches `databaseId,createdAt`, with no branch/ref, actor, head SHA, package label, or unique request marker. The fallback at `scripts/release-beta-smoke.ts:269` therefore accepts any newer `npm-telegram-beta-e2e.yml` manual run in the repo. A focused simulation with two new IDs returned the later ID, which would be wrong if the earlier new ID was this helper’s dispatch and a second maintainer/process dispatched the same workflow seconds later.
- Impact: the helper can poll and download the artifact for a different package/ref, then print a Telegram report that did not validate the beta under review. For release smoke automation, false validation evidence is worse than failing closed.
- Suggested fix: make the fallback match a unique dispatch marker. A robust path would add a generated request ID input or `run-name` marker to `.github/workflows/npm-telegram-beta-e2e.yml`, pass it from `release-beta-smoke.ts`, and list/match `displayTitle` or another exposed field. At minimum, filter by dispatch start time, `--branch`/head branch for `options.ref` where applicable, and triggering actor, then choose the matching/oldest eligible run rather than newest across all refs.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the selector logic: importing `selectNewestDispatchedRunId` and passing two post-snapshot runs returns the newest one regardless of which run the current process dispatched. I did not live-dispatch the workflow because this is a report-only review and dispatching would mutate GitHub Actions state.
- Is this the best way to solve the issue? Yes, matching a unique marker is the safest fix direction. Branch/actor/time filters reduce the race but still cannot prove exact identity when two same-actor runs are close together.
