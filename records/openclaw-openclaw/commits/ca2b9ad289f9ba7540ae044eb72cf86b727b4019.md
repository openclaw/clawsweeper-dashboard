# Commit ca2b9ad

## Summary

I found one remaining auth bug in the per-model override fix: transport now honors a model-level `baseUrl`, but auth resolution still only receives the provider-level config plus `modelApi`. That means a model can be resolved to one endpoint while auth is still chosen for another endpoint.

## Findings

### Medium: per-model `baseUrl` is dropped during auth resolution

- Kind: bug
- File: `src/agents/model-auth.ts`
- Line: 975
- Evidence: `getApiKeyForModel` forwards `modelApi: params.model.api`, but it does not forward `params.model.baseUrl`. The synthetic-auth path then builds hook context only from `resolveProviderConfig(...)`, so plugin/local auth decisions still see `providerConfig.baseUrl`, not the selected model’s effective `baseUrl`. Meanwhile `src/agents/pi-embedded-runner/model.ts:914` now lets `configuredModel?.baseUrl` win for fallback transport.
- Impact: A mixed provider can resolve a selected model to `http://127.0.0.1:11434/v1` but still fail auth because the provider-level base URL is remote. If the provider has an API key, the same split causes the provider-level key to be used for the model-level endpoint.
- Suggested fix: Carry the effective model `baseUrl` into auth resolution, or pass an effective provider/model transport descriptor into `resolveApiKeyForProvider`. Synthetic local auth and provider auth hooks should evaluate the same `api`/`baseUrl` pair that transport will use.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A one-off `node --import tsx` probe showed `resolveModel(...)` resolving a configured fallback model to `api: "openai-completions"` and `baseUrl: "http://127.0.0.1:11434/v1"`, while `getApiKeyForModel(...)` with the same model and a remote provider-level `baseUrl` threw `No API key found for provider "mixed"`.
- Is this the best way to solve the issue? yes. The fix direction should make auth consume the already-selected model transport facts, rather than adding another provider-level fallback branch.
