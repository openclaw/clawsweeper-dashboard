# Commit bd6035d

## Summary

Found one regression in the runtime registry reuse path. Gateway startup now loads built bundled plugin artifacts, but later compatible default-mode runtime lookups do not consider that startup registry compatible and can re-import the source TS entry again.

## Findings

### Medium: Later runtime lookups can fall back to the broken source entry after built-artifact startup succeeds

- Kind: regression
- File: `src/plugins/loader.ts`
- Line: 780
- Evidence: `loadGatewayPlugins` now always passes `preferBuiltPluginArtifacts: true` at `src/gateway/server-plugins.ts:605`. `buildCacheKey` includes that flag as `prefer-built-artifacts` vs `source-default` at `src/plugins/loader.ts:779`, while the gateway-bindable compatibility bridge at `src/plugins/loader.ts:1221` only adds `runtimeOptions.allowGatewaySubagentBinding` and does not also try `preferBuiltPluginArtifacts: true`. Runtime provider lookup uses `getLoadedRuntimePluginRegistry(...loadOptions...) ?? getRuntimePluginRegistryForLoadOptions(loadOptions)` at `src/plugins/providers.runtime.ts:316`, and `getLoadedRuntimePluginRegistry` requires cache-key compatibility for scoped active lookups at `src/plugins/active-runtime-registry.ts:84`. I reproduced this with a temp bundled plugin whose source `index.ts` throws and whose `dist-runtime` `index.js` loads: the gateway-style load returned `loaded`, then a later matching default load returned `error: Error: source TS loaded` and was not the same registry.
- Impact: This undermines the commit’s startup fix for follow-up runtime loads. A source checkout can start successfully from built artifacts, then fail later when provider/tool/runtime resolution requests the same scoped plugin without the new flag. It can also duplicate plugin registration side effects instead of reusing the active gateway registry.
- Suggested fix: Preserve the existing gateway-bindable reuse behavior for this new mode. For example, when `getCompatibleActivePluginRegistry` or the reusable-cache bridge tests a default/snapshot request against a gateway-bindable active registry, also try the same candidate with `preferBuiltPluginArtifacts: true`. Alternatively, propagate the flag into runtime load-option builders that are meant to reuse the gateway startup registry.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The inline smoke created `extensions/startup-artifact-test/index.ts` that throws and `dist-runtime/extensions/startup-artifact-test/index.js` that registers successfully, then called `loadOpenClawPlugins` first with gateway-style `{ preferBuiltPluginArtifacts: true, runtimeOptions: { allowGatewaySubagentBinding: true } }` and then with matching default options. The second load re-imported source and failed.
- Is this the best way to solve the issue? yes for the fix direction: keep cache-key separation for source-default vs built-artifact direct loads, but teach the gateway-bindable active-registry compatibility path that the built-artifact startup registry satisfies later matching default/snapshot runtime lookups.
