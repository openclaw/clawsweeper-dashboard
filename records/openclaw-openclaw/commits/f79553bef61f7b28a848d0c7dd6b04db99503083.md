# Commit f79553b

## Summary

Found one regression: the commit makes every no-prompt Claude CLI credential read file-only, but runtime fallback paths still use `allowKeychainPrompt: false`. That reopens the keychain-only macOS runtime failure fixed by #73682.

## Findings

### Medium: No-prompt runtime Claude auth no longer reuses cached Keychain credentials

- Kind: regression
- File: `src/agents/cli-credentials.ts`
- Line: 463
- Evidence: The previous parent had an early return that allowed `readClaudeCliCredentialsCached({ allowKeychainPrompt: false })` to reuse an already-cached positive `:keychain` credential. This commit removes that branch and keys no-prompt reads as `${credentialsPath}:file`, so they call `readClaudeCliCredentials` with `allowKeychainPrompt: false` and return `null` when `~/.claude/.credentials.json` is absent. The updated test now asserts exactly that outcome in `src/agents/cli-credentials.test.ts:275`.
- Impact: This is broader than `models.authStatus`. Runtime fallback still loads auth with `allowKeychainPrompt: false` in `src/agents/model-fallback.ts:775`, and cooldown refresh does the same at `src/agents/model-fallback.ts:384`. On macOS setups where Claude CLI credentials live only in Keychain, a cron/runtime lane can miss fresh cached interactive Claude auth and fall back to the existing `No credentials found for profile "anthropic:claude-cli"` failure mode. The changelog still documents the opposite contract at `CHANGELOG.md:91`.
- Suggested fix: Restore positive cached-Keychain reuse for runtime no-prompt callers, or split the policy into two modes: status/read-only paths can request strict file-only reads, while runtime no-prompt paths can reuse already-cached Keychain credentials without triggering a new prompt.
- Confidence: high
