# Commit 21a92ea0f6

## Summary

Found one reliability gap: the new stale-listener recovery is still global app-message liveness, so direct-message traffic can mask the group-only inbound failure this commit is meant to recover.

## Findings

### Medium: Direct messages still mask stale group inbound after reconnect

- Kind: reliability
- File: `extensions/whatsapp/src/connection-controller.ts`
- Line: 334
- Evidence: `monitorWebChannel` calls `controller.noteInbound()` for every delivered `WebInboundMsg` without distinguishing direct vs group traffic (`extensions/whatsapp/src/auto-reply/monitor.ts:310`). `noteInbound()` then refreshes `lastInboundAt` and clears `openedAfterRecentInbound` for any inbound message (`extensions/whatsapp/src/connection-controller.ts:334`). The watchdog only checks total app silence using that single `lastInboundAt` timestamp (`extensions/whatsapp/src/connection-controller.ts:607`). The new regression test covers the case where no app messages arrive after the 408 reconnect, but not the reported case where DMs keep arriving while group messages stop (`extensions/whatsapp/src/auto-reply.web-auto-reply.connection-and-logging.e2e.test.ts:516`).
- Impact: The canonical issue context for #66920 says DMs continued while group inbound silently died. In that state, each DM refreshes the global liveness timestamp, so the watchdog will treat the listener as healthy and will not force the reconnect needed to restore group delivery. Group messages can still be permanently missed until another failure or manual restart.
- Suggested fix: Track liveness separately for group delivery or known active group subscriptions, or keep a post-408 recovery window that direct DMs cannot clear when the risk being mitigated is group inbound loss. Add a regression where a post-408 listener receives continued transport frames and direct DMs, but no group upserts, and assert the recovery reconnect still happens.
- Confidence: high
