# Commit d5c094f

## Summary

Found one regression in the CLI `models list` configured/default path: it now shows models outside an explicit `agents.defaults.models` allowlist.

## Findings

### Low: `models list` bypasses the configured model allowlist

- Kind: regression
- File: `src/commands/models/list.row-sources.ts`
- Line: 176
- Evidence: `appendConfiguredModelRowSources` first appends `agents.defaults.models` entries, but then unconditionally appends every explicit `models.providers.*.models` row and every auth-backed catalog row via `appendConfiguredProviderRows` and `appendAuthenticatedCatalogRows`. That ignores the documented allowlist behavior where `agents.defaults.models` wins when present (`docs/concepts/models.md:66`, `docs/gateway/protocol.md:608`), and the Gateway path has an explicit regression test for that contract (`src/gateway/server.models-voicewake-misc.test.ts:578`).
- Impact: users with a deliberately restricted model allowlist now see disallowed provider models in `openclaw models list`. I reproduced this with an isolated config containing only `agents.defaults.models["openai/gpt-allowed"]` plus a configured `minimax` provider model; `openclaw models list --json` returned both `openai/gpt-allowed` and the non-allowlisted `minimax/MiniMax-M2.7-highspeed`. That can mislead CLI users or automation into selecting a model that runtime policy will later reject.
- Suggested fix: mirror the Gateway visibility semantics in the CLI row source. When `agents.defaults.models` has exact entries, keep the list limited to those entries; when it has `provider/*`, append catalog/provider rows only for those wildcard providers; only use the new “explicit provider rows + auth-backed catalog rows” behavior when no allowlist is configured.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Use an isolated config with one exact `agents.defaults.models` allowlist entry and one configured provider model outside that allowlist, then run `openclaw models list --json`; the outside model is included.
- Is this the best way to solve the issue? yes. The safer fix is to apply the same allowlist-aware visibility rule already used by the Gateway configured view instead of appending the new row sources unconditionally.
