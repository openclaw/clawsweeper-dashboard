## Findings

### Medium: Codex migration provider tests fail after removing the cached-plugin warning

- Kind: regression
- File: `extensions/codex/src/migration/provider.test.ts`
- Line: 213
- Evidence: `pnpm test src/commands/migrate/output.test.ts extensions/codex/src/migration/provider.test.ts -- --reporter=verbose` fails 3 tests. All failures are stale expectations for `"Codex cached plugin bundles remain manual-review only."` at `extensions/codex/src/migration/provider.test.ts:213`, `extensions/codex/src/migration/provider.test.ts:369`, and `extensions/codex/src/migration/provider.test.ts:428`, while the commit removed that warning from `extensions/codex/src/migration/plan.ts:426`.
- Impact: the extension migration test shard is red on current `main`, which blocks reliable CI/release validation even if the warning removal is intended.
- Suggested fix: update the Codex provider test expectations to match the new warning contract, or restore the warning if the tests reflect intended behavior.
- Confidence: high

### Medium: `openclaw migrate apply --yes` now suppresses all migration warnings

- Kind: regression
- File: `src/commands/migrate/output.ts`
- Line: 106
- Evidence: `formatMigrationPreview` includes `formatPlanWarnings(plan)` at `src/commands/migrate/output.ts:98`, but `formatMigrationResult` does not. `writeApplyResult` only logs `formatMigrationResult(result)` at `src/commands/migrate/output.ts:217`. For `opts.yes`, `migrateApplyCommand` skips the preview branch at `src/commands/migrate.ts:367`, and `runMigrationApply` plans and applies directly before writing only the result at `src/commands/migrate/apply.ts:64` and `src/commands/migrate/apply.ts:122`.
- Impact: documented/direct workflows like `openclaw migrate apply codex --yes` no longer show warnings such as unverified app-backed plugin planning, archive-only Codex config/hooks, skipped secrets, or manual-review state. Interactive apply still previews warnings, but non-interactive apply is exactly where users cannot rely on the prompt preview.
- Suggested fix: include warnings in apply output when no preview was emitted, or have `runMigrationApply` emit plan warnings before applying in `--yes` mode.
- Confidence: high

### Low: Hidden config conflicts can make the preview say `0 conflicts` before apply aborts

- Kind: bug
- File: `src/commands/migrate/output.ts`
- Line: 18
- Evidence: the header counts only non-hidden items at `src/commands/migrate/output.ts:18`, so `config` items are excluded from the displayed conflict count. Apply still aborts on `plan.summary.conflicts` at `src/commands/migrate/output.ts:200`. A config-only conflict can therefore render as `0 conflicts` in the preview while `assertConflictFreePlan` later throws `Migration has 1 conflict`.
- Impact: users get an inconsistent preview and no visible conflicting item to inspect, especially because config items are now hidden unconditionally.
- Suggested fix: either include hidden conflicts in the header conflict count, surface hidden conflict items, or add an explicit hidden-config conflict line.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The test regression reproduces with the exact command above. The warning suppression follows directly from the `--yes` control flow and can be covered with a unit test where a provider returns `warnings` and `migrateApplyCommand(..., { yes: true })` should emit them.
- Is this the best way to solve the issue? yes for the test failure: align tests with the intended warning contract. For warning output, the safest fix is to preserve “warnings are seen before mutation” for `--yes` without reintroducing duplicate warnings after an interactive preview.
