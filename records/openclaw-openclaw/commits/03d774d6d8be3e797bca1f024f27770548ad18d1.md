# Commit 03d774d6

## Summary

Found one medium-confidence regression risk in the new shared active-state helper: it treats persisted terminal session status as always newer than the live active-run flag, but gateway code computes `hasActiveRun` from current abort controllers independently of persisted status.

## Findings

### Medium: terminal persisted status can suppress a genuinely active recovered run

- Kind: regression
- File: `ui/src/ui/session-run-state.ts`
- Line: 9
- Evidence: `isSessionRunActive()` returns `false` for any present non-`running` status before consulting `hasActiveRun`. That helper now drives `hasAbortableSessionRun()` and `reconcileChatRunFromCurrentSessionRow()`, so a row shaped like `{ status: "done", hasActiveRun: true }` is always treated as inactive. In the gateway, `sessions.list` overlays `hasActiveRun` from live `chatAbortControllers` separately from persisted row `status` (`src/gateway/server-methods/sessions.ts:944`), and `chat.send` registers the abort controller before later lifecycle `start` events update/persist `status: "running"` (`src/gateway/server-methods/chat.ts:2298`, `src/gateway/server-chat.ts:979`).
- Impact: after a page refresh, reconnect, or session reload during that active-before-start-status window, Control UI can hide the Stop/recovered active-run UI, enable terminal-state UI such as compaction, or clear local run tracking as `done` even though the gateway still has an abortable active run for that session.
- Suggested fix: avoid making terminal `status` globally authoritative over `hasActiveRun`. Either have the gateway project live active rows as `status: "running"` when an abort controller exists, or carry enough active-run identity/timestamps to let the UI distinguish stale completion flags from a currently active controller before suppressing abort/recovery UI.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Unclear. I did not run a live refresh/reconnect race, but the source contract shows the fields can diverge: active-run state is live controller state, while status is persisted lifecycle state.
- Is this the best way to solve the issue? Unclear. The safest direction is to make the data contract explicit, preferably in gateway projection; otherwise the UI helper needs separate semantics for display cleanup versus abortability/recovery.
