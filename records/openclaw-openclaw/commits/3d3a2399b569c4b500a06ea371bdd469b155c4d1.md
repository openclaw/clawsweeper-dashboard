# Commit 3d3a239

## Summary

Found two low-severity diagnostics regressions in recovery/activity bookkeeping. Both are actionable, but neither looks like a high/critical runtime correctness or security issue.

## Findings

### Low: Aborted recoveries that also release queued work leave stale diagnostic queue depth

- Kind: bug
- File: `src/logging/diagnostic-session-recovery-coordinator.ts`
- Line: 111
- Evidence: `applyRecoveryOutcomeToDiagnosticState` now clears queue depth only through `recoveryOutcomeClearsQueuedSessionState()`, which returns true for `released` and `noop/no_active_work` only. `recoverStuckDiagnosticSession()` can return `status: "aborted"` with `released > 0` when an active run aborts but does not drain and the lane is reset. A focused probe with `queueDepth: 3` and an outcome `{ status: "aborted", released: 3, drained: false }` left the state as `{"state":"idle","queueDepth":2,"generation":8}`.
- Impact: after recovery has reset the lane, diagnostics can continue reporting queued work that no longer exists. That keeps heartbeat/liveness work alive and makes diagnostic labels misleading until later state churn or pruning.
- Suggested fix: clear queued diagnostic state for any recovery outcome with `recoveryOutcomeReleasedCount(outcome) > 0`, including `aborted`, while preserving the new `noop/no_active_work` clear behavior.
- Confidence: high

### Low: `run.completed` clears reply-run diagnostic ownership too early

- Kind: bug
- File: `src/logging/diagnostic-run-activity.ts`
- Line: 237
- Evidence: the commit converts embedded activity to a keyed owner set, but `recordRunCompleted()` still calls `activity.activeEmbeddedRuns.clear()`. A focused probe started normal embedded work plus reply work for the same session, emitted `run.completed`, and the snapshot became `{"lastProgressAgeMs":1,"lastProgressReason":"run:completed"}` with no `activeWorkKind`, even though the reply work owner had not ended.
- Impact: post-run reply work can be misclassified as no active work if it lasts past the stuck-session threshold. Recovery runtime still checks the reply-run registry before lane reset, so this is mostly false stuck diagnostics and skipped recovery noise, not an immediate lane reset/data-loss path.
- Suggested fix: have `run.completed` remove only the underlying run owner, not every `activeEmbeddedRuns` key. Preserve `reply:*` work keys until `markDiagnosticEmbeddedRunEnded(..., clearRunActivity: false)` runs for the reply operation.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Both were reproduced with focused `node --import tsx` probes against current `main`.
- Is this the best way to solve the issue? yes for the first finding. For the second, the safest direction is to make owner cleanup granular rather than clearing the whole set.
