# Commit 335f870

## Summary

Found one live-test regression in the Codex CLI backend path. The new minimal tool profile hides `cron`, but the same test path still runs the default MCP cron preflight.

## Findings

### Medium: Codex CLI live backend can no longer run its default MCP cron probe

- Kind: regression
- File: `src/gateway/gateway-cli-backend.live.test.ts`
- Line: 261
- Evidence: `useMinimalToolsProfile` is enabled for `providerId === "codex-cli" && !schemaProbePluginPath`, then writes `tools.profile: "minimal"` into the live config. The Codex CLI backend still declares `defaultMcpProbe: true` in `extensions/openai/cli-backend.ts:12`, so `enableCliMcpProbe` remains true unless the environment explicitly sets `OPENCLAW_LIVE_CLI_BACKEND_MCP_PROBE=0`. That path calls `verifyCliCronMcpLoopbackPreflight` at `src/gateway/gateway-cli-backend.live.test.ts:551`, which requires `tools/list` to expose `cron`. But `cron` is only in the `coding` profile at `src/agents/tool-catalog.ts:233`, while `minimal` only allows `session_status` at `src/agents/tool-catalog.ts:193`. I also confirmed with an ad hoc loopback check that a config containing `{ tools: { profile: "minimal" } }` exposes only `["session_status"]` and `hasCron:false`.
- Impact: Native/manual Codex CLI backend live runs that rely on the backend default MCP probe will fail in the loopback preflight with `mcp loopback tools/list did not expose cron`, even though the probe passed before this commit under the unrestricted/default tool policy.
- Suggested fix: Do not apply the minimal profile when `enableCliMcpProbe` is true, or explicitly disable the MCP probe in the same live-smoke configuration. If the probe must still run, use a profile/allow override that includes `cron`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the failing preflight condition: `resolveCliBackendLiveTest("codex-cli")` returns `defaultMcpProbe:true`, and an MCP loopback check under `tools.profile:"minimal"` exposes no `cron`.
- Is this the best way to solve the issue? Yes: the test config and the probe expectation need to agree. Either keep the MCP probe and allow `cron`, or keep the minimal profile and disable/gate the MCP probe.
