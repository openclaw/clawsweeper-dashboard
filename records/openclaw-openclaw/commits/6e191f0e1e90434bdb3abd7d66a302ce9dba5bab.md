# Commit 6e191f0e1e

## Summary

Found two low-severity diagnostic regressions in the new LM Studio JSON wrapping. Both are actionable because they make current failure messages point users at the wrong cause.

## Findings

### Low: Malformed model-list JSON is still surfaced as “not reachable” in setup

- Kind: bug
- File: `extensions/lmstudio/src/models.fetch.ts`
- Line: 108
- Evidence: `fetchLmstudioModels()` wraps invalid `/api/v1/models` JSON, but the outer catch returns `reachable: false`. The setup path then treats every `!discovery.reachable` result as “LM Studio could not be reached” / `LM Studio not reachable` in `extensions/lmstudio/src/setup.ts:210`.
- Impact: A running LM Studio server that returns HTTP 200 with malformed model JSON causes setup to tell the user to start LM Studio or run `lms server start`, hiding the new owned parse error this commit intended to expose.
- Suggested fix: Preserve a distinct parse-failure state for successful HTTP responses, for example `reachable: true`, `status`, and `error`, or a typed failure reason. Then make setup prefer the owned malformed-JSON message before falling through to unreachable/no-model guidance.
- Confidence: high

### Low: Non-parser body failures are mislabeled as malformed JSON

- Kind: reliability
- File: `extensions/lmstudio/src/models.fetch.ts`
- Line: 107
- Evidence: Both new catch blocks wrap every `response.json()` rejection as malformed JSON. A body timeout or aborted response after headers are received is not malformed JSON, but current code reports `LM Studio model list returned malformed JSON` and `LM Studio model load returned malformed JSON`.
- Impact: Slow or interrupted local LM Studio responses can be diagnosed as bad JSON instead of timeout/transport failures, which sends users and support down the wrong repair path.
- Suggested fix: Only wrap parser failures, such as `SyntaxError`, as malformed JSON. Re-throw abort, timeout, and transport/body-read errors unchanged or wrap them with timeout/transport-specific owned errors.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A local TSX probe showed malformed model-list JSON through `promptAndConfigureLmstudioInteractive()` produces the note `LM Studio could not be reached...` and throws `WizardCancelledError LM Studio not reachable`. A second probe showed a `TimeoutError` thrown from the model-list and model-load JSON body path is wrapped as malformed JSON.
- Is this the best way to solve the issue? yes. The safe fix direction is to distinguish parse failures from reachability/body-read failures at the low-level fetch boundary, then propagate that typed reason to setup/runtime callers.
