# Commit 4647400

## Summary

Found one regression in the new default Discord progress-preview path: default previews start, but the default path does not suppress standalone text-only tool progress messages, so tool-heavy Discord replies can produce both a preview message and separate tool-result messages.

## Findings

### Medium: Default Discord progress previews can duplicate tool progress messages

- Kind: regression
- File: `extensions/discord/src/preview-streaming.ts`
- Line: 14
- Evidence: The commit makes missing Discord `streaming`/`streamMode` resolve to `"progress"` in `extensions/discord/src/preview-streaming.ts:14`. That creates a draft stream in `extensions/discord/src/monitor/message-handler.draft-preview.ts:43`, but `suppressDefaultToolProgressMessages` is still computed through the shared helper in `extensions/discord/src/monitor/message-handler.draft-preview.ts:77`. The shared helper calls `resolveChannelPreviewStreamMode(entry, "off")` and returns `false` for an unset config at `src/plugin-sdk/channel-streaming.ts:589`, before it considers that Discord’s effective mode is now progress. As a result, `extensions/discord/src/monitor/message-handler.process.ts:664` passes `undefined` instead of `true`, and text-only tool results continue through `src/auto-reply/reply/dispatch-from-config.ts:1354` to `dispatcher.sendToolResult` at `src/auto-reply/reply/dispatch-from-config.ts:1372`.
- Impact: With default Discord config, a tool-using run can update the new progress preview and still emit separate tool-result/status messages. That contradicts the changelog/docs promise that work appears in one edited Discord message, and can add noisy duplicate Discord sends/rate-limit pressure for the now-default path.
- Suggested fix: In the Discord draft controller, base suppression on the effective Discord stream mode, e.g. force `suppressDefaultToolProgressMessages` true when `discordStreamMode === "progress"` and a draft stream exists, while preserving the explicit `off` path and media/error/approval exceptions in `dispatch-from-config`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Use default Discord config, process an automatic-source-delivery Discord DM whose agent emits tool progress and a text-only tool result. The controller creates the default progress draft, but reply options omit `suppressDefaultToolProgressMessages`, so `dispatch-from-config` still queues the text-only tool result.
- Is this the best way to solve the issue? yes. The safer fix is local to Discord’s effective-mode handling; changing the shared helper default would affect other channels that intentionally default differently.
