# Commit 109493b

## Summary

Found one actionable compatibility regression in the new plugin payload smoke check. It can fail a valid installed plugin package that uses the documented `openclaw.runtimeExtensions` contract, causing `openclaw update` to exit before restart even though the gateway would load the plugin from its built runtime entry.

## Findings

### Medium: payload smoke check rejects valid runtimeExtensions packages

- Kind: compatibility
- File: `src/cli/update-cli/plugin-payload-validation.ts`
- Line: 105
- Evidence: `runPluginPayloadSmokeCheck` iterates `resolvePackageExtensionEntries(manifest)` and stats `path.join(installPath, extensionRel)`, requiring the source `openclaw.extensions` file to exist. Existing install validation allows that source entry to be absent when a matching `openclaw.runtimeExtensions[index]` built file exists (`src/plugins/package-entry-resolution.ts:154` checks source with `requireExisting: false`, then runtime with `requireExisting: true`). Runtime discovery also resolves `runtimeExtensions` through `resolvePackageRuntimeExtensionSources` (`src/plugins/package-entry-resolution.ts:618`). Docs explicitly recommend `"extensions": ["./src/index.ts"]` plus `"runtimeExtensions": ["./dist/index.js"]` for installed packages (`docs/plugins/sdk-entrypoints.md:14`, `docs/tools/plugin.md:212`).
- Impact: A plugin package with `package.json` containing `openclaw.extensions: ["./src/index.ts"]`, `openclaw.runtimeExtensions: ["./dist/index.js"]`, and only `dist/index.js` shipped is valid by the existing install/discovery contract, but this commit reports `missing-extension-entry` and turns post-core convergence into `status: "error"`. That blocks update completion and leaves a managed package-update flow before the gateway restart path.
- Suggested fix: Reuse the package-entry validation/runtime-resolution helpers for the smoke check, or at least honor `openclaw.runtimeExtensions` when present and keep the same package-boundary checks as `package-entry-resolution.ts`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A temp package with `extensions: ["./src/index.ts"]`, `runtimeExtensions: ["./dist/index.js"]`, and only `dist/index.js` produced `missing-extension-entry` from `runPluginPayloadSmokeCheck`, while the existing plugin install/discovery tests confirm this shape is valid.
- Is this the best way to solve the issue? yes: the smoke check should share the existing plugin package entry contract instead of maintaining a second, narrower interpretation.
