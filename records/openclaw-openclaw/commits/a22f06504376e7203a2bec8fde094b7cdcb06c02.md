# Commit a22f065

## Summary

Found one concrete bug: the new Slack exact-read path still fails for exact thread replies, which is the main behavior the commit claims to fix.

## Findings

### Medium: Exact Slack thread reply reads can return no message

- Kind: bug
- File: `extensions/slack/src/actions.ts`
- Line: 266
- Evidence: When `messageId` is set, `readSlackMessages` builds `{ inclusive: true, latest: exactMessageId, oldest: undefined }` and forces `limit: 1`; the thread path then calls `conversations.replies` with the parent `threadId` and filters the returned page to `message.ts === exactMessageId`. Slack documents `conversations.replies` time pagination as returning the earliest messages in the range first, while `conversations.history` returns most recent first. With a parent thread timestamp older than the target reply timestamp, `conversations.replies({ ts: parent, latest: reply, inclusive: true, limit: 1 })` can return the parent as the single earliest message; this code then filters it out and returns `messages: []`. The added test mocks both parent and reply despite `limit: 1`, so it does not model the documented API behavior. 
- Impact: `openclaw message read --channel slack --target <channel> --thread-id <parent-ts> --message-id <reply-ts>` and agent `message(action: "read", ...)` calls for Slack thread-reply URLs can still fail to fetch the requested reply, leaving issue #53943 effectively unresolved for thread replies.
- Suggested fix: For exact thread reply reads, use a lower bound around the requested reply timestamp, e.g. `oldest: exactMessageId`, `inclusive: true`, `limit: 1`, then keep the exact `ts` filter. Alternatively set both bounds to the exact timestamp if Slack accepts that consistently for `conversations.replies`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Use a Slack thread where parent `threadId` is older than reply `messageId`, then call the Slack read action with both values. The current code requests the page ending at the reply with `limit: 1`; Slack’s documented replies ordering means the parent is the first item in that window, and the exact filter drops it.
- Is this the best way to solve the issue? yes. The fix should stay in `readSlackMessages`, because the dispatcher and CLI now correctly forward `messageId`; the broken behavior is the Slack API bounds used by the helper.
