# Commit 2000227e9e22

## Summary

The commit correctly preserves native Ollama assistant `tool_calls[].id` values through ingest and replay, but it also changes the native Ollama replay policy so the embedded-runner request-time sanitizer block no longer runs. That block was the only per-request path applying tool-result pairing repair for internal tool-call cycles.

## Findings

### Medium: Native Ollama skips request-time tool-result pairing repair

- Kind: regression
- File: `src/agents/pi-embedded-runner/run/attempt.ts`
- Line: 2788
- Evidence: Native Ollama now returns a replay policy with `sanitizeToolCallIds: false` and no `toolCallIdMode` in `extensions/ollama/index.ts:70`. `shouldApplyReplayToolCallIdSanitizer` therefore returns false in `src/agents/pi-embedded-runner/run/attempt.tool-call-normalization.ts:888`, so `runEmbeddedAttempt` skips the whole wrapper at `src/agents/pi-embedded-runner/run/attempt.ts:2788`. The skipped wrapper is also where `repairToolUseResultPairing: transcriptPolicy.repairToolUseResultPairing` is passed to `sanitizeReplayToolCallIdsForStream` at `src/agents/pi-embedded-runner/run/attempt.ts:2797`; that helper applies `sanitizeToolUseResultPairing` at `src/agents/pi-embedded-runner/run/attempt.tool-call-normalization.ts:909`. The merged transcript default still has `repairToolUseResultPairing: true` in `src/agents/transcript-policy.ts:55`.
- Impact: A native Ollama turn with a missing, displaced, or orphaned tool result inside the agent loop is no longer repaired before the next provider request. Before this commit, native Ollama used strict ID sanitization, so this per-request wrapper ran and could synthesize/drop/move tool results. After this commit, the same repair is silently skipped, which can surface as provider rejection or broken continuation on the exact Ollama/Gemini tool replay path this commit is trying to stabilize.
- Suggested fix: Decouple pairing repair from strict ID rewriting. Keep `sanitizeToolCallIds: false` for native Ollama IDs, but still run a request-time pairing repair wrapper when `transcriptPolicy.repairToolUseResultPairing` is true, or make the existing wrapper gate true when either ID sanitization or pairing repair is needed.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the source-level regression: a direct probe shows native Ollama policy has `sanitizeToolCallIds:false`, the wrapper gate returns false, while `sanitizeReplayToolCallIdsForStream(..., repairToolUseResultPairing:true)` would add the missing `toolResult` if reached. Live provider rejection was not reproduced.
- Is this the best way to solve the issue? The fix direction is clear: preserve native IDs without tying pairing repair to ID sanitization. The safer shape is a separate pairing-repair pass so native Ollama does not re-enter strict ID rewriting.
