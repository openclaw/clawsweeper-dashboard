## Findings

### Low: Changelog example drops the regex literal-dot escape

- Kind: regression
- File: `CHANGELOG.md`
- Line: 395
- Evidence: The commit changes the existing exec approvals example from `argPattern: "^safe\\.py$"` to `argPattern: "^safe\.py$"`. In a JavaScript-style config string, the single-backslash form evaluates to `^safe.py$`, so the dot becomes a wildcard; a focused Node check showed `new RegExp("^safe.py$").test("safeXpy") === true`, while `new RegExp("^safe\\.py$").test("safeXpy") === false`.
- Impact: Users copying the changelog example for the security-relevant `argPattern` may get a broader allowlist pattern than intended, or an invalid escape in JSON-shaped config.
- Suggested fix: Restore the original example escape as `argPattern: "^safe\\.py$"` while keeping the new compaction changelog entry.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. The string-evaluation smoke check above demonstrates the copied pattern no longer encodes a literal dot.
- Is this the best way to solve the issue? Yes. This is an unrelated changelog-only regression; restoring the previous escape is the narrow fix.
