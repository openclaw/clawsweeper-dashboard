# Commit 5457462e62

## Summary

Found one realtime Discord voice race that can leave short authorized speaker turns stuck or dropped when the speaker stops before async ingress authorization finishes.

## Findings

### Medium: Realtime voice can miss speaker-end finalization while waiting for authorization

- Kind: reliability
- File: `extensions/discord/src/voice/manager.ts`
- Line: 582
- Evidence: In realtime modes, `handleSpeakingStart` awaits `resolveDiscordVoiceIngressContext` before it subscribes to the Discord receiver or calls `beginVoiceCapture` (`extensions/discord/src/voice/manager.ts:582`). The separate `"end"` handler only calls `scheduleCaptureFinalize` (`extensions/discord/src/voice/manager.ts:426`), and `scheduleVoiceCaptureFinalize` returns immediately when no active capture stream exists yet (`extensions/discord/src/voice/capture-state.ts:102`). `resolveDiscordVoiceIngressContext` can perform async guild/member lookups before returning (`extensions/discord/src/voice/ingress.ts:35`, `extensions/discord/src/voice/ingress.ts:38`).
- Impact: For a short utterance, or a slow Discord member/guild fetch, `"speaking.end"` can fire during the await. Because no capture exists yet, no finalize timer is recorded. After authorization returns, the code subscribes with `EndBehaviorType.Manual` and waits for `decodeOpusStreamChunks` to finish, but the missed end event means nothing later destroys that stream. The realtime turn can hang, keep the speaker marked active, and prevent later speech from that user from being captured correctly.
- Suggested fix: Preserve the â€śauthorize before forwarding audioâ€ť behavior, but close the race by tracking an end event that arrives while realtime authorization is pending and applying the normal finalize/destroy path immediately after `beginVoiceCapture`; alternatively subscribe/begin capture before the await and destroy the stream without forwarding audio if authorization fails.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Configure `voice.mode: "talk-buffer"` or `"bidi"`, make `fetchGuild`/`fetchMember` slow, emit `speaking.start`, then emit `speaking.end` before `resolveDiscordVoiceIngressContext` resolves. `scheduleCaptureFinalize` observes no active capture and does nothing; after the await, the manual receiver stream has no finalize timer.
- Is this the best way to solve the issue? yes for the suggested direction. The core invariant should be that every accepted `speaking.start` has a matching finalize/cleanup path even when authorization is async.
