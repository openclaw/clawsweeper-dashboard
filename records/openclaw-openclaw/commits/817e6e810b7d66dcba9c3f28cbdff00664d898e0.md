# Commit 817e6e8

## Summary

Found one reliability regression in the new transcript-only announce path: it bypasses the existing `agent.wait` timeout and can deliver stale A2A announce messages after the configured announce timeout.

## Findings

### Medium: Transcript-only announce steps ignore `announceTimeoutMs`

- Kind: reliability
- File: `src/agents/tools/agent-step.ts`
- Line: 67
- Evidence: When `transcriptMessage` is present, `runAgentStep` awaits `agentCommandFromIngress(...)` directly and never uses `params.timeoutMs`. The non-transcript branch still calls `waitForAgentRunAndReadUpdatedAssistantReply` with `Math.min(params.timeoutMs, 60_000)` at `src/agents/tools/agent-step.ts:103`. `runSessionsSendA2AFlow` passes `announceTimeoutMs` into this transcript path for the final announce step at `src/agents/tools/sessions-send-tool.a2a.ts:125`. Because no `timeout` is passed to `agentCommandFromIngress`, nested lanes default to `timeoutSecondsRaw = 0` in `src/agents/agent-command.ts:328`, which `src/agents/timeout.ts:25` treats as the maximum timer-safe “no timeout” value.
- Impact: Before this commit, a slow announce-synthesis turn would stop affecting the A2A flow after the announce wait timeout and return no channel send. After this commit, the background A2A flow can wait until the direct agent run eventually completes and then send the announce reply late, potentially minutes or hours after the original `sessions_send` timeout window. A stuck announce run can also keep the fire-and-forget A2A promise alive far longer than intended.
- Suggested fix: Preserve the transcript override but restore the timeout contract for the direct path. For example, wrap `agentCommandFromIngress` with an `AbortController`/timer using `Math.min(params.timeoutMs, 60_000)`, pass `abortSignal`, and return `undefined` on timeout; or add an internal gateway-only path that accepts `transcriptMessage` while keeping the existing `agent.wait` behavior.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: a focused unit test can stub `agentCommandFromIngress` to resolve after a delay longer than `timeoutMs`, call `runAgentStep({ transcriptMessage: "", timeoutMs: 10, ... })`, and assert that it returns the late payload instead of timing out. Existing tests cover forwarding `transcriptMessage` but do not assert timeout behavior.
- Is this the best way to solve the issue? yes: the timeout belongs at the new direct execution branch, because the branch exists specifically to avoid exposing `transcriptMessage` through public gateway params.
