## Findings

### Medium: Runtime parity tier metadata does not drive gate selection

- Kind: test_gap
- File: `extensions/qa-lab/src/agentic-parity.ts`
- Line: 89
- Evidence: `qa/scenarios/runtime/first-hour-20-turn.md:7` marks the new 20-turn scenario as `runtimeParityTier: standard`, and its objective says it verifies the required 20-turn maintainer gate. But the release runtime parity workflow still runs `pnpm openclaw qa suite --parity-pack agentic --runtime-pair pi,codex` in `.github/workflows/openclaw-release-checks.yml:875`, and `--parity-pack agentic` expands only the hard-coded `QA_AGENTIC_PARITY_SCENARIO_IDS` list in `extensions/qa-lab/src/agentic-parity.ts:3`; that list does not include `runtime-first-hour-20-turn`. The generic selector in `extensions/qa-lab/src/suite-planning.ts:79` only filters by explicit IDs/live-lane requirements, not `runtimeParityTier`.
- Impact: Maintainers can run the documented `qa-parity` release group and get a green runtime-parity lane without exercising the newly added 20-turn same-session depth scenario. That leaves the exact long-session regression this commit is meant to catch outside the standard gate.
- Suggested fix: Add a runtime-parity tier selector to the CLI/workflow, or make the runtime parity release lane expand scenarios with `runtimeParityTier: standard` while keeping `soak` excluded unless explicitly requested. Add a CLI/unit test that proves `runtime-first-hour-20-turn` is selected for the standard runtime-pair gate and `runtime-soak-100-turn` is not.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The focused CLI tests passed and still assert `--parity-pack agentic` expands only the hard-coded agentic list; source inspection shows the release runtime-pair command uses that pack and no tier selector.
- Is this the best way to solve the issue? yes. The new metadata should become the selection contract; otherwise it is only catalog decoration and can drift from CI/release behavior.
