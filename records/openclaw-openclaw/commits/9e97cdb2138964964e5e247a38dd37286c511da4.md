# Commit 9e97cdb

## Summary

Found one low-severity regression risk: Discord now treats every zero-result shared outbound delivery as a failed final reply, but shared outbound can return zero results for intentional `message_sending` hook cancellation.

## Findings

### Low: Intentional outbound hook cancellation is reported as Discord final-delivery failure

- Kind: regression
- File: `extensions/discord/src/monitor/reply-delivery.ts`
- Line: 208
- Evidence: `deliverDiscordReply` now throws whenever `deliverOutboundPayloads(...)` returns an empty result list. Shared outbound explicitly allows `message_sending` hooks to cancel outgoing messages (`src/plugins/hooks.ts:1014`) and returns `{ cancelled: true }` from `applyMessageSendingHook` when a hook returns `{ cancel: true }` (`src/infra/outbound/deliver.ts:779`). The outbound loop handles that by `continue` without sending or adding a result (`src/infra/outbound/deliver.ts:1074`-`1087`). That exception is then counted as `failedCounts.final` by the dispatcher and Discord sets the error reaction (`extensions/discord/src/monitor/message-handler.process.ts:805`-`815`).
- Impact: A plugin or hook that intentionally suppresses a Discord outbound reply can now produce an error reaction and delivery-failure log instead of a normal cancellation/no-send outcome. This also diverges from the generic dispatcher path, where cancellations are subtracted via `cancelledCounts` rather than surfaced as failures.
- Suggested fix: Distinguish “send attempted but no message id returned” from “delivery suppressed before send.” One narrow option is to set an `attemptedDelivery` flag via `deliverOutboundPayloads`’s `onPayload` callback and only throw on `results.length === 0` when a payload reached the actual delivery phase. A fuller fix would expose a structured suppressed/cancelled result from shared outbound.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. Register a `message_sending` hook that returns `{ cancel: true }`, then run an automatic Discord final reply. The source path deterministically returns no outbound result before any send attempt, and this commit converts that zero-result cancellation into a final delivery failure.
- Is this the best way to solve the issue? The commit is a good fix for adapter paths that silently accept a final reply but return no delivered message. The safer version should preserve that behavior while excluding known pre-send suppression/cancellation paths.
