# Commit 55cfe00

## Summary

Found two lifecycle regressions in the new auth-profile watcher rewarm path. The main issue is that the watcher is registered as a shutdown sidecar too late, so shutdown/restart can still leave it running.

## Findings

### Medium: Auth-profile watcher can be orphaned when shutdown races post-ready setup

- Kind: reliability
- File: `src/gateway/server-startup-post-attach.ts`
- Line: 1054
- Evidence: The auth watcher is created inside a detached `sidecarsPromise.then(async ...)` after several dynamic imports, then appended to `sidecarsResult.postReadySidecars`. But `onPostReadySidecars` already captured that array earlier in the sidecars-ready path, and close clears `runtimeState.postReadySidecars` before stopping the captured handles at `src/gateway/server.impl.ts:977` and `src/gateway/server.impl.ts:1673`. If close starts before this detached callback reaches line 1054, the late push goes into an array that has already been stopped and removed from runtime state.
- Impact: A gateway close/restart immediately after sidecars become ready can still leave the chokidar auth-profile watcher alive, keeping file descriptors/event-loop handles around after shutdown. This directly undermines the commit’s “close on shutdown” fix.
- Suggested fix: Register a placeholder/handle in `postReadySidecars` before `onPostReadySidecars` can fire, or route late sidecar registration through a callback that checks `closePreludeStarted` and stops immediately when the gateway is already closing. Avoid appending lifecycle-owned handles to the sidecar array from a detached post-resolution continuation.
- Confidence: high

### Low: Auth failure rewarm uses the startup config after hot reloads

- Kind: regression
- File: `src/gateway/server-startup-post-attach.ts`
- Line: 1032
- Evidence: `scheduleAuthMapRewarm` always calls `warmCurrentProviderAuthState(params.cfgAtStart)`. Hot reload explicitly clears and warms the current config at `src/gateway/server-reload-handlers.ts:93` and `src/gateway/server-reload-handlers.ts:503`, but the global auth-profile failure hook is not re-bound with `nextConfig`.
- Impact: After a config hot reload changes agents/providers/model catalog inputs, an auth-profile failure clears the current prepared auth map and then publishes a map for the old startup config. The fingerprint checks prevent wrong auth answers, but current-config model listing falls back to slow provider discovery until another reload path refreshes it.
- Suggested fix: Make the failure hook/rewarm path read the current runtime config, or update the hook and watcher ownership when hot reload installs `nextConfig`.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes. For the shutdown race, delay the auth watcher dynamic import or `watchAuthProfilesForChanges`, let `sidecarsPromise` resolve, call gateway `close()`, then release the delayed import and assert the watcher’s `stop` is not called. For the stale-config case, hot reload to a distinct config, invoke `markAuthProfileFailure`, and observe the rewarm call still receives the startup config.
- Is this the best way to solve the issue? yes for the first finding: lifecycle handles should be registered before shutdown can clear the sidecar list, or registered through a close-aware owner. For the second, the safer direction is to use the current runtime config rather than preserving `cfgAtStart` in a global failure hook.
