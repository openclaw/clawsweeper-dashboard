# Commit ac75d6f

## Summary

Found one regression: the new reply-chain prompt block is not recognized by the metadata stripper used before displaying stored user messages.

## Findings

### Medium: Reply-chain metadata can leak into user-visible chat history

- Kind: privacy
- File: `src/auto-reply/reply/inbound-meta.ts`
- Line: 311
- Evidence: This commit adds the new sentinel `Reply chain of current user message (untrusted, nearest first):`, but `src/auto-reply/reply/strip-inbound-meta.ts:21` still lists only the older sentinels and explicitly says that list must stay in sync with `buildInboundUserContextPrefix`. A direct smoke confirmed `stripInboundMetadata(buildInboundUserContextPrefix({ ReplyChain: [...] }) + "\n\nActual user text")` returns the reply-chain JSON block plus the user text.
- Impact: AI-facing metadata now survives stripping in TUI/webchat/macOS/session-history surfaces. For Telegram reply chains this can expose quoted message bodies, sender IDs/usernames, and media refs/paths in user-visible history instead of showing only the actual user message.
- Suggested fix: Add the new reply-chain sentinel to `INBOUND_META_SENTINELS` and add builder-compatibility tests that pass a generated `ReplyChain` prefix through `stripInboundMetadata` and `stripLeadingInboundMetadata`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: the direct builder/stripper smoke on current `main` returned `containsReplyChain: true` and showed the full reply-chain JSON before `Actual user text`.
- Is this the best way to solve the issue? yes: the existing stripper already uses a central sentinel list for this exact contract; the new block just needs to be registered there and covered by tests.
