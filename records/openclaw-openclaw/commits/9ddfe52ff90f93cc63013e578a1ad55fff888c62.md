# Commit 9ddfe52

## Summary

Found two reliability issues in the new Google Meet listen-health probe path. Both are actionable but not high/critical severity.

## Findings

### Medium: Agent tool timeout can expire before `test_listen` can return its structured result

- Kind: reliability
- File: `extensions/google-meet/index.ts`
- Line: 442
- Evidence: `callGoogleMeetGatewayFromTool` always sets the gateway RPC timeout from `resolveGoogleMeetToolGatewayTimeoutMs(config)`, which only considers config join/voice-call timeouts. The new `test_listen` action forwards user `timeoutMs` to the gateway method, and runtime clamps it up to 120000 ms, but the agent-facing gateway call still defaults to 60000 ms. A tool call like `{"action":"test_listen","timeoutMs":120000}` can therefore time out in the gateway client before `GoogleMeetRuntime.testListen` returns `listenTimedOut`.
- Impact: Agents may get a generic RPC timeout instead of the intended `listenVerified` / `listenTimedOut` / manual-action health payload, especially for quiet meetings or intentionally longer probes.
- Suggested fix: Make the tool RPC timeout action-aware and include normalized `raw.timeoutMs` for `test_listen`, plus join overhead. For example, use `max(base, timeoutMs + config.chrome.joinTimeoutMs + buffer)` with the same 120s cap semantics as runtime.
- Confidence: high

### Medium: Reused transcribe sessions can verify old caption backlog as fresh listen movement

- Kind: reliability
- File: `extensions/google-meet/src/runtime.ts`
- Line: 673
- Evidence: `testListen` takes `start = transcriptCheckpoint(existingSession?.chrome?.health)` before `join()` refreshes a reusable transcribe session. For transcribe sessions, caption lines are accumulated in the browser page and only copied into `session.chrome.health` when OpenClaw refreshes browser health. If captions arrived after the last OpenClaw status refresh but before `test_listen` starts, the stale in-memory checkpoint can be `0`; then `join()` refreshes the reusable tab and `hasTranscriptAdvanced` returns true immediately even if no caption moved during this probe.
- Impact: The yes/no listen probe can report `listenVerified: true` from pre-existing caption text, so an operator or agent may believe current listening is healthy when the meeting has gone quiet or captioning has stopped.
- Suggested fix: When an existing reusable transcribe session is found, refresh its caption/browser health before taking the checkpoint, then wait for movement beyond that refreshed checkpoint.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the timeout path by calling the agent tool with `test_listen` and a `timeoutMs` larger than the fixed tool RPC timeout while the meeting stays quiet. The stale-checkpoint issue is reproducible with a reused transcribe tab whose browser caption observer has accumulated lines since the last runtime health refresh.
- Is this the best way to solve the issue? Yes: keep timeout budgeting at the agent gateway call boundary, and take the transcript checkpoint after refreshing reusable session state.
