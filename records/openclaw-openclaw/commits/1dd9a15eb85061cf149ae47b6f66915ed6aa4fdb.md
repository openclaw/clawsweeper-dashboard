# Commit 1dd9a15

## Summary

Found one concrete follow-up issue in the Discord target-resolution path: the new explicit `discord:channel:<id>` parse can still be overridden by a live user directory hit before sending.

## Findings

### Medium: Explicit Discord channel targets can still resolve to a user directory hit

- Kind: privacy
- File: `extensions/discord/src/target-resolver.ts`
- Line: 24
- Evidence: `parseDiscordTarget()` now correctly parses `discord:channel:555` as `{ kind: "channel", id: "555" }`, but `resolveDiscordTarget()` still sets `shouldLookup` through `isExplicitUserLookup()`, which treats every `discord:` prefix as a user lookup. Because direct channel parses are not returned at `directParse && directParse.kind !== "channel"`, the resolver calls `listDiscordDirectoryPeersLive()` and returns the first user match before falling back to the parsed channel.
- Impact: A send to an explicit channel target like `discord:channel:<channelId>` can be misdirected to a DM if Discord member search returns a user whose username/nickname/display name matches that query. That can leak channel-intended content to the wrong user.
- Suggested fix: Treat `discord:channel:` as an explicit channel target in `resolveDiscordTarget()` and return the direct parse before user-directory lookup. Add a focused test that mocks `listDiscordDirectoryPeersLive()` to return a user for `discord:channel:555` and asserts the resolver still returns `channel:555`.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the deterministic code path: mock `listDiscordDirectoryPeersLive()` to return `[{ kind: "user", id: "user:999" }]`, then call `resolveDiscordTarget("discord:channel:555", { cfg, accountId: "default" }, { defaultKind: "channel" })`; current code returns user `999` instead of channel `555`. I did not run a live Discord API collision.
- Is this the best way to solve the issue? Yes. The parser already knows the target is explicitly a channel, so the resolver should not run user lookup for that case.
