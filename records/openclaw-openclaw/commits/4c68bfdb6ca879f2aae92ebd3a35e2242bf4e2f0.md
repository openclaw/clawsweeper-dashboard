# Commit 4c68bfdb6c

## Summary

Found two release-validation filter bugs. The intended focused QA rerun works for `rerun_group=qa-live`, but the same newly advertised QA filter breaks broader release-check dispatches and accepts typoed QA lane names without failing.

## Findings

### Medium: QA live filters are still forwarded to the live/E2E reusable workflow on broad release-check runs

- Kind: regression
- File: `.github/workflows/openclaw-release-checks.yml`
- Line: 434
- Evidence: `live_suite_filter` now advertises QA lane values such as `qa-live-matrix,qa-live-telegram` at `.github/workflows/openclaw-release-checks.yml:57`, but `live_repo_e2e_release_checks` still runs for `rerun_group=all` and forwards the raw value to `openclaw-live-and-e2e-checks-reusable.yml` at `.github/workflows/openclaw-release-checks.yml:420` and `.github/workflows/openclaw-release-checks.yml:434`. That reusable workflow validates any non-empty filter against live/E2E suite IDs only and fails when the filter is not in that list at `.github/workflows/openclaw-live-and-e2e-checks-reusable.yml:347` and `.github/workflows/openclaw-live-and-e2e-checks-reusable.yml:446`.
- Impact: A manually dispatched `OpenClaw Release Checks` run with default `rerun_group=all`, or a `Full Release Validation` run with `rerun_group=release-checks`, fails if the operator uses the newly documented `live_suite_filter=qa-live-matrix` or `qa-live-matrix,qa-live-telegram` to skip Slack. That blocks the broad release validation path instead of only filtering QA live lanes.
- Suggested fix: Split the inputs, or derive separate outputs such as `live_e2e_suite_filter` and `qa_live_lane_filter`. When the filter contains QA lane tokens, apply it only to the QA live jobs and pass an empty live/E2E filter to the reusable workflow for broad groups.
- Confidence: high

### Low: Unknown QA lane tokens are silently ignored after any recognized QA token

- Kind: reliability
- File: `.github/workflows/openclaw-release-checks.yml`
- Line: 231
- Evidence: The parser only handles known `case` branches and has no default rejection path. If an operator enters `qa-live-matrix,qa-live-telegarm`, the recognized Matrix token sets `qa_filter_seen=true`, then the misspelled Telegram token is ignored, and the outputs become Matrix `true`, Telegram `false`, Slack `false` at `.github/workflows/openclaw-release-checks.yml:258`. For `rerun_group=qa-live`, no live/E2E filter validation runs, skipped QA jobs are accepted by the summary at `.github/workflows/openclaw-release-checks.yml:1015`.
- Impact: A typo can produce a green focused release check with intended QA coverage silently omitted.
- Suggested fix: Collect unrecognized tokens and fail when `rerun_group` includes `qa`/`qa-live` and the filter is being interpreted as a QA lane selector. At minimum, reject tokens starting with `qa-` that do not match the accepted aliases.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Static workflow tracing shows the broad-run failure path, and a focused shell simulation of the reusable filter validation rejects `qa-live-matrix` as not matching any runnable live/E2E suite. A parser simulation also shows `qa-live-matrix,qa-live-telegarm` selecting only Matrix without an error.
- Is this the best way to solve the issue? yes. Separating live/E2E suite filtering from QA live lane filtering avoids overloading one input across two validators; explicit unknown-token rejection prevents false-green focused QA runs.
