# Commit 3c4b482

## Summary

Found one concrete reply-chain bug that persists on current `main`: Telegram `external_reply` messages can hydrate from the current chat’s cache even though Telegram marks them as replies to another chat or forum topic.

## Findings

### Medium: External replies can pull unrelated same-chat cached messages

- Kind: bug
- File: `extensions/telegram/src/message-cache.ts`
- Line: 278
- Evidence: `resolveReplyMessage` collapses `reply_to_message` and `external_reply` into one value, then `buildTelegramReplyChain` always looks up `replyMessage.message_id` using `params.chatId`. The installed `@grammyjs/types` definitions distinguish these cases: `reply_to_message` is for the same chat/thread, while `external_reply` “may come from another chat or forum topic” and carries its own optional `chat` plus `message_id`.
- Impact: If chat A has cached message `42`, and a later message in chat A externally replies to a channel/supergroup message that also has `message_id: 42`, the reply chain will use chat A’s cached message instead of the external reply metadata. That can give the agent the wrong quoted context or media reference and produce incorrect replies.
- Suggested fix: Preserve reply source when resolving replies. Only use the current-chat cache for `reply_to_message`; for `external_reply`, either use `external_reply.chat.id` as the cache scope when present or skip same-chat cache hydration and normalize the embedded external reply info directly.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes conceptually: cache a current-chat message with id `N`, then process a Telegram update in the same chat whose `external_reply.message_id` is also `N` but whose `external_reply.chat.id` is different. The code path will query `cache.get({ chatId: currentChatId, messageId: N })` before normalizing the external reply.
- Is this the best way to solve the issue? Yes: the cache key needs the true reply source scope. Treating external replies as current-chat replies is the root mismatch.
