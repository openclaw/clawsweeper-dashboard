# Commit 0e8cb3d

## Summary

Found one Parallels smoke harness regression. The commit changed timeout setup from a provider-agnostic `models.providers.<id>.timeoutSeconds` write to an OpenAI-only full provider config helper, so Anthropic and MiniMax lanes silently skip the intended 600s model timeout.

## Findings

### Low: Non-OpenAI Parallels lanes no longer configure the model timeout

- Kind: regression
- File: `scripts/e2e/parallels/provider-auth.ts`
- Line: 91
- Evidence: `providerTimeoutConfigJson()` returns `""` for every provider except `openai`, and all call sites skip timeout setup when that string is empty. Before this commit, the macOS/Linux/npm-update/Windows helpers set `models.providers.${providerId}.timeoutSeconds` for any valid provider id. Current `main` still has the OpenAI-only guard. A focused eval confirmed `providerTimeoutConfigJson("anthropic/claude-sonnet-4-6", "macos")` and `providerTimeoutConfigJson("minimax/MiniMax-M2.7", "macos")` both return empty while the configured harness timeout is 600.
- Impact: `--provider anthropic` and `--provider minimax` Parallels smoke agent turns fall back to the runner’s default 120s LLM idle timeout instead of the intended 600s/provider override. Slow first responses can now fail the macOS, Linux, Windows, and aggregate npm-update provider lanes despite the harness timeout envs.
- Suggested fix: keep the OpenAI full provider config path where needed, but preserve the old provider-agnostic timeout write for non-OpenAI providers, for example by returning a structured config operation and emitting `config set models.providers.<providerId>.timeoutSeconds <seconds> --strict-json` when no OpenAI catalog injection is needed.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the config omission: evaluate `providerTimeoutConfigJson()` for `anthropic/*` or `minimax/*` and inspect the call sites that skip on empty config. A live VM timeout failure would require a slow provider response, but the timeout configuration regression itself is deterministic.
- Is this the best way to solve the issue? Yes: restoring the non-OpenAI nested timeout write preserves the prior contract without needing to synthesize complete Anthropic/MiniMax provider configs.
