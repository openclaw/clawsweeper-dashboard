# Commit a3f2f3a

## Summary

Found one low-severity contract bug in the newly added ClawHub package security helper. The helper’s declared response type and test fixture require a top-level `state`, but the default live ClawHub endpoint currently returns a `{ package, version }` payload with scan data nested under the version, so callers would get `undefined` for a field TypeScript says is always present.

## Findings

### Low: `fetchClawHubPackageSecurity` does not match the live ClawHub response shape

- Kind: bug
- File: `src/infra/clawhub.ts`
- Line: 118
- Evidence: `ClawHubPackageSecurityResponse` requires `state` at `src/infra/clawhub.ts:121`, and the new test mocks that shape at `src/infra/clawhub.test.ts:305`. A live check against `https://clawhub.ai/api/v1/packages/%40openclaw%2Fdiagnostics-otel/versions/2026.3.22/security` via the new helper returned `state: null` and top-level keys only `["package","version"]`.
- Impact: Any future caller that trusts `response.state` for install, moderation, warning, or display decisions will read `undefined` despite the type promising a concrete security state. That can hide scan state or make downstream policy code branch incorrectly.
- Suggested fix: Align the type and test fixture with the current package security endpoint payload, or change the endpoint/parser to normalize the live `{ package, version }` response into an explicit top-level security summary before returning it.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: `pnpm exec tsx -e "import { fetchClawHubPackageSecurity } from './src/infra/clawhub.ts'; void (async () => { const r = await fetchClawHubPackageSecurity({ name: '@openclaw/diagnostics-otel', version: '2026.3.22', timeoutMs: 20000 }); console.log(JSON.stringify({ state: (r as { state?: unknown }).state ?? null, topLevelKeys: Object.keys(r).sort() }, null, 2)); })();"` prints `state: null` and only `package`/`version` top-level keys.
- Is this the best way to solve the issue? unclear: if the ClawHub service is about to change, keeping the helper may be fine, but current `main` should not expose a required field that the default service does not return.
