# Commit d06f0a0

## Summary

Found one regression in the PowerShell git installer path. The patch correctly tries to make `git pull` best-effort, but it also treats a failed `git status` as a clean checkout and then runs `pnpm` inside whatever directory `-GitDir` points at.

## Findings

### Medium: Failed git status now falls through to pnpm in a non-git directory

- Kind: regression
- File: `scripts/install.ps1`
- Line: 575
- Evidence: `try { $dirty = git -C $RepoDir status --porcelain 2>$null } catch {}` leaves `$dirty = $null` when `git status` fails, then `if (-not $dirty)` takes the clean-checkout branch and the function continues to `& $pnpmCommand -C $RepoDir install`, `ui:build`, and `build` at `scripts/install.ps1:594`. A focused PowerShell probe with `$PSNativeCommandUseErrorActionPreference = $true` showed the base commit stopped on `git status` exit 128 with no `pnpm` calls, while this commit returned success and invoked `pnpm -C <non-git-dir> install`, `ui:build`, and `build`.
- Impact: `install.ps1 -InstallMethod git` can now run package install/build scripts in an existing directory that is not an OpenClaw checkout, then install an `openclaw.cmd` wrapper pointing at that unrelated directoryŌĆÖs `dist\entry.js`. This can produce false-success installs and unexpected script execution for the default/custom git directory.
- Suggested fix: validate that an existing `RepoDir` is a git/OpenClaw checkout before the update block, or treat `git status` failure as a hard failure/not-clean state. Keep only `git pull --rebase` best-effort after status succeeds. Add a regression test in `test/scripts/install-ps1.test.ts` with `$PSNativeCommandUseErrorActionPreference = $true` and a fake `pnpm` asserting that non-git directories do not reach `pnpm`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Use a non-git existing `RepoDir`, set `$PSNativeCommandUseErrorActionPreference = $true`, stub `pnpm`, and call `Install-OpenClawFromGit`; the merged commit logs `install`, `ui:build`, and `build`, while the base commit stops before `pnpm`.
- Is this the best way to solve the issue? yes. The intended behavior is best-effort pull, not best-effort checkout validation; status failure should not be interpreted as a clean checkout.
