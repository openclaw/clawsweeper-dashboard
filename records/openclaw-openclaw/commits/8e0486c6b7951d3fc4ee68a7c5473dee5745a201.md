# Commit 8e0486c6b7

## Summary

Found two remaining timeout-contract regressions in the Codex dynamic-tool bridge. The commit fixes `timeoutMs`, `image_generate`, and the global `tools.media.image.timeoutSeconds` path, but still misses tool contracts that use `timeoutSeconds` and the `image` tool’s per-model media timeout overrides.

## Findings

### Medium: Dynamic tools with `timeoutSeconds` still hit the 30s Codex bridge default

- Kind: regression
- File: `extensions/codex/src/app-server/run-attempt.ts`
- Line: 1649
- Evidence: `readDynamicToolCallTimeoutMs()` only reads `arguments.timeoutMs`, then falls back to configured special cases or `CODEX_DYNAMIC_TOOL_TIMEOUT_MS` at 30s. Codex exposes each OpenClaw tool’s real schema as `inputSchema`, so tools such as `sessions_send` advertise `timeoutSeconds`, not `timeoutMs`. `sessions_send` then converts that value into the actual wait budget and passes it to `waitForAgentRunAndReadUpdatedAssistantReply`.
- Impact: A Codex call like `sessions_send` with `timeoutSeconds: 120` is a valid tool call and the tool will wait up to 120s, but the Codex bridge still returns a failed dynamic-tool response after 30s. This preserves the same false-timeout class reported in #79810 for non-image tools.
- Suggested fix: teach the bridge to honor known per-call timeout fields such as `timeoutSeconds` and `runTimeoutSeconds` where those are actual tool wait budgets, or move timeout hints into tool metadata so the Codex bridge does not guess from argument names.
- Confidence: high

### Medium: `image` dynamic-tool calls ignore per-model media timeout overrides

- Kind: regression
- File: `extensions/codex/src/app-server/run-attempt.ts`
- Line: 1666
- Evidence: the new `toolName === "image"` branch only reads `tools.media.image.timeoutSeconds` and otherwise uses 60s. The actual `image` tool resolves a more specific timeout from matching `tools.media.image.models[]` or shared `tools.media.models[]` entries before falling back to the global image timeout, and existing tests verify a matching model entry uses 300000 ms.
- Impact: Configs that follow the documented per-model media timeout pattern, for example slow Ollama image models with `tools.media.image.models[].timeoutSeconds: 300`, still get aborted by Codex at 60s or the lower global image timeout. The provider call can legitimately still be in flight, recreating the false failure and retry fragmentation this commit is meant to stop.
- Suggested fix: share the `image` tool’s timeout resolution with the Codex bridge, or carry a prepared tool-timeout hint from tool construction. At minimum, the bridge should consider matching `tools.media.image.models[]` and shared `tools.media.models[]` image entries before applying the 60s default.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes, by source path: call Codex `item/tool/call` for `sessions_send` with `timeoutSeconds > 30`, or for `image` with only a matching per-model media timeout above 60s. The bridge deadline resolves lower than the tool’s own deadline before execution starts.
- Is this the best way to solve the issue? No. The commit is a good partial fix, but the safer direction is to keep the bridge deadline aligned with the tool-owned timeout contract instead of duplicating only selected timeout config branches.
