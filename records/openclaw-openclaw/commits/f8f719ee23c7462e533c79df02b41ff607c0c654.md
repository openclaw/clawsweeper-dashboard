# Commit f8f719e

## Summary

Found one regression in the Google realtime voice lazy bridge wrapper: early bridge calls now throw before the dynamically imported provider has been created, while the previous concrete bridge buffered early audio.

## Findings

### Medium: Lazy Google realtime bridge drops early audio before provider import completes

- Kind: regression
- File: `extensions/google/index.ts`
- Line: 228
- Evidence: `createLazyGoogleRealtimeVoiceBridge()` returns a wrapper whose `sendAudio`, `setMediaTimestamp`, `triggerGreeting`, `submitToolResult`, and `acknowledgeMark` all call `requireBridge()`. Until `connect()` finishes `loadGoogleRealtimeVoiceProvider().then(provider.createBridge(...))`, `bridge` is undefined and `sendAudio()` throws `Google realtime voice bridge is not connected`. The concrete Google bridge still intentionally buffers pre-ready audio in `extensions/google/realtime-voice-provider.ts:457`, pushing frames into `pendingAudio` until the session is connected/configured. Voice-call starts `bridge.connect()` asynchronously at `extensions/voice-call/src/webhook/realtime-handler.ts:380`, while inbound media frames call `bridge.sendAudio(...)` immediately at `extensions/voice-call/src/webhook/realtime-handler.ts:182`.
- Impact: Google-backed realtime voice calls can lose the first audio frames, and log them as websocket parse failures, whenever the caller starts speaking before the lazy dynamic import has completed. This is a behavioral regression from the old eager `buildGoogleRealtimeVoiceProvider()` registration, where `sendAudio()` before connection was accepted and queued.
- Suggested fix: make the lazy wrapper preserve the concrete bridge contract: buffer audio until `loadBridge()` resolves, make timestamp/mark methods no-op until the real bridge exists, and flush buffered audio into the concrete bridge once it is created. Alternatively, eagerly create the concrete bridge in `createBridge()` and defer only the module import at registration time if startup isolation still holds.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: configure the voice-call plugin to use Google realtime voice, open a realtime websocket, send the `start` event, then send a `media` event before the wrapper’s dynamic import has resolved. The current wrapper throws from `extensions/google/index.ts:218`, while the previous concrete bridge would queue at `extensions/google/realtime-voice-provider.ts:457`.
- Is this the best way to solve the issue? yes: the safest fix is to keep the lazy import but make the wrapper match the old `RealtimeVoiceBridge` behavior for pre-connect audio and benign pre-ready calls.
