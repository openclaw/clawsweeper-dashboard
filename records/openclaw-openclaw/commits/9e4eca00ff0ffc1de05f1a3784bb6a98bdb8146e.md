# Commit 9e4eca00ff

## Summary

Found one regression outside the Slack approval normalization itself: the commit changes OpenClaw’s public secret-file helper to swallow all fs-safe validation errors.

## Findings

### Medium: `tryReadSecretFileSync` now hides invalid secret-file errors

- Kind: regression
- File: `src/infra/secret-file.ts`
- Line: 17
- Evidence: This commit replaces the direct re-export of `@openclaw/fs-safe/secret` with a wrapper that catches every thrown error and returns `undefined`. The pinned `@openclaw/fs-safe@0.2.7` contract only returns `undefined` for blank or missing paths; for rejected symlinks, non-files, oversized files, empty files, and other invalid secret-file states it throws `FsSafeError`. Current `src/infra/secret-file.test.ts:114` still asserts that `tryReadSecretFileSync(file, ..., { rejectSymlink: true })` throws for a symlink.
- Impact: Public SDK exports through `src/plugin-sdk/secret-file-runtime.ts`, `src/plugin-sdk/core.ts`, and `src/plugin-sdk/channel-core.ts` now silently turn invalid or rejected configured secret files into “missing”. That weakens the helper contract and will make the existing secret-file regression test fail once dependencies are installed.
- Suggested fix: Remove the wrapper and re-export `tryReadSecretFileSync` from fs-safe again. If a specific channel/status path needs “configured but unavailable” behavior, handle that at that call site without changing the public helper’s validation semantics.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. With dependencies installed, run `node scripts/run-vitest.mjs src/infra/secret-file.test.ts -t "throws from the try helper"`. I could not run Vitest in this checkout because `vitest/package.json` is missing, but I verified the pinned `@openclaw/fs-safe@0.2.7` package directly with a symlink fixture and it throws `FsSafeError`.
- Is this the best way to solve the issue? yes. The underlying fs-safe helper already implements the intended “undefined only for absent input/missing file” behavior, so the safest fix is to restore the re-export.
