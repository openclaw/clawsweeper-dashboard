# Commit 3e2a2c7b74

## Summary

The message-event path is fixed, but Slack slash commands still route with the raw config. A Slack route binding written with the newly documented target syntax can therefore route normal messages to the bound agent while `/openclaw` commands in the same Slack peer still fall back to `main`.

## Findings

### Medium: Slash command routing still misses target-style Slack route bindings

- Kind: bug
- File: `extensions/slack/src/monitor/slash.ts`
- Line: 581
- Evidence: The commit normalizes Slack route bindings only inside `extensions/slack/src/monitor/message-handler/prepare-routing.ts:135`, before message events call `resolveAgentRoute`. Slack slash commands have their own route calls at `extensions/slack/src/monitor/slash.ts:581` and `extensions/slack/src/monitor/slash.ts:652`, both passing `cfg` directly with raw `command.channel_id` / `command.user_id`. A focused source check confirms generic routing still misses `peer.id: "channel:C0AJUGWG5L6"` against runtime peer `C0AJUGWG5L6`, returning `agentId: "main"` and `matchedBy: "default"`.
- Impact: Operators using the now-documented Slack target forms can get inconsistent routing: inbound Slack messages hit the configured agent, but slash commands and native command menus in the same channel/DM can run against `main`, using the wrong session/workspace/agent defaults.
- Suggested fix: Reuse the same Slack route-binding normalization for slash routing. Prefer extracting the normalizer from `prepare-routing.ts` into a Slack-owned routing helper, then call it from both message and slash paths. Add a slash regression test for `channel:C...`, `user:U...`, and ideally `<@U...>`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Configure `bindings[].match.peer.id` as `channel:C0AJUGWG5L6`, then route a Slack slash command from `command.channel_id=C0AJUGWG5L6`; the slash path calls generic `resolveAgentRoute` without the new normalizer and falls back to `main`.
- Is this the best way to solve the issue? yes. The normalizer belongs in the Slack plugin boundary, but it needs to be shared by all Slack route-entry paths, not only message preparation.
