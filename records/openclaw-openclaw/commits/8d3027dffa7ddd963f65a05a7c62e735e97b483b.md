# Commit 8d3027d

## Summary

Found one concrete concurrency/data-loss risk in the new doctor-only sidecar migration. The focused regression suite passes, but the migration writes auth-profile files from a stale pre-prompt snapshot without using the auth store lock.

## Findings

### Medium: Doctor OAuth sidecar migration can overwrite concurrent auth-profile updates

- Kind: data_loss
- File: `src/commands/doctor-auth-oauth-sidecar.ts`
- Line: 581
- Evidence: `maybeRepairLegacyOAuthSidecarProfiles` loads each candidate store and keeps its `raw` object before prompting at lines 502-539, then later mutates that same object and writes it with `saveJsonFile(store.authPath, store.raw)` at line 581. This path does not acquire `AUTH_STORE_LOCK_OPTIONS` or reload the auth store under the lock, unlike normal auth-profile writes in `src/agents/auth-profiles/store.ts` and stale-shadow doctor repair.
- Impact: If a live Gateway, login, refresh, or another CLI updates `auth-profiles.json` while doctor is prompting or decrypting, doctor can overwrite that newer file with the earlier snapshot plus migrated tokens. That can drop newly added profiles, refreshed tokens, state fields, or auth metadata. The backup at line 577 copies the current file, but the active store is still reverted and the migrated sidecar may then be removed.
- Suggested fix: Do the prompt before locking, then for each store acquire the auth-profile file lock, reload the current raw file inside the lock, re-resolve sidecar-backed profiles from that current raw data, apply migration, and save. Only mark sidecars as migrated/removable after the locked save succeeds from the fresh file.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. Use a prompter that pauses at `confirmAutoFix`, modify the same `auth-profiles.json` from another writer, then resume; the migration writes the pre-prompt `store.raw` snapshot and drops the concurrent change.
- Is this the best way to solve the issue? Yes. Auth-profile persistence already has a lock contract, and this repair is now a writer to that same file.
