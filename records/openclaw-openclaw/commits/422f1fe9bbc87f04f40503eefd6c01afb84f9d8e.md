# Commit 422f1fe

## Summary

Found one concrete regression in the postpublish dependency verifier. The changed whitelist makes the verifier accept root-owned imports of externalized/plugin-local packages even when the root package does not declare them, and an existing focused test now fails on current `main`.

## Findings

### Medium: Root-owned runtime imports can bypass dependency manifest verification

- Kind: regression
- File: `scripts/openclaw-npm-postpublish-verify.ts`
- Line: 339
- Evidence: The new `OPTIONAL_OR_EXTERNALIZED_RUNTIME_IMPORTS` check runs before `declaredRuntimeDeps` and before the bundled-extension ownership check. That means any root `dist/*.js` importer of `@lancedb/lancedb`, `matrix-js-sdk`, `@matrix-org/matrix-sdk-crypto-nodejs`, etc. is accepted even when `package.json` has no dependency and the source is not inside an owning `//#region extensions/<id>/` chunk.
- Evidence: `OPENCLAW_VITEST_MAX_WORKERS=1 pnpm test test/openclaw-npm-postpublish-verify.test.ts test/release-check.test.ts` fails at `test/release-check.test.ts:331`; the existing “still requires root deps for root-owned installed chunks” case now receives `[]` instead of the expected missing dependency error for `dist/root-runtime.js` importing `@lancedb/lancedb`.
- Impact: A published package with root-owned code importing one of these externalized packages can pass postpublish verification without declaring the runtime dependency, then fail at runtime for users. It also leaves the focused tooling test suite red on `main`.
- Suggested fix: Remove the blanket skip, or scope it to proven plugin-owned/externalized chunks by checking the owning extension manifest/region before accepting the import. Root-owned imports should still require a root dependency or optionalDependency.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: run `OPENCLAW_VITEST_MAX_WORKERS=1 pnpm test test/openclaw-npm-postpublish-verify.test.ts test/release-check.test.ts`.
- Is this the best way to solve the issue? yes: preserve the existing owner-aware dependency exception and avoid a package-name-only bypass for root-owned code.
