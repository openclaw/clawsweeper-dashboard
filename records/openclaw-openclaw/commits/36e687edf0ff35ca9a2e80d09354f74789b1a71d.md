# Commit 36e687e

## Summary

Found one compatibility regression risk in the new channel plugin `tool-discovery` path. Focused tests covering the changed files passed on current `main`.

## Findings

### Medium: `tool-discovery` now runs `registerFull` despite the public full-only contract

- Kind: compatibility
- File: `src/plugin-sdk/core.ts`
- Line: 526
- Evidence: `resolvePluginTools` now loads plugins with `activate: false` and `toolDiscovery: true` in `src/plugins/tools.ts:136`. In that mode, both channel entry helpers call `registerFull` immediately and return before channel/runtime wiring: `src/plugin-sdk/core.ts:526` calls `registerFull?.(api)` before `api.registerChannel(...)` and `setRuntime?.(...)`; `src/plugin-sdk/channel-entry-contract.ts:497` does the same for bundled channel entries before `loadChannelPlugin`, `registerChannel`, and `setChannelRuntime`. The current public docs still say `registerFull` only runs for `api.registrationMode === "full"` and tell plugin authors to put sockets, clients, workers, services, and other runtime-only work behind full-only paths in `docs/plugins/sdk-entrypoints.md:130`.
- Impact: A channel plugin following the documented SDK contract can put runtime-only setup, side effects, or code that assumes `setRuntime` has already run inside `registerFull`. After this commit, ordinary per-run plugin tool resolution can execute that code in a non-activating snapshot and without runtime wiring, causing crashes or unintended work even when the plugin only expected full gateway activation to reach that callback.
- Suggested fix: Preserve `registerFull` as full-only and add a separate narrow callback for tool-discovery registrations, then move bundled channel tool registration to that callback. If the intended contract is to let `registerFull` run in `tool-discovery`, update the SDK/docs and at least wire runtime before invoking it, though that still leaves full-only side-effect compatibility risk.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes from the code path: define a channel plugin with `setRuntime` storing runtime and `registerFull` reading that store or starting full-only work, enable it, then call `resolvePluginTools`. The new path reaches `registerFull` with `api.registrationMode === "tool-discovery"` before `setRuntime`.
- Is this the best way to solve the issue? The safest direction is a separate tool-discovery registration seam. That keeps the performance goal without changing the documented meaning of `registerFull`.
