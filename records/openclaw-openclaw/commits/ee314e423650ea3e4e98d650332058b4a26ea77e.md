# Commit ee314e4236

## Summary

Found one migration ordering bug that can widen Telegram group reply triggering during legacy config repair.

## Findings

### Medium: Telegram-specific mention gate can be overwritten by the global routing migration

- Kind: security
- File: `src/commands/doctor/shared/legacy-config-migrations.channels.ts`
- Line: 420
- Evidence: `apply` runs `migrateRoutingGroupChat` before `migrateTelegramRequireMention`. With `{ routing: { groupChat: { requireMention: false } }, channels: { telegram: { requireMention: true, botToken: "x" } } }`, the first migration writes `channels.telegram.groups."*".requireMention = false`; the later Telegram-specific migration then deletes `channels.telegram.requireMention` because the wildcard group entry is already set. A focused smoke confirmed the migrated config resolves `resolveChannelGroupRequireMention({ channel: "telegram" })` to `false`.
- Impact: A user who had a global legacy default of “do not require mention” but overrode Telegram to require mentions loses the stricter Telegram override after `openclaw doctor --fix`. In allowed/open Telegram groups, unmentioned messages can then trigger the agent unexpectedly.
- Suggested fix: Migrate `channels.telegram.requireMention` before applying `routing.groupChat.requireMention`, or have the routing migration treat `channels.telegram.requireMention` as an existing channel-specific value and skip writing the global default for Telegram. Add a regression test where the two legacy values conflict in both directions.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: run a focused migration smoke with conflicting `routing.groupChat.requireMention` and `channels.telegram.requireMention`, then resolve the resulting Telegram group mention policy.
- Is this the best way to solve the issue? yes: preserving the more specific channel setting matches normal config precedence and avoids widening group triggers.
