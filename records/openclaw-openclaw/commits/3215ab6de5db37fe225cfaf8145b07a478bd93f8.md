# Commit 3215ab6

## Summary

Found two heartbeat regressions in the changed behavior: default heartbeat enablement is now reported for all agents but the runner still schedules only the default agent, and the new task-block stripper still drops some non-task `HEARTBEAT.md` directives.

## Findings

### Medium: Default heartbeat config still does not schedule non-default agents

- Kind: regression
- File: `src/infra/heartbeat-runner.ts`
- Line: 254
- Evidence: `isHeartbeatEnabledForAgent()` now returns `true` for any agent when `agents.defaults.heartbeat` exists (`src/infra/heartbeat-summary.ts:41`), matching docs and changelog (`docs/gateway/heartbeat.md:117`, `CHANGELOG.md:28`). But `resolveHeartbeatAgents()` still returns only `resolveDefaultAgentId(cfg)` when no per-agent heartbeat block exists (`src/infra/heartbeat-runner.ts:254`), and `startHeartbeatRunner()` builds its runnable state from that list (`src/infra/heartbeat-runner.ts:1504`). A focused `tsx` smoke with `list: [{id:"main"}, {id:"ops"}]` and `agents.defaults.heartbeat.every` dispatched only `[{"agentId":"main"}]`.
- Impact: non-default agents appear heartbeat-enabled in status/health summaries, but scheduled and untargeted wake heartbeats never run for them. Targeted wake requests for those agents also miss because `state.agents` does not contain them.
- Suggested fix: when no `agents.list[].heartbeat` entries exist but `agents.defaults.heartbeat` is configured, return every configured agent from `resolveHeartbeatAgents()` with merged defaults. Keep the current default-agent fallback only when there is no agent list or no heartbeat config.
- Confidence: high

### Low: Top-level bullet directives after `tasks:` are still stripped

- Kind: regression
- File: `src/infra/heartbeat-runner.ts`
- Line: 718
- Evidence: `stripHeartbeatTasksBlock()` treats any trimmed line starting with `-` as part of the task block even when it is top-level and unindented (`src/infra/heartbeat-runner.ts:717`). A focused `runHeartbeatOnce()` smoke with `HEARTBEAT.md` containing a valid task block followed by `- Keep this top-level directive` produced a task prompt where `containsDirective` was `false`.
- Impact: common markdown checklist/directive bullets placed after a `tasks:` block are omitted from the heartbeat prompt, so task-mode heartbeats can ignore user instructions that this commit intended to preserve.
- Suggested fix: make the stripper indentation-sensitive: only drop list items and task fields that are actually inside the `tasks:` block, and terminate on any top-level non-task line, including top-level bullets. Add a test for a top-level bullet after the task block.
- Confidence: high
