# Commit fc7a531

## Summary

Found one concrete regression risk in the reload fallback: a successful `kickstart -k` can make the helper report success even when the required plist reload did not happen.

## Findings

### Medium: Reload fallback can hide a failed plist reload and keep stale launchd state

- Kind: regression
- File: `src/daemon/launchd-restart-handoff.ts`
- Line: 181
- Evidence: Reload mode explicitly needs `bootout` + `bootstrap` because the code says `kickstart` alone keeps already-loaded stdout/stderr/stdin paths at `src/daemon/launchd-restart-handoff.ts:154`. The new fallback runs `launchctl kickstart -k "$service_target"` after `bootstrap` fails, then overwrites `status` with the kickstart exit code at `src/daemon/launchd-restart-handoff.ts:180`. If `bootout` is ignored/fails, or the new 3-second poll times out while launchd still has the old service definition loaded, `bootstrap` can fail because the service is still loaded and `kickstart -k` can succeed against that stale loaded definition. The script then logs `restart done`, even though the new plist was not bootstrapped.
- Impact: The gateway stays alive, but reload semantics are silently lost. That can leave old launchd-managed stdio paths, wrapper/env state, or other plist changes active after a restart that the caller treats as completed.
- Suggested fix: Do not turn a reload bootstrap failure into full success solely because `kickstart` succeeds. Either retry `bootout`/poll/`bootstrap`, or use `kickstart` only as a keep-alive fallback while preserving/logging the bootstrap failure as “reload not applied”.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Unclear. A live macOS reproduction would need the reload handoff path where `launchctl bootout` leaves the service loaded past the wait loop, or otherwise fails in a way this script ignores, then `bootstrap` fails and `kickstart -k` succeeds.
- Is this the best way to solve the issue? The poll is a good fix direction for the original race. The risky part is only status handling after the fallback: availability recovery and “plist reload applied” should not be collapsed into the same success state.
