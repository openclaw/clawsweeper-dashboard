# Commit 0b6db06d

## Summary

Current `main` already has a follow-up fix (`bc924889be`) for the package-image regression in the reviewed commit, but one Open WebUI live-lane planner regression remains: retry override settings are ignored for the default selected Open WebUI lane.

## Findings

### Low: Open WebUI selected lane ignores `OPENCLAW_DOCKER_ALL_LIVE_RETRIES`

- Kind: reliability
- File: `scripts/lib/docker-e2e-plan.mjs`
- Line: 387
- Evidence: `resolveDockerE2ePlan` builds selectable lanes by deduping `allReleasePathLanes({ includeOpenWebUI })` before `retriedMainLanes`, so with the default `includeOpenWebUI=true`, `OPENCLAW_DOCKER_ALL_LANES=openwebui` selects the release-path Open WebUI lane before the copy that passed through `applyLiveRetries`. A dry run with `OPENCLAW_DOCKER_ALL_LANES=openwebui OPENCLAW_DOCKER_ALL_LIVE_RETRIES=0 ... node scripts/test-docker-all.mjs` prints `Live retries: 0` but the lane summary still shows `openwebui(... retries=1)`. The same command with `OPENCLAW_DOCKER_ALL_LIVE_RETRIES=3` also shows `retries=1`; setting `OPENCLAW_DOCKER_ALL_INCLUDE_OPENWEBUI=0` lets the main-lane copy be selected and shows `retries=3`.
- Impact: Operators cannot disable or increase retries for targeted Open WebUI reruns under the default planner inputs. This makes the scheduler output contradict its configured live retry count and can either add an unwanted rerun or fail to add requested retries for a live, external-service-backed smoke.
- Suggested fix: Apply `applyLiveRetries` after final lane selection for selected and release-path lanes, or construct/dedupe release-path Open WebUI lanes with the requested retry count before they enter `selectableLanes`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: run `OPENCLAW_DOCKER_ALL_LANES=openwebui OPENCLAW_DOCKER_ALL_LIVE_RETRIES=0 OPENCLAW_DOCKER_ALL_DRY_RUN=1 OPENCLAW_DOCKER_ALL_PREFLIGHT=0 OPENCLAW_DOCKER_ALL_BUILD=0 node scripts/test-docker-all.mjs` and compare the logged `Live retries: 0` with `openwebui(... retries=1)`.
- Is this the best way to solve the issue? yes: retry configuration is planner metadata, so normalizing retries after the planner chooses lanes keeps selected, release-path, main, and tail live lanes consistent without changing the Open WebUI smoke itself.
