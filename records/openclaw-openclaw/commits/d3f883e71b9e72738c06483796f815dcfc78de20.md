# Commit d3f883e

## Summary

Found one reliability issue in the new manual report-repo initialization path. The default one-lane manual run for this commit succeeded, but the multi-lane publish path can still fail when more than one matrix lane publishes around the first creation of `main`.

## Findings

### Medium: Multi-lane report publishing can lose the first empty-repo initialization race

- Kind: reliability
- File: `.github/workflows/openclaw-performance.yml`
- Line: 372
- Evidence: The workflow has three matrix lanes, and scheduled runs allow at least `mock-provider` and `mock-deep-profile` to publish. Each lane independently initializes `.artifacts/clawgrit-reports`, checks whether `origin/main` exists, commits its lane output, then runs a single `git -C "$reports_root" push origin HEAD:main` without fetching/rebasing/retrying. A focused local reproduction with two worktrees following the new empty-remote path showed the first push creating `main` and the second failing with `! [rejected] HEAD -> main (fetch first)`.
- Impact: The first scheduled or multi-lane manual publish against an empty `openclaw/clawgrit-reports` repo can still fail one or more lanes after Kova has produced reports. That leaves missing lane reports and `latest-<lane>.json` pointers, and marks the workflow failed even though the report repo initialization was the change’s purpose.
- Suggested fix: Move publishing into a single aggregate job after artifacts are uploaded, or add a bounded push retry that fetches `origin/main` when the push is rejected, rebases/merges the local lane commit, and retries. The retry should also handle the transition from no remote branch to a newly-created `main`.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes for the Git failure mode: two independent local checkouts initialized from an empty bare remote, each committing `openclaw-performance/...`, reproduce the second push rejection. The exact scheduled timing in GitHub Actions was not observed.
- Is this the best way to solve the issue? unclear. A single aggregate publisher is the most robust because it avoids branch contention; a fetch/rebase/retry loop is a smaller workflow-only fix.
