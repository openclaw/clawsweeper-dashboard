# Commit 9ef37d1

## Summary

Found one low-severity consistency bug in the Google model-id normalization surface. The changed Google plugin helper now canonicalizes `gemini-3-pro-preview`, but the parallel Plugin SDK helper still returns the stale id.

## Findings

### Low: Public SDK Google model normalizer was left behind

- Kind: compatibility
- File: `src/plugin-sdk/provider-model-id-normalize.ts`
- Line: 4
- Evidence: `extensions/google/model-id.ts:4` now maps both `gemini-3-pro` and `gemini-3-pro-preview` to `gemini-3.1-pro-preview`, but `src/plugin-sdk/provider-model-id-normalize.ts:4` only maps `gemini-3-pro`. The public `src/plugin-sdk/google-model-id.ts` and `src/plugin-sdk/image-generation-core.ts` exports still flow through this older helper. A focused live check confirms the split: `{"sdk":"gemini-3-pro-preview","plugin":"gemini-3.1-pro-preview"}`.
- Impact: callers using the public SDK normalizer can keep routing the deprecated `gemini-3-pro-preview` id while the bundled Google provider routes the same input to `gemini-3.1-pro-preview`. That can make third-party/plugin-facing behavior drift from bundled Google behavior and leave stale model ids in SDK consumers.
- Suggested fix: update `normalizeGooglePreviewModelId` to handle `gemini-3-pro-preview`, add the matching assertion in `src/plugin-sdk/provider-model-id-normalize.test.ts`, or centralize the Google plugin and SDK normalizers so this alias table cannot drift.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes: compare `normalizeGooglePreviewModelId("gemini-3-pro-preview")` from `src/plugin-sdk/provider-model-id-normalize.ts` with `normalizeGoogleModelId("gemini-3-pro-preview")` from `extensions/google/model-id.ts`.
- Is this the best way to solve the issue? Yes: the safest fix is to keep the public SDK helper aligned with the bundled provider normalization contract.
