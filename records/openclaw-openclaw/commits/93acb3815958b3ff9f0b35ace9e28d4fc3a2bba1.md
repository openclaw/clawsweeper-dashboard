# Commit 93acb38159

## Summary

Found one concrete Tool Search dispatch bug that still matters on current `main`: name-based lookup can silently choose the wrong catalog entry when two cataloged tools share the same name.

## Findings

### Medium: Ambiguous name lookup can dispatch to the wrong catalog tool

- Kind: bug
- File: `src/agents/tool-search.ts`
- Line: 760
- Evidence: `registerToolSearchCatalog` stores entries by full id and raw name, but `findEntry` returns the first catalog entry whose `id` or `name` matches the requested string. Current PI wiring can append client tools into the same Tool Search catalog, and the current client conflict check intentionally allows client tools to coexist with unrelated plugin tool names in `src/agents/pi-embedded-runner/run/attempt.ts:1777`. I reproduced this with a focused `node --import tsx` probe: a plugin tool and a client tool both named `same_name` were cataloged; `tool_call` with `{ id: "same_name" }` executed the client tool, while `{ id: "openclaw:fake-plugin:same_name" }` executed the plugin tool.
- Impact: The structured fallback and code bridge both document/accept “id or name” lookup. If a model uses a bare tool name after search/describe, Tool Search can invoke a different same-named implementation than intended, including a client-delegated tool instead of an OpenClaw/plugin tool or vice versa. That is a wrong-tool side effect, not just a display issue.
- Suggested fix: Keep full ids callable, but make bare-name lookup fail closed when more than one catalog entry has that name. Alternatively, remove name aliases from `tool_call`/`openclaw.tools.call` and require the id returned by search/describe.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The focused `tsx` probe cataloged same-named plugin and client tools, then showed `{ id: "same_name" }` called the client entry while the plugin was reachable only by full id.
- Is this the best way to solve the issue? yes. Rejecting ambiguous bare names preserves the compact search API while preventing silent misrouting; requiring ids everywhere is stricter but also safe.
