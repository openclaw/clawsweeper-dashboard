# Commit 9a816f4

## Summary

Found one actionable test-gap regression: the updated assertion now blesses the Docker prod-store seed pipeline, but the pipeline can fail upstream and still exit successfully under Docker shell semantics.

## Findings

### Medium: Docker prod-store seed failures are hidden by the tracked pipeline

- Kind: test_gap
- File: `src/dockerfile.test.ts`
- Line: 96
- Evidence: The test now only checks that `Dockerfile:127` contains `pnpm list --prod --depth Infinity --json | node scripts/list-prod-store-packages.mjs | xargs -r pnpm store add`. Running that exact pipeline without `pipefail`, matching Docker’s default shell behavior, printed a `TypeError: roots is not iterable` from `scripts/list-prod-store-packages.mjs` and still ended with `pipeline_exit=0` because `xargs -r` received no input and succeeded. With `pipefail`, the same pipeline failed because `pnpm list --prod --depth Infinity --json` emitted an `{ "error": { "code": "EMFILE", ... } }` object instead of a dependency list.
- Impact: Docker builds can silently skip the prod-store seeding step and continue to `pnpm prune --prod --config.offline=true`, making the runtime image build depend on incidental cache state or fail later with a less direct error. The revised test passes while missing that failure mode.
- Suggested fix: Avoid the pipe for failure-sensitive steps, for example write `pnpm list` output to a temp file, run `node scripts/list-prod-store-packages.mjs` as a separate `&&` command, then feed the resulting package list to `xargs`. Update the test to exercise failure propagation or assert the non-pipeline command shape.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Run `pnpm list --prod --depth Infinity --json | node scripts/list-prod-store-packages.mjs | xargs -r pnpm store add; printf 'pipeline_exit=%s\n' "$?"` from the repo root after installing dependencies. It reports the helper `TypeError` but exits `0`.
- Is this the best way to solve the issue? yes. The build step needs command-level failure propagation; string-matching the pipeline cannot prove that.
