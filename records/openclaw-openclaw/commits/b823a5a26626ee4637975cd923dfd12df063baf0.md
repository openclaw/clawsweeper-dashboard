# Commit b823a5a2

## Summary

Found one lifecycle regression: the new approval/action-required Live Activity can be immediately dismissed by the operator session disconnect callback, which undercuts the commit’s main attention-state behavior.

## Findings

### Medium: Operator disconnect can immediately dismiss approval-needed Live Activities

- Kind: regression
- File: `apps/ios/Sources/Model/NodeAppModel.swift`
- Line: 2129
- Evidence: `applyGatewayConnectionProblem` now starts an attention Live Activity for `needsPairingApproval` or `pauseReconnect` at `apps/ios/Sources/Model/NodeAppModel.swift:1912`. In the pairing-required path, the node loop then cancels and disconnects the operator gateway at `apps/ios/Sources/Model/NodeAppModel.swift:2338` and `apps/ios/Sources/Model/NodeAppModel.swift:2344`. The operator `onDisconnected` callback now unconditionally calls `LiveActivityManager.shared.endActivity(reason: "operator_disconnected")` at `apps/ios/Sources/Model/NodeAppModel.swift:2129`. The underlying `GatewayChannelActor` invokes disconnect handlers on connect/receive failures at `apps/shared/OpenClawKit/Sources/OpenClawKit/GatewayChannel.swift:353` and `apps/shared/OpenClawKit/Sources/OpenClawKit/GatewayChannel.swift:791`.
- Impact: approval-required reconnects can briefly create the intended “Approval needed” Live Activity and then dismiss it as part of the same failure-handling flow, so users may never see the attention state while the reconnect loop is paused.
- Suggested fix: do not end Live Activities from the secondary operator-session disconnect callback. Let the node gateway lifecycle and explicit disconnect/background-idle paths own Live Activity dismissal, or at minimum guard the operator callback so it does not end when `lastGatewayProblem?.needsPairingApproval == true` or `lastGatewayProblem?.pauseReconnect == true`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Drive a gateway connection into a pairing-required or pause-reconnect error while an operator session exists or is attempting to connect; the node loop calls `showAttention`, then the operator disconnect path can call `endActivity`.
- Is this the best way to solve the issue? yes. The operator session is a secondary chat/talk/config session, while the Live Activity represents the primary gateway connection/attention lifecycle. Removing or guarding the operator-session dismissal keeps ownership aligned and avoids racing the new attention state.
