# Commit cce12697a1

## Summary

Found one authorization regression in the new doctor migration: multi-account channels can inherit a newly materialized parent `groupAllowFrom` even when an account explicitly had `allowFrom: []` to block group fallback.

## Findings

### Medium: Doctor can broaden group access for accounts with explicit empty `allowFrom`

- Kind: security
- File: `src/commands/doctor/shared/allowfrom-fallback-migration.ts`
- Line: 61
- Evidence: The migration first copies channel `allowFrom` to channel `groupAllowFrom`, then skips an account when `ownAllowFrom.length === 0` and the parent has `groupAllowFrom`. For channels using `resolveMergedAccountConfig`, that skipped account still inherits the new parent `groupAllowFrom`. A focused Signal repro changed `channels.signal.accounts.work.allowFrom: []` from blocked before repair to allowed after repair because the account inherited `channels.signal.groupAllowFrom`.
- Impact: Running `openclaw doctor --fix` can widen group sender authorization for a multi-account channel. With a parent `allowFrom: ["*"]`, an account that previously blocked group fallback via an explicit empty account allowlist can become open to all group senders after repair.
- Suggested fix: Distinguish absent own DM allowlist from an explicit empty own DM allowlist. When materializing a parent `groupAllowFrom`, preserve explicit account overrides, for example by writing account-level `groupAllowFrom: []` for accounts with explicit empty own `allowFrom`, or by otherwise preventing inherited parent `groupAllowFrom` from changing those accounts’ effective group access.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. I used `maybeRepairGroupAllowFromFallback`, `resolveSignalAccount`, and `resolveSignalAccessState` with `channels.signal.allowFrom=["+15555550123"]` and `channels.signal.accounts.work.allowFrom=[]`; before repair Signal blocked the group sender with `group_policy_empty_allowlist`, after repair it allowed the same sender with `group_policy_allowed`.
- Is this the best way to solve the issue? yes directionally: the migration should preserve the pre-repair effective access for every account, including explicit empty account-level allowlists that override parent fallback.
