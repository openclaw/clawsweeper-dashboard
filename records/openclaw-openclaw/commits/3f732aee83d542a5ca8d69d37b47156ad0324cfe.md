# Commit 3f732aee83

## Summary

Found one remaining user-visible status bug in the changed `runSessionKey` path: explicit `sessionKey: "current"` now uses the live run session, but an omitted `sessionKey` still uses the stale sandbox/policy key.

## Findings

### Medium: `session_status` with omitted `sessionKey` still resolves to the stale sandbox key

- Kind: bug
- File: `src/agents/tools/session-status-tool.ts`
- Line: 353
- Evidence: `createSessionStatusTool` starts with `requestedKeyRaw = requestedKeyParam ?? opts?.agentSessionKey`, so when the model calls `session_status({})`, the newly threaded `opts.runSessionKey` is ignored. In the fixed Telegram direct shape, `runEmbeddedAttempt` passes `sessionKey: sandboxSessionKey` and separately passes `runSessionKey: params.sessionKey` when they differ (`src/agents/pi-embedded-runner/run/attempt.ts:887`). The explicit `"current"` path rewrites to `runSessionKey`, but the omitted-argument path does not.
- Impact: The tool schema makes `sessionKey` optional, and existing tests treat no-arg `session_status` as “current session” (`src/agents/openclaw-tools.session-status.test.ts:395`). In the #76708 shape where both `agent:main:telegram:default:direct:<id>` and `agent:main:main` exist, `session_status({})` will still report the stale Telegram direct row/model while `session_status({ sessionKey: "current" })` reports the live run. That can preserve the same wrong-model answer this commit intended to fix whenever the model omits the optional field.
- Suggested fix: Treat omitted `sessionKey` as semantic current when `opts.runSessionKey` is present, while preserving the existing explicit-key visibility behavior. A focused regression test should mirror the new `"current"` test but call `tool.execute(..., {})` and expect `agent:main:main`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. Construct `createSessionStatusTool({ agentSessionKey: "agent:main:telegram:default:direct:1234", runSessionKey: "agent:main:main" })`, seed both store rows, and call `execute("call", {})`; the code resolves the stale `agentSessionKey` before any `runSessionKey` rewrite.
- Is this the best way to solve the issue? Yes. The narrow fix is to apply `runSessionKey` to semantic current resolution for omitted `sessionKey`, without allowing explicit cross-session keys to bypass visibility.
