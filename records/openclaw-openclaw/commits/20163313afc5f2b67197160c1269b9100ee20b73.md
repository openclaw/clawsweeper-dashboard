# Commit 2016331

## Summary

Found one concrete regression: the QQ Bot chunked-upload unit tests now make real DNS/network attempts after the production code switched COS PUTs from `fetch` to `fetchWithSsrFGuard`.

## Findings

### Medium: QQ Bot chunked-upload tests no longer mock COS PUTs

- Kind: regression
- File: `extensions/qqbot/src/engine/api/media-chunked.ts`
- Line: 568
- Evidence: `putToPresignedUrl` now calls `fetchWithSsrFGuard({ url: presignedUrl, ... })`. The existing test stub only replaces `globalThis.fetch` (`extensions/qqbot/src/engine/api/media-chunked.test.ts:84`), but the guard resolves/pins DNS before dispatch. Running `pnpm test src/infra/tmp-openclaw-dir.test.ts src/infra/archive.test.ts src/infra/fs-safe-defaults.test.ts extensions/qqbot/src/engine/api/media-chunked.test.ts` passes the three infra files, then fails 3 QQ Bot cases with `Error: getaddrinfo ENOTFOUND cos.example.com`.
- Impact: the extension-messaging test lane is red on current `main`, and the unit test attempts a real lookup for the fixture hostname instead of staying hermetic.
- Suggested fix: mock `openclaw/plugin-sdk/ssrf-runtime` in `media-chunked.test.ts`, or add a narrow injectable fetch-guard seam for `ChunkedMediaApi`/`putToPresignedUrl` and assert the guarded request shape there.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: run `pnpm test extensions/qqbot/src/engine/api/media-chunked.test.ts` or the combined focused command above after dependencies are installed.
- Is this the best way to solve the issue? yes: the production switch to the SSRF guard is reasonable, but tests must mock the new guard boundary rather than only `globalThis.fetch`.
