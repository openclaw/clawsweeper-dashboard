# Commit 2bdbec8

## Summary

Found one low-severity status/fallback observability regression. The alias equivalence fix is too broad: it also suppresses real fallback state when a CLI backend is configured as a fallback route for the same canonical model.

## Findings

### Low: Same-model CLI fallback is hidden as if no fallback happened

- Kind: bug
- File: `src/auto-reply/fallback-state.ts`
- Line: 156
- Evidence: `areRuntimeModelRefsEquivalent()` canonicalizes `claude-cli/<model>` to `anthropic/<model>` in `src/agents/model-runtime-aliases.ts:100`, and `resolveFallbackTransition()` now uses that equivalence to set `fallbackActive` false. A focused read-only check with selected `anthropic/claude-opus-4-7`, active `claude-cli/claude-opus-4-7`, and a `429 rate limit` attempt returned `fallbackActive: false`, `stateChanged: false`, and `buildFallbackNotice(...) === null`. The runner only persists fallback state and emits fallback lifecycle/notices when `fallbackTransitioned` is true in `src/auto-reply/reply/agent-runner.ts:1388` and `src/auto-reply/reply/agent-runner.ts:1631`.
- Impact: OpenClaw documents CLI backends as valid fallback targets that run only after primary models fail (`docs/gateway/cli-backends.md:65`). If a user falls back from Anthropic API to Claude CLI for the same model id, `/status` and verbose notices now report no fallback, hiding the actual runtime/auth/billing route and the primary failure reason.
- Suggested fix: Keep alias equivalence for the configured-runtime case, but require evidence that the active CLI route is the selected model’s configured runtime before suppressing fallback state. If the CLI provider came from the fallback chain, preserve the fallback notice even when the canonical provider/model id matches.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Direct import check: `resolveFallbackTransition({ selectedProvider: "anthropic", selectedModel: "claude-opus-4-7", activeProvider: "claude-cli", activeModel: "claude-opus-4-7", attempts: [{ provider: "anthropic", model: "claude-opus-4-7", reason: "rate_limit", error: "429 rate limit" }] })` returns no active fallback.
- Is this the best way to solve the issue? yes. The fix should distinguish “selected canonical model executed through its configured CLI runtime” from “selected model failed and a CLI backend fallback handled the turn”; only the first case should be display-equivalent.
