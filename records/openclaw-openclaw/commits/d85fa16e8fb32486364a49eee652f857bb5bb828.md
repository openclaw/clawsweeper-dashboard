# Commit d85fa16

## Summary

Found one actionable regression risk in the new transcript echo suppression. The audio suppression pieces and nearby Google Meet tests otherwise look consistent on current `main`.

## Findings

### Medium: Echo matcher can drop real follow-up questions that quote the agent

- Kind: regression
- File: `extensions/google-meet/src/realtime.ts`
- Line: 208
- Evidence: `isGoogleMeetLikelyAssistantEchoTranscript` treats a user transcript as echo when `userNormalized.includes(assistantNormalized)`. In agent strategy, `onTranscript` then returns before `enqueueAgentConsultForUserTranscript`. A helper-equivalent check returns `true` for recent assistant text `The launch is blocked by API quotas.` and user text `The launch is blocked by API quotas. What should we do next?`.
- Impact: In the default `realtime.strategy: "agent"` path, a participant who repeats or quotes the agent’s previous answer before asking a follow-up can be silently ignored, so the meeting receives no agent consult or spoken answer.
- Suggested fix: Only suppress containment matches when the user transcript is mostly just the assistant text, or require a tight extra-token budget. Do not suppress when the user transcript contains meaningful new trailing/leading question content.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes: call `isGoogleMeetLikelyAssistantEchoTranscript` with a recent assistant transcript and a user transcript that contains that assistant sentence plus a follow-up question; the runtime path then skips `enqueueAgentConsultForUserTranscript`.
- Is this the best way to solve the issue? unclear: the echo suppressor is needed, but the containment branch should distinguish pure echo from quote-plus-follow-up.
