# Commit 23eb44b

## Summary

Found one actionable macOS/privacy/reliability issue in the new no-provider `models auth list` path.

## Findings

### Medium: `models auth list` can unexpectedly read/prompt for external CLI Keychain credentials

- Kind: privacy
- File: `src/commands/models/auth-list.ts`
- Line: 100
- Evidence: `modelsAuthListCommand` calls `ensureAuthProfileStore(agentDir, undefined)` when `--provider` is omitted. That falls through to external CLI overlay with no `allowKeychainPrompt: false`. The external CLI scope then includes an existing stored external profile, and Codex/Claude credential readers invoke macOS Keychain unless `allowKeychainPrompt === false` (`src/agents/cli-credentials.ts:255`, `src/agents/cli-credentials.ts:433`). The provider-filtered path is safe because `externalCliDiscoveryForProviderAuth` defaults `allowKeychainPrompt` to false.
- Impact: A read-only listing command documented as showing saved profiles can trigger a macOS Keychain prompt or read external CLI secret material into memory just because an existing `openai-codex:default` or `claude-cli` profile is present. This can block scripts and violates the repo’s documented read-only/status credential policy.
- Suggested fix: Always pass an explicit non-prompting external discovery policy for this command, including the no-provider case. If the command should list only saved persisted profiles, use the no-external/read-only store helper. If it should include external runtime profiles, use an existing/config-scoped discovery mode with `allowKeychainPrompt: false`, and add a test for `modelsAuthListCommand({ json: true })` verifying the no-provider path does not allow keychain prompting.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes by code path; live reproduction needs macOS with a persisted external CLI OAuth profile. Run `openclaw models auth list` without `--provider`; the no-provider branch passes no store options, so external CLI credential reads are allowed to use Keychain.
- Is this the best way to solve the issue? Yes: the least invasive fix is to make the list path’s external discovery policy explicit and non-prompting, matching nearby auth/status behavior.
