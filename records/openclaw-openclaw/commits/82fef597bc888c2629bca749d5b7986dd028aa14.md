# Commit 82fef597

## Summary

Found one regression in plugin config id normalization. The commit changed the alias scan from bundled metadata to all discovered plugins, which lets an external/global plugin provider id override a bundled plugin id in `plugins.allow`, `plugins.deny`, and `plugins.entries`.

## Findings

### Medium: External provider aliases can hijack bundled plugin allow/deny entries

- Kind: regression
- File: `src/plugins/config-state.ts`
- Line: 51
- Evidence: `getBundledPluginAliasLookup()` now loops over `discoverOpenClawPlugins({}).candidates` and writes every manifest id, provider id, and legacy id into one `Map`; provider aliases are assigned with `lookup.set(normalizedProviderId, manifest.id)` at line 67. Since discovery includes global plugins after bundled plugins, a global plugin with `providers: ["anthropic"]` overwrites the bundled `anthropic -> anthropic` alias. `normalizePluginsConfig()` then applies that alias to allow/deny lists and entry keys via `normalizeList()` / `normalizePluginEntries()`, and activation policy uses those normalized ids for deny and allow decisions.
- Impact: A config meant to deny or allow a bundled plugin by id can target the wrong plugin after an external plugin is installed. I reproduced this with a temp global plugin `rogue-anthropic` declaring `providers: ["anthropic"]`: `normalizePluginId("anthropic")` returned `"rogue-anthropic"`, and `normalizePluginsConfig({ deny: ["anthropic"] }).deny` became `["rogue-anthropic"]`. That means bundled Anthropic would no longer be denied, while `allow: ["anthropic"]` would explicitly allow the external plugin and exclude the bundled one when an allowlist is active.
- Suggested fix: Keep this alias table bundled-only, or enforce deterministic owner precedence so non-bundled provider/legacy aliases cannot overwrite bundled plugin ids. At minimum, do not overwrite an existing alias for a concrete bundled plugin id/provider unless the later record is the same plugin or an explicit config override path with clear precedence.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A `node --import tsx` temp-fixture smoke created a global plugin under an isolated `OPENCLAW_CONFIG_PATH` and showed `normalizePluginId("anthropic")` resolving to `rogue-anthropic`.
- Is this the best way to solve the issue? yes for the narrow failure mode: restore bundled-only alias ownership or preserve bundled precedence in this normalization helper. Broader external provider alias support should use registry owner maps with explicit conflict handling, not this unconditional map overwrite.
