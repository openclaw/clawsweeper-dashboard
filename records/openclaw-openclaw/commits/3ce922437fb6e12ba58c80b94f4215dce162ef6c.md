# Commit 3ce9224

## Summary

Found one edge-case regression in the new selectable model-map runtime discovery. A default-agent model override can mask a default selectable OpenAI model that still resolves to Codex for other agents, so startup/doctor can still miss Codex.

## Findings

### Medium: Default selectable model runtimes are resolved only in the default-agent scope

- Kind: bug
- File: `src/agents/harness-runtimes.ts`
- Line: 138
- Evidence: `pushModelMapRefs(config.agents?.defaults?.models)` resolves default model-map keys without an `agentId`, so `resolveModelRuntimePolicy` evaluates them in the configured default agent’s scope. Runtime policy resolution checks `agentEntry?.models` before `agents.defaults.models`, so a default agent override such as `"openai/gpt-5.5": { agentRuntime: { id: "pi" } }` suppresses the default map’s implicit Codex runtime. The collector then only scans each agent’s own `agent.models`, not the inherited `agents.defaults.models`, so a second agent that inherits the default selectable OpenAI model is missed.
- Impact: In a multi-agent config, `worker` can select `agents.defaults.models["openai/gpt-5.5"]` and runtime selection resolves it to `{ runtime: "codex" }`, but `collectConfiguredAgentHarnessRuntimes` returns `[]`. That means plugin auto-enable, startup planning, and doctor install repair can still fail to load/repair Codex for the exact class of selectable model config this commit is meant to fix.
- Suggested fix: When scanning `agents.defaults.models`, resolve those refs for every configured agent scope, or otherwise make the default-map scan union the runtimes required by all agents that inherit it. Keep the single unscoped scan for configs without an agent list.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused probe with:
  `agents.defaults.models["openai/gpt-5.5"] = {}`, `agents.list[0] = { id: "main", default: true, models: { "openai/gpt-5.5": { agentRuntime: { id: "pi" } } } }`, and `agents.list[1] = { id: "worker" }` printed `collector []`, while `resolveAgentHarnessPolicy(... agentId: "worker")` printed `{ runtime: 'codex', runtimeSource: 'implicit' }`.
- Is this the best way to solve the issue? yes. The collector is the shared source for auto-enable/startup/doctor, so it should mirror the same per-agent policy lookup used at runtime instead of evaluating default selectable entries in only one agent scope.
