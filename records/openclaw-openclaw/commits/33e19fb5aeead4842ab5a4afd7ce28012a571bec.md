# Commit 33e19fb5ae

## Summary

Found two actionable issues. The main one is a committed test regression: the release scan test now expects reviewed `dist/*` findings even though the current npm dry-run packages do not include `dist` files in a fresh checkout.

## Findings

### Medium: Release scan test fails when plugin packages have no built `dist`

- Kind: regression
- File: `src/plugins/npm-install-security-scan.release.test.ts`
- Line: 24
- Evidence: The commit adds `dist/*` entries to `REVIEWED_PUBLISHABLE_CRITICAL_FINDINGS`, but the test also asserts every reviewed key was actually observed at line 171. In a fresh current-main checkout, `npm pack --dry-run --json --ignore-scripts` for `extensions/acpx`, `extensions/codex`, `extensions/google-meet`, and `extensions/voice-call` reports no `dist/` files, so the reviewed set only contains the existing `src/*` findings.
- Impact: `pnpm test src/security/skill-scanner.test.ts src/plugins/npm-install-security-scan.release.test.ts` fails. This breaks focused changed-test/local validation for this touched release test unless a build has populated those package `dist` outputs first.
- Suggested fix: Do not require generated `dist` reviewed findings unless those files are present in the packed file list, or make the test build/stage the expected dist artifacts before asserting them. Keep the “unexpected criticals fail” assertion strict.
- Confidence: high

### Low: Template literal interpolation comments are still used as source-rule context

- Kind: bug
- File: `src/security/skill-scanner.ts`
- Line: 261
- Evidence: `stripCommentsForHeuristics` treats backticks like plain quoted strings, so it never strips comments inside `${...}` interpolation code. A focused smoke with valid JS containing `const env = process.env;` and a template interpolation whose body has only `// fetch("https://example.invalid")` still returns a critical `env-harvesting` finding.
- Impact: The scanner can still block a clean plugin when the only network-send context is a comment inside a template interpolation. This is narrower than the fixed inline/block comment cases, but it contradicts the intended “ignore scanner comment context” behavior for valid JavaScript.
- Suggested fix: Track template-literal interpolation state and strip comments inside `${...}`, or use a TypeScript/JavaScript lexer instead of a hand-rolled string/comment scanner. Add a regression test for `// fetch()` inside template interpolation.
- Confidence: medium
