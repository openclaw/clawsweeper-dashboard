# Commit 39f8109

## Summary

Found multiple actionable regressions in the TypeScript smoke-script port. The highest-risk issue is secret exposure on failure paths: provider API keys are now placed in argv/script text, and the shared command helper prints full argv when a command fails.

## Findings

### High: Failed guest commands can print provider API keys into logs

- Kind: privacy
- File: `scripts/e2e/parallels/host-command.ts`
- Line: 54
- Evidence: `run()` reports failures with `die(\`command failed (${status}): ${[command, ...args].join(" ")}\`)`. New callers place provider secrets directly in those args, for example Linux onboard uses `${this.auth.apiKeyEnv}=${this.auth.apiKeyValue}` at `scripts/e2e/parallels/linux-smoke.ts:545`, macOS does the same through `MacosGuest` env args at `scripts/e2e/parallels/guest-transports.ts:50`, and npm-update embeds secrets in inline bash/PowerShell scripts at `scripts/e2e/parallels/npm-update-smoke.ts:409`, `scripts/e2e/parallels/npm-update-smoke.ts:457`, and `scripts/e2e/parallels/npm-update-smoke.ts:501`. A focused dummy check confirmed the helper prints sensitive argv on nonzero exit.
- Impact: Any failing auth/onboard/agent/update phase can leak OpenAI/Anthropic/Minimax keys into terminal logs, Testbox logs, or copied failure reports.
- Suggested fix: Do not put provider keys in argv. Use guest temp secret files/stdin or `spawnSync` env where possible, and make `run()` redact known secret patterns before printing command metadata.
- Confidence: high

### Medium: Expected phase failures now abort the whole script instead of producing lane summaries

- Kind: regression
- File: `scripts/e2e/parallels/host-command.ts`
- Line: 54
- Evidence: `run()` calls `process.exit(1)` through `die()` on nonzero exit, but `PhaseRunner.phaseReturns()` only catches thrown errors at `scripts/e2e/parallels/phase-runner.ts:41`. The macOS and Windows upgrade prechecks expect `runRefOnboard()` failures to be captured as `latest-ref-fail` (`scripts/e2e/parallels/macos-smoke.ts:481`, `scripts/e2e/parallels/windows-smoke.ts:428`), yet a failed guest command exits the Node process before `phaseReturns()` can return `false`. A focused check with `phaseReturns(... run("bash", ["-lc", "exit 9"]) ...)` exited immediately instead of printing the caught result.
- Impact: Known-baseline precheck failures and ordinary guest command failures can skip JSON summaries, lane status marking, server cleanup, and later phases.
- Suggested fix: Change `run()` to throw a redacted command error instead of exiting; reserve `die()` for top-level argument/preflight failures.
- Confidence: high

### Medium: Shared package locking was dropped, making parallel Parallels packs race on `dist`

- Kind: concurrency
- File: `scripts/e2e/parallels/package-artifact.ts`
- Line: 31
- Evidence: `ensureCurrentBuild()` accepts `lockDir` but immediately discards it with `void input.lockDir`. `packOpenClaw()` then runs `pnpm build`, writes package dist inventory, optionally stages bundled plugin runtime deps, and runs `npm pack` without the old `openclaw-parallels-build.lock` guard.
- Impact: Independent Parallels lanes run in parallel by design; without the lock, two lanes can mutate or package `dist` concurrently and produce flaky or mixed tarballs.
- Suggested fix: Reintroduce a cross-process lock around the full build/inventory/stage/pack critical section, including stale-lock handling from the former shell helper.
- Confidence: high

### Medium: macOS upgrade-only runs still build and pack current main unnecessarily

- Kind: regression
- File: `scripts/e2e/parallels/macos-smoke.ts`
- Line: 409
- Evidence: `needsHostTgz()` returns `true` whenever `targetPackageSpec` is empty, so `--mode upgrade` with no target package still calls `packOpenClaw()` at `scripts/e2e/parallels/macos-smoke.ts:316`. The old shell path only packed for fresh/both or an explicit target-package upgrade; default macOS upgrade is latest stable -> `openclaw update --channel dev` and does not need a host tarball.
- Impact: A release-to-dev upgrade smoke can fail before touching the VM if the local checkout cannot build/package, even though that lane should validate the guest updater against GitHub main.
- Suggested fix: Make `needsHostTgz()` mode-aware: for `upgrade` without `--target-package-spec`, skip packing/server startup.
- Confidence: high

### Low: npm-update Linux fallback can choose an unsupported old Ubuntu VM

- Kind: compatibility
- File: `scripts/e2e/parallels/npm-update-smoke.ts`
- Line: 632
- Evidence: When `Ubuntu 24.04.3 ARM64` is missing, the aggregate wrapper now returns `names.find((name) => /ubuntu/i.test(name))`. The previous resolver selected the closest Ubuntu VM with major version `>= 24`.
- Impact: Hosts with both old and current Ubuntu guests can pick an arbitrary older VM based on `prlctl` JSON order, breaking the Linux lane on an unsupported baseline.
- Suggested fix: Reuse the version-aware fallback from the standalone Linux smoke resolver for the npm-update aggregate.
- Confidence: medium

### Low: npm-update same-guest updates no longer verify update status

- Kind: test_gap
- File: `scripts/e2e/parallels/npm-update-smoke.ts`
- Line: 402
- Evidence: The macOS update path runs `openclaw update`, checks `openclaw --version`, then goes straight to gateway restart/status. The Windows and Linux update paths follow the same pattern. The documented contract for `pnpm test:parallels:npm-update` includes verifying `openclaw update status --json`.
- Impact: A regression that installs the right files but fails to persist channel/install metadata can pass the aggregate update smoke.
- Suggested fix: Add `openclaw update status --json` checks after each same-guest update and assert the expected channel/install kind for the target.
- Confidence: high
