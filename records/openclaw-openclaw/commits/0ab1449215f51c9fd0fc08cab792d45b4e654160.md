# Commit 0ab1449

## Summary

Found one medium-confidence abort race in the touched dispatch path. The focused tests added by the commit pass, but they do not cover a routed block-streaming send already in progress when `/stop` aborts the dispatch operation.

## Findings

### Medium: Routed block replies can still ignore dispatch aborts when a block timeout signal is present

- Kind: reliability
- File: `src/auto-reply/reply/dispatch-from-config.ts`
- Line: 2066
- Evidence: The new abort-aware block callback checks `isDispatchOperationAborted()` before delivery, but the routed path still calls `sendPayloadAsync(normalizedPayload, context?.abortSignal, false)`. `sendPayloadAsync` then chooses `abortSignal ?? dispatchAbortOperation?.abortSignal`, so any block context signal masks the dispatch abort signal. The block pipeline always supplies its own timeout `AbortController.signal` to `onBlockReply` in `src/auto-reply/reply/block-reply-pipeline.ts:149`, so cross-provider routed block delivery receives the timeout signal instead of the `/stop` dispatch signal.
- Impact: If `/stop` lands after the pre-send abort check but while `routeReply` / `sendDurableMessageBatch` is still pending, dispatch can resolve as aborted while the routed block send continues and may still deliver a visible late block reply. That undercuts the commit’s stated “late replies are cleared instead of leaking after stop” behavior for routed block streaming.
- Suggested fix: Compose the block context signal with `dispatchAbortOperation?.abortSignal` for routed block sends, for example with `AbortSignal.any` where available, so either the block timeout or the dispatch abort cancels delivery.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Unclear. The code path is concrete, but I did not build a new harness test. A focused repro would make `shouldRouteToOriginating` true, make `routeReply` hang after block delivery starts, abort the reply operation, and assert the route signal aborts before send completion.
- Is this the best way to solve the issue? Yes. Composing the two signals preserves the existing block timeout behavior while adding the dispatch abort ownership this commit intends.
