The commit makes the live cache release job much more likely to time out before a normal successful run can finish.

## Findings

### Medium: live-cache release validation can now be canceled before a known-successful run duration

- Kind: reliability
- File: `.github/workflows/openclaw-live-and-e2e-checks-reusable.yml`
- Line: 458
- Evidence: The commit lowers `validate_release_live_cache` from `timeout-minutes: 40` to `20`, and each attempt from `12m` to `8m` at line 495. Recent Actions evidence shows this lane can legitimately take longer: run `25480566662` had `validate_release_live_cache` start at `2026-05-07T06:49:05Z` and complete successfully at `2026-05-07T07:10:30Z`; its log shows `live-cache attempt 1/3` at `06:49:18Z` and `[live-cache] all regression floors satisfied` at `07:10:25Z`. - Impact: A live-cache validation that is still making progress and can pass will now be killed by the job timeout, or by the 8-minute attempt timeout, turning release validation into a false failure.
- Suggested fix: Keep the outer job timeout above the observed successful runtime plus setup margin, or split the cache probe into smaller provider/lane shards before applying an 8-minute per-attempt cap.
- Confidence: high
