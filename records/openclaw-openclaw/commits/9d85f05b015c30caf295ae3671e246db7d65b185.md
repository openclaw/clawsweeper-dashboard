# Commit 9d85f05

## Summary

Found one Matrix regression. Telegram looks safe because `sendMessageTelegram` chunks text internally, but Matrix also embeds the full presentation as custom event content, so removing the text cap can make otherwise chunkable presentation messages fail Matrix’s event-size limit.

## Findings

### Medium: Matrix presentation metadata can exceed the event-size limit

- Kind: regression
- File: `extensions/matrix/src/outbound.ts`
- Line: 108
- Evidence: Matrix presentation rendering stores the adapted presentation under `com.openclaw.presentation` in `channelData.matrix.extraContent` at `extensions/matrix/src/outbound.ts:75`, and `sendMessageMatrix` merges `opts.extraContent` into the first Matrix event unchanged at `extensions/matrix/src/matrix/send.ts:237`. This commit removed `limits.text.maxLength: 4000` from both Matrix outbound adapters, so `adaptMessagePresentationForChannel` no longer truncates text/context/title blocks before that metadata is attached. A local size probe with a 70,000-character presentation text block produced 74,139 bytes of Matrix event content before the rest of the Matrix event envelope/signatures; the Matrix spec requires complete events to be at most 65,536 bytes and says additional keys are only bounded by that total event limit. Source: Matrix Specification, Client-Server API size limits, https://spec.matrix.org/latest/client-server-api/#size-limits. ([spec.matrix.org](https://spec.matrix.org/latest/client-server-api/))
- Impact: A long presentation text block that should now be deliverable via Matrix body chunking can still fail at the first Matrix send because the unchunked custom presentation metadata makes the event too large. This breaks the commit’s intent for Matrix and can fail the whole outbound delivery path for long presentation payloads.
- Suggested fix: Keep chunkable fallback text intact, but bound the Matrix-specific `com.openclaw.presentation` metadata separately before attaching it to `extraContent`, or omit large text fields from Matrix custom metadata while preserving the full fallback body for `sendMessageMatrix` chunking.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: send a Matrix presentation payload with a large text block, for example 70,000 ASCII characters. Current rendering preserves that whole block in `com.openclaw.presentation`; a local byte-length probe showed the content alone exceeds Matrix’s 65,536-byte event limit.
- Is this the best way to solve the issue? yes: restoring the old generic presentation text cap would avoid the oversized metadata, but it would also undo the intended body chunking behavior. A Matrix-only metadata cap/omission preserves the behavior change while keeping Matrix event content within protocol limits.
