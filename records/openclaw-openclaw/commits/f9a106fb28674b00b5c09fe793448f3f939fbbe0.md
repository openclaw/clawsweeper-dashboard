# Commit f9a106fb28

## Summary

Found one CI reliability regression: the reviewed commit caps the whole OpenAI package-update Docker job at 30 minutes, leaving no room for the lane-level work it runs or the workflow setup/artifact steps around it.

## Findings

### Medium: OpenAI package-update job can time out before its own lane budget and artifacts complete

- Kind: reliability
- File: `.github/workflows/openclaw-live-and-e2e-checks-reusable.yml`
- Line: 621
- Evidence: The commit changes `package-update-openai` from `timeout_minutes: 180` to `timeout_minutes: 30`, and the job uses that value directly as `timeout-minutes: ${{ matrix.timeout_minutes }}`. The OpenAI package-update chunk runs `install-e2e-openai` and `codex-on-demand`; in the reviewed commit, `install-e2e-openai` already allows a 1500-second agent turn inside the command, while the scheduler fallback lane timeout is 120 minutes. In current `main`, the dry plan still shows this chunk includes `codex-on-demand` with `timeoutMs: 1800000` (30 minutes), and current `main` has tightened the matrix job further to 20 minutes. GitHub documents `jobs.<job_id>.timeout-minutes` as the maximum time before GitHub automatically cancels the whole job, so this cap applies to checkout, setup, planning, image pulls, credential validation, the lane pool, summarization, and artifact upload, not just the lane process. ([docs.github.com](https://docs.github.com/en/actions/reference/workflows-and-actions/workflow-syntax?s=09&utm_source=openai))
- Impact: A valid slow OpenAI package-update run can be canceled by the outer Actions job before the scheduler reports a lane timeout, writes `summary.json`/`failures.json`, or uploads the Docker E2E artifacts needed for diagnosis and rerun commands. That makes release validation flaky and less debuggable.
- Suggested fix: Set the job timeout above the slowest lane timeout plus setup/cleanup margin, or reduce the scheduler lane timeout and command-level timeouts to a value comfortably below the job timeout. For the current lane shape, the job should be above 30 minutes, not equal to or below it.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The focused dry plan `OPENCLAW_DOCKER_ALL_PROFILE=release-path OPENCLAW_DOCKER_ALL_CHUNK=package-update-openai OPENCLAW_DOCKER_ALL_RELEASE_PROFILE=stable OPENCLAW_DOCKER_ALL_INCLUDE_OPENWEBUI=true node scripts/test-docker-all.mjs --plan-json` shows the chunk still contains a 30-minute `codex-on-demand` lane. In Actions, dispatch the reusable workflow with release-path suites enabled and no `docker_lanes`; if the OpenAI chunk approaches its lane timeout, the outer job cancels first.
- Is this the best way to solve the issue? yes. The workflow-level timeout should be an outer guard with enough margin for all expected steps, while the scheduler owns per-lane timeout reporting and artifacts.
