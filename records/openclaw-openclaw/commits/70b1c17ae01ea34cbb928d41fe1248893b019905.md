# Commit 70b1c17

## Summary

Found one regression in the new manifest-id preference. The fix handles WeCom/Yuanbao, but it also lets an installed plugin that merely claims an existing bundled channel id win over the bundled owner without declaring `preferOver`.

## Findings

### Medium: Channel claims can bypass the documented `preferOver` replacement gate

- Kind: security
- File: `src/config/plugin-auto-enable.shared.ts`
- Line: 254
- Evidence: The changed fallback now returns `claims[0]?.plugin.id` whenever any manifest claims a configured channel. In a focused smoke with configured `channels.slack`, restrictive `plugins.allow: ["telegram"]`, and registry order `[replacement-slack(global), slack(bundled)]`, `applyPluginAutoEnable` produced `plugins.allow: ["telegram", "replacement-slack"]` and `plugins.entries.replacement-slack.enabled = true`. The selected id is then persisted through `ensurePluginAllowlisted` at `src/config/plugin-auto-enable.shared.ts:959`.
- Impact: A discoverable non-bundled plugin can become eligible to load just by claiming a configured bundled channel id, even when the operator’s allowlist did not include it and the plugin did not declare `channelConfigs.<id>.preferOver`. That weakens `plugins.allow` as an approved-plugin inventory and conflicts with the documented duplicate-channel contract in `docs/plugins/manifest.md` and `docs/tools/plugin.md`.
- Suggested fix: Keep the new behavior for alias/id split channels that have no bundled owner claim, but when a claim id matches the normalized built-in channel id, prefer that built-in/bundled owner unless another claimant explicitly declares `preferOver` for it.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The focused `pnpm exec tsx` smoke above directly exercised current `main` with a fake manifest registry and showed `replacement-slack` being enabled and allowlisted.
- Is this the best way to solve the issue? yes. The narrow fix is in candidate selection: distinguish “built-in catalog alias with no bundled owner” from “real bundled channel owner plus duplicate claimant”; keep `preferOver` as the replacement mechanism.
