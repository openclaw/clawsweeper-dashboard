# Commit 892f611065

## Summary

Found one low-severity reliability issue in the new Mantis Telegram Desktop evidence packager: relative lane output directories are resolved against the harness checkout instead of the passed lane worktree root.

## Findings

### Low: Relative lane output dirs fail from the harness checkout

- Kind: reliability
- File: `scripts/mantis/build-telegram-desktop-proof-evidence.mjs`
- Line: 218
- Evidence: `writeTelegramDesktopProofEvidence` resolves `--baseline-output-dir` and `--candidate-output-dir` with `path.resolve(args.*_output_dir)`, so relative values are interpreted relative to the current process cwd. The Mantis prompt passes separate `--baseline-repo-root` / `--candidate-repo-root`, and the Telegram proof skill documents session output dirs as repo-relative `.artifacts/...` paths.
- Impact: if the Mantis agent follows the repo-relative output-dir pattern after running the proof inside each detached worktree, the packager looks for `.artifacts/.../telegram-user-crabbox-session-summary.json` under the harness checkout, not under the baseline/candidate worktree. The workflow then fails before producing/publishing the evidence comment.
- Suggested fix: resolve lane output dirs relative to the corresponding lane repo root when they are not absolute, e.g. `path.isAbsolute(outputDir) ? path.resolve(outputDir) : path.resolve(repoRoot, outputDir)`, and add a test using repo-relative lane output dirs.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. I created temp baseline/candidate worktrees with valid repo-relative summary artifact paths; invoking the script with `--baseline-output-dir .artifacts/...` failed with `ENOENT` under the harness checkout. The same temp setup passed when the lane output dirs were absolute.
- Is this the best way to solve the issue? yes. The script already requires each lane repo root and resolves summary artifact paths relative to that root, so applying the same rule to the lane output dir keeps the CLI contract consistent without changing artifact publication behavior.
