## Summary

Found one low-severity QA report regression. The suite execution behavior looks covered by focused tests, but runtime-parity QA reports can now describe the worker topology incorrectly.

## Findings

### Low: Runtime-parity suite reports no longer mark isolated workers

- Kind: bug
- File: `extensions/qa-lab/src/suite.ts`
- Line: 769
- Evidence: `runQaRuntimeParitySuite` writes artifacts without passing `isolatedWorkers`, while `extensions/qa-lab/src/qa-channel-transport.ts:105` now treats only `isolatedWorkers === true` as isolated. Before this commit, qa-channel report notes used `concurrency > 1`, so a default multi-scenario runtime-parity run could report isolated concurrent workers. After this commit, the same path falls through to `"Scenarios run serially in one gateway worker."` even though runtime-parity cells are launched through per-cell `runQaSuite(...)` workers.
- Impact: Runtime-parity QA artifacts can misstate how the suite ran, which weakens release/debug evidence when operators inspect reports for concurrency and isolation behavior.
- Suggested fix: Pass `isolatedWorkers: true` from the runtime-parity artifact writer, or add a runtime-parity-specific report note that accurately describes per-cell isolated workers.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. Run a runtime-pair QA suite with more than one selected scenario and inspect `qa-suite-report.md`; the report-note branch in `qa-channel-transport.ts` will use the serial one-worker text because `suite.ts` does not pass `isolatedWorkers` in the runtime-parity artifact call.
- Is this the best way to solve the issue? Yes, if the existing note is meant to remain generic. A more precise alternative is a separate runtime-parity note because that path launches runtime cells rather than one ordinary gateway worker per scenario.
