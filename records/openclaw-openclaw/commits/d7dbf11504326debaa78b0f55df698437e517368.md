## Summary

Found one low-severity regression risk in the new managed npm-root manifest flow.

## Findings

### Low: npm rewrites the managed root spec immediately after the helper preserves it

- Kind: regression
- File: `src/plugins/install.ts`
- Line: 1203
- Evidence: `resolveManagedNpmRootDependencySpec` writes the intended exact version or tag into `~/.openclaw/npm/package.json` (`src/infra/npm-managed-root.ts:45`), but `installPluginFromNpmSpec` then runs `npm install ... <spec>` against that same root (`src/plugins/install.ts:1192-1204`). A focused npm check showed `package.json` changes from `"is-odd":"3.0.0"` to `"is-odd":"^3.0.0"` when the command includes `is-odd@3.0.0`, and from `"latest"` to `"^3.0.1"` when the command includes `is-odd@latest`. Running `npm install --prefix <root>` without the package argument preserved the exact/tag dependency strings.
- Impact: the managed npm root no longer reliably records exact/tag plugin specs after a successful install. The durable OpenClaw install record still preserves the requested spec, and `package-lock.json` usually limits immediate drift, so this is not a high-severity break. But if the managed root lockfile is missing/stale during a later plugin repair/install, npm can resolve an already-installed exact/tag plugin through the widened caret range, while OpenClaw still believes the recorded exact/tag install is intact. This conflicts with the documented ŌĆ£exact versions and explicit tags are preservedŌĆØ update contract in `docs/plugins/manage-plugins.md:80-90`.
- Suggested fix: after `upsertManagedNpmRootDependency`, run npm against the manifest (`npm install --prefix <npmRoot>`) instead of passing `spec` again, or otherwise pass npm save options that prevent rewriting the dependency value. Add an integration-style test in `src/plugins/install.npm-spec.test.ts` that asserts the final managed root `package.json` keeps exact versions and tags.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes. In a temporary directory, write `{"private":true,"dependencies":{"is-odd":"3.0.0"}}`, run `npm install --prefix "$tmp" is-odd@3.0.0 --omit=dev --ignore-scripts --no-audit --no-fund --loglevel=error`, and observe `package.json` rewritten to `^3.0.0`. Running the same install without the trailing package spec preserves `3.0.0`.
- Is this the best way to solve the issue? yes, likely. Installing from the manifest after the helper writes it matches the helperŌĆÖs purpose and avoids npmŌĆÖs save-prefix rewrite.
