# Commit c1996f5

## Summary

Found one concrete risk in the new Speech buffer normalizer: the downmix divides by the total hardware channel count, which can make the common pro-interface case of “one live mic channel, many silent channels” much quieter before it reaches Apple Speech.

## Findings

### Medium: Sparse multichannel mic input is attenuated by the full device channel count

- Kind: bug
- File: `apps/macos/Sources/OpenClaw/SpeechAudioBufferNormalizer.swift`
- Line: 35
- Evidence: `speechCompatibleBuffer(from:)` routes any `channelCount > 2` buffer through `downmixFloatBuffer` when possible, and that loop sums every channel then multiplies by `1.0 / Float(channelCount)`. On a 16-channel interface with one active mic channel and 15 silent channels, a sample with amplitude `1.0` becomes `0.0625`. The added test fills all four channels, so it does not cover the sparse active-channel case. Apple documents `floatChannelData` as channel pointers for the buffer’s channel count with `frameLength` valid samples, and the Speech request consumes the appended `AVAudioPCMBuffer`, so this attenuation is exactly what Speech receives. citeturn0search0turn0search2
- Impact: The commit targets pro audio interfaces, where a single microphone is commonly plugged into one input on a device exposing many channels. This can leave Push-to-Talk, Talk Mode, Voice Wake, and the tester still producing empty or unreliable transcripts, while the existing RMS meters can still look healthy because they read the original first channel.
- Suggested fix: Do not normalize by total hardware channel count. Prefer a selected/preferred input channel when available, or mix only active channels based on per-channel RMS with clipping/limiting. Add tests for 4/8/16-channel buffers where only channel 0 or channel 1 contains speech.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes for the downmix math: construct a deinterleaved 16-channel float `AVAudioPCMBuffer` with only one populated channel and call `SpeechAudioBufferNormalizer.speechCompatibleBuffer(from:)`; the output is divided by 16. Live confirmation that this still causes empty Apple Speech transcripts needs macOS hardware.
- Is this the best way to solve the issue? Unclear. The shared normalizer is the right surface, but the mix policy should preserve speech level for sparse multichannel interfaces instead of averaging in silent channels.
