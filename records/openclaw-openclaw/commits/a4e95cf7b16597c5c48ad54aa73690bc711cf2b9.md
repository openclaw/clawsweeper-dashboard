# Commit a4e95cf

## Summary

Found one low-severity reliability issue in the new base64 size guard: it allocates a full normalized copy of the untrusted media payload before enforcing the max byte limit.

## Findings

### Low: Base64 limit check still makes an unbounded full-string copy

- Kind: reliability
- File: `src/cli/nodes-camera.ts`
- Line: 181
- Evidence: `estimateDecodedBase64Bytes()` calls `base64.replace(/\s+/g, "")`, which allocates another string proportional to the node-supplied payload before the new `maxBytes` check can reject it. The affected write helper is reached by camera and screen node media paths, including `src/cli/nodes-cli/register.screen.ts:57`, `src/cli/nodes-cli/register.camera.ts:174`, and `src/agents/tools/nodes-tool-media.ts:374`. The repo already has `src/media/base64.ts:1`, whose comment explicitly avoids `trim()`/`replace()` because they allocate a second potentially huge string before `Buffer.from`.
- Impact: A paired or compromised node returning an oversized base64 media payload can still force a large extra allocation in the OpenClaw CLI or agent process before rejection. The commit bounds file writes, but the rejection path can still create avoidable memory pressure or OOM risk.
- Suggested fix: Replace the local estimator with `estimateBase64DecodedBytes` from `src/media/base64.ts`, or use the same non-allocating loop locally.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes; mock a node media payload with base64 over the 250 MB decoded limit and call `writeBase64ToFile` or a screen/camera node path. I did not run a full-size reproduction because the failure mode is resource exhaustion.
- Is this the best way to solve the issue? yes; the existing shared estimator is designed for this exact pre-`Buffer.from` budget check and avoids the extra copy.
