# Commit 0e76dafe

## Summary

Found one low-severity test regression. The runtime code on current `main` still removes pending Telegram startup probe waiters on abort, but this commit weakens the regression test so it would no longer catch the deadlock it is named to prevent.

## Findings

### Low: Queued abort test no longer proves the queued startup settles while probe slots are full

- Kind: test_gap
- File: `extensions/telegram/src/channel.gateway.test.ts`
- Line: 502
- Evidence: Before this commit, the test aborted the queued account and immediately awaited `queued.task` before releasing the two occupied startup probe slots. After this commit, it aborts the queued account, releases the occupied probes at lines 504-506, then awaits `queued.task` as part of `Promise.all` at line 507. The production contract is in `extensions/telegram/src/startup-probe-limiter.ts:71`: the abort listener removes the pending waiter and rejects it immediately. The fallback dequeue path at `extensions/telegram/src/startup-probe-limiter.ts:48` also rejects already-aborted waiters after a slot opens, so the new test would still pass if the immediate abort listener stopped working.
- Impact: A future regression that only abandons the queued startup after an active probe slot releases would pass this test, but the real deadlock case is when the active startup probes remain hung and the queued aborted account never settles.
- Suggested fix: Keep the non-hanging shape, but assert settlement before releasing active probes. For example, attach a `queuedSettled` flag to `queued.task.finally(...)`, call `abortQueued.abort()`, `await waitForCondition(() => queuedSettled, ...)`, and release active probes in a `finally` block so failures clean up.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. By static path comparison: a limiter that drops the `abortSignal.addEventListener` rejection but keeps the `drainStartupProbeWaiters()` aborted-waiter branch would satisfy the new test once the active probes are released, while failing the old immediate-settlement assertion and the real hung-probe scenario.
- Is this the best way to solve the issue? Yes. The safer fix is to restore the pre-release assertion with a bounded wait and cleanup, avoiding the old indefinite test hang while still checking the intended contract.
