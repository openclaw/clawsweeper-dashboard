# Commit 4c40686

## Summary

Found one concrete trust-boundary regression in the new Codex reserved-command ownership check.

## Findings

### Medium: Global Codex plugins can spoof official package metadata to claim trusted `/codex`

- Kind: security
- File: `src/plugins/registry.ts`
- Line: 283
- Evidence: `isOfficialCodexPluginRecord` now returns true for any `origin: "global"` plugin whose `PluginRecord.packageName` is exactly `@openclaw/codex`. That field is threaded from `manifestRecord.packageName` in `src/plugins/loader.ts:1772`, and discovery sources it from the plugin package’s own `package.json` `name` field at `src/plugins/discovery.ts:573`. Auto-discovered global plugins from the user plugin directory are also `origin: "global"` (`src/plugins/discovery.ts:1171`), and non-bundled/non-workspace plugins default enabled (`src/plugins/config-activation-shared.ts:292`). The new test accepts a global `/tmp/.openclaw/extensions/codex` record solely because it carries `packageName: "@openclaw/codex"` (`src/plugins/contracts/host-hooks.contract.test.ts:164`), while the negative test only omits `packageName` and does not cover package-name spoofing (`src/plugins/contracts/host-hooks.contract.test.ts:199`).
- Impact: A globally installed or manually placed plugin with manifest id `codex` and `package.json` name `@openclaw/codex` can bypass the reserved command guard and register `/codex` as host-trusted. Trusted reserved command owners receive privileged command context such as `senderIsOwner` and diagnostics routing/upload flags (`src/plugins/commands.ts:249`, `src/plugins/commands.ts:256`, `src/plugins/commands.ts:263`, `src/plugins/commands.ts:270`). This weakens the intended “only bundled or verified official Codex package” boundary to self-attested package metadata.
- Suggested fix: Do not treat `package.json#name` as provenance. Propagate verified install provenance into `PluginRecord` and require the Codex record to come from a trusted install record, for example managed npm under the actual `node_modules/@openclaw/codex` package or a ClawHub record whose canonical package/source metadata is `@openclaw/codex`. Add a negative contract test for `origin: "global"`, `id: "codex"`, `packageName: "@openclaw/codex"`, and an unverified global extensions path.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. Construct a global plugin record with `id: "codex"`, `origin: "global"`, `packageName: "@openclaw/codex"`, and a non-`node_modules/@openclaw/codex` root, then register a command with `ownership: "reserved"`. The committed contract test already demonstrates that this record is accepted.
- Is this the best way to solve the issue? Yes. The intended behavior is provenance-based trust for official package commands, so the check should bind to install provenance, not package-authored metadata.
