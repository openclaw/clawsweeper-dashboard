# Commit 0fcf2c64

## Summary

Found one concrete hole in the persisted-turn replay fix. The new guard is applied after explicit overflow compaction, but the adjacent retry path for attempt-level compaction still retries the original prompt after the current inbound message may already have been persisted.

## Findings

### Medium: attempt-level compaction retry can still replay the persisted inbound turn

- Kind: bug
- File: `src/agents/pi-embedded-runner/run.ts`
- Line: 1503
- Evidence: This commit records `lastPersistedCurrentMessageId` via `onUserMessagePersisted` and uses it only in the explicit `contextEngine.compact()` success path at `src/agents/pi-embedded-runner/run.ts:1647`, where it switches to the continuation prompt and sets `suppressNextUserMessagePersistence = true`. The earlier branch for `hadAttemptLevelCompaction` at `src/agents/pi-embedded-runner/run.ts:1503` still does `continue` after logging “retrying prompt without additional compaction”; unless `preflightRecovery?.source === "mid-turn"`, it leaves `nextAttemptPromptOverride` unset and leaves `suppressNextUserMessagePersistence` false.
- Impact: If `runEmbeddedAttempt` persists the channel user turn, performs SDK/Pi attempt-level compaction, then returns a context-overflow prompt/assistant error with `compactionCount > 0`, the next loop iteration receives the original `params.prompt` and can append the same `currentMessageId` again. That is the same stale-turn replay shape described in issue `#76424`, just through the in-attempt compaction branch instead of the explicit compaction branch covered by the new test.
- Suggested fix: Apply the same `params.currentMessageId === lastPersistedCurrentMessageId` check in the `hadAttemptLevelCompaction` branch. When true, set `nextAttemptPromptOverride = MID_TURN_PRECHECK_CONTINUATION_PROMPT` and `suppressNextUserMessagePersistence = true` before continuing.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes, with a focused unit variant of `src/agents/pi-embedded-runner/run.overflow-compaction.loop.test.ts`: make the first mocked attempt invoke `onUserMessagePersisted`, return a likely overflow `promptError`, and set `compactionCount: 1`; current code will take the `hadAttemptLevelCompaction` branch and call the second attempt with the original prompt and no suppression.
- Is this the best way to solve the issue? yes; the explicit compaction branch already establishes the intended policy, and the attempt-level compaction branch is the same “already persisted current turn, retry after compaction” state transition.
