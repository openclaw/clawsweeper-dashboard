# Commit 624d920

## Summary

Found one regression in the focused Full Release Validation rerun verifier: `rerun_group=npm-telegram` can now go green without dispatching the Telegram package child when no package spec is supplied.

## Findings

### Medium: `npm-telegram` focused reruns can pass without running any validation

- Kind: regression
- File: `.github/workflows/full-release-validation.yml`
- Line: 1009
- Evidence: The `npm_telegram` job only runs for `rerun_group=npm-telegram` when `npm_telegram_package_spec` or `release_package_spec` is non-empty (`.github/workflows/full-release-validation.yml:663`). The new required-child mapping marks `ci`, `plugin-prerelease`, and release-check groups as required, but has no required flag for `npm-telegram` (`.github/workflows/full-release-validation.yml:976`). The verifier then treats a skipped `npm_telegram` job with no run id as optional unconditionally (`.github/workflows/full-release-validation.yml:1009`). A shell simulation of the verifier’s no-run-id branch exits `0` for `RERUN_GROUP=npm-telegram` with all child run ids empty.
- Impact: A manually dispatched focused Telegram package rerun that omits `release_package_spec`/`npm_telegram_package_spec` can finish successfully while proving nothing. This contradicts the documented contract that `npm-telegram` requires one of those specs and can leave a misleading green Full Release Validation run/manifest.
- Suggested fix: Add a required `npm_telegram_required` flag set for `RERUN_GROUP=npm-telegram`, and pass it into the skipped-child check. Alternatively, fail earlier with a clear input validation error when `rerun_group=npm-telegram` has no package spec.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Dispatch Full Release Validation with `rerun_group=npm-telegram` and no `release_package_spec` or `npm_telegram_package_spec`; statically, the `npm_telegram` job condition is false, all other focused groups are now optional, and the verifier accepts skipped `npm_telegram`.
- Is this the best way to solve the issue? yes. The parent verifier already encodes required child workflows per focused rerun group; adding `npm_telegram_required` preserves that design and closes the missing case.
