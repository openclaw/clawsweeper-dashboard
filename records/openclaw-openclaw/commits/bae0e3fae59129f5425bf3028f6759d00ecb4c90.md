# Commit bae0e3f

## Summary

Found one release-tooling regression: the helper now lets an internal `npm install` write to stdout, which breaks the `--pack-dry-run` JSON stdout contract for plugins that need portable optional bundled dependencies.

## Findings

### Medium: Internal optional dependency install can corrupt pack JSON stdout

- Kind: regression
- File: `scripts/lib/plugin-npm-package-manifest.mjs`
- Line: 297
- Evidence: The changed line sets the helper’s internal optional bundled dependency install to `stdio: ["ignore", "inherit", "inherit"]`. `scripts/plugin-npm-publish.sh` intentionally sends normal logs to stderr in `--pack-dry-run` mode (`scripts/plugin-npm-publish.sh:21`) and then runs `npm pack --dry-run --json --ignore-scripts` through this helper (`scripts/plugin-npm-publish.sh:142`), so stdout is expected to be the pack JSON. A focused temp-directory check with the same npm install flags on Node `v24.15.0` / npm `11.12.1` showed npm still writes `added 1 package in ...` to stdout with `--loglevel=error`.
- Impact: For a plugin whose bundled dependency graph has a missing optional dependency, `scripts/plugin-npm-publish.sh --pack-dry-run <plugin>` will emit npm’s install summary before the `npm pack --json` payload. Any release automation or maintainer workflow parsing that stdout as JSON will fail even though the pack itself succeeded.
- Suggested fix: Keep the internal install stdout quiet or route it to stderr, for example restore `["ignore", "ignore", "inherit"]` here, or use `pipe` and forward any useful npm install output to stderr after the JSON-producing command’s stdout contract is preserved.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Use a publishable plugin fixture with a bundled dependency whose optional dependency is skipped by `npm ci`, then run the `npm install --force --omit=dev --omit=peer --legacy-peer-deps --ignore-scripts --no-audit --no-fund --package-lock=false --save=false --loglevel=error ...` path; npm writes an install summary to stdout, which the changed `inherit` forwards into the wrapper’s stdout.
- Is this the best way to solve the issue? yes. The helper’s internal dependency preparation is not the JSON-producing operation, so its stdout should not share the caller stdout channel.
