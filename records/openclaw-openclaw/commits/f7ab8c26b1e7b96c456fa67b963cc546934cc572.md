# Commit f7ab8c26

## Summary

Found one concrete regression in the new Codex context projection reserve mapping. The main scaling fix works in the focused tests, but a floor-only compaction config is now interpreted as the whole projection reserve instead of as a floor guard.

## Findings

### Medium: Floor-only reserve config removes Codex projection headroom

- Kind: regression
- File: `extensions/codex/src/app-server/context-engine-projection.ts`
- Line: 97
- Evidence: `resolveCodexContextEngineProjectionReserveTokens()` returns `configuredReserveTokensFloor` directly when `reserveTokens` is absent. With `{ agents: { defaults: { compaction: { reserveTokensFloor: 0 } } } }`, the resolver returns `0`, and `resolveCodexContextEngineProjectionMaxChars({ contextTokenBudget: 80_000, reserveTokens: 0 })` raises the rendered-context cap from the default-reserved `240_000` chars to `320_000` chars. The added test at `extensions/codex/src/app-server/context-engine-projection.test.ts:157` locks in that floor-only `0` behavior.
- Impact: `reserveTokensFloor` is documented and implemented in Pi settings as a minimum floor for `reserveTokens`, with `0` disabling only the floor guard. On the native Codex path, the same floor-only config now disables all projection reserve headroom, so large context-engine assemblies can consume space intended for developer instructions, current prompt, tools, and output. That can turn a previously valid native Codex context-engine turn into an oversized request or provider context-window failure.
- Suggested fix: treat floor-only config as a floor over the default projection reserve, e.g. use `Math.max(DEFAULT_PROJECTION_RESERVE_TOKENS, configuredReserveTokensFloor)` when `reserveTokens` is absent. Keep `reserveTokens: 0, reserveTokensFloor: 0` as the explicit way to request zero reserve.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Call the new resolver with `reserveTokensFloor: 0` and no `reserveTokens`; it returns `0` and expands an 80k-token projection cap from `240000` to `320000` chars.
- Is this the best way to solve the issue? yes. The bug is in config interpretation, not the context projection renderer itself; preserving the new budget-aware cap while resolving floor-only config as a floor keeps the intended fix.
