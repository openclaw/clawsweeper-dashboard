# Commit ef0dbcf

## Summary

Found one concrete gap in the current-tab export guard: `GET /snapshot` remains outside the new route-level policy gate, despite the changelog claiming snapshot is covered.

## Findings

### Medium: `/snapshot` can still collect from a blocked current tab through CDP fallback

- Kind: security
- File: `extensions/browser/src/browser/routes/agent.snapshot.ts`
- Line: 502
- Evidence: The new guard only runs through `withRouteTabContext({ enforceCurrentUrlAllowed: true })` in `extensions/browser/src/browser/routes/agent.shared.ts:127`, but `GET /snapshot` still resolves `profileCtx` and `tab` directly at `extensions/browser/src/browser/routes/agent.snapshot.ts:505` and `extensions/browser/src/browser/routes/agent.snapshot.ts:519`. Non-Chrome-MCP snapshot paths can then fall back to direct CDP reads via `snapshotRoleViaCdp` at `extensions/browser/src/browser/routes/agent.snapshot.ts:647` or `snapshotAria` at `extensions/browser/src/browser/routes/agent.snapshot.ts:749`; those CDP helpers accept only a `wsUrl` and have no current-page URL policy input (`extensions/browser/src/browser/cdp.ts:456`, `extensions/browser/src/browser/cdp.ts:921`).
- Impact: Under a strict browser SSRF policy, a selected tab already sitting on a blocked private-network URL can still be snapshotted through the CDP fallback path before any route-level current-tab URL denial. That leaves the same tab-scoped read/export class partially unfixed for snapshot output.
- Suggested fix: Route `GET /snapshot` through `withRouteTabContext` with `enforceCurrentUrlAllowed: true` before plan execution and add `/snapshot` to the blocked-current-tab HTTP regression matrix.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Configure strict browser SSRF, set the selected tab URL to a private/loopback URL such as `http://127.0.0.1:8080/admin`, and request `/snapshot` on a path that uses CDP fallback, such as no Playwright or Playwright role-snapshot failure. Source inspection shows collection proceeds without the new route-level gate.
- Is this the best way to solve the issue? yes. The shared helper is already the commit’s intended execution gate; applying it to `/snapshot` avoids backend-specific drift and matches the new route matrix pattern.
