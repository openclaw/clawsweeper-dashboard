# Commit 87f43ca

## Summary

Found one concrete regression: the new ClawHub trust flag is passed into `installPluginFromArchive`, but the archive installer drops it before scanning the extracted package. The intended fix therefore does not work for real ClawHub archive installs.

## Findings

### Medium: Trusted ClawHub archive installs still fail the dangerous-code scanner

- Kind: regression
- File: `src/plugins/install.ts`
- Line: 901
- Evidence: `src/plugins/clawhub.ts:956` passes `trustedSourceLinkedOfficialInstall` into `installPluginFromArchive`, but `installPluginFromArchive` rebuilds the params at `src/plugins/install.ts:899` through `pickPackageInstallCommonParams` without including `params.trustedSourceLinkedOfficialInstall`. The scanner only receives the override when it reaches `src/plugins/install.ts:696`, so this archive path loses it.
- Impact: official source-linked ClawHub plugins that trigger a critical built-in scanner pattern still fail with `security_scan_blocked`, which is the exact workflow this commit is trying to allow.
- Suggested fix: forward `trustedSourceLinkedOfficialInstall: params.trustedSourceLinkedOfficialInstall` in the object passed at `src/plugins/install.ts:901`, and add an archive-focused regression test beside the existing forced-unsafe archive test.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused smoke created a minimal plugin archive containing `child_process.spawn("google-chrome", [])` and called `installPluginFromArchive({ trustedSourceLinkedOfficialInstall: true })`; it returned `ok: false`, `code: "security_scan_blocked"`.
- Is this the best way to solve the issue? yes. The trust decision is already computed in ClawHub and the package-dir scanner already honors it; the missing hop is only the archive extraction wrapper.
