# Commit 53c2dbe

## Summary

Found one scoped-registry regression. The new wider-registry reuse can return an unfiltered active registry to callers that requested a scoped plugin set. At least the web provider shared runtime path then maps that wider registry using the original caller params, not the resolved scoped plugin ids, so unrelated providers can leak into a scoped provider resolution.

## Findings

### Medium: Scoped web provider resolution can return unrelated providers from a reused wider registry

- Kind: regression
- File: `src/plugins/loader.ts`
- Line: 1080
- Evidence: `getCompatibleActivePluginRegistry()` now returns `activeRegistry` when `scopedPluginLoadOptionsMatchWiderActiveCacheKey()` sees the requested `onlyPluginIds` inside a wider active registry. That preserves cache-key freshness, but it does not preserve the scoped registry shape. `src/plugins/web-provider-runtime-shared.ts:99` maps a compatible registry with `onlyPluginIds: params.onlyPluginIds`; when the scope was computed internally by `resolveCandidatePluginIds()` and the original params had no explicit scope, mapping is unfiltered.
- Impact: Runtime web search/fetch provider discovery can include providers from plugins outside the computed scope, including providers excluded by `origin` or manifest-candidate filtering. That can select or credential-check the wrong web provider and route web-search/web-fetch behavior through an unrequested plugin.
- Suggested fix: Either remove this wider-registry reuse from the generic `resolveRuntimePluginRegistry()` path, or return a scoped registry projection/filter all capability arrays before satisfying a scoped request. At minimum, web provider resolution should map with the resolved `loadOptions.onlyPluginIds`, not only the original `params.onlyPluginIds`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. I ran a focused `node --import tsx` smoke that seeded an active unscoped registry containing `alpha` and `beta`, then called `resolvePluginWebProviders()` with `resolveCandidatePluginIds()` returning only `alpha`; the result included both `alpha-fetch` and `beta-fetch`.
- Is this the best way to solve the issue? unclear. The central loader optimization is risky unless all returned registry surfaces are scoped. A central scoped projection is safer than requiring every caller to remember that a scoped registry request may now return a wider registry.
