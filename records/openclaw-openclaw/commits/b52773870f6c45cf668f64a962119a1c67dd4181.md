# Commit b527738

## Summary

Found one release-blocking workflow regression in the OpenClaw release publish path.

## Findings

### High: Release publish deletes the preflight manifest before validating it

- Kind: regression
- File: `.github/workflows/openclaw-release-publish.yml`
- Line: 124
- Evidence: The workflow downloads `openclaw-npm-preflight-${{ inputs.tag }}` into `preflight-manifest` at lines 124-132, then runs `actions/checkout@v6` at lines 134-139, then validates `preflight-manifest/preflight-manifest.json` at lines 145-158. `actions/checkout` documents `clean` as defaulting to true and running `git clean -ffdx && git reset --hard HEAD` before fetching, which removes untracked directories such as the downloaded artifact directory. ([github.com](https://github.com/actions/checkout?utm_source=openai)) A local temp-repo check confirmed `git clean -ffdx` removes an untracked `preflight-manifest/` directory.
- Impact: `OpenClaw Release Publish` with the default `publish_openclaw_npm=true` will fail in `resolve_release_target` before dispatching plugin npm, ClawHub, or OpenClaw npm publish, because the manifest downloaded from the preflight run is gone by the time the validation step runs.
- Suggested fix: Move `Download OpenClaw npm preflight manifest` after `Checkout release tag`, or download it outside the checkout workspace, for example under `$RUNNER_TEMP`, and validate it from there. Keeping checkout cleaning enabled is safer than setting `clean: false`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Dispatch `OpenClaw Release Publish` with `publish_openclaw_npm=true` and a valid `preflight_run_id`; the artifact download happens before checkout, checkout cleans untracked files, and the manifest validation then fails as missing.
- Is this the best way to solve the issue? yes. Reordering the download after checkout, or using `$RUNNER_TEMP`, preserves the clean checkout contract while keeping the preflight provenance validation.
