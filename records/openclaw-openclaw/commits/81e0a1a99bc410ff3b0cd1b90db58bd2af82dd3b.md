# Commit 81e0a1a99b

## Summary

Found two iMessage message-loss regressions: one affects all live iMessage echo handling after the persisted text echo TTL was widened to 12 hours, and one affects opt-in catchup for self-chat rows.

## Findings

### Medium: 12-hour persisted text echo matches can drop legitimate repeated user messages

- Kind: data_loss
- File: `extensions/imessage/src/monitor/persisted-echo-cache.ts`
- Line: 19
- Evidence: The commit widens persisted echo retention to 12 hours, and `hasPersistedIMessageEcho` still returns true on text equality alone at `extensions/imessage/src/monitor/persisted-echo-cache.ts:237`. `createSentMessageCache.has` consults that persisted store before the in-memory message-id mismatch short-circuit at `extensions/imessage/src/monitor/echo-cache.ts:72`, and inbound processing drops the message as `"echo"` when this matches at `extensions/imessage/src/monitor/inbound-processing.ts:397`. A focused smoke confirmed an outbound `"OK"` at 12:00 still makes `cache.has(... { text: "OK", messageId: "user-rowid-999" })` return `true` at 15:00.
- Impact: A real user who sends the same text the agent sent earlier in the same iMessage conversation can be silently dropped for up to 12 hours. This regresses the existing text-only guard’s stated intent to keep text fallback short to avoid suppressing legitimate repeated user text.
- Suggested fix: Keep long retention only for durable message-id/guid matches, or persist separate TTL/type metadata so text-only fallback remains short. If a lookup includes a nonmatching inbound message id, do not fall through to 12-hour text equality unless the self-chat `skipIdShortCircuit` case explicitly asks for that behavior.
- Confidence: high

### Medium: Catchup skips self-chat messages before the live self-chat gate can process them

- Kind: bug
- File: `extensions/imessage/src/monitor/catchup.ts`
- Line: 372
- Evidence: Catchup advances the cursor and skips every `row.isFromMe` before invoking dispatch. The bridge sets that flag directly from `payload.is_from_me` at `extensions/imessage/src/monitor/catchup-bridge.ts:186`. The live iMessage gate is more nuanced: `resolveIMessageInboundDecision` dispatches real self-chat user messages when `sender`, `chat_identifier`, and `destination_caller_id` identify the self-chat, while still dropping normal outbound echoes at `extensions/imessage/src/monitor/inbound-processing.ts:279`; the existing regression test asserts that behavior at `extensions/imessage/src/monitor/self-chat-dedupe.test.ts:346`. A focused smoke of `performIMessageCatchup` with an `isFromMe` row produced `dispatches=0`, `skippedFromMe=1`, and advanced the cursor past the row.
- Impact: Users who send messages to their own iMessage thread while the gateway is down lose those messages on catchup. Because the cursor advances, the skipped self-chat row is not retried.
- Suggested fix: Do not make the generic replay loop decide `is_from_me`; feed these rows through the same live handler and let `resolveIMessageInboundDecision` classify self-chat, outbound echo, and normal from-me rows. If summary metrics still need `skippedFromMe`, return a classified dispatch result from the bridge instead of duplicating self-chat policy in catchup.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Both findings were reproduced with focused local smokes against the current `main` source: the echo cache returns a text-only match after 3 hours, and catchup skips an `isFromMe` row without dispatching.
- Is this the best way to solve the issue? yes for direction. Both fixes should preserve the existing owner boundary: keep the iMessage replay feature local, but reuse the existing live inbound decision for `is_from_me` and keep long echo retention tied to stable ids rather than broad text equality.
