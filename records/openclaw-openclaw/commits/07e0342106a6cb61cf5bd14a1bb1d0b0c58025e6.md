# Commit 07e0342

## Summary

Found one supply-chain/security regression in the new trust calculation. The issue is still present on current `main`.

## Findings

### Medium: External catalog packages can receive official-package scanner trust during npm updates

- Kind: security
- File: `src/plugins/update.ts`
- Line: 443
- Evidence: `isTrustedSourceLinkedOfficialNpmUpdate` accepts any entry returned by `getOfficialExternalPluginCatalogEntry` and only compares npm package names before forwarding `trustedSourceLinkedOfficialInstall` into `installPluginFromNpmSpec` at `src/plugins/update.ts:1078`. That trust flag suppresses critical built-in scan blocking in `src/plugins/install-security-scan.runtime.ts:608`. But the catalog helper includes entries whose JSON metadata is explicitly `"source": "external"`, for example WeCom and Yuanbao in `scripts/lib/official-external-channel-catalog.json:6` and `scripts/lib/official-external-channel-catalog.json:34`.
- Impact: A tracked external catalog npm install can be updated with an explicit npm package spec, which docs allow for moving a pinned plugin back to the default release line. For such a package-name-matched external entry, this commit can pass `trustedSourceLinkedOfficialInstall: true`, allowing critical scanner findings without `--dangerously-force-unsafe-install` and logging it as an official OpenClaw package. That weakens the intended install/update safety boundary for third-party catalog packages.
- Suggested fix: Require `entry.source === "official"` before granting `trustedSourceLinkedOfficialInstall`, and add a regression test for a `source: "external"` catalog entry with a matching npm package. Keep third-party catalog trust tied to exact integrity/verification metadata, not the official-package bypass.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. Use a tracked npm install record for an external catalog package such as `@wecom/wecom-openclaw-plugin`, then run `openclaw plugins update @wecom/wecom-openclaw-plugin`; selection resolves the package back to the tracked plugin, `effectiveSpec` matches the catalog package name, and the updater passes the official trust flag.
- Is this the best way to solve the issue? Yes. The narrow fix is to gate the bypass on catalog source/verification, preserving the intended official OpenClaw false-positive path without widening it to external packages.
