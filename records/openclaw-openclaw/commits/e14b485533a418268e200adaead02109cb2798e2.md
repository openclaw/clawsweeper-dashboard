# Commit e14b485

## Summary

Found one actionable security hardening issue in the new agentic Mantis workflow: the autonomous Codex step receives a write-scoped GitHub token even though the prompt explicitly forbids GitHub mutation and the workflow already uses a separate GitHub App token for publishing evidence.

## Findings

### Medium: Agentic proof step receives unnecessary write-scoped `GH_TOKEN`

- Kind: security
- File: `.github/workflows/mantis-telegram-desktop-proof.yml`
- Line: 338
- Evidence: the workflow grants `contents: write`, `issues: write`, and `pull-requests: write` globally at lines 39-42, then passes `GH_TOKEN: ${{ github.token }}` into the `openai/codex-action` step at line 338. That same step runs with `sandbox: danger-full-access` and `codex-args: '["--full-auto"]'` at lines 351-353, while the prompt says “Do not post GitHub comments or reviews” and “Do not commit, push, label, merge, or edit PR metadata” in `.github/codex/prompts/mantis-telegram-desktop-proof.md:12`.
- Impact: prompt text is not an enforcement boundary. If the agent is misled by PR content, maintainer instructions, or a model/tooling error, it can use `gh` or GitHub API calls to push contents, edit PRs, or post comments with the workflow token. The intended publication path already uses `actions/create-github-app-token` at `.github/workflows/mantis-telegram-desktop-proof.yml:381` and `scripts/mantis/publish-pr-evidence.mjs`, so the agent does not need write credentials to produce the manifest.
- Suggested fix: split permissions by job and keep the Codex agent token read-only, or remove `GH_TOKEN` from the agent step entirely if unauthenticated/local `git diff` inspection is enough. Keep write permissions only for the resolver reaction job if needed and for the publishing step via the Mantis GitHub App token.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? No live reproduction; this is a secret-bearing workflow. The static path is clear: triggering the workflow reaches the Codex action with a write-scoped `GH_TOKEN` in its environment.
- Is this the best way to solve the issue? Yes. Reducing job/token permissions preserves the workflow behavior while making the prompt’s “do not mutate GitHub” rule enforceable by credentials.
