# Commit 5b063c2

## Summary

This change moves legacy config detection out of the normal read path, but two downstream paths still depend on normal snapshots carrying either full runtime semantics or legacy issue classification. The result is one data-loss-prone startup recovery path and one CLI runtime-default regression.

## Findings

### High: Gateway startup can overwrite legacy configs from last-known-good instead of sending them to doctor

- Kind: data_loss
- File: `src/config/io.ts`
- Line: 1771
- Evidence: invalid snapshots now always set `legacyIssues: []`, even when the validation issue is a known legacy key such as `memorySearch`. `shouldAttemptLastKnownGoodRecovery()` only skips plugin-local invalidity and otherwise returns true for any invalid snapshot. Gateway startup then calls `recoverConfigFromLastKnownGood()` for invalid snapshots. A focused smoke reproduced this: after promoting a valid config, replacing the config with `{ memorySearch: ... }` produced `valid=false`, `issues=["memorySearch"]`, `legacyIssues=[]`, and `recoverConfigFromLastKnownGood()` restored the old config, removing `memorySearch`.
- Impact: upgrading with a legacy-but-repairable config can cause gateway startup to restore a stale last-known-good file instead of preserving the current file and instructing the user to run `openclaw doctor --fix`. This can discard current config changes; a clobbered backup exists, but the active config and running gateway move back to stale settings.
- Suggested fix: keep enough legacy classification on normal snapshots for recovery policy, or have recovery policy re-run legacy detection before allowing whole-file restore. Legacy repairable config should not be treated like arbitrary corruption.
- Confidence: high

### Medium: CLI bootstrap pins snapshot-mode config and drops load-time runtime defaults

- Kind: regression
- File: `src/cli/program/config-guard.ts`
- Line: 97
- Evidence: `ensureConfigReady()` now seeds the process runtime snapshot with `snapshot.runtimeConfig`. That value comes from `readConfigFileSnapshot()`, which materializes with `"snapshot"` mode. Snapshot mode explicitly excludes compaction and context-pruning defaults, while load mode includes them. A focused smoke with an Anthropic profile showed direct `getRuntimeConfig()` returned `compaction.mode=safeguard`, `contextPruning.mode=cache-ttl`, and Anthropic cache retention, but after `ensureConfigReady(["status"])`, `getRuntimeConfig()` returned all three as `undefined`.
- Impact: commands that run after normal CLI bootstrap can see a weaker runtime config than the previous `getRuntimeConfig()` load path. For agent/message surfaces, this can disable default compaction safeguard and Anthropic context pruning/cache settings for the process.
- Suggested fix: pin a load-equivalent runtime config, not the snapshot-mode config, or avoid setting the global runtime snapshot from `ensureConfigReady()` unless it has passed through the same load-time finalization/default path as `loadConfig()`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. I used focused temporary-home smoke checks: one compared direct `getRuntimeConfig()` with `ensureConfigReady()`-pinned `getRuntimeConfig()`, and one promoted a last-known-good config then verified a legacy `memorySearch` config was auto-restored away because `legacyIssues` was empty.
- Is this the best way to solve the issue? no. The owner-boundary direction is right, but normal read snapshots still feed startup recovery and runtime snapshot caching. Those consumers need either legacy classification or load-equivalent runtime materialization.
