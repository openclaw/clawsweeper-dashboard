# Commit 9dedc4d

## Summary

Found one auth-routing regression in the new OpenAI PI profile ordering path. The Codex-first case is handled, but API-key-first configs can now rotate to a Codex OAuth profile while still running the OpenAI API transport.

## Findings

### Medium: OpenAI PI auth rotation can try Codex OAuth against the OpenAI API transport

- Kind: regression
- File: `src/agents/pi-embedded-runner/run.ts`
- Line: 750
- Evidence: `resolveSelectedOpenAIPiRuntimeProvider()` only switches `provider` to `openai-codex` before the run when the run already has a Codex profile or `auth.order.openai` starts with one (`src/agents/pi-embedded-runner/run.ts:560`, `src/agents/pi-embedded-runner/run.ts:593`). But the later `profileOrder` now always expands PI OpenAI auth providers through `listOpenAIAuthProfileProvidersForAgentRuntime()` (`src/agents/pi-embedded-runner/run.ts:750`), and the API-key-first case intentionally returns `["openai", "openai-codex"]` while leaving the selected runtime provider as `openai` (`src/agents/openai-codex-routing.ts:140`, `src/agents/openai-codex-routing.test.ts:87`). A focused static check produced: `selected: "openai"`, `authProviders: ["openai","openai-codex"]`, `profileOrder: ["openai:backup","openai-codex:work"]`.
- Impact: With `auth.order.openai: ["openai:backup", "openai-codex:work"]`, an expired/invalid/rate-limited OpenAI API-key profile can rotate to `openai-codex:work`, but the run keeps `provider === "openai"` and the OpenAI runtime model. That sends a ChatGPT/Codex OAuth token down the OpenAI API path instead of switching to the Codex runtime, so the documented backup profile is likely to fail instead of recovering the run.
- Suggested fix: Either do not append `openai-codex` candidates while the selected PI runtime provider remains `openai`, or make auth-profile advancement re-resolve/switch the runtime provider and model when the active candidate crosses from `openai:*` to `openai-codex:*`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Configure OpenAI PI with `auth.order.openai` as `["openai:backup", "openai-codex:work"]`, make `openai:backup` unusable, and the runner’s profile rotation reaches `openai-codex:work` without updating the already-selected `openai` runtime provider/model.
- Is this the best way to solve the issue? yes. The runtime provider has to remain aligned with the credential namespace; otherwise auth rotation mixes incompatible transport and token types.
