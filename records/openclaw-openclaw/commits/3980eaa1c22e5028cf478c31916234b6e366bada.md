## Summary

Found one low-severity reliability issue in the newly added one-off component wait helper. The targeted Discord tests pass after installing dependencies.

## Findings

### Low: one-off interaction replies can lose the component wait behind the new background GET lane

- Kind: reliability
- File: `extensions/discord/src/internal/interactions.ts`
- Line: 229
- Evidence: `replyAndWaitForComponent()` posts the initial interaction reply, then does `fetchReply()` before registering `waitForMessageComponent()`. That fetch is a GET, so `RequestClient` classifies it as `background`, and this same commit lets stale background requests be dropped after the lane timeout. A user click that arrives before the fetch returns also reaches `dispatchInteraction()` before the one-off map entry exists, so it is not resolved by the new one-off path.
- Impact: a command using the new Carbon-style helper can successfully send the Discord message but then fail to wait for its component, especially under REST backlog or a fast user click. The user may see an unhandled component interaction and the command may later time out.
- Suggested fix: make initial interaction replies request Discord’s callback response (`with_response=true`) and build the `Message` from that response before registering the one-off wait. If retaining the fetch fallback, avoid scheduling that fetch as droppable background work for this helper.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes: use `replyAndWaitForComponent()` with a mocked REST `post` that resolves and a delayed `get`; dispatch the matching component interaction before the delayed `fetchReply()` resolves. The one-off wait has not been registered yet, so the component is not consumed by `resolveOneOffComponent()`.
- Is this the best way to solve the issue? yes: using the interaction callback response follows the upstream Carbon shape and removes the extra droppable GET from the critical wait setup path.
