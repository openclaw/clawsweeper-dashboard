# Commit 65d7232

## Summary

The commit adds recursive inline-eval detection for shell positional carriers, but it still misses an allowlisted unquoted `$*` carrier that the allowlist accepts as direct positional argv. In strict inline-eval mode, that leaves a command-executing `find -exec` form approval-free when the carried executable is already allowlisted.

## Findings

### High: Unquoted `$*` positional carriers still bypass strict inline-eval approval

- Kind: security
- File: `src/infra/command-analysis/risks.ts`
- Line: 141
- Evidence: `resolveShellPositionalCarrierPlan` accepts `$*` in the carrier regex but then returns `null` when the normalized token is `star`. The allowlist side still treats the same direct carrier pattern as valid at `src/infra/exec-approvals-allowlist.ts:1019`, resolves the carried executable path, and can satisfy the allowlist at `src/infra/exec-approvals-allowlist.ts:1143`. A focused probe on current `main` showed `sh -c '$0 $*' find . -exec id {} ';'` produced `analysisOk: true`, `allowlistSatisfied: true`, and `inlineFlag: null`, while the equivalent `$@` form produced `inlineFlag: "-exec"`.
- Impact: With `strictInlineEval` enabled, gateway/node approval enforcement only asks explicitly when `detectPolicyInlineEval` returns a hit. A user with an allowlisted `find` path can carry `find -exec ...` through `sh -c '$0 $*' ...` without the strict inline-eval approval boundary firing.
- Suggested fix: Align risk detection with the allowlist carrier contract. Either unwrap unquoted `$*` as an all-args carrier, or stop `isDirectShellPositionalCarrierInvocation` from accepting `$*` carriers for allowlist persistence/matching. Keep quoted `"$*"` separate, since it collapses args into one word.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The focused TS probe imported `evaluateShellAllowlist`, `detectPolicyInlineEval`, and `detectInlineEvalArgv`; with a temp allowlisted `find`, `$@` was flagged and unquoted `$*` was not. A shell semantics probe also showed unquoted `$*` passes `.` `-exec` `id` `{}` `;` as separate argv entries, while quoted `"$*"` passes one combined arg.
- Is this the best way to solve the issue? yes. The bug is a contract mismatch between the allowlist recognizer and the risk detector; making one side stop accepting `$*`, or making the detector conservatively inspect unquoted `$*`, closes the approval gap at the shared boundary.
