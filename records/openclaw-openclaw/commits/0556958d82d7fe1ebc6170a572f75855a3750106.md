# Commit 0556958d82

## Summary

Found one likely macOS launch regression in the switch from SwiftUI `Settings` to a named `Window`.

## Findings

### Medium: Settings window can open automatically on app launch

- Kind: regression
- File: `apps/macos/Sources/OpenClaw/MenuBar.swift`
- Line: 83
- Evidence: The commit replaces the special `Settings { ... }` scene with `Window("OpenClaw Settings", id: "settings")`, but does not add `.defaultLaunchBehavior(.suppressed)`. OpenClaw is an `LSUIElement` menu-bar app, and this settings window is meant to be opened explicitly through `openWindow`. Apple documents `.defaultLaunchBehavior(.suppressed)` as the way to prevent a scene from presenting itself when there are no restored scenes, and SwiftUI menu-bar app reports match this exact `Window`/`WindowGroup` auto-open behavior on launch. ([developer.apple.com](https://developer.apple.com/documentation/swiftui/scenelaunchbehavior?utm_source=openai))
- Impact: Users may see the Settings window every time the macOS app launches, instead of OpenClaw staying in the menu bar until Settings is requested. That is especially disruptive for launch-at-login.
- Suggested fix: Add `.defaultLaunchBehavior(.suppressed)` to the settings `Window` scene. The package minimum is macOS 15, so the modifier is available.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? No live macOS proof in this Linux runner. Targeted repro: build/run the macOS app on macOS 15+ with no restored windows and observe whether the Settings window appears immediately.
- Is this the best way to solve the issue? Yes, if the app should keep this SwiftUI `Window`; suppressing default launch behavior is the native fix for a manually opened secondary scene.
