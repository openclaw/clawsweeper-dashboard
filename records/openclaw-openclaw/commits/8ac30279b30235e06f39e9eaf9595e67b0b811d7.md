# Commit 8ac3027

## Summary

Found one reliability regression in the new plural tool-call adjacency scan. The leak-stripping behavior itself works for the targeted case, but the implementation adds superlinear synchronous work for ordinary prose containing many unmatched `<function_calls>` / `<tool_calls>` mentions.

## Findings

### Medium: Repeated literal plural tool-call tags can make delivery sanitization quadratic

- Kind: reliability
- File: `src/shared/text/assistant-visible-text.ts`
- Line: 382
- Evidence: The new code computes `matchingCloseStart` with `findMatchingToolCallCloseIndex(...)` for every plural wrapper before checking whether the new delivery option is enabled and before checking whether the tag payload is actually strip-worthy XML/JSON. `findMatchingToolCallCloseIndex` scans forward to the next matching close tag, so repeated literal/unmatched `<function_calls>` mentions rescan the remaining text each time.
- Impact: This runs in user-visible delivery sanitization. `src/agents/pi-embedded-utils.ts:34` routes assistant extraction through `sanitizeAssistantVisibleText`, and `src/agents/pi-embedded-subscribe.handlers.messages.ts:513` calls it on partial assistant messages during streaming, then again at message end at `src/agents/pi-embedded-subscribe.handlers.messages.ts:675`. A user can prompt a model to output many literal `<function_calls>` lines; in a local probe, 1,000 lines took 319.9ms and 3,000 lines took 2738.7ms for `sanitizeAssistantVisibleText`, while preserving the text. Larger or repeated streaming partials can block the event loop for seconds.
- Suggested fix: Do the cheap payload/option checks before calling `findMatchingToolCallCloseIndex`, and avoid the forward close scan entirely for prose-like plural tags. A single-pass adjacency check or a scan only after an XML/JSON payload candidate would preserve the leak fix without penalizing ordinary literal mentions.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Run a local `node --import tsx` probe that calls `sanitizeAssistantVisibleText` on `Array.from({ length: 3000 }, (_, i) => \`<function_calls> literal ${i}\`).join("\n")`; the current code takes about 2.7s on this checkout.
- Is this the best way to solve the issue? yes. The failure comes from an avoidable forward scan on non-payload prose; moving/gating that scan is narrower than changing the sanitizer contract.
