## Summary

Found one low-severity test reliability regression in the new Slack artifact-preservation test. The runtime behavior itself looks aligned with the commit intent.

## Findings

### Low: New Convex failure test depends on host QA credential environment

- Kind: reliability
- File: `extensions/qa-lab/src/live-transports/slack/slack-live.runtime.test.ts`
- Line: 112
- Evidence: The new test calls `runSlackQaLive({ credentialSource: "convex", credentialRole: "ci" })` and asserts the failure details contain `Missing OPENCLAW_QA_CONVEX_SITE_URL`. But `runSlackQaLive` delegates to `acquireQaCredentialLease`, which reads `process.env` directly. If `OPENCLAW_QA_CONVEX_SITE_URL` is present, the first failure becomes `Missing OPENCLAW_QA_CONVEX_SECRET_CI for CI credential access.` instead.
- Impact: Maintainer machines or CI jobs that already define Convex QA broker env can fail this unit test or, with fuller credentials, attempt real broker acquisition from a unit test. This makes the new artifact-preservation coverage environment-dependent.
- Suggested fix: Isolate the test from host env before calling `runSlackQaLive`, or add an injectable credential/env seam and force a deterministic acquisition failure in the test.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: `OPENCLAW_QA_CONVEX_SITE_URL=https://example.invalid pnpm test extensions/qa-lab/src/live-transports/slack/slack-live.runtime.test.ts` fails at line 112 with the missing-secret message instead of the expected missing-site message.
- Is this the best way to solve the issue? yes: the test is intended to prove artifact writing on acquisition failure, so it should control the acquisition failure deterministically rather than depending on ambient `OPENCLAW_QA_CONVEX_*` state.
