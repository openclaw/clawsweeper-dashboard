# Commit 36df0d93b9

## Summary

Found two actionable compatibility/reliability issues in the landed LAN pairing repair. The main iOS path now accepts private-LAN bootstrap setup codes, but the Swift session persistence gate still treats those `ws://` endpoints as untrusted, so successful bootstrap handoff tokens are dropped.

## Findings

### Medium: Private-LAN iOS bootstrap pairing drops handed-off device tokens

- Kind: reliability
- File: `apps/shared/OpenClawKit/Sources/OpenClawKit/GatewayChannel.swift`
- Line: 549
- Evidence: This commit allows insecure setup/manual hosts when `LoopbackHost.isLocalNetworkHost(...)` is true, including `ws://openclaw.local` and `ws://192.168...` in `DeepLinks.swift:119` and `DeepLinks.swift:146`. But `GatewayChannel.shouldPersistBootstrapHandoffTokens()` still returns true only for `wss://` or loopback hosts, and `persistIssuedDeviceToken()` returns without storing the server-issued token when that guard fails.
- Impact: A private-LAN iOS setup-code connection can complete once with the single-use bootstrap token, then fail after disconnect/app restart because the bootstrap token is cleared and no handed-off node/operator device token was persisted.
- Suggested fix: Extend `shouldPersistBootstrapHandoffTokens()` to trust the same local/private host policy as setup-code parsing, e.g. `LoopbackHost.isLocalNetworkHost(host)`, and add a `ws://openclaw.local` or `ws://192.168.1.20` regression test that asserts bootstrap `deviceToken` and `deviceTokens` are stored.
- Confidence: high

### Low: Setup-code `.local` allowance is not supported by Android parser

- Kind: compatibility
- File: `src/pairing/setup-code.ts`
- Line: 100
- Evidence: The shared setup-code resolver now accepts `.local` cleartext hosts, and docs say mobile `ws://` setup codes support private LAN and `.local` hosts. Current Android parsing still rejects every cleartext non-loopback host at `apps/android/app/src/main/java/ai/openclaw/app/ui/GatewayConfigResolver.kt:152`, with tests explicitly asserting `ws://gateway.local:18789` is rejected.
- Impact: `openclaw qr` or `/pair` can now issue a `.local` setup code that iOS accepts but Android rejects before auth, despite the user-facing “mobile pairing” docs.
- Suggested fix: Either align Android’s cleartext allowlist with the shared/iOS private-LAN `.local` policy, or narrow generator/docs language to the iOS-only support surface.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. Source path: generate or paste a bootstrap setup code for `ws://openclaw.local:18789`, parse it through `GatewayConnectDeepLink`, connect via `GatewayNodeSession`, receive bootstrap handoff tokens, and observe `shouldPersistBootstrapHandoffTokens()` returning false because the URL is `ws://` and non-loopback.
- Is this the best way to solve the issue? Yes for the first finding: the persistence trust boundary should match the newly accepted local/private bootstrap transport boundary. For Android, either runtime alignment or explicit platform scoping would be safe.
