# Commit 207bcd6

## Summary

Found one installer regression in the new rc-file prepend logic. It still exists on current `main`.

## Findings

### Medium: PATH prepending replaces symlinked shell rc files

- Kind: data_loss
- File: `scripts/install.sh`
- Line: 1380
- Evidence: `persist_shell_path_prepend` and `fix_npm_permissions` now build `${rc}.openclaw-tmp` and `mv "$tmp_rc" "$rc"` over `$HOME/.bashrc` / `$HOME/.zshrc`. That replaces the rc path itself. A focused smoke with `$HOME/.bashrc -> dotfiles/bashrc` showed `symlink=no`, the dotfiles target unchanged, and the new plain `$HOME/.bashrc` beginning with the OpenClaw PATH export.
- Impact: users who manage `.bashrc` or `.zshrc` as symlinks into a dotfiles repo lose that link during install. Future dotfiles updates no longer affect the real shell startup file, and the change is hard to notice because the generated copy still contains the old content.
- Suggested fix: update the resolved rc target when `$rc` is a symlink, or use a helper that prepends while preserving symlinks and file metadata. Add tests for symlinked `.bashrc` / `.zshrc` for both the Node path and npm prefix paths.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Create a temp home with `.bashrc` symlinked to another file, source `scripts/install.sh` with `OPENCLAW_INSTALL_SH_NO_RUN=1`, run `activate_supported_node_on_path` with a supported fake `node`, then check `[[ -L "$HOME/.bashrc" ]]`; it becomes false.
- Is this the best way to solve the issue? yes. The intended prepend behavior can be kept, but the write path needs to preserve symlinked rc files instead of moving a temp file over the symlink.
