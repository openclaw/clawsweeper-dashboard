## Summary

Found one low-severity release-operator regression: the workflows now require `full_release_validation_run_id`, but the release docs still show publish commands and input guidance that omit it.

## Findings

### Low: Release publish docs still give commands that fail the new required gate

- Kind: regression
- File: `docs/reference/RELEASING.md`
- Line: 626
- Evidence: `openclaw-release-publish.yml` now fails early when `publish_openclaw_npm=true` and `FULL_RELEASE_VALIDATION_RUN_ID` is empty (`.github/workflows/openclaw-release-publish.yml:109`). The beta/stable publish examples in `docs/reference/RELEASING.md:626`, `docs/reference/RELEASING.md:636`, and `docs/reference/RELEASING.md:646` still pass only `preflight_run_id` and `npm_dist_tag`. The workflow input list at `docs/reference/RELEASING.md:672` also omits `full_release_validation_run_id`, and the stable sequence at `docs/reference/RELEASING.md:722` still says to save and pass only `preflight_run_id`.
- Impact: Release operators following the documented commands will hit `publish_openclaw_npm=true requires full_release_validation_run_id` instead of starting the publish orchestration. This fails closed before publishing, so it is operational friction rather than an unsafe publish path.
- Suggested fix: Update the release docs examples, input list, and stable/beta sequence to tell operators to save the successful Full Release Validation run ID and pass `-f full_release_validation_run_id=<successful-full-release-validation-run-id>`. Consider extending the workflow-contract test to assert the docs include the new input.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Dispatching any documented `OpenClaw Release Publish` command with default `publish_openclaw_npm=true` omits `full_release_validation_run_id`, so the validate step exits at `.github/workflows/openclaw-release-publish.yml:109`.
- Is this the best way to solve the issue? yes. The workflow behavior is intentionally fail-closed; the fix is to update operator-facing docs to match the new contract.
