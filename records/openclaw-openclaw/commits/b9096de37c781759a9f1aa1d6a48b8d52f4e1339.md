# Commit b9096de

## Summary

Found two low-severity reliability regressions in the Parallels smoke test changes. No security or supply-chain issues found.

## Findings

### Low: New timeout-default test fails when Parallels timeout env vars are set

- Kind: reliability
- File: `test/scripts/parallels-smoke-model.test.ts`
- Line: 505
- Evidence: `runTsEval` inherits `process.env` at `test/scripts/parallels-smoke-model.test.ts:38`, while the new assertion expects hard-coded defaults from `resolveParallelsModelTimeoutSeconds`. That helper intentionally reads `OPENCLAW_PARALLELS_<PLATFORM>_MODEL_TIMEOUT_S` and `OPENCLAW_PARALLELS_MODEL_TIMEOUT_S` at `scripts/e2e/parallels/provider-auth.ts:80`. Reproduced with `OPENCLAW_PARALLELS_MODEL_TIMEOUT_S=1200 pnpm test test/scripts/parallels-smoke-model.test.ts -- -t "gives GPT-5.5 enough Parallels model time"`, which fails because the helper returns `1200` for all platforms instead of `{ linux: 600, macos: 900, windows: 900 }`.
- Impact: maintainer machines or CI lanes that export the same timeout overrides used by these smoke scripts will get an unrelated unit-test failure.
- Suggested fix: run this default assertion with a sanitized env, clearing `OPENCLAW_PARALLELS_MODEL_TIMEOUT_S` and all platform-specific `OPENCLAW_PARALLELS_*_MODEL_TIMEOUT_S` keys, or make the assertion explicitly test env override behavior separately.
- Confidence: high

### Low: Linux smoke now caps the whole agent run below the previous phase budget

- Kind: regression
- File: `scripts/e2e/parallels/linux-smoke.ts`
- Line: 721
- Evidence: the commit adds `--timeout ${resolveParallelsModelTimeoutSeconds("linux")}` to the Linux agent command, whose default remains `600` seconds, while the phase budget is now `1500` seconds at `scripts/e2e/parallels/linux-smoke.ts:347` and `:376`. `openclaw agent --timeout` is parsed as the overall agent run timeout in `src/agents/agent-command.ts:328`, then used to schedule the embedded run abort in `src/agents/pi-embedded-runner/run/attempt.ts:2400`. Before this commit, the Linux smoke did not pass `--timeout`, so the outer 900-second phase was the effective cap for non-provider-overridden agent runs.
- Impact: a slow Linux guest/model can now be aborted by OpenClaw after 600 seconds even though the smoke phase allows 1500 seconds. This especially affects non-OpenAI provider lanes, because `providerTimeoutConfigJson` only writes OpenAI provider timeouts.
- Suggested fix: give the Linux `openclaw agent --timeout` an overall-run budget that includes startup/request overhead, or avoid passing it for providers that do not receive the matching provider timeout config.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes for the env-sensitive test failure, using the focused command above. The Linux budget issue is static code evidence from the current timeout plumbing; reproducing the live failure would require a slow Parallels guest/model run.
- Is this the best way to solve the issue? yes for the test isolation fix. For the Linux smoke budget, the safest direction is to separate model request timeout from whole-agent timeout so the outer command does not undercut the intended smoke budget.
