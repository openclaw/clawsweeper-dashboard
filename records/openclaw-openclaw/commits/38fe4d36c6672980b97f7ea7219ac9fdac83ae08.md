# Commit 38fe4d3

## Summary

Found one provider contract regression: the new generic `suppressResponse` realtime tool-result option is wired through the Gateway and implemented for OpenAI, but Google Live realtime ignores it and will still schedule a model response.

## Findings

### Low: Google realtime ignores `suppressResponse`

- Kind: bug
- File: `extensions/google/realtime-voice-provider.ts`
- Line: 601
- Evidence: `src/talk/provider-types.ts:52` adds `suppressResponse`, `src/gateway/talk-realtime-relay.ts:393` forwards the option to provider bridges, and `docs/gateway/protocol.md:380` documents it as satisfying a provider call without starting another assistant response. Discord now relies on that for late forced consults at `extensions/discord/src/voice/realtime.ts:854`. The Google realtime bridge accepts the shared `RealtimeVoiceToolResultOptions` but only checks `willContinue`; for consult tools it unconditionally sets `functionResponse.scheduling = "WHEN_IDLE"` at `extensions/google/realtime-voice-provider.ts:601`. The installed `@google/genai` 2.0.1 types define `SILENT` as “Only add the result to the conversation context, do not interrupt or trigger generation,” while `WHEN_IDLE` prompts output generation.
- Impact: A Discord or Gateway realtime session using the Google Live provider can still speak a stale “already delivered” tool result after OpenClaw has already delivered the forced answer, so the commit’s stale-consult suppression only works for OpenAI-backed realtime sessions.
- Suggested fix: In the Google realtime bridge, when `options?.suppressResponse === true` for the agent consult tool, set `functionResponse.scheduling = "SILENT"` instead of `WHEN_IDLE`, clear the pending call like a final result, and add a Google provider test mirroring the OpenAI suppressed-result test.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. Create a Google `gateway-relay` realtime session, receive an `openclaw_agent_consult` tool call, then submit `options: { suppressResponse: true }`; current code sends a Google function response scheduled as `WHEN_IDLE` instead of `SILENT`.
- Is this the best way to solve the issue? Yes. The new option is provider-level behavior, and Google’s SDK already exposes the silent scheduling mode needed to implement the shared contract.
