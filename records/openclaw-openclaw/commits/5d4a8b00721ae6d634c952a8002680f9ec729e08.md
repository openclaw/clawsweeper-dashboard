# Commit 5d4a8b0

## Summary

Found one scoped regression in the new trajectory-specific timeout surface: it only applies to the PI harness step name, so Codex trajectory flush cleanup still ignores `OPENCLAW_TRAJECTORY_FLUSH_TIMEOUT_MS`.

## Findings

### Medium: Codex trajectory flush ignores the new trajectory timeout variable

- Kind: bug
- File: `src/agents/run-cleanup-timeout.ts`
- Line: 42
- Evidence: `resolveAgentCleanupStepTimeoutMs` only reads `OPENCLAW_TRAJECTORY_FLUSH_TIMEOUT_MS` when `params.step === "pi-trajectory-flush"`. The Codex app-server trajectory flush call sites use `step: "codex-trajectory-flush-startup-failure"` and `step: "codex-trajectory-flush"` in `extensions/codex/src/app-server/run-attempt.ts:1578` and `extensions/codex/src/app-server/run-attempt.ts:1840`, so they fall through to the general env/default path. A focused resolver proof returned `{"pi":25000,"codex":10000,"startup":10000,"defaultMs":10000}` with only `OPENCLAW_TRAJECTORY_FLUSH_TIMEOUT_MS=25000`.
- Impact: Operators following the new trajectory docs can still see Codex trajectory cleanup time out at the old 10s default on slow stores, even though the change advertises a trajectory-specific knob. The broad `OPENCLAW_AGENT_CLEANUP_TIMEOUT_MS` can work around it, but that changes all cleanup steps rather than only trajectory flushes.
- Suggested fix: Make the trajectory-specific timeout apply to all first-party trajectory flush cleanup steps, either by an explicit allowlist for `pi-trajectory-flush`, `codex-trajectory-flush`, and `codex-trajectory-flush-startup-failure`, or by passing an explicit trajectory timeout option from trajectory flush call sites.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Running the timeout resolver with `OPENCLAW_TRAJECTORY_FLUSH_TIMEOUT_MS=25000` returns `25000` for `pi-trajectory-flush` but `10000` for both Codex trajectory flush step names.
- Is this the best way to solve the issue? yes. The bug is in timeout classification, not trajectory writing itself; applying the trajectory timeout to all first-party trajectory flush cleanup steps preserves the new config contract without broadening unrelated cleanup behavior.
