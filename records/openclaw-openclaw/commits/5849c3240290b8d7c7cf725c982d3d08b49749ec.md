# Commit 5849c32

## Summary

Found one low-severity regression: the new non-interactive OpenAI provider path can still install/enable Codex even when the selected OpenAI model is explicitly pinned to the PI runtime. Startup planning preserves that override, but the install helper does not.

## Findings

### Low: Codex install ignores explicit PI runtime pins

- Kind: regression
- File: `src/commands/codex-runtime-plugin-install.ts`
- Line: 17
- Evidence: `selectedModelShouldEnsureCodexRuntimePlugin` only delegates to `modelSelectionShouldEnsureCodexPlugin`, which checks the model/provider and custom OpenAI base URL but does not consult `agents.defaults.models["openai/..."].agentRuntime`. The new non-interactive path calls this helper whenever provider setup returns an OpenAI default model in `src/commands/onboard-non-interactive/local/auth-choice.plugin-providers.ts:200`. A focused check with a config containing `agents.defaults.model.primary = "openai/gpt-5.5"` and `agents.defaults.models["openai/gpt-5.5"].agentRuntime.id = "pi"` returned `true` from `selectedModelShouldEnsureCodexRuntimePlugin`, while the added startup test explicitly expects Codex not to load for that same PI pin in `src/plugins/channel-plugin-ids.test.ts:1435`.
- Impact: users who deliberately pin an OpenAI model to PI can still get an automatic npm install and config enablement of the Codex plugin during non-interactive OpenAI onboarding. That contradicts the changelog/PR behavior claim that manual PI runtime overrides are preserved, adds an unexpected network/package install, and leaves an enabled plugin that startup later skips.
- Suggested fix: make the Codex install predicate share the runtime-policy check used by `collectConfiguredAgentHarnessRuntimes` in `src/agents/harness-runtimes.ts:58`, or otherwise skip install when `resolveModelRuntimePolicy` resolves the selected model to an explicit non-Codex runtime such as `pi`. Add a non-interactive onboarding test for the PI-pinned OpenAI model case.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes at the helper level: evaluating `selectedModelShouldEnsureCodexRuntimePlugin` with an OpenAI default model plus a PI model-runtime pin returns `true`, even though startup planning’s regression test says that same config should not require Codex.
- Is this the best way to solve the issue? Yes: the runtime-policy decision already exists; the install decision should reuse it so install, repair, and startup agree on when Codex is actually required.
