# Commit 706132c

## Summary

Found one low-severity reliability issue in the new CPU profile option: timed-out benchmark samples can still report success while producing no CPU profile.

## Findings

### Low: Timed-out startup samples silently miss the requested CPU profile

- Kind: reliability
- File: `scripts/bench-gateway-startup.ts`
- Line: 682
- Evidence: The commit adds `--cpu-prof`, `--cpu-prof-dir`, and `--cpu-prof-name` to the spawned Gateway process, but the existing stop path sends `SIGTERM` and escalates to `SIGKILL` after 2 seconds at `scripts/bench-gateway-startup.ts:520`. Node’s own help says `--cpu-prof` writes the profile “before exit”; a focused benchmark smoke with a nonresponsive entry produced `signal: "SIGKILL"` and only `result.json`, with no `.cpuprofile`, while the script exited 0.
- Impact: The option is most useful when startup is slow or wedged, but that timeout path can leave users with no profile and no failure or warning, despite the changelog saying the benchmark writes per-run V8 CPU profiles.
- Suggested fix: When `cpuProfDir` is set, either extend the graceful shutdown window enough for profile flush, fail/warn if the expected profile is missing, or record profile path/status in the sample output so missing profiles are visible.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes, for the timeout path: run the benchmark with `--cpu-prof-dir`, a short `--timeout-ms`, and an entry that does not serve readiness and ignores `SIGTERM`; the sample exits via `SIGKILL` and no profile is written.
- Is this the best way to solve the issue? unclear; the safest minimal fix is to detect and report missing profiles, while a longer CPU-profile shutdown grace would make the feature more useful for slow-but-responsive shutdowns.
