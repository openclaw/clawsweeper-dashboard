Found one current-main regression from the reviewed commit. Several other risky areas from the commit have follow-up fixes on `main`; this compaction regression still remains.

## Findings

### Medium: Built-in compaction can request more output tokens than the model supports

- Kind: reliability
- File: `src/agents/compaction.ts`
- Line: 328
- Evidence: The reviewed commit removed the `reserveTokens` clamp before calling `generateSummary`; current `main` still passes `params.reserveTokens` directly into the locked Pi summarizer. The locked dependency `@mariozechner/pi-coding-agent@0.73.0` computes `maxTokens = Math.floor(0.8 * reserveTokens)` in `dist/core/compaction/compaction.js`, and `@mariozechner/pi-ai@0.73.0` passes explicit `options.maxTokens` through instead of clamping to `model.maxTokens`. OpenClaw config allows unbounded nonnegative `agents.defaults.compaction.reserveTokens` / `reserveTokensFloor` at `src/config/zod-schema.agent-defaults.ts:176`, and `src/agents/pi-settings.ts:100` preserves high configured values for large-context models.
- Impact: A user with a large-context model and high compaction reserve, for example `reserveTokens: 300_000` with a model output cap of `128_000`, will make built-in compaction ask the provider for `240_000` output tokens. Providers that reject over-cap `max_tokens` will fail summarization during context-overflow recovery, so long-running sessions can get stuck at the point compaction is supposed to recover them.
- Suggested fix: Restore the removed clamp near `summarizeChunks`, or clamp the output budget immediately before calling Pi so `Math.floor(0.8 * reserveTokens) <= model.maxTokens` for every chunk and merge summary. Re-add the deleted reserve-token clamp test.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Configure built-in compaction with a high `agents.defaults.compaction.reserveTokens` or `reserveTokensFloor`, use a model whose `maxTokens` is lower than `0.8 * reserveTokens`, and trigger `summarizeInStages()` on an overflowing transcript. The deleted test `src/agents/compaction.reserve-tokens-clamping.test.ts` already encoded this exact path with `reserveTokens: 300_000` and `maxTokens: 128_000`.
- Is this the best way to solve the issue? yes. The boundary that knows both OpenClaw’s configured reserve and the resolved model output cap is the OpenClaw wrapper around Pi summarization; clamping there avoids relying on every provider implementation to reject or normalize invalid output budgets consistently.
