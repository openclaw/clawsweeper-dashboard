# Commit 543518b

## Summary

Found one low-severity QA contract weakness. The production Matrix approval path was not changed, but the live QA scenario can now pass without proving the second approval was actually delivered to an approver DM.

## Findings

### Low: target=both QA accepts any non-origin room as the DM approval

- Kind: test_gap
- File: `extensions/qa-matrix/src/runners/contract/scenario-runtime-approval.ts`
- Line: 202
- Evidence: `runApprovalChannelTargetBothScenario` still computes candidate DM rooms from topology and passes them as `roomIds`, but the new fallback in `waitForObservedApprovalEvent` returns true for any observed SUT `m.room.message` with the expected approval kind/id after only excluding `context.roomId`. It does not require `event.roomId` to be one of the candidate DM rooms, nor verify that the runtime room is a DM with the configured approver. The updated unit fixture demonstrates this by accepting `!driver-runtime-dm:matrix-qa.test`, which is not in the candidate topology.
- Impact: The scenario title/contract says `target=both` delivers channel and DM metadata once, but a Matrix native delivery regression that sends the approver copy to the wrong non-origin room could still satisfy this QA scenario as long as the driver observes the event and the approval id matches.
- Suggested fix: Keep accepting runtime-created rooms, but verify the fallback room is an approver DM before returning it. For example, resolve the approver DM room through Matrix direct-room metadata/membership, or have the runtime delivery path expose/record the prepared approver DM room id for the QA scenario to match.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes for the false-positive path: the helper’s fallback only checks sender/type/kind/id and `excludedRoomIds`, so an event with the same approval id in any other observed room would be accepted. I did not run a live homeserver misrouting scenario.
- Is this the best way to solve the issue? unclear. The safest direction is to preserve support for runtime-created DM room ids while adding a real DM/approver-room proof instead of reverting to static topology-only matching.
