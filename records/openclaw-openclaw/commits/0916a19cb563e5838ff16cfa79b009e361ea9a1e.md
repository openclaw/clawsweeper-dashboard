## Findings

### Medium: Package patch guard is not wired into normal PR CI

- Kind: supply_chain
- File: `.github/workflows/ci.yml`
- Line: 1395
- Evidence: The commit adds `deps:patches:check` to local `pnpm check` and `pnpm check:changed`, but current PR CI expands check shards manually in `.github/workflows/ci.yml`. The `preflight-guards` case runs conflict markers, tool display, host env policy, and duplicate coverage only; the `dependencies` case runs deadcode checks only. A repository search found `deps:patches:check` only in `package.json`, `scripts/check.mjs`, and `scripts/check-changed.mjs`, plus release/test-performance workflows, not the main PR CI shard.
- Impact: A PR can add `pnpm-workspace.yaml` `patchedDependencies` and a tracked `patches/*.patch` file while passing the normal CI check shards, so the commit does not actually “block new package patches” before merge.
- Suggested fix: Add `pnpm deps:patches:check` to the main CI check shard, preferably `preflight-guards` or `dependencies`, and add/update a workflow test that asserts the PR CI shard invokes the guard.
- Confidence: high
