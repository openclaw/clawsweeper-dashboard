# Commit b2fb2d9

## Summary

Found one compatibility/reliability regression risk in the new manifest catalog normalization path. Current bundled tests still pass, but the public SDK helper can now emit different provider model ids depending on ambient plugin metadata, not just the manifest catalog it was asked to convert.

## Findings

### Medium: Manifest provider config output now depends on ambient plugin normalization policies

- Kind: compatibility
- File: `src/plugin-sdk/provider-catalog-shared.ts`
- Line: 129
- Evidence: `buildManifestCatalogModel` now calls `normalizeConfiguredProviderCatalogModelId(providerId, model.id)`. That helper calls `normalizeStaticProviderModelId` without disabling manifest normalization, and `normalizeStaticProviderModelId` consults `normalizeProviderModelIdWithManifest`, which loads the current plugin metadata snapshot or scans plugin metadata when no explicit plugin list is supplied. A focused smoke check showed the same catalog row emitting `legacy` with no current snapshot and `current` after setting an ambient snapshot with `modelIdNormalization.providers.example.aliases.legacy = current`.
- Impact: `buildManifestModelProviderConfig` is a public plugin SDK helper for converting a plugin’s own manifest catalog into runtime provider config. After this commit, a plugin’s emitted runtime model ids can be rewritten by the active plugin metadata environment, including another plugin’s normalization policy for the same provider id, rather than being a pure conversion of the supplied `catalog`. That can make third-party provider configs nondeterministic and can send the wrong model id to the provider if manifest normalization aliases are meant only for user/config ref compatibility.
- Suggested fix: keep this conversion input-local. For the intended Gemini fix, normalize only the built-in Google retired ids here, or call a helper that disables manifest-backed normalization and then applies the nested `google/` Gemini alias rule.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. A one-off `pnpm exec tsx -e ...` check called `buildManifestModelProviderConfig` on `{ id: "legacy" }`; output was `legacy` before setting current plugin metadata and `current` after setting an ambient model-id alias policy for provider `example`.
- Is this the best way to solve the issue? The fix direction is clear: avoid ambient manifest policy lookup inside manifest-to-runtime conversion. The exact helper shape is maintainer-owned because this SDK area already has built-in Google normalization and plugin manifest normalization seams.
