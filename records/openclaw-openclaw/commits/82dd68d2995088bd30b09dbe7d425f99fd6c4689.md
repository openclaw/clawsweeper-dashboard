# Commit 82dd68d

## Summary

Found one idempotency regression in the runtime alias postbuild path. Current `main` has a follow-up fix for the `runtime-model-auth` collision case, but the legacy compat aliases added by this commit still poison later alias scans.

## Findings

### Medium: Re-running runtime-postbuild deletes stable runtime aliases after legacy compat aliases exist

- Kind: reliability
- File: `scripts/runtime-postbuild.mjs`
- Line: 67
- Evidence: `writeLegacyRootRuntimeCompatAliases()` writes files such as `runtime-plugins.runtime-fLHuT7Vs.js` that match `ROOT_RUNTIME_ALIAS_PATTERN`. On a later `writeStableRootRuntimeAliases()` pass, those generated compat files are grouped as additional candidates for `runtime-plugins.runtime.js`; no unique candidate is selected, so line 105 removes the stable alias. A focused temp harness on current `main` showed: first pass created `runtime-plugins.runtime.js`, first pass created legacy aliases, second stable-alias pass left `runtime-plugins.runtime.js` missing.
- Impact: Any source-checkout path that re-runs runtime-postbuild without a fresh `dist` clean can remove stable runtime aliases that current rewritten chunks import, causing later lazy runtime imports to fail with missing module errors.
- Suggested fix: Exclude generated legacy compat wrappers from `writeStableRootRuntimeAliases()` and `rewriteRootRuntimeImportsToStableAliases()` candidate grouping, preferably by detecting the exact generated `export * from "./<stable-alias>";` wrapper content rather than only filename. Add an idempotency test that runs stable aliases, legacy compat aliases, then stable aliases again.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The temp harness created `runtime-plugins.runtime-NewHash.js`, ran `writeStableRootRuntimeAliases()`, ran `writeLegacyRootRuntimeCompatAliases()`, then ran `writeStableRootRuntimeAliases()` again; the final check showed `runtime-plugins.runtime.js` was deleted.
- Is this the best way to solve the issue? yes. The problem is candidate classification, not alias writing in general, so filtering generated compat wrappers preserves the commit’s ambiguity guard while making postbuild idempotent.
