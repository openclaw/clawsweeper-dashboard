## Summary

Found one actionable Windows test-isolation regression in the new PowerShell completion coverage. Current `main` has a follow-up commit that fixed the PowerShell profile-directory selection for `pwsh` vs Windows PowerShell, but the tests introduced here still use the live Windows `USERPROFILE`/platform state.

## Findings

### Medium: PowerShell completion tests can modify the real Windows profile

- Kind: data_loss
- File: `src/cli/completion-runtime.test.ts`
- Line: 92
- Evidence: The test creates temp `homeDir` and `stateDir`, then sets only `process.env.HOME` and `OPENCLAW_STATE_DIR` before calling `installCompletion("powershell", true, "openclaw")` at `src/cli/completion-runtime.test.ts:100`. On Windows, `resolveCompletionProfilePath("powershell")` uses `env.USERPROFILE || home` at `src/cli/completion-runtime.ts:180`, so a normal Windows test process with `USERPROFILE=C:\Users\<user>` writes `Documents\PowerShell\Microsoft.PowerShell_profile.ps1` under the real user profile instead of the temp directory. A read-only Node path probe with `HOME=C:\Temp\openclaw-completion-home-123` and `USERPROFILE=C:\Users\runneradmin` resolves to `C:\Users\runneradmin\Documents\PowerShell\Microsoft.PowerShell_profile.ps1`. Microsoft’s PowerShell docs confirm this profile file is the CurrentUserCurrentHost startup profile path used by PowerShell, so writing it changes user shell startup behavior. ([learn.microsoft.com](https://learn.microsoft.com/en-us/powershell/module/microsoft.powershell.core/about/about_profiles?source=post_page---------------------------&view=powershell-7.4&utm_source=openai))
- Impact: Running the focused test on Windows can create or rewrite a real PowerShell profile outside the test temp tree, leaving a stale `. '<temp cache>\openclaw.ps1'` line after the temp cache is deleted. The companion wizard test at `src/wizard/setup.completion.test.ts:104` also assumes the non-Windows PowerShell profile path after setting only `HOME`, so it is likely to fail on Windows rather than proving the intended behavior.
- Suggested fix: Save and restore `USERPROFILE` in these tests and set it to the temp home before calling production install/profile code. For the wizard note test, avoid a live-platform expectation by setting all relevant env inputs consistently, adding an injectable profile resolver, or branching the expected path through the same controlled resolver options used by the Windows path unit test.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Run `node scripts/run-vitest.mjs src/cli/completion-runtime.test.ts src/wizard/setup.completion.test.ts` on Windows with a normal `USERPROFILE`; `completion-runtime.test.ts` will resolve the profile under `USERPROFILE\Documents\PowerShell` instead of the test temp home.
- Is this the best way to solve the issue? yes. The least risky fix is test isolation: set and restore `USERPROFILE` alongside `HOME`, and keep platform-specific expected paths under controlled inputs rather than live process globals.
