# Commit 6350dd5a

## Summary

Found one changed-gate false negative around deleted release-metadata files.

## Findings

### Medium: Release-metadata deletions can route to the shortcut while its guard ignores them

- Kind: bug
- File: `scripts/changed-lanes.mjs`
- Line: 239
- Evidence: This commit adds `D` to the staged/worktree diff filters, so deleting a release metadata file now reaches lane detection. `detectChangedLanes()` then classifies paths such as `CHANGELOG.md` as `releaseMetadata`, but the release-metadata guard recomputes its own diff with `--diff-filter=ACMR` in `scripts/check-release-metadata-only.mjs:55`, excluding deletions. A focused temp-repo smoke with a staged `CHANGELOG.md` deletion produced `check:changed --dry-run --staged` => `lanes=release metadata`, then `check-release-metadata-only --staged` => `[release-metadata] ok (0 files)` with exit status 0.
- Impact: A deletion of a release metadata file can avoid the normal changed-gate lanes and also avoid the release-metadata guard. For files like `CHANGELOG.md` or version metadata, this can let a destructive deletion look like a valid release-metadata-only change.
- Suggested fix: Include `D` in `scripts/check-release-metadata-only.mjs` or pass the already-detected path list into the guard, then fail explicitly when a release metadata path is deleted. Add staged/range deletion tests for at least `CHANGELOG.md` and one version file.
- Confidence: high
