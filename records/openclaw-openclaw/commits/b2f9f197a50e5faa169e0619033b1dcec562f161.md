# Commit b2f9f197

## Summary

Found one low-severity diagnostic bug in the new WhatsApp group-drop guidance for multi-account configs. Runtime routing and sender authorization still look unchanged.

## Findings

### Low: Unregistered-group diagnostic points named-account users at the wrong config scope

- Kind: bug
- File: `extensions/whatsapp/src/auto-reply/monitor/group-gating.ts`
- Line: 118
- Evidence: The new drop log always says to add the group JID to `channels.whatsapp.groups`. But WhatsApp account merging lets `channels.whatsapp.accounts.<id>.groups` replace the root groups map for named accounts (`extensions/whatsapp/src/account-config.ts:50-51`), and the docs state the same replacement contract (`docs/channels/whatsapp.md:652`). For a `work` account with its own `groups` map, adding the dropped group only to root `channels.whatsapp.groups` will not admit it.
- Impact: Operators using multi-account WhatsApp with account-scoped groups can follow the new diagnostic exactly and still see the group dropped, making the diagnostic misleading in a supported configuration.
- Suggested fix: Make the drop guidance account-aware, or phrase it as the effective account scope. For example, when `msg.accountId` has an explicit account-level groups map, point to `channels.whatsapp.accounts.<accountId>.groups`; otherwise point to root `channels.whatsapp.groups`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Source path: configure a named WhatsApp account with `channels.whatsapp.accounts.work.groups` excluding `123@g.us`; process a group message for `accountId: "work"` from `123@g.us`; `applyGroupGating` drops it and logs root-only guidance even though the account-level groups map is the effective allowlist.
- Is this the best way to solve the issue? yes. The fix should stay in diagnostics only and preserve the existing group admission and sender authorization contracts.
