# Commit d922edd

## Summary

Found one low-severity regression in interactive Codex migration: archive-only Codex plans can no longer be applied from the normal interactive path.

## Findings

### Low: Interactive Codex migration skips archive-only plans before confirmation

- Kind: regression
- File: `src/commands/migrate.ts`
- Line: 192
- Evidence: `hasSelectedCodexMigrationWork` only counts planned `skill/copy` and `plugin/install` items, and `migrateDefaultCommand` returns before confirmation when that returns false. But Codex planning still emits planned archive items for `config.toml` and `hooks/hooks.json`, and Codex apply archives those items into the migration report. When a source Codex home has only archiveable config/hooks and no selectable skills/plugins, both selector functions return the original plan without prompting, then the new guard logs “No Codex skills or native Codex plugins selected” and returns without applying the archive items.
- Impact: Users running `openclaw migrate codex` interactively on a config/hooks-only Codex source cannot create the migration report/archive from the normal prompt flow. This regresses the documented “Codex config and hook files are archive-only” preservation behavior; `--yes` still applies, but the preview-first interactive path silently exits before the apply confirmation.
- Suggested fix: Gate the early return on an explicit empty selector choice, or check for no remaining planned items at all rather than no planned skill/plugin items. Archive/config/manual planned work should still reach the apply confirmation when no selector was shown.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Use a Codex source with `config.toml` or `hooks/hooks.json` but no migratable skills or source-installed curated plugins, then run `openclaw migrate codex` in a TTY. `buildCodexMigrationPlan` creates planned archive items, both selectors return without changes, and `shouldSkipCodexApplyAfterInteractiveSelection` exits before `promptYesNo`.
- Is this the best way to solve the issue? yes. The fix should keep the intended “empty selection exits cleanly” behavior, but distinguish “the user selected no skill/plugin work” from “there were no skill/plugin selectors and archive work remains planned.”
