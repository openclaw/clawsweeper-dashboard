# Commit a9aafc84

## Summary

Found one regression in git installer ref resolution: exact tag installs can now resolve to a same-named branch.

## Findings

### Medium: Same-named branches now override exact release tags

- Kind: regression
- File: `scripts/install-cli.sh`
- Line: 426
- Evidence: `checkout_git_openclaw_ref` now checks `ls-remote --heads origin "$ref"` and checks out `origin/$ref` before fetching or checking `refs/tags/${ref}`. The same order exists in `scripts/install.sh:1972`. The live remote currently has both `refs/heads/v2026.5.12-beta.6` at `5bf1258d5dba1a0b5f1327f088356fcdf81fe0c5` and `refs/tags/v2026.5.12-beta.6^{}` at `c4292e053d7c86361700b3acb59cb51e4a310f2d`, so `OPENCLAW_VERSION=2026.5.12-beta.6 --install-method git` installs the moving branch instead of the exact release tag. A synthetic temp-repo check confirmed the base commit checked the tag commit, while this commit checks the branch commit.
- Impact: Exact version installs are no longer reproducible when a release tag collides with a branch name. Users requesting an older beta tag can silently build an unrelated branch tip.
- Suggested fix: Preserve tag precedence without fetching all tags for branch installs. For non-`main` refs, first check/fetch the exact tag ref only, e.g. `refs/tags/${ref}:refs/tags/${ref}`, and detach using `refs/tags/${ref}^{commit}`. Only fall through to the branch-only `--no-tags` fetch when no exact tag exists.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: create a remote with `refs/tags/v-collision` and `refs/heads/v-collision` pointing to different commits, source `scripts/install-cli.sh`, then call `checkout_git_openclaw_ref <checkout> v-collision`; current code checks out the branch commit.
- Is this the best way to solve the issue? yes: exact tag lookup/fetch preserves the old version-tag contract while still avoiding a full tag fetch for ordinary moving branch installs.
