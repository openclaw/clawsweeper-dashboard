# Commit acb2f91

## Summary

Found one compatibility regression in the new provider validator: it rejects configured provider ids with uppercase characters even though the runtime has accepted them case-insensitively.

## Findings

### Low: Provider validation is stricter than runtime case normalization

- Kind: compatibility
- File: `src/config/validation.ts`
- Line: 1057
- Evidence: The new validator trims the raw configured value and checks `allowedValues.includes(trimmed)` at `src/config/validation.ts:1050-1058`. Runtime provider selection, unchanged from the parent commit, uses `normalizeLowercaseStringOrEmpty(search.provider)` before matching providers at `src/web-search/runtime.ts:152-158`, so values like `"Brave"` or `"GROK"` previously selected `brave`/`grok`.
- Impact: Existing configs that worked because runtime normalized provider ids can now fail config validation and block load/startup with `unknown web_search provider`, even though the provider is otherwise valid.
- Suggested fix: Normalize the configured provider with the same lowercase normalization used by runtime before comparing against manifest ids. Ideally add a regression test for `provider: "Brave"` or `provider: "GROK"`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. Use a config with `tools.web.search.provider: "Brave"`: the validator compares `"Brave"` against lowercase manifest ids, while runtime normalizes it to `"brave"`.
- Is this the best way to solve the issue? Yes. Matching validator normalization to runtime preserves the new typo detection without narrowing a previously accepted config form.
