# Commit de47989

## Summary

Found one high-confidence security regression. The commit newly recognizes abbreviated PowerShell `-Command` switch forms like `-co` and `/c`, but the shared extractor still takes only the next argv token as the PowerShell payload. PowerShell executes following tokens as part of the command, so allowlist mode can approve a safe first token and still run unapproved trailing PowerShell command text.

## Findings

### High: PowerShell `-Command` prefixes approve only the first token but execute the full tail

- Kind: security
- File: `src/infra/shell-inline-command.ts`
- Line: 15
- Evidence: `expandPowerShellSwitchPrefixForms("command", "c")` now accepts `-co`, `/c`, and other prefixes as inline command flags, but `resolveInlineCommandMatch` still returns only `argv[i + 1]` for non-rest flags. On current `main`, `resolveSystemRunCommandRequest({ command: ["pwsh","-co","/usr/bin/printf","ok",";","/usr/bin/id"] })` returns `shellPayload: "/usr/bin/printf"`, and `evaluateSystemRunAllowlist` with only `{ pattern: "/usr/bin/printf" }` returns `allowlistSatisfied: true`. The policy then allows execution and `resolveSystemRunExecArgv` keeps the original argv: `["pwsh","-co","/usr/bin/printf","ok",";","/usr/bin/id"]`.
- Impact: In allowlist mode, a user with `/usr/bin/printf` or another benign executable allowlisted can run additional unallowlisted PowerShell command text through newly recognized forms such as `pwsh -co /usr/bin/printf ok ; /usr/bin/id`. Local PowerShell 7.4.14 executed that exact shape and ran `/usr/bin/id`.
- Suggested fix: Treat PowerShell `-Command` and all recognized aliases/prefixes as rest-of-argv payload carriers, similar to `-CommandWithArgs`, or fail closed when any trailing argv exists after a PowerShell `-Command` value. The allowlist analysis must see `/usr/bin/printf ok ; /usr/bin/id`, not only `/usr/bin/printf`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Run the function-level check above, or execute `pwsh -NoLogo -NoProfile -co /usr/bin/printf ok ';' /usr/bin/id` and observe that PowerShell runs the trailing command. Microsoft’s `about_Pwsh` docs also state that when `-Command` is a string, it must be the last parameter because following arguments are interpreted as part of the command to execute. ([learn.microsoft.com](https://learn.microsoft.com/en-us/powershell/module/microsoft.powershell.core/about/about_pwsh?view=powershell-7.5&utm_source=openai))
- Is this the best way to solve the issue? yes. The approval boundary is the extracted shell payload, so the safe fix is to make the payload match PowerShell execution semantics before allowlist evaluation.
