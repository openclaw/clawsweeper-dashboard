# Commit 70ebf48

## Summary

Found one concrete issue: the commit updates `isChannelConfigured()` semantics, but the plugin auto-enable path still does not see `channels.<id>: { enabled: true }` as a configured channel. That leaves the reported plugin-backed channel startup path unfixed unless the old `accounts: {}` workaround is still present.

## Findings

### Medium: Enabled-only channel config still does not auto-enable the channel plugin

- Kind: bug
- File: `src/channels/config-presence.ts`
- Line: 40
- Evidence: `src/config/channel-configured-shared.ts:18` now treats `{ enabled: true }` as configured, but `src/config/plugin-auto-enable.shared.ts:270` gathers auto-enable channel ids through `listPotentialConfiguredChannelPresenceSignals()`, whose `hasMeaningfulChannelConfig()` still returns true only for keys other than `enabled` at `src/channels/config-presence.ts:40`.
- Impact: Startup/plugin activation paths call `applyPluginAutoEnable()` before loading plugin-backed channels. For a manifest that owns `openclaw-weixin`, an ad hoc probe showed `isChannelConfigured({ channels: { "openclaw-weixin": { enabled: true } } }) === true`, but `applyPluginAutoEnable()` produced no candidates and no `plugins.entries.openclaw-weixin.enabled`. With the workaround shape `{ enabled: true, accounts: {} }`, auto-enable did enable the plugin. This preserves the failure mode described in issue `#81323` for enabled-only channel config.
- Suggested fix: Make the channel presence scanner share the same enabled-only semantics as `hasMeaningfulChannelConfigShallow()`, or have `collectConfiguredChannelIds()` enumerate `cfg.channels` directly and defer final acceptance to `isChannelConfigured()`. Add a plugin auto-enable regression for a third-party manifest channel with only `{ enabled: true }`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The direct probe was:
  `node_modules/.bin/tsx -e '<imports applyPluginAutoEnable/isChannelConfigured; manifestRegistry owns openclaw-weixin; config has channels.openclaw-weixin.enabled=true>'`
  It returned `isChannelConfiguredEnabledOnly: true`, `enabledOnlyAutoEnabled: null`, and `accountsWorkaroundAutoEnabled: true`.
- Is this the best way to solve the issue? yes. The broken contract is the duplicated “meaningful channel config” predicate; consolidating it prevents the configured-state and auto-enable paths from drifting again.
