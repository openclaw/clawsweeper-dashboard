# Commit 3ce8746b27e1

## Summary

Found two actionable regressions in the new full-release validation flow. The main one is still present on current `main`: documented release branch/tag validation from trusted `main` will fail the new unconditional child `headSha == TARGET_SHA` verifier.

## Findings

### Medium: Trusted-ref release branch/tag validation now fails the child SHA verifier

- Kind: regression
- File: `.github/workflows/full-release-validation.yml`
- Line: 670
- Evidence: Child workflows are dispatched from `CHILD_WORKFLOW_REF=${{ github.ref_name }}` via `gh workflow run ... --ref "$CHILD_WORKFLOW_REF"` at `.github/workflows/full-release-validation.yml:199`, while the release target is passed separately as inputs such as `target_ref="$TARGET_SHA"` and `ref="$TARGET_SHA"` at `.github/workflows/full-release-validation.yml:262` and `.github/workflows/full-release-validation.yml:466`. The new summary verifier then compares every child run’s workflow `headSha` to the resolved target SHA at `.github/workflows/full-release-validation.yml:670`. Current docs still instruct release branch/tag validation from `--ref main -f ref=release/YYYY.M.D` and state children run from the trusted ref, normally `main`, even when the target points at an older release branch or tag (`docs/reference/RELEASING.md:251`, `docs/reference/RELEASING.md:280`). `git ls-remote` also shows `main` and `release/2026.4.30` currently resolve to different SHAs.
- Impact: A documented release validation run from trusted `main` against a release branch or tag will dispatch successful children from `main`, then fail the umbrella summary because those child runs’ `headSha` is the harness SHA rather than the target release SHA. That blocks the normal release validation path.
- Suggested fix: Only require child `headSha == TARGET_SHA` for the pinned exact-SHA helper path, for example a `release-ci/<sha>-<timestamp>` workflow ref or when the workflow ref itself resolves to `TARGET_SHA`. For trusted `main` release branch/tag validation, verify child `headSha` against the resolved harness/workflow ref and keep validating the target through the existing child inputs (`target_ref`, `ref`, `expected_sha`).
- Confidence: high

### Low: Documented helper pass-through syntax rejects `-- -f key=value`

- Kind: bug
- File: `scripts/full-release-validation-at-sha.mjs`
- Line: 74
- Evidence: The helper usage advertises `[-- -f key=value ...]` at `scripts/full-release-validation-at-sha.mjs:13`, but after seeing `--`, the parser treats each remaining token independently and expects every token to contain `=`. A focused dry run failed with `Unsupported extra argument after --: -f` for `pnpm ci:full-release --sha 3ce8746b27e11f946822229094210d64bcbd38ed --dry-run -- -f release_profile=stable`. The non-`--` form, `-f release_profile=stable`, works.
- Impact: Operators following the helper’s own usage text cannot override workflow inputs with the advertised separator form; the command fails before dispatch.
- Suggested fix: Either support paired `-f key=value` tokens inside the post-`--` loop, or change the usage text to only advertise supported forms such as `-f key=value` before `--` or `-- key=value`.
- Confidence: high

- Is this the best way to solve the issue? yes for the fix direction: the verifier needs to distinguish “exact pinned commit proof” from “trusted harness validating a separate release target.”
