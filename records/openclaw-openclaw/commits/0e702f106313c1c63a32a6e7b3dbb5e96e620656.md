# Commit 0e702f1063

## Summary

Found one concrete residual auth-rotation bug in the trusted-proxy generation work.

## Findings

### Medium: trusted-proxy local password fallback sessions ignore password rotation

- Kind: security
- File: `src/gateway/server/ws-shared-generation.ts`
- Line: 40
- Evidence: Trusted-proxy mode intentionally accepts local direct password fallback in `src/gateway/auth.ts:469`, and docs tell internal direct clients to use `gateway.auth.password` in this mode. The new session generation hashes token/password only when `auth.mode` is exactly `"token"` or `"password"`; for `auth.mode === "trusted-proxy"` it hashes only proxy policy and `gateway.trustedProxies`, omitting `auth.password` at `src/gateway/server/ws-shared-generation.ts:46`. The config-write detector mirrors that omission by comparing only trusted-proxy policy and proxy source lists at `src/gateway/server-methods/config-write-flow.ts:63`. A focused `tsx` helper check showed old vs new trusted-proxy passwords produce the same generation.
- Impact: A local internal WebSocket authenticated with the old trusted-proxy fallback password can keep its cached session and scopes after `gateway.auth.password` is rotated.
- Suggested fix: Include the effective trusted-proxy fallback password in the generation/change comparison when present, or track password-authenticated trusted-proxy fallback sessions with a separate password-based generation. Add regression coverage for password rotation in trusted-proxy mode.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes; source inspection plus a focused helper check. Start trusted-proxy mode with `gateway.auth.password`, connect locally using that password, rotate the password, and the existing session’s generation remains unchanged.
- Is this the best way to solve the issue? yes; password fallback is a documented auth credential, so it should participate in the same stale-session invalidation as other shared secrets.
