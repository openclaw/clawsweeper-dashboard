# Commit 4f80cc1943

## Summary

Found two current, actionable availability regressions in the provider-auth warm lifecycle. Current `main` has fixed the original provider-scope leakage with config/workspace/generation/logout/reload guards, but model listing can still re-enter the measured slow synchronous auth path after startup and after the 10s prepared-state TTL.

## Findings

### High: `/models` can still hit the slow auth sweep after the 10s prepared-state TTL

- Kind: reliability
- File: `src/agents/model-provider-auth.ts`
- Line: 29
- Evidence: Current `main` expires the prepared provider-auth state after `PREPARED_PROVIDER_AUTH_STATE_TTL_MS = 10_000`, then `hasAuthForModelProvider` falls through to `hasRuntimeAvailableProviderAuth` and `ensureAuthProfileStore` (`src/agents/model-provider-auth.ts:75`, `src/agents/model-provider-auth.ts:104`). `createProviderAuthChecker` is per request, so the PR’s measured ~20s provider loop can recur on later `/models` calls. The current test explicitly proves the fallback after 10,001ms (`src/agents/model-provider-auth.test.ts:125`).
- Impact: Long-running gateways can bring back the Discord picker ack failure this patch set is meant to remove; Discord’s 3s interaction window is still exceeded when the slow path runs.
- Suggested fix: Keep the last prepared map on the hot path and refresh asynchronously, or invalidate only from known auth/config/plugin/logout mutations instead of time-based synchronous expiry.
- Confidence: high

### High: Gateway logs ready and accepts channel work before the startup warm is usable

- Kind: reliability
- File: `src/gateway/server-startup-post-attach.ts`
- Line: 1021
- Evidence: Channels start before `gateway ready` (`src/gateway/server-startup-post-attach.ts:477`), `gateway ready` is logged at `src/gateway/server-startup-post-attach.ts:979`, and only then does the fire-and-forget continuation call `warmCurrentProviderAuthState`. The PR body for #84816 reports the one-time warm at about 49s and the pre-existing auth sweep at about 20s; that work runs after channels are available.
- Impact: A model picker request arriving just after gateway readiness but before warm completion can still block on the old discovery path, or contend with the warm itself, causing the same user-visible interaction timeout.
- Suggested fix: Do not expose channel model-picker handling until a usable provider-auth state exists, or make handlers ack/defer before any warm-dependent work and use a nonblocking pending/stale snapshot path.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Start a gateway with the large provider catalog from #84816, trigger `/models` immediately after `gateway ready` but before `provider auth state pre-warmed`, then trigger again more than 10s after warm completion; source and tests show both paths can bypass the prepared map.
- Is this the best way to solve the issue? yes. The safest direction is lifecycle-owned stale-while-refresh or explicit invalidation, not a synchronous TTL fallback on the user-facing model-listing path.
