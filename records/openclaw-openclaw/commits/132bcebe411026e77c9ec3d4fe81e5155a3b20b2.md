# Commit 132bceb

## Summary

Found one low-severity regression in live-test model filtering. The production config normalization change looks acyclic and equivalent for Google model refs, but the live matcher and gateway explicit lookup now drop manifest-owned model-id normalization for non-Google providers.

## Findings

### Low: Explicit live model filters no longer honor non-Google manifest model normalization

- Kind: regression
- File: `src/agents/live-target-matcher.ts`
- Line: 18
- Evidence: `parseModelTarget` now calls `normalizeLiveTargetModelId`, which only rewrites Google providers. Before this commit it called `normalizeStaticProviderModelId`, which also applied manifest policies such as `extensions/nvidia/openclaw.plugin.json` `prefixWhenBare: "nvidia"`. A focused smoke on current `main` showed `normalizeStaticProviderModelId("nvidia", "nemotron-3-super-120b-a12b")` returns `nvidia/nemotron-3-super-120b-a12b`, but `createLiveTargetMatcher({ modelFilter: new Set(["nvidia/nemotron-3-super-120b-a12b"]) }).matchesModel("nvidia", "nvidia/nemotron-3-super-120b-a12b")` returns `false`.
- Impact: documented explicit live filters like `OPENCLAW_LIVE_MODELS="provider/model"` / `OPENCLAW_LIVE_GATEWAY_MODELS="provider/model"` can silently select zero candidates for manifest-normalized non-Google models, including NVIDIA bare-prefixed models and provider aliases in other manifests. The direct and gateway live suites then report no candidates / no keys instead of proving the requested model.
- Suggested fix: keep the import cycle fix, but restore generic manifest-policy normalization through a dependency-light helper or a prepared manifest-policy input to the live matcher and gateway explicit parser. Add a regression test for a non-Google manifest-normalized provider such as NVIDIA.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: the one-off matcher smoke above reproduces it without network or credentials.
- Is this the best way to solve the issue? yes, if live filters are meant to follow the same provider/model ref contract as normal model resolution. If the new intended contract is Google-only normalization, the live docs and explicit-filter tests should say that and explicit filters selecting no models should fail loudly.
