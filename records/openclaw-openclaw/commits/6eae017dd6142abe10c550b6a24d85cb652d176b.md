# Commit 6eae017

## Summary

Found one concrete issue: the new detection covers PI API-provider function identities, but the production PI Coding Agent default stream is an anonymous wrapper around `streamSimple`, so default embedded sessions can still be classified as `session-custom` and bypass the boundary-aware transport path this commit intended to restore.

## Findings

### Medium: Production PI default stream still falls through as custom

- Kind: bug
- File: `src/agents/pi-embedded-runner/stream-resolution.ts`
- Line: 40
- Evidence: `isDefaultPiStreamFnForModel()` now recognizes `streamSimple`, `getApiProvider(api).streamSimple`, and `getApiProvider(api).stream`. However, `@mariozechner/pi-coding-agent@0.73.0` creates the session agent with `streamFn: async (...) => streamSimple(...)`, after resolving auth and retry settings in `node_modules/@mariozechner/pi-coding-agent/dist/core/sdk.js`. That wrapper is captured from the created session in `src/agents/pi-embedded-runner/run/attempt.ts:1859`, and it is not identical to the global `streamSimple` or the PI API-provider stream functions. The resolver therefore skips the boundary-aware branch at `src/agents/pi-embedded-runner/stream-resolution.ts:122` and returns `currentStreamFn` at `src/agents/pi-embedded-runner/stream-resolution.ts:139`.
- Impact: The real embedded-runner default path can still bypass OpenClaw’s boundary-aware transport for supported APIs, including the transport policy and runtime credential path that this commit was meant to enforce. The added regression test uses `getApiProvider("openai-completions")?.streamSimple` directly, so it does not cover the actual default stream function produced by `createAgentSession()`.
- Suggested fix: Carry an explicit “PI default session stream” fact from session creation into `resolveEmbeddedAgentStreamFn`, or otherwise mark the default wrapper intentionally, instead of relying only on function identity. Keep provider-owned streams and deliberate custom session streams as higher-precedence paths.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes, by source inspection against the locked dependency and current call path: `createAgentSession()` installs an anonymous wrapper, `resolveEmbeddedAgentBaseStreamFn()` captures it, and the new identity check cannot match it.
- Is this the best way to solve the issue? The current fix direction is only partial. Matching PI provider functions is useful, but the safer production fix needs an explicit default-stream marker/fact rather than more dependency-internal function identity guesses.
