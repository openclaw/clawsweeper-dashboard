## Findings

### Medium: Docker attestation verifier accepts omitted `artifactType` with any manifest media type

- Kind: supply_chain
- File: `scripts/verify-docker-attestations.mjs`
- Line: 89
- Evidence: The commit changed the verifier to check only `artifactType` when that field exists. If `artifactType` is omitted, the inspected attestation `mediaType` is no longer validated. A focused in-memory call with `mediaType: "application/vnd.not-a-manifest+json"`, no `artifactType`, and the two required predicate annotations returned `[]` errors. Docker’s attestation docs show omitted-`artifactType` attestations as OCI image manifests, and the storage docs say standard OCI/Docker manifest properties still apply; layers should be `application/vnd.in-toto+json`, with unknown layer media types ignored. ([docs.docker.com](https://docs.docker.com/build/metadata/attestations/?utm_source=openai))
- Impact: `docker-release.yml` uses this script as the release check for GHCR image SBOM/provenance attestations. A malformed or unexpected attestation manifest can now satisfy the release gate as long as it carries the expected reference annotations and predicate annotations, reducing the check’s ability to catch broken Docker/BuildKit/registry output before publishing.
- Suggested fix: Restore a fallback manifest-shape check when `artifactType` is omitted, accepting only known Docker/OCI image manifest media types. Also consider counting predicates only from supported in-toto attestation layers.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Running `collectDockerAttestationErrors` with an attestation object that omits `artifactType`, uses an invalid top-level `mediaType`, and includes the required predicate annotations returns `[]`.
- Is this the best way to solve the issue? yes. The compatibility goal can be preserved by accepting omitted `artifactType` only when the manifest still has an expected Docker/OCI manifest media type.
