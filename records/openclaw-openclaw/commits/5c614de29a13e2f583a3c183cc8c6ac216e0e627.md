## Findings

### Medium: Direct slash handlers still capture longer skill command names

- Kind: bug
- File: `src/auto-reply/reply/commands-models.ts`
- Line: 652
- Evidence: The new test asserts `/modelsy` should not be captured by `/models` in `src/auto-reply/reply/commands-slash-parse.test.ts:36`, but the production `/models` handler does not use `parseSlashCommandOrNull`. It still gates with `commandBodyNormalized.startsWith("/models")` at `src/auto-reply/reply/commands-models.ts:652`; `resolveModelsCommandReply` has the same broad check at line 485. `/allowlist` has the same direct-prefix pattern at `src/auto-reply/reply/commands-allowlist.ts:79`. These handlers are registered built-ins at `src/auto-reply/reply/commands-handlers.runtime.ts:61` and `:74`.
- Impact: Prompt-style skill commands such as `/models-list`, `/modelsy`, or `/allowlist-check` can still be consumed by the shorter built-in handler after skill rewrite, returning built-in replies like unknown provider or allowlist output instead of allowing the intended skill invocation to continue. That leaves the linked prefix-collision bug only partially fixed while the commit closes #84572.
- Suggested fix: Add a shared slash command token-boundary helper and use it in both the shared parser and direct handlers, at least `/models` and `/allowlist`; add handler-level tests for longer slash tokens falling through.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Source inspection shows the direct `/models` and `/allowlist` handlers still use broad `startsWith` checks, and `get-reply-inline-actions.ts:550` later runs built-in handlers with the original slash command after prompt-style skill rewrite.
- Is this the best way to solve the issue? yes. A shared boundary helper avoids fixing only one parser while leaving direct handlers with a different command-token contract.
