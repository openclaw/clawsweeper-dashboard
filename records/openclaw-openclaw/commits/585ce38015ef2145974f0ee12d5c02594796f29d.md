## Summary

Found one actionable regression: the new Telegram topic coverage hangs in source-mode tests because the changed target-normalization path falls back to loading the real bundled Telegram plugin when no channel plugin is loaded.

## Findings

### Medium: Telegram target fallback hangs the new regression tests

- Kind: regression
- File: `src/infra/outbound/target-normalization.ts`
- Line: 25
- Evidence: `resolveChannelPluginForTargetRead()` now falls back from `getLoadedChannelPluginForRead(channelId)` to `getChannelPlugin(channelId)`. With an empty active registry, `getChannelPlugin("telegram")` did not return within a 20s focused smoke. The two new commit tests that exercise this path also timed out at 120s:
  - `src/infra/outbound/message-action-runner.core-send.test.ts:127`
  - `src/agents/tools/message-tool.test.ts:431`
- Impact: Focused CI/local test runs for these changed files fail after multi-minute timeouts, so the committed regression coverage does not pass and can block changed/full gates. It also means the intended Telegram-topic fix is not actually proven by the new tests.
- Suggested fix: Keep these core/agent tests on a lightweight test plugin or a narrow target-grammar artifact instead of loading the full bundled Telegram plugin through `getChannelPlugin()`. If production needs unloaded-plugin target grammar, expose/load a small public artifact for `normalizeTarget`/`looksLikeId` rather than the full channel plugin.
- Confidence: high

- Do we have a high-confidence way to reproduce the issue? yes. Run `pnpm test src/infra/outbound/message-action-runner.core-send.test.ts src/agents/tools/message-tool.test.ts`; both new tests time out.
- Is this the best way to solve the issue? yes. The failing path needs narrow Telegram target grammar, not full Telegram plugin loading in hot core tests.
