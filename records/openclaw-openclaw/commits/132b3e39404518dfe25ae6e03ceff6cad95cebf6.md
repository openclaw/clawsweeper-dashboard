# Commit 132b3e3

## Summary

Found two regressions in the new fallback capture path. The more important one bypasses explicit bundled plugin disablement for capability providers.

## Findings

### Medium: Bundled capability capture ignores explicit plugin disablement

- Kind: security
- File: `src/plugins/capability-provider-runtime.ts`
- Line: 391
- Evidence: `loadCapabilityProviderEntries()` first asks `resolveRuntimePluginRegistry(loadOptions)`, but when that returns no entries it directly calls `loadBundledCapabilityRuntimeRegistry({ pluginIds, env, pluginSdkResolution })` without the user config. That capture helper builds its own enablement config from `pluginIds` in `src/plugins/bundled-capability-runtime.ts:230`, while `withBundledPluginEnablementCompat()` intentionally preserves explicit existing entries such as `{ enabled: false }` in `src/plugins/bundled-compat.ts:45`.
- Impact: A user can explicitly disable a bundled plugin, but capability lookup can still run the plugin register path and return its provider. Focused smoke: with `plugins.entries.microsoft.enabled=false`, the normal snapshot had no speech providers, but `resolvePluginCapabilityProvider({ key: "speechProviders", providerId: "microsoft" })` returned `microsoft`. The same happened for disabled `memory-core`, returning the `local` memory embedding provider.
- Suggested fix: Keep the capture fallback under the same enablement decision as the snapshot load. Either pass and honor the effective config in `loadBundledCapabilityRuntimeRegistry`, or skip direct capture for plugin ids that are explicitly disabled in the caller config.
- Confidence: high

### Medium: Model provider config now collapses TTS auto provider discovery

- Kind: regression
- File: `src/plugins/capability-provider-runtime.ts`
- Line: 475
- Evidence: When no active speech providers are loaded, the new code derives `requestedSpeechProviders` from `collectRequestedSpeechProviderIds()`. That helper includes `cfg.models.providers` keys at `src/plugins/capability-provider-runtime.ts:265`, then `resolveRequestedCapabilityCompatPluginIds()` narrows the fallback load to only those provider ids. Focused smoke: `listSpeechProviders({})` returned 11 bundled speech providers, while `listSpeechProviders({ models: { providers: { openai: {} } } })` returned only `openai`.
- Impact: TTS auto mode with `messages.tts.provider` unset can lose normal speech fallbacks whenever a model provider config exists. That is a common config shape, and it can prevent separately configured TTS providers, including env-configured providers, from being considered.
- Suggested fix: Do not treat `models.providers` as a speech request when TTS has no explicit provider selection. Keep full speech provider discovery for auto-selection, or narrow only for explicit `messages.tts.provider` lookups while preserving configured fallback candidates.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The disabled-plugin repro is a focused `pnpm exec tsx` check that prints an empty snapshot and then `resolved:microsoft`; the memory provider variant prints an empty snapshot and then `resolved:local`.
- Is this the best way to solve the issue? yes for the disablement bypass: the capture path must honor the same config policy as the snapshot path. For the TTS narrowing issue, the safest direction is to separate explicit speech provider requests from unrelated model provider config.
