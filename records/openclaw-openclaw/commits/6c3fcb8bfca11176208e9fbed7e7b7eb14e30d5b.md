## Summary

Found one likely OpenAI video edit contract regression. Current `main` still contains the issue; the follow-up commit only fixed a test typecheck optional-chain access.

## Findings

### Medium: Video edit requests send unsupported `model` field

- Kind: bug
- File: `extensions/openai/video-generation-provider.ts`
- Line: 369
- Evidence: The new video-to-video path builds multipart `POST /videos/edits` and sets `prompt`, `model`, and `video`. OpenAI’s generated JS SDK 6.38.0 defines `VideoEditParams` as only `prompt` plus `video: Uploadable | { id }`, and its `Videos.edit` method posts that body to `/videos/edits` as multipart. The public API reference likewise describes creating a video edit from a source video, with request body fields for `prompt` and `video`, not `model`. `postMultipartRequest` forwards the `FormData` body unchanged. citeturn5fetch0
- Impact: OpenAI video-to-video requests can be rejected by the API as an unknown/unsupported `model` parameter, so the newly routed `/videos/edits` flow may still fail for real uploaded edits even though the unit test expects the extra field.
- Suggested fix: Remove `form.set("model", model)` from the `/videos/edits` branch and update the test to assert only `prompt` and `video` for edits. Keep `model` on `POST /videos` create/image-reference requests.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? Yes: call `video_generate` with `model=openai/sora-2` and a local video reference; inspect the outgoing multipart body or run against OpenAI with edit access. The current unit test already proves the body includes `model`.
- Is this the best way to solve the issue? Yes: it aligns OpenClaw’s edit request shape with the OpenAI SDK/API contract while preserving model selection for text/image create requests.
