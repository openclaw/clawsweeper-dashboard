# Commit dd60c6bc4d59

## Summary

Found one low-severity QA mock regression. The new Telegram stream special case can stale-match an older transcript marker and override a later streaming prompt’s exact reply directive.

## Findings

### Low: Telegram stream marker branch can override later quiet-streaming prompts

- Kind: bug
- File: `extensions/qa-lab/src/providers/mock-openai/server.ts`
- Line: 1530
- Evidence: the new branch checks `QA_STREAMING_PROMPT_RE.test(allInputText)` and `allInputText.includes(QA_TELEGRAM_STREAM_SINGLE_MARKER)`, then always returns `QA-TELEGRAM-STREAM-SINGLE-OK`. `allInputText` is the whole request history, not the latest user prompt, and this branch runs before the generic streaming exact-reply branch. A focused smoke with prior input `Quiet streaming QA check. Reply exactly: QA-TELEGRAM-STREAM-SINGLE-OK` followed by latest input `Quiet streaming QA check: reply exactly \`QA_STREAMING_OK\`.` returned `QA-TELEGRAM-STREAM-SINGLE-OK`.
- Impact: a later mock-backed quiet/partial streaming QA check in the same transcript can fail with a stale Telegram token even though the latest prompt asks for a different exact marker. This can create false release-QA failures in shared mock conversations.
- Suggested fix: scope the Telegram special case to the latest prompt, or require the latest extracted exact reply directive to equal `QA_TELEGRAM_STREAM_SINGLE_MARKER`; otherwise let the generic `exactReplyDirective` streaming branch handle it.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes: start `startQaMockOpenAiServer`, send `/v1/responses` with `stream: false`, first user input containing `QA-TELEGRAM-STREAM-SINGLE-OK`, and second/latest user input asking for ``QA_STREAMING_OK``. Current main returns the stale Telegram marker.
- Is this the best way to solve the issue? yes: the special case should be latest-turn or latest-directive scoped, matching existing exact-marker guards elsewhere in this mock.
