# Commit 4373103

## Summary

Found one low-severity reliability edge case in the new terminal cleanup path. The happy path is covered and passes, including the Discord message-handler tests, but a transient remove failure can still leave the stale reaction permanently tracked as removed.

## Findings

### Low: Failed terminal cleanup removals are not retried by the delayed clear

- Kind: reliability
- File: `src/channels/status-reactions.ts`
- Line: 344
- Evidence: The commit adds `await removeActiveEmojis({ keepEmoji: emoji })` inside `finishWithEmoji()`. `removeActiveEmojis()` catches removal errors but deletes that emoji from `activeEmojis` in `finally` at line 235. In the Discord/Slack `removeAckAfterReply` flow, `setDone()`/`setError()` runs first, then `clear()` is scheduled after the hold period. A focused smoke on current `main` reproduced the tracking loss: after `thinking` is active, making the first `removeReaction(🤔)` throw during `setDone()`, then calling `clear()`, produced only `set:🤔`, `set:👍`, `remove:🤔`, `remove:👍`; the later clear never retried `🤔`.
- Impact: A transient Discord/Slack remove failure or rate-limit during the newly added immediate terminal cleanup can still leave the non-terminal reaction permanently visible. This is close to the bug this commit is fixing, and with `removeAckAfterReply: true` the delayed clear no longer gets a chance to clean up the failed non-terminal emoji because it was removed from tracking already.
- Suggested fix: Keep failed removals in `activeEmojis` so a later `clear()`/`restoreInitial()` can retry, and have adapters normalize benign “not found/no reaction” cases to success before reaching the controller. Alternatively, track failed removals separately and drain them during terminal clear.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes for the controller state machine with an in-memory adapter; live Discord was not run. The reproduction is: activate `thinking`, make `removeReaction(thinking)` throw once during `setDone()`, then call `clear()` and observe that `thinking` is not retried.
- Is this the best way to solve the issue? unclear; the safest direction is to avoid deleting failed removals from tracking, but adapters may need to distinguish expected “already gone” errors from retryable failures.
