# Commit 5a0d6c7

## Summary

Found one concrete regression in the session-memory filename change. The new default filename uses only minute precision, so repeated `/new` or `/reset` commands in the same minute can silently overwrite earlier memory captures.

## Findings

### Medium: Default session-memory filenames can overwrite earlier captures from the same minute

- Kind: data_loss
- File: `src/hooks/bundled/session-memory/handler.ts`
- Line: 221
- Evidence: This commit makes LLM filename slugs opt-in (`hookConfig?.llmSlug === true`), so the default path now falls back to `localTimestamp.timeSlug`. That slug is only `HHMM`, and the filename is `${dateStr}-${slug}.md`. The write then replaces that target through `writeFileWithinRoot`; the pinned writer uses `os.replace`/rename semantics, so an existing file at the same path is overwritten.
- Impact: Two `/new` or `/reset` commands for the same workspace within one local minute both write `YYYY-MM-DD-HHMM.md`. The second capture silently replaces the first, losing the earlier session-memory artifact. Backgrounding the hook also makes close-together resets more likely to overlap on the same target.
- Suggested fix: Make the default filename collision-resistant without requiring `llmSlug`, for example include seconds plus a short session id/run id suffix, or allocate a unique suffix when the target already exists. Add a regression test that fires two same-minute events and expects two memory files.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Use session-memory with default config, send two `/new` or `/reset` events with timestamps in the same minute for the same workspace, then inspect `<workspace>/memory`; current code can produce only one `YYYY-MM-DD-HHMM.md` target for both events.
- Is this the best way to solve the issue? yes. Keeping LLM slugs opt-in is fine for responsiveness, but the local fallback needs uniqueness because memory capture is archival and should not replace prior captures.
