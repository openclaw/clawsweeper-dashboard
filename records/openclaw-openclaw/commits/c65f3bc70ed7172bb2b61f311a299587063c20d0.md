# Commit c65f3bc

## Summary

Found one Slack callback authorization regression: the new `commands.allowFrom` path does not honor Slack’s accepted `slack:` / `user:`-prefixed user IDs, so authorized users can be delivered to plugin callbacks as unauthorized.

## Findings

### Medium: Slack plugin callbacks reject prefixed `commands.allowFrom` identities

- Kind: regression
- File: `extensions/slack/src/monitor/events/interactions.block-actions.ts`
- Line: 693
- Evidence: The new Slack plugin callback path delegates `commands.allowFrom` cases to `resolveCommandAuthorization()`. A focused runtime probe showed `commands.allowFrom.slack: ["slack:U123", "user:U456", "U789"]` authorizes only bare `U789`; `U123` and `U456` return `isAuthorizedSender=false`. The channel’s own Slack allowlist matcher accepts all three forms.
- Impact: Slack users configured with provider-prefixed command identities can still reach the callback, but plugin handlers receive `ctx.auth.isAuthorizedSender: false`. Plugins that gate resume/action/model controls on this flag will disable or reject legitimate operator actions.
- Suggested fix: Normalize Slack `commands.allowFrom` entries with the same Slack-specific candidates used by `resolveSlackAllowListMatch()`, or strip `slack:` / `user:` in Slack command-auth formatting. Add callback tests for `slack:U...` and `user:U...`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: configure `commands.allowFrom.slack` with `slack:U123`, trigger a Slack plugin block action as `U123`, and observe `ctx.auth.isAuthorizedSender === false`.
- Is this the best way to solve the issue? yes: the callback path should use Slack-native allowlist normalization so its new dynamic auth matches existing Slack allowlist contracts.
