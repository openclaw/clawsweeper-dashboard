# Commit 84a3b50

## Summary

Found one active regression in the POSIX Parallels OpenAI SSE smoke config path.

## Findings

### Medium: POSIX smoke writes dotted model IDs as unescaped config paths

- Kind: regression
- File: `scripts/e2e/parallels/npm-update-scripts.ts`
- Line: 39
- Evidence: The commit writes `agents.defaults.models.${modelId}` for OpenAI models. The default OpenAI model is `openai/gpt-5.5`, and `config set` splits paths on `.`, so this becomes `agents.defaults.models["openai/gpt-5"]["5"]` instead of `agents.defaults.models["openai/gpt-5.5"]`. I reproduced with the same path against a temp config containing the normal `models set openai/gpt-5.5` output; `openclaw config set --batch-json '[{"path":"agents.defaults.models.openai/gpt-5.5",...}]' --strict-json` failed with `Config validation failed: agents.defaults.models.openai/gpt-5: Invalid input: expected object, received array`. The issue was introduced in the POSIX call sites at `scripts/e2e/parallels/linux-smoke.ts:706`, `scripts/e2e/parallels/macos-smoke.ts:990`, and `scripts/e2e/parallels/npm-update-scripts.ts:39`; current `main` still has the equivalent path in `scripts/e2e/parallels/provider-auth.ts:138`.
- Impact: The macOS/Linux Parallels OpenAI SSE agent-turn checks can fail at config setup before exercising the intended SSE transport, especially for the default `openai/gpt-5.5` model.
- Suggested fix: Build the path with bracket notation, for example `agents.defaults.models["openai/gpt-5.5"]`, or otherwise escape `.` in model ids before passing paths to `config set` / batch mode.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: the temp-config `openclaw config set --batch-json` check above reproduces the same parser failure without needing a Parallels guest.
- Is this the best way to solve the issue? yes: bracket path notation matches the existing config path parser and preserves model IDs containing dots.
