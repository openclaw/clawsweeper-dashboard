# Commit 2e495b0

## Summary

Found one concrete bug in the wildcard allowlist handling. The wildcard sentinel is preserved as intended, but current selection checks also accept the sentinel itself as a runnable model.

## Findings

### Medium: `provider/*` can be selected and persisted as model `*`

- Kind: bug
- File: `src/agents/model-selection-shared.ts`
- Line: 973
- Evidence: The commit stores provider wildcard entries as `provider/*` in `allowedKeys`, then `isModelKeyAllowedBySet()` returns `true` immediately when the candidate key is exactly present. Current `src/auto-reply/reply/model-selection-directive.ts:402` passes direct `/model` selections through that helper, so `raw: "vllm/*"` with `allowedModelKeys: new Set(["vllm/*"])` resolves to `{"provider":"vllm","model":"*"}`. `src/auto-reply/reply/directive-handling.persist.ts:266` then persists that selection via `applyModelOverrideToSessionEntry()`, which writes `modelOverride = "*"`.
- Impact: A user can accidentally switch a session, OpenAI-compatible header, or one-off override to the wildcard marker itself. That leaves the session pointed at a non-model (`vllm/*`, `openai-codex/*`, etc.), so later replies can fail until the model override is reset.
- Suggested fix: Treat wildcard keys as allowlist metadata only. Either make `isModelKeyAllowedBySet()` return false when the requested model segment is exactly `*`, while still matching `provider/*` for concrete `provider/model` keys, or add a separate concrete-selection helper and use it on all user-facing override paths.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: on current `main`, `pnpm exec tsx -e '...'` calling `resolveModelDirectiveSelection({ raw: "vllm/*", allowedModelKeys: new Set(["vllm/*"]), ... })` returns `selection: { provider: "vllm", model: "*" }`.
- Is this the best way to solve the issue? yes: the wildcard sentinel should remain in `allowedKeys` for concrete model matching, but direct selection of the sentinel should be rejected centrally so all override surfaces share the same behavior.
