# Commit bce56ba

## Summary

Found one low-severity presentation regression in migration result output: archive failures can now render with the archive glyph instead of the error/conflict glyph.

## Findings

### Low: Archive migration failures are rendered as archived rows

- Kind: bug
- File: `src/commands/migrate/output.ts`
- Line: 184
- Evidence: `formatItemPrefix` returns `📖 ` for every `item.kind === "archive"` before checking `mode === "result"` and `RESULT_STATUS_GLYPHS`. Archive apply can concretely return `error` for missing source/symlink/copy failures and `conflict` for existing-file races in `src/plugin-sdk/migration-runtime.ts:137` and `src/plugin-sdk/migration-runtime.ts:161`.
- Impact: a failed archive row in the final migration output is visually marked as “filed away” instead of failed or conflicted. The command still fails via the summary/error path, and the row still includes the reason text, so this is not data loss, but it can mislead users scanning a multi-item migration result.
- Suggested fix: preserve result failure glyphs before applying the special archive glyph, for example by returning `❌`/`⚠️` for `error` and `conflict` in result mode before the `archive` kind override.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A formatter smoke with an archive item `{ status: "error", reason: "missing source or target" }` renders `📖 config.toml (missing source or target)`.
- Is this the best way to solve the issue? yes. The commit’s intended archive glyph applies to successful archived rows, while failures should continue using the status glyph contract.
