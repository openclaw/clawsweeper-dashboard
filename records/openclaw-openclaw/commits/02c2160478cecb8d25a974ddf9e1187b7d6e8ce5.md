# Commit 02c2160

## Summary

Found one supply-chain regression: moving the sandbox Dockerfiles out of the repository root dropped them from the current Dependabot Docker update scope.

## Findings

### Medium: Moved sandbox Dockerfiles are no longer covered by Dependabot Docker updates

- Kind: supply_chain
- File: `.github/dependabot.yml`
- Line: 117
- Evidence: The commit moves `Dockerfile.sandbox` and `Dockerfile.sandbox-browser` to `scripts/docker/sandbox/Dockerfile` and `scripts/docker/sandbox/Dockerfile.browser`; both still pin `debian:bookworm-slim` by digest at line 3. The Dependabot config still has only one Docker updater with `directory: /` at `.github/dependabot.yml:117-118`. GitHub’s Dependabot docs define `directory` as the location of the manifest/definition files and show Docker `directory: "/"` as looking for a Dockerfile in the root directory; current dependabot-core Docker fetching lists candidate Dockerfiles from the configured directory contents, not recursively from the whole repository. ([docs.github.com](https://docs.github.com/en/code-security/dependabot/working-with-dependabot/dependabot-options-reference))
- Impact: Future Debian base image digest updates for the default sandbox and browser sandbox images can be missed. Those images install OS packages used by sandboxed agent/tool execution, so stale base images reduce the value of the digest-pinning and automated security update workflow.
- Suggested fix: Add Docker Dependabot coverage for `/scripts/docker/sandbox` using either another `package-ecosystem: docker` entry or `directories` including both `/` and `/scripts/docker/sandbox`, and add/adjust a test so the moved sandbox Dockerfile directory is asserted as covered.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Static reproduction is enough: compare `.github/dependabot.yml:117-118` with the moved Dockerfile paths and Dependabot’s documented/source behavior for `directory`.
- Is this the best way to solve the issue? yes. The Dockerfiles intentionally moved; the safer fix is to extend the dependency updater scope rather than move them back or rely on manual digest audits.
