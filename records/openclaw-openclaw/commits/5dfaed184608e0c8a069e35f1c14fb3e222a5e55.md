# Commit 5dfaed1

## Summary

Found one current regression in the new post-compaction guard wiring: it is still enabled when `tools.loopDetection` is absent, even though the shared loop-detection switch defaults to off and the docs/maintainer guidance say guardrails stay off unless explicitly enabled.

## Findings

### Medium: Post-compaction guard ignores the default-off loop-detection policy

- Kind: reliability
- File: `src/agents/pi-embedded-runner/run.ts`
- Line: 800
- Evidence: `resolveToolLoopDetectionConfig()` returns `undefined` when no `tools.loopDetection` config exists (`src/agents/tool-loop-detection-config.ts:15`), but the runner passes `{ enabled: resolvedLoopDetectionConfig?.enabled !== false }`, which evaluates to `true` for absent config. A focused probe on current `main` with `cfg: {}` armed the guard and produced `compaction_loop_persisted` after three identical post-compaction tool outcomes. This conflicts with `src/config/types.tools.ts:175`, `src/agents/tool-loop-detection.ts:33`, and `docs/tools/loop-detection.md:10` / `docs/tools/loop-detection.md:124`, which define loop detection as default-off. It also conflicts with the maintainer decision in #77474 to keep tool-loop detection opt-in.
- Impact: default installs can now abort post-compaction runs with `compaction_loop_persisted` even when the operator never enabled loop detection. That is surprise blocking behavior in the exact policy area this commit is trying to centralize under `tools.loopDetection.enabled`.
- Suggested fix: pass an explicitly defaulted switch, e.g. enable the post-compaction guard only when `resolvedLoopDetectionConfig?.enabled === true`, or share the same default resolution used by `detectToolCallLoop()` before constructing the guard. Add a runner test for absent `tools.loopDetection` proving no abort.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: run the guard with `resolveToolLoopDetectionConfig({ cfg: {}, agentId: "main" })`, then observe the same `{toolName,argsHash,resultHash}` three times; current main aborts.
- Is this the best way to solve the issue? yes: the safest fix is to make the post-compaction guard consume the same defaulted master switch semantics as the rest of loop detection, instead of treating only explicit `enabled: false` as disabled.
