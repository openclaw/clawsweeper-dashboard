# Commit eb2701e

## Summary

Found one concrete regression in the package dist inventory change: the inventory can now record files that `npm pack` intentionally excludes, which can make the installed package fail its own dist verification.

## Findings

### Medium: Inventory can reference plugin `node_modules` files that npm never packs

- Kind: regression
- File: `src/infra/package-dist-inventory.ts`
- Line: 86
- Evidence: The commit removed the `dist/extensions/node_modules/**` and `dist/extensions/*/node_modules/**` omitted-subtree rules, so `isPackagedDistPath()` now returns true for those files. But `package.json:34` and `package.json:35` still exclude those same paths from the npm package. A temp-package smoke confirmed the mismatch: `writePackageDistInventory()` wrote `dist/extensions/demo/node_modules/left-pad/index.js` into `dist/postinstall-inventory.json`, while `npm pack --json --dry-run --ignore-scripts` omitted that file and packed only the inventory and manifest. Simulating the installed package then produced `missing packaged dist file dist/extensions/demo/node_modules/left-pad/index.js`.
- Impact: If a release/build workspace has stale legacy bundled-plugin `node_modules` debris under `dist/extensions/<plugin>/node_modules`, the generated package inventory can name files that are absent from the actual npm tarball. After install/update, `collectInstalledGlobalPackageErrors()` will report missing packaged dist files and can fail global install verification for a package that npm packed exactly as configured.
- Suggested fix: Keep the “do not mask legacy debris” behavior by rejecting these `dist/extensions/**/node_modules` trees before writing the inventory, or restore the npm-omitted subtree exclusion and add a separate prepack/release-time error for legacy plugin dependency debris. The inventory itself should only contain paths that can actually ship in the tarball.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Create `dist/extensions/demo/node_modules/left-pad/index.js`, run `writePackageDistInventory()`, then run `npm pack --json --dry-run --ignore-scripts` with the existing package `files` exclusion shape; the inventory includes the file and the pack list omits it.
- Is this the best way to solve the issue? no. The intent to stop hiding legacy dependency debris is sound, but recording npm-excluded files in the shipped inventory creates a new package self-verification failure. Rejecting the debris before inventory generation is safer.
