# Commit 64e28a6

## Summary

Found one compatibility regression in the metadata stripper: historical session messages with the old reply-context sentinel are no longer stripped.

## Findings

### Medium: Historical reply-context metadata can leak into visible history

- Kind: regression
- File: `src/auto-reply/reply/strip-inbound-meta.ts`
- Line: 25
- Evidence: This commit changes the generated reply-context label from `Replied message (untrusted, for context):` to `Reply target of current user message (untrusted, for context):` in `src/auto-reply/reply/inbound-meta.ts:272`, but `INBOUND_META_SENTINELS` now recognizes only the new label. `SENTINEL_FAST_RE` is built from that list, so old persisted messages skip parsing entirely. Focused check on current `main`: `stripInboundMetadata(oldBlock)` returned the full old metadata block, while the new label stripped to `Visible user text`.
- Impact: Existing stored sessions created before this commit can display AI-facing reply metadata in UI/chat history surfaces that call `stripInboundMetadata`, including `ui/src/ui/chat/message-normalizer.ts:377` and gateway sanitization at `src/gateway/chat-sanitize.ts:87`. The leaked block can include quoted-message body and sender label that the code comment says “must never surface in user-visible chat history.”
- Suggested fix: Keep generating the new label, but keep the old sentinel as a legacy accepted strip marker, ideally with a regression test that old `Replied message (untrusted, for context):` blocks still strip.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: on current `main`, `pnpm exec tsx --eval ...stripInboundMetadata(oldBlock/newBlock)...` produced `{"old":"Replied message ... Visible user text","new":"Visible user text"}`.
- Is this the best way to solve the issue? yes: preserving legacy strip sentinels is narrow and backwards-compatible; it avoids changing the new prompt wording while protecting persisted transcripts.
