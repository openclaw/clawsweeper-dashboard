# Commit ffae8f3

## Summary

Found two low-severity regressions in the xAI malformed-JSON handling change: one newly added test fails in-order because it is served from the shared x_search cache, and one timeout path is now misclassified as malformed JSON.

## Findings

### Low: x_search malformed JSON test is bypassed by the previous test’s cache entry

- Kind: reliability
- File: `extensions/xai/x-search.test.ts`
- Line: 322
- Evidence: The prior test executes `query: "latest post from huntharo"` at `extensions/xai/x-search.test.ts:286`, which writes to the process-global x_search cache. The new malformed-JSON test uses the same query at `extensions/xai/x-search.test.ts:322`. Since `extensions/xai/x-search.ts:198` returns cached payloads before fetching, the malformed `response.json()` mock is never called. Running `node scripts/run-vitest.mjs extensions/xai/web-search.test.ts extensions/xai/x-search.test.ts extensions/xai/code-execution.test.ts` failed exactly this assertion: the promise resolved with a cached result instead of rejecting.
- Impact: The focused xAI test lane is red, and the new regression test does not actually prove malformed x_search JSON handling when run with the surrounding suite.
- Suggested fix: Use a unique query/cache key for the malformed-JSON test, set `cacheTtlMinutes: 0`, or clear the x_search test cache between cases.
- Confidence: high

### Low: response-body aborts are reported as malformed JSON instead of xAI web search timeouts

- Kind: regression
- File: `extensions/xai/src/web-search-shared.ts`
- Line: 113
- Evidence: `requestXaiWebSearch` now calls `readProviderJsonResponse(...)`, whose implementation catches every `response.json()` rejection and throws `<label>: malformed JSON response`. That wrapper runs before `wrapXaiWebSearchError(...)`, so an `AbortError` raised while reading the response body is no longer recognized as a timeout. A focused probe confirmed this path prints `xAI web search failed: malformed JSON response` for a `response.json()` `AbortError`, while the existing timeout wrapper test expects AbortErrors to become `xAI web search timed out after ...`.
- Impact: Slow or aborted xAI response bodies can be surfaced to users as malformed provider JSON instead of the existing timeout guidance.
- Suggested fix: Make `readProviderJsonResponse` rethrow abort/timeout errors before wrapping parse failures, or locally unwrap/rethrow abort causes before `wrapXaiWebSearchError` classifies the error.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The cache/test issue reproduces with the focused Vitest command above. The timeout-classification issue reproduces with a direct `readProviderJsonResponse` plus `wrapXaiWebSearchError` probe using a `response.json()` `AbortError`.
- Is this the best way to solve the issue? yes for the test cache collision. For timeout classification, the safest direction is to narrow malformed-JSON wrapping so abort/timeout failures preserve their original classification.
