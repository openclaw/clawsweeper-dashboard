# Commit 1b62168

## Summary

Found one low-severity incomplete rejection path in the DeepInfra data-URL handling. The focused tests added by the commit pass, but an empty `data:*;base64,` video payload still bypasses the new base64 validator and is returned as a URL-only generated video.

## Findings

### Low: Empty DeepInfra base64 data URLs bypass the malformed-base64 rejection

- Kind: bug
- File: `extensions/deepinfra/video-generation-provider.ts`
- Line: 71
- Evidence: `parseVideoDataUrl` only enters the new `canonicalizeBase64` path when `/^data:([^;,]+);base64,(.+)$/u` matches. For `data:video/webm;base64,`, the `(.+)` group does not match, so `parseVideoDataUrl` returns `undefined`; `extractDeepInfraVideoAsset` then falls through and returns `{ url: normalizedUrl, mimeType: "video/mp4", fileName: "video-1.mp4" }`. A focused `node -e` regex check confirmed this input produces `no data-url match`.
- Impact: A provider response with an empty generated data URL is treated as a successful URL-only video instead of a malformed generated payload. Runtime validation accepts assets with `url`, so the agent/tool path can report a generated video and attach a broken `data:` URL.
- Suggested fix: Make data-URL parsing distinguish “not a data URL” from “malformed data URL”. For example, match with `(.*)` and let `canonicalizeBase64("")` reject, or explicitly throw when `normalizedUrl.startsWith("data:")` but parsing fails.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes: mock DeepInfra `video_url` as `data:video/webm;base64,`; the parser regex misses it before the new validator runs.
- Is this the best way to solve the issue? yes: the fix should stay in `parseVideoDataUrl`/`extractDeepInfraVideoAsset` so all DeepInfra data URL failures are provider-owned errors.
