# Commit 6a65ea8

## Summary

The main reply path now preserves fresh post-compaction token totals, but the queued followup path still runs preflight compaction and then persists stale usage without the new preservation flag. I also found a smaller freshness-contract regression when a “fresh” usage source contains no prompt/context tokens.

## Findings

### Medium: Followup queue still stales fresh post-compaction totals

- Kind: regression
- File: `src/auto-reply/reply/followup-runner.ts`
- Line: 401
- Evidence: `createFollowupRunner` calls `runPreflightCompactionIfNeeded` at `src/auto-reply/reply/followup-runner.ts:256`, but the later `persistRunSessionUsage` call at `src/auto-reply/reply/followup-runner.ts:401` does not pass `preserveFreshTotalTokensOnStaleUsage`. The new `persistSessionUsageUpdate` stale branch only preserves freshness when that flag is true; otherwise it sets `totalTokensFresh=false` even if `incrementCompactionCount` just stored a fresh `tokensAfter` value. A focused probe confirmed `incrementCompactionCount(... tokensAfter: 42000)` followed by stale usage persistence without the flag leaves `totalTokens: 42000` but `totalTokensFresh: false`.
- Impact: queued followup/queue-mode runs can still fall back to transcript re-estimation after every preflight compaction, so the repeated auto-compaction loop fixed for `runReplyAgent` can remain for followup-drained messages.
- Suggested fix: track whether `runPreflightCompactionIfNeeded` changed the session in `followup-runner.ts` and pass `preserveFreshTotalTokensOnStaleUsage: true` to `persistRunSessionUsage` for that turn, matching `agent-runner.ts`. Add a followup-runner regression test.
- Confidence: high

### Low: Fresh usage can be marked fresh with no token snapshot

- Kind: bug
- File: `src/auto-reply/reply/session-usage.ts`
- Line: 160
- Evidence: `hasFreshContextSnapshot` is true for any `lastCallUsage` object, but `deriveSessionTotalTokens` returns `undefined` when there are no positive prompt tokens (`src/agents/usage.ts:262`). The new branch still writes `patch.totalTokensFresh = true`. A focused probe with `lastCallUsage: { output: 7 }` persisted a row with `totalTokensFresh: true` and no `totalTokens`.
- Impact: this violates the session freshness contract. Consumers that look at the raw freshness flag, or status paths that allow non-stale fallback when `totalTokensFresh !== false` (`src/status/status-message.ts:584`), can report accumulated input/output as current context instead of treating the snapshot as unknown.
- Suggested fix: restore the numeric guard: only write `totalTokensFresh=true` when `totalTokens` is a finite positive number; otherwise mark stale or leave the existing value according to the explicit preservation case. Add an output-only `lastCallUsage` test.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The first finding is reproducible by simulating compaction token persistence followed by stale usage persistence without the new flag; the second is reproducible with an output-only `lastCallUsage`.
- Is this the best way to solve the issue? yes for the first finding; the followup path should thread the same preservation state as the main runner. For the second, the safer fix is to restore the previous numeric freshness guard.
