# Commit b180b8ae

## Summary

Found one sanitizer gap: the new `<function_response>` handling strips multiline plain-output blocks, but still leaks same-line plain-output wrappers.

## Findings

### Medium: Same-line `<function_response>` plain output still leaks

- Kind: bug
- File: `src/shared/text/assistant-visible-text.ts`
- Line: 312
- Evidence: `shouldStripStandaloneResult` only strips `function_response` when `isStandaloneOpeningTagLine(...)` is true. That helper requires the character after the opening tag to be a newline or end-of-string, so `<function_response>raw command output</function_response>` is preserved. Focused probe:
  `sanitizeUserFacingText("Before\n<function_response>raw command output</function_response>\nAfter")` returned `"Before\n<function_response>raw command output</function_response>\nAfter"`.
- Impact: A workflow/model that emits function responses on one line still exposes the internal response wrapper and raw tool output in final replies, transcript mirrors, and streaming/user-facing sanitizer paths.
- Suggested fix: Treat line-leading `<function_response>...</function_response>` blocks as strip candidates even when the plain response begins on the same line, while keeping the existing prose guard for examples like `<function_response> is the response wrapper.` Add regression coverage for same-line plain output.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes. The one-line `node --import tsx` sanitizer probe above reproduces the leak on current `main`.
- Is this the best way to solve the issue? Yes. The fix belongs in the shared XML sanitizer so `sanitizeUserFacingText`, assistant extraction, final reply payloads, and transcript mirrors inherit the same behavior.
