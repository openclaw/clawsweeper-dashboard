# Commit f969ae4

## Summary

Found one reliability issue in the new beta fallback path. The happy-path behavior and added tests cover beta resolution/fallback results, but the new fallback installer calls are outside the existing exception boundary.

## Findings

### Medium: Beta fallback installer exceptions can escape the plugin update loop

- Kind: reliability
- File: `src/plugins/update.ts`
- Line: 1094
- Evidence: The main live installer call is wrapped in `try/catch` and converts thrown exceptions into a per-plugin `"error"` outcome at `src/plugins/update.ts:1077`. The new fallback call after a missing beta npm release runs at `src/plugins/update.ts:1094` outside that catch. The same pattern exists for dry-run fallback at `src/plugins/update.ts:915`. `installPluginFromNpmSpec` can throw after metadata resolution, for example during runtime loading, filesystem setup, or process execution paths (`src/plugins/install.ts:1074`, `src/plugins/install.ts:1180`).
- Impact: If `pkg@beta` is missing but the fallback default/latest install hits an exceptional runtime/filesystem/process error, `updateNpmInstalledPlugins` rejects instead of returning a structured plugin outcome. In `openclaw update`, that can abort post-update plugin sync abruptly, skip remaining plugin outcomes, and produce a worse failure than the pre-existing non-beta update path, which caught the same installer exception.
- Suggested fix: Put the fallback npm and ClawHub installer calls inside the same `try/catch` handling as the primary install, or move installer invocation into a small helper that returns `{ ok: false, error }` for thrown fallback exceptions and formats the error against the actual fallback spec.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? yes: use a beta-channel npm install record with a default spec, make the first installer result look like `ETARGET`/missing beta, then make the fallback installer throw. Current code will reject from the fallback call instead of appending an outcome.
- Is this the best way to solve the issue? yes: the existing function already treats primary installer throws as per-plugin failures, so applying the same exception boundary to fallback calls preserves the established contract.
