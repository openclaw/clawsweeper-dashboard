# Commit a9bf94c

## Summary

Found one low-severity CLI behavior bug in the new capture controls. The held-session path honors `--record-fps`, but the one-command `probe` path accepts the same option and then silently ignores it.

## Findings

### Low: `--record-fps` is ignored by one-command Telegram Crabbox proof

- Kind: bug
- File: `scripts/e2e/telegram-user-crabbox-proof.ts`
- Line: 1852
- Evidence: `--record-fps` is parsed into `opts.recordFps` at `scripts/e2e/telegram-user-crabbox-proof.ts:292`, and the held-session recorder uses it in `ffmpeg -framerate ${opts.recordFps}` at `scripts/e2e/telegram-user-crabbox-proof.ts:1317`. The default one-command `probe` path still records through `crabbox artifacts video` at `scripts/e2e/telegram-user-crabbox-proof.ts:1852` and only passes provider, target, id, duration, and output; `opts.recordFps` is never used there.
- Impact: `pnpm qa:telegram-user:crabbox -- --record-fps <n>` appears accepted but does not affect the recording for the quick proof path. Operators trying to tune proof size or match the documented 24fps capture behavior can get a different Crabbox default without any warning.
- Suggested fix: Pass the supported frame-rate option through to `crabbox artifacts video`, or reject `--record-fps` for the one-command path and document that it only applies to held sessions. If `artifacts video` has no FPS control, use the same remote `ffmpeg` recorder path for quick proofs.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Static command construction is enough: the quick proof path builds the `crabbox artifacts video` argv without `opts.recordFps`, while the held-session path uses the value directly.
- Is this the best way to solve the issue? yes, assuming Crabbox supports a video FPS flag. If not, explicit rejection is safer than silently accepting an ignored option.
