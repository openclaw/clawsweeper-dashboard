# Commit 1caf3ae

## Summary

Found one supply-chain issue in the new Mantis workflow tool install path. The basic functional path installs and the focused workflow test passes, but the workflow now executes an npm install script and installs its downloaded binary into a later secret-bearing job path.

## Findings

### Medium: Secret-bearing proof workflow trusts an unsigned runtime ffmpeg download

- Kind: supply_chain
- File: `.github/workflows/mantis-telegram-desktop-proof.yml`
- Line: 342
- Evidence: The workflow runs `pnpm --dir "$media_tools" --config.dangerouslyAllowAllBuilds=true add ffmpeg-static@5.2.0 ffprobe-static@3.1.0`, then `sudo install`s the resulting binaries to `/usr/local/bin/ffmpeg` and `/usr/local/bin/ffprobe`. A focused install smoke with pnpm `10.33.0` showed `ffmpeg-static` executes `node install.js`; the package source downloads the ffmpeg asset from the package’s GitHub release URL, writes it to the package path, and `chmod`s it without a hard-coded checksum verification. The later Codex agent step receives Crabbox, Convex, GitHub, and OpenAI credentials at `.github/workflows/mantis-telegram-desktop-proof.yml:391`, and the prompt tells the agent to use the workflow-provided local `ffmpeg`/`ffprobe`. The proof script then invokes `command: "ffmpeg"` during finish at `scripts/e2e/telegram-user-crabbox-proof.ts:926`.
- Impact: A compromised npm package, compromised release asset, or malicious transitive install-time behavior can place a trojan media binary on the runner before the secret-bearing agent step. That binary would be executed during proof generation with access to the job workspace and later-step environment, so it could exfiltrate credentials or tamper with the generated evidence.
- Suggested fix: Do not run npm lifecycle scripts to fetch proof tooling inside this secret-bearing workflow. Prefer a trusted runner/container image with `ffmpeg` and `ffprobe` preinstalled, or fetch exact static assets with pinned SHA-256 verification before installing them. If npm remains the source, use repo-managed lockfile/frozen resolution and avoid `dangerouslyAllowAllBuilds`; keep any required binary fetch behind an explicit checksum-verifying script.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Running the workflow install command in a temp directory with pnpm `10.33.0` reproduced the lifecycle script execution and produced runnable `ffmpeg`/`ffprobe` binaries.
- Is this the best way to solve the issue? yes for the risk direction. The exact delivery mechanism can be image-based or checksum-based, but the important fix is to remove unsigned runtime package-script media tooling from the secret-bearing job path.
