# Commit 2b01bcf6c8

## Summary

Found one macOS LaunchAgent install regression in the new service env planning refactor.

## Findings

### Medium: launchd drops `.env` credentials when an auth profile references the same env key

- Kind: regression
- File: `src/daemon/service-env-render-policy.ts`
- Line: 36
- Evidence: `src/commands/daemon-install-helpers.ts:451` adds state-dir `.env` entries, then `src/commands/daemon-install-helpers.ts:455` adds auth-profile env refs. `addServiceEnvPlanEntries` stores only one entry per normalized key at `src/daemon/service-env-plan.ts:71`, so an auth-profile ref for `OPENAI_API_KEY` overwrites the prior `state-dotenv` entry. The launchd render policy then only re-adds entries whose last source is `state-dotenv` at `src/daemon/service-env-render-policy.ts:36`, so the managed `.env` value is not written into the LaunchAgent env file.
- Impact: On macOS, users who keep provider keys in state-dir `.env` and also have auth profiles pointing at the same env vars can install a managed Gateway that starts without those provider credentials. The install plan still records `OPENCLAW_SERVICE_MANAGED_ENV_KEYS=OPENAI_API_KEY`, but `OPENAI_API_KEY` itself is absent, so provider auth fails under launchd.
- Suggested fix: Keep per-source plan entries, or make the launchd render policy consult the state-dir `.env` source directly instead of the last writer in `entriesByNormalizedKey`. It should preserve the old behavior: re-add a state `.env` value for launchd when that key is managed and not superseded by config env precedence.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused `node --import tsx -e` check with a temp `HOME`, `.openclaw/.env` containing `OPENAI_API_KEY=dotenv-openai`, `platform: "darwin"`, and an auth profile `keyRef` for `OPENAI_API_KEY` produced `{"managed":"OPENAI_API_KEY"}` with no `value`. The same check without the auth-profile ref produced `{"value":"dotenv-openai","managed":"OPENAI_API_KEY"}`.
- Is this the best way to solve the issue? yes. The bug is in source tracking, not launchd file writing; preserving the state-dotenv source separately avoids weakening the intended precedence rules.
