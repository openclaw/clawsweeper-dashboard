# Commit 480af03

## Summary

Found one concrete compatibility regression in the mirrored Codex tool-result shape. The focused Codex tests pass, but they now assert a `toolResult` content block shape that is outside the upstream `ToolResultMessage` contract and is ignored by existing text-based tool-result consumers.

## Findings

### Medium: Mirrored tool results persist a non-standard content block that drops output in replay consumers

- Kind: compatibility
- File: `extensions/codex/src/app-server/event-projector.ts`
- Line: 1047
- Evidence: `createToolResultMessage()` writes `role: "toolResult"` with `content: [{ type: "toolResult", ..., content: text, text }]`. The upstream `@mariozechner/pi-ai` `ToolResultMessage` contract allows only `(TextContent | ImageContent)[]` for `content`; current replay/truncation code follows that contract. For example, OpenAI Responses replay only reads `msg.content.filter((item) => item.type === "text")` and falls back to `"(see attached image)"` when no text block exists, while `getToolResultTextLength()` counts only `type === "text"` blocks.
- Impact: A Codex transcript mirrored by this commit can later be consumed by generic OpenClaw history paths, model-switch replay, context guards, or tools expecting standard `ToolResultMessage` text blocks. Those paths will see the tool result as having no text, so actual command/browser/MCP output can be lost or undercounted even though the raw JSON contains it.
- Suggested fix: Persist standard tool-result content, e.g. `content: [{ type: "text", text }]`, and keep IDs/tool metadata on the top-level `toolCallId`/`toolName` fields or in `details` if needed. Add a regression test that reloads the mirrored transcript and verifies generic tool-result text extraction or OpenAI Responses replay sees the actual output.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Mirror any Codex tool result from this commit, then pass the resulting `toolResult` message through a consumer that reads standard text blocks, such as `getToolResultTextLength()` or OpenAI Responses replay; the output text is ignored because the only block has `type: "toolResult"`, not `type: "text"`.
- Is this the best way to solve the issue? yes. The narrow fix is to keep the Codex-owned projection but emit the standard `ToolResultMessage` shape so existing transcript consumers continue to work.
