# Commit 212a326

## Summary

Found one regression in the tarball checker: extracting the package and then reading entries via `fs.readFileSync` follows archive symlinks, so a crafted tarball can make CI/local package checks read files outside the tarball or hang on special files.

## Findings

### Medium: Tarball checker follows extracted symlinks outside the package

- Kind: security
- File: `scripts/check-openclaw-package-tarball.mjs`
- Line: 132
- Evidence: This commit changed `readTarEntry()` from `tar -xOf` for a named archive member to reading the extracted filesystem path. `fs.readFileSync(candidate, "utf8")` follows symlinks. I built a tarball where `package/dist/postinstall-inventory.json` is a symlink to a temporary host file containing `SECRET_TOKEN=abc123`; the checker failed with stderr including that content in the JSON parse error. A second probe symlinked the same tar entry to an endless device and `timeout 4s node scripts/check-openclaw-package-tarball.mjs ...` exited `124`, while direct `tar -xOf` for that symlink returned immediately with zero bytes.
- Impact: Package Acceptance and Docker package validation can consume tarballs from npm, URL, or workflow artifacts. A malicious or compromised package candidate can leak readable host-file snippets into logs or hang the package check until the outer job timeout.
- Suggested fix: Avoid reading extracted paths for tar members. Keep selective archive reads, or reject symlinks/hardlinks/special files before reading and open only regular files under a realpath-checked extraction root. Also cleanup the extraction dir in a `finally`.
- Confidence: high
