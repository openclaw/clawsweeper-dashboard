# Commit 310c853

## Summary

Found one concrete provenance-widening bug in the new trusted tool media merge path.

## Findings

### Medium: Trusted tool provenance is applied to pre-existing untrusted reply media

- Kind: security
- File: `src/agents/pi-embedded-runner/run/tool-media-payloads.ts`
- Line: 29
- Evidence: `mergeAttemptToolMediaPayloads` merges existing visible reply media with trusted tool media into one payload at `src/agents/pi-embedded-runner/run/tool-media-payloads.ts:23`, then sets `trustedLocalMedia` on that whole payload at `src/agents/pi-embedded-runner/run/tool-media-payloads.ts:29`. The new flag is passed from the attempt result through `src/agents/pi-embedded-runner/run/attempt.ts:4280` and `src/agents/pi-embedded-runner/run.ts:2519`. WebChat treats `payload.trustedLocalMedia === true` as payload-wide for every media URL when preserving display media and embedding local audio (`src/gateway/server-methods/chat-reply-media.ts:13`, `src/gateway/server-methods/chat-reply-media.ts:23`, `src/gateway/server-methods/chat-webchat-media.ts:88`).
- Impact: if the assistant final reply already contains an untrusted local audio `MEDIA:` path and a trusted TTS/tool audio attachment is pending, the untrusted path inherits trusted provenance. A focused `node --import tsx` check produced one payload with both `named-by-model.mp3` and `tts.opus`, `trustedLocalMedia: true`, and WebChat emitted attachment blocks for both paths. This bypasses the intended distinction from the prior WebChat media hardening: untrusted audio-looking paths should go through staging/read-policy normalization, not be preserved as trusted local media because another attachment was trusted.
- Suggested fix: keep trusted tool media in a separate payload when the first visible reply already has media that is not itself trusted, or track provenance per URL and only set `trustedLocalMedia` when every media URL in the merged payload is trusted.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Use `mergeAttemptToolMediaPayloads({ payloads: [{ text: "done", mediaUrls: [modelNamedAudio] }], toolMediaUrls: [trustedTtsAudio], toolAudioAsVoice: true, toolTrustedLocalMedia: true })`, then pass the result to `buildWebchatAssistantMessageFromReplyPayloads`; both URLs are treated as trusted local audio attachments.
- Is this the best way to solve the issue? yes. Splitting trusted tool media from pre-existing reply media preserves the new TTS/tool behavior without widening the trust label to unrelated assistant-provided media.
