# Commit 37a2538

## Summary

Found one low-severity regression in the Slack status-reaction fix: the tool-only path now works only when `messages.statusReactions.enabled` is explicitly `true`, while Slack’s existing contract says unset should behave as enabled when ack reactions are active.

## Findings

### Low: Default-on Slack status reactions are still skipped in tool-only rooms

- Kind: regression
- File: `extensions/slack/src/monitor/message-handler/prepare.ts`
- Line: 566
- Evidence: `allowToolOnlyStatusReaction` requires `cfg.messages?.statusReactions?.enabled === true`; when the flag is unset and a mentioned Slack channel turn resolves to `message_tool_only`, `shouldSendAckReaction` becomes false. Dispatch then cannot start status reactions because it requires `prepared.ackReactionPromise`. This conflicts with the existing Slack/Discord behavior documented in `src/config/schema.help.ts:1720` and `docs/gateway/config-agents.md:1292`, and with dispatch’s own `enabled !== false` default-on check at `extensions/slack/src/monitor/message-handler/dispatch.ts:308`.
- Impact: users relying on default Slack status reactions still get no lifecycle reaction feedback in the default group/channel tool-only mode unless they explicitly add `messages.statusReactions.enabled: true`.
- Suggested fix: let mentioned or command-bypassed tool-only Slack turns use the same `enabled !== false` status-reaction gate as dispatch, while keeping the existing privacy guard for unmentioned always-on room traffic.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes, by source path: mentioned Slack channel turn, default `messages.groupChat.visibleReplies: "message_tool"`, ack reaction active, `messages.statusReactions` unset.
- Is this the best way to solve the issue? Yes, if the intended Slack contract remains “unset is enabled when ack reactions are active”; otherwise docs/config help need to be changed to require explicit opt-in for this tool-only case.
