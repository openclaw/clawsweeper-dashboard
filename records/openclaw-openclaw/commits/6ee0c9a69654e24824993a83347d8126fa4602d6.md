# Commit 6ee0c9a

## Summary

Found one full-path auth-profile regression: the commit teaches `resolveSessionAuthProfileOverride` that normal OpenAI API-key credentials are valid Codex fallback credentials, but earlier session validators still clear those profiles before that helper can preserve them.

## Findings

### Medium: Codex fallback API-key overrides are still cleared before session override resolution

- Kind: regression
- File: `src/auto-reply/reply/model-selection.ts`
- Line: 323
- Evidence: The new helper in `src/agents/auth-profiles/order.ts:66` returns `true` for `{ provider: "openai-codex", credential: { type: "api_key", provider: "openai" } }`, and `src/agents/auth-profiles/session-override.ts:32` now uses it. But `src/auto-reply/reply/model-selection.ts:319` builds accepted providers for the default OpenAI Codex runtime as only `["openai-codex"]`, then line 323 compares the stored profile’s raw normalized provider (`"openai"`) against that list and calls `clearSessionAuthProfileOverride`. The same raw-provider validator exists in `src/agents/agent-command.ts:881` through `src/agents/agent-command.ts:887`.
- Impact: A user-pinned or fallback-selected normal OpenAI API-key profile for an OpenAI/Codex session can be deleted before `resolveSessionAuthProfileOverride` runs. That defeats the new compatibility rule and can send the next run back to the blocked Codex subscription profile, or silently discard the user’s chosen fallback credential.
- Suggested fix: Reuse `isStoredCredentialCompatibleWithAuthProvider` in both pre-validation paths, or centralize validation so these gates share the same credential compatibility contract as `resolveSessionAuthProfileOverride`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused runtime predicate probe showed default `openai` resolves to Codex runtime, accepted providers become `["openai-codex"]`, profile provider `"openai"` is rejected by the current validators, while the new compatibility helper accepts the same credential.
- Is this the best way to solve the issue? yes. The commit already introduced the correct shared predicate; applying it to the pre-validation gates keeps the contract consistent instead of adding another Codex-specific special case.
