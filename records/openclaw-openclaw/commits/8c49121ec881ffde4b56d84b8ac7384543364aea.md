# Commit 8c49121

## Summary

Found one remaining provider-selection bug in the changed fallback alias guard. The commit fixes the reported OpenCode Go/OpenRouter collision when the requested provider differs from the configured default provider, but the same explicit provider/model can still be rewritten when that explicit provider is also the configured default provider.

## Findings

### Medium: Explicit default-provider selections can still be rewritten through a colliding bare alias

- Kind: bug
- File: `src/agents/model-fallback.ts`
- Line: 594
- Evidence: `resolveFallbackCandidates` now accepts a bare alias if `normalizedPrimary.provider === defaultProvider`. `defaultProvider` is derived from the configured primary model at line 566, so an explicit `opencode-go/deepseek-v4-pro` selection is still treated as alias-eligible whenever the configured primary provider is also `opencode-go`. A read-only probe on current `main` with both `openrouter/deepseek/deepseek-v4-pro` alias `deepseek-v4-pro` and `opencode-go/deepseek-v4-pro` configured, plus `agents.defaults.model.primary = opencode-go/kimi-k2.6`, produced:
  `[{ "provider": "openrouter", "model": "deepseek/deepseek-v4-pro" }, { "provider": "opencode-go", "model": "kimi-k2.6" }]`
- Impact: A fully provider-qualified user selection can still run against the wrong provider, sending the prompt to OpenRouter instead of the explicitly selected OpenCode Go route. This is the same class of wrong-provider behavior the changelog says is fixed, just with the selected provider matching the configured default provider.
- Suggested fix: Preserve whether the original model selection included an explicit provider before fallback resolution, and only allow cross-provider bare-alias resolution when the provider was omitted. If that context is unavailable here, prefer the parsed provider/model candidate unless the alias resolves within the same provider, and keep slash-form alias behavior covered by tests.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. The one-off `pnpm exec tsx -e` probe against `__testing.resolveFallbackCandidates` reproduces the wrong first candidate without live provider credentials.
- Is this the best way to solve the issue? yes. `agent-command.ts` parses explicit `--model provider/model` selections into `provider` and `model` before calling `runWithModelFallback`, so fallback resolution needs either explicitness metadata or stricter alias rules to avoid losing that intent.
