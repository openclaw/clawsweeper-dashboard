# Commit 8c49121

## Summary

Found one concrete edge case where the fix still rewrites an explicitly selected provider/model to another provider when the selected provider is also the configured default provider.

## Findings

### Medium: Explicit selections on the default provider can still be hijacked by a bare alias

- Kind: bug
- File: `src/agents/model-fallback.ts`
- Line: 591
- Evidence: `resolveFallbackCandidates` still accepts a cross-provider bare alias whenever `normalizedPrimary.provider === defaultProvider` (`src/agents/model-fallback.ts:591`). CLI explicit model refs are already split into `provider` and `model` before reaching fallback (`src/agents/agent-command.ts:816`, `src/agents/agent-command.ts:1020`). A source-level check with default primary `opencode-go/kimi-k2.6`, alias `openrouter/deepseek/deepseek-v4-pro -> deepseek-v4-pro`, and explicit candidate `opencode-go/deepseek-v4-pro` returned `{"provider":"openrouter","model":"deepseek/deepseek-v4-pro"}` as the first candidate.
- Impact: Users who set `opencode-go` as their default provider can still explicitly select `opencode-go/deepseek-v4-pro` and run OpenRouter instead when OpenRouter owns the same bare alias. This is the same wrong-provider failure mode as #79325, just with the explicit provider equal to the configured default provider.
- Suggested fix: Preserve whether the original selection was provider-qualified, or pass the raw model selection into fallback resolution. Only apply a cross-provider bare alias for truly bare selections; explicit `provider/model` refs should remain exact unless a full `provider/model` alias intentionally matches. Add a regression where `primary` is `opencode-go/kimi-k2.6`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Source-level reproduction calls `__testing.resolveFallbackCandidates` with default provider `opencode-go`, explicit provider `opencode-go`, model `deepseek-v4-pro`, and the OpenRouter bare alias; it selects OpenRouter first.
- Is this the best way to solve the issue? yes. The failure comes from losing explicitness after parsing; preserving that fact is safer than guessing from `provider === defaultProvider`.
