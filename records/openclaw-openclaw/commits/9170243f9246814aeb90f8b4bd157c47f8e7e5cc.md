# Commit 9170243

## Summary

Found two low-severity CLI regressions in the new `channels list` surface: hidden catalog entries can leak into `--all`, and the removed `--no-usage` flag is still silently accepted by the routed fast path.

## Findings

### Low: `--all` lists hidden QA-only catalog channels

- Kind: bug
- File: `src/commands/channels/list.ts`
- Line: 206
- Evidence: `catalogOnlyLines` includes every catalog entry not already rendered, with no exposure filter, then JSON emits each entry at lines 235-241. `extensions/qa-channel/package.json:43` marks QA Channel exposure as `configured: false`, `setup: false`, and `docs: false`, but `pnpm openclaw channels list --all --json` includes `"qa-channel": { "installed": false, "origin": "installable" }`.
- Impact: `openclaw channels list --all` can advertise internal/synthetic channels that explicitly opted out of setup/docs/configured surfaces, making them look user-installable.
- Suggested fix: filter catalog-only entries through the intended exposure predicate before rendering, likely `isChannelVisibleInSetup(entry.meta)` for installable/configurable catalog rows, and add a regression test for `qa-channel`.
- Confidence: high

### Low: Removed `--no-usage` flag is ignored instead of rejected

- Kind: compatibility
- File: `src/cli/program/route-args.ts`
- Line: 251
- Evidence: `parseChannelsListRouteArgs` always returns `{ json, all }`; `src/cli/program/route-specs.ts:31` treats any truthy parse result as runnable, so Commander never sees unknown flags. `src/cli/channels-cli.ts:119` no longer registers `--no-usage`, but `pnpm openclaw channels list --no-usage --json` exits 0 and prints `{ "chat": {} }`.
- Impact: The commitâ€™s documented migration behavior says stale callers get a clear unknown-option error, but the actual fast path silently accepts the removed flag. That hides stale automation and makes CLI behavior differ from Commanderâ€™s declared contract.
- Suggested fix: make the routed parser validate allowed flags/positionals and return `null` on `--no-usage` or other unknown args so Commander rejects them, or emit the same unknown-option error in the route parser. Add a route test for `--no-usage`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes: `pnpm openclaw channels list --all --json` shows `qa-channel`, and `pnpm openclaw channels list --no-usage --json` succeeds instead of failing.
- Is this the best way to solve the issue? yes for the suggested direction: both bugs are in the new listing/route boundary, not in channel setup or catalog metadata.
