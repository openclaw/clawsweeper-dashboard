# Commit 9ff5250

## Summary

Found one concrete regression risk in the new “native source auth” path: source Codex app-server probes skip OpenClaw auth bridging, but still inherit `OPENAI_API_KEY` / `CODEX_API_KEY` from the OpenClaw process environment.

## Findings

### Medium: source migration probes can still use inherited API-key auth instead of source Codex auth

- Kind: bug
- File: `extensions/codex/src/migration/source.ts`
- Line: 244
- Evidence: `sourceCodexAppServerStartOptions()` sets source `CODEX_HOME` and `HOME`, but does not set `clearEnv` for `OPENAI_API_KEY` or `CODEX_API_KEY`. The source requests then pass `authProfileId: null` at `extensions/codex/src/migration/source.ts:270`. That skips auth-profile login in `applyCodexAppServerAuthProfile()` at `extensions/codex/src/app-server/auth-bridge.ts:301`, but the stdio transport still builds the child environment by copying `process.env` first in `extensions/codex/src/app-server/transport-stdio.ts:52`.
- Impact: A user with source `$CODEX_HOME` logged into ChatGPT subscription auth but with `OPENAI_API_KEY` or `CODEX_API_KEY` exported can have migration readiness evaluated under inherited API-key auth. For app-backed plugins, `account/read` can then appear non-ChatGPT and the plugin is skipped with `codex_subscription_required` / related source-account gating, even though the source Codex home has the needed native auth. This undercuts the commit’s source/destination auth isolation fix.
- Suggested fix: For source migration app-server probes, clear inherited Codex/OpenAI API-key env vars. The narrowest fix is adding `clearEnv: ["CODEX_API_KEY", "OPENAI_API_KEY"]` to `sourceCodexAppServerStartOptions()`, or making the `authProfileId: null` source-auth mode clear those vars before spawn while still preserving explicit `CODEX_HOME` and `HOME`.
- Confidence: medium

## Details

- Do we have a high-confidence way to reproduce the issue? Yes, with a source Codex home logged into ChatGPT auth and an exported `OPENAI_API_KEY`; I did not run that live because this checkout does not have that source credential setup. The static path is direct: source start options lack `clearEnv`, transport inherits `process.env`, and `authProfileId: null` only disables OpenClaw login.
- Is this the best way to solve the issue? Yes. Clearing only API-key auth env vars for source probes preserves the intended native `$CODEX_HOME` account while avoiding target/process auth contamination.
