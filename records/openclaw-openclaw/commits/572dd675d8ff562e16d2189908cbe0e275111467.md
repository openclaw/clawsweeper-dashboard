# Commit 572dd675

## Summary

Found one concrete regression: the new read-side repair for provider-wrapped session overrides also clears any existing session auth-profile override. That can silently move a repaired OpenRouter session from the user-selected profile to the first/default profile, or to no forwarded profile.

## Findings

### Medium: Provider-wrapper repair drops session auth profile overrides

- Kind: regression
- File: `src/sessions/model-overrides.ts`
- Line: 148
- Evidence: `repairProviderWrappedModelOverride` calls `applyModelOverrideToSessionEntry` without passing the existing `authProfileOverride`. The generic helper treats missing `profileOverride` as an instruction to delete `authProfileOverride`, `authProfileOverrideSource`, and `authProfileOverrideCompactionCount` at `src/sessions/model-overrides.ts:100`. The repair is wired before dispatch consumes the stored override at `src/agents/agent-command.ts:782`.
- Impact: a stale entry like `providerOverride: "anthropic"`, `modelOverride: "claude-haiku-4.5"`, `modelProvider: "openrouter"`, `model: "anthropic/claude-haiku-4.5"`, plus `authProfileOverride: "openrouter:work"` is repaired to the right OpenRouter model but loses the user-selected auth profile. Later attempt setup only honors `sessionEntry.authProfileOverride` when it is still present; otherwise it auto-selects from auth order/store in `src/agents/command/attempt-execution.ts:133`. This can use the wrong OpenRouter account/key or fail where only the explicit profile was usable.
- Suggested fix: preserve the existing auth-profile fields during repair and let the existing post-selection auth validation in `agent-command` clear them only if they are incompatible with the repaired provider. A narrow option is to snapshot and restore `authProfileOverride`, `authProfileOverrideSource`, and `authProfileOverrideCompactionCount` around the repair, or add an explicit preserve-auth option to `applyModelOverrideToSessionEntry`.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. A focused `pnpm exec tsx -e` invocation against `repairProviderWrappedModelOverride` showed `updated: true` and the corrected model fields while `authProfileOverride`, `authProfileOverrideSource`, and `authProfileOverrideCompactionCount` became `undefined`.
- Is this the best way to solve the issue? yes. The repair is a state-shape cleanup, not a user request to change credentials. Preserving the profile through the repair keeps the existing validator responsible for clearing incompatible profiles.
