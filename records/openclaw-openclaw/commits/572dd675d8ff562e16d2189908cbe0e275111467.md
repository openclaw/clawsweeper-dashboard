# Commit 572dd675

## Summary

Found one regression: the new default-based repair path can erase a legitimate direct-provider session override when the configured default is a wrapper provider for the same vendor/model id.

## Findings

### Medium: Default-wrapper repair clears legitimate direct-provider overrides

- Kind: regression
- File: `src/sessions/model-overrides.ts`
- Line: 176
- Evidence: `repairProviderWrappedModelOverride` treats any stored override where `defaultModel === "${overrideProvider}/${overrideModel}"` and `defaultProvider !== overrideProvider` as stale, then calls `applyModelOverrideToSessionEntry(... isDefault: true)`, which deletes the user’s `providerOverride` and `modelOverride`. A focused probe on current `main` showed this exact behavior: an entry with `providerOverride: "anthropic"` and `modelOverride: "claude-opus-4-6"` is cleared when the default is `openrouter/anthropic/claude-opus-4-6`, even though direct Anthropic is a documented valid provider/model selection in `docs/providers/anthropic.md:54` and `docs/providers/anthropic.md:64`.
- Impact: users who intentionally pin a session to direct Anthropic while their global default uses OpenRouter for the same Anthropic model will silently lose that session override on the next agent run. The session then routes through the configured wrapper provider instead, which can change credentials, billing, latency, provider policy, and failure behavior.
- Suggested fix: do not repair the default-match shape without additional proof that the override is corrupt. The runtime-field branch has that proof (`modelProvider` plus wrapped `model`); the default-only branch should either require similar persisted runtime evidence, or verify that the direct override provider/model is not a valid registered selection before clearing it. If the state is ambiguous and user-sourced, preserve the explicit override.
- Confidence: high

## Details

- Do we have a high-confidence way to reproduce the issue? yes. Run `repairProviderWrappedModelOverride` with `{ providerOverride: "anthropic", modelOverride: "claude-opus-4-6", modelOverrideSource: "user" }` and defaults `{ defaultProvider: "openrouter", defaultModel: "anthropic/claude-opus-4-6" }`; current code returns `updated: true` and deletes the override.
- Is this the best way to solve the issue? no. The runtime-backed repair is well targeted, but the default-only branch conflates a corrupt OpenRouter override with a valid direct-provider override that happens to match the wrapped default model string.
