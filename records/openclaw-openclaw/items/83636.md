## Summary

Keep open: current main still has a source-level path where trusted dynamic TTS media is queued, then blocked by message-tool-only source delivery before Discord or Telegram channel send is reached.

## What This Changes

The issue reports that successful dynamic TTS tool media is staged but not delivered in Discord/Telegram message-tool-only channel contexts.

## Best Possible Solution

Route trusted dynamic TTS media through an explicit internal deliver-despite-source-suppression path, or equivalent source-media reply, while keeping ordinary final text private.

## Reproduction Assessment

Yes, by source inspection: a successful TTS details.media result is queued as pending tool media, then current message_tool_only suppression returns before block or final media delivery. I did not run the live channel flow in this read-only review.

## Solution Assessment

No, the current implementation does not satisfy the existing TTS auto-delivery contract. The narrow maintainable fix is to mark or route dynamic trusted TTS media as an internal deliverable payload without making ordinary final text public.
