## Summary

Keep open: the report matches a current-main source path where TTS tool media is queued, then suppressed by message-tool-only source delivery before Discord or Telegram send is reached.

## What This Changes

Ensure successful dynamic TTS tool media is delivered to Discord/Telegram-originating message_tool_only channel contexts instead of being suppressed with private final replies.

## Best Possible Solution

Route trusted dynamic TTS media through an explicit internal deliver-despite-source-suppression path, or equivalent message-tool media reply, while keeping ordinary final text private.

## Reproduction Assessment

Yes, by source inspection: a successful tts details.media result is queued as pending tool media, then current message_tool_only suppression returns before block/final media delivery. I did not run the live channel flow in this read-only pass.

## Solution Assessment

No, the current implementation does not satisfy the existing TTS auto-delivery contract. The narrow maintainable fix is to mark or route dynamic TTS media as an internal deliverable payload without making ordinary final text public.
