## Label Justifications

- `impact:message-loss`: The successful TTS tool result stages audio but the originating channel never receives the media send.

## Summary

Keep open: current main still queues trusted TTS media but routes it through block/final reply paths that are suppressed in `message_tool_only` channel contexts, matching the reporter's Discord and Telegram failure evidence.

## What This Changes

Deliver successful dynamic TTS tool media to originating Discord/Telegram channels even when normal final replies are message-tool-only.

## Best Possible Solution

Route only trusted dynamic TTS media through an explicit internal suppression-bypass or equivalent source-media delivery path, while preserving private ordinary final text in message-tool-only channel turns.

## Reproduction Assessment

Yes. The issue has two concrete runtime traces, and source inspection shows successful TTS media is queued, then block/final delivery is suppressed before a Discord or Telegram send is reached.

## Solution Assessment

No, the current implementation does not satisfy the existing TTS auto-delivery contract. The narrow fix is to make trusted dynamic TTS media deliverable despite source-reply suppression without making ordinary final text public.
