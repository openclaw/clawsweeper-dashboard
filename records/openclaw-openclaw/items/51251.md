## Summary

Keep open: current main and v2026.6.1 partially address the silent part with status/docs and auto-fallback cleanup, but explicit and legacy session `modelOverride` values are still intentionally preserved across `/new`/`/reset`, so the remaining behavior needs a maintainer product decision rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to stop session-level model overrides from silently shadowing updated config defaults across restarts and `/new`, or to provide clear visibility and reset controls.

## Best Possible Solution

Choose the intended contract for explicit session model selections, then align `/new`/`/reset`, `/model default` or any bulk reset command, status wording, docs, and regression tests with that contract.

## Reproduction Assessment

Yes, from source inspection: current reset code and tests preserve user or legacy `modelOverride` entries while clearing only auto fallback pins. I did not run a live gateway repro in this read-only review.

## Solution Assessment

Unclear: the current auto-vs-user split is a deliberate partial fix, but it does not settle whether `/new` should mean a clean model default or preserve an explicit session pin. A maintainer needs to choose that product contract before a narrow implementation is safe.
