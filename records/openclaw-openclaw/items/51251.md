## Summary

Keep open; current main partially addresses the silent UX by clearing auto-sourced fallback pins and surfacing pinned-session status, but explicit and legacy session model selections still intentionally survive restart, `/new`, and `sessions.reset`, so the remaining behavior needs a maintainer model-selection contract decision.

## What This Changes

The issue requests safer handling and visibility for persisted session `modelOverride` state after `/model`, `/new`, `/reset`, gateway restart, and config default changes.

## Best Possible Solution

Define one source-aware model-selection contract for explicit, legacy, and auto session selections, then align reset, status, doctor, `models set`, and any clear or reconcile command with that contract.

## Reproduction Assessment

Yes, at source level. Seed a session with a user or legacy `modelOverride`, change the configured default, then `/new` or `sessions.reset` still preserves that selection while `resolveSessionModelRef` continues to prefer it over the new default.

## Solution Assessment

Unclear. Unconditionally clearing explicit pins conflicts with current docs and tests, so the maintainable path is a maintainer-approved source contract plus explicit clear, migrate, reconcile, or status/doctor UX.
