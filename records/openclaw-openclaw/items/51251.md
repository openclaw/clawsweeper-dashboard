## Label Justifications

- `P2`: The issue affects model routing correctness and UX for existing sessions, but it is not an outage, security bypass, or data-loss emergency.
- `impact:session-state`: The core problem is persisted session model state remaining active after config-default changes and reset flows.
- `impact:auth-provider`: The stale state controls effective provider/model routing, not only display text.

## Summary

Keep open: current main and v2026.5.22 still preserve user and legacy session model overrides ahead of changed config defaults, while only auto-fallback pins are cleared and visibility has improved. The remaining behavior needs a maintainer policy decision before a safe fix PR.

## What This Changes

The issue asks OpenClaw to stop stale session-level model overrides from silently winning over changed config defaults, especially across gateway restart and `/new` or `/reset`, and to add clearer reset/visibility paths.

## Best Possible Solution

Define one source-aware model-selection contract across `/model`, `/new`, `/reset`, `models set`, status, doctor, and any inherit/reset command, then update code, docs, and focused tests to match that chosen policy.

## Reproduction Assessment

Yes, at source level. A session entry with `modelOverride` or legacy override state still resolves before `resolveDefaultModelForAgent`, and `/new`/`/reset` preserve user or legacy pins through `resolveResetPreservedSelection`.

## Solution Assessment

Unclear. The current partial solution improves visibility and clears auto pins, but maintainers still need to choose whether explicit or legacy session pins should survive reset, restart, and default-model changes.
