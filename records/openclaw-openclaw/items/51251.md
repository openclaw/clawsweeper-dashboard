## Summary

Keep open: current main and the latest release still preserve user and legacy session model pins across reset/restart, while only auto fallback pins are cleared; the remaining behavior needs a maintainer policy decision because changing it could break intentional per-session model selections.

## What This Changes

The issue asks OpenClaw to stop stale session-level `modelOverride` state from silently outranking changed config defaults, especially after gateway restart and `/new`.

## Best Possible Solution

Define one source-aware model-selection contract for explicit, legacy, auto, and default-derived session model state, then align `/new`, `/reset`, `/status`, `session_status(model=default)`, doctor/docs, and any clear/inherit command with that contract.

## Reproduction Assessment

Yes, at source level. Seed a session with a user or legacy `modelOverride`, change the configured default, then `/new` or `sessions.reset` preserves that selection while `resolveSessionModelRef` prefers it over the new default.

## Solution Assessment

Unclear. Unconditionally clearing explicit pins conflicts with current docs, release behavior, and tests, so the maintainable path is a maintainer-approved source contract plus explicit clear, migrate, reconcile, or status/doctor UX.
