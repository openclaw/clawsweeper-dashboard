## Summary

Keep open: current main and the latest release include useful visibility and auto-fallback cleanup, but user and legacy session model pins still intentionally survive restart and reset, so the remaining behavior needs a maintainer model-selection policy decision.

## What This Changes

Request that session-level model overrides stop silently masking changed config defaults, especially across gateway restarts and `/new`, with clear visibility and reset behavior.

## Best Possible Solution

Define one source-aware model-selection contract for explicit, legacy, auto, and runtime session model state, then align reset, status, doctor, `models set`, and any clear or inherit command with that contract.

## Reproduction Assessment

Yes, at source level. Seed a session with a user or legacy `modelOverride`, change the configured default, then `/new` or `sessions.reset` still preserves that selection while `resolveSessionModelRef` continues to prefer it over the new default.

## Solution Assessment

Unclear. Unconditionally clearing explicit pins conflicts with current docs, release behavior, and tests, so the maintainable path is a maintainer-approved source contract plus explicit clear, migrate, reconcile, or status/doctor UX.
