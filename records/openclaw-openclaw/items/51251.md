## Summary

Keep open. Current main intentionally preserves explicit and legacy session-level model overrides across resets/restarts, while only auto fallback overrides are cleared; `/status` still shows the selected model without labeling whether it came from config or a session override, so the remaining issue is a real UX/product contract decision rather than a cleanup close.

## What This Changes

The issue asks OpenClaw to stop stale session-level `modelOverride` state from silently masking changed config defaults across gateway restarts and new sessions, or to make that state visible and clearable.

## Best Possible Solution

Define the session model override contract explicitly, then align reset/restart behavior, `/status` source labeling, and clear/inherit controls with that contract.

## Reproduction Assessment

Yes. Source and tests give a high-confidence path: store a session with user or legacy `modelOverride`, change the config default, then use `/new`, `/reset`, `sessions.reset`, or restart and observe that the stored override still has precedence.

## Solution Assessment

No. Current main is a partial solution for auto fallback stickiness, but it does not resolve the UX problem of explicit or legacy session overrides masking config changes without source labeling or a clear/inherit workflow.
