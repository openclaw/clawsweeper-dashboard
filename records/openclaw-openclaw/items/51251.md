## Summary

Keep open. Current main partially fixes auto-sourced fallback stickiness, but it deliberately preserves explicit and legacy session model overrides through resets/restarts and `/status` still does not label the model source, so the remaining behavior needs a maintainer UX/product decision rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to prevent or expose stale session-level `modelOverride` state after `/model`, config default changes, gateway restarts, and `/new`, including a clear or inherit workflow.

## Best Possible Solution

Define the session model override contract explicitly, then align reset/restart persistence, `/status` source labeling, and clear/inherit commands with that contract.

## Reproduction Assessment

Yes. Source and tests provide a high-confidence path: create a stored session with `modelOverrideSource: "user"` or no source, then trigger `/new` or `sessions.reset` and observe that the override is preserved while only `"auto"` overrides are cleared.

## Solution Assessment

No. Current main is only a partial solution for auto fallback stickiness; the remaining request needs a maintainer decision on whether explicit or legacy pins should survive and how users should see or clear them.
