## Summary

Keep open: current main and v2026.6.6 still preserve explicit or legacy session model pins across `/new` and Gateway resets, while only auto-fallback pins are cleared. The remaining request is a compatibility-sensitive model/session-state decision rather than fully implemented cleanup.

## What This Changes

The issue asks OpenClaw to stop stale session-level model overrides from silently taking precedence after default-model changes, especially across restarts and `/new`, or provide clear reset/visibility tooling.

## Best Possible Solution

Maintain a source-aware model selection contract: keep intentional session pins visible, provide an explicit clear/reset path if maintainers want one, and cover `/new`, Gateway reset, status, docs, and upgrade behavior with focused tests.

## Reproduction Assessment

Yes, source-reproducible: current main's reset helper preserves user/legacy `modelOverride`, `/new` and `sessions.reset` call that helper, and tests lock the explicit-vs-auto split. I did not run a live gateway restart reproduction.

## Solution Assessment

Unclear without maintainer product direction: automatically clearing explicit `/model` pins on restart or `/new` is not clearly the best fix. A safer solution is an explicit source-aware clear/reset UX plus status/docs/tests that preserve intentional selections unless maintainers choose a contract change.
