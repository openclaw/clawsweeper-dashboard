## Summary

Keep open. Current main intentionally preserves user and legacy session model overrides while clearing only auto-sourced fallback pins, and `/status` still shows the selected model without its source, so the remaining request needs maintainer UX/product judgment rather than cleanup closure.

## What This Changes

The issue asks for session-level `/model` pins to stop silently shadowing updated config defaults across restarts and resets, or at least to be visible and clearable.

## Best Possible Solution

Define the override contract explicitly, keep intentional pins understandable, then align `/status` source labeling, clear/inherit UX, and reset/restart persistence with that contract.

## Reproduction Assessment

Yes. Source and tests give a high-confidence reproduction path: create a stored session with `modelOverrideSource: "user"` or no source, run `/new` or `sessions.reset`, and the override remains while only `"auto"` overrides are cleared.

## Solution Assessment

Unclear. Automatically making config defaults beat explicit session overrides is not clearly the best fix because current docs treat user and legacy overrides as exact selections; a safer maintainer path is source labeling plus an explicit clear/inherit mechanism.
