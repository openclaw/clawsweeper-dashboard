## Summary

Keep open. Current main intentionally preserves explicit and legacy session-level model overrides through resets and restarts, and the remaining problem is a real UX/product gap around visibility, clear/inherit semantics, and whether `/new` should keep user model pins.

## What This Changes

The issue asks OpenClaw to prevent stale session-level `modelOverride` values from silently masking updated config defaults, especially after gateway restart and `/new`.

## Best Possible Solution

Define the session model override contract explicitly, then make reset/restart behavior, `/status` source labeling, and clear or inherit commands match that contract consistently.

## Reproduction Assessment

Yes. Source and tests give a high-confidence reproduction path: create a stored session with user or legacy `modelOverride`, then trigger `/new`, `/reset`, or `sessions.reset` and observe that the override is preserved while only auto-sourced overrides are cleared.

## Solution Assessment

No. Current main is only a partial solution for auto fallback stickiness; it does not resolve the remaining UX problem of explicit or legacy session overrides masking changed config defaults without clear source visibility.
