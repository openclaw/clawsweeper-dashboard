## Summary

Keep open: current main and v2026.6.6 still deliberately preserve explicit and legacy session model overrides across `/new`, `/reset`, and gateway reset; the silent-status part now has better visibility and `/model default`, but the requested reset/default-precedence behavior remains a compatibility-sensitive product decision.

## What This Changes

The issue requests clearer and resettable handling for persisted session-level model overrides after config default changes, gateway restarts, and session resets.

## Best Possible Solution

Keep the current source-aware contract unless maintainers choose a new policy: explicit pins stay visible and clearable, auto fallback state clears on reset, and any bulk reset or default-precedence change should be designed with docs, status, doctor, and upgrade tests together.

## Reproduction Assessment

Yes, source-reproducible: current main preserves user/legacy `modelOverride` through `/new` and Gateway reset, and model resolution prefers that persisted selection over configured defaults. I did not run a live gateway restart reproduction.

## Solution Assessment

Unclear without maintainer product direction: clearing explicit `/model` pins automatically is not obviously the best fix because current docs treat them as intentional exact selections. A safer path is source-aware visibility plus explicit clear/reset UX unless maintainers choose to change the reset contract.
