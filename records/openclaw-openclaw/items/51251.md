## Summary

Keep open; current main partially addresses the silent UX by clearing auto fallback pins and showing pinned-session status, but it still intentionally preserves explicit and legacy session model selections, so the remaining reset/restart/bulk-clear behavior needs a maintainer model-selection contract decision.

## What This Changes

The issue asks for stale session `modelOverride` state to stop silently winning over changed config defaults, especially across gateway restarts, `/new`, and reset flows.

## Best Possible Solution

Define one source-aware model-selection contract for explicit, legacy, and auto session selections, then align reset, status, doctor, and clear/reconcile commands to that contract.

## Reproduction Assessment

Yes, at source level. Seed a session with a user or legacy `modelOverride`, change the configured default, then `/new` or `sessions.reset` still preserves that selection while only auto-sourced fallback pins are cleared.

## Solution Assessment

Unclear. Unconditionally clearing explicit pins conflicts with current docs and tests, so the maintainable path is a maintainer-approved source contract plus explicit clear, migrate, or reconcile UX.
