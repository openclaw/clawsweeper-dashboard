## Summary

Keep open. Current main intentionally preserves user and legacy session model overrides while clearing only auto fallback overrides, so the remaining request is a model-override UX/product decision rather than a cleanup-close candidate.

## What This Changes

The issue asks for session model overrides set by `/model` to stop silently shadowing updated config defaults across restarts and `/new`, or to become clearly visible and easy to clear.

## Best Possible Solution

Define the session model-override contract explicitly, then align reset/restart persistence, status source labeling, and explicit clear/reset tooling with that contract.

## Reproduction Assessment

Yes. Source and tests give a high-confidence reproduction path: create a stored session with `modelOverrideSource: "user"` or no source, run `/new` or `sessions.reset`, and the override remains while only `"auto"` overrides are cleared.

## Solution Assessment

No. Current main is a partial, intentional source-aware policy for auto fallback stickiness; the remaining behavior needs a maintainer decision on explicit/legacy pin persistence, status wording, and clear/reset surfaces.
