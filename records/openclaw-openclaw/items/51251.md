## Summary

Keep open. Current main partially addresses auto-fallback stickiness, but explicit `/model` selections and legacy `modelOverride` entries still intentionally survive `/new`, `/reset`, `sessions.reset`, and gateway restarts, while `/status` still does not label the model as a session override versus a config default.

## What This Changes

The issue asks for session-level model overrides set by `/model` to stop silently masking updated config defaults across gateway restarts and new sessions, or at least become visible and clearable.

## Best Possible Solution

Decide the user-facing model override contract explicitly, then implement it consistently across `/new`, `/reset`, `sessions.reset`, `/status`, docs, and a clear/reset command path. A conservative end state would preserve intentional pins but make them visible in `/status` and provide documented `/model clear` or `/model inherit` plus a bulk session-store cleanup path.

## Reproduction Assessment

Yes. The current source and tests provide a high-confidence reproduction path: create a stored session with user or legacy `modelOverride`, run `/new`/`/reset` or `sessions.reset`, and observe that the override is preserved while auto-sourced overrides are cleared.

## Solution Assessment

No. The current implementation is only a partial fix for auto fallback overrides; it does not resolve the report's main UX problem for explicit or legacy session overrides masking config defaults.
