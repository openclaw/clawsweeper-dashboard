## Summary

Keep open: current main and v2026.6.6 intentionally preserve explicit or legacy session model pins across restarts and resets, while only auto-created fallback pins are cleared. Visibility and per-session clear paths now exist, but the remaining bulk-clear/config-precedence behavior is a compatibility-sensitive product decision with related open work.

## What This Changes

The issue asks for session-level `/model` overrides not to silently survive restarts and `/new`/`/reset`, plus clearer status output and a way to clear overrides.

## Best Possible Solution

Keep the current explicit-pins-are-user-selection contract unless maintainers deliberately choose a UX change; any change should go through the canonical session model/auth state resolver with status/docs/tests and a clear upgrade story.

## Reproduction Assessment

No live gateway restart reproduction was run. Source and tests show the current behavior: explicit or legacy `modelOverride` values are preserved across `/new`/`sessions.reset`, while auto-created fallback overrides are cleared.

## Solution Assessment

Unclear: automatically clearing explicit `/model` pins on restart or `/new` is not currently the best fix without maintainer product direction. The safer end state is an explicit clear/reset UX and any broader precedence change handled through the central session model/auth resolver.
