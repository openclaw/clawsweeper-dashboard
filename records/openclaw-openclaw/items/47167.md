## Label Justifications

- `P2`: The request affects plugin visibility for skipped channel messages but does not report a core outage.
- `impact:message-loss`: Skipped group messages are intentionally not delivered to plugin message hooks, preventing passive archive plugins from recording them.
- `impact:session-state`: The requested archive visibility affects what historical conversation state plugins can preserve for later retrieval.
- `impact:security`: An early passive inbound hook would expose pre-gating message content to plugins, so access-control and data exposure boundaries matter.

## Summary

Keep this open: current main and v2026.6.6 still lack the requested early read-only plugin hook, and the existing shipped `ingest` path is only a partial internal-hook workaround. The same-author implementation PR remains open and conflicting, so the remaining action is maintainer review of that plugin API/security boundary, not cleanup closure.

## What This Changes

The issue requests a read-only early inbound plugin hook so archive plugins can observe group messages skipped by `requireMention` without creating sessions or replies.

## Best Possible Solution

Keep archive behavior plugin-owned, and land or replace the linked PR with a narrow read-only plugin hook that preserves current `message_received` semantics and clearly documents the security boundary.

## Reproduction Assessment

Yes at source level: the Telegram `requireMention` skip path returns before the shared `message_received` plugin hook, and current hook types/docs do not include an early raw/observed inbound plugin hook. I did not run a live channel repro in this read-only review.

## Solution Assessment

Yes, pending maintainer approval: a separate read-only observation hook is a better fit than moving `message_received` earlier because it avoids changing existing plugin hook semantics. The open PR is the concrete path to review, but the API and security boundary still need maintainer sign-off.
