## Summary

Keep open. Current main already has the Control UI provider/model picker behavior, but it does not have the proposed top-level `llm` config shorthand, and the PR patch has source-reproducible config merge and validation blockers.

## What This Changes

The PR adds a top-level `llm` `openclaw.json` shorthand that expands into custom model provider/default-model config and changes the Control UI chat model picker to send qualified `provider/model` refs.

## Best Possible Solution

Decide whether OpenClaw should support this shorthand at all; if yes, rebase it onto the current config/model-picker architecture, preserve existing model objects/fallbacks, validate ref-safe ids and provider APIs, and add generated schema/docs/changelog plus focused config tests.

## Reproduction Assessment

The feature request itself is not a bug reproduction, but the patch defects have high-confidence source paths: existing fallback-only model config is overwritten, reused provider ids can create dangling primary refs, and slash-containing provider ids parse incorrectly.

## Solution Assessment

No. The UI direction is already handled on current main, but the config shorthand implementation is not yet the narrowest maintainable solution because it adds public config surface without preserving existing defaults or validating values against the model-ref/API contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve fallbacks when seeding llm primary:** `src/config/defaults.ts:591`
  - body: If `agents.defaults.model` is already an object with `fallbacks` but no `primary`, this assignment replaces the entire model object with `{ primary }`. Adding the shorthand then silently removes the user's fallback chain, contradicting the non-destructive behavior described by the new helper.
  - confidence: 0.91
- **[P2] Avoid dangling primary refs for existing providers:** `src/config/defaults.ts:574-576`
  - body: When `models.providers[providerId]` already exists, the new provider entry and its `llm.model` are dropped, but the code can still set `agents.defaults.model.primary` to that model ref. A config with provider `deepseek` containing only `deepseek-reasoner` plus `llm.model: deepseek-chat` would point the primary at a model that was never added.
  - confidence: 0.9
- **[P2] Reject provider ids that cannot form model refs:** `src/config/zod-schema.core.ts:271`
  - body: `provider_id` is used as the provider key and then embedded in `${providerId}/${modelId}`. Values containing `/` pass validation here, but runtime parsing splits on the first slash, so `foo/bar/baz` resolves as provider `foo` and model `bar/baz` instead of the injected `models.providers["foo/bar"]`.
  - confidence: 0.88
- **[P2] Validate llm provider values before casting:** `src/config/zod-schema.core.ts:266`
  - body: The new schema accepts any non-empty `llm.provider`, then `applyLlmConfig()` casts it into `ModelProviderConfig["api"]`. Typos such as `openai-compatable` therefore survive config validation and create provider config with an API value the model runtime does not recognize; this should be constrained to the two shorthands plus the existing `MODEL_APIS`.
  - confidence: 0.86
