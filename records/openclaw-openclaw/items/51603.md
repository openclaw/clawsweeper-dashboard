## Summary

The branch still contains a unique root `llm` config feature that is not on current main, so it should not be closed as obsolete; however the picker slice has been superseded by merged main work and the config slice has blocking correctness, config-contract, secret-surface, and proof gaps.

## What This Changes

The PR adds a root `llm` shorthand config field that expands into model provider/default config and changes the Control UI model picker to submit provider-qualified model refs.

## Best Possible Solution

If maintainers want this shorthand, rebuild it as a focused current-main config PR with strict validation, non-destructive merging, generated schema/help/docs, secret-surface coverage, and tests; otherwise rely on the existing `models.providers` plus `agents.defaults` contract and the already-merged picker fix.

## Reproduction Assessment

Yes for the PR defects by source inspection: the missing import, fallback overwrite, dangling model-ref path, validation gaps, and missing secret-surface coverage are deterministic in the diff. The root `llm` shorthand itself is a new feature, so there is no current-main bug reproduction path for that request.

## Solution Assessment

No. The picker problem is already solved on current main by https://github.com/openclaw/openclaw/pull/61382, and the remaining shorthand should be a current-main config/security design change rather than this mixed stale branch.

- **[P1] Import the primary model resolver before using it:** `src/config/defaults.ts:588`
  - body: `applyLlmConfig()` calls `resolveAgentModelPrimaryValue`, but this branch never imports that symbol from `src/config/model-input.ts`. The config build/typecheck will fail before the new shorthand can run.
  - confidence: 0.96
- **[P1] Preserve existing model defaults when seeding primary:** `src/config/defaults.ts:591`
  - body: When `agents.defaults.model` already has `fallbacks` or `timeoutMs` but no `primary`, this assignment replaces the whole object with `{ primary }`, silently dropping configured fallback behavior on every load.
  - confidence: 0.93
- **[P2] Add the shorthand model before pointing primary at it:** `src/config/defaults.ts:572-574`
  - body: If `models.providers[providerId]` already exists, the branch preserves that provider unchanged but can still set `agents.defaults.model.primary` to `providerId/modelId`. When the preserved provider lacks that model, the default points at a model that cannot resolve.
  - confidence: 0.91
- **[P2] Reject provider IDs that cannot form model refs:** `src/config/zod-schema.core.ts:270`
  - body: `provider_id` is later used as the left side of a `provider/model` ref, but the schema allows values containing `/`. A value like `foo/bar` is stored under one provider key while model-ref parsing resolves it as provider `foo`.
  - confidence: 0.89
- **[P2] Restrict llm.provider to supported API values:** `src/config/zod-schema.core.ts:265`
  - body: `llm.provider` accepts any non-empty string and is cast into `ModelProviderConfig["api"]`. Typos pass validation and reach downstream provider selection as unsupported API ids instead of failing at config load.
  - confidence: 0.87
- **[P2] Require base_url before materializing a provider:** `src/config/zod-schema.core.ts:268`
  - body: `base_url` is optional, but `applyLlmConfig()` materializes `baseUrl: ""` when omitted even though provider config expects a real endpoint for custom providers. A schema-valid `llm` block can create an invalid runtime provider.
  - confidence: 0.84
- **[P2] Cover llm.api_key in secret surfaces:** `src/config/zod-schema.core.ts:267`
  - body: The PR adds credential-bearing `llm.api_key`, but only registers the Zod field. It needs the same target-registry, configure/apply, audit, redaction, generated help, and credential-surface coverage as `models.providers.*.apiKey`.
  - confidence: 0.82
- **[P3] Add the public config contract docs:** `src/config/types.openclaw.ts:95-96`
  - body: A new root config key needs generated schema/help and user-facing configuration docs so operators and config tooling have a stable contract for the shorthand.
  - confidence: 0.76
