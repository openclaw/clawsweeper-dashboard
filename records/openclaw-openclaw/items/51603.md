Keep open; the branch is not mergeable as-is because the picker half is superseded by merged current-main work, while the remaining root shorthand is a new credential-bearing config feature with deterministic source-level defects and missing real behavior proof.

## What This Changes

The branch adds a root `llm` config shorthand for custom model providers and changes the Control UI model picker to send provider-qualified model refs.

## Best Possible Solution

Keep the shipped picker implementation; if maintainers want a root `llm` shorthand, land it as a rebased config/security design change in current materialization with strict validation, SecretRef coverage, docs, tests, and real proof.

## Reproduction Assessment

Yes for the PR defects by source inspection: the dangling primary ref, destructive defaults overwrite, loose validation, optional base URL, and missing secret-surface coverage are deterministic in the branch. The root shorthand itself is a new feature, not a current-main bug reproduction.

## Solution Assessment

No. The picker fix already landed in https://github.com/openclaw/openclaw/pull/61382, and the remaining shorthand should be a rebased config/security design PR rather than this mixed stale branch.

Full review comments:

- **[P1] Preserve existing model defaults when seeding primary:** `src/config/defaults.ts:591`
  - body: When `agents.defaults.model` already has `fallbacks`, `timeoutMs`, or other model-default fields but no `primary`, assigning `{ primary: modelRef }` replaces the whole object and drops configured failover behavior during config load.
  - confidence: 0.93
- **[P2] Add the shorthand model before setting primary:** `src/config/defaults.ts:574-576`
  - body: If `models.providers[providerId]` already exists, this preserves that provider unchanged but still points `agents.defaults.model.primary` at `providerId/modelId`; when the preserved provider lacks that model, the default cannot resolve.
  - confidence: 0.91
- **[P2] Reject provider IDs that cannot form model refs:** `src/config/zod-schema.core.ts:271`
  - body: `provider_id` is later used as the left side of a `provider/model` ref, but the schema allows values containing `/`; those parse as a different provider/model pair at runtime and miss the injected provider entry.
  - confidence: 0.89
- **[P2] Restrict llm.provider to supported API values:** `src/config/zod-schema.core.ts:266`
  - body: `llm.provider` accepts any non-empty string and is cast into `ModelProviderConfig["api"]`, so typos pass config validation and fail later in provider selection instead of being rejected at load time.
  - confidence: 0.87
- **[P2] Require base_url before materializing a provider:** `src/config/zod-schema.core.ts:269`
  - body: `base_url` is optional, but `applyLlmConfig()` materializes `baseUrl: ""`; the normal provider schema requires a non-empty base URL, so a schema-valid `llm` block can create an invalid provider.
  - confidence: 0.84
- **[P2] Cover llm.api_key in secret surfaces:** `src/config/zod-schema.core.ts:268`
  - body: The PR adds credential-bearing `llm.api_key`, but it does not add target-registry, configure/apply, audit, generated docs, or equivalent handling for that credential path comparable to `models.providers.*.apiKey`.
  - confidence: 0.82
