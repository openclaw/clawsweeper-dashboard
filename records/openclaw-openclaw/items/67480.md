## Summary

Keep open: current main still has the nested skills-root bug, and this PR is useful but not ready because the new probe is direct-only, scans the configured root without the existing cap, and omits the required user-facing changelog entry.

## What This Changes

The PR adds an early-exit root-validity probe to skills discovery, passes the skill-file byte cap into that probe, and adds a regression test for a direct skill beside a nested `skills/` decoy.

## Best Possible Solution

Revise this PR so nested-root detection first proves the configured directory has a loadable direct or one-level grouped skill using the same containment, size, and scan-limit rules as normal discovery, then add the changelog entry and regression coverage for both layouts.

## Reproduction Assessment

Yes. Current main can be reproduced with a configured skills root containing `my-skill/SKILL.md` plus `skills/decoy/SKILL.md`; the resolver redirects to `dir/skills` first, so the root skill is skipped. A grouped-root variant with `group/my-skill/SKILL.md` plus `skills/decoy/SKILL.md` reproduces the remaining gap in this PR.

## Solution Assessment

No. Validating the configured root before redirecting is the right direction, but the current diff does not mirror documented grouped discovery, bypasses the directory scan cap, and lacks the required changelog entry.
