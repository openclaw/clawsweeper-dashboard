## Summary

Keep open: current main already clears stale per-session context windows when a model override changes, but the reported stale DeepSeek V4 `contextWindow: 200000` config is still a plausible current-main path because explicit configured windows are treated as authoritative.

## What This Changes

The issue reports that context usage stays near the old model's percentage after switching from MiniMax M2.7 to DeepSeek V4 Flash, with the local DeepSeek model metadata showing a 200K window instead of 1M.

## Best Possible Solution

Keep the shipped model-switch cache clearing, and add a narrow `openclaw doctor --fix` repair for known stale DeepSeek V4 200K model metadata so official 1M catalog values are restored without changing intentional `contextTokens` caps.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: a DeepSeek V4 config row with `contextWindow: 200000` is returned as the model limit and then used by the UI context notice. I did not live-run the Windows/npm setup.

## Solution Assessment

No, closing as fully implemented would miss the reported stale local config path. The maintainable fix is a doctor/config repair for known stale official DeepSeek V4 metadata, not a runtime override of all explicit user settings.
