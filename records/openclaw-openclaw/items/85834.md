## Summary

Keep open. Current main already clears stale per-session context-token caches when the selected model changes and the bundled DeepSeek V4 catalog says 1,000,000 tokens, but source inspection still shows lower explicit DeepSeek model `contextWindow` values can be preserved despite docs saying built-in catalog metadata should win, which can keep the context usage percentage wrong for legacy configs.

## What This Changes

The issue requests that the context usage UI recompute against the newly selected model's context window after switching models, specifically from MiniMax M2.7 to DeepSeek V4 Flash.

## Best Possible Solution

Keep the current model-switch cache clearing, then make legacy or lower DeepSeek V4 `contextWindow` metadata converge to the bundled 1M catalog through a documented merge/doctor path while preserving explicit `contextTokens` caps.

## Reproduction Assessment

Yes, at source level: the UI percent comes from `session.contextTokens`, and current config/merge policy can preserve a stale explicit DeepSeek V4 `contextWindow` below the bundled 1M catalog value. I did not establish a live Windows UI repro in this read-only pass.

## Solution Assessment

Unclear. Clearing session cached context tokens on model switch is already the right behavior, but the remaining fix needs a maintainer decision on merge precedence versus a targeted `openclaw doctor --fix` legacy-config repair.
