## Summary

Keep this PR open: the shared-loader fix is still useful for the linked Telegram filename bug, but the current patch rewrites any local filename that happens to match the media-store suffix pattern and still lacks real transport proof.

## What This Changes

The PR changes `loadWebMediaInternal` so local outbound media filenames are derived with `extractOriginalFilename` instead of the raw basename.

## Best Possible Solution

Land one shared-loader fix that strips the suffix only for media-store-backed paths, preserves ordinary local filenames, adds focused regression coverage, and includes redacted live transport proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: media-store paths can be staged as `name---uuid.ext`, current main forwards the local basename as `fileName`, and Telegram passes that name into `InputFile`; I did not run a live Telegram delivery repro.

## Solution Assessment

No, not as written. The shared loader is the right layer, but the decoder needs to be scoped to media-store-backed paths and backed by regression plus live transport proof before this is the best fix.
