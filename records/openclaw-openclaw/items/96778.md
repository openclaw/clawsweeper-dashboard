## Summary

Keep this PR open: it targets a valid linked Telegram/media filename bug at the shared media boundary, but the patch reimplements the existing media-store filename extractor with broader semantics and only provides test/mock proof instead of real Telegram behavior proof.

## What This Changes

The PR adds a filename-stripping helper in `src/media/web-media.ts` and applies it to `loadWebMediaInternal` filenames before channel delivery.

## Best Possible Solution

Reuse `extractOriginalFilename` at the shared `loadWebMediaInternal` filename boundary, keep the storage ID format unchanged, add focused regression coverage, and provide redacted live Telegram proof.

## Reproduction Assessment

Yes, from source inspection: media-store saves original names as `name---uuid.ext`, `loadWebMediaInternal` currently returns the staged basename, and Telegram uses that value as the `InputFile` filename. I did not run live Telegram delivery in this read-only review.

## Solution Assessment

No: the shared media boundary is the right layer, but this patch is not the best shape because it duplicates and broadens the already-tested `extractOriginalFilename` behavior. The safer fix is to reuse that helper and cover the `loadWebMedia` path directly.
