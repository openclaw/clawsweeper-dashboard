## Summary

Keep open. Current main still codifies a native-first Jiti default while the published Discord plugin package is source-only and its native-command source keeps the `.js` sibling import pattern that fails under Node native TypeScript loading.

## What This Changes

Discord npm installs should avoid crash-looping when source-loaded native command modules import `.js` sibling specifiers on Node versions with native TypeScript stripping.

## Best Possible Solution

Make plugin loader Jiti options path-aware so source TypeScript loads and their dynamic imports stay on the Jiti transform path, while retaining native loading only for already-built JavaScript artifacts.

## Reproduction Assessment

Yes at source level. I did not run the live gateway, but the current source, the published Discord tarball, and the Jiti dependency path line up with the reported Node 22.6+ `.js` to `.ts` resolution failure.

## Solution Assessment

Yes, with a narrow variant of proposed option A. Route `buildPluginLoaderJitiOptions` through resolved/path-aware `tryNative` behavior or default source loaders to `false`, and keep explicit native loading for built JavaScript only.

## Repair Work Prompt

Fix openclaw/openclaw issue #76555: Discord npm source installs can crash-loop on Node 22.6+ when `native-command.ts` imports `./native-command.args.js` through a Jiti loader configured with `tryNative: true`. Keep the change scoped to plugin loader/source-module Jiti behavior; do not add config flags, release/version bumps, dependency overrides, or broad package publishing changes. Prefer making `buildPluginLoaderJitiOptions` accept or derive path-aware `tryNative` behavior so source `.ts` loads and nested dynamic imports stay on Jiti's transform path, while preserving the native fast path for built `dist/*.js` artifacts where current tests require it. Add regression coverage for a source TypeScript plugin/Discord runtime module that imports a `.js` sibling backed only by `.ts`, update tests that currently assert the hardcoded `tryNative: true`, and add a single-line user-facing `CHANGELOG.md` fix entry. Validate with targeted plugin loader/channel loader/Discord tests and then the changed gate in Testbox per repo policy.
