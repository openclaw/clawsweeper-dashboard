## Summary

Keep this issue open. Current main still has the native-first jiti default in `buildPluginLoaderJitiOptions`, and the published Discord package for the reported release ships TypeScript source with `.js` sibling specifiers, so the reported Node/jiti failure mode has not been ruled out or fixed on main.

## What This Changes

The issue asks OpenClaw to stop Discord channel npm installs from crash-looping when source TypeScript is loaded through jiti with native loading enabled.

## Best Possible Solution

Keep compiled JavaScript artifacts on the native fast path, but make every source TypeScript plugin/channel load use jiti's transform path and add a regression test around a Discord-style `.js` sibling import from `.ts` source.

## Reproduction Assessment

No high-confidence live gateway reproduction was run in this read-only pass. Source inspection does show the failing ingredients: a source-only Discord npm package, `.js` specifiers pointing at `.ts` siblings, and native-enabled jiti paths that can bypass jiti's source resolver.

## Solution Assessment

Unclear until patched, but the narrowest maintainable direction is to route `buildPluginLoaderJitiOptions` or its channel-plugin callers through `resolvePluginLoaderTryNative` for the actual module path instead of leaving source loads able to inherit `tryNative: true`. Shipping prebuilt Discord dist would also work, but it is broader packaging work.
