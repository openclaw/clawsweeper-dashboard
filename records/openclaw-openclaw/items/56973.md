## Summary

Keep open. Current main still has the reported naming collision: both channel registries export `normalizeChannelId`, and current callers still alias those imports to recover the canonical meanings. The cleanup is valid, but the safest fix should preserve the existing plugin SDK compatibility alias instead of removing a public export outright.

## What This Changes

The issue requests removing ambiguous exported `normalizeChannelId` wrappers and updating callers to use `normalizeChatChannelId` or `normalizeAnyChannelId` directly.

## Best Possible Solution

Use canonical internal names everywhere: export/import `normalizeChatChannelId` for bundled chat IDs and `normalizeAnyChannelId` for registered plugin channel IDs, remove unused internal wrappers, and keep any public plugin SDK `normalizeChannelId` compatibility alias deprecated until a versioned breaking-removal path exists.

## Reproduction Assessment

Yes. A static source check against current main reproduces the issue: both exported wrappers remain, and at least two current callers alias both imports to distinguish the return types.

## Solution Assessment

Partly. The requested mechanical rename is the right direction for internal code, but outright public removal is not the best fix because `openclaw/plugin-sdk/channel-targets` and `channel-runtime` currently expose `normalizeChannelId`; preserving a deprecated alias is the safer path.
