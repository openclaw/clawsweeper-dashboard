## Summary

Keep open. Current main still has both ambiguous exported normalizeChannelId wrappers and the reported aliased imports, while the plugin-facing name is still part of the published plugin SDK surface and contract tests, so this needs maintainer compatibility direction rather than cleanup closure.

## What This Changes

Remove ambiguous normalizeChannelId wrapper exports and switch internal callers to canonical normalizeChatChannelId and normalizeAnyChannelId names.

## Best Possible Solution

Use canonical internal imports for normalizeAnyChannelId and normalizeChatChannelId while retaining or formally deprecating the public SDK normalizeChannelId alias until maintainers choose a compatible removal window.

## Reproduction Assessment

Yes. Source inspection on current main shows both exported wrappers and the aliased imports in sessions-send-helpers.ts; no runtime scenario is needed to reproduce the naming collision.

## Solution Assessment

Unclear. Canonicalizing internal imports is a good cleanup, but removing the plugin-facing wrapper outright is not the safest solution while published SDK subpaths and contract tests still expose that symbol.
