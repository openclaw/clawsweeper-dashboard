## Summary

Keep open. Current main and the latest release still contain both exported `normalizeChannelId` wrappers and the reported aliasing pattern, while the plugin-facing wrapper is also exposed through a public plugin SDK subpath that needs maintainer compatibility direction before removal.

## What This Changes

The issue requests replacing ambiguous `normalizeChannelId` wrapper exports with existing canonical channel-normalization names.

## Best Possible Solution

Canonicalize internal imports to `normalizeChatChannelId` and `normalizeAnyChannelId`, while retaining or formally deprecating the public SDK `normalizeChannelId` alias until maintainers choose a compatible removal window.

## Reproduction Assessment

Yes. Source inspection of current main shows both exported wrappers and the aliased imports in `src/agents/tools/sessions-send-helpers.ts`; no runtime scenario is needed to reproduce the naming collision.

## Solution Assessment

Unclear as written. Internal canonicalization is the right cleanup direction, but outright public wrapper removal is not the safest path while `openclaw/plugin-sdk/channel-targets` and its contract test still expose that symbol.
