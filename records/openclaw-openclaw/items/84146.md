## Summary

Keep open. The transport error-message fix is source-backed and useful, but this external PR still only supplies unit-test proof and also carries a second cron string-null behavior change with compatibility-sensitive provider failover implications.

## What This Changes

The branch changes transport stream finalization to throw `output.errorMessage` for errored or aborted streams, adds unit coverage, and also maps cron `agentId`/`sessionKey` string `"null"` inputs to `null` with tests.

## Best Possible Solution

Land a scoped fix that preserves useful provider error text with a non-empty generic fallback, and either split the cron string-null change or explicitly cover it with proof and maintainer acceptance.

## Reproduction Assessment

Yes, by source inspection. Current main hardcodes the finalizer error while the same transport output shape carries `errorMessage`, and the cron normalizer handles JSON null but preserves string `"null"`.

## Solution Assessment

Partly. Threading non-empty `output.errorMessage` is the right narrow direction, but the PR should preserve a generic fallback for blank values and either split or fully prove the cron sentinel behavior before merge.
