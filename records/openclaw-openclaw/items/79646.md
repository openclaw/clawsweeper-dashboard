## Summary

Keep open: current main still closes blockquote style spans after `paragraph_close` appends the paragraph separator, so Telegram native blockquote HTML can include the reported trailing blank line. The related open newline and blockquote-prefix items do not supersede this exact span-boundary bug.

## What This Changes

The issue asks OpenClaw to stop Telegram blockquote spans from including the paragraph separator emitted inside a quoted paragraph.

## Best Possible Solution

Fix shared Markdown IR span boundaries so container blockquote styles exclude the separator between the quote and following block, then add focused IR and Telegram formatter regressions.

## Reproduction Assessment

Yes, by source inspection. The current `markdownToTelegramHtml` path for `> `gpt`\n\nbody` reaches `paragraph_close` before `blockquote_close`, and the current source closes the style after the separator has been appended; I did not run a live Telegram send in this read-only review.

## Solution Assessment

Yes on direction, with care around the exact patch. The narrow maintainable fix belongs in the shared Markdown IR span boundary logic, not Telegram post-processing, and should preserve current list and multi-block spacing semantics.

Fix https://github.com/openclaw/openclaw/issues/79646 on current main. The observable bug is that Telegram HTML blockquotes include the trailing paragraph separator emitted by `paragraph_close` when a blockquote is followed by another block, e.g. `> `gpt`\n\nbody` renders with the blank line inside `<blockquote>`. Keep the fix in shared Markdown IR/span-boundary logic, not as Telegram-only HTML string post-processing. Add focused regression coverage proving the blockquote style span excludes the separator and `markdownToTelegramHtml("> `gpt`\n\nbody")` closes `</blockquote>` before the separator/body.
