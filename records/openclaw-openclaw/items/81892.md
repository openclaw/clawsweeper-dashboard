## Summary

Current `main` still has a clear Matrix delivery gap: reasoning payloads are produced when reasoning is enabled, but Matrix final delivery drops `isReasoning` replies and the Matrix handler does not wire the available reasoning-stream callbacks. The report is recent, source-reproducible, and not superseded by the older inverse leak fix.

## What This Changes

The issue asks Matrix to deliver reasoning blocks when `/reasoning on` is enabled, instead of behaving the same as `/reasoning off`.

## Best Possible Solution

Implement Matrix reasoning delivery through the existing reply/reasoning callback contracts, making `/reasoning on` produce a separate visible reasoning message while preserving `/reasoning off` suppression and treating `/reasoning stream` according to Matrix’s non-token-streaming constraints.

## Reproduction Assessment

Yes from source inspection and the reporter’s concrete Matrix steps: current code creates `isReasoning` replies for `reasoningLevel: "on"`, then Matrix drops them. I did not perform a live Matrix reproduction in this read-only sweep.

## Solution Assessment

Yes: the narrow fix is in the Matrix channel delivery path, using existing reasoning payloads or stream callbacks. A new channel-level config option is not the best first fix because the generic `/reasoning` contract already carries the user’s intent.
