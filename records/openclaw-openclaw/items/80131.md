## Summary

Keep open: this is a recent, concrete gateway performance report, and current main only partially covers the implicated paths. The Google Vertex service-account auth path is still owned by open Google Vertex work, while tool construction/schema normalization still runs on the embedded-attempt path and needs fresh profiling before a safe fix is chosen.

## What This Changes

Request to reduce gateway TTFT by caching/warming Google Vertex service-account auth and memoizing static plugin tool loading/schema serialization across requests.

## Best Possible Solution

Split this into measured perf work: credential-type-aware Google Vertex ADC/token caching with rotation-safe keys and warmup, plus stable tool descriptor/provider payload caching with explicit invalidation and benchmark proof.

## Reproduction Assessment

No high-confidence live reproduction was run for the exact timing numbers. Source inspection does confirm the relevant current-main auth, tool construction, and schema-normalization paths remain on the request/attempt path.

## Solution Assessment

Unclear. The caching direction is plausible, but the best fix needs maintainer design around service-account versus authorized-user ADC, credential rotation, plugin hot reload, dynamic tool context, and provider payload semantics.
