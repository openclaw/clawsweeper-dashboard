## Summary

Keep open: current main has partial tool-descriptor caching and authorized-user Vertex token caching, but it does not cover the reported service-account Vertex auth path or all per-request tool normalization/bundling work.

## What This Changes

The issue requests reducing gateway time-to-first-token by caching Google Vertex service-account auth resolution and static plugin tool bundling work across requests.

## Best Possible Solution

Split this into measured performance work: credential-type-aware Vertex service-account auth caching/warmup, stable provider/tool payload caching with explicit invalidation, and benchmark coverage against a heavy plugin-tool setup.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection does confirm the current request path still resolves Vertex headers through an authorized-user-only helper and still rebuilds/normalizes/wraps tool definitions per embedded attempt.

## Solution Assessment

Unclear. The caching direction is plausible, but the best fix needs maintainer design around service-account versus authorized-user credentials, token rotation, plugin reload, request-scoped tool context, and provider-specific payload caching.
