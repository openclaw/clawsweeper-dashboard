## Summary

Keep open: current main contains partial work in this area, but it does not fully cover the reported service-account Vertex auth path or the provider tool-bundling/serialization costs called out in the profiling report.

## What This Changes

Requests reducing gateway TTFT by caching Google Vertex auth resolution and memoizing plugin/tool definition work instead of rebuilding it on each embedded request.

## Best Possible Solution

Split this into measured perf work: credential-type-aware Vertex runtime auth caching/warmup, stable tool descriptor/provider payload caching with explicit invalidation, and focused regression plus benchmark proof.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection does confirm the relevant request-time paths still build/normalize tools and convert provider tool declarations, while the exact latency numbers need the reported Mac/Vertex setup or a targeted benchmark.

## Solution Assessment

Unclear. The proposed caching direction is plausible, but the best fix needs maintainer design around credential rotation, service-account versus authorized-user ADC behavior, plugin hot reload, dynamic tool context, and provider payload semantics.
