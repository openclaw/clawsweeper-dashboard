## Summary

Keep open: current main has partial related caching, but it does not terminally fix the reported Google Vertex service-account auth latency or the repeated tool construction, schema normalization, and bundle-prep costs.

## What This Changes

The issue requests reducing gateway TTFT by caching Google Vertex service-account auth and repeated plugin/tool construction or bundling work between embedded requests.

## Best Possible Solution

Split this into measured performance work: service-account-aware Vertex auth reuse/warmup, stable tool descriptor/provider payload caching with explicit invalidation, and focused benchmark/regression proof.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection confirms the current-main paths still resolve Vertex headers and rebuild/normalize tool surfaces per embedded attempt, but the reported latency numbers need the same Mac/Vertex service-account or a targeted benchmark setup.

## Solution Assessment

Unclear. The caching direction is plausible, but the best fix needs maintainer design around service-account versus authorized-user credentials, plugin reload invalidation, dynamic tool context, provider payload semantics, and fresh perf proof.
