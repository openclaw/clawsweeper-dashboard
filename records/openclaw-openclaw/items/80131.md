## Summary

Keep open. Current main still shows the reported auth preparation, plugin-tool construction, and bundled-tool materialization on the embedded-run request path, while the existing caches are narrower than the Vertex service-account and whole-tool-bundle behavior requested here.

## What This Changes

The issue requests reducing gateway time-to-first-token by caching or prewarming Vertex auth and avoiding repeated plugin/tool bundle construction per embedded run.

## Best Possible Solution

Capture current-main substage timings for the reported Vertex and heavy-tool setup, then land scoped auth prewarming/token reuse and tool/bundle materialization caching only where invalidation and per-request policy semantics are preserved.

## Reproduction Assessment

No high-confidence live reproduction was run. The reporter supplied repeated v2026.5.7 timings, and current source confirms the same auth and tool/bundle preparation surfaces remain on the embedded-run request path.

## Solution Assessment

Unclear. Singleton auth and module-scope tool memoization are plausible directions, but the maintainable fix should be driven by fresh profiling and scoped cache boundaries rather than a broad cache that could bypass runtime auth, plugin reload, or tool policy state.
