## Summary

Keep open: current main has partial caching work in the same area, but it still does not cover the reported Google Vertex service-account auth path or the repeated provider/tool payload preparation costs described by the profiler.

## What This Changes

The issue asks OpenClaw to reduce gateway time-to-first-token by caching Google Vertex service-account auth and static tool descriptor/provider payload work across requests.

## Best Possible Solution

Split this into measured perf work: credential-type-aware Vertex runtime auth caching/warmup, stable tool descriptor/provider payload caching with explicit invalidation, and focused benchmark proof on current main.

## Reproduction Assessment

No, we do not have a high-confidence live reproduction of the reported latency numbers. Source inspection does confirm the relevant request-time auth and tool-preparation paths still exist, but the exact timings need the reported Mac/Vertex/heavy-plugin setup or a targeted benchmark.

## Solution Assessment

Unclear. The proposed caching direction is plausible, but the best fix needs maintainer design around credential rotation, service-account versus authorized-user ADC behavior, plugin hot reload, dynamic tool context, and provider payload semantics.
