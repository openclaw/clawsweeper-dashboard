## Summary

Keep open: current main still has the placeholder-only Telegram document fallback, and this PR is a useful narrow direction, but the proposed patch still loses fallback details in multi-entry debounce bursts and is missing the required user-facing changelog entry.

## What This Changes

The PR changes Telegram inbound media resolution and body rendering to surface document download fallback reasons, with focused resolver and body-format tests.

## Best Possible Solution

Land a revised Telegram plugin fix that carries both the document placeholder and unavailable reason through resolver, body rendering, single-entry debounce, and multi-entry debounce paths, with focused regression coverage and a changelog entry.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. Source inspection shows current main returns `null` for failed document `getFile` paths and has no body option for an unavailable reason, so a failed inbound document can render as `<media:document>` without explanation.

## Solution Assessment

Is this the best way to solve the issue? No, not as currently proposed. The resolver/body strategy is the narrow maintainable direction, but the patch must preserve fallback details through multi-entry debounce bursts and add the required changelog entry.
