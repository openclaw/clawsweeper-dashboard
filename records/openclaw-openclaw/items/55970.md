## What This Changes

The PR changes Telegram failed-document media resolution and body rendering so failed document downloads can preserve `<media:document>` and append a file-too-large or generic download-failed reason.

## Best Possible Solution

Land or replace a rebased Telegram plugin fix that carries document unavailable reasons through resolver, single and multi-entry debounce, media-group, reply-context, and body-rendering paths, with focused regression tests plus redacted real Telegram proof.

## Reproduction Assessment

Yes, source-level. Current main returns `null` for failed Telegram document `getFile` calls, and the PR's pathless fallback object can still lose its reason through multi-entry debounce, media-group, and reply-media paths without a live account.

## Solution Assessment

No, not as currently proposed. The resolver/body direction is narrow and maintainable, but the implementation must cover every handler path that consumes `resolveMedia` before it can solve the linked Telegram bug.

Overall confidence: 0.88

Full review comments:

- **[P2] Propagate fallback reasons through debounce bursts:** `extensions/telegram/src/bot-handlers.runtime.ts:1089`
  - body: The patch stores `mediaUnavailableText` on queued entries, but multi-entry `onFlush` still builds options from message id and timing only. A text or caption plus failed document has no saved media path, so it is debounce-eligible; when it coalesces with another message, the synthetic message reaches `resolveTelegramInboundBody` without the fallback reason and the ambiguous attachment failure returns.
  - confidence: 0.9
- **[P2] Forward media-group fallback reasons:** `extensions/telegram/src/bot-handlers.runtime.ts:403`
  - body: Media-group handling now skips pathless document fallback objects with `media?.path`, but it only increments skipped media and never carries the new unavailable reason into the final message options. Failed documents in a media group can still reach the agent as skipped or ambiguous media instead of the explicit reason this PR promises.
  - confidence: 0.83
- **[P2] Keep fallback objects out of reply media:** `extensions/telegram/src/bot/delivery.resolve-media.ts:297`
  - body: `resolveMedia` can now return a truthy object with no `path`, so all consumers must guard on `media?.path` before constructing `TelegramMediaRef`. The rebased PR needs the same guard for reply-chain media or replied-to failed documents can produce undefined media paths in reply context.
  - confidence: 0.78
