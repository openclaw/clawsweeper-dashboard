## Summary

Keep this PR open. Current main still lacks the Telegram document-unavailable reason, and the PR is a useful narrow fix, but the proposed patch still has a correctness gap where the new fallback reason can be lost during multi-entry inbound debounce flushing.

## What This Changes

The PR changes Telegram inbound document media resolution and message body rendering so failed document downloads keep the document placeholder and append a file-too-large or download-failed reason, with focused resolver/body tests.

## Best Possible Solution

Revise the Telegram extension patch so failed inbound document downloads produce a structured unavailable reason, body rendering preserves user text plus <media:document> plus that reason, and both single-entry and multi-entry debounce paths intentionally propagate or combine the reason with focused tests.

## Reproduction Assessment

Yes. The static path is high-confidence: with the PR shape, a text plus failed-document entry has no allMedia and is debounce-eligible; if another same-key entry joins the flush, the multi-entry processMessage options omit mediaUnavailableText, so the final body loses the explanation.

## Solution Assessment

No, not as currently proposed. The direction is the narrowest maintainable fix for the user-visible ambiguity, but it is incomplete until debounce batching preserves the new unavailable-reason metadata.
