## Summary

Keep this PR open: current main still has the placeholder-only Telegram document failure behavior, and the PR's resolver/body direction is useful but incomplete because debounce bursts can still drop the new unavailable reason and the required changelog entry is missing.

## What This Changes

The PR changes Telegram inbound document failure handling to preserve `<media:document>`, append a file-too-large or generic download-failed reason, and add focused resolver/body tests.

## Best Possible Solution

Land a revised Telegram plugin fix that carries the structured document-unavailable reason from media resolution through final body rendering and every inbound debounce path, with focused single-entry and multi-entry tests plus an Unreleased changelog entry.

## Reproduction Assessment

Yes. A high-confidence seam reproduction is to make Telegram `getFile` fail for an inbound document with either the Bot API file-too-large error or exhausted retries, then build the inbound body and observe that current main only has placeholder fallback without a reason.

## Solution Assessment

No, not as currently proposed. The resolver/body approach is narrow and maintainable, but it must propagate or merge the unavailable reason in multi-entry debounce flushes before it fully solves the reported behavior.
