## Summary

Keep open: current main still lacks unavailable-reason plumbing, and this PR head adds the right resolver/body direction but remains blocked by propagation gaps plus missing real Telegram proof.

## What This Changes

The PR changes Telegram media resolution and body context so failed inbound document downloads can preserve `<media:document>` and append a file-too-large or download-failed reason, with focused tests.

## Best Possible Solution

Land a rebased Telegram plugin fix that carries document-unavailable reasons through resolver, body rendering, debounce, and media-group paths, backed by focused tests and redacted real Telegram proof.

## Reproduction Assessment

Yes, source-reproducible: on current main, a Telegram document whose getFile reports file-too-large or exhausts retries returns null, which leaves the handler with an empty media list and no unavailable-reason option. I did not run a live Telegram transfer in this read-only review.

## Solution Assessment

No, not as currently proposed. The resolver/body approach is the right narrow direction, but the patch must preserve the reason through multi-entry debounce and media-group handling and include real Telegram proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Propagate fallback reasons through debounce bursts:** `extensions/telegram/src/bot-handlers.runtime.ts:1092`
  - body: The new `mediaUnavailableText` is stored on queued entries, but multi-entry flushes build synthetic options without merging `entry.options`. A caption/text plus failed document has no saved media and is debounce-eligible, so batching it with another update drops the unavailable reason this PR is meant to surface.
  - confidence: 0.91
- **[P2] Forward media-group fallback reasons:** `extensions/telegram/src/bot-handlers.runtime.ts:406`
  - body: This branch now ignores failed document results that only carry `unavailableText`, and the later media-group `processMessage` call still passes no options. A failed document in that path can therefore reach body rendering without the explicit fallback reason.
  - confidence: 0.82
