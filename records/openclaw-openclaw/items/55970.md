## Summary

Keep open. The PR is a useful narrow direction for the linked Telegram attachment bug, but it is still dirty against main, lacks real Telegram proof, and source review shows the new unavailable reason can still be dropped on debounced, grouped, and reply-context paths.

## What This Changes

The PR changes Telegram inbound document media resolution and body rendering so failed document downloads can preserve `<media:document>` and append a file-too-large or download-failed reason.

## Best Possible Solution

Land or replace a rebased Telegram plugin fix that carries document unavailable reasons through resolver, single and multi-entry debounce, media-group, reply-context, and body-rendering paths, with focused regression tests plus redacted real Telegram proof.

## Reproduction Assessment

Yes. Source inspection shows current main returns null for failed Telegram document getFile calls, and the PR's pathless fallback object can still lose its reason through multi-entry debounce, media-group, and reply-media paths without needing a live account to see the code path.

## Solution Assessment

No, not as currently proposed. The resolver/body direction is narrow and maintainable, but the implementation must cover every handler path that consumes resolveMedia before it can solve the linked Telegram bug.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Propagate fallback reasons through debounce bursts:** `extensions/telegram/src/bot-handlers.runtime.ts:1089`
  - body: The patch stores `mediaUnavailableText` on queued entries, but multi-entry `onFlush` still builds options from `messageIdOverride` and timing only. A text or caption plus failed document has no saved media and is debounce-eligible, so batching it with another entry drops the unavailable reason this PR promises to surface.
  - confidence: 0.91
- **[P2] Forward media-group fallback reasons:** `extensions/telegram/src/bot-handlers.runtime.ts:403`
  - body: When a document inside a media group hits the new fallback return, `media?.path` is false, so the handler counts it as skipped and calls `processMessageWithReplyChain` without any `mediaUnavailableText`. Grouped failed documents can therefore still reach the agent without the explicit reason this PR is adding.
  - confidence: 0.88
- **[P2] Keep fallback objects out of reply media:** `extensions/telegram/src/bot/delivery.resolve-media.ts:300`
  - body: This change makes `resolveMedia` return a truthy object without `path`, but the reply-chain resolver still treats any truthy result as a saved `TelegramMediaRef`. Replying to a failed document can push `path: undefined` into reply media instead of omitting it or carrying a clear unavailable reason.
  - confidence: 0.82
