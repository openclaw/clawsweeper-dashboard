## Summary

Keep this PR open, but it is not merge-ready: the intended Telegram fallback wording is useful, yet the current patch still drops the new reason in debounce and media-group paths and has only test-level proof rather than real Telegram behavior proof.

## What This Changes

The PR changes Telegram inbound document failure handling to preserve `<media:document>` and append a file-too-large or download-failed reason, with focused resolver/body tests.

## Best Possible Solution

Land a rebased Telegram plugin fix that carries document-unavailable reasons through resolver, body rendering, debounce, and media-group handling, backed by focused tests and redacted real Telegram proof.

## Reproduction Assessment

Yes, source-reproducible: mock a Telegram document whose ctx.getFile rejects with `file is too big` or exhausts retries; current main returns null and has no mediaUnavailableText path. I did not run a live Telegram transfer in this read-only review.

## Solution Assessment

No, not as currently proposed. The resolver/body direction is narrow, but this patch still loses the reason through debounce and media-group paths and lacks real Telegram proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

- **[P2] Propagate fallback reasons through debounce bursts:** `extensions/telegram/src/bot-handlers.runtime.ts:1092`
  - body: The new `mediaUnavailableText` is stored on queued entries, but multi-entry flushes build synthetic options without merging `entry.options`. A caption/text plus failed document has no saved media and is debounce-eligible, so batching it with another update drops the unavailable reason this PR is meant to surface.
  - confidence: 0.91

  - body: This branch now ignores failed document results that only carry `unavailableText`, and the later media-group processing call still passes no options. A failed document in that path can therefore reach body rendering without the explicit fallback reason.
  - confidence: 0.82
