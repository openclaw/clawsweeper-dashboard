## Summary

Keep open: the PR targets a valid Telegram document fallback bug, but the current patch still drops the new fallback reason in debounce burst and media-group paths, and the external PR has only test-based proof rather than after-fix real Telegram behavior proof.

## What This Changes

The PR adds Telegram document download fallback text plumbing, resolver/body tests, and retry tests so failed inbound documents keep `<media:document>` plus a file-too-large or download-failed reason.

## Best Possible Solution

Land a rebased Telegram plugin fix that carries unavailable reasons through resolver, debounce, media-group, and body rendering paths, backed by focused tests and redacted real Telegram proof.

## Reproduction Assessment

Yes, source-reproducible: mock a Telegram document whose `ctx.getFile()` rejects with `file is too big` or exhausts retries; current main returns `null` and existing retry coverage expects that null fallback. I did not run a live Telegram transfer in this read-only review.

## Solution Assessment

No, not as currently proposed. The resolver/body direction is narrow and maintainable, but the patch still loses the reason through debounce burst and media-group paths and lacks after-fix real Telegram proof.

  - body: The patch stores `mediaUnavailableText` on queued entries, but multi-entry debounce flushes still synthesize options without merging `entry.options`.   - confidence: 0.91

  - confidence: 0.82
