## Summary

Keep open. The PR targets a real Telegram document fallback bug, but the proposed patch still drops the new unavailable reason in burst and media-group paths, and the external PR has only focused test claims rather than real after-fix behavior proof.

## What This Changes

The PR changes Telegram inbound media resolution, message-context rendering, and focused tests so failed document downloads can append a file-too-large or download-failed reason while keeping the `<media:document>` placeholder.

## Best Possible Solution

Land a revised Telegram plugin fix that carries document-unavailable reasons through resolver, body rendering, single-entry debounce, multi-entry debounce, and media-group handling, with focused tests and real redacted behavior proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: make Telegram `getFile` fail for a document with `file is too big` or exhausted retries, then current main returns no media and can render only placeholder text without a reason.

## Solution Assessment

No, not as currently proposed. The resolver/body strategy is the right narrow direction, but the patch must preserve the reason through multi-entry debounce and media-group paths and provide real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve fallback reasons in debounce bursts:** `extensions/telegram/src/bot-handlers.runtime.ts:1089`
  - body: The new `mediaUnavailableText` is stored on each debounce entry, but only the single-entry flush spreads entry options. A caption/text plus failed-document message is debounceable because it has text and no media path; if it batches with another entry, the multi-entry synthetic message calls `processMessageWithReplyChain` without the stored option, so the final body again loses the unavailable reason.
  - confidence: 0.91
- **[P2] Carry document fallback reasons through media groups:** `extensions/telegram/src/bot-handlers.runtime.ts:403`
  - body: This guard now skips a failed document result because it has no `path`, but `processMediaGroup` never forwards the result's `unavailableText` to `processMessageWithReplyChain`. Telegram albums can contain documents, so a failed document in a media group still reaches body rendering without the explicit unavailable reason this PR is meant to add.
  - confidence: 0.82
