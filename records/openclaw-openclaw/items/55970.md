## Summary

Keep open. The PR targets a real, source-reproducible Telegram document fallback bug, but the current branch still drops the new fallback reason in debounce and media-group paths and lacks required after-fix real Telegram proof.

## What This Changes

The PR adds Telegram document download unavailable-reason plumbing and focused tests so failed inbound documents can render `<media:document>` plus file-too-large or download-failed text.

## Best Possible Solution

Land a rebased Telegram plugin fix that carries document-unavailable reasons through resolver, body rendering, debounce, and media groups, backed by focused tests and redacted real Telegram proof.

## Reproduction Assessment

Yes, source-reproducible: mock a Telegram document whose `ctx.getFile()` throws `file is too big` or exhausts retries; current main returns `null`, enqueues empty media, and has no `mediaUnavailableText` path. I did not run a live Telegram transfer in this read-only review.

## Solution Assessment

No, not as currently proposed. The resolver/body direction is narrow and maintainable, but the patch must preserve the reason through multi-entry debounce and media-group handling before merge.
