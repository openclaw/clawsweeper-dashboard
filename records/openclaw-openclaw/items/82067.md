## Summary

Keep open: the PR is protected by `security` and `maintainer` labels and the current head still has a source-reproducible owner-downgrade compatibility regression in the cron adapter.

## What This Changes

The PR moves queued system-event owner downgrades from prompt-visible `System (untrusted):` text to structured `forceSenderIsOwnerFalse` metadata across queue, reply, heartbeat, gateway, cron, agent, plugin producer, scanner, tests, and changelog surfaces.

## Best Possible Solution

Keep the structured metadata design, but map legacy cron `trusted: false` to the downgrade bit in the gateway cron adapter and add a regression test for that adapter path before maintainer/security review lands the PR.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: call the PR's `CronServiceDeps.enqueueSystemEvent` adapter with `{ trusted: false }` and it forwards no `forceSenderIsOwnerFalse` while setting `trusted: true`.

## Solution Assessment

No, not yet. The structured metadata direction is maintainable, but the cron adapter must preserve the deprecated compatibility alias before this security-sensitive change is safe to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve legacy cron trust downgrades:** `src/gateway/server-cron.ts:298-299`
  - body: `buildGatewayCronService` still ignores `opts.trusted` and derives the forwarded authority only from `forceSenderIsOwnerFalse`. A legacy cron caller using `{ trusted: false }` is silently upgraded to `{ trusted: true }` with no downgrade metadata, so externally sourced cron text can wake with owner semantics instead of least privilege. Map `opts.trusted === false` to the new downgrade bit and cover that adapter path.
  - confidence: 0.92
