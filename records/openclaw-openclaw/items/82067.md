## Summary

Keep open: this active security/maintainer PR has a blocking cron compatibility regression. The structured metadata direction is sound, and the PR body includes sufficient after-fix proof, but the current diff can drop legacy `trusted: false` owner downgrades in the gateway cron adapter.

## What This Changes

The PR moves queued system-event owner downgrades from prompt-visible `System (untrusted):` text to structured `forceSenderIsOwnerFalse` metadata across queue, reply, heartbeat, gateway, cron, agent, plugin producer, scanner, tests, and changelog surfaces.

## Best Possible Solution

Preserve the structured metadata approach, but first map legacy cron `trusted: false` to the new downgrade bit, cover that adapter path with a regression test, and then land after maintainer/security review.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence repro path: call the PR's gateway cron dependency adapter with `{ trusted: false }` and it forwards no `forceSenderIsOwnerFalse` while setting `trusted: true`; no executable test was run because this was a read-only review.

## Solution Assessment

No, not yet. The structured metadata design is the right maintainable fix, but the cron adapter must preserve the deprecated `trusted: false` compatibility alias before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve legacy cron trust downgrades:** `src/gateway/server-cron.ts:298-299`
  - body: `buildGatewayCronService` now ignores `opts.trusted` and derives `trusted` only from `forceSenderIsOwnerFalse`. A cron producer still using the compatibility alias `{ trusted: false }` will be forwarded as `trusted: true` with no downgrade metadata, so the queued event can run with owner semantics. Please map `opts.trusted === false` to the new downgrade bit and add a legacy-caller regression test.
