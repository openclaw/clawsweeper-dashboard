## Summary

Keep open. This draft PR has a protected `maintainer` label, and the diff has blocking authority and SDK-migration concerns: it removes the current owner-downgrade path for non-exec untrusted system-event wakes while plugin SDK callers still pass `trusted: false`.

## What This Changes

The PR removes `SystemEvent.trusted` / `SystemEventOptions.trusted`, normalizes queued system-event prompt formatting to `System:`, and drops queued-event trust from reply and heartbeat sender-owner authority.

## Best Possible Solution

Keep the PR open and either preserve/deprecate the SDK option while migrating all callers, or replace the old trust metadata with a concrete authority source in the same change before removing the downgrade behavior.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: compare the PR’s `ForceSenderIsOwnerFalse: hasExecCompletion` change with current command auth’s use of that flag, then run the affected heartbeat and system-event tests. I did not run tests because this review is required to keep the checkout read-only.

## Solution Assessment

No. The cleanup is not yet the best solution because it removes authority and SDK plumbing without a replacement authority source or complete caller migration.
