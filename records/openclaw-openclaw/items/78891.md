## Summary

Current main and the shipped release already handle this failure mode: provider `content_filter` finish reasons become agent errors, cron isolated runs treat those failures as job errors, and run history records delivery status separately.

## What This Changes

Cron-triggered announce deliveries should not report a green run when the provider ends with `finish_reason: content_filter` and no user message is delivered.

## Best Possible Solution

Keep the shipped error and delivery-status behavior; any new report should include the full `cron.runs` record so maintainers can distinguish a model-finish failure from a QQ adapter delivery issue.

## Reproduction Assessment

No. The current-main and release source path maps `content_filter` to an agent error, and cron now records fatal isolated run failures as `status: error` rather than clearing them as successful.

## Solution Assessment

Yes. The existing implementation is the narrow maintainable fix because it uses the generic provider-error and cron-run-status pipeline while keeping delivery outcome in the separate `deliveryStatus` field.
