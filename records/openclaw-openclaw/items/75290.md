## Summary

Keep open. Current main has narrowed one subset by carrying runtime `toolsAllow` into plugin tool materialization, but normal embedded runs still build `createOpenClawCodingTools(...)` on every run and `resolvePluginTools` still invokes registered plugin tool factories without per-plugin timing, a plugin-only embedded opt-out, or safe memoization.

## What This Changes

The issue requests investigation and fixes for repeated `core-plugin-tools` embedded-run latency, including factory-cost amortization, a plugin-tool opt-out, and per-plugin timing diagnostics.

## Best Possible Solution

Keep this as the canonical `core-plugin-tools` per-run factory-cost tracker. The next maintainable implementation should add low-overhead per-plugin factory timing and an explicit plugin-tool opt-out for embedded runs first, then consider narrowly cached static tool definitions only after the plugin tool contract defines safe invalidation and per-run wrappers.

## Reproduction Assessment

Yes. The issue and follow-up comments provide repeated prep-stage traces where `core-plugin-tools` is hot and drops near zero when plugin tools are skipped, and current main still follows the source path that invokes plugin factories during embedded tool construction.

## Solution Assessment

No current-main solution fully solves it. The safest repair path is instrumentation plus a threaded opt-out before any broad `(pluginId, sandboxKey)` factory-result memoization, because factories receive run-specific context.
