## Summary

Keep open. Current main has meaningful partial fixes for this latency cluster, including scoped plugin-tool discovery, per-plugin factory timing, and optional media factory gating, but normal embedded runs still construct the tool set per run and still invoke scoped plugin tool factories at request time; the public plugin-tool opt-out and any safe amortization remain unresolved.

## What This Changes

The issue asks to reduce repeated embedded-run `core-plugin-tools` prep latency, expose safer opt-outs/diagnostics, and avoid per-run plugin or core tool factory cost where possible.

## Best Possible Solution

Keep this as the canonical tracker and land focused fixes that avoid constructing unavailable or disallowed tools, preserve per-plugin timing, and memoize only factories proven not to capture request-scoped context.

## Reproduction Assessment

Yes. The issue and comments include repeated traces across installs plus a current-main warm-turn data point, and the source still shows request-time tool materialization even though this review did not run a live gateway trace.

## Solution Assessment

Yes. Keeping it open is the best current outcome because main has partial observability and gating fixes, while broad factory-output caching still needs context-safety review and a documented opt-out remains unresolved.
