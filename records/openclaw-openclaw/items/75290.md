## Summary

Keep open. Current main has narrowed one subset by carrying explicit `toolsAllow` into plugin tool materialization, but normal embedded runs still build the full OpenClaw tool set and still invoke every registered plugin tool factory during `core-plugin-tools`; the latest maintainer comment also explicitly says the remaining factory-cost, opt-out, and timing work is not proven fixed.

## What This Changes

The issue requests investigation and optimization for repeated embedded-run `core-plugin-tools` latency, including factory-cost amortization, a plugin-tool opt-out, and per-plugin timing visibility.

## Best Possible Solution

Keep this as the canonical remaining `core-plugin-tools` factory-cost tracker: preserve the current explicit-allowlist narrowing, add per-plugin factory timing, thread a deliberate plugin-tool opt-out for embedded runs, and defer memoization until the plugin tool context contract has a safe reusable-definition model.

## Reproduction Assessment

Yes. The issue includes repeated prep-stage traces plus a `disablePluginTools: true` control, and current main still follows the source path that resolves plugin tools and calls factories for normal embedded runs.

## Solution Assessment

No full current-main solution exists yet. The partial `toolsAllow` propagation is a good narrow fix, but the safest remaining approach is instrumentation and an explicit skip path before attempting broad factory-result caching.
