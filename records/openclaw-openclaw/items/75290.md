## Summary

Keep open. Current main still resolves plugin tools during each normal embedded tool-construction pass and still calls every registered plugin tool factory without per-plugin timing, safe factory-result memoization, or an embedded/run-level plugin-only opt-out; the recent allowlist threading and registry reuse work narrows related cases but does not close this report.

## What This Changes

The issue requests reducing repeated `core-plugin-tools` prep latency by diagnosing and avoiding per-run plugin tool factory cost, exposing a plugin-tool opt-out, and splitting timing by plugin.

## Best Possible Solution

Keep this as the focused `core-plugin-tools` factory-cost tracker. Add low-overhead per-plugin factory timing and a threaded plugin-tool opt-out first, then consider narrowly caching static plugin tool definitions only after a safe invalidation and per-run wrapper contract exists.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes: the issue provides repeated prep-stage traces plus a `disablePluginTools: true` control where the stage drops near zero, and current source still follows the same factory-call path during embedded tool construction.

## Solution Assessment

Is this the best way to solve the issue? No current-main solution fully solves it. The safest maintainable direction is instrumentation plus an explicit skip path before any broad `(pluginId, sandboxKey)` memoization.
