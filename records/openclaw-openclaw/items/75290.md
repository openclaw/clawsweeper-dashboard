## Summary

Keep open. Current main has useful partial fixes for discovery, scoped registry reuse, and per-plugin factory timing, but the normal embedded-run path still constructs tools per run and still calls each plugin tool factory at request time; the remaining opt-out or safe amortization work is not implemented yet.

## What This Changes

The issue requests reducing repeated `core-plugin-tools` embedded-run latency, adding a plugin-tool opt-out, and exposing per-plugin factory timing for diagnosis.

## Best Possible Solution

Keep this as the canonical tracker until current main both identifies slow plugin factories in logs and provides a documented way for agents or runs that do not need plugin tools to skip plugin-tool materialization. Memoization should be added only for specific factories or keys proven stable by those traces.

Implement a narrow fix for #75290: reduce embedded-run `core-plugin-tools` latency for agents/runs that do not need plugin tools. Current main already has per-plugin factory timing, so do not add another timing system and do not attempt broad factory-result memoization without explicit evidence. Use the existing internal `disablePluginTools` seam in `src/agents/openclaw-tools.ts` / `src/agents/openclaw-plugin-tools.ts` and thread a documented public or effective-policy opt-out so plugin tool factories are not materialized when plugin tools cannot be used or are explicitly disabled for that agent/run.
