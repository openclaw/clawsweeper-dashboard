## Summary

Close as implemented on current main. The post-report plugin-tool discovery fix now resolves plugin tools in a lightweight `tool-discovery` mode, skips channel runtime hydration during that path, and prefers built bundled plugin artifacts, which directly targets the per-run `core-plugin-tools` cost reported here.

## What This Changes

The issue asks to address repeated embedded-run `core-plugin-tools` latency from per-run plugin tool factory resolution, with follow-up questions about memoization, plugin-tool opt-out, and per-plugin timing.

## Best Possible Solution

Keep the current main implementation and ship it in the next release; if `core-plugin-tools` remains hot after a build containing 36e687edf0, open a narrower follow-up with fresh per-plugin traces rather than keeping this pre-fix umbrella open.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the affected releases: the issue includes repeated `core-plugin-tools` prep traces and a `disablePluginTools` control, while current-main inspection verifies the fixed source path rather than re-running a live gateway workload.

## Solution Assessment

Is this the best way to solve the issue? Yes. Avoiding source/jiti and channel-runtime hydration during plugin tool discovery is narrower and safer than broad factory-result memoization because it reduces the fixed per-run cost without weakening per-run context isolation.
