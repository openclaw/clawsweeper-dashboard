## Summary

Keep open. Current main has important mitigations for this latency cluster, including context-keyed plugin descriptor caching, per-plugin timing, allowlist-aware core-tool skipping, and media/PDF pre-gating, but it still constructs available built-in media/web tools per attempt and the overlapping built-in factory memoization PR remains open.

## What This Changes

The issue asks to reduce per-run `core-plugin-tools` prep latency, add safe amortization or opt-out controls, and expose finer timing for embedded runs.

## Best Possible Solution

Keep this as the canonical `core-plugin-tools` tracker until fresh current-main traces show warm embedded runs are cheap, or land/reject a rebased equivalent of #73039 with tests and live timing proof.

## Reproduction Assessment

Yes for the affected releases: the issue and comments provide repeated 2026.4.27/2026.4.29 stage traces and a `disablePluginTools` control. For current main, the reproduction is only partially static because the remaining built-in factory cost depends on available media/web providers and needs a live timing run.

## Solution Assessment

Yes, keeping the issue open is the best current outcome. Current main uses safer context-keyed descriptor caching and availability gating, but closing would be premature while available built-in factories still run per attempt and the direct memoization PR is open.
