## Summary

Keep open. Current main contains partial plugin-tool discovery and scoped registry-reuse fixes, but the embedded-run hot path still constructs OpenClaw coding tools on every run and `resolvePluginTools` still invokes each registered plugin tool factory at request time; maintainer discussion also reopened this as the canonical remaining tracker after a current-main warm-turn repro.

## What This Changes

The issue reports persistent 10-12s `core-plugin-tools` prep latency on every embedded run despite a cached plugin registry, and asks for factory-cost amortization, a plugin-tool opt-out, and per-plugin timing.

## Best Possible Solution

Keep this as the canonical tracker for the remaining `core-plugin-tools` factory-cost work. The next maintainable outcome is per-plugin factory timing plus a documented plugin-tool-only opt-out path, then narrowly keyed amortization only where timing data proves it is safe.

## Repair Work Prompt

Fix the remaining `core-plugin-tools` per-run factory-cost issue tracked at https://github.com/openclaw/openclaw/issues/75290. Current main already has tool-discovery and scoped registry-reuse mitigations, so do not replace them or add broad hidden factory-result caching as the first step. Add per-plugin timing around plugin tool factory materialization in `resolvePluginTools` so traces/logs can identify heavy plugin factories, and thread a documented plugin-tool-only opt-out for embedded runs/agent config where plugin tools are not needed. Keep core tool behavior unchanged, preserve existing `toolsAllow` semantics, and avoid changing plugin manifest semantics, dependency installation, or runtime activation policy. Add focused regression coverage for the new timing/opt-out plumbing and update user-facing docs plus `CHANGELOG.md` with reporter credit. If memoization is attempted, require a narrow key that accounts for sandbox, workspace, session/delivery context, gateway binding, config, and auth-sensitive state, with tests proving stale context cannot leak.
