## Summary

Keep open. Current main has reduced some adjacent prep-latency causes, especially channel describe paths and media/web provider discovery, but the central hot path reported here still builds the OpenClaw tool set on every embedded run and still calls each registered plugin tool factory during each resolution pass.

## What This Changes

This issue requests reducing repeated `core-plugin-tools` prep latency from per-run plugin tool factory resolution, with possible memoization, opt-out controls, and per-plugin timing.

## Best Possible Solution

Keep this open and fix it with a measured, context-safe approach: add per-plugin factory timing, skip plugin tool resolution when the final run policy explicitly excludes plugin tools, and only cache or split factory outputs after the plugin tool contract distinguishes static definitions from per-run wrappers.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for triage: the issue includes repeated `core-plugin-tools` timing logs and a control case where `disablePluginTools: true` removes the stage cost, and current source still follows the reported per-resolution factory path.

## Solution Assessment

Is this the best way to solve the issue? No current-main solution fully addresses the generic hot path yet. The recent lazy-load fixes are useful, but raw `(pluginId, sandboxKey)` memoization is not sufficient without a context-safe contract or a precise invalidation key.
