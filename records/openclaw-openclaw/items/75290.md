## Summary

Keep open. Current main still has the reported embedded-run hot path and still materializes plugin tools by invoking each registered plugin factory during tool resolution; the related #75520/#75521 work covers a narrower scoped-registry reload problem, not the full factory-cost, opt-out, and per-plugin timing requests here.

## What This Changes

The issue asks to reduce repeated `core-plugin-tools` prep latency by avoiding per-run plugin tool factory cost where safe, exposing a plugin-tool skip path for runs that do not need plugin tools, and adding per-plugin timing diagnostics.

## Best Possible Solution

Keep this as the canonical `core-plugin-tools` per-run factory-cost tracker. Review or land the narrower #75521 registry-reuse PR separately, then add measured per-plugin factory timings, thread an explicit plugin-tool opt-out through embedded tool construction, and only add memoization after the plugin tool contract has a safe cache key or reusable-definition/per-run-wrapper split.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the reported behavior: the issue includes repeated prep-stage traces plus a `disablePluginTools: true` control where the stage drops to about zero, and current main still follows the same source path that calls plugin factories during embedded tool construction.

## Solution Assessment

Is this the best way to solve the issue? No current-main solution exists yet. The safest direction is instrumentation and an explicit skip path first; broad `(pluginId, sandboxKey)` memoization should wait until context isolation is proven.
