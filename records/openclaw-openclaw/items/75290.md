## Summary

Keep open. Current main has useful partial fixes, including tool-discovery registry reuse and plugin factory timing traces, but normal embedded runs still materialize plugin tools on every run and still invoke each registered plugin tool factory at request time. The maintainer discussion also explicitly reopened this as the canonical remaining tracker after a warm current-main repro.

## What This Changes

Investigate and reduce repeated `core-plugin-tools` prep latency in embedded runs by identifying per-plugin factory cost, adding an opt-out where appropriate, and considering safe amortization.

## Best Possible Solution

Keep this as the canonical tracker for the remaining embedded-run plugin-tool prep work. The desired end state is a documented plugin-tools-only opt-out for agents/runs that do not need plugin tools, plus targeted lazy factory or narrowly keyed caching changes only for factories proven context-stable by the new timing traces.

## Reproduction Assessment

Yes. The issue has repeated prep-stage traces, a `disablePluginTools` control comparison, and a later current-main warm-turn repro; current source still maps the stage to per-run tool creation and request-time plugin factory invocation.

## Solution Assessment

Partially. Per-plugin timing is now implemented and is the right first step, but broad memoization is not yet proven safe; the narrower next fix is to thread a documented opt-out and use timing data to target any cache/lazy-loading work.
