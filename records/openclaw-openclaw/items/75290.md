## Summary

Keep open. Current main contains substantial partial fixes after the 2026.4.29 reports, but this is still the canonical tracker for live `core-plugin-tools` latency because normal embedded runs still build the tool set per attempt and an overlapping built-in factory memoization PR remains open.

## What This Changes

The issue asks to reduce per-run `core-plugin-tools` prep latency, add safe factory amortization or opt-out controls, and expose finer timing for embedded runs.

## Best Possible Solution

Keep this as the canonical latency tracker until current-main live traces show warm embedded runs are cheap or a focused PR lands the remaining built-in-tool factory amortization or plugin-tool opt-out safely.

## Reproduction Assessment

Yes for affected releases: multiple reporters provided `core-plugin-tools` stage traces on 2026.4.27/2026.4.29, plus a current-main-era warm-turn repro before the later cache/pre-gating changes. This review did not run a live gateway trace against d919281.

## Solution Assessment

Unclear. The current context-keyed plugin factory cache and manifest/auth pre-gating are safer than broad request-agnostic caching, but the remaining built-in factory memoization and opt-out direction should be reviewed with #73039 and fresh timing data.
