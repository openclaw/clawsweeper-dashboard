## Summary

Keep open. Current main has substantial partial mitigation and observability for this path, but normal embedded runs still build the OpenClaw tool set per attempt, configured media/PDF factories remain request-time when available, there is no documented plugin-tool-only opt-out, and the related media/web factory memoization PR is still open.

## What This Changes

The issue requests reducing per-run `core-plugin-tools` latency in embedded runs by amortizing plugin and built-in tool factory work, exposing a plugin-tool opt-out, and adding per-plugin/per-tool timing.

## Best Possible Solution

Keep this as the canonical tracker until context-safe factory amortization or a documented plugin-tool opt-out lands, ships, and is validated with warm embedded-run traces showing `core-plugin-tools` no longer dominates normal turns.

## Reproduction Assessment

Yes, for the reported released builds and earlier current-main snapshots: the issue contains repeated `core-plugin-tools` traces plus `disablePluginTools` controls and later same-shape datapoints. Exact reproduction on 9bedcff is unclear without a live warm-turn trace because current main now has new allowlist, descriptor-cache, and timing mitigations.

## Solution Assessment

Unclear. The current context-keyed descriptor cache and allowlist-gated media planning are safer than broad request-scoped tool-object memoization, but the best final fix likely needs the remaining media/web factory amortization from #73039 or an equivalent plus a documented cheap opt-out.
