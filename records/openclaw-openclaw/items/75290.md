## Summary

Keep open. Current main has partial discovery and registry-reuse fixes, but the remaining reported problem is still visible in source: normal embedded runs still construct the tool set per run and `resolvePluginTools` still invokes every registered plugin tool factory for that request, with no per-plugin factory timing, public plugin-tools-only opt-out, or safe factory-result amortization.

## What This Changes

Request to reduce and instrument repeated `core-plugin-tools` prep latency by avoiding per-run plugin tool factory cost, adding a plugin-tools opt-out, and splitting timing by plugin.

## Best Possible Solution

Keep this as the canonical tracker and land a narrow first repair that adds per-plugin factory timing plus a documented plugin-tools-only opt-out for embedded runs; only add memoization after timing identifies context-invariant factory work that can be cached without leaking per-run state.

## Reproduction Assessment

Yes. The issue and later comments provide repeated prep-stage traces, including a current-main ARM repro, and source inspection verifies the same per-run factory path still exists on current main.

## Solution Assessment

Unclear for broad memoization. The current discovery and registry-reuse fixes are useful but incomplete; the safer next solution is targeted instrumentation and an opt-out, then a separately justified cache keyed only on safe, context-invariant inputs.

## Repair Work Prompt

Fix the remaining first slice of #75290 without broad factory-result memoization. Current main still builds `createOpenClawCodingTools(...)` for each embedded run and `resolvePluginTools(...)` still calls `entry.factory(params.context)` for each registered plugin tool. Add per-plugin factory timing under the existing `OPENCLAW_PLUGIN_LOAD_PROFILE=1` profiling contract, preserving the documented `[plugin-load-profile] phase=<X> plugin=<Y> elapsedMs=<N> ... source=<S>` field order. Then expose a conservative plugin-tools-only opt-out for normal embedded runs, preferably in the existing tool policy/config surface so global and per-agent config can skip plugin tool materialization while leaving plugin runtime, channels, providers, and the core `message` tool available. Thread that opt-out through `runEmbeddedAttempt`/`createOpenClawCodingTools` to the existing `createOpenClawTools({ disablePluginTools: true })` path. Do not disable all plugins, do not remove channel runtime behavior, do not cache factory results yet, and preserve per-run context isolation. Update config types/schema/help, docs, focused tests, and CHANGELOG with user-facing credit. Treat #75290 as canonical; use #75408/#73428 only as supporting latency context and #75520 as the already-partial registry-reuse fix.
