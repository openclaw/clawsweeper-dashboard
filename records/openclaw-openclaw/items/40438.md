## Summary

Keep this PR open, but it is not merge-ready: the requested exclusion feature is still useful, while the branch is stale against current memory indexing architecture, lacks real behavior proof, exposes compatibility-sensitive defaults, and includes an unrelated runtime hook.

## What This Changes

This PR adds a `memorySearch.excludeDirs` config option, default directory exclusions, status/schema/help/type plumbing, and tests for pruning directory basenames during memory indexing.

## Best Possible Solution

Define one backend-consistent memory traversal exclusion contract in the active memory-host SDK and QMD collection/update path, document exact default and override semantics, and keep unrelated network behavior out of the branch.

## Reproduction Assessment

Yes, by source inspection. Current main still recursively scans configured builtin `extraPaths` without a user exclusion parameter, and QMD maps those paths to plain collections with no traversal exclusion policy; I did not run an end-to-end indexing command.

## Solution Assessment

No. The direction is useful, but this branch is not the best current-main solution because it modifies stale legacy files, leaves QMD traversal semantics unresolved, exposes a public config key prematurely, and mixes in unrelated network behavior.
