## Summary

Keep open: current main already covers the reported timeoutSeconds failure, but the central Claude ACP auth request remains unresolved because OpenClaw has no explicit per-harness env-file or SecretRef contract for injecting CLAUDE_CODE_OAUTH_TOKEN into ACP child processes.

## What This Changes

Request safe auth env forwarding for Claude ACP sessions, with a secondary report that timeoutSeconds was rejected by acpx in the reported release.

## Best Possible Solution

Define an explicit ACP harness auth/env contract, likely per-agent env or SecretRef-backed configuration plus docs, that can pass Claude OAuth env safely without broad automatic secret-file sourcing; keep the current timeout suppression behavior.

## Reproduction Assessment

Yes for the source-level boundary: current main only passes process.env into the Claude ACP wrapper and does not load ~/.openclaw/secrets/anthropic.env. I did not live-run Claude ACP with credentials, and the timeoutSeconds failure no longer matches current main.

## Solution Assessment

Unclear as proposed: automatically sourcing a secrets file is not obviously the safest credential boundary. A maintainer-approved per-harness env or SecretRef contract with docs and real Claude ACP proof would be the maintainable path.
