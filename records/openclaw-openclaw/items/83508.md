## Summary

Keep open: current main still lacks a supported ACPX/Claude harness credential forwarding contract for vaulted Claude OAuth, so gateway-spawned `sessions_spawn(runtime: "acp", agentId: "claude")` can still fail unless operators manually inject the ACPX auth environment into the gateway process. The separate per-call timeout part of the report has been handled, but the auth/provider boundary remains unresolved and security-sensitive.

## What This Changes

The issue asks OpenClaw to make ACPX Claude harness authentication work from gateway-spawned sessions using vaulted OAuth or another supported SecretRef/env contract, instead of requiring manual gateway environment injection.

## Best Possible Solution

Add an explicit ACPX plugin-owned, SecretRef-aware per-agent harness auth/env contract, document the supported `ACPX_AUTH_*` operator path, and avoid silent auto-sourcing of conventional secrets files.

## Reproduction Assessment

Yes. Source inspection shows no OpenClaw env/auth forwarding path into ACPX Claude sessions, and the reporter's workaround of injecting `ACPX_AUTH_CLAUDE_CODE_OAUTH_TOKEN` into the gateway environment matches the dependency contract.

## Solution Assessment

No final implementation should be chosen automatically. The best repair is an explicit SecretRef-aware ACPX auth/env contract or clearly documented `ACPX_AUTH_*` setup, not silent sourcing of local secrets files.
