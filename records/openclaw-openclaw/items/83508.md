## Summary

Keep open: current main still lacks a supported ACPX/Claude harness-auth forwarding contract from OpenClaw SecretRefs or env files. The reporter’s follow-up confirms that making `ACPX_AUTH_CLAUDE_CODE_OAUTH_TOKEN` visible to the gateway process works, and the timeout side report appears handled on current main, so the remaining work is a credential-boundary and docs/config decision.

## What This Changes

The issue asks OpenClaw to let gateway-spawned Claude ACP sessions use vaulted Claude OAuth credentials and also reports an unsupported ACPX timeout control path.

## Best Possible Solution

Define and document an explicit ACPX plugin-owned harness-auth contract, preferably SecretRef-aware and per agent, and avoid silently loading conventional secret files into external harness environments.

## Reproduction Assessment

Yes, from source inspection: `sessions_spawn` reaches ACP session creation without adding env from OpenClaw secrets, and the Claude wrapper only inherits the gateway process environment. I did not run a live Claude/acpx OAuth session in this read-only review.

## Solution Assessment

No for silent auto-sourcing: the maintainable path is an explicit ACPX per-agent auth/env contract with a documented security boundary. The issue’s timeout sub-report appears already handled on current main for Claude ACP commands.
