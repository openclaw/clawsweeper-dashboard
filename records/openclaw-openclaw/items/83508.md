## Summary

Keep open: current main still has no supported ACPX/Claude harness SecretRef, env-file, or documented `ACPX_AUTH_*` setup path for gateway-spawned Claude ACP sessions. The timeout sub-report is already fixed on current main, but the credential-forwarding request is a security-sensitive product/API decision, not conservative cleanup closure.

## What This Changes

The issue requests a supported way for OpenClaw to make Claude OAuth credentials available to gateway-spawned ACP Claude harnesses, and also reports a `timeoutSeconds` ACP control mismatch.

## Best Possible Solution

Add an explicit ACPX plugin-owned per-agent harness auth contract, preferably SecretRef-aware and documented with the existing `ACPX_AUTH_*` convention, and avoid silent auto-sourcing of fixed secret files.

## Reproduction Assessment

Yes for the source-level gap: current main does not read or resolve a Claude harness env file or SecretRef before spawning ACPX Claude, while upstream acpx only promotes `ACPX_AUTH_*` variables already present in the gateway process environment. I did not run a live Claude OAuth/acpx session.

## Solution Assessment

No for silent auto-sourcing of `~/.openclaw/secrets/anthropic.env`; the safer solution is an explicit per-agent ACPX auth/env or SecretRef contract plus docs. The reported `timeoutSeconds` mismatch is already handled on current main and should not drive this remaining fix.
