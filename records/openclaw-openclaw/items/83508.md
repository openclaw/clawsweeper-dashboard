## Summary

Keep open: current main still has no supported ACPX Claude harness SecretRef/env-file forwarding contract. The reporter's `ACPX_AUTH_CLAUDE_CODE_OAUTH_TOKEN` gateway-env workaround confirms a viable lower-level route, but making OpenClaw load or forward OAuth material is a security/product decision; the timeout sub-report is already handled on current main.

## What This Changes

Request a supported ACP Claude harness auth/env forwarding path so `sessions_spawn` can use Claude Max OAuth tokens without manual gateway environment surgery.

## Best Possible Solution

Add an explicit ACPX plugin-owned harness auth/env contract, preferably SecretRef-aware and per agent, and document the existing `ACPX_AUTH_*` gateway-env workaround without silently auto-sourcing conventional secret files.

## Reproduction Assessment

Yes, source-level: the Claude wrapper only inherits the gateway process environment, and the ACP control plane does not pass a resolved env payload into session creation. I did not run a live Claude Max OAuth repro in this read-only review.

## Solution Assessment

No for silent auto-sourcing from a fixed secrets file; that would make credential exposure implicit. The safer solution is an explicit per-agent ACPX env or SecretRef contract, with docs for the current `ACPX_AUTH_*` gateway-env workaround.
