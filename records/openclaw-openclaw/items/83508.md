## Summary

Keep open: current main covers the `timeoutSeconds` failure, but the Claude ACP credential-forwarding contract is still missing. The reporter’s verified `ACPX_AUTH_CLAUDE_CODE_OAUTH_TOKEN` gateway-env workaround points to a real supported-path/documentation gap, and choosing automatic env-file sourcing versus an explicit SecretRef/per-agent env contract is a security/product decision.

## What This Changes

The issue requests a supported ACP Claude harness auth path for gateway-spawned sessions, plus notes a `timeoutSeconds` ACPX control-forwarding failure.

## Best Possible Solution

Add an explicit ACPX plugin-owned per-agent harness auth/env contract, preferably SecretRef-aware and documented with `ACPX_AUTH_*`, while keeping the current timeout suppression unchanged.

## Reproduction Assessment

Yes, source-level: gateway ACP session creation passes no env, the generated Claude wrapper only copies `process.env`, and acpx only promotes `ACPX_AUTH_*` variables already present in that process. I did not run a live Claude OAuth session.

## Solution Assessment

No for silent auto-source behavior: it would blur the credential boundary. The maintainable direction is an explicit per-agent ACPX env or SecretRef contract plus docs for the existing `ACPX_AUTH_*` convention.
