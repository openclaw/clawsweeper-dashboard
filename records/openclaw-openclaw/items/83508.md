## Summary

Keep open: current main still does not provide an OpenClaw-documented or config-backed way to move vaulted Claude OAuth material into gateway-spawned ACP Claude harnesses. The reporter’s follow-up usefully confirms that `ACPX_AUTH_CLAUDE_CODE_OAUTH_TOKEN` works when injected into the gateway process, but choosing whether OpenClaw should document that path or add a SecretRef-aware env contract is a product/security decision.

## What This Changes

The issue asks OpenClaw to make ACP Claude sessions spawned by `sessions_spawn` work with vaulted Claude OAuth credentials and also reports a timeout config mismatch.

## Best Possible Solution

Document the current `ACPX_AUTH_*` gateway-environment path now, and only add a per-agent SecretRef-aware ACPX harness env contract if maintainers explicitly accept that credential boundary; avoid auto-sourcing a conventional secrets file silently.

## Reproduction Assessment

Yes, at source level: `sessions_spawn` reaches ACP session creation without an env payload, and the Claude wrapper only inherits the gateway process environment. I did not run a live Claude OAuth spawn, but the reporter’s follow-up confirms that adding the `ACPX_AUTH_CLAUDE_CODE_OAUTH_TOKEN` env var to the gateway process fixes the observed failure.

## Solution Assessment

No complete solution is on main yet. The immediate low-risk path is documentation for the existing `ACPX_AUTH_*` contract; adding env-file or SecretRef forwarding needs maintainer security/product review before implementation.
