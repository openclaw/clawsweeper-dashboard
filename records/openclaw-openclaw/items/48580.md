## Summary

Keep this open. Current main has meaningful ACPX lifecycle hardening and the pinned `acpx` package now reports dead/idle status, but there is still no high-confidence proof that the reported Codex `sessions new` dead-PID state is fully reconciled at creation time; the closest terminal-truth fix remains an open PR with unresolved proof and security concerns.

## What This Changes

The issue requests that ACPX Codex session creation either satisfy Codex's terminal needs or mark the session failed/closed when the spawned process exits immediately.

## Best Possible Solution

Rework or land a sanitized ACPX terminal-state reconciliation at the supported `acpx` runtime boundary, then close this after real Codex ACP proof shows failed or lost session state is reported accurately.

## Reproduction Assessment

No. The historical logs and current source make the state-drift bug plausible, but I did not reproduce the exact `stdin is not a terminal` / dead-PID session against current main, `acpx` 0.7.0, and the current Codex ACP adapter.

## Solution Assessment

Unclear. The terminal-truth direction is likely the right fix shape, but the open implementation needs sanitized durable metadata, real behavior proof, and alignment with the published `acpx` runtime boundary before this can be considered solved.
