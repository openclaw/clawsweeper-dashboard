## Summary

Keep this open: current main has substantial ACPX lifecycle hardening and newer ACPX/Codex adapter pins, but I could not prove that direct `acpx codex sessions new` no longer leaves a dead Codex process represented by an open session record. The remaining terminal/liveness contract is already being explored in related open PRs and needs maintainer/security review rather than cleanup closure.

## What This Changes

The issue asks ACPX Codex session creation to either satisfy the Codex CLI terminal requirement or reconcile session metadata when the spawned process exits immediately.

## Best Possible Solution

Define an ACPX-owned terminal/liveness contract that records failed or dead starts consistently, reports them clearly, and avoids persisting secret-bearing runtime metadata.

## Reproduction Assessment

No high-confidence live reproduction was established. The original steps are concrete and current source still shows open session records being reconciled separately from dead-process status/reconnect handling, but I did not run the reported Codex TTY failure on current main.

## Solution Assessment

No, closing from the merged lifecycle hardening alone would be premature. Current main improves cleanup, but the narrow maintainable path is owner-reviewed terminal-state and PID-liveness handling in the ACPX runtime boundary.
