## Summary

Keep this open. Current main has stronger ACPX lifecycle cleanup and a newer `acpx` dependency, but I did not find proof that the reported Codex `sessions new` dead-PID state is reconciled at creation time, and the direct terminal-truth work remains open with proof/security blockers.

## What This Changes

The issue asks ACPX/Codex session creation to handle TTY-required startup failure or accurately mark the session closed/failed when the spawned process exits immediately.

## Best Possible Solution

Land or supersede the terminal-state reconciliation at the supported ACPX runtime boundary, with sanitized durable metadata and real Codex ACP proof that failed or lost session state is reported accurately.

## Reproduction Assessment

No. Source and dependency inspection make the session-state drift plausible, but I did not establish a high-confidence live reproduction of the exact `stdin is not a terminal` dead-PID state on current main.

## Solution Assessment

Unclear. Lease-backed cleanup and the newer `acpx` status/reconnect semantics are useful mitigations, but the direct terminal-truth approach still needs sanitized metadata, proof, and boundary alignment before it is the best fix.
