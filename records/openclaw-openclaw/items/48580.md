## What This Changes

The issue reports that `acpx codex sessions new` can save an open Codex ACP session whose recorded PID has already exited after a TTY startup failure.

## Best Possible Solution

Reconcile session records with process/lifecycle truth at the supported acpx runtime boundary, with redacted durable diagnostics and real Codex ACP proof for a failed startup and a lost/dead session.

## Reproduction Assessment

No high-confidence live reproduction was established. Current source and `acpx@0.7.0` inspection make the dead-PID/session-record drift plausible, but I did not run the exact Codex TTY failure path on current main.

## Solution Assessment

Unclear. Lease-backed cleanup and newer acpx reconnect semantics are useful mitigations, but the maintainable fix should reconcile terminal/process truth at the acpx runtime boundary with proof and redaction rather than relying only on startup cleanup.
