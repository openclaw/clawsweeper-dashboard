## Summary

Keep this open: current main has useful ACPX lease cleanup and ships a newer `acpx` runtime, but source and dependency inspection still do not prove that a newly-created Codex session whose process dies immediately is reconciled to closed/lost at creation or status time.

## What This Changes

The issue reports that `acpx codex sessions new` can create a Codex ACP session marked open with a PID even though the spawned process has already exited because the Codex CLI needs a TTY.

## Best Possible Solution

Reconcile ACPX session records with process and terminal truth at the supported acpx runtime boundary, with redacted diagnostics and real Codex ACP proof for failed startup and dead-session recovery.

## Reproduction Assessment

No high-confidence live reproduction was established. Source and `acpx@0.7.0` inspection make the dead-PID/session-record drift plausible, but the exact Codex TTY failure was not run against current main.

## Solution Assessment

Unclear. The merged lease-backed cleanup is a useful mitigation, but the best fix should prove creation/status terminal truth at the acpx runtime boundary rather than relying only on startup or close-time cleanup.
