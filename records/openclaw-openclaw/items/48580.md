## Summary

Keep this open. Current main has ACP turn retry and dead cached-handle cleanup, but it does not prove the reported ACPX Codex session-create drift is fixed; the pinned `acpx@0.6.1` session creation path still records `closed: false` and a PID before the client is closed, and the terminal-truth work that directly targets this class of drift is still an open PR.

## What This Changes

The issue asks for ACPX Codex session creation to either satisfy Codex's TTY/runtime requirements or mark the session closed/failed when the child process exits immediately.

## Best Possible Solution

Review and land the active terminal-truth path, or ship an upstream ACPX fix that records authoritative terminal state during session creation and marks sessions failed/closed when the agent process exits before a usable session exists; then bump the bundled ACPX dependency and add a regression test for the Codex no-TTY/early-exit create path.

## Reproduction Assessment

Yes. The issue gives a concrete path (`acpx codex sessions new --name test`, then compare session JSON with the OS process table), and current `acpx@0.6.1` source still shows how `closed: false` plus a stale PID can be persisted around client shutdown; live proof was not run in this read-only checkout.

## Solution Assessment

No current-main solution is complete yet. The narrow maintainable fix is terminal-truth/liveness gating in ACPX session creation/status, with OpenClaw consuming the fixed ACPX package or the already-open terminal-truth PR once reviewed.
