## Summary

Keep this open: current main has meaningful ACPX lifecycle hardening, newer ACPX/Codex adapter pins, and dead-status handling, but the direct `acpx codex sessions new` path still has no proven current-main fix for an immediately dead Codex process being represented by an open session record.

## What This Changes

The issue reports `acpx codex sessions new` creating a Codex session record that remains open even though the spawned Codex process immediately exits because stdin is not a terminal.

## Best Possible Solution

Define and land an ACPX-owned terminal/liveness contract, preferably in upstream `acpx` with OpenClaw wrapper glue only, so failed or dead Codex starts are reported consistently without misleading open session records or secret-bearing artifacts.

## Reproduction Assessment

No high-confidence live reproduction was established. The original `acpx codex sessions new --name test` path is concrete, and source/dependency inspection shows open records and dead status remain separate, but I did not run the macOS Codex TTY failure against current main.

## Solution Assessment

No, closing from the merged lifecycle hardening alone is not the best solution. The maintainable path is owner-reviewed terminal-state and PID-liveness handling in the ACPX runtime boundary, with redaction rules for any durable artifacts.
