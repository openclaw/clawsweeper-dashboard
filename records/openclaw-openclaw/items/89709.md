## Summary

Keep open: current main and the v2026.5.28 tag still have a source-proven ranged-usage path where untimestamped historical usage can be included in daily/model totals, so the reported dashboard inflation is actionable rather than already implemented.

## What This Changes

The issue reports that Dashboard Usage daily model-call totals show historical cumulative data instead of date-filtered daily usage after upgrading to v2026.5.28.

## Best Possible Solution

Make ranged usage summaries exclude or separately quarantine untimestamped usage from daily/model/provider totals unless the caller explicitly asks for all-time usage, and cover the behavior with a regression test.

## Reproduction Assessment

Yes, by source inspection: a ranged usage request can keep untimestamped usage entries because both cache and live session summary builders only reject entries outside the range when a timestamp exists. I did not run a live WSL2 dashboard repro in this read-only review.

## Solution Assessment

Yes for keeping this open; the narrow fix should be in the shared usage summary/cache aggregation path, with UI tests only if the displayed top-model cards need an additional selected-day guard.
