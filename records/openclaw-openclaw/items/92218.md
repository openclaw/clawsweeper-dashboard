## Label Justifications

- `P2`: The bug blocks a supported QMD memory workflow and produces misleading diagnostics, but its blast radius is limited to combined QMD and fallback failure states.
- `impact:session-state`: The disabled tool prevents retrieval from configured persistent agent memory.
- `impact:auth-provider`: The failure is surfaced as missing embedding-provider credentials even though QMD is the selected backend.

## Summary

Keep this issue open. Current main intentionally falls back from unavailable QMD to builtin memory, but initial QMD failure and cooldown paths discard the QMD failure reason when builtin initialization also fails, leaving users with misleading embedding-provider errors. The earlier related report found a configuration workaround, but it predates this current fallback implementation and does not resolve the source-level failure.

## What This Changes

Preserve an actionable QMD failure when the documented builtin fallback cannot initialize because embedding providers are unavailable.

## Best Possible Solution

Keep the documented QMD-to-builtin fallback, but when builtin initialization also fails, return the original actionable QMD reason with concise fallback diagnostics instead of exposing only unrelated provider-key errors.

## Reproduction Assessment

Yes at source level. Select QMD, make QMD manager creation fail or enter its failure cooldown, and leave builtin embeddings unavailable; current main then returns provider-readiness errors without preserving the QMD cause.

## Solution Assessment

Yes. Preserving the documented fallback while combining its failure diagnostics with the primary QMD error is narrower and safer than removing fallback or requiring unrelated embedding configuration.
