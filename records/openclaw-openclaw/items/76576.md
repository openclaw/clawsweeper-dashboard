## Summary

Keep open. Current main still has a source-level Gateway runtime gap: a selected external context-engine plugin can be enabled and visible in plugin inventory, but omitted from the Gateway startup snapshot used by agent turns, so `resolveContextEngine()` can hit the reported fallback warning before the plugin registration code runs.

## What This Changes

The issue reports that `plugins.slots.contextEngine: "lossless-claw"` is loaded/enabled but not registered at QQBot message runtime, causing fallback to the legacy context engine.

## Best Possible Solution

Keep this issue open until the generic Gateway startup/runtime plan includes the selected non-legacy context-engine plugin before any agent or channel-triggered turn resolves the active engine, with regression coverage for a `kind: "context-engine"` plugin that lacks `activation.onStartup`.

## Reproduction Assessment

Yes. This is source-reproducible on current main with `plugins.slots.contextEngine: "lossless-claw"` and the published `lossless-claw@0.9.2` manifest: the plugin can be omitted from `startup.pluginIds`, then `ensureRuntimePluginsLoaded()` scopes runtime loading before `resolveContextEngine()` emits the reported fallback.

## Solution Assessment

Yes. The narrow maintainable fix is generic plugin planning/runtime loading for the selected context-engine slot, not a QQBot-specific or `lossless-claw`-specific special case.
