## Summary

Keep open. Current main still has a source-level startup/load-plan gap: a selected external context-engine plugin can be enabled and present in the installed plugin index, but omitted from the gateway startup plugin IDs, so agent/QQBot turns can load only the scoped startup set and then fall back when `resolveContextEngine()` cannot find the selected engine.

## What This Changes

The issue reports that a configured and enabled `lossless-claw` context-engine plugin logs a fallback to the legacy engine during QQBot message handling instead of being registered and used.

## Best Possible Solution

Teach the generic gateway startup/runtime plugin plan to include the selected non-legacy `plugins.slots.contextEngine` plugin as a required runtime dependency before any agent turn or compaction resolves the active engine.

## Reproduction Assessment

Yes. This is source-reproducible with `plugins.slots.contextEngine: "lossless-claw"` and the published `lossless-claw@0.9.2` manifest, which lacks `activation.onStartup`; current main can omit it from `startup.pluginIds` and then hit the exact resolver fallback warning.

## Solution Assessment

Yes. The narrow maintainable fix is to include the selected context-engine plugin in the generic startup/runtime load plan, not to add QQBot-specific behavior or a `lossless-claw` special case.
