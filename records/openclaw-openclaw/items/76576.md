## Summary

Keep open. Current main still has a source-level path where channel-triggered turns load only the Gateway startup plugin scope before resolving the selected context engine; that scope can include QQBot while omitting the non-legacy `plugins.slots.contextEngine` plugin, which produces the exact fallback warning reported for `lossless-claw`.

## What This Changes

The issue reports that a loaded and enabled `lossless-claw` context-engine plugin selected through `plugins.slots.contextEngine` is not registered during QQBot turns, causing fallback to `legacy`.

## Best Possible Solution

Add the selected non-legacy context-engine plugin to the Gateway/runtime plugin load scope before context-engine resolution, using generic plugin-id normalization and existing enable/deny rules rather than a `lossless-claw` special case.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a QQBot-triggered turn loads the scoped runtime plugin registry, then `resolveContextEngine()` falls back when the selected non-legacy context-engine plugin was omitted from that scoped load.

## Solution Assessment

Yes. The narrow maintainable fix is to include the configured context-engine slot owner in the startup/runtime plugin scope and add fake-plugin regression coverage; changing the external plugin or hard-coding `lossless-claw` would be the wrong boundary.

## Repair Work Prompt

Fix the selected context-engine runtime loading bug reported in openclaw/openclaw#76576. Current main can scope live channel/agent runtime plugin loading to Gateway `startup.pluginIds`, while `plugins.slots.contextEngine` non-legacy plugins are only included by `resolveEffectivePluginIds`; as a result `lossless-claw` can appear installed/enabled but not be registered when `resolveContextEngine()` runs. Add a generic selected-context-engine plugin inclusion to the Gateway startup/runtime plugin plan, honoring plugin enablement, deny/disabled entries, aliases/normalization, and keeping the default `legacy` engine out of plugin loads. Do not hard-code `lossless-claw`, QQBot, or external package names. Add regression coverage with generated/fake plugin metadata for a configured channel plus a selected context-engine plugin, including exclusions for `legacy`, globally disabled plugins, denylisted plugins, and disabled entries. Update CHANGELOG.md with a single-line user-facing fix entry. Validate with targeted plugin startup/runtime tests and the changed gate in Testbox.
