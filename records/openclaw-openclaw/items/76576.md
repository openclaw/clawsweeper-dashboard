## Summary

Keep open. Current main still has a source-level plugin startup/load-plan gap: selected external context-engine plugins can be omitted from gateway startup runtime plugin IDs, so a QQBot-dispatched agent run can load only the scoped startup set and then log the reported fallback when `resolveContextEngine()` cannot find the selected engine.

## What This Changes

The issue reports that a globally installed `lossless-claw` context-engine plugin is loaded and selected, but QQBot handling warns that it is not registered and falls back to `legacy`.

## Best Possible Solution

Teach gateway plugin startup/runtime loading to treat a non-legacy `plugins.slots.contextEngine` selection as a required runtime plugin dependency before agent runs or compaction resolve the active engine.

## Reproduction Assessment

Yes, source-reproducible. With a non-legacy external `plugins.slots.contextEngine` plugin that lacks `activation.onStartup` and is not a memory-startup plugin, current main can omit it from `startup.pluginIds`, load only that scoped set, and then hit the resolver fallback warning.

## Solution Assessment

Yes. The narrow maintainable fix is to include the selected context-engine plugin in the generic plugin startup/runtime load plan, not to add QQBot-specific behavior or a `lossless-claw` special case.
