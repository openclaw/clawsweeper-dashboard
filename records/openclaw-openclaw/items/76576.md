## Summary

Keep open. Current main still has a source-level registration gap where the selected external context-engine plugin can be enabled and present in effective plugin resolution, but absent from the startup-scoped runtime load set used before `resolveContextEngine()` runs.

## What This Changes

The issue reports that a configured external `lossless-claw` context-engine plugin is loaded/enabled but not registered at runtime, causing fallback to `legacy`.

## Best Possible Solution

Selected non-legacy context-engine plugins should be part of the generic startup/runtime dependency plan before any agent turn, compaction path, or subagent path resolves the active engine.

## Reproduction Assessment

Yes. Source inspection shows a current-main path where `plugins.slots.contextEngine: "lossless-claw"` can be omitted from `startup.pluginIds`, then agent runtime loading scopes to that list before resolving the configured engine.

## Solution Assessment

Yes. The narrow maintainable fix is to include the selected context-engine plugin in the generic runtime/startup load plan, without adding QQBot-specific behavior or a `lossless-claw` special case.

## Repair Work Prompt

Fix the context-engine plugin startup/runtime registration regression reported in openclaw/openclaw#76576. A selected non-legacy `plugins.slots.contextEngine` plugin such as `lossless-claw` can be enabled and present in effective plugin resolution but omitted from `startup.pluginIds`; agent and compaction paths then call `ensureRuntimePluginsLoaded()` with that startup-scoped `onlyPluginIds` list before `resolveContextEngine()` runs, causing the configured engine to be unregistered and fallback to `legacy`. Keep the fix generic: do not special-case QQBot or `lossless-claw`, and do not add a new config option. Update the gateway startup/runtime plugin plan so the selected, enabled, non-default context-engine plugin is treated as a required runtime dependency, respecting global plugin disablement, denylist, and disabled entry semantics. Add focused regression coverage around startup plugin planning/runtime loading and update CHANGELOG.md because this is a user-facing regression fix.
