## Summary

Keep open. Current main and v2026.5.7 support a cheaper single Dream Diary model through `dreaming.model` and `dreaming.execution.defaults.model`, but the requested direct `dreaming.execution.model` shape and ordered fallback-model chain are still absent and require maintainer API direction.

## What This Changes

The issue requests Dreaming execution config that can choose a cheap/background model and an ordered fallback model chain.

## Best Possible Solution

Choose and document one Dreaming model/fallback contract, then align the manifest schema, resolver, subagent invocation contract, docs, and focused regression tests around that contract.

## Reproduction Assessment

Yes for the config/API gap via source inspection: current main rejects direct `dreaming.execution.model` and fallback-list shapes while carrying only one optional model through resolver/runtime. I did not run a live cron validation path.

## Solution Assessment

Unclear. The requested shape may be useful, but it overlaps existing `dreaming.model` and `dreaming.execution.defaults.model`; maintainers should choose a single stable public fallback contract before implementation.
