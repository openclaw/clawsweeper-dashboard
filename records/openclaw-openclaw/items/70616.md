## Summary

Keep open. Current main supports a single Dream Diary model override through `dreaming.model` and `dreaming.execution.defaults.model`, plus one retry to the session default, but the requested direct `dreaming.execution.model` shape and ordered fallback-model chain are still absent and need a maintainer API decision.

## What This Changes

The issue requests a Dreaming configuration execution field that can select cheaper background models and provide ordered fallback models.

## Best Possible Solution

Choose and document one stable Dreaming execution/fallback contract, then align the manifest schema, resolver, subagent invocation contract, docs, and focused regression tests around that contract.

## Reproduction Assessment

Yes for the config/API gap by source inspection: current main rejects direct `dreaming.execution.model` and fallback-list shapes while carrying only one optional model through resolver/runtime. I did not run a live Dreaming cron path because the read-only source contract is enough to prove the remaining feature gap.

## Solution Assessment

Unclear. The requested shape may be useful, but it overlaps existing `dreaming.model` and `dreaming.execution.defaults.model`; maintainers should choose a single stable public fallback contract before implementation.
