## Summary

Keep open. Current main supports cheaper Dream Diary routing through `dreaming.model`, `dreaming.execution.defaults.model`, per-phase `execution.model`, and one retry to the session default, but it still rejects the requested direct `dreaming.execution.model` shape and has no configured fallback-model chain.

## What This Changes

Request to add memory-core `dreaming.execution` config for cheaper/background Dream Diary model selection and ordered fallback models.

## Best Possible Solution

Choose the public memory-core Dreaming execution contract, then either document the existing `dreaming.model` path as canonical or implement a backward-compatible fallback chain through schema, resolver, runtime, docs, and tests.

## Reproduction Assessment

Yes. Source inspection on current main shows the strict manifest rejects the requested direct `dreaming.execution.model` and fallback-list shape, while runtime fallback only retries with the session default.

## Solution Assessment

Unclear. The current shipped `dreaming.model` and `execution.defaults.model` design partially solves cheaper model routing, but adding a direct execution field and ordered fallback chain needs a maintainer API decision before implementation.
