## Summary

Keep open. Current main has only partial dreaming model override support and still does not expose direct `dreaming.execution.model` or configured fallback chains; the issue discussion includes a concrete schema-rejection reproduction, and the current docs/tests confirm the retry path only goes to the session default model.

## What This Changes

Request to extend memory-core dreaming configuration so background dreaming/Dream Diary runs can specify execution model overrides and ordered fallback models.

## Best Possible Solution

Keep this open for a maintainer API decision. If accepted, implement a backward-compatible fallback surface across the strict manifest, resolver types, Dream Diary attempt order, subagent trust/allowlist handling, docs/config reference, and focused tests; otherwise document the existing `dreaming.model` and session-default retry behavior as the canonical supported path.

## Reproduction Assessment

Yes. Static current-main inspection gives a high-confidence reproduction: the requested direct `dreaming.execution.model` and fallback list are not schema properties under strict validation, and the runtime attempt order has no configured fallback chain.

## Solution Assessment

No. The current `dreaming.model` and `execution.defaults.model` behavior is a useful partial solution, but it does not solve the requested direct execution shape or configured fallback models; the safest next step is to choose the public config contract first.
