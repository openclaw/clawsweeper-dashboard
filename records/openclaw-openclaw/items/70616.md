## Summary

Keep open. Current main and v2026.5.7 cover the cheaper single-model use case through `dreaming.model` and nested execution defaults, but the requested direct `dreaming.execution.model` shape and ordered fallback-model chain are still absent and need maintainer API judgment.

## What This Changes

Request adds a Dreaming execution config surface for selecting cheaper background models and configured fallback models.

## Best Possible Solution

Choose and document one Dreaming model/fallback contract, then align schema, resolver, subagent invocation, docs, and regression tests around that contract.

## Reproduction Assessment

Yes for the config/API gap via source inspection: current main rejects the requested direct `dreaming.execution.model` and fallback-list shape and carries only one optional model through the resolver/runtime. A live cron validation run was not performed.

## Solution Assessment

Unclear. The requested shape may be useful, but it overlaps with existing `dreaming.model` and `dreaming.execution.defaults.model`, so maintainers should choose the stable public contract before implementation.
