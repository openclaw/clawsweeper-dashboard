## Summary

Keep open. Current main partially addresses cheaper Dream Diary model selection through `dreaming.model`, `execution.defaults.model`, and one retry to the session default, but it still does not expose the requested direct `dreaming.execution.model` shape or a configured fallback-model chain.

## What This Changes

Add memory-core dreaming configuration support for direct execution model overrides and explicit fallback models for background dreaming runs.

## Best Possible Solution

Choose the public memory-core dreaming execution contract, then either document the existing `dreaming.model` path as canonical or implement a backward-compatible fallback chain through the manifest, resolver, Dream Diary attempts, docs, and tests.

## Reproduction Assessment

Yes. Static current-main inspection gives a high-confidence reproduction: direct `dreaming.execution.model` and configured fallback lists are absent from the strict manifest/resolver, while runtime retry only falls back to the session default.

## Solution Assessment

Unclear. The current `dreaming.model` implementation is a useful partial solution, but it is not the requested direct execution/fallback shape; the narrowest maintainable next step is a maintainer API decision before code changes.
