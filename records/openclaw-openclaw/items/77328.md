## Summary

Keep open: the PR is aimed at a valid linked bug, but the latest head still does not make the generated plugin-skill links reliably usable and leaves stale owned symlinks behind.

## What This Changes

The PR adds plugin skill symlink publication, routes a generated plugin-skills directory into skill loading, adds regression coverage, and updates the changelog for #77296.

## Best Possible Solution

Publish plugin-declared skills to an agent-readable generated root while preserving the documented low-precedence tier, then update owned symlinks and cleanup inactive plugin links deterministically.

## Reproduction Assessment

Yes, source-based. Current main resolves enabled plugin skill directories for prompt loading, but it does not publish a conventional file-discovery path, and the linked issue includes a concrete ENOENT path for the missing SKILL.md.

## Solution Assessment

No. The direction is maintainable, but the current patch writes symlinked children into a root that the existing loader does not load, and it does not repair or remove generated links when active plugin state changes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Make the published plugin root loadable:** `src/agents/skills/workspace.ts:643`
  - body: Adding `pluginSkillsDir` to `mergedExtraDirs` does not make the generated symlinks usable: the local skill loader only lists `Dirent.isDirectory()` children and verifies opened SKILL.md realpaths stay under the configured root, so symlinked children under `~/.openclaw/plugin-skills` are skipped or rejected. The snapshot therefore still falls back to the original plugin dirs and does not prove the agent-readable discovery path is fixed.
  - confidence: 0.9
- **[P2] Replace owned links when plugin targets move:** `src/agents/skills/plugin-skills.ts:187-190`
  - body: The new `plugin-skills` directory is described as fully OpenClaw-owned, but an existing symlink for an active skill name that points at an old plugin path is only logged and skipped. Because cleanup also skips active names, a reinstall or plugin path move leaves a stale generated link and can continue producing ENOENT or old prompt content.
  - confidence: 0.88
- **[P2] Remove inactive plugin links whose targets still exist:** `src/agents/skills/plugin-skills.ts:222-225`
  - body: The stale-link pass only deletes generated links when the target path no longer exists. Disabling a plugin, changing the selected memory slot, or otherwise deactivating a plugin can leave the target directory on disk, so the generated discovery link remains even though that plugin skill is no longer active.
  - confidence: 0.84
- **[P3] Correct the changelog path:** `CHANGELOG.md:52`
  - body: The changelog says plugin skills are published into `~/.openclaw/skills/`, but the current implementation publishes into `~/.openclaw/plugin-skills/`. Release notes should not point users or maintainers at the wrong generated root.
  - confidence: 0.93
