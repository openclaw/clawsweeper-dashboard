## Summary

This PR addresses a real linked bug, but the current patch should not merge as-is because it promotes plugin skills through a higher-precedence managed root and mutates unmanaged symlinks without ownership tracking.

## What This Changes

The PR adds managed-root symlink publication for plugin-declared skills, regression tests, a workspace resolver parameter, and a changelog entry.

## Best Possible Solution

Publish plugin-declared skills to the agent discovery surface while preserving their documented low-precedence tier and tracking generated entries before replacing or deleting symlinks.

## Reproduction Assessment

Yes, source-based. Current main resolves enabled plugin skill paths and merges them into extraDirs, but it does not publish them into the global file-discovery roots described in the linked bug.

## Solution Assessment

No. The direction is valid, but this implementation should preserve plugin-skill precedence and add ownership-aware symlink repair/cleanup before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep plugin skills below bundled skills:** `src/agents/skills/plugin-skills.ts:198`
  - body: Creating plugin symlinks under `managedSkillsDir` exposes third-party plugin skills through a higher-precedence managed/local discovery root. For same-name conflicts, path-based skill consumers can now prefer the plugin copy even though the docs say plugin skills stay in the low-precedence extra-dir tier below bundled and managed skills.
  - confidence: 0.88
- **[P2] Update owned links when plugin targets move:** `src/agents/skills/plugin-skills.ts:187-190`
  - body: If an existing symlink for the same skill name points at an old plugin target, this branch logs and skips it, and the later cleanup skips active names. A reinstall or path move can therefore leave the agent following a stale link and still hitting ENOENT.
  - confidence: 0.86
- **[P2] Prove ownership before deleting managed symlinks:** `src/agents/skills/plugin-skills.ts:224-225`
  - body: The stale cleanup deletes any dangling symlink under the managed skills root that is not in the active plugin target map. A user-managed symlink to a temporarily unavailable path would be removed even though OpenClaw did not create it.
  - confidence: 0.9
