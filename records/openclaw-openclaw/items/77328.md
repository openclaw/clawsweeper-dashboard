## Summary

Keep open: the PR addresses a valid plugin-skill discovery bug, but the proposed patch is incomplete and introduces precedence/persistence problems around managed skill symlinks.

## What This Changes

The PR adds a plugin-skill symlink publisher called from skill resolution, threads the managed skills directory through the workspace loader, adds symlink-focused tests, and updates the changelog.

## Best Possible Solution

Expose enabled plugin-declared skills to agent file discovery using the same contained skill-root semantics as the workspace loader, while preserving managed/user precedence and hiding entries when plugins are disabled.

## Reproduction Assessment

Yes. A browser-style plugin manifest declares `skills: ["./skills"]`, the actual `SKILL.md` lives in a child directory, and the PR publisher skips that root because `./skills/SKILL.md` does not exist.

## Solution Assessment

No. Publishing into managed discovery paths can be a workable direction, but this patch must resolve concrete child skill directories and track plugin-owned links without violating managed skill precedence.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Publish child skills from manifest roots:** `src/agents/skills/plugin-skills.ts:124-126`
  - body: Most bundled plugins declare `skills: ["./skills"]`, with the actual `SKILL.md` in child directories such as `skills/browser-automation/SKILL.md`. This guard skips those parent roots entirely, so the PR still does not publish the stock plugin skills named in the report.
  - confidence: 0.95
- **[P2] Preserve existing managed skill symlinks:** `src/agents/skills/plugin-skills.ts:147-157`
  - body: If a user-managed skill is itself a symlink, `readlinkSync` succeeds and this code unlinks it whenever a plugin skill has the same basename. That lets plugin publication replace a higher-precedence managed override instead of preserving the documented managed/local skill source.
  - confidence: 0.88
- **[P2] Remove inactive plugin links while preserving user links:** `src/agents/skills/plugin-skills.ts:195-199`
  - body: The cleanup only removes inactive symlinks when their target no longer exists. Disabling a plugin leaves the plugin directory on disk, so a previously published symlink remains under the managed skills root and the disabled plugin skill can still be discovered.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff changes prompt-skill discovery by creating managed symlinks and needs ownership safeguards before merge.

Concerns:

- **[medium] Plugin publication can replace user-managed symlinked skills:** `src/agents/skills/plugin-skills.ts:147`
  - body: A plugin skill with the same basename as an existing managed symlink causes the existing symlink to be unlinked and replaced, which can change the prompt instructions an agent reads from a higher-precedence local skill source.
  - confidence: 0.86
