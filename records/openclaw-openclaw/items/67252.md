## Summary

Keep open. Current `main` and the latest release still clear Feishu Docx replacement content with one non-paginated `documentBlock.list` call, while the SDK contract is paginated and an external fork PR with the narrow fix remains open rather than merged into `openclaw/openclaw`.

## What This Changes

The issue requests that Feishu Docx `write` fully replace large documents by paginating `clearDocumentContent` before deleting existing blocks.

## Best Possible Solution

Land a focused Feishu plugin fix that paginates `clearDocumentContent` until no next page remains, deletes the full top-level range, and adds a regression test for multi-page block-list responses during `write`.

## Reproduction Assessment

Yes, at source level. A high-confidence unit reproduction would return first-page root children plus a `page_token` from `documentBlock.list`, then more root children on the next page; current `writeDoc` only makes the first list call before deleting.

## Solution Assessment

Yes. Paginating `clearDocumentContent` using the SDK's `page_token`/`has_more` contract and covering it with a regression test is the narrow maintainable fix; maintainers should review or import the linked external PR before creating duplicate repair work.
