## Summary

Keep open. Current main and the latest tagged release still clear Feishu Docx content with a single non-paginated `documentBlock.list` call before replacement content is inserted, while the open external fork PR that references this issue has not landed in `openclaw/openclaw`.

## What This Changes

The issue requests that Feishu `write` fully replace large documents by paginating `clearDocumentContent` before deleting old top-level blocks.

## Best Possible Solution

Land a focused Feishu plugin fix that paginates `clearDocumentContent` until no next page token remains and adds a regression test for multi-page block-list responses during `write`.

## Reproduction Assessment

Yes at source level. A high-confidence mocked path is for `documentBlock.list` to return root children plus `data.page_token` on the first call and more root children on the next page; current `writeDoc` calls the list method once, so `batchDelete` only receives the first-page count.

## Solution Assessment

Yes. Paginating `clearDocumentContent` with the SDK's `page_token` contract is the narrow maintainable fix; because an external PR with closing syntax is already open, maintainers should review/import it rather than create a duplicate repair by default.
