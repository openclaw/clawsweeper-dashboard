## Summary

Keep open. Current main and the latest release still clear Feishu document content with a single non-paginated `documentBlock.list` call before `writeDoc` inserts replacement content; the issue discussion and external fork PR are useful prior art, but no fix has landed in `openclaw/openclaw`.

## What This Changes

The issue requests that the Feishu `feishu_doc` write action fully replace large existing documents by paginating `clearDocumentContent` before deleting old blocks and inserting new content.

## Best Possible Solution

Land a focused Feishu plugin fix in `extensions/feishu/src/docx.ts`: paginate `client.docx.documentBlock.list` inside `clearDocumentContent` until no next page token remains, delete the full top-level range, and add regression coverage where `feishu_doc` write receives multiple block-list pages before inserting replacement content.

## Reproduction Assessment

Yes. A mocked reproduction is high confidence: make `documentBlock.list` return a first page with `data.page_token` and additional root blocks on a second page, then current `writeDoc` calls the list method once and deletes only the first page count.

## Solution Assessment

Yes. Paginating `clearDocumentContent` is the narrowest maintainable fix for the reported replace-all behavior; the external fork patch is directionally right, but the in-repo fix should include a multi-page regression test.
