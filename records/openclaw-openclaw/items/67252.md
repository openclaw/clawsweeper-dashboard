## Label Justifications

- `P2`: This is a normal-priority user-facing Feishu plugin bug with a clear narrow code path and limited large-document blast radius.
- `impact:data-loss`: The reported write operation can leave stale old document blocks mixed with replacement content, corrupting the intended document state.

## Summary

Keep open. Current main and the latest release still use a single non-paginated Feishu Docx block-list call before replacement writes, while the bundled skill promises replace-all behavior and the locked SDK exposes a paginated contract; the open external fork PR is useful prior art but has not landed in openclaw/openclaw.

## What This Changes

The issue asks for `feishu_doc` `write` to clear every existing top-level block in large Feishu documents before inserting replacement content.

## Best Possible Solution

Update the Feishu plugin's Docx clear helper to paginate `documentBlock.list` before `batchDelete`, then add regression coverage for multi-page `write` replacement.

## Reproduction Assessment

Yes, at source level. A focused unit reproduction can mock `documentBlock.list` returning first-page root children plus a `page_token`, then second-page root children; current main only deletes the first page before inserting replacement content.

## Solution Assessment

Yes. Paginating `clearDocumentContent` with the SDK `page_token`/`has_more` contract is the narrow maintainable fix, and the external fork PR is useful prior art but target-repo regression coverage is still needed.
