## Summary

Keep open. Current main and the latest release still clear Feishu Docx replacement content with one non-paginated `documentBlock.list` call; the Feishu SDK exposes `page_token`/`has_more`, and the linked external fork fix remains open rather than merged into `openclaw/openclaw`.

## What This Changes

The issue reports that Feishu `feishu_doc` `write` only deletes the first paginated block-list page before inserting replacement content on large documents.

## Best Possible Solution

Land a focused Feishu plugin fix that paginates `clearDocumentContent` until no next page token remains and adds a regression test for multi-page block-list responses during `write`.

## Reproduction Assessment

Yes at source level. A high-confidence mocked path is for `documentBlock.list` to return root children plus `data.page_token` on the first call and more root children on the next page; current `writeDoc` calls the list method once, so `batchDelete` only receives the first-page count.

## Solution Assessment

Yes. Paginating `clearDocumentContent` with the SDK's `page_token` contract and adding regression coverage is the narrow maintainable fix; maintainers should review or import the linked external fork PR before authorizing a duplicate replacement.
