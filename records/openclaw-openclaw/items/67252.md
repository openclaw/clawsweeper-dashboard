## Summary

Keep open. Current main and the latest release still clear Feishu Docx replacement content with one non-paginated `documentBlock.list` call before `writeDoc` inserts replacement content, while the SDK contract is paginated and an open external fork PR already proposes the narrow fix.

## What This Changes

The issue requests that Feishu document `write` paginate block listing so replacing a large existing document clears all top-level blocks before inserting new content.

## Best Possible Solution

Land a focused Feishu plugin fix that paginates `clearDocumentContent` until no next page token remains, deletes the full top-level range, and adds regression coverage for multi-page `documentBlock.list` responses during `write`.

## Reproduction Assessment

Yes, at source level. A focused unit reproduction can return first-page root children plus a `page_token`, then second-page root children; current `writeDoc` only makes the first list call before deleting.

## Solution Assessment

Yes. Paginating `clearDocumentContent` using the SDK `page_token`/`has_more` contract is the narrow maintainable fix, with the external fork PR as useful prior art but target-repo regression coverage still needed.
