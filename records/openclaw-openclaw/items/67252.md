## Summary

Keep open. Current main still clears Feishu Docx content with a single non-paginated `documentBlock.list` response before `feishu_doc` write inserts replacement content, while the bundled skill promises replace-all behavior and the locked Feishu SDK exposes paginated `page_token` results. The EronFan comment and external fork PR are useful prior art, but that PR remains open outside `openclaw/openclaw` and the fix has not landed here.

## What This Changes

The issue reports that `feishu_doc` write only clears the first page of Feishu Docx blocks before inserting replacement content, leaving old blocks in large documents.

## Best Possible Solution

Land a focused Feishu plugin fix in `extensions/feishu/src/docx.ts`: paginate `client.docx.documentBlock.list` inside `clearDocumentContent` until no next page token remains, delete the full top-level range, and add regression coverage where `feishu_doc` write receives multiple block-list pages before inserting replacement content.

## Reproduction Assessment

Yes. A source-level mocked reproduction is high confidence: make `documentBlock.list` return a first page with `data.page_token` and additional root blocks on a second page, then current `writeDoc` will call the list method once and delete only the first page count.

## Solution Assessment

Yes. Paginating `clearDocumentContent` is the narrowest maintainable fix for the reported write behavior; the external fork patch is directionally right, but the in-repo solution should include regression coverage before landing.
