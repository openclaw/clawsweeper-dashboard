## Summary

Keep open. Current `main` still clears Feishu Docx content with a single non-paginated `documentBlock.list` call before replacement content is inserted, and an external open PR already proposes the narrow fix but has not landed in `openclaw/openclaw`.

## What This Changes

Fix `feishu_doc` write so replacing a large document clears every top-level block, not only the first paginated block-list response.

## Best Possible Solution

Review and land the linked external fix PR or apply the same focused Feishu plugin change in `extensions/feishu/src/docx.ts`, with regression coverage for a multi-page block-list response during `write`.

## Reproduction Assessment

Yes. A high-confidence mocked reproduction is to have `documentBlock.list` return root children plus `data.page_token` on the first call and more root children on the next page; current `writeDoc` calls the list method once, so `batchDelete` only receives the first-page count.

## Solution Assessment

Yes. Paginating `clearDocumentContent` until no next page token remains is the narrowest maintainable fix for the replace-all contract; the linked external PR is directionally right but should be reviewed or imported with a regression test before this closes.
