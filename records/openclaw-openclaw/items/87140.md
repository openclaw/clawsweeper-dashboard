- `P2`: This is a normal-priority feature request for a real macOS Push-to-Talk limitation with bounded impact and no broken default workflow.
- `impact:auth-provider`: The requested behavior centers on STT provider/backend selection, Gateway media audio routing, and local/private ASR model choice.

## Summary

Keep open: current main and the latest release still route macOS Push-to-Talk through Apple Speech and forward only recognized text, while the requested configurable STT backend would add native app, Gateway/provider, config, and security policy surface that needs maintainer design.

## What This Changes

The issue requests configurable speech-to-text backends for native macOS Push-to-Talk while preserving Apple Speech as the default.

## Best Possible Solution

Design a narrow macOS PTT backend selector that keeps Apple Speech as the default, reuses existing Gateway media or Talk transcription contracts where maintainable, and documents privacy, config, permission, and failure-state behavior.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source, docs, and latest-release inspection confirm native macOS PTT is Apple Speech-backed and not configurable through tools.media.audio today.

## Solution Assessment

Unclear until maintainers choose the product/API shape. Reusing existing Gateway media-understanding or Talk transcription contracts looks safer than a separate macOS-only ASR protocol, but config, privacy, and overlay behavior need owner review.
