## Summary

Keep open: current main and the latest release still route native macOS Push-to-Talk through Apple Speech, while `tools.media.audio` and Talk transcription remain separate Gateway/provider paths. The request is valid, but w2jmoe's capability-routing comment and the microphone-audio/provider surface make this a maintainer product/API/security decision rather than conservative cleanup.

## What This Changes

The issue requests a configurable speech-to-text backend for native macOS Push-to-Talk instead of always using Apple Speech.

## Best Possible Solution

Design a narrow PTT backend selector that keeps Apple Speech as the default, reuses existing Gateway media-audio or Talk transcription contracts where maintainers choose, and documents permissions, privacy, credentials, and failure-state behavior before implementation.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source and latest-release inspection give a high-confidence behavior check that native macOS PTT is Apple Speech-backed and not configurable through `tools.media.audio`.

## Solution Assessment

Unclear until maintainers choose the product/API shape. Reusing existing Gateway media-audio or Talk transcription contracts looks safer than adding a parallel macOS-only ASR stack, but the config, privacy, and overlay-state behavior need owner review.
