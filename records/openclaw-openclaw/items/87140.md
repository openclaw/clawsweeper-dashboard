## Summary

Keep open: current main and v2026.6.10 still use Apple Speech for native macOS Push-to-Talk, and the existing `tools.media.audio`/Talk transcription paths are separate Gateway/provider surfaces. The request is valid but needs maintainer product/API/security design before implementation.

## What This Changes

Request native macOS Push-to-Talk to support a configurable STT backend, preferably reusing OpenClaw media audio while keeping Apple Speech as the default.

## Best Possible Solution

Design a narrow native PTT STT boundary that keeps Apple Speech as the default, reuses Gateway media-audio or Talk transcription contracts where maintainers choose, and documents credential, network, permission, fallback, and overlay-state behavior before implementation.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source and latest-release inspection give a high-confidence behavior check that native macOS PTT is Apple Speech-backed and not configurable through `tools.media.audio`.

## Solution Assessment

Unclear until maintainers choose the product/API/security shape. Reusing existing Gateway media-audio or Talk transcription contracts looks safer than adding a parallel macOS-only ASR stack, but the config, privacy, and overlay-state behavior need owner review.
