## Summary

Keep open: current main and v2026.6.8 still run macOS Push-to-Talk through Apple Speech and forward recognized text, while Gateway `tools.media.audio` and Talk transcription remain separate provider paths. The request is valid, but it needs maintainer product, API, and security design before implementation.

## What This Changes

Request to let native macOS Push-to-Talk use a configurable STT backend, including Gateway `tools.media.audio` or local/private ASR, while keeping Apple Speech as the default.

## Best Possible Solution

Design a narrow macOS PTT STT backend contract that preserves Apple Speech by default, reuses existing Gateway media-understanding or Talk provider contracts where appropriate, and documents privacy, credentials, transport policy, overlay states, and failure behavior.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source, docs, related search, and v2026.6.8 inspection give a high-confidence feature-gap check showing PTT is Apple Speech-backed and not configurable through `tools.media.audio`.

## Solution Assessment

Unclear until maintainers choose the product/API shape. Reusing existing Gateway media-understanding or Talk transcription contracts looks safer than a parallel macOS-only ASR protocol, but the final owner boundary and security posture need review.
