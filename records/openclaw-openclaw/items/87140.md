## Summary

Current source confirms the requested backend seam is still missing, so this should not close as implemented or ClawHub-only. The remaining decision is product/API shape: native macOS capture lives in the app, while existing Gateway media and Talk transcription surfaces may be reusable but are not wired into Push-to-Talk today.

## What This Changes

Request configurable speech-to-text backends for native macOS Push-to-Talk while preserving Apple Speech as the default live-partial path.

## Best Possible Solution

Design a narrow native PTT backend selector that keeps Apple Speech as the default, reuses an existing Gateway transcription surface where possible, and documents the security/permission behavior for any remote or local backend.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source inspection confirms current main's native PTT path is Apple Speech-backed and not configurable through `tools.media.audio`.

## Solution Assessment

Unclear until maintainers choose the product/API shape. Reusing existing Gateway transcription or Talk session surfaces looks preferable to inventing a separate macOS-only ASR protocol, but the config, privacy, and overlay-state behavior need owner review.
