## Summary

Keep open: current main still hard-wires macOS Push-to-Talk capture to Apple Speech and forwards only recognized text, while the existing `tools.media.audio` and plugin runtime transcription surfaces are separate Gateway/media-understanding paths. The request remains useful, but it needs maintainer product/API review rather than an automated fix PR.

## What This Changes

Add a configurable speech-to-text backend for native macOS Push-to-Talk, including a Gateway-backed batch transcription option while keeping Apple Speech as the default.

## Best Possible Solution

Design a narrow macOS PTT backend selector that keeps Apple Speech as the default, reuses Gateway mediaUnderstanding or Talk transcription where appropriate, and documents config, permissions, and failure-state behavior.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source and docs inspection confirm current main's native PTT path is Apple Speech-backed and not configurable through `tools.media.audio`.

## Solution Assessment

Unclear until maintainers choose the product/API shape. Reusing existing Gateway mediaUnderstanding or Talk transcription surfaces looks preferable to a separate macOS-only ASR protocol, but config, privacy, and overlay-state behavior need owner review.
