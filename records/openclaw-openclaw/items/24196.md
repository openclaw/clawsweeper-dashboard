## Summary

Keep open: current main and v2026.6.6 still expose Discord outbound uploads through scalar `mediaUrl`/`filename` paths, and multi-media payloads are still sequenced into separate sends rather than one native multi-file Discord message.

## What This Changes

The issue requests native Discord support for sending up to Discord's per-message multi-file attachment limit instead of exposing only one attachment per message.

## Best Possible Solution

Add a Discord-owned ordered attachment batch send path that posts up to the verified Discord per-message file limit per request, splits overflow into follow-up messages, and preserves existing scalar callers plus media safety checks.

## Reproduction Assessment

Yes, at source level: a Discord payload with multiple `mediaUrls` is planned as separate media units and the Discord sender itself only accepts one `mediaUrl` and builds one uploaded file. I did not run a live Discord send in this read-only review.

## Solution Assessment

Yes. The narrow maintainable solution is native batching in the Discord plugin while preserving the existing scalar send contract; a new config knob or generic gateway-wide batching behavior is not needed for this request.
