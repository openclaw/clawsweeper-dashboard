## Summary

Keep open. Current main still routes ordinary Discord uploads through scalar `mediaUrl`/`filename` paths, and shared outbound delivery still fans multiple media URLs into repeated one-file sends, so native multi-attachment Discord messages remain unimplemented.

## What This Changes

Allow Discord outbound messages to attach multiple files in one native Discord message instead of exposing only one file per send.

## Best Possible Solution

Implement Discord-owned multi-file batching for ordered attachments, splitting overflow into follow-up messages while keeping existing scalar send behavior compatible.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a payload with two `mediaUrls` is planned as two media units and delivered through two `sendMedia` calls, while the normal Discord action/runtime/send stack has no multi-attachment input shape.

## Solution Assessment

Yes. A Discord-owned native multi-attachment send path is the narrowest maintainable direction; shared/core changes should be limited to an additive typed seam if Discord needs to receive an ordered batch without changing other channels.
