## Summary

Keep open: current main and the latest release still route Discord uploads through scalar `mediaUrl` paths and one-file REST upload bodies. The issue remains the canonical tracker because https://github.com/openclaw/openclaw/pull/96709 is open but unmerged, while the earlier attempt at https://github.com/openclaw/openclaw/pull/94122 is closed unmerged.

## What This Changes

The issue requests native Discord delivery of up to ten file attachments in a single message instead of exposing only one attachment per send.

## Best Possible Solution

Land a Discord-plugin-owned native batching implementation with focused tests and real Discord proof, then close this issue from the merged PR.

## Reproduction Assessment

Yes, at source level: current main still accepts and forwards scalar Discord `mediaUrl` values, builds one uploaded file, and fans shared `mediaUrls` into separate sends. I did not run a live Discord bot round trip in this read-only review.

## Solution Assessment

Yes: a Discord-owned native batch send is the maintainable fix boundary, while broad core fanout changes or a new config knob would be riskier unless the plugin cannot express batching locally.
