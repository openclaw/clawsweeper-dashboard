## Summary

Keep open. Current main still accepts multiple media references upstream, but Discord delivery remains scalar or sends one file per message, so the requested native multi-file Discord message behavior is not implemented.

## What This Changes

Request to let Discord sends attach multiple files in one message up to Discord's native attachment limit.

## Best Possible Solution

Add a Discord-owned multi-attachment send path that preserves scalar inputs, batches ordered media up to the verified per-message cap, splits overflow into follow-up messages, and updates focused tests, docs, and changelog.

## Reproduction Assessment

Yes, at source level. Current main can carry multiple media references into outbound payloads, but regular Discord action/runtime/send paths reduce delivery to one mediaUrl or sequential one-file sends.

## Solution Assessment

Yes. A Discord-owned batching implementation is the narrowest maintainable direction; avoid a broad shared outbound rewrite or a new config option unless maintainers explicitly want a tunable cap.
