## Summary

Keep open. Current main still models ordinary Discord uploads as one `mediaUrl`/`filename` and shared outbound multi-media payloads are still planned as repeated one-file sends, matching the prior maintainer and ClawSweeper comments that this feature remains unimplemented.

## What This Changes

The issue requests native Discord support for sending up to Discord’s per-message attachment limit instead of exposing or delivering only one file per message.

## Best Possible Solution

Add Discord-owned multi-file batching that accepts ordered attachments, sends up to the verified Discord per-message limit in one `files` array, splits overflow into follow-up messages, and keeps scalar send behavior compatible.

## Reproduction Assessment

Yes for the current limitation. Source inspection on current main shows multi-media outbound payloads become multiple `sendMedia` calls, while the Discord action/runtime/send stack only accepts one `mediaUrl`.

## Solution Assessment

Yes. A Discord-owned native multi-attachment path is the narrowest maintainable direction; shared/core changes should be limited to an additive typed seam only if Discord needs to receive ordered batches from existing outbound payloads.
