## Summary

Keep open. Current main still carries multiple outbound media references, but the Discord send paths reduce ordinary uploads to one scalar media URL or fan them out as repeated one-file sends instead of one native Discord multi-file message.

## What This Changes

Allow Discord sends to attach up to Discord's native multi-file message limit in one message instead of exposing or delivering only one file at a time.

## Best Possible Solution

Add a Discord-owned multi-attachment send path that preserves scalar inputs, batches ordered media up to the verified Discord cap, splits overflow into follow-up messages, and updates focused tests, docs, and changelog.

## Reproduction Assessment

Yes, at source level. Current main can carry multiple media references into outbound payloads, but regular Discord action/runtime/send paths reduce delivery to one mediaUrl or repeated one-file sends.

## Solution Assessment

Yes. A Discord-owned batching implementation is the narrowest maintainable direction; avoid a broad shared outbound rewrite or a new config option unless maintainers explicitly want a tunable cap.
