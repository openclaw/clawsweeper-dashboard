## Summary

Keep open. Current main still exposes Discord outbound uploads through scalar `mediaUrl`/`filename` paths and shared outbound delivery still fans multi-media payloads out into repeated single-file sends, so the requested native multi-attachment Discord message behavior remains unimplemented.

## What This Changes

The issue requests native Discord support for sending up to Discord's per-message maximum number of file attachments instead of exposing only one file attachment per message.

## Best Possible Solution

Add native multi-file sending in the bundled Discord plugin: accept ordered multi-attachment input while preserving existing scalar `media`/`path`/`filePath`/`filename` behavior, load each file through existing outbound media helpers and access checks, send up to the verified Discord per-message attachment limit in one `files` array, split overflow into follow-up messages, and document the behavior.

## Reproduction Assessment

Yes. Static reproduction is high confidence: a payload with two `mediaUrls` is currently planned as two media units and delivered through two `sendMedia` calls, while the Discord action/runtime/send stack has no multi-attachment input shape.

## Solution Assessment

Yes. A Discord-owned multi-attachment send path is the narrowest maintainable fix; shared/core changes should be limited to a typed outbound adapter seam only if needed to avoid changing other channels' behavior.
