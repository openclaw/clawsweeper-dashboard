## Summary

Keep open. Current main still carries multiple media references upstream, but the Discord send/action path remains scalar or fans media out as one file per message, so the requested native multi-file Discord message behavior is not implemented.

## What This Changes

The issue requests native Discord support for sending up to Discord's per-message attachment limit instead of exposing only one file attachment per message.

## Best Possible Solution

Add a Discord-owned multi-attachment send path that batches ordered media up to the verified Discord cap, splits overflow into follow-up messages, and preserves existing scalar inputs and media safety checks.

## Reproduction Assessment

Yes, at source level. Current main can carry multiple media references into outbound payloads, but regular Discord action/runtime/send paths reduce delivery to one `mediaUrl` or sequential one-file sends.

## Solution Assessment

Yes. A Discord-owned batching implementation is the narrowest maintainable direction; avoid a broad shared outbound rewrite or new config option unless maintainers explicitly want a tunable cap.
