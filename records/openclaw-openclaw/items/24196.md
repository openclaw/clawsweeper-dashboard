## Summary

Keep open. Current main has multi-media payload plumbing, but Discord still converts normal sends to scalar `mediaUrl` uploads or sequential one-file sends, so native Discord multi-attachment delivery remains unimplemented.

## What This Changes

Issue requests native Discord support for sending up to Discord's per-message attachment limit instead of exposing only one file attachment.

## Best Possible Solution

Add a Discord-owned multi-attachment send path that preserves scalar inputs, batches ordered files up to the verified Discord per-message cap, splits overflow, and updates focused tests, docs, and changelog.

## Reproduction Assessment

Yes, at source level. Current main accepts multi-media payloads upstream, but the Discord action/runtime/send helper paths still reduce normal uploads to one mediaUrl or repeated one-file sends; I did not run a live Discord delivery.

## Solution Assessment

Yes. A Discord-owned batching path is the narrowest maintainable solution; avoid a broad core fan-out rewrite and avoid a new config option unless maintainers explicitly want a tunable cap.
