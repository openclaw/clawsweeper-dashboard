## Summary

Keep open. Current main still routes ordinary Discord message uploads through scalar `mediaUrl`/`filename` inputs and one-file upload sends, while shared `mediaUrls` delivery still fans out as repeated media sends instead of one native Discord multi-attachment message.

## What This Changes

The issue requests allowing Discord messages to send multiple file attachments in one native message instead of one file per message.

## Best Possible Solution

Add a Discord-owned multi-attachment send path that preserves existing scalar media inputs, batches ordered files up to the verified Discord per-message cap, splits overflow, and updates focused tests, docs, and changelog.

## Reproduction Assessment

Yes. Source inspection on current `main` shows the standard Discord send action/runtime and shared outbound delivery paths are scalar or fan out `mediaUrls` into repeated one-file sends; I did not run a live Discord send in this read-only review.

## Solution Assessment

Yes. A Discord-owned batching path is the narrowest maintainable solution; avoid a broad core rewrite and avoid adding config unless maintainers explicitly want a tunable cap.
