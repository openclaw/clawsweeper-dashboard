## Summary

Keep open: current main and the latest release still route Discord file sends through scalar `mediaUrl`/`filename` paths and build one-file upload bodies, while the attempted fix at https://github.com/openclaw/openclaw/pull/94122 is closed unmerged and no longer owns the work.

## What This Changes

Requested Discord support for sending up to Discord's native maximum file attachments in one message instead of requiring one message per file.

## Best Possible Solution

Implement a Discord plugin-owned native batch upload path that accepts ordered multi-attachment input, sends up to the verified per-message limit in one Discord `files` array, splits overflow without replaying successful batches, preserves scalar aliases, updates focused tests/docs, and leaves release notes to PR-body or release tooling.

## Reproduction Assessment

Yes, at source level: current main accepts and forwards scalar Discord `mediaUrl` values, `sendDiscordMedia` builds one uploaded file, and shared outbound planning expands `mediaUrls` into separate media sends. I did not run a live Discord bot round trip in this read-only review.

## Solution Assessment

Yes. A Discord-owned native batch send is the maintainable fix; a broad core fanout change or new config knob is riskier unless the Discord plugin cannot express batching locally.
