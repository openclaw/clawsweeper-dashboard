- `impact:message-loss`: The current path fragments a multi-file response across separate messages instead of preserving the intended attachment group.

## Summary

Current main and v2026.6.5 still send ordinary Discord media through a scalar one-file request, while the only documented multi-file path is the separate interactive `media-gallery` surface; the requested native batching remains useful and unimplemented.

## What This Changes

Allow ordinary Discord outbound messages to include multiple file attachments in one native message, splitting overflow into ordered follow-up messages.

## Best Possible Solution

Add an ordered multi-attachment input at the Discord adapter/send boundary, load every file through existing guarded media helpers, send verified-size batches as one native `files` array, and preserve the current scalar path as the one-item case without adding configuration.

## Reproduction Assessment

Yes, at source level with high confidence: current main accepts one ordinary Discord media URL, constructs a one-file request, and plans multiple generic URLs as separate delivery units. A live Discord account is useful for final proof but is not needed to establish the missing capability.

## Solution Assessment

Yes. A Discord-plugin-owned batching seam is the narrowest maintainable solution; changing generic outbound fan-out or adding attachment-count configuration would create unnecessary cross-channel or config surface.
