## Summary

Keep open. Current main still routes ordinary Discord uploads through scalar `mediaUrl` paths or sequential one-file sends, so native multi-file Discord message delivery remains unimplemented and no related PR appears to supersede this issue.

## What This Changes

Request Discord message sending support up to the platform's native multi-file attachment behavior instead of exposing only one file per normal send.

## Best Possible Solution

Add a Discord-owned multi-attachment send path that preserves scalar inputs, batches ordered files up to the verified Discord per-message cap, splits overflow, and updates focused tests, docs, and changelog.

## Reproduction Assessment

Yes, at source level. Current main accepts multi-media payloads upstream, but the Discord action/runtime/send helper paths still reduce normal uploads to one `mediaUrl` or repeated one-file sends.

## Solution Assessment

Yes. A Discord-owned batching path is the narrowest maintainable solution; avoid a broad core fan-out rewrite and avoid a new config option unless maintainers explicitly want a tunable cap.
