## Summary

Keep open. Current main still exposes Discord message uploads through scalar `mediaUrl`/`filename` paths and fans shared `mediaUrls` out as repeated one-file sends, so the requested native multi-file Discord message behavior remains unimplemented.

## What This Changes

Allow Discord messages to send multiple file attachments up to Discord's native per-message limit instead of one attachment per send.

## Best Possible Solution

Implement Discord-owned multi-attachment batching that accepts ordered attachments, sends up to the verified Discord per-message limit in one `files` array, splits overflow into follow-up messages, and keeps existing scalar send behavior compatible.

## Reproduction Assessment

Yes for the current limitation: source inspection on current main shows multi-media payloads are planned into separate media sends, while the Discord action/runtime/send stack accepts only one `mediaUrl`. I did not run a live Discord send in this read-only review.

## Solution Assessment

Yes. A Discord-owned native multi-attachment path is the narrowest maintainable direction; shared/core changes should be limited to an additive typed seam only if the Discord plugin cannot otherwise receive ordered batches.
