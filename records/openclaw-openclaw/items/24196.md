## Summary

Keep open. Current main and the latest release still carry multiple media references upstream, but Discord delivery remains scalar or sequences them as separate one-file sends; no linked or searched PR supersedes this issue.

## What This Changes

Allow the bundled Discord plugin to send multiple file attachments in one native Discord message instead of exposing only one file/send or splitting files into separate messages.

## Best Possible Solution

Add a Discord-owned batching path that loads each attachment through existing media helpers, sends files together up to the verified Discord cap, splits overflow into follow-up messages, and keeps scalar inputs compatible.

## Reproduction Assessment

Yes, at source level. Current main preserves multiple attachment hints upstream, but the Discord action/send path still accepts one `mediaUrl` and existing helpers sequence multi-media payloads as separate sends.

## Solution Assessment

Yes. A Discord-owned batching implementation is the narrowest maintainable solution; avoid a new config knob or broad shared outbound behavior change unless the plugin cannot express batching locally.
