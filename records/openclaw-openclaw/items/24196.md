## Summary

Keep open. Current main still accepts and sends Discord media through scalar `mediaUrl`/`filename` paths or one-file-at-a-time media sequencing, so the requested native multi-attachment Discord message behavior remains unimplemented.

## What This Changes

Request native Discord delivery of multiple file attachments in one message instead of exposing only one attachment per Discord send.

## Best Possible Solution

Add a Discord-owned batching path that loads each attachment through existing media helpers, sends files together up to the verified Discord cap, splits overflow into follow-up messages, and keeps scalar inputs compatible.

## Reproduction Assessment

Yes, at source level. Current main can carry multiple media references upstream, but Discord action/send paths still accept one `mediaUrl` and shared helpers sequence multiple media URLs as separate sends.

## Solution Assessment

Yes. A Discord-owned batching implementation is the narrowest maintainable solution; avoid a broad shared outbound rewrite or a new config knob unless maintainers explicitly want tunable limits.
