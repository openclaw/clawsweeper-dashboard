## Summary

Keep open. Current main can carry multiple media references upstream, but the Discord action/send paths still accept one `mediaUrl`/`filename` or fan media out as one file per message, and no merged or canonical open item owns this exact remaining work.

## What This Changes

Request native Discord sends to accept multiple file attachments per message instead of exposing only a single attachment path.

## Best Possible Solution

Add a Discord-owned multi-attachment send path that loads each attachment through existing media helpers, batches files in one Discord `files` array up to the verified cap, splits overflow into follow-up messages, and keeps existing scalar inputs compatible.

## Reproduction Assessment

Yes, at source level. The message tool and runner can carry multiple media references, but Discord action/send types accept one `mediaUrl` and shared outbound planning currently splits multiple media URLs into separate sends.

## Solution Assessment

Yes. A Discord-owned batching implementation is the narrowest maintainable solution; avoid a broad shared outbound rewrite or a new config knob unless maintainers explicitly want tunable limits.
