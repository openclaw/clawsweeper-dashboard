## Summary

Keep open. Current main and the latest shipped tag still expose Discord outbound uploads through scalar media fields and one-file-at-a-time fan-out, while the reporter's attached implementation sketch explicitly stopped before handler/send integration and no open PR specifically implements native Discord multi-file batching.

## What This Changes

The issue requests native Discord message sends that attach up to Discord's 10-file-per-message limit instead of exposing or delivering only one file at a time.

## Best Possible Solution

Implement Discord-owned native multi-attachment batching that keeps scalar sends compatible, batches ordered attachments up to Discord's verified limit, and splits overflow into follow-up messages.

## Reproduction Assessment

Yes for the current limitation: source inspection on current main shows ordinary Discord sends accept one media reference and shared `mediaUrls` are delivered as repeated one-file sends. I did not run a live Discord send in this read-only review.

## Solution Assessment

Yes. A Discord-owned batching path is the narrowest maintainable direction; shared/core changes should be limited to an additive typed seam only if the plugin cannot otherwise receive ordered batches.
