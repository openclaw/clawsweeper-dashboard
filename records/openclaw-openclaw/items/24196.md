## Summary

Keep open. Current main still accepts multiple media references upstream, but regular Discord send/action paths reduce delivery to scalar `mediaUrl` calls or sequential one-file sends, so the native multi-file Discord message behavior remains unimplemented.

## What This Changes

The issue requests Discord messages to support multiple file attachments up to Discord's per-message limit instead of exposing only one file per message.

## Best Possible Solution

Add a Discord-owned multi-attachment send path that preserves scalar inputs, batches ordered media up to the verified Discord per-message cap, splits overflow, and updates focused tests, docs, and changelog.

## Reproduction Assessment

Yes, at source level. Current main can carry multiple media references into outbound payloads, but the regular Discord action/runtime/send paths reduce delivery to one `mediaUrl` or sequential one-file sends.

## Solution Assessment

Yes. A Discord-owned batching implementation is the narrowest maintainable direction; avoid a broad shared outbound fan-out rewrite or a new config option unless maintainers explicitly want a tunable cap.
