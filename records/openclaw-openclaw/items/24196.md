- `impact:message-loss`: The affected surface is Discord file delivery, where multi-file attachments cannot be delivered in the single native message shape users requested.

## Summary

Keep open. Current main still represents multiple outbound media items upstream, but Discord delivery still funnels them through scalar `mediaUrl`/`filename` sends or one-file-at-a-time sequencing, so the requested native multi-file Discord message behavior remains unimplemented.

## What This Changes

Allow Discord delivery to send multiple file attachments in one native Discord message up to Discord's verified per-message cap, instead of only scalar single-file paths or separate messages.

## Best Possible Solution

Add a Discord-owned batching path that loads each attachment through existing media helpers, sends files together up to the verified Discord cap, splits overflow into follow-up messages, and keeps scalar inputs compatible.

## Reproduction Assessment

Yes, at source level. Current main can carry multiple media references upstream, but Discord action/send paths still accept one `mediaUrl` and shared helpers sequence multiple media URLs as separate sends.

## Solution Assessment

Yes. A Discord-owned batching implementation is the narrowest maintainable solution; avoid a broad shared outbound rewrite or a new config knob unless maintainers explicitly want tunable limits.
