## Summary

Keep open. Current main still carries multiple media references through shared send payloads, but Discord delivery reduces them to one scalar `mediaUrl` or repeated one-file sends instead of one native multi-file Discord message.

## What This Changes

The issue requests Discord sends to support multiple file attachments in one message up to Discord's native per-message limit.

## Best Possible Solution

Add a Discord-owned multi-attachment send path that preserves scalar inputs, batches ordered media up to the verified per-message cap, splits overflow into follow-up messages, and updates focused tests, docs, and changelog.

## Reproduction Assessment

Yes, at source level. Current main can carry multiple media references into outbound payloads, but the regular Discord action/runtime/send paths reduce delivery to one `mediaUrl` or sequential one-file sends.

## Solution Assessment

Yes. A Discord-owned batching implementation is the narrowest maintainable direction; avoid a broad shared outbound fan-out rewrite or a new config option unless maintainers explicitly want a tunable cap.
