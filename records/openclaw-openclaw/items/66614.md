## Summary

Keep open. Current main and latest release still use fence-only safe-break handling in the shared block streaming chunker, while the already-open fix path is https://github.com/openclaw/openclaw/pull/73981 with related integration coverage tracked by https://github.com/openclaw/openclaw/issues/66605.

## What This Changes

The issue asks OpenClaw to stop block streaming from splitting markdown tables across separate channel messages.

## Best Possible Solution

Land a repaired shared EmbeddedBlockChunker table-span safe-break fix, add the subscribeEmbeddedPiSession markdown-table integration case from https://github.com/openclaw/openclaw/issues/66605, and preserve credit from https://github.com/openclaw/openclaw/pull/66568.

## Reproduction Assessment

Yes, at source level. Feeding `Intro\n\n| A | B |\n|---|---|\n| 1 | 2 |\n\nOutro` through EmbeddedBlockChunker with flushOnParagraph enabled still reaches a paragraph-break path that only checks `isSafeFenceBreak`.

## Solution Assessment

Yes. The narrow maintainable solution belongs in the shared chunker/table-span layer because channel-level table conversion happens after chunk boundaries are already selected.
