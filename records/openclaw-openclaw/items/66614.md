## Summary

Keep open. Current main and v2026.5.12 still use fence-only safe-break handling in the shared block streaming chunker, and the remaining fix path is already represented by https://github.com/openclaw/openclaw/pull/73981 with integration coverage tracked by https://github.com/openclaw/openclaw/issues/66605.

## What This Changes

The issue asks for block streaming to keep a markdown table-containing assistant reply together instead of splitting it into separate channel messages around paragraph boundaries.

## Best Possible Solution

Repair and land the shared `EmbeddedBlockChunker` table-safe break fix, with the subscribe integration guard from https://github.com/openclaw/openclaw/issues/66605 and preserved credit from https://github.com/openclaw/openclaw/pull/66568.

## Reproduction Assessment

Yes, at source level. Feeding `Intro\n\n| A | B |\n|---|---|\n| 1 | 2 |\n\nOutro` through the current flush-on-paragraph chunker reaches a blank-line split path that checks only fence spans, not table spans.

## Solution Assessment

Yes, the shared chunker/table-span layer is the right fix surface. Channel-specific table conversion runs too late to choose safe streaming chunk boundaries consistently across channels.
