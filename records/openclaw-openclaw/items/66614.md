## Summary

Keep open. Current main still has fence-only safe-break handling in the shared block streaming chunker, with no markdown table span parser or table-safe boundary guard; the related open PRs are implementation candidates, not shipped behavior, and #66344 uses closing syntax for this issue.

## What This Changes

The issue requests that block streaming preserve markdown tables as atomic chunks across streaming channel messages instead of splitting on paragraph breaks around or inside tables.

## Best Possible Solution

Keep this issue open until a narrow shared chunker fix lands. The desired end state is table span detection in the shared markdown/chunker layer, table-safe guards for paragraph/newline/sentence/whitespace/max-length break paths, and the #66605 integration coverage, while preserving useful implementation credit from #66568.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to instantiate EmbeddedBlockChunker with flushOnParagraph and append text like `Intro\n\n| A | B |\n|---|---|\n| 1 | 2 |\n\nOutro`; current main will treat the table-adjacent blank lines as safe because only fence spans are checked.

## Solution Assessment

No current-main solution is present. The maintainable fix is table-span safe-break logic in the shared block chunker plus regression tests; channel-level convertMarkdownTables handling is too late because the chunker already chose message boundaries.
