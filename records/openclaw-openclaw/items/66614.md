## Label Justifications

- `P2`: The bug breaks table rendering across streaming channels but does not crash the runtime or block all messaging.
- `impact:message-loss`: The affected behavior changes one intended assistant reply into fragmented channel messages with inconsistent table rendering.

## Summary

Keep open. Current main and the latest release still use fence-only safe-break handling in the shared block streaming chunker, so the reported table-splitting path remains present; the active fix path is the open replacement PR plus the companion integration-test issue.

## What This Changes

The issue asks OpenClaw to keep Markdown tables intact when block-streamed replies are split across channel messages.

## Best Possible Solution

Land a shared `EmbeddedBlockChunker` table-span safe-break fix with the subscribe integration guard from https://github.com/openclaw/openclaw/issues/66605, then close this issue after the merged fix is on main or shipped.

## Reproduction Assessment

Yes, at source level. Feeding `Intro\n\n| A | B |\n|---|---|\n| 1 | 2 |\n\nOutro` through the current flush-on-paragraph path reaches paragraph-break logic that checks only fence spans, not table spans.

## Solution Assessment

Yes, the shared chunker/table-span layer is the right fix surface. Channel-specific table conversion happens after chunk boundaries are chosen, so it cannot reliably preserve tables across all streaming channels.
