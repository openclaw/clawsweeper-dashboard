## Summary

Keep open. Current main still has fence-only safe-break handling in the shared block streaming chunker, no table-span parser or table-safe guard, and the related implementation/coverage work remains open rather than shipped.

## What This Changes

The issue reports that block streaming treats Markdown table boundaries as safe paragraph breaks, splitting one assistant reply into multiple channel messages.

## Best Possible Solution

Land a narrow shared chunker/markdown span fix that detects table spans, guards all safe-break paths, and adds the #66605 integration case while preserving useful credit from #66568.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to append `Intro\n\n| A | B |\n|---|---|\n| 1 | 2 |\n\nOutro` to EmbeddedBlockChunker with flushOnParagraph enabled; current main treats the table-adjacent blank lines as safe because only fence spans are checked.

## Solution Assessment

Yes, the reported fix direction is the right boundary. The maintainable solution belongs in the shared block chunker/table-span layer because channel-level convertMarkdownTables runs after the chunker has already chosen message boundaries.
