## Summary

Keep open. Current main and the latest release still use fence-only safe-break handling in the shared block streaming chunker, and an open replacement fix path already exists at https://github.com/openclaw/openclaw/pull/73981 with related integration coverage tracked by https://github.com/openclaw/openclaw/issues/66605.

## What This Changes

The issue requests table-aware block streaming so markdown tables emitted in one assistant response are not split across separate channel messages.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/73981 with a shared EmbeddedBlockChunker table-span safe-break fix plus the integration guard from https://github.com/openclaw/openclaw/issues/66605, preserving credit from https://github.com/openclaw/openclaw/pull/66568.

## Reproduction Assessment

Yes, at source level. Feeding `Intro\n\n| A | B |\n|---|---|\n| 1 | 2 |\n\nOutro` through the current flushOnParagraph chunker still reaches a paragraph-break path that only checks fence spans, not table spans.

## Solution Assessment

Yes, the shared chunker/table-span layer is the right fix surface. A channel-by-channel conversion workaround would run after chunk boundaries are already selected and would not address all streaming channels consistently.
