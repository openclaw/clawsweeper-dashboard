## Summary

Keep open. Current main still has fence-only safe-break handling in the shared block streaming chunker, with no markdown table span parser or table-safe boundary guard; the open related PRs are implementation candidates, not shipped behavior.

## What This Changes

The issue reports that block streaming treats markdown-table paragraph boundaries as safe flush points, causing one assistant reply with text, table, and following text to be delivered as multiple messages across streaming channels.

## Best Possible Solution

Keep this issue open and review or land a narrow shared chunker fix, likely via #73981 or #66344 while preserving the useful implementation credit from #66568. The end state should add table span detection in the shared markdown/chunker layer, guard EmbeddedBlockChunker paragraph/newline/sentence/whitespace/max-length break paths with table-safe boundaries, and include the integration coverage requested by #66605.

## Reproduction Assessment

Yes. A focused reproduction is to instantiate EmbeddedBlockChunker with flushOnParagraph, append text containing `Intro\n\n| A | B |\n|---|---|\n| 1 | 2 |\n\nOutro`, and drain; current source will consider the table-adjacent blank lines safe because only fence spans are checked.

## Solution Assessment

No current-main solution is present. convertMarkdownTables() is not the right fix for this path because block streaming appends raw assistant chunks directly into EmbeddedBlockChunker; the maintainable fix is table-span safe-break logic in the shared chunker plus regression tests.
