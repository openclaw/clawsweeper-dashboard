## Summary

Keep this PR open: the bug direction is valid, but the branch retries compaction only on the non-completion steer path and leaves the linked generated-completion active-wake path treating compaction as a failed wake.

## What This Changes

The PR adds a bounded compaction retry loop to active requester steering and two regression tests in subagent announce delivery.

## Best Possible Solution

Share a compaction-aware queue helper across both active requester wake paths, preserve transcript-wait best-effort fallback after compaction, and prove it with an expectsCompletionMessage generated-media regression plus real behavior evidence.

## Reproduction Assessment

Yes. Source inspection shows expectsCompletionMessage dispatch uses sendSubagentAnnounceDirectly first, and that active wake path still treats a compacting queue outcome as a failed wake on PR head.

## Solution Assessment

No. The retry idea is right, but the current patch applies it to only one steering path and should instead share the compaction retry with the generated-completion active wake while preserving the transcript-wait fallback.
