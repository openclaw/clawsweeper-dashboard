## Summary

Keep open: current main still lacks proactive active-channel retry/failover/recovery notifications and the requested notification config; existing fallback visibility is lifecycle/status data plus verbose-only final notices. The open implementation at https://github.com/openclaw/openclaw/pull/69399 is narrower and still under review, so it does not close this broader policy request.

## What This Changes

Request proactive user-channel notifications and configuration for LLM API errors, fallback transitions, and optional recovery back to the primary model.

## Best Possible Solution

Define one canonical sanitized retry/failover notification policy in the core reply-runner/model-fallback path, coordinate it with https://github.com/openclaw/openclaw/pull/69399, and document/test the default-false config surface.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request rather than broken documented behavior. Source inspection and current e2e tests give high confidence that non-verbose active-channel retry/failover notices are absent today.

## Solution Assessment

Yes in direction: the gateway/reply-runner fallback path is the right boundary because it sees failed attempts and fallback transitions before the fallback response. The exact config location, redaction, timing, recovery semantics, and interaction with the open narrower PR need maintainer judgment.
