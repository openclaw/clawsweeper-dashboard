## Summary

Keep open. Current main has fallback attempt tracking, lifecycle events, and verbose-only final fallback notices, but it still lacks proactive non-verbose active-channel retry/failover notices and the requested notification policy/config; the related open PR is narrower than this issue.

## What This Changes

The issue requests proactive user-facing notices for provider API failures, delayed retries, model failover, and optional recovery during LLM fallback.

## Best Possible Solution

Define a sanitized core retry/fallback notice policy in the reply-runner/model-fallback path, coordinate it with https://github.com/openclaw/openclaw/pull/69399, and add config/docs plus regression coverage for non-verbose active-channel delivery.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken documented behavior. Source inspection and current tests show fallback exists, but non-verbose active-channel retry/failover notices are intentionally absent.

## Solution Assessment

Partly yes: gateway-level active-channel delivery is the right boundary because the fallback runner sees failed-attempt and transition signals before the fallback response. The exact config surface, defaults, redaction, retry timing, recovery behavior, and overlap with https://github.com/openclaw/openclaw/pull/69399 need maintainer judgment.
