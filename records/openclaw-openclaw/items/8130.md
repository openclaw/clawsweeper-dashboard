## Summary

Keep open: current main still has fallback lifecycle/log/status visibility and verbose-only final reply notices, but it does not proactively send active-channel retry/failover/recovery notices for normal users and it has no requested notification config. The open PR and related issues cover narrower or broader slices, so this remains useful as the user-notification policy thread.

## What This Changes

Request proactive active-channel notifications and optional config when LLM provider errors, retries, fallback transitions, or recovery occur.

## Best Possible Solution

Define one default-false, sanitized notification policy in the core reply-runner/model-fallback path, covering delayed retry notices, successful fallback notices, recovery notices, docs, and regression tests while coordinating with the narrower opt-in notice PR and hook/context work.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature/config request. Source inspection gives high confidence that current main still lacks non-verbose proactive active-channel retry/failover/recovery notices and the requested config surface.

## Solution Assessment

Yes in direction: the reply-runner/model-fallback boundary is the right place because it sees failed attempts and fallback transitions before the fallback response. The exact config key, default, redaction, delayed retry timing, recovery behavior, and coordination with related work need maintainer review.
