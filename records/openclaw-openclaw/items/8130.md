## Summary

Keep this open. Current main still has fallback lifecycle/status visibility and verbose-only final notices, but it still does not proactively notify the active non-verbose user channel during provider retry/failover and the requested notification policy/config surface is absent.

## What This Changes

Request active-channel retry, failover, and optional recovery notifications when configured LLM fallbacks are used after provider/API errors.

## Best Possible Solution

Add a sanitized, configurable fallback-notice policy in the core reply-runner/model-fallback path, with regression coverage proving non-verbose users receive retry/fallback notices without duplicating existing observability.

## Reproduction Assessment

Not applicable: this is a feature request, not broken documented behavior. Source inspection and existing e2e assertions show current non-verbose fallback replies intentionally remain silent while lifecycle, /status, logs, and verbose observability exist.

## Solution Assessment

Partly yes: gateway-level active-channel delivery is the right boundary because the fallback runner sees failed-attempt and successful-fallback transitions before the fallback model starts. The exact config shape, default policy, and message wording still need maintainer judgment.
