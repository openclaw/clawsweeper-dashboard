## Summary

Keep this open. Current main still has model fallback tracking, lifecycle events, status visibility, and verbose-only final fallback notices, but it does not proactively notify non-verbose active-channel users during provider retry/failover and the requested notification config is absent.

## What This Changes

The issue requests proactive active-channel notifications and optional config for LLM retry, failover, and recovery events.

## Best Possible Solution

Add a configurable, sanitized active-channel notice policy in the core reply-runner/model-fallback path, with tests proving non-verbose users receive retry/fallback notices without duplicating verbose or lifecycle observability.

## Reproduction Assessment

Not applicable as a feature request. Source inspection and existing e2e coverage give high confidence that current non-verbose fallback behavior remains silent while lifecycle and verbose observability exist.

## Solution Assessment

Yes for the core boundary, but the exact config/default policy needs maintainer judgment. A gateway-level implementation is more reliable than model self-announcement because the fallback runner has error and fallback signals before the fallback model starts.
