## Summary

Keep open. Current main still has fallback lifecycle signals and verbose-only final fallback notices, but it does not proactively notify the active non-verbose channel during retry/failover and does not expose the requested retry/failover notification config; the related open PR is narrower than this request.

## What This Changes

Request proactive user-facing notifications for provider API errors, fallback model switches, and optional recovery, controlled by new retry/failover notification config.

## Best Possible Solution

Define and implement a sanitized, configurable retry/fallback notice policy in the core reply-runner/model-fallback path, with docs and regression coverage for non-verbose active-channel delivery.

## Reproduction Assessment

Not applicable: this is a feature request, not broken documented behavior. Source inspection and existing e2e assertions show non-verbose fallback replies intentionally stay silent while lifecycle events and verbose notices exist.

## Solution Assessment

Partly yes: gateway-level active-channel delivery is the right boundary because the fallback runner sees failed-attempt and successful-fallback transitions before the fallback model runs. The exact config placement, defaults, redaction, retry delay, recovery behavior, and relation to https://github.com/openclaw/openclaw/pull/69399 still need maintainer judgment.
