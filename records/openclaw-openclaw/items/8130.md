## Summary

Keep open: current main still only exposes fallback visibility through lifecycle events, /status, and verbose-mode final notices, not proactive non-verbose active-channel retry/failover notifications or the requested notification policy config.

## What This Changes

Request proactive active-channel notifications when provider API calls retry, fail over to a backup model, and optionally recover to the primary model.

## Best Possible Solution

Define a sanitized core retry/failover notice policy in the reply-runner/model-fallback path, with config/docs/tests for non-verbose active-channel delivery and coordinated handling of the narrower open notice PR.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request rather than broken documented behavior. Source inspection and current tests give high confidence that non-verbose active-channel retry/failover notices are intentionally absent today.

## Solution Assessment

Yes in direction: gateway-level delivery is the right boundary because the fallback runner sees failed attempts and transitions before the fallback response. The exact config surface, defaults, delay timing, recovery behavior, and redaction policy need maintainer judgment before implementation.
