## Summary

Keep this open. Current main has model fallback, lifecycle events, and verbose-only final fallback notices, but non-verbose active-channel users still do not get proactive retry/failover notifications and the requested notification config surface is absent.

## What This Changes

The issue requests proactive user-channel notifications and config for provider API retry delays, model failover, and optional primary-model recovery.

## Best Possible Solution

Add a configurable, sanitized active-channel notice policy in the core reply-runner/model-fallback path, with tests proving non-verbose users receive retry/fallback notices without duplicating verbose or lifecycle observability.

## Reproduction Assessment

Not applicable as a feature request. Source inspection and existing e2e coverage give high confidence that current non-verbose fallback behavior remains silent while lifecycle/verbose observability exists.

## Solution Assessment

Yes for the core boundary, but the exact config/default policy needs maintainer judgment. A gateway-level implementation is more reliable than model self-announcement because it can notify before the fallback model starts running.
