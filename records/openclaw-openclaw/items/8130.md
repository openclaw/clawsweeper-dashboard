## Summary

Keep this issue open. Current main has model fallback hooks, lifecycle events, Control UI-style state, and verbose-only final fallback notices, but it still does not proactively notify normal active-channel users during provider retry/failover and the requested notification config is absent.

## What This Changes

Request opt-in active-channel notifications for provider API retry, failover, and recovery events, plus config knobs for delayed retry and failover notices.

## Best Possible Solution

Implement opt-in, sanitized retry/fallback notices in the core reply-runner/model-fallback delivery path while preserving existing lifecycle, verbose, and fallback-state observability.

## Reproduction Assessment

Yes. Source inspection and existing e2e coverage provide a high-confidence path: force runWithModelFallback to return a fallback model with `resolvedVerboseLevel: "off"`, and the final payload lacks `Model Fallback:` while lifecycle fallback events still fire.

## Solution Assessment

Yes. A core reply-runner/model-fallback implementation is the narrowest maintainable direction because the low-level hooks already exist and prompt/plugin workarounds only notice the switch after the fallback model is running.

## Repair Work Prompt

Implement opt-in proactive model retry/failover notices for active user channels. Reuse the existing runWithModelFallback onError/onFallbackStep signals and reply-runner block reply delivery; do not change fallback selection policy or make notices default-on without an explicit existing default. Add strict config support for a failover notification policy, including an enabled flag and delayed retry notice threshold, in the agents defaults model/fallback config area with schema/help/docs alignment. Notices must use sanitized provider/model/error summaries, avoid raw provider error text or credentials, and dedupe against existing verbose final fallback notices, lifecycle events, fallback state, Control UI indicators, and all-models-failed payloads. Add/update regression coverage proving non-verbose users can receive opted-in retry/fallback notices while current verbose-off behavior remains unchanged when the option is disabled. Add a user-facing CHANGELOG.md entry and update public docs for the new config.
