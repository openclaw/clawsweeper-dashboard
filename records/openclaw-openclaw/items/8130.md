## Summary

Keep open. Current main still has fallback tracking, lifecycle events, and verbose-only final fallback notices, but it does not proactively notify the active non-verbose user channel during retry/failover and does not expose the requested retry/failover notification policy; related open PRs are useful but narrower than this issue.

## What This Changes

Request a configurable way to notify users in the active channel when provider API errors, model failover, and optional recovery happen during LLM fallback.

## Best Possible Solution

Define a sanitized core retry/fallback notice policy in the reply-runner/model-fallback path, coordinate it with the narrower open notice and hook PRs, and add docs plus regression coverage for non-verbose active-channel delivery.

## Reproduction Assessment

Not applicable: this is a feature request, not broken documented behavior. Source inspection and current e2e coverage show fallback exists but non-verbose active-channel retry/fallback notices are intentionally absent.

## Solution Assessment

Partly yes: gateway-level active-channel delivery is the right boundary because the fallback runner sees failed-attempt and transition signals before the fallback response. The exact config surface, defaults, redaction, retry timing, recovery behavior, and overlap with related PRs need maintainer judgment.
