## Summary

Keep open. Current main still records fallback attempts and emits lifecycle/verbose final notices, but it does not proactively notify the active non-verbose user channel during retry/failover and does not expose the requested retry/failover notification policy; the open related PR is narrower than this issue.

## What This Changes

Request proactive active-channel notifications when LLM API errors trigger retries, fallback model switches, and optional recovery, with configurable retry/failover notification policy.

## Best Possible Solution

Define a sanitized core retry/fallback notice policy in the reply-runner/model-fallback path, coordinate it with the narrower open notice PR, and add config/docs plus regression coverage for non-verbose active-channel delivery.

## Reproduction Assessment

Not applicable: this is a feature request, not broken documented behavior. Source inspection and current tests show fallback exists, but non-verbose active-channel retry/failover notices are intentionally absent.

## Solution Assessment

Partly yes: gateway-level active-channel delivery is the right boundary because the fallback runner sees failed-attempt and transition signals before the fallback response. The exact config surface, defaults, redaction, retry timing, recovery behavior, and overlap with https://github.com/openclaw/openclaw/pull/69399 need maintainer judgment.
