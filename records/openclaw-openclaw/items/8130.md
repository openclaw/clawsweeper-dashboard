## Summary

Keep open. Current main still has fallback tracking, lifecycle events, verbose-only final fallback notices, and diagnostic docs, but it does not proactively notify the active non-verbose user channel during retry/failover and the requested retry/failover notification config remains absent.

## What This Changes

The issue requests proactive active-channel notifications when model retries, failovers, or recovery happen after LLM provider/API errors.

## Best Possible Solution

Define a sanitized core retry/fallback notice policy in the reply-runner/model-fallback path, coordinate it with the narrower open fallback-notice work, and add docs plus regression coverage for non-verbose active-channel delivery.

## Reproduction Assessment

Not applicable: this is a feature request, not broken documented behavior. Source inspection and current e2e coverage show fallback exists but non-verbose active-channel retry/failover notices are intentionally absent.

## Solution Assessment

Partly yes: gateway-level active-channel delivery is the right boundary because the fallback runner sees failed-attempt and transition signals before the fallback response. The exact config surface, defaults, redaction, retry timing, recovery behavior, and coordination with related open work need maintainer judgment.
