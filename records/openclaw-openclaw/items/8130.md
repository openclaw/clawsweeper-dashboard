## Summary

Keep this issue open. Current main has model fallback state, lifecycle/UI observability, and verbose-only final fallback notices, but it still does not proactively notify the active user channel during retry/failover for normal non-verbose replies and the requested failover notification config is absent.

## What This Changes

Request configurable user-facing notifications when LLM provider calls retry, fail over to backup models, and optionally recover.

## Best Possible Solution

Implement this in the core reply-runner/model-fallback path: add a backwards-compatible notification policy, wire retry/fallback transitions into active-channel block reply delivery, reuse sanitized fallback summaries, preserve lifecycle/verbose observability, and cover non-verbose channel behavior with tests and docs.

## Reproduction Assessment

Yes. The current source-level reproduction is the existing e2e pattern that stubs `runWithModelFallback` to return a fallback model with `resolvedVerboseLevel: "off"`; the test asserts no `Model Fallback:` text reaches the reply payload even though fallback lifecycle events fire.

## Solution Assessment

Yes, a core reply-runner/model-fallback implementation is the best fit because the low-level fallback hooks already exist and an agent prompt or plugin workaround can only notice after the fallback model is running. The safer implementation is opt-in/backwards-compatible, sanitized, and deduped against existing verbose/lifecycle output.
