## Summary

Keep this open: current main still has fallback tracking, lifecycle events, logs, and verbose-only final notices, but it does not send proactive retry/failover notifications to the active user channel for non-verbose users or expose the requested notification config.

## What This Changes

Request configurable active-channel notifications when LLM API errors cause retries, model failover, and optional primary-model recovery.

## Best Possible Solution

Keep this as the canonical feature request and implement a core reply-runner failover notice policy that sends sanitized retry/fallback notices through active-channel delivery, with config/docs/tests aligned.

## Solution Assessment
