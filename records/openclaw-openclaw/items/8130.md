## Summary

Keep this issue open. Current main still has fallback detection, lifecycle/Control UI visibility, and verbose-only final fallback notices, but it does not send proactive retry/failover messages to the active user channel for normal non-verbose replies and the requested config surface is absent.

## What This Changes

The issue requests configurable active-channel notifications when LLM API retries or model fallback occurs.

## Best Possible Solution

Implement opt-in, sanitized retry/fallback notices in the core reply-runner/model-fallback path, reusing existing fallback summaries and preserving current lifecycle, verbose, and Control UI observability.

## Reproduction Assessment

Yes. Source inspection and existing e2e tests show a high-confidence path: stub `runWithModelFallback` to return a fallback model with `resolvedVerboseLevel: "off"`, and the final payload lacks `Model Fallback:` while lifecycle events still fire.

## Solution Assessment

Yes. A core reply-runner/model-fallback implementation is the narrowest maintainable path because the low-level fallback hooks already exist and prompt/plugin workarounds can only react after the fallback model is running.
