## Summary

Keep open. Current main still has internal fallback signals, lifecycle events, and verbose-only fallback notices, but it does not proactively notify the active non-verbose user channel during retry/failover and does not expose the requested retry/failover notification config; https://github.com/openclaw/openclaw/pull/69399 and https://github.com/openclaw/openclaw/pull/70990 are useful but narrower than this issue.

## What This Changes

The issue requests configurable active-channel notices for API errors, fallback model switches, and optional primary-model recovery during LLM provider failover.

## Best Possible Solution

Define and implement a sanitized, configurable retry/fallback notice policy in the core reply-runner/model-fallback path, coordinated with the narrower open notice and hook PRs, with docs and regression coverage for non-verbose active-channel delivery.

## Reproduction Assessment

Not applicable: this is a feature request, not broken documented behavior. Source inspection and current e2e coverage show fallback exists but non-verbose active-channel retry/fallback notices are intentionally absent.

## Solution Assessment

Partly yes: gateway-level active-channel delivery is the right boundary because the fallback runner sees failed-attempt and transition signals before the fallback response. The exact config surface, defaults, redaction, retry timing, recovery behavior, and relationship to https://github.com/openclaw/openclaw/pull/69399 and https://github.com/openclaw/openclaw/pull/70990 need maintainer judgment.
