## Summary

Keep open. Current main still has internal fallback signals, lifecycle/status visibility, and verbose-only final fallback notices, but it does not proactively notify the active non-verbose channel during retry/failover or expose the requested retry/failover notification config; the open related PR only covers a narrower opt-in final notice path.

## What This Changes

Request proactive active-channel notifications and configuration for LLM API retry, failover, and optional recovery events.

## Best Possible Solution

Define and implement a sanitized, configurable retry/fallback notice policy in the core reply-runner/model-fallback path, with docs and regression coverage for non-verbose active-channel delivery.

## Reproduction Assessment

Not applicable: this is a feature request, not broken documented behavior. Source inspection and existing e2e assertions show non-verbose fallback replies intentionally stay silent while lifecycle events, /status, logs, verbose notices, and Web UI observability exist.

## Solution Assessment

Partly yes: gateway-level active-channel delivery is the right boundary because the fallback runner sees failed-attempt and successful-fallback transitions before the fallback model runs. The exact config placement, defaults, redaction, delayed retry, and recovery policy still need maintainer judgment; https://github.com/openclaw/openclaw/pull/69399 is a narrower possible subset, not the full solution.
