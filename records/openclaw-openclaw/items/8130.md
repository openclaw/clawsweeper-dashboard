## Summary

Keep open. Current main still has model fallback tracking, lifecycle events, and verbose-only final fallback notices, but it does not proactively notify the active non-verbose user channel during provider retry/failover and the requested retry/failover notification policy is still absent.

## What This Changes

Request proactive active-channel notifications when LLM provider/API failures trigger retries, model failover, and optional recovery.

## Best Possible Solution

Define a sanitized core retry/fallback notice policy in the reply-runner/model-fallback path, add the chosen config/docs surface, and cover non-verbose active-channel delivery with regression tests.

## Reproduction Assessment

Not applicable: this is a feature request rather than broken documented behavior. Source inspection and current e2e coverage show fallback works, but non-verbose active-channel retry/fallback notices are intentionally absent.

## Solution Assessment

Partly yes: gateway-level delivery is the right boundary because the fallback runner sees failed attempts and transitions before the fallback response. The exact config shape, defaults, redaction, retry timing, recovery behavior, and overlap with prior closed PR approaches need maintainer judgment.
