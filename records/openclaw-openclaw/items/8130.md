## Summary

Keep this open: current main still lacks proactive active-channel retry/failover notifications and the requested model-failover notification config. The closed related reports look like satellites, while this issue remains the canonical UX gap.

## What This Changes

Request proactive user-facing channel notices when provider errors cause retries or model failover, optionally controlled by model failover notification config.

## Best Possible Solution

Add an opt-in core failover-notice policy in the reply-runner/model-fallback path, using existing fallback hooks and block reply delivery with sanitized messages, schema/help/docs updates, and regression coverage.

## Reproduction Assessment

Not applicable as a strict bug reproduction; this is a feature request. Source and tests do show the current behavior with high confidence: fallback notices are lifecycle/verbose-only and non-verbose channel delivery is not wired.

## Solution Assessment

Yes in direction, but not ready for automatic cleanup: the maintainable place is the core reply-runner/model-fallback path, not a SOUL.md workaround. Maintainers still need to choose the user-facing config shape, defaults, delay policy, and sanitized wording.
