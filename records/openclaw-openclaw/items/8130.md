## Summary

Keep open. Current main still has model fallback tracking, lifecycle events, and verbose-only final fallback notices, but it does not proactively notify the active non-verbose user channel during retry/failover and does not expose the requested retry/failover notification policy; the related open notice PR is useful but narrower than this issue.

## What This Changes

Request proactive user-channel notifications and optional config when LLM API errors trigger retries, model failover, and recovery.

## Best Possible Solution

Define a sanitized core retry/fallback notice policy in the reply-runner/model-fallback path, coordinate it with the narrower open notice/error-detail PRs, and add docs plus regression coverage for non-verbose active-channel delivery.

## Reproduction Assessment

Not applicable: this is a feature request, not broken documented behavior. Source inspection and current e2e coverage show fallback exists, but non-verbose active-channel retry/fallback notices are intentionally absent.

## Solution Assessment

Partly yes: gateway-level active-channel delivery is the right boundary because the fallback runner sees failed-attempt and transition signals before the fallback response. The exact config surface, defaults, redaction, retry timing, recovery behavior, and overlap with https://github.com/openclaw/openclaw/pull/69399 need maintainer judgment.
