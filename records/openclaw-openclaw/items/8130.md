## Summary

Keep this open. Current main still has fallback signals, lifecycle/status visibility, Web UI indicators, and verbose-only final notices, but it does not proactively notify the active non-verbose channel during retry/failover and it lacks the requested notification config surface.

## What This Changes

Request active-channel retry, failover, and optional recovery notifications plus config for LLM provider API errors and fallback model switches.

## Best Possible Solution

Define and implement a sanitized, configurable retry/fallback notice policy in the core reply-runner/model-fallback path, with docs and regression coverage for non-verbose active-channel delivery.

## Reproduction Assessment

Not applicable: this is a feature request, not broken documented behavior. Source inspection and existing e2e assertions show non-verbose fallback replies intentionally stay silent while lifecycle, /status, logs, verbose notices, and Web UI observability exist.

## Solution Assessment

Partly yes: gateway-level active-channel delivery is the right boundary because the fallback runner sees failed-attempt and successful-fallback transitions before the fallback model runs. The exact config, defaults, redaction, and recovery policy still need maintainer judgment.
