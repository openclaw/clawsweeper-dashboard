## Summary

Keep this open. Current main still has model fallback tracking, lifecycle events, `/status` visibility, and verbose-only final fallback notices, but it does not proactively notify non-verbose active-channel users during provider retry/failover and the requested notification config is absent.

## What This Changes

Request proactive active-channel notices when provider errors or retries trigger LLM fallback, plus optional recovery notices and config.

## Best Possible Solution

Add a sanitized, configurable active-channel fallback-notice policy in the core reply-runner/model-fallback path, with tests for non-verbose retry/fallback notices and no duplicate verbose or lifecycle observability.

## Reproduction Assessment

Not applicable because this is a feature request, not a broken documented behavior. Source inspection and existing e2e assertions show current non-verbose fallback replies intentionally remain silent while lifecycle and verbose observability exist.

## Solution Assessment

Yes for the core boundary, but the exact config/default policy still needs maintainer judgment. A gateway-level implementation is more reliable than model self-announcement because the fallback runner sees failed-attempt and successful-fallback transitions before the fallback model starts.
