## Summary

Keep open: current main still lacks proactive active-channel retry/failover/recovery notices and the requested failover notification config; existing fallback visibility is lifecycle/log/status data plus verbose-only final reply notices. The related open PR and narrower issue cover only part of this policy, so this issue remains the broader user-notification thread.

## What This Changes

The issue requests configurable user-facing channel notifications when LLM API calls fail, when OpenClaw switches to a fallback model, and optionally when the primary model recovers.

## Best Possible Solution

Define one canonical default-false, sanitized retry/failover/recovery notification policy in the core reply-runner/model-fallback path, coordinate it with the narrower opt-in notice PR and hook/context work, then document and test the config surface.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature/config request. Source inspection and current tests give high confidence that non-verbose active-channel retry/failover/recovery notices are absent today.

## Solution Assessment

Yes in direction: the reply-runner/model-fallback path is the right boundary because it sees failed attempts and fallback transitions before the fallback response. The exact config key, defaults, redaction, timing, recovery semantics, and coordination with the open narrower PR need maintainer judgment.
