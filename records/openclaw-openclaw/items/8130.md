## Summary

Keep open: current `main` still only exposes fallback visibility through lifecycle/status data and verbose final notices, not proactive non-verbose active-channel retry/failover notifications. The linked implementation at https://github.com/openclaw/openclaw/pull/69399 is narrower and open, so it does not close the broader retry/failover/recovery notification policy requested here.

## What This Changes

Request proactive user-channel notifications and optional config for provider retry, fallback, and recovery events during model failover.

## Best Possible Solution

Define a sanitized core retry/failover notice policy in the reply-runner/model-fallback path, with config, docs, and tests for non-verbose active-channel delivery coordinated with the narrower open fallback-notice PR.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request rather than broken documented behavior. Source inspection and current e2e tests give high confidence that non-verbose active-channel retry/failover notices are absent today.

## Solution Assessment

Yes in direction: gateway-level delivery is the right boundary because the fallback runner sees failed attempts and transitions before the fallback response. The exact config surface, default behavior, delay timing, recovery behavior, and redaction policy need maintainer judgment before implementation.
