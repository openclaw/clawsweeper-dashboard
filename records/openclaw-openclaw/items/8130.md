## Summary

Keep open: current main still lacks proactive non-verbose active-channel retry/failover notices and the requested notification policy config; existing visibility is lifecycle/status/verbose final notices, while https://github.com/openclaw/openclaw/pull/69399 is narrower and does not close this issue.

## What This Changes

The issue requests configurable user-facing notifications when provider/API errors trigger retries, model failover, and optional recovery to the primary model.

## Best Possible Solution

Define a sanitized core retry/failover notice policy in the reply-runner/model-fallback path, with config/docs/tests for non-verbose active-channel delivery and coordinated handling of the narrower open notice PR.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request rather than broken documented behavior. Source inspection and current e2e tests give high confidence that non-verbose active-channel retry/failover notices are absent today.

## Solution Assessment

Yes in direction: gateway-level delivery is the right boundary because the fallback runner sees failed attempts and transitions before the fallback response. The exact config surface, defaults, delay timing, recovery behavior, and redaction policy need maintainer judgment before implementation.
