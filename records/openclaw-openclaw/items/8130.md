## Summary

Keep this open. Current main still has model fallback, lifecycle events, and verbose-only final fallback notices, but it does not proactively notify non-verbose active-channel users during retry/failover and the requested notification config surface is absent.

## What This Changes

The issue requests proactive active-channel notifications when model provider retries or fallback occur, plus optional configuration for failover and retry notice timing.

## Best Possible Solution

Add a configurable, sanitized active-channel notice policy in the core reply-runner/model-fallback path, with tests proving non-verbose users receive retry/fallback notices without duplicating verbose or lifecycle observability.

## Reproduction Assessment

Not applicable as a feature request. Source inspection and existing e2e coverage give high confidence that current non-verbose fallback behavior remains silent while lifecycle/verbose observability exists.

## Solution Assessment

Yes for the core boundary, but the exact config/default policy needs maintainer judgment. A gateway-level implementation is more reliable than model self-announcement because the fallback runner has error/fallback signals before the fallback model starts.
