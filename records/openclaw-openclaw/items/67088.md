Fix the macOS dashboard false no-GUI bug from https://github.com/openclaw/openclaw/issues/67088. On current main, `resolveBrowserOpenCommand()` returns `ssh-no-display` before the Darwin `open` branch when stale `SSH_CLIENT`, `SSH_TTY`, or `SSH_CONNECTION` variables exist and `DISPLAY`/`WAYLAND_DISPLAY` are absent. Add focused regression coverage, preferably in `src/infra/browser-open.test.ts` unless the existing onboard-helper barrel test is the better local pattern: Darwin with SSH vars and no display should resolve to `open`; Linux with SSH vars and no display should still resolve `ssh-no-display`; Windows hardening tests must remain green. Credit the prior closed attempts https://github.com/openclaw/openclaw/pull/67110 and https://github.com/openclaw/openclaw/pull/67115 in the replacement PR body if their approach is reused.
