## Summary

Keep open. Current main still checks `SSH_*` plus no display before the macOS `open` branch, so Darwin shells with stale SSH variables still resolve to `ssh-no-display`; the two linked fix PRs are closed unmerged.

## What This Changes

The issue reports that `openclaw dashboard` on macOS treats stale `SSH_*` environment variables as proof of no GUI and prints the remote-opening hint instead of using the native browser-open path.

## Best Possible Solution

Land a narrow current-main patch that lets Darwin reach the native `open` path despite stale SSH variables, keeps Linux and Windows behavior intact, adds focused regression tests, and includes the user-facing changelog entry during landing.

## Reproduction Assessment

Yes, from source inspection: on Darwin with any `SSH_*` variable and no `DISPLAY` or `WAYLAND_DISPLAY`, current main returns `ssh-no-display` before the macOS branch. I did not run a live macOS shell in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is in `resolveBrowserOpenCommand()`, with regression coverage for Darwin stale SSH variables and preserved Linux/Windows behavior.
