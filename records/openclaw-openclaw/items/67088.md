## Summary

Keep open. Current main still applies the SSH/no-display early return on macOS before the Darwin `open` path, so the dashboard can still print the reported `No GUI detected` hint when stale `SSH_*` variables are present. PR #67115 is an open unmerged implementation candidate that references this issue with closing syntax, so cleanup should wait for that PR or an equivalent fix to land.

## What This Changes

Fix the dashboard browser-opening heuristic so macOS with lingering `SSH_*` environment variables falls through to `/usr/bin/open` instead of showing the SSH tunnel hint.

## Best Possible Solution

Land or replace PR #67115 with a narrow fix in `src/infra/browser-open.ts` that makes the SSH/no-display guard Linux-only or otherwise excludes `darwin`, then add focused regression tests for Darwin with stale `SSH_*` variables and no display resolving to `open`, plus Linux SSH without display still returning `ssh-no-display`.

## Reproduction Assessment

Yes. The source path is deterministic: on `darwin` with any `SSH_CLIENT`, `SSH_TTY`, or `SSH_CONNECTION` set and no `DISPLAY`/`WAYLAND_DISPLAY`, `resolveBrowserOpenCommand()` returns `ssh-no-display` before checking the macOS `open` branch, and `dashboardCommand()` turns that into the reported hint.

## Solution Assessment

Yes, the narrowest maintainable fix is in the shared browser-open resolver rather than in dashboard output formatting. PR #67115 is directionally the right fix, but the landing path should include regression tests that lock macOS and Linux behavior separately.
