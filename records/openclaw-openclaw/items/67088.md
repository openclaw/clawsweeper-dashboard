## Summary

Keep open. Current main and `v2026.5.18` still send macOS shells with stale `SSH_*` variables and no display env to the `ssh-no-display` fallback before the native `open` branch, and all linked fix PRs are closed unmerged.

## What This Changes

Running `openclaw dashboard` on macOS should still open the native browser when stale `SSH_*` variables are present instead of showing the headless SSH hint.

## Best Possible Solution

Change the resolver so Darwin is not short-circuited by stale SSH env vars, add focused macOS stale-SSH and Linux headless regression tests, and close this issue only after the fix lands with real macOS dashboard/open proof or maintainer override.

## Reproduction Assessment

Yes. Source inspection on current main shows Darwin with stale `SSH_*` variables and no `DISPLAY`/`WAYLAND_DISPLAY` returns `ssh-no-display` before the native `open` branch; I did not run a live macOS desktop command in this read-only review.

## Solution Assessment

Yes. Excluding Darwin from the SSH/no-display short-circuit, or equivalently restricting that short-circuit to Linux, is the narrow maintainable fix while preserving Linux and Windows behavior.
