## Summary

Keep open. Current main still returns `ssh-no-display` before the macOS `open` path when stale `SSH_*` variables are present, and both linked fix PRs are closed unmerged, so this remains a valid narrow bug.

## What This Changes

The issue asks OpenClaw to avoid falsely treating macOS shells with stale `SSH_*` variables as no-GUI environments when running `openclaw dashboard`.

## Best Possible Solution

Land a current-main patch that lets Darwin reach the native `open` path despite stale SSH env vars, preserves Linux no-display and Windows hardened launch behavior, and adds focused regression coverage plus a user-facing changelog entry.

## Reproduction Assessment

Yes from source inspection: on Darwin with any `SSH_*` variable and no `DISPLAY`/`WAYLAND_DISPLAY`, current main returns `ssh-no-display` before the macOS `open` branch. I did not run a live macOS shell in this read-only sweep.

## Solution Assessment

Yes. The narrow maintainable fix is in `resolveBrowserOpenCommand()`, with tests for Darwin stale SSH vars, preserved Linux no-display behavior, and preserved Windows browser-open hardening.
