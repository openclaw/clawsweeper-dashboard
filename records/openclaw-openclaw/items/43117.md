## Summary

Keep open. Current main implements Gemini Embedding 2, configurable dimensions, and opt-in image/audio indexing from memorySearch.extraPaths, but chat and Telegram attachments in session history are still flattened to text and never become structured embedding inputs.

## What This Changes

Add multimodal memorySearch support for Gemini Embedding 2, including attachment/image indexing, mixed-modal recall, and configurable embedding output dimensions.

## Best Possible Solution

Keep the shipped extraPaths implementation and add a narrowly scoped session-attachment indexing path that preserves eligible media metadata, resolves files through existing media and sandbox boundaries, builds structured EmbeddingInput chunks only behind provider/size/privacy gates, and documents the distinction from extraPaths multimodal memory.

## Reproduction Assessment

Yes. A source-level reproduction path is clear: a transcript message with MediaPath/MediaPaths but little or no descriptive text is reduced by buildSessionEntry to sanitized text only, while the same eligible file under extraPaths can become an inline-data embedding chunk.

## Solution Assessment

Yes, if narrowed to the remaining session-memory workflow. The provider and extraPaths pieces are already implemented, so the maintainable fix is to bridge existing attachment metadata into session indexing rather than reworking the Gemini provider.
