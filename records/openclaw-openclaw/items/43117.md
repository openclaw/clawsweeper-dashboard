## Summary

Keep open. Current main covers Gemini Embedding 2, output dimensions, and opt-in image/audio indexing from `memorySearch.extraPaths`, but it still does not index media attached to session/chat history, and the broader video/PDF part of the request is not implemented.

## What This Changes

Request to extend `memorySearch` so Gemini Embedding 2 can index multimodal conversation attachments, expose configurable output dimensionality, and return cross-modal search results.

## Best Possible Solution

Keep the shipped Gemini and `extraPaths` work, then add a narrowly scoped opt-in session-attachment indexing path that preserves eligible media metadata, respects existing media/sandbox boundaries, produces structured `EmbeddingInput` chunks, and documents the difference from `extraPaths` multimodal memory.

## Reproduction Assessment

Yes. A source-level reproduction path is clear: a transcript message with only `MediaPath`/`MediaPaths` and little or no text is reduced to no searchable embedding content, while an eligible image/audio file under `memorySearch.extraPaths` can become a structured inline-data embedding chunk.

## Solution Assessment

Unclear until maintainers settle the privacy and scope decision for session media. The current provider and `extraPaths` implementation is the right base, but the remaining fix should bridge transcript media into existing structured embedding inputs rather than reworking the Gemini provider.
