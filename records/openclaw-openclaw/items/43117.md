- `P2`: This is a normal-priority memory-search feature gap with useful partial support already shipped and limited blast radius to opt-in multimodal memory users.
- `impact:session-state`: The issue affects what media can become searchable memory context and therefore what later `memory_search` can retrieve.
- `impact:security`: The remaining attachment/media path would persist and send user media to a remote embedding provider, which is sensitive-data handling.

## Summary

Keep open: current main and the latest release already cover `outputDimensionality` plus opt-in image/audio indexing from `memorySearch.extraPaths`, but PDF/video modalities and remembered inbound-attachment indexing remain missing and need maintainer product/security direction.

## What This Changes

Requested Gemini Embedding 2 multimodal memory search support for media attachments, mixed-modal retrieval, and configurable embedding dimensionality.

## Best Possible Solution

Keep the shipped image/audio `extraPaths` foundation, then split or design the remaining PDF/video modality contract and explicit remembered-attachment ingestion path with provider limits, consent/retention rules, docs, and focused tests.

## Reproduction Assessment

Yes at source level: current main accepts only image/audio multimodal memory under `extraPaths`, while the config schema and classifier exclude PDF/video and session transcript indexing extracts text blocks only. I did not run a live Gemini account because this read-only review has no provider credentials.

## Solution Assessment

No automatic fix is appropriate yet. The existing image/audio implementation is a safe partial foundation, but the remaining work needs a maintainer-approved media retention and provider-format contract before code changes.
