## Summary

Keep open. Current main covers Gemini Embedding 2, configurable dimensions, and opt-in image/audio indexing from memorySearch.extraPaths, but it still does not index chat or Telegram attachments from stored session history; that path continues to flatten transcripts to sanitized text only.

## What This Changes

Request support for multimodal memory search with gemini-embedding-2-preview, including attachment indexing, configurable output dimensions, and cross-modal recall.

## Best Possible Solution

Keep the shipped extraPaths implementation, then narrow the remaining work to session-memory attachment indexing: preserve eligible MediaPath/MediaPaths plus MIME metadata from transcripts, resolve files through existing managed-media and sandbox boundaries, build structured EmbeddingInput chunks behind provider capability and size gates, cover Telegram/chat recall, and document the difference between file-backed multimodal memory and conversation-history attachment recall.

## Reproduction Assessment

Yes. A source-level reproduction path is clear: a transcript message with MediaPath/MediaPaths but no descriptive text will be reduced to no structured embedding input by buildSessionEntry, while the same file under extraPaths can produce an inline-data embedding chunk.

## Solution Assessment

Yes, if narrowed to the remaining session-memory workflow. The provider and extraPaths pieces are already implemented, so the maintainable fix is to bridge existing attachment metadata into session indexing rather than reworking the Gemini provider or broadening core memory behavior indiscriminately.
