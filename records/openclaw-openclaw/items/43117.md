## Best Possible Solution

Extend the existing structured-input `extraPaths` foundation with explicit PDF/video modality support and a separate opt-in retained-media ingestion contract for chat attachments, including deletion and provider-upload limits.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection gives high confidence that current main supports Gemini dimensions plus image/audio `extraPaths`, while PDF/video and retained chat-attachment indexing remain absent.

## Solution Assessment

Unclear until maintainers settle the media-retention and provider-upload contract. The existing structured embedding path is the right base, but broad attachment indexing should not be added without explicit opt-in, deletion semantics, and current Gemini API proof.

## Risks / Open Questions

- Automatically indexing inbound attachments would change media retention, deletion, audit, consent, and search-result exposure expectations for conversation history.
- Expanding beyond image/audio would upload more raw user media to a remote embedding provider, so MIME, size, provider, and opt-in boundaries need an explicit policy.
- The upstream Gemini model name and request contract appear to have evolved from preview toward stable naming, so implementation should verify the current API before hard-coding more preview-only behavior.
