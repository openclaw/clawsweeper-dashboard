## Summary

Keep open, but this PR is not merge-ready: the linked "Model is not allowed" symptom already has a documented allowlist explanation, and the patch removes the provider auth guard for every baseUrl-backed provider instead of only providers with proven synthetic/local auth.

## What This Changes

The PR relaxes `isWritableProviderConfig` so providers with configured models are written to `models.json` when they have a `baseUrl`, even without an `apiKey`, and adds an explanatory comment.

## Best Possible Solution

Narrow the planner exception to providers with proven runtime synthetic/local auth, preserve the existing unauthenticated remote-provider filter, add focused planner coverage, and attach redacted Docker/Ollama after-fix proof.

## Reproduction Assessment

Yes for the patch defect: source inspection shows the PR would include any baseUrl-only provider, while the existing planner test expects unauthenticated providers with model rows to be filtered. The linked Docker/Ollama user flow itself was not live-reproduced.

## Solution Assessment

No: the current patch is too broad because it treats every baseUrl-only provider as writable. A safer fix would reuse or mirror runtime synthetic/local auth eligibility and leave unauthenticated remote providers filtered.
