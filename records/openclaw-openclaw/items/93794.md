## Summary

Close: current main already restores readable Telegram default delivery by routing normal replies through standard sendMessage/HTML unless richMessages is explicitly enabled. ## What This Changes

The issue reports a v2026.6.8 regression where Telegram Web renders OpenClaw bot replies as unsupported messages instead of readable agent text.

## Best Possible Solution

Keep the PR https://github.com/openclaw/openclaw/pull/93279 behavior: standard Telegram sendMessage/HTML by default, Bot API rich messages only behind explicit richMessages opt-in, and release that fix in the next patch train.

## Reproduction Assessment

Yes for the reported released build: v2026.6.8 source routes normal text through sendRichMessage, and multiple commenters confirmed the unsupported-message symptom. Current main no longer follows that default path; I did not run a live Telegram Web replay in this read-only review.

## Solution Assessment

Yes. The current main fix is the narrow owner-boundary solution because it restores the compatible sendMessage/HTML default and keeps richer Bot API delivery as an explicit opt-in rather than adding a second workaround path.
