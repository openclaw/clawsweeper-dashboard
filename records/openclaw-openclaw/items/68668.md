## Summary

Keep open. Current main and the latest shipped release still have the unguarded browser-cleanup wrapper path, and the open closing PR https://github.com/openclaw/openclaw/pull/68669 has not merged yet.

## What This Changes

The issue requests deduping `cleanupBrowserSessionsForLifecycleEnd` so embedded subagent completions do not invoke the browser cleanup wrapper twice for the same `runId`.

## Best Possible Solution

Review or repair and land https://github.com/openclaw/openclaw/pull/68669, or an equivalent guarded lifecycle change, ensuring duplicate completions skip only the browser cleanup wrapper and still run the completion tail.

## Reproduction Assessment

Yes, at source level. Current main starts both the in-process embedded listener and gateway wait path for one registered run, and both can call `completeSubagentRun(..., triggerCleanup: true)` while the browser cleanup wrapper has no per-run dispatch marker.

## Solution Assessment

Yes, the narrow maintainable direction is a lifecycle-controller guard for browser cleanup dispatch. The open PR should still preserve runtime retirement and announce cleanup on duplicate completion paths before it lands.
