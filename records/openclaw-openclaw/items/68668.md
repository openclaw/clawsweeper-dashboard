## Summary

Keep open. Current main still has the unguarded browser cleanup wrapper path, and the issue is already paired with open closing PR #68669, so it should remain open until that implementation or an equivalent fix is merged or rejected.

## What This Changes

The issue requests a per-run guard so embedded subagent completion cannot invoke `cleanupBrowserSessionsForLifecycleEnd` twice for the same `runId`.

## Best Possible Solution

Review and land #68669, or an equivalent narrow fix, with a synchronous per-record browser cleanup dispatch guard, a rearm reset, and regression coverage in the subagent lifecycle tests. Once that PR merges, this issue can close as implemented.

## Reproduction Assessment

Yes. Current main has a high-confidence source-level reproduction path: two embedded completion paths can call `completeSubagentRun` for the same run with `triggerCleanup: true`, and the cleanup wrapper branch has no per-run guard.

## Solution Assessment

Yes. A synchronous per-record dispatch guard colocated with the existing `endedHookEmittedAt` pattern is the narrowest maintainable fix for wrapper-level duplication; changing the browser plugin would not remove duplicated normalization, best-effort cleanup, facade dispatch, or warning overhead.
