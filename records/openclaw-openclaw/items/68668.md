## Summary

Keep open. Current main still has the unguarded browser-cleanup wrapper path, and the issue is paired with the open closing PR https://github.com/openclaw/openclaw/pull/68669, so it should close only after that PR or an equivalent fix lands.

## What This Changes

The issue requests deduping `cleanupBrowserSessionsForLifecycleEnd` wrapper dispatch when embedded subagent completion reaches the same `runId` through two completion paths.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/68669, or an equivalent lifecycle-controller guard that dedupes only browser cleanup dispatch, resets on rearm, and preserves the existing completion tail.

## Reproduction Assessment

Yes, source-level reproduction is high confidence. Current main starts both the in-process listener and gateway wait path for a registered subagent, and both can call `completeSubagentRun(..., triggerCleanup: true)` while the browser cleanup wrapper has no per-run dispatch marker.

## Solution Assessment

Yes, the proposed per-run dispatch marker is the narrow maintainable direction. The implementation must suppress only the browser cleanup wrapper and keep the existing retirement and announce-cleanup tail intact.
