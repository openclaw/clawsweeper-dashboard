## Summary

Keep open. Current main still has two embedded subagent completion paths that can call the same unguarded browser cleanup wrapper for one run, and the open linked PR #68669 uses closing syntax, so this issue should remain open until that implementation is reviewed and either merged or closed.

## What This Changes

Deduplicate per-run browser session lifecycle cleanup in embedded subagent completion so `cleanupBrowserSessionsForLifecycleEnd` is not invoked twice for the same `runId`.

## Best Possible Solution

Review #68669 as the implementation candidate, verify it adds a synchronous per-record cleanup dispatch guard with a rearm reset and regression coverage, then merge it or choose an equivalent dedupe point before closing this issue.

## Reproduction Assessment

Yes. Current main has a high-confidence source-level reproduction path: invoke `completeSubagentRun` twice in parallel for the same run with `triggerCleanup: true`, matching the two production completion paths, and the wrapper call is reachable both times without a browser-cleanup dispatch flag.

## Solution Assessment

Yes. A per-record synchronous dispatch guard colocated with the existing ended-hook dedupe pattern is the narrowest maintainable fix; changing the browser plugin would not remove duplicated wrapper normalization, best-effort cleanup, facade dispatch, or warning overhead.
