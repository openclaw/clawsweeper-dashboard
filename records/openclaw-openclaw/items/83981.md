## Summary

Keep open: current main still constructs `sessions_yield` with an active `onYield` callback before factory-level policy gating, and the only linked fix PR was closed unmerged after proof and compatibility concerns. Source inspection supports the reported policy-boundary bug, but I did not establish a live end-to-end bypass run.

## What This Changes

The issue reports that `sessions_yield` remains available and callable when the effective tool allow/deny policy excludes it.

## Best Possible Solution

Add construction-time gating for `sessions_yield` that honors deny and truly restrictive allow policies without breaking `tools.alsoAllow`, keep downstream filtering as defense in depth, and add focused regression coverage plus real restricted-policy proof.

## Reproduction Assessment

Yes for source-level reproduction: current main still builds `sessions_yield` with `onYield` before a factory-level policy check, and the embedded allow filter returns all tools when the runtime allowlist is undefined. No live end-to-end bypass run was established in this read-only review.

## Solution Assessment

Yes in direction: construction-time gating plus regression tests is the narrow maintainable fix. The replacement must avoid the prior PR's `alsoAllow` compatibility pitfall and should include proof from a real restricted-policy setup.
