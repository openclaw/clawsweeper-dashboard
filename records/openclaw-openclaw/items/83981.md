## Summary

Keep open: current main still unconditionally creates `sessions_yield` with an active `onYield` callback before the factory policy gates used for other tools, and the only linked fix PR was closed unmerged. ## What This Changes

The issue asks OpenClaw to remove `sessions_yield` from the effective tool surface and reject invocations when tool policy excludes it.

## Best Possible Solution

Land a corrected narrow fix that gates `sessions_yield` construction and invocation on the effective policy, preserves documented default and `alsoAllow` semantics, and adds regression plus redacted real-session proof for denied and narrow-allow configurations.

## Reproduction Assessment

Partially: source inspection gives a high-confidence path because current main constructs `sessions_yield` with `onYield` even after reading deny policy inputs, and the deny-only embedded filter can leave the tool set unfiltered. I did not establish a live model or gateway run where the final model-visible surface bypasses every downstream filter.

## Solution Assessment

Yes in direction: construction-time gating plus downstream policy filtering as defense in depth is the narrow maintainable fix. The previous PR’s approach still needs the documented `alsoAllow` compatibility issue resolved and real restricted-policy proof before this issue can close.
