## Summary

Keep open: current main and the latest release still leave the documented WebChat targetless `message.send` path dependent on `sourceReplyDeliveryMode`, and the open linked fix PR is the right owner until it is merged or closed.

## What This Changes

The issue asks for targetless WebChat `message.send` calls during an active run to route through the documented current-run internal source-reply sink instead of failing with a target-required error.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/96850, or replace it with the same narrow message-tool construction fix that preserves ordinary WebChat final replies as automatic.

## Reproduction Assessment

Yes: from source inspection, current main documents the targetless WebChat sink but the message tool can still pass an undefined delivery mode, the sink requires `message_tool_only`, and normalization then throws the reported target error. I did not run a live WebChat session in this read-only review.

## Solution Assessment

Yes for the narrow fix shape in the linked PR; no for the broad issue-body gateway suggestion, because current tests intentionally keep ordinary WebChat room-event final delivery automatic. The maintainable solution is to default internal/WebChat message tools into the sink only where the message tool is actually being constructed, while preserving explicit `automatic` overrides.
