## Summary

Keep open: current main and the latest release still show the documented WebChat targetless-send gap, and two open closing PRs now own the fix path. The issue should close only after one candidate merges and the duplicate branch is handled.

## What This Changes

The issue requests that targetless `message.send` from an active WebChat run use the documented current-run internal source-reply sink instead of failing target validation.

## Best Possible Solution

Land one narrow `createMessageTool` source-reply-mode fix, preserve ordinary WebChat automatic final replies, then close the sibling candidate and this issue after merge.

## Reproduction Assessment

Yes from source inspection: current main documents the WebChat targetless source-reply sink, but implicit WebChat message tools do not set the mode that the sink requires before target validation. I did not run a live WebChat session in this read-only review.

## Solution Assessment

Yes: the narrow fix is to supply `message_tool_only` at the WebChat message-tool construction boundary while preserving automatic final replies for ordinary WebChat runs. A broad gateway room-event change would be riskier because current tests intentionally keep those WebChat turns automatic.
