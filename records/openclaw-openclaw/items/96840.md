## Summary

Keep open: current main still has a source-level mismatch where ordinary WebChat room-event tool resolution stays automatic, while targetless `message.send` only reaches the documented internal source-reply sink when `sourceReplyDeliveryMode` is already `message_tool_only`.

## What This Changes

The issue asks OpenClaw to make targetless WebChat `message.send` from an active run project into the same chat instead of failing target validation.

## Best Possible Solution

Keep ordinary WebChat final replies automatic, but ensure WebChat runs that actually expose or require `tools.message` can route a targetless `send` through the current-run internal sink with focused regression coverage.

## Reproduction Assessment

Yes from source inspection: current main leaves implicit WebChat room events without `message_tool_only`, the internal sink requires that mode, and the next validation step throws the reported target error. I did not run a live WebChat session in this read-only review.

## Solution Assessment

No for the exact broad suggested patch: making all WebChat room events message-tool-only would conflict with current automatic final-delivery tests. The best fix is a narrower mode/tool-resolution repair that preserves automatic WebChat finals while enabling the documented targetless sink for message-tool-only WebChat runs.
