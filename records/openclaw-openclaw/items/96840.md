## Summary

Keep open: current main still has the documented WebChat source-reply sink, but ordinary WebChat room-event tool resolution remains automatic while the targetless internal sink only accepts `message_tool_only` runs.

## What This Changes

The issue requests that targetless WebChat `message.send` calls during an active WebChat run route to the current chat instead of failing for lack of a target.

## Best Possible Solution

Preserve ordinary WebChat final-answer delivery, but add a narrow current-run WebChat message-tool path or clarified mode handoff so targetless sends only use the internal sink when the active run exposes or requires `tools.message`.

## Reproduction Assessment

Yes, by source inspection: ordinary WebChat tool resolution leaves `sourceReplyDeliveryMode` automatic, the internal sink requires `message_tool_only`, and the fallback target validation throws the reported error. I did not run a live WebChat session in this read-only review.

## Solution Assessment

No for the issue's broad suggested patch, because making all WebChat room events message-tool-only conflicts with current automatic-delivery tests. The best fix is narrower: keep automatic WebChat finals while allowing the documented current-run sink only for the active message-tool path.
