## Summary

Keep open: current main and the latest release still leave WebChat room-event tool resolution on automatic delivery while the docs promise a targetless active-run `message.send` should use the internal source-reply sink.

## What This Changes

Targetless WebChat `message.send` should route to the current-run source-reply sink instead of requiring an explicit target.

## Best Possible Solution

Set the source-reply ownership mode for active WebChat source-reply tool resolution, keep explicit routes on the normal path, and update the gateway/outbound regression tests so WebChat remains an internal current-run sink rather than a reusable outbound channel.

## Reproduction Assessment

Yes, at source level: current gateway tool resolution leaves WebChat source-reply mode unset, and the outbound sink requires that mode before accepting a targetless current-source send. I did not run a live WebChat session in this read-only review.

## Solution Assessment

Yes, with a scope caveat: the right repair is at the gateway/tool-resolution ownership boundary that passes `sourceReplyDeliveryMode` into the message tool. The fix should stay limited to active current-run WebChat source replies, not turn WebChat into a normal outbound channel or add config.
