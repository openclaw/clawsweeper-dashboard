## Summary

Keep open: current main still leaves the documented WebChat targetless `message.send` path dependent on an explicit `message_tool_only` mode, and the issue has open closing PRs that should be reviewed or landed before this issue closes.

## What This Changes

The issue reports that targetless `message(action="send")` fails in active WebChat sessions even though WebChat docs say the current-run internal source-reply sink should project it into the same chat.

## Best Possible Solution

Land one narrow `createMessageTool` boundary fix, preferably the already proof-positive https://github.com/openclaw/openclaw/pull/97167 if maintainers agree, then close the duplicate PR and this issue after merge.

## Reproduction Assessment

Yes, source-reproducible: current main has the documented sink, the sink requires `message_tool_only`, and the fallback validation throws the reported target error when that mode is absent. I did not run a live WebChat session in this read-only review.

## Solution Assessment

Yes for keeping this issue open while fixes are unmerged. The best fix appears to be the narrow message-tool-boundary defaulting approach in https://github.com/openclaw/openclaw/pull/97167 rather than closing this issue before a candidate reaches main.
