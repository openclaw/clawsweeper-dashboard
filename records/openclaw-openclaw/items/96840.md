## Summary

Keep open: current main and latest release still leave ordinary WebChat room-event tool resolution without `message_tool_only`, while the documented internal source-reply sink requires that mode before accepting a targetless `message.send`. The related merged work added the sink, but no open or merged canonical fix owns this remaining gateway/tool-resolution mismatch.

## What This Changes

Targetless WebChat `message.send` during an active run should project into the same WebChat conversation instead of failing for lack of an explicit target.

## Best Possible Solution

Repair the gateway/tool-resolution source-reply mode boundary so WebChat runs that actually expose or require `tools.message` can use the existing current-run internal sink, without making WebChat a reusable outbound channel or changing ordinary automatic final replies.

## Reproduction Assessment

Yes from source inspection: current main leaves implicit WebChat room events without `message_tool_only`, the internal sink requires that mode, and the next validation path throws the reported target error. I did not run a live WebChat session because this review is read-only.

## Solution Assessment

No for the exact broad suggested patch: making all WebChat room events message-tool-only would conflict with current automatic final-delivery tests. The best fix is a narrower mode/tool-resolution repair that preserves automatic WebChat finals while enabling the documented targetless sink for message-tool-only WebChat runs.

## Repair Work Prompt

Fix the WebChat targetless message tool source-reply bug reported in https://github.com/openclaw/openclaw/issues/96840. In current main, `src/gateway/tool-resolution.ts` keeps WebChat room events out of inferred `message_tool_only`, while `src/infra/outbound/internal-source-reply.ts` and `src/infra/outbound/message-action-runner.ts` can already route targetless active-run sends through the internal source sink when that mode is present. Add the smallest owner-boundary fix and regression tests so an active WebChat run with the message tool/source-reply contract can call `message(action="send")` without `target`, `to`, `channelId`, or `targets` and get `channel=webchat`, `target=current-run`, `sourceReplySink=internal-ui`. Preserve ordinary automatic final assistant delivery for normal direct WebChat turns, preserve explicit outbound channel/target sends, and do not make WebChat a reusable outbound channel or `lastChannel`. Do not add config/env options or changelog edits.
