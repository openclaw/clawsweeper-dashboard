## Summary

## What This Changes

Fix targetless `message.send` in active WebChat runs so it uses the current-run internal source-reply sink instead of failing target validation.

## Best Possible Solution

Land one focused message-tool-construction fix that defaults only internal WebChat message tools to `message_tool_only` while preserving explicit `automatic` mode and ordinary WebChat final replies, then close this issue through the merged PR.

## Reproduction Assessment

Yes from source inspection: current main leaves internal WebChat message-tool sends without `message_tool_only`, the internal sink requires that mode, and the fallback normalization path throws the reported target error. I did not run a live WebChat session because this review is read-only and tests can write artifacts.

## Solution Assessment

Yes for the narrower open-PR shape, not for the issue body's broad gateway default. The best fix is to set the mode where the internal WebChat message tool is constructed while preserving automatic WebChat final replies and explicit override behavior.
