## Summary

Current main still has the reported host-side behavior: context-engine assembly can replace the active prompt messages, but the embedded runner still passes the pre-assembly snapshot into the hard overflow precheck, and the helper still lets the larger unwindowed estimate win. The issue also has an open closing PR, #74255, while the linked lossless-claw issue/PRs cover companion LCM-side hardening rather than the OpenClaw host fix.

## What This Changes

The issue requests that context-engine assembled messages be treated as prompt-authoritative for overflow admission instead of hard-gating on the larger pre-assembly history by default.

## Best Possible Solution

Land or adapt #74255 so successful context-engine assembly is prompt-authoritative by default, while any raw pre-assembly transcript pressure becomes explicit opt-in overflow policy or separate maintenance telemetry with regression coverage for both paths.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: provide a small assembled message set and a larger pre-assembly snapshot, then exercise the current embedded precheck path or helper and observe that the larger unwindowed estimate drives compaction.

## Solution Assessment

Yes. The requested direction is the narrow maintainable fix: preserve overflow protection for the actual prompt by default and require an explicit context-engine opt-in before raw pre-assembly history participates in hard admission gating.
