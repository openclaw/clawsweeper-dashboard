## Summary

Keep open. Current main still contains the reported browser tab-selection race, the linked fix PR is closed unmerged, and no open replacement PR was found.

## What This Changes

Fix browser tab selection so `ensureTabAvailable()` does not throw `BrowserTabNotFoundError` when a newly opened local-managed tab appears before its CDP `webSocketDebuggerUrl`.

## Best Possible Solution

Land a focused browser plugin repair that captures the bootstrap `openTab()` result, waits briefly for `wsUrl` discovery when filtering empties visible tabs, tolerates transient list failures during that wait, and falls back to the latest usable target-id tab without folding in the separate AbortSignal concern.

## Reproduction Assessment

Yes. A deterministic reproduction is to have `listTabs()` return `[]`, have `openTab("about:blank")` return a created tab, then have the next `listTabs()` return that tab without `wsUrl`; current main filters it out and throws `BrowserTabNotFoundError`.

## Solution Assessment

Yes. The narrowest maintainable fix belongs in `extensions/browser/src/browser/server-context.selection.ts` with colocated regression tests, using #63355 as source context while keeping the unrelated Playwright AbortSignal cleanup separate.

Fix the browser plugin `ensureTabAvailable()` wsUrl race reported in https://github.com/openclaw/openclaw/issues/63343, using the closed https://github.com/openclaw/openclaw/pull/63355 as source context but making a clean narrow branch. In `extensions/browser/src/browser/server-context.selection.ts`, capture the `openTab("about:blank")` return value, avoid unnecessary second listing when tabs already exist, poll briefly when `supportsPerTabWs` filtering leaves no candidates while tabs exist, tolerate transient `listTabs()` failures during that polling window, fall back to the latest non-empty tab snapshot without `wsUrl`, and use the `openTab()` result directly if discovery stays empty.
