## Summary

Keep open. Current main still has the reported `ensureTabAvailable` race shape: the `openTab("about:blank")` return value is ignored, per-tab websocket selection filters out tabs without `wsUrl`, and the related fix PR was closed unmerged rather than superseded by a landed repair.

## What This Changes

The issue requests a browser bridge fix so `ensureTabAvailable()` can tolerate CDP tab discovery and `webSocketDebuggerUrl` lag after opening or recovering tabs.

## Best Possible Solution

Land a narrow browser plugin repair that keeps the newly opened tab as a fallback candidate, waits briefly for delayed `wsUrl` discovery, and adds regression tests for both empty-list and missing-`wsUrl` selection paths.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source: current `ensureTabAvailable()` can open a tab, discard that result, relist before CDP has populated `wsUrl`, filter all per-tab candidates out, and throw `BrowserTabNotFoundError`; I did not run the live Chrome timing race.

## Solution Assessment

Is this the best way to solve the issue? Yes, the narrowest fix is in browser tab selection rather than a new config path: preserve the opened-tab candidate, tolerate delayed `wsUrl`, and test the selection behavior directly.
