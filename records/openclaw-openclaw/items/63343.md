## Summary

Keep open. Current main still has the reported browser selection race: `ensureTabAvailable()` discards the opened tab, immediately re-lists tabs, filters local-managed tabs without `wsUrl`, and can still throw `BrowserTabNotFoundError`; the linked fix PR is closed unmerged.

## What This Changes

Requests a browser plugin fix so `ensureTabAvailable()` does not throw `tab not found` when local-managed CDP tab discovery temporarily omits `wsUrl`.

## Best Possible Solution

Patch the browser plugin selection path to preserve the opened target and tolerate delayed `wsUrl` discovery with focused regression tests, without adding config or moving ownership into core.

## Reproduction Assessment

Yes at source level: if the first tab list is empty, `openTab()` can return a target, the next list can still omit `wsUrl`, and local-managed selection filters all candidates before throwing. I did not run a live Chrome timing repro in this read-only pass.

## Solution Assessment

Yes, the best fix is in `ensureTabAvailable()` because the failure is selection state losing a known opened tab and treating delayed `wsUrl` as no tab. A bounded poll plus fallback should be covered by deterministic tests around the existing browser plugin seam.
