## Summary

Keep open. Current main and the latest stable release still contain the reported browser tab-selection race, and the related fix PR was closed unmerged without an open replacement owning the repair.

## What This Changes

The issue requests a browser plugin fix for `ensureTabAvailable()` so newly opened tabs remain selectable when CDP tab discovery or per-tab `wsUrl` population lags.

## Best Possible Solution

Land a focused browser plugin repair that keeps the bootstrap `openTab()` result, waits briefly for `wsUrl` discovery when filtering empties candidates, tolerates transient tab-list failures, and falls back to the latest usable target-id tab without mixing in the separate AbortSignal concern.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: make the first `listTabs()` empty, let `openTab("about:blank")` return a target, then have the next `listTabs()` return no tab or a tab without `wsUrl`; local-managed filtering leaves no candidate and reaches `BrowserTabNotFoundError`.

## Solution Assessment

Yes. The narrowest maintainable fix belongs in `extensions/browser/src/browser/server-context.selection.ts` with colocated regression tests, using the closed PR as implementation context while keeping the unrelated Playwright AbortSignal follow-up separate.
