## Summary

Keep open. Current main and the latest release still contain the reported tab-selection race, the linked fix PR is closed unmerged, and search found no canonical replacement PR currently owning the fix.

## What This Changes

Fix the browser bridge race where `ensureTabAvailable()` can discard the bootstrap `openTab()` result and then filter out local-managed tabs while CDP has not populated `wsUrl`.

## Best Possible Solution

Land a focused browser plugin fix that keeps the bootstrap `openTab()` result, waits briefly for `wsUrl` discovery when filtering empties candidates, tolerates transient tab-list failures, and falls back to the latest usable target-id tab without mixing in the separate AbortSignal concern.

## Reproduction Assessment

Yes. From current source, mock `listTabs()` as `[]`, have `openTab("about:blank")` return a target, then have the next `listTabs()` return that target without `wsUrl`; local-managed filtering empties candidates and reaches `BrowserTabNotFoundError`.

## Solution Assessment

Yes. The narrowest maintainable solution is in `extensions/browser/src/browser/server-context.selection.ts` with colocated regression tests, using the closed PR as context but keeping the repair isolated from unrelated branch history and the separate abort-signal follow-up.
