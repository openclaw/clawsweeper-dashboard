## Summary

Keep open. Current main still contains the reported browser tab-selection race, and the linked fix PR was closed unmerged after the author said the branch had unrelated commits.

## What This Changes

Fix a local-managed browser tab-selection race where `ensureTabAvailable()` discards the bootstrap `openTab()` result and filters out tabs whose `wsUrl` has not appeared yet.

## Best Possible Solution

Land a focused browser plugin fix that captures the bootstrap `openTab()` result, waits briefly for `wsUrl` discovery, tolerates transient tab-list failures, falls back to the latest usable target-id tab, and keeps the separate Playwright `AbortSignal` concern out of scope.

## Reproduction Assessment

Yes, from source inspection rather than a live run. Configure `listTabs()` to return `[]`, have `openTab("about:blank")` return a target, then have the next `listTabs()` return that tab without `wsUrl`; current main filters candidates empty and throws `BrowserTabNotFoundError`.

## Solution Assessment

Yes. The narrowest maintainable fix belongs in `extensions/browser/src/browser/server-context.selection.ts` with colocated regression tests, using the closed PR as source context without pulling in unrelated branch history or the separate abort-signal cleanup.
