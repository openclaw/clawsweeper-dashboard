## Summary

Keep open. Current main and the latest release still have the reported `ensureTabAvailable()` race, and https://github.com/openclaw/openclaw/pull/91463 is an open closing PR for this issue rather than a merged fix.

## What This Changes

The issue reports a browser plugin race where `ensureTabAvailable()` can throw `BrowserTabNotFoundError` when CDP target discovery or `webSocketDebuggerUrl` lags a newly opened local-managed tab.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/91463 if its patch and proof hold up, or close that PR and keep this issue as the canonical browser-selection bug until a replacement fix lands.

## Reproduction Assessment

Yes, at source level: current `main` still discards the opened tab, immediately re-lists, filters local-managed candidates without `wsUrl`, and throws when no candidate remains. I did not run a live Chrome timing repro in this read-only pass.

## Solution Assessment

Yes, the best current solution path is to fix `createProfileSelectionOps()` because that is where the known target is lost and the `wsUrl` filter turns delayed metadata into `tab not found`. The linked open PR is the active candidate and should be reviewed or replaced rather than closing this issue early.
