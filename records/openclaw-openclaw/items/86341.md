## Summary

Keep open for review/merge: current main and the latest release still advertise `cron add --at +duration` while the current parser rejects that plus form, and this PR applies a narrow parser-side fix with targeted regression coverage and terminal proof.

## What This Changes

The PR changes cron CLI one-shot schedule parsing so `--at +duration` is accepted alongside bare durations, while keeping the shared duration parser strict and adding parser/CLI regression tests.

## Best Possible Solution

Land the narrow parser and regression-test fix after normal CI and maintainer gates so the CLI help and validator agree while other duration consumers stay strict.

## Reproduction Assessment

Yes. Source inspection on current main shows the help advertises `+duration`, while `parseDurationMs` only accepts digit-leading durations and `resolveDirectSchedule` rejects `parseAt` null results.

## Solution Assessment

Yes. The PR fixes the mismatch at the `parseAt` boundary, which is the narrowest maintainable place to accept `+duration` for one-shot `--at` without loosening `--every`, `--stagger`, or other shared duration callers.
