## What This Changes

The PR updates the cron CLI one-shot schedule parser to accept a leading plus on relative `--at` durations and adds parser/CLI regression coverage.

## Best Possible Solution

Land this narrow parser/test fix after required CI and maintainer review, with any release changelog handled during maintainer landing rather than as contributor rework.

## Reproduction Assessment

Yes, source-reproducible: current main and v2026.5.22 advertise `+duration`, while `parseAt` delegates the raw value to a digit-only duration regex and then throws the reported `Invalid --at` error. I did not execute the mutating CLI path in this read-only review.

## Solution Assessment

Yes. Accepting one optional leading plus inside `parseAt` is the narrowest maintainable fix because it honors the existing help contract while keeping the shared duration parser strict for callers such as `--every`.
