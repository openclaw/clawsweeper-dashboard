## Summary

Keep this PR open for normal maintainer merge review. The latest head appears to address the earlier CLI-lane gap and I did not find a blocking correctness defect; the main remaining concern is the intentional session-state behavior change when host timeouts return `ok:false` while a non-cooperating plugin may still be cleaning itself up.

## What This Changes

The branch adds a shared safety-timeout and abort-signal wrapper for plugin-owned `ContextEngine.compact()` and wires it into pi-embedded, CLI, and Codex ownsCompaction lanes with regression tests.

## Best Possible Solution

Land the bounded host-side compaction wrapper after normal merge gates, while keeping plugin authors responsible for honoring the new optional abort signal and documenting/tuning long compaction behavior through the existing timeout setting.

## Reproduction Assessment

Yes by source inspection, though I did not execute a current-main repro because this review is read-only. Current main directly awaits plugin `ContextEngine.compact()` in the affected lanes without passing an abort signal, so a never-settling promise would hang the await.

## Solution Assessment

Yes. Reusing the existing compaction timeout helper, adding an optional abort signal, and applying it to the pi-embedded, CLI, and Codex ownsCompaction lanes is the narrow maintainable fix; the remaining question is maintainer acceptance of the timeout failure semantics.

## Risks / Open Questions

- This intentionally changes plugin-owned compaction failure semantics: after the configured safety timeout, OpenClaw returns control with a failed compaction result while a plugin that ignores the abort signal may still need to clean up background work.
- Existing plugins with legitimately long compactions may now fail after `agents.defaults.compaction.timeoutSeconds` instead of waiting indefinitely, so maintainers should be comfortable with the upgrade behavior and operator tuning path before merge.
