## Label Justifications

- `P1`: A hung plugin-owned compaction can stall an active agent turn and break real agent/channel workflows.
- `merge-risk: 🚨 compatibility`: The PR intentionally changes long-running plugin compactions from unbounded waits to timeout failures under the existing compaction timeout setting.

## Summary

Keep this PR open for the armed automerge flow: source inspection confirms the current-main hang risk, and the patch covers the pi-embedded, CLI, and Codex harness plugin-owned compaction paths without a blocking correctness finding. The remaining merge question is intentional timeout/session-state compatibility, not a ClawSweeper repair defect.

## What This Changes

The PR adds a shared safety-timeout wrapper for plugin-owned `ContextEngine.compact()` calls and wires it into pi-embedded, CLI, and Codex harness compaction paths with abort-signal contract updates, tests, and a changelog entry.

## Best Possible Solution

Land the host-side bounded compaction change after exact-head gates pass, keeping the existing timeout config as the operator knob and documenting that plugins should honor `abortSignal`.

## Reproduction Assessment

Yes. Source inspection of current main shows the native compaction path is timeout-wrapped while plugin-owned queued, recovery, CLI, and Codex lanes directly await `ContextEngine.compact()` with no timeout or abort signal.

## Solution Assessment

Yes. Reusing the existing compaction timeout helper and threading an optional abort signal through the context-engine contract is the narrowest maintainable fix for the host-side stall; the main caution is the deliberate compatibility shift for very slow plugins.
