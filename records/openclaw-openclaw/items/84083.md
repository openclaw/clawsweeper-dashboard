## Summary

This PR targets a real P1 hang and the latest head fixes the earlier timeout-signal cancellation gap, but it still leaves the new public `ContextEngine.compact()` timeout contract broader than the implementation because CLI budget compaction still awaits plugin compaction directly.

## What This Changes

The branch adds a shared safety-timeout wrapper for plugin-owned context-engine compaction, wires it into pi-embedded and Codex ownsCompaction lanes, threads an optional `abortSignal`, and adds focused tests.

## Best Possible Solution

Keep the shared timeout helper, but make the SDK promise true by wrapping the remaining CLI budget compaction path or explicitly narrowing the contract, with a focused regression test for that path before merge.

## Reproduction Assessment

Yes by source inspection: current main directly awaits plugin `ContextEngine.compact()` in the affected lanes, and the CLI budget path still has that shape, so a never-settling plugin promise would hang the await. I did not execute tests because this review is read-only.

## Solution Assessment

No, not as currently proposed: the shared helper is the right direction, but the public contract should not say every host compact call is bounded while CLI budget compaction remains unwrapped. The narrow fix is to wrap that path or narrow the contract before merge.
