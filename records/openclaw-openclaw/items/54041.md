## Summary

Keep open. Current main still has Anthropic beta-header/cache/service-tier plumbing and OpenAI-only server-side compaction, but it does not implement the requested Anthropic `context_management` request payloads, compaction/context-editing stream/replay handling, `pause_after_compaction`, or a `memory_20250818` provider-defined tool path. The related issues cover narrower slices, so this broader Anthropic provider design item is not fully superseded.

## What This Changes

Request native or passthrough support for Anthropic context-management beta features, including server-side compaction, context editing, beta headers, and the `memory_20250818` Memory Tool.

## Best Possible Solution

Keep this as a bundled Anthropic provider design item. The maintainable end state is an opt-in Anthropic configuration/API contract that validates and injects `context_management.edits`, sends required beta headers only when enabled, preserves compaction blocks and context-edit metadata through streaming/replay, handles pause/resume behavior with post-compaction reinjection, and treats `memory_20250818` as a deliberate provider-owned tool or a carefully designed provider-defined-tool SDK seam with filesystem protections.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. There is a high-confidence current-main verification path: source search plus inspection of the Anthropic wrapper, provider registration, request builder, tool conversion, stream parser, docs, and OpenAI-only compaction policy show the requested Anthropic paths are absent.

## Solution Assessment

Unclear for the final implementation, but keeping this open is the right triage outcome. Header passthrough alone is not the best complete fix because compaction blocks, context-edit metadata, replay semantics, pause/resume behavior, and Memory Tool filesystem execution all need provider-owned design.
