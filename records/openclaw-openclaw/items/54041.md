## Summary

Keep open. Current main still lacks Anthropic `context_management` request-body wiring, compaction/context-editing response handling, `pause_after_compaction` semantics, and a direct provider-defined Memory Tool path; the related open compaction trackers do not fully cover context editing or the memory-tool security boundary.

## What This Changes

Request native or passthrough support for Anthropic beta context management, server-side compaction, context editing, beta headers, and Memory Tool provider-defined tools.

## Best Possible Solution

Keep a maintainer-owned Anthropic provider design open: coordinate the compaction/cache work with the existing trackers, add or preserve explicit context-editing tracking, and keep Memory Tool support on the ClawHub/plugin path unless a concrete core SDK gap appears.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source, docs, tests, related-issue state, and current Anthropic docs give a high-confidence absence check for the requested current-main support.

## Solution Assessment

Unclear for final implementation. Header passthrough alone is not enough because request validation, compaction blocks, context-edit metadata, stop reasons, replay behavior, post-compaction reinjection, and Memory Tool filesystem execution all need provider/product design.
