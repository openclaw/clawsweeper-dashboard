## Summary

Keep open. Current main and the latest release still only provide Anthropic beta-header/service-tier handling plus OpenAI-specific server-side compaction; Anthropic `context_management` payload shaping, context-editing lifecycle, and memory-tool semantics remain product/security-sensitive design work.

## What This Changes

The issue requests native or passthrough support for Anthropic context-management beta features, including server-side compaction, context editing, beta headers, and the memory tool.

## Best Possible Solution

Keep this open for maintainer-owned Anthropic context-management design: provider/model params should be explicit, local compaction should remain stable unless opted in, memory-tool behavior should stay plugin/ClawHub unless a core API gap is proven, and stream/replay/security semantics need a deliberate contract.

## Reproduction Assessment

Yes, at source level: current main directly shows the Anthropic request and config paths lack the requested context-management passthrough/native lifecycle. This is a missing capability rather than a broken existing behavior contract.

## Solution Assessment

No, the issue's broad native-or-passthrough menu is not yet a best-fix design. A maintainer decision should choose the provider/config/security boundary before an implementation picks field names or enables provider-defined memory behavior.
