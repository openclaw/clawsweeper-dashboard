## Summary

Keep open. Current main partially covers arbitrary Anthropic beta-header passthrough via `params.anthropicBeta`, but it still lacks Anthropic `context_management` request-body shaping, compaction stop/replay/stream handling, context-edit response metadata handling, and a `memory_20250818` provider-defined tool plus client-side handler. Related #10213 and #65287 overlap only the compaction/cache slices, while closed #48558 covers only the Memory Tool slice, so this broader Anthropic provider design request is not superseded.

## What This Changes

The issue requests native or passthrough support for Anthropic beta context-management features, including Memory Tool, server-side compaction, context editing, and related beta headers.

## Best Possible Solution

Keep this as a bundled Anthropic provider design item. The end state should add documented opt-in params, validated Anthropic `context_management.edits` payload shaping, required beta-header handling only when enabled, replay/stream support for compaction blocks and context-edit metadata, `pause_after_compaction` continuation behavior, and a deliberate Memory Tool implementation or generic provider-defined-tool seam with filesystem safeguards.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. There is a high-confidence current-main verification path: source search plus request-builder, tool-conversion, stream-parser, docs, and OpenAI-only compaction policy inspection show the requested Anthropic runtime paths are absent.

## Solution Assessment

No for a header-only solution; that would not handle compaction blocks, `compaction` stop handling, `applied_edits`, replay, or Memory Tool execution. A provider-owned opt-in design with tests and docs is the narrowest maintainable direction for this issue.
