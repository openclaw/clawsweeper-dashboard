## Summary

Keep open. Current main still lacks Anthropic `context_management` payload support, compaction/context-editing stream semantics, `pause_after_compaction`, and a `memory_20250818` provider-defined tool or handler path. Related #10213 and #65287 cover narrower compaction/cache work, while closed #48558 covers only the Memory Tool slice, so none fully supersedes this broader bundled Anthropic provider design request.

## What This Changes

Requests bundled Anthropic support or passthrough for Memory Tool, server-side compaction, context editing, required beta headers, and provider-defined tool types.

## Best Possible Solution

Keep this as a bundled Anthropic provider design item. The end state should add documented opt-in model params, validated Anthropic `context_management.edits` payload shaping for compaction and context editing, required beta-header handling only when enabled, stream/replay support for compaction blocks and `applied_edits`, `pause_after_compaction` continuation behavior, and a deliberate Memory Tool design as either a provider-owned tool implementation or a generic provider-defined-tool SDK seam.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. There is a high-confidence current-main verification path: source search plus request-builder, tool-conversion, stream-parser, docs, and OpenAI-only compaction policy inspection show the requested Anthropic paths are absent.

## Solution Assessment

Unclear for the final implementation, but keeping this open is the right triage outcome. Header passthrough alone would not solve the request because compaction blocks, `compaction` stop handling, context-edit metadata, replay, and Memory Tool filesystem execution all need provider-owned design.
