## Summary

Keep open: the PR targets a real warning-noise bug from #76095, but the current patch is partial and can still surface persisted disabled-plugin diagnostics while also hiding compatibility warnings for active same-root replacement records.

## What This Changes

Threads normalized plugin config into manifest compatibility diagnostics, suppresses `channelConfigs` warnings for disabled plugin entries, skips same-physical-plugin replacement diagnostics, and adds two manifest-registry tests.

## Best Possible Solution

Deduplicate and suppress only the `channelConfigs` warning by diagnostic key and effective plugin activation across fresh and installed-index diagnostics, while preserving one compatibility warning for active replacement records.

## Reproduction Assessment

Yes. #76095 gives exact startup output and config steps, and source inspection shows a high-confidence path through installed-index diagnostic replay, plugin metadata snapshots with `includeDisabled: true`, and config validation warning surfacing.

## Solution Assessment

No. The helper-level early return is not enough for persisted diagnostics, and removing replacement diagnostics wholesale is broader than needed; filtering/deduping the specific warning at the replay/emission boundary is safer.

  - body: The early return only blocks diagnostics generated in this registry pass. `loadPluginManifestRegistryForInstalledIndex` seeds `params.index.diagnostics` before rebuilding candidates, and metadata/config validation calls it with `includeDisabled: true`, so an existing `without channelConfigs metadata` warning for a disabled plugin can still be surfaced after restart.
