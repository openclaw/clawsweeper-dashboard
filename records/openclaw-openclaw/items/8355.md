## Label Justifications

- `P2`: This is a normal-priority voice-call latency feature with clear source-level scope but a provider-contract design dependency rather than an urgent regression.

## Summary

Keep open. Current main still batches the classic Voice Call STT to embedded-agent to telephony TTS path before Twilio playback; realtime voice is documented and implemented as a separate bypass mode, and the related sentence-level streaming PR attempts are closed unmerged.

## What This Changes

Request a provider-generic streaming TTS pipeline that starts voice-call playback from sentence-level LLM output instead of waiting for complete text and complete telephony audio buffers.

## Best Possible Solution

Add an additive provider-generic streaming telephony speech contract and wire classic Voice Call through embedded-agent partial replies, sentence splitting, ordered provider audio streaming, Twilio/Telnyx playback adapters, barge-in/abort handling, fallback behavior, and focused tests.

## Reproduction Assessment

Yes, source-level. Current main awaits one embedded-agent result, one telephony audio buffer, and then chunks the completed buffer to the media stream; no live carrier setup was run.

## Solution Assessment

Yes at the product direction level, but not as a one-off automated patch. The maintainable route is a shared streaming telephony speech contract plus voice-call adapters, while realtime voice remains a separate bypass architecture.
