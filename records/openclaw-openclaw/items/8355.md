## Summary

Keep open: current main still batches the classic Voice Call STT-to-agent-to-telephony-TTS path, and the newer realtime voice mode is a separate full-duplex bypass rather than the requested sentence-level LLM-to-TTS pipeline.

## What This Changes

The issue requests a provider-generic streaming telephony TTS pipeline that starts synthesis and playback from sentence-level LLM output before the full response is complete.

## Best Possible Solution

Design and land an additive provider-generic streaming telephony speech contract plus Voice Call wiring that consumes embedded-agent partial replies, splits complete sentences, streams provider audio chunks where available, and preserves barge-in, fallback, and ordering across Twilio and Telnyx.

## Reproduction Assessment

Yes, by source inspection: current main awaits a complete embedded-agent result, then a complete telephony TTS buffer, then Twilio chunking. I did not place a live carrier call.

## Solution Assessment

Yes. Keeping this open is the right current outcome because realtime voice is a separate bypass, and the maintainable fix is a provider-generic telephony streaming contract plus Voice Call integration rather than a Twilio-only shortcut.
