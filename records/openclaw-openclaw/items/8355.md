## Summary

Keep open: current main and the latest release still batch the classic Voice Call STT -> LLM -> TTS path, while realtime voice is a separate bypass and the prior streaming-TTS PR attempts are closed unmerged.

## What This Changes

Add provider-generic sentence-level streaming to the classic Voice Call pipeline so TTS audio can begin before the full LLM response and full TTS buffer complete.

## Best Possible Solution

Add an additive provider-generic streaming telephony speech contract and wire classic Voice Call through partial replies, sentence splitting, ordered audio streaming, playback adapters, barge-in abort handling, buffered fallback, and focused tests.

## Reproduction Assessment

Yes, source-level. Current main awaits one embedded-agent result, one telephony audio buffer, and then chunks the completed buffer to the media stream; no live carrier timing setup was run.

## Solution Assessment

Yes at the product direction level, but not as a one-off automated patch. The maintainable route is a shared streaming telephony speech contract plus classic voice-call adapters, while realtime voice remains a separate bypass architecture.
