## Summary

Keep open: current main still batches the classic Voice Call STT -> LLM -> telephony TTS path, and the maintainer discussion already identifies this issue as the canonical provider-generic sentence-level streaming feature rather than something solved by realtime voice mode or Discord TTS streaming.

## What This Changes

The issue requests a sentence-level LLM-to-TTS-to-telephony-audio pipeline for Voice Call to reduce first-audio latency.

## Best Possible Solution

Design an additive streaming telephony speech contract plus Voice Call wiring that uses embedded-agent partial replies, sentence splitting, provider audio chunk streaming where available, and provider adapters with ordering, abort, fallback, and barge-in coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows the classic Voice Call path awaits a complete embedded-agent result, a complete telephony TTS buffer, and then Twilio chunking; no live carrier call was placed.

## Solution Assessment

Yes. Keeping this open is the best current outcome because realtime voice is a separate bypass, while this request needs a provider-generic streaming telephony speech contract rather than a Twilio- or ElevenLabs-only shortcut.
