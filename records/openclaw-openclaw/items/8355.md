## Summary

Keep open. Current main and the latest release still batch the classic Voice Call STT to LLM to TTS path; realtime voice is a separate low-latency bypass, and the closed streaming PRs are useful context but not merged fix paths.

## What This Changes

Request a provider-generic streaming TTS voice-call pipeline that starts TTS from sentence-level LLM output and streams audio chunks to call transports.

## Best Possible Solution

Keep this as the canonical provider-generic classic Voice Call latency feature, implemented through an additive streaming telephony/speech contract, voice-call sentence segmentation, transport playback adapters, and ordering, abort, fallback, and provider compatibility tests.

## Reproduction Assessment

Yes, at source level: current main still awaits a completed embedded-agent result and a completed telephony audio Buffer before Twilio chunks audio. I did not run a live Twilio or Telnyx call to reproduce the exact 5-8 second latency measurement.

## Solution Assessment

Yes, keeping this open is the right outcome; realtime voice is a separate bypass and the closed streaming PRs are not merged fixes. The best fix is a provider-generic streaming telephony contract plus voice-call wiring, not a Twilio-only or provider-specific branch.
