## Summary

Keep open. Current main and v2026.6.8 still batch the classic Voice Call STT -> agent -> telephony TTS path; realtime voice and ordinary non-telephony TTS streaming do not implement the requested provider-generic sentence-level telephony pipeline.

## What This Changes

Request provider-generic sentence-level streaming TTS for classic voice calls so partial LLM output can reach TTS/audio playback before the full response is complete.

## Best Possible Solution

Keep this as the canonical provider-generic classic voice-call latency feature and design an additive streaming telephony/speech contract with sentence segmentation, transport playback adapters, fallback, abort, ordering, and live carrier proof.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.8 still await a completed embedded-agent result and a completed telephony audio Buffer before Twilio sends audio chunks. I did not replay a live Twilio or Telnyx call to measure the exact latency numbers.

## Solution Assessment

Yes, keeping this open is the best current triage outcome. Realtime voice is a separate bypass, and the maintainable fix is a provider-generic streaming telephony contract plus classic voice-call wiring rather than a provider-specific shortcut.
