## Summary

Keep open. Current main and v2026.6.8 still batch the classic Voice Call STT -> agent -> telephony TTS path; realtime voice and ordinary ElevenLabs streaming do not implement the requested provider-generic sentence-level telephony pipeline.

## What This Changes

The issue requests a provider-generic sentence-level LLM-to-TTS-to-audio streaming pipeline for voice-call latency reduction.

## Best Possible Solution

Keep this issue as the canonical provider-generic classic voice-call latency feature, implemented through an additive streaming telephony/speech contract, sentence segmentation, transport playback adapters, and ordering, abort, fallback, and provider compatibility tests.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.8 still await a completed embedded-agent result and a completed telephony audio Buffer before Twilio sends audio chunks. I did not run a live Twilio or Telnyx call to measure the exact latency numbers.

## Solution Assessment

Yes, keeping this open is the best current triage outcome. The maintainable implementation is a provider-generic streaming telephony contract plus classic voice-call wiring, not treating realtime voice or ordinary ElevenLabs non-telephony streaming as the fix.
