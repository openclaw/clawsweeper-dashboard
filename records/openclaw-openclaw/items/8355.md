## Summary

Keep open. Current main and the latest release still batch the classic Voice Call path through one completed agent response, one completed telephony TTS buffer, and only then Twilio audio chunking; realtime voice is a separate bypass, and the prior streaming implementation was closed unmerged.

## What This Changes

The issue requests a provider-generic sentence-level streaming LLM to TTS to audio pipeline for classic voice-call flows.

## Best Possible Solution

Keep this as the canonical provider-generic classic Voice Call latency feature, implemented through an additive streaming telephony/speech contract, voice-call sentence segmentation, Twilio/Telnyx playback adapters, and ordering, abort, fallback, and provider compatibility tests while keeping realtime voice documented as a separate bypass mode.

## Reproduction Assessment

Yes, at source level: current main still awaits a completed embedded-agent result and then a completed telephony audio Buffer before Twilio chunks audio. I did not run a live Twilio/Telnyx call or measure the reported 5-8 second delay in this read-only review.

## Solution Assessment

Yes, keeping this open is the right direction, but the fix is not a narrow bug patch. The maintainable solution is an additive provider-generic streaming telephony contract and voice-call pipeline, not treating realtime voice as equivalent or landing the closed provider-specific attempt unchanged.
