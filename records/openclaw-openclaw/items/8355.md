## Summary

Keep open: current main and latest v2026.6.6 still batch the classic Voice Call STT -> embedded agent -> telephony TTS path, while the newer realtime voice mode is a separate bypass architecture and prior streaming-TTS implementation PRs closed unmerged.

## What This Changes

The issue requests provider-generic sentence-level LLM output piped into streaming telephony TTS for voice calls to reduce first-audio latency.

## Best Possible Solution

Keep this as the canonical provider-generic classic Voice Call latency feature, implemented as an additive streaming telephony speech contract plus voice-call-owned sentence orchestration and transport playback adapters.

## Reproduction Assessment

Yes. Source inspection on current main shows the classic path still waits for a completed embedded-agent response and a completed telephony TTS buffer before Twilio frame playback; I did not replay a live phone call.

## Solution Assessment

Yes, keeping this open is the right outcome. The best fix is not to treat realtime.enabled as equivalent, but to add a provider-generic streaming telephony TTS path with ordering, fallback, abort, and Twilio/Telnyx proof.
