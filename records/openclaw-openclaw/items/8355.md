## Summary

Keep open. Current main and the latest release still batch the classic voice-call STT -> LLM -> TTS path, realtime voice is a separate full-duplex bypass, and the prior streaming PR was closed unmerged, so the provider-generic sentence-level telephony TTS pipeline remains real work.

## What This Changes

The issue requests a provider-generic sentence-level streaming LLM-to-TTS-to-call-audio pipeline for the classic voice-call path.

## Best Possible Solution

Keep this as the canonical provider-generic classic Voice Call latency feature: add an additive streaming telephony speech contract, wire voice-call partial replies through sentence segmentation, support provider audio chunks for Twilio/Telnyx with buffered fallback, and preserve barge-in, abort, ordering, and failure tests while leaving realtime voice as a separate bypass mode.

## Reproduction Assessment

Yes, at source level: current main still awaits one completed embedded-agent result, one completed telephony TTS buffer, and then chunks that buffer to Twilio. I did not replay a live call or independently measure the exact latency numbers.

## Solution Assessment

Yes on the requested direction, but it needs maintainer API design rather than a narrow bug patch. The best fix is an additive provider-generic streaming telephony seam with voice-call-owned sentence segmentation and compatibility fallback, not treating full-duplex realtime mode as the same feature.
