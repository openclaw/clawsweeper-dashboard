## Summary

Keep open. Current main and the latest release still batch the classic Voice Call STT -> embedded agent -> telephony TTS path; realtime voice is a separate bypass, and the previous streaming implementation attempts are closed unmerged.

## What This Changes

The issue requests a provider-generic sentence-level streaming pipeline from LLM output through telephony TTS audio for lower-latency voice calls.

## Best Possible Solution

Keep this as the canonical classic voice-call latency feature and implement it with an additive streaming telephony/speech contract, sentence segmentation, provider audio chunk streaming where available, transport playback adapters, and ordering, abort, fallback, and compatibility tests.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.8 still await a completed embedded-agent result and completed telephony audio Buffer before Twilio sends audio chunks. I did not run a live Twilio or Telnyx call to measure the exact latency numbers.

## Solution Assessment

Yes for triage: keeping this open is the best outcome because realtime voice is a separate bypass and the prior streaming PRs are closed unmerged. The maintainable implementation needs a maintainer-approved provider-generic telephony streaming API and playback/fallback shape.
