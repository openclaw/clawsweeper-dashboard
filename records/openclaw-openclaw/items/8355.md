- `P2`: This is a normal-priority voice-call latency feature with clear source-level scope but shared provider-contract and playback-semantics design work.

## Summary

Source inspection confirms the classic voice-call path is still buffered from embedded-agent response through telephony TTS before Twilio playback, while realtime voice is a separate bypass mode; keep this open as the canonical provider-generic streaming telephony TTS feature.

## What This Changes

The issue requests a sentence-level classic voice-call pipeline that streams LLM text into provider TTS audio and then into telephony playback instead of waiting for full text and full audio buffers.

## Best Possible Solution

Add an additive provider-generic streaming telephony speech contract and wire classic Voice Call through embedded-agent partial replies, sentence splitting, ordered provider audio streaming, Twilio/Telnyx playback adapters, barge-in/abort handling, buffered fallback behavior, and focused tests.

## Reproduction Assessment

Yes, source-level. Current main awaits one embedded-agent result, one telephony audio buffer, and then chunks the completed buffer to the media stream; no live carrier timing setup was run.

## Solution Assessment

Yes at the product direction level, but not as a one-off automated patch. The maintainable route is a shared streaming telephony speech contract plus classic voice-call adapters, while realtime voice remains a separate bypass architecture.
