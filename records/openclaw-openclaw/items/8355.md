## Summary

Keep open. Current main still batches the classic Voice Call STT -> embedded-agent -> telephony TTS path, while realtime voice is a separate low-latency bypass and the prior streaming-TTS implementation PR was closed unmerged. The Telnyx comment also supports keeping this as a provider-generic feature rather than a Twilio-only patch.

## What This Changes

Request a provider-generic sentence-level streaming TTS pipeline for voice calls so partial LLM output can be synthesized and played before the full response completes.

## Best Possible Solution

Add an explicit provider-generic streaming telephony TTS contract and wire classic Voice Call through embedded-agent partial reply callbacks, sentence splitting, provider audio streaming, ordered Twilio/Telnyx playback, abort/barge-in handling, and fallback tests.

## Reproduction Assessment

Yes, source-level. The current classic path awaits one generated text result, one telephony audio buffer, and then chunks the completed buffer to the media stream; no live carrier setup was run.

## Solution Assessment

Yes at the product direction level, but it needs maintainer design before implementation. The maintainable route is a shared streaming telephony speech contract plus voice-call provider adapters, not treating the separate realtime voice bypass as the fix.
