## Summary

Keep open. Current main and the latest release still batch the classic Voice Call STT -> embedded-agent -> telephony TTS path, while `realtime.enabled` is a separate full-duplex bypass and the prior streaming-TTS PRs were closed unmerged.

## What This Changes

Request to add a sentence-level streaming LLM-to-TTS-to-audio pipeline for classic voice-call flows so playback can start before the full assistant response and full TTS buffer are complete.

## Best Possible Solution

Add an explicit provider-generic streaming telephony TTS contract and wire classic Voice Call through embedded-agent partial replies, sentence splitting, provider audio streaming, ordered Twilio/Telnyx playback, abort/barge-in handling, fallback behavior, and focused tests.

## Reproduction Assessment

Yes, source-level. Current main awaits one generated text result, one telephony audio buffer, and then chunks the completed buffer to the media stream; no live carrier setup was run.

## Solution Assessment

Yes at the product direction level, but it needs maintainer design before implementation. The maintainable route is a shared streaming telephony speech contract plus voice-call provider adapters, not treating the separate realtime voice bypass as the fix.
