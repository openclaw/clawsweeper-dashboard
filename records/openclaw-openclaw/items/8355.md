## Label Justifications

- `P2`: Classic voice calls remain functional, but multi-second first-audio latency is a material conversational-quality limitation affecting real users.
- `impact:other`: The concrete impact is delayed audible agent responses during voice calls, which does not fit the narrower owned impact categories.

## Summary

Current main still batches the classic voice-call path at both the agent-response and telephony-TTS stages; realtime voice is a separate bypass, and the prior streaming PR closed unmerged, so the provider-generic latency feature remains necessary but needs a maintainer-owned API and fallback decision.

## What This Changes

Add provider-generic sentence-level LLM-to-TTS audio streaming to classic voice calls while preserving ordered playback, barge-in cancellation, and fallback behavior.

## Best Possible Solution

Add the smallest additive async audio-chunk capability at the shared speech-provider seam, keep sentence segmentation and orchestration inside the voice-call plugin, support transports through explicit capabilities, and retain the current completed-buffer path as the tested fallback when streaming is unavailable or fails before playback begins.

## Reproduction Assessment

Yes, at source level: current main awaits a completed agent result, then a completed telephony audio buffer, before playback can consume audio. This review did not independently measure the reported 5–8 second live latency.

## Solution Assessment

Yes on the provider-generic direction, but the proposed interface replacement is not yet the best exact shape; an additive stream capability with voice-call-owned segmentation and a buffered compatibility fallback is narrower and safer.

## Risks / Open Questions

- The permanent contract crosses a shipped Plugin SDK/provider boundary, so maintainers must decide whether streaming is an additive speech capability, a voice-call-local runtime facade, or another supported API shape.
- Sentence segmentation and concurrent synthesis can regress ordering, spoken-output filtering, barge-in cancellation, zero-delivery detection, and provider fallback unless those existing contracts are explicitly carried forward.
- Twilio and Telnyx have different playback capabilities; promising provider-generic behavior requires a clear capability model and tested buffered fallback for providers or transports that cannot accept live audio chunks.
