## Summary

Keep open. Current `main` and latest release `v2026.6.10` still buffer the classic Voice Call STT -> embedded agent -> telephony TTS path before sending Twilio audio chunks; realtime voice and the open early-reply PR are adjacent latency paths, not this provider-generic sentence-level telephony streaming pipeline.

## What This Changes

The issue requests a provider-generic sentence-level streaming pipeline from LLM output through telephony TTS to voice-call audio playback.

## Best Possible Solution

Keep this as the canonical classic voice-call latency feature and design an additive provider-generic telephony streaming contract with sentence segmentation, audio chunk adapters, and ordering, abort, fallback, and compatibility tests.

## Reproduction Assessment

Yes, at source level: current `main` and `v2026.6.10` await a completed embedded-agent result and completed telephony audio `Buffer` before Twilio sends chunks. I did not run a live Twilio or Telnyx call to measure the exact latency numbers.

## Solution Assessment

Yes for triage: keeping this open is the best outcome because realtime voice is a separate bypass, prior streaming PRs are closed unmerged, and the open adjacent PR targets early block replies for compaction latency rather than this full provider-generic telephony streaming pipeline.
