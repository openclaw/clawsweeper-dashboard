## Summary

Keep open. Current main still buffers the classic Voice Call STT to embedded-agent to telephony TTS path before Twilio sends audio chunks; newer speech streaming and realtime voice support are adjacent infrastructure, not the requested provider-generic sentence-level telephony pipeline.

## What This Changes

The issue requests a provider-generic sentence-level LLM to TTS to telephony-audio streaming pipeline for voice calls to reduce first-audio latency.

## Best Possible Solution

Keep this as the canonical provider-generic classic voice-call latency feature and design an additive telephony streaming speech contract plus Twilio/Telnyx playback semantics before implementation.

## Reproduction Assessment

Yes, at source level: the current classic voice-call path awaits a completed embedded-agent result and completed telephony audio Buffer before Twilio sends chunks. I did not run a live Twilio or Telnyx call to measure the exact reported latency numbers.

## Solution Assessment

Yes for triage: keeping this open is the best outcome because realtime voice is a separate bypass and prior streaming PRs are closed unmerged. The maintainable fix needs a maintainer-approved provider-generic telephony streaming API and playback/fallback shape.
