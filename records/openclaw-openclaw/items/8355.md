## Summary

Current main still batches the classic Voice Call STT → agent → TTS path, while realtime voice is a separate bypass and the prior implementation PR closed unmerged; this remains the canonical provider-generic latency feature and needs maintainer API/product direction before implementation.

## What This Changes

Add provider-generic sentence-level streaming from partial agent output through telephony TTS playback to reduce voice-call first-audio latency.

## Best Possible Solution

Extend the shared speech capability with streaming telephony output, then have the voice-call plugin feed sentence-safe partial spoken output into provider-neutral ordered playback with abort, barge-in, and buffered fallback coverage.

## Reproduction Assessment

Yes, at source level. Current main still completes the agent response and telephony audio buffer before Twilio begins frame delivery, directly demonstrating the architectural latency condition; this read-only review did not run a live call to reproduce the reported timing numbers.

## Solution Assessment

Yes in direction, but not as direct ElevenLabs or Twilio special-casing. The narrowest maintainable solution is a shared streaming telephony speech contract plus voice-call orchestration that preserves the existing output-safety, queue, abort, and fallback invariants.
