## Summary

Keep open. Current main still batches the classic Voice Call STT -> LLM -> TTS path, and the discussion already distinguishes this canonical feature from realtime voice and from the closed unmerged streaming PR.

## What This Changes

Request sentence-level LLM-to-TTS audio streaming in the Voice Call plugin so callers hear earlier response audio on classic STT -> LLM -> TTS calls.

## Best Possible Solution

Keep this as the canonical classic Voice Call latency feature and design an additive provider-generic streaming telephony speech contract with sentence splitting, provider audio chunk streams, Twilio/Telnyx adapters, barge-in/abort handling, fallback behavior, and focused contract plus integration tests.

## Reproduction Assessment

Yes. A high-confidence source reproduction path exists on current main by tracing `handleInboundResponse()` through `generateVoiceResponse()`, `manager.speak()`, `playTtsViaStream()`, and `synthesizeForTelephony()` to see the complete-text and complete-buffer waits before audio chunks are sent.

## Solution Assessment

Unclear because there is no current patch to accept. The maintainable direction is an additive provider-generic streaming contract and voice-call integration, not closing this as covered by realtime voice.
