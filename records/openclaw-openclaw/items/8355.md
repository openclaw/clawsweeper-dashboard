## Summary

Keep open. Current main and v2026.6.8 still batch the classic Voice Call STT -> LLM -> TTS path; ordinary ElevenLabs TTS streaming and the realtime voice bypass do not implement the requested provider-generic sentence-level telephony pipeline.

## What This Changes

The issue requests a provider-generic streaming text-to-speech pipeline for voice calls so sentence-level LLM output can be synthesized and played before the full response is complete.

## Best Possible Solution

Keep this as the canonical provider-generic classic Voice Call latency feature, implemented through an additive streaming telephony/speech contract, voice-call sentence segmentation, transport playback adapters, and ordering, abort, fallback, and provider compatibility tests.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.8 still await a completed embedded-agent result and a completed telephony audio Buffer before Twilio sends audio chunks. I did not run a live Twilio or Telnyx call to measure the exact latency numbers.

## Solution Assessment

Yes, keeping this open is still the best current outcome. Realtime voice is a separate low-latency bypass, and the maintainable fix is a provider-generic streaming telephony contract plus classic voice-call wiring rather than a Twilio-only shortcut.
