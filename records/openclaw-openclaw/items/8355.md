## Summary

Keep open. Current main and v2026.6.10 still buffer the classic Voice Call STT -> embedded agent -> telephony TTS path before Twilio sends audio chunks; the newer generic speech streaming support does not wire a sentence-level telephony pipeline, and https://github.com/openclaw/openclaw/pull/94015 is only an adjacent early-reply/compaction fix.

## What This Changes

Request a classic voice-call pipeline that streams LLM output at sentence granularity into TTS/audio playback instead of waiting for complete text and complete audio buffers.

## Best Possible Solution

Keep this as the canonical classic voice-call latency feature and implement an additive provider-generic streaming telephony/speech contract with sentence segmentation, audio chunk adapters, and ordering, abort, fallback, and compatibility tests.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.10 still await a completed embedded-agent result and completed telephony audio Buffer before Twilio sends chunks. I did not run a live Twilio or Telnyx call to measure the exact latency numbers.

## Solution Assessment

Yes for triage: keeping this open is the best outcome because realtime voice is a separate bypass and the prior streaming PRs are closed unmerged. The maintainable implementation needs a maintainer-approved provider-generic telephony streaming API and playback/fallback shape.
