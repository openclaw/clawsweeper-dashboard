## Summary

Keep open. Current main still batches the classic Voice Call STT -> embedded-agent -> telephony TTS path, the realtime voice mode is a separate full-duplex bypass, and the known streaming-TTS implementation attempts remain closed unmerged.

## What This Changes

Request a provider-generic sentence-level streaming TTS pipeline for voice calls that overlaps LLM generation, TTS synthesis, and Twilio/Telnyx playback.

## Best Possible Solution

Add a provider-generic streaming telephony TTS contract and wire classic Voice Call through embedded-agent partial replies, sentence splitting, ordered provider audio streaming, Twilio/Telnyx playback adapters, barge-in/abort handling, fallback behavior, and focused tests.

## Reproduction Assessment

Yes, source-level. Current main awaits one generated text result, one telephony audio buffer, and then chunks the completed buffer to the media stream; no live carrier setup was run.

## Solution Assessment

Yes at the product direction level, but implementation needs maintainer design first. The maintainable route is a shared streaming telephony speech contract plus provider adapters, not treating the separate realtime voice bypass as the fix.
