## Summary

Keep open. Current main still buffers the classic Voice Call STT -> LLM -> TTS path, while realtime voice is a separate bypass and the prior streaming-TTS PR was closed unmerged.

## What This Changes

The issue requests a provider-generic streaming voice-call pipeline that sends sentence-level LLM output into streaming TTS audio before the full response is complete.

## Best Possible Solution

Keep this as the canonical classic Voice Call latency feature and design an additive provider-generic streaming telephony speech contract with sentence splitting, provider audio chunk streams, playback adapters, fallback behavior, and focused contract plus integration tests.

## Reproduction Assessment

Yes. A high-confidence source trace on current main follows `handleInboundResponse()` through `generateVoiceResponse()`, `manager.speak()`, `playTtsViaStream()`, and `synthesizeForTelephony()` to show complete-text and complete-buffer waits before audio chunks are sent.

## Solution Assessment

Unclear because there is no current patch to accept. The narrow maintainable direction is an additive provider-generic streaming speech contract and voice-call integration, not closing this as covered by realtime voice.
