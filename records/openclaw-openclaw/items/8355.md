## Summary

Keep open: current main and v2026.5.22 still batch the classic Voice Call STT -> LLM -> TTS path. Realtime voice is documented as a separate bypass, and prior streaming-TTS PRs were closed unmerged, so the provider-generic sentence-level pipeline remains real product and contract work.

## What This Changes

Request a sentence-level streaming pipeline for classic voice-call responses so LLM output can feed TTS and telephony audio before the full response is complete.

## Best Possible Solution

Add an additive provider-generic streaming telephony speech contract and wire classic Voice Call through partial replies, sentence splitting, ordered audio streaming, playback adapters, barge-in/abort handling, buffered fallback, docs, and focused tests.

## Reproduction Assessment

Yes, source-level. Current main awaits one embedded-agent result, one telephony audio buffer, and then chunks the completed buffer to the media stream; no live carrier timing setup was run.

## Solution Assessment

Yes at the product direction level, if implemented as a provider-generic additive contract. A narrow Twilio-only or ElevenLabs-only shortcut would not be the best maintainable solution because Telnyx and shared speech-provider semantics are part of the request.
