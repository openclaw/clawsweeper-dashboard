## Summary

Keep open. Current main still buffers the classic Voice Call STT -> LLM -> TTS path, and the newer realtime voice mode is a separate bypass rather than the provider-generic sentence-level TTS pipeline requested here.

## What This Changes

Request a provider-generic Voice Call streaming pipeline that feeds sentence-level LLM output into streaming TTS and immediate Twilio/Telnyx audio playback.

## Best Possible Solution

Keep this as the canonical Voice Call latency feature and design an additive streaming telephony contract with sentence splitting, embedded-agent partial reply callbacks, provider audio chunk streaming, abort/barge-in handling, and Twilio/Telnyx ordering tests.

## Reproduction Assessment

Yes, at source level. The current main path still returns one completed voice response, synthesizes one completed telephony buffer, and only then chunks it to Twilio Media Streams.

## Solution Assessment

No current implementation solves it yet. The maintainable direction is a provider-generic streaming telephony contract and Voice Call wiring, not treating realtime voice as the same feature or landing a one-provider shim.
