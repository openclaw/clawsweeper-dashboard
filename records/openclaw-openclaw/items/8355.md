## Summary

Keep open: current main still batches the classic Voice Call path from completed agent response to completed telephony TTS buffer before Twilio chunking, and the shipped realtime voice mode is a separate bypass rather than this provider-generic sentence-level pipeline.

## What This Changes

The issue requests a provider-generic streaming Voice Call pipeline that sends sentence-level LLM output through streaming TTS to telephony audio playback.

## Best Possible Solution

Keep this as the canonical provider-generic classic Voice Call latency feature and design an additive streaming speech contract plus voice-call adapters that preserve barge-in, abort, fallback, ordering, and provider boundaries.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes: source inspection on current main shows the classic path waits for a complete agent result, a complete telephony TTS Buffer, and then Twilio chunking; I did not place a live carrier call.

## Solution Assessment

Is this the best way to solve the issue? Yes for triage: keeping this open is the right outcome because realtime voice is a separate full-duplex bypass, while this request needs an additive provider-generic streaming TTS contract rather than a Twilio- or ElevenLabs-only shortcut.
