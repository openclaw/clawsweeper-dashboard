## Summary

Keep open. Current main still batches the classic Voice Call STT -> LLM -> TTS path, and maintainer discussion already distinguishes this requested sentence-level pipeline from the newer realtime voice bypass and from the closed unmerged implementation PR.

## What This Changes

The issue requests a provider-generic sentence-level streaming TTS pipeline for voice calls so partial LLM output can be synthesized and played before the full response and audio buffer are complete.

## Best Possible Solution

Keep this as the canonical classic Voice Call latency feature and implement an additive streaming telephony speech contract with voice-call sentence splitting, provider audio chunk streams, Twilio/Telnyx playback adapters, abort/barge-in semantics, fallback behavior, and focused contract plus integration tests.

## Reproduction Assessment

Yes. A high-confidence source reproduction path exists on current main by tracing `handleInboundResponse()` through `generateVoiceResponse()`, `manager.speak()`, `playTtsViaStream()`, and `synthesizeForTelephony()` to see the complete-text and complete-buffer waits before playback chunks begin.

## Solution Assessment

Unclear for implementation because there is no current patch to accept. The maintainable direction is an additive provider-generic streaming contract and voice-call integration, not closing this as covered by realtime voice.
