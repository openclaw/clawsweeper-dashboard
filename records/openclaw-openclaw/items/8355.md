## Summary

Keep open. Current main and v2026.6.10 still batch the classic Voice Call STT -> embedded agent -> telephony TTS path; realtime voice and the open early-payload PR are adjacent latency work, not the requested provider-generic sentence-level TTS pipeline.

## What This Changes

Request a provider-generic classic Voice Call streaming TTS pipeline that turns streamed LLM sentences into telephony audio chunks instead of buffering full text and full audio first.

## Best Possible Solution

Keep this as the canonical classic Voice Call latency feature and implement an additive provider-generic telephony streaming contract with sentence segmentation, transport playback adapters, and ordering, abort, fallback, and compatibility tests.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.10 still await a completed embedded-agent result and completed telephony audio Buffer before Twilio sends chunks. I did not run a live Twilio or Telnyx call to measure latency.

## Solution Assessment

Yes for triage: keeping this open is the best outcome because realtime voice is a separate bypass and the prior streaming implementation attempts closed unmerged. The implementation itself needs a maintainer-approved provider-generic telephony streaming API and playback/fallback shape.
