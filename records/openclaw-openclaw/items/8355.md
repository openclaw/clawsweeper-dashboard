## Summary

Keep open. Current main still buffers the classic Voice Call STT -> LLM -> TTS path, while realtime voice is documented and wired as a separate full-duplex bypass rather than sentence-level streaming for the classic pipeline.

## What This Changes

Add a provider-generic sentence-level LLM-to-TTS-to-audio streaming pipeline for classic Voice Call flows so callers hear initial audio before full response synthesis finishes.

## Best Possible Solution

Design an additive provider-generic streaming telephony speech contract, then wire Voice Call with sentence splitting, provider audio chunk streams, Twilio/Telnyx playback adapters, fallback behavior, barge-in cancellation, and focused contract plus integration tests.

## Reproduction Assessment

Yes, as a source-level reproduction of the current limitation. The current main trace follows `handleInboundResponse()` -> `generateVoiceResponse()` -> `manager.speak()` -> `playTtsViaStream()` -> `synthesizeForTelephony()` and shows complete-text plus complete-buffer waits before audio chunks are sent.

## Solution Assessment

Unclear as an implementation because there is no current patch to accept. The maintainable direction is an additive provider-generic streaming speech contract and Voice Call integration, not closing this as covered by realtime voice.
