## Summary

Keep open: current main still implements classic Voice Call as a buffered STT -> complete agent response -> complete telephony TTS Buffer -> Twilio chunking path, while realtime voice is a separate bypass and the prior implementation PR is closed unmerged.

## What This Changes

Add a provider-generic streaming TTS pipeline for voice calls so LLM output can be sentence-split, synthesized, and played before the full response completes.

## Best Possible Solution

Keep this as the canonical provider-generic classic Voice Call latency feature and design an additive streaming speech contract plus Twilio/Telnyx playback adapters that preserve barge-in, abort, fallback, ordering, and provider capability boundaries.

## Reproduction Assessment

Yes. Source inspection on current main shows the classic path waits for a complete agent result, a complete telephony TTS Buffer, and then Twilio chunking; I did not run a live carrier call.

## Solution Assessment

Yes. Keeping this open is the right outcome because realtime voice solves a different full-duplex bypass, while the requested fix needs an additive provider-generic streaming TTS contract rather than a Twilio- or ElevenLabs-only shortcut.
