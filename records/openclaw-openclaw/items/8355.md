## Summary

Keep open. Current main still batches the classic Voice Call STT -> LLM -> TTS path, while maintainer comments already distinguish this feature from realtime voice and from the closed unmerged streaming PR.

## What This Changes

Request a provider-generic sentence-level Voice Call pipeline that streams LLM output into TTS and telephony audio playback to reduce first-audio latency.

## Best Possible Solution

Keep this as the canonical classic Voice Call latency feature and implement an additive streaming telephony speech contract with sentence splitting, provider audio chunk streams, Twilio/Telnyx playback adapters, barge-in/abort semantics, fallback behavior, and focused contract plus integration tests.

## Reproduction Assessment

Yes. A high-confidence source reproduction path exists on current main by tracing `handleInboundResponse()` through `generateVoiceResponse()`, `manager.speak()`, `playTtsViaStream()`, and `synthesizeForTelephony()` to see the complete-text and complete-buffer waits before playback chunks begin.

## Solution Assessment

Unclear because there is no current patch to accept. The maintainable direction is an additive provider-generic streaming contract and voice-call integration, not closing this as covered by realtime voice.
