## Summary

Keep open. Current main still implements the classic Voice Call path as STT final transcript -> one completed agent response -> one completed telephony TTS buffer -> Twilio chunk playback, while the newer realtime voice mode is documented and validated as a separate full-duplex bypass. A maintainer already identified this issue as the canonical provider-generic sentence-level LLM -> TTS -> audio streaming feature after the related implementation PR was closed unmerged.

## What This Changes

The issue requests a provider-generic sentence-level streaming TTS pipeline for voice calls so LLM partial output can be synthesized and played before the full response and full audio buffer are complete.

## Best Possible Solution

Keep this as the canonical classic Voice Call latency feature. The desired end state is an additive, versioned streaming speech/telephony contract with voice-call wiring for agent partial replies, sentence splitting, provider audio chunk streams, Twilio/Telnyx playback adapters, barge-in/abort semantics, fallback behavior, and focused contract plus integration tests; realtime voice should remain documented as a separate low-latency mode.

## Reproduction Assessment

Yes. A high-confidence source reproduction path exists on current main: trace `handleInboundResponse()` through `generateVoiceResponse()`, `manager.speak()`, `playTtsViaStream()`, and `synthesizeForTelephony()` to see that the classic path waits for a complete text result and complete telephony audio buffer before caller playback chunks begin.

## Solution Assessment

Unclear for an implementation because there is no current patch to accept. Closing as implemented would be incorrect; the narrow maintainable direction is an additive provider-generic streaming contract and voice-call integration, not treating realtime voice as the same solution.
