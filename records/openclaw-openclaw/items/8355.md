## Label Justifications

- `P2`: This is a meaningful voice-call performance feature with limited blast radius, not an emergency regression.
- `impact:other`: The reported user impact is multi-second first-audio latency that makes conversations feel unnatural.

## Summary

Keep open: current main and the latest release still batch the classic Voice Call STT→agent→TTS path, while realtime voice is a separate full-duplex architecture and the prior streaming implementation closed unmerged.

## What This Changes

Requests provider-generic sentence-level streaming from LLM output through telephony TTS to voice-call audio playback.

## Best Possible Solution

Implement an ordered, abortable Voice Call partial-reply pipeline that reuses shared speech streaming, adds only narrowly necessary telephony contracts, supports Twilio and Telnyx adapters, and retains buffered fallback.

## Reproduction Assessment

Yes at source level: current main completes both agent text generation and telephony synthesis before Twilio begins playback. This read-only review did not run a live call latency measurement.

## Solution Assessment

No, not exactly as proposed: the outcome is valid, but the narrowest maintainable solution should reuse the existing shared speech-streaming API and preserve structured spoken-output and playback lifecycle contracts rather than wholesale replacing the current interfaces.
