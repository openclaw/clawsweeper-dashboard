## Summary

Keep open. Current main and the latest release still batch the classic Voice Call STT -> LLM -> TTS path; realtime voice is a separate low-latency bypass, and the prior streaming TTS PRs are closed unmerged design input rather than fix paths.

## What This Changes

The issue requests a provider-generic sentence-level streaming pipeline that overlaps voice-call LLM output, TTS synthesis, and telephony audio playback to reduce first-audio latency.

## Best Possible Solution

Keep this as the canonical provider-generic classic Voice Call latency feature and design an additive streaming telephony/speech contract with sentence segmentation, Twilio/Telnyx playback adapters, provider streaming implementations, and ordering/abort/fallback tests.

## Reproduction Assessment

Yes, at source level: current main still awaits one completed embedded-agent result and one completed telephony audio Buffer before Twilio sends the response chunks. I did not run a live Twilio or Telnyx call to reproduce the exact latency number.

## Solution Assessment

Yes, keeping this open is the right outcome; realtime voice is a separate bypass and the closed streaming PRs are not merged fixes. The best fix is a provider-generic streaming telephony contract plus voice-call wiring, not a Twilio-only or provider-specific branch.
