## Summary

Keep open: current main still lacks a first-class WhatsApp listen-only/hooks-only mode. WhatsApp `message_received` hooks are opt-in inside the admitted inbound reply path, while access control, read receipts, pairing replies, and reactions still need an explicit product/security design before a no-agent/no-outbound ingestion mode can ship.

## What This Changes

Request a WhatsApp account or route mode that emits inbound `message_received` hooks without agent runs, LLM calls, replies, read receipts, typing, reactions, polls, or pairing messages.

## Best Possible Solution

Define a WhatsApp-owned, explicit ingestion mode that admits configured DMs/groups to privacy-gated hook emission, bypasses agent dispatch and all outbound side effects, and documents validation, status, and live-proof expectations.

## Reproduction Assessment

Not applicable: this is a feature/config request, not a broken documented behavior. Source inspection gives high confidence that current main lacks the requested listen-only mode and only emits WhatsApp message hooks inside existing admitted inbound processing.

## Solution Assessment

Unclear: the requested direction is plausible, but the exact config shape, privacy boundary, and side-effect guarantees need maintainer product/security judgment. The maintainable path is a WhatsApp-owned mode plus tests/docs/live proof, not a generic core workaround.
