## Summary

Keep open: current main and the latest release have the narrower WhatsApp `message_received` hook opt-in, but they still do not provide a listen-only/hooks-only mode that bypasses agent/session/model routing and all outbound WhatsApp side effects. The linked read-only PR is closed unmerged and was scoped to no-outbound behavior, not the remaining no-agent hook pipeline.

## What This Changes

The issue requests a WhatsApp listen-only/hooks-only mode that emits inbound `message_received` hooks without creating agent runs or any outbound WhatsApp activity.

## Best Possible Solution

Define a WhatsApp-owned account or route mode that emits opted-in inbound hooks before conversational routing, skips agent/model execution, and suppresses all outbound WhatsApp side effects by construction.

## Reproduction Assessment

Yes from source inspection for the missing capability: current main exposes hook opt-in, but the hook path is behind access-control admission and then continues into agent dispatch. I did not run live WhatsApp because this is a read-only cleanup review.

## Solution Assessment

No current implementation is sufficient. The best direction is a WhatsApp-owned mode, but maintainers still need to choose the durable config/routing contract and how it relates to the closed unmerged `readOnly` PR.
