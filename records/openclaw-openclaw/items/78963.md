## Summary

Keep open: current main still lacks a first-class WhatsApp listen-only/hooks-only mode. Existing `message_received` hooks are privacy opt-in, run only after inbound access admission, and are followed by the normal inbound reply turn, so the request is a new WhatsApp channel/config mode rather than an already-fixed bug.

## What This Changes

The issue requests a WhatsApp listen-only/hooks-only inbound mode that emits plugin hooks without agent sessions, model calls, read receipts, typing, reactions, pairing replies, or other outbound activity.

## Best Possible Solution

Design an explicit WhatsApp account/route ingestion mode in the WhatsApp plugin that admits configured DMs/groups to hook emission, bypasses agent dispatch and all outbound side effects, and documents validation, privacy opt-in, and status semantics.

## Reproduction Assessment

Not applicable: this is a feature/config request, not a report of broken existing behavior. Source inspection on current main shows no listen-only mode, and existing hooks are tied to the admitted inbound processing path.

## Solution Assessment

Unclear: the direction is plausible, but the exact config shape and safety boundaries need maintainer product/security judgment. The maintainable path is likely a WhatsApp-owned mode with tests and docs, not a generic core bypass.
