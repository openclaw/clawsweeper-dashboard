## Summary

Keep open: current main and latest release have the narrower WhatsApp `message_received` hook opt-in, but still lack a listen-only/hooks-only mode that bypasses access-control blocking, agent/session/model routing, and all outbound WhatsApp side effects. The linked read-only PR is closed unmerged and was explicitly scoped to no-outbound behavior, leaving the no-agent hook pipeline unresolved.

## What This Changes

The issue requests a WhatsApp listen-only/hooks-only inbound mode that emits plugin hooks without creating agent runs, calling models, or sending outbound WhatsApp activity.

## Best Possible Solution

Define a WhatsApp-owned account or route mode that emits opted-in inbound hooks before conversational routing, skips agent/model execution, and suppresses all outbound WhatsApp side effects by construction.

## Reproduction Assessment

Yes from source inspection: current main has a hook opt-in, but the path is behind admission and continues into agent dispatch rather than stopping after hook emission. I did not run live WhatsApp because this cleanup review is read-only.

## Solution Assessment

No current implementation is sufficient. The best direction is a WhatsApp-owned mode, but maintainers still need to choose the durable config/routing contract and whether to reuse any part of the closed read-only design.
