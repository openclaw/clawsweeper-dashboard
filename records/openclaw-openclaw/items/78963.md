## Summary

Keep open: current main still lacks a first-class WhatsApp listen-only/hooks-only mode. The existing hook support is an opt-in inside admitted inbound reply processing, and the path still proceeds into agent dispatch while access control and outbound side effects need an explicit product/security design.

## What This Changes

The issue requests a WhatsApp listen-only/hooks-only mode that receives inbound group and DM messages, emits `message_received` hooks, and guarantees no agent run or outbound WhatsApp activity.

## Best Possible Solution

Define a WhatsApp-owned ingestion mode that explicitly admits configured DMs/groups to privacy-gated hook emission, bypasses agent dispatch, disables every outbound side effect, and documents config, status, tests, and live proof expectations.

## Reproduction Assessment

Not applicable: this is a feature/config request, not broken documented behavior. Source inspection gives high confidence that current main lacks the requested listen-only mode and only emits WhatsApp message hooks inside admitted inbound processing.

## Solution Assessment

Unclear: the requested direction is plausible, but the exact config shape, privacy boundary, and side-effect guarantees need maintainer product/security judgment. The maintainable path is a WhatsApp-owned mode plus focused tests, docs, and live proof, not a generic core workaround.
