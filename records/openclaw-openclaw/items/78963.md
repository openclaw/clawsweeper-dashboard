## Summary

Keep open: current main and v2026.6.10 still have only the narrower WhatsApp `message_received` hook opt-in, while the requested listen-only/hooks-only mode still lacks a terminal no-agent/no-model/no-outbound path for admitted or access-blocked inbound messages.

## What This Changes

Request a first-class WhatsApp account or route mode that emits inbound `message_received` hooks without creating agent runs, calling models, or sending any WhatsApp outbound activity.

## Best Possible Solution

Define a WhatsApp-owned account or route mode that emits opted-in inbound hooks before conversational routing, avoids session/model dispatch, and suppresses all outbound WhatsApp side effects by construction.

## Reproduction Assessment

Yes, at source level: current main has hook opt-in, but access-denied messages return before hooks and admitted messages still pass through channel turn/session machinery. I did not run live WhatsApp because this cleanup review is read-only.

## Solution Assessment

Unclear: the requested invariant is maintainable, but the best permanent config/routing contract still needs WhatsApp/channel owner judgment. The current hook opt-in and the closed read-only PR are only partial slices.
