## Summary

Keep open: current main and the latest release still implement/document oldest-first round-robin auth-profile ordering, not MRU/last-success stickiness. The 402 classifier and profile-rotation path have improved, but the exact live multi-key silent-402 UX still needs proof and the ordering change needs maintainer product direction.

## What This Changes

The issue requests MRU-sticky API-key auth profile ordering after session reset and silent intermediate 402 failover across multiple same-provider API keys.

## Best Possible Solution

Define one compatibility-safe auth-profile selection policy around explicit order, session overrides, cooldowns, and lastUsed/lastGood state, then back it with focused tests, docs, and live multi-key 402 proof before changing defaults.

## Reproduction Assessment

Partly yes: current source and tests reproduce the ordering half by ignoring lastGood and using oldest-first lastUsed. The live multi-key 402 chat-rendering path has not been replayed, so full reproduction is not high-confidence.

## Solution Assessment

No for a simple default flip: current source and docs intentionally preserve round-robin behavior. The safer path is a maintainer-approved policy or mode, with tests and live proof for the 402 UX path.
