## Summary

Keep open. Current main and v2026.5.12 still intentionally use oldest-first round-robin auth-profile ordering, while the requested MRU/last-success behavior and any new ordering-mode config need maintainer policy; the 402 failover path has related fixes but the exact visible intermediate-error report is not fully proven closed.

## What This Changes

The issue asks for API-key auth profile selection to prefer the most recently successful key after a new session and to suppress intermediate 402 errors while failover retries another key.

## Best Possible Solution

Maintainers should choose and document a supported auth-profile selection policy, likely a config-backed MRU/last-success mode or an explicit default change, then add final-only 402 failover coverage for successful profile retries.

## Reproduction Assessment

Partly yes. Source and tests give a high-confidence path for the ordering half because main still prefers oldest lastUsed over lastGood; the exact live multi-key 402 rendering sequence was not reproduced.

## Solution Assessment

Unclear. A blind default flip would conflict with current docs/tests, so a maintainer-approved config-backed MRU/last-success mode plus final-only 402 failover coverage is the safer maintainable direction.
