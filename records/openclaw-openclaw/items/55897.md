## Summary

Keep open. Current `main` and the latest release still intentionally document and test oldest-first round-robin auth-profile ordering, while 402 failover has related classifier and surfacing fixes but not a focused proof that the reported multi-key intermediate 402 rendering is fully closed.

## What This Changes

Request MRU/last-success ordering for multiple API-key auth profiles after session reset, plus silent intermediate 402 failover while another key is retried.

## Best Possible Solution

Maintainers should choose and document a supported auth-profile selection policy, then add focused coverage for a config-backed or reason-aware MRU mode plus final-only 402 failover behavior without accidentally breaking documented round-robin rotation.

## Reproduction Assessment

Partly yes. Source and tests reproduce the ordering half because current main still prefers oldest `lastUsed` over `lastGood`; the exact live multi-key 402 rendering sequence was not reproduced.

## Solution Assessment

Unclear. A default MRU flip would conflict with current docs and tests, so a maintainer-approved config-backed or reason-aware mode plus final-only 402 coverage is the safer path.
