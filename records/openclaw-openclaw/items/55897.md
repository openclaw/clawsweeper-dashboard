## Summary

Keep open. Current main still intentionally resolves automatic auth-profile order as oldest-first round-robin and explicitly does not prioritize `lastGood`, so the requested MRU-sticky behavior after `/new` remains unresolved. The 402 classification and retry/fallback surface is stronger than the reporter's 2026.3.24 environment, but related #48623 and #50217 do not fully supersede this issue because neither changes initial API-key ordering policy on current main.

## What This Changes

The issue requests MRU-sticky API-key profile selection after session reset and silent intermediate 402 billing failover across multiple API keys for the same provider.

## Best Possible Solution

Keep this issue open until maintainers choose a supported auth-profile selection policy. The maintainable endpoint is either an explicit MRU/last-success policy option or a deliberate API-key-only default change, with schema/docs updates and focused coverage for `/new`, cooldown skipping, profile rotation, and final-only 402 surfacing.

## Reproduction Assessment

Yes for the ordering half: current unit coverage and source show a high-confidence reproduction where `lastGood` points to the newer profile but oldest `lastUsed` still wins. The exact provider-specific visible-402 stream is only partially reproduced by current tests, which cover classified 402 fallback but not all intermediate rendering shapes.

## Solution Assessment

No, current main is not the best solution for the reported multi-key quota workflow because it preserves round-robin even after a key has just succeeded. The safer path is a maintainer-approved policy choice, preferably an explicit config option if preserving existing round-robin defaults matters.
