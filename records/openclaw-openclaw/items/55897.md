## Summary

Keep open. Current main and the latest release still preserve oldest-first round-robin auth-profile ordering, so the MRU sticky behavior is not implemented; 402 classification and rotation have improved, but the exact multi-key intermediate chat-rendering path still needs focused proof.

## What This Changes

The issue requests MRU sticky selection for multiple API-key auth profiles after `/new` and silent intermediate 402 handling during auth-profile failover.

## Best Possible Solution

Maintainers should choose a documented auth-profile selection policy that preserves compatibility, then add regression or live proof that 402 failover retries do not render intermediate provider errors.

## Reproduction Assessment

Yes for the ordering half: current source, tests, and docs still show oldest-first `usageStats.lastUsed` ordering rather than MRU/`lastGood`. The exact live 402 chat-rendering sequence has not been reproduced.

## Solution Assessment

Unclear. Flipping the default to MRU would conflict with current docs and tests, so the safer solution is a maintainer-approved policy/config mode plus final-only 402 failover coverage.
