## Summary

Keep open. Current main still implements and documents oldest-first round-robin auth-profile ordering, while the 402 retry/fallback path is improved but not enough to close the reporter's MRU-sticky plus final-only-error request.

## What This Changes

The issue asks OpenClaw to prefer the most recently successful API-key auth profile after session reset and to avoid rendering intermediate 402 errors while profile failover is still trying other keys.

## Best Possible Solution

Choose and document a supported auth-profile selection policy, then implement either an explicit MRU/last-success mode or a deliberate API-key default change with schema/docs/tests and final-only 402 failover coverage.

## Reproduction Assessment

Partly. Source and tests give a high-confidence reproduction for the ordering half because current main still prefers oldest `lastUsed` over `lastGood`; the exact provider-specific visible intermediate 402 stream was not replayed against current main.

## Solution Assessment

No, not as an automatic cleanup fix. The narrow code path is clear, but changing selection from documented round-robin to MRU/last-success is a product/config policy decision before implementation.
