## Summary

Keep open. Current main still intentionally uses non-MRU, oldest-first round-robin auth-profile ordering when no explicit order is configured, and a regression test asserts that `lastGood` does not win over that ordering. The 402 classifier and failover paths have improved since the reported release, but the requested MRU-sticky API-key behavior remains unresolved and the related open/closed items do not fully supersede the combined MRU plus final-only 402 rendering request.

## What This Changes

The issue requests MRU-sticky API-key auth-profile selection after a session reset and silent intermediate 402 failover so only the final exhausted error is rendered.

## Best Possible Solution

Keep the issue open until maintainers choose the supported selection policy: either make API-key profiles MRU/lastGood sticky after success or add an explicit MRU-versus-round-robin configuration option. Add focused coverage for `/new` selection, cooldown skipping, rotation through 402/billing failures, and final-only user-visible error surfacing, then update the model-failover and auth-order docs.

## Reproduction Assessment

Yes for the ordering half: the existing `resolveAuthProfileOrder` regression test reproduces that `lastGood` is ignored in favor of older `lastUsed`. Unclear for the user-visible 402 half without a live or full embedded streaming reproduction that proves intermediate billing errors are rendered before a later profile succeeds.

## Solution Assessment

Unclear. The requested MRU default may be the right user outcome for quota-backed API keys, but it conflicts with the current documented round-robin policy; the safer maintainable path is a maintainer-approved policy/config choice plus tests that separately cover final-only 402 surfacing.
