## Summary

Keep open: current main and v2026.6.8 still intentionally use oldest-first round-robin auth-profile ordering and ignore `lastGood`; the 402 classifier/failover path has improved, but the requested MRU-sticky policy and live silent-402 behavior remain unimplemented product decisions.

## What This Changes

Request MRU-sticky `api_key` auth-profile selection after session reset and silent intermediate 402 failover across multiple keys.

## Best Possible Solution

Make a maintainer decision on auth-profile selection semantics, then either document the existing explicit `auth.order` path or implement a tested MRU/last-success mode with docs, upgrade proof, and live 402 suppression proof.

## Reproduction Assessment

Partly yes: current source and tests show the ordering half by ignoring `lastGood` and using oldest-first `lastUsed`. The live three-key 402 chat-rendering path has not been replayed.

## Solution Assessment

No for an automatic default flip: current code and docs intentionally define round-robin ordering. The safer solution is a maintainer-approved policy or mode, plus focused tests and live proof for the 402 UX path.
