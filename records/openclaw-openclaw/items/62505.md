## Summary

Keep open. Current main still has the narrowed reliability bug from the discussion: background exec completion requests an `exec-event` wake, but heartbeat dispatch is still treated as disabled when the configured interval is `0m`, and the linked repair PR #67913 was closed unmerged.

## What This Changes

The issue reports a regression where OpenAI Codex-backed agents stop completing or reporting background work, later narrowed to lost exec-event delivery when heartbeat cadence is disabled.

## Best Possible Solution

Add a narrow heartbeat dispatch fix so targeted `exec-event` wakes with captured delivery context can run even when periodic cadence is `0m`, with regression tests for scheduler targeting and completion relay.

## Reproduction Assessment

Yes for the narrowed current-main bug: configure `agents.defaults.heartbeat.every` to `0m`, let a long exec yield to background, then finish with output, and the code path requests an `exec-event` wake that the heartbeat runner still treats as disabled. The broader Codex planning/final-answer symptom is not fully reproducible from static issue data alone.

## Solution Assessment

Yes, the best current fix is a small heartbeat exec-event bypass rather than reintroducing a separate codex-cli backend or changing phase handling broadly. The implementation should keep global heartbeat disable semantics intact.

## Repair Work Prompt

Fix the disabled-heartbeat exec-event delivery bug for this issue. Observable bug: when a background exec finishes after the foreground yield and `agents.defaults.heartbeat.every` or the target agent heartbeat interval is `"0m"`, `notifySessionExit`/`emitExecSystemEvent` enqueue an `exec-event` wake but the heartbeat runner skips dispatch as disabled, so the original chat never receives completion output. Expected boundary: allow targeted `exec-event` wakes with session/delivery context to run as one-shot dispatches even when periodic heartbeat cadence is disabled, while preserving the global heartbeat enable switch and normal disabled periodic heartbeat behavior. Use PR #67913 only as source context; do not reintroduce codex-cli, do not broaden OpenAI/Codex phase handling, and do not change unrelated heartbeat policy. Add regression tests for scheduler session-scoped exec wakes with `every: "0m"` and for completion relay with captured delivery context. Add a single-line `CHANGELOG.md` fix entry because this is user-facing reliability behavior.
