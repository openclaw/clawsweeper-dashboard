## Summary

Keep open. Current main still drops the narrowed background-exec completion path when `heartbeat.every` is `0m`: exec completion requests an `exec-event` wake, but the direct runner and scheduler both require a non-null heartbeat interval, and related PR #67913 is closed unmerged.

## What This Changes

Restore Codex/coding-agent background task completion so long-running work produces final progress or output instead of status-only planning updates.

## Best Possible Solution

Allow targeted `exec-event` wakes with captured delivery context to run once even when periodic heartbeat cadence is disabled, while preserving global heartbeat disable semantics and leaving broader phase/planning changes to separate repros.

## Reproduction Assessment

Yes for the narrowed bug: configure `agents.defaults.heartbeat.every` to `0m`, let a long exec background and finish with output, and the source path requests an `exec-event` wake that the runner still treats as disabled. The broader Codex planning-only symptom is not fully reproducible from static issue data alone.

## Solution Assessment

Yes, a narrow heartbeat `exec-event` dispatch fix is the best current path; reintroducing a separate codex-cli backend or broadly changing assistant phase handling would be less targeted without a separate reproduction.
