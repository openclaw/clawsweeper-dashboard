## Summary

Keep open: current main and the latest release still have the source-level disabled-heartbeat exec completion gap, and the linked fix candidate at https://github.com/openclaw/openclaw/pull/79869 is open, conflicting, and previously flagged for delivery-policy concerns.

## What This Changes

The issue reports a regression where Codex-backed coding agents stop after vague progress updates; the narrowed source path is background exec completion not being delivered when `heartbeat.every` is `0m`.

## Best Possible Solution

Review, refresh, or replace https://github.com/openclaw/openclaw/pull/79869 with a narrow fix that delivers targeted exec completion events when periodic heartbeats are disabled while preserving explicit delivery suppression, cooldown policy, and redacted real behavior proof.

## Reproduction Assessment

Yes at source level: with `heartbeat.every: "0m"`, current main resolves the interval to null and skips heartbeat execution while background exec completion still requests an `exec-event` wake. I did not run a live gateway/Codex scenario in this read-only review.

## Solution Assessment

Unclear for the open fix candidate: the heartbeat runner boundary is plausible, but prior review and the earlier closed attempt show maintainers still need to settle whether exec completions should bypass disabled heartbeat semantics or move to a separate system-event delivery path.
