Keep open: the PR addresses a real Microsoft Teams thread-isolation class, but it is not merge-ready because the branch is dirty, lacks redacted after-fix Teams proof, and still has blocking correctness issues around effective thread-binding policy and stable thread-root identity. Current main already contains some overlapping Teams thread routing/idempotency fixes, while the PR still has potentially useful remaining mention, media, history, and debounce work that needs a focused rebase or replacement.

## What This Changes

The branch modifies the Microsoft Teams plugin to isolate channel-thread sessions/history/debounce, target outbound thread replies, preserve non-bot mentions, adjust Graph media fallback, add Teams tests, and includes unrelated formatting churn across other plugins/core files.

## Best Possible Solution

Rebase or replace this with a narrow Microsoft Teams-only patch that preserves current main's thread-root/idempotent routing, adds effective-policy-aware thread-scoped history/debounce, retains any proven mention/media fixes, and includes focused tests plus redacted real Teams proof.

## Reproduction Assessment

Yes, source-level: current main still computes pending history and inbound debounce from the normalized Teams channel id, so same-sender messages or pending history can still cross threads. I did not run a live Teams tenant in this read-only review.

## Solution Assessment

No. The direction is useful, but the current PR is not the best fix because it bypasses effective thread-binding policy, uses replyToId instead of the Teams thread root, and needs a rebase around current main's shipped thread-routing behavior.
