## Summary

This PR should stay open: current main has landed the basic Teams thread-session and proactive-thread fixes, but the branch still carries unique debounce/history/mention/attachment work and has blocking correctness/proof gaps before merge.

## What This Changes

The branch changes the Microsoft Teams plugin to isolate channel-thread sessions/history/debounce, target outbound thread replies, preserve non-bot mentions, broaden attachment fallback, and add related tests, with unrelated formatting churn across other plugins and core files.

## Best Possible Solution

Rebase or replace this with a narrow MS Teams-only change on current main that preserves shipped thread-root, proactive routing, and idempotent key behavior while adding effective-policy-aware thread-scoped history/debounce and any retained mention/attachment fixes with focused tests and real Teams proof.

## Reproduction Assessment

Yes. The linked bug gives a concrete two-thread Teams reproduction, and current main still has a source-level reproduction for pending-history and debounce mixing because both keys omit the Teams thread root; I did not run a live Teams tenant in this review.

## Solution Assessment

No. The direction is useful, but this branch is stale and its global-only gate plus `activity.replyToId`-derived thread identity do not match current main's effective policy and thread-root behavior.

## Review Findings

Full review comments:

- **[P2] Honor the effective thread-binding policy:** `extensions/msteams/src/monitor-handler/message-handler.ts:391-392`
  - body: The new gate only checks `cfg.session?.threadBindings?.enabled === true`, but OpenClaw's shared policy supports channel/account overrides and treats thread bindings as enabled by default. A default Teams install, or one using `channels.msteams.threadBindings.enabled`, would still use the unthreaded session/history/debounce path and keep the cross-thread bleed this PR is meant to fix.
  - confidence: 0.87
- **[P2] Derive the thread key from the Teams thread root:** `extensions/msteams/src/monitor-handler/message-handler.ts:392-393`
  - body: This scopes the new session/history/debounce keys to `activity.replyToId`, but the stable Teams channel-thread root is the `;messageid=` value already parsed as `conversationMessageId`. For deep replies, `replyToId` can point at a nested parent, so one visible Teams thread can split across multiple OpenClaw session keys instead of staying isolated as one lane.
  - confidence: 0.78
