## Summary

Keep this PR open. Current main and v2026.4.29 already cover several central MS Teams thread fixes, including per-thread session suffixes, stored thread roots, proactive thread conversation IDs, feedback session routing, and safer Graph media fallback, but the PR is not fully obsolete: current main still keys inbound debounce and pending group history only by normalized conversation ID, so same-sender activity across two Teams threads can still mix before dispatch. The proposed patch also needs adjustment before merge because it checks only the global thread binding switch and misses the channel/account-scoped thread binding contract.

## What This Changes

The PR changes the Microsoft Teams plugin to add thread-aware session/history/debounce routing, preserve non-bot mentions in agent text, target outbound thread replies, adjust attachment fallback behavior, and add related tests with incidental formatting churn across other plugins.

## Best Possible Solution

Rebase or replace this PR with a narrow MS Teams-only change on current main: preserve the shipped threadId/proactive ;messageid= routing, add thread-aware debounce and pending-history keys, decide whether non-bot mention preservation is still desired, honor the effective thread binding policy, and cover #66771-style double-thread-key prevention if the same path is touched.

## Reproduction Assessment

Yes. #58615 gives a concrete two-thread Teams reproduction, and current main still has a high-confidence code-level reproduction for one remaining class: two quick messages from the same sender in different channel threads share the same debounce key and pending history key because both omit thread identity.

## Solution Assessment

No. The direction is useful, but this branch is stale relative to current main and the patch's global-only threadBindings check is not the narrowest maintainable fix because OpenClaw's effective thread binding policy supports channel and account overrides.
