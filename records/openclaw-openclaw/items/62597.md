## Summary

Keep this PR open. Current main still performs the uncached SQLite indexed-content preflight on memory search, so the PR remains relevant; the patch direction looks narrow and functionally reasonable, but it needs the required changelog entry and would benefit from focused cache invalidation coverage before merge.

## What This Changes

The PR adds an in-memory cached boolean for `MemoryIndexManager.hasIndexedContent()`, populates it from the SQLite preflight or chunk writes, and invalidates it on chunk deletes, index resets, and safe-reindex database swaps.

## Best Possible Solution

Land a small memory-core-only change that caches `hasIndexedContent()` within the manager lifecycle, invalidates the cache on all chunk/table mutations and DB-handle swaps, includes the required changelog entry, and then closes the linked performance issue after merge.

## Reproduction Assessment

Yes. Current main provides a high-confidence source reproduction: repeated `MemoryIndexManager.search()` calls enter `hasIndexedContent()`, which prepares and runs the chunks/FTS existence queries each time before search preflight.

## Solution Assessment

Yes, with minor follow-up. An in-memory manager cache invalidated by memory-core's owned write/delete/reindex paths is the narrowest maintainable fix; adding changelog and focused tests is safer than moving this into core or changing search semantics.
