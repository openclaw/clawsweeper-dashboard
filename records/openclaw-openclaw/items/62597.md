## Summary

Keep open. Current main still performs direct SQLite indexed-content probes from the memory search path, and this PR still contains useful, focused cache work for the linked performance issue, but it needs real behavior proof and a current-main refresh before merge.

## What This Changes

Adds an in-memory memory-core `hasIndexedContent()` cache and invalidates it on chunk writes, deletes, index reset, and safe-reindex DB swaps.

## Best Possible Solution

Refresh the branch on current memory-core, keep the manager-local cache, invalidate it on every chunk/table mutation and DB handle replacement, add focused regression coverage, and attach redacted runtime proof showing repeated searches avoid redundant probes.

## Reproduction Assessment

Yes. Source inspection on current main shows each `search()` reaches `hasIndexedContent()`, which prepares direct `chunks` and FTS SQLite existence probes; a focused regression can count prepare/query calls across repeated searches.

## Solution Assessment

Yes, with one rebase adjustment. A manager-local cache is the narrow maintainable solution, but it must be invalidated on all mutation and DB replacement paths, including readonly recovery.
