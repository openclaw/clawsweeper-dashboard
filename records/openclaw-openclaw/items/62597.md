## Summary

Keep open. Current main still performs the repeated SQLite indexed-content preflight, and this PR is a narrow plausible fix, but it still needs the required changelog entry and exact-head validation before merge.

## What This Changes

The PR adds an in-memory cache for `MemoryIndexManager.hasIndexedContent()`, sets it after chunk writes, and invalidates it on chunk deletes, index resets, and DB-handle swaps.

## Best Possible Solution

Land a memory-core-only cache with invalidation on all chunk/table mutations and DB-handle swaps, add the required changelog/test coverage, then close #62513 after merge.

## Reproduction Assessment

Yes. Current main provides a high-confidence source reproduction: repeated `MemoryIndexManager.search()` calls enter `hasIndexedContent()`, which prepares and runs chunks/FTS existence queries before deciding whether to search.

## Solution Assessment

Yes, with minor follow-up. A manager-lifetime in-memory cache invalidated by memory-core's owned write/delete/reindex paths is the narrowest maintainable fix; adding changelog and focused tests is safer than changing search semantics.
