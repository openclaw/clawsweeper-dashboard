## Summary

Keep open. Current main and the latest release still perform direct SQLite indexed-content probes from memory search, and this PR is the active implementation path for the linked issue, but it still needs real behavior proof and a refresh against current memory-core code before merge.

## What This Changes

The PR adds a nullable memory-core cache for `hasIndexedContent()`, populates it from SQLite probes, sets it true after chunk writes, and invalidates it on deletes, reset, and safe-reindex DB swaps.

## Best Possible Solution

Refresh the PR on current main, clear the cache on every chunk/table mutation and DB handle replacement, add focused regression coverage, and provide redacted runtime proof that repeated searches avoid redundant probes.

## Reproduction Assessment

Yes. Source inspection on current main shows each `search()` reaches `hasIndexedContent()`, which prepares direct SQLite existence probes; a focused regression can count prepare/query calls across repeated searches.

## Solution Assessment

Yes, with a rebase adjustment. A manager-local indexed-content cache is the narrow maintainable solution if it is invalidated on writes, deletes, resets, safe-reindex DB swaps, and readonly-recovery DB replacement.
