## Summary

Keep open. The cache change is still useful and not present on current main, but this PR is not merge-ready because it lacks after-fix real behavior proof, is conflicting with main, and needs its cache lifecycle refreshed for the current readonly-recovery DB reopen path.

## What This Changes

This PR adds an in-memory `hasIndexedContent()` cache in memory-core, sets it after chunk writes, and clears it on deletes, index reset, and safe-reindex DB swaps.

## Best Possible Solution

Refresh or replace the branch with a manager-local cache invalidated on every indexed-table mutation and active DB handle replacement, backed by focused cache tests and redacted repeated-search proof.

## Reproduction Assessment

Yes. Source inspection on current main shows repeated `search()` calls reach `hasIndexedContent()`, which prepares direct SQLite existence probes; I did not run a runtime benchmark because this review is read-only.

## Solution Assessment

Yes, with a refresh. A manager-local cache is the narrow maintainable solution, but it needs invalidation across writes, deletes, resets, safe reindex, and readonly-recovery DB replacement before merge.
