## Summary

Keep open: current main still has the repeated SQLite preflight probes and this PR is a focused implementation lane, but the branch is now conflicting, lacks after-fix real behavior proof, and should account for the newer readonly-recovery DB reopen path before merge.

## What This Changes

This PR adds an in-memory cache for `MemoryIndexManager.hasIndexedContent()`, sets it after chunk writes, and clears it around chunk deletes, reset, and safe-reindex DB swaps.

## Best Possible Solution

Refresh the branch on current main, keep the cache local to memory-core, invalidate it on every chunk/table mutation and DB handle replacement, add focused regression coverage, and provide redacted runtime proof that repeated searches avoid redundant probes.

## Reproduction Assessment

Yes, source-reproducible. Current main reaches `hasIndexedContent()` from `search()`, and that method prepares direct SQLite existence probes on each call; I did not run a live benchmark because this review is read-only.

## Solution Assessment

Yes, with a rebase adjustment. A manager-lifetime cache invalidated by memory-core's write/delete/reset/DB-swap paths is the narrow maintainable fix, but the current readonly-recovery DB reopen path should be included before merge.
