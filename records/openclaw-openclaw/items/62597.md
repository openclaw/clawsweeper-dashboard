## Summary

Keep open. The PR targets a real repeated SQLite probe still present on current main, but it needs a branch refresh, contributor real behavior proof, and a small cache invalidation adjustment before it is merge-ready.

## What This Changes

The PR adds an in-memory `hasIndexedContent()` result cache in memory-core and invalidates it on chunk writes, deletes, reset, and safe-reindex DB swaps.

## Best Possible Solution

Refresh the branch on current memory-core, keep the manager-local cache, invalidate it on every chunk/table mutation and DB handle replacement, add focused regression coverage, and attach redacted runtime proof showing repeated searches avoid redundant probes.

## Reproduction Assessment

Yes. Source inspection on current main shows each `search()` reaches `hasIndexedContent()`, which prepares direct SQLite existence probes; a focused regression can count prepare/query calls across repeated searches.

## Solution Assessment

Yes, with a rebase adjustment. A manager-local cache is the narrow maintainable solution, but it must be invalidated on all mutation and DB replacement paths, including readonly recovery.
