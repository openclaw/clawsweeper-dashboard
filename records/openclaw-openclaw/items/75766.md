## Summary

Keep open. The Docker healthcheck change is aimed at a real operator bug, but the PR is incomplete for the bundled Docker flow: Compose still carries its own shorter healthcheck grace period, the linked report describes cold starts that may exceed 120s, and the user-facing fix is missing a changelog entry.

## What This Changes

The branch changes the root Dockerfile OCI HEALTHCHECK start grace period from 15s to 120s while leaving the probe command and other healthcheck options unchanged.

## Best Possible Solution

Land a narrow Docker operator fix that uses one issue-backed grace period across both the image HEALTHCHECK and bundled Compose healthcheck, keeps `/healthz` and the existing interval/timeout/retry semantics, adds a static guard for the Dockerfile/Compose contract, and records the user-facing fix in the Unreleased changelog.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: current main has a 15s Dockerfile start period and a 20s Compose start period, Docker healthcheck semantics make those values affect when failures count, and the linked bug gives a concrete `docker run` plus `docker ps` unhealthy-state check; no live Docker run was executed in this read-only review.

## Solution Assessment

No. Bumping only the Dockerfile value is the right direction, but this PR is not the best complete fix because the bundled Compose override remains short, the chosen 120s value is not proven against the linked cold-start envelope, and the release notes would miss the operator-visible change.
