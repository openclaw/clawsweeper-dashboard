## Summary

Keep open. The PR is aimed at a valid Docker healthcheck timing bug, but the one-line Dockerfile change is incomplete for the bundled Docker Compose path because `docker-compose.yml` still defines its own short `start_period`, and the user-facing Docker fix is missing the required changelog entry.

## What This Changes

The PR changes the image-level Dockerfile OCI `HEALTHCHECK` start period from 15 seconds to 120 seconds while leaving the `/healthz` probe command and other Dockerfile healthcheck options unchanged.

## Best Possible Solution

Land a narrow Docker/operator fix that aligns the image-level and bundled Compose healthcheck grace periods, keeps `/healthz` as the liveness probe, adds the required changelog entry, and validates the static Dockerfile/Compose contract plus a built-image or Compose smoke when feasible.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: current main has `--start-period=15s` in `Dockerfile`, bundled Compose has `start_period: 20s`, Docker docs say healthcheck failures can mark containers unhealthy, and the linked issue provides a concrete `docker run` plus `docker ps` check; no live Docker run was executed in this read-only review.

## Solution Assessment

No, not as currently scoped. Bumping the image `HEALTHCHECK` is the right direction, but the bundled Compose override and missing changelog make this patch incomplete for the documented Docker flow; a small follow-up should update Compose and release notes without changing gateway probe semantics.
