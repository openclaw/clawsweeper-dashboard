## Summary

Keep open: this is a fresh, specific QA-lab report tied to the open Codex-vs-Pi parity work, and current main does not yet contain the reported runtime-axis token-efficiency gate. The linked PR head does flag absolute token deltas as failures, while the original token-efficiency tracker also described a symmetric absolute-delta policy, so savings-vs-regression semantics and the `runtime-tool-fs-read` overhead need maintainer review rather than cleanup closure.

## What This Changes

Clarify the QA-lab live token-efficiency gate so Codex savings are not treated like regressions, and investigate the high-token `runtime-tool-fs-read` live row.

## Best Possible Solution

Define the live token-efficiency policy in the active parity work, update https://github.com/openclaw/openclaw/pull/80323 and its tests to match that policy, and keep `runtime-tool-fs-read` as a live efficiency triage row until the tool-loop/root-cause evidence is classified.

## Reproduction Assessment

No high-confidence current-main reproduction exists because current main does not include the reported runtime-axis token-efficiency mode. There is a source-level reproduction on the linked PR head for the savings flagging because `Math.abs(delta)` marks negative deltas over threshold as flagged.

## Solution Assessment

Unclear: making only positive Codex-over-Pi deltas fail may be the right policy if the gate is Codex-regression-only, but it conflicts with the open Phase 4 tracker's symmetric absolute-delta wording. The fs.read overhead should be triaged from live artifacts before changing runtime behavior.
