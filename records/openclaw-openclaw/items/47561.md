## Summary

Keep open: current main still derives the active DM session key from the current routing configuration, and `/new` resets only the currently resolved key. The existing dmScope migration coverage handles main-to-isolated delivery metadata, but not the reverse isolated-to-main case described here; related Control UI work explicitly left this channel routing-mode drift open.

## What This Changes

Make `/new` remain a durable fresh session for a channel conversation across routing-mode changes, or explicitly migrate/warn when the fresh session is superseded.

## Best Possible Solution

Add a shared routing/session continuity guard that detects when a recent `/new` key for the same conversation is about to be superseded by a routing-mode change, then either performs a deliberate rebind/migration or surfaces an operator-visible diagnostic.

## Reproduction Assessment

Yes, at source level. Current routing tests and implementation show the same direct Telegram-style peer resolves to a per-channel key under `dmScope="per-channel-peer"` and back to `agent:main:main` under `dmScope="main"`, while `/new` persists only the currently resolved key.

## Solution Assessment

Unclear. The fix likely belongs in shared routing/session state rather than a Telegram-only patch, but maintainers need to choose whether route changes should migrate, quarantine, or warn before a narrow implementation is safe.
