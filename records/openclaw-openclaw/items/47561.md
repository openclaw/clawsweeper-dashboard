## Summary

Keep this open: current main still resolves the same Telegram DM to different session keys when `session.dmScope` changes, and `/new` only rotates the currently resolved key. The merged Control UI fix in #75988 explicitly left this channel routing-drift case open rather than superseding it.

## What This Changes

Ensure `/new` stays authoritative, migrates, or warns when channel routing policy changes so a chat does not silently resume an older session.

## Best Possible Solution

Add a generic session-continuity guard at the routing/session boundary so a recent `/new` for the same conversation cannot be silently superseded by an older row after routing mode changes, with focused channel coverage for Telegram and any shared plugin path.

## Reproduction Assessment

No live high-confidence reproduction was run. Source inspection gives a clear reproduction shape: a Telegram DM can resolve to a per-channel key before the config change and to `agent:main:main` after reverting `dmScope`, where the older row is eligible for reuse.

## Solution Assessment

Unclear. The narrow code location is identifiable, but maintainers should choose whether the durable fix is conversation binding, migration/rebinding, or a warning because each has different cross-channel compatibility behavior.
