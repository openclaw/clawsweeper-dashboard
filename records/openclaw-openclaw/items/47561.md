## Summary

Keep open. Current main still selects Telegram DM session keys from the active `session.dmScope`, so reverting from isolated direct-message routing back to the unified main route can strand the fresh `/new` session and resume the older main session. The existing cleanup code only handles the opposite direction, and the related Feishu/DingTalk comments are useful adjacent signals rather than proof of a single already-shipped fix.

## What This Changes

The issue asks OpenClaw to keep a `/new` session authoritative, migrate/rebind it, or warn when later DM routing-mode changes would resume an older session.

## Best Possible Solution

Persist enough reset/conversation metadata for direct-message `/new` and `/reset` so later dmScope changes can either rebind the fresh session to the new canonical key or emit an explicit diagnostic/warning, with regression coverage for both main-to-isolated cleanup and isolated-to-main supersession.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: route the same direct Telegram peer with `dmScope: per-channel-peer` to get `agent:main:telegram:direct:<peer>`, then change `dmScope` to `main` and the same route resolves to `agent:main:main`; `initSessionState` then loads the entry for that new key with no fresher-session lookup.

## Solution Assessment

No current solution is present on main. The narrowest maintainable repair is a generic session/routing-layer guard plus channel tests, avoiding a Telegram-only special case unless the generic metadata proves insufficient.

## Repair Work Prompt

Current main can create a fresh direct-message session under `session.dmScope: per-channel-peer`, then after config returns to `dmScope: main` route the same conversation to the older `agent:<agentId>:main` entry. Implement a conservative generic session/routing-layer repair: persist or derive enough reset/conversation metadata so a recent direct-message `/new` or `/reset` is not silently superseded by a dmScope lane change. Do not change default `dmScope: main` sharing for unrelated DMs. Add regression coverage for per-channel-peer -> main after `/new`, keep existing main -> isolated delivery-route cleanup tests green, and include Telegram coverage plus any generic session-key tests needed.
