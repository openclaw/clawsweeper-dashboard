## Summary

Keep this open: current main still derives the active DM session key directly from the current `session.dmScope`, and the existing migration guard only handles stale main-route metadata when moving away from unified mode, not the reported scoped-session-to-unified rollback.

## What This Changes

Ensure `/new` remains an authoritative fresh start, or is explicitly migrated or warned about, when a channel conversation's session-routing mode changes.

## Best Possible Solution

Define a generic route-drift policy for explicit reset events: preserve or migrate the most recent reset for the same conversation when safe, and otherwise emit a clear diagnostic before resuming an older routed session.

## Reproduction Assessment

Yes by source inspection. A Telegram DM resolves to a scoped key under `dmScope: "per-channel-peer"`, but the same route resolves to `agent:main:main` after `dmScope` returns to `main`, with no reverse migration or warning path in current main.

## Solution Assessment

No current patch solves it. The existing guard only clears stale main-route delivery when moving away from unified mode, so the maintainable fix should first settle the desired cross-routing reset semantics.
