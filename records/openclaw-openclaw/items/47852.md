## Summary

Keep open. Current main partially supports the use case through flat per-agent `agents.list[].params` for `temperature` and `maxTokens`, but it still does not accept the requested `generation` block or generically forward `topP`/`top_p`, `topK`/`top_k`, presence penalties, or frequency penalties through the main agent stream path.

## What This Changes

Request per-agent configurable generation parameters in `openclaw.json`, including a `generation` block for temperature, maxTokens, topP/topK, and repetition penalties.

## Best Possible Solution

Keep this open until maintainers choose the public config contract. The narrow implementation should either extend the existing documented `agents.list[].params` surface or add a documented `agents.list[].generation` alias, then normalize camelCase and snake_case sampling keys and forward only fields supported by the selected provider/transport, coordinated with #32496.

## Reproduction Assessment

Yes. This is a feature/config gap rather than a failing runtime bug: a static check shows `agents.list[].generation` is absent from the strict schema, while `agents.list[].params` is accepted but the generic stream wrapper only forwards a narrow parameter set.

## Solution Assessment

Unclear. The requested `generation` block is reasonable, but current main already has a public `params` surface, so the best fix depends on maintainer choice between extending that contract and introducing a new alias.
