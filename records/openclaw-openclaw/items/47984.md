## Summary

Keep open. Current main still lacks Discord channelEdit appliedTags support, but this PR is not merge-ready because explicit empty arrays are dropped before the REST layer, the branch predates the current shared parser shape, and the external PR lacks real behavior proof.

## What This Changes

The PR adds Discord channelEdit appliedTags plumbing through types, dispatchers, REST body mapping, and a dispatcher test so agents can update forum/media thread tags.

## Best Possible Solution

Implement this on the current Discord plugin shared channel-edit path: add appliedTags to the send type, parse camelCase and preferably snake_case while preserving explicit empty arrays, map it to applied_tags, and cover non-empty, empty, and dispatch/send paths.

## Reproduction Assessment

Yes, source-level. Current main has no channelEdit appliedTags path, and the PR's empty-array clearing path is provably swallowed by readStringArrayParam before the REST body is built; I did not run a live Discord repro.

## Solution Assessment

No. The direction is right, but the patch should be adapted to current main's shared parser and use a parser that preserves appliedTags: [] instead of routing through readStringArrayParam unchanged.
