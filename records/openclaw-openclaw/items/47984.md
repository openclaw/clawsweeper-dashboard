## Best Possible Solution

Refit the feature onto `readDiscordChannelEditParams` and `editChannelDiscord`, preserve explicit `appliedTags: []`, add focused non-empty and empty-array coverage, and attach redacted real Discord proof.

Yes, source-level. Current main lacks `appliedTags` in the shared Discord channel-edit parser and REST PATCH body, and the PR branch’s use of `readStringArrayParam` makes explicit `appliedTags: []` reproducibly collapse to `undefined`.

No. The API direction is right, but the current branch targets stale dispatch paths and should preserve field presence in the shared parser before mapping `applied_tags` in the REST edit path.
