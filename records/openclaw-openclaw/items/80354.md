## Summary

Keep open. The narrowed PR targets a real embedded-runner timeout path, but the new bypass is still unreachable when an explicit run or agent timeout above 120s is present, and the supplied proof only exercises the helper rather than a live embedded/Gemini turn.

## What This Changes

The PR adds a `thinkingEnabled` input to embedded LLM idle-timeout resolution, passes non-off `thinkLevel` from `runEmbeddedAttempt`, and adds resolver tests for the new bypass.

## Best Possible Solution

Keep the embedded timeout fix narrow, make the thinking bypass override only the default idle watchdog while preserving explicit hard caps, and require redacted live embedded/Gemini proof before merge.

## Reproduction Assessment

Yes, source-level reproduction is high confidence from current main plus the linked report: cloud runs fall back to the 120s idle watchdog without considering non-off thinking, and the linked logs show repeated roughly 125s Gemini failures. I did not run a live Gemini reproduction in this read-only review.

## Solution Assessment

No as submitted. The direction is plausible, but the bypass should not remain after the branches that clamp explicit longer run or agent timeouts back to the default idle watchdog.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Honor thinking before default timeout caps:** `src/agents/pi-embedded-runner/run/llm-idle-timeout.ts:187-193`
  - body: The new `thinkingEnabled` check runs after `runTimeoutMs` and `agents.defaults.timeoutSeconds` have already returned through `clampImplicitTimeoutMs`, so `thinkingEnabled: true` with an explicit 300s run/agent timeout still resolves to the 120s default before reaching this bypass. That leaves slow thinking runs vulnerable to the same false idle timeout even when the operator sets a longer hard cap. Move the thinking-specific bypass ahead of the implicit default clamp, or add a thinking-specific path that preserves explicit bounds without reapplying the default watchdog.
  - confidence: 0.84
