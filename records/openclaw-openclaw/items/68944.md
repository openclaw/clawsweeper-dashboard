## Summary

Keep open: current main still has a credible post-challenge silent-failure path in the Gateway client, and the discussion already confirms the Google Chat cross-reference was unrelated.

## What This Changes

The issue reports CLI subcommands hanging after the Gateway sends `connect.challenge` because the CLI never emits the signed `connect` request.

## Best Possible Solution

Add a narrow Gateway client fix that separates JSON parse failures from event handling, reports and closes/reset-settles failures while assembling or sending the signed connect request, and keeps Ed25519 device-token auth unchanged.

## Reproduction Assessment

Yes. The issue includes concrete WebSocket traces, and current main has a deterministic unit path: inject a throw during `sendConnect()` after `connect.challenge` and verify no `connect` frame is sent while the broad handler catch hides the error.

## Solution Assessment

Yes. Client-side error propagation and prompt CLI settlement around `sendConnect()` is the narrow maintainable fix; gateway idle timeouts or HMAC fallback would not address a client that fails before emitting the signed connect request.
