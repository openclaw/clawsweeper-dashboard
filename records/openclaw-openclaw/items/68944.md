## Summary

Keep open. Current main still has a source-visible silent failure path in the Gateway WebSocket challenge response, the issue is recent with corroborating user reports, and the only linked Google Chat PR is confirmed unrelated.

## What This Changes

The issue reports that gateway-backed CLI commands hang after receiving `connect.challenge` because no authenticated connect request is emitted.

## Best Possible Solution

Surface challenge-reply construction failures through `onConnectError` with clear logging and socket close behavior, keep JSON parse errors separate, and preserve the existing device-auth nonce/signature contract without adding an HMAC fallback.

## Reproduction Assessment

No for the reporter's exact current-main environment: I did not establish a live Windows or Linux hang reproduction. Source inspection does provide a concrete repro path for the silent failure class by making device signature or public-key conversion throw during `connect.challenge` handling and observing that current code logs only a debug parse error before sending no connect request.

## Solution Assessment

Yes for the narrow first repair: route device connect payload/signing/public-key errors to the normal connect-error path and add regression coverage. A broader Gateway CPU-spin or auth-policy change should stay separate unless fresh logs prove that is the actual remaining root cause.
