## What This Changes

The issue reports gateway-backed CLI commands hanging after receiving `connect.challenge` but before emitting the signed connect request.

## Best Possible Solution

GatewayClient should propagate and settle post-challenge connect assembly failures, narrow JSON parse handling away from event dispatch, and keep the existing Ed25519 challenge protocol unchanged.

Yes for a high-confidence source-level reproduction path: force a synchronous error inside `sendConnect()` after `connect.challenge` and before `request("connect", params)`, and current main can skip emitting the connect frame while logging only a debug parse error. ## Solution Assessment

Yes. The narrow maintainable fix is to surface and settle `sendConnect()` assembly failures and narrow the parse catch; a Gateway idle timeout or HMAC fallback would not fix a client that fails before emitting the signed connect request.
