## Summary

Keep open: current main still has a source-provable post-challenge failure path where `sendConnect()` can throw before emitting the signed `connect` request, and that exception is still swallowed by `handleMessage()` as a debug parse error. The accidental Google Chat link was corrected in discussion, and the adjacent pre-hello-close PR does not cover this connect-assembly/error-propagation gap.

## What This Changes

The issue reports CLI gateway commands hanging after receiving `connect.challenge` and before sending the signed Gateway connect request.

## Best Possible Solution

Catch and report Gateway client connect-assembly failures, narrow JSON parse handling so non-parse errors are not swallowed, and make CLI gateway calls settle on connect errors while preserving the existing Ed25519 device-auth protocol.

## Reproduction Assessment

Yes, by source inspection: forcing a synchronous failure inside `sendConnect()` after `connect.challenge` exercises a current-main path where no connect frame is emitted and the exception is swallowed as a debug parse error. I did not reproduce the reporter's exact Windows/Node runtime.

## Solution Assessment

Yes. The narrow maintainable fix is client-side error propagation and request/socket settlement around `sendConnect()`; a gateway idle timeout or HMAC fallback would not fix a client that fails before emitting the signed connect request.
