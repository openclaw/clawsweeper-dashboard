## What This Changes

The issue reports Gateway-backed CLI commands hanging after receiving `connect.challenge` because the client never emits the signed `connect` request.

## Best Possible Solution

Add a focused Gateway client repair that separates JSON parse errors from event handling, reports failures while assembling or sending the signed connect request, closes or resets the socket so CLI callers settle promptly, and preserves the existing device-token security model.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction shape: make post-challenge connect-frame assembly throw after `connect.challenge` and current main can mark the connect as sent without emitting the `connect` request while the broad handler catch logs only a debug parse error. I did not run a live Windows/Node 24 reproduction in this read-only review.

## Solution Assessment

Yes. The narrowest maintainable fix is client-side error propagation and socket settlement around `sendConnect()` plus regression coverage; gateway idle timeouts or HMAC fallback would not address a client that fails before emitting the signed connect request.
