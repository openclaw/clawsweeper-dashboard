## Summary

Keep open: current main and the latest shipped tag still allow a post-challenge `sendConnect()` assembly failure to be swallowed by the broad message handler catch after the challenge timer is cleared, so gateway-backed CLI commands can wait for the outer timeout without a useful connect-auth error. The mistaken Google Chat cross-reference and older closed handshake-timeout/package-walk reports cover different intervals.

## What This Changes

The issue reports that gateway-backed CLI commands hang after receiving a WebSocket `connect.challenge` because the client never emits the signed connect request.

## Best Possible Solution

GatewayClient should propagate and settle post-challenge connect assembly failures, close or reject promptly, narrow `handleMessage()` parse-error handling, and keep the existing Ed25519 nonce protocol unchanged.

## Reproduction Assessment

Yes for a high-confidence source-level reproduction path: force a synchronous error inside `sendConnect()` after `connect.challenge` and before `request("connect", params)`, and current main can skip emitting the connect frame while logging only a debug parse error. I did not reproduce the reporter’s exact Windows runtime.

## Solution Assessment

Yes. The narrow maintainable fix is to propagate and settle `sendConnect()` assembly failures and narrow the parse catch; a Gateway idle timeout or HMAC fallback would not fix a client that fails before emitting the signed connect request.
