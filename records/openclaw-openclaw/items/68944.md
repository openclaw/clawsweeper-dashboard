## Summary

Keep open: current main and the latest release still have the source-proven post-challenge failure shape where a thrown `connect.challenge` handler can prevent the signed connect request from being sent while the CLI waits for a timeout. The latest discussion says a PR is planned, but no merged or open fixing PR is linked yet.

## What This Changes

The issue reports gateway-backed CLI commands hanging after receiving `connect.challenge` because the client never emits the signed connect request.

## Best Possible Solution

GatewayClient should separate JSON parsing from challenge dispatch, surface post-challenge connect assembly failures through `onConnectError` or equivalent CLI-visible failure handling, and preserve the existing signed-device protocol.

## Reproduction Assessment

Yes for a source-level reproduction: force a synchronous error in `sendConnect()` after `connect.challenge` and before `request("connect", params)`, and current main can drop the connect frame under the broad `handleMessage()` catch. I did not reproduce the original Windows scheduled-task runtime.

## Solution Assessment

Yes. The narrow maintainable fix is to propagate and settle `sendConnect()` assembly failures and narrow the message parse catch; a Gateway idle timeout or HMAC fallback would not fix a client that fails before emitting the signed connect request.
