## Summary

Keep this open. The discussion already rules out the accidental Google Chat PR cross-reference, and current main still has the post-challenge silent-failure shape: connect-frame assembly can throw before any `connect` request is sent, while the challenge handler catches that as a debug-only parse error and leaves the CLI wrapper waiting for hello, close, or its generic timeout.

## What This Changes

The issue requests a fix for gateway-backed CLI commands hanging after the WebSocket gateway sends `connect.challenge` but before the CLI emits the signed `connect` frame.

## Best Possible Solution

Implement a focused Gateway client repair: isolate JSON parsing from event handling, catch and report failures while assembling/sending the post-challenge connect frame, close or reset the socket cleanly, and add regression coverage proving a thrown device-signing or public-key-normalization failure after `connect.challenge` does not leave the CLI waiting silently.

## Reproduction Assessment

Yes. The issue provides a concrete handshake trace, and current main has a deterministic unit-level reproduction path by forcing connect-frame assembly to throw after a valid `connect.challenge` and observing that no connect frame is sent under the broad handler catch.

## Solution Assessment

Yes, a narrow client-side error propagation fix is the best direction. Gateway-side idle timeout tuning and HMAC fallback would not address the observed failure before the CLI emits the signed connect request.

## Repair Work Prompt

Fix openclaw/openclaw#68944. The observable bug is that gateway-backed CLI commands can hang after receiving `connect.challenge` when the post-challenge client path throws before sending the signed `connect` request. Keep the fix in the generic Gateway client/CLI transport boundary. In `src/gateway/client.ts`, do not let synchronous failures from auth selection, device payload construction, `signDevicePayload()`, public-key normalization, or connect request-frame construction get swallowed by the generic `handleMessage()` parse-error catch. Narrow `handleMessage()` so parse/validation failures remain parse errors, and route connect-frame failures through `onConnectError`, actionable logging, and a clean socket close/reset. Wire `src/gateway/call.ts` to settle promptly on `onConnectError` if needed. Do not change the protocol, add HMAC fallback, touch the unrelated Google Chat PR path, or log tokens/private keys/signature payloads. Add focused regression coverage in `src/gateway/client.test.ts` and, if wrapper wiring changes, `src/gateway/call.test.ts`: simulate a valid `connect.challenge` followed by a thrown device signing or public-key normalization step and assert the CLI/client surfaces a connect error, closes/resets the socket, and does not leave an open socket with no connect frame. Add a user-facing `CHANGELOG.md` fix entry crediting @WaMaSeDu and @mj-deving if the final behavior change is user-facing.
