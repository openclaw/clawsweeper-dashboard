## Summary

Keep open: current main still has a source-provable Gateway client path where a post-challenge connect assembly failure can prevent the `connect` request from being emitted and be swallowed as a debug-only parse error. The discussion also corrected the accidental Google Chat cross-reference, so there is no merged or open related PR that covers this CLI handshake path.

## What This Changes

The issue reports CLI subcommands hanging after receiving a Gateway `connect.challenge` without sending the signed post-challenge `connect` request.

## Best Possible Solution

Implement a focused Gateway client fix that reports and settles post-challenge connect assembly failures, narrows parse-error handling, and preserves the existing Ed25519 device-auth protocol with regression coverage.

## Reproduction Assessment

Yes, for the source-level failure path: force a synchronous error inside `sendConnect()` after `connect.challenge` and current main can skip emitting `connect` while logging only a debug parse error. I did not reproduce the reporter's exact Windows/Node runtime.

## Solution Assessment

Yes. The narrow maintainable fix is client-side error propagation and request/socket settlement around `sendConnect()`, plus regression tests; a Gateway idle timeout or HMAC fallback would not fix a client that fails before emitting the signed connect request.
