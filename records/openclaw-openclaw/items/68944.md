## What This Changes

Fix gateway-backed CLI commands that stall after receiving `connect.challenge` by surfacing post-challenge client/signing failures and closing or rejecting cleanly.

## Best Possible Solution

Add a narrow Gateway client repair that separates JSON parse errors from event handling, catches and reports failures while assembling or sending the signed connect request, closes or resets the socket so CLI callers settle promptly, and preserves the existing device-token security model.

## Reproduction Assessment

Yes. The issue has a concrete WebSocket trace, and current main has a deterministic unit-level path: make post-challenge connect-frame assembly throw after `connect.challenge` and verify no `connect` frame is sent while the broad handler catch hides the error.

## Solution Assessment

Yes. The best fix is client-side error propagation and settlement around `sendConnect()`; gateway-side idle timeouts or HMAC fallback would not address a client that fails before emitting the signed `connect` request.

## Repair Work Prompt

Create a focused fix PR for https://github.com/openclaw/openclaw/issues/68944. The observable bug is that gateway-backed CLI commands can receive `connect.challenge` but fail before emitting the signed `connect` reply, leaving the command waiting for hello-ok/timeout with no useful error. Keep the fix inside the Gateway client/CLI RPC boundary. In `src/gateway/client.ts`, isolate JSON.parse failures from event handling, wrap post-challenge connect-frame assembly and send work (`selectConnectAuth`, device payload construction, `signDevicePayload`, `publicKeyRawBase64UrlFromPem`, and `request("connect")`) so synchronous failures call `onConnectError`, log clearly, and close/reset the socket instead of leaving `connectSent` and the cleared challenge timer stuck. In `src/gateway/call.ts`, ensure gateway-backed CLI RPCs reject promptly on connect errors or the resulting socket close. Add regression tests in `src/gateway/client.test.ts` for a thrown signing/public-key-normalization path after `connect.challenge` proving no silent wait, and add `src/gateway/call.test.ts` coverage if the wrapper behavior changes. Do not add HMAC fallback, relax device-auth signing, or turn this into a broad Gateway timeout retune. Add a user-facing `CHANGELOG.md` fix entry crediting the non-bot reporters, likely `Thanks @WaMaSeDu and @mj-deving`.
