Current `main` can swallow synchronous failures while assembling the signed post-challenge `connect` request: `GatewayClient.handleMessage()` calls `sendConnect()` inside the broad parse-error catch, `sendConnect()` marks `connectSent` and clears the challenge timer before building/signing the connect params, and `executeGatewayRequestWithScopes()` does not receive `onConnectError`. Implement a focused fix in the gateway client/call path that reports and settles post-challenge connect assembly failures with a clear error, narrows parse-error handling so non-JSON failures are not mislabeled, closes or stops the socket as needed so CLI calls do not hang until the outer timeout, and preserves the existing Ed25519 device-auth protocol. Do not add HMAC fallback, a new config flag, or gateway-side policy changes. Add regression coverage for a forced synchronous connect assembly/signing failure after `connect.challenge`, including the one-shot CLI call wrapper behavior, and keep the existing happy-path challenge/nonce coverage.
