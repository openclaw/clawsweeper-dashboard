## Summary

Keep open: current main still has the post-challenge silent-failure shape in the Gateway client, and the unrelated Google Chat PR cross-reference was already corrected in discussion.

## What This Changes

The issue reports CLI gateway commands hanging after receiving `connect.challenge` without emitting the signed `connect` request.

## Best Possible Solution

Add a focused Gateway client repair that separates JSON parse failures from event handling, reports and closes on connect assembly failures, settles CLI callers promptly, and preserves the existing Ed25519 device-auth model.

## Reproduction Assessment

Yes, by source inspection: make connect-frame assembly throw after `connect.challenge` on current main and `handleMessage()` can swallow it as a debug parse error after `connectSent` is already set. I did not run a live Windows/Node 24 reproduction.

## Solution Assessment

Yes. The narrow maintainable fix is client-side error propagation and socket settlement around `sendConnect()` plus regression coverage; gateway idle timeouts or an HMAC fallback would not fix a client that fails before emitting the signed connect request.
