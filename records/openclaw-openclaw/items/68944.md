## What This Changes

The issue reports gateway-backed CLI commands hanging after `connect.challenge` because the CLI never emits the signed Gateway connect request.

## Best Possible Solution

Add focused Gateway client error propagation and settlement around post-challenge connect assembly, narrow parse-error handling, and keep the existing Ed25519 challenge/nonce protocol unchanged.

## Reproduction Assessment

Yes for a source-level reproduction path: force a synchronous error inside `sendConnect()` after `connect.challenge` and before `request("connect", params)`, and current main can skip emitting the connect frame while logging only a debug parse error. ## Solution Assessment

Yes. The narrow maintainable fix is to propagate and settle `sendConnect()` assembly failures and narrow the parse catch; a Gateway idle timeout or HMAC fallback would not fix a client that fails before emitting the signed connect request.
