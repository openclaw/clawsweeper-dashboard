## Summary

Keep open: current main still has a source-provable post-challenge failure path where `sendConnect()` can fail before emitting the signed `connect` request and `handleMessage()` swallows that failure as a debug parse error. The accidental Google Chat PR link was corrected in discussion, and the later closed package-walk duplicate does not cover this remaining client assembly/error propagation gap.

## What This Changes

The issue reports CLI gateway commands hanging after `connect.challenge` because the client never sends the signed connect reply.

## Best Possible Solution

Implement a focused Gateway client fix that reports and settles connect assembly failures, narrows the parse catch, and preserves the existing Ed25519 device-auth protocol with regression coverage.

## Reproduction Assessment

Yes, by source inspection: forcing a synchronous failure inside `sendConnect()` after `connect.challenge` exercises a current-main path where no `connect` frame is emitted and the exception is swallowed as a debug parse error. I did not reproduce the reporter's exact Windows/Node runtime.

## Solution Assessment

Yes. The narrow maintainable fix is client-side error propagation and socket/request settlement around `sendConnect()`, plus regression tests; a gateway idle timeout or HMAC fallback would not fix a client that fails before emitting the signed connect request.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/68944. The CLI Gateway client can hang after receiving `connect.challenge` if `sendConnect()` throws before `request("connect", params)` sends the signed connect frame; `handleMessage()` currently catches that as a debug parse error. Keep the existing Ed25519 device-auth/protocol contract. In `src/gateway/client.ts`, handle connect assembly/signing/public-key/request-send failures explicitly, route them through `onConnectError`, and close/settle the socket so callers do not wait for the outer timeout. Narrow `handleMessage()` so JSON parse failures are the only parse errors and connect assembly failures are not mislabeled. If needed, wire `src/gateway/call.ts` so CLI Gateway requests observe connect errors. Add focused regression tests for a post-challenge `sendConnect()` failure and for CLI request settlement. Do not add HMAC fallback, a broad tracing feature, gateway-side migration, or plugin changes. Add a user-facing changelog entry because this is a CLI/Gateway bug fix.
