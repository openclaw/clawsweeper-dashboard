## What This Changes

The PR adds a Control UI startup listener that writes `--keyboard-inset-bottom`, uses it in mobile chat composer spacing, and adds unit/browser layout coverage for the inset behavior.

## Best Possible Solution

Keep the VisualViewport CSS-variable approach, but gate it to focused text entry and non-zoomed viewport state, rebase cleanly, and add redacted browser or device proof that the focused composer clears the virtual keyboard.

No high-confidence live current-main reproduction was established. Source inspection shows current main has safe-area-only composer spacing and the closed iOS report gives clear steps, while the PR's false-positive risk is source-reproducible from the VisualViewport calculation.
