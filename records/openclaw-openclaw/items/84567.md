## Label Justifications

- `P1`: The issue describes a deterministic current-release regression that blocks recurring isolated cron workflows for a real Codex OAuth setup.
- `impact:auth-provider`: The reported failure is specific to `openai/gpt-5.5` routed through the Codex/OpenAI OAuth app-server runtime while a non-Codex model works.

## Summary

Current source still supports the reported failure shape: isolated cron starts a 60s pre-run watchdog before `onExecutionStarted`, while Codex app-server setup can still occur before that callback and the default shared-client factory does not pass the app-server request timeout into initialize. The fresh 0.132.0 follow-up makes this a current compatibility/debugging issue rather than a stale duplicate of the older initialize-timeout report.

## What This Changes

The issue requests a fix for Codex-backed isolated cron `agentTurn` jobs timing out before runner start on OpenClaw 2026.5.18 with `openai/gpt-5.5` routed through the bundled Codex OAuth app-server runtime.

## Best Possible Solution

Instrument and fix the Codex app-server setup path used by isolated cron so it either starts the runner normally or fails at a bounded, phase-specific Codex startup/auth/thread step, with regression coverage and live OAuth proof before closing.

## Reproduction Assessment

No high-confidence live reproduction was run in this read-only pass. The issue body gives detailed current-release steps and current source confirms the exact watchdog/diagnostic path can fire before runner start if Codex setup stalls.

## Solution Assessment

Unclear until the Codex setup stall is isolated. The best fix should not only rely on the existing outer watchdog; it should bound and identify the Codex app-server initialize/auth/thread phase and prove the `openai/gpt-5.5` OAuth isolated-cron path in a real environment.
