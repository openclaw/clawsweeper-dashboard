## Summary

Keep open: current main and the latest release still leave `openclaw node run` on a direct WS/WSS connection path, with SSH left to an external tunnel; the remote token-resolution part is now covered, but the first-class node-host SSH lifecycle remains unresolved.

## What This Changes

Request first-class SSH transport support for `openclaw node run` and reliable challenge auth for node pairing through a direct WSS reverse proxy.

## Best Possible Solution

Honor the existing `gateway.remote` SSH settings in the node-host lifecycle, supervise the tunnel for foreground and installed service modes, and keep direct WSS remote challenge auth covered by tests plus real deployment proof.

## Reproduction Assessment

Yes for the remaining SSH transport gap: current main exposes only direct host/port/TLS node-host inputs and builds a direct WS/WSS URL. I did not establish a live nginx reverse-proxy reproduction for the older pairing failure.

## Solution Assessment

Unclear. Reusing the existing SSH helper and `gateway.remote` settings is the likely path, but maintainers should first decide node service lifecycle, reconnect behavior, SSH identity handling, and loopback trust semantics.
