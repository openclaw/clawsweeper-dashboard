## Summary

Keep open: current main and the latest release still do not provide managed SSH transport for `openclaw node run` or `openclaw node install`, while the reverse-proxy challenge-auth path remains security-sensitive and only partially source-confirmed.

## What This Changes

The issue requests first-class SSH transport support for node hosts and a fix for reverse-proxy challenge authentication so remote nodes can create approvable pairing requests.

## Best Possible Solution

Define the remote node-host contract explicitly, then either add operator-visible managed SSH using the existing tunnel helper or keep external tunnels as the supported model and repair only the credential/pairing bug with focused live proof.

## Reproduction Assessment

Partially: source inspection confirms `openclaw node run` still constructs a direct `ws://` or `wss://` URL and does not start the SSH tunnel helper. The reverse-proxy challenge-auth symptom was not live-reproduced in this read-only pass.

## Solution Assessment

Unclear: managed node-host SSH is a plausible solution, but the best fix depends on whether maintainers want node-host-owned SSH lifecycle or an explicitly documented external tunnel model. Any auth change should preserve or intentionally revise the documented local-mode exception.
