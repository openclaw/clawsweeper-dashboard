## Summary

Keep open. Current main still lacks first-class SSH transport for the node host: the foreground CLI, persisted node-host gateway config, runner, docs, and installed-service argument builder all model direct host/port/TLS connections. The remote-token signing path appears better covered now, but the reported reverse-proxy challenge case still needs live confirmation and security-aware design rather than cleanup closure.

## What This Changes

The issue requests SSH transport support for `openclaw node run` and `openclaw node install`, plus reliable non-loopback WSS challenge auth so remote node pairing can be approved behind a reverse proxy.

## Best Possible Solution

Add a reviewed node-host SSH transport path that reuses the existing SSH port-forward helper for foreground and installed-service flows, connects `GatewayClient` only to the local loopback tunnel endpoint, preserves remote token/password and device-pairing semantics, updates node docs, and coordinates the plaintext WebSocket/tunnel security model with #26250.

## Reproduction Assessment

Yes for the SSH-transport gap: current source and docs show node-host foreground and service flows use only direct host/port/TLS settings. Unclear for the reverse-proxy challenge-auth failure on current main because the review verified source/tests but did not run an nginx/podman live reproduction.

## Solution Assessment

No, the current direct-only node-host path is not the best complete solution for the reported workflow. The maintainable fix is node-host-owned SSH tunnel lifecycle with explicit loopback endpoint validation and credential/device-pairing coverage, reviewed with the related security issue.
