## Summary

Keep open: current `main` and the latest release still do not provide first-class SSH transport for `openclaw node run`, and the remaining auth/pairing behavior needs a maintainer product/security decision rather than stale cleanup.

## What This Changes

The issue asks for managed SSH transport support in `openclaw node run` and for remote WebSocket challenge auth to populate the correct gateway credential so node pairing can be approved behind a TLS reverse proxy.

## Best Possible Solution

Define the remote node-host contract explicitly: either add first-class SSH transport using the existing tunnel helper with clear daemon lifecycle and security semantics, or document external tunnels as the supported path, then add a focused credential/pairing regression test for any remaining remote-mode auth failure.

## Reproduction Assessment

Yes for the missing SSH transport: current source and latest release show `openclaw node run` only supports direct host/port/TLS connections. The reverse-proxy challenge-auth failure is source-reproducible only in part, because current remote-mode credential resolution can populate a token, while local-mode node-host behavior still intentionally strips `gateway.remote.*`.

## Solution Assessment

Unclear: reusing `startSshPortForward` is a plausible implementation primitive, but the best fix depends on whether maintainers want node-host-owned SSH lifecycle or an explicitly documented external tunnel model. Any auth fix should preserve the documented local-mode exception unless maintainers change that contract.
