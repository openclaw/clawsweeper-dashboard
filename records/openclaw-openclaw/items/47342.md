## Summary

Keep open: current main still lacks first-class node-host SSH transport, although the shared credential path now covers remote token resolution; the adjacent open tunnel PR adds a separate CLI command and does not settle `node run` integration.

## What This Changes

Request to let `openclaw node run` use managed SSH transport for remote gateways and ensure node-host challenge auth works through non-loopback WSS/reverse-proxy connections.

## Best Possible Solution

Add a maintained node-host SSH transport path that consumes existing `gateway.remote` SSH settings, supervises tunnel lifetime for foreground and installed services, and keeps remote challenge-auth behavior covered by regression tests and docs.

## Reproduction Assessment

Yes for the remaining SSH transport gap: current source shows `node run` accepts only direct host/port/TLS inputs and `runNodeHost` builds a direct WS/WSS URL. I did not reproduce the older reverse-proxy pairing failure live.

## Solution Assessment

Unclear. Reusing `startSshPortForward` is the likely implementation direction, but maintainers should first decide node service lifecycle, reconnect behavior, SSH identity handling, and security semantics.
