## Summary

Keep open: current main and v2026.6.10 still expose `openclaw node run` as a direct host/port/TLS connection, while managed SSH tunneling exists only in gateway-status/probe code and docs still tell node-host users to run SSH manually. The reverse-proxy credential/pairing path has improved on current main, but it still needs a live current-main repro and security/product review before changing node-host transport behavior.

## What This Changes

The issue requests managed SSH transport support for `openclaw node run` plus reliable reverse-proxy challenge auth that creates an approvable node pairing request.

## Best Possible Solution

Decide the node-host transport product/security boundary first, then either keep the explicit user-managed tunnel path documented or add an owner-reviewed managed SSH path with foreground/service lifecycle handling and live reverse-proxy pairing proof.

## Reproduction Assessment

Partly: source and docs show a high-confidence reproduction for the missing managed SSH node-host transport, because `openclaw node run` only supports direct host/port/TLS options. The reverse-proxy challenge-auth failure is not live-reproduced on current main.

## Solution Assessment

Unclear as written: managed SSH is a plausible solution for the transport gap, but the credential/pairing side appears partly improved and the permanent fix needs a maintainer-approved security and lifecycle design.
