## Summary

Keep open: current main still has no first-class SSH transport path for `openclaw node run`, while the reverse-proxy token-resolution part appears partially covered by the shared node-host credential path added after the report.

## What This Changes

Request first-class SSH transport support for node hosts and reliable challenge auth when a node connects to a remote gateway through non-loopback WSS/reverse-proxy paths.

## Best Possible Solution

Add a maintained node-host SSH transport path that consumes existing `gateway.remote` SSH settings, supervises tunnel lifetime for foreground and installed services, and preserves tested remote challenge-auth behavior.

## Reproduction Assessment

Yes for the remaining source-level SSH transport gap: current main exposes only direct host/port/TLS node-host inputs and builds a direct WS/WSS URL. I did not establish a live nginx reverse-proxy reproduction for the older pairing failure.

## Solution Assessment

Unclear. Reusing `startSshPortForward` is the likely implementation direction, but maintainers should first decide node service lifecycle, reconnect behavior, SSH identity handling, and loopback trust semantics.
