## Summary

Keep open: current main and v2026.6.6 still do not provide managed SSH transport for `openclaw node run`/`install`, and the reverse-proxy auth/pairing path still needs maintainer product and security judgment rather than stale cleanup.

## What This Changes

Request to add managed SSH transport to node-host commands and resolve remote WSS challenge-auth/pairing behavior for node hosts behind reverse proxies.

## Best Possible Solution

Define the remote node-host contract explicitly, then either add operator-visible managed SSH using the existing tunnel helper or document external tunnels as the supported path and repair only the credential/pairing bug with focused regression coverage.

## Reproduction Assessment

Yes for the missing SSH transport: current source and v2026.6.6 show node-host commands only support direct host/port/TLS connections. The reverse-proxy challenge-auth path is only partially source-reproducible without a live current-version proxy setup.

## Solution Assessment

Unclear: `startSshPortForward` is a plausible implementation primitive, but the best fix depends on whether maintainers want node-host-owned SSH lifecycle or an explicitly documented external tunnel model. Any auth change should preserve the documented local-mode exception unless maintainers intentionally change that contract.
