## Summary

Keep open: current main still leaves node host on direct WS/WSS host/port wiring and the docs still require an external SSH tunnel; the remote-token and pairing-request parts look improved, but the requested first-class node-host SSH transport remains missing.

## What This Changes

Request to add first-class SSH transport for `openclaw node run` and ensure non-loopback node challenge auth creates an approvable pairing request.

## Best Possible Solution

Add a maintained node-host SSH transport path that reuses the existing tunnel helper/config, supervises tunnel lifetime for foreground and installed services, and keeps non-loopback device-pairing challenge behavior covered by regression tests.

## Reproduction Assessment

Yes for the remaining SSH transport gap from source inspection: `node run` only exposes direct host/port/TLS wiring and `runNodeHost` builds a direct `ws`/`wss` URL. I did not establish a live nginx or SSH reproduction for the reverse-proxy pairing symptom.

## Solution Assessment

Unclear. Reusing `startSshPortForward` is the likely technical direction, but maintainers should decide service lifecycle, reconnect behavior, SSH identity handling, and security semantics before implementation.
