## Summary

Keep this open. Current main still lacks first-class SSH transport for `openclaw node run` and installed node services, while the reverse-proxy pairing/auth portion is only partly addressed by current device-pairing code, docs, and tests and was not live-reproduced for the reported proxy topology.

## What This Changes

The issue requests SSH transport support for `openclaw node run` and a fix for node challenge authentication when connecting to a gateway through non-loopback `wss://`.

## Best Possible Solution

Make an explicit maintainer call on whether node hosts should own managed SSH tunnels; if yes, reuse the existing SSH tunnel helper in node run/install with clear service lifecycle semantics, while keeping device-pairing approval through the current device queue.

## Reproduction Assessment

Partially yes from source: current main shows `openclaw node run` and installed node services still use a direct host/port/TLS connection and have no SSH tunnel setup. No, for the reverse-proxy pairing failure: I did not reproduce that network path, and current device-pairing code/tests point to a pending device request for non-loopback node connects.

## Solution Assessment

Unclear. Reusing the gateway SSH tunnel machinery is plausible, but adding managed SSH to node host foreground and service modes changes node lifecycle and remote-access policy, so it needs maintainer product judgment before a repair PR.
