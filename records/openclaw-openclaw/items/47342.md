## Summary

Keep open: current main has improved the remote-token challenge-auth path, but the node host still lacks first-class SSH transport and still documents an external tunnel workaround for this setup.

## What This Changes

The issue requests SSH transport support for `openclaw node run` and reliable remote challenge authentication behind a TLS reverse proxy.

## Best Possible Solution

Add a maintained node-host SSH transport path that consumes the existing remote SSH config, supervises tunnel lifetime for foreground and installed services, and keeps non-loopback challenge-auth behavior covered by regression tests.

## Reproduction Assessment

Yes for the remaining SSH transport gap: current source shows `node run` only accepts direct host/port/TLS inputs and `runNodeHost` builds a direct WS/WSS URL. I did not establish a live reverse-proxy pairing reproduction for the older auth symptom.

## Solution Assessment

Unclear. Reusing `startSshPortForward` is the likely technical direction, but maintainers should first decide node service lifecycle, reconnect behavior, SSH identity handling, and security semantics.
