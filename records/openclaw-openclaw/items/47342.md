## Summary

Keep open: current main has improved the remote-token and pairing-request paths, but `openclaw node run` still has no first-class SSH transport and still requires operators to manage an external tunnel for this setup.

## What This Changes

The issue requests first-class SSH transport support for `openclaw node run` and reliable non-loopback WSS challenge auth/pairing behind a reverse proxy.

## Best Possible Solution

Add a maintained node-host SSH transport path that reuses the existing tunnel helper/config, supervises tunnel lifetime for foreground and installed services, and keeps non-loopback pairing challenge behavior covered by regression tests.

## Reproduction Assessment

Yes for the remaining SSH transport gap: current source shows `node run` only accepts direct host/port/TLS inputs and `runNodeHost` builds a direct WS/WSS URL. I did not establish a live nginx or SSH reproduction for the original reverse-proxy pairing symptom.

## Solution Assessment

Unclear. Reusing `startSshPortForward` is the likely technical direction, but maintainers should first decide node service lifecycle, reconnect behavior, SSH identity handling, and security semantics.
