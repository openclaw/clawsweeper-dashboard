## Summary

Keep open: current main has improved the remote-token and pairing-request parts of the report, but the central requested node-host SSH transport is still missing and current docs still tell operators to run an external tunnel.

## What This Changes

The issue requests first-class SSH transport support for `openclaw node run`/installed node hosts and reliable non-loopback WSS challenge auth for remote node pairing.

## Best Possible Solution

Add a maintained node-host SSH transport path that reuses the existing tunnel helper/config, supervises tunnel lifetime for foreground and installed node services, and keeps non-loopback device-pairing challenge behavior covered by regression tests.

## Reproduction Assessment

Yes for the remaining gap from source inspection: `node run` only exposes direct host/port/TLS wiring and `runNodeHost` builds a direct `ws`/`wss` URL while docs require an external SSH tunnel. I did not establish a live nginx/SSH reproduction.

## Solution Assessment

Unclear. Reusing `startSshPortForward` is the likely technical direction, but maintainers should decide the service lifecycle, reconnect, SSH identity, and security semantics before implementation.
