## Summary

Keep open. Current main still routes `openclaw node run` and installed node services through direct host/port/TLS settings only; the remote-token signing path has better source/test coverage now, but the reported reverse-proxy pairing failure is not live-confirmed and the remaining SSH transport work is security-sensitive.

## What This Changes

The issue requests first-class SSH transport support for the node host plus reliable challenge authentication when a node connects through a TLS reverse proxy.

## Best Possible Solution

Add a reviewed node-host SSH transport path that reuses the existing SSH port-forward helper for foreground and installed-service flows, connects only to the local loopback tunnel endpoint, preserves remote token/password and device-pairing semantics, and documents the supported setup.

## Reproduction Assessment

Yes for the SSH-transport gap: current source and docs show node-host foreground and service paths are direct host/port/TLS only. The reverse-proxy challenge failure is unclear on current main because nearby source/tests were inspected but no nginx/podman live reproduction was run.

## Solution Assessment

Unclear as an implementation plan: first-class node-host SSH support is a plausible direction, but the best fix needs maintainer-reviewed SSH lifecycle, loopback endpoint validation, and reverse-proxy pairing regression coverage rather than a cleanup close.
