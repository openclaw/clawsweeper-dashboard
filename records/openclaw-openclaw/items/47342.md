## Summary

Keep open: current main appears to have improved the remote-token and pairing-request path, but the node host still only builds a direct host/port WebSocket connection and the docs still require users to run an external SSH tunnel.

## What This Changes

Requests first-class SSH tunnel transport for `openclaw node run`/`openclaw node install` and reliable node device-pairing challenge auth through a TLS reverse proxy.

## Best Possible Solution

Add a maintained node-host SSH transport path that reuses the existing tunnel helper/config where appropriate, supervises tunnel lifetime with `node run` and installed services, and keeps the current device-pairing challenge behavior covered by regression tests.

## Reproduction Assessment

Yes for the remaining gap from source inspection: `node run` only exposes direct host/port/TLS wiring and `runNodeHost` builds a direct `ws`/`wss` URL, while the docs instruct users to create the SSH tunnel manually. I did not establish a live nginx/SSH reproduction.

## Solution Assessment

Unclear. Reusing the existing SSH tunnel helper is the likely direction, but maintainers should decide how node-host-owned tunnels interact with daemon install, reconnection, SSH identity handling, and existing `gateway.remote.*` config.
