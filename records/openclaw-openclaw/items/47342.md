## Summary

Keep open: current main and the latest release still require external SSH tunneling for the CLI node host, while the reverse-proxy challenge-auth side is only partially source-confirmed and remains security/product-sensitive.

## What This Changes

The issue requests managed SSH transport for `openclaw node run`/`install` and a fix for reverse-proxy `wss://` node challenge auth that fails without a visible pending pairing request.

## Best Possible Solution

Define the remote node-host contract first, then either integrate the existing SSH helper with explicit lifecycle/security proof or keep external tunnels supported and fix only the verified auth/pairing bug.

## Reproduction Assessment

Partially yes: source inspection reproduces the missing managed SSH path because node run still constructs a direct WebSocket URL and never calls the SSH tunnel helper. The reverse-proxy challenge-auth failure was not live-reproduced on current main.

## Solution Assessment

Unclear: managed node-host SSH is plausible, but the best fix depends on maintainer direction for node-owned SSH lifecycle versus documented external tunnels. Any auth repair should preserve or deliberately revise the documented local/remote credential contract.
