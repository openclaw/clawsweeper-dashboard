## Summary

Keep open: current main still has a direct-only node-host connection path with no first-class SSH transport or service lifecycle, while the remote-token challenge-auth part appears partially covered by current credential tests.

## What This Changes

The issue requests first-class SSH transport support for `openclaw node run` and a fix for remote reverse-proxy node pairing challenge auth.

## Best Possible Solution

Add a maintained node-host SSH transport path that consumes existing `gateway.remote` SSH settings, supervises tunnel lifetime for foreground and installed services, and preserves tested remote challenge-auth behavior.

## Reproduction Assessment

Yes for the source-level SSH transport gap: current main exposes only direct host/port/TLS node-host inputs and builds a direct WS/WSS URL. No for the nginx reverse-proxy challenge-auth symptom; I did not establish a live reproduction, and current tests only cover the credential-resolution portion.

## Solution Assessment

Unclear. Reusing `startSshPortForward` is the likely implementation direction, but maintainers should decide node service lifecycle, reconnect behavior, SSH identity handling, credential precedence, and loopback trust semantics first.
