## Summary

Keep open. Current `main` still leaves the node-host SSH transport gap in place: `openclaw node run`, persisted node config, runner URL construction, and installed-service argument generation are all direct host/port/TLS paths. The remote-token signing path is now covered better, but the reverse-proxy challenge case still needs live confirmation and security-aware design rather than cleanup closure.

## What This Changes

The issue requests first-class SSH transport for `openclaw node run` plus reliable challenge authentication for node hosts connecting through a TLS reverse proxy.

## Best Possible Solution

Add a reviewed node-host SSH transport path that reuses the existing SSH port-forward helper for foreground and installed-service flows, connects `GatewayClient` only to the local loopback tunnel endpoint, preserves remote token/password and device-pairing semantics, and updates node/remote docs.

## Reproduction Assessment

Yes for the SSH-transport gap: current source and docs show node-host foreground and service flows use direct host/port/TLS settings only. Unclear for the reverse-proxy challenge failure on current `main` because the source/tests were inspected but no nginx/podman live reproduction was run.

## Solution Assessment

No, the current direct-only node-host path is not the best complete solution for the reported workflow. The maintainable fix is node-host-owned SSH tunnel lifecycle with explicit loopback endpoint validation and credential/device-pairing coverage, reviewed alongside #26250.
