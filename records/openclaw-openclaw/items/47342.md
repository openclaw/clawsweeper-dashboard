## Summary

Keep open: current main still lacks a first-class SSH transport path for `openclaw node run`, while the reverse-proxy token-resolution portion appears partly covered by newer shared node-host credential handling.

## What This Changes

The issue requests SSH transport support for `openclaw node run` and reliable node-host challenge auth when connecting to a remote gateway through a TLS reverse proxy.

## Best Possible Solution

Add a maintained node-host SSH transport path that consumes the existing `gateway.remote` SSH settings, supervises tunnel lifetime for foreground and installed services, and preserves tested remote challenge-auth behavior.

## Reproduction Assessment

Yes for the remaining SSH transport gap: current main exposes only direct host/port/TLS node-host inputs and builds a direct WS/WSS URL. I did not establish a live nginx reverse-proxy reproduction for the older pairing failure.

## Solution Assessment

Unclear. Reusing `startSshPortForward` is the likely implementation direction, but maintainers should first decide node service lifecycle, reconnect behavior, SSH identity handling, and loopback trust semantics.
