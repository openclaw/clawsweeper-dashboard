## Summary

Keep open: current main and v2026.6.6 still use raw `chmodSync` in the agent DB path, and this PR is a small, plausible extension of the merged shared-state chmod-less volume fix. It still needs non-mock real behavior proof, plus maintainer/security acceptance that the same chmod-less fallback policy should apply to per-agent auth/profile state.

## What This Changes

The PR exports the shared state database chmod-tolerance helper, uses it for agent database directory and sidecar permission hardening, and adds agent DB regression coverage for chmod-less filesystems.

## Best Possible Solution

Reuse the merged chmod-less helper for the agent DB if maintainers accept the same policy there, but require non-mock runtime proof or an explicit maintainer override before merge.

## Reproduction Assessment

Yes, at source level rather than from a live volume. Current main calls raw `chmodSync` in the agent DB open and write paths, so an ENOTSUP or EPERM from chmod will propagate; the added mocked regression demonstrates that exact path.

## Solution Assessment

Yes, conditionally. Reusing the already-narrow shared-state helper is cleaner than adding a new env var or wrapping broad callers because it keeps path validation and unexpected chmod failures fatal, but the branch still needs real runtime proof before merge.
