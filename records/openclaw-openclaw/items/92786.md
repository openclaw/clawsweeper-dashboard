- `P1`: The PR targets an agent/auth startup crash class on chmod-less mounted filesystems, which can block real users from running agents.

## Summary

Keep this PR open as the active fix: current main still has raw agent-DB chmod calls, while the branch reuses the already-merged shared-state chmod policy, adds focused regression coverage, and now includes sufficient real terminal proof.

## What This Changes

The PR exports the shared state database best-effort chmod helper, applies it to agent database directory and sidecar hardening, and adds agent database chmod-failure regression tests.

## Best Possible Solution

Reuse the already accepted shared chmod policy for the agent DB, keep the focused regression test, and merge after ordinary maintainer review and exact-head gates.

## Reproduction Assessment

Yes, at source level: current main calls raw chmodSync on the per-agent DB open path, so an ENOTSUP or eligible EPERM chmod-less filesystem error propagates before the agent database opens. The contributor's terminal proof also shows the same before/after behavior through the real agent DB and auth-profile reader paths, though I did not run a live mount in this read-only review.

## Solution Assessment

Yes: reusing the merged shared-state helper is the narrowest maintainable fix because it keeps one chmod policy, preserves fail-closed behavior for EACCES and ordinary ownership failures, and avoids a new env/config workaround. Wrapping individual agent callers would duplicate policy and miss post-open or transaction hardening paths.
