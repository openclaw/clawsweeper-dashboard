## Summary

Keep open. Current main and the latest release still manage macOS Gateway install as a per-user LaunchAgent, while the remaining request is first-class system LaunchDaemon install/status for boot-before-login always-on Macs; the TMPDIR portion is partially addressed, but the privileged launchd scope contract still needs product and security review.

## What This Changes

The issue asks for macOS Gateway `install` and `status` support for a system LaunchDaemon so the Gateway can start before user login and report system-domain service state correctly.

## Best Possible Solution

Define one documented opt-in macOS user/system launchd scope model that keeps LaunchAgent as the default, then align install, status, stop, uninstall, doctor guidance, docs, and regression tests around that contract.

## Reproduction Assessment

Yes, at source level: current main routes macOS Gateway install/status through the user LaunchAgent path and `gui/<uid>` domain with no managed system-scope install option. I did not run a live reboot-without-login macOS scenario.

## Solution Assessment

Unclear as a final implementation plan. LaunchDaemon support is directionally right for always-on Macs, but maintainers should choose the CLI scope, privilege model, service user, environment, coexistence, and lifecycle/status semantics first.
