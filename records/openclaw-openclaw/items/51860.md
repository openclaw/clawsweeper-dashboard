## Summary

Keep open. Current main and v2026.6.10 still manage macOS Gateway install/status through the user LaunchAgent path and GUI launchd domain; the open lifecycle companion overlaps but does not supersede this install/status and boot-without-login request.

## What This Changes

The issue requests macOS Gateway install/status support for system LaunchDaemons so the Gateway can start at boot before user login and report that state accurately.

## Best Possible Solution

Define one explicit macOS launchd user/system scope contract across install, status, stop, uninstall, doctor, docs, and tests, keeping user LaunchAgent as the safe default and adding only an audited explicit LaunchDaemon path or guidance flow.

## Reproduction Assessment

Yes at source level. Current main still resolves Darwin Gateway install/status through the LaunchAgent registry, user LaunchAgents plist path, and gui/<uid> launchctl calls; I did not run a live macOS boot-without-login scenario in this read-only review.

## Solution Assessment

No. A bare --system flag is a plausible direction, but the maintainable fix is a shared launchd scope model covering privilege, execution user, coexistence, lifecycle, status, doctor, docs, and tests.
