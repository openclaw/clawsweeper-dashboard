## Summary

Keep open. Current main still installs and manages the macOS gateway as a per-user LaunchAgent; LaunchDaemons are only discovered as extra services in deep status scans, so the boot-without-login request remains unimplemented.

## What This Changes

Request support for installing and managing the macOS gateway as a system LaunchDaemon with accurate status reporting.

## Best Possible Solution

Design an explicit opt-in macOS system-service mode that installs and manages /Library/LaunchDaemons with UserName, durable TMPDIR, matching lifecycle/status/doctor/docs support, and coordination with #14251.

## Reproduction Assessment

Yes. The reporter's steps are concrete, and current-main source inspection shows gateway install/status/lifecycle still target the user LaunchAgent path and gui launchd domain while LaunchDaemon handling is only best-effort --deep discovery.

## Solution Assessment

Yes in direction, but this needs maintainer design first. An explicit opt-in system mode is safer than automatic detection because it changes privilege, ownership, launchd domain behavior, and service environment handling.
