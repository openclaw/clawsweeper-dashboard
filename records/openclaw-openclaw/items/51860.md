## Summary

Keep open. Current main and v2026.6.10 still manage macOS Gateway install/status as a user LaunchAgent in the GUI launchd domain; system LaunchDaemon handling remains a product and security decision, with related lifecycle and duplicate-manager reports still open.

## What This Changes

Requests macOS Gateway support for installing and reporting a system LaunchDaemon so the Gateway can start before user login and status will not misreport system-domain services as missing.

## Best Possible Solution

Define one explicit macOS launchd user/system scope contract across install, status, stop, uninstall, doctor, docs, and tests, keeping user LaunchAgent as the safe default and adding only an audited explicit LaunchDaemon path or guidance flow.

## Reproduction Assessment

Yes, at source level. Current main still resolves Darwin Gateway install/status through the LaunchAgent registry, user LaunchAgents plist path, and `gui/<uid>` launchctl calls; I did not run a live macOS boot-without-login scenario in this read-only review.

## Solution Assessment

No, the bare `--system` flag sketch is only a plausible direction. The maintainable solution is a shared launchd scope model covering privilege, execution user, coexistence, lifecycle, status, doctor, docs, and tests.
