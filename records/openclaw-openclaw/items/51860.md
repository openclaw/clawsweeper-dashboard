## Label Justifications

- `P2`: This is a concrete macOS Gateway reliability improvement with real always-on impact, but it is platform-scoped and blocked on product/security design rather than an emergency regression.
- `impact:security`: The requested solution would manage root-owned LaunchDaemon plists, service-user ownership, sudo guidance, and system-domain launchctl operations.
- `impact:message-loss`: When the Gateway does not start after an unattended reboot, channel messages such as WhatsApp traffic can go unanswered until a user logs in.

## Summary

Keep open. Current main and the latest release still manage macOS Gateway installs as per-user LaunchAgents, while the requested opt-in LaunchDaemon install/status mode remains a real always-on reliability gap that needs maintainer product and security review before implementation.

## What This Changes

The issue requests an opt-in macOS system LaunchDaemon install/status mode so the Gateway can start before user login and report system-domain services correctly.

## Best Possible Solution

Define a documented opt-in macOS launchd scope model, keep the existing LaunchAgent default, then implement LaunchDaemon install/status/lifecycle/doctor/docs/tests as one coherent service-management design.

## Reproduction Assessment

Yes, at source level: current main routes macOS install/status through a per-user LaunchAgent path and gui/<uid> launchd domain, with no supported system-scope install/status option. I did not run a live reboot-without-login macOS scenario.

## Solution Assessment

Unclear as a final implementation plan. LaunchDaemon support is directionally right for this use case, but maintainers should first choose the CLI scope, service user, sudo UX, environment, lifecycle, and status semantics.
