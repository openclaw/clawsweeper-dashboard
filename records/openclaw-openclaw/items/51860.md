## Label Justifications

- `P2`: This is a normal-priority macOS setup reliability gap with meaningful impact for always-on gateway users but limited platform scope.
- `impact:security`: Managed LaunchDaemon support crosses elevated privilege, root-owned service metadata, execution-user, and environment-boundary decisions.
- `impact:message-loss`: When the gateway does not start before login after reboot, channel traffic can be unavailable until someone logs into the Mac.

## Summary

Keep open. Current main and v2026.6.8 still manage macOS gateway installs/status as a user LaunchAgent; the requested managed LaunchDaemon path remains a security-sensitive product decision.

## What This Changes

Request support for macOS system LaunchDaemon gateway install/status so the gateway can start before user login and report system-domain services accurately.

## Best Possible Solution

Define one explicit macOS launchd user/system scope contract across install, status, stop, uninstall, doctor, docs, and tests, keeping user LaunchAgent as the safe default and adding only an explicit audited LaunchDaemon path or guidance flow.

## Reproduction Assessment

Yes. At source level, current main resolves Darwin gateway install/status through the LaunchAgent registry, user LaunchAgents plist path, and GUI-domain launchctl calls; I did not run a live macOS boot-without-login scenario.

## Solution Assessment

No. A bare --system flag is only a plausible direction; the maintainable fix is a shared launchd scope model covering privilege, execution user, coexistence, lifecycle, status, doctor, docs, and tests.
