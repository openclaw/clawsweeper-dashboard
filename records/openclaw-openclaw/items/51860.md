## Label Justifications

- `P2`: This is a concrete macOS Gateway reliability improvement with real always-on impact, but it is platform-scoped and blocked on product/security design rather than an emergency regression.
- `impact:security`: The requested solution would manage root-owned LaunchDaemon plists, service-user ownership, sudo guidance, and system-domain launchctl operations.
- `impact:message-loss`: When the Gateway does not start after an unattended reboot, channel traffic such as WhatsApp messages can go unanswered until a user logs in.

## Summary

Keep open. Current main has hardened the macOS service TMPDIR path, but Gateway install/status still use a per-user LaunchAgent and `gui/<uid>` launchd domain; first-class LaunchDaemon install/status remains a real always-on capability that needs product and security review.

## What This Changes

The issue requests macOS Gateway support for an opt-in system LaunchDaemon install/status mode so the service can start before user login.

## Best Possible Solution

Define one documented opt-in macOS launchd scope model, keep LaunchAgent as the default, then implement LaunchDaemon install/status/lifecycle/doctor/docs/tests as a coherent service-management design.

## Reproduction Assessment

Yes, at source level: current main routes macOS Gateway install/status through `$HOME/Library/LaunchAgents` and `gui/<uid>` with no managed system-scope install/status option. I did not run a live reboot-without-login macOS scenario.

## Solution Assessment

Unclear as a final implementation plan. LaunchDaemon support is directionally right for always-on Macs, but maintainers should choose the CLI scope, privilege model, service user, environment, coexistence, and lifecycle/status semantics first.
