- `P2`: This is a normal-priority macOS gateway reliability gap with meaningful always-on setup impact but limited platform scope and pending product/security decisions.
- `impact:security`: Managed LaunchDaemon support crosses elevated privilege, root-owned service metadata, execution-user, sudo/operator flow, and durable environment-boundary choices.
- `impact:message-loss`: When the gateway does not start before login after reboot, remote channel traffic cannot reach the assistant until someone logs into the Mac.

## Summary

Keep open. Current main and v2026.6.10 still manage the macOS gateway service as a user LaunchAgent, while `gateway status --deep` only discovers extra system services and does not provide a managed LaunchDaemon install/status mode.

## What This Changes

Requests a macOS system LaunchDaemon install/status mode so the gateway can start before user login and report system-domain launchd state accurately.

## Best Possible Solution

Define one explicit macOS launchd user/system scope contract across install, status, stop, uninstall, doctor, docs, and tests, keeping user LaunchAgent as the safe default and adding only an audited explicit LaunchDaemon path or guidance flow.

## Reproduction Assessment

Yes at source level. Current main still resolves Darwin gateway install/status through the LaunchAgent registry, user LaunchAgents plist path, and gui/<uid> launchctl calls; I did not run a live macOS boot-without-login scenario in this read-only review.

## Solution Assessment

No. A bare --system flag is only a plausible direction; the maintainable fix is a shared launchd scope model covering privilege, execution user, coexistence, lifecycle, status, doctor, docs, and tests.
