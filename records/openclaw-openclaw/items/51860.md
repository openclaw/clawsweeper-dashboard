- `P2`: This is a normal-priority macOS setup reliability and lifecycle gap with meaningful impact for always-on gateway users but limited platform scope.
- `impact:security`: Managed LaunchDaemon support crosses elevated privilege, root-owned service metadata, execution-user, and environment-boundary decisions.
- `impact:message-loss`: When the gateway does not start before login after reboot, channel traffic can be unavailable until someone logs into the Mac.

## Summary

Keep open. Current main and the latest release still manage macOS gateway install/status through the user LaunchAgent path and GUI launchd domain; deep status can discover system services, but the requested managed LaunchDaemon install/status path remains a security-sensitive product decision.

## What This Changes

The issue requests macOS gateway LaunchDaemon install/status support so the gateway can start before user login and report system-level launchd state accurately.

## Best Possible Solution

Define one explicit macOS launchd scope contract across install, status, stop, uninstall, doctor, docs, and tests, keeping user LaunchAgent as the safe default and adding only an explicit audited system LaunchDaemon path or guidance flow.

## Reproduction Assessment

Yes, at source level with high confidence: current main resolves Darwin gateway install/status through the LaunchAgent registry, user LaunchAgents plist path, and GUI-domain launchctl calls. I did not run a live macOS boot-without-login scenario in this read-only review.

## Solution Assessment

No, an isolated --system flag sketch is only a plausible direction. The maintainable fix is one shared launchd scope model coordinated with https://github.com/openclaw/openclaw/issues/14251 so privilege, execution user, coexistence, lifecycle, status, doctor, docs, and tests do not diverge.
