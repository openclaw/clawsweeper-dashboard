## Label Justifications

- `P2`: This is a concrete macOS Gateway reliability improvement with real always-on impact but limited platform scope and product/security design still pending.
- `impact:security`: The requested solution would manage root-owned LaunchDaemon plist files, service user ownership, and elevated launchctl operations.
- `impact:message-loss`: When the Gateway does not start after an unattended reboot, channel interactions such as WhatsApp messages can go unanswered until a user logs in.

## Summary

Keep open. Current main still treats the macOS Gateway as a per-user LaunchAgent and only offers diagnostic system-service scanning, so the requested LaunchDaemon install/status mode remains unresolved and is not safely auto-fixable without a maintainer product/security decision.

## What This Changes

The issue requests opt-in macOS LaunchDaemon install and status support for the Gateway so it can start at boot without a user login.

## Best Possible Solution

Define one documented opt-in macOS user/system launchd model, then implement LaunchDaemon install/status with `UserName`, durable environment handling, safe privileged guidance, lifecycle alignment with the open stop/uninstall tracker, docs, doctor guidance, and focused regression tests.

## Reproduction Assessment

Yes, source-level: current main still routes macOS Gateway install/status through a per-user LaunchAgent path and `gui/<uid>` launchd domain with no supported system-scope install/status option. I did not run a live reboot-without-login macOS scenario.

## Solution Assessment

Unclear as a final implementation plan. LaunchDaemon support is directionally right for this use case, but the exact CLI scope, privilege model, ownership, environment, status, and lifecycle semantics need maintainer design first.
