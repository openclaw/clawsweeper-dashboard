## Label Justifications

- `P2`: This is a concrete macOS Gateway reliability improvement with real always-on impact but limited platform scope and product/security design still pending.
- `impact:security`: The requested fix would manage root-owned LaunchDaemon plist files, service-user ownership, and elevated launchctl operations.
- `impact:message-loss`: When the Gateway does not start after an unattended reboot, channel interactions can go unanswered until a user logs in.

## Summary

Keep open. Current main and the latest shipped release still expose the macOS Gateway as a per-user LaunchAgent; `status --deep` can scan for extra services, but there is still no managed LaunchDaemon install/status scope.

## What This Changes

Add macOS system LaunchDaemon support for Gateway install/status so the service can start before user login and report system-domain state correctly.

## Best Possible Solution

Define one documented opt-in macOS user/system launchd model, then align install, status, stop/uninstall, doctor guidance, docs, and regression tests around that contract.

## Reproduction Assessment

Yes, source-level: current main routes macOS Gateway install/status through `~/Library/LaunchAgents` and `gui/<uid>` with no supported system LaunchDaemon install/status option. I did not run a live reboot-without-login macOS scenario.

## Solution Assessment

Unclear as a final implementation plan. LaunchDaemon support is directionally right for this use case, but the CLI scope, privilege model, ownership, environment, status, and lifecycle semantics need maintainer design first.
