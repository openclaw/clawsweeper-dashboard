- `impact:security`: System LaunchDaemon support crosses privilege, root-owned service metadata, execution-user, and environment-boundary decisions.
- `impact:message-loss`: When the gateway does not start before login after reboot, channel traffic such as WhatsApp/Tailscale-routed access can be unavailable until someone logs into the Mac.

## Summary

Keep open. Current main and the latest release still manage the macOS gateway as a user LaunchAgent; `status --deep` can discover system services, but managed LaunchDaemon install/status remains a security-sensitive product decision.

## What This Changes

The issue requests macOS system LaunchDaemon support for gateway install/status so the gateway can start before user login and report system-domain installs accurately.

## Best Possible Solution

Define one explicit macOS launchd scope contract across install, status, stop, uninstall, doctor, docs, and tests; keep user LaunchAgent as the safe default and add only an explicit, audited system LaunchDaemon path.

## Reproduction Assessment

Yes, at source level with high confidence. Current main manages only the user LaunchAgent path and GUI domain, and docs still say headless macOS users need a custom LaunchDaemon that is not shipped.

## Solution Assessment

No, an isolated `--system` flag sketch is only a plausible fix. The best solution is a shared launchd scope model with https://github.com/openclaw/openclaw/issues/14251 so install, status, lifecycle, privilege, and dual-install behavior do not diverge.
