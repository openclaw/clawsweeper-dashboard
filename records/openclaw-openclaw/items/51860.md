## Summary

Keep open. Current main still manages the macOS gateway as a per-user LaunchAgent, while this issue is the canonical open install/status request for an opt-in system LaunchDaemon mode; the remaining work needs maintainer design because it adds privileged service scope and CLI surface.

## What This Changes

The issue asks for macOS gateway install/status support for system-level LaunchDaemons so the gateway can start at boot without a user login.

## Best Possible Solution

Define one documented opt-in macOS user/system launchd service model, then implement LaunchDaemon install/status with safe privilege guidance, `UserName`, durable environment handling, doctor/lifecycle coordination, docs, and regression tests.

## Reproduction Assessment

Yes for source reproduction: current main still resolves the managed macOS gateway service to `$HOME/Library/LaunchAgents` and `gui/<uid>`, with no system-scope install/status option. I did not run a live macOS reboot-without-login scenario.

## Solution Assessment

Unclear as a final implementation plan. Adding an explicit LaunchDaemon path is the right direction, but the exact CLI flag, privilege model, ownership, TMPDIR, doctor, and lifecycle/status semantics should be decided with the related lifecycle work.
