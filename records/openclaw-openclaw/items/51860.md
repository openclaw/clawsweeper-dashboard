## Summary

Keep open. Current main still implements the managed macOS gateway service as a per-user LaunchAgent; `status --deep` can discover extra system services and macOS TMPDIR handling has been hardened, but there is still no supported LaunchDaemon install/status lifecycle contract.

## What This Changes

Request macOS gateway install/status support for system-level LaunchDaemon mode so the gateway can start before user login and report system-domain services accurately.

## Best Possible Solution

Define one opt-in macOS LaunchDaemon service mode spanning install, status, stop, uninstall, plist ownership, `UserName`, durable environment/TMPDIR handling, sudo guidance, doctor behavior, docs, and regression tests.

## Reproduction Assessment

Yes for source-level reproduction: current main registers install without a system-scope flag, uses `$HOME/Library/LaunchAgents` and `gui/<uid>`, and only scans `/Library/LaunchDaemons` in `status --deep`. I did not run a live macOS reboot/no-login scenario.

## Solution Assessment

Unclear as a final implementation until maintainers choose the launchd scope contract. The requested direction is plausible, but the maintainable fix should coordinate install/status/lifecycle/doctor/docs rather than adding an isolated path check.
