## Summary

Keep open. Current main still treats the managed macOS gateway service as a per-user LaunchAgent and only discovers LaunchDaemons as best-effort extras, so the boot-without-login install/status request remains unresolved despite adjacent TMPDIR hardening.

## What This Changes

The issue requests adding a macOS system LaunchDaemon installation/status mode for the gateway so it starts at boot before login and reports system-domain services correctly.

## Best Possible Solution

Design an explicit macOS system-service mode that installs and manages /Library/LaunchDaemons/ai.openclaw.gateway.plist with safe ownership, UserName, durable environment, and matching status/start/stop/restart/uninstall/doctor/docs support, coordinated with #14251 rather than closing this as already handled.

## Reproduction Assessment

Yes. The reporter's steps are concrete, and current-main static inspection reproduces the product gap: install/status/lifecycle still target the LaunchAgent path and GUI launchd domain, while LaunchDaemon discovery is only a deep extra-service scan. I did not run a live macOS boot test in this read-only review.

## Solution Assessment

Yes in direction, but it needs maintainer design. An explicit opt-in system mode is safer than automatic detection because it changes privilege, ownership, and launchd domain behavior; the fix should include lifecycle/status/doctor support, not only plist generation.
