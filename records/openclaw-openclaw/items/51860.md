## Summary

Keep open. Current main still installs and manages the macOS Gateway as a per-user LaunchAgent; `status --deep` has extra-service scanning, and the macOS TMPDIR issue has been improved, but there is still no supported LaunchDaemon install/status mode for boot-without-login setups.

## What This Changes

Request macOS `openclaw gateway install/status` support for an opt-in system LaunchDaemon under `/Library/LaunchDaemons` with user ownership and durable environment handling.

## Best Possible Solution

Define one documented opt-in macOS user/system launchd model, then implement LaunchDaemon install/status and align stop/uninstall, doctor guidance, docs, and regression tests around that contract.

## Reproduction Assessment

Yes, source-level: current main still resolves the managed macOS Gateway service to `$HOME/Library/LaunchAgents` and `gui/<uid>` with no system-scope install/status option. I did not run a live macOS reboot-without-login scenario.

## Solution Assessment

Unclear as a final implementation plan. LaunchDaemon support is the right product direction for this use case, but the exact CLI flag, privilege model, ownership, environment, status, and lifecycle semantics need maintainer design first.
