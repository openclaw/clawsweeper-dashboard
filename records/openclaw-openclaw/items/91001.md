## Summary

Keep open: current main still has the same source-level identity mismatch reported by the earlier ClawSweeper review, so local indexes built with `memorySearch.local.modelPath` can remain reported as dirty after rebuild.

## What This Changes

The issue reports that local memory-search embeddings remain marked dirty after a forced rebuild because `openclaw memory status` still reports `Dirty: yes`.

## Best Possible Solution

Make status-only memory identity resolution use the same local provider model identity that indexing writes, then add regression coverage for custom `memorySearch.local.modelPath`.

## Reproduction Assessment

Yes, source-level: local indexing persists `local.modelPath` as the model identity, while status-only identity can expect `settings.model` or the adapter default before provider initialization. I did not run the full Docker/GGUF repro in this read-only pass.

## Solution Assessment

Yes, the best fix is a narrow memory-core identity repair so status and indexing agree on the local provider model; adding a new config option or fallback path would be the wrong layer.

Current source path: a local memory index built with `memorySearch.provider: "local"` and `memorySearch.local.modelPath` writes `meta.model` from `provider.model`, and the local worker provider reports that model as `local.modelPath`; plain/status-only identity resolution currently expects `settings.model` or the adapter default before provider init, so status can keep reporting dirty after rebuild. Keep the fix inside the memory-core/local identity path, preferably by making `resolveCurrentIndexIdentityState` derive the same configured local model identity as provider initialization. Do not add a config option, runtime fallback, changelog entry, or broad provider refactor.
