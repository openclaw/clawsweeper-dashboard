## Summary

Keep open. Current main still truncates chat-history display text at an 8,000-character default before the larger byte-budget checks, so the reported WebChat/Control UI behavior remains valid. The companion fix PR is still open and unmerged, so this issue should stay open until that PR or an equivalent gateway fix lands.

## What This Changes

`chat.history` should preserve long assistant text that fits the configured history and per-message byte budgets instead of truncating each text field at the default character cap.

## Best Possible Solution

Update the shared chat display projection so long assistant text is preserved when the serialized history entry fits the byte budget, keep the existing config/RPC override semantics, make UTF-8 and JSON-escape-heavy truncation degrade to a visible prefix instead of the oversized placeholder, and add focused gateway/UI regression coverage plus docs/config-help updates for the chosen default.
