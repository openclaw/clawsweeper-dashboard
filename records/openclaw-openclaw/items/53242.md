## Summary

Keep open. Current main still truncates WebChat history text before the serialized byte budget, and the linked closing PR remains open and conflicting while the newer 4,000-character Control UI datapoint is already folded into this canonical issue.

## What This Changes

The issue asks OpenClaw to preserve long assistant text in `chat.history` when it fits the configured history and single-message byte budgets instead of truncating it via a low character cap.

## Best Possible Solution

Align `src/gateway/chat-display-projection.ts`, `chat.history` byte budgeting, and Control UI request policy so configured WebChat history limits are honored, with focused gateway and UI regression tests.

## Reproduction Assessment

Yes, source-reproducible. A 50k ASCII assistant message fits under the 128 KiB single-message byte guard, while current main applies the 8,000 default cap or the Control UI's 4,000 RPC cap before that guard runs.

## Solution Assessment

No, the original one-line 64k bump is not the best complete solution. The durable fix should update the shared projection helper and align serialized-byte budgeting with the gateway config and Control UI request policy.
