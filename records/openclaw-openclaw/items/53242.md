## Summary

Keep open. Current main still truncates chat.history display text before the larger byte-budget checks, and the normal Control UI/WebChat history request still uses that default path. The open companion PR #53243 is explicitly paired with this issue and remains unmerged, so this issue should stay open until that PR or an equivalent gateway fix lands.

## What This Changes

The issue requests that chat.history preserve long assistant messages that fit the configured history and per-message byte budgets instead of truncating them at a small per-field character cap.

## Best Possible Solution

Update the shared chat display projection limit so the default no longer preempts messages that fit the byte budgets, preserve the existing config and RPC override semantics, make truncation byte-aware for UTF-8-heavy content, add a default-path regression test for long assistant text, and sync config help/docs plus changelog if the shipped default changes.

## Reproduction Assessment

Yes. A current-main chat.history response containing an assistant text field longer than 8,000 characters will be truncated in chat-display-projection before the 128 KiB per-message cap and 6 MiB history budget are considered; the Control UI default request does not override maxChars.

## Solution Assessment

Unclear for the exact proposed one-line cap bump: raising the cap addresses the ASCII reproduction, but the maintainable fix should also be byte-aware or derived from the budget. The paired PR #53243 is the right item to review because it claims to cover both the 50k ASCII case and UTF-8-heavy fallback behavior.
