## Summary

Keep open. Current main still truncates default chat.history text before the larger byte-budget checks, and the provided GitHub context shows the paired fix PR #53243 is open, unmerged, and explicitly closes this issue if it lands.

## What This Changes

The issue asks for chat.history to preserve long assistant text that still fits the history and per-message byte budgets instead of truncating it at a much smaller text-field cap.

## Best Possible Solution

Land PR #53243 or an equivalent gateway fix that raises or derives the default display text cap, keeps truncation byte-aware so UTF-8-heavy messages do not collapse to the oversized placeholder, adds default-path regression coverage, and updates config help/docs plus changelog for the user-facing default change.

## Reproduction Assessment

Yes. A current-main chat.history response with an assistant text field longer than 8,000 characters on the default Control UI path will be truncated before the 128 KiB single-message cap is considered; a 50k ASCII message fits that byte cap but still hits the earlier character cap.

## Solution Assessment

Unclear for the issue's original one-line 64k-only proposal. The safer solution is the paired PR's broader shape or equivalent: increase the default while making truncation byte-aware and covering both ASCII and UTF-8-heavy messages in gateway tests.
