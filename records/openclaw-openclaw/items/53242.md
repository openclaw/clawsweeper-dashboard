## Summary

Keep open. Current main and v2026.5.22 still truncate `chat.history` text through the shared projection cap and Control UI request cap before the larger byte-budget guards, and the direct closing PR is still open and conflicting.

## What This Changes

`chat.history` should preserve long assistant text that fits the serialized history budgets instead of truncating it at the projection or request character cap.

## Best Possible Solution

Land or replace the direct fix so shared history projection preserves text that fits serialized budgets, while keeping bounded WebChat payload behavior explicit and leaving the broader full-message reader to its canonical issue.

## Reproduction Assessment

Yes, at source level. A 50k ASCII assistant message is below the 128 KiB single-message cap, but current main can truncate it first through the 8,000-character projection default or Control UI's 4,000-character request cap.

## Solution Assessment

No, the original one-line cap bump is not the best complete solution on current main. The durable fix should update the shared projection behavior, handle byte-heavy text safely, and settle the Control UI request-cap policy.
