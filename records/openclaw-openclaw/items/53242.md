## Summary

Keep open. Current main still applies the default `chat.history` text-field cap before the byte-budget checks, and the supplied context shows paired PR #53243 is open, unmerged, and explicitly closes this issue if it lands.

## What This Changes

The issue asks for long assistant text in `chat.history` to be preserved when it still fits the configured history and single-message byte budgets.

## Best Possible Solution

Land PR #53243 or an equivalent gateway fix that raises or derives the default cap, keeps truncation byte-aware for UTF-8-heavy text, adds default-path regression coverage, and updates config help/docs plus changelog for the user-facing default change.

## Reproduction Assessment

Yes. Source inspection of current main shows WebChat calls `chat.history` without a `maxChars` override, the default cap is 8,000 characters, and projection truncates text before the 128 KiB per-message byte cap, so a 50k ASCII assistant message would still be cut on the default path.

## Solution Assessment

Yes for keeping this issue open until the paired PR is resolved. The narrow maintainable fix is PR #53243 or equivalent, because it addresses both the too-low default and the UTF-8 byte-cap fallback instead of only changing one constant.
