## Summary

Keep open. Current main still truncates WebChat history text before the serialized byte budget, and the linked closing PR remains open/conflicting rather than merged.

## What This Changes

The issue asks for `chat.history` and WebChat to stop prematurely truncating long assistant text that still fits the configured history byte budgets.

## Best Possible Solution

Align `src/gateway/chat-display-projection.ts`, serialized-byte budgeting, and Control UI request policy so configured WebChat history limits are honored, with focused gateway and UI regression tests.

## Reproduction Assessment

Yes, source-reproducible. A 50k ASCII assistant message fits under the 128 KiB per-message byte guard, but current main applies the 8,000 default projection cap or the Control UI's 4,000 RPC cap before that guard runs.

## Solution Assessment

No, the original one-line 64k bump is not the best complete fix. The durable solution should update the shared projection helper, budget against serialized message size, and settle the Control UI/config cap policy.
