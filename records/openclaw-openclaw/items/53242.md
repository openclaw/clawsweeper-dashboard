## Summary

Keep open. Current main and the latest release still apply character caps before the serialized byte-budget guards, Control UI still requests a 4,000-character history cap, and the linked closing PR remains open rather than merged.

## What This Changes

The issue requests that `chat.history` preserve long assistant text when it still fits the configured history and per-message budgets.

## Best Possible Solution

Review, rebase, or replace https://github.com/openclaw/openclaw/pull/53243 so the shared projection helper, serialized-byte safeguards, and Control UI request policy preserve readable long assistant history without unbounded payload growth.

## Reproduction Assessment

Yes, source-reproducible. A 50k ASCII assistant message fits under the 128 KiB serialized single-message cap, but current `chat.history` can truncate it first through the 8,000-character projection default or the Control UI's 4,000-character request cap.

## Solution Assessment

No, the original one-line cap bump is not the best complete solution on current main. The durable fix should update the shared projection behavior, budget against serialized message bytes, and settle the Control UI maxChars/config policy.
