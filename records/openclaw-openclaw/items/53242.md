## Summary

Keep open. Current main still applies a character projection cap before the serialized byte-budget guards, Control UI still requests a lower 4,000-character cap, and the linked closing PR is still open and conflicting.

## What This Changes

The issue requests that `chat.history` stop cutting long assistant text at fixed small character caps when the message still fits the configured history and single-message byte budgets.

## Best Possible Solution

Review, rebase, or replace https://github.com/openclaw/openclaw/pull/53243 so the gateway projection cap, serialized-byte budgeting, and Control UI request policy preserve long assistant text within intended budgets, while coordinating separately with the full-message reader work for oversized cases.

## Reproduction Assessment

Yes, source-reproducible. A 50k ASCII assistant message fits under the 128 KiB serialized single-message cap, but current main can truncate it first through the 8,000-character projection default or the Control UI's 4,000-character request cap.

## Solution Assessment

No, the original one-line 12k-to-64k bump is not a complete current-main fix. The durable solution should update the shared projection behavior and Control UI request policy while preserving byte-budget safeguards.
