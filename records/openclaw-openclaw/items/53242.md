## Summary

Keep open. Current main still applies a low character cap before the serialized byte budget, and the paired closing PR at https://github.com/openclaw/openclaw/pull/53243 is still open and conflicting; the newer Control UI 4,000-character datapoint is valid and already folded into this issue as the canonical tracker.

## What This Changes

The issue requests aligning `chat.history` truncation so long assistant messages that fit the history byte budget are not cut off in WebChat/Control UI.

## Best Possible Solution

Implement the fix in `src/gateway/chat-display-projection.ts`, preserve text that fits the serialized history budget, truncate byte-heavy text to a visible prefix, and align gateway default/config/UI cap policy with focused gateway and UI tests.

## Reproduction Assessment

Yes, source-reproducible. A 50k ASCII assistant message fits under the 128 KiB per-message byte guard, but current main applies the 8,000 default cap or the Control UI's 4,000 RPC cap before that guard runs.

## Solution Assessment

No, the original one-line constant bump is not the best complete fix. The durable path should update the shared projection helper and align serialized-byte budgeting with the gateway/config/UI cap policy.
