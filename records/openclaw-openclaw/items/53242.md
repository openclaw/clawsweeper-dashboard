## Summary

## What This Changes

The issue reports that chat.history truncates long assistant messages before the configured history and per-message byte budgets should require truncation.

## Best Possible Solution

Keep this issue open until the narrow chat.history cap fix lands or is replaced, then leave the broader full-message reader work to https://github.com/openclaw/openclaw/issues/84651 and https://github.com/openclaw/openclaw/pull/84670.

## Reproduction Assessment

Yes. Source inspection and byte math show that a 50k ASCII assistant message fits the 128 KiB single-message cap, but current chat.history can truncate it first through the 8,000-character projection default or Control UI's 4,000-character request cap.

## Solution Assessment

Unclear. The linked narrow PR is the right shape for the original cap bug, but maintainers still need to settle the Control UI maxChars policy and whether the broader full-message reader is the preferred user-facing recovery path.
