## Summary

Keep open. Current main still applies a character projection cap before the serialized history byte budget, and the linked closing PR remains open rather than merged.

## What This Changes

The issue asks OpenClaw to stop prematurely truncating long `chat.history` assistant text that still fits the configured history and per-message byte budgets.

## Best Possible Solution

Rework or replace the linked PR on current main so the shared projection helper, serialized-byte budgeting, and Control UI request policy honor the intended WebChat history limits with focused gateway and UI regression coverage.

## Reproduction Assessment

Yes, source-reproducible. A 50k ASCII assistant message fits under the 128 KiB serialized single-message cap, but current main can truncate it first at the 8,000 projection default or the Control UI's 4,000 request cap.

## Solution Assessment

No, the original one-line cap bump is not the best complete fix. The durable path should update the shared projection helper, account for serialized message bytes, and settle the Control UI/config cap policy.
