## Summary

Keep open. Current main and the latest release still truncate `chat.history` text fields before the larger byte-budget guards, and the direct closing PR for this report is still open rather than merged.

## What This Changes

The issue asks for `chat.history` to preserve long assistant text when the message still fits the configured history and single-message budgets.

## Best Possible Solution

Land or replace the current-main fix so shared history projection preserves text that fits serialized budgets, while keeping bounded WebChat payload behavior explicit.

## Reproduction Assessment

Yes. Source inspection and byte math show that a 50k ASCII assistant message fits the 128 KiB single-message cap, but current `chat.history` can truncate it first through the 8,000-character projection default or Control UI's 4,000-character request cap.

## Solution Assessment

No, the original one-line cap bump is not the best complete solution on current main. The durable fix should update the shared projection behavior, budget against serialized message bytes, and settle the Control UI `maxChars` policy.
