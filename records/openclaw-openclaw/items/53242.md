## Summary

Keep open. Current main and the latest shipped release still truncate `chat.history` text by a low character cap before the serialized byte-budget guards, and the paired closing PR is open, conflicting, and already reviewed as needing rework on the shared projection path.

## What This Changes

The issue asks for `chat.history` to preserve long assistant messages that fit the configured history budget instead of truncating visible text at a low per-field character cap.

## Best Possible Solution

Keep this issue as the canonical tracker, then rework or replace the paired PR so the fix lives in `src/gateway/chat-display-projection.ts`, preserves text within the serialized history budget, aligns the Control UI/config cap policy, and adds focused gateway/UI regression proof.

## Reproduction Assessment

Yes, source-reproducible. A 50k ASCII assistant text can fit below the 128 KiB per-message byte guard, but current main applies the 8,000 gateway default or the Control UI's 4,000 request cap before the serialized byte-budget guards run.

## Solution Assessment

No, the originally proposed one-line cap bump is not the complete fix. The maintainable solution should update the shared projection helper, budget visible prefixes against serialized message size, and decide the Control UI/config cap policy together.
