## Summary

Keep open. Current main does not clearly implement the session-store write-back requested here, but this PR should not merge as-is because the proposed write-backs can clobber concurrent session-store updates and the runtime-dependency deferral can race cold plugin imports.

## What This Changes

The PR writes normalized and maintained session-store data back to `sessions.json` from `loadSessionStore` and defers bundled plugin runtime-dependency installation with `setImmediate`, on top of an old release-train-sized branch diff.

## Best Possible Solution

Salvage this as a narrow current-main fix: persist normalization or explicit maintenance results only through the session-store lock/atomic-write path with stale-snapshot protection, add focused regression tests for disk write-back and concurrent updates, and handle any remaining plugin startup latency in the current plugin dependency-loading architecture rather than deferring the old installer callback blindly.

## Reproduction Assessment

Yes for the PR review findings: the races are visible from the PR head source, where `loadSessionStore` writes a stale parsed snapshot directly and the loader imports the plugin before the `setImmediate` installer can run. The production delay itself is plausible from the PR body, but I did not live-reproduce it in this read-only pass.

## Solution Assessment

No. The requested direction is valid, but this implementation bypasses the session-store concurrency contract and moves npm runtime-dependency installation outside the synchronous lock/error boundary; the safer fix is a locked atomic write-back plus a current-main plugin-dependency design review.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Use the session store lock for write-back:** `src/config/sessions/store-load.ts:138`
  - body: `loadSessionStore` is called by many readers outside `withSessionStoreLock`, so writing the parsed snapshot directly can overwrite a newer `updateSessionStore` save that lands after this read. Persisting normalization or maintenance needs to go through the locked/atomic path, or re-read/compare before replacing the file; the maintenance write-back below repeats the same issue.
  - confidence: 0.87
- **[P2] Keep runtime deps installed before plugin import:** `src/plugins/loader.ts:1511`
  - body: Deferring the installer to `setImmediate` lets `prepareBundledPluginRuntimeLoadRoot` return before missing deps are materialized, and this loader imports the plugin module later in the same tick. On a cold install, plugins that import those deps can fail before the background install runs, and install errors also escape the surrounding load error path.
  - confidence: 0.89

## Security Review

Status: needs_attention

Summary: The diff touches the bundled runtime-dependency installation path and moves npm installation outside the intended lock/error boundary.

Concerns:

- **[medium] Runtime dependency install leaves guarded boundary:** `src/plugins/loader.ts:1511`
  - body: Wrapping the installer in `setImmediate` means `ensureBundledPluginRuntimeDeps` can release its filesystem lock and finish its activity tracking before `installBundledRuntimeDeps` runs. That can allow concurrent uncoordinated installs into the same root and hides install failures from the plugin-load error path.
  - confidence: 0.82
