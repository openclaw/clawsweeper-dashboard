## Summary

Keep open. The underlying plugin context gap is still valid on current main, but this PR is not safe to merge as-is: it adds a new completion hook with missing runner support and broken streaming semantics, omits account identity from one promised agent hook phase, and needs a refresh against the current hook type split.

## What This Changes

The PR changes plugin hook mappers/types/tests and agent runner code to propagate conversation/account identity across hook contexts, and adds an `after_agent_complete` reinject/suppress wrapper around reply-agent execution.

## Best Possible Solution

Refresh the branch against current main and narrow the change to the supported hook contract: add `conversationId` and `accountId` to `src/plugins/hook-types.ts` and `src/agents/harness/hook-context.ts`, wire `params.messageTo` and `params.agentAccountId` through embedded and native hook contexts, preserve the current channel-owned inbound conversation resolver, and add focused regression tests. Drop or redesign `after_agent_complete` unless maintainers explicitly approve a streaming-safe API distinct from `before_agent_finalize` and `reply_dispatch`.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: current main's agent hook context/builders omit `conversationId` and `accountId`, and the PR diff calls a missing `runAfterAgentComplete` method while placing suppress/reinject logic after streaming can already emit blocks.

## Solution Assessment

No. The identity propagation direction is valid, but the PR's new completion wrapper is not the narrowest maintainable solution on current main; the safer path is to wire the missing fields through existing hook phases and handle final-answer revision through the documented hook surfaces unless a new API is deliberately designed.

  - body: The wrapper awaits `runAgentTurnWithFallback()` before invoking `after_agent_complete`, but block streaming can enqueue or send chunks during that inner run. In streaming mode, `suppress` cannot reliably stop delivery and `reinject` can duplicate partial output.
  - confidence: 0.86
- **[P2] Pass accountId through to agent_end hooks:** `src/agents/pi-embedded-runner/run/attempt.ts:2813`
  - body: The PR adds `accountId` to `PluginHookAgentContext`, but the `agent_end` context only adds `conversationId`. Multi-account plugins using `agent_end` would still see `ctx.accountId` as undefined, so the promised consistent account scoping remains false for that phase.
  - confidence: 0.92
- **[P2] Use the resolved origin conversation for native commands:** `src/auto-reply/reply/agent-runner.ts:382`
  - body: This hook context uses `sessionCtx.To` for `conversationId`, while the existing reply routing helpers prefer `OriginatingTo` over `To`. Native Telegram or Discord commands can have synthetic `To` values, so this reports a different conversation than the inbound/embedded paths for the same turn.
