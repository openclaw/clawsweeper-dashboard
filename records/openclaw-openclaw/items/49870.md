- `P2`: The PR addresses a real plugin hook consistency bug but affects a bounded plugin/agent hook surface rather than an urgent production outage.

## Summary

Keep open, but this PR is not merge-ready. It is the canonical same-author fix path for the closed hook-context report, but the branch is conflicting, lacks real behavior proof, and introduces blocking hook-runner, delivery, session-context, and security-boundary problems.

## What This Changes

The PR passes conversationId/accountId through plugin hook contexts and adds an after_agent_complete hook wrapper around reply-agent execution.

## Best Possible Solution

Rebase and split the safe context-field fix from the new completion-hook API: extend the shared hook context helper for accountId/conversationId, and handle any after-agent completion behavior through the existing before_agent_finalize contract or a separately reviewed, gated, pre-delivery API.

## Reproduction Assessment

Yes for the PR-introduced blockers: source inspection of the diff and current main shows the missing hook-runner method, incomplete accountId propagation, post-delivery hook timing, and conversation-access gap. The original multi-account Slack/Discord behavior still lacks contributor real setup proof in this PR.

## Solution Assessment

No. The context consistency goal is valid, but bundling it with an incomplete new after_agent_complete lifecycle API is not the narrowest maintainable fix; a smaller helper-based context patch plus a separately reviewed completion-hook design is safer.

  - confidence: 0.91
- **[P1] Defer delivery before suppressing or reinjecting:** `src/auto-reply/reply/agent-runner-hooks.ts:46-50`
  - body: The wrapper waits for `runAgentTurnWithFallback()` to finish before asking the hook whether to suppress or reinject. In block-streaming paths the inner run can already enqueue or flush chunks, so the advertised pre-delivery decision can be too late and can duplicate or fail to suppress output.
  - confidence: 0.88

  - body: This patch adds `accountId` to the agent hook context contract, but the `agent_end` call site still only passes `conversationId`. Multi-account plugins using `agent_end` for cleanup, metrics, or state will still see no account id in that phase.
  - confidence: 0.93
- **[P2] Use the originating conversation for completion hooks:** `src/auto-reply/reply/agent-runner.ts:381-383`
