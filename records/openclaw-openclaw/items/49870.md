## Summary

Keep open. The identity-context gap still exists on current main, but this PR is not mergeable as-is because it is stale against the current hook contract and introduces a new completion hook with blocking correctness issues.

## What This Changes

The PR changes plugin hook mappers/types/tests and embedded/native agent runner paths to propagate conversation/account identity and adds an `after_agent_complete` suppress/reinject hook loop.

## Best Possible Solution

Refresh the branch against current main and narrow it to the supported hook contract: add `conversationId` and `accountId` to the shared hook context helpers and existing runner call sites, preserve the current channel-owned inbound resolver, and handle final-answer revision through `before_agent_finalize`/`reply_dispatch` unless maintainers explicitly approve a new completion hook.

## Reproduction Assessment

Yes. Source inspection shows current main still omits `conversationId` and `accountId` from agent hook contexts while carrying `messageTo` and `agentAccountId` in runner params, and the PR diff has concrete compile/runtime issues around the new hook runner.

## Solution Assessment

No. Identity propagation is the right direction, but this PR's new `after_agent_complete` wrapper is not the narrowest maintainable fix on current main and is unsafe around streaming delivery.

## Review Findings

Full review comments:

- **[P1] Add the missing after_agent_complete runner:** `src/auto-reply/reply/agent-runner-hooks.ts:65`
  - body: `runAgentTurnWithHooks` calls `hookRunner.runAfterAgentComplete`, but the PR never adds that method to `createHookRunner` or its returned object. Since hook runners are typed from that factory, this branch fails type checking and would throw if the hook path ran at runtime.
  - confidence: 0.94
- **[P1] Run completion decisions before block replies can stream:** `src/auto-reply/reply/agent-runner-hooks.ts:48-52`
  - body: The new wrapper awaits `runAgentTurnWithFallback()` before invoking `after_agent_complete`, but block streaming can enqueue or send chunks during that inner run. In streaming mode, `suppress` cannot reliably stop delivery and `reinject` can duplicate partial output.
  - confidence: 0.86
- **[P2] Pass accountId through to agent_end hooks:** `src/agents/pi-embedded-runner/run/attempt.ts:2813`
  - body: The PR adds `accountId` to the agent hook context contract but the `agent_end` context only supplies `conversationId`. Multi-account plugins using `agent_end` would still see `ctx.accountId` as undefined, so the promised account scoping remains inconsistent.
  - confidence: 0.92
- **[P2] Use the resolved origin conversation for native commands:** `src/auto-reply/reply/agent-runner.ts:382`
  - body: The added native reply-agent hook context derives `conversationId` from `sessionCtx.To`, while existing origin-routing helpers prefer `OriginatingTo` for native Telegram or Discord commands. That can report a synthetic command target instead of the real conversation and break cross-phase correlation.
  - confidence: 0.84
