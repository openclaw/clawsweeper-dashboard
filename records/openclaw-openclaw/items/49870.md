## Summary

Keep open. Current main still does not fully provide the requested accountId/conversationId consistency, and the linked issue was closed specifically so this PR remains the canonical review point; however, this patch is not mergeable as-is because the new hook runner is missing and the new post-run hook semantics break for block-streaming replies.

## What This Changes

The PR passes conversationId/accountId through more plugin hook contexts, removes generic inbound conversationId derivation, adds an after_agent_complete hook loop with suppress/reinject behavior, and updates message hook mapper coverage.

## Best Possible Solution

Replace or rework this PR against current main so existing agent hook contexts get accountId/conversationId through the shared hook-context helper, inbound claim uses a consistent channel-owned resolver, and any post-final-answer hook is designed around the current before_agent_finalize/delivery pipeline with docs, changelog, and regression coverage.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path for the PR defects: typecheck/build would reject runAfterAgentComplete on the hook runner, and block-streaming replies can be emitted before the proposed after_agent_complete decision runs.

## Solution Assessment

No. The context consistency goal is valid, but this patch adds a new hook after the delivery path has already been allowed to stream and misses required runner/context wiring; current main’s hook-agent-context helper and before_agent_finalize/next-turn seams are safer integration points.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Run the hook before streaming replies can escape:** `src/auto-reply/reply/agent-runner-hooks.ts:48-52`
  - body: `after_agent_complete` is supposed to suppress or reinject a response, but this wrapper only calls it after `runAgentTurnWithFallback()` returns. With block streaming enabled, the inner run can already enqueue or send chunks through the block reply pipeline, so `suppress` cannot reliably stop delivery and `reinject` can duplicate partial output.
  - confidence: 0.93
- **[P2] Implement the after_agent_complete runner method:** `src/auto-reply/reply/agent-runner-hooks.ts:65-82`
  - body: This calls `hookRunner.runAfterAgentComplete(...)`, but the PR only registers the hook name and handler type; it does not add a corresponding method to `createHookRunner` or return it. Since the runner type comes from `createHookRunner`, this will fail typecheck and would throw at runtime if reached.
  - confidence: 0.96
- **[P2] Pass accountId into agent_end hooks:** `src/agents/pi-embedded-runner/run/attempt.ts:2813-2816`
  - body: The PR adds `accountId` to `PluginHookAgentContext`, but the `agent_end` call site in the embedded attempt runner still only passes `conversationId`. Multi-account plugins observing `agent_end` would still see `ctx.accountId` as undefined, so the new consistency contract remains false for that phase.
  - confidence: 0.94
- **[P2] Use the originating target for post-run hook context:** `src/auto-reply/reply/agent-runner.ts:376-383`
  - body: For native Telegram and Discord commands, `sessionCtx.To` can be a synthetic target such as `slash:...`; the real conversation is carried in `OriginatingTo`. Building `channelKey` and `conversationId` from `sessionCtx.To` makes this new hook disagree with the inbound and embedded-runner contexts for the same turn.
  - confidence: 0.9
