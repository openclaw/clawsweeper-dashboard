## Summary

Keep this PR open: the linked issue was closed specifically in favor of this PR, current main still lacks the requested agent hook `conversationId`/`accountId` contract, and the branch has blocking correctness, privacy, streaming, and proof gaps before it can merge.

## What This Changes

The branch passes conversation/account identifiers through plugin hook contexts, removes inbound-claim conversation derivation, and adds an `after_agent_complete` reply-agent hook wrapper.

## Best Possible Solution

Rebase and split the useful context-field fix from any new completion-hook API, then land only a privacy-gated, streaming-safe hook design with focused tests and real runtime proof.

## Reproduction Assessment

Yes. The review blockers are source-reproducible from the PR diff and current main: the branch calls a missing hook-runner method, omits `accountId` from one `agent_end` context, exposes raw response text without the current conversation-access gate, and runs suppress/reinject after streamable replies can be queued.

## Solution Assessment

No. The context-consistency direction is useful, but combining it with a new raw-response completion hook is not the narrowest maintainable fix while privacy, runner wiring, streaming semantics, and real behavior proof remain incomplete.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Gate after_agent_complete as conversation access:** `src/plugins/types.ts:1546`
  - body: The new hook exposes the full assistant `response` to plugin handlers, but the patch only registers the hook name/type and does not add it to the conversation-content gate. Non-bundled plugins could read raw conversation text without `allowConversationAccess=true`.
  - confidence: 0.89
- **[P1] Run completion hooks before streaming replies can send:** `src/auto-reply/reply/agent-runner-hooks.ts:47-51`
  - body: The wrapper awaits `runAgentTurnWithFallback()` before checking suppress/reinject. With block streaming, that inner run can already enqueue or send reply chunks, so `suppress` is too late and `reinject` can duplicate or strand partial output.
  - confidence: 0.86
- **[P2] Add runAfterAgentComplete to the hook runner:** `src/auto-reply/reply/agent-runner-hooks.ts:65`
  - body: This call is made on `HookRunner`, but the patch does not add a matching method to `createHookRunner()` or its returned object. The branch will fail type checking and would throw at runtime if forced through.
  - confidence: 0.94
- **[P2] Pass accountId through agent_end contexts:** `src/agents/pi-embedded-runner/run/attempt.ts:2813`
  - body: The PR adds `accountId` to `PluginHookAgentContext`, but the `agent_end` call site only adds `conversationId`. Multi-account plugins observing `agent_end` still cannot distinguish the bot account.
  - confidence: 0.91
- **[P2] Use the originating target for hook keys:** `src/auto-reply/reply/agent-runner.ts:379`
  - body: For native Telegram/Discord command turns, `sessionCtx.To` can be a synthetic command target while the real conversation lives in `OriginatingTo`. Building the new hook key from `sessionCtx.To` makes this phase disagree with inbound contexts.
  - confidence: 0.78
