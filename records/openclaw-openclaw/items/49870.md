## Summary

Keep open: current main still has the hook-context consistency gap, but this branch is not safe to merge because it mixes the useful context fix with a new raw-response hook that lacks runner support, conversation-access gating, and streaming-safe semantics.

## What This Changes

This PR passes inbound-claim conversation IDs through directly, adds conversation/account fields to agent hook contexts, and adds an `after_agent_complete` wrapper around reply-agent runs.

## Best Possible Solution

Rebase against current main and narrow the fix through the shared hook-context helpers with focused tests; either drop `after_agent_complete` or add it as a fully gated, documented, runner-backed, streaming-safe hook.

## Reproduction Assessment

Yes for the PR defects by source inspection: the proposed code calls an unimplemented runner method, misses `accountId` in one agent hook context, and runs suppress/reinject logic after block replies can stream. The underlying current-main context mismatch is also source-reproducible by comparing the inbound and message hook mappers.

## Solution Assessment

No. The context-consistency direction is useful, but this branch is not the narrowest maintainable fix while it also introduces a raw-response completion hook without privacy, runner, docs, and streaming semantics settled.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Gate after_agent_complete as conversation access:** `src/plugins/types.ts:1546`
  - body: The new hook exposes the full agent `response` to plugin handlers, but the patch only registers the hook name and handler type. Non-bundled plugins could receive raw conversation content without the `allowConversationAccess` opt-in that protects `llm_output`, `before_agent_finalize`, and `agent_end`.
  - confidence: 0.89
- **[P1] Run completion hooks before block replies stream:** `src/auto-reply/reply/agent-runner-hooks.ts:47-51`
  - body: `after_agent_complete` is supposed to suppress or reinject a response, but this wrapper invokes it only after `runAgentTurnWithFallback()` returns. In block-streaming mode, reply chunks can already be enqueued or sent, so suppression arrives too late and reinjection can duplicate partial output.
  - confidence: 0.86
- **[P2] Add runAfterAgentComplete to the hook runner:** `src/auto-reply/reply/agent-runner-hooks.ts:65`
  - body: This call is made on `HookRunner`, but the patch does not add a matching method to `createHookRunner()` or its returned object. The branch will fail type checking and would throw at runtime if forced through.
  - confidence: 0.94
- **[P2] Pass accountId to agent_end contexts:** `src/agents/pi-embedded-runner/run/attempt.ts:2813`
  - body: The PR adds `accountId` to `PluginHookAgentContext`, but the `agent_end` call site only adds `conversationId`. Multi-account plugins observing `agent_end` still cannot distinguish the bot account, so the promised consistency is incomplete.
  - confidence: 0.91
- **[P2] Use the originating target for native commands:** `src/auto-reply/reply/agent-runner.ts:379`
  - body: For native Telegram/Discord command turns, `sessionCtx.To` can be a synthetic command target while the real conversation is in `OriginatingTo`. Building the new hook key from `sessionCtx.To` makes this phase disagree with inbound and embedded-agent contexts for the same turn.
  - confidence: 0.78
