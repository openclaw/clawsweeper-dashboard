## Summary

Keep open. The linked hook-context bug is still source-reproducible on current main and this PR is the canonical same-author fix path, but the branch is not mergeable as-is because it adds an incomplete `after_agent_complete` path, leaves one agent hook context inconsistent, and lacks real behavior proof.

## What This Changes

The PR changes plugin hook mapping and agent runner hook contexts so `conversationId` is passed consistently and `accountId` is exposed, while also adding an `after_agent_complete` hook wrapper.

## Best Possible Solution

Rebase onto current main, split or align `after_agent_complete` with https://github.com/openclaw/openclaw/pull/48100, then land a focused hook-context fix with regression coverage and redacted multi-account runtime proof.

## Reproduction Assessment

Yes, by source inspection. Current main still has an inbound-claim fallback that rewrites conversation ids and an agent hook context/helper that does not expose `conversationId` or `accountId`.

## Solution Assessment

No. The context-consistency direction is right, but this branch mixes it with an incomplete new lifecycle hook and needs to either remove that scope or make it match the existing hook-runner and prompt-injection policy contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Add the missing after_agent_complete runner method:** `src/auto-reply/reply/agent-runner-hooks.ts:64`
  - body: `getGlobalHookRunner()` returns `HookRunner = ReturnType<typeof createHookRunner>`, but this PR calls `hookRunner.runAfterAgentComplete(...)` without adding that method to `createHookRunner` or its return object. The branch will fail typechecking, and would throw at runtime if the hook path were reached.
  - confidence: 0.98
- **[P1] Gate reinjection as prompt injection:** `src/plugins/types.ts:1393`
  - body: The new hook can return `reinject` with `injectContext`, which rewrites the next prompt, but the PR only registers `after_agent_complete` as a normal hook. Plugins configured with `hooks.allowPromptInjection=false` would still be able to modify prompts once the runner method is added, so this hook needs to be classified with the prompt-injection hooks.
  - confidence: 0.86
- **[P1] Run suppress/reinject before streaming chunks can flush:** `src/auto-reply/reply/agent-runner-hooks.ts:48-52`
  - body: The wrapper waits for `runAgentTurnWithFallback()` to finish before firing `after_agent_complete`, but block-streaming runs can already enqueue or flush reply chunks inside that call. In streaming mode, a plugin returning `suppress` cannot reliably stop the answer and `reinject` can duplicate already-sent partial output.
  - confidence: 0.88
- **[P2] Pass accountId through to agent_end hooks:** `src/agents/pi-embedded-runner/run/attempt.ts:2813`
  - body: The PR adds `accountId` to the shared agent hook context for multi-account isolation, but the `agent_end` call site only adds `conversationId`. Plugins that do cleanup, metrics, or state updates in `agent_end` still cannot distinguish two bot accounts in the same conversation.
  - confidence: 0.93
- **[P2] Use the originating target for after_agent_complete:** `src/auto-reply/reply/agent-runner.ts:384`
  - body: For native commands, `sessionCtx.To` can be a synthetic command target while `OriginatingTo` carries the real conversation. Building `channelKey` and `conversationId` from `sessionCtx.To` makes this new hook disagree with the inbound and embedded-agent contexts for the same turn.
  - confidence: 0.86
