## Summary

Keep open: current main still does not fully solve the hook context problem, but this PR is not mergeable as-is because it adds a new completion hook without runner support, privacy gating, or streaming-safe semantics.

## What This Changes

The PR attempts to make plugin hook conversation/account context consistent and adds an `after_agent_complete` hook loop around reply agent runs.

## Best Possible Solution

Rework against current main: add `accountId` and a stable conversation field through the shared agent hook context helper with focused tests, keep inbound normalization in channel/plugin resolvers, and either drop `after_agent_complete` or add it with runner support, conversation-access gating, docs, and streaming-safe behavior.

## Reproduction Assessment

Yes for the PR defects by source inspection: the proposed code calls a hook runner method that is not implemented, omits `accountId` from one agent hook context, and runs suppression/reinject logic after block streaming can already send output. I did not run the suite because this was a read-only review and `pnpm` was unavailable.

## Solution Assessment

No. The context-consistency fix is useful, but this branch mixes it with a new completion-hook API that lacks privacy gating, hook runner support, and a streaming-safe insertion point.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Gate after_agent_complete as conversation access:** `src/plugins/types.ts:1546`
  - body: The new hook hands the full agent response to plugin handlers, but the patch only registers the hook name and handler type. Since it is not added to the conversation-access gate, a non-bundled plugin could read raw replies without the `allowConversationAccess` opt-in that protects `llm_output`, `before_agent_finalize`, and `agent_end`.
  - confidence: 0.89
- **[P1] Run completion hooks before block replies stream:** `src/auto-reply/reply/agent-runner-hooks.ts:47-51`
  - body: `runAgentTurnWithFallback` receives the live `blockReplyPipeline`, and block streaming can enqueue or directly send text before this wrapper calls `after_agent_complete`. A plugin returning `suppress` or `reinject` would arrive after partial output is already delivered, so the new hook semantics break when block streaming is enabled.
  - confidence: 0.86
- **[P2] Add runAfterAgentComplete to the hook runner:** `src/auto-reply/reply/agent-runner-hooks.ts:65`
  - body: This call is made on `HookRunner`, but the patch does not add a corresponding method to `createHookRunner` or its returned object. The new file will fail type checking and would throw at runtime if forced through.
  - confidence: 0.94
- **[P2] Pass accountId to agent_end contexts:** `src/agents/pi-embedded-runner/run/attempt.ts:2813`
  - body: The PR adds `accountId` to `PluginHookAgentContext`, but this `agent_end` context only adds `conversationId`. Multi-account plugins observing `agent_end` still cannot distinguish the bot account, so the promised consistency remains false for that phase.
  - confidence: 0.91
- **[P2] Use the originating target for native commands:** `src/auto-reply/reply/agent-runner.ts:379`
  - body: For native Telegram/Discord command turns, `sessionCtx.To` can be a synthetic command target while the real conversation is carried by the originating target. Building the new hook key from `sessionCtx.To` makes this phase disagree with inbound and embedded-agent hook contexts for the same turn.
  - confidence: 0.78

## Security Review
