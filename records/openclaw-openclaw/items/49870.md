## Summary

Keep open: the PR addresses a real plugin hook consistency problem, but the proposed branch is not safe to merge against current main because it adds a response-bearing hook without the matching runner implementation, has pre-delivery/block-streaming semantics problems, and is now behind the current hook architecture.

## What This Changes

The PR makes plugin hook contexts expose more consistent conversation/account identifiers and adds an after_agent_complete reply hook with reinject/suppress behavior.

## Best Possible Solution

Rework the context consistency fix on current main, using the existing hook-types/hooks registry and conversation-access gates, and decide whether the existing before_agent_finalize hook should cover the response mutation use case instead of adding a second post-run hook.

## Reproduction Assessment

Yes. Source inspection plus the PR diff show a high-confidence failure path: the new code calls runAfterAgentComplete, but the hook runner does not define or return that method, and the block-streaming ordering issue follows from current reply-delivery enqueue behavior.

## Solution Assessment

No. The current PR is not the best mergeable solution because it introduces a new hook with incomplete runner wiring, missing context fields, unresolved access gating, and delivery timing that conflicts with block streaming; a current-main rework should align with before_agent_finalize or a properly gated pre-delivery hook.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Implement the after_agent_complete runner method:** `src/auto-reply/reply/agent-runner-hooks.ts:65`
  - body: The new wrapper calls hookRunner.runAfterAgentComplete(), but the PR only registers the hook name/types and never adds the method to createHookRunner's returned object. Because HookRunner is derived from createHookRunner, this is a TypeScript error and would also throw at runtime if reached.
  - confidence: 0.96
- **[P1] Run the hook before block replies can stream:** `src/auto-reply/reply/agent-runner-hooks.ts:50`
  - body: The hook fires only after runAgentTurnWithFallback() returns, but block streaming can enqueue or send chunks during that inner run. A plugin returning suppress or reinject therefore cannot reliably prevent already-delivered partial replies and can duplicate output.
  - confidence: 0.9
- **[P1] Gate after_agent_complete as a conversation hook:** `src/plugins/types.ts:1393`
  - body: The new hook exposes the final response text and can alter delivery, but the PR only adds it to the hook name/handler list. It should be treated like the existing conversation hooks so non-bundled plugins need explicit allowConversationAccess before receiving conversation content.
  - confidence: 0.82
- **[P2] Pass accountId through to agent_end hooks:** `src/agents/pi-embedded-runner/run/attempt.ts:2813`
  - body: The PR adds accountId to PluginHookAgentContext but the agent_end call site only adds conversationId. Multi-account plugins using agent_end for cleanup or state will still see ctx.accountId as undefined, so the stated consistency contract remains false for that phase.
  - confidence: 0.91
- **[P2] Use the originating target for reply hook context:** `src/auto-reply/reply/agent-runner.ts:351`
  - body: The new channelKey/conversationId construction uses sessionCtx.To, which can be a synthetic native-command target. For Telegram/Discord native commands this can diverge from the real OriginatingTo conversation, so plugins correlating hook phases still miss the same turn.
  - confidence: 0.84
