## Summary

Keep open. Current main only partially addresses the inbound conversation-normalization half through channel-owned resolvers; the public agent hook context and embedded/native hook call sites still do not expose `conversationId` or `accountId`. The PR still targets a real plugin contract gap, but it is not safe to merge as-is because the added `after_agent_complete` surface is incomplete and conflicts with current finalization/dispatch behavior.

## What This Changes

The PR changes plugin hook mappers/types and agent runners to pass raw inbound `conversationId`, add `conversationId`/`accountId` to agent hook contexts, and introduce an `after_agent_complete` suppress/reinject wrapper around reply-agent execution.

## Best Possible Solution

Refresh the PR against current main and narrow it to the supported plugin hook contract: add `conversationId` and `accountId` to `src/plugins/hook-types.ts` and `src/agents/harness/hook-context.ts`, wire `params.messageTo` and `params.agentAccountId` through embedded and native lifecycle hook contexts, keep the channel-owned inbound resolver now on main, and add focused regression tests. Drop `after_agent_complete` unless maintainers explicitly approve a streaming-safe design that is distinct from `before_agent_finalize` and `reply_dispatch`.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: inspect current `PluginHookAgentContext` and the embedded/native hook context builders and the requested fields are absent; inspect the PR diff and `runAfterAgentComplete` is called without a runner implementation.

## Solution Assessment

No. The identity propagation direction is valid, but the PR's added `after_agent_complete` wrapper is not the narrowest maintainable fix on current main; the safer path is to wire the missing fields through existing hook phases and redesign or omit the new hook.
