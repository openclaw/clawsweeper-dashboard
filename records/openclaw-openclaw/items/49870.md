## Summary

Keep open: current main still shows the hook-context inconsistency this PR is trying to fix, but the branch is not mergeable as-is because it couples the context fix to an incomplete and ungated response/reinject hook and still lacks real behavior proof.

## What This Changes

The PR updates plugin hook mapper and agent runner contexts to pass conversationId/accountId consistently and adds an after_agent_complete hook loop with suppress/reinject behavior.

## Best Possible Solution

Split the context-consistency fix from lifecycle-hook API work, land accountId/conversationId through the current shared hook-context helper with focused regression coverage and runtime proof, and handle response revision through before_agent_finalize or a separate API proposal.

## Reproduction Assessment

Yes, from source inspection: current main still derives inbound_claim conversationId through resolveInboundConversation while message hooks use canonical.conversationId, and PluginHookAgentContext still lacks accountId/conversationId.

## Solution Assessment

No. The context-consistency direction is sound, but this branch is not the best mergeable shape because it couples that fix to an incomplete lifecycle hook with unresolved security and delivery semantics.

Overall confidence: 0.93

Full review comments:

- **[P1] Add the missing after_agent_complete runner:** `src/auto-reply/reply/agent-runner-hooks.ts:65-82`
  - body: `runAgentTurnWithHooks` calls `hookRunner.runAfterAgentComplete(...)`, but the PR does not add that method to `createHookRunner` or its returned object. TypeScript will reject the call, and registered hooks would throw at runtime.
  - confidence: 0.98
- **[P1] Gate reinjection as prompt injection:** `src/plugins/types.ts:1554-1555`
  - body: The new hook result can set `reinject` with `injectContext`, which appends to a later prompt, but the hook is registered as an ordinary hook. Plugins with prompt injection disabled could still mutate prompts through this path.
  - confidence: 0.9
- **[P1] Require conversation access for completion hooks:** `src/plugins/types.ts:1549`
  - body: The new event exposes the assistant `response`, but the hook is not classified with current conversation-access gated hooks. A non-bundled plugin could observe conversation content without the explicit opt-in required by comparable hooks.
  - confidence: 0.91
- **[P1] Run completion hooks before block replies can stream:** `src/auto-reply/reply/agent-runner-hooks.ts:48-52`
  - body: The wrapper waits for `runAgentTurnWithFallback()` to finish before calling the hook. In block-streaming setups, that inner call can already enqueue or send chunks, so later `suppress` or `reinject` results cannot reliably retract or replace content.
  - confidence: 0.88
- **[P2] Pass accountId through to agent_end hooks:** `src/agents/pi-embedded-runner/run/attempt.ts:2813-2817`
  - body: The PR adds `accountId` to the shared agent hook context goal, but the `agent_end` call site still omits it. Plugins using `agent_end` for cleanup, metrics, or state updates would still be unable to distinguish multiple bot accounts.
  - confidence: 0.93
- **[P2] Use the originating target for completion context:** `src/auto-reply/reply/agent-runner.ts:382`
  - body: For native commands, `sessionCtx.To` can be a synthetic command target while `OriginatingTo` carries the real conversation. Building this hook's `conversationId` and `channelKey` from `sessionCtx.To` can make it disagree with other hook phases for the same turn.
  - confidence: 0.84
