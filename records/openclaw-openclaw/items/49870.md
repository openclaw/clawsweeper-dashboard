Full review comments:

- **[P1] Add the missing completion-hook runner:** `src/auto-reply/reply/agent-runner-hooks.ts:66-82`
  - body: `runAgentTurnWithHooks` calls `hookRunner.runAfterAgentComplete(...)`, but the PR does not add that method to `createHookRunner` or its returned object. TypeScript will reject the call, and a registered hook would be undefined at runtime.
  - confidence: 0.98
- **[P1] Run completion decisions before block replies stream:** `src/auto-reply/reply/agent-runner-hooks.ts:47-52`
  - body: The wrapper waits for `runAgentTurnWithFallback()` to finish before calling the hook. In block-streaming setups, the inner runner can already enqueue or send chunks, so later `suppress` or `reinject` cannot reliably retract or replace content.
  - confidence: 0.88
- **[P1] Require access gates for completion hooks:** `src/plugins/types.ts:1538-1553`
  - body: The new hook exposes assistant response text and can reinject prompt context, but the PR registers it as an ordinary hook. Non-bundled plugins should need the same conversation-access and prompt-injection gates used for comparable sensitive hooks.
  - confidence: 0.9
- **[P2] Pass accountId through to agent_end hooks:** `src/agents/pi-embedded-runner/run/attempt.ts:2813-2817`
  - body: The PR adds `accountId` to the shared agent hook context goal, but the `agent_end` call site still omits it. Plugins using `agent_end` for cleanup, metrics, or state updates would still be unable to distinguish multiple bot accounts.
  - confidence: 0.93
- **[P2] Use the originating target for completion context:** `src/auto-reply/reply/agent-runner.ts:380-383`
  - body: For native commands, `sessionCtx.To` can be a synthetic command target while `OriginatingTo` carries the real conversation. Building this hook's `conversationId` and `channelKey` from `sessionCtx.To` can make it disagree with other hook phases for the same turn.
  - confidence: 0.84
