## Summary

Keep open: current main still has the hook context inconsistency, but this PR is not mergeable as-is because it lacks real behavior proof and mixes the context fix with incomplete, security-sensitive `after_agent_complete` semantics.

## What This Changes

The branch passes `conversationId` and `accountId` through plugin hook contexts and adds an `after_agent_complete` reply-agent wrapper with suppress/reinject behavior.

## Best Possible Solution

Split or refit the context-consistency fix onto current hook-type architecture, align or remove the `after_agent_complete` portion in favor of https://github.com/openclaw/openclaw/pull/48100, and add regression coverage plus redacted runtime proof.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main still normalizes `inbound_claim` conversation IDs differently and the agent hook context helper omits `conversationId` and `accountId`. I did not run live Slack or Discord proof under the read-only review constraint.

## Solution Assessment

No. The context-consistency direction is sound, but this branch is not the best mergeable solution because it combines the bug fix with an incomplete `after_agent_complete` feature that needs current-main refit, policy gating, and streaming semantics work.

  - body: `runAgentTurnWithHooks` calls `hookRunner.runAfterAgentComplete(...)`, but the PR does not add that method to `createHookRunner` or its returned object. This will fail typechecking and would throw once an `after_agent_complete` hook is registered.

  - body: The new hook result can set `reinject` with `injectContext`, which rewrites a later prompt, but the PR registers the hook as an ordinary hook. Plugins with prompt injection disabled could still mutate prompts through this path.
  - confidence: 0.9
- **[P1] Require conversation access for response hooks:** `src/plugins/types.ts:1549`
  - body: The new event exposes the assistant `response`, but the hook is not classified with current conversation-access gated hooks. A non-bundled plugin could observe conversation content without the explicit `allowConversationAccess` opt-in required by comparable hooks.
  - confidence: 0.91
- **[P1] Prevent streaming output before hook verdicts:** `src/auto-reply/reply/agent-runner-hooks.ts:48-52`
  - body: The wrapper waits for `runAgentTurnWithFallback()` to finish before calling the hook, but block streaming can enqueue or send chunks inside that call. A later `suppress` or `reinject` result cannot reliably retract content already delivered to the channel.
  - confidence: 0.88
- **[P2] Pass accountId through to agent_end hooks:** `src/agents/pi-embedded-runner/run/attempt.ts:2808-2817`
  - body: The PR adds `accountId` to the shared agent hook context goal, but the `agent_end` call site still only supplies `conversationId`. Plugins using `agent_end` for cleanup, metrics, or state updates would still be unable to distinguish multiple bot accounts.
  - confidence: 0.93
- **[P2] Use the originating target for completion hook context:** `src/auto-reply/reply/agent-runner.ts:380-383`
  - body: For native commands, `sessionCtx.To` can be a synthetic command target while `OriginatingTo` carries the real conversation. Building `channelKey` and `conversationId` from `sessionCtx.To` makes this new hook disagree with inbound and embedded-agent contexts for the same turn.
  - confidence: 0.84
