## Summary

Keep open: the underlying context-consistency bug is still present on current main, but this PR is not mergeable as-is because it mixes that fix with an incomplete and policy-sensitive `after_agent_complete` API, misses one agent hook account path, and lacks real behavior proof.

## What This Changes

The PR changes plugin hook mappers and agent/reply runners to pass `conversationId` and `accountId` through hook contexts, and adds an `after_agent_complete` hook with suppress/reinject behavior.

## Best Possible Solution

Split the context-consistency fix from lifecycle-hook API work, land `conversationId`/`accountId` through the shared hook-context helper with focused regression coverage and runtime proof, and handle response revision through `before_agent_finalize` or a separate reviewed API proposal.

## Reproduction Assessment

Yes. Source inspection on current main shows `inbound_claim` still resolves and strips conversation IDs differently from `message_received`, and agent hook contexts still omit `conversationId` and `accountId`.

## Solution Assessment

No. The context-consistency direction is sound, but this branch is not the best mergeable solution because it combines that fix with an incomplete completion hook that needs policy gating, delivery-order semantics, and a current-main API refit.

Overall confidence: 0.93

Full review comments:

- **[P1] Add the missing after_agent_complete runner:** `src/auto-reply/reply/agent-runner-hooks.ts:65-82`
  - body: `runAgentTurnWithHooks` calls `hookRunner.runAfterAgentComplete(...)`, but this PR does not add that method to `createHookRunner` or its returned object. This will fail typechecking and would throw once an `after_agent_complete` hook is registered.
  - confidence: 0.98
- **[P1] Gate reinjection as prompt injection:** `src/plugins/types.ts:1554-1555`
  - body: The new hook result can set `reinject` with `injectContext`, which appends to a later prompt, but the PR registers the hook as an ordinary hook. Plugins with prompt injection disabled could still mutate prompts through this path.
  - confidence: 0.9
- **[P1] Require conversation access for response hooks:** `src/plugins/types.ts:1549`
  - body: The new event exposes the assistant `response`, but the hook is not classified with current conversation-access gated hooks. A non-bundled plugin could observe conversation content without the explicit `allowConversationAccess` opt-in required by comparable hooks.
  - confidence: 0.91
- **[P1] Defer streaming until after completion hook verdicts:** `src/auto-reply/reply/agent-runner-hooks.ts:48-52`
  - body: The wrapper waits for `runAgentTurnWithFallback()` to finish before calling the hook, but block streaming can enqueue or send chunks inside that call. A later `suppress` or `reinject` result cannot reliably retract content already delivered to the channel.
  - confidence: 0.88
- **[P2] Pass accountId through to agent_end hooks:** `src/agents/pi-embedded-runner/run/attempt.ts:2813-2817`
  - body: The PR adds `accountId` to the shared agent hook context goal, but the `agent_end` call site still omits it. Plugins using `agent_end` for cleanup, metrics, or state updates would still be unable to distinguish multiple bot accounts.
  - confidence: 0.93
- **[P2] Use the originating target for completion hook context:** `src/auto-reply/reply/agent-runner.ts:382`
  - body: For native commands, `sessionCtx.To` can be a synthetic command target while `OriginatingTo` carries the real conversation. Building this new hook's `conversationId` from `sessionCtx.To` makes it disagree with inbound and embedded-agent contexts for the same turn.
  - confidence: 0.84
