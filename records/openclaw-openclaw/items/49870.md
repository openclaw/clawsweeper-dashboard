## Summary

Keep open: the linked report is already routed to this PR and current main still lacks the full requested hook identifiers, but the branch is not merge-ready because its new after-agent hook is incomplete and some context wiring remains inconsistent.

## What This Changes

The PR passes raw inbound `conversationId`, adds `conversationId`/`accountId` to agent hook contexts, and introduces an `after_agent_complete` hook wrapper around reply-agent execution.

## Best Possible Solution

Land a smaller, current-main-based fix that exposes stable account/conversation identifiers across all existing agent hook phases, and handle any new pre-delivery response hook as a separately designed plugin API with streaming-safe semantics and live proof.

## Reproduction Assessment

Yes, source-reproducible: current main still lacks `accountId`/`conversationId` on `PluginHookAgentContext`, and inbound-claim conversation IDs still pass through separate resolver/fallback normalization. I did not run tests because this review is read-only and tests may create artifacts.

## Solution Assessment

No. Exposing consistent identifiers is the right direction, but this PR mixes that bug fix with a new `after_agent_complete` hook whose runner, streaming, and origin-context semantics are not correct enough to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Implement the after-agent hook runner:** `src/auto-reply/reply/agent-runner-hooks.ts:65`
  - body: This calls `hookRunner.runAfterAgentComplete`, but the PR only registers the hook name/types. `createHookRunner` on the PR head does not implement or return that method, so the branch will fail type checking and would throw when a handler is registered.
  - confidence: 0.96
- **[P1] Gate streamed replies before suppressing or reinjecting:** `src/auto-reply/reply/agent-runner-hooks.ts:48-52`
  - body: The wrapper waits for `runAgentTurnWithFallback` to return before running `after_agent_complete`, but that inner call receives the live block reply pipeline and can already enqueue/flush streamed chunks. A hook returning `suppress` cannot reliably stop output, and `reinject` can duplicate partial replies.
  - confidence: 0.9
- **[P2] Derive after-agent conversation from the origin target:** `src/auto-reply/reply/agent-runner.ts:382`
  - body: For native commands, `sessionCtx.To` can be a synthetic command target while the real chat is in `OriginatingTo`. Using `sessionCtx.To` for both `channelKey` and `conversationId` makes this new hook disagree with the embedded-agent origin routing used for the same turn.
  - confidence: 0.86
- **[P2] Pass accountId through to agent_end hooks:** `src/agents/pi-embedded-runner/run/attempt.ts:2816`
  - body: The PR adds `accountId` to the shared agent hook context and supplies it for `before_prompt_build`, but the `agent_end` context still omits it. Plugins using `agent_end` for cleanup or metrics would still be unable to scope multi-account conversations.
  - confidence: 0.94
