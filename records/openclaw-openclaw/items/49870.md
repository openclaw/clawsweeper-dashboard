## Summary

Keep open: current main still shows the linked hook-context bug, but this PR is not merge-ready because it couples the fix to an incomplete completion hook with policy and delivery-semantics issues, and it lacks required real behavior proof.

## What This Changes

The PR passes inbound `conversationId` through directly, adds `conversationId` and `accountId` to agent hook contexts, and introduces an `after_agent_complete` hook loop with suppress/reinject behavior.

## Best Possible Solution

Split the context-consistency bug fix from lifecycle-hook API work, land account/conversation fields through the current shared hook-context helpers with focused regression coverage and runtime proof, and handle response revision through the existing finalize hook surface or a separate API proposal.

## Reproduction Assessment

Yes, by source inspection: current `main` still maps `message_received` from `canonical.conversationId` while `inbound_claim` goes through `resolveInboundConversation`, and the current agent hook context type/helper omit `conversationId` and `accountId`. I did not run a live Slack or Discord scenario in this read-only review.

## Solution Assessment

No. The context-consistency direction is a valid bug fix, but this PR is not the best mergeable shape because it couples that fix to a new `after_agent_complete` API with unresolved security and delivery semantics.

Full review comments:

- **[P1] Add the missing after-agent-complete runner:** `src/auto-reply/reply/agent-runner-hooks.ts:65-82`
  - body: `runAgentTurnWithHooks` calls `hookRunner.runAfterAgentComplete(...)`, but the PR only registers the hook name/type and never adds a corresponding method to `createHookRunner` or its return object. `HookRunner` is inferred from that factory, so this call will fail type checking and registered hooks would throw at runtime.
  - confidence: 0.98
- **[P1] Gate reinjection as prompt injection:** `src/plugins/types.ts:1554-1555`
  - body: The new hook result can set `reinject` with `injectContext`, which is appended to the next agent prompt, but the hook is not classified with the existing prompt-injection policy gates. A plugin with prompt injection disabled could still mutate later prompts through this new path.
  - confidence: 0.9
- **[P1] Require conversation access for completion hooks:** `src/plugins/types.ts:1549`
  - body: The new event exposes the assistant response text, but the hook is registered as an ordinary hook rather than one requiring the existing conversation-access opt-in for non-bundled plugins. That lets plugins observe conversation content without the explicit policy used by comparable hooks.
  - confidence: 0.91
- **[P1] Run completion hooks before block replies can stream:** `src/auto-reply/reply/agent-runner-hooks.ts:48-52`
  - body: The wrapper waits for `runAgentTurnWithFallback()` to finish before asking plugins whether to suppress or reinject. With block streaming enabled, the inner run can already enqueue or send chunks, so a later suppress result cannot reliably prevent visible output and reinjection can duplicate partial content.
  - confidence: 0.88
- **[P2] Pass accountId through to agent_end hooks:** `src/agents/pi-embedded-runner/run/attempt.ts:2813-2817`
  - body: The PR adds `accountId` to the shared agent hook context goal, but the `agent_end` call site still only supplies `conversationId`. Plugins using `agent_end` for cleanup, metrics, or per-account state still cannot distinguish multiple bot accounts.
  - confidence: 0.93
- **[P2] Use the originating target for completion context:** `src/auto-reply/reply/agent-runner.ts:382`
  - body: For native commands, `sessionCtx.To` can be a synthetic command target while `OriginatingTo` carries the real conversation. Building `channelKey` and `conversationId` from `sessionCtx.To` makes this hook disagree with other phases for the same turn.
  - confidence: 0.84
