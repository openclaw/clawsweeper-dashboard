## Summary

Keep open. Current main does not fully implement the requested account/conversation hook-context contract, and this PR remains unsafe to merge because it adds an ungated raw-response completion hook with missing runner and streaming semantics.

## What This Changes

The PR changes plugin hook mappers and agent hook contexts to expose more consistent conversation/account identifiers, while adding an `after_agent_complete` wrapper around reply-agent turns.

## Best Possible Solution

Rebase against the current hook SDK and either land a narrow, tested context-field contract or design a separate gated, documented completion hook with streaming-safe semantics.

## Reproduction Assessment

Yes. The PR defects are source-reproducible from the diff and current hook runner: the branch calls an unimplemented runner method, omits `accountId` in one `agent_end` context, and runs suppress/reinject logic after streamable replies can be queued.

## Solution Assessment

No. The context-consistency direction is useful, but combining it with a new raw-response completion hook is not the narrowest maintainable fix while privacy, runner wiring, and streaming behavior are incomplete.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Gate after_agent_complete as conversation access:** `src/plugins/types.ts:1546`
  - body: The new hook exposes the full agent `response` to plugin handlers, but the patch only registers the hook name/type and does not add it to the conversation-content gate. Non-bundled plugins could read raw conversation text without `allowConversationAccess`.
  - confidence: 0.89
- **[P1] Run completion hooks before block replies stream:** `src/auto-reply/reply/agent-runner-hooks.ts:47-51`
  - body: The wrapper awaits `runAgentTurnWithFallback()` before checking suppress/reinject. With block streaming, that inner run can already enqueue or send reply chunks, so `suppress` is too late and `reinject` can duplicate partial output.
  - confidence: 0.86
- **[P2] Add runAfterAgentComplete to the hook runner:** `src/auto-reply/reply/agent-runner-hooks.ts:65`
  - body: This call is made on `HookRunner`, but the patch does not add a matching method to `createHookRunner()` or its returned object. The branch will fail type checking and would throw at runtime if forced through.
  - confidence: 0.94
- **[P2] Pass accountId to agent_end contexts:** `src/agents/pi-embedded-runner/run/attempt.ts:2813`
  - body: The PR adds `accountId` to `PluginHookAgentContext`, but the `agent_end` call site only adds `conversationId`. Multi-account plugins observing `agent_end` still cannot distinguish the bot account.
  - confidence: 0.91
- **[P2] Use the originating target for native commands:** `src/auto-reply/reply/agent-runner.ts:379`
  - body: For native Telegram/Discord command turns, `sessionCtx.To` can be a synthetic command target while the real conversation lives in `OriginatingTo`. Building the new hook key from `sessionCtx.To` makes this phase disagree with inbound contexts.
  - confidence: 0.78
