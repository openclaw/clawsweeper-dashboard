## Summary

Keep open: current main still shows the hook-context inconsistency, but this PR is not mergeable as-is because the branch adds an incomplete `after_agent_complete` lifecycle hook, misses real behavior proof, and introduces prompt-injection/streaming policy defects.

## What This Changes

The PR passes `conversationId`/`accountId` through plugin agent and inbound-claim hook contexts and adds an `after_agent_complete` hook wrapper around reply-agent runs.

## Best Possible Solution

Rebase onto current main, split the `after_agent_complete` feature into the canonical lifecycle-hook work, and land a focused hook-context fix with regression coverage plus redacted multi-account runtime proof.

## Reproduction Assessment

Yes. Source inspection of current main shows inbound-claim conversation ids are still resolved through channel/fallback normalization, while the shared agent hook context still lacks `conversationId` and `accountId`.

## Solution Assessment

No. The context-consistency direction is sound, but this branch is not the narrowest maintainable fix because it bundles an incomplete `after_agent_complete` hook that should be split or aligned with https://github.com/openclaw/openclaw/pull/48100.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Add the missing after_agent_complete runner method:** `src/auto-reply/reply/agent-runner-hooks.ts:64`
  - body: `getGlobalHookRunner()` returns `HookRunner = ReturnType<typeof createHookRunner>`, but this PR calls `hookRunner.runAfterAgentComplete(...)` without adding that method to `createHookRunner` or its returned object. The branch will fail typechecking and would throw at runtime if this hook path is reached.
  - confidence: 0.98
- **[P1] Gate reinjection as prompt injection:** `src/plugins/types.ts:1393`
  - body: The new hook can return `reinject` with `injectContext`, which rewrites the next prompt, but the PR registers `after_agent_complete` as a normal hook. Plugins configured with `hooks.allowPromptInjection=false` would still be able to modify prompts once the runner method exists.
  - confidence: 0.86
- **[P1] Run suppress and reinject before block replies can stream:** `src/auto-reply/reply/agent-runner-hooks.ts:48-52`
  - body: The wrapper waits for `runAgentTurnWithFallback()` to finish before firing `after_agent_complete`, but block-streaming runs can enqueue or flush reply chunks inside that call. In streaming mode, `suppress` cannot reliably stop the answer and `reinject` can duplicate already-sent partial output.
  - confidence: 0.88
- **[P2] Pass accountId through to agent_end hooks:** `src/agents/pi-embedded-runner/run/attempt.ts:2813`
  - body: The PR adds `accountId` to the shared agent hook context for multi-account isolation, but the `agent_end` call site only adds `conversationId`. Plugins doing cleanup, metrics, or state updates in `agent_end` still cannot distinguish two bot accounts in the same conversation.
  - confidence: 0.93
- **[P2] Use the originating target for after_agent_complete:** `src/auto-reply/reply/agent-runner.ts:384`
  - body: For native commands, `sessionCtx.To` can be a synthetic command target while `OriginatingTo` carries the real conversation. Building `channelKey` and `conversationId` from `sessionCtx.To` makes this new hook disagree with inbound and embedded-agent contexts for the same turn.
  - confidence: 0.86
