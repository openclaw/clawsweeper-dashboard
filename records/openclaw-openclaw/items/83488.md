## Label Justifications

- `P1`: The report describes a broken Discord channel workflow where subsequent messages hang for minutes and are effectively dropped until manual state surgery.
- `impact:message-loss`: Queued channel messages can be lost or left unanswered while the lane repeatedly retries an unresumable session.
- `impact:session-state`: The bug centers on stale persisted session state in sessions.json causing the wrong sessionId/sessionFile to be reused.

## Summary

Keep open: current main still reuses a fresh session entry by sessionId/freshness alone and carries the old sessionFile forward when the sessionId is unchanged, so the reported failed lane with a missing transcript is not cleared by the current code path.

## What This Changes

The issue asks OpenClaw to discard an unresumable per-channel session lane whose entry is failed and points at a missing sessionFile, then allocate a fresh sessionId for the next turn.

## Best Possible Solution

Add a narrow recoverability check before sessionId selection so a failed entry with a resolved missing transcript file is treated as a new session while preserving safe channel metadata, then cover that gateway path with a regression test.

## Reproduction Assessment

Yes, source-reproducible: seed a fresh per-channel session entry with status failed, an existing sessionId, and a missing sessionFile, then invoke the gateway agent path for that sessionKey; current source selects the old sessionId and preserves the stale file pointer. I did not run the live Discord/claude-cli setup in this read-only review.

## Solution Assessment

No, the current implementation is not the best solution because it gates reuse on freshness only. The requested file-existence check is directionally right, but it should live at the gateway/session-entry reuse boundary and resolve paths through the existing session path helpers.
