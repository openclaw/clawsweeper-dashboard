## Label Justifications

- `P1`: The issue describes a Discord channel lane that repeatedly hangs for minutes and blocks replies until manual sessions.json editing.
- `impact:message-loss`: Subsequent channel messages can be dropped or left unanswered while the lane retries an unresumable session.
- `impact:session-state`: The root problem is stale persisted session state causing the wrong sessionId/sessionFile pair to be reused.

## Summary

Keep open: current main still selects an existing fresh session by sessionId/freshness alone and preserves the stored sessionFile when that sessionId is reused, so a failed lane with a missing transcript remains a valid source-backed bug.

## What This Changes

The issue reports that failed per-channel session lanes with missing transcript files are resumed instead of replaced with a fresh session.

## Best Possible Solution

Treat a failed session entry with a resolved missing transcript as unresumable before choosing the sessionId, then add a focused gateway regression test that preserves safe delivery metadata while allocating a fresh session.

## Reproduction Assessment

Yes, source-reproducible: seed a fresh per-channel entry with status failed, an existing sessionId, and a missing sessionFile, then invoke the gateway agent path for that session key; current source still selects the old sessionId and preserves the stale file pointer. I did not run the live Discord/claude-cli setup in this read-only review.

## Solution Assessment

No; the current implementation is not the best solution because reuse is gated on freshness only. The fix should live at the gateway/session-entry reuse boundary and use existing session path resolution before deciding whether the entry is resumable.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/83488. On current main, a per-channel session entry with `status: "failed"`, a fresh `updatedAt`, an existing `sessionId`, and a missing resolved `sessionFile` is treated as reusable, so the gateway keeps retrying the unresumable session. Add a narrow recoverability check before gateway sessionId selection in `src/gateway/server-methods/agent.ts`, using existing session path helpers from `src/config/sessions/*`, so failed entries whose resolved transcript is missing allocate a fresh sessionId and do not carry forward the stale `sessionFile`; preserve delivery/channel/plugin metadata that remains safe. Add or update focused regression coverage in `src/gateway/server-methods/agent.test.ts` and, if helper behavior changes, `src/config/sessions/sessions.test.ts`. Add a `CHANGELOG.md` fix entry because this is user-visible. Do not introduce a new config option, startup migration, or broad lane watchdog in this fix.
