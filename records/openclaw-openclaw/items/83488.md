## Label Justifications

- `P1`: The reported Discord lane can hang for minutes on every subsequent message until manual sessions.json cleanup.
- `impact:message-loss`: Messages sent to the wedged channel can be left unanswered while the gateway retries an unresumable session.
- `impact:session-state`: The bug is caused by stale persisted session state reusing a missing transcript-backed sessionId.

## Summary

Keep open: current main still treats a fresh failed lane as reusable by sessionId/freshness alone, so a failed entry with a missing transcript path can keep the channel on the stale sessionId.

## What This Changes

Treat failed per-channel session lanes whose persisted sessionFile is missing as unresumable and rotate to a fresh sessionId on the next turn.

## Best Possible Solution

Add a narrow recoverability check at the gateway session-entry reuse boundary, resolve the stored sessionFile through existing path helpers, and rotate only failed entries whose resolved transcript file is missing.

## Reproduction Assessment

Yes, source-reproducible: seed a fresh per-channel entry with status failed, a sessionId, and a missing sessionFile, then send a gateway agent turn for the same sessionKey; current source selects the old sessionId. I did not run the focused Vitest or live Discord/claude-cli setup in this read-only review.

## Solution Assessment

No, the current implementation is not the best way to solve it because freshness alone is not a recoverability check. The narrow maintainable fix is to reject reuse for failed entries whose resolved transcript file is missing, while preserving channel metadata for the fresh entry.
