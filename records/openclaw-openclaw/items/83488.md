## Label Justifications

- `P1`: The report describes a broken Discord channel workflow where messages hang for minutes and are effectively dropped until manual state surgery.
- `impact:message-loss`: The affected channel can leave follow-up messages unanswered while retrying an unresumable session lane.
- `impact:session-state`: The root state is a stale sessions.json entry that reuses the wrong sessionId/sessionFile pairing.

## Summary

Keep open: current main still reuses a fresh failed lane by sessionId/freshness alone and preserves the stale sessionFile when the sessionId is unchanged, so the reported missing-transcript lane is not recovered.

## What This Changes

The issue requests that a failed per-channel session lane whose sessionFile is missing be discarded so the next channel message allocates a fresh sessionId.

## Best Possible Solution

Add a narrow recoverability check before sessionId selection so a failed entry with a resolved missing transcript file is treated as a new session while preserving safe metadata and adding focused regression coverage.

## Reproduction Assessment

Yes, at source level. Seed a fresh per-channel session entry with status failed, an existing sessionId, and a missing sessionFile, then invoke the gateway agent path for that sessionKey; current main selects the old sessionId and preserves the stale file pointer.

## Solution Assessment

Yes, the best fix is a narrow gateway/session-entry recoverability check rather than a new config or Discord-specific workaround. The check should resolve the persisted path through existing session path helpers before deciding the entry is unresumable.
