## Summary

Keep open for normal maintainer handling: this is a narrow security-sensitive Codex sandbox hardening PR, and I found no blocking correctness or supply-chain regression. The only actionable issue I found is low-priority docs alignment for the canonical Codex harness config docs.

## What This Changes

The PR clamps Codex app-server sandboxing from danger-full-access to workspace-write when an OpenClaw sandbox is active, while adding targeted tests, docs notes, and a changelog entry.

## Best Possible Solution

Land the narrow runtime clamp after maintainer review, and align the Codex harness config docs with the active-sandbox exception so operators see the same contract in the canonical reference.

## Reproduction Assessment

Yes by source inspection: current main passes appServer.sandbox through to Codex thread and turn requests, and danger-full-access maps to Codex dangerFullAccess even after an enabled OpenClaw sandbox is resolved. I did not run a live Docker/Codex repro.

## Solution Assessment

Yes, the proposed runtime clamp is the narrowest maintainable fix because it only narrows danger-full-access when OpenClaw sandboxing is active and preserves non-sandboxed plus already-narrower Codex sandbox behavior. The docs should also update the canonical Codex harness reference.
