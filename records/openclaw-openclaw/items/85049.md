## Label Justifications

- `P1`: Cleartext provider credential exposure in gateway logs can affect real operators and deserves urgent security handling without making the core runtime unusable.
- `impact:security`: The report is about sensitive credentials being persisted to local logs after a denied tool call.
- `impact:auth-provider`: The exposed values are provider API keys and access tokens used for model/tool authentication.

## Summary

Keep open: current main has general tool-log redaction, but denied exec failures still flow through raw_params logging, and the current redaction pattern can miss JSON-escaped quoted env assignments for non-sk token values, so the credential-log exposure remains actionable.

## What This Changes

Scrub, truncate, or stop logging denied exec raw_params so inline env-var credentials do not persist to gateway.err.log.

## Best Possible Solution

Treat this as a gateway logging security fix: prevent raw denied exec command values from reaching persistent logs, and add regression coverage for JSON-escaped quoted env assignments plus provider token prefixes.

## Reproduction Assessment

Yes, source-level: denied exec throws through the generic tool failure logger, which includes raw_params, and the current redaction pattern misses escaped quoted env assignments for several token shapes. I did not run a live gateway repro because this review was required to keep the checkout and logs read-only.

## Solution Assessment

Unclear which exact mitigation maintainers want, but the requested direction is sound: either redact command fields before logging, log a bounded hash/preview, or suppress raw_params on denial, with focused regression tests for the reported command shape.
