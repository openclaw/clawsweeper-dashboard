## Summary

Keep open. The merged related PR shipped useful no-visible-reply handling, but source inspection still leaves the reported observability gap for normal replies from a reused automatic fallback: current main emits lifecycle/status signals, not a `model fallback decision` gateway log for that path, and active fallback-pin policy PRs are still open.

## What This Changes

This issue requests gateway-log telemetry whenever an embedded run uses a fallback provider instead of the configured primary.

## Best Possible Solution

Land one coherent auto-fallback recovery and telemetry policy that makes reused automatic fallback selections observable or temporary, while preserving explicit user model overrides and existing per-attempt fallback logs.

## Reproduction Assessment

Yes, at source level: a session with `modelOverrideSource: "auto"` and a fallback active can start on the fallback provider, so the failed-primary `model fallback decision` path does not run. The live Slack/Codex timeout trigger was not reproduced here.

## Solution Assessment

No, the merged no-visible-reply fix is not the complete solution for this issue because normal fallback replies can still avoid the requested gateway decision log. The safer path is to choose the open failback policy or add explicit reused-auto-fallback telemetry.
