## Summary

Keep open. Current main still logs `model fallback decision` for fallback-loop failures, but a persisted `modelOverrideSource: "auto"` fallback can become the next embedded run's first candidate and bypass the failed-primary log path; the related https://github.com/openclaw/openclaw/issues/64902 tracks transcript audit persistence, not this gateway-log gap.

## What This Changes

Add observable gateway fallback telemetry when an embedded run dispatches an auto-selected fallback provider instead of the configured primary.

## Best Possible Solution

Keep fallback persistence, but add a bounded `model fallback decision` or equivalent telemetry event when an auto fallback selection is reused as the run's starting model while the configured primary differs.

## Reproduction Assessment

Yes for a source-level path: a session entry with `modelOverrideSource: "auto"` pointing at the configured fallback will make that fallback the first candidate, so the failed-primary decision log is not emitted. The exact Slack/Codex transient outage still needs a real setup capture.

## Solution Assessment

Yes. The narrow maintainable fix is telemetry at the auto fallback-selection reuse boundary or equivalent fallback-step emission, without changing fallback persistence, candidate ordering, or strict user override semantics.
