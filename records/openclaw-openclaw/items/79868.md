## Summary

Keep open. The merged related PR handles the no-visible-reply variant, but current main can still reuse an auto-persisted fallback as the first dispatched model and produce a normal reply without emitting the requested `model fallback decision` gateway log.

## What This Changes

The issue asks OpenClaw to log a fallback decision whenever an embedded run uses a fallback provider instead of the configured primary model.

## Best Possible Solution

Add sanitized gateway-visible fallback telemetry for reused auto-fallback selections while preserving fallback persistence, candidate ordering, cooldown behavior, and user-selected model override semantics.

## Reproduction Assessment

Yes at source level: a session with `modelOverrideSource: "auto"` can dispatch the fallback as the first candidate, so the failed-primary and candidate-succeeded fallback log path does not fire. The reported live Slack/Codex transient trigger still needs real setup capture.

## Solution Assessment

Yes. The narrow maintainable fix is telemetry at the auto-fallback reuse boundary or equivalent fallback-step emission, not a change to model selection, fallback persistence, or user override policy.
