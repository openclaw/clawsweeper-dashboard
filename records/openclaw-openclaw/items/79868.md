## Summary

Keep open. Current main fixed an adjacent silent no-reply fallback symptom, but source inspection still shows reused auto fallback state can start a later run on the fallback model and avoid the `model fallback decision` log path requested here.

## What This Changes

The issue requests gateway telemetry whenever an embedded run uses a fallback provider instead of the configured primary model.

## Best Possible Solution

Add explicit telemetry for reused auto fallback selections, or land the chosen temporary-fallback recovery policy, so configured-primary bypasses are operator-visible without treating user-selected overrides as fallback.

## Reproduction Assessment

Yes, at source level. A session with auto fallback provenance can make the fallback provider/model candidate 0, and a successful candidate 0 path does not emit the failed-primary `model fallback decision` log.

## Solution Assessment

Unclear until maintainers choose the fallback-pin policy. The merged no-visible-reply fix is useful but does not fully solve the requested gateway fallback decision telemetry.
