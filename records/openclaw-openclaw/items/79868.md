## Summary

Keep open. Current main still preserves auto fallback selections across later embedded runs and only emits `model fallback decision` from the active fallback loop, so a reused auto fallback can dispatch first without the requested gateway log line.

## What This Changes

Embedded runs can use a configured fallback provider without a gateway `model fallback decision` log when the configured primary provider is bypassed mid-dispatch.

## Best Possible Solution

Add explicit telemetry when an auto-sourced fallback selection is reused as the starting model, while preserving current fallback persistence and user override semantics.

## Reproduction Assessment

Yes for a source-level path: a session entry with `modelOverrideSource: "auto"` and a fallback provider/model is selected before dispatch, then `runWithModelFallback` sees that fallback as the first candidate and does not emit a failed-primary decision. The original live Slack/Codex outage trigger was not reproduced here.

## Solution Assessment

Yes. The narrow maintainable fix is observability at the stored auto-fallback reuse boundary or equivalent fallback-step emission, not changing user override behavior or broadly redesigning fallback persistence.
