## Summary

Keep open. Current main still has a source-level path where an auto-persisted fallback selection can become the next embedded run's starting model, so the normal failed-primary `model fallback decision` log path is skipped even though the configured primary is being bypassed.

## What This Changes

The issue requests gateway telemetry whenever an embedded run uses a fallback provider instead of `agents.defaults.model.primary`.

## Best Possible Solution

Keep the existing fallback persistence behavior, but emit explicit telemetry for reused `modelOverrideSource: "auto"` fallback selections or an equivalent fallback-step event without changing user override semantics.

## Reproduction Assessment

Yes for a source-level path: a session whose `modelOverrideSource` is `auto` and whose override points at a configured fallback will dispatch that provider as the first candidate, so the failed-primary fallback log does not fire. The live Slack/Codex outage trigger itself still needs a real setup capture.

## Solution Assessment

Yes. The narrow maintainable fix is telemetry at the auto fallback-selection reuse boundary, not a change to fallback ordering, persistence, or user-selected model behavior.
