## Summary

Keep open. Current main still has a source-level path where an auto-persisted fallback provider becomes the first embedded-run candidate, so the existing failed-primary `model fallback decision` log path never fires even though the configured primary is being bypassed.

## What This Changes

The issue requests explicit gateway fallback telemetry when an embedded run uses a fallback provider instead of the configured primary.

## Best Possible Solution

Keep fallback persistence, but emit explicit sanitized telemetry when an auto-sourced fallback selection is reused as the starting embedded-run model and differs from the configured primary.

## Reproduction Assessment

Yes for a high-confidence source-level path: store a session entry with `modelOverrideSource: "auto"` pointing at a configured fallback, then run an embedded reply and observe that the fallback is candidate zero so the failed-primary log path is skipped. The original Slack/Codex outage timing has not been live-reproduced here.

## Solution Assessment

Yes. The best fix is narrow observability at the auto fallback-selection reuse boundary or equivalent fallback-step emission, without changing model routing, fallback persistence, or user override semantics.
