## Summary

Keep open. Current main still allows an auto-persisted fallback selection to become the first candidate on a later embedded run, while `model fallback decision` logging only fires for skip/fail/probe paths or success after a prior attempt, so the reported silent fallback path remains plausible from source. Related open PRs address sticky fallback-pin policy, but they do not close this telemetry issue yet.

## What This Changes

The issue asks for gateway telemetry whenever an embedded run uses a configured fallback provider instead of the configured primary without emitting a `model fallback decision` log.

## Best Possible Solution

Keep auto fallback recovery visible by emitting structured fallback telemetry when an auto-owned fallback selection is reused, while preserving strict handling for explicit user model overrides.

## Reproduction Assessment

Yes for a source-level path: a session with `modelOverrideSource: "auto"` pointing at a configured fallback can dispatch that fallback as the first candidate, so the failed-primary log path is not entered. The reported live Slack/Codex outage trigger still needs environment-specific capture.

## Solution Assessment

Yes. The narrow maintainable fix is telemetry at the auto fallback reuse boundary, or an equivalent fallback-step emission, without changing explicit user override semantics or broad model-selection policy.
