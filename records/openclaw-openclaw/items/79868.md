## Summary

Keep open. Current main still has a source-level path where an auto-persisted fallback selection can become the next embedded run's starting model, so the configured primary is bypassed without producing the existing failed-primary `model fallback decision` log.

## What This Changes

The issue asks OpenClaw to log an explicit fallback decision whenever embedded dispatch uses a configured fallback provider instead of the configured primary.

## Best Possible Solution

Keep fallback persistence, but emit explicit fallback telemetry when an auto fallback override is reused as the starting model and the configured primary is being bypassed, without changing user override semantics.

## Reproduction Assessment

Yes, for a high-confidence source-level path: a session with `modelOverrideSource: "auto"` pointing at the configured fallback is dispatched as candidate one, so the failed-primary decision log does not fire. The exact live Slack/Codex outage trigger still needs a real setup capture.

## Solution Assessment

Yes. The narrow maintainable fix is telemetry at the reused auto-fallback boundary or equivalent fallback-step emission, not a broad change to fallback ordering or user-selected model behavior.
