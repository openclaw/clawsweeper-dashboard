## Summary

Keep open. Current main and the merged silent-fallback PR cover the no-visible-reply case, but they do not fully cover the reported successful-reply path where a persisted auto fallback starts the run on the fallback provider and therefore avoids the failed-primary `model fallback decision` log. The related recovery-policy PRs are still open and competing, so this needs maintainer follow-up rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to emit fallback-decision telemetry whenever an embedded run uses a fallback provider instead of the configured primary provider.

## Best Possible Solution

Add explicit fallback-decision telemetry for reused auto fallback selections, or land a maintainer-chosen recovery policy that makes those reused selections impossible while preserving accurate operator logs.

## Reproduction Assessment

Yes for a source-level path: a session entry with `modelOverrideSource: "auto"` can make the fallback provider the starting model, and current `model fallback decision` logging only fires during actual fallback-loop decisions. I did not reproduce the reporter's Windows Docker plus Slack plus Codex outage live.

## Solution Assessment

No, current main is not a complete solution for this issue. The merged no-visible-reply fix is useful but narrower than the reported successful-reply telemetry gap; the maintainable fix is narrow telemetry at the reused auto-fallback boundary or a chosen recovery-policy change.
