## Summary

Keep open: current main improved the related no-visible-reply path, but it still does not emit a `model fallback decision` log when an auto-persisted fallback is already the first candidate and succeeds with a visible reply.

## What This Changes

The issue asks OpenClaw to log a fallback decision whenever an embedded run uses a fallback provider instead of the configured primary.

## Best Possible Solution

Add narrow telemetry for reused auto fallback selections, ideally emitting `model_fallback_decision` or an equivalent fallback-step event when the active provider/model differs from the configured origin because of auto fallback provenance, without changing fallback routing semantics.

## Reproduction Assessment

Yes, at source level: a session entry with auto fallback provenance can start the run on the fallback model, and the first-candidate success path in `runWithModelFallback` does not log a fallback decision. I did not run the reporter's live Slack/Codex/Docker setup.

## Solution Assessment

No, current main is not the full fix for this issue. The merged related PR is useful for empty fallback replies, but the maintainable solution is a focused decision-log or fallback-step emission for reused auto fallback provenance.
