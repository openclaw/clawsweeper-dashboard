## Summary

Keep open. Current main still has a source-level path where an embedded turn can start from a persisted `modelOverrideSource: "auto"` fallback, succeed as the first candidate, and avoid the `model fallback decision` log path; the merged https://github.com/openclaw/openclaw/pull/80917 covers silent/no-visible-reply fallback outcomes, not the reported visible-reply telemetry gap.

## What This Changes

The issue asks OpenClaw to emit gateway fallback telemetry when an embedded run uses a configured fallback provider instead of the configured primary model.

## Best Possible Solution

Add sanitized gateway/subsystem telemetry when an embedded run reuses an auto fallback origin and the active provider/model differs from the configured origin, while preserving current fallback persistence and user override semantics.

## Reproduction Assessment

Yes, at source level: a session entry with `modelOverrideSource: "auto"` and a fallback provider/model becomes the first candidate, and a successful first candidate does not enter the existing `model fallback decision` log branch. I did not reproduce the original Slack/Codex Docker timeout live.

## Solution Assessment

Yes. The narrow maintainable fix is telemetry at the auto fallback reuse or fallback-transition boundary; https://github.com/openclaw/openclaw/pull/80917's no-visible-reply guard and https://github.com/openclaw/openclaw/pull/75270's sticky-fallback policy are adjacent, not a replacement for the gateway log gap.
