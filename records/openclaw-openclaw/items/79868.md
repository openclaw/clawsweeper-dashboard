## Summary

Keep open. Current main has improved fallback visibility for no-reply outcomes, but a persisted auto fallback can still start a later embedded run on the fallback provider without emitting a `model fallback decision` gateway log for a successful visible reply.

## What This Changes

Embedded runs should emit fallback telemetry whenever the actual provider differs from the configured primary.

## Best Possible Solution

Add a bounded, sanitized fallback decision log or equivalent gateway-visible event when auto fallback provenance makes the selected origin differ from the active provider/model, while leaving sticky-vs-rollback fallback policy to its own track.

## Reproduction Assessment

Yes, at source level: a session with `modelOverrideSource: "auto"` and fallback-origin fields can dispatch the fallback as the first candidate, so the current failed-primary decision log path is skipped. I did not rerun the reporter's live Slack/Windows/Codex setup.

## Solution Assessment

Yes, the narrow maintainable fix is telemetry at the auto fallback reuse or selected-vs-active mismatch boundary. It should not change fallback ordering, rollback policy, or user override semantics.
