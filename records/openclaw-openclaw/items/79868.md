## Summary

Keep open. Current main still allows an auto-persisted fallback selection to become the next embedded run's starting model, so the existing failed-primary `model fallback decision` log path can be bypassed in the way reported.

## What This Changes

The issue asks OpenClaw to log observable fallback telemetry when an embedded run uses a fallback provider instead of the configured primary due to an auto fallback selection.

## Best Possible Solution

Keep auto fallback persistence, but emit explicit telemetry when an auto-sourced fallback selection is reused so operators can see that the configured primary is being bypassed without changing user override semantics.

## Reproduction Assessment

Yes for a source-level path: create a session entry with `modelOverrideSource: "auto"` and provider/model set to a configured fallback, then run the embedded reply path. The fallback becomes the first candidate, so the current failed-primary `model fallback decision` log does not fire.

## Solution Assessment

Yes. The narrow maintainable fix is telemetry at the auto fallback reuse boundary, or an equivalent fallback-step emission, while preserving exact behavior for user-selected model overrides.
