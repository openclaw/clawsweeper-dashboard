## Summary

Keep open. Current main merged the adjacent no-visible-reply fallback fix, but a persisted `modelOverrideSource: "auto"` fallback can still be the first successful candidate and avoid the `model fallback decision` path, which matches the reported silent gateway-log gap for successful replies.

## What This Changes

The issue requests gateway telemetry whenever an embedded run uses a fallback provider instead of the configured primary model.

## Best Possible Solution

Keep fallback persistence, but add explicit sanitized telemetry for reused auto fallback selections so operators can see when the configured primary is being bypassed without changing user-selected model semantics.

## Reproduction Assessment

Yes, for a source-level path: a session whose `modelOverrideSource` is `auto` and points at the fallback will dispatch that fallback as the first candidate, and current main only logs `candidate_succeeded` after a prior attempt/cooldown. I did not live-reproduce the reporter's Slack/Codex transient trigger.

## Solution Assessment

Yes, the best fix is narrow telemetry at the auto fallback reuse boundary or equivalent fallback-step emission. The merged no-visible-reply guard is useful but does not solve the successful-reply gateway-log gap.
