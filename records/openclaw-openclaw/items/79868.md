## Summary

Keep open. Current main logs fallback decisions for failures inside a fallback loop, but source inspection shows an auto-persisted fallback selection can become the next embedded run's starting model and avoid that failed-candidate log path, which matches the reported silent Anthropic usage.

## What This Changes

The issue asks OpenClaw to emit gateway-visible model fallback telemetry whenever an embedded run bypasses the configured primary and uses a fallback provider.

## Best Possible Solution

Keep fallback persistence, but add explicit telemetry for reused auto fallback selections so operators can see why a configured primary is being bypassed without misreporting user-selected model overrides.

## Reproduction Assessment

Yes for a source-level path: a session with `modelOverrideSource: "auto"` pointing at the configured fallback will dispatch that provider as the first candidate, so the current failed-primary fallback log does not fire. The reported live Slack/Codex outage trigger still needs a real setup capture.

## Solution Assessment

Yes, the maintainable fix is narrow telemetry at the auto fallback-selection boundary or equivalent fallback-step emission, not a broad change to model selection or user override semantics.
