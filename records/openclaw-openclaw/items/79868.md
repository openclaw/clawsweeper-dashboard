## Summary

Keep open. Current main still preserves auto fallback overrides and can start a later embedded turn on the fallback provider; the fallback decision log path only fires for skips, failures, cooldown probes, or successes after earlier fallback activity, so the reported silent reused-fallback path remains plausible.

## What This Changes

The issue asks OpenClaw to emit gateway fallback telemetry whenever an embedded run uses a fallback provider instead of the configured primary.

## Best Possible Solution

Add explicit fallback decision telemetry for auto-sourced fallback reuse while preserving sticky fallback behavior and strict user override semantics.

## Reproduction Assessment

Yes for a source-level current-main path: a session entry with `modelOverrideSource: "auto"` and a fallback provider/model starts `runWithModelFallback` on that fallback, and first-candidate success does not emit `model fallback decision`. The exact live Slack/Codex outage trigger was not reproduced.

## Solution Assessment

Yes. The narrow maintainable fix is telemetry at the auto fallback reuse boundary or equivalent fallback-step emission, not changing candidate selection, fallback persistence, or user override semantics.
