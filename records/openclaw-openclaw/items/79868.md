## Summary

Keep open. Current main still has a source-level path where an auto-persisted fallback selection is reused as the next embedded run's starting model, so the configured primary is bypassed without the failed-candidate `model fallback decision` log the reporter expected.

## What This Changes

The issue asks OpenClaw to emit gateway fallback telemetry whenever an embedded run uses a fallback provider instead of the configured primary.

## Best Possible Solution

Keep auto fallback persistence, but add explicit telemetry when a `modelOverrideSource: "auto"` fallback selection is reused and differs from the configured primary, without changing user override semantics.

## Reproduction Assessment

Yes for a source-level path: a session entry with `modelOverrideSource: "auto"` pointing at the configured fallback is preserved, selected as the next run's first candidate, and can succeed without a failed-primary decision log. The reporter's exact live Slack/Codex transient trigger was not reproduced here.

## Solution Assessment

Yes. The narrow maintainable fix is telemetry at the reused auto fallback-selection boundary or equivalent fallback-step emission, not a change to fallback ordering, persistence, or user-selected model behavior.
