## Best Possible Solution

Use one canonical shared Telegram rich-message preparation fix, preferably this PR if current-head checks pass, then close or retarget the parallel formatter-level candidate.

## Solution Assessment

Yes, this is the best current fix shape: putting newline materialization in shared rich HTML preparation covers both Markdown-rendered and explicit rich HTML callers without touching `sendRichMessageDraft` or the rich-disabled fallback. The formatter-level alternative is narrower and should not land in parallel.
