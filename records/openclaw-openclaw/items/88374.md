## Summary

Keep this PR open for human gateway/security review. It fixes a real custom-node-id bug with supplied real CLI proof, but the current patch makes `client.instanceId` authoritative without a signed node-id contract and changes persisted pairing ownership/adoption behavior while the branch is conflicting against current `main`.

## What This Changes

The PR changes gateway node identity resolution so `openclaw node run --node-id` is honored across pairing, registry/listing, pending work, node events, invoke-result validation, and paired-node ownership metadata.

## Best Possible Solution

Choose the gateway identity contract first, preferably a signed custom-node-id protocol/auth shape if maintainers want custom ids to become canonical node identities, then land one refreshed implementation with upgrade coverage for existing paired-node state.

## Reproduction Assessment

Yes, at source level. Current main passes `--node-id` as `client.instanceId`, while gateway registry and pairing reconciliation still key node identity from `device.id ?? client.id`; I did not run a live gateway/node smoke in this read-only review.

## Solution Assessment

No. The resolver fixes the visible symptom, but custom node IDs should be signed or otherwise gateway-verified before becoming the runtime identity boundary.
