## What This Changes

The PR changes gateway node identity handling so signed explicit node IDs are used across pairing, registry, catalog/listing, pending work, node events, invoke-result checks, and node-host restart state.

## Best Possible Solution

Land the signed-instance-ID path after gateway/security owners confirm the v4 auth contract, the legacy owner-checked adoption behavior, and the upgrade fallback are acceptable.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main sends `--node-id` as `client.instanceId`, while registry, pending, and invoke-result paths use the device ID before the client ID. I did not run the live current-main repro in this read-only review.

## Solution Assessment

Yes, this is likely the right shape because it avoids blindly trusting `client.instanceId` and instead binds explicit node IDs to a v4 device-auth signature plus owner device ID. The remaining question is owner/security acceptance of the compatibility and migration tradeoffs, not a concrete code defect I found.
