## Summary

Keep open: current main and v2026.6.1 still ignore the CLI node ID at the gateway pairing boundary, while this PR appears to fix the linked bug but needs maintainer/security acceptance of the identity and storage semantics before merge.

## What This Changes

The PR centralizes gateway node identity resolution around `client.instanceId`, adds `ownerDeviceId` pairing/adoption, and updates registry, catalog, pending work, node events, invoke-result checks, and regression tests.

## Best Possible Solution

Land the centralized identity fix after gateway/security owners accept authenticated `instanceId` as the node identity and required checks pass, preserving the owner-scoped adoption and metadata safeguards.

## Reproduction Assessment

Yes, at source level: current `main` sends CLI `--node-id` as `client.instanceId`, while gateway pairing/registry paths read `device.id ?? client.id`. I did not run a live gateway in this read-only review, but the PR body supplies after-fix CLI/gateway proof.

## Solution Assessment

Yes, this appears to be the right fix shape: a centralized resolver removes duplicated identity fallbacks, and owner-scoped adoption handles the upgrade/security boundary. A signed protocol redesign would be broader and is not required for this bug fix if maintainers accept authenticated instance IDs.
