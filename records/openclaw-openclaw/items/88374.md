## Summary

Keep this PR open because it addresses a real, source-reproducible gateway node ID bug and includes sufficient terminal proof, but the latest head is still not merge-ready: the new resolver trusts raw `client.instanceId` when callers omit options, which undercuts the signed-node-id auth boundary the PR is adding.

## What This Changes

The PR changes gateway node identity handling so signed custom node IDs flow through pairing, registry, catalog/listing, pending work, events, invoke-result checks, and node-host restart state.

## Best Possible Solution

Make raw instance-ID trust opt-in to verified v4 device auth or an already resolved post-handshake `nodeIdentity`, keep the owner-checked adoption path, remove duplicate tests, then get gateway/security owner acceptance for the upgrade behavior.

## Reproduction Assessment

Yes from source inspection: current main sends `--node-id` as `client.instanceId`, but registry, pending, event, and invoke-result paths still resolve the node from the device/client ID. I did not run a live repro because this review is read-only.

## Solution Assessment

No, not yet. The PR is in the right owner boundary and covers important sibling paths, but the resolver should default to not trusting raw instance IDs before this auth-boundary change is merge-ready.
