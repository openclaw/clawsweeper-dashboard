## Summary

Keep open: the underlying custom node-id bug is still source-reproducible and this PR has useful real-behavior proof, but the patch should not land as-is because it makes unsigned `client.instanceId` authoritative for gateway node identity. The open signed replacement path is not merged or proof-cleared enough to safely close this branch as superseded.

## What This Changes

The PR changes gateway node identity handling so `openclaw node run --node-id` is resolved from `client.instanceId`, adds owner-bound pairing/adoption metadata, and updates gateway, pairing, node-host, and regression tests around custom node IDs.

## Best Possible Solution

Use a signed top-level custom node ID or another gateway-verified identity source, while preserving the owner-scoped adoption and metadata safeguards that make the upgrade path safe.

## Reproduction Assessment

Yes at source level. Current main sends `--node-id` as `client.instanceId`, but gateway pairing and registry still key node identity from `device.id ?? client.id`.

## Solution Assessment

No. The centralized resolver fixes the symptom, but custom IDs should be signed or gateway-verified before they become the runtime identity for pairing, pending work, node events, and invoke results.
