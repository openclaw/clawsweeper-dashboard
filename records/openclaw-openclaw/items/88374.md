## Summary

Keep open for maintainer review: current main and the latest release still ignore the custom node ID at the gateway pairing/registry boundary, and this PR appears to cover the right gateway/node paths, but the identity and upgrade semantics need gateway/security-owner acceptance before merge.

## What This Changes

The PR centralizes gateway node identity around authenticated `client.instanceId`, adds `ownerDeviceId` pairing/adoption, and updates registry, catalog, pending work, node events, invoke-result checks, and regression tests.

## Best Possible Solution

Land the centralized identity fix after gateway/security owners accept authenticated `instanceId` as the node identity and the owner-scoped adoption/metadata safeguards remain intact.

## Reproduction Assessment

Yes. At source level, current main sends `--node-id` as `client.instanceId` while gateway pairing and registry paths still key off `device.id ?? client.id`; the PR body also supplies after-fix CLI/gateway proof.

## Solution Assessment

Yes, this appears to be the right fix shape if maintainers accept the identity boundary: a centralized resolver removes duplicated fallbacks, and owner-scoped adoption handles the upgrade path. A signed-node-ID protocol would be broader and should be a separate maintainer decision if required.
