## Summary

Keep this PR open: current main still does not honor the custom node id, and the branch is aimed at the right gateway/node identity boundary, but the patch should not merge while the new resolver trusts raw `client.instanceId` by default outside the v4/nodeIdentity path.

## What This Changes

The PR changes gateway node identity handling so explicit signed node ids flow through pairing, registry, catalog/listing, pending work, events, invoke-result checks, and node-host restart state.

## Best Possible Solution

Make raw instance-id trust opt-in to v4 verification or an already resolved `nodeIdentity`, keep the owner-checked migration path, remove PR-specific comment lore, then get gateway/security owner acceptance before landing.

Yes from source inspection: current main sends `--node-id` as `client.instanceId`, but registry, pairing, pending, event, and invoke-result paths still resolve the node from the device/client id. I did not run the live repro because this review is read-only.
