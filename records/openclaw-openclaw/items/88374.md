## Summary

Keep this PR open, but it is not merge-ready: it fixes the visible custom node-id symptom by promoting `client.instanceId`, while current auth only signs device/client metadata and not that field. The signed replacement PR is related, but it is still open, conflicting, and marked as needing proof, so it is not a safe supersession target yet.

## What This Changes

The PR changes gateway node identity resolution so `openclaw node run --node-id` is honored across pairing, registry, pending work, node events, invoke results, and owner-bound pairing metadata.

## Best Possible Solution

Use a signed top-level custom node ID or another gateway-verified identity source, while preserving owner-bound adoption and metadata safeguards for the upgrade path.

## Reproduction Assessment

Yes, at source level. Current main forwards `--node-id` as `client.instanceId`, but gateway pairing and registry still key node identity from `device.id ?? client.id`.

## Solution Assessment

No. The resolver fixes the visible symptom, but custom node IDs should be signed or otherwise gateway-verified before becoming the runtime identity boundary.
