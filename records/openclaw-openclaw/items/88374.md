## Summary

Keep this PR open but not merge-ready: current main still does not honor the custom node ID, this branch has useful proof and owner-scoped pairing work, but it makes unsigned `client.instanceId` authoritative for gateway node identity and the signed replacement PR is still conflicting and lacks sufficient real packaged proof.

## What This Changes

The PR changes gateway node identity resolution so `openclaw node run --node-id` values flow through pairing, registry, catalog/listing, pending work, node events, and invoke-result checks.

## Best Possible Solution

Use a signed top-level custom node ID or another gateway-verified identity source, while preserving the owner-scoped adoption and metadata safeguards that make the upgrade path safe.

## Reproduction Assessment

Yes at source level, but not from a live run in this review. Current main sends `--node-id` as `client.instanceId`, while gateway registry and pairing reconciliation still key nodes from `device.id ?? client.id`.

## Solution Assessment

No. The centralized resolver fixes the visible symptom, but custom node IDs should be signed or otherwise gateway-verified before they become runtime identity across pairing, pending work, events, and invoke results.
