## Summary

Keep this PR open, but it needs changes before merge. The source work targets a real auth-profile cooldown problem, yet the proposed raw-HTTP-400 guard conflicts with current main's classifier semantics, where 400/422 responses are deliberately message-inspected before falling back to `format`; the PR would suppress cooldowns for non-format failures such as billing, auth, and rate limits.

## What This Changes

The PR adds status-aware guards to auth-profile failure marking, plumbs prompt-stage error text into the embedded runner for HTTP status extraction, adds unit tests for the new cooldown skip behavior, and adds a changelog entry.

## Best Possible Solution

Keep the narrow fix direction, but make the auth-profile mutation decision depend on the resolved failure classification: skip profile mutation for confirmed `format` payload-shape failures, preserve cooldown/disable behavior for `billing`, `auth`, `rate_limit`, and other specific reasons even when the HTTP status is 400, and cover prompt plus assistant failure paths with focused tests.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: with the PR diff, `markAuthProfileFailure({ reason: "billing", statusCode: 400 })` or `reason: "rate_limit", httpStatus: 400` would return before updating auth-profile usage state, contradicting current main's message-first 400 classification and non-timeout failure policy.

## Solution Assessment

No. The PR solves part of the false-cooldown problem, but the safest maintainable solution is to reuse the existing classifier output and skip only when the resolved reason is `format`, not when any raw HTTP status is 400.

Full review comments:

- **[P1] Do not skip cooldowns for every HTTP 400:** `src/agents/auth-profiles/usage.ts:847-848`
  - body: This early return ignores the already-classified `reason` whenever a caller forwards raw status 400. Current main treats 400/422 as ambiguous and preserves message-derived `billing`, `auth`, `rate_limit`, and `model_not_found` reasons before falling back to `format`, so a quota/auth-broken profile that surfaces as 400 would stay eligible for retry instead of being cooled down or disabled. Gate the skip on a confirmed `format` classification instead of raw status alone.
  - confidence: 0.92
- **[P2] Parse the actual HTTP status before using it for policy:** `src/agents/pi-embedded-runner/run.ts:533-536`
  - body: The regex takes the first 4xx/5xx-looking number in `errorText`. If a provider or proxy includes another number before the real status, such as a request count before a final 429, the cooldown policy can receive a false 400 and skip a real rate-limit/server failure. Use a structured status source or a helper with deterministic tested semantics before feeding status into auth-profile policy.
  - confidence: 0.78
