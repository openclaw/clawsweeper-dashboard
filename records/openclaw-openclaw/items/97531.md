## Summary

Keep open: current main and v2026.6.10 cover registered active embedded runs before Gateway compaction, but they do not prove the narrower Codex-native liveness/write-lock requirement from #97531 is fully handled when `thread/compact/start` is reached.

## What This Changes

Issue #97531 asks for `sessions.compact` to reject, defer, or guard Codex app-server-backed sessions when an active turn, pending tool calls, fresh write lock, or app-server sidecar/lease is present.

## Best Possible Solution

Keep native Codex compaction, but require the Gateway/Codex boundary to prove the target session is idle or return a retryable active-session/deferred result before starting native compaction.

## Reproduction Assessment

No high-confidence live current-main reproduction was run here; source inspection shows broad active-run interruption is present, but the manual Codex native path still sends `thread/compact/start` once reached and upstream Codex does not expose a liveness response for that request.

## Solution Assessment

Unclear: a Codex-owned liveness/write-lock guard or retryable `SESSION_ACTIVE`/defer result is the safer direction, but maintainers should decide whether abort-before-compact is sufficient or active turns must be rejected instead.
