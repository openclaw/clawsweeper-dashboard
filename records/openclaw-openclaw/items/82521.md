## Summary

This should stay open: the docs describe subscription-first `auth.order.openai` routing for `openai/gpt-*`, but current main still has an explicit PI path where `openai-codex:*` is only used after an auth profile is already selected. The issue also identifies a real docs/CLI mismatch for the advertised `--device-code` login flag.

## What This Changes

The issue asks OpenClaw to make documented `openai/gpt-*` plus `auth.order.openai` select `openai-codex:*` subscription auth for Codex/PI fallback and to fix the documented device-code login path.

## Best Possible Solution

Align runtime and docs: explicit PI OpenAI runs should honor the documented `auth.order.openai` subscription-first `openai-codex:*` profile before API-key fallback, with focused regression coverage and corrected device-code login documentation or CLI aliasing.

Yes, there is a high-confidence source-level reproduction path: configure `openai/gpt-5.5` with explicit PI runtime and `auth.order.openai` containing `openai-codex:*` before `openai:*`; current main resolves generic OpenAI auth before the Codex PI routing guard can engage. ## Solution Assessment

Yes, the best fix is a narrow runtime/docs alignment, not a close: select or forward the documented Codex auth profile for explicit PI OpenAI runs before falling back to OpenAI API-key auth, and fix the advertised device-code login path.
