## Summary

Keep open. The stale-response replay bug is still plausible on current main and this PR targets the right surface, but the proposed `.slice(prePromptMessageCount)` fix is not merge-ready because that index can be stale after compaction and the PR also lacks the required changelog entry.

## What This Changes

The PR scopes embedded-runner `lastAssistant` lookup to messages after `prePromptMessageCount` and adds a payload-builder regression test for the idle-timeout/no-assistant case.

## Best Possible Solution

Use a compaction-safe current-turn assistant selection path for reply payload and failover handling, while preserving whole-session `lastAssistant` only where whole-session metadata is intended.

## Reproduction Assessment

Yes for code-level reproduction. A prior assistant plus empty current `assistantTexts` can flow through whole-session `lastAssistant` into payload fallback, and the PR’s proposed fix can be falsified with a compacted snapshot whose length is less than the original `prePromptMessageCount`.

## Solution Assessment

No. The PR fixes the non-compacted stale-replay case, but applying a pre-compaction index to a later snapshot is not the narrowest maintainable solution; the safer fix is to derive or pass a compaction-aware current-turn assistant.
