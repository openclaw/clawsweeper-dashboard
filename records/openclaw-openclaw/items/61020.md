## Summary

Keep open: current main still has the stale `lastAssistant` replay source, but this PR is not mergeable as-is because the proposed raw index boundary can fail after compaction and the external PR lacks real behavior proof.

## What This Changes

The PR scopes embedded-runner `lastAssistant` selection to messages after `prePromptMessageCount` and adds an idle-timeout payload regression test.

## Best Possible Solution

Use a compaction-safe current-turn assistant selection path for payload and failover handling, preserving whole-session `lastAssistant` only where whole-session metadata is intended.

## Reproduction Assessment

Yes for source-level reproduction. Current main can select a prior assistant from the full session snapshot and pass it to payload fallback when current `assistantTexts` is empty; I did not run a live idle-timeout scenario.

## Solution Assessment

No. Scoping to current-turn output is the right direction, but this patch applies a pre-compaction count to a later snapshot; the safer fix should derive or carry the current-attempt assistant across compaction safely.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Derive the current assistant without stale indices:** `src/agents/pi-embedded-runner/run/attempt.ts:1911`
  - body: The new `.slice(prePromptMessageCount)` applies a count captured before the prompt to `messagesSnapshot` selected after compaction. If compaction removed earlier history, that count can start beyond the compacted snapshot and `lastAssistant` becomes `undefined` even when the current turn produced assistant output, dropping payload and failover context.
