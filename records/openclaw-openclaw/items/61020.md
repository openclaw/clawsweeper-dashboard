## Summary

Keep open. The stale-replay bug is valid from source inspection, but this PR's `prePromptMessageCount` slice is compaction-sensitive and can drop a real current-turn assistant; the PR also still lacks after-fix real behavior proof.

## What This Changes

The PR changes embedded-runner `lastAssistant` selection to slice from `prePromptMessageCount` and adds a payload-builder idle-timeout regression test.

## Best Possible Solution

Derive a compaction-safe terminal assistant for the current attempt and pass it to payload/failover paths, with focused regression coverage for idle-timeout/no-token and compaction-after-output cases.

## Reproduction Assessment

Yes for source-level reproduction: current main can select a prior assistant from the full snapshot, feed it into payload fallback when `assistantTexts` is empty, and return those payloads on a prompt idle timeout. I did not run a live idle-timeout reproduction.

## Solution Assessment

No. Scoping to current-turn output is the right direction, but `.slice(prePromptMessageCount)` applies a pre-prompt index to a post-compaction selected snapshot; the fix should derive a boundary from the selected snapshot or another compaction-safe current-attempt marker.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Use a compaction-safe assistant boundary:** `src/agents/pi-embedded-runner/run/attempt.ts:1911`
  - body: The new `.slice(prePromptMessageCount)` applies a count captured before prompt submission to `messagesSnapshot` selected after compaction may have removed earlier history. If that happens, the lookup can skip the current turn's assistant and pass `undefined` downstream even though this attempt produced output; derive the boundary from the selected snapshot or another compaction-safe current-turn marker.
  - confidence: 0.88
