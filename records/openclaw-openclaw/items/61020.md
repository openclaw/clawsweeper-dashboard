## Summary

Keep open: the stale idle-timeout replay bug is still source-reproducible on current main and the latest release, but this PR should not merge as-is because its index boundary is compaction-sensitive and it lacks real after-fix behavior proof.

## What This Changes

The PR scopes embedded-runner `lastAssistant` lookup to messages after `prePromptMessageCount` and adds a payload-builder idle-timeout regression test.

## Best Possible Solution

Use a compaction-safe current-turn assistant selection path for payload and failover handling, preserving whole-session `lastAssistant` only where whole-session metadata is intended.

## Reproduction Assessment

Yes for source-level reproduction. Current main can select a prior assistant from the full session snapshot, pass it into payload fallback when `assistantTexts` is empty, and include those payloads on a prompt idle timeout.

## Solution Assessment

No. Scoping to current-turn output is the right direction, but applying a pre-prompt count to a post-compaction snapshot is not the safest maintainable fix.
