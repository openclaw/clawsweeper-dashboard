## Summary

Keep open. The PR targets a real stale-response replay path because current main still derives `lastAssistant` from the whole session snapshot and `buildEmbeddedRunPayloads` can use it as fallback text, but the proposed one-line index slice is not safe across compaction and should not merge as-is.

## What This Changes

This PR scopes embedded-runner `lastAssistant` lookup to messages after `prePromptMessageCount` and adds a payload-builder regression test for the idle-timeout/no-assistant case.

## Best Possible Solution

Use a compaction-safe current-turn assistant selection path for reply payload/failover handling, keep whole-session `lastAssistant` only where whole-session metadata is intended, and cover both no-token idle timeout and compacted-history current-assistant cases with focused tests plus a changelog entry.

## Reproduction Assessment

Yes for code-level reproduction. A prior assistant plus empty current `assistantTexts` can flow through `lastAssistant` into payload fallback, and the proposed fix can be falsified with a compacted snapshot whose length is less than the original `prePromptMessageCount`.

## Solution Assessment

No. The PR fixes the non-compacted stale-replay case, but `.slice(prePromptMessageCount)` is not the narrowest maintainable solution because it uses a pre-compaction index against a possibly compacted snapshot.
