## Summary

Keep open. The stale `lastAssistant` replay path is still source-reproducible on current main, but this PR is not mergeable as-is because the proposed `prePromptMessageCount` slice can drop real current-turn assistant output after compaction, and the external PR has no after-fix real behavior proof.

## What This Changes

The PR scopes embedded-runner `lastAssistant` selection to messages after `prePromptMessageCount` and adds an idle-timeout payload regression test.

## Best Possible Solution

Use a compaction-safe current-turn assistant selection path for reply payload and failover handling, while preserving whole-session `lastAssistant` only where whole-session metadata is intended.

## Reproduction Assessment

Yes for source-level reproduction. Current main can select a prior assistant from the full session snapshot and pass it to payload fallback when current `assistantTexts` is empty, matching the idle-timeout/no-token stale replay path; I did not run a live idle-timeout scenario.

## Solution Assessment

No. Scoping to current-turn output is the right direction, but this specific patch applies a pre-compaction index to a later selected snapshot; the safer fix is a compaction-aware current-attempt assistant derivation or handoff.

## Review Findings

Overall correctness: patch is incorrect
