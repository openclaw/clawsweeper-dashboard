## Summary

Keep open: current main and the latest release still let unsanitized inbound metadata become the TUI session picker title, and the linked fix branches are either stale/conflicting or cover only adjacent title-precedence work.

## What This Changes

The issue reports TUI session picker titles using gateway-injected inbound metadata instead of a sanitized message or display name.

## Best Possible Solution

Land a narrow Gateway/TUI title fix on current main that strips inbound metadata before deriving titles and lets computed row displayName win across sync, async, and tool session-list paths, with regression coverage and real `sessions.list` or TUI proof.

## Reproduction Assessment

Yes for source-level reproduction: a user transcript whose first message starts with `Sender (untrusted metadata):` flows through `extractTextFromContent` without the inbound sanitizer, and the TUI picker prefers that derived title over displayName. I did not run a live TUI/Gateway session in this read-only review.

## Solution Assessment

Yes, the reported fix direction is the narrow maintainable path: reuse the existing inbound metadata sanitizer and pass the computed row displayName into title derivation. The stale linked PRs are not the best landing vehicle unless rebased and expanded to current main's sync, async, and sibling session-list paths.
