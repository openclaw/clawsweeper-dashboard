## Summary

Keep open: current main and v2026.6.10 still let the TUI session picker prefer a transcript-derived title while the title extractor does not strip gateway-injected inbound metadata, and the direct same-repository fix PR is closed unmerged. The open adjacent title-precedence PR only partially overlaps, so this issue remains the canonical tracker for the concrete Gateway/TUI metadata-title bug.

## What This Changes

Fix TUI `/sessions` titles so gateway-injected inbound metadata is stripped and computed display names win before transcript-derived first-user text.

## Best Possible Solution

Land or adopt one narrow Gateway fix that strips inbound metadata before title extraction and threads computed displayName through both sync and async derived-title hydration, with focused regression coverage and real Gateway/TUI proof.

## Reproduction Assessment

Yes, at source level: the TUI requests derived titles and renders `derivedTitle` before `displayName`, while current Gateway title extraction returns first-user text without using the inbound metadata sanitizer. I did not run a live TUI/Gateway scenario in this read-only review.

## Solution Assessment

Yes: the best fix is the narrow gateway/session-title boundary, not broad persisted-message stripping. The closed broad persistence PR confirms maintainers want this solved at title extraction rather than by rewriting canonical user messages.
