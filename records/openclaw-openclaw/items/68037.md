## Summary

Keep open. Current main and the latest shipped tag still map HTTP MCP tool-result content blocks down to `type` and `text`, while https://github.com/openclaw/openclaw/pull/68046 is the open, unmerged closing PR for this exact gateway bug.

## What This Changes

The issue asks the HTTP MCP gateway to preserve valid non-text MCP tool-result content blocks instead of reducing them to text-only shapes.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/68046 or a narrow replacement that preserves valid MCP text, image, audio, embedded resource, and resource_link blocks while retaining safe text fallback behavior for malformed entries.

## Reproduction Assessment

Yes. A loopback tool returning `{ content: [{ type: "image", data, mimeType }] }` through HTTP MCP `tools/call` deterministically reaches `normalizeToolCallContent`, which currently returns an image-typed object without the required top-level payload fields.

## Solution Assessment

Yes, fixing `normalizeToolCallContent` is the right boundary; the maintainable version should preserve only valid MCP content block shapes and fall back to text for malformed or unknown entries.
