## Summary

Keep open. Current `main` and the latest release still strip non-text MCP content-block payload fields in the HTTP `tools/call` result path, and the open closing PR at https://github.com/openclaw/openclaw/pull/68046 should be reviewed, landed, or replaced before this issue closes.

## What This Changes

The issue asks the HTTP MCP gateway to preserve valid image, audio, resource, and resource_link tool-result content blocks instead of normalizing every block to `{type, text}`.

## Best Possible Solution

Review and land a focused HTTP MCP gateway fix, preferably the linked PR or an equivalent replacement that preserves valid MCP content blocks and adds regression coverage for non-text results.

## Reproduction Assessment

Yes by source inspection: `tools/call` returns `normalizeToolCallContent(result)`, and the current helper drops every non-`type`/`text` field from array content blocks. I did not run a live HTTP MCP client in this read-only pass.

## Solution Assessment

Yes, the narrow fix is to preserve valid MCP content blocks at `normalizeToolCallContent` and cover image/audio/resource/resource_link cases. The issue should stay open until the linked PR or an equivalent fix lands.
