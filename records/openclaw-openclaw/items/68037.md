## Summary

Keep open. Current main still drops non-text MCP content-block payload fields on the HTTP `tools/call` response path, and the open closing PR #68046 is the right implementation candidate, so this issue should not be closed before that PR or an equivalent fix lands.

## What This Changes

The issue reports that HTTP MCP `tools/call` normalization should preserve image, audio, resource, and resource_link content block payload fields instead of rewriting every block to text-shaped content.

## Best Possible Solution

Review and land #68046 or an equivalent focused fix that preserves valid MCP text, image, audio, embedded resource, and resource_link blocks on the HTTP `tools/call` path, keeps fallback text wrapping for strings and malformed entries, and adds regression coverage for non-text block preservation.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: a loopback tool returning `{ content: [{ type: "image", data, mimeType }] }` through HTTP MCP `tools/call` is mapped to a block containing only `type` and `text`; the browser screenshot path naturally returns that image content shape.

## Solution Assessment

Yes. The narrow maintainable solution belongs in `src/gateway/mcp-http.handlers.ts` at the content-normalization boundary, with tests for each MCP content block variant; the open PR is the right candidate, subject to maintainer review of pass-through versus stricter variant copying.
