- `impact:data-loss`: Valid tool-result payload fields are silently dropped before they reach the MCP client.

## Summary

Keep open. Current main and the latest stable release still drop non-text MCP content-block payload fields in the HTTP `tools/call` result path, and the open closing PR at https://github.com/openclaw/openclaw/pull/68046 should be reviewed or replaced before this issue closes.

## What This Changes

The issue reports that the HTTP MCP gateway normalizes every tool-result content block down to `{type, text}`, dropping image, audio, resource, and resource_link payload fields.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/68046 with a normalizer that preserves SDK-valid MCP content blocks, falls back for malformed entries, and adds HTTP `tools/call` regression coverage.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: a loopback tool returning `{ content: [{ type: "image", data, mimeType }] }` through HTTP MCP `tools/call` reaches `normalizeToolCallContent`, which currently emits only `type` and `text`.

## Solution Assessment

Yes. The right fix boundary is the HTTP MCP tool-result normalizer; the maintainable solution should preserve only valid MCP text, image, audio, embedded resource, and resource_link shapes and fall back to text for malformed or unknown entries.
