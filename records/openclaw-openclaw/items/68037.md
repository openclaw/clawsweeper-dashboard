## Summary

Keep open. Current `main` and latest release still drop non-text MCP content-block payload fields in the HTTP `tools/call` response path, and the open closing PR at https://github.com/openclaw/openclaw/pull/68046 should be reviewed or replaced before this issue closes.

## What This Changes

HTTP MCP gateway should preserve image, audio, resource, and resource_link fields in `tools/call` results instead of normalizing every content block to `{ type, text }`.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/68046 with a normalizer that preserves SDK-valid MCP content blocks, falls back for malformed entries, and adds HTTP `tools/call` regression coverage.

## Reproduction Assessment

Yes. A source-level reproduction is clear: a loopback tool returning `{ content: [{ type: "image", data, mimeType }] }` through HTTP MCP `tools/call` reaches `normalizeToolCallContent`, which currently returns an image-typed object without the required payload fields.

## Solution Assessment

Yes. Fixing `normalizeToolCallContent` is the right boundary, but the maintainable implementation should preserve only valid MCP text, image, audio, embedded resource, and resource_link shapes and fall back to text for malformed or unknown entries.
