## Summary

Keep open. Current main and the latest release still strip non-text MCP content-block fields in the HTTP `tools/call` response path, and https://github.com/openclaw/openclaw/pull/68046 is an open, unmerged closing PR for the same gateway bug.

## What This Changes

The issue requests that HTTP MCP `tools/call` results preserve image, audio, embedded resource, and resource_link payload fields instead of normalizing every block to `type` and `text`.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/68046 with a normalizer that preserves valid MCP content blocks, falls back for malformed entries, and adds HTTP `tools/call` regression coverage.

## Reproduction Assessment

Yes. A loopback tool returning `{ content: [{ type: "image", data, mimeType }] }` through HTTP MCP `tools/call` reaches `normalizeToolCallContent`, which currently returns an image-typed object without the required payload fields.

## Solution Assessment

Yes, fixing `normalizeToolCallContent` is the right boundary. The maintainable version should preserve only valid MCP text, image, audio, embedded resource, and resource_link shapes and fall back to text for malformed or unknown entries.
