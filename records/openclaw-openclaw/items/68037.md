## Summary

Keep open. Current main still maps HTTP MCP tool-result content blocks down to only `type` and `text`, so image/audio/resource payload fields are still dropped; open PR #68046 remains the focused closing implementation candidate.

## What This Changes

The issue requests that the HTTP MCP `tools/call` normalizer preserve non-text MCP content block fields instead of reducing every result block to `type`/`text`.

## Best Possible Solution

Review and land #68046, or an equivalent focused fix, so the HTTP MCP normalizer preserves valid text, image, audio, embedded resource, and resource_link blocks while keeping text fallback behavior for strings or malformed entries.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: any loopback tool returning `{ content: [{ type: "image", data, mimeType }] }` through HTTP MCP `tools/call` is transformed into an image-typed block without top-level `data` or `mimeType`.

## Solution Assessment

Yes. The narrow maintainable fix belongs in `normalizeToolCallContent` with regression coverage for MCP content block variants; the linked open PR is already the right implementation candidate.
