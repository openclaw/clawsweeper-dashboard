## Summary

Keep open. Current main still strips non-text MCP content block payload fields on the HTTP `tools/call` response path, and https://github.com/openclaw/openclaw/pull/68046 is the open closing implementation candidate.

## What This Changes

Preserve valid MCP non-text content blocks returned by HTTP MCP `tools/call` instead of rewriting every tool result to a text-shaped object.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/68046 or a narrow replacement that preserves valid MCP text, image, audio, embedded resource, and resource_link blocks while retaining safe text fallback behavior for malformed entries.

## Reproduction Assessment

Yes. A loopback tool returning `{ content: [{ type: "image", data, mimeType }] }` through HTTP MCP `tools/call` deterministically reaches `normalizeToolCallContent`, which currently returns an image-typed object without the required top-level payload fields.

## Solution Assessment

Yes. The narrow maintainable fix belongs at `normalizeToolCallContent` with regression coverage for the HTTP `tools/call` path and the MCP content block variants; the linked PR is the right review path, not early issue closure.
