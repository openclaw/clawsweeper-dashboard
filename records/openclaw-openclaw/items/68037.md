## Summary

Keep open. Current main still drops non-text MCP content-block payload fields on the HTTP `tools/call` response path, and #68046 is an open, unmerged closing PR for the same bug, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue asks the HTTP MCP gateway to preserve image, audio, embedded resource, and resource_link content-block fields in tool-call results instead of normalizing every block to only `type` and `text`.

## Best Possible Solution

Review and revise #68046 as the focused implementation candidate: preserve valid MCP text, image, audio, embedded resource, and resource_link blocks, fall back to text for strings or malformed entries, add regression coverage on the HTTP `tools/call` path, and let the issue close after that fix lands.

## Reproduction Assessment

Yes. A source-level reproduction is clear: any loopback tool returning `{ content: [{ type: "image", data, mimeType }] }` through HTTP MCP `tools/call` is mapped to a block containing only `type` and `text`; the browser screenshot path naturally returns that image content shape.

## Solution Assessment

Yes, the narrow maintainable fix belongs in `src/gateway/mcp-http.handlers.ts` at the content normalizer boundary. The linked PR is the right implementation candidate, but the safest final shape should preserve only valid MCP content block variants rather than blindly passing through every object with a string `type`.
