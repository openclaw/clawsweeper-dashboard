## Summary

Keep open. Current main still strips non-text MCP content-block payload fields on the HTTP `tools/call` path, and open PR #68046 is already the focused closing implementation candidate.

## What This Changes

The issue asks the HTTP MCP gateway to preserve image, audio, embedded resource, and resource_link content-block fields instead of normalizing every tool result to `type` and `text`.

## Best Possible Solution

Review and land #68046, or an equivalent focused fix, so the HTTP MCP normalizer preserves valid MCP content blocks while retaining safe text fallback behavior for strings or malformed entries.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is any loopback tool returning `{ content: [{ type: "image", data, mimeType }] }` through HTTP MCP `tools/call`; current main maps that block down to `type` and `text` before returning the JSON-RPC result.

## Solution Assessment

Yes. The narrow maintainable fix belongs in `normalizeToolCallContent` with endpoint regression coverage; because #68046 is open and references this issue with closing syntax, this issue should close only after that PR or an equivalent fix merges.
