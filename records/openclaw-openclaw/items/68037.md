## Summary

Keep open. Current main still drops the top-level payload fields from non-text MCP content blocks on the HTTP `tools/call` path, and open PR #68046 is already the focused closing implementation candidate.

## What This Changes

Preserve image, audio, embedded-resource, and resource_link MCP content fields in HTTP gateway `tools/call` results instead of normalizing every block to `{type,text}`.

## Best Possible Solution

Review and land #68046, or an equivalent focused fix, so the HTTP MCP normalizer preserves valid text, image, audio, embedded resource, and resource_link blocks while keeping text fallback behavior for strings or malformed entries.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: any loopback tool returning `{ content: [{ type: "image", data, mimeType }] }` through HTTP MCP `tools/call` is transformed into an image-typed block without top-level `data` or `mimeType`.

## Solution Assessment

Yes. The narrow maintainable fix belongs at `normalizeToolCallContent`, with regression coverage for each MCP content block variant; the open PR is the right implementation candidate and this issue should close only after a fix lands.
