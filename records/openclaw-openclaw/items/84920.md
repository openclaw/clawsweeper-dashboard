## Label Justifications

- `P1`: A real Windows Control UI image workflow is broken when uploaded media reaches OpenClaw but cannot be read by the image tool.
- `impact:message-loss`: The issue is about an uploaded image attachment becoming inaccessible to the agent/tool flow even though it was saved successfully.

## Summary

Keep open: current main still has a source-visible path where the image tool rewrites `media://inbound/...` through `workspaceDir` before the shared media-store resolver can map it to `.openclaw/media/inbound`, matching the Windows path in the report.

## What This Changes

The issue reports that Control UI image uploads saved as `media://inbound` refs later resolve under the Windows workspace as `media:\inbound`, causing the image tool to miss the stored file.

## Best Possible Solution

Preserve `media://inbound` as a media-store reference through image-tool loading, or resolve it with the shared media-reference helper before workspace-relative path handling, while keeping ordinary workspace-relative image paths working.

## Reproduction Assessment

Yes, at source level: current `image-tool.ts` resolves non-file/non-http strings against `workspaceDir` without excluding `media://` refs, and a `path.win32.resolve` probe reproduces the malformed Windows path. I did not perform a full Windows Gateway/Control UI run in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is in the image/media reference boundary, not a new Control UI feature or config option. Add a regression where `media://inbound/<id>` is passed to the image tool with `workspaceDir` set.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/84920. Current `src/agents/tools/image-tool.ts` treats `media://inbound/<id>` as acceptable but then resolves it against `options.workspaceDir`, producing malformed paths such as `C:\\Users\\...\\workspace\\media:\\inbound\\...` before `loadWebMedia` can map the media-store URI to `.openclaw/media/inbound`. Preserve raw media-store refs through the shared media loader, or resolve them with the existing media-reference helper before workspace-relative path handling. Keep ordinary relative workspace image paths working; do not change the Control UI upload contract, media store layout, or local-root security policy. Add focused regression coverage where the image tool receives `media://inbound/<id>` with `workspaceDir` set, and include a changelog fix entry if this repair PR is promoted.
