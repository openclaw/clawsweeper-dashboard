## Label Justifications

- `P1`: The issue reports a real user-facing Control UI image-understanding workflow failure where uploaded images cannot be read by the image tool.
- `impact:message-loss`: The uploaded image reference reaches OpenClaw but is effectively misrouted to a malformed local path before the image tool can consume it.

## Summary

Close as duplicate: the Windows Control UI report adds useful logs, but the failure path matches the existing open media-store image-tool resolver issue at https://github.com/openclaw/openclaw/issues/74123, which already has open fix PRs targeting the same malformed `workspace/media:/inbound` behavior.

## What This Changes

Fix Control UI image uploads on Windows so `media://inbound` references resolve to the managed media store path instead of a malformed workspace path.

## Best Possible Solution

Track the remaining resolver work under https://github.com/openclaw/openclaw/issues/74123 and review or consolidate the open fixes so managed `media://inbound` refs are resolved before any workspace or sandbox path join.

## Reproduction Assessment

Yes, source-level. Current main passes `workspaceDir` into the image tool, the image-tool resolver does not exclude `media://inbound` before workspace resolution, and a read-only Node check reproduces the reported Windows malformed path shape.

## Solution Assessment

Yes. Closing this satellite as a duplicate is the best triage path because the same root cause is already tracked in the canonical open issue with open PRs targeting the relevant resolver seams.
