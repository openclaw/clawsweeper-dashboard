## Summary

Keep open: the Android assistant hardening goal is useful, but this branch is not mergeable as-is because it conflicts with current main's draft-only exported ASK_OPENCLAW policy, can drop fresh restored assistant launches, and still lacks real Android behavior proof.

## What This Changes

The PR adds prompt trimming/capping, assistant-launch fingerprinting, SavedStateHandle-backed assistant routing state, and Android unit tests for assistant/App Action launches.

## Best Possible Solution

Rebase or replace the branch so prompt capping and SavedStateHandle restoration are kept, ASK_OPENCLAW remains draft-only, restored-launch suppression is freshness-aware, and redacted Android runtime proof is added.

## Reproduction Assessment

Yes. Source review of the PR head shows ASK_OPENCLAW would auto-send while current main and tests require draft-only handling, and the restored-launch check drops any intent with a matching fingerprint; I did not run device-level Android proof in this read-only review.

## Solution Assessment

No. The hardening direction is reasonable, but this branch needs a rebase and code changes to preserve current security behavior and avoid fresh-launch suppression before it is the narrow maintainable fix.
