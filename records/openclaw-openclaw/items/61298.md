## Label Justifications

- `P1`: The PR targets a broken user-facing assistant flow, but its restoration gate can itself silently drop a real invocation.
- `merge-risk: 🚨 message-delivery`: The new fingerprint equality check directly decides whether an Android assistant request reaches the chat routing path.

## Summary

This remains useful work because current main still accepts unbounded App Action prompts and keeps assistant routing transient, but the PR is not merge-ready: its payload-only restoration fingerprint can suppress a legitimate new Android assistant invocation, the branch conflicts with current main’s rewritten activity initialization, and the contributor supplied no real-device lifecycle proof.

## What This Changes

The PR caps and trims Android App Action prompts, fingerprints handled assistant launches across activity restoration, and persists transient chat-routing state through SavedStateHandle.

## Best Possible Solution

Rebase onto current main, preserve bounded prompt sanitization and state restoration, but replace payload-only suppression with an explicit consumed-launch lifecycle identity that cannot hide a fresh invocation; then prove restored replay prevention and a fresh identical launch on a real Android device.

## Reproduction Assessment

No high-confidence real-device reproduction is provided; source inspection does establish the ambiguous identical-intent path, but actual task/process restoration remains unverified.

## Solution Assessment

No. Prompt bounding and SavedStateHandle persistence are appropriate, but payload equality is not a reliable consumed-launch identity and the implementation must be adapted to current main’s deferred initialization flow.
