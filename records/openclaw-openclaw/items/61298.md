## Summary

Keep open: the Android assistant hardening direction is useful, but this PR head is not mergeable as-is because it reintroduces auto-send for exported ASK_OPENCLAW prompts, can suppress fresh assistant launches with a fingerprint-only stale check, and only provides unit/compile proof rather than real Android runtime evidence.

## What This Changes

The PR trims and caps Android App Action prompts, fingerprints/restores assistant launch state, persists assistant routing state through SavedStateHandle, and adds Android unit tests.

## Best Possible Solution

Rebase or replace the branch so prompt capping and SavedStateHandle restoration are kept, ASK_OPENCLAW remains draft-only, restored-launch suppression is freshness-aware, and redacted Android runtime proof is added.

## Reproduction Assessment

Yes for the PR defects by source review: PR head sets ASK_OPENCLAW prompts to auto-send and returns early for matching restored fingerprints. I did not run device-level Android proof in this read-only review.

## Solution Assessment

No. The hardening direction is reasonable, but this branch is not the best fix until it preserves current exported prompt confirmation behavior and distinguishes restored launches from fresh invocations.
