## Summary

Keep open. The PR contains useful Android assistant hardening work, but the current branch would regress the current ASK_OPENCLAW confirmation boundary, its restored-launch guard can still suppress fresh launches, and the PR body only provides unit/compile proof rather than real Android runtime proof.

## What This Changes

The PR branch changes Android assistant/App Action launch parsing and lifecycle handling by adding prompt capping, assistant-launch fingerprinting, SavedStateHandle-backed routing state, and unit tests.

## Best Possible Solution

Rework or replace the branch so prompt capping and SavedStateHandle restoration are kept, ASK_OPENCLAW remains draft-only, restored-launch suppression is freshness-aware, and redacted Android runtime proof is added.

## Reproduction Assessment

Yes. Source review of the PR head reproduces the two review findings: ASK_OPENCLAW prompts become auto-send, and restored-launch suppression returns early on a matching fingerprint; I did not run real Android device or emulator proof in this read-only review.

## Solution Assessment

No. The hardening direction is reasonable, but this branch is not the best mergeable fix until it preserves the current security boundary and distinguishes restored stale launches from fresh identical launches.
