## Summary

Keep this PR open: it targets a real heartbeat scheduler bug, but the current patch throttles a stale schedule into repeated 2-second wakes instead of repairing the stale schedule state, and the latest proof is still fake-timer/CI output rather than real runtime evidence.

## What This Changes

The PR adds a 2 s floor to heartbeat scheduler refires when next due time is in the past, adds fake-timer scheduler tests, adjusts a Discord voice test mock signature, and adds a changelog entry.

## Best Possible Solution

Repair or clear stale heartbeat nextDueMs after terminal skipped/deferred scheduled outcomes so the runner returns to the configured cadence, with focused regression coverage and redacted real runtime proof.

## Reproduction Assessment

Yes, by source inspection: current main can leave nextDueMs in the past after a skipped/disabled scheduled outcome and then re-arm scheduleNext with delay 0. I did not establish a live Raspberry Pi or Docker crash-loop reproduction in this read-only review.

## Solution Assessment

No. The timer floor prevents an immediate setTimeout(0) hot loop, but it still wakes repeatedly every 2 seconds for stale schedules instead of restoring the configured heartbeat cadence.
