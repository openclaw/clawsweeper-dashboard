## Summary

This PR should stay open: the branch addresses the immediate zero-delay timer symptom, but it does not repair the stale heartbeat schedule state that feeds the loop, and the external PR still has only mock/unit proof for the reported Raspberry Pi/Docker failure.

## What This Changes

The PR adds a 2 s heartbeat scheduler refire floor for past-due timers, fake-timer scheduler tests, a Discord voice mock signature tweak, and a changelog entry.

## Best Possible Solution

Repair stale heartbeat schedule state after terminal skipped/disabled scheduled outcomes so agents return to configured cadence, then prove CPU and polling recovery with redacted real gateway evidence.

## Reproduction Assessment

Yes for the source-level scheduler hot-loop path: current main can leave `nextDueMs` stale after `skipped/disabled`, compute a 0 ms delay, and re-arm from `finally`. I did not establish a live Raspberry Pi/Docker crash-loop reproduction in this read-only review.

## Solution Assessment

No. A 2 s floor avoids the tightest timer loop, but the maintainable fix is to advance, clear, or recompute the stale schedule state instead of polling stale agents every 2 seconds.
