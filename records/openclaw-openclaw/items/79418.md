## Summary

Keep open: the PR targets a real source-level heartbeat scheduler hot-loop, but the patch only changes a stale due time from a 0 ms loop into repeated 2 s polling and still lacks real gateway/RPi/Docker behavior proof.

## What This Changes

The PR adds a 2 s minimum delay for past-due heartbeat scheduler timers, fake-timer scheduler coverage, a Discord voice test mock signature tweak, and a changelog entry.

## Best Possible Solution

Repair stale heartbeat schedule state after terminal skipped or deferred scheduled outcomes so agents return to the configured cadence, with focused regression coverage and redacted real gateway proof.

## Reproduction Assessment

Yes, by source inspection: current main can leave nextDueMs in the past and re-arm scheduleNext with a 0 ms delay after skipped/disabled scheduled outcomes. I did not establish a live Raspberry Pi or Docker crash-loop reproduction in this read-only review.

## Solution Assessment

No. A 2 s floor avoids the tightest setTimeout(0) loop, but the maintainable fix is to repair or clear the stale schedule state instead of polling stale agents every 2 seconds.
