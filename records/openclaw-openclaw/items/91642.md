## What This Changes

The PR updates the docs i18n Go script so doc-mode `--allow-partial` continues later queued files after a per-file translation error while preserving default fail-fast behavior.

## Best Possible Solution

Land the bounded docs-i18n loop change after normal maintainer CI/merge handling, keeping `--allow-partial` as the explicit opt-in for successful later docs to survive per-file translation failures.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: on current main, doc-mode sequential processing returns immediately on the first per-file error and parallel processing cancels the worker context after an error, so later queued docs are skipped. I did not run tests because this review is read-only.

## Solution Assessment

Yes, this appears to be the narrow maintainable fix: it keeps the output-writing contract in processFileDoc, passes the existing allowPartial option into the doc loops, and preserves fail-fast/cancellation behavior. Moving the behavior into postprocessing or processFileDoc would not let later files regenerate and would risk mixing failure handling with output writes.
