## Summary

Keep open. The PR targets a real QQBot stale-binding bug with a narrow runtime approach, but it still lacks required real QQBot after-fix proof and the new access-stage regression test currently fails type/lint checks.

## What This Changes

The PR adds a QQBot active runtime-config provider, wires it into inbound routing and outbound dispatch, tightens the gateway config type, adds QQBot regression tests, adjusts one Codex plugin test cast, and records the fix in the changelog.

## Best Possible Solution

Land the QQBot per-inbound runtime config lookup after repairing the access-stage regression harness and adding redacted live QQBot/log proof that a binding edit routes correctly without restart.

## Reproduction Assessment

Yes. The linked bug provides concrete QQBot C2C steps and routing logs, and current-main source still passes the startup config object into a resolver that caches by config reference; I did not run a live QQBot account.

## Solution Assessment

Mostly yes, but not in this mergeable form. Resolving active runtime config at the QQBot inbound boundary is the narrow maintainable fix, while the regression test harness and real behavior proof still need follow-up.
