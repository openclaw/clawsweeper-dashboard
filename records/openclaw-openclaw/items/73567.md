The PR adds a QQBot active runtime-config provider, wires fresh config into per-inbound context construction and outbound dispatch, adds QQBot regression tests and a changelog entry, tightens the gateway cfg type, and removes a redundant Codex plugin test cast.

## Best Possible Solution

## Reproduction Assessment

## Solution Assessment

Yes. Resolving the active runtime config once at the QQBot inbound boundary is the narrowest maintainable fix and matches the existing Telegram per-event routing pattern; broad route-cache invalidation would be a less targeted change.
