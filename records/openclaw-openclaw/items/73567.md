## Summary

Keep open: the PR targets a real QQBot dynamic-binding bug and the implementation direction is narrow, but it is not merge-ready because the new access-stage regression test is written against a Promise instead of the resolved result, and the external real-behavior proof gate is still unsatisfied.

## What This Changes

The PR adds a QQBot active runtime-config provider, uses it for inbound routing and outbound dispatch, tightens the gateway config type, adds QQBot regression tests, adjusts one Codex plugin test cast, and records the fix in the changelog.

## Best Possible Solution

Land the QQBot-side per-inbound active runtime config lookup after repairing the regression test harness and adding redacted real QQBot/log proof; no broad core cache invalidation is needed.

## Reproduction Assessment

Yes. The linked bug supplies concrete QQBot C2C steps and logs, and source inspection on current main shows QQBot routes with the startup config while the resolver caches by config object; I did not run a live QQBot account in this read-only review.

## Solution Assessment

Mostly yes, but this PR is not currently the best mergeable form. The active-config lookup at the QQBot inbound boundary is the narrow maintainable fix, but the regression test must await the async stage and provide its access dependency before the solution is safe to land.
