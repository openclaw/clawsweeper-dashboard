## Summary

Keep open. The QQBot active-config fix is the right narrow direction for the stale-binding bug, but this PR still has a blocking regression-test bug and only test/CI proof rather than real QQBot after-fix proof.

## What This Changes

The branch adds a QQBot active runtime-config provider, wires inbound routing and outbound dispatch to use it per message, tightens gateway config typing, adds QQBot regression tests, adjusts one Codex test cast, and updates the changelog.

## Best Possible Solution

Land the QQBot per-inbound runtime-config lookup after fixing the async access-stage regression test and adding redacted live QQBot/log proof; no broad core routing-cache change is needed.

## Reproduction Assessment

Yes. The linked report at https://github.com/openclaw/openclaw/issues/69546 gives concrete QQBot C2C steps and before/after routing logs, and current main still passes the startup `ctx.cfg` into routing while `bindings` are no-restart config; I did not exercise a live QQBot account in this read-only review.

## Solution Assessment

Mostly yes. Resolving active runtime config at the QQBot inbound boundary is the narrow maintainable fix, but this PR is not yet the best mergeable form because its regression test does not await the async stage and real behavior proof is missing.
