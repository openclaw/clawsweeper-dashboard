## Label Justifications

- `P1`: Internal tool-call scaffolding can leak into visible channel messages across configured channels, making this an urgent channel-delivery safety bug rather than low-risk polish.
- `impact:security`: The issue is about preventing internal tool XML, commands, and operational context from being exposed to chat participants.

## Summary

Keep open: current main already sanitizes normal final reply payloads in shared reply normalization, but the outbound/message-tool path still depends on per-channel `sanitizeText` hooks and Feishu/Telegram do not provide one, so XML tool-call scaffolding can still reach user-visible channel sends.

## What This Changes

The issue asks for Feishu and other non-Discord outbound text to run through `sanitizeAssistantVisibleText()` so internal tool-call XML does not leak into chat messages.

## Best Possible Solution

Add a shared outbound visible-text sanitization step, or consistently wire `sanitizeAssistantVisibleText()` into affected channel outbound adapters, with focused tests for Feishu and Telegram message-tool sends plus formatting-preservation cases.

## Reproduction Assessment

Yes at source level: a message-tool send with `<function_calls>` or `<invoke name="exec">` can reach Feishu or Telegram outbound delivery because those adapters lack `sanitizeText`, while shared delivery only calls that hook when present. I did not run a live Feishu or Telegram account reproduction in this read-only review.

## Solution Assessment

Yes, the maintainable direction is to reuse the existing assistant-visible sanitizer in the shared outbound/channel adapter layer rather than adding a new config flag. The exact placement should avoid double-sanitizing already-normalized final replies and preserve channel formatting contracts.
