## Label Justifications

- `P1`: Internal reasoning/tool scaffolding can reach real user-facing channels, which is an urgent agent/channel workflow and privacy failure.
- `impact:security`: The issue is about leaking internal assistant scaffolding, reasoning, and runtime/tool traces into external chat surfaces.
- `impact:message-loss`: Raw scaffolding can corrupt, hide, or replace the intended user-visible answer in messaging channels.

## Summary

Keep open: current main and the latest release still leave Feishu without a channel-level assistant-visible sanitizer hook, and the linked PRs/issues only cover partial channel or tag-shape slices of the broader leak.

## What This Changes

The issue asks OpenClaw to apply the canonical assistant-visible text sanitizer to Feishu and other non-Discord outbound channel messages so internal tool, reasoning, and trace scaffolding is not delivered to users.

## Best Possible Solution

Adopt a maintainer-approved canonical outbound sanitizer boundary, or add equivalent `sanitizeText` hooks for every uncovered channel adapter, with Feishu plain text/card coverage and focused regressions for tool XML, reasoning, relevant-memory, and trace-line leaks.

## Reproduction Assessment

Yes, source-reproducible: Feishu’s outbound adapter lacks `sanitizeText`, and its send/card functions build payloads from raw text while the shared delivery path only calls a sanitizer when the channel provides one. I did not run a live Feishu credentialed repro.

## Solution Assessment

No, current main is not the best final state because it has the canonical sanitizer and some per-channel use, but Feishu and related channel slices still lack guaranteed application. The best fix is a maintainer-approved shared boundary or complete per-channel hook coverage with regression tests.
