## Summary

Keep open. Current main still hardcodes `fastMode: true` for hidden commitment extraction, which feeds the MiniMax fast-mode wrapper and can rewrite `MiniMax-M2.7` to `MiniMax-M2.7-highspeed`; the linked closing PR is still open and conflicting, so this should close only after that fix or an equivalent one lands.

## What This Changes

The issue reports that commitment extraction forces MiniMax fast mode and fails for MiniMax Coding Plan users who have `MiniMax-M2.7` but not `MiniMax-M2.7-highspeed`.

## Best Possible Solution

Pass the resolved fast-mode state for the selected commitment extractor model into `runEmbeddedPiAgent`, defaulting to false for MiniMax base-plan users while preserving explicit fast-mode configuration, then land the linked PR or an equivalent replacement with regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows commitment extraction always passes `fastMode: true`, and the MiniMax wrapper rewrites `MiniMax-M2.7` to `MiniMax-M2.7-highspeed` when that flag is true; the reporter also supplied matching live HTTP 500 logs.

## Solution Assessment

Yes for the bug-fix direction. The narrowest maintainable fix is to reuse `resolveFastModeState` for the extractor model rather than add a commitments-specific config field or drop configured fast-mode behavior entirely.
