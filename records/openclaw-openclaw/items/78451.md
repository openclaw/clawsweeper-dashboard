## Summary

This should stay open: current main still forces hidden commitment extraction into fast mode, and the MiniMax provider intentionally rewrites MiniMax-M2.7 to MiniMax-M2.7-highspeed when fast mode is true.

## What This Changes

The issue asks OpenClaw to stop commitment extraction from forcing MiniMax-M2.7-highspeed on MiniMax plans that only support MiniMax-M2.7.

## Best Possible Solution

Make hidden commitment extraction stop forcing fast mode and add a regression test showing MiniMax-M2.7 is not upgraded to highspeed unless the existing agent/model fast-mode configuration explicitly enables it.

## Reproduction Assessment

Yes from source inspection: enabling commitments with MiniMax-M2.7 reaches `runEmbeddedPiAgent` with `fastMode: true`, and the MiniMax stream wrapper rewrites that model to MiniMax-M2.7-highspeed. I did not run a live MiniMax non-highspeed account path, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

No, the current implementation is not the best solution because the hidden extractor overrides the user's fast-mode policy. The narrow maintainable fix is to derive fast mode through the existing agent/model policy or omit the override, not to add a new commitments config knob by default.
