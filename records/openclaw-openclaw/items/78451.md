## Summary

Keep open. Current main and the latest release still force `fastMode: true` for hidden commitment extraction, and MiniMax fast mode rewrites `MiniMax-M2.7` to `MiniMax-M2.7-highspeed`, matching the reported non-highspeed-plan failure.

## What This Changes

Commitment extraction should not force MiniMax highspeed when the configured MiniMax plan/model only supports `MiniMax-M2.7`.

## Best Possible Solution

Stop forcing fast mode in commitment extraction and rely on the existing agent/model fast-mode controls, with regression coverage that MiniMax extraction stays on `MiniMax-M2.7` unless fast mode is explicitly enabled.

## Reproduction Assessment

Yes, by source inspection: hidden commitment extraction always passes `fastMode: true`, and the MiniMax wrapper rewrites `MiniMax-M2.7` to `MiniMax-M2.7-highspeed`. I did not run a live non-highspeed MiniMax account, but the code path matches the reported logs.

## Solution Assessment

Yes, the narrow maintainable fix is to remove the unconditional extractor fast-mode override or derive it from existing fast-mode settings. Adding a new commitments-specific fast-mode config would be broader than needed.
