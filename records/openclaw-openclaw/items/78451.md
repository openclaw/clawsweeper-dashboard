## Summary

Keep open. Current main and the latest shipped release still force `fastMode: true` in hidden commitment extraction, which drives the MiniMax wrapper to rewrite `MiniMax-M2.7` to `MiniMax-M2.7-highspeed`; the linked closing PR remains open and conflicting, so this should not close before that fix or an equivalent one lands.

## What This Changes

Commitment extraction should stop forcing MiniMax highspeed mode for users whose MiniMax plan only supports the base `MiniMax-M2.7` model.

## Best Possible Solution

Pass the existing resolved fast-mode state for the selected extractor model into commitment extraction, defaulting to false for MiniMax base-plan users while preserving explicit fast-mode configuration, then land the linked PR or an equivalent replacement with regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows hidden commitment extraction always passes `fastMode: true`, and the MiniMax stream wrapper rewrites `MiniMax-M2.7` to `MiniMax-M2.7-highspeed` when that flag is true; the reporter also supplied matching live error logs.

## Solution Assessment

Yes for the overall direction, but the narrowest maintainable fix is to use the existing fast-mode resolver rather than add a commitments-specific config field or drop configured fast-mode behavior entirely.
