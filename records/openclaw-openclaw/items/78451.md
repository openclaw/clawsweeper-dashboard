## Summary

Keep open. Current main and the latest release still pass `fastMode: true` from hidden commitment extraction, and the open closing PR remains the canonical implementation path rather than a reason to close this issue before merge.

## What This Changes

The issue asks OpenClaw commitments extraction to stop forcing MiniMax fast-mode/highspeed model selection for users whose MiniMax plan only supports the base MiniMax-M2.7 model.

## Best Possible Solution

Land or replace `https://github.com/openclaw/openclaw/pull/78459` with a fix that replaces the hardcoded true with the shared resolved fast-mode state and regression coverage for default-off plus explicit opt-in behavior.

## Reproduction Assessment

Yes, at source level: current main passes `fastMode: true` in the hidden commitment extractor, and the MiniMax wrapper rewrites `MiniMax-M2.7` to `MiniMax-M2.7-highspeed` when that flag is true. I did not run a live MiniMax OAuth account in this read-only review.

## Solution Assessment

Yes, the fix belongs in commitment extraction by using the shared fast-mode decision path rather than adding a new commitments-specific config field. The issue should stay open until the linked PR or an equivalent fix lands.
