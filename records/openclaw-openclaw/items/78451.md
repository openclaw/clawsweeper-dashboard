## Summary

Keep open. Current `main` and the latest release still force `fastMode: true` in hidden commitment extraction, and the linked closing PR is open, so this issue should close only after that PR or an equivalent fix lands.

## What This Changes

Commitment extraction for MiniMax M2.7 forces fast mode, which routes background extraction to the unsupported `MiniMax-M2.7-highspeed` variant on non-highspeed plans.

## Best Possible Solution

Land `https://github.com/openclaw/openclaw/pull/78459` or an equivalent fix that removes the hardcoded true by using the shared fast-mode decision path and covers default-off plus explicit opt-in behavior.

## Reproduction Assessment

Yes from source and reporter logs: current `main` passes `fastMode: true` in the commitment extractor, and the MiniMax wrapper rewrites `MiniMax-M2.7` to `MiniMax-M2.7-highspeed` when that flag is true. I did not run a live MiniMax OAuth account in this read-only review.

## Solution Assessment

Yes, the fix belongs in commitment extraction rather than a new commitments-specific config surface. The safest shape is to remove the unconditional true while preserving the shared resolved fast-mode behavior for explicit opt-ins.
