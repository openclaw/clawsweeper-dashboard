## Summary

Keep open. Current main and the latest release still hardcode `fastMode: true` for hidden commitment extraction, and the linked closing PR is still open and conflicting, so this issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue reports that commitment extraction forces MiniMax fast mode and requests `MiniMax-M2.7-highspeed` even when the operator’s plan only supports `MiniMax-M2.7`.

## Best Possible Solution

Land the linked PR or an equivalent replacement that passes the existing resolved fast-mode state into commitment extraction and adds regression coverage for default-false MiniMax behavior and configured-fast behavior.

## Reproduction Assessment

Yes. Source inspection on current main shows hidden commitment extraction always passes `fastMode: true`, and the MiniMax wrapper rewrites `MiniMax-M2.7` to `MiniMax-M2.7-highspeed` when that flag is true; the reporter supplied matching live HTTP 500 logs.

## Solution Assessment

Yes for fixing this issue by changing commitment extraction, but the narrowest maintainable implementation should use `resolveFastModeState(...).enabled` for the selected extractor model instead of adding a commitments-specific config field or dropping configured fast mode entirely.
