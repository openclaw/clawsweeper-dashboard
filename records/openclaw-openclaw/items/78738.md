## Summary

Keep open: current main still has best-effort approval delivery and followup paths that log transport/outbound failures without turning them into an agent-visible approval failure; the adjacent open PR only suppresses stale warning noise.

## What This Changes

The issue asks OpenClaw to return a visible exec approval delivery error to the agent when Discord approval dispatch or followup delivery fails.

## Best Possible Solution

Make failed approval prompt delivery and failed async followup delivery transition into an existing visible approval failure path, with regression coverage for forwarding, native Discord delivery, and followup fallback failures.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: injected outbound/session-followup failures flow into log-only catch paths or best-effort notices instead of a returned approval failure. I did not run a live Discord/WebSocket scenario in this read-only review.

## Solution Assessment

Yes, surfacing these failures through the existing no-approval-route or approval-request-failed mechanisms is the narrow maintainable direction. A pure log suppression change would not solve the reported dropped-command behavior.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/78738. Current main can accept or continue exec approval requests even when the actual Discord/native forwarding or async followup delivery fails, leaving only warning logs such as `exec approval followup dispatch failed` or `Outbound not configured for channel: discord`. Keep the change inside the existing exec approval runtime/followup flow; do not add a config option or broaden channel policy. Make initial approval forwarding/native delivery failures produce a visible approval failure/denial through existing `no-approval-route` or `approval-request-failed` semantics when no prompt was actually delivered, and make async followup dispatch failures surface to the agent/user when possible instead of only `logWarn`. Add regression coverage for generic forwarding failure, native approval zero-delivered-target failure, and followup fallback failure. Preserve the intent of https://github.com/openclaw/openclaw/pull/66685 if it lands first: stale `unknown or expired approval id` noise can remain suppressed, but real channel/transport/session failures must stay visible. Add a single-line CHANGELOG.md Fixes entry because this is user-facing behavior.
