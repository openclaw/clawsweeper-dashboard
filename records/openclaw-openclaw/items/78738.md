## Summary

Keep this open: current main and the latest release still accept configured approval forwarding before proving that the approval prompt was delivered, so all-target dispatch failures can still leave the agent waiting instead of receiving an immediate visible failure. The closed unmerged repair https://github.com/openclaw/openclaw/pull/82506 points at the right surface, while the open https://github.com/openclaw/openclaw/pull/66685 only suppresses expired follow-up warning noise and does not own this central bug.

## What This Changes

The issue asks OpenClaw to return an explicit exec approval delivery failure to the agent instead of silently leaving approval-gated commands pending when Discord or another configured approval route cannot dispatch.

## Best Possible Solution

Update the shared approval delivery contract so configured-target request delivery waits for the initial delivery outcome, fails visibly when every attempted route fails, preserves pending behavior when at least one route succeeds, and covers exec plus plugin approval siblings with focused regression tests.

## Reproduction Assessment

Yes for a source-level reproduction: inject or mock a configured approval target whose durable delivery returns `status: "failed"`, and current `handleRequested` can still return true before the failure reaches `handlePendingApprovalRequest`. I did not establish a live Discord/WebSocket failure path in this read-only pass.

## Solution Assessment

No, current main is not the best solution because it only fast-fails missing routes, not configured routes whose first delivery fails. A current-code version of the https://github.com/openclaw/openclaw/pull/82506 approach is a better fit than the retry-only https://github.com/openclaw/openclaw/pull/78766 or warning-only https://github.com/openclaw/openclaw/pull/66685 paths.
