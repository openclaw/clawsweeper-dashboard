## Summary

Keep this open: current main and the latest release still count configured approval forwarding as accepted before proving that any target actually received the prompt, so all-target dispatch failures can still strand an approval-gated exec request.

## What This Changes

The issue reports that exec approval delivery failures can leave approval-gated commands stranded without returning an error to the agent.

## Best Possible Solution

Make approval forwarding report delivery success only after at least one configured route actually accepts the pending prompt, and return a visible failure/null decision when every explicit route fails.

## Reproduction Assessment

Yes, from source inspection: an approval request with configured forwarding targets can return accepted before the target send succeeds. If every target send later fails, the gateway has already skipped the no-route failure path.

## Solution Assessment

Yes, a narrow fix in approval forwarding/request handling is the best path. A retry-only follow-up fix is too late for the initial prompt delivery failure because the gateway has already accepted the pending request.

## Repair Work Prompt

Fix the exec/plugin approval forwarding delivery-failure bug described in https://github.com/openclaw/openclaw/issues/78738 and related follow-up https://github.com/openclaw/openclaw/issues/89217. On current main, `createExecApprovalForwarder().handleRequested(...)` returns true after scheduling pending delivery even if all configured target sends later fail, so `handlePendingApprovalRequest(...)` skips the no-route/null-decision path and an approval-gated exec can remain pending without a visible agent error. Change the owner-boundary path so configured explicit/native approval targets count as delivered only when at least one route actually accepts the prompt; if every configured route fails or outbound is unavailable, clear/expire the pending request and surface a visible null/error result to the requester instead of falling back silently. Preserve successful session/target forwarding, iOS push behavior, and legitimate turn-source fallback. Add focused regression tests for all-target delivery failure and no-route behavior after explicit dispatch failure. Do not edit CHANGELOG.md.
