## Summary

Keep open: the QQBot retry hardening is focused and current main still uses a fixed retry sleep, but this PR is not merge-ready because the advertised max delay is not a true cap after jitter, the new helper still fails lint, the posted proof only exercises the pure helper, and maintainers should accept the 5-minute cooldown tradeoff.

## What This Changes

The PR adds QQBot background token-refresh exponential backoff, jitter, circuit-breaker defaults, a retry-delay helper, and helper-level tests.

## Best Possible Solution

Keep the QQBot-owned retry hardening, but clamp the final jittered delay to the configured maximum, replace Math.pow with exponentiation, and require redacted runtime proof of the background refresh failure and recovery path before merging the chosen cooldown defaults.

## Reproduction Assessment

Yes at source level: current main retries QQBot background token-refresh failures with a fixed 5-second sleep in the catch path. I did not live-force a bots.qq.com outage or observe a full runtime recovery sequence.

## Solution Assessment

No, not as submitted: the QQBot token manager is the right owner boundary, but the final delay can exceed the advertised cap and the real background-loop behavior still needs proof. A safer version clamps after jitter, clears lint, and demonstrates failure/recovery with redacted runtime output.
