## Summary

Keep open: current main and the latest release still use the fixed 5-second QQBot background token-refresh retry, so the useful mitigation is not obsolete, but this PR is not merge-ready because the configured max delay can be exceeded, lint fails on the new exponentiation code, runtime-loop proof is still insufficient, and the five-minute cooldown is an availability policy tradeoff.

## What This Changes

The PR adds QQBot token-refresh exponential backoff, jitter, circuit-breaker retry defaults, a retry-delay helper, and helper-level tests.

## Best Possible Solution

Land a narrowed QQBot-owned retry hardening change that clamps final jittered delays, clears lint, covers the background loop, and includes redacted runtime/log proof plus explicit maintainer acceptance of cooldown defaults.

## Reproduction Assessment

Yes at source level: current main and the latest release retry QQBot background token-refresh failures with a fixed 5-second sleep in the catch path. I did not live-force a Tencent outage or run a QQBot account through the full recovery sequence in this read-only review.

## Solution Assessment

No, not as submitted: the QQBot token manager is the right owner boundary, but the final delay can exceed the advertised cap, lint is failing, and runtime-loop behavior still needs proof. A safer version clamps after jitter and demonstrates failure/recovery with redacted output.
