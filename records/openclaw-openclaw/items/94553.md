## Summary

Keep this PR open, but not merge-ready: the QQBot token manager is the right owner boundary for retry hardening, yet the patch still lets jitter exceed the documented max delay, leaves a diff-caused lint failure, and only proves the pure helper instead of the wired background refresh loop.

## What This Changes

The PR adds optional QQBot background token refresh retry settings, a delay helper with exponential backoff/jitter/cooldown behavior, and helper-level tests.

## Best Possible Solution

Keep the QQBot-owned retry hardening, but make maxRetryDelayMs a true final cap, clear the lint failure, and require redacted runtime proof of the background loop through failure and recovery before merging the chosen defaults.

## Reproduction Assessment

Yes at source level: current main retries QQBot background token-refresh failures with a fixed 5-second sleep in the catch path. I did not live-force a bots.qq.com outage or observe a full recovery sequence.

## Solution Assessment

No, not as submitted: the QQBot token manager is the right owner boundary, but the final delay can exceed the advertised cap and the real background-loop behavior still needs proof. A safer version clamps after jitter, clears lint, and demonstrates failure/recovery with redacted runtime output.
