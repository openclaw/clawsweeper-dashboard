Keep this PR open. The branch adds QQBot token-refresh retry timing options, an exported exponential-backoff/jitter/cooldown helper, and helper-level tests.

## Best Possible Solution

Land a QQBot-local retry hardening change after clamping final jittered delays, clearing lint, preserving bounded response reads, adding real background-refresh proof, and getting maintainer acceptance for the cooldown default.

Yes at source level: current main and the latest release retry QQBot background token-refresh failures with a fixed 5-second sleep in the catch path. I did not live-force a Tencent outage or run a QQBot account through the full recovery sequence in this read-only review.

No, not as submitted: the QQBot token manager is the right owner boundary, but the final delay can exceed the advertised maximum, lint is failing, and runtime-loop behavior still needs proof. A safer version clamps after jitter, uses the lint-preferred exponentiation operator, and demonstrates failure and recovery with redacted output.
