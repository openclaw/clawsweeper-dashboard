## Summary

Keep open. This is a draft, stacked feature PR whose central recommend-reset path is not implemented on current main, and the proposed runtime wiring has blocking validation-contract issues that need author/maintainer follow-up before merge.

## What This Changes

The branch adds a loop-aware compaction guard stack with guard config, transcript-tail detection/scoring, guarded compaction instruction augmentation, a post-compaction validator, and internal recommend-reset diagnostics.

## Best Possible Solution

Rebase or consolidate the compaction-guard series onto current main, fix the validator runtime data contract, add real behavior proof, and have maintainers decide the reset recommendation versus snapshot/restore policy.

## Reproduction Assessment

Source-reproducible. This is a feature PR rather than a bug report, and the blocking review findings are visible directly on PR head: compact.ts hard-codes compaction counts while the safeguard details never produce unresolvedItems.

## Solution Assessment

No. The direction is plausible, but this patch is not yet the best mergeable form because the runtime validator receives misleading or missing evidence and the broader reset/recovery semantics still need maintainer judgment.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Pass real compaction evidence to the validator:** `src/agents/pi-embedded-runner/compact.ts:434-435`
  - body: The runtime always calls validatePostCompaction with compactionCountBefore: 0 and compactionCountAfter: 1, so the validator treats every guarded compaction as having count evidence. Since validatePostCompaction only emits usage-not-improved when the count did not increase and usage also did not improve, a no-op compaction with unchanged token pressure can validate as healthy and skip the recommend-reset diagnostic this PR is trying to surface. Pass real before/after evidence or change the validator contract so non-improved usage can fail without this hard-coded success.
  - confidence: 0.86
- **[P2] Populate unresolved items before validating them:** `src/agents/pi-extensions/compaction-safeguard.ts:1117-1124`
  - body: compact.ts passes safeguardDetails.unresolvedItems into validatePostCompaction, but the safeguard result never sets unresolvedItems in its compaction details. That makes the pending-items-missing branch dead in the real post-compaction hook path, so losing unresolved work will not contribute to the recommend-reset decision even though the validator and tests cover it. Extract and include unresolved items, or remove this runtime input until it has a producer.
  - confidence: 0.78
