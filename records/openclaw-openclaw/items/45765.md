## Summary

Keep open. Current main and the latest release still append an extra `.openclaw` segment after an explicit `OPENCLAW_HOME` that already points at `.openclaw`, and an open related PR is still the active implementation path rather than a merged fix.

## What This Changes

The issue requests that `OPENCLAW_HOME=~/.openclaw` write config and state directly under that directory instead of creating `~/.openclaw/.openclaw`.

## Best Possible Solution

Land a focused compatibility fix that bypasses the extra `.openclaw` append only when an explicit home path already names a known state directory, applies that rule consistently to state and config helpers, and adds regression coverage for `.openclaw`, legacy dirnames, tilde expansion, and preserved parent-home behavior.

## Reproduction Assessment

Yes. A static trace on current main is enough: explicit `OPENCLAW_HOME` is resolved as the effective home, then `resolveStateDir()` and `resolveConfigDir()` both append `.openclaw` again.

## Solution Assessment

Yes, the best fix is a narrow basename/state-dir guard rather than treating every explicit `OPENCLAW_HOME` as the final config/state directory, because docs and tests currently preserve parent-home semantics for non-state-dir paths.
