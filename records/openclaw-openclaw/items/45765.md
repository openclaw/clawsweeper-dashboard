## Summary

Keep open because current main still source-reproduces the nested config/state path behavior, and the open related implementation PR has not merged and is too broad/incomplete as written.

## What This Changes

The issue reports that setting `OPENCLAW_HOME` to an existing `.openclaw` directory causes OpenClaw to write config/state under a nested `.openclaw/.openclaw` directory.

## Best Possible Solution

Land a narrow basename/state-dir guard in both `resolveStateDir()` and `resolveConfigDir()` with regression coverage for `.openclaw`, legacy state dir names, tilde expansion, and parent-home negative cases.

## Reproduction Assessment

Yes. Source tracing on current main is enough: `OPENCLAW_HOME=/home/user/.openclaw` is preserved by the home resolver, then both state and config helpers append another `.openclaw` segment.

## Solution Assessment

Yes for the narrow basename/state-dir guard discussed in the issue; no for the current related PR’s broad direct-home approach because it changes documented parent-home semantics and does not fix `resolveStateDir()`.
