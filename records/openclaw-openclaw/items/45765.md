## Label Justifications

- `P2`: This is a valid config/state path regression with limited blast radius, but it can misplace mutable OpenClaw state for users who point `OPENCLAW_HOME` at an existing state directory.
- `impact:session-state`: The issue is about config, agents, workspace, and other mutable state being written under the wrong state root.

## Summary

Keep open: current main and the latest release still append `.openclaw` after an explicit `OPENCLAW_HOME` that already points at `.openclaw`, so the reported nested state/config root remains a valid narrow bug.

## What This Changes

The issue reports that `OPENCLAW_HOME=/home/user/.openclaw` makes OpenClaw write config and state under `/home/user/.openclaw/.openclaw` instead of the configured state root.

## Best Possible Solution

Add a narrow known-state-directory basename guard for explicit `OPENCLAW_HOME` in state and config resolution, preserve documented parent-home behavior, and cover `.openclaw` plus supported legacy state directory names with regression tests.

## Reproduction Assessment

Yes, at source level. `OPENCLAW_HOME=/home/user/.openclaw` is preserved as the effective home, then both `resolveStateDir()` and `resolveConfigDir()` append another `.openclaw` when direct overrides are absent.

## Solution Assessment

Yes for the narrow basename/state-dir guard discussed in the issue. Returning every explicit `OPENCLAW_HOME` directly is not the best fix because current docs and tests preserve parent-home semantics for normal replacement-home paths.
