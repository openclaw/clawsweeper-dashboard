## Summary

Keep open. Current main still reproduces the nested `.openclaw/.openclaw` path for explicit `OPENCLAW_HOME` values that already name the state directory, and open PR #45784 is the active implementation path rather than a merged fix.

## What This Changes

The issue requests a fix so `OPENCLAW_HOME=/home/tpam/.openclaw` writes config and state directly under that directory instead of `/home/tpam/.openclaw/.openclaw`.

## Best Possible Solution

Review, revise, and land #45784 or a replacement that applies a narrow explicit-home basename guard consistently to state and config helpers with regression coverage for `.openclaw`, legacy state dir names, tilde expansion, and unchanged parent-home behavior.

## Reproduction Assessment

Yes. Static tracing is enough: `OPENCLAW_HOME=/home/tpam/.openclaw` is resolved as the effective home, then `resolveStateDir()` and `resolveConfigDir()` append `.openclaw` again.

## Solution Assessment

Yes for a narrow state-dir basename guard. It fixes the reported case while preserving the documented behavior that non-state-dir `OPENCLAW_HOME` values act as replacement home directories.
