## Label Justifications

- `P2`: This is a valid config/state path regression with limited blast radius, but it can misplace mutable OpenClaw state for users who point `OPENCLAW_HOME` at an existing state directory.
- `impact:session-state`: The issue is about config, agents, workspace, and other mutable state being written under the wrong state root.

## Summary

Keep open: current main and the latest release still source-reproduce the nested `.openclaw/.openclaw` path when `OPENCLAW_HOME` already points at a state directory, and the active related PR is not yet a complete safe fix.

## What This Changes

The issue reports that `OPENCLAW_HOME=/home/tpam/.openclaw` causes config and state to be written under `/home/tpam/.openclaw/.openclaw` instead of the configured state root.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/45784 with a narrow known-state-directory basename guard in both `resolveStateDir()` and `resolveConfigDir()`, preserving parent-home semantics and adding regression coverage.

## Reproduction Assessment

Yes, at source level. `OPENCLAW_HOME=/home/user/.openclaw` is preserved by the effective-home resolver, then both state and config helpers append another `.openclaw` segment when direct overrides are absent.

## Solution Assessment

Yes for the narrow basename/state-dir guard discussed in the issue; no for returning every explicit `OPENCLAW_HOME` directly, because current docs and tests preserve parent-home semantics.
