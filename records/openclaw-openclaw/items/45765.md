## Summary

Keep open because current main still source-reproduces the nested state/config path behavior, and the related open implementation PR has not merged. The report is concrete and narrow; cleanup should route through the linked PR or a replacement fix rather than closing the issue early.

## What This Changes

The issue asks OpenClaw to avoid creating `~/.openclaw/.openclaw` when `OPENCLAW_HOME` is already set to a `.openclaw` state directory.

## Best Possible Solution

Land a narrow basename guard in both `resolveStateDir()` and `resolveConfigDir()` with regression coverage for `.openclaw`, legacy state dir names, tilde expansion, and the `/srv/app -> /srv/app/.openclaw` negative case.

## Reproduction Assessment

Yes. Source tracing on current main is enough: `OPENCLAW_HOME=/home/tpam/.openclaw` is preserved by the home resolver, then both state and config helpers append another `.openclaw`.

## Solution Assessment

Yes for the narrow basename-guard direction discussed in the issue; no for a broad reinterpretation of every explicit `OPENCLAW_HOME` value as the final state/config directory because current tests preserve parent-home semantics.
