## Summary

This issue should stay open: current main and the latest release still append `.openclaw` under an explicit `OPENCLAW_HOME`, and the linked closing fix at https://github.com/openclaw/openclaw/pull/87255 is open but conflicting.

## What This Changes

The issue reports that setting `OPENCLAW_HOME` to an existing `.openclaw` directory makes OpenClaw create and use a nested `.openclaw/.openclaw` state/config directory.

## Best Possible Solution

Rebase and review the linked fix PR, or land an equivalent narrow replacement that guards explicit known state-directory basenames across state-dir, config-dir, and config-candidate resolution while preserving normal parent-home behavior.

## Reproduction Assessment

Yes, from source inspection: `OPENCLAW_HOME` becomes the effective home, then the current state/config resolvers append `.openclaw` without checking whether the explicit home already names a state directory. I did not run a live onboarding flow in this read-only review.

## Solution Assessment

Yes, the best fix shape is a narrow explicit-home state-dir basename guard shared by state, config, and config-candidate resolution, with tests that preserve non-state-dir parent-home behavior. The currently linked PR appears to target that shape but needs rebase/review before the issue can close.
