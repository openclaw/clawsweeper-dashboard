## Summary

Keep open: current main and the latest release still append `.openclaw` under an explicit `OPENCLAW_HOME` that already names `.openclaw`, and the open closing PR has not merged yet.

## What This Changes

The issue asks OpenClaw to avoid creating `~/.openclaw/.openclaw` when `OPENCLAW_HOME` already points at the `.openclaw` state directory.

## Best Possible Solution

Land or replace the linked fix with a basename guard for explicit state-dir `OPENCLAW_HOME` plus upgrade-safe doctor recovery for already-created nested state.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `OPENCLAW_HOME=/home/user/.openclaw` resolves as the effective home, then current state/config helpers append another `.openclaw`.

## Solution Assessment

Yes, with upgrade safety. A known-state-directory basename guard is the narrow source fix, and already-created nested state should be handled by doctor recovery rather than runtime read-through fallback.
