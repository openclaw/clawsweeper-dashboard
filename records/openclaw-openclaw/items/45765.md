## Summary

Current main and the latest release still route an explicit `OPENCLAW_HOME=/.../.openclaw` through helpers that append another `.openclaw`, and an open unmerged PR already owns the fix path, so this issue remains necessary.

## What This Changes

The issue reports that setting `OPENCLAW_HOME` to an existing `.openclaw` state directory causes config and state paths to nest under `.openclaw/.openclaw`.

## Best Possible Solution

Land or replace the linked narrow resolver fix so explicit `OPENCLAW_HOME` values that already name a known state directory are used directly while non-state-dir homes keep the existing parent-home convention.

## Reproduction Assessment

Yes. Source inspection shows `OPENCLAW_HOME=/home/user/.openclaw` becomes the effective home and both current config/state resolvers append another `.openclaw` segment.

## Solution Assessment

Yes. A narrow known-state-directory basename guard at the config/state path-resolution boundary, with regression tests, is the maintainable fix; the open linked PR owns that path but currently needs conflict handling.
