## Summary

This issue should stay open: current main and the latest release still route an explicit `OPENCLAW_HOME=/path/.openclaw` through helpers that append another `.openclaw`, and the active linked repair PR has not merged. The issue is source-reproducible and already has a concrete fix path in https://github.com/openclaw/openclaw/pull/87255.

## What This Changes

The issue reports that setting `OPENCLAW_HOME` to an existing `.openclaw` directory causes onboarding/config resolution to write under a nested `.openclaw/.openclaw` directory.

## Best Possible Solution

Land a narrow known-state-directory basename guard for explicit `OPENCLAW_HOME` in state dir, config dir, and default config candidate resolution, while preserving parent-home behavior for other custom homes.

## Reproduction Assessment

Yes. Source inspection shows `OPENCLAW_HOME=/home/user/.openclaw` flows through `resolveRequiredHomeDir()` and is then passed to `newStateDir()` / `path.join(home, ".openclaw")`, producing the nested path.

## Solution Assessment

Yes. The narrow guard proposed by the linked PR is the best current shape because it fixes explicit state-dir homes without adding a new option or changing non-state-dir `OPENCLAW_HOME` behavior.
