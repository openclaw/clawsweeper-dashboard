## Summary

Keep open: current main still source-reproduces the nested `.openclaw/.openclaw` path for `OPENCLAW_HOME` values that already end in `.openclaw`. The linked open implementation at https://github.com/openclaw/openclaw/pull/45784 is relevant, but it only changes `resolveConfigDir()` and broadens the documented `OPENCLAW_HOME` parent-home contract.

## What This Changes

The issue asks OpenClaw to avoid creating a nested `.openclaw/.openclaw` state/config directory when `OPENCLAW_HOME` already points at `.openclaw`.

## Best Possible Solution

Land or replace the related fix with a narrow basename/state-dir guard in both `resolveStateDir()` and `resolveConfigDir()`, with regression tests for `.openclaw`, legacy state dir names, tilde expansion, and non-state-dir parent homes.

## Reproduction Assessment

Yes, at source level. `OPENCLAW_HOME=/home/user/.openclaw` flows through the effective-home resolver unchanged, then both state and config helpers append another `.openclaw` segment.

## Solution Assessment

Yes for the narrow basename/state-dir guard discussed in the issue. No for returning every explicit `OPENCLAW_HOME` directly, because current docs and tests preserve parent-home semantics.
