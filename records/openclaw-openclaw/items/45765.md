## Summary

## What This Changes

The issue reports that setting `OPENCLAW_HOME` to an existing `.openclaw` state directory makes onboarding/config/state paths nest under `.openclaw/.openclaw`.

## Best Possible Solution

Add a narrow known-state-directory basename guard for explicit `OPENCLAW_HOME` in both state and config resolution, preserving parent-home behavior and covering `.openclaw` plus supported legacy state directory names.

## Reproduction Assessment

Yes, at source level. `OPENCLAW_HOME=/home/user/.openclaw` is preserved by `resolveEffectiveHomeDir()`, then both `resolveStateDir()` and `resolveConfigDir()` append another `.openclaw` when direct overrides are absent.

## Solution Assessment

Yes for the narrow basename/state-dir guard discussed in the issue. Returning every explicit `OPENCLAW_HOME` directly is not the best fix because current docs and tests preserve parent-home semantics for normal replacement-home paths.
