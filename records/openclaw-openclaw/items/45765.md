## Summary

Keep open: current main still source-reproduces the nested path, and the open related PR is incomplete because it only changes config-dir resolution and broadens `OPENCLAW_HOME` semantics.

## What This Changes

The issue asks OpenClaw to stop creating `$OPENCLAW_HOME/.openclaw` when `OPENCLAW_HOME` already points at a state directory such as `~/.openclaw`.

## Best Possible Solution

Land a narrow basename/state-dir guard in both `resolveStateDir()` and `resolveConfigDir()` with regression coverage while preserving parent-home behavior.

## Reproduction Assessment

Yes, at source level. `OPENCLAW_HOME=/home/user/.openclaw` flows through the home resolver unchanged, then both state and config helpers append another `.openclaw`.

## Solution Assessment

Yes for the narrow basename/state-dir guard discussed in the issue. No for returning every explicit `OPENCLAW_HOME` directly, because current docs/tests preserve parent-home semantics.
