## Summary

Keep open. Current `origin/main` still source-reproduces the nested `.openclaw/.openclaw` path, and the related open PR is an implementation candidate rather than a merged fix.

## What This Changes

The issue reports that `OPENCLAW_HOME` pointing at an existing `.openclaw` directory causes config and state paths to be resolved under a nested `.openclaw/.openclaw` directory.

## Best Possible Solution

Land a narrow basename guard in both `resolveStateDir()` and `resolveConfigDir()` with regression coverage for `OPENCLAW_HOME=~/.openclaw`, while preserving `/srv/app -> /srv/app/.openclaw` parent-home behavior.

## Reproduction Assessment

Yes. Source tracing on current `origin/main` is enough: `OPENCLAW_HOME=~/.openclaw` becomes the resolved home, then the state and config helpers append `.openclaw` again.

## Solution Assessment

No, the currently open PR is not the best fix as-is because it only changes `resolveConfigDir()` and treats every explicit `OPENCLAW_HOME` as the final config directory. The safer solution is a narrow state-dir basename guard in both affected helpers.
