## Summary

Keep open: current `main` and the latest release still resolve explicit `OPENCLAW_HOME=/path/.openclaw` through helpers that append another `.openclaw`, and the open closing PR is still the active fix path rather than a merged implementation.

## What This Changes

The issue reports that `OPENCLAW_HOME` values already pointing at a `.openclaw` state directory make onboarding/config writes land under a nested `.openclaw/.openclaw` path.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/87255 with a compatibility-safe fix that adds a narrow explicit state-dir basename guard, keeps parent-home semantics for ordinary `OPENCLAW_HOME`, updates all resolver/candidate paths together, and includes doctor-owned recovery for already-nested state.

Yes. Source inspection shows `resolveStateDir`, `resolveConfigDir`, and default config candidates still append `.openclaw` after resolving explicit `OPENCLAW_HOME`, which reproduces the reported nested path without needing a live onboarding run.

## Solution Assessment

Yes for keeping this issue open behind the active linked PR. The best code path is a narrow state-dir-basename guard plus doctor recovery for already-nested data, not a broad direct-`OPENCLAW_HOME` behavior change or runtime fallback.
