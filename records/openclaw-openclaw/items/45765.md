## Label Justifications

- `P2`: This is a valid path-resolution bug with limited blast radius, but it can misplace config and state for users who set `OPENCLAW_HOME` to an existing state directory.
- `impact:session-state`: The issue is about config, agents, workspace, and mutable state being written under the wrong state root.

## Summary

Keep open: current main and the latest release still source-reproduce the nested `.openclaw/.openclaw` path when `OPENCLAW_HOME` already points at `.openclaw`. The open related implementation is relevant but not sufficient as-is because it only changes config-dir resolution and broadens the documented parent-home contract.

## What This Changes

Fix `OPENCLAW_HOME` values that already point at `.openclaw` so onboarding, config, and state files are written there instead of under a second `.openclaw` directory.

## Best Possible Solution

Land or replace the related fix with a narrow basename/state-dir guard in both `resolveStateDir()` and `resolveConfigDir()`, preserving parent-home semantics and adding regression coverage for `.openclaw`, legacy state-dir names, tilde expansion, and non-state-dir parent homes.

## Reproduction Assessment

Yes, at source level. `OPENCLAW_HOME=/home/user/.openclaw` is preserved by the effective-home resolver, then both state and config helpers append another `.openclaw` segment.

## Solution Assessment

Yes for the narrow basename/state-dir guard discussed in the issue; no for returning every explicit `OPENCLAW_HOME` directly, because current docs and tests preserve parent-home semantics.
