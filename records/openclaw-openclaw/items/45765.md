## Summary

Keep open: current main and the latest release still route explicit `OPENCLAW_HOME=/path/.openclaw` through config/state helpers that append another `.openclaw`, while https://github.com/openclaw/openclaw/pull/87255 is the open linked fix path rather than a merged implementation.

## What This Changes

The issue reports that `openclaw onboard` writes config and state under `$OPENCLAW_HOME/.openclaw` when `OPENCLAW_HOME` already points at `.openclaw`.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/87255 with a compatibility-safe fix that updates state, config-dir, and candidate resolution together and handles already-nested state through `openclaw doctor --fix`.

## Reproduction Assessment

Yes. Source inspection shows `resolveStateDir`, `resolveConfigDir`, and default config candidates still append `.openclaw` after resolving explicit `OPENCLAW_HOME`, which reproduces the reported nested path without a live onboarding run.

## Solution Assessment

Yes for keeping this issue open behind the active linked PR. The best code path is a narrow state-dir-basename guard plus doctor recovery for already-nested data, not a broad direct-`OPENCLAW_HOME` behavior change or runtime fallback.
