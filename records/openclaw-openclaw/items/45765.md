## Summary

Keep open because current main still source-reproduces the nested state/config path, and the only open implementation PR remains unmerged and incomplete for the affected state-dir path.

## What This Changes

Avoid creating `OPENCLAW_HOME/.openclaw` when `OPENCLAW_HOME` already points at an `.openclaw` state directory.

## Best Possible Solution

Land a narrow state-dir basename guard in both `resolveStateDir()` and `resolveConfigDir()` with regression coverage, while preserving parent-home behavior such as `/srv/app -> /srv/app/.openclaw`.

## Reproduction Assessment

Yes. Source tracing on current main is enough: `OPENCLAW_HOME=/home/tpam/.openclaw` is preserved by the home resolver, then both state and config helpers append another `.openclaw`.

## Solution Assessment

Yes for the narrower basename-guard direction discussed on the issue; no for the currently open one-file PR as-is because it changes only `resolveConfigDir()` and broadens explicit-home semantics.
