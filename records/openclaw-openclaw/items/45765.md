## Summary

Keep open: current `main` and the latest release still have the source-reproducible nested path behavior, and the open linked PR is the active fix candidate that should land or be replaced before this issue closes.

## What This Changes

The issue asks OpenClaw to avoid writing config and state under `$OPENCLAW_HOME/.openclaw` when `OPENCLAW_HOME` already points at a state directory such as `.openclaw`.

## Best Possible Solution

Review, rebase, and land the linked fix or a narrow replacement that guards explicit state-directory basenames across state-dir, config-dir, and config-candidate resolution while adding conflict-safe doctor recovery for already-created nested state.

## Reproduction Assessment

Yes. Source inspection against current `main` shows `OPENCLAW_HOME=/home/user/.openclaw` flows through `resolveRequiredHomeDir()` and is then appended with `.openclaw` in both state and config resolution; I did not run a live onboarding wizard in this read-only review.

## Solution Assessment

Yes for the central fix boundary, with one important upgrade condition. A narrow basename guard in the shared path resolvers is the maintainable fresh-write fix, but the merge-ready solution should also include doctor-owned recovery for existing nested state rather than a runtime fallback.
