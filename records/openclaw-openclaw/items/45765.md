## Summary

Keep open. Current main still source-reproduces the nested `.openclaw/.openclaw` path, and the open related PR is an implementation path rather than a merged fix.

## What This Changes

The issue asks for `OPENCLAW_HOME` values that already point at `.openclaw` to be used directly instead of creating a nested `.openclaw/.openclaw` directory.

## Best Possible Solution

Review and land #45784 or a replacement that adds a narrow explicit-home state-dir basename guard in both state and config helpers with regression coverage while preserving non-state-dir parent-home behavior.

## Reproduction Assessment

Yes. Source tracing on current main is enough: `OPENCLAW_HOME=~/.openclaw` becomes the resolved home, then `resolveStateDir()` and `resolveConfigDir()` append `.openclaw` again.

## Solution Assessment

Yes for a narrow state-dir basename guard in both helpers. Treating all explicit `OPENCLAW_HOME` values as final config directories would be broader than the established parent-home contract.
