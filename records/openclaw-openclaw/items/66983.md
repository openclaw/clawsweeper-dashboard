## Summary

Keep open. Current main still does not define a first-class web/browser node platform, and device-less node-role browser sessions remain intentionally rejected even when shared-token, trusted-proxy, or Control UI break-glass paths are available. The request is valid, but the auth/pairing part is security-sensitive product design work rather than a safe cleanup close.

## What This Changes

Request browser-tab A2UI canvas node support through an explicit web platform policy, a secure node-role auth/pairing path for reverse-proxied browsers, and documentation for that setup.

## Best Possible Solution

Define an explicit Gateway web canvas-node contract: a narrow web platform command policy for A2UI/canvas, a reviewed browser-node enrollment/auth model for reverse-proxied deployments, regression tests for Origin/device identity/node pairing behavior, and public docs for the supported browser setup.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: connect as role "node" with shared auth but no device identity and current policy returns reject-device-required; existing tests cover that rejection and the node-role exclusions for Control UI bypasses.

## Solution Assessment

No, not as the proposed blanket dangerouslyDisableDeviceAuth-equivalent or shared-token bypass. The safer path is a deliberate web node enrollment mechanism with constrained commands and explicit origin/device identity semantics.
