## Summary

Keep open. Current main still lacks an explicit web/browser node platform contract, and the node-role browser enrollment/auth part remains a security-sensitive design question rather than a cleanup-close candidate.

## What This Changes

Request support for browser tabs to register as A2UI canvas node surfaces, including explicit web canvas command policy and a node-role auth/pairing path for reverse-proxied browser clients.

## Best Possible Solution

Define a reviewed Gateway web canvas-node contract: explicit web platform command policy, constrained browser-node enrollment/auth for reverse-proxied deployments, regression tests for Origin/device identity/pairing behavior, and public docs for the supported setup.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: connect as role "node" with platform "web", shared auth, and no device identity, and current policy returns reject-device-required; browser-origin node auto-approval is also explicitly blocked.

## Solution Assessment

No, not as a blanket dangerouslyDisableDeviceAuth-equivalent or shared-token bypass for node role. The safer solution is a purpose-built web node enrollment flow with canvas-only scope and explicit origin/device semantics.
