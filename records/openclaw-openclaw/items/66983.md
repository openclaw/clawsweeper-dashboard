## Summary

Keep open. Current main still lacks a deliberate web/browser canvas-node contract, and the requested node-role browser enrollment path remains intentionally blocked by device-identity and trusted-proxy policy.

## What This Changes

The issue requests first-class browser tabs as A2UI canvas node surfaces, including explicit web platform command policy, node-role auth/enrollment support, and docs for the pattern.

## Best Possible Solution

Define a reviewed Gateway web canvas-node contract with explicit web platform policy, constrained browser-node enrollment/auth, canvas-only command scope, regression tests for origin/device identity/pairing behavior, and public docs.

## Reproduction Assessment

Yes, at source level. A role node browser-style client with platform web, shared auth, and no device identity is not admitted as a first-class web canvas node: platform policy is only unknown fallback and missing device identity reaches reject-device-required for node role.

## Solution Assessment

No, not as a blanket dangerouslyDisableDeviceAuth-equivalent or shared-token bypass for node role. The maintainable path is a purpose-built browser canvas-node enrollment design with explicit identity, origin, revocation, and canvas-only semantics.
