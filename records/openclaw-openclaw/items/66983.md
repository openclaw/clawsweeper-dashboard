## Summary

Keep open. Current main still does not define a first-class web/browser canvas-node contract, and the requested browser node enrollment path remains blocked by the Gateway device-identity and trusted-proxy pairing policy.

## What This Changes

The request is to add first-class browser canvas-node support, including web platform canvas command defaults, browser node enrollment/auth, and web canvas-node docs.

## Best Possible Solution

Define a reviewed Gateway web canvas-node contract with explicit web platform policy, constrained browser-node enrollment/auth, canvas-only command scope, regression tests for origin/device identity/pairing behavior, and public docs coordinated with the browser action bridge work.

## Reproduction Assessment

Yes, at source level. A role "node" browser-style client with `platform: "web"`, shared auth, and no device identity is not admitted as a first-class web canvas node: the platform policy only falls through to unknown, and missing device identity reaches `reject-device-required` for node role.

## Solution Assessment

No, not as a blanket `dangerouslyDisableDeviceAuth` equivalent or shared-token bypass for node role. The maintainable path is a purpose-built browser canvas-node enrollment design with explicit identity, origin, revocation, trusted-proxy, and canvas-only semantics.
