## Summary

Keep open. Current main still lacks a deliberate web/browser canvas-node platform and the node-role device identity policy still rejects device-less browser node enrollment, so this remains valid feature/security design work rather than cleanup.

## What This Changes

Request first-class browser tabs or web clients as A2UI canvas node surfaces, including web platform defaults, node-role auth/enrollment support, and docs.

## Best Possible Solution

Define a reviewed Gateway web canvas-node contract with explicit platform policy, constrained browser-node identity/auth, canvas-only command scope, revocation/origin handling, regression tests, and docs coordinated with the related action bridge.

## Reproduction Assessment

Yes, at source level. A role node browser-style client with platform web and no device identity is not admitted as a first-class web canvas node: platform policy falls into unknown and missing device identity reaches reject-device-required for node role.

## Solution Assessment

No, not as a blanket dangerouslyDisableDeviceAuth equivalent or shared-token node bypass. The maintainable path is a purpose-built browser canvas-node enrollment design with explicit identity, origin, revocation, trusted-proxy, and canvas-only semantics.
