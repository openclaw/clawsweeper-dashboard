## Summary

The PR adds a TUI `/upload` slash command, file-browser overlay, file-context prepending, theme/selectors wiring, and component tests.

## Best Possible Solution

Repair the branch to match current TUI dependencies, bound regular-file reads, and make pending uploads retry-safe; then maintainers can decide whether the core TUI should ship the new command.

Yes for the PR blockers: PR-head source shows the stale imports, pending-context lifecycle defects, and unbounded local reads. There is no current-main bug path because `/upload` is a new feature absent from main.

## Solution Assessment

No. The feature direction may be useful, but this implementation needs current-main rebasing, bounded regular-file validation, and safer pending-context lifecycle before it is the narrow maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Use the current pi-tui package:** `src/tui/components/file-browser-select-list.ts:10`
  - body: Current main depends on `@earendil-works/pi-tui`, but this added component imports `@mariozechner/pi-tui`. Rebasing this branch as-is leaves the new file importing a package outside the current dependency surface.
  - confidence: 0.93
- **[P1] Keep pending uploads until send succeeds:** `src/tui/tui-command-handlers.ts:572`
  - body: `pendingFileContext` is cleared before `client.sendChat` resolves. If the send fails, retrying drops the selected file context and sends only the typed message, causing user-visible data loss.
  - confidence: 0.9
- **[P1] Clear stale uploads when replacement attaches nothing:** `src/tui/tui-command-handlers.ts:278-282`
  - body: When a later `/upload` confirms but every selected path is skipped or unreadable, the previous `pendingFileContext` remains queued. The next message can silently include old files after a failed replacement upload.
  - confidence: 0.88
- **[P1] Reject non-regular or oversized files before reading:** `src/tui/tui-command-handlers.ts:257`
  - body: The upload handler calls `readFileSync` on selected paths without regular-file or size validation. Device nodes, FIFOs, sockets, or very large files can block the TUI or consume unbounded memory before binary detection runs.
  - confidence: 0.87
