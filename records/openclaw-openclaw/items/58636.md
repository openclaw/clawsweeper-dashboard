## Summary

Keep open. The feature is not implemented on current main, and the PR is useful, but the branch still has merge-blocking local file-read and queued-context lifecycle bugs that should be repaired before maintainer review or landing.

## What This Changes

The PR adds a TUI `/upload` slash command with a file-browser overlay, selector/theme wiring, pending file-context injection into the next chat message, and component tests.

## Best Possible Solution

Repair the feature on top of current main: add bounded regular-file validation before reading, preserve queued upload context across failed sends, clear stale queued context when an upload attaches nothing, document `/upload` in the TUI docs, and add the required user-facing changelog entry while keeping the scope limited to TUI behavior.

## Reproduction Assessment

Yes. The remaining bugs have high-confidence source-level reproductions from the PR head: selecting a huge or special path reaches `readFileSync` before validation, a rejected `sendChat` loses queued file context, and confirming only skipped/unreadable files leaves older context queued.

## Solution Assessment

No. The feature direction fits the TUI and does not need protocol changes, but this PR is not the best merge path until file reads are bounded and the queued-context lifecycle matches user retry and replacement behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Validate files before reading them:** `src/tui/tui-command-handlers.ts:257`
  - body: `readFileSync(filePath)` runs before any regular-file check or size cap. Since the browser can surface non-directory entries and users can select very large files, this can block the TUI or exhaust memory before binary detection can skip anything.
  - confidence: 0.9
- **[P2] Clear queued context only after a successful send:** `src/tui/tui-command-handlers.ts:572`
  - body: `pendingFileContext` is nulled before `client.sendChat` resolves. If the gateway or provider send rejects, the selected files are lost and retrying sends only the typed message instead of the queued context.
  - confidence: 0.86
- **[P2] Clear stale context when no files attach:** `src/tui/tui-command-handlers.ts:278-282`
  - body: If a user has a previous upload queued and then confirms a replacement upload where every selected path is skipped or fails to read, this branch leaves the old `pendingFileContext` intact. The next message can silently send stale file contents.
  - confidence: 0.84
- **[P3] Document the new TUI command:** `src/tui/commands.ts:117`
  - body: This adds a user-facing slash command, but the TUI docs still list the complete command set without `/upload`. Add the command and its file-browser workflow to `docs/web/tui.md` so shipped docs match the UI.
  - confidence: 0.78
- **[P3] Add the required changelog entry:** `src/tui/commands.ts:117`
  - body: This is a user-facing feature, but the PR does not update `CHANGELOG.md`. OpenClaw policy requires a `### Changes` entry for new user-visible behavior, credited to the contributor.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The patch introduces a local file-read surface that needs bounded regular-file validation before merge.

Concerns:

- **[medium] Unbounded local file read:** `src/tui/tui-command-handlers.ts:257`
  - body: The upload handler reads each selected path synchronously before checking file type or size. A selected special file, FIFO, or very large file can block or exhaust the TUI process, so the handler should stat/open only regular files and enforce a size cap before full reads.
