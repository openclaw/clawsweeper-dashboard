## Summary

Keep open: current main does not implement the TUI `/upload` feature and no canonical duplicate was found, but the PR branch still has source-reproducible merge blockers in the TUI package import, upload lifecycle, stale pending context handling, and regular-file validation.

## What This Changes

The branch adds a TUI `/upload` slash command, a keyboard-driven file-browser select component, theme wiring, and tests that attach selected local file contents to the next chat message.

## Best Possible Solution

Keep the feature branch open, rebase onto current TUI code, repair upload lifecycle and file validation, add command-handler regression tests, then let maintainers decide whether this TUI file-context workflow should ship.

## Reproduction Assessment

Yes for the review blockers: PR-head source shows the premature context clear, stale replacement path, old package import, and non-regular file predicate, and a local stat check confirms the special-file case. The broader `/upload` behavior is a new feature, not a bug reproduction.

## Solution Assessment

No. The direction may be useful, but this implementation should not merge until it imports the current TUI package, preserves pending uploads across failed sends, clears stale context on empty replacement uploads, and restricts reads to regular files.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Import pi-tui from the current package:** `src/tui/components/file-browser-select-list.ts:10`
  - body: Current main has moved TUI code and dependencies to `@earendil-works/pi-tui`, but this new component still imports `@mariozechner/pi-tui`. Since `package.json` no longer provides the old package name, the rebased PR will fail to resolve this module unless the new file uses the current package.
  - confidence: 0.93
- **[P1] Preserve pending uploads until send succeeds:** `src/tui/tui-command-handlers.ts:570-572`
  - body: `pendingFileContext` is cleared before `client.sendChat` resolves. If the gateway or provider rejects the send, the catch path reports the failure but the selected files are already gone, so retrying sends only the typed text. Clear the pending context after `sendChat` accepts the message, or restore it in the catch path.
  - confidence: 0.9
- **[P1] Clear stale context when no files attach:** `src/tui/tui-command-handlers.ts:278-279`
  - body: A confirmed upload that produces no `blocks` leaves the previous `pendingFileContext` intact. If every newly selected path is skipped as binary or fails to read, the next message silently includes the older files instead of nothing. Reset the pending context when a new confirmed upload starts or when no files attach.
  - confidence: 0.88
- **[P1] Filter the picker to regular files:** `src/tui/components/file-browser-select-list.ts:77-81`
  - body: Every non-directory entry is shown and selectable as a file, but the handler later calls `readFileSync` on the selected path. Device nodes, FIFOs, and sockets are non-directories too; selecting one can block or consume memory before binary detection runs. Only surface regular files, and validate again before reading.
  - confidence: 0.87
