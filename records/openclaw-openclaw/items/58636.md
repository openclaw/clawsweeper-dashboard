## Summary

Keep open: current main still lacks the proposed TUI `/upload` feature, and the PR has credible real-behavior screenshots, but the branch is not mergeable as-is due to stale TUI imports plus retry, stale-context, and unbounded local-file-read defects.

## What This Changes

The PR adds a TUI `/upload` slash command with a file-browser overlay that reads selected local files and prepends their contents to the next chat message, plus theme wiring and component tests.

## Best Possible Solution

Repair the branch against current TUI contracts, bound uploads to regular size-limited files, make pending context retry-safe, then let maintainers decide whether the TUI local-file upload UX belongs in core.

## Reproduction Assessment

Yes for the PR blockers: PR-head source shows stale `pi-tui` imports, pre-send pending-context clearing, stale context after empty replacement uploads, and non-regular entries flowing into synchronous reads. There is no current-main bug reproduction path because `/upload` is a new feature.

## Solution Assessment

No. The feature direction is plausible, but this implementation should first be rebased to current TUI package/backend contracts and should bound local file reads while preserving queued uploads until send success.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Use the current pi-tui package:** `src/tui/components/file-browser-select-list.ts:10`
  - body: Current main only pins and imports `@earendil-works/pi-tui`, but this added component imports `@mariozechner/pi-tui`. As written, the branch cannot resolve against current main's TUI dependency surface.
  - confidence: 0.93
- **[P1] Preserve pending uploads until send succeeds:** `src/tui/tui-command-handlers.ts:572`
  - body: `pendingFileContext` is nulled before `client.sendChat` resolves. If the send fails, retrying drops the selected file context and sends only the typed message, causing user-visible data loss.
  - confidence: 0.9
- **[P1] Clear stale uploads when nothing attaches:** `src/tui/tui-command-handlers.ts:278-282`
  - body: When a later `/upload` confirms but every selected path is skipped or unreadable, this branch leaves the previous `pendingFileContext` queued. The next message can silently include old files after a failed replacement upload.
  - confidence: 0.88
- **[P1] Restrict uploads to regular bounded files:** `src/tui/components/file-browser-select-list.ts:77-81`
  - body: Every non-directory entry is selectable, but the handler later calls `readFileSync` on selected paths. Device nodes, FIFOs, sockets, and very large files can block the TUI or consume unbounded memory before binary detection runs.
  - confidence: 0.87
