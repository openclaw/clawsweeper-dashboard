## Summary

Keep this PR open, but it still needs changes before merge: source review shows blocking TUI import, upload lifecycle, stale-context, and file-validation defects. The screenshots do show the intended WSL2 `/upload` flow, so the proof gate looks satisfied for the visible behavior.

## What This Changes

This PR adds a TUI `/upload` slash command, a file-browser selector with theme wiring, pending file-context injection into the next message, and file-browser component tests.

## Best Possible Solution

Repair the PR branch against the current TUI package and command-handler contract, add upload lifecycle and regular-file validation coverage, then have maintainers decide whether core TUI should ship `/upload`.

## Reproduction Assessment

Yes for the PR blockers: PR-head source shows the stale package import, premature pending-context clear, stale replacement path, and non-regular file predicate. The broader `/upload` behavior is a new feature, so there is no current-main bug reproduction path.

## Solution Assessment

No. The feature direction may be useful, but this implementation should first rebase to the current TUI package, preserve pending uploads until `sendChat` succeeds, clear stale context on empty replacement uploads, and restrict reads to regular files.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Import pi-tui from the current package:** `src/tui/components/file-browser-select-list.ts:10`
  - body: Current main imports and depends on `@earendil-works/pi-tui`, but this new component imports `@mariozechner/pi-tui`. The old package is no longer in the current dependency surface, so a rebased branch will fail to resolve this module unless the new code uses the current package.
  - confidence: 0.93
- **[P1] Preserve pending uploads until send succeeds:** `src/tui/tui-command-handlers.ts:570-572`
  - body: `pendingFileContext` is cleared before `client.sendChat` resolves. If the gateway or provider rejects the send, the catch path reports the failure but the selected files are already gone, so retrying sends only the typed text.
  - confidence: 0.9
- **[P1] Clear stale context when a replacement upload attaches nothing:** `src/tui/tui-command-handlers.ts:278-279`
  - body: A confirmed upload that produces no `blocks` leaves the previous `pendingFileContext` intact. If every newly selected path is skipped as binary or fails to read, the next message silently includes the older files instead of no attachment.
  - confidence: 0.88
- **[P1] Filter selectable entries to regular files:** `src/tui/components/file-browser-select-list.ts:77-81`
  - body: Every non-directory entry is shown and selectable as a file, but the handler later calls `readFileSync` on the selected path. Device nodes, FIFOs, and sockets are non-directories too; selecting one can block or consume memory before binary detection runs.
  - confidence: 0.87
