## Summary

Keep open. Current main still lacks the requested `/upload` TUI feature and the PR has useful visible proof, but the patch is not merge-ready because it is stale against current TUI package contracts and has retry/stale-context and local-file read safety bugs.

## What This Changes

The PR adds a TUI `/upload` command, a file-browser multi-select overlay, theme wiring, upload handling that prepends selected local file contents to the next chat message, and component tests.

## Best Possible Solution

Repair the branch against current TUI contracts with bounded regular-file handling and retry-safe pending context, then have maintainers decide whether to ship the local-file upload UX.

## Reproduction Assessment

Yes for the PR blockers: PR-head source shows stale pi-tui imports, pending context cleared before awaited send, failed replacement uploads preserving old context, and non-regular entries treated as files. The underlying `/upload` behavior is a new feature, not a current-main bug reproduction path.

## Solution Assessment

No. The feature direction is plausible, but this implementation should first be rebased to current `@earendil-works/pi-tui` and `TuiBackend` contracts, preserve queued uploads until send success, clear stale uploads on failed replacements, and restrict reads to regular bounded files.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Use the current pi-tui package:** `src/tui/components/file-browser-select-list.ts:10`
  - body: Current main only pins and imports `@earendil-works/pi-tui`, but this new component imports `@mariozechner/pi-tui`. A rebase onto main will fail to resolve or mix incompatible TUI component types until these imports are updated across the PR's touched TUI files.
  - confidence: 0.93
- **[P1] Preserve pending uploads until send succeeds:** `src/tui/tui-command-handlers.ts:572`
  - body: `pendingFileContext` is nulled before `client.sendChat` resolves. If the send fails, retrying drops the selected file context and sends only the typed message, which is user-visible data loss.
  - confidence: 0.9
- **[P1] Clear stale uploads when nothing attaches:** `src/tui/tui-command-handlers.ts:278-282`
  - body: When a later `/upload` confirms but every selected path is skipped or unreadable, this branch leaves the previous `pendingFileContext` queued. The next message can silently include old files after the user attempted a replacement upload that attached nothing.
  - confidence: 0.88
- **[P1] Restrict uploads to regular bounded files:** `src/tui/components/file-browser-select-list.ts:77-81`
  - body: Every non-directory entry is selectable, but the handler later reads selected paths synchronously. Device nodes, FIFOs, sockets, and very large files can block the TUI or consume unbounded memory before binary detection runs.
  - confidence: 0.87

## Security Review

Status: needs_attention

Summary: The diff adds local file reads and currently lets special or very large filesystem entries reach synchronous reads.
