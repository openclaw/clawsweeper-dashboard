## Summary

Keep open: the PR adds a plausible new TUI file-context feature, but the current branch still has source-review blockers around the stale TUI package import, upload context lifecycle, and filesystem entry validation before maintainers can judge the feature itself.

## What This Changes

The PR adds a `/upload` TUI slash command, a file-browser selector, theme wiring, and tests so selected local file contents are prepended to the next chat message.

## Best Possible Solution

Keep the branch open, repair the upload lifecycle and file validation defects, rebase onto the current TUI dependency surface, add handler regression coverage, then route the new file-context workflow for maintainer product approval.

## Reproduction Assessment

Yes for the review blockers: PR-head source shows the stale package import, premature context clear, stale replacement path, and non-regular file predicate. The broader `/upload` behavior is a new feature, so there is no current-main bug reproduction.

## Solution Assessment

No. The feature direction may be useful, but this implementation should not merge until it imports the current TUI package, preserves queued files across failed sends, clears stale context on empty replacement uploads, and restricts reads to regular files.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Import pi-tui from the current package:** `src/tui/components/file-browser-select-list.ts:10`
  - body: Current main imports and depends on `@earendil-works/pi-tui`, but this new component still imports `@mariozechner/pi-tui`. Since the old package is no longer present in `package.json` or the lockfile, the rebased PR will fail to resolve this module.
  - confidence: 0.93
- **[P1] Preserve pending uploads until send succeeds:** `src/tui/tui-command-handlers.ts:570-572`
  - body: `pendingFileContext` is cleared before `client.sendChat` resolves. If the gateway or provider rejects the send, the catch path reports the failure but the selected files are already gone, so retrying sends only the typed text. Clear the pending context after `sendChat` accepts the message, or restore it in the catch path.
  - confidence: 0.9
- **[P1] Clear stale context when no files attach:** `src/tui/tui-command-handlers.ts:278-279`
  - body: A confirmed upload that produces no `blocks` leaves the previous `pendingFileContext` intact. If every newly selected path is skipped as binary or fails to read, the next message silently includes the older files instead of nothing. Reset the pending context when a new confirmed upload starts or when no files attach.
  - confidence: 0.88
- **[P1] Filter the picker to regular files:** `src/tui/components/file-browser-select-list.ts:77-81`
  - body: Every non-directory entry is shown and selectable as a file, but the handler later calls `readFileSync` on the selected path. Device nodes, FIFOs, and sockets are non-directories too; selecting one can block or consume memory before binary detection runs. Only surface regular files, and validate again before reading.
  - confidence: 0.87
