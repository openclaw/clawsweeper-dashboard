## Best Possible Solution

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep queued uploads until send succeeds:** `src/tui/tui-command-handlers.ts:572`
  - body: `pendingFileContext` is cleared before `client.sendChat` resolves. If the send fails due to a timeout, disconnect, or provider error, the user cannot retry with the same file context because the next send only contains the typed message; clear it only after a successful send.
  - confidence: 0.91
- **[P1] Clear stale upload context when nothing attaches:** `src/tui/tui-command-handlers.ts:278-282`
  - body: When a replacement `/upload` selection contains only skipped or unreadable files, `blocks.length` stays zero and the previous `pendingFileContext` remains queued. That can attach old file contents after the user attempted to replace them; clear pending context before processing or explicitly on the no-blocks path.
  - confidence: 0.9
- **[P1] Filter selectable entries to regular files:** `src/tui/components/file-browser-select-list.ts:76-82`
  - body: Every non-directory entry is currently treated as a file, but the upload handler later calls `readFileSync` on selected paths. Selecting FIFOs, devices, or similar entries can block indefinitely or consume unbounded data, so the browser and handler should require `stat.isFile()` before allowing a path through.
  - confidence: 0.88
- **[P2] Use the current TUI package import:** `src/tui/components/file-browser-select-list.ts:10`
  - body: Current main depends on `@earendil-works/pi-tui` and no longer uses `@mariozechner/pi-tui`. This new component still imports the old package, so the rebased PR will fail or drift unless the added TUI imports are updated to the current package.
  - confidence: 0.84
