## Summary

Keep open. The feature is plausible, but this PR is stale against current TUI code, has blocking correctness/security issues in the file-read path and send path, and still needs real proof that selected file contents reach an actual sent prompt.

## What This Changes

The PR adds a TUI `/upload` slash command with an interactive file browser for selecting local files and prepending their contents to the next user message.

## Best Possible Solution

Rebase the feature onto current TUI main, preserve the existing send/session contract, harden file validation and pending-context lifecycle, then require redacted real-run proof that the next message contains the selected file context.

## Reproduction Assessment

Yes for the blocking review findings by source inspection: the PR branch imports a removed package, drops current sendChat fields, and reads selected paths without regular-file or size validation. I did not run the TUI because this was a read-only review.

## Solution Assessment

No. The requested UX is useful, but this branch should be rebased and hardened around current TUI contracts rather than merging the stale send path and unbounded file reader.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use the current TUI package import:** `src/tui/components/file-browser-select-list.ts:10`
  - body: Current main only declares `@earendil-works/pi-tui`, but this new component imports `@mariozechner/pi-tui`. After rebasing onto main this will fail module resolution/typecheck; update the new TUI imports to the current package name.
  - confidence: 0.96
- **[P1] Preserve current chat send session/run fields:** `src/tui/tui-command-handlers.ts:585-587`
  - body: This `sendChat` block is based on an older handler and drops current main's `sessionId: state.currentSessionId` plus the post-send `state.pendingChatRunId = runId`. That regresses local session continuity and Esc aborts during the pre-event wait; wrap the upload payload around the current send path instead.
  - confidence: 0.94
- **[P1] Validate selected paths before reading them:** `src/tui/tui-command-handlers.ts:257`
  - body: `readFileSync(filePath)` runs before any regular-file or size check, and the browser classifies every non-directory as selectable. Selecting a FIFO/device node or a huge file can hang or OOM the TUI; verify regular files and enforce a size cap before reading.
  - confidence: 0.9
- **[P1] Clear stale context when an upload attaches nothing:** `src/tui/tui-command-handlers.ts:278-282`
  - body: If a prior upload queued context and the next `/upload` confirms only unreadable or binary paths, `blocks.length` is zero and the old `pendingFileContext` remains queued. Clear pending context before processing a replacement upload or explicitly on the no-attachments path.
  - confidence: 0.88
- **[P2] Keep queued file context when send fails:** `src/tui/tui-command-handlers.ts:570-572`
  - body: The code nulls `pendingFileContext` before `client.sendChat` resolves, so a transient send failure discards the selected files and a retry sends only the typed text. Clear it after a successful send, or restore it in the catch path.
  - confidence: 0.83
