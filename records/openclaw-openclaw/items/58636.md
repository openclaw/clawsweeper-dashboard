## Summary

Keep open: current main does not implement the TUI `/upload` workflow, and this PR is still useful but not merge-ready. The branch has concrete source-level blockers around the current TUI package import, pending upload lifecycle, and filesystem entry validation.

## What This Changes

The PR adds a TUI `/upload` command, file-browser selector, theme wiring, pending file-context injection into the next chat message, and file-browser component tests.

## Best Possible Solution

Rebase or repair the branch against the current TUI package and command-handler contract, add handler coverage for upload lifecycle and file validation, then have maintainers decide whether core TUI should ship `/upload`.

## Reproduction Assessment

Yes for the PR blockers: PR-head source shows the stale package import, premature pending-context clear, stale replacement path, and non-regular file predicate. The broader `/upload` behavior is a new feature, so there is no current-main bug reproduction path.

## Solution Assessment

No. The feature direction may be useful, but this implementation should first rebase to the current TUI package, preserve pending uploads until `sendChat` succeeds, clear stale context on empty replacement uploads, and restrict reads to regular files.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Import pi-tui from the current package:** `src/tui/components/file-browser-select-list.ts:10`
  - body: Current main imports and depends on `@earendil-works/pi-tui`, but this new component imports `@mariozechner/pi-tui`. A rebased branch will fail to resolve the stale package unless the new TUI code uses the current package.
  - confidence: 0.93
- **[P1] Preserve pending uploads until send succeeds:** `src/tui/tui-command-handlers.ts:570-572`
  - body: `pendingFileContext` is cleared before `client.sendChat` resolves. If the gateway or provider rejects the send, the catch path reports the failure but the selected files are already gone, so retrying sends only the typed text.
  - confidence: 0.9
- **[P1] Clear stale context when a replacement upload attaches nothing:** `src/tui/tui-command-handlers.ts:278-279`
  - body: A confirmed upload that produces no `blocks` leaves the previous `pendingFileContext` intact. If every newly selected path is skipped or fails to read, the next message silently includes the older files instead of no attachment.
  - confidence: 0.88
- **[P1] Filter selectable entries to regular files:** `src/tui/components/file-browser-select-list.ts:77-81`
  - body: Every non-directory entry is shown and selectable as a file, but the handler later calls `readFileSync` on the selected path. Device nodes, FIFOs, and sockets are non-directories too; selecting one can block or consume memory before binary detection runs.
  - confidence: 0.87

## Security Review

Status: needs_attention

Summary: The diff introduces local file reads and currently allows non-regular filesystem entries to reach synchronous reads.
