## Summary

Keep open. Current main still lacks the TUI `/upload` feature, and this PR contains useful implementation work with real TUI screenshots, but PR head still has source-reproducible blockers in file validation and pending upload context lifecycle.

## What This Changes

The PR adds a TUI `/upload` slash command, file-browser overlay/theme wiring, and component tests to prepend selected local file contents to the next TUI message.

## Best Possible Solution

Repair this PR or a narrow replacement with bounded regular-file reads, correct pending-context lifecycle, focused handler coverage, TUI docs, and maintainer-owned changelog handling while preserving the TUI-only scope.

## Reproduction Assessment

Yes. By source inspection of PR head, selecting a huge or special non-directory path reaches `readFileSync` before validation, a failed `sendChat` drops queued attachments, and an all-skipped replacement upload leaves prior context queued.

## Solution Assessment

No. The TUI-only `/upload` direction is reasonable, but this implementation should validate regular files and size before reading and should preserve or clear pending context only at the intended lifecycle points.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Validate files before reading them:** `src/tui/tui-command-handlers.ts:257`
  - body: `readFileSync(filePath)` runs before checking that the selected path is a regular file or applying any size cap. Because the browser treats every non-directory entry as selectable, choosing a FIFO/device node or very large file can block or exhaust the TUI before binary detection runs.
  - confidence: 0.9
- **[P2] Clear queued uploads only after send succeeds:** `src/tui/tui-command-handlers.ts:570-572`
  - body: `pendingFileContext` is nulled before `client.sendChat` resolves. If the gateway/provider send rejects, retrying the same message loses the selected file context and sends only the typed text.
  - confidence: 0.86
- **[P2] Clear stale context when upload attaches no files:** `src/tui/tui-command-handlers.ts:278-282`
  - body: When a replacement upload is confirmed but all selected paths are skipped or fail to read, this branch leaves the previous `pendingFileContext` intact. The next message can silently send stale file contents.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff introduces an unbounded synchronous local file-read path in the TUI upload handler.

Concerns:

- **[medium] Unbounded local file read:** `src/tui/tui-command-handlers.ts:257`
