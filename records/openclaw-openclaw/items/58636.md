## Summary

Keep open. Current main still lacks `/upload`, and the PR adds a useful visible TUI feature with screenshot proof, but the branch is stale against current TUI contracts and has blocking correctness and file-read safety issues.

## What This Changes

The PR adds a TUI `/upload` slash command with a file-browser overlay, theme wiring, tests, and pending file-context prepending for the next chat message.

## Best Possible Solution

Rebase the feature onto current main, keep it scoped to the TUI, preserve queued uploads until send success, clear stale uploads on failed replacement attempts, restrict reads to regular bounded files, then let maintainers decide product fit.

## Reproduction Assessment

Yes for the PR blockers: PR-head source shows the stale package import, pending context cleared before awaited send, empty upload not clearing prior context, and non-regular entries selectable. The underlying `/upload` behavior is a new feature, not a current-main bug reproduction path.

## Solution Assessment

No. The feature direction may be useful, but this implementation should first be rebased to current TUI contracts and repaired for pending-context lifecycle and regular-file validation before maintainers judge whether to ship it.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Use the current pi-tui package:** `src/tui/components/file-browser-select-list.ts:10`
  - body: Current main imports TUI APIs from `@earendil-works/pi-tui`, but the new file browser still imports `@mariozechner/pi-tui`. A rebased branch will fail to resolve or split component types until these imports are updated consistently.
  - confidence: 0.93
- **[P1] Keep uploaded context until sendChat succeeds:** `src/tui/tui-command-handlers.ts:570-572`
  - body: `pendingFileContext` is cleared before `client.sendChat` resolves. If the gateway or provider rejects the send, retrying loses the selected file context and sends only the typed message.
  - confidence: 0.9
- **[P1] Clear old uploads when a replacement attaches nothing:** `src/tui/tui-command-handlers.ts:278-282`
  - body: When a later `/upload` confirms but every selected path is skipped or unreadable, this branch leaves the previous `pendingFileContext` queued. The next message can silently include old files after the user attempted to replace them with no attachment.
  - confidence: 0.88
- **[P1] Limit upload selection to regular files:** `src/tui/components/file-browser-select-list.ts:77-81`
  - body: Every non-directory entry is selectable, but the handler later calls `readFileSync` on selected paths. Device nodes, FIFOs, sockets, and similar entries can block the TUI or consume unbounded memory before binary detection runs.
  - confidence: 0.87
