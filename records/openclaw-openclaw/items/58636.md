## Summary

Keep open: the TUI upload feature is still missing on current main, and this PR remains useful, but the patch has blocking file-read and queued-context lifecycle bugs before it can merge.

## What This Changes

The PR adds a TUI `/upload` slash command, file-browser overlay, theme wiring, and component tests so selected local files can be prepended to the next TUI message.

## Best Possible Solution

Repair this PR or a narrow replacement with bounded regular-file reads, correct pending-context lifecycle, focused handler tests, TUI docs, and a maintainer-owned changelog entry while preserving the stated TUI-only scope.

## Reproduction Assessment

Yes, by source inspection of PR head: selecting a huge or special file reaches `readFileSync` before validation, failed `sendChat` drops queued attachments, and an all-skipped replacement upload leaves older context queued.

## Solution Assessment

No. The TUI-only direction is reasonable, but this implementation is not the best merge path until file reads are stat/size bounded and pending upload context is cleared only at the intended lifecycle points.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Validate files before reading them:** `src/tui/tui-command-handlers.ts:257`
  - body: `readFileSync(filePath)` runs before checking that the selected path is a regular file or applying any size cap. Because the browser treats every non-directory entry as selectable, choosing a FIFO/device node or very large file can block or exhaust the TUI before binary detection runs.
  - confidence: 0.9
- **[P2] Clear queued uploads only after send succeeds:** `src/tui/tui-command-handlers.ts:570-572`
  - body: `pendingFileContext` is nulled before `client.sendChat` resolves. If the gateway/provider send rejects, retrying the same message loses the selected file context and sends only the typed text.
  - confidence: 0.86
- **[P2] Clear stale context when upload attaches no files:** `src/tui/tui-command-handlers.ts:278-282`
  - body: When a replacement upload is confirmed but all selected paths are skipped or fail to read, this branch leaves the previous `pendingFileContext` intact. The next message can silently send stale file contents.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: Security review needs attention because the patch introduces an unbounded local file-read path in the TUI upload handler.

Concerns:

- **[medium] Unbounded local file read:** `src/tui/tui-command-handlers.ts:257`
  - body: The upload handler synchronously reads each selected path before checking regular-file status or enforcing a size limit. A selected special file or very large file can block or exhaust the TUI process, so the handler should stat and bound reads before loading file contents.
  - confidence: 0.88
