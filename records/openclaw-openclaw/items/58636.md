## Summary

Keep this PR open, but it needs changes before merge. Current main still lacks `/upload`, and the PR has real TUI proof, but the branch is stale against the current TUI package and has pending-context and filesystem-read blockers.

## What This Changes

The PR adds a TUI `/upload` command, a file-browser selector and theme, pending file-context injection into the next chat message, and component tests.

## Best Possible Solution

Repair the branch against the current TUI package and command-handler contract, constrain uploads to safe regular-file reads, preserve pending upload context across failed sends, and then let maintainers decide whether core TUI should ship `/upload`.

## Reproduction Assessment

Yes for the PR blockers: PR-head source shows the stale package import, premature pending-context clear, stale replacement path, and non-regular file predicate. The broader `/upload` behavior is a new feature, so there is no current-main bug reproduction path.

## Solution Assessment

No. The feature direction is plausible, but this implementation should first rebase to the current package and command-handler surface, preserve upload context until send succeeds, clear stale context on empty replacement uploads, and restrict reads to regular files.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Import pi-tui from the current package:** `src/tui/components/file-browser-select-list.ts:10`
  - body: Current main depends on `@earendil-works/pi-tui`, but this new component imports `@mariozechner/pi-tui`. A rebased branch will fail to resolve the old package unless the new TUI code uses the current dependency.
  - confidence: 0.93
- **[P1] Preserve pending uploads until send succeeds:** `src/tui/tui-command-handlers.ts:570-572`
  - body: `pendingFileContext` is cleared before `client.sendChat` resolves. If the gateway or provider rejects the send, the selected files are lost and retrying sends only the typed text.
  - confidence: 0.9
- **[P1] Clear stale context when replacement upload attaches nothing:** `src/tui/tui-command-handlers.ts:278-282`
  - body: A confirmed upload that produces no `blocks` leaves the previous `pendingFileContext` intact. If every newly selected path is skipped or fails to read, the next message silently includes older files instead of no attachment.
  - confidence: 0.88
- **[P1] Filter selectable entries to regular files:** `src/tui/components/file-browser-select-list.ts:77-81`
  - body: Every non-directory entry is shown as a selectable file, but the handler later calls `readFileSync` on the selected path. Device nodes, FIFOs, and sockets can block or consume memory before binary detection runs.
  - confidence: 0.87

## Security Review
