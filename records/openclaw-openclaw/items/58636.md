## Summary

Keep open: the feature is plausible and the screenshots show the TUI flow, but the current patch still has blocking upload lifecycle and filesystem-safety defects that can drop selected context, leak stale context, or hang the TUI on non-regular files.

## What This Changes

The PR adds a TUI `/upload` slash command, file-browser select component, theme wiring, and tests to attach selected local file contents to the next chat message.

## Best Possible Solution

Keep the feature branch open, fix the upload lifecycle and regular-file validation, add focused regression tests, then rebase onto the current TUI code before maintainer product review.

## Reproduction Assessment

Yes for the review blockers: source inspection of the PR head shows the pending-context lifecycle paths and non-regular file selection path, and a local stat check confirms the special-file predicate gap. The broader feature itself is not a bug reproduction; it is a new TUI capability.

## Solution Assessment

No. The direction may be useful, but this implementation should not merge until it preserves pending uploads across send failures, clears stale pending context when a replacement upload attaches nothing, and restricts reads to regular files.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve pending uploads until send succeeds:** `src/tui/tui-command-handlers.ts:570-572`
  - body: `pendingFileContext` is cleared before `client.sendChat` resolves. If the gateway or provider rejects the send, the catch path reports the failure but the selected files are already gone, so retrying sends only the typed text. Clear the pending context only after `sendChat` accepts the message, or restore it in the catch path.
  - confidence: 0.9
- **[P1] Clear stale context when no files attach:** `src/tui/tui-command-handlers.ts:278-279`
  - body: A confirmed upload that produces no `blocks` leaves the previous `pendingFileContext` intact. If every newly selected path is skipped as binary or fails to read, the next message silently includes the older files instead of nothing. Reset the pending context when a new confirmed upload starts or when no files attach.
  - confidence: 0.88
- **[P1] Filter the picker to regular files:** `src/tui/components/file-browser-select-list.ts:77-81`
  - body: Every non-directory entry is shown and selectable as a file, but the handler later calls `readFileSync` on the selected path. Device nodes, FIFOs, and sockets are non-directories too; selecting something like `/dev/zero` can block or consume memory before the binary check runs. Only surface `stat.isFile()` entries, and consider validating again before reading.
  - confidence: 0.87

## Security Review

Status: needs_attention

Summary: The patch introduces local file reads and needs a regular-file guard before merge to avoid local denial-of-service behavior from special filesystem entries.

Concerns:

- **[medium] Unbounded reads from special files:** `src/tui/components/file-browser-select-list.ts:77`
  - body: The browser treats every non-directory as a selectable file and the handler reads selected paths with `readFileSync`; character devices or FIFOs can block or stream indefinitely, freezing the TUI or exhausting memory before binary detection can help.
  - confidence: 0.86
