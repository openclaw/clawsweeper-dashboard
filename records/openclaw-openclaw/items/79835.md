## Summary

Keep this PR open: it targets a real image-model routing bug and has useful live proof, but the patch still has source-level correctness gaps around providerless imageModel refs and temporary auth-profile handling.

## What This Changes

The PR adds Gateway image attachment detection, imageModel override plumbing through auto-reply model selection, MIME sniffing hardening for attachments, and tests for image versus non-image routing.

## Best Possible Solution

Keep the imageModel auto-switch direction, but canonicalize the temporary override before dispatch and make the automatic provider switch non-mutating for stored session auth/profile state.

## Reproduction Assessment

Yes. Source inspection on current main shows chat.send only checks the session model's image support and offloads images for text-only models, while the docs say imageModel is the intended fallback when the primary cannot accept images.

## Solution Assessment

No. The PR is close, but the current patch is not the best complete fix because it validates providerless imageModel refs with one provider context and dispatches them with another, and it can still mutate persisted auth-profile state during a temporary image-model switch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Canonicalize the image override before dispatch:** `src/gateway/server-methods/chat.ts:2243`
  - body: Here `modelOverride` remains the raw `imageModelPrimary`, even though the code below resolves it with `imageModelProvider` only for validation. `getReplyFromConfig` later resolves providerless `opts.modelOverride` with the normal default provider, so a config like a text-only default plus providerless imageModel can be validated as the image provider but executed as `<defaultProvider>/<imageModel>`. Pass the canonical `provider/model` from `overrideRef`, or carry the provider context through reply options.
  - confidence: 0.88
- **[P2] Preserve stored auth profiles during image overrides:** `src/auto-reply/reply/model-selection.ts:269-271`
  - body: This only skips the cleanup inside model selection, but the same run still calls `resolveSessionAuthProfileOverride` with the temporary image provider and that helper clears incompatible persisted session auth profiles. A user with a provider-specific session auth profile can lose it after one image upload. The image override should ignore incompatible auth for this turn or choose a compatible image-model profile without mutating the stored session selection.
  - confidence: 0.78
