## Summary

Keep open: the PR addresses the right Gateway/Dashboard imageModel routing bug and the supplied live proof covers the happy path, but current head still has providerless-image-model and allowlist-routing regressions that should block merge.

## What This Changes

The PR adds Gateway image attachment sniffing plus per-turn imageModel modelOverride/fallback plumbing through auto-reply, with tests for image and non-image attachments.

## Best Possible Solution

Keep the Gateway-owned imageModel switch, but compute a canonical usable override before attachment parsing and preserve the existing text-only offload path when no allowed vision override can run.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path for both the original bug and the PR regressions: configure a text-only primary plus `agents.defaults.imageModel`, send an image via `chat.send`, and inspect the modelOverride/attachment routing decisions.

## Solution Assessment

No. The Gateway/reply boundary is the right place to solve this, but the PR needs to canonicalize providerless overrides and make inline image routing depend on an actually usable, allowlisted vision model.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Canonicalize providerless image overrides before dispatch:** `src/gateway/server-methods/chat.ts:2714`
  - body: Here `modelOverride` can still be the raw providerless `agents.defaults.imageModel.primary`, for example `moondream`. Gateway may validate it with the image-model provider, but `getReplyFromConfig` later resolves `opts.modelOverride` with the session default provider, so a text primary like `deepseek/...` can dispatch as `deepseek/moondream` instead of the configured `ollama/moondream`. Pass the canonical provider/model ref into `replyOptions`.
  - confidence: 0.92
- **[P2] Don't inline images when the override will be rejected:** `src/gateway/server-methods/chat.ts:2311-2312`
  - body: `supportsImages` is set from `imageModelSupportsImages` before the downstream allowlist check can reject `modelOverride`. With a text-only primary and `agents.defaults.models` that excludes the configured image model, attachment parsing keeps the image inline, then auto-reply falls back to the text model with inline images instead of preserving the MediaPaths offload path. Decide the usable allowlisted override before setting `supportsImages`.
  - confidence: 0.86
