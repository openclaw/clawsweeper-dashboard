## Summary

Keep open. This PR is aimed at a real documented behavior gap and includes sufficient live proof, but the current patch still has two P2 routing gaps that can send image turns through the wrong model path.

## What This Changes

The PR adds Dashboard/Gateway image attachment sniffing, temporary imageModel overrides with canonicalized fallbacks, auth-profile safeguards, and regression tests for image versus non-image attachments.

## Best Possible Solution

Keep the Gateway-owned temporary image override, but choose an allowed vision primary/fallback before inline image parsing and keep image turns on imageModel fallbacks during runtime retries.

## Reproduction Assessment

Yes. Current main source shows Dashboard chat attachments are parsed using the resolved session model path, with text-only sessions routing images to MediaPaths and no GetReplyOptions modelOverride path.

## Solution Assessment

No. The Gateway override boundary is the right direction, but the patch must enforce allowlist before preserving inline images and must carry imageModel fallbacks into runtime retry resolution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Gate inline image parsing on an allowed override:** `src/gateway/server-methods/chat.ts:2351`
  - body: Here `supportsImages` becomes true as soon as the configured imageModel looks vision-capable. The reply layer can still reject that same `modelOverride` because it is not in `agents.defaults.models`; when that happens, `replyOptions.images` remains populated and the default text model gets inline images instead of the existing MediaPaths/media-understanding path. Resolve an allowed imageModel primary/fallback before setting `supportsImages`, and leave `routeImageOffloadsAsMediaPaths` true when none is allowed.
  - confidence: 0.86
- **[P2] Carry image fallbacks into runtime retries:** `src/auto-reply/reply/get-reply.ts:976`
  - body: Only the applied-override flag is passed into `runPreparedReply`; `FollowupRun` still has no prepared image fallback list, and `resolveModelFallbackOptions` derives retries from `agents.defaults.model.fallbacks`. If the selected vision model fails, the image turn can retry normal text fallbacks with images attached instead of `agents.defaults.imageModel.fallbacks`.
  - confidence: 0.87
