## Summary

Keep open: the branch has useful live proof and appears to address the earlier MIME spoof/OOXML findings, but the latest auth-profile change still drops valid same-provider session auth profiles during image overrides.

## What This Changes

The branch adds Gateway image-attachment sniffing and temporary auto-reply `imageModel` overrides/fallbacks so Dashboard image uploads can use a vision-capable model while non-image attachments stay on the session model.

## Best Possible Solution

Keep the Gateway image override design, but resolve a provider-correct turn-scoped auth profile: preserve same-provider profiles, suppress mismatched cross-provider profiles, and leave the stored session profile untouched.

## Reproduction Assessment

Yes for the remaining PR blocker by source inspection: PR head clears `authProfileId` for every image override, while current runner auth scoping supports preserving provider-matching profiles. I did not live-run the original Dashboard flow, but the contributor supplied usable after-fix proof for it.

## Solution Assessment

No, not as-is. Passing a temporary image-model override from Gateway is a reasonable narrow fix, but auth handling needs provider-correct profile resolution before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve same-provider auth profiles for image overrides:** `src/auto-reply/reply/get-reply-run.ts:885-886`

  - confidence: 0.86
