## Summary

Keep this PR open: it is addressing a real documented imageModel routing bug, but the latest head still has two merge-blocking source gaps around allowlist-gated inline image handling and imageModel retry fallbacks.

## What This Changes

The PR adds Gateway image attachment sniffing, a temporary imageModel modelOverride path, model override fallback handling, auth-profile safeguards, and regression tests.

## Best Possible Solution

Land a narrowed Gateway/reply fix that proves an allowlisted vision model or imageModel fallback before inline image parsing, then keeps image turns on the imageModel fallback chain during runtime retries.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main documents imageModel fallback behavior but Dashboard attachment routing only considers the session model’s image support and has no modelOverride path.

## Solution Assessment

No, not yet. The Gateway-owned temporary override is the right direction, but the patch must gate inline images on an allowlisted vision selection and thread imageModel fallbacks into retry resolution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Keep images on the media path when the override is blocked:** `src/gateway/server-methods/chat.ts:2351-2352`
  - body: Here `supportsImages` can become true from the configured imageModel before the reply layer verifies that the same override or fallback is allowed by `agents.defaults.models`. If `getReplyFromConfig` rejects the override, the turn falls back to the text model while `replyOptions.images` still carries inline images, bypassing the safer MediaPaths/media-understanding path. Resolve an allowlisted vision model or fallback before setting `supportsImages`, or keep image offloads routed as MediaPaths when none is allowed.
  - confidence: 0.86
- **[P2] Carry image fallbacks into runtime retries:** `src/auto-reply/reply/get-reply.ts:976`
  - body: The PR prepares `modelOverrideFallbacks`, but the prepared run only receives `hasAppliedImageModelOverride`; `FollowupRun.run` and `resolveModelFallbackOptions` still derive retries from the normal model fallback contract. If the selected vision model fails, an image turn can retry text fallbacks with inline images instead of the configured imageModel fallbacks. Add the prepared image fallback list to the run and fallback resolver path.
  - confidence: 0.87
