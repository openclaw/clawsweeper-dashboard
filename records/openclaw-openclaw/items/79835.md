## Summary

Keep open: the latest PR head still clears the run auth profile for every image-model override, so same-provider image turns can bypass a valid user-selected auth profile. The contributor supplied sufficient live Dashboard proof for the main image/non-image routing behavior, but credential routing needs a narrow fix before merge.

## What This Changes

The PR adds Gateway image-attachment sniffing and temporary `imageModel` override/fallback handling for Dashboard chat, plus auto-reply helper coverage and Gateway tests.

## Best Possible Solution

Keep the image override design, but resolve a provider-scoped turn auth profile: preserve same-provider profiles, suppress mismatched cross-provider profiles, and avoid mutating the stored session profile for the temporary image turn.

## Reproduction Assessment

Yes for the remaining blocker by source inspection: PR head clears `authProfileId` for every image override, while current auth scoping and resolver code preserve provider-valid session profiles. I did not run a live same-provider auth-profile turn.

## Solution Assessment

No, not as-is. The Gateway image override path is a reasonable narrow fix, but the run auth handling should be provider-scoped instead of clearing all profiles whenever an image override is active.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve same-provider auth profiles for image overrides:** `src/auto-reply/reply/get-reply-run.ts:885-886`
  - body: During an image override on the same provider, this unconditionally drops the session auth profile. Existing auth scoping keeps that profile when the run provider matches, so a selected account/profile can be silently replaced by default credentials for image turns; clear only mismatched cross-provider profiles or resolve a provider-correct turn profile.
