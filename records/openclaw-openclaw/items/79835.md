## Summary

Keep open: the current PR head still has two source-proven routing regressions and only test-based proof, so it is not ready for merge.

## What This Changes

The PR adds Gateway image attachment sniffing, imageModel override/fallback plumbing through auto-reply, and related tests.

## Best Possible Solution

Route chat.send image attachments through the current session model when it supports vision; otherwise switch only to a resolved, allowlisted imageModel/fallback and keep unresolved or text-only cases on the MediaPaths/media-understanding path.

## Reproduction Assessment

No high-confidence live reproduction is present. Source inspection of PR head shows the PR-introduced regression paths, but the PR body only reports tests rather than a real Dashboard/Gateway run.

## Solution Assessment

No. The narrow maintainable fix should preserve existing session-model routing, verify the image override resolves to a runnable vision model, and reuse the existing offload path when it does not.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Preserve vision-capable session models:** `src/gateway/server-methods/chat.ts:2216-2219`
  - body: This assigns `imageModel` as `modelOverride` whenever image attachments are present, before checking whether the selected session/channel model already supports images. The documented contract is fallback routing only when the selected/default model cannot accept image input, so a user on a vision-capable model would be silently switched for every image turn.
  - confidence: 0.74
- **[P2] Resolve image overrides before keeping attachments inline:** `src/gateway/server-methods/chat.ts:2233-2235`
  - body: When override resolution fails, this branch still uses the raw override so `resolveModelSupportsVision` can match it against `imageModelConfig` and mark `supportsImages` true. `getReplyFromConfig` later drops the unresolved override and runs the default model, so a text-only session can receive inline `images` instead of the existing MediaPaths/media-understanding path.
  - confidence: 0.87
