## Summary

Keep this PR open: it addresses a real documented imageModel routing gap and the Gateway attachment sniffing direction is sound, but the current branch drops the configured imageModel fallback list after selecting the primary override, so it is not merge-ready yet.

## What This Changes

The PR adds Gateway image-attachment sniffing plus auto-reply modelOverride plumbing so Dashboard/WebChat image turns can temporarily use `agents.defaults.imageModel` when the selected session model is text-only.

## Best Possible Solution

Land the Gateway-owned imageModel auto-switch after carrying imageModel fallbacks into the run fallback contract and adding focused coverage for the primary-allowed fallback case.

## Reproduction Assessment

Yes by source inspection: current docs promise imageModel routing when the primary model cannot accept images, while current main only evaluates the selected session model during Dashboard chat attachment dispatch. I did not run a live pre-fix turn in this read-only review.

## Solution Assessment

No, not as-is. The Gateway decision point is the right owner boundary, but the implementation should preserve `agents.defaults.imageModel.fallbacks` for actual model failover, not only use them when the primary override is allowlist-blocked.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve imageModel fallbacks after selecting the override:** `src/auto-reply/reply/get-reply.ts:335-337`
  - body: When the imageModel primary is allowlisted, this branch sets `provider`/`model` and drops `opts.modelOverrideFallbacks`. Please carry the filtered image fallback list into the run fallback options and cover the primary-allowed case.
