## Summary

## What This Changes

The PR adds Gateway content-sniffed image attachment detection, per-turn imageModel override/fallback plumbing, and tests for image versus non-image attachments.

## Best Possible Solution

Keep the PR open and require a catalog-backed vision guard plus redacted diagnostic proof for image, PDF, and fake-metadata routing in a real Gateway run.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source inspection: configure agents.defaults.imageModel.primary to a catalog-known text-only or mistyped model, send an actual image through chat.send, and the PR can still return supportsImages=true from imageModel config after catalog support is not confirmed.

## Solution Assessment

Is this the best way to solve the issue? No. The branch should require a resolved, allowlisted, catalog-confirmed vision model before keeping images inline for the override; otherwise it should keep the existing media-understanding offload path.
