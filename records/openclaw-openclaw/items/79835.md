## Summary

Keep open: the bug target is plausible and overlaps with https://github.com/openclaw/openclaw/pull/38881, but this PR head is not merge-ready because it introduces two image-routing regressions and the supplied proof is only tests/checks, not observed Gateway or Dashboard behavior.

## What This Changes

The PR adds Gateway image attachment sniffing, passes image model override/fallback options into auto-reply, and adds model-routing tests.

## Best Possible Solution

Rework the branch so `imageModel` is fallback routing for true image attachments only: preserve selected vision-capable models, use only resolved allowlisted imageModel primary/fallback refs, and keep unresolved or text-only cases on the existing MediaPaths/media-understanding path.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No live runtime reproduction is present, but source inspection gives a high-confidence reproduction path for the PR-introduced regressions: image attachments with a vision-capable selected model, or with an unresolved imageModel override, exercise the bad branches.

## Solution Assessment

Is this the best way to solve the issue? No. The direction is useful, but the current patch should be revised to preserve existing vision-capable routing and resolve image overrides before using them to decide inline image handling.

  - confidence: 0.88
