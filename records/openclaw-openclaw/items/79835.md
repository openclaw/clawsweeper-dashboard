## Summary

Keep open: this is a plausible bug-fix area, but the current PR still introduces source-backed regressions in image-model routing and the supplied screenshots only prove the image-upload path, not the non-image or fake-metadata cases claimed by the patch.

## What This Changes

The PR adds Gateway image attachment sniffing, per-turn image `modelOverride`/fallback plumbing into auto-reply, and tests for image versus non-image Dashboard chat attachments.

## Best Possible Solution

Keep the shared attachment parser as the MIME source of truth and add any direct per-turn imageModel switch only after preserving native vision-capable session models and requiring catalog-confirmed vision support.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source inspection: configure a vision-capable primary plus `agents.defaults.imageModel` and send an image through `chat.send`, or configure an unresolved/text-only imageModel and observe the PR path setting image support before catalog-backed validation.

## Solution Assessment

Is this the best way to solve the issue? No. The narrower maintainable fix is to preserve the selected session model when it already supports images, and only use a resolved catalog-backed imageModel or existing media-understanding route when the selected model cannot accept images.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve vision-capable session models:** `src/gateway/server-methods/chat.ts:2216-2219`
  - body: This assigns `modelOverride` from `imageModel` whenever an image attachment is present, before checking whether the selected session model already supports images. The documented contract says `imageModel` is only for primaries that cannot accept images, so users on a vision-capable model would be silently switched for every image turn.
  - confidence: 0.86
- **[P2] Require catalog-backed vision support:** `src/auto-reply/reply/image-model-helpers.ts:300-307`
  - body: This returns true solely because the ref appears in `imageModel` config before checking the catalog. An unresolved alias or configured text-only imageModel can therefore make Gateway keep images inline while auto-reply falls back to the default/text-only path instead of the safer media-understanding route.
  - confidence: 0.88
