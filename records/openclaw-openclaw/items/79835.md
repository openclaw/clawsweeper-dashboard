## Summary

Keep open: the PR addresses a real imageModel routing gap and now has visible live proof, but the latest head still treats imageModel configuration itself as vision support after catalog lookup does not confirm image input support.

## What This Changes

The PR adds Gateway attachment sniffing and per-turn imageModel modelOverride/fallback plumbing with auto-reply and Gateway tests for image versus non-image chat attachments.

## Best Possible Solution

Require the chosen imageModel primary or fallback to resolve and be catalog-confirmed as image-capable before marking supportsImages true; otherwise route image attachments through the existing media-understanding path.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source inspection: configure agents.defaults.imageModel.primary to a catalog-known text-only or mistyped model, send an actual image through chat.send, and the PR path can still mark supportsImages true from imageModel config after catalog support is not confirmed.

## Solution Assessment

Is this the best way to solve the issue? No. The PR improves attachment detection, but the maintainable fix is to make the imageModel switch depend on a resolved, allowlisted, catalog-confirmed vision model before keeping images inline.
