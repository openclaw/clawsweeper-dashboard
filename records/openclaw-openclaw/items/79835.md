## Summary

Keep open: the image-model override direction is useful and the supplied live proof is sufficient, but the latest head still breaks the existing OOXML attachment MIME contract and now risks forwarding a stored primary-provider auth profile into a temporary image-provider run.

## What This Changes

The PR adds Gateway attachment content sniffing and a temporary auto-reply `imageModel` override so Dashboard image uploads can run on a configured vision model while non-image attachments stay on the session model.

## Best Possible Solution

Keep the Gateway image override design, but preserve specific non-image MIME recovery and make temporary image-provider auth use that provider's profile or no profile without mutating the stored session profile.

## Reproduction Assessment

Yes for the PR-introduced blockers: PR CI reproduces the `.xlsx` MIME regression, and source inspection shows the cross-provider image override can forward the stored primary-provider auth profile. I did not independently live-reproduce the original image-model bug on current main, but the contributor supplied usable after-fix live proof.

## Solution Assessment

No, not as-is. Passing a temporary image-model override from Gateway is a reasonable fix direction, but the shared MIME fallback and auth-profile handling need narrower repairs before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Preserve filename-derived OOXML MIME on generic sniff:** `src/gateway/chat-attachments.ts:273-274`
  - body: This clears the filename-derived MIME whenever sniffing returns a generic container, so an `.xlsx` upload with no provided MIME now falls through to `application/zip` instead of `application/vnd.openxmlformats-officedocument.spreadsheetml.sheet`. The existing Gateway attachment test fails on this PR for that exact case; keep rejecting misleading image filenames without dropping specific non-image Office labels.
  - confidence: 0.94
- **[P2] Do not forward stale auth profiles to image overrides:** `src/auto-reply/reply/get-reply-run.ts:883-885`
  - body: When an image override switches providers, this branch sets `authProfileId` to the stored session profile from the original provider. The runner then treats it as scoped to the image-provider run because `run.provider` has already changed, so a session pinned to one provider can pass that provider's auth profile into a different provider's image turn. Preserve the stored session profile without forwarding it to the temporary image-provider run, or resolve a provider-correct turn-scoped profile.
  - confidence: 0.87
