## Summary

Keep open: current PR head still regresses the documented imageModel fallback semantics in https://github.com/openclaw/openclaw/pull/79835. The diff can switch away from a vision-capable primary model and can also treat unresolved or text-only imageModel refs as vision-capable before catalog validation; the supplied proof does not visibly cover the non-image or fake-metadata cases.

## What This Changes

The PR adds Gateway attachment content sniffing, threads image modelOverride/fallback options into auto-reply, and adds tests for image versus non-image chat.send attachment routing.

## Best Possible Solution

Preserve native vision-capable session models first, then use only a resolved catalog-backed imageModel or fallback when the selected model cannot accept images; otherwise keep the existing MediaPaths/media-understanding route.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source inspection: configure a vision-capable primary plus agents.defaults.imageModel and send an image through chat.send, or configure an unresolved/text-only imageModel and observe the PR path setting supportsImages before catalog-backed validation.

## Solution Assessment

Is this the best way to solve the issue? No. The narrow maintainable fix is to check the selected session model’s image capability first, then switch only to a resolved vision-capable imageModel when the primary cannot accept images.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve vision-capable session models:** `src/gateway/server-methods/chat.ts:2216-2219`
  - body: This assigns modelOverride from imageModel whenever an image attachment is present, before checking whether the selected session model already supports images. The documented contract says imageModel is only for primaries that cannot accept images, so users on a vision-capable model would be silently switched for every image turn.
  - confidence: 0.86
- **[P2] Require catalog-backed vision support:** `src/auto-reply/reply/image-model-helpers.ts:300-307`

An unresolved alias or configured text-only imageModel can therefore make Gateway keep images inline, while getReply falls back to the default/text-only path instead of MediaPaths/media-understanding.
  - confidence: 0.88
