## Summary

Keep open: the PR targets a real imageModel routing gap, but the current patch is not mergeable because it can override already-vision-capable session models and can inline images without proving the override is a runnable vision model.

## What This Changes

The PR adds Gateway image attachment sniffing, threads imageModel override/fallback options into auto-reply, and adds tests around image versus non-image chat.send attachments.

## Best Possible Solution

Keep the imageModel path as fallback routing only: preserve a selected vision-capable model, switch only to a resolved allowlisted imageModel or fallback that the catalog says supports image input, and otherwise keep the existing MediaPaths/media-understanding route.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source inspection: send chat.send with an image while the active session model already supports images, or configure an unresolved/non-vision imageModel and observe the PR path keeping images inline anyway.

## Solution Assessment

Is this the best way to solve the issue? No. The safer fix is to reuse the current session-model capability check first, then apply imageModel only after resolving and validating a runnable vision-capable override.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve vision-capable session models:** `src/gateway/server-methods/chat.ts:2216-2219`
  - body: This sets imageModel as the modelOverride whenever an image attachment is present, before checking whether the selected session/channel model already supports images. The documented contract is fallback routing only when the primary cannot accept images, so users on a vision-capable model would be silently switched for every image turn.
  - confidence: 0.84
- **[P2] Require catalog-backed vision support:** `src/auto-reply/reply/image-model-helpers.ts:300-307`
  - body: This helper returns true just because the provider/model appears in imageModel config, before checking the catalog. An unresolved alias or configured text-only imageModel can therefore make Gateway keep images inline, while getReply later falls back to the default model instead of using the MediaPaths/media-understanding path.
  - confidence: 0.88
