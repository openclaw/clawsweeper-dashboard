## Summary

Keep open: the image-model override direction is plausible and the contributor supplied live proof, but the patch regresses the existing non-image attachment parser contract. CI shows `src/gateway/chat-attachments.test.ts` now returns `application/zip` for an `.xlsx` upload where current main expects the specific OOXML MIME type.

## What This Changes

The branch adds Dashboard/Gateway image attachment sniffing and a temporary `imageModel` model override path so image uploads can switch from a text-only session model to a configured vision-capable image model while non-image attachments stay on the default model.

## Best Possible Solution

Keep the Gateway image-model override approach, but narrow the MIME trust change so generic container sniffing rejects misleading image filenames without losing specific non-image OOXML filename-derived MIME; then rerun the focused parser and gateway image override tests.

## Reproduction Assessment

Yes for the PR-introduced regression: the current PR CI fails the existing `.xlsx` MIME recovery test, and the changed `trustedLabelMime` branch explains the failure. For the original image override bug, the contributor supplied live proof, but I did not independently run a current-main live repro.

## Solution Assessment

No, not as-is. Passing a temporary image-model override from Gateway is a reasonable fix direction, but the shared attachment parser change is too broad and breaks established non-image attachment MIME behavior.
