## Summary

Keep open: this PR targets a valid imageModel routing bug, but the current head still treats a configured imageModel as vision-capable after the catalog fails to confirm image support, which can route real images inline to a text-only or mistyped model.

## What This Changes

The PR adds Gateway attachment sniffing and imageModel override/fallback plumbing for Dashboard chat, plus auto-reply and Gateway tests for image versus non-image attachments.

## Best Possible Solution

Require the selected imageModel primary or fallback to resolve through the allowlist and catalog as image-capable before setting supportsImages; otherwise keep the existing media-understanding offload path.

## Reproduction Assessment

Yes from source inspection: configure imageModel.primary to a catalog-known text-only or mistyped model, send an image through chat.send, and the helper can still return true from config membership after catalog support is not confirmed.

## Solution Assessment

No. The diff moves in the right direction, but the maintainable fix is to make catalog-confirmed vision support required before Gateway keeps images inline for the override.
