## Summary

For https://github.com/openclaw/openclaw/pull/79835, keep open: the PR addresses a real Gateway image-model routing gap, but the latest head still applies the override before confirmed vision support and the proof is screenshot-only for hidden routing decisions.

## What This Changes

The PR adds Gateway image attachment sniffing, per-turn imageModel modelOverride/fallback plumbing, and auto-reply/Gateway tests for image vs non-image attachments.

## Best Possible Solution

Gate the one-turn imageModel override on a resolved model that is catalog-confirmed vision-capable; otherwise keep images on the existing offloaded media-understanding path and preserve the current session/default model.

## Reproduction Assessment

Yes. Source inspection of the PR head gives a high-confidence path: configure a catalog-known text-only model as `agents.defaults.imageModel.primary`, send an image via `chat.send`, and the branch still passes `modelOverride` even though image support is false.

## Solution Assessment

No. The direction is plausible, but the override must be assigned only after vision validation succeeds; otherwise a misconfigured imageModel can become the main text-only reply model.
