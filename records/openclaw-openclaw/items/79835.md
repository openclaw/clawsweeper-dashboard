## Summary

Keep open: this PR addresses a real docs-backed Gateway image-model switching bug and includes live proof, but the new attachment detector still trusts an image filename after content sniffing identifies a generic ZIP/container, so renamed DOCX/ZIP content can still trigger the image-model override.

## What This Changes

The branch adds Gateway image-attachment detection plus auto-reply `modelOverride` plumbing, fallback filtering, and regression tests so Dashboard image sends can use `agents.defaults.imageModel` when the active model lacks vision.

## Best Possible Solution

Keep the Gateway-owned image-model switch, but tighten MIME trust so generic-sniffed non-image containers cannot become image attachments through filename metadata, then land with the focused Gateway and auto-reply tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure a text-only primary plus a vision-capable `agents.defaults.imageModel`, send an image through Dashboard `chat.send`, and current main dispatches without a `modelOverride`.

## Solution Assessment

No. The fix belongs at the Gateway/reply boundary, but the proposed attachment detection still has a filename-trust hole that can route renamed non-image containers through the vision override path.
