## Summary

Keep open: this is a plausible Gateway/auto-reply fix area, but the patch introduces model-routing regressions and the external PR currently provides only test-based proof rather than after-fix real behavior evidence.

## What This Changes

This PR adds Gateway content-sniffed image attachment detection, imageModel override/fallback plumbing into auto-reply, and related auto-reply/Gateway tests.

## Best Possible Solution

Route chat.send image attachments through the current session model when it supports vision, otherwise switch only to a resolved, allowlisted imageModel/fallback and keep unresolved or text-only cases on the existing MediaPaths/media-understanding path.

## Reproduction Assessment

No high-confidence live reproduction is present. Source inspection does show the PR-introduced regression paths, but the PR body only reports tests rather than a real Dashboard/Gateway run.

## Solution Assessment

No. The narrow maintainable fix should preserve existing session-model routing, verify the image override resolves to a runnable vision model, and reuse the existing offload path when it does not.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Resolve image overrides before keeping attachments inline:** `src/gateway/server-methods/chat.ts:2234`
  - body: When override resolution fails, this branch still uses the raw override so `resolveModelSupportsVision` can match it against `imageModelConfig` and mark `supportsImages` true. `getReplyFromConfig` later drops the unresolved override and runs the default model, so a text-only session can receive inline `images` instead of the existing MediaPaths/media-understanding path. Keep images offloaded unless the override or one of its fallbacks resolves to a runnable vision model.
  - confidence: 0.87
- **[P2] Preserve vision-capable session models:** `src/gateway/server-methods/chat.ts:2215-2218`
  - body: This assigns `imageModel` whenever an image attachment is present, before checking whether the selected session/channel model already supports images. The documented contract is fallback routing only when the selected/default model cannot accept image input, so a user on a vision-capable model would be silently switched to `agents.defaults.imageModel` for every image turn.
  - confidence: 0.74
