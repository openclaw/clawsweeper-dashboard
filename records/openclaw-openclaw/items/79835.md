## Summary

Keep open: the PR is trying to fix a real imageModel/Gateway routing gap, but the current head still introduces image-routing regressions and only supplies test/CI output rather than real Gateway behavior proof.

## What This Changes

The PR adds Gateway image attachment content sniffing, passes imageModel override/fallback options into auto-reply, and adds tests for image versus non-image chat.send routing.

## Best Possible Solution

Keep imageModel as fallback routing for true image attachments: preserve the current selected model when it supports vision, otherwise use a resolved allowlisted imageModel/fallback or fall back to MediaPaths/media-understanding.

## Reproduction Assessment

No high-confidence live reproduction is present. Source inspection of the PR head gives high-confidence reproduction paths for the introduced regressions: an image turn on a vision-capable selected model, and an image turn with an unresolved imageModel override.

## Solution Assessment

No. The patch should first preserve selected vision-capable models, resolve and allowlist imageModel candidates before treating them as runnable, and keep unresolved/text-only cases on the existing MediaPaths path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve vision-capable session models:** `src/gateway/server-methods/chat.ts:2216-2219`
  - body: This assigns imageModel as modelOverride whenever image attachments are present, before checking whether the selected session/channel model already supports images. The documented contract is fallback routing only when the primary model cannot accept images, so users on a vision-capable selected model would be silently switched for every image turn.
  - confidence: 0.84
- **[P2] Resolve image overrides before keeping images inline:** `src/gateway/server-methods/chat.ts:2233-2235`
  - body: When override resolution fails, this branch still feeds the raw override string into vision detection, which can match imageModelConfig and keep attachments inline. getReplyFromConfig later drops that unresolved override and runs the default model, so a text-only default can receive inline images instead of the existing MediaPaths/media-understanding path.
  - confidence: 0.9
