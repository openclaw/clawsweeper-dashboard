## Summary

Keep open: the latest head addresses the earlier unconditional switch to imageModel, but it still treats a configured imageModel as vision-capable even when the catalog does not confirm image support.

## What This Changes

The PR adds Gateway image-attachment sniffing, per-turn imageModel modelOverride/fallback plumbing, and auto-reply/Gateway tests for image vs non-image attachments.

## Best Possible Solution

Keep the per-turn switch only when the selected model lacks image support and the chosen imageModel primary or fallback resolves to an allowlisted, catalog-confirmed vision model; otherwise route image attachments through the existing media-understanding path.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source inspection: configure agents.defaults.imageModel.primary to a catalog-known text-only model, send an actual image through chat.send, and the PR path can still return true from imageModel config after catalog support is not confirmed.

## Solution Assessment

Is this the best way to solve the issue? No. The latest head fixed the unconditional model switch, but the safer solution is to require a resolved catalog-backed vision model before Gateway keeps images inline for the override.
