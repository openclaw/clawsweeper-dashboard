## Summary

Keep this PR open: it addresses a real source-reproducible bug in Dashboard/Gateway image routing and has useful live proof, but the patch still has a blocking fallback-plumbing bug before it should merge.

## What This Changes

The PR adds Gateway image attachment sniffing and threads a temporary image-model override through auto-reply so Dashboard image turns can use `agents.defaults.imageModel` when the session model lacks vision.

## Best Possible Solution

Land the Gateway-owned temporary image-model override after wiring imageModel fallback retries into the agent run fallback policy and getting the required checks green.

## Reproduction Assessment

Yes, by source inspection: current main has a clear path where Dashboard attachments are parsed against the session model and dispatched without an image-model override, despite the documented imageModel contract. I did not run a live failing pre-fix turn in this read-only review.

## Solution Assessment

Mostly yes: putting the switch in Gateway before attachment parsing is the right ownership boundary for Dashboard `chat.send`. It is not complete until `agents.defaults.imageModel.fallbacks` are used for runtime retry/failover when the temporary image override is active.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Use image fallbacks after accepting the override:** `src/auto-reply/reply/get-reply.ts:335-337`
  - body: The PR computes `modelOverrideFallbacks`, but after an allowed primary override is selected here the list is never carried into `followupRun.run` or `resolveModelFallbackOptions`. If the selected vision model fails, the run still falls back through the regular model fallback policy, which can retry a text-only default fallback with the same images instead of the configured `agents.defaults.imageModel.fallbacks`.

## Repair Work Prompt

On the PR branch for https://github.com/openclaw/openclaw/pull/79835, fix the image model override fallback policy. When Gateway passes `modelOverrideFallbacks` for an image turn and `getReplyFromConfig` accepts the temporary image model override, the agent run fallback runtime must retry through the configured/canonicalized `agents.defaults.imageModel.fallbacks`, not the regular `agents.defaults.model.fallbacks`. Preserve the one-turn nature of the override, do not persist session model/auth changes, keep the MIME/content sniffing behavior in `src/gateway/server-methods/chat.ts`, and add focused regression coverage for a failing vision override falling back to an imageModel fallback instead of a text-only default fallback. Do not ask the contributor for a changelog entry; maintainer landing can handle release notes if this user-facing fix lands.
