## Summary

Keep open: the PR addresses a real current-main imageModel routing gap and has convincing live proof, but the patch has a blocking allowlist/offload regression that should be fixed before merge.

## What This Changes

The branch adds Gateway content-sniffed image attachment detection, threads image-model overrides and fallbacks through auto-reply, and adds focused tests for Dashboard imageModel override routing.

## Best Possible Solution

Resolve the allowed image override or allowed fallback before Gateway keeps images inline; when no allowed override will apply, preserve the existing MediaPaths/media-understanding path for text-only sessions.

## Reproduction Assessment

Yes by source inspection: current main lacks a chat.send modelOverride path despite the documented imageModel contract. The PR’s remaining allowlist regression is also source-reproducible with a text-only primary, a vision imageModel excluded by agents.defaults.models, and an image attachment.

## Solution Assessment

No as-is. Moving the decision into Gateway is the right ownership boundary, but the image-capable parsing decision must be tied to the same allowlist-selected override that auto-reply will actually run.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep images offloaded when the override is rejected:** `src/gateway/server-methods/chat.ts:2323-2324`
  - body: This marks the turn image-capable based only on the configured imageModel’s vision support, but `getReplyFromConfig` later rejects `modelOverride` when it is outside `agents.defaults.models` and no allowed fallback exists. In that case Gateway has already kept the image inline instead of offloading it to `MediaPaths`, so a text-only session model can receive image payloads and the current media-understanding fallback path is bypassed. Gate this on an allowed override/fallback, or keep `routeImageOffloadsAsMediaPaths` true when no allowed override will be applied.
  - confidence: 0.88
