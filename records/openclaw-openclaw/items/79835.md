## Summary

Keep open: the PR addresses a real Dashboard/Gateway image-model routing bug and includes credible live proof, but the latest head still has two P2 correctness blockers around allowlist-gated inline images and image-model fallback retries.

## What This Changes

The PR adds Gateway image attachment sniffing, a temporary imageModel modelOverride path, auth-profile safeguards, and regression tests for image versus non-image Dashboard attachments.

## Best Possible Solution

Land a narrowed Gateway/reply fix that only inlines images after an allowlisted vision model or fallback is selected, and keeps image turns on the imageModel fallback chain during runtime retries.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main routes Dashboard attachments using the session model's image support and has no modelOverride path, while the docs already define imageModel as the fallback when the primary cannot accept images.

## Solution Assessment

No, not yet. The Gateway-owned temporary override is the right direction, but the patch must prove the override/fallback is allowlisted before inline parsing and must thread imageModel fallbacks through runtime retry resolution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Gate inline image parsing on an allowed override:** `src/gateway/server-methods/chat.ts:2351-2352`
  - body: Here `supportsImages` becomes true from the configured imageModel's vision capability before the reply layer has proved the same override is allowed. If `agents.defaults.models` blocks that imageModel and no allowed fallback is found, `getReplyFromConfig` falls back to the text model while `replyOptions.images` still contains inline images, bypassing the existing MediaPaths/media-understanding path. Resolve an allowed vision primary/fallback in Gateway before setting `supportsImages`, or keep `routeImageOffloadsAsMediaPaths` true when none is allowed.
  - confidence: 0.86
- **[P2] Carry image fallbacks into runtime retries:** `src/auto-reply/reply/get-reply.ts:976`
  - body: The PR prepares `modelOverrideFallbacks`, but only passes `hasAppliedImageModelOverride` into `runPreparedReply`, and `FollowupRun` still resolves retries from normal model fallbacks. If the selected vision model fails, an image turn can retry text fallbacks with inline images instead of the configured imageModel fallbacks. Add the prepared image fallback list to the run and fallback resolver path.
  - confidence: 0.87
