## Summary

Keep open: the updated proof is inspectable, but the patch still introduces routing regressions against the documented imageModel fallback contract.

## What This Changes

This PR adds Gateway image attachment sniffing, imageModel modelOverride/fallback plumbing into auto-reply, and tests for image versus non-image chat.send attachments.

## Best Possible Solution

Preserve native vision models first, then use only a resolved catalog-backed imageModel or fallback when the selected model cannot accept images; otherwise keep the current MediaPaths/media-understanding route.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source inspection: send chat.send with an image while the active session model already supports images, or configure an unresolved/non-vision imageModel and observe the PR path keeping images inline instead of using MediaPaths/media-understanding.

## Solution Assessment

Is this the best way to solve the issue? No. The safer fix is to reuse the current session-model capability check first, then apply imageModel only after resolving and validating a runnable vision-capable override.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve vision-capable session models:** `src/gateway/server-methods/chat.ts:2216-2219`
  - body: This assigns imageModel as the modelOverride whenever an image attachment is present, before checking whether the selected session/channel model already supports images. The documented contract is fallback routing only when the primary cannot accept images, so users on a vision-capable model would be silently switched for every image turn.
  - confidence: 0.84
- **[P2] Require catalog-backed vision support:** `src/auto-reply/reply/image-model-helpers.ts:300-307`
  - body: This helper returns true just because the provider/model appears in imageModel config before checking the catalog. An unresolved alias or configured text-only imageModel can therefore make Gateway keep images inline, while getReply later falls back to the default model instead of using the MediaPaths/media-understanding path.
  - confidence: 0.88
