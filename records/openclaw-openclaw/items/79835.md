## Summary

Keep open: this PR targets a plausible Gateway image-routing bug, but the current diff still introduces image model routing regressions and the contributor proof is test-only rather than a real Gateway/Dashboard run.

## What This Changes

The PR adds Gateway attachment MIME sniffing, imageModel override/fallback plumbing through auto-reply, and tests for image versus non-image chat.send attachments.

## Best Possible Solution

Keep imageModel as fallback routing for true image attachments: preserve the current session model when it supports vision, use only a resolved allowlisted imageModel/fallback otherwise, and fall back to MediaPaths/media-understanding when no runnable vision override exists.

## Reproduction Assessment

No high-confidence live reproduction is present. Source inspection of the PR head gives a high-confidence reproduction path for the introduced routing regressions: image attachments with a vision-capable session model, or with an unresolved imageModel override, exercise the bad branches.

## Solution Assessment

No. The narrow maintainable fix should preserve current selected vision-capable model routing, resolve and allowlist image overrides before treating them as vision-capable, and keep unresolved/text-only cases on the existing MediaPaths path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Preserve vision-capable session models:** `src/gateway/server-methods/chat.ts:2216-2219`
  - body: This assigns imageModel as modelOverride whenever image attachments are present, before checking whether the selected session/channel model already supports images. The documented contract is fallback routing only when the selected/default model cannot accept image input, so a user on a vision-capable model would be silently switched for every image turn.
  - confidence: 0.74

  - confidence: 0.87
