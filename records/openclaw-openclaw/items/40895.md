## Summary

Keep open. Current main still lowers Discord mention requirements for bot-owned autoThread threads and active bound thread sessions before the no-mention drop path can run, so the reported follow-up replies can still bypass `requireMention`. The related #52568 is open and useful, but it has not yet replaced this issue with an implemented policy.

## What This Changes

Discord thread follow-up messages should honor `requireMention` instead of staying implicitly active after an initial mention, bot-owned autoThread thread, or bound thread session.

## Best Possible Solution

Define an explicit Discord thread-reply mention policy, preferably coordinated with #52568 and aligned with Slack's `thread.requireExplicitMention` concept where appropriate, then update Discord preflight, autoThread ownership handling, bound-thread session behavior, config types, strict schema/help metadata, docs, and regression tests together.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: create a bot-owned autoThread thread or active bound Discord thread session with `requireMention: true`, send a follow-up without a mention, and current preflight resolves `shouldRequireMention` to false before the no-mention drop branch.

## Solution Assessment

Unclear. The strict per-message rule would stop the reported bypass, but current tests encode thread continuity for some workflows, so the narrow maintainable fix is an explicit documented Discord thread-reply mention policy rather than a silent behavior flip.
