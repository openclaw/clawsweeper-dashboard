## Summary

Keep open. Current main still stores and searches memory-lancedb data through one unpartitioned LanceDB table, while the fresh issue discussion and prior related reports describe a concrete cross-user/cross-agent privacy problem. The KinthAI comment and earlier mem0/user-pool context are useful implementation evidence, but no current OpenClaw source path applies a trusted partition key yet.

## What This Changes

Add multi-user or partition isolation to memory-lancedb so memory writes, searches, deletes, auto-recall, and auto-capture do not mix results across users or partitions.

## Best Possible Solution

Implement this in the bundled memory-lancedb plugin with a trusted partition key on stored rows, scoped store/search/list/delete/duplicate checks, coverage for auto-recall and auto-capture across two partitions, a migration/default policy for existing rows, and docs that define the first supported partition source and CLI behavior. Keep any core or SDK changes additive and generic.

## Reproduction Assessment

Yes. A high-confidence reproduction path exists by inspection: configure two users or agents against the same memory-lancedb dbPath, store memory from one, then recall/search from the other; current main has no row field or query filter that can prevent cross-scope results.

## Solution Assessment

No current solution is implemented. Metadata filtering or LanceDB namespace-style isolation looks like the narrowest maintainable direction, but it should derive scope from trusted gateway/plugin context rather than a prompt or tool argument.
