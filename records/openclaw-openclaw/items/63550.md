## Label Justifications

- `P1`: A shared memory-lancedb table can leak or contaminate memory across real multi-user deployments, which is an urgent broken memory workflow.
- `impact:security`: The reported failure crosses a privacy and sensitive-data boundary between user partitions.
- `impact:session-state`: The affected data is persisted memory/context state that can drift between users.

## Summary

Keep open. Current main still lacks structural partitioning for memory-lancedb rows and queries, and the related isolation PRs remain open implementation tracks rather than merged fixes for the cross-user memory boundary.

## What This Changes

The issue requests multi-user or partition-aware isolation for memory-lancedb storage and search so one user partition's memories cannot appear in another user's results.

## Best Possible Solution

Land one structural isolation design that scopes memory-lancedb writes, duplicate checks, search/recall, forget/delete, CLI/operator paths, auto-capture, auto-recall, docs, migration behavior, and two-user regression proof through a trusted partition contract.

## Reproduction Assessment

Yes, source-level. Configure two users or agents with the same memory-lancedb dbPath, store a memory from one scope, then recall/search from the other; current main has no stored partition field, query filter, or scoped DB selection to prevent cross-scope results.

## Solution Assessment

Unclear for the exact implementation because no fix is merged yet. The maintainable direction is a single structural isolation contract, with maintainer decisions on identity, defaults, and migration before code lands.
