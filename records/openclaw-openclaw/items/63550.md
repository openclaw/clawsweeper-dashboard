## Summary

Keep open. Current main still stores and searches memory-lancedb rows through one unpartitioned LanceDB table, and the open agent-scoping PR is a partial draft implementation track rather than a merged fix for the broader user/partition isolation problem.

## What This Changes

Request structural multi-user or partition isolation for memory-lancedb so memory writes and searches do not mix results across users or partitions.

## Best Possible Solution

Implement structural isolation in memory-lancedb using a trusted partition key across writes, vector searches, duplicate checks, forget/delete, auto-recall, auto-capture, CLI/operator paths, migration/default visibility, docs, and regression tests.

## Reproduction Assessment

Yes, source-level. Configure two users or agents against the same memory-lancedb dbPath, store memory from one, then recall/search from the other; current main has no row field or query filter that can prevent cross-scope results.

## Solution Assessment

No current solution is implemented. Metadata filtering, namespace-style isolation, or trusted per-partition storage are plausible directions, but maintainers need to choose the identity and migration semantics before a safe fix can land.
