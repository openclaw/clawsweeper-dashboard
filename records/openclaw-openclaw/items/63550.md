## Label Justifications

- `P1`: The issue describes cross-user memory recall leakage in multi-user deployments, which can expose private memory/session state in a core agent workflow.
- `impact:security`: The reported failure mode can leak sensitive remembered preferences, decisions, contacts, or other user data across partitions.
- `impact:session-state`: The affected surface is persistent memory/search state being stored and retrieved without user or partition isolation.

## Summary

Keep open: current main and the latest release still leave `memory-lancedb` on a single shared LanceDB table with no user or partition key, and the only open implementation attempt is still conflicting and proof-blocked. This is canonical security/product work, not conservative cleanup material.

## What This Changes

Request support for user or partition isolation in memory search so `memory-lancedb` storage and results do not mix memories across users.

## Best Possible Solution

Keep this issue as the canonical tracker and land a maintainer-approved isolation design with gateway-owned identity, explicit migration/default policy, two-user read/write/search tests, and docs for the chosen memory plugin path.

## Reproduction Assessment

Yes, at source level: current `memory-lancedb` writes rows without a user or partition key and searches the shared `memories` table without a metadata filter. I did not run a live two-user LanceDB setup in this read-only review.

## Solution Assessment

Unclear. The proposed options are plausible, but the best fix likely needs a gateway-owned partition identity plus explicit upgrade and migration policy rather than only `dbPath` templating or plugin-local filtering.
