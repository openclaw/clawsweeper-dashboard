## Label Justifications

- `P1`: The issue describes cross-user memory search leakage in multi-user deployments, which can expose private memory state in a core agent workflow.
- `impact:security`: The reported failure mode can leak sensitive remembered preferences, decisions, contacts, or other user data across partitions.
- `impact:session-state`: The affected surface is persistent memory/search state being stored and recalled without user or partition isolation.

## Summary

Keep open: current main and v2026.6.10 still leave memory-lancedb on a shared LanceDB store with no user, partition, or scope key, and the open implementation PR is still conflicting and proof-blocked. Related memory-scope reports overlap, but they do not safely replace this user/partition tracker.

## What This Changes

The issue requests multi-user or partition isolation for memory-lancedb storage and search so one user's memories cannot appear in another user's results.

## Best Possible Solution

Keep this as the user/partition isolation tracker and land a maintainer-approved memory-lancedb design with gateway-owned identity, scoped storage/query behavior, migration or upgrade policy, focused two-user tests, and redacted real setup proof.

## Reproduction Assessment

Yes, at source level: current memory-lancedb writes rows without a user or partition key and searches the shared memories table without a metadata filter. I did not run a live two-user LanceDB setup in this read-only review.

## Solution Assessment

Unclear. The proposed options are plausible, but the best fix likely needs a gateway-owned partition identity plus explicit storage, upgrade, and memory-management policy rather than only dbPath templating or plugin-local filtering.
