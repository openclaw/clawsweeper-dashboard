## Label Justifications

- `P1`: The issue describes cross-user memory recall leakage in multi-user deployments, which can expose private memory/session state and affects a core agent workflow.
- `impact:session-state`: The affected behavior is memory storage/search state being mixed across user partitions.
- `impact:security`: The reported failure mode can expose sensitive preferences, decisions, contacts, or other remembered data across users.

## Summary

Keep open: current main and the latest release still use a single shared `memory-lancedb` table with no user or partition key, while the related implementation PR remains open but not merge-ready. The issue is useful canonical security/product work, not conservative cleanup material.

## What This Changes

The issue requests partition-aware memory storage and search so `memory-lancedb` results cannot cross user partitions.

## Best Possible Solution

Keep this issue open as the canonical tracker, then land a maintainer-approved isolation design with explicit identity ownership, upgrade/migration policy, tests for two-user read/write/search isolation, and docs for the chosen memory plugin path.

## Reproduction Assessment

Yes, at source level: current `memory-lancedb` writes rows without a user or partition key and searches the single `memories` table without a metadata filter. I did not run a live two-user LanceDB setup in this read-only review.

## Solution Assessment

Unclear. The proposed options are plausible, but the best fix likely needs a gateway-owned partition identity and explicit migration/default policy rather than only dbPath templating or a plugin-local filter.
