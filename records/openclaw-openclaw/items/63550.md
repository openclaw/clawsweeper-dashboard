## Summary

Keep open. Current main and the latest release still leave memory-lancedb on one shared LanceDB table/dbPath with unpartitioned writes, recalls, forgets, CLI queries, auto-recall, and auto-capture; the related implementation track at https://github.com/openclaw/openclaw/pull/47277 is still open, conflicting, and not a merged fix. KinthAI's comments add useful production context, but the remaining work needs maintainer product and security review before code is queued.

## What This Changes

The issue requests user or partition isolation for memory-lancedb storage and search so memories from one user partition cannot appear in another user's results.

## Best Possible Solution

Land one maintainer-approved structural isolation design for memory-lancedb that scopes writes, duplicate checks, search/recall, forget/delete, CLI/operator paths, auto-capture, auto-recall, docs, migration behavior, and two-user regression proof through a trusted partition contract.

## Reproduction Assessment

Yes, at source level. Configure two users or agents with the same memory-lancedb dbPath and current main has one shared table, no partition field, and unscoped search/store/delete/hook paths.

## Solution Assessment

Unclear for the exact implementation because no fix is merged yet. The best direction is a single structural isolation contract with maintainer decisions on identity, compatibility defaults, and migration before code lands.
