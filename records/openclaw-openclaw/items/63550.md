## Summary

Keep open: current main and v2026.6.10 still leave memory-lancedb rows and recall on one shared LanceDB table with no user, sender, agent, or partition scope, and the live implementation candidate is still conflicting and proof-blocked.

## What This Changes

Request to add multi-user or partition isolation to memory-lancedb storage and search so one user's memories do not appear in another user's recall results.

## Best Possible Solution

Keep this issue as the canonical tracker until maintainers approve a memory isolation design with gateway-owned identity, scoped read/write/search coverage, explicit legacy-row policy, docs, and real two-user proof.

## Reproduction Assessment

Yes, at source level: current memory-lancedb writes rows without a user or partition key and searches the shared memories table without a metadata filter. I did not run a live two-user LanceDB setup in this read-only review.

## Solution Assessment

Unclear. The proposed options are plausible, but the best fix likely needs gateway-owned partition identity plus explicit upgrade and legacy-memory policy rather than only dbPath templating or plugin-local filtering.
