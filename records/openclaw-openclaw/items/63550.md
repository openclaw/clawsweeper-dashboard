- `impact:security`: The report is about sensitive remembered preferences, decisions, contacts, or other user data surfacing across partitions.

## Summary

Keep open: current main and v2026.6.10 still leave `memory-lancedb` storage and recall on one shared LanceDB table with no user, sender, agent, or partition key. The open implementation attempt is still conflicting and proof-blocked, so this remains a product/security design item rather than conservative cleanup.

## What This Changes

Add multi-user or partition-aware isolation to `memory-lancedb` so memory writes and searches do not mix results across configured users or partitions.

## Best Possible Solution

Keep this issue as the canonical tracker until maintainers approve a memory isolation design with gateway-owned identity, explicit legacy-row policy, scoped read/write/search coverage, docs, and real two-user proof.

## Reproduction Assessment

Yes, at source level: current `memory-lancedb` writes rows without a user or partition key and searches the shared `memories` table without a metadata filter. I did not run a live two-user LanceDB setup in this read-only review.

## Solution Assessment

Unclear. The proposed options are plausible, but the best fix likely needs gateway-owned partition identity plus explicit upgrade and legacy-memory policy rather than only `dbPath` templating or plugin-local filtering.
