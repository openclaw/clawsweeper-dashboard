## Summary

Keep open. Current main still stores, searches, recalls, and forgets memory-lancedb rows through one unpartitioned LanceDB table, while the related isolation PRs are open implementation tracks rather than merged fixes for the multi-user privacy boundary.

## What This Changes

Request support for trusted user or partition isolation in memory-lancedb so shared LanceDB stores do not mix memory writes and search results across users.

## Best Possible Solution

Implement structural memory isolation through one trusted partition contract across memory-lancedb writes, vector search, duplicate checks, forget/delete, auto-recall, auto-capture, CLI/operator paths, docs, migration behavior, and regression plus real two-user proof.

## Reproduction Assessment

Yes, source-level. Configure two users or agents against the same memory-lancedb dbPath, store from one scope, then recall/search from the other; current main has no row field, query filter, or scoped DB selection that can prevent cross-scope results.

## Solution Assessment

No current solution is implemented on main. The open PR directions are plausible, but maintainers still need to choose identity, migration/default visibility, backend parity, and docs before a safe privacy boundary can land.
