## Summary

Keep open. Current main still stores and searches memory-lancedb rows through one unpartitioned LanceDB table, and the issue discussion describes a concrete cross-user privacy problem that is not solved by current code or docs.

## What This Changes

Add multi-user or partition isolation to memory-lancedb so writes, searches, deletes, auto-recall, and auto-capture do not mix results across users.

## Best Possible Solution

Implement partitioning in the bundled memory-lancedb plugin using a trusted partition key, scoped queries/mutations, migration/default-read policy, regression tests for two partitions, and docs for supported scope semantics.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure two users or agents against the same memory-lancedb dbPath, store memory from one, and recall/search from the other; current main has no row field or query filter that can prevent cross-scope results.

## Solution Assessment

No current solution is implemented. Metadata filtering or namespace-style isolation is likely the narrowest maintainable direction, but the scope must come from trusted gateway/plugin context rather than prompt or tool arguments.
