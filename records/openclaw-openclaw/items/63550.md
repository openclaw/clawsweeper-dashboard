## Label Justifications

- `P1`: A shared memory-lancedb table can leak or contaminate persisted memories across real multi-user deployments, which is an urgent memory workflow and privacy boundary issue.
- `impact:security`: The report concerns cross-partition exposure of sensitive stored user memory and preferences.
- `impact:session-state`: The affected data is persisted memory/context state that can drift between users or partitions.

## Summary

Keep open. Current main still uses one shared memory-lancedb table with no partition key or scoped query path, and the two related isolation PRs are still open implementation tracks rather than merged fixes.

## What This Changes

Add multi-user or partition-aware isolation to memory-lancedb so stored memories and search results do not mix across user partitions sharing a database.

## Best Possible Solution

Land one structural isolation design that scopes memory-lancedb writes, duplicate checks, search/recall, forget/delete, CLI/operator paths, auto-capture, auto-recall, docs, migration behavior, and two-user proof through a trusted runtime partition contract.

## Reproduction Assessment

Yes, at source level. Configure two users or agents with the same memory-lancedb dbPath, store a memory from one scope, then recall/search from the other; current main has no stored partition field, query filter, or scoped DB selection to prevent cross-scope results.

## Solution Assessment

Unclear for the exact implementation because no fix has landed. The maintainable direction is a single structural isolation contract with maintainer decisions on identity, defaults, migration, and how it relates to the two open implementation PRs.
