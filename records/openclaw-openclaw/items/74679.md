## Summary

Keep open. Current main still has the reported routing hazard: Chat Completions resolves the target agent from `model`, but a raw `x-openclaw-session-key` is forwarded as the only execution selector, and downstream session routing treats non-`agent:<id>:` keys as belonging to the default agent. The docs also state that this header fully controls session routing, and related open PR #69060 intentionally preserves that override semantics, so this needs maintainer API-contract judgment rather than cleanup closure.

## What This Changes

The issue requests that `/v1/chat/completions` honor `model: "openclaw/<agentId>"` routing even when the request also sends `x-openclaw-session-key`.

## Best Possible Solution

Define the public precedence contract for full-routing session keys versus agent-scoped stable session hints, then make Chat Completions and OpenResponses consistent with regression coverage for raw headers, canonical `agent:<id>:` headers, model-selected agents, `x-openclaw-agent-id`, `user`, and previous-response/session reuse.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a `/v1/chat/completions` request with `model: "openclaw/beta"` plus raw `x-openclaw-session-key: my-session-key`; current main forwards the raw key without `agentId`, and execution resolves that raw key to the default agent.

## Solution Assessment

Unclear. The requested behavior may be the right operator experience, but it conflicts with the documented full-override meaning of `x-openclaw-session-key` and with related PR #69060's proposed preservation of that precedence; a safer fix needs an explicit maintainer contract decision first.

- The issue body mentions a `sessionKey` body field, but current Chat Completions code and docs use OpenAI `user` for stable body-driven sessions and do not read a `sessionKey` body field.
