## Summary

Keep open. Current main still lets raw `x-openclaw-session-key` bypass `model: "openclaw/<agentId>"` command routing, while the docs currently describe that header as a full session-routing override, so this needs an API precedence decision rather than conservative cleanup.

## What This Changes

The issue asks for `/v1/chat/completions` to honor `model: "openclaw/<agentId>"` routing when `x-openclaw-session-key` supplies the session key.

## Best Possible Solution

Define one OpenAI-compatible HTTP session precedence contract, then update Chat Completions, Responses, docs, and regression coverage together.

## Reproduction Assessment

Yes. Source inspection on current main is enough: send a raw `x-openclaw-session-key` with `model: "openclaw/<agentId>"`, and the raw key reaches ingress without an `agentId` override, so non-agent keys resolve to the default agent.

## Solution Assessment

Unclear. Making the model always win would match the report, but it conflicts with documented full-override semantics and overlaps #69060's agent-scoped stable-session header, so maintainers should decide the precedence contract first.
