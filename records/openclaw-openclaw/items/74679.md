## Summary

Keep open. Current main still has the source-level mismatch: the gateway resolves `model: "openclaw/<agentId>"` to an agent, but a raw `x-openclaw-session-key` is passed through unchanged and the command layer later derives the runtime agent from that raw key. The docs also expose a contract conflict, so this needs an API precedence decision rather than conservative cleanup.

## What This Changes

Make `/v1/chat/completions` honor the `model: "openclaw/<agentId>"` agent target even when the session key is supplied via `x-openclaw-session-key`.

## Best Possible Solution

Define one explicit OpenAI-compatible HTTP session precedence contract, then update Chat Completions, Responses, docs, and regression tests consistently; coordinate the decision with #69060 so the stable scoped-session story does not diverge.

## Reproduction Assessment

Yes, at source level. A request with `model: "openclaw/beta"` and raw `x-openclaw-session-key: my-session-key` resolves `agentId` as `beta`, passes `my-session-key` to the command layer, and then falls back to the default agent because the key is not agent-scoped.

## Solution Assessment

Unclear. Making the model always win would match the report, but it conflicts with the documented `x-openclaw-session-key` full-override contract and the direction of #69060, so the narrowest maintainable fix needs maintainer precedence approval first.
