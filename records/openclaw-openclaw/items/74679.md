## Summary

Keep open. Current main still lets a raw `x-openclaw-session-key` take precedence over model-derived agent scoping, while the docs and related PR context preserve that header as an explicit routing override, so this is an unresolved API precedence decision rather than conservative cleanup.

## What This Changes

The issue asks for `/v1/chat/completions` to honor `model: "openclaw/<agentId>"` agent routing even when `x-openclaw-session-key` is supplied.

## Best Possible Solution

Define one explicit OpenAI-compatible HTTP session precedence contract, then update Chat Completions, Responses, docs, and regression tests consistently.

## Reproduction Assessment

Yes. Source inspection on current main shows a request with `model: "openclaw/cathy"` and raw `x-openclaw-session-key: my-session-key` resolves the selected agent separately, then passes only the raw key into command execution, where the runtime agent is derived from that key and falls back to the default agent.

## Solution Assessment

Unclear. Making the model always win would address the report, but it would revise the documented full-override behavior and overlaps #69060, so the maintainable fix needs an explicit precedence decision first.
