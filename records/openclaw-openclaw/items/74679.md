## Summary

Keep open. Current main still has the reported routing hazard, and the existing docs plus related PR context make this an API precedence decision rather than a safe cleanup close.

## What This Changes

The issue asks for `/v1/chat/completions` to honor `model: "openclaw/<agentId>"` routing when `x-openclaw-session-key` is also supplied as a request header.

## Best Possible Solution

Define the public precedence contract for explicit full-routing session keys versus agent-scoped stable session hints, then make Chat Completions and Responses consistent with regression coverage for raw headers, `agent:<id>:` headers, `model` agent routing, `x-openclaw-agent-id`, and previous-response/session reuse.

## Reproduction Assessment

Yes. A source-level reproduction is a Chat Completions request with `model: "openclaw/beta"` plus raw `x-openclaw-session-key: my-session-key`; current main forwards that raw key, and downstream session resolution maps unscoped keys to the default `main` agent.

## Solution Assessment

Unclear. The reporter's expected behavior may be better for callers, but it conflicts with the current documented meaning of `x-openclaw-session-key` and with open PR #69060's stated preservation of that precedence, so a maintainer should choose the API contract first.
