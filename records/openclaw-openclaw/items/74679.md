## Summary

Keep open. Current main still has the reported routing hazard: the Gateway resolves the model-selected agent, but a raw `x-openclaw-session-key` remains a full session-routing override and the downstream agent command receives no explicit `agentId`, so non-`agent:<id>:` keys resolve to the default agent. The related open PR #69060 preserves that existing header precedence, so this needs maintainer API-contract direction before an automated fix.

## What This Changes

The issue asks for `/v1/chat/completions` to honor `model: "openclaw/<agentId>"` routing even when callers provide the session key through the `x-openclaw-session-key` header.

## Best Possible Solution

Define the public precedence contract for full-routing session keys versus agent-scoped stable session hints, then make Chat Completions and OpenResponses consistent with regression tests for raw headers, canonical `agent:<id>:` headers, model-selected agents, `x-openclaw-agent-id`, and previous-response/session reuse.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a Chat Completions request with `model: "openclaw/beta"` plus raw `x-openclaw-session-key: my-session-key`; current main forwards the raw key without an explicit agent id, and downstream execution resolves that raw key to the default agent.

## Solution Assessment

Unclear. The requested behavior may be better for callers, but it conflicts with the current documented meaning of `x-openclaw-session-key` and with #69060's stated preservation of that precedence, so the safest path is maintainer API-contract clarification before implementation.
