## Summary

Keep open. Current main still has the reported routing hazard, and the related open PR preserves the existing full-override meaning of `x-openclaw-session-key` rather than resolving this issue directly, so this needs maintainer API-contract judgment instead of cleanup closure.

## What This Changes

The issue requests that `/v1/chat/completions` honor the `model: "openclaw/<agentId>"` agent target even when a raw `x-openclaw-session-key` header is supplied.

## Best Possible Solution

Define one public precedence model for full-routing session keys versus agent-scoped stable session hints, then update Chat Completions and OpenResponses together with regression tests for raw headers, canonical `agent:<id>:` headers, model-selected agents, `x-openclaw-agent-id`, and previous-response/session reuse.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a Chat Completions request with `model: "openclaw/beta"` plus raw `x-openclaw-session-key: my-session-key`; current main forwards the raw key without an explicit `agentId`, and execution resolves that unscoped key to the default agent.

## Solution Assessment

Unclear. The requested behavior may be the right caller experience, but it conflicts with the current documented full-override contract and with #69060’s stated approach, so the safest next step is maintainer API-contract review before code changes.
