## Summary

Keep open. Current main still has the reported routing hazard: `/v1/chat/completions` resolves the model-selected agent, but a raw `x-openclaw-session-key` remains a full session-routing override and the downstream agent command does not receive an explicit `agentId`, so non-`agent:<id>:` keys resolve to the default agent. The open related PR #69060 also intentionally preserves that header precedence, so this needs maintainer API-contract judgment before a code change.

## What This Changes

The issue asks for `/v1/chat/completions` to honor `model: "openclaw/<agentId>"` routing even when the caller supplies `x-openclaw-session-key` as the session key carrier.

## Best Possible Solution

Define the public precedence contract for full-routing session keys versus agent-scoped stable session hints, then make Chat Completions and OpenResponses consistent with regression tests for raw headers, canonical `agent:<id>:` headers, model-selected agents, `x-openclaw-agent-id`, and previous-response/session reuse.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a Chat Completions request with `model: "openclaw/beta"` plus raw `x-openclaw-session-key: my-session-key`; current main forwards the raw key without an explicit agent id, and downstream execution resolves that raw key to the default agent.

## Solution Assessment

Unclear. The reported behavior may be the better operator experience, but it conflicts with the current documented meaning of `x-openclaw-session-key` and with #69060's stated preservation of that precedence, so a maintainer should settle the API contract before implementation.
