## Summary

Keep open. Current main still has the reported routing hazard, but the desired fix changes the public precedence contract for `x-openclaw-session-key`, which current docs and related PR #69060 both treat as a full session-routing override.

## What This Changes

The issue requests that `/v1/chat/completions` honor `model: "openclaw/<agentId>"` routing even when `x-openclaw-session-key` is supplied as the session key transport.

## Best Possible Solution

Define the public precedence contract for full-routing session keys versus agent-scoped stable session hints, then make Chat Completions and OpenResponses consistent with regression tests for raw headers, canonical `agent:<id>:` headers, model-selected agents, `x-openclaw-agent-id`, and previous-response/session reuse.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a Chat Completions request with `model: "openclaw/beta"` plus raw `x-openclaw-session-key: my-session-key`; current main forwards the raw key without an explicit agent id, so downstream execution resolves it to the default agent.

## Solution Assessment

Unclear. The reporter's requested behavior may be the better operator experience, but it conflicts with the current documented meaning of `x-openclaw-session-key` and with #69060's stated preservation of that precedence, so a maintainer should settle the API contract before implementation.
