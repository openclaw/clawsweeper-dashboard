## Summary

Keep open. Current main still has the raw `x-openclaw-session-key` versus `model: "openclaw/<agentId>"` routing mismatch, and related #69060 makes this an API precedence decision rather than conservative cleanup.

## What This Changes

The issue requests that `x-openclaw-session-key` on `/v1/chat/completions` not cause raw session keys to bypass `model: "openclaw/<agentId>"` agent routing.

## Best Possible Solution

Define and document one OpenAI-compatible HTTP session precedence contract, then update Chat Completions, OpenResponses, and regression tests together.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is `model: "openclaw/beta"` with raw `x-openclaw-session-key: my-session-key`; current main returns the raw key, passes no agent override, and downstream resolution maps that key to `main`.

## Solution Assessment

Unclear. Making the model-selected agent always win would match the reporter expectation, but it conflicts with the documented full-override behavior and #69060, so the safer path is an explicit maintainer API-contract decision first.
