## Summary

Keep open: the feature is plausible, but the patch has a blocking session-state bug in the restore path and the claimed Telegram proof is not actually attached or linked in the live PR body/comments.

## What This Changes

The PR adds a `/resume` slash command, bounded session history metadata, transcript restore logic, docs/help/registry entries, and focused tests for listing and switching previous sessions.

## Best Possible Solution

Land a minimal `/resume` only after it resets stale backend bindings and has inspectable real Telegram proof, while keeping broader named snapshot/load/fork semantics tracked by https://github.com/openclaw/openclaw/issues/13700.

## Reproduction Assessment

Yes for the blocking review finding: source inspection shows `/resume` spreads the existing stored entry while changing sessionId/sessionFile, and current backend reuse reads preserved cliSessionBindings/cliSessionIds/claudeCliSessionId. This is not a current-main bug reproduction because `/resume` is a new feature in the PR.

## Solution Assessment

No as currently implemented. A resume command is a reasonable direction, but it must reset or intentionally restore backend runtime metadata instead of preserving stale bindings from the pre-resume session.
