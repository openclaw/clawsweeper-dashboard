## Best Possible Solution

## Solution Assessment

No as submitted. Runtime fallback is the right Telegram plugin boundary for explicit richMessages users, but the implementation needs chunk-state safety, edit parity, a narrower classifier, and live Telegram proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Avoid replaying already sent rich chunks:** `extensions/telegram/src/send.ts:858`
  - body: The new catch wraps the whole rich chunk helper, but that helper records each successful rich chunk before moving to the next one. If a later chunk matches the unsupported classifier, the fallback resends the full regular chunk plan and can duplicate already-visible content; fall back only before any rich chunk succeeds or resume from the remaining chunks.
  - confidence: 0.9
- **[P2] Cover the durable rich edit path:** `extensions/telegram/src/send.ts:1632`
  - body: This fallback covers sends and draft edits, but editMessageTelegram still calls richRawApi.editMessageText directly when richMessages is enabled. Opt-in rich edits can still fail on the same unsupported-rich condition instead of retrying through the regular HTML edit path.
  - confidence: 0.87
- **[P2] Narrow the unsupported-rich matcher:** `extensions/telegram/src/network-errors.ts:276`
  - body: The bare UNSUPPORTED alternative can match unrelated Telegram API errors and turn them into legacy sends or edits. Keep the classifier scoped to known rich-message/client-unsupported responses so validation errors still surface instead of being hidden by fallback delivery.
  - confidence: 0.8
