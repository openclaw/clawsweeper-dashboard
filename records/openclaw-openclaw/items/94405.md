Overall correctness: patch is incorrect

Full review comments:

- **[P1] Avoid replaying the whole message after partial rich sends:** `extensions/telegram/src/send.ts:858`
  - body: The new catch wraps the entire rich chunk loop, but that loop records each successful rich chunk before continuing. If a later chunk matches the unsupported classifier, the fallback resends the full regular chunk plan and can duplicate already-visible content; only fall back before any rich chunk succeeds or resume from the remaining chunks.
  - confidence: 0.9
- **[P2] Cover the remaining rich edit path:** `extensions/telegram/src/send.ts:845-850`
  - body: This adds a fallback around the send path in `send.ts`, but `editMessageTelegram` still calls `richRawApi.editMessageText` directly when `richMessages` is enabled. Opt-in rich edits can still fail on the same unsupported rich-message condition instead of retrying through the regular HTML edit path.
  - confidence: 0.86
- **[P2] Narrow the unsupported-rich matcher:** `extensions/telegram/src/network-errors.ts:276`
  - body: The bare `UNSUPPORTED` alternative can match unrelated Telegram API errors and turn them into legacy sends or edits. Keep the classifier scoped to known rich-message/client-unsupported responses so validation errors still surface instead of being hidden by fallback delivery.
  - confidence: 0.8
