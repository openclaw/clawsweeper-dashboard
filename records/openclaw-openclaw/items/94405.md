## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Avoid replaying the whole message after partial rich sends:** `extensions/telegram/src/send.ts:858`
  - body: The new catch retries the full regular chunk plan after `sendTelegramRichTextChunks` throws, but that helper records each successful rich chunk before continuing. If a later rich chunk hits the unsupported classifier, fallback can duplicate already-visible content; only fall back before any rich chunk succeeds or resume from the unsent chunks.
  - confidence: 0.9
- **[P2] Cover the durable rich edit path:** `extensions/telegram/src/send.ts:845-850`
  - body: This fallback only wraps selected send and draft paths, while `editMessageTelegram` still calls `richRawApi.editMessageText` directly when `richMessages` is enabled. Opt-in rich edits can still fail on the same unsupported-rich condition instead of retrying through the regular HTML edit path.
  - confidence: 0.86
- **[P2] Narrow the unsupported-rich matcher:** `extensions/telegram/src/network-errors.ts:276`
  - body: The bare `UNSUPPORTED` alternative can match unrelated Telegram API errors and turn them into legacy sends or edits. Keep the classifier scoped to known rich-message/client-unsupported responses so validation errors still surface instead of being hidden by fallback delivery.
  - confidence: 0.8
