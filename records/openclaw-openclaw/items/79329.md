## Summary

Keep open: current main still skips isolated cron runs during primary local-provider preflight before the configured fallback chain can run. The related retry PR only addresses slow/cold local providers, and the broader routing issue was closed in favor of keeping this as the canonical fallback-preflight bug.

## What This Changes

The issue requests that isolated cron model preflight try configured fallback models before marking a scheduled run skipped when the local primary provider is unreachable.

## Best Possible Solution

Make cron preflight evaluate the effective fallback candidate chain, or defer skip until the normal fallback path is exhausted, while preserving strict `fallbacks: []` semantics and the local endpoint probe cache.

## Reproduction Assessment

Yes, at source level. An unavailable local primary makes `src/cron/isolated-agent/run.ts` return `skipped` before `src/cron/isolated-agent/run-executor.ts` constructs the model fallback runner.

## Solution Assessment

Yes. The narrow maintainable fix is to apply existing cron fallback policy to the preflight decision before skipping; the retry-only PR helps cold starts but does not solve configured cloud fallback bypass.
