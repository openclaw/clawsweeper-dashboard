## Summary

Keep open: current main still runs isolated cron local-provider preflight against only the resolved primary and returns `skipped` before the model fallback runner can use configured cloud fallbacks.

## What This Changes

The issue asks for isolated cron model preflight to honor configured model fallbacks when a local primary provider is unreachable.

## Best Possible Solution

Make cron preflight evaluate the effective model candidate chain and skip only when every preflight-relevant candidate is unavailable, while preserving `fallbacks: []`, local endpoint cache behavior, and request-storm protection.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: an isolated cron run with an unreachable local primary will return `skipped` in `prepareCronRunContext` before the fallback runner is constructed. I did not execute a live cron run in this read-only review.

## Solution Assessment

Yes, the issue's preferred direction is the narrow maintainable fix: centralize chain-aware preflight before entering the runner instead of disabling preflight or adding a separate request-time fallback path.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/79329. In isolated cron, local-provider model preflight currently probes only the resolved primary and returns `skipped` before configured fallbacks can run. Update the cron preflight path so the effective model candidate chain is considered: payload `fallbacks` must override configured fallbacks, `fallbacks: []` must remain strict, cloud/non-local candidates should make the preflight pass, and local fallback candidates should be probed with the existing endpoint cache before the run is skipped. Keep the fix scoped to cron isolated-agent preflight/fallback planning; do not broaden into post-invocation fallback issues such as https://github.com/openclaw/openclaw/issues/44353 or embedded-agent timeout behavior in https://github.com/openclaw/openclaw/issues/74985. Add regression coverage for local primary unavailable plus healthy cloud fallback, local primary unavailable plus strict empty fallback list, and any docs/changelog updates required for user-facing behavior.
