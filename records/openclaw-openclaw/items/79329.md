## Summary

Keep open: current main still preflights only the resolved primary model for isolated cron runs and returns `skipped` before the configured fallback chain can reach the model fallback runner. The closed, unmerged fix attempt at https://github.com/openclaw/openclaw/pull/79335 remains useful repair context, and the later production comment confirms the same failure mode for another provider setup.

## What This Changes

The issue reports that isolated cron model preflight checks only the configured primary model and skips the run when that local endpoint is unreachable instead of trying configured cloud fallbacks.

## Best Possible Solution

Make cron preflight evaluate the effective model candidate chain and skip only after every preflight-relevant candidate is unavailable, while preserving payload fallback precedence, explicit `fallbacks: []`, alias resolution, and local endpoint probe caching.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an isolated cron run with an unreachable local primary returns `skipped` in `prepareCronRunContext` before `createCronPromptExecutor` can pass fallbacks to `runWithModelFallback`.

## Solution Assessment

Yes. The narrow maintainable fix is chain-aware cron preflight before entering the runner, not disabling preflight or adding a new configuration knob.
