## Summary

Keep open. Current main still allows raw explicit external session keys and Gateway agent/session resolution to disagree; related fixes cover UUID/Open WebUI or Voice Call subsets, but the structured raw-key split remains unimplemented.

## What This Changes

Raw external session keys should keep resolving to the same OpenClaw session after Gateway sessions_send or nested agent activity touches the conversation.

## Best Possible Solution

Add a shared session-store alias rule so existing raw external rows either remain durable aliases or are resolved consistently by both inbound and Gateway paths before pruning.

## Reproduction Assessment

Yes, at source level: seed a row under a raw key like conversation:pair:user_a::user_b:space:space_123, resolve or write through Gateway with that key, and then send another inbound turn using the same explicit SessionKey. Current main can migrate/prune the raw row before the inbound lookup.

## Solution Assessment

Yes. The best fix is a generic raw-key alias/lookup contract with structured-key regression coverage; UUID-only or plugin-specific fixes leave this broader issue open.

## Repair Work Prompt

Fix the raw-vs-canonical session-key split reported in https://github.com/openclaw/openclaw/issues/63005. Implement a generic session-store alias/lookup rule so a raw external key with an existing active row, such as conversation:pair:user_a::user_b:space:space_123, cannot be pruned away by Gateway sessions.resolve, sessions_send/chat.send, nested agent activity, session patch/reset, or node-event writeback while future inbound traffic still resolves by that raw key. Do not make this a UUID-only, Voice Call-only, or service-specific workaround; coordinate with the narrower Open WebUI UUID PR at https://github.com/openclaw/openclaw/pull/55351 and preserve the current canonical agent-scoped behavior for normal channel keys where no raw alias exists. Add focused regression tests for structured raw external keys and include a user-facing CHANGELOG.md fix entry unless the final patch is test-only.
