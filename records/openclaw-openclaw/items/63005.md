## Summary

Keep open. Current main still canonicalizes arbitrary non-agent raw keys into the default agent namespace and can prune the original raw key, while inbound explicit session handling still looks up the raw key directly; the UUID and Voice Call related items cover narrower subsets, not this structured external-key case.

## What This Changes

The issue requests stable continuity for raw external session keys after Gateway, nested-agent, or `sessions_send` canonicalization touches the same logical conversation.

## Best Possible Solution

Add a shared session-store alias/lookup contract so existing raw external rows remain durable aliases or resolve consistently before Gateway pruning, without adding a new config knob or narrowing the fix to UUIDs only.

## Reproduction Assessment

Yes, at source level: seed a session row under `conversation:pair:user_a::user_b:space:space_123`, resolve or write through Gateway with that key, then send another inbound turn using the same explicit `SessionKey`. Current main can promote/prune the raw row before the inbound lookup, causing the next inbound turn to take the new-session path.

## Solution Assessment

Yes. A generic raw-key alias or consistent lookup rule in the shared session-store boundary is the narrowest maintainable fix; UUID-only or plugin-specific canonicalization leaves this broader structured-key bug open.
