## Summary

Keep open. Current main still has the raw-vs-canonical session-key mismatch: explicit inbound keys can stay raw while Gateway resolution and sessions_send paths resolve the same logical key to an agent-prefixed canonical key and can remove the raw store row.

## What This Changes

The issue requests preserving session continuity for stable external raw session keys after sessions_send or nested agent activity touches the same conversation.

## Best Possible Solution

Add a generic session-store alias contract so raw external keys and Gateway agent/session-tool resolution either share one durable key or preserve a raw alias that future inbound traffic can still resolve.

## Reproduction Assessment

Yes, at source level. Seed a store row under a raw structured key, resolve or send through Gateway with that key so it migrates to agent:<default>:<raw> and prunes the raw row, then send another inbound turn with the same explicit SessionKey; initSessionState will miss the raw row and start a new session.

## Solution Assessment

Yes. The narrow maintainable fix is a shared raw-key alias rule with regression coverage for structured external keys, not another UUID-only or service-specific workaround.

## Repair Work Prompt

Fix the raw external session-key continuity bug from https://github.com/openclaw/openclaw/issues/63005. Preserve stable explicit raw keys such as conversation:pair:user_a::user_b:space:space_123 across Gateway sessions.resolve, sessions.send/sessions_send, nested agent activity, and later inbound initSessionState. Implement a generic alias/canonicalization rule in the session-store/Gateway boundary: either raw external keys and canonical agent keys must resolve to the same durable store entry, or raw aliases that inbound paths depend on must not be pruned. Do not add a service-specific UUID/WebUI-only exception, do not add a config flag, and do not change unrelated channel/plugin routing policy. Add focused regression coverage for a structured non-agent raw key seeded in sessions.json, resolved/sent through Gateway, then reused by inbound session initialization without creating a new sessionId. Add a CHANGELOG.md fix entry because this is user-facing session continuity behavior.
