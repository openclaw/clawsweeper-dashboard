## Summary

Keep open. Current main still has the raw-vs-canonical session-key mismatch described here: inbound paths can continue using a raw external key while Gateway resolution and agent/session-send paths canonicalize the same key to `agent:<default>:...` and prune the raw row.

## What This Changes

The issue requests preserving one logical session for stable raw external session keys after `sessions_send` or nested agent activity touches the same conversation.

## Best Possible Solution

Define a generic session-store alias rule so raw external keys and Gateway agent/session-tool resolution either share one durable key or preserve a raw alias that future inbound traffic can still resolve.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to seed a fresh store row under a raw key like `conversation:pair:user_a::user_b:space:space_123`, resolve or run through Gateway with that key, then send another inbound turn using the same explicit `SessionKey`; current code can migrate/prune the raw row before the inbound lookup.

## Solution Assessment

Yes. The narrow maintainable fix is a shared raw-key alias contract with regression coverage for structured external keys, not another service-specific UUID or WebUI exception.
