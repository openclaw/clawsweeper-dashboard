## Summary

Keep open. Current main still has the raw-vs-canonical session-key mismatch: explicit inbound keys can remain raw while Gateway/session-tool paths canonicalize the same key to an agent-scoped key and then prune the raw candidate.

## What This Changes

Report asks OpenClaw to preserve continuity for stable raw external session keys when Gateway/session-tool paths canonicalize the same conversation.

## Best Possible Solution

Add a generic session-store alias/lookup contract so explicit raw external keys and agent-scoped Gateway keys resolve to the same durable entry, with regression coverage for structured raw keys and sessions_send/agent flows.

## Reproduction Assessment

Yes, source-level: seed a store row under a raw key like conversation:pair:user_a::user_b:space:space_123, let sessions.resolve or an agent/session-tool path migrate/prune it to the canonical agent key, then send another inbound turn with the same explicit raw SessionKey. The next inbound lookup can miss the existing entry and create a fresh sessionId.

## Solution Assessment

Yes. The narrow maintainable fix is a shared raw-key alias or canonical lookup rule with regression coverage, not another service-specific UUID/WebUI exception.
