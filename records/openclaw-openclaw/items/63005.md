## Summary

Keep open. Current main still has inconsistent raw-key handling: explicit inbound `SessionKey` traffic can remain under the raw external key while Gateway agent/session resolution canonicalizes that same key to an agent-scoped key and prunes the raw alias, allowing the next inbound turn to miss the existing session.

## What This Changes

The issue requests stable continuity for durable external raw session keys after `sessions_send` or nested agent activity touches the same conversation.

## Best Possible Solution

Define one generic durable external raw-key alias rule so inbound initialization and Gateway agent/session-tool resolution either use the same stored key or preserve a raw alias that future inbound traffic can resolve.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to seed a session store with a raw key such as `conversation:pair:user_a::user_b:space:space_123`, run Gateway agent or `sessions.resolve` resolution with that key, then inspect that the raw row can be promoted/pruned before the next inbound raw-key lookup.

## Solution Assessment

Yes. The narrow maintainable fix is a shared raw-key alias/store contract with regression coverage for structured external keys, not another service-specific UUID or UI workaround.
