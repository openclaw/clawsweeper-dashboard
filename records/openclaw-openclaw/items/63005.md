## Summary

Keep open. Current main still allows a raw external session key to be looked up and written by inbound paths while Gateway agent/session resolution can promote the same row to an `agent:<default>:...` key and prune the raw alias, so the next inbound turn can miss the prior session and create a new `sessionId`.

## What This Changes

Ensure stable external raw session keys keep resolving to the same session after `sessions_send` or nested agent activity touches the conversation.

## Best Possible Solution

Define one generic session-store alias contract for external raw keys so inbound initialization and Gateway agent/session-tool resolution either use the same durable key or preserve a raw alias that future inbound traffic can still resolve.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to seed a store row under a raw key like `conversation:pair:user_a::user_b:space:space_123`, invoke `sessions.resolve` or the Gateway `agent` path with that key, then send another inbound turn with the same explicit `SessionKey`; current code can canonicalize/prune the raw row before the raw inbound lookup.

## Solution Assessment

Yes. The narrow maintainable fix is a shared session-store key/alias rule with regression coverage for structured raw external keys, not another service-specific UUID/WebUI exception.
