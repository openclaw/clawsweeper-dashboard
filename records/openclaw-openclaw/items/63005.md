## Summary

Keep open. Current main still has the reported raw-versus-canonical session-key mismatch: inbound explicit SessionKey values can remain as raw external keys, while gateway/session-tool paths canonicalize the same bare key to an agent-prefixed key and may prune the raw row before the next inbound turn.

## What This Changes

The issue requests stable external raw session keys to keep using the same OpenClaw session after sessions_send or nested agent activity touches the conversation.

## Best Possible Solution

Add a generic session-store key contract so inbound explicit SessionKey handling and gateway/session-tool resolution agree for external raw keys, or preserve durable raw aliases when future inbound traffic is expected to keep using those keys. The fix should avoid service-specific exceptions and should cover a conversation:pair-style key through sessions.resolve or sessions_send without rotating sessionId or transcript.

## Reproduction Assessment

Yes. Seed or create a session under a raw key such as conversation:pair:user_a::user_b:space:space_123, resolve or send to it through the gateway/session-tool path, then send another inbound turn with the same raw SessionKey and compare sessionId/sessionFile reuse.

## Solution Assessment

Yes. The narrow maintainable fix is a shared raw-key continuity rule between inbound session initialization and gateway/session-tool resolution; the related closed #49202 UUID/WebUI case is narrower and does not cover arbitrary structured external keys.
