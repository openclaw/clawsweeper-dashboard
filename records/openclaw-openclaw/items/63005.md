## Summary

Keep open. Current main still has the raw-versus-canonical session-key mismatch described in the report: inbound explicit `SessionKey` values can remain raw while gateway/session-tool resolution canonicalizes the same bare key and can prune the raw row.

## What This Changes

The issue requests preserving stable raw external session keys across inbound turns after `sessions_send` or nested-agent activity touches the same conversation.

## Best Possible Solution

Define one generic contract for external raw session keys so inbound initialization and gateway/session-tool resolution either agree on the same durable key or preserve a raw alias that future inbound traffic can still use.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to create a store entry under `conversation:pair:user_a::user_b:space:space_123`, run `sessions.resolve` or `sessions_send` with that key, then send another inbound turn with the same explicit `SessionKey`; current code can migrate/prune the raw row before the next inbound raw lookup.

## Solution Assessment

Yes. The maintainable path is a narrow shared session-store key/alias fix with regression coverage for structured raw external keys, rather than another service-specific exception like the already closed WebUI UUID case.

## Repair Work Prompt

Fix the raw external session-key split described in https://github.com/openclaw/openclaw/issues/63005. Preserve continuity for arbitrary explicit inbound `SessionKey` values such as `conversation:pair:user_a::user_b:space:space_123` after `sessions.resolve`, `sessions_send`, or nested-agent activity touches the same logical conversation. Keep the fix generic: either make inbound explicit-key initialization and gateway/session-tool resolution agree on one durable key, or keep a durable raw alias when future inbound traffic is expected to continue using that raw key. Do not add service-specific Open WebUI-only handling, do not weaken existing `main`/`global` alias behavior, deleted-agent rejection, case-variant cleanup, or spawned-session visibility. Add regression coverage that seeds or creates a raw structured key, resolves/sends through the gateway/session-tool path, then verifies the next inbound raw-key turn reuses the same `sessionId` and transcript instead of creating a new one. Add a user-facing `CHANGELOG.md` fix entry; no dependency or release/version changes.
