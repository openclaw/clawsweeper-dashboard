## Summary

Keep this open: current main has relevant opt-in safeguards for tool-loop pressure and transcript repair, but I did not find a default protection that clearly covers the reported long sequence of varied exec calls ending in an ineffective or missing user reply.

## What This Changes

The issue asks OpenClaw to prevent long exec-heavy model loops from losing the final user-visible reply, especially when lossless-claw reports missing tool results and context pressure is high.

## Best Possible Solution

Define a core-owned behavior for long tool-heavy Pi runs that preserves tool results, compacts or aborts before context exhaustion, and returns a clear partial summary or failure message instead of an ineffective final reply.

## Reproduction Assessment

No, there is not yet a high-confidence current-main reproduction. The report has detailed operational symptoms, and source inspection shows relevant guards are opt-in and pattern-based, but the exact Feishu/lossless/qclaw run is not deterministic from the provided data.

## Solution Assessment

Unclear. Existing opt-in knobs may mitigate the context-pressure part, but the requested default exec-loop interruption and meaningful partial-summary behavior need a maintainer decision before a narrow implementation is safe.
