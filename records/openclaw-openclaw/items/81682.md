## Summary

Keep open: current main has partial safeguards for repeated tool calls and post-compaction loops, but I did not find a default, current-main behavior that clearly covers this Feishu/lossless-claw run with many exec calls, missing tool results, and an ineffective final reply.

## What This Changes

The issue requests reliable handling of long Feishu private-chat exec tool-call loops with lossless-claw so tool results are preserved and the user receives a meaningful response instead of silence or truncation.

## Best Possible Solution

Define a core-owned reliability behavior for tool-heavy Pi runs that preserves tool results, aborts or compacts before context exhaustion, and returns a clear partial result or failure message while respecting the existing opt-in loop-detection policy unless maintainers change it.

## Reproduction Assessment

No, there is not a high-confidence end-to-end current-main reproduction. Source inspection does show the relevant default gaps: rolling loop detection and mid-turn precheck are opt-in, post-compaction guarding is narrow, and channel sessions do not synthesize completion text for tool-only runs.

## Solution Assessment
