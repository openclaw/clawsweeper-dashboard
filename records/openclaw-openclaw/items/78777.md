## Summary

Keep open: current main still starts the plain Docker image without an image-level init wrapper, and this PR is the active narrow implementation for the zombie-process report at https://github.com/openclaw/openclaw/issues/77885. The latest diff addresses the nested-init concern with Tini subreaper mode, includes docs/changelog coverage, and I found no blocking review finding.

## What This Changes

The branch installs Debian `tini`, sets `ENTRYPOINT ["tini", "-s", "--"]`, and updates Docker/Fly docs plus the changelog for container process reaping and signal forwarding.

## Best Possible Solution

Land the image-level Tini wrapper after Docker and Compose validation, preserving the docs/changelog updates and letting the linked issue close when the PR merges.

## Reproduction Assessment

No high-confidence live current-main reproduction was run. Source inspection shows the plain image uses a direct Node `CMD` without image-level init, while the linked report supplies observed zombie accumulation in that topology.

## Solution Assessment

Yes: putting Tini at the Dockerfile ENTRYPOINT with `-s` fixes the container launch boundary without adding app-level process management, and it aligns with Tini and Compose behavior for nested init paths.
