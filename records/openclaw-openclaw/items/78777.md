## Summary

Keep open. This targets a real Docker runtime issue and the image-level init direction is sound, but this external PR still lacks after-fix real behavior proof and needs small release/docs polish before merge.

## What This Changes

Adds Debian `tini` to the runtime Docker image, sets it as the image entrypoint, and updates Docker/Fly install docs to describe the init wrapper.

## Best Possible Solution

Land a verified Docker image-level init wrapper with a changelog entry and aligned Docker/Fly/Compose docs once a real Docker smoke proves PID 1 and orphan reaping behavior.

## Reproduction Assessment

No live current-main reproduction was run here, but the source path is clear: the root image currently has no image-level init wrapper while the paired issue reports long-running zombie accumulation and repo-owned Compose/Podman paths already opt into init handling.

## Solution Assessment

Yes in direction: fixing the container launch boundary with Tini is narrower and more maintainable than adding process-reaping logic to the Node app. The PR still needs real Docker proof plus small release/docs polish before it is the best mergeable version.
