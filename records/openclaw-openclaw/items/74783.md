## Summary

Keep this PR open: it addresses a real `health.connection` gap, but current `main` already has the merged #76111 authenticated WebSocket ping loop, and this PR would add a second 5s protocol ping loop instead of reusing one owner.

## What This Changes

This PR adds optional per-client `health.connection` telemetry for authenticated Gateway WebSocket health responses, with RTT tracking from ping/pong control frames plus docs, changelog, and Gateway tests.

## Best Possible Solution

Expose per-client `health.connection` from the health RPC while keeping exactly one authenticated WebSocket ping/pong path responsible for keepalive and RTT collection.

## Reproduction Assessment

Yes. Current main still lacks `health.connection` fields and returns shared health snapshots directly, while it also now has a 25s authenticated socket ping interval from #76111; the PR adds the requested overlay but schedules another ping loop.

## Solution Assessment

No, not as currently written. The per-response overlay is the right boundary, but RTT tracking should be wired into the landed authenticated WebSocket ping owner or replace it with one shared owner.
