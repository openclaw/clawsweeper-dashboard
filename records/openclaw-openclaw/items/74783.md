## Summary

Keep open. The PR implements the requested telemetry shape and addresses the delayed-pong concern, but the current diff appears to break existing direct health-handler expectations by adding a synthetic `connection: false` block when no WebSocket connection state is present.

## What This Changes

The PR adds per-client Gateway health connection telemetry, WebSocket ping/pong RTT tracking in gateway maintenance, health response types/docs, a changelog entry, and focused gateway tests.

## Best Possible Solution

Land one Gateway-owned authenticated WebSocket ping path that feeds per-client health telemetry only when real connection state exists, while leaving the shared service health cache and non-connection callers unchanged.

## Reproduction Assessment

Yes. Current-main source inspection reproduces the feature gap, and the PR diff gives a high-confidence source-level failure path against existing `server-methods.test.ts` assertions when the handler is called without socket state.

## Solution Assessment

No, not yet. The per-response overlay is the right boundary, but it should be gated to real WebSocket connection health state or the affected direct-handler tests/callers need to model an authenticated WebSocket client explicitly.
