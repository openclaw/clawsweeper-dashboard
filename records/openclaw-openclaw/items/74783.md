## Summary

Keep this PR open: it adds a still-missing gateway health telemetry feature, but the current head can reject valid delayed pong samples and incorrectly report an active connection as stale.

## What This Changes

The PR adds per-client WebSocket `health.connection` telemetry, gateway ping/pong RTT tracking, exported health types, docs, tests, and a changelog entry.

## Best Possible Solution

Keep the per-response overlay design, but track pending or bounded recent ping payloads so delayed valid pongs still refresh connection health, then land one shared protocol-ping path.

## Reproduction Assessment

Yes for the review finding: send pings at 0ms and 5000ms, then receive the valid 0ms pong at 6000ms; the latest-timestamp equality rejects it. For the original feature request, reproduction is not applicable because current-main source inspection proves the fields are absent.

## Solution Assessment

No, not as currently written. The overlay boundary is maintainable, but pong matching should tolerate delayed valid samples within the stale window instead of accepting only the newest ping timestamp.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Accept delayed pong samples within the stale window:** `src/gateway/server/connection-health.ts:32-34`
