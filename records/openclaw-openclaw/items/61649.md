## Summary

Keep open. Current main supports Gmail/gog setup and watcher startup, but it still lacks the requested deterministic revoked-token recovery path, lifecycle diagnostics for gog auth failures, and Gmail read/send plus Drive validation closure signal.

## What This Changes

Request a sustainable recovery and validation flow for revoked or expired Google/gog OAuth used by Gmail and Drive automation on headless OpenClaw instances.

## Best Possible Solution

Add an OpenClaw-owned Gmail/gog operability path: documented revoked-token recovery, actionable watcher/run diagnostics, and an operator validation flow that proves Gmail read, Gmail send, and Drive access after reauth.

## Reproduction Assessment

No high-confidence live reproduction for the exact invalid_grant token state. Static inspection does reproduce the repository gap: current docs, skill text, watcher code, and tests do not provide the requested recovery runbook or validation closure signal.

## Solution Assessment

No, current main is not the best solution yet because gog OAuth failures remain mostly raw process output. The safer initial fix is narrow docs plus diagnostics/validation around existing gog commands, leaving token storage and upstream auth-flow semantics unchanged until maintainer review.
