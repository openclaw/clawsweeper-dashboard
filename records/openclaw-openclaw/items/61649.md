## Summary

Keep open. Current main still has Gmail/gog setup, watcher startup, and model-OAuth doctor coverage, but it does not provide the deterministic revoked-token recovery path, Gmail/Drive validation closure signal, or auth-state lifecycle UX requested here; the overlapping open #43557 covers an intermittent cron/keyring symptom rather than superseding this broader operability report.

## What This Changes

The issue requests a deterministic Google/Gmail/Drive OAuth recovery path with clearer noninteractive auth-state UX and post-reauth validation for production-like instances.

## Best Possible Solution

Keep this open and add an OpenClaw-owned Gmail/gog operability path: a documented revoked-token/headless recovery runbook, actionable auth-error diagnostics in the Gmail watcher/run path, and an operator-driven validation checklist or command flow that proves Gmail read, Gmail send, and Drive access after reauth without prematurely changing token storage.
