## Summary

Keep open. Current main and the latest release still expose Gmail/gog setup and watcher paths, but OpenClaw does not yet provide the requested revoked-token recovery runbook, auth-state lifecycle diagnostics, or Gmail/Drive validation closure signal; the related open autoheal PR is not sufficient as submitted and needs maintainer judgment.

## What This Changes

Request deterministic Google/Gmail/Drive OAuth recovery guidance, stronger gog auth-state diagnostics, loop protection, and post-recovery Gmail/Drive validation for production-like instances.

## Best Possible Solution

Use upstream gog auth doctor/remote/manual primitives in an OpenClaw-owned runbook and diagnostics, then add a safe validation path for Gmail/Drive recovery while leaving token exchange/storage fixes in gogcli unless a missing OpenClaw seam is proven.

## Reproduction Assessment

No high-confidence live reproduction for the exact revoked-token state. Source inspection does reproduce the repository gap: current docs, skill guidance, watcher diagnostics, and tests do not provide the requested recovery path or validation closure signal.

## Solution Assessment

Unclear as a full solution because the issue spans docs, diagnostics, validation UX, and upstream gog behavior. A focused docs/diagnostics pass around `gog auth doctor --check` and supported reauth commands looks like the narrowest maintainable first step.
