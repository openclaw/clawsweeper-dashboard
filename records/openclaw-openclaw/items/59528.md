Keep open: the reported OpenShell sandbox status bug still exists on current main, and the open linked fix PR is the right active path rather than a reason to close the issue before merge.

## What This Changes

The issue reports that documented OpenShell sandbox setup leaves OpenClaw showing OpenShell-backed sandbox runtimes as stopped even when the OpenShell sandbox is ready.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/87696, or replace it with an equivalent narrower sandbox-backend activation path, then close this issue after the fix reaches main.

## Reproduction Assessment

Yes, at source level: current main leaves sandbox commands on the default loadPlugins:"never" path, OpenShell only registers its manager during full plugin activation, and listSandboxContainers reports running:false when the manager is absent. I did not perform the original WSL/OpenShell live run.

## Solution Assessment

Mostly yes: the linked PR is the narrowest fix using the current CLI/plugin registry policy surface, while a future sandbox-backend-only activation scope would be a lower-blast-radius alternative if maintainers reject full plugin activation for sandbox commands.
