## Summary

Keep open. Current main still has the source-level failure path described in the issue: `openclaw sandbox list` and `openclaw sandbox recreate` can run without full plugin runtime activation, so an OpenShell-backed registry entry has no registered OpenShell manager and is rendered as stopped.

## What This Changes

The issue asks for the bundled OpenShell plugin to create and report OpenShell-backed sandboxes correctly instead of leaving them stopped or invisible to `openshell sandbox list`.

## Best Possible Solution

Add a generic sandbox-backend activation path for `openclaw sandbox list` and `openclaw sandbox recreate`, with regression coverage for an OpenShell-style backend entry and no OpenShell-specific policy hardcoded into core.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: an OpenShell registry entry listed through `openclaw sandbox list` can reach `listSandboxContainers` without the OpenShell manager registered, causing `running: false`.

## Solution Assessment

Yes for the proven stopped-status path. The narrow maintainable fix is to load or activate sandbox backend managers before sandbox CLI status/removal operations through the existing generic plugin/backend seam; any remaining first-turn create flake should be handled with separate logs after that change.

## Repair Work Prompt

Fix OpenShell sandbox CLI status/removal activation for https://github.com/openclaw/openclaw/issues/59528. Current main lets `openclaw sandbox list` and `openclaw sandbox recreate` use the default CLI policy (`loadPlugins: "never"`), so OpenShell registry entries can be listed without the OpenShell backend manager registered and appear stopped. Add a generic sandbox-backend activation path for sandbox lifecycle CLI operations, likely by updating the CLI command catalog/startup policy for `sandbox list` and `sandbox recreate` or an equivalent generic seam. Do not hardcode OpenShell-specific behavior in core beyond using normal plugin/backend registration. Preserve `sandbox explain` behavior unless it genuinely needs plugin runtime activation. Add focused regression coverage proving an OpenShell-style backend manager is available before list/recreate status/removal, plus policy/startup tests.
