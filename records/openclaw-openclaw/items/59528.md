Keep open. Current main still lets `openclaw sandbox list` inspect OpenShell registry entries without loading the full plugin registration that provides the OpenShell backend manager, so the stopped-status symptom remains source-reproducible.

## What This Changes

The issue reports that the OpenShell sandbox backend can create or see a remote sandbox while OpenClaw continues to show the sandbox runtime as stopped after enabling the plugin.

## Best Possible Solution

Add a generic sandbox CLI backend activation path for sandbox lifecycle/status commands before registry inspection, with regression coverage, without hardcoding OpenShell in core.

## Reproduction Assessment

Yes, source-level with high confidence rather than live WSL/OpenShell proof. `openclaw sandbox list` resolves through a command path with plugin loading disabled, OpenShell only registers its manager during full plugin registration, and managerless entries are displayed as stopped.

## Solution Assessment

Yes for the known stopped-status symptom. The maintainable fix is a generic sandbox CLI plugin/backend activation path, not an OpenShell-specific hardcode or a config migration.
