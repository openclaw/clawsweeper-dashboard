## Summary

Keep open. Current main still has the source-level OpenShell status path described in the prior ClawSweeper review: the sandbox CLI can inspect an OpenShell registry entry without loading the plugin that registers the OpenShell backend manager, so the entry is rendered as stopped instead of being described through OpenShell.

## What This Changes

The issue reports that an OpenShell-backed sandbox created by OpenClaw is shown as stopped by `openclaw sandbox list` while OpenShell may have a ready sandbox.

## Best Possible Solution

Add a generic sandbox-backend activation path for sandbox CLI lifecycle/status commands so plugin-provided backend managers are available before list and recreate inspect registry entries.

## Reproduction Assessment

Yes, as a high-confidence source-level reproduction rather than a live WSL/OpenShell run. On current main, `openclaw sandbox list` can run with plugin loading disabled, leaving the OpenShell backend manager unregistered and causing the registry entry to be shown as stopped.

## Solution Assessment

Yes for the known stopped-status symptom. The maintainable direction is generic sandbox CLI backend activation, not an OpenShell-specific hardcode in core.
