## Summary

Keep open. Current main and the latest stable tag still have the OpenShell sandbox CLI status path described in the earlier ClawSweeper review: `openclaw sandbox list` can inspect an OpenShell registry entry without loading the plugin that registers the OpenShell backend manager, so the entry is rendered as stopped.

## What This Changes

The issue reports that enabling the OpenShell sandbox plugin leaves OpenClaw-created OpenShell sandboxes shown as stopped or absent from `openclaw sandbox list` even when OpenShell later reports the sandbox ready.

## Best Possible Solution

Load or activate configured sandbox backend plugins for sandbox lifecycle/status CLI commands before registry inspection, then add focused regression coverage for OpenShell manager registration and the missing-manager stopped fallback.

## Reproduction Assessment

Yes, as a high-confidence source-level reproduction rather than a live WSL/OpenShell run. On current main, `openclaw sandbox list` can run with plugin loading disabled, leaving the OpenShell backend manager unregistered and causing registry entries for that backend to be displayed as stopped.

## Solution Assessment

Yes for the known stopped-status symptom. The maintainable fix is a generic sandbox backend activation path for sandbox CLI lifecycle/status commands, not an OpenShell-specific hardcode in core.
