## Summary

Keep open: current main and the latest release still leave sandbox CLI commands on the no-plugin startup path, while OpenShell only registers its sandbox backend manager during full plugin registration. The open linked pull request https://github.com/openclaw/openclaw/pull/87696 is the active candidate fix, so this issue should wait for that review/landing decision rather than be closed or queued again.

## What This Changes

The issue reports that OpenShell-backed sandbox entries stay stopped or invisible through OpenClaw sandbox commands after enabling the bundled OpenShell plugin.

## Best Possible Solution

Review and land or revise the linked fix so sandbox CLI commands can register OpenShell sandbox backend managers before reading OpenShell-backed registry entries.

## Reproduction Assessment

Yes at source level: sandbox CLI policy does not load plugins, OpenShell registers its manager only during full plugin registration, and missing managers make listSandboxContainers report stopped entries. I did not run a live WSL plus OpenShell daemon reproduction in this read-only review.

## Solution Assessment

Unclear as a final merge shape: the linked PR fixes the implicated path, but all-plugin activation for sandbox commands is broader than the failing backend-manager registration path. A backend-scoped activation path or explicit maintainer acceptance of broad activation would settle the issue.
