## Summary

Keep open. Current main still matches the reported failure path: the sandbox CLI does not opt into plugin loading, while OpenShell registers its sandbox backend manager only during full plugin registration, so `openclaw sandbox list` can render an OpenShell registry entry as stopped without querying OpenShell.

## What This Changes

The issue asks OpenClaw to make the bundled OpenShell sandbox backend create, list, and report sandbox runtime status correctly after enabling `agents.defaults.sandbox.backend = "openshell"`.

## Best Possible Solution

Add a generic sandbox-backend activation path for `openclaw sandbox list` and `openclaw sandbox recreate`, preferably through command-path plugin loading or a lightweight typed backend-manager registration seam. Keep core extension-agnostic, avoid hardcoding OpenShell, and add regression coverage with an OpenShell-style backend registry entry that proves the CLI reaches the registered manager instead of falling back to `running: false`.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: on current main, `openclaw sandbox list` resolves under `loadPlugins: "never"`, so an OpenShell registry entry can be listed without the OpenShell manager registered and is then reported as `running: false`; no live WSL/OpenShell run was performed.

## Solution Assessment

Yes. The narrow maintainable solution is to repair the sandbox CLI/backend lifecycle so enabled sandbox-backend plugins are loaded before status/removal operations; the related global-registry-scope PR is adjacent but does not solve the missing CLI plugin activation by itself.
