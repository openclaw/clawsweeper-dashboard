## Summary

Keep open. Current main still does not load plugins for `openclaw sandbox` commands, while OpenShell only registers its backend manager during full plugin registration; that leaves `openclaw sandbox list` able to read OpenShell registry entries but unable to ask the OpenShell manager for live status, so entries can still be shown as stopped.

## What This Changes

The issue reports that enabling the OpenShell sandbox backend leaves `openclaw sandbox list` showing the runtime as stopped or absent instead of reflecting the OpenShell sandbox state.

## Best Possible Solution

Add a generic sandbox CLI backend activation path for sandbox lifecycle/status commands before registry inspection, with regression coverage and without hardcoding OpenShell in core.

## Reproduction Assessment

Yes, source-level with high confidence. `openclaw sandbox` falls through to the default no-plugin CLI policy, OpenShell only registers its manager in full plugin registration, and managerless registry entries are displayed as stopped.

## Solution Assessment

Yes for the known stopped-status symptom. The maintainable fix is a generic sandbox CLI plugin/backend activation path, not an OpenShell-specific hardcode or config migration.
