Keep open: the reported OpenShell sandbox status bug is still present on current main and latest release, and the linked open PR is the active fix candidate rather than a reason to close the issue before merge.

## What This Changes

The issue reports that the documented OpenShell sandbox backend can be enabled and used, but `openclaw sandbox list` reports the OpenShell-backed runtime as stopped or missing instead of showing the live running state.

## Best Possible Solution

Review and either land https://github.com/openclaw/openclaw/pull/87696 after accepting the sandbox CLI activation policy, or replace it with a narrower sandbox-backend activation scope before closing this issue.

## Reproduction Assessment

Yes, at source level. Current main has no sandbox command catalog entry, so sandbox commands inherit `loadPlugins: "never"`; OpenShell only registers its manager during full plugin registration; `listSandboxContainers()` reports `running: false` when that manager is absent.

## Solution Assessment

Mostly yes. The linked PR is the narrowest fix using the current CLI/plugin registry policy surface, but the all-enabled-plugin activation scope is a maintainer availability tradeoff that could be replaced by a narrower sandbox-backend activation scope.
