## Summary

Keep open: current main and the latest release still leave `openclaw sandbox` on the no-plugin CLI startup path, so OpenShell can create/register a runtime in the gateway path while sandbox CLI status falls back to `running: false`; the open linked pull request is the active candidate and needs maintainer review of the plugin-activation scope.

## What This Changes

The issue reports that the documented OpenShell sandbox backend creates or owns sandboxes but `openclaw sandbox list` reports them as stopped and may not match `openshell sandbox list`.

## Best Possible Solution

Land or revise a maintainer-approved fix that registers the needed sandbox backend managers before sandbox CLI status/recreate reads registry entries, without unnecessarily making sandbox commands depend on unrelated plugin activation.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: sandbox CLI policy does not load plugins, OpenShell registers its backend manager only during full plugin registration, and `listSandboxContainers()` reports entries stopped when that manager is absent.

## Solution Assessment

Unclear as a final merge shape. Loading plugins for sandbox commands is a plausible fix, but the current candidate needs maintainer acceptance of broad plugin activation or a narrower sandbox-backend registration path.
