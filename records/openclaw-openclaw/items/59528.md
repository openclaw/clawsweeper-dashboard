## Summary

Keep open. Current main still has a source-level failure path where sandbox CLI status/removal commands can run without plugin runtime activation, leaving an OpenShell registry entry with no registered backend manager and rendering it as stopped.

## What This Changes

Fix the OpenShell-backed sandbox path so OpenClaw sandbox status and lifecycle commands agree with the OpenShell runtime after the bundled plugin is enabled.

## Best Possible Solution

Add a generic sandbox-backend activation path for `openclaw sandbox list` and `openclaw sandbox recreate`, with regression coverage for an OpenShell-style backend entry and no OpenShell-specific policy hardcoded into core.

Yes. A high-confidence source-level reproduction exists: `openclaw sandbox list` currently falls through to `loadPlugins: "never"`, so an OpenShell registry entry can be listed without the OpenShell manager registered and then reported as `running: false`.

## Solution Assessment

Yes for the known stopped-status path. The narrow maintainable fix is to load or activate sandbox backend managers before CLI status/removal operations through a generic plugin/backend seam; if first-turn creation still fails afterward, that needs separate logs.

## Repair Work Prompt

Fix the OpenShell sandbox CLI status/recreate activation bug for https://github.com/openclaw/openclaw/issues/59528. Current behavior: `openclaw sandbox list` and `openclaw sandbox recreate` resolve through the CLI default `loadPlugins: "never"` path because the command catalog has no sandbox policy, while OpenShell registers its sandbox backend manager only during full plugin registration. As a result, `listSandboxContainers()` can see an `openshell` registry entry with no registered manager and reports `running: false` without querying OpenShell. Implement a generic solution so sandbox lifecycle commands load or activate the relevant sandbox backend managers before listing/removing runtimes. Prefer command-path plugin loading for `sandbox list`/`sandbox recreate` or a narrow typed backend-manager activation seam; do not hardcode OpenShell IDs, config defaults, or recovery policy in core. Preserve existing behavior for `sandbox explain` unless it needs the same activation. Add regression coverage that proves an OpenShell-style backend registry entry reaches the registered manager instead of falling back to `running: false`, and add/adjust CLI startup policy tests for the sandbox commands. Add a single-line user-facing changelog fix entry with reporter credit if policy permits.
