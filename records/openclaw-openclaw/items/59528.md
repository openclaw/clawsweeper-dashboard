## Summary

Keep open. Current main and the latest shipped release still have the source-level path described in the prior ClawSweeper review: `openclaw sandbox list` can inspect an OpenShell registry entry without loading the plugin that registers the OpenShell backend manager, so OpenClaw reports the runtime as stopped instead of asking OpenShell for status.

## What This Changes

The issue reports that an enabled OpenShell sandbox backend creates or records a sandbox but `openclaw sandbox list` keeps showing it as stopped on WSL/Ubuntu with OpenClaw 2026.4.1 and OpenShell 0.0.20.

## Best Possible Solution

Add a generic sandbox-backend activation path for sandbox CLI lifecycle/status commands so plugin-provided backend managers are available before list/recreate inspects registry entries.

## Reproduction Assessment

Yes, as a high-confidence source-level reproduction rather than a live WSL/OpenShell run. On current main, `openclaw sandbox list` inherits `loadPlugins: "never"`, so an OpenShell registry entry can be listed without the OpenShell backend manager and rendered as stopped.

## Solution Assessment

Yes for the known stopped-status symptom. The narrow maintainable fix is a generic plugin/backend activation path for sandbox CLI operations, without hardcoding OpenShell-specific policy in core.
