## Summary

Keep open. Current main still has a generic CLI/plugin activation gap where `openclaw sandbox list` and `recreate` can inspect an OpenShell-backed registry entry before the OpenShell backend manager is registered, causing the CLI to report it as stopped.

## What This Changes

The issue reports that enabling the bundled OpenShell plugin leaves OpenClaw-created OpenShell sandboxes shown as stopped or missing from `openshell sandbox list` on OpenClaw 2026.4.1 with OpenShell 0.0.20.

## Best Possible Solution

Add a generic sandbox-backend activation path for sandbox CLI status and recreate operations, preserving the plugin boundary and avoiding OpenShell-specific policy in core.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: run the sandbox CLI against an OpenShell registry entry without a full plugin registry load, and `getSandboxBackendManager('openshell')` is null so the row is rendered as stopped.

## Solution Assessment

Yes for the known stopped-status path. The narrow maintainable fix is to activate sandbox backend managers before CLI status/removal operations through a generic plugin/backend seam; any remaining first-turn creation failure should be handled with separate logs.
