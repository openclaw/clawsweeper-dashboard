## Summary

Keep open: the PR addresses plausible xAI auth/video/attribution failures, but this review found a merge-blocking CORS/security defect and the external after-fix proof is still test-only.

## What This Changes

The PR changes xAI OAuth callback/token handling, OAuth-aware sidecar auth, video submit/poll behavior, provider attribution headers, and related tests.

## Best Possible Solution

Keep the focused xAI fixes, but enforce the callback CORS policy in the server path, add a denial regression test, and require redacted live xAI OAuth/video/proxy evidence before merge.

## Reproduction Assessment

Yes for the review finding: source inspection shows the patched callback server still emits broad CORS headers before the new allowlist. No high-confidence live reproduction is supplied for the xAI service bugs because the PR proof is tests/checks plus before evidence.

## Solution Assessment

No. The xAI direction is plausible, but the CORS implementation is not safe until the old broad helper is removed or made allowlist-aware and the PR includes real after-fix proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Enforce the callback CORS allowlist:** `src/plugin-sdk/provider-auth-runtime.ts:119-124`
  - body: This allowlist block does not actually restrict the callback server because `applyOAuthCallbackCorsHeaders()` already ran above it and echoes any HTTP(S) `Origin`, including `Access-Control-Allow-Private-Network`.
