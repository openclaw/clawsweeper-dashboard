## Summary

Keep this PR open: the xAI fix direction is useful, but the patch has blocking correctness/security issues and the external-PR proof gate is not met because the after-fix evidence is tests/checks plus assertions, not real runtime output.

## What This Changes

The PR changes xAI OAuth login, xAI OAuth-aware audio/video sidecar auth, xAI video submit/poll handling, xAI User-Agent attribution, and the SDK loopback OAuth callback helper.

## Best Possible Solution

Keep the useful xAI auth/video fixes, but preserve caller/upstream video defaults, make the OAuth CORS allowlist actually control emitted CORS headers, and require redacted after-fix live proof before merge.

## Reproduction Assessment

Yes at source level for several reported problems: current main’s xAI video poll parser is closed over a small status set, and TTS/realtime selection currently checks direct API-key paths only. I did not live-run xAI OAuth or paid video generation, so the real service behavior still needs contributor proof.

## Solution Assessment

No. The direction is plausible, but this implementation overrides optional xAI video defaults and leaves the new OAuth CORS allowlist ineffective because the old broad CORS helper still runs first.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve caller defaults for xAI video requests:** `extensions/xai/video-generation-provider.ts:249-256`
  - body: The PR now sends `duration`, `aspect_ratio`, and `resolution` for every generate/reference request even when the caller omitted them. xAI documents prompt-only text-to-video and says image-to-video defaults to the input image aspect ratio unless `aspect_ratio` is specified, so this silently changes output shape/cost and can stretch image-to-video inputs. Only send these fields when OpenClaw has an explicit caller value or a documented product default.
  - confidence: 0.88
- **[P1] Route OAuth CORS headers through the allowlist:** `src/plugin-sdk/provider-auth-runtime.ts:120-124`
  - body: The new allowlist is ineffective because `applyOAuthCallbackCorsHeaders(req, res)` still runs before `requestOrigin` is checked and already echoes any valid HTTP(S) `Origin`. For non-allowlisted origins, `requestOrigin` is undefined, but `Access-Control-Allow-Origin` has already been set, so the SDK comment and xAI security hardening are false. Make the helper accept the resolved origin or skip the broad CORS helper when an allowlist is configured, and cover the full server path in a test.
  - confidence: 0.95

## Security Review

Status: needs_attention

Summary: The patch adds a security-sensitive CORS allowlist but does not enforce it because the previous broad Origin echo still runs first.

Concerns:

- **[medium] OAuth callback allowlist does not constrain CORS:** `src/plugin-sdk/provider-auth-runtime.ts:120`
