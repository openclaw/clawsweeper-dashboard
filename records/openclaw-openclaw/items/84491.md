- `impact:message-loss`: The affected behavior is missing intermediate replies and, under Feishu block streaming, missing all visible replies.

## Summary

Keep open: the no-`blockStreaming` part matches documented final-only behavior, but the Feishu `blockStreaming: true` total-loss report is actionable because docs say completed blocks should flush before the final reply and current source still carries prior silent-drop guardrails around that path.

## What This Changes

The issue asks Feishu and QQ Bot external channels to deliver assistant text before tool calls and reports total reply loss when Feishu block streaming is enabled.

## Best Possible Solution

Keep default final-only delivery when block streaming is off, and fix the documented opt-in block-streaming path so Feishu block text and the final reply are delivered without duplicate finals or silent drops.

## Reproduction Assessment

No live Feishu/QQ Bot reproduction was run. Source and docs give a medium-confidence reproduction target: `channels.feishu.blockStreaming: true` is documented to flush completed blocks, while current tests and history show this path has previously caused silent drops.

## Solution Assessment

Yes for the actionable part: the maintainable fix is to repair the existing documented block-streaming delivery path and add regression coverage, not to make no-`blockStreaming` channels emit intermediate text by default.
