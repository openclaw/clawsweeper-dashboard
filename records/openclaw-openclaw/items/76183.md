## Summary

Keep this PR open, but do not merge as-is. The Active Memory fast-fail direction is useful and matches the replacement work from #75761, but the new terminal detection can misclassify non-empty memory_search results as empty when debug.hits is 0 for the memory backend and supplement/wiki results are still present.

## What This Changes

The branch adds Active Memory terminal memory_search polling, timeout-boilerplate filtering, focused regressions, and a changelog fix entry.

## Best Possible Solution

Keep the early timeout and unavailable/empty fast-fail behavior, but only classify memory_search as empty when the returned result set is actually empty or the tool explicitly reports unavailable.

## Reproduction Assessment

Yes. A focused regression can write a memory_search toolResult with details.results containing a supplement hit and debug.hits set to 0; the new watcher would currently resolve empty before the subagent can use the returned result.

## Solution Assessment

No. The overall approach is narrow and maintainable, but the zero-hit predicate needs to account for non-empty results instead of trusting debug.hits alone.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Do not drop non-empty search results:** `extensions/active-memory/index.ts:1601`
  - body: Memory Core builds `debug.hits` from raw memory hits before appending supplement/wiki results to `results`, so a `memory_search` response can have `debug.hits === 0` and still return usable results. This condition treats that as terminal empty, aborting the recall subagent before it can summarize those results; require the returned `results` array to be empty before fast-failing on zero hits.
  - confidence: 0.88
