## Summary

Keep this PR open: it is useful Active Memory timeout/empty-recall work, but the current diff still has a P2 correctness bug and the automation trail was explicitly paused for human review.

## What This Changes

The branch adds Active Memory transcript polling to fast-fail terminal zero-hit or unavailable memory search recall, filters timeout boilerplate, adds focused regressions, and records a changelog fix entry.

## Best Possible Solution

Keep the fast-fail design, but only classify memory_search as empty when the actual returned result set is empty or the tool explicitly reports unavailable, then re-review after maintainer-approved branch repair.

## Reproduction Assessment

Yes. A focused regression can write a `memory_search` tool result with `details.results` containing a supplement/wiki hit and `debug.hits` set to 0; the current PR predicate would resolve empty before the subagent can summarize that hit.

## Solution Assessment

No. The overall approach is narrow, but the zero-hit predicate should account for non-empty `details.results` instead of trusting `debug.hits` alone.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Do not drop non-empty search results:** `extensions/active-memory/index.ts:1601`
  - body: Memory Core records `debug.hits` from raw memory hits before appending supplement/wiki results to `details.results`, so a `memory_search` response can have `debug.hits === 0` while `results` is non-empty. This branch treats that as terminal empty and aborts recall before the subagent can summarize useful results; require the returned results array to be empty before fast-failing on zero hits.
