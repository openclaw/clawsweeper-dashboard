## Summary

Keep this PR open. The branch addresses a real Active Memory timeout/empty-search problem, but the new terminal-search watcher would incorrectly resolve recall as empty when Memory Core returns supplemental/wiki results with `debug.hits === 0`.

## What This Changes

The PR adds Active Memory transcript polling to fast-fail terminal empty or unavailable memory search recall paths, filters timeout boilerplate, adds focused regressions, and updates the changelog.

## Best Possible Solution

Keep the timeout and unavailable fast-fail design, but only classify zero-hit search as empty when the final returned `details.results` set is actually empty, with a regression for non-empty results plus `debug.hits: 0`.

## Reproduction Assessment

Yes. A focused regression can write a `memory_search` tool result with non-empty `details.results` and `debug.hits: 0`, then verify Active Memory waits for the subagent summary instead of resolving empty.

## Solution Assessment

No. The overall PR direction is narrow and maintainable, but the terminal zero-hit predicate needs to account for the final `results` array before it is safe to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Do not drop non-empty search results:** `extensions/active-memory/index.ts:1601`
  - body: Memory Core computes `debug.hits` from raw memory hits before appending supplemental/wiki hits into `results`, so this fast-fail can abort a recall that already found usable results. Require the returned `results` array to be empty before treating `debug.hits === 0` as terminal empty.
