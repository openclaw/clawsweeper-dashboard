## Label Justifications

## Summary

Keep open: the PR addresses a real gateway lifecycle bug, but it needs a small retry/fallback safety fix and non-mock real behavior proof before merge.

## What This Changes

The PR stores deferred gateway lifecycle errors as structured timer/event/options state and adds regression tests for error-to-tool cleanup and error-to-start retry ordering.

## Best Possible Solution

Keep the structured pending-error state, preserve every supported retry/fallback cancellation signal, and merge only with regression coverage plus real behavior proof for the stuck-running path.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: lifecycle error schedules deferred cleanup, then a later non-error event clears the timer on current main. I did not run the live Telegram plus llama.cpp timeout path in this read-only review.

## Solution Assessment

Not yet. The structured pending-error approach is the right narrow fix, but the patch should preserve fallback retry cancellation or prove that lifecycle fallback is no longer a real continuation signal.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Preserve fallback lifecycle events from finalizing early:** `src/gateway/server-chat.ts:814`
  - body: The current grace path is documented to protect fallback or retry reuse of the same runId, and the existing tests model a lifecycle `fallback` event after a lifecycle error. With this condition, `fallback` no longer clears the pending terminal error, so a fallback attempt that runs past the grace window can be finalized as the old error before it finishes. Please include `fallback` as a continuation signal or add proof/tests showing real fallback retries always emit lifecycle `start` first.
  - confidence: 0.78
