## Summary

Current main and the latest release provide a default-off rollout-based package auto-updater, but they still lack the requested wall-clock/cron scheduling, approval and notify-only modes, channel-targeted notices, and post-restart version/changelog/health reporting. The closed implementation PR is not a viable replacement, while the open Control UI item covers discoverability rather than the core engine workflow, so this issue remains the canonical product-design request.

## What This Changes

Add a first-class configurable auto-update workflow with scheduled checks, confirmation or notify-only modes, targeted notifications, and post-restart reporting.

## Best Possible Solution

Keep one core update engine, add a narrowly designed operator-approval and notification layer around the existing update execution path, persist only canonical restart/update state in SQLite, and leave Control UI discoverability to the separate UI tracker.

## Reproduction Assessment

Not applicable; this is a feature request rather than broken defined behavior. Current source, tests, and docs give a high-confidence comparison showing which requested capabilities are absent.

## Solution Assessment

Unclear as proposed: the user problem is valid, but the full update.auto block risks config sprawl and overlapping responsibilities. The maintainable direction is to retain scheduling and eligibility in the existing updater while designing approval and delivery through canonical update execution and typed channel-action surfaces.

## Risks / Open Questions

- Unattended installation of new code needs an explicit maintainer decision on whether approval is per update, per channel, or a standing opt-in, including recovery when confirmation cannot be delivered.
- Channel targeting and approval callbacks must preserve authorization boundaries and avoid sending upgrade controls or operational details to the wrong conversation.
- Post-restart status must use canonical SQLite-owned state and defined health semantics rather than adding workspace files, sidecars, or a second updater path.
- Adding all proposed fields directly under update.auto would expand an already large config surface; maintainers should first decide which behavior belongs in updater policy versus existing update.run and presentation/action seams.
