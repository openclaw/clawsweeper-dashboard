## Summary

Keep open. Current main and the shipped updating docs have a default-off package auto-updater, but it is still the narrower delay/jitter/beta-cadence implementation and does not cover scheduled timezone/cron checks, confirm or notify-only modes, channel-targeted notifications, or post-restart reporting.

## What This Changes

Request to make auto-updates a first-class `update.auto` workflow with configurable schedule/timezone, confirm/silent/notify-only modes, channel notifications, and post-update reporting.

## Best Possible Solution

Keep this as the canonical updater engine/workflow request: extend the existing package updater with schema/types/help/docs/tests for the requested fields, add timezone-aware scheduling, implement notify-only/confirm/silent flows through configured channel targets, persist restart update state, and report old/new version, changelog context, and health after restart while leaving the Control UI manager to the separate follow-up.

## Reproduction Assessment

Not applicable as a runtime bug. Source inspection gives high confidence that the requested feature is absent because current main only implements the narrower package auto-updater contract and docs/tests match that behavior.

## Solution Assessment

Yes for the product direction, but not as a single unsliced patch. The maintainable path is to extend the existing updater engine and config contract, not resurrect the closed PR wholesale or move this into the UI-only follow-up.
