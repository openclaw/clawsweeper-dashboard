## Summary

Keep open: current main ships a narrower default-off package auto-updater, but it still lacks the requested wall-clock/cron scheduling, confirm or notify-only modes, channel-targeted notifications, and rich post-restart reporting. The discussion also points the same way: Bortlesboat confirmed the existing primitives are usable, #12896 is closed unmerged, and #65011 is a separate UI/discoverability follow-up rather than a replacement for this engine/workflow request.

## What This Changes

The issue asks to replace DIY cron and BOOT.md update orchestration with a built-in `update.auto` workflow for scheduling, confirmation, notifications, and post-update reporting.

## Best Possible Solution

Keep this as the canonical updater engine/workflow request, extend the existing package updater plus restart-sentinel/update delivery surfaces after maintainers settle the approval, scheduling, and notification contract, and leave #65011 as the separate UI/discoverability follow-up.

## Reproduction Assessment

Not applicable as a feature request. There is a high-confidence source verification path: current schema, types, runtime, docs, and tests show the requested workflow fields and behaviors are still absent.

## Solution Assessment

Yes, keeping this open and building on the existing updater/restart-sentinel surfaces is the maintainable direction. Closing as implemented would hide the remaining scheduling, approval, notification, and reporting contract behind a narrower package auto-apply feature.
