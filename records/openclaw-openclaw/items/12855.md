## Summary

Keep open: current main and the latest release have a useful default-off package auto-updater, but the requested first-class workflow still needs maintainer product/security design before implementation.

## What This Changes

Request adds a built-in auto-update workflow with schedule/timezone controls, confirm/notify/silent modes, channel-targeted notifications, and post-restart reporting.

## Best Possible Solution

Keep this as the canonical updater engine/workflow tracker and design one supervised update workflow around `update.run` or the external-updater direction, with approval modes, safe delivery, durable restart reporting, docs, schema/help, and focused tests.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken existing behavior report. Source inspection gives high confidence that current main lacks the requested config fields, approval modes, scheduler, notification flow, and chat update command surface.

## Solution Assessment

Unclear as proposed: the current package updater is a useful foundation, but directly adding all requested config risks sprawling a compatibility-sensitive surface. A maintainer-approved design should keep scheduling/eligibility, approval, delivery, and restart reporting on one supervised update engine.
