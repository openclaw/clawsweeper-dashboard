## Summary

The requested workflow remains necessary: current main provides default-off package auto-apply, managed restart handoff, and health verification, but still lacks operator confirmation or notify-only modes, timezone-aware wall-clock scheduling, stable channel targeting, and user-facing post-restart version/health notification. The closed implementation PR did not merge, while the open Control UI and rollback issues cover adjacent UX and safety work rather than replacing this engine/workflow request.

## What This Changes

Add a first-class updater workflow with configurable scheduling, operator approval modes, notification routing, and post-restart update reporting.

## Best Possible Solution

Keep one native updater engine and extend it with a narrowly specified, authorized operator workflow that reuses the existing restart sentinel, health verification, approval primitives, and typed channel actions; keep Control UI discovery and backup/rollback work in their existing adjacent trackers.

## Reproduction Assessment

Not applicable: this is a feature request, and current source, tests, docs, and shipped-release inspection directly establish which workflow capabilities are present and absent.

## Solution Assessment

No, not exactly as the proposed all-in-one `update.auto` block; the best maintainable direction is a narrower authorized workflow layered on the existing updater and restart-health control plane, with stable typed notification targets.

## Risks / Open Questions

- A confirmation flow must bind approval to an authorized operator identity and resist replay or approval from an unrelated inbound conversation.
- Notification routing such as a generic “last” target can become stale or ambiguous and could send privileged update prompts or status to the wrong channel or user.
- The proposal expands a compatibility-sensitive config surface and crosses scheduling, privileged execution, persistent restart state, channel delivery, and health reporting boundaries.
- Silent updating and post-update reporting need explicit failure, recovery, and partial-success semantics so an update handoff is not reported as healthy before verification completes.
