## Summary

Keep open: current main and v2026.6.8 ship a narrower default-off package auto-updater, but the requested scheduled confirmation/notify workflow, channel-targeted post-restart reporting, and on-demand chat update surface remain unimplemented and still need maintainer product/security design.

Request a first-class auto-update workflow with configurable schedule, confirmation or notify-only modes, silent mode, notification target, and post-update version/health reporting.

## Best Possible Solution

Keep this as the canonical updater engine/workflow tracker until maintainers choose the approval policy, scheduling API, channel-targeting model, and restart-state storage, then implement it on top of the supervised update.run/update handoff path with schema, docs, and tests.

Unclear as an exact implementation: the capability is useful and belongs near the core updater, but the proposed update.auto expansion needs maintainer product/security design before code should land.
