## Summary

Keep open: current main and v2026.6.6 ship a default-off package auto-updater, but the requested scheduled approval/notify workflow, timezone-aware schedule, channel-targeted post-update notification, and old/new version health reporting are still missing. The linked implementation PR is closed unmerged, and the related update-safety/UI issues are adjacent rather than replacements.

## What This Changes

Add a first-class configurable auto-update workflow with schedule/timezone, confirm/notify-only/silent modes, post-update channel notifications, and old/new version health reporting.

## Best Possible Solution

Keep this as the canonical updater engine/workflow request and design one core path that composes existing update execution, scheduling, outbound delivery, restart continuation, and health checks while keeping silent apply explicitly opt-in.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/config request. The capability gap is high-confidence from current source, tests, docs, latest release inspection, and the closed-unmerged linked PR.

## Solution Assessment

No, the original all-in-one config proposal is not yet proven to be the best permanent design. Maintainers should first decide how update.run, scheduler/cron, outbound delivery, restart-state persistence, and health checks compose without creating a parallel updater.
