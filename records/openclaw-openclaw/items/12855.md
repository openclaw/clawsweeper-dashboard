## Summary

Keep open: current main has a real default-off package auto-updater, and Bortlesboat's live-host note/guide confirms it is usable today, but it still does not implement the requested scheduled confirm/notify workflow, channel-targeted notifications, or rich post-restart reporting.

## What This Changes

Request adding a first-class `update.auto` workflow with scheduled checks, confirm/notify-only/silent modes, channel-targeted notifications, and post-restart version, changelog, and health reporting.

## Best Possible Solution

Keep this as the canonical updater engine/workflow request and extend the existing package updater plus restart-sentinel/update delivery surfaces after maintainers settle the approval, scheduling, and notification contract; leave #65011 as the separate UI/discoverability follow-up.

## Reproduction Assessment

Not applicable as a feature request. There is a high-confidence source verification path: current schema, types, runtime, docs, and tests show the requested workflow fields and behaviors are still absent.

## Solution Assessment

No, closing as implemented would not be the best way to solve the issue because current main only ships a narrower package auto-apply policy. The maintainable path is to build on the existing updater and restart-sentinel/update delivery surfaces after the product and safety contract is settled.
