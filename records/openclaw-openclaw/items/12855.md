## Summary

The requested updater workflow remains necessary: current main and the latest release ship a default-off package auto-updater with managed restart handoff, but still lack confirmation or notify-only modes, wall-clock/cron scheduling with timezone selection, channel-targeted user notifications, and post-restart old/new version reporting. The closed implementation PR did not merge, while the open Control UI and safe-update issues cover adjacent UX and rollback work rather than replacing this engine/workflow request.

## What This Changes

This issue requests a first-class `update.auto` workflow with configurable schedule, confirmation/silent/notify-only modes, timezone handling, and post-update notifications.

## Best Possible Solution

Keep this issue open as the canonical updater engine/workflow request, extend the existing package updater with a maintainer-designed approval/scheduling/notification contract, and leave Control UI discovery plus backup/rollback policy in their adjacent trackers.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request, and current source, tests, docs, and release-tag inspection directly establish which workflow capabilities are present and absent.

## Solution Assessment

Unclear without maintainer direction: the requested direction is plausible, but the exact all-in-one `update.auto` shape may sprawl, so the best maintainable solution should layer explicit approval and typed delivery on the existing updater and restart-health control plane.
