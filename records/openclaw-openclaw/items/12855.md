## Summary

Keep open: current main has useful default-off package auto-update primitives, but it does not implement the requested first-class workflow for scheduled checks, confirm/notify-only modes, channel-targeted notifications, or rich post-restart reporting. The linked UI/discoverability issue is related but does not replace this engine/workflow request.

## What This Changes

The issue requests a built-in `update.auto` workflow with opt-in scheduling, timezone support, confirm/silent/notify-only modes, channel notification targeting, and post-update status reporting.

## Best Possible Solution

Keep this as the canonical updater engine/workflow request. Extend the existing package auto-updater with schema/types/help/docs/tests for the requested mode, schedule, timezone, notification, and restart-report fields; implement timezone-aware scheduling and approval/notify flows through the existing channel delivery surfaces; and keep #65011 focused on UI/onboarding/discoverability.

## Reproduction Assessment

Not applicable as a feature request, but there is a high-confidence current-main verification path: inspect the config schema/type, updater runtime, docs, and tests listed above. Those checks show the requested workflow fields and behaviors are still absent.

## Solution Assessment

No, closing as implemented would not be the best way to solve the issue because current main only ships a narrower package auto-apply policy. The maintainable path is to build on the existing updater and restart-sentinel/update.run surfaces rather than treat the UI follow-up or cron workaround as a replacement.
