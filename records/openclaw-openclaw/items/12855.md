## Summary

Keep open. Current main has a real default-off package auto-updater, but the requested first-class workflow still is not implemented: scheduled wall-clock or cron checks, timezone selection, confirm/notify-only modes, channel-targeted notifications, and post-restart changelog/health reporting remain missing. The discussion also points the overlapping operator-grant request back here as the canonical engine/workflow tracker, while https://github.com/openclaw/openclaw/issues/65011 is only the separate UI/discoverability follow-up.

## What This Changes

The issue requests built-in configurable auto-update scheduling with confirm, silent, and notify-only modes plus post-update notifications.

## Best Possible Solution

Keep this as the canonical updater engine/workflow tracker, extend `update.auto` with validated mode/schedule/timezone/notification fields, implement timezone-aware scheduling and routed confirm/notify flows, persist restart state, and report version/changelog/health after restart while leaving the Update Manager UI to the separate follow-up.

## Reproduction Assessment

Not applicable as a runtime bug. The product gap is high-confidence by source inspection: current main supports only the narrower default-off package auto-updater policy and lacks the requested scheduled confirmation/notification workflow.

## Solution Assessment

Yes, keeping this open is the best current handling. The maintainable path is to build on the existing updater with additive config, gateway, scheduling, and notification contracts rather than closing it behind the narrower shipped package auto-updater or the separate UI follow-up.
