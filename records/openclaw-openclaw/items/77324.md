## Summary

Keep open: current main still has separate live-delivery and durable-transcript paths, so the reported UI-visible-but-not-persisted assistant turn remains a plausible contract violation rather than implemented cleanup.

## What This Changes

The issue reports that WebChat/Control UI can render streamed assistant messages that are missing from the durable session transcript.

## Best Possible Solution

Add guarded WebChat agent-run transcript reconciliation: after a final live assistant event, verify the Pi-managed assistant turn exists and only then append an idempotent fallback or surface an unsaved warning.

## Reproduction Assessment

No, not with high confidence from an end-to-end current-main run. Source inspection does show a reproducible divergence shape: live final assistant text can be rendered from chat events while durable history depends on a separate Pi transcript write.

## Solution Assessment

Yes for keeping this open, but the issue's render-blocking suggestion is not the narrowest fix. The safer direction is post-run reconciliation or an idempotent fallback that preserves live streaming and avoids duplicate persisted assistant turns.
