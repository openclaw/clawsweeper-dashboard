## Summary

Keep open: the report describes a real contract violation, and current main still has separate live-delivery and durable-transcript paths where normal agent-run assistant text is displayed from live events while Pi is expected to persist the model-visible assistant turn separately.

## What This Changes

Persist or clearly flag assistant messages streamed in Control UI/WebChat so the durable session transcript includes the assistant turns the user saw.

## Best Possible Solution

Add a guarded transcript reconciliation path for WebChat agent-run finals: verify Pi persisted the assistant turn, append an idempotent SessionManager-backed fallback or surface an unsaved warning only when the durable transcript is still missing it, and cover the failure with regression tests.

## Reproduction Assessment

No, not with high confidence from an end-to-end current-main run. Source inspection does show the relevant divergence path: live assistant finals can be rendered from gateway chat events while durable history depends on a separate Pi transcript write.

## Solution Assessment

No, the issue's suggested UI render-blocking approach is not the safest primary fix because live streaming is intentionally separate from transcript persistence. The narrower maintainable fix is a post-run reconciliation/fallback that only writes when the Pi-managed assistant turn is still absent.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/77324. WebChat/Control UI can show streamed assistant output from live `chat` events while the durable session transcript lacks the corresponding assistant message, so the next turn loses context. Implement a guarded reconciliation/fallback for normal agent-run final outputs: after an agent-run final/done, verify the active session transcript contains the Pi-managed assistant turn matching the live final/run before accepting the run as durable; if it is still absent, append via the existing SessionManager-backed assistant transcript append helper with an idempotency key, or surface/log a clear unsaved warning if append is impossible. Do not blindly mirror every agent-run final, do not duplicate normal Pi `message_end` entries, and keep existing non-agent, media, and abort transcript behavior intact. Add regression coverage that simulates agent-run live final delivery with no Pi assistant write and proves `chat.history`/`sessions_history` can recover the assistant turn, plus coverage that normal Pi-persisted turns are not duplicated. Add a user-facing changelog fix entry.
