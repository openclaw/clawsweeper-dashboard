## Summary

This PR should stay open for human handling, but it is not merge-ready: the branch adds a new skill setup hook while leaking ambient process env to skill-authored scripts, omitting the ClawHub install/update paths named by the request, and still lacking real install/update proof.

## What This Changes

The PR adds SKILL.md `metadata.openclaw.setup` parsing and runs a declared bundle-relative setup script after the metadata install path succeeds.

## Best Possible Solution

Land one canonical setup-hook implementation only after it isolates env, composes with metadata and ClawHub install/update flows, preserves rollback semantics, documents the SKILL.md contract, and includes redacted real install/update proof.

## Reproduction Assessment

Yes for the PR defects by source inspection: the proposed runner inherits `process.env` through `runCommandWithTimeout`, and the hook is wired only into the metadata-installer path. For the linked feature request itself, current-main bug reproduction is not applicable because this adds a new capability.

## Solution Assessment

No. The direction may be viable, but this PR is not the best merge candidate until it enforces the declared env contract, covers the actual install/update flows, and settles the security/product contract.

Full review comments:

- **[P1] Pass only the declared setup env to scripts:** `src/agents/skills-install.ts:494`
  - body: `runCommandWithTimeout` merges the supplied `env` over `process.env`, so this still gives every `setup.script` access to unrelated gateway secrets. That contradicts the promised contract of only `SKILL_DIR`, `OPENCLAW_HOOK_KIND`, and allowlisted `requires.env` values.
  - confidence: 0.97

  - confidence: 0.9
