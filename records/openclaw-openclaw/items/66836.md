## Summary

Keep open. Current main still lacks a sandbox initialization timeout, so the PR targets a real gap, but this revision is not mergeable because it rolls back the sandbox browser security epoch, leaves backend startup cancellation optional and partly unconsumed, documents a remote-backend default the runtime does not apply, and needs rebase care around current sandbox browser security plumbing.

## What This Changes

The PR adds an `agents.defaults.sandbox.initTimeoutMs` config field, wraps sandbox context initialization in a timeout with AbortController plumbing through selected Docker/browser paths, adjusts an OpenShell e2e availability guard, and adds sandbox timeout tests.

## Best Possible Solution

Revise or replace the PR so sandbox initialization has one documented timeout contract, preserves the current browser security epoch and current-main browser security behavior, makes init cancellation a required backend startup contract, propagates the signal through Docker/SSH/OpenShell/prune/browser probes, keeps focused timeout regression tests, and separates unrelated OpenShell e2e guard changes unless they are necessary.

## Reproduction Assessment

Yes. Current main's missing feature is reproducible by static source search and inspection of `resolveSandboxContext`; the PR defects are also statically reproducible from the PR head lines above, so a live hung Docker daemon is not needed to confirm the blockers.

## Solution Assessment

No. The direction is appropriate, but this revision is not the best way to solve it until the security epoch rollback, backend cancellation contract, runtime/default mismatch, and abort propagation gaps are fixed on top of current main.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Restore the current sandbox browser security epoch:** `src/agents/sandbox/constants.ts:42`
  - body: This changes `SANDBOX_BROWSER_SECURITY_HASH_EPOCH` from current main's `2026-04-05-cdp-source-range` back to the older no-sandbox marker. Because `ensureSandboxBrowser` includes this epoch in the config hash and container labels, older browser containers can be treated as current and skip the recreate needed for the CDP source-range hardening era.
  - confidence: 0.94
- **[P2] Make backend startup consume the init abort signal:** `src/agents/sandbox/backend.types.ts:31`
  - body: Adding `abortSignal?: AbortSignal` to the backend factory params does not make startup cancellable: factories can ignore it, and the PR head's Docker factory still calls `ensureSandboxContainer` without passing the signal. In the hung-Docker case, inspect/create/start/setup work can continue after the new timeout rejects.
  - confidence: 0.91
- **[P2] Match the documented remote backend timeout default:** `src/agents/sandbox/context.ts:152-156`
  - body: The public config type documents a 300000 ms default for SSH/remote backends, but this code falls back to the single 60000 ms default whenever the setting is unset. Slow remote/OpenShell startup will therefore fail after 60s despite the new config surface saying it gets five minutes by default.
  - confidence: 0.88
- **[P2] Abort browser Docker probes during init cancellation:** `src/agents/sandbox/browser.ts:167`
  - body: The new `abortSignal` parameter is not applied to the initial browser container state probe, Docker network driver/gateway probes, or mapped-port reads. If Docker hangs at those points, the outer race rejects while the underlying Docker child can keep running, which is the zombie-work case this PR is meant to prevent.
  - confidence: 0.84
