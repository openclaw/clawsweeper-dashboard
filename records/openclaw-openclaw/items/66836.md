## Summary

Keep open: the timeout direction is useful and current main still lacks it, but this branch is not mergeable as-is because it conflicts with current main, drops or bypasses current sandbox browser security contracts, has incomplete cancellation propagation, and lacks real Docker behavior proof.

## What This Changes

The PR adds `agents.defaults.sandbox.initTimeoutMs`, wraps `resolveSandboxContext` in a timeout with `AbortController` cancellation, threads cancellation through sandbox Docker/browser paths, adjusts an OpenShell e2e guard, and adds sandbox timeout tests.

## Best Possible Solution

Rebase onto current main, preserve the current browser relay auth and image-contract checks, make timeout cancellation a real backend/prune/browser contract, document the config defaults, and provide redacted real Docker proof.

## Reproduction Assessment

Yes for source-level review: current main has no sandbox init timeout, and the PR-head merge blockers are visible from source inspection. No live hung Docker daemon reproduction or after-fix runtime proof was established.

## Solution Assessment

No. The timeout approach is useful, but this branch is not the best mergeable solution until it is rebased, preserves current browser security contracts, completes cancellation propagation, and shows real Docker behavior proof.

  - confidence: 0.94
- **[P1] Preserve CDP relay authentication:** `src/agents/sandbox/browser.ts:70`
  - body: The PR-head CDP probe fetches `/json/version` without the current Authorization header and the generated browser profile no longer carries an authenticated CDP URL. That is incompatible with the current relay-auth sandbox browser contract and would break or bypass the hardened relay path.
  - confidence: 0.92
- **[P2] Pass the init abort signal into Docker backend creation:** `src/agents/sandbox/docker-backend.ts:35-40`
  - body: `resolveSandboxContextInner` passes `abortSignal` to the backend factory, but `createDockerSandboxBackend` drops it when calling `ensureSandboxContainer`. A hung Docker inspect/create/start/setup can therefore continue after the new timeout rejects.
  - confidence: 0.93
- **[P2] Make browser Docker probes observe the init signal:** `src/agents/sandbox/browser.ts:179`
  - body: The browser path still starts with `dockerContainerState` and later `readDockerPort` calls that do not accept or receive the init signal. If Docker hangs at those points, the outer timeout can return while detached Docker child work remains.
  - confidence: 0.88
- **[P2] Make prune consume the init signal:** `src/agents/sandbox/context.ts:229`
  - body: Racing `maybePruneSandboxes(cfg)` against the signal only unblocks the caller; PR-head prune internals still take no signal and remove runtimes through managers without cancellation. Timeout during prune can still leave background Docker work running.
  - confidence: 0.86
- **[P2] Align timeout default docs with runtime:** `src/config/types.agents-shared.ts:50`
  - body: The public type comment promises a 300000 ms default for SSH/remote backends, but `resolveSandboxContext` falls back to a single 60000 ms default whenever `initTimeoutMs` is unset. Slow remote/OpenShell startup would fail earlier than the new config contract says.
  - confidence: 0.9
