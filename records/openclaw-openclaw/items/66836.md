## Summary

Keep open: current main still lacks a sandbox initialization timeout, but this PR is not mergeable as-is because it predates current sandbox browser CDP relay-auth/image-contract hardening, leaves cancellation gaps, and lacks required real Docker behavior proof.

## What This Changes

The PR adds `agents.defaults.sandbox.initTimeoutMs`, wraps sandbox context initialization in a timeout with `AbortSignal` plumbing through sandbox Docker/browser paths, adjusts an OpenShell e2e guard, and adds timeout tests.

## Best Possible Solution

Rebase on current main, preserve CDP relay auth and browser image contracts, make timeout/cancellation a real backend/browser/prune contract, document the config surface, and add redacted real Docker proof.

Yes for source-level review: current main has direct awaits in `resolveSandboxContext` without an init timeout, and the PR-head defects are visible from source. No live hung Docker daemon reproduction was established in this read-only pass.

## Solution Assessment

No. The timeout direction is useful, but this branch is not the best mergeable implementation until it is rebased, preserves current browser security contracts, completes cancellation propagation, and shows real behavior proof.

  - body: This branch sets `SANDBOX_BROWSER_SECURITY_HASH_EPOCH` back to `2026-02-28-no-sandbox-env`, while current main uses `2026-05-12-cdp-relay-auth`. Reusing the older epoch can treat pre-auth browser containers as current after rebase or merge.

  - body: The PR-head image check accepts any existing sandbox browser image, but current main rejects images without the current `org.openclaw.sandbox-browser.contract` label. Dropping that check would allow stale images without the relay-auth contract to run.
  - confidence: 0.93
- **[P1] Keep CDP relay authentication in browser startup:** `src/agents/sandbox/browser.ts:70`
  - body: Current main creates a CDP auth token, passes it to the container, builds an authenticated CDP URL, and sends the Authorization header while waiting for CDP. The PR-head browser path has no equivalent, so it is incompatible with the current relay-auth browser contract.
  - confidence: 0.9
- **[P2] Forward the init abort signal into Docker backend startup:** `src/agents/sandbox/docker-backend.ts:35-40`
  - body: `resolveSandboxContextInner` passes `abortSignal` to the backend factory, but the Docker backend ignores it when calling `ensureSandboxContainer`. A hung `docker inspect/create/start/setup` can therefore continue after the new timeout rejects.
  - confidence: 0.93
- **[P2] Make browser Docker probes cancellation-aware:** `src/agents/sandbox/browser.ts:179`
  - body: The new `abortSignal` parameter is not used for the initial browser `dockerContainerState`, network driver/gateway probes, or mapped-port reads. If Docker hangs at those points, the outer timeout can return while Docker child work keeps running.
  - confidence: 0.88
- **[P2] Make prune consume the init signal:** `src/agents/sandbox/context.ts:229`
  - body: Racing `maybePruneSandboxes(cfg)` against the signal only unblocks the caller; prune still accepts no signal and its backend removals call Docker without cancellation. Timeout during prune can still leave detached Docker work running.
  - confidence: 0.86
- **[P2] Align the documented backend timeout defaults with runtime:** `src/config/types.agents-shared.ts:50`
  - body: The public type comment promises a 300000 ms default for SSH/remote backends, but the runtime fallback always uses the single 60000 ms default when `initTimeoutMs` is unset. Slow remote/OpenShell startup would fail earlier than the new config contract says.
  - confidence: 0.9
