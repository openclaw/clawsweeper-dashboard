## Summary

Keep open. Current main still lacks a sandbox initialization timeout, so the central change is not obsolete, but this PR head is not mergeable as-is because it rolls back current sandbox browser security epoch behavior, leaves cancellation gaps in backend/browser/prune paths, has a documented/default timeout mismatch, and still lacks real hung-Docker behavior proof.

## What This Changes

The PR adds `agents.defaults.sandbox.initTimeoutMs`, wraps sandbox context initialization in a timeout, threads `AbortSignal` through parts of sandbox Docker/browser startup, relaxes an OpenShell e2e guard, and adds sandbox timeout tests.

## Best Possible Solution

Revise the PR on current main so sandbox initialization has one documented timeout contract, preserves current browser security epochs, propagates cancellation through backend/browser/prune work, and includes focused tests plus redacted real behavior proof.

## Reproduction Assessment

Yes for source-level review, not for live Docker behavior: current main has no init timeout surface, and the PR-head defects are visible in source. I did not run a hung Docker daemon test in this read-only review.

## Solution Assessment

No. The timeout direction is useful, but this revision is not the best fix until it preserves current security behavior, makes cancellation a real backend/browser/prune contract, resolves the default mismatch, and shows real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep the current browser security epoch:** `src/agents/sandbox/constants.ts:42`
  - body: This line sets `SANDBOX_BROWSER_SECURITY_HASH_EPOCH` back to `2026-02-28-no-sandbox-env`, while current main uses the newer `2026-05-12-cdp-relay-auth` epoch in browser config hashing. Older browser containers can be treated as current and skip the recreate needed for current CDP relay auth/source-range hardening.
  - confidence: 0.96
- **[P2] Require backend startup to consume the init signal:** `src/agents/sandbox/backend.types.ts:31`
  - body: Making `abortSignal` optional lets backend factories compile while ignoring the timeout signal, and this PR's Docker backend still calls `ensureSandboxContainer` without forwarding it. In the hung-Docker case, backend inspect/create/start/setup work can continue after the new timeout rejects.
  - confidence: 0.93
- **[P2] Use backend-aware timeout defaults:** `src/agents/sandbox/context.ts:152-156`
  - body: The public config type documents a 300000 ms default for SSH/remote backends, but this fallback applies the single 60000 ms default whenever `initTimeoutMs` is unset. Slow remote/OpenShell startup will fail after 60s despite the new config surface saying it has five minutes by default.
  - confidence: 0.9
- **[P2] Pass abort signals to browser Docker probes:** `src/agents/sandbox/browser.ts:167`
  - body: The new `abortSignal` parameter is not applied to the initial browser container state probe, network driver/gateway probes, or mapped-port reads. If Docker hangs at those points, the outer timeout rejects while the underlying Docker child can keep running, which is the zombie-work case this PR is meant to prevent.
  - confidence: 0.88
- **[P2] Make prune itself cancellation-aware:** `src/agents/sandbox/context.ts:229`
  - body: Racing `maybePruneSandboxes(cfg)` against the signal only unblocks `resolveSandboxContext`; the prune implementation still accepts no signal and its backend removals can keep running in the background. That leaves the init timeout's cooperative-cancellation guarantee incomplete when pruning hits a stuck Docker daemon.
  - confidence: 0.85
