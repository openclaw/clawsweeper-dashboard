## Summary

Keep this PR open because current main still lacks a sandbox initialization timeout and cooperative cancellation path, so the proposed work is not obsolete. The current PR revision is not mergeable as-is: it rolls back the sandbox browser security hash epoch, leaves backend startup cancellation optional and incomplete, and documents a remote-backend timeout default that the runtime does not implement.

## What This Changes

The PR adds an `agents.defaults.sandbox.initTimeoutMs` config/schema field, wraps sandbox context initialization in a timeout with AbortController plumbing through selected Docker/browser calls, adds timeout regression tests, and changes an OpenShell e2e availability guard.

## Best Possible Solution

Revise this PR or a replacement so sandbox initialization has a single documented timeout contract, preserves the current browser security epoch and current-main browser security plumbing, makes init cancellation a real backend startup contract, propagates the signal through Docker/SSH/OpenShell/prune/browser/network/port operations, keeps focused timeout regression tests, and separates unrelated OpenShell e2e guard changes unless they are required for the fix.

## Reproduction Assessment

Yes. For current main, static source inspection reproduces the missing feature: there is no `initTimeoutMs` config and `resolveSandboxContext` has no timeout wrapper. For PR review, the blocking defects are also statically reproducible from the PR head lines called out above; no live hung Docker daemon is required to confirm them.

## Solution Assessment

No. The direction is appropriate, but this revision is not the best way to solve the issue until it preserves the shipped sandbox browser security epoch, makes cancellation mandatory and consumed by backend startup, aligns documented defaults with runtime behavior, and rebases cleanly onto current sandbox browser code.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Restore the current sandbox browser security epoch:** `src/agents/sandbox/constants.ts:42`
  - body: This rolls `SANDBOX_BROWSER_SECURITY_HASH_EPOCH` back from the current CDP source-range marker to the older no-sandbox marker. Existing browser containers can then be treated as current and skip the recreate that the source-range hardening relies on, so keep the newest epoch value.
  - confidence: 0.94
- **[P2] Make backend startup consume the init abort signal:** `src/agents/sandbox/backend.types.ts:31`
  - body: Adding `abortSignal?: AbortSignal` to the backend factory params does not make backend startup cancellable: the signal is optional, and the Docker factory still calls `ensureSandboxContainer` without passing it. In the hung-Docker case this leaves inspect/create/start/setup work running after the new timeout fires.
  - confidence: 0.91
- **[P2] Match the documented remote-backend timeout default:** `src/agents/sandbox/context.ts:153-156`
  - body: The public config type documents a 300000 ms default for SSH/remote backends, but `resolveSandboxContext` always falls back to the single 60000 ms default. Slow remote/OpenShell startup will therefore fail after 60s even though the new config surface says it gets five minutes by default.
  - confidence: 0.88
- **[P2] Abort browser Docker probes during init cancellation:** `src/agents/sandbox/browser.ts:299-302`
  - body: The browser setup still derives CDP source ranges through Docker network probes and then reads mapped ports without any abort signal. If Docker hangs at these points, the outer race rejects but the underlying Docker child can keep running, which is the zombie-work case this PR is meant to prevent.
  - confidence: 0.84
