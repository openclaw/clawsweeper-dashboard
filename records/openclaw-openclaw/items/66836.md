## Summary

Keep open: the timeout direction is useful and current main still lacks it, but this PR is outdated against current sandbox browser security behavior, has incomplete cancellation plumbing, reports merge conflicts, and lacks real Docker proof.

## What This Changes

The PR adds a configurable sandbox initialization timeout, threads abort signals through parts of sandbox Docker/browser startup, relaxes one OpenShell e2e guard, and adds timeout tests.

## Best Possible Solution

Rebase on current main, preserve relay-auth and browser image contracts, propagate cancellation through backend/browser/prune operations, align the config contract, and add redacted real Docker proof.

## Reproduction Assessment

Yes for source-level review: current main directly awaits sandbox Docker/browser setup without an init timeout, and the PR-head blockers are visible from source. No live hung Docker daemon reproduction was established in this read-only pass.

## Solution Assessment

No. The timeout approach is useful, but this PR is not the best mergeable solution until it is rebased, preserves current browser security contracts, completes abort propagation, and shows real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Keep the current browser security epoch:** `src/agents/sandbox/constants.ts:42`
  - body: This branch sets `SANDBOX_BROWSER_SECURITY_HASH_EPOCH` back to `2026-02-28-no-sandbox-env`, while current main uses the relay-auth epoch. Reusing the older epoch can treat pre-auth browser containers as current after rebase or merge.
  - confidence: 0.96
- **[P1] Preserve the browser image contract check:** `src/agents/sandbox/browser.ts:124-130`
  - body: `ensureSandboxBrowserImage` now accepts any successful `docker image inspect`, but current main rejects images without the expected `org.openclaw.sandbox-browser.contract` label. That would allow stale browser images without the current relay-auth contract to run.
  - confidence: 0.93
- **[P1] Keep CDP relay authentication in browser startup:** `src/agents/sandbox/browser.ts:89-121`
  - body: The PR-head browser path no longer carries the CDP auth token into the resolved CDP URL or readiness fetch headers. Current relay-auth browser containers require that auth plumbing, so this would break or bypass the current browser contract.
  - confidence: 0.9
- **[P2] Require and forward backend abort signals:** `src/agents/sandbox/backend.types.ts:31`
  - body: `CreateSandboxBackendParams.abortSignal` is optional, and the Docker backend still calls `ensureSandboxContainer` without forwarding it. A hung backend startup can therefore continue after the init timeout rejects.
  - confidence: 0.92
- **[P2] Make browser Docker probes cancellation-aware:** `src/agents/sandbox/browser.ts:167`
  - body: Adding `abortSignal` to `ensureSandboxBrowser` is not enough because the initial container-state probe, network driver/gateway reads, and mapped-port reads still run without a signal. A timeout can unblock the caller while those Docker operations keep running.
  - confidence: 0.86
- **[P2] Make prune consume the init signal:** `src/agents/sandbox/context.ts:229`
  - body: Racing `maybePruneSandboxes(cfg)` against the signal only unblocks the caller; prune still accepts no signal and its backend removals call Docker without cancellation, leaving detached work in the hung-daemon case this PR targets.
  - confidence: 0.88
- **[P2] Align the documented timeout defaults with runtime:** `src/config/types.agents-shared.ts:48-51`
  - body: The public type comment promises a 300000 ms default for SSH/remote backends, but the runtime fallback uses a single 60000 ms default when `initTimeoutMs` is unset. Slow remote/OpenShell startup would fail earlier than the new config contract says.
  - confidence: 0.9
