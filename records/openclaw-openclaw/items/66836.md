## Summary

Keep open: the timeout direction is useful and is not implemented on current main, but this external PR is currently conflicting, lacks required real Docker proof, and still has security and cancellation blockers.

## What This Changes

The PR adds `agents.defaults.sandbox.initTimeoutMs`, wraps sandbox context initialization in a timeout, threads `AbortSignal` through some Docker/browser startup paths, relaxes an OpenShell e2e guard, and adds sandbox timeout tests.

## Best Possible Solution

Rework the PR on current main, keep CDP relay-auth/security epoch and image contract checks, make timeout/cancellation a real backend/browser/prune contract, document the config surface, and add redacted real Docker proof.

## Reproduction Assessment

Yes for source-level review: current main has direct awaits in `resolveSandboxContext` without an init timeout, and the branch's remaining defects are visible in PR-head source. No live hung Docker daemon reproduction was established in this read-only pass.

## Solution Assessment

No. The timeout guard is useful, but this revision is not the best fix until it is rebased, preserves current browser security contracts, propagates cancellation through backend/prune/browser probes, and shows real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Keep the current browser security epoch:** `src/agents/sandbox/constants.ts:42`
  - body: This branch sets `SANDBOX_BROWSER_SECURITY_HASH_EPOCH` back to `2026-02-28-no-sandbox-env`, while current main uses `2026-05-12-cdp-relay-auth` for the browser config hash/recreate trigger. Reusing the older epoch can let pre-auth/CDP-relay-hardening browser containers be treated as current after the branch is rebased or merged.
  - confidence: 0.96
- **[P1] Preserve the current browser image contract check:** `src/agents/sandbox/browser.ts:124-134`
  - body: The PR-head `ensureSandboxBrowserImage` only checks that the image exists, but current main validates the `org.openclaw.sandbox-browser.contract` label against `SANDBOX_BROWSER_IMAGE_CONTRACT_EPOCH`. Dropping that check would allow stale browser images without the current CDP relay-auth contract to run.
  - confidence: 0.92
- **[P2] Forward the init abort signal into Docker backend startup:** `src/agents/sandbox/docker-backend.ts:35-40`
  - body: `resolveSandboxContextInner` passes `abortSignal` to the backend factory, but the Docker backend ignores it when calling `ensureSandboxContainer`. A hung `docker inspect/create/start/setup` in backend startup can therefore continue after the new timeout rejects, leaving the zombie work this PR is meant to avoid.
  - confidence: 0.93
- **[P2] Make browser Docker probes cancellation-aware:** `src/agents/sandbox/browser.ts:179`
  - body: The new `abortSignal` parameter is not used for the initial browser `dockerContainerState`, network driver/gateway probes, or mapped-port reads. If Docker hangs at those points, the outer timeout returns while the child Docker process can keep running in the background.
  - confidence: 0.88
- **[P2] Make prune itself consume the init signal:** `src/agents/sandbox/context.ts:229`
  - body: Racing `maybePruneSandboxes(cfg)` against the signal only unblocks the caller; prune still accepts no signal and its backend removals call Docker without cancellation. Timeout during prune can still leave detached Docker work running.
  - confidence: 0.86
- **[P2] Use backend-aware timeout defaults:** `src/agents/sandbox/context.ts:152-156`
  - body: The public type comment says SSH/remote backends default to five minutes, but the runtime fallback here always uses the single 60 second default when `initTimeoutMs` is unset. Slow remote/OpenShell startup would fail earlier than the new config contract promises.
  - confidence: 0.9
