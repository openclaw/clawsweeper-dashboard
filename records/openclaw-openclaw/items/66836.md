## Summary

Keep open: current main still lacks a sandbox initialization timeout, but this PR revision is not mergeable because it rolls back the current sandbox browser security epoch, leaves important Docker/prune startup work outside the abort contract, has a documented/default timeout mismatch, and lacks after-fix real behavior proof from a real setup.

## What This Changes

The branch adds `agents.defaults.sandbox.initTimeoutMs`, wraps sandbox context initialization in a timeout with AbortController plumbing across sandbox Docker/browser paths, relaxes an OpenShell e2e guard, and adds sandbox timeout tests.

## Best Possible Solution

Revise or replace the PR on current main so sandbox initialization has one documented timeout contract, preserves the current browser security epoch, requires and propagates init cancellation through backend/Docker/prune/browser startup work, and includes focused regression tests plus redacted real behavior proof.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main has no sandbox init timeout, and the PR defects are visible from the PR-head source without needing to run a live hung Docker daemon. I did not execute a live Docker hang test because this review is read-only.

## Solution Assessment

No. The direction is appropriate for the reported hang, but this revision is not the best fix until it preserves current security behavior, makes cancellation a real backend/browser/prune contract, resolves the timeout default mismatch, and shows real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve the current browser security epoch:** `src/agents/sandbox/constants.ts:42`
  - body: This changes `SANDBOX_BROWSER_SECURITY_HASH_EPOCH` from current main's `2026-04-05-cdp-source-range` back to `2026-02-28-no-sandbox-env`. Because `ensureSandboxBrowser` includes that epoch in the browser config hash and labels, older browser containers can be treated as current and skip the recreate needed for the CDP source-range hardening era.
  - confidence: 0.95
- **[P2] Require backend startup to consume the init signal:** `src/agents/sandbox/backend.types.ts:31`
  - body: Making `abortSignal` optional lets backend factories compile while ignoring the timeout signal, and the PR head's Docker backend still calls `ensureSandboxContainer` without passing it. In the hung-Docker case, inspect/create/start/setup work can continue after the new timeout rejects.
  - confidence: 0.92
- **[P2] Use the documented remote backend timeout default:** `src/agents/sandbox/context.ts:152-156`
  - body: The public config type documents a 300000 ms default for SSH/remote backends, but this code falls back to the single 60000 ms default whenever `initTimeoutMs` is unset. Slow remote/OpenShell startup will fail after 60s despite the new config surface saying it gets five minutes by default.
  - confidence: 0.89
- **[P2] Thread the abort signal into browser Docker probes:** `src/agents/sandbox/browser.ts:167`
  - body: The new `abortSignal` parameter is not applied to the initial browser container state probe, network driver/gateway probes, or mapped-port reads. If Docker hangs at those points, the outer timeout rejects while the underlying Docker child can keep running, which is the zombie-work case this PR is meant to prevent.
  - confidence: 0.87
- **[P2] Make prune itself cancellation-aware:** `src/agents/sandbox/context.ts:229`
  - body: Racing `maybePruneSandboxes(cfg)` against the signal only unblocks the caller; the prune implementation still has no signal parameter and its backend removals can keep running in the background. That leaves the init timeout's cooperative-cancellation guarantee incomplete when pruning hits a stuck Docker daemon.
  - confidence: 0.84
