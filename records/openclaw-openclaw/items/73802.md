## Summary

Keep open. Current main still matches the reported multi-account Discord approval gap: native Discord delivery is account-scoped before approver or delivery target handling, so a request originating from another Discord account can be filtered out even though a configured Discord approver can resolve it manually. The newer fallback work reduces silent failures, but it does not deliver the native card/buttons requested here.

## What This Changes

The issue reports that Discord exec approvals create pending approval IDs and accept manual `/approve` commands, but do not deliver native Discord approval cards/buttons to the configured approval account in a multi-account setup.

## Best Possible Solution

Define the Discord multi-account native approval contract. If cross-account approval surfaces are supported, route cards only to explicitly configured approval accounts with clear authorization, disclosure, and regression coverage; otherwise document origin-account-only native delivery and keep the same-chat manual fallback reliable and explicit.

## Reproduction Assessment

Yes. A source-level reproduction is to create a Discord approval request with `turnSourceChannel: "discord"` and `turnSourceAccountId` set to one account while evaluating native delivery for the configured approval account; current main rejects the native path before approver-DM routing, while the `/approve` command path can still authorize the configured approver.

## Solution Assessment

Unclear. Relaxing the account gate is the narrow mechanical change, but it is not necessarily the best fix until maintainers decide whether native Discord approval cards may cross account boundaries and what command-disclosure rules apply.
