## Summary

Keep open. Current main still rejects Discord native approval delivery when the approval request is bound to a different Discord account than the configured native approval account, matching the reported multi-account setup where manual `/approve` works but no native card appears. Later fallback work improves the manual path, but does not implement cross-account native card delivery, which needs a maintainer security/product decision.

## What This Changes

The issue requests restoring native Discord exec approval cards/buttons for a multi-account Discord setup where approval creation and manual `/approve` succeed but no native approval card is delivered.

## Best Possible Solution

Define the Discord multi-account native approval contract first. If cross-account approval surfaces are supported, route cards only to explicitly configured approver/native accounts with authorization and disclosure tests; otherwise document origin-account-only delivery and keep the same-chat manual fallback clear and unsuppressed.

## Reproduction Assessment

Yes. Configure Discord native approvals only on one account and create an approval request whose `turnSourceAccountId` is another Discord account; current main predicts no native card for the configured account while manual `/approve` remains authorized for configured approvers.

## Solution Assessment

Unclear. Relaxing the account gate would be the narrow code change, but it is not clearly the best solution until maintainers decide whether cross-account native cards are allowed and how command text/control disclosure should work.
