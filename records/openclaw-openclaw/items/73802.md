## Summary

Keep open. Current main still matches the reported multi-account Discord approval failure: a native approval request bound to one Discord account is filtered out before the configured approval account can plan origin or DM delivery, while manual Discord `/approve` remains authorized for configured approvers.

## What This Changes

Discord should deliver native exec approval cards/buttons to the configured approval surface in a multi-account setup instead of requiring the operator to type `/approve` manually.

## Best Possible Solution

Define the Discord multi-account approval routing contract. If cross-account approval surfaces are supported, route native cards to the explicitly configured approval account with authorization, disclosure, and regression tests; otherwise document origin-account-only native delivery and make the same-chat manual fallback reliable and explicit for this configuration.

## Reproduction Assessment

Yes. At source level, configure native Discord approvals on one account and create an approval request with `turnSourceChannel: "discord"` and a different `turnSourceAccountId`; current main predicts no native card for the configured approval account while manual `/approve` can still authorize the approver.

## Solution Assessment

Unclear. The narrow mechanical change would be to relax the account gate for configured approver delivery, but that is not clearly the best solution until maintainers decide whether Discord native approval cards may cross account boundaries and what disclosure rules apply.
