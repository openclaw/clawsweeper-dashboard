## Summary

Keep open. Current main still rejects Discord native approval handling when the approval request is bound to a different Discord account than the configured native approver account, so manual `/approve` can work while native card/button delivery is skipped; the newer fallback-notice fix does not resolve that native routing gap.

## What This Changes

The issue requests restoration of Discord native exec approval cards/buttons when approvals are created and manual `/approve` succeeds in a multi-account Discord setup.

## Best Possible Solution

Define the multi-account Discord native approval contract, then either route cards only to explicitly configured approver accounts with regression coverage or document origin-account-only native delivery while preserving the manual fallback.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a Discord approval request with `turnSourceChannel: "discord"` and `turnSourceAccountId` set to a different account than the configured approver account; current main rejects native handling before delivery target resolution while `/approve` authorization can still accept the configured approver.

## Solution Assessment

Unclear. Removing the account gate is the narrow mechanical fix, but the best maintainable solution needs maintainer policy on cross-account command disclosure and approval authorization before changing delivery behavior.
