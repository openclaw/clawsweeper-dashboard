## Summary

Keep open. Current main still matches the reported Discord multi-account native approval gap: native delivery is account-scoped before origin or approver-DM routing, so a request from one Discord account can be skipped even though a configured approver can resolve it manually with `/approve`. The newer fallback work reduces silent failures, but it still falls back to a typed command rather than delivering the requested native card/buttons.

## What This Changes

Restore Discord native exec approval card/button delivery for a configured Discord approver surface when manual `/approve` works but no native card appears.

## Best Possible Solution

Define the Discord multi-account native approval contract. If cross-account approval surfaces are supported, route native cards only to explicitly configured approval accounts with clear authorization, command-disclosure rules, and regression coverage; otherwise document origin-account-only native delivery and keep the same-chat manual fallback reliable and explicit.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a Discord approval request with `turnSourceChannel: "discord"` and `turnSourceAccountId` set to a different account than the configured native approver account; current main rejects native handling before resolving approver-DM targets, while the `/approve` command path can still authorize the configured approver.

## Solution Assessment

Unclear. Relaxing the Discord account gate is the narrow mechanical route to native cards, but it is not necessarily the best fix until maintainers decide whether approval cards may cross Discord account boundaries and how command text/control disclosure should work.
