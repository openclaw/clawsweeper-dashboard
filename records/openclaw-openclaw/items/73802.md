## Summary

Keep open. Current main still rejects native Discord approval delivery when the approval request is bound to a different Discord account than the configured native approval account, matching the reported multi-account setup where manual approval works but no card appears.

## What This Changes

The issue reports that Discord exec approval IDs are created and manual `/approve` works, but the native Discord approval card/buttons are not delivered in a multi-account Discord setup.

## Best Possible Solution

Define the intended Discord multi-account native approval contract, then implement either safe cross-account approver delivery or an explicit documented origin-account-only fallback with regression coverage.

## Reproduction Assessment

Yes, source-level. A request with `turnSourceChannel: "discord"` and `turnSourceAccountId` set to a different account than the configured native approval account is rejected before origin, approver-DM, or runtime delivery planning.

## Solution Assessment

Unclear. Relaxing the account gate or changing fallback suppression is mechanically narrow, but the best maintainable fix needs an explicit policy for whether command-bearing Discord approval cards may cross account boundaries.
