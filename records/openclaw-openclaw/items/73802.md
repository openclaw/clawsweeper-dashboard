## Summary

Keep open. Current main still rejects Discord native approval delivery when the approval request is bound to a different Discord account than the configured native approval account, which matches the reported multi-account setup where manual `/approve` succeeds but no native card appears. Broadening that route is security-sensitive because native approval cards expose command text and approval controls across Discord accounts.

## What This Changes

The issue requests restoring Discord native exec approval cards/buttons for a multi-account setup where approvals are created and manual `/approve` works, but the configured Discord approval surface receives no card.

## Best Possible Solution

Define the Discord multi-account approval routing contract first. If cross-account approval surfaces are supported, route native cards to the configured approver account with explicit authorization checks and regression tests; otherwise document origin-account-only delivery and make the same-chat manual fallback explicit for this configuration.

## Reproduction Assessment

Yes. At source level, configure Discord native approvals only on one account and create an approval request whose `turnSourceAccountId` is another Discord account; current main predicts no native card for the configured approval account while manual `/approve` can still authorize the configured approver.

## Solution Assessment

Unclear. The narrow code change would be to relax the account gate for configured approver delivery, but that is not clearly the best solution until maintainers decide whether Discord native cards may cross account boundaries and what disclosure rules apply.
