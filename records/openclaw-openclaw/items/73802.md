## Summary

Keep open. Current main still gates Discord native approval delivery on the request's source account before resolving origin or approver-DM targets, so the reported multi-account setup can still approve manually while skipping the native Discord card/buttons.

## What This Changes

Discord exec approvals can be manually resolved with `/approve`, but the native approval card/buttons are not delivered in a multi-account Discord setup after the prior approval-routing fix.

## Best Possible Solution

Define the Discord multi-account native approval contract, then either route cards only to explicitly configured approver accounts with regression coverage or document origin-account-only native delivery while keeping manual fallback reliable.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a Discord approval request with `turnSourceChannel: "discord"` and `turnSourceAccountId` set to a different account than the configured native approver account; current main rejects native handling while `/approve` authorization can still accept the configured approver.

## Solution Assessment

Unclear. Removing the account gate is the narrow mechanical route to native cards, but the maintainable fix needs maintainer policy on cross-account command disclosure and authorization before changing delivery behavior.
