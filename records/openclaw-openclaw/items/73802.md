## Summary

Keep open. Current main still rejects Discord native approval delivery when the approval request is bound to a different Discord account than the account with native approvals configured, which matches the reported multi-account setup where manual `/approve` succeeds but no native card appears.

## What This Changes

Request Discord native exec approval cards/buttons to be delivered to the configured Discord approval surface when approval creation and manual `/approve` already work.

## Best Possible Solution

Define the Discord multi-account approval routing contract, then either route native cards to the configured approver account with explicit authorization checks and regression tests, or document origin-account-only delivery and make the same-chat manual fallback clear for this setup.

## Reproduction Assessment

Yes. At source level, configure native Discord approvals only on one account and create an approval request whose `turnSourceAccountId` is another Discord account; current main predicts no native card while manual `/approve` can still authorize the configured approver.

## Solution Assessment

Unclear. The narrow code change would be to relax the account gate for configured approver delivery, but maintainers should first decide whether native approval cards may cross Discord account boundaries and what disclosure rules apply.
