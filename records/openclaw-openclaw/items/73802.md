## Summary

Keep open. Current main still preserves the Discord multi-account native approval gap: native card delivery is rejected when the request source account differs from the configured native approver account, while the configured Discord approver can still resolve the same approval manually with `/approve`.

## What This Changes

The issue asks OpenClaw to restore native Discord exec approval card/button delivery for configured approvers instead of requiring a typed `/approve` fallback.

## Best Possible Solution

Define the multi-account Discord approval routing contract, then either route cards only to explicitly enabled and authorized approver-account targets with regression coverage, or document that native delivery is origin-account-only while keeping the manual fallback reliable.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a Discord approval request with `turnSourceChannel: "discord"` and `turnSourceAccountId` set to a different account than the configured native approver account; current main rejects native handling before target resolution while `/approve` remains authorized for the configured approver.

## Solution Assessment

Unclear. Relaxing the account gate is the narrow mechanical path to cards, but the best fix needs maintainer judgment on cross-account command disclosure and approval routing before implementation.
