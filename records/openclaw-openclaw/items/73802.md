## Summary

Keep open. The latest reporter retest shows the simple same-account Discord approval path works on `2026.5.7`, but the original multi-account setup is still source-reproducible: current main rejects native Discord approval delivery when the approval request is bound to a different Discord account than the configured native approval account.

## What This Changes

The issue asks OpenClaw to restore native Discord exec approval cards/buttons instead of requiring manual `/approve` commands in a multi-account Discord approval setup.

## Best Possible Solution

Define the intended Discord multi-account native approval contract, then either implement safe cross-account approver delivery or document and enforce an origin-account-only fallback with regression coverage.

## Reproduction Assessment

Yes, source-level. A request with `turnSourceChannel: "discord"` and `turnSourceAccountId` set to a different account than the configured native approval account is rejected before origin, approver-DM, or runtime delivery planning.

## Solution Assessment

Unclear. Relaxing the account gate is mechanically narrow, but the best maintainable fix needs an explicit policy for command-bearing Discord approval cards crossing account boundaries and should preserve a manual fallback when native routing is suppressed.
