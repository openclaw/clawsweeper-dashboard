## Summary

Keep open. Current main still matches the reported Discord multi-account approval failure: a request bound to one Discord account is rejected by the native approval gate before the configured approval account can plan origin or DM delivery, while the text `/approve` path remains authorized for configured Discord approvers.

## What This Changes

The issue requests restoration of native Discord exec approval card/button delivery when approvals are configured on a Discord approval account and manual `/approve` still works.

## Best Possible Solution

Define the Discord multi-account native approval contract. If cross-account approval surfaces are supported, route cards to the explicitly configured approval account with authorization, disclosure, and regression tests; otherwise document origin-account-only native delivery and make the same-chat manual fallback reliable and explicit for this configuration.

## Reproduction Assessment

Yes. At source level, configure native Discord approvals on one account and create a request with `turnSourceChannel: "discord"` and a different `turnSourceAccountId`; current main predicts no native card for the configured approval account while manual `/approve` remains accepted for configured approvers.

## Solution Assessment

Unclear. A narrow code change could relax the account gate for configured approver delivery, but that is not clearly the best fix until maintainers decide whether Discord native approval cards may cross account boundaries and what disclosure rules apply.
