## Summary

Keep open. The reporter's latest same-account retest shows the broad missing-card regression is fixed in current release, but the original multi-account setup still hits an intentional source-account gate; deciding whether Xavier should receive native approval cards for Forge-originated requests needs maintainer approval-routing and security policy.

## What This Changes

Discord exec approval requests should deliver native Discord approval cards/buttons instead of forcing manual `/approve` commands.

## Best Possible Solution

Define the Discord multi-account native approval routing contract, then either implement safe cross-account approver delivery or document and enforce origin-account-only behavior with fallback and regression coverage.

## Reproduction Assessment

Yes, source-level. A Discord approval request with `turnSourceChannel: "discord"` and `turnSourceAccountId` set to a different account than the configured native approval account is rejected before native origin or approver-DM delivery planning.

## Solution Assessment

Unclear. The desired user outcome is valid, but the safest implementation needs an explicit approval-routing policy for command-bearing cards across Discord account boundaries rather than a mechanical gate relaxation.
