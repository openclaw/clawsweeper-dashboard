## Summary

Current main still has the source-account gate that explains the reported multi-account Discord behavior, so this is not safe to close as implemented or obsolete. Manual `/approve` authorization is intentionally separate from native card delivery, and changing the native card route across Discord accounts needs maintainer/product approval because approval cards expose command details and controls.

## What This Changes

The issue requests restoring Discord native exec approval cards/buttons for a multi-account Discord setup where approval IDs are created and manual `/approve` works but no native card is delivered.

## Best Possible Solution

Define the Discord multi-account approval routing contract explicitly. If cross-account native approval surfaces are supported, route cards only to authorized configured approval accounts with tests for account mismatch, approver filtering, and manual `/approve` parity; otherwise document origin-account-only native delivery and make the fallback message precise for this configuration.

## Reproduction Assessment

Yes. Configure Discord native approvals on one account, create an approval request whose `turnSourceAccountId` is another Discord account, and current source/tests show the native card path is skipped while manual `/approve` can still authorize the approval.

## Solution Assessment

Unclear. Delivering native cards to the configured approval account would match the reporter's workflow, but the narrow maintainable fix depends on a maintainer-approved cross-account disclosure and authorization model.
