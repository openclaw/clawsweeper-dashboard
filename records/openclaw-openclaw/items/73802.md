## Summary

Keep open. Current main still has the Discord multi-account native approval gap: native card delivery is gated by the request's source Discord account before approver-DM targets are resolved, while configured approvers can still resolve the same approval manually with `/approve`.

## What This Changes

Discord exec approvals should deliver native Discord approval cards/buttons to the configured approval surface instead of requiring manual `/approve` commands in the reported multi-account setup.

## Best Possible Solution

Define the Discord multi-account native approval contract, then either route cards only to explicitly configured approver accounts with regression coverage or document origin-account-only native delivery while keeping the deterministic manual fallback reliable.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a Discord approval request with `turnSourceChannel: "discord"` and `turnSourceAccountId` set to a different account than the configured native approver account; current main rejects native handling before approver-DM resolution while `/approve` authorization still uses the configured approver list.

## Solution Assessment

Unclear. Relaxing the account gate is the narrow mechanical path to native cards, but the best fix needs maintainer judgment because Discord approval cards include command text and approval controls that may cross account boundaries.
