## Summary

Keep open. Current main still has the Discord multi-account native approval gap: native card delivery is rejected on the source-account gate before the configured approver-DM account can route a card, while the manual `/approve` path remains separately authorized.

## What This Changes

The issue asks for Discord native exec approval cards/buttons to be delivered when approvals are created in a multi-account Discord setup where manual `/approve` still works.

## Best Possible Solution

Define the Discord multi-account native approval contract, then either route cards only to explicitly configured approver accounts with regression tests or document origin-account-only native delivery while keeping the manual fallback reliable.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a Discord approval request with `turnSourceChannel: "discord"` and a `turnSourceAccountId` different from the configured native approver account; current main rejects native handling before approver-DM target resolution, while configured `/approve` authorization still succeeds.

## Solution Assessment

Unclear. Relaxing the account gate is the narrow mechanical path to restore cards, but the best fix needs maintainer judgment on cross-account approval disclosure and authorization before changing routing.
