## Summary

Current main still matches the reported Discord multi-account native approval gap. Discord native approval handling rejects a request bound to a different Discord account before origin or approver-DM targets are resolved, while the text `/approve` path remains separately authorized for configured Discord approvers. Recent fallback work keeps the typed command reachable, but it does not deliver the requested native approval card/buttons.

## What This Changes

The issue asks for Discord native exec approval cards/buttons to be delivered to the configured approval surface instead of requiring the operator to type `/approve` manually after an approval is created.

## Best Possible Solution

Define the Discord multi-account native approval contract. If cross-account approval surfaces are supported, route native cards only to explicitly configured approval accounts with clear authorization, command-disclosure rules, and regression coverage; otherwise document origin-account-only native delivery and keep the same-chat manual fallback reliable and explicit.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a Discord approval request with `turnSourceChannel: "discord"` and `turnSourceAccountId` set to a different account than the configured native approver account; current main rejects native handling before resolving approver-DM targets, while `/approve` remains authorized for the configured approver.

## Solution Assessment

Unclear. Relaxing the Discord account gate is the narrow mechanical path to native cards, but it is not necessarily the best fix until maintainers decide whether approval cards may cross Discord account boundaries and how command/control disclosure should work.
