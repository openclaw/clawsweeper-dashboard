## Summary

Keep open. Current main still rejects native Discord approval delivery when the approval request is bound to a different Discord account than the configured native approval account, which matches the reported multi-account symptom where manual `/approve` works but no native card is delivered.

## What This Changes

The issue requests restored Discord native exec approval card/button delivery when Discord exec approvals are configured and manual `/approve` currently works as a workaround.

## Best Possible Solution

Define the intended Discord multi-account native approval contract, then implement either safe cross-account approver delivery or an explicit documented origin-account-only fallback with regression coverage.

## Reproduction Assessment

Yes, source-level. A request with `turnSourceChannel: "discord"` and `turnSourceAccountId` set to a different account than the configured native approval account is rejected before origin, approver-DM, or runtime delivery planning.

## Solution Assessment

Unclear. Relaxing the account gate is the narrow mechanical change, but the best maintainable fix needs an explicit policy for when Discord approval cards may cross account boundaries.
