## Summary

Keep open. The one-account Discord approval path now has a fresh successful retest on current main and 2026.5.7, but the reported multi-account shape is still source-reproducible: a request bound to a different Discord account than the configured native approval account is rejected before native card delivery planning.

## What This Changes

Discord exec approvals should deliver native approve/deny cards in the configured Discord approval surface instead of leaving users with only manual `/approve` commands.

## Best Possible Solution

Define the supported Discord multi-account approval contract, then either safely route native cards to the configured approval account or document and test origin-account-only fallback behavior.

## Reproduction Assessment

Yes, source-level. A request with `turnSourceChannel: "discord"` and `turnSourceAccountId` different from the configured native approval account is rejected before origin, approver-DM, or runtime delivery planning, although the reporter's later one-account retest now passes.

## Solution Assessment

Unclear. A mechanical relaxation of the account gate is narrow, but the maintainable fix needs a security/product decision on whether command-bearing Discord approval cards may cross account boundaries.
