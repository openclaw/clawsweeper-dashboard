## Summary

Current main still leaves the reported Discord multi-account native approval-card path unresolved. The Discord native approval gate rejects delivery when the approval request is bound to a different Discord account than the configured native approval account, while manual Discord `/approve` remains available for configured approvers, matching the issue discussion.

## What This Changes

The issue requests restoration of Discord native exec approval cards/buttons for a multi-account Discord setup where approval IDs are created and manual `/approve` succeeds but no native card is delivered.

## Best Possible Solution

Define the Discord multi-account approval-routing contract. If cross-account native approval surfaces are supported, route native cards to the explicitly configured approval account while preserving approver authorization checks and adding regression coverage; otherwise document origin-account-only native delivery and make the fallback message explicit for this configuration.

## Reproduction Assessment

Yes. At source level, configure native approvals on one Discord account and create an approval request whose `turnSourceAccountId` is another account; current main predicts no native card while the command tests show a configured Discord approver can still resolve the approval manually.

## Solution Assessment

Unclear. Relaxing the account gate is the obvious mechanical fix, but it is not clearly the best solution until maintainers decide the security and UX contract for cross-account native approval delivery.
