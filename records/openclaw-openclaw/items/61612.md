## Summary

Keep open. This PR is a plausible CLI performance improvement with benchmark proof, but the current patch is not safe to merge because it can drop runtime-only plugin CLI help and can remove pairing channel data from help output.

## What This Changes

The branch adds early CLI help paths for built-in and plugin subcommands, switches help-only plugin CLI registration toward metadata loading, and adds focused tests around those paths.

## Best Possible Solution

Implement a descriptor- and activation-aware subcommand help fast path on current main while preserving runtime plugin help fallback and pairing channel registry population.

## Reproduction Assessment

Yes for the review findings. Source inspection of the PR diff and current contracts shows runtime-only plugin CLI help can be skipped, and pairing help snapshots channel registry data during registration.

## Solution Assessment

No. The fast-path direction is reasonable, but this patch over-broadly uses metadata-only plugin help loading and skips pairing registry setup that current behavior still depends on.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve runtime plugin help fallback:** `src/cli/run-main.ts:345`
  - body: Passing `helpOnly` for every help invocation routes fallback plugin command registration through metadata-only entries. Runtime-only third-party plugin CLIs that do not expose `cli-metadata` can lose `openclaw <plugin> --help`, so keep the full runtime fallback unless the primary command is known descriptor-backed.
  - confidence: 0.88
- **[P2] Keep pairing help channel data:** `src/cli/program/register.subclis.ts:232-235`
  - body: This short-circuits `pairing --help` before plugin CLI registration, but `registerPairingCli()` snapshots `listPairingChannels()` into option and argument help. Plugin-backed channels can disappear from help text, so preserve the registration needed for pairing help.
  - confidence: 0.86
