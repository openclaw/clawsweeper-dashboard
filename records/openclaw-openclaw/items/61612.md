## Summary

Keep this PR open, but it still needs changes before merge. The fast-path direction is useful, yet the current branch is stale against the CLI/plugin registration architecture and still has compatibility regressions in plugin help fallback and pairing help registration.

## What This Changes

The branch adds CLI subcommand help fast-path helpers, skips plugin bootstrap for pairing/plugins help, adds a helpOnly metadata-only plugin CLI path, and adds focused CLI/plugin tests.

## Best Possible Solution

Keep the performance goal, but rebase onto current main and make the fast path descriptor-aware while preserving full non-activating plugin CLI fallback and complete pairing help data.

## Reproduction Assessment

Yes for the review findings: the PR source shows the blanket helpOnly fallback and the pairing early return, while current tests/docs/source show runtime CLI fallback and pairing channel snapshot dependencies. I did not live-run the CLI in this read-only review.

## Solution Assessment

No. The requested optimization is reasonable, but this implementation is broader than the plugin compatibility contract; a narrower descriptor/manifest-owned fast path with conservative fallback is the maintainable route.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve runtime plugin CLI fallback for help:** `src/cli/run-main.ts:345`
  - body: The normal fallback path now passes `helpOnly` for every help invocation, so when the fast path misses it only loads `cli-metadata` registrations. Plugins that still expose CLI commands from the full non-activating runtime registrar can lose `openclaw <command> --help`; keep metadata-only loading for known descriptor-backed roots, but preserve the full CLI fallback for other plugin roots.
  - confidence: 0.87
- **[P2] Keep pairing help registration after channel plugins:** `src/cli/program/register.subclis.ts:235-237`
  - body: This early return registers `pairing` before plugin CLI registration runs. `registerPairingCli()` snapshots `listPairingChannels()` into help-visible `--channel` option and argument text, so plugin-backed pairing channels can be omitted or render as an empty channel list unless pairing help gets a metadata-only channel source or keeps the needed plugin registration.
  - confidence: 0.78
