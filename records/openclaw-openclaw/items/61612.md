## What This Changes

The PR adds CLI help fast paths for built-in and plugin subcommands, metadata-only help registration, pairing/plugins help shortcuts, and focused CLI/plugin tests.

## Best Possible Solution

Implement a descriptor- and activation-aware subcommand help fast path on current main while preserving runtime plugin CLI fallback and pairing channel registry population.

## Reproduction Assessment

Yes for the review findings: source inspection of the PR diff and current plugin/pairing CLI contracts shows the metadata-only fallback and pairing shortcut can drop existing help behavior. I did not run live CLI commands because this was a read-only review.

## Solution Assessment

No. The fast-path goal is sound, but this diff is not the narrowest maintainable solution on current main; it should use descriptor/activation-aware narrowing with full-loader fallback and preserve pairing channel help data.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve runtime plugin help fallback:** `src/cli/run-main.ts:342`
  - body: Passing `helpOnly` for every help invocation sends fallback plugin command registration through metadata-only entries. Plugins that expose CLI commands only from their runtime entry can lose `openclaw <plugin> --help`, so keep the full runtime fallback unless the command is known descriptor-backed.
  - confidence: 0.88
- **[P2] Keep pairing help channel data:** `src/cli/program/register.subclis.ts:231-235`
  - body: This short-circuits `pairing --help` before plugin CLI registration, but `registerPairingCli()` snapshots `listPairingChannels()` into option and argument help. Plugin-backed channels can disappear from help text, so preserve the registration needed for pairing help.
  - confidence: 0.86
