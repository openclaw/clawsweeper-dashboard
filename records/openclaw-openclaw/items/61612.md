## Summary

Keep open: the performance goal is valid and the PR has useful proof, but the current branch still has concrete CLI compatibility blockers and is behind later main-branch plugin-loader work.

## What This Changes

The branch adds CLI help fast paths and metadata-only plugin help registration for subcommand `--help` paths, with focused CLI/plugin tests.

## Best Possible Solution

Implement a descriptor- and activation-aware subcommand help fast path on current main while preserving runtime plugin CLI fallback and pairing channel registry population.

## Reproduction Assessment

Yes for the review findings: source inspection of the PR branch and current plugin/pairing contracts shows the metadata-only fallback and pairing shortcut can drop existing help behavior. I did not run live CLI commands because this was a read-only review.

## Solution Assessment

No. The fast-path goal is sound, but this branch should be rebased or replaced with a current-main implementation that uses descriptor/activation-aware narrowing while keeping full-loader fallback and pairing channel help data.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve runtime plugin help fallback:** `src/cli/run-main.ts:344-345`
  - body: Passing `helpOnly` for every help invocation sends fallback plugin command registration through metadata-only entries. Plugins that expose CLI commands only from their runtime entry can lose `openclaw <plugin> --help`, so keep the full runtime fallback unless the command is known descriptor-backed.
  - confidence: 0.88
- **[P2] Keep pairing help channel data:** `src/cli/program/register.subclis.ts:235-237`
  - body: This exits `pairing --help` before plugin CLI registration, but `registerPairingCli()` snapshots `listPairingChannels()` into option and argument help. Plugin-backed channels can disappear from help text, so preserve the registration needed for pairing help.
  - confidence: 0.86
