## Summary

Keep open: the performance direction is useful, but the patch is not correct as-is because plugin help fallback is forced through metadata-only loading and can drop supported command-only plugin CLIs.

## What This Changes

The PR adds built-in and plugin subcommand help fast paths plus focused tests around CLI/plugin registration.

## Best Possible Solution

Rebuild the help performance work on current main using manifest/descriptor-owned command lookup while preserving the documented full-runtime fallback for command-only plugin CLI registrations.

## Reproduction Assessment

Yes for the review finding by source inspection. The PR routes help-time plugin registration through metadata-only loading even though current docs and tests preserve command-only plugin CLI registrations through the full loader compatibility path.

## Solution Assessment

No. The performance approach is reasonable, but the implementation should use current main's plugin-owner resolution and only take metadata-only paths when that will not drop supported command-only plugin CLIs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Plugins that rely on the documented command-only eager compatibility path can lose `openclaw <plugin> --help`, so keep a full-runtime fallback unless metadata proves the command is descriptor-backed.
