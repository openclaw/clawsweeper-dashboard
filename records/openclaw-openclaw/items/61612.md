## Summary

Keep open: current main still lacks the proposed subcommand help fast path, but this PR has compatibility regressions in plugin help fallback and plugin-backed pairing help, plus a missing changelog entry.

## What This Changes

The PR adds subcommand help fast-path helpers, metadata-only plugin CLI help registration, pairing/plugins help shortcuts, and focused CLI/plugin tests.

## Best Possible Solution

Keep the fast-path direction, but make it descriptor-aware and conservative: avoid full runtime for known descriptor-backed plugin help while preserving full-loader fallback and plugin-populated pairing help.

## Reproduction Assessment

Yes for the review findings: current source and tests show the commands-only plugin fallback contract and the pairing help channel snapshot dependency. Live confirmation would run help against a commands-only plugin and a plugin-backed pairing channel, but this review stayed read-only.

## Solution Assessment

No. The proposed fast path is useful, but the global helpOnly fallback and pairing shortcut are broader than the existing compatibility contract; a descriptor-aware fast path with runtime fallback is safer.

- **[P2] Preserve runtime fallback for plugin help:** `src/cli/run-main.ts:342`
  - body: Passing `helpOnly` for every help invocation forces the normal fallback path onto metadata-only CLI loading. Plugins that expose CLI commands only from their runtime entry remain supported by current docs/tests, so their `openclaw <plugin> --help` command can disappear; keep the full-runtime fallback unless the command is known to be descriptor-backed.
