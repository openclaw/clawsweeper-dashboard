## Summary

Keep open: the performance direction is useful, but the current PR should not merge because it can drop runtime-only plugin CLI help and can remove plugin-populated pairing channel help data. The branch is also stale against current main's refactored CLI/plugin registration surface, so a narrow current-main repair or replacement is safer than applying the diff as-is.

## What This Changes

The PR adds CLI help fast paths for built-in and plugin subcommands and routes plugin CLI help registration through metadata-only loading with focused tests.

## Best Possible Solution

Keep the fast-help performance goal, but implement a descriptor-aware current-main path that preserves runtime-only plugin CLI compatibility and plugin-backed pairing channel help.

## Reproduction Assessment

Yes for the review findings: source inspection shows the PR sends fallback help through metadata-only plugin CLI registration, while current tests/docs show runtime-only plugin CLI commands still exist. Pairing help is also source-reproducible because registerPairingCli snapshots channel names during registration.

## Solution Assessment

No. The fast-path idea is worthwhile, but the current diff is too broad for the compatibility contract; a conservative descriptor-aware implementation with preserved full-loader fallback and pairing channel registration is safer.
