## Summary

Keep open: the performance goal is useful, but the patch still has merge-blocking CLI/plugin-contract issues around metadata-only plugin help fallback and core-owned plugin command mapping.

## What This Changes

The PR adds CLI subcommand help fast paths, a help-only plugin CLI registration mode, and focused tests for built-in and plugin help startup.

## Best Possible Solution

Rebuild the help performance work on current main using manifest/descriptor-owned command lookup while preserving the full non-activating loader fallback for supported command-only plugin CLIs.

## Reproduction Assessment

Yes for the review findings by source inspection. The PR routes help-time fallback registration through metadata-only loading while current docs and tests preserve commands-only api.registerCli compatibility through the full loader.

## Solution Assessment

No. The performance direction is reasonable, but the current implementation should use the existing plugin owner-resolution path and keep full-loader compatibility when metadata ownership is not enough.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve command-only plugin help fallback:** `src/cli/run-main.ts:345`
  - body: Passing `helpOnly` for all fallback help invocations makes plugin registration use only CLI metadata. Current SDK docs and tests still support commands-only `api.registerCli` entries, so those plugin commands can lose `openclaw <plugin> --help`; keep the full non-activating loader fallback unless descriptor or manifest ownership proves metadata-only loading is safe.
  - confidence: 0.86
- **[P2] Resolve plugin help owners from metadata:** `src/cli/run-main.ts:115-117`
  - body: Hard-coding `memory -> memory-core` in core makes the fast path stale when command ownership moves, is disabled, or becomes external. Use the existing manifest/descriptor plugin CLI owner lookup before narrowing `onlyPluginIds` so core stays plugin-agnostic.
  - confidence: 0.78
