## Summary

Keep open: the performance goal is useful, but the current patch still has source-reproducible compatibility blockers around plugin CLI help fallback and pairing help channel data, and the branch is dirty against current main.

## What This Changes

The branch adds CLI subcommand help fast paths and metadata-only plugin help registration, with focused tests for built-in/plugin help loading.

## Best Possible Solution

Implement a descriptor- and activation-aware subcommand help fast path on current main while preserving runtime plugin help fallback and pairing channel registry population.

## Reproduction Assessment

Yes for the review findings: source inspection of the PR diff and current contracts shows runtime-only plugin CLI help can be skipped, and pairing help snapshots channel registry data during registration.

## Solution Assessment

No. The fast-path direction is sound, but this patch over-broadly uses metadata-only help loading and skips pairing registry setup that current behavior still depends on.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve runtime plugin help fallback:** `src/cli/run-main.ts:345`
  - body: Passing `helpOnly` for every help invocation sends fallback plugin command registration through metadata-only entries. Plugins that expose CLI commands only from their runtime entry can lose `openclaw <plugin> --help`, so keep the full runtime fallback unless the command is known descriptor-backed.
  - confidence: 0.88
- **[P2] Keep pairing help channel data:** `src/cli/program/register.subclis.ts:232-235`
  - body: This short-circuits `pairing --help` before plugin CLI registration, but `registerPairingCli()` snapshots `listPairingChannels()` into option and argument help. Plugin-backed channels can disappear from help text, so preserve the registration needed for pairing help.
  - confidence: 0.86
