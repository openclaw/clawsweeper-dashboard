## Summary

Keep open for maintainer handling: the current PR head still introduces blocking process-execution, process-cleanup, shell-selection, type/runtime, and command-trust regressions, and it has no real Apple Silicon behavior proof.

## What This Changes

The branch adds Apple Silicon platform utilities, modifies process execution, shell selection, and kill-tree behavior, adds hybrid runtime/thread-pool/tensor helpers, and adds docs plus a verification script.

## Best Possible Solution

Narrow this into benchmark-backed macOS ARM64-only improvements that preserve Node defaults, shell preferences, process-group safety, and Windows command trust boundaries; move the hybrid runtime API into a separate design discussion.

## Reproduction Assessment

Yes for the blocking patch defects: PR-head source inspection plus a local Node execFile maxBuffer probe show the regressions and type/runtime blockers. No high-confidence reproduction exists for the claimed Apple Silicon performance benefit because no real Apple Silicon proof is attached.

## Solution Assessment

No. The PR is too broad and changes sensitive runtime contracts; a maintainable solution would be a smaller measured patch that keeps existing defaults and adds focused regression coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.97

Full review comments:

- **[P1] Preserve runExec default maxBuffer capacity:** `src/process/exec.ts:115`
  - body: Object-options callers that omit maxBuffer now get OPTIMAL_BUFFER_SIZE instead of leaving the value undefined for Node's default cap. That lowers successful output capacity to 64 KiB or 128 KiB and can make existing commands fail with ERR_CHILD_PROCESS_STDIO_MAXBUFFER.
  - confidence: 0.98
- **[P1] Use the trusted Windows cmd wrapper:** `src/process/exec.ts:146`
  - body: The PR calls process.env.ComSpec for Windows batch wrapping, while current main resolves a trusted System32 cmd.exe path and tests that ComSpec is ignored. This weakens the command execution trust boundary.
  - confidence: 0.95
- **[P1] Clean up detached timeout process groups:** `src/process/exec.ts:253`
  - body: The Apple Silicon path sets detached = true, creating a new process group, but timeout and no-output-timeout cleanup still calls child.kill("SIGKILL") on only the direct child. Grandchildren can survive cleanup on the platform this PR is trying to improve.
  - confidence: 0.95
- **[P1] Type spawn options before adding detached:** `src/process/exec.ts:253`
  - body: spawnOptions is inferred from an object that has no detached property, then the PR assigns spawnOptions.detached. Under the repo's strict TypeScript settings this new branch does not typecheck as written.
  - confidence: 0.9
- **[P1] Keep kill-tree group kills tied to actual detachment:** `src/process/kill-tree.ts:20-33`
  - body: The PR removes the detached option and gates group SIGKILL on Apple Silicon detection. That loses the current contract where callers pass whether the child was actually detached, so Linux and service-managed macOS paths can leak children or target the wrong process group.
  - confidence: 0.96
- **[P2] Respect configured SHELL before preferring zsh:** `src/agents/shell-utils.ts:61-64`
  - body: The Apple Silicon branch returns zsh before the existing SHELL lookup. Users who explicitly configured bash or another usable shell on macOS ARM64 would silently get zsh instead.
  - confidence: 0.93
- **[P2] Replace CommonJS require in the ESM helper:** `src/utils/platform.ts:25`
  - body: This package is ESM/NodeNext, but the new platform helper calls require() inside runtime functions. Those paths can throw at runtime, and the shell helper lookup points the dependency in the wrong direction.
  - confidence: 0.93
- **[P2] Mark dynamic-import test bodies async:** `src/utils/platform.test.ts:54`
  - body: This test body and several later callbacks use await import() inside non-async functions, which is a TypeScript syntax error. The new test file will not compile until those callbacks are async or rewritten.
  - confidence: 0.99
- **[P1] Define thread-pool sizing helpers:** `src/utils/thread-pool.ts:25-29`
  - body: getDefaultThreadPoolConfig() calls getPhysicalCpuCount() and getPerformanceCoreCount(), but this file does not import or define either symbol. The new thread-pool module cannot typecheck as written.
  - confidence: 0.98
- **[P1] Use an ESM-safe worker entrypoint:** `src/utils/thread-pool.ts:83-84`
  - body: The new thread pool constructs Worker(__filename), but OpenClaw is an ESM/NodeNext package and __filename is not available in ESM runtime modules. Creating the worker can fail before any task runs.
  - confidence: 0.9
- **[P1] Preserve thread-pool task resolvers:** `src/utils/thread-pool.ts:181-185`
  - body: run() stores each task's real Promise handlers, but submitTaskToWorker() immediately overwrites that active task entry with no-op callbacks. Completed tasks would not settle the Promise returned to callers.
  - confidence: 0.94
- **[P1] Add worker-side execute-task handling:** `src/utils/thread-pool.ts:175-179`
  - body: The main thread posts execute-task messages to workers, but this module never defines a worker-side parentPort message loop to execute the task and emit completion or error messages. Dispatched work can remain pending forever.
  - confidence: 0.92
- **[P2] Stop calling missing thread-pool exports:** `src/utils/tensor-ops.ts:396-405`
  - body: The tensor helpers destructure getOptimalParallelFactor and getOptimalBatchSize from ./thread-pool.js, but thread-pool.ts exports neither name. Calling these helpers will fail at runtime.
  - confidence: 0.91
