## Label Justifications

- `P2`: The report describes a normal-priority install/runtime regression with limited blast radius to non-root packaged gateway deployments and affected plugins.
- `impact:other`: The impact is plugin availability loss during gateway startup, which is maintainer-visible but outside the more specific owned impact categories.

## Summary

Current main and shipped releases no longer have the gateway-runtime dependency install path that caused non-root EACCES under packaged plugin directories; the package-manager-owned plugin install model from the merged dependency refactor covers the reported failure.

## What This Changes

The issue reports that non-root gateway startup could fail to load acpx, browser, and whatsapp because bundled plugin runtime dependency installation attempted to create node_modules under root-owned dist extension directories.

## Best Possible Solution

Keep the shipped package-manager-owned plugin dependency model and ask for a fresh report only if a current release still starts a package-manager child process under packaged dist extension directories during gateway startup.

## Reproduction Assessment

Yes for the historical bug: the report gives concrete 2026.4.26 root-owned global install plus non-root systemd-user gateway steps and EACCES logs. I did not reproduce a current-main failure because source and release evidence show the runtime install path is gone.

## Solution Assessment

Yes. The package-manager-owned model with explicit plugin install/update handling and legacy cleanup is the maintainable fix; adding a new runtime --prefix or chown workaround would reintroduce a second dependency path.
