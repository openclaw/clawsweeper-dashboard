## Label Justifications

- `P2`: The report described a real plugin-loading regression with limited blast radius that is now fixed in shipped releases.
- `impact:other`: The affected behavior was silent loss of selected plugin availability rather than data loss, security, crash-loop, session-state, auth-provider, or direct message-loss evidence.

## Summary

Current main and shipped releases no longer have the gateway-runtime per-plugin dependency install path that caused EACCES under root-owned bundled plugin directories; the matching dependency-handling refactor is in stable v2026.5.2 and later.

## What This Changes

Fix the regression where a non-root gateway could lose bundled plugins because runtime dependency installs attempted to write `node_modules` under root-owned `dist/extensions/*` directories.

## Best Possible Solution

Keep the shipped package-manager-owned plugin dependency model and ask for a fresh current-release report only if a gateway still launches npm or writes under `dist/extensions/*/node_modules`.

## Reproduction Assessment

Yes for the historical 2026.4.26 failure: the report includes concrete root-owned npm-global install steps and EACCES logs. I did not reproduce a current-main failure because current source, tests, and release provenance show the runtime install path is gone.

## Solution Assessment

Yes. The shipped fix removes gateway startup writes into packaged extension directories and uses explicit install/update/doctor flows under managed plugin roots, which is safer than documenting `chown` or adding another runtime prefix mode.
