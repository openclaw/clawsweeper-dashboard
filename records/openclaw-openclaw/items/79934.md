## Summary

Keep open. The PR is a plausible feature/API seam for SQLite transcript projections, but the current patch misses an existing OpenResponses tool identity shape and adds a supply-chain-sensitive unpinned codegen path; it also depends on the still-open database-first refactor branch.

## What This Changes

The PR adds SQLite transcript projection helpers, active-branch projection loading, runtime SDK exports, tests, docs, and changelog coverage for companion rebuilds from canonical transcript rows.

## Best Possible Solution

Settle the database-first base, then merge a narrow projection follow-up that covers OpenResponses `call_id`/`function_call_output`, keeps codegen packages pinned or separately reviewed, and preserves canonical SQLite rows as the durable source.

## Reproduction Assessment

Not applicable as a current-main bug reproduction: this is a new feature/API seam. The PR-specific defect is source-reproducible by comparing the new extraction keys with current OpenResponses `call_id` schema and transcript repair handling.

## Solution Assessment

No. The direction is maintainable, but the current patch should not merge until it handles `call_id`/`function_call_output`, pins or splits the codegen execution path, and is rebased onto a settled database-first base.
