## Summary

Keep open. Current main has partial session-scoped effective-tool visibility through `tools.effective` and the Control UI, but it still does not provide strict declared-vs-effective inheritance, declared/removed tool comparison, or a general per-layer clipping reason trace.

## What This Changes

Request strict agent tool inheritance plus per-session visibility into declared, effective, removed, and blocked tools with structured reasons for runtime clipping.

## Best Possible Solution

Design one generic policy-resolution trace that records declared inputs, effective tools, removed tools, and removal layer/reason, expose it through the typed Gateway protocol, Control UI, and docs, then add strict inheritance as an explicit fail-closed mode for layers that cannot honor declared parity.

## Reproduction Assessment

Yes. A high-confidence static reproduction is available on current main: configure or inspect a session whose runtime tool list is reduced by sandbox/subagent/provider policy, then compare the current `tools.effective` result and schema against the requested declared/removed/blocked reason fields; those fields and the strict config key are absent.

## Solution Assessment

No. The current `tools.effective` implementation is useful partial visibility, but it is not the best final solution because it exposes only the effective list; the maintainable fix is an additive protocol-level trace first, with strict inheritance layered on top after security/runtime precedence is explicit.
