## Summary

Keep open: current main now has partial runtime-effective tool visibility via `/tools` and `tools.effective`, but the requested strict inheritance mode and a structured per-layer clipping/reason trace are still not implemented and need maintainer design because several restriction layers are intentional safety boundaries.

## What This Changes

The issue requests strict parity between declared agent tools and effective session tools, plus per-session visibility into effective/removed tools and structured reasons when a tool is unavailable.

## Best Possible Solution

Treat this as a product/API design item: extend the runtime tool diagnostics contract to expose declared-vs-effective differences and reason provenance, and decide whether any strict mode can safely fail closed instead of overriding safety layers.

## Reproduction Assessment

Not applicable as a single bug reproduction. Source and docs confirm runtime policy layers can intentionally narrow tools after agent-level configuration, but the issue does not provide a current-main session transcript or config that reproduces the specific `exec` denial.

## Solution Assessment
