## Summary

Keep open: current main provides partial runtime-effective tool visibility through `tools.effective` and `/tools`, but the requested strict inheritance mode plus declared-vs-effective removed-tool provenance and structured clipping reasons are still absent and security-sensitive.

## What This Changes

The issue requests strict tool inheritance plus per-session visibility into declared, effective, and clipped tools with structured unavailability reasons.

## Best Possible Solution

Define a maintainer-approved diagnostics contract that exposes declared, effective, and removed tools with reason provenance, while any strict mode fails closed and preserves safety-layer precedence.

## Reproduction Assessment

No high-confidence live reproduction of the specific `exec` denial was established. Source and docs do confirm the remaining gap: layered clipping is intentional, `tools.effective` is available-tool oriented, and strict parity plus removed-tool reason metadata are not implemented.

## Solution Assessment

No for an automated one-pass fix. The maintainable path needs product and security design first, with diagnostics and fail-closed precedence before any strict inheritance mode is added.
