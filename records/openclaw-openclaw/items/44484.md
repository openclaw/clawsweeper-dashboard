## Summary

Keep open. Current main has useful session-scoped effective-tool visibility, but it still lacks strict declared-vs-effective inheritance, declared/removed tool comparison, and a general per-layer clipping reason trace.

## What This Changes

Request strict tool inheritance plus per-session visibility into declared, effective, removed, and blocked tools with structured removal reasons.

## Best Possible Solution

Add an additive policy-resolution trace that records declared inputs, effective tools, removed tools, and removal layer/reason, then layer strict inheritance on top as an explicit fail-closed mode.

## Reproduction Assessment

Yes. Source inspection on current main shows the effective session path and schema expose only the final effective inventory, while the requested strict inheritance key and declared/removed/reason fields are absent.

## Solution Assessment

No. The current `tools.effective` path is a useful partial solution, but the requested single strict mode should not land before the policy precedence and trace contract are designed across runtime, protocol, UI, and docs.
