## What This Changes

The issue asks OpenClaw to expose declared versus effective session tools, explain which runtime layer removed a tool, and optionally enforce strict declared/effective parity.

## Best Possible Solution

Define an additive declared/effective/removed-tool diagnostics contract with per-layer reason provenance, then add any strict mode only as an opt-in, fail-closed, security-reviewed config surface by reviewing or replacing the related open PR.

## Reproduction Assessment

Yes at source level: current main lacks the requested strict inheritance key, declared/effective/removed-tool schema fields, and general blocked-by contract. I did not establish a live reproduction of the exact exec denial session in this read-only review.

## Solution Assessment

No for an automated one-pass fix: strict inheritance and blocked-by semantics define cross-layer tool authority and need a product/security contract first. The safer path is additive diagnostics with fail-closed precedence before any parity-enforcement mode.
