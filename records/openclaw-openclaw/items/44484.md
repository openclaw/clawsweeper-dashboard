## Summary

Keep open: current main and the latest release expose a session-scoped effective tool inventory, but they still do not provide strict inheritance, declared-versus-effective diffs, removed-tool provenance, or a general blocked-by contract; the related implementation PR is open and not merge-ready.

## What This Changes

The issue requests strict tool inheritance, effective tool-set visibility, and structured reasons when runtime/session layers clip declared agent tools.

## Best Possible Solution

Define an additive declared/effective/removed-tool diagnostics contract with per-layer reason provenance, then add any strict mode only as an opt-in, fail-closed, security-reviewed config surface.

## Reproduction Assessment

Yes at source level: current main lacks the requested strict inheritance key, declared/effective/removed-tool schema fields, and general blocked-by contract. I did not establish a live reproduction of the exact exec denial session in this read-only review.

## Solution Assessment

No for an automated one-pass fix: strict inheritance and blocked-by semantics define cross-layer tool authority and need a product/security contract first. The safer path is additive diagnostics with fail-closed precedence before any parity-enforcement mode.
