## What This Changes

The PR changes both auto-reply dispatcher entry points so channel-supplied `beforeDeliver` hooks compose with the canonical `reply_payload_sending` plus `message_sending` gate and adds dispatcher regression tests.

## Best Possible Solution

Land one canonical hook-composition fix that runs `message_sending` exactly once per visible reply across plain, buffered, Telegram native, and downstream delivery paths, with focused tests and redacted live or equivalent channel proof.

## Reproduction Assessment

Yes from source inspection: current main and `v2026.6.6` compose channel `beforeDeliver` only with `reply_payload_sending`, and Telegram supplies identity `beforeDeliver` hooks on real reply paths. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No, not as-is. The shared dispatcher layer is the right place to fix the bypass, but this patch needs exactly-once `message_sending` ownership so Telegram/native `deliverReplies` paths do not run the hook both before and during platform delivery.
