## Summary

Keep open. Current main still has the all-or-nothing doctor migration path described in the report, and the adjacent related issues are not duplicates because this issue is specifically about the manual `doctor --fix` escape hatch dropping safe legacy migrations when unrelated validation errors remain.

## What This Changes

Make `openclaw doctor --fix` persist safe legacy config migrations even when unrelated plugin/provider validation issues still need separate repair.

## Best Possible Solution

Keep the issue open and fix the doctor config flow so legacy migrations are an independently committable repair step with regression coverage for a migrated config that still has an unrelated provider validation error.

## Reproduction Assessment

Yes from source inspection: a migrated config that still fails plugin-aware validation returns `config: null`, so the caller retains the unmigrated candidate and the later write can fail on the unrelated issue. I did not run a live `doctor --fix` repro because this review is read-only.

## Solution Assessment

Yes, with one caveat: the narrow fix should persist only the safe legacy migration delta and keep unresolved validation errors visible. Returning the migrated object alone may not be sufficient because the final write path still rejects invalid candidates.
