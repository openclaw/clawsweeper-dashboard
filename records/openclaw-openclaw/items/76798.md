## Summary

Current main still has the reported all-or-nothing behavior, so this should stay open. The source path applies legacy migrations, discards the migrated candidate when full plugin-aware validation still fails, then the write path validates the whole config again before persisting, so an unrelated provider/plugin issue can still block safe legacy cleanup.

## What This Changes

`doctor --fix` should persist safe legacy config migrations even when unrelated config validation issues remain.

## Best Possible Solution

Make legacy doctor migrations independently committable, ideally as an atomic doctor repair step that persists schema-safe legacy cleanup while reporting unresolved plugin/provider issues separately.

## Reproduction Assessment

Yes, by source inspection: a config with `agents.defaults.llm` plus an unavailable `tools.web.search.provider` reaches `migrateLegacyConfig`, loses the migrated candidate on validation failure, and then the write path rejects the whole candidate before persistence. I did not run a live `doctor --fix` because this review is read-only.

## Solution Assessment

No, the current implementation is not the best way to solve the issue because it couples safe legacy migrations to unrelated plugin-aware validation. The safer fix is a doctor-scoped partial/atomic migration commit with regression coverage for a legacy key plus an unrelated provider error.

## Repair Work Prompt

Fix the `doctor --fix` legacy migration catch-22 reported in openclaw/openclaw#76798. On current main, `migrateLegacyConfig` returns `config: null` whenever post-migration plugin-aware validation still has an unrelated issue, and `applyLegacyCompatibilityStep` then keeps the unmigrated candidate; later `writeConfigFile` rejects the whole candidate, so safe legacy migrations such as removing `agents.defaults.llm` never reach disk when an unrelated provider/plugin validation error remains. Implement a narrow doctor-owned fix that makes safe legacy migrations independently committable without adding a general invalid-config write bypass. Add regression coverage for a raw config containing both `agents.defaults.llm` and an unavailable `tools.web.search.provider: "brave"`, proving the legacy key is removed/persisted or carried to the write candidate while the unrelated provider issue is still reported. Do not change plugin/provider semantics, do not add a config option, and stop for maintainer review if the only feasible approach is a broad write-path bypass for arbitrary invalid configs. Add a user-facing changelog entry because this changes `doctor --fix` recovery behavior.
