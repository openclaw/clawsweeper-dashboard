## Summary

Keep open. Current main still has the all-or-nothing doctor migration path, and the linked open PR #76800 is the right active implementation candidate rather than a reason to close before merge.

## What This Changes

`doctor --fix` should persist safe legacy config migrations even when unrelated plugin/provider validation errors remain.

## Best Possible Solution

Land #76800 or an equivalent narrow fix with regression coverage for a legacy migration plus an unrelated provider validation error.

## Reproduction Assessment

Yes. Source inspection gives a deterministic path: a config with a legacy migration and a remaining provider validation error makes `migrateLegacyConfig` return null, so the caller keeps the unmigrated candidate.

## Solution Assessment

Yes. The narrow maintainable fix is to make safe legacy migrations independently committable while keeping remaining validation errors visible, rather than broadening config writes generally.
