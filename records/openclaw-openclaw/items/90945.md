## Summary

This issue is still necessary: current main and v2026.6.6 still lack the all-queue SQLite startup recovery sweep requested for `channel_ingress_events`, while the direct fixing PR remains open and unmerged.

## What This Changes

The issue reports that stale `claimed` rows in the SQLite `channel_ingress_events` queue survive gateway crashes and silently deadlock Telegram DM ingress.

## Best Possible Solution

Land or revise the linked channel/gateway fix so stale SQLite ingress claims are safely released without dropping live claims, with regression coverage for dead owners, null lane keys, PID-style owners, and claim-token races.

## Reproduction Assessment

Yes. Source inspection shows durable `claimed` rows can persist in `channel_ingress_events` and current main has no all-queue startup sweep; the issue also has multiple independent production reproductions, though I did not run a live Telegram crash test in this read-only review.

## Solution Assessment

Yes. The best current path is to keep this issue open and review/land or revise the linked fixing PR, because neither current main nor the latest release implements the missing SQLite startup recovery.
