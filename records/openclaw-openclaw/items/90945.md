## Label Justifications

- `P1`: The issue describes a real Telegram channel workflow deadlock where inbound messages are accepted but not delivered after a crash/restart.
- `impact:message-loss`: The reported failure leaves Telegram messages stuck in `claimed`/`pending` ingress rows with no reply delivered to the user.
- `impact:session-state`: The failure is caused by stale persisted ingress claim state surviving a crashed session or gateway process.

## Summary

Keep open. Current main and v2026.6.6 still lack a gateway-wide SQLite stale-claim sweep, and this issue is already paired with the open clean implementation PR at https://github.com/openclaw/openclaw/pull/90989, so closure should wait for that PR to merge or be replaced.

## What This Changes

Recover stale SQLite `channel_ingress_events` claims after gateway/session crashes so Telegram inbound messages do not remain stuck behind `claimed` rows.

## Best Possible Solution

Keep this issue as the canonical SQLite stale-claim tracker until https://github.com/openclaw/openclaw/pull/90989 lands or is replaced, then close it with the merged fix proof and carry any periodic/PID-reuse remainder into a narrower follow-up if needed.
