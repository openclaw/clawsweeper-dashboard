## Label Justifications

- `P1`: The report describes a real Telegram channel workflow becoming silently unresponsive after a crash or restart.
- `impact:message-loss`: Inbound Telegram messages can remain acknowledged upstream but undelivered in OpenClaw while rows are stuck behind claimed SQLite events.
- `impact:session-state`: The stuck claim leaves channel/session processing state stale across gateway restarts until an operator edits SQLite manually.

## Summary

Keep open: current main and the latest release still lack recovery for stale SQLite `channel_ingress_events` claims, and the open linked PR is the canonical fix candidate. The independent reports in the comments strengthen the bug signal rather than making this cleanup-closeable.

## What This Changes

Recover stale claimed SQLite channel ingress events after a gateway or session crash so Telegram DMs do not silently deadlock.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/90989, or replace it with the same shared queue/gateway recovery shape that releases stale SQLite claims with age, owner-liveness, and claim-token race guards.

## Reproduction Assessment

Yes, at source level. Current main has stored claim fields and per-queue recovery but no startup/global caller for stale SQLite claims, and multiple comments report the same claimed/pending row pattern across releases and platforms.

## Solution Assessment

Yes, the requested shared recovery direction is the right owner boundary. A Telegram-only workaround would miss the shared `channel_ingress_events` queue and other channel users of the same contract.
