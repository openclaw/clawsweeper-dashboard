## Label Justifications

- `P1`: The reported failure can make a Telegram DM session silently stop processing inbound messages for affected users.
- `impact:message-loss`: The issue is about inbound Telegram updates being stuck in the durable ingress queue instead of being delivered.
- `impact:session-state`: The failure mode depends on stale persisted claim/session state after a gateway or worker crash.

## Summary

Current main already recovers stale Telegram SQLite ingress claims before draining pending updates, so the reported v2026.6.1 deadlock is fixed on main but not yet in a tagged release.

## What This Changes

The issue reports that SQLite-backed `channel_ingress_events` rows can remain permanently `claimed` after a gateway or session crash, causing Telegram DM ingress to stop responding until manual database repair.

## Best Possible Solution

Keep the current main recovery path and make sure the next release ships it; no separate core issue remains unless a build after the fix timestamp still leaves claimed rows stuck.

## Reproduction Assessment

Yes, at source level: the release tag lacks the drain-time recovery, while current main has a regression test that creates a claimed Telegram update and verifies it is recovered with no stuck claims. I did not run a live Telegram crash repro in this read-only review.

## Solution Assessment

Yes. Recovering in the Telegram isolated ingress drain with lane and live-process guards is narrower than a blind startup sweep and matches the queue ownership boundary.
