## Label Justifications

- `P1`: The report describes a real channel workflow deadlock where Telegram messages are accepted but replies stop until manual SQLite intervention.
- `impact:message-loss`: The affected Telegram updates are ingested but remain undelivered behind stuck claimed rows.
- `impact:session-state`: The failure is caused by stale persisted ingress/session claim state surviving gateway or session crashes.

## Summary

Keep open. Current main and v2026.6.6 still lack gateway-wide recovery for stale SQLite `channel_ingress_events` claims, and the issue has an open proof-supplied closing PR that should land or be replaced before this is closed.

## What This Changes

The issue reports Telegram DMs becoming silently unresponsive because SQLite `channel_ingress_events` rows left `claimed` after a gateway or session crash are never swept back to `pending`.

## Best Possible Solution

Keep this issue open until https://github.com/openclaw/openclaw/pull/90989 or an equivalent token-guarded stale-claim recovery lands, then close it as implemented.

## Reproduction Assessment

Yes, source-reproducible but not locally executed. Current main has only queue-local `recoverStaleClaims` when a caller invokes it, while startup runs no ingress sweep; multiple comments provide live v2026.6.1 and v2026.6.5-beta.1 SQLite states.

## Solution Assessment

Yes for the central issue. The narrow maintainable path is shared SQLite ingress-claim recovery before channels start, with token and owner guards; review or extend the existing PR rather than opening a second repair lane.
