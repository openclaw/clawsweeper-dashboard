## Label Justifications

- `P1`: The issue describes a real Telegram channel workflow deadlock where inbound user messages remain queued or claimed after a crash.
- `impact:message-loss`: Stuck claimed ingress rows cause Telegram inbound messages to be silently undelivered until manual SQLite repair.
- `impact:session-state`: The failure is stale persisted queue/session state surviving gateway restarts and blocking future processing.

## Summary

Keep open: current main and the latest release still lack a gateway-wide SQLite ingress-claim recovery path, while multiple reports reproduce stuck `channel_ingress_events` claims after Telegram/gateway crashes. The direct fixing PR is open and linked, so this issue should remain open until that PR lands or is replaced.

## What This Changes

The issue requests recovery for stale SQLite `channel_ingress_events` claims so Telegram inbound messages do not deadlock after a session or gateway crash.

## Best Possible Solution

Land or replace the shared channel/gateway recovery from https://github.com/openclaw/openclaw/pull/90989, ensuring stale dead-owner SQLite claims are released safely without duplicating live message delivery.

## Reproduction Assessment

Yes. Multiple issue comments give real-environment reproductions, and source inspection of current main shows no startup or global SQLite claim recovery path for already-claimed rows.

## Solution Assessment

Yes, the shared queue/gateway layer is the right owner boundary for the main bug. Maintainers should still review whether startup-only recovery is sufficient for the container PID-reuse and periodic-recovery concerns raised in the comments.
