- `P1`: The issue describes a silent Telegram inbound deadlock after crashes in real deployments, affecting active channel workflows now.
- `impact:message-loss`: Inbound Telegram messages are acknowledged or queued internally but never dispatched to the agent while stale claimed rows block the queue.
- `impact:session-state`: The root state is a stale persisted SQLite claim/session queue record that survives restarts and blocks later work.

## Summary

Keep this issue open: current main still lacks a global SQLite `channel_ingress_events` stale-claim recovery path, and the live closing PR is still open and unmerged. The report is well-supported by source inspection plus multiple independent Telegram/container reproductions.

## What This Changes

Recover stale `channel_ingress_events` SQLite claims after gateway or session crashes so Telegram ingress does not deadlock behind claimed rows.

## Best Possible Solution

Land or revise the linked recovery PR so stale SQLite ingress claims are reclaimed with owner/token guards, with regression coverage for null `lane_key`, dead owners, and container PID reuse behavior.

## Reproduction Assessment

Yes. The source path is clear without running a live Telegram setup: stale `claimed` rows are only recoverable through per-queue caller paths, while gateway startup has no global SQLite ingress sweep; multiple independent comments give live reproduction data.

## Solution Assessment

Yes, a core SQLite ingress recovery path is the right owner boundary because the stuck data lives in shared state, not only the Telegram spool. The linked PR should be reviewed or updated rather than closing this issue before merge.
