## Summary

Keep open. Current main and the latest stable release still lack a gateway-wide recovery path for stale SQLite ingress claims, and the issue is already paired with an open closing PR that should be reviewed or landed before this issue closes.

## What This Changes

Recover stale claimed SQLite channel ingress events after gateway or session crashes so Telegram DMs do not deadlock behind permanently claimed rows.

## Best Possible Solution

Review and land the shared, token-guarded SQLite ingress claim recovery in https://github.com/openclaw/openclaw/pull/90989, folding in any required container PID-reuse or periodic-reaper scope before closing this issue.

Yes, at source level. Current main exposes tokened claimed rows and queue-local stale recovery, but no gateway-wide startup sweep, and the issue has multiple independent live reports showing the same claimed-row backlog surviving restarts.

## Solution Assessment

Yes, the best direction is a shared channel ingress recovery path with age, owner-liveness, and claim-token guards rather than a Telegram-only SQL workaround; maintainers still need to decide whether periodic recovery belongs in the same patch.
