- `P1`: The issue reports a real broken Telegram channel workflow where inbound messages are silently stranded after crashes or restarts.
- `impact:message-loss`: Stale claimed ingress rows can cause Telegram messages to be acked upstream but never delivered to the agent.

## Summary

Keep open: current main still has the reported SQLite ingress stale-claim gap, and an open linked fix PR is the active implementation path rather than a reason to close the issue before merge.

## What This Changes

The issue asks OpenClaw to recover stale `channel_ingress_events` SQLite claims after gateway/session crashes so Telegram DMs do not silently deadlock.

## Best Possible Solution

Land or revise https://github.com/openclaw/openclaw/pull/90989 so the shared SQLite ingress queue recovers dead-owner claimed rows before channels can deadlock, then close this issue after the fix merges.

## Reproduction Assessment

Yes. I did not run a live Telegram crash, but current source plus multiple independent issue comments show the stale `claimed` SQLite row path and the PID-based recovery skip that can keep rows blocked.

## Solution Assessment

Yes, the best current direction is a shared queue/gateway startup recovery path rather than a Telegram-only workaround, with maintainer review for the container PID and periodic-recovery edge cases.
