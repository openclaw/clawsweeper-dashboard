- `P1`: The issue reports a silent Telegram inbound deadlock affecting real deployments until operators manually edit SQLite state.

## Summary

Keep open: current main still lacks a gateway startup or periodic sweep for stale SQLite `channel_ingress_events` claims, and the linked closing PR is still open rather than merged.

## What This Changes

The issue reports Telegram inbound messages deadlocking after a crash because SQLite channel ingress rows remain `claimed` until manually reset.

## Best Possible Solution

Land a bounded core channels/gateway recovery that safely reclaims stale claimed ingress rows across channel queues, with https://github.com/openclaw/openclaw/pull/90989 as the active candidate, then close after merge.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: leave a row in `channel_ingress_events` as `claimed`, restart current main, and no startup task resets it before channel startup. I did not run the live Telegram crash path in this read-only review.

## Solution Assessment

Unclear until the linked PR is reviewed: a startup sweep matches the central repro, but the comments add periodic recovery and PID-reuse edge cases that should be checked before calling it complete.
