## Label Justifications

- `P1`: The issue reports a broken Telegram inbound workflow where messages are accepted into OpenClaw state but no longer delivered to the agent after a crash or restart.
- `impact:message-loss`: The reported failure strands Telegram inbound messages in the durable ingress queue behind stale claimed rows.
- `impact:session-state`: The deadlock is caused by persisted session/channel queue state remaining in an unrecovered `claimed` state.

## Summary

Current main still lacks startup-wide recovery for stale SQLite `channel_ingress_events` claims, and the issue is paired with an open fixing PR that has not merged yet. Keep this issue open until https://github.com/openclaw/openclaw/pull/90989 lands or maintainers choose a different canonical fix.

## What This Changes

Recover stale `channel_ingress_events` rows left in `claimed` state after a gateway/session crash so Telegram inbound delivery does not deadlock.

## Best Possible Solution

Land a reviewed fix through https://github.com/openclaw/openclaw/pull/90989 or a replacement that safely reclaims stale SQLite ingress claims without duplicating or dropping active channel work.

## Reproduction Assessment

Yes, with source-level confidence: current main has queue-level stale-claim recovery but no startup-wide sweep, and multiple issue comments report the same stuck `claimed` rows in released builds. I did not run a live Telegram crash/restart reproduction in this read-only review.

## Solution Assessment

Yes for keeping it open: the maintainable path is to review and land the linked recovery PR or an equivalent replacement, not close the issue before the fix merges. The PR should still account for the restart-overlap and PID-reuse evidence before landing.
