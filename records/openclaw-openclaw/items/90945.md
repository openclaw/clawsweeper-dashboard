## Summary

This is a valid P1 delivery bug still present on current main. The issue should stay open because the clean, proof-bearing PR https://github.com/openclaw/openclaw/pull/90989 explicitly closes it and remains unmerged; closing now would break the issue/PR tracking contract.

## What This Changes

Recover stale SQLite channel ingress claims after a gateway or session crash so Telegram delivery does not remain silently blocked.

## Best Possible Solution

Land one shared SQLite ingress recovery implementation with exact claim-token race protection, container-safe owner identity, and a clearly tested cadence that covers the reproduced restart and long-running orphan cases.

## Reproduction Assessment

Yes. Independent VM, container, and Kubernetes reports show stale `claimed` rows with pending backlogs, and releasing those rows to `pending` restores delivery; source inspection confirms current main has no global recovery path.

## Solution Assessment

Yes at the ownership boundary: recovery belongs in the shared SQLite ingress queue rather than Telegram-specific code. Before merge, the linked PR should explicitly settle PID reuse and whether startup-only recovery covers all reproduced failure modes.
