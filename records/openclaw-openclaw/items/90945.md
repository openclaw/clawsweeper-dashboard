## Label Justifications

- `P1`: The issue describes a real, silent Telegram inbound deadlock where healthy-looking deployments stop processing user messages.
- `impact:message-loss`: Telegram updates can be acknowledged upstream but remain stuck internally behind claimed SQLite ingress rows.
- `impact:session-state`: The failure is caused by stale claimed queue/session ownership state persisting after a crash or restart.

## Summary

Keep open: current main still has queue-local SQLite claim recovery but no startup or periodic sweep for stale `channel_ingress_events` claims, and the linked fix PR is still open rather than merged.

## What This Changes

The issue requests recovery for stale SQLite channel ingress claims after crashes so Telegram DMs do not silently deadlock.

## Best Possible Solution

Land a narrow core SQLite ingress recovery that reclaims stale `channel_ingress_events` claims without stealing live work, covers null lane keys, and includes focused gateway startup or periodic recovery tests.

## Reproduction Assessment

Yes from source inspection and multiple user reports, though I did not run a live Telegram crash repro in this read-only review. Current main has stale-claim helpers but no runtime startup or periodic recovery path for the SQLite `channel_ingress_events` rows.

## Solution Assessment

Yes, fixing the core SQLite ingress queue recovery path is the right layer. The existing open PR is the current implementation path, but maintainers should confirm whether startup-only recovery handles the PID-namespace and in-process restart cases raised in comments.
