- `impact:message-loss`: Inbound Telegram messages are acknowledged upstream but can be stranded in OpenClaw's SQLite ingress queue until manual database recovery.

## Summary

Keep open. The report is valid on current main and the latest release, and the open proof-supplied fix PR at https://github.com/openclaw/openclaw/pull/90989 references this issue with closing syntax, so this issue should stay open until that PR lands or is replaced.

## What This Changes

The issue reports stale SQLite `channel_ingress_events` claims surviving a gateway or session crash and deadlocking Telegram inbound messages until the rows are manually reset.

## Best Possible Solution

Review and land the shared gateway startup recovery in https://github.com/openclaw/openclaw/pull/90989, preserving age, live-owner, and exact claim-token guards, then close this issue as implemented.

## Reproduction Assessment

Yes. Source inspection shows current main lacks the startup sweep, and the issue discussion includes independent v2026.6.1 and v2026.6.5-beta.1 reproductions with claimed SQLite rows and pending backlogs after restarts.

## Solution Assessment

Yes. The best path is the shared gateway startup recovery in the linked PR rather than a Telegram-only workaround or another manual database repair path.
