## Summary

Keep open: the underlying Bun launcher bug is valid, but this PR still needs real Bun runtime proof and the current patch weakens the direct-vs-transitive module-miss guard by not checking Bun's referrer.

## What This Changes

The PR updates `openclaw.mjs` to treat Bun-shaped module-not-found errors as direct optional launcher import misses by checking `err.specifier` and relative-specifier error messages.

## Best Possible Solution

Land a narrow launcher fix that recognizes Bun direct optional-import misses by matching both `specifier` and the launcher `referrer`, with focused regression coverage and redacted live Bun CLI proof.

## Reproduction Assessment

Yes. The linked report gives concrete `bun openclaw.mjs --version` steps, and current main still has the Node-only direct-miss helper on the optional warning-filter bootstrap path; I did not run Bun locally because it is not installed in this checkout.

## Solution Assessment

No, not as written. The fix should use Bun's `referrer` as well as `specifier` so the helper continues to swallow only direct launcher import misses instead of same-specifier transitive failures.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Check Bun referrer before swallowing direct misses:** `openclaw.mjs:63-65`
  - body: `isDirectModuleNotFoundError` is the guard that keeps transitive import failures from being treated as missing launcher output. Bun ResolveMessage exposes both `specifier` and `referrer`, but this branch returns true on `specifier` alone, so a nested module that misses the same relative path can be swallowed as an optional launcher miss. Please require the Bun `referrer` to point at the launcher before accepting the specifier or relative-message fallback.
  - confidence: 0.78
