## Summary

Keep open: this PR targets a valid Bun launcher regression, but the patch still weakens direct-vs-transitive module-miss detection by accepting Bun `specifier` and message matches without checking the Bun `referrer`; it also still lacks required real Bun after-fix proof.

## What This Changes

Adds Bun `err.specifier` and relative-message matching in `openclaw.mjs` so missing optional `warning-filter` imports are treated as direct module misses.

## Best Possible Solution

Land a narrow launcher fix that recognizes Bun direct optional-import misses by matching both specifier and launcher referrer, with focused regression coverage and redacted live Bun CLI proof.

## Reproduction Assessment

Yes, source-backed: the linked bug gives `bun openclaw.mjs --version`, and current main still rethrows Bun-shaped optional `warning-filter` misses because the helper lacks Bun `specifier`/`referrer` handling. I did not run Bun in this read-only review.

## Solution Assessment

No, not as written. The fix direction is right, but the Bun-specific branches should require the launcher `referrer` instead of accepting any matching relative specifier or message text.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Scope Bun direct misses to the launcher referrer:** `openclaw.mjs:63-75`
  - body: `isDirectModuleNotFoundError` is the guard that keeps transitive module-resolution failures from being treated as missing launcher output. The new Bun branches return true for any `ERR_MODULE_NOT_FOUND` with the same relative specifier or message, without checking Bun's `referrer`, so a nested import failure for that same specifier could be swallowed as the optional launcher miss.
