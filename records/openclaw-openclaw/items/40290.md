## Summary

Keep open: current main still blocks the intended `docker.env` credential-like keys, and this PR is not safe to merge as-is because it bypasses skill eligibility, drops existing sanitizer options, and lacks the required changelog entry. Earlier review comments correctly caught the `/root` denylist regression, and the latest PR commit appears to have addressed that specific issue.

## What This Changes

The PR adds a `forceAllowKeys` sanitizer path for sandbox Docker env vars, wires it into Docker create args, adds sandbox env validation code, and changes skill filtering so agent-listed skills can bypass eligibility checks.

## Best Possible Solution

Revise this into a narrow sandbox Docker env fix that force-allows admin-configured `docker.env` keys while preserving caller sanitizer options, keeps skill eligibility gates intact, adds regression tests, and records the user-facing fix in the changelog.

## Reproduction Assessment

Yes. Static source review gives a high-confidence path: current docs direct users to `sandbox.docker.env` for skill API keys, while current main sanitizes those keys through a credential blocklist; on the PR head, the changed skill filter and sanitizer call introduce the review blockers.

## Solution Assessment

No. The `docker.env` force-allow direction is appropriate, but this patch should preserve existing sanitizer options and leave `shouldIncludeSkill` as the prerequisite for all skill visibility.
