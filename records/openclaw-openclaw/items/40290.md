## Summary

Keep open: the branch targets a real documented Docker sandbox env bug, but the current PR still introduces security-sensitive regressions in skill visibility and sanitizer policy, lacks contributor real-behavior proof, and is currently conflicting.

## What This Changes

The PR adds a `forceAllowKeys` sanitizer option for configured Docker sandbox env vars and changes explicit skill filters to bypass eligibility checks.

## Best Possible Solution

Revise this into a narrow `docker.env` sanitizer exception that preserves `envSanitizationOptions`, keeps `shouldIncludeSkill` as the prerequisite for skill visibility, adds focused regression tests, and includes redacted real Docker sandbox proof.

## Reproduction Assessment

Yes, at source level. Current docs direct sandboxed skill API keys to `agents.defaults.sandbox.docker.env`, while current main sends that map through the sanitizer that blocks API-key-shaped names; I did not run a live Docker sandbox in this read-only review.

## Solution Assessment

No. The Docker `docker.env` force-allow direction is plausible, but the current PR should not bypass skill eligibility gates or discard existing sanitizer policy options.
