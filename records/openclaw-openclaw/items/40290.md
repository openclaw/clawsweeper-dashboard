## Summary

Keep open: current main still blocks credential-shaped keys in documented Docker sandbox env, so the core bug is real, but this PR still has blocking correctness/security findings and no contributor-owned real behavior proof. The related open PR at https://github.com/openclaw/openclaw/pull/44569 covers config `env.vars` forwarding, not this `docker.env` sanitizer exception.

## What This Changes

The branch adds `forceAllowKeys` to `sanitizeEnvVars`, applies it to Docker config env, adds a sandbox env validation helper, and changes explicit skill filtering to bypass eligibility checks.

## Best Possible Solution

Revise this into a narrow `docker.env` credential exception that preserves caller sanitizer options, keeps `shouldIncludeSkill` as the prerequisite gate, adds focused regression tests, and includes redacted Docker sandbox proof.

## Reproduction Assessment

Yes, at source level. Current main sends configured Docker env through the credential blocklist while the docs direct sandboxed skill API keys to `agents.defaults.sandbox.docker.env`; I did not run live Docker proof in this read-only review.

## Solution Assessment

No. The `docker.env` force-allow direction is plausible, but the current PR should preserve `envSanitizationOptions`, leave skill eligibility gates intact, and provide real sandbox proof before merge.

Merge the existing options through and scope the bypass to the documented Docker env credential exception.
