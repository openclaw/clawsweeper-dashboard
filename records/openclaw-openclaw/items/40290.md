## Summary

Keep open: current main still blocks the documented Docker sandbox env-var use case, but this PR is not mergeable as-is because it bypasses skill eligibility, drops existing sanitizer policy options, and lacks contributor real behavior proof.

## What This Changes

The PR adds a `forceAllowKeys` sanitizer option for configured Docker env vars and changes explicit skill filters to bypass eligibility checks.

## Best Possible Solution

Revise this into a narrow `docker.env` sanitizer exception that preserves caller sanitizer options, keeps `shouldIncludeSkill` as the prerequisite for skill visibility, adds focused regression tests, and includes redacted real Docker sandbox proof.

## Reproduction Assessment

Yes, at source level. Current main sends configured Docker env through the credential blocklist while docs direct sandboxed skill API keys to `agents.defaults.sandbox.docker.env`; I did not run a live Docker sandbox in this read-only review.

## Solution Assessment

No. The Docker `docker.env` force-allow direction is plausible, but this PR should preserve `envSanitizationOptions` and keep eligibility gates before explicit skill filtering.

## Review Findings

Overall correctness: patch is incorrect

- **[P1] Preserve skill eligibility before explicit filters:** `src/agents/skills/workspace.ts:79-81`
  - body: The explicit-name shortcut runs before `shouldIncludeSkill`, which is the gate for `skills.entries.<key>.enabled === false`, bundled allowlists, and runtime `requires` checks. Carry the existing options through and scope the bypass to the documented Docker env credential exception.
  - confidence: 0.88
