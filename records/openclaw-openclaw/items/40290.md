## Summary

Keep this PR open, but do not merge as-is. The sandbox env bypass addresses a real current-main gap, but the PR branch still weakens existing skill eligibility gates and drops caller-provided sanitizer policy when adding forced env keys.

## What This Changes

The PR adds `forceAllowKeys` to sandbox env sanitization and Docker create args, adds an env validation helper, and changes skill filtering so explicitly listed skills bypass eligibility checks.

## Best Possible Solution

Revise this into a narrow sandbox env fix: forced keys should apply only to resolved operator-configured `agents.*.sandbox.docker.env` keys, merge with existing sanitizer options, preserve `shouldIncludeSkill` as the prerequisite for agent skill allowlists, and add focused tests for both the env bypass and preserved skill gates.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence reproduction path: on the PR head, a disabled or runtime-ineligible skill listed in an agent allowlist bypasses `shouldIncludeSkill`, and a Docker create-args caller using `strictMode` or custom sanitizer patterns loses that policy.

## Solution Assessment

No. The env bypass direction is plausible, but this PR is not the best way to solve it because it mixes an unrelated skill policy bypass into sandbox env handling and replaces existing sanitizer options instead of extending them.
