## Summary

Keep open. Current main still blocks credential-like `agents.*.sandbox.docker.env` keys, so the PR targets a real sandbox env gap, but the current branch remains incorrect because it weakens skill eligibility gates and drops the existing sanitizer-options contract.

## What This Changes

The PR adds a `forceAllowKeys` sanitizer option for sandbox Docker env values, changes Docker create-arg env sanitization, adds an env validation helper, and lets explicit agent skill filters bypass skill eligibility.

## Best Possible Solution

Revise this into a narrow sandbox env fix: preserve `shouldIncludeSkill` as the prerequisite for all skill allowlists, merge any existing sanitizer options when adding forced Docker env keys, add focused tests for blocked/default/forced env behavior and preserved skill gates, and include a one-line changelog entry.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence path: on the PR head, a skill with `skills.entries.<name>.enabled: false` that remains in an agent skill filter would bypass `shouldIncludeSkill`, and any `buildSandboxCreateArgs` caller-provided sanitizer options are replaced by the forced-key object.

## Solution Assessment

No. The `docker.env` bypass direction is plausible, but this PR is not the best current solution because it mixes in an unrelated skill eligibility bypass and drops an existing sanitizer API contract instead of extending it.
