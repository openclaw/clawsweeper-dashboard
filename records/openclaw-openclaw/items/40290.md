## Summary

Keep open. Current main still blocks credential-like `sandbox.docker.env` values even though the docs direct sandboxed skill API keys there, but this PR is not safe to merge because it bypasses skill eligibility gates and drops existing sanitizer options.

## What This Changes

The PR adds a `forceAllowKeys` sandbox env sanitizer option, applies it to Docker config env values, adds an env validation helper, and lets explicit skill filters bypass eligibility checks.

## Best Possible Solution

Land a narrow sandbox env fix that force-allows only admin-configured Docker env keys, preserves existing sanitizer options, keeps skill eligibility gates intact, adds focused regression tests, and records the user-facing fix in the changelog.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main blocks `*_API_KEY` from `sandbox.docker.env`, and on the PR head a disabled or runtime-ineligible skill listed in `skillFilter` would be exposed while caller-provided sanitizer options are dropped.

## Solution Assessment

No. The `docker.env` bypass direction is plausible, but the patch mixes in an unrelated skill-gate policy bypass and replaces existing sanitizer options instead of extending them.
