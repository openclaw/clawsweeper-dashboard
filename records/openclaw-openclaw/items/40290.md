## Summary

Keep open. The underlying `docker.env` sanitizer bug is still valid on current main, but this PR still has merge-blocking issues: it bypasses skill eligibility gates, drops caller-provided sanitizer options, and lacks after-fix real behavior proof.

## What This Changes

The PR adds `forceAllowKeys` to sandbox env sanitization for admin-configured Docker env vars and changes agent skill filtering so explicit skill names bypass eligibility checks.

## Best Possible Solution

Revise this into a narrow sandbox Docker env fix that force-allows explicitly configured `docker.env` keys while preserving caller sanitizer options, leaving `shouldIncludeSkill` as the prerequisite for all skill visibility, and adding focused regression coverage.

## Reproduction Assessment

Yes, at source level. Current docs direct users to `agents.defaults.sandbox.docker.env` for sandboxed skill API keys, while current main passes that map through a sanitizer that blocks API-key-shaped names.

## Solution Assessment

No. The `docker.env` force-allow direction is appropriate, but this patch should merge existing sanitizer options and preserve skill eligibility gates instead of broadening skill visibility.
