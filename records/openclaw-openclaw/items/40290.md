## Summary

Keep open: the underlying Docker sandbox env problem is still source-reproducible on current main, but this PR is not safe to merge as written because it bypasses existing skill eligibility gates, weakens sanitizer option plumbing, and lacks after-fix real behavior proof.

## What This Changes

The PR adds a `forceAllowKeys` sanitizer option for admin-configured Docker env vars and changes explicit skill filters to bypass skill eligibility checks.

## Best Possible Solution

Revise this into a narrow Docker `docker.env` allowlist fix that preserves sanitizer extension points and keeps `shouldIncludeSkill` as the prerequisite for skill visibility, with focused regression coverage and redacted real sandbox proof.

## Reproduction Assessment

Yes, at source level. Current docs direct sandboxed skill API keys to `agents.defaults.sandbox.docker.env`, while current main sends that map through a sanitizer that blocks API-key-shaped names; I did not run a live Docker sandbox in this read-only review.

## Solution Assessment

No. The Docker `docker.env` force-allow direction is appropriate, but the PR should not bypass skill eligibility gates or discard existing sanitizer policy options.
