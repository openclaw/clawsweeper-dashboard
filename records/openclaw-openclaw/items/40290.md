## Summary

Keep this PR open: the reported sandbox `docker.env` problem is still present on current main, but the branch is not merge-ready because it drops current sandbox env-marker behavior, ignores existing sanitizer options, and re-enables skills that config explicitly disables. The adjacent open env-var work in #44569/#57427 is related but does not supersede this exact `docker.env` blocklist fix.

## What This Changes

The PR adds `forceAllowKeys` to sandbox env sanitization, uses it for Docker `cfg.env`, adds a sandbox env validator export, and makes explicit skill filters bypass runtime eligibility checks.

## Best Possible Solution

Rebase or replace the PR with a narrow sandbox fix that preserves current env-marker and sanitizer-option behavior, keeps `enabled:false` as an absolute skill disable, adds focused regression tests, and includes a changelog entry.

## Reproduction Assessment

Yes, this is source-reproducible on current main: sandbox Docker env is passed through the default sanitizer, and the default blocklist rejects `*_API_KEY` even though docs direct sandboxed skill API keys to `agents.defaults.sandbox.docker.env`. I did not run a live Docker repro in this read-only review.

## Solution Assessment

No, not as currently proposed. The `forceAllowKeys` direction is plausible for the documented `docker.env` path, but the branch needs to preserve existing sandbox env policy hooks and skill disable semantics before it is the narrow maintainable fix.
