## Summary

Keep open: the Docker `docker.env` sanitizer problem is source-reproducible and the PR points in a useful direction, but the current patch still broadens skill visibility past eligibility gates, drops caller-supplied sanitizer policy, and lacks external real behavior proof.

## What This Changes

The PR adds a `forceAllowKeys` sanitizer bypass for configured Docker sandbox env vars and changes explicit agent skill filters to bypass eligibility checks.

## Best Possible Solution

Revise this into a narrow Docker `docker.env` allowlist fix that preserves sanitizer extension points, keeps `shouldIncludeSkill` as the prerequisite for skill visibility, adds focused regression coverage, and includes redacted real sandbox proof.

## Reproduction Assessment

Yes, at source level. Current docs direct sandboxed skill API keys to `agents.defaults.sandbox.docker.env`, while current main sends that map through a sanitizer that blocks API-key-shaped names; I did not run a live Docker sandbox in this read-only review.

## Solution Assessment

No. The Docker `docker.env` force-allow direction is appropriate, but the PR should not bypass skill eligibility gates or replace existing sanitizer policy options.
