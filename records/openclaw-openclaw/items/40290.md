## Summary

Keep open: the core sandbox-env idea is still useful, and the latest PR commit addressed the earlier `/root` blocklist regression and strict-mode gap, but the current patch still introduces concrete skill-gating and sanitizer-option regressions that should block merge.

## What This Changes

The PR adds a `forceAllowKeys` sanitizer option for admin-configured sandbox Docker env vars, wires it into Docker create args, adds sandbox env validation code, and lets explicit skill filters bypass normal skill eligibility.

## Best Possible Solution

Revise this into a narrow sandbox Docker env fix that merges forced admin keys with existing sanitizer options, keeps `shouldIncludeSkill` as the prerequisite for all skill visibility, adds regression tests, and records the user-facing fix in the changelog.

## Reproduction Assessment

Yes. Static review gives a high-confidence path: on the PR head, a disabled or runtime-ineligible skill listed in an agent skill filter would be exposed, and any caller-provided `envSanitizationOptions` would be dropped by `buildSandboxCreateArgs`.

## Solution Assessment

No. The `docker.env` force-allow direction is plausible, but this patch is not the best solution because it mixes in a skill-gate bypass and regresses the sanitizer options contract instead of extending it.
