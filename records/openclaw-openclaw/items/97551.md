## Summary

Keep open for maintainer review: current main and the latest release still capture cloneable debug-proxy responses with an unbounded `clone().arrayBuffer()`, while this PR adds a bounded metadata-only path with sufficient real behavior proof. The remaining decision is whether maintainers accept the intentional 16 MiB diagnostics cutoff.

## What This Changes

Bounds debug proxy response body capture to 16 MiB, records over-cap responses as metadata-only, and adds focused proxy-capture runtime tests.

## Best Possible Solution

Land the bounded metadata-only capture behavior after maintainers accept the 16 MiB diagnostic cutoff and required checks pass.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main and v2026.6.10 call `response.clone().arrayBuffer()` without a byte cap during debug proxy capture. I did not execute an OOM repro in this read-only review, but the PR body includes after-fix real-session proof.

## Solution Assessment

Yes, this is the best current fix shape: the shared proxy-capture runtime covers global fetch, guarded fetch, and plugin capture callers. Reusing the media-core bounded reader is less suitable here because the checked dependency path tees cloned response bodies and the shared helper awaits stream cancellation on overflow.
