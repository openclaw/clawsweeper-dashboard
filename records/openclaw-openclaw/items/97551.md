## Summary

Keep open: current main and the latest release still capture cloneable debug-proxy responses with unbounded `response.clone().arrayBuffer()`, while this PR is focused, proof-backed, and has no blocking code-review findings. The remaining question is maintainer acceptance of the intentional 16 MiB diagnostics cutoff.

## What This Changes

The PR adds a 16 MiB cap to debug proxy response-body capture, records oversized bodies as metadata-only, and extends proxy-capture runtime tests.

## Best Possible Solution

Land the bounded metadata-only capture behavior after maintainers accept the fixed diagnostics cutoff and normal merge gates pass.

## Reproduction Assessment

Yes. Source inspection shows current main and v2026.6.10 call `response.clone().arrayBuffer()` without a byte cap during debug proxy response capture; I did not run an OOM repro in this read-only review.

## Solution Assessment

Yes. The shared proxy-capture runtime is the narrow owner boundary for global fetch and guarded-fetch capture, and the local bounded reader is justified because the shared media helper awaits cancellation on overflow.
