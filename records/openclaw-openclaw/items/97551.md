## What This Changes

The PR bounds debug proxy captured response bodies at 16 MiB, records oversized bodies as metadata-only, and adds focused proxy-capture runtime tests.

## Best Possible Solution

Land the bounded metadata-only capture behavior after maintainers accept the 16 MiB diagnostic cutoff and required checks pass.

## Reproduction Assessment

Yes from source inspection: current main captures cloneable HTTP responses by calling `response.clone().arrayBuffer()` without a byte cap, reachable from global and guarded fetch capture paths. I did not run a live OOM repro because this review is read-only, but the PR body includes after-fix real-session proof.

## Solution Assessment

Yes. The best fix is in the shared proxy-capture runtime because global fetch, guarded fetch, and plugin capture seams all converge there; the media-core bounded reader was checked and is a poorer fit for this tee-clone path because it awaits `reader.cancel()` on overflow.
