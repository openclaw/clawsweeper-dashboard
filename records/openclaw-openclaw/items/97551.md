## Summary

Keep open. The patch appears to address a real source-backed OOM risk in debug proxy capture, but this external PR still needs after-fix real behavior proof before merge, and maintainers should explicitly accept the new 16 MiB diagnostics cutoff.

## What This Changes

The PR bounds debug proxy captured response bodies at 16 MiB, records oversized bodies as metadata-only, and adds focused runtime tests for declared and chunked oversized responses.

## Best Possible Solution

Land a bounded metadata-only capture path after real debug-proxy proof is added, keeping the cap local and avoiding a new config surface unless maintainers choose a different cutoff.

## Reproduction Assessment

Yes from source inspection: current main reaches captureHttpExchange from debug proxy fetch paths and buffers cloneable responses with response.clone().arrayBuffer() without a byte cap. I did not run a live OOM reproduction because this review is read-only.

## Solution Assessment

Yes, the proposed owner boundary is the right one: cap the shared proxy-capture response body read while preserving request/response metadata. Reusing readResponseWithLimit would be riskier here because that helper awaits cancel on overflow, while this tee-clone path needs fire-and-forget cancellation.
