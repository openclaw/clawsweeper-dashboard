## Summary

Keep open; this focused bug fix is still necessary because current main and the latest release still buffer captured clone response bodies without a byte cap, while the PR patch, tests, and real-session proof are strong. The remaining maintainer decision is accepting the intentional 16 MiB diagnostics cutoff before normal merge gates pass.

## What This Changes

The PR adds a 16 MiB bounded reader for debug proxy response capture, records oversized bodies as metadata-only, and adds focused runtime tests.

## Best Possible Solution

Land the bounded metadata-only capture behavior after maintainers accept the 16 MiB diagnostic cutoff and required checks are green.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main and v2026.6.10 call response.clone().arrayBuffer() without a byte cap during debug proxy capture; I did not execute an OOM repro in this read-only review.

## Solution Assessment

Yes. This is the best current fix shape because the shared proxy-capture runtime covers global fetch, guarded fetch, and plugin capture callers, while the shared media-core reader is less suitable for this tee branch because it awaits cancel on overflow.
