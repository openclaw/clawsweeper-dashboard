## Summary

Keep open. Current main still has the compatibility path described in the report: runtime config normalization can migrate `claude-cli/*` default model and allowlist entries to canonical `anthropic/*`, while isolated cron validates `payload.model` against the effective allowlist and can reject the original legacy ref. The later #75854 alias report is related but does not fully cover this legacy runtime-prefix migration edge.

## What This Changes

The issue requests that `agents.defaults.models` entries using `claude-cli/*` remain visible and accepted by isolated cron `payload.model` validation after the 2026.4.29 runtime/model migration.

## Best Possible Solution

Add a backward-compatible allowlist alias path for legacy runtime model refs after runtime config normalization, with focused regression coverage proving `payload.model: "claude-cli/<model>"` remains accepted and still selects Claude CLI execution semantics.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to normalize a config whose `agents.defaults.model.primary` is `claude-cli/...` and whose allowlist contains both `claude-cli/*` and `anthropic/*`, then resolve an isolated cron `payload.model: "claude-cli/..."`; current tests already prove the legacy allowlist key is removed.

## Solution Assessment

Unclear because there is no patch yet. The narrow maintainable direction is to keep canonical runtime/model migration intact and preserve legacy refs as accepted aliases in the allowlist/catalog or cron resolver, rather than reverting the migration.
