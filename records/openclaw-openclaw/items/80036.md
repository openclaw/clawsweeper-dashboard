## Summary

Keep open. Current main still supports Chrome MCP existing-session profiles, but status readiness is transport-only while `tabs`, snapshot, screenshot, and evaluate still depend on page-level Chrome MCP calls that can fail or time out separately. The older closed macOS attach issue does not cover this later-stage failure because this report reaches `cdpReady` before page tools hang.

## What This Changes

The issue asks OpenClaw to make macOS Chrome MCP `profile=user` page operations work after attach readiness, or fail quickly with specific diagnostics.

## Best Possible Solution

Restore reliable `user` profile tab discovery and page operations for supported macOS Chrome sessions, and make status distinguish transport attach from page/tool readiness with actionable timeout diagnostics.

## Reproduction Assessment

No, not at end-to-end confidence from this read-only review. The source and tests clearly show the readiness-contract gap, and the reporter supplied concrete macOS timeout logs, but the macOS Chrome 148 page-tool hang still needs live verification.

## Solution Assessment

Not applicable because there is no proposed patch. The narrowest maintainable path is a live macOS reproduction first, then either a Chrome MCP page discovery/tool fix or an explicit status split between transport readiness and page readiness.
