## Summary

Keep open. Current main still separates Chrome MCP transport readiness from page/tool readiness, so the reported macOS case where `profile=user` shows ready while `tabs`, snapshot, screenshot, and evaluate time out remains plausible and unsolved; the older https://github.com/openclaw/openclaw/issues/65125 was an earlier attach/CDP failure closed by loopback fixes, not this later-stage page-tool timeout.

## What This Changes

Investigate macOS Chrome MCP existing-session user-profile readiness reporting success while tab, snapshot, screenshot, and evaluation tools time out.

## Best Possible Solution

The browser plugin should make `profile=user` either reliably complete page discovery, snapshot, screenshot, and evaluate on supported Chrome MCP sessions, or clearly report separate transport-ready, page-discovery, permission, and tool-timeout states.

## Reproduction Assessment

No high-confidence end-to-end macOS reproduction was established in this read-only review. There is a source-level reproduction of the readiness gap: current code and tests allow status to report ready from Chrome MCP transport availability while page reachability and page tools are separate calls.

## Solution Assessment

Not applicable because there is no proposed patch. The narrowest maintainable path is to live-reproduce on macOS Chrome 148, then fix Chrome MCP page discovery/tool execution or intentionally split status into transport readiness versus page readiness with actionable diagnostics.
