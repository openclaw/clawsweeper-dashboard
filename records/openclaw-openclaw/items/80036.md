## Summary

Keep open. Current main improved the misleading status surface by adding `pageReady`, but the central reported failure remains unresolved: Chrome MCP existing-session `profile=user` can still attach at transport level while page-level tools fail, and the merged follow-up explicitly left the underlying Chrome 148/user-profile page-tool timeout untested and unfixed.

## What This Changes

The issue reports that Chrome MCP existing-session `profile=user` on macOS reports transport readiness while page tools such as tabs, snapshot, screenshot, and evaluate time out.

## Best Possible Solution

Live-reproduce against macOS Chrome 148 using the real signed-in `user` profile, then either fix Chrome MCP page discovery/execution for existing sessions or make the status/CLI/tool errors explicitly guide users when page tools are blocked by Chrome permission-gated debugging or target selection.

## Reproduction Assessment

No high-confidence end-to-end reproduction was established in this review. Source inspection and reporter logs clearly identify the transport-ready/page-tool-timeout path, but the exact Chrome 148 signed-in user-profile hang still needs live macOS proof.

## Solution Assessment

Unclear because there is no proposed patch for the underlying page-tool failure. The merged `pageReady` change is a good diagnostic improvement, but the best fix likely needs live root-cause work in Chrome MCP page discovery or more actionable user-profile failure handling.
