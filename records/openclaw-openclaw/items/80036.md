## Summary

Keep open. Current main and the latest release still allow Chrome MCP existing-session `status` to report transport readiness without proving page reachability, while the docs define `tabs` and `snapshot` as part of the successful `user` profile smoke test; the related open status-timeout PR does not cover the reported signed-in macOS page-tool hangs.

## What This Changes

The issue reports that Chrome MCP existing-session `profile=user` on macOS reports ready while tab listing, snapshot, screenshot, and script evaluation all time out.

## Best Possible Solution

Fix the browser plugin so existing-session `user` profiles either reliably control the selected page on macOS Chrome 148 or clearly separate transport readiness from page/tool readiness with actionable diagnostics.

## Reproduction Assessment

No high-confidence end-to-end reproduction was established in this review. Source, docs, and tests do show the ready-status/page-reachability gap, and the reporter supplied concrete macOS v2026.5.7 logs for the page-tool timeouts.

## Solution Assessment

Not applicable because there is no proposed patch for this issue. The narrowest maintainable path is a live macOS Chrome 148 reproduction, then either fixing the Chrome MCP page discovery/tool path or changing the status contract to distinguish transport attach from page readiness.
