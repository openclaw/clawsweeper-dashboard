## Summary

Keep open. Current main still has a real readiness gap for Chrome MCP existing-session profiles: status is derived from transport attach, while the documented success path and affected operations require page-level Chrome MCP tools; the recent status-timeout fix in https://github.com/openclaw/openclaw/pull/77473 does not cover this reported page-tool timeout.

## What This Changes

The issue requests investigation and repair for macOS Chrome MCP `profile=user` sessions that report ready but time out on tabs, snapshot, screenshot, and script evaluation.

## Best Possible Solution

Live-reproduce on macOS Chrome 148 against current main, then fix the Chrome MCP page discovery/tool path or explicitly split transport status from page readiness with actionable diagnostics.

## Reproduction Assessment

No high-confidence end-to-end current-main reproduction yet. Source inspection shows the readiness gap and the reported tool paths, and the reporter supplied concrete macOS logs, but the exact Chrome 148 page-tool timeout still needs live proof.

## Solution Assessment

Unclear because there is no proposed patch. The recent status-probe fix in https://github.com/openclaw/openclaw/pull/77473 is useful but not sufficient for page-tool timeouts; the safer direction is live reproduction followed by a focused browser plugin fix or an explicit status/page-readiness contract change.
