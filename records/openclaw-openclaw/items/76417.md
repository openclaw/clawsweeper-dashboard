## Summary

Keep open: current main still starts Discord native typing from the reply-start/model-loop pipeline, while the requested early inbound cue remains missing and Telegram already has the precedent.

## What This Changes

Send a best-effort Discord native typing indicator immediately after an inbound message is accepted, before queueing and other pre-dispatch work.

## Best Possible Solution

Add a Discord-owned early native typing cue at the accepted inbound/queue boundary, with regression coverage and a changelog entry, while preserving existing typing-mode and keepalive semantics.

## Reproduction Assessment

Yes, from source inspection: accepted Discord messages are queued before processDiscordMessage runs, and native typing is only started later through the reply-start callback. I did not run a live Discord bot in this read-only review.

## Solution Assessment

Yes: mirroring Telegram's fire-and-forget early cue inside the Discord plugin is the narrow maintainable fix, without changing global typing-mode semantics or adding config.

## Repair Work Prompt

Fix the Discord typing indicator delay reported in https://github.com/openclaw/openclaw/issues/76417 by sending a best-effort native typing cue after Discord preflight accepts and routes an inbound message and before enqueueing `processDiscordMessage`. Keep the change Discord-owned; do not change global `typingMode` semantics, add config, or replace the existing reply-start keepalive/cleanup path. Skip the early cue for preflight-dropped, unauthorized, bot-authored, replay-aborted, and debounced-away messages. Reuse existing Discord typing/rest plumbing or a small local helper; failures must be fire-and-forget and verbose-log only. Add focused regression coverage in the Discord message-handler queue tests, keep existing process/typing tests green, and add a single-line `CHANGELOG.md` entry because this is user-facing. Stop if the fix would require a core/plugin SDK change or new product policy.
