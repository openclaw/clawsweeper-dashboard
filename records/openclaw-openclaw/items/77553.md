## Summary

Keep open: the PR targets a valid session_status bug, but the current patch still lets stored session entry model overrides win in the real status renderer, and the attached after-fix proof is only test output.

## What This Changes

The PR adds a `runtimeModelOverride` option to `session_status`, passes `${modelProvider}/${modelId}` from `createOpenClawTools`, adds two focused mocked tests, and updates `CHANGELOG.md`.

## Best Possible Solution

Thread the active current-run provider/model into the actual status renderer inputs for semantic current requests, keep explicit non-current lookups unchanged, and prove the heartbeat output after the fix.

## Reproduction Assessment

Yes, at source level. Current main resolves `heartbeat.model` into an active provider/model before the run and passes that identity near tool construction, while `session_status` still renders from stored session/default model state without a current-run override path.

## Solution Assessment

No. Passing the active model into tool construction is the right direction, but this patch applies it only as `primaryModelLabelOverride`; the production renderer still gives stored `sessionEntry` model overrides precedence.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Make runtime overrides beat stored entry models:** `src/agents/tools/session-status-tool.ts:626-629`
  - body: When the current session has a stored `providerOverride` or `modelOverride`, this patch only changes `primaryModelLabelOverride`. `buildStatusMessage` still selects `sessionEntry.providerOverride`/`modelOverride` before `agent.model.primary`, so the heartbeat case can keep rendering the stale stored model instead of the active runtime model. Apply the runtime override at the renderer's selected provider/model or session-entry boundary and cover that production path.
  - confidence: 0.87
