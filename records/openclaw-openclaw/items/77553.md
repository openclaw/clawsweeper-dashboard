## Summary

Keep open: this PR targets a valid session_status bug, but the current patch still lets stored session model overrides win in the production status renderer and it lacks required real behavior proof from an actual heartbeat run.

## What This Changes

The PR adds a runtimeModelOverride option to session_status, threads the active provider/model from createOpenClawTools, adds two focused tests, and updates CHANGELOG.md.

## Best Possible Solution

Thread the active runtime provider/model through the status renderer boundary for semantic current requests, keep explicit non-current lookups unchanged, and prove the real heartbeat output after the fix.

## Reproduction Assessment

Yes, at source level. Current main resolves heartbeat.model into an active provider/model before the run, while session_status builds its displayed model from stored session entry/default status state without an active-run override path.

## Solution Assessment

No. Passing the active model into tool construction is the right direction, but the patch applies it only to primaryModelLabelOverride while buildStatusMessage still gives stored sessionEntry model overrides precedence; the fix should update the real renderer inputs and cover that production path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Make runtime overrides beat stored entry models:** `src/agents/tools/session-status-tool.ts:626-629`
  - body: When a current session has a stored providerOverride or modelOverride, this patch only changes primaryModelLabelOverride. buildStatusMessage still selects sessionEntry.providerOverride/modelOverride before agent.model.primary, so the heartbeat case can keep rendering the stale stored model instead of the active runtime model. Apply the runtime override at the status entry/provider/model boundary, or teach the renderer an explicit current-run override, and cover the production renderer path.
  - confidence: 0.87
