## Summary

Keep open: the PR targets a valid bug and threads the active run model in the right general direction, but the patch is not sufficient because the real status renderer can still prefer stored session overrides over the new runtime label.

## What This Changes

This PR adds a `runtimeModelOverride` option to the session status tool, threads `options.modelProvider/modelId` into it, adds two focused session-status tests, and updates `CHANGELOG.md`.

## Best Possible Solution

Make the active runtime provider/model supersede stored/default display state only for semantic `current` requests, preserve explicit session lookup behavior, and cover the production status-renderer precedence.

## Reproduction Assessment

Yes, at source level. Current `main` resolves `heartbeat.model` into the active run provider/model, but `session_status` does not receive that runtime model and the renderer can select stored session overrides instead.

## Solution Assessment

No for the current patch. The proposed data flow is narrow, but the override must reach the renderer or sanitized entry state because `buildStatusMessage` still prioritizes stored `sessionEntry` model fields.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Make runtime overrides beat stored entry models:** `src/agents/tools/session-status-tool.ts:626-629`
  - body: When the current session has a stored `providerOverride` or `modelOverride`, this change only passes the runtime model as `primaryModelLabelOverride`. `buildStatusText` forwards that as `agent.model.primary`, but `buildStatusMessage` still selects `sessionEntry.providerOverride` and `sessionEntry.modelOverride` first, so the stale stored model can keep rendering instead of the active heartbeat model. Apply the runtime override at the renderer/entry boundary or otherwise prevent stored entry overrides from winning for semantic `current` requests.
  - confidence: 0.87
