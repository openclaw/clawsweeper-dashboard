## Summary

Keep open. The bug is source-reproducible and the PR is directionally narrow, but the current patch has a renderer-precedence blocker: it threads a primary label while still passing a session entry whose stored model overrides can win in the real status renderer.

## What This Changes

The PR adds a runtime model override option to `session_status`, threads the active provider/model from `createOpenClawTools`, adds current/non-current coverage, and updates the changelog.

## Best Possible Solution

Make the active runtime provider/model supersede stored/default display state only for semantic current requests, keep explicit session lookups unchanged, and cover the real status-renderer precedence.

## Reproduction Assessment

Yes, at source level. Current main resolves `heartbeat.model` into the active run provider/model, but `session_status` has no active-run model input and derives the displayed card from session entry/config instead.

## Solution Assessment

No for the current patch. The proposed boundary is narrow, but the override must reach or sanitize the real renderer state because `buildStatusMessage` prioritizes stored session model overrides over `agent.model.primary`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Make runtime overrides beat stored entry models:** `src/agents/tools/session-status-tool.ts:631-633`
  - body: When a current session has a stored `providerOverride` or `modelOverride`, this override is only passed as `primaryModelLabelOverride`. `buildStatusText` forwards that as `agent.model.primary`, but `buildStatusMessage` still selects `sessionEntry.modelOverride` first, so the stale-entry case can keep rendering the stored model instead of the active heartbeat model. Apply the runtime override at the entry/renderer boundary, or otherwise ensure stored entry overrides cannot win for semantic `current` requests.
  - confidence: 0.87
