## Summary

The prior ClawSweeper finding still applies after the author’s rebase-only update: the fix threads an active model label, but the production renderer can still prefer the stored session override and show the stale model.

## What This Changes

The PR adds a `runtimeModelOverride` option to `session_status`, threads `options.modelProvider/options.modelId` from OpenClaw tool construction, adds focused tests, and records a changelog fix for #77493.

## Best Possible Solution

Make the active runtime provider/model override the final rendered status model only for semantic `current` requests, while explicit session lookups continue to report stored session intent.

## Reproduction Assessment

Yes, at source level. Current main resolves `heartbeat.model` into the active reply provider/model, but `session_status({"sessionKey":"current"})` has no active-run model input and builds its status display from stored session/config state.

## Solution Assessment

No for the posted patch. Threading the runtime label is the right narrow direction, but it must also override or sanitize the `sessionEntry` state consumed by `buildStatusMessage`, because that renderer currently prefers stored entry overrides over `agent.model.primary`.

## Review Findings
