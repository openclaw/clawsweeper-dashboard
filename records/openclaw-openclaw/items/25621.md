## Summary

Keep open: current main still throws for unsupported Gateway service platforms before either read-only status path can degrade, and the related implementation PR at https://github.com/openclaw/openclaw/pull/25690 is closed unmerged.

## What This Changes

The issue requests that OpenBSD and other unsupported platforms stop crashing `openclaw status` when Gateway service management is unavailable.

## Best Possible Solution

Add a conservative unsupported-platform Gateway service adapter or status-only guard so read-only status reports unknown/not-installed while lifecycle operations reject clearly with manual-start guidance.

## Reproduction Assessment

Yes, source-level with high confidence. On an unsupported `process.platform` such as `openbsd` or `aix`, `resolveGatewayService()` throws and both status paths call it before their fallback handlers can run.

## Solution Assessment

Yes for the status crash. A narrow unsupported-platform adapter or status-only guard is the maintainable fix; full OpenBSD service-manager support and generic shell template generation should be separate follow-up work.

Fix https://github.com/openclaw/openclaw/issues/25621. On unsupported `process.platform` values such as `openbsd` or `aix`, `openclaw status` and `openclaw gateway status` must not throw from `resolveGatewayService()` before status output can render. Implement either a conservative unsupported-platform `GatewayService` adapter or focused status-only guard: read-only methods should report safe unknown/not-installed state, while lifecycle/mutating operations should reject clearly with manual-start guidance. Do not add OpenBSD service-manager support, generated shell service templates, broad Android Gateway hosting, or new config knobs.
