## Summary

Keep open: current main still throws for unsupported Gateway service platforms before the read-only status fallback paths can render, and the related implementation PR was closed unmerged.

## What This Changes

OpenBSD users want `openclaw status` and related Gateway status commands to render diagnostic output instead of crashing when managed Gateway service install is unsupported.

## Best Possible Solution

Add a conservative unsupported-platform Gateway service adapter or status-only guard so read-only status reports unknown/not-installed while lifecycle/install operations reject with clear manual-start guidance.

## Reproduction Assessment

Yes, source-level with high confidence. On an unsupported `process.platform` such as `openbsd` or `aix`, `resolveGatewayService()` throws, and both status paths call it before their fallback handlers can run.

## Solution Assessment

Yes for the crash. A narrow unsupported-platform adapter or status-only guard is the maintainable fix; full OpenBSD service-manager support or generic shell templates should be separate follow-up work.

## Repair Work Prompt

Fix the unsupported-platform Gateway service status crash for OpenBSD and other platforms outside `darwin`, `linux`, and `win32`. Current main throws in `resolveGatewayService()` before `openclaw status` and `openclaw gateway status` can render degraded diagnostics. Keep the fix narrow: either return a conservative unsupported-platform `GatewayService` adapter whose read-only methods report safe unknown/not-installed state while mutating lifecycle methods reject clearly with manual-start guidance, or guard status-only call sites before resolving managed service adapters. Do not add full OpenBSD service-manager support, broad shell-template generation, dependency changes, or new config flags. Add regression coverage for both top-level status and gateway status on an unsupported platform, update the resolver test that currently expects the throw, and add a user-facing changelog entry.
