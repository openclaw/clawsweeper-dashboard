## Summary

Keep open: current main and the latest release still throw for unsupported Gateway service platforms before the read-only status fallback paths can render. The linked implementation PR was closed unmerged, so the narrow status crash remains actionable.

## What This Changes

Make `openclaw status` and `openclaw gateway status` usable on OpenBSD and other unsupported platforms instead of crashing with a Gateway service install error.

## Best Possible Solution

Add a conservative unsupported-platform Gateway service adapter or status-only guard so read-only status reports unknown/not-installed while lifecycle operations reject with clear manual-start guidance.

## Solution Assessment

Yes for the crash. A narrow unsupported-platform adapter or status-only guard is the maintainable fix; full OpenBSD service-manager support and generated shell templates should be separate follow-up work.

## Repair Work Prompt

Fix the unsupported Gateway service platform status crash tracked by https://github.com/openclaw/openclaw/issues/25621. Current `resolveGatewayService()` throws for platforms outside `darwin`, `linux`, and `win32`, and both `openclaw status` plus `openclaw gateway status` resolve it before their degraded read-only status paths can run. Implement either a conservative unsupported-platform `GatewayService` adapter whose read-only methods report safe unknown/not-installed state and whose lifecycle methods reject clearly, or a status-only guard before resolving managed service adapters. Do not add OpenBSD/FreeBSD service-manager support, Android/Termux support, new config, release packaging, or generated generic shell scripts in this PR. Add regression coverage for unsupported `process.platform` on both status paths, keep supported macOS/systemd/schtasks behavior covered, and add a user-facing changelog fix entry if behavior changes are shipped.
