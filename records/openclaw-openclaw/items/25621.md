## Summary

Keep open. Current main still throws for unsupported Gateway service platforms before the read-only status degradation paths can run, and the related implementation PR was closed unmerged, so the OpenBSD status crash remains an actionable source fix.

## What This Changes

The issue asks for `openclaw status` and related Gateway status commands to work on OpenBSD or other unsupported service platforms, with clear manual service guidance instead of a stack trace.

## Best Possible Solution

Add a conservative unsupported-platform Gateway service adapter, or guard the status-only call sites before resolving managed adapters, so read-only status reports safe unknown/not-installed state while lifecycle/install operations reject with clear unsupported-platform manual-start guidance.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to set `process.platform` to an unsupported value such as `aix` or OpenBSD and call `resolveGatewayService()`; the existing unit test asserts the throw, and both status paths call that resolver before their degradation handlers.

## Solution Assessment

Yes. A narrow unsupported-platform adapter or status-only guard is the best maintainable fix; implementing OpenBSD rc.d service management would be broader than the reported status crash and should be a separate design.
