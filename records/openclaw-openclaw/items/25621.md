## Summary

Keep open: current main and v2026.5.20 still throw for unsupported Gateway service platforms before read-only status fallback paths can render, and the linked focused fix PR was closed unmerged.

## What This Changes

Make read-only Gateway status commands degrade gracefully on OpenBSD and other unsupported platforms instead of crashing during managed service resolution.

## Best Possible Solution

Add a conservative unsupported-platform Gateway service adapter or a status-only guard so read-only status reports unknown/not-installed while lifecycle operations reject clearly.

## Reproduction Assessment

Yes. Source inspection on current main shows unsupported `process.platform` values such as `openbsd` or `aix` throw in `resolveGatewayService()`, and both status paths call it before fallback handlers can run.

## Solution Assessment

Yes for the status crash. A narrow unsupported-platform adapter or status-only guard is the maintainable fix; full OpenBSD service-manager support or generic shell-template generation should remain separate follow-up work.
