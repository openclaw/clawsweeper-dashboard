## Summary

Keep open: current `origin/main` still throws for unsupported Gateway service platforms before either `openclaw status` or `openclaw gateway status` can render degraded read-only diagnostics, and the linked implementation PR is closed unmerged.

## What This Changes

The issue asks for status commands to avoid crashing on OpenBSD and other unsupported service platforms, with useful manual-service guidance instead of a stack trace.

## Best Possible Solution

Add a conservative unsupported-platform Gateway service adapter or status-only guard so read-only status reports unknown/not-installed while lifecycle operations reject clearly with manual-start guidance.

## Reproduction Assessment

Yes, source-level with high confidence. On an unsupported `process.platform` such as `openbsd` or `aix`, `resolveGatewayService()` throws and both status paths call it before their fallback handlers can run.

## Solution Assessment

Yes for the status crash. A narrow unsupported-platform adapter or status-only guard is the maintainable fix; full OpenBSD service-manager support and generated shell templates should be separate follow-up work.

On unsupported managed-service platforms such as OpenBSD/AIX, `openclaw status` and `openclaw gateway status` must not crash before rendering read-only diagnostics. Add either a conservative unsupported-platform `GatewayService` adapter whose read-only methods report safe unknown/not-installed state and whose mutating methods reject clearly with manual-start guidance, or guard the status-only call sites before resolving managed service adapters.
