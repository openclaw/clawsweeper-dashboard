## Summary

Keep open: current main and the latest release still throw for unsupported Gateway service platforms before the read-only status fallback paths can run, and the related implementation PR was closed unmerged.

## What This Changes

The issue asks for OpenBSD and other unsupported service platforms to get non-crashing status output plus clearer manual service-start guidance instead of an unsupported-platform stack trace.

## Best Possible Solution

Add a conservative unsupported-platform Gateway service adapter, or guard status-only call sites before resolving managed adapters, so status reports safe unknown/not-installed state while lifecycle/install commands reject with clear manual-start guidance.

## Reproduction Assessment

Yes, source-level. Set `process.platform` to an unsupported value such as `aix` or OpenBSD and call `resolveGatewayService()`; the current unit test asserts the throw, and both status paths call that resolver before their fallback handlers.

## Solution Assessment

Yes for the status crash. A narrow unsupported-platform adapter or status-only guard is the maintainable fix; full OpenBSD rc.d management or generated shell templates should be separate follow-up work if maintainers want it.

Keep the change narrow: make `openclaw status` and `openclaw gateway status` work when `process.platform` is not `darwin`, `linux`, or `win32` by either adding a conservative unsupported-platform `GatewayService` adapter or guarding status-only call sites before resolving managed adapters.
