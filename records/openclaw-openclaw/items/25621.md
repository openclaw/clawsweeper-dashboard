## Summary

Keep open: current main and v2026.5.7 still throw for unsupported Gateway service platforms before read-only status diagnostics can degrade, and the related implementation PR is closed unmerged.

## What This Changes

OpenBSD and other unsupported-platform users want `openclaw status` and related read-only Gateway diagnostics to work instead of crashing on Gateway service resolution.

## Best Possible Solution

Add a conservative unsupported-platform Gateway service adapter or status-only guard so read-only status reports unknown/not-installed while lifecycle operations reject clearly; keep generic shell templates or OpenBSD service-manager support separate.

## Reproduction Assessment

Yes, source-level with high confidence. On unsupported `process.platform` values such as `openbsd` or `aix`, `resolveGatewayService()` throws and both status paths call it before their fallback handlers can run.

## Solution Assessment

Yes for the status crash. A narrow unsupported-platform adapter or status-only guard is the maintainable fix; full OpenBSD service support or generated shell templates should be separate follow-up work.
