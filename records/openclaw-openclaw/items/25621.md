## Summary

Keep open. Current main still throws before both status paths can render degraded output on OpenBSD or any platform outside macOS, Linux, and Windows, and the related implementation PR was closed unmerged. The issue remains narrow and actionable.

## What This Changes

The issue asks for `openclaw status` and related gateway status commands to avoid crashing on OpenBSD or other unsupported service platforms and to provide a manual-service fallback path.

## Best Possible Solution

Add a narrow unsupported-platform service/status behavior: status-only paths should render safe unknown/not-installed service state on unsupported platforms, while install/restart/stop/uninstall continue to fail with clear manual-run guidance. Do not add OpenBSD service-manager support unless that is designed separately.

## Repair Work Prompt

Current main throws from `resolveGatewayService()` on platforms outside `darwin`, `linux`, and `win32`, which prevents `openclaw status` and `openclaw gateway status` from rendering degraded diagnostics. Implement a narrow fix: either return a conservative unsupported-platform `GatewayService` adapter whose read-only methods report safe unknown/not-installed state while mutating methods reject with clear manual-run guidance, or guard the status-only call sites before resolving managed service adapters. Do not add OpenBSD service-manager support, do not change macOS/Linux/Windows service behavior, and do not broaden lifecycle permissions.
