Keep open: current main and the latest release still throw for unsupported Gateway service platforms before read-only status fallback paths can render, and the related implementation PR was closed unmerged.

## What This Changes

The issue asks for unsupported Gateway service platforms such as OpenBSD to let read-only status commands work instead of crashing, with clearer manual lifecycle guidance.

## Best Possible Solution

Add a conservative unsupported-platform Gateway service adapter or status-only guard so read-only status reports unknown/not-installed while lifecycle operations reject with clear manual-start guidance.

## Solution Assessment

Yes for the crash. A narrow unsupported-platform adapter or status-only guard is the maintainable fix; full OpenBSD service-manager support and generated shell templates should be separate follow-up work.

Fix https://github.com/openclaw/openclaw/issues/25621 with a narrow Gateway service/status change. Current main throws from `resolveGatewayService()` on unsupported platforms like OpenBSD, causing `openclaw status` and `openclaw gateway status` to fail before read-only fallback paths can render. Implement either a conservative unsupported-platform `GatewayService` adapter whose read-only methods report safe unknown/not-installed state while mutating lifecycle methods reject with a clear manual-start message, or guard status-only call sites before resolving managed service adapters. Cover both `openclaw status` and `openclaw gateway status` on an unsupported `process.platform` such as `aix`/`openbsd`. Preserve existing macOS launchd, Linux systemd, and Windows schtasks behavior.
