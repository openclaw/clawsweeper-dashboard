## Summary

Keep open: current main and the latest release still throw on unsupported Gateway service platforms before status fallback paths can render, and the related implementation PR #25690 was closed unmerged.

## What This Changes

Make `openclaw status` and `openclaw gateway status` work on OpenBSD and other unsupported service-manager platforms instead of crashing during Gateway service resolution.

## Best Possible Solution

Add a conservative unsupported-platform Gateway service adapter, or guard status-only call sites before resolving managed adapters, so read-only status reports unknown/not-installed while install/lifecycle commands reject with clear manual-start guidance.

## Reproduction Assessment

Yes, source-level. Set `process.platform` to an unsupported value such as `aix` or OpenBSD and call `resolveGatewayService()` or either status path; current code throws before status fallback handlers run.

## Solution Assessment

Yes for the status crash. A narrow unsupported-platform adapter or status-only guard is the maintainable fix; full OpenBSD rc.d management or generated shell templates should be separate follow-up work.

## Repair Work Prompt

Fix the unsupported-platform Gateway service status crash reported in https://github.com/openclaw/openclaw/issues/25621. On platforms outside darwin/linux/win32, `openclaw status` and `openclaw gateway status` must not fail before rendering status output. Prefer either a conservative unsupported-platform `GatewayService` adapter whose read-only methods return safe unknown/not-installed state and whose mutating methods reject with a clear manual-start message, or focused status-only guards before resolving managed service adapters. Do not add OpenBSD rc.d/service-manager support, generic shell-template generation, new config options, or broad service-manager abstractions in this repair. Add regression coverage for `resolveGatewayService()` and both status paths under an unsupported `process.platform`; update `CHANGELOG.md` because the behavior is user-facing.
