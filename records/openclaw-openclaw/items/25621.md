## Summary

Keep open: current main still throws for unsupported Gateway service platforms before the status fallback paths can render, and the related implementation PR is closed unmerged.

## What This Changes

Request `openclaw status` and related Gateway status commands to work on OpenBSD or other unsupported service platforms, with clearer manual service guidance.

## Best Possible Solution

Add a conservative unsupported-platform Gateway service adapter or status-only guard so read-only status reports unknown/not-installed while mutating service lifecycle commands reject with clear manual-start guidance.

## Reproduction Assessment

Yes, source-level with high confidence. On an unsupported `process.platform` such as `openbsd` or `aix`, `resolveGatewayService()` throws, and both status paths call it before their fallback handlers can run.

## Solution Assessment

Yes for the crash. A narrow unsupported-platform adapter or status-only guard is the maintainable fix; full OpenBSD service-manager support or generic shell templates should be separate follow-up work.

Implement a conservative unsupported-platform Gateway service adapter, or guard the status-only call sites before resolving managed adapters.
