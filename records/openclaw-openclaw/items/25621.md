## Summary

Keep open: current main still throws for unsupported Gateway service platforms before the read-only status fallback can run, and the related implementation PR was closed unmerged.

## What This Changes

Make `openclaw status` and related Gateway status diagnostics work on OpenBSD or other unsupported service platforms instead of crashing.

## Best Possible Solution

Add a conservative unsupported-platform Gateway service adapter or status-only guard so read-only status reports unknown/not-installed while lifecycle operations reject clearly.

## Reproduction Assessment

Yes, source-level with high confidence. On unsupported `process.platform` values such as `openbsd` or `aix`, `resolveGatewayService()` throws and both status paths call it before their fallback handlers can run.

## Solution Assessment

Yes for the status crash: a narrow unsupported-platform adapter or status-only guard is the maintainable fix. Full OpenBSD service-manager support or generated shell templates should be separate follow-up work.

## Repair Work Prompt

Fix this issue by making read-only Gateway status paths tolerate unsupported service platforms such as OpenBSD without adding full OpenBSD service-manager support. On unsupported `process.platform`, `openclaw status` and `openclaw gateway status` should render degraded/unknown service state instead of throwing `Gateway service install not supported on ...`; mutating lifecycle operations such as install, uninstall, stop, restart, and stage must still reject with a clear unsupported-platform error. Prefer either a conservative unsupported-platform `GatewayService` adapter or a status-only guard, and keep supported darwin/linux/win32 behavior unchanged. Add or update regression coverage for `resolveGatewayService()`, `getDaemonStatusSummary()`, and `gatherDaemonStatus()` on an unsupported platform. Add a single user-facing changelog Fixes entry if repo policy still requires one. Do not implement a generic shell service template in this repair; that is separate product work.
