## What This Changes

The issue requests that `openclaw status` and related diagnostics degrade gracefully on OpenBSD instead of throwing from Gateway service resolution.

## Best Possible Solution

Add a conservative unsupported-platform Gateway service adapter or status-only guard so read-only status reports unknown/not-installed while lifecycle operations reject clearly.

## Reproduction Assessment

Yes, at source level. Current main throws for unsupported `process.platform` values such as OpenBSD or AIX in `resolveGatewayService()`, and both status paths call that resolver before their fallback handlers can run.

## Solution Assessment

Yes for the status crash. A narrow unsupported-platform adapter or status-only guard is the maintainable fix; full OpenBSD service-manager support or generic shell-template generation should be separate follow-up work.

On unsupported `process.platform` values such as OpenBSD/AIX, `openclaw status` and `openclaw gateway status` must not throw from `resolveGatewayService()` before diagnostic output can render. Either add a conservative unsupported-platform `GatewayService` adapter whose read-only methods report safe unknown/not-installed state while lifecycle/mutating methods reject with a clear unsupported-platform message, or guard the status-only call sites before resolving managed service adapters.
