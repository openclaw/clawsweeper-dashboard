## Summary

Keep open: the issue has the protected `maintainer` label, and current `main` still enforces the reported four-slot `oc://` limit and scalar-only JSONC leaf writes. The remaining work is a maintainer API/UX decision about the public `oc://` grammar, JSON object replacement semantics, and SecretRef migration guidance.

## What This Changes

The issue asks for `openclaw path` to support deeper `openclaw.json` paths and object-valued replacements so common nested config and SecretRef edits do not require hand-editing JSONC.

## Best Possible Solution

Make a maintainer decision on the public `oc://` contract, then either add scoped deep `openclaw.json` and JSON object replacement support with regression tests or improve docs/errors to point users to dotted `oc://` subsegments and `openclaw config set`/`config patch` for shape-changing config edits.

## Reproduction Assessment

Yes from source inspection: current `parseOcPath` still rejects more than four slash slots, and `setOcPath` still refuses object/array leaf replacement through JSONC scalar coercion. I did not run the CLI because this review is read-only, but current tests and shipped `v2026.5.22` source match the reported behavior.

## Solution Assessment

Unclear: raising the global depth cap is not obviously the best fix because current docs and resolver code already support dotted JSONC subsegments. The safer path is a maintainer choice between a scoped grammar change, a shape-changing JSON value verb/flag, and better docs/error hints for existing config commands.
