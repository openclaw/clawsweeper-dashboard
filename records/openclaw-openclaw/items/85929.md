## Summary

This issue should stay open: it has the protected `maintainer` label, and current `main` still rejects the reported slash-deep `oc://openclaw.json/...` shape before filesystem access. Source and docs also show a supported dotted-subsegment JSONC path model, so the permanent fix is a maintainer API/UX choice rather than an automatic close or repair.

## What This Changes

Requests `openclaw path` support or clearer guidance for editing nested `openclaw.json` keys beyond the four slash-slot `oc://` cap.

## Best Possible Solution

Keep the four-slot contract unless maintainers explicitly want a grammar change; add `openclaw.json` nested-path examples and an `OC_PATH_TOO_DEEP` hint that points users to dotted JSONC subsegments or `openclaw config set`.

## Reproduction Assessment

Yes from source inspection: current `parseOcPath` rejects more than four slash slots, and the reported path has seven slots. I did not execute the CLI in this read-only review, but current tests assert the same `OC_PATH_TOO_DEEP` behavior.

## Solution Assessment

No, raising the global depth cap is not clearly the best fix yet because current docs and JSONC code already support deep paths through dotted subsegments. The safer path is maintainer choice plus docs/error guidance, or a deliberately scoped `openclaw.json` syntax change with tests.
