## What This Changes

The issue requests making `openclaw path set/get` work for nested `openclaw.json` edits by relaxing the four-slot `oc://` depth cap and supporting JSON object leaf replacement or clearer documented alternatives.

## Best Possible Solution

Make an explicit maintainer choice for the `openclaw path` UX: either keep the four-slot grammar with better nested JSONC examples and `OC_PATH_TOO_DEEP` hints, or add a scoped JSON value/replacement mode with focused parser, SecretRef, docs, and regression coverage.

## Reproduction Assessment

Yes. Source inspection shows `parseOcPath` rejects more than four slash slots and current tests assert `OC_PATH_TOO_DEEP`; the JSONC setter also rejects object/array replacement through scalar leaf coercion.

## Solution Assessment

Unclear. Raising the global depth cap alone is not clearly the best fix because current docs and code already support dotted JSONC subsegments; the object-value ask likely needs a deliberate `--json`/replacement mode or a documented fallback.
