## Summary

This issue should stay open: it has the protected `maintainer` label, current `main` and the latest release still have the reported `OC_PATH_TOO_DEEP` and object-value limits, and the right fix is a maintainer decision about the public `oc://` grammar and JSONC value-replacement contract.

## What This Changes

Issue asks `openclaw path` to support deeply nested `openclaw.json` paths and JSON object leaf replacements, or at least document and hint supported fallbacks.

## Best Possible Solution

Maintainers should choose the path CLI contract: either keep the four-slot grammar and improve docs/error hints for dotted JSONC/config commands, or add a deliberately scoped deep-JSONC and JSON-object replacement mode with focused tests.

## Reproduction Assessment

Yes from source inspection: current `parseOcPath` rejects more than four slash slots, and `setStructuredLeaf` rejects object/array leaf replacement by design. I did not execute the CLI in this read-only review, but current tests and release-tag source match the reported failures.

## Solution Assessment

Unclear: raising the global depth cap alone is not clearly the best fix because JSONC dotted subsegments already support deep nesting inside the documented grammar. The object-value ask is a separate CLI/API contract choice that likely needs a scoped flag or sibling verb plus SecretRef-focused tests.
