## Summary

Keep this open: the protected `maintainer` label requires explicit maintainer handling, and current `main` plus the latest release still have the reported `oc://` slash-depth cap and the JSONC object/array leaf replacement limitation. Existing dotted JSONC subsegments and `openclaw config set` provide partial workarounds, but they do not fully answer the requested byte-preserving `openclaw path` workflow.

## What This Changes

The issue requests better `openclaw path` support for deeply nested `openclaw.json` paths, JSON object leaf replacements, documentation, and more actionable `OC_PATH_TOO_DEEP` guidance.

## Best Possible Solution

Have maintainers choose the public `oc://` contract: either keep four slash slots and add stronger docs/error hints for dotted JSONC paths, or add a scoped deep JSONC/openclaw.json mode plus explicit JSON object replacement support with tests and docs.

## Reproduction Assessment

Yes, from source inspection. `parseOcPath` rejects more than four slash slots, tests assert `OC_PATH_TOO_DEEP`, and JSONC leaf replacement currently refuses object/array shape changes through `coerceJsoncLeaf`.

## Solution Assessment

No, raising the global depth cap alone is not clearly the best fix because current JSONC paths already support dotted subsegments and the later comment adds an object-value mode request. The maintainable answer needs a deliberate CLI/API contract choice plus focused docs and tests.
