## Summary

Keep this PR open: current main still has no EasyRunner platform guide, but the proposed docs are not merge-ready because they conflict with current gateway security and container deployment contracts.

## What This Changes

The PR adds a new EasyRunner platform deployment guide and registers it in the docs sidebar and platform index.

## Best Possible Solution

Revise the guide against current Docker and gateway security contracts, validate it on EasyRunner, and then let maintainers decide whether EasyRunner belongs in core platform docs.

## Reproduction Assessment

Yes for the review findings: source inspection of the PR head and current main shows the origin allowlist, device-auth, first-start config, and update-flow problems. I did not run a live EasyRunner deployment.

## Solution Assessment

No. The current patch is not the best way to solve this because it documents a dangerous auth bypass and an unverified first-start/update flow instead of current explicit config and security defaults.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Configure the public Control UI origin:** `docs/platforms/easyrunner.md:102`
  - body: The guide tells users to open `https://your-domain.com`, but the compose/config example never adds that exact browser origin to `gateway.controlUi.allowedOrigins`. Current non-loopback Control UI access requires explicit origins, so users can hit `origin not allowed` even after the container starts.
  - confidence: 0.92
- **[P1] Remove the device-auth bypass requirement:** `docs/platforms/easyrunner.md:262-280`
  - body: This section says `dangerouslyDisableDeviceAuth` is required behind a reverse proxy, but current security docs classify that flag as a break-glass severe downgrade. The guide should not normalize disabling Control UI device identity checks for a normal EasyRunner deployment.
  - confidence: 0.95
- **[P2] Create gateway config before describing it:** `docs/platforms/easyrunner.md:108-113`
  - body: The guide claims first start automatically creates durable `gateway.mode`, `trustedProxies`, device-auth, and plugin-slot config. The Docker command runs with `--allow-unconfigured`, which explicitly does not repair config, so the guide needs a pre-start config/onboarding step or a mounted config example.
  - confidence: 0.9
- **[P2] Rebuild the image during updates:** `docs/platforms/easyrunner.md:202-208`
  - body: The update command pulls source and redeploys but never rebuilds `localhost/openclaw:latest`. Because the compose file runs that local image tag, EasyRunner can keep launching the old image after users follow this update flow.
  - confidence: 0.87
