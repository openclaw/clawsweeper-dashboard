## Summary

Keep open: the EasyRunner docs are not implemented on current main, and this branch would still publish incorrect and security-sensitive gateway setup guidance for public Control UI access.

## What This Changes

This PR adds an EasyRunner deployment guide and links it from docs navigation and the platforms index.

## Best Possible Solution

A publishable guide should match current container gateway setup: pre-start config, explicit public allowedOrigins, no device-auth bypass, update-time image rebuilds, and verified EasyRunner deployment proof.

## Reproduction Assessment

Yes for the review findings: source inspection of the PR head and current main shows the missing public origin, unsafe device-auth guidance, unsupported durable-config claim, stale update flow, and incorrect memory-core workaround. I did not run a live EasyRunner deployment.

## Solution Assessment

No. The current patch is not the best way to solve the docs gap because it mixes an otherwise useful platform guide with unsafe and outdated gateway setup instructions.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Configure the public Control UI origin:** `docs/platforms/easyrunner.md:102`
  - body: The guide tells users to open `https://your-domain.com`, but neither the compose example nor the config section adds that origin to `gateway.controlUi.allowedOrigins`. Current startup seeding only covers localhost/127.0.0.1, so browser sessions from the public EasyRunner domain will fail origin checks.
  - confidence: 0.92
- **[P1] Remove the device-auth bypass requirement:** `docs/platforms/easyrunner.md:280`
  - body: This documents `dangerouslyDisableDeviceAuth` as required for reverse-proxy access, but current security docs classify it as break-glass only and a severe security downgrade. The guide should use normal token/password or trusted-proxy setup instead of telling production users to disable device identity checks.
  - confidence: 0.96
- **[P2] Create gateway config before describing it:** `docs/platforms/easyrunner.md:108-113`
  - body: The guide says first start automatically creates durable config with gateway mode, trusted proxies, device-auth bypass, and memory settings. The Docker command uses `gateway --allow-unconfigured`, which bypasses the guard and does not repair or persist that config, so users need an explicit pre-start config/onboard step or mounted config example.
  - confidence: 0.9
- **[P2] Rebuild the local image during updates:** `docs/platforms/easyrunner.md:203-208`
  - body: The update flow pulls source and redeploys but never rebuilds `localhost/openclaw:latest`. EasyRunner can keep launching the previously built image, so the maintenance steps should rebuild before `er app deploy`.
  - confidence: 0.86
- **[P2] Remove the memory-core workaround:** `docs/platforms/easyrunner.md:236-248`
  - body: The troubleshooting section says the default `memory-core` plugin is not available in the container and tells users to set `plugins.slots.memory` to `none`. Current plugin docs list memory-core as included in OpenClaw, so this would unnecessarily disable the default memory plugin instead of documenting a real fix.
  - confidence: 0.82
